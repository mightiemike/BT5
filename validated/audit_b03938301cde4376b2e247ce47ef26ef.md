Audit Report

## Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary

`DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool." However, `beforeAddLiquidity` silently discards the `sender` argument and checks `owner` (the position beneficiary) instead. Because `owner` is caller-supplied via `MetricOmmPoolLiquidityAdder.addLiquidityExactShares`, any unprivileged address can bypass the allowlist by nominating any allowlisted address as `owner` while themselves remaining non-allowlisted. The admin-configured security boundary is fully defeated.

## Finding Description

`ExtensionCalling._beforeAddLiquidity` encodes and forwards both `sender` and `owner` to every configured extension: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` names the first parameter (the actual depositing `sender`) as unnamed/discarded and gates only on `owner`: [2](#0-1) 

The admin-facing setter names the parameter `depositor`, confirming the intent was to gate by the depositing actor, not the position beneficiary: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` and discards the second (recipient) parameter: [4](#0-3) 

The bypass path is `MetricOmmPoolLiquidityAdder.addLiquidityExactShares`, which accepts a fully attacker-controlled `owner` and only validates `owner != address(0)`: [5](#0-4) 

`_addLiquidity` then passes `positionOwner` (the attacker-supplied value) directly to the pool as the `owner` argument: [6](#0-5) 

Because the extension checks `allowedDepositor[pool][owner]` and `owner` is attacker-supplied, the gate is trivially satisfied by naming any allowlisted address as `owner`. The actual payer (`msg.sender`, stored as `payer` in the transient context) is never validated against the allowlist.

## Impact Explanation

The deposit allowlist is an admin-configured security boundary used for KYC-gated pools, institutional-only pools, or pools where uncontrolled LP entry enables price manipulation. Because the check fires on `owner` rather than `sender`, any unprivileged address can deposit into the pool by nominating an allowlisted address as the position owner. The actual payer is never validated. This is a direct admin-boundary break bypassed by an unprivileged path, matching the allowed impact category.

## Likelihood Explanation

The bypass requires no special privilege, no flash loan, and no oracle manipulation. Any EOA or contract can call `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with an allowlisted `owner` address. Allowlisted addresses are public (emitted in `AllowedToDepositSet` events). Likelihood is high.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the first, currently unnamed parameter) instead of `owner`, mirroring the pattern in `SwapAllowlistExtension`:

```solidity
// Fixed:
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate by both the depositor and the position owner, both should be checked.

## Proof of Concept

```
Setup:
  - Pool deployed with DepositAllowlistExtension as beforeAddLiquidity hook
  - Admin calls setAllowedToDeposit(pool, alice, true)
  - Bob (not allowlisted) holds token0 and token1 and has approved LiquidityAdder

Attack:
  1. Bob calls:
       liquidityAdder.addLiquidityExactShares(
           pool,
           alice,   // allowlisted owner; Bob is the actual payer
           salt,
           deltas,
           maxAmount0,
           maxAmount1,
           ""
       )

  2. LiquidityAdder calls pool.addLiquidity(alice, salt, deltas, abi.encode(KIND_PAY), "")

  3. Pool calls _beforeAddLiquidity(sender=LiquidityAdder, owner=alice, ...)

  4. DepositAllowlistExtension checks allowedDepositor[pool][alice] → true → passes

  5. Pool issues callback to LiquidityAdder; LiquidityAdder pulls tokens from Bob (payer)

  6. Alice receives the LP position; Bob's tokens are consumed.
     The allowlist check on the actual depositor (Bob) was never performed.
```

The allowlist is fully bypassed. Any non-allowlisted address can deposit into a restricted pool by supplying any allowlisted address as `owner`.
