### Title
`SwapAllowlistExtension` checks the router address instead of the actual user, allowing any user to bypass per-user swap restrictions via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

The `SwapAllowlistExtension` gates swaps by checking the `sender` argument passed by the pool's `beforeSwap` hook. That argument is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool sees the **router** as `msg.sender`, not the actual user. The extension therefore checks whether the router is allowlisted, not whether the individual user is allowlisted. Any user who is not on the allowlist can bypass the restriction by calling through the router if the router itself is allowlisted.

### Finding Description

**Actor binding in `SwapAllowlistExtension`**

The extension's `beforeSwap` hook receives `sender` and checks it against the per-pool allowlist:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension caller) and `sender` is the first argument the pool passes to the hook.

**What the pool passes as `sender`**

In `MetricOmmPool.swap()`, the pool calls `_beforeSwap(msg.sender, recipient, ...)`:

```solidity
_beforeSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    ...
);
``` [2](#0-1) 

So `sender` in the extension equals whoever called `pool.swap()`.

**What the router passes as the pool's `msg.sender`**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly:

```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
);
``` [3](#0-2) 

The pool's `msg.sender` is the **router contract address**, not the end user. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

**Contrast with `DepositAllowlistExtension`**

The deposit extension correctly checks `owner`, which is a separate parameter the pool passes independently of `msg.sender`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [4](#0-3) 

The pool's `addLiquidity` passes `owner` as a distinct argument, so the extension can gate the actual economic actor regardless of who the payer/operator is. No equivalent "actual user" argument exists in the swap path.

### Impact Explanation

A pool admin who deploys a curated pool (e.g., KYC-gated, institution-only) and configures `SwapAllowlistExtension` with a per-user allowlist will naturally also allowlist the `MetricOmmSimpleRouter` so that approved users can trade through the standard interface. Once the router is allowlisted, **any** address—including those explicitly excluded from the per-user allowlist—can bypass the restriction by calling `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the router. The curation policy is completely defeated. This constitutes a broken core pool functionality and admin-boundary break: the pool admin's configured access control is bypassed by an unprivileged path.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface documented and deployed alongside the protocol. A pool admin who wants to restrict swaps to specific users but still allow those users to use the standard router must allowlist the router. This is the expected operational configuration. The bypass is therefore reachable in every realistic curated-pool deployment.

### Recommendation

Pass the actual initiating user through the swap hook so the extension can gate the correct actor. One approach: add an `initiator` field to the `beforeSwap` signature (analogous to `owner` in `beforeAddLiquidity`) and have the router forward `msg.sender` (the end user) as that field. Alternatively, the extension can read the payer from the router's transient callback context, but this creates an undesirable coupling between the extension and a specific router implementation.

A simpler short-term fix: document that `SwapAllowlistExtension` gates the **direct caller of `pool.swap()`** (i.e., the router), not the end user, and that per-user enforcement requires calling the pool directly or using a router that forwards the user identity.

### Proof of Concept

```solidity
// Pool is configured with SwapAllowlistExtension.
// Admin allowlists the router (so approved users can trade via the standard interface).
// Admin does NOT allowlist attacker.

swapExtension.setAllowedToSwap(address(pool), address(router), true);
// attacker is NOT in the allowlist

// Direct call from attacker → reverts (correct)
vm.prank(attacker);
pool.swap(attacker, true, 1000, 0, "", "");
// → NotAllowedToSwap

// Router call from attacker → succeeds (bypass)
vm.prank(attacker);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: address(token0),
    tokenOut: address(token1),
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: attacker,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// → succeeds: extension sees sender = router (allowlisted), not attacker
```

The extension checks `allowedSwapper[pool][router]` which is `true`, so the swap proceeds even though `attacker` is not on the allowlist. [5](#0-4) [2](#0-1) [6](#0-5)
