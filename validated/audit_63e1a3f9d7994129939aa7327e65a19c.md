### Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original user. The extension therefore checks whether the **router** is allowlisted, not whether the **user** is allowlisted. This creates a binary failure: either the router is allowlisted (any user bypasses the gate) or it is not (allowlisted users cannot use the router at all).

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the extension's caller) and `sender` is whatever address called `pool.swap()`: [3](#0-2) 

When a user routes through `MetricOmmSimpleRouter`, the call chain is:

```
user → MetricOmmSimpleRouter.exactInput*()
     → pool.swap(msg.sender = router, ...)
     → extension.beforeSwap(sender = router, ...)
     → allowedSwapper[pool][router]  ← user identity is lost
```

The allowlist mapping is keyed by `(pool, swapper)` and is intended to gate individual user addresses: [4](#0-3) 

The pool admin sets per-user entries via `setAllowedToSwap(pool, swapper, allowed)`: [5](#0-4) 

Because the router is a public, permissionless contract, two mutually exclusive failure modes exist:

1. **Router NOT allowlisted**: Allowlisted users who route through `MetricOmmSimpleRouter` are blocked — the router's address fails the check. Core swap functionality is broken for the intended user population.
2. **Router IS allowlisted** (admin adds it to enable router-mediated swaps): `allowedSwapper[pool][router] = true` passes for every caller of the router, so any unprivileged user can bypass the allowlist entirely by routing through the public router.

The `DepositAllowlistExtension` does not share this flaw — it correctly gates by `owner` (the position owner, not the payer/sender): [6](#0-5) 

### Impact Explanation

A pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties (e.g., KYC-gated, institutional, or whitelist-only pools) loses that restriction entirely once the router is allowlisted. Any unprivileged address can execute swaps against LP assets by calling the public `MetricOmmSimpleRouter`. LP principal is exposed to unrestricted trading that the pool admin explicitly intended to prevent. This is a direct admin-boundary break with fund-impacting consequences: unauthorized swaps drain LP assets at oracle-derived prices.

### Likelihood Explanation

The trigger is fully unprivileged — `MetricOmmSimpleRouter` is a public contract with no access control. The only precondition is that the pool admin allowlists the router, which is the natural action any admin would take when they want allowlisted users to be able to use the standard periphery router. The flaw is structural and requires no special timing or oracle conditions.

### Recommendation

The extension must recover the original user identity rather than relying on the pool's `msg.sender`. Two approaches:

1. **Pass the original user through `extensionData`**: The router encodes the actual user address in `extensionData`; the extension decodes and checks it. This requires a trusted encoding convention.
2. **Check `sender` against the allowlist at the router level and revert before calling the pool**: The router reads `isAllowedToSwap(pool, msg.sender)` and reverts if the user is not allowed, so the pool call never reaches the extension with a disallowed user.
3. **Preferred — use a transient-storage or callback pattern**: Store the original user in transient storage before the pool call and read it inside the extension hook, similar to how `MetricOmmPoolLiquidityAdder` uses `_setPayContext` / `_clearPayContext` for payer tracking. [7](#0-6) 

### Proof of Concept

1. Pool is deployed with `SwapAllowlistExtension` configured on `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is intended to swap.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — necessary so Alice can use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInput*(...)` targeting the pool.
5. Router calls `pool.swap(recipient=bob, ...)` with `msg.sender = router`.
6. Pool calls `extension.beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
8. Bob's swap executes against LP assets. The allowlist is fully bypassed.
