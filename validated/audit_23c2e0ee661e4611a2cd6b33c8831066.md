Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of originating user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][originalUser]`. A pool admin who allowlists the router to enable router-mediated swaps for legitimate users inadvertently opens the gate to every user, completely nullifying per-user curation.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender`:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`. When invoked via `MetricOmmSimpleRouter`, `msg.sender` is the router contract address, not the end user.

**Step 2 — `ExtensionCalling._beforeSwap` forwards unchanged:**

`ExtensionCalling._beforeSwap` encodes `sender` (the router) into the `abi.encodeCall` payload and dispatches it to every configured extension via `_callExtensionsInOrder`. The original user's address is never present in this call.

**Step 3 — Extension checks the wrong identity:**

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool (the extension is called by the pool), and `sender` = router. The effective check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`.

**Step 4 — Router never forwards the originating user:**

`MetricOmmSimpleRouter.exactInputSingle` stores `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext(..., msg.sender, ...)`), but calls `pool.swap(params.recipient, ...)` directly. The pool's `swap()` signature fixes `sender = msg.sender` (the router). There is no mechanism to pass the originating user's address as `sender`.

**Existing guards are insufficient:** `BaseMetricExtension.beforeSwap` has an `onlyPool` modifier that only verifies the caller is a registered pool — it does not validate which user initiated the swap. No other guard in the extension or pool checks the originating user's identity.

**Exploit flow (Case B — router allowlisted):**
1. Pool admin deploys pool with `SwapAllowlistExtension` and allowlists Alice: `allowedSwapper[pool][alice] = true`
2. Pool admin also allowlists the router so Alice can use it: `allowedSwapper[pool][router] = true`
3. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`
4. Router calls `pool.swap(...)` → pool sees `msg.sender = router`
5. Pool calls `_beforeSwap(sender=router, ...)`
6. Extension checks `allowedSwapper[pool][router] == true` → passes
7. Bob's swap executes on the curated pool

## Impact Explanation

When the router is allowlisted (the only way to enable router-mediated swaps for legitimate users), the per-user allowlist is completely bypassed. Any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant) and swap on a pool designed to exclude them. This breaks the core invariant of the `SwapAllowlistExtension` — "only allowlisted users may swap on this pool" — and allows unauthorized users to drain LP value or violate the pool's economic design. This constitutes broken core pool functionality causing potential loss of funds to LPs.

## Likelihood Explanation

The trigger is fully unprivileged. Any user who knows the pool uses `SwapAllowlistExtension` and that the router is allowlisted can call `MetricOmmSimpleRouter.exactInputSingle` with no special permissions. The router is a public, documented periphery contract. The pool admin's only recourse is to not allowlist the router at all, which breaks the UX for all legitimate users who need router-mediated swaps (multi-hop paths, WETH-unwrap flows). The two failure modes are binary and unavoidable given the current design.

## Recommendation

The extension must check the originating user, not the immediate caller of `pool.swap()`. The preferred fix is to have `pool.swap()` accept an explicit `swapper` parameter (separate from `msg.sender`) that the router fills with `msg.sender` (the originating user), and have `SwapAllowlistExtension.beforeSwap` check that field. As a short-term mitigation, the extension documentation must warn that allowlisting the router opens the gate to all users, and curated pools should not allowlist the router until a protocol-level fix is in place.

## Proof of Concept

```
// Foundry integration test outline:
// 1. Deploy pool with SwapAllowlistExtension
// 2. Pool admin: swapExtension.setAllowedToSwap(pool, alice, true)
// 3. Pool admin: swapExtension.setAllowedToSwap(pool, address(router), true)
//    (necessary to allow Alice to use the router)
// 4. vm.prank(bob); // bob is NOT allowlisted
//    router.exactInputSingle(ExactInputSingleParams({pool: pool, ...}));
// 5. Assert: swap succeeds (no NotAllowedToSwap revert)
//    → Bob bypassed the allowlist entirely via the router
```

The existing test `FullMetricExtension.t.sol::test_allowedSwapSucceeds` confirms the direct-call path works correctly (allowlisting `callers[0]` which is the direct `msg.sender`), but no test covers the router-mediated path where `msg.sender` of `pool.swap()` differs from the originating user.
