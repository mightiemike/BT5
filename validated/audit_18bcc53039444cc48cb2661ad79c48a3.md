Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Breaking Allowlist Enforcement — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap()` passes `msg.sender` (the immediate caller) as `sender` to every extension hook. When users swap through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the router, so `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This produces two fund-impacting failure modes: (1) if the router is allowlisted, every unprivileged user on the internet bypasses the curation gate; (2) if individual users are allowlisted but the router is not, those users are completely locked out of swapping because EOAs cannot call `pool.swap()` directly (the pool requires a `metricOmmSwapCallback` implementation from `msg.sender`).

## Finding Description

**Root cause — pool passes `msg.sender` as `sender`:**

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the first argument: [1](#0-0) 

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()`, the pool's `msg.sender` is the router contract, not the end user: [2](#0-1) 

**Extension checks the wrong address:**

`SwapAllowlistExtension.beforeSwap` uses the `sender` argument (which is the router) to look up `allowedSwapper[pool][sender]`: [3](#0-2) 

The lookup is therefore `allowedSwapper[pool][router]`, never `allowedSwapper[pool][user]`.

**EOAs cannot bypass the router:** After executing the swap, the pool calls `IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(...)` on the direct caller. EOAs cannot implement this interface, so they are forced to use the router. There is no practical direct-call alternative for end users. [4](#0-3) 

**Multi-hop paths are equally affected:** All router entry points (`exactInput` loop, `exactOutputSingle`, `exactOutput` recursive callback) call `pool.swap()` with the router as `msg.sender`: [5](#0-4) [6](#0-5) 

## Impact Explanation

**Lockout (no admin misconfiguration required):** A pool admin deploys a pool with `SwapAllowlistExtension` and allowlists specific users via `setAllowedToSwap(pool, alice, true)`. Alice calls `router.exactInputSingle(...)`. The extension checks `allowedSwapper[pool][router] == false` and reverts with `NotAllowedToSwap`. Alice cannot swap at all — she cannot call `pool.swap()` directly because the pool requires a callback implementation. The core swap flow is completely broken for all individually-allowlisted users, matching the "broken core pool functionality causing unusable swap flows" impact criterion.

**Bypass (router allowlisted):** If the admin allowlists the router as a trusted intermediary (`setAllowedToSwap(pool, router, true)`), every unprivileged address on the internet can call `router.exactInputSingle(...)` and pass the extension check, trading against LP positions on a pool intended to be closed to them. This is a direct loss of LP principal to unauthorized counterparties.

## Likelihood Explanation

The lockout path requires zero attacker action — it is the default outcome whenever a pool admin uses `SwapAllowlistExtension` with per-user allowlisting (the extension's documented purpose). Any allowlisted user who attempts to use the canonical router interface triggers the lockout. The bypass path requires only that the admin allowlists the router, which is a natural configuration decision given that the router is the primary user-facing interface. The unit tests for `SwapAllowlistExtension` call the extension directly with `vm.prank(address(pool))` and pass the swapper address explicitly, so the router path is entirely untested and the mismatch is invisible in CI: [7](#0-6) 

## Recommendation

The pool should forward the economically relevant actor as `sender`. The standard approach is a trusted-forwarder pattern: the router encodes the original `msg.sender` into `extensionData`, and the extension decodes and verifies it (checking that `msg.sender` of `beforeSwap` is a factory-registered router before trusting the forwarded address). Alternatively, the pool can accept an explicit `sender` override parameter validated by a signed permit. At minimum, the `SwapAllowlistExtension` NatSpec must document that `sender` is the direct `pool.swap()` caller, not the end user, so pool admins can make an informed configuration decision.

## Proof of Concept

**Lockout path (no attacker required):**
1. Deploy a pool with `SwapAllowlistExtension` in the `beforeSwap` slot.
2. Call `extension.setAllowedToSwap(pool, alice, true)` — allowlist Alice directly (intended use).
3. Alice calls `router.exactInputSingle({pool: pool, recipient: alice, ...})`.
4. Router calls `pool.swap(...)` — pool's `msg.sender` is the router.
5. Pool calls `extension.beforeSwap(router, ...)`. Extension checks `allowedSwapper[pool][router] == false` → `NotAllowedToSwap` revert.
6. Alice cannot swap through the router. She cannot call `pool.swap()` directly (no callback implementation). Alice is completely locked out despite being explicitly allowlisted.

**Bypass path:**
1. Same setup; call `extension.setAllowedToSwap(pool, router, true)` instead.
2. As a non-allowlisted EOA (`attacker`), call `router.exactInputSingle({pool: pool, recipient: attacker, ...})`.
3. Extension checks `allowedSwapper[pool][router] == true` → passes.
4. Attacker receives tokens from a curated pool that was supposed to be closed to them.
