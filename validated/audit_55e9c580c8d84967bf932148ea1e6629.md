Audit Report

## Title
SwapAllowlistExtension Gates on Router Address Instead of Real User, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap()` sets to its own `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end user. A pool admin who allowlists the router address to enable standard periphery access inadvertently grants every user on the internet the ability to bypass the allowlist, completely voiding the access-control boundary.

## Finding Description
The root cause is a three-step identity loss:

1. `MetricOmmPool.swap()` passes `msg.sender` verbatim as the `sender` argument to `_beforeSwap()`. [1](#0-0) 

2. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value from step 1 â€” the router address, not the end user. [2](#0-1) 

3. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` at the pool level. The original caller's address is stored only in transient callback state via `_setNextCallbackContext` and is never forwarded to the extension. [3](#0-2) 

The same identity loss applies to `exactOutputSingle` and `exactOutput` entry points. [4](#0-3) 

No existing guard compensates for this: the extension has no mechanism to recover the original caller, and the pool does not forward any caller-identity data beyond the raw `msg.sender`. [5](#0-4) 

## Impact Explanation
This is a direct admin-boundary break. The `SwapAllowlistExtension` is the primary access-control boundary for curated pools. When the router is allowlisted (the natural operational step to enable standard periphery access for allowlisted users), the check `allowedSwapper[pool][router] == true` passes for every caller of the router, regardless of whether that caller was ever individually allowlisted. Any unprivileged user can execute swaps on pools restricted to specific counterparties (KYC, institutional, compliance-gated), enabling unauthorized trading, potential bad-price extraction, and regulatory breach on curated pools. Severity: **High**.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical, documented swap entry point. Any pool admin deploying `SwapAllowlistExtension` who also wants allowlisted users to access the pool through the standard router will allowlist the router address â€” this is the expected operational pattern. No special knowledge or adversarial setup is required; any user calling `exactInputSingle` through the router on such a pool silently bypasses the guard. The failure mode is triggered by normal, expected usage.

## Recommendation
The extension must check the original user's identity, not the intermediary's. The cleanest fix: the router encodes `abi.encode(msg.sender)` as a prefix in `extensionData` before calling `pool.swap()`; the extension decodes and checks that address when `msg.sender` (the pool) is the caller. This requires a convention between router and extension but preserves the allowlist invariant without breaking the existing call path. Alternatively, document and enforce that `SwapAllowlistExtension` is incompatible with router-mediated swaps and revert at the extension level when called from a known router address.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (intending to allow standard periphery access for allowlisted users)
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  - attacker calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  - Router calls pool.swap(...) â€” msg.sender at pool = router address
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true â†’ PASSES
  - Attacker's swap executes on the curated pool despite never being allowlisted

Wrong value: allowedSwapper[pool][router] = true is evaluated at line 37 of
SwapAllowlistExtension.sol instead of allowedSwapper[pool][attacker] = false.
The allowlist is completely bypassed for any user who routes through the router.
```
