### Title
Phantom-Index Merkle Proof Bypass via Odd-Length Leaf Duplication — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` accepts an attacker-supplied `tx_index` with no upper-bound check against the actual number of transactions in the block. Because `compute_root_from_merkle_proof` is a pure positional computation and the Bitcoin Merkle tree duplicates the last leaf for odd-length transaction lists, the proof for the real last transaction at index `n-1` produces an identical root when replayed with the phantom index `n`. The function returns `true` for a transaction position that does not exist in the committed block.

---

### Finding Description

**`compute_root_from_merkle_proof` — no bounds check on `transaction_position`** [1](#0-0) 

The function accepts any `transaction_position` value and uses only its parity at each level to decide left/right ordering. There is no validation that the position is within `[0, tx_count)`.

**`merkle_proof_calculator` — last-leaf duplication for odd-length lists** [2](#0-1) 

When the transaction list has odd length `n`, the last element is appended as its own duplicate before pairing. This means the sibling of the real last leaf at index `n-1` (even) is `tx[n-1]` itself.

**`verify_transaction_inclusion` — passes `tx_index` directly to the proof function** [3](#0-2) 

The only guards are: confirmations check, block-in-mainchain check, and non-empty proof check. There is no check that `tx_index < number_of_transactions`. The contract does not store transaction counts — only the Merkle root.

**Concrete arithmetic for a 3-tx block `[A, B, C]`:**

Proof for real index 2 (`merkle_proof_calculator([A,B,C], 2)`):
- Level 0: `[A, B, C]` → odd → pad to `[A, B, C, C]`; position 2 is even, sibling = `C`; proof step 1 = `C`
- Level 1: `[hash(A,B), hash(C,C)]`; position 1 is odd, sibling = `hash(A,B)`; proof step 2 = `hash(A,B)`
- Result: `proof = [C, hash(A,B)]`

`compute_root_from_merkle_proof(C, 2, [C, hash(A,B)])`:
- pos=2 (even): `compute_hash(C, C)` = `hash(C,C)`; pos→1
- pos=1 (odd): `compute_hash(hash(A,B), hash(C,C))` = root ✓

`compute_root_from_merkle_proof(C, 3, [C, hash(A,B)])`:
- pos=3 (odd): `compute_hash(proof[0]=C, current=C)` = `compute_hash(C, C)` = `hash(C,C)`; pos→1
- pos=1 (odd): `compute_hash(hash(A,B), hash(C,C))` = root ✓

Both calls return the identical canonical Merkle root because `compute_hash(C, C)` is the same regardless of argument order when both arguments are equal. The phantom index `n` produces the same intermediate hash as the real index `n-1`.

### Impact Explanation

`verify_transaction_inclusion` returns `true` for `tx_index = n` (a position that does not exist in the block) when the block has an odd number of transactions `n`. Any downstream contract or application that relies on this function to confirm a transaction's inclusion in a specific block position can be deceived into accepting a fabricated inclusion claim. This breaks the core SPV invariant the contract is designed to enforce.

---

### Likelihood Explanation

- `verify_transaction_inclusion` is a public NEAR contract method with no `#[private]` or `#[trusted_relayer]` restriction. [4](#0-3) 
- Any NEAR account can call it when the contract is not paused.
- The attacker needs only: a canonical block with an odd transaction count stored in the mainchain (a routine condition), the real last transaction's hash, and its legitimate proof — all of which are publicly available from any Bitcoin node.
- No privileged role, leaked key, or social engineering is required.
- The deprecated status of `verify_transaction_inclusion` does not prevent calls; it is still deployed and executable. `verify_transaction_inclusion_v2` also calls it internally after its coinbase check, so the phantom-index path is reachable through v2 as well. [5](#0-4) 

---

### Recommendation

1. **Add an upper-bound check on `tx_index`** in `verify_transaction_inclusion`. Since the contract does not store transaction counts, the proof length encodes the tree depth: the maximum valid index is `2^(proof.len()) - 1`. Reject any `tx_index >= 2^(merkle_proof.len())` or, more precisely, enforce `tx_index < (1 << merkle_proof.len())` and additionally require that the index is consistent with the proof length for the claimed tree size.
2. **Store or accept the transaction count** as part of `ProofArgs` and validate `tx_index < tx_count` before calling `compute_root_from_merkle_proof`.
3. **Add a bounds check inside `compute_root_from_merkle_proof`** itself so the library cannot be misused by other callers.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
