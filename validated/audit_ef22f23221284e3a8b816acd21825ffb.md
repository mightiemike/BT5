### Title
Rewards Pool Balance Permanently Locked With No Withdrawal Path — (`x/tieredrewards/types/keys.go`, `x/tieredrewards/keeper/abci.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

The `rewards_pool` module account in `x/tieredrewards` accumulates tokens but exposes no governance-gated message to withdraw the remaining balance. Tokens can only leave the pool via two indirect paths: bonus reward claims (requires active delegated positions) and `BeginBlocker` top-ups (requires a fee-collector shortfall). When all positions exit and the fee collector has no shortfall, any remaining pool balance is permanently locked with no recovery path.

---

### Finding Description

The `RewardsPoolName = "rewards_pool"` module account is defined as the sole source of bonus rewards. [1](#0-0) 

The pool is funded externally — anyone can send tokens to the module address via a standard `bank.Send`. The module's `MsgServer` interface exposes fourteen messages, none of which is a pool-withdrawal message: [2](#0-1) 

The only two drain paths are:

**Path 1 — Bonus reward claims** (`processEventsAndClaimBonus`): sends tokens from `rewards_pool` to a position owner. This path is only reachable when at least one delegated position exists and the pool has sufficient balance. [3](#0-2) 

**Path 2 — BeginBlocker top-up** (`topUpBaseRewards`): transfers the fee-collector shortfall from `rewards_pool` to `x/distribution`. This path is only triggered when `TargetBaseRewardsRate > 0` **and** the fee collector balance is below the target per-block reward. [4](#0-3) 

When both drain paths are inactive simultaneously — no active positions and no fee-collector shortfall — the pool balance is unreachable. Governance can reach this state by marking all tiers `CloseOnly` (blocking new positions) and waiting for all existing positions to exit. Once that state is reached, any remaining pool balance is permanently locked.

The `BeforeRedelegationSlashed` hook compounds this by silently forfeiting accrued bonus rewards into the pool when the pool is insufficient, further increasing the stranded balance: [5](#0-4) 

---

### Impact Explanation

Tokens sent to `rewards_pool` — whether by the protocol, a governance proposal, or any external account — can only be recovered through normal operational drain. If the system is wound down (all tiers `CloseOnly`, all positions exited) while the fee collector has no shortfall, the remaining pool balance is permanently locked in the module account with no on-chain recovery mechanism. The corrupted value is the `rewards_pool` module account balance.

---

### Likelihood Explanation

Medium. The scenario requires governance to mark all tiers `CloseOnly` and for all positions to exit — both realistic steps in a protocol sunset. The fee-collector-no-shortfall condition is also realistic on a high-fee chain or when `x/mint` inflation is disabled. The `BeforeRedelegationSlashed` silent-forfeit path can strand additional funds without any governance action.

---

### Recommendation

Add a governance-gated `MsgWithdrawFromPool(authority, recipient, amount)` message to `x/tieredrewards` that allows the `x/gov` module account to transfer any remaining `rewards_pool` balance to a specified address. This mirrors the pattern used by other Cosmos SDK modules (e.g., `x/distribution` community pool spend) and eliminates the permanent-lock risk.

---

### Proof of Concept

1. Governance funds `rewards_pool` with a large amount via `bank.Send` to the module address.
2. Governance submits `MsgUpdateTier` for every tier setting `CloseOnly = true` — blocking new positions.
3. All existing position owners call `MsgTriggerExitFromTier` → `MsgTierUndelegate` → `MsgWithdrawFromTier` (or `MsgExitTierWithDelegation`). All positions are deleted.
4. The chain has sufficient transaction fees so the fee collector never falls below the `TargetBaseRewardsRate` target — `topUpBaseRewards` never fires.
5. Query `RewardsPoolBalances`: the pool still holds the original funded amount.
6. No message in the `MsgServer` interface can move those tokens. The balance is permanently locked. [6](#0-5) [7](#0-6)

type MsgServer interface {
	// UpdateParams defines a governance operation for updating the x/tieredrewards module
	// parameters. The authority is hard-coded to the x/gov module account.
	UpdateParams(context.Context, *MsgUpdateParams) (*MsgUpdateParamsResponse, error)
	// AddTier creates a new tier. Authority-gated.
	AddTier(context.Context, *MsgAddTier) (*MsgAddTierResponse, error)
	// UpdateTier updates an existing tier. Authority-gated.
	UpdateTier(context.Context, *MsgUpdateTier) (*MsgUpdateTierResponse, error)
	// DeleteTier removes a tier. Authority-gated.
	DeleteTier(context.Context, *MsgDeleteTier) (*MsgDeleteTierResponse, error)
	// LockTier locks tokens into a tier, optionally delegating and/or triggering exit immediately.
	LockTier(context.Context, *MsgLockTier) (*MsgLockTierResponse, error)
	// CommitDelegationToTier allows a commitment of a delegation to a tier.
	CommitDelegationToTier(context.Context, *MsgCommitDelegationToTier) (*MsgCommitDelegationToTierResponse, error)
	// TierUndelegate begins undelegating a position's tokens from its validator.
	TierUndelegate(context.Context, *MsgTierUndelegate) (*MsgTierUndelegateResponse, error)
	// TierRedelegate moves a position's delegation from one validator to another.
	TierRedelegate(context.Context, *MsgTierRedelegate) (*MsgTierRedelegateResponse, error)
	// AddToTierPosition adds tokens to an existing position.
	AddToTierPosition(context.Context, *MsgAddToTierPosition) (*MsgAddToTierPositionResponse, error)
	// TriggerExitFromTier starts the exit commitment for a position.
	TriggerExitFromTier(context.Context, *MsgTriggerExitFromTier) (*MsgTriggerExitFromTierResponse, error)
	// ClearPosition clears exit state for a position so the owner can add funds again.
	ClearPosition(context.Context, *MsgClearPosition) (*MsgClearPositionResponse, error)
	// ClaimTierRewards claims base and bonus rewards for one or more delegated positions.
	ClaimTierRewards(context.Context, *MsgClaimTierRewards) (*MsgClaimTierRewardsResponse, error)
	// WithdrawFromTier withdraws locked tokens after the exit commitment has elapsed,
	// sends the position's amount back to the owner, and deletes the position.
	WithdrawFromTier(context.Context, *MsgWithdrawFromTier) (*MsgWithdrawFromTierResponse, error)
	// ExitTierWithDelegation exits a tier position by transferring the module's
	// delegation back to the owner on the same validator (no unbonding period).
	// Supports partial exits. Deletes the position if the full amount is transferred.
	ExitTierWithDelegation(context.Context, *MsgExitTierWithDelegation) (*MsgExitTierWithDelegationResponse, error)
}
