Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Enabling Full Allowlist Bypass — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`, who is called back to provide tokens) and instead checks `owner` (the LP-share recipient address supplied by the caller). Any attacker controlling one allowlisted address can deposit from a completely unauthorized address, fully defeating the restriction the extension is designed to enforce.

## Finding Description
In `MetricOmmPool.addLiquidity` (L191), the pool calls `_beforeAddLiquidity(msg.sender, owner, ...)`, passing the real caller as `sender` and the caller-supplied share recipient as `owner`. `ExtensionCalling._beforeAddLiquidity` (L95–98) faithfully encodes both into the hook calldata via `abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, ...))`.

`DepositAllowlistExtension.beforeAddLiquidity` (L32–42) receives `sender` as its first positional argument but leaves it unnamed (discarded), then evaluates only `allowedDepositor[msg.sender][owner]`. The actual token-providing address (`sender`) is never checked.

By contrast, `SwapAllowlistExtension.beforeSwap` (L31–41) correctly names and checks `sender`, ignoring the `recipient`. The structural inconsistency means the deposit guard checks the wrong actor.

Exploit path:
1. `attackerEOA` (not allowlisted) calls `pool.addLiquidity(attackerContract, salt, deltas, callbackData, extensionData)`.
2. Extension evaluates `allowedDepositor[pool][attackerContract]` → `true` → hook passes.
3. Pool issues the `metricOmmSwapCallback` to `attackerEOA` (the real `msg.sender`) to pull tokens.
4. LP shares are minted to `attackerContract`.
5. `attackerContract` calls `removeLiquidity(attackerContract, ...)` — the `msg.sender == owner` guard at L206 passes — recovering the tokens.

No existing guard prevents this: the allowlist check is the only gate, and it evaluates the wrong address.

## Impact Explanation
**Allowlist bypass (unauthorized deposit):** An unprivileged address that was never granted allowlist access can deposit into a restricted pool by nominating any allowlisted address as `owner`. This constitutes broken core pool functionality — the deposit restriction is entirely ineffective — and enables unauthorized LP positions in pools intended to be permissioned. The attacker retains full control of the position through `attackerContract` and can withdraw at will, representing direct loss of the pool admin's access-control invariant and potential fund-flow impact in pools where the allowlist is used to enforce KYC, whitelist, or counterparty restrictions.

**False block (router pattern broken):** If a pool admin allowlists a router as the authorized depositor but users are the `owner`, every router-mediated deposit reverts because `allowedDepositor[pool][user]` is `false` even though the router (the actual token provider) is allowlisted. This renders the pool unusable for any router-based liquidity flow, constituting broken core pool functionality.

## Likelihood Explanation
High. The bypass requires only that the attacker controls one allowlisted address — achievable by any participant who was ever granted allowlist access, or by social engineering a single allowlisted address. No oracle manipulation, price conditions, or privileged roles are needed. The false-block scenario triggers automatically and unconditionally whenever a router is used, which is the standard periphery pattern in this protocol.

## Recommendation
Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension.beforeSwap`:

```solidity
// Fix: check sender (actual token provider), not owner (share recipient)
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

  allowedDepositor[pool][attackerContract] = true
  allowedDepositor[pool][attackerEOA]      = false

Attack:
  // Step 1: attackerEOA calls addLiquidity with owner = attackerContract
  pool.addLiquidity(
      attackerContract,   // owner ← allowlisted, passes the guard
      salt,
      deltas,
      callbackData,
      extensionData
  );
  // msg.sender = attackerEOA (not allowlisted, but never checked)

  // Step 2: DepositAllowlistExtension.beforeAddLiquidity checks
  //         allowedDepositor[pool][attackerContract] → true → no revert

  // Step 3: Pool calls attackerEOA.metricOmmSwapCallback(...) to pull tokens
  //         attackerEOA transfers tokens to pool

  // Step 4: LP shares minted to attackerContract

  // Step 5: attackerContract calls removeLiquidity
  pool.removeLiquidity(attackerContract, salt, deltas, extensionData);
  // msg.sender == owner → passes NotPositionOwner check → tokens recovered

Result: attackerEOA (unauthorized) successfully deposited into a restricted pool.
