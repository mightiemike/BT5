Audit Report

## Title
`SwapAllowlistExtension` gates on router address instead of end-user, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. If the pool admin allowlists the router to enable router-mediated swaps for legitimate users, the per-user allowlist is silently bypassed for every user, including those explicitly excluded.

## Finding Description

**Root cause — wrong identity checked by the guard**

`SwapAllowlistExtension.beforeSwap` performs:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the pool calls the extension) and `sender` is the first argument the pool passes, which is `msg.sender` of `pool.swap()`.

**How the pool sets `sender`**

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
``` [2](#0-1) 

**How the router calls the pool**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the `msg.sender` of that call:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [3](#0-2) 

So when a user routes through the router, the extension receives `sender = router_address`, not the actual end-user.

**The bypass**

1. Pool admin deploys pool with `SwapAllowlistExtension` to restrict swaps to `alice`.
2. Admin calls `setAllowedToSwap(pool, alice, true)`.
3. Admin also calls `setAllowedToSwap(pool, router, true)` so that alice can use the router.
4. `bob` (not allowlisted) calls `router.exactInputSingle(...)` targeting the pool.
5. Router calls `pool.swap(...)` → pool calls `extension.beforeSwap(sender=router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. Bob successfully swaps in the restricted pool.

**Contrast with `DepositAllowlistExtension`**

The deposit allowlist correctly ignores `sender` and checks `owner` instead — the position owner explicitly passed by the caller — which is stable regardless of routing intermediary:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
``` [4](#0-3) 

The swap allowlist has no equivalent stable identity to fall back on.

## Impact Explanation
Any user can bypass a pool's per-user swap allowlist by routing through `MetricOmmSimpleRouter`. The pool admin cannot simultaneously (a) restrict swaps to specific counterparties and (b) support router-mediated swaps. If they allowlist the router to satisfy (b), the restriction in (a) is nullified entirely. Unauthorized users gain unrestricted swap access to a pool that was intended to be private, exposing LPs to swap flow they explicitly opted out of. In an oracle-priced pool this translates directly to LP value leakage from uninvited counterparties trading against the pool's liquidity — a direct loss of LP principal matching the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact categories.

## Likelihood Explanation
The scenario is highly likely in practice. Pools using `SwapAllowlistExtension` are explicitly designed for restricted counterparty access. The router is the standard user-facing entry point; pool admins who want allowlisted users to have a normal UX will allowlist the router. No documentation or NatSpec on `SwapAllowlistExtension` warns that allowlisting the router opens the gate to all users. The bypass requires no special privileges — any EOA can call the router with zero preconditions beyond the admin having allowlisted the router.

## Recommendation
The `SwapAllowlistExtension` must gate on an identity that is stable across routing intermediaries. Two viable approaches:

1. **Check `recipient` instead of `sender`**: For swaps, the `recipient` is the address that receives output tokens and is set by the end-user. Checking `allowedSwapper[pool][recipient]` would gate on the economic beneficiary rather than the routing intermediary. This is consistent with how `DepositAllowlistExtension` uses `owner` (the economic beneficiary) rather than `sender`.

2. **Decode end-user from `extensionData`**: Require the router to embed the actual caller's address in `extensionData` and verify it in the extension. The extension would decode the address and check it against the allowlist.

Option 1 is simpler and directly mirrors the `DepositAllowlistExtension` pattern.

## Proof of Concept

```
Setup:
  pool = deploy pool with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)
  admin calls setAllowedToSwap(pool, router, true)   // to let alice use the router

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({
      pool: pool,
      recipient: bob,
      zeroForOne: true,
      amountIn: X,
      ...
    })

  → router calls pool.swap(recipient=bob, ...)          [MetricOmmSimpleRouter.sol L72-80]
  → pool calls extension.beforeSwap(sender=router, ...) [MetricOmmPool.sol L230-240]
  → extension checks allowedSwapper[pool][router] == true ✓ [SwapAllowlistExtension.sol L37]
  → swap executes; bob receives output tokens

Result:
  bob swapped in a pool he was explicitly excluded from.
  The allowlist invariant is broken.
```
