Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` evaluates the router address as the swapper, allowing any user to bypass per-pool swap allowlists via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so `sender` resolves to the router address rather than the actual user. Any pool admin who allowlists the router to permit allowlisted users to trade through it simultaneously opens the gate to every non-allowlisted user, completely defeating the extension's access control guarantee.

## Finding Description

**Step 1 — Pool captures `msg.sender` as `sender`:**
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 231, passing the immediate caller of `pool.swap()` as the `sender` argument. [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` verbatim:**
`ExtensionCalling._beforeSwap` encodes `sender` directly into the `abi.encodeCall` for `IMetricOmmExtensions.beforeSwap` with no transformation. [2](#0-1) 

**Step 3 — Extension checks the wrong address:**
`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When the router is the caller, `sender` is the router address, not the user. [3](#0-2) 

**Step 4 — Router calls `pool.swap()` directly:**
`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making the router contract the `msg.sender` of the pool call. The actual user's address (`msg.sender` of `exactInputSingle`) is never forwarded to the pool as `sender`. [4](#0-3) 

**Step 5 — Same pattern in `exactInput` and `exactOutputSingle`/`exactOutput`:**
`exactInput` also calls `pool.swap()` directly in a loop, and `exactOutputSingle` does the same, all with the router as `msg.sender` to the pool. [5](#0-4) 

**Root cause:** The `sender` parameter in `beforeSwap` represents the immediate pool caller (the router), not the economic actor whose identity the allowlist is meant to gate. There is no mechanism in the call path to propagate the original user's address to the extension.

**Contrast with `DepositAllowlistExtension`:** The deposit extension checks `owner` (the position owner explicitly supplied by the caller), which is not subject to router indirection. The swap extension has no equivalent identity-preserving parameter. [6](#0-5) 

**The impossible dilemma for pool admins:**

| Router allowlist state | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot use the router at all |
| Router **allowlisted** | Every non-allowlisted user bypasses the guard via the router |

No configuration simultaneously allows allowlisted users to use the router and blocks non-allowlisted users.

## Impact Explanation

A pool deploying `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC'd addresses, institutional participants, or protocol-controlled addresses) has that restriction completely defeated by any user routing through `MetricOmmSimpleRouter`. The non-allowlisted user receives full swap execution — including access to pool liquidity and oracle-priced fills — identical to an allowlisted user. This is a direct admin-boundary break: an unprivileged path bypasses the pool admin's configured access control, and the allowlist invariant the extension is designed to enforce is broken. This matches the "Allowlist path" smart audit pivot explicitly: "swap allowlist checks must cover the exact actor/action intended and cannot be bypassed through router."

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard supported periphery path, callable permissionlessly by any address. No special role, token balance, or prior state is required beyond having tokens to swap. The bypass requires a single `exactInputSingle` call targeting the restricted pool. Any user aware of the pool's existence can execute this. The bypass is repeatable on every block.

## Recommendation

The extension must identify the economic actor — the address whose funds are at risk and whose identity the allowlist is meant to gate — not the immediate caller of `pool.swap`. Two viable approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension, and the extension must verify the data source is trustworthy (e.g., by checking `msg.sender` is a known router).
2. **Make the extension router-aware**: The extension checks `sender` when `sender` is not a known router; when `sender` is a known router, it requires the router to attest the real user via `extensionData` and validates that claim.

A simpler short-term mitigation is to document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and must only be used with direct pool calls, but this does not fix the underlying flaw.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is intended to swap.
3. Admin calls `setAllowedToSwap(pool, router, true)` — necessary so Alice can use `MetricOmmSimpleRouter`.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)` with `msg.sender = router`.
6. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender=router, ...)`.
7. `SwapAllowlistExtension.beforeSwap` receives `sender = router`; evaluates `allowedSwapper[pool][router]` → `true`.
8. Bob's swap executes successfully despite not being on the allowlist.

**Confirming the dilemma:** If the admin omits step 3 (does not allowlist the router), Alice's call to `exactInputSingle` also fails at step 7 with `NotAllowedToSwap`, confirming there is no valid configuration.

A Foundry integration test can reproduce this by: deploying the pool with the extension, setting Alice as the only allowed swapper, setting the router as allowed, then calling `exactInputSingle` from a Bob address and asserting the call succeeds (swap executes) rather than reverting.
