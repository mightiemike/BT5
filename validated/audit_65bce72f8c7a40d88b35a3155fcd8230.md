### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Validation Required to Prevent 64-Byte Merkle Forgery - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR entry point despite being deprecated. It omits the coinbase Merkle proof check that `verify_transaction_inclusion_v2` introduces specifically to block the 64-byte transaction Merkle forgery attack. Any unprivileged caller can invoke v1 directly, bypassing the only guard that prevents a crafted internal Merkle node from being accepted as a valid transaction.

---

### Finding Description

The contract exposes two SPV proof verification entry points:

- `verify_transaction_inclusion` (v1) — marked `#[deprecated]`, but still `pub` and gated only by `#[pause]`. It validates a Merkle proof by computing the root from `tx_id` and comparing it to the block's `merkle_root`. The sole structural guard is `require!(!args.merkle_proof.is_empty(), ...)`.
- `verify_transaction_inclusion_v2` — the secure replacement. Before delegating to v1, it first validates a coinbase Merkle proof at index 0 against the same `merkle_root`, which closes the 64-byte forgery path. [1](#0-0) 

The `#[deprecated]` attribute in Rust emits only a compiler warning; it does not restrict runtime callability. The function remains a fully reachable NEAR method for any external account. [2](#0-1) 

The contract's own inline warning acknowledges the consequence:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

The coinbase proof check that v2 enforces — and that v1 entirely lacks — is: [4](#0-3) 

The analog to the reported wXTZ issue is direct: just as the token contract skips approval checks when `from` is the contract itself (a special-case bypass), this contract skips the coinbase proof check when the caller invokes v1 directly (a deprecated-path bypass). In both cases, a missing validation in a reachable code path allows an unprivileged actor to obtain a result they should not be entitled to.

---

### Impact Explanation

Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion` (v1) to gate an action — releasing bridged funds, minting tokens, unlocking collateral — can be deceived into accepting a forged Bitcoin transaction inclusion proof. The attacker does not need any privileged role; they only need to know a confirmed block hash stored in the contract's main chain and the ability to construct a 64-byte value that hashes to an existing internal Merkle node.

The corrupted invariant: `verify_transaction_inclusion` returns `true` for a `tx_id` that does not correspond to any real Bitcoin transaction.

---

### Likelihood Explanation

The 64-byte Merkle forgery technique is well-documented (referenced in the contract's own comments via the BitMEX blog post). The attack requires no special access — only knowledge of a block's Merkle tree structure, which is public Bitcoin data. The entry point is callable by any NEAR account. The only precondition is that the contract is not paused.

---

### Recommendation

**Short term:** Remove `verify_transaction_inclusion` (v1) as a public NEAR entry point. If backward compatibility is required, gate it with an access-control role so only trusted callers can invoke it, or add the coinbase proof check directly to v1.

**Long term:** Avoid retaining deprecated public entry points that omit security-critical validation. The existence of two callable verification paths with different security properties creates a permanent bypass surface. A single, hardened entry point is easier to reason about and audit.

---

### Proof of Concept

1. Identify any confirmed Bitcoin block `B` whose hash `H` is recorded in the contract's `mainchain_header_to_height` map (publicly queryable via `get_block_hash_by_height`).
2. Obtain block `B`'s full transaction list from a Bitcoin node. Compute the Merkle tree. Select any internal node `N` at depth `d` whose 64-byte preimage (left child || right child) is the attacker's chosen fake "transaction."
3. Construct `ProofArgs`:
   - `tx_id` = hash of the crafted 64-byte value (i.e., `N` itself, treated as a leaf)
   - `tx_block_blockhash` = `H`
   - `tx_index` = the position of `N` in the tree at depth `d`, mapped to a leaf index
   - `merkle_proof` = the sibling path from `N` up to the root (valid, derived from real block data)
   - `confirmations` = any value ≤ `gc_threshold`
4. Call `verify_transaction_inclusion` directly (not v2) with the above args.
5. The function computes `compute_root_from_merkle_proof(N, tx_index, merkle_proof)`, which equals the real `merkle_root` of block `B`, and returns `true` — confirming a transaction that never existed on Bitcoin. [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
