### Title
Linked Signer Can Replace Itself via `LinkSigner` Transaction, Permanently Hijacking Subaccount Signing Authority — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the currently-registered linked signer (a session key / trading bot key) can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. Once replaced, the attacker's address becomes the new linked signer and can authorize withdrawals, liquidations, transfers, NLP mints/burns, and order matching on behalf of the victim subaccount.

---

### Finding Description

The fast-path handler for `LinkSigner` in `EndpointTx.processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes the current linked signer as an accepted signer when `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 172–184
function validateSignature(
    bytes32 sender,
    bytes32 digest,
    bytes memory signature,
    bool allowLinkedSigner
) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if the recovered address equals either the subaccount owner **or** the linked signer:

```solidity
// Verifier.sol lines 291–304
function validateSignature(
    bytes32 sender,
    address linkedSigner,
    bytes32 digest,
    bytes memory signature
) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),
        ERR_INVALID_SIGNATURE
    );
}
``` [3](#0-2) 

The nonce required to construct the transaction is publicly readable via `Endpoint.getNonce(address)`. [4](#0-3) 

The `linkedSigners` mapping is the sole source of truth for delegated signing authority across all signed transaction types. [5](#0-4) 

---

### Impact Explanation

Once a linked signer replaces itself with an attacker-controlled address, the attacker's address is accepted as a valid signer for every `allowLinkedSigner = true` transaction type: `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, `TransferQuote`, `MintNlp`, `BurnNlp`, and `MatchOrders`. [6](#0-5) 

The attacker can immediately drain the victim's collateral via `WithdrawCollateral` or `TransferQuote` to an address they control. The subaccount owner retains the ability to recover via the slow-mode `LinkSigner` path (which uses `validateSender` instead of signature validation), but only after the 3-day `SLOW_MODE_TX_DELAY`. [7](#0-6) 

During that 3-day window, all collateral is at risk.

---

### Likelihood Explanation

Linked signers are explicitly designed as session keys for trading bots and automated systems — hot wallets with elevated exposure. A compromised linked signer key (e.g., via a leaked `.env` file, a malicious npm package in a trading bot, or a phishing attack on a bot operator) is a realistic and common threat. The attacker needs only the linked signer's private key and the current nonce (public on-chain) to execute the attack in a single sequencer batch.

---

### Recommendation

`LinkSigner` transactions should only be authorized by the subaccount owner (the address encoded in the first 20 bytes of the subaccount identifier), never by the current linked signer. Change the `allowLinkedSigner` flag to `false` for the `LinkSigner` handler:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only the subaccount owner may change the linked signer
);
``` [8](#0-7) 

---

### Proof of Concept

1. Alice links a session key `sessionKey` to her subaccount `aliceSubaccount` via a legitimate `LinkSigner` transaction.
2. Attacker compromises `sessionKey` (e.g., via a leaked trading bot `.env`).
3. Attacker reads `Endpoint.getNonce(address(aliceSubaccount[:20]))` to get Alice's current nonce `n`.
4. Attacker uses `sessionKey` to sign a `LinkSigner` transaction: `{sender: aliceSubaccount, signer: attackerAddress, nonce: n}`.
5. Attacker submits this to the sequencer (or waits for the sequencer to include it).
6. `processTransactionImpl` validates the signature — `sessionKey == getLinkedSigner(aliceSubaccount)` — and sets `linkedSigners[aliceSubaccount] = attackerAddress`.
7. Attacker immediately signs a `WithdrawCollateral` transaction with `attackerAddress`, draining Alice's collateral before she can respond via slow mode.
