### Title
Fork Retarget Uses Mainchain Ancestor Instead of Fork Lineage — (`contract/src/litecoin.rs`, `contract/src/lib.rs`)

---

### Summary

`get_next_work_required` in `litecoin.rs` calls `get_header_by_height(first_block_height)` to obtain the timestamp of the interval-start block for retarget calculation. `get_header_by_height` is implemented to read exclusively from `mainchain_height_to_header`. When the block being validated is a fork candidate whose divergence point is **before** `first_block_height`, the fork's own block at that height exists only in `headers_pool` (not in the mainchain index), so the contract silently uses the mainchain block's timestamp instead of the fork's true ancestor. An attacker can exploit this to submit a fork whose retarget block carries `bits` that the contract accepts but the real Litecoin network would reject.

---

### Finding Description

**Root cause — `get_header_by_height` is mainchain-only:**

`store_fork_header` inserts only into `headers_pool`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [1](#0-0) 

`store_block_header` (mainchain path) additionally writes to `mainchain_height_to_header`:

```rust
fn store_block_header(&mut self, header: &ExtendedHeader) {
    self.mainchain_height_to_header
        .insert(&header.block_height, &header.block_hash);
    ...
}
``` [2](#0-1) 

`get_header_by_height` reads **only** from the mainchain index:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [3](#0-2) 

**Root cause — retarget uses that mainchain-only lookup:**

At every 2016-block retarget boundary, `get_next_work_required` computes `first_block_height` and calls `get_header_by_height`:

```rust
let first_block_height = prev_block_header.block_height - blocks_to_go_back;
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [4](#0-3) 

The same acknowledged defect exists in `dogecoin.rs` with an explicit TODO:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [5](#0-4) 

**Call chain for fork blocks:**

`submit_block_header` → `check_target` → `check_pow` → `get_next_work_required` → `get_header_by_height` [6](#0-5) [7](#0-6) 

This path is taken for **every** submitted header, including fork candidates, before `submit_block_header_inner` decides whether to route to mainchain or fork storage. [8](#0-7) 

---

### Impact Explanation

The contract computes `expected_bits` for the fork's retarget block using the mainchain block's timestamp at `first_block_height` rather than the fork's own ancestor at that height. The attacker can:

1. Choose a fork divergence point just before `first_block_height` so the fork's block at that height has an attacker-controlled timestamp.
2. Compute what `bits` the contract will accept (using the mainchain ancestor's timestamp).
3. Mine a retarget block with those `bits` (valid scrypt PoW against the attacker-chosen target).
4. The real Litecoin network computes `bits` from the fork's true ancestor — a different value — and would reject the block.
5. The contract accepts and stores the fork. If the fork accumulates more `chain_work` than the mainchain (feasible on testnet where `pow_allow_min_difficulty_blocks = true` allows minimum-difficulty blocks), `reorg_chain` promotes it to canonical.
6. A downstream bridge relying on this light client treats the fork's transactions as finalized, enabling double-spend or fund theft.

---

### Likelihood Explanation

On Litecoin **testnet** the `pow_allow_min_difficulty_blocks` flag is `true`, meaning any block whose timestamp exceeds `prev.time + 2 × target_spacing` is accepted at minimum difficulty. An attacker can therefore build a long fork cheaply, place the divergence point one block before `first_block_height`, and mine the single retarget block with the manipulated `bits`. No privileged role, leaked key, or social engineering is required — only the ability to submit headers through the public `submit_blocks` relayer path. [9](#0-8) 

---

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` with an ancestor walk that follows `prev_block_hash` links through `headers_pool` (the same mechanism used by `get_prev_header`). This ensures the retarget calculation always uses the block that is actually `blocks_to_go_back` steps behind `prev_block_header` in the candidate chain, regardless of what the mainchain index contains at that height. The existing `get_prev_header` already does this correctly for the min-difficulty walk on lines 69–71 of `litecoin.rs`. [10](#0-9) 

---

### Proof of Concept

1. Deploy the Litecoin-testnet variant of the contract.
2. Submit a canonical chain up to height `H + 2015` (one block before a retarget boundary at `H + 2016`). Record the mainchain block at height `H` — call its timestamp `T_main`.
3. Construct a fork that diverges at height `H - 1`. The fork's block at height `H` has timestamp `T_fork ≠ T_main` (e.g., `T_fork = T_main + 3600`). Submit fork blocks `H` through `H + 2015` using min-difficulty (space timestamps > 2× target spacing).
4. Compute `expected_bits_contract` using `T_main` (what the contract will compute) and `expected_bits_real` using `T_fork` (what the real network computes). These will differ.
5. Mine a fork block at height `H + 2016` with `bits = expected_bits_contract` and valid scrypt PoW against that target.
6. Submit it. The contract's `check_pow` passes (`expected_bits == block_header.bits`). A real Litecoin node rejects it (`bits` mismatch).
7. Continue extending the fork with min-difficulty blocks until `chain_work` exceeds the mainchain. `reorg_chain` promotes the fork to canonical, and `verify_transaction_inclusion` will now confirm transactions in the fraudulent fork.

        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {

            // chainwork > highScore

            );

            self.store_block_header(&current_header);

        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission

                .headers_pool
                .get(&self.mainchain_tip_blockhash)

            if current_header.chain_work > total_main_chain_chainwork {
