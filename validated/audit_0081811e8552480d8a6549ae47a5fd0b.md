### Title
Unchecked ERC20 `transferFrom` Return Value Enables Token Theft from DDA Contracts - (File: `core/contracts/ContractOwner.sol`)

### Summary
`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value. If the USDC `transferFrom` silently returns `false` instead of reverting, the function continues to withdraw USDCe from the DDA and send it to the caller — who paid nothing. Any unprivileged caller on chain 57073 (Ink) can drain USDCe from any DDA that holds a balance.

### Finding Description
`ContractOwner.replaceUsdcEWithUsdc` is a migration helper that is callable by any external address on chain 57073. It is designed to atomically swap USDCe held in a `DirectDepositV1` (DDA) contract for USDC provided by the caller:

1. Read `balance = usdcE.balanceOf(directDepositV1)`
2. Pull `balance` USDC from `msg.sender` into the DDA via `transferFrom`
3. Withdraw `balance` USDCe from the DDA to `ContractOwner` via `DirectDepositV1.withdraw`
4. Send `balance` USDCe to `msg.sender` via `safeTransfer`

Step 2 uses a raw `transferFrom` call whose boolean return value is never checked:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // return value ignored
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
```

If `transferFrom` returns `false` without reverting (as is possible for ERC20 tokens that do not revert on failure), execution continues. Steps 3 and 4 still execute: USDCe is drained from the DDA and sent to the caller, who contributed no USDC.

The rest of the codebase consistently uses `ERC20Helper.safeTransfer` / `ERC20Helper.safeTransferFrom` (which check return values via low-level `call`), making this a localised deviation from the established safe-transfer pattern. [1](#0-0) [2](#0-1) 

### Impact Explanation
A successful exploit drains USDCe from a victim's DDA without the attacker paying any USDC. The DDA's USDCe balance represents collateral that the subaccount owner deposited or intends to deposit into the protocol. Loss of that balance is a direct, irreversible asset loss for the subaccount owner. The `ContractOwner` contract itself also temporarily holds the USDCe between steps 3 and 4, so a reentrancy-capable token could compound the damage, though the primary impact is the unchecked-return-value path. [3](#0-2) 

### Likelihood Explanation
The function carries no `onlyOwner` or similar access modifier — any address on chain 57073 can call it for any subaccount that has a non-zero DDA address. The only precondition is that the DDA holds a non-zero USDCe balance. The exploitability depends on whether the specific USDC deployment at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink chain can return `false` rather than revert. Circle's canonical USDC reverts on failure; however, the judge in the referenced report explicitly noted that ERC20 tokens may fail for reasons beyond balance (e.g., blacklisting), and the Nado codebase itself treats unchecked return values as a correctness requirement everywhere else. The open call surface and the concrete asset-loss path make this medium likelihood. [4](#0-3) 

### Recommendation
Replace the raw `transferFrom` call with the protocol's own `ERC20Helper.safeTransferFrom`, which is already imported and used throughout `ContractOwner`:

```solidity
// Before
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [2](#0-1) 

### Proof of Concept
1. A DDA for `subaccount` holds 10,000 USDCe.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` on chain 57073 with zero USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, 10000)` returns `false` (no revert) because the attacker granted no allowance.
4. The return value is not checked; execution continues.
5. `DirectDepositV1(directDepositV1).withdraw(usdcE)` transfers 10,000 USDCe to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(attacker, 10000)` sends 10,000 USDCe to the attacker.
7. Attacker receives 10,000 USDCe having paid 0 USDC; the subaccount owner's DDA is emptied. [5](#0-4)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

**File:** core/contracts/libraries/ERC20Helper.sol (L23-42)
```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
