### Title
Unchecked `transferFrom` Return Value Enables Draining `usdcE` Without Providing `usdc` — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` as a raw call without checking its return value. The contract already imports and applies `ERC20Helper` (which wraps transfers safely), but this one call bypasses it. If the USDC token at the hardcoded address returns `false` on a failed transfer rather than reverting, any unprivileged caller can receive `usdcE` tokens from the `DirectDepositV1` contract without actually providing `usdc`.

---

### Finding Description

`ContractOwner` declares `using ERC20Helper for IERC20Base` at line 24, which provides `safeTransfer` and `safeTransferFrom` wrappers that check the return value. However, inside `replaceUsdcEWithUsdc`, the inbound leg of the swap uses a raw `.transferFrom()` call:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [1](#0-0) 

The function then unconditionally proceeds to:
1. Call `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` — pulling `usdcE` out of the DDA to `msg.sender`
2. Call `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — sending `usdcE` to `msg.sender` [2](#0-1) 

There is no access control on this function — no `onlyOwner` or `onlyDeployer` modifier — making it callable by any address. [3](#0-2) 

The safe wrapper that should have been used is already available in the library: [4](#0-3) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (Ink chain, `chainid == 57073`) returns `false` on a failed `transferFrom` (e.g., due to insufficient allowance or balance) rather than reverting, the caller receives `usdcE` tokens from the `DirectDepositV1` contract without having transferred any `usdc`. The `usdcE` balance drained equals whatever `usdcE` is held in the DDA at the time of the call. This is a direct asset theft from the protocol's deposit infrastructure.

---

### Likelihood Explanation

The function is restricted to `block.chainid == 57073` (Ink), so it only applies on that deployment. Standard Circle USDC typically reverts on failure, but the token address is hardcoded to a chain-specific deployment whose exact behavior must be verified. The function has no access control, so any unprivileged caller can attempt it. The likelihood is **medium**: the trigger requires a non-reverting failure path in the specific USDC deployment, but the entry path is fully open to any caller.

---

### Recommendation

Replace the raw `.transferFrom()` call with the `safeTransferFrom` wrapper already available via `ERC20Helper`:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [5](#0-4) 

Also audit the unchecked `approve()` calls at lines 254, 530–531 in `ContractOwner.sol` and line 92 in `DirectDepositV1.sol` for the same class of issue. [6](#0-5) [7](#0-6) 

---

### Proof of Concept

1. A `DirectDepositV1` exists for some `subaccount` holding a non-zero `usdcE` balance.
2. Attacker calls `replaceUsdcEWithUsdc(subaccount)` on Ink (`chainid == 57073`) with zero USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` returns `false` (no revert) because the attacker has no allowance.
4. Execution continues: `DirectDepositV1.withdraw(usdcE)` sends `usdcE` to `msg.sender`; `safeTransfer` sends `usdcE` to `msg.sender`.
5. Attacker receives `balance` worth of `usdcE` tokens at zero cost. [8](#0-7)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
