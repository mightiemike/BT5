### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Unauthorized Depositors to Bypass the Allowlist - (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` is intended to gate who may call `addLiquidity` on a restricted pool. However, it validates the LP position `owner` address instead of the actual transaction initiator (`sender`). Because `addLiquidity` imposes no `msg.sender == owner` constraint, any unauthorized address can call `addLiquidity(authorizedOwner, ...)` and pass the allowlist check, fully bypassing the deposit restriction.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address and passes both `msg.sender` (as `sender`) and `owner` to the extension hook: [1](#0-0) 

The pool enforces `msg.sender == owner` only for `removeLiquidity`, not for `addLiquidity`: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` parameter (first `address` argument) and instead validates `owner`: [3](#0-2) 

The allowlist mappings are keyed `allowedDepositor[pool][depositor]`, where the intent is that `depositor` is the address actually calling `addLiquidity`. Because the check uses `owner` instead of `sender`, an attacker who is **not** on the allowlist can call `addLiquidity(authorizedOwner, ...)`, causing the guard to evaluate the authorized owner's allowlist entry rather than the attacker's.

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly validates `sender` (the actual swap initiator): [4](#0-3) 

---

### Impact Explanation

- **Allowlist bypass**: Any address not on the depositor allowlist can add liquidity to a pool configured as restricted, as long as it supplies an `owner` address that is on the allowlist.
- **Unauthorized pool state mutation**: The attacker injects liquidity into bins, altering `binTotals.scaledToken0/1` and per-bin balances. If the pool also runs `OracleValueStopLossExtension`, the injected liquidity shifts per-share metrics and high-watermark state, potentially suppressing or triggering the stop-loss guard for legitimate LPs.
- **Forced LP position on authorized owner**: The authorized `owner` receives LP shares they did not request, which they cannot prevent and which may expose them to impermanent loss or stop-loss events.

This breaks the core pool access-control invariant that the pool admin established by deploying the `DepositAllowlistExtension`.

---

### Likelihood Explanation

- `addLiquidity` has no `require(msg.sender == owner)` guard, so any EOA or contract can supply an arbitrary `owner`.
- The attacker only needs to know one address that is on the allowlist (e.g., a publicly visible LP or the pool admin themselves).
- No special privilege, flash loan, or oracle manipulation is required.

---

### Recommendation

Change `beforeAddLiquidity` to validate `sender` (the actual caller) rather than `owner`:

```solidity
// current (wrong):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}

// fixed:
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` and adds only `authorizedLP` to the allowlist.
2. `attacker` (not on allowlist) calls:
   ```solidity
   pool.addLiquidity(
       authorizedLP,   // owner — on the allowlist
       salt,
       deltas,
       callbackData,   // attacker's callback provides the tokens
       extensionData
   );
   ```
3. Pool calls `beforeAddLiquidity(attacker, authorizedLP, ...)`.
4. Extension evaluates `allowedDepositor[pool][authorizedLP]` → `true` → no revert.
5. Attacker's callback transfers tokens into the pool; LP shares are minted to `authorizedLP`.
6. The deposit allowlist has been fully bypassed without any privileged access.
