### Title
NFT Denom Creator Frontrunning via Permissionless `MsgIssueDenom` - (File: x/nft/keeper/keeper.go)

### Summary
The `x/nft` module's `IssueDenom` handler is fully permissionless: any account can claim any denom ID as its permanent `creator`. Because the `creator` field is set to `msg.Sender` at first-write and can never be changed, an attacker who observes a pending `MsgIssueDenom` in the mempool can submit a competing transaction with a higher fee, claim the target denom ID first, and permanently become its creator — gaining exclusive minting and editing rights over all NFTs in that collection.

### Finding Description

`MsgIssueDenom` accepts a caller-supplied `id` and records `msg.Sender` as the permanent `creator` of that denom: [1](#0-0) 

`SetDenom` writes the denom only if the ID does not yet exist, and there is no mechanism to transfer or update the creator afterward: [2](#0-1) 

The `creator` field is set directly from the caller's address: [3](#0-2) 

`IsDenomCreator` is the sole authorization gate for both `MintNFT` and `EditNFT`: [4](#0-3) [5](#0-4) 

There is no access control on `IssueDenom` — no whitelist, no governance gate, no proof-of-ownership of the ID string. The CLI entry point confirms any account can submit it: [6](#0-5) 

### Impact Explanation

The corrupted on-chain value is `Denom.Creator`. Once an attacker claims a denom ID:

- They hold the **exclusive and permanent** right to mint NFTs into that collection (`MintNFT` enforces `IsDenomCreator`).
- They hold the **exclusive and permanent** right to edit NFT metadata in that collection (`EditNFT` enforces `IsDenomCreator`).
- The legitimate project is permanently denied that denom ID; `SetDenom` returns `ErrInvalidDenom` on any subsequent attempt to register the same ID.
- The attacker can mint counterfeit NFTs under a well-known denom ID (e.g., a brand name), deceiving users who trust the denom ID as a provenance signal.

**Impact: Medium** — NFT ownership/creator bypass; permanent loss of a denom namespace for the legitimate creator; potential for counterfeit minting under a trusted denom ID.

### Likelihood Explanation

Cosmos SDK mempools order transactions by fee (gas price). An attacker who monitors the public mempool for `MsgIssueDenom` transactions can trivially submit a competing transaction with a marginally higher fee targeting the same denom ID. No privileged access, leaked keys, or social engineering is required — only a standard signed transaction from any funded account.

**Likelihood: Medium** — Requires mempool monitoring and a competing transaction, but is straightforward for any technically capable actor targeting a high-value denom ID.

### Recommendation

Bind denom ID creation to the sender's address by deriving the denom ID deterministically from the creator's address (e.g., `sha256(creator || sequence)`) or by requiring the denom ID to be prefixed with the creator's address. This eliminates the race condition because no two senders can produce the same derived ID. Alternatively, add a governance-controlled whitelist of permitted denom creators if open issuance is not intended.

### Proof of Concept

1. Legitimate project broadcasts `MsgIssueDenom{Id: "mycollection", Name: "My Collection", Sender: "cro1legitimate..."}` with fee `1000basecro`.
2. Attacker observes the pending transaction in the mempool.
3. Attacker broadcasts `MsgIssueDenom{Id: "mycollection", Name: "My Collection", Sender: "cro1attacker..."}` with fee `1001basecro`.
4. Attacker's transaction is included first; `SetDenom` writes `Creator = "cro1attacker..."`.
5. Legitimate project's transaction fails: `denomID mycollection has already exists`.
6. Attacker calls `MsgMintNFT{DenomId: "mycollection", Sender: "cro1attacker...", Recipient: <victim>}` — `IsDenomCreator` passes, counterfeit NFTs are minted under the trusted denom ID.
7. The legitimate project has no recourse; there is no `UpdateDenomCreator` or equivalent message. [2](#0-1) [4](#0-3)

```go
func (k Keeper) SetDenom(ctx sdk.Context, denom types.Denom) error {
	if k.HasDenomID(ctx, denom.Id) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomID %s has already exists", denom.Id)
	}

	if k.HasDenomNm(ctx, denom.Name) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomName %s has already exists", denom.Name)
	}

	store := ctx.KVStore(k.storeKey)
	bz := k.cdc.MustMarshal(&denom)
	store.Set(types.KeyDenomID(denom.Id), bz)
	store.Set(types.KeyDenomName(denom.Name), []byte(denom.Id))
	return nil
```
