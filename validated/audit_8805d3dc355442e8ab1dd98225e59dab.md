### Title
Host-function gas-key exec-fee uses wire-format key length instead of trie-id length for ML-DSA-65 keys, producing a ~59× fee divergence from the transaction path - (`runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The host functions `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` compute the **exec-side** gas fee by passing `public_key_len as usize` (the raw borsh wire length of the key as supplied by the contract) to `gas_key_transfer_exec_fee` / `gas_key_add_key_exec_fee`. The transaction path for the same actions correctly passes `action.public_key.trie_id_len()`. For ed25519 and secp256k1 the two values are identical, so no divergence exists today. For ML-DSA-65 they diverge by a factor of ~59 (wire = 1953 bytes, trie-id = 33 bytes), causing the exec fee to be massively overcharged when a contract invokes these host functions with a post-quantum key.

---

### Finding Description

`gas_key_transfer_exec_fee` is documented as charging for "the access key trie key length + estimated value length (what the receiver needs to read/write in the trie)." [1](#0-0) 

It receives `public_key_len` and feeds it directly into `access_key_key_len(account_id_len, public_key_len)`, which computes the trie key length. For ML-DSA-65 the trie stores a 33-byte SHA3-256 hash (tag=3 + 32-byte digest), not the 1953-byte borsh-encoded full pubkey.

The **transaction path** in `exec_fee()` correctly converts:

```rust
// runtime/runtime/src/config.rs line 347-354
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
``` [2](#0-1) 

The **host-function path** in both the wasmer and wasmtime runners does not perform this conversion:

```rust
// runtime/near-vm-runner/src/logic/logic.rs line 3091-3096
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← wire length, not trie_id_len
);
``` [3](#0-2) 

The same pattern appears in the wasmtime runner: [4](#0-3) 

And for the add-gas-key host functions, `gas_key_add_key_exec_fee` is called with `public_key_len as usize` in the host-function path: [5](#0-4) 

while the transaction path uses `trie_id_len()`: [6](#0-5) 

The divergence is rooted in the key-scheme split: `PublicKey::len()` returns 1953 for ML-DSA-65 (borsh wire form), while `PublicKey::trie_id_len()` returns 33 (hash form actually stored in the trie). [7](#0-6) 

The storage-stake and runtime-path fee code was explicitly updated to call `trie_id_len()`: [8](#0-7) 

The host-function fee path was not updated.

---

### Impact Explanation

For an ML-DSA-65 gas key, the per-byte exec-fee component of `gas_key_transfer_exec_fee` is computed over `access_key_key_len(account_id_len, 1953)` bytes instead of `access_key_key_len(account_id_len, 33)` bytes — a difference of 1920 bytes per call. For `gas_key_add_key_exec_fee` with `N` nonces the overcharge is `1920 × N` extra bytes. At the `gas_key_byte` exec rate this translates to a ~59× overcharge on the per-byte exec component for every contract-initiated gas-key operation that uses an ML-DSA-65 key. The overcharge:

1. Makes ML-DSA-65 gas-key operations via host functions prohibitively expensive relative to the same operations submitted as transactions, breaking the fee-parity invariant that the test suite explicitly verifies for ed25519 keys.
2. Can cause contract calls that budget gas correctly for the transaction path to run out of gas when the same action is dispatched via a host function, silently breaking contract logic.
3. Creates a permanent, deterministic fee-domain mismatch that is part of the on-chain execution record once `PostQuantumSignatures` is active.

---

### Likelihood Explanation

The bug is reachable by any unprivileged contract author once `ProtocolFeature::PostQuantumSignatures` is enabled. The contract supplies the borsh-encoded public key bytes and `public_key_len`; the runtime uses that length directly for the exec-fee calculation without converting to the trie-id length. No privileged role is required. The only precondition is the protocol feature gate.

---

### Recommendation

In `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` (in both `logic/logic.rs` and `wasmtime_runner/logic.rs`), decode the public key first, then use `decoded_key.trie_id_len()` for the exec-fee calculation instead of `public_key_len as usize`. The send-fee calculation may legitimately use the wire length (it charges for bytes the sender transmits), but the exec fee must use the trie-id length to reflect actual receiver-side trie I/O cost, matching the transaction path.

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }

    /// `access_key_storage_usage` uses `public_key.trie_id_len()`, which for
    /// ML-DSA-65 is the SHA3-256 hash form (1 tag + 32 bytes = 33 bytes),
    /// not the 1953-byte borsh-encoded full pubkey. The 32-byte digest is the
    /// same size as an ed25519 key, so the storage usage of an ML-DSA-65
    /// access key matches an ed25519 one exactly - rather than the
    /// ~1920-byte-larger figure a raw pubkey would produce.
    #[test]
    fn test_ml_dsa_65_access_key_storage_scales() {
        let config = RuntimeConfig::test();
        let access_key = AccessKey::full_access();

        let ed_pk: PublicKey =
            near_crypto::SecretKey::from_seed(near_crypto::KeyType::ED25519, "alice-ed")
                .public_key();
        let pq_pk: PublicKey =
            near_crypto::SecretKey::from_seed(near_crypto::KeyType::MLDSA65, "alice-pq")
                .public_key();

        let ed_usage = access_key_storage_usage(&config.fees, &ed_pk, &access_key);
        let pq_usage = access_key_storage_usage(&config.fees, &pq_pk, &access_key);

        // The trie-id encoded lengths are identical:
        // ML-DSA-65: [tag=3] + 32-byte SHA3-256 hash = 33 bytes.
        // ED25519:   [tag=0] + 32-byte raw pubkey    = 33 bytes.
        // So storage usage matches exactly. Stored raw, the ML-DSA-65 pubkey
        // would be 1953 bytes and the usage would be ~1920 bytes larger.
        assert_eq!(
            pq_usage, ed_usage,
            "ML-DSA-65 storage usage ({pq_usage}) must match ed25519 ({ed_usage}); \
             both are 33-byte trie ids"
        );
    }
