Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates the Router Address Instead of the End User, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the pool's `msg.sender` — the router — not the originating user. When a pool admin allowlists the router (a natural step to support standard UX), every on-chain user can bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`, rendering the extension's access control completely ineffective.

## Finding Description

**Root cause — actor mismatch in the allowlist check:**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist: [1](#0-0) 

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the `msg.sender` seen by the pool: [3](#0-2) 

The same pattern holds for `exactInput`, `exactOutputSingle`, and `exactOutput` — in every case the pool receives `msg.sender = router`.

**Exploit flow:**

```
allowedSwapper[pool][router]   ← what is actually checked
allowedSwapper[pool][end_user] ← what the admin intended to check
```

1. Pool admin deploys pool with `SwapAllowlistExtension` to restrict trading.
2. Pool admin allowlists Alice: `setAllowedToSwap(pool, alice, true)`.
3. Pool admin allowlists the router so Alice can use the standard UI: `setAllowedToSwap(pool, router, true)`.
4. Bob (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` — `msg.sender` to pool = router.
6. Pool calls `_beforeSwap(router, bob_recipient, ...)`.
7. Extension checks `allowedSwapper[pool][router] == true` → passes.
8. Bob's swap executes in the curated pool despite never being allowlisted.

**Why existing guards fail:**

The `allowAllSwappers` escape hatch is a separate flag. The `allowedSwapper` mapping is keyed by `(pool, sender)` where `sender` is always the immediate caller of `pool.swap()`. There is no mechanism in the current extension to distinguish between a direct swap from an allowlisted user and a router-mediated swap from an arbitrary user. [4](#0-3) 

## Impact Explanation

The `SwapAllowlistExtension` is broken for any pool that uses the standard router. Once the router is allowlisted (required for standard UX), the per-user allowlist provides zero protection — any unprivileged user can trade in a pool configured to restrict access to specific counterparties. Pools designed to restrict adverse-selection flow (e.g., limiting to known market makers) will silently accept all public order flow, directly exposing LPs to losses they configured the extension to prevent. This constitutes broken core pool functionality causing potential loss of LP assets.

## Likelihood Explanation

The trigger requires the pool admin to allowlist the router, which is a natural and expected configuration step for any operator who wants allowlisted users to access the standard router UI. No privileged attacker capability is required — any normal user of the router can exploit this. The condition is realistic and repeatable on every pool that uses both `SwapAllowlistExtension` and the standard router.

## Recommendation

Replace the `sender` check with a check on the end user's identity. Two approaches:

1. **Pass the originating user through `extensionData`**: Require callers (including the router) to encode the real user address in `extensionData`. The extension decodes and verifies it. The router already forwards `extensionData` unchanged (`params.extensionData` is passed directly to `pool.swap`), so this is compatible with the existing call path. [5](#0-4) 

2. **Router-aware fallback**: If `sender` is a known router, extract the real user from `extensionData` and check that address instead of the router address.

Either fix must ensure the allowlist gates the economically relevant actor (the end user who controls the trade), not the intermediate contract relaying the call.

## Proof of Concept

```solidity
// Setup
// 1. Deploy pool with SwapAllowlistExtension
// 2. setAllowedToSwap(pool, alice, true)
// 3. setAllowedToSwap(pool, router, true)  // natural step for standard UX

// Attack
// 4. Bob (not allowlisted) calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: bob,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    tokenIn: token0,
    extensionData: ""
}));

// 5. router calls pool.swap() → msg.sender to pool = router
// 6. pool calls _beforeSwap(router, bob, ...)
// 7. extension: allowedSwapper[pool][router] == true → passes
// 8. Bob's swap executes; Bob was never allowlisted
```

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist only Alice and the router, then assert that Bob can successfully call `router.exactInputSingle` and receive output tokens despite not being in `allowedSwapper`.
