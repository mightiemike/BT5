Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address as `sender`, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` at the pool — the immediate caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the original user. A pool admin who allowlists the router (the only way to enable router-mediated swaps for approved users) inadvertently opens the pool to every caller of the router, breaking the per-user access-control invariant.

## Finding Description

**Root cause — pool passes `msg.sender` as `sender`:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`. [1](#0-0) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, `msg.sender` at the pool is the router address. [2](#0-1) 

**`ExtensionCalling._beforeSwap` forwards `sender` unchanged:** [3](#0-2) 

**`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`:**

`msg.sender` here is the pool; `sender` is the first argument — the router address when the swap is router-mediated. The check passes if the router is allowlisted, regardless of who called the router. [4](#0-3) 

**Multi-hop paths compound the issue:**

In `exactInput`, intermediate hops use `address(this)` (the router itself) as the effective `sender` for all pools beyond the first. [5](#0-4) 

**Contrast with `DepositAllowlistExtension`**, which correctly checks `owner` (the economically relevant identity) rather than `sender` (the transport layer): [6](#0-5) 

**Exploit flow:**

1. Pool admin deploys pool with `SwapAllowlistExtension`.
2. Pool admin calls `setAllowedToSwap(pool, user1, true)` — allowlists `user1`.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — required to let `user1` use the router.
4. Attacker (`user2`, not allowlisted) calls `router.exactInputSingle({pool: pool, recipient: attacker, ...})`.
5. Router calls `pool.swap(...)` → pool sees `msg.sender = router` → `_beforeSwap(sender=router, ...)`.
6. Extension checks `allowedSwapper[pool][router] == true` → passes.
7. Attacker's swap executes in a pool they are not allowlisted for.

No existing guard prevents this: `SwapAllowlistExtension` has no concept of trusted routers or original callers, and `extensionData` forwarded by the router carries no authenticated user identity. [7](#0-6) 

## Impact Explanation

The swap allowlist is the sole on-chain access-control mechanism for restricted pools (KYC-gated, institutional-only, strategy-restricted). Allowlisting the router — the natural and expected configuration for any pool whose approved users interact via the standard periphery — sets `allowedSwapper[pool][router] = true`, which causes the extension to pass for **any** caller of the router. The access-control invariant ("only approved addresses may swap") is broken for all router-mediated paths. Unauthorized users can trade in pools intended to be restricted, undermining any compliance or strategy guarantees that depend on the allowlist. This constitutes broken core pool functionality (the allowlist hook) causing unauthorized fund flows through restricted pools.

## Likelihood Explanation

The scenario requires the pool admin to allowlist the router, which is the natural and expected configuration for any restricted pool whose approved users are expected to interact via `MetricOmmSimpleRouter`. There is no in-protocol warning that allowlisting the router is equivalent to opening the pool to all users. The bypass is reachable by any unprivileged address with no special setup beyond calling the public router entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`). It is repeatable indefinitely.

## Recommendation

Gate the allowlist on the **original user's identity**, not the immediate caller. Two concrete approaches:

1. **Extension-data forwarding with trusted-router verification**: Have the router encode the original `msg.sender` into `extensionData`. In `SwapAllowlistExtension.beforeSwap`, maintain a `trustedRouter` mapping. When `sender` is a trusted router, decode the real user from `extensionData` and check `allowedSwapper[pool][realUser]`. When `sender` is not a trusted router, check `allowedSwapper[pool][sender]` as today.

2. **Follow the `DepositAllowlistExtension` pattern**: Gate the economically relevant identity (the address that benefits from the swap / pays for it), not the transport layer. This requires the pool to expose the original initiator, or the router to authenticate it via `extensionData`.

## Proof of Concept

```
// Setup
pool admin deploys pool with SwapAllowlistExtension
pool admin calls setAllowedToSwap(pool, user1, true)   // allowlist user1
pool admin calls setAllowedToSwap(pool, router, true)  // enable router for user1

// Attack
attacker (user2, not allowlisted) calls:
  router.exactInputSingle({pool: pool, recipient: attacker, amountIn: X, ...})

// Execution trace
router.exactInputSingle()
  → pool.swap(recipient=attacker, ...)
      msg.sender at pool = router
    → _beforeSwap(sender=router, ...)
      → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          allowedSwapper[pool][router] == true  ← passes
      → swap executes for attacker

// Result: user2 swaps in a pool they are not allowlisted for
```

Foundry test plan: deploy `SwapAllowlistExtension`, configure a pool with it, allowlist `user1` and the router, then call `router.exactInputSingle` from an address that is not allowlisted and assert the swap succeeds (no `NotAllowedToSwap` revert).
