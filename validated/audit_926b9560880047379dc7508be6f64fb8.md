Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, making per-user swap allowlists bypassable or non-functional — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` as the address that called `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract, not the end user. A pool admin who allowlists the router address (a natural configuration) inadvertently grants every unprivileged user swap access. Conversely, a pool admin who allowlists individual user addresses will find those users blocked when they use the standard router, breaking the pool's intended swap functionality.

## Finding Description

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` where `msg.sender` is whoever called `pool.swap()`. [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the `sender` argument to every configured extension. [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly as `msg.sender`, so the pool receives `sender = address(router)`: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates by `owner` (the position owner, a separate argument), not by the intermediary caller — demonstrating the intended pattern exists in the codebase but was not applied to the swap path: [6](#0-5) 

## Impact Explanation

Two fund-impacting outcomes arise from the same root cause:

**Bypass (primary):** A pool admin allowlists the router address — the natural configuration for "allow swaps through the official periphery." Because the extension checks the router, not the end user, every unprivileged user can swap against a pool intended to be KYC-gated, institutional-only, or whitelist-only. LP funds are transferred at oracle prices to unauthorized counterparties with no access control enforced. This is a direct loss of LP principal through unauthorized swap execution.

**Broken allowlist (secondary):** A pool admin allowlists individual user addresses. Those users call the router. The router is not in the allowlist, so `NotAllowedToSwap` reverts for every allowlisted user going through the standard periphery. The pool's swap functionality is completely broken for its intended participants — a broken core pool functionality impact.

Both outcomes meet the Sherlock threshold for High severity: direct LP fund loss and broken core swap functionality.

## Likelihood Explanation

The bypass requires no privileged action by the attacker. Allowlisting the router is a standard, expected configuration for any pool that wants to accept swaps through the official periphery. Once the router is allowlisted, any user with a token approval can call `exactInputSingle` and the extension passes. The broken-allowlist scenario is triggered by the equally natural configuration of allowlisting individual users. Both misconfigurations are the default expected usage of the extension.

## Recommendation

Pass the originating end user through the swap path so the extension can gate the correct identity. The cleanest fix is to add a `swapInitiator` field to the `beforeSwap` hook arguments, populated from a transient storage slot set by the router before calling `pool.swap()`. A simpler short-term fix: have the router write `msg.sender` into a dedicated transient slot before each `pool.swap()` call, and have `SwapAllowlistExtension.beforeSwap` read that slot (with a fallback to `sender` when the slot is unset, for direct pool callers). The `DepositAllowlistExtension` pattern — receiving the actual actor as a distinct argument (`owner`) rather than relying on the intermediary caller — is the correct model to follow.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls: swapExtension.setAllowedToSwap(pool, address(router), true)
    // admin intent: "allow swaps through the official router"
    // actual effect: any user can swap

Attack:
  attacker (not in any per-user allowlist) calls:
    router.exactInputSingle({pool: pool, tokenIn: token0, ...})

  Execution trace:
    router.exactInputSingle()
      → pool.swap(recipient, zeroForOne, amount, ...) [msg.sender = router]
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[pool][router] == true  ✓  (passes)
        → swap executes, LP funds transferred to attacker

Result:
  Attacker swaps against a pool intended to be access-controlled.
  LP funds are transferred at oracle prices to an unauthorized counterparty.
  The per-user allowlist is completely bypassed.

Foundry test outline:
  1. Deploy pool with SwapAllowlistExtension configured.
  2. Pool admin calls setAllowedToSwap(pool, address(router), true).
  3. Attacker (EOA not in allowlist) calls router.exactInputSingle(...).
  4. Assert swap succeeds and attacker receives output tokens.
  5. Assert allowedSwapper[pool][attacker] == false (confirming bypass).
```
