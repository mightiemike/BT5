### Title
Duplicate-Leaf Phantom Index Bypass in `verify_transaction_inclusion_v2` — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

The coinbase-proof mitigation in `verify_transaction_inclusion_v2` does not prevent an attacker from claiming inclusion of a nonexistent transaction at a phantom index in any odd-length Bitcoin block. The `compute_root_from_merkle_proof` function has no bounds check on `transaction_position`, and the length-equality guard between the two proofs does not constrain the target index to a real leaf position. An unprivileged NEAR caller can obtain a `true` return for a transaction that does not exist in the block.

---

### Finding Description

**Root cause — `compute_root_from_merkle_proof` is position-agnostic:** [1](#0-0) 

The function accepts any `transaction_position` value and simply uses `position % 2` to decide left/right at each level. There is no check that the position is within the actual number of transactions in the block.

**`verify_transaction_inclusion` has no index bounds check:** [2](#0-1) 

The only guard is that the proof is non-empty. `tx_index` is passed directly to `compute_root_from_merkle_proof` without any validation against the block's actual transaction count.

**The v2 mitigation only checks proof-length equality and coinbase validity:** [3](#0-2) 

Neither guard prevents the phantom-index attack:
- `merkle_proof.len() == coinbase_merkle_proof.len()` — ensures equal depth, not a valid index.
- The coinbase `compute_root_from_merkle_proof` check — confirms the coinbase is in the block, but places no constraint on the target `tx_index`.

**The Bitcoin Merkle duplicate-leaf property:**

For an odd-length block with `N` transactions, Bitcoin's Merkle tree duplicates the last leaf: `H(tx_{N-1}, tx_{N-1})`. This means the proof for the real last transaction at index `N-1` (even position) and the phantom position `N` (odd position) produce identical intermediate hashes and therefore the same root, because:

- Real: `position=N-1` (even) → `H(tx_{N-1}, tx_{N-1})` (sibling = self)
- Phantom: `position=N` (odd) → `H(sibling=tx_{N-1}, current=tx_{N-1})` = same hash

**Concrete 3-tx block walkthrough:**

Block transactions: `[tx0 (coinbase), tx1, tx2]`

Merkle tree:
```
Root = H(H(tx0,tx1), H(tx2,tx2))
         L1[0]           L1[1]
```

Valid proof for `tx2` at index `2`: `proof = [tx2, H(tx0,tx1)]`

Attack: submit `tx_id=tx2`, `tx_index=3`, `merkle_proof=[tx2, H(tx0,tx1)]`

`compute_root_from_merkle_proof(tx2, 3, [tx2, H(tx0,tx1)])`:
- Step 1: position `3` is **odd** → `H(tx2, tx2)` = `L1[1]`; position becomes `1`
- Step 2: position `1` is **odd** → `H(H(tx0,tx1), L1[1])` = `Root`; position becomes `0`

Result equals `header.merkle_root` → function returns `true`.

Coinbase proof (valid, length 2): `compute_root_from_merkle_proof(tx0, 0, [tx1, H(tx2,tx2)]) == Root` ✓

Length check: both proofs have length 2 ✓

All three guards in `verify_transaction_inclusion_v2` pass, and the function returns `true` for a transaction at index `3` that does not exist in a 3-transaction block.

---

### Impact Explanation

Any consumer of `verify_transaction_inclusion_v2` that relies on its `true` return to authorize an action (e.g., bridging, settlement, payment release) can be deceived into accepting a nonexistent transaction as confirmed in a real Bitcoin block. This directly breaks the core invariant of the BTC light client: that a `true` result means the given `tx_id` is genuinely included at the claimed position.

---

### Likelihood Explanation

The attack requires only:
1. A real Bitcoin block with an odd number of transactions (extremely common — the majority of Bitcoin blocks have an odd tx count).
2. Knowledge of the coinbase tx hash and its valid Merkle proof (fully public on-chain data).
3. The ability to call `verify_transaction_inclusion_v2` as any unprivileged NEAR account (the function is `#[pause]`-gated but otherwise open).

No privileged role, leaked key, or social engineering is required.

---

### Recommendation

1. **Enforce index bounds in `verify_transaction_inclusion`**: The contract does not store the transaction count per block, so this requires either storing it at submission time or deriving the maximum valid index from the proof depth (`tx_index < 2^proof_depth`). The tighter check is: `tx_index < 2^(merkle_proof.len())` AND `tx_index` must be a position that is reachable without relying on the duplicate-leaf padding.

2. **Reject odd-position phantom indices in v2**: In `verify_transaction_inclusion_v2`, after validating the coinbase proof, additionally verify that `tx_index` is not a phantom duplicate by checking that `tx_index` is even when it equals the sibling of itself (i.e., detect when `tx_index % 2 == 1` and `merkle_proof[0] == tx_id`).

3. **Preferred fix**: Store the transaction count in the block header record at submission time and reject any `tx_index >= tx_count`.

### Proof of Concept

```rust
// 3-tx block: [coinbase, tx1, tx2]
// Merkle root = H(H(coinbase, tx1), H(tx2, tx2))
//
// Valid proof for tx2 at index 2: [tx2, H(coinbase, tx1)]
// Attack: submit tx_id=tx2, tx_index=3, same proof

let coinbase = /* real coinbase hash */;
let tx1      = /* real tx1 hash */;
let tx2      = /* real tx2 hash */;

let l1_left  = double_sha256(coinbase || tx1);
let l1_right = double_sha256(tx2 || tx2);
let root     = double_sha256(l1_left || l1_right);

// Coinbase proof (valid, depth 2)
let coinbase_proof = vec![tx1, l1_right];
// compute_root_from_merkle_proof(coinbase, 0, coinbase_proof) == root ✓

// Phantom proof (depth 2, same length)
let phantom_proof = vec![tx2, l1_left];
// compute_root_from_merkle_proof(tx2, 3, phantom_proof) == root ✓

verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id: tx2,
    tx_block_blockhash: block_hash,
    tx_index: 3,           // nonexistent — block only has indices 0,1,2
    merkle_proof: phantom_proof,
    coinbase_tx_id: coinbase,
    coinbase_merkle_proof: coinbase_proof,
    confirmations: 1,
})
// Returns: true  ← false inclusion accepted
```

The key arithmetic: at step 1, `position=3` (odd) causes `H(proof[0]=tx2, current=tx2)` = `H(tx2,tx2)` = `l1_right`, identical to what the real index-2 proof produces at step 1 with `position=2` (even) causing `H(current=tx2, proof[0]=tx2)`. Both paths converge to the same intermediate value and thus the same root. [4](#0-3) [5](#0-4) [3](#0-2)

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
