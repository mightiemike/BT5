### Title
Receipt Size Invariant Bypassed via `output_data_receivers` Appended After Validation in `promise_return` Path — (`runtime/runtime/src/lib.rs`)

### Summary

The nearcore runtime validates a newly-created action receipt's Borsh-serialized size against `max_receipt_size` (4,194,304 bytes) at creation time. However, when a contract uses `promise_return` (returning a `ReceiptIndex`), the runtime subsequently appends the parent receipt's `output_data_receivers` directly into the child receipt's field — **after** the size check has already passed. This post-validation mutation causes the child receipt to exceed `max_receipt_size` with no re-validation, allowing oversized receipts to enter the network and be forwarded across shards. The bug is acknowledged in the codebase itself (issue #12606) but remains unpatched in production.

### Finding Description

**Step 1 — Validation gate (passes at creation time):**

In `validate_receipt`, the size check is performed only in `ValidateReceiptMode::NewReceipt`:

```rust
// runtime/runtime/src/verifier.rs:533-541
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 =
        borsh::object_length(receipt).unwrap().try_into()...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
``` [1](#0-0) 

**Step 2 — Post-validation mutation (the root cause):**

After a function call executes and returns `ReturnData::ReceiptIndex` (i.e., `promise_return`), the runtime appends the parent receipt's `output_data_receivers` into the already-validated child receipt:

```rust
// runtime/runtime/src/lib.rs:1028-1035
ReceiptEnum::Action(new_action_receipt)
| ReceiptEnum::PromiseYield(new_action_receipt) => new_action_receipt
    .output_data_receivers
    .extend_from_slice(&action_receipt.output_data_receivers()),
ReceiptEnum::ActionV2(new_action_receipt)
| ReceiptEnum::PromiseYieldV2(new_action_receipt) => new_action_receipt
    .output_data_receivers
    .extend_from_slice(&action_receipt.output_data_receivers()),
``` [2](#0-1) 

No re-validation of the receipt's Borsh size occurs after this mutation. The receipt is then forwarded or buffered as-is.

**Step 3 — Workaround acknowledges the bug:**

The congestion control layer explicitly clamps oversized receipts to `max_receipt_size` for accounting purposes, but the actual oversized receipt bytes are still forwarded:

```rust
// runtime/runtime/src/congestion_control.rs:413-427
// There is a bug which allows to create receipts that are above the size limit.
// See https://github.com/near/nearcore/issues/12606
let max_receipt_size = apply_state.config.wasm_config.limit_config.max_receipt_size;
if size > max_receipt_size {
    size = max_receipt_size;
}
``` [3](#0-2) 

**Step 4 — `ExistingReceipt` mode is permanently weakened:**

Because oversized receipts can now exist in state, the `ExistingReceipt` validation mode must skip the size check entirely, permanently weakening the protocol's receipt validation invariant for all receipts already in flight:

```rust
// runtime/runtime/src/verifier.rs:573-585
/// 2) There is a bug which allows to create receipts that are above the size limit.
///    Runtime has to handle them gracefully until the receipt size limit bug is fixed.
///    See https://github.com/near/nearcore/issues/12606 for details.
ExistingReceipt,
``` [4](#0-3) 

**Attacker-controlled path:**

A contract author crafts a function call that:
1. Creates a new action receipt (promise C) with `args` sized to exactly `max_receipt_size - base_receipt_overhead` bytes — this passes `NewReceipt` validation.
2. Uses `promise_return(C)` inside a promise chain `[A -then-> B]`, so that when A executes, C inherits B's `output_data_receivers`.
3. The appended `DataReceiver` entries (each containing an `AccountId` + `CryptoHash`) push C's Borsh size above `max_receipt_size`.

The test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` explicitly demonstrates this path and confirms the oversized receipt enters the network: [5](#0-4) 

### Impact Explanation

- **Protocol invariant broken**: `max_receipt_size` is a consensus-level limit. Receipts exceeding it can be included in chunks and forwarded cross-shard, violating the protocol's size guarantee.
- **Congestion control accounting corrupted**: The workaround in `try_forward` silently clamps the reported size to `max_receipt_size`, meaning the actual bytes forwarded exceed the declared bandwidth grant. This undermines the bandwidth scheduler's correctness.
- **State sync and witness size**: Oversized receipts inflate storage proofs and chunk witnesses beyond their intended bounds (`per_receipt_storage_proof_size_limit`, `main_storage_proof_size_soft_limit`).
- **Permanent weakening of validation**: The `ExistingReceipt` mode must permanently skip size checks, meaning any future receipts that enter state via this path are never re-validated.

### Likelihood Explanation

Any unprivileged user who can deploy a contract can trigger this. The exploit requires:
1. Deploying a contract that calls `promise_create` + `promise_then` + `promise_return` with a large argument blob.
2. Sizing the blob to `max_receipt_size - base_receipt_overhead` (a fixed, computable constant).

No privileged role, validator access, or special network position is required. The test contract `near_test_contracts::rs_contract()` already contains the `max_receipt_size_promise_return_method1` method that demonstrates this.

### Recommendation

Re-validate the receipt's Borsh size **after** appending `output_data_receivers` in the `promise_return` path in `runtime/runtime/src/lib.rs`. If the post-mutation size exceeds `max_receipt_size`, the `promise_return` should be treated as a failure (similar to how `NewReceiptValidationError::ReceiptSizeExceeded` is returned for directly oversized receipts). This mirrors the fix pattern already applied to yield receipts, which are validated after construction.

### Proof of Concept

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is a complete, runnable proof of concept. It:
1. Deploys a contract.
2. Calls `max_receipt_size_promise_return_method1` with `args_size = max_receipt_size - base_receipt_size`.
3. Asserts via `assert_oversized_receipt_occurred` that a receipt with size above `max_receipt_size` was observed in the chain's incoming receipts — confirming the invariant is broken in production. [6](#0-5)

// A function call will generate a new receipt. Size of this receipt will be equal to
// `max_receipt_size`, it'll pass validation, but then `output_data_receivers` will be modified and
// the receipt's size will go above max_receipt_size. The receipt should be rejected, but currently
// isn't because of a bug (See https://github.com/near/nearcore/issues/12606)
// Runtime shouldn't die when it encounters a receipt with size above `max_receipt_size`.
#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
