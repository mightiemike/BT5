### Title
Wrong `MAX_FUTURE_BLOCK_TIME_MTP` constant (5400 s vs. Zcash protocol's 129600 s) causes valid Zcash headers to be permanently rejected — (`contract/src/zcash.rs`, `btc-types/src/network.rs`)

### Summary

The contract enforces a 90-minute (5400 s) MTP-based future-time window, but the Zcash reference implementation (`src/main.h` in zcash/zcash) defines `MAX_FUTURE_BLOCK_TIME_MTP = 129600` seconds (36 hours). Any canonical Zcash block whose timestamp falls in the range `(MTP + 5400, MTP + 129600]` is valid under Zcash consensus but is permanently rejected by the contract.

### Finding Description

`btc-types/src/network.rs` line 11 defines:

```rust
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60; // 5400 s
```

`contract/src/zcash.rs` `check_pow` uses this constant:

```rust
require!(
    block_header.time
        <= next_work_result.prev_block_median_time_past + MAX_FUTURE_BLOCK_TIME_MTP,
    "time-too-far-ahead-of-mtp: ..."
);
```

The contract's own comment on the line immediately above this `require!` states the Zcash protocol value is 129600 seconds (36 hours), confirming the constant is wrong by a factor of 24×. The question's premise that the Zcash reference uses 7200 s is itself incorrect — the actual Zcash value is 129600 s — but the direction of the bug is real and the discrepancy is even larger than claimed.

### Impact Explanation

- Any valid Zcash block with `time ∈ (MTP + 5400, MTP + 129600]` is accepted by every Zcash full node but rejected by the contract with `"time-too-far-ahead-of-mtp"`.
- If such a block is part of the canonical Zcash chain, the relayer's `submit_blocks` call will always revert at that height.
- The contract's tip is permanently stuck; no further blocks can be submitted because every subsequent block requires the rejected block as its predecessor.
- This breaks the core invariant: the contract must accept exactly the set of headers a Zcash full node accepts.

### Likelihood Explanation

Zcash's 75-second block target and 36-hour MTP window mean timestamps can legitimately drift well beyond 90 minutes of MTP during periods of low hash rate or network disruption. This is not a theoretical edge case — it is an explicitly permitted protocol range. The relayer has no way to work around it; the constant is hardcoded in the contract.

### Recommendation

Change the constant in `btc-types/src/network.rs` to match the Zcash reference:

```rust
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 129600; // 36 hours, matches Zcash MAX_FUTURE_BLOCK_TIME_MTP
```

The `MAX_FUTURE_BLOCK_TIME_LOCAL` constant (currently `2 * 60 * 60 = 7200 s`) is the local-clock check and is separate; it should remain as-is or be verified independently against the Zcash reference `nMaxFutureBlockTime`.

### Proof of Concept

A contract unit test demonstrating the bug:

```rust

// Contract panics with "time-too-far-ahead-of-mtp"

```
