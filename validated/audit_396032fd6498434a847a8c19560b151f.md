Audit Report

## Title
L2 Price Providers Lack Sequencer Uptime Feed Check, Allowing Stale Oracle Prices to Reach Pool Swaps — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are explicitly named and designed for L2 deployment but contain no Chainlink Sequencer Uptime Feed check. Their sole oracle-health guard is a pure timestamp comparison in `_isStale`. When the Arbitrum sequencer is down, oracle `refTime` freezes while `block.timestamp` advances via the L1 forced-inclusion path, allowing a stale bid/ask price to pass the staleness check and reach pool swaps, causing direct LP principal loss.

## Finding Description

Both contracts implement `_isStale` as a pure timestamp subtraction with no sequencer liveness signal:

`PriceProviderL2.sol` lines 135–150:
```solidity
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;
}
``` [1](#0-0) 

The constructor permits `MAX_TIME_DELTA` up to 7 days: [2](#0-1) 

`_getBidAndAskPrice` passes the result of `_isStale` as the only gate before delivering bid/ask to the pool: [3](#0-2) 

The identical pattern exists in `ProtectedPriceProviderL2`: [4](#0-3) [5](#0-4) 

**Exploit path:**
1. Sequencer goes offline at time `T₀`; last oracle push sets `refTime = T₀`, price = `P₀`.
2. Market price moves to `P₁` (e.g., +20%) during outage; no oracle update reaches the chain.
3. At `T₀ + 24h`, attacker submits a swap via the Arbitrum L1 delayed inbox (permissionless forced inclusion).
4. Transaction executes with `block.timestamp ≈ T₀ + 24h`.
5. `_isStale` evaluates: `(T₀ + 24h) − T₀ = 24h`. If `MAX_TIME_DELTA ≥ 24h` (e.g., 2 days), check passes → **not stale**.
6. `_getBidAndAskPrice` / `_computeBidAsk` return bid/ask derived from stale `P₀`.
7. Attacker buys the asset at `P₀` (20% below current market `P₁`), profiting directly at LP expense.

No sequencer uptime feed, circuit-breaker, or equivalent liveness signal exists anywhere in either L2 provider.

## Impact Explanation

The exact wrong value is the bid/ask pair returned from `_getBidAndAskPrice` / `_computeBidAsk`: it is derived from a stale oracle price `P₀` that no longer reflects market reality. This stale bid/ask is delivered directly to the pool swap, enabling the attacker to trade against LP liquidity at an arbitrarily outdated price. This constitutes **bad-price execution** and **direct loss of LP principal**, both of which are explicitly listed allowed impacts. Severity is High given the magnitude of potential LP loss (proportional to price movement during sequencer downtime, which can be substantial).

## Likelihood Explanation

- Arbitrum sequencer outages have occurred multiple times in 2023–2024, establishing real-world precedent.
- `MAX_TIME_DELTA` up to 7 days is permitted by the constructor; any value ≥ 24h makes the attack viable.
- The Arbitrum forced-inclusion delay is ~24h and the path is fully permissionless — any address can submit after the delay.
- No sequencer uptime feed or circuit-breaker exists anywhere in the L2 provider path.

Likelihood: **Medium** (requires sequencer downtime + `MAX_TIME_DELTA ≥ 24h`, both realistic conditions).

## Recommendation

Add a Chainlink L2 Sequencer Uptime Feed check at the top of `_getBidAndAskPrice` / `_computeBidAsk` in both contracts. If the sequencer is reported down or the uptime feed answer timestamp is itself stale, return `(0, type(uint128).max)` to halt swaps:

```solidity
(, int256 answer, uint256 startedAt, ,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0) return (0, type(uint128).max);
if (block.timestamp - startedAt < GRACE_PERIOD) return (0, type(uint128).max);
```

Additionally, cap `MAX_TIME_DELTA` to a value well below the forced-inclusion delay (e.g., ≤ 1 hour) to reduce the exploitable window even without the uptime feed.

## Proof of Concept

**Setup:** Deploy `PriceProviderL2` on Arbitrum fork with `MAX_TIME_DELTA = 2 days`. Configure a pool using this provider.

**Steps:**
1. At time `T₀`, simulate sequencer downtime by stopping oracle updates. Last oracle push: `refTime = T₀`, `mid = P₀`.
2. Warp `block.timestamp` to `T₀ + 24h` (simulating forced-inclusion execution).
3. Call `getBidAndAskPrice()` on the provider.
4. Assert: `_isStale(T₀, T₀ + 24h, 2 days, ...)` returns `false` (24h < 2 days).
5. Assert: returned bid/ask is derived from stale `P₀`, not current market price `P₁`.
6. Execute a swap at the stale price and verify LP balance decreases by the arbitrage amount.
