### Title
Unnecessary Non-Empty Merkle Proof Guard Blocks Valid SPV Verification for Single-Transaction Blocks - (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` contains an unnecessary `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard that rejects mathematically valid SPV proofs for blocks containing only one transaction. This is a direct analog to H-2: just as `require(left > 0)` blocked a valid tax refund when `left == 0` was a legitimate state, this guard blocks a valid proof when an empty proof is the correct proof.

### Finding Description

In `verify_transaction_inclusion`, after confirming the block is on the mainchain and has enough confirmations, the contract checks:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

It then calls `compute_root_from_merkle_proof` to verify the proof:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

The underlying `compute_root_from_merkle_proof` implementation iterates over the proof slice and, if it is empty, simply returns `transaction_hash` unchanged:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {   // loop body never executes when empty
        ...
    }
    current_hash   // returns transaction_hash as-is
}
``` [3](#0-2) 

For a Bitcoin block containing exactly one transaction (the coinbase), the Merkle tree has a single leaf. The Merkle root **is** the coinbase transaction hash, and the correct Merkle proof is the empty list `[]`. Calling `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` correctly returns `coinbase_tx_id`, which equals `merkle_root`. The guard fires before this computation is reached, unconditionally rejecting the valid proof.

`verify_transaction_inclusion_v2` is also affected because it delegates to `verify_transaction_inclusion` via `self.verify_transaction_inclusion(args.into())`: [4](#0-3) 

For a single-transaction block, both `merkle_proof` and `coinbase_merkle_proof` are empty. The length-equality check in `verify_transaction_inclusion_v2` passes (both are length 0), the coinbase proof check passes (empty proof correctly reconstructs the root), but the inner call to `verify_transaction_inclusion` then panics on the empty-proof guard.

### Impact Explanation

Any downstream NEAR contract or user that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to prove a transaction in a single-transaction Bitcoin block receives a hard panic (`"Merkle proof is empty"`) instead of `true`. The proof is mathematically valid and the block is on the mainchain with sufficient confirmations — the rejection is caused solely by the unnecessary guard. This breaks the SPV guarantee for a real class of Bitcoin blocks (empty blocks mined by pools are common on mainnet) and can cause downstream bridges or applications to permanently fail to process legitimate transactions.

### Likelihood Explanation

Empty blocks (blocks with only the coinbase transaction) are a routine occurrence on Bitcoin mainnet — mining pools frequently produce them when a new block arrives before the mempool can be assembled. Any relayer that submits such a block to the light client will store it successfully, but any subsequent SPV proof call for the coinbase transaction of that block will be unconditionally rejected. The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion`.

### Recommendation

Remove the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard entirely. The subsequent `compute_root_from_merkle_proof` call already handles the empty-proof case correctly and will return `false` (via the equality check against `merkle_root`) for any invalid empty proof, so the guard provides no additional security while blocking valid single-transaction proofs.

### Proof of Concept

1. A Bitcoin block is mined with only the coinbase transaction. Its Merkle root equals the coinbase tx hash `C`.
2. The relayer submits this block header; it is stored in `headers_pool` with `merkle_root = C`.
3. A caller invokes `verify_transaction_inclusion` with `tx_id = C`, `tx_index = 0`, `merkle_proof = []`, and sufficient confirmations.
4. The contract reaches line 315: `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` — **panics**.
5. Had the guard been absent, `compute_root_from_merkle_proof(C, 0, &[])` returns `C`, which equals `header.block_header.merkle_root`, and the function correctly returns `true`. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
