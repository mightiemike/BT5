Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks caller-supplied `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` who provides tokens) and gates only on `owner`, a free caller-supplied argument to `MetricOmmPool.addLiquidity`. Any address not on the allowlist can bypass the guard by passing any allowlisted address as `owner`, defeating the pool admin's curation policy entirely.

## Finding Description

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` with no validation and forwards `msg.sender` as `sender` and the raw `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both `sender` and `owner` and passes them to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` names the first parameter (the actual depositor `sender`) as `address` with no identifier — it is discarded — and gates only on `owner`: [3](#0-2) 

The sibling `SwapAllowlistExtension.beforeSwap` demonstrates the correct pattern, checking `sender` (the actual swapper) and discarding the second address argument: [4](#0-3) 

Because `owner` is a free parameter with no on-chain constraint tying it to `msg.sender`, an attacker calls `pool.addLiquidity(allowedLP, salt, deltas, callbackData, extensionData)`. The extension evaluates `allowedDepositor[pool][allowedLP]` → `true` and does not revert. `LiquidityLib.addLiquidity` then pulls tokens from the attacker via callback and credits the position to `allowedLP`. The attacker cannot recover the tokens because `removeLiquidity` enforces `msg.sender == owner`: [5](#0-4) 

## Impact Explanation

This is an admin-boundary break: the pool admin's sole mechanism for restricting who may deposit (the `DepositAllowlistExtension`) fails open for every `addLiquidity` call. Any unprivileged address can deposit into a curated pool. The attacker's tokens are permanently transferred to `allowedLP`'s position (direct loss of the attacker's principal), and the allowlist invariant — the only on-chain enforcement of pool curation — is completely defeated.

## Likelihood Explanation

No special role or privilege is required. The bypass requires only knowing one allowlisted address, which is publicly readable from `allowedDepositor`. Any EOA or contract can execute this with a single transaction. The condition is always satisfiable whenever the pool has at least one allowlisted depositor, and is repeatable indefinitely.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual token provider) and discard `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address /*owner*/, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

1. Deploy pool with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(pool, allowedLP, true)`.
3. `attacker` (not on the allowlist) calls `pool.addLiquidity(allowedLP, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(attacker /*sender*/, allowedLP /*owner*/, ...)`.
5. Extension evaluates `allowedDepositor[pool][allowedLP]` → `true` → does not revert.
6. `LiquidityLib.addLiquidity` executes with `owner = allowedLP`, pulling tokens from `attacker` via callback.
7. Position is credited to `allowedLP`; `attacker` has deposited into a pool that was supposed to reject them.
8. `attacker` cannot recover tokens because `removeLiquidity` requires `msg.sender == owner` (i.e., `attacker != allowedLP`).
9. The deposit allowlist is bypassed with zero privileges; attacker's principal is lost.
