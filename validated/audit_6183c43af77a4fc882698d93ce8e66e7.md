### Title
`build_streamer_message()` silently drops execution outcomes for stale shard IDs at resharding boundaries, permanently omitting them from `StreamerMessage` — (`chain/indexer/src/streamer/mod.rs`)

### Summary

In `build_streamer_message()`, execution outcomes are fetched for **all** shards (including stale/old shard IDs carried by old chunk headers at resharding boundaries), but the per-chunk processing loop only consumes outcomes for **new chunks the node tracks**. Outcomes for stale shard IDs are left in `shards_outcomes` after the loop and are silently dropped — never placed into any `IndexerShard.receipt_execution_outcomes`. The code acknowledges this with a warning log and a `TODO`, but provides no fix. The result is that every `StreamerMessage` produced at a resharding boundary is permanently missing a subset of execution outcomes, with no way for downstream consumers to detect or recover the gap.

### Finding Description

`build_streamer_message()` in `chain/indexer/src/streamer/mod.rs` follows this sequence:

**Step 1 — fetch all outcomes (all shards, including stale):**
```rust

```
This calls `GetExecutionOutcomesForBlock`, handled by `ViewClientActor`, which calls `ChainStore::get_block_execution_outcomes()`:


`iter_raw()` returns **all** chunk headers — both new chunks and old (repeated) chunk headers. At a resharding boundary, the block carries old chunk headers for the parent shard IDs that are no longer in the new layout. Any execution outcomes stored under those stale shard IDs (e.g., delayed receipts processed in the last block of the old epoch) are included in `shards_outcomes`.

**Step 2 — fetch only new, tracked chunks:**

Stale shard IDs from old chunk headers are filtered out here.

**Step 3 — per-chunk loop only removes outcomes for tracked new chunks:**

```rust
    .remove(&header.shard_id)
    .expect("execution outcomes for given shard should be present");
```

Only the shard IDs of new, tracked chunks are removed from `shards_outcomes`.

**Step 4 — leftover outcomes are silently dropped:**
```rust

    let leftover_outcomes: usize = shards_outcomes.values().map(Vec::len).sum();
```

The outcomes are logged as a warning and then dropped. They are **never** placed into any `IndexerShard.receipt_execution_outcomes`. The `StreamerMessage` is returned with a structural gap. The code's own comment names the two triggering conditions:

> (a) the indexer's `ShardTracker` excludes a shard the node itself tracked  
> (b) the post-resharding edge case where a stale shard id is no longer part of the new layout

Both are acknowledged as requiring a "proper fix" with a `TODO`, mirroring the Cork Protocol's `FIXME: this is only temporary` comment that was never resolved.

### Impact Explanation

Every indexer built on NEAR Indexer Framework (`near-indexer`, NEAR Lake, etc.) that processes blocks at a resharding boundary will receive a `StreamerMessage` with missing `receipt_execution_outcomes`. The missing outcomes correspond to real on-chain events — receipts executed in the last block(s) of the old epoch under the parent shard IDs. Downstream consumers (DeFi protocols, NFT marketplaces, balance trackers) that rely on the completeness of `StreamerMessage` will silently compute incorrect state: missed token transfers, missed contract events, incorrect account balances. Because the gap is permanent (the block is finalized and the `StreamerMessage` for it will always be missing those outcomes), there is no self-healing path. The only recovery is a full re-index with a patched indexer.

### Likelihood Explanation

This is triggered deterministically at every resharding event — a public protocol upgrade boundary. NEAR has already undergone multiple resharding events (Simple Nightshade V1–V6, per the deprecated feature list in `core/primitives-core/src/version.rs`), and dynamic resharding is actively being developed. Any user whose transaction or receipt is processed in the parent shard during the last block of the old epoch contributes an outcome that will be dropped. No special privilege is required; ordinary user transactions suffice.

### Recommendation

In `build_streamer_message()`, after the per-chunk loop, iterate over any remaining entries in `shards_outcomes` and route them into the appropriate `IndexerShard`. For stale shard IDs (case b), map the old shard ID to its child shard(s) using the new shard layout's split map and append the outcomes there. For untracked shards (case a), align the indexer's `ShardTracker` with the node's actual tracked shards so no outcomes are fetched that cannot be attributed to a chunk. Remove the `TODO` only after both cases are handled.

### Proof of Concept

At a resharding boundary block `B` (last block of old epoch):

1. `get_block_execution_outcomes(B)` iterates `block.chunks().iter_raw()`, which includes old chunk headers for parent shard IDs `[P0, P1, ...]`. Any delayed receipts executed under `P0` in block `B` produce outcomes stored in `DBCol::OutcomeIds` keyed by `(B, P0)`. These are returned in `shards_outcomes[P0]`.

2. `fetch_block_new_chunks(B)` skips old chunk headers (`is_new_chunk` = false for `P0`). `shards_outcomes[P0]` is never removed by the per-chunk loop.

3. After the loop, `shards_outcomes` is non-empty. The warning fires:
   ```
   "execution outcomes left after streaming all chunks; they are not included in the streamer message"
   leftover_shards=[P0], leftover_outcomes=N
   ```

4. `StreamerMessage` is returned. `shards[*].receipt_execution_outcomes` contains no entry for `P0`'s outcomes. Downstream consumers see `N` fewer outcomes than actually occurred on-chain for block `B`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```rust
    pub(crate) async fn fetch_block_new_chunks(
        &self,
        block: &BlockView,
        shard_tracker: &ShardTracker,
    ) -> Result<Vec<ChunkView>, FailedToFetchData> {
        tracing::debug!(target: INDEXER, height = block.header.height,  "fetch chunks for block");
        let mut futures = futures::stream::FuturesUnordered::new();
        for chunk in &block.chunks {
            if !chunk.is_new_chunk(block.header.height) {
                continue;
            }
            let cares_about_shard = shard_tracker
                .cares_about_shard_checked(&block.header.prev_hash, chunk.shard_id)
                .map_err(|err| {
                    FailedToFetchData::String(format!(
                        "failed to determine shard tracking for shard {} at block {}: {err}",
                        chunk.shard_id, block.header.hash,
                    ))
                })?;
            if !cares_about_shard {
                continue;
            }
            futures.push(self.fetch_single_chunk(chunk.chunk_hash));
        }
        let mut chunks = Vec::<ChunkView>::with_capacity(futures.len());
        while let Some(chunk) = futures.next().await {
            chunks.push(chunk?);
        }
        Ok(chunks)
    }
```
