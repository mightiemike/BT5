### Title
Compile-Time `ADDRESS_REGISTRAR_ACCOUNT_ID` Hardcoded to Mainnet Account Breaks ERC-20 Emulation on Testnet/Localnet — (`runtime/near-wallet-contract/implementation/wallet-contract/src/ADDRESS_REGISTRAR_ACCOUNT_ID`)

---

### Summary

The ETH wallet contract bakes `address-map.near` (a mainnet account) into every network variant of its WASM at compile time via `include_str!`. The `build.rs` correctly varies `CHAIN_ID` per network (397/398/399) but never varies `ADDRESS_REGISTRAR_ACCOUNT_ID`. As a result, the testnet and localnet WASM binaries both call `address-map.near` for address-registrar lookups, which is the wrong account on those networks, silently breaking ERC-20 transfer emulation and EOA base-token-transfer address checks for all ETH-implicit accounts on non-mainnet deployments.

---

### Finding Description

The wallet contract reads its address-registrar account ID from a source file at compile time:

```rust
const ADDRESS_REGISTRAR_ACCOUNT_ID: &str = std::include_str!("ADDRESS_REGISTRAR_ACCOUNT_ID");
``` [1](#0-0) 

That file contains exactly one value:

```
address-map.near
``` [2](#0-1) 

The `build.rs` produces three separate WASM artifacts — one per network — by mutating `CHAIN_ID` before each Docker build and restoring it afterward:

```rust
std::fs::write(&chain_id_path, chain_id.to_string().into_bytes())?;
docker_build(...)?;
std::fs::write(&chain_id_path, chain_id_content)?;
``` [3](#0-2) 

`ADDRESS_REGISTRAR_ACCOUNT_ID` is **never touched** by `build.rs`. All three artifacts — `wallet_contract_mainnet.wasm`, `wallet_contract_testnet.wasm`, `wallet_contract_localnet.wasm` — embed `address-map.near` identically. [4](#0-3) 

At runtime, when an ETH-implicit account performs an ERC-20 transfer or an EOA base-token transfer that requires an address check, the wallet contract dispatches a cross-contract call to the hardcoded registrar:

```rust
let account_id = ADDRESS_REGISTRAR_ACCOUNT_ID
    .trim()
    .parse()
    .unwrap_or_else(|_| env::panic_str("Invalid address registrar"));
ext_registrar::ext(account_id).with_static_gas(REGISTRAR_LOOKUP_GAS)
``` [5](#0-4) 

On testnet and localnet, `address-map.near` either does not exist or is an unrelated account. The `lookup` call therefore returns `None` or fails, causing the address-check callback to treat every address as unregistered, which silently misdirects or drops ERC-20 transfers.

The runtime dispatches the correct per-network WASM via `wallet_contract_magic_bytes`:

```rust
match chain_id {
    chains::MAINNET => MAINNET.magic_bytes(),
    chains::TESTNET => TESTNET.magic_bytes(),
    _ => LOCALNET.magic_bytes(),
}
``` [6](#0-5) 

But the per-network WASM distinction only covers `CHAIN_ID` (EIP-155 replay protection); the registrar account ID is identical across all three.

The test harness itself confirms the correct pattern: it **does** update `ADDRESS_REGISTRAR_ACCOUNT_ID` before building the contract under test, then restores it — proving the file must differ per deployment:

```rust
tokio::fs::write(address_registrar_account_id_path(BASE_DIR), contract.id().as_bytes()).await?;
// ...
tokio::fs::write(registrar_id_path, &original_registrar_id).await?;
``` [7](#0-6) 

The production `build.rs` never performs this substitution for testnet or localnet builds.

---

### Impact Explanation

Any ETH-implicit account on testnet or localnet that attempts an ERC-20 transfer (which requires a registrar lookup to resolve the token contract's named account) will have the cross-contract call routed to `address-map.near` — the wrong account. The lookup returns `None`, the address-check callback treats the target as unregistered, and the transfer is either silently misdirected or the relayer is incorrectly banned. ERC-20 emulation is effectively non-functional for ETH-implicit accounts on testnet.

**Severity: Medium** — Core ETH-implicit account functionality (ERC-20 transfers) is broken on testnet; mainnet is unaffected.

---

### Likelihood Explanation

**Likelihood: Medium** — The testnet wallet contract is a live, protocol-level global contract. Any testnet user or developer exercising ERC-20 transfer emulation via an ETH-implicit account will trigger this path. The bug is structural (compile-time constant) and cannot be patched without redeploying the global contract.

---

### Recommendation

In `build.rs`, apply the same per-network substitution pattern to `ADDRESS_REGISTRAR_ACCOUNT_ID` that is already applied to `CHAIN_ID`:

```rust
const MAINNET_REGISTRAR: &str = "address-map.near";
const TESTNET_REGISTRAR: &str = "address-map.testnet"; // or the actual testnet account
const LOCALNET_REGISTRAR: &str = "address-map.test.near"; // or sandbox account

fn build_contract(dir, contract_name, output, chain_id, registrar_id) {
    // write CHAIN_ID ...
    std::fs::write(&registrar_id_path, registrar_id.as_bytes())?;
    docker_build(...)?;
    // restore both files
}
```

Alternatively, make `ADDRESS_REGISTRAR_ACCOUNT_ID` a runtime parameter passed to the contract's `init` method, eliminating the compile-time coupling entirely.

### Proof of Concept

1. `build.rs` defines three network builds, varying only `CHAIN_ID`: [8](#0-7) 

2. `ADDRESS_REGISTRAR_ACCOUNT_ID` is never modified by `build.rs`; the source file always contains `address-map.near`: [2](#0-1) 

3. The constant is baked into the WASM at compile time: [1](#0-0) 

4. At runtime on testnet, the wallet contract calls `address-map.near.lookup(address)` for every ERC-20 transfer address check — a mainnet-only account — returning `None` or failing, breaking the transfer: [9](#0-8) 

5. The test harness proves the correct fix: it writes the actual deployed registrar account ID into the file before building, confirming the file must be network-specific: [10](#0-9)
