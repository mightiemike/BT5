Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against the per-pool allowlist, where `sender` is `msg.sender` at the pool level. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original user. This creates an irreconcilable configuration dilemma: allowlisting the router grants access to all users, while not allowlisting it blocks all router-based swaps including those from legitimately allowlisted users.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension via `abi.encodeCall`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the `msg.sender` from the pool's perspective: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

The extension never sees the original `msg.sender` of the router call — it only ever sees the router's address. No existing guard in `SwapAllowlistExtension` or `ExtensionCalling` compensates for this: the `sender` field is passed through without any original-initiator tracking, and the extension has no mechanism to decode or verify the true caller from `extensionData`.

## Impact Explanation
A curated pool (e.g., KYC-gated, institutional-only, or regulatory-restricted) that deploys `SwapAllowlistExtension` to restrict who may trade can be bypassed by any unprivileged user simply by calling `MetricOmmSimpleRouter.exactInputSingle` or any multi-hop variant. If the pool admin allowlists the router (necessary for allowlisted users to swap via the router), every user — including non-allowlisted ones — can call the router and the extension will approve the swap because `allowedSwapper[pool][router] == true`. The non-allowlisted user executes a real swap, receives output tokens, and the pool's LP assets are consumed in an unauthorized trade. This is a direct loss of the pool's curation guarantee and constitutes unauthorized access to LP-owned liquidity, matching the "broken core pool functionality causing loss of funds" impact criterion.

## Likelihood Explanation
The router is a public, permissionless contract. Any user can call it at any time with no special privileges, flash loans, or multi-step setup. The bypass is reachable on every swap through the router on any pool that has `SwapAllowlistExtension` in its `BEFORE_SWAP_ORDER` and has allowlisted the router. Likelihood is **High**.

## Recommendation
The extension must gate on the original user, not the intermediary caller. The preferred fix is to have the router encode the original `msg.sender` into `extensionData` and have the extension decode and check it when `sender` is a known trusted router. Alternatively, the pool could pass the original initiator as a separate field. A partial mitigation is to check both `sender` and `recipient` and require at least one to be allowlisted, though this does not cover all cases. The cleanest production fix requires the router to forward the real user address in `extensionData` and the extension to decode it when `sender` matches a registered trusted-router address.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension in BEFORE_SWAP_ORDER.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — necessary so that allowlisted users can swap via the router.
3. Non-allowlisted attacker calls:
     router.exactInputSingle(ExactInputSingleParams{
       pool: pool,
       tokenIn: token0,
       zeroForOne: true,
       amountIn: 1_000,
       ...
     })
4. Pool calls pool.swap(...) with msg.sender = router.
5. _beforeSwap passes sender = router to SwapAllowlistExtension.
6. Extension checks allowedSwapper[pool][router] == true → passes.
7. Attacker receives token1 output. Swap settles. Allowlist is bypassed.

Alternatively, if the admin does NOT allowlist the router:
3b. Allowlisted user calls router.exactInputSingle(...)
4b. Extension checks allowedSwapper[pool][router] == false → reverts NotAllowedToSwap.
    Allowlisted user cannot use the router at all.
```
