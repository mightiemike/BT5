### Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router**, not the end user. The extension therefore checks whether the **router** is allowlisted, not the actual trader. If the pool admin allowlists the router (a natural action to "enable the standard periphery"), every user — including those the allowlist was meant to exclude — can bypass the per-user gate.

---

### Finding Description

**Hook plumbing — wrong-actor binding on the swap allowlist path.**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, ...))
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` against the per-pool allowlist:

```solidity
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant), the router calls `pool.swap(...)` directly:

```solidity
// MetricOmmSimpleRouter.sol
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ..., params.extensionData
);
```

At that point `msg.sender` inside the pool is the **router contract**, so `sender` delivered to the extension is the router address. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Two concrete failure modes:**

| Admin intent | What admin does | Actual result |
|---|---|---|
| Allow only KYC'd users | Allowlists Alice, Bob | Alice/Bob are blocked when using the router (router not allowlisted); they must call the pool directly |
| Enable the standard periphery | Allowlists the router address | Every user — including non-KYC'd — can swap through the router; per-user allowlist is fully bypassed |

The second mode is the direct bypass: once the router is allowlisted, the extension's `allowedSwapper` mapping is dead code for all router-mediated swaps.

---

### Impact Explanation

A curated pool (e.g., restricted to institutional counterparties, KYC'd addresses, or protocol-owned accounts) relies on `SwapAllowlistExtension` as its sole swap gate. If the pool admin allowlists the router to enable the standard periphery UX, any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps against the pool. The pool's token reserves are exposed to unauthorized traders, directly impacting LP principal and fee revenue. This is a broken core pool functionality / admin-boundary break with direct loss of LP assets.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap entry point. A pool admin who deploys a curated pool and wants users to interact through the standard periphery will naturally allowlist the router. The bypass requires no special privilege, no flash loan, and no unusual token behavior — any EOA can call `exactInputSingle` on the router pointing at the restricted pool.

---

### Recommendation

The extension must check the **economic actor** (the end user), not the intermediary. Two options:

1. **Pass the original user through the router.** Add a `swapper` field to the swap call or use a separate authenticated channel (e.g., transient storage, similar to how the router already stores `_getPayer()`). The pool or extension then checks that identity.

2. **Check `recipient` instead of `sender` for the allowlist.** If the pool's design guarantees that the recipient is always the end user, the extension can gate on `recipient`. However, this changes the semantics and may not hold for multi-hop paths.

The cleanest fix is option 1: the router stores the original `msg.sender` in transient storage before calling the pool, and the extension reads it via a trusted accessor — mirroring the existing `_getPayer()` / `_getTokenToPay()` pattern already used for callback settlement.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured as beforeSwap hook
  - Pool admin calls setAllowedToSwap(pool, alice, true)       // Alice is KYC'd
  - Pool admin calls setAllowedToSwap(pool, router, true)      // Enable standard periphery

Attack (Bob, not allowlisted):
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({
         pool: restrictedPool,
         recipient: bob,
         zeroForOne: true,
         amountIn: X,
         ...
     })
  2. Router calls pool.swap(bob, true, X, ...) — msg.sender = router
  3. Pool calls _beforeSwap(router, bob, ...)
  4. Extension checks allowedSwapper[pool][router] → true (router is allowlisted)
  5. Swap executes; Bob receives tokens from the restricted pool

Result:
  - Bob bypassed the per-user allowlist entirely
  - allowedSwapper[pool][bob] was never consulted
  - LP funds in the restricted pool are exposed to unauthorized traders
