### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable On-Chain, Bypassing the 64-Byte Merkle Forgery Protection Added in v2 — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is marked `#[deprecated]` in Rust but is still a fully public NEAR contract method. The `#[deprecated]` attribute only emits compiler warnings for Rust callers; it imposes no on-chain access restriction. Any unprivileged NEAR account can call v1 directly, bypassing the coinbase Merkle proof validation that `verify_transaction_inclusion_v2` was specifically introduced to enforce. This enables the documented 64-byte transaction Merkle proof forgery attack against any consumer of the v1 endpoint.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction Merkle proof forgery vulnerability (referenced in the code's own doc comment and at https://www.bitmex.com/blog/64-Byte-Transactions). The fix works by requiring the caller to also supply a coinbase Merkle proof, which is validated before delegating to v1: [1](#0-0) 

The coinbase proof check is the only guard against the forgery. However, v1 itself remains a public, unguarded NEAR method: [2](#0-1) 

The `#[deprecated]` Rust attribute is a **compiler-level hint only**. It does not translate into any on-chain access restriction. The method is `pub`, carries no `#[private]` attribute, and is not paused by default (only `#[pause]` is present, which requires an active PauseManager action to engage). Any NEAR account — including an unprivileged attacker — can invoke it directly via a cross-contract call or a direct transaction.

The function's own documentation acknowledges the risk:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

The Merkle proof verifier in `merkle-tools` performs no structural check to distinguish a leaf (real transaction hash) from an internal node: [4](#0-3) 

This means an attacker can craft a `tx_id` that is actually a 64-byte internal Merkle node hash, supply a valid-looking `merkle_proof` path, and receive `true` from v1 for a transaction that does not exist in the block.

---

### Impact Explanation

Any recipient contract or off-chain application that calls `verify_transaction_inclusion` (v1) directly — or that an attacker routes through v1 by constructing a cross-contract call — will receive a forged `true` result for a non-existent Bitcoin transaction. This corrupts the proof-verification result that the light client is designed to provide, enabling fraudulent cross-chain operations such as claiming BTC deposits that never occurred, unlocking bridged assets, or triggering settlement logic based on fabricated on-chain evidence.

The corrupted value is the boolean return of `verify_transaction_inclusion`, which downstream contracts treat as authoritative confirmation of Bitcoin transaction inclusion.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. Any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte forgery technique is publicly documented (linked in the contract's own source). The only prerequisite is knowledge of a valid Bitcoin block's Merkle root, which is public information stored in the contract's `headers_pool`. Likelihood is **high** for any attacker aware of the forgery technique.

---

### Recommendation

Remove the public accessibility of v1 entirely. The simplest fix is to add `#[private]` to `verify_transaction_inclusion`, which in NEAR SDK restricts callers to `env::current_account_id()` (i.e., only the contract itself, as used internally by v2):

```rust
#[private]
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
```

Alternatively, remove v1 entirely and inline its logic into v2, eliminating the bypass surface. If backward compatibility must be preserved for existing callers, the PauseManager should immediately pause v1 as a stopgap.

---

### Proof of Concept

1. Identify any Bitcoin block whose hash is stored in `mainchain_header_to_height` (public state).
2. Obtain the block's Merkle tree. Find any internal node `N` at depth `d` whose 32-byte hash, when used as `tx_id` with a `merkle_proof` of length `d`, computes to the block's `merkle_root` via `compute_root_from_merkle_proof`.
3. Call `verify_transaction_inclusion` directly on the NEAR contract:
   ```json
   {
     "tx_id": "<internal_node_hash>",
     "tx_block_blockhash": "<target_block_hash>",
     "tx_index": <crafted_index>,
     "merkle_proof": ["<sibling_hashes...>"],
     "confirmations": 1
   }
   ```
4. The function returns `true` for a transaction that does not exist.
5. `verify_transaction_inclusion_v2` would have rejected this call at the coinbase proof check step (`require!(...coinbase merkle proof... == header.block_header.merkle_root)`), but v1 has no such guard. [5](#0-4)

```rust
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
