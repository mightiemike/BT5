### Title
Oracle Price Used for Perpetual Position Health Valuation Enables Excess Collateral Withdrawal During Basis Deviation — (File: `core/contracts/PerpEngine.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`PerpEngine.calculatePositionPnl` values perpetual positions using the stored oracle price (`_risk(productId).priceX18`), not the perpetual mark price. This oracle price feeds directly into `getHealth`, which gates `withdrawCollateral`. During backwardation (mark price < oracle price), long positions appear more profitable than they are on-chain, allowing an unprivileged user to withdraw excess collateral. The shortfall is later absorbed by the insurance fund.

---

### Finding Description

In `PerpEngine.sol`, `calculatePositionPnl` computes unrealized PnL as:

```solidity
int128 priceX18 = _risk(productId).priceX18;
positionPnl = priceX18.mul(balance.amount) + balance.vQuoteBalance;
``` [1](#0-0) 

`_risk(productId).priceX18` is the oracle price stored in `RiskHelper.RiskStore`, updated by sequencer-submitted `UpdatePrice` transactions. It reflects the external spot price of the underlying asset, not the perpetual contract's own mark price. [2](#0-1) 

This same oracle price is used by `getHealthContribution` for perp positions (via the `RiskStore.priceX18` field), which is aggregated in `Clearinghouse.getHealth`: [3](#0-2) 

`withdrawCollateral` enforces a health gate using this oracle-priced health value: [4](#0-3) 

Trades are matched at the perpetual mark price (order book price). When a user opens a long position at mark price `P_mark`, their `vQuoteBalance` is set to `-P_mark * amount`. Health is then computed as `P_oracle * amount + vQuoteBalance`. When `P_oracle > P_mark` (backwardation), the health calculation overstates the position's value by `(P_oracle - P_mark) * amount`, allowing the user to withdraw collateral that does not correspond to any real economic value.

The protocol does track the basis (`avgPriceDiffs`) via `PerpTick` for funding rate purposes, confirming that mark price and oracle price are treated as distinct values — but only oracle price is used for health: [5](#0-4) 

---

### Impact Explanation

A user with a large long perp position can withdraw collateral equal to `(P_oracle - P_mark) * positionSize * (1 - initialWeight)` beyond what their position's true mark-price value supports. After withdrawal, when the oracle price corrects downward (or the basis persists), the position becomes undercollateralized. The protocol's insurance fund absorbs the deficit. This is a direct, quantifiable asset loss from the protocol, proportional to the basis magnitude and position size.

---

### Likelihood Explanation

Backwardation is a routine market condition in perpetual markets, particularly during high selling pressure or funding rate inversions. A basis of 2–5% is common; during volatile periods it can exceed 10%. The attack is permissionless: any existing depositor with a long perp position can execute it by timing a `WithdrawCollateral` or `WithdrawCollateralV2` transaction to a period of elevated basis. No privileged access, sequencer compromise, or governance action is required.

---

### Recommendation

Use the perpetual mark price — derived from the on-chain order book state or a TWAP of recent trade prices — for health and PnL calculations. The oracle price should be reserved for funding rate computation and liquidation price determination, which are its correct roles in a perpetuals system. At minimum, apply a conservative haircut to the oracle price when computing initial health for long positions to bound the exploitable basis window.

---

### Proof of Concept

1. User opens a long position of 10 BTC-PERP at mark price $48,000. `vQuoteBalance = -480,000`. User deposits $48,000 collateral (1× leverage).
2. BTC-PERP enters backwardation: mark price remains $48,000, oracle price updates to $50,000 via `UpdatePrice`.
3. On-chain health = `$50,000 × 10 + (−$480,000)` = **$20,000** (using oracle price).
4. True health at mark price = `$48,000 × 10 + (−$480,000)` = **$0**.
5. User submits a `WithdrawCollateralV2` transaction to withdraw $20,000 of quote collateral.
6. `getHealth` returns $20,000 ≥ 0 → check passes. [4](#0-3) 
7. User receives $20,000 of quote that has no backing in the position's true mark-price value.
8. Oracle price corrects to $48,000. User's position is now undercollateralized by $20,000.
9. Liquidation or socialization forces the insurance fund to cover the $20,000 deficit.
10. Remaining depositors and the insurance fund bear the loss; the attacker has extracted $20,000 risk-free.
