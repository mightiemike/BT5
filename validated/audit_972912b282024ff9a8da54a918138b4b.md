### Title
Unchecked `approve` Return Value in `DirectDepositV1.creditDeposit` Silently Fails for Non-Standard ERC20 Tokens - (File: `core/contracts/DirectDepositV1.sol`)

### Summary
`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without checking the boolean return value. For non-standard ERC20 tokens that return `false` on a failed approval instead of reverting, the approval silently fails. The function then proceeds to call `endpoint.depositCollateralWithReferral`, which will attempt a `transferFrom` against an allowance that was never actually set, causing the deposit to fail. Because `creditDeposit` has no access control, any caller can trigger this path.

### Finding Description
`DirectDepositV1.creditDeposit()` iterates over all spot product IDs, checks the DDA's token balance, and for each non-zero balance calls:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(
    subaccount,
    productId,
    uint128(balance),
    "-1"
);
```

The `approve` call uses the raw `IIERC20Base.approve` interface and discards the `bool` return value. The `IIERC20Base` interface explicitly declares `approve` as returning `bool`:

```solidity
function approve(address spender, uint256 amount) external returns (bool);
```

Yet the return value is never inspected. The rest of the codebase — `ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` — uses low-level `call` with explicit return-value decoding to handle exactly this class of non-standard tokens. `creditDeposit` does not use this safe wrapper for the approval step.

The `Endpoint` (via `EndpointTx`) uses `using ERC20Helper for IERC20Base` and calls `handleDepositTransfer`, which internally uses `safeTransferFrom`. If the approval was silently skipped, `safeTransferFrom` will revert because the allowance was never set, causing the entire `creditDeposit` call to revert. For a token that also silently fails on `transferFrom` (returning `false` without reverting), the endpoint's `safeTransferFrom` wrapper would still catch it and revert — but if a future integration path bypasses the safe wrapper, the deposit could be credited without actual token movement.

### Impact Explanation
The immediate, confirmed impact is a **denial-of-service on the DDA deposit path** for any ERC20 token that returns `false` on `approve` without reverting. Because `creditDeposit` iterates over all product IDs in a single loop, a single non-standard token in the product list causes the entire call to revert, blocking deposits for all tokens in that invocation. Users whose subaccount relies on the DDA mechanism cannot credit their deposits. Funds remain in the DDA contract but cannot be forwarded to the protocol.

The secondary, latent impact is **solvency corruption**: if the endpoint's internal transfer path is ever changed to use a raw (unchecked) `transferFrom`, a non-standard token that also returns `false` on `transferFrom` would allow `clearinghouse.depositCollateral` → `spotEngine.updateBalance` to credit the subaccount without any actual token movement, inflating protocol balances.

### Likelihood Explanation
Non-standard ERC20 tokens (e.g., BNB, OMG, and others that return `false` instead of reverting) are well-documented and actively deployed. The Nado protocol is designed to support multiple spot collateral tokens. Any such token added as a product would trigger this path. `creditDeposit` is callable by any address with no access control, so no privileged position is required to reach the vulnerable code.

### Recommendation
Replace the raw `approve` call in `creditDeposit` with a safe wrapper analogous to `ERC20Helper.safeTransfer`. Either:

1. Add a `safeApprove` function to `ERC20Helper` (or `DirectDepositV1`'s internal helpers) that uses a low-level `call` and checks the return value, mirroring the existing `safeTransfer` pattern already in the codebase.
2. Or use OpenZeppelin's `SafeERC20.safeApprove` / `forceApprove`.

The fix should also reset the allowance to zero before setting it (to handle tokens like USDT that revert on non-zero-to-non-zero approvals), consistent with the pattern already used in `ContractOwner.wrapVaultAsset` (lines 530–531).

**Root cause — unchecked `approve`:** [1](#0-0) 

**`IIERC20Base.approve` declares a `bool` return that is discarded:** [2](#0-1) 

**Contrast: the rest of the codebase uses safe wrappers that check return values:** [3](#0-2) 

**`EndpointTx` uses `ERC20Helper` for all token operations, meaning a failed approval causes a revert at the `safeTransferFrom` step:** [4](#0-3) 

**`creditDeposit` has no access control — any caller can trigger the path:** [5](#0-4) 

**Attack path:**
1. A non-standard ERC20 token (returning `false` on `approve`) is listed as a spot product.
2. Tokens accumulate in the DDA contract (e.g., via direct transfer or native wrapping).
3. Any caller invokes `creditDeposit()`.
4. `token.approve(address(endpoint), balance)` returns `false`; the return value is ignored.
5. `endpoint.depositCollateralWithReferral(...)` is called; the endpoint's `safeTransferFrom` reverts because the allowance was never set.
6. The entire `creditDeposit` call reverts, blocking all deposits for all tokens in that loop iteration.

```text
    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }
```

```text
        IERC20Base self,
        address to,
```
