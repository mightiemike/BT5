Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the originating user, enabling full allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension` is intended to gate swaps by individual user address. However, its `beforeSwap` hook checks `sender`, which is the pool's `msg.sender` — the `MetricOmmSimpleRouter` contract address for all router-mediated swaps, not the originating user. A pool admin who allowlists the router (the natural action to enable router-based swaps for their allowlisted users) inadvertently grants every user access to the restricted pool, completely defeating the per-user allowlist.

## Finding Description

In `SwapAllowlistExtension.beforeSwap`, the check is:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension caller) and `sender` is the first argument forwarded by the pool. [1](#0-0) 

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the first argument to `_beforeSwap`: [2](#0-1) 

In `ExtensionCalling._beforeSwap`, this `sender` is forwarded directly to the extension: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, the pool's `msg.sender` is the **router contract**, not the originating user. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`: [4](#0-3) 

This is structurally asymmetric with `DepositAllowlistExtension.beforeAddLiquidity`, which correctly checks `owner` (the position owner, not the adder contract), because `addLiquidity` passes both `msg.sender` (adder) and `owner` (actual user) as separate arguments: [5](#0-4) [6](#0-5) 

The `beforeSwap` interface has no equivalent separate "actual user" parameter — only `sender` (the pool's `msg.sender`) and `recipient`: [7](#0-6) 

## Impact Explanation

Two concrete fund-impacting consequences follow:

**1. Complete allowlist bypass:** A pool admin who allowlists the router address — the natural action to let their allowlisted users swap via the standard periphery — inadvertently grants every user on-chain access to the restricted pool. Any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle` and the extension will pass because `allowedSwapper[pool][router] == true`. Unauthorized swaps execute at oracle-derived prices, draining LP assets.

**2. Broken core swap flow for allowlisted users:** If the admin allowlists only specific user addresses (not the router), those users cannot swap through the router at all — the extension sees `sender = router` and reverts with `NotAllowedToSwap`. The standard periphery path is permanently broken for the intended users, forcing direct pool calls.

Both outcomes constitute broken core pool functionality and direct loss of LP assets, meeting Sherlock High/Medium thresholds.

## Likelihood Explanation

The bypass requires the pool admin to allowlist the router. This is a natural and expected configuration step: a pool admin who deploys a restricted pool and wants their allowlisted users to use the standard periphery will allowlist the router. The extension provides no mechanism to gate individual users through the router, so the admin has no correct alternative. The trigger is a predictable admin action, not a malicious one. Once the router is allowlisted, the unprivileged bypass (any user routing through the allowlisted router) requires no special access and is repeatable by any address.

## Recommendation

The `beforeSwap` hook should gate the actual initiating user, not the intermediary. Two options:

1. **Pass the real user via `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.

2. **Mirror the deposit pattern:** Add a `swapper` parameter to the swap hook (analogous to `owner` in the deposit hook) that the pool populates with the economically relevant actor. This is a protocol-level change but closes the gap cleanly.

Until fixed, pool admins must not allowlist the router address in `SwapAllowlistExtension`; allowlisted users must call `pool.swap()` directly.

## Proof of Concept

```
Setup:
  pool deployed with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)       // alice is the intended user
  admin calls setAllowedToSwap(pool, router, true)      // to let alice use the router

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

  router calls:
    pool.swap(bob, zeroForOne, amount, priceLimit, "", extensionData)
    // pool's msg.sender = router

  pool calls:
    extension.beforeSwap(sender=router, ...)
    // checks allowedSwapper[pool][router] == true  ← passes

  Result:
    bob's swap executes at oracle price, draining LP assets
    per-user allowlist is completely bypassed
```

Foundry test plan: deploy pool with `SwapAllowlistExtension`, configure alice and router as allowlisted, call `exactInputSingle` from bob's address, assert the swap succeeds (bypass confirmed). Then remove router from allowlist, assert alice's router call reverts (broken flow confirmed).
