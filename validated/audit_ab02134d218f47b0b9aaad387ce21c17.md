### Title
`MsgTierUndelegate` Blocked by Insufficient Bonus Pool — Position Owner Cannot Recover Principal — (`x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`MsgTierUndelegate` unconditionally calls `claimRewards` before executing the staking unbond. `claimRewards` propagates `ErrInsufficientBonusPool` as a hard failure. When the bonus pool is empty and a position has accrued non-zero bonus, the entire `TierUndelegate` transaction reverts, leaving the position owner unable to undelegate their principal regardless of whether they have satisfied every exit precondition.

---

### Finding Description

In `TierUndelegate` the reward-settlement call precedes the actual unbond: [1](#0-0) 

`claimRewards` delegates to `processEventsAndClaimBonus`: [2](#0-1) 

Inside `processEventsAndClaimBonus`, after computing `totalBonus`, the pool balance is checked: [3](#0-2) 

`sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` when the module account balance is less than the owed bonus: [4](#0-3) 

That error propagates back through `claimRewards` → `TierUndelegate` as a hard `return nil, err`, so the staking `Undelegate` call on line 182 is never reached: [5](#0-4) 

The position owner's principal — held at the position's dedicated delegator address — remains locked in the staking module with no exit path until the pool is externally replenished.

The ADR-006 documentation explicitly acknowledges this behaviour as a design choice: [6](#0-5) 

However, listing `Undelegate` alongside `ClaimTierRewards` in the "fail atomically, retry later" category conflates two fundamentally different operations: claiming a bonus (discretionary) and recovering principal (non-discretionary). The Audius H11 report identified exactly this pattern — coupling an exit path to a balance check that can fail — as the root cause.

---

### Impact Explanation

A position owner who has:
1. Triggered exit (`MsgTriggerExitFromTier`), and
2. Waited for the full `ExitDuration` to elapse,

satisfies every stated precondition for undelegation. Nevertheless, if the bonus pool is empty and any bonus has accrued, `MsgTierUndelegate` reverts. The owner's principal is locked at the position's delegator address with no alternative exit path (`MsgExitTierWithDelegation` also calls `claimRewards` and has the same failure mode). The only recovery is waiting for governance to replenish the pool — an indefinite, externally-controlled timeline.

The exact corrupted invariant: the position's delegator address holds bonded tokens that the owner cannot retrieve despite having completed the full exit lifecycle.

---

### Likelihood Explanation

The bonus pool is a finite module account funded by governance. It can reach zero through:

1. **Natural depletion** — high aggregate bonus accrual across all positions outpaces governance top-ups.
2. **Attacker-induced depletion** — any unprivileged account that holds one or more large positions can call `MsgClaimTierRewards` to drain the pool, then immediately prevent all other position owners from undelegating. The attacker's own positions are unaffected (they already claimed). No special role is required; a standard `MsgClaimTierRewards` transaction is the entry point. [7](#0-6) 

The attack is low-cost relative to impact: the attacker only needs enough locked tokens to accrue a bonus that exceeds the remaining pool balance.

---

### Recommendation

Decouple principal recovery from bonus payment in `TierUndelegate`. When `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the handler should either:

- **Forfeit the accrued bonus** (advance `LastBonusAccrual` and `LastEventSeq` without paying) and proceed with the unbond, or
- **Skip bonus settlement entirely** in the undelegate path and require the owner to claim rewards separately before or after undelegating.

The `BeforeRedelegationSlashed` hook already demonstrates the correct pattern — it silently forfeits bonus when the pool is insufficient rather than halting the operation: [8](#0-7) 

The same "forfeit rather than block" logic should be applied to `TierUndelegate` (and `MsgExitTierWithDelegation`) so that a depleted pool can never prevent a position owner from recovering their principal.

---

### Proof of Concept

1. Alice locks `N` tokens in a tier with `TriggerExitImmediately = true`. Bonus accrues over time.
2. Bob (attacker) holds a large position and calls `MsgClaimTierRewards`, draining the bonus pool to zero.
3. Alice's `ExitUnlockAt` elapses. She submits `MsgTierUndelegate`.
4. Execution path: `TierUndelegate` → `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → `ErrInsufficientBonusPool` → transaction reverts.
5. Alice's principal remains bonded at her position's delegator address. She cannot undelegate, cannot `ExitTierWithDelegation` (same failure), and cannot `WithdrawFromTier` (position is still delegated). Her tokens are locked until governance replenishes the pool.

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewards(ctx context.Context, pos types.PositionState) (types.PositionState, sdk.Coins, sdk.Coins, error) {
	if !pos.IsDelegated() {
		return pos, sdk.NewCoins(), sdk.NewCoins(), nil
	}

	base, err := k.claimBaseRewards(ctx, pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	return pos, base, bonus, nil
}

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
