### Title
Compromised Linked Signer Can Re-Link to Attacker-Controlled Address Before User Revocation Takes Effect — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The sequencer-path `LinkSigner` transaction is validated with `allowLinkedSigner = true`, meaning the **current linked signer** can authorize a new `LinkSigner` transaction to replace itself with any address. Because the only user-controlled on-chain revocation path (slow mode) carries a mandatory 3-day delay, a compromised linked signer can re-link to an attacker-controlled address before the revocation takes effect, giving the attacker a persistent signing authority over the victim's subaccount.

---

### Finding Description

In `EndpointTx.processTransactionImpl()`, the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves to `validateSignature(sender, digest, sig, true)`, which passes `getLinkedSigner(sender)` to the verifier as an accepted signer:

```solidity
// EndpointTx.sol lines 172–184
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

This means the **current linked signer** can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address, including an attacker-controlled one.

The only user-controlled on-chain revocation path is the slow-mode `LinkSigner`, which enforces a hard-coded 3-day delay:

```solidity
// EndpointTx.sol lines 376–380
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
    ...
});
``` [3](#0-2) 

The slow-mode `LinkSigner` path correctly restricts authorization to the subaccount owner via `validateSender`, so the linked signer cannot use slow mode:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    validateSender(txn.sender, sender);   // ← only subaccount owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
``` [4](#0-3) 

The asymmetry is the root cause: the sequencer path lets the linked signer authorize its own replacement; the only immediate user-controlled revocation path has a 3-day delay.

---

### Impact Explanation

Once the attacker re-links to address `B`, `B` can sign:

- `WithdrawCollateral` / `WithdrawCollateralV2` (with `sendTo == address(0)`, which permits linked signer authorization per line 447) to drain collateral.
- `TransferQuote` to move quote balances to an attacker-controlled subaccount.
- `MintNlp` / `BurnNlp` to manipulate NLP positions.

All of these accept linked signer signatures with `allowLinkedSigner = true`: [5](#0-4) [6](#0-5) 

The result is direct, complete theft of the victim's on-chain collateral. The 3-day slow-mode window gives the attacker ample time to execute all draining transactions before the revocation is enforced.

**Impact: 5 / 10** — Full collateral loss for any user whose linked signer key is compromised.

---

### Likelihood Explanation

Linked signers are the standard mechanism for automated trading bots and API integrations. Hot wallet key compromise (phishing, server breach, leaked `.env`) is a realistic and common event. The attacker needs only the compromised linked signer's private key and the ability to submit a transaction to the sequencer — no privileged protocol access is required.

**Likelihood: 4 / 10** — Realistic for any user operating a linked signer on an internet-connected machine.

---

### Recommendation

Change the `allowLinkedSigner` flag for the sequencer-path `LinkSigner` handler from `true` to `false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only subaccount owner may change the linked signer
);
```

This mirrors the slow-mode path's restriction and ensures that only the subaccount owner (the Ethereum address embedded in the `bytes32` subaccount) can authorize a linked signer change, regardless of path.

---

### Proof of Concept

1. Alice has subaccount `0xALICE_SUBACCOUNT` with linked signer `S` (an API hot wallet).
2. Attacker compromises `S`'s private key.
3. Alice discovers the breach and submits a slow-mode `LinkSigner` with `signer = address(0)`. This is queued with `executableAt = now + 3 days`.
4. Attacker, holding `S`, signs a sequencer-path `LinkSigner` transaction: `{sender: 0xALICE_SUBACCOUNT, signer: 0xATTACKER_B, nonce: N}`.
5. Sequencer processes the attacker's transaction immediately (no delay). `linkedSigners[0xALICE_SUBACCOUNT] = 0xATTACKER_B`.
6. Attacker uses `0xATTACKER_B` to sign `WithdrawCollateralV2` transactions (with `sendTo = address(0)`, so `allowLinkedSigner = true`) and drains Alice's collateral.
7. After 3 days, Alice's slow-mode revocation executes and sets `linkedSigners[0xALICE_SUBACCOUNT] = address(0)` — but the funds are already gone.

The broken invariant: a user's intent to revoke a linked signer can be defeated by the linked signer itself, because the sequencer-path `LinkSigner` accepts the linked signer's own signature as authorization to overwrite the linked signer mapping.
