### Title
Overly-Restrictive Pre-Check in `verify_transaction_inclusion` Rejects Valid Single-Transaction Block Proofs — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` contains a pre-check (`require!(!args.merkle_proof.is_empty())`) that is stricter than the underlying Merkle computation requires. For a Bitcoin block containing exactly one transaction (the coinbase), the correct Merkle proof is legitimately empty, and the underlying `compute_root_from_merkle_proof` handles that case correctly. The pre-check causes the call to panic before the computation is ever reached, rejecting a provably valid inclusion proof. `verify_transaction_inclusion_v2` is also affected because it delegates to `verify_transaction_inclusion` after its own checks pass.

---

### Finding Description

In `verify_transaction_inclusion`, after confirming the block exists and has enough confirmations, the contract executes:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

The underlying `compute_root_from_merkle_proof` iterates over the proof slice. When the slice is empty, the loop body never executes and the function returns `transaction_hash` unchanged:

```rust
let mut current_hash = transaction_hash;
for proof_hash in merkle_proof {   // never entered when proof is empty
    ...
}
current_hash                       // returns transaction_hash as-is
``` [2](#0-1) 

In Bitcoin, when a block contains exactly one transaction (the coinbase), the Merkle root is defined as the coinbase transaction hash itself, and the correct inclusion proof is the empty vector. Calling `compute_root_from_merkle_proof(tx_id, 0, &[])` returns `tx_id`. If `tx_id == header.block_header.merkle_root`, the verification is correct and should return `true`. The `require!` at line 315 prevents this path from ever being reached.

`verify_transaction_inclusion_v2` is equally affected. For a single-transaction block:

1. The length equality pre-check passes: `0 == 0`.
2. The coinbase proof check passes: `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id`, which equals `merkle_root` for a single-tx block.
3. The function then delegates to `verify_transaction_inclusion`, which panics at line 315. [3](#0-2) 

---

### Impact Explanation

Any cross-chain application or downstream NEAR contract that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to confirm a Bitcoin transaction residing in a single-transaction block will receive a panic instead of a valid `true` result. The contract's core SPV verification guarantee is broken for this class of blocks. If the consuming contract gates fund release or state transitions on a `true` return, those operations become permanently inaccessible for affected transactions, even though the on-chain Bitcoin data is fully valid and the proof is correct.

---

### Likelihood Explanation

Single-transaction blocks (coinbase only) occur on Bitcoin mainnet during periods of very low mempool activity and are routine on testnets and regtest environments. The condition is externally determined by the state of the Bitcoin network, not by any attacker action. Any unprivileged NEAR account can trigger the panic by submitting a valid `ProofArgs` or `ProofArgsV2` for such a block with an empty `merkle_proof`. No special role, key, or social engineering is required.

---

### Recommendation

Remove the overly-restrictive pre-check at line 315. The underlying `compute_root_from_merkle_proof` already handles the empty-proof case correctly by returning the transaction hash unchanged. The correctness check is already performed by the equality comparison on line 318–322:

```rust
// Remove this line:
// require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0)

### Proof of Concept

**Setup:** A Bitcoin block is submitted to the contract containing exactly one transaction (the coinbase). Its Merkle root equals the coinbase transaction hash `C`.

**Call:**
```
verify_transaction_inclusion_v2({
    tx_id:                  C,
    tx_block_blockhash:     <valid block hash in mainchain>,
    tx_index:               0,
    merkle_proof:           [],          // correct empty proof
    coinbase_tx_id:         C,
    coinbase_merkle_proof:  [],          // correct empty proof
    confirmations:          1,
})
```

**Execution trace:**
1. `merkle_proof.len() == coinbase_merkle_proof.len()` → `0 == 0` ✓ passes.
2. `compute_root_from_merkle_proof(C, 0, &[])` → returns `C` == `merkle_root` ✓ passes.
3. Delegates to `verify_transaction_inclusion`.
4. `require!(!args.merkle_proof.is_empty())` → `!true` → **panics** with `"Merkle proof is empty"`.

**Expected result:** `true` (the coinbase is trivially included in its own block).
**Actual result:** Transaction aborted. [4](#0-3) [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
