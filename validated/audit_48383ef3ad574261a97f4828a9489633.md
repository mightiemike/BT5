### Title
`ValidResourceBounds::AllResources` silently downgraded to `ValidResourceBounds::L1Gas` in protobuf deserialization when `l2_gas` and `l1_data_gas` are zero, producing a divergent transaction hash across the P2P sync boundary — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` classifies a transaction as `L1Gas` whenever the deserialized `l2_gas` and `l1_data_gas` values are both zero, regardless of whether the original transaction was submitted and hashed as `AllResources`. Because `get_tip_resource_bounds_hash` includes the `L1_DATA_GAS` resource element in the Poseidon preimage for `AllResources` but omits it for `L1Gas`, the transaction hash computed by a syncing node differs from the hash computed and stored by the proposing node. This is a canonicalization invariant break at the protobuf serialization boundary.

---

### Finding Description

**Root cause — protobuf deserialization (`transaction.rs` lines 417–436):**

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // ...
        let l1_data_gas = value.l1_data_gas.unwrap_or_default(); // silently defaults to zero
        // ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← type downgrade
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
``` [1](#0-0) 

**Submission path — always `AllResources`:**

Every transaction entering via the RPC gateway is stored as `AllResources`. `RpcInvokeTransactionV3.resource_bounds` is typed `AllResourceBounds`, and the conversion to `InvokeTransactionV3` unconditionally wraps it:

```rust
resource_bounds: ValidResourceBounds::AllResources(tx.resource_bounds),
``` [2](#0-1) 

A transaction with `AllResourceBounds { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` is accepted by the stateless validator (the test suite explicitly marks `valid_l1_gas` — only `l1_gas` non-zero — as a passing case): [3](#0-2) 

**Hash divergence — `get_tip_resource_bounds_hash`:**

```rust
pub fn get_tip_resource_bounds_hash(resource_bounds: &ValidResourceBounds, tip: &Tip)
    -> Result<Felt, StarknetApiError>
{
    // L1 and L2 always included.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    // L1_DATA_GAS only for AllResources:
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // ← omitted
        ValidResourceBounds::AllResources(all) => {
            vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?]     // ← included
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
``` [4](#0-3) 

For the same transaction data:

| Side | `ValidResourceBounds` variant | Poseidon preimage elements |
|---|---|---|
| Proposing node (submission) | `AllResources` | `tip, L1_GAS, L2_GAS(0), L1_DATA_GAS(0)` |
| Syncing node (after protobuf round-trip) | `L1Gas` | `tip, L1_GAS, L2_GAS(0)` |

The two hashes are structurally different. The `L1_DATA_GAS(0)` element is present in the `AllResources` preimage and absent in the `L1Gas` preimage.

**Serialization round-trip is not preserved:**

The serializer for `L1Gas` explicitly emits `l1_data_gas: Some(ResourceBounds::default().into())` (zero), so `L1Gas` round-trips correctly. But `AllResources` with zero `l2_gas` and `l1_data_gas` serializes identically to `L1Gas` on the wire, yet was hashed differently at submission time: [5](#0-4) 

**Contrast with JSON deserialization (correct behavior):**

The JSON/serde path uses key *presence* to distinguish the two variants: if the `L1DataGas` key is present in the map (even with a zero value), the result is `AllResources`; only if the key is *absent* does it fall back to `L1Gas`. The protobuf path uses value comparison instead of presence, breaking the invariant: [6](#0-5) 

---

### Impact Explanation

A syncing node that receives a block containing a V3 transaction with `AllResources { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` will:

1. Deserialize the transaction as `ValidResourceBounds::L1Gas`.
2. Compute a transaction hash that differs from the hash stored in the block by the proposing node.
3. Either reject the block (if hash validation is enforced), causing a sync stall, or store the transaction with the wrong hash and execute it under `L1Gas` fee/gas-accounting semantics (`GasVectorComputationMode::NoL2Gas`) instead of `AllResources` semantics (`GasVectorComputationMode::All`).

The second outcome produces wrong receipts, wrong fee charges, and wrong state — matching the "Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input" and "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload" impact categories.

---

### Likelihood Explanation

Any user can submit a V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas` non-zero). The gateway stateless validator explicitly accepts this. The transaction will be included in a block and propagated over P2P sync, triggering the deserialization mismatch on every syncing peer. No special privilege is required.

---

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, use the *presence* of the `l1_data_gas` field (i.e., `Option::is_some`) rather than its *value* to decide the variant, mirroring the JSON deserialization logic:

```rust
Ok(if value.l1_data_gas.is_none() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

This preserves the `AllResources` type for any transaction that explicitly provided an `l1_data_gas` field on the wire, even if its value is zero, matching the canonical hash preimage used at submission time.

---

### Proof of Concept

1. Submit a V3 invoke transaction with `resource_bounds = AllResourceBounds { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: { 0, 0 }, l1_data_gas: { 0, 0 } }`.
2. The gateway computes hash H1 using `get_tip_resource_bounds_hash` with `AllResources` — preimage includes `L1_DATA_GAS(0)`.
3. The transaction is included in a block and serialized to protobuf for P2P sync.
4. The receiving node calls `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`: `l1_data_gas.is_zero() && l2_gas.is_zero()` → returns `L1Gas(l1_gas)`.
5. The receiving node computes hash H2 using `get_tip_resource_bounds_hash` with `L1Gas` — preimage omits `L1_DATA_GAS`.
6. H1 ≠ H2. The receiving node has a divergent view of the transaction hash, causing either block rejection or incorrect execution under `NoL2Gas` fee semantics.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
