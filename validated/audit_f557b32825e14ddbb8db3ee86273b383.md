### Title
Profitable Perp Position Allows Full Quote Collateral Withdrawal, Leaving Protocol Exposed to Insurance Fund Bad Debt — (`core/contracts/Clearinghouse.sol`)

---

### Summary

In `Clearinghouse::withdrawCollateral`, the post-withdrawal health check uses `INITIAL` health type, which includes unrealized perp PnL (via `vQuoteBalance` in each perp balance). A user with a sufficiently profitable perp position can withdraw their entire USDC quote balance. If the position later reverses and becomes liquidatable, the insurance fund must absorb the resulting bad debt while the user retains the withdrawn USDC.

---

### Finding Description

`withdrawCollateral` in `Clearinghouse.sol` performs a health check after debiting the spot balance:

```solidity
IProductEngine.HealthType healthType = sender == X_ACCOUNT
    ? IProductEngine.HealthType.PNL
    : IProductEngine.HealthType.INITIAL;

require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [1](#0-0) 

`getHealth` aggregates contributions from both the `SpotEngine` and `PerpEngine`:

```solidity
health = spotEngine.getHealthContribution(subaccount, healthType);
...
health += perpEngine.getHealthContribution(subaccount, healthType);
``` [2](#0-1) 

For the `INITIAL` health type, the perp engine's health contribution for a long position of `amount` at current price `P'` with entry `vQuoteBalance = -amount * P_entry` is:

```
perp_health = amount * P' * longWeightInitial + vQuoteBalance
            = amount * (P' * longWeightInitial - P_entry)
``` [3](#0-2) 

`_getWeightX18` returns the actual `longWeightInitialX18` for `INITIAL` type (not `ONE`), but the `vQuoteBalance` — which encodes unrealized PnL — is still added directly to health without any haircut: [4](#0-3) 

When `P'` is significantly above `P_entry`, the perp health contribution becomes large and positive, allowing the user to withdraw their entire USDC quote balance while the health check still passes.

---

### Impact Explanation

After the user withdraws all USDC, if the perp price reverses and maintenance health drops below zero, `liquidateSubaccountImpl` is triggered. For perp liquidations, the payment is settled entirely through `vQuoteBalance` adjustments — no actual USDC is seized from the liquidatee:

```solidity
perpEngine.updateBalance(txn.productId, txn.liquidatee, -txn.amount, v.liquidationPayment);
perpEngine.updateBalance(txn.productId, txn.sender, txn.amount, -v.liquidationPayment);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -v.liquidationFees);
``` [5](#0-4) 

If the liquidatee's `vQuoteBalance` is deeply negative after position closure, `_finalizeSubaccount` uses the insurance fund to cover it:

```solidity
int128 insuranceCover = MathHelper.min(v.insurance, -quoteBalance.amount);
if (insuranceCover > 0) {
    v.insurance -= insuranceCover;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.liquidatee, insuranceCover);
}
``` [6](#0-5) 

The user has already extracted real USDC. The protocol's insurance fund absorbs the bad debt. If the insurance fund is insufficient, `socializeSubaccount` is called, spreading losses across all other participants. [7](#0-6) 

---

### Likelihood Explanation

The attack path is fully permissionless and requires no special privileges. Any trader who opens a perp position and observes sufficient unrealized profit can immediately submit a `withdrawCollateral` transaction for the full USDC balance. The sequencer processes this transaction, and the on-chain health check at line 419 passes. The attacker then waits for the position to reverse naturally (or can close it themselves after withdrawal, locking in the profit while having already extracted the collateral). [8](#0-7) 

---

### Recommendation

The `withdrawCollateral` function should not allow unrealized perp PnL to substitute for actual deposited collateral. Two approaches:

1. **Require a minimum quote balance**: After withdrawal, assert that the remaining quote balance is at least equal to the initial margin requirement for all open perp positions, independent of unrealized PnL.
2. **Exclude unrealized PnL from withdrawal health**: Use a separate health type (e.g., `PNL` type with `weight = ONE` but excluding `vQuoteBalance` from perp contributions) for the withdrawal check, so only settled, realized value counts toward collateral adequacy.

---

### Proof of Concept

1. User deposits 1000 USDC (quote balance = 1000).
2. User opens a long perp position: 1 ETH at $1000 entry → `vQuoteBalance = -1000`, `amount = 1`.
3. ETH price rises to $2000. Perp health contribution (INITIAL, `longWeightInitial = 0.9`):
   `1 * 2000 * 0.9 + (-1000) = 800`. Total INITIAL health = `1000 (USDC) + 800 = 1800`.
4. User calls `withdrawCollateral(QUOTE_PRODUCT_ID, 1000)`. Post-withdrawal health = `0 + 800 = 800 ≥ 0` → **passes**.
5. ETH price drops to $800. Maintenance health:
   `1 * 800 * maintenanceWeight + (-1000) < 0` → **liquidatable**.
6. Liquidator calls `liquidateSubaccountImpl`. Perp position closed via vQuote adjustments. Liquidatee has zero USDC. Insurance fund covers the negative quote balance.
7. User retains the 1000 USDC withdrawn in step 4. Protocol suffers a loss equal to the bad debt covered by insurance. [1](#0-0) [9](#0-8)

    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }
