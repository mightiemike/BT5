### Title
Merkle Second-Preimage Forgery in `verify_transaction_inclusion` Allows Fake Transaction Inclusion Proof — (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` (v1) function performs no validation that the supplied `tx_id` is a leaf node (a real transaction hash) rather than an internal Merkle tree node. An unprivileged NEAR caller can supply a crafted internal-node hash together with a valid Merkle path that hashes up to the stored `merkle_root`, causing the contract to return `true` for a transaction that was never included in the block.

---

### Finding Description

The vulnerability class in the external report is **insufficient data-context validation**: the wallet SDK accepts the wallet UI in an unintended embedding context (iframe) because it never checks whether the rendering environment is a popup. The analog here is identical in structure: `verify_transaction_inclusion` accepts an unintended input context (an internal Merkle tree node) as a valid transaction identifier because it never checks whether the supplied hash is a leaf.

The contract's own developer documentation explicitly acknowledges this:

> **Important**: This function is vulnerable to the standard Bitcoin merkle tree second-preimage attack — it may return `true` for an internal node hash rather than a real transaction hash. Block headers do not contain the transaction count, so proof depth cannot be validated on-chain. [1](#0-0) 

The function is still a live, callable public NEAR contract method. The `#[deprecated]` Rust attribute emits a compiler warning only; it does not remove the method from the deployed WASM ABI. The v2 variant (`verify_transaction_inclusion_v2`) mitigates the separate 64-byte transaction attack by requiring a coinbase proof, but it still delegates the actual transaction proof check to v1 via `self.verify_transaction_inclusion(args.into())`, leaving the second-preimage path open for the `tx_id` argument. [2](#0-1) 

The coinbase-length equality check in v2 (`args.merkle_proof.len() == args.coinbase_merkle_proof.len()`) does not close the second-preimage path: an attacker who targets an internal node at depth *d* of a tree of depth *D* simply supplies a proof of length *d* for both the coinbase and the target, where the coinbase proof is independently valid at that depth. [3](#0-2) 

---

### Impact Explanation

Any downstream NEAR contract or dapp that calls `verify_transaction_inclusion` (v1) to gate a financial action (e.g., releasing bridged funds, minting wrapped tokens, settling a bet) will receive `true` for a transaction that does not exist. The attacker does not need to mine a block or control any privileged role; they only need to know the `merkle_root` of any already-stored block header and construct a valid internal-node Merkle path to it. This is a **proof-verification forgery** with direct financial impact on any consumer of the verification result.

---

### Likelihood Explanation

The entry point is a public, permissionless view/call function. The Bitcoin Merkle second-preimage attack is well-documented and the construction of a valid internal-node proof requires only knowledge of the stored block's `merkle_root` and the Merkle tree structure, both of which are public. No privileged access, key material, or social engineering is required. Likelihood is **high** for any deployment where a consumer contract relies on v1 without off-chain tx validation.

---

### Recommendation

- **Short term**: Remove or hard-disable (not just deprecate) the v1 `verify_transaction_inclusion` public method from the deployed contract. Any caller that still needs it must be migrated to v2.
- **Long term**: In `verify_transaction_inclusion_v2`, add an explicit check that the `tx_index` is consistent with the proof length (i.e., `tx_index < 2^merkle_proof.len()`), and document that callers must independently verify the raw transaction bytes to confirm `tx_id` is a real leaf. Consider storing the transaction count per block header to enable on-chain depth validation.

---

### Proof of Concept

1. Identify any block already stored in the contract with a known `merkle_root` and at least 2 transactions (so internal nodes exist).
2. Let the block's Merkle tree have depth *D*. Pick any internal node `N` at depth *d < D*. `N` is the hash of two child subtree roots.
3. Construct a Merkle path of length *d* from `N` up to `merkle_root` using the sibling hashes at each level — all values are public from the block's transaction set.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash)
   - `tx_block_blockhash` = the stored block hash
   - `tx_index` = the index corresponding to `N`'s position
   - `merkle_proof` = the *d*-element path constructed above
   - `confirmations = 0`
5. The contract recomputes `compute_root_from_merkle_proof(N, tx_index, &merkle_proof)`, arrives at `merkle_root`, matches the stored header, and returns `true` — despite `N` not being any real transaction. [4](#0-3) [5](#0-4)

    /// Verifies that a transaction is included in a block at a given block height,
    /// with an additional coinbase merkle proof validation.
    /// This is needed to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param tx_id transaction identifier
    /// @param tx_block_blockhash block hash at which transaction is supposedly included
    /// @param tx_index index of transaction in the block's tx merkle tree
    /// @param merkle_proof merkle tree path (concatenated LE sha256 hashes) (does not contain initial transaction_hash and merkle_root)
    /// @param coinbase_tx_id coinbase transaction hash
    /// @param coinbase_merkle_proof merkle proof for the coinbase transaction (must have the same length as merkle_proof)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if tx_id is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Panics
    /// - If `merkle_proof` and `coinbase_merkle_proof` have different lengths
    /// - If `tx_block_blockhash` is not found in the headers pool
    /// - If coinbase merkle proof does not match the block's merkle root
    /// - If the required number of confirmations exceeds the number of stored blocks
    /// - If the block does not belong to the current main chain
    /// - If there are not enough confirmed blocks
    #[pause]
    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
