### Title
Duplicate-Leaf Merkle Phantom-Index Forgery — False Transaction Inclusion for Non-Existent Position - (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` returns `true` for a phantom `tx_index` that does not correspond to any real transaction in an odd-length block. Because `compute_root_from_merkle_proof` is purely positional and the contract stores no transaction count, an attacker can supply the last real transaction hash at the duplicated (phantom) index and produce the correct Merkle root, bypassing the only inclusion check.

---

### Finding Description

**Root cause — no tx_index bounds check against actual transaction count.**

The contract stores only block headers (containing `merkle_root`). Neither `verify_transaction_inclusion` nor `verify_transaction_inclusion_v2` validates that `tx_index < number_of_transactions_in_block`. [1](#0-0) 

The Merkle verifier is purely arithmetic: given `(hash, position, proof)` it recomputes a root with no knowledge of tree size. [2](#0-1) 

**Why the phantom index produces the correct root.**

For a 3-tx block `[tx0, tx1, tx2]`, `merkle_proof_calculator` pads the odd-length level by duplicating the last element: [3](#0-2) 

This yields the padded tree `[tx0, tx1, tx2, tx2]` with root `R = H(H(tx0,tx1), H(tx2,tx2))`.

The proof for `tx2` at real index **2** is `P = [tx2, H(tx0,tx1)]`.

Tracing `compute_root_from_merkle_proof(tx2, 2, P)`:
- step 1, pos=2 (even): `H(tx2, tx2)`, pos→1
- step 2, pos=1 (odd): `H(H(tx0,tx1), H(tx2,tx2))` = **R** ✓

Tracing `compute_root_from_merkle_proof(tx2, 3, P)` (phantom index):
- step 1, pos=3 (odd): `H(tx2, tx2)`, pos→1
- step 2, pos=1 (odd): `H(H(tx0,tx1), H(tx2,tx2))` = **R** ✓

The same proof `P` and hash `tx2` produce the correct root at both index 2 (real) and index 3 (phantom). The only difference between the two paths is whether the first combine is `H(current, proof[0])` or `H(proof[0], current)` — but since `proof[0] == tx2 == current`, both orderings yield `H(tx2, tx2)`, making the two paths arithmetically identical.

**The coinbase guard does not prevent this.**

`verify_transaction_inclusion_v2` adds a coinbase proof check at hardcoded index 0: [4](#0-3) 

For the same 3-tx block, the valid coinbase proof for `tx0` at index 0 is `[tx1, H(tx2,tx2)]` (length 2), matching the phantom tx proof length (also 2). The length-equality guard passes: [5](#0-4) 

The coinbase check is satisfied with a fully legitimate proof. The phantom tx proof then passes the final root comparison in `verify_transaction_inclusion`.

---

### Impact Explanation

`verify_transaction_inclusion_v2` returns `true` for `(tx_id=tx2, tx_index=3)` in a 3-tx block where index 3 does not exist. Any downstream consumer that trusts this return value to confirm "transaction `tx2` is at position 3 in block B" receives a false attestation. If the caller uses `tx_index` to locate outputs, derive payment proofs, or gate state transitions, the phantom position breaks the invariant that a verified `(tx_id, tx_index)` pair corresponds to a real on-chain transaction slot.

---

### Likelihood Explanation

The attack requires only:
1. A real confirmed block with an odd transaction count (extremely common on Bitcoin mainnet).
2. Knowledge of the last transaction hash in that block (publicly available).
3. A valid coinbase proof for that block (also publicly computable from block data).

No privileged role, no key compromise, no special relayer access. Any NEAR account can call `verify_transaction_inclusion_v2` directly. [6](#0-5) 

---

### Recommendation

Add a `tx_count` field to the stored block metadata and reject any proof where `tx_index >= tx_count`. Alternatively, enforce that `tx_index` is strictly less than `2^(proof_length - 1)` only when the proof length implies a tree that would not require duplication at the claimed index — though storing the actual transaction count is the cleaner fix. The coinbase-proof guard alone is insufficient because it validates a different position and does not constrain the range of valid `tx_index` values.

---

### Proof of Concept

```
Block transactions: [tx0, tx1, tx2]   (3 txs, odd-length)

Padded tree:        [tx0, tx1, tx2, tx2]
Level 1:            [H(tx0,tx1),  H(tx2,tx2)]
Root R:              H(H(tx0,tx1), H(tx2,tx2))

Coinbase proof (tx0 at index 0):
  merkle_proof_calculator([tx0,tx1,tx2], 0) = [tx1, H(tx2,tx2)]
  compute_root_from_merkle_proof(tx0, 0, [tx1, H(tx2,tx2)]) = R  ✓

Phantom proof (tx2 at phantom index 3):
  merkle_proof = [tx2, H(tx0,tx1)]   ← same proof as for real index 2
  compute_root_from_merkle_proof(tx2, 3, [tx2, H(tx0,tx1)])
    pos=3 (odd):  H(tx2, tx2),          pos→1
    pos=1 (odd):  H(H(tx0,tx1), H(tx2,tx2)) = R  ✓

Call verify_transaction_inclusion_v2 with:
  tx_id                = tx2
  tx_block_blockhash   = <valid block hash>
  tx_index             = 3          ← phantom, no real tx here
  merkle_proof         = [tx2, H(tx0,tx1)]
  coinbase_tx_id       = tx0
  coinbase_merkle_proof= [tx1, H(tx2,tx2)]
  confirmations        = 0

Result: true   ← false positive for non-existent transaction slot
```

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
