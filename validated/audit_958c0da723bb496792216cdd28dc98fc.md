### Title
Lack of Incentivization for `executeSlowModeTransaction()` Callers Enables Griefing via Slow Mode Queue Spam — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`executeSlowModeTransaction()` in `Endpoint.sol` is a permissionless function that processes queued slow mode transactions. Callers pay gas but receive no reward. The `SLOW_MODE_FEE` collected from submitters accumulates in `slowModeFees` but is never distributed to executors. This mirrors the Divergence Protocol `settle()` issue: rational actors will not execute slow mode transactions on behalf of others, forcing the protocol to centralize execution, and enabling a griefing attack where an adversary spams cheap slow mode submissions to create an unprofitable execution backlog.

---

### Finding Description

`executeSlowModeTransaction()` is declared with no access control and no caller reward:

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);
    nSubmissions += 1;
    slowModeConfig = _slowModeConfig;
}
``` [1](#0-0) 

When a user submits a slow mode transaction, `SLOW_MODE_FEE` ($1) is charged from the submitter and added to the protocol-owned `slowModeFees` counter:

```solidity
chargeSlowModeFee(_getQuote(), sender);
slowModeFees += SLOW_MODE_FEE;
``` [2](#0-1) 

The `SLOW_MODE_FEE` constant is $1 (1,000,000 in 6-decimal USDC): [3](#0-2) 

However, `slowModeFees` is never distributed to the `msg.sender` of `executeSlowModeTransaction()`. The executor pays gas and receives nothing. There is no mechanism anywhere in `Endpoint.sol` or `EndpointTx.sol` that transfers any portion of `slowModeFees` to the executor.

Slow mode transactions are queued with a 3-day delay before they become executable by external callers: [4](#0-3) 

---

### Impact Explanation

- **Centralization**: Because there is no economic incentive for external parties to call `executeSlowModeTransaction()`, the sequencer/protocol owner must execute the queue themselves. This is the same centralization risk identified in the Divergence report.
- **Griefing via queue spam**: An attacker can submit many slow mode transactions (each costs $1 in `SLOW_MODE_FEE`), building a large queue. The protocol is then forced to execute all of them at gas cost per call. On networks where gas cost per execution exceeds $1, each execution is a net loss for the protocol. The attacker's cost is bounded ($1/tx), but the protocol's cost is unbounded in aggregate.
- **Withdrawal delay amplification**: Legitimate users waiting for slow mode withdrawals (e.g., `WithdrawCollateral`) depend on the queue being processed. If the queue is spammed and the protocol is the only executor, user withdrawals are delayed.

---

### Likelihood Explanation

The attack is low-cost and permissionless. Any unprivileged user can call `submitSlowModeTransaction()` with a valid transaction type (e.g., `WithdrawCollateral`) paying only $1 per entry. The 3-day delay before execution does not prevent queue buildup. The attacker does not need any special role, linked signer, or privileged access. [5](#0-4) 

---

### Recommendation

Distribute a portion of the collected `slowModeFees` to the `msg.sender` of `executeSlowModeTransaction()` as an execution bounty. This creates a self-sustaining incentive: submitters pay a fee, executors earn a reward, and the protocol no longer needs to centralize queue processing. Spam protection (e.g., raising `SLOW_MODE_FEE` or adding per-address rate limits) should accompany this change to prevent the bounty itself from being gamed.

---

### Proof of Concept

1. Attacker calls `submitSlowModeTransaction()` 1,000 times with valid `WithdrawCollateral` payloads, paying $1 each ($1,000 total).
2. After 3 days, 1,000 entries become executable.
3. No external party calls `executeSlowModeTransaction()` because there is no reward.
4. The protocol sequencer must call it 1,000 times. On a network where each call costs >$1 in gas, the protocol loses money on every execution.
5. Legitimate users' slow mode withdrawals are delayed or blocked behind the spam queue.
6. The attacker's cost is fixed at $1,000; the protocol's cost scales with gas price and queue depth. [1](#0-0) [6](#0-5) [7](#0-6)

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
