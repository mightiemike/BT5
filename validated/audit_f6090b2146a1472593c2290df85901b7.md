### Title
`calculate_next_base_gas_price` and `calculate_next_l2_gas_price_for_fin` Always Read `gas_target` and `gas_price_max_change_denominator` from `VersionedConstants::latest_constants()` Instead of the Block-Version-Appropriate Constants, Producing Wrong `next_l2_gas_price` at Version Boundaries - (`File: crates/apollo_consensus_orchestrator/src/fee_market/mod.rs`)

---

### Summary

`calculate_next_l2_gas_price_for_fin` fetches `gas_target` from `VersionedConstants::latest_constants()` at call time, and `calculate_next_base_gas_price` fetches `gas_price_max_change_denominator` and `max_block_size` from the same dynamic source. Neither function accepts a `StarknetVersion` parameter. Because `gas_target` changes substantially across orchestrator versioned-constant files (3.2 B → 4.0 B → 1.5 B → 1.04 B), any node running software whose `LATEST` version differs from the version under which a block was originally built will compute a different `next_l2_gas_price`. The same pattern appears in `sequencer_consensus_context.rs` for `fee_proposal_window_size` and `fee_proposal_margin_ppt`.

---

### Finding Description

**Root cause — dynamic fetch instead of version-snapshotted constant**

In `calculate_next_l2_gas_price_for_fin`:

```rust
// crates/apollo_consensus_orchestrator/src/fee_market/mod.rs  line 70
let gas_target = VersionedConstants::latest_constants().gas_target;
```

In `calculate_next_base_gas_price`:

```rust
// crates/apollo_consensus_orchestrator/src/fee_market/mod.rs  line 92
let versioned_constants = VersionedConstants::latest_constants();
// line 94 – assertion uses versioned_constants.max_block_size
// line 125 – denominator uses versioned_constants.gas_price_max_change_denominator
let denominator =
    gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
```

Neither function receives a `StarknetVersion` argument. The `apollo_versioned_constants::VersionedConstants` struct is keyed by `StarknetVersion` and the `gas_target` field differs across every shipped JSON file:

| Version | `gas_target` | `max_block_size` |
|---------|-------------|-----------------|
| V0_14_0 | 3 200 000 000 | 4 000 000 000 |
| V0_14_1 | 4 000 000 000 | 5 000 000 000 |
| V0_14_2 | 1 500 000 000 | 5 800 000 000 |
| V0_14_3 | 1 040 000 000 | 5 800 000 000 |
| V0_14_4 | 1 040 000 000 | 5 800 000 000 |

The EIP-1559 price-change formula is:

```
denominator = gas_target × gas_price_max_change_denominator
price_change = (price × |gas_used − gas_target|) / denominator
```

Using `gas_target = 1.04 B` (V0_14_4) instead of `gas_target = 4.0 B` (V0_14_1) makes the denominator ~3.85× smaller, so the price moves ~3.85× faster per block than the protocol intends for that version.

The same `latest_constants()` pattern appears in `sequencer_consensus_context.rs` for `fee_proposal_window_size` and `fee_proposal_margin_ppt`, and in `validate_proposal.rs` for the `fee_proposal_bounds` check:

```rust
// sequencer_consensus_context.rs  line 431
VersionedConstants::latest_constants().fee_proposal_window_size,

// sequencer_consensus_context.rs  line 490
VersionedConstants::latest_constants().fee_proposal_margin_ppt,

// validate_proposal.rs  line 403
VersionedConstants::latest_constants().fee_proposal_margin_ppt,
```

**Trigger path**

1. Network is running at Starknet version V0_14_1 (`gas_target = 4 B`).
2. A software upgrade ships V0_14_2 binaries (`gas_target = 1.5 B`, a 62.5 % drop).
3. The proposer upgrades first; it calls `calculate_next_l2_gas_price_for_fin` with `gas_target = 1.5 B`.
4. Validators still on V0_14_1 binaries call the same function with `gas_target = 4 B`.
5. Both sides compute a different `next_l2_gas_price` and embed it in their view of the block header.
6. Consensus diverges, or the stored `next_l2_gas_price` is wrong, propagating the error into every subsequent block's starting gas price.

Even on a fully-upgraded network, re-execution of historical blocks (e.g., during state sync or proof generation) will produce a wrong `next_l2_gas_price` because the re-executing node always uses `latest_constants()` rather than the constants that were active when the block was originally built.

---

### Impact Explanation

`next_l2_gas_price` is an economic parameter: it is the base fee charged to every transaction in the next block. Computing it with the wrong `gas_target` denominator causes:

- **Incorrect fee accounting** — the base fee rises or falls at the wrong rate, directly affecting transaction costs for all users.
- **Consensus split at upgrade boundaries** — proposer and validators running different software versions compute different `next_l2_gas_price` values, potentially causing valid blocks to be rejected or invalid blocks to be accepted.
- **Wrong committed state** — the `next_l2_gas_price` stored in the block header is wrong, and every subsequent block inherits the error as its starting price.

This matches the allowed impact: *"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

---

### Likelihood Explanation

`gas_target` has already changed four times across the five shipped versioned-constant files. Any future upgrade that changes `gas_target` or `gas_price_max_change_denominator` will trigger the divergence. No special attacker capability is required; the inconsistency is triggered automatically by the normal software-upgrade process.

---

### Recommendation

Pass the block's `StarknetVersion` into `calculate_next_l2_gas_price_for_fin` and `calculate_next_base_gas_price`, and resolve the constants with `VersionedConstants::get(&starknet_version)?` instead of `VersionedConstants::latest_constants()`. Apply the same fix to the `fee_proposal_window_size` and `fee_proposal_margin_ppt` lookups in `sequencer_consensus_context.rs` and `validate_proposal.rs`.

```rust
// proposed signature
pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    starknet_version: &StarknetVersion,   // <-- add
    override_l2_gas_price_fri: Option<u128>,
    min_l2_gas_price_per_height: &[PricePerHeight],
    fee_actual: Option<GasPrice>,
) -> GasPrice {
    let vc = VersionedConstants::get(starknet_version)
        .unwrap_or_else(|_| VersionedConstants::latest_constants());
    let gas_target = vc.gas_target;
    ...
}
```
