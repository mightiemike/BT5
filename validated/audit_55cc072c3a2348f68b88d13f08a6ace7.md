### Title
`TransferQuote` Registers and Funds Sanctioned Recipient Subaccount Without Sanctions Check — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `TransferQuote` fast-path handler in `EndpointTx.sol` calls `_recordSubaccount` on the recipient and then transfers quote tokens to them via `clearinghouse.transferQuote`, but never calls `requireUnsanctioned` on the recipient. Every other subaccount-registration path enforces a sanctions check before registration. This inconsistency allows a non-sanctioned sender to register a sanctioned address as a valid protocol subaccount and credit it with funds, bypassing the on-chain sanctions enforcement.

---

### Finding Description

The `depositCollateralWithReferral` entry point explicitly guards both the depositor and the depositee:

```solidity
// Endpoint.sol
requireUnsanctioned(msg.sender);   // line 134
requireUnsanctioned(sender);       // line 135
``` [1](#0-0) 

`submitSlowModeTransactionImpl` similarly calls `requireUnsanctioned(sender)` before queuing any slow-mode transaction: [2](#0-1) 

However, the `TransferQuote` branch inside `processTransactionImpl` (the sequencer fast path) does neither:

```solidity
_recordSubaccount(signedTx.tx.recipient);   // line 598 — registers recipient
validateSignedTx(                            // line 599 — validates SENDER only
    signedTx.tx.sender, ...
);
...
clearinghouse.transferQuote(signedTx.tx);   // line 614 — credits recipient
``` [3](#0-2) 

`_recordSubaccount` unconditionally writes the recipient into `subaccountIds`, making it pass all future `requireSubaccount` checks: [4](#0-3) 

The analog to STRX's `forceAllow()` is direct: just as `forceAllow()` assumed the address being reinstated was previously compliant (when it may only have been denylisted), `TransferQuote` assumes the recipient is a valid, unsanctioned address when it may be sanctioned. Both functions combine two state-mutating actions — registration and asset credit — without independently verifying the compliance status of the affected address.

---

### Impact Explanation

A sanctioned address that has never deposited and would be blocked by `requireUnsanctioned` in every normal entry point can:

1. **Receive quote-token funds** — `clearinghouse.transferQuote` credits the sanctioned subaccount with real protocol balance.
2. **Gain a registered subaccount** — `_recordSubaccount` sets `subaccountIds[recipient] != 0`, so the address now passes `requireSubaccount` in all subsequent on-chain checks.
3. **Interact via the sequencer fast path** — `processTransactionImpl` (the sequencer path) does not call `requireUnsanctioned` anywhere; only `requireSubaccount` is checked via `validateSignedTx`. A registered sanctioned address therefore passes all on-chain fast-path validation.

The corrupted state is: `subaccountIds[sanctionedSubaccount]` becomes non-zero and the clearinghouse balance for that subaccount becomes positive, both without the sanctions gate that every other registration path enforces.

---

### Likelihood Explanation

The trigger requires a non-sanctioned sender to submit a `TransferQuote` naming a sanctioned address as recipient. This can happen:

- **Unknowingly** — the sender does not know the recipient is sanctioned.
- **Collusively** — the sanctioned party controls a second, non-sanctioned address and uses it to bootstrap the sanctioned subaccount.

The sequencer processes the transaction on-chain; the on-chain code contains no recipient sanctions gate. Whether the sequencer's off-chain logic independently checks recipient sanctions is outside the on-chain scope and not guaranteed by the contract code.

---

### Recommendation

Add a `requireUnsanctioned` check on the recipient address inside the `TransferQuote` handler, mirroring the pattern used in `depositCollateralWithReferral`:

```solidity
// Before _recordSubaccount and clearinghouse.transferQuote
requireUnsanctioned(address(uint160(bytes20(signedTx.tx.recipient))));
```

Alternatively, move `_recordSubaccount` to after the sanctions check so that a failed check never partially mutates state.

---

### Proof of Concept

1. Address `S` is sanctioned (present in the `ISanctionsList`). It has no subaccount registered (`subaccountIds[S_subaccount] == 0`).
2. Address `B` is non-sanctioned with a registered subaccount and a positive quote balance.
3. `B` submits a `TransferQuote` transaction to the sequencer with `recipient = S_subaccount` and some `amount`.
4. Sequencer calls `processTransaction` → `processTransactionImpl`.
5. On-chain: `_recordSubaccount(S_subaccount)` executes — `subaccountIds[S_subaccount]` is now non-zero.
6. On-chain: `clearinghouse.transferQuote` credits `S_subaccount` with `amount` of quote tokens.
7. `S` now has a registered subaccount with funds. All future `requireSubaccount(S_subaccount)` checks pass. `S` can interact with the sequencer fast path (trades, `LinkSigner`, etc.) without ever passing a `requireUnsanctioned` gate. [5](#0-4) [1](#0-0)

    function _recordSubaccount(bytes32 subaccount) internal {
        if (subaccountIds[subaccount] == 0) {
            subaccountIds[subaccount] = ++numSubaccounts;
            subaccounts[numSubaccounts] = subaccount;
        }
    }

    function requireSubaccount(bytes32 subaccount) internal view {
        require(
            subaccount == X_ACCOUNT ||
                subaccount == N_ACCOUNT ||
                (subaccountIds[subaccount] != 0),
            ERR_REQUIRES_DEPOSIT
        );
    }
