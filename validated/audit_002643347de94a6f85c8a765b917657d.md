### Title
Changing `withdrawPool` in `Clearinghouse.sol` Enables Fast-Withdrawal Replay Attacks Against the New Pool — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.setWithdrawPool()` allows the owner to replace the active `WithdrawPool` contract. The new pool starts with a completely empty `markedIdxs` mapping and `minIdx = 0`. Any fast withdrawal that was already processed and marked in the **old** pool is invisible to the new pool, allowing the original recipient to replay the same signed fast-withdrawal transaction and drain tokens from the new pool a second time.

---

### Finding Description

`Clearinghouse` stores a mutable `withdrawPool` address and exposes a setter:

```solidity
// Clearinghouse.sol L750-753
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));
    withdrawPool = _withdrawPool;
}
``` [1](#0-0) 

All token transfers for withdrawals are routed through this address:

```solidity
// Clearinghouse.sol L377-385
function handleWithdrawTransfer(...) internal virtual {
    token.safeTransfer(withdrawPool, uint256(amount));
    BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
}
``` [2](#0-1) 

The replay-prevention state lives entirely inside each `WithdrawPool` instance. `submitFastWithdrawal` guards against double-spend using two checks:

```solidity
// BaseWithdrawPool.sol L86-88
require(!markedIdxs[idx], "Withdrawal already submitted");
require(idx > minIdx, "idx too small");
markedIdxs[idx] = true;
``` [3](#0-2) 

And `submitWithdrawal` (called by the sequencer path) sets `minIdx` to the most recently sequenced withdrawal index:

```solidity
// BaseWithdrawPool.sol L122-131
markedIdxs[idx] = true;
minIdx = idx;
handleWithdrawTransfer(token, sendTo, amount);
``` [4](#0-3) 

When `setWithdrawPool` replaces the pool, the new contract instance has `markedIdxs` entirely empty and `minIdx = 0`. Every `idx` that was legitimately processed and marked in the old pool is now **unmarked** in the new pool, and `idx > 0` trivially satisfies the `minIdx` guard.

---

### Impact Explanation

A user who previously submitted a fast withdrawal with index `idx = X` and received their tokens can, after the pool is replaced, call `submitFastWithdrawal` again with the same `transaction` bytes and `signatures` (both are public on-chain data). The new pool sees `markedIdxs[X] == false` and `X > minIdx (0)`, so it passes both guards, marks the index, and transfers tokens to the user a second time. This drains pre-funded liquidity from the new pool, directly stealing ERC-20 collateral from liquidity providers or other users whose withdrawals are pending in the new pool.

The corrupted invariant is: *each `(idx, transaction)` pair must be processed at most once across the lifetime of the protocol*. Replacing the pool breaks this invariant because the deduplication state is not migrated.

---

### Likelihood Explanation

The precondition is an owner call to `setWithdrawPool`, which is a plausible operational action (pool upgrade, migration, bug fix). Once the new pool is deployed and funded — a necessary step before fast withdrawals can resume — any prior fast-withdrawal recipient can immediately replay their transaction. The replay requires only public on-chain data (transaction bytes + signatures already emitted/stored), no privileged access, and no special timing beyond the pool being funded.

---

### Recommendation

1. **Make `withdrawPool` immutable** (preferred): deploy a new `Clearinghouse` proxy if the pool must change, preserving the deduplication invariant.
2. **Migrate `markedIdxs` and `minIdx`**: if mutability is required, require the new pool to be initialized with `minIdx` set to at least the highest `idx` ever processed by the old pool, so the `idx > minIdx` guard blocks all historical replays.
3. **Alternatively**, enforce `require(markedIdxs[idx] == false)` at the `Clearinghouse` level (not just inside the pool) using a global nonce registry that survives pool replacement.

---

### Proof of Concept

1. User A submits a fast withdrawal: `submitFastWithdrawal(idx=500, txBytes, sigs)` against `OldPool`. Tokens are transferred to User A. `OldPool.markedIdxs[500] = true`, `OldPool.minIdx = 500`.
2. Owner calls `Clearinghouse.setWithdrawPool(NewPool)`. `NewPool.markedIdxs` is empty; `NewPool.minIdx = 0`.
3. Liquidity providers fund `NewPool` so fast withdrawals can resume.
4. User A calls `NewPool.submitFastWithdrawal(idx=500, txBytes, sigs)`.
   - `NewPool.markedIdxs[500] == false` → passes.
   - `500 > NewPool.minIdx (0)` → passes.
   - Verifier accepts the original signatures (transaction bytes and idx are unchanged).
   - `NewPool` transfers tokens to User A again.
5. User A has received double the tokens; `NewPool` liquidity is drained by the replayed amount. [1](#0-0) [5](#0-4)
