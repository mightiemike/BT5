Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the address that actually provides tokens via callback) and enforces the allowlist only against `owner` (the caller-supplied LP position recipient). Because `owner` is freely chosen by any caller, any address can bypass the deposit allowlist by specifying an allowlisted address as `owner`. This fully defeats the access-control invariant the extension is designed to enforce.

## Finding Description
`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` argument and forwards `msg.sender` as `sender` to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both `sender` and `owner` and passes them to the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` leaves the first parameter (`sender`) unnamed, discarding it entirely, and checks only `owner`: [3](#0-2) 

The admin-facing setter names the gated subject `depositor`, confirming the intended guard is on the token-providing caller, not the position recipient: [4](#0-3) 

Since `owner` is a free argument chosen by the caller, any unauthorized address `bob` can pass the check by supplying any allowlisted address `alice` as `owner`. The pool then invokes `bob`'s callback to pull tokens from `bob`, and mints the LP position to `alice`. No existing guard prevents this: `addLiquidity` has no `msg.sender == owner` restriction (unlike `removeLiquidity`), and the extension hook never inspects `sender`. [5](#0-4) 

## Impact Explanation
The deposit allowlist guard is completely bypassed. Any unauthorized address can inject liquidity into a pool administratively restricted to specific depositors (e.g., institutional-only, KYC-gated, or whitelist-only). This breaks the core access-control invariant the extension enforces. Additionally, `removeLiquidity` enforces `msg.sender == owner`, so the allowlisted address that received the unwanted LP position must act to unwind it — constituting a griefing vector against allowlisted LPs. [6](#0-5) 

## Likelihood Explanation
The bypass requires a single `addLiquidity` call with `owner` set to any allowlisted address. No privileged access, flash loan, or multi-step setup is needed. Any address that can call the pool or a router that calls the pool can trigger this. Likelihood is **High**.

## Recommendation
Replace the `owner` check with a `sender` check in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This ensures the guard applies to the address that actually provides tokens, matching the intent of `setAllowedToDeposit`.

## Proof of Concept
1. Pool is deployed with `DepositAllowlistExtension`; `allowAllDepositors[pool] = false`.
2. Admin calls `setAllowedToDeposit(pool, alice, true)`. Only `alice` is allowlisted.
3. `bob` (not allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `extension.beforeAddLiquidity(bob, alice, ...)`.
5. Check evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
6. `bob`'s callback is invoked; `bob` transfers tokens into the pool.
7. `alice` receives the LP position she never requested.
8. `alice` must call `removeLiquidity` herself to unwind the position (enforced by `msg.sender == owner` check in `removeLiquidity`).
