### Title
`py_deploy_account` Binds `contract_address` from Python-Supplied `sender_address` Without Cryptographic Recomputation at the Python-to-Rust Conversion Boundary — (`File: crates/native_blockifier/src/py_deploy_account.rs`)

---

### Summary

The `py_deploy_account` conversion function populates the `contract_address` field of the executable `DeployAccountTransaction` directly from the Python object's `sender_address` attribute, without independently deriving it from the transaction's cryptographic parameters. The canonical Rust path (`DeployAccountTransaction::create`) always recomputes the address from `(class_hash, salt, constructor_calldata, deployer=0)`. The Python wrapper skips this step, creating a conversion boundary where the wrong identity source is bound into the executable object — an exact structural analog to M-13's incorrect `Sender` binding during a cross-layer conversion.

---

### Finding Description

In `py_deploy_account`, the `contract_address` of the resulting `DeployAccountTransaction` is taken verbatim from the Python object's `sender_address` attribute:

```rust
let contract_address =
    ContractAddress::try_from(py_attr::<PyFelt>(py_tx, "sender_address")?.0)?;
Ok(DeployAccountTransaction { tx, tx_hash, contract_address })
``` [1](#0-0) 

The canonical Rust path for constructing an executable `DeployAccountTransaction` is `DeployAccountTransaction::create`, which independently computes the address:

```rust
let contract_address = deploy_account_tx.calculate_contract_address()?;
let tx_hash = deploy_account_tx.calculate_transaction_hash(chain_id, &deploy_account_tx.version())?;
Ok(Self { tx: deploy_account_tx, tx_hash, contract_address })
``` [2](#0-1) 

The `contract_address` field is then used in two critical places in the blockifier:

1. **Constructor execution** — `storage_address: self.contract_address()` determines where the new account contract is deployed in state: [3](#0-2) 

2. **`TransactionInfo` construction** — `sender_address: self.contract_address()` is the address used for nonce checking, fee charging, and account `__validate__` dispatch: [4](#0-3) 

The transaction hash for a `DeployAccount` V3 transaction commits to the cryptographically-derived contract address (computed from `class_hash`, `contract_address_salt`, `constructor_calldata`, and `deployer=0`): [5](#0-4) 

The Python wrapper also trusts the Python-provided `hash_value` without recomputation: [1](#0-0) 

This means both the `tx_hash` and `contract_address` fields of the executable object are sourced from the Python layer without any Rust-side verification against the transaction's own parameters.

---

### Impact Explanation

If the Python-provided `sender_address` diverges from the cryptographically-derived address — due to a Python/Rust implementation discrepancy in `calculate_contract_address`, a version mismatch between the Python OS and the Rust blockifier, or a bug in the Python sequencer — the blockifier will:

- Deploy the account contract at the wrong storage address (wrong state written to the Patricia-Merkle trie).
- Charge fees from the wrong address (wrong balance deduction).
- Call `__validate__` on the wrong account contract (wrong signature verification, potentially accepting an invalid transaction or rejecting a valid one).
- Produce a receipt and state diff that are inconsistent with the transaction hash, which commits to the correct cryptographic address.

This matches the impact category: **Wrong state, receipt, or revert result from blockifier/execution logic for accepted input**, and **Transaction conversion logic binds the wrong signer/executable payload**.

---

### Likelihood Explanation

The trigger requires a divergence between the Python layer's address computation and the Rust layer's `calculate_contract_address`. Such divergence can arise from:

- A protocol upgrade that changes the address derivation formula in one layer but not the other.
- A subtle difference in how `constructor_calldata` is hashed (e.g., length prefix handling) between the Python Cairo runner and the Rust implementation.
- A bug introduced in the Python sequencer's address pre-computation.

The Python sequencer is trusted infrastructure, so this is not directly exploitable by an unprivileged RPC user in isolation. However, the broken canonicalization invariant means any divergence between layers silently produces wrong state rather than a detectable error, because the Rust blockifier never cross-checks the Python-provided address against the transaction parameters.

---

### Recommendation

In `py_deploy_account`, after constructing the inner `tx` object, independently recompute the contract address using the Rust `calculate_contract_address` implementation and assert it matches the Python-provided `sender_address`:

```rust
let derived_address = tx.calculate_contract_address()?;
let python_address = ContractAddress::try_from(py_attr::<PyFelt>(py_tx, "sender_address")?.0)?;
if derived_address != python_address {
    return Err(NativeBlockifierInputError::ContractAddressMismatch {
        derived: derived_address,
        provided: python_address,
    });
}
Ok(DeployAccountTransaction { tx, tx_hash, contract_address: derived_address })
```

This mirrors the pattern used in `DeployAccountTransaction::create` and closes the conversion boundary gap. [2](#0-1) 

---

### Proof of Concept

1. Python sequencer computes `sender_address = A` for a `DeployAccount` transaction (e.g., using a slightly different hash of `constructor_calldata`).
2. Rust `calculate_contract_address` would compute address `B ≠ A` for the same parameters.
3. `py_deploy_account` sets `contract_address = A` (Python-provided) without verification.
4. Blockifier executes the constructor at storage address `A`, writes class hash and nonce to `A`.
5. The transaction hash (also Python-provided) commits to address `A`, so the hash/address pair is internally consistent from the Python perspective.
6. However, any external verifier (e.g., the Starknet OS prover, a full node re-executing the block) that independently computes the contract address from the transaction parameters will derive `B`, not `A`.
7. Result: the sequencer's committed state diff places the account at `A`, but the proof system expects it at `B` — a state/proof inconsistency that can cause proof failure or, if the prover also trusts the sequencer's address, silently wrong state. [6](#0-5)

pub fn py_deploy_account(py_tx: &PyAny) -> NativeBlockifierResult<DeployAccountTransaction> {
    let version = py_attr::<PyFelt>(py_tx, "version")?.0;
    // TODO(Dori): Make TransactionVersion an enum and use match here.
    let tx = if version == Felt::ONE {
        let py_deploy_account_tx: PyDeployAccountTransactionV1 = py_tx.extract()?;
        let deploy_account_tx = DeployAccountTransactionV1::from(py_deploy_account_tx);
        Ok(starknet_api::transaction::DeployAccountTransaction::V1(deploy_account_tx))
    } else if version == Felt::THREE {
        let py_deploy_account_tx: PyDeployAccountTransactionV3 = py_tx.extract()?;
        let deploy_account_tx = DeployAccountTransactionV3::try_from(py_deploy_account_tx)?;
        Ok(starknet_api::transaction::DeployAccountTransaction::V3(deploy_account_tx))
    } else {
        Err(NativeBlockifierInputError::UnsupportedTransactionVersion {
            tx_type: TransactionType::DeployAccount,
            version: version.to_biguint(),
        })
    }?;

    let tx_hash = TransactionHash(py_attr::<PyFelt>(py_tx, "hash_value")?.0);
    let contract_address =
        ContractAddress::try_from(py_attr::<PyFelt>(py_tx, "sender_address")?.0)?;
    Ok(DeployAccountTransaction { tx, tx_hash, contract_address })
