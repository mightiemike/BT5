### Title
Per-Segment `TruncateInt()` in Bonus Reward Accumulation Causes Dust Loss for Delegators - (File: `x/tieredrewards/keeper/bonus_rewards.go`)

### Summary

`computeSegmentBonus` truncates each time-segment's bonus to `math.Int` independently. When `processEventsAndClaimBonus` sums N segments, up to N units of the smallest denomination (basecro) are silently discarded instead of being paid to the position owner.

### Finding Description

`computeSegmentBonus` computes the bonus for a single time segment and immediately calls `TruncateInt()` to convert the result from `math.LegacyDec` to `math.Int`:

```go
tokens := pos.Delegation.Shares.Mul(tokensPerShare)

return tokens.
    Mul(tier.BonusApy).
    MulInt64(durationSeconds).
    QuoInt64(types.SecondsPerYear).
    TruncateInt()   // ← truncates fractional part of THIS segment
``` [1](#0-0) 

`processEventsAndClaimBonus` calls `computeSegmentBonus` once per validator event and once for the final open segment, accumulating the already-truncated integers:

```go
totalBonus := math.ZeroInt()
for _, entry := range events {
    ...
    bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
    totalBonus = totalBonus.Add(bonus)   // adds already-truncated value
    ...
}
...
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
totalBonus = totalBonus.Add(bonus)
``` [2](#0-1) 

Because each call to `computeSegmentBonus` independently truncates its fractional part, the sum of N segments loses up to N units of precision. The mathematically correct approach is to accumulate all segments in `math.LegacyDec` and call `TruncateInt()` exactly once on the total.

The constant `SecondsPerYear = 31_557_600` is large relative to typical segment durations, making the fractional remainder of each segment non-trivial. [3](#0-2) 

### Impact Explanation

Every call to `MsgClaimTierRewards` (or any reward-settling handler such as `MsgTierUndelegate`, `MsgTierRedelegate`, `MsgAddToTierPosition`, `MsgClearPosition`, `MsgExitTierWithDelegation`) that processes N validator events loses up to N basecro of bonus rewards that should have been paid to the position owner. The discarded dust remains in the rewards pool rather than being transferred to the owner. The contract fails to deliver the exact promised bonus return, but no value leaves the system.

### Likelihood Explanation

Any delegator with a tier position on a validator that has experienced at least one slash, unbond, or bond event before the delegator claims will be affected. Validators are routinely slashed and jailed/unjailed on live networks. A position that has accumulated several events before claiming (e.g., two slashes and a rebond = 3 segments) loses up to 2 basecro per claim. The loss is proportional to the number of pending events, not to the stake size, so it affects all position sizes equally.

### Recommendation

Change `computeSegmentBonus` to return `math.LegacyDec` instead of `math.Int`, accumulate all segment values in `math.LegacyDec` inside `processEventsAndClaimBonus`, and call `TruncateInt()` exactly once on the final total before constructing `bonusCoins`. This mirrors the mathematically correct single-truncation pattern and eliminates per-segment precision loss.

### Proof of Concept

Given:
- `shares = 1_000_000_000` (1 basecro worth of shares at 1:1 rate)
- `tokensPerShare = 1.0`
- `bonusApy = 0.04` (4%)
- Two equal segments of `SecondsPerYear / 2` seconds each (total = 1 full year)

**Current behaviour (two separate calls):**
```
seg1 = floor(1e9 × 1.0 × 0.04 × (31557600/2) / 31557600)
     = floor(1e9 × 0.04 × 0.5)
     = floor(20_000_000.0) = 20_000_000

seg2 = floor(1e9 × 1.0 × 0.04 × (31557600/2) / 31557600)
     = 20_000_000

total_paid = 20_000_000 + 20_000_000 = 40_000_000
```

**Correct behaviour (single call over full year):**
```
total = floor(1e9 × 1.0 × 0.04 × 31557600 / 31557600)
      = floor(40_000_000.0) = 40_000_000
```

In this clean example the values happen to be exact. With non-round durations (e.g., `durationSeconds` not a multiple of `SecondsPerYear`), each segment independently truncates its fractional remainder. For example, if each segment's exact value is `X.7`, two segments pay `floor(X.7) + floor(X.7) = 2X` instead of `floor(2X + 1.4) = 2X + 1`, losing 1 basecro. With N segments the loss is up to N−1 basecro per claim.
