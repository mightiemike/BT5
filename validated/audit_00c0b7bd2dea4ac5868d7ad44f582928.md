### Title
Fork Difficulty Desynchronization: Dogecoin Per-Block Retarget Uses Stale Mainchain Ancestor Timestamp Instead of Fork Ancestor Timestamp — (`contract/src/dogecoin.rs`)

---

### Summary

In `get_next_work_required` inside `contract/src/dogecoin.rs`, when validating fork (non-mainchain) blocks, the difficulty calculation fetches the "first block" timestamp via `get_header_by_height`, which always resolves against the **mainchain** height-to-header index. For any fork block that is two or more blocks deep from the fork point, this returns a **different block** than the fork's actual ancestor at that height. The resulting timespan used for Digishield retargeting is computed from the wrong block's timestamp, causing the expected `bits` value to be incorrect. This is a direct cross-module desynchronization: the fork chain's actual state (its own ancestor timestamps) is decoupled from the derived value (the required difficulty) that is computed from it.

---

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` (lines 229–298) handles Dogecoin's per-block Digishield difficulty adjustment (active after block height 145,000, where `difficulty_adjustment_interval = 1`). For each new block, it computes `height_first = prev_block_header.block_height - blocks_to_go_back` and then fetches the timestamp of the block at that height: [1](#0-0) 

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

The TODO comment at line 291 explicitly flags this as a known unresolved correctness concern. [2](#0-1) 

`get_header_by_height` is implemented in `contract/src/lib.rs` and **always** resolves against `mainchain_height_to_header`: [3](#0-2) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

For a fork block at height `H+2` where the fork diverged at height `H`:

- `prev_block_header` = fork block at `H+1` (correctly fetched via `prev_block_hash` chain walk)
- `height_first = H`
- `get_header_by_height(H)` → **mainchain block at H** (timestamp `T_main`)
- Correct value → **fork block at H** (timestamp `T_fork`, attacker-controlled)

The Digishield timespan is computed as: [4](#0-3) 

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
```

`first_block_time` is `T_main` (mainchain), but should be `T_fork` (fork ancestor). These diverge whenever the fork block at `H` has a different timestamp than the mainchain block at `H`, which is always possible since the attacker controls the fork block's timestamp (subject only to the MTP check).

The `check_pow` call that invokes this function happens **before** `submit_block_header_inner` and `reorg_chain`, so the mainchain state is fully intact and consistently wrong for fork blocks at depth ≥ 2. [5](#0-4) 

---

### Impact Explanation

**Incorrect `bits` accepted for fork blocks.** If the attacker sets `T_fork < T_main` (fork block at `H` has an earlier timestamp than the mainchain block at `H`, which is valid as long as it exceeds MTP), then:

- Used timespan: `T_fork+1 - T_main` (shorter, because `T_main > T_fork`)
- Correct timespan: `T_fork+1 - T_fork` (longer)
- Shorter timespan → Digishield produces a **harder** (lower) target → higher `bits` value required
- `work_from_bits` for harder bits yields **more work per block**

The fork accumulates more `chain_work` than it legitimately earned. Since reorg is triggered when `current_header.chain_work > total_main_chain_chainwork`, an attacker can manufacture a fork that appears to have more accumulated work than the honest mainchain, triggering a false chain reorganization. [6](#0-5) 

After a false reorg, `mainchain_tip_blockhash` points to the attacker's fork tip. All subsequent calls to `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` validate SPV proofs against the attacker's chain, not the honest Bitcoin chain. Downstream NEAR contracts consuming these verification results would accept fraudulent transaction inclusion proofs. [7](#0-6) 

- **Impact**: High — canonical chain corruption; SPV proof verification results become untrustworthy
- **Likelihood**: Medium — requires submitting a multi-block fork (≥2 blocks deep from fork point), which is a standard relayer operation reachable by any unprivileged NEAR caller with the relayer role

---

### Likelihood Explanation

The `submit_blocks` entry point is accessible to any account granted the trusted relayer role, which is a normal operational role with no privileged key assumption. Submitting a fork is an explicitly supported protocol flow (the contract stores fork headers and handles reorgs). A fork 2+ blocks deep is the minimum required to trigger the bug, and this is a routine scenario in any active chain. The attacker only needs to craft block timestamps within the MTP constraint, which is straightforward.

---

### Recommendation

Replace the `get_header_by_height` lookup with a chain walk that follows `prev_block_hash` links from `prev_block_header` backward by `blocks_to_go_back` steps, using `get_prev_header`. This ensures the timestamp used for difficulty calculation is always the fork's actual ancestor, not the mainchain block at the same height:

```rust
// Instead of:
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;

// Use:
let mut ancestor = prev_block_header.clone();
for _ in 0..blocks_to_go_back {
    ancestor = blocks_getter.get_prev_header(&ancestor.block_header);
}
let first_block_time = ancestor.block_header.time;
```

This mirrors the correct approach used in `zcash.rs` and `bitcoin.rs`'s `pow_allow_min_difficulty_blocks` path, which walk the chain via `get_prev_header` rather than looking up by height. [8](#0-7) 

---

### Proof of Concept

**Setup**: Dogecoin mainchain at height 145,002 (per-block Digishield active). Mainchain block at height 145,001 has timestamp `T_main = 1700000100`.

**Attack**:

1. Attacker submits fork block at height 145,001 (diverging from mainchain at 145,000) with timestamp `T_fork = 1700000000` (100 seconds earlier than `T_main`, above MTP). This passes `check_pow` since MTP < `T_fork`.

2. Attacker submits fork block at height 145,002 with timestamp `T_fork+1 = 1700000060`.

3. For fork block at height 145,003:
   - `prev_block_header` = fork block at 145,002 (correct, via `prev_block_hash`)
   - `height_first = 145,001`
   - `get_header_by_height(145,001)` → **mainchain block**, timestamp `T_main = 1700000100`
   - Used timespan: `1700000060 - 1700000100 = -40` → clamped to `min_timespan` → hardest allowed difficulty
   - Correct timespan: `1700000060 - 1700000000 = 60` → normal difficulty

4. Fork block at 145,003 is accepted with the hardest allowed `bits`, yielding maximum `work_from_bits` per block. [9](#0-8) 

5. After enough such blocks, `fork.chain_work > mainchain.chain_work` → `reorg_chain` is triggered → `mainchain_tip_blockhash` is updated to the attacker's fork tip. [10](#0-9) 

6. All subsequent `verify_transaction_inclusion` calls validate against the attacker's chain.
