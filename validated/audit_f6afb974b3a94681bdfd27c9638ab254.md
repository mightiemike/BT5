### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase-Proof Security Check Added in `verify_transaction_inclusion_v2` — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle-proof forgery vulnerability by requiring a coinbase proof. However, the original `verify_transaction_inclusion` function remains `pub` and is directly callable by any unprivileged NEAR account. An attacker can invoke it directly, skipping the coinbase-proof guard entirely, and obtain a `true` return value for a forged transaction-inclusion claim.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces two additional invariants over the deprecated v1:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — ensures tree depth parity.
2. The coinbase transaction's Merkle proof must reconstruct the block's `merkle_root` — anchors the tree depth to a known-valid leaf. [1](#0-0) 

These two checks together prevent an attacker from supplying an internal Merkle-tree node as `tx_id`, because the coinbase proof forces the verifier to confirm the tree has the correct depth.

`verify_transaction_inclusion_v2` then delegates to `verify_transaction_inclusion`: [2](#0-1) 

The problem is that `verify_transaction_inclusion` is itself `pub` with only a `#[pause]` gate — no role restriction, no access-control attribute: [3](#0-2) 

Its only guard against a forged proof is:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [4](#0-3) 

There is no coinbase-proof check. Any NEAR account can call `verify_transaction_inclusion` directly, supply an internal Merkle-tree node hash as `tx_id`, and receive `true`.

The contract's own docstring acknowledges this:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [5](#0-4) 

The `#[deprecated]` Rust attribute only emits a compiler warning to the contract's own Rust callers; it does not restrict on-chain invocation. Any NEAR account — including a malicious bridge consumer — can call the method at runtime.

---

### Impact Explanation

Consumer contracts (bridges, atomic-swap protocols, cross-chain lending) that call `verify_transaction_inclusion` — either because they have not migrated to v2, or because an attacker deliberately targets the v1 entry point — will accept a forged proof of transaction inclusion. The attacker can claim that an arbitrary 32-byte value (crafted to be an internal Merkle-tree node) is a confirmed transaction in a real Bitcoin block, causing the consumer contract to release funds or execute privileged logic based on a non-existent Bitcoin transaction.

The corrupted value is the boolean result of `verify_transaction_inclusion`: it returns `true` for a `tx_id` that is not a leaf-level transaction hash, breaking the core SPV guarantee the light client is designed to provide.

---

### Likelihood Explanation

The entry point is fully open to any unprivileged NEAR caller when the contract is not paused. No staking, no role, no deposit beyond storage cost is required. The 64-byte transaction forgery technique is publicly documented (referenced in the contract's own comments: https://www.bitmex.com/blog/64-Byte-Transactions). Any bridge or DeFi protocol consuming this light client that has not explicitly migrated to `verify_transaction_inclusion_v2` is immediately exploitable.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` (make it `pub(crate)` or `fn`) so it can only be called internally by `verify_transaction_inclusion_v2`. The security check (coinbase proof) must be non-bypassable from any external entry point. Alternatively, add a hard `env::panic_str` at the top of `verify_transaction_inclusion` when called externally, or remove the function from the public ABI entirely and inline its logic into `verify_transaction_inclusion_v2`.

---

### Proof of Concept

1. Identify a real Bitcoin block `B` with merkle root `R` and at least two transactions.
2. Compute the hash of an internal Merkle-tree node `N` at depth `d` (e.g., the hash of `tx0 || tx1`). `N` is 32 bytes and is a valid `H256`.
3. Construct a valid Merkle proof `P` that reconstructs `R` starting from `N` at position `p` (treating `N` as if it were a leaf).
4. Call `verify_transaction_inclusion` on-chain:
   ```
   verify_transaction_inclusion({
     tx_id: N,               // internal node, not a real txid
     tx_block_blockhash: B,
     tx_index: p,
     merkle_proof: P,        // valid proof for internal node N
     confirmations: 1
   })
   ```
5. The function computes `compute_root_from_merkle_proof(N, p, P)` which equals `R` (the real merkle root), so the comparison succeeds and the function returns `true`.
6. The coinbase-proof check in `verify_transaction_inclusion_v2` is never reached because the attacker called v1 directly. [6](#0-5) [7](#0-6)
