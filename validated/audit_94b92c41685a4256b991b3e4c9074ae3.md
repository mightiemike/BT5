### Title
User Principal Permanently Locked When Bonus Pool Is Empty — (`x/tieredrewards/keeper/msg_server.go`)

### Summary

Both exit paths from a tiered-rewards position — `MsgTierUndelegate` and `MsgExitTierWithDelegation` — unconditionally call `claimRewards` before executing the exit. If the bonus pool (`rewards_pool` module account) is empty and the position has accrued non-zero bonus rewards, both calls fail with `ErrInsufficientBonusPool`. Because `MsgWithdrawFromTier` requires the position to be undelegated first, a user whose exit commitment has fully elapsed can be permanently blocked from recovering their principal with no user-controlled escape path.

### Finding Description

`MsgTierUndelegate` in `x/tieredrewards/keeper/msg_server.go` calls `claimRewards` before issuing the staking `Undelegate`:

```go
pos, _, _, err = ms.claimRewards(ctx, pos)
if err != nil {
    return nil, err
}
```

`claimRewards` calls `processEventsAndClaimBonus`, which calls `sufficientBonusPoolBalance`. If the pool balance is less than the accrued bonus, it returns `ErrInsufficientBonusPool` and the entire transaction reverts — the position stays delegated.

`MsgExitTierWithDelegation` (the only other exit path) has the identical pattern:

```go
pos, _, _, err = ms.claimRewards(ctx, pos)
if err != nil {
    return nil, err
}
```

`MsgWithdrawFromTier` requires the position to be undelegated (`ErrPositionDelegated` is returned otherwise), so it cannot be used to bypass the stuck state.

The result is a complete deadlock: the user's principal is held in the position's per-position delegator address, the exit commitment has elapsed, but neither exit path can proceed until an external party replenishes the pool.

The ADR explicitly documents this as intentional for user-driven paths ("fail atomically, user retries after pool is replenished"), but provides no time-bounded guarantee that the pool will ever be replenished, and no fallback that lets the user forfeit bonus and recover principal.

By contrast, the `BeforeRedelegationSlashed` hook already handles the empty-pool case by silently forfeiting bonus to avoid a chain halt — the same pattern is not applied to user exit paths.

### Impact Explanation

A user who has locked tokens via `MsgLockTier` or `MsgCommitDelegationToTier`, triggered exit, and waited for the full `ExitDuration` to elapse cannot recover their principal if the `rewards_pool` module account is empty and their position has accrued non-zero bonus. The locked tokens (bond denom, e.g. `basecro`) remain at the position's per-position delegator address indefinitely. There is no user-controlled transaction that bypasses the reward-claim requirement.

### Likelihood Explanation

The bonus pool is funded by an external party (governance proposal or direct transfer). The `BeginBlocker` top-up only transfers from the fee collector when `TargetBaseRewardsRate > 0` and there is a shortfall in base rewards — it does not guarantee the bonus pool stays solvent. As the pool is shared across all positions and all tiers, a period of high claim activity or a governance decision to stop funding the pool can drain it to zero. Any position that accrued even 1 unit of bonus during its lock period will be blocked.

### Recommendation

Allow `MsgTierUndelegate` and `MsgExitTierWithDelegation` to proceed even when the bonus pool is insufficient, forfeiting the unpayable bonus rewards rather than reverting the entire transaction. This mirrors the existing behavior in `BeforeRedelegationSlashed`. The principal recovery path must never be gated on pool solvency.

### Proof of Concept

1. User calls `MsgLockTier` with `amount = MinLockAmount`, `trigger_exit_immediately = true`.
2. Time advances past `ExitDuration` (e.g. 365 days). Bonus accrues.
3. Bonus pool is empty (never funded, or drained by prior claims or BeginBlocker top-ups).
4. User calls `MsgTierUndelegate`:
   - `validateUndelegatePosition` passes (exit triggered, elapsed, delegated, owner match).
   - `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → returns `ErrInsufficientBonusPool`.
   - Transaction reverts. Position remains delegated.
5. User calls `MsgExitTierWithDelegation`:
   - Same `claimRewards` call fails identically.
6. User calls `MsgWithdrawFromTier`:
   - `validateWithdrawFromTier` returns `ErrPositionDelegated` because step 4 never completed.
7. Principal is permanently locked until an external party funds the pool.

```go
// claimRewards claims base and bonus rewards for a single position.
// Returns:
//   - position: the updated position state with reward checkpoints advanced;
//   - base: base rewards paid to the owner for this position in this call;
//   - bonus: bonus rewards paid to the owner for this position in this call;
func (k Keeper) claimRewards(ctx context.Context, pos types.PositionState) (types.PositionState, sdk.Coins, sdk.Coins, error) {
	if !pos.IsDelegated() {
		return pos, sdk.NewCoins(), sdk.NewCoins(), nil
	}

	base, err := k.claimBaseRewards(ctx, pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	return pos, base, bonus, nil
}
```

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```

```go
func (k Keeper) validateUndelegatePosition(ctx context.Context, pos types.PositionState, owner string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.IsDelegated() {
		return types.ErrPositionNotDelegated
	}

	if !pos.HasTriggeredExit() {
		return types.ErrExitNotTriggered
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if !pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
		return types.ErrExitLockDurationNotReached
	}

	// skip check for zero amount as we want those positions to be able to close their position properly

	return nil
}
```
