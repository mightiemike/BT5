### Title
Stale `lastLiquidationFees` Across Subaccounts Corrupts Insurance Accounting During Finalization — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a persistent global storage variable in `ClearinghouseStorage` that is written by every call to `_handleLiquidationPayment` and read by `_finalizeSubaccount`. Because it is never scoped to a specific liquidatee, a liquidation of subaccount A contaminates the finalization of a completely different subaccount B, causing `v.canLiquidateMore` to be computed with the wrong fee value. This leads to either premature socialization of losses onto depositors (when the insurance fund could have covered them) or a blocked finalization.

---

### Finding Description

`ClearinghouseStorage` declares a single shared slot:

```solidity
int128 internal lastLiquidationFees;
``` [1](#0-0) 

Every call to `_handleLiquidationPayment` unconditionally overwrites this slot with the fees from the most recent liquidation step, regardless of which subaccount is being liquidated:

```solidity
lastLiquidationFees = v.liquidationFees;
``` [2](#0-1) 

When `_finalizeSubaccount` is later called for a **different** subaccount, it reads this stale value to compute the effective insurance available for socialization:

```solidity
v.insurance = insurance;
v.insurance -= lastLiquidationFees;          // stale: from a different subaccount
v.canLiquidateMore = (quoteBalance.amount + v.insurance) > 0;
``` [3](#0-2) 

The comment in the code explains the intended purpose: fees from the *last liquidation step of the same subaccount* should be withheld from the insurance pool during socialization to prevent a re-entrancy-like blocking condition. But because `lastLiquidationFees` is global and never reset between subaccounts, it carries the value from whichever subaccount was most recently liquidated — not the one being finalized.

The `v.canLiquidateMore` flag gates a critical safety check:

```solidity
if (v.canLiquidateMore) {
    for (uint32 i = 1; i < v.spotIds.length; ++i) {
        ...
        require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
    }
}
``` [4](#0-3) 

If `v.canLiquidateMore` is incorrectly `false` (because stale `lastLiquidationFees` is too large), this check is skipped entirely, allowing a subaccount with remaining spot liabilities to be socialized even though the insurance fund had sufficient balance to cover them.

---

### Impact Explanation

**Incorrect socialization of losses:** When `lastLiquidationFees` is inflated by a prior large liquidation of a different subaccount, `v.insurance` is underestimated. `v.canLiquidateMore` becomes `false`, the spot-asset-zero check is bypassed, and `spotEngine.socializeSubaccount` is called. This spreads the liquidatee's remaining spot liabilities across all depositors via a reduction in `cumulativeDepositsMultiplierX18`, permanently diluting their balances — even though the insurance fund had enough to cover the shortfall. [5](#0-4) 

**Blocked finalization (secondary):** If `lastLiquidationFees` is stale-low (e.g., 0 from a prior finalization), `v.canLiquidateMore` is incorrectly `true`, the spot-asset check fires, and a legitimate finalization reverts with `ERR_NOT_FINALIZABLE_SUBACCOUNT`.

---

### Likelihood Explanation

Any liquidator can submit `LiquidateSubaccount` transactions. The sequencer processes them in submission order. A liquidator who:
1. Liquidates a large position on subaccount A (setting `lastLiquidationFees` to a large value), then
2. Submits a finalization (`productId == type(uint32).max`) for a different underwater subaccount B

will trigger the contaminated path. No privileged access is required beyond the ability to submit signed liquidation transactions, which is open to any participant. [6](#0-5) 

---

### Recommendation

Scope `lastLiquidationFees` to the specific liquidatee being processed. The simplest fix is to pass it as a local variable through the liquidation call chain rather than storing it in persistent state. Alternatively, key it by liquidatee: `mapping(bytes32 => int128) internal lastLiquidationFees`. This ensures that when `_finalizeSubaccount` reads the value for subaccount B, it only sees fees generated by prior liquidation steps of subaccount B, not of any other subaccount.

---

### Proof of Concept

1. Subaccount A holds a large perp position. Subaccount B is underwater with spot liabilities and zero perp positions.
2. Liquidator calls `liquidateSubaccountImpl` for subaccount A (large position) → `lastLiquidationFees` is set to a large value `F`.
3. Liquidator immediately calls `liquidateSubaccountImpl` for subaccount B with `productId == type(uint32).max` (finalization).
4. Inside `_finalizeSubaccount`: `v.insurance = insurance - F`. If `insurance < F + |quoteBalance|`, then `v.canLiquidateMore = false`.
5. The spot-asset-zero check is skipped. `spotEngine.socializeSubaccount(txn.liquidatee)` is called.
6. All depositors in the spot pool for subaccount B's liability token have their `cumulativeDepositsMultiplierX18` reduced — a permanent loss — even though `insurance` (before subtracting the stale `F`) was sufficient to cover the shortfall. [7](#0-6)
