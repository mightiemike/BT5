### Title
`topUpBaseRewards` drains shared pool without bonus reservation, causing permanent bonus loss on redelegation slash — (`x/tieredrewards/keeper/abci.go`, `x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

The `RewardsPoolName` module account is the single funding source for both base-reward top-ups (BeginBlocker) and bonus reward payouts (user-driven). `topUpBaseRewards` can drain the entire pool balance without reserving anything for pending bonus obligations. When the pool is empty and `BeforeRedelegationSlashed` fires, `slashRedelegationPosition` swallows `ErrInsufficientBonusPool` but still persists the advanced bonus accrual checkpoints via `setPositionWithState`, permanently forfeiting the position owner's accrued bonus for that period with no recovery path.

---

### Finding Description

**Root cause — missing reservation in `topUpBaseRewards`:**

`topUpBaseRewards` computes a shortfall and, when the pool balance is less than the shortfall, drains the entire pool: [1](#0-0) 

There is no check that any amount must be reserved for bonus rewards owed to existing tier positions. The pool can be reduced to zero every block when fees are low relative to `TargetBaseRewardsRate`.

**Root cause — checkpoint advancement before balance check in `processEventsAndClaimBonus`:**

Inside `processEventsAndClaimBonus`, the position's in-memory state is mutated — `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded` are all advanced — before the pool balance check at line 230: [2](#0-1) [3](#0-2) 

When the pool is empty, the function returns `(nil, ErrInsufficientBonusPool)` with `pos` already mutated.

**Root cause — error swallowed and mutated state persisted in `slashRedelegationPosition`:**

`slashRedelegationPosition` deliberately swallows `ErrInsufficientBonusPool` to avoid a chain halt: [4](#0-3) 

After swallowing the error, the function calls `setPositionWithState` with the already-mutated `pos` (checkpoints advanced, bonus not paid): [5](#0-4) 

The advanced checkpoints are persisted to state. The accrual window for the forfeited bonus is now closed — the user cannot retry because `LastBonusAccrual` has already been moved forward past the period.

**Contrast with user-driven paths:**

For all user-driven message handlers (`MsgClaimTierRewards`, `MsgTierUndelegate`, `MsgTierRedelegate`, `MsgAddToTierPosition`, `MsgClearPosition`), `processEventsAndClaimBonus` is called and if the pool is insufficient the entire transaction fails atomically — state is not committed and the user retries later: [6](#0-5) 

The slash hook path has no such rollback. The asymmetry is exact: one path (user-driven) preserves the user's ability to retry; the other path (slash hook) silently advances checkpoints and permanently forfeits the bonus.

---

### Impact Explanation

A tier position owner with an active redelegation permanently loses all accrued bonus rewards for the period between `LastBonusAccrual` and the slash block time. The `LastBonusAccrual` checkpoint is advanced to `blockTime` without payment, closing the accrual window. There is no mechanism to reclaim the forfeited bonus. The corrupted value is the `LastBonusAccrual` field of the persisted `Position` proto and the corresponding unreceived `sdk.Coins` from `RewardsPoolName`.

---

### Likelihood Explanation

Requires two concurrent conditions, both of which are normal protocol events:

1. **Pool drained by BeginBlocker:** Occurs automatically whenever the fee collector balance falls below `TargetBaseRewardsRate × TotalBonded / BlocksPerYear`. In low-fee periods this happens every block.
2. **Validator slash with active tier redelegation:** Double-signing or extended downtime triggers `BeforeRedelegationSlashed`. Any tier position that has called `MsgTierRedelegate` and whose redelegation has not yet matured is exposed.

Neither condition requires attacker action. The combination is realistic in production, particularly during low-activity periods when fee revenue is insufficient to cover the target base rate.

---

### Recommendation

Decouple checkpoint advancement from the payment attempt. In `processEventsAndClaimBonus`, only advance `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded` after `SendCoinsFromModuleToAccount` succeeds. Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, do **not** call `setPositionWithState` with the mutated `pos`; instead restore the original checkpoints so the user can claim the bonus later once the pool is replenished.

---

### Proof of Concept

1. User creates a tier position and calls `MsgTierRedelegate` to validator V; a `RedelegationMapping` entry is created.
2. BeginBlocker runs with low fee-collector balance and high `TargetBaseRewardsRate`; `topUpBaseRewards` drains the entire `RewardsPoolName` balance to zero.
3. Validator V double-signs; the Cosmos SDK staking module calls `BeforeRedelegationSlashed` with the position's `unbondingId`.
4. `slashRedelegationPosition` is entered; `processEventsAndClaimBonus` is called on the position.
5. Inside `processEventsAndClaimBonus`, the loop advances `LastEventSeq` and the final segment advances `LastBonusAccrual` to `blockTime` and updates `LastKnownBonded` — all in-memory on `pos`.
6. `sufficientBonusPoolBalance` finds the pool empty and returns `ErrInsufficientBonusPool`; no coins are transferred.
7. Back in `slashRedelegationPosition`, the error is swallowed; `setPositionWithState(ctx, pos, nil)` persists the mutated `pos` with advanced checkpoints.
8. The user's accrued bonus for the entire period since their last claim is permanently lost. Any subsequent `MsgClaimTierRewards` call will compute zero bonus for the already-checkpointed window. [7](#0-6) [8](#0-7) [9](#0-8)
