### Title
Reverting Extension Permanently Blocks `removeLiquidity` and `swap` With No Recovery Path — (`metric-core/contracts/ExtensionCalling.sol`)

---

### Summary

`ExtensionCalling._callExtensionsInOrder` propagates any revert from a configured extension directly to the caller. Because all seven extension slots are stored as **immutables** set at construction time, there is no admin path to unset a broken extension. If any extension configured on `BEFORE_REMOVE_LIQUIDITY_ORDER`, `AFTER_REMOVE_LIQUIDITY_ORDER`, `BEFORE_SWAP_ORDER`, or `AFTER_SWAP_ORDER` begins reverting consistently (e.g., due to an oracle failure inside `OracleValueStopLossExtension` or `PriceVelocityGuardExtension`), the corresponding pool operation is permanently bricked. LP funds become unwithdrawable with no on-chain recovery mechanism — a strictly worse outcome than the AlgebraVirtualPool analog, where `activeIncentive` can be unset.

---

### Finding Description

`_callExtensionsInOrder` iterates through the packed extension order and calls each configured extension with no error handling:

```solidity
// ExtensionCalling.sol lines 75-86
function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;
    while (true) {
        uint256 extensionIndex = order & 0x7;
        if (extensionIndex == 0) break;
        address extension = _extensionAddress(extensionIndex);
        if (extension == address(0)) revert PanicEmptyExtension();
        CallExtension.callExtension(extension, data);  // ← no try/catch
        order >>= 3;
    }
}
``` [1](#0-0) 

This is called unconditionally from every hook point — `_beforeRemoveLiquidity`, `_afterRemoveLiquidity`, `_beforeSwap`, `_afterSwap`, `_beforeAddLiquidity`, `_afterAddLiquidity`:

```solidity
// MetricOmmPool.sol lines 199-212
function removeLiquidity(...) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) ... {
    ...
    _beforeRemoveLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Removed, amount1Removed) = LiquidityLib.removeLiquidity(...);
    _afterRemoveLiquidity(msg.sender, owner, salt, deltas, amount0Removed, amount1Removed, extensionData);
}
``` [2](#0-1) 

All seven extension addresses are **immutable**, set once at construction and never changeable:

```solidity
// ExtensionCalling.sol lines 17-35
address internal immutable EXTENSION_1;
...
address internal immutable EXTENSION_7;
uint256 internal immutable BEFORE_REMOVE_LIQUIDITY_ORDER;
uint256 internal immutable AFTER_REMOVE_LIQUIDITY_ORDER;
...
``` [3](#0-2) 

The pool's only recovery lever is `setPause`, which sets `pauseLevel` and is checked by the `whenNotPaused` modifier — but `whenNotPaused` is only applied to `swap`, **not** to `removeLiquidity` or `addLiquidity`:

```solidity
// MetricOmmPool.sol line 224
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ...
// MetricOmmPool.sol line 200
function removeLiquidity(...) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) ...
``` [4](#0-3) 

So even pausing the pool does not unblock `removeLiquidity` if its extension hook is reverting.

The periphery ships concrete extensions that depend on external oracles:

```
metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol
metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol
```



Either of these, if configured on `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER`, will permanently block LP withdrawals if the underlying oracle becomes unavailable, returns stale/zero data, or the extension itself has a bug.

---

### Impact Explanation

- **LP funds locked**: If `removeLiquidity` is blocked, LPs cannot withdraw their principal. If the market moves adversely during the outage, they suffer direct, unrecoverable losses.
- **Swap DoS**: If `beforeSwap`/`afterSwap` extensions revert, the pool is entirely unusable for trading.
- **No recovery path**: Unlike AlgebraPool (where `activeIncentive` can be unset by admin), Metric OMM extension addresses are immutable. There is no `setExtension`, no factory override, and no pause mechanism that covers `removeLiquidity`. The pool is permanently bricked for the affected operation.

Impact: **High** — direct loss of LP principal (locked funds), broken core withdraw flow.

---

### Likelihood Explanation

- Extensions like `OracleValueStopLossExtension` and `PriceVelocityGuardExtension` depend on external price feeds. Oracle outages, feed deprecation, or returning zero/stale prices are realistic operational events.
- The trigger requires no privileged action by an attacker — a normal oracle failure or extension bug is sufficient.
- The pool deployer choosing to configure these extensions is the intended use case, not a malicious setup assumption.

Likelihood: **Medium** — requires an oracle or extension failure, which is a realistic operational risk for any pool using these guards.

---

### Recommendation

1. **Short term**: Wrap `CallExtension.callExtension` in a try/catch inside `_callExtensionsInOrder`. On revert, emit an event and continue (or revert with a descriptive error that allows the factory to respond). At minimum, apply this to `_beforeRemoveLiquidity` and `_afterRemoveLiquidity` so LP withdrawals are never permanently blocked by an extension.

2. **Long term**: Make extension orders mutable (factory-controlled with a timelock) so a broken extension can be removed from the order without redeploying the pool. Alternatively, implement a factory-level emergency bypass that can zero out a specific extension slot.

3. **Immediate**: Apply `whenNotPaused` to `addLiquidity` and `removeLiquidity` as well, so that pausing the pool at least prevents further state corruption while the extension issue is diagnosed — though this does not unblock LP withdrawals.

---

### Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension` configured on `BEFORE_REMOVE_LIQUIDITY_ORDER` (index packed into `extensionOrders.beforeRemoveLiquidity`).
2. LPs add liquidity normally.
3. The oracle feed used by `OracleValueStopLossExtension` becomes unavailable (returns stale data, reverts, or returns zero).
4. `OracleValueStopLossExtension.beforeRemoveLiquidity` reverts when it calls the oracle.
5. `_callExtensionsInOrder` propagates the revert (no try/catch at line 83 of `ExtensionCalling.sol`).
6. Every call to `removeLiquidity` reverts.
7. The factory calls `setPause(1)` — this only blocks `swap` (line 224 of `MetricOmmPool.sol`), not `removeLiquidity`.
8. Extension addresses are immutable — no `setExtension` exists.
9. LP funds are permanently locked with no on-chain recovery path.
