### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Forgery Protection — (`contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` (v1) is still a live, unrestricted public NEAR contract method. Any unprivileged caller can invoke it directly, bypassing the coinbase Merkle proof validation that `verify_transaction_inclusion_v2` was specifically introduced to enforce. This is a direct structural analog to the Open Dollar bug: a lower-level function is reachable without going through the higher-level guard that enforces the critical invariant.

### Finding Description

The contract exposes two proof-verification entry points:

- `verify_transaction_inclusion_v2` — the **guarded** path. It first validates a coinbase Merkle proof against the block's `merkle_root`, then delegates to v1. This guard was added to prevent the 64-byte transaction Merkle forgery attack (CVE-2017-12842).
- `verify_transaction_inclusion` (v1) — the **unguarded** path. It performs only a raw `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == merkle_root` comparison, with no check that `tx_id` is a real leaf rather than an internal Merkle tree node. [1](#0-0) 

The `#[deprecated]` Rust attribute generates only a **compiler warning**; it does not remove the function from the compiled WASM binary or restrict on-chain access. The function carries `pub` visibility and the `#[pause]` attribute — meaning it is callable by any NEAR account whenever the contract is not paused. [2](#0-1) 

The v2 guard that is bypassed: [3](#0-2) 

The docstring for v1 explicitly acknowledges the gap: *"We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification."* — but no on-chain mechanism enforces that callers use the higher level. [4](#0-3) 

The underlying Merkle computation in `merkle-tools` is a pure hash traversal with no leaf/node distinction: [5](#0-4) 

### Impact Explanation

An attacker who knows the internal structure of a real Bitcoin block's Merkle tree can compute an internal node value `N = SHA256d(left_child || right_child)`. Because `N` is 32 bytes (same as a real `txid`), the attacker can supply it as `tx_id` to `verify_transaction_inclusion` with a crafted `merkle_proof` and `tx_index` such that `compute_root_from_merkle_proof(N, index, proof) == real_merkle_root`. The function returns `true`, falsely asserting that a transaction with ID `N` is confirmed in that block.

Any bridge, dApp, or cross-chain protocol that calls this contract and trusts the boolean result to authorize an action (e.g., minting wrapped BTC, releasing escrowed funds) can be drained by an attacker who fabricates a Bitcoin "payment" that never occurred. [6](#0-5) 

### Likelihood Explanation

- The entry path requires no privileged role, no leaked key, and no social engineering — any NEAR account can call `verify_transaction_inclusion` directly.
- The 64-byte attack is well-documented and has known tooling. An attacker only needs a real mainchain block (already stored in `headers_pool`) and the ability to compute its Merkle tree internals from public Bitcoin data.
- The contract's own documentation acknowledges the risk and points to v2 as the fix, but leaves v1 reachable.

### Recommendation

Remove `verify_transaction_inclusion` from the public NEAR ABI entirely, or gate it with an access-control role that prevents unprivileged callers from invoking it. The simplest fix is to change `pub fn verify_transaction_inclusion` to `fn verify_transaction_inclusion` (private), since `verify_transaction_inclusion_v2` already calls it internally via `self.verify_transaction_inclusion(args.into())`. [7](#0-6) 

### Proof of Concept

```
// Attacker calls the NEAR contract directly, bypassing v2:
//
// 1. Pick any real block B already in the mainchain (tx_block_blockhash = B.hash).
// 2. From public Bitcoin data, obtain the full tx list of B and compute its Merkle tree.
// 3. Choose any internal node N at depth d (N = SHA256d(left || right)).
// 4. Construct a merkle_proof of length d-1 that walks from N up to merkle_root.
//    This proof is valid because N IS a node in the real tree.
// 5. Set tx_index to the position that corresponds to N's subtree.
//
// Call (via near-cli or any NEAR SDK client):
//   verify_transaction_inclusion({
//     tx_id:             N,          // internal node, not a real txid
//     tx_block_blockhash: B.hash,
//     tx_index:          <computed>,
//     merkle_proof:      <proof from N to root>,
//     confirmations:     1,
//   })
//
// Result: returns `true` — the contract asserts that transaction N
// is confirmed in block B, even though no such transaction exists.
//
// A bridge contract that calls verify_transaction_inclusion (v1) and
// mints/releases funds on `true` is now exploitable for any amount
// the attacker can construct a plausible-looking fake payment for.

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
