### Title
Fork Difficulty Validated Against Stale Mainchain State Instead of Fork's Own Ancestor - (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`)

---

### Summary

When validating the difficulty of fork-chain blocks, the contract always fetches the boundary block via `get_header_by_height`, which reads exclusively from `mainchain_height_to_header`. This map is never updated during fork submission; it only reflects the current canonical chain. As a result, the difficulty check for every fork block beyond the first is performed against the **mainchain's** historical block at the interval boundary, not the fork's own ancestor at that height. This is a direct cross-module desynchronization: a snapshot of mainchain state is used as a fixed reference while the fork's own chain history diverges from it.

---

### Finding Description

In `get_next_work_required` (Dogecoin), the difficulty boundary block is retrieved as:

```rust

    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

The same pattern appears in the Bitcoin build:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [2](#0-1) 

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [3](#0-2) 

`mainchain_height_to_header` is only written by `store_block_header` (mainchain path) and `reorg_chain`. Fork blocks are stored only in `headers_pool` via `store_fork_header`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [4](#0-3) 

Fork blocks are **never** inserted into `mainchain_height_to_header` before a reorg is confirmed. Therefore, `get_header_by_height` always returns the mainchain block at the requested height, even when called during fork-block difficulty validation.

**Concrete desynchronization for Dogecoin (post-block 145,000, per-block retarget):**

After block 145,000, `difficulty_adjustment_interval = 1`, so every block triggers a retarget using the block one step back:

```rust
let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
let difficulty_adjustment_interval = if new_difficulty_protocol { 1 } else { ... };
``` [5](#0-4) 

Suppose the fork diverges at mainchain height `H`. The fork block at `H` has attacker-controlled timestamp `T_H_fork`. When the contract validates the fork block at height `H+1`:

- `prev_block_header` = fork block at `H` (timestamp `T_H_fork`)
- `height_first = H - 1`
- `get_header_by_height(H-1)` → **mainchain** block at `H-1` (timestamp `T_{H-1}_main`)
- **Correct** `first_block_time` = fork block at `H-1`'s timestamp = `T_{H-1}_fork`

The Digishield calculation:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
``` [6](#0-5) 

uses `T_{H-1}_main` instead of `T_{H-1}_fork`. If the attacker sets `T_{H-1}_fork > T_{H-1}_main` (up to `MAX_FUTURE_BLOCK_TIME_LOCAL` = 2 hours ahead), the actual `modulated_timespan` computed by the contract is **larger** than the correct value, yielding an **easier target** for the fork block at `H+1`.

The timestamp constraints that bound the manipulation:

```rust
require!(
    block_header.time > get_median_time_past(prev_block_header.clone(), self),
    "time-too-old: block's timestamp is too early"
);
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
    "time-too-new: block timestamp too far in the future"
);
``` [7](#0-6)

---

### Impact Explanation

Fork blocks are accepted with a difficulty target computed from the **mainchain's** boundary block timestamp rather than the fork's own ancestor's timestamp. For Dogecoin (per-block retarget), this desynchronization applies to every fork block from the second one onward. An attacker who sets the fork's divergence-point block timestamp up to 2 hours ahead of the mainchain block at the same height causes the contract to compute an easier `bits` target for subsequent fork blocks. The contract then requires the attacker to provide a `bits` field matching this easier target (`expected_bits == block_header.bits`), and the PoW check passes against the easier target. If the fork accumulates sufficient chain work (even at the reduced per-block work implied by the easier `bits`), `reorg_chain` promotes it to the mainchain. Once promoted, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` return `true` for transactions in the incorrectly-difficulty-validated fork, enabling false SPV proofs to downstream NEAR contracts. [8](#0-7) 

---

### Likelihood Explanation

For Dogecoin (the most affected build), the attacker only needs to create a fork diverging one block back from the current tip — a standard fork submission scenario. No privileged role is required; `submit_blocks` is callable by any trusted relayer, and the `trusted_relayer` macro's bypass roles are not needed for this path. The timestamp manipulation window (up to 2 hours) is bounded by `MAX_FUTURE_BLOCK_TIME_LOCAL` and the Digishield clamping (`max_timespan = retarget_timespan * 1.5`), limiting the target relaxation to roughly 1.5×. This reduces the hash rate needed to outpace the mainchain from 51% to approximately 34%, a meaningful reduction for a live chain.

---

### Recommendation

Replace `get_header_by_height` in `get_next_work_required` with a traversal that walks backward through `headers_pool` via `prev_block_hash` links from the fork's `prev_block_header` until reaching the required height. This ensures the boundary block used for difficulty calculation is always the fork's own ancestor, not the mainchain's block at that height. The `BlocksGetter` trait should expose a `get_ancestor_at_height` method that performs this traversal. [9](#0-8) 

---

### Proof of Concept

1. Mainchain has blocks at heights `H-2`, `H-1`, `H` with timestamps `T_{H-2}`, `T_{H-1}`, `T_H` (all normal, ~60 s apart for Dogecoin).
2. Attacker submits a fork block at height `H-1` with timestamp `T_{H-1}_fork = T_{H-1} + 7200` (2 hours ahead, maximum allowed by `MAX_FUTURE_BLOCK_TIME_LOCAL`). This passes MTP and future-time checks.
3. Attacker submits a fork block at height `H` with timestamp `T_H_fork = T_{H-1}_fork + 45` (45 s later, within MTP bounds).
4. Attacker submits a fork block at height `H+1`. The contract calls `get_next_work_required`:
   - `height_first = H - 1`
   - `get_header_by_height(H-1)` → mainchain block, timestamp `T_{H-1}` (not `T_{H-1}_fork`)
   - `modulated_timespan` (actual) = `T_H_fork - T_{H-1}` = `(T_{H-1} + 7200 + 45) - T_{H-1}` = 7245 s → clamped to `max_timespan = 90 s`
   - `modulated_timespan` (correct) = `T_H_fork - T_{H-1}_fork` = 45 s → clamped to `min_timespan = 45 s`
   - Actual target = `prev_bits_target × 90 / 60` (1.5× easier)
   - Correct target = `prev_bits_target × 45 / 60` (0.75× harder)
5. The contract accepts the fork block at `H+1` with `bits` encoding the 1.5× easier target. The attacker mines this block with ~2× less work than the correct difficulty requires.
6. Repeating for subsequent fork blocks, the attacker builds a fork with systematically under-validated difficulty. If chain work surpasses the mainchain tip, `reorg_chain` promotes the fork, and `verify_transaction_inclusion` begins returning `true` for transactions in the attacker-controlled chain. fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
```
