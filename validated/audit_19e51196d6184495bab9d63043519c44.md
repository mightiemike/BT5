### Title
Native ETH Permanently Locked in `DirectDepositV1` — Recovery Path Broken Because `ContractOwner` Cannot Receive ETH - (File: `core/contracts/ContractOwner.sol`, `core/contracts/DirectDepositV1.sol`)

---

### Summary

`ContractOwner` has no `receive()` function, yet `DirectDepositV1.withdrawNative()` unconditionally sends ETH to `msg.sender` (which is `ContractOwner`). This makes the entire native-ETH recovery path permanently revert. Any native ETH that ends up in a `DirectDepositV1` instance — whether from a constructor-time wrapping failure on a pre-funded CREATE2 address, or from a force-send via `selfdestruct` — is locked forever with no on-chain escape.

---

### Finding Description

`DirectDepositV1` is deployed deterministically via CREATE2 with a fixed salt of `bytes32(uint256(1))` inside `ContractOwner.createDirectDepositV1`. Its constructor attempts to wrap any pre-existing ETH balance:

```solidity
uint256 balance = address(this).balance;
if (balance != 0) {
    (bool success, ) = wrappedNative.call{value: balance}("");
    if (!success) {
        emit NativeTokenTransferFailed(balance);   // does NOT revert
    }
}
```

If the wrap call fails, the ETH is silently left in the contract. Additionally, any actor can force-send ETH to `DirectDepositV1` via `selfdestruct`, bypassing the `receive()` hook entirely.

The intended recovery path is `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`:

```solidity
if (token == address(0)) {
    uint256 preBalance = address(this).balance;
    DirectDepositV1(directDepositV1).withdrawNative();   // ← always reverts
    uint256 postBalance = address(this).balance;
    require(postBalance > preBalance, "empty");
    (bool success, ) = msg.sender.call{value: postBalance - preBalance}("");
    require(success, "xfer");
}
```

`withdrawNative()` in `DirectDepositV1` is:

```solidity
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
```

`msg.sender` here is `ContractOwner`. A grep across all production Solidity files confirms `ContractOwner` declares no `receive()` or `fallback()` function and inherits only from `EIP712Upgradeable` and `OwnableUpgradeable`, neither of which defines one. The low-level ETH transfer therefore fails unconditionally, causing `withdrawNative()` to revert with `"Failed to transfer native token to owner"`, which propagates and reverts `withdrawFromDirectDepositV1` entirely.

---

### Impact Explanation

Any native ETH that reaches a `DirectDepositV1` instance — through constructor-time wrapping failure on a pre-funded CREATE2 address, or through a `selfdestruct` force-send — is permanently locked. The owner-only recovery function `withdrawFromDirectDepositV1(..., address(0))` is permanently non-functional. There is no alternative on-chain path to recover native ETH from `DirectDepositV1`.

---

### Likelihood Explanation

Two concrete triggers exist:

1. **Pre-funded CREATE2 address + wrapping failure**: Because the salt is fixed (`bytes32(uint256(1))`), the `DirectDepositV1` address for any subaccount is deterministic and computable before deployment. An actor can send ETH to that address before `createDirectDepositV1` is called. If `wrappedNative.call` fails at constructor time (e.g., during a chain incident or if `wrappedNative` is temporarily non-functional), the ETH is silently retained and permanently locked.

2. **`selfdestruct` force-send**: Any unprivileged caller can deploy a contract, fund it with ETH, and `selfdestruct` it targeting a `DirectDepositV1` address. This bypasses `receive()` and deposits ETH directly into the contract balance. The ETH is then irrecoverable.

Both paths are reachable by an unprivileged external actor with no special permissions.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

Alternatively, refactor `withdrawNative()` to accept an explicit `address payable recipient` parameter and pass `payable(msg.sender)` from the `ContractOwner` call site, forwarding ETH directly to the final recipient rather than staging it through `ContractOwner`.

---

### Proof of Concept

1. Compute the deterministic address of `DirectDepositV1` for a target subaccount using the fixed salt `bytes32(uint256(1))`.
2. `selfdestruct` a funded contract to that address, depositing ETH directly into its balance (bypassing `receive()`).
3. Call `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` as the owner.
4. Execution reaches `DirectDepositV1.withdrawNative()`, which calls `ContractOwner.call{value: balance}("")`.
5. `ContractOwner` has no `receive()` function → the call returns `success = false`.
6. `withdrawNative()` reverts: `"Failed to transfer native token to owner"`.
7. The entire recovery transaction reverts. ETH remains locked in `DirectDepositV1` permanently.
