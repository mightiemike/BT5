### Title
`SwapAllowlistExtension` gates on the immediate pool caller (`sender = router`) rather than the end-user, allowing any unprivileged user to bypass per-user swap restrictions on curated pools — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `swap()` call. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the **router address**, not the end-user. If a pool admin allowlists the router (a natural step to enable router-mediated swaps for their allowlisted users), every user — including non-allowlisted ones — can bypass the per-user gate by routing through the router.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap`, the pool calls:

```solidity
_beforeSwap(msg.sender, recipient, zeroForOne, ...);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` then forwards that value verbatim as the `sender` argument to every configured extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
``` [2](#0-1) 

**Step 2 — Router is the immediate caller of `pool.swap`.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly. The pool therefore sees `msg.sender = address(router)`, not the end-user's address. [3](#0-2) 

**Step 3 — Extension checks the wrong actor.**

`SwapAllowlistExtension.beforeSwap` evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [4](#0-3) 

Here `msg.sender` is the pool (correct) and `sender` is the router address (wrong — it should be the end-user). The lookup becomes `allowedSwapper[pool][router]`.

**Step 4 — The bypass.**

A pool admin who wants to allow specific KYC'd users to swap via the router will:
1. Allowlist individual users: `setAllowedToSwap(pool, userA, true)`.
2. Also allowlist the router so those users can use it: `setAllowedToSwap(pool, router, true)`.

Once the router is allowlisted, **any** address can call `router.exactInputSingle(pool, ...)` and the extension will pass — because it only checks `allowedSwapper[pool][router]`, which is `true`. The actual end-user identity is never inspected.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., KYC'd institutions, whitelisted market makers) loses that restriction entirely for any user who routes through the public `MetricOmmSimpleRouter`. Non-allowlisted users can drain liquidity, front-run allowlisted traders, or violate the pool's compliance model. This breaks the core curation invariant the extension was deployed to enforce and constitutes an admin-boundary break reachable by an unprivileged path.

---

### Likelihood Explanation

The trigger requires the pool admin to allowlist the router — a natural and expected action for any pool that wants to support router-mediated swaps for its allowlisted users. The router is a first-party periphery contract. Any pool that combines `SwapAllowlistExtension` with router support is vulnerable. The attacker needs no privileged access; a standard `exactInputSingle` call suffices.

---

### Recommendation

The extension must gate on the **end-user identity**, not the immediate caller. Two options:

1. **Pass the original `msg.sender` through the router.** The router could include the originating user in `extensionData`, and the extension could decode and verify it. However, this is forgeable unless the pool enforces the encoding.

2. **Check `sender` only for direct pool calls; for router calls, require the router to attest the real user.** A cleaner approach is to have the router pass the real user as `recipient` and have the extension check `recipient` instead of `sender` — but this requires a protocol-level convention.

3. **Preferred fix:** Redesign `SwapAllowlistExtension.beforeSwap` to check `recipient` (the economic beneficiary) rather than `sender` (the technical caller), or introduce a trusted-forwarder pattern where the router signs the real user identity into `extensionData` and the extension verifies it.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, alice, true)       // alice is KYC'd
  pool admin calls setAllowedToSwap(pool, router, true)      // enable router for alice

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ..., recipient: bob})

Result:
  pool.swap(msg.sender=router, recipient=bob, ...)
  _beforeSwap(sender=router, ...)
  extension checks allowedSwapper[pool][router] == true  → passes
  bob receives output tokens despite never being allowlisted
```

The `DepositAllowlistExtension` does **not** share this flaw — it gates on `owner` (the LP position recipient), which is correctly preserved through the `MetricOmmPoolLiquidityAdder` path. [5](#0-4) [6](#0-5)
