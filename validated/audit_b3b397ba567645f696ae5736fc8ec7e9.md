### Title
`ENABLE_SECP_OPS` Flag Guard Bypassed via 4-Byte Opcode Path, Allowing Ungated Secp Verification — (`File: src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` contains two separate dispatch paths for secp signature verification. The 1-byte opcode path (opcodes 64, 65) is correctly gated behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (`0x13d61f00`, `0x1c3a8f00`) dispatches the identical `op_secp256k1_verify` / `op_secp256r1_verify` functions **without any flag check**. An attacker-controlled CLVM program can invoke secp verification via the 4-byte path regardless of whether `ENABLE_SECP_OPS` is set, bypassing the intended soft-fork gate and producing consensus divergence.

---

### Finding Description

In `src/chia_dialect.rs`, `ChiaDialect::op()` handles operator dispatch in two sequential branches:

**Branch 1 — 4-byte opcodes (lines 157–183):**

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // ← no ENABLE_SECP_OPS check
        0x1c3a8f00 => op_secp256r1_verify,   // ← no ENABLE_SECP_OPS check
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);  // early return
}
```

**Branch 2 — 1-byte opcodes (lines 248–249):**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The 4-byte branch returns early before the 1-byte branch is ever reached. The `ENABLE_SECP_OPS` guard on opcodes 64/65 is therefore irrelevant for any caller using the 4-byte encoding. The same `op_secp256k1_verify` and `op_secp256r1_verify` functions are invoked unconditionally via the 4-byte path.

This is structurally identical to the reported bug: the first check (4-byte path) returns early without applying the flag guard that the second path (1-byte path) correctly enforces. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

**Consensus divergence / flag bypass.** The `ENABLE_SECP_OPS` flag is documented as the gate for secp opcodes:

> "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)." [3](#0-2) 

However, the flag does not gate the 4-byte opcode path. Before `ENABLE_SECP_OPS` activates:

- Opcode `0x40` (64) → `unknown_operator` → returns nil (consensus) or hard error (mempool with `NO_UNKNOWN_OPS`)
- Opcode `0x13d61f00` → `op_secp256k1_verify` → **always executes**, can raise `EvalErr::Secp256Failed`

A CLVM program embedding `0x13d61f00` with an invalid secp256k1 signature will **fail** on nodes running this code (regardless of `ENABLE_SECP_OPS`), but **succeed** (return nil) on older nodes that treat the same 4-byte opcode as an unknown operator. This is a direct consensus split.

Additionally, `MEMPOOL_MODE` sets `NO_UNKNOWN_OPS` but does **not** set `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

In mempool mode, opcode 64 is rejected as unimplemented, but `0x13d61f00` still executes secp verification. This creates an asymmetric enforcement surface: the same cryptographic check is reachable via one encoding but not the other, undermining the mempool's stricter validation intent.

---

### Likelihood Explanation

The 4-byte secp opcode values (`0x13d61f00`, `0x1c3a8f00`) are valid CLVM atom bytes that any attacker can embed directly in a serialized CLVM program. No special privilege is required. The attacker only needs to serialize a CLVM expression whose operator atom is the 4-byte sequence `\x13\xd6\x1f\x00` and supply any three atom arguments (pubkey, message, signature). If the signature is invalid, the program raises `EvalErr::Secp256Failed` on nodes with this code, while older nodes return nil — a reachable, externally-triggered consensus split. [5](#0-4) 

---

### Recommendation

Add an `ENABLE_SECP_OPS` flag check inside the 4-byte opcode branch, mirroring the guard already present on the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both encoding paths are governed by the same flag, eliminating the bypass. [6](#0-5) 

---

### Proof of Concept

1. Serialize a CLVM program whose operator atom is the 4-byte sequence `\x13\xd6\x1f\x00` (opcode `0x13d61f00`) with three atom arguments: any 33-byte pubkey, a 32-byte message, and a deliberately invalid 64-byte signature.
2. Run the program via `run_program` with a `ChiaDialect` constructed **without** `ENABLE_SECP_OPS` (e.g., `ChiaDialect::new(ClvmFlags::empty())`).
3. Observe: `op_secp256k1_verify` executes and returns `EvalErr::Secp256Failed` — secp verification ran despite the flag being absent.
4. Run the same program with opcode `0x40` (1-byte encoding of opcode 64) under the same flags.
5. Observe: the 1-byte path falls through to `unknown_operator`, returning nil — no secp verification occurs.

The two encodings of the same logical operator produce different results under identical flags, confirming the flag guard is applied inconsistently across the two dispatch paths. [7](#0-6) [8](#0-7)

```rust
pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
