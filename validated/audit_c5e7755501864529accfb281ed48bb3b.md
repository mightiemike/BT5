Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and evaluates only `owner`, a free parameter supplied by the caller. Because any caller can set `owner` to any allowlisted address, the allowlist never fires against the actual depositing party. The unauthorized depositor's tokens are permanently locked under `owner`'s position with no recovery path.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` discards the first argument entirely (unnamed `address`) and checks only `owner`: [2](#0-1) 

Since `owner` is a free parameter chosen by the caller, any address can pass the guard by setting `owner` to any address that the pool admin has previously allowlisted. The actual depositor (`sender`) is never checked. By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender`: [3](#0-2) 

After the guard passes, `LiquidityLib.addLiquidity` executes, bin balances and `binTotals` are updated, and LP shares are credited to `owner`. The unauthorized depositor's tokens are pulled via the modify-liquidity callback. Recovery is impossible because `removeLiquidity` enforces `msg.sender == owner`: [4](#0-3) 

## Impact Explanation
Two concrete impacts result:
1. **Allowlist bypass**: any pool configured as restricted (KYC-gated, institutional-only, etc.) accepts deposits from any address — the core access-control invariant of `DepositAllowlistExtension` is broken.
2. **Permanent principal loss**: the unauthorized depositor's tokens are locked in the pool under `owner`'s position. The depositor has no LP shares and cannot call `removeLiquidity` (`NotPositionOwner` reverts). This constitutes direct, irrecoverable loss of user principal, meeting Critical/High severity thresholds.

## Likelihood Explanation
Exploitability requires only knowing one allowlisted address, which is observable on-chain via `AllowedToDepositSet` events or direct `allowedDepositor` mapping reads. No special role, flash loan, or oracle manipulation is needed — a plain `addLiquidity` call with `owner` set to any allowlisted address suffices. Every pool deploying `DepositAllowlistExtension` with a non-empty allowlist is affected.

## Recommendation
Check `sender` (the first parameter) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  bob is NOT allowlisted

Attack:
  bob calls pool.addLiquidity(
      owner  = alice,   // allowlisted — guard passes
      salt   = 0,
      deltas = { binIdxs: [0], shares: [1e18] },
      callbackData = "",
      extensionData = ""
  )

  Inside beforeAddLiquidity:
    sender (bob) is silently discarded (unnamed parameter)
    owner  (alice) passes allowedDepositor[pool][alice] == true

  LiquidityLib.addLiquidity executes:
    - bin balances updated
    - positionBinShares[keccak(alice, 0, 0)] += 1e18
    - callback fires, pulling bob's tokens into the pool

Result:
  - bob's tokens are in the pool
  - alice holds the LP position
  - bob cannot call removeLiquidity (NotPositionOwner revert at msg.sender != owner)
  - pool's deposit restriction is silently bypassed
```
