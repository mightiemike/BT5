### Title
Stop-Loss Guard Uses Arithmetic Mid Instead of Protocol's Canonical Geometric Mid, Allowing Value Extraction Beyond Configured Drawdown — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the **arithmetic mean** `(bid + ask) / 2`, while every other component in the protocol uses the **geometric mean** `sqrt(bid * ask)` (via `SwapMath.midAndSpreadFeeX64FromBidAsk`). By the AM-GM inequality, arithmetic mean ≥ geometric mean for any bid < ask. This systematic discrepancy miscalibrates the per-bin value metrics and their watermarks, allowing value extraction beyond the LP-configured drawdown when the oracle spread changes between swaps.

---

### Finding Description

In `_afterSwapOracleStopLoss`, the mid price used to value LP holdings is:

```solidity
// OracleValueStopLossExtension.sol line 218
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

The protocol's canonical mid price, used by the pool for every swap, is the geometric mean:

```solidity
// SwapMath.sol line 70
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
``` [2](#0-1) 

The same geometric mid is used in `PriceVelocityGuardExtension.beforeSwap` and in `MetricOmmPool.swap` itself: [3](#0-2) [4](#0-3) 

The stop-loss value metrics are:

```
metricToken0 = t0/shares + t1 / (mid * shares)
metricToken1 = t0 * mid / shares + t1/shares
``` [5](#0-4) 

Because arithmetic mid ≥ geometric mid, the stop-loss always uses a **higher** mid than the pool does. This means:
- `metricToken0` is **underestimated** (divides by a larger mid)
- `metricToken1` is **overestimated** (multiplies by a larger mid)

The watermark is set from the first call's metric value. If the oracle spread is wide when the watermark is set, the arithmetic mid is materially higher than the geometric mid, so the `metricToken0` watermark is set **lower** than it should be. When the spread later narrows (arithmetic mid converges toward geometric mid), `metricToken0` rises — masking genuine value extraction that the geometric-mid watermark would have caught.

---

### Impact Explanation

**Concrete scenario (false negative — stop-loss fails to protect LPs):**

| Parameter | Value |
|---|---|
| Initial balances | t0 = 1000, t1 = 1000, shares = 1000 |
| Initial spread | 50 % (bid = 0.75, ask = 1.25) |
| Arithmetic mid | 1.000 |
| Geometric mid | √(0.75 × 1.25) ≈ 0.968 |
| Configured drawdown | 10 % |

**Watermark set (first swap):**
- Arithmetic: `metricToken0 = 1 + 1/1.0 = 2.000` → watermark = 2.000, threshold = 1.800
- Geometric: `metricToken0 = 1 + 1/0.968 = 2.033` → watermark = 2.033, threshold = 1.830

**After 10 % value extraction (t0 = 900, t1 = 900) and spread narrows to 2 %:**
- Arithmetic mid ≈ 1.000 → `metricToken0 = 0.9 + 0.9 = 1.800` → exactly at threshold → **no revert**
- Geometric mid ≈ 0.9999 → `metricToken0 ≈ 1.800` vs threshold 1.830 → **revert triggered**

The stop-loss configured to block 10 % drawdown silently allows the full 10 % extraction because the arithmetic watermark was set 1.65 % lower than the geometric watermark. The attacker's extraction is indistinguishable from a benign spread-narrowing event.

The `_applyWatermark` and `_checkAndUpdateWatermarks` logic is otherwise correct; the sole root cause is the wrong mid formula on line 218. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

- Every pool that opts into `OracleValueStopLossExtension` is affected.
- Oracle spreads are not constant; they widen during volatility and narrow during calm periods. The discrepancy is therefore a normal operating condition, not an edge case.
- No privileged action is required: any ordinary swap triggers `afterSwap`, which calls `_afterSwapOracleStopLoss`.
- The magnitude of the miscalibration scales with the spread: for a 50 % spread the AM-GM gap is ~3.3 %; for a 20 % spread it is ~0.5 %. Pools using `PriceProvider` / `PriceProviderL2` (which allow spreads up to `ORACLE_BPS - 1 = 9999 bps`) can exhibit large discrepancies. [8](#0-7) 

---

### Recommendation

Replace the arithmetic mean on line 218 with the protocol's canonical geometric mid:

```solidity
// Before (wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64), uint256(askPriceX64)
);
```

This is identical to what `MetricOmmPool.swap` and `PriceVelocityGuardExtension.beforeSwap` already do, and ensures the stop-loss measures LP value at the same price the pool uses for swap settlement.

---

### Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension`, drawdown = 10 % (`drawdownE6 = 100_000`).
2. Set oracle: bid = 0.75 Q64, ask = 1.25 Q64 (50 % spread). Arithmetic mid = 1.0 Q64, geometric mid ≈ 0.968 Q64.
3. Execute a swap (any direction). The watermark for `metricToken0` is set to `2.000` (arithmetic) instead of `2.033` (geometric). Threshold = `1.800` instead of `1.830`.
4. Reduce bin balances by 10 % (simulate value extraction via a `zeroForOne = true` swap).
5. Narrow oracle spread to 2 % (bid = 0.99, ask = 1.01). Arithmetic mid ≈ 1.000, geometric mid ≈ 0.9999.
6. Execute a `zeroForOne = true` swap. `metricToken0 = 0.9 + 0.9 = 1.800` — exactly at the arithmetic threshold — **no revert**. With the geometric watermark (threshold = 1.830), this same state would have reverted with `OracleStopLossTriggered`.

The stop-loss configured to protect against 10 % drawdown silently passes a 10 % extraction event. [9](#0-8) [2](#0-1)
