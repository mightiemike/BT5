I have traced the full validation path in `set_state_header` and the downstream `set_state_finalize` chain. Here is my analysis.

---

## Tracing the Validation Path

### `set_state_header` checks (in order)

**Check 1-2: `validate_chunk_proofs`**

`validate_chunk_proofs` verifies the chunk's internal hash consistency and that transactions/receipts match the chunk's own roots. It does **not** compare `chunk.shard_id()` against the caller-supplied `shard_id` parameter. [1](#0-0) 

**Check 3a: `verify_path` against `chunk_headers_root`**

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { ... }
```

This verifies that `ChunkHashHeight(chunk.hash, chunk.height_included)` is a leaf in the block's Merkle chunk tree. The Merkle path encodes the leaf's *position* implicitly through sibling hashes, but **the code never checks which position the path corresponds to**. A chunk from shard A at position `index(A)` with its valid proof for `index(A)` will pass this check even when the request says `shard_id = B`. [2](#0-1) 

**Check 3b: `verify_path` for `prev_chunk_header`**

Same structure — verifies the prev chunk is in the prev block's chunk tree, again without checking that the position corresponds to the requested shard. [3](#0-2) 

**Check 4: Receipt proofs**

The receipt hash is computed as `CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` where `shard_id` is the **request's** shard_id (B), not the chunk's `shard_id` (A). The receipt proofs are verified against the block's `prev_chunk_outgoing_receipts_root`, which is independent of the chunk's shard_id. A malicious peer can supply valid incoming receipt proofs for shard B while embedding shard A's chunk. [4](#0-3) 

**Check 5: `validate_state_root_node`**

```rust
let chunk_inner = chunk.take_header().take_inner();
if matches!(
    self.runtime_adapter.validate_state_root_node(
        shard_state_header.state_root_node(),
        chunk_inner.prev_state_root(),
    ), ...
```

This checks `hash(state_root_node.data) == chunk_inner.prev_state_root()`. `chunk_inner` is shard A's chunk, so `prev_state_root()` is shard A's state root. The attacker supplies a `state_root_node` matching shard A's state root. No `shard_id` is involved. [5](#0-4) [6](#0-5) 

**Storage**

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```

The key uses the **request's** `shard_id` (B), but the stored value contains shard A's chunk and shard A's `prev_state_root`. [7](#0-6)

## The Missing Check

There is **no** explicit assertion anywhere in `set_state_header` that `chunk.shard_id() == shard_id`. The function signature accepts both independently and never cross-validates them. [8](#0-7)

## Downstream Impact in `set_state_finalize`

`set_state_finalize(shard_id=B, sync_hash)` reads the stored header via `get_state_header(B, sync_hash)`, which returns the poisoned header (shard A's chunk). It then calls `chain_update.set_state_finalize(B, sync_hash, shard_state_header)`:

```rust
let shard_uid = shard_id_to_uid(epoch_manager, shard_id=B, epoch_id)?;  // B's uid
let apply_result = self.runtime_adapter.apply_chunk(
    RuntimeStorageConfig::new(chunk_header.prev_state_root(), true),  // shard A's root
    ...
    ApplyChunkShardContext { shard_uid, ... },  // shard B's uid
    ...
)?;
```

Shard A's `prev_state_root` does not exist in shard B's trie storage, so `apply_chunk` fails with a storage error. The node is stuck — the poisoned header persists in `DBCol::StateHeaders` under `StateHeaderKey(B, sync_hash)`, and every retry of `set_state_finalize` will fail until the node explicitly clears all downloaded state. ## Reachability

The `StateHeaderValidationRequest` is sent by `StateSyncDownloader::ensure_shard_header` after downloading a header from a peer. Any peer in the network (unprivileged) can serve a crafted `ShardStateSyncResponseHeader`. No validator or node-admin privilege is required. [11](#0-10) [12](#0-11)

## Verdict

### Title
Missing `chunk.shard_id()` vs. request `shard_id` cross-check in `set_state_header` allows cross-shard state header injection — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a `ShardStateSyncResponseHeader` whose embedded chunk's `shard_id` field is never compared against the caller-supplied `shard_id`. A malicious peer can submit a header containing a real chunk from shard A (with its valid Merkle proof for shard A's position) in response to a state-sync request for shard B. All five validation steps pass, and the header is committed to `DBCol::StateHeaders` under `StateHeaderKey(B, sync_hash)` carrying shard A's `prev_state_root`. Subsequent `set_state_finalize(B, sync_hash)` attempts to apply shard A's state root against shard B's `ShardUId`, fails with a storage error, and leaves the node permanently stuck in state sync for shard B until the entire state-sync cache is manually cleared.

### Finding Description
The exact missing guard is:

```rust
// After validate_chunk_proofs, before verify_path:
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match request shard_id".into()
    ));
}
```

Without it, the `verify_path` check (step 3a) only proves the chunk hash appears *somewhere* in the block's Merkle chunk tree — it does not prove the chunk occupies the position corresponding to `shard_id`. The receipt-proof check (step 4) uses the request's `shard_id` for `ReceiptList(shard_id, receipts)`, so a malicious peer can supply valid receipt proofs for shard B independently. The `validate_state_root_node` check (step 5) uses `chunk_inner.prev_state_root()` from the submitted chunk (shard A), so it passes with shard A's state root node.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
