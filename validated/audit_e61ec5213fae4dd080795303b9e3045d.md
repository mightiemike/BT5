Audit Report

## Title
DepositAllowlistExtension Checks Position Owner Instead of Actual Depositor, Allowing Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual caller who pays tokens) and validates only `owner` (the LP position recipient) against the allowlist. Any unprivileged address can bypass the deposit allowlist by calling `addLiquidity` with an allowlisted address as `owner`, depositing into a restricted pool without authorization. This directly contradicts the contract's stated invariant: *"Gates `addLiquidity` by depositor address, per pool."*

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as separate arguments to `_beforeAddLiquidity`: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but leaves it unnamed and unused. It checks only `owner`: [2](#0-1) 

The check `allowedDepositor[msg.sender][owner]` passes whenever `owner` is allowlisted, regardless of who `sender` is. The actual depositor — the address paying tokens via the swap callback — is never validated. By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender`: [3](#0-2) 

The contract's own NatSpec names the gated entity as "depositor address": [4](#0-3) 

## Impact Explanation
A pool admin deploys `DepositAllowlistExtension` to restrict liquidity provision to a curated set (e.g., KYC'd institutional LPs). Any unprivileged address bypasses this control by specifying any allowlisted address as `owner`. The unauthorized caller pays tokens via the callback (called on `msg.sender` of the pool), while the allowlisted address receives LP shares. The pool admin's access control invariant is broken: unauthorized parties can inject liquidity into restricted pools, manipulating bin balances and pool state in ways the admin did not authorize. This is an admin-boundary break by an unprivileged path.

## Likelihood Explanation
Exploitation requires no special privileges, no flash loans, and no complex setup. The `allowedDepositor` mapping is public and readable on-chain. Any address can call `addLiquidity` on a pool with this extension, specifying any allowlisted address as `owner`. The attack is repeatable at will.

## Recommendation
Check `sender` (the actual depositor/caller) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the depositor and the position owner, both should be checked independently.

## Proof of Concept
```
Setup:
  - Pool P has DepositAllowlistExtension configured
  - allowedDepositor[P][alice] = true
  - bob is NOT on the allowlist

Attack:
  1. bob calls P.addLiquidity(owner=alice, salt=0, deltas=..., callbackData=..., extensionData=...)
  2. Extension checks allowedDepositor[P][alice] → true → no revert
  3. LiquidityLib calls back on bob (msg.sender of pool) to collect tokens
  4. bob pays tokens; alice receives LP shares
  5. bob has successfully deposited into a restricted pool without being allowlisted
```
