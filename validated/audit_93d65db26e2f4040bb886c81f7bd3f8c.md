Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address as `sender` instead of actual end user, allowing allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swap access by checking the `sender` argument against `allowedSwapper[pool][sender]`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` equals the router address (the direct `msg.sender` of `pool.swap()`), not the actual end user. Any pool that enables router-based swaps must allowlist the router, and once the router is allowlisted, every user — including those explicitly excluded — can bypass the per-user allowlist by routing through the router.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs its access check at line 37:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension caller) and `sender` is the first argument forwarded from `MetricOmmPool.swap()` via `ExtensionCalling._beforeSwap`. In `MetricOmmPool.swap()` (line 230–231), `_beforeSwap` is called with `msg.sender` as the first argument — i.e., the direct caller of `pool.swap()`.

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly (line 72–80 of `MetricOmmSimpleRouter.sol`). The actual end user is stored only in transient storage via `_setNextCallbackContext` for the payment callback and is never forwarded to the extension. Therefore, `sender` received by `beforeSwap` is the router address, not the real user.

For any pool using `SwapAllowlistExtension` that also supports router-based swaps, the pool admin must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, `allowedSwapper[pool][router]` is `true` for every call routed through it, regardless of whether the actual end user is individually allowed or explicitly excluded.

Exploit path:
1. Pool deployed with `SwapAllowlistExtension` on `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps.
3. Pool admin never adds Alice (or explicitly sets `setAllowedToSwap(pool, alice, false)`).
4. Alice calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, ...)` — `msg.sender` = router.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Alice successfully swaps on a pool she was excluded from.

## Impact Explanation
This is an admin-boundary break: the pool admin's configured per-user swap allowlist — the primary curation mechanism for pools restricting trading to specific counterparties (KYC'd addresses, institutional partners) — is silently defeated by the standard periphery path. Any disallowed user can trade on a curated pool by routing through `MetricOmmSimpleRouter`, bypassing the access control the pool admin intended to enforce. This constitutes broken core pool functionality and a direct violation of the admin-configured access boundary.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard documented periphery entry point for swaps. Any pool deploying `SwapAllowlistExtension` and expecting users to swap through the router — the normal usage pattern — is immediately affected. No special attacker capability is required: any user simply calls `exactInputSingle` or `exactInput` on the router. The precondition (router allowlisted) is a necessary operational requirement, not an unusual configuration.

## Recommendation
The `beforeSwap` hook must check the actual end user, not the intermediary router. The cleanest fix is for the router to encode the original caller (`msg.sender`) into `extensionData` before forwarding to the pool. `SwapAllowlistExtension.beforeSwap` would then decode and verify that address when present, falling back to `sender` for direct pool calls. This requires a convention between the router and the extension but avoids relying on `recipient` (which is unreliable for multi-hop paths where intermediate recipients are the router itself).

## Proof of Concept
1. Deploy pool with `SwapAllowlistExtension` configured on `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)`.
3. Pool admin does NOT call `setAllowedToSwap(pool, alice, true)` (Alice is not allowlisted).
4. Alice calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: alice, ...})`.
5. Router executes `IMetricOmmPoolActions(pool).swap(alice, ...)` with `msg.sender` = router.
6. Pool calls `_beforeSwap(msg.sender=router, ...)` → extension receives `sender=router`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
8. Alice's swap completes on a pool she was never authorized to trade on.

A Foundry integration test can confirm this by asserting that Alice's `exactInputSingle` call succeeds when `allowedSwapper[pool][alice]` is `false` but `allowedSwapper[pool][router]` is `true`. [1](#0-0) [2](#0-1) [3](#0-2)
