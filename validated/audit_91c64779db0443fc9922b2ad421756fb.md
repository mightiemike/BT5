### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual User, Making the Allowlist Bypassable or the Standard Swap Path Unusable — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's `msg.sender` — the `MetricOmmSimpleRouter` — not the originating user. Any pool that configures a per-user swap allowlist is broken in one of two ways: allowlisted users cannot swap through the standard router, or the pool admin is forced to allowlist the router itself, which lets every unprivileged user bypass the allowlist entirely.

---

### Finding Description

**Call chain for a router-mediated swap:**

```
User (Alice) → MetricOmmSimpleRouter.exactInputSingle()
    → pool.swap(recipient, ..., extensionData)   // msg.sender = router
        → _beforeSwap(msg.sender=router, ...)
            → SwapAllowlistExtension.beforeSwap(sender=router, ...)
```

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (correct key) and `sender` is the router (wrong actor): [3](#0-2) 

The router never exposes the originating user's address to the pool or the extension. In `exactInputSingle` and `exactInput`, the router calls `pool.swap(params.recipient, ...)` with itself as `msg.sender`: [4](#0-3) 

The `DepositAllowlistExtension` does **not** share this flaw because it ignores `sender` and checks `owner` (the position owner explicitly passed through the call), which the liquidity adder correctly threads through: [5](#0-4) 

---

### Impact Explanation

Two mutually exclusive failure modes arise for any pool that configures `SwapAllowlistExtension` with a per-user list:

**Mode A — Allowlisted users cannot use the standard router (broken core swap flow).**
The pool admin allowlists specific user addresses (e.g., Alice, Bob). When Alice calls `exactInputSingle` through `MetricOmmSimpleRouter`, the extension sees `sender = router`, which is not on the allowlist, and reverts `NotAllowedToSwap`. Alice must implement `IMetricOmmSwapCallback` herself and call the pool directly — the standard periphery swap path is unusable for her.

**Mode B — Router allowlisted ⇒ allowlist fully bypassed.**
To unblock router users, the pool admin allowlists the router address. Now every unprivileged user can call `exactInputSingle` through the router and the extension passes, defeating the entire purpose of the allowlist. Any actor the admin intended to exclude can swap freely.

Both outcomes are contest-relevant: Mode A is broken core pool functionality on the standard swap path; Mode B is an admin-boundary break where an unprivileged path bypasses a configured access control.

---

### Likelihood Explanation

The trigger is the normal, documented user flow. Any user of an allowlisted pool who reaches for `MetricOmmSimpleRouter` — the only provided periphery swap contract — hits this immediately. No special timing, no privileged access, no exotic token behavior is required. The router is the intended entry point for all end users.

---

### Recommendation

Pass the originating user's address through the swap call so the extension can gate on it. Two options:

1. **Encode the real payer in `extensionData`** — the router writes `msg.sender` into `extensionData` before calling `pool.swap`; the extension decodes and checks it. This requires the extension to trust the router, which must be verified via `BaseMetricExtension.onlyPool` or a separate router registry.

2. **Add a `payer` field to the pool's swap interface** — the pool accepts an explicit `payer` address (analogous to how `addLiquidity` separates `msg.sender` from `owner`) and passes it to extensions as the authoritative actor. The router sets `payer = msg.sender`.

Option 2 is the cleaner invariant: every guard must key authorization to the same actor that the economic action is attributed to.

---

### Proof of Concept

```solidity
// Setup
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
// Pool admin allowlists Alice directly (not the router)
ext.setAllowedToSwap(address(pool), alice, true);

// Alice tries to swap through the standard router
vm.prank(alice);
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: alice,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp + 1,
    extensionData: ""
}));
// ↑ Reverts with NotAllowedToSwap because the extension sees sender=router, not alice.

// Conversely: admin allowlists the router to unblock alice
ext.setAllowedToSwap(address(pool), address(router), true);
// Now bob (not allowlisted) can also swap freely through the router:
vm.prank(bob);
router.exactInputSingle(...); // succeeds — allowlist fully bypassed
```

The root cause is that `SwapAllowlistExtension.beforeSwap` receives `sender = address(router)` in both cases, never `alice` or `bob`. [6](#0-5) [7](#0-6) [8](#0-7)
