### Title
NEAR `env::block_timestamp_ms()` Used as Real-Time Proxy in Bitcoin Future-Timestamp Validation — (`contract/src/bitcoin.rs`, `contract/src/dogecoin.rs`, `contract/src/litecoin.rs`, `contract/src/zcash.rs`)

---

### Summary

All four chain-specific `check_pow` implementations use `env::block_timestamp_ms()` — the NEAR blockchain's current block timestamp — as a proxy for real-world time when enforcing the "time-too-new" rule. Because NEAR's block timestamp is set by validators and can drift ahead of real time, the effective 2-hour future-timestamp window can be silently widened, allowing a relayer to submit Bitcoin block headers with timestamps that exceed the intended 2-hour limit relative to actual wall-clock time.

---

### Finding Description

In `check_pow` for Bitcoin, Dogecoin, Litecoin, and Zcash, the contract validates that a submitted Bitcoin block's timestamp is not more than `MAX_FUTURE_BLOCK_TIME_LOCAL` (7200 seconds / 2 hours) ahead of "current time":

```rust
// contract/src/bitcoin.rs:35-38
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
    "time-too-new: block timestamp too far in the future"
);
```

`env::block_timestamp_ms()` returns the NEAR blockchain's current block timestamp in milliseconds, set by the NEAR block producer. This is not guaranteed to match real-world time. NEAR's protocol allows block timestamps to be ahead of real time within certain bounds. If NEAR's block timestamp is ahead of real time by `Δ` seconds, the effective future-timestamp window becomes `2h + Δ` instead of the intended `2h`. A relayer can observe the NEAR block timestamp and craft a Bitcoin block header with a timestamp that passes the on-chain check but exceeds the intended 2-hour limit relative to actual wall-clock time.

The same pattern appears identically in all four chain implementations: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

The constant `MAX_FUTURE_BLOCK_TIME_LOCAL` is defined as `2 * 60 * 60` (7200 seconds): [5](#0-4) 

The check is intended to mirror Bitcoin Core's `CheckBlockHeader` rule, which compares the block timestamp against the node's local system clock. In Bitcoin Core, "local time" is actual wall-clock time. In this contract, "local time" is `env::block_timestamp_ms()` — the NEAR chain's block timestamp — which is a fundamentally different quantity.

---

### Impact Explanation

A relayer can submit a Bitcoin block header with a timestamp more than 2 hours ahead of real time, causing the light client to accept a block that the real Bitcoin network would reject. This corrupts the light client's canonical chain state: `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` are all updated based on the accepted header. [6](#0-5) 

Downstream contracts consuming `verify_transaction_inclusion_v2` results would receive incorrect proof outcomes based on a chain view that diverges from the real Bitcoin network. [7](#0-6) 

---

### Likelihood Explanation

Low for Bitcoin mainnet (requires valid PoW, which is computationally infeasible for an attacker). Meaningfully higher for Dogecoin testnet or Litecoin testnet where PoW difficulty is much lower and `pow_allow_min_difficulty_blocks` is `true`. [8](#0-7) 

NEAR's timestamp drift is typically bounded to seconds or minutes by NEAR's protocol, limiting the practical window of exploitation. However, the vulnerability is structural and present in all four chain implementations without any documentation or mitigation.

---

### Recommendation

Replace `env::block_timestamp_ms()` with a more robust approach. Options include:

1. **Use MTP-only validation**: The contract already enforces `block_header.time > get_median_time_past(...)`. The "time-too-new" check against local time is a secondary defense. Document that this check is relative to NEAR's block timestamp, not real-world time, and accept the bounded drift as a known limitation.
2. **Add a configurable clock skew tolerance**: Allow the contract owner to configure an additional buffer that accounts for expected NEAR timestamp drift.
3. **Document the behavior explicitly**: At minimum, add inline comments and README documentation clarifying that `env::block_timestamp_ms()` is NEAR's block timestamp, not wall-clock time, and that the effective future-timestamp window may be wider than 2 hours by the amount of NEAR's timestamp drift.

---

### Proof of Concept

1. Observe NEAR's current block timestamp: `T_near = real_time + Δ` (where `Δ` is NEAR's timestamp drift ahead of real time, e.g., 60 seconds).
2. Craft a Bitcoin block header (on Dogecoin testnet for feasible PoW) with `time = T_near + 7200 - 1` (just within the allowed window).
3. Submit the block header via `submit_blocks()`.
4. The check passes: `block_header.time (= real_time + Δ + 7199) <= current_timestamp (= real_time + Δ) + 7200`.
5. But the Bitcoin block's timestamp is `real_time + Δ + 7199` — more than 2 hours ahead of real time.
6. The real Bitcoin/Dogecoin network would reject this block, but the light client accepts it, updating `mainchain_tip_blockhash` to point to an invalid header and corrupting all subsequent SPV proof verifications.
