### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable Without Coinbase Merkle Proof Validation, Enabling 64-Byte Transaction Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is still a live, unpermissioned public entry point on the NEAR contract. It omits the coinbase merkle proof check that was introduced in `verify_transaction_inclusion_v2` specifically to close the 64-byte transaction Merkle forgery path. Any unprivileged NEAR caller can invoke the old function with a crafted proof and receive `true` for a transaction that was never included in any Bitcoin block.

---

### Finding Description

`verify_transaction_inclusion` (lines 288–323 of `contract/src/lib.rs`) reconstructs the Merkle root from a caller-supplied `tx_id` and `merkle_proof` and compares it against the stored block's `merkle_root`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

There is no check that `tx_id` corresponds to a leaf (a real transaction) rather than an internal Merkle node. The 64-byte transaction forgery technique (https://www.bitmex.com/blog/64-Byte-Transactions) exploits exactly this gap: a 64-byte blob can be crafted whose double-SHA256 hash equals an internal Merkle node at depth D. Supplying that hash as `tx_id` with a proof of length D−1 causes the function to reconstruct the correct Merkle root and return `true`.

`verify_transaction_inclusion_v2` (lines 347–369) was added to close this hole by requiring a coinbase merkle proof of the same length, anchoring the proof depth to a known-valid leaf:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
``` [2](#0-1) 

However, the old function was never removed. It carries only a `#[deprecated]` attribute and a `#[pause]` gate (which is inactive by default). It remains `pub` and callable by any NEAR account with no role requirement. [3](#0-2) 

This is the direct analog of the external report's pattern: a function that should have been overridden/replaced with a correct implementation was not, so the default (incomplete) implementation remains reachable and returns incorrect results to callers.

---

### Impact Explanation

Any NEAR smart contract that gates an action on `verify_transaction_inclusion` returning `true` (e.g., releasing bridged funds, minting wrapped tokens, updating cross-chain state) can be deceived into accepting a forged transaction inclusion proof. The corrupted value is the proof result: the function returns `true` for a transaction that does not exist in the claimed Bitcoin block.

---

### Likelihood Explanation

Medium. The function is publicly callable by any NEAR account without any role or deposit requirement. The 64-byte forgery technique is well-documented and the construction of a valid-looking fake proof requires only off-chain computation against a known block's Merkle tree — no mining or privileged access is needed. The primary mitigating factor is that the function is deprecated and documented with a warning, so new integrations should prefer `verify_transaction_inclusion_v2`. Existing integrations using the old function remain fully exposed.

---

### Recommendation

Remove `verify_transaction_inclusion` entirely from the contract, or replace its body with an unconditional `env::panic_str("use verify_transaction_inclusion_v2")`. If backward ABI compatibility is strictly required, apply the same coinbase merkle proof validation that `verify_transaction_inclusion_v2` performs before delegating to the shared proof logic.

---

### Proof of Concept

1. Identify any Bitcoin block stored in the light client with a known Merkle tree of depth D (e.g., a block with 4 transactions has depth 2).
2. Locate an internal Merkle node `N` at depth D−1 (the hash of two concatenated child hashes).
3. Craft a 64-byte blob whose double-SHA256 equals `N`; call this `fake_txid`.
4. Construct a Merkle proof of length D−1 that walks from `N` up to the Merkle root.
5. Call `verify_transaction_inclusion` on-chain:
   - `tx_id = fake_txid`
   - `tx_block_blockhash` = hash of the target block
   - `tx_index` = the index corresponding to `N`'s position
   - `merkle_proof` = the D−1 element proof
   - `confirmations = 1`
6. The function returns `true`. No such transaction exists in the block. [4](#0-3)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
