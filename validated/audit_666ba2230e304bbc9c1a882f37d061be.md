### Title
Phantom Transaction-Position Collision in Merkle Proof Verification — (`merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` accepts an arbitrary `transaction_position` without validating it against the actual number of transactions in the block. Because `merkle_proof_calculator` duplicates the last leaf when the transaction count is odd, the last real transaction and its phantom duplicate position produce **identical proofs and identical computed roots**. Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` therefore return `true` for a `tx_index` that does not correspond to any real transaction, violating the uniqueness invariant of inclusion proofs.

---

### Finding Description

`merkle_proof_calculator` pads an odd-length leaf set by appending a copy of the last hash: [1](#0-0) 

For a block with N transactions where N is odd, this creates a phantom leaf at position N that is byte-for-byte identical to the real leaf at position N-1. Because the sibling of position N-1 is the duplicated copy of itself, the proof vector for position N-1 and position N are **identical**: `[T_last, hash(left_subtree)]`.

`compute_root_from_merkle_proof` then computes the root purely from `(transaction_hash, transaction_position, merkle_proof)` with no bounds check: [2](#0-1) 

Tracing both positions through the loop:

| Step | position N-1 (even) | position N (odd) |
|------|---------------------|-----------------|
| 1 | `hash(T_last, T_last)` | `hash(T_last, T_last)` |
| 2 | `hash(left, result)` | `hash(left, result)` |

The computed root is identical for both positions. Neither `verify_transaction_inclusion` nor `verify_transaction_inclusion_v2` adds a bounds check on `tx_index`: [3](#0-2) [4](#0-3) 

`ProofArgs.tx_index` is a caller-supplied `u64` with no server-side range enforcement: [5](#0-4) 

---

### Impact Explanation

Any consumer contract or bridge that calls `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` and relies on the returned `true` to mean "transaction T exists at position `tx_index` in this block" receives a false guarantee. An attacker can present `tx_index = N` (phantom) for a block that only has N transactions (indices 0..N-1), and the call returns `true`. Protocols that use `tx_index` to distinguish transaction roles — e.g., coinbase detection (index 0), output-position routing in atomic swaps, or cross-chain bridge replay guards keyed on `(block_hash, tx_index)` — can be deceived into accepting a fabricated position as canonical.

---

### Likelihood Explanation

The precondition is a Bitcoin block with an odd transaction count. The overwhelming majority of mainnet blocks satisfy this (coinbase + any even number of user transactions = odd total). The attacker needs only publicly available block data (the last transaction hash and its sibling path) and the ability to call the NEAR contract as an unprivileged account. No privileged role, key material, or social engineering is required.

---

### Recommendation

Add an upper-bound check on `transaction_position` inside `compute_root_from_merkle_proof`: the maximum valid leaf index for a proof of length `L` is `2^L - 1`, and the caller-supplied position must be strictly less than the actual transaction count. Since the contract does not store transaction counts, the recommended fix is to enforce `transaction_position < 2^merkle_proof.len()` as a minimum guard, and to document that callers must additionally supply the transaction count so the contract can reject phantom positions (`tx_index >= tx_count`).

---

### Proof of Concept

Consider a block with 3 transactions `[T0, T1, T2]`:

- Merkle tree (after duplication): `[T0, T1, T2, T2]`
- Merkle root: `hash(hash(T0,T1), hash(T2,T2))`
- Proof for T2 at position 2: `[T2, hash(T0,T1)]`
- Proof for T2 at position 3 (phantom): `[T2, hash(T0,T1)]` ← identical

An attacker calls:
```
verify_transaction_inclusion({
    tx_id: T2,
    tx_block_blockhash: <block_hash>,
    tx_index: 3,          // phantom — no real tx here
    merkle_proof: [T2, hash(T0,T1)],
    confirmations: 1
})
```

`compute_root_from_merkle_proof(T2, 3, [T2, hash(T0,T1)])` computes:
1. position=3 (odd) → `hash(T2, T2)`, position→1
2. position=1 (odd) → `hash(hash(T0,T1), hash(T2,T2))` = merkle_root

The function returns `true` for a transaction index that does not exist in the block, producing the same verification result as the legitimate call with `tx_index: 2`. Two distinct `(tx_id, tx_index)` inputs — one real, one phantom — are rendered equivalent by the verifier, directly mirroring the coin-denomination collision described in the reference report.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
