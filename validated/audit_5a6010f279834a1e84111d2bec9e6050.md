### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Bypasses Coinbase Proof Validation, Enabling 64-Byte Merkle Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, public NEAR entry point despite being deprecated. It skips the coinbase Merkle proof validation that `verify_transaction_inclusion_v2` enforces. Any unprivileged NEAR caller can invoke v1 directly with a forged `tx_id` (an internal Merkle tree node) and receive `true`, while the same forged proof would be rejected by v2. Downstream contracts that consume the boolean result cannot distinguish which version was called and will act on a fraudulent proof.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability. It enforces two checks that v1 does not:

1. **Coinbase proof length parity**: `args.merkle_proof.len() == args.coinbase_merkle_proof.len()`
2. **Coinbase Merkle proof validity**: `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_proof) == header.merkle_root` [1](#0-0) 

Only after both checks pass does v2 delegate to v1: [2](#0-1) 

v1, however, performs **no coinbase validation at all**. It only checks that the supplied `tx_id`'s Merkle path reconstructs to the stored `merkle_root`: [3](#0-2) 

The contract's own `# Warning` comment acknowledges this:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [4](#0-3) 

Despite this acknowledgment, v1 carries no `#[private]` attribute, no role guard, and no `#[access_control]` restriction beyond the shared `#[pause]` decorator. It is reachable by any NEAR account. [5](#0-4) 

The asymmetry is exact: v2 enforces the coinbase proof check; v1 silently skips it. An attacker who calls v1 directly bypasses the only mitigation for the 64-byte forgery attack.

---

### Impact Explanation

A downstream bridge or cross-chain contract that calls `verify_transaction_inclusion` to gate fund releases or state transitions will receive `true` for a forged proof. The attacker does not need to produce a real Bitcoin transaction — only an internal Merkle tree node whose path reconstructs to a real block's `merkle_root`. This allows fabricating "proof" of a Bitcoin deposit or payment that never occurred, enabling theft of bridged assets or unauthorized state changes in any consumer contract. [6](#0-5) 

---

### Likelihood Explanation

- The entry point is public with no role restriction.
- The 64-byte Merkle forgery technique is well-documented (BitMEX research, referenced in the v2 docstring at line 332).
- The attacker only needs a valid Bitcoin block header already stored in `headers_pool` — no privileged access, no key material, no social engineering.
- The attack is fully off-chain constructible: pick any stored block, find an internal node whose subtree path reconstructs `merkle_root`, submit to v1. [7](#0-6) 

---

### Recommendation

Remove `verify_transaction_inclusion` entirely from the public interface, or gate it with `#[private]` so it is only callable internally (as v2 already does via `self.verify_transaction_inclusion(args.into())`). Any external caller must be forced through `verify_transaction_inclusion_v2`. If backward compatibility is required, replicate the coinbase proof validation inside v1 before the Merkle root comparison.

---

### Proof of Concept

1. Identify any Bitcoin block already accepted into `headers_pool` with known `merkle_root = R`.
2. Locate an internal Merkle tree node `N` in that block's transaction tree (e.g., the parent of two leaf transactions). `N` is a 32-byte SHA256d hash.
3. Construct `merkle_proof` such that `compute_root_from_merkle_proof(N, index, merkle_proof) == R`. This is trivially possible because `N` is already an intermediate node in the real tree — its sibling path to the root is the proof.
4. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = <block hash>`, `tx_index = index`, `merkle_proof = <sibling path>`, `confirmations = 1`.
5. The function returns `true` because `compute_root_from_merkle_proof(N, index, proof) == R` holds by construction, even though `N` is not a real transaction hash.
6. Any downstream contract gating on this boolean now believes a non-existent Bitcoin transaction was confirmed. [8](#0-7) [3](#0-2)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
