### Title
`ValidResourceBounds` protobuf deserialization silently downgrades `AllResources` V3 transactions to `L1Gas` when l2_gas and l1_data_gas are zero, producing a divergent transaction hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion in `crates/apollo_protobuf/src/converters/transaction.rs` uses a value-based heuristic to decide between the `L1Gas` and `AllResources` enum variants. When a V3 transaction carries `AllResources` with `l2_gas = 0` and `l1_data_gas = 0` (but non-zero `l1_gas`), the deserialization silently produces `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. Because `get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` hashes a different number of elements for each variant, the transaction hash recomputed from the deserialized data diverges from the canonical hash that was computed at submission time. This breaks hash canonicalization across the p2p sync boundary and causes the RPC to return an authoritative-looking wrong hash for any such transaction.

---

### Finding Description

**Root cause — non-canonical deserialization heuristic** [1](#0-0) 

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong for AllResources V3 txs
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The serialization direction is lossless — it always emits all three fields: [2](#0-1) 

But the deserialization direction is lossy: `AllResources(l1_gas, l2_gas=0, l1_data_gas=0)` → protobuf → `L1Gas(l1_gas)`. The variant information is destroyed.

**Why the variant matters for hashing**

`get_tip_resource_bounds_hash` branches on the variant and hashes a different number of elements: [3](#0-2) 

- `L1Gas` path: hashes `[tip, L1_GAS_bounds, L2_GAS_bounds(zero)]` — **3 elements**
- `AllResources` path: hashes `[tip, L1_GAS_bounds, L2_GAS_bounds(zero), L1_DATA_GAS_bounds(zero)]` — **4 elements**

These produce different Poseidon digests even when the numeric values of the bounds are identical. The canonical hash for a V3 `AllResources` transaction (the one the account signed and the gateway stored) always includes the fourth element. After protobuf round-trip the hash is recomputed with only three elements.

**The same heuristic is duplicated in the RPC layer** [4](#0-3) 

`From<ResourceBoundsMapping> for ValidResourceBounds` applies the identical `l1_data_gas.is_zero() && l2_gas.is_zero()` test, so any node serving such a transaction via `starknet_getTransactionByHash` reconstructs it with the wrong variant and therefore the wrong hash preimage.

**Attacker-controlled trigger**

The gateway's stateless validator only requires `max_possible_fee > 0`: [5](#0-4) 

For `AllResources` with `l2_gas = 0` and `l1_data_gas = 0`, `max_possible_fee = l1_gas.max_amount × l1_gas.max_price_per_unit`. Any non-zero `l1_gas` satisfies this check. No privilege is required; any user can submit such a transaction.

**Conversion path that recomputes the hash**

When a `ConsensusTransaction` is received from a peer, `convert_consensus_tx_to_internal_consensus_tx` calls `convert_rpc_tx_to_internal`, which recomputes the hash from the deserialized body: [6](#0-5) 

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
```

`InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` delegates to `get_invoke_transaction_v3_hash` via the `InvokeTransactionV3Trait` impl on `InternalRpcInvokeTransactionV3`, which always wraps the stored `AllResourceBounds` as `ValidResourceBounds::AllResources`: [7](#0-6) 

So the consensus path (which uses `AllResourceBounds` throughout) is **not** affected. The divergence occurs specifically in the p2p state-sync path, which uses `ValidResourceBounds` deserialized from `transaction.rs`.

---

### Impact Explanation

**High — RPC returns an authoritative-looking wrong value.**  
A syncing node that deserializes a block containing such a transaction via the p2p sync path reconstructs it with `L1Gas` semantics. The hash it computes and stores differs from the canonical hash. Any subsequent `starknet_getTransactionByHash` or `starknet_getTransactionReceipt` call returns a transaction body whose hash preimage does not match the stored hash, breaking client-side signature verification and receipt matching.

**High — Mempool/gateway admission divergence.**  
If the wrong hash is stored, a re-submitted transaction with the same fields will be treated as a duplicate under the wrong hash, or the correct hash will not be found, causing incorrect admission decisions.

---

### Likelihood Explanation

The trigger is a standard V3 invoke transaction with only `l1_gas` set (the most common pre-0.13.3 usage pattern for users who have not yet migrated to multi-resource bounds). Such transactions are accepted by the gateway today. The bug is latent for every such transaction that passes through the p2p sync path.

---

### Recommendation

Remove the value-based heuristic. The protobuf `ResourceBounds` message already carries all three fields; the variant should be encoded explicitly (e.g., a boolean `is_all_resources` field, or by always deserializing to `AllResources` for V3 transactions). At minimum, the deserialization should always produce `AllResources` when all three fields are present in the wire message, regardless of whether their values are zero:

```rust
// Always AllResources when the wire message carries all three fields.
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

Apply the same fix to `From<ResourceBoundsMapping> for ValidResourceBounds` in `crates/apollo_rpc/src/v0_8/transaction.rs`.

---

### Proof of Concept

```
1. Submit via RPC:
   InvokeV3 {
     resource_bounds: AllResources { l1_gas: {max_amount:1, max_price:1}, l2_gas: ZERO, l1_data_gas: ZERO },
     ...
   }

2. Gateway computes hash H_all using AllResources path:
   poseidon([INVOKE, version, sender, tip_resource_hash_4_elements, ..., calldata_hash])
   where tip_resource_hash_4_elements = poseidon([tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)])

3. Transaction included in block B with hash H_all.

4. Peer syncs block B via p2p. Protobuf ResourceBounds arrives with l2_gas=0, l1_data_gas=0.
   TryFrom heuristic fires → ValidResourceBounds::L1Gas(l1_gas).

5. Hash recomputed as H_l1 using L1Gas path:
   tip_resource_hash_3_elements = poseidon([tip, L1_GAS_packed, L2_GAS_packed(0)])
   H_l1 ≠ H_all  (different Poseidon digest due to missing fourth element)

6. Hash validation: expected H_all, got H_l1 → FAIL.
   Syncing node rejects the block or stores the wrong hash.
   starknet_getTransactionByHash(H_all) returns wrong data or not found.
```

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
