### Title
Protobuf `ValidResourceBounds` Deserialization Uses Resource-Bound Values as Version Discriminant, Misclassifying `AllResources` V3 Transactions as `L1Gas` and Producing a Divergent Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation in the P2P state-sync path uses the zero-ness of the `l2_gas` and `l1_data_gas` resource-bound values as the sole discriminant to decide whether to produce `ValidResourceBounds::L1Gas` (pre-0.13.3) or `ValidResourceBounds::AllResources` (0.13.3+). This is a dual-purpose use of the same fields: their numeric values are the resource bounds, and their zero-ness determines the transaction version/type. A valid V3 transaction with `AllResources` bounds where both `l2_gas = 0` and `l1_data_gas = 0` is accepted by the gateway, included in a block, and then misclassified as `L1Gas` by every syncing node. Because `get_tip_resource_bounds_hash` produces structurally different output for the two variants (3 vs 4 Poseidon inputs), the transaction hash computed from the misclassified representation diverges from the canonical hash signed by the sender.

---

### Finding Description

**Root cause — dual-purpose field in protobuf deserialization:** [1](#0-0) 

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← pre-0.13.3 path
        } else {
            ValidResourceBounds::AllResources(...)       // ← 0.13.3+ path
        })
    }
}
```

The `l2_gas` and `l1_data_gas` fields serve two roles simultaneously:
1. Their numeric values are the actual resource bounds.
2. Their zero-ness is the discriminant that selects the `ValidResourceBounds` variant.

This is the exact structural analog to the external bug: `_maxPaidAmount` was used both as the ETH `value` sent and as the `qty` parameter; zeroing it for non-native tokens silently zeroed the quantity. Here, zeroing `l2_gas`/`l1_data_gas` for a legitimate `AllResources` transaction silently downgrades the variant to `L1Gas`.

**Where this converter is invoked in the state-sync path:** [2](#0-1) 

`TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` calls `ValidResourceBounds::try_from(...)` directly. This is reached from: [3](#0-2) 

`TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)` → `TryFrom<protobuf::TransactionWithReceipt> for FullTransaction`, which is the P2P block-sync deserialization path.

**The hash divergence:**

`get_tip_resource_bounds_hash` branches on the variant: [4](#0-3) 

- `AllResources`: Poseidon over `[tip, l1_gas_packed, l2_gas_packed(0), l1_data_gas_packed(0)]` — **4 elements**
- `L1Gas`: Poseidon over `[tip, l1_gas_packed, l2_gas_packed(0)]` — **3 elements**

The 4th element `get_concat_resource(l1_data_gas_zero, L1_DATA_GAS)` is **non-zero** even when amounts are zero, because it encodes the `L1_DATA_GAS` resource-name bytes in the upper 56 bits of the packed felt: [5](#0-4) 

Therefore the two hashes are always distinct for the same `l1_gas` value.

**The consensus path is unaffected** because `ConsensusTransaction::RpcTransaction` uses `RpcTransaction` → `AllResourceBounds` via the separate `rpc_transaction.rs` converter, which always produces `AllResources` and requires all three fields: [6](#0-5) 

**The trigger is valid and accepted by the gateway.** The stateless validator explicitly accepts `AllResourceBounds` with `l2_gas = 0` and `l1_data_gas = 0`: [7](#0-6) 

---

### Impact Explanation

A syncing node that receives a `TransactionWithReceipt` for a V3 transaction with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` stores it with `ValidResourceBounds::L1Gas(X)`. Consequences:

1. **Wrong transaction hash** — any code path that recomputes the hash from the stored `InvokeTransactionV3` (e.g., `validate_transaction_hash`) produces a hash that does not match the canonical hash signed by the sender and committed in the block.
2. **Wrong gas vector computation mode** — `get_gas_vector_computation_mode()` returns `NoL2Gas` instead of `All`: [8](#0-7) 
3. **Wrong RPC simulation/estimation** — `starknet_simulateTransactions` and `starknet_estimateFee` re-execute the stored transaction under `L1Gas` mode, skipping L2 and L1-data gas bound checks and using a different fee computation path: [9](#0-8) 

This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

Any unprivileged user can submit a V3 invoke, declare, or deploy-account transaction with `AllResourceBounds { l1_gas: non_zero, l2_gas: 0, l1_data_gas: 0 }`. The gateway's stateless validator accepts this. The transaction is included in a block. Every node that syncs the block via P2P will misclassify it. No special privileges or malformed bytes are required.

---

### Recommendation

Separate the version/type discriminant from the resource-bound values. The correct fix is to carry an explicit version tag through the protobuf deserialization path (e.g., from the enclosing transaction version field) and use that to select the `ValidResourceBounds` variant, rather than inferring it from whether the bound values happen to be zero. The TODO comment already acknowledges this: [10](#0-9) 

Until the TODO is resolved, the immediate fix is to always produce `AllResources` when `l1_data_gas` is present in the protobuf message (even if zero), and only fall back to `L1Gas` when `l1_data_gas` is absent (`None`), which is the actual 0.13.2 wire signal:

```rust
let l1_data_gas_opt = value.l1_data_gas;
...
Ok(match l1_data_gas_opt {
    None => ValidResourceBounds::L1Gas(l1_gas),   // truly pre-0.13.3: field absent
    Some(_) => ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }),
})
```

The serializer already sets `l1_data_gas: Some(ResourceBounds::default().into())` for `L1Gas` transactions: [11](#0-10) 

so this change would require also updating the serializer to emit `l1_data_gas: None` for `L1Gas` transactions to preserve round-trip fidelity.

---

### Proof of Concept

1. Submit a V3 invoke transaction with `resource_bounds = AllResourceBounds { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: { 0, 0 }, l1_data_gas: { 0, 0 } }` to the gateway. It is accepted and included in a block. The gateway computes the hash using `AllResources` (4-element Poseidon input).

2. A second node syncs the block via P2P. The `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` converter sees `l2_gas.is_zero() && l1_data_gas.is_zero()` and stores `ValidResourceBounds::L1Gas(l1_gas)`.

3. Call `starknet_simulateTransactions` on the syncing node for that transaction hash. The blockifier executes it under `GasVectorComputationMode::NoL2Gas` instead of `All`, producing a different fee result than the originating node.

4. Call `validate_transaction_hash` on the stored transaction. It recomputes the hash using `get_tip_resource_bounds_hash` with `L1Gas` (3-element Poseidon input) and obtains a value that does not match the hash stored in the block header, demonstrating the canonical hash divergence.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

fn get_concat_resource(
    resource_bounds: &ResourceBounds,
    resource_name: &ResourceName,
) -> Result<Felt, StarknetApiError> {
    let max_amount = resource_bounds.max_amount.0.to_be_bytes();
    let max_price = resource_bounds.max_price_per_unit.0.to_be_bytes();
    let concat_bytes =
        [[0_u8].as_slice(), resource_name.as_slice(), max_amount.as_slice(), max_price.as_slice()]
            .concat();
    Ok(Felt::from_bytes_be(&concat_bytes.try_into().expect("Expect 32 bytes")))
}
