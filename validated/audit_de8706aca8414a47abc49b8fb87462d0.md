### Title
`finalize_epoch` fallback path stores stale `shard_layout` and `last_resharding` in `EpochInfoV5`, silently dropping a scheduled dynamic shard split — (`chain/epoch-manager/src/lib.rs`)

---

### Summary

In `EpochManager::finalize_epoch`, when `proposals_to_epoch_info` fails with `EpochError::ThresholdError` or `EpochError::NotEnoughValidators`, the error-recovery path clones `next_epoch_info` (epoch N+1's `EpochInfo`) and increments only its `epoch_height`. It does **not** apply the freshly-computed `next_next_shard_layout` or the freshly-computed `last_resharding` value. When `ProtocolFeature::DynamicResharding` is active, `next_epoch_info` is an `EpochInfoV5` whose `shard_layout` field is the layout for epoch N+1. The fallback therefore persists epoch N+1's layout as the authoritative layout for epoch N+2, silently discarding any shard split that was scheduled for that epoch.

---

### Finding Description

`finalize_epoch` computes two values before calling `proposals_to_epoch_info`:

```rust
// chain/epoch-manager/src/lib.rs  lines 926-929
let has_same_shard_layout = next_next_shard_layout == next_shard_layout;
let last_resharding = (!has_same_shard_layout)
    .then(|| next_epoch_info.epoch_height() + 1)
    .or_else(|| next_epoch_info.last_resharding());
```

Both values are then passed into `proposals_to_epoch_info`:

```rust
// lines 938-950
let next_next_epoch_info = match proposals_to_epoch_info(
    &next_next_epoch_config, rng_seed, &next_epoch_info,
    all_proposals, validator_kickout, validator_reward, minted_amount,
    next_next_epoch_version,
    next_next_shard_layout.clone(),   // ← correct split layout
    &strategy,
    last_resharding,                  // ← correct cooldown epoch
) {
    Ok(next_next_epoch_info) => next_next_epoch_info,
    Err(EpochError::ThresholdError { stake_sum, num_seats }) => {
        let mut epoch_info = EpochInfo::clone(&next_epoch_info); // ← clones N+1
        *epoch_info.epoch_height_mut() += 1;
        epoch_info   // shard_layout = N+1's layout; last_resharding = N+1's value
    }
    Err(EpochError::NotEnoughValidators { num_validators, num_shards }) => {
        let mut epoch_info = EpochInfo::clone(&next_epoch_info); // ← clones N+1
        *epoch_info.epoch_height_mut() += 1;
        epoch_info   // same problem
    }
    Err(err) => return Err(err),
};
```

`EpochInfo::new` creates `EpochInfoV5` whenever `ProtocolFeature::DynamicResharding` is enabled, storing `shard_layout` and `last_resharding` as first-class fields. The fallback bypasses `EpochInfo::new` entirely, so the two fields are never updated to the values that were computed for epoch N+2. The resulting `EpochInfoV5` is then persisted as the authoritative record for epoch N+2 via `save_epoch_info`.

The dual-source lookup in `get_shard_layout` reads `EpochInfo::shard_layout()` first for V5 epochs, so the stale value becomes the definitive answer for all subsequent callers:

```rust
// chain/epoch-manager/src/lib.rs  lines 1759-1771
pub fn get_shard_layout(&self, epoch_id: &EpochId) -> Result<ShardLayout, EpochError> {
    let epoch_info = self.get_epoch_info(epoch_id)?;
    if let Some(shard_layout) = epoch_info.shard_layout() {
        Ok(shard_layout.clone())   // ← returns N+1's layout for epoch N+2
    } else { ... }
}
```

The `last_resharding` divergence is equally damaging. When a split is scheduled (`has_same_shard_layout = false`), the correct value is `next_epoch_info.epoch_height() + 1`. The fallback stores `next_epoch_info.last_resharding()` instead — which may be `None` or an older epoch height — causing `can_reshard` to permit a back-to-back resharding that the documentation explicitly marks as unsafe.

---

### Impact Explanation

**Shard split silently dropped.** A split committed to `BlockHeaderInnerRestV6.shard_split` and propagated through `BlockInfoV4.shard_split` is fully validated by `validate_block_shard_split`, but the resulting `EpochInfoV5.shard_layout` for epoch N+2 retains the pre-split layout. Every consumer of `get_shard_layout(epoch_N+2)` — account-to-shard routing, chunk producer assignment, state-sync, memtrie pre-loading — operates on the wrong layout. Transactions are routed to the wrong shard, state witnesses are built against the wrong shard set, and the split's child shards are never created.

**Cooldown invariant broken.** With `last_resharding` wrong, `can_reshard` may return `true` immediately after a split epoch, enabling back-to-back reshardings. The documentation in `DynamicReshardingConfig` states this is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`.

**Scope:** High — incorrect shard layout persisted in consensus-critical `EpochInfo` DB record; all nodes that replay `finalize_epoch` from the same block will derive the same wrong layout, so the divergence is network-wide and permanent for that epoch.

---

### Likelihood Explanation

`ThresholdError` fires when `stake_sum < num_seats` — reachable if a large fraction of validators unstake in the same epoch. `NotEnoughValidators` fires when `num_validators < num_shards * minimum_validators_per_shard` — directly reachable by the shard split itself if the post-split shard count exceeds the validator-to-shard ratio. Both are documented as real production scenarios (the warn log says "all validators tried to unstake?"). Neither requires any privileged role beyond normal validator unstaking, which is a public protocol action. The interaction is at the public upgrade boundary where `DynamicResharding` (protocol version 85, stable) first activates and schedules its first split.

---

### Recommendation

In both fallback arms, apply `next_next_shard_layout` and `last_resharding` to the cloned `EpochInfo` before returning it. Because `EpochInfoV5` exposes `shard_layout` as a public field, the fix is straightforward:

```rust
Err(EpochError::ThresholdError { stake_sum, num_seats }) => {
    tracing::warn!(...);
    let mut epoch_info = EpochInfo::clone(&next_epoch_info);
    *epoch_info.epoch_height_mut() += 1;
    // Apply the computed shard layout and cooldown to the fallback epoch info.
    if let EpochInfo::V5(ref mut v5) = epoch_info {
        v5.shard_layout = next_next_shard_layout.clone();
        v5.last_resharding = last_resharding;
    }
    epoch_info
}
```

Apply the same patch to the `NotEnoughValidators` arm. Alternatively, extract a helper that constructs the minimal fallback `EpochInfoV5` directly from the computed fields rather than cloning `next_epoch_info`.

---

### Proof of Concept

1. Enable `DynamicResharding` (protocol version 85). Configure `force_split_shards` so a split is scheduled at the end of epoch N.
2. Arrange for `proposals_to_epoch_info` to return `ThresholdError` or `NotEnoughValidators` during `finalize_epoch` for epoch N. The `NotEnoughValidators` path is reachable by setting `minimum_validators_per_shard` such that the post-split shard count exceeds the available validator count.
3. Observe that `get_shard_layout(epoch_N+2)` returns the pre-split layout (same as epoch N+1) instead of the derived `ShardLayoutV3`.
4. Observe that `get_epoch_info(epoch_N+2).last_resharding()` returns `None` or the stale value from epoch N+1 instead of `epoch_N+2_height`.
5. Confirm that `can_reshard` immediately returns `true` for epoch N+2, violating the cooldown invariant.
