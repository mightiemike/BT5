### Title
Deprecated `verify_transaction_inclusion` Publicly Callable Without Coinbase Depth-Binding, Enabling Internal-Node Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is marked `#[deprecated]` but remains a fully public, unpermissioned NEAR contract method gated only by `#[pause]`. When the contract is not paused, any NEAR account can call it directly. The underlying `compute_root_from_merkle_proof` performs no proof-depth validation and no leaf-vs-internal-node distinction. An attacker can supply an internal Merkle tree node hash as `tx_id` with a proof that is one level shorter than the real tree depth, and the function will compute the correct Merkle root and return `true` — falsely asserting that a non-existent transaction is included in the block.

---

### Finding Description

`verify_transaction_inclusion` is declared `pub` with only `#[pause]` — no `#[private]`, no role check, no access-control guard: [1](#0-0) 

The sole Merkle validation is: [2](#0-1) 

`compute_root_from_merkle_proof` iterates over the supplied proof hashes and hashes upward. It has no concept of expected tree depth, no minimum proof length beyond the `is_empty()` guard, and no check that the starting hash is a leaf: [3](#0-2) 

**Concrete attack for a 4-transaction block `[T0, T1, T2, T3]`:**

```
Level 0 (leaves): T0, T1, T2, T3
Level 1 (internal): H01 = hash(T0,T1),  H23 = hash(T2,T3)
Level 2 (root):    R  = hash(H01, H23)
```

Attacker calls:
```
verify_transaction_inclusion(
    tx_id             = H01,   // internal node, not a real tx
    tx_index          = 0,
    merkle_proof      = [H23], // one level shorter than the real proof
    tx_block_blockhash = <any valid mainchain block hash>,
    confirmations     = 0,
)
```

`compute_root_from_merkle_proof(H01, 0, [H23])` = `hash(H01, H23)` = `R` = `header.block_header.merkle_root` → returns **`true`**.

`H01` is not a transaction; it is an internal node. The function returns `true` for a fabricated inclusion claim.

The `verify_transaction_inclusion_v2` fix closes this by requiring a coinbase proof of the **same length** as the tx proof and verifying it independently against the Merkle root: [4](#0-3) 

If the attacker shortens the tx proof by one level, they must also shorten the coinbase proof by one level, but then the coinbase proof would start from an internal node rather than the real coinbase txid, causing the coinbase root check to fail. The v1 endpoint has no such binding.

---

### Impact Explanation

Any downstream NEAR contract or off-chain system that calls `verify_transaction_inclusion` (v1) to gate BTC-backed logic — e.g., bridge unlocks, atomic swaps, proof-of-payment — can be fed a forged inclusion proof. The attacker needs only public knowledge of a real block's Merkle tree (all internal node hashes are computable from the public transaction list) and a NEAR account. No privileged role, no staking, no key compromise is required.

---

### Likelihood Explanation

The contract is not paused in normal production operation. The method is publicly exported in the WASM ABI. Internal node hashes for any Bitcoin block are trivially computable from the block's transaction list (publicly available on-chain). The attack requires a single NEAR view or change call with crafted Borsh-encoded arguments. Likelihood is high whenever any consumer of this contract uses the v1 endpoint.

---

### Recommendation

1. **Remove `verify_transaction_inclusion` from the public ABI.** Add `#[private]` to it, or delete the method entirely. The `#[deprecated]` annotation does not prevent external callers.
2. **Enforce all callers to use `verify_transaction_inclusion_v2`.** The coinbase depth-binding in v2 is the correct mitigation.
3. **Add a proof-depth lower bound** in `compute_root_from_merkle_proof` or in the v1/v2 wrappers to reject proofs shorter than `floor(log2(known_tx_count))`.

---

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
