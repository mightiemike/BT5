### Title
`DepositAllowlistExtension` gates by LP position `owner` instead of transaction `sender`, allowing non-allowlisted callers to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` checks the `owner` parameter (LP position owner) against the allowlist instead of the `sender` parameter (the actual caller/depositor). Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint, any non-allowlisted address can bypass the deposit allowlist by supplying an allowlisted address as `owner`.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` argument and gates on `owner`: [1](#0-0) 

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`msg.sender` here is the pool (the extension caller). The check is therefore `allowedDepositor[pool][owner]` — it never inspects who actually called the pool.

The pool passes `msg.sender` as `sender` and the user-supplied `owner` as `owner`: [2](#0-1) 

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

Critically, `addLiquidity` has **no** `msg.sender == owner` guard: [3](#0-2) 

(Compare `removeLiquidity`, which does enforce `msg.sender == owner`:) [4](#0-3) 

The sibling `SwapAllowlistExtension` correctly gates on `sender` (the actual caller): [5](#0-4) 

The inconsistency confirms the deposit extension checks the wrong actor. The contract's own NatSpec states it "Gates `addLiquidity` by depositor address" — the depositor is the caller (`sender`), not the LP position owner.

---

### Impact Explanation

A non-allowlisted address can call `pool.addLiquidity(allowlistedAddress, salt, deltas, ...)`. The extension sees `owner = allowlistedAddress` → allowlisted → passes. The non-allowlisted caller pays tokens via the swap callback and the pool's liquidity state is mutated (bin balances, `binTotals`, per-bin shares) even though the caller was never permitted to interact with the curated pool. The LP shares are minted to the allowlisted address, but the non-allowlisted address has still:

1. **Bypassed the curation policy** — the pool admin's deposit allowlist is rendered ineffective.
2. **Mutated pool state** — bin balances and `binTotals` change, which can shift the oracle-value metrics monitored by `OracleValueStopLossExtension`, potentially triggering or suppressing stop-loss guards on subsequent swaps.
3. **Forced LP positions** on allowlisted addresses without their consent, which those addresses cannot immediately undo without calling `removeLiquidity` themselves.

This is a direct admin-boundary break: an unprivileged path bypasses a pool-admin-configured access control.

---

### Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with `owner` set to any allowlisted address. No special permissions, flash loans, or oracle manipulation are needed. Any non-allowlisted address can trigger this at any time.

---

### Recommendation

Gate on `sender` (the actual caller), consistent with `SwapAllowlistExtension`:

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` configured on `BEFORE_ADD_LIQUIDITY_ORDER`.
2. Pool admin allowlists only address **B** via `setAllowedToDeposit(pool, B, true)`.
3. Non-allowlisted address **A** calls:
   ```solidity
   pool.addLiquidity(
       B,          // owner — allowlisted, passes the check
       salt,
       deltas,
       callbackData,
       extensionData
   );
   ```
4. `_beforeAddLiquidity(msg.sender=A, owner=B, ...)` is called → extension checks `allowedDepositor[pool][B]` → `true` → no revert.
5. A's callback pays the tokens; bin balances and `binTotals` are updated; LP shares are minted to B.
6. A has successfully deposited into the curated pool despite never being allowlisted, mutating pool state and bypassing the admin's curation policy.
