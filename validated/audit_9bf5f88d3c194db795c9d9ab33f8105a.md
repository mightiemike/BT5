### Title
Liquidatee Can Evade Full Liquidation by Reducing Position Before Liquidator's Transaction Executes - (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

In `ClearinghouseLiq.liquidateSubaccountImpl`, the `_assertLiquidationAmount` function performs a hard revert when `txn.amount > maxLiquidatable`. A liquidatee whose position is under maintenance health can submit a trade to reduce their position by the minimum increment (1 `sizeIncrement`) before the liquidator's transaction is sequenced. This causes the liquidator's transaction to revert with `ERR_NOT_LIQUIDATABLE_AMT`, and the liquidatee can repeat this indefinitely to delay or fully evade liquidation.

---

### Finding Description

When a liquidator calls `liquidateSubaccountImpl`, the function reads the liquidatee's current on-chain position balance to compute `maxLiquidatable`: [1](#0-0) 

For a perp position, `maxLiquidatable = perpAmount` (the current balance). For a spot position, `maxLiquidatable = spotAmount`. The function then enforces: [2](#0-1) 

If `txn.amount` (the amount the liquidator specified off-chain) exceeds `maxLiquidatable` (the current on-chain balance), the entire liquidation reverts with `ERR_NOT_LIQUIDATABLE_AMT`.

The liquidatee can exploit this by submitting a trade through `OffchainExchange.matchOrders` to reduce their position by exactly 1 `sizeIncrement` before the liquidator's transaction is processed. Because `OffchainExchange.isHealthy` unconditionally returns `true`, there is no health gate preventing an underwater subaccount from trading: [3](#0-2) 

The health checks at the end of `matchOrders` therefore never block the position-reducing trade: [4](#0-3) 

After the trade executes, the liquidatee's position is `original - 1 sizeIncrement`. The liquidator's transaction, which was built against the original position size, now fails the `_assertLiquidationAmount` check and reverts. The liquidatee can repeat this pattern for every new liquidation attempt.

---

### Impact Explanation

A liquidatee can continuously evade liquidation by racing each liquidation attempt with a minimum-size position reduction. During the delay:

- The position may become further underwater as prices move adversely, increasing the eventual bad-debt exposure for the insurance fund.
- All competing liquidators are simultaneously DoS'd, not just one, because the position size they all read off-chain is stale after the reduction.
- For large positions, the gas cost of each 1-increment reduction is negligible relative to the collateral saved by avoiding liquidation.

The corrupted protocol invariant is: **a subaccount that is under maintenance health must be liquidatable**. The liquidatee breaks this invariant by making every liquidation attempt revert while keeping the position in an unhealthy state.

---

### Likelihood Explanation

The attack is realistic because:

1. The liquidatee controls their own subaccount and can submit a trade at any time through the `Endpoint`.
2. `OffchainExchange.isHealthy` always returns `true`, so no on-chain gate prevents an underwater subaccount from trading.
3. In Nado's sequencer model, the liquidatee only needs to submit their reduction transaction before the liquidator's transaction is sequenced — not before it is broadcast to a mempool. A liquidatee monitoring their own health can react the moment their position crosses the maintenance threshold.
4. The minimum cost per evasion is one trade fee on 1 `sizeIncrement`, which is economically trivial for any position large enough to attract a liquidator.

---

### Recommendation

Replace the hard revert in `_assertLiquidationAmount` with a cap: if `txn.amount` exceeds `maxLiquidatable`, silently clamp it to `maxLiquidatable` rather than reverting. This mirrors the standard fix applied in analogous protocols and ensures the liquidator always liquidates the maximum available amount regardless of concurrent position changes.

```solidity
// Instead of:
require(maxLiquidatable >= txn.amount, ERR_NOT_LIQUIDATABLE_AMT);

// Use:
if (txn.amount > maxLiquidatable) {
    txn.amount = maxLiquidatable;
}
require(txn.amount != 0, ERR_NOT_LIQUIDATABLE_AMT);
```

The same clamping logic should be applied to the negative-amount branch (`maxLiquidatable <= txn.amount`).

---

### Proof of Concept

1. Liquidatee holds a perp position of `N` units (e.g., `100e18`) and is under maintenance health.
2. Liquidator reads the position off-chain: `txn.amount = 100e18`.
3. Liquidatee submits a reduce-only order for `1 sizeIncrement` through `OffchainExchange.matchOrders`. Because `isHealthy` always returns `true`, the trade succeeds. Position is now `100e18 - sizeIncrement`.
4. Sequencer processes the reduction before the liquidation.
5. Liquidator's `liquidateSubaccountImpl` call reaches `_assertLiquidationAmount`:
   - `maxLiquidatable = 100e18 - sizeIncrement`
   - `txn.amount = 100e18`
   - `require(maxLiquidatable >= txn.amount)` → **reverts with `ERR_NOT_LIQUIDATABLE_AMT`**
6. Liquidatee repeats step 3 for every new liquidation attempt, indefinitely delaying liquidation while the position accrues further losses. [5](#0-4) [6](#0-5) [3](#0-2)
