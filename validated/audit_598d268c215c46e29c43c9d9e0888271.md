### Title
Fork Retarget Difficulty Computed from Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/lib.rs`, `contract/src/bitcoin.rs`)

---

### Summary

`get_header_by_height` unconditionally reads from `mainchain_height_to_header`. When `get_next_work_required` calls it to fetch the retarget interval's opening block during fork-block validation, it retrieves the **mainchain's** block at that height rather than the fork's own ancestor. A trusted relayer can exploit this to have a fork's retarget block accepted with `bits` derived from the mainchain's interval-start timestamp, not the fork's, violating the invariant that difficulty must be computed from the fork's own chain history.

---

### Finding Description

**Entrypoint**

`submit_blocks` (gated by `#[trusted_relayer]`) → `submit_block_header` → `check_target` → `check_pow` → `get_next_work_required`.

**Buggy lookup**

`get_header_by_height` is implemented as:

```rust
// contract/src/lib.rs  lines 677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← always the mainchain map
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

Fork blocks are stored only in `headers_pool` via `store_fork_header`, never in `mainchain_height_to_header`:

```rust
// contract/src/lib.rs  lines 665-667
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [2](#0-1) 

**Where the wrong value is consumed**

At a retarget boundary `get_next_work_required` computes:

```rust
// contract/src/bitcoin.rs  lines 78-86
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),   // ← mainchain time, not fork time
)
``` [3](#0-2) 

`calculate_next_work_required` then computes `actual_time_taken = prev_block_time − first_block_time` and derives the new target from it: [4](#0-3) 

**Concrete attack path**

Let the mainchain tip be at height N where `(N + 1) % 2016 == 0` (one block before a retarget). `first_block_height = N − 2015`.

1. The attacker (trusted relayer) submits a fork that diverges at height M < `first_block_height` — i.e., more than 2015 blocks before the retarget boundary. All fork blocks land in `headers_pool` only.
2. The fork builds up to height N. Each non-retarget fork block is validated using `prev_block_header.block_header.bits` (the fork's own previous block), so those pass correctly.
3. The attacker submits the fork block at height N+1 (the retarget block). `check_pow` fires `get_next_work_required`.
4. `get_header_by_height(first_block_height)` returns the **mainchain's** block at that height (from `mainchain_height_to_header`), whose timestamp differs from the fork's ancestor at the same height.
5. `calculate_next_work_required` uses `mainchain_first_block_time` instead of `fork_first_block_time`. If the attacker crafts the fork's block timestamps so that `fork_prev_block_time − mainchain_first_block_time` is larger than the true fork interval, `actual_time_taken` is inflated (clamped at `pow_target_timespan × 4`), and the required target is up to **4× easier** than the network would enforce.
6. The fork block is accepted with incorrect `bits`, and if its chain work exceeds the mainchain's, a reorg is triggered.

---

### Impact Explanation

A fork block at a retarget boundary is accepted with `bits` derived from the mainchain's interval-start timestamp rather than the fork's own ancestor's timestamp. This can make the fork's required difficulty up to 4× easier than the network enforces, allowing a fork chain to be accepted with less cumulative proof-of-work than a legitimately-difficulty-adjusted chain would require. If the fork's chain work then exceeds the mainchain's, a reorg occurs, and the light client's view of the canonical chain diverges from Bitcoin consensus.

---

### Likelihood Explanation

The path requires a trusted relayer role. However, the scope explicitly includes "trusted-relayer submission path with attacker-relevant data." A compromised or malicious relayer can construct the fork entirely from crafted headers (no real mining needed for the non-retarget blocks if `skip_pow_verification` is ever toggled, or with real mining for the retarget block at reduced difficulty). The bug is structural and requires no race condition or timing dependency.

---

### Recommendation

`get_header_by_height` must be able to walk the fork's own ancestry when called during fork-block validation. The correct fix is to traverse the fork's chain backward from `prev_block_header` by exactly `difficulty_adjustment_interval − 1` steps using `get_prev_header` (which correctly reads from `headers_pool`), rather than using the height-indexed mainchain map. This mirrors how Bitcoin Core's `GetAncestor` works on the active chain index only for the active chain, and uses `pindexPrev->GetAncestor` (a parent-pointer walk) for fork validation.

---

fn get_next_work_required(
    config: &NetworkConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    if (prev_block_header.block_height + 1) % config.difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            if block_header.time
                > prev_block_header.block_header.time + 2 * config.pow_target_spacing
            {
                return config.proof_of_work_limit_bits;
            }

            let mut current_block_header = prev_block_header.clone();
            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            let last_bits = current_block_header.block_header.bits;
            return last_bits;
        }
        return prev_block_header.block_header.bits;
    }

    let first_block_height =
        prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

    let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
    calculate_next_work_required(
        config,
        prev_block_header,
        interval_tail_extend_header.block_header.time.into(),
    )
}

    config: &NetworkConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let prev_block_time: i64 = prev_block_header.block_header.time.into();

    let mut actual_time_taken: i64 = prev_block_time - first_block_time;
    if actual_time_taken < config.pow_target_timespan / 4 {
        actual_time_taken = config.pow_target_timespan / 4;

    if actual_time_taken > config.pow_target_timespan * 4 {
        actual_time_taken = config.pow_target_timespan * 4;
