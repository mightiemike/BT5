### Title
`FeeEstimation.overall_fee` silently includes L2 gas cost while `l2_gas_consumed` is absent from the serialized response — (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`tx_execution_output_to_fee_estimation` sets `overall_fee` from `receipt.fee`, which is computed by `GasVector::cost()` and includes `l2_gas * l2_gas_price`. The `FeeEstimation` struct serializes `l2_gas_price` but omits `l2_gas_consumed` (acknowledged TODO at line 104). For any v3 transaction executing in SierraGas mode (`AllResources` bounds), the RPC response for `starknet_estimateFee` / `starknet_simulateTransactions` returns an `overall_fee` that is provably larger than `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`, contradicting both the struct's own doc comment and the published OpenAPI spec.

### Finding Description

**Step 1 — `receipt.fee` includes l2_gas.**

`TransactionReceipt::from_params` computes the fee via:

```rust
tx_context.tx_info.get_fee_by_gas_vector(&block_info, gas, effective_tip)
``` [1](#0-0) 

which delegates to `GasVector::cost()`:

```rust
for (gas, price, resource) in [
    (self.l1_gas,      gas_prices.l1_gas_price,      Resource::L1Gas),
    (self.l1_data_gas, gas_prices.l1_data_gas_price,  Resource::L1DataGas),
    (self.l2_gas,      tipped_l2_gas_price,           Resource::L2Gas),  // ← included
] { ... }
``` [2](#0-1) 

So `receipt.fee = l1_gas*l1_gas_price + l1_data_gas*l1_data_gas_price + l2_gas*(l2_gas_price+tip)`.

**Step 2 — `tx_execution_output_to_fee_estimation` copies `receipt.fee` verbatim.**

```rust
Ok(FeeEstimation {
    gas_consumed:      gas_vector.l1_gas.0.into(),
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l2_gas_price,
    overall_fee: tx_execution_output.execution_info.receipt.fee,  // ← full fee
    ...
})
``` [3](#0-2) 

**Step 3 — `l2_gas_consumed` is absent; the doc comment is wrong.**

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
/// The total amount of fee. This is equal to:
/// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
pub overall_fee: Fee,
``` [4](#0-3) 

The OpenAPI spec repeats the same false invariant:

> "equals to gas_consumed\*gas_price + data_gas_consumed\*data_gas_price" [5](#0-4) 

**Step 4 — Trigger path is fully unprivileged.**

Any caller of `starknet_estimateFee` or `starknet_simulateTransactions` with a v3 `INVOKE` transaction using `AllResources` bounds (which activates `GasVectorComputationMode::All` and produces `gas.l2_gas > 0`) reaches this code path. No operator or admin privilege is required. [6](#0-5) 

### Impact Explanation

The RPC response for fee estimation is authoritative-looking but internally inconsistent: `overall_fee` cannot be reconstructed from the other fields in the response. A client that trusts the documented formula `overall_fee = gas_consumed*l1_gas_price + data_gas_consumed*l1_data_gas_price` will compute a value that is **less** than the actual `overall_fee` by exactly `l2_gas * l2_gas_price`. This is a concrete, non-zero, protocol-visible divergence for every v3 SierraGas transaction. Wallets and tooling that use this formula to display or validate fee breakdowns will show incorrect values.

### Likelihood Explanation

Every v3 `INVOKE` transaction with `AllResources` bounds (the standard for Cairo 1 contracts under Starknet ≥ 0.13/0.14) triggers this. The condition `l2_gas > 0` is the normal case, not an edge case.

### Recommendation

Add `l2_gas_consumed: Felt` to `FeeEstimation` (resolving the TODO at line 104), populate it from `gas_vector.l2_gas.0.into()`, and update the `overall_fee` doc comment and the OpenAPI `FEE_ESTIMATE` schema to reflect the three-term formula. Until then, the doc comment should at minimum be corrected to avoid misleading clients.

### Proof of Concept

```rust
// In tx_execution_output_to_fee_estimation, after construction:
let reconstructed = gas_vector.l1_gas.0 as u128 * l1_gas_price.get().0
    + gas_vector.l1_data_gas.0 as u128 * l1_data_gas_price.get().0;
// When l2_gas > 0:
assert_eq!(fee_estimation.overall_fee.0, reconstructed); // FAILS
// overall_fee.0 == reconstructed + gas_vector.l2_gas.0 as u128 * l2_gas_price.get().0
```
