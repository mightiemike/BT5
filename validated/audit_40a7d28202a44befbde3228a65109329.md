### Title
Incomplete Subaccount Settlement in `delistProduct` Permanently Locks User Collateral — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.delistProduct()` only settles perp positions for subaccounts explicitly enumerated in the admin-supplied `txn.subaccounts` array. Any subaccount omitted from that list retains an open position on the delisted product with no on-chain path to close it, and the stale price used in subsequent health calculations can prevent those users from ever withdrawing their spot collateral.

---

### Finding Description

`Clearinghouse.delistProduct()` enforces two preconditions — the product must be a perp, and the supplied price must match the current oracle price — then iterates over `txn.subaccounts` and closes each position at that price:

```solidity
// Clearinghouse.sol lines 294-325
function delistProduct(bytes calldata transaction) external onlyEndpoint {
    IEndpoint.DelistProduct memory txn = abi.decode(transaction[1:], (IEndpoint.DelistProduct));
    require(productToEngine[txn.productId] == _perpEngine(), ERR_INVALID_PRODUCT);
    require(txn.priceX18 == _getPriceX18(txn.productId), ERR_INVALID_PRICE);
    IPerpEngine perpEngine = _perpEngine();
    for (uint256 i = 0; i < txn.subaccounts.length; i++) {
        IPerpEngine.Balance memory balance = perpEngine.getBalance(txn.productId, txn.subaccounts[i]);
        int128 baseDelta = -balance.amount;
        int128 quoteDelta = -baseDelta.mul(txn.priceX18);
        perpEngine.updateBalance(txn.productId, txn.subaccounts[i], baseDelta, quoteDelta);
        ...
    }
}
``` [1](#0-0) 

There is **no on-chain invariant** that verifies every subaccount holding a non-zero balance in the delisted product is present in `txn.subaccounts`. The list is assembled off-chain by the deployer and submitted through the slow-mode queue. Any subaccount omitted from the list retains its open perp position after the product is delisted.

Once delisted, the sequencer stops submitting `UpdatePrice` transactions for that product. The health engine continues to value the position using the last stored `priceX18` from `_risk().value[productId]`:

```solidity
// BaseEngine.sol lines 162-176
function _calculateProductHealth(...) internal returns (int128 health) {
    RiskHelper.Risk memory risk = _risk(productId);
    (int128 amount, int128 quoteAmount) = _getBalance(productId, subaccount);
    int128 weight = RiskHelper._getWeightX18(risk, amount, healthType);
    health += quoteAmount;
    if (amount != 0) {
        if (weight == 2 * ONE) { return -INF; }
        health += amount.mul(weight).mul(risk.priceX18);
    }
}
``` [2](#0-1) 

The stale price is used indefinitely. If the omitted subaccount holds a perp position with negative `vQuoteBalance` (unrealized loss), the health contribution is negative. The withdrawal gate in `Clearinghouse.withdrawCollateral()` then permanently blocks the user:

```solidity
// Clearinghouse.sol line 419
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [3](#0-2) 

Because the product is no longer traded on the exchange after delisting, the user has no on-chain mechanism to close the position and restore their health. Their spot collateral is permanently locked.

---

### Impact Explanation

Any subaccount omitted from `txn.subaccounts` during `delistProduct` is left with:

1. An open perp position on a product that can no longer be traded or price-updated.
2. A health score computed from a stale price that may diverge arbitrarily from real market value.
3. No on-chain path to close the position (normal order matching requires the sequencer to route orders to the product, which stops after delisting).

If the stale-price health contribution is negative and large enough to push total health below zero, `withdrawCollateral` reverts unconditionally. The user's entire spot collateral balance — including assets deposited in unrelated products — is locked with no recovery path.

A secondary impact: if the stale price is significantly lower than the real price, the subaccount may appear under-maintenance and become liquidatable at an incorrect valuation, causing direct asset loss. [4](#0-3) 

---

### Likelihood Explanation

Medium. The deployer must enumerate all affected subaccounts off-chain before calling `delistProduct`. With a large or growing user base, omissions are operationally plausible — especially for subaccounts that opened positions close to the delisting announcement window. The 3-day slow-mode delay (`SLOW_MODE_TX_DELAY`) provides a warning window, but users who are unaware, or who cannot act within the window, are silently harmed with no on-chain recourse. [5](#0-4) 

---

### Recommendation

1. **On-chain completeness check**: Before executing the settlement loop, verify that the total open interest for the product across all provided subaccounts equals the engine's global `totalDepositsNormalized` / `totalBorrowsNormalized` for that product, or require that the product's aggregate open interest reaches zero after the loop.
2. **User self-settlement path**: Add a permissionless function that allows any subaccount to settle its own position on a delisted product at the last recorded price, analogous to the per-subaccount settlement already performed inside `delistProduct`.
3. **Guard withdrawal on delisted products**: Alternatively, exclude delisted product health contributions from the withdrawal health check, so users can at minimum recover their spot collateral even if their perp position cannot be closed.

---

### Proof of Concept

1. User A deposits USDC (spot collateral) and opens a short perp position on product P with negative `vQuoteBalance` (unrealized loss).
2. Admin queues `delistProduct` for product P via `ContractOwner.delistProduct()`, supplying a `subaccounts` array that omits User A.
3. After the 3-day slow-mode delay, the sequencer processes the transaction. User A's position is not settled; their `vQuoteBalance` remains negative.
4. The sequencer stops submitting `UpdatePrice` for product P. `_risk().value[P].priceX18` is now stale.
5. User A calls `WithdrawCollateral` for their USDC balance.
6. `Clearinghouse.withdrawCollateral()` calls `getHealth(sender, INITIAL)`, which calls `BaseEngine.getHealthContribution()`, which calls `_calculateProductHealth(P, sender, INITIAL)`.
7. The stale price and negative `vQuoteBalance` produce a negative health contribution large enough to make total health < 0.
8. The call reverts at `require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH)`.
9. User A cannot withdraw any collateral. No on-chain function exists to close the position on the delisted product. [1](#0-0) [6](#0-5)
