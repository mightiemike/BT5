### Title
Checked Arithmetic on Cumulative Funding Accumulators Causes Permanent Protocol Halt - (`File: core/contracts/PerpEngineState.sol`)

---

### Summary

`cumulativeFundingLongX18` and `cumulativeFundingShortX18` are cumulative accumulators that follow the TWAP pattern — only the *difference* between the current value and a stored snapshot matters. Under Solidity 0.8's default checked arithmetic, the plain `+=` and `-=` operations on these `int128` accumulators will revert on overflow instead of wrapping, permanently halting funding updates and all balance mutations for the affected perpetual product.

---

### Finding Description

`IPerpEngine.State` holds two cumulative funding accumulators: [1](#0-0) 

These are incremented in `updateStates()` with plain checked `+=`: [2](#0-1) 

And again in `socializeSubaccount()` with no `unchecked` block: [3](#0-2) 

The consumer of these accumulators, `_updateBalance()`, computes the funding owed as a plain checked subtraction: [4](#0-3) 

This is the canonical TWAP/TWAV accumulator pattern: accumulate indefinitely, then take the difference between the current value and a per-position snapshot (`balance.lastCumulativeFundingX18`). The design is only correct if the arithmetic is allowed to wrap — but Solidity 0.8 reverts on overflow by default, and no `unchecked` block is present at any of these sites.

The `socializeSubaccount` path is the most dangerous trigger. `fundingPerShare` is computed via `MathSD21x18.div`, which permits results up to `int128` max (~1.7×10³⁸): [5](#0-4) 

A single extreme socialization event (very small `openInterest`, large negative `vQuoteBalance`) can produce a `fundingPerShare` large enough that `cumulativeFundingLongX18 += fundingPerShare` overflows `int128` and reverts. Once this state is reached, every subsequent call to `updateStates()` and every call to `_updateBalance()` (which is invoked on every trade and liquidation) reverts for that product.

---

### Impact Explanation

If `cumulativeFundingLongX18` or `cumulativeFundingShortX18` overflows:

1. `updateStates()` reverts permanently for the affected product — funding payments can never be applied again.
2. `_updateBalance()` reverts on every call — all trades, liquidations, and PnL settlements for that product are permanently bricked.
3. Positions in the affected product are frozen: users cannot close, liquidators cannot liquidate, and the clearinghouse cannot settle PnL.

The corrupted state variable is `State.cumulativeFundingLongX18` / `State.cumulativeFundingShortX18` for the affected `productId`.

---

### Likelihood Explanation

The `updateStates()` path alone is extremely unlikely to overflow `int128` given the `MAX_DAILY_FUNDING_RATE` cap of 2%/day and the `dt < 7 * SECONDS_PER_DAY` guard: [6](#0-5) 

However, the `socializeSubaccount()` path has no equivalent rate cap. `fundingPerShare` is bounded only by `int128` max (enforced inside `MathSD21x18.div`). In a scenario where open interest has been reduced to a very small value and a large underwater position is being socialized, a single call can add a value close to `int128` max to `cumulativeFundingLongX18`. If the accumulator is already non-zero, the addition overflows and the product is permanently halted. This is a realistic scenario during extreme market stress or cascading liquidations.

---

### Recommendation

Wrap all additions and subtractions to `cumulativeFundingLongX18` and `cumulativeFundingShortX18` in `unchecked` blocks, and do the same for the `diffX18` subtraction in `_updateBalance`. This is the standard fix for TWAP-style accumulators and is exactly what Uniswap v2/v3 do:

```solidity
// In updateStates():
unchecked {
    state.cumulativeFundingLongX18 += paymentAmount;
    state.cumulativeFundingShortX18 += paymentAmount;
}

// In socializeSubaccount():
unchecked {
    state.cumulativeFundingLongX18 += fundingPerShare;
    state.cumulativeFundingShortX18 -= fundingPerShare;
}

// In _updateBalance():
unchecked {
    int128 diffX18 = cumulativeFundingAmountX18 - balance.lastCumulativeFundingX18;
}
```

The correctness of the funding calculation is preserved because only the *difference* between the current accumulator and the per-position snapshot is used, and two's-complement wrap-around cancels correctly in the subtraction.

---

### Proof of Concept

1. A perpetual product has `openInterest = 1e12` (very small, e.g., after most positions are closed) and one remaining position with `vQuoteBalance = -1.5e38` (deeply underwater after a price crash).
2. The clearinghouse calls `socializeSubaccount()` for that position.
3. `fundingPerShare = -(-1.5e38).div(1e12)` — `MathSD21x18.div` computes `(1.5e38 * 1e18) / 1e12 = 1.5e44`, which exceeds `int128` max and reverts inside `div`. Adjust: with `vQuoteBalance = -1.5e26` and `openInterest = 1e12`, `fundingPerShare = 1.5e26 * 1e18 / 1e12 = 1.5e32`.
4. If `cumulativeFundingLongX18` is already at `1.6e38` (from prior funding accumulation), then `1.6e38 + 1.5e32 ≈ 1.6e38`, which is still within range. But if `cumulativeFundingLongX18 = 1.699e38` and `fundingPerShare = 1e36`, the addition `1.699e38 + 1e36 > 1.7e38 = int128 max` reverts.
5. From this point forward, every call to `updateStates()` for this product reverts at line 131, and every call to `_updateBalance()` (trades, liquidations) reverts at line 34-35 if the diff overflows — permanently bricking the product. [7](#0-6) [8](#0-7)
