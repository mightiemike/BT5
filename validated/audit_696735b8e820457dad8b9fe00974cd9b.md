### Title
Rebasing Token Deposit Accounting Desynchronization Enables Protocol Insolvency and Fund Theft — (`core/contracts/Clearinghouse.sol`, `core/contracts/EndpointStorage.sol`)

---

### Summary

`Clearinghouse.depositCollateral` records the caller-supplied nominal `amount` as the user's internal balance without measuring the actual token balance change. If a rebasing token is configured as a spot collateral asset, a post-deposit negative rebase causes the clearinghouse's real token holdings to fall below the sum of all recorded user balances. An attacker who front-runs the rebase and withdraws first drains the clearinghouse, leaving later depositors unable to recover their full balance.

---

### Finding Description

**Deposit path — no actual-balance measurement:**

`EndpointStorage.handleDepositTransfer` transfers the caller-supplied `amount` from the user to the clearinghouse and immediately passes that same `amount` upstream: [1](#0-0) 

No before/after `balanceOf` snapshot is taken. The value that reaches `Clearinghouse.depositCollateral` is always the nominal parameter, never the actual received quantity.

`depositCollateral` then scales and records that nominal value directly into the `SpotEngine` internal ledger: [2](#0-1) 

`amountRealized` is derived purely from `txn.amount` — the number the user signed, not the number of tokens the clearinghouse actually holds.

**Solvency check is blind to actual token balances:**

`SpotEngine.assertUtilization` — the only on-chain solvency guard — compares `totalDepositsNormalized` against `totalBorrowsNormalized`, both of which are internal accounting figures: [3](#0-2) 

It never calls `token.balanceOf(clearinghouse)`. A rebase that silently shrinks the clearinghouse's real token holdings does not trigger this check.

**Withdrawal transfers the nominal recorded amount:**

`BaseWithdrawPool.handleWithdrawTransfer` sends exactly the amount the sequencer specifies: [4](#0-3) 

After a negative rebase the clearinghouse may not hold enough tokens to honour all recorded balances, yet each individual withdrawal call will succeed until the pool is drained.

---

### Impact Explanation

**Negative rebase → attacker-controlled insolvency / fund theft:**

1. Attacker and victim each deposit 1 000 rebasing tokens. Clearinghouse holds 2 000 tokens; internal ledger records 2 000 total.
2. A negative rebase halves all holder balances. Clearinghouse now holds 1 000 actual tokens; internal ledger still records 2 000.
3. Attacker submits a withdrawal for their 1 000 recorded tokens. The transfer succeeds (1 000 ≤ 1 000 available). Clearinghouse is now empty.
4. Victim's withdrawal for their 1 000 recorded tokens reverts — no tokens remain.

The attacker deposited 1 000 pre-rebase tokens (worth, say, $1 000) and withdrew 1 000 post-rebase tokens (worth $2 000 in a supply-contraction model), netting $1 000 at the victim's expense. The victim loses their entire deposit.

**Positive rebase → permanent token lock:**

Extra tokens accrued by the clearinghouse after a positive rebase are never credited to any user balance and cannot be recovered through any protocol path, permanently locking them.

---

### Likelihood Explanation

The protocol is designed to support arbitrary ERC20 spot assets configured by the owner via `SpotEngine.addOrUpdateProduct`. Any rebasing token added as a product immediately exposes all depositors to this vulnerability. The attacker needs only to monitor the rebasing token's rebase schedule (which is typically public and predictable) and submit a withdrawal transaction immediately after the rebase event. No privileged access is required; the entry path is the standard `WithdrawCollateral` signed transaction processed by the sequencer.

---

### Recommendation

Adopt a before/after balance measurement in `handleDepositTransfer`, recording only the actual increase in the clearinghouse's token balance as the credited amount:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal returns (uint256 actualReceived) {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    actualReceived = token.balanceOf(address(clearinghouse)) - before;
}
```

Pass `actualReceived` through to `depositCollateral` so the internal ledger reflects the true token quantity held. Alternatively, explicitly disallow rebasing tokens at the `addOrUpdateProduct` level with a documented invariant that only fixed-supply ERC20 tokens are supported.

### Proof of Concept

```
Setup:
  - Deploy a simplified rebasing token (AMPL-style) where balanceOf(holder)
    = gonBalance[holder] / gonsPerFragment.
  - Configure it as a spot product in SpotEngine.

Step 1 — Attacker deposits 1 000 tokens (gonsPerFragment = 1):
  endpoint.depositCollateralWithReferral(attackerSubaccount, productId, 1000, "");
  // SpotEngine records +1000 for attacker
  // clearinghouse.balanceOf = 1000

Step 2 — Victim deposits 1 000 tokens:
  endpoint.depositCollateralWithReferral(victimSubaccount, productId, 1000, "");
  // SpotEngine records +1000 for victim
  // clearinghouse.balanceOf = 2000

Step 3 — Negative rebase (gonsPerFragment = 2):
  rebasingToken.rebase(2);
  // clearinghouse.balanceOf = 1000  (halved)
  // SpotEngine internal totals unchanged: still 2000

Step 4 — Attacker withdraws 1 000 (their full recorded balance):
  // Sequencer processes WithdrawCollateral for attacker, amount = 1000
  // handleWithdrawTransfer transfers 1000 tokens → succeeds
  // clearinghouse.balanceOf = 0

Step 5 — Victim withdraws 1 000:
  // handleWithdrawTransfer attempts to transfer 1000 tokens
  // clearinghouse.balanceOf = 0 → transfer reverts
  // Victim's 1000-token balance is permanently locked
```

The attacker deposited 1 000 pre-rebase tokens and withdrew 1 000 post-rebase tokens, effectively stealing 500 tokens of real value from the victim. The `assertUtilization` check passes throughout because it never inspects `token.balanceOf(clearinghouse)`. [1](#0-0) [5](#0-4) [3](#0-2) [4](#0-3)
