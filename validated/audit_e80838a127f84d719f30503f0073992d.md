### Title
Unpermissioned `run_mainchain_gc` Permanently Deletes Blocks Within Proof Confirmation Window — (`contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` carries no caller-identity gate. Any unprivileged NEAR account can call it at any time and force the contract to delete every mainchain block older than `gc_threshold` from the tip in a single transaction. Because `verify_transaction_inclusion` only checks `args.confirmations <= gc_threshold` — not that the target block itself is within the surviving GC window — a block that has accumulated more than `gc_threshold` confirmations is silently eligible for deletion even though a caller with a low `confirmations` value would otherwise be entitled to verify it. Once deleted, the block is permanently unrecoverable, causing a permanent proof lockout.

---

### Finding Description

**Access control on `run_mainchain_gc`**

`submit_blocks` carries both `#[pause]` and `#[trusted_relayer]`, requiring the caller to be an active trusted relayer. [1](#0-0) 

`run_mainchain_gc` carries only `#[pause(except(roles(Role::UnrestrictedRunGC)))]` — no `#[trusted_relayer]` attribute. Any NEAR account may call it whenever the contract is not paused. [2](#0-1) 

**GC deletion logic**

When called, GC computes `total_amount_to_remove = amount_of_headers_we_store - gc_threshold` and removes up to `min(total_amount_to_remove, batch_size)` of the oldest mainchain blocks. Passing `u64::MAX` as `batch_size` removes all `total_amount_to_remove` blocks in one call, leaving exactly `gc_threshold` blocks (heights `tip - gc_threshold + 1` through `tip`). [3](#0-2) 

**Insufficient guard in `verify_transaction_inclusion`**

The only GC-related guard is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [4](#0-3) 

This checks the *requested* confirmation count, not the *actual distance* of the target block from the tip. A block at height `H` where `tip - H + 1 > gc_threshold` is outside the surviving GC window and will be deleted, yet the guard passes as long as `args.confirmations <= gc_threshold`.

The subsequent lookup panics with `"block does not belong to the current main chain"` if the block has been GC'd: [5](#0-4) 

---

### Impact Explanation

**Concrete scenario** (`gc_threshold = 100`):

| Step | State |
|------|-------|
| Chain has 200 blocks (heights 1–200), tip = 200 | Block at height 50 is stored; `tip - 50 + 1 = 151` confirmations |
| Legitimate user prepares `verify_transaction_inclusion` with `confirmations = 6` | `6 ≤ 100` ✓, `151 ≥ 6` ✓ — would succeed |
| Adversary calls `run_mainchain_gc(u64::MAX)` | GC removes heights 1–100; block at height 50 is permanently deleted |
| Legitimate user calls `verify_transaction_inclusion` | Panics: "block does not belong to the current main chain" |

The block cannot be re-submitted: its predecessor is also gone, and even if it could be re-added it would be immediately re-GC'd. The lockout is permanent.

The same attack applies to `verify_transaction_inclusion_v2`, which delegates to `verify_transaction_inclusion` after the coinbase proof check. [6](#0-5) 

---

### Likelihood Explanation

The attack requires no special role, no staked funds, and no knowledge beyond the contract's public ABI. Any NEAR account can execute it in a single transaction. The only prerequisite is that the chain has accumulated more than `gc_threshold` blocks since the last GC run — a normal operational condition on a live deployment.

---

### Recommendation

1. **Add a caller-identity gate to `run_mainchain_gc`**: apply `#[trusted_relayer]` (or a dedicated role check) so only authorized accounts can trigger manual GC, consistent with `submit_blocks`.

2. **Fix the confirmation window check in `verify_transaction_inclusion`**: replace the current guard with one that also verifies the target block is within the surviving GC window:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "confirmations exceeds gc_threshold"
);
// NEW: ensure the target block has not been (or will not be) GC'd
require!(
    heaviest_block_header.block_height
        .saturating_sub(target_block_height) + 1
        <= self.gc_threshold,
    "target block is outside the GC retention window"
);
```

Both fixes are needed: (1) prevents adversarial triggering; (2) closes the logical gap between the GC invariant and the proof guard.
