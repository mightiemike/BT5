### Title
`ContractCodeResponse::V2` Accepted Without Signature Verification Before `SignedContractCodeResponse` Activates — (`File: chain/client/src/stateless_validation/validate.rs`)

---

### Summary

`validate_contract_code_response` gates only the **signature check** on `ProtocolFeature::SignedContractCodeResponse`, but never drops a `ContractCodeResponse::V2` message that arrives before the feature activates. Any peer can craft a Borsh-discriminant-`1` (`V2`) response carrying arbitrary contract bytes, send it to a chunk validator in a pre-feature epoch, and have it accepted and stored without any signature verification. The analogous message types (`PartialEncodedContractDeploys`, `ChunkContractAccesses`, `VersionedPartialEncodedStateWitness`) all have an explicit version-based drop gate (`should_drop_for_version` / `witness_version_mismatch`) that rejects the wrong wire version for the epoch; `ContractCodeResponse` has no such gate.

---

### Finding Description

`ContractCodeResponse` is a versioned Borsh enum:

```rust
pub enum ContractCodeResponse {
    V1(ContractCodeResponseV1) = 0,
    V2(ContractCodeResponseV2) = 1,
}
``` [1](#0-0) 

The sender selects V1 or V2 based on the feature flag:

```rust
if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
    ContractCodeResponseV2::encode(...).map(Self::V2)
} else {
    ContractCodeResponseV1::encode(...).map(Self::V1)
}
``` [2](#0-1) 

The receiver validates the response in `validate_contract_code_response`:

```rust
pub fn validate_contract_code_response(...) -> Result<ChunkRelevance, Error> {
    let key = response.chunk_production_key();
    require_relevant!(validate_chunk_relevant(epoch_manager, key, store)?);
    let protocol_version = epoch_manager.get_epoch_protocol_version(&key.epoch_id)?;
    if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
        validate_witness_contract_code_response_signature(epoch_manager, response)?;
    }
    Ok(ChunkRelevance::Relevant)
}
``` [3](#0-2) 

When the feature is **not** enabled, the function returns `Relevant` for **any** `ContractCodeResponse` variant — including `V2` — without checking the signature. Borsh deserialization of `V2` (discriminant `1`) is unconditional and succeeds regardless of protocol version.

The handler then stores the contracts directly:

```rust
fn handle_contract_code_response(&self, response: ContractCodeResponse) -> Result<(), Error> {
    if !validate_contract_code_response(...)?. is_relevant() { return Ok(()); }
    let key = response.chunk_production_key().clone();
    let contracts = response.decompress_contracts()?;
    self.partial_witness_tracker.store_accessed_contract_codes(key, contracts)
}
``` [4](#0-3) 

**Contrast with the correct pattern used for every other versioned contract-distribution message type.** `handle_partial_encoded_contract_deploys` calls `should_drop_for_version` before any validation:

```rust
if self.should_drop_for_version(
    &partial_deploys.chunk_production_key(),
    matches!(&partial_deploys, PartialEncodedContractDeploys::V2(_)),
    "partial encoded contract deploys",
) { return Ok(()); }
``` [5](#0-4) 

`should_drop_for_version` calls `version_mismatch`, which drops a V2 message before `EarlyKickout` activates and a V1 message after:

```rust
pub(super) fn version_mismatch(version: Option<ProtocolVersion>, is_v2: bool) -> bool {
    let Some(version) = version else { return false; };
    is_v2 != ProtocolFeature::EarlyKickout.enabled(version)
}
``` [6](#0-5) 

`handle_contract_code_response` has **no equivalent gate**. The `SignedContractCodeResponse` feature controls only whether the signature is checked, not whether a V2 wire message is accepted at all.

---

### Impact Explanation

A malicious peer crafts a `ContractCodeResponse::V2` (Borsh byte `0x01` as discriminant) carrying arbitrary compressed contract bytes and any `responder` account ID, targeting a valid `ChunkProductionKey` for an in-progress chunk. Before `SignedContractCodeResponse` activates:

1. `validate_chunk_relevant` passes (the key is valid).
2. The feature gate is false → signature check is skipped entirely.
3. `store_accessed_contract_codes` stores the attacker-controlled bytes under the chunk's key.
4. When the validator later attempts state witness validation, the stored contract bytes do not match the expected code hash, causing validation to fail.
5. The validator cannot produce a valid chunk endorsement for that chunk, degrading liveness.

Because `V2` carries a `responder` field that is never verified pre-feature, the attacker does not need to impersonate any specific validator — any arbitrary account ID is accepted.

**Scope:** Chunk validation liveness. Safety (chain fork) is not directly threatened because the wrong bytes are caught by hash comparison during witness validation, but the validator is unable to recover without re-requesting the code, and the poisoned cache entry may block that recovery path.

---

### Likelihood Explanation

- Any peer connected to a chunk validator can send a `ContractCodeResponseMessage`; no validator stake or privileged role is required.
- The attack window is the epoch(s) before `SignedContractCodeResponse` stabilizes. The feature is currently listed as a nightly-only protocol feature.
- Constructing a valid `ChunkProductionKey` requires only observing the network (epoch ID, shard ID, and height are public).
- The attack is a single crafted Borsh message; no computational work is required.

---

### Recommendation

Apply the same version-based drop gate used by `handle_partial_encoded_contract_deploys` and `handle_partial_encoded_state_witness` to `handle_contract_code_response`. Concretely, before calling `validate_contract_code_response`, check whether the wire version of the response matches the epoch's protocol version and drop it if not:

```rust
fn handle_contract_code_response(&self, response: ContractCodeResponse) -> Result<(), Error> {
    // Drop V2 before SignedContractCodeResponse activates (mirrors the
    // should_drop_for_version gate on deploys and witnesses).
    let key = response.chunk_production_key();
    if let Ok(version) = self.epoch_manager.get_epoch_protocol_version(&key.epoch_id) {
        let is_v2 = matches!(response, ContractCodeResponse::V2(_));
        let expect_v2 = ProtocolFeature::SignedContractCodeResponse.enabled(version);
        if is_v2 != expect_v2 {
            return Ok(());
        }
    }
    if !validate_contract_code_response(self.epoch_manager.as_ref(), &response, self.runtime.store())?
        .is_relevant()
    {
        return Ok(());
    }
    let key = response.chunk_production_key().clone();
    let contracts = response.decompress_contracts()?;
    self.partial_witness_tracker.store_accessed_contract_codes(key, contracts)
}
```

Additionally, add a unit test analogous to `deploys_receiver_gate_drops_v2_pre_kickout` that verifies a `ContractCodeResponse::V2` arriving before `SignedContractCodeResponse` activates is silently dropped.

### Proof of Concept

**Pre-condition:** The network epoch's protocol version is below `ProtocolFeature::SignedContractCodeResponse.protocol_version()`.

**Steps:**

1. Attacker connects as a peer to a chunk validator node.
2. Attacker observes a valid `ChunkProductionKey` `(epoch_id, shard_id, height_created)` for an active chunk.
3. Attacker constructs a `ContractCodeResponse::V2` in Borsh:
   - Byte `0x01` (V2 discriminant)
   - A `ContractCodeResponseV2Inner` with the observed `next_chunk` key, any `responder` account ID (e.g., `"attacker.near"`), arbitrary compressed contract bytes, and the literal string `"ContractCodeResponseV2Inner"` as `signature_differentiator`
   - Any 64-byte ed25519 signature (e.g., all zeros)
4. Attacker sends this as a `ContractCodeResponseMessage` to the validator.
5. `validate_contract_code_response` is called:
   - `validate_chunk_relevant` returns `Relevant` (valid key).
   - `ProtocolFeature::SignedContractCodeResponse.enabled(version)` is `false` → `validate_witness_contract_code_response_signature` is **not called**.
   - Returns `Ok(ChunkRelevance::Relevant)`.
6. `store_accessed_contract_codes` stores the attacker's arbitrary bytes under the chunk's key.
7. The validator's subsequent state witness validation for that chunk fails because the stored contract bytes do not hash to the expected `CodeHash`, preventing chunk endorsement. [3](#0-2) [4](#0-3) [7](#0-6)

#[derive(Debug, Clone, PartialEq, Eq, BorshSerialize, BorshDeserialize, ProtocolSchema)]
#[borsh(use_discriminant = true)]
#[repr(u8)]
pub enum ContractCodeResponse {
    V1(ContractCodeResponseV1) = 0,
    V2(ContractCodeResponseV2) = 1,
}

impl ContractCodeResponse {
    pub fn encode(
        next_chunk: ChunkProductionKey,
        contracts: &Vec<CodeBytes>,
        signer: &ValidatorSigner,
        protocol_version: ProtocolVersion,
    ) -> std::io::Result<Self> {
        if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
            ContractCodeResponseV2::encode(next_chunk, contracts, signer).map(Self::V2)
        } else {
            ContractCodeResponseV1::encode(next_chunk, contracts).map(Self::V1)
        }
    }

    pub fn chunk_production_key(&self) -> &ChunkProductionKey {
        match self {
            Self::V1(v1) => &v1.next_chunk,
            Self::V2(v2) => &v2.inner.next_chunk,
        }
    }

    /// Account that produced this response. Available only for signed variants.
    pub fn responder(&self) -> Option<&AccountId> {
        match self {
            Self::V1(_) => None,
            Self::V2(v2) => Some(&v2.inner.responder),
        }
    }

    pub fn decompress_contracts(&self) -> std::io::Result<Vec<CodeBytes>> {
        let compressed_contracts = match self {
            Self::V1(v1) => &v1.compressed_contracts,
            Self::V2(v2) => &v2.inner.compressed_contracts,
        };
        compressed_contracts.decode().map(|(data, _size)| data)
    }

    /// Verifies the signature for signed variants. Returns `false` for
    /// unsigned variants since there is nothing to verify.
    pub fn verify_signature(&self, public_key: &PublicKey) -> bool {
        match self {
            Self::V1(_) => false,
            Self::V2(v2) => v2.verify_signature(public_key),
        }
    }
