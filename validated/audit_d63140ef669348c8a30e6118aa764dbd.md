### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `swap` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract, not the actual user. If the pool admin allowlists the router address (required for any router-mediated swap to succeed), every user on the network can bypass the per-user allowlist by routing through the public router.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` (and every other router entry point) calls `pool.swap(...)` directly, making the router the `msg.sender` to the pool: [4](#0-3) 

The router stores the actual user in transient storage for the payment callback only; it is never surfaced to the extension layer: [5](#0-4) 

This creates an irresolvable dilemma for the pool admin:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all |
| **Allowlist the router** | Every user on the network can bypass the per-user allowlist |

There is no configuration that simultaneously allows allowlisted users to trade through the router while blocking non-allowlisted users.

### Impact Explanation

Any unprivileged user can trade on a curated pool that was intended to be restricted to specific counterparties (e.g., institutional market makers, KYC-verified addresses). If the pool offers favorable oracle-anchored pricing, lower fees, or special bin configurations, unauthorized users can extract value from it. The pool admin's access-control invariant is permanently broken as long as the router is allowlisted.

### Likelihood Explanation

`MetricOmmSimpleRouter` is the standard, publicly documented entry point for swaps. Any user who discovers the pool can call `exactInputSingle` or `exactInput` with no special privileges. The bypass requires zero setup beyond a normal swap call.

### Recommendation

The `sender` argument passed to `beforeSwap` must represent the economic actor, not the intermediary. Two complementary fixes:

1. **In the router**: store the originating user in transient storage and expose it via a standardized accessor (e.g., `IMetricOmmSwapOriginator`) so extensions can read the real swapper during the hook.
2. **In `SwapAllowlistExtension`**: if the pool is called by a known router, read the originating user from the router's transient context instead of trusting `sender` directly. Alternatively, document that the extension is incompatible with router-mediated flows and enforce this at pool creation time.

### Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension (BEFORE_SWAP_ORDER = extension 1)
  admin allowlists only `allowedUser`
  admin also allowlists `router` (required for router swaps to work)

Attack:
  vm.prank(unauthorizedUser);
  router.exactInputSingle(ExactInputSingleParams({
      pool: address(pool),
      tokenIn: token0,
      ...
  }));
  // Pool calls extension.beforeSwap(sender=router, ...)
  // allowedSwapper[pool][router] == true  →  check passes
  // unauthorizedUser trades successfully on the curated pool
```

The extension's `allowedSwapper[pool][router] == true` check passes for every caller of the router, regardless of whether the actual user was ever allowlisted. [6](#0-5) [7](#0-6)
