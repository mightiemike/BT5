Audit Report

## Title
`OracleValueStopLossExtension` Stop-Loss Bypassed via Stale `lastDecayTs` When Bin Transitions from Empty to Active — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`_afterSwapOracleStopLoss` skips bins with `totalShares == 0` without updating `lastDecayTs`, freezing the decay clock for the duration of the empty period. When liquidity returns and the next swap crosses the bin, the inflated `dt` drives `_decayed` to return 0, causing `_applyWatermark` to treat any live metric as a new high and report no breach. The stop-loss is bypassed for that swap, allowing a trade at an oracle price that would otherwise have been blocked.

## Finding Description

In `_afterSwapOracleStopLoss`, the loop skips any bin whose `totalShares` is zero: [1](#0-0) 

`_checkAndUpdateWatermarks` — the only writer of `hwmS.lastDecayTs` — is therefore never reached for empty bins: [2](#0-1) 

`executeOracleStopLossHighWatermarks` sets `lastDecayTs` to `block.timestamp` at execution time: [3](#0-2) 

After the bin drains to zero, `lastDecayTs` is frozen at that admin-set timestamp. When liquidity returns and the next swap crosses the bin, `dt` spans the entire dormant period: [4](#0-3) 

`_decayed` collapses the watermark to zero when `ratePerSecondE8 * dt >= 1e8`: [5](#0-4) 

With `hwm = 0`, `_applyWatermark` always satisfies `metric >= hwm` (uint256 comparison), returning `(metric, false)` — no breach: [6](#0-5) 

The swap proceeds without triggering `OracleStopLossTriggered`, and `_checkAndUpdateWatermarks` then resets the watermark to the current live metric, so the stop-loss resumes normally only after the bypass has already occurred. The test `test_skipsEmptyBins` explicitly confirms that a bin with `shares=0` is skipped and its watermark storage is never touched: [7](#0-6) 

## Impact Explanation

The stop-loss is the primary on-chain guard preventing LP value from being extracted below the configured drawdown floor. Bypassing it for even one swap allows a trader to execute at an oracle price that the admin explicitly prohibited, constituting a direct loss of LP principal. This matches the allowed impact gate: bad-price execution reaching a pool swap and broken core pool functionality causing loss of funds.

## Likelihood Explanation

All preconditions are reachable through normal, unprivileged pool operation: a pool admin configures a non-zero `decayPerSecondE8`; LPs routinely remove all liquidity from a bin; swaps continue crossing adjacent bins while the empty bin's clock is frozen; after `ceil(1e8 / decayPerSecondE8)` seconds (≈20 days at `decayPerSecondE8 = 58`) the watermark is fully decayed; any LP re-adds liquidity; any trader then executes a swap crossing the bin. No privileged access or malicious setup is required.

## Recommendation

Inside `_afterSwapOracleStopLoss`, when a bin is detected as empty (`totalShares == 0`), still advance the decay clock without touching the watermark values:

```solidity
if (totalShares == 0) {
    highWatermarks[pool_][binIdxs[i]].lastDecayTs = uint32(block.timestamp);
    continue;
}
```

This ensures `dt` on the next active-bin check reflects only the time since the last swap traversal, not the entire dormant period, while leaving the stored watermark magnitudes unchanged.

## Proof of Concept

```
1. Deploy pool with OracleValueStopLossExtension, decayPerSecondE8 = 58.
2. Admin calls executeOracleStopLossHighWatermarks(pool, binIdx=1, hwm0=1000, hwm1=1000).
   → highWatermarks[pool][1] = {token0:1000, token1:1000, lastDecayTs: T0}
3. All LPs remove liquidity from bin 1 → totalShares[1] = 0.
4. Swaps cross bins 0 and 2 for 20+ days; bin 1 is skipped each iteration.
   → lastDecayTs for bin 1 remains T0.
5. LP adds liquidity back to bin 1 at time T0 + 1_728_000 s (20 days).
6. Attacker calls swap crossing bin 1.
   → dt = 1_728_000 s
   → factor = 58 * 1_728_000 = 100_224_000 >= 1e8
   → _decayed(1000, 58, 1_728_000) = 0
   → _applyWatermark(metricT0, 0, floorMultiplier) → (metricT0, false)
   → OracleStopLossTriggered is never emitted; swap proceeds at the unfavorable oracle price.
   → Watermark is reset to current live metric; subsequent swaps are protected.
7. LP value is extracted from bin 1 below the configured drawdown floor on that swap.
```
