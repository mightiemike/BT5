Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument forwarded from the pool, which is `msg.sender` of the `pool.swap()` call — the router contract address when users route through `MetricOmmSimpleRouter`. Any pool admin who allowlists the router to permit router-mediated swaps simultaneously opens the pool to every user on the network, completely defeating per-user access control.

## Finding Description

**Root cause — wrong identity checked by the allowlist hook**

`MetricOmmPool.swap()` calls `_beforeSwap` with `msg.sender` as the `sender` argument: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first positional argument and checks it against the per-pool allowlist using `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [2](#0-1) 

**Attack path through the router**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly: [3](#0-2) 

When this call reaches the pool, `msg.sender` is the router contract. The `sender` forwarded to `SwapAllowlistExtension.beforeSwap` is therefore the router address, not the end-user who called `exactInputSingle`.

**The forced all-or-nothing choice**

A pool admin who wants to allow *any* router-mediated swap must allowlist the router address. Once the router is allowlisted, `allowedSwapper[pool][router]` is `true`, and the extension passes for every user who routes through it — the per-user restriction is completely bypassed. There is no mechanism to allowlist the router for specific users only, because the extension never sees the actual user's address in the router flow.

The same structural problem applies to multi-hop `exactInput` paths: [4](#0-3) 

**Existing guards are insufficient**

The `allowAllSwappers` flag and `allowedSwapper` mapping both key on the immediate pool caller (`sender`), not the economic actor. There is no secondary check on `recipient` or any mechanism to decode the true originator from `extensionData`. [5](#0-4) 

## Impact Explanation

A pool configured with `SwapAllowlistExtension` and a curated allowlist (e.g., a KYC-gated pool, a private institutional pool, or a pool restricted to a single market-maker) can be freely accessed by any unprivileged user simply by calling `MetricOmmSimpleRouter.exactInputSingle` or `exactInput`. The attacker receives real token output from the pool's liquidity at oracle-anchored prices. LP providers suffer direct loss of principal because swaps they intended to restrict to trusted counterparties are executed by arbitrary actors. This meets the Sherlock High severity threshold: direct loss of user principal with no special privileges required.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point. Any pool that deploys `SwapAllowlistExtension` and also wants to support router-mediated swaps — the normal production configuration — must allowlist the router, triggering the bypass. The attacker needs no special privileges, no flash loan, and no oracle manipulation. A single standard router call suffices, and the attack is repeatable indefinitely.

## Recommendation

The extension should gate on the economically relevant actor, not the immediate pool caller. Two sound approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a trusted router convention and that the pool admin only allowlists trusted routers that enforce this encoding.
2. **Check `recipient` instead of (or in addition to) `sender`**: For swap allowlists the recipient is the party receiving value; gating on recipient prevents value extraction by unauthorized parties regardless of routing path. In `beforeSwap`, the second argument (currently ignored with `address`) is `recipient` and is already available.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured (allowAll = false).
  - Admin allowlists only address(0xALICE) as an approved swapper.
  - Admin also allowlists address(router) so that Alice can use the router.

Attack:
  1. Bob (address(0xBOB), not on the allowlist) calls:
       router.exactInputSingle({pool: pool, ..., recipient: bob})
  2. Router calls pool.swap(bob, ...) — msg.sender = router.
  3. Pool calls _beforeSwap(router, bob, ...).
  4. SwapAllowlistExtension.beforeSwap(sender=router, ...) checks
       allowedSwapper[pool][router] → true → passes.
  5. Swap executes; Bob receives token output from the restricted pool.

Result: Bob, an unprivileged address, extracts value from a pool that was
        intended to be restricted to Alice only.

Foundry test skeleton:
  - Deploy extension, pool (allowAll=false), router.
  - vm.prank(admin); extension.setAllowedToSwap(pool, alice, true);
  - vm.prank(admin); extension.setAllowedToSwap(pool, address(router), true);
  - vm.prank(bob); router.exactInputSingle(...);  // must NOT revert — demonstrates bypass.
  - Assert bob received token output despite not being on the allowlist.
```
