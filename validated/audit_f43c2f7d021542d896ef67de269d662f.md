### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling Merkle Internal-Node Forgery to Bypass SPV Proof Verification - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` is still a live, publicly callable NEAR entry point despite being marked `#[deprecated]`. It performs no depth or leaf-validity check on the supplied `tx_id`, so an unprivileged caller can supply an internal Merkle tree node as the "transaction hash" and construct a shorter proof path that still hashes to the real block's `merkle_root`. The function returns `true` for a transaction that was never included in any Bitcoin block. Any downstream NEAR contract that gates fund releases on this return value can be drained by an attacker with no privileged access.

### Finding Description

**Root cause — `verify_transaction_inclusion` is still reachable:**

The function is annotated `#[deprecated]` in Rust, which only emits a compiler warning to Rust callers. It has no effect on external NEAR callers invoking the method by name over RPC. The function remains `pub` and is gated only by `#[pause]`, which is inactive by default. [1](#0-0) 

**Root cause — no leaf-validity check in `compute_root_from_merkle_proof`:**

`merkle_tools::compute_root_from_merkle_proof` blindly walks the supplied proof path from whatever `transaction_hash` is given. It does not verify that the starting hash is a leaf (real transaction) rather than an internal node, and it does not validate proof depth against the tree size. [2](#0-1) 

**Root cause — `verify_transaction_inclusion` does not validate `tx_id` is a leaf:**

The only check before calling `compute_root_from_merkle_proof` is that the proof is non-empty. There is no assertion that `tx_id` corresponds to a real transaction rather than an internal node. [3](#0-2) 

The code's own `# Warning` comment acknowledges this: *"This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."* [4](#0-3) 

**Exploit path — concrete Merkle internal-node forgery:**

Consider a real Bitcoin block `B` already in the contract's mainchain with four transactions `[T0, T1, T2, T3]` and Merkle root `R`:

```
        R = hash(N01, N23)
       / \
    N01   N23
    / \   / \
   T0 T1 T2 T3
```

The attacker:
1. Computes `N01 = double_sha256(T0 || T1)` — public Bitcoin data.
2. Calls `verify_transaction_inclusion` with:
   - `tx_id = N01`
   - `tx_block_blockhash = B`
   - `tx_index = 0`
   - `merkle_proof = [N23]`
3. `compute_root_from_merkle_proof(N01, 0, [N23])` computes `hash(N01, N23) = R`, which equals `header.block_header.merkle_root`.
4. The function returns `true` — a forged proof for a non-existent transaction.

No privileged keys, no leaked secrets, no social engineering required. The attacker only needs knowledge of any real block's transaction set, which is public.

### Impact Explanation

The BTC light client's `verify_transaction_inclusion` is the primary SPV API consumed by downstream NEAR dApps (bridges, payment gates, cross-chain protocols) to authorize fund releases when a Bitcoin payment is "proven." A forged `true` result allows an attacker to claim funds on NEAR for a Bitcoin transaction that never occurred — the direct analog to the QuailFinance unauthorized fund withdrawal. The corrupted value is the proof result returned to the consuming contract, which treats it as a trusted authorization signal. [5](#0-4) 

### Likelihood Explanation

Likelihood is **high**. The entry point is public, requires no special role, and the attack requires only reading a real Bitcoin block's transaction list (freely available). The `#[deprecated]` marker provides zero runtime protection. Any integrator that has not migrated to `verify_transaction_inclusion_v2` is immediately exploitable.

### Recommendation

1. **Remove `verify_transaction_inclusion` entirely** or gate it with an explicit runtime panic so it cannot be called externally. A Rust `#[deprecated]` attribute does not prevent NEAR RPC invocation.
2. In `compute_root_from_merkle_proof`, enforce that the proof length matches the expected tree depth (derivable from the transaction count stored or passed in), so an internal-node input at a shallower depth is rejected.
3. Ensure all downstream integrators are migrated to `verify_transaction_inclusion_v2`, which enforces equal proof depth via the coinbase proof length check. [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
