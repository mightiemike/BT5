### Title
`ValidResourceBounds::AllResources { l2_gas: 0, l1_data_gas: 0 }` Silently Collapses to `L1Gas` Across Protobuf Boundary, Producing a Different Transaction Hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` silently converts `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` into `L1Gas(X)`. Because `get_tip_resource_bounds_hash` produces a **different Poseidon hash** for these two variants (3 resource felts vs. 2), the same transaction content yields two distinct `TransactionHash` values depending on which side of the protobuf boundary computed it. This is the sequencer-native analog of the ECDSA malleability bug: two different serialized representations of the same logical object are accepted and produce different identity values, breaking the canonicalization invariant that every transaction has exactly one hash.

---

### Finding Description

**Root cause — hash domain split in `get_tip_resource_bounds_hash`**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` branches on the `ValidResourceBounds` variant to decide how many resource felts to include in the Poseidon hash:

```
L1Gas(X)                                    → Poseidon(tip, L1_GAS_concat(X), L2_GAS_concat(0))
AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 } → Poseidon(tip, L1_GAS_concat(X), L2_GAS_concat(0), L1_DATA_GAS_concat(0))
```

The two inputs differ in length (2 vs. 3 resource felts), so the Poseidon outputs are distinct. This means `get_invoke_transaction_v3_hash` — which calls `get_tip_resource_bounds_hash` — produces hash **H_all** for `AllResources` and hash **H_l1 ≠ H_all** for `L1Gas`, even when the numeric values of all three resource bounds are identical. [1](#0-0) 

**Trigger — protobuf deserializer collapses the variant**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` applies the following rule:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [2](#0-1) 

When a transaction carrying `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is serialized to protobuf (all three fields present but l2 and l1_data are zero) and then deserialized, the condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is true and the result is `L1Gas(X)`. The variant has changed, but no error is raised.

**Serialization confirms the round-trip is lossy**

`From<ValidResourceBounds> for protobuf::ResourceBounds` always emits all three fields:

```rust
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),
    },
``` [3](#0-2) 

So `AllResources { l2_gas: 0, l1_data_gas: 0 }` → protobuf (all three fields present, two are zero) → `L1Gas`. The round-trip is not identity-preserving.

**End-to-end hash divergence path**

1. A user submits an Invoke V3 transaction via RPC with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The gateway's `convert_rpc_tx_to_internal` calls `tx_without_hash.calculate_transaction_hash(&self.chain_id)`, which calls `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `AllResources` → hash **H_all** is stored in `InternalRpcTransaction.tx_hash`. [4](#0-3) 

2. The transaction is propagated via the P2P sync/consensus protobuf protocol. The receiving node deserializes `ValidResourceBounds` as `L1Gas(X)` (variant collapse above).

3. The receiving node recomputes the transaction hash from the deserialized struct → `get_tip_resource_bounds_hash` with `L1Gas` → hash **H_l1 ≠ H_all**.

4. The `transaction_commitment` in the block header is a Merkle/Poseidon commitment over the per-transaction leaf elements, which include `transaction_hash`. [5](#0-4) 

5. The originating node commits `transaction_commitment(H_all, …)` while the syncing node commits `transaction_commitment(H_l1, …)`. These differ, so `calculate_block_hash` produces different block hashes on the two nodes. [6](#0-5) 

**Attacker-controlled input**

The `RpcInvokeTransactionV3` struct carries `resource_bounds: AllResourceBounds`. An unprivileged user can submit any combination of `l1_gas`, `l2_gas`, `l1_data_gas` values. Setting `l2_gas = 0` and `l1_data_gas = 0` while keeping `l1_gas` non-zero passes the gateway's stateless resource-bounds check (fee > 0) and the `min_gas_price` check on `l2_gas.max_price_per_unit` when `min_gas_price = 0` (a valid configuration). [7](#0-6) 

Even when the gateway rejects such transactions, a malicious or buggy sequencer can include them in a block directly; syncing nodes receive them via the protobuf sync protocol without gateway validation, triggering the hash divergence.

---

### Impact Explanation

**Critical — Wrong receipt/state from blockifier for accepted input.** The `tx_hash` embedded in the block's `transaction_commitment` differs between the proposing node and syncing nodes. The block hash computed by `calculate_block_hash` diverges, so syncing nodes reject or misidentify the block. Any RPC query for a receipt, event, or trace by `tx_hash` returns wrong or missing data on nodes that deserialized via protobuf.

**High — Transaction conversion binds the wrong hash.** The `InternalRpcTransaction.tx_hash` computed at the gateway is `H_all`, but after a protobuf round-trip the same transaction is identified by `H_l1`. Mempool deduplication, consensus proposal matching, and block-level receipt indexing all use `tx_hash` as the canonical key; the two values refer to the same transaction but are treated as distinct objects.

---

### Likelihood Explanation

The condition `l2_gas = 0 && l1_data_gas = 0` is reachable by any user who submits an Invoke V3 transaction with only L1 gas bounds set (a common pattern for pre-0.13.3 style transactions submitted through the new API). The protobuf sync path is exercised on every block sync between nodes. The only mitigating factor is that the gateway's `min_gas_price` check on `l2_gas.max_price_per_unit` may reject such transactions when `min_gas_price > 0`; however, this is a configuration-dependent gate, not a protocol-level invariant, and the sync path bypasses it entirely.

---

### Recommendation

**Option A (preferred — fix the deserializer).** In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not collapse to `L1Gas` when all three fields are explicitly present in the protobuf message. Reserve `L1Gas` only for the legacy case where `l1_data_gas` is absent (`None`), which is the original intent of the `TODO(Shahak)` comment:

```rust
// Only collapse to L1Gas when l1_data_gas is absent (legacy 0.13.2 wire format).
let l1_data_gas = match value.l1_data_gas {
    None => {
        // Legacy: treat as L1Gas.
        return Ok(ValidResourceBounds::L1Gas(l1_gas));
    }
    Some(v) => ResourceBounds::try_from(v)?,
};
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

**Option B — fix the hash function.** Make `get_tip_resource_bounds_hash` always emit all three resource felts regardless of variant, using zeros for absent fields. This makes the hash canonical across both variants when the numeric values are equal.

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds,
};
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
use starknet_api::transaction::fields::Tip;

let l1_gas = ResourceBounds { max_amount: 1000.into(), max_price_per_unit: 1.into() };

// Representation 1: AllResources with zero l2 and l1_data
let all_resources = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas,
    l2_gas: ResourceBounds::default(),    // zero
    l1_data_gas: ResourceBounds::default(), // zero
});

// Representation 2: L1Gas — produced by protobuf round-trip of Representation 1
let l1_gas_only = ValidResourceBounds::L1Gas(l1_gas);

let tip = Tip(0);
let hash_all = get_tip_resource_bounds_hash(&all_resources, &tip).unwrap();
let hash_l1  = get_tip_resource_bounds_hash(&l1_gas_only,  &tip).unwrap();

// These are DIFFERENT — same economic content, different transaction hashes
assert_ne!(hash_all, hash_l1,
    "AllResources{{l2=0,l1_data=0}} and L1Gas produce the same hash — invariant holds");
// The assert_ne passes: the hashes diverge, confirming the vulnerability.
```

The protobuf round-trip that triggers the collapse:

```rust
// Serialize AllResources { l2_gas: 0, l1_data_gas: 0 } → protobuf
let proto: protobuf::ResourceBounds = all_resources.into();
// proto.l1_data_gas = Some(zeros), proto.l2_gas = Some(zeros)

// Deserialize → L1Gas  (the collapse)
let roundtripped = ValidResourceBounds::try_from(proto).unwrap();
assert!(matches!(roundtripped, ValidResourceBounds::L1Gas(_)));
// Hash computed from roundtripped == hash_l1 ≠ hash_all

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
