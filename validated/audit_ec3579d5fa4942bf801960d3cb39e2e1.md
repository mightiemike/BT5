### Title
`ValidResourceBounds` Protobuf Deserialization Silently Downgrades `AllResources` to `L1Gas` Variant, Producing a Different Transaction Hash Preimage for Declare V3 Transactions with Zero L2/L1-Data Gas Bounds — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to decide between the `L1Gas` and `AllResources` variants. A Declare V3 transaction submitted via the gateway with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` has its hash computed over **three** resource-bound entries. When the same transaction is later received by a peer via P2P state sync and deserialized through `DeclareTransactionV3Common`, the same wire bytes produce `ValidResourceBounds::L1Gas(X)`, and the hash is recomputed over **two** resource-bound entries. The two Poseidon hashes are structurally different, breaking hash canonicalization across the gateway→storage→P2P boundary.

---

### Finding Description

**Step 1 — Gateway path always uses `AllResources`.**

`InternalRpcDeclareTransactionV3` stores `resource_bounds: AllResourceBounds` (never `ValidResourceBounds`). Its `DeclareTransactionV3Trait` implementation wraps it unconditionally:

```rust
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
``` [1](#0-0) 

The transaction hash is therefore always computed with the `AllResources` branch of `get_tip_resource_bounds_hash`, which includes the `L1_DATA_GAS` packed field:

```rust
ValidResourceBounds::AllResources(all_resources) => {
    vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
}
``` [2](#0-1) 

So for `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` the fee-fields hash is:

```
poseidon(tip, pack(L1_GAS, X), pack(L2_GAS, 0), pack(L1_DATA_GAS, 0))   // 4 elements
```

**Step 2 — P2P state-sync path may produce `L1Gas`.**

When a peer deserializes the same transaction from protobuf, it goes through `TryFrom<protobuf::DeclareV3Common> for DeclareTransactionV3Common`, which calls:

```rust
let resource_bounds = ValidResourceBounds::try_from(
    value.resource_bounds.ok_or(missing("DeclareV3Common::resource_bounds"))?,
)?;
``` [3](#0-2) 

That converter applies a value-based heuristic:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [4](#0-3) 

For the transaction above (`l2_gas=0, l1_data_gas=0`), the result is `ValidResourceBounds::L1Gas(X)`.

**Step 3 — `L1Gas` variant omits `L1_DATA_GAS` from the hash preimage.**

```rust
ValidResourceBounds::L1Gas(_) => vec![],   // no L1_DATA_GAS entry
``` [2](#0-1) 

So the fee-fields hash recomputed on the receiving peer is:

```
poseidon(tip, pack(L1_GAS, X), pack(L2_GAS, 0))   // 3 elements
```

**Step 4 — Divergent values.**

| Side | Variant | Fee-fields hash input length | Hash value |
|---|---|---|---|
| Gateway (submitter) | `AllResources` | 4 elements | `H_all` |
| P2P peer (state sync) | `L1Gas` | 3 elements | `H_l1` ≠ `H_all` |

Because `get_invoke_transaction_v3_hash` / `get_declare_transaction_v3_hash` chain the fee-fields hash into the full transaction hash, the full transaction hash also diverges. [5](#0-4) 

**Step 5 — The developers are aware of the symptom but have not fixed the root cause.**

The consensus test explicitly works around it by injecting a non-zero L2 gas amount before serialization:

```rust
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    ...
    resource_bounds.l2_gas.max_amount = GasAmount(1);
``` [6](#0-5) 

The workaround is absent from the production state-sync deserialization path.

---

### Impact Explanation

A Declare V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` (valid per the gateway's stateless validator, which accepts any combination of non-zero resource bounds) is committed to a block with hash `H_all`. When a syncing peer deserializes the block from protobuf, it recomputes hash `H_l1 ≠ H_all`. Any hash-equality check — transaction commitment verification, deduplication lookup, or `validate_transaction_hash` — will fail, causing the peer to reject a valid block or misidentify the transaction. This is a transaction-conversion boundary where the wrong hash is bound to the executable payload.

**Matching impact**: *High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.*

---

### Likelihood Explanation

The trigger is a legitimately valid Declare V3 transaction with zero L2 and L1-data gas bounds. Such transactions are accepted by the gateway today (the stateless validator only requires at least one non-zero bound). Any user or attacker can submit one. The protobuf round-trip silently changes the variant without any error, making the divergence invisible until hash verification fails downstream.

---

### Recommendation

Replace the value-based heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit version/type tag carried in the protobuf message, or always deserialize into `AllResources` when all three fields are present on the wire (matching the serialization side, which always emits all three fields for `L1Gas` too):

```rust
// Serialization (already emits all three fields for L1Gas):
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),       // zero
    l1_data_gas: Some(ResourceBounds::default().into()), // zero
},
``` [7](#0-6) 

Since the serializer always emits all three fields, the deserializer should always produce `AllResources` when all three fields are present, preserving the original variant. Alternatively, carry an explicit `is_all_resources: bool` flag in the protobuf message.

---

### Proof of Concept

```
1. Submit Declare V3 tx to gateway:
   resource_bounds = AllResourceBounds { l1_gas: {max_amount:1, max_price:1}, l2_gas: {0,0}, l1_data_gas: {0,0} }

2. Gateway computes:
   fee_hash_A = poseidon(tip=0, pack(L1_GAS,1,1), pack(L2_GAS,0,0), pack(L1_DATA_GAS,0,0))
   tx_hash_A  = poseidon(DECLARE, version, sender, fee_hash_A, ...)

3. Block committed with tx_hash_A.

4. Peer receives block via P2P, deserializes DeclareV3Common:
   l2_gas.is_zero() && l1_data_gas.is_zero()  →  ValidResourceBounds::L1Gas({1,1})

5. Peer computes:
   fee_hash_B = poseidon(tip=0, pack(L1_GAS,1,1), pack(L2_GAS,0,0))   // only 3 elements
   tx_hash_B  = poseidon(DECLARE, version, sender, fee_hash_B, ...)

6. tx_hash_B ≠ tx_hash_A  →  hash verification fails, block rejected.
```
