### Title
Odd-Width Merkle Tree Duplicate-Sibling Allows Phantom Index Verification — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`)

---

### Summary

`compute_root_from_merkle_proof` accepts any caller-supplied `tx_index` without bounding it against the actual transaction count stored in the block. In Bitcoin's Merkle tree construction, when the leaf count is odd the last leaf is duplicated. This means the same sibling sequence verifies two distinct index values (`N-1` and `N`) for the same `tx_id`, causing `verify_transaction_inclusion_v2` to return `true` for a phantom transaction position that does not exist in the block.

---

### Finding Description

**Merkle tree construction (Bitcoin protocol):** For a block with an odd number of transactions `[T0, T1, T2]`, the tree is built by duplicating the last leaf:

```
Level 0 (leaves): [T0, T1, T2, T2*]   (* = implicit duplicate)
Level 1:          [H(T0,T1), H(T2,T2)]
Root:             H(H(T0,T1), H(T2,T2))
```

**`compute_root_from_merkle_proof` trace for `tx_id=T2, tx_index=2` (real):** [1](#0-0) 

- `position=2` (even) → `hash = H(T2, proof[0])` where `proof[0]=T2` → `H(T2,T2)`, `position=1`
- `position=1` (odd) → `hash = H(proof[1], H(T2,T2))` where `proof[1]=H(T0,T1)` → root ✓

**Same proof, `tx_id=T2, tx_index=3` (phantom):**

- `position=3` (odd) → `hash = H(proof[0], T2)` where `proof[0]=T2` → `H(T2,T2)`, `position=1`
- `position=1` (odd) → `hash = H(proof[1], H(T2,T2))` → root ✓

The proof `[T2, H(T0,T1)]` is valid for **both** `(T2, index=2)` and `(T2, index=3)`. Index 3 is a phantom — it does not correspond to any real transaction in the block.

**`verify_transaction_inclusion_v2` does not close this gap.** The coinbase check only validates that `coinbase_tx_id` at index `0` hashes to the stored merkle root: [2](#0-1) 

It does not validate that `tx_index` is within `[0, actual_tx_count)`. The contract stores only the merkle root in `LightHeader`, never the transaction count: [3](#0-2) 

After the coinbase check passes, execution falls through to `verify_transaction_inclusion` which calls `compute_root_from_merkle_proof` with the attacker-supplied `tx_index`: [4](#0-3) 

`compute_root_from_merkle_proof` performs no bounds check on `transaction_position`: [1](#0-0) 

`ProofArgsV2.tx_index` is fully attacker-controlled with no server-side range validation: [5](#0-4) 

---

### Impact Explanation

Any downstream bridge, unlock, mint, or withdrawal contract that calls `verify_transaction_inclusion_v2` and uses `(tx_id, tx_index)` as its deduplication key will accept the phantom proof as a second, independent inclusion event for the same transaction. This allows an attacker to trigger a second payout/unlock for a single on-chain Bitcoin event — a direct economic replay with Critical impact.

Even systems that deduplicate on `tx_id` alone are exposed to a weaker form: the light client's stated invariant ("returns `true` only for transactions that exist at the claimed position") is broken, which undermines the security model for any consumer that relies on the index for routing or accounting.

---

### Likelihood Explanation

- Odd transaction counts are the norm in Bitcoin blocks (roughly 50% of all blocks).
- The attack requires no special privileges: `verify_transaction_inclusion_v2` is a public, unpermissioned view call.
- The attacker only needs a real block already accepted by the light client, a real `coinbase_tx_id` with a valid coinbase proof, and the last transaction hash as the repeated sibling.
- No relayer compromise, key leak, or DAO role is required.

---

### Recommendation

The contract does not store transaction counts, so a pure on-chain bounds check is not directly possible without a protocol change. Mitigations include:

1. **Require `tx_index < merkle_proof.len()^2` upper-bound heuristic** — reject any index that implies more leaves than the proof depth can represent for a non-padded tree (i.e., `tx_index < 2^proof_len - 1` is necessary but not sufficient).
2. **Require the caller to supply the transaction count** and verify it is consistent with the proof length and index, then check `tx_index < tx_count`.
3. **Document the invariant gap** and require all downstream consumers to deduplicate strictly on `tx_id`, never on `(tx_id, tx_index)`.

---

### Proof of Concept

```
Block: 3 transactions [coinbase=C, T1, T2]
Merkle tree (Bitcoin): [C, T1, T2, T2]  (T2 duplicated)
  Level 1: [H(C,T1), H(T2,T2)]
  Root R = H(H(C,T1), H(T2,T2))

Coinbase proof (valid, index=0):
  coinbase_tx_id = C
  coinbase_merkle_proof = [T1, H(T2,T2)]
  → compute_root_from_merkle_proof(C, 0, [T1, H(T2,T2)]) == R  ✓

Phantom proof (attacker-supplied, index=3):
  tx_id = T2
  tx_index = 3
  merkle_proof = [T2, H(C,T1)]
  → position=3 (odd): hash = H(T2, T2), position=1
  → position=1 (odd): hash = H(H(C,T1), H(T2,T2)) = R  ✓

verify_transaction_inclusion_v2 returns true for (T2, index=3)
— a position that does not exist in the block.
```

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
