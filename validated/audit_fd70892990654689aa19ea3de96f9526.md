### Title
Fork Retarget Ancestor Confusion: `get_header_by_height` Reads Mainchain Instead of Fork Lineage — (`contract/src/litecoin.rs`, `contract/src/lib.rs`)

---

### Summary

When validating a fork block that falls on a Litecoin retarget boundary, `get_next_work_required` calls `get_header_by_height(first_block_height)` to obtain the interval-start block's timestamp. That function unconditionally reads from `mainchain_height_to_header` — the mainchain index — never from the fork's own ancestor chain. An attacker who controls fork headers can craft a fork whose ancestor at `first_block_height` carries a timestamp that differs from the mainchain block at that height, causing the contract to accept a `bits` value the real Litecoin network would reject.

---

### Finding Description

**Root cause — `get_header_by_height` is mainchain-only:** [1](#0-0) 

The function exclusively queries `mainchain_height_to_header`. Fork blocks are stored only in `headers_pool` via `store_fork_header`, which never writes to the height index: [2](#0-1) 

So for any fork block, `get_header_by_height(first_block_height)` silently returns the **mainchain** block at that height, not the fork's ancestor.

**Where this is called during fork validation:** [3](#0-2) 

`first_block_height` is computed as `prev_block_header.block_height - blocks_to_go_back` (2016 for Litecoin mainnet after the first retarget). If the fork diverges at or before that height, the fork's block at `first_block_height` is a fork block — invisible to `get_header_by_height` — so the mainchain block's timestamp is used instead.

**The same bug is acknowledged in `dogecoin.rs`:** [4](#0-3) 

The TODO comment explicitly flags that reading by height from the mainchain without walking the fork ancestor is potentially incorrect.

**Call chain:**

`submit_blocks` → `submit_block_header` → `check_target` → `check_pow` → `get_next_work_required` → `get_header_by_height` [5](#0-4) [6](#0-5) 

---

### Impact Explanation

The contract stores a fork whose retarget `bits` value is computed from the wrong ancestor timestamp. Specifically:

- `actual_time_taken = prev_block_time − first_block_time` uses the **mainchain** `first_block_time`.
- If the fork's ancestor at `first_block_height` carries a later timestamp than the mainchain block (which the attacker controls), the contract sees a larger `actual_time_taken`, yielding a higher target (easier difficulty).
- `calculate_next_work_required` clamps the ratio to `[1/4, 4×]`, so the maximum difficulty reduction is **4×**. [7](#0-6) 

The contract accepts the fork block; the real Litecoin network would reject it because it uses the fork's actual ancestor timestamp and would compute a harder `bits`. A downstream bridge that trusts the contract's mainchain state would treat confirmations built on this invalid fork as final.

---

### Likelihood Explanation

**Barriers:**

1. `submit_blocks` is gated by `#[trusted_relayer]` (line 168), so an attacker cannot call it directly without being a registered relayer or having `Role::UnrestrictedSubmitBlocks`. [8](#0-7) 

2. The fork must diverge at or before `first_block_height`, requiring the attacker to mine ≥ 2016 Litecoin blocks with valid scrypt PoW at current mainnet difficulty before the retarget block — substantial but not impossible for a well-resourced miner.

3. The timestamp manipulation is bounded by `MAX_FUTURE_BLOCK_TIME_LOCAL` (2 hours), limiting the achievable difficulty reduction in practice.

**Realistic path:** The attacker creates a fork on the actual Litecoin P2P network with crafted headers (valid scrypt PoW for the claimed `bits`). The legitimate relayer, which monitors the Litecoin network and submits headers it observes, picks up the fork and calls `submit_blocks`. The relayer does not independently re-validate `bits` against the protocol — that is the contract's job — so the malformed fork reaches the contract.

---

### Recommendation

Replace the height-based lookup with a parent-pointer walk along the fork's own lineage. At retarget time, walk backwards from `prev_block_header` using `get_prev_header` for exactly `blocks_to_go_back` steps to obtain the true interval-start block, regardless of whether those blocks are on the mainchain or a fork:

```rust
// Instead of:
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);

// Walk the fork's own ancestor chain:
let mut cursor = prev_block_header.clone();
for _ in 0..blocks_to_go_back {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
// cursor is now the true interval-start ancestor on this fork
calculate_next_work_required(config, prev_block_header, cursor.block_header.time.into())
```

This matches the reference Litecoin implementation, which walks `pindexFirst` backwards via `pprev` pointers — always following the candidate chain, never a height index.

### Proof of Concept

**Setup:** Litecoin mainnet contract initialized at a retarget boundary. Mainchain is at height `N` (a multiple of 2016). `first_block_height = N − 2016`.

1. **Mine fork divergence.** Starting from the mainchain block at height `N − 2017`, mine an alternative block `F_{N−2016}` with timestamp `T_fork = T_mainchain_at_(N−2016) + 7000 s` (within the 2-hour future-time window). This block has valid scrypt PoW at the current difficulty.

2. **Extend the fork.** Mine fork blocks `F_{N−2015}` through `F_{N−1}`, each inheriting the previous block's `bits` (no retarget). All have valid scrypt PoW.

3. **Compute the crafted `bits`.** The contract will call `get_header_by_height(N − 2016)` and receive the **mainchain** block with timestamp `T_mainchain`. Compute `actual_time_taken = T_{N−1} − T_mainchain` (larger than the real value by 7000 s). Derive the resulting `new_target` and encode it as `bits_crafted`.

4. **Mine the retarget block.** Mine `F_N` with `bits = bits_crafted` and valid scrypt PoW for that (easier) target.

5. **Submit via relayer.** Broadcast the fork to the Litecoin P2P network. The relayer detects the fork and calls `submit_blocks([F_{N−2016}, ..., F_N])`. The contract validates each block; at `F_N` it calls `get_header_by_height(N − 2016)`, gets the mainchain block's timestamp, computes `bits_crafted`, and accepts the block.

6. **Verify the discrepancy.** Query a reference Litecoin node: it computes `bits` using `F_{N−2016}.time = T_fork`, producing a harder target. The contract accepted a `bits` value the real network rejects.

7. **Trigger reorg (if desired).** Continue mining fork blocks at the easier post-retarget difficulty. Once the fork's `chain_work` exceeds the mainchain's, `reorg_chain` promotes the invalid fork to mainchain. Any bridge querying `verify_transaction_inclusion` against a block on this fork receives a `true` result for a transaction the real Litecoin network never confirmed. [9](#0-8)

fn calculate_next_work_required(
    config: &NetworkConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let prev_block_time: i64 = prev_block_header.block_header.time.into();

    let mut actual_time_taken: i64 = prev_block_time - first_block_time;
    if actual_time_taken < config.pow_target_timespan / 4 {
        actual_time_taken = config.pow_target_timespan / 4;
    }
    if actual_time_taken > config.pow_target_timespan * 4 {
        actual_time_taken = config.pow_target_timespan * 4;
    }

    let mut new_target = target_from_bits(prev_block_header.block_header.bits);

    let shift: bool = new_target.bits() > config.pow_limit.bits() - 1;
    if shift {
        new_target = new_target >> 1;
    }

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(actual_time_taken).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target = new_target
        / U256::from(<i64 as TryInto<u64>>::try_into(config.pow_target_timespan).unwrap());

    if shift {
        new_target = new_target << 1;
    }

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
