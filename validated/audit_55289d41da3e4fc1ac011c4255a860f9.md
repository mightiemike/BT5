### Title
Swap Allowlist Bypassed for All Router-Mediated Swaps — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[pool][sender]`. The pool passes `msg.sender` as `sender`, which is the **router contract address** when a user swaps via `MetricOmmSimpleRouter`. The hook therefore checks whether the router is allowlisted, not the original user. If the router is allowlisted (the only way for allowlisted users to use it), every user — including non-allowlisted ones — can bypass the gate by routing through the router.

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the hook.**

In `MetricOmmPool.swap`, the hook dispatcher is called with `msg.sender`: [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards that value unchanged.** [2](#0-1) 

**Step 3 — The extension checks `allowedSwapper[pool][sender]` where `sender` is the router.**

```
allowedSwapper[msg.sender][sender]
              ^^^^^^^^^^^  ^^^^^^
              pool addr    router addr  ← NOT the original EOA
``` [3](#0-2) 

**Step 4 — The router calls `pool.swap()` directly, making itself `msg.sender` at the pool.** [4](#0-3) 

The same applies to `exactOutputSingle` and both multi-hop variants (`exactInput`, `exactOutput`). [5](#0-4) 

**Resulting states:**

| Router allowlisted? | Allowlisted user via router | Non-allowlisted user via router |
|---|---|---|
| Yes | Passes (correct) | **Passes — bypass** |
| No | **Reverts — broken** | Reverts |

There is no configuration in which the allowlist correctly gates individual users through the router.

### Impact Explanation

A pool admin who restricts swaps to a specific set of addresses (e.g., KYC'd counterparties, whitelisted market makers) must also allowlist the router for those users to access it. Once the router is allowlisted, any unprivileged address can call `exactInputSingle` / `exactOutputSingle` / `exactInput` / `exactOutput` and the `beforeSwap` hook will pass because it sees the router — not the caller — as the swapper. The allowlist protection is completely nullified for all router-mediated swaps.

### Likelihood Explanation

The router is the standard public entrypoint for swaps. Any pool that uses `SwapAllowlistExtension` and also permits router access is immediately vulnerable. No special timing, oracle manipulation, or two-transaction sequence is required — a single `exactInputSingle` call from any address suffices.

### Recommendation

The extension must check the **original initiator**, not the immediate caller of `pool.swap`. Options:

1. Pass the original `msg.sender` through the router as an explicit parameter in `extensionData`, and have the extension decode and verify it (requires a trusted router identity check inside the extension).
2. Have the pool record the original initiator in transient storage before calling extensions, and expose it to extensions via a pool read.
3. Require that the pool admin allowlist individual users directly and never allowlist the router contract itself — but this must be enforced and documented, as it breaks router usability.

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension.
2. Admin calls setAllowedToSwap(pool, router, true)  // to let allowlisted users use the router
3. Non-allowlisted EOA `attacker` calls:
     router.exactInputSingle({pool: pool, ...})
4. Pool calls _beforeSwap(msg.sender=router, ...)
5. Extension checks allowedSwapper[pool][router] → true → passes
6. Attacker's swap executes despite not being on the allowlist.
```

The "watermark boundary" framing in the question is a red herring — no oracle math or boundary condition is involved. The bypass is structural and works on every router call unconditionally.
