### Title
Linked Signer Can Hijack Subaccount Signing Authority via `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.sol`, the sequencer-submitted `LinkSigner` transaction is validated with `allowLinkedSigner = true`, meaning the **current linked signer** can sign a `LinkSigner` transaction to replace itself with any arbitrary address. The subaccount owner never authorized this delegation. The slow-mode path for the same operation correctly restricts it to the subaccount owner, making this an inconsistent and exploitable privilege escalation.

---

### Finding Description

In `EndpointTx.sol`, the fast (sequencer-submitted) path for `TransactionType.LinkSigner` calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which — when `allowLinkedSigner = true` — passes `getLinkedSigner(sender)` to the verifier as an accepted signer: [2](#0-1) 

This means the verifier accepts a signature from **either** the subaccount owner **or** the current linked signer to authorize a `LinkSigner` mutation. After validation, the mapping is unconditionally overwritten: [3](#0-2) 

By contrast, the slow-mode path for the same `LinkSigner` transaction uses `validateSender`, which enforces that only the Ethereum address matching the subaccount (i.e., the owner) can submit it: [4](#0-3) 

The `validateSender` check requires `address(uint160(bytes20(txn.sender))) == sender`, which is the subaccount owner's address — not the linked signer's address. The two paths are inconsistent: the slow-mode path correctly restricts `LinkSigner` to the owner, while the fast path allows the linked signer to perform the same mutation. [5](#0-4) 

---

### Impact Explanation

A linked signer (lower-privilege role) can replace itself — or any existing linked signer — with an attacker-controlled address, without the subaccount owner's knowledge or consent. Once the new address is installed as the linked signer, it can sign any transaction the protocol accepts from a linked signer: trades, withdrawals (`WithdrawCollateral`), quote transfers (`TransferQuote`), and NLP operations. This constitutes unauthorized subaccount mutation leading to direct asset loss for the victim.

The `linkedSigners` mapping is the sole on-chain record of signing authority delegation: [6](#0-5) 

Corrupting it permanently grants an attacker signing authority over the victim's subaccount.

---

### Likelihood Explanation

Any address that has ever been granted linked-signer status for a subaccount can exploit this. The attacker does not need to compromise the owner's key — they only need to have been trusted as a linked signer at any point. The sequencer will include a validly signed `LinkSigner` transaction in a batch without any additional checks. The owner has no on-chain mechanism to detect or prevent this before the batch is finalized.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in the fast path, consistent with the slow-mode path which already enforces owner-only authorization:

```solidity
// EndpointTx.sol — fast path for LinkSigner
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // was: true — linked signer must NOT be able to change the linked signer
);
```

Only the subaccount owner (the address whose private key corresponds to the subaccount bytes32) should be permitted to authorize a `LinkSigner` mutation.

---

### Proof of Concept

1. Alice owns subaccount `alice_subaccount` and sets Bob (`0xBob`) as her linked signer via a legitimate `LinkSigner` transaction.
2. Bob constructs a new `LinkSigner` transaction: `{ sender: alice_subaccount, signer: 0xAttacker, nonce: N }`.
3. Bob signs this transaction with his own key (`0xBob`).
4. Bob submits the signed transaction to the sequencer (or waits for the sequencer to include it).
5. In `EndpointTx.sol`, `validateSignedTx(..., allowLinkedSigner: true)` accepts Bob's signature because `getLinkedSigner(alice_subaccount) == 0xBob`.
6. `linkedSigners[alice_subaccount]` is overwritten to `0xAttacker`.
7. `0xAttacker` can now sign `WithdrawCollateral` or `TransferQuote` transactions on behalf of Alice's subaccount, draining her collateral. [1](#0-0) [7](#0-6)
