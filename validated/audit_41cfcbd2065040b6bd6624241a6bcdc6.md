Audit Report

## Title
`SwapAllowlistExtension` Evaluates Router Address as Swapper Identity, Making Per-User Allowlist Enforcement Impossible Through `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the direct caller) as the `sender` argument to every extension hook. When swaps are routed through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. No configuration exists that simultaneously permits allowlisted users to use the router and blocks non-allowlisted users from doing the same.

## Finding Description
**Root cause — pool passes `msg.sender` (the router) as `sender`:**

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

**`SwapAllowlistExtension` keys its allowlist on that `sender`:**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()` — the router, not the originating user.

**The router never forwards the original user's address:**

`exactInputSingle` calls `pool.swap()` directly with no user identity propagation: [4](#0-3) 

`exactInput` does the same for every hop: [5](#0-4) 

`exactOutputSingle` and `exactOutput` follow the same pattern: [6](#0-5) [7](#0-6) 

**The two broken states:**

| Pool admin decision | Consequence |
|---|---|
| Allowlist the router | Every user on the network bypasses the allowlist via the router |
| Do not allowlist the router | Individually allowlisted users cannot use the router at all |

There is no configuration that simultaneously (a) permits allowlisted users to use the router and (b) blocks non-allowlisted users from doing the same. The wrong value is `allowedSwapper[pool][router]` being evaluated in place of `allowedSwapper[pool][user]`.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of addresses (e.g., KYC'd counterparties, protocol-owned bots, or whitelisted market makers) loses that restriction entirely for any user who calls `MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput`. The attacker receives the same execution as an allowlisted user — consuming LP liquidity at oracle-anchored prices — without ever being added to the allowlist. This is a direct loss of the pool's access-control invariant with fund-impacting consequences for LP providers on restricted pools, constituting broken core pool functionality causing loss of funds.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any user who encounters a revert on direct `pool.swap()` due to the allowlist will naturally try the router next. No privileged access, no special token, and no admin cooperation is required. The bypass is a single public function call, repeatable by any address.

## Recommendation
Two sound mitigations exist:

1. **Authenticated `originSender` via `extensionData`:** Extend the router to encode and sign the originating user's address in `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and verify it. This keeps the check in the extension layer without changing the pool interface.

2. **Gate at the router level:** Deploy a router variant that enforces the allowlist before calling `pool.swap()`, and configure the pool to only accept calls from that trusted router (checked in `beforeSwap` via `msg.sender` alone, without relying on `sender` for user identity).

Until one of these is applied, `SwapAllowlistExtension` should document that it **only gates direct `pool.swap()` calls** and is incompatible with `MetricOmmSimpleRouter`.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (required so that allowlisted users can use the router)
  - Alice (address not in allowlist) calls:
      router.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(recipient, ...)
  - Pool calls _beforeSwap(msg.sender=router, ...)
  - Extension evaluates: allowedSwapper[pool][router] == true  ✓
  - Alice's swap executes — allowlist completely bypassed

Alternatively, without allowlisting the router:
  - Bob (address in allowlist) calls router.exactInputSingle(...)
  - Extension evaluates: allowedSwapper[pool][router] == false  ✗
  - Bob's swap reverts — allowlisted user blocked from using the router
```

Foundry test outline:
1. Deploy pool with `SwapAllowlistExtension` as a `beforeSwap` extension.
2. Call `setAllowedToSwap(pool, router, true)` from pool admin.
3. Call `router.exactInputSingle(...)` from an address not in the allowlist.
4. Assert the swap succeeds (demonstrating the bypass).
5. Alternatively, call `setAllowedToSwap(pool, router, false)` and call from an allowlisted user through the router; assert it reverts.
