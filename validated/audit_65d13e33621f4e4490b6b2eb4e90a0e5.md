### Title
Storage Deposits Paid by Relayers for GC'd Block Headers Are Permanently Locked With No Withdrawal Mechanism ã (`contract/src/lib.rs`)

---

### Summary

`submit_blocks` charges relayers a NEAR storage deposit for each batch of block headers they submit. When `run_mainchain_gc` later removes those headers from on-chain storage, the freed NEAR tokens accumulate in the contract's own balance. No function exists to withdraw or redistribute these freed deposits. They are permanently locked.

---

### Finding Description

`submit_blocks` is `#[payable]` and enforces a storage deposit from the caller: [1](#0-0) 

The sequence inside `submit_blocks` is:
1. Capture `initial_storage = env::storage_usage()` before any writes.
2. Insert all new headers into `headers_pool` and the two mainchain maps.
3. Call `run_mainchain_gc(num_of_headers)`, which **removes** old headers from storage.
4. Compute `diff_storage_usage = env::storage_usage().saturating_sub(initial_storage)`.
5. Charge the caller `required_deposit = storage_byte_cost * diff_storage_usage`. [2](#0-1) 

`run_mainchain_gc` removes entries from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`: [3](#0-2) 

In NEAR Protocol, when storage entries are removed, the NEAR tokens that were locked to cover that storage are released back to the **contract account's own balance** ã not to the original depositor. Because `diff_storage_usage` uses `saturating_sub`, if GC frees more bytes than the new headers consume, the net charge to the current caller is zero. The freed tokens from prior relayers' deposits simply accumulate in the contract's balance.

There is no `withdraw`, `transfer`, or governance-callable function anywhere in `contract/src/lib.rs` that can extract this accumulated balance. The only `Promise::new(...).transfer(...)` in the entire contract is the per-call excess refund to the current caller: [4](#0-3) 

This refund only covers the overpayment within a single call. It does not return the storage deposit of a prior relayer whose blocks were later GC'd.

The same issue applies to blocks removed during chain reorganizations in `reorg_chain`: [5](#0-4) 

---

### Impact Explanation

NEAR tokens paid by relayers for storage of block headers that are subsequently garbage-collected or displaced by reorgs are permanently locked in the contract's balance. With the recommended `gc_threshold = 52704` (approximately one year of Bitcoin blocks), the first cohort of relayers who funded storage for those 52704 headers will never recover their deposits. At NEAR's storage cost of ~10 NEAR per MB, and each `ExtendedHeader` entry consuming on the order of several hundred bytes across three maps, the locked amount grows to a material sum over the contract's lifetime. No privileged role ã not `DAO`, `PauseManager`, `CodeDeployer`, nor any other defined `Role` ã has a path to recover these funds.

---

### Likelihood Explanation

This is a certainty, not a probability. Every deployment that runs long enough for GC to trigger will accumulate locked NEAR. The recommended `gc_threshold` guarantees GC runs continuously in production. The absence of a withdrawal function is a structural gap, not a configuration issue.

---

### Recommendation

Add a governance-callable (e.g., `Role::DAO`-gated) function that transfers the contract's excess balance ã the difference between `env::account_balance()` and the minimum balance required to cover current `env::storage_usage()` ã to a designated treasury or back to relayers. For example:

```rust
#[access_control_any(roles(Role::DAO))]
pub fn withdraw_excess_storage_refund(&mut self, recipient: AccountId) -> Promise {
    let min_balance = env::storage_byte_cost()
        .saturating_mul(env::storage_usage().into());
    let excess = env::account_balance().saturating_sub(min_balance);
    require!(excess > NearToken::from_near(0), "No excess balance");
    Promise::new(recipient).transfer(excess)
}
```

---

### Proof of Concept

1. Relayer A calls `submit_blocks` with 52704 headers, paying the full storage deposit for all of them. These are stored in `headers_pool` and the two mainchain maps.
2. The chain advances. Relayer B calls `submit_blocks` with another batch. Inside the call, `run_mainchain_gc` removes the oldest headers (the ones Relayer A paid for). `diff_storage_usage` is 0 (GC freed as much as was added), so Relayer B pays nothing.
3. The NEAR tokens Relayer A paid for those now-deleted headers are in the contract's balance. No function exists to return them to Relayer A or to any governance address.
4. Repeat indefinitely. The contract's balance grows monotonically relative to its actual storage obligation, with no recovery path.
