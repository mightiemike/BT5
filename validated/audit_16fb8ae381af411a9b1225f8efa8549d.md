### Title
Equihash Solution Validation Return Value Ignored — Invalid Zcash Blocks Accepted - (File: `contract/src/zcash.rs`)

### Summary
In the Zcash `check_pow` function, `equihash::is_valid_solution` returns a `Result<bool, _>`. The code handles the `Err` variant (by panicking) but silently discards the `Ok(bool)` validity result. A block with a structurally well-formed but cryptographically invalid Equihash solution returns `Ok(false)`, which is dropped. The block passes PoW verification and is accepted into the contract's `headers_pool`.

### Finding Description

In `contract/src/zcash.rs`, `check_pow` ends with:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
```

`equihash::is_valid_solution` has the signature `fn(...) -> Result<bool, Error>`. The `.unwrap_or_else(|e| panic)` call handles only the `Err` path. When the function returns `Ok(false)` — meaning the solution is structurally parseable but cryptographically invalid — `.unwrap_or_else` extracts the `false` value and the entire expression evaluates to `false`, which is then **discarded as an unused statement**. No `require!` or assertion wraps the result. The block proceeds to `submit_block_header_inner` and is stored.

This is the exact same bug class as M-04: a boolean-returning validation helper is called, its return value is not acted upon, and the invalid entity is accepted. [1](#0-0) 

The call site in `submit_block_header` (non-dogecoin path) invokes `check_target` → `check_pow` only when `skip_pow_verification` is `false`, which is the production setting: [2](#0-1) 

### Impact Explanation

An attacker can submit a Zcash block header whose `solution` field is syntactically valid (passes `equihash` parsing, returning `Ok(false)`) but cryptographically invalid. The block bypasses the only Equihash check and is stored in `headers_pool`. If its `chain_work` exceeds the current tip, it becomes the mainchain tip via `reorg_chain`. Downstream callers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will then verify SPV proofs against a fraudulent canonical chain, returning `true` for transactions that were never confirmed on the real Zcash chain.

**Corrupted invariant:** Every block in `headers_pool` must have a valid Equihash solution. This invariant is broken. [3](#0-2) 

### Likelihood Explanation

**Medium.** The `submit_blocks` entry point is callable by any trusted relayer (or, depending on configuration, any account). Constructing a structurally parseable but invalid Equihash solution is straightforward — the attacker only needs to produce bytes that the `equihash` crate can parse without returning `Err`, while the actual graph-based validity check returns `false`. The attacker also controls `chain_work` indirectly through the `bits` field, allowing them to manufacture a tip with arbitrarily high claimed work. [4](#0-3) 

### Recommendation

Wrap the call in a `require!` that asserts the returned `bool` is `true`:

```rust
// Before (broken):
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });

// After (fixed):
let is_valid = equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
require!(is_valid, "Invalid Equihash solution");
``` [5](#0-4) 

### Proof of Concept

1. Construct a Zcash `Header` with valid `bits`, valid `time`, valid `version >= 4`, and a `solution` field that is syntactically well-formed (correct byte length for n=200, k=9) but does not satisfy the Equihash graph constraint — causing `equihash::is_valid_solution` to return `Ok(false)`.
2. Call `submit_blocks([header])` on the deployed Zcash contract with `skip_pow_verification = false`.
3. `check_pow` is entered; `equihash::is_valid_solution` returns `Ok(false)`; the `false` is silently dropped.
4. `submit_block_header_inner` stores the header. If `chain_work` is crafted to exceed the current tip, `reorg_chain` promotes it to mainchain tip.
5. A subsequent call to `verify_transaction_inclusion_v2` with a fabricated Merkle proof against this fraudulent tip returns `true`. [6](#0-5)

    #[payable]

    #[trusted_relayer]
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
