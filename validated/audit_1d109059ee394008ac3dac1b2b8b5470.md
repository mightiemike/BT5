### Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[pool][sender]`. When a user routes through `MetricOmmSimpleRouter`, the pool's `swap()` is called by the router, so `msg.sender` inside the pool is the router address — and that router address is what gets forwarded as `sender` to the extension. The allowlist therefore checks the router's address, not the actual user's address. A pool admin who allowlists the router (the only way to let any allowlisted user trade through the router) simultaneously opens the gate to every non-allowlisted user.

### Finding Description

**Call chain:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)`.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(recipient, zeroForOne, amount, priceLimit, "", params.extensionData)`.
   At this point `msg.sender` inside the pool is the **router address**.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` — passing the router as `sender`.
4. `ExtensionCalling._beforeSwap` encodes and dispatches to `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
5. The extension evaluates:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool, `sender` = **router**. The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

**The dilemma this creates for the pool admin:**

| Admin choice | Effect |
|---|---|
| Do NOT allowlist the router | Allowlisted users cannot trade through the router at all |
| Allowlist the router | Every user on earth can bypass the allowlist via the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users. The guard is structurally broken for any router-mediated path.

**Relevant code locations:**

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1)

### Impact Explanation

Any non-allowlisted user can execute swaps on a curated pool that is supposed to restrict trading to a specific set of addresses. The allowlist — the sole access-control mechanism on the swap path — is fully bypassed. This breaks the core invariant of curated pools and allows unauthorized price-taking, front-running, or draining of LP value on pools whose operators believed access was restricted. Impact is **High**: broken core pool functionality and unauthorized swap execution on production pools.

### Likelihood Explanation

The router is the standard, documented periphery entry point for swaps. Any pool operator who deploys a curated pool and wants allowlisted users to be able to use the router must allowlist the router address. The moment they do, the bypass is live and reachable by any unprivileged address with no special setup. Likelihood is **High**.

### Recommendation

The extension must recover the original user's address rather than relying on the `sender` argument forwarded by the pool. Two sound approaches:

1. **Pass the original caller through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a trusted router convention.
2. **Check `recipient` or use a dedicated identity field**: Redesign the hook interface to carry a separate `originator` field that the pool populates from a transient-storage context set by the router before calling `swap`, analogous to how the router already stores the payer in transient storage (`_setNextCallbackContext`).
3. **Allowlist at the router level**: The router enforces its own allowlist before forwarding to the pool, and the pool's extension trusts only calls from the router. This requires the router to be a trusted, non-upgradeable contract.

### Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)   // alice is allowed
  admin calls setAllowedToSwap(pool, router, true)  // required so alice can use the router

Attack:
  bob (not allowlisted) calls:

      pool: pool,
      recipient: bob,
      zeroForOne: true,
      amountIn: X,

    → pool calls _beforeSwap(msg.sender=router, ...)
    → extension checks allowedSwapper[pool][router] == true  ✓
    → swap executes for bob despite bob not being allowlisted

Result: bob successfully swaps on a pool that was supposed to deny him.
