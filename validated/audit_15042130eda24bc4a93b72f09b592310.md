### Title
`compute_exempted_kickout` Uses Inflated `total_stake` Denominator Including Already-Kicked-Out Validators, Making `min_keep_stake` Threshold Unreachable — (`chain/epoch-manager/src/lib.rs`)

---

### Summary

In `EpochManager::compute_validators_to_reward_and_kickout`, the `total_stake` denominator used to compute `min_keep_stake` includes the stake of validators who are already scheduled for kickout from the **previous** epoch (`prev_validator_kickout`). However, those same validators are explicitly **skipped** in `compute_exempted_kickout` when accumulating `exempted_stake`. This creates an inflated denominator against an unreachable numerator — the exact structural analog of the Party `burn`/`totalVotingPower` invariant break — and can make the `min_keep_stake` threshold permanently unreachable, causing all remaining validators to be over-exempted and preventing legitimate kickouts.

---

### Finding Description

At the end of epoch T, `collect_blocks_info` calls `compute_validators_to_reward_and_kickout` to determine which validators should be kicked out for epoch T+2.

**Step 1 — `total_stake` is computed from all epoch-T validators, including those already kicked out:** [1](#0-0) 

The loop at line 445 iterates over `epoch_info.validators_iter()` (epoch T's full validator set) and unconditionally accumulates every validator's stake into `total_stake` at line 469. This set **includes** validators who appear in `prev_validator_kickout` — validators who were kicked out at the end of epoch T-1 and will not be in epoch T+1, but are still present in epoch T's validator set (because epoch T's set was fixed at the end of epoch T-2).

**Step 2 — `prev_validator_kickout` is sourced from `next_epoch_info` (epoch T+1):** [2](#0-1) 

`prev_validator_kickout = next_epoch_info.validator_kickout()` contains validators removed at the end of epoch T-1. They are still active in epoch T, so their stake is counted in `total_stake`.

**Step 3 — `min_keep_stake` is derived from the inflated `total_stake`:** [3](#0-2) 

`min_keep_stake = total_stake * exempt_perc / 100`, where `exempt_perc = 100 - validator_max_kickout_stake_perc`.

**Step 4 — The exemption loop skips `prev_validator_kickout` validators, making the threshold unreachable:** [4](#0-3) 

The loop iterates over all validators sorted by online ratio and skips any validator in `prev_validator_kickout`. So `exempted_stake` can only accumulate from the **remaining** validators (those not already kicked out), but `min_keep_stake` was computed against the **full** stake including the kicked-out validators.

**Concrete scenario with default `validator_max_kickout_stake_perc = 30`:**

- Epoch T validators: A (1000), B (1000), C (1000), D (3000) — total = 6000
- D was kicked out at end of epoch T-1 → in `prev_validator_kickout`
- `exempt_perc = 70`, `min_keep_stake = 6000 × 70/100 = 4200`
- Exemption loop can only accumulate from A, B, C → max `exempted_stake` = 3000
- 3000 < 4200 → threshold **unreachable** → A, B, C all exempted
- **Result:** No validators are kicked out in epoch T+2, even if A, B, C all performed below threshold

Without D's stake in the denominator: `min_keep_stake = 3000 × 70/100 = 2100`, which is reachable, and the correct subset of A/B/C would be exempted while the worst performer is kicked out.

---

### Impact Explanation

The `validator_max_kickout_stake_perc` safety mechanism is designed to ensure at most X% of **active** validator stake is removed per epoch, preventing network destabilization. By including already-kicked-out validators' stake in `total_stake`, the effective protection threshold is computed against a denominator that is larger than the reachable exemption pool. When the kicked-out validators' combined stake exceeds `validator_max_kickout_stake_perc`% of total stake, the `min_keep_stake` threshold becomes permanently unreachable, causing **all** remaining validators to be exempted from kickout — even those who performed far below the block/chunk production thresholds. This allows chronically underperforming validators to remain in the active set indefinitely, degrading network liveness and chunk production reliability.

**Severity: High** — the invariant that `validator_max_kickout_stake_perc` correctly bounds per-epoch kickouts is broken; the mechanism silently over-exempts validators with no error or warning.

---

### Likelihood Explanation

This triggers deterministically whenever a validator holding more than `validator_max_kickout_stake_perc`% of total stake (default: 30%) is kicked out in epoch T-1. On mainnet, large validators routinely hold significant stake fractions. A single large validator being kicked out (e.g., for running an old protocol version via `ProtocolVersionTooOld`) is sufficient to trigger the condition for the following epoch's kickout computation. No privileged action is required beyond normal protocol operation.

---

### Recommendation

Exclude validators in `prev_validator_kickout` from `total_stake` when computing `min_keep_stake`, so the denominator reflects only the stake that can actually contribute to the exemption pool:

```rust
// In compute_validators_to_reward_and_kickout, change:
total_stake = total_stake.checked_add(v.stake()).unwrap();

// To:
if !prev_validator_kickout.contains_key(account_id) {
    total_stake = total_stake.checked_add(v.stake()).unwrap();
}
```

This ensures `min_keep_stake` is always reachable by the non-kicked-out validators, preserving the intended semantics of `validator_max_kickout_stake_perc`.

---

### Proof of Concept

The divergent values are exact and traceable:

1. `epoch_info.validators_iter()` at line 445 yields all epoch-T validators including those in `prev_validator_kickout`.
2. `total_stake` at line 469 accumulates their stake unconditionally.
3. `min_keep_stake` at lines 386–390 is `total_stake × exempt_perc / 100`.
4. The exemption loop at lines 392–403 skips `prev_validator_kickout` members, capping `exempted_stake` at `total_stake − (stake of prev_validator_kickout members)`.
5. When `stake(prev_validator_kickout) > validator_max_kickout_stake_perc% × total_stake`, we get `exempted_stake_max < min_keep_stake`, the loop exhausts all candidates, and every remaining validator is added to `exempted_validators`.
6. Back in `compute_validators_to_reward_and_kickout` at line 520, every validator is in `exempted_validators`, so the kickout map remains empty — no validators are kicked out regardless of their block/chunk production stats. [1](#0-0) [5](#0-4) [6](#0-5) [7](#0-6)
