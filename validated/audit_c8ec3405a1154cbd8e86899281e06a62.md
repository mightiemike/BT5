### Title
Liquidatee Continues to Accrue Spot Borrow Interest and Perp Funding Between Sequential Liquidation Steps - (`core/contracts/SpotEngineState.sol`, `core/contracts/PerpEngineState.sol`)

---

### Summary

Nado's liquidation of a multi-product subaccount requires multiple sequential `LiquidateSubaccount` transactions. Between those transactions the sequencer routinely submits `SpotTick` and `PerpTick` transactions that advance the global borrow-interest and funding-rate accumulators for **all** subaccounts, including those mid-liquidation. There is no mechanism to freeze accrual for a subaccount once it becomes undercollateralized. As a result, the liquidatee's spot liabilities grow and their perp `vQuoteBalance` worsens between liquidation steps, causing them to receive less residual value than they would if accrual were paused at the moment liquidation began.

---

### Finding Description

**Spot borrow interest (primary analog)**

`SpotEngineState.updateStates(dt)` iterates over every product and calls `_updateState`, which compounds `cumulativeBorrowsMultiplierX18` by the per-second borrow rate raised to the power of `dt`:

```
borrowRateMultiplierX18 = (ONE + borrowerRateX18).pow(int128(dt));
state.cumulativeBorrowsMultiplierX18 = state.cumulativeBorrowsMultiplierX18.mul(borrowRateMultiplierX18);
``` [1](#0-0) 

Individual balances are stored normalized; the real balance is recovered lazily as `amountNormalized * cumulativeBorrowsMultiplierX18`. A subaccount with a negative (borrowed) spot balance therefore has its effective liability grow every time `updateStates` is called, regardless of whether it is currently being liquidated. [2](#0-1) 

`updateStates` is triggered by the sequencer via a `SpotTick` transaction processed inside `EndpointTx.processTransactionImpl`:

```solidity
} else if (txType == IEndpoint.TransactionType.SpotTick) {
    ...
    spotEngine.updateStates(dt);
``` [3](#0-2) 

**Perp funding (secondary analog)**

`PerpEngineState.updateStates` advances `cumulativeFundingLongX18` and `cumulativeFundingShortX18` for every product with non-zero open interest:

```solidity
state.cumulativeFundingLongX18 += paymentAmount;
state.cumulativeFundingShortX18 += paymentAmount;
``` [4](#0-3) 

A perp `Balance` stores `lastCumulativeFundingX18`; the accrued funding is settled into `vQuoteBalance` lazily in `_updateBalance`. A subaccount with an open perp position continues to accrue funding between liquidation steps, worsening its `vQuoteBalance`. [5](#0-4) 

**Liquidation is multi-step with no accrual freeze**

`liquidateSubaccountImpl` processes one product per call and has no flag or guard that would prevent `updateStates` from running between calls: [6](#0-5) 

The sequencer batches transactions via `submitTransactionsChecked`. A typical liquidation batch looks like:

```
[SpotTick, PerpTick, LiquidateSubaccount(productA), SpotTick, PerpTick, LiquidateSubaccount(productB), ..., LiquidateSubaccount(finalize)]
``` [7](#0-6) 

Every `SpotTick`/`PerpTick` in that batch advances the global accumulators, growing the liquidatee's liabilities and worsening their perp PnL before the next liquidation step executes.

---

### Impact Explanation

**Impact: Medium.**

For spot liabilities: the liquidatee's effective borrow balance grows between steps. When the liquidator removes the (now larger) liability, the liquidatee's quote balance is debited by `liquidationPrice × (original_liability + Δ)` instead of `liquidationPrice × original_liability`. The extra deduction `liquidationPrice × Δ` is a direct loss to the liquidatee that would not exist if accrual were frozen at liquidation onset.

For perp positions: adverse funding accrual between steps reduces `vQuoteBalance`, directly reducing the liquidatee's residual quote after finalization.

In both cases the loss is bounded by the sequencer's tick frequency multiplied by the number of liquidation steps required. For a subaccount with many products this can span multiple sequencer batches, making the cumulative loss non-trivial at high utilization (high borrow rates) or high funding rates.

---

### Likelihood Explanation

**Likelihood: Medium.**

Any subaccount holding both spot liabilities and perp positions (or multiple spot liabilities across different products) requires more than one `LiquidateSubaccount` transaction to fully close. The sequencer always interleaves `SpotTick`/`PerpTick` with liquidation transactions as part of normal operation. This condition is met for every multi-product liquidation, which is the common case for leveraged traders.

---

### Recommendation

Snapshot the liquidatee's normalized borrow balances and perp `lastCumulativeFundingX18` at the start of the first liquidation step and use those snapshots for all subsequent steps, or introduce a per-subaccount "liquidation-in-progress" flag that causes `_updateBalanceNormalized` and `_updateBalance` to skip accrual for flagged subaccounts until finalization is complete.

---

### Proof of Concept

1. Subaccount `A` holds: 100 USDC borrowed (negative spot balance) + 1 BTC-PERP long position. It becomes undercollateralized.
2. Liquidator submits batch:
   - `SpotTick(t=0)` → `cumulativeBorrowsMultiplierX18` advances; A's effective USDC liability grows from 100 to 100.001 USDC.
   - `PerpTick(t=0)` → `cumulativeFundingLongX18` advances; A's BTC-PERP `vQuoteBalance` decreases by funding payment.
   - `LiquidateSubaccount(sender=liquidator, liquidatee=A, productId=BTC-PERP)` → perp position closed at discount.
   - `SpotTick(t=1)` → liability grows further to 100.002 USDC.
   - `LiquidateSubaccount(sender=liquidator, liquidatee=A, productId=USDC-SPOT)` → spot liability liquidated at the now-larger amount; A's quote balance is debited by `liquidationPrice × 100.002` instead of `liquidationPrice × 100`.
   - `LiquidateSubaccount(sender=liquidator, liquidatee=A, productId=uint32.max)` → finalization.
3. A receives less residual quote than if accrual had been frozen at step 1. The difference is `liquidationPrice × 0.002 USDC` per tick interval, multiplied by the number of ticks between steps. [8](#0-7) [9](#0-8) [10](#0-9)

    function updateStates(uint128 dt, int128[] calldata avgPriceDiffs)
        external
        onlyEndpoint
    {
        int128 dtX18 = int128(dt).fromInt();
        for (uint32 i = 0; i < avgPriceDiffs.length; i++) {
            uint32 productId = productIds[i];
            State memory state = states[productId];
            if (state.openInterest == 0) {
                continue;
            }
            require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
            {
                int128 indexPriceX18 = _risk(productId).priceX18;

                // cap this price diff
                int128 priceDiffX18 = avgPriceDiffs[i];

                int128 maxPriceDiff = MAX_DAILY_FUNDING_RATE.mul(indexPriceX18);

                if (priceDiffX18.abs() > maxPriceDiff) {
                    // Proper sign
                    priceDiffX18 = (priceDiffX18 > 0)
                        ? maxPriceDiff
                        : -maxPriceDiff;
                }

                int128 paymentAmount = priceDiffX18.mul(dtX18).div(ONE_DAY_X18);
                state.cumulativeFundingLongX18 += paymentAmount;
                state.cumulativeFundingShortX18 += paymentAmount;

                emit FundingPayment(
                    productId,
                    dt,
                    state.openInterest,
                    paymentAmount
                );
            }
            _setState(productId, state);
            emit PriceQuery(productId);
        }
    }
