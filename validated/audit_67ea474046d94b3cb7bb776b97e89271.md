### Title
Incorrect `alreadyMatched` Computation Using Current Fill Price Corrupts Minimum Fee Threshold Across Partial Fills — (File: `core/contracts/OffchainExchange.sol`)

### Summary

`OffchainExchange.applyFee` receives `alreadyMatched` (denominated in quote) computed by multiplying the **current** maker's price against the cumulative `filledAmounts` mapping, which is stored in **base** units. When a taker order is partially filled across multiple matches at different maker prices, this conversion produces an incorrect quote-denominated `alreadyMatched`, corrupting the minimum-fee threshold calculation and causing either under-collection of protocol fees or over-charging of taker fees.

### Finding Description

`filledAmounts` is a mapping from order digest to a cumulative **base** amount filled: [1](#0-0) 

At the call site for `applyFee`, `alreadyMatched` is computed as: [2](#0-1) 

The parameter comment explicitly states `alreadyMatched` is **in quote**: [3](#0-2) 

The conversion `maker.order.priceX18.mul(filledAmounts[ordersInfo.taker.digest])` uses the **current fill's maker price** to convert the cumulative base amount to quote. If the taker order was previously partially filled at a different maker price, the actual quote already matched differs from this computed value.

Inside `applyFee`, `alreadyMatched` gates the minimum-fee threshold: [4](#0-3) 

- Line 526: the flat minimum fee is only applied when `alreadyMatched == 0` (i.e., first fill).
- Lines 536–538: `feeApplied` is computed as `abs(alreadyMatched + matchQuote) - market.minSize`, clamped to `matchQuote.abs()`.

When `alreadyMatched` is wrong (because the prior fill was at a different price), `feeApplied` is miscalculated, directly corrupting `orderInfo.fee` and `orderInfo.quoteDelta`.

**Concrete scenario:**

1. Taker places a buy order for 10 ETH. `market.minSize` = $100.
2. Fill 1: 5 ETH matched at maker price $2,900. Actual quote matched = $14,500. `filledAmounts[digest] = 5e18`.
3. Fill 2: 5 ETH matched at maker price $3,100. `alreadyMatched` is computed as `-(3100 * 5) = -$15,500` instead of the correct `-$14,500`.
4. `feeApplied = abs(-15500 + (-15500)) - 100 = 30900` — clamped to `matchQuote.abs() = 15500`. The fee basis is correct by coincidence only when the delta is small, but the threshold boundary is shifted, causing the minimum-fee exemption window to be applied incorrectly.

More critically, if the price difference is large enough that `abs(alreadyMatched + matchQuote) < market.minSize` evaluates differently than it should, `feeApplied` becomes 0 when it should be positive (fee under-collected) or positive when it should be 0 (fee over-charged).

### Impact Explanation

**Fee accounting corruption.** The protocol either:
- Under-collects taker fees (protocol revenue loss), when the overestimated `alreadyMatched` pushes `feeApplied` below the true value, or
- Over-charges the taker (user asset loss), when the underestimated `alreadyMatched` inflates `feeApplied`.

The corrupted `orderInfo.fee` flows directly into `updateCollectedFees` and `_updateBalances`, meaning the on-chain balance state is permanently wrong for every affected fill. [5](#0-4) 

### Likelihood Explanation

**High.** Any taker order that is partially filled across two or more matches at different maker prices triggers this. This is the normal execution path in any active market with multiple makers at different price levels. No special privileges are required — any unprivileged trader submitting a taker order larger than the best maker's available size will trigger this on the second fill.

### Recommendation

Track `filledAmounts` in **quote** units rather than base units, or maintain a separate `filledQuoteAmounts` mapping. Replace the `alreadyMatched` computation:

```solidity
// Current (incorrect): converts base filledAmounts using current fill price
-maker.order.priceX18.mul(filledAmounts[ordersInfo.taker.digest])

// Corrected: track cumulative quote directly
-filledQuoteAmounts[ordersInfo.taker.digest]
```

And update the accumulation at the end of each fill to add `ordersInfo.taker.quoteDelta.abs()` (pre-fee quote) to `filledQuoteAmounts`.

### Proof of Concept

1. Deploy `OffchainExchange` with `market.minSize = 100e18` (100 USDC).
2. Place taker buy order: 10 ETH, limit price $3,200.
3. Place maker sell order A: 5 ETH at $2,900. Match → `filledAmounts[takerDigest] = 5e18`. Actual quote matched = $14,500.
4. Place maker sell order B: 5 ETH at $3,100. `alreadyMatched = -(3100e18 * 5e18 / 1e18) = -15,500e18` (wrong; should be `-14,500e18`).
5. In `applyFee`: `feeApplied = abs(-15500e18 + (-15500e18)) - 100e18 = 30,900e18`, clamped to `15,500e18`. The minimum-fee threshold boundary is evaluated against a stale/wrong quote basis, producing a fee amount that does not correspond to the actual quote traded in fill 2.
6. Observe that `orderInfo.fee` and the resulting `collectedFees` differ from the correct value computed against the true cumulative quote of $14,500 + $15,500 = $30,000. [2](#0-1) [1](#0-0)
