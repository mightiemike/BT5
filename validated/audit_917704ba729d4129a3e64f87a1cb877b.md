### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Guard — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) was deprecated in favour of `verify_transaction_inclusion_v2`, which adds a mandatory coinbase Merkle proof check to mitigate the 64-byte transaction Merkle-proof forgery vulnerability. However, the v1 function is still declared `pub` and remains a live, callable NEAR entry point. Rust's `#[deprecated]` attribute emits only a compiler warning; it does not restrict runtime access. Any unprivileged NEAR caller can invoke v1 directly, skipping the coinbase guard entirely and obtaining a `true` result for a forged transaction inclusion proof.

---

### Finding Description

`lib.rs` is 1 180 lines long. Within it, two proof-verification entry points coexist:

**v1 — still publicly reachable:** [1](#0-0) 

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
```

**v2 — the intended secure path:** [2](#0-1) 

v2 first verifies the coinbase Merkle proof against the block's `merkle_root`, then delegates to v1 via `self.verify_transaction_inclusion(args.into())`. The coinbase check is the only mechanism that prevents an attacker from supplying an internal Merkle-tree node (≤ 64 bytes) as a fake `tx_id`.

v1 performs no such check: [3](#0-2) 

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

Because `pub` is not removed and `#[deprecated]` has no runtime effect on NEAR, v1 is a fully reachable entry point for any caller.

---

### Impact Explanation

An attacker can forge a transaction inclusion proof for a transaction that was never broadcast:

1. Pick any confirmed mainchain block whose `merkle_root` is known.
2. Construct a `ProofArgs` where `tx_id` is an internal Merkle-tree node hash and `merkle_proof` is a valid sibling path to that node (the classic 64-byte attack described at https://www.bitmex.com/blog/64-Byte-Transactions).
3. Call `verify_transaction_inclusion` directly (bypassing v2).
4. The function returns `true`.

Any recipient contract that calls `verify_transaction_inclusion` — or that an attacker can route through it — will accept the forged proof as valid, enabling fraudulent cross-chain settlement, double-spend claims, or bridge fund releases.

---

### Likelihood Explanation

The entry point is unconditionally public, requires no privileged role, no staking, and no special NEAR account. The 64-byte Merkle forgery technique is well-documented and the required inputs (a real block hash, its Merkle root, and a crafted sibling path) are all derivable from public Bitcoin chain data. Likelihood is **high** for any deployment where a downstream contract consumes the v1 result.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or gate it with `#[private]` so it is only callable internally (as it already is from v2). The function should not be an externally reachable NEAR entry point. Alternatively, delete the function body and have v2 call the shared logic directly without re-exposing the insecure path.

---

### Proof of Concept

```
# Attacker constructs a ProofArgs (Borsh-encoded) with:
#   tx_id          = hash of an internal Merkle node in block B
#   tx_block_blockhash = hash of block B (on mainchain, enough confirmations)
#   tx_index       = index matching the internal node position
#   merkle_proof   = valid sibling path from that node to merkle_root
#   confirmations  = 1

near call <contract> verify_transaction_inclusion \
  '{"args": <borsh-encoded ProofArgs>}' \
  --accountId attacker.near

# Returns: true
# No coinbase proof was required; the 64-byte forgery succeeds.
```

The root cause is directly traceable to the monolithic 1 180-line `lib.rs`: [4](#0-3) 

The deprecated function was left as a live public entry point inside the large file, making the open attack surface easy to overlook during review — the exact security risk the external report identifies as the consequence of excessive file size and insufficient modularisation.

    #[deprecated(
        since = "0.5.0",

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
