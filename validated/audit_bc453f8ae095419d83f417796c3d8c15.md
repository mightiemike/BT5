## Summary
`_applyWatermark` uses a strict `<` comparison to detect drawdown floor breaches. When the post-swap per-share metric lands exactly on the floor (`metric == hwm * floorMultiplier / E6`), `breached` is set to `false` and the swap settles without reverting. The stop-loss fails open at the one value that is its hard boundary, allowing LP principal to drain by exactly the configured drawdown amount.

## Finding Description
In `_applyWatermark` at line 334:

```solidity
breached = metric < (hwm * floorMultiplier) / E6;
```

`floorMultiplier = E6 - drawdownE6`, so the floor is `hwm * (1 - drawdown)`. When `metric` equals the floor exactly, the expression evaluates to `false` because `metric < metric` is always false. `_checkAndUpdateWatermarks` (lines 270â€“278) then skips the revert and writes `hwm` (not `metric`) back to storage, leaving the watermark unchanged. The next swap faces the same floor comparison against the same `hwm`, so the boundary trade's loss is permanently committed with no self-correction.

## Impact Explanation
The stop-loss is the primary on-chain mechanism preventing LP value from draining beyond the configured drawdown. A swap landing the metric exactly on the floor is the worst-case trade the protection was designed to block. Allowing it to settle means LP principal leaks by exactly one full drawdown unit (e.g., 5% of token0-denominated bin value for `drawdownE6 = 50_000`) beyond the configured cap. This constitutes a direct loss of LP principal above Sherlock thresholds for pools with material bin balances.

## Likelihood Explanation
No privileged access is required. Any unprivileged trader can observe the current watermark and bin balances on-chain, compute the exact swap size that moves the metric to the floor off-chain (the `afterSwap` hook receives `packedSlot0Final` and the oracle mid, making the final metric fully predictable), and submit the crafted swap. The condition is repeatable on any pool with a configured stop-loss and non-zero bin liquidity.

## Recommendation
Change the strict less-than to less-than-or-equal in `_applyWatermark` (line 334):

```solidity
// Before
breached = metric < (hwm * floorMultiplier) / E6;

// After
breached = metric <= (hwm * floorMultiplier) / E6;
```

This ensures the stop-loss fires when the metric reaches the floor, matching the invariant that value per share cannot fall by more than `drawdownE6 / E6` from the watermark.
