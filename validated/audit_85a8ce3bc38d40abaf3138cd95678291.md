Audit Report

## Title
`SwapAllowlistExtension` allowlist bypassed via router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is always the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` mediates a swap, `sender` equals the router's address. If the pool admin allowlists the router so that allowlisted users can use it, every user — including non-allowlisted ones — can bypass the guard by routing through the router, because the extension cannot distinguish end users behind the same router address.

## Finding Description

`SwapAllowlistExtension.beforeSwap` gates swaps using the `sender` argument forwarded by the pool: [1](#0-0) 

`MetricOmmPool.swap` always passes `msg.sender` (the direct caller of `pool.swap()`) as the first argument to `_beforeSwap`: [2](#0-1) 

`MetricOmmSimpleRouter` calls `pool.swap()` directly in all four entry points — `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` — making itself `msg.sender` to the pool in every case: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

The result is that `sender` seen by the extension is always the router address, not the end user. The pool admin cannot simultaneously allow allowlisted users to use the router (requires allowlisting the router address) and block non-allowlisted users from using the router (impossible once the router is allowlisted, since all users share the same router address).

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to specific counterparties (e.g., KYC'd addresses or curated market makers) is fully bypassed by any user routing through `MetricOmmSimpleRouter`. Non-allowlisted users can execute swaps against LP positions that were intended to be accessible only to trusted parties, constituting unauthorized swap execution against LP assets — a direct loss-of-principal path for LPs who deployed capital under the assumption that only trusted counterparties could trade against them.

## Likelihood Explanation

The bypass is triggered whenever the pool admin allowlists the router address, which is a natural and expected configuration: allowlisted users who want to perform multi-hop swaps or use slippage-protected wrappers must go through the router, so the admin allowlists it. The moment the router is allowlisted, the guard is open to all users. No malicious admin action is required — the vulnerability is a structural design flaw in identity propagation between the router and the extension.

## Recommendation

The `sender` forwarded to extensions must reflect the economic actor (the end user), not the intermediary contract. Two options:

**Option A — Router encodes original caller in `extensionData`:** Encode `msg.sender` into `extensionData` at the router level and have `SwapAllowlistExtension` decode and verify it.

**Option B — Pool exposes an `originSender` parameter (preferred):** Add an `originSender` field to the `beforeSwap` hook signature. The pool sets it to `msg.sender` for direct calls; the router passes the original `msg.sender` explicitly. `SwapAllowlistExtension` then checks `originSender` instead of `sender`.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only allowedUser is allowlisted.
// Admin also allowlists the router so allowedUser can use it.
swapExtension.setAllowedToSwap(address(pool), allowedUser, true);
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Attack: non-allowlisted user bypasses the guard via the router.
vm.prank(nonAllowlistedUser);
// Direct call reverts with NotAllowedToSwap ✓

// Router call succeeds — extension sees sender=router, which IS allowlisted:
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: address(token0),
    tokenOut: address(token1),
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: nonAllowlistedUser,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// ↑ succeeds — allowlist bypassed
```
