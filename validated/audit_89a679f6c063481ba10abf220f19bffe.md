### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Merkle Forgery Protection Added in `verify_transaction_inclusion_v2` — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof. However, the original `verify_transaction_inclusion` function is still publicly callable by any unprivileged NEAR account. An attacker or a downstream consumer can bypass the coinbase-proof guard entirely by invoking the deprecated entry point directly, obtaining a `true` result for a forged proof.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces a coinbase Merkle proof check before delegating to `verify_transaction_inclusion`:

```
// verify_transaction_inclusion_v2 (lines 347-369)
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);

#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [1](#0-0) 

The deprecated `verify_transaction_inclusion` is annotated `#[deprecated]` and `#[pause]`, but carries **no** `#[private]` guard, making it fully reachable by any NEAR caller:

```
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [2](#0-1) 

The function itself performs only a standard Merkle root comparison with no coinbase proof requirement:

```
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [3](#0-2) 

The 64-byte forgery attack works because Bitcoin's Merkle tree does not distinguish between leaf nodes (transactions) and internal nodes. A 64-byte internal node can be presented as a `tx_id`, and a valid Merkle path to the root can be constructed for it. Without the coinbase proof check, the function cannot detect this forgery and returns `true`.

---

### Impact Explanation

Any NEAR contract that calls `verify_transaction_inclusion` directly — whether because it was integrated before v2 existed, or because an attacker deliberately routes through the deprecated path — receives a `true` result for a forged transaction inclusion proof. This breaks the core security guarantee of the light client: that a `true` return means a real Bitcoin transaction was confirmed. Downstream contracts that gate asset releases, bridge operations, or other state changes on this result can be deceived into acting on a transaction that never existed.

**Impact: High** — the corrupted value is the proof result (`bool`) returned to a consuming contract, which is the primary output of the light client's SPV verification system.

---

### Likelihood Explanation

**Likelihood: High.** The bypass requires no privilege, no leaked key, and no social engineering. Any NEAR account can call `verify_transaction_inclusion` directly in a single transaction. Existing integrations that predate v2 are already exposed without any attacker action. A malicious actor who knows the deprecated function exists can trivially route around the protection.

---

### Recommendation

Apply `#[private]` to `verify_transaction_inclusion` so it can only be called from within the contract itself (i.e., from `verify_transaction_inclusion_v2`). Alternatively, remove the public function entirely and inline its logic into `verify_transaction_inclusion_v2`. The deprecation notice alone provides no enforcement and is insufficient as a security boundary.

---

### Proof of Concept

1. Attacker identifies a 64-byte internal Merkle node `N` in a Bitcoin block whose header is tracked by the light client.
2. Attacker constructs a `ProofArgs` with `tx_id = N`, a valid `tx_index`, and a `merkle_proof` path that correctly reconstructs the block's `merkle_root` from `N`.
3. Attacker calls `verify_transaction_inclusion(args)` directly on the NEAR contract — bypassing `verify_transaction_inclusion_v2` and its coinbase proof requirement entirely.
4. The contract computes `compute_root_from_merkle_proof(N, tx_index, merkle_proof)`, which equals `header.block_header.merkle_root`, and returns `true`.
5. Any downstream NEAR contract that trusts this result treats the forged `tx_id` as a confirmed Bitcoin transaction and executes its dependent logic (e.g., releasing bridged funds). [4](#0-3) [5](#0-4)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
