Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Original User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender = msg.sender` of `pool.swap()`, which is the router contract when users route through `MetricOmmSimpleRouter`. If the pool admin allowlists the router address (a natural action to support periphery usage), every unprivileged user can bypass the per-user allowlist by calling any router function, because the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This renders the allowlist invariant completely broken for any pool that supports the standard periphery router.

## Finding Description
**Root cause:** `MetricOmmPool.swap()` passes its own `msg.sender` as the `sender` argument to `_beforeSwap` (lines 230–231 of `MetricOmmPool.sol`). When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router stores the original user only in transient callback context for payment purposes (`_setNextCallbackContext`), then calls `pool.swap()` directly (lines 71–80 of `MetricOmmSimpleRouter.sol`). The pool receives `msg.sender = router` and forwards it to `_beforeSwap(router, ...)`.

`SwapAllowlistExtension.beforeSwap` then evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is the router. The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The original user's address is never visible to the extension.

**Why existing guards fail:** There is no mechanism in the pool or extension interface to propagate the original caller through the router. The `extensionData` field is user-controlled and not authenticated. All four router entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) call `pool.swap()` with `msg.sender = router`, so all are equally affected.

## Impact Explanation
When a pool admin allowlists the router to support periphery usage, the per-user allowlist is fully bypassed: any unprivileged EOA can swap on a pool intended to restrict swaps to KYC'd or protocol-approved addresses by routing through `MetricOmmSimpleRouter`. Swaps drain LP reserves at oracle prices; loss is bounded only by pool liquidity. This is a broken core pool functionality / allowlist invariant failure with direct fund impact.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard documented periphery entry point. A pool admin configuring `SwapAllowlistExtension` who also wants users to use the router will naturally allowlist the router address — there is no on-chain signal that this opens the pool to everyone. The bypass requires no special privilege: any EOA calls `router.exactInputSingle()` with the target pool. The condition is highly likely to occur in production.

## Recommendation
The extension must gate on the economically relevant actor — the original user — not the intermediary contract. Two viable approaches:

1. **Pass original caller through `extensionData` with router-side encoding:** The router encodes `msg.sender` into a dedicated authenticated slot of `extensionData` before calling the pool; the extension decodes and verifies it. Requires a convention between router and extension.
2. **Add `originalSender` to the hook interface:** Add an `originalSender` parameter to `beforeSwap` that the pool populates from a transient-storage context set by the router before calling `pool.swap()`. This is the cleanest separation.

Until fixed, pool admins must not allowlist the router address on pools that use `SwapAllowlistExtension` with per-user restrictions.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin: setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  pool admin: setAllowedToSwap(pool, router, true)  // router allowlisted to support periphery

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})
      → _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, bob, tokenIn)  // bob stored for payment only
      → pool.swap(bob_recipient, zeroForOne, amount, ...)   // msg.sender = router
        → _beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓ passes
          → swap executes for bob

Result:
  bob swaps on a pool that was supposed to block him.
  allowedSwapper[pool][bob] was never checked.
  All four router entry points (exactInputSingle, exactInput, exactOutputSingle, exactOutput)
  are equally affected as all call pool.swap() with msg.sender = router.
```
