### Title
Fast Withdrawal Fee Charged to `msg.sender` Instead of `sendTo` Recipient — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

In `BaseWithdrawPool.submitFastWithdrawal`, when the resolved recipient `sendTo` differs from `msg.sender`, the fast-withdrawal fee is incorrectly pulled from `msg.sender` (the caller) rather than being deducted from the user's `transferAmount`. This is the exact same wrong-party-for-asset-movement pattern as M-09, applied to fee collection instead of reward distribution.

---

### Finding Description

`submitFastWithdrawal` resolves the intended recipient from the signed withdrawal transaction: [1](#0-0) 

The fee branch then reads: [2](#0-1) 

Two asymmetric paths exist:

| Condition | Fee source | User receives |
|---|---|---|
| `sendTo == msg.sender` | Deducted from `transferAmount` | `transferAmount − fee` |
| `sendTo != msg.sender` | Pulled from `msg.sender` via `safeTransferFrom` | Full `transferAmount` |

In the second branch — which is the **normal** fast-withdrawal case where a third-party provider calls on behalf of the user — the caller is charged the fee while the user receives the full amount. The fee should always be deducted from `transferAmount` (the user's withdrawal proceeds), regardless of who calls the function.

The `sendTo` value is resolved from the signed transaction:

- For `WithdrawCollateral` (V1): `sendTo = address(uint160(bytes20(signedTx.tx.sender)))` [3](#0-2) 
- For `WithdrawCollateralV2`: `sendTo = signedTx.tx.sendTo` if non-zero, else sender address [4](#0-3) 

In both cases `sendTo` is the user's address, not the caller's. The caller is a fast-withdrawal provider who is a distinct third party.

---

### Impact Explanation

1. **Fee charged to wrong party**: The fast-withdrawal provider (`msg.sender`) pays the protocol fee out of pocket instead of it being deducted from the user's withdrawal amount.
2. **User receives excess funds**: The user receives the full `transferAmount` without bearing the fee cost, breaking the intended fee model.
3. **Composability broken**: Any external contract or off-chain system that integrates fast withdrawals and expects the fee to be deducted from `transferAmount` will observe incorrect net amounts.
4. **Economic disincentive**: Providers calling `submitFastWithdrawal` when `sendTo != msg.sender` must hold and approve fee tokens, making the feature economically unattractive and potentially non-functional for the standard third-party provider use case.

---

### Likelihood Explanation

The `sendTo != msg.sender` branch is the **primary** fast-withdrawal path: a third-party provider calls the function on behalf of a user whose address is embedded in the signed transaction. Every such call triggers the bug. The fee magnitude is non-trivial — it is the maximum of a proportional rate and five times the standard withdrawal fee: [5](#0-4) 

---

### Recommendation

Remove the conditional and always deduct the fee from `transferAmount`:

```diff
-    if (sendTo == msg.sender) {
-        require(transferAmount > uint128(fee), "Fee larger than balance");
-        transferAmount -= uint128(fee);
-    } else {
-        safeTransferFrom(token, msg.sender, uint128(fee));
-    }
+    require(transferAmount > uint128(fee), "Fee larger than balance");
+    transferAmount -= uint128(fee);
```

This ensures the fee is always borne by the user's withdrawal proceeds, regardless of who calls the function, consistent with the `sendTo == msg.sender` case.

---

### Proof of Concept

1. User signs a `WithdrawCollateralV2` transaction: `sender = userSubaccount`, `sendTo = userAddress`, `amount = 1000e6`.
2. Fast-withdrawal provider (`providerAddress`) calls `submitFastWithdrawal(idx, transaction, signatures)`.
3. `resolveFastWithdrawal` returns `sendTo = userAddress`, `transferAmount = 1000e6`.
4. `fee = fastWithdrawalFeeAmount(...)` computes, e.g., `5e6`.
5. Since `sendTo (userAddress) != msg.sender (providerAddress)`, the code executes `safeTransferFrom(token, providerAddress, 5e6)` — pulling the fee from the provider.
6. `handleWithdrawTransfer(token, userAddress, 1000e6)` sends the **full** amount to the user.
7. Result: provider loses `5e6` in fees; user receives `1000e6` instead of `995e6`. The fee model is inverted. [6](#0-5)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
