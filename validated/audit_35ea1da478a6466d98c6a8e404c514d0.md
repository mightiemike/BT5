### Title
Linked Signer Race Condition Allows Malicious Signer to Drain Subaccount Before Revocation — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type in Nado's fast-mode path is processed with `allowLinkedSigner=true`, meaning the **current linked signer can sign a new `LinkSigner` transaction on behalf of the subaccount owner**. This creates a direct analog to the ERC20 `approve` race condition: when a subaccount owner submits a `LinkSigner` to revoke or replace their linked signer, the current linked signer can race to consume the same nonce first — either by submitting a `WithdrawCollateral` to drain funds, or a competing `LinkSigner` to keep themselves authorized — before the revocation is mined.

---

### Finding Description

In `EndpointTx.sol`, the fast-mode handler for `LinkSigner` transactions calls `validateSignedTx` with `allowLinkedSigner=true`:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // ← allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

This means the current linked signer (`Bob`) can sign a `LinkSigner` transaction on behalf of Alice's subaccount. The nonce used is Alice's address-level nonce:

```solidity
function validateNonce(bytes32 sender, uint64 nonce) internal virtual {
    require(
        nonce == nonces[address(uint160(bytes20(sender)))]++,
        ERR_WRONG_NONCE
    );
}
``` [2](#0-1) 

Because the nonce is shared between Alice's own transactions and any transaction Bob signs on her behalf, Bob can submit a transaction consuming nonce `N` before Alice's revocation (also at nonce `N`) is processed by the sequencer.

The `WithdrawCollateral` fast-mode handler also uses `allowLinkedSigner=true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true
);
``` [3](#0-2) 

The linked signer mapping is stored in `EndpointStorage`:

```solidity
mapping(bytes32 => address) internal linkedSigners;
``` [4](#0-3) 

---

### Impact Explanation

A malicious linked signer can:

1. **Drain collateral**: Submit `WithdrawCollateral(sender=alice_sub, amount=MAX, nonce=N)` signed by Bob before Alice's `LinkSigner(signer=address(0), nonce=N)` is processed. Bob's transaction consumes nonce `N`, Alice's revocation fails with `ERR_WRONG_NONCE`.
2. **Prevent revocation indefinitely**: Submit `LinkSigner(sender=alice_sub, signer=Bob, nonce=N)` to keep themselves as linked signer. Alice's revocation fails; Bob repeats the attack on every subsequent revocation attempt.
3. **Redirect funds**: Use `TransferQuote(sender=alice_sub, recipient=bob_sub, amount=MAX, nonce=N)` to move quote balances to a Bob-controlled subaccount.

The `TransferQuote` fast-mode handler also uses `allowLinkedSigner=true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true
);
``` [5](#0-4) 

The corrupted state delta is: `spotEngine` balance for Alice's subaccount reduced to zero, or `linkedSigners[alice_sub]` permanently locked to Bob's address.

---

### Likelihood Explanation

- The linked signer pattern is a documented, supported feature of the protocol. Users are expected to link signers for automated trading bots or API keys.
- A malicious linked signer (e.g., a compromised API key, a rogue trading bot, or a counterparty who was granted signing rights) can monitor the sequencer's pending transaction queue for a revocation attempt and submit a competing transaction immediately.
- The sequencer processes transactions in submission order. If Bob submits before Alice's revocation is included in a batch, Bob's transaction is processed first.
- No privileged access is required beyond the linked signer role itself, which is granted by the user.
- The attack is repeatable: every time Alice increments her nonce to retry the revocation, Bob can consume the new nonce again.

---

### Recommendation

1. **Disallow linked signers from signing `LinkSigner` transactions**: Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type. Only the subaccount owner's own key should be able to change the linked signer mapping. This is the direct mitigation — analogous to requiring `allowed[msg.sender][spender] == 0` before a new `approve`.

```solidity
// Change from:
validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, true);
// To:
validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, false);
``` [6](#0-5) 

2. **Consider a time-lock or two-step revocation**: Introduce a pending revocation state that the linked signer cannot override, giving the subaccount owner a guaranteed revocation window.

3. **Separate nonce spaces**: Use a separate nonce counter for owner-signed vs. linked-signer-signed transactions so that a linked signer cannot consume the owner's revocation nonce.

---

### Proof of Concept

**Setup**: Alice's subaccount is `alice_sub`. `linkedSigners[alice_sub] = Bob`. Alice's current nonce is `N`.

**Step 1**: Alice decides to revoke Bob. She signs and submits to the sequencer:
```
LinkSigner { sender: alice_sub, signer: address(0), nonce: N }
signed by: Alice's private key
```

**Step 2**: Bob observes the pending revocation in the sequencer queue. Bob signs and submits:
```
WithdrawCollateral { sender: alice_sub, productId: QUOTE, amount: MAX_BALANCE, nonce: N }
signed by: Bob's private key (as linked signer for alice_sub)
```

**Step 3**: The sequencer processes Bob's `WithdrawCollateral` first (Bob submitted it earlier in the batch). Nonce `N` is consumed. Alice's entire collateral balance is withdrawn to Bob's address.

**Step 4**: Alice's `LinkSigner` transaction is processed next. `validateNonce` checks `N == nonces[alice_address]` but nonces is now `N+1`. Transaction reverts with `ERR_WRONG_NONCE`.

**Step 5**: Alice submits `LinkSigner { nonce: N+1 }`. Bob repeats Step 2 with `WithdrawCollateral { nonce: N+1 }`. The cycle continues until Alice has no funds left.

The root cause — `allowLinkedSigner=true` on `LinkSigner` — also enables the variant where Bob submits `LinkSigner { sender: alice_sub, signer: Bob, nonce: N }` to consume Alice's nonce while keeping himself as the authorized signer, blocking revocation entirely. [1](#0-0) [2](#0-1) [4](#0-3)
