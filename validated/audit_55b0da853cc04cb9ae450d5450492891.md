### Title
Duplicate-Leaf Merkle Padding Allows Phantom Transaction Inclusion Proof — (`btc-types/src/contract_args.rs`, `merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` accepts a `tx_index` that points to a phantom duplicate-leaf position in an odd-length block's Merkle tree. Because the contract stores only the block header (containing `merkle_root`) and never the transaction count, there is no guard preventing an attacker from supplying `tx_index = N+1` for the last transaction `tx[N]` in an odd-length block. The same Merkle proof that is valid for the real position `N` is also valid for the phantom position `N+1`, so the function returns `true` for a transaction that does not exist.

---

### Finding Description

Bitcoin's Merkle tree construction duplicates the last leaf when the number of transactions is odd. For a 3-transaction block `[tx0, tx1, tx2]`:

```
Level 0 (leaves): [tx0, tx1, tx2, tx2*]   ← tx2 duplicated
Level 1:          [H(tx0,tx1), H(tx2,tx2*)]
Level 2 (root):   H(H(tx0,tx1), H(tx2,tx2*))  = merkle_root
```

`merkle_proof_calculator` implements this duplication: [1](#0-0) 

`compute_root_from_merkle_proof` uses only `transaction_position % 2` to decide left/right at each level: [2](#0-1) 

For `tx2` at real position `2` (even), the proof is `[tx2, H(tx0,tx1)]`:
- Step 1: pos=2 (even) → `hash(tx2, tx2)` = `H(tx2,tx2)`
- Step 2: pos=1 (odd) → `hash(H(tx0,tx1), H(tx2,tx2))` = `merkle_root` ✓

For the **phantom** position `3` (odd), the **same proof** `[tx2, H(tx0,tx1)]` also works:
- Step 1: pos=3 (odd) → `hash(tx2, tx2)` = `H(tx2,tx2)` ← same intermediate result
- Step 2: pos=1 (odd) → `hash(H(tx0,tx1), H(tx2,tx2))` = `merkle_root` ✓

`verify_transaction_inclusion_v2` performs three checks: [3](#0-2) 

1. **Length equality** of `merkle_proof` and `coinbase_merkle_proof` — passes (both length 2).
2. **Coinbase proof** against `header.block_header.merkle_root` — passes (coinbase proof is independent of `tx_index`).
3. Delegates to `verify_transaction_inclusion`, which only checks `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.block_header.merkle_root` — passes for phantom index 3. [4](#0-3) 

The contract stores only the block header (with `merkle_root`), never the transaction count. `ProofArgsV2.tx_index` is an unconstrained `u64`: [5](#0-4) 

There is no check that `tx_index < num_transactions_in_block` anywhere in the verification path.

---

### Impact Explanation

`verify_transaction_inclusion_v2` returns `true` for a phantom transaction at index `N+1` that does not exist in the block. Any on-chain or off-chain consumer that relies on this function to confirm a Bitcoin transaction was included in a block (e.g., bridge unlock, cross-chain settlement, proof-of-payment) can be deceived into accepting a fabricated inclusion proof. This is a direct false transaction inclusion verification result.

---

### Likelihood Explanation

- No privileged access is required. `verify_transaction_inclusion_v2` is a public NEAR contract view/call.
- Any odd-length Bitcoin block (the majority of real blocks) can be used.
- The attacker only needs the block's transaction list (publicly available) to construct the proof.
- The coinbase proof is straightforward to compute from the same public data.
- The attack is deterministic and requires no brute force.

---

### Recommendation

Add a `tx_count` field to the block header storage or to `ProofArgsV2`, and enforce `tx_index < tx_count` before accepting the proof. Alternatively, require that the `merkle_proof` length equals `ceil(log2(tx_count))` and that `tx_index < tx_count`, where `tx_count` is committed to at block submission time. The coinbase-proof mitigation already present addresses the 64-byte forgery attack but does **not** address phantom-index forgery.

---

### Proof of Concept

**3-transaction block:** `[tx0 (coinbase), tx1, tx2]`

Compute:
- `H01 = double_sha256(tx0 || tx1)`
- `H22 = double_sha256(tx2 || tx2)`
- `merkle_root = double_sha256(H01 || H22)`

Submit the block header (with this `merkle_root`) to the contract.

Call `verify_transaction_inclusion_v2` with:
```
ProofArgsV2 {
    tx_id:                  tx2,
    tx_block_blockhash:     block_hash,
    tx_index:               3,          // phantom — no tx exists here
    merkle_proof:           [tx2, H01], // valid proof for real index 2, also valid for index 3
    coinbase_tx_id:         tx0,
    coinbase_merkle_proof:  [tx1, H22], // valid coinbase proof
    confirmations:          0,
}
```

Verification trace:
1. Length check: 2 == 2 ✓
2. Coinbase: `compute_root(tx0, 0, [tx1, H22])` → `hash(tx0,tx1)=H01` → `hash(H01,H22)=merkle_root` ✓
3. Tx proof: `compute_root(tx2, 3, [tx2, H01])` → pos=3 odd: `hash(tx2,tx2)=H22` → pos=1 odd: `hash(H01,H22)=merkle_root` ✓

**Result: `true` returned for a non-existent phantom transaction.** [6](#0-5) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
