Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap()` checks the `sender` argument forwarded by the pool, which resolves to `msg.sender` of the `pool.swap()` call — the router contract, not the originating user. A pool admin who allowlists the router so that legitimate users can swap through it inadvertently opens the gate to every unpermissioned address, since any caller can route through the same public `MetricOmmSimpleRouter`. The allowlist is therefore rendered completely ineffective for any pool that supports router-mediated swaps.

## Finding Description
`MetricOmmPool.swap()` unconditionally passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // router address when called via router
    recipient, ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap()` encodes that value verbatim and forwards it to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is the router, so the check resolves to `allowedSwapper[pool][router]`, never `allowedSwapper[pool][actualUser]`.

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, making itself `msg.sender` to the pool: [4](#0-3) 

This creates an inescapable dilemma:

| Router allowlisted? | Effect |
|---|---|
| **No** | Every router-mediated swap reverts — allowlisted users cannot use the router |
| **Yes** | Every router-mediated swap succeeds — any unpermissioned user bypasses the gate |

There is no mechanism in the extension, pool, or router to recover the originating user's address. The `extensionData` field passed by the router is simply forwarded from the caller's `params.extensionData` with no router-injected identity.

## Impact Explanation
`SwapAllowlistExtension` is the sole on-chain mechanism for restricting swap access on a pool. Once the router is allowlisted (a necessary step for any allowlisted user to trade via the router), the gate is open to the entire public. Any unpermissioned address can execute swaps on pools intended to be KYC-gated, whitelist-only, or partner-only. LPs who deposited under the assumption that only vetted counterparties could trade against them are exposed to the full universe of public traders, with direct fund-impacting consequences on adverse selection, fee dynamics, and pool-specific risk controls. This constitutes broken core pool functionality with direct LP principal impact.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps; most real users route through it. The bypass requires no special privileges, no flash loans, and no multi-transaction setup — a single `exactInputSingle` call suffices. Pool admins who want their allowlisted users to be able to use the router will naturally add the router to the allowlist, unknowingly opening the gate to everyone. The mismatch is invisible in normal unit testing because extension tests call it directly from a mock pool with a controlled `sender`, never through the router.

## Recommendation
The extension must gate the economic actor (the originating user), not the intermediate dispatcher. Two complementary approaches:

1. **Pass the originating user through `extensionData`:** Have the router encode `msg.sender` into `extensionData` before forwarding to the pool. The extension decodes and verifies that address against the allowlist. This requires a convention between router and extension but keeps the pool interface unchanged.

2. **Add a dedicated `originalSender` field to the extension interface:** Extend `IMetricOmmExtensions.beforeSwap` with an `originalSender` parameter that the pool populates from a trusted transient-storage slot set by the router (similar to how the router already stores payer context in transient storage for the swap callback). The extension then checks `allowedSwapper[pool][originalSender]`.

Either approach must be paired with a check that the supplied identity is actually the address that called the router, to prevent spoofing via crafted `extensionData`.

## Proof of Concept
```solidity
// Setup:
// 1. Deploy pool with SwapAllowlistExtension
// 2. Admin allowlists Alice (legitimate user) and the router
//    extension.setAllowedToSwap(pool, alice, true);
//    extension.setAllowedToSwap(pool, address(router), true); // needed so Alice can use router
// 3. Bob (not allowlisted) calls the router directly

function testBypassViaRouter() public {
    // Bob is NOT in the allowlist
    assertFalse(extension.isAllowedToSwap(address(pool), bob));

    // But the router IS allowlisted (admin added it so Alice could use the router)
    assertTrue(extension.isAllowedToSwap(address(pool), address(router)));

    // Bob routes through the router — extension sees sender=router, not bob
    vm.prank(bob);
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: token0,
            recipient: bob,
            zeroForOne: true,
            amountIn: 1000,
            amountOutMinimum: 0,
            priceLimitX64: type(uint128).max,
            deadline: block.timestamp,
            extensionData: ""
        })
    );
    // Swap succeeds — allowlist bypassed
}
```
