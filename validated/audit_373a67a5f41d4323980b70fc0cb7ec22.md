### Title
Unbounded `latest_protocol_version` in Block Header Enables Forced Protocol Upgrade to `u32::MAX`, Causing Permanent Chain Halt — (File: `chain/chain/src/chain.rs`)

---

### Summary

The `latest_protocol_version` field present in every `BlockHeader` variant (V1–V7) is validated in `validate_header` only with a **lower bound** — there is no upper bound. A coalition of block producers controlling >80% of stake can set this field to `u32::MAX` (4,294,967,295). The epoch manager's protocol-version voting logic accepts this value verbatim, schedules an "upgrade" to `u32::MAX` for epoch T+2, and then the lower-bound check in `validate_header` permanently rejects every subsequent block produced by any honest validator binary (whose `PROTOCOL_VERSION` is at most 200), halting the chain.

---

### Finding Description

**Step 1 — The unvalidated field.**

`latest_protocol_version` is a `u32` field in every `BlockHeaderInnerRest*` struct: [1](#0-0) 

It is exposed via: [2](#0-1) 

**Step 2 — Only a lower bound is checked.**

`validate_header` enforces that `latest_protocol_version` is not *below* the current epoch version, but imposes **no upper bound**: [3](#0-2) 

A block producer can freely set `latest_protocol_version = u32::MAX` and the block will pass this check (since `u32::MAX >= epoch_protocol_version`).

**Step 3 — Raw value flows into `BlockInfo` without clamping.**

`BlockInfo::from_header` copies the field verbatim: [4](#0-3) [5](#0-4) 

**Step 4 — `EpochInfoAggregator` records it in `version_tracker`.** [6](#0-5) 

**Step 5 — `collect_blocks_info` selects the winning version with no upper bound.**

The version with the most stake wins and becomes `next_next_epoch_version`. No cap is applied: [7](#0-6) 

**Step 6 — All validators below `u32::MAX` are kicked out.** [8](#0-7) 

**Step 7 — Epoch T+2 is committed with `protocol_version = u32::MAX`.**

`next_next_epoch_version` is passed directly to `proposals_to_epoch_info` and stored as the epoch's protocol version: [9](#0-8) [10](#0-9) 

**Step 8 — Chain halt.**

Once epoch T+2 is active with `epoch_protocol_version = u32::MAX`, the lower-bound check at `validate_header` rejects every block from every honest validator (whose binary has `PROTOCOL_VERSION ≤ 200 < u32::MAX`). No further blocks can be produced. The chain halts permanently.

The current stable `PROTOCOL_VERSION` is 86, nightly is 156, and Spice is 200 — all far below `u32::MAX`: [11](#0-10) 

---

### Impact Explanation

**Critical.** Once the malicious epoch is finalized, the chain cannot produce any further blocks. Every honest validator's block header will fail the lower-bound check in `validate_header` because their `latest_protocol_version` (≤ 200) is less than `u32::MAX`. The damage is irreversible without a coordinated out-of-band hard fork.

---

### Likelihood Explanation

The attack requires a coalition of block producers holding >80% of total block-producer stake to vote for `u32::MAX` across one epoch. This is a high threshold, but:

- A single entity controlling a supermajority (e.g., a compromised or colluding validator set) can execute it unilaterally.
- The attack is silent — setting `latest_protocol_version = u32::MAX` in a signed block header is indistinguishable from a normal block until the epoch boundary.
- There is no on-chain mechanism to detect or reject the anomalous value before the epoch is finalized.

---

### Recommendation

Add an upper-bound check in `validate_header` in `chain/chain/src/chain.rs`, immediately after the existing lower-bound check:

```rust
// Existing lower bound
if header.latest_protocol_version() < epoch_protocol_version {
    return Err(Error::InvalidProtocolVersion);
}
// NEW: upper bound — reject versions beyond what any supported binary can claim
if header.latest_protocol_version() > PROTOCOL_VERSION {
    return Err(Error::InvalidProtocolVersion);
}
```

Alternatively, clamp `next_next_epoch_version` in `collect_blocks_info` (`chain/epoch-manager/src/lib.rs`) to `min(voted_version, PROTOCOL_VERSION)` before it is stored as the epoch's protocol version.

---

### Proof of Concept

1. A coalition of validators with >80% of block-producer stake produces blocks in epoch T with `latest_protocol_version = 4294967295` (`u32::MAX`).
2. Each such block passes `validate_header` because `u32::MAX >= epoch_protocol_version` (e.g., 86).
3. `BlockInfo::from_header` stores `latest_protocol_version = u32::MAX` for each such block.
4. `EpochInfoAggregator::update_tail` records `version_tracker[producer_id] = u32::MAX`.
5. At the end of epoch T, `collect_blocks_info` selects `u32::MAX` as `next_next_epoch_version` (it has >80% stake weight and exceeds the upgrade threshold).
6. All validators with `version < u32::MAX` (i.e., every honest validator) are added to `validator_kickout` with `ProtocolVersionTooOld`.
7. Epoch T+2 is saved with `protocol_version = u32::MAX`.
8. When epoch T+2 becomes active, `validate_header` rejects every block from every honest validator because `latest_protocol_version (≤ 200) < u32::MAX`. The chain halts permanently.
