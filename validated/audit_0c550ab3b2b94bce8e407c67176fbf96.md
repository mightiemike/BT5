### Title
NEAR Validator-Manipulable `env::block_timestamp_ms()` Used as Trusted Local Time in Bitcoin Header Timestamp Validation — (File: `contract/src/bitcoin.rs`)

---

### Summary

The `check_pow` function in `contract/src/bitcoin.rs` uses `env::block_timestamp_ms()` — a NEAR-validator-controlled on-chain value — as the authoritative "local time" reference to enforce the future-timestamp bound on submitted Bitcoin block headers. A malicious NEAR validator can manipulate this value when including a `submit_blocks` transaction, causing the contract to accept Bitcoin headers whose timestamps exceed the intended `MAX_FUTURE_BLOCK_TIME_LOCAL` bound. Accepted headers with inflated timestamps corrupt difficulty adjustment calculations, producing incorrect chainwork values that corrupt fork-choice decisions and SPV proof verification results.

---

### Finding Description

In `contract/src/bitcoin.rs`, `check_pow` enforces that a submitted Bitcoin block header's timestamp is not too far in the future:

```rust
// contract/src/bitcoin.rs, lines 35–39
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
    "time-too-new: block timestamp too far in the future"
);
```

`env::block_timestamp_ms()` is the timestamp of the NEAR block in which the transaction is executed. NEAR validators set this value; it is not derived from any external oracle or cryptographic commitment. A NEAR validator can set `env::block_timestamp_ms()` to an artificially high value (within NEAR protocol bounds) when including a `submit_blocks` call in a NEAR block.

When the validator inflates `env::block_timestamp_ms()`, the effective upper bound `current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL` is raised by the same delta. A Bitcoin header whose `time` field exceeds the honest upper bound — but falls within the inflated bound — passes the check and is stored in `headers_pool` as a valid header.

The same pattern appears in `contract/src/zcash.rs` at lines 48–52, where `env::block_timestamp_ms()` is used identically for the Zcash `check_pow` path.

The stored header's inflated `time` value then feeds directly into difficulty adjustment. In `calculate_next_work_required` (`contract/src/bitcoin.rs`, lines 90–118):

```rust
let mut actual_time_taken: i64 = prev_block_time - first_block_time;
```

If `prev_block_time` is inflated due to an accepted future-timestamp header, `actual_time_taken` is larger than the true elapsed time, causing the computed difficulty target to be lower (easier) than it should be. The resulting `bits` value is stored in the `ExtendedHeader` and used to compute `chain_work` via `work_from_bits(header.bits)` in `submit_block_header` (`contract/src/lib.rs`, lines 505–507):

```rust
let (current_block_computed_chain_work, overflow) = prev_block_header
    .chain_work
    .overflowing_add(work_from_bits(header.bits));
```

A lower difficulty target produces a lower `work_from_bits` value (since `work_from_bits` computes `target.inverse()`), meaning the corrupted chain accumulates less chainwork per block than it should. This corrupts the canonical chain determination in `submit_block_header_inner` (`contract/src/lib.rs`, lines 562–566):

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
```

---

### Impact Explanation

An accepted Bitcoin header with an inflated timestamp corrupts the `actual_time_taken` used in difficulty retargeting. This produces an incorrect `bits` value for the next difficulty epoch, which in turn produces incorrect `chain_work` values for all subsequent headers. Corrupted `chain_work` values directly control which chain is treated as the canonical mainchain. SPV proof verification (`verify_transaction_inclusion` and `verify_transaction_inclusion_v2`) checks that the target block belongs to `mainchain_header_to_height`, which is determined by the corrupted fork-choice logic. A recipient contract consuming a `true` result from `verify_transaction_inclusion` on a corrupted canonical chain may act on a fraudulent transaction inclusion proof.

---

### Likelihood Explanation

The attack requires a malicious NEAR validator to manipulate `env::block_timestamp_ms()` when including a `submit_blocks` transaction. The `submit_blocks` entrypoint is restricted to trusted relayers via `#[trusted_relayer]`, but the timestamp manipulation is applied at the NEAR block level by the validator — independently of who submitted the transaction. A colluding trusted relayer submitting a Bitcoin header with a timestamp beyond the honest bound, combined with a malicious NEAR validator inflating `env::block_timestamp_ms()` to cover the gap, is the concrete trigger. This is the direct analog of the original report: a validator manipulating an on-chain time value to influence a security-critical check. Likelihood is medium-low given the requirement for a malicious NEAR validator, but the attack is structurally identical to the reported vulnerability class.

---

### Recommendation

Replace `env::block_timestamp_ms()` with a manipulation-resistant time source for the future-timestamp check. The standard Bitcoin approach is to use the Median Time Past (MTP) of the previous 11 blocks — already computed in `get_median_time_past` (`contract/src/utils.rs`) — as the reference time, rather than the local clock. Alternatively, bound the future-timestamp check against the MTP of the submitted header's ancestor chain, which is fully determined by the submitted Bitcoin headers and cannot be manipulated by a NEAR validator.

---

### Proof of Concept

1. A trusted relayer prepares a Bitcoin block header with `time = T_honest + MAX_FUTURE_BLOCK_TIME_LOCAL + Δ` (where `Δ > 0` is the desired excess).
2. The relayer submits `submit_blocks([header])` to the NEAR contract.
3. A malicious NEAR validator includes this transaction in a NEAR block with `env::block_timestamp_ms()` set such that `env::block_timestamp_ms() / 1000 >= T_honest + Δ`.
4. Inside `check_pow` (`contract/src/bitcoin.rs`, line 35), `current_timestamp = env::block_timestamp_ms() / 1000 >= T_honest + Δ`.
5. The check `block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL` evaluates to `T_honest + MAX_FUTURE_BLOCK_TIME_LOCAL + Δ <= T_honest + Δ + MAX_FUTURE_BLOCK_TIME_LOCAL`, which is `true`.
6. The header passes and is stored in `headers_pool` with its inflated `time`.
7. At the next difficulty adjustment boundary (every 2016 blocks for Bitcoin), `calculate_next_work_required` computes `actual_time_taken` using this inflated `prev_block_time`, producing a lower difficulty target and lower `chain_work` per block for all subsequent headers.
