### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` Permanently Blocks Finalization When `openInterest == 0` — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` divides by `state.openInterest` without a zero-guard. When all open positions in a perp market are closed after an underwater subaccount's position has been liquidated to zero, `openInterest` reaches 0 while the subaccount still carries a negative `vQuoteBalance`. Any subsequent finalization attempt reverts permanently, locking the bad debt and blocking insurance fund recovery.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, the socialization path computes:

```solidity
int128 fundingPerShare = -balance.vQuoteBalance.div(
    state.openInterest   // ← no zero-guard
);
```

`MathSD21x18.div` explicitly enforces `require(y != 0, ERR_DIV_BY_ZERO)`. [1](#0-0) 

`openInterest` is maintained as the sum of absolute position sizes across all participants, updated in `_updateBalance`: [2](#0-1) 

A subaccount with `balance.amount == 0` contributes nothing to `openInterest`. The finalization precondition in `_finalizeSubaccount` **requires** `balance.amount == 0` for every perp product of the liquidatee: [3](#0-2) 

Therefore, the liquidatee's own position never prevents `openInterest` from reaching 0. If every other participant also closes their position, `openInterest == 0` while the liquidatee still holds a negative `vQuoteBalance` (residual loss after position closure). The division at line 166–168 then reverts: [4](#0-3) 

This revert propagates through `_finalizeSubaccount` (line 386–389) and `liquidateSubaccountImpl` (line 620), making every future finalization attempt for that subaccount revert as well: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

The underwater subaccount can never be finalized. Its bad debt is permanently locked in the protocol. The insurance fund cannot cover or socialize the loss. `SpotEngine.socializeSubaccount` (which would spread the remaining liability across depositors) is also never reached, so depositors are not made whole either. This constitutes a permanent lock of insurance fund recovery and protocol-controlled assets.

---

### Likelihood Explanation

The attack requires only normal market operations — opening positions, executing liquidations, and closing positions — all of which are permissionless. A single attacker controlling two accounts (one long, one liquidator) can deterministically manufacture the `openInterest == 0` condition in any perp market before triggering finalization. No admin access, oracle manipulation, or governance capture is needed. The attack is especially easy in thin or newly listed markets.

---

### Recommendation

Add a zero-guard before the division in `socializeSubaccount`. If `state.openInterest == 0` and `balance.vQuoteBalance < 0`, there are no other participants to socialize against; the loss should be absorbed entirely by the insurance fund or handled as a special case (e.g., skip socialization and zero out the balance, letting the insurance/spot socialization path cover it):

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; zero out and let
        // insurance/spot socialization absorb the residual loss.
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

---

### Proof of Concept

1. Deploy the protocol with one perp product (productId = P).
2. Account A opens long 1 BTC; Account V opens short 1 BTC. `openInterest == 2`.
3. Price drops; V becomes underwater (maintenance health < 0).
4. Liquidator L liquidates V's entire short position. After full liquidation: V has `balance.amount == 0`, `vQuoteBalance < 0`; L holds `balance.amount == -1`. `openInterest == 2`.
5. A (sell) and L (buy) execute a matching trade, closing both positions. `openInterest == 0`.
6. Call `liquidateSubaccountImpl` with `txn.productId == type(uint32).max` (finalization path) targeting V.
7. `_finalizeSubaccount` passes the `balance.amount == 0` check, reaches `perpEngine.socializeSubaccount(V, insurance)`.
8. Inside `socializeSubaccount`: `balance.vQuoteBalance < 0` is true, insurance is insufficient, execution reaches `fundingPerShare = -balance.vQuoteBalance.div(state.openInterest)` with `state.openInterest == 0`.
9. `MathSD21x18.div` reverts with `"DBZ"`. The finalization transaction reverts. V's bad debt is permanently locked.
