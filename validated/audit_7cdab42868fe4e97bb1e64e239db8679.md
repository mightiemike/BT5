Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual End User, Allowing Any User to Bypass the Swap Allowlist When the Router Is Allowlisted — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the actual end user. If the pool admin allowlists the router address, every user — including those not individually allowlisted — can bypass the per-user gate by going through the router.

## Finding Description
**Pool passes `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, ...)` at line 231: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim as the `sender` argument to every configured extension: [2](#0-1) 

**Extension checks `sender`, not the real user.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` = pool and `sender` = whoever called `pool.swap()`: [3](#0-2) 

**Router replaces the real user as `sender`.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly, making the router contract `msg.sender` at the pool: [4](#0-3) 

At the moment `pool.swap` executes, `msg.sender` is the router. The pool therefore passes `sender = router` to the extension. The extension checks `allowedSwapper[pool][router]` — the actual end user's address is never consulted. The same flaw applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**`DepositAllowlistExtension` does not share this flaw** — it checks `owner` (the position owner supplied by the caller), which is independent of the intermediary contract: [5](#0-4) 

## Impact Explanation
Any user who is not individually allowlisted can trade on a pool that is supposed to be restricted (e.g., KYC-gated, institutional-only) simply by routing through `MetricOmmSimpleRouter`. Because the pool is oracle-anchored and the allowlist is the only access barrier, the bypass gives the non-allowlisted user full swap access: they can extract token output from LP bins at oracle-derived prices, causing direct LP principal loss on a pool whose operators believed it was access-controlled. The allowlist guard — the sole mechanism protecting LP assets on a restricted pool — is rendered ineffective for all router-mediated paths. This constitutes a direct loss of user principal (LP assets) above Sherlock thresholds, qualifying as High severity.

## Likelihood Explanation
A pool admin who deploys a restricted pool and also wants to support the standard router UX will naturally add the router to the allowlist. The admin's mental model ("the router is a trusted intermediary; individual users are still gated") is not what the code enforces. This is a plausible and realistic misconfiguration. The router is a public, permissionless contract, so once it is allowlisted, the bypass is available to any address on-chain with no further preconditions. No special privileges or insider knowledge are required.

## Recommendation
The extension must check the identity of the actual end user, not the intermediary. Two viable approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` (the real user) into `extensionData` before calling `pool.swap`. The extension decodes and checks that address instead of (or in addition to) `sender`.

2. **Check `recipient` instead of `sender`**: For swap allowlists the economically relevant actor is often the recipient of output tokens. Checking `recipient` would not be spoofable by the router (the user controls `params.recipient`), though it shifts the semantic from "who initiates" to "who receives".

Either way, the extension must be updated to gate the address the pool admin actually intends to restrict, and the router must be updated to forward that address faithfully.

## Proof of Concept
```
Setup
─────
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Admin allowlists alice (KYC'd user):
       setAllowedToSwap(pool, alice, true)
3. Admin allowlists the router (to support router UX for alice):
       setAllowedToSwap(pool, router, true)

Attack
──────
4. bob (not individually allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: bob, ...})

5. Router calls pool.swap(bob, ...) — msg.sender at pool = router.

6. Pool calls _beforeSwap(sender=router, ...).

7. Extension evaluates:
       allowedSwapper[pool][router]  →  true   ✓
   The check passes. bob's address is never consulted.

8. Swap executes. bob receives output tokens from LP bins.
   LPs suffer a loss on a pool they believed was KYC-restricted.
```

Foundry test outline:
- Deploy pool with `SwapAllowlistExtension` configured.
- Call `setAllowedToSwap(pool, router, true)` as pool admin.
- Call `MetricOmmSimpleRouter.exactInputSingle` from an address not in the allowlist.
- Assert the swap succeeds (no `NotAllowedToSwap` revert), confirming the bypass.
