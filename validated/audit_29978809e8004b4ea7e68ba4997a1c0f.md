Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Authenticates the Router Address Instead of the Originating User, Allowing Any Caller to Bypass a Pool's Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user. A pool admin who allowlists the router (the only way to permit legitimate router-based swaps) simultaneously grants every unpermissioned user the ability to bypass the allowlist by calling the same public router, silently voiding the curation guarantee the extension is meant to enforce.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever address called `pool.swap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the pool's `msg.sender`: [3](#0-2) 

The same pattern applies to `exactInput` (all hops) and `exactOutputSingle`: [4](#0-3) 

The extension therefore sees the router's address as the swapper identity. The pool admin faces an impossible choice: not allowlisting the router breaks all router-based swaps for legitimate users; allowlisting the router opens the allowlist to every caller of the public router. There is no existing guard in `SwapAllowlistExtension` or `MetricOmmPool` that recovers the original economic actor's identity from `extensionData` or any other source.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to KYC'd or otherwise curated addresses is fully bypassable by any unpermissioned user calling `MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, or `exactOutputSingle`. The attacker receives the same swap execution — same oracle price, same bins, same output tokens — as an allowlisted direct caller. The allowlist guard is silently voided for the entire pool, constituting a broken core pool functionality and an admin-boundary break: the pool admin's access-control configuration is bypassed by an unprivileged path without any special privilege, custom contract, or unusual token behavior.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Any user who calls the router against a pool that has allowlisted the router (required for legitimate use) bypasses the guard with a single standard router call. No special privilege, flash loan, or unusual token behavior is required. The bypass is repeatable on every block.

## Recommendation

The pool must convey the original economic actor's identity to the extension. Two sound approaches:

1. **Pass the originating user through `extensionData`**: Require the router to ABI-encode `msg.sender` (the user) into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it. This requires a convention between router and extension but is the cleanest long-term fix, especially if a standardized `extensionData` header is adopted across all periphery contracts.

2. **Gate on `recipient` instead of `sender`**: If the pool admin's intent is to restrict who receives output tokens, changing the extension to check the `recipient` argument (the second parameter of `beforeSwap`) would correctly identify the beneficiary regardless of routing path, without requiring any router changes.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (required so that legitimate users can use the router)
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  1. attacker calls MetricOmmSimpleRouter.exactInputSingle({
         pool: curated_pool,
         recipient: attacker,
         zeroForOne: true,
         amountIn: X,
         ...
     })
  2. Router calls pool.swap(attacker, true, X, ...) with msg.sender = router
  3. Pool calls _beforeSwap(router, attacker, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  5. Swap executes; attacker receives output tokens at oracle price
  6. Allowlist bypassed; attacker was never individually permitted

Foundry test outline:
  - deployPool with SwapAllowlistExtension configured
  - vm.prank(poolAdmin); ext.setAllowedToSwap(pool, router, true)
  - vm.prank(attacker); router.exactInputSingle(...)
  - assert swap succeeds and attacker receives tokens
  - assert ext.isAllowedToSwap(pool, attacker) == false
```
