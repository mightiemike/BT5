### Title
Unauthenticated `ContractCodeResponse::V1` Accepted Pre-`SignedContractCodeResponse` Feature — (`chain/client/src/stateless_validation/validate.rs`)

### Summary

`validate_contract_code_response` conditionally skips all sender-authentication when the epoch's protocol version is below `SignedContractCodeResponse` (version 85). Any network peer can send a `ContractCodeResponse::V1` (Borsh discriminant `0x00`) carrying arbitrary contract bytecode, which is accepted and stored in the partial-witness tracker without any identity or integrity check. A validator that uses this poisoned code to execute the state transition will compute a wrong state root and fail to endorse the chunk.

### Finding Description

`validate_contract_code_response` in `chain/client/src/stateless_validation/validate.rs` reads the epoch's protocol version and only calls `validate_witness_contract_code_response_signature` when `ProtocolFeature::SignedContractCodeResponse` is enabled:

```rust
pub fn validate_contract_code_response(...) -> Result<ChunkRelevance, Error> {
    let key = response.chunk_production_key();
    require_relevant!(validate_chunk_relevant(epoch_manager, key, store)?);
    let protocol_version = epoch_manager.get_epoch_protocol_version(&key.epoch_id)?;
    if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
        validate_witness_contract_code_response_signature(epoch_manager, response)?;
    }
    Ok(ChunkRelevance::Relevant)   // ← returned unconditionally pre-feature
}
``` [1](#0-0) 

When the feature is **not** enabled, the function returns `ChunkRelevance::Relevant` with no authentication. The caller immediately decompresses and stores the contracts:

```rust
fn handle_contract_code_response(&self, response: ContractCodeResponse) -> Result<(), Error> {
    if !validate_contract_code_response(...)?. is_relevant() { return Ok(()); }
    let key = response.chunk_production_key().clone();
    let contracts = response.decompress_contracts()?;
    self.partial_witness_tracker.store_accessed_contract_codes(key, contracts)
}
``` [2](#0-1) 

The `ContractCodeResponse` enum has two variants. `V1` carries no signature field and `verify_signature` on it always returns `false` (it is never called pre-feature):

```rust
pub enum ContractCodeResponse {
    V1(ContractCodeResponseV1) = 0,   // ← no signature, no responder field
    V2(ContractCodeResponseV2) = 1,   // ← signed
}
``` [3](#0-2) 

`ContractCodeResponseV1` contains only `next_chunk: ChunkProductionKey` and `compressed_contracts: CompressedContractCode` — no identity field whatsoever: [4](#0-3) 

`SignedContractCodeResponse` is assigned protocol version 85 (the current stable version), meaning the unauthenticated path is active during the epoch-transition window when the epoch protocol version is still 84: [5](#0-4) 

By contrast, `validate_contract_code_request` (the inbound request path) **always** verifies the requester's signature, with no protocol-version gate: [6](#0-5) 

This asymmetry is the exact analog of the Ethereum bridge bug: the request side checks its "return value" (signature), but the response side does not, so the state (stored contract code) is updated as if the operation succeeded even when the sender is unauthenticated.

### Impact Explanation

During the epoch whose protocol version is still 84 (the transition epoch immediately before version 85 activates), any network peer — not just a legitimate chunk producer — can:

1. Craft a `ContractCodeResponse::V1` (Borsh prefix `0x00`) for any in-flight `ChunkProductionKey`.
2. Send it to a chunk validator before the legitimate response arrives.
3. The validator stores the attacker-supplied bytecode in its partial-witness tracker.
4. When the validator executes the state transition using this wrong code, it computes a different state root and fails to endorse the chunk.

If enough validators for a shard are targeted simultaneously, the chunk accumulates insufficient endorsements, causing a missed chunk or chain stall for that shard during the transition epoch.

### Likelihood Explanation

The attack window is one epoch (~12 hours on mainnet). The attacker needs only a standard network peer connection — no validator stake, no privileged role. The `ContractCodeResponse` message is unicast to the requesting validator, so the attacker must race or intercept the legitimate response, but since the validator's `ContractCodeRequest` is sent to a randomly chosen chunk producer and the response travels over the p2p network, a well-connected adversarial peer can win the race. The `processed_contract_code_requests` deduplication is keyed on `(ChunkProductionKey, requester)` and lives on the **request** side, not the response side, so there is no deduplication guard against multiple unsolicited responses.

### Recommendation

Remove the protocol-version gate from `validate_contract_code_response`. The `V1` variant carries no signature, so the only safe policy is to reject it unconditionally once the feature is stabilized, or to require that any accepted response be `V2`. Concretely:

```rust
pub fn validate_contract_code_response(...) -> Result<ChunkRelevance, Error> {
    let key = response.chunk_production_key();
    require_relevant!(validate_chunk_relevant(epoch_manager, key, store)?);
    // Always require a signed response; V1 is rejected by the inner check.
    validate_witness_contract_code_response_signature(epoch_manager, response)?;
    Ok(ChunkRelevance::Relevant)
}
```

`validate_witness_contract_code_response_signature` already returns an error for unsigned (`V1`) responses via the `response.responder()` → `None` branch, so this change is backward-safe for any epoch already running version 85. [7](#0-6) 

### Proof of Concept

1. Connect a peer to a mainnet/testnet validator node during the epoch-transition window (epoch protocol version = 84).
2. Observe a `ContractCodeRequest` being sent by the validator (visible in logs or via network sniffing).
3. Craft a `ContractCodeResponse::V1` with the same `ChunkProductionKey` but with `compressed_contracts` encoding arbitrary (wrong) bytecode.
4. Send it to the validator before the legitimate chunk producer responds.
5. The validator calls `validate_contract_code_response` → feature not enabled → returns `Relevant` → stores the poisoned code.
6. The validator executes the state transition with the wrong code, computes a mismatched state root, and fails to produce a chunk endorsement.
7. Repeat for enough validators on the shard to prevent the chunk from reaching the endorsement threshold.

pub enum ContractCodeResponse {
    V1(ContractCodeResponseV1) = 0,
    V2(ContractCodeResponseV2) = 1,
}

impl ContractCodeResponse {
    pub fn encode(
        next_chunk: ChunkProductionKey,
        contracts: &Vec<CodeBytes>,
        signer: &ValidatorSigner,
        protocol_version: ProtocolVersion,
    ) -> std::io::Result<Self> {
        if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
            ContractCodeResponseV2::encode(next_chunk, contracts, signer).map(Self::V2)
        } else {
            ContractCodeResponseV1::encode(next_chunk, contracts).map(Self::V1)
        }
    }

    pub fn chunk_production_key(&self) -> &ChunkProductionKey {
        match self {
            Self::V1(v1) => &v1.next_chunk,
            Self::V2(v2) => &v2.inner.next_chunk,
        }
    }

    /// Account that produced this response. Available only for signed variants.
    pub fn responder(&self) -> Option<&AccountId> {
        match self {
            Self::V1(_) => None,
            Self::V2(v2) => Some(&v2.inner.responder),
        }
    }

    pub fn decompress_contracts(&self) -> std::io::Result<Vec<CodeBytes>> {
        let compressed_contracts = match self {
            Self::V1(v1) => &v1.compressed_contracts,
            Self::V2(v2) => &v2.inner.compressed_contracts,
        };
        compressed_contracts.decode().map(|(data, _size)| data)
    }

    /// Verifies the signature for signed variants. Returns `false` for
    /// unsigned variants since there is nothing to verify.
    pub fn verify_signature(&self, public_key: &PublicKey) -> bool {
        match self {
            Self::V1(_) => false,
            Self::V2(v2) => v2.verify_signature(public_key),
        }
    }
