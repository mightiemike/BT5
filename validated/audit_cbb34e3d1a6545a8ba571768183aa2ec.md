Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` validates `owner` instead of `sender`, allowing any non-allowlisted address to deposit into a curated pool — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual depositor/payer) and validates only the `owner` argument (the position recipient). Because `MetricOmmPool.addLiquidity` accepts an arbitrary caller-supplied `owner` with no constraint that `owner == msg.sender`, any non-allowlisted address can bypass the deposit gate by supplying an allowlisted address as `owner`. This breaks the core invariant that only allowlisted addresses may deposit into a curated pool.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` then ignores `sender` (first parameter, unnamed) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper), not the recipient: [3](#0-2) 

The bypass is also reachable through `MetricOmmPoolLiquidityAdder.addLiquidityExactShares`, which accepts an explicit `owner` argument and calls `pool.addLiquidity(owner, ...)` on behalf of the caller: [4](#0-3) 

`_validateOwner` only rejects `address(0)`, so any allowlisted address is a valid bypass target: [5](#0-4) 

**Exploit path:**
1. Bob (non-allowlisted) calls `pool.addLiquidity(alice /*allowlisted*/, salt, deltas, callbackData, extensionData)`.
2. Pool calls `extension.beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`.
3. Extension evaluates `allowedDepositor[pool][alice]` → `true` → returns selector, no revert.
4. Bob's tokens are pulled via callback; Alice receives the position shares.
5. Bob has interacted with a curated pool without being allowlisted, altering `curBinIdx`, `curPosInBin`, and `binTotals`.

## Impact Explanation

The deposit allowlist is the pool admin's mechanism to restrict who may interact with a curated pool. Because the check is bound to `owner` rather than `sender`, the restriction is entirely ineffective against the actual payer. A non-allowlisted address can deposit tokens into the pool, moving the bin cursor (`curBinIdx`, `curPosInBin`) and altering liquidity distribution across bins — state changes the admin intended to gate. The non-allowlisted depositor can also grief any allowlisted owner by minting positions into their account without consent. Because `removeLiquidity` enforces `msg.sender == owner`, the attacker cannot reclaim the deposited tokens, but the pool's internal accounting (`binTotals`, per-bin shares) is permanently altered in ways that affect all subsequent swaps and LP redemptions. This constitutes broken core pool functionality causing alteration of pool state that the admin intended to restrict. [6](#0-5) 

## Likelihood Explanation

The bypass requires only that the attacker know one allowlisted address (trivially discoverable on-chain via `AllowedToDepositSet` events or `allowedDepositor` reads) and call `pool.addLiquidity(allowlisted_address, ...)` or the equivalent `LiquidityAdder` path. No special role, flash loan, or multi-step setup is needed. The attack is repeatable by any unprivileged address at any time the pool is not paused. [7](#0-6) 

## Recommendation

Bind the allowlist check to `sender` (the actual depositor), consistent with `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate by position owner (not payer), the contract name, NatDoc, and admin tooling must be updated to reflect that, and a separate sender-level check should be added. [2](#0-1) 

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` as a configured extension.
2. Pool admin calls `extension.setAllowedToDeposit(pool, alice, true)` — Alice is allowlisted; Bob is not.
3. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)` directly.
4. Pool calls `extension.beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`.
5. Extension evaluates `allowedDepositor[pool][alice]` → `true` → returns selector, no revert.
6. Bob's deposit executes: tokens are pulled from Bob, Alice receives the position shares.
7. Bob has interacted with a curated pool without being allowlisted, altering bin state and LP accounting.

The same result is achieved via `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, extensionData)` called by Bob. [4](#0-3)
