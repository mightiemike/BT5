Audit Report

## Title
`SwapAllowlistExtension.beforeSwap()` evaluates router address instead of real swapper, enabling allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap()` passes `msg.sender` (the immediate caller) as `sender` to `_beforeSwap()`. When `MetricOmmSimpleRouter` intermediates a swap, `msg.sender` at the pool is the router contract, not the end user. `SwapAllowlistExtension.beforeSwap()` therefore checks the router's allowlist status, not the real swapper's, rendering per-user allowlists ineffective for all router-mediated swaps.

## Finding Description
`MetricOmmPool.swap()` captures `msg.sender` and forwards it as `sender` to `_beforeSwap()`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle()` stores the real user in transient storage only for the payment callback, and never passes it to `pool.swap()`: [3](#0-2) 

The full call path is:
```
forbiddenUser → MetricOmmSimpleRouter.exactInputSingle()
              → pool.swap(recipient, ...)        // msg.sender = router
              → _beforeSwap(msg.sender=router, ...)
              → SwapAllowlistExtension.beforeSwap(sender=router, ...)
              → checks allowedSwapper[pool][router] → passes (router is allowlisted)
```

For router-mediated swaps to function at all, the pool admin must allowlist the router address. Once the router is allowlisted, every user — including those explicitly excluded — can bypass the per-user gate by calling through the router. The guard at line 37 of `SwapAllowlistExtension.sol` evaluates the wrong address in every router-mediated swap. [4](#0-3) 

## Impact Explanation
**Allowlist bypass (fund-impacting):** A pool admin deploying `SwapAllowlistExtension` to restrict swaps to KYC'd or whitelisted addresses must allowlist the router for normal operation. Once the router is allowlisted, any excluded user can bypass the per-user gate by routing through `MetricOmmSimpleRouter`, allowing unauthorized or toxic flow into pools explicitly designed to exclude it, directly harming LP value. This breaks the core security guarantee of the extension and constitutes a broken core pool functionality / admin-boundary break with direct fund impact.

**DoS (unusable swap flow):** If the admin does not allowlist the router, every individually-allowlisted user who swaps through the router is denied with `NotAllowedToSwap`, making the router path permanently broken for all allowlisted pools.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any pool deploying `SwapAllowlistExtension` and expecting users to interact through the router encounters this mismatch on every router-mediated swap. The behavior is deterministic, requires no special conditions, and is triggered by any unprivileged user calling `exactInputSingle()` or `exactInput()` on an allowlist-gated pool.

## Recommendation
The `sender` identity forwarded through the hook chain must reflect the actual end user, not the immediate caller of `pool.swap()`. Two viable approaches:

1. **Extend the swap interface**: Add an explicit `swapper` parameter to `pool.swap()` that the router populates with `msg.sender` before calling the pool. The pool passes this value (rather than its own `msg.sender`) as `sender` to extension hooks.

2. **Extension-side resolution**: Have `SwapAllowlistExtension.beforeSwap()` accept an `extensionData` payload that the router encodes with the real user address, verified against a trusted forwarder registry or router-signed attestation.

Option 1 is simpler and keeps the trust model intact because the pool still controls what `sender` value reaches the extension.

## Proof of Concept
1. Deploy `SwapAllowlistExtension` and a pool with it configured on `beforeSwap`.
2. Allowlist `allowedUser` and `address(router)` via `setAllowedToSwap`; do NOT allowlist `forbiddenUser`.
3. Call `pool.swap()` directly as `forbiddenUser` → reverts with `NotAllowedToSwap` (guard works correctly for direct calls).
4. Call `router.exactInputSingle()` as `forbiddenUser` → succeeds, swap executes, `forbiddenUser` receives output tokens.

The extension evaluates `allowedSwapper[pool][router]` (true) instead of `allowedSwapper[pool][forbiddenUser]` (false), confirming the bypass. The submitted Foundry test (`SwapAllowlistBypassViaRouterTest`) directly demonstrates both behaviors.
