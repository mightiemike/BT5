### Title
Incorrect Difficulty Validation for Fork Blocks Due to Mainchain-Only Height Lookup — (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`)

---

### Summary

The `get_next_work_required` function in both `dogecoin.rs` and `bitcoin.rs` fetches the first block of the difficulty adjustment interval using `get_header_by_height`, which unconditionally returns the **mainchain** block at that height. When the function is called during fork-block validation, the fork's actual ancestor at that height may be a different block. This desynchronization causes the difficulty calculation to use the wrong timestamp, allowing an unprivileged relayer to submit fork blocks whose PoW satisfies an incorrectly computed (easier) target.

---

### Finding Description

In `get_next_work_required` inside `contract/src/dogecoin.rs`, after computing `height_first`, the code calls:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

The `get_header_by_height` implementation always resolves through `mainchain_height_to_header`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

Fork blocks are stored only in `headers_pool` (via `store_fork_header`), never in `mainchain_height_to_header`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [3](#0-2) 

So when a fork block at height `H` is being validated, `get_header_by_height(H-1)` returns the **mainchain** block at `H-1`, not the fork's actual ancestor at `H-1`. The same pattern exists in `bitcoin.rs`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [4](#0-3) 

The developers themselves flagged this with a `TODO` comment, confirming the known desynchronization.

For Dogecoin after block 145,000, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `height_first = prev_block_header.block_height - 1`. This means **every** fork block submitted after the divergence point triggers the wrong lookup. [5](#0-4) 

The difficulty formula is:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
``` [6](#0-5) 

A larger `first_block_time` → smaller `modulated_timespan` → harder target. A smaller `first_block_time` → larger `modulated_timespan` → easier target (higher `new_target`). By contrast, Zcash's `zcash_get_next_work_required` correctly walks the chain via `get_prev_header` (following `prev_block_hash`), so it is not affected. [7](#0-6) 

---

### Impact Explanation

An attacker who submits a fork diverging at height `D` can craft the fork block at `D` with a timestamp `T_fork > T_main` (where `T_main` is the mainchain block's timestamp at `D`). For the next fork block at `D+1`, the contract computes difficulty using `T_main` (the mainchain value) instead of `T_fork` (the fork's actual ancestor). Because `T_main < T_fork`, the computed timespan is larger, the target is higher (easier), and the attacker's fork block at `D+1` satisfies a weaker PoW requirement than the protocol mandates. This cascades: every subsequent fork block benefits from the same incorrect lookup, making a sustained chain reorg progressively cheaper. A successful reorg corrupts `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` — the canonical state that all SPV proof consumers depend on.

---

### Likelihood Explanation

- The entry point is the public, unprivileged `submit_blocks` call, reachable by any NEAR account.
- For Dogecoin (post-block 145,000), the bug fires for **any** fork that diverges more than one block back — a routine scenario during normal chain operation.
- No leaked keys, privileged roles, or social engineering are required.
- The `TODO` comment in the production code confirms the developers identified but did not resolve the issue.

---

### Recommendation

Replace the `get_header_by_height` call with an ancestor walk using `get_prev_header` (which follows `prev_block_hash` through `headers_pool`), mirroring the correct approach already used in `zcash_get_next_work_required`. This ensures the difficulty calculation always uses the block that is the true ancestor of the block being validated, regardless of whether it is on the mainchain or a fork.

---

### Proof of Concept

1. Mainchain has blocks at heights `…, H-1 (timestamp T_main), H`.
2. Attacker calls `submit_blocks` with a fork block at height `H-1` carrying timestamp `T_fork = T_main + 3600` (1 hour later, within the 2-hour future-time limit). This fork block is stored only in `headers_pool`.
3. Attacker calls `submit_blocks` with a fork block at height `H`. The contract calls `check_pow` → `get_next_work_required` → `get_header_by_height(H-1)`.
4. `get_header_by_height(H-1)` resolves through `mainchain_height_to_header`, returning the mainchain block with `T_main`, not the fork block with `T_fork`.
5. `modulated_timespan` is computed as `T_fork_H - T_main` instead of the correct `T_fork_H - T_fork`. Since `T_main < T_fork`, the contract's timespan is 3600 seconds larger than correct.
6. `calculate_next_work_required` produces a higher (easier) target. The attacker's fork block at `H` satisfies this weaker PoW and is accepted.
7. Repeated across many blocks, the attacker accumulates enough chainwork to trigger `reorg_chain`, replacing the canonical chain with the attacker's fork.
