### Title
Linked Signer Can Permanently Hijack Any Subaccount via `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is processed with `allowLinkedSigner = true`. This means the **current linked signer** of a subaccount can sign a new `LinkSigner` transaction to replace the linked signer with any address — including themselves. Because the linked signer can also sign `WithdrawCollateral` and other fund-moving transactions, this creates a path for a malicious or compromised linked signer to permanently seize control of a victim's subaccount and drain its collateral.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch is: [1](#0-0) 

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes the current linked signer to the verifier as an acceptable signer: [2](#0-1) 

```solidity
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
```

This means the **linked signer** — not just the subaccount owner — can produce a valid signature for a `LinkSigner` transaction. The transaction then unconditionally overwrites `linkedSigners[signedTx.tx.sender]` with the attacker-chosen value.

By contrast, the **slow-mode path** for `LinkSigner` correctly restricts this to the subaccount owner only, using `validateSender` which checks `address(uint160(bytes20(txn.sender))) == msg.sender`: [3](#0-2) 

The fast path has no equivalent restriction.

---

### Impact Explanation

A malicious or compromised linked signer can:

1. Sign a `LinkSigner` transaction setting `signedTx.tx.signer` to themselves (or any attacker-controlled address).
2. This permanently replaces the linked signer entry for the victim's subaccount.
3. The attacker retains the ability to sign `WithdrawCollateral`, `WithdrawCollateralV2` (when `sendTo == address(0)`), `TransferQuote`, `MintNlp`, `BurnNlp`, and `LiquidateSubaccount` transactions — all of which also accept `allowLinkedSigner = true`.
4. If the victim attempts to revoke by submitting their own `LinkSigner` transaction, the attacker can immediately re-link themselves with the next nonce, creating an unresolvable race condition at the sequencer level.

The concrete corrupted state: `linkedSigners[victimSubaccount]` is permanently set to the attacker's address, giving the attacker full signing authority over the subaccount's collateral.

---

### Likelihood Explanation

Linked signers are a standard feature used for trading bots, automated strategies, and third-party integrations. Any user who has ever linked a signer is exposed. The attacker only needs to:

- Be (or compromise) the current linked signer of any subaccount with non-zero collateral.
- Know the current nonce of the subaccount owner's address (publicly queryable via `getNonce`).
- Submit a single signed `LinkSigner` transaction through the sequencer.

No admin access, no governance capture, and no leaked private keys are required beyond the linked signer key itself.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner should be permitted to change the linked signer, consistent with the slow-mode path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // linked signer must NOT be allowed to change the linked signer
);
```

---

### Proof of Concept

1. Alice owns subaccount `alice_subaccount` and has linked `bot_address` as her signer.
2. `bot_address` is compromised by an attacker.
3. Attacker queries `getNonce(alice_address)` → returns `N`.
4. Attacker constructs a `LinkSigner` transaction:
   - `sender = alice_subaccount`
   - `signer = bytes32(uint256(uint160(attacker_address)))`
   - `nonce = N`
5. Attacker signs this transaction with `bot_address`'s private key.
6. Attacker submits the transaction to the sequencer.
7. `processTransactionImpl` accepts the signature (because `allowLinkedSigner = true` and `bot_address` is the current linked signer).
8. `linkedSigners[alice_subaccount]` is now set to `attacker_address`.
9. Attacker signs a `WithdrawCollateral` transaction (also `allowLinkedSigner = true`) to drain Alice's collateral.
10. Alice cannot recover: any revocation transaction she submits can be immediately countered by the attacker re-linking themselves with the next nonce.
