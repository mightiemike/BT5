### Title
Hash Domain Divergence in V3 Transaction Simulation: `ResourceBoundsMapping` → `ValidResourceBounds` Collapses `AllResources` to `L1Gas`, Producing Wrong Transaction Hash - (File: `crates/apollo_rpc/src/v0_8/transaction.rs`)

---

### Summary

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in the RPC layer silently collapses `AllResources{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` to `L1Gas(X)`. Because `get_tip_resource_bounds_hash` produces a structurally different Poseidon hash for `L1Gas` (two resource felts: L1+L2) versus `AllResources` (three resource felts: L1+L2+L1_DATA), the transaction hash computed during `starknet_simulateTransactions` diverges from the hash computed during gateway submission for any V3 transaction with zero L2 gas and zero L1 data gas bounds. The simulation returns an authoritative-looking wrong transaction hash via the `get_execution_info` syscall.

---

### Finding Description

**Gateway submission path** accepts `RpcInvokeTransactionV3` / `RpcDeclareTransactionV3`, which carry `resource_bounds: AllResourceBounds` — always the `ValidResourceBounds::AllResources` variant. [1](#0-0)  The hash is computed via `get_tip_resource_bounds_hash` with `AllResources`, which chains **three** resource felts: `[tip, concat(L1_GAS, amount, price), concat(L2_GAS, 0, 0), concat(L1_DATA_GAS, 0, 0)]`. [2](#0-1) 

**RPC simulation path** accepts `BroadcastedTransaction` with `resource_bounds: ResourceBoundsMapping`. When converted to `ValidResourceBounds`:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)   // ← silently collapses to L1Gas
        } else {
            Self::AllResources(...)
        }
    }
}
``` [3](#0-2) 

A transaction with `l2_gas=0` and `l1_data_gas=0` becomes `L1Gas`. `get_tip_resource_bounds_hash` then chains only **two** resource felts — the `L1_DATA_GAS` felt is absent: [4](#0-3) 

```
L1Gas  → poseidon(tip, concat(L1_GAS,X,P), concat(L2_GAS,0,0))
AllRes → poseidon(tip, concat(L1_GAS,X,P), concat(L2_GAS,0,0), concat(L1_DATA_GAS,0,0))
```

These are **different** Poseidon outputs. The resulting transaction hash diverges from the gateway-computed hash.

The gateway stateless validator accepts V3 transactions with zero L2 gas and zero L1 data gas (only one resource bound needs to be non-zero), so this divergence is reachable for any such transaction. [5](#0-4) 

The simulation path calls `tx.try_into()` on each `BroadcastedTransaction`, which computes the hash from the (now-collapsed) `L1Gas` bounds: [6](#0-5) 

The `InternalRpcInvokeTransactionV3` and `InternalRpcDeclareTransactionV3` internal types always store `resource_bounds: AllResourceBounds` and always return `ValidResourceBounds::AllResources` from their trait implementations, confirming the gateway path never produces `L1Gas` for V3 transactions: [7](#0-6) 

---

### Impact Explanation

`starknet_simulateTransactions` computes and returns a wrong transaction hash. The hash is passed to the contract via the `get_execution_info` syscall. Any contract that uses the transaction hash in its `__execute__` function (e.g., for replay protection, event emission, or cross-contract calls that verify the hash) will observe a different value during simulation than during actual execution. This causes simulation to return authoritative-looking wrong execution results, wrong traces, and potentially wrong fee estimates — matching the **High** impact: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

Any V3 transaction with `l2_gas=0` and `l1_data_gas=0` triggers this divergence. This is a common pattern for transactions that only consume L1 gas. Any unprivileged user can trigger this by calling `starknet_simulateTransactions` with such a transaction. No special privileges or malicious peer behavior is required.

---

### Recommendation

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion must not collapse `AllResources` with zero l2_gas and l1_data_gas to `L1Gas` in the simulation path. Options:

1. **Preferred**: Change `BroadcastedInvokeV3Transaction` and `BroadcastedDeclareV3Transaction` to carry `resource_bounds: AllResourceBounds` directly (matching `RpcInvokeTransactionV3` / `RpcDeclareTransactionV3`), eliminating the lossy `ResourceBoundsMapping` conversion entirely.
2. **Alternative**: In the simulation conversion path, always produce `AllResources` for V3 transactions regardless of whether l2_gas and l1_data_gas are zero, preserving the hash domain used at submission time.
