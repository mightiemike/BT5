### Title
Double Reward Claim via Duplicate Position IDs in `ClaimTierRewards` — (File: `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`ClaimTierRewards` accepts a list of position IDs in a single message, validates ownership for each in a loop **without updating state between iterations**, then processes all collected positions in a single batch call. If the same position ID appears more than once in `msg.PositionIds`, the position is collected twice with identical pre-claim state and rewards are disbursed twice.

---

### Finding Description

In `ClaimTierRewards` (`x/tieredrewards/keeper/msg_server.go`), the handler first iterates over all submitted position IDs, reads each position from the KV store, validates ownership, and appends to a local `positions` slice. Only after the loop completes does it call `claimRewardsAndUpdatesPositions` to actually update state and disburse rewards.

```go
// x/tieredrewards/keeper/msg_server.go lines 434–448
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    ...
    if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
        return nil, err
    }
    positions = append(positions, pos)
}

totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
```

Because `getPositionState` reads from the KV store and no write-back occurs inside the loop, submitting `[posId, posId]` causes the same position to be appended twice with identical pre-claim state. `claimRewardsAndUpdatesPositions` then processes both entries, disbursing rewards for the same position twice.

This is structurally identical to M-05: the "current state" flag (here: the position's reward checkpoint) is not invalidated within the loop, so the same authority (position ownership) can be exercised multiple times in one transaction.

---

### Impact Explanation

A position owner can claim tier rewards multiple times per transaction by repeating the same position ID `N` times in `msg.PositionIds`. Each repetition causes an additional reward disbursement from the module's reward pool, draining it faster than intended and giving the attacker `N×` the legitimate reward amount. The corrupted value is the module reward pool balance and the position's bonus checkpoint state.

---

### Likelihood Explanation

Any position owner can construct and broadcast a `MsgClaimTierRewards` with duplicate entries in `PositionIds`. No privileged role, leaked key, or social engineering is required. The message is a standard Cosmos SDK transaction reachable from the CLI or any client. The only mitigation would be deduplication in `msg.Validate()`, which is not visible in the collected code.

---

### Recommendation

Deduplicate `msg.PositionIds` in `MsgClaimTierRewards.Validate()`, or update each position's state (checkpoint) inside the collection loop before appending it to the `positions` slice, so that a second read of the same ID within the same transaction reflects the already-claimed state and fails validation.

---

### Proof of Concept

1. Owner holds position with ID `42`.
2. Owner broadcasts `MsgClaimTierRewards{Owner: owner, PositionIds: [42, 42, 42]}`.
3. The loop at lines 435–446 reads position `42` from the KV store three times (same pre-claim state each time) and appends three identical `PositionState` structs to `positions`.
4. `claimRewardsAndUpdatesPositions` processes all three entries, disbursing rewards three times.
5. Owner receives `3×` the legitimate reward amount in a single transaction. [1](#0-0) [2](#0-1)
