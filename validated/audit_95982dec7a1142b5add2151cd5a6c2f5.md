### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc()` Enables Draining usdcE Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` is an unguarded external function that performs a 1:1 swap of usdcE held in a `DirectDepositV1` contract for USDC provided by the caller. The USDC inbound leg uses a raw `transferFrom` call whose return value is never checked, while the usdcE outbound leg uses the safe wrapper. If the USDC token returns `false` on failure instead of reverting, the function silently skips the USDC pull and still delivers usdcE to the caller, draining the `DirectDepositV1` contract.

---

### Finding Description

`ContractOwner` imports and activates `ERC20Helper` (`using ERC20Helper for IERC20Base`), which provides `safeTransfer` and `safeTransferFrom` wrappers that revert on a `false` return value. Every other token transfer in the contract uses these safe wrappers. The single exception is the USDC inbound transfer inside `replaceUsdcEWithUsdc`:

```solidity
// core/contracts/ContractOwner.sol  lines 608-620
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    address payable directDepositV1 = directDepositV1Address[subaccount];
    require(directDepositV1 != address(0), "no dda");
    address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
    address usdc  = 0x2D270e6886d130D724215A266106e6832161EAEd;
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← raw call, return ignored
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));       // usdcE → ContractOwner
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);                 // usdcE → caller
    }
}
```

The execution sequence is:
1. Pull `balance` USDC from `msg.sender` into `directDepositV1` — **return value silently ignored**.
2. Withdraw all usdcE from `directDepositV1` into `ContractOwner`.
3. Forward that usdcE to `msg.sender`.

Steps 2 and 3 execute unconditionally regardless of whether step 1 succeeded. There is no access control beyond the chain-ID guard; any address on chain 57073 can call this function.

The inconsistency is structural: `safeTransfer` is used for the usdcE outbound leg (line 618) but the raw interface method is used for the USDC inbound leg (line 616), bypassing the return-value check that `ERC20Helper.safeTransferFrom` would enforce. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on chain 57073 (Ink) returns `false` on an insufficient-balance or insufficient-allowance condition rather than reverting, an attacker with zero USDC (or zero allowance) can:

- Call `replaceUsdcEWithUsdc` for any subaccount whose `directDepositV1` holds usdcE.
- Receive the full usdcE balance of that `DirectDepositV1` contract at zero cost.
- Repeat for every registered subaccount.

The corrupted asset delta is the complete usdcE balance of every affected `DirectDepositV1` contract. These balances represent user collateral awaiting deposit into the protocol; their loss is permanent and unrecoverable. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

The function is callable by any unprivileged address on chain 57073 with no further preconditions. The exploitability is gated solely on whether the specific USDC deployment on Ink returns `false` on failure rather than reverting. Circle's canonical USDC implementation reverts, which would prevent exploitation in the current deployment. However:

- The hardcoded token address is chain-specific and could be a bridged or wrapped variant with different revert semantics.
- The code defect is present regardless of current token behavior and would become immediately exploitable if the token is upgraded or replaced.
- The function carries no access control, so no privilege escalation is required. [5](#0-4) 

---

### Recommendation

Replace the raw `transferFrom` call with the `safeTransferFrom` wrapper already available via `ERC20Helper`, consistent with every other transfer in the contract:

```diff
- IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
+ IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [2](#0-1)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
