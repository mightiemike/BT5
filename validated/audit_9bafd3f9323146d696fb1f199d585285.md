### Title
Missing Parent Health Check After Margin Transfer in `createIsolatedSubaccount` — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.createIsolatedSubaccount` deducts an attacker-controlled margin amount from the parent subaccount's `QUOTE_PRODUCT_ID` balance but never asserts that the parent remains above initial health afterward. The analogous collateral-reduction path (`Clearinghouse.transferQuote`) enforces this invariant explicitly. The omission allows a parent subaccount with open positions to be driven below initial health in a single signed transaction, with no revert.

---

### Finding Description

In `OffchainExchange.createIsolatedSubaccount`, after the margin is extracted from the order appendix bits `[127:64]` via `_isolatedMargin`, the function unconditionally deducts it from the parent:

```solidity
// OffchainExchange.sol:1074-1087
int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
}
return newIsolatedSubaccount;
``` [1](#0-0) 

No health check follows. Compare with `Clearinghouse.transferQuote`, which performs the identical balance mutation and then enforces:

```solidity
// Clearinghouse.sol:247-249
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -toTransfer);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.recipient, toTransfer);
require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH);
``` [2](#0-1) 

`_isAboveInitial` is defined as:

```solidity
// Clearinghouse.sol:639-642
function _isAboveInitial(bytes32 subaccount) internal returns (bool) {
    return getHealth(subaccount, IProductEngine.HealthType.INITIAL) >= 0;
}
``` [3](#0-2) 

The `EndpointTx.sol` dispatch path for `CreateIsolatedSubaccount` also adds no compensating check after the call returns:

```solidity
// EndpointTx.sol:619-631
} else if (txType == IEndpoint.TransactionType.CreateIsolatedSubaccount) {
    ...
    bytes32 newIsolatedSubaccount = IOffchainExchange(offchainExchange)
        .createIsolatedSubaccount(txn, getLinkedSigner(txn.order.sender));
    _recordSubaccount(newIsolatedSubaccount);
}
``` [4](#0-3) 

The margin value is fully attacker-controlled: `_isolatedMargin` reads bits `[127:64]` of the appendix and scales by `10**12`, with no upper bound enforced anywhere in the function. [5](#0-4) 

The signature check only verifies that the signer is the subaccount owner or a linked signer — the attacker signs their own order, so this passes trivially. [6](#0-5) 

---

### Impact Explanation

A parent subaccount that holds open perp positions (requiring initial margin) can have its entire quote balance drained into an isolated subaccount in one transaction. After the transfer the parent's initial health is negative, yet no revert occurs. The parent is now immediately eligible for liquidation at a discount. If the parent's positions cannot be fully covered, the shortfall is absorbed by the insurance fund, creating bad debt. The isolated subaccount retains the transferred quote and is a separate liquidation entity, so the attacker's funds are not at risk from the parent's liquidation.

---

### Likelihood Explanation

The path is reachable by any user who owns a parent subaccount with open positions. It requires only a validly signed `CreateIsolatedSubaccount` transaction with a large margin value in the appendix — no privileged access, no oracle manipulation, no governance capture. The sequencer processes the transaction normally.

---

### Recommendation

Add a parent health assertion immediately after the margin transfer in `createIsolatedSubaccount`, mirroring the pattern in `transferQuote`:

```solidity
// After the spotEngine.updateBalance calls at line 1087
if (margin > 0) {
    ...
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
    require(
        IClearinghouse(clearinghouse).getHealth(
            txn.order.sender,
            IProductEngine.HealthType.INITIAL
        ) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
``` [7](#0-6) 

---

### Proof of Concept

```
1. Deploy protocol with SpotEngine, PerpEngine, Clearinghouse, OffchainExchange.
2. Deposit 1000 USDC quote to parent subaccount P.
3. Open a perp position on P requiring 900 USDC initial margin.
   → P.health(INITIAL) = 100 (above zero, valid).
4. Craft a CreateIsolatedSubaccount order for P:
   - appendix bits [127:64] = 1000  (encodes margin = 1000 * 10^12 = 1000 USDC)
   - isolated bit set, valid expiry
   - signed by P's private key.
5. Submit via processTransactionImpl → CreateIsolatedSubaccount dispatch.
6. createIsolatedSubaccount executes:
   - spotEngine.updateBalance(QUOTE_PRODUCT_ID, P, -1000 USDC)  ← P quote = 0
   - spotEngine.updateBalance(QUOTE_PRODUCT_ID, isolatedSub, +1000 USDC)
   - no health check → returns without revert.
7. Assert: clearinghouse.getHealth(P, INITIAL) < 0  ← parent is undercollateralized.
8. Liquidator can now liquidate P at a discount; isolated subaccount retains 1000 USDC.
```
