### Title
Unvalidated `tx_id` Enables Merkle Internal-Node Forgery in `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` accepts a caller-supplied `tx_id` without validating that it is a leaf node (a real transaction hash) rather than an internal Merkle tree node. Any unprivileged NEAR caller can supply an internal-node hash as `tx_id` with a valid sibling proof, causing the function to return `true` for a transaction that was never included in the block.

---

### Finding Description

The analog vulnerability class is **incomplete field validation**: just as the Uniswap wallet accepts a `value`, `nonce`, and `deadline` without displaying them for user verification, this contract accepts `tx_id` without validating that it corresponds to a real leaf-level transaction.

`verify_transaction_inclusion` accepts five caller-controlled fields in `ProofArgs`: `tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, and `confirmations`. [1](#0-0) 

The function validates:
- `tx_block_blockhash` — must be in the mainchain [2](#0-1) 
- `confirmations` — must not exceed `gc_threshold` [3](#0-2) 
- `merkle_proof` — must not be empty [4](#0-3) 

It does **not** validate that `tx_id` is a leaf node. The Merkle root computation is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [5](#0-4) 

`compute_root_from_merkle_proof` is a pure hash-chain computation that is equally valid whether `transaction_hash` is a leaf or an internal node. [6](#0-5) 

The function's own documentation acknowledges the broken invariant:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [7](#0-6) 

The `#[deprecated]` attribute is a Rust compiler warning only — it does not prevent the function from being called at runtime. The function remains public and `#[pause]`-gated (not access-controlled), so any NEAR account can invoke it. [8](#0-7) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate fund releases or authorization decisions can be fooled into accepting a fabricated transaction. The attacker supplies an internal Merkle node hash as `tx_id` with a valid sibling proof for that node. The function returns `true`, and the downstream contract acts on a transaction that was never included in the block. The corrupted value is the **proof result** — a `true` return for a non-existent transaction — which is the exact state delta that downstream consumers trust.

---

### Likelihood Explanation

The attack requires no privileged role, no private key, and no social engineering. The attacker only needs:
1. Any real mainchain block (public information).
2. The block's transaction list (public, derivable from the Bitcoin network).
3. The ability to call a public NEAR contract method.

Any downstream contract that has not migrated from `verify_transaction_inclusion` to `verify_transaction_inclusion_v2` is directly exploitable. The deprecated function is still live on-chain.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or add a runtime `env::panic_str` guard so it cannot be called. Downstream contracts must migrate to `verify_transaction_inclusion_v2`, which adds the coinbase proof check that prevents internal-node forgery by enforcing equal proof depths. [9](#0-8) 

---

### Proof of Concept

1. Attacker identifies a real mainchain block `B` with at least two transactions (`T0` = coinbase, `T1`).
2. Attacker computes the internal Merkle node `N = H(T0 ‖ T1)` — the parent of the first two leaves.
3. Attacker constructs a valid Merkle sibling proof for `N` at position `0` in the next tree level (proof length = `tree_depth − 1`).
4. Attacker calls `verify_transaction_inclusion` with:
   - `tx_id = N` (internal node hash, not a real transaction)
   - `tx_block_blockhash = B`
   - `tx_index = 0`
   - `merkle_proof = [sibling hashes for N upward to root]`
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, 0, proof)` equals `B`'s stored `merkle_root`. [6](#0-5) 
6. The function returns `true`.
7. A downstream contract that trusted this result releases funds or grants authorization for a transaction that was never broadcast or mined.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
