### Title
Fee-on-Transfer Token Inflates Subaccount Collateral Balance Beyond Actual Received Tokens — (`File: core/contracts/EndpointStorage.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`handleDepositTransfer` in `EndpointStorage.sol` performs two sequential ERC20 transfers using the nominal caller-supplied `amount` without measuring actual received balances. The same nominal `amount` is then encoded into the slow-mode queue and used by `Clearinghouse.depositCollateral` to credit the subaccount. For any fee-on-transfer token, the Clearinghouse credits more than it actually holds, creating an unbacked balance that can be used for trading or withdrawal.

---

### Finding Description

The deposit flow is:

**Step 1 — `Endpoint.depositCollateralWithReferral`** enqueues a slow-mode transaction carrying the raw caller-supplied `amount`:

```
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)          // nominal, not measured
);
// ...
DepositCollateral({ sender: subaccount, productId: productId, amount: amount })
``` [1](#0-0) 

**Step 2 — `handleDepositTransfer`** in `EndpointStorage` executes two transfers with the same nominal value, with no before/after balance check:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);          // user → Endpoint
    safeTransferTo(token, address(clearinghouse), amount); // Endpoint → Clearinghouse
}
``` [2](#0-1) 

For a token with a 10% transfer fee, the first hop delivers 90 tokens to `Endpoint`, and the second hop delivers 81 tokens to `Clearinghouse`. Yet `amount = 100` is what gets stored in the slow-mode queue.

**Step 3 — `Clearinghouse.depositCollateral`** credits the subaccount using the queued nominal `txn.amount`, not the actual on-chain balance delta:

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [3](#0-2) 

The same pattern applies to `depositInsurance` via `submitSlowModeTransactionImpl`:

```solidity
handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
``` [4](#0-3) 

followed by `clearinghouse.depositInsurance(transaction)` which credits `insurance` using `txn.amount`:

```solidity
int128 amount = int128(txn.amount) * int128(multiplier);
insurance += amount;
``` [5](#0-4) 

---

### Impact Explanation

The Clearinghouse's internal ledger (`spotEngine` balance for the depositor's subaccount) is credited with the nominal `amount` while the actual token balance held is strictly less. The gap is permanently unbacked. The inflated balance:

- Passes health checks (`getHealth`) because health is computed from ledger balances, not on-chain token balances.
- Can be used to open leveraged positions or withdraw collateral via `withdrawCollateral`, draining tokens that belong to other depositors.
- Accumulates across multiple deposits, compounding the insolvency.

The `manualAssert` invariant check compares `insurance` against the stored value but does not compare the Clearinghouse's actual ERC20 token balance against the sum of all subaccount ledger balances, so the discrepancy is not caught on-chain. [6](#0-5) 

---

### Likelihood Explanation

Any unprivileged user can trigger this by calling `Endpoint.depositCollateral` or `Endpoint.depositCollateralWithReferral` with a fee-on-transfer token that has been listed as a supported spot product. The protocol does not whitelist or validate token behavior at deposit time. The attacker needs no special role, no governance access, and no leaked keys — only a wallet and a supported fee-on-transfer token. [7](#0-6) 

---

### Recommendation

Replace the fixed-amount double-transfer in `handleDepositTransfer` with a balance-delta measurement:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), token.balanceOf(address(this)));
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    require(actualReceived == amount, ERR_FOT_TOKEN); // or use actualReceived downstream
}
```

The `actualReceived` value must be propagated into the slow-mode `DepositCollateral` struct so that `Clearinghouse.depositCollateral` credits only what was truly received. Alternatively, restrict supported collateral tokens to a whitelist that excludes fee-on-transfer and rebasing tokens.

---

### Proof of Concept

1. A fee-on-transfer token `FOT` (10% fee per transfer) is listed as a supported spot product.
2. Attacker calls `Endpoint.depositCollateral(subaccountName, productId, 100e18)`.
3. `handleDepositTransfer` runs:
   - `safeTransferFrom(FOT, attacker, Endpoint, 100e18)` → Endpoint receives **90e18**.
   - `safeTransferTo(FOT, Endpoint, Clearinghouse, 100e18)` → Clearinghouse receives **81e18** (fee taken again); if Endpoint only holds 90e18, the call attempts to send 100e18 and reverts — or if the token allows it, sends 90e18 and Clearinghouse receives 81e18.
4. Slow-mode queue records `DepositCollateral{amount: 100e18}`.
5. After the slow-mode delay, `Clearinghouse.depositCollateral` credits the subaccount with `100e18 * multiplier`.
6. Attacker's subaccount now shows **100e18** in the ledger while Clearinghouse holds only **81e18** of `FOT`.
7. Attacker calls `withdrawCollateral` for 100e18, draining tokens that belong to other users. [8](#0-7) [9](#0-8)
