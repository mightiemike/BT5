### Title
`ResourceBoundsMapping → ValidResourceBounds` Silently Downgrades V3 Transactions to `L1Gas` Mode in RPC Simulation and Fee Estimation — (File: `crates/apollo_rpc/src/v0_8/transaction.rs`)

---

### Summary

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in the RPC layer silently produces `ValidResourceBounds::L1Gas` for any V3 transaction whose `l1_data_gas` and `l2_gas` fields are both zero. The gateway path, by contrast, always binds V3 transactions to `ValidResourceBounds::AllResources`. The two paths therefore execute the same user-supplied transaction under different gas-accounting modes, causing `simulateTransactions` and `estimateFee` to return authoritative-looking but wrong values.

---

### Finding Description

**Conversion that introduces the split**

`crates/apollo_rpc/src/v0_8/transaction.rs` lines 188–199 implement:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← legacy mode
        } else {
            Self::AllResources(AllResourceBounds {
                l1_gas: value.l1_gas,
                l1_data_gas: value.l1_data_gas,
                l2_gas: value.l2_gas,
            })
        }
    }
}
``` [1](#0-0) 

`BroadcastedTransaction` (the type accepted by `estimateFee` / `simulateTransactions`) carries `resource_bounds: ResourceBoundsMapping`. When the RPC handler converts it to an `ExecutableTransactionInput` for blockifier execution, this `From` impl is invoked. Any V3 invoke, declare, or deploy-account transaction whose user sets `l2_gas = 0` and `l1_data_gas = 0` is therefore silently reclassified as a pre-0.13.3 `L1Gas` transaction.

**Gateway path never does this**

The gateway receives `RpcTransaction`, whose V3 variants carry `resource_bounds: AllResourceBounds` directly. The converter always wraps them as `ValidResourceBounds::AllResources`:

```rust
// InternalRpcInvokeTransactionV3 / InternalRpcDeclareTransactionV3
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
``` [2](#0-1) [3](#0-2) 

**Downstream divergence in gas accounting**

`ValidResourceBounds` controls the `GasVectorComputationMode`:

```rust
pub fn get_gas_vector_computation_mode(&self) -> GasVectorComputationMode {
    match self {
        Self::AllResources(_) => GasVectorComputationMode::All,
        Self::L1Gas(_)        => GasVectorComputationMode::NoL2Gas,
    }
}
``` [4](#0-3) 

`NoL2Gas` means L2 gas is neither tracked nor charged during execution. The fee-check branch for `L1Gas` only validates L1 gas consumption:

```rust
ValidResourceBounds::L1Gas(l1_bounds) => {
    FeeCheckReport::check_l1_gas_amount_within_bounds(l1_bounds, gas_vector, tx_context)
}
``` [5](#0-4) 

whereas `AllResources` checks all three resources including L2 gas:

```rust
ValidResourceBounds::AllResources(all_resource_bounds) => {
    FeeCheckReport::check_all_gas_amounts_within_bounds(
        &all_resource_bounds.to_max_amounts(),
        gas_vector,
    )
}
``` [6](#0-5) 

**Transaction-hash divergence**

`get_tip_resource_bounds_hash` includes `l1_data_gas` in the Poseidon preimage only for `AllResources`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [7](#0-6) 

So the hash computed inside the simulation for the same V3 transaction differs from the hash the gateway will assign when the transaction is actually submitted.

The same incorrect downgrade is present in the protobuf converter used for P2P propagation:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else { ... })
``` [8](#0-7) 

---

### Impact Explanation

A caller invoking `starknet_simulateTransactions` or `starknet_estimateFee` with a V3 transaction that has `l2_gas = 0` and `l1_data_gas = 0` receives results computed under `L1Gas / NoL2Gas` mode:

1. **Wrong success/failure prediction.** The simulation succeeds even when the contract consumes L2 gas, because L2 gas is not tracked. The identical transaction submitted to the gateway executes under `AllResources / All` mode and immediately fails with `MaxGasAmountExceeded` for L2 gas (bound is 0, actual > 0).

2. **Wrong fee estimate.** L2 gas costs are omitted from the `overall_fee` returned by `estimateFee`, so the estimate is systematically too low.

3. **Wrong transaction hash.** The hash embedded in simulation traces omits the `l1_data_gas` field from the Poseidon preimage, producing a hash that will never match the on-chain transaction hash.

All three are "authoritative-looking wrong values" returned by the RPC.

---

### Likelihood Explanation

The trigger is unprivileged and trivially reachable: any caller can submit a V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` to the public RPC endpoints. In Starknet ≥ 0.13.3, Sierra contracts consume L2 gas by default, so the discrepancy is encountered by ordinary contract interactions whenever a user or wallet sets only an L1 gas bound.

---

### Recommendation

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion should not infer the variant from whether the bounds happen to be zero. For V3 transactions, `AllResources` must always be produced regardless of the bound values:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        Self::AllResources(AllResourceBounds {
            l1_gas: value.l1_gas,
            l1_data_gas: value.l1_data_gas,
            l2_gas: value.l2_gas,
        })
    }
}
```

The same fix must be applied to `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs`. The `L1Gas` variant should only be constructed when the transaction version is explicitly pre-0.13.3 (V1/V0), not inferred from zero-valued fields.

### Proof of Concept

```
1. Craft a V3 InvokeTransaction with:
     resource_bounds.l1_gas   = { max_amount: 1_000_000, max_price_per_unit: 1 }
     resource_bounds.l2_gas   = { max_amount: 0, max_price_per_unit: 0 }
     resource_bounds.l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }
   targeting any Sierra contract entry-point that executes at least one step
   (any real contract will consume L2 gas under Starknet ≥ 0.13.3).

2. Call starknet_simulateTransactions with this transaction.
   → ResourceBoundsMapping::from() fires the zero-check branch,
     produces ValidResourceBounds::L1Gas.
   → GasVectorComputationMode::NoL2Gas: L2 gas is not tracked.
   → Fee check: only L1 gas is validated.
   → Simulation returns SUCCEEDED with a fee estimate that omits L2 gas cost.
   → The transaction hash in the trace is computed without l1_data_gas in the
     Poseidon preimage.

3. Submit the identical transaction to starknet_addInvokeTransaction (gateway).
   → RpcInvokeTransactionV3 carries AllResourceBounds directly.
   → ValidResourceBounds::AllResources is used throughout.
   → GasVectorComputationMode::All: L2 gas is tracked.
   → Fee check: l2_gas.max_amount = 0 < actual_l2_gas → REVERTED /
     MaxGasAmountExceeded.
   → On-chain transaction hash includes l1_data_gas in the preimage → differs
     from the hash returned in step 2.
```
