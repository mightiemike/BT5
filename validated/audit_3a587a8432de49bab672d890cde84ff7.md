Audit Report

## Title
Router-as-sender bypasses per-user swap allowlist in `SwapAllowlistExtension` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool always sets to its own `msg.sender` — the router contract, not the end user. If the pool admin allowlists the router (required for any user to swap via the router), every EOA that calls through the router passes the check regardless of whether their own address is individually allowlisted, completely defeating the per-user access control.

## Finding Description

The call chain is fully confirmed in production code:

1. `MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` directly — making the pool see `msg.sender = router`. [1](#0-0) 

2. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, forwarding the router address as `sender`. [2](#0-1) 

3. `ExtensionCalling._beforeSwap` encodes and forwards that `sender` value to the extension unchanged. [3](#0-2) 

4. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][router]`. If the router is allowlisted, the check passes for every caller of the router. [4](#0-3) 

The existing unit tests for `SwapAllowlistExtension` call `extension.beforeSwap(swapper, ...)` with `vm.prank(address(pool))` directly, bypassing the router layer entirely, so they do not catch this bypass. [5](#0-4) 

## Impact Explanation

A pool admin deploys a curated pool with `SwapAllowlistExtension` intending only KYC'd or individually approved EOAs to swap. The admin allowlists the router (the only way to enable router-based swaps). Any non-allowlisted EOA can call `router.exactInputSingle(...)` and the extension sees `sender = router` (allowlisted), permitting the swap. The per-user access control is entirely non-functional when a router is used. This constitutes broken core pool functionality — the swap curation invariant the extension is designed to enforce is violated — with direct, repeatable fund impact on the pool's LPs.

## Likelihood Explanation

Allowlisting the router is the expected operational pattern; without it, no user can swap via the router at all. The bypass requires only a standard `exactInputSingle` call available to any EOA. No privileged access, no malicious setup, no non-standard tokens, and no special conditions are required. Any attacker who knows the pool uses `SwapAllowlistExtension` can exploit this immediately.

## Recommendation

The extension must verify the originating user, not the intermediary. The most robust fix is to have the router encode `msg.sender` into `extensionData` and have the extension decode and verify it against the allowlist. Alternatively, if the semantic goal is "only allowlisted addresses receive output," check the `recipient` argument (the second parameter of `beforeSwap`) instead of `sender`. A third option is to document and enforce that `SwapAllowlistExtension` is incompatible with router intermediaries and require direct pool calls only (e.g., by checking `sender == tx.origin`, with known limitations). [6](#0-5) 

## Proof of Concept

```solidity
// Foundry integration test
function test_routerBypassesSwapAllowlist() public {
    // Pool admin allowlists the router (required for router-based swaps)
    vm.prank(admin);
    extension.setAllowedToSwap(address(pool), address(router), true);

    // Attacker EOA is NOT individually allowlisted
    assertFalse(extension.isAllowedToSwap(address(pool), attacker));

    // Attacker calls through the router
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        recipient: attacker,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    }));

    // Swap succeeds: pool sees sender=router (allowlisted), revert never fires
    assertGt(token1.balanceOf(attacker), 0);
}
```

The pool's `_beforeSwap` receives `sender = address(router)`. Since `allowedSwapper[pool][router] == true`, the `NotAllowedToSwap` revert is never triggered despite the attacker's EOA not being allowlisted. [2](#0-1) [4](#0-3)
