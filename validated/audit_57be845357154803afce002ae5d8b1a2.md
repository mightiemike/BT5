### Title
Caller-Controlled `confirmations` Parameter Bypasses Confirmation-Depth Security Requirement — (`contract/src/lib.rs`, `btc-types/src/contract_args.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations` field that is not cryptographically bound to the block header or Merkle proof. The contract enforces only an upper bound (`confirmations <= gc_threshold`) but no lower bound. Any unprivileged NEAR caller can set `confirmations = 0`, making the confirmation-depth check trivially pass for any block, including the chain tip itself.

---

### Finding Description

`ProofArgs.confirmations` and `ProofArgsV2.confirmations` are free u64 fields supplied entirely by the caller at call time. [1](#0-0) 

Inside `verify_transaction_inclusion`, the contract checks:

```
(heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
    >= args.confirmations
``` [2](#0-1) 

The only other guard is an upper-bound check:

```
args.confirmations <= self.gc_threshold
``` [3](#0-2) 

There is no minimum enforced. With `confirmations = 0`, the expression `depth + 1 >= 0` is always `true` for a `u64`, so the check is unconditionally satisfied regardless of how recently the block was mined. The same path is reachable through `verify_transaction_inclusion_v2`, which delegates to `verify_transaction_inclusion` after converting `ProofArgsV2` into `ProofArgs`: [4](#0-3) 

The `confirmations` value is never committed to in the Merkle proof, the block header, or any on-chain signature. It is a pure caller assertion with no cryptographic binding.

---

### Impact Explanation

Any dApp or bridge contract that calls `verify_transaction_inclusion{_v2}` and relies on the caller to supply a meaningful `confirmations` value receives a `true` result for a transaction that may be in the most recent block or even subject to an active reorg. This breaks the SPV security model: the confirmation depth is the only mechanism preventing acceptance of transactions that will be reversed by a chain reorganization. With `confirmations = 0`, an attacker can present a transaction in a freshly mined (or even the tip) block and receive a `true` verification result, enabling double-spend attacks against any consuming contract that gates asset release on this call.

---

### Likelihood Explanation

The entry path requires no privileges. Any NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` directly with `confirmations = 0`. The attack is deterministic and requires no brute force, timing, or key material. Any consuming contract that does not independently enforce a minimum confirmation count is immediately exploitable.

---

### Recommendation

Enforce a protocol-level minimum confirmation count inside `verify_transaction_inclusion`. Either:

1. Require `args.confirmations >= MIN_CONFIRMATIONS` (e.g., 6 for Bitcoin mainnet) before proceeding, or
2. Remove `confirmations` from the caller-supplied arguments entirely and hard-code or store it as a contract-level configuration parameter that only a privileged role can update.

This mirrors the fix applied in the referenced report: the unconstrained parameter must either be removed from caller control or bounded on both sides by the contract itself.

---

### Proof of Concept

1. A transaction `tx_id` is included in the current chain-tip block (`tx_block_blockhash = mainchain_tip_blockhash`).
2. Attacker calls:
   ```
   verify_transaction_inclusion({
     tx_id: <valid tx hash>,
     tx_block_blockhash: <tip block hash>,
     tx_index: <correct index>,
     merkle_proof: <valid proof>,
     confirmations: 0          // ← attacker-controlled, no lower bound
   })
   ```
3. The upper-bound check passes: `0 <= gc_threshold`. [3](#0-2) 
4. The depth check evaluates `(tip_height - tip_height) + 1 >= 0` → `1 >= 0` → `true`. [2](#0-1) 
5. The Merkle proof is valid, so the function returns `true`.
6. The consuming contract releases assets for a transaction with zero confirmation depth, which may be reversed by a reorg.
