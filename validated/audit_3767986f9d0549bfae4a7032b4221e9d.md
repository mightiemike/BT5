### Title
Unconstrained `confirmations` Parameter Allows Zero-Confirmation Transaction Verification - (File: `contract/src/lib.rs`)

### Summary
The `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` functions accept a caller-supplied `confirmations` value with no minimum bound enforced. Any unprivileged NEAR caller can pass `confirmations = 0`, causing the depth check to trivially pass for any mainchain transaction, defeating the finality guarantee the parameter is meant to enforce.

### Finding Description
In `contract/src/lib.rs`, the confirmation depth check is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

The left-hand side is a `u64` expression that evaluates to at least `1` for any block in the mainchain (since `target_block_height <= heaviest_block_header.block_height`). When `args.confirmations = 0`, the comparison becomes `(some u64 >= 1) >= 0`, which is always `true` in Rust's unsigned arithmetic. The contract enforces only an **upper** bound:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

There is no corresponding **lower** bound. The `confirmations` field is part of `ProofArgs` / `ProofArgsV2` in `btc-types/src/contract_args.rs`, which is fully caller-controlled with no validation at the type level. `verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` via `args.into()`, so it inherits the same flaw. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
A consuming contract that calls `verify_transaction_inclusion` to gate fund releases or cross-chain actions receives `true` for any transaction included in any mainchain block when `confirmations = 0` is passed. This means a Bitcoin transaction that appears in the chain tip — with zero depth, fully reversible — is treated as finalized. A double-spend attack becomes feasible: submit a Bitcoin transaction, immediately prove inclusion with `confirmations = 0`, trigger the consuming contract's action, then reorganize or replace the Bitcoin transaction before it achieves real finality. [4](#0-3) 

### Likelihood Explanation
Medium. The `confirmations` parameter is fully caller-controlled. Any consuming contract that either (a) exposes the `confirmations` value to its own callers, (b) defaults it to `0`, or (c) fails to enforce a minimum before calling the light client is directly exploitable by an unprivileged NEAR account. No privileged role, key material, or social engineering is required — only a valid Merkle proof for a mainchain block.

### Recommendation
Enforce a minimum of `1` confirmation at the entry point of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "Confirmations must be at least 1");
```

This mirrors the fix recommended for the StEthPriceFeed analog: add an explicit lower bound to prevent the parameter from being set to a value that trivially satisfies the security check. [5](#0-4) 

### Proof of Concept
1. Deploy the BTC light client (Bitcoin feature, mainnet, `skip_pow_verification = false`).
2. Submit a valid block header at height `H` containing transaction `tx_id`.
3. Call `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = hash of block `H`
   - `tx_id` = the transaction hash
   - `merkle_proof` = valid Merkle path
   - `confirmations = 0`
4. The upper-bound check passes (`0 <= gc_threshold`). The depth check evaluates `(H - H) + 1 = 1 >= 0` → `true`. The Merkle proof check passes. The function returns `true`.
5. The transaction has zero confirmations and is fully reversible on the Bitcoin network, yet the light client certifies it as verified. [6](#0-5)

```rust
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```
