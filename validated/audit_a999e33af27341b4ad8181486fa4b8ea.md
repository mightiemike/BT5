### Title
Wrong Ancestor Used for Retarget Difficulty on Deep Forks — (`contract/src/bitcoin.rs` / `contract/src/lib.rs`)

---

### Summary

`get_next_work_required` fetches the retarget-interval start block via `blocks_getter.get_header_by_height(first_block_height)`. The `BlocksGetter` implementation resolves this call through `mainchain_height_to_header` — the mainchain-only height index. When a fork diverges **before** `first_block_height`, the fork's ancestor at that height is stored only in `headers_pool` (not in `mainchain_height_to_header`), so the lookup silently returns the **mainchain** block at that height. The difficulty for the fork's retarget block is therefore computed from the wrong ancestor's timestamp, breaking the invariant that each chain's difficulty is derived from its own ancestor chain.

---

### Finding Description

**Root cause — `get_header_by_height` is mainchain-only:** [1](#0-0) 

The implementation exclusively consults `mainchain_height_to_header`. Fork blocks are stored only in `headers_pool` (via `store_fork_header`), never in `mainchain_height_to_header`. [2](#0-1) 

**Contrast with `get_prev_header`**, which correctly follows the fork's hash-linked ancestor chain: [3](#0-2) 

**The retarget path that triggers the bug:** [4](#0-3) 

When `(prev_block_header.block_height + 1) % difficulty_adjustment_interval == 0`, `first_block_height` is computed as `prev_block_header.block_height - 2015`. If the fork diverged before that height, `get_header_by_height(first_block_height)` returns the **mainchain** block at that height, not the fork's ancestor. `calculate_next_work_required` then uses the mainchain block's timestamp, producing an `expected_bits` value that does not correspond to the fork's own ancestor chain.

**The check that is fooled:** [5](#0-4) 

`check_pow` enforces `expected_bits == block_header.bits`. Because `expected_bits` was derived from the wrong ancestor, a fork block whose `bits` matches the mainchain-derived target (rather than the fork-derived target) passes this check.

**Chainwork is accumulated from the submitted `bits`:** [6](#0-5) 

If the mainchain-derived target is easier (higher target value, lower work per block), the attacker's fork blocks carry less work per block but are accepted. If the mainchain-derived target is harder, the attacker's fork blocks carry more work per block and are accepted with inflated chainwork. Either way, the fork's chainwork is computed from bits that were validated against the wrong difficulty rule.

**Reorg path:** [7](#0-6) 

Once the fork's `chain_work` exceeds the mainchain's, `reorg_chain` is called, overwriting `mainchain_tip_blockhash` and the entire height-to-header index with the fork's blocks.

---

### Impact Explanation

A trusted relayer can submit a deep fork (diverging before the current retarget interval start) whose retarget block carries `bits` derived from the mainchain's ancestor timestamp rather than the fork's. The contract accepts the block, accumulates chainwork from those bits, and — if the fork's total chainwork exceeds the mainchain's — promotes the fork to mainchain. This corrupts `mainchain_tip_blockhash`, `chain_work`, and `mainchain_height_to_header`, directly breaking the light client's canonical-chain identity. All downstream consumers (transaction inclusion proofs, bridge relays) operate on a chain whose difficulty history is fabricated.

---

### Likelihood Explanation

The precondition — a fork that diverges more than 2016 blocks before a retarget boundary — is realistic for any long-running fork scenario the relay is designed to handle. The trusted relayer is the normal, production submission path (`submit_blocks`). No additional privilege beyond the trusted-relayer role is required. The attacker controls the `bits` field and the block timestamps in the submitted headers, so the necessary conditions can be constructed deliberately.

---

### Recommendation

Replace the height-based lookup with a hash-linked ancestor walk. At the retarget boundary, walk back `difficulty_adjustment_interval - 1` steps from `prev_block_header` using `get_prev_header` (which correctly follows `prev_block_hash` pointers through `headers_pool`). This mirrors Bitcoin Core's own implementation and ensures the interval-start block is always the fork's true ancestor regardless of what the mainchain index contains.

```rust
// Instead of:
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);

// Walk back through the fork's own ancestor chain:
let mut cursor = prev_block_header.clone();
for _ in 0..(config.difficulty_adjustment_interval - 1) {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let interval_tail_extend_header = cursor;
```

`get_header_by_height` should either be removed from the `BlocksGetter` trait or clearly documented as mainchain-only and never called during fork validation.

---

### Proof of Concept

**State setup:**
- Initialize with genesis at height `N` (a retarget boundary, `N % 2016 == 0`).
- Submit mainchain blocks `N` through `N + 2016`. The mainchain block at height `N` has timestamp `T_main`.
- Submit fork blocks diverging at height `N - 1` (before the retarget interval start), so the fork's block at height `N` has timestamp `T_fork ≠ T_main`.
- Continue the fork through heights `N` to `N + 2015`.

**Trigger:**
- Submit fork block at height `N + 2016` (the retarget boundary).
- `get_next_work_required` computes `first_block_height = N + 2016 - 2016 = N`.
- `get_header_by_height(N)` returns the **mainchain** block at height `N` (timestamp `T_main`).
- `calculate_next_work_required` uses `T_main`, producing `expected_bits_mainchain`.
- The fork block's `bits` is set to `expected_bits_mainchain` (not what `T_fork` would require).
- `check_pow` passes: `expected_bits_mainchain == block_header.bits`.

**Assert the invariant is broken:**
- Independently compute what `bits` the fork's own ancestor chain requires: use `T_fork` in `calculate_next_work_required` → `expected_bits_fork`.
- Assert `expected_bits_fork ≠ expected_bits_mainchain` (holds whenever `T_fork ≠ T_main`).
- The contract accepted a block whose `bits` does not match its own ancestor chain's required difficulty.

**Reorg:**
- If `expected_bits_mainchain` encodes a harder target than `expected_bits_fork`, the fork block carries more chainwork per block than it should. Submit enough fork blocks to exceed the mainchain's total chainwork; `reorg_chain` fires, corrupting `mainchain_tip_blockhash`.
