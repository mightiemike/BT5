### Title
Unsigned `ContractCodeResponse::V2` Accepted Without Signature Verification Below Protocol Version 85 — (`chain/client/src/stateless_validation/validate.rs`)

---

### Summary

`validate_contract_code_response` gates signature verification solely on `ProtocolFeature::SignedContractCodeResponse` being enabled (protocol version ≥ 85). It does **not** check whether the received variant is `V2`. A malicious peer can craft a borsh-encoded `ContractCodeResponse::V2` (discriminant `0x01`) with an arbitrary or zeroed signature and send it to a chunk-validator node running at any protocol version below 85. The function returns `Ok(ChunkRelevance::Relevant)` without ever calling `verify_signature`, and the injected contract bytes are stored into the witness-reconstruction pipeline.

---

### Finding Description

The gate in `validate_contract_code_response` is:

```rust
if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
    validate_witness_contract_code_response_signature(epoch_manager, response)?;
}
``` [1](#0-0) 

`SignedContractCodeResponse` maps to protocol version 85: [2](#0-1) 

The `ContractCodeResponse` enum has two variants with explicit borsh discriminants:

```rust
pub enum ContractCodeResponse {
    V1(ContractCodeResponseV1) = 0,   // no signature
    V2(ContractCodeResponseV2) = 1,   // carries signature + responder
}
``` [3](#0-2) 

`ContractCodeResponseV2` always carries a `signature` field and a `responder` account: [4](#0-3) 

The legitimate encoder only emits V2 when the feature is enabled: [5](#0-4) 

However, the receiver's validator does **not** mirror this invariant. When `protocol_version < 85`, the entire `validate_witness_contract_code_response_signature` call is skipped, regardless of which variant arrived. A peer can send discriminant `0x01` (V2) with a zeroed 64-byte signature and any `responder` account ID; the function returns `Ok(Relevant)`.

The accepted response is then immediately consumed:

```rust
let contracts = response.decompress_contracts()?;
self.partial_witness_tracker.store_accessed_contract_codes(key, contracts)
``` [6](#0-5) 

`validate_witness_contract_code_response_signature` — which would have checked that `responder` is a real chunk producer and that the signature over the borsh-serialized inner is valid — is never reached: [7](#0-6) 

---

### Impact Explanation

Any network peer (no validator or chunk-producer privileges required) can inject arbitrary contract bytes into the witness-reconstruction cache of a chunk-validator node for any currently-relevant `ChunkProductionKey`. The injected bytes are stored under the key and used during witness validation. This can cause a chunk validator to reconstruct a witness with wrong contract code, producing a divergent validation result — either a false rejection of a valid witness or acceptance of an incorrect one.

---

### Likelihood Explanation

The `ChunkProductionKey` (epoch ID, shard ID, height) is fully public from the chain. Any peer connected to a chunk-validator node can send a `ContractCodeResponseMessage`. The only prerequisite is knowing a currently-relevant key, which is trivially observable. The attack window is every epoch where the network protocol version is below 85.

---

### Recommendation

The variant and the feature gate must be checked together. Two equivalent fixes:

1. **Reject V2 when the feature is off** — add a check before the feature gate:
   ```rust
   if matches!(response, ContractCodeResponse::V2(_))
       && !ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version)
   {
       return Err(Error::Other("V2 response received before feature activation".into()));
   }
   ```

2. **Always verify the signature when the variant is V2** — move the signature check to be variant-driven rather than feature-driven:
   ```rust
   match response {
       ContractCodeResponse::V2(_) => {
           validate_witness_contract_code_response_signature(epoch_manager, response)?;
       }
       ContractCodeResponse::V1(_) => {
           if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
               return Err(Error::Other("V1 response after feature activation".into()));
           }
       }
   }
   ```

Option 2 also closes the symmetric gap where a V1 response could be sent after the feature activates.

### Proof of Concept

```rust
// Differential test (pseudo-code):
// 1. Build a ContractCodeResponseV2 with zeroed signature and arbitrary contract bytes.
// 2. Call validate_contract_code_response with epoch protocol_version = 84.
//    Assert: returns Ok(ChunkRelevance::Relevant)  ← signature never checked
// 3. Call validate_contract_code_response with epoch protocol_version = 85.
//    Assert: returns Err("Invalid witness contract code response signature")
```

The divergent value is concrete: `ContractCodeResponse::V2` (borsh discriminant `0x01`) with `signature = [0u8; 64]` returns `Ok(Relevant)` at protocol version 84 and `Err` at protocol version 85, demonstrating that the feature gate — not the variant — controls whether the invariant is enforced.

fn validate_witness_contract_code_response_signature(
    epoch_manager: &dyn EpochManagerAdapter,
    response: &ContractCodeResponse,
) -> Result<(), Error> {
    let Some(responder) = response.responder() else {
        return Err(Error::Other(
            "Unsigned contract code response in epoch where signature is required".to_owned(),
        ));
    };
    let key = response.chunk_production_key();
    let chunk_producers =
        epoch_manager.get_epoch_chunk_producers_for_shard(&key.epoch_id, key.shard_id)?;
    if !chunk_producers.contains(responder) {
        return Err(Error::Other(format!(
            "Contract code response responder {responder} is not a chunk producer for shard {} in epoch {:?}",
            key.shard_id, key.epoch_id,
        )));
    }
    let validator = epoch_manager.get_validator_by_account_id(&key.epoch_id, responder)?;
    if !response.verify_signature(validator.public_key()) {
        return Err(Error::Other("Invalid witness contract code response signature".to_owned()));
    }
    Ok(())
```
