### Title
Stale `digestToSubaccount` and `filledAmounts` After Isolated Subaccount Close Allows Margin-Free Position Opening - (`core/contracts/OffchainExchange.sol`)

---

### Summary

`_tryCloseIsolatedSubaccount` clears the subaccount's slot in `isolatedSubaccounts`, `isolatedSubaccountsMask`, and `parentSubaccounts`, but never clears `digestToSubaccount[digest]` or `filledAmounts[digest]`. When the same order is resubmitted via `createIsolatedSubaccount`, the function hits the early-return guard at line 1009 and returns the stale (closed) subaccount without transferring any margin. Subsequent `matchOrders` calls then fill against a zero-margin isolated subaccount while `filledAmounts[digest]` carries over the prior cycle's accumulated fills, incorrectly reducing the remaining fillable amount.

---

### Finding Description

**Root cause — `_tryCloseIsolatedSubaccount` does not clear digest-keyed state:**

`_tryCloseIsolatedSubaccount` (lines 160–208) performs the following cleanup:

```
isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);   // line 202
isolatedSubaccounts[parent][id] = bytes32(0);                // line 203
parentSubaccounts[subaccount]   = bytes32(0);                // line 204
```

It does **not** clear:
- `digestToSubaccount[digest]` — still maps the order digest to the now-closed subaccount
- `filledAmounts[digest]` — still holds the accumulated fill amount from the previous cycle [1](#0-0) 

**Early-return in `createIsolatedSubaccount` skips margin transfer:**

```solidity
if (digestToSubaccount[digest] != bytes32(0)) {
    return digestToSubaccount[digest];   // returns stale closed subaccount
}
```

Because `digestToSubaccount[digest]` was never cleared, this guard fires on re-submission. The margin transfer block (lines 1074–1087) is never reached, so the isolated subaccount receives **zero margin** while the parent's balance is unchanged. [2](#0-1) [3](#0-2) 

**`matchOrders` routes fills to the zero-margin subaccount using stale `filledAmounts`:**

`matchOrders` redirects the sender to the isolated subaccount via the still-populated `digestToSubaccount` mapping (lines 673–674), then `_validateOrder` subtracts the stale `filledAmounts[digest]` from `order.amount` (lines 435–436), reducing the remaining fillable quantity. If the prior cycle filled 100%, `order.amount` becomes 0 and the order is rejected outright (line 467). If it filled 50%, only 50% remains fillable — against a subaccount with no margin. [4](#0-3) [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Two distinct broken invariants, both with direct asset impact:

1. **Margin not transferred on re-create**: The trader's margin remains in the parent subaccount. The isolated subaccount that receives the fill has a zero quote balance. Any fill executed against it creates an undercollateralized position — the isolated subaccount owes quote it does not have, constituting a protocol-level bad debt.

2. **Stale `filledAmounts` reduces fillable quantity**: The remaining fillable amount is permanently reduced by the prior cycle's fills. If the order was fully filled before close, it is completely unfillable after re-create, meaning the trader cannot open the intended position at all despite the margin being available in the parent.

Combined: the trader either (a) cannot open the position at all (funds effectively locked in parent, order permanently exhausted), or (b) opens a position with zero margin in the isolated subaccount, creating bad debt in the protocol.

---

### Likelihood Explanation

The path is reachable through the normal sequencer-processed transaction flow (`CreateIsolatedSubaccount` → `MatchOrders` → `CloseIsolatedSubaccount` → `CreateIsolatedSubaccount` → `MatchOrders`). No privileged access is required. Any trader who partially fills an isolated order, has the subaccount closed (e.g., via liquidation or explicit `CloseIsolatedSubaccount`), and then resubmits the same signed order triggers this path. The same signed order bytes produce the same EIP-712 digest, so the stale mapping is deterministically hit.

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, after clearing the subaccount slot mappings, also clear the digest-keyed state. This requires iterating over (or storing) the digests associated with the subaccount, or storing a reverse mapping from subaccount to its originating digest(s):

```solidity
// After line 204:
delete digestToSubaccount[digestForSubaccount];
delete filledAmounts[digestForSubaccount];
delete digestToMargin[digestForSubaccount];
```

A reverse mapping `subaccountToDigest[subaccount]` should be populated in `createIsolatedSubaccount` at line 1072 and used here for the cleanup. Alternatively, `createIsolatedSubaccount` must check not only that `digestToSubaccount[digest] != bytes32(0)` but also that the returned subaccount is still active (i.e., `parentSubaccounts[subaccount] != bytes32(0)`), and if not, proceed with a fresh subaccount creation and margin transfer.

### Proof of Concept

State-machine test on unmodified Hardhat setup:

1. Trader calls `CreateIsolatedSubaccount` with `order` (amount=100, margin=M). Assert: `digestToSubaccount[digest] = isoSubacc`, isolated subaccount quote balance = M, parent quote balance decremented by M.
2. Sequencer submits `MatchOrders` filling 50 units. Assert: `filledAmounts[digest] = 50`, isolated subaccount has a perp position of 50.
3. Sequencer submits `CloseIsolatedSubaccount(isoSubacc)` (position must be 0 first — close the perp position via another `MatchOrders`, then close). Assert: `parentSubaccounts[isoSubacc] = 0`, `isolatedSubaccounts[parent][id] = 0`, parent quote balance restored. **Assert: `digestToSubaccount[digest]` is still set (not zero). `filledAmounts[digest]` is still 50.**
4. Trader resubmits `CreateIsolatedSubaccount` with the identical `order`. Assert: function returns `isoSubacc` immediately (early return). Assert: parent quote balance **unchanged** (no margin deducted). Assert: isolated subaccount quote balance = 0.
5. Sequencer submits `MatchOrders` for the remaining 50 units. Assert: `_validateOrder` computes `order.amount = 100 - 50 = 50` (stale `filledAmounts`). Fill executes against `isoSubacc` with 0 quote balance. Assert: isolated subaccount quote balance is now negative (bad debt). Assert: `filledAmounts[digest] = 100`. [7](#0-6) [2](#0-1) [8](#0-7)
