### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Allowlist Bypass via Router — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so `sender` = router address. If the router is allowlisted (required for router-mediated swaps to function), every user can bypass the per-user allowlist by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (correct), and `sender` is the first argument forwarded by the pool from its own `msg.sender` — i.e., whoever called `pool.swap()`. [1](#0-0) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
// MetricOmmSimpleRouter.sol lines 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
``` [2](#0-1) 

The pool's `msg.sender` is the router, so the pool passes the router address as `sender` to `_beforeSwap`, which then forwards it to the extension: [3](#0-2) 

The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

This creates an irreconcilable conflict:

| Router allowlisted? | Effect |
|---|---|
| **Yes** (needed for router swaps to work) | Every user bypasses the per-user allowlist by routing through the router |
| **No** | Allowlisted users cannot swap through the router at all |

There is no configuration that simultaneously (a) allows router-mediated swaps and (b) enforces per-user restrictions. The same structural problem exists in the multi-hop `exactInput` path: [4](#0-3) 

---

### Impact Explanation

A pool admin who deploys `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC'd addresses, whitelisted market makers, or to prevent adverse selection against LPs) cannot enforce that restriction for any user who routes through `MetricOmmSimpleRouter`. If the router is allowlisted — the only way to make router-mediated swaps functional — the allowlist is completely bypassed for all users. This breaks the core pool access-control functionality and can expose LP funds to unauthorized trading activity the admin explicitly intended to block.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any pool that (a) configures `SwapAllowlistExtension` and (b) needs to support router-mediated swaps must allowlist the router, triggering the bypass. The attacker path requires no special privilege: any user can call the public router.

---

### Recommendation

The extension must gate the **economic actor**, not the intermediary. Two viable approaches:

1. **Forward the original caller in `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap`. The extension decodes and checks that address. This requires a convention between router and extension.

2. **Check `recipient` instead of `sender`** (if the pool's design guarantees recipient = economic actor): Evaluate whether `recipient` more reliably identifies the end user for allowlist purposes.

3. **Dedicated router-aware extension**: Create a `SwapAllowlistExtension` variant that accepts a trusted forwarder list and decodes the real caller from a signed or transient-storage-backed context.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // only Alice allowed
3. Pool admin calls setAllowedToSwap(pool, router, true)  // router must be allowed for router swaps
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, tokenIn: ..., ...})
5. Router calls pool.swap(...) — pool's msg.sender = router
6. Pool calls extension.beforeSwap(router, ...)
7. Extension checks: allowedSwapper[pool][router] == true  → passes
8. Bob's swap executes successfully despite not being allowlisted.

Conversely, if step 3 is omitted:
4. Alice (allowlisted) calls router.exactInputSingle(...)
5. Router calls pool.swap(...) — pool's msg.sender = router
6. Extension checks: allowedSwapper[pool][router] == false → reverts NotAllowedToSwap
7. Alice cannot swap through the router even though she is allowlisted.
```
