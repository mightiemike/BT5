### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables USDC.e Drain Without USDC Payment - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the boolean return value. If the USDC token on chain 57073 returns `false` on failure rather than reverting, any caller can drain USDC.e from a victim's DDA without providing USDC in exchange.

---

### Finding Description

The function `replaceUsdcEWithUsdc` is a publicly callable migration helper that swaps USDC.e held in a Direct Deposit Account (DDA) for USDC. Its logic is:

1. Pull USDC from `msg.sender` into the DDA via `transferFrom` (unchecked return value).
2. Withdraw USDC.e from the DDA to `ContractOwner` via `DirectDepositV1.withdraw()`.
3. Send the USDC.e to `msg.sender` via `safeTransfer` (return value IS checked here).

The critical flaw is in step 1:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [1](#0-0) 

The return value of this call is silently discarded. If the USDC token implementation on chain 57073 (Ink mainnet) returns `false` on a failed transfer rather than reverting — a pattern seen in older or non-standard ERC20 tokens — execution continues unconditionally into steps 2 and 3. The USDC.e is then transferred out of the DDA and delivered to the attacker, with no USDC deposited in return.

By contrast, the outbound USDC.e transfer in step 3 uses `safeTransfer` (via `ERC20Helper`), which does check the return value. The asymmetry between the two token calls is the root cause. [2](#0-1) 

The `ERC20Helper.safeTransferFrom` wrapper that correctly validates return values exists in the codebase and is used elsewhere, but was not applied here. [3](#0-2) 

---

### Impact Explanation

If the USDC token on chain 57073 returns `false` on failure (e.g., insufficient balance or missing approval) rather than reverting:

- An attacker calls `replaceUsdcEWithUsdc(victimSubaccount)` with no USDC balance or approval.
- `transferFrom` returns `false`; the return value is ignored; execution continues.
- `DirectDepositV1.withdraw()` moves the victim DDA's entire USDC.e balance to `ContractOwner`.
- `safeTransfer` sends that USDC.e to the attacker.
- **Net result**: attacker receives victim's USDC.e for free; DDA is drained.

The corrupted asset delta is the full USDC.e balance of any DDA on chain 57073 that holds USDC.e. There is no cap on the number of DDAs that can be targeted in sequence.

---

### Likelihood Explanation

Likelihood is **low**. The function is chain-gated to `block.chainid == 57073` (Ink mainnet). The USDC token at the hardcoded address `0x2D270e6886d130D724215A266106e6832161EAEd` on that chain may revert on failure rather than returning `false`. However:

- The behavior of this specific bridged USDC deployment cannot be confirmed from the codebase alone.
- The function is callable by any unprivileged address with no preconditions beyond the chain check.
- The pattern is a known ERC20 integration pitfall; the codebase itself acknowledges this by using `safeTransferFrom` everywhere else.

---

### Recommendation

Replace the raw `transferFrom` call with the `safeTransferFrom` wrapper already present in `ERC20Helper`, consistent with every other token transfer in the codebase:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [3](#0-2) 

---

### Proof of Concept

**Precondition**: DDA for `victimSubaccount` holds 10,000 USDC.e. USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on failed `transferFrom` rather than reverting.

1. Attacker (with zero USDC balance) calls `ContractOwner.replaceUsdcEWithUsdc(victimSubaccount)` on chain 57073.
2. `balance = IERC20Base(usdcE).balanceOf(directDepositV1)` → 10,000.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, 10000)` → returns `false` (attacker has no USDC). Return value ignored.
4. `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` → 10,000 USDC.e transferred from DDA to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, 10000)` → 10,000 USDC.e sent to attacker.
6. Attacker holds 10,000 USDC.e. DDA holds 0. No USDC was deposited. [2](#0-1)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
