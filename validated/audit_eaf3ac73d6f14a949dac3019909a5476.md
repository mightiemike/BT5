### Title
Arithmetic Overflow in `op_unknown` Cost Multiplier Produces Undercharged Execution Cost — (File: `src/more_ops.rs`)

### Summary
In `op_unknown`, the final cost is computed by multiplying a base cost (bounded by `max_cost`) by an attacker-controlled `cost_multiplier` derived from the opcode bytes. This multiplication is performed in unchecked `u64` arithmetic in Rust release mode. When the product overflows `u64`, it wraps to a small value. The only post-multiplication guard checks `cost > u32::MAX`, which does not detect overflow. A wrapped value ≤ `u32::MAX` is returned as the legitimate cost, causing the CLVM evaluator to accept a program as cheap when it should be rejected as exceeding the cost limit.

### Finding Description

In `src/more_ops.rs`, `op_unknown` computes the cost for unknown opcodes in consensus (lenient) mode: [1](#0-0) 

The `cost_multiplier` is decoded from the opcode atom bytes and is at most `u32::MAX` (4,294,967,295). The base `cost` is computed from the arguments according to `cost_function` (0–3), then checked against `max_cost`: [2](#0-1) 

The sequence is:
1. `check_cost(cost, max_cost)` — passes if `cost ≤ max_cost` (e.g., ≤ 11 billion for Chia)
2. `cost *= cost_multiplier + 1` — **unchecked u64 multiplication**; wraps silently in release mode
3. `if cost > u32::MAX` — only checks the *wrapped* value, not whether overflow occurred

`Cost` is `u64`: [3](#0-2) 

**Concrete overflow path:**
- `cost_multiplier = 0x7FFFFFFF` → `cost_multiplier + 1 = 2^31 = 2,147,483,648`
- Opcode bytes: `[0x7F, 0xFF, 0xFF, 0xFF, 0x00]` (5 bytes; first byte ≠ `0xFF` so not reserved; last byte `0x00` → `cost_function = 0`)
- With `cost_function = 1` and enough atom arguments, drive base `cost` to exactly `2^33 = 8,589,934,592` (within Chia's ~11 billion block limit)
- `2^33 * 2^31 = 2^64 ≡ 0 (mod 2^64)` → wrapped cost = **0**
- `0 ≤ u32::MAX` → returns `Ok(Reduction(0, nil))` — **zero cost charged**

The `assert!(cost > 0)` at line 258 fires only on the *pre-multiplication* value, not after: [4](#0-3) 

### Impact Explanation

`op_unknown` is invoked in consensus mode for every unknown opcode in a CLVM program. The returned cost is accumulated into the program's total cost in `run_program`: [5](#0-4) 

If the cost of an unknown op wraps to 0 (or a small value), the program's total cost is underreported. A program that should be rejected for exceeding `max_cost` is instead accepted. This breaks the cost-metering invariant that prevents DoS attacks on Chia full nodes. An attacker can craft a block containing programs with many such unknown ops, each appearing nearly free, while consuming significant real CPU/memory resources on every validating node.

### Likelihood Explanation

The opcode bytes are fully attacker-controlled CLVM input. The `cost_multiplier` is decoded directly from those bytes with no cap below `u32::MAX`. Driving the base `cost` to a specific multiple of `2^33` requires crafting the argument list (atom count and sizes), which is also attacker-controlled. The overflow is deterministic and reproducible across all nodes running in release mode (the standard deployment mode). No privileged access is required — any transaction submitter can include such a program.

### Recommendation

Replace the unchecked multiplication with a checked or saturating variant, and reject on overflow:

```rust
// Before (line 261):
cost *= cost_multiplier + 1;

// After:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

Alternatively, validate that `cost * (cost_multiplier + 1) ≤ u32::MAX` before performing the multiplication, using integer arithmetic that cannot overflow.

### Proof of Concept

```
Opcode bytes: [0x7F, 0xFF, 0xFF, 0xFF, 0x41]
  - op[0..4] = [0x7F, 0xFF, 0xFF, 0xFF] → cost_multiplier = 0x7FFFFFFF = 2147483647
  - op[4] = 0x41 = 0b01000001 → cost_function = (0x41 & 0xC0) >> 6 = 1 (arith-like)

Arguments: enough atoms totaling ~26,843,545 bytes
  cost = ARITH_BASE_COST + n*ARITH_COST_PER_ARG + total_bytes*ARITH_COST_PER_BYTE
       = 99 + n*320 + 26843545*3
  Tune n and total_bytes so cost = 2^33 = 8589934592

check_cost(8589934592, max_cost=11000000000) → Ok  (passes)
cost *= (2147483647 + 1) = cost * 2^31
     = 2^33 * 2^31 = 2^64 ≡ 0 (mod 2^64)  [u64 wraps in release mode]

if 0 > u32::MAX → false
→ Ok(Reduction(0, nil))   // zero cost returned for a high-cost operation
```

The program's total cost is undercharged by up to `max_cost` per such opcode invocation, allowing an attacker to include arbitrarily many such opcodes in a block while staying under the reported cost limit.
