### Title
`DISABLE_OP` in `MEMPOOL_MODE` Disables Opcode 60 (`modpow`) in Mempool but Not Consensus, Creating a Coin-Locking Divergence ŌĆö (`src/chia_dialect.rs`)

---

### Summary

The `DISABLE_OP` flag is permanently wired into `MEMPOOL_MODE` but absent from consensus-mode flag sets. This causes opcode 60 (`op_modpow`) to be silently rejected by the mempool while being fully accepted by consensus. A coin whose puzzle uses opcode 60 can be committed on-chain (consensus accepts it), but the owner can never spend it through the mempool (mempool rejects it), locking the coin's value with no recovery path short of finding a cooperative farmer who bypasses the mempool entirely.

---

### Finding Description

In `src/chia_dialect.rs`, `MEMPOOL_MODE` is defined as:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)      // ŌåÉ present only here
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

`DISABLE_OP` carries no documentation and is not exported as a hardfork activation flag. Its sole effect is in the operator dispatch table:

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
```

Every other flag in `MEMPOOL_MODE` (`NO_UNKNOWN_OPS`, `LIMIT_HEAP`, `CANONICAL_INTS`, `LIMIT_SOFTFORK`) enforces a stricter subset of what consensus also enforces. `DISABLE_OP` is different: it **inverts** the validity of opcode 60 between the two evaluation contexts.

| Evaluation context | `DISABLE_OP` set? | Opcode 60 result |
|---|---|---|
| Mempool (`MEMPOOL_MODE`) | Yes | `EvalErr::Unimplemented` |
| Consensus (no `DISABLE_OP`) | No | `op_modpow` executes normally |

Because a coin's puzzle hash is just a hash of the puzzle bytes, a coin whose puzzle body calls opcode 60 can be created on-chain without any mempool involvement (e.g., via a coinbase reward or a farmer-direct inclusion). Once that coin exists, every spend attempt submitted through the standard mempool path is rejected unconditionally, regardless of whether the solution is correct.

The entry path is fully attacker-controlled: an adversary can craft a puzzle that calls opcode 60, send funds to its puzzle hash, and the victim's only recovery options are (a) finding a farmer willing to include the spend bundle out-of-band, or (b) waiting for a protocol change that removes `DISABLE_OP` from `MEMPOOL_MODE`.

---

### Impact Explanation

- **Corrupted result**: the mempool returns `EvalErr::Unimplemented` for a spend that consensus would accept and execute correctly, producing a valid `Reduction`.
- **Coin-locking**: any coin whose puzzle evaluates opcode 60 is permanently unspendable through the mempool. The owner's funds are locked for an indefinite period with no protocol-level unbonding mechanism.
- **Consensus divergence**: a farmer running consensus-mode validation can include the spend in a block; full nodes that re-validate in consensus mode accept it; but any node that re-runs mempool-mode validation on the same spend would reject it, creating a split view of validity.

---

### Likelihood Explanation

- `DISABLE_OP` is unconditionally present in `MEMPOOL_MODE` with no activation-height guard or hardfork flag to remove it.
- Opcode 60 (`modpow`) is a documented, named operator in the dispatch table and is reachable in consensus mode; developers writing puzzles in consensus-mode test environments will observe it working.
- There is no warning, compile-time guard, or runtime signal that opcode 60 is mempool-banned. A developer testing against a consensus-mode runner will ship a puzzle that silently breaks in production.
- The `DISABLE_OP` flag is not exported to Python callers in `wheel/src/api.rs` under a self-documenting name, so Python-side callers cannot easily detect or replicate the restriction.

---

### Recommendation

1. **Document the intent**: add a comment to `DISABLE_OP` explaining which operator(s) it disables and why it belongs in `MEMPOOL_MODE` but not consensus mode.
2. **Align mempool and consensus**: if `modpow` is intended to be permanently unavailable, remove it from the consensus dispatch table entirely so both modes agree. If it is a transitional restriction, introduce a hardfork activation flag (parallel to `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`) that removes `DISABLE_OP` from `MEMPOOL_MODE` at the correct block height.
3. **Export the flag**: if `DISABLE_OP` must remain, export it from `wheel/src/api.rs` alongside the other `ClvmFlags` constants so Python callers can detect and replicate the restriction in their own validation logic.

---

### Proof of Concept

**Trigger**: run the same CLVM program under mempool flags and consensus flags and observe divergent results.

```python
import clvm_rs

# Puzzle: (modpow (q . 2) (q . 10) (q . 1000))  ŌåÆ opcode 60
puzzle_hex = bytes.fromhex("ff3cff02ff0aff8203e880")  # serialized form
args_hex   = bytes.fromhex("80")                        # nil env

CONSENSUS_FLAGS = 0          # no DISABLE_OP
MEMPOOL_FLAGS   = clvm_rs.MEMPOOL_MODE  # includes DISABLE_OP

cost_c, result_c = clvm_rs.run_serialized_chia_program(
    puzzle_hex, args_hex, 10_000_000, CONSENSUS_FLAGS
)
print("Consensus:", cost_c, result_c)   # succeeds, returns 1024 mod 1000 = 24

try:
    clvm_rs.run_serialized_chia_program(
        puzzle_hex, args_hex, 10_000_000, MEMPOOL_FLAGS
    )
except Exception as e:
    print("Mempool:", e)   # EvalErr::Unimplemented ŌĆö spend rejected
```

A coin locked with this puzzle is accepted by consensus (block is valid) but permanently rejected by the mempool (spend bundle never propagates). The owner's funds are locked with no protocol-level recovery path, directly mirroring the Babylon staking-transaction lock described in the reference report.
