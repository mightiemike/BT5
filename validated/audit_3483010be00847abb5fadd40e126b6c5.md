Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` validates the `sender` argument passed by the pool, which equals `msg.sender` of the `pool.swap(...)` call. When users route through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the router is allowlisted, every user — including those explicitly excluded — can bypass the per-user swap gate by routing through the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the `msg.sender` of that call: [3](#0-2) 

The allowlist check therefore resolves to `allowedSwapper[pool][router]`. If the pool admin has allowlisted the router, the check passes for every user who calls the router, regardless of whether that individual user is on the allowlist. The same substitution occurs in `exactInput`, `exactOutputSingle`, and `exactOutput`: [4](#0-3) 

There is no mechanism in the extension or the pool to recover the original `msg.sender` of the router call. The `extensionData` field is passed through unchanged from the router to the pool to the extension, but the extension does not read it for identity verification.

`DepositAllowlistExtension` is not affected because it checks the `owner` parameter (the position owner), not the `sender` (the payer/caller): [5](#0-4) 

## Impact Explanation
A pool admin who deploys a pool with `SwapAllowlistExtension` intends to restrict swapping to a specific set of addresses (e.g., KYC-verified counterparties, whitelisted market makers, or compliance-gated participants). If the admin also allowlists the router as a trusted intermediary — a natural and expected operational step — the per-user gate is silently voided. Any address can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps against the pool. This breaks the core access-control invariant the extension is designed to enforce and constitutes broken core pool functionality with direct fund-flow consequences: unauthorized parties can execute swaps that the pool admin explicitly intended to block.

## Likelihood Explanation
The trigger requires two conditions that commonly co-occur in production: (1) a pool is deployed with `SwapAllowlistExtension` configured in `beforeSwapOrder`, and (2) the pool admin allowlists `MetricOmmSimpleRouter` as a permitted swapper. Both conditions are operationally natural. The router is the canonical user-facing entry point; allowlisting it while restricting direct pool calls is a standard pattern. No privileged escalation, malicious setup, or non-standard token is required — any unprivileged user can call the public router.

## Recommendation
`SwapAllowlistExtension.beforeSwap` should gate on the end user, not the intermediary. Two options:

1. **Pass the real user through the router**: Add a `realSender` field to `extensionData` that the router populates with `msg.sender` before calling `pool.swap`. The extension reads and validates this field (with the pool verifying the router's identity before trusting the payload).

2. **Check `sender` against a router registry and then require the router to attest the real user**: The extension rejects any `sender` that is a known router unless the `extensionData` carries a signed or transient-storage-backed real-user identity.

The simplest safe fix is to document that the router must never be allowlisted as a swapper and that per-user allowlisting requires direct pool calls — but this is fragile and should be enforced in code.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension in beforeSwapOrder
  allowedSwapper[pool][router] = true   // admin allowlists the router
  allowedSwapper[pool][alice] = false   // alice is explicitly excluded

Attack:
  alice calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  → router calls pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)
    → pool calls _beforeSwap(msg.sender=router, ...)
      → SwapAllowlistExtension.beforeSwap(sender=router, ...)
        → allowedSwapper[pool][router] == true  → passes
  → swap executes; alice receives output tokens

Result:
  alice, who is explicitly excluded from the allowlist, successfully swaps
  against the pool through the router, bypassing the admin-configured guard.
```
