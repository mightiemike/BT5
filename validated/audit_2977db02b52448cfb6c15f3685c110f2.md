### Title
Checkpoint Advanced Before Pool Check in `processEventsAndClaimBonus` Causes Permanent Bonus Reward Loss on Redelegation Slash — (`x/tieredrewards/keeper/claim_rewards.go`, `x/tieredrewards/keeper/slash.go`)

### Summary
When `BeforeRedelegationSlashed` fires and the bonus pool is insufficient, `processEventsAndClaimBonus` advances `pos.LastBonusAccrual` (via `applyBonusAccrualCheckpoint`) **before** checking pool sufficiency. The `ErrInsufficientBonusPool` error is then swallowed in `slashRedelegationPosition`, and the position is persisted with the advanced checkpoint. The accrued-but-unpaid bonus segment is permanently lost.

### Finding Description

The ordering bug is in `processEventsAndClaimBonus`: [1](#0-0) 

`applyBonusAccrualCheckpoint` mutates `pos.Position.LastBonusAccrual` at line 215. The pool sufficiency check at line 230 comes **after** this mutation. When the pool is insufficient, the function returns `ErrInsufficientBonusPool` with the checkpoint already advanced on the in-memory `pos`.

In `slashRedelegationPosition`, the error is explicitly swallowed: [2](#0-1) 

Because `processEventsAndClaimBonus` received `&pos` (a pointer), the mutation to `pos.LastBonusAccrual` persists in the caller's local `pos`. The function then calls `setPositionWithState` with the mutated `pos`: [3](#0-2) 

Both the full-slash path (line 71) and the partial-slash path (line 77) persist the position with the advanced checkpoint, permanently erasing the unpaid bonus segment from the position's history.

### Impact Explanation

Position owners permanently lose accrued bonus rewards for the time segment between their previous `LastBonusAccrual` and the block time of the slash. The segment is computed, the checkpoint is advanced past it, but no coins are transferred. On the next claim, `processEventsAndClaimBonus` will start from the already-advanced checkpoint and will not recompute the lost segment.

### Likelihood Explanation

The bonus pool is a module account funded externally. It can reach near-zero through normal operation (many positions claiming rewards). A validator slash with active redelegations is a normal chain event. No attacker control of a validator is required — any position that happens to be in a redelegation when the pool is low is affected. The pool does not need to be fully drained; it only needs to be insufficient for the specific bonus amount owed to the slashed position.

### Recommendation

Move `applyBonusAccrualCheckpoint` to **after** the successful `SendCoinsFromModuleToAccount` call, so the checkpoint only advances when the bonus is actually paid. Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, explicitly revert the checkpoint mutation on `pos` before persisting (e.g., by saving `pos.LastBonusAccrual` before the call and restoring it on error).

### Proof of Concept

1. Fund the `RewardsPoolName` module account with a small amount (e.g., 1 token).
2. Create a position with a large delegation that has accrued significant bonus rewards (e.g., 100 tokens worth).
3. Begin a redelegation for that position.
4. Trigger `BeforeRedelegationSlashed` (via the staking hook) for the redelegation's unbonding ID.
5. Assert: `pos.LastBonusAccrual` has been advanced to block time (checkpoint moved).
6. Assert: the position owner's balance has **not** increased by the bonus amount.
7. Call `claimRewards` for the position — assert that the previously accrued segment yields zero bonus (it was skipped).

The key assertion is that `LastBonusAccrual` after the slash equals the block time of the slash, while the owner received 0 bonus coins for the accrued segment. [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
