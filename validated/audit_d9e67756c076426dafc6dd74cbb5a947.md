Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Originating User, Allowing Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps using the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` of `pool.swap()`, so the extension checks whether the **router** is allowlisted rather than the originating user. If the pool admin allowlists the router to enable normal trading, every address in the world can bypass the curated pool's swap gate by routing through the public router.

## Finding Description

The full call chain is confirmed in production code:

1. `MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` directly — the router is `msg.sender` inside `MetricOmmPool.swap`.

2. `MetricOmmPool.swap` (L230–240) calls `_beforeSwap(msg.sender, recipient, ...)`, forwarding the router address as `sender`.

3. `ExtensionCalling._beforeSwap` (L149–177) ABI-encodes `sender = router` and dispatches to the extension.

4. `SwapAllowlistExtension.beforeSwap` (L37) executes:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
       revert IMetricOmmPoolActions.NotAllowedToSwap();
   }
   ```
   Here `msg.sender` = pool (correct), but `sender` = **router address**. The check reads `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

Two exploitable outcomes follow:

- **Bypass**: If the pool admin allowlists the router (the natural operational choice so that normal users can trade), every address can swap through the router and pass the gate, defeating the curation entirely.
- **Lockout**: If the pool admin does not allowlist the router, every allowlisted user who uses the router is incorrectly blocked.

The same structural flaw applies to `exactInput` (multi-hop), `exactOutputSingle`, and `exactOutput`, because in every case the router is the direct caller of `pool.swap()`. The existing `onlyPool` guard in `BaseMetricExtension` only verifies that the caller is a registered pool — it does not fix the identity of the `sender` argument passed into the hook.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of addresses loses that restriction entirely for any user routing through `MetricOmmSimpleRouter`. The attacker pays no special cost: they call the public router with the target pool address. Tokens flow out of the pool at oracle-derived prices to an address the pool admin never intended to allow. This is a direct loss of the pool's curation invariant — unauthorized swaps drain LP-owned token reserves at live oracle prices, constituting a broken core pool functionality and an admin-boundary bypass reachable by any unprivileged trader.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface for the protocol. Any user who reads the docs or inspects deployed contracts will find it. No privileged access, no special setup, and no front-running is required. The bypass is reachable in a single transaction by any EOA or contract. The only precondition is that the pool admin has allowlisted the router — a natural and expected operational step for any pool that intends to support router-mediated trading.

## Recommendation

The extension must gate the **originating user**, not the immediate caller of `pool.swap()`. The preferred fix is to check `recipient` instead of `sender` in `beforeSwap`, since `recipient` is the address that receives output tokens and is the economically relevant actor — it is already passed as the second argument to `beforeSwap`. Alternatively, redesign the allowlist to require the router to embed the real user identity in `extensionData` and have the extension decode and verify it. A documentation-only mitigation (marking the extension incompatible with router flows) is insufficient given the public router is the primary trading interface.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `alice` is allowlisted.
// Pool admin allowlists the router so normal users can trade:
//   allowedSwapper[pool][router] = true
//   allowedSwapper[pool][bob]    = false  (bob is not allowlisted)

// Bob (not allowlisted) calls the router:
router.exactInputSingle(ExactInputSingleParams({
    pool: curated_pool,
    recipient: bob,
    zeroForOne: true,
    amountIn: 1000e6,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    tokenIn: token0,
    extensionData: ""
}));

// Inside MetricOmmPool.swap():
//   msg.sender = router → _beforeSwap(sender=router, recipient=bob, ...)
//
// Inside SwapAllowlistExtension.beforeSwap(sender=router, ...):
//   allowedSwapper[pool][router] == true  → check passes
//   → Bob's swap executes at oracle price, draining LP reserves
//   → Bob was never allowlisted; the curation invariant is broken
```

A Foundry integration test can reproduce this by deploying a pool with `SwapAllowlistExtension`, allowlisting only the router address (simulating the admin enabling router trading), then calling `router.exactInputSingle` from an address not in `allowedSwapper` and asserting the swap succeeds when it should revert.
