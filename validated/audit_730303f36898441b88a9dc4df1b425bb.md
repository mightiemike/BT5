### Title
Fork Difficulty Retarget Uses Mainchain Block Timestamp Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When a fork block falls on a difficulty retarget boundary, the difficulty adjustment functions call `get_header_by_height` to retrieve the timestamp of the first block in the retarget interval. The `BtcLightClient` implementation of `get_header_by_height` always reads from the **mainchain** height index, not from the fork's actual ancestor chain. If the fork diverged before the retarget boundary, the fork's true ancestor at that height is a different block than the mainchain block at that height. The difficulty target for the fork block is therefore computed using the wrong timestamp, allowing an attacker to submit fork blocks validated against an incorrectly computed (potentially easier) target.

---

### Finding Description

The `BlocksGetter` trait has two methods: `get_prev_header` (walks the chain via `prev_block_hash` links) and `get_header_by_height` (looks up by height index).

`BtcLightClient::get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

This unconditionally reads from `mainchain_height_to_header` — the mainchain index.

During difficulty adjustment for Bitcoin and Litecoin, when `(prev_block_height + 1)` is a multiple of `difficulty_adjustment_interval`, the code calls:

```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [2](#0-1)

When a fork diverges at height `F` and a retarget boundary falls at height `R > F`, the fork's actual ancestor at height `R` is a **different block** than the mainchain block at height `R`. The `get_header_by_height` call returns the mainchain block's timestamp, not the fork ancestor's timestamp. `calculate_next_work_required` then computes a target based on this wrong timestamp, producing a difficulty target that does not correspond to the fork's actual chain history.

The analog to the external report is direct: just as rewards were computed using the last-updated level (stale state) instead of the current level, here the difficulty target is computed using the mainchain block's timestamp (stale/wrong state) instead of the fork's actual ancestor's timestamp (correct current state).

---

### Impact Explanation

`calculate_next_work_required` uses the timestamp difference between the retarget interval's first and last blocks to scale the difficulty target:

```rust
let mut actual_time_taken: i64 = prev_block_time - first_block_time;
// clamped to [pow_target_timespan/4, pow_target_timespan*4]
let new_target = target_from_bits(prev_block_header.block_header.bits);
new_target * actual_time_taken / pow_target_timespan
``` [5](#0-4) 

If the mainchain block at `first_block_height` has a timestamp that is **earlier** than the fork's true ancestor at that height, `actual_time_taken` is inflated, producing a **higher (easier) target**. The attacker's fork blocks are then validated against this easier target, requiring less real PoW. The fork's `chain_work` is accumulated using the correct per-block work formula, but the blocks themselves were accepted with insufficient PoW, so the accumulated `chain_work` is inflated relative to honest effort.

Once the fork's `chain_work` exceeds the mainchain's `chain_work`, `submit_block_header_inner` triggers `reorg_chain`, replacing the canonical chain with the attacker's fork:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [6](#0-5) 

A successful fraudulent reorg corrupts `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` — the canonical chain state used by `verify_transaction_inclusion` for SPV proofs. Any downstream NEAR contract relying on SPV proofs from this light client would accept proofs against a fraudulent chain.

---

### Likelihood Explanation

The precondition is that a fork diverges at or before a retarget boundary (every 2016 blocks for Bitcoin/Litecoin, every block for Dogecoin post-145k). For Bitcoin and Litecoin this is a periodic event; for Dogecoin it is constant. An attacker who has been building a private fork since before a retarget boundary can exploit this on every retarget that falls within the fork. The entry point is the permissionless `submit_blocks()` call — no privileged role is required.

---

### Recommendation

Replace `get_header_by_height` in the difficulty adjustment path with a function that walks the fork's actual ancestor chain via `get_prev_header` links to find the block at the target height. This is the correct approach referenced in the Dogecoin TODO comment. The `get_header_by_height` shortcut is only safe when the block being validated is guaranteed to extend the mainchain tip (i.e., `prev_block_header.block_hash == mainchain_tip_blockhash`), and must not be used for fork blocks.

---

### Proof of Concept

1. The mainchain contains blocks at heights 0–2016, with the retarget block at height 2016. The mainchain block at height 1 has timestamp `T_main_1`.

2. An attacker submits a fork block at height 1 (diverging from genesis) with timestamp `T_fork_1`, where `T_fork_1 > T_main_1`. The fork block passes PoW at the genesis difficulty.

3. The attacker builds fork blocks at heights 2–2015 at the genesis difficulty (no retarget applies yet).

4. At height 2016, the retarget fires. `get_next_work_required` calls `get_header_by_height(1)`, which returns the **mainchain** block at height 1 with timestamp `T_main_1`.

5. `actual_time_taken = T_fork_2015 - T_main_1`. Because `T_main_1 < T_fork_1`, this is larger than the true fork interval `T_fork_2015 - T_fork_1`, producing a higher (easier) target for the fork's block 2016.

6. The attacker mines fork block 2016 against this easier target, accumulating more `chain_work` per block than the mainchain required.

7. The attacker continues extending the fork. Eventually `fork.chain_work > mainchain.chain_work`, triggering `reorg_chain` via `submit_blocks()`.

8. The canonical chain is now the attacker's fork. `verify_transaction_inclusion` returns `true` for transactions in the attacker's fraudulent blocks. [7](#0-6) [8](#0-7)
