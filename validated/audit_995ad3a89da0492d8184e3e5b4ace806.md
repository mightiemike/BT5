### Title
Unbounded `ExitDuration` in Tier Validation Allows Governance to Permanently Lock User Funds - (File: x/tieredrewards/types/tier.go)

### Summary

`Tier.Validate()` enforces only a lower bound (`ExitDuration > 0`) on the exit lock duration, with no upper bound. A governance proposal passing `MsgUpdateTier` or `MsgAddTier` with an arbitrarily large `ExitDuration` (up to `math.MaxInt64` nanoseconds ≈ 292 years) causes every subsequent `MsgTriggerExitFromTier` call to set `Position.ExitUnlockAt` to a time far in the future, permanently blocking users from undelegating or withdrawing their staked tokens.

### Finding Description

In `x/tieredrewards/types/tier.go`, `Tier.Validate()` checks:

```go
if t.ExitDuration <= 0 {
    return fmt.Errorf("exit duration must be positive")
}
```

There is no upper bound. Compare this to `BonusApy`, which has an explicit cap with a comment explaining the rationale:

```go
// Cap BonusApy at 100% to prevent governance from draining the rewards pool.
if t.BonusApy.GT(math.LegacyOneDec()) {
    return fmt.Errorf("bonus apy must not exceed 1.0 (100%%): got %s", t.BonusApy)
}
```

No equivalent guard exists for `ExitDuration`.

When a user calls `MsgTriggerExitFromTier`, the keeper executes:

```go
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
```

which calls:

```go
func (p *Position) TriggerExit(blockTime time.Time, duration time.Duration) {
    p.ExitTriggeredAt = blockTime
    p.ExitUnlockAt = blockTime.Add(duration)
}
```

`ExitUnlockAt` is the sole gate for `MsgTierUndelegate`, `MsgWithdrawFromTier`, and `MsgExitTierWithDelegation`. All three require `CompletedExitLockDuration(blockTime)` to be true, i.e., `block_time >= ExitUnlockAt`. With `ExitDuration = math.MaxInt64` (≈ 292 years), `ExitUnlockAt` is set to approximately year 2316, making these messages permanently unreachable.

The attack is compounded when governance simultaneously sets `CloseOnly = true` on the same tier in the same `MsgUpdateTier` proposal. `CloseOnly` blocks `MsgClearPosition` (the only escape hatch for users who have already triggered exit), trapping them with no recourse:

- `MsgClearPosition` → blocked by `CloseOnly`
- `MsgTierUndelegate` / `MsgWithdrawFromTier` / `MsgExitTierWithDelegation` → blocked by `ExitUnlockAt` far in the future

The corrupted state is `Position.ExitUnlockAt` — set to an unreachable future timestamp, permanently freezing the position's principal (staked tokens held by the per-position delegator account).

### Impact Explanation

Users who have locked tokens into the affected tier and triggered exit (or who trigger exit after the update) cannot recover their principal. Their staked tokens remain delegated under the module's per-position delegator account indefinitely. This is a direct, irreversible loss of liquidity and effective loss of access to user funds. The corrupted value is `Position.ExitUnlockAt` for every position in the affected tier that has a triggered exit.

**Impact: High**

### Likelihood Explanation

Requires a governance proposal (`MsgUpdateTier`) to pass with a majority vote. This is a realistic governance-attack vector: a malicious or negligent proposal that sets `ExitDuration` to `math.MaxInt64` (or any multi-decade value) combined with `CloseOnly = true` would pass if token holders do not scrutinize the numeric field. The absence of an on-chain upper bound means the chain itself provides no safety net — the only protection is off-chain social review of governance proposals.

**Likelihood: Low** (governance majority required), but the protocol provides zero on-chain defense.

### Recommendation

Add an upper bound to `ExitDuration` in `Tier.Validate()`, analogous to the existing cap on `BonusApy`. A reasonable maximum (e.g., 4 years) should be chosen and enforced:

```go
const MaxExitDuration = 4 * 365 * 24 * time.Hour

if t.ExitDuration > MaxExitDuration {
    return fmt.Errorf("exit duration must not exceed %s: got %s", MaxExitDuration, t.ExitDuration)
}
```

This mirrors the fix applied in the referenced external report (adding an upper limit to the fee parameter) and is consistent with the pattern already used for `BonusApy` in the same file.

### Proof of Concept

1. Governance submits and passes `MsgUpdateTier` with:
   ```json
   {
     "id": 1,
     "exit_duration": "9223372036854775807ns",
     "bonus_apy": "0.04",
     "min_lock_amount": "1000",
     "close_only": true
   }
   ```
2. Any user with an existing position calls `MsgTriggerExitFromTier` (allowed even in `CloseOnly` mode).
3. `pos.ExitUnlockAt` is set to `blockTime + 292 years`.
4. User attempts `MsgClearPosition` → rejected: `ErrTierIsCloseOnly`.
5. User attempts `MsgTierUndelegate` → rejected: exit lock duration not elapsed.
6. User attempts `MsgExitTierWithDelegation` → rejected: exit lock duration not elapsed.
7. User's staked tokens are permanently inaccessible.

**Root cause:** `Tier.Validate()` at [1](#0-0)  checks only `ExitDuration > 0` with no upper bound, while `BonusApy` has an explicit cap at line 28–30 for the same reason this guard is needed. [2](#0-1) 

**Trigger path:** `MsgUpdateTier` → `msgServer.UpdateTier` → `tier.Validate()` (passes with any positive duration) → `keeper.SetTier` stores the unbounded value. [3](#0-2) 

**Corrupted state:** `Position.ExitUnlockAt = blockTime.Add(duration)` where `duration` can be up to `math.MaxInt64` nanoseconds. [4](#0-3)
