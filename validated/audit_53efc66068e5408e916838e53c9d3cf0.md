### Title
ML-DSA-65 Gas-Key Host Functions Charge Fees Against Raw Borsh Length Instead of Trie-ID Length — (`File: runtime/near-vm-runner/src/logic/logic.rs`)

### Summary

The three gas-key host functions (`promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`) pass the raw contract-supplied `public_key_len` (the borsh-encoded wire length) directly into `gas_key_transfer_send_fee`, `gas_key_transfer_exec_fee`, and `gas_key_add_key_exec_fee`. The runtime's own `exec_fee` / `total_send_fees` path for the same actions correctly calls `action.public_key.trie_id_len()`. For ML-DSA-65 keys these two lengths diverge by a factor of ~59 (1953 bytes wire vs 33 bytes trie-id), producing a deterministic, protocol-level gas-pricing inconsistency that is triggered by any unprivileged contract once `ProtocolFeature::PostQuantumSignatures` is active.

---

### Finding Description

**Two distinct length semantics exist for `PublicKey::MLDSA65`:**

| Method | ED25519 | SECP256K1 | ML-DSA-65 |
|---|---|---|---|
| `PublicKey::len()` (borsh wire) | 33 | 65 | **1953** |
| `PublicKey::trie_id_len()` (on-trie hash) | 33 | 65 | **33** |

The runtime's action-fee helpers are documented and implemented to accept the *trie-id* length:

```rust
// runtime/runtime/src/config.rs  line 114-116
TransferToGasKey(action) => {
    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.trie_id_len())
        .total()
}
// line 347-349
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
``` [1](#0-0) [2](#0-1) 

The host-function path in `logic.rs` instead passes the raw `public_key_len` argument (the byte count the contract wrote to Wasm memory, i.e. the borsh-encoded full pubkey length):

```rust
// runtime/near-vm-runner/src/logic/logic.rs  lines 3091-3095

    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← raw borsh length, not trie_id_len

``` [3](#0-2) 

The same pattern appears in `promise_batch_action_add_gas_key_with_full_access` (line 3158) and `promise_batch_action_add_gas_key_with_function_call` (line 3229), and is duplicated verbatim in the wasmtime runner: [4](#0-3) [5](#0-4) [6](#0-5) 

The fee functions themselves are correct; the divergence is entirely in what length is passed to them:

### Impact Explanation

For an ML-DSA-65 gas key, `public_key_len` supplied by the contract is 1953 bytes; `trie_id_len()` is 33 bytes. Both `gas_key_transfer_send_fee` and `gas_key_transfer_exec_fee` (and `gas_key_add_key_exec_fee`) multiply `gas_key_byte` by this length. The host-function path therefore charges **~59× more gas** per byte than the transaction path for the same action. Concretely:

- A contract calling `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 key is charged `gas_key_byte × 1953` instead of `gas_key_byte × 33` for both the send and exec components.
- The same action submitted as a `TransferToGasKey` transaction action is charged `gas_key_byte × 33`.
- The two paths produce **different deterministic gas costs for identical protocol actions**, breaking the gas-pricing invariant that host-function and transaction-action fees must agree.
- Any contract that budgets gas for this host function based on the transaction-path cost will run out of gas, making ML-DSA-65 gas keys effectively unusable from contracts.

---

### Likelihood Explanation

The bug is reachable by any unprivileged contract once `ProtocolFeature::PostQuantumSignatures` activates. No privileged role is required. The contract simply passes a borsh-serialized ML-DSA-65 `PublicKey` to the host function — the standard calling convention — and the wrong length flows automatically into the fee calculation. The discrepancy is deterministic and reproducible on every invocation.

---

### Recommendation

Replace `public_key_len as usize` with the decoded key's `trie_id_len()` when calling the exec-fee helpers in all three host functions (both `logic.rs` and `wasmtime_runner/logic.rs`). The decoded key is already available at that point (`public_key.decode()?`):

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
// And:   let decoded_key = public_key.decode()?;

let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
-   public_key_len as usize,
+   decoded_key.trie_id_len(),
);
```

Apply the same fix to `gas_key_add_key_exec_fee` calls in `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`.

---

### Proof of Concept

The divergence is directly observable from the existing test infrastructure. The runtime-path fee for a `TransferToGasKey` action with an ML-DSA-65 key calls `action.public_key.trie_id_len()` = 33: [9](#0-8) 

The host-function path passes `public_key_len as usize`. For an ML-DSA-65 key serialized with `borsh::to_vec`, `public_key_len` = `1 + ML_DSA_65_PUBLIC_KEY_LENGTH` = 1953. Substituting into `gas_key_transfer_exec_fee`:

- `trie_key_len = access_key_key_len(account_id_len, 1953)` — inflated by 1920 bytes vs the correct `access_key_key_len(account_id_len, 33)`
- `per_byte = gas_key_byte × (trie_key_len + estimated_value_len)` — ~59× larger

The same inflation applies to `gas_key_add_key_exec_fee` via `nonce_key_len = access_key_key_len(account_id_len, 1953) + size_of::<NonceIndex>()`. [10](#0-9)

pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }

pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =

        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
```
