### Title
Insufficient Bonus Pool Blocks Undelegation and Exit Operations, Locking User Principal - (File: `x/tieredrewards/keeper/msg_server.go`)

### Summary

In `x/tieredrewards`, the message handlers for `MsgTierUndelegate`, `MsgTierRedelegate`, `MsgAddToTierPosition`, `MsgClearPosition`, and `MsgExitTierWithDelegation` all call `claimRewards()` — which internally calls `processEventsAndClaimBonus()` — **before** performing the actual delegation mutation. If the module's bonus rewards pool (`RewardsPoolName`) has insufficient balance to cover accrued bonus, `processEventsAndClaimBonus()` returns `ErrInsufficientBonusPool` and the entire transaction fails atomically. This means a user with accrued bonus rewards cannot undelegate, redelegate, or exit their position when the pool is empty, permanently locking their principal until an external party replenishes the pool.

### Finding Description

Every exit and mutation path in `x/tieredrewards` mandates reward settlement before the delegation state is modified. In `TierUndelegate`:

```go
pos, _, _, err = ms.claimRewards(ctx, pos)
if err != nil {
    return nil, err   // blocks undelegation entirely
}
```

`claimRewards` calls `processEventsAndClaimBonus`, which at line 230 checks pool sufficiency:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err   // ErrInsufficientBonusPool propagates up
}
```

The same pattern appears in `TierRedelegate` (line 229), `AddToTierPosition` (line 314), and `ClearPosition`. The ADR-006 design document explicitly acknowledges this: *"User-driven paths (ClaimTierRewards, AddToPosition, Undelegate, Redelegate, ClearPosition): fail atomically. User retries after pool is replenished."*

Critically, the user has **no alternative exit path** when all mutation messages are blocked. The `BeforeRedelegationSlashed` hook already demonstrates the correct pattern — it silently forfeits bonus when the pool is insufficient to avoid chain halt — but this safety valve is absent from user-driven paths.

### Impact Explanation

A user's principal (staked tokens held via the position's per-position delegator address) is locked inside the tier module. They cannot:
- Undelegate via `MsgTierUndelegate`
- Exit instantly via `MsgExitTierWithDelegation`
- Redelegate to a different validator via `MsgTierRedelegate`
- Cancel their exit via `MsgClearPosition`

The corrupted invariant is: **a user who has satisfied all exit conditions (exit triggered, exit duration elapsed) must still be able to recover their principal**. When the pool is empty, this invariant is broken. The user's principal remains delegated from the position's delegator address indefinitely, with no self-service recovery path.

### Likelihood Explanation

The bonus pool is funded externally (governance/admin via `MsgFundRewardsPool` or the BeginBlocker top-up). The pool can become empty through normal operation: many users claiming simultaneously, the BeginBlocker top-up failing when the fee collector is empty, or governance choosing not to replenish a sunset tier. A `CloseOnly` tier (governance-sunsetted) is a particularly acute scenario: governance blocks new entries but the pool may not be funded, trapping all existing positions.

### Recommendation

Mirror the `BeforeRedelegationSlashed` pattern for user-driven exit paths: when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool` inside `TierUndelegate` and `ExitTierWithDelegation`, forfeit the bonus silently (log the loss) and proceed with the undelegation/exit rather than reverting the entire transaction. Bonus forfeiture on an empty pool is preferable to permanent principal lockup.

### Proof of Concept

1. User calls `MsgLockTier` with `trigger_exit_immediately = true`, locking tokens for the minimum exit duration.
2. Time passes; bonus accrues. The bonus pool is drained to zero (by other claimants or simply never funded).
3. Exit duration elapses (`block_time >= ExitUnlockAt`). User is entitled to exit.
4. User submits `MsgTierUndelegate`:
   - `validateUndelegatePosition` passes (exit triggered, elapsed, delegated).
   - `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → returns `ErrInsufficientBonusPool`.
   - Transaction reverts. Undelegation does not occur.
5. User submits `MsgExitTierWithDelegation`:
   - Same `claimRewards` call at the top of the handler fails identically.
6. User's principal remains locked in the tier module with no self-service recovery path.
