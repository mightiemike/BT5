### Title
`verify_transaction_inclusion` Skips Coinbase Merkle Proof Safety Check, Enabling 64-Byte Transaction Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function remains a live, publicly callable NEAR entry point. It lacks the coinbase Merkle proof validation that `verify_transaction_inclusion_v2` was specifically introduced to enforce. Any unprivileged NEAR caller can invoke `verify_transaction_inclusion` directly, bypassing the safety check entirely, and receive a `true` result for a fabricated `tx_id` that corresponds to an internal Merkle tree node rather than a real transaction.

---

### Finding Description

The contract exposes two public SPV verification functions:

**`verify_transaction_inclusion`** (the deprecated one): [1](#0-0) 

Its own documentation explicitly warns: [2](#0-1) 

**`verify_transaction_inclusion_v2`** (the safe replacement): [3](#0-2) 

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the [64-byte transaction Merkle proof forgery vulnerability](https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring a coinbase Merkle proof of the same length, which prevents an attacker from presenting an internal Merkle tree node as a valid transaction leaf: [4](#0-3) 

However, `verify_transaction_inclusion` is still decorated with `#[pause]` (callable when the contract is not paused) and is a fully public function. In Rust, `#[deprecated]` is a **compiler lint warning only** — it does not restrict callers at runtime or on-chain. Any unprivileged NEAR account can call `verify_transaction_inclusion` directly, completely bypassing the coinbase proof safety check.

The underlying Merkle proof computation in `merkle-tools` performs no leaf-vs-internal-node distinction: [5](#0-4) 

`compute_root_from_merkle_proof` accepts any 32-byte value as `transaction_hash` and will happily compute a root that matches the block's `merkle_root` if the attacker supplies a crafted internal node hash and a matching proof path.

**Analog to M-03**: Just as `safeTransferFrom` claimed to perform the `onERC721Received` safety check but silently delegated to `_transfer` which skipped it entirely, `verify_transaction_inclusion` claims to verify transaction inclusion but silently omits the coinbase proof validation that is the only guard against 64-byte forgery. The safety check exists in the codebase — it is simply unreachable via the still-public deprecated path.

---

### Impact Explanation

Any downstream consumer — a bridge, a payment oracle, a DeFi protocol — that calls `verify_transaction_inclusion` instead of `verify_transaction_inclusion_v2` can be deceived into accepting a forged transaction proof. An attacker can:

1. Identify a target block with a known Merkle tree structure.
2. Select an internal Merkle node hash `N` at depth `D` (the double-SHA256 of two concatenated 32-byte child hashes — a 64-byte preimage).
3. Treat `N` as the `tx_id` and construct a Merkle proof path from depth `D` up to the root.
4. Call `verify_transaction_inclusion` with this crafted `tx_id`, `tx_index`, and `merkle_proof`.
5. `compute_root_from_merkle_proof(N, crafted_index, crafted_proof)` equals the block's `merkle_root`.
6. The function returns `true` — a forged transaction inclusion proof is accepted.

This enables an attacker to claim that an arbitrary Bitcoin transaction occurred in a real block when it did not, which is the canonical impact of the 64-byte Merkle forgery attack. In a bridge or payment-verification context this directly enables theft of funds.

---

### Likelihood Explanation

**High.** The function is publicly callable by any NEAR account with no access control beyond the global pause flag. The 64-byte Merkle forgery technique is fully documented and the required preimage (two concatenated child hashes) is computable from any publicly observable Bitcoin block. The only barrier is knowing the Merkle tree of a real block, which is public data.

---

### Recommendation

Remove the public entry point for `verify_transaction_inclusion` or restrict its visibility to `pub(crate)` / private so it can only be called internally by `verify_transaction_inclusion_v2`. Alternatively, inline the coinbase proof check directly into `verify_transaction_inclusion` so both code paths are equally safe. The `#[deprecated]` attribute alone provides no on-chain protection.

---

### Proof of Concept

1. Attacker selects a real mainchain block `B` whose hash is stored in the contract, with a known Merkle tree of `2^k` transactions.
2. Attacker picks an internal node `N = double_sha256(child_left || child_right)` at depth `k-1` (one level above the leaves).
3. Attacker constructs `merkle_proof` = the sibling path from depth `k-1` up to the root (length `k-1` hashes), and sets `tx_index` to the position of `N` at that depth.
4. Attacker calls:
   ```
   verify_transaction_inclusion({
       tx_id: N,
       tx_block_blockhash: B,
       tx_index: crafted_index,
       merkle_proof: crafted_proof,
       confirmations: 1,
   })
   ```
5. Inside the function, `compute_root_from_merkle_proof(N, crafted_index, crafted_proof)` traverses the proof path and arrives at the correct `merkle_root` of block `B`.
6. The comparison `== header.block_header.merkle_root` succeeds and the function returns `true`.
7. The attacker has proven inclusion of a non-existent transaction `N` in a real Bitcoin block.

The coinbase check in `verify_transaction_inclusion_v2` would have caught this because the coinbase proof (always at index 0, always at full tree depth `k`) would have a different length than the attacker's `k-1`-length proof, causing the length-equality `require!` to panic — but that check is entirely absent from the still-public `verify_transaction_inclusion`. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
