### Title
`paid_fee_on_l1` Hardcoded to `Fee(1)` in Consensus and RPC L1 Handler Conversion Silently Bypasses Fee Validation - (File: crates/apollo_transaction_converter/src/transaction_converter.rs)

### Summary

The `convert_consensus_l1_handler_to_internal_l1_handler` function unconditionally substitutes `Fee(1)` for the actual `paid_fee_on_l1` value when converting a `ConsensusTransaction::L1Handler` into an executable `L1HandlerTransaction`. The same substitution exists in the RPC re-execution path. Because `ConsensusTransaction::L1Handler` is typed as `transaction::L1HandlerTransaction` — which carries no `paid_fee_on_l1` field — the converter cannot recover the real value and silently injects a constant. The blockifier's only fee guard (`if paid_fee == Fee(0)`) is therefore always satisfied, and the central blob written for proof generation always records `paid_fee_on_l1 = 1` regardless of what was actually paid on L1.

### Finding Description

`ConsensusTransaction::L1Handler` wraps `transaction::L1HandlerTransaction`, which has no `paid_fee_on_l1` field:

```
// transaction::L1HandlerTransaction
{ version, nonce, contract_address, entry_point_selector, calldata }
```

`executable_transaction::L1HandlerTransaction` does carry the field:

```
{ tx: transaction::L1HandlerTransaction, tx_hash, paid_fee_on_l1 }
```

When the converter bridges the two, it cannot recover the real fee and hardcodes a constant:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs  line 473-483
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
```

The blockifier's only guard against an unpaid L1 handler is:

```rust
// crates/blockifier/src/transaction/l1_handler_transaction.rs  line 103-113
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(...));
}
```

`Fee(1) != Fee(0)` is always true, so the guard is permanently bypassed for every L1 handler that arrives through the consensus path.

The identical substitution exists in the RPC re-execution path:

```rust
// crates/apollo_rpc/src/v0_8/api/mod.rs  line 424-428
starknet_api::transaction::Transaction::L1Handler(value) => {
    // todo(yair): This is a temporary solution until we have a better way to get the l1 fee.
    let paid_fee_on_l1 = Fee(1);
    Ok(ExecutableTransactionInput::L1Handler(value, paid_fee_on_l1, false))
}
```

The central blob serializer faithfully copies whatever `paid_fee_on_l1` is present in the executable transaction:

```rust
// crates/apollo_consensus_orchestrator/src/cende/central_objects.rs  line 383-393
impl From<L1HandlerTransaction> for CentralL1HandlerTransaction {
    fn from(tx: L1HandlerTransaction) -> CentralL1HandlerTransaction {
        CentralL1HandlerTransaction {
            ...
            paid_fee_on_l1: tx.paid_fee_on_l1,   // always 1 in consensus path
        }
    }
}
```

So the central blob — the authoritative record used for proof generation and the feeder gateway — always records `paid_fee_on_l1 = 1` for every L1 handler sequenced through consensus, regardless of the actual fee paid on L1.

### Impact Explanation

**Conversion boundary produces a wrong executable payload (High).** The `paid_fee_on_l1` field in the executable `L1HandlerTransaction` is a protocol-level value that is supposed to reflect the fee committed on L1. By substituting `Fee(1)` the converter binds the wrong value into the executable object, satisfying the scope criterion "Transaction conversion … binds the wrong … executable payload."

**RPC re-execution returns an authoritative-looking wrong value (High).** Any call to `starknet_getTransactionReceipt`, `starknet_simulateTransactions`, or trace APIs that re-executes a historical L1 handler will use `paid_fee_on_l1 = 1` instead of the real value. The fee check will always pass, and the receipt will reflect execution that used a fabricated fee field.

**Central blob / proof input carries wrong field value.** The blob written to the cende endpoint records `paid_fee_on_l1 = 1` for every consensus-path L1 handler. If any future OS version or verifier uses this field, the proof input will be inconsistent with the on-chain record.

### Likelihood Explanation

Every L1 handler transaction that passes through the consensus path or is re-executed via the RPC is affected unconditionally. No special attacker capability is required; the substitution is structural and applies to all such transactions in production.

### Recommendation

1. Add `paid_fee_on_l1: Fee` to `ConsensusTransaction::L1Handler` (or wrap it in a dedicated struct) and propagate it through the protobuf `L1Handler` message so validators can recover the real value.
2. Update `convert_consensus_l1_handler_to_internal_l1_handler` to use the field from the consensus message instead of the constant.
3. For the RPC path, read `paid_fee_on_l1` from the stored transaction receipt or the L1 event log rather than substituting `Fee(1)`.
4. Add an assertion or metric that fires whenever `Fee(1)` is used as a fallback, so the workaround is visible in production telemetry until the proper fix lands.

### Proof of Concept

1. An L1 handler transaction is emitted on Ethereum with any `paid_fee_on_l1` value (e.g., the actual fee is `Fee(1_000_000)`).
2. The proposer includes it in a block proposal as `ConsensusTransaction::L1Handler(tx)` where `tx: transaction::L1HandlerTransaction` carries no fee field.
3. Each validator calls `convert_consensus_tx_to_internal_consensus_tx` → `convert_consensus_l1_handler_to_internal_l1_handler`, which constructs `executable_transaction::L1HandlerTransaction { ..., paid_fee_on_l1: Fee(1) }`.
4. The blockifier executes the transaction and evaluates `if paid_fee == Fee(0)` → `Fee(1) != Fee(0)` → check passes unconditionally.
5. The central blob records `paid_fee_on_l1: 0x1` instead of `0xF4240`.
6. An RPC caller invoking `starknet_getTransactionReceipt` for the same transaction triggers `stored_txn_to_executable_txn`, which again injects `Fee(1)`, re-executes with the wrong fee field, and returns a receipt whose fee accounting diverges from the on-chain record.
