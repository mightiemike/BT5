### Title
Undercharged Execution Cost in `op_multiply` Due to Unit Mismatch in Running-Size Accumulator — (File: src/more_ops.rs)

---

### Summary

`op_multiply` tracks the running byte-size of the multiplication accumulator (`l0`) using `limbs_for_int(&total)`, which returns the number of bignum limbs (not bytes). The cost formula `(l0 + l1) * MUL_LINEAR_COST_PER_BYTE + (l0 * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER` is calibrated for byte lengths. For any multi-argument multiplication where intermediate results exceed one limb (> 4 bytes for 32-bit limbs), `l0` is systematically smaller than the true byte size, causing the cost charged for each subsequent multiplication step to be significantly lower than the actual computational work performed.

---

### Finding Description

In `op_multiply`, the first argument's size is correctly obtained in bytes from `int_atom`:

```rust
(total, l0) = int_atom(a, arg, "*")?;  // l0 = byte length
```

After each subsequent multiplication, `l0` is updated:

```rust
l0 = limbs_for_int(&total);  // l0 = bignum limb count, NOT bytes
```

The cost for the next step is then computed as:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

where `l1` is always in bytes (`buf.len()` or `len_for_value(val)`). The constant names `MUL_LINEAR_COST_PER_BYTE` and `MUL_SQUARE_COST_PER_BYTE_DIVIDER` confirm the formula is calibrated for byte-sized inputs. After the first multiplication step, `l0` is in limbs while `l1` remains in bytes — a mixed-unit formula.

The divergence is confirmed by comparing with `op_unknown` cost function 2, which is documented as "computed like operator mul" but correctly uses `l0 += l1` (bytes throughout):

```rust
// op_unknown cost_function == 2
l0 += l1;  // l0 stays in bytes
```

vs `op_multiply`:

```rust
l0 = limbs_for_int(&total);  // l0 switches to limbs
```

For a 256-byte intermediate product (2048 bits), `limbs_for_int` returns approximately 64 (32-bit limbs), while the correct byte length is 256 — a 4× undercount. The quadratic term `(l0 * l1) / 128` is then undercharged by ~16×.

The test cases in `op-tests/test-more-ops.txt` only exercise small numbers (≤ 4 bytes) where one limb equals one byte, so the divergence is invisible in existing tests. The bug only manifests for 3+ argument multiplications where intermediate results exceed one limb.

---

### Impact Explanation

An attacker-controlled CLVM program using `op_multiply` (opcode 18) with three or more large-number arguments will be charged significantly less cost than the actual computation requires. For 256-byte inputs, the undercharge on each step after the first is approximately:

- **Correct cost** (bytes): `(512 + 256) × 6 + (512 × 256) / 128 = 4608 + 1024 = 5632`
- **Actual charged** (limbs): `(64 + 256) × 6 + (64 × 256) / 128 = 1920 + 128 = 2048`

This is a ~2.75× undercharge per step, growing with the number of arguments and the size of intermediate results. A program chaining many large multiplications can consume far more CPU than its declared cost budget permits, breaking the cost-as-resource-limit invariant that protects the Chia mempool and block validation from computational DoS.

Additionally, `op_unknown` cost function 2 (which is supposed to mirror `op_multiply`) charges a different amount for the same inputs, creating a consensus divergence between nodes that evaluate unknown ops vs known multiply ops.

---

### Likelihood Explanation

The entry path is direct: any caller of `run_program` (including the Chia full node processing spend bundles) that evaluates attacker-supplied CLVM bytes containing opcode 18 with three or more large-integer arguments will trigger the undercharge. No special flags or configuration are required. The `ChiaDialect` routes opcode 18 unconditionally to `op_multiply`. The only constraint is that each input must be ≤ 256 bytes and the running product ≤ 1024 limbs (~4096 bytes), which still allows substantial undercharging across many steps.

---

### Recommendation

Replace `limbs_for_int(&total)` with a byte-length computation of the running product, consistent with how `l1` is measured. The simplest fix is to compute the byte length of the result directly:

```rust
// After: total *= operand;
l0 = byte_length_of_number(&total);  // must return bytes, not limbs
```

Alternatively, align the cost formula to use limbs throughout (adjusting the constants accordingly), or verify that `limbs_for_int` is intentionally returning bytes and rename it to eliminate the ambiguity. Also align `op_unknown` cost function 2 to use the same `l0`-update logic as `op_multiply`.

---

### Proof of Concept

A CLVM program `(* A B C)` where `A`, `B`, `C` are each 256-byte integers:

1. Step 1 (`A × B`): `l0 = 256` (bytes), `l1 = 256` (bytes). Cost charged: `885 + (256+256)×6 + (256×256)/128 = 885 + 3072 + 512 = 4469`. `l0` updated to `limbs_for_int(A×B) ≈ 64` (limbs for ~512-byte result).
2. Step 2 (`(A×B) × C`): `l0 = 64` (limbs), `l1 = 256` (bytes). Cost charged: `885 + (64+256)×6 + (64×256)/128 = 885 + 1920 + 128 = 2933`. **Correct cost** (if `l0` were 512 bytes): `885 + (512+256)×6 + (512×256)/128 = 885 + 4608 + 1024 = 6517`.

Undercharge on step 2 alone: **3584 cost units** (~55% of the correct cost). With more arguments or larger inputs, the undercharge compounds. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

pub fn op_multiply(
    a: &mut Allocator,
    mut input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost: Cost = MUL_BASE_COST;
    let mut first_iter: bool = true;
    let mut total: Number = 1.into();
    let mut l0: usize = 0;
    while let Some((arg, rest)) = a.next(input) {
        input = rest;
        if first_iter {
            (total, l0) = int_atom(a, arg, "*")?;
            if l0 > 256 {
                return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
            }
            first_iter = false;
            continue;
        }

        cost += MUL_COST_PER_OP;
        #[cfg(not(feature = "no-fastpath"))]
        match a.node(arg) {
            NodeVisitor::Buffer(buf) => {
                let l1 = buf.len() as u64;
                if l1 > 256 {
                    return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
                }
                cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
                cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
                check_cost(cost, max_cost)?;

                total *= number_from_u8(buf);
            }
            NodeVisitor::U32(val) => {
                let l1 = len_for_value(val) as u64;
                cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
                cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
                check_cost(cost, max_cost)?;

                total *= val;
            }
            NodeVisitor::Pair(_, _) => {
                Err(EvalErr::InvalidOpArg(
                    arg,
                    "Requires Int Argument: *".to_string(),
                ))?;
            }
        }
        #[cfg(feature = "no-fastpath")]
        {
            let (n1, l1) = int_atom(a, arg, "*")?;
            let l1 = l1 as u64;
            if l1 > 256 {
                return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
            }
            cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
            cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
            check_cost(cost, max_cost)?;

            total *= n1;
        }
        l0 = limbs_for_int(&total);
        if l0 > 1024 {
            return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
        }
    }
    let total = a.new_number(total)?;
    Ok(malloc_cost(a, cost, total))
