### Title
`DepositAllowlistExtension` Gates `owner` Instead of `sender`, Allowing Unauthorized Callers to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool." However, its `beforeAddLiquidity` hook silently ignores the `sender` parameter (the actual caller of `pool.addLiquidity`) and instead checks the `owner` parameter (the position beneficiary). Any unprivileged address can bypass the allowlist by calling `addLiquidity` with an allowlisted `owner`, while the `SwapAllowlistExtension` correctly gates the `sender` for swaps. This is a direct analog to the external report's pattern: the wrong party is checked, so the guard is misapplied.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook: [1](#0-0) 

```
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the actual caller (the `sender`); `owner` is the position beneficiary supplied by the caller. Both are forwarded to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives both but discards `sender` (first parameter is unnamed) and only checks `owner`: [3](#0-2) 

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swap initiator) and ignores `recipient`: [4](#0-3) 

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

The admin-facing setter names the allowlisted address `depositor`, confirming the intent is to gate the depositing party, not the position owner: [5](#0-4) 

---

### Impact Explanation

A pool admin deploys `DepositAllowlistExtension` to restrict which addresses may call `addLiquidity` (e.g., KYC compliance, curated LP set, or router-only access). Because the hook checks `owner` rather than `sender`, any unprivileged address can call `pool.addLiquidity(owner = allowlisted_address, ...)` and pass the guard. The unauthorized caller pays the tokens (via the modify-liquidity callback) and creates a position credited to the allowlisted `owner`. The pool admin's access control is entirely bypassed for the calling party. Additionally, the allowlisted `owner` receives an unsolicited position in bins they did not choose, which constitutes a griefing vector: the attacker can force-deposit at an unfavorable bin or price moment, and the victim must actively remove the position to recover their tokens.

---

### Likelihood Explanation

- The bypass requires only a valid allowlisted `owner` address, which is publicly readable from `allowedDepositor` or `allowAllDepositors` mappings.
- No special privilege is needed; any EOA or contract can call `pool.addLiquidity` directly.
- The `MetricOmmPoolLiquidityAdder` exposes `addLiquidityExactShares(pool, owner, ...)` where `owner` is caller-supplied, making the bypass trivially accessible through the standard periphery: [6](#0-5) 

---

### Recommendation

Change `DepositAllowlistExtension.beforeAddLiquidity` to check the `sender` parameter (the actual caller) instead of `owner`, mirroring the pattern used by `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is instead to gate the position `owner` (beneficiary), the contract name, NatSpec, and setter parameter name (`depositor`) must be corrected to avoid misuse by pool admins who expect caller-gating semantics.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; pool admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is not allowlisted.
2. Bob calls `pool.addLiquidity(owner = alice, salt = 0, deltas = {...}, callbackData = ..., extensionData = "")` directly.
3. Pool calls `_beforeAddLiquidity(msg.sender=Bob, owner=alice, ...)`.
4. Extension evaluates `allowedDepositor[pool][alice]` → `true` → hook passes.
5. Pool executes the liquidity addition; Bob's callback pays the tokens; a position is minted under `(alice, 0, binIdx)`.
6. Bob has interacted with the restricted pool and created an unsolicited position for Alice, despite never being allowlisted. [3](#0-2) [7](#0-6) [4](#0-3)
