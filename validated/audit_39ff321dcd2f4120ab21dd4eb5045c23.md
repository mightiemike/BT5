### Title
Unprivileged Caller Can Front-Run `verify_transaction_inclusion` by Calling `run_mainchain_gc` to Prune the Target Block — (`contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` carries no caller-identity restriction. Any unprivileged NEAR account can invoke it with an arbitrary `batch_size` and permanently delete mainchain block headers from on-chain storage. An adversary who observes a pending `verify_transaction_inclusion` cross-contract call in the NEAR transaction pool can front-run it by calling `run_mainchain_gc` first, removing the target block before the verification executes, causing it to panic and return failure to the consumer.

---

### Finding Description

`submit_blocks` is gated by `#[trusted_relayer]`, which enforces that only registered, active relayers may write headers to the chain. [1](#0-0) 

`run_mainchain_gc`, by contrast, carries only `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. That attribute only blocks calls when the contract is **paused**; during normal (unpaused) operation it imposes no restriction on the caller identity whatsoever. [2](#0-1) 

When the mainchain has grown beyond `gc_threshold`, the function removes the oldest blocks from both `mainchain_header_to_height` and `headers_pool` (via `remove_block_header`), up to `batch_size` entries. [3](#0-2) 

`remove_block_header` deletes the entry from both lookup maps: [4](#0-3) 

`verify_transaction_inclusion` then looks up the target block in `mainchain_header_to_height` and panics if it is absent: [5](#0-4) 

It also looks up the header in `headers_pool` and panics if absent: [6](#0-5) 

The attack path is therefore:

1. The chain has grown past `gc_threshold`, making the oldest blocks eligible for GC.
2. A consumer contract (bridge, atomic-swap, lending protocol) submits a cross-contract call to `verify_transaction_inclusion` for a block near the GC boundary. This call is visible in the NEAR transaction pool before it is included in a block.
3. An adversary observes the pending call, reads the `tx_block_blockhash` argument, and immediately calls `run_mainchain_gc(u64::MAX)` — no role, no stake, no deposit required.
4. GC executes first, removing the target block from both maps.
5. The victim's `verify_transaction_inclusion` panics: *"block does not belong to the current main chain"*, returning failure to the consumer.

The CLAUDE.md developer note confirms the design intent and the gap: `run_mainchain_gc` is documented as a public, manually-triggerable function with no mention of caller restriction. [7](#0-6) 

---

### Impact Explanation

Any consumer contract that relies on `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to gate an irreversible action (releasing bridged funds, confirming an atomic swap leg, unlocking collateral) can have that verification forcibly failed by an adversary with zero privileges. The adversary does not need to forge a proof or break cryptography — they only need to call an unrestricted public function before the victim's transaction is included. The corrupted state is the removal of a valid mainchain block from `mainchain_header_to_height` and `headers_pool`, which permanently prevents SPV verification of any transaction in that block for the lifetime of the contract deployment.

---

### Likelihood Explanation

- The attack requires the target block to be beyond the `gc_threshold` window (i.e., the chain must have grown by more than `gc_threshold` blocks since the target block was mined).
- For Bitcoin mainnet with the recommended `gc_threshold = 52704` (~1 year), the window is large but bounded to old transactions.
- For faster chains (Dogecoin ~1-minute blocks, `gc_threshold = 52704` ≈ 36 days; or any deployment with a smaller threshold), the eligible window opens much sooner.
- NEAR's transaction pool is publicly observable, making the pending call trivially detectable.
- The adversary pays only standard NEAR gas — no stake, no role, no deposit.
- Likelihood is **medium-high** for any deployment where consumer contracts verify transactions in blocks older than `gc_threshold`.

---

### Recommendation

Add a caller-identity guard to `run_mainchain_gc` equivalent to the one on `submit_blocks`. The simplest fix is to restrict it to a dedicated role (e.g., a new `GCManager` role, or reuse `Role::DAO`/`Role::RelayerManager`) using the existing `near_plugins` access-control infrastructure, so that only authorized accounts can trigger manual GC. The automatic GC path inside `submit_blocks` is already safe because `submit_blocks` itself is gated by `#[trusted_relayer]`. [2](#0-1) 

---

### Proof of Concept

```
1. Deploy contract with gc_threshold = 100.
2. Trusted relayer submits 201 blocks (heights 0–200).
   → Chain stores 201 blocks; 101 are now eligible for GC (heights 0–100).
3. Bridge contract submits a cross-contract call:
     verify_transaction_inclusion({ tx_block_blockhash: hash_of_block_50, ... })
   This call enters the NEAR transaction pool and is publicly visible.
4. Adversary (any NEAR account, no role) calls:
     run_mainchain_gc({ batch_size: 18446744073709551615 })
   GC removes heights 0–100 from mainchain_height_to_header and headers_pool.
5. Bridge's verify_transaction_inclusion executes next:
     mainchain_header_to_height.get(&hash_of_block_50)
     → None → panic: "block does not belong to the current main chain"
6. Bridge receives a failure receipt and rejects the user's deposit.
```
