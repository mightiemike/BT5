### Title
Systematic Cost Undercharge in `op_multiply` Due to `limbs_for_int` Undercounting Atom Byte Length - (File: src/more_ops.rs)

### Summary

`op_multiply` updates its running size estimate `l0` using `limbs_for_int(&total)`, which counts only magnitude bytes via `v.bits().div_ceil(8)`. For any intermediate product whose most-significant magnitude byte has bit 7 set (e.g., 128, 255, 32768, -129), the CLVM atom representation requires one extra leading byte (a `0x00` sign byte for positives, or an extra byte for negatives), but `limbs_for_int` returns a value that is 1 less than `atom_len`. Every subsequent multiplication step then uses an underestimated `l0`, causing the cost charged to be systematically lower than the true cost of the operation.

### Finding Description

In `src/more_ops.rs`, `op_multiply` tracks the byte-size of the running product in `l0` and uses it to compute the cost of each multiplication step:

```rust
// lines 615-616, 623-624
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

After each step, `l0` is updated at line 649:

```rust
l0 = limbs_for_int(&total);
```

`limbs_for_int` is defined at line 100–102:

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

`BigInt::bits()` returns the number of bits in the **magnitude** (absolute value), ignoring the sign. For a positive number like 255 (`0xff`), `bits()` = 8, so `limbs_for_int` returns 1. But the CLVM atom representation of 255 is `[0x00, 0xff]` — 2 bytes — because the high bit of `0xff` would otherwise be interpreted as a sign bit. The same discrepancy applies to any positive number whose most-significant magnitude byte has bit 7 set (128–255, 32768–65535, etc.) and to negative numbers like -129 whose magnitude also has bit 7 set.

The existing test at lines 104–116 explicitly documents this:

```rust
let expected = if !bytes.is_empty() && bytes[0] == 0 {
    bytes.len() - 1  // limbs_for_int strips the leading zero
} else {
    bytes.len()
};
assert_eq!(limbs_for_int(&bigint), expected);
```

So for `[0x00, 0xff]` (= 255), `limbs_for_int` returns 1, while `atom_len` returns 2. The cost formula uses the wrong (smaller) `l0` for all subsequent multiplication steps.

By contrast, the `op_unknown` cost-function-2 path (lines 223–242) uses `l0 += l1` — a different (overestimating) heuristic — making the two cost models inconsistent with each other.

### Impact Explanation

The cost charged for `op_multiply` is systematically lower than the true computational cost whenever an intermediate product has its high magnitude bit set. The undercharge per affected step is:

```
delta = 1 * l1 * MUL_LINEAR_COST_PER_BYTE + l1 / MUL_SQUARE_COST_PER_BYTE_DIVIDER
      = l1 * 6 + l1 / 128
```

For `l1` near the 256-byte limit, this is up to ~1538 cost units per step. Over a chain of multiplications, the accumulated undercharge allows a program whose true cost exceeds `max_cost` to pass the cost check. This breaks the cost-accounting invariant that is the primary DoS defense of the CLVM interpreter, and could cause consensus divergence between nodes if any node applies a corrected cost model in a future update.

### Likelihood Explanation

The trigger is straightforward: any attacker-controlled CLVM program that uses `*` with arguments crafted so that intermediate products land in the high-bit-set range (e.g., `(* 0x7f 0x02 ...)` produces 254 = `0xfe` after the first step, causing `limbs_for_int` to return 1 instead of 2 for the next step). This is trivially achievable with attacker-controlled CLVM bytes submitted to any node running `run_program`.

### Recommendation

Replace `limbs_for_int(&total)` at line 649 with the actual CLVM atom byte length of the intermediate product. The simplest fix is to allocate the intermediate result and call `a.atom_len(...)`, or to replicate the logic of `atom_len` inline: add 1 to `limbs_for_int` whenever the most-significant magnitude byte has bit 7 set (i.e., when `total.bits() % 8 == 0 && total != 0` for positives, and analogously for negatives).

Alternatively, align `op_multiply`'s cost model with the `op_unknown` cost-function-2 path, which uses `l0 += l1` (though that model overestimates and has its own issues).

### Proof of Concept

**Concrete trigger** — three-argument multiply where the first intermediate product is 254 (`0xfe`):

```
(* 0x7f 0x02 0x02)
```

Step 1 (first arg, no cost): `total = 127`, `l0 = atom_len(0x7f) = 1`  
Step 2 (`0x02`, l1 = 1):
- cost += (1 + 1) × 6 = 12; cost += (1 × 1) / 128 = 0
- `total = 254`; `l0 = limbs_for_int(254) = 1` ← **should be 2** (atom is `[0x00, 0xfe]`)

Step 3 (`0x02`, l1 = 1):
- cost += (1 + 1) × 6 = **12** ← **should be (2 + 1) × 6 = 18**
- cost += (1 × 1) / 128 = 0 ← **should be (2 × 1) / 128 = 0** (same here, but linear term is wrong)
- Undercharge: **6 cost units** for this step alone

For large operands near the 256-byte limit, the per-step undercharge reaches ~1538 cost units, and a crafted program with many such steps can exceed the true cost budget while passing the `check_cost` gate. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
