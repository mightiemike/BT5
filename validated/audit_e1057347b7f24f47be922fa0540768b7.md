### Title
`mintNlp` Is Not Charged Its Respective Proportional Mint Fee While `burnNlp` Is — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`burnNlp` deducts a proportional 0.1% fee from the user's returned quote amount, but the symmetric `mintNlp` operation deducts no equivalent proportional fee from the deposited quote before computing the NLP amount issued. This asymmetric fee treatment allows any user to enter the NLP pool at full oracle price while only paying a fee on exit, inconsistent with the protocol's own fee model.

---

### Finding Description

In `Clearinghouse.sol`, `burnNlp` explicitly computes and deducts a proportional burn fee before crediting the user:

```solidity
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);  // 0.1%, min $1
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
```

The `mintNlp` function performs no analogous deduction. It converts the full `quoteAmount` to NLP at oracle price with no fee applied:

```solidity
int128 quoteAmount = int128(txn.quoteAmount);
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
// No mint fee deducted
spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, nlpAmount);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -quoteAmount);
```

Both `MintNlp` and `BurnNlp` transaction types in `EndpointTx.sol` charge only the flat `HEALTHCHECK_FEE` ($1) at the sequencer layer, which is a gas-deterrent fee, not an economic protocol fee. The proportional economic fee exists only on burn. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

A user minting a large NLP position avoids the proportional fee entirely. For a $1,000,000 mint, the missing fee is $1,000 (0.1%). This fee, when charged on burn, is retained within the NLP pool (it is simply not credited back to the user — the NLP pool's quote balance is not reduced by the burn fee). The protocol therefore under-collects revenue on every NLP mint. The `burnFee` is credited to the NLP pool's liquidity, so the asymmetry also means NLP pool LPs receive less fee income than the protocol design intends, since the mint-side contribution is zero. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

High. Every `MintNlp` transaction submitted by any unprivileged user through the sequencer or slow-mode path benefits from the missing fee. No special privileges are required. The entry path is `Endpoint.submitTransactions` → `EndpointTx.processTransactionImpl` → `Clearinghouse.mintNlp`. The `MintNlp` transaction type is a standard signed user transaction validated with `validateSignedTx`, reachable by any subaccount holder. [7](#0-6) [8](#0-7) 

---

### Recommendation

Apply a proportional mint fee in `mintNlp` symmetric to the burn fee in `burnNlp`. Specifically, deduct a fee from `quoteAmount` before computing `nlpAmount`:

```solidity
int128 mintFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = quoteAmount - mintFee;
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
```

The fee should be retained in the NLP pool (not credited to the user's quote balance), consistent with how `burnFee` is handled in `burnNlp`. [2](#0-1) 

---

### Proof of Concept

1. User holds $1,000,000 USDC in their subaccount quote balance.
2. User submits a `MintNlp` transaction for `quoteAmount = 1_000_000e18`.
3. `Clearinghouse.mintNlp` computes `nlpAmount = 1_000_000e18 / oraclePriceX18` — full amount, no fee.
4. User's quote balance decreases by exactly $1,000,000; NLP balance increases by the full equivalent.
5. User immediately submits `BurnNlp` for the same NLP amount.
6. `Clearinghouse.burnNlp` computes `burnFee = max(1e18, quoteAmount / 1000)` ≈ $1,000.
7. User receives back $999,000 — net cost of the round-trip is $1,000 (burn fee only).
8. Had a symmetric mint fee existed, the round-trip cost would have been ~$2,000.
9. The protocol collects only half the intended fee revenue per mint+burn cycle. [1](#0-0) [9](#0-8)
