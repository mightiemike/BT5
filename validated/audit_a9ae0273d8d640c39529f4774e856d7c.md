### Title
Unscoped `GenericAuthorization` for `MsgTransfer` Allows Grantee to Drain All Grantor NFTs Cross-Chain — (`x/nft-transfer/keeper/packet.go`, `x/nft-transfer/types/msgs.go`)

---

### Summary

The `nft-transfer` module implements no custom `Authorization` type for `MsgTransfer`. A grantee holding a `GenericAuthorization` grant for `MsgTransfer` can submit a `MsgExec` with `msg.Sender = grantor`, pass the `sender.Equals(owner)` ownership check in `createOutgoingPacket`, and IBC-transfer any NFT owned by the grantor to an attacker-controlled receiver address — with no restriction on classID or tokenID.

---

### Finding Description

**`GetSigners` returns `msg.Sender` (the grantor):** [1](#0-0) 

The authz ante handler uses `GetSigners()` to identify the grantor and validates that the grantee holds a grant for `MsgTransfer` from that grantor. With `GenericAuthorization`, this check passes for any parameters — no classID or tokenID scope is enforced.

**`createOutgoingPacket` ownership check uses `msg.Sender` (the grantor):** [2](#0-1) 

Because `sender` is derived from `msg.Sender` (the grantor) and the grantor is also the NFT owner, `sender.Equals(owner)` is trivially true. The check does not verify that the actual transaction signer (the grantee) is authorized for this specific classID/tokenID.

**No custom `Authorization` type exists in the module:**

A search across all `x/nft-transfer/**/*.go` files returns zero matches for `Authorization`, `authz`, or `GenericAuthorization`. Unlike the bank module (`SendAuthorization` with spend limits) or the staking module (`StakeAuthorization` with validator allow/deny lists), the nft-transfer module provides no scoped authorization primitive.



**`Transfer` → `SendTransfer` → `createOutgoingPacket` call chain:** [3](#0-2) [4](#0-3) 

---

### Impact Explanation

A grantee with a `GenericAuthorization` for `MsgTransfer` can:
1. Enumerate all NFTs owned by the grantor.
2. For each NFT, submit `MsgExec{msgs: [MsgTransfer{Sender: grantor, ClassId: X, TokenIds: [Y], Receiver: attacker}]}`.
3. The ownership check passes (grantor owns the NFT), the NFT is escrowed/burned, and the IBC packet delivers it to the attacker's address on the destination chain.

The grantor loses all NFTs. There is no on-chain mechanism to scope the grant to specific classIDs or tokenIDs.

---

### Likelihood Explanation

Realistic. Authz grants for `MsgTransfer` are a natural feature for NFT marketplaces, automation bots, or custodial services. A grantor granting a marketplace contract or operator permission to transfer NFTs on their behalf would not expect the grant to cover all their NFTs unconditionally. The attack requires only that the grantor has issued such a grant — no privileged access, no key compromise.

---

### Recommendation

Implement a custom `NFTTransferAuthorization` type (analogous to `SendAuthorization`) that:
- Accepts an allow-list of `(classID, tokenID)` pairs or at minimum a classID allow-list.
- Implements `cosmos/cosmos-sdk/x/authz.Authorization` with an `Accept` method that validates the specific classID and tokenIDs in the submitted `MsgTransfer` against the scoped allow-list.
- Registers the type in the module codec so it can be used in `MsgGrant`.

Until then, users should be warned that any `GenericAuthorization` for `MsgTransfer` is completely unscoped.

---

### Proof of Concept

```
1. grantor owns NFT (classA, tok1)
2. grantor submits MsgGrant{
       granter: grantor,
       grantee: grantee,
       grant: GenericAuthorization{msg: "/chainmain.nft_transfer.v1.MsgTransfer"}
   }
3. grantee submits MsgExec{
       grantee: grantee,
       msgs: [MsgTransfer{
           Sender:        grantor,
           ClassId:       "classA",
           TokenIds:      ["tok1"],
           Receiver:      attacker_address_on_chain_B,
           SourcePort:    "nft-transfer",
           SourceChannel: "channel-0",
           ...
       }]
   }
4. authz ante: grantee has GenericAuthorization for MsgTransfer from grantor → PASS
5. createOutgoingPacket: sender=grantor, owner=grantor → sender.Equals(owner) → PASS
6. tok1/classA escrowed; IBC packet sent; attacker receives NFT on chain B
``` [5](#0-4)
