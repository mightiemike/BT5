Audit Report

## Title
SwapAllowlistExtension checks router address instead of end-user, enabling complete per-user allowlist bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool, which is `msg.sender` at the pool level — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates a swap, it is the direct caller, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. A pool admin who allowlists the router to enable standard periphery UX inadvertently grants every on-chain user the ability to bypass the per-user swap allowlist entirely.

## Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension is called by the pool via `_callExtensionsInOrder`) and `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`, which passes it verbatim from the pool's own `sender` parameter — i.e., `msg.sender` at the pool level. [1](#0-0) [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with the router as `msg.sender`:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,   // recipient = user-supplied
    params.zeroForOne,
    ...
  );
``` [3](#0-2) 

The same pattern holds for `exactOutputSingle` (L136-137) and `exactInput` (L104-112). [4](#0-3) 

At the pool level, `msg.sender` is the router contract. The pool passes this as `sender` to `_beforeSwap`, which passes it to the extension. The allowlist lookup therefore becomes `allowedSwapper[pool][router]` — the end-user's address is never checked.

`DepositAllowlistExtension` does not share this flaw: it checks `owner` (the position owner explicitly supplied by the caller), which correctly identifies the economic beneficiary even when a router or operator is the `sender`. [5](#0-4) 

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties, institutional LPs, whitelisted protocols) loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. LP assets deposited under the assumption that only curated counterparties could trade are exposed to uncurated counterparties. This constitutes broken core pool functionality (curation invariant violated) and a direct loss of LP assets to uncurated counterparties, matching the "Broken core pool functionality causing loss of funds" and "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" impact categories.

## Likelihood Explanation

The trigger is a valid, semi-trusted operational action: a pool admin allowlisting the router so their curated pool is accessible via the standard periphery. This is the expected step for any pool that wants to support the router UX. The admin has no on-chain signal that doing so opens the allowlist to all users. Any unprivileged user can then exploit it by simply routing through `MetricOmmSimpleRouter` — no special privileges, no front-running, no complex setup required.

## Recommendation

The extension must receive and check the economic actor (the end-user), not the intermediary. Two complementary fixes:

1. **Router-side**: The router should accept an explicit `swapper` parameter and forward it to the pool as a separate field, or the pool's `swap` interface should accept a `payer` address distinct from `msg.sender` (mirroring the `owner`/`sender` split already present in `addLiquidity`).
2. **Extension-side**: Until the interface is extended, document clearly that allowlisting the router is equivalent to `setAllowAllSwappers(pool, true)`. Alternatively, `SwapAllowlistExtension` could check `recipient` instead of `sender` when the intent is to gate who benefits from the swap, though this is an imperfect substitute.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice]  = true   // only alice should trade
  allowedSwapper[pool][router] = true   // admin enables router UX

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, tokenIn: weth, tokenOut: token1, ...})

  router calls:
    pool.swap(recipient=bob, ...)   // msg.sender at pool = router

  pool calls _beforeSwap(sender=router, ...)
  extension checks:
    allowedSwapper[pool][router] == true  → passes

  Result: bob's swap executes against LP assets despite not being allowlisted.
  The per-user curation invariant is broken; any user can trade via the router.
```
