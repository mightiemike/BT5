### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Per-User Swap Gate - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the router is allowlisted for a pool (a natural configuration for any pool that wants to support router-mediated swaps), every user—regardless of their individual allowlist status—can execute swaps on that restricted pool by calling the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs its identity check as follows:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (the extension is called by the pool). `sender` is the first argument, which `MetricOmmPool.swap` sets to `msg.sender` of the pool call:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool receives `msg.sender = router`. It passes `router` as `sender` to the extension. The extension then evaluates `allowedSwapper[pool][router]`—the router's allowlist status—not the actual user's.

The router is a public, permissionless contract. Any pool admin who wants to support router-mediated swaps for their allowlisted users must allowlist the router address. Once the router is allowlisted, the check `allowedSwapper[pool][router] == true` passes for every call that arrives through the router, regardless of who the actual end user is. The per-user allowlist is completely bypassed.

The `recipient` parameter (second argument to `beforeSwap`) is ignored by `SwapAllowlistExtension`, and `extensionData` is not decoded, so there is no secondary path through which the actual user identity reaches the guard.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swapping to a curated set of addresses (e.g., KYC'd market makers, whitelisted counterparties, or protocol-controlled addresses) loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The unauthorized swapper can:

- Execute swaps against pool liquidity at oracle-derived prices, extracting value from LPs who deposited under the assumption that only vetted counterparties could trade.
- Drain one side of the pool if the oracle price diverges from the market, since the allowlist was the only barrier preventing adversarial flow.

This is a direct loss of LP principal and a broken core pool invariant (the access-controlled swap gate).

---

### Likelihood Explanation

The bypass requires the router to be allowlisted for the target pool. This is a natural and expected configuration: any pool admin who wants allowlisted users to be able to use the standard periphery router must allowlist the router. The moment they do, the per-user gate collapses to a single shared entry. The attacker needs no special privileges—only the ability to call a public periphery function.

---

### Recommendation

The extension must check the identity of the economic actor, not the immediate caller of the pool. Two approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated change to the router and extension.

2. **Check `sender` against the allowlist and require direct pool calls for restricted pools**: Document that pools using `SwapAllowlistExtension` must not allowlist shared router contracts, and that users must call `pool.swap()` directly. This is a deployment/operational constraint, not a code fix, and is fragile.

3. **Preferred fix—check `sender` (the actual user) and never allowlist the router**: The extension is already checking `sender`. The root problem is that the router replaces the user's address with its own. A cleaner fix is to have the router forward the original `msg.sender` as part of a signed or verified `extensionData` field, and have the extension verify and decode it. Alternatively, restrict the `SwapAllowlistExtension` to pools where users are expected to call the pool directly.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  - allowedSwapper[pool][router] = true   (admin allowlists the router)
  - allowedSwapper[pool][alice]  = false  (alice is NOT allowlisted)

Attack:
  1. alice calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, ...) with msg.sender = router
  3. Pool calls _beforeSwap(router, recipient, ...)
  4. Extension evaluates allowedSwapper[pool][router] == true → passes
  5. Swap executes; alice receives output tokens.

Expected: revert NotAllowedToSwap (alice is not allowlisted)
Actual:   swap succeeds (router is allowlisted, alice's identity is never checked)
