### Title
Missing Deprecated-Product Check in `_settlePositivePerpPnl()` Causes Incorrect PnL Settlement During Liquidation — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`ClearinghouseLiq._settlePositivePerpPnl()` iterates over every registered perp product and settles positive PnL against the liquidator without checking whether a perp product has been deprecated (`longWeightInitialX18 == 0`). This is the direct Nado analog of the external report: a loop over all items in a registry that updates accounting state while ignoring a disabled/paused flag on individual items.

---

### Finding Description

The protocol supports deprecating a product by setting its `longWeightInitialX18` to `0` via `BaseEngine._addOrUpdateProduct()`. This is already used as a skip-guard in several loops over spot products. For example, `_assertCanLiquidateLiability` explicitly skips deprecated spot products:

```solidity
// ClearinghouseLiq.sol line 237
if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
    continue;
}
``` [1](#0-0) 

The same guard is applied in `_finalizeSubaccount` for spot products: [2](#0-1) 

However, `_settlePositivePerpPnl()` — which is called during spot-liability and spread liquidations — iterates over **all** perp product IDs with no deprecated check:

```solidity
// ClearinghouseLiq.sol lines 420-436
uint32[] memory productIds = perpEngine.getProductIds();
for (uint32 i = 0; i < productIds.length; ++i) {
    uint32 productId = productIds[i];
    int128 positionPnl = perpEngine.getPositionPnl(productId, txn.liquidatee);
    if (positionPnl > 0) {
        _settlePnlAgainstLiquidator(txn, productId, positionPnl, spotEngine, perpEngine);
    }
}
``` [3](#0-2) 

`_settlePnlAgainstLiquidator` directly mutates both the liquidatee's and the liquidator's `vQuoteBalance` and quote balance: [4](#0-3) 

`getPositionPnl` computes PnL using the stored `priceX18` for the product: [5](#0-4) 

A deprecated perp product's price is stale (it is no longer updated by the sequencer), so the computed PnL is unreliable. The settlement is still executed unconditionally.

The call path from an unprivileged liquidator is:

```
Endpoint (sequencer-submitted tx from liquidator)
  → Clearinghouse.liquidateSubaccount
    → ClearinghouseLiq.liquidateSubaccountImpl  [line 598]
      → _settlePositivePerpPnl  [line 637]
``` [6](#0-5) 

---

### Impact Explanation

When a deprecated perp product has a non-zero position for the liquidatee with positive PnL (computed from a stale price), `_settlePositivePerpPnl` will:

1. Reduce the liquidatee's `vQuoteBalance` for the deprecated product.
2. Reduce the liquidator's quote balance by the same amount.
3. Increase the liquidatee's quote balance.

This corrupts the quote balances of both the liquidatee and the liquidator. The liquidator is forced to absorb PnL from a product they did not intend to interact with, and the liquidatee receives an unearned quote credit. The broken invariant is: **only active (non-deprecated) products should participate in PnL settlement during liquidation**.

---

### Likelihood Explanation

The condition requires: (a) at least one perp product has been deprecated (`longWeightInitialX18 = 0`), and (b) the liquidatee holds a position in that deprecated product with positive PnL at the stale price. Both conditions are realistic over the protocol's lifetime as markets are added and retired. Any liquidator triggering a spot-liability or spread liquidation against such a liquidatee will hit this path.

---

### Recommendation

Add the same deprecated-product guard used for spot products in `_settlePositivePerpPnl`:

```diff
uint32[] memory productIds = perpEngine.getProductIds();
for (uint32 i = 0; i < productIds.length; ++i) {
    uint32 productId = productIds[i];
+   if (perpEngine.getRisk(productId).longWeightInitialX18 == 0) {
+       continue; // skip deprecated perp products
+   }
    int128 positionPnl = perpEngine.getPositionPnl(productId, txn.liquidatee);
    if (positionPnl > 0) {
        _settlePnlAgainstLiquidator(txn, productId, positionPnl, spotEngine, perpEngine);
    }
}
```

Apply the same guard consistently in the perp-product loops inside `_finalizeSubaccount` (lines 313–338 and 346–366). [7](#0-6) 

---

### Proof of Concept

1. Admin deprecates perp product `P` by calling `PerpEngine.addOrUpdateProduct(P, ..., riskStore)` with `riskStore.longWeightInitial = 0`. Price for `P` is now stale.
2. Liquidatee holds a position in `P` with `balance.amount > 0` and `balance.vQuoteBalance < 0` such that `priceX18_stale * amount + vQuoteBalance > 0` (positive PnL at stale price).
3. Liquidatee also holds a spot liability, making them under-maintenance.
4. Liquidator submits a `LiquidateSubaccount` transaction with `txn.amount < 0` targeting the spot liability.
5. `liquidateSubaccountImpl` calls `_settlePositivePerpPnl`.
6. The loop reaches product `P`, computes positive PnL using the stale price, and calls `_settlePnlAgainstLiquidator`.
7. Liquidator's quote balance is reduced by the stale-price PnL; liquidatee's quote balance is increased by the same amount — both incorrectly.
