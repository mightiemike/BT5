### Title
Accrued Bonus Rewards Permanently Lost When Pool Is Insufficient During Redelegation Slash - (`x/tieredrewards/keeper/slash.go`)

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` fails with `ErrInsufficientBonusPool` during the `BeforeRedelegationSlashed` hook, the error is deliberately swallowed to avoid a chain halt. However, `processEventsAndClaimBonus` has already advanced the position's bonus checkpoints in memory and decremented event reference counts on-chain before returning the error. The position is then persisted with those advanced checkpoints, making the accrued bonus permanently unclaimable — the rewards are silently forfeited with no recovery path.

### Finding Description

`processEventsAndClaimBonus` modifies the in-memory `pos` and on-chain event state **before** it checks pool sufficiency: [1](#0-0) 

The loop at lines 172–199 walks all pending validator events, updates `pos.LastEventSeq` in memory, and **decrements event reference counts on-chain** (potentially garbage-collecting them). After the loop, `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` advance the remaining checkpoints in memory. Only then does the function check pool balance: [2](#0-1) 

When `ErrInsufficientBonusPool` is returned, the in-memory `pos` already carries fully-advanced checkpoints (`LastEventSeq`, `LastBonusAccrual`, `LastKnownBonded`) and the on-chain events have been decremented.

Back in `slashRedelegationPosition`, this error is caught and swallowed: [3](#0-2) 

Execution then continues with the modified `pos` and the position is persisted via `setPositionWithState`: [4](#0-3) 

The saved position now has checkpoints that have advanced past the accrual period for which bonus was earned. The bonus pool still holds the funds (they were never transferred), but the position can never claim them again because:
1. `LastEventSeq` has moved past the relevant events (which may have been GC'd)
2. `LastBonusAccrual` has advanced to `blockTime`, closing the accrual window
3. There is no mechanism to re-open or replay a forfeited accrual window

The ADR explicitly documents this as intentional for chain-halt avoidance, but the consequence is a permanent, silent loss of user funds: [5](#0-4) 

### Impact Explanation

A position owner who has accrued bonus rewards during a redelegation period permanently loses those rewards if the bonus pool is insufficient at the moment a validator slash fires. The pool retains the funds but the position's checkpoints have moved past the claimable window — the accrued value is tracked (in the pool) but permanently inaccessible to the user. This is a direct loss of earned rewards with no recovery path.

### Likelihood Explanation

The scenario requires two concurrent conditions:
1. A position is in the redelegation period (21-day unbonding window after `MsgTierRedelegate`)
2. The bonus pool is insufficient at the moment the source validator is slashed

Both are realistic protocol events. The bonus pool can be depleted by normal BeginBlocker top-ups and concurrent user claims. Validator slashing is a standard protocol mechanism. The combination is not exotic — any validator slash during a period of pool depletion triggers this for all redelegating positions on that validator.

### Recommendation

In `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is encountered, the position's bonus checkpoints must **not** be advanced. The fix requires either:

1. Saving the pre-call checkpoint state and restoring it on `ErrInsufficientBonusPool` before persisting the position, so the user can retry the claim later once the pool is replenished; or
2. Separating checkpoint advancement from payment inside `processEventsAndClaimBonus` so that checkpoints only advance after a successful transfer.

The current design advances checkpoints and decrements event ref counts before the pool check, which makes rollback on error impossible without explicit state restoration.

### Proof of Concept

1. User calls `MsgTierRedelegate` — position enters 21-day redelegation period; `RedelegationMappings[unbondingId] = positionId` is set.
2. Bonus accrues on the destination validator over several days.
3. The bonus pool is drained (e.g., by other users claiming or BeginBlocker top-ups).
4. The source validator is slashed — `BeforeRedelegationSlashed` fires → `slashRedelegationPosition`.
5. `processEventsAndClaimBonus` walks events, advances `pos.LastEventSeq` and `pos.LastBonusAccrual` in memory, decrements event ref counts on-chain, then returns `ErrInsufficientBonusPool`.
6. The error is swallowed at `slash.go:56`; execution continues with the modified `pos`.
7. `setPositionWithState` persists the position with advanced checkpoints.
8. Pool is later replenished, but the user calls `MsgClaimTierRewards` and receives zero bonus — `processEventsAndClaimBonus` finds no events since `LastEventSeq` and computes zero for the segment starting at `LastBonusAccrual` (which is now `blockTime` of the slash). The accrued bonus is permanently lost.
