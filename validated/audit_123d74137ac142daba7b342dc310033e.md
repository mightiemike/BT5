### Title
Sequencer Downtime Causes Instant Liquidations Upon Resumption via Unbounded `dt` in SpotTick/PerpTick Processing — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

When the Nado sequencer resumes after a prolonged downtime, it submits `SpotTick` and `PerpTick` transactions carrying the full accumulated time delta since the last tick. Because there is no cap on `dt` and no post-resumption grace period, all interest and funding-rate accrual is applied atomically in a single block. Subaccounts that were healthy before the downtime can become instantly undercollateralized, and liquidators can immediately liquidate them before users have any opportunity to repay or add collateral.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, `SpotTick` and `PerpTick` are processed as follows:

```solidity
// SpotTick
uint128 dt = t.spotTime == 0 ? 0 : txn.time - t.spotTime;
spotEngine.updateStates(dt);
t.spotTime = txn.time;
times = t;

// PerpTick
uint128 dt = t.perpTime == 0 ? 0 : txn.time - t.perpTime;
perpEngine.updateStates(dt, txn.avgPriceDiffs);
t.perpTime = txn.time;
times = t;
``` [1](#0-0) 

The `txn.time` value is sequencer-supplied and entirely unconstrained. `dt` is the raw difference between the new time and the last recorded time — there is no maximum, no smoothing, and no grace-period flag.

All user-facing transactions (deposits, withdrawals, position adjustments) flow exclusively through `submitTransactionsChecked`, which enforces `msg.sender == sequencer`. [2](#0-1) 

This means that while the sequencer is offline, users have no on-chain path to improve their health. The only censorship-resistance escape hatch is the slow-mode queue, but:

1. Deposits via `depositCollateral()` are queued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (3 days).
2. The slow-mode queue is FIFO; a user cannot skip ahead to process their own deposit.
3. Even after 3 days, `executeSlowModeTransaction()` processes the *next* queued entry, not a specific user's entry. [3](#0-2) [4](#0-3) 

When the sequencer resumes, it naturally submits `SpotTick`/`PerpTick` with the current wall-clock time. The resulting `dt` equals the entire downtime duration. `spotEngine.updateStates(dt)` and `perpEngine.updateStates(dt, ...)` apply the full accumulated interest and funding rates in a single atomic call. Subaccounts that were above the maintenance threshold before the downtime can drop below it instantly.

Liquidation is then immediately available to any unprivileged liquidator via `LiquidateSubaccount`:

```solidity
require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
``` [5](#0-4) 

There is no post-resumption grace period, no check for an abnormally large `dt`, and no mechanism to delay liquidations after a large time jump.

---

### Impact Explanation

**High.** A user with a spot borrow position or a perp position that was healthy before the downtime can be liquidated at a penalty price the moment the sequencer resumes, with no opportunity to repay debt or add collateral. The liquidation penalty (`LIQUIDATION_FEE_FRACTION = 50%` of the spread) represents a direct, irreversible asset loss. [6](#0-5) 

---

### Likelihood Explanation

**Low.** The scenario requires a prolonged sequencer downtime (days to weeks) sufficient for interest or funding-rate accrual to push previously healthy subaccounts below the maintenance threshold. This mirrors the original report's likelihood assessment: the precondition is an extended operational pause, not a routine event, but it is a realistic failure mode for any sequencer-based system.

---

### Recommendation

1. **Cap `dt` per tick.** Enforce a maximum `dt` (e.g., `SECONDS_PER_DAY`) per `SpotTick`/`PerpTick` call. If more time has elapsed, require multiple sequential ticks to catch up, giving users time to react between each one.
2. **Post-resumption grace period.** After a `dt` exceeding a threshold (e.g., 1 day), set a protocol-level flag that delays liquidation eligibility for a configurable grace window, allowing users to repay or add collateral.
3. **Slow-mode deposit priority.** Allow users with pending slow-mode deposits to have those deposits credited before liquidation checks are applied to their subaccount.

---

### Proof of Concept

1. User holds a spot borrow position; maintenance health = +5 (just above threshold).
2. Sequencer goes offline for 30 days. User cannot repay or add collateral.
3. Sequencer resumes; submits `SpotTick` with `txn.time = t.spotTime + 30 days`.
4. `dt = 30 * 86400` seconds is passed to `spotEngine.updateStates(dt)`.
5. Thirty days of interest accrues atomically; user's maintenance health drops to −20.
6. `isUnderMaintenance(user)` returns `true`.
7. Liquidator calls `LiquidateSubaccount` in the same block; user is liquidated at a penalty price with no recourse. [7](#0-6) [8](#0-7)

    function submitTransactionsChecked(
        uint64 idx,
        bytes[] calldata transactions,
        bytes32 e,
        bytes32 s,
        uint8 signerBitmask
    ) external {
        validateSubmissionIdx(idx);
        require(msg.sender == sequencer);
        // TODO: if one of these transactions fails this means the sequencer is in an error state
        // we should probably record this, and engage some sort of recovery mode

        bytes32 digest = keccak256(abi.encode(idx));
        for (uint256 i = 0; i < transactions.length; ++i) {
            digest = keccak256(abi.encodePacked(digest, transactions[i]));
        }
        verifier.requireValidSignature(digest, e, s, signerBitmask);

        for (uint256 i = 0; i < transactions.length; i++) {
            bytes calldata transaction = transactions[i];
            processTransaction(transaction);
            nSubmissions += 1;
        }
    }
