### Title
Unprivileged `run_mainchain_gc` Caller Can Permanently Freeze Bridge Proofs by Pruning the Retention-Boundary Block â€” (`contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a fully public NEAR contract method with no caller-identity guard. Any account can invoke it at any time. When the stored chain length exceeds `gc_threshold`, the function removes the oldest blocks from all three storage maps (`mainchain_height_to_header`, `mainchain_header_to_height`, `headers_pool`). Because `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` require the target block to be present in both `mainchain_header_to_height` and `headers_pool`, pruning a block that a downstream bridge is about to prove against makes that proof permanently and irrecoverably impossible.

---

### Finding Description

**Access control on `run_mainchain_gc`** [1](#0-0) 

The only gate is `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. When the contract is not paused â€” the normal production state â€” any NEAR account can call this method with an arbitrary `batch_size`.

**What GC deletes** [2](#0-1) 

For each height in `[start_removal_height, end_removal_height)`, `remove_block_header` is called: [3](#0-2) 

This removes the block from **both** `mainchain_header_to_height` and `headers_pool`. There is no soft-delete, no tombstone, and no way to re-insert a pruned block.

**Why a pruned block makes a proof permanently unprovable**

`verify_transaction_inclusion` (and v2 which delegates to it) does: [4](#0-3) 

If the block was pruned, `mainchain_header_to_height.get` returns `None` â†’ unconditional panic `"block does not belong to the current main chain"`. The block can never be re-added because its `ExtendedHeader` is also gone from `headers_pool`.

**Why `reorg_chain` also breaks when the fork point is pruned** [5](#0-4) 

The walk-back loop terminates only when it finds a block in `mainchain_header_to_height`. If the fork-point block was pruned, the loop walks past it and calls `headers_pool.get(&prev_block_hash)` on a block that no longer exists â†’ unconditional panic `"previous fork block should be there"`. Because NEAR transactions are atomic, the panic reverts all state, but the fork-point block remains permanently absent, so every future attempt to submit the winning fork also panics.

---

### Impact Explanation

A downstream bridge contract that holds user deposits and calls `verify_transaction_inclusion` to release them will be permanently unable to verify any deposit whose inclusion block was pruned. The funds locked behind that proof are frozen with no recovery path: the block cannot be re-submitted (its parent chain is gone), and the proof function always panics.

This matches the **Theft or permanent freezing of funds** Critical impact category.

---

### Likelihood Explanation

The preconditions are:

1. The chain has grown at least one block beyond `gc_threshold` (so `total_amount_to_remove â‰Ą 1`). With the recommended `gc_threshold = 52704` this happens continuously in production.
2. A bridge proof is pending for a block at or near `initial_height`. This is realistic whenever a user delays claiming a deposit, a relayer has downtime, or a dispute-resolution window is long.
3. The attacker calls `run_mainchain_gc` before the proof is submitted. Since the function is public and costs only gas, the attacker can front-run the proof submission by monitoring the mempool or simply calling GC repeatedly.

No privileged role, no mining, and no social engineering are required.

---

### Recommendation

Restrict `run_mainchain_gc` to trusted callers only (e.g., add a `trusted_relayer` or dedicated `Role::GCManager` guard), mirroring the access control already applied to `submit_blocks`: [6](#0-5) 

Alternatively, remove the standalone public entrypoint entirely and keep GC as an internal step called only from within `submit_blocks`, where it is already invoked: [7](#0-6) 

---

### Proof of Concept

Exact interleaving that triggers the freeze (workspace test, `skip_pow_verification = true`, `gc_threshold = 3`):

```
// State: mainchain has 4 blocks (heights 0-3), gc_threshold = 3
// â†’ amount_of_headers_we_store = 4 > 3, so total_amount_to_remove = 1

// Step 1 (bridge): bridge has a pending proof for block at height 0
//   tx_block_blockhash = hash_of_block_0

// Step 2 (attacker): call run_mainchain_gc(1) from any account
//   â†’ removes height 0 from mainchain_height_to_header,
//      mainchain_header_to_height, and headers_pool
//   â†’ mainchain_initial_blockhash advances to height 1

// Step 3 (bridge): submit verify_transaction_inclusion with tx_block_blockhash = hash_of_block_0
//   â†’ mainchain_header_to_height.get(&hash_of_block_0) returns None
//   â†’ panics: "block does not belong to the current main chain"
//   â†’ proof is permanently unprovable; funds are frozen
```

