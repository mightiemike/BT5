Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, allowing any address to bypass per-pool swap allowlists via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives the `sender` argument from the pool, which is the pool's own `msg.sender` — the immediate caller of `pool.swap`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. A pool admin who allowlists the router (the necessary step to let curated users interact via the standard periphery) inadvertently grants every on-chain address the ability to swap in the restricted pool.

## Finding Description
`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- pool's msg.sender, not the originating user
  recipient,
  ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks this value against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the pool's `msg.sender`. The originating user's address is stored only in transient callback context for payment settlement and is never forwarded to the pool or any extension:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(
  params.recipient,
  ...
);
```

The allowlist check therefore evaluates `allowedSwapper[pool][router]` — the router address — rather than the originating user. Once the router is allowlisted (required for any curated user to use the standard periphery), the check passes for every caller. The same mismatch applies to `exactInput` (intermediate hops use `address(this)` as payer), `exactOutputSingle`, and `exactOutput`.

`DepositAllowlistExtension` does not share this flaw because it checks the `owner` argument (the LP position beneficiary), which is explicitly passed through by the caller and represents the economically relevant actor.

## Impact Explanation
The `SwapAllowlistExtension` is the sole on-chain mechanism for restricting swap access to a curated set of addresses (e.g., KYC-verified users, institutional counterparties). Once the router is allowlisted, the gate is fully open to every address on-chain. Unauthorized traders can execute swaps at the pool's oracle-anchored prices, causing adverse selection against LPs and extracting value from a pool designed to serve only a restricted set of counterparties. This constitutes a direct loss of LP principal and a complete break of the pool's intended access-control invariant — a High severity impact under Sherlock thresholds.

## Likelihood Explanation
The scenario requires the pool admin to allowlist the router. This is the natural and expected configuration for any curated pool whose allowlisted users are expected to interact through the standard periphery (`MetricOmmSimpleRouter` is the primary user-facing swap interface). A pool admin who does not allowlist the router effectively makes the router unusable for their pool — an undocumented and non-obvious restriction. The probability that at least one production pool admin takes this step is high, making exploitation straightforward and repeatable by any on-chain address.

## Recommendation
The extension must check the economically relevant actor — the address that initiated the transaction — not the immediate `msg.sender` of `pool.swap`. Two viable approaches:

1. **Router-forwarded identity**: Add an optional `swapper` field to `extensionData` that the router encodes as `msg.sender` before forwarding. `SwapAllowlistExtension.beforeSwap` decodes and checks this field when present, falling back to `sender` for direct pool calls.

2. **Pool-level sender override**: Extend the pool's `swap` interface with an explicit `swapper` parameter (separate from `recipient`) that the router populates with `msg.sender`. Extensions receive this value instead of the pool's `msg.sender`.

Either fix must be applied consistently across `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput`.

## Proof of Concept
```
Setup
─────
1. Deploy MetricOmmPool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is the curated user
3. Pool admin calls setAllowedToSwap(pool, router, true)  // needed so alice can use the router

Attack
──────
4. bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool:      <restricted pool>,
           recipient: bob,
           ...
       })

5. Router calls pool.swap(bob, ...) — msg.sender inside pool = router.

6. Pool calls _beforeSwap(sender=router, ...).

7. SwapAllowlistExtension checks allowedSwapper[pool][router] → true (step 3).

8. Swap executes. bob receives tokens from the restricted pool.

Result: bob, who was never allowlisted, successfully swaps in a pool
        intended to be restricted to alice only.
```
