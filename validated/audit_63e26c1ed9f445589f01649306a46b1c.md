### Title
Undercharged Execution in `op_keccak256` — Per-Iteration Cost Guard Uses Stale Byte Count, Omitting Current Argument's Bytes - (File: src/keccak256_ops.rs)

---

### Summary

`op_keccak256` performs its intermediate cost guard using `byte_count` accumulated from *previous* loop iterations, not including the current argument's bytes. This is the exact same class of bug as the external report: a threshold check uses a partial/stale value instead of the cumulative total. The result is that the guard passes when it should fail, allowing the interpreter to hash one argument's worth of data beyond what the declared cost limit permits.

---

### Finding Description

In `src/keccak256_ops.rs`, `op_keccak256` accumulates argument bytes in `byte_count` and checks cost inside the argument loop. The check fires *before* the current argument's length is added to `byte_count`:

```rust
// src/keccak256_ops.rs lines 26-33
cost += KECCAK256_COST_PER_ARG;
check_cost(
    cost + byte_count as Cost * KECCAK256_COST_PER_BYTE,  // byte_count is STALE
    max_cost,
)?;
let blob = atom(a, arg, "keccak256")?;
byte_count += blob.as_ref().len();   // updated AFTER the guard
hasher.update(blob);
``` [1](#0-0) 

For the first argument, `byte_count = 0`, so the guard reduces to `check_cost(cost + 0, max_cost)`. If `max_cost` equals exactly `KECCAK256_BASE_COST + KECCAK256_COST_PER_ARG = 50 + 160 = 210`, the guard passes unconditionally regardless of how large the argument blob is. The function then hashes the full blob, computes the correct final cost (`210 + blob_len * 2`), and returns it. The caller's loop-level check then rejects the result — but the hashing work has already been performed.

**Contrast with `op_sha256`**, which correctly updates cost *before* the guard:

```rust
// src/more_ops.rs lines 403-405
let blob = atom(a, arg, "sha256")?;
cost += blob.as_ref().len() as Cost * SHA256_COST_PER_BYTE;
check_cost(cost, max_cost)?;
``` [2](#0-1) 

`op_sha256` reads the blob first, adds its byte cost to `cost`, then checks. `op_keccak256` inverts this order, leaving the current argument's byte cost out of the guard entirely.

The same correct pattern appears in `binop_reduction` (used by `logand`/`logior`/`logxor`) and in `op_unknown` cost-function-1, both of which update the byte accumulator *before* calling `check_cost`: [3](#0-2) [4](#0-3) 

`op_keccak256` is the sole outlier.

---

### Impact Explanation

An attacker-controlled CLVM program can call `keccak256` with a single large atom argument while engineering the remaining cost budget to be exactly `KECCAK256_BASE_COST + KECCAK256_COST_PER_ARG = 210`. The stale-byte-count guard passes (`210 ≤ 210`), the interpreter hashes the full blob (potentially megabytes), and only then does the caller's loop-level check (`cost > effective_max_cost`) reject the result. The hashing computation is unrecoverable — it has already been performed. This is a **DoS via undercharged execution**: the attacker causes O(N) hashing work while paying O(1) declared cost units.

The final `Reduction` cost returned is always arithmetically correct; the bug is exclusively in the early-termination guard, which is the only mechanism preventing the operator from doing unbounded work before the caller can intervene. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

`keccak256` is reachable from any attacker-supplied CLVM byte sequence under softfork extension 1 or with `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`. No special privileges are required. The attacker only needs to craft a program that (a) burns cost down to exactly 210 remaining units before invoking `keccak256`, and (b) passes a large atom as the first argument. Both are trivially achievable with standard CLVM primitives. The attack is repeatable across any full node that evaluates the transaction. [7](#0-6) 

---

### Recommendation

Read the blob and update `byte_count` *before* calling `check_cost`, mirroring the pattern in `op_sha256`:

```rust
// Fixed op_keccak256 inner loop body
cost += KECCAK256_COST_PER_ARG;
let blob = atom(a, arg, "keccak256")?;
byte_count += blob.as_ref().len();
check_cost(
    cost + byte_count as Cost * KECCAK256_COST_PER_BYTE,
    max_cost,
)?;
hasher.update(blob);
```

This ensures the guard always reflects the cumulative byte cost including the current argument, consistent with every other multi-argument operator in the codebase.

### Proof of Concept

```
; Attacker-controlled CLVM program (pseudo-notation)
; Assume total max_cost = 1000, program burns 790 cost first,
; leaving remaining budget = 210 when keccak256 is reached.

(keccak256 <1-MB-atom>)
; Inside op_keccak256:
;   cost = 50 + 160 = 210
;   check_cost(210 + 0*2, 210) => 210 <= 210 => PASSES
;   blob = 1,048,576 bytes
;   byte_count = 1,048,576
;   hasher.update(1MB)   <-- full hashing work performed
; After loop:
;   cost = 210 + 1,048,576*2 = 2,097,362
;   returns Reduction(2,097,394, hash_node)
; Caller: cost += 2,097,394 => cost > 1000 => CostExceeded
; But 1 MB was hashed for free.
```

The attacker pays a transaction fee calibrated to `max_cost = 1000` but forces every validating full node to perform ~1 MB of Keccak-256 hashing per transaction submission. [8](#0-7)

pub fn op_keccak256(
    a: &mut Allocator,
    mut input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = KECCAK256_BASE_COST;

    let mut byte_count: usize = 0;
    let mut hasher = Keccak256::new();
    while let Some((arg, rest)) = a.next(input) {
        input = rest;
        cost += KECCAK256_COST_PER_ARG;
        check_cost(
            cost + byte_count as Cost * KECCAK256_COST_PER_BYTE,
            max_cost,
        )?;
        let blob = atom(a, arg, "keccak256")?;
        byte_count += blob.as_ref().len();
        hasher.update(blob);
    }
    cost += byte_count as Cost * KECCAK256_COST_PER_BYTE;
    new_atom_and_cost(a, cost, &hasher.finalize())
