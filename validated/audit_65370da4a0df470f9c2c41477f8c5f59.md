### Title
Division by Zero in `PerpEngine.socializeSubaccount` Blocks Liquidation Finalization When `openInterest == 0` — (File: `core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` without a zero-guard. When called during liquidation finalization via `ClearinghouseLiq._finalizeSubaccount`, if all positions in a product are closed (`openInterest == 0`) but the liquidatee still carries negative `vQuoteBalance`, the division reverts with `"DBZ"`, permanently blocking finalization and leaving bad debt unresolved in the protocol.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, the socialization logic computes a per-share funding adjustment:

```solidity
// actually socialize if still not enough
if (balance.vQuoteBalance < 0) {
    // socialize across all other participants
    int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
    ...
}
``` [1](#0-0) 

`MathSD21x18.div` enforces a hard revert when the denominator is zero:

```solidity
function div(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        require(y != 0, ERR_DIV_BY_ZERO);
        ...
    }
}
``` [2](#0-1) 

`state.openInterest` tracks the total open interest for a product. It reaches zero when all participants have closed their positions. However, `vQuoteBalance` represents realized PnL that persists independently of position size. A subaccount can have `balance.amount == 0` (position closed) while retaining `balance.vQuoteBalance < 0` (realized loss not yet settled). If the liquidatee is in this state and no other participant holds an open position in that product, `openInterest == 0` while `vQuoteBalance < 0` — the exact condition that triggers the division by zero.

This function is called unconditionally from `ClearinghouseLiq._finalizeSubaccount`:

```solidity
v.insurance = perpEngine.socializeSubaccount(
    txn.liquidatee,
    v.insurance
);
``` [3](#0-2) 

`_finalizeSubaccount` is invoked from `liquidateSubaccountImpl` when `txn.productId == type(uint32).max`: [4](#0-3) 

The `liquidateSubaccountImpl` call is reached via `delegatecall` from `Clearinghouse.liquidateSubaccount`, which is an `onlyEndpoint` function reachable by the sequencer processing any liquidator's signed transaction: [5](#0-4) 

---

### Impact Explanation

When the revert occurs, the entire finalization transaction is rolled back. The liquidatee's negative `vQuoteBalance` cannot be socialized across remaining participants, and the subaccount cannot be finalized. This leaves unresolved bad debt permanently frozen in the protocol's accounting state — `balance.vQuoteBalance` remains negative, `state.availableSettle` is not adjusted, and the subaccount is never cleared. This is a solvency/accounting corruption: bad debt that the protocol's socialization mechanism is designed to distribute is instead irrecoverably stuck.

---

### Likelihood Explanation

The condition (`openInterest == 0` with `vQuoteBalance < 0`) arises naturally in low-liquidity markets or during product wind-down when the liquidatee is the last participant. A sophisticated liquidatee can engineer this defensively: open a position in a thin market, accumulate a realized loss (close the position, keeping negative `vQuoteBalance`), wait for all other participants to exit, then become unhealthy. Any subsequent finalization attempt by a liquidator will revert, permanently shielding the subaccount from finalization and leaving bad debt in the system.

---

### Recommendation

Add an explicit zero-guard before the division in `PerpEngine.socializeSubaccount`. If `openInterest == 0`, there are no other participants to socialize the loss against; the remaining bad debt should be absorbed by the insurance fund or written off:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest > 0) {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
    }
    // write off remaining bad debt regardless
    balance.vQuoteBalance = 0;
}
``` [6](#0-5) 

---

### Proof of Concept

1. Product `P` has `state.openInterest == 0` (all participants have closed positions).
2. Liquidatee holds `balance.amount == 0`, `balance.vQuoteBalance == -1000e18` for product `P` (realized loss from a previously closed position).
3. Protocol insurance is insufficient to fully cover the loss (`insurance < 1000e18`), so after the insurance cover step, `balance.vQuoteBalance` remains negative.
4. Liquidatee's maintenance health drops below zero (e.g., due to other positions).
5. Liquidator submits `LiquidateSubaccount` with `productId == type(uint32).max` to finalize the subaccount.
6. Execution path: `Clearinghouse.liquidateSubaccount` → `delegatecall` → `liquidateSubaccountImpl` → `_finalizeSubaccount` → `perpEngine.socializeSubaccount`.
7. Inside `socializeSubaccount`, `balance.vQuoteBalance < 0` is true; after insurance deduction it remains negative; the code reaches `(-X).div(0)`.
8. `MathSD21x18.div` reverts with `"DBZ"`.
9. The entire transaction reverts. No subsequent finalization attempt can succeed as long as `openInterest == 0` for product `P`.
10. Bad debt is permanently frozen; the subaccount is never finalized. [7](#0-6) [2](#0-1)

                int128 insuranceCover = MathHelper.min(
                    insurance,
