Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address as `sender` instead of the originating user, making per-user allowlist enforcement incorrect for all router-mediated swaps â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap()` passes `msg.sender` (the router) as `sender` to `_beforeSwap`, which forwards it unchanged to `SwapAllowlistExtension.beforeSwap`. The extension then checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`. This produces a structural misbinding: if the router is not allowlisted, even legitimately allowlisted users cannot swap through the router; if the router is allowlisted, every user â€” including non-allowlisted ones â€” can bypass the per-user gate.

## Finding Description

**Root cause â€” pool passes `msg.sender` (the router) as `sender`:**

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the first argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged into the extension call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()` â€” the router, not the user: [3](#0-2) 

**Router calls `pool.swap()` directly, making itself `msg.sender` to the pool:**

`exactInputSingle` calls `pool.swap()` with no forwarding of the original caller: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

**Two broken outcomes, both caused by the same misbinding:**

1. **Router not allowlisted:** `allowedSwapper[pool][router]` is `false`, so every router-mediated swap reverts â€” including swaps by users who are individually allowlisted. Allowlisted users are locked out of the standard periphery entry point.
2. **Router allowlisted:** `allowedSwapper[pool][router]` is `true`, so every router-mediated swap succeeds â€” including swaps by users who are not individually allowlisted. The per-user gate is fully bypassed.

Neither outcome matches the intended per-user access control. The extension's `allowedSwapper` mapping is keyed on the intermediary address, not the economic actor.

## Impact Explanation
The `SwapAllowlistExtension` is a core pool functionality component for curated/restricted pools. The misbinding renders it non-functional for all router-mediated swaps: either allowlisted users are incorrectly blocked (broken swap flow) or non-allowlisted users gain unauthorized access (allowlist bypass enabling unauthorized trading on restricted pools). Both outcomes constitute broken core pool functionality. In the bypass scenario, unauthorized users can trade on pools designed to restrict access to KYC'd or whitelisted counterparties, constituting a direct violation of the pool admin's access control intent and potential loss of LP value through unauthorized swaps.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard documented periphery entry point. The misbinding is unconditional â€” it triggers on every router-mediated swap regardless of amounts, tokens, or timing. No special privileges, flash loans, or multi-step setup are required. Any user who calls `exactInputSingle` on a pool with `SwapAllowlistExtension` triggers the broken check. The bypass scenario (router allowlisted) is a natural admin action when the admin wants allowlisted users to be able to use the router.

## Recommendation
The `sender` forwarded to extensions must represent the originating user, not the intermediary. Two options:

1. **In the router (preferred):** Before calling `pool.swap()`, store the original `msg.sender` in a dedicated transient storage slot (analogous to how `_setNextCallbackContext` already stores the payer). Expose a getter so the pool or extension can read the true initiator. Pass this value as `sender` to `_beforeSwap` instead of `msg.sender`.

2. **In the extension:** Accept the real user address via `extensionData` (ABI-encoded by the router), verify it against a router-signed or router-provided value, and use that for the allowlist check. This requires router cooperation and is more complex to secure.

## Proof of Concept

```solidity
// Pool configured with SwapAllowlistExtension
// allowedSwapper[pool][alice] = true  (alice is the only allowed swapper)
// allowedSwapper[pool][router] = false (router not explicitly allowlisted)

// Direct swap by alice â€” works correctly:
vm.prank(alice);
pool.swap(alice, true, 1000, 0, "", "");  // passes: allowedSwapper[pool][alice] = true

// Router swap by alice â€” incorrectly reverts:
vm.prank(alice);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
router.exactInputSingle(...);  // fails: checks allowedSwapper[pool][router] = false

// Admin allowlists router to fix alice's access:
// allowedSwapper[pool][router] = true

// Now router swap by bob (not allowlisted) â€” incorrectly succeeds:
vm.prank(bob);
router.exactInputSingle(...);  // passes: checks allowedSwapper[pool][router] = true
// bob bypasses the per-user allowlist entirely
```
