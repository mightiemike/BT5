### Title
SwapAllowlistExtension gates the router address instead of the actual user for router-mediated swaps, making per-user allowlist bypassable - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which is the `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` of `pool.swap()`, so the extension checks whether the **router** is allowlisted ŌĆö not the actual user. If the pool admin allowlists the router to enable router-mediated swaps for their permitted users, every unpermitted user can bypass the per-user allowlist by calling the router.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[pool][sender]`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(recipient, ...)`: [4](#0-3) 

At that point `msg.sender` of `pool.swap()` is the router, so `sender` arriving at the extension is the router address. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

Contrast this with `DepositAllowlistExtension`, which correctly checks `owner` ŌĆö the economically relevant LP position holder ŌĆö regardless of who calls `addLiquidity`: [5](#0-4) 

The swap allowlist has no equivalent mechanism to recover the actual user identity through the router.

### Impact Explanation

A pool admin who deploys a private pool with `SwapAllowlistExtension` and then allowlists the router (the natural step to enable router-mediated swaps for their permitted users) inadvertently opens the pool to **all** users. Any address can call `router.exactInputSingle` or `router.exactInput` targeting the pool; the extension sees `sender = router`, finds it allowlisted, and permits the swap. The per-user allowlist is entirely bypassed for router-mediated paths. Unauthorized traders can extract value from the pool's LP positions through unrestricted swaps, constituting a direct loss of LP principal.

### Likelihood Explanation

The bypass requires the pool admin to allowlist the router ŌĆö a reasonable and expected action for any pool that intends to support router-mediated swaps alongside the allowlist. The Sherlock audit target description for this exact path states: *"Because public users may enter through the router, the hook must gate the same actor the pool designers thought they were allowlisting."* The mismatch between intent and implementation is reachable by any unprivileged user with no special setup beyond the pool admin performing the natural configuration step.

### Recommendation

Pass the original user identity through the swap path so the extension can gate the actual swapper. Two options:

1. **Preferred**: Change `beforeSwap` to check `recipient` (the address receiving swap output) instead of `sender`, since `recipient` is set by the user even through the router. Alternatively, add a dedicated `swapper` field to `extensionData` that the router populates with `msg.sender` before forwarding.
2. **Alternative**: Mirror the deposit allowlist pattern ŌĆö add an explicit `swapper` parameter to the pool's `swap` signature (analogous to `owner` in `addLiquidity`) so the caller can declare the economically relevant identity and the extension can gate on it.

### Proof of Concept

1. Deploy pool with `SwapAllowlistExtension`.
2. Pool admin calls `setAllowedToSwap(pool, userA, true)` ŌĆö only `userA` is permitted.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` ŌĆö router allowlisted to enable router-mediated swaps for `userA`.
4. `userB` (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(userB, ...)` ŌĆö `msg.sender` of `pool.swap()` = router.
6. Extension evaluates `allowedSwapper[pool][router]` = `true` ŌåÆ swap proceeds.
7. `userB` successfully trades in the private pool, bypassing the per-user allowlist entirely.
