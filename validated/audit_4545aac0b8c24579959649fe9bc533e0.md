### Title
Unvalidated HTTP Redirect Following in Bitcoin RPC Client Enables Server-Side Request Forgery — (`relayer/src/bitcoin_client.rs`)

---

### Summary

The relayer's custom `minreq`-based HTTP transport sends JSON-RPC POST requests to the configured Bitcoin node without disabling HTTP redirect following. If the Bitcoin node (or a network path to it) is compromised, an attacker can respond with an HTTP 3xx redirect pointing to a local service on the relayer host. The `minreq` client will automatically follow the redirect and re-issue the POST request to the attacker-chosen local address, enabling SSRF against any HTTP service co-located with the relayer.

---

### Finding Description

`CustomMinreqHttpTransport` is the sole HTTP transport used by the relayer to communicate with the Bitcoin full node. It constructs `minreq::Request` objects and calls `.send()` with no redirect policy configured:

```rust
// relayer/src/bitcoin_client.rs  lines 44-57
let req = match &self.basic_auth {
    Some(auth) => minreq::Request::new(minreq::Method::Post, &self.url)
        .with_timeout(self.timeout.as_secs())
        .with_header("Authorization", auth)
        .with_headers(self.headers.clone())
        .with_json(&req)?,
    None => minreq::Request::new(minreq::Method::Post, &self.url)
        .with_timeout(self.timeout.as_secs())
        .with_json(&req)?,
};
let resp = req.send()?;
```

`minreq` follows HTTP redirects by default (up to its built-in limit). There is no call to `.with_max_redirects(0)` or any equivalent guard anywhere in the transport. The `url` field is taken directly from `config.bitcoin.endpoint` — a plain `String` with no scheme enforcement — meaning the connection may be unencrypted HTTP. [1](#0-0) 

The same transport is used for every Bitcoin RPC call the relayer makes: `get_block_count`, `get_block_hash`, `get_block_header`, `get_aux_block_header`, etc. [2](#0-1) 

The endpoint is loaded from operator configuration with no URL-scheme or host validation: [3](#0-2) 

---

### Impact Explanation

An attacker who can inject a redirect response on the Bitcoin RPC channel (by compromising the node, performing a MITM on a plaintext HTTP connection, or poisoning DNS) can redirect the relayer's POST request to any HTTP service reachable from the relayer host — including:

- The NEAR RPC endpoint (default port 3030) running on the same machine, causing it to receive a malformed or crafted JSON body.
- Local admin or monitoring HTTP interfaces that accept unauthenticated POST requests.
- Any other co-located service that acts on POST bodies.

Because the relayer holds a funded NEAR signing key and continuously submits transactions, a successful redirect to the local NEAR RPC could be used to submit crafted transactions or view calls that the attacker controls, corrupting the light-client's canonical chain state or draining the relayer's NEAR balance.

---

### Likelihood Explanation

The Bitcoin node is typically accessed over plaintext HTTP (the `endpoint` field is an unvalidated string; Bitcoin Core's default RPC port 8332 uses HTTP). A network-level attacker on the path between the relayer and the node — or an attacker who has compromised the node itself — can inject a single redirect response. This matches the exact threat model described in the reference report ("attacker gains control over the [upstream] API"). No privileged contract role is required; the vulnerability is entirely in the off-chain relayer process.

---

### Recommendation

In `CustomMinreqHttpTransport::request`, disable redirect following by calling `.with_max_redirects(0)` on every `minreq::Request` before calling `.send()`:

```rust
let req = minreq::Request::new(minreq::Method::Post, &self.url)
    .with_max_redirects(0)          // ← add this
    .with_timeout(self.timeout.as_secs())
    .with_json(&req)?;
```

Apply the same guard to both the authenticated and unauthenticated branches. Additionally, validate that `config.bitcoin.endpoint` uses the `http://` or `https://` scheme and, where possible, enforce HTTPS to eliminate plaintext MITM as an attack vector. [4](#0-3) 

---

### Proof of Concept

1. Operator runs the relayer with `bitcoin.endpoint = "http://btc-node.internal:8332"`.
2. Attacker performs a MITM on the plaintext HTTP connection (or compromises the node).
3. When the relayer calls `get_block_count` (or any other RPC), the attacker intercepts the POST and responds:
   ```
   HTTP/1.1 307 Temporary Redirect
   Location: http://127.0.0.1:3030/
   ```
4. `minreq` follows the 307 (which preserves the POST method and body) and re-issues the JSON-RPC request to `http://127.0.0.1:3030/` — the local NEAR RPC.
5. The NEAR RPC receives an unexpected POST body; depending on the endpoint and body content, it may execute a view call or return data that the attacker can observe, or the attacker can craft the redirect target to be any other local HTTP service that accepts POST requests.

struct CustomMinreqHttpTransport {
    url: String,
    timeout: std::time::Duration,

impl CustomMinreqHttpTransport {
    fn request<R>(&self, req: impl serde::Serialize) -> Result<R, jsonrpc::minreq_http::Error>
    where
        R: for<'a> serde::de::Deserialize<'a>,
    {
        let req = match &self.basic_auth {
            Some(auth) => minreq::Request::new(minreq::Method::Post, &self.url)
                .with_timeout(self.timeout.as_secs())
                .with_header("Authorization", auth)
                .with_headers(self.headers.clone())
                .with_json(&req)?,
            None => minreq::Request::new(minreq::Method::Post, &self.url)
                .with_timeout(self.timeout.as_secs())
                .with_json(&req)?,
        };

        // Send the request and parse the response. If the response is an error that does not
        // contain valid JSON in its body (for instance if the bitcoind HTTP server work queue
        // depth is exceeded), return the raw HTTP error so users can match against it.
        let resp = req.send()?;
