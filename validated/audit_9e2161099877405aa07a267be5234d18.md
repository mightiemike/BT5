### Title
Inconsistent Uptime Metric Between Kickout-Exemption Sorting and Reward Calculation Causes Incorrect Validator Exemption — (File: chain/epoch-manager/src/validator_stats.rs)

### Summary

`get_sortable_validator_online_ratio` — used to rank validators for kickout exemption — always passes `None` for the endorsement cutoff threshold, computing raw uptime. `get_validator_online_ratio` — used in reward calculation — receives `Some(chunk_validator_only_kickout_threshold)`, binarizing the endorsement contribution to 0 or 1. The two functions therefore evaluate the same validator's performance with different metrics, so the validator selected for kickout exemption (highest raw uptime) is not necessarily the validator that would receive the highest reward (highest binarized uptime). This is the direct nearcore analog of the external report's pattern: applying a transformation separately to one component of a sum before aggregating produces a different result than applying it to the whole.

### Finding Description

In `compute_validators_to_reward_and_kickout`, validators are sorted for kickout exemption by `get_sortable_validator_online_ratio`: [1](#0-0) 

This function unconditionally calls `get_validator_online_ratio(stats, None)` — no cutoff — so the endorsement ratio enters the average as its raw value. [2](#0-1) 

In `finalize_epoch`, the reward calculation passes `endorsement_cutoff_threshold: Some(epoch_config.chunk_validator_only_kickout_threshold)`: [3](#0-2) 

Inside `calculate_reward`, `get_validator_online_ratio` is called with that cutoff, binarizing the endorsement contribution to 0 (below threshold) or 1 (above threshold): [4](#0-3) 

The binarization logic in `get_endorsement_ratio`: [5](#0-4) 

The match arm for a validator doing all three duties (blocks + chunks + endorsements) shows how the endorsement term enters the average: [6](#0-5) 

With the cutoff active, `produced_endorsements / expected_endorsements` is replaced by `0/1` or `1/1`. Without the cutoff (sorting path), the raw ratio is used. The two paths therefore produce divergent uptime values for the same validator whenever endorsements straddle the threshold.

The exemption selection in `compute_validators_to_reward_and_kickout` uses the raw-uptime sort: [7](#0-6) 

### Impact Explanation

When `validator_max_kickout_stake_perc` is exceeded and the exemption mechanism fires, the validator with the highest **raw** uptime is protected. If that validator's endorsements are just below the cutoff threshold, their **binarized** uptime collapses (endorsement term → 0), pulling the average below `online_min_threshold` (90/100 on mainnet). That validator stays in the set but earns **zero reward**. A competing validator with slightly lower raw uptime but endorsements just above the threshold would have earned a reward — but is kicked out instead. The result is a wasted validator slot and an incorrect reward distribution, directly analogous to the external report's incorrect margin calculation caused by applying a factor separately to realized vs. unrealized balances.

Concrete numbers with `chunk_validator_only_kickout_threshold = 70`, `online_min_threshold = 90/100`, `block_producer_kickout_threshold = 80`:

| Validator | blocks | chunks | endorsements | Raw uptime | Binarized uptime | Reward |
|-----------|--------|--------|--------------|------------|------------------|--------|
| X (kicked out, exempted) | 79% | 79% | 69% | 75.7% | 52.7% | **0** |
| Y (kicked out, not exempted) | 79% | 79% | 71% | 76.3% | 86.0% | **would earn reward** |

Sorting by raw uptime: Y (76.3%) > X (75.7%) → Y is exempted, X is kicked out. But Y's binarized uptime (86%) is still below `online_min_threshold` (90%), so Y also earns 0 reward. The exemption protects the wrong validator and the reward calculation confirms neither earns anything — yet the validator that *would* have earned a reward under a consistent metric is the one removed from the set.

### Likelihood Explanation

The exemption mechanism triggers whenever the total stake of validators to be kicked out exceeds `validator_max_kickout_stake_perc` (30% on mainnet per epoch config files). The endorsement cutoff threshold is always active in production (`finalize_epoch` always passes `Some(chunk_validator_only_kickout_threshold)`). Any epoch with a network-wide performance dip that pushes multiple validators below kickout thresholds simultaneously can trigger this path. The divergence between raw and binarized uptime is largest for validators whose endorsement ratio is near the cutoff boundary — a common scenario during partial network degradation.

### Recommendation

Update `get_sortable_validator_online_ratio` to accept and forward the same `endorsement_cutoff_threshold` that is used in reward calculation, so that kickout-exemption sorting and reward calculation evaluate validators with an identical uptime metric. Aggregate the endorsement contribution consistently — either always raw or always binarized — before comparing validators for exemption. [8](#0-7) 

### Proof of Concept

The divergence is directly observable from the existing unit tests. `test_average_uptime_ratio_with_endorsement_cutoff_not_passed` shows that with `endorsement_cutoff = Some(50)` and `endorsement = 42/100`, the binarized uptime is `(98/100 + 76/100 + 0) / 3 = 58%`: [9](#0-8) 

But `get_sortable_validator_online_ratio` — called with `None` — returns `(98/100 + 76/100 + 42/100) / 3 = 72%` for the same stats: [10](#0-9) 

The same `BlockChunkValidatorStats` produces 72% for kickout-exemption sorting and 58% for reward calculation — a 14-percentage-point divergence that can flip which validator is exempted and whether they earn a reward, exactly mirroring the external report's realized/unrealized balance inconsistency.

/// Computes the overall online (uptime) ratio of the validator for sorting.
/// The reason for this function is that U256 used in the core implementation
/// cannot be used with `Ratio<U256>` for sorting since it does not implement `num_integer::Integer`.
/// Instead of having a full-blown implementation of `U256`` for `num_integer::Integer`
/// we wrap the value in a `BigInt` for now.
/// TODO: Implement `num_integer::Integer` for `U256` and remove this function.
/// cspell:words bigdenom bignumer
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);
    let mut bytes: [u8; size_of::<U256>()] = [0; size_of::<U256>()];
    ratio.numer().to_little_endian(&mut bytes);
    let bignumer = BigUint::from_bytes_le(&bytes);
    ratio.denom().to_little_endian(&mut bytes);
    let bigdenom = BigUint::from_bytes_le(&bytes);
    BigRational::new(bignumer.try_into().unwrap(), bigdenom.try_into().unwrap())
}

fn get_endorsement_ratio(stats: &ValidatorStats, cutoff_threshold: Option<u8>) -> (u64, u64) {
    let (numer, denom) = if stats.expected == 0 {
        debug_assert_eq!(stats.produced, 0);
        (0, 0)
    } else if let Some(threshold) = cutoff_threshold {
        if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
    } else {
        (stats.produced, stats.expected)
    };
    (numer, denom)
}
