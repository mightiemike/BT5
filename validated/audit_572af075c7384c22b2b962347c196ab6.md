### Title
Swap Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool, which is `msg.sender` to the pool (i.e., the router contract), not the actual end-user. When a non-allowlisted user routes through `MetricOmmSimpleRouter`, the extension sees the router address as the swapper. If the pool admin has allowlisted the router (required for any allowlisted user to use the router), every user — including non-allowlisted ones — can bypass the curated-pool gate by routing through the router.

---

### Finding Description

**Root cause — wrong actor bound in `beforeSwap`**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist: [1](#0-0) 

`msg.sender` inside the extension is the pool (correct key for the mapping), and `sender` is whatever the pool forwarded. The pool always forwards its own `msg.sender`: [2](#0-1) 

`ExtensionCalling._beforeSwap` passes that value straight through: [3](#0-2) 

**Router path**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly: [4](#0-3) 

When the router is the caller, `msg.sender` to the pool is the router address. The pool therefore passes `sender = router` to the extension. The extension evaluates:

```
allowedSwapper[pool][router]   // checked
allowedSwapper[pool][real_user] // never checked
```

**The dilemma for the pool admin**

| Router allowlisted? | Allowlisted user via router | Non-allowlisted user via router |
|---|---|---|
| No | Blocked (broken UX) | Blocked |
| Yes | Passes | **Also passes — bypass** |

There is no configuration in which the pool admin can allow allowlisted users to use the router while blocking non-allowlisted users. Allowlisting the router is an all-or-nothing gate that admits every caller.

The `DepositAllowlistExtension` does **not** share this flaw — it ignores `sender` and checks `owner`, which the liquidity adder passes as the actual position owner: [5](#0-4) 

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs, or protocol-controlled accounts) can be fully bypassed by any user routing through `MetricOmmSimpleRouter`. The attacker receives the same oracle-anchored execution as an allowlisted user, draining LP value or front-running in a pool the admin intended to gate. This breaks the core curation invariant and constitutes unauthorized access to restricted pool liquidity.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public swap entrypoint documented and deployed for end-users. Any pool admin who enables the router for allowlisted users (the only way to give allowlisted users router access) simultaneously opens the gate to all users. The bypass requires no special privileges, no flash loans, and no unusual token behavior — only a standard router call.

---

### Recommendation

The extension must resolve the actual end-user, not the intermediary. Two sound approaches:

1. **Pass `tx.origin` as `sender` from the pool** — fragile and generally discouraged.
2. **Require the router to forward the real user** — add a `swapper` field to `extensionData` that the router populates with `msg.sender`, and have the extension decode and check that field instead of the `sender` argument.
3. **Check `sender` only when `sender` is an EOA; otherwise decode from `extensionData`** — the router already forwards `extensionData` untouched, so the real user can be embedded there by the router and verified by the extension.

The cleanest fix is option 2: the router encodes `msg.sender` into `extensionData` before calling the pool, and `SwapAllowlistExtension.beforeSwap` decodes and checks that value when `sender` is a known router address.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the only allowed swapper
  allowedSwapper[pool][router] = true  // admin must set this for alice to use the router

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

  router calls:
    pool.swap(bob, zeroForOne, amount, priceLimit, "", extensionData)
    // msg.sender to pool = router

  pool calls:
    _beforeSwap(sender=router, ...)

  extension checks:
    allowedSwapper[pool][router] == true  ✓  → swap proceeds

Result:
  bob swaps successfully in a pool he is not allowlisted for.
  The allowlist is completely bypassed.
```
