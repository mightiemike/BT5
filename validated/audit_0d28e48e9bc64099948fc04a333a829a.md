### Title
Linked Signer Can Overwrite Its Own Entry in `linkedSigners` Without Subaccount Owner Consent — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.sol`, the sequencer-path handler for `LinkSigner` transactions calls `validateSignedTx` with `allowLinkedSigner = true`. This means the **current linked signer** — not the subaccount owner — can sign and submit a `LinkSigner` transaction that replaces the `linkedSigners[subaccount]` entry with any arbitrary address. Because the linked signer already has authority to sign `WithdrawCollateral` transactions (also `allowLinkedSigner = true`), a compromised linked signer can atomically rotate itself to an attacker-controlled address and then drain the subaccount, all before the owner can respond via the 3-day slow-mode path.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch decodes a `SignedLinkSigner` struct and calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes the current linked signer to the verifier as an acceptable signer when `allowLinkedSigner` is `true`:

```solidity
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

This means the linked signer's signature alone is sufficient to mutate `linkedSigners[subaccount]` to any target address — no subaccount-owner signature is required.

By contrast, the **slow-mode** path for `LinkSigner` correctly enforces ownership via `validateSender`, which requires `address(uint160(bytes20(txn.sender))) == sender` (i.e., the first 20 bytes of the subaccount must equal `msg.sender`, the actual owner):

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

```solidity
function validateSender(bytes32 txSender, address sender) internal view {
    require(
        address(uint160(bytes20(txSender))) == sender ||
            sender == address(this),
        ERR_SLOW_MODE_WRONG_SENDER
    );
}
``` [4](#0-3) 

The slow-mode path is safe; the sequencer path is not.

---

### Impact Explanation

The `linkedSigners` mapping is a critical security variable: it designates which address may sign on behalf of a subaccount for all operations that pass `allowLinkedSigner = true`, including `WithdrawCollateral`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // allowLinkedSigner for WithdrawCollateral
);
``` [5](#0-4) 

A compromised linked signer can:
1. Sign a `LinkSigner` transaction pointing to an attacker-controlled address → sequencer writes `linkedSigners[victim] = attacker`.
2. Immediately sign a `WithdrawCollateral` transaction with the new attacker key → funds are drained.

The subaccount owner's only recourse is the slow-mode `LinkSigner` path, which carries a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`):

```solidity
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
``` [6](#0-5) 

By the time the owner's revocation is executable, the attacker has already drained the account. **Impact: complete theft of all collateral held in the subaccount.**

---

### Likelihood Explanation

Linked signers are explicitly designed to be hot wallets or automated trading bots — keys that are online, frequently used, and therefore more likely to be compromised than the cold-wallet subaccount owner key. Any key-leak, phishing, or malware event targeting the linked signer key is sufficient to trigger this exploit. No admin access, governance capture, or sequencer compromise is required. The attacker only needs to obtain the linked signer's private key and submit two sequencer-path transactions.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner (whose address is embedded in the first 20 bytes of the `sender` bytes32) should be permitted to change the linked signer. The slow-mode path already enforces this correctly via `validateSender`; the sequencer path should match that invariant.

```diff
- validateSignedTx(
-     signedTx.tx.sender,
-     signedTx.tx.nonce,
-     transaction,
-     signedTx.signature,
-     true
- );
+ validateSignedTx(
+     signedTx.tx.sender,
+     signedTx.tx.nonce,
+     transaction,
+     signedTx.signature,
+     false   // only owner may change linked signer
+ );
``` [7](#0-6) 

---

### Proof of Concept

1. Alice (owner `0xAlice`) sets Bob (`0xBob`) as her linked signer via a valid `LinkSigner` transaction. `linkedSigners[alice_subaccount] = 0xBob`.
2. Attacker obtains `0xBob`'s private key (hot wallet compromise).
3. Attacker constructs a `SignedLinkSigner` transaction: `sender = alice_subaccount`, `signer = 0xAttacker`, signed by `0xBob`.
4. Attacker submits this to the sequencer. `validateSignedTx` accepts `0xBob`'s signature because `allowLinkedSigner = true` and `getLinkedSigner(alice_subaccount) == 0xBob`. State: `linkedSigners[alice_subaccount] = 0xAttacker`.
5. Attacker constructs a `SignedWithdrawCollateral` transaction: `sender = alice_subaccount`, signed by `0xAttacker`.
6. Sequencer processes it; `validateSignedTx` accepts `0xAttacker`'s signature because `getLinkedSigner(alice_subaccount) == 0xAttacker`. Alice's collateral is withdrawn to the attacker.
7. Alice submits a slow-mode `LinkSigner` to revoke, but `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (3 days). Funds are already gone. [1](#0-0) [8](#0-7)
