### Title
`CongestionInfo.delayed_receipts_gas` Underflow or Permanent Miscounting Across Protocol-Upgrade Fee Boundary — (`runtime/runtime/src/congestion_control.rs`)

---

### Summary

When a receipt is pushed into the delayed-receipt queue or outgoing buffer, its congestion gas is computed with the **current** `RuntimeConfig` fee parameters and added to the `CongestionInfo.delayed_receipts_gas` accumulator. When the receipt is later popped, the gas is **recomputed** with whatever `RuntimeConfig` is active at that moment. If a protocol upgrade changes fee parameters between push and pop, the subtracted value diverges from the added value, breaking the invariant that `delayed_receipts_gas` is an exact running sum. This mirrors the external bug exactly: a value is recorded at one "exchange rate" and must be settled at a different one.

---

### Finding Description

The congestion control system maintains `CongestionInfo.delayed_receipts_gas` as a running sum of gas across all receipts in the delayed queue. The invariant is:

```
delayed_receipts_gas = Σ compute_receipt_congestion_gas(r, config_at_push_time)
```

**Push path** (`DelayedReceiptQueueWrapper::push`, line 846):

```rust
let gas = compute_receipt_congestion_gas(&receipt, &config)?;
// ...
self.new_delayed_gas = self.new_delayed_gas.checked_add(gas)...;
self.queue.push_back(trie_update, &receipt)?;
```

When `config.use_state_stored_receipt == false` (the pre-feature state), the receipt is stored as a plain `ReceiptOrStateStoredReceipt::Receipt` — **no gas is frozen in metadata**.

**Pop path** (`DelayedReceiptQueueWrapper::pop`, line 895):

```rust
let delayed_gas = receipt_congestion_gas(&receipt, &config)?;
```

`receipt_congestion_gas` dispatches on the variant:

```rust
match receipt {
    ReceiptOrStateStoredReceipt::Receipt(receipt) =>
        compute_receipt_congestion_gas(receipt, config),   // recomputed with CURRENT config
    ReceiptOrStateStoredReceipt::StateStoredReceipt(receipt) =>
        Ok(receipt.metadata().congestion_gas),             // frozen at push time
}
```

For old-style receipts the gas is **recomputed with the current config**. If a protocol upgrade between push and pop changes any fee parameter consumed by `action_receipt_congestion_gas` (exec fees, send fees — all drawn from `config.fees`), the value subtracted differs from the value added.

**Underflow branch** (`CongestionInfo::remove_delayed_receipt_gas`, line 281–292):

```rust
inner.delayed_receipts_gas = inner
    .delayed_receipts_gas
    .checked_sub(gas.as_gas().into())
    .ok_or_else(|| RuntimeError::UnexpectedIntegerOverflow(
        "remove_delayed_receipt_gas".into()))?;
```

`checked_sub` returns `None` when the subtracted value exceeds the accumulator, propagating `RuntimeError::UnexpectedIntegerOverflow` up through `process_delayed_receipts` → `apply` → chunk application, halting chunk processing for that shard.

The `StateStoredReceipt` path (introduced by the `use_state_stored_receipt` protocol feature) was designed to freeze gas in metadata and avoid this problem for **newly pushed** receipts. However, receipts already in the queue at the time the feature activates remain as plain `Receipt` objects and are still subject to the divergence on the next fee-changing upgrade.

---

### Impact Explanation

**High.** Two outcomes depending on direction of fee change:

1. **Fee increases** (new config computes higher gas than old config): `remove_delayed_receipt_gas` underflows → `RuntimeError::UnexpectedIntegerOverflow` → chunk application fails → the shard cannot produce valid chunks → effective chain halt for that shard until the delayed queue drains past the affected receipts (which it cannot, because processing fails).

2. **Fee decreases** (new config computes lower gas than old config): the accumulator is never fully drained even when the queue is empty → `delayed_receipts_gas > 0` permanently → the shard reports false incoming congestion → other shards throttle forwarding to it indefinitely, degrading throughput.

Both outcomes are triggered automatically at the epoch boundary of a protocol upgrade, requiring no further user action after the initial receipt submission.

---

### Likelihood Explanation

**Medium.** The preconditions are:

1. Receipts must be in the delayed queue stored as plain `Receipt` (i.e., pushed before `use_state_stored_receipt` was enabled, or on a node that had the feature disabled).
2. A subsequent protocol upgrade must change any fee parameter consumed by `compute_receipt_congestion_gas` — specifically exec fees (`total_prepaid_exec_fees`) or send fees (`total_prepaid_send_fees`) for the action types present in the queued receipts.
3. The affected receipts must not have been processed before the upgrade epoch boundary.

Any user can submit transactions that create delayed receipts (by flooding a shard past its gas limit). The protocol upgrade boundary is a public, scheduled event. The combination is reachable without privileged access.

---

### Recommendation

1. **Freeze gas at push time unconditionally.** The `StateStoredReceipt` mechanism already does this for new receipts. Ensure that when `use_state_stored_receipt` is enabled, a one-time migration re-writes all existing plain `Receipt` entries in the delayed queue and outgoing buffers as `StateStoredReceipt` with gas computed under the **current** config, so the accumulator is consistent before any future fee change.

2. **Guard fee-parameter changes with a queue-drain check.** Before activating any protocol feature that alters `compute_receipt_congestion_gas` output, verify (or assert in migration code) that no plain `Receipt` objects remain in any shard's delayed queue or outgoing buffer.

3. **Add a reconciliation step in `bootstrap_congestion_info`.** Call `bootstrap_congestion_info` (which recomputes all gas from scratch) at the first chunk of a new protocol version that changes fees, replacing the carried-over `CongestionInfo` from the previous epoch.

---

### Proof of Concept

**Scenario (no privileged role required):**

1. At protocol version N (before `use_state_stored_receipt`), a user submits enough function-call transactions to shard S to overflow its gas limit. The excess receipts are pushed to the delayed queue as plain `Receipt`. Each receipt with 300 Tgas attached contributes `G1 = exec_fee_v_N + send_fee_v_N + 300 Tgas` to `delayed_receipts_gas`.

2. Protocol upgrades to version N+1. The upgrade increases `new_action_receipt` exec fee by δ. The `RuntimeConfig` for version N+1 is now active.

3. At the first chunk of epoch N+1, `process_delayed_receipts` pops a receipt. `receipt_congestion_gas` dispatches to `compute_receipt_congestion_gas(receipt, config_N+1)`, returning `G2 = G1 + δ`.

4. `apply_congestion_changes` calls `congestion.remove_delayed_receipt_gas(G2)`. Since `delayed_receipts_gas` holds only `G1` (or a sum based on `G1` values), `G2 > delayed_receipts_gas` → `checked_sub` returns `None` → `RuntimeError::UnexpectedIntegerOverflow("remove_delayed_receipt_gas")`.

5. Chunk application for shard S fails. The shard cannot advance. All further withdrawals and cross-shard receipts targeting shard S are stuck — the exact analog of the external report's "entire withdrawal flow will be stuck because this epoch can't be settled."

/// Get the receipt gas from the receipt that was retrieved from the state.
/// If it is a [Receipt], the gas will be computed.
/// If it s the [StateStoredReceipt], the size will be read from the metadata.
pub(crate) fn receipt_congestion_gas(
    receipt: &ReceiptOrStateStoredReceipt,
    config: &RuntimeConfig,
) -> Result<Gas, IntegerOverflowError> {
    match receipt {
        ReceiptOrStateStoredReceipt::Receipt(receipt) => {
            compute_receipt_congestion_gas(receipt, config)
        }
        ReceiptOrStateStoredReceipt::StateStoredReceipt(receipt) => {
            Ok(receipt.metadata().congestion_gas)
        }
    }
}

fn action_receipt_congestion_gas(
    receipt: &Receipt,
    config: &RuntimeConfig,
    action_receipt: VersionedActionReceipt,
) -> Result<Gas, IntegerOverflowError> {
    let prepaid_exec_gas =
        total_prepaid_exec_fees(config, &action_receipt.actions(), receipt.receiver_id())?
            .gas
            .checked_add(config.fees.fee(ActionCosts::new_action_receipt).exec_fee().gas)
            .ok_or(IntegerOverflowError)?;
    // account for gas guaranteed to be used for creating new receipts
    let prepaid_send_cost = total_prepaid_send_fees(config, &action_receipt.actions())?;
    let prepaid_gas = prepaid_exec_gas.checked_add_result(prepaid_send_cost.gas)?;

    // account for gas potentially used for dynamic execution
    let gas_attached_to_fns = total_prepaid_gas(&action_receipt.actions())?;
    let gas = gas_attached_to_fns.checked_add_result(prepaid_gas)?;

    Ok(gas)

        self.queue.push_back(trie_update, &receipt)?;
        Ok(())

    pub(crate) fn pop(
        &mut self,
        trie_update: &mut TrieUpdate,
        config: &RuntimeConfig,
    ) -> Result<Option<ReceiptOrStateStoredReceipt<'_>>, RuntimeError> {
        // While processing receipts, we need to keep track of the gas and bytes
        // even for receipts that may be filtered out due to a resharding event
        loop {
            // Check proof size limit before each receipt is popped.
            if trie_update.trie.check_proof_size_limit_exceed() {
                break;
            }
            let Some(receipt) = self.queue.pop_front(trie_update)? else {
                break;
            };
            let delayed_gas = receipt_congestion_gas(&receipt, &config)?;
            let delayed_bytes = receipt_size(&receipt)? as u64;
            self.removed_delayed_gas =
                self.removed_delayed_gas.checked_add(delayed_gas).ok_or(IntegerOverflowError)?;
            self.removed_delayed_bytes = self
                .removed_delayed_bytes
                .checked_add(delayed_bytes)
                .ok_or(IntegerOverflowError)?;

            // Track gas and bytes for receipt above and return only receipt that belong to the shard.
            if self.receipt_filter_fn(&receipt) {
                return Ok(Some(receipt));
            }
        }
        Ok(None)
    }
