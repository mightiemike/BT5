### Title
Duplicate-Last-Leaf Merkle Proof Accepts Phantom `tx_index` — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`, `contract/src/lib.rs::verify_transaction_inclusion`)

---

### Summary

`compute_root_from_merkle_proof` performs no bounds check on `transaction_position` against the actual number of transactions in the block. In an odd-width Bitcoin Merkle tree the last leaf is duplicated during construction, which means the proof generated for the real last leaf (index N-1) is mathematically identical to a proof for the phantom duplicate leaf (index N). An unprivileged caller can therefore pass `tx_index = N` with the same `tx_id` and `merkle_proof` that are valid for `tx_index = N-1`, and the function returns `true` for a position that does not correspond to any real transaction.

---

### Finding Description

**Root cause — `merkle-tools/src/lib.rs`**

`compute_root_from_merkle_proof` iterates over the supplied proof hashes, choosing left/right placement purely from `current_position % 2`, then halves the position each step: [1](#0-0) 

There is no check that `transaction_position` is less than the number of leaves in the tree. The contract stores only the block header (which contains `merkle_root`) — it never stores the transaction count — so no upper-bound guard can be applied at the call site either.

**Why the duplicate-leaf makes two positions equivalent**

For a 3-transaction block `[T0, T1, T2]`, Bitcoin's construction duplicates the last leaf:

```
Level 0 (leaves): T0  T1  T2  T2   ← T2 duplicated
Level 1:          H(T0,T1)  H(T2,T2)
Level 2 (root):   H(H(T0,T1), H(T2,T2))
```

The canonical proof for T2 at index 2 is `[T2, H(T0,T1)]`.

*Verification at index 2 (real):*
- pos=2 (even) → `H(T2, proof[0]=T2)` = `H(T2,T2)`; pos→1
- pos=1 (odd)  → `H(proof[1]=H(T0,T1), H(T2,T2))` = root ✓

*Verification at index 3 (phantom):*
- pos=3 (odd)  → `H(proof[0]=T2, T2)` = `H(T2,T2)`; pos→1
- pos=1 (odd)  → `H(proof[1]=H(T0,T1), H(T2,T2))` = root ✓

Both calls return the same root and therefore both return `true` from `verify_transaction_inclusion`.

**Call path**

`verify_transaction_inclusion` is a public, unpermissioned function (only `#[pause]`-gated): [2](#0-1) 

The attacker controls every field of `ProofArgs`: [3](#0-2) 

`verify_transaction_inclusion_v2` also delegates to `verify_transaction_inclusion` after its coinbase check, so it inherits the same flaw: [4](#0-3) 

---

### Impact Explanation

Any downstream bridge, mint, unlock, or withdrawal contract that uses `(tx_id, tx_block_blockhash, tx_index)` as its replay-protection key can be double-spent:

1. Attacker submits a legitimate deposit transaction T2 (index 2 in a 3-tx block) → bridge marks `(T2, block, 2)` as used and releases funds.
2. Attacker calls `verify_transaction_inclusion` again with the identical `tx_id=T2`, same `merkle_proof`, but `tx_index=3` → returns `true`.
3. Bridge sees `(T2, block, 3)` as unused → releases funds a second time.

The light client itself has no state that records which `(tx_id, tx_index)` pairs have been verified, so the replay is invisible at the contract level.

---

### Likelihood Explanation

- `verify_transaction_inclusion` is callable by any account with no role requirement.
- The attacker needs only a real transaction in an odd-width block (extremely common on Bitcoin mainnet) and the standard SPV proof for it.
- No privileged access, no leaked keys, no social engineering required.
- The deprecated v1 function remains live and callable; v2 is also affected via delegation.

---

### Recommendation

1. **Enforce `tx_index < 2^proof_length`** — the proof length encodes the tree depth; the maximum valid leaf index for a proof of length `k` is `2^k - 1`. Reject any `tx_index >= 2^(merkle_proof.len())`.
2. **Reject even-position last-leaf proofs where `proof[0] == tx_id`** — this is the fingerprint of the duplicate-leaf case; a legitimate sibling is never equal to the transaction itself unless the tree was constructed with duplication.
3. **Require callers to supply the total transaction count** and validate `tx_index < tx_count` inside `compute_root_from_merkle_proof` or at the `verify_transaction_inclusion` layer.
4. Downstream bridges must use `tx_id` alone (not `tx_index`) as the replay-protection key, or must independently validate that `tx_index` is within the real transaction count obtained from a trusted source.

### Proof of Concept

```
Tree: [T0, T1, T2]  (3 transactions, odd width)
Merkle root = H(H(T0,T1), H(T2,T2))

Canonical proof for T2 at index 2:
  merkle_proof = [T2, H(T0,T1)]

Call 1 — legitimate:
  verify_transaction_inclusion({ tx_id: T2, tx_index: 2,
      merkle_proof: [T2, H(T0,T1)], ... })
  → compute_root_from_merkle_proof returns root → true ✓

Call 2 — phantom index:
  verify_transaction_inclusion({ tx_id: T2, tx_index: 3,
      merkle_proof: [T2, H(T0,T1)], ... })
  → compute_root_from_merkle_proof:
      pos=3 (odd)  → H(T2, T2)         ; pos=1
      pos=1 (odd)  → H(H(T0,T1),H(T2,T2)) = root
  → returns root → true ✓

Both calls return true. A bridge keyed on (tx_id, tx_index) pays out twice.
```

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
