### Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the depositor allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension` is intended to gate which addresses may deposit capital into a pool. Its `beforeAddLiquidity` hook receives both `sender` (the actual `msg.sender` of `addLiquidity`, i.e., the capital provider) and `owner` (the LP share recipient). The hook ignores `sender` entirely and only checks whether `owner` is allowlisted. Because `owner` is a caller-controlled parameter with no access restriction on `addLiquidity`, any non-allowlisted attacker can pass an allowlisted address as `owner`, satisfy the check, and inject their own capital into the pool.

### Finding Description
`pool.addLiquidity` accepts a free `owner` parameter and places no restriction on who may call it:

```solidity
// MetricOmmPool.sol
function addLiquidity(address owner, uint80 salt, ...) external nonReentrant(...) {
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    ...
}
``` [4](#0-3) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both `sender` (the real caller) and `owner` to the extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [5](#0-4) 

`DepositAllowlistExtension.beforeAddLiquidity` then silently drops `sender` and gates only on `owner`:

```solidity
function beforeAddLiquidity(address, address owner, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

An attacker who is not in `allowedDepositor[pool]` calls `pool.addLiquidity(owner=allowlistedAddr, ...)`. The check evaluates `allowedDepositor[pool][allowlistedAddr]` → `true`. The hook returns success. The pool then invokes `metricOmmModifyLiquidityCallback` on the attacker (the real `msg.sender`), pulling the attacker's tokens. LP shares are minted to `allowlistedAddr`. The allowlist gate is completely bypassed.

Note: the periphery `MetricOmmPoolLiquidityAdder` does call `_validateOwner(owner)` before forwarding, but this only prevents `owner=address(0)` — it does not prevent `owner` from being set to an arbitrary allowlisted third party. [6](#0-5)  The attacker bypasses the periphery entirely and calls the pool directly.

### Impact Explanation
The depositor allowlist — the sole mechanism for pool curation — is rendered ineffective. Any address can deposit capital into a restricted pool by nominating an allowlisted address as `owner`. Unauthorized capital enters the pool on every such call, breaking the admin-configured invariant that only approved depositors may provide liquidity. The allowlisted address receives unwanted LP shares; the attacker's capital is now inside the pool. This constitutes a broken admin-boundary and broken core pool functionality under the contest rules.

### Likelihood Explanation
The attack requires only a direct call to `pool.addLiquidity` with a known allowlisted address as `owner`. No privileged access, no special token behavior, and no off-chain data are needed. Any allowlisted address is publicly discoverable via `AllowedToDepositSet` events or `allowedDepositor` view calls. Likelihood is high.

### Recommendation
Check `sender` (the actual capital provider) instead of `owner` (the share recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```
