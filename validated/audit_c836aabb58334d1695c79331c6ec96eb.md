### Title
Fork Difficulty Validation Uses Mainchain Ancestor Timestamps Instead of Fork Ancestor Timestamps — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When validating the difficulty of a submitted fork block, the contract's `get_next_work_required` function fetches the boundary block of the difficulty period via `get_header_by_height`, which unconditionally reads from `mainchain_height_to_header`. During a fork submission, the fork's true ancestor at that height is a different block stored only in `headers_pool`. The contract therefore computes `expected_bits` using the mainchain block's timestamp rather than the fork's actual ancestor's timestamp — a direct cross-module desynchronization between the fork-submission path and the difficulty-validation path.

---

### Finding Description

The difficulty adjustment calculation for all three non-Zcash chains (Bitcoin, Litecoin, Dogecoin) requires the timestamp of the block at the **start** of the current difficulty period. This is fetched via `get_header_by_height`:

**Bitcoin** (`contract/src/bitcoin.rs`, lines 78–86):
```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [1](#0-0) 

**Litecoin** (`contract/src/litecoin.rs`, lines 86–88):
```rust
let first_block_height = prev_block_header.block_height - blocks_to_go_back;
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [2](#0-1) 

**Dogecoin** (`contract/src/dogecoin.rs`, lines 286–295) — the developers themselves flagged this with a TODO:
```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [3](#0-2) 

`get_header_by_height` is implemented as:
```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [4](#0-3) 

`mainchain_height_to_header` is only updated when a block is promoted to the mainchain. Fork blocks are stored only in `headers_pool` via `store_fork_header`: [5](#0-4) 

Therefore, when `check_pow` is called for a fork block whose chain diverged before the difficulty period boundary, `get_header_by_height(first_block_height)` returns the **mainchain** block at that height, not the fork's true ancestor. The two blocks can have different timestamps, producing a different `expected_bits` than the real Bitcoin network would compute for the fork.

The exact check that enforces the computed value:
```rust
require!(
    expected_bits == block_header.bits,
    "bad-diffbits: incorrect proof of work"
);
``` [6](#0-5) 

---

### Impact Explanation

The desynchronization corrupts the difficulty gate for fork blocks in two directions:

1. **Attacker-favorable (easier target accepted):** If the mainchain's boundary block has a later timestamp than the fork's true ancestor at the same height, `actual_time_taken` is computed as smaller, yielding a harder (lower) `expected_bits`. But if the mainchain's boundary block has an *earlier* timestamp, `actual_time_taken` is larger, yielding an easier (higher target) `expected_bits`. An attacker who controls the fork's ancestor timestamps can engineer the contract to compute an easier `expected_bits` than the real network would require. They then mine fork blocks satisfying this easier target, which the contract accepts as valid. Because chainwork is accumulated from the submitted `bits` field (`work_from_bits(block_header.bits)`), blocks with an artificially easy difficulty carry less chainwork per block — but the attacker can compensate by submitting more blocks, eventually triggering a chain reorganization to a weaker fork. [7](#0-6) 

2. **Denial of valid fork:** If the desynchronization causes the contract to compute a harder `expected_bits` than the real network, a legitimately mined fork block is rejected, preventing the contract from tracking a real Bitcoin chain reorganization.

Both outcomes corrupt the canonical chain mapping maintained by `mainchain_height_to_header` / `mainchain_tip_blockhash`, which is the root of trust for `verify_transaction_inclusion` and `verify_transaction_inclusion_v2`. [8](#0-7) 

---

### Likelihood Explanation

- The trigger condition — a fork that diverges before a difficulty adjustment boundary — is a normal operational scenario for any chain reorganization that spans a retarget epoch.
- The attacker-controlled entry point is `submit_blocks`, callable by any trusted relayer (or any account if the trusted-relayer gate is bypassed via `Role::UnrestrictedSubmitBlocks`).
- The attacker does not need privileged access beyond the ability to submit block headers; they only need to mine a fork chain whose boundary-block ancestor has a timestamp that differs from the mainchain's block at the same height.
- The Dogecoin module's own TODO comment confirms the developers recognized this as an open correctness question, indicating it has not been resolved.

---

### Recommendation

Replace the `get_header_by_height` call in all three `get_next_work_required` implementations with an ancestor-walk that follows `prev_block_hash` links through `headers_pool` starting from `prev_block_header`, descending exactly `blocks_to_go_back` steps. This ensures the boundary block is always the fork's true ancestor, not the mainchain block at the same height. The `get_prev_header` helper already supports this traversal: [9](#0-8) 

---

### Proof of Concept

1. Mainchain has blocks at heights 0–2016 (Bitcoin). Block at height 0 has timestamp `T0_main`.
2. Attacker submits a fork starting at height 0 with a genesis block whose timestamp is `T0_fork ≠ T0_main`. Fork blocks 1–2015 are submitted (all stored in `headers_pool` only, not in `mainchain_height_to_header`).
3. Attacker submits fork block at height 2016. `check_pow` calls `get_next_work_required`, which calls `get_header_by_height(0)`. This returns the **mainchain** block with timestamp `T0_main`.
4. `calculate_next_work_required` computes `actual_time_taken` using `T0_main` instead of `T0_fork`. If `T0_main < T0_fork`, the computed target is easier than the real network requires.
5. The attacker has pre-mined fork block 2016 with `bits` matching the contract's (incorrect) `expected_bits`. The block passes `expected_bits == block_header.bits` and the PoW hash check.
6. If the fork accumulates sufficient chainwork, `reorg_chain` is triggered, replacing the mainchain mapping with the attacker's fork — corrupting all subsequent `verify_transaction_inclusion` results. [10](#0-9)
