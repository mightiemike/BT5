### Title
Internal-Node Hash Accepted as Valid Transaction by `verify_transaction_inclusion` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) performs no check that the supplied `tx_id` is a leaf node of the Merkle tree. Any unprivileged NEAR caller can supply an internal node hash with a shortened proof (one level shorter than the real tree depth) and receive `true` in return, enabling a forged transaction-inclusion claim.

---

### Finding Description

`verify_transaction_inclusion` is publicly callable — it carries `#[pause]` but no `#[private]`, no `#[trusted_relayer]`, and no role guard. Its only Merkle check is: [1](#0-0) 

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

`compute_root_from_merkle_proof` is a pure hash-chain computation with no notion of tree depth or leaf-vs-internal-node distinction: [2](#0-1) 

For a 4-transaction tree `[A, B, C, D]`:

```
        root = SHA256d(AB || CD)
       /                        \
  AB = SHA256d(A||B)       CD = SHA256d(C||D)
    /     \                   /     \
   A       B                 C       D
```

An attacker who knows `AB` and `CD` (both are computable from the public block) can call:

- `tx_id    = AB`
- `tx_index = 0`
- `merkle_proof = [CD]`

`compute_root_from_merkle_proof(AB, 0, [CD])` computes `SHA256d(AB || CD) = root`, which equals `header.block_header.merkle_root`. The function returns `true`.

The function's own docstring acknowledges this: [3](#0-2) 

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."

The `#[deprecated]` Rust attribute is a **compiler warning only** — it does not prevent runtime calls. The function remains fully reachable on-chain. [4](#0-3) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` and acts on a `true` result (e.g., releasing funds, minting tokens, unlocking a bridge transfer) can be deceived into accepting a hash that was never a real Bitcoin transaction. The attacker needs only:

1. A canonical block with ≥ 2 transactions stored in the light client.
2. Knowledge of the block's transaction hashes (all public on-chain data).
3. The ability to call a public NEAR contract method.

No privileged role, no key compromise, and no special tooling are required.

---

### Likelihood Explanation

- The function is public and callable by any NEAR account.
- All inputs needed to construct the forged proof are publicly available from the Bitcoin blockchain.
- The attack is deterministic and requires no brute-force or probabilistic steps.
- Any integrator that has not yet migrated from v1 to v2 is immediately exposed.

---

### Recommendation

1. **Remove `verify_transaction_inclusion` (v1) entirely**, or add a hard `env::panic_str` body that unconditionally aborts, so it can no longer return a meaningful result.
2. If removal is not immediately possible, add a proof-depth guard: require `merkle_proof.len()` to equal `ceil(log2(tx_count))`, rejecting proofs that are shorter than the full tree depth.
3. Enforce migration to `verify_transaction_inclusion_v2`, which requires a coinbase proof of equal depth, preventing the level-skipping attack.

---

### Proof of Concept

```rust
// 4-tx Merkle tree: [A, B, C, D]
// AB = SHA256d(A || B),  CD = SHA256d(C || D)
// root = SHA256d(AB || CD)  <-- stored as block's merkle_root

let ab = compute_hash(&tx_a, &tx_b);   // internal node
let cd = compute_hash(&tx_c, &tx_d);   // internal node

// Attacker calls verify_transaction_inclusion with:
//   tx_id    = ab          (internal node, NOT a real tx)
//   tx_index = 0
//   merkle_proof = [cd]    (1-element proof, tree depth is 2)
//
// compute_root_from_merkle_proof(ab, 0, [cd])
//   => SHA256d(ab || cd) == root  => returns true
assert!(compute_root_from_merkle_proof(ab.clone(), 0, &vec![cd]) == root);

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
