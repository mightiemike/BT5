### Title
`verify_transaction_inclusion` Lacks Coinbase Merkle Proof Validation Present in `verify_transaction_inclusion_v2`, Enabling Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function remains callable by any unprivileged NEAR account and omits the coinbase Merkle proof validation that was added in `verify_transaction_inclusion_v2` specifically to close the 64-byte transaction Merkle proof forgery vulnerability. An attacker can supply a crafted `tx_id` that is an internal Merkle tree node rather than a real transaction hash and receive a `true` result, forging proof of a transaction that was never included in the block.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to fix a known class of Merkle proof forgery. It enforces two guards that the v1 function does not:

1. **Equal-length proof arrays** — `merkle_proof.len() == coinbase_merkle_proof.len()` ensures the claimed transaction sits at the same tree depth as the coinbase.
2. **Coinbase Merkle proof validation** — the coinbase transaction (index 0) is independently verified against the block's `merkle_root` before the target transaction proof is checked. [1](#0-0) 

The v1 function performs neither check: [2](#0-1) 

The code itself acknowledges the gap with an explicit warning:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

Despite being marked `#[deprecated]`, the function carries no access restriction beyond the shared `#[pause]` guard and is fully reachable by any unprivileged caller. [4](#0-3) 

---

### Impact Explanation

Any dApp or bridge that calls `verify_transaction_inclusion` to gate a high-value action (token release, cross-chain settlement, etc.) can be deceived into accepting a forged proof. The function returns `bool`, so a `true` result is treated as authoritative confirmation of transaction inclusion. The corrupted proof result is the exact state delta: a non-existent transaction is certified as present in a canonical Bitcoin block.

---

### Likelihood Explanation

The 64-byte transaction Merkle proof forgery is a well-documented, publicly known attack (referenced in the contract's own deprecation notice and at `https://www.bitmex.com/blog/64-Byte-Transactions`). The entry point requires no privilege — any NEAR account can call `verify_transaction_inclusion` with a borsh-serialized `ProofArgs`. The attacker only needs a real block hash present in the mainchain and a crafted 64-byte internal node as `tx_id`.

---

### Recommendation

Remove `verify_transaction_inclusion` entirely from the contract, or add a hard panic (`env::panic_str`) inside it so it is unreachable at runtime regardless of the `#[deprecated]` annotation. All callers must migrate to `verify_transaction_inclusion_v2`. [5](#0-4) 

---

### Proof of Concept

1. Identify a real block hash `B` present in `mainchain_header_to_height` (readable via `get_block_hash_by_height`).
2. Retrieve the block's `merkle_root` from `headers_pool`.
3. Construct a 64-byte value `N` that, when hashed with a chosen sibling set, produces the block's `merkle_root`. (Standard 64-byte forgery construction — the 64-byte blob is interpreted as two 32-byte child hashes by the Merkle verifier.)
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N`
   - `tx_block_blockhash = B`
   - `tx_index` and `merkle_proof` matching the forged path
   - `confirmations = 1`
5. The function computes `compute_root_from_merkle_proof(N, index, proof)`, which equals `merkle_root`, and returns `true` — certifying a transaction that does not exist. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
