### Title
Zero-Amount Token Transfer in `submitFastWithdrawal` and `handleWithdrawTransfer` Blocks Withdrawals for Non-Standard Tokens — (File: `core/contracts/BaseWithdrawPool.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal` calls `safeTransferFrom(token, msg.sender, uint128(fee))` without checking whether `fee > 0`. When `fastWithdrawalFeeAmount` returns zero (reachable for dust-amount withdrawals with `withdrawFeeX18 = 0`), this zero-value `transferFrom` reverts on tokens that reject zero-value transfers, permanently blocking that fast-withdrawal submission. Separately, `Clearinghouse.handleWithdrawTransfer` and `BaseWithdrawPool.handleWithdrawTransfer` both call `token.safeTransfer(…, uint256(amount))` without a zero-amount guard, exposing the same revert path when a slow-mode `WithdrawCollateral` with `amount = 0` is processed.

---

### Finding Description

**Instance 1 — `BaseWithdrawPool.submitFastWithdrawal` (line 108)** [1](#0-0) 

When `sendTo != msg.sender`, the submitter pays the fee via:

```solidity
safeTransferFrom(token, msg.sender, uint128(fee));
``` [2](#0-1) 

`fee` is computed by `fastWithdrawalFeeAmount`: [3](#0-2) 

For a 6-decimal token with `withdrawFeeX18 = 0`:
- `multiplier = 10^12`
- `proportionalFeeX18 = 0.001 × amount × 10^12`
- `feeX18 / multiplier = 0` when `amount < 1000` (i.e., `< $0.001`)

So `fee = 0` is reachable. `safeTransferFrom(token, msg.sender, 0)` is then called with no zero-guard. For any collateral token that reverts on zero-value transfers, the entire `submitFastWithdrawal` call reverts.

**Instance 2 — `Clearinghouse.handleWithdrawTransfer` (line 383) and `BaseWithdrawPool.handleWithdrawTransfer` (line 189)** [4](#0-3) [5](#0-4) 

Neither checks `amount > 0` before calling `token.safeTransfer`. A user can submit a slow-mode `WithdrawCollateral` with `amount = 0` via `submitSlowModeTransaction`: [6](#0-5) 

The slow-mode fee is charged unconditionally: [7](#0-6) 

When the queued transaction is later executed, `withdrawCollateral` calls `handleWithdrawTransfer(token, sendTo, 0, idx)`, which calls `token.safeTransfer(withdrawPool, 0)`. For a non-standard token, this reverts. The revert is silently swallowed by `_executeSlowModeTransaction`'s try/catch: [8](#0-7) 

The user's collateral balance is never updated (the `spotEngine.updateBalance` call after `handleWithdrawTransfer` is never reached), but the $1 slow-mode fee is permanently consumed. [9](#0-8) 

---

### Impact Explanation

**Instance 1**: Any third-party submitter calling `submitFastWithdrawal` for a dust-amount withdrawal (fee rounds to 0) against a non-standard collateral token will have their call revert. The fast-withdrawal path is blocked for that `(idx, token)` combination until the user submits it themselves (`sendTo == msg.sender`). Time-sensitive fast withdrawals are delayed.

**Instance 2**: A user submitting a zero-amount slow-mode withdrawal against a non-standard token loses their $1 `SLOW_MODE_FEE` with no recourse. The withdrawal is silently dropped, the `markedIdxs[idx]` slot is not consumed (since `submitWithdrawal` is never reached), but the fee is already collected and non-refundable. [10](#0-9) 

---

### Likelihood Explanation

The Nado protocol supports multiple collateral tokens configured via `SpotEngine`. Any token added with `withdrawFeeX18 = 0` and non-standard zero-transfer behavior triggers Instance 1 for dust amounts. Instance 2 is reachable by any user who submits a zero-amount slow-mode withdrawal — no special privilege required. Both paths are externally reachable without sequencer or owner involvement.

---

### Recommendation

Add zero-amount guards before all token transfers:

```solidity
// In submitFastWithdrawal (BaseWithdrawPool.sol)
if (fee > 0) {
    safeTransferFrom(token, msg.sender, uint128(fee));
}

// In handleWithdrawTransfer (BaseWithdrawPool.sol and Clearinghouse.sol)
if (amount > 0) {
    token.safeTransfer(to, uint256(amount));
}
```

Also add a `require(amount > 0)` guard at the entry of `withdrawCollateral` in `Clearinghouse.sol` to reject zero-amount withdrawals before any fee or transfer logic executes.

---

### Proof of Concept

**Instance 1 (fast withdrawal DoS):**
1. A collateral token `T` that reverts on zero-value transfers is listed with `withdrawFeeX18 = 0`.
2. User signs a `WithdrawCollateral` transaction for `amount = 500` (< 1000 units, i.e., < $0.001 for 6-decimal token).
3. Third party calls `submitFastWithdrawal(idx, transaction, signatures)` with `sendTo != msg.sender`.
4. `fastWithdrawalFeeAmount` returns `0` (integer division truncates).
5. `safeTransferFrom(T, msg.sender, 0)` is called — token `T` reverts.
6. The fast withdrawal is blocked; the user must wait for the slow-mode

    function fastWithdrawalFeeAmount(
        IERC20Base token,
        uint32 productId,
        uint128 amount
    ) public view returns (int128) {
        uint8 decimals = token.decimals();
        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
        int128 amountX18 = int128(amount) * int128(multiplier);

        int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
        int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;

        int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
        return feeX18 / int128(multiplier);
    }
