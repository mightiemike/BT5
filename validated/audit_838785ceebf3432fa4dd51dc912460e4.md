### Title
ML-DSA-65 gas-key host-function fee uses borsh key length instead of trie identifier length, causing ~59× overcharge vs transaction path — (`runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The host-function implementations of `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` pass the raw caller-supplied `public_key_len` (the borsh-encoded byte length of the public key) to `gas_key_transfer_send_fee`, `gas_key_transfer_exec_fee`, and `gas_key_add_key_exec_fee`. For ML-DSA-65 keys this value is **1953 bytes** (1 tag + 1952 key bytes). The transaction path for the same actions calls those same fee helpers with `action.public_key.trie_id_len()`, which for ML-DSA-65 is **33 bytes** (1 tag + 32 SHA3-256 hash bytes). The two paths diverge by a factor of ~59× for every ML-DSA-65 gas-key action issued from a contract.

---

### Finding Description

`PublicKey::trie_id_len()` was introduced precisely because ML-DSA-65 access keys are stored in the trie by their 32-byte hash, not by the 1952-byte raw key. The storage-fee and gas-fee paths in the runtime were updated to call `trie_id_len()` instead of `len()` everywhere that touches trie-byte-priced costs.

**Transaction path** (`runtime/runtime/src/config.rs`, lines 114–116 and 347–353):

```rust
TransferToGasKey(action) => {
    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.trie_id_len())
        .total()
}
// ...
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
```

**Host-function path** (`runtime/near-vm-runner/src/logic/logic.rs`, lines 3091–3096):

```rust
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← raw borsh length, not trie_id_len()
);
```

The same pattern appears in the wasmtime runner (`runtime/near-vm-runner/src/wasmtime_runner/logic.rs`, lines 3323–3325 and 3395–3400) and for `promise_batch_action_add_gas_key_with_function_call` (lines 3499–3504 / logic.rs 3226–3231).

For ED25519 and SECP256K1 the two lengths coincide (`len() == trie_id_len()`), so the discrepancy is invisible today. Once `PostQuantumSignatures` activates (protocol version 85), any contract that issues a gas-key action with an ML-DSA-65 public key via a host function is charged:

| Metric | Transaction path | Host-function path |
|---|---|---|
| `public_key_len` fed to fee helpers | 33 bytes (`trie_id_len()`) | 1953 bytes (borsh `len()`) |
| Ratio | 1× | ~59× |

The `gas_key_transfer_exec_fee` helper computes `trie_key_len = access_key_key_len(account_id_len, public_key_len)` and multiplies by `gas_key_byte` exec rate. Feeding 1953 instead of 33 inflates every per-byte exec-gas charge by ~59×. The `gas_key_transfer_send_fee` helper multiplies `gas_key_byte` send rate by `public_key_len` directly, producing the same ~59× overcharge on the send side.

The existing fee-parity integration test (`test_gas_key_fee_parity` in `test-loop-tests/src/tests/gas_keys.rs`) only exercises ED25519 keys, so the divergence for ML-DSA-65 is not caught.

---

### Impact Explanation

Any on-chain contract that calls `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key after `PostQuantumSignatures` activates will burn ~59× more gas than the equivalent transaction-path operation. This:

1. **Breaks the fee-parity protocol invariant** that the same action costs the same gas whether submitted as a transaction or dispatched from a contract.
2. **Makes ML-DSA-65 gas-key operations via host functions economically infeasible** — a contract budgeted for the transaction-path cost will run out of gas.
3. **Is unprivileged-user-controlled**: any deployed contract can trigger the overcharge by passing a valid ML-DSA-65 borsh-encoded public key (1953 bytes) to the host function.

The exact divergent value is `1953 - 33 = 1920` extra bytes fed to every per-byte fee multiplier per gas-key host-function call.

---

### Likelihood Explanation

The condition is reachable as soon as `PostQuantumSignatures` is enabled (protocol version 85) and a contract issues a gas-key action with an ML-DSA-65 key. No privileged role is required; any unprivileged account can deploy such a contract. The discrepancy is silent — the host function succeeds, the action is appended, and the overcharge is only visible in `gas_burnt`.

---

### Recommendation

In all three host-function implementations, decode the public key first and use `trie_id_len()` on the decoded key when calling the fee helpers, mirroring the transaction path:

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_pk = public_key.decode()?;
let pk_fee_len = decoded_pk.trie_id_len();

let send = gas_key_transfer_send_fee(&self.fees_config, sir, pk_fee_len);
let exec = gas_key_transfer_exec_fee(&self.fees_config, receiver_id.len(), pk_fee_len);
```

Apply the same fix to `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call` (both `logic.rs` and `wasmtime_runner/logic.rs`). Extend `test_gas_key_fee_parity` to cover ML-DSA-65 keys.

---

### Proof of Concept

**Exact divergent values for a `TransferToGasKey` with an ML-DSA-65 key:**

- `public_key_len` as seen by the host function: `1 + 1952 = 1953` bytes (borsh tag + raw key)
- `trie_id_len()` used by the transaction path: `1 + 32 = 33` bytes (borsh tag + SHA3-256 hash)
- Ratio: `1953 / 33 ≈ 59.2×`

**Fee formula** (`gas_key_transfer_exec_fee`, `core/parameters/src/cost.rs` lines 833–846):

```
trie_key_len = COLUMN_PREFIX(1) + account_id_len + SEPARATOR(1) + public_key_len
per_byte_exec = gas_key_byte_exec_rate × (trie_key_len + AccessKey::min_gas_key_borsh_len())
```

With `account_id_len = 10` (e.g. `"alice.near"`):
- Transaction path: `trie_key_len = 1 + 10 + 1 + 33 = 45`
- Host-function path: `trie_key_len = 1 + 10 + 1 + 1953 = 1965`

The exec per-byte component is inflated by `(1965 + value_len) / (45 + value_len)` ≈ 43× before the `gas_key_byte` rate is applied. The send per-byte component is inflated by `1953 / 33 ≈ 59×`.

```rust
pub fn gas_key_transfer_send_fee(
    cfg: &RuntimeFeesConfig,
    sender_is_receiver: bool,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).send_fee(sender_is_receiver);
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .send_fee(sender_is_receiver)
        .checked_mul(public_key_len as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
```

```rust
    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
```
