### Title
Merkle Proof Length Drives Loop Termination Instead of Transaction Position Bits, Enabling Proof Forgery via Shortened Branch — (`merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` terminates based on the length of the attacker-supplied `merkle_proof` slice rather than the bit-width of `transaction_position`. The still-active (non-gated) `verify_transaction_inclusion` endpoint passes all three of `tx_id`, `tx_index`, and `merkle_proof` directly from caller-supplied arguments into this function without validating that `merkle_proof.len()` equals the tree depth implied by `tx_index`. An unprivileged NEAR caller can supply an internal Merkle tree node as `tx_id` together with a proof one level shorter than the real tree depth, causing the verifier to accept the internal node as a proven leaf and return `true` for a transaction that does not exist.

---

### Finding Description

**Root cause — `merkle-tools/src/lib.rs` lines 42–49:**

```rust
for proof_hash in merkle_proof {          // terminates at merkle_proof.len()
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
```

The loop runs exactly `merkle_proof.len()` times. There is no check that this length equals `floor(log2(tx_index + 1))` or any other value derived from `transaction_position`. [1](#0-0) 

**Reachable entry point — `contract/src/lib.rs` lines 288–323:**

`verify_transaction_inclusion` (v1) is a `pub` NEAR method decorated only with `#[pause]` and `#[deprecated]`. The `#[deprecated]` Rust attribute emits a compile-time warning; it does not gate the method at runtime. Any unprivileged NEAR account can call it directly. [2](#0-1) 

The only proof-related guard inside v1 is:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

No check constrains `merkle_proof.len()` to the depth of the tree. [3](#0-2) 

All three forgery inputs — `tx_id`, `tx_index`, `merkle_proof` — come directly from `ProofArgs`, which is fully caller-controlled: [4](#0-3)

### Impact Explanation

Consider a Bitcoin block with four transactions `tx0, tx1, tx2, tx3` and tree depth 2:

```
merkle_root = H(H(tx0, tx1), H(tx2, tx3))
node_01     = H(tx0, tx1)
node_23     = H(tx2, tx3)
```

An attacker calls `verify_transaction_inclusion` with:

| Field | Value |
|---|---|
| `tx_id` | `node_01` (an internal node, not a real txid) |
| `tx_index` | `0` |
| `merkle_proof` | `[node_23]` (length 1; correct depth is 2) |

The function executes one iteration:

```
position=0 → current_hash = H(node_01, node_23) = merkle_root  ✓
```

The function returns `true`, asserting that `node_01` is a confirmed transaction in the block. `node_01` is not a transaction; it is an intermediate hash. Any consumer contract that calls `verify_transaction_inclusion` and acts on a `true` result — e.g., releasing bridged funds, crediting a deposit, or unlocking an asset — can be deceived into processing a fabricated event. The attacker needs no privileged role and no control over the Bitcoin chain; all required values (`node_01`, `node_23`) are computable from the public block data of any already-confirmed block.

---

### Likelihood Explanation

- The attack requires only a block with more than one transaction (i.e., virtually every Bitcoin block).
- All inputs are computable from public chain data; no mining or key material is needed.
- `verify_transaction_inclusion` remains callable on-chain. Consumer contracts that integrated before v2 existed, or that call v1 directly, are exposed.
- The contract's own warning acknowledges the internal-node risk but delegates responsibility to callers, creating a systemic assumption gap.

---

### Recommendation

**Short term:** Add a length check in `compute_root_from_merkle_proof` that asserts `merkle_proof.len()` equals the number of bits needed to represent `transaction_position` (i.e., `floor(log2(transaction_position + 1))` for a standard Bitcoin Merkle tree, or equivalently rewrite the loop to terminate when `current_position` reaches 0 or 1 rather than when the proof slice is exhausted):

```rust
// Terminate on position, not on proof length
let mut current_position = transaction_position + 1; // generalized index
for proof_hash in merkle_proof {
    require!(current_position > 1, "proof longer than tree depth");
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
require!(current_position == 1, "proof shorter than tree depth");
```

**Long term:** Remove `verify_transaction_inclusion` (v1) entirely from the on-chain interface, or gate it with an explicit access-control role so it cannot be called by unprivileged accounts. All callers should migrate to `verify_transaction_inclusion_v2`, which indirectly constrains proof length through the coinbase check.

### Proof of Concept

Given any mainchain Bitcoin block with ≥ 2 transactions (e.g., block at height H with `merkle_root` R):

1. Fetch the block's transaction list. Compute `node_01 = double_sha256(tx0 || tx1)` and `node_23 = double_sha256(tx2 || tx3)` (or the appropriate sibling for the chosen subtree).
2. Call `verify_transaction_inclusion` on the NEAR contract:
   ```json
   {
     "tx_id":             "<node_01 hex>",
     "tx_block_blockhash":"<block hash hex>",
     "tx_index":          0,
     "merkle_proof":      ["<node_23 hex>"],
     "confirmations":     1
   }
   ```
3. The contract returns `true`. The supplied `tx_id` is an internal Merkle node, not a real transaction. The proof has length 1 instead of the correct depth 2. The broken invariant is that `compute_root_from_merkle_proof` accepted a proof whose length is not equal to the depth of the leaf in the tree. [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
