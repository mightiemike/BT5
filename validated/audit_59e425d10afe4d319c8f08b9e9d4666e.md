Audit Report

## Title
Missing Minimum Timelock Validation in `initialize()` Allows Pool Admin to Bypass LP-Protection Timelock — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension.initialize()` validates `drawdownE6` and `decayPerSecondE8` but applies no lower bound to the `timelock` parameter, silently accepting `timelock = 0`. With a zero timelock, the pool admin can propose and execute any stop-loss parameter change — including setting `drawdownE6 = 0` to fully disable the circuit-breaker — atomically in a single transaction, voiding the LP-protection guarantee the contract's own NatSpec promises.

## Finding Description

**Root cause — missing validation in `initialize`:**

`initialize()` decodes three fields and validates only the first two:

```solidity
// Lines 56–62
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);    // enforces drawdownE6 <= 1e6
_validateDecay(decayPerSecondE8); // enforces decayPerSecondE8 <= 1e8
// ← no _validateTimelock; timelock = 0 accepted silently
oracleStopLossConfig[pool] = PoolStopLossConfig({
    drawdownE6: drawdownE6, decayPerSecondE8: decayPerSecondE8, timelock: timelock, initialized: true
});
```

**How the timelock guard collapses to a no-op:**

`_afterTimelock` computes `block.timestamp + 0 = block.timestamp` when `timelock = 0`:

```solidity
// Lines 297–299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`_requireElapsed` then checks `block.timestamp < block.timestamp`, which is always `false`, so the revert never fires:

```solidity
// Lines 301–303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

Every timelocked setter — `proposeOracleStopLossDrawdown` / `executeOracleStopLossDrawdown`, `proposeOracleStopLossDecay` / `executeOracleStopLossDecay`, and `proposeOracleStopLossHighWatermarks` / `executeOracleStopLossHighWatermarks` — routes through these two helpers, so all become instant.

**Exploit path — disabling the stop-loss entirely:**

```
proposeOracleStopLossDrawdown(pool, 0)  // executeAfter = block.timestamp
executeOracleStopLossDrawdown(pool)     // block.timestamp < block.timestamp == false → passes
```

After execution, `oracleStopLossConfig[pool].drawdownE6 == 0`. The `afterSwap` hook short-circuits at line 217:

```solidity
// Lines 215–217
PoolStopLossConfig memory cfg = oracleStopLossConfig[pool_];
uint256 drawdown = cfg.drawdownE6;
if (drawdown == 0) return;   // all stop-loss checks bypassed for every subsequent swap
```

All subsequent swaps bypass the stop-loss; LP value can be drained freely with no circuit-breaker.

**`proposeOracleStopLossTimelock` also lacks validation**, so a pool admin can also propose a new timelock of `0` (line 78–83) without any lower-bound check, compounding the issue.

## Impact Explanation

The `OracleValueStopLossExtension` is the primary on-chain mechanism protecting LP principal from value drain. Its NatSpec states: *"Drawdown and decay changes are timelocked so LPs can react."* With `timelock = 0` that guarantee is void. The pool admin can:

1. **Disable the stop-loss entirely** (`drawdownE6 → 0`) in one atomic transaction, then execute a large swap that drains LP value with no circuit-breaker.
2. **Maximize decay** (`decayPerSecondE8 → 1e8`) instantly, collapsing all watermarks to zero within one second and removing every per-bin threshold.
3. **Zero out watermarks** for any bin instantly, resetting the high-water reference so the next swap sees no breach regardless of actual value loss.

All three paths result in direct loss of LP principal. This is an admin-boundary break: the pool admin exceeds the timelock constraint the protocol explicitly imposes on them.

## Likelihood Explanation

Pool creation via `MetricOmmPoolFactory.createPool` is public — any caller can supply `extensionInitData` with `timelockSeconds = 0`. The `admin` field in `PoolParameters` becomes `poolAdmin[pool]` on the factory, so the pool creator is also the pool admin. No special privilege beyond pool creation is required. The missing check is a single omitted bounds validation analogous to the validated `drawdownE6` and `decayPerSecondE8` fields. A deployer who misreads the field as "number of blocks," omits it, or intentionally sets it to zero produces the same exploitable outcome. Once deployed, the pool admin can exploit the condition at any time without warning to LPs.

## Recommendation

Add a minimum timelock constant and enforce it in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MIN_TIMELOCK = 1 days; // 86400 seconds

function _validateTimelock(uint256 t) private pure {
    if (t < MIN_TIMELOCK) revert OracleStopLossTimelockTooShort(t);
}
```

Call `_validateTimelock(timelock)` alongside `_validateDrawdown` and `_validateDecay` in `initialize` (line 58), and again inside `proposeOracleStopLossTimelock` (line 81) before storing `newTimelock`.
