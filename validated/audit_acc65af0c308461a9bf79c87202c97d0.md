### Title
CEI Violation in `withdrawCollateral()`: Token Transfer Before Balance Update and Health Check — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral()` performs an external token transfer via `handleWithdrawTransfer()` before updating the subaccount's balance in `SpotEngine` and before executing the health check. This violates the Checks-Effects-Interactions (CEI) pattern and creates a cross-function reentrancy window. Because no reentrancy guard exists anywhere in the protocol, a recipient contract with a token transfer hook (e.g., ERC-777 `tokensReceived`) can re-enter the protocol through the publicly callable `Endpoint.executeSlowModeTransaction()` and execute a second withdrawal against the same, not-yet-decremented balance.

---

### Finding Description

In `Clearinghouse.withdrawCollateral()`, the execution order is:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` — external token transfer to `sendTo`
2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)` — balance decrement
3. **Line 413** — `spotEngine.assertUtilization(productId)` — utilization check
4. **Line 419** — `require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH)` — health check [1](#0-0) 

`handleWithdrawTransfer` first transfers tokens to `withdrawPool`, then calls `BaseWithdrawPool.submitWithdrawal`, which in turn calls `handleWithdrawTransfer(token, sendTo, amount)` — delivering tokens to `sendTo` before any state has been updated. [2](#0-1) [3](#0-2) 

There is **no reentrancy guard** anywhere in the protocol — confirmed by the absence of any `nonReentrant` modifier or `ReentrancyGuard` import across all production contracts.

The public entry point `Endpoint.executeSlowModeTransaction()` has no access control and no reentrancy protection: [4](#0-3) 

This function calls `_executeSlowModeTransaction` → `this.processSlowModeTransaction` → `EndpointTx.processSlowModeTransactionImpl`, which can dispatch a `WithdrawCollateral` slow mode transaction back into `Clearinghouse.withdrawCollateral`: [5](#0-4) 

---

### Impact Explanation

An attacker who controls a `sendTo` contract (possible via `WithdrawCollateralV2`, which allows an arbitrary `sendTo` address) and whose token supports transfer callbacks (ERC-777 `tokensReceived`) can:

1. Receive tokens in the callback while `sender`'s balance in `SpotEngine` is still at its pre-withdrawal value.
2. Call `Endpoint.executeSlowModeTransaction()` from within the callback to process a pre-staged slow mode `WithdrawCollateral` for the same subaccount.
3. The second `withdrawCollateral` call reads the original (not yet decremented) balance, passes the health check, and transfers a second full withdrawal amount.
4. After the callback returns, the first call's `spotEngine.updateBalance` runs, leaving the subaccount with a corrupted (doubly decremented) balance — but the attacker has already received double the collateral.

**Corrupted state delta**: `SpotEngine` balance for `sender` is decremented twice for a single actual balance, enabling theft of collateral equal to one full withdrawal amount. [6](#0-5) 

---

### Likelihood Explanation

- `WithdrawCollateralV2` allows the user to specify an arbitrary `sendTo` address, giving the attacker full control over the recipient contract.
- `Endpoint.executeSlowModeTransaction()` is unconditionally public with no caller restriction.
- The attacker only needs a slow mode withdrawal queued at least `SLOW_MODE_TX_DELAY` (3 days) in advance — a realistic setup.
- No reentrancy guard exists anywhere in the codebase.
- The attack is triggered entirely by an unprivileged user through supported contract entrypoints (`WithdrawCollateralV2` + `executeSlowModeTransaction`).

The prerequisite of a token with transfer callbacks (ERC-777 or similar) is the only constraint. If any listed collateral token supports such hooks, the vulnerability is directly exploitable.

---

### Recommendation

Move `handleWithdrawTransfer` to **after** all state updates and health checks in `withdrawCollateral`:

```solidity
function withdrawCollateral(...) public virtual onlyEndpoint {
    // 1. Checks
    require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
    require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
    ...

    // 2. Effects
    int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
    int128 amountRealized = -int128(amount) * int128(multiplier);
    spotEngine.updateBalance(productId, sender, amountRealized);
    spotEngine.assertUtilization(productId);
    require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
    emit ModifyCollateral(amountRealized, sender, productId);

    // 3. Interactions (last)
    handleWithdrawTransfer(token, sendTo, amount, idx);
}
```

Additionally, add OpenZeppelin's `ReentrancyGuardUpgradeable` to `Clearinghouse` and `Endpoint` and apply `nonReentrant` to `withdrawCollateral` and `executeSlowModeTransaction`.

---

### Proof of Concept

1. Attacker deploys a malicious contract `MaliciousRecipient` implementing ERC-777 `tokensReceived`.
2. Attacker submits a slow mode `WithdrawCollateral` for their subaccount via `Endpoint.submitSlowModeTransaction`. Waits 3+ days.
3. Attacker submits a signed `WithdrawCollateralV2` transaction with `sendTo = address(MaliciousRecipient)` to the sequencer.
4. Sequencer calls `Endpoint.submitTransactionsChecked` → `processTransactionImpl` → `Clearinghouse.withdrawCollateral`.
5. `handleWithdrawTransfer` → `token.safeTransfer(withdrawPool)` → `WithdrawPool.submitWithdrawal` → `token.safeTransfer(MaliciousRecipient, amount)`.
6. ERC-777 `tokensReceived` fires on `MaliciousRecipient`. Inside the hook, attacker calls `Endpoint.executeSlowModeTransaction()`.
7. Slow mode `WithdrawCollateral` is processed. `spotEngine.updateBalance` has not yet run for the first withdrawal, so the balance is still full. Health check passes. Second withdrawal transfers `amount` tokens to the attacker.
8. Callback returns. First withdrawal's `spotEngine.updateBalance` runs, decrementing the balance a second time.
9. Attacker has received `2 × amount` tokens while the subaccount balance reflects only one decrement — net theft of `amount` tokens from the protocol. [7](#0-6) [8](#0-7) [4](#0-3)
