### Title
Linked Signer Can Self-Replace via `LinkSigner` Transaction Due to `allowLinkedSigner=true` — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner=true`. This allows the **current linked signer** — not just the subaccount owner — to authorize a new `LinkSigner` transaction, permanently replacing themselves (or any address) as the linked signer. The invariant that only the subaccount owner should control the linked signer relationship is broken.

---

### Finding Description

In `EndpointTx.sol`, the fast-path handler for `LinkSigner` transactions is:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner=true` calls `validateSignature`, which passes `getLinkedSigner(sender)` as the accepted signer to `verifier.validateSignature`:

```solidity
// EndpointTx.sol lines 172–184
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if the recovered address equals **either** the subaccount owner **or** the linked signer:

```solidity
// Verifier.sol lines 297–303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means: if `linkedSigners[victimSubaccount] = attacker1`, then attacker1 can sign a `LinkSigner{sender: victimSubaccount, signer: attacker2}` transaction, and the on-chain validation will accept it, writing `linkedSigners[victimSubaccount] = attacker2`.

**Contrast with the slow-mode path**, which correctly enforces owner-only authorization for `LinkSigner`:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // msg.sender must be subaccount owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The slow-mode path uses `validateSender` (owner check), while the fast-path uses `allowLinkedSigner=true` — an inconsistency that creates the vulnerability.

---

### Impact Explanation

A compromised or malicious linked signer can:
1. Sign a `LinkSigner` transaction naming any attacker-controlled address as the new signer.
2. The new signer can then sign `WithdrawCollateral`, `TransferQuote`, and all other `allowLinkedSigner=true` transactions on behalf of the victim subaccount.
3. The original owner loses control unless they submit a corrective `LinkSigner` via the slow-mode path (which requires on-chain `msg.sender` ownership), but by then assets may already be drained.

The exact storage mutation is: `linkedSigners[victimSubaccount]` is overwritten from `attacker1` to `attacker2` without the subaccount owner's consent.

---

### Likelihood Explanation

Any party that has ever been granted linked signer access (e.g., a trading bot key, a compromised API key, a former delegate) can execute this attack. Linked signer keys are typically hot keys with weaker security than the subaccount owner key. The attack requires only a valid signed `LinkSigner` transaction submitted through the normal sequencer flow — no special privileges beyond already holding the linked signer key.

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only subaccount owner may change the linked signer
);
```

This aligns the fast-path with the slow-mode path's `validateSender` owner check and enforces the invariant that only the subaccount owner controls the linked signer relationship.

---

### Proof of Concept

```solidity
// 1. Setup: victim subaccount has attacker1 as linked signer
linkedSigners[victimSubaccount] = attacker1;

// 2. Attacker1 crafts and signs a LinkSigner tx replacing themselves with attacker2
IEndpoint.LinkSigner memory lsTx = IEndpoint.LinkSigner({
    sender: victimSubaccount,
    signer: bytes32(uint256(uint160(attacker2))),
    nonce: currentNonce
});
bytes memory sig = sign(attacker1PrivKey, EIP712digest(lsTx));

// 3. Sequencer includes it — validateSignedTx passes because attacker1 == getLinkedSigner(victimSubaccount)
// 4. State after: linkedSigners[victimSubaccount] == attacker2
assert(linkedSigners[victimSubaccount] == attacker2);

// 5. attacker2 now signs WithdrawCollateral on behalf of victimSubaccount — accepted
```
