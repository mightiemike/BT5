### Title
Reduce-Only Constraint Validated on Local Copy But Not Enforced at Execution, Allowing Position Flip — (`core/contracts/OffchainExchange.sol`)

---

### Summary

In `OffchainExchange.sol`, the reduce-only order constraint is evaluated inside `_validateOrder` on a **local copy** of the order amount. However, `matchOrders` executes the trade using the **original uncapped** `taker.order.amount`, meaning the reduce-only cap computed during validation is silently discarded. A user can sign a reduce-only order whose stated amount exceeds their current position size, pass on-chain validation, and have the full uncapped amount executed — flipping their position from long to short (or vice versa) in violation of the reduce-only invariant.

---

### Finding Description

In `_validateOrder`, when `_isReduceOnly(order.appendix)` is true, the function reads the current position and caps a local `order` variable:

```solidity
IEndpoint.Order memory order = signedOrder.order;   // local copy
...
if (_isReduceOnly(order.appendix)) {
    int128 amount = callState.isPerp
        ? callState.perp.getBalance(...).amount
        : callState.spot.getBalance(...).amount;
    if ((order.amount > 0) == (amount > 0)) {
        order.amount = 0;
    } else if (order.amount > 0) {
        order.amount = MathHelper.min(order.amount, -amount);
    } else if (order.amount < 0) {
        order.amount = MathHelper.max(order.amount, -amount);
    }
}
return ... (order.amount != 0) ...;
``` [1](#0-0) 

The capping is applied to `order`, a **local `memory` copy** of `signedOrder.order`. Because `order` is a freshly allocated struct (`IEndpoint.Order memory order = signedOrder.order`), writes to `order.amount` do not propagate back to `signedOrder.order` or to the caller's `taker` variable in `matchOrders`.

Back in `matchOrders`, after `_validateOrder` returns `true`, the execution path computes the fill delta directly from the **original, uncapped** `taker.order.amount`:

```solidity
if (taker.order.amount < 0) {
    ordersInfo.taker.amountDelta = MathHelper.max(
        taker.order.amount,
        -maker.order.amount
    );
} else if (taker.order.amount > 0) {
    ordersInfo.taker.amountDelta = MathHelper.min(
        taker.order.amount,
        -maker.order.amount
    );
}
``` [2](#0-1) 

The reduce-only cap computed in `_validateOrder` is never applied to `taker.order.amount` before this point. The result is that the fill can exceed the user's current position size, crossing zero and opening a position in the opposite direction — the exact outcome the reduce-only flag is designed to prevent.

---

### Impact Explanation

A user holding a long position of `N` units can sign a reduce-only sell order for `2N` units. `_validateOrder` caps the local copy to `N` and returns `true` (non-zero). `matchOrders` then executes for `2N` units (bounded only by the maker's available amount), flipping the user's position from `+N` to `-N`. This violates the reduce-only invariant: the position is not merely reduced — it is reversed. The post-trade health check (`require(isHealthy(...))`) only reverts if the resulting position is unhealthy; a healthy flipped position passes through, leaving the user with an unintended directional exposure. [3](#0-2) 

**Impact: MEDIUM** — Violates a core order-type invariant; can produce unintended position reversals and unexpected P&L exposure for the signing user.

---

### Likelihood Explanation

**Likelihood: MEDIUM** — The user controls the signed `amount` field and the `reduce-only` bit in `appendix`. No privileged role or sequencer compromise is required: the user submits a validly signed order; the sequencer forwards it to `matchOrders` as normal. The sequencer may rely on on-chain enforcement of the reduce-only cap rather than duplicating the check off-chain, making this reachable in practice.

---

### Recommendation

Apply the reduce-only cap to the **actual order amount used for execution**, not only to the local validation copy. The simplest fix is to propagate the capped amount back before computing `ordersInfo.taker.amountDelta`:

```solidity
// After _validateOrder, re-apply reduce-only cap to taker.order.amount
if (_isReduceOnly(taker.order.appendix)) {
    int128 pos = callState.isPerp
        ? callState.perp.getBalance(callState.productId, taker.order.sender).amount
        : callState.spot.getBalance(callState.productId, taker.order.sender).amount;
    if (taker.order.amount < 0)
        taker.order.amount = MathHelper.max(taker.order.amount, -pos);
    else if (taker.order.amount > 0)
        taker.order.amount = MathHelper.min(taker.order.amount, -pos);
}
```

Alternatively, refactor `_validateOrder` to return the capped amount so `matchOrders` can use it directly.

---

### Proof of Concept

1. User holds a perp long position: `balance.amount = +5`.
2. User signs a reduce-only sell order: `order.amount = -10`, `_isReduceOnly(appendix) == true`.
3. Sequencer calls `matchOrders` with a maker offering `-10` units.
4. `_validateOrder` (line 452–453): local `order.amount = MathHelper.max(-10, -5) = -5`; returns `true` (non-zero). [4](#0-3) 
5. `matchOrders` (line 745–749): `ordersInfo.taker.amountDelta = MathHelper.max(-10, -(-10)) = -10`. [5](#0-4) 
6. `_updateBalances` applies `baseDelta = -10` to the user's account. [6](#0-5) 
7. User's position is now `-5` — a full reversal — despite the reduce-only flag. The health check passes if the short position is within margin. [7](#0-6)
