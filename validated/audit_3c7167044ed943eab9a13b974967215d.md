### Title
Consensus-Path `convert_consensus_tx_to_internal_consensus_tx` Skips OS-Enforced DA-Mode and Field Constraints Enforced by Gateway `StatelessTransactionValidator` — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

The gateway admission path enforces several OS-level invariants (DA mode must be L1, paymaster data must be empty, account deployment data must be empty) via `StatelessTransactionValidator::validate()`. The consensus proposal-validation path — `handle_proposal_part` → `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal` — performs none of these checks. A malicious block proposer can inject transactions that violate these invariants directly through the consensus wire, bypassing every gateway guard. Validator nodes accept and forward them to the batcher, which executes them and commits the resulting state. Because the Starknet OS enforces the same invariants at prove-time, the committed block cannot be proved, producing wrong committed state.

---

### Finding Description

**Gateway path** — `GenericGateway::add_tx_inner` calls `self.stateless_tx_validator.validate(&tx)` before any conversion:

```
validate_nonce_data_availability_mode  // OS enforces L1 only
validate_fee_data_availability_mode    // OS enforces L1 only
validate_empty_account_deployment_data // OS enforces empty
validate_empty_paymaster_data          // OS enforces empty
validate_resource_bounds               // max_l2_gas_amount cap
validate_tx_size                       // calldata / signature / proof limits
```

Each of the first four is documented with the explicit rationale: *"The Starknet OS enforces … We add this validation here in the gateway to prevent transactions from failing the OS."*

**Consensus path** — `handle_proposal_part` calls:

```rust
transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
```

which dispatches to `convert_rpc_tx_to_internal`:

```rust
async fn convert_rpc_tx_to_internal(
    &self,
    tx: RpcTransaction,
) -> TransactionConverterResult<(InternalRpcTransaction, Option<(ProofFacts, Proof)>)> {
    let (tx_without_hash, proof_data) = match tx {
        RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => {
            // ← no DA-mode check, no paymaster check, no deployment-data check
            (InternalRpcTransactionWithoutTxHash::Invoke(tx.into()), proof_data)
        }
        RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => { /* compiled_class_hash only */ }
        RpcTransaction::DeployAccount(RpcDeployAccountTransaction::V3(tx)) => { /* address only */ }
    };
    let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
    Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
}
```

The `From<RpcInvokeTransactionV3> for InternalRpcInvokeTransactionV3` conversion faithfully copies the attacker-controlled DA-mode fields into the internal representation:

```rust
impl From<RpcInvokeTransactionV3> for InternalRpcInvokeTransactionV3 {
    fn from(tx: RpcInvokeTransactionV3) -> Self {
        Self {
            nonce_data_availability_mode: tx.nonce_data_availability_mode, // L2 preserved
            fee_data_availability_mode:   tx.fee_data_availability_mode,   // L2 preserved
            paymaster_data:               tx.paymaster_data,               // non-empty preserved
            account_deployment_data:      tx.account_deployment_data,      // non-empty preserved
            ...
        }
    }
}
```

The DA-mode value is then embedded in the transaction hash via `concat_data_availability_mode` and forwarded to the batcher as an `InternalConsensusTransaction` with no further sanitisation.

**Invariant broken**: the gateway enforces `nonce_data_availability_mode == L1` and `fee_data_availability_mode == L1` on every transaction it admits. The consensus path admits the same transaction type (`RpcTransaction`) with no such check, creating two code paths that both produce `InternalRpcTransaction` objects but only one of which enforces the OS constraint — the exact structural pattern of the veRAACToken `lock`/`increase` split.

---

### Impact Explanation

The Starknet OS enforces DA mode = L1 at prove-time. A block that contains a committed transaction with `nonce_data_availability_mode = L2` or `fee_data_availability_mode = L2` will produce a state diff the OS cannot prove, yielding wrong committed state and a broken block. Additionally, non-empty `paymaster_data` or `account_deployment_data` fields are OS-rejected for the same reason; their presence in an executed transaction corrupts the provable state. This maps to **Critical — wrong state/receipt/revert result from blockifier/execution logic for accepted input**.

---

### Likelihood Explanation

Any network participant who wins the block-proposer role (an unprivileged, rotating role in BFT consensus) can craft a `ConsensusTransaction::RpcTransaction` with `nonce_data_availability_mode = DataAvailabilityMode::L2`. No special privilege beyond being a proposer is required. The malicious transaction travels the consensus wire, is deserialized by every validator, and enters `handle_proposal_part` → `convert_consensus_tx_to_internal_consensus_tx` without any DA-mode gate.

---

### Recommendation

Apply the same OS-invariant checks inside `convert_rpc_tx_to_internal` (or in a dedicated consensus-path validator called before conversion) that `StatelessTransactionValidator::validate` applies in the gateway path. Concretely:

- Reject any `RpcTransaction` whose `nonce_data_availability_mode != DataAvailabilityMode::L1`.
- Reject any `RpcTransaction` whose `fee_data_availability_mode != DataAvailabilityMode::L1`.
- Reject any Invoke/Declare transaction with non-empty `paymaster_data`.
- Reject any Invoke/Declare transaction with non-empty `account_deployment_data`.

These checks are cheap, stateless, and already exist in `StatelessTransactionValidator`; they should be extracted into a shared function callable from both the gateway and the consensus converter.

---

### Proof of Concept

1. A malicious proposer constructs an `RpcInvokeTransactionV3` with `nonce_data_availability_mode = DataAvailabilityMode::L2` (wire value `1`) and a valid signature over the resulting hash (which includes the L2 DA-mode bits via `concat_data_availability_mode`).

2. The proposer wraps it as `ConsensusTransaction::RpcTransaction` and streams it to validator nodes inside a `ProposalPart::Transactions` batch.

3. Each validator calls:
   ```
   handle_proposal_part
     └─ transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
          └─ convert_rpc_tx_to_internal(tx)   // no DA-mode check here
               └─ InternalRpcInvokeTransactionV3 { nonce_data_availability_mode: L2, ... }
   ```
   No error is returned; the transaction is accepted.

4. The `InternalConsensusTransaction` is forwarded to the batcher via `send_txs_for_proposal`. The batcher executes it and commits the resulting state, which contains an L2-DA-mode nonce update.

5. At prove-time the Starknet OS rejects the transaction because it enforces `nonce_data_availability_mode == L1`, producing an unprovable block with wrong committed state.

    /// The Starknet OS enforces that the nonce data availability mode is L1. We add this validation
    /// here in the gateway to prevent transactions from failing the OS.
    fn validate_nonce_data_availability_mode(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let expected_da_mode = DataAvailabilityMode::L1;
        let da_mode = *tx.nonce_data_availability_mode();
        if da_mode != expected_da_mode {
            return Err(StatelessTransactionValidatorError::InvalidDataAvailabilityMode {
                field_name: "nonce".to_string(),
            });
        };

        Ok(())
    }

    /// The Starknet OS enforces that the fee data availability mode is L1. We add this validation

                let proof_data = if tx.proof_facts.is_empty() {

            RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => {
                let ClassHashes { class_hash, executable_class_hash_v2 } =

                    self.class_manager_client.add_class(tx.contract_class).await?;
                // TODO(Aviv): Ensure that we do not want to
                // allow declare with compiled class hash v1.
                if tx.compiled_class_hash != executable_class_hash_v2 {
                    return Err(TransactionConverterError::ValidateCompiledClassHashError(
                        ValidateCompiledClassHashError::CompiledClassHashMismatch {
                            computed_class_hash: executable_class_hash_v2,
                            supplied_class_hash: tx.compiled_class_hash,
                        },
                    ));
                }
                (
                    InternalRpcTransactionWithoutTxHash::Declare(InternalRpcDeclareTransactionV3 {
                        sender_address: tx.sender_address,
                        compiled_class_hash: tx.compiled_class_hash,
                        signature: tx.signature,
                        nonce: tx.nonce,
                        class_hash,
                        resource_bounds: tx.resource_bounds,
                        tip: tx.tip,
                        paymaster_data: tx.paymaster_data,
                        account_deployment_data: tx.account_deployment_data,
                        nonce_data_availability_mode: tx.nonce_data_availability_mode,
                        fee_data_availability_mode: tx.fee_data_availability_mode,
                    }),
                    None,
                )
            }
            RpcTransaction::DeployAccount(RpcDeployAccountTransaction::V3(tx)) => {
                let contract_address = tx.calculate_contract_address()?;
                (
                    InternalRpcTransactionWithoutTxHash::DeployAccount(
                        InternalRpcDeployAccountTransaction {
                            tx: RpcDeployAccountTransaction::V3(tx),
                            contract_address,
                        },
                    ),
                    None,
                )
            }
        };
        let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
        Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
