### Title
`limbs_for_int` Mixes Unsigned Bit-Count Bytes With CLVM Signed-Encoding Lengths, Causing Systematic Cost Undercharging in `op_ash`, `op_lsh`, and `op_multiply` — (File: `src/more_ops.rs`)

---

### Summary

`limbs_for_int` computes the byte size of a `Number` using `v.bits().div_ceil(8)`, which yields the **unsigned** byte count. CLVM, however, uses **signed** big-endian encoding: any positive integer whose most-significant bit is 1 requires a leading `0x00` byte, making its actual atom length one byte larger than `limbs_for_int` returns. This mismatch is the direct analog of the external report's "different decimal precisions added without normalization" class: two quantities measured in different units (`l0` from `int_atom` = actual CLVM atom length; `l1` from `limbs_for_int` = unsigned byte count) are summed in cost formulas without conversion. The result is systematic cost undercharging for any shift or multiply operation whose output has its high bit set.

---

### Finding Description

**Root cause — `limbs_for_int` (lines 100–102):**

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

`BigInt::bits()` returns the number of bits in the **absolute value**, so for `v = 128` (0x80):
- `bits()` = 8 → `div_ceil(8)` = **1**
- Actual CLVM signed atom: `[0x00, 0x80]` → length = **2**

The function undercounts by 1 for every positive integer whose most-significant byte has its high bit set.

**Affected site 1 — `op_ash` (lines 927–930):**

```rust
let l1 = limbs_for_int(&v);          // unsigned byte count — may be 1 short
let r = a.new_number(v)?;
let cost = ASHIFT_BASE_COST + ((l0 + l1) as Cost) * ASHIFT_COST_PER_BYTE;
Ok(malloc_cost(a, cost, r))
```

`l0` comes from `int_atom` → `a.atom_len(args)` (actual CLVM length). `l1` comes from `limbs_for_int` (unsigned length). They are added together in the cost formula without any unit reconciliation. When the shift result has its high bit set, `l1` is 1 less than the actual output atom length, undercharging by `ASHIFT_COST_PER_BYTE` = 3 cost units.

**Affected site 2 — `op_lsh` (lines 996–999):**

Identical pattern: `l1 = limbs_for_int(&v)` fed into `LSHIFT_COST_PER_BYTE` formula, undercharging by 3 cost units per affected output.

**Affected site 3 — `op_multiply` (line 649 and cost lines 615–616, 623–624):**

```rust
l0 = limbs_for_int(&total);   // updated after each multiplication step
if l0 > 1024 {
    return Err(...);
}
```

After the first multiplication, `l0` is refreshed via `limbs_for_int`. If `total` has its high bit set, `l0` is 1 less than the actual CLVM atom length. This propagates into the next iteration's cost formula:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

undercharging by `MUL_LINEAR_COST_PER_BYTE` (6) plus `l1/128` per step. Additionally, the size-limit guard `l0 > 1024` can be bypassed by exactly 1 byte: a product whose actual CLVM encoding is 1025 bytes passes as 1024.

---

### Impact Explanation

CLVM cost accounting is consensus-critical on the Chia blockchain. Every full node must compute identical costs for every program. The undercharge is small per operation (3–6 cost units), but:

1. **Spec/implementation divergence**: If any reference implementation (e.g., Python clvm) computes output sizes using actual CLVM signed atom lengths, it will compute a strictly higher cost for the same program. Nodes running the two implementations will disagree on whether a program exceeds the block cost limit, producing a **consensus split**.
2. **Budget bypass**: An attacker who knows the undercharge can craft programs that consume slightly more real computation than the declared cost budget permits, weakening the DoS protection the cost model provides.
3. **Size-limit bypass in `op_multiply`**: A product that is actually 1025 bytes in CLVM encoding passes the `l0 > 1024` guard, allowing one extra byte of intermediate state beyond the intended cap.

---

### Likelihood Explanation

Triggering the mismatch requires only that a shift or multiply operation produce a result whose most-significant byte has its high bit set (e.g., any result in the range [2^(8k−1), 2^(8k)) for integer k). This is trivially achievable with attacker-controlled CLVM bytes: `(ash 1 7)` produces 128 = `[0x00, 0x80]` and immediately demonstrates the undercount. No privileged access is required; any CLVM program submitted to a node is sufficient.

---

### Recommendation

Replace `limbs_for_int` with a function that computes the actual CLVM signed encoding length:

```rust
fn limbs_for_int(v: &Number) -> usize {
    // CLVM uses signed big-endian; a leading 0x00 is needed when the
    // high bit of the most-significant byte is set (positive numbers only).
    let bits = v.bits() as usize;
    if bits == 0 {
        return 0;
    }
    let unsigned_bytes = bits.div_ceil(8);
    // If bits is an exact multiple of 8, the high bit of the MSB is set
    // for a positive number, requiring a sign byte.
    if v.sign() == Sign::Plus && bits % 8 == 0 {
        unsigned_bytes + 1
    } else {
        unsigned_bytes
    }
}
```

Alternatively, allocate the number first and use `a.atom_len(ptr)` as the authoritative length, consistent with how `int_atom` measures input lengths.

### Proof of Concept

**Shift undercharge (op_ash):**

CLVM program: `(ash 1 7)` → result = 128

- `limbs_for_int(128)`: `bits(128)` = 8, `div_ceil(8)` = **1**
- Actual CLVM atom: `[0x00, 0x80]`, length = **2**
- Cost charged: `ASHIFT_BASE_COST + (1 + 1) * 3` = `596 + 6` = **602**
- Cost correct: `ASHIFT_BASE_COST + (1 + 2) * 3` = `596 + 9` = **605**
- Undercharge: **3 cost units**

**Multiply size-limit bypass (op_multiply):**

Craft a sequence of multiplications whose product `total` satisfies:
- `limbs_for_int(total)` = 1024 (bits = 8192, all set, high bit of MSB set)
- Actual CLVM atom length = 1025

The guard `l0 > 1024` evaluates `1024 > 1024` = false, so execution continues with a 1025-byte intermediate value, bypassing the intended 1024-byte cap. The subsequent cost step uses `l0 = 1024` instead of 1025, undercharging by `MUL_LINEAR_COST_PER_BYTE` (6) plus `l1/128`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
