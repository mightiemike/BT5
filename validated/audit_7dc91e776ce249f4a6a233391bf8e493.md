### Title
`ENABLE_SECP_OPS` Flag Check Missing on 4-Byte Secp Opcode Dispatch Path - (`File: src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` via two separate encoding paths. The 1-byte opcode path (opcodes 64 and 65) correctly gates execution behind the `ENABLE_SECP_OPS` flag. The 4-byte opcode path (`0x13d61f00` and `0x1c3a8f00`) dispatches the same functions with **no flag check at all**, allowing any attacker-controlled CLVM program to invoke secp signature verification regardless of whether `ENABLE_SECP_OPS` is set.

---

### Finding Description

`ChiaDialect::op()` contains two distinct dispatch branches for secp operators.

**Branch 1 — 1-byte opcodes (correctly gated):**

```rust
// src/chia_dialect.rs lines 248-249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

If `ENABLE_SECP_OPS` is absent, opcodes 64 and 65 fall through to `unknown_operator`, which in `NO_UNKNOWN_OPS` mode (i.e., mempool mode) returns `Err(EvalErr::Unimplemented(o))`.

**Branch 2 — 4-byte opcodes (ungated):**

```rust
// src/chia_dialect.rs lines 175-182
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

No `ENABLE_SECP_OPS` check is present. The function is dispatched and executed unconditionally.

`MEMPOOL_MODE` is defined as:

```rust
// src/chia_dialect.rs lines 72-76
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

`ENABLE_SECP_OPS` is absent from `MEMPOOL_MODE`. This means in mempool mode, 1-byte secp opcodes are rejected, but 4-byte secp opcodes execute freely. The same asymmetry applies to any caller that sets flags without `ENABLE_SECP_OPS`.

The pattern is consistent with how other flag-gated operators are handled: `op_modpow` (opcode 60) checks `DISABLE_OP`, `op_keccak256` (opcode 62) checks `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, and `op_sha256_tree` (opcode 63) checks `ENABLE_SHA256_TREE`. The 4-byte secp path is the sole exception that performs no flag check.

---

### Impact Explanation

**Vuln class**: Flag/operator wiring error — missing access control check on an alternate opcode encoding path.

**Corrupted result**: `op_secp256k1_verify` and `op_secp256r1_verify` execute and return `Reduction(cost, nil)` in execution contexts where `ENABLE_SECP_OPS` is not set and the operations are supposed to be blocked. The flag-based access control invariant is broken: the same cryptographic operation is simultaneously forbidden (via 1-byte encoding) and permitted (via 4-byte encoding) under identical flag state.

**Concrete impact**:
- In mempool mode, a program using 4-byte secp opcodes passes validation while an equivalent program using 1-byte opcodes is rejected. This creates a mempool bypass for secp operations.
- If `ENABLE_SECP_OPS` is a hardfork activation flag, nodes that have not yet activated the hardfork will still execute secp operations submitted via 4-byte opcodes, creating a consensus divergence vector between nodes at different flag states.
- The cost is correctly charged in both paths, so there is no undercharging, but the execution gate is absent.

**Impact: 4**

---

### Likelihood Explanation

The 4-byte opcode encoding is part of the documented "unknown operators with assigned cost" format described in the inline comment at line 158. The encoding is deterministic and derivable from the cost formula. Any attacker who can submit attacker-controlled CLVM bytes — via the mempool, a puzzle, or the Python API (`run_serialized_chia_program`) — can craft a program using `0x13d61f00` or `0x1c3a8f00` as the operator atom. No special privileges, social engineering, or compromised infrastructure are required.

**Likelihood: 4**

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the guard already present on the 1-byte path:

```rust
// src/chia_dialect.rs — 4-byte dispatch block
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both encoding paths enforce the same access control policy and that `ENABLE_SECP_OPS` is the single authoritative gate for secp operations regardless of how the operator is encoded.

### Proof of Concept

**Entry path**: Submit the following CLVM program to a node or mempool running with `MEMPOOL_MODE` (which lacks `ENABLE_SECP_OPS`):

```
; Operator atom: 4-byte encoding 0x13d61f00 = op_secp256k1_verify
; Program: (0x13d61f00 pubkey msg sig)
(0x13d61f00
  <33-byte secp256k1 pubkey>
  <32-byte message hash>
  <64-byte signature>
)
```

**Expected behavior** (with correct access control): The program is rejected with `Unimplemented` because `ENABLE_SECP_OPS` is not in `MEMPOOL_MODE`.

**Actual behavior**: The 4-byte opcode branch at lines 175–182 matches `0x13d61f00`, dispatches `op_secp256k1_verify` without any flag check, and returns `Reduction(1300000, nil)` — the operation executes successfully.

**Contrast**: The identical operation encoded as 1-byte opcode 64 falls through to `unknown_operator` at line 251, which returns `Err(EvalErr::Unimplemented(o))` under `NO_UNKNOWN_OPS`, correctly blocking it.

The root cause is exclusively in `src/chia_dialect.rs` lines 175–182. [1](#0-0) [2](#0-1) [3](#0-2)
