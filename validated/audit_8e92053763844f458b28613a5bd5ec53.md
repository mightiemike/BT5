### Title
`ChunkEndorsementInnerV1` Signed Payload Omits Shard/Epoch/Height — Block-Body Signature Reusable Across Shards in Same Block - (File: `core/primitives/src/stateless_validation/chunk_endorsement.rs`)

### Summary

The `ChunkEndorsementInnerV1` struct — the only data actually signed and embedded in the block body — contains only the `chunk_hash` and a static string differentiator. The `shard_id`, `epoch_id`, and `height_created` fields that uniquely identify the chunk's context are placed in a **separate** `ChunkEndorsementMetadata` struct signed under a **different** signature (`metadata_signature`). The block-body verification path (`validate_chunk_endorsements_in_block`) calls `ChunkEndorsement::validate_signature`, which reconstructs only the `inner` payload (chunk_hash + differentiator) and ignores the metadata signature entirely. A malicious block producer can therefore substitute the `inner` signature from one shard's slot into another shard's slot in the same block, provided both chunks happen to share the same `chunk_hash` — a condition that is structurally possible when two shards produce identical empty or near-identical chunks.

### Finding Description

`ChunkEndorsementInnerV1` is defined as:

```rust
struct ChunkEndorsementInnerV1 {
    chunk_hash: ChunkHash,
    signature_differentiator: SignatureDifferentiator, // always "ChunkEndorsement"
}
```

The `shard_id`, `epoch_id`, and `height_created` live only in `ChunkEndorsementMetadata`, which is signed separately and stored alongside the inner signature in the full `ChunkEndorsementV2` message. However, when a block producer assembles a block, it strips the full `ChunkEndorsementV2` down to just the raw `Signature` (the `inner` signature) and places it in the block body's `ChunkEndorsementSignatures` vector. The block-body validator then calls:

```rust
ChunkEndorsement::validate_signature(
    chunk_header.chunk_hash().clone(),
    signature,
    validator.public_key(),
)
```

which reconstructs `ChunkEndorsementInnerV1 { chunk_hash, "ChunkEndorsement" }` and verifies only that. The `metadata_signature` — which binds the validator's identity to a specific `shard_id`, `epoch_id`, and `height_created` — is **never checked** in this path.

The two-signature design in `ChunkEndorsementV2::verify` (which checks both `inner` and `metadata`) is only exercised during the P2P gossip validation path (`validate_chunk_endorsement_signature` → `endorsement.verify()`). Once the block producer has collected and cached the raw `inner` signatures, the block-body path only re-verifies the `inner` signature against the chunk hash.

### Impact Explanation

If two shards in the same block produce chunks with the same `chunk_hash` (e.g., both are empty chunks — `CryptoHash::default()` or any identical content), a validator's `inner` signature for shard A is cryptographically valid for shard B. A malicious block producer can copy the signature from shard A's slot into shard B's slot. The block-body verifier will accept it because it only checks `sig.verify(borsh(chunk_hash || "ChunkEndorsement"), pubkey)`, which is identical for both shards. This allows a block producer to fabricate the appearance of a 2/3-stake endorsement for a shard whose state witness was never actually validated by the claimed validators.

**Impact:** A malicious block producer can include a chunk for shard B in a block with forged endorsement stake, bypassing the stateless validation safety guarantee that 2/3 of stake has verified the chunk's state witness. This is a consensus-safety violation: invalid state transitions can be finalized.

### Likelihood Explanation

The precondition — two shards producing chunks with the same `chunk_hash` — occurs naturally and frequently for empty chunks. In a multi-shard network where some shards are idle, multiple shards will produce empty chunks in the same block, all hashing to the same value. This is not a rare edge case; it is the normal steady-state for any shard that has no transactions. The attack requires only a malicious block producer, which is an unprivileged role reachable by any validator with sufficient stake.

### Recommendation

Include `shard_id`, `epoch_id`, and `height_created` directly inside `ChunkEndorsementInnerV1` so that the signed payload is unique per (shard, epoch, height) triple. The block-body verification path must then reconstruct the inner from the chunk header's own fields rather than from the endorsement message, ensuring the signature cannot be transplanted across shards. Alternatively, unify the two-signature scheme so that the block-body path verifies the same combined payload that the gossip path verifies.

### Proof of Concept

1. Network has two shards (shard 0 and shard 1), both idle → both produce empty chunks with `chunk_hash = H(empty)`.
2. Validator V is assigned to shard 0. V sends a valid `ChunkEndorsement::V2` for shard 0 to the block producer. The `inner` signature is `sig = sign(borsh(H(empty) || "ChunkEndorsement"))`.
3. Malicious block producer assembles a block. For shard 1's endorsement slot at V's position, it inserts `sig` (copied from shard 0's slot) instead of a real endorsement from a shard-1 validator.
4. `validate_chunk_endorsements_in_block` calls `ChunkEndorsement::validate_signature(H(empty), sig, V.pubkey)` for shard 1's slot. This reconstructs `ChunkEndorsementInnerV1 { chunk_hash: H(empty), signature_differentiator: "ChunkEndorsement" }` and verifies `sig` against it — which succeeds, because `sig` was produced over exactly this payload.
5. The block is accepted with V's stake counted toward shard 1's endorsement, even though V never validated shard 1's state witness.

**Exact divergent bytes:** The Borsh-serialized `ChunkEndorsementInnerV1` for shard 0 and shard 1 are byte-for-byte identical when both chunks are empty, so the same 64-byte ed25519 signature satisfies both verification calls. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

#[derive(Debug, Clone, PartialEq, Eq, BorshSerialize, BorshDeserialize, ProtocolSchema)]
pub struct ChunkEndorsementV2 {
    // This is the part of the chunk endorsement that signed and included in the block header
    inner: ChunkEndorsementInnerV1,
    // This is the signature of the inner field, to be included in the block header
    signature: Signature,
    // This consists of the metadata for chunk endorsement used in validation
    metadata: ChunkEndorsementMetadata,
    // Metadata signature is used to validate that the metadata is produced by the expected validator
    metadata_signature: Signature,
}

impl ChunkEndorsementV2 {
    fn verify(&self, public_key: &PublicKey) -> bool {
        let inner = borsh::to_vec(&self.inner).unwrap();
        let metadata = borsh::to_vec(&self.metadata).unwrap();
        self.signature.verify(&inner, public_key)
            && self.metadata_signature.verify(&metadata, public_key)
    }

fn validate_chunk_endorsement_signature(
    epoch_manager: &dyn EpochManagerAdapter,
    endorsement: &ChunkEndorsement,
