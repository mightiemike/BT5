### Title
Merkle Proof Replay Across Chains and Contracts Enables Double-Claiming of Airdrop Tokens — (File: `core/contracts/Airdrop.sol`)

---

### Summary

`Airdrop._verifyProof` constructs the Merkle leaf using only `sender` and `totalAmount`. It omits `block.chainid` and `address(this)`. If the same Merkle root is registered on multiple chains or across two `Airdrop` contract instances on the same chain, any valid proof is replayable verbatim, allowing a user to drain the same allocation from every deployment.

---

### Finding Description

The leaf hash in `_verifyProof` is computed as:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(keccak256(abi.encode(sender, totalAmount)))
);
``` [1](#0-0) 

Neither `block.chainid` nor `address(this)` is encoded into the leaf. The `claimed` mapping that guards against re-use is per-contract storage:

```solidity
mapping(uint32 => mapping(address => uint256)) internal claimed;
``` [2](#0-1) 

A claim recorded on one chain or one contract instance has no effect on the `claimed` state of any other deployment. Because the leaf is identical across all deployments, the same `(week, sender, totalAmount, proof[])` tuple passes `MerkleProof.verify` on every instance that holds the same root. [3](#0-2) 

---

### Impact Explanation

A user who is entitled to `totalAmount` tokens for a given week can claim that amount once per chain and once per `Airdrop` contract instance that holds the same Merkle root. If Nado operates on N chains (or redeploys the `Airdrop` contract after an upgrade while keeping the same roots), the user can extract `N × totalAmount` tokens in total. The asset delta is direct token loss from each `Airdrop` contract's balance. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** Multi-chain deployment is a standard pattern for DeFi protocols; Nado's own `OffchainExchange` already encodes `block.chainid` in its order digest, confirming multi-chain awareness. Airdrop Merkle roots are typically identical across chains because the eligible set of users is the same. A new `Airdrop` contract deployed after an upgrade would also share historical roots registered via `registerMerkleRoot`. Both scenarios are realistic operational conditions, not edge cases. [5](#0-4) 

---

### Recommendation

Bind the leaf to the specific deployment by including `block.chainid` and `address(this)` in the encoded data:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(
        keccak256(
            abi.encode(block.chainid, address(this), sender, totalAmount)
        )
    )
);
```

The Merkle roots must be regenerated with leaves that include these fields. Existing roots without chain/contract binding should be considered invalid and replaced before any multi-chain deployment.

---

### Proof of Concept

1. Nado deploys `Airdrop` on Chain A and Chain B.
2. Owner calls `registerMerkleRoot(1, root)` on both contracts with the same `root`.
3. Alice is entitled to 1000 tokens. Her leaf is `keccak256(bytes.concat(keccak256(abi.encode(alice, 1000))))` — identical on both chains.
4. Alice calls `claim([{week:1, totalAmount:1000, proof:[...]}])` on Chain A → succeeds, `claimed[1][alice] = 1000` on Chain A.
5. Alice calls the same `claim` call on Chain B → `claimed[1][alice]` on Chain B is still `0`, `MerkleProof.verify` passes with the same proof, Alice receives another 1000 tokens.
6. Alice has claimed 2000 tokens against a 1000-token entitlement. [6](#0-5)

    function _verifyProof(
        uint32 week,
        address sender,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        require(claimed[week][sender] == 0, "Already claimed.");
        require(
            merkleRoots[week] != bytes32(0),
            "Week hasn't been registered."
        );
        require(
            !ISanctionsList(sanctions).isSanctioned(sender),
            "address is sanctioned."
        );
        bytes32 leaf = keccak256(
            bytes.concat(keccak256(abi.encode(sender, totalAmount)))
        );
        bool isValidLeaf = MerkleProof.verify(proof, merkleRoots[week], leaf);
        require(isValidLeaf, "Invalid proof.");
        claimed[week][sender] = totalAmount;
    }

    function _claim(
        uint32 week,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        _verifyProof(week, msg.sender, totalAmount, proof);
        SafeERC20.safeTransfer(IERC20(token), msg.sender, totalAmount);
        emit Claim(msg.sender, week, totalAmount);
    }
```
