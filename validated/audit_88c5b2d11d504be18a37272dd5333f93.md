Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks LP-position `owner` instead of transaction `sender`, allowing any address to bypass the deposit allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead validates the caller-supplied `owner` argument against `allowedDepositor`. Because `owner` is freely chosen by the caller, any address not on the allowlist can pass an allowlisted address as `owner` and the hook will not revert. The deposit allowlist is fully bypassed.

## Finding Description
`MetricOmmPool.addLiquidity` passes the real caller as the first argument to the extension hook:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but leaves it unnamed (discarded) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

The sibling `SwapAllowlistExtension.beforeSwap` applies the correct pattern — it names and checks `sender` (the actual caller) and discards the second argument (`recipient`):

```solidity
// SwapAllowlistExtension.sol lines 31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [3](#0-2) 

The exploit path is: attacker (not on allowlist) calls `pool.addLiquidity(owner = <allowlisted address>, ...)`. The hook evaluates `allowedDepositor[pool][allowlistedAddress]` which is `true`, so no revert occurs. The attacker's `msg.sender` is never checked. The attacker provides tokens via the liquidity callback; LP shares are credited to the allowlisted `owner`.

## Impact Explanation
The deposit allowlist — the sole on-chain mechanism restricting who may add liquidity to the pool — is completely non-functional. Any unprivileged address can deposit freely by supplying any allowlisted address as `owner`. If the allowlist is used for regulatory compliance (KYC/AML), sanctioned or non-compliant addresses can deposit without restriction. Additionally, the allowlisted `owner` receives LP shares they did not request, constituting a griefing vector (forced exposure increase without consent). Because extension addresses are immutable post-deployment, the pool admin has no on-chain remedy once deployed.

## Likelihood Explanation
Exploitation requires only: (1) knowledge of one allowlisted address, which is publicly readable from `allowedDepositor` storage, and (2) the ability to call `addLiquidity` directly or through any router. No special privilege, flash loan, or oracle manipulation is needed. The attacker must supply tokens (they pay), so the attack is not directly profitable, but it is trivially executable by any address seeking to circumvent access control.

## Recommendation
Replace the unnamed first argument with `sender` and check it instead of `owner`, matching the pattern used by `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

## Proof of Concept
```solidity
// Assume pool has DepositAllowlistExtension configured.
// Alice (allowlisted), Bob (not allowlisted).

// Bob calls addLiquidity with owner = Alice.
pool.addLiquidity(
    /* owner */        address(alice),
    /* salt */         0,
    /* deltas */       liquidityDelta,
    /* callbackData */ "",
    /* extensionData */""
);
// Hook checks allowedDepositor[pool][alice] == true → no revert.
// Bob pays tokens via callback; Alice's LP position is credited.
// Bob successfully bypassed the allowlist.
```

A Foundry integration test can confirm this by: deploying a pool with `DepositAllowlistExtension`, allowlisting Alice but not Bob, then calling `addLiquidity` from Bob with `owner = alice` and asserting the call succeeds (no `NotAllowedToDeposit` revert).
