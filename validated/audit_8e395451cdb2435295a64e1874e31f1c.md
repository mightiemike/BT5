Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of originating user, allowing full allowlist bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension` gates swaps by checking the `sender` argument passed from the pool, which is `msg.sender` at the time `pool.swap` is called. When `MetricOmmSimpleRouter` is the caller, `sender` is the router's address, not the originating user's address. Any pool admin who allowlists the router (required for legitimate allowlisted users to use it) inadvertently grants every user unrestricted swap access to the pool.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

In `exactInputSingle`, the router calls `pool.swap` directly with no originating-user argument. The originating user (`msg.sender`) is stored only in the internal callback context for payment settlement and is never surfaced to the pool or extensions: [4](#0-3) 

The same pattern holds for `exactInput` (all hops): [5](#0-4) 

The result: the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. If the router is allowlisted (required for any legitimate user to use it), the check passes for every caller regardless of their individual allowlist status.

## Impact Explanation
A pool admin who deploys a curated pool with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC-verified counterparties or whitelisted market makers) loses that protection entirely for any user who routes through `MetricOmmSimpleRouter`. The non-allowlisted user receives output tokens and pays input tokens exactly as a legitimate allowlisted user would. This is a direct bypass of a core access-control mechanism, constituting broken core pool functionality and an admin-boundary break where an unprivileged actor circumvents a pool admin's enforced restriction.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary supported swap interface for end users. Any pool admin who wants allowlisted users to be able to use the router must allowlist the router itself. Once the router is allowlisted, the bypass is available to every user with no special setup, no privileged access, and no non-standard token behavior required. The attacker only needs to call a standard router function.

## Recommendation
The pool must surface the originating user's identity to extensions. Two viable approaches:

1. **Router passes the originating user via `extensionData`**: Define a standard encoding (e.g., a leading `address` in `extensionData`) that the router always prepends, and have `SwapAllowlistExtension.beforeSwap` decode and check that address instead of `sender`. The extension must verify `msg.sender` is the expected pool to prevent spoofing by direct callers.

2. **Pool exposes a `swapWithOriginator` entry point**: Add an optional `originator` parameter to `swap`; the router passes `msg.sender` there; extensions receive it as a distinct argument. Direct callers pass `address(0)` to fall back to `msg.sender`.

Either fix must be applied consistently across all router entry points, including the recursive `_exactOutputIterateCallback` hops.

## Proof of Concept
```
Setup:
  Pool P configured with SwapAllowlistExtension E
  Pool admin allowlists user A:   allowedSwapper[P][A] = true
  Pool admin allowlists router R: allowedSwapper[P][R] = true
    (required so A can use the router)

Attack (user B, not allowlisted):
  1. B calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})
  2. Router calls P.swap(recipient=B, ...) — msg.sender inside P is R
  3. P calls _beforeSwap(sender=R, ...)
  4. SwapAllowlistExtension checks allowedSwapper[P][R] → true → passes
  5. Swap executes; B receives output tokens

Result:
  B successfully swaps on a pool restricted to A only.
  The allowlist provides zero protection against router-mediated swaps.
```
