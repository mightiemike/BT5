### Title
4-byte secp operator encoding bypasses `ENABLE_SECP_OPS` flag guard — (File: `src/chia_dialect.rs`)

---

### Summary

In `ChiaDialect::op()`, the 4-byte secp operator encodings (`0x13d61f00` for `secp256k1_verify` and `0x1c3a8f00` for `secp256r1_verify`) are dispatched directly to the secp verification functions **without** checking `ClvmFlags::ENABLE_SECP_OPS`. The 1-byte encodings (opcodes 64 and 65) correctly require this flag. An early return in the `op_len == 4` branch exits before the flag-guarded dispatch table is reached, mirroring the external report's pattern of a validation step being skipped due to a conditional early return.

---

### Finding Description

`ChiaDialect::op()` has two structurally separate dispatch paths for secp operators:

**Path 1 — 4-byte operators (lines 157–183):**

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);  // ← early return, no flag check
}
```

**Path 2 — 1-byte operators (lines 248–249):**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The `return` at line 182 exits the function before the `ENABLE_SECP_OPS`-guarded match arm is ever reached. The 4-byte path dispatches `op_secp256k1_verify` and `op_secp256r1_verify` unconditionally, regardless of whether `ENABLE_SECP_OPS` is set in the dialect flags.

The root cause is a **flag/operator wiring error**: the guard check (`ENABLE_SECP_OPS`) is enforced for the 1-byte encoding but is entirely absent from the 4-byte encoding of the same operations.

---

### Impact Explanation

`ENABLE_SECP_OPS` is the hard-fork activation flag for the 1-byte secp opcode shorthand. When it is **not** set (i.e., before the hard fork activates, or in mempool-mode configurations that have not yet enabled it), the intent is that secp verification via the new 1-byte opcodes is unavailable. However, an attacker-controlled CLVM program can use the 4-byte encoding `0x13d61f00` or `0x1c3a8f00` as the operator atom to invoke the identical secp verification logic unconditionally.

This produces **consensus divergence**: a node running with `ENABLE_SECP_OPS` unset will:
- Correctly reject a program using opcode `0x40` (64) or `0x41` (65) as an unknown operator
- Incorrectly **execute** the same secp verification when the program uses the 4-byte encoding

Two nodes with different flag configurations will therefore reach different conclusions about the validity of the same CLVM program, breaking consensus equivalence. The corrupted result is the boolean validity of the secp signature check — it is accepted on one code path and rejected on the other for the same underlying operation.

---

### Likelihood Explanation

The 4-byte encoding is a valid CLVM atom and can be embedded in any attacker-submitted puzzle or solution. No special privileges are required. Any caller that passes attacker-controlled CLVM bytes to `run_program` with a `ChiaDialect` that does not set `ENABLE_SECP_OPS` is reachable. The encoding values (`0x13d61f00`, `0x1c3a8f00`) are documented in the source comments, making them trivially discoverable.

---

### Recommendation

Add an `ENABLE_SECP_OPS` check inside the `op_len == 4` branch before dispatching to the secp functions, mirroring the guard already present in the 1-byte branch:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures the flag gate is enforced uniformly for both encodings of the same operation, eliminating the control-flow bypass.

---

### Proof of Concept

A CLVM program of the form `(0x13d61f00 pubkey msg sig)` — where `0x13d61f00` is a 4-byte atom used as the operator — submitted to a `ChiaDialect` instantiated **without** `ENABLE_SECP_OPS`:

1. `op_len == 4` → enters the 4-byte branch at line 157.
2. `opcode == 0x13d61f00` → matches `op_secp256k1_verify` at line 176.
3. `return f(...)` at line 182 → executes secp256k1 signature verification and returns a result.
4. The `ENABLE_SECP_OPS` check at line 248 is **never reached**.

The identical program using opcode `0x40` (1-byte encoding for 64) falls through to `unknown_operator` under the same flag configuration, confirming the asymmetric enforcement. [1](#0-0) [2](#0-1)
