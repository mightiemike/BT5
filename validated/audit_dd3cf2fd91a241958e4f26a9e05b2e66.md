### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Allowing Bypass of 64-Byte Merkle Forgery Mitigation — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR method despite being superseded by `verify_transaction_inclusion_v2`. The v2 function was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability. Because v1 remains reachable by any unprivileged caller, an attacker can bypass the coinbase-proof guard entirely by routing their forged proof through v1, obtaining a `true` inclusion result for a transaction that was never confirmed in a Bitcoin block.

---

### Finding Description

`verify_transaction_inclusion_v2` was added to mitigate the well-known 64-byte transaction Merkle proof forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring a coinbase Merkle proof of equal length, validating that the coinbase transaction hashes to the block's `merkle_root` before delegating to v1: [1](#0-0) 

However, v1 is not removed or access-gated — it is only annotated `#[deprecated]`, which is a Rust compiler hint with no runtime enforcement: [2](#0-1) 

The function remains `pub` and is exposed as a NEAR contract method. Any caller can invoke it directly, skipping the coinbase validation entirely: [3](#0-2) 

The v1 path only checks that `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.merkle_root`. It does not verify that `tx_id` corresponds to a real leaf (transaction) rather than an internal Merkle node: [4](#0-3) 

The `compute_root_from_merkle_proof` function in `merkle-tools` is a pure positional hash-chain computation with no structural constraints: [5](#0-4) 

---

### Impact Explanation

The analog to the cross-function signature replay is exact: v2 adds a domain-specific guard (coinbase proof) that v1 lacks, but both functions accept the same `ProofArgs` data (v2 converts `ProofArgsV2` into `ProofArgs` via `.into()`). An attacker who constructs a forged 64-byte internal-node proof that v2 would reject (coinbase check fails) can submit the same proof directly to v1 and receive `true`.

Any downstream NEAR contract or bridge that calls `verify_transaction_inclusion` to gate a token release, cross-chain action, or settlement will be deceived into treating an unconfirmed or fabricated Bitcoin transaction as confirmed. The corrupted value is the `bool` return of `verify_transaction_inclusion` — it becomes `true` for a proof that encodes an internal Merkle node rather than a real transaction.

---

### Likelihood Explanation

The entry path requires no privileges: any NEAR account can call `verify_transaction_inclusion` with a crafted `ProofArgs`. The 64-byte forgery technique is publicly documented and has known tooling. The only prerequisite is that the attacker knows a valid `tx_block_blockhash` that is on the current mainchain and has enough confirmations — both are public on-chain state readable via `get_block_hash_by_height` or `get_last_n_blocks_hashes`. [6](#0-5) 

---

### Recommendation

Remove `verify_transaction_inclusion` as a public NEAR method. It should be made `pub(crate)` or `fn` (private) so it is only reachable as an internal helper called by `verify_transaction_inclusion_v2`. The `#[deprecated]` attribute provides no access control at the NEAR runtime level.

---

### Proof of Concept

1. Attacker reads a recent mainchain block hash `B` via `get_block_hash_by_height(h)`.
2. Attacker constructs a 64-byte value `fake_tx` that equals the concatenation of two real child hashes at some internal Merkle node of block `B`, such that `double_sha256(fake_tx)` equals a known internal node hash `N`.
3. Attacker builds a `merkle_proof` path from `N` up to `merkle_root` of block `B` (all values are public from the Bitcoin network).
4. Attacker calls `verify_transaction_inclusion({ tx_id: fake_tx_hash, tx_block_blockhash: B, tx_index: <matching position>, merkle_proof: [...], confirmations: 1 })` directly — bypassing v2.
5. `compute_root_from_merkle_proof(fake_tx_hash, position, proof)` returns `merkle_root` of `B`.
6. The function returns `true` for a transaction that does not exist. [7](#0-6) [8](#0-7)

    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
