### Title
Cross-Channel Refund Failure: `MintNFT` Creator Mismatch Causes Permanent NFT Loss on Timeout/Ack-Error — (`x/nft-transfer/keeper/packet.go`)

---

### Summary

When a chain acts as a **sink** (burns the voucher on send) and the outgoing packet uses a **different channel** than the one through which the NFT originally arrived, the `refundPacketToken` path calls `MintNFT` with the escrow address of the *sending* channel. However, the denom's stored creator is the escrow address of the *receiving* channel from the original arrival. The `IsDenomCreator` check inside `MintNFT` fails, the error propagates, and the burned voucher is permanently unrecoverable.

---

### Finding Description

**Step 1 — NFT arrives via channel-A, denom creator is set to `escrowAddress_A`.**

In `processReceivedPacket`, when `isAwayFromOrigin = true` (NFT moving away from origin, i.e., this chain is a sink), the denom is issued with the **destination channel's** escrow address as creator:

```go
escrowAddress := types.GetEscrowAddress(packet.GetDestPort(), packet.GetDestChannel())
// escrowAddress = escrowAddress_A
k.nftKeeper.IssueDenom(ctx, voucherClassID, voucherClassID, "", data.ClassUri, escrowAddress)
``` [1](#0-0) 

The denom `voucherClassID` is now permanently stored with `Creator = escrowAddress_A`.

**Step 2 — User sends the voucher back via channel-B (different channel), NFT is burned.**

`createOutgoingPacket` resolves `fullClassPath = "transfer/channel-A/nftClass"` and evaluates:

```go
isAwayFromOrigin := types.IsAwayFromOrigin("transfer", "channel-B", "transfer/channel-A/nftClass")
// = !strings.HasPrefix("transfer/channel-A/nftClass", "transfer/channel-B/") = false
``` [2](#0-1) 

Because `isAwayFromOrigin = false`, the chain is treated as a sink and the voucher is **burned** via `BurnNFTUnverified` — which has no denom-creator check:

```go
k.nftKeeper.BurnNFTUnverified(ctx, classID, tokenID, sender)
``` [3](#0-2) 

**Step 3 — Timeout or ack-error triggers `refundPacketToken` with `escrowAddress_B`.**

`OnTimeoutPacket` and `OnAcknowledgementPacket` both call `refundPacketToken`: [4](#0-3) 

Inside `refundPacketToken`, the escrow address is derived from the **packet's source channel** (channel-B):

```go
escrowAddress := types.GetEscrowAddress(packet.GetSourcePort(), packet.GetSourceChannel())
// escrowAddress = escrowAddress_B  ← WRONG channel
``` [5](#0-4) 

Since `isAwayFromOrigin = false` (sink), the refund path calls:

```go
k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, sender)
``` [6](#0-5) 

**Step 4 — `MintNFT` calls `IsDenomCreator`, which fails.**

```go
func (k Keeper) MintNFT(..., sender, owner sdk.AccAddress) error {
    _, err := k.IsDenomCreator(ctx, denomID, sender)  // sender = escrowAddress_B
    if err != nil {
        return err  // ← FAILS: creator is escrowAddress_A
    }
    ...
}
``` [7](#0-6) 

`IsDenomCreator` compares `denom.Creator` (= `escrowAddress_A`) against `escrowAddress_B`:

```go
if !creator.Equals(address) {
    return types.Denom{}, sdkerrors.Wrapf(types.ErrUnauthorized, "%s is not the creator of %s", address, denomID)
}
``` [8](#0-7) 

The error propagates back through `refundPacketToken` → `OnTimeoutPacket` / `OnAcknowledgementPacket`. The burned voucher is never reminted. **The NFT is permanently lost.**

---

### Impact Explanation

The invariant that every timeout/ack-error on a sink-chain send must successfully remint the burned voucher to the sender is violated. The sender's NFT is burned in step 2 and cannot be recovered. This is a **total, permanent loss** of the NFT for the sender — matching the Critical scope: "unintentional burn or permanent lock of economically valuable NFTs."

No governance or privileged action is required. The user triggers this entirely through normal IBC send operations.

---

### Likelihood Explanation

- Two IBC channels between the same pair of chains (or different chains sharing the same NFT class path prefix) is a common, supported configuration.
- Channel upgrades, re-routes, or simply having multiple relayers open parallel channels make this scenario reachable in production.
- The user does not need to be malicious — this can happen accidentally when a user sends via a non-canonical channel.
- No special permissions, governance votes, or operator compromise are required.

---

### Recommendation

In `refundPacketToken`, when the chain is a sink (must remint), do **not** use `MintNFT` (which enforces a denom-creator check). Instead, use `MintNFTUnverified`, which skips the creator check and is already used for genesis initialization:

```go
// Instead of:
k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, sender)

// Use:
k.nftKeeper.MintNFTUnverified(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", sender)
``` [9](#0-8) 

Alternatively, the denom creator could be updated to a canonical module-level address (not a per-channel escrow address) so that any channel's escrow address can mint. The root design issue is that `IssueDenom` binds the creator to a specific channel's escrow address, but the refund path can arrive via a different channel.

---

### Proof of Concept

```
Chain Y state setup:
  1. NFT "nftClass/token1" sent from chain X → chain Y via channel-A
     → processReceivedPacket: IssueDenom("ibc/HASH", creator=escrowAddr_A)
     → MintNFT("ibc/HASH", "token1", ..., escrowAddr_A, userY)
     → State: denom "ibc/HASH".Creator = escrowAddr_A, owner = userY

  2. userY sends "ibc/HASH/token1" back via channel-B (different channel)
     → fullClassPath = "transfer/channel-A/nftClass"
     → IsAwayFromOrigin("transfer","channel-B","transfer/channel-A/nftClass") = false
     → BurnNFTUnverified("ibc/HASH", "token1", userY) succeeds
     → Packet sent on channel-B

  3. Packet times out (or counterparty returns ack-error)
     → OnTimeoutPacket → refundPacketToken
     → escrowAddress = GetEscrowAddress("transfer","channel-B") = escrowAddr_B
     → isAwayFromOrigin = false → MintNFT("ibc/HASH","token1","","uri","",escrowAddr_B,userY)
     → IsDenomCreator("ibc/HASH", escrowAddr_B): escrowAddr_A ≠ escrowAddr_B → ERROR
     → refundPacketToken returns error
     → "token1" is permanently lost
```

// MintNFTUnverified mints an NFT without verifying if the owner is the creator of denom
// Needed during genesis initialization
func (k Keeper) MintNFTUnverified(ctx sdk.Context, denomID, tokenID, tokenNm, tokenURI, tokenData string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	if k.HasNFT(ctx, denomID, tokenID) {
		return sdkerrors.Wrapf(types.ErrNFTAlreadyExists, "NFT %s already exists in collection %s", tokenID, denomID)
	}

	k.setNFT(
		ctx, denomID,
		types.NewBaseNFT(
			tokenID,
			tokenNm,
			owner,
			tokenURI,
			tokenData,
		),
	)
	k.setOwner(ctx, denomID, tokenID, owner)
	k.increaseSupply(ctx, denomID)

	return nil
}
```
