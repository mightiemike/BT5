### Title
Removed Refund Mechanism in Silent-Failure Slow-Mode Deposit Path Causes Permanent Collateral Lockup - (File: `core/contracts/Endpoint.sol`)

---

### Summary

When a user calls `depositCollateral()`, their tokens are transferred to the clearinghouse **immediately and unconditionally**, but the actual on-chain credit is deferred into a slow-mode queue processed up to 3 days later. If the deferred `processSlowModeTransaction` call reverts for any reason, the failure is silently swallowed by a `try/catch` block whose catch body explicitly has no refund logic — confirmed by the in-code comment `// try return funds now removed`. The user's tokens are permanently locked in the clearinghouse with no recovery path.

---

### Finding Description

`depositCollateralWithReferral()` in `Endpoint.sol` performs two distinct, non-atomic operations:

**Step 1 — Immediate token transfer** (lines 144–148):
```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
```
This calls `safeTransferFrom(user → clearinghouse)` immediately and unconditionally. The user's tokens leave their wallet at this point.

**Step 2 — Deferred credit** (lines 152–165): A `SlowModeTx` is enqueued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (3 days). The actual `clearinghouse.depositCollateral()` call — which credits the user's subaccount balance — only happens when this queued transaction is later executed. [1](#0-0) 

When the slow-mode tx is eventually executed via `_executeSlowModeTransaction()`, the processing call is wrapped in a `try/catch`:

```solidity
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed
}
``` [2](#0-1) 

The catch block only re-raises on OOG. For all other reverts, it silently discards the failure. The comment `// try return funds now removed` is a direct admission that a refund path previously existed and was deliberately deleted, leaving no recovery mechanism.

Inside `processSlowModeTransactionImpl`, the `DepositCollateral` branch calls `clearinghouse.depositCollateral(txn)`: [3](#0-2) 

`Clearinghouse.depositCollateral()` can revert under several conditions reachable within the 3-day window:

- `_decimals(txn.productId)` reverts if the product's token address is zero (e.g., after a product is delisted or reconfigured) — `require(address(token) != address(0), ERR_INVALID_PRODUCT)` in `_decimals`.
- `require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED)` — if subaccount classification changes.
- `require(decimals <= MAX_DECIMALS)` — if token metadata changes. [4](#0-3) 

When any of these revert, the slow-mode tx is consumed (`delete slowModeTxs[_slowModeConfig.txUpTo++]` at line 194), the failure is swallowed, and the user's tokens — already sitting in the clearinghouse — are never credited to any subaccount. There is no re-queue, no refund, and no admin recovery function. [5](#0-4) 

---

### Impact Explanation

A user's deposited ERC-20 collateral is permanently locked inside the `Clearinghouse` contract. The tokens are held by the clearinghouse (`handleDepositTransfer` sends them there), but no subaccount balance is ever credited, so the user cannot withdraw them. There is no admin sweep, no re-deposit path, and no fallback. The asset delta is: user loses `amount` tokens of `productId` with zero protocol-side credit.

---

### Likelihood Explanation

The 3-day `SLOW_MODE_TX_DELAY` window creates a realistic gap for state changes. Product delisting is an explicitly supported protocol operation (`DelistProduct` transaction type in `EndpointTx.sol`), and it can be triggered by the owner via slow mode at any time. A user who deposits a spot asset that is subsequently delisted within 3 days will have their deposit slow-mode tx fail silently. The comment `// try return funds now removed` confirms the developers were aware of this failure mode and removed the safety net, making this a known-but-unmitigated risk. Likelihood is **Medium** — requires a product lifecycle event within the 3-day window, which is an infrequent but realistic protocol operation. [6](#0-5) 

---

### Recommendation

Restore a refund mechanism in the `catch` block of `_executeSlowModeTransaction`. For `DepositCollateral` slow-mode transactions specifically, if `processSlowModeTransaction` reverts (non-OOG), the clearinghouse should transfer the deposited tokens back to the original sender. Alternatively, record failed deposit txs in a claimable mapping so users can self-recover their funds. The 3-day deferred execution model is inherently racy with protocol state changes, so the refund path is a necessary invariant, not an optional optimization.

---

### Proof of Concept

1. User calls `Endpoint.depositCollateral(subaccountName, productId=X, amount=1000e6)`.
2. `handleDepositTransfer` immediately moves 1000 USDC from user → clearinghouse. A `SlowModeTx` is enqueued with `executableAt = now + 3 days`.
3. Within 3 days, the protocol owner calls `submitSlowModeTransaction(DelistProduct{productId: X, ...})` and the sequencer processes it, removing product X from the engine.
4. After 3 days, anyone calls `executeSlowModeTransaction()`. `_executeSlowModeTransaction` dequeues the deposit tx and calls `this.processSlowModeTransaction(sender, tx)`.
5. Inside `processSlowModeTransactionImpl`, `clearinghouse.depositCollateral(txn)` is called. `_decimals(X)` calls `spotEngine.getConfig(X).token`, which now returns `address(0)` for the delisted product, causing `require(address(token) != address(0), ERR_INVALID_PRODUCT)` to revert.
6. The `catch` block fires. `gasleft()` is well above 250000. The catch body executes `// try return funds now removed` — i.e., nothing.
7. The slow-mode tx slot is already deleted (line 194). The 1000 USDC sits in the clearinghouse forever. The user's subaccount balance for product X is 0. No withdrawal is possible. [7](#0-6) [8](#0-7)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
```

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
```
