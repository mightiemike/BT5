### Title
Gas-Key Exec-Fee Uses Wire Length Instead of Trie-ID Length for ML-DSA-65 Keys in Host Functions — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call` host functions compute the nonce-write exec fee using the raw wire length of the borsh-encoded public key (`public_key_len as usize`). For ED25519 and SECP256K1 keys the wire length equals the trie-ID length, so the fee is correct. For ML-DSA-65 keys the wire length is 1953 bytes but the on-trie identifier is only 33 bytes (a SHA3-256 hash), causing a ~59× overcharge in the exec fee per nonce. The transaction path (`permission_exec_fees` in `runtime/runtime/src/config.rs`) correctly calls `public_key.trie_id_len()`, creating an exact, deterministic fee-parity violation between the two code paths.

---

### Finding Description

`gas_key_add_key_exec_fee` computes the cost of writing per-nonce trie entries for a gas key. Its `public_key_len` parameter feeds directly into `access_key_key_len`, which determines the nonce trie-key length: [1](#0-0) 

The transaction path in `permission_exec_fees` passes the correct trie-ID length: [2](#0-1) 

The host-function path in `promise_batch_action_add_gas_key_with_full_access` passes the raw wire length instead: [3](#0-2) 

The same mistake appears in `promise_batch_action_add_gas_key_with_function_call`: [4](#0-3) 

And in both wasmtime counterparts: [5](#0-4) [6](#0-5) 

The design document explicitly states this invariant:

> `PublicKey::trie_id_len()` is a new contract that all storage-cost code must respect. Callers that still use `len()` for trie-storage costing will misprice ML-DSA-65 keys by ~1900 bytes. [7](#0-6) 

For ML-DSA-65 keys, `trie_id_len()` returns 33 bytes while `len()` (and the wire `public_key_len`) returns 1953 bytes: [8](#0-7) 

The fee-parity test that would catch this only exercises ED25519 keys, where wire length equals trie-ID length and the bug is invisible: [9](#0-8) 

---

### Impact Explanation

When both `GasKeys` and `PostQuantumSignatures` protocol features are active, any contract that calls `promise_batch_action_add_gas_key_with_full_access` or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key will have its exec fee computed against a 1953-byte key length instead of the correct 33-byte trie-ID length. The overcharge is `(1953 − 33) × gas_key_byte_exec_fee × num_nonces` per action. This can cause gas exhaustion for contracts that budget based on the documented trie-ID size, breaking the ability to add ML-DSA-65 gas keys via host functions. It also breaks the fee-parity invariant between the transaction path and the host-function path for the same operation.

---

### Likelihood Explanation

Both `GasKeys` and `PostQuantumSignatures` are stabilized features in the codebase. Any contract author who reads the protocol documentation and attempts to add an ML-DSA-65 gas key via a host function will encounter the overcharge. The trigger is unprivileged: any contract call suffices. The fee-parity test exists but only covers ED25519, so the regression is not caught by the existing test suite.

---

### Recommendation

Replace `public_key_len as usize` with `public_key.trie_id_len()` in all four call sites of `gas_key_add_key_exec_fee` inside the host-function implementations. The decoded `PublicKey` is already available at that point (as `public_key.decode()?` is called immediately after), so `trie_id_len()` can be called on the decoded key before the fee computation. Add a variant of `test_gas_key_fee_parity` that uses `KeyType::MLDSA65` to pin the invariant.

---

### Proof of Concept

For an ML-DSA-65 key with `num_nonces = 4` and a 10-character account ID:

- **Transaction path** (`permission_exec_fees`): `nonce_key_len = access_key_key_len(10, 33) + 2` → fee based on ~45 bytes per nonce.
- **Host-function path** (`promise_batch_action_add_gas_key_with_full_access`): `nonce_key_len = access_key_key_len(10, 1953) + 2` → fee based on ~1965 bytes per nonce.

The divergence is ~1920 bytes × `gas_key_byte` exec fee × 4 nonces. At the mainnet `gas_key_byte` exec rate, this is a deterministic overcharge that will exhaust the gas budget of any contract that sizes its prepaid gas against the documented 33-byte trie-ID length.

pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
