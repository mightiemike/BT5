### Title
Missing Cross-Validation Between `BouncerWeights::receipt_l2_gas` and `VersionedConstants::max_block_size` Causes Systematically Wrong EIP-1559 Fee Market Pricing - (`File: crates/blockifier/src/bouncer.rs` / `crates/apollo_versioned_constants/src/lib.rs`)

---

### Summary

`BouncerWeights::receipt_l2_gas` (batcher config, set per-node) and `VersionedConstants::max_block_size` (orchestrator versioned constants, set per protocol version) are two independently configurable values that **must be equal** for the EIP-1559 fee market to produce correct gas prices. Both values are documented with "Must stay in sync" comments, but no runtime or startup validation enforces this invariant. When they diverge — for example, after a protocol upgrade changes `max_block_size` without a corresponding update to the batcher config — the fee market produces systematically wrong L2 gas prices for every subsequent block.

---

### Finding Description

`BouncerWeights::receipt_l2_gas` is the block-closing capacity limit for the economic gas metric in the Batcher: [1](#0-0) 

Its default value is hardcoded to `GasAmount(5800000000)` with the comment "Must stay in sync with orchestrator_versioned_constants' max_block_size": [2](#0-1) 

`VersionedConstants::max_block_size` in the orchestrator versioned constants is the reference block size used by the EIP-1559 fee market: [3](#0-2) 

The comment there also says "Must stay in sync with BouncerWeights receipt_l2_gas." The `config_schema.json` description for `receipt_l2_gas` says "Should equal max_block_size": [4](#0-3) 

The EIP-1559 fee market function `calculate_next_base_gas_price` uses `gas_target` (a fraction of `max_block_size`) as the denominator for price adjustment: [5](#0-4) 

The `gas_target` is read from `VersionedConstants::latest_constants()` inside the fee market, while `receipt_l2_gas` is read from the batcher's `BouncerConfig`. These are two separate configuration sources with no cross-validation: [6](#0-5) 

The versioned constants JSON files show that `max_block_size` has changed across protocol versions (e.g., 4000000000 → 5000000000 → 5800000000): [7](#0-6) 

The `validate_node_config` cross-member validation in `node_config.rs` performs many consistency checks but contains no check that `batcher_config.block_max_capacity.receipt_l2_gas == VersionedConstants::max_block_size`: [8](#0-7) 

---

### Impact Explanation

If `block_max_capacity.receipt_l2_gas` is set lower than `VersionedConstants::gas_target` (e.g., after a protocol upgrade raises `max_block_size` and `gas_target` but the operator forgets to update the batcher config), every block closes before accumulating enough `receipt_l2_gas` to reach the fee market's target. The fee market then permanently sees under-utilization and drives the L2 gas price to the configured minimum, regardless of actual network demand. Conversely, if `receipt_l2_gas` is set higher than `max_block_size`, blocks can exceed the fee market's reference size, causing the fee market to permanently see over-utilization and drive prices up without bound. Both cases produce **incorrect fee, gas, and resource accounting with direct economic impact** on all users submitting transactions.

This matches the allowed impact: *"Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

---

### Likelihood Explanation

The two values live in different configuration namespaces (`batcher_config` vs. versioned constants JSON files) and are updated by different actors (node operator vs. protocol upgrade). The versioned constants have already changed `max_block_size` three times across five versions (0.14.0–0.14.4). Each such upgrade creates an opportunity for the values to diverge if the operator does not simultaneously update the batcher config. The external bug report describes exactly this class of accidental misconfiguration.

---

### Recommendation

Add a startup cross-validation (in `validate_node_config` or at `Bouncer::new` time) that asserts:

```
batcher_config.block_max_capacity.receipt_l2_gas
    == VersionedConstants::latest_constants().max_block_size
```

Return a hard `ConfigError` (not just a log warning) if the invariant is violated, mirroring the pattern already used for other cross-member constraints in `cross_member_validations`. Additionally, consider deriving `receipt_l2_gas` directly from `VersionedConstants::max_block_size` at construction time rather than requiring it to be set independently.

---

### Proof of Concept

1. Start a node with the default config but override the batcher bouncer config:
   ```
   batcher_config.static_config.block_builder_config.bouncer_config
     .block_max_capacity.receipt_l2_gas = 500000000
   ```
   (below the current `gas_target` of 1040000000 in v0.14.3/v0.14.4)

2. The Bouncer closes every block after accumulating only 500000000 units of `receipt_l2_gas`.

3. `calculate_next_l2_gas_price_for_fin` is called with `l2_gas_used ≤ 500000000`, while `gas_target = 1040000000`. Since `gas_used < gas_target` on every block, the EIP-1559 formula always decreases the price: [9](#0-8) 

4. The gas price converges to `min_gas_price` and stays there permanently, regardless of actual transaction volume. All fee estimates returned by the RPC are wrong (too low), and the economic signal of the fee market is broken.

5. No error is logged or returned at node startup because `validate_node_config` contains no check for this invariant.

/// Compute the next L2 gas price (for the fin or for updating state). Respects override when set.
pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    override_l2_gas_price_fri: Option<u128>,
    min_l2_gas_price_per_height: &[PricePerHeight],
    fee_actual: Option<GasPrice>,
) -> GasPrice {
    if let Some(override_value) = override_l2_gas_price_fri {
        info!(
            "L2 gas price ({}) is not updated, remains on override value of {override_value} fri",
            current_l2_gas_price.0
        );
        return GasPrice(override_value);
    }
    let gas_target = VersionedConstants::latest_constants().gas_target;
    let config_min = get_min_gas_price_for_height(height, min_l2_gas_price_per_height);
    let effective_min = match fee_actual {
        Some(fa) => GasPrice(max(config_min.0, fa.0)),
        None => config_min,
    };
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
}
