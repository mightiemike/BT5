### Title
Tier Bonus Rate Applied Retroactively to Historical Segments on Claim — (`File: x/tieredrewards/keeper/claim_rewards.go`)

### Summary
`processEventsAndClaimBonus` fetches the **current** tier parameters at claim time and applies them uniformly to every historical bonded segment since the position's last claim. If governance increases a tier's bonus rate between a position's last claim and the current claim, the position owner receives the higher rate retroactively over the entire unclaimed window — including the period that predates the rate increase. This is the direct analog of the MixinRefunds frontrun: instead of buying at the old price and getting refunded at the new price, a position holder simply delays claiming until after a governance-approved rate increase and receives the higher rate for the full historical window.

---

### Finding Description

In `processEventsAndClaimBonus`, the tier is loaded once at the start of the function using the current on-chain state:

```go
tier, err := k.getTier(ctx, pos.TierId)   // line 167 — current tier, not creation-time tier
```

This single `tier` object is then passed into `computeSegmentBonus` for **every** historical validator-event segment:

```go
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
// line 178 — current tier rate applied to a historical segment
```

And again for the trailing live segment after all events are replayed:

```go
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
// line 211 — same current tier rate
```

The tier's bonus rate (the multiplier that scales delegation value into bonus tokens) is therefore the **current** governance-set value, not the value that was in effect during each segment. No snapshot of the tier rate is stored per-position or per-event.

Governance can update any tier's parameters at any time via `SetTier`:

```go
func (k Keeper) SetTier(ctx context.Context, tier types.Tier) error {
    if err := tier.Validate(); err != nil { return err }
    if err := k.Tiers.Set(ctx, tier.Id, tier); err != nil { ... }
    return nil
}
// x/tieredrewards/keeper/tier.go lines 27–35
```

There is no mechanism that forces all open positions to checkpoint their accrued bonus before a tier update takes effect.

---

### Impact Explanation

A position holder who has not claimed rewards since block T₀ and whose tier bonus rate is increased at block T₁ (T₀ < T₁ < T_claim) will receive:

```
bonus = BonusRate_new × Σ(segment_value × segment_duration)
```

for **all** segments from T₀ to T_claim, including the segment [T₀, T₁] that predates the rate increase. The correct calculation for that segment should use `BonusRate_old`. The difference:

```
windfall = (BonusRate_new − BonusRate_old) × segment_value[T₀,T₁] × (T₁ − T₀)
```

is paid from the `RewardsPoolName` module account to the position owner without any corresponding contribution. With a large delegation and a long unclaimed window, this windfall can be substantial and drains the shared bonus pool at the expense of other participants.

---

### Likelihood Explanation

Governance proposals to adjust tier bonus rates are a normal operational event (tuning incentives). Any position holder who monitors the governance queue can simply refrain from claiming rewards while a rate-increase proposal is in the voting period, then claim immediately after it passes. No special permissions, private keys, or mempool manipulation are required — the entry path is the standard `MsgClaimTierRewards` transaction signed by the position owner.

---

### Recommendation

Before applying a tier parameter update, force-checkpoint all open positions in that tier by calling `claimRewardsAndUpdateTierPositions` (which already exists for this purpose) as part of the `SetTier` governance execution path. This ensures every position's accrued bonus is settled at the old rate before the new rate takes effect, eliminating the retroactive windfall.

Alternatively, store a per-position snapshot of the tier's bonus rate at the time of the last claim and use that snapshot — not the current tier — when computing bonus for historical segments.

---

### Proof of Concept

1. Tier 1 has `BonusRate = 5%`. Alice creates a position with 1,000,000 basecro and does not claim rewards.
2. After 30 days, a governance proposal to increase Tier 1's `BonusRate` to `10%` passes.
3. Alice immediately submits `MsgClaimTierRewards`.
4. `processEventsAndClaimBonus` fetches the current tier (`BonusRate = 10%`) and applies it to the full 30-day window.
5. Alice receives `10% × 1,000,000 × 30 days` worth of bonus instead of the correct `5% × 1,000,000 × 30 days`, extracting the difference from the shared `RewardsPoolName` account.
