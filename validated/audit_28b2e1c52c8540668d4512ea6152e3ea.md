Audit Report

## Title
Stop-Loss Watermarks Retroactively Collapsed to Zero on Decay-Rate Change — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`executeOracleStopLossDecay` writes the new `decayPerSecondE8` directly into `oracleStopLossConfig` without settling per-bin `BinHighWatermarks` at the old rate. The next `afterSwap` call applies the new rate over the full accumulated `dt = block.timestamp − lastDecayTs`, which includes the entire timelock waiting period plus any prior idle time. When `newRate × dt ≥ 1e8`, `_decayed` returns 0 for every touched bin, permanently disabling the stop-loss guard until watermarks are organically rebuilt.

## Finding Description
`executeOracleStopLossDecay` (lines 139–147) stores the new rate and clears the pending schedule but never touches `lastDecayTs` in any `BinHighWatermarks` entry:

```solidity
oracleStopLossConfig[pool_].decayPerSecondE8 = decay;   // new rate stored
(sched.pendingDecayPerSecondE8, sched.pendingDecayExecuteAfter) = (0, 0);
// lastDecayTs in highWatermarks[pool][*] is never updated
```

On the next swap, `_checkAndUpdateWatermarks` (lines 267–268) computes:

```solidity
BinHighWatermarks storage hwmS = highWatermarks[pool_][binIdx];
uint256 dt = block.timestamp - hwmS.lastDecayTs;
```

`dt` spans from the last swap (before the rate change) to the current block, covering the full timelock period and any idle time. `_decayed` (lines 319–323) then applies the new rate to this full `dt`:

```solidity
uint256 factor = ratePerSecondE8 * dt;
if (factor >= E8) return 0;   // collapses watermark
```

Once `hwm = 0`, `_applyWatermark` (lines 328–336) always returns `(metric, false)` because `metric >= 0` is unconditionally true for a `uint256`, so `breached` is never set and the stop-loss never fires for those bins.

The existing `_validateDecay` guard only enforces `decayPerSecondE8 ≤ E8`; it does not prevent retroactive over-application of the rate across accumulated idle time.

## Impact Explanation
The `OracleValueStopLossExtension` is the primary on-chain guard preventing LP value-per-share from falling below the configured drawdown floor. Collapsing all watermarks to zero silently disables this guard for every bin touched by the next swap. During the resulting unguarded window, any swap that would otherwise have been blocked (e.g., one driven by a stale or manipulated oracle price) executes freely, directly draining LP token balances. This is broken core pool safety functionality with a direct path to loss of LP principal, satisfying the "broken core pool functionality causing loss of funds" impact gate.

## Likelihood Explanation
The trigger is the pool admin executing a timelocked decay-rate change — a routine administrative action explicitly supported by the protocol. The pool admin need not act maliciously; a well-intentioned admin setting a rate of 1157 E8/s (≈100%/day, well within the `E8` cap) will collapse watermarks if the pool has been idle for two days (one day idle before proposal + one-day timelock). The timelock is supposed to give LPs time to react, but the retroactive application collapses watermarks instantaneously at execution time, not gradually, defeating the protection the timelock was designed to provide. This constitutes a fund-impacting cap failure: the `E8` cap on decay rate is rendered ineffective by the retroactive application.

## Recommendation
Before writing the new decay rate, settle all active watermarks at the old rate. Because iterating all bins on-chain is impractical, the safest fix is to record the rate-change timestamp and apply a two-segment decay in `_decayed`: `oldRate × (changeTs − lastDecayTs) + newRate × (block.timestamp − changeTs)`. Alternatively, snapshot `lastDecayTs = block.timestamp` for every bin at execution time (requires an enumerable bin set), or require that `executeOracleStopLossDecay` resets `lastDecayTs` on all bins via an off-chain-supplied list verified on-chain.

## Proof of Concept
1. Deploy pool with `OracleValueStopLossExtension`, `drawdownE6 = 50_000`, `decayPerSecondE8 = 0`, `timelock = 86_400` (1 day).
2. Execute swaps; bin 0 watermarks are set to `hwm0 = hwm1 = 1e6`, `lastDecayTs = T0`.
3. Pool goes idle for 1 day (no swaps).
4. Pool admin calls `proposeOracleStopLossDecay(pool, 1_157)` at `T0 + 86_400`.
5. After the 1-day timelock, admin calls `executeOracleStopLossDecay(pool)` at `T0 + 172_800`.
6. Next swap triggers `afterSwap` → `_checkAndUpdateWatermarks`: `dt = 172_800`, `factor = 1_157 × 172_800 = 199_929_600 > 1e8` → `_decayed` returns 0 for both `hwm0` and `hwm1`.
7. `_applyWatermark(metric, 0, floorMultiplier)` → `metric >= 0` always true → `breached = false`. Stop-loss never fires.
8. Subsequent swaps that drain LP value below the 5% drawdown floor proceed without revert.
