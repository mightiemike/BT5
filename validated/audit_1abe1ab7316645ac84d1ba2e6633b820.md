### Title
Residual Allowance from Skipped Deposit Permanently Bricks `creditDeposit()` for Non-Standard ERC20 Tokens — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` before invoking `endpoint.depositCollateralWithReferral(...)`. When the deposit amount falls below the protocol minimum, `depositCollateralWithReferral` silently returns early without pulling any tokens, leaving a non-zero allowance. On the next invocation of `creditDeposit()`, tokens like USDT — which require the allowance to be reset to zero before setting a new non-zero value — will revert on the `approve` call, permanently bricking the DDA for that token.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`, for each registered product token the function:

1. Reads the DDA's token balance.
2. Calls `token.approve(address(endpoint), balance)` unconditionally.
3. Calls `endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1")`. [1](#0-0) 

In `Endpoint.depositCollateralWithReferral`, if `isValidDepositAmount` returns `false` (i.e., the amount is below the minimum deposit threshold), the function returns **early** without calling `handleDepositTransfer` and without consuming any allowance: [2](#0-1) 

This means the full `balance` allowance set in step 2 remains unconsumed. On the next call to `creditDeposit()` for the same token, `token.approve(address(endpoint), newBalance)` is called while the current allowance is still non-zero. For USDT-like tokens that enforce `require(currentAllowance == 0 || newAllowance == 0)`, this call reverts, causing the entire `creditDeposit()` loop to revert and permanently preventing any future deposits through this DDA for that token.

There is a second, lower-likelihood path: `balance` is a `uint256` but `depositCollateralWithReferral` accepts only `uint128 amount`. If `balance > type(uint128).max`, the endpoint pulls only `uint128(balance)` tokens, leaving a residual allowance of `balance - uint128(balance)`, triggering the same revert on the next call. [3](#0-2) 

---

### Impact Explanation

The DDA for the affected subaccount is permanently bricked for the targeted token. All future calls to `creditDeposit()` revert at the `approve` step for that token, meaning:

- Legitimate user funds already sitting in the DDA cannot be credited to the subaccount.
- The only recovery path is `ContractOwner.withdrawFromDirectDepositV1()`, which requires owner action and does not restore normal deposit functionality.

The corrupted state is the ERC20 allowance mapping: `token.allowance(DDA, endpoint) != 0` while `creditDeposit()` assumes it is always zero at entry. [4](#0-3) 

---

### Likelihood Explanation

`creditDeposit()` has no access control — any caller can invoke it via `ContractOwner.creditDepositV1(subaccount)` (also permissionless) or directly on the DDA. [5](#0-4) 

An attacker can:
1. Send a dust amount of USDT (below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT`) to the DDA address.
2. Call `creditDepositV1(subaccount)` — this sets the allowance but the deposit is skipped.
3. The DDA is now bricked for USDT.

The minimum deposit check in `depositCollateralWithReferral` is explicitly designed to not revert (to avoid blocking other assets), making the silent skip — and thus the residual allowance — an intended code path that is reliably reachable. [6](#0-5) 

---

### Recommendation

Reset the allowance to zero before setting a new value, mirroring the correct pattern already used in `ContractOwner.wrapVaultAsset()`:

```solidity
// In DirectDepositV1.creditDeposit(), replace:
token.approve(address(endpoint), balance);

// With:
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
```

Alternatively, use `safeIncreaseAllowance` after first resetting to zero, or use `forceApprove` from OpenZeppelin's `SafeERC20` library, which handles the reset-to-zero pattern internally. [7](#0-6) 

---

### Proof of Concept

1. A DDA exists for `subaccount` (created via `ContractOwner.createDirectDepositV1`).
2. USDT is a registered spot product in `SpotEngine`.
3. Attacker sends `1 wei` of USDT to the DDA address.
4. Attacker calls `ContractOwner.creditDepositV1(subaccount)`.
   - `creditDeposit()` executes: `token.approve(endpoint, 1)` — allowance is now `1`.
   - `depositCollateralWithReferral(subaccount, productId, 1, "-1")` is called.
   - `isValidDepositAmount` returns `false` (1 wei is below minimum).
   - Function returns early. Allowance remains `1`.
5. Legitimate user sends `1000 USDT` to the DDA and calls `creditDepositV1(subaccount)`.
   - `creditDeposit()` executes: `token.approve(endpoint, 1000e6)`.
   - USDT reverts: current allowance is `1 != 0`.
   - `creditDeposit()` reverts. Funds are stuck. [8](#0-7)

    function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
        external
        onlyOwner
    {
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        if (token == address(0)) {
            uint256 preBalance = address(this).balance;
            DirectDepositV1(directDepositV1).withdrawNative();
            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");
            (bool success, ) = msg.sender.call{value: postBalance - preBalance}(
                ""
            );
            require(success, "xfer");
        } else {
            uint256 preBalance = IERC20Base(token).balanceOf(address(this));
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));
            require(postBalance > preBalance, "empty");
            IERC20Base(token).safeTransfer(
                msg.sender,
                postBalance - preBalance
            );
        }
    }
