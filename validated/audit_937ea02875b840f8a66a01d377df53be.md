### Title
Linked Signer Can Hijack Subaccount by Replacing Its Own Authorization — (`core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction handler in `EndpointTx.sol` validates the transaction signature with `allowLinkedSigner = true`, meaning the **current linked signer** is accepted as a valid signer for a `LinkSigner` transaction. This allows a malicious linked signer to replace itself with any attacker-controlled address in `linkedSigners[subaccount]`, without the subaccount owner's knowledge or consent. The slow-mode path correctly restricts this operation to the subaccount owner, but the fast-mode path does not.

---

### Finding Description

**Root cause — asymmetric authorization in fast-mode vs. slow-mode `LinkSigner` handling:**

In the fast-mode (sequencer-submitted) path, `LinkSigner` is handled at: [1](#0-0) 

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes `getLinkedSigner(sender)` as the accepted signer: [2](#0-1) 

This means the **existing linked signer** is a valid signer for a `LinkSigner` transaction that overwrites `linkedSigners[signedTx.tx.sender]` with any arbitrary address.

The `LinkSigner` struct is: [3](#0-2) 

```solidity
struct LinkSigner {
    bytes32 sender;   // victim subaccount
    bytes32 signer;   // new signer — attacker-controlled
    uint64 nonce;
}
```

By contrast, the slow-mode path correctly restricts `LinkSigner` to the subaccount owner: [4](#0-3) 

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // requires msg.sender == subaccount owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

The slow-mode uses `validateSender`, which enforces that `msg.sender` is the actual owner address of the subaccount. The fast-mode has no equivalent restriction — it accepts the linked signer as a valid authorizer for changing the linked signer.

---

### Impact Explanation

A malicious linked signer can:

1. Craft a `SignedLinkSigner` with `sender = victim subaccount`, `signer = attacker address`, sign it with the current linked signer key.
2. Submit it to the sequencer for inclusion in a `submitTransactionsChecked` batch.
3. After execution, `linkedSigners[victim subaccount]` is overwritten with the attacker's address.
4. The attacker's address is now the linked signer and can sign all trading operations (orders, `TransferQuote`, etc.) on behalf of the victim subaccount.
5. The attacker can drain the subaccount via unfavorable trades or direct quote transfers to an attacker-controlled subaccount.

The corrupted state is `linkedSigners[subaccount]` in `EndpointStorage`. The asset delta is the full balance of the victim subaccount across all products. [5](#0-4) 

---

### Likelihood Explanation

Any user who has ever granted a linked signer (e.g., for API trading) is exposed. The linked signer is a commonly used feature for programmatic trading. A malicious or compromised linked signer key can exploit this with a single sequencer-submitted transaction. No admin access, governance capture, or privileged key is required — only possession of the current linked signer's private key.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the fast-mode `LinkSigner` handler, consistent with the slow-mode path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This ensures that only the subaccount owner (whose address is embedded in `signedTx.tx.sender`) can authorize a change to their own linked signer, matching the invariant enforced in the slow-mode path. [6](#0-5) 

---

### Proof of Concept

**Preconditions:**
- Alice (`0xAlice...`) has subaccount `aliceSub = bytes32(abi.encodePacked(address(0xAlice), uint96(0)))`.
- Alice previously set Bob (`0xBob...`) as her linked signer via a legitimate `LinkSigner` slow-mode transaction.
- `linkedSigners[aliceSub] == 0xBob`.

**Attack steps:**

1. Bob constructs:
   ```solidity
   IEndpoint.LinkSigner memory lsTx = IEndpoint.LinkSigner({
       sender: aliceSub,
       signer: bytes32(bytes20(0xAttacker)),
       nonce: currentNonce(aliceSub)
   });
   IEndpoint.SignedLinkSigner memory signed = IEndpoint.SignedLinkSigner({
       tx: lsTx,
       signature: bobSign(digest(lsTx))
   });
   ```
2. Bob submits `abi.encodePacked(uint8(TransactionType.LinkSigner), abi.encode(signed))` to the sequencer.
3. Sequencer includes it in `submitTransactionsChecked`.
4. `validateSignedTx(aliceSub, nonce, txn, bobSignature, true)` passes — Bob is the current linked signer.
5. `linkedSigners[aliceSub]` is now `0xAttacker`.
6. Attacker signs a `TransferQuote` from `aliceSub` to `attackerSub`, draining Alice's quote balance. [1](#0-0) [7](#0-6)
