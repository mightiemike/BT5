Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the end-user, allowing any unprivileged caller to bypass the swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is always `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. Because the router is a single shared public contract, allowlisting it grants every on-chain address the ability to bypass the per-user allowlist, fully defeating the pool admin's intended access control.

## Finding Description
`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension via `_callExtensionsInOrder`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is in `allowedSwapper[msg.sender][sender]`, where `msg.sender` inside the extension is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The same pattern holds for `exactInput` (all hops): [5](#0-4) 

And for `exactOutputSingle` and `exactOutput` recursive hops via `_exactOutputIterateCallback`: [6](#0-5) 

The router stores the originating user only in transient storage for payment purposes (`_setNextCallbackContext` with `msg.sender` as payer), but this information is never surfaced to the pool or extension as the swap initiator. The extension has no mechanism to distinguish which end-user triggered the router call. The allowlist entry `allowedSwapper[pool][router]` is either present or absent for all users simultaneously — there is no configuration that allows only allowlisted users to swap through the router while blocking others.

## Impact Explanation
This is an admin-boundary break. The pool admin's intended invariant — "only approved addresses may swap in this pool" — is fully defeated. Any address can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) targeting a pool with `SwapAllowlistExtension` configured. If the pool admin has added the router to the allowlist (the only way to let allowlisted users use the router), every address on-chain can swap in the restricted pool. The corrupted value is the `allowedSwapper[pool][sender]` extension decision: it evaluates `true` for the router address regardless of who the actual end-user is, causing the extension to return `IMetricOmmExtensions.beforeSwap.selector` instead of reverting with `NotAllowedToSwap`.

## Likelihood Explanation
The router is a public, permissionless contract requiring no special role, token balance, or prior state. Any user who discovers a pool uses `SwapAllowlistExtension` can immediately route through `MetricOmmSimpleRouter` to bypass it. The bypass is active whenever the pool admin has allowlisted the router — which is the only practical configuration for allowlisted users who want router access. Likelihood is high in any real deployment where the allowlist is intended to restrict access while still supporting router-based UX.

## Recommendation
Pass the economically meaningful actor — the end user — as `sender` to the extension, not the immediate `msg.sender` of `pool.swap()`. One approach: the router stores the originating user in transient storage (it already uses transient storage for callback context via `_setNextCallbackContext`) and exposes it through a trusted interface that the pool reads before invoking extensions, or the pool accepts an explicit `originator` parameter that the router fills with `msg.sender` before calling `pool.swap()`. Alternatively, `SwapAllowlistExtension` could be redesigned to gate on the payer identity surfaced through the callback (already stored in transient storage as `_getPayer()`) rather than the `sender` hook argument.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — required so that allowlisted user Alice can use the router.
3. Bob (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` — pool's `msg.sender` = router address.
5. Pool calls `_beforeSwap(msg.sender=router, ...)` → extension receives `sender=router`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
7. Bob's swap executes successfully despite not being on the allowlist.

The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` since all entry points call `pool.swap()` directly from the router contract. [7](#0-6) [8](#0-7)
