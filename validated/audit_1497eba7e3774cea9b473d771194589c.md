### Title
Lack of Slippage Check in ERC4626 Vault Deposit in `wrapVaultAsset` Could Lead to Users' Collateral Loss - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.wrapVaultAsset()` converts a user's underlying ERC4626 asset tokens (held in their `DirectDepositV1` address) into ERC4626 vault shares and deposits them as collateral. The function calls `IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1)` with no minimum-shares check. The only guard is `previewDeposit(assetBalance) != 0`, which only ensures non-zero output — not a minimum acceptable amount. Because `wrapVaultAsset` is also callable by any unprivileged external address (no access control modifier), an adversary can time the call to execute when the vault's exchange rate is unfavorable, causing the user to receive fewer vault shares than expected for their underlying assets.

---

### Finding Description

`ContractOwner.wrapVaultAsset()` performs the following steps:

1. Reads `assetBalance` — the underlying token balance held in the user's `DirectDepositV1` contract.
2. Calls `previewDeposit(assetBalance)` and proceeds only if the result is non-zero.
3. Pulls the underlying tokens from `DirectDepositV1` into `ContractOwner` via `DirectDepositV1.withdraw()`.
4. Approves the ERC4626 vault and calls `IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1)`.

Step 4 has no minimum-shares guard. The `previewDeposit` check at step 2 is not a slippage check — it only gates on non-zero output. The actual shares minted by `deposit()` can be arbitrarily lower than `previewDeposit` suggested if the vault's exchange rate changes between the preview and the actual deposit call.

Additionally, `wrapVaultAsset` carries no `onlyOwner` or equivalent modifier, so any external caller can invoke it for any subaccount at any time. [1](#0-0) 

The `IERC4626Base` interface exposes only `deposit`, `asset`, and `previewDeposit` — there is no `minSharesOut` parameter available through this interface, and none is enforced at the call site. [2](#0-1) 

---

### Impact Explanation

The vault shares minted by `deposit()` are sent directly to `directDepositV1`, which then calls `endpoint.depositCollateralWithReferral()` to credit the user's subaccount. If the exchange rate is degraded at execution time, the user's subaccount receives fewer vault-share units of collateral than their underlying assets were worth. This directly reduces the user's collateral balance, lowers their subaccount health score, and can push them closer to (or into) liquidation — a concrete, measurable asset delta against the user. [3](#0-2) 

---

### Likelihood Explanation

The function is `external` with no access control, so any unprivileged caller can trigger it for any subaccount at any time. An adversary can monitor the mempool for users sending underlying tokens to their `DirectDepositV1` address, wait for or cause an unfavorable exchange rate event in the ERC4626 vault (e.g., a loss realization, a first-depositor inflation attack on a low-TVL vault, or a fee-on-deposit vault), and then call `wrapVaultAsset` to lock in the unfavorable rate. The protocol does not specify which ERC4626 vaults are integrated, so vaults susceptible to exchange rate manipulation must be considered in scope. [4](#0-3) 

---

### Recommendation

1. **Add a `minSharesOut` parameter** to `wrapVaultAsset` and revert if the actual shares returned by `deposit()` are below it:

```solidity
function wrapVaultAsset(bytes32 subaccount, uint32 productId, uint256 minSharesOut) external {
    ...
    uint256 sharesReceived = IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
    require(sharesReceived >= minSharesOut, "Slippage too high");
}
```

2. **Restrict the caller** to the subaccount owner (i.e., `require(address(bytes20(subaccount)) == msg.sender)`) so only the affected user can trigger the wrap, preventing adversarial timing.

---

### Proof of Concept

1. Alice sends 1000 USDC (the ERC4626 underlying) to her `DirectDepositV1` address.
2. The ERC4626 vault currently has an exchange rate of 1 USDC → 1 share (`previewDeposit(1000e6) = 1000e18`).
3. Before Alice or the off-chain relayer calls `wrapVaultAsset`, an attacker observes the pending state and triggers a loss event in the vault (e.g., via a first-depositor inflation attack on a low-TVL vault), degrading the exchange rate to 1 USDC → 0.5 shares.
4. The attacker calls `ContractOwner.wrapVaultAsset(alice_subaccount, productId)` — no access control blocks this.
5. `previewDeposit(1000e6)` returns a non-zero value (500e18), so the guard passes.
6. `IERC4626Base(tokenAddr).deposit(1000e6, directDepositV1)` mints only 500 shares instead of the expected 1000.
7. Alice's subaccount is credited with 500 vault shares worth of collateral instead of 1000, halving her effective collateral and potentially triggering liquidation. [5](#0-4)
