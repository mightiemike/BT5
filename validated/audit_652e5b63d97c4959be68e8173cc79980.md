Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Trader, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender` at the time `swap()` is called. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. Any admin who allowlists the router to enable legitimate router-mediated swaps simultaneously opens the gate to every non-allowlisted user, completely nullifying the allowlist policy for the primary public entrypoint.

## Finding Description

**Root cause — three confirmed code facts:**

1. `MetricOmmPool.swap` passes its own `msg.sender` (the router) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

2. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()` — the router, not the actual user: [2](#0-1) 

3. `MetricOmmSimpleRouter.exactInputSingle` stores the real `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext`), but never forwards it to the pool or extension. The pool call carries no originator identity: [3](#0-2) 

**Exploit flow:**

```
attacker → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(recipient, ...)          // msg.sender = router
         → _beforeSwap(sender=router, ...)
         → SwapAllowlistExtension.beforeSwap
         → allowedSwapper[pool][router] == true  ← checked, NOT attacker
         → swap executes
```

The admin faces an inescapable dilemma: not allowlisting the router blocks all legitimate router users; allowlisting the router opens the gate to every non-allowlisted address. There is no configuration that simultaneously permits router-mediated swaps for allowlisted users and blocks non-allowlisted users. The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) [5](#0-4) 

## Impact Explanation

The allowlist policy is completely nullified for all router-mediated swap paths, which are the canonical public entrypoints. Any non-allowlisted user can execute swaps on pools intended to be restricted (e.g., KYC-gated, regulatory-gated, or counterparty-restricted pools) by routing through `MetricOmmSimpleRouter`. This directly matches the "Allowlist path" smart audit pivot: "swap allowlist checks must cover the exact actor/action intended and cannot be bypassed through router." The broken invariant is `allowedSwapper[pool][actual_trader]` — the extension evaluates the wrong address, permitting swaps that should be rejected. This constitutes broken core pool access-control functionality with direct fund-impacting consequences for restricted pools.

## Likelihood Explanation

The router is the primary documented user-facing entrypoint. Any operator deploying a pool with `SwapAllowlistExtension` who wants allowlisted users to be able to use the router must allowlist the router address — this is the only operational path available. The bypass condition is therefore the expected production configuration, making exploitation trivially reachable by any unprivileged address with no special capability required beyond calling the public router.

## Recommendation

Pass the original caller's identity through the extension data or as a dedicated parameter so the extension gates the economic actor rather than the intermediary. Concrete options:

- Have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that value when `sender` is a known router address.
- Add an explicit `originator` parameter to `pool.swap()` that the router populates with its `msg.sender`, and have extensions gate on `originator` instead of `sender`.
- Alternatively, maintain a registry of trusted routers in the extension; when `sender` is a trusted router, decode the actual trader from `extensionData` and apply the allowlist check to that address.

// Exploit: attacker (not on allowlist) routes through the router
vm.prank(attacker);

    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""

    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
