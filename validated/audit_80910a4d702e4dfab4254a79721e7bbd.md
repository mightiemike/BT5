### Title
Zero-Signature Fast Withdrawal Authorization Bypass During Key Rotation Window — (`core/contracts/Verifier.sol`)

---

### Summary

`requireValidTxSignatures` enforces `nSignatures == nSigner`. When `nSigner == 0` (all pubkeys deleted), passing `signatures = []` satisfies this check trivially, allowing `submitFastWithdrawal` to execute with zero cryptographic authorization. This is exploitable during any key rotation window where the owner deletes all existing keys before assigning replacements.

---

### Finding Description

**`requireValidTxSignatures` when `nSigner == 0` and `signatures = []`:** [1](#0-0) 

- `nSignatures` starts at `0`
- The `for` loop body never executes (array length is 0)
- `require(0 == 0, "not enough signatures")` → **passes**

**`checkQuorum` when `nSigner == 0` and `signerBitmask == 0`:** [2](#0-1) 

- `nSigned` stays `0`
- `return 0 * 2 > 0` → **returns `false`**

The two authorization paths are arithmetically inconsistent at the `nSigner == 0` boundary. `checkQuorum` correctly rejects the zero-signer state; `requireValidTxSignatures` does not.

**`submitFastWithdrawal` calls only `requireValidTxSignatures`:** [3](#0-2) 

There is no secondary quorum check. If `requireValidTxSignatures` passes, the token transfer executes unconditionally.

**The vulnerable state is created by `deletePubkey`:** [4](#0-3) 

Each call decrements `nSigner`. After all assigned keys are deleted, `nSigner == 0`. This is a legitimate operational action (key rotation), not a malicious one. The attacker does not need to control the owner — they only need to observe the on-chain state and submit a transaction during the window between the last `deletePubkey` and the first `assignPubKey`.

---

### Impact Explanation

An attacker can drain the `BaseWithdrawPool` of any token balance by submitting a crafted `submitFastWithdrawal` call with `signatures = []` during the rotation window. The call passes all guards (`markedIdxs`, `minIdx`, `requireValidTxSignatures`) and transfers the full requested `amount` to an attacker-controlled address. This directly destroys protocol solvency and constitutes unauthorized asset extraction.

---

### Likelihood Explanation

Key rotations are infrequent but are a documented operational procedure. The window is one or more blocks between the last `deletePubkey` and the first `assignPubKey`. An attacker monitoring the mempool or chain state can detect the `DeletePubkey` event and front-run or race the `assignPubKey` transaction. The attack requires no special privileges and is executable by any EOA.

---

### Recommendation

Add an explicit guard in `requireValidTxSignatures` rejecting the zero-signer state:

```solidity
require(nSigner > 0, "no signers registered");
require(nSignatures == nSigner, "not enough signatures");
```

Additionally, enforce atomic key rotation at the operational level (add new keys before removing old ones), and consider adding a `require(nSigner > 0)` guard in `submitFastWithdrawal` itself as defense-in-depth.

---

### Proof of Concept

```solidity
// State-machine test on unmodified Hardhat setup
// 1. Deploy Verifier, assign pubkeys at index 0 and 1
verifier.assignPubKey(0, px0, py0);  // nSigner = 1
verifier.assignPubKey(1, px1, py1);  // nSigner = 2

// 2. Owner performs key rotation — deletes both keys
verifier.deletePubkey(0);  // nSigner = 1
verifier.deletePubkey(1);  // nSigner = 0  ← vulnerable window opens

// 3. Attacker calls submitFastWithdrawal with empty signatures
//    (any valid idx > minIdx, any well-formed withdrawal transaction)
withdrawPool.submitFastWithdrawal(idx, withdrawalTxBytes, new bytes[](0));
// requireValidTxSignatures: nSignatures(0) == nSigner(0) → passes
// Funds transferred to attacker's address

    function checkQuorum(uint8 signerBitmask) internal view returns (bool) {
        uint256 nSigned = 0;
        for (uint256 i = 0; i < 8; ++i) {
            bool signed = ((signerBitmask >> i) & 1) == 1;
            if (signed) {
                if (isPointNone(pubkeys[i])) {
                    return false;
                }
                nSigned += 1;
            }
        }
        return nSigned * 2 > nSigner;
    }

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }
```
