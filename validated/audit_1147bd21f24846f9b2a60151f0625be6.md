### Title
Compromised Linked Signer Can Perpetuate Its Own Authorization Indefinitely via Self-Re-Linking — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in `EndpointTx.sol` is processed with `allowLinkedSigner = true`, meaning the **currently active linked signer can sign a new `LinkSigner` transaction** to replace itself with any address. Once a linked signer key is compromised, the attacker can continuously re-link a new attacker-controlled address before the legitimate owner can revoke access, creating a persistent, irrevocable authorization state — the on-chain analog of a wallet session that remains unlocked after the browser is closed.

---

### Finding Description

The `linkedSigners` mapping in `EndpointStorage.sol` stores a delegated signing address per subaccount with no expiry, no timeout, and no automatic invalidation. [1](#0-0) 

When the sequencer processes a `LinkSigner` transaction via `processTransactionImpl`, it calls `validateSignedTx` with `allowLinkedSigner = true`: [2](#0-1) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the authorized signer via `getLinkedSigner` and passes it to `verifier.validateSignature`, which accepts a signature from **either** the subaccount owner address **or** the current linked signer: [3](#0-2) [4](#0-3) 

This means the current linked signer can unilaterally issue a new `LinkSigner` transaction — signed with its own key — to overwrite `linkedSigners[subaccount]` with a new attacker-controlled address. The owner's nonce is consumed by the attacker's transaction, causing any concurrent revocation attempt by the legitimate owner to fail with `ERR_WRONG_NONCE`. [5](#0-4) 

The slow-mode path for `LinkSigner` does enforce `validateSender` (requiring `msg.sender` to match the subaccount owner address), which would allow the owner to eventually revoke via slow mode: [6](#0-5) 

However, slow-mode transactions are subject to a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`): [7](#0-6) 

During those 3 days, the attacker retains full linked-signer authority and can drain the account or re-link again via the fast path before the slow-mode revocation executes.

---

### Impact Explanation

With persistent linked signer access, the attacker can sign any transaction type that accepts `allowLinkedSigner = true`:

- **`TransferQuote`** (`allowLinkedSigner = true`): Transfer quote-asset balances from the victim's subaccount to an attacker-controlled registered subaccount — direct asset theft. [8](#0-7) 

- **`WithdrawCollateral` (V1)** (`allowLinkedSigner = true`): Initiate withdrawals (funds go to the subaccount owner's address, limiting direct theft here, but usable for griefing/fee drain). [9](#0-8) 

- **`MintNlp` / `BurnNlp`** (`allowLinkedSigner = true`): Manipulate NLP positions at attacker-chosen oracle prices, causing value loss. [10](#0-9) 

- **`LiquidateSubaccount`** (`allowLinkedSigner = true`): Initiate liquidations of other accounts, earning fees at the victim's expense. [11](#0-10) 

- **`LinkSigner` (self-re-link)**: Perpetuate the authorization indefinitely, defeating any owner revocation attempt via the fast path. [2](#0-1) 

The corrupted state is `linkedSigners[subaccount]` — an authorization mapping that should be exclusively owner-controlled but is writable by the entity it authorizes.

---

### Likelihood Explanation

Likelihood is **Medium**. The attack requires an initial linked signer key compromise (device loss, key leak, phishing). This is a realistic threat model for hot-wallet linked signers, which are the primary use case for the `LinkSigner` feature. Once the key is compromised, the persistence attack requires no special privileges — any address holding the compromised key can execute it permissionlessly through the standard sequencer submission path.

---

### Recommendation

1. **Disallow linked signer self-modification**: Change `allowLinkedSigner` to `false` for `LinkSigner` transaction processing. Only the subaccount owner (the address embedded in the `bytes32 sender`) should be permitted to change the linked signer.

   ```solidity
   // In processTransactionImpl, LinkSigner branch:
   validateSignedTx(
       signedTx.tx.sender,
       signedTx.tx.nonce,
       transaction,
       signedTx.signature,
       false  // was: true — linked signer must NOT be able to re-link
   );
   ```

2. **Add an expiry field to `LinkSigner`**: Introduce a `uint64 expiresAt` field in the `LinkSigner` struct and enforce it in `getLinkedSigner`, so linked signer authorizations automatically expire — analogous to session timeouts.

3. **Enforce owner-only revocation in the fast path**: Alternatively, add a separate `RevokeSigner` transaction type that only accepts a signature from the subaccount owner address (never the linked signer), ensuring revocation cannot be front-run by the compromised key.

---

### Proof of Concept

1. Alice sets `linkedSigners[alice_subaccount] = hot_wallet` via a valid `LinkSigner` transaction. Alice's nonce advances to `N`.
2. `hot_wallet` key is compromised. Attacker now controls `hot_wallet`.
3. Alice attempts to revoke: signs `LinkSigner(sender=alice_subaccount, signer=0x0, nonce=N)` and submits to the sequencer.
4. Attacker simultaneously signs `LinkSigner(sender=alice_subaccount, signer=attacker_wallet, nonce=N)` using `hot_wallet` and submits to the sequencer.
5. Sequencer processes the attacker's transaction first (both are valid; sequencer ordering is not guaranteed to favor the owner). Alice's nonce is now `N+1`; her revocation fails with `ERR_WRONG_NONCE`.
6. Attacker now controls `linkedSigners[alice_subaccount] = attacker_wallet`. Attacker signs `TransferQuote(sender=alice_subaccount, recipient=attacker_subaccount, amount=X)` — funds are transferred.
7. Alice submits a slow-mode `LinkSigner` revocation. It is queued with `executableAt = block.timestamp + 3 days`. During those 3 days, the attacker continues operating with full linked-signer authority and can drain remaining assets or re-link again via the fast path. [2](#0-1) [5](#0-4) [7](#0-6)
