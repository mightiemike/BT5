### Title
Silently-Dropped Slow Mode `DepositCollateral` Transaction Permanently Locks User Tokens — (`File: core/contracts/Endpoint.sol`)

---

### Summary

`_executeSlowModeTransaction` deletes a slow mode transaction from the queue **before** attempting to execute it, and wraps execution in a `try/catch` that silently discards any revert. For `DepositCollateral` slow mode entries, the user's ERC-20 tokens are already held by the `Endpoint` contract at queue time. If execution reverts for any reason (e.g., product delisted in the intervening window), the transaction is permanently gone, the tokens are permanently locked in the contract, and the user has no resubmission path.

---

### Finding Description

`depositCollateralWithReferral` in `Endpoint.sol` first pulls tokens from the caller into the contract, then enqueues a `SlowModeTx`: [1](#0-0) 

Later, `_executeSlowModeTransaction` deletes the entry from `slowModeTxs` **before** calling `processSlowModeTransaction`: [2](#0-1) 

Execution is then attempted inside a `try/catch` that silently swallows any non-OOG revert. The comment at line 226 explicitly acknowledges that a fund-return path was deliberately removed: [3](#0-2) 

Inside `processSlowModeTransactionImpl`, the `DepositCollateral` branch calls `clearinghouse.depositCollateral(txn)` with no further guard: [4](#0-3) 

If `clearinghouse.depositCollateral` reverts — for example because the product was delisted via a `DelistProduct` slow mode tx that executed in the same 3-day window — the outer `catch` block discards the error, the `SlowModeTx` record is already gone, and the tokens remain in the `Endpoint` contract with no accounting entry and no recovery function.

---

### Impact Explanation

A user's deposited ERC-20 tokens are permanently locked inside the `Endpoint` contract. The `slowModeTxs` mapping entry has been deleted, so there is no on-chain record to replay. No refund function exists. The user cannot resubmit because the tokens are already in the contract and the queue slot is gone. This is a direct, concrete asset loss for any affected depositor.

---

### Likelihood Explanation

The slow mode queue has a 3-day execution delay: [5](#0-4) 

During that window, an owner-submitted `DelistProduct` slow mode tx can execute first, removing the product. When the `DepositCollateral` tx then executes, `clearinghouse.depositCollateral` reverts on the now-invalid product. The sequencer can also trigger this path via `ExecuteSlowMode` at any ordering it chooses. The 3-day window makes the race realistic, not theoretical.

---

### Recommendation

Restore the fund-return path that was removed (acknowledged by the `// try return funds now removed` comment). On a caught revert for `DepositCollateral`, the contract should transfer the deposited tokens back to the original sender. Alternatively, do not delete the `SlowModeTx` entry until after successful execution, so a failed tx can be retried or refunded.

---

### Proof of Concept

1. User calls `depositCollateralWithReferral` for product `P`. Tokens are transferred to `Endpoint`; a `SlowModeTx` is enqueued at index `N`.
2. Owner submits a `DelistProduct` slow mode tx for product `P`. It executes first (owner can submit and the sequencer can order it ahead).
3. The sequencer (or anyone after 3 days) calls `executeSlowModeTransaction` for index `N`.
4. `_executeSlowModeTransaction` executes: `delete slowModeTxs[N]` fires at line 194, permanently removing the record.
5. `processSlowModeTransaction` → `clearinghouse.depositCollateral` reverts because product `P` no longer exists.
6. The `catch` block at line 207 swallows the revert. No funds are returned.
7. User's tokens remain in `Endpoint` with no accounting entry and no recovery path. [6](#0-5)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
