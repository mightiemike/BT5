### Title
`SwapAllowlistExtension.beforeSwap` Checks Wrong Actor (`sender` = Router) Instead of End User, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which is `msg.sender` of `pool.swap()` — the direct pool caller (e.g., `MetricOmmSimpleRouter`), not the actual end user. When a pool admin allowlists the router to enable router-based swaps for their curated pool, every unprivileged user can bypass the per-user allowlist by routing through the router. This is structurally inconsistent with `DepositAllowlistExtension`, which correctly checks `owner` (the actual position owner) rather than `sender` (the LiquidityAdder intermediary).

---

### Finding Description

**Root cause — wrong actor bound in `beforeSwap`:**

`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap()`: [1](#0-0) 

`_beforeSwap` forwards it verbatim to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[pool][sender]`: [3](#0-2) 

When `MetricOmmSimpleRouter` calls `pool.swap()`, `msg.sender` from the pool's perspective is the **router address**, not the end user. So `sender` = router, and the allowlist check is `allowedSwapper[pool][router]`.

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` ignores `sender` entirely and checks `owner` — the actual position owner passed explicitly by the caller: [4](#0-3) 

`pool.addLiquidity` passes the explicit `owner` parameter (the real user) as the second argument: [5](#0-4) 

So the deposit allowlist correctly gates by user identity regardless of which intermediary calls the pool. The swap allowlist does not.

**Exploit path:**

1. Pool admin deploys a curated pool with `SwapAllowlistExtension` configured as `beforeSwap` extension.
2. Pool admin allowlists specific user addresses: `setAllowedToSwap(pool, userA, true)`.
3. Pool admin also allowlists the router so that allowlisted users can use the standard periphery: `setAllowedToSwap(pool, router, true)`.
4. Non-allowlisted `userB` calls `MetricOmmSimpleRouter.exactInput(...)` targeting the curated pool.
5. The router calls `pool.swap(recipient, ...)` — `msg.sender` from the pool's view is the router.
6. `_beforeSwap(router, recipient, ...)` is called; the extension checks `allowedSwapper[pool][router]` = `true`.
7. The swap succeeds. `userB` has bypassed the per-user allowlist entirely.

The pool admin's intent was to gate by individual user identity. The actual check gates by the direct pool caller. Allowlisting the router to support legitimate users inadvertently opens the gate to all users.

---

### Impact Explanation

A curated pool's swap allowlist is silently bypassed by any unprivileged user who routes through `MetricOmmSimpleRouter` once the router is allowlisted. The pool admin cannot simultaneously (a) allow their allowlisted users to use the router and (b) block non-allowlisted users from using the router. The curation invariant — that only approved addresses may swap — is broken. Unauthorized users can execute swaps, draining LP-provided liquidity at oracle-quoted prices, causing direct loss of LP assets and curation failure.

Severity: **High** — broken core pool functionality (allowlist guard fails open), direct loss of LP assets on curated pools.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the standard supported periphery swap path. A pool admin who configures `SwapAllowlistExtension` and wants their allowlisted users to use the router will naturally call `setAllowedToSwap(pool, router, true)`. The inconsistency with `DepositAllowlistExtension` (which checks `owner`, not `sender`) makes this mistake highly likely — a pool admin who observes that the deposit allowlist correctly gates by user identity will assume the swap allowlist behaves identically. There is no warning in the interface or NatDoc that `sender` is the direct pool caller rather than the end user. [6](#0-5) 

---

### Recommendation

`SwapAllowlistExtension.beforeSwap` should check the actual economic actor. Because the pool's swap model uses a callback where `msg.sender` of `pool.swap()` is the payer/router, the extension should either:

1. **Check `recipient`** if the intent is to gate by who receives output tokens (consistent with how the router passes the end user as `recipient`), or
2. **Require the pool to pass the originating user** via `extensionData` (a signed or router-attested identity), or
3. **Document clearly** that `sender` is the direct pool caller and that allowlisting the router opens the gate to all users — and provide a separate per-user-gated extension that reads identity from `extensionData`.

The simplest fix consistent with `DepositAllowlistExtension`'s pattern is to check `recipient` (the second argument, currently ignored with `address`):

```solidity
// Current (wrong actor):
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {

// Fixed (check recipient — the end user in router flows):
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
``` [3](#0-2)

Attack:
  userB (NOT allowlisted) calls:
    MetricOmmSimpleRouter.exactInput(pool, zeroForOne, amount, priceLimit, ...)

  Router internally calls:
    pool.swap(userB /*recipient*/, zeroForOne, amount, priceLimit, callbackData, extensionData)
    → msg.sender from pool's view = router

  Pool calls:
    _beforeSwap(router /*sender*/, userB /*recipient*/, ...)

  SwapAllowlistExtension.beforeSwap checks:
    allowedSwapper[pool][router] == true  ✓  → no revert

  Swap executes. userB receives output tokens.
  userB has bypassed the per-user allowlist.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
