### Title
Slow-Mode `WithdrawCollateral` Path Bypasses `withdrawFeeX18` Charged by Sequencer Path — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The Nado protocol has two distinct execution paths for `WithdrawCollateral` transactions: the sequencer path (`processTransactionImpl`) and the slow-mode path (`processSlowModeTransactionImpl`). The sequencer path correctly reads `withdrawFeeX18` from `SpotEngine` config and charges it to the withdrawing subaccount. The slow-mode path processes the identical withdrawal operation but never reads or charges `withdrawFeeX18` at all. Any user can exploit the slow-mode path to withdraw collateral while paying only the flat `SLOW_MODE_FEE` (in quote) and avoiding the per-product `withdrawFeeX18` entirely.

---

### Finding Description

**Sequencer path** (`processTransactionImpl`, lines 413–436):

```solidity
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,
    signedTx.tx.productId
);
clearinghouse.withdrawCollateral(
    signedTx.tx.sender,
    signedTx.tx.productId,
    signedTx.tx.amount,
    address(0),
    nSubmissions
);
``` [1](#0-0) 

The sequencer path reads `withdrawFeeX18` from `SpotEngine.getConfig(productId)` and charges it before executing the withdrawal.

**Slow-mode path** (`processSlowModeTransactionImpl`, lines 217–229):

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(
        txn.sender,
        txn.productId,
        txn.amount,
        address(0),
        nSubmissions
    );
``` [2](#0-1) 

The slow-mode path calls `clearinghouse.withdrawCollateral` directly with no `chargeFee` call and no read of `withdrawFeeX18` from `SpotEngine`. The only cost to the user is the flat `SLOW_MODE_FEE` charged at submission time in quote tokens. [3](#0-2) 

`clearinghouse.withdrawCollateral` itself does not charge `withdrawFeeX18` either — it only deducts the balance and checks health. [4](#0-3) 

The `withdrawFeeX18` for the quote product is initialized to `ONE` (1e18 in 18-decimal fixed-point) in `SpotEngine.initialize`, representing a non-trivial per-withdrawal fee. [5](#0-4) 

---

### Impact Explanation

**Impact: High**

Any user can bypass `withdrawFeeX18` on any product by routing their withdrawal through the slow-mode path. The fee is charged in the product's own token (e.g., 1 USDC per withdrawal for the quote product), while the slow-mode fee is a separate flat charge in quote. For high-value or frequent withdrawals, the attacker saves the full `withdrawFeeX18` on every withdrawal. The sequencer's fee revenue — which flows to `X_ACCOUNT` via `claimSequencerFees` — is permanently reduced. The broken invariant is: *every `WithdrawCollateral` execution must charge `withdrawFeeX18` from `SpotEngine` config*.

---

### Likelihood Explanation

**Likelihood: High**

The slow-mode path is a standard, publicly documented censorship-resistance mechanism. Any user who knows the protocol can call `submitSlowModeTransaction` with a `WithdrawCollateral` payload, wait the 3-day `SLOW_MODE_TX_DELAY`, and call `executeSlowModeTransaction`. No special privileges, leaked keys, or sequencer compromise are required. The only cost is the flat slow-mode fee, which is structurally different from and likely smaller than `withdrawFeeX18` for non-trivial withdrawals.

---

### Recommendation

In `processSlowModeTransactionImpl`, add the same `chargeFee` call that the sequencer path uses before delegating to `clearinghouse.withdrawCollateral`:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
    // Add this:
    chargeFee(
        txn.sender,
        spotEngine.getConfig(txn.productId).withdrawFeeX18,
        txn.productId
    );
    clearinghouse.withdrawCollateral(
        txn.sender,
        txn.productId,
        txn.amount,
        address(0),
        nSubmissions
    );
```

This ensures both execution paths enforce the same `withdrawFeeX18` sourced from `SpotEngine` config, eliminating the desynchronization.

---

### Proof of Concept

1. User holds a balance in product `P` with `withdrawFeeX18 = F > 0` configured in `SpotEngine`.
2. User encodes a `WithdrawCollateral{sender: subaccount, productId: P, amount: A}` payload (no signature needed for slow mode).
3. User calls `Endpoint.submitSlowModeTransaction(payload)` — pays only `SLOW_MODE_FEE` in quote.
4. After `SLOW_MODE_TX_DELAY` (3 days), user (or anyone) calls `Endpoint.executeSlowModeTransaction()`.
5. `processSlowModeTransactionImpl` decodes the payload, calls `validateSender` (passes), then calls `clearinghouse.withdrawCollateral` directly — **`F` is never charged**.
6. User receives full `A` tokens without paying `withdrawFeeX18 = F`, saving `F` per withdrawal compared to the sequencer path.

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
