### Title
Unprotected `initialize()` in Core Proxy Contracts Enables Front-Running During Deployment — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse`, `Endpoint`, `SpotEngine`, `PerpEngine`, and `OffchainExchange` are deployed as `TransparentUpgradeableProxy` contracts. Their `initialize()` functions carry no caller restriction and none of these contracts call `_disableInitializers()` in their constructors. When the proxy is deployed and `initialize()` is submitted as a separate transaction — the standard pattern evidenced by `BaseProxyManager.registerRegularProxy()` — an attacker can front-run the initialization, seize ownership, and inject malicious configuration into the core protocol.

---

### Finding Description

The KelpDAO report identifies a two-step proxy deployment pattern (upgrade implementation → call `initialize` in a separate transaction) that allows a third party to intervene and take governance control. The exact same structural flaw exists in Nado's core contracts.

`Clearinghouse.initialize()` is declared `external initializer` with no `msg.sender` check: [1](#0-0) 

The `initializer` modifier only prevents *re-initialization*; it does not restrict *who* calls it first. The first caller on the proxy wins ownership.

Critically, `Clearinghouse` (and `Endpoint`, `SpotEngine`, `PerpEngine`, `OffchainExchange`) have **no constructor calling `_disableInitializers()`**. The grep across all production Solidity files confirms `_disableInitializers()` appears only in `BaseProxyManager`, `ContractOwner`, `Verifier`, `Airdrop`, and `BaseWithdrawPool` — not in any of the five core proxy contracts: [2](#0-1) [3](#0-2) 

`Clearinghouse` is explicitly managed as a `TransparentUpgradeableProxy` by `BaseProxyManager`: [4](#0-3) 

`registerRegularProxy` registers an *already-deployed* proxy, confirming that proxy deployment and `initialize()` are separate operations: [5](#0-4) 

`_migrateRegularProxy` similarly calls only `upgradeTo` with no atomic initialization: [6](#0-5) 

`Endpoint.initialize()` carries the same flaw — no caller restriction, no `_disableInitializers()`: [7](#0-6) 

`SpotEngine.initialize()` and `PerpEngine.initialize()` delegate to `BaseEngine._initialize()` which uses `initializer` but has no caller guard: [8](#0-7) 

`OffchainExchange.initialize()` is identical in structure: [9](#0-8) 

---

### Impact Explanation

If an attacker front-runs `Clearinghouse.initialize()`:

1. `__Ownable_init()` sets the attacker as owner.
2. `clearinghouseLiq` is set to an attacker-controlled contract. `liquidateSubaccount()` uses `delegatecall` to `clearinghouseLiq`, so the attacker's code executes in `Clearinghouse`'s storage context and can drain all collateral: [10](#0-9) 

3. `withdrawPool` is set to an attacker-controlled contract. Every user withdrawal is routed through `handleWithdrawTransfer` → `withdrawPool.submitWithdrawal`, sending funds to the attacker: [11](#0-10) 

4. The legitimate deployer's `initialize()` call reverts permanently (already initialized). The protocol is unrecoverable without redeployment.

If `Endpoint.initialize()` is front-run, the attacker controls `sequencer`, `clearinghouse`, and `verifier`, giving them full control over transaction processing and signature verification.

**Corrupted state**: `clearinghouseLiq` (delegatecall target), `withdrawPool` (withdrawal recipient), `owner` (governance), `sequencer` (transaction gating), `verifier` (signature authority).

---

### Likelihood Explanation

Medium. The attack requires only mempool monitoring and a higher-gas-price transaction during the deployment window — no privileged access, no key compromise, no social engineering. It is executable by any unprivileged on-chain actor. The window exists because `BaseProxyManager.registerRegularProxy()` confirms that proxies are deployed and registered before initialization, creating an observable gap in the mempool. The attack is one-shot and irreversible once executed.

---

### Recommendation

1. Add `_disableInitializers()` to the constructors of `Clearinghouse`, `Endpoint`, `SpotEngine`, `PerpEngine`, and `OffchainExchange` to protect the implementation contracts.
2. Use `upgradeAndCall` (or pass initialization calldata to the `TransparentUpgradeableProxy` constructor) so that proxy deployment and `initialize()` execute atomically in a single transaction, eliminating the front-running window.
3. Add a deployer-address check to `initialize()` (analogous to `ContractOwner.initialize()`'s `require(_deployer == msg.sender, "expected deployed to initialize")`): [12](#0-11) 

---

### Proof of Concept

```
1. Attacker monitors mempool.
2. Sees TransparentUpgradeableProxy deployment for Clearinghouse.
3. Attacker submits (higher gas):
     Clearinghouse(proxyAddr).initialize(
         attackerEndpoint,
         quoteToken,
         maliciousClearinghouseLiq,   // delegatecall target
         0,
         maliciousWithdrawPool         // withdrawal recipient
     )
4. Attacker's tx mines first; attacker is now owner with malicious config.
5. Legitimate deployer's initialize() reverts ("Initializable: contract is already initialized").
6. All user deposits flow into the compromised Clearinghouse.
7. On any withdrawal: Clearinghouse.withdrawCollateral() → handleWithdrawTransfer()
   → maliciousWithdrawPool.submitWithdrawal() → funds sent to attacker.
8. On any liquidation: Clearinghouse.liquidateSubaccount() → delegatecall(maliciousClearinghouseLiq)
   → arbitrary code in Clearinghouse storage context → full collateral drain.
```

    function registerRegularProxy(string memory name, address proxy)
        external
        override
        onlyOwner
    {
        require(proxies[name] == address(0), "already registered.");
        address impl = _getImpl(proxy);
        contractNames.push(name);
        proxies[name] = proxy;
        pendingImpls[name] = impl;
        pendingHashes[name] = _getCodeHash(impl.code);
        codeHashes[name] = pendingHashes[name];
        if (_isEqual(name, CLEARINGHOUSE)) {
            proxyManagerHelper.registerClearinghouse(proxy);
            address clearinghouseLiq = _getClearinghouseLiqImpl();
            pendingImpls[CLEARINGHOUSE_LIQ] = clearinghouseLiq;
            pendingHashes[CLEARINGHOUSE_LIQ] = _getCodeHash(
                clearinghouseLiq.code
            );
            codeHashes[CLEARINGHOUSE_LIQ] = pendingHashes[CLEARINGHOUSE_LIQ];
        } else if (_isEqual(name, ENDPOINT)) {
            proxyManagerHelper.registerEndpoint(proxy);
            address endpointTx = _getEndpointTxImpl();
            pendingImpls[ENDPOINT_TX] = endpointTx;
            pendingHashes[ENDPOINT_TX] = _getCodeHash(endpointTx.code);
            codeHashes[ENDPOINT_TX] = pendingHashes[ENDPOINT_TX];
        }
    }
