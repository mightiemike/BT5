### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so `sender` = router. If the router is allowlisted (which is required for router-mediated swaps to function), every user — including non-allowlisted ones — can bypass the gate by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (correct). `sender` is the first argument the pool passes to `_beforeSwap`, which is the pool's own `msg.sender` — i.e., whoever called `pool.swap(...)`. [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes `sender` directly from the pool's caller:

```solidity
// metric-core/contracts/ExtensionCalling.sol
function _beforeSwap(
    address sender,   // = pool's msg.sender
    address recipient,
    ...
) internal {
    _callExtensionsInOrder(
        BEFORE_SWAP_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
    );
}
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInput*(...)`, the router calls `pool.swap(...)` — making the router the pool's `msg.sender`. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

This creates an inescapable dilemma for any pool that uses `SwapAllowlistExtension` and also wants to support router-mediated swaps:

| Router allowlisted? | Effect |
|---|---|
| **Yes** | Every user bypasses the allowlist by routing through the router |
| **No** | Allowlisted users cannot use the router at all (broken functionality) |

The analog to the external report is exact: just as `GaugeController` passes the raw `periodWeight` where a scaled value is required, `SwapAllowlistExtension` checks the router address where the actual user address is required. In both cases the guard receives the wrong value and its protection is nullified.

---

### Impact Explanation

A pool deployer who configures `SwapAllowlistExtension` to restrict swaps to a set of KYC'd, institutional, or protocol-controlled addresses cannot enforce that restriction for router-mediated swaps. Any unprivileged user can call `MetricOmmSimpleRouter` and trade on the restricted pool, bypassing the intended access control. This breaks the core invariant that the allowlist gates the economically relevant actor (the user whose funds move), and allows unauthorized parties to drain or manipulate pool liquidity.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the standard public entry point for swaps. Any pool that enables `SwapAllowlistExtension` and also needs to support the router must allowlist the router, which immediately opens the bypass to all users. The trigger requires no special privilege — any address can call the router.

---

### Recommendation

The extension must check the identity of the actual end-user, not the intermediary. Two complementary fixes:

1. **In `SwapAllowlistExtension.beforeSwap`**: check `recipient` (the address that receives tokens and is economically accountable) instead of `sender`, or require the pool to pass the originating user via `extensionData`.

2. **In `MetricOmmSimpleRouter`**: forward the caller's address explicitly in `extensionData` so the extension can decode and verify the true user. The extension should then decode and check that address rather than `sender`.

The cleanest fix is to have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension` decode and gate on that value, making the check router-aware.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured in BEFORE_SWAP_ORDER.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // only alice is allowed
3. Pool admin calls setAllowedToSwap(pool, router, true)  // router must be allowlisted
                                                           // for router-mediated swaps
4. bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInput(pool, bob, zeroForOne, amount, priceLimit, "")
5. Router calls pool.swap(bob, zeroForOne, amount, priceLimit, "")
       → pool's msg.sender = router
       → _beforeSwap(sender=router, recipient=bob, ...)
       → extension checks allowedSwapper[pool][router] == true  ✓
       → swap succeeds — bob bypasses the allowlist entirely
``` [3](#0-2) [4](#0-3) [5](#0-4)
