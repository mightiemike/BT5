Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Enabling Full Allowlist Bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which is the direct `msg.sender` of `MetricOmmPool.swap`. When users route through `MetricOmmSimpleRouter`, `sender` equals the router's address, not the original user. A pool admin who allowlists the router to enable router-mediated swaps for their curated users inadvertently opens the pool to every unprivileged caller of the same public router, completely defeating the access-control invariant the extension was deployed to enforce.

## Finding Description

**Root cause — wrong actor bound in the allowlist check**

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap`: [2](#0-1) 

**How the router breaks identity**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly: [3](#0-2) 

Because the router is the immediate caller of `pool.swap`, `msg.sender` inside the pool equals the router's address. The `sender` forwarded to the extension is therefore the router, not the original user. The allowlist check becomes `allowedSwapper[pool][router_address]`.

The same structural problem applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [4](#0-3) 

**The forced configuration trap**

A pool admin who wants allowlisted users to use the router must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, the check `allowedSwapper[pool][router]` passes for every call arriving through the router — regardless of who the original caller is. The allowlist no longer distinguishes between `alice` (allowlisted) and `charlie` (not allowlisted) when both route through the same public contract. [5](#0-4) 

## Impact Explanation
Any unprivileged user can bypass the swap allowlist on a curated pool by routing through `MetricOmmSimpleRouter`. Curated pools designed for specific users (KYC'd participants, institutional LPs, whitelisted protocols) become accessible to all callers. Unauthorized swaps drain LP inventory at oracle prices, causing direct loss of LP principal and breaking the core access-control invariant the extension was deployed to enforce. This constitutes a broken core pool functionality causing loss of funds and an admin-boundary break where an unprivileged path bypasses the pool admin's configured access control.

## Likelihood Explanation
High. `MetricOmmSimpleRouter` is the canonical, publicly documented swap entry point. A pool admin who deploys a swap-allowlisted pool and wants their allowlisted users to use the router has no choice but to allowlist the router address — this is the natural and expected operational step. Once taken, the bypass is trivially reachable by any user with no special privileges, no flash loan, and no admin interaction. The only precondition is the router being allowlisted, which is the direct consequence of the pool admin trying to make the allowlist work with the supported periphery.

## Recommendation
The extension must check the original economic actor, not the intermediary. Concrete options:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Requires a coordinated convention between router and extension.
2. **Factory-level incompatibility guard**: The factory rejects pool configurations that pair `SwapAllowlistExtension` with a known router address in the allowlist, or documents that the extension is incompatible with router-mediated flows and enforces direct-call-only usage.
3. **Dedicated router with identity forwarding**: Deploy a router variant that passes the original caller's identity in a verifiable way (e.g., signed message or trusted forwarder pattern) that the extension can validate.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls:
     swapExtension.setAllowedToSwap(pool, alice, true);
     swapExtension.setAllowedToSwap(pool, address(router), true);
   (second call is required so alice can use the router)
3. charlie (not allowlisted) calls:
     router.exactInputSingle({pool: pool, recipient: charlie, ...})
4. Router calls pool.swap(...) — msg.sender inside pool = router.
5. SwapAllowlistExtension.beforeSwap receives sender = router.
6. Check: allowedSwapper[pool][router] == true → passes.
7. Charlie's swap executes on the curated pool despite never being allowlisted.
```

The bypass requires zero privileged access from charlie. The only precondition — the router being allowlisted — is the natural consequence of the pool admin trying to make the allowlist work with the supported periphery.
