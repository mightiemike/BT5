### Title
Child Shard `ChunkExtra` Inherits Stale `proposed_split` from Parent During Dynamic Resharding — (`chain/chain/src/resharding/manager.rs`)

### Summary

During dynamic resharding, `ReshardingManager::process_memtrie_resharding_storage_update` creates child shard `ChunkExtra` records by cloning the parent's `ChunkExtra` and only updating `state_root` and `congestion_info`. The `proposed_split` field is never reset to `None`. When `min_epochs_between_resharding = 1` (the minimum permitted value), the parent shard can legitimately propose a split in epoch N+1 (the preparation epoch between the resharding decision and its execution). The child shards created at the epoch N+2 boundary then inherit a non-`None` `proposed_split` that refers to the now-defunct parent shard. The child shards' first chunk application computes `proposed_split = None` (because `is_next_block_possibly_last_in_epoch` returns false for the first block of a new epoch), causing `validate_chunk_with_chunk_extra_and_receipts_root` to fail with `InvalidChunkHeaderShardSplit` — permanently stalling chunk production for the child shards.

### Finding Description

In `process_memtrie_resharding_storage_update`, child `ChunkExtra` is constructed by cloning the parent's record and patching only two fields:

```rust
// TODO(resharding): set all fields of `ChunkExtra`. ...
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
``` [1](#0-0) 

The `proposed_split` field of `ChunkExtraV5` is silently carried over from the parent: [2](#0-1) 

The `proposed_split` field is the authoritative source used by chunk-header validation:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [3](#0-2) 

The `proposed_split` value stored in `ChunkExtra` is computed during chunk application by `compute_proposed_split`, which gates on `is_next_block_possibly_last_in_epoch` and `can_reshard`: [4](#0-3) 

With `min_epochs_between_resharding = 1`, `can_reshard` returns `true` in epoch N+1 (one epoch after the resharding decision in epoch N). If the parent shard is above the memory threshold and `is_next_block_possibly_last_in_epoch` returns `true` for the last few blocks of epoch N+1, `compute_proposed_split` returns `Some(TrieSplit{...})` and stores it in the parent's `ChunkExtra`. The resharding manager then clones this `ChunkExtra` into both child shards without clearing `proposed_split`.

The child shards' first chunk application in epoch N+2 calls `compute_proposed_split` and returns `None` (the first block of a new epoch is never near the epoch boundary). The stored `ChunkExtra` says `Some(...)`, the chunk header says `None` — validation fails permanently.

The documentation explicitly acknowledges the incomplete clone: [5](#0-4) 

And separately documents the unsafe back-to-back resharding scenario that arises from this exact stale-field propagation: [6](#0-5) 

The `is_next_block_possibly_last_in_epoch` function is documented to produce false positives, meaning the parent's `proposed_split` can be non-`None` even for blocks that are not actually the last of the epoch: [7](#0-6) 

### Impact Explanation

If triggered, the child shards' `ChunkExtra.proposed_split` permanently disagrees with every chunk header the chunk producer can generate. No valid chunk can be produced for the affected child shards. This halts those shards of the network. The divergent Borsh-serialized value is the `Option<TrieSplit>` field inside `ChunkExtraV5` — `Some(boundary_account, left_mem, right_mem)` in the stored record versus `None` in every subsequent chunk header.

### Likelihood Explanation

Requires `min_epochs_between_resharding = 1` (the minimum non-zero value the documentation permits) and a shard above the memory split threshold during epoch N+1. The cooldown is intended as the sole guard against this scenario, but the guard is insufficient at its minimum value. The root cause — the incomplete clone — is present unconditionally in production code.

### Recommendation

In `process_memtrie_resharding_storage_update`, after cloning the parent `ChunkExtra`, explicitly reset `proposed_split` to `None` for each child shard:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
*child_chunk_extra.proposed_split_mut() = None;  // child starts with no pending split
```

This removes the dependency on `min_epochs_between_resharding` being large enough to suppress a parent split proposal in epoch N+1, and closes the invariant gap regardless of config.

### Proof of Concept

1. Configure `DynamicReshardingConfig` with `min_epochs_between_resharding = 1` and a low `memory_usage_threshold` so the parent shard crosses it.
2. Trigger a resharding: epoch N last block embeds `shard_split` in `BlockHeaderInnerRestV6`; `finalize_epoch` stores the new `ShardLayoutV3` for epoch N+2.
3. In epoch N+1, the parent shard's chunk application near the epoch boundary calls `compute_proposed_split`. `can_reshard` returns `true` (1 epoch elapsed since epoch N). `is_next_block_possibly_last_in_epoch` returns `true`. `proposed_split = Some(TrieSplit{boundary, left_mem, right_mem})` is stored in the parent's `ChunkExtra`.
4. At the first block of epoch N+2, `start_resharding` → `split_shard` → `process_memtrie_resharding_storage_update` clones the parent's `ChunkExtra` (which has `proposed_split = Some(...)`) into both child shards and saves them via `chain_store_update.save_chunk_extra`.
5. The child shard's chunk producer calls `apply_transactions` → `compute_proposed_split` → returns `None` (first block of epoch, not near boundary). The produced chunk header carries `proposed_split = None`.
6. `validate_chunk_with_chunk_extra_and_receipts_root` compares `prev_chunk_extra.proposed_split() = Some(...)` against `chunk_header.proposed_split() = None` and returns `Err(InvalidChunkHeaderShardSplit(...))`.
7. Chunk production for both child shards is permanently blocked; the affected shards halt.

    /// Like `is_produced_block_last_in_epoch`, but checks if the **next** block will be last
    /// in the epoch, not the one currently produced. Due to how block finalization works, this
    /// method **can produce false positives**, but will never return a false negative.
    /// Used to check if proposed_split should be computed for a shard.
    ///
    /// Parameters:
    ///  - `block_height`: the height of the block being produced
    ///  - `parent_hash`: hash of the parent block (the block we're building on top of)
    pub fn is_next_block_possibly_last_in_epoch(
        &self,
        block_height: BlockHeight,
        parent_hash: &CryptoHash,
    ) -> Result<bool, EpochError> {
        // Avoid checking the wrong epoch if parent is the last block of an epoch.
        if self.is_next_block_epoch_start(parent_hash)? {
            return Ok(false);
        }
        let parent_info = self.get_block_info(parent_hash)?;
        let epoch_first_block = parent_info.epoch_first_block();
        // At block_height+1, last_final_height is at most block_height-1.
        // This can lead to false positive result, if previous block doesn't get finalized
        // before the next block is produced, but it will never give a false negative.
        let max_last_final_height = block_height.saturating_sub(1);
        self.is_next_block_in_next_epoch_impl(
            block_height + 1,
            max_last_final_height,
            epoch_first_block,
        )
    }
