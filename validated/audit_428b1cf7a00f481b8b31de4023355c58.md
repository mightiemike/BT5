### Title
Python Object Pointer Address Aliasing in `clvm_tree_to_lazy_node` Produces Wrong Tree Structure — (`File: wheel/src/api.rs`)

### Summary

`clvm_tree_to_lazy_node` uses raw CPython object pointer addresses (`pyobj.as_ptr() as usize`) as the sole identity key in `identity_map`. When a custom `CLVMStorage` implementation returns uncached child objects from `.pair`, those children can be freed mid-traversal (their only live reference was the `Bound<'py, PyAny>` on the work stack, which is dropped after processing). CPython immediately reuses freed small-object addresses. A subsequent node in the same traversal can land at the same address, causing `identity_map` to silently return the `NodePtr` for the **old, already-freed** node instead of the new one. The resulting `LazyNode` encodes a structurally wrong CLVM tree, and any serialization of it (`ser_2026`, `ser_backrefs`, `ser_legacy`) produces bytes that do not represent the intended program.

---

### Finding Description

`clvm_tree_to_lazy_node` (`wheel/src/api.rs`, lines 193–301) traverses an arbitrary Python `CLVMStorage` tree and builds a Rust `Allocator`-backed `LazyNode`. It maintains three maps:

```
identity_map : HashMap<usize, NodePtr>   // Python id() → NodePtr
atom_map     : HashMap<Vec<u8>, NodePtr> // atom bytes → NodePtr
pair_map     : HashMap<(NodePtr,NodePtr), NodePtr>
```

When a pair node is encountered and its children are not yet processed, the code pushes three work items onto the stack (LIFO):

```rust
stack.push(WorkItem::BuildPair { id, left_id, right_id });
if !right_done { stack.push(WorkItem::Visit(right)); }
if !left_done  { stack.push(WorkItem::Visit(left));  }
```

`left_id` and `right_id` are captured as raw `usize` pointer values. The actual `Bound<'py, PyAny>` objects (`left`, `right`) are **moved** into the `Visit` work items. Once `Visit(left)` is popped and processed, the `Bound` is consumed and dropped at the end of the match arm, decrementing the CPython reference count. If no other Python-side reference holds the child alive (i.e., the parent's `.pair` property returns freshly-allocated wrappers without caching them), the child object is freed immediately.

CPython's small-object allocator (`obmalloc`) reuses freed addresses within the same arena. A subsequent `Visit` for a different node can receive a new Python object at the exact same address. When that new object is processed, `identity_map.insert(new_id, new_node_ptr)` **overwrites** the entry for `left_id`. Later, when `BuildPair` is processed:

```rust
let l = identity_map[&left_id];   // returns NodePtr for the NEW object, not the original left child
let r = identity_map[&right_id];
allocator.new_pair(l, r)
```

The pair is built with the wrong left child `NodePtr`. The `LazyNode` returned to Python encodes a structurally incorrect CLVM tree.

---

### Impact Explanation

The corrupted `NodePtr` is baked into the `Allocator` that backs the returned `LazyNode`. Every downstream serializer (`ser_2026`, `ser_backrefs`, `ser_legacy`) reads the `Allocator` through the fixed `NodePtr` stored in `LazyNode`:

```rust
// wheel/src/api.rs:180-182
fn ser_2026(py: Python, node: &LazyNode, level: u32) -> PyResult<Py<PyBytes>> {
    let buf = serialize_2026(node.allocator(), node.node(), level).map_err(eval_to_py)?;
```

The serialized bytes will represent a different CLVM program than the one the caller intended. If those bytes are used as a puzzle reveal or solution in a Chia spend bundle, the resulting tree hash will differ from what validators compute from the original Python tree, causing **consensus divergence** between nodes that serialize via `clvm_tree_to_lazy_node` and those that use a different path. The `Program.to_bytes_2026()` method calls `clvm_tree_to_lazy_node(self)` directly, making this reachable from the standard Python API.

---

### Likelihood Explanation

The standard `Program` and `CLVMTree` implementations both cache their `.pair` result (`_pair` attribute), so the children remain alive through the parent for the duration of traversal. The bug is therefore **not triggered** by these standard types. However:

1. The function signature accepts any `CLVMStorage`-conforming Python object (duck-typed protocol, no enforcement).
2. A custom implementation that computes `.pair` lazily without caching (e.g., a streaming deserializer, a proxy object, or a test harness) will expose the bug.
3. CPython's `obmalloc` reuses freed addresses within microseconds on any platform; no special timing is required once the refcount hits zero.
4. The `clvm_tree_to_lazy_node` function is exported as a public Python API and is documented for use with arbitrary `CLVMStorage` objects.

Likelihood is **medium** for any caller using a non-caching `CLVMStorage` implementation, and **low** for callers using only `Program` or `CLVMTree`.

---

### Recommendation

Replace raw pointer addresses with a safe identity scheme that keeps the Python objects alive for the duration of the traversal:

- **Short term**: Store the actual `Bound<'py, PyAny>` (or an `Py<PyAny>` owning reference) in `BuildPair` alongside the `usize` key, ensuring the child objects remain alive until the pair is constructed. This prevents the refcount from dropping to zero mid-traversal.
- **Long term**: Audit all `CLVMStorage`-accepting APIs for the same pattern. Document that `.pair` implementations must return stable (cached) references, or enforce this at the API boundary by eagerly collecting all nodes before building the `identity_map`.

### Proof of Concept

```python
import gc
from clvm_rs.clvm_rs import clvm_tree_to_lazy_node, ser_legacy
from clvm_rs.clvm_storage import CLVMStorage

class FreshPairNode(CLVMStorage):
    """Returns uncached wrapper objects from .pair — no _pair cache."""
    def __init__(self, left_bytes, right_bytes):
        self.atom = None
        self._left_bytes = left_bytes
        self._right_bytes = right_bytes

    @property
    def pair(self):
        # Fresh allocation every call — no caching
        class Leaf(CLVMStorage):
            def __init__(self, b):
                self.atom = b
            @property
            def pair(self): return None
        return (Leaf(self._left_bytes), Leaf(self._right_bytes))

root = FreshPairNode(b"left_child_data", b"right_child_data")

# Force GC pressure so CPython reuses freed addresses
gc.collect()

lazy = clvm_tree_to_lazy_node(root)
result = ser_legacy(lazy)

# Verify: the serialized bytes may encode the wrong left or right child
# due to address aliasing in identity_map during traversal.
print(result.hex())
```

The exact wrong output depends on CPython's allocator state, but the structural corruption is deterministic once address reuse occurs. Under address-sanitizer or with `gc.collect()` calls between traversal steps, the aliasing can be made reliably reproducible.
