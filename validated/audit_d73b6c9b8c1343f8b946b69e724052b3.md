Audit Report

## Title
`uint32` Overflow in `_afterTimelock` Allows Pool Admin to Bypass Stop-Loss Timelock Immediately — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`_afterTimelock` computes `uint32(block.timestamp + oracleStopLossConfig[pool_].timelock)`. When `timelock` is set to any value exceeding `type(uint32).max − block.timestamp` (≈ 2.54 billion seconds at current timestamps), the `uint256` addition overflows the `uint32` cast and wraps to a timestamp already in the past. `_requireElapsed` then passes unconditionally, allowing the pool admin to propose and execute any stop-loss parameter change in the same block, with zero LP reaction time.

## Finding Description

The root cause is in `_afterTimelock`:

```solidity
// L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `timelock` is `uint32` (promoted to `uint256` for the addition). The sum is silently truncated to `uint32`. No upper-bound validation exists on `timelock` in either `initialize` (L56-58, only `drawdownE6` and `decayPerSecondE8` are validated) or `proposeOracleStopLossTimelock` (L78-84, no cap on `newTimelock`).

The guard that enforces the delay:

```solidity
// L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(executeAfter, block.timestamp);
}
```

When `executeAfter` wraps to a value already less than `block.timestamp`, this check passes unconditionally. All four proposal paths (`proposeOracleStopLossTimelock`, `proposeOracleStopLossDrawdown`, `proposeOracleStopLossDecay`, `proposeOracleStopLossHighWatermarks`) call `_afterTimelock` and store the wrapped value as their respective `executeAfter` fields.

**Exploit path:**
1. Pool admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)` — appears to be a 136-year protective setting.
2. Waits through the current timelock once, then calls `executeOracleStopLossTimelock`.
3. Now `oracleStopLossConfig[pool_].timelock = type(uint32).max`. Every subsequent `_afterTimelock` call computes `uint32(~1,753,000,000 + 4,294,967,295) = uint32(6,047,967,295) ≈ 1,752,999,999` — a timestamp in the past.
4. Pool admin calls `proposeOracleStopLossDrawdown(pool, 0)` then `executeOracleStopLossDrawdown(pool)` in the **same block**. `_requireElapsed` passes immediately.
5. `drawdownE6 = 0` disables the stop-loss check entirely (L217: `if (drawdown == 0) return;`), allowing value-leaking swaps that `OracleStopLossTriggered` would otherwise block.

## Impact Explanation

The timelock is the sole LP-protection mechanism against sudden stop-loss parameter changes by the pool admin. With the overflow active, the pool admin can disable the stop-loss (`drawdownE6 = 0`) in the same block as the proposal, giving LPs zero advance notice and no opportunity to withdraw before value-leaking swaps drain their positions. This is a direct admin-boundary break: the pool admin bypasses a timelock that LPs were explicitly relying on for principal protection.

## Likelihood Explanation

The pool admin must wait through one honest timelock period to install the overflowing `timelock` value. After that single wait, **all future proposals on that pool are immediately executable** with no further delay. The deceptive aspect is that `type(uint32).max` (≈ 136 years) appears to be an extremely conservative protective setting, yet it produces an `executeAfter` already in the past.

## Recommendation

Cast `block.timestamp` to `uint32` **before** adding the timelock, so the addition stays within `uint32` arithmetic and wraps predictably (matching the pattern used in similar timelock implementations):

```diff
 function _afterTimelock(address pool_) private view returns (uint32) {
-    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
+    return uint32(block.timestamp) + oracleStopLossConfig[pool_].timelock;
 }
```

Additionally, add an upper-bound validation on `timelock` in both `initialize` and `proposeOracleStopLossTimelock` (e.g., `require(timelock <= 365 days)`) to prevent any future overflow path.

## Proof of Concept

```solidity
function test_timelockOverflowBypassesDelay() public {
    OracleValueStopLossExtension freshExt = new OracleValueStopLossExtension(address(factoryStub));
    MockExtensionExtsloadPool freshPool = new MockExtensionExtsloadPool(address(factoryStub), MIN_SHARES);
    factoryStub.setPoolAdmin(address(freshPool), admin);
    vm.prank(address(factoryStub));
    freshExt.initialize(address(freshPool), abi.encode(uint32(500_000), uint32(0), uint32(7 days)));

    vm.startPrank(admin);
    // Step 1: install overflowing timelock (looks like 136-year protection)
    freshExt.proposeOracleStopLossTimelock(address(freshPool), type(uint32).max);
    vm.warp(block.timestamp + 7 days);
    freshExt.executeOracleStopLossTimelock(address(freshPool));

    // Step 2: propose drawdown = 0 (disables stop-loss)
    // _afterTimelock now returns a wrapped past timestamp
    freshExt.proposeOracleStopLossDrawdown(address(freshPool), 0);

    // Step 3: execute in the SAME block — no warp needed
    freshExt.executeOracleStopLossDrawdown(address(freshPool));

    (uint32 dd,,,) = freshExt.oracleStopLossConfig(address(freshPool));
    assertEq(dd, 0); // stop-loss silently disabled with zero LP notice
    vm.stopPrank();
}
```
