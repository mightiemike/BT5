### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Depositors to Bypass Curated-Pool Guard — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool." Its `beforeAddLiquidity` hook receives two actor addresses — `sender` (the actual `msg.sender` of the `addLiquidity` call, i.e., the depositor) and `owner` (the LP position owner, which is a freely chosen parameter). The implementation silently drops `sender` (it is unnamed) and checks only `owner`. Because `MetricOmmPool.addLiquidity` accepts an arbitrary `owner` and `MetricOmmPoolLiquidityAdder` explicitly supports depositing on behalf of a third-party owner, any non-allowlisted user can bypass the guard by nominating an allowlisted address as `owner`.

---

### Finding Description

**Wrong actor checked in the hook.**

`DepositAllowlistExtension.beforeAddLiquidity` ignores the `sender` parameter entirely and gates on `owner`: [1](#0-0) 

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The first parameter (`sender`) is unnamed and never read.

**Pool passes `msg.sender` as `sender` and an arbitrary address as `owner`.**

`MetricOmmPool.addLiquidity` accepts `owner` as a free caller-supplied parameter and forwards `msg.sender` as `sender` to the extension: [2](#0-1) 

```solidity
function addLiquidity(address owner, ...) external {
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    ...
}
```

The extension dispatch confirms both actors are forwarded: [3](#0-2) 

**Periphery router explicitly supports depositing on behalf of a third-party owner.**

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts an explicit `owner` that differs from `msg.sender` (the payer): [4](#0-3) 

The adder's own test suite confirms this is a supported, tested flow: [5](#0-4) 

**Attack path:**

1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists address `A`: `extension.setAllowedToDeposit(pool, A, true)`.
2. Non-allowlisted user `B` calls `adder.addLiquidityExactShares(pool, A, salt, deltas, maxAmount0, maxAmount1, "")`.
3. The adder calls `pool.addLiquidity(A, salt, deltas, ...)`.
4. The pool calls `extension.beforeAddLiquidity(adder_address, A, ...)`.
5. The extension evaluates `allowedDepositor[pool][A]` → `true` → passes.
6. `B`'s tokens are pulled (B is the payer stored in transient context); `A` receives the LP position.
7. `B`'s tokens have entered the curated pool without `B` ever being allowlisted.

The same bypass works when `B` calls `pool.addLiquidity(A, ...)` directly, since the pool imposes no restriction on who may nominate an `owner`.

---

### Impact Explanation

The deposit allowlist is structurally bypassed. Any non-allowlisted user who knows one allowlisted address can deposit tokens into the curated pool. The pool admin's curation policy — whether for regulatory (KYC/AML), risk, or access-control reasons — is not enforced. Additionally, non-allowlisted users can grief allowlisted LPs by depositing tokens into their positions without consent, causing unexpected LP exposure and position-size changes the LP did not authorize.

---

### Likelihood Explanation

The `MetricOmmPoolLiquidityAdder` explicitly supports and tests the "deposit on behalf of another owner" flow. Allowlisted addresses are typically discoverable on-chain (via `AllowedToDepositSet` events). No special privilege is required — any EOA or contract can call the adder with an arbitrary `owner`.

---

### Recommendation

Check `sender` (the actual depositor) instead of `owner` (the LP position owner):

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension.beforeSwap`, which checks `sender` (the actual swap initiator): [6](#0-5)

### Proof of Concept

```solidity
// Foundry test demonstrating the bypass
function test_nonAllowlistedUserBypassesDepositGuard() public {
    address allowlistedLP = makeAddr("allowlistedLP");
    address attacker      = makeAddr("attacker");

    // Admin allowlists only allowlistedLP
    vm.prank(admin);
    depositExtension.setAllowedToDeposit(address(pool), allowlistedLP, true);

    // Fund attacker and approve adder
    deal(address(token0), attacker, 1_000_000);
    deal(address(token1), attacker, 1_000_000);
    vm.startPrank(attacker);
    token0.approve(address(adder), type(uint256).max);
    token1.approve(address(adder), type(uint256).max);

    // Attacker deposits with owner = allowlistedLP — extension checks owner, passes
    LiquidityDelta memory d = ...;
    adder.addLiquidityExactShares(address(pool), allowlistedLP, 1, d, 1_000_000, 1_000_000, "");
    vm.stopPrank();

    // Attacker's tokens entered the pool; allowlistedLP holds the position
    // The deposit allowlist was bypassed — attacker was never allowlisted
}
```

The existing test `test_exactShares_canAddOnBehalfOfAnotherOwner` already demonstrates the mechanics without an allowlist; adding the extension confirms the bypass. [7](#0-6)
