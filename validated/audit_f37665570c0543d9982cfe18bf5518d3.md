### Title
`SwapAllowlistExtension` gates on the direct pool caller (`sender`) instead of the originating user, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` becomes the router address. If the pool admin allowlists the router — a necessary step for allowlisted users to use the router — every user can bypass the per-user allowlist by routing through the router. This is the direct analog of the external bug: two code paths that are supposed to enforce the same guard check different identities, so one path silently bypasses the restriction.

---

### Finding Description

**The mismatch in identity checked by the two swap paths**

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension is called by the pool), and `sender` is whatever address the pool passes as the first argument to `beforeSwap`. The pool always passes its own `msg.sender`:

```solidity
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap()
    recipient,
    ...
``` [2](#0-1) 

**Path 1 — direct swap:** user calls `pool.swap()` → `sender = user` → `allowedSwapper[pool][user]` is checked. ✓

**Path 2 — router swap:** user calls `router.exactInputSingle()` → router calls `pool.swap()` → `sender = router` → `allowedSwapper[pool][router]` is checked. ✗ [3](#0-2) 

The pool admin faces an inescapable catch-22:

| Router allowlisted? | Allowlisted users can use router? | Non-allowlisted users can bypass? |
|---|---|---|
| No | ✗ | ✗ |
| Yes | ✓ | **✓ (bypass)** |

**Contrast with `DepositAllowlistExtension`**, which correctly checks `owner` (the economic beneficiary of the deposit), not `sender` (the direct caller):

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
``` [4](#0-3) 

The deposit allowlist correctly identifies the economic actor (`owner`). The swap allowlist uses `sender`, which collapses to the router address on the router path — the exact same two-mapping mismatch pattern as the external bug, where `approve`/`allowance` tracked one mapping while `increaseAllowance`/`decreaseAllowance` tracked another.

---

### Impact Explanation

When the pool admin allowlists the router (the only way to let allowlisted users use the router), the `SwapAllowlistExtension` guard is completely defeated for all router-mediated swaps. Any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle()` or `exactInput()` against the restricted pool and the extension will pass because `allowedSwapper[pool][router] == true`. LPs in a restricted pool — e.g., one designed for KYC-gated counterparties, or one where the admin is trying to prevent specific actors from extracting value — are exposed to unrestricted swaps from any address, directly undermining the LP protection the extension was deployed to provide.

---

### Likelihood Explanation

Any pool that (a) deploys `SwapAllowlistExtension` to restrict swappers and (b) wants those allowlisted users to be able to use the standard router must allowlist the router. This is the expected operational configuration. Once the router is allowlisted, the bypass is available to every address on-chain with no further preconditions.

---

### Recommendation

The `beforeSwap` interface passes `sender` (direct pool caller) and `recipient` (output recipient) but not the originating EOA. Two remediation paths:

1. **Check `recipient` instead of `sender`** — for swaps, the economic beneficiary is the `recipient` (analogous to `owner` in deposits). This mirrors the correct pattern in `DepositAllowlistExtension` and closes the router bypass, since the recipient is always the user-supplied address regardless of routing.

2. **Pass originating user via `extensionData`** — require the router to embed the originating user's address in `extensionData` and have the extension decode and verify it. This is more flexible but requires router cooperation.

Option 1 is the minimal, consistent fix that aligns the swap allowlist with the deposit allowlist's correct identity-checking pattern.

---

### Proof of Concept

```
1. Admin deploys pool with SwapAllowlistExtension.
2. Admin: extension.setAllowedToSwap(pool, userA, true)
          extension.setAllowedToSwap(pool, router, true)   ← required for userA to use router
3. Non-allowlisted userB calls:
       router.exactInputSingle({pool: pool, recipient: userB, ...})
4. Router calls pool.swap(userB, ...) with msg.sender = router.
5. Pool calls _beforeSwap(router, userB, ...).
6. Extension evaluates: allowedSwapper[pool][router] == true → passes.
7. Swap executes. userB bypassed the per-user allowlist.
``` [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)
