Audit Report

## Title
`SwapAllowlistExtension` Allowlist Fully Bypassed via `MetricOmmSimpleRouter` â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap()` sets to its own `msg.sender` â€” the immediate caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. Any pool admin who allowlists the router to enable router-mediated swaps for legitimate users simultaneously grants every unprivileged address the ability to bypass the allowlist entirely, as the extension never sees the true originator.

## Finding Description

**Root cause â€” `sender` is the immediate caller, not the economic actor:**

`MetricOmmPool.swap()` passes `msg.sender` (the direct caller of `pool.swap()`) as the `sender` argument to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value unchanged into the encoded extension call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Router makes itself the `sender`:**

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, so the pool's `msg.sender` is the router address, not the end user: [4](#0-3) 

The same applies to all other router entry points â€” `exactInput` (all hops), `exactOutputSingle`, and `exactOutput` (including the recursive callback path in `_exactOutputIterateCallback`): [5](#0-4) [6](#0-5) 

**Forced dilemma for pool admins:**

| Admin choice | Effect on allowlisted users | Effect on non-allowlisted users |
|---|---|---|
| Do **not** allowlist the router | Router-mediated swaps revert for everyone | Correctly blocked |
| **Allowlist the router** | Router-mediated swaps work | **Bypass: any user calls the router and the check passes** |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same. The `allowedSwapper[pool][attacker]` mapping is never consulted when the router is the intermediary.

## Impact Explanation

A pool protected by `SwapAllowlistExtension` is intended to restrict swap counterparties to vetted addresses (e.g., KYC'd, institutional, or protocol-controlled accounts). LP providers deposit into such a pool with the expectation that only those addresses trade against their liquidity. Once the router is allowlisted â€” the only way to make the router usable for legitimate participants â€” any address can call `MetricOmmSimpleRouter.exactInputSingle()` and execute a swap. The allowlist check passes because it sees the router's address, not the caller's. LP funds are exposed to the full universe of counterparties, defeating the purpose of the restricted pool and exposing LPs to adverse selection, front-running, or other attacks the allowlist was designed to prevent. This constitutes a direct loss of the protection over LP principal and broken core pool functionality (the allowlist extension is rendered ineffective).

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard, documented periphery entry point for swaps. Any pool admin who wants allowlisted users to be able to use the router must allowlist the router address â€” this is the expected operational configuration, not an edge case. Once that configuration is in place, the bypass is reachable by any unprivileged address with zero preconditions: no special role, no tokens beyond the swap input, no prior interaction with the pool.

## Recommendation

The extension must gate the **economic actor**, not the immediate caller of `pool.swap()`. Two complementary fixes:

1. **In `MetricOmmSimpleRouter`**: encode `msg.sender` (the end user) into `extensionData` on every `pool.swap()` call, so the extension can decode and verify the true originator.

2. **In `SwapAllowlistExtension.beforeSwap()`**: when `sender` is a known/trusted router address, decode the originating user from `extensionData` and check that address against `allowedSwapper`. Fall back to checking `sender` directly for non-router callers.

The cleanest fix is option 1 combined with option 2: the router encodes `msg.sender` into `extensionData`, and `SwapAllowlistExtension` decodes and checks it when `sender` is a recognized router, ensuring the true economic actor is always gated.

## Proof of Concept

```solidity
// Setup
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
address pool = factory.deployPool(..., ext, ...);

// Admin allowlists the router so legitimate users can use it
ext.setAllowedToSwap(pool, address(router), true);
// Admin does NOT allowlist attacker
// allowedSwapper[pool][attacker] == false

// Attacker (not on allowlist) calls the router
vm.prank(attacker);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool,
        recipient: attacker,
        tokenIn: token0,
        zeroForOne: true,
        amountIn: 1e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    })
);
// Swap succeeds: beforeSwap checked allowedSwapper[pool][router] == true
// Attacker swapped on a pool they were explicitly excluded from
```

The `beforeSwap` hook receives `sender = address(router)`, finds it allowlisted at `allowedSwapper[pool][router] == true`, and returns the success selector. The attacker's identity (`allowedSwapper[pool][attacker] == false`) is never consulted. [7](#0-6) [8](#0-7)
