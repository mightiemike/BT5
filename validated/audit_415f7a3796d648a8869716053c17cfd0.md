### Title
Fast Withdrawal Fee Rounds to Zero for Low-Decimal Tokens via Fragmented Withdrawals - (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.fastWithdrawalFeeAmount()` converts a fee computed in X18 fixed-point back to native token decimals by integer-dividing by `multiplier = 10^(18 - decimals)`. For tokens with low decimal precision (e.g., GUSD at 2 decimals, `multiplier = 10^16`), any `feeX18` value below `10^16` truncates to zero. A user can split a large withdrawal into many small ones, each below the rounding threshold, and pay zero fast-withdrawal fee on every chunk.

---

### Finding Description

In `fastWithdrawalFeeAmount`:

```solidity
int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
int128 amountX18 = int128(amount) * int128(multiplier);

int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;

int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
return feeX18 / int128(multiplier);          // ← truncates to 0
``` [1](#0-0) 

For GUSD (2 decimals), `multiplier = 10^16`. The returned fee is zero whenever `feeX18 < 10^16`. Both the proportional fee and the minimum fee (`5 * withdrawFeeX18`) can satisfy this condition for small `amount` values.

In `submitFastWithdrawal`, when `sendTo == msg.sender` (user acting as their own LP), the fee is deducted from `transferAmount`:

```solidity
if (sendTo == msg.sender) {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);   // fee = 0 → no deduction
}
fees[productId] += fee;               // fee = 0 → no protocol revenue
``` [2](#0-1) 

When `sendTo != msg.sender` (third-party LP), the LP pays the fee to the protocol:

```solidity
} else {
    safeTransferFrom(token, msg.sender, uint128(fee));  // fee = 0 → LP pays nothing
}
``` [3](#0-2) 

In both cases, a zero fee means the protocol collects nothing.

---

### Impact Explanation

**Impact: Low**

The protocol loses fast-withdrawal fee revenue. For a GUSD market with `FAST_WITHDRAWAL_FEE_RATE = 0.1%`, any single withdrawal below ~10 GUSD produces a zero proportional fee. If `withdrawFeeX18` is also configured below `2 × 10^15` (i.e., the minimum fee also truncates), every small withdrawal is completely fee-free. An attacker who splits a 1,000 GUSD withdrawal into 200 chunks of 5 GUSD each pays zero fee instead of the expected ~1 GUSD total fee. The protocol's `fees[productId]` accumulator is never incremented, permanently losing that revenue.

---

### Likelihood Explanation

**Likelihood: Medium**

- `submitFastWithdrawal` is `public` with no minimum-amount guard. [4](#0-3) 
- The only on-chain constraint is that each `idx` must be unique and increasing, which does not prevent many small withdrawals. [5](#0-4) 
- Low-decimal tokens (USDC at 6, GUSD at 2) are common collateral assets in DEX protocols.
- A user acting as their own LP (self-service fast withdrawal, `sendTo == msg.sender`) has a direct financial incentive to split withdrawals.

---

### Recommendation

Add a zero-fee guard after computing the fee, analogous to the external report's recommendation:

```solidity
int128 feeNative = feeX18 / int128(multiplier);
require(feeNative > 0, "Withdrawal amount too small for fee calculation");
return feeNative;
```

Alternatively, enforce a protocol-level minimum withdrawal amount in `submitFastWithdrawal` that guarantees `fastWithdrawalFeeAmount` always returns a non-zero value for supported tokens.

---

### Proof of Concept

**Setup**: GUSD (2 decimals) is a supported collateral. `withdrawFeeX18 = 10^14`. `FAST_WITHDRAWAL_FEE_RATE = 0.001e18` (0.1%).

**Calculation for a single chunk of 5 GUSD (`amount = 500`, i.e., 5.00 GUSD in 2-decimal units)**:

```
multiplier = 10^(18-2) = 10^16
amountX18  = 500 * 10^16 = 5 * 10^18

proportionalFeeX18 = 0.001 * 5e18 = 5e15
minFeeX18          = 5 * 1e14     = 5e14

feeX18 = max(5e15, 5e14) = 5e15

fee = 5e15 / 10^16 = 0   ← truncates to zero
```

**Attack**: User signs 200 withdrawal transactions each for 5 GUSD. Each call to `submitFastWithdrawal` returns `fee = 0`. The user withdraws 1,000 GUSD total and pays zero fast-withdrawal fee. The expected fee at 0.1% would have been ~1 GUSD. `fees[productId]` remains unchanged throughout.
