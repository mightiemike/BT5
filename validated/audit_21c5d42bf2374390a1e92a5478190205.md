### Title
`addLiquidity` is not protected by `whenNotPaused` guard, allowing deposits into a paused pool — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

`MetricOmmPool.swap()` is gated by the `whenNotPaused` modifier which checks `pauseLevel != 0`, but `addLiquidity()` and `removeLiquidity()` carry no such guard. When the pool is paused (admin or protocol pause), swaps are blocked but liquidity operations proceed freely. This is directly analogous to M-09: one major pool action is unprotected while its siblings are guarded.

### Finding Description

`MetricOmmPool` stores a `pauseLevel` (0 = active, 1 = admin-paused, 2 = protocol-paused) and enforces it via the `whenNotPaused` modifier:

```solidity
// metric-core/contracts/MetricOmmPool.sol:174-177
modifier whenNotPaused() {
    _checkNotPaused();
    _;
}

// line 643-645
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

`swap()` carries this modifier:
```solidity
// line 224
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ...
```

But `addLiquidity()` and `removeLiquidity()` do not:
```solidity
// line 188
function addLiquidity(...) external nonReentrant(PoolActions.ADD_LIQUIDITY) ...

// line 199-202
function removeLiquidity(...) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) ...
```

The pool is paused precisely when the admin or protocol detects an unsafe condition (oracle anomaly, price manipulation, discovered bug). During that window, the pool's bin balances and price state may be in an inconsistent or exploitable configuration. Allowing `addLiquidity` during this window lets any caller deposit tokens into a pool whose state the pause was meant to freeze.

### Impact Explanation

When a pool is paused due to a detected oracle manipulation or price anomaly:

1. Swaps are blocked — the attacker cannot immediately exploit the bad price.
2. `addLiquidity` is **not** blocked — the attacker deposits tokens into the pool, acquiring shares at the current (potentially skewed) bin balances.
3. The pool is unpaused after the issue is addressed.
4. The attacker's shares, minted during the skewed state, may represent a disproportionate claim on the pool's assets relative to what they deposited, allowing them to `removeLiquidity` and extract more value than they put in — at the expense of other LPs whose shares are diluted.

Additionally, any configured extension guard (e.g., `DepositAllowlistExtension`, `OracleValueStopLossExtension`) that fires on `beforeAddLiquidity` / `afterAddLiquidity` hooks still executes during a pause, but the core invariant — that the pool is frozen — is violated because the pool's bin totals and share accounting are mutated by `addLiquidity` even while `pauseLevel != 0`.

This constitutes broken core pool functionality and potential LP principal loss above Sherlock medium thresholds.

### Likelihood Explanation

- Any caller (or any allowlisted depositor if `DepositAllowlistExtension` is configured) can call `addLiquidity` at any time, including during a pause.
- Pool pauses are triggered by the admin or protocol in response to detected anomalies — exactly the moments when the pool state is most sensitive.
- No special privilege or setup is required beyond having tokens to deposit.

### Recommendation

Add `whenNotPaused` to `addLiquidity`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
```

For `removeLiquidity`, consider whether LP exits during a pause are intentionally permitted (a common DeFi pattern). If so, document it explicitly. If not, apply `whenNotPaused` there as well.

### Proof of Concept

1. Pool admin detects an oracle anomaly and calls `factory.setPause(pool, 1)` → `pauseLevel = 1`.
2. `swap()` now reverts with `PoolPaused()` for all callers.
3. Attacker calls `addLiquidity(owner=attacker, ...)` — **succeeds** because no `whenNotPaused` check exists.
4. `LiquidityLib.addLiquidity` executes, minting shares and updating `binTotals.scaledToken0/1` at the current (anomalous) bin state.
5. Admin resolves the anomaly and calls `factory.setPause(pool, 0)` → `pauseLevel = 0`.
6. Attacker calls `removeLiquidity` and redeems shares that were minted against skewed bin balances, extracting more value than deposited.
