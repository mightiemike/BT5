### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` validates the `owner` parameter (the address that will hold the resulting liquidity position) rather than the `sender` parameter (the actual caller who provides tokens via callback). Because `MetricOmmPool.addLiquidity` places no constraint that `msg.sender == owner`, any non-allowlisted address can bypass the deposit guard entirely by specifying an allowlisted address as `owner`.

---

### Finding Description

In `MetricOmmPool.addLiquidity`, the pool calls the extension hook with `msg.sender` as `sender` and the freely-chosen `owner` as the position recipient:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

There is no `msg.sender == owner` requirement in `addLiquidity` (contrast with `removeLiquidity` at line 206: `if (msg.sender != owner) revert NotPositionOwner()`).

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (first parameter, unnamed) and instead validates `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The extension's own documentation (`isAllowedToDeposit`, `setAllowedToDeposit`, error `NotAllowedToDeposit`) and the wiki description ("Gates `addLiquidity` by **depositor** address") all indicate the intent is to restrict who **provides tokens** — i.e., the caller (`sender`). The implementation checks the wrong address.

**Attack path:**

1. Pool admin deploys a pool with `DepositAllowlistExtension` to enforce that only allowlisted addresses can deposit (e.g., KYC/AML compliance).
2. `allowedDepositor[pool][alice] = true`; attacker `bob` is not on the allowlist.
3. `bob` calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. The extension receives `sender = bob`, `owner = alice`. It checks `allowedDepositor[pool][alice]` → `true` → passes.
5. `bob` satisfies the token callback, depositing tokens into the pool. `alice` receives the position.
6. The allowlist has been completely bypassed: `bob`'s tokens entered the pool despite `bob` not being allowlisted.

---

### Impact Explanation

The `DepositAllowlistExtension` is rendered entirely ineffective as an access control gate. Any non-allowlisted address can deposit tokens into the pool by nominating any allowlisted address as `owner`. The pool admin's configured guard — the primary mechanism for restricting liquidity provision — is bypassed by any unprivileged caller. In regulated or permissioned pool deployments (the stated use case), this breaks the core invariant that only approved depositors contribute liquidity, and non-compliant funds enter the pool's accounting.

---

### Likelihood Explanation

The bypass requires no special privileges, no flash loan, and no oracle manipulation. Any address can call `addLiquidity` with an arbitrary `owner`. The only prerequisite is knowing one allowlisted address (trivially discoverable from on-chain events emitted by `setAllowedToDeposit`). Likelihood is **high**.

---

### Recommendation

Change `beforeAddLiquidity` to validate `sender` (the actual token-providing caller) instead of `owner`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

This mirrors the correct pattern in `SwapAllowlistExtension.beforeSwap`, which already validates `sender` (the swap caller) rather than `recipient`.

### Proof of Concept

```solidity
// Assume pool has DepositAllowlistExtension configured.
// alice is allowlisted; bob is not.
// allowedDepositor[pool][alice] = true
// allowedDepositor[pool][bob]   = false (default)

// Bob calls addLiquidity specifying alice as owner.
// Extension checks allowedDepositor[pool][alice] → true → no revert.
// Bob's metricOmmSwapCallback (or equivalent liquidity callback) transfers tokens.
// Alice receives the position; bob's tokens are in the pool.
// The allowlist did not block bob.
pool.addLiquidity(
    alice,          // owner — checked by extension (passes)
    salt,
    deltas,
    callbackData,   // bob's callback pays the tokens
    extensionData
);
