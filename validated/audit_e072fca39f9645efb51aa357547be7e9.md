### Title
Crafted IBC Packet ClassId Bypasses Direction Check, Enabling Unauthorized NFT Unescrow and Double-Spend — (`x/nft-transfer/keeper/packet.go`)

---

### Summary

A malicious counterparty chain can craft an IBC NFT packet whose `ClassId` begins with the source port/channel prefix, causing `IsAwayFromOrigin` to return `false` (back-to-origin) even though no legitimate return transfer occurred. This triggers `RemoveClassPrefix` → `TransferOwner` from the escrow address, unescrowin an NFT that was legitimately escrowed when the victim chain sent it to the attacker. The attacker retains the voucher on their chain, achieving a double-spend.

---

### Finding Description

`processReceivedPacket` determines transfer direction by calling `IsAwayFromOrigin` with the packet's source port/channel against the attacker-controlled `data.ClassId`: [1](#0-0) 

`IsAwayFromOrigin` is a pure string prefix check: [2](#0-1) 

`ValidateBasic` on `NonFungibleTokenPacketData` validates only that `ClassId` is non-blank, token/URI counts match, and sender/receiver are valid addresses. It performs **no check** that `ClassId` does not start with the source port/channel prefix: [3](#0-2) 

When `IsAwayFromOrigin` returns `false`, the code strips the source prefix and calls `TransferOwner` from the escrow address to the attacker-controlled receiver: [4](#0-3) 

`TransferOwner` only checks that `srcOwner` is the current owner of the NFT — it has no IBC-level invariant check: [5](#0-4)

### Impact Explanation

**Concrete double-spend scenario (multi-hop):**

1. Chain A → Chain B (victim): sends native NFT `baseClass` via channel-1. B receives and mints voucher `ibc/{hash("nft/channel-B1/baseClass")}`.
2. Chain B → Chain C (attacker): sends `ibc/{hash("nft/channel-B1/baseClass")}` via channel-BC. B escrows it at `GetEscrowAddress("nft", "channel-BC")`; packet `ClassId = "nft/channel-B1/baseClass"`.
3. Chain C receives and mints its own voucher `ibc/{hash("nft/channel-CB/nft/channel-B1/baseClass")}`.
4. **Attack:** Chain C crafts a fake return packet to Chain B with `ClassId = "nft/channel-CB/nft/channel-B1/baseClass"`, source port `nft`, source channel `channel-CB`.
5. Chain B evaluates: `IsAwayFromOrigin("nft", "channel-CB", "nft/channel-CB/nft/channel-B1/baseClass")` → `HasPrefix` is `true` → returns `false` (back-to-origin). `RemoveClassPrefix` yields `"nft/channel-B1/baseClass"`. `voucherClassID = ibc/{hash("nft/channel-B1/baseClass")}`. `escrowAddress = GetEscrowAddress("nft", "channel-BC")`. `TransferOwner` succeeds because the NFT **is** escrowed there from step 2.
6. Chain B unescrows the NFT to the attacker's receiver. Chain C still holds its voucher. **Double-spend achieved.** [6](#0-5) 

---

### Likelihood Explanation

Any chain that has an established IBC channel with the victim chain and has received at least one legitimate NFT transfer (creating an escrowed NFT on the victim chain) can execute this attack. The attacker controls the counterparty chain's state, so IBC light client verification passes. No governance, privileged keys, or social engineering are required — only control of a counterparty chain node, which is the standard IBC threat model for cross-chain trust.

---

### Recommendation

In `OnRecvPacket` / `processReceivedPacket`, validate that the incoming `data.ClassId` does **not** start with the source port/channel prefix before entering the back-to-origin branch. Specifically, reject packets where `strings.HasPrefix(data.ClassId, GetClassPrefix(packet.GetSourcePort(), packet.GetSourceChannel()))` is `true` unless the packet was legitimately committed on the counterparty chain as a return transfer (i.e., the counterparty burned the voucher). At minimum, add a `ValidateBasic`-level check that rejects a `ClassId` that begins with the source port/channel prefix, mirroring the ICS-721 spec requirement that the sending chain strips its own prefix before sending.   packet.SourceChannel = "channel-CB"

  data.Receiver        = attacker_address

  IsAwayFromOrigin("nft","channel-CB","nft/channel-CB/nft/channel-B1/baseClass")

  TransferOwner(ctx, ibc/{hash("nft/channel-B1/baseClass")}, tokenID,

  → SUCCESS: NFT transferred to attacker

func IsAwayFromOrigin(sourcePort, sourceChannel, fullClassPath string) bool {
	prefixClassID := GetClassPrefix(sourcePort, sourceChannel)
	return !strings.HasPrefix(fullClassPath, prefixClassID)
}

func (nftpd NonFungibleTokenPacketData) ValidateBasic() error {
	if strings.TrimSpace(nftpd.ClassId) == "" {
		return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
	}

	if len(nftpd.TokenIds) == 0 {
		return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
	}

	if len(nftpd.TokenIds) != len(nftpd.TokenUris) {
		return newsdkerrors.Wrap(ErrInvalidPacket, "tokenIds and tokenUris lengths do not match")
	}

	if strings.TrimSpace(nftpd.Sender) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "sender address cannot be blank")
	}

	// decode the sender address
	if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
	}

	if strings.TrimSpace(nftpd.Receiver) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "receiver address cannot be blank")
	}

	// decode the receiver address
	if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
	}
	return nil
}

func (k Keeper) TransferOwner(
	ctx sdk.Context, denomID, tokenID string, srcOwner, dstOwner sdk.AccAddress,
) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, srcOwner)
	if err != nil {
		return err
	}

	nft.Owner = dstOwner.String()

	k.setNFT(ctx, denomID, nft)
	k.swapOwner(ctx, denomID, tokenID, srcOwner, dstOwner)
	return nil
}
