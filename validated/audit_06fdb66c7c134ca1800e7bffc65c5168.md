Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the actual swapper, allowing any router user to bypass per-user swap restrictions — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the router) as `sender` to `_beforeSwap`, so `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actualUser]`. A pool admin who allowlists the router to enable router-mediated swaps for specific users inadvertently grants swap access to every router caller. By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates the `owner` parameter, which `MetricOmmPoolLiquidityAdder` explicitly passes as the position owner.

## Finding Description
`MetricOmmPool.swap` at line 230–240 calls `_beforeSwap(msg.sender, ...)`. When invoked via `MetricOmmSimpleRouter.exactInputSingle` (line 72–80), `msg.sender` inside the pool is the router contract. `ExtensionCalling._beforeSwap` (line 149–177) forwards this value as `sender` to the extension. `SwapAllowlistExtension.beforeSwap` (line 37) then evaluates `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][router]`. If the router is allowlisted, the check passes for any caller of the router regardless of individual allowlist status. The deposit path avoids this because `MetricOmmPoolLiquidityAdder.addLiquidity` (line 195) explicitly passes `positionOwner` as the `owner` argument, and `DepositAllowlistExtension.beforeAddLiquidity` (line 38) checks `allowedDepositor[msg.sender][owner]` — the economically relevant party. No analogous mechanism exists for swaps; the router does not encode the original `msg.sender` into `extensionData` or any other parameter visible to the extension.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict swaps to specific addresses (e.g., KYC-verified users) is silently opened to all router users once the router is allowlisted. Any non-allowlisted address can execute swaps in a pool intended to be access-controlled, defeating the admin-configured restriction. This constitutes a broken core pool access-control mechanism reachable by an unprivileged caller.

## Likelihood Explanation
Medium. The pool admin must allowlist the router — a natural and expected operational step to allow allowlisted users to trade via the router. No privileged attacker role is required; any public address can exploit the bypass once the router is allowlisted. The inconsistency between deposit and swap allowlist identity checks makes this easy to overlook during configuration.

## Recommendation
1. **Preferred**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension` decode and verify it, analogous to how `positionOwner` is passed explicitly in `addLiquidity`.
2. **Alternative**: Add an explicit `swapper` parameter to the pool's `swap` function (mirroring `owner` in `addLiquidity`) so extensions can gate the economically relevant party rather than the direct caller.
3. **Minimum**: Document clearly that allowlisting the router grants swap access to all router users, not just individually allowlisted addresses.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` attached to the `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is individually allowlisted.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — router is allowlisted so Alice can use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, ...)` — `msg.sender` inside the pool is the router.
6. Pool calls `_beforeSwap(router, ...)` — `sender` = router address (MetricOmmPool.sol L230–231).
7. Extension evaluates `allowedSwapper[pool][router]` = `true` → no revert (SwapAllowlistExtension.sol L37).
8. Bob's swap executes successfully in a pool he was never individually allowlisted for.
