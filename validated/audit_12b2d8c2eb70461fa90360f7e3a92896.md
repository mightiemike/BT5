### Title
Fee Rate Returns Zero Instead of Default for New Products When Non-Default Tier Mask Is Set — (`core/contracts/OffchainExchange.sol`)

### Summary
`getTierFeeRateX18` uses a per-tier bitmask (`nonDefaultFeeTierMask`) to decide whether to return a stored fee rate or the protocol default (2 bps taker fee). When `updateTierFeeRates` is called with `productId == QUOTE_PRODUCT_ID`, it marks a tier as non-default and populates `feeRates[tier][productId]` for all currently registered products. Products added after this call have no stored fee rate (`feeRates[tier][newProductId] == 0`), yet the tier is already flagged as non-default. As a result, `getTierFeeRateX18` returns `{makerRateX18: 0, takerRateX18: 0}` for new products instead of the 2 bps default, causing users with non-default fee tiers to trade fee-free on those products.

### Finding Description

`getTierFeeRateX18` implements a two-path return based on `nonDefaultFeeTierMask`: [1](#0-0) 

When the tier bit is set, the function returns the raw storage value `feeRates[tier][productId]` directly. The default 2 bps path is only taken when the bit is **not** set.

`nonDefaultFeeTierMask` is a per-tier (not per-product) flag. It is set unconditionally in `updateTierFeeRates` whenever any fee rate is written for a tier: [2](#0-1) 

When `txn.productId == QUOTE_PRODUCT_ID`, the function iterates over all currently registered spot and perp products and writes `feeRates[txn.tier][productId]` for each one: [3](#0-2) 

After this call, the tier bit is permanently set in `nonDefaultFeeTierMask`. If a new product is subsequently added via `SpotEngine.addOrUpdateProduct` or `PerpEngine.addOrUpdateProduct`, `feeRates[tier][newProductId]` is never populated. Because the tier bit is already set, `getTierFeeRateX18` takes the first branch and returns the zero-initialized storage value `{makerRateX18: 0, takerRateX18: 0}` instead of the 2 bps default.

This value is consumed by `getUserFeeRateWithBuilder` → `applyFee` on every `matchOrders` call: [4](#0-3) [5](#0-4) 

With `feeInfo.feeRate == 0`, `keepRateX18 = ONE`, `newMeteredQuote = meteredQuote`, and `orderInfo.fee = 0`. The user pays nothing.

### Impact Explanation
Any user whose address has been assigned a non-default fee tier (via `updateFeeTier`) will pay zero taker fees when trading on a product added after the tier's fee rates were last bulk-updated. The protocol's `collectedFees` for that market accumulates nothing, and the fee revenue that would normally flow to the protocol via `dumpFees` is permanently lost. The exact corrupted value is `marketInfo[newProductId].collectedFees`, which remains 0 instead of accumulating 2 bps per trade. [6](#0-5) 

### Likelihood Explanation
This is a realistic operational sequence: the protocol sets custom fee tiers for market makers or VIP users, bulk-updates fee rates across all products, and then lists new trading pairs. Any user with a non-default tier who trades on the new product before a per-product `updateTierFeeRates` is issued exploits the gap. No special permissions or exploits are required from the user side — they simply submit orders through the normal `matchOrders` path. [7](#0-6) 

### Recommendation
Change `nonDefaultFeeTierMask` from a per-tier flag to a per-(tier, product) mapping, or add a fallback in `getTierFeeRateX18` that returns the default when the stored `takerRateX18` is zero:

```solidity
function getTierFeeRateX18(uint32 tier, uint32 productId)
    public view returns (FeeRates memory)
{
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        FeeRates memory stored = feeRates[tier][productId];
        if (stored.takerRateX18 != 0 || stored.makerRateX18 != 0) {
            return stored;
        }
    }
    return FeeRates({
        makerRateX18: 0,
        takerRateX18: 200_000_000_000_000
    });
}
```

Alternatively, `updateTierFeeRates` should be called again for each new product at the time of listing.

### Proof of Concept
1. Sequencer calls `updateTierFeeRates({tier: 1, productId: QUOTE_PRODUCT_ID, makerRateX18: 0, takerRateX18: 100_000_000_000_000})` — sets 1 bps taker fee for tier 1 on all existing products; sets `nonDefaultFeeTierMask |= (1 << 1)`.
2. Owner calls `SpotEngine.addOrUpdateProduct(productId=5, ...)` — new product registered; `feeRates[1][5]` is never written, remains `{0, 0}`.
3. User with `feeTiers[user] == 1` submits a taker order on product 5 via `matchOrders`.
4. `getUserFeeRateWithBuilder` calls `getTierFeeRateX18(1, 5)` → `nonDefaultFeeTierMask & 2 != 0` → returns `feeRates[1][5]` = `{makerRateX18: 0, takerRateX18: 0}`.
5. `applyFee` computes `keepRateX18 = ONE - 0 = ONE`; `orderInfo.fee = 0`. User pays zero taker fees.
6. `collectedFees` for product 5 remains 0; protocol receives no fee revenue from this trade. [1](#0-0) [8](#0-7)
