Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address as `sender` instead of the real user, enabling allowlist bypass or breaking all router-mediated swaps — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap` passes its own `msg.sender` as `sender` to `_beforeSwap`, and `SwapAllowlistExtension.beforeSwap` gates on that value. When `MetricOmmSimpleRouter` calls `pool.swap(...)`, the pool's `msg.sender` is the router contract, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This produces two mutually exclusive fund-impacting failure modes: allowlist fully bypassed (if router is allowlisted) or all router-mediated swaps broken for legitimate users (if router is not allowlisted).

## Finding Description

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` value unchanged into the extension call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the pool's `msg.sender`. The real user is stored only in transient callback context for payment, never forwarded to the extension: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. The extension therefore evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. No existing guard corrects this: the pool interface carries no separate "originating EOA" field, and the extension has no mechanism to decode the real user from `extensionData`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the explicit position-owner argument), not `sender`, because `addLiquidity` carries a distinct `owner` parameter that periphery contracts can set to the real user: [5](#0-4) 

The swap path has no equivalent explicit-user argument, so the mismatch is structural.

## Impact Explanation

Two mutually exclusive outcomes, both fund-impacting:

**Mode A — Allowlist bypass (High):** Pool admin allowlists the router (the only way to let any user swap through the standard periphery). Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle`; the extension sees `sender = router`, which is allowlisted, and permits the swap. The curated pool's per-user access control is completely defeated — unauthorized users trade on pools intended for KYC'd or restricted participants.

**Mode B — Broken core swap path (High):** Pool admin does not allowlist the router (correctly intending to gate individual users). Every allowlisted user who calls through the router receives `NotAllowedToSwap`. The standard periphery swap path is entirely unusable for the pool, breaking core swap functionality for all legitimate users and matching the "broken core pool functionality" allowed impact.

## Likelihood Explanation

Likelihood is high. `MetricOmmSimpleRouter` is the standard swap entrypoint for EOAs. Any pool deploying `SwapAllowlistExtension` immediately encounters one of the two failure modes on the first router-mediated swap. No special timing, privileged access, or exotic token behavior is required — a single `exactInputSingle` call reproduces the issue.

## Recommendation

The pool must carry the originating user's identity through to the extension. Two complementary fixes:

1. **Router-side:** Encode the real `msg.sender` in `extensionData` for each `pool.swap` call, and have `SwapAllowlistExtension.beforeSwap` decode and check it when present.
2. **Extension-side:** `SwapAllowlistExtension.beforeSwap` should decode a trusted sender from `extensionData` (e.g., `abi.decode(extensionData, (address))`) and fall back to the raw `sender` only for direct (non-router) calls. The pool interface could also be extended to carry an originating EOA field analogous to `owner` in `addLiquidity`.

The `DepositAllowlistExtension` pattern — gating on the economically relevant actor passed explicitly, not the intermediary — should be adopted for the swap allowlist.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension`. Set `allowAllSwappers[pool] = false`. Call `setAllowedToSwap(pool, alice, true)` — only Alice is allowlisted.

2. **Mode A (bypass):** Call `setAllowedToSwap(pool, router, true)` to permit router-mediated swaps. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle` targeting the pool. Pool calls `_beforeSwap(router, ...)`. Extension checks `allowedSwapper[pool][router] == true` → swap succeeds. Bob bypasses the allowlist entirely.

3. **Mode B (broken path):** Do not allowlist the router. Alice (allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle`. Pool calls `_beforeSwap(router, ...)`. Extension checks `allowedSwapper[pool][router] == false` → `NotAllowedToSwap` revert. Alice cannot use the standard periphery path despite being explicitly allowlisted. [6](#0-5) [7](#0-6) [4](#0-3)
