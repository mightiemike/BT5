### Title
Silent Slow-Mode Transaction Drop Causes Permanent Loss of Deposited Collateral — (File: `core/contracts/Endpoint.sol`)

---

### Summary
In `Endpoint.sol`, `_executeSlowModeTransaction` deletes a queued slow-mode transaction from storage **before** attempting to process it. If processing reverts, the catch block silently discards the transaction with no fund-recovery path. For `DepositCollateral` slow-mode entries, the user's tokens have already been transferred to the contract at queue time, so a silent drop permanently locks those tokens with no recourse. The comment `// try return funds now removed` in the catch block explicitly confirms the recovery mechanism was intentionally removed.

---

### Finding Description

`depositCollateralWithReferral` in `Endpoint.sol` performs two sequential actions:

1. Transfers the user's tokens into the Endpoint contract via `handleDepositTransfer`.
2. Enqueues a `DepositCollateral` slow-mode transaction. [1](#0-0) 

Later, when `_executeSlowModeTransaction` runs (callable by anyone via `executeSlowModeTransaction()`), it:

1. Reads the queued transaction.
2. **Deletes it from storage** (`delete slowModeTxs[_slowModeConfig.txUpTo++]`).
3. Wraps execution in a `try/catch`.
4. On failure, the catch block does nothing — the comment `// try return funds now removed` confirms the refund path was removed. [2](#0-1) 

If `processSlowModeTransaction` reverts for any reason (product delisted between deposit and execution, user address sanctioned after deposit, `clearinghouse.depositCollateral` reverts due to a state change, etc.), the slow-mode entry is gone from the queue and the tokens remain locked in the Endpoint contract forever. There is no admin recovery function, no re-queue mechanism, and no refund path.

---

### Impact Explanation

A user's deposited collateral tokens are permanently locked inside the Endpoint contract. The user loses the full deposited amount with no on-chain recovery path. The corrupted state is: `token.balanceOf(endpoint)` increases by `amount`, but the corresponding `spotEngine` balance for the user's subaccount is never credited. The asset delta is a permanent loss of `amount` tokens for the depositor.

---

### Likelihood Explanation

**Medium.** The slow-mode queue has a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). During that window, several realistic conditions can cause `processSlowModeTransaction` to revert:

- The deposited product is delisted via `DelistProduct` before the slow-mode tx executes.
- The depositor's address is added to the sanctions list between deposit and execution.
- Any revert inside `clearinghouse.depositCollateral` due to a state change in the 3-day window.

These are not exotic conditions — product lifecycle changes and sanctions additions are normal protocol operations.

---

### Recommendation

Restore a fund-recovery path inside the catch block of `_executeSlowModeTransaction`. Specifically, for `DepositCollateral` slow-mode transactions that fail, the contract should refund the deposited token amount back to the original sender. Alternatively, failed slow-mode transactions should be re-queued or flagged for admin recovery rather than silently discarded.

---

### Proof of Concept

1. User calls `depositCollateralWithReferral(subaccount, productId, amount, referral)`.
   - `handleDepositTransfer` moves `amount` tokens from user → Endpoint.
   - A `DepositCollateral` slow-mode tx is enqueued at index `txCount`. [1](#0-0) 

2. Within the 3-day window, the owner calls `submitSlowModeTransaction` with a `DelistProduct` tx for `productId`, which executes immediately via the sequencer.

3. After 3 days, anyone calls `executeSlowModeTransaction()`.
   - `_executeSlowModeTransaction` deletes the entry: `delete slowModeTxs[_slowModeConfig.txUpTo++]`.
   - `try this.processSlowModeTransaction(...)` calls `clearinghouse.depositCollateral(txn)`.
   - `clearinghouse.depositCollateral` reverts because `productId` is delisted.
   - The catch block executes: no refund, no re-queue. [3](#0-2) 

4. The user's `amount` tokens remain in the Endpoint contract permanently. The slow-mode entry no longer exists in `slowModeTxs`. There is no function to recover the funds.
