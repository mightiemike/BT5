Looking at the `SwapAllowlistExtension`, `MetricOmmPool.swap`, `ExtensionCalling._beforeSwap`, and `MetricOmmSimpleRouter` call chain reveals a concrete wrong-actor binding issue.

---

### Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the actual user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of the pool's `swap` call. When `MetricOmmSimpleRouter` mediates the swap, `sender` is the router address, not the end user. A pool admin who allowlists the router to enable router-mediated swaps for their curated users inadvertently opens the pool to every user who calls through the router.

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension dispatcher.**

In `MetricOmmPool.swap`:

```solidity
_beforeSwap(
    msg.sender,   // ← this is the router when the router calls swap
    recipient,
    ...
);
``` [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards that value as the first argument to every configured extension.** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router.**

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [3](#0-2) 

**Step 4 — `MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap` directly, making the router the `msg.sender` seen by the pool.** [4](#0-3) 

The result is a binary, inescapable choice for the pool admin:

| Admin action | Effect |
|---|---|
| Do **not** allowlist the router | No user can swap through the router, even allowlisted ones |
| **Allowlist the router** | **Every user** can swap through the router, bypassing individual allowlist entries |

There is no configuration that allows only the admin's curated set of users to swap through the router while blocking others.

### Impact Explanation

A pool deployed with `SwapAllowlistExtension` for regulatory or curation purposes (e.g., KYC-gated trading) must allowlist the router to let its approved users access the standard periphery. Once the router is allowlisted, any address — including addresses explicitly absent from the allowlist — can call `MetricOmmSimpleRouter.exactInputSingle` and execute a swap on the curated pool. The allowlist is completely ineffective for the router path. This is a direct policy bypass with fund-flow consequences: unauthorized counterparties trade against LP positions that were intended to be restricted.

### Likelihood Explanation

Medium-to-high. Any curated pool that wants to support the standard router (the primary user-facing entry point) must allowlist it. The admin's intent — "allow my approved users to use the router" — is reasonable and expected. The unintended consequence — "this also allows every other user" — is non-obvious from the extension's documented behavior ("Gates `swap` by swapper address, per pool").

### Recommendation

The extension must gate the **end user**, not the intermediary. Two viable approaches:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires the extension to trust that the router is the caller (verified via `msg.sender == router`).
2. **Separate router-level allowlist from user-level allowlist**: Introduce a two-tier check — `allowedSwapper[pool][sender]` for direct calls, and a separate `allowedRouterUser[pool][decodedUser]` for router calls — so the admin can allowlist the router as a transport without opening the user gate.

### Proof of Concept

```
1. Admin deploys pool with SwapAllowlistExtension.
2. Admin calls setAllowedToSwap(pool, alice, true)   // alice is KYC'd
3. Admin calls setAllowedToSwap(pool, router, true)  // to let alice use the router
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(...); pool's msg.sender = router.
6. Extension checks allowedSwapper[pool][router] → true.
7. Bob's swap executes on the curated pool. Allowlist bypassed.
``` [5](#0-4) [6](#0-5) [7](#0-6)
