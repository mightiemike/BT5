### Title
Unprivileged Caller Can Aggressively Prune Mainchain Block Headers via `run_mainchain_gc` - (File: contract/src/lib.rs)

### Summary

`run_mainchain_gc` is a public, state-mutating function on `BtcLightClient` that carries no caller authorization check. Any unprivileged NEAR account can call it with an arbitrarily large `batch_size`, immediately deleting all mainchain block headers that exceed `gc_threshold` in a single transaction. This permanently corrupts the light client's historical header state, breaks SPV proof verification for affected blocks, and can cause any pending chain reorg to fail irrecoverably.

### Finding Description

`submit_blocks` is correctly gated behind `#[trusted_relayer]`, which enforces that only registered, staked relayers may submit headers. However, `run_mainchain_gc` — which permanently deletes mainchain block headers from all three storage maps — carries only `#[pause(except(roles(Role::UnrestrictedRunGC)))]`:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

The `#[pause]` attribute controls whether the function is callable when the contract is **paused**. It imposes no restriction on the caller when the contract is running normally. Any NEAR account — with no role, no stake, no registration — can call `run_mainchain_gc(u64::MAX)` at any time.

Inside the function, `batch_size` is attacker-controlled and is only capped by `min(total_amount_to_remove, batch_size)`, where `total_amount_to_remove = amount_of_headers_we_store - gc_threshold`. An attacker supplying `u64::MAX` forces the maximum eligible deletion in a single call — the same deletion that the relayer would spread across many `submit_blocks` calls over time.

The deletion is permanent and irreversible:

```rust
self.remove_block_header(blockhash);           // removes from mainchain_header_to_height + headers_pool
self.mainchain_height_to_header.remove(&height);
```

`remove_block_header` erases the entry from both `mainchain_header_to_height` and `headers_pool`. Once gone, those headers cannot be recovered without redeploying from a known-good state.

The contrast with `submit_blocks` is exact: `submit_blocks` calls `self.run_mainchain_gc(num_of_headers)` internally after each batch, spreading GC across many authorized relayer calls. The public `run_mainchain_gc` entry point allows any caller to collapse that gradual process into a single unauthenticated call.

### Impact Explanation

**Broken SPV proof verification.** `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the block header in `headers_pool` by blockhash. If the block was GC'd by the attacker, the lookup panics with `"cannot find requested transaction block"`. Any downstream contract or user relying on SPV proofs for blocks in the deleted range receives a permanent failure.

**Irrecoverable chain reorg failure.** As documented in `contract/CLAUDE.md`: *"If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound` when it cannot walk the chain back to the common ancestor."* An attacker who calls `run_mainchain_gc(u64::MAX)` immediately after a fork is detected can ensure the reorg walk panics, permanently locking the chain on the wrong tip.

**Corrupted `mainchain_initial_blockhash`.** The function updates `self.mainchain_initial_blockhash` to the new oldest block. Forcing this pointer forward prematurely narrows the window of verifiable history, making the light client useless for any transaction older than the new initial block.

### Likelihood Explanation

The attack requires no privileged keys, no staking, no registration, and no special knowledge beyond the contract's public ABI. Any NEAR account with enough gas can execute it. The function is callable at any time the contract is not paused. The attacker's only cost is gas for the transaction.

### Recommendation

Add the same `#[trusted_relayer]` guard (or an equivalent role check) to `run_mainchain_gc` that protects `submit_blocks`. If the intent is to allow any caller to trigger GC (e.g., to let anyone help with storage management), the function should at minimum enforce a maximum `batch_size` cap and/or require a small deposit to prevent griefing. The simplest fix matching the existing pattern is:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
#[trusted_relayer]   // add this
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

### Proof of Concept

1. Deploy the contract and initialize it with a `gc_threshold` of, say, 100 blocks.
2. Have the legitimate relayer submit 200 blocks (so 100 are eligible for GC).
3. From any unprivileged NEAR account (no role granted), call:
   ```
   run_mainchain_gc(batch_size: 18446744073709551615)
   ```
4. All 100 eligible blocks are deleted in one transaction.
5. Call `verify_transaction_inclusion` for any transaction in one of the deleted blocks — the call panics with `"cannot find requested transaction block"`.
6. Submit a fork header that requires walking back through the deleted range — the reorg panics with `"PrevBlockNotFound"`.
