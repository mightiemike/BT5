### Title
Negative `canSettle` in `settlePnl` Inflates `availableSettle` and Drains Spot QUOTE — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.settlePnl` computes `canSettle` via `MathHelper.min(positionPnl, state.availableSettle)`. When a subaccount holds a perp position with negative PnL and `state.availableSettle > 0`, `canSettle` is negative. The function then applies `state.availableSettle -= canSettle` (inflating it) and `balance.vQuoteBalance -= canSettle` (improving the perp balance), while `Clearinghouse._settlePnl` credits the subaccount's spot QUOTE with the negative `amountSettled` (decreasing it). The `availableSettle` pool is permanently inflated, allowing other traders to extract more settled quote than the protocol has insurance backing for.

---

### Finding Description

**Entry path:**

`Clearinghouse.settlePnl` (line 629) is `onlyEndpoint`. The sequencer processes user-submitted `SettlePnl` requests without filtering on the sign of the caller's PnL. It calls `_settlePnl` (line 617), which calls `perpEngine.settlePnl` (line 620) and then credits the subaccount's spot QUOTE with the returned `amountSettled`.

**Root cause — `getSettlementState`:** [1](#0-0) 

`MathHelper.min` is a plain signed comparison with no non-negativity guard: [2](#0-1) 

When `positionPnl < 0` and `state.availableSettle > 0`, `canSettle = positionPnl` (negative).

**Root cause — `settlePnl` applies the negative value without a guard:** [3](#0-2) 

- `state.availableSettle -= canSettle` → subtracting a negative value **increases** `availableSettle`.
- `balance.vQuoteBalance -= canSettle` → **increases** the perp vQuote balance (position PnL goes to zero).
- `totalSettled += canSettle` → returns a **negative** amount to the Clearinghouse.

**Root cause — Clearinghouse credits a negative `amountSettled` to spot QUOTE:** [4](#0-3) 

`spotEngine.updateBalance(QUOTE_PRODUCT_ID, subaccount, amountSettled)` with `amountSettled < 0` **decreases** the subaccount's spot QUOTE balance.

**Net per-call accounting:**

| Variable | Change |
|---|---|
| `state.availableSettle` | +\|positionPnl\| (inflated) |
| `balance.vQuoteBalance` | +\|positionPnl\| (position PnL → 0) |
| Spot QUOTE balance | −\|positionPnl\| |
| Trader net wealth | 0 |

The trader's own wealth is unchanged (perp improves, spot worsens by the same amount). However, `availableSettle` is permanently inflated by `|positionPnl|` per call. After one call the position PnL becomes exactly zero, so the same subaccount cannot repeat the inflation on the same position — but any subaccount with a fresh negative-PnL position can do it again.

**No health check exists after `settlePnl`:** [5](#0-4) 

There is no `require(getHealth(...) >= 0)` guard, so the call succeeds regardless of the resulting spot QUOTE balance.

---

### Impact Explanation

`availableSettle` is the settlement pool funded by insurance contributions via `socializeSubaccount`: [6](#0-5) 

Its invariant is that it must never exceed the sum of insurance contributions. Inflating it allows traders with positive PnL to call `settlePnl` and extract spot QUOTE that has no insurance backing, directly breaking protocol solvency. The more negative-PnL positions exist across subaccounts, the larger the inflation and the larger the shortfall imposed on the protocol.

---

### Likelihood Explanation

The attack requires only a subaccount with a negative-PnL perp position and a non-zero `availableSettle`. Both conditions are routine in normal protocol operation. The sequencer processes `SettlePnl` requests without sign-checking the caller's PnL. No special privileges, governance access, or sequencer compromise is needed.

---

### Recommendation

In `PerpEngine.settlePnl`, skip (or clamp to zero) any product where `canSettle <= 0`:

```solidity
// In the while loop, after calling getSettlementState:
if (canSettle <= 0) {
    productIds >>= 32;
    continue;
}
```

Alternatively, clamp inside `getSettlementState`:

```solidity
availableSettle = MathHelper.max(
    0,
    MathHelper.min(
        calculatePositionPnl(balance, productId),
        state.availableSettle
    )
);
``` [7](#0-6)

```solidity

// - state.availableSettle = 1e18 (funded by prior insurance contribution)
// - Attacker subaccount has perp position with positionPnl = -0.5e18
// - Attacker subaccount has spot QUOTE balance = 1e18

// Step 1: Attacker submits SettlePnl request to sequencer for their subaccount.
// Sequencer calls Clearinghouse.settlePnl → _settlePnl → perpEngine.settlePnl.

// Inside getSettlementState:
//   canSettle = MathHelper.min(-0.5e18, 1e18) = -0.5e18

// Inside PerpEngine.settlePnl:
//   state.availableSettle -= (-0.5e18)  → availableSettle = 1.5e18  (INFLATED)
//   balance.vQuoteBalance -= (-0.5e18)  → vQuoteBalance += 0.5e18   (positionPnl → 0)
//   totalSettled = -0.5e18

// Inside Clearinghouse._settlePnl:
//   spotEngine.updateBalance(QUOTE_PRODUCT_ID, attacker, -0.5e18)
//   → attacker spot QUOTE = 0.5e18  (decreased)

// Result:
//   availableSettle = 1.5e18  (was 1e18, inflated by 0.5e18)
//   A legitimate trader with +0.5e18 positionPnl can now settle against
//   the inflated pool, extracting 0.5e18 spot QUOTE that has no backing.

// Assert: availableSettle_after (1.5e18) > availableSettle_before (1e18) — invariant broken.
// Assert: totalSettled (-0.5e18) < 0 — invariant broken.
