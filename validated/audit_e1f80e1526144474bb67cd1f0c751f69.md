### Title
Missing Access Control on `run_mainchain_gc()` Allows Any Caller to Prune Canonical Chain State - (File: `contract/src/lib.rs`)

### Summary
`run_mainchain_gc()` is a public, state-mutating function that permanently removes mainchain block headers from contract storage. It is called internally by the access-controlled `submit_blocks()` function, but is also directly callable by any unprivileged NEAR account with no caller identity check. This is a direct structural analog to the reported `releaseTokens()` missing access control: a function intended to be invoked only through a privileged path is exposed as an unrestricted public entrypoint.

### Finding Description

`submit_blocks()` is guarded by `#[trusted_relayer]`, restricting it to accounts holding `Role::DAO` or `Role::UnrestrictedSubmitBlocks`. [1](#0-0) 

Inside `submit_blocks()`, GC is triggered internally: [2](#0-1) 

However, `run_mainchain_gc()` is also exposed as a standalone public function: [3](#0-2) 

Its only attribute is `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. The `#[pause]` macro from `near-plugins` exclusively checks whether the contract is in a paused state — it does **not** enforce any caller identity restriction. When the contract is unpaused (its normal operating state), any NEAR account can call `run_mainchain_gc(batch_size)` with an arbitrary `batch_size`.

The function mutates three critical state fields:
- Removes entries from `mainchain_height_to_header` and calls `remove_block_header` (which removes from `headers_pool`)
- Overwrites `mainchain_initial_blockhash` to point to a newer block [4](#0-3) 

### Impact Explanation

**Reorg prevention (canonical chain corruption):** The contract's own documentation explicitly states: *"If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound` when it cannot walk the chain back to the common ancestor."* [5](#0-4) 

An attacker monitoring the mempool can call `run_mainchain_gc` with a large `batch_size` immediately before or during a legitimate chain reorganization, removing the blocks needed to walk back to the common ancestor. This permanently freezes the light client on the wrong chain tip, as the reorg can never complete once those blocks are deleted.

**SPV proof invalidation:** `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up `mainchain_header_to_height` to confirm a block belongs to the current main chain. [6](#0-5) 

Aggressively GC-ing blocks invalidates SPV proofs for older transactions that downstream contracts depend on, even when those transactions are genuinely confirmed.

### Likelihood Explanation

**High.** The function is unconditionally reachable by any NEAR account when the contract is unpaused. No staking, no role, no deposit is required. The attacker only needs to know the contract address and call `run_mainchain_gc` with `batch_size = u64::MAX`. The attack is cheap (single transaction) and the damage (deleted storage entries) is irreversible.

### Recommendation

Add a caller identity check to `run_mainchain_gc()` analogous to the `#[trusted_relayer]` guard on `submit_blocks()`. The function should only be callable by accounts holding a privileged role (e.g., `Role::DAO`, `Role::UnrestrictedSubmitBlocks`, or a new dedicated `GCManager` role), or it should be made `pub(crate)` and removed from the public NEAR ABI entirely, since its intended invocation path is already through `submit_blocks()`.

### Proof of Concept

1. Deploy the contract and initialize it with a `gc_threshold` of, say, 100 blocks.
2. Have a trusted relayer submit 200 blocks, bringing the mainchain to height 200 (100 blocks eligible for GC).
3. Suppose a fork exists at height 150 that is about to accumulate more chainwork than the main chain.
4. Before the reorg transaction lands, an attacker calls:
   ```
   run_mainchain_gc(batch_size: 18446744073709551615)
   ```
   from any unprivileged account.
5. The function removes blocks at heights 0–99 (the 100 eligible blocks), updating `mainchain_initial_blockhash` to height 100.
6. When the relayer now submits the fork tip to trigger the reorg, the contract attempts to walk back to the common ancestor at height 149, but blocks 100–149 may have been removed, causing a `PrevBlockNotFound` panic.
7. The light client is permanently stuck on the stale main chain tip; the fork can never be applied. [7](#0-6)

```rust
    #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
```
