### Title
`verify_transaction_inclusion` Accepts Unvalidated `tx_id` Enabling Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The public `verify_transaction_inclusion` function (v1) accepts a caller-supplied `tx_id` and passes it directly into `compute_root_from_merkle_proof` without validating that it is a genuine leaf node (transaction hash) rather than an internal Merkle tree node. An unprivileged NEAR caller can supply an internal node hash as `tx_id` with a correspondingly shorter proof, causing the function to return `true` for a transaction that does not exist in the block.

---

### Finding Description

`verify_transaction_inclusion` in `contract/src/lib.rs` performs the following steps:

1. Confirms the block is on the main chain and has enough confirmations.
2. Checks `!args.merkle_proof.is_empty()`.
3. Calls `merkle_tools::compute_root_from_merkle_proof(args.tx_id, tx_index, &args.merkle_proof)` and compares the result to `header.block_header.merkle_root`. [1](#0-0) 

There is no check that `args.tx_id` is a leaf-level hash. `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` simply iterates over the proof hashes and combines them with the current hash using left/right ordering determined by `current_position % 2`: [2](#0-1) 

Because Bitcoin's Merkle tree uses the same double-SHA256 hash function for both leaf nodes and internal nodes, an internal node `I = H(left_txid ‖ right_txid)` is itself a valid 32-byte value (H256). An attacker can supply `tx_id = I` with a proof one level shorter than the full tree depth, and `compute_root_from_merkle_proof` will hash `I` upward to the correct `merkle_root`, returning `true`.

The code itself documents this gap: [3](#0-2) 

The `#[deprecated]` Rust attribute is a compile-time lint only; the function remains fully callable at runtime by any NEAR account. The `#[pause]` guard is not activated by default. [4](#0-3) 

The v2 function (`verify_transaction_inclusion_v2`) partially mitigates this by requiring `merkle_proof.len() == coinbase_merkle_proof.len()`, which forces the target proof to have the same depth as the coinbase proof (leaf depth). However, v1 remains independently callable and unprotected. [5](#0-4) 

---

### Impact Explanation

Any downstream contract or application that calls `verify_transaction_inclusion` (v1) and acts on a `true` result — e.g., releasing funds, minting tokens, or authorizing a cross-chain action — can be deceived into treating a non-existent transaction as confirmed. The attacker does not need any privileged role; they only need to know the Merkle tree structure of a real Bitcoin block (publicly available) and submit a crafted `ProofArgs` call.

**Impact: High** — Proof-verification forgery; a recipient contract consuming the `true` result has no way to distinguish a forged proof from a genuine one.

---

### Likelihood Explanation

The attack requires:
- Knowledge of any confirmed Bitcoin block's Merkle tree (public data).
- A single NEAR transaction calling `verify_transaction_inclusion`.

No special access, staking, or timing is required. The function is open to any NEAR account while unpaused.

**Likelihood: High**

---

### Recommendation

1. **Remove or gate v1**: Either delete `verify_transaction_inclusion` or restrict it to privileged roles so unprivileged callers cannot reach it.
2. **Enforce proof depth equality in v1**: Add `require!(args.merkle_proof.len() == expected_depth)` where `expected_depth` is derived from a validated coinbase proof, mirroring the v2 approach.
3. **Migrate all callers to v2**: Ensure no production consumer still calls the deprecated v1 endpoint.

---

### Proof of Concept

Given a confirmed Bitcoin block with four transactions `[A, B, C, D]`:

```
Merkle tree:
  root = H(H(A,B), H(C,D))
  I    = H(A, B)            ← internal node, 32-byte H256 value
```

Attacker calls `verify_transaction_inclusion` with:

```
tx_id             = I  (= H(A,B), a real 32-byte value, not a real txid)
tx_block_blockhash = <any confirmed block hash containing A,B,C,D>
tx_index          = 0
merkle_proof      = [ H(C,D) ]   ← one element, not two
confirmations     = 1
```

Inside `compute_root_from_merkle_proof`:
- `current_hash = I`, `current_position = 0`
- Iteration 1: `0 % 2 == 0` → `current_hash = H(I, H(C,D)) = H(H(A,B), H(C,D)) = root`

Result equals `header.block_header.merkle_root` → function returns **`true`**, falsely asserting that transaction `I` is confirmed in the block. [6](#0-5) [7](#0-6)

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
