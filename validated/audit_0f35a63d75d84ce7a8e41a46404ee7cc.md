### Title
`pending_tier3_requests` entry removed on single routing failure, invalidating all other in-flight T3 authorizations for the same peer — (`chain/network/src/peer_manager/peer_manager_actor.rs`)

### Summary

`pending_tier3_requests` is a `PeerId → Instant` map (one entry per peer) used to authorize inbound Tier-3 connections. Multiple independent state-sync requests (`StateRequestHeader` + N `StateRequestPart`) can be dispatched to the same peer. Each dispatch overwrites the single map entry. If any one of those dispatches fails to route, the code calls `pending_tier3_requests.remove(&peer_id)`, erasing the authorization for **all** other already-sent requests to that peer. The peer's subsequent inbound T3 connections are then rejected with `UnexpectedTier3Connection`, stalling state sync.

### Finding Description

Both `StateRequestHeader` and `StateRequestPart` handlers follow the same pattern:

```rust
self.state.pending_tier3_requests.insert(peer_id.clone(), self.clock.now());
if !self.state.send_message_to_peer(...) {
    self.state.pending_tier3_requests.remove(&peer_id);   // ← removes the whole entry
    return NetworkResponses::RouteNotFound;
}
``` [1](#0-0) [2](#0-1) 

The map is keyed only by `PeerId`. When `select_host_for_header` and `select_host_for_part` both return the same peer (the common case when few snapshot hosts exist), each call overwrites the single entry. If the Nth dispatch fails, `remove` wipes the entry that was authorizing all N-1 already-sent requests.

The inbound T3 validation gate checks only `contains_key`:

```rust
if !self.pending_tier3_requests.contains_key(&peer_info.id) {
    return Err(RegisterPeerError::UnexpectedTier3Connection);
}
``` [3](#0-2) 

Once the entry is gone, every T3 connection attempt from that peer is rejected, regardless of how many valid requests are still outstanding.

The analog to the external bug is exact:

| Knox auction | nearcore |
|---|---|
| `epochsByBuyer[epoch]` — one entry per epoch | `pending_tier3_requests[peer_id]` — one entry per peer |
| User places two orders in same epoch; only one entry added | Node sends header + N part requests to same peer; each overwrites the single entry |
| Cancelling one order removes the epoch entry | One failed routing call removes the peer entry |
| Other orders in that epoch become untracked | Other pending T3 connections from that peer are rejected |

### Impact Explanation

State sync stalls. A syncing node that has already dispatched a header request and several part requests to the same snapshot host loses all T3 authorizations for that host if any single subsequent dispatch fails (e.g., the T2 connection drops between dispatches). The host's inbound T3 connections are rejected, and the node must wait for the `PENDING_TIER3_REQUEST_TIMEOUT` cleanup cycle before retrying. A malicious snapshot host can exploit this deliberately: accept the T2 routing of the first request, drop the T2 connection before the second, then reconnect — causing the remove — and repeat, permanently preventing the victim from completing state sync against that host.

### Likelihood Explanation

Reachable in normal operation whenever a single snapshot host serves multiple shards or multiple parts. The `select_host_for_header` and `select_host_for_part` functions both draw from `hosts_for_shard`, which will return the same peer when only one host is available for a shard. Any archival node advertising state snapshots can trigger this without validator privileges. [4](#0-3) [5](#0-4) 

### Recommendation

Replace the scalar `PeerId → Instant` map with a reference-counted structure: `PeerId → (count: usize, earliest_sent_at: Instant)`. Increment the count on each successful dispatch; decrement (and remove only when count reaches zero) on each routing failure. This mirrors the correct fix for the Knox bug: "only remove the `epochsByBuyer` record when **all** orders for that epoch are cancelled."

### Proof of Concept

1. Node needs to sync state; only peer **P** advertises the required snapshot.
2. `StateRequestHeader` dispatched to P → `pending_tier3_requests[P] = t1`; T2 send succeeds.
3. `StateRequestPart(part=0)` dispatched to P → `pending_tier3_requests[P] = t2` (overwrites); T2 send succeeds.
4. `StateRequestPart(part=1)` dispatched to P → `pending_tier3_requests[P] = t3` (overwrites); T2 send **fails** (P dropped the T2 connection).
5. `pending_tier3_requests.remove(P)` executes — entry gone.
6. P reconnects on T3 to deliver the header and part 0 → `validate_new_connection` returns `UnexpectedTier3Connection`; both connections rejected.
7. State sync for this node stalls until timeout and retry. [6](#0-5) [7](#0-6) [8](#0-7)

    pub fn select_host_for_part(
        &mut self,
        sync_hash: &CryptoHash,
        shard_id: ShardId,
        part_id: u64,
    ) -> Option<PeerId> {
        self.update_current_state_sync_hash(sync_hash);

        let selector =
            self.peer_selector.entry((shard_id, part_id)).or_insert(PartPeerSelector::default());

        // Insert more hosts into the selector if needed
        let available_hosts = self.hosts_for_shard.get(&shard_id)?;
        if selector.tried_everybody() && selector.len() < available_hosts.len() {
            let mut new_peers = BinaryHeap::new();
            let already_included = selector.peer_set();

            for peer_id in available_hosts {
                if already_included.contains(peer_id) {
                    continue;
                }

                let score = priority_score(peer_id, shard_id, part_id);

                // Wrap entries with `Reverse` so that we pop the *least* desirable options
                new_peers.push(std::cmp::Reverse(StatePartHost {
                    peer_id: peer_id.clone(),
                    score,
                    num_requests: 0,
                }));

                if new_peers.len() > self.part_selection_cache_batch_size {
                    new_peers.pop();
                }
            }

            selector.insert_peers(new_peers.drain().map(|e| e.0));
        }

        let res = selector.next();
        res
    }
