### Title
Gateway admits `DeclareV3` transactions with non-empty `account_deployment_data` that the OS unconditionally rejects - (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary
The gateway validates that `paymaster_data` is empty for all V3 transaction types because the Starknet OS enforces this constraint. An identical OS-level constraint exists for `account_deployment_data` in declare transactions (`assert account_deployment_data_size = 0`), but the gateway has no corresponding rejection check. Any `DeclareV3` transaction with non-empty `account_deployment_data` passes gateway admission and enters the mempool, yet is guaranteed to fail at OS execution time.

### Finding Description

**Gateway-side (missing validation):**

`StatelessTransactionValidator::validate_empty_paymaster_data` rejects all V3 transactions whose `paymaster_data` is non-empty, with the explicit rationale: *"The Starknet OS enforces that the paymaster data is empty. We add this validation here in the gateway to prevent transactions from failing the OS."* [1](#0-0) 

No analogous `validate_empty_account_deployment_data` function exists for `DeclareV3` transactions. The `RpcDeclareTransactionV3` struct carries the field, and the Rust hash function `get_declare_transaction_v3_hash` includes it in the Poseidon hash chain: [2](#0-1) 

**OS-side (unconditional rejection):**

The OS Cairo code in `transaction_hash.cairo` unconditionally asserts the field is absent for every declare transaction before computing the hash:

```cairo
// TODO(Noa, 01/01/2026): remove the following `assert` once the field is supported.
assert account_deployment_data_size = 0;
``` [3](#0-2) 

**The divergence:** The Rust blockifier computes a hash `H1` that incorporates the non-empty `account_deployment_data`. The OS Cairo execution hits the `assert account_deployment_data_size = 0` trap and aborts before producing any hash. The transaction is guaranteed to fail at execution time, yet the gateway admitted it.

**Contrast with `paymaster_data`:** The gateway correctly rejects non-empty `paymaster_data` for the same OS-enforcement reason. The `account_deployment_data` constraint is identical in the OS but has no gateway mirror. [4](#0-3) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

Any unprivileged user can submit a `DeclareV3` transaction with non-empty `account_deployment_data`. The gateway accepts it, the mempool stores it, and the blockifier schedules it. The OS then aborts execution unconditionally. The gateway's own stated invariant — *"prevent transactions from failing the OS"* — is violated for this field. At scale this enables cheap, unfiltered injection of transactions that consume mempool slots and blockifier execution budget while always reverting.

### Likelihood Explanation

**High — certain to occur.** The `account_deployment_data` field is a standard part of the `DeclareV3` wire format exposed in the public RPC schema. Any client that populates it (e.g., a future account-abstraction wallet following the spec) will have its declare transactions silently accepted by the gateway and then fail at execution. The OS assertion has a `TODO` dated 2026-01-01, confirming the constraint is active and intentional for the current protocol version.

### Recommendation

Add a `validate_empty_account_deployment_data` check in `StatelessTransactionValidator` for `DeclareV3` transactions, mirroring `validate_empty_paymaster_data`:

```rust
/// The Starknet OS enforces that account_deployment_data is empty in declare transactions.
/// We add this validation here to prevent transactions from failing the OS.
fn validate_empty_account_deployment_data_for_declare(
    tx: &RpcTransaction,
) -> StatelessTransactionValidatorResult<()> {
    if let RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) = tx {
        if !tx.account_deployment_data.is_empty() {
            return Err(StatelessTransactionValidatorError::NonEmptyField {
                field_name: "account_deployment_data".to_string(),
            });
        }
    }
    Ok(())
}
```

Remove or update this check when the OS TODO is resolved and `account_deployment_data` is supported in declare transactions.

### Proof of Concept

1. Construct a `RpcDeclareTransactionV3` with `account_deployment_data: AccountDeploymentData(vec![Felt::ONE])` and valid class/signature fields.
2. Submit to the gateway endpoint.
3. `validate_empty_paymaster_data` passes (paymaster is empty). No other check inspects `account_deployment_data`.
4. The transaction is admitted to the mempool with a hash `H1` computed by `get_declare_transaction_v3_hash` over the non-empty field.
5. The blockifier schedules execution; the OS Cairo code reaches `assert account_deployment_data_size = 0` and aborts.
6. The transaction reverts; the sender loses the fee. The gateway's admission invariant is broken. [1](#0-0) [3](#0-2) [2](#0-1) [4](#0-3)

    /// The Starknet OS enforces that the paymaster data is empty. We add this validation here in
    /// the gateway to prevent transactions from failing the OS.
    fn validate_empty_paymaster_data(
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let paymaster_data = match tx {
            RpcTransaction::DeployAccount(RpcDeployAccountTransaction::V3(tx)) => {
                &tx.paymaster_data
            }
            RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => &tx.paymaster_data,
            RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => &tx.paymaster_data,
        };

        if paymaster_data.is_empty() {
            Ok(())
        } else {
            Err(StatelessTransactionValidatorError::NonEmptyField {
                field_name: "paymaster_data".to_string(),
            })
        }
    }
