Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Any Caller to Bypass the Configured Swap Guard — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the immediate caller of `pool.swap()` (the `sender` argument), not the originating end user. When `MetricOmmSimpleRouter` is used, the router contract becomes `sender`. If the pool admin allowlists the router to enable router-mediated swaps for authorized users, every unprivileged caller can bypass the swap allowlist by routing through the same public router, as the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][endUser]`.

## Finding Description
`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool (the pool calls the extension) and `sender` is whoever called `pool.swap()` directly.

`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap` (lines 230–232). When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly (lines 72–80), making the router the `msg.sender` of `pool.swap()`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][endUser]`.

This creates an irreconcilable dilemma: if the pool admin does not allowlist the router, authorized users cannot use the router at all; if the pool admin allowlists the router (the necessary operational step), every unprivileged caller can bypass the allowlist via the same public router. The `extensionData` field is forwarded from the router to the extension but `SwapAllowlistExtension` never reads it, so the end user's identity is permanently lost.

## Impact Explanation
The swap allowlist is the pool admin's primary mechanism for restricting pool access (KYC/AML compliance, protocol-only pools, staged rollouts). Once the router is allowlisted — a necessary step for any authorized user to use the standard router — the guard is fully neutralized for all router-mediated paths. Any unprivileged caller can execute swaps on a pool explicitly configured to block them. This is a direct admin-boundary break: a pool admin-configured guard is bypassed by an unprivileged path through a public periphery contract.

## Likelihood Explanation
The bypass requires the pool admin to have allowlisted the router address, which is the expected operational step whenever authorized users need to use the standard router. Any user who knows the router address (a public, deployed contract) can then exploit the bypass. No special privileges, flash loans, or oracle manipulation are required — only a direct call to `MetricOmmSimpleRouter.exactInputSingle()`.

## Recommendation
`SwapAllowlistExtension` must identify the originating end user, not the immediate pool caller. Two viable approaches:

1. **`extensionData` forwarding**: The router encodes `msg.sender` into `extensionData` before calling the pool. The extension decodes and checks that address against the allowlist. Requires coordinated changes to the router and extension.
2. **Two-tier allowlist**: Introduce a separate set of approved routers. If `sender` is an approved router, decode the end user from `extensionData` and check that address against the per-user allowlist.

The current design where `sender` is the immediate pool caller is fundamentally incompatible with a router-mediated access-control model.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Pool admin allowlists user A directly: `setAllowedToSwap(pool, userA, true)`.
3. Pool admin allowlists the router so user A can use it: `setAllowedToSwap(pool, router, true)`.
4. User B (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(...)` — `msg.sender` of `pool.swap()` = router address.
6. Pool calls `extension.beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
8. User B's swap executes on the restricted pool, bypassing the configured guard entirely.
