### Title
Unbounded Loop in `reorg_chain` Can Exceed NEAR Gas Limit, Permanently Stalling Fork Promotion — (`File: contract/src/lib.rs`)

### Summary
`reorg_chain` contains two loops whose iteration count is proportional to the depth of a fork relative to the main chain. Because fork depth is unbounded within a single NEAR transaction, a sufficiently deep reorganization causes the call to exhaust the NEAR gas budget (~300 TGas), revert with no state change, and leave the contract permanently unable to promote the heavier fork to the canonical chain.

### Finding Description

`reorg_chain` is invoked from `submit_block_header_inner` whenever a fork header's cumulative chainwork exceeds the current main chain tip's chainwork. [1](#0-0) 

Inside `reorg_chain`, two loops run whose bounds are set by the fork depth — i.e., the number of headers that diverge from the common ancestor:

**Loop 1** — removes main-chain blocks above the fork tip height: [2](#0-1) 

**Loop 2** — walks the entire fork chain backwards from the fork tip to the common ancestor, promoting each fork block into the main chain and evicting the displaced main-chain block: [3](#0-2) 

Neither loop has an internal cap. Each iteration performs multiple `LookupMap` storage reads and writes (`insert`, `remove`, `contains_key`), which are among the most gas-expensive operations on NEAR. The fork depth is bounded only by `gc_threshold` (default 52,704 blocks per the recommended init comment). [4](#0-3) 

A fork of even a few hundred blocks, each iteration doing 3–5 storage operations, will exhaust the 300 TGas per-transaction limit. When the transaction reverts, the fork remains in `headers_pool` with higher chainwork than the main chain tip, but `mainchain_tip_blockhash` is never updated. Every subsequent `submit_blocks` call that extends this fork will re-enter `reorg_chain` and fail identically, permanently blocking the promotion.

### Impact Explanation

Once the reorg is blocked, `mainchain_tip_blockhash` points to a stale block on the lighter chain. All downstream consumers of the light client are affected:

- `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` check confirmations and Merkle roots against the stale canonical chain. [5](#0-4) 

Transactions on the true (heavier) chain cannot be verified; transactions on the abandoned lighter chain can be falsely verified as confirmed. This is a **fork-choice corruption** and **proof-verification integrity failure**, not merely a performance issue.

### Likelihood Explanation

Bitcoin and its derivatives (Dogecoin, Litecoin, Zcash) all support by-design chain reorganizations. A reorg of 100–500 blocks is rare but protocol-valid and has occurred historically. The relayer's job is to faithfully submit all valid headers; it has no mechanism to detect that the resulting reorg depth will exceed the gas budget before submitting the tip. The condition is therefore reachable through normal, non-malicious protocol operation.

### Recommendation

Apply the same fix class as the Visor patch: process the reorg in bounded batches rather than in a single unbounded loop. Specifically:

1. Add a `reorg_batch_size` parameter (or reuse `batch_size`) to `reorg_chain` and break after processing that many headers, storing a "pending reorg cursor" in contract state.
2. Expose a separate `continue_reorg` entry point (analogous to `run_mainchain_gc`) that resumes from the cursor across multiple transactions.
3. Alternatively, cap the maximum fork depth the contract will accept in a single `submit_blocks` call and require the relayer to submit fork headers in bounded batches.

### Proof of Concept

1. Initialize the contract with `gc_threshold = 52704`.
2. Submit 500 main-chain headers via `submit_blocks` (fork A, becomes canonical).
3. Submit 499 fork headers via `submit_blocks` starting from the common ancestor (fork B, stored in `headers_pool` but not promoted because chainwork < main chain).
4. Submit the 500th fork-B header whose cumulative chainwork now exceeds fork A's tip.
5. `submit_block_header_inner` detects `current_header.chain_work > total_main_chain_chainwork` and calls `reorg_chain`. [6](#0-5) 

6. `reorg_chain`'s `while` loop must walk all 500 fork-B headers back to the common ancestor, performing ~3 storage ops per iteration (~1500 storage ops total). This exhausts the NEAR gas budget and the transaction reverts.
7. `mainchain_tip_blockhash` remains at fork-A's tip. All subsequent calls to `verify_transaction_inclusion` with fork-B block hashes return `false` even though fork B is the true canonical chain. [7](#0-6)

    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
