Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on caller-controlled `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and instead checks `owner`, which is a free caller-supplied argument. Because `owner` is not authenticated, any unprivileged address can pass any allowlisted address as `owner` and satisfy the allowlist check, injecting liquidity into a curated pool without ever being allowlisted. This breaks the core pool invariant that only allowlisted depositors may modify LP state on a curated pool.

## Finding Description
`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)` where `msg.sender` is the real caller and `owner` is a caller-supplied parameter with no authentication constraint. `ExtensionCalling._beforeAddLiquidity` encodes both and dispatches to the extension via `abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, ...))`. Inside `DepositAllowlistExtension.beforeAddLiquidity`, the `sender` positional argument is unnamed and ignored; the check reads `allowedDepositor[msg.sender][owner]`, where `msg.sender` is the pool and `owner` is the attacker-controlled value. An attacker reads any allowlisted address from the public `allowedDepositor` mapping, passes it as `owner`, and the check passes. `SwapAllowlistExtension.beforeSwap` correctly gates on `sender` and ignores `recipient`, confirming the asymmetry in `DepositAllowlistExtension` is a defect. Existing guards (`nonReentrant`, `LiquidityDeltaLengthMismatch`) do not address actor identity.

## Impact Explanation
Any address can add liquidity to a pool whose admin deployed `DepositAllowlistExtension` to restrict depositors. The attacker modifies bin totals and bin distribution without authorization, breaking the admin-boundary invariant. With a cooperating allowlisted address as `owner`, the attacker can later recover deposited tokens via `owner.removeLiquidity()`, making the bypass economically free. Even without collusion, the unauthorized LP position corrupts pool state (bin balances, `binTotals.scaledToken0/1`, `_positionBinShares`), constituting broken core pool functionality with direct fund-impacting consequences.

## Likelihood Explanation
Exploitation requires only reading one allowlisted address from the public `allowedDepositor` mapping and calling `addLiquidity` with that address as `owner`. No special privileges, flash loans, or complex setup are needed. Any user can trigger this on any pool using `DepositAllowlistExtension`, at any time, repeatedly.

## Recommendation
Gate on `sender`, not `owner`, consistent with `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```solidity
// Pool has DepositAllowlistExtension; only `alice` is allowlisted.
// Bob (not allowlisted) reads allowedDepositor[pool][alice] == true from chain.

pool.addLiquidity(
    alice,        // owner — satisfies allowedDepositor[pool][alice] == true
    salt,
    deltas,
    callbackData, // Bob's contract provides tokens in the swap callback
    extensionData
);
// Result: Bob's tokens enter the pool under alice's position.
// alice.removeLiquidity() (with collusion) recovers them.
// Without collusion: bin totals and bin distribution are still corrupted by an unauthorized actor.
```
