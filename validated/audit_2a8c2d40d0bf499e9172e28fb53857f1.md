Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of End User, Breaking Allowlist Invariant — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which propagates it to `SwapAllowlistExtension.beforeSwap`. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end user. The extension therefore evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`, producing two fund-impacting failure modes: a full allowlist bypass when the router is allowlisted, and a lockout of individually-allowlisted users from the primary swap interface when it is not.

## Finding Description

**Root cause — pool passes immediate caller, not end user:**

In `MetricOmmPool.swap()` (lines 230–240), the pool calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
_beforeSwap(
    msg.sender,   // direct caller of pool.swap() — the router, not the user
    recipient,
    ...
);
```

**Extension checks that argument against its allowlist:**

`SwapAllowlistExtension.beforeSwap` (lines 37–39) uses `msg.sender` (the pool) as the mapping key and `sender` (the value passed by the pool) as the swapper:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

**Router is the direct caller of `pool.swap()`:**

`MetricOmmSimpleRouter.exactInputSingle` (lines 72–80) calls `pool.swap()` directly with no forwarding of the original `msg.sender`:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The same pattern holds in `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput`/`_exactOutputIterateCallback` (line 220). In every case, the pool's `msg.sender` is the router.

**Result:** The extension evaluates `allowedSwapper[pool][router]`, never `allowedSwapper[pool][user]`.

**Bypass path:** A pool admin who allowlists the router (a natural action for a "trusted intermediary") inadvertently opens the pool to every address on the internet. Any EOA calls `router.exactInputSingle(...)` → pool sees `msg.sender = router` → `allowedSwapper[pool][router] == true` → swap executes. The curation invariant collapses entirely.

**Lockout path:** A pool admin who allowlists individual users (the intended use case) and does not allowlist the router will silently lock those users out of the router. Any allowlisted user calls `router.exactInputSingle(...)` → pool sees `msg.sender = router` → `allowedSwapper[pool][router] == false` → `NotAllowedToSwap` revert. The primary swap interface is unusable for the pool's intended participants.

**Existing guards are insufficient:** There is no mechanism in the pool or router to forward the original `msg.sender` to the extension. The extension has no fallback logic to inspect `extensionData` for a verified caller identity.

## Impact Explanation

- **Bypass path:** Unauthorized counterparties trade against LP positions on curated pools. This is a direct loss of LP principal because the pool's curation invariant — which determines who may trade against LP-provided liquidity — is completely bypassed. Severity: Critical/High.
- **Lockout path:** Individually allowlisted users cannot access the router (the canonical, documented swap interface). LP assets are inaccessible to legitimate swappers, breaking core swap functionality. Severity: Medium/High.

Both outcomes are fund-impacting and match the allowed impact gate: broken core pool functionality causing loss of funds or unusable swap flows, and admin-boundary break where the allowlist check fails to enforce the intended access control.

## Likelihood Explanation

- The router is the canonical, documented swap interface for end users; virtually all user-facing swaps go through it.
- No special setup is required beyond the normal use of `SwapAllowlistExtension` (allowlisting individual users or the router).
- The pool admin has no on-chain signal that the allowlist is evaluated against the router address rather than the user.
- The bypass path requires only that the admin allowlists the router — a natural and expected configuration for a pool that wants to allow router-based swaps.
- The lockout path requires only that the admin allowlists individual users and does not allowlist the router — the primary intended use case of the extension.
- The unit tests for `SwapAllowlistExtension` call the extension directly with `vm.prank(address(pool))` and pass the swapper address explicitly, never exercising the router path, so the mismatch is untested and undetected.

## Recommendation

The pool must pass the economically relevant actor — the end user — as `sender` to extension hooks, not the immediate `msg.sender`. Concrete options:

1. **Trusted-forwarder pattern:** Allow the caller to supply a `sender` override in `extensionData`; the pool validates it (e.g., via EIP-712 permit or a registry of trusted routers) and passes the verified user address to extensions.
2. **Router forwards original sender:** The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it against a registry of trusted routers before using it as the swapper identity.
3. **Documentation minimum:** At minimum, add explicit NatSpec to `SwapAllowlistExtension` and the router warning that the allowlist gates the direct `pool.swap()` caller, not the end user, so pool admins can make an informed configuration decision.

## Proof of Concept

**Bypass path:**
1. Deploy a pool with `SwapAllowlistExtension` in the `beforeSwap` slot.
2. Pool admin calls `extension.setAllowedToSwap(pool, router, true)` — allowlists the router as a trusted intermediary.
3. Non-allowlisted EOA `attacker` calls `router.exactInputSingle({pool: pool, recipient: attacker, ...})`.
4. Pool calls `_beforeSwap(msg.sender=router, ...)`. Extension checks `allowedSwapper[pool][router] == true` → passes.
5. Swap executes. `attacker` receives tokens from a pool closed to them.

**Lockout path:**
1. Same setup, but pool admin calls `extension.setAllowedToSwap(pool, alice, true)` — does not allowlist the router.
2. `alice` calls `router.exactInputSingle({pool: pool, ...})`.
3. Pool calls `_beforeSwap(msg.sender=router, ...)`. Extension checks `allowedSwapper[pool][router] == false` → `NotAllowedToSwap` revert.
4. `alice` is locked out of the router despite being individually allowlisted; she must call `pool.swap()` directly (no slippage protection, no multi-hop) or cannot swap at all.
