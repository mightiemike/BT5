### Title
`submit_blocks` Is Effectively Uncallable by `Role::UnrestrictedSubmitBlocks` When Paused Due to Stricter Inner Guard on `run_mainchain_gc` - (File: `contract/src/lib.rs`)

---

### Summary

`submit_blocks` is decorated with `#[pause]`, which is intended to be bypassable by accounts holding `Role::UnrestrictedSubmitBlocks`. However, `submit_blocks` internally calls `self.run_mainchain_gc(...)`, which carries its own `#[pause(except(roles(Role::UnrestrictedRunGC)))]` guard. Because the `near-plugins` `#[pause]` macro injects its check directly into the function body (not only at the ABI boundary), the inner guard fires against the original `predecessor_account_id` even on internal calls. A caller with only `Role::UnrestrictedSubmitBlocks` passes the outer guard but is rejected by the inner one, making `submit_blocks` completely non-functional when the contract is paused — exactly the scenario the role was designed to handle.

---

### Finding Description

`submit_blocks` is the sole write path for relayers to advance the on-chain Bitcoin header chain. It is annotated with two guards:

```
#[pause]           // bypassable by Role::UnrestrictedSubmitBlocks
#[trusted_relayer] // bypassable by Role::DAO, Role::UnrestrictedSubmitBlocks
pub fn submit_blocks(&mut self, ...) -> PromiseOrValue<()> {
    ...
    self.run_mainchain_gc(num_of_headers);  // line 181
    ...
}
``` [1](#0-0) 

`run_mainchain_gc` is a separate public method with its own, narrower pause exception:

```
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }
``` [2](#0-1) 

The `near-plugins` `#[pause]` macro expands to a guard block injected at the top of the function body. It calls `self.pa_is_paused(feature)` and then checks `self.acl_has_role(predecessor, bypass_role)` against `env::predecessor_account_id()`. This check runs unconditionally whenever the function is entered — whether via an external NEAR transaction or an internal Rust call (`self.run_mainchain_gc(...)`). The `predecessor_account_id` is the original external caller in both cases.

The two roles are disjoint by design:

| Role | Defined purpose |
|---|---|
| `UnrestrictedSubmitBlocks` | "Allows to use contract API even after contract is paused" — bypasses `#[pause]` on `submit_blocks` |
| `UnrestrictedRunGC` | "Allows to use `run_mainchain_gc` API on a paused contract" | [3](#0-2) 

A relayer granted only `Role::UnrestrictedSubmitBlocks` will:
1. Pass the `#[pause]` check on `submit_blocks` ✓
2. Pass the `#[trusted_relayer]` check ✓
3. Process all submitted headers ✓
4. **Panic** inside `run_mainchain_gc` because the injected pause guard finds the caller lacks `Role::UnrestrictedRunGC` ✗

The call to `self.run_mainchain_gc(num_of_headers)` at line 181 is unconditional — it executes on every `submit_blocks` invocation regardless of whether GC is actually needed. [4](#0-3) 

---

### Impact Explanation

When the contract is paused, `submit_blocks` is completely non-functional for any account holding only `Role::UnrestrictedSubmitBlocks`. The entire purpose of that role — to allow a designated relayer to keep the Bitcoin header chain advancing during a pause — is nullified. The canonical chain tip (`mainchain_tip_blockhash`) stops advancing, and all downstream consumers of `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` will be unable to confirm new Bitcoin transactions. The only accounts that can successfully call `submit_blocks` while paused are those holding `Role::DAO` or both `Role::UnrestrictedSubmitBlocks` AND `Role::UnrestrictedRunGC` simultaneously — a combination that is not documented or enforced by the role model.

---

### Likelihood Explanation

The contract explicitly provides `Role::UnrestrictedSubmitBlocks` as the designated bypass for paused block submission. Operators will naturally grant this role to relayer accounts without also granting `Role::UnrestrictedRunGC`, since the documentation and role names give no indication that both are required. The bug is triggered the first time a relayer with only `Role::UnrestrictedSubmitBlocks` calls `submit_blocks` while the contract is paused — a realistic operational scenario (e.g., emergency pause during a security incident while keeping the relay alive).

---

### Recommendation

Extract the GC logic into a private helper function that carries no pause guard, and call that helper from both `submit_blocks` and `run_mainchain_gc`:

```rust
// New private helper — no pause guard
fn run_mainchain_gc_inner(&mut self, batch_size: u64) { /* existing GC logic */ }

// Public entry point keeps its own pause guard
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    self.run_mainchain_gc_inner(batch_size);
}

// submit_blocks calls the private helper directly
pub fn submit_blocks(&mut self, ...) {
    ...
    self.run_mainchain_gc_inner(num_of_headers); // no inner pause check
    ...
}
```

Alternatively, add `Role::UnrestrictedSubmitBlocks` to the except-roles list of `run_mainchain_gc`:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC, Role::UnrestrictedSubmitBlocks)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }
```

---

### Proof of Concept

1. Deploy the contract and pause it via a `PauseManager` account.
2. Grant an account `Role::UnrestrictedSubmitBlocks` only (not `Role::UnrestrictedRunGC`).
3. Have that account call `submit_blocks` with a valid batch of headers and sufficient deposit.
4. The call passes the `#[pause]` and `#[trusted_relayer]` guards on `submit_blocks`, processes all headers, then panics inside `run_mainchain_gc` with the near-plugins pause rejection message.
5. The transaction is rolled back: no headers are stored, the canonical tip is unchanged, and the relayer's deposit is consumed by gas. [5](#0-4) [2](#0-1)
