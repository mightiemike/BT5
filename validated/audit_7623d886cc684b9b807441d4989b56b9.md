### Title
No Minimum Liquidation Profit Allows Tiny Positions to Accumulate Bad Debt — (`core/contracts/ClearinghouseStorage.sol`, `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

Nado's liquidation incentive scales linearly with position size and is subject to truncating integer division. For sufficiently small positions, the liquidator's profit rounds to zero and does not cover on-chain gas costs. Because no minimum position size is enforced in the liquidation path, tiny undercollateralized positions can accumulate indefinitely, eventually creating bad debt that drains the insurance fund or is socialized across all participants.

---

### Finding Description

In Nado, a liquidator's economic incentive is the **price discount** they receive when taking over a liquidated position. The discount is computed in `getLiqPriceX18` inside `ClearinghouseStorage.sol`:

```solidity
int128 penaltyX18 = (RiskHelper._getWeightX18(
    risk, amount, IProductEngine.HealthType.MAINTENANCE
) - ONE) / 5;
if (penaltyX18.abs() < MIN_NON_SPREAD_LIQ_PENALTY_X18) {
    penaltyX18 = -MIN_NON_SPREAD_LIQ_PENALTY_X18; // floor at 0.5%
}
return (risk.priceX18.mul(ONE + penaltyX18), risk.priceX18);
``` [1](#0-0) 

The liquidation fee that goes to the insurance fund (and the liquidator's net profit) is then computed in `_handleLiquidationPayment`:

```solidity
v.liquidationFees = (v.oraclePriceX18 - v.liquidationPriceX18)
    .mul(LIQUIDATION_FEE_FRACTION)
    .mul(txn.amount);
``` [2](#0-1) 

`LIQUIDATION_FEE_FRACTION` is 50%, meaning the liquidator retains the other 50% of the discount as profit. Both `.mul()` calls use `MathSD21x18.mul`, which performs **truncating integer division**:

```solidity
function mul(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        int256 result = (int256(x) * y) / ONE_X18;
        ...
    }
}
``` [3](#0-2) 

For a position with `amount = 1` (1 wei), oracle price `1e18`, and minimum penalty `5e15` (0.5%):

```
liquidatorProfit = 5e15 * 5e17 * 1 / 1e18 / 1e18 = 2.5e33 / 1e36 = 0
```

The profit truncates to **zero**. Even for non-trivial but small positions (e.g., `amount = 1e9`, the minimum `sizeIncrement`), the absolute profit is on the order of `1e7` (i.e., `$0.00000001`), far below any realistic gas cost.

Critically, `_assertLiquidationAmount` enforces that the liquidation amount is a multiple of `sizeIncrement`, but **does not enforce a minimum absolute position size**:

```solidity
require(txn.amount % perpSizeIncrement == 0, ERR_INVALID_LIQUIDATION_AMOUNT);
maxLiquidatable = perpAmount;
``` [4](#0-3) 

While `minSize` is stored in `OffchainExchange.marketInfo` and accessible via `getMinSize()`, it is **not consulted anywhere in the liquidation path** (`_assertLiquidationAmount`, `_handleLiquidationPayment`, or `liquidateSubaccountImpl`). [5](#0-4) 

Furthermore, even if `minSize` is enforced during order matching, partial liquidations can reduce a position to below `minSize` — the remaining stub is then unliquidatable in practice.

---

### Impact Explanation

Tiny undercollateralized positions yield zero or sub-gas-cost profit for liquidators. These positions will not be liquidated by rational external actors. As losses accumulate:

1. The **insurance fund** (`insurance` storage variable) is drawn down to cover bad debt during `_finalizeSubaccount`.
2. If the insurance fund is exhausted, `socializeSubaccount` is called, which reduces the cumulative funding index for all profitable positions — effectively **stealing from all winning traders** to cover the insolvent loser. [6](#0-5) 

The constants confirm there is no minimum absolute liquidation reward enforced:

```solidity
int128 constant MIN_NON_SPREAD_LIQ_PENALTY_X18 = ONE / 200; // 0.5% (percentage, not absolute)
int128 constant MIN_SPREAD_LIQ_PENALTY_X18 = ONE / 400;     // 0.25%
``` [7](#0-6) 

These are **percentage floors**, not absolute dollar floors. A 0.5% discount on a $0.01 position is still $0.00005 — far below gas cost.

---

### Likelihood Explanation

This can occur in two ways:

1. **Organically**: Traders on DEXes routinely open high-leverage positions with small account values. A $20 position at 10x leverage with a 0.5% liquidation discount yields only $0.05 for the liquidator — marginal even on L2s.

2. **Deliberately**: An attacker can open thousands of positions at `minSize` across different subaccounts. Each position costs only the margin requirement plus gas. The attacker accepts the loss on each position, knowing they will not be liquidated. The aggregate bad debt can be substantial.

Nado is deployed on Ink (an EVM L2), where gas costs are low but not zero. The attack threshold is lower than on mainnet, but the vulnerability remains exploitable.

---

### Recommendation

1. **Enforce a minimum absolute liquidation profit**: In `_assertLiquidationAmount`, require that `penaltyX18.mul(txn.amount.abs()) >= MIN_LIQUIDATION_PROFIT` where `MIN_LIQUIDATION_PROFIT` is a configurable dollar-denominated floor (e.g., `$1`).

2. **Enforce `minSize` in the liquidation path**: In `_assertLiquidationAmount`, check that `txn.amount.abs() >= getMinSize(productId)` to prevent liquidation of sub-minimum stubs.

3. **Round up the liquidation fee**: Use ceiling division when computing `v.liquidationFees` so that even tiny positions yield a non-zero fee, ensuring the insurance fund is not silently depleted.

---

### Proof of Concept

**Setup**: A perp product with `sizeIncrement = 1e9`, `minSize = 1e9`, oracle price `$1.00`, maintenance weight `0.9` (10% margin requirement).

**Attack**:
1. Attacker opens 10,000 positions across 10,000 subaccounts, each with `amount = 1e9` (minimum size) and just enough margin to pass initial health.
2. Price moves adversely; all positions become undercollateralized (maintenance health < 0).
3. Liquidation profit per position: `(1e18 - 9e17) * 5e17 * 1e9 / 1e18 / 1e18 = 5e7` ≈ `$0.00000005`. Gas cost on Ink: ~`$0.01`.
4. No rational liquidator submits liquidation transactions.
5. All 10,000 positions accumulate bad debt. Total bad debt = `10,000 * $1 * 10% = $1,000`.
6. Insurance fund is drawn down by `$1,000`; if exhausted, losses are socialized.

The attacker's cost is only the margin posted (which they lose) plus gas for 10,000 open transactions. The protocol absorbs the bad debt. [1](#0-0) [8](#0-7) [9](#0-8)
