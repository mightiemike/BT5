### Title
Non-injective `ValidResourceBounds` protobuf/RPC deserialization collapses `AllResources{l2=0,l1_data=0}` to `L1Gas`, producing a divergent transaction hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf and RPC deserializers for `ValidResourceBounds` apply a non-injective mapping: any `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is silently collapsed to `L1Gas(X)`. Because `get_tip_resource_bounds_hash` produces structurally different hashes for these two variants (the `AllResources` path appends an extra non-zero `L1_DATA_GAS` felt to the poseidon preimage), a V3 transaction submitted with `AllResources` and zero l2/l1_data bounds will carry hash H1 in the sequencer's DB, but any node that deserializes it over P2P or RPC will recompute hash H2 ≠ H1. This breaks `validate_transaction_hash` on syncing peers and causes the RPC to serve an authoritative-looking wrong hash.

### Finding Description

**Step 1 — Hash domain split on `ValidResourceBounds` variant.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` builds the poseidon preimage differently depending on the enum variant:

```
L1Gas(X)        → poseidon([tip, l1_gas_felt, l2_gas_felt])           // 3 elements
AllResources{X,0,0} → poseidon([tip, l1_gas_felt, l2_gas_felt, l1_data_gas_felt]) // 4 elements
```

The extra element `l1_data_gas_felt` for zero bounds is **not zero**: `get_concat_resource` encodes the resource name `L1_DATA` (7 bytes = `0x4C315F44415441`) into the high bits of the felt, producing `0x004C315F44415441_0000000000000000_00000000000000000000000000000000` — a non-zero value. The two hashes are therefore provably distinct. [1](#0-0) [2](#0-1) 

**Step 2 — Non-injective protobuf deserialization.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` maps both `L1Gas(X)` and `AllResources{X,0,0}` to the same output:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← collapses AllResources{X,0,0} here
} else {
    ValidResourceBounds::AllResources(...)
})
``` [3](#0-2) 

The same collapse exists in the RPC layer:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)   // ← same collapse
        } else { ... }
    }
}
``` [4](#0-3) 

**Step 3 — Gateway always stores V3 transactions as `AllResources`.**

`InternalRpcInvokeTransactionV3` and `InternalRpcDeclareTransactionV3` carry `AllResourceBounds` (not `ValidResourceBounds`). Their `From` impls into `InvokeTransactionV3` / `DeclareTransactionV3` always wrap with `ValidResourceBounds::AllResources(...)`:

```rust
impl From<InternalRpcInvokeTransactionV3> for InvokeTransactionV3 {
    fn from(tx: ...) -> Self {
        Self { resource_bounds: ValidResourceBounds::AllResources(tx.resource_bounds), ... }
    }
}
``` [5](#0-4) [6](#0-5) 

So any V3 transaction submitted via the gateway with `l2_gas = 0` and `l1_data_gas = 0` is stored as `AllResources{X,0,0}` and its hash H1 is computed over 4 poseidon elements.

**Step 4 — P2P sync round-trip breaks hash.**

The P2P server serializes `FullTransaction` (which contains `Transaction::Invoke(InvokeTransaction::V3{resource_bounds: AllResources{X,0,0}})`) to protobuf via `From<ValidResourceBounds> for protobuf::ResourceBounds`, which correctly emits explicit zero fields for l2_gas and l1_data_gas. [7](#0-6) 

The receiving peer deserializes via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` and gets `L1Gas(X)`. When `validate_transaction_hash` is called on the deserialized `Transaction`, it recomputes H2 (3 poseidon elements) ≠ H1 (4 poseidon elements), causing the block/transaction to be rejected. [8](#0-7) 

### Impact Explanation

- **P2P sync rejects valid blocks**: Any block containing a V3 transaction with `AllResources{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` will fail `validate_transaction_hash` on syncing peers because the deserialized variant is `L1Gas(X)` and produces a different hash. This is a liveness failure for syncing nodes.
- **RPC returns authoritative-looking wrong hash**: The RPC `From<ResourceBoundsMapping> for ValidResourceBounds` applies the same collapse. A client that queries such a transaction via `starknet_getTransactionByHash` and recomputes the hash from the returned `resource_bounds` will get H2 ≠ H1 — the stored hash — making the RPC response internally inconsistent.

Both match the allowed impact scope: *"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"* and *"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

### Likelihood Explanation

Any V3 transaction submitted via the gateway with `l2_gas = 0` and `l1_data_gas = 0` (but non-zero `l1_gas`) triggers this. This is a valid, accepted transaction — the gateway's `ZeroResourceBounds` check only requires at least one resource bound to be non-zero. Pre-0.13.3 style transactions (L1-gas-only) submitted as V3 `AllResources` with zero l2/l1_data bounds are a natural real-world case. The trigger is unprivileged and requires no special access.

### Recommendation

The protobuf and RPC deserializers must not infer the `ValidResourceBounds` variant from the numeric values of the bounds. Instead, the variant must be preserved explicitly. Options:

1. **Add a discriminant field to the protobuf schema** (e.g., a boolean `is_all_resources`) so the deserializer can reconstruct the correct variant without inspecting the values.
2. **Always deserialize as `AllResources`** when all three fields are present in the protobuf message (since the serializer always emits all three fields), and only fall back to `L1Gas` when the l2_gas field is absent (the pre-0.13.3 wire format).
3. **Remove the `L1Gas` variant entirely** from the internal representation and always use `AllResources`, treating `L1Gas` as `AllResources{l2_gas: 0, l1_data_gas: 0}` with a separate flag for hash computation mode.

The same fix must be applied to `From<ResourceBoundsMapping> for ValidResourceBounds` in `crates/apollo_rpc/src/v0_8/transaction.rs`.

### Proof of Concept

```
1. Submit a V3 invoke transaction to the gateway with:
   resource_bounds = AllResourceBounds {
       l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
   }

2. Gateway stores it as Transaction::Invoke(InvokeTransaction::V3 {
       resource_bounds: ValidResourceBounds::AllResources(...),
       ...
   }) with hash H1 = poseidon([INVOKE, version, sender, poseidon([tip, l1_felt, l2_felt, l1_data_felt]), ...])
   where l1_data_felt = 0x004C315F44415441_0000000000000000_00000000000000000000000000000000 ≠ 0

3. P2P server serializes to protobuf: l1_gas=X, l2_gas=0, l1_data_gas=0 (all fields present).

4. Syncing peer deserializes:
   TryFrom<protobuf::ResourceBounds> for ValidResourceBounds
   → l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → ValidResourceBounds::L1Gas(X)   ← WRONG VARIANT

5. validate_transaction_hash recomputes:
   H2 = poseidon([INVOKE, version, sender, poseidon([tip, l1_felt, l2_felt]), ...])
   (l1_data_felt is absent)

6. H1 ≠ H2 → validate_transaction_hash returns false → block rejected.
```

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

fn get_concat_resource(
    resource_bounds: &ResourceBounds,
    resource_name: &ResourceName,
) -> Result<Felt, StarknetApiError> {
    let max_amount = resource_bounds.max_amount.0.to_be_bytes();
    let max_price = resource_bounds.max_price_per_unit.0.to_be_bytes();
    let concat_bytes =
        [[0_u8].as_slice(), resource_name.as_slice(), max_amount.as_slice(), max_price.as_slice()]
            .concat();
    Ok(Felt::from_bytes_be(&concat_bytes.try_into().expect("Expect 32 bytes")))
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
