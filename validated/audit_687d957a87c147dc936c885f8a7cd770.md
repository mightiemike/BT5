Audit Report

## Title
Router-Mediated Swaps Pass Router Address as `sender` to `SwapAllowlistExtension::beforeSwap`, Breaking Per-User Allowlist Gating — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension::beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the pool's own `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. The hook therefore checks the router's allowlist status rather than the user's, making per-user allowlisting incoherent for any router-mediated swap. This produces two mutually exclusive failure modes: allowlisted users cannot swap via the router, or allowlisting the router to restore functionality opens the pool to all users.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the hook.**

`MetricOmmPool::swap` calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

`ExtensionCalling::_beforeSwap` forwards that value unchanged as the first argument to every configured extension: [2](#0-1) 

**Step 2 — The hook checks `allowedSwapper[pool][sender]`.**

`SwapAllowlistExtension::beforeSwap` uses `msg.sender` (the pool) as the pool key and the passed `sender` argument as the swapper identity: [3](#0-2) 

**Step 3 — The router is the direct caller of `pool.swap()`.**

`MetricOmmSimpleRouter::exactInputSingle` calls `pool.swap(...)` directly. At that call site, `msg.sender` inside the pool is the router, not the end user: [4](#0-3) 

The original `msg.sender` is stored only in transient callback context for payment purposes via `_setNextCallbackContext` and is never forwarded to the pool or the hook: [5](#0-4) 

The same pattern applies to `exactInput` (line 103–112), `exactOutputSingle` (line 135–137), and `exactOutput` (line 165–181), all of which call `pool.swap()` with the router as `msg.sender`. [6](#0-5) 

**Resulting identity mismatch:**

| Path | `sender` seen by hook | Allowlist check |
|---|---|---|
| Direct `pool.swap()` | user EOA | `allowedSwapper[pool][user]` ✓ |
| Via `exactInputSingle` | router contract | `allowedSwapper[pool][router]` ✗ |
| Via `exactInput` (hop > 0) | router contract | `allowedSwapper[pool][router]` ✗ |
| Via `exactOutput` (recursive) | pool (prior hop) | `allowedSwapper[pool][pool]` ✗ |

## Impact Explanation

Two mutually exclusive failure modes exist for any pool that configures `SwapAllowlistExtension`:

1. **Allowlisted users cannot use the router.** If the pool admin allowlists specific user addresses, every router-mediated swap by those users reverts with `NotAllowedToSwap` because the hook sees the router's address. The router — the primary public swap interface — is effectively unusable for allowlisted pools.

2. **Allowlisting the router opens the pool to all users.** If the pool admin allowlists the router address to restore router functionality, the allowlist is entirely bypassed: any unprivileged user can call `exactInputSingle` and swap in a pool intended to be restricted. The per-user gating the extension was designed to enforce is nullified.

There is no configuration that simultaneously allows router-mediated swaps and enforces per-user identity. This constitutes broken core pool functionality — the `SwapAllowlistExtension` is a first-party, production extension whose primary purpose (per-user swap gating) is rendered inoperable for the protocol's primary swap interface.

## Likelihood Explanation

The `SwapAllowlistExtension` is a first-party production extension in `metric-periphery`. The router is the primary public swap interface. Any pool that deploys with this extension and expects users to interact via the router will immediately encounter one of the two failure modes above. No special attacker capability is required — a normal user calling `exactInputSingle` triggers the mismatch. The condition is deterministic and repeatable.

## Recommendation

The pool must propagate the original initiator through the hook call chain, or the extension must accept an authenticated initiator via `extensionData`. Two concrete options:

1. **Pass the original initiator as a dedicated hook argument.** Add an `initiator` field alongside `sender` in the `beforeSwap` signature. For direct calls `initiator == sender`; for router calls the router encodes the original `msg.sender` into `extensionData` and the pool/hook reads it. This requires a protocol-level convention.

2. **Extension-level router awareness.** `SwapAllowlistExtension::beforeSwap` checks: if `sender` is a known trusted router, decode the real initiator from `extensionData` and check that address instead. This is simpler but requires the extension to maintain a trusted-router registry.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is allowlisted
  allowedSwapper[pool][router] = false // router is not

Attack path (broken functionality):
  alice calls router.exactInputSingle({pool: pool, ...})
  → router calls pool.swap(recipient, ...)   // msg.sender = router
  → pool calls _beforeSwap(msg.sender=router, ...)
  → SwapAllowlistExtension.beforeSwap(sender=router, ...)
  → allowedSwapper[pool][router] == false → revert NotAllowedToSwap

Alice is allowlisted but cannot use the router.

Bypass path (if admin allowlists router to fix above):
  allowedSwapper[pool][router] = true

  mallory (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  → router calls pool.swap(...)   // msg.sender = router
  → allowedSwapper[pool][router] == true → swap succeeds

Mallory bypasses the per-user allowlist entirely.
```
