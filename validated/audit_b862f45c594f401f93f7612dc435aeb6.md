### Title
`verify_transaction_inclusion_v2` Architecturally Depends on the Deprecated `verify_transaction_inclusion`, Guaranteeing Future Proof-Verification Breakage - (File: contract/src/lib.rs)

---

### Summary

`verify_transaction_inclusion_v2` is the contract's officially recommended, "safe" SPV proof endpoint. However, its implementation delegates its core proof-result computation entirely to `verify_transaction_inclusion`, the function it is supposed to replace. Because `verify_transaction_inclusion` carries a formal `#[deprecated]` annotation directing all callers to use v2, any future upgrade that removes the deprecated function will silently break v2 as well — the exact same "system stops working after a planned protocol change" pattern as M-03.

---

### Finding Description

`verify_transaction_inclusion` is marked deprecated since v0.5.0: [1](#0-0) 

The deprecation notice explicitly instructs all callers to migrate to `verify_transaction_inclusion_v2`. However, `verify_transaction_inclusion_v2` does not re-implement the proof logic — it validates only the coinbase sub-proof and then unconditionally delegates the full proof result to the deprecated function: [2](#0-1) 

The `#[allow(deprecated)]` attribute suppresses the compiler warning, masking the dependency entirely. The result is a circular architectural trap:

- Callers of v1 are told "use v2 instead."
- v2 internally calls v1 for every proof result it returns.
- The actual proof computation — Merkle root recomputation and comparison against `header.block_header.merkle_root` — lives exclusively in v1: [3](#0-2) 

If v1 is removed in any future upgrade (the natural and stated lifecycle of a deprecated function), v2 loses its entire proof-computation body and will panic or produce no result. Every downstream consumer that migrated to v2 on the strength of the deprecation notice will break simultaneously.

---

### Impact Explanation

The broken invariant is the **proof result** returned by `verify_transaction_inclusion_v2`. Any NEAR contract or off-chain client that calls v2 to gate a cross-chain action (e.g., releasing funds, minting wrapped assets, confirming a Bitcoin deposit) will receive a panic instead of a boolean after v1 is removed. The canonical proof-verification path — the only on-chain mechanism for confirming Bitcoin transaction inclusion — becomes permanently non-functional without a new contract deployment. This matches the M-03 impact class: a core system function stops working due to a planned removal of a mechanism it secretly depends on.

---

### Likelihood Explanation

The deprecation annotation is a formal, versioned, in-code commitment (`since = "0.5.0"`) that v1 will be removed. The `#[allow(deprecated)]` suppressor in v2 means the dependency is invisible to routine code review and CI lint passes. Any developer executing the natural deprecation lifecycle — removing v1 in a future version — will not see a compile error or warning pointing to v2's dependency, and will ship a broken v2 without realizing it. The likelihood of this occurring during a routine upgrade is high precisely because the tooling that would normally catch it has been silenced.

---

### Recommendation

Inline the proof computation directly into `verify_transaction_inclusion_v2` rather than delegating to v1. Specifically, move the Merkle root recomputation and confirmation-count checks from `verify_transaction_inclusion` into v2's body. Once v2 is self-contained, v1 can be safely removed without any risk of breaking v2. The `#[allow(deprecated)]` suppressor in v2 should be treated as a red flag and removed as part of this refactor.

---

### Proof of Concept

1. A downstream contract calls `verify_transaction_inclusion_v2` with a valid `ProofArgsV2`.
2. v2 validates the coinbase sub-proof (lines 358–365) and then calls `self.verify_transaction_inclusion(args.into())` (line 368).
3. A future upgrade removes `verify_transaction_inclusion` as part of the deprecation lifecycle.
4. After the upgrade, every call to `verify_transaction_inclusion_v2` panics at line 368 because the callee no longer exists in the deployed WASM.
5. All SPV proof verification for the light client is permanently broken until a new contract version is deployed and migrated. [4](#0-3) [5](#0-4)
