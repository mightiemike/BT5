### Title
No Slippage Protection on NLP Mint and Burn — Oracle Price Accepted Without User-Defined Output Bounds - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`mintNlp` and `burnNlp` in `Clearinghouse.sol` compute output amounts dynamically from a sequencer-supplied `oraclePriceX18`. The user's signed transaction commits only to an input amount (`quoteAmount` for mint, `nlpAmount` for burn) with no field to enforce a minimum output. If the oracle price shifts between when the user signs and when the sequencer executes, the user receives fewer NLP tokens or less quote than expected, with no on-chain mechanism to revert the transaction.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, when a `MintNlp` transaction is processed, the sequencer supplies `signedTx.oraclePriceX18` alongside the user's signed `MintNlp` struct: [1](#0-0) 

This oracle price is then passed directly into `Clearinghouse.mintNlp`, where the NLP output is computed as:

```
nlpAmount = quoteAmount.div(oraclePriceX18)
``` [2](#0-1) 

The user's signed `MintNlp` struct contains only `sender`, `nonce`, and `quoteAmount`. There is no `minNlpOut` field. The user commits to spending `quoteAmount` of quote but has no on-chain guarantee about how many NLP tokens they will receive.

The same pattern applies to `burnNlp`: the user signs a `BurnNlp` specifying `nlpAmount` to burn, but the quote returned is computed from the sequencer-supplied oracle price with no `minQuoteOut` guard: [3](#0-2) 

---

### Impact Explanation

A user who signs a `MintNlp` transaction expecting to receive `N` NLP tokens at price `P` may instead receive significantly fewer tokens if the oracle price has risen by the time the sequencer executes the batch. The quote is fully deducted from the user's subaccount regardless. For `burnNlp`, a user burning `Y` NLP tokens may receive substantially less quote if the oracle price has fallen. In both cases, the user's subaccount balance is permanently reduced with no recourse.

The corrupted state delta is: `spotEngine` balance for `NLP_PRODUCT_ID` (mint) or `QUOTE_PRODUCT_ID` (burn) is updated at an oracle price the user never agreed to. [4](#0-3) 

---

### Likelihood Explanation

The Nado sequencer batches and submits transactions off-chain. There is an inherent delay between when a user signs a `MintNlp` or `BurnNlp` transaction and when the sequencer includes it in a batch. During high volatility, the NLP oracle price can move materially within this window. No malicious actor is required — an honest sequencer using the current oracle price at execution time is sufficient to trigger the loss. This is a realistic scenario for any active user of the NLP system.

---

### Recommendation

- **Short term:** Add `minNlpOut` to the `MintNlp` struct and `minQuoteOut` to the `BurnNlp` struct. In `Clearinghouse.mintNlp`, assert `nlpAmount >= txn.minNlpOut` after computing the output. In `burnNlp`, assert the computed quote return meets `txn.minQuoteOut` before applying balance updates.
- **Long term:** Apply the same pattern to any future operation where an output amount is dynamically computed from a sequencer-provided price or oracle value, ensuring users can always bound their worst-case execution.

---

### Proof of Concept

1. Alice signs a `MintNlp` transaction with `quoteAmount = 1000e18` when the NLP oracle price is `10e18` (expecting ~100 NLP tokens).
2. The transaction enters the sequencer's queue.
3. Before the sequencer submits the batch, the NLP oracle price rises to `20e18`.
4. The sequencer submits the batch with `oraclePriceX18 = 20e18`.
5. `Clearinghouse.mintNlp` computes `nlpAmount = 1000e18 / 20e18 = 50` NLP tokens.
6. Alice's quote balance is debited by `1000e18` but she receives only 50 NLP instead of the ~100 she expected.
7. No revert occurs; Alice has no `minNlpOut` field in her signed transaction to trigger one. [5](#0-4) [1](#0-0)

    function mintNlp(
        IEndpoint.MintNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.quoteAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 quoteAmount = int128(txn.quoteAmount);
        int128 nlpAmount = quoteAmount.div(oraclePriceX18);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] >= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, -nlpAmount);

        spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -quoteAmount);
        _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);

        require(
            getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
