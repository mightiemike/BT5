### Title
Difficulty Retarget Uses Mainchain Ancestor Instead of Fork-Chain Ancestor, Enabling Incorrect PoW Acceptance on Fork Blocks - (File: `contract/src/dogecoin.rs`)

---

### Summary

During Dogecoin difficulty adjustment, `get_next_work_required` fetches the retarget boundary block via `get_header_by_height`, which always resolves to the **mainchain** block at that height. When the block being validated is on a **fork** that diverged before the retarget boundary, the mainchain block at that height is a different block from the fork's actual ancestor. This causes the difficulty calculation to use the wrong `first_block_time`, producing an incorrect expected `bits` value — the exact same bug class as M-04 (using a global/current state value instead of the correct context-specific value).

---

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes the expected difficulty for a new block. When a retarget is due, it walks back `blocks_to_go_back` blocks from `prev_block_header` to find the boundary block's timestamp: [1](#0-0) 

```rust
let height_first = prev_block_header
    .block_height
    .checked_sub(blocks_to_go_back)
    ...;

// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

The `get_header_by_height` implementation resolves the height exclusively through `mainchain_height_to_header`: [2](#0-1) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

When a fork block is submitted whose chain diverged **before** `height_first`, the mainchain block at `height_first` is a completely different block from the fork's actual ancestor at that height. The `first_block_time` used in `calculate_next_work_required` is therefore wrong, producing an incorrect expected `bits` value.

The code itself acknowledges this with the TODO comment at line 291, confirming the developer was uncertain whether this is correct.

---

### Impact Explanation

An attacker can craft a sequence of fork headers where the fork diverges before a retarget boundary. Because `first_block_time` is taken from the mainchain block (not the fork's ancestor), the computed expected `bits` differs from what the fork's actual difficulty should be. The `check_pow` call: [3](#0-2) 

...validates `block_header.bits == expected_bits` using this corrupted expected value. A crafted fork block with an artificially low difficulty (easy-to-mine `bits`) can pass validation if the wrong `first_block_time` happens to produce a matching expected `bits`. This corrupts the fork-choice invariant: the contract may accept and store fork blocks with insufficient PoW, and if their accumulated `chain_work` exceeds the mainchain, a reorg is triggered — promoting a low-work fork to the canonical chain.

---

### Likelihood Explanation

The entry path is fully unprivileged: any NEAR account can call `submit_blocks()`. The attacker needs to:
1. Submit a valid fork chain that diverges before a retarget boundary (every 60 blocks in Dogecoin post-145000 with `difficulty_adjustment_interval = 1`, meaning every block is a retarget).
2. Craft the fork's `bits` field to match the corrupted expected value derived from the mainchain's `first_block_time`.

The mainchain's `first_block_time` is publicly readable on-chain, so the attacker can compute the exact `bits` value that will pass. The likelihood is medium-high for a motivated attacker on Dogecoin testnet (where `pow_allow_min_difficulty_blocks` is true, making the min-difficulty path also reachable).

---

### Recommendation

Replace `get_header_by_height` with an ancestor walk that follows `prev_block_hash` links from `prev_block_header` back `blocks_to_go_back` steps, using `get_prev_header` in a loop. This ensures the retarget boundary block is the fork chain's actual ancestor, not the mainchain block at that height:

```rust
let mut ancestor = prev_block_header.clone();
for _ in 0..blocks_to_go_back {
    ancestor = blocks_getter.get_prev_header(&ancestor.block_header);
}
let first_block_time = ancestor.block_header.time;
```

This mirrors how `get_prev_header` is already used correctly in the min-difficulty walk at lines 263–270. [4](#0-3) 

---

### Proof of Concept

1. Mainchain is at height 200 (post-145000 protocol, so `difficulty_adjustment_interval = 1`, meaning every block is a retarget).
2. Attacker submits a fork block at height 150 (diverging from mainchain at height 149).
3. Attacker submits fork blocks 151–200 on the fork chain.
4. For each fork block at height `H`, `get_next_work_required` calls `get_header_by_height(H-1)`, which returns the **mainchain** block at `H-1` — not the fork's ancestor.
5. The mainchain block at `H-1` has a different `time` than the fork's ancestor at `H-1`.
6. `calculate_next_work_required` produces an expected `bits` based on the wrong time delta.
7. The attacker crafts fork block `H` with `bits` matching this corrupted expected value and mines it at that (potentially lower) difficulty.
8. The contract accepts the block. If the fork's cumulative `chain_work` exceeds the mainchain, a reorg occurs, promoting the low-work fork to canonical status. [5](#0-4) [6](#0-5)

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
```
