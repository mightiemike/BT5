### Title
Fork Difficulty Calculation Uses Mainchain Timestamps Instead of Fork Ancestor Timestamps — (`contract/src/dogecoin.rs`)

### Summary

In `get_next_work_required` (Dogecoin), and analogously in the Bitcoin and Litecoin modules, the difficulty retarget calculation fetches the "first block" of the retarget window via `blocks_getter.get_header_by_height(height_first)`. This function always resolves against the **mainchain** index (`mainchain_height_to_header`), not the fork being validated. When a fork block is submitted, `prev_block_header` is the fork's tip (attacker-supplied timestamp), but `first_block_time` is the mainchain block at the same height (a different block from a different chain). The two timestamps are from different chains, creating a cross-module desynchronization that corrupts the expected-difficulty computation for every fork block.

The difficulty formula then computes: [5](#0-4) 

`modulated_timespan` is `fork_prev_block.time − mainchain_block_at_(height−1).time`. These are timestamps from two different chains. The attacker controls `fork_prev_block.time` (within the 2-hour future-timestamp window enforced by `MAX_FUTURE_BLOCK_TIME_LOCAL`): [6](#0-5) 

By setting fork block timestamps near the maximum allowed value, the attacker inflates `modulated_timespan`, which is clamped to `max_timespan = retarget_timespan + retarget_timespan/2`. This produces a higher target (easier difficulty) than the correct fork difficulty. The contract then enforces `expected_bits == block_header.bits`, so it accepts fork blocks whose `bits` field encodes this artificially easy target.

The same structural defect exists in Bitcoin and Litecoin: [7](#0-6) [8](#0-7) 

For Bitcoin/Litecoin the 2016-block retarget interval limits exposure to deep forks. For Dogecoin post-145,000, **every single fork block** is affected.

### Impact Explanation

An attacker who can submit fork blocks can craft fork timestamps to make the contract compute an artificially easy `expected_bits`. The contract then:

1. Accepts fork blocks whose `bits` field encodes a lower difficulty than the correct fork difficulty requires.
2. Stores these blocks in `headers_pool` with their (reduced) `chain_work`.
3. If the fork's cumulative `chain_work` exceeds the mainchain tip's `chain_work`, `reorg_chain` is triggered, replacing the canonical chain with the attacker's fork.

The broken invariant is: **all accepted headers must satisfy the difficulty that their own chain's retarget window dictates**. Because the retarget window borrows a timestamp from the mainchain instead of the fork's true ancestor, this invariant is violated. The canonical chain pointer (`mainchain_tip_blockhash`) and the height-to-hash index can be corrupted, causing `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to return incorrect results for downstream consumers.

### Likelihood Explanation

`submit_blocks` is gated by `#[trusted_relayer]`, which is an economic staking mechanism (not a closed privileged role). Any account that stakes the required NEAR can become a trusted relayer. A malicious relayer, or a legitimate relayer whose node is compromised, can submit adversarial fork headers. For Dogecoin, the desynchronization affects every fork block after height 145,000, making the attack surface continuous rather than edge-case. The timestamp manipulation is bounded but sufficient to shift the difficulty by up to 50% (the `max_timespan` cap), which is a meaningful reduction in required PoW.

### Recommendation

Replace the mainchain-indexed lookup with a true ancestor traversal along the fork chain. Instead of:

```rust
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

Walk backwards from `prev_block_header` by `blocks_to_go_back` steps using `get_prev_header`, which follows `prev_block_hash` links and therefore stays within the fork's own ancestry regardless of what the mainchain contains at the same height. This mirrors how Bitcoin Core's `GetAncestor` works and eliminates the cross-chain timestamp contamination.

### Proof of Concept

Consider a Dogecoin deployment at height 200,000 (new difficulty protocol active, interval = 1).

**Setup:**
- Mainchain tip at height N, mainchain block at height N−1 has `time = T_main`.
- Attacker submits a fork block at height N whose `prev_block_hash` points to the mainchain block at height N−1 (fork diverges here).

**Attack:**
1. Attacker sets fork block at height N's `time = T_main + MAX_FUTURE_BLOCK_TIME_LOCAL` (2 hours ahead of NEAR block time, the maximum allowed).
2. Contract calls `get_next_work_required` for the next fork block (height N+1):
   - `prev_block_header.block_header.time` = attacker's fork block time = `T_main + 7200`
   - `first_block_time` = `get_header_by_height(N−1)` = mainchain block at N−1 = `T_main`
   - `modulated_timespan` = 7200 seconds, clamped to `max_timespan = 60 + 30 = 90` seconds (for Dogecoin's 60s target, max is 1.5×).
3. The resulting `expected_bits` encodes a target ~50% higher (easier) than the correct fork difficulty.
4. Attacker mines the next fork block to this easier target, submits it; the contract accepts it because `expected_bits == block_header.bits`.
5. Repeat: each subsequent fork block benefits from the same desynchronization, since `get_header_by_height` always returns the mainchain block at that height, not the fork's ancestor.
6. With each fork block requiring ~33% less PoW than correct, the attacker accumulates chainwork faster relative to the PoW expended, eventually triggering `reorg_chain` and corrupting the canonical chain state.
