### Title
Missing `SetWithdrawAddr` Restoration in `InitGenesis` Silently Misdirects Base Rewards to Inaccessible Delegator Address - (File: x/tieredrewards/keeper/genesis.go)

### Summary

`InitGenesis` loads existing tier positions from genesis state but never calls `routeBaseRewardsToOwner` (`SetWithdrawAddr`) for any of them. After a chain export/import or upgrade that re-runs `InitGenesis`, every pre-existing delegated position loses its distribution withdraw-address routing. Subsequent `MsgClaimTierRewards` calls silently send all base rewards to the position's per-position delegator address (`posDelAddr`) — a module-derived address the owner does not control — instead of to the owner.

### Finding Description

When a position is created via `MsgLockTier` or `MsgCommitDelegationToTier`, `createDelegatedPosition` calls `routeBaseRewardsToOwner`, which calls `distributionKeeper.SetWithdrawAddr(ctx, posDelAddr, ownerAddr)`. [1](#0-0) 

This routes all future `WithdrawDelegationRewards` calls for `posDelAddr` to the owner's bank account. The comment in `claimBaseRewards` explicitly documents this assumption: [2](#0-1) 

`claimBaseRewards` then calls `WithdrawDelegationRewards(ctx, posDelAddr, valAddr)` and relies entirely on the distribution module's stored withdraw-address mapping to route the coins to the owner: [3](#0-2) 

`InitGenesis`, however, iterates all positions and calls only `setPosition` for each — it never calls `routeBaseRewardsToOwner`: [4](#0-3) 

The distribution module's `DelegatorsWithdrawAddress` store is a separate store from the tieredrewards module. It is populated by `SetWithdrawAddr` at position-creation time, but `InitGenesis` does not restore it. After a genesis export/import (zero-height genesis) or any upgrade that re-runs `InitGenesis`, the distribution module's withdraw-address entries for all `posDelAddr` accounts are absent. The Cosmos SDK distribution module defaults to returning the delegator address itself when no mapping is stored, so `WithdrawDelegationRewards(posDelAddr, valAddr)` sends rewards to `posDelAddr` — not to the owner.

### Impact Explanation

For every delegated position that existed before the genesis restart, all base rewards claimed via `MsgClaimTierRewards` are silently transferred to `posDelAddr` instead of the owner. `posDelAddr` is a deterministically derived address that the owner does not hold a private key for and cannot sign transactions from. The rewards are inaccessible to the owner during the entire active position lifecycle. They are only recoverable when the position is fully withdrawn via `MsgWithdrawFromTier`, which sweeps `SpendableCoins` from `posDelAddr` to the owner: [5](#0-4) 

Until withdrawal, every `MsgClaimTierRewards` call succeeds with no error but delivers zero base rewards to the owner. The `EventBaseRewardsClaimed` event is emitted with the correct amount, creating a false impression that rewards were paid. The corrupted value is the owner's bank balance: it does not increase by the base reward amount, while `posDelAddr`'s balance silently accumulates the misdirected coins.

### Likelihood Explanation

This is triggered by any zero-height genesis export/import or chain upgrade that re-initializes the tieredrewards module state. These are standard, documented operational procedures for Cosmos chains. Any operator performing a chain migration after positions have been created will trigger this condition for all existing positions. The attacker-controlled entry path is simply submitting a `MsgClaimTierRewards` transaction after such a restart — a normal, unprivileged user action.

### Recommendation

In `InitGenesis`, after loading each delegated position, call `routeBaseRewardsToOwner` to restore the distribution withdraw-address mapping:

```go
for _, pos := range data.Positions {
    // ... existing delegation and validation logic ...
    if del != nil {
        ownerAddr, err := sdk.AccAddressFromBech32(pos.Owner)
        if err != nil {
            panic(err)
        }
        posDelAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
        if err != nil {
            panic(err)
        }
        if err := k.routeBaseRewardsToOwner(ctx, posDelAddr, ownerAddr); err != nil {
            panic(err)
        }
    }
    // ... setPosition ...
}
``` [4](#0-3) 

### Proof of Concept

1. Create a delegated tier position via `MsgLockTier`. Verify `GetDelegatorWithdrawAddr(posDelAddr)` returns `ownerAddr`.
2. Export chain state and re-import (zero-height genesis), re-running `InitGenesis`.
3. Verify `GetDelegatorWithdrawAddr(posDelAddr)` now returns `posDelAddr` (the default — mapping is gone).
4. Submit `MsgClaimTierRewards` for the position. The call succeeds and emits `EventBaseRewardsClaimed` with a non-zero amount.
5. Observe that `ownerAddr` bank balance did not increase; `posDelAddr` bank balance increased by the base reward amount.
6. The owner's base rewards are stranded at `posDelAddr` until `MsgWithdrawFromTier` is called. [6](#0-5)
