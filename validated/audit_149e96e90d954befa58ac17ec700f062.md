### Title
Malicious IBC Counterparty Can Drain Escrowed NFTs via Crafted ClassID Prefix in `processReceivedPacket` — (`x/nft-transfer/keeper/packet.go`)

---

### Summary

`processReceivedPacket` determines whether an incoming IBC packet is a "return-to-origin" unescrow by checking only whether `data.ClassId` starts with the source port/channel prefix string. A malicious counterparty chain can craft a packet whose `ClassId` begins with its own `srcPort/srcChannel/` prefix, causing `IsAwayFromOrigin` to return `false`, then stripping that prefix to resolve to a legitimate native denom that has real NFTs sitting in the destination chain's escrow account. `TransferOwner` is then called with the escrow address as `srcOwner`, which passes all keeper-level checks, and the victim's NFT is transferred to the attacker's receiver address.

The branch is entered whenever `IsAwayFromOrigin(srcPort, srcChannel, data.ClassId)` is `false`, i.e., whenever `data.ClassId` starts with `"srcPort/srcChannel/"`. The code then:

1. Strips that prefix to get `unprefixedClassID`.
2. Calls `ParseClassTrace(unprefixedClassID).IBCClassID()` — if `unprefixedClassID` contains no `/`, `Path` is empty and `IBCClassID()` returns the raw string unchanged (no hashing).
3. Calls `TransferOwner(voucherClassID, tokenID, escrowAddress, receiver)`.

There is **no check** that `data.ClassId` was ever committed by the destination chain in a prior outgoing packet, and **no check** that the resulting `voucherClassID` was legitimately escrowed via this specific channel.

**`ValidateBasic()` (`x/nft-transfer/types/packet.go` lines 41–71) does not validate classID format** — it only checks non-empty string, non-empty tokenIds, matching lengths, and valid bech32 addresses. A classID of `"nft/channel-1/nativeClass"` passes without error.

**`TransferOwner` (`x/nft/keeper/keeper.go` lines 121–137)** only checks:
- `HasDenomID(denomID)` — passes because `"nativeClass"` is a real denom on chain D.
- `IsOwner(denomID, tokenID, srcOwner)` — passes because the NFT was legitimately escrowed at `escrowAddress = GetEscrowAddress(destPort, destChannel)` by a prior honest user.

Both guards are satisfied by the attacker's crafted packet.

---

### Impact Explanation

An attacker who controls an IBC counterparty chain can unescrow any NFT that any user has previously sent from chain D to that counterparty via the shared channel. The attacker specifies their own address as `data.Receiver` and receives the victim's NFT. The original owner loses their asset permanently. The escrow invariant — that only a legitimate return packet from the chain that received the NFT can trigger unescrow — is broken.

---

### Likelihood Explanation

The precondition is that the attacker controls the counterparty chain (or can submit arbitrary packets on it). This is realistic: IBC channels can be opened permissionlessly to attacker-operated chains. Once a single honest user sends an NFT to the attacker's chain, the attacker can immediately drain it. The attack requires no governance access, no private key compromise, and no social engineering — only the ability to commit a packet on the counterparty chain, which the attacker controls by definition.

---

### Recommendation

In the `else` branch of `processReceivedPacket`, before calling `TransferOwner`, verify that the resolved `voucherClassID` was legitimately escrowed via the incoming channel. Concretely:

- Maintain a per-channel registry of classIDs that were escrowed outbound (or equivalently, verify that a class trace record exists whose path begins with the destination port/channel and whose hash matches the expected IBC class ID).
- Alternatively, require that `data.ClassId` (after prefix removal) resolves to an IBC class ID (`ibc/<hash>`) whose stored trace path begins with `destPort/destChannel/`, proving the denom originated from this chain and was sent via this specific channel — not an arbitrary native denom.

   - `data.ClassId = "nft/channel-1/nativeClass"`
   - `data.TokenIds = ["token1"]`
   - `data.Receiver = attacker_bech32_address`

**Chain D receives the packet:**
5. `OnRecvPacket` → `OnRecvPacket` keeper → `processReceivedPacket`.
6. `IsAwayFromOrigin("nft","channel-1","nft/channel-1/nativeClass")` → `false` (string has prefix `"nft/channel-1/"`).
7. `escrowAddress = GetEscrowAddress("nft","channel-0")` — the address holding `"token1"`.
8. `RemoveClassPrefix("nft","channel-1","nft/channel-1/nativeClass")` → `"nativeClass"`.

10. `TransferOwner(ctx, "nativeClass", "token1", escrowAddress, attacker)` — succeeds because `escrowAddress` owns `"token1"`.
11. Attacker now owns `"token1"`. User A's NFT is gone.

	if err != nil {

	k.swapOwner(ctx, denomID, tokenID, srcOwner, dstOwner)
	return nil

func GetEscrowAddress(portID, channelID string) sdk.AccAddress {
	// a slash is used to create domain separation between port and channel identifiers to
	// prevent address collisions between escrow addresses created for different channels
	contents := fmt.Sprintf("%s/%s", portID, channelID)

	// ADR 028 AddressHash construction
	preImage := []byte(Version)
	preImage = append(preImage, 0)
	preImage = append(preImage, contents...)
	hash := sha256.Sum256(preImage)
	return hash[:20]
