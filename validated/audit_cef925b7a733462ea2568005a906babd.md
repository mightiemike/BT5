### Title
Stale `cumulativeBorrowsMultiplierX18` Used in Health Checks Without Prior `updateStates` Call Enables Liquidation Bypass — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`ClearinghouseLiq.liquidateSubaccountImpl()` performs its maintenance-health check and all subsequent balance computations using a `cumulativeBorrowsMultiplierX18` that may be arbitrarily stale, because `SpotEngineState.updateStates()` is never called before the health check. This is the direct Nado analog of the reported bug: just as `collectFees()` used a stale `cumulativeInterestRate` without first calling `updateBalances()`, Nado's liquidation path uses stale cumulative multipliers without first refreshing spot-engine state.

---

### Finding Description

`SpotEngineState` tracks borrow interest through two compounding multipliers stored in `State`:

```
cumulativeDepositsMultiplierX18
cumulativeBorrowsMultiplierX18
```

These are updated **only** inside `_updateState()`, which is invoked exclusively from `updateStates(uint128 dt)`: [1](#0-0) 

`updateStates` carries the `onlyEndpoint` modifier, so it can only be called by the sequencer as a discrete transaction. Between sequencer calls the multipliers sit frozen in storage.

Every balance read in the protocol goes through `balanceNormalizedToBalance()`: [2](#0-1) 

which multiplies the stored `amountNormalized` by whichever multiplier is currently in `states[productId]`. `getBalance()` loads that state directly from storage with no freshness guarantee: [3](#0-2) 

`liquidateSubaccountImpl` is the externally reachable entry point: [4](#0-3) 

Its very first substantive check is: [5](#0-4) 

`isUnderMaintenance()` delegates to `getHealthFromClearinghouse()` → `Clearinghouse.getHealth()` → `spotEngine.getHealthContribution()`, which ultimately calls `getBalance()` on the stale state. No call to `updateStates` precedes any of this. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

For a subaccount with a **negative** spot balance (a borrower / liability):

```
balance.amount = amountNormalized × cumulativeBorrowsMultiplierX18
```

Because `cumulativeBorrowsMultiplierX18` grows monotonically with time (interest compounds), a stale (lower) value makes `balance.amount` **less negative** than the true current debt. The health contribution is therefore overstated. `isUnderMaintenance()` returns `false` even though the account is genuinely below the maintenance threshold when the correct, up-to-date multiplier is applied.

Consequence: `liquidateSubaccountImpl` reverts at line 603 with `ERR_NOT_LIQUIDATABLE`. The undercollateralised borrower cannot be liquidated. The longer the gap since the last `updateStates` call, the larger the discrepancy. In a high-utilisation, high-borrow-rate environment the accrued interest between sequencer ticks can be material, leaving the protocol holding unrecoverable bad debt.

The same stale multiplier also corrupts the balance deltas applied inside `_handleLiquidationPayment` when a liquidation does eventually proceed, meaning the liquidation payment is computed against an understated liability. [8](#0-7) 

---

### Likelihood Explanation

Medium. The sequencer calls `updateStates` periodically, but the gap is non-zero and protocol-controlled, not enforced on-chain. Any liquidator who submits a liquidation transaction in a batch that does not include a preceding `updateStates` call will observe the stale-state behaviour. In periods of high utilisation (borrow rate can reach the `interestLargeCapX18` cap of `1.0` annualised per `SpotEngine.sol` line 28), even a few hours of staleness produces a measurable understatement of debt, sufficient to flip a borderline account from liquidatable to apparently healthy. [9](#0-8) 

---

### Recommendation

`updateStates` should be called (or its equivalent state-refresh logic inlined) for all relevant spot products **before** any health check or balance computation inside `liquidateSubaccountImpl`. Because `updateStates` is `onlyEndpoint`, the sequencer must be required by protocol convention (or enforced on-chain via a staleness guard) to include a state-update step in every batch that contains a liquidation transaction. A staleness guard — e.g., `require(block.timestamp - lastUpdateTime[productId] < MAX_STALENESS)` — would make the invariant explicit and revert liquidations that would otherwise silently use stale interest state.

---

### Proof of Concept

1. Borrower opens a leveraged spot position; their maintenance health is slightly above zero.
2. Significant time passes; `updateStates` has not been called. `cumulativeBorrowsMultiplierX18` is stale and lower than the true current value.
3. With the correct current multiplier the borrower's debt has grown enough that maintenance health < 0.
4. Liquidator submits `liquidateSubaccount` through the endpoint.
5. `liquidateSubaccountImpl` calls `isUnderMaintenance(txn.liquidatee)`.
6. `getHealth()` calls `spotEngine.getHealthContribution()` → `getBalance()` → `balanceNormalizedToBalance(states[productId], ...)` using the stale `cumulativeBorrowsMultiplierX18`.
7. The borrower's debt is understated; health appears ≥ 0; `isUnderMaintenance()` returns `false`.
8. `require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE)` reverts.
9. The borrower remains unliquidated; bad debt accumulates until the sequencer eventually calls `updateStates`. [10](#0-9) [11](#0-10)

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }
