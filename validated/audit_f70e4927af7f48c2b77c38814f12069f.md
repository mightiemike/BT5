Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original Swapper, Enabling Complete Allowlist Bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router address, not the original trader. If the pool admin allowlists the router to support periphery-mediated swaps, every unprivileged user can bypass the allowlist entirely by routing through the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- whoever called pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension via `_callExtensionsInOrder`. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` is used, it calls `pool.swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ...);
```

This makes the router the `msg.sender` seen by the pool, so the extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][original_user]`. The same pattern applies to `exactInput` (L104), `exactOutputSingle` (L136), and `exactOutput` (L165). The router never propagates the original caller's identity; the pool has no mechanism to receive it.

This creates two mutually exclusive failure modes:
- **Router not allowlisted**: Every allowlisted user who calls through the router is blocked — broken core functionality.
- **Router allowlisted** (admin enables router support): Every unprivileged user bypasses the allowlist by routing through the router — complete policy bypass.

No configuration simultaneously allows specific users to use the router while blocking others.

## Impact Explanation
A pool deploying `SwapAllowlistExtension` to create a curated trading environment (KYC-gated, institutional-only, or low-toxicity flow) cannot enforce that policy for router-mediated swaps. If the admin allowlists the router to support the standard periphery path, any unprivileged user can trade on the pool. Unrestricted toxic or arbitrage flow against a pool designed for curated flow causes direct LP value leakage through adverse selection — a fund-impacting consequence meeting the Medium threshold. This is broken core extension functionality with direct LP exposure.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical periphery entry point. Pool admins who deploy a swap-allowlisted pool and want their allowlisted users to access the standard router will naturally allowlist the router address. This is the expected configuration and is reachable by any unprivileged user with a single public call to any of the four router entry points.

## Recommendation
The router must propagate the original caller's identity to the pool so the extension can gate the economically relevant actor. Two viable approaches:

1. **Extension-data convention**: The router encodes `msg.sender` into `extensionData` for each hop, and `SwapAllowlistExtension` decodes and checks that field instead of (or in addition to) `sender`.
2. **Pool-level originator field**: Add an optional `originator` parameter to `pool.swap()` that the router populates with `msg.sender`; the pool forwards it to extensions alongside `sender`.

Either approach must be enforced consistently across all router entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) and the extension must validate that the encoded originator matches the actual payer context to prevent spoofing.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is the intended gated user
3. Pool admin calls setAllowedToSwap(pool, router, true)  // admin enables router support
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(recipient, ...) with msg.sender = router
6. Pool calls _beforeSwap(sender=router, ...)
7. SwapAllowlistExtension checks allowedSwapper[pool][router] → true
8. Bob's swap executes — allowlist completely bypassed
```
