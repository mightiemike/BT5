### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables USDC.e Drain Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` is a permissionless function that calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value. If the USDC token at the hardcoded address on chain 57073 (Ink) is non-reverting on failure, an attacker with zero USDC balance or allowance can drain USDC.e from any `DirectDepositV1` contract without providing any USDC in exchange.

---

### Finding Description

`replaceUsdcEWithUsdc` is callable by any external address with no access control modifier. Its logic is:

1. Read the USDC.e balance held by `directDepositV1`.
2. Pull USDC from `msg.sender` into `directDepositV1` via `transferFrom`.
3. Withdraw USDC.e from `directDepositV1` to `ContractOwner` via `DirectDepositV1.withdraw`.
4. Send that USDC.e from `ContractOwner` to `msg.sender` via `safeTransfer`.

Step 2 uses a bare `transferFrom` call whose boolean return value is silently discarded:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [1](#0-0) 

The very next line uses the safe wrapper `safeTransfer`, demonstrating that the codebase is aware of the pattern but inconsistently applied it here:

```solidity
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
``` [2](#0-1) 

The rest of the codebase consistently uses `ERC20Helper.safeTransferFrom`, which performs a low-level call and requires both `success == true` and a truthy decoded return value:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [3](#0-2) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on chain 57073 returns `false` instead of reverting on a failed transfer (non-compliant ERC20), an attacker can:

- Call `replaceUsdcEWithUsdc(subaccount)` for any subaccount whose `directDepositV1` holds USDC.e.
- The `transferFrom` silently fails (no USDC leaves the attacker).
- `DirectDepositV1.withdraw` moves all USDC.e from `directDepositV1` to `ContractOwner`.
- `safeTransfer` sends that USDC.e to the attacker.

The attacker receives USDC.e for free. The `directDepositV1` contract is drained of its USDC.e balance, and the subaccount owner loses their pending deposit. [4](#0-3) 

---

### Likelihood Explanation

The function has no access control — any unprivileged caller can invoke it on chain 57073. The only precondition is that a `directDepositV1` contract for the target subaccount holds a nonzero USDC.e balance. The likelihood depends on whether the specific USDC deployment at the hardcoded address is reverting or non-reverting on failure. Circle's canonical USDC does revert, but the address is a hardcoded chain-specific deployment whose exact behavior must be verified. The structural vulnerability is present regardless.

---

### Recommendation

Replace the bare `transferFrom` call with the `safeTransferFrom` wrapper already used throughout the codebase:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [5](#0-4) 

---

### Proof of Concept

1. A `DirectDepositV1` contract exists for `subaccount` on chain 57073 and holds 1000 USDC.e.
2. Attacker (with 0 USDC and 0 allowance) calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)`.
3. `balance = IERC20Base(usdcE).balanceOf(directDepositV1)` → 1000.
4. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, 1000)` → returns `false`, no revert, no USDC moved.
5. `DirectDepositV1(directDepositV1).withdraw(usdcE)` → 1000 USDC.e transferred to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(attacker, 1000)` → 1000 USDC.e transferred to attacker.
7. Attacker receives 1000 USDC.e at zero cost. The subaccount's pending deposit is wiped. [4](#0-3)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

**File:** core/contracts/libraries/ERC20Helper.sol (L23-42)
```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
