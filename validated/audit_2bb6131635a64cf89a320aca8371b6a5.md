### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Silent Failure and Potential usdcE Drain — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the returned boolean. If the call returns `false` instead of reverting, execution continues: the DDA's usdcE is withdrawn to `ContractOwner` and forwarded to `msg.sender` — without the caller ever having provided USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is a public migration helper on Ink (chain 57073) that is intended to swap usdcE held in a DDA for USDC on a 1-for-1 basis. The function:

1. Reads the usdcE balance of the target DDA.
2. Calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` — **return value discarded**.
3. Calls `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` — pulls all usdcE to `ContractOwner`.
4. Calls `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — sends usdcE to caller. [1](#0-0) 

The `IERC20Base` interface explicitly declares `transferFrom` as returning `bool`: [2](#0-1) 

The project's own `ERC20Helper.safeTransferFrom` exists precisely to handle this pattern — it low-level calls and requires `success && (data.length == 0 || abi.decode(data, (bool)))`: [3](#0-2) 

`replaceUsdcEWithUsdc` bypasses `ERC20Helper` and calls `transferFrom` directly, discarding the return value. [4](#0-3) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink returns `false` on a failed transfer (e.g., insufficient balance or allowance) rather than reverting, the function silently proceeds. The attacker receives the full usdcE balance of the victim DDA without having transferred any USDC. This is a direct, concrete asset delta: usdcE tokens are drained from the DDA and delivered to the attacker at zero cost.

---

### Likelihood Explanation

The function has **no access control** — any address on Ink (chain 57073) can call it for any subaccount that has a deployed DDA with a non-zero usdcE balance. The only gate is the chain ID check. The USDC token address used (`0x2D270e6886d130D724215A266106e6832161EAEd`) is a deployment-specific contract whose exact revert-vs-return-false behavior on Ink is not guaranteed to match mainnet Circle USDC. The code pattern is unconditionally unsafe regardless of token behavior. [5](#0-4) 

---

### Recommendation

Replace the bare `transferFrom` call with the project's existing `safeTransferFrom` helper from `ERC20Helper`:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [3](#0-2) 

---

### Proof of Concept

1. A subaccount `S` has a deployed DDA (`directDepositV1Address[S] != address(0)`) holding `N` usdcE tokens.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(S)` on Ink (chain 57073) with zero USDC balance or zero allowance.
3. `IERC20Base(usdc).transferFrom(attacker, dda, N)` returns `false` — **not checked, execution continues**.
4. `DirectDepositV1(dda).withdraw(usdcE)` transfers all `N` usdcE from the DDA to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, N)` sends `N` usdcE to the attacker.
6. Attacker has received `N` usdcE without providing any USDC. The DDA is drained. [6](#0-5)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
