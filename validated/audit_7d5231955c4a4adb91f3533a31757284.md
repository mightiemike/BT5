### Title
`SwapAllowlistExtension` Bypass via Router: Allowlisted Router Grants Unrestricted Swap Access to All Users — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap(...)` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router**, not the end user. Any pool admin who allowlists the router (to enable router-mediated swaps for their allowlisted users) inadvertently grants unrestricted swap access to every user who calls through the router, completely defeating the per-user allowlist.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension caller) and `sender` is the first argument forwarded by the pool — which is `msg.sender` of the pool's own `swap` call. The pool passes its `msg.sender` as `sender` to `_beforeSwap` / `_afterSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L281-295
_afterSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    ...
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle(...)`, the router calls `pool.swap(recipient, ...)` with `msg.sender = router`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
    );
``` [3](#0-2) 

So the extension receives `sender = router`, and the check becomes `allowedSwapper[pool][router]`. If the router is allowlisted, **every user** who calls through the router passes the check, regardless of whether they are individually allowlisted.

The project's own audit-target file explicitly flags this identity mismatch:

> *"Because public users may enter through the router, the hook must gate the same actor the pool designers thought they were allowlisting."*
> *"Test direct swaps and router swaps on allowlisted pools and assert the hook cannot be bypassed by routing through an intermediate public contract."* [4](#0-3) 

---

### Impact Explanation

A pool admin who deploys a `SwapAllowlistExtension`-gated pool and allowlists the router (a natural step to let their approved users access the pool via the standard periphery) opens the pool to **all** users. Any non-allowlisted address can call `router.exactInputSingle` / `exactInput` / `exactOutput` and swap freely. LP capital in a pool intended to be private or KYC-restricted is exposed to the full public, enabling unauthorized liquidity drain, sandwich attacks, and violation of any access-control invariant the pool admin intended to enforce.

---

### Likelihood Explanation

The trigger requires only that the pool admin allowlists the router — a routine operational step for any pool that wants to support the standard periphery. No privileged exploit, no malicious setup, no non-standard token. Any user with a standard ERC-20 balance can execute the bypass by calling the public router.

---

### Recommendation

The `SwapAllowlistExtension` must gate the **actual end user**, not the immediate caller of `pool.swap`. Two viable approaches:

1. **Extension-data forwarding**: Have the router encode the original `msg.sender` in `extensionData`; the extension decodes and checks that address. This requires a coordinated change to both the router and the extension.
2. **Direct-only enforcement**: Document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and revert if `sender` is a known router address, forcing allowlisted users to call the pool directly.

Additionally, add an integration test that deploys a pool with `SwapAllowlistExtension`, allowlists the router, and asserts that a non-allowlisted user calling through the router is **rejected**.

---

### Proof of Concept

```
1. Deploy MetricOmmPool with SwapAllowlistExtension as a beforeSwap hook.
2. Pool admin calls setAllowedToSwap(pool, user1, true)       // allowlist user1
3. Pool admin calls setAllowedToSwap(pool, router, true)      // allowlist router for UX
4. user2 (not allowlisted) calls:
       router.exactInputSingle({pool: pool, tokenIn: token0, ...})
5. Router calls pool.swap(recipient, ...) — msg.sender = router
6. Pool calls extension.beforeSwap(sender=router, ...)
7. Extension evaluates: allowedSwapper[pool][router] == true  → passes
8. Swap executes. user2 successfully bypasses the per-user allowlist.
```

The allowlist invariant is broken: `allowedSwapper[pool][user2]` is `false`, yet user2 swaps. Every non-allowlisted address can replicate step 4 at will.
