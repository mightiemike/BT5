### Title
Permanent Loss of Deposited Collateral When Slow-Mode `DepositCollateral` Transaction Fails Silently — (`core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` transfers user tokens to the `Clearinghouse` contract **before** queuing a slow-mode `DepositCollateral` transaction. If that queued transaction later fails during execution, the `_executeSlowModeTransaction` catch block silently discards the failure — a comment in the code explicitly reads `// try return funds now removed` — leaving the transferred tokens permanently locked in `Clearinghouse` with no recovery path.

---

### Finding Description

The deposit flow in `Endpoint.depositCollateralWithReferral` is a two-phase operation:

**Phase 1 — Immediate token transfer** (`Endpoint.sol` lines 144–148):
```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
```
`handleDepositTransfer` calls `safeTransferFrom(token, from, amount)` then `safeTransfer(token, address(clearinghouse), amount)`, moving real ERC-20 tokens from the user into the `Clearinghouse` contract immediately.

**Phase 2 — Deferred accounting** (`Endpoint.sol` lines 150–166):
A `DepositCollateral` slow-mode transaction is queued with a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). When eventually executed, `processSlowModeTransactionImpl` calls `clearinghouse.depositCollateral(txn)` to credit the user's subaccount balance.

The execution of the queued transaction happens inside `_executeSlowModeTransaction` (`Endpoint.sol` lines 185–229):

```solidity
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed
}
```

If `processSlowModeTransaction` reverts for any reason, the catch block does nothing. The queued transaction has already been deleted from storage (`delete slowModeTxs[_slowModeConfig.txUpTo++]` at line 194), so it cannot be retried. The tokens are already in `Clearinghouse` but the user's subaccount is never credited.

`clearinghouse.depositCollateral` can revert if:
- `_decimals(txn.productId)` reverts because the token address for the product returns `address(0)` (e.g., after a product configuration change or upgrade during the 3-day window)
- `spotEngine.updateBalance` reverts due to an arithmetic overflow or product-level invariant violation
- Any other state change during the 3-day delay that invalidates the deposit parameters

The comment `// try return funds now removed` is direct evidence that a fund-return mechanism was intentionally removed, leaving the catch block as a silent discard.

---

### Impact Explanation

Deposited ERC-20 tokens are permanently locked in the `Clearinghouse` contract. The user's subaccount balance is never credited. There is no retry mechanism (the slow-mode entry is deleted before execution), no admin recovery function scoped to this case, and no event emitted on failure. The loss is total and irreversible for the affected deposit amount.

---

### Likelihood Explanation

The 3-day `SLOW_MODE_TX_DELAY` creates a meaningful window during which protocol state can change. Any upgrade, product configuration change, or engine-level invariant shift that causes `clearinghouse.depositCollateral` to revert will trigger the bug. The `DirectDepositV1.creditDeposit()` function calls `depositCollateralWithReferral` for every product ID, increasing the surface area. The vulnerability is reachable by any unprivileged user who deposits collateral via the slow-mode path.

---

### Recommendation

Restore a fund-return mechanism in the catch block of `_executeSlowModeTransaction`. When a `DepositCollateral` slow-mode transaction fails, the contract should identify the deposited token and amount from the queued transaction data and return the tokens to the original sender before discarding the entry. This is the exact mitigation recommended in the analogous Stargate report: wrap the accounting call in a try/catch and refund the user on failure.

---

### Proof of Concept

1. User calls `Endpoint.depositCollateralWithReferral(subaccount, productId, amount, referral)`.
2. `handleDepositTransfer` executes: `amount` tokens move from user → `Endpoint` → `Clearinghouse`. Real ERC-20 balance is now in `Clearinghouse`.
3. A `DepositCollateral` slow-mode entry is queued at index `txCount`, with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (3 days).
4. During the 3-day window, a protocol upgrade or product configuration change causes `clearinghouse.depositCollateral` to revert (e.g., `_decimals(productId)` reverts because `spotEngine.getConfig(productId).token` now returns `address(0)`).
5. After 3 days, `executeSlowModeTransaction()` is called (by anyone).
6. `_executeSlowModeTransaction` deletes the entry from `slowModeTxs` and calls `this.processSlowModeTransaction(txn.sender, txn.tx)` inside a try/catch.
7. `processSlowModeTransaction` → `clearinghouse.depositCollateral` reverts.
8. The catch block fires. `gasleft()` checks pass (not an OOG). Execution falls through to `// try return funds now removed` — no action taken.
9. The slow-mode entry is gone. The tokens remain in `Clearinghouse`. The user's subaccount balance is `0`. Funds are permanently lost. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```text
    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
```
