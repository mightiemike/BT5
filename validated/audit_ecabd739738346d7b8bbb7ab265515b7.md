Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of End User, Enabling Full Per-User Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of the pool's `swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. The extension therefore evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`, making the per-user allowlist trivially bypassable by any trader who routes through the public router.

## Finding Description
In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- whoever called pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling` forwards this `sender` value directly to each registered extension. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool address and `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` is used, the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
  );
```

The pool's `msg.sender` is the router address. The extension evaluates `allowedSwapper[pool][router]`, never consulting `allowedSwapper[pool][user]`. The same misbinding applies to `exactInput` (intermediate hops use `address(this)` as payer) and `exactOutput` (recursive callback path calls `pool.swap()` from within the router's callback, so `msg.sender` at the pool is again the router).

## Impact Explanation
A pool admin who deploys a `SwapAllowlistExtension` and also wants users to access the pool via the supported router must add the router to the allowlist (`allowedSwapper[pool][router] = true`). Once the router is allowlisted, every user — including those explicitly excluded — can bypass the per-user gate by routing through `MetricOmmSimpleRouter`. The allowlist provides zero protection against router-mediated access. Unauthorized traders can execute swaps on curated pools, breaking the access-control invariant the extension is designed to enforce. This constitutes a broken core pool functionality / admin-boundary break where an unprivileged path nullifies a pool admin's access control configuration.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary supported periphery path for end users. Any pool admin who deploys a `SwapAllowlistExtension` and also wants normal UX faces an impossible choice: block the router entirely (breaking normal UX) or allowlist it (nullifying the per-user gate). An attacker needs only to call the public router with a valid swap — no special privileges, no front-running, no oracle manipulation. The exploit is repeatable and requires no setup beyond the pool admin having allowlisted the router for normal operation.

## Recommendation
**Short term**: Have the router encode the original `msg.sender` in `extensionData` and have the extension decode and check it. Alternatively, add a `payer` or `originator` field to the swap interface so the pool can forward the economically relevant actor to extensions.

**Long term**: Establish a protocol-level convention that extension hooks receive the address of the actor who initiated the economic action (the end user), not the address of the intermediary contract that called the pool. The `sender` argument passed to extension hooks should be the original initiator, not the immediate caller.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin allowlists Alice: `setAllowedToSwap(pool, alice, true)`.
3. Pool admin allowlists the router to support normal UX: `setAllowedToSwap(pool, router, true)`.
4. Eve (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, ...)` — pool's `msg.sender` is the router.
6. Pool calls `extension.beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router] == true` → no revert.
8. Eve's swap executes successfully on the curated pool, bypassing the allowlist entirely.

The invariant broken: `allowedSwapper[pool][eve] == false` should have blocked Eve's trade, but the extension never reads that mapping entry.
