### Title
Kickout-Exemption Sort Uses Raw Endorsement Ratio While Reward Calculation Uses Binarized Ratio, Causing Inconsistent Validator Uptime Values - (`chain/epoch-manager/src/validator_stats.rs`, `chain/epoch-manager/src/lib.rs`)

---

### Summary

`get_sortable_validator_online_ratio` always calls `get_validator_online_ratio` with `endorsement_cutoff_threshold = None`, producing a raw endorsement ratio. The reward calculation in `finalize_epoch` passes `endorsement_cutoff_threshold = Some(chunk_validator_only_kickout_threshold)`, binarizing the endorsement contribution to 0 or 1. The same validator's uptime is therefore computed with two different formulas depending on which code path executes, causing the kickout-exemption sort to rank validators by a metric that diverges significantly from the metric used to compute their actual rewards.

---

### Finding Description

**Path 1 — Exemption sort (no cutoff):** [1](#0-0) 

`get_sortable_validator_online_ratio` hard-codes `None` as the `endorsement_cutoff_threshold`, so the raw endorsement ratio (e.g. 69/100) flows into the uptime average. [2](#0-1) 

This raw-ratio sort determines which validators are exempted from kickout via `compute_exempted_kickout`.

**Path 2 — Reward calculation (cutoff applied):** [3](#0-2) 

`finalize_epoch` sets `endorsement_cutoff_threshold = Some(chunk_validator_only_kickout_threshold)` (70 on mainnet). [4](#0-3) 

`calculate_reward` passes this cutoff to `get_validator_online_ratio`, which binarizes the endorsement contribution: below threshold → 0, at or above → 1. [5](#0-4) 

**Divergent value — concrete example** (blocks=100/100, chunks=100/100, endorsements=69/100, threshold=70, `online_min_threshold`=0.9):

| Path | Endorsement contribution | Uptime ratio | Reward |
|---|---|---|---|
| Exemption sort (`None`) | 69/100 | ≈ 0.897 | — |
| Reward calc (`Some(70)`) | 0/1 (binarized) | ≈ 0.667 | **ZERO** (below 0.9 min) |

For a validator with 71% endorsement the reward-path uptime is 1.0 (full reward), while the sort-path uptime is ≈ 0.903 — nearly identical to the 69% validator's sort position of 0.897. The two validators are ranked almost the same in the exemption sort, but their reward outcomes are maximally different (zero vs. full).

---

### Impact Explanation

The exemption algorithm (`compute_exempted_kickout`) is designed to protect the highest-uptime validators from being kicked out when network instability would otherwise remove too many validators at once. Because the sort uses the raw endorsement ratio instead of the binarized one, a validator whose effective reward-uptime is zero (endorsement just below the cutoff) can rank nearly as high as a validator with full effective uptime (endorsement just above the cutoff). This causes:

1. **Wrong validators exempted from kickout**: A validator with 69% endorsement (zero effective uptime, should be kicked out) may be exempted while a validator with 71% endorsement (full effective uptime, should stay) is kicked out.
2. **Incorrect validator set composition**: The network retains underperforming validators and ejects well-performing ones, degrading liveness and security guarantees.
3. **Reward inconsistency**: The exempted validator receives zero reward despite being protected from kickout, creating an economic inconsistency that persists across epochs.

---

### Likelihood Explanation

The `chunk_validator_only_kickout_threshold` is 70 on mainnet (confirmed in epoch config JSON and CHANGELOG). Any chunk-validator-only node whose endorsement ratio falls in the range [0%, 70%) triggers the divergence. Network jitter, temporary connectivity issues, or deliberate positioning near the threshold are all realistic triggers. The `validator_max_kickout_stake_perc` cap (100 on mainnet, lower in some configs) is the gate that activates the exemption sort; when it is below 100, the bug is reachable on every epoch boundary where enough validators are below threshold. [6](#0-5) 

---

### Recommendation

`get_sortable_validator_online_ratio` should accept and forward the same `endorsement_cutoff_threshold` that `finalize_epoch` uses for reward calculation. Concretely:

```rust
// validator_stats.rs
pub(crate) fn get_sortable_validator_online_ratio(
    stats: &BlockChunkValidatorStats,
    endorsement_cutoff_threshold: Option<u8>,   // add parameter
) -> BigRational {
    let ratio = get_validator_online_ratio(stats, endorsement_cutoff_threshold);
    // ... rest unchanged
}
```

And in `compute_validators_to_reward_and_kickout`, pass `Some(config.chunk_validator_only_kickout_threshold)` instead of the implicit `None`. [7](#0-6) 

---

### Proof of Concept

Using the existing test harness in `chain/epoch-manager/src/tests/mod.rs`, construct two validators with identical block/chunk stats but endorsement ratios straddling the cutoff (e.g. 69% and 71%), set `validator_max_kickout_stake_perc` low enough to activate exemption, and call `compute_validators_to_reward_and_kickout`. Observe that the 69%-endorsement validator is ranked higher (or equal) in the exemption sort than the 71%-endorsement validator, despite the former receiving zero reward in `calculate_reward` (called separately with `endorsement_cutoff_threshold = Some(70)`) and the latter receiving full reward. The divergence between the two uptime values — 0.897 vs. 0.667 for the 69% validator — is the exact inconsistency. [8](#0-7) [9](#0-8)

fn get_endorsement_ratio(stats: &ValidatorStats, cutoff_threshold: Option<u8>) -> (u64, u64) {
    let (numer, denom) = if stats.expected == 0 {
        debug_assert_eq!(stats.produced, 0);
        (0, 0)
    } else if let Some(threshold) = cutoff_threshold {
        if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
    } else {
        (stats.produced, stats.expected)
    };
    (numer, denom)
