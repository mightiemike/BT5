### Title
Linked Signer Can Irrevocably Entrench Its Own Access by Re-Linking Before Revocation — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.sol`, the fast-path `LinkSigner` transaction is validated with `allowLinkedSigner = true`, meaning the currently active linked signer is accepted as a valid signer for a `LinkSigner` transaction. This allows a compromised linked signer to front-run any revocation attempt by the subaccount owner by submitting its own `LinkSigner` transaction that re-delegates signing authority to a fresh address it controls. Because the linked signer can also sign `WithdrawCollateral` and `TransferQuote` transactions, this translates directly into permanent, irrevocable access to drain the subaccount.

---

### Finding Description

**Root cause — `allowLinkedSigner = true` on `LinkSigner` processing:**

In the fast-path transaction processor, the `LinkSigner` branch calls `validateSignedTx` with the fourth argument set to `true`:

```solidity
// EndpointTx.sol lines 581–590
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // <-- allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` forwards this flag to `validateSignature`, which in turn passes `getLinkedSigner(sender)` as the accepted signer to `Verifier.validateSignature`:

```solidity
// EndpointTx.sol lines 172–184
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`Verifier.validateSignature` accepts either the subaccount owner address or the linked signer address:

```solidity
// Verifier.sol lines 297–303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

**Consequence:** The linked signer key itself is sufficient to sign a valid `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with any address the attacker chooses. The subaccount owner's revocation transaction (also a `LinkSigner` tx, signed by the owner key) and the attacker's re-link transaction are both valid on-chain. Whichever the sequencer includes last wins. Because the attacker controls a live key and can continuously submit re-link transactions, they can perpetually maintain signing authority over the subaccount.

**Contrast with the slow-mode path:** The slow-mode `LinkSigner` handler at line 237 uses `validateSender(txn.sender, sender)`, which checks that `msg.sender` is the actual owner address — the linked signer cannot use this path. The vulnerability is exclusive to the fast (sequencer-batched) path. [4](#0-3) 

---

### Impact Explanation

A compromised linked signer can:

1. **Prevent permanent revocation** — continuously front-run any `LinkSigner` revocation by the owner with its own `LinkSigner` re-delegation to a fresh address it controls.
2. **Drain the subaccount** — the linked signer is accepted for `WithdrawCollateral` and `TransferQuote` transactions (both also use `allowLinkedSigner = true`), so it can withdraw all collateral or transfer quote balances to an attacker-controlled subaccount.
3. **Manipulate positions** — the linked signer can place, cancel, or match orders on behalf of the subaccount.

The corrupted state is `linkedSigners[subaccount]` — once the attacker controls this mapping entry, the subaccount owner loses the ability to reclaim sole signing authority. The concrete asset delta is the full collateral balance of the affected subaccount.

---

### Likelihood Explanation

Linked signers are a standard, documented feature used by API traders and automated strategies. Any user who has ever set a linked signer and whose linked signer key is later compromised (e.g., leaked API key, compromised hot wallet) faces this scenario. The attacker needs only a valid linked signer key and the ability to submit transactions to the sequencer — both are normal, unprivileged capabilities. No admin access, governance capture, or cryptographic break is required.

---

### Recommendation

Remove `allowLinkedSigner = true` from the fast-path `LinkSigner` handler. Only the subaccount owner (the address embedded in the `sender` bytes32) should be permitted to change the linked signer. The slow-mode path already enforces this correctly via `validateSender`. The fast path should be brought into alignment:

```solidity
// Change allowLinkedSigner from true to false
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may re-link
);
``` [1](#0-0)

### Proof of Concept

1. Alice owns subaccount `S` and has set `linkedSigners[S] = attacker_key_1`.
2. `attacker_key_1` is compromised. Alice submits a fast-path `LinkSigner` transaction signed by her owner key, setting `linkedSigners[S] = address(0)`.
3. The attacker, controlling `attacker_key_1`, simultaneously submits a fast-path `LinkSigner` transaction signed by `attacker_key_1`, setting `linkedSigners[S] = attacker_key_2` (another address they own). This transaction passes `validateSignedTx(..., true)` because `attacker_key_1 == getLinkedSigner(S)` at the time of validation.
4. If the attacker's transaction is sequenced after Alice's, `linkedSigners[S]` is now `attacker_key_2`. Alice's revocation is undone.
5. The attacker repeats step 3 indefinitely with fresh keys, permanently maintaining signing authority.
6. Using `attacker_key_2`, the attacker submits a `WithdrawCollateral` transaction (also `allowLinkedSigner = true`) to drain Alice's collateral to an attacker-controlled address. [5](#0-4) [6](#0-5) [7](#0-6)
