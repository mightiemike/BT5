### Title
Missing Lower-Bound Validation on `confirmations` Allows Zero-Confirmation SPV Proof Acceptance — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` parameter with no lower-bound check. When `confirmations = 0` is supplied, the confirmation-depth guard reduces to `X >= 0` — always true for `u64` — making the check trivially pass for any block in the main chain, including the chain tip. This is the direct analog of the multisig threshold=0 bug: the contract validates only the upper bound (`confirmations <= gc_threshold`) but not the lower bound (`confirmations >= 1`), causing the critical security check to be unconditionally bypassed.

### Finding Description

`ProofArgs.confirmations` is a `u64` field that is fully caller-controlled: [1](#0-0) 

Inside `verify_transaction_inclusion`, the contract validates only the upper bound: [2](#0-1) 

Then the confirmation-depth guard is: [3](#0-2) 

When `args.confirmations = 0`:
- `0 <= gc_threshold` → always passes (upper-bound check)
- `(heaviest_height - target_height + 1) >= 0` → always passes for `u64` (depth check)

Both guards are trivially satisfied. The function proceeds to Merkle proof verification and returns `true` for any transaction in any main-chain block, **including the chain tip**, which has zero subsequent confirmations and is fully exposed to reorganization.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase proof check, so it inherits the same flaw: [4](#0-3) 

### Impact Explanation

A downstream NEAR contract that calls `verify_transaction_inclusion_v2` (the current recommended API) with a user-supplied or attacker-influenced `confirmations` value of `0` will receive `true` for a transaction that exists only in the tip block — a block with no subsequent work and full double-spend exposure. The contract's core security guarantee — that a verified transaction has at least N blocks of accumulated PoW protecting it — is completely nullified. Any bridge, payment gate, or cross-chain settlement contract consuming this result could be defrauded by a double-spend attack against a zero-confirmation transaction.

### Likelihood Explanation

The entry path is direct and requires no privilege: any NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with `confirmations = 0`. The attack is realistic in any integration where the `confirmations` value is derived from user input or a configurable parameter that an attacker can influence. The missing check is a single missing `require!` statement, making it easy to overlook and easy to exploit.

### Recommendation

**Short term:** Add an explicit lower-bound guard at the start of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "confirmations must be at least 1");
```

**Long term:** Extend the test suite to cover adversarial boundary values for all numeric parameters in `ProofArgs` and `ProofArgsV2`, including `confirmations = 0`, `tx_index = u64::MAX`, and empty `merkle_proof` with `confirmations = 0`.

### Proof of Concept

1. Deploy the contract with any valid genesis block and `gc_threshold >= 1`.
2. Submit one block header (the tip, height = genesis + 1).
3. Call `verify_transaction_inclusion_v2` with:
   - `tx_block_blockhash` = the tip block hash
   - `confirmations = 0`
   - A valid Merkle proof for any transaction in that block
4. The upper-bound check passes: `0 <= gc_threshold`.
5. The depth check passes: `(tip_height - tip_height + 1) = 1 >= 0`.
6. The function returns `true` for a transaction with zero confirmations — a transaction that can be reorganized away in the next block.

The root cause is the missing `require!(args.confirmations >= 1, ...)` guard, directly parallel to the missing `require!(threshold >= 1, ...)` in the multisig report.
