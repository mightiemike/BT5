### Title
ML-DSA-65 Gas-Key Host Functions Use Wire Length Instead of Trie-ID Length for Exec-Fee Calculation, Causing Massive Overcharge and Potential Out-of-Gas for Contract-Initiated PQ Gas Keys — (`runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call` host functions pass the raw wire length of the public key (`public_key_len as usize`, which is 1953 bytes for ML-DSA-65) to `gas_key_add_key_exec_fee`. The correct value is the on-trie identifier length (`trie_id_len()` = 33 bytes for ML-DSA-65). The transaction-conversion path (`permission_exec_fees`) already uses `public_key.trie_id_len()` correctly. The two paths diverge by exactly 1920 bytes per nonce for every ML-DSA-65 gas key added via a contract call, producing a fee that is ~59× too large per nonce and can cause out-of-gas failures for any contract that tries to add a PQ gas key.

---

### Finding Description

When the `PostQuantumSignatures` and `GasKeys` protocol features activate (both at protocol version 85), ML-DSA-65 access keys are stored in the trie by their 32-byte SHA3-256 hash (33 bytes including the type tag), not by the 1952-byte raw public key. This is the `trie_id_len()` / `PublicKeyHandle::MlDsa65` design.

The runtime's **transaction-conversion path** correctly accounts for this:

```rust
// runtime/runtime/src/config.rs  (permission_exec_fees)
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // ← 33 for ML-DSA-65
    gas_key_info.num_nonces,
);
``` [1](#0-0) 

The **host-function path** used by contracts does not:

```rust
// runtime/near-vm-runner/src/logic/logic.rs
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // ← 1953 for ML-DSA-65 (wire length, not trie id)
    num_nonces,
);
``` [2](#0-1) 

The same bug appears in the Wasmtime runner: [3](#0-2) 

`gas_key_add_key_exec_fee` uses the supplied key length to compute the per-nonce trie-key size (`account_id_len + public_key_len + size_of::<NonceIndex>()`). For ML-DSA-65 the actual trie key is 33 bytes wide, but the host function supplies 1953, inflating the per-nonce fee by 1920 bytes × `per_byte_fee` × `num_nonces`.

The actual trie key length is defined as:

```rust
// core/primitives/src/trie_key.rs
pub fn gas_key_nonce_key_len(account_id: &AccountId, key_handle: &PublicKeyHandle) -> usize {
    access_key_key_len(account_id.len(), key_handle.trie_id_len()) + size_of::<NonceIndex>()
}
``` [4](#0-3) 

The design document explicitly calls out that callers must use `trie_id_len()` rather than `len()`:

> **`PublicKey::trie_id_len()` is a new contract** that all storage-cost code must respect. Callers that still use `len()` for trie-storage costing will misprice ML-DSA-65 keys by ~1900 bytes. [5](#0-4) 

The host functions received `public_key_len` as the raw borsh-encoded wire length (1 tag byte + 1952 raw key bytes = 1953), which is what the contract passes in. They never convert it to the on-trie identifier length before forwarding it to the fee function.

---

### Impact Explanation

For every ML-DSA-65 gas key added by a contract via `promise_batch_action_add_gas_key_with_full_access` or `promise_batch_action_add_gas_key_with_function_call`:

- The exec fee is computed using 1953 bytes instead of 33 bytes per nonce.
- Overcharge per nonce ≈ 1920 bytes × `per_byte_storage_fee`.
- With `num_nonces` up to `u16::MAX` (65535), the total overcharge can be enormous.
- If the contract's attached gas does not cover the inflated fee, the host function returns `GasExceeded` and the entire promise batch fails — making ML-DSA-65 gas keys **unreachable via contract calls** in practice.
- The same operation submitted as a direct transaction succeeds and is priced correctly, creating a broken invariant between the two code paths.

---

### Likelihood Explanation

Both `GasKeys` and `PostQuantumSignatures` stabilized at protocol version 85 (the current stable version). Any contract that attempts to programmatically add an ML-DSA-65 gas key — a use case explicitly supported by the feature — will hit this overcharge. The trigger is entirely user-controlled (submit a contract call with an ML-DSA-65 pubkey to the gas-key host function). No privileged role is required.

---

### Recommendation

Replace `public_key_len as usize` with `public_key.trie_id_len()` in both host-function call sites, after the public key has been decoded:

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key.trie_id_len(),   // ← use trie id length, not wire length
    num_nonces,
);
```

Apply the same fix in `runtime/near-vm-runner/src/wasmtime_runner/logic.rs` for both `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`.

---

### Proof of Concept

1. Deploy a contract that calls `promise_batch_action_add_gas_key_with_full_access` with an ML-DSA-65 public key (1953 bytes) and `num_nonces = 100`.
2. Observe that the exec fee charged is computed with key length 1953 instead of 33, inflating the nonce-storage component by `100 × 1920 × per_byte_fee`.
3. Submit the identical `AddKey(GasKeyFullAccess)` action as a direct transaction with the same ML-DSA-65 key and `num_nonces = 100`.
4. Observe that the transaction-conversion fee uses `trie_id_len()` = 33 and is ~59× cheaper for the nonce component.
5. The divergence is exactly `(1953 - 33) × num_nonces × per_byte_fee` gas units.

The exact divergent values are: wire length = 1953 (`PublicKey::MLDSA65` borsh encoding), trie id length = 33 (`[tag=3] || sha3_256(domain || raw_pubkey)`). [6](#0-5) [7](#0-6)
