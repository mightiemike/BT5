### Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any User to Bypass the Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool. The pool always sets `sender = msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the end-user. The allowlist therefore checks whether the router is permitted, not whether the individual user is permitted. Any user can bypass a curated pool's swap allowlist by calling through the router.

---

### Finding Description

**Pool passes its own `msg.sender` as `sender` to the extension:** [1](#0-0) 

```solidity
_beforeSwap(
  msg.sender,   // ← always the immediate caller of pool.swap()
  recipient,
  ...
);
```

**Extension checks `sender` (the pool's `msg.sender`) against the allowlist:** [2](#0-1) 

```solidity
function beforeSwap(address sender, ...)
  external view override returns (bytes4)
{
  if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
  }
  ...
}
```

Here `msg.sender` is the pool (correct), and `sender` is what the pool forwarded — the pool's own `msg.sender`.

**Router calls `pool.swap()` directly, making itself the pool's `msg.sender`:** [3](#0-2) 

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
  );
```

The router stores the real user in transient callback context (for payment), but calls `pool.swap()` directly. The pool therefore sees `msg.sender = router`, and the extension receives `sender = router`.

The allowlist check becomes:
```
allowedSwapper[pool][router]   // router address, not the end-user
```

The pool admin faces an impossible choice:
- **Do not allowlist the router** → all router-mediated swaps fail, even for legitimately allowlisted users.
- **Allowlist the router** → every user on the network can bypass the allowlist by routing through the router.

The same structural problem exists for `exactInput` (multi-hop) and `exactOutputSingle`/`exactOutput`. [4](#0-3) 

For multi-hop `exactInput`, intermediate hops use `address(this)` (the router) as payer, making the router the pool's `msg.sender` for every hop.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, whitelisted market makers) provides **zero protection** against any user who calls through `MetricOmmSimpleRouter`. The attacker does not need any special privilege — the router is a public, permissionless contract. The allowlist policy is completely bypassed, allowing unauthorized users to trade against the pool's liquidity, which constitutes a broken core pool functionality and a direct policy violation with fund-impacting consequences (unauthorized parties drain LP value through swaps the pool admin explicitly intended to block).

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entry point for the protocol. Any user who knows the pool has a swap allowlist can trivially route through the router to bypass it. No special knowledge, capital, or timing is required. The bypass is deterministic and repeatable on every block.

---

### Recommendation

The extension must gate on the **economically relevant actor** — the end-user — not the immediate caller of `pool.swap()`. Two approaches:

1. **Pass the original user through the extension payload**: The router encodes `msg.sender` into `extensionData`, and the `SwapAllowlistExtension` decodes and checks it. This requires a protocol-level convention for the payload format.

2. **Check `recipient` instead of `sender`**: For single-hop swaps the recipient is often the user, but this breaks for multi-hop paths where intermediate recipients are the router itself.

3. **Preferred — router-aware allowlist**: The extension checks `sender` first; if `sender` is a registered router, it decodes the real user from `extensionData` and checks that address instead. The factory registry of approved routers makes this trustworthy.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, alice, true)
  - Pool admin does NOT allowlist bob or the router

Attack:
  1. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient=bob, ...)
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → false
     → revert NotAllowedToSwap()

  Now pool admin allowlists the router so alice can use it:
  5. Pool admin calls setAllowedToSwap(pool, router, true)
  6. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  7. Router calls pool.swap(recipient=bob, ...)
  8. Pool calls _beforeSwap(sender=router, ...)
  9. SwapAllowlistExtension checks allowedSwapper[pool][router] → true → PASSES
  10. bob's swap executes successfully despite bob never being allowlisted
```

The allowlist is either broken for all router users (step 4) or bypassed by all users (step 10). There is no configuration that enforces per-user allowlisting through the router. [5](#0-4) [6](#0-5) [3](#0-2)
