### Title
Missing `safeApprove` Wrapper in `DirectDepositV1.creditDeposit()` Causes Permanent Deposit Lock for Non-Standard ERC20 Tokens — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` using a raw high-level ABI call on the custom `IIERC20Base` interface. The same contract implements a `safeTransfer()` helper that uses a low-level `.call()` to tolerate tokens that do not return a `bool` from `transfer()`, but no equivalent `safeApprove` exists anywhere in the codebase. For any ERC20 token that does not return a `bool` from `approve()` (e.g., USDT on certain chains), the high-level call will revert on ABI decoding, permanently breaking the deposit credit path for that token in that `DirectDepositV1` instance.

---

### Finding Description

`DirectDepositV1.creditDeposit()` is a publicly callable function (no access modifier) that iterates over all spot product tokens, approves the endpoint for the full balance, and calls `depositCollateralWithReferral`. [1](#0-0) 

The `approve` at line 92 is a direct high-level Solidity call against the `IIERC20Base` interface, which declares `approve` as returning `bool`: [2](#0-1) 

When the underlying token does not return any data from `approve()` (non-standard ERC20), the Solidity ABI decoder attempts to decode an empty return buffer as `bool` and reverts. This is the canonical USDT-class failure mode.

By contrast, the same contract already implements a `safeTransfer` helper using a low-level `.call()` with the `data.length == 0 || abi.decode(data, (bool))` guard: [3](#0-2) 

The `ERC20Helper` library used elsewhere in the protocol provides `safeTransfer` and `safeTransferFrom` with the same low-level pattern, but **no `safeApprove` equivalent exists**: [4](#0-3) 

A secondary instance exists in `ContractOwner.wrapVaultAsset()` (also no access control), which calls `assetToken.approve(tokenAddr, 0)` and `assetToken.approve(tokenAddr, assetBalance)` directly on `IERC20Base` without a safe wrapper: [5](#0-4) 

---

### Impact Explanation

A user sends a non-standard ERC20 token (one whose `approve()` emits no return value) to a `DirectDepositV1` address. Every subsequent call to `creditDeposit()` reverts at the `approve` line. The tokens accumulate in the `DirectDepositV1` contract and cannot be credited to the subaccount. Recovery requires the `ContractOwner` multisig to call `withdrawFromDirectDepositV1()`, meaning the user's collateral is locked until privileged intervention. The subaccount never receives the deposit, so any positions or margin requirements dependent on that collateral are unmet.

A second scenario: if `balance > type(uint128).max`, the `depositCollateralWithReferral` call uses `uint128(balance)` (truncated), but the approval was for the full `uint256 balance`. The residual non-zero allowance means that on the next `creditDeposit()` call, a USDT-like token (which reverts on non-zero-to-non-zero `approve`) will revert, permanently breaking the deposit path for that token in that DDA instance.

---

### Likelihood Explanation

`creditDeposit()` carries no access control and is the primary mechanism for crediting deposits in the `DirectDepositV1` flow. Any token added to the spot engine that does not conform to the ERC20 `bool`-return convention will trigger this silently. The protocol already handles this correctly for `transfer` (via `safeTransfer`) but omits the same protection for `approve`, making this an inconsistency that is easy to overlook when onboarding new collateral tokens.

---

### Recommendation

Add a `safeApprove` helper to `ERC20Helper` (or inline it in `DirectDepositV1`) using the same low-level `.call()` pattern already used for `safeTransfer`:

```solidity
function safeApprove(IERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Replace the raw `token.approve(address(endpoint), balance)` in `DirectDepositV1.creditDeposit()` with this safe variant. Apply the same fix to `ContractOwner.wrapVaultAsset()` for the `assetToken.approve` calls.

---

### Proof of Concept

1. Deploy or identify a `DirectDepositV1` instance for a subaccount that holds USDT (or any ERC20 whose `approve()` returns no value).
2. Call `creditDeposit()` on the `DirectDepositV1` contract from any EOA (no access control).
3. Execution reaches line 92: `token.approve(address(endpoint), balance)`.
4. The Solidity runtime issues a high-level call expecting a `bool` return. USDT returns no data.
5. ABI decoding of the empty return buffer as `bool` reverts.
6. The entire `creditDeposit()` call reverts; the token balance remains in the `DirectDepositV1` contract.
7. Every future call to `creditDeposit()` for this token also reverts.
8. The subaccount never receives the collateral credit; the user's funds are locked until the `ContractOwner` multisig manually calls `withdrawFromDirectDepositV1()`. [1](#0-0) [6](#0-5)
