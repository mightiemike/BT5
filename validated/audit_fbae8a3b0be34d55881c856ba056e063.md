### Title
Depleted `RewardsPoolName` Module Account Permanently Blocks Undelegation and Reward Claims - (`x/tieredrewards/keeper/claim_rewards.go`, `x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

The `x/tieredrewards` module's `processEventsAndClaimBonus` function checks the `RewardsPoolName` module account balance before paying bonus rewards and reverts with `ErrInsufficientBonusPool` if insufficient. The pool is funded only at genesis and is continuously drained by the BeginBlocker (`topUpBaseRewards`) and by bonus reward payouts. There is no on-chain mechanism to automatically refill it. Once depleted, every user operation that internally calls `claimRewards` — including `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, and `ExitTierWithDelegation` — fails for any user with accrued non-zero bonus rewards, permanently locking their staked funds.

---

### Finding Description

`processEventsAndClaimBonus` computes the bonus owed to a position and then calls `sufficientBonusPoolBalance` before transferring from the `RewardsPoolName` module account: [1](#0-0) 

`sufficientBonusPoolBalance` reads the live balance of the module account and returns `ErrInsufficientBonusPool` if it cannot cover the bonus: [2](#0-1) 

The pool is drained from two directions every block and on every claim:

1. **BeginBlocker** (`topUpBaseRewards`) sends from `RewardsPoolName` to the distribution module every block where fee income falls short of `TargetBaseRewardsRate`: [3](#0-2) 

2. **`processEventsAndClaimBonus`** sends from `RewardsPoolName` directly to position owners: [4](#0-3) 

There is no `MsgFundRewardsPool` message, no automatic mint-to-pool path, and no other code path that sends tokens *into* `RewardsPoolName`. The pool can only be seeded at genesis or via an out-of-band governance bank send.

Critically, `claimRewards` is called as a mandatory sub-step inside five state-mutating messages: [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) 

The BeginBlocker, by contrast, handles an empty pool gracefully (logs and returns `nil`): [10](#0-9) 

This asymmetry means the BeginBlocker silently drains the pool to zero, after which every user with accrued bonus rewards is hard-blocked from undelegating or exiting.

---

### Impact Explanation

Once `RewardsPoolName` is depleted:

- `MsgTierUndelegate` fails — users cannot undelegate their staked tokens.
- `MsgTierRedelegate` fails — users cannot move stake to a different validator.
- `MsgExitTierWithDelegation` fails — users cannot exit their tier position with their delegation.
- `MsgAddToTierPosition` fails — users cannot add to an existing position.
- `MsgClearPosition` fails — users cannot clear a triggered exit.
- `MsgClaimTierRewards` fails — users cannot claim any rewards.

Any user who has accrued non-zero bonus rewards (i.e., any delegated position in a bonded validator after any time has passed) is affected. Their staked principal is locked in the protocol with no exit path until governance manually refunds the pool. This directly breaks the solvency and liveness invariants of the tiered rewards system.

---

### Likelihood Explanation

The pool is drained every block by `topUpBaseRewards` whenever fee income is below `TargetBaseRewardsRate`. On a chain with low transaction volume (common in early operation), this shortfall occurs every block, draining the pool at a rate proportional to total bonded stake. The pool will reach zero within a predictable number of blocks after genesis. No attacker action is required; normal chain operation causes the depletion. Any user who then tries to undelegate after the pool is empty triggers the failure.

---

### Recommendation

1. **Decouple bonus reward payment from mandatory claim steps**: Make `processEventsAndClaimBonus` skip payment (rather than revert) when the pool is insufficient, recording the owed amount as a claimable debt. This unblocks `TierUndelegate`, `TierRedelegate`, and `ExitTierWithDelegation`.
2. **Add an automatic refill path**: Wire the inflation module or a governance `MsgFundRewardsPool` message to top up `RewardsPoolName` so it cannot be silently drained to zero.
3. **Align BeginBlocker and claim-path error handling**: Both should either hard-fail or both should degrade gracefully; the current asymmetry is the root cause of the lock-up.

---

### Proof of Concept

1. Deploy the chain with a finite genesis allocation to `RewardsPoolName`.
2. A user calls `MsgLockTier` to create a delegated position in a bonded validator.
3. Wait for enough blocks that `topUpBaseRewards` drains `RewardsPoolName` to zero (guaranteed when fee income < `TargetBaseRewardsRate`).
4. After at least one block of bonded time, the user's position has accrued non-zero `totalBonus` (computed in `processEventsAndClaimBonus`).
5. The user submits `MsgTierUndelegate`. The call chain is:

   `TierUndelegate` → `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → returns `ErrInsufficientBonusPool` → entire tx reverts.

6. The user's staked tokens remain locked. `MsgTierRedelegate` and `MsgExitTierWithDelegation` fail identically. The user has no exit path until governance refunds the pool.

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```
