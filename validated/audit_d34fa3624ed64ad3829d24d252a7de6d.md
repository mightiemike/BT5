### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Security Check Enforced by `verify_transaction_inclusion_v2` - (File: `contract/src/lib.rs`)

### Summary
The contract exposes two public entry points for transaction inclusion verification. `verify_transaction_inclusion_v2` enforces a coinbase merkle proof check to prevent the 64-byte transaction Merkle proof forgery attack. The original `verify_transaction_inclusion` (v1) skips this check entirely and remains fully callable on-chain. Any NEAR caller can invoke v1 directly to obtain a `true` verification result for a forged proof, bypassing the security fix that v2 was introduced to provide.

### Finding Description

The contract comment on `verify_transaction_inclusion` explicitly states the security motivation for v2:

> "Use `verify_transaction_inclusion_v2` instead, which includes coinbase merkle proof validation to mitigate the 64-byte transaction Merkle proof forgery vulnerability"

`verify_transaction_inclusion_v2` enforces this by computing the coinbase merkle root and requiring it to match the block's `merkle_root` before proceeding: [1](#0-0) 

It then delegates to v1 via `self.verify_transaction_inclusion(args.into())`: [2](#0-1) 

However, v1 is still a fully public, `#[pause]`-gated contract method: [3](#0-2) 

The `#[deprecated]` Rust attribute is a compiler lint only. It generates a warning for Rust callers at compile time but does **not** restrict on-chain access. Any NEAR account can call `verify_transaction_inclusion` directly via a NEAR RPC transaction, completely skipping the coinbase proof check.

v1 performs only a basic Merkle root comparison with no coinbase validation: [4](#0-3) 

The `ProofArgs` struct accepted by v1 contains no `coinbase_tx_id` or `coinbase_merkle_proof` fields, so the check cannot be applied even in principle: [5](#0-4) 

This is the direct analog to H-01: two code paths for the same logical operation (transaction inclusion verification) apply inconsistent validation rules. v2 applies the full security check; v1 applies none. An attacker selects the lenient path.

### Impact Explanation

The 64-byte transaction forgery attack (https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to craft an input that is accepted as a valid Merkle proof for a transaction that was never included in a Bitcoin block. By calling `verify_transaction_inclusion` directly with such a crafted `ProofArgs`, the attacker receives `true` from the contract. Any downstream protocol (bridge, cross-chain asset release, conditional payment) that trusts this return value will act on a false Bitcoin transaction confirmation, enabling theft or unauthorized state transitions.

**Impact:** High — forged transaction inclusion proofs accepted as valid.
**Likelihood:** High — the entry point is public, requires no privilege, and the forgery technique is well-documented.

### Likelihood Explanation

The attack requires only:
1. Constructing a 64-byte input that collides with an internal Merkle tree node in a real Bitcoin block (a known, documented technique).
2. Calling `verify_transaction_inclusion` on the NEAR contract with that input and the corresponding real block hash.

No privileged role, leaked key, or social engineering is needed. The function is callable by any NEAR account.

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface entirely, or gate it with an access-control role that prevents unprivileged callers from invoking it. The `#[deprecated]` attribute alone provides no on-chain protection. Since `verify_transaction_inclusion_v2` already calls v1 internally after performing the coinbase check, the safest fix is to make v1 `pub(crate)` or `private`, ensuring all external callers are forced through the v2 path.

### Proof of Concept

1. Identify a real Bitcoin block whose `merkle_root` is already stored in the contract (i.e., the block is on the tracked mainchain).
2. Construct a 64-byte value `fake_tx_id` such that `compute_root_from_merkle_proof(fake_tx_id, index, proof) == block.merkle_root` using the known 64-byte Merkle forgery technique (treating an internal node hash as a leaf).
3. Call `verify_transaction_inclusion` on the NEAR contract with:
   - `tx_id = fake_tx_id`
   - `tx_block_blockhash = <real block hash>`
   - `tx_index = <crafted index>`
   - `merkle_proof = <crafted proof>`
   - `confirmations = 1`
4. The contract returns `true` at line 318–322 because the Merkle root matches, with no coinbase proof check performed. [6](#0-5) 

A recipient contract consuming this `true` result would incorrectly treat the forged transaction as confirmed on Bitcoin.
