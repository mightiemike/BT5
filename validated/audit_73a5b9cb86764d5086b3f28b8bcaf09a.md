### Title
Division by Zero in `PerpEngine.socializeSubaccount` When `openInterest == 0` — (`File: core/contracts/PerpEngine.sol`)

### Summary

`PerpEngine.socializeSubaccount` divides by `state.openInterest` to compute a per-share funding adjustment when socializing a negative `vQuoteBalance`. If the liquidatee was the sole remaining participant in a perp market (or all other participants have closed their positions), `openInterest` is zero at the time of socialization, causing an unconditional revert with "DBZ". This permanently blocks finalization of the insolvent subaccount, preventing the protocol from clearing bad debt and updating the insurance fund.

### Finding Description

In `ClearinghouseLiq._finalizeSubaccount`, before calling `perpEngine.socializeSubaccount`, the code requires that all of the liquidatee's perp position amounts are zero:

```solidity
require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
``` [1](#0-0) 

Once those positions are closed, the liquidatee's contribution to `openInterest` has already been subtracted. If the liquidatee was the only open-interest participant in a given perp product, `state.openInterest` is now `0` for that product.

`perpEngine.socializeSubaccount` is then called. It calls `getStateAndBalance`, which applies `_updateBalance(state, balance, 0, 0)`. With `balance.amount == 0`, this is a no-op on `openInterest`, so `state.openInterest` remains `0`. The function then checks `balance.vQuoteBalance < 0` — which can be true because the closed position may have accumulated unrealized losses — and reaches:

```solidity
int128 fundingPerShare = -balance.vQuoteBalance.div(
    state.openInterest
);
``` [2](#0-1) 

`MathSD21x18.div` enforces `require(y != 0, ERR_DIV_BY_ZERO)`:

```solidity
function div(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        require(y != 0, ERR_DIV_BY_ZERO);
``` [3](#0-2) 

The call reverts unconditionally. There is no guard on `state.openInterest` before the division, and no early-exit path for the case where `openInterest == 0`.

The full call chain is:

`Endpoint → Clearinghouse.liquidateSubaccount → (delegatecall) ClearinghouseLiq.liquidateSubaccountImpl → _finalizeSubaccount → perpEngine.socializeSubaccount → DIV BY ZERO` [4](#0-3) [5](#0-4) 

### Impact Explanation

When the revert occurs, the entire `liquidateSubaccountImpl` transaction fails. The insolvent subaccount cannot be finalized: its bad debt is never socialized, the insurance fund is never updated, and the subaccount's accounting state is permanently stuck. Any subsequent attempt to finalize the same subaccount will hit the same revert as long as `openInterest` for that product remains zero. This is a solvency/accounting corruption: bad debt that should be absorbed by the insurance fund or socialized across remaining participants is instead frozen in an unresolvable state.

### Likelihood Explanation

The condition is reachable whenever a perp product has only one active participant (the liquidatee) at the time of finalization. This is realistic for newly listed or thinly traded perp markets, or when all other participants have closed their positions before the liquidation is finalized. No special privileges are required; any user can be the sole participant in a perp market through normal trading activity.

### Recommendation

Before dividing by `state.openInterest`, add a guard that skips the socialization step when `openInterest == 0`. If there are no other participants to socialize the loss to, the remaining negative `vQuoteBalance` should be fully absorbed by the insurance fund (or written off), and `balance.vQuoteBalance` should be set to zero:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No participants to socialize to; absorb via insurance or write off
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

### Proof of Concept

1. Deploy the protocol with a perp product (e.g., `productId = 2`).
2. Alice opens a long perp position; no other user opens a position in this product. `openInterest = Alice.amount`.
3. The price moves against Alice; her `vQuoteBalance` becomes negative.
4. Alice's account becomes insolvent (maintenance health < 0).
5. The sequencer submits a `LiquidateSubaccount` transaction to close Alice's perp position (`amount = 0` after liquidation). `openInterest` drops to `0`.
6. The sequencer submits a second `LiquidateSubaccount` with `productId = type(uint32).max` to finalize Alice.
7. `_finalizeSubaccount` passes the `balance.amount == 0` check, then calls `perpEngine.socializeSubaccount`.
8. Inside `socializeSubaccount`, `balance.vQuoteBalance < 0` is true and `state.openInterest == 0`.
9. `(-balance.vQuoteBalance).div(0)` reverts with `"DBZ"`.
10. Alice's subaccount can never be finalized; bad debt is permanently frozen.
