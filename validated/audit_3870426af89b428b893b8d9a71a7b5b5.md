### Title
Malicious P2P Peer Can Poison Transaction Hash Index via Unverified `transaction_hash` Field in `TryFrom<protobuf::TransactionInBlock>` - (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)` converter blindly trusts the `transaction_hash` field supplied by a remote peer. The p2p sync client then stores this peer-controlled value directly into the local transaction index without ever recomputing or verifying the hash against the transaction body. A malicious peer can therefore cause a syncing node to store an arbitrary `TransactionHash` for any transaction, permanently poisoning the local index and causing `starknet_getTransactionByHash` (and related RPC methods) to return authoritative-looking but incorrect data.

---

### Finding Description

**Step 1 – Converter accepts peer-supplied hash verbatim.**

In `TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)`: [1](#0-0) 

The `tx_hash` is extracted directly from `value.transaction_hash` (a field the remote peer controls). The transaction body is decoded separately into `transaction`, but **no step computes `transaction.calculate_transaction_hash(...)` and compares it to `tx_hash`**. The function returns `Ok((transaction, tx_hash))` at line 183 with the peer-supplied hash intact. [2](#0-1) 

**Step 2 – The deserialization chain propagates the tainted hash.**

`TryFrom<protobuf::TransactionWithReceipt> for FullTransaction` calls the above converter and places the result directly into `FullTransaction { transaction, transaction_output, transaction_hash }`: [3](#0-2) 

`TryFrom<protobuf::TransactionsResponse> for DataOrFin<FullTransaction>` then wraps this into the type consumed by the p2p sync client: [4](#0-3) 

**Step 3 – The p2p sync client stores the tainted hash with an explicit TODO acknowledging the missing check.**

In `TransactionStreamFactory::parse_data_for_block`: [5](#0-4) 

The comment `// TODO(eitan): Validate transaction hash from untrusted sources` is a developer acknowledgment that this validation is intentionally deferred and currently absent. The peer-supplied `transaction_hash` is pushed directly into `block_body.transaction_hashes`.

**Step 4 – The block body (with the tainted hash) is committed to storage.** [6](#0-5) 

`append_body` writes the `BlockBody` — including `transaction_hashes` — to the LMDB storage. The storage also builds a reverse index mapping each `TransactionHash` to its `TransactionIndex`.

**Step 5 – RPC serves the tainted hash as authoritative.**

`get_transaction_by_hash` looks up the transaction by the stored (attacker-supplied) hash and returns it: [7](#0-6) 

`get_block_w_full_transactions` zips stored transactions with stored hashes and returns them: [8](#0-7) 

**No block-hash cross-check closes the gap.** The block header stores a `transaction_commitment` (a Merkle root over transaction hashes), but the p2p sync client never verifies that the received transaction hashes match this commitment before writing them to storage. The header sync and transaction sync are independent pipelines. [9](#0-8) 

---

### Impact Explanation

A malicious peer sends a `TransactionsResponse` protobuf message containing a valid transaction body but with `transaction_hash` set to an arbitrary `Felt` value (e.g., `0xdeadbeef...`). After deserialization and storage:

1. The local node's transaction index maps `0xdeadbeef...` → the transaction, not the correct hash.
2. `starknet_getTransactionByHash(correct_hash)` returns `TRANSACTION_HASH_NOT_FOUND`.
3. `starknet_getTransactionByHash(0xdeadbeef...)` returns the transaction body, presenting a hash-content binding that is false.
4. `starknet_getBlockWithTxs` returns the block with all transactions carrying attacker-chosen hashes.

This directly satisfies the allowed impact: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."** and **"High. RPC execution … returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

Any node that enables p2p sync and connects to at least one malicious peer is vulnerable. No special privileges are required — the attacker only needs to be a peer in the p2p network. The attack is deterministic and requires no brute force. The TODO comment confirms the developers are aware the check is missing.

---

### Recommendation

In `parse_data_for_block` (or in the converter itself), after decoding the transaction body, recompute the transaction hash using `calculate_transaction_hash` with the appropriate chain ID and version, and reject (disconnect/penalize) the peer if the recomputed hash does not match the peer-supplied `transaction_hash`. The `validate_transaction_hash` function in `crates/starknet_api/src/transaction_hash.rs` already handles historical hash variants and is the appropriate tool: [10](#0-9) 

---

### Proof of Concept

```rust
// Pseudocode – runnable as a Rust unit test in apollo_protobuf
use apollo_protobuf::protobuf::{self, transaction_in_block};
use starknet_api::transaction::{Transaction, TransactionHash};
use starknet_api::felt;

let arbitrary_felt = felt!("0xdeadbeefdeadbeefdeadbeefdeadbeef");

// Build a minimal valid InvokeV1 body
let invoke_v1 = transaction_in_block::InvokeV1 {
    sender: Some(/* valid address protobuf */),
    max_fee: Some(/* valid felt */),
    signature: Some(protobuf::AccountSignature { parts: vec![] }),
    calldata: vec![],
    nonce: Some(/* valid felt */),
};

let tx_in_block = protobuf::TransactionInBlock {
    transaction_hash: Some(arbitrary_felt.into()),
    txn: Some(transaction_in_block::Txn::InvokeV1(invoke_v1)),
};

let (transaction, transaction_hash) =
    <(Transaction, TransactionHash)>::try_from(tx_in_block).unwrap();

// Assert: stored hash equals the attacker-supplied value
assert_eq!(transaction_hash.0, arbitrary_felt);

// Assert: recomputed hash differs from the attacker-supplied value
let chain_id = ChainId::Mainnet;
let recomputed = transaction.calculate_transaction_hash(&chain_id, &transaction.version()).unwrap();
assert_ne!(recomputed, transaction_hash); // This passes – the binding is broken
```

impl TryFrom<protobuf::TransactionsResponse> for DataOrFin<FullTransaction> {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::TransactionsResponse) -> Result<Self, Self::Error> {
        let Some(transaction_message) = value.transaction_message else {
            return Err(missing("TransactionsResponse::transaction_message"));

                tx_with_receipt,

                Ok(DataOrFin(Some(result)))
            }
            protobuf::transactions_response::TransactionMessage::Fin(_) => Ok(DataOrFin(None)),
        }
    }
