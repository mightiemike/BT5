### Title
Stale `lastLiquidationFees` Cross-Account Contamination Triggers Unnecessary Socialization — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a global storage variable that is written by every `_handleLiquidationPayment` call and never reset between accounts. When `_finalizeSubaccount` is called for account B immediately after a large liquidation of account A, it subtracts account A's fees from `v.insurance` before deciding whether to call `spotEngine.socializeSubaccount`. If the stale fee value is large enough, `v.insurance` appears insufficient even when the actual `insurance` balance fully covers account B's deficit, causing `spotEngine.socializeSubaccount` to fire and permanently reduce `cumulativeDepositsMultiplierX18` for all spot depositors.

---

### Finding Description

`lastLiquidationFees` is declared as a persistent storage slot in `ClearinghouseStorage`: [1](#0-0) 

It is written unconditionally at the end of every `_handleLiquidationPayment` call, regardless of which account is being liquidated: [2](#0-1) 

The design comment (lines 581–585) explains the intent: when insurance is insufficient to make an account healthy, the fees just added by the final liquidation step should be excluded so that socialization is not blocked. The mechanism is correct for a single-account sequence, but the variable is never scoped to the account being finalized.

In `_finalizeSubaccount`, the socialization decision is made using this global value: [3](#0-2) 

`v.insurance` is then used as the budget passed to `perpEngine.socializeSubaccount` and as the basis for the `spotEngine.socializeSubaccount` guard: [4](#0-3) 

The `lastLiquidationFees` amount is added back at line 410 before writing to storage, so the `insurance` storage slot is not permanently drained. However, the intermediate `v.insurance` value used at line 407 is already deflated, and `spotEngine.socializeSubaccount` has already been called by that point.

`spotEngine.socializeSubaccount` permanently reduces `cumulativeDepositsMultiplierX18` for every spot product where the liquidatee holds a negative balance: [5](#0-4) 

---

### Impact Explanation

When `spotEngine.socializeSubaccount(B)` fires unnecessarily:

- `cumulativeDepositsMultiplierX18` is reduced for every spot product in which account B holds a negative quote balance.
- Every depositor's effective balance is permanently diluted proportionally.
- The insurance fund retains the correct amount (the add-back at line 410 restores it), but the depositor loss is irreversible — there is no mechanism to undo a `cumulativeDepositsMultiplierX18` reduction.

The magnitude of the depositor loss equals the portion of account B's deficit that insurance could have covered but did not, because `v.insurance` was artificially deflated by the stale `lastLiquidationFees`.

---

### Likelihood Explanation

The precondition is:

1. A large liquidation of any account A sets `lastLiquidationFees` to value `L`.
2. Before any subsequent `_handleLiquidationPayment` call resets `lastLiquidationFees`, a finalization transaction for account B is processed where `insurance - L < |quoteBalance(B)| < insurance`.

This does not require a malicious liquidator. During a market stress event, multiple accounts are liquidated in rapid succession. Any liquidator who submits a large liquidation followed by a finalization of a different account in the same sequencer batch satisfies the precondition. A malicious liquidator can deliberately engineer this by choosing the order of their submitted transactions.

---

### Recommendation

`lastLiquidationFees` should only be subtracted from `v.insurance` when the finalization is for the same account whose last liquidation step set the variable. One correct fix is to also store the `liquidatee` address alongside `lastLiquidationFees` and only apply the subtraction when `txn.liquidatee == lastLiquidationFeesAccount`. Alternatively, reset `lastLiquidationFees` to zero at the start of `_finalizeSubaccount` (or after each finalization) so it cannot bleed across accounts.

---

### Proof of Concept

```
State before:
  insurance = 1_000e18
  lastLiquidationFees = 900e18   ← set by prior liquidation of account A

Account B state entering _finalizeSubaccount:
  quoteBalance.amount = -400e18  (deficit)
  all perp/spot positions already closed

Execution trace (_finalizeSubaccount for B):
  v.insurance = 1_000e18 - 900e18 = 100e18          // line 369
  v.canLiquidateMore = (-400e18 + 100e18) > 0 = false // line 370
  v.insurance = perpEngine.socializeSubaccount(B, 100e18)
              = 100e18  (no perp losses)              // line 386
  insuranceCover = min(100e18, 400e18) = 100e18       // line 395
  v.insurance = 100e18 - 100e18 = 0                  // line 400
  spotEngine.updateBalance(QUOTE, B, +100e18)         // B deficit now -300e18
  v.insurance <= 0  →  spotEngine.socializeSubaccount(B) FIRES  // line 407-408
    → cumulativeDepositsMultiplierX18 reduced by 300e18 / totalDepositsNormalized
  v.insurance += 900e18  →  insurance = 900e18        // lines 410-411

Correct behavior (if lastLiquidationFees were 0):
  v.insurance = 1_000e18
  insuranceCover = min(1_000e18, 400e18) = 400e18
  v.insurance = 600e18 > 0  →  NO socialization
  insurance = 600e18
```

Assert: `spotEngine.socializeSubaccount` was called despite `insurance (1_000e18) > deficit (400e18)`, permanently reducing `cumulativeDepositsMultiplierX18` for all spot depositors.
