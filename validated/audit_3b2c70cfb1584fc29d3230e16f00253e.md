### Title
Swapped secp256k1/secp256r1 Operator Dispatch and Cost Constants - (File: src/chia_dialect.rs, src/secp_ops.rs)

### Summary
The 4-byte opcode dispatch table in `ChiaDialect::op` maps the secp256k1 and secp256r1 verification operators to the wrong opcodes, and the cost constants `SECP256K1_VERIFY_COST` / `SECP256R1_VERIFY_COST` are swapped relative to the specification documented in the inline comment. This is a direct operator-wiring error: the wrong cryptographic verification function is invoked for each 4-byte opcode, and secp256k1_verify is undercharged by 550,000 cost units per call.

### Finding Description

In `src/chia_dialect.rs`, the comment at lines 172–174 documents the intended cost assignment for the two secp 4-byte opcodes:

```
// the secp operators have a fixed cost of 1850000 and 1300000,
// which makes the multiplier 0x1c3a8f and 0x0cf84f (there is an
// implied +1) and cost function 0
```

The comment lists costs in the same order as the match arms that follow:

```rust
0x13d61f00 => op_secp256k1_verify,   // first arm
0x1c3a8f00 => op_secp256r1_verify,   // second arm
```

Decoding the 4-byte opcode format (multiplier = top 3 bytes, cost = multiplier + 1):

| Opcode | Multiplier | Encoded cost |
|---|---|---|
| `0x13d61f00` | `0x13d61f` = 1,299,999 | **1,300,000** |
| `0x1c3a8f00` | `0x1c3a8f` = 1,849,999 | **1,850,000** |

The comment says the first operator (secp256k1_verify) costs **1,850,000** (multiplier `0x1c3a8f`). But the actual opcode wired to `op_secp256k1_verify` is `0x13d61f00`, which encodes cost **1,300,000**. The opcode that encodes cost 1,850,000 (`0x1c3a8f00`) is wired to `op_secp256r1_verify` instead.

This is confirmed by `src/secp_ops.rs`:

```rust
const SECP256R1_VERIFY_COST: Cost = 1850000;   // line 11
const SECP256K1_VERIFY_COST: Cost = 1300000;   // line 12
```

The cost constants are swapped relative to the specification: per the comment, secp256k1 should cost 1,850,000 and secp256r1 should cost 1,300,000, but the code has them reversed.

The consequence is twofold:

1. **Wrong function dispatched for each 4-byte opcode.** The comment states opcode `0x1c3a8f00` (multiplier `0x1c3a8f`) belongs to secp256k1_verify. The code dispatches it to `op_secp256r1_verify`. A CLVM coin that encodes opcode `0x1c3a8f00` with a secp256k1 pubkey/signature will have its signature verified against the P-256 (secp256r1) curve, causing a guaranteed verification failure for any valid secp256k1 credential.

2. **secp256k1_verify is undercharged.** `op_secp256k1_verify` charges `SECP256K1_VERIFY_COST = 1,300,000` instead of the specified 1,850,000 — a 42% undercharge per call.

### Impact Explanation

**Consensus divergence / undercharged execution.** Any CLVM program that uses the 4-byte opcode `0x1c3a8f00` expecting secp256k1 verification will receive secp256r1 verification instead. Valid secp256k1 signatures will be rejected; the coin is permanently unspendable. Conversely, secp256k1_verify via opcode `0x13d61f00` is undercharged by 550,000 cost units per invocation. An attacker can pack more secp256k1 verification calls into a fixed cost budget than the protocol intends, performing more elliptic-curve work per block than nodes are designed to handle.

### Likelihood Explanation

The 4-byte opcode path is reachable by any attacker who submits a CLVM spend bundle containing opcode bytes `0x13d61f00` or `0x1c3a8f00`. No special privileges are required. The undercharge is exploitable in any mode that does not set `NO_UNKNOWN_OPS` (consensus mode), and the wrong-function dispatch affects any coin that was compiled against the documented opcode assignment.

### Recommendation

Swap the two match arms in `src/chia_dialect.rs` so that opcode `0x1c3a8f00` (cost 1,850,000) dispatches to `op_secp256k1_verify` and opcode `0x13d61f00` (cost 1,300,000) dispatches to `op_secp256r1_verify`, consistent with the inline comment. Simultaneously correct the constants in `src/secp_ops.rs`:

```rust
const SECP256K1_VERIFY_COST: Cost = 1850000;
const SECP256R1_VERIFY_COST: Cost = 1300000;
```

Verify against the canonical Chia CHIP that specifies these opcode assignments before deploying.

### Proof of Concept

**Trigger path (attacker-controlled CLVM bytes):**

1. Craft a CLVM program whose operator atom is the 4-byte sequence `[0x1c, 0x3a, 0x8f, 0x00]` with arguments `(secp256k1_pubkey, msg_hash, secp256k1_sig)`.
2. Submit the spend bundle to a Chia node running this code.
3. `ChiaDialect::op` matches `op_len == 4`, reads `opcode = 0x1c3a8f00`, and dispatches to `op_secp256r1_verify` (line 177 of `chia_dialect.rs`).
4. `op_secp256r1_verify` attempts to parse the secp256k1 pubkey as a P-256 point — this fails or produces a wrong result, rejecting a legitimately signed coin.

**Undercharge path:**

1. Craft a CLVM program with N calls to opcode `0x13d61f00` (secp256k1_verify), each with a valid secp256k1 signature.
2. Each call charges `SECP256K1_VERIFY_COST = 1,300,000` instead of the intended 1,850,000.
3. Within a fixed `max_cost` budget, the attacker fits `max_cost / 1,300,000` secp256k1 verifications instead of the intended `max_cost / 1,850,000` — approximately 42% more cryptographic work per cost unit.
