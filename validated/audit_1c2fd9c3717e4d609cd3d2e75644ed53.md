### Title
`Role::UnrestrictedSubmitBlocks` Cannot Bypass Pause on `submit_blocks` — (`File: contract/src/lib.rs`)

### Summary
`Role::UnrestrictedSubmitBlocks` is documented as allowing callers to use the contract API even after the contract is paused. However, `submit_blocks` carries a bare `#[pause]` attribute with no `except` clause, so the role's bypass capability is never checked during the pause guard. Holders of `UnrestrictedSubmitBlocks` are permanently blocked from submitting headers whenever the contract is paused, making the role non-functional for its stated purpose — an exact structural analog to M-02.

### Finding Description

The `Role` enum documents `UnrestrictedSubmitBlocks` as:

> "Allows to use contract API even after contract is paused" [1](#0-0) 

The `#[trusted_relayer]` macro applied to the `impl` block lists `UnrestrictedSubmitBlocks` as a `bypass_role`, meaning holders skip the trusted-relayer stake check: [2](#0-1) 

However, `submit_blocks` is decorated with a bare `#[pause]` — no `except(roles(...))` clause: [3](#0-2) 

The `#[pause]` macro from `near-plugins` injects a guard at the top of the function body. Without an `except` clause, the guard unconditionally panics when the feature is paused, regardless of the caller's roles. Because `#[pause]` appears before `#[trusted_relayer]` in the attribute list, the pause guard fires first; the trusted-relayer bypass (and therefore the `UnrestrictedSubmitBlocks` bypass) is never reached.

Contrast this with `run_mainchain_gc`, which correctly wires its analogous role: [4](#0-3) 

`UnrestrictedRunGC` works as intended; `UnrestrictedSubmitBlocks` does not.

### Impact Explanation

When a `PauseManager` pauses the contract (a routine security response), every account holding `UnrestrictedSubmitBlocks` is blocked from calling `submit_blocks` despite the role's explicit guarantee. The Bitcoin light client stops advancing its canonical chain tip. Any downstream NEAR DApp calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` during the pause will operate on a stale chain tip, potentially accepting or rejecting proofs based on an outdated mainchain state. The role is completely inert for its documented purpose — it bypasses only the trusted-relayer stake check, not the pause guard it was designed to bypass.

**Impact: Medium**

### Likelihood Explanation

The contract already defines and grants `PauseManager` authority, and pausing is an expected operational action during incidents or upgrades. Any pause event immediately triggers the broken invariant. The `UnrestrictedSubmitBlocks` role exists precisely because operators anticipated needing to submit blocks during a pause; the misconfiguration will be discovered the first time a pause is issued and a role holder attempts to use it.

**Likelihood: Medium**

### Recommendation

Change the `#[pause]` attribute on `submit_blocks` to include an `except` clause mirroring the pattern already used on `run_mainchain_gc`:

```rust
#[payable]
#[pause(except(roles(Role::UnrestrictedSubmitBlocks, Role::DAO)))]
#[trusted_relayer]
pub fn submit_blocks(
    &mut self,
    #[serializer(borsh)] headers: Vec<BlockHeader>,
) -> PromiseOrValue<()> {
```

This makes the pause guard consistent with the role's documented semantics and with the existing `UnrestrictedRunGC` pattern.

### Proof of Concept

1. Deploy the contract with a `PauseManager` account `pm.near` and grant `UnrestrictedSubmitBlocks` to `relayer.near`.
2. `pm.near` calls `pa_pause_feature("submit_blocks")` (or the equivalent near-plugins pause entry point).
3. `relayer.near` calls `submit_blocks` with a valid batch of headers.
4. The call panics at the `#[pause]` guard before the `#[trusted_relayer]` bypass is evaluated — `relayer.near` is rejected despite holding `UnrestrictedSubmitBlocks`.
5. Calling `run_mainchain_gc` from an `UnrestrictedRunGC` holder succeeds under the same pause, confirming the asymmetry between the two roles. [5](#0-4) [6](#0-5) [4](#0-3)
