### Title
Duplicate Position ID in `MsgClaimTierRewards` Enables Double-Claiming of Bonus Rewards - (File: `x/tieredrewards/keeper/msg_server.go`)

### Summary
`ClaimTierRewards` loads all requested positions from state into a local slice **before** processing any of them. If a caller submits the same position ID twice in `msg.PositionIds`, both entries are populated with the same pre-claim stale state. The second iteration then recomputes and pays out the same bonus rewards a second time, draining the `RewardsPoolName` module account beyond what is owed.

### Finding Description

In `ClaimTierRewards` (`x/tieredrewards/keeper/msg_server.go`, lines 429–468), all positions are loaded from the KV store in a first pass and appended to a local `positions` slice: [1](#0-0) 

Only after all positions are loaded does the function call `claimRewardsAndUpdatesPositions`, which iterates over the in-memory slice: [2](#0-1) 

Inside `claimRewardsAndUpdatesPositions`, each position is processed sequentially — bonus rewards are computed from the position's `LastEventSeq` / `LastBonusAccrual` fields, paid out via `bankKeeper.SendCoinsFromModuleToAccount`, and then the updated position is written back to the KV store: [3](#0-2) 

The bonus computation and payout happen here: [4](#0-3) 

The critical flaw: if `msg.PositionIds = [X, X]`, both `positions[0]` and `positions[1]` are loaded from state with the **same stale** `LastEventSeq` and `LastBonusAccrual`. When `positions[0]` is processed, the KV store is updated (checkpoint advanced), but `positions[1]` still holds the pre-claim snapshot. The second iteration recomputes the identical bonus window and issues a second `SendCoinsFromModuleToAccount` transfer.

The `claimBaseRewards` path (which calls `distributionKeeper.WithdrawDelegationRewards`) is not double-exploitable because the distribution module itself tracks pending rewards and returns zero on the second call. However, the **bonus rewards** path is fully controlled by the stale in-memory position state and has no such external guard. [5](#0-4) 

### Impact Explanation

An attacker who owns a position with accrued bonus rewards submits `MsgClaimTierRewards` with `PositionIds = [posId, posId, posId, ...]` (N duplicates). They receive N times the bonus owed for that position in a single transaction. The `RewardsPoolName` module account is drained at N× the legitimate rate, directly stealing bonus rewards that belong to all other position holders.

**Corrupted value**: `RewardsPoolName` module account balance (bonus pool) and the per-position `LastBonusAccrual` / `LastEventSeq` checkpoints.

### Likelihood Explanation

Any position owner can trigger this with a standard `MsgClaimTierRewards` transaction — no privileged role, no leaked key, no social engineering. The only precondition is having a position with non-zero accrued bonus rewards and the pool having sufficient balance (which is the normal operating state). The attack is a single transaction.

### Recommendation

Deduplicate `msg.PositionIds` before loading positions, or reload each position from the KV store inside `claimRewardsAndUpdatesPositions` rather than operating on a pre-loaded snapshot. The simplest fix is to add a uniqueness check in `MsgClaimTierRewards.Validate()` or at the top of `ClaimTierRewards`:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, id := range msg.PositionIds {
    if _, dup := seen[id]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", id)
    }
    seen[id] = struct{}{}
}
```

Alternatively, restructure `claimRewardsAndUpdatesPositions` to re-fetch each position from the KV store inside the loop (after prior iterations have committed their updates), mirroring the check-effects-interactions pattern.

### Proof of Concept

1. Attacker holds position ID `42` with accrued bonus rewards of 1000 `basecro`.
2. Attacker submits:
   ```
   MsgClaimTierRewards{
       Owner:       "cro1attacker...",
       PositionIds: [42, 42, 42, 42, 42],
   }
   ```
3. `ClaimTierRewards` loads position `42` five times from state — all five copies carry `LastBonusAccrual = T0`, `LastEventSeq = S0`.
4. `claimRewardsAndUpdatesPositions` iterates:
   - i=0: computes 1000 `basecro` bonus (T0→now), sends to attacker, writes updated position (checkpoint T1) to KV store.
   - i=1: uses stale copy (T0, S0), computes 1000 `basecro` again, sends to attacker again.
   - i=2–4: same, three more times.
5. Attacker receives 5000 `basecro` instead of 1000, draining 4000 `basecro` from the rewards pool. [6](#0-5) [3](#0-2)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}

func (k Keeper) processEventsAndClaimBonus(ctx context.Context, pos *types.PositionState) (sdk.Coins, error) {
	// Rewards should have been claimed before undelegation
	if !pos.IsDelegated() {
		return sdk.NewCoins(), nil
	}

	valAddr, err := sdk.ValAddressFromBech32(pos.Delegation.ValidatorAddress)
	if err != nil {
		return nil, err
	}

	events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	blockTime := sdkCtx.BlockTime()

	totalBonus := math.ZeroInt()
	// Use the persisted bonded state from the last replay, not a hardcoded default.
	// This prevents overpaying bonus for unbonded gaps between claims.
	bonded := pos.LastKnownBonded
	segmentStart := pos.LastBonusAccrual

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	for _, entry := range events {
		evt := entry.Event

		if bonded {
			// Compute bonus for the bonded segment [segmentStart, eventTime]
			// using the snapshot rate at the event.
			bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
			totalBonus = totalBonus.Add(bonus)
		}

		// Update bonded state based on event type.
		switch evt.EventType {
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND:
			bonded = false
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND:
			bonded = true
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_SLASH:
			// Slash doesn't change bonded state.
		}

		segmentStart = evt.Timestamp
		pos.UpdateLastEventSeq(entry.Seq)

		// Decrement reference count.
		if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
			return nil, err
		}
	}
