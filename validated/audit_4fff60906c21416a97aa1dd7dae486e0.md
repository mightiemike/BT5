### Title
`ValidResourceBounds` Protobuf Round-Trip Collapses `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` to `L1Gas(X)`, Producing a Different Transaction Hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserialization of `ValidResourceBounds` uses a value-based heuristic (`l1_data_gas.is_zero() && l2_gas.is_zero()`) to decide which enum variant to reconstruct. This is not a lossless round-trip: a transaction carrying `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` serializes to the same protobuf bytes as `L1Gas(X)`, but deserializes back as `L1Gas(X)`. Because `get_tip_resource_bounds_hash` hashes a different number of elements for each variant, the transaction hash computed before serialization differs from the hash computed after deserialization. Any component that recomputes the hash from the deserialized object — including RPC simulation/tracing, block-sync hash verification, and fee estimation — will produce an authoritative-looking wrong value.

---

### Finding Description

**Serialization side** (`From<ValidResourceBounds> for protobuf::ResourceBounds`, lines 471–490):

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),      // always ResourceBounds::default() = 0
    l1_data_gas: Some(ResourceBounds::default().into()), // always 0
},
```

`AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` serializes identically: all three fields present, `l2_gas` and `l1_data_gas` both zero. The two variants are **indistinguishable in the wire format**.

**Deserialization side** (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, lines 417–436):

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

When both decoded fields are zero the deserializer always picks `L1Gas`, regardless of which variant was originally serialized.

**Hash divergence** (`get_tip_resource_bounds_hash`, `transaction_hash.rs` lines 188–210):

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 3 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 4 elements total
    }
});
Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
```

`AllResources(X, 0, 0)` hashes `[tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)]` — four poseidon inputs.  
`L1Gas(X)` hashes `[tip, L1_GAS_packed, L2_GAS_packed(0)]` — three poseidon inputs.  
The two hashes are **always distinct** (different-length poseidon inputs produce different outputs).

---

### Impact Explanation

Any code path that reconstructs a `ValidResourceBounds` from protobuf and then calls `calculate_transaction_hash` will compute a hash that does not match the hash computed at submission time. Concretely:

- **Block-sync / P2P**: `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` and `TryFrom<protobuf::DeployAccountV3> for DeployAccountTransactionV3` both call `ValidResourceBounds::try_from(...)`. A syncing node that recomputes the transaction hash from the deserialized struct will obtain a different value than the sequencer stored, causing hash-verification failures or silently storing a wrong hash.
- **RPC `starknet_simulateTransactions` / `starknet_estimateFee`**: These endpoints re-execute transactions from storage. If the stored `ValidResourceBounds` was collapsed to `L1Gas`, the gas-vector computation mode switches from `All` to `NoL2Gas`, changing fee calculation and the value returned by the `get_execution_info` syscall to the contract — a wrong authoritative value.
- **`get_execution_info` syscall divergence**: The contract sees a different `n_resource_bounds` (2 vs 3) and a missing `L1_DATA_GAS` entry, potentially altering contract logic and producing a different state.

This matches the allowed impact: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"** and **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

The trigger condition is a V3 transaction where `l2_gas.is_zero() && l1_data_gas.is_zero()` after protobuf decoding, meaning both `max_amount = 0` and `max_price_per_unit = 0` for each of those two resources. The gateway stateless validator enforces `l2_gas.max_price_per_unit >= min_gas_price`; if `min_gas_price > 0` on a given deployment, `l2_gas.max_price_per_unit` cannot be zero and the bug does not trigger through the normal submission path. However:

1. Deployments with `min_gas_price = 0` (testnet, local nodes, or nodes with `validate_resource_bounds: false`) accept such transactions directly.
2. The block-sync path receives transactions from peers without re-running gateway validation; a peer that produced a block under a `min_gas_price = 0` configuration can deliver such a transaction to a node running any configuration.
3. The `data_gas_accounts` backward-compatibility list already demonstrates that zero-`l1_data_gas` transactions are a known, intentional protocol state.

---

### Recommendation

Replace the value-based heuristic with an explicit discriminator. Options:

1. **Add a boolean field** `is_l1_gas_only: bool` to the `protobuf::ResourceBounds` message and set/check it during (de)serialization.
2. **Use a protobuf `oneof`** to encode `L1Gas` and `AllResources` as distinct message variants, making the wire format unambiguous.
3. **Short-term**: In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, always produce `AllResources` when all three fields are present (even if zero), and produce `L1Gas` only when `l2_gas` is absent (`None`) in the protobuf message — relying on the existing `unwrap_or_default` fallback as the discriminator.

---

### Proof of Concept

```
1. Construct AllResourceBounds { l1_gas: {max_amount:1, max_price:1}, l2_gas: {0,0}, l1_data_gas: {0,0} }.
2. Wrap in ValidResourceBounds::AllResources and call get_tip_resource_bounds_hash → hash_A (4 poseidon inputs).
3. Serialize via From<ValidResourceBounds> for protobuf::ResourceBounds.
4. Deserialize via TryFrom<protobuf::ResourceBounds> for ValidResourceBounds
   → produces ValidResourceBounds::L1Gas({max_amount:1, max_price:1}).
5. Call get_tip_resource_bounds_hash on the result → hash_B (3 poseidon inputs).
6. Assert hash_A != hash_B  ← always true; the transaction hash has changed across the protobuf boundary.
```

    type Error = ProtobufConversionError;

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
