Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the end-user, allowing any user to bypass the swap allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist, but `MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as `sender` to extensions. When swaps are routed through `MetricOmmSimpleRouter`, the pool receives the router as `msg.sender`, so the extension checks the router address rather than the end-user. Any pool admin who allowlists the router to support approved users trading through the standard periphery inadvertently opens the allowlist to every address on the network.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` verbatim as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` checks that `sender` value against `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the immediate caller of the pool: [3](#0-2) 

The same structural problem exists for `exactInput` multi-hop paths, where intermediate hops use `address(this)` (the router) as the payer: [4](#0-3) 

The extension has no mechanism to inspect the original `msg.sender` of the router call. Once the router is added to `allowedSwapper[pool]`, the check `allowedSwapper[pool][router] == true` passes for every call arriving through the router, regardless of who the actual end-user is. The `allowAllSwappers` bypass path is separate and not relevant here — the core issue is that the per-address allowlist is keyed on the wrong identity.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` is a restricted venue where only explicitly approved addresses may trade. Once the router is allowlisted (the only way to let approved users trade through the standard periphery), the restriction is silently voided for the entire public. Any address can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the router targeting the restricted pool and the extension will pass. This is a direct loss of the access-control invariant the pool admin paid to enforce, exposing LP funds to trades from counterparties the pool was designed to exclude (e.g., non-KYC'd users, sanctioned addresses, or competitors). This constitutes broken core pool functionality causing potential loss of funds and falls under the admin-boundary break category.

## Likelihood Explanation
Every pool that (a) deploys `SwapAllowlistExtension` and (b) wants its approved users to trade through the standard router must allowlist the router. This is the normal operational path for any production restricted pool — requiring every approved user to call `pool.swap` directly is not a realistic operational model. The misconfiguration is therefore not an edge case; it is the expected configuration for any pool using both the allowlist extension and the periphery router together. No special attacker capability is required beyond calling a public router function.

## Recommendation
The extension must gate the economic actor, not the immediate caller. Two viable approaches:

1. **Router-forwarded identity via `extensionData`**: The router encodes `msg.sender` (the end-user) into `extensionData` before calling `pool.swap`. The extension decodes and checks that address when `sender` is a recognized router. This requires a convention between the router and the extension, and a registry of trusted routers in the extension.

2. **Originator field in the swap call path**: Add an `originator` field to the swap parameters so the pool can pass the true initiator to extensions without relying on `tx.origin`. The extension then checks `originator` instead of `sender`.

Until fixed, pool admins must not allowlist the router on pools with `SwapAllowlistExtension` active; approved users must call `pool.swap` directly.

## Proof of Concept
```
Setup:
  pool P configured with SwapAllowlistExtension E
  pool admin calls E.setAllowedToSwap(P, alice, true)   // alice is approved
  pool admin calls E.setAllowedToSwap(P, router, true)  // router allowlisted so alice can use it

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: P, recipient: bob, ...})

  Execution trace:
    router.exactInputSingle  (msg.sender = bob)
      pool.swap(sender = router, ...)   // pool sees msg.sender = router
        E.beforeSwap(sender = router, ...)
          allowedSwapper[pool][router] == true  → passes, no revert

  Result: bob swaps on a pool he was never approved for.
```

Foundry test outline:
1. Deploy pool P with `SwapAllowlistExtension` E configured.
2. Call `E.setAllowedToSwap(P, alice, true)` and `E.setAllowedToSwap(P, router, true)`.
3. As `bob` (not in allowlist), call `router.exactInputSingle({pool: P, ...})`.
4. Assert the call succeeds (no `NotAllowedToSwap` revert), confirming bob bypassed the gate.
5. Confirm that calling `pool.swap(...)` directly as `bob` reverts with `NotAllowedToSwap`.
