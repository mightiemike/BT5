Audit Report

## Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Originating User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` as the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates the swap, that caller is the router contract, not the originating user. The allowlist therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. If the pool admin allowlists the router so that legitimate users can trade through it, every non-allowlisted user gains unrestricted swap access by routing through the same public router.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` — the immediate caller — as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that value against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without forwarding the originating user's identity:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

At the point of the pool call, `msg.sender` inside the pool is the router address. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originating_user]`. Two broken outcomes follow:

1. **Bypass**: The admin must allowlist the router for any router-mediated swap to succeed. Once the router is allowlisted, every user — including those explicitly excluded — can swap by calling the router.
2. **Lockout**: If the admin does not allowlist the router, every allowlisted user is silently blocked from using the router, forcing direct pool calls and losing slippage protection, deadline enforcement, and multi-hop routing.

`DepositAllowlistExtension` does not share this flaw because it checks the explicit `owner` parameter, which `MetricOmmPoolLiquidityAdder` preserves faithfully from the originating caller.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to KYC'd counterparties, whitelisted market makers, or protocol-controlled addresses loses that restriction entirely for any user routing through `MetricOmmSimpleRouter`. LP funds are exposed to swaps from actors the pool admin explicitly intended to exclude. This is a direct admin-boundary break and broken core pool functionality: the extension's sole purpose — gating swap access — is rendered ineffective on the standard router path.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard periphery entry point for swaps. No special privileges, flash loans, or multi-step setup are required. Any user aware of the allowlist restriction can trivially bypass it by calling `exactInputSingle` or `exactInput` on the public router instead of calling the pool directly. The bypass is reachable on every swap through the router.

## Recommendation

The allowlist must gate the economically relevant actor, not the immediate caller of `pool.swap()`.

**Option A — Pass the originating user explicitly.** Add a `swapper` parameter to `pool.swap()` (analogous to `owner` in `addLiquidity`) that the router populates with `msg.sender` before forwarding to the pool. The extension then checks that field instead of `sender`.

**Option B — Allowlist the router and enforce identity inside the router.** Gate the router itself with an on-router allowlist that verifies `msg.sender` before forwarding to the pool. This keeps the pool interface unchanged but requires the router to be trusted and non-upgradeable.

**Option C — Check `recipient` as a proxy.** If the pool interface cannot change, the extension can check `recipient` instead of `sender`. This is imperfect but closer to the economic actor for single-hop swaps.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the only allowed swapper
  allowedSwapper[pool][router] = true  // admin must allowlist router for alice to use it

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

  router calls:
    pool.swap(bob, zeroForOne, amount, limit, "", extensionData)
    // msg.sender inside pool = router

  pool calls _beforeSwap(sender=router, ...)

  extension checks:
    allowedSwapper[pool][router] == true → passes

  bob's swap executes despite not being on the allowlist
