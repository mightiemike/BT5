Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist, but the pool always passes its own `msg.sender` (the direct caller of `pool.swap()`) as `sender`. When `MetricOmmSimpleRouter` intermediates the swap, `sender` is the router's address, not the end user's. A pool admin who allowlists the router to support router-mediated swaps for approved users simultaneously opens the pool to every user on the network, because the extension has no mechanism to distinguish which end user is behind the router call.

## Finding Description
**Root cause — pool passes `msg.sender` as `sender` to the hook:**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap` with its own `msg.sender` as the first argument: [1](#0-0) 

**Extension checks only the direct pool caller:**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` = pool address and `sender` = whoever called `pool.swap()`: [2](#0-1) 

**Router calls `pool.swap()` with itself as `msg.sender`:**

`MetricOmmSimpleRouter.exactInputSingle` stores the end user's address only in transient callback context (for payment), and calls `pool.swap()` directly — making the pool see `msg.sender = router_address`: [3](#0-2) 

The end user's identity (`msg.sender` in the router) is stored via `_setNextCallbackContext` solely for payment purposes and is never forwarded to the pool or extension as the swap initiator.

**Broken invariant:**

| Call path | `sender` seen by extension | Allowlist check |
|---|---|---|
| User → `pool.swap()` directly | `user_address` | `allowedSwapper[pool][user_address]` |
| User → `router.exactInputSingle()` → `pool.swap()` | `router_address` | `allowedSwapper[pool][router_address]` |

If the pool admin allowlists the router (necessary for approved users to use router UX), the check degenerates to "is the router allowlisted?" — which is true for every user who routes through it. No existing guard in the extension or pool prevents this; the extension has no access to transient callback context and no way to recover the true end user identity.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict swaps to specific counterparties (e.g., KYC'd institutions, whitelisted market makers) is fully open to any user who routes through `MetricOmmSimpleRouter`. The pool admin cannot simultaneously allow approved users to use the router and prevent non-approved users from swapping. Unauthorized users gain full swap access to a pool whose LP providers deposited under the assumption that only approved counterparties would trade against them, exposing LP principal to adversarial flow (arbitrage, MEV, toxic order flow) that the allowlist was designed to exclude. This is a direct loss-of-LP-value impact and constitutes an admin-boundary break where an unprivileged path bypasses a pool admin's access control configuration.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary public entry point for swaps. Any pool admin who wants approved users to enjoy router UX (slippage protection, deadline checks, multi-hop) must allowlist the router address. This is the expected production configuration, making the bypass reachable by any unprivileged user with no special setup, no privileged role, and no capital requirement beyond the swap itself. The attack is repeatable indefinitely.

## Recommendation
The extension must verify the end user's identity, not the intermediary's. Two viable approaches:

1. **Pass end-user identity through `extensionData`**: The router encodes `msg.sender` (the end user) into `extensionData`; the extension decodes and checks it. The pool admin controls which router is allowlisted and can require a trusted router that always encodes the caller correctly.

2. **Two-level check**: Introduce a shared interface between the router and the extension — if `sender` is a known router, additionally verify the payer stored in the router's transient context. This requires the extension to call back into the router to retrieve the true initiator.

Option 1 is simpler and does not require changes to the pool or router's transient storage interface, only cooperation from a trusted router implementation.

## Proof of Concept
```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension
  pool admin calls: extension.setAllowedToSwap(pool, alice, true)
  pool admin calls: extension.setAllowedToSwap(pool, router, true)
    ↑ necessary so alice can use the router

Attack:
  charlie (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: charlie, ...})

  router executes (MetricOmmSimpleRouter.sol L71-80):
    _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, charlie, tokenIn)
    pool.swap(charlie, zeroForOne, amount, limit, "", extensionData)
    // msg.sender in pool = router_address

  pool executes (MetricOmmPool.sol L230-240):
    _beforeSwap(router_address, charlie, ...)

  extension checks (SwapAllowlistExtension.sol L37):
    allowedSwapper[pool][router_address] == true  ✓  → no revert

  Result: charlie swaps successfully in a pool he was never approved for.
```
