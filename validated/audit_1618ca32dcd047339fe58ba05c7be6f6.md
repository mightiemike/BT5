Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of actual user, enabling allowlist bypass — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which is `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the EOA. Allowlisting the router to enable standard swap flows grants every unprivileged user access to the restricted pool, fully defeating the allowlist. Not allowlisting the router breaks the primary swap interface for individually allowlisted users.

## Finding Description
In `MetricOmmPool.sol`, `pool.swap()` calls `_beforeSwap(msg.sender, recipient, ...)` where `msg.sender` is the direct caller of `swap()`. [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this `sender` value unchanged to each extension. [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`. [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()`, the router is `msg.sender` of that call, so `sender = router`, not the EOA user. [4](#0-3) 

The same applies to `exactInput` (multi-hop) and `exactOutputSingle`/`exactOutput`. [5](#0-4) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the actual position owner, second parameter), not `sender` (the intermediary caller, first parameter). [6](#0-5) 

The `beforeSwap` hook signature provides `recipient` as the second parameter, but `recipient` is the output destination chosen by the caller — it is not a reliable proxy for the actual user identity either.

## Impact Explanation
A pool deploying `SwapAllowlistExtension` (e.g., KYC-gated or access-restricted) faces an irreconcilable dilemma: allowlisting the router grants every unprivileged user the ability to execute swaps against the restricted pool via `MetricOmmSimpleRouter`, draining liquidity or executing trades the pool admin intended to block. Not allowlisting the router makes the standard periphery swap path unusable for individually allowlisted users. Both outcomes break the core invariant that the allowlist restricts swaps to approved addresses — constituting broken core pool functionality and potential unauthorized access to restricted liquidity.

## Likelihood Explanation
Any pool that activates `SwapAllowlistExtension` and expects users to route through `MetricOmmSimpleRouter` (the standard swap interface) is affected. The bypass requires no special privileges — any unprivileged user can call the router. The extension is a production periphery contract explicitly designed for deployment.

## Recommendation
Change `SwapAllowlistExtension.beforeSwap` to check a field representing the actual user. The simplest approach mirrors `DepositAllowlistExtension`: identify which parameter carries the true user identity. Since the `beforeSwap` hook does not currently expose the originating user, the router should encode `msg.sender` into `extensionData` and the extension should decode and verify it. Alternatively, restructure the hook interface to pass the originating user explicitly (analogous to `owner` in `beforeAddLiquidity`).

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` active on `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to permit router-based swaps.
3. Pool admin does **not** allowlist `alice` (unprivileged EOA).
4. `alice` calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
5. Router calls `pool.swap(recipient, ...)` — `msg.sender` of `pool.swap()` is the router.
6. `_beforeSwap(router, recipient, ...)` is invoked; `sender = router`.
7. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` → `true`.
8. Swap succeeds. `alice` has bypassed the allowlist without being individually approved.
