### Title
Linked Signer Can Unilaterally Replace Itself via Fast-Path `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

---

### Summary

The fast-path `LinkSigner` transaction handler in `EndpointTx.processTransactionImpl` validates the signature with `allowLinkedSigner = true`, permitting the **current linked signer** to sign and submit a `LinkSigner` transaction that replaces itself with any arbitrary address. This is an invalid-validation analog to the external report: a state-replacement operation (signer replacement) lacks the necessary check that only the subaccount owner should be authorized to mutate the linked-signer slot.

---

### Finding Description

Two separate code paths handle `LinkSigner` transactions in `EndpointTx.sol`:

**Slow-mode path** (`processSlowModeTransactionImpl`, lines 232–239):
```solidity
validateSender(txn.sender, sender);   // msg.sender must be the owner address
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```
`validateSender` enforces that only the Ethereum address embedded in `txn.sender` (i.e., the subaccount owner) can submit this transaction. This is correct.

**Fast-path** (`processTransactionImpl`, lines 576–590):
```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true              // ← allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
```

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which in turn calls `verifier.validateSignature` passing `getLinkedSigner(sender)` as the permitted signer. The verifier accepts the signature if it was produced by **either** the subaccount owner **or** the current linked signer. [1](#0-0) [2](#0-1) [3](#0-2) 

Because `allowLinkedSigner = true` is appropriate for trading operations (`WithdrawCollateral`, `MintNlp`, `BurnNlp`, `TransferQuote`) but not for the authorization-structure mutation that `LinkSigner` performs, the fast path is missing the validation that only the subaccount owner may change the linked signer. The slow-mode path enforces this correctly; the fast path does not. [4](#0-3) 

---

### Impact Explanation

A linked signer that is compromised or malicious can sign a `LinkSigner` transaction naming any attacker-controlled address as the new signer. Once the sequencer processes this transaction on-chain:

- `linkedSigners[victim_subaccount]` is overwritten with the attacker's address.
- The attacker's address is now accepted as a valid signer for all subsequent fast-path operations on that subaccount: `WithdrawCollateral`, `WithdrawCollateralV2`, `MintNlp`, `BurnNlp`, `TransferQuote`, `LiquidateSubaccount`, and order matching.
- The attacker can drain the subaccount's collateral or manipulate its positions before the owner can react.

The corrupted state is `linkedSigners[subaccount]` — a concrete, persistent on-chain mapping that governs signing authority over all assets in the subaccount. [5](#0-4) 

---

### Likelihood Explanation

Linked signers are the standard mechanism for automated trading bots in Nado. Bot keys are stored on internet-connected servers and are high-value targets. A key compromise, a malicious third-party trading service, or a rogue bot operator can all trigger this path without any additional on-chain privilege. The attacker only needs to submit a validly signed `LinkSigner` transaction to the sequencer — a normal user-facing entry point.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` case in `processTransactionImpl`, so only the subaccount owner's key can authorize a linked-signer change via the fast path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the owner may change the linked signer
);
```

This aligns the fast path with the slow-mode path's intent and removes the ability of a linked signer to mutate its own authorization slot.

---

### Proof of Concept

1. Alice (subaccount owner `0xAlice`) links signer Bob (`0xBob`) via a slow-mode `LinkSigner` transaction. `linkedSigners[Alice_subaccount] = 0xBob`.
2. Bob's key is compromised by an attacker who controls Eve (`0xEve`).
3. The attacker uses Bob's key to sign a `SignedLinkSigner` transaction: `{ sender: Alice_subaccount, signer: 0xEve, nonce: N }`.
4. The attacker submits this signed transaction to the sequencer's normal submission endpoint.
5. The sequencer calls `processTransactionImpl` on-chain with this transaction.
6. `validateSignedTx(..., true)` passes: Bob's signature is valid because `allowLinkedSigner = true` and `getLinkedSigner(Alice_subaccount) == 0xBob`.
7. `linkedSigners[Alice_subaccount] = 0xEve` is written.
8. Eve signs a `SignedWithdrawCollateralV2` transaction draining Alice's collateral. `validateSignedTx(..., true)` passes because Eve is now the linked signer. [6](#0-5) [7](#0-6)
