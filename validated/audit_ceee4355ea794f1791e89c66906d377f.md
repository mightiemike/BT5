### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Drain Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

### Summary
`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` at line 616 without checking the return value. If the USDC token returns `false` instead of reverting on a failed transfer, the function continues to withdraw usdcE from the DirectDepositV1 account and send it to the caller — without the caller having actually deposited any USDC. Any unprivileged caller on chain 57073 can drain usdcE from any DDA that holds a balance.

### Finding Description
`ContractOwner` declares `using ERC20Helper for IERC20Base` at line 24, giving it access to the `safeTransferFrom` wrapper that checks both the call success and the decoded boolean return value. However, `replaceUsdcEWithUsdc` bypasses this wrapper and calls the raw `transferFrom` directly:

```solidity
// core/contracts/ContractOwner.sol line 616
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));   // sends usdcE → ContractOwner
IERC20Base(usdcE).safeTransfer(msg.sender, balance);              // sends usdcE → caller
```

The return value of `transferFrom` is silently discarded. If it returns `false`, execution falls through to the two subsequent lines, which unconditionally move usdcE out of the DDA and into the caller's wallet.

The `ERC20Helper.safeTransferFrom` wrapper (lines 23–42 of `ERC20Helper.sol`) would have caught this by requiring `success && (data.length == 0 || abi.decode(data, (bool)))`. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
If `transferFrom` returns `false` without reverting:

1. The caller provides zero USDC.
2. `DirectDepositV1.withdraw` sends the full usdcE balance of the DDA to `ContractOwner` (ContractOwner is the owner of the DDA, so the `onlyOwner` check passes).
3. `safeTransfer` sends that usdcE to the caller.

The DDA loses its entire usdcE balance. The corrupted asset delta is the usdcE balance of the targeted DirectDepositV1 account. Because `replaceUsdcEWithUsdc` accepts any `subaccount` argument, every DDA on chain 57073 that holds usdcE is at risk. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
The function is `external` with no access-control modifier — only a `block.chainid == 57073` guard. Any wallet on the Ink chain can call it. The attack requires only that the targeted USDC implementation returns `false` on a failed transfer rather than reverting. Several ERC20 implementations (including older USDC variants and many bridged stablecoins) exhibit this behavior. The hardcoded USDC address `0x2D270e6886d130D724215A266106e6832161EAEd` is a bridged token on a relatively new chain, making its exact revert-vs-return-false behavior a deployment-time risk that the code does not guard against. [6](#0-5) 

### Recommendation
Replace the raw `transferFrom` call with the `safeTransferFrom` wrapper already available via `ERC20Helper`:

```solidity
// Before (line 616)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ERC20Helper.safeTransferFrom` uses a low-level `call` and requires both call success and a truthy decoded return value, matching the pattern used everywhere else in the codebase. [2](#0-1) 

### Proof of Concept
1. A DDA for `subaccount` holds `N` usdcE tokens.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` on chain 57073.
3. `IERC20Base(usdc).transferFrom(attacker, dda, N)` is called. USDC returns `false` (attacker has no allowance or balance). Return value is not checked; execution continues.
4. `DirectDepositV1(dda).withdraw(usdcE)` executes: usdcE balance (`N` tokens) is transferred from the DDA to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, N)` executes: `N` usdcE tokens are sent to the attacker.
6. Attacker receives `N` usdcE; DDA balance is zero; attacker paid nothing. [1](#0-0) [5](#0-4)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);

        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;

        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

**File:** core/contracts/libraries/ERC20Helper.sol (L23-42)
```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
