Audit Report

## Title
`SwapAllowlistExtension` Allowlist Bypassed via Router When Router Is Allowlisted — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so the extension checks whether the **router** is allowlisted rather than the actual end user. A pool admin who allowlists the router — the natural step to enable router-mediated access for allowlisted users — inadvertently grants unrestricted swap access to every address on the network.

## Finding Description
**Root cause — pool passes its own `msg.sender` as `sender` to the extension:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` at line 231, forwarding its own `msg.sender` (the direct caller of `pool.swap()`) as the `sender` argument. `ExtensionCalling._beforeSwap` then encodes this value and dispatches it to every configured extension via `IMetricOmmExtensions.beforeSwap(sender, ...)`.

`SwapAllowlistExtension.beforeSwap` performs:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

**Exploit path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a whitelist of addresses.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` to allow allowlisted users to reach the pool through `MetricOmmSimpleRouter`.
3. Any unprivileged user (not on the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. The router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly — making the router the `msg.sender` of `pool.swap()`.
5. The pool calls `_beforeSwap(msg.sender=router, ...)`, which calls `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
6. The extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. The actual end user's address is never checked.

**Existing guards are insufficient:** The extension has no mechanism to recover the original end user's address. The `sender` field is structurally bound to `msg.sender` of `pool.swap()` throughout the call chain (`MetricOmmPool.sol:231`, `ExtensionCalling.sol:165`), and the router does not pass the originating user's address through any other channel visible to the extension.

## Impact Explanation
The swap allowlist — the sole access-control mechanism for restricting who may trade on a pool — is completely defeated for any pool that allowlists the router. Any unprivileged user can execute swaps on a pool intended to be restricted, draining liquidity or trading at favorable oracle-derived prices against LPs who expected a closed pool. This constitutes broken core pool functionality and an admin-boundary break: the pool admin's configured access control is bypassed by an unprivileged path.

## Likelihood Explanation
The precondition (router is allowlisted) is the natural and expected operational state for any pool that uses both `SwapAllowlistExtension` and `MetricOmmSimpleRouter`. Any user who discovers the router is allowlisted can immediately exploit this with a standard router call. No special privileges, flash loans, or complex setup are required. The attack is repeatable on every block.

## Recommendation
The extension must check the true originating user, not the intermediate router. Two viable approaches:

1. **Pass originating caller through `extensionData`:** Have the router encode `msg.sender` into `extensionData` before calling `pool.swap()`, and have `SwapAllowlistExtension.beforeSwap` decode and verify it. This requires the router to be trusted to populate this field honestly (which it is, as a periphery contract).

2. **Check `recipient` instead of `sender`:** If the pool's design intent is that the recipient is the economically relevant party, gate on `recipient` rather than `sender`. However, this changes semantics and must be evaluated against the full extension design.

3. **Disallow router allowlisting:** Document that allowlisting the router address is unsafe and add a guard in `setAllowedToSwap` that rejects known router addresses.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension, only alice is allowlisted
swapAllowlist.setAllowedToSwap(pool, alice, true);
// Admin also allowlists the router to let alice use it
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attack: bob (not allowlisted) routes through the router
vm.startPrank(bob);
token0.approve(address(router), type(uint256).max);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: bob,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    extensionData: "",
    deadline: block.timestamp
}));
// Bob's swap succeeds — allowlist bypassed
```

The pool calls `_beforeSwap(msg.sender=router, ...)` at `MetricOmmPool.sol:231`, the extension checks `allowedSwapper[pool][router]` which is `true`, and bob's swap is never checked.
