### Title
Merkle Tree Second-Preimage Attack Enables Proof Forgery for Non-Existent Transactions — (File: `contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

Any unprivileged NEAR caller can invoke `verify_transaction_inclusion` (and `verify_transaction_inclusion_v2`) with a crafted Merkle proof in which `tx_id` is an internal tree node hash rather than a real transaction hash. Because `compute_root_from_merkle_proof` never validates proof depth against the actual transaction count, the function returns `true` for a transaction that does not exist. A downstream bridge or settlement contract that trusts this result will authorize a cross-chain payment for a Bitcoin transfer that never occurred.

---

### Finding Description

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` iterates over the supplied sibling hashes and hashes upward from whatever starting value is given:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;
    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
    current_hash
}
``` [1](#0-0) 

The function receives no information about how many transactions are in the block. Bitcoin block headers store only the Merkle root, not the transaction count, so the contract cannot independently derive the expected proof depth. An attacker can therefore supply an internal node hash (e.g., `H(H(T0) || H(T1))` for a 4-transaction block) as `tx_id` together with a proof of depth `D-1` instead of `D`. The computed root still equals the real Merkle root, so the equality check passes and the function returns `true`.

`verify_transaction_inclusion` (v1) is a public view function with no role restriction — only the `#[pause]` guard, which is inactive during normal operation:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    ...
    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root
}
``` [2](#0-1) 

`verify_transaction_inclusion_v2` adds a coinbase proof check intended to mitigate the 64-byte transaction forgery, but it does **not** fix the second-preimage attack. The coinbase check calls the same `compute_root_from_merkle_proof` with `coinbase_tx_id` at index 0:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

The only additional constraint is `merkle_proof.len() == coinbase_merkle_proof.len()`. An attacker satisfies this by using the same internal-node trick for both proofs: set `coinbase_tx_id = I01` (an internal node at depth 1, position 0) with `coinbase_merkle_proof = [I23]`, and set `tx_id = I01` (or any other internal node at the same depth) with `merkle_proof = [I23]`. Both proofs have length 1, both produce the real Merkle root, the length check passes, and v2 also returns `true` for a non-existent transaction.

The contract itself documents this residual risk in `CLAUDE.md`:

> **Important**: This function is vulnerable to the standard Bitcoin merkle tree second-preimage attack — it may return `true` for an internal node hash rather than a real transaction hash. Block headers do not contain the transaction count, so proof depth cannot be validated on-chain. [4](#0-3) 

---

### Impact Explanation

Any NEAR contract that acts as a bridge or settlement layer and calls `verify_transaction_inclusion` (or v2) to confirm a Bitcoin deposit before releasing funds will accept a forged proof. The attacker never needs to broadcast a real Bitcoin transaction. They observe a real block already stored in the light client, derive an internal node hash and a shortened proof offline, then call the view function to obtain a `true` result, which they relay to the bridge contract to claim the corresponding NEAR-side assets. The corrupted value is the **proof result** (`bool`) returned by the SPV verification path — the canonical chain state itself is not altered, but the authorization decision made by every downstream consumer of this API is.

---

### Likelihood Explanation

The attack requires no role, no stake, no privileged key, and no mining power. The attacker only needs:
1. A block already accepted by the light client (publicly observable on-chain).
2. Offline computation of an internal node hash and a shortened Merkle proof (trivial arithmetic).
3. A single NEAR view call to `verify_transaction_inclusion`.

The technique is a well-documented property of Bitcoin's Merkle tree design (CVE-2012-2459 family). Any bridge that integrates this contract as its SPV oracle is immediately exploitable.

---

### Recommendation

1. **Require proof depth to match `ceil(log2(tx_count))`**: The transaction count must be supplied by the caller and validated against the proof length. Because the block header does not contain the count, callers must provide it alongside a commitment (e.g., a coinbase-path that encodes the count, as used by BIP-141 witness commitments).
2. **Reject 64-byte internal nodes as leaf inputs**: Validate that `tx_id` is not 64 bytes (the size of a concatenated pair), which is the canonical signal of a second-preimage candidate.
3. **Deprecate and gate v1**: Remove or hard-disable `verify_transaction_inclusion` (v1) at the contract level rather than relying on a Rust `#[deprecated]` attribute that has no on-chain enforcement.
4. **Document the residual risk of v2 explicitly in its public interface**, not only in `CLAUDE.md`.

---

### Proof of Concept

Consider a real Bitcoin block stored in the light client with 4 transactions `[T0, T1, T2, T3]`:

```
Level 0 (leaves): H0=H(T0), H1=H(T1), H2=H(T2), H3=H(T3)
Level 1:          I01=H(H0||H1),        I23=H(H2||H3)
Level 2 (root):   R = H(I01||I23)
```

**Attack against v1 (`verify_transaction_inclusion`)**:

```
tx_id          = I01          # internal node, not a real transaction
tx_index       = 0
merkle_proof   = [I23]        # length 1 instead of 2
```

`compute_root_from_merkle_proof(I01, 0, [I23])` = `H(I01 || I23)` = `R` ✓ → returns `true`. [5](#0-4) 

**Attack against v2 (`verify_transaction_inclusion_v2`)**:

```
coinbase_tx_id        = I01   # same internal node
coinbase_merkle_proof = [I23] # length 1
tx_id                 = I01
tx_index              = 0
merkle_proof          = [I23] # length 1 — satisfies len equality check
```

Coinbase check: `compute_root_from_merkle_proof(I01, 0, [I23])` = `R` ✓ (passes `require!`).
Tx check (via v1): same computation → `true`. [6](#0-5) 

The attacker presents this `true` result to a bridge contract as proof that a Bitcoin deposit transaction `I01` was confirmed, and claims the corresponding NEAR-side funds — despite no such transaction ever existing on Bitcoin.
