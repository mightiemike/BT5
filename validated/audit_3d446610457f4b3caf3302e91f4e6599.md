### Title
Uncleared ERC20 Allowance in `creditDeposit()` Permanently Blocks USDT Deposits via Residual Allowance After Silent Return - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. `Endpoint.depositCollateralWithReferral()` contains an explicit silent-return path that skips `handleDepositTransfer` when the deposit amount is below the minimum threshold — leaving the allowance unconsumed. For USDT-like tokens that revert on `approve` when the current allowance is non-zero, this permanently breaks the `creditDeposit()` path for that token in the affected contract.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`, for each spot product the contract approves the endpoint and then calls `depositCollateralWithReferral`:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(
    subaccount,
    productId,
    uint128(balance),
    "-1"
);
``` [1](#0-0) 

No allowance reset to zero precedes the `approve` call. For USDT (and other non-standard ERC20s), calling `approve(spender, nonZeroValue)` when the current allowance is already non-zero reverts.

The residual allowance is created by a concrete path inside `Endpoint.depositCollateralWithReferral()`:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked when there are
    // multiple assets awaiting credit but one of them is below the minimum deposit amount.
    // we can just skip the deposit and continue with the next asset.
    return;
}
handleDepositTransfer(...);
``` [2](#0-1) 

When `isValidDepositAmount` returns false, the function silently returns **without** calling `handleDepositTransfer`, meaning no `transferFrom` is executed and the allowance set by `creditDeposit()` is never consumed. The USDT allowance from `DirectDepositV1` to `endpoint` is now stuck at a non-zero value.

On any subsequent call to `creditDeposit()`, the line:

```solidity
token.approve(address(endpoint), balance);
``` [3](#0-2) 

reverts for USDT because the current allowance is non-zero. The entire `creditDeposit()` call reverts, permanently blocking the deposit path for USDT in that `DirectDepositV1` instance.

`creditDeposit()` has no access control — it is callable by any external account: [4](#0-3) 

`ERC20Helper` provides `safeTransfer` and `safeTransferFrom` but no `safeApprove`, and the `IIERC20Base` interface used inside `DirectDepositV1` calls raw `approve` with no reset guard: [5](#0-4) 

---

### Impact Explanation

User USDT deposited to a `DirectDepositV1` contract becomes permanently uncreditable through the `creditDeposit()` flow. The tokens are locked in the contract and cannot be credited to the user's on-chain subaccount. The multisig owner can call `withdrawFromDirectDepositV1` to recover tokens, but the intended deposit flow is irreversibly broken for that contract instance and token. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** USDT is a standard collateral token in perpetual DEX protocols. The trigger condition — a deposit amount below the minimum — can occur naturally (user sends a small test amount) or be deliberately induced by any unprivileged attacker sending dust USDT to a victim's `DirectDepositV1` address before `creditDeposit()` is called. The `creditDeposit()` function is permissionless, so the attacker can also call it themselves to trigger the stuck-allowance state.

---

### Recommendation

Reset the allowance to zero before setting a new approval inside `creditDeposit()`:

```solidity
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
```

This mirrors the pattern already correctly applied in `ContractOwner.wrapVaultAsset()`: [7](#0-6) 

---

### Proof of Concept

1. A `DirectDepositV1` contract exists for a user's subaccount (created via `ContractOwner.createDirectDepositV1`).
2. An attacker (or the user accidentally) sends 1 wei of USDT to the `DirectDepositV1` address — an amount below `MIN_DEPOSIT_AMOUNT` / `MIN_FIRST_DEPOSIT_AMOUNT`.
3. Anyone calls `DirectDepositV1.creditDeposit()`.
4. The loop reaches the USDT product: `token.approve(endpoint, 1)` — USDT allowance is now 1.
5. `endpoint.depositCollateralWithReferral(subaccount, productId, 1, "-1")` is called.
6. Inside, `isValidDepositAmount` returns `false` (amount too small); the function silently returns without calling `handleDepositTransfer`. USDT allowance remains at 1.
7. The user later sends a legitimate amount of USDT (e.g., 100 USDT) to the same `DirectDepositV1` contract.
8. `creditDeposit()` is called again. The loop reaches USDT: `token.approve(endpoint, 100e6)` — **USDT reverts** because the current allowance (1) is non-zero.
9. The entire `creditDeposit()` transaction reverts.
10. The user's 100 USDT is stuck in the `DirectDepositV1` contract and cannot be credited to their subaccount through the normal flow. [8](#0-7)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

                uint8(TransactionType.DepositCollateral),

                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
