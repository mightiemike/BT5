### Title
Pending stop-loss parameter proposals store absolute `executeAfter` timestamps, allowing a pool admin to bypass a newly-increased timelock — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` timelocks drawdown, decay, and watermark changes so LPs can react before harmful parameters take effect. However, every proposal records an **absolute** `executeAfter = block.timestamp + currentTimelock` at proposal time. When the admin subsequently increases the timelock, all in-flight proposals retain their old, shorter absolute deadline and can be executed immediately after that shorter window — bypassing the protection the new timelock was supposed to provide.

---

### Finding Description

`_afterTimelock` computes the absolute execution deadline at proposal time:

```solidity
// OracleValueStopLossExtension.sol L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

Every propose function calls this and stores the result:

```solidity
// L106-108  (drawdown proposal)
uint32 executeAfter = _afterTimelock(pool_);
sched.pendingDrawdownE6 = uint32(newMaxDrawdownE6);
sched.pendingDrawdownExecuteAfter = executeAfter;
```

```solidity
// L80-83  (timelock proposal)
uint32 executeAfter = _afterTimelock(pool_);
sched.pendingTimelock = newTimelock;
sched.pendingTimelockExecuteAfter = executeAfter;
```

`executeOracleStopLossDrawdown` (and the analogous decay/watermark executors) only checks whether `block.timestamp >= pendingDrawdownExecuteAfter` — it never re-derives the deadline from the **current** timelock:

```solidity
// L112-120
function executeOracleStopLossDrawdown(address pool_) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    if (sched.pendingDrawdownExecuteAfter == 0) revert OracleStopLossNoPendingDrawdown(pool_);
    _requireElapsed(sched.pendingDrawdownExecuteAfter);   // ← absolute stored value only
    ...
}
```

This is structurally identical to the `StakedlvlUSD` cooldown bug: both store an absolute end-time at initiation, so a later change to the governing duration has no effect on already-queued operations.

**Attack path (pool admin, semi-trusted):**

1. Pool is deployed with `timelock = 1 day`. LPs add liquidity trusting the stop-loss.
2. Admin calls `proposeOracleStopLossDrawdown(pool, 1_000_000)` (100 % drawdown → stop-loss floor = 0, effectively disabled). `pendingDrawdownExecuteAfter = T + 1 day`.
3. Admin simultaneously calls `proposeOracleStopLossTimelock(pool, 30 days)`. `pendingTimelockExecuteAfter = T + 1 day`.
4. LPs observe the timelock-increase proposal and conclude future changes will carry 30-day protection; they do not urgently exit.
5. At `T + 1 day`, admin calls `executeOracleStopLossTimelock` (timelock → 30 days) **and** `executeOracleStopLossDrawdown` (drawdown → 100 %).
6. Stop-loss is now disabled. Adversarial swaps can drain the pool; LPs had only 1 day of real warning, not 30.

The same path applies to `proposeOracleStopLossDecay` and `proposeOracleStopLossHighWatermarks`.

---

### Impact Explanation

Disabling the drawdown guard (`drawdownE6 = E6`) sets `floorMultiplier = 0`, making `_applyWatermark` never report a breach:

```solidity
// L334
breached = metric < (hwm * floorMultiplier) / E6;  // hwm * 0 / E6 = 0, always false
```

With the stop-loss silenced, adversarial swaps can extract value from every bin without triggering `OracleStopLossTriggered`. LP principal is directly at risk.

---

### Likelihood Explanation

The pool admin is semi-trusted; the timelock exists precisely to constrain admin power. The exploit requires only two simultaneous `propose*` calls followed by two `execute*` calls after the old short window — no external oracle manipulation, no special token behavior, no privileged factory role beyond pool admin. Any pool deployed with a short initial timelock is permanently exposed to this race until the timelock is long enough that LPs can monitor and exit.

---

### Recommendation

Store the **proposal timestamp** instead of the absolute deadline, and recompute `executeAfter` from the **current** timelock at execution time:

```solidity
// In PoolStopLossSchedule, replace pendingDrawdownExecuteAfter with:
uint32 pendingDrawdownProposedAt;

// In proposeOracleStopLossDrawdown:
sched.pendingDrawdownProposedAt = uint32(block.timestamp);

// In executeOracleStopLossDrawdown:
uint32 executeAfter = sched.pendingDrawdownProposedAt
    + oracleStopLossConfig[pool_].timelock;   // uses CURRENT timelock
_requireElapsed(executeAfter);
```

Apply the same change to the decay, watermark, and timelock-change schedulers. This mirrors the recommended fix in the external report: store the start time and derive the end time dynamically, so any increase to the governing duration is automatically honoured for all in-flight proposals.

### Proof of Concept

```solidity
function testTimelockBypassViaAbsoluteExecuteAfter() public {
    // Pool starts with 1-day timelock
    vm.prank(factory);
    extension.initialize(address(pool),
        abi.encode(uint32(100_000), uint32(58), uint32(1 days)));

    // Admin proposes: disable stop-loss (drawdown = 100%) AND increase timelock to 30 days
    vm.startPrank(admin);
    extension.proposeOracleStopLossDrawdown(address(pool), 1_000_000); // 100%
    extension.proposeOracleStopLossTimelock(address(pool), uint32(30 days));
    vm.stopPrank();

    // Only 1 day passes — LPs expected 30 days of protection
    vm.warp(block.timestamp + 1 days);

    // Admin executes both; drawdown change only had 1-day protection
    vm.startPrank(admin);
    extension.executeOracleStopLossTimelock(address(pool));  // timelock → 30 days
    extension.executeOracleStopLossDrawdown(address(pool));  // drawdown → 100% (stop-loss off)
    vm.stopPrank();

    (uint32 drawdown,,,) = extension.oracleStopLossConfig(address(pool));
    assertEq(drawdown, 1_000_000); // stop-loss disabled after only 1 day
    // LPs had no 30-day window; stop-loss is now permanently silenced
}
