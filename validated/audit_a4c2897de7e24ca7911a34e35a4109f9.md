Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Binds to Router Address Instead of End-User, Allowing Any User to Bypass Per-User Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the router is `msg.sender` of `pool.swap()`, so the extension evaluates `allowedSwapper[pool][router]` — a single boolean covering every user of the router — rather than the originating user's address. A pool admin who allowlists the router to enable router-based swaps on a curated pool inadvertently grants swap access to every user, completely bypassing the per-user allowlist.

## Finding Description

**Root cause — extension checks direct caller, not economic actor:**

In `SwapAllowlistExtension.beforeSwap` (L31–41), `msg.sender` is the pool and `sender` is whoever called `pool.swap()`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

**Pool passes its own `msg.sender` as `sender`:**

`MetricOmmPool.swap` (L230–240) passes `msg.sender` — whoever called `pool.swap()` — directly into `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap()
```

**Router never forwards user identity:**

`MetricOmmSimpleRouter.exactInputSingle` (L72–80) calls `pool.swap()` without encoding the originating user anywhere in the call:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData   // ← user-supplied, not router-injected identity
    );
```

The same pattern applies to `exactInput` (L104–112), `exactOutputSingle` (L135–137), and `exactOutput` (L165–181).

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension` as `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps for approved users.
3. Attacker (not individually allowlisted) calls `router.exactInputSingle(...)`.
4. Router calls `pool.swap(...)` — pool sees `msg.sender = router`.
5. Pool calls `_beforeSwap(router, ...)` → extension checks `allowedSwapper[pool][router] == true` → no revert.
6. Swap executes; attacker receives output tokens.

**Existing guards are insufficient:** The `isAllowedToSwap` view function returns `true` for the router without revealing that this grants access to all users. There is no mechanism in the extension or router to authenticate the originating user.

## Impact Explanation
Any unprivileged EOA can execute swaps on a pool explicitly configured to restrict access to a named set of users, by routing through `MetricOmmSimpleRouter`. LP assets in the curated pool are exposed to trades from unauthorized counterparties. The pool's core access-control invariant — that only allowlisted addresses may swap — is completely broken whenever the router is allowlisted. This is a direct allowlist bypass through the router, which is an explicitly listed Smart Audit Pivot ("cannot be bypassed through router").

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary supported periphery path for end-user swaps. A pool admin running a curated pool who wants to support the standard router will naturally call `setAllowedToSwap(pool, router, true)`. The system provides no documentation warning against this, and the bypass requires no special privileges — any EOA can call the router. The mistake is easy to make and the attack is trivially repeatable.

## Recommendation
The extension must gate on the **economic actor** (end user), not the **call-chain intermediary** (router). Two complementary fixes:

1. **Router side:** Encode `msg.sender` into `extensionData` in a standard, authenticated way (e.g., ABI-encoded with a router-specific prefix) so allowlist extensions can recover the true initiator.
2. **Extension side:** Parse a `swapper` field from `extensionData` when `sender` is a known router, and verify `allowedSwapper[pool][swapper]` instead of `allowedSwapper[pool][sender]`.

Until resolved, pool admins must not allowlist the router on curated pools; users must call `pool.swap()` directly, forfeiting router safety features.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  - Pool admin: setAllowedToSwap(pool, router, true)
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  - attacker calls router.exactInputSingle(ExactInputSingleParams{
        pool:      curated_pool,
        recipient: attacker,
        ...
    })

Trace:
  router.exactInputSingle()                          [msg.sender = attacker]
    → pool.swap(recipient=attacker, ...)             [msg.sender = router]
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (no revert)
      → swap executes, attacker receives output tokens

Result:
  attacker swaps on a pool configured to block them.
  Per-user allowlist is completely bypassed.
```

Foundry test: deploy pool with `SwapAllowlistExtension`, configure `setAllowedToSwap(pool, router, true)`, call `router.exactInputSingle` from an address not individually allowlisted, assert no revert and tokens transferred.
