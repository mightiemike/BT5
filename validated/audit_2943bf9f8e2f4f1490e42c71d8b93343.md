### Title
Deprecated `verify_transaction_inclusion` Remains Fully Callable On-Chain, Bypassing Coinbase Proof Forgery Protection — (`File: contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is marked `#[deprecated]` but remains a fully exposed public NEAR contract method. Any unprivileged caller — including an external NEAR contract — can invoke it directly, bypassing the coinbase merkle proof validation that `verify_transaction_inclusion_v2` was introduced to enforce. This is a direct analog to the 0x delegated-stake-weight bypass: just as an external contract could represent delegated ZRX as operator-owned stake to obtain 100% weight instead of 90%, any NEAR caller can route around the v2 protection layer by calling v1 directly, obtaining a `true` inclusion result for a forged 64-byte transaction proof.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability (the "CVE-2012-2459 / BIP-0037 second-preimage" class attack documented at https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring the caller to also supply a coinbase merkle proof, which anchors the tree depth and prevents an attacker from substituting an internal node hash as a leaf `tx_id`. [1](#0-0) 

However, v1 is still deployed as a public contract method: [2](#0-1) 

The `#[deprecated]` Rust attribute is a **compiler-only hint**. It emits a warning when the function is called from Rust source code, but it has zero effect on NEAR's ABI or on-chain dispatch. The function is compiled into the WASM binary and registered as a callable contract method identically to any other `pub fn`. Any NEAR account — EOA or contract — can call `verify_transaction_inclusion` by name with a `ProofArgs` payload, with no role check and no coinbase proof requirement. [3](#0-2) 

The only guard on v1 is `#[pause]`, which is a liveness control, not a security boundary. When the contract is unpaused (its normal operating state), v1 is fully open.

---

### Impact Explanation

An attacker constructs a `ProofArgs` where `tx_id` is the hash of an **internal merkle tree node** (not a real transaction hash). Because v1 performs no coinbase proof check and no tree-depth validation, `compute_root_from_merkle_proof` will reconstruct the correct merkle root from the internal node, and `verify_transaction_inclusion` returns `true`. [4](#0-3) 

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) — whether by design, by mistake, or because an attacker routes a cross-contract call through it — receives a forged `true` result for a transaction that was never broadcast or confirmed on Bitcoin. If that downstream contract gates asset release, bridge withdrawals, or state transitions on this result, those actions execute on a fabricated proof.

The CLAUDE.md explicitly acknowledges this attack surface exists but frames it as a caller responsibility: [5](#0-4) 

The problem is that the contract itself provides no enforcement: v1 is equally accessible to all callers, so the "caller must use v2" policy is unenforceable at the protocol level.

---

### Likelihood Explanation

- No privileged role is required to call `verify_transaction_inclusion`.
- No deposit is required (it is a view-compatible read function).
- The function name is documented in the public ABI and in the relayer client: [6](#0-5) 

- Any external NEAR contract that integrates the light client can call v1 directly, either intentionally (to avoid the coinbase proof overhead) or because it was written before v2 existed and was never updated.
- The 64-byte forgery technique is well-documented and the construction of a valid internal-node proof is straightforward for anyone with access to a Bitcoin block's raw transaction data.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the deployed contract's public ABI entirely. In NEAR/Rust, this means either:

1. **Change visibility**: Make the function `pub(crate)` so it is only callable internally (by `verify_transaction_inclusion_v2`), removing it from the on-chain ABI.
2. **Inline the logic**: Fold v1's body directly into v2 and delete the standalone v1 method.

The `#[deprecated]` attribute alone provides no on-chain protection and should not be relied upon as a security boundary.

---

### Proof of Concept

1. Deploy the contract (Bitcoin feature, `skip_pow_verification = false`).
2. Submit a real Bitcoin block containing at least two transactions so the merkle tree has internal nodes.
3. Compute the hash of an internal merkle node `N` (e.g., the parent of tx0 and tx1).
4. Construct `ProofArgs { tx_id: N, tx_block_blockhash: <that block's hash>, tx_index: <index that places N at the root via the remaining siblings>, merkle_proof: <siblings above N>, confirmations: 0 }`.
5. Call `verify_transaction_inclusion` directly (not v2) from any unprivileged NEAR account.
6. The call returns `true` — `N` is accepted as a confirmed transaction inclusion, despite `N` not being a real transaction hash.

The same call routed through `verify_transaction_inclusion_v2` would require a valid coinbase proof anchoring the tree depth, which would reject this crafted input. [7](#0-6)

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub async fn verify_transaction_inclusion(
        &self,
        transaction_hash: H256,
        transaction_position: usize,
        transaction_block_blockhash: H256,
        merkle_proof: Vec<H256>,
        confirmations: u64,
    ) -> Result<bool, Box<dyn std::error::Error + Send + Sync>> {
        let args = btc_types::contract_args::ProofArgs {
            tx_id: transaction_hash,
            tx_block_blockhash: transaction_block_blockhash,
            tx_index: transaction_position.try_into()?,
            merkle_proof,
            confirmations,
        };

        let tx_hash = self
            .submit_tx(
                self.sign_tx(VERIFY_TRANSACTION_INCLUSION, to_vec(&args)?, 0, None)
                    .await?,
            )
            .await?;
        let response = self.get_tx_status(tx_hash).await?;
        Self::parse_verify_response(response, VERIFY_TRANSACTION_INCLUSION)
    }
