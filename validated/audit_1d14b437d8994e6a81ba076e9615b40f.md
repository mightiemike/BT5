### Title
Stale Non-Zero ERC20 Approval in `DirectDepositV1.creditDeposit` Permanently Bricks USDT Deposits - (File: core/contracts/DirectDepositV1.sol)

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. When `Endpoint.depositCollateralWithReferral` silently returns early (below-minimum deposit), the approval is never consumed. Any subsequent call to `creditDeposit()` with a non-zero balance will attempt a non-zero → non-zero `approve` for tokens like USDT, which reverts, permanently bricking the DDA for that token.

### Finding Description

`creditDeposit()` iterates over all registered spot product IDs, and for each token with a non-zero balance it unconditionally calls:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
``` [1](#0-0) 

`Endpoint.depositCollateralWithReferral` contains an explicit early-return path that skips `handleDepositTransfer` (and therefore skips the `transferFrom` that would consume the allowance) when the deposit amount is below the protocol minimum:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked ...
    return;
}
handleDepositTransfer(...);
``` [2](#0-1) 

After this early return, the DDA holds a non-zero allowance for the endpoint. On the next call to `creditDeposit()` with any non-zero balance, `token.approve(endpoint, newBalance)` is called while the existing allowance is still non-zero. USDT (and other tokens with the same guard) reverts on any `approve` call that changes a non-zero allowance to another non-zero value. The DDA is permanently unable to process deposits for that token.

`creditDeposit()` has no access control — it is callable by any external address: [3](#0-2) 

`ContractOwner.creditDepositV1` also exposes this path publicly: [4](#0-3) 

Note: `ContractOwner.wrapVaultAsset` correctly avoids this pattern by resetting to zero first, demonstrating the fix is known within the codebase: [5](#0-4) 

### Impact Explanation

Any DDA deployed for a subaccount that holds USDT (or any token with the non-zero → non-zero `approve` guard) becomes permanently unable to credit deposits once the stale allowance is set. Tokens sent to the DDA accumulate but can never be forwarded to the endpoint. While the owner can recover tokens via `withdraw()`, the deposit functionality is irreversibly broken for that DDA instance, requiring redeployment and migration.

### Likelihood Explanation

The trigger is straightforward and requires no special privilege. An attacker (or even an accidental deposit below the minimum threshold) can set the stale allowance. The minimum deposit check in `depositCollateralWithReferral` is a normal protocol guard that fires for any small transfer. USDT is a standard collateral asset in perpetuals protocols. Likelihood is **medium-high**.

### Recommendation

Reset the allowance to zero before setting a new non-zero allowance in `creditDeposit()`:

```solidity
if (balance != 0) {
    token.approve(address(endpoint), 0);
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(...);
}
```

This mirrors the pattern already used correctly in `ContractOwner.wrapVaultAsset` at lines 530–531.

### Proof of Concept

1. A DDA is created for `subaccount` via `ContractOwner.createDirectDepositV1`.
2. Attacker transfers 1 wei of USDT to the DDA address (below `MIN_DEPOSIT_AMOUNT`).
3. Attacker calls `creditDeposit()` (no access control). The DDA calls `usdt.approve(endpoint, 1)` — allowance is now 1. `depositCollateralWithReferral` returns early; `transferFrom` is never called; allowance remains 1.
4. A legitimate user later sends 1000 USDT to the DDA and calls `creditDeposit()`. The DDA calls `usdt.approve(endpoint, 1000)` — USDT reverts because existing allowance (1) is non-zero.
5. Every future call to `creditDeposit()` for USDT reverts. The DDA is bricked for USDT deposits. [3](#0-2)

        for (uint256 i = 0; i < productIds.length; i++) {

                );

        }
    }
