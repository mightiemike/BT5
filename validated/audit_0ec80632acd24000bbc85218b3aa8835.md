### Title
Spot Base Asset Balance Permanently Locked on Isolated Subaccount Closure — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`_tryCloseIsolatedSubaccount` only sweeps `perpEngine.vQuoteBalance` and `spotEngine.getBalance(QUOTE_PRODUCT_ID)` back to the parent. It never reads or transfers `spotEngine.getBalance(productId, subaccount)` — the non-quote spot base asset. Because the closing gate (`perpEngine.getBalance(productId, subaccount).amount == 0`) is trivially satisfied for any spot product (spot products carry no balance in `perpEngine`), the function will close a spot isolated subaccount immediately, even while it holds a non-zero base asset balance, orphaning those assets permanently.

---

### Finding Description

`_tryCloseIsolatedSubaccount` reads the perp engine for the product encoded in the isolated subaccount's address: [1](#0-0) 

For a spot product, `perpEngine.getBalance(spotProductId, subaccount)` always returns a zero struct — spot products are not tracked in `perpEngine`. The `balance.amount == 0` guard therefore passes unconditionally, and the function proceeds to close the subaccount.

Inside the closure block, only two balances are swept to the parent:

1. `perpEngine.vQuoteBalance` (lines 173–185) — always zero for a spot product.
2. `spotEngine.getBalance(QUOTE_PRODUCT_ID, subaccount)` (lines 187–201) — the USDC/quote balance. [2](#0-1) 

`spotEngine.getBalance(productId, subaccount)` — the base asset (e.g. ETH) — is never read or transferred. After the three storage clears at lines 202–204, `parentSubaccounts[subaccount]` is zeroed and the isolated subaccount is fully de-registered. The base balance has no recovery path.

Isolated subaccounts are not restricted to perp products. `createIsolatedSubaccount` accepts any `txn.productId`: [3](#0-2) 

`_getCallState` routes spot products to `callState.isPerp = false`, and `_updateBalances` then calls `callState.spot.updateBalance(callState.productId, subaccount, baseDelta, quoteDelta)`, crediting the isolated subaccount with a real spot base balance after a fill: [4](#0-3) 

`tryCloseIsolatedSubaccount` is reachable through three production paths without any precondition that the base balance be zero:

- **`CloseIsolatedSubaccount` sequencer transaction** — `EndpointTx.sol` line 639, callable at any time after a fill.
- **Liquidation finalization** — `ClearinghouseLiq.sol` lines 620–625, called when `_finalizeSubaccount` returns true.
- **`delistProduct`** — `Clearinghouse.sol` lines 319–323, called for each subaccount in the delist list. [5](#0-4) 

---

### Impact Explanation

Any spot base asset balance held by a spot isolated subaccount at the moment of closure is permanently locked in the now-orphaned subaccount. `parentSubaccounts[subaccount]` is cleared, so `transferQuote` (which requires a valid parent link for isolated subaccounts) cannot be used to recover it. No other withdrawal or sweep path exists for an orphaned isolated subaccount. The user's base asset is irrecoverably lost.

---

### Likelihood Explanation

The `CloseIsolatedSubaccount` transaction is a normal sequencer operation. Any spot isolated subaccount that has received a partial or full fill and is then closed — whether by the sequencer, by liquidation, or by a delist — will silently drop the base balance. No special attacker capability is required; the bug is triggered by the ordinary lifecycle of a spot isolated order.

---

### Recommendation

Before clearing the parent link, also sweep the spot base balance for the traded product:

```solidity
int128 baseBalance = spotEngine.getBalance(productId, subaccount).amount;
if (baseBalance != 0) {
    spotEngine.updateBalance(productId, subaccount, -baseBalance);
    spotEngine.updateBalance(productId, parent, baseBalance);
}
```

This should be inserted between the existing `quoteBalance` sweep (lines 187–201) and the storage-clear block (lines 202–204). Additionally, consider adding an explicit guard that rejects closure when `productId` is a spot product and `spotEngine.getBalance(productId, subaccount).amount != 0`, to make the invariant explicit and auditable. [6](#0-5) 

---

### Proof of Concept

1. Deploy the protocol with a spot market (e.g. ETH, `productId = 2`).
2. Call `CreateIsolatedSubaccount` with `productId = 2` and a non-zero `isolatedMargin`. This creates `isoSubaccount` and transfers USDC margin to it.
3. Submit a `MatchOrders` transaction that fills the isolated order. `_updateBalances` credits `isoSubaccount` with `+1e18` ETH at `productId = 2` in `spotEngine`.
4. Submit a `CloseIsolatedSubaccount` transaction for `isoSubaccount`.
5. `_tryCloseIsolatedSubaccount` runs: `perpEngine.getBalance(2, isoSubaccount).amount == 0` → true. USDC is swept to parent. ETH balance at `productId = 2` is not touched. `parentSubaccounts[isoSubaccount]` is cleared.
6. Assert: `spotEngine.getBalance(2, isoSubaccount).amount == 1e18` (non-zero, orphaned). Assert: `parentSubaccounts[isoSubaccount] == bytes32(0)` (no recovery path). [7](#0-6)

        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);

        }

            productId,

        if (balance.amount == 0) {

            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(

        }
    }
