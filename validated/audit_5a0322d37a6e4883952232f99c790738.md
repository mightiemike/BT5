### Title
Mutable `ExitDuration` in `MsgUpdateTier` Allows Governance to Retroactively Extend Lock Period After User Deposits — (`x/tieredrewards/keeper/msg_server_auth.go`)

---

### Summary

`MsgUpdateTier` allows governance to change `ExitDuration` on a live tier at any time, including after users have already locked funds. Because `ExitDuration` is read from the tier at the moment a user calls `MsgTriggerExitFromTier` — not at lock time — any governance change to `ExitDuration` silently rewrites the exit commitment that existing position holders will face when they eventually trigger their exit. Users committed funds under one set of rules and may be forced to wait a materially different period before they can recover them.

---

### Finding Description

`UpdateTier` in `x/tieredrewards/keeper/msg_server_auth.go` writes the new tier directly to state with no guard on `ExitDuration` changes for existing positions:

```go
func (ms msgServer) UpdateTier(ctx context.Context, msg *types.MsgUpdateTier) (*types.MsgUpdateTierResponse, error) {
    oldTier, err := ms.getTier(ctx, msg.Tier.Id)
    ...
    if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
        // Only BonusApy change triggers a settlement sweep
        ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id)
    }
    ms.SetTier(ctx, msg.Tier)   // ExitDuration written unconditionally
    ...
}
``` [1](#0-0) 

When a user later calls `MsgTriggerExitFromTier`, the handler reads `tier.ExitDuration` live from state and stamps `ExitUnlockAt = blockTime + tier.ExitDuration` onto the position:

```go
tier, err := ms.getTier(ctx, pos.TierId)
...
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
``` [2](#0-1) 

`TriggerExit` simply computes `ExitUnlockAt = blockTime + duration`:

```go
func (p *Position) TriggerExit(blockTime time.Time, duration time.Duration) {
    p.ExitTriggeredAt = blockTime
    p.ExitUnlockAt = blockTime.Add(duration)
}
``` [3](#0-2) 

`ExitDuration` is **never snapshotted at lock time**. The `Position` proto stores only `ExitTriggeredAt` and `ExitUnlockAt` (set at trigger time), not the duration the user relied on when locking. [4](#0-3) 

The design intent is that `ExitDuration` applies to new positions, as confirmed by the integration test that verifies the new duration applies to new positions after an update: [5](#0-4) 

However, the same governance action equally affects all existing positions that have not yet triggered exit — there is no distinction between "new" and "existing" positions in the `UpdateTier` handler.

---

### Impact Explanation

**Scenario — ExitDuration increased:**

```
Initial: Tier 1 ExitDuration = 1 year
User A locks 10,000 CRO expecting a 1-year exit commitment
Governance passes MsgUpdateTier: ExitDuration = 5 years
User A calls TriggerExitFromTier
ExitUnlockAt = now + 5 years  (not 1 year)
User A's funds are locked for 4 additional years beyond their expectation
```

**Scenario — ExitDuration decreased (favors exiting users, harms protocol):**

```
Initial: Tier 1 ExitDuration = 5 years (designed to incentivize long-term staking)
Users lock large amounts expecting 5-year commitment
Governance passes MsgUpdateTier: ExitDuration = 1 day
All pre-existing position holders can now exit in 1 day
Long-term staking incentive is retroactively nullified
```

The corrupted value is `Position.ExitUnlockAt` — the concrete timestamp that gates `MsgTierUndelegate`, `MsgWithdrawFromTier`, and `MsgExitTierWithDelegation`. All three check `CompletedExitLockDuration(blockTime)` which reads `ExitUnlockAt` directly. [6](#0-5) 

---

### Likelihood Explanation

The trigger path is a standard governance proposal (`MsgUpdateTier`) submitted and approved through the normal `x/gov` flow — the same path used in production integration tests. [7](#0-6) 

The upgrade test itself deliberately shortens `ExitDuration` via `MsgUpdateTier` on a live position to speed up test execution, demonstrating the mechanism is reachable and effective on existing positions: [8](#0-7) 

Any governance participant (validator set majority or token-weighted quorum) can submit and pass such a proposal. No leaked keys or social engineering are required.

---

### Recommendation

**Option 1 — Snapshot `ExitDuration` at lock time.** Add an `exit_duration` field to `Position` and populate it at creation. `TriggerExitFromTier` reads from the position, not the tier.

**Option 2 — Protect existing positions on `ExitDuration` change.** In `UpdateTier`, when `ExitDuration` changes, iterate all positions in the tier that have not yet triggered exit and record the old duration on each position before overwriting the tier.

**Option 3 — Time-lock `ExitDuration` changes.** Require a minimum notice period (e.g., equal to the current `ExitDuration`) before a new `ExitDuration` takes effect, giving existing holders time to trigger exit under the old rules.

Note: `BonusApy` changes already have a settlement sweep (`claimRewardsAndUpdateTierPositions`) as a mitigation. An analogous protection for `ExitDuration` is absent. [9](#0-8) 

---

### Proof of Concept

1. Governance creates Tier 1 with `ExitDuration = 1 year`, `BonusApy = 4%`.
2. User calls `MsgLockTier(tier_id=1, amount=10000, validator=V)` — position created, `ExitTriggeredAt` and `ExitUnlockAt` are zero (not yet exiting).
3. Governance submits and passes `MsgUpdateTier(id=1, exit_duration=5 years, bonus_apy=4%, ...)`.
   - `UpdateTier` sees `BonusApy` unchanged → no settlement sweep.
   - `SetTier` writes `ExitDuration = 5 years` to state unconditionally.
4. User calls `MsgTriggerExitFromTier(position_id=P)`.
   - Handler reads `tier.ExitDuration = 5 years`.
   - `pos.TriggerExit(blockTime, 5 years)` → `ExitUnlockAt = now + 5 years`.
5. User is now locked for 5 years. `MsgTierUndelegate` and `MsgExitTierWithDelegation` both gate on `CompletedExitLockDuration(blockTime)` which will return `false` for 5 years.
6. The user committed funds under a 1-year rule; the protocol enforces a 5-year rule.

func (ms msgServer) UpdateTier(ctx context.Context, msg *types.MsgUpdateTier) (*types.MsgUpdateTierResponse, error) {
	if err := ms.requireAuthority(msg.Authority); err != nil {
		return nil, err
	}

	oldTier, err := ms.getTier(ctx, msg.Tier.Id)
	if err != nil {
		return nil, err
	}

	if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
		if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
			return nil, err
		}
	}

	if err := ms.SetTier(ctx, msg.Tier); err != nil {
		return nil, err
	}

	if err := ms.emitTierChangedEvent(ctx, types.TierChangeAction_TIER_CHANGE_ACTION_UPDATE, msg.Tier); err != nil {
		return nil, err
	}

	return &types.MsgUpdateTierResponse{}, nil
}
