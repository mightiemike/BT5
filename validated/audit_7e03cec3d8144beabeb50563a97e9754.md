### Title
Silent Transfer Success on Destroyed or Undeployed Token Contracts Enables Phantom Collateral Deposits - (File: `core/contracts/libraries/ERC20Helper.sol`)

---

### Summary

`ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` perform low-level `.call()` without verifying that the target token contract exists. Per the Solidity/EVM specification, a call to a non-existent account returns `success = true` with empty return data. Because the guard condition `success && (data.length == 0 || abi.decode(data, (bool)))` passes for empty return data, both functions silently succeed when the token has been destroyed or has not yet been deployed. This library is used in the deposit and withdrawal paths of the Nado protocol, meaning a destroyed token can produce phantom collateral credits or silently swallow withdrawal transfers.

---

### Finding Description

`ERC20Helper` is the sole transfer primitive used across the protocol:

```solidity
// core/contracts/libraries/ERC20Helper.sol lines 14–20
(bool success, bytes memory data) = address(self).call(
    abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [1](#0-0) 

The same pattern is repeated in `safeTransferFrom`: [2](#0-1) 

When `address(self)` has no deployed bytecode (destroyed via `selfdestruct`, or not yet deployed), the EVM returns `(true, "")`. The require at line 17–20 evaluates `true && (0 == 0 || ...)` → passes. No revert occurs and no tokens move.

`EndpointStorage` imports and activates this library for all deposit-path transfers: [3](#0-2) 

`handleDepositTransfer` calls both `safeTransferFrom` and `safeTransferTo` through `ERC20Helper`: [4](#0-3) 

`Clearinghouse` activates the same library for the withdrawal path: [5](#0-4) 

`handleWithdrawTransfer` calls `token.safeTransfer` through `ERC20Helper`: [6](#0-5) 

`BaseWithdrawPool` also activates `ERC20Helper` and calls `token.safeTransfer` in `handleWithdrawTransfer`: [7](#0-6) [8](#0-7) 

Additionally, `DirectDepositV1` contains its own inline copy of the same vulnerable pattern: [9](#0-8) 

---

### Impact Explanation

**Deposit path (attacker gains phantom collateral):**

After a listed spot token is destroyed, an attacker calls `depositCollateral` with that token. `handleDepositTransfer` invokes `ERC20Helper.safeTransferFrom` (no tokens move, call silently succeeds) and then `ERC20Helper.safeTransferTo` (same). The Endpoint then queues a `DepositCollateral` transaction; the sequencer processes it and `Clearinghouse.depositCollateral` credits the attacker's subaccount with `amountRealized`: [10](#0-9) 

The attacker now holds phantom collateral backed by nothing. They can use it to open positions, pass health checks, or withdraw other real assets from the protocol, directly draining solvent users and the insurance fund.

**Withdrawal path (victim loses balance silently):**

If a token is destroyed after a user initiates a withdrawal, `handleWithdrawTransfer` calls `token.safeTransfer` through `ERC20Helper`. The call silently succeeds, the user's on-chain balance is debited, but no tokens are received. The user permanently loses their collateral. [11](#0-10) 

---

### Likelihood Explanation

ERC-20 tokens can be destroyed via `selfdestruct` if their implementation includes such a function (common in older or custom tokens). Tokens that are registered as products but not yet deployed (e.g., pre-announced listings) are also vulnerable. The deposit entry point (`depositCollateral` / `depositCollateralWithReferral`) is publicly callable by any unprivileged user with no preconditions beyond having a subaccount. The only guard in `getToken` checks `address(token) != address(0)`, which does not detect a destroyed contract: [12](#0-11) 

Likelihood is **medium**: it requires a listed token to be destroyed or undeployed, which is not a common event but is a realistic edge case for any protocol supporting multiple spot assets over time.

---

### Recommendation

Add a contract existence check before each low-level call in `ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom`:

```solidity
function safeTransfer(IERC20Base self, address to, uint256 amount) internal {
    require(address(self).code.length > 0, "ERC20Helper: token not deployed");
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Apply the same guard to `safeTransferFrom` and to the inline `safeTransfer` in `DirectDepositV1`. Long-term, prefer OpenZeppelin's `SafeERC20` which includes this check. [13](#0-12) 

---

### Proof of Concept

1. Token `T` is listed as a spot product in the Nado protocol (non-zero address, valid `productId`).
2. `T`'s contract is destroyed via `selfdestruct` (or was never deployed at the registered address).
3. Attacker calls `Endpoint.depositCollateralWithReferral(subaccount, productId_T, 1_000_000e18, "")`.
4. `EndpointStorage.handleDepositTransfer` executes:
   - `ERC20Helper.safeTransferFrom(T, attacker, endpoint, 1_000_000e18)` → `address(T).call(...)` returns `(true, "")` → require passes, no tokens move.
   - `ERC20Helper.safeTransferTo(T, clearinghouse, 1_000_000e18)` → same, passes silently.
5. Sequencer processes the queued `DepositCollateral` transaction; `Clearinghouse.depositCollateral` credits attacker's subaccount with `1_000_000e18 * multiplier` units of product `productId_T`.
6. Attacker's health check passes (positive phantom balance). Attacker calls `withdrawCollateral` for the quote token (USDC), draining real assets from the clearinghouse. [4](#0-3) [14](#0-13)
