### Title
Linked Signer Can Unilaterally Replace Itself, Enabling Persistent Unauthorized Subaccount Control — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction handler in `EndpointTx.sol` validates the transaction with `allowLinkedSigner = true`. This means the currently-linked signer — not just the subaccount owner — can sign and submit a `LinkSigner` transaction to replace the linked signer with any arbitrary address. A malicious or compromised linked signer can therefore permanently entrench its own access (or hand it to a colluding address) without the subaccount owner's consent, analogous to one admin removing all other admins in the reference report.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the permitted signer as:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts a signature from either the subaccount owner address **or** the linked signer address:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
         (recovered == linkedSigner))
``` [3](#0-2) 

Because the linked signer is accepted as a valid signer for `LinkSigner` itself, the linked signer can submit a sequencer-processed `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with any address it chooses — including a fresh attacker-controlled address.

The slow-mode `LinkSigner` path does **not** share this flaw: it uses `validateSender`, which enforces that `msg.sender` matches the wallet address embedded in the subaccount bytes32, so only the actual owner's EOA can submit it. [4](#0-3) 

---

### Impact Explanation

Once the linked signer replaces itself with an attacker-controlled address (`Carol`), `Carol` becomes the authorized signer for the victim subaccount. `Carol` can then:

- Sign `WithdrawCollateral` (V1) transactions — funds are sent to the subaccount owner's wallet address, so direct theft via V1 is blocked.
- Sign `WithdrawCollateralV2` with `sendTo == address(0)` — again funds go to the owner's wallet.
- Sign `MintNlp` / `BurnNlp` transactions, causing the subaccount to enter or exit NLP positions at sequencer-chosen oracle prices, potentially at a loss.
- Sign `LiquidateSubaccount` transactions using the victim subaccount as the liquidator, charging it the `LIQUIDATION_FEE` on every call.
- Sign `TransferQuote` transactions — constrained to same-address-prefix subaccounts, so cross-wallet theft is blocked, but intra-user subaccount balances can be drained.
- Sign any future `MatchOrders` as taker at unfavorable prices, draining the subaccount's collateral to a counterparty.

The most critical impact is **persistent unauthorized control**: the owner's only recourse is a slow-mode `LinkSigner` transaction, which is queued with a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). During that window the attacker-controlled linked signer remains active and can continue to act. [5](#0-4) 

---

### Likelihood Explanation

Linked signers are the standard mechanism for automated trading bots and third-party portfolio managers in perpetual DEX protocols. Any user who has linked a signer to their subaccount is exposed. The attack requires only that the linked signer (or an attacker who has compromised the linked signer's private key) submit one sequencer transaction — a realistic and low-cost operation. No admin access, governance capture, or social engineering is required.

---

### Recommendation

Remove `allowLinkedSigner = true` from the fast-mode `LinkSigner` handler. The linked signer should be permitted to perform trading operations on behalf of the subaccount, but it must not be permitted to mutate its own authorization. Only the subaccount owner (the address embedded in the subaccount bytes32) should be able to change the linked signer.

Change:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // ← remove linked signer permission here
);
```

To:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only subaccount owner may change linked signer
);
``` [6](#0-5) 

---

### Proof of Concept

1. Alice owns subaccount `alice_subaccount = bytes32(abi.encodePacked(alice_addr, subaccountName))`.
2. Alice submits a slow-mode `LinkSigner` transaction setting `linkedSigners[alice_subaccount] = bob_addr`.
3. Bob (the linked signer, now malicious) constructs a `SignedLinkSigner` struct:
   - `tx.sender = alice_subaccount`
   - `tx.signer = bytes32(uint256(uint160(carol_addr)))` (attacker-controlled)
   - `tx.nonce = nonces[alice_addr]` (current nonce)
4. Bob signs the EIP-712 digest of this struct with his private key.
5. Bob submits the transaction to the sequencer.
6. The sequencer calls `submitTransactionsChecked`, which calls `processTransactionImpl`.
7. `validateSignedTx(..., true)` recovers Bob's address, matches it against `getLinkedSigner(alice_subaccount) == bob_addr` — validation passes.
8. `linkedSigners[alice_subaccount]` is overwritten with `carol_addr`.
9. Carol is now the authorized linked signer for Alice's subaccount.
10. Alice's only recourse is a slow-mode `LinkSigner` with a 3-day execution delay, during which Carol retains full signing authority. [1](#0-0) [7](#0-6)
