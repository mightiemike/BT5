Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Position `owner` Instead of Transaction `sender`, Allowing Any Address to Bypass the Deposit Allowlist ŌĆö (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual transaction caller) and gates access on `owner` (the LP position recipient) instead. Because `MetricOmmPool.addLiquidity` imposes no constraint that `msg.sender == owner`, any address not on the allowlist can deposit into a permissioned pool by naming any allowlisted address as the LP position `owner`, completely nullifying the pool admin's access control intent.

## Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` for the LP position and passes `msg.sender` as `sender` and `owner` as `owner` to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

Unlike `removeLiquidity`, which enforces `msg.sender == owner` before proceeding, `addLiquidity` has no such constraint ŌĆö any caller may credit shares to any `owner` address: [2](#0-1) 

The extension hook receives `(sender, owner, ŌĆ”)` in that order. `DepositAllowlistExtension.beforeAddLiquidity` names the first parameter with no identifier (discarded) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

The contract's own NatSpec states it "Gates `addLiquidity` by depositor address", and `setAllowedToDeposit`/`isAllowedToDeposit` are framed around the depositor concept ŌĆö but the actual depositor (`sender`) is never checked: [4](#0-3) 

**Exploit path:**
1. Pool admin configures `DepositAllowlistExtension` and sets `allowedDepositor[pool][alice] = true`. Bob is not allowlisted.
2. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. The pool calls `_beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`.
4. The extension evaluates `allowedDepositor[pool][alice]` ŌåÆ `true` ŌåÆ no revert.
5. Bob's tokens are transferred to the pool via the callback; alice receives LP shares.
6. Bob has bypassed the deposit allowlist entirely.

## Impact Explanation

The deposit allowlist is completely defeated. The pool admin's intent to restrict which addresses may deposit is nullified ŌĆö any address, including sanctioned or otherwise excluded actors, can inject tokens and alter bin state. Additionally, the allowlisted victim (`alice`) receives LP shares in bins she never chose, and the attacker can repeat the deposit continuously to grief. This constitutes a broken admin-boundary control: an unprivileged caller bypasses a pool-admin-configured access restriction.

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with `owner` set to any allowlisted address. No privileged access, no special token, no flash loan is required. The attacker must supply tokens (which enter the pool), making the attack economically rational for griefing or pool-state manipulation. The barrier is trivially low and the attack is repeatable.

## Recommendation

Replace the unnamed first parameter with `sender` and check it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the design intent is to allow a whitelisted router to deposit on behalf of any owner, both `sender` and `owner` should be checked with explicit policy (e.g., `sender` must be allowlisted, or `sender == owner`).

## Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  bob is NOT on the allowlist

Attack:
  bob calls pool.addLiquidity(
      owner        = alice,   // allowlisted ŌĆö passes the guard
      salt         = 0,
      deltas       = { binIdxs: [0], shares: [1e18] },
      callbackData = ...,     // bob pays tokens in the callback
      extensionData = ""
  )

Result:
  - beforeAddLiquidity checks allowedDepositor[pool][alice] ŌåÆ true ŌåÆ no revert
  - bob's tokens are transferred to the pool
  - alice receives 1e18 LP shares in bin 0
  - bob has bypassed the deposit allowlist entirely
```

Foundry test: deploy pool with `DepositAllowlistExtension`, allowlist `alice`, call `addLiquidity` from `bob` with `owner = alice`, assert no revert and that `alice`'s position shares increased while `bob`'s token balance decreased.
