### Title
Unsynchronized `number_of_blocks_for_mean` Between `L1GasPriceScraperConfig` and `L1GasPriceProviderConfig` Causes Wrong L1 Gas Price Mean and Incorrect Fee Accounting - (`crates/apollo_l1_gas_price_config/src/config.rs`)

---

### Summary

The `L1GasPriceScraperConfig` and `L1GasPriceProviderConfig` each carry an independent `number_of_blocks_for_mean` field with no cross-validation enforcing they are equal. When they diverge, the scraper fetches fewer historical L1 blocks than the provider needs to compute a correct mean, causing the provider to silently fall back to a shorter window and produce a systematically wrong L1 gas price. This wrong price is then embedded in every block proposal's `ProposalInit`, affecting fee charging for all transactions in those blocks.

---

### Finding Description

The external bug's core invariant is: **a tracked/stored value diverges from the actual value because external changes are not accounted for in the tracking logic**. The sequencer analog is: the scraper's `number_of_blocks_for_mean` controls how far back in L1 history the scraper starts, while the provider's `number_of_blocks_for_mean` controls the window size used to compute the mean gas price. These two values are **separate config fields** with no enforcement that they match.

The code comment in `config.rs` explicitly acknowledges this:

```rust
// TODO(guyn): find a way to synchronize the value of number_of_blocks_for_mean
// with the one in L1GasPriceProviderConfig. In the end they should both be loaded
// from VersionedConstants.
``` [1](#0-0) 

The scraper uses its own `number_of_blocks_for_mean` to determine the startup block:

```rust
let latest = latest.saturating_sub(
    self.config.number_of_blocks_for_mean * self.config.startup_num_blocks_multiplier,
);
``` [2](#0-1) 

The provider uses its own `number_of_blocks_for_mean` to compute the mean:

```rust
let num_blocks = usize::try_from(self.config.number_of_blocks_for_mean)
    .expect("number_of_blocks_for_mean is too large to fit into a usize");

let first_index = if last_index >= num_blocks {
    last_index - num_blocks
} else {
    warn!("Not enough history to calculate the mean gas price...");
    L1_GAS_PRICE_PROVIDER_INSUFFICIENT_HISTORY.increment(1);
    0
};
``` [3](#0-2) 

The provider's ring buffer is sized by `storage_limit`, not `number_of_blocks_for_mean`, and the two configs are loaded independently from separate JSON files: [4](#0-3) [5](#0-4) 

**Concrete divergence scenario**: If `L1GasPriceScraperConfig.number_of_blocks_for_mean = 100` but `L1GasPriceProviderConfig.number_of_blocks_for_mean = 300`, the scraper only pre-fetches 200 historical blocks (100 × 2 multiplier) at startup. The provider then tries to average over 300 blocks but only has ≤200 available, silently falling back to a shorter window and emitting a wrong mean. The wrong mean is then used in `ProposalInit` gas prices for every block.

The wrong gas price flows into block production via `convert_to_sn_api_block_info`, which constructs `BlockInfo` from `ProposalInit` fields including `l1_gas_price_fri` and `l1_data_gas_price_fri`: [6](#0-5) 

These prices are then used by the blockifier's fee checks to charge users: [7](#0-6) 

---

### Impact Explanation

When `L1GasPriceScraperConfig.number_of_blocks_for_mean < L1GasPriceProviderConfig.number_of_blocks_for_mean`, the provider silently computes the mean over fewer blocks than intended. This produces a wrong L1 gas price (WEI) that is embedded in every `ProposalInit`. The wrong WEI price is then converted to FRI and used as the authoritative `l1_gas_price_fri` / `l1_data_gas_price_fri` for all transactions in the block. Users are charged incorrect fees — either systematically undercharged (if the shorter window happens to be lower) or overcharged (if it is higher), with no error or revert. This matches the **"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact"** critical impact category.

---

### Likelihood Explanation

The two configs are loaded from separate JSON files with no cross-validation in `validate_node_config` or `cross_member_validations`. An operator who tunes `l1_gas_price_provider_config.number_of_blocks_for_mean` without also updating `l1_gas_price_scraper_config.number_of_blocks_for_mean` (or vice versa) silently introduces the divergence. The codebase itself acknowledges the risk with a `TODO` comment. The default values happen to match (both 300), but there is no runtime enforcement. [8](#0-7) 

---

### Recommendation

1. **Add a cross-validation** in `SequencerNodeConfig::cross_member_validations` (or a dedicated validator) that asserts `l1_gas_price_scraper_config.number_of_blocks_for_mean == l1_gas_price_provider_config.number_of_blocks_for_mean` and returns a `ConfigError` if they differ.

2. **Long-term**: move `number_of_blocks_for_mean` into `VersionedConstants` as the TODO comment suggests, so both components read from a single source of truth. [1](#0-0) 

---

### Proof of Concept

**Setup**: Deploy the sequencer with:
```json
// l1_gas_price_provider_config.json
{ "l1_gas_price_provider_config.number_of_blocks_for_mean": 300 }

// l1_gas_price_scraper_config.json
{ "l1_gas_price_scraper_config.number_of_blocks_for_mean": 50 }
```

**Execution**:
1. Scraper starts, computes `start_block = latest - 50 * 2 = latest - 100`. It feeds only ~100 blocks into the provider's ring buffer.
2. Provider receives a `get_price_info(timestamp)` call. `last_index` resolves to ~100. `num_blocks = 300`. Since `100 < 300`, the `else` branch fires:
   ```
   warn!("Not enough history to calculate the mean gas price. Using blocks 0-99, inclusive.");
   first_index = 0;
   actual_number_of_blocks = 100;
   ```
3. The mean is computed over 100 blocks but divided by `actual_number_of_blocks = 100` — so the mean is numerically correct for 100 blocks but represents a **different (shorter) window** than the protocol-intended 300-block window. If L1 gas prices have been rising, the 100-block mean is higher than the 300-block mean, causing all users to be overcharged.
4. This wrong price is placed in `ProposalInit.l1_gas_price_wei` and propagated to all validators, who accept it because they perform no independent price verification.
5. The blockifier charges every transaction in the block using this wrong price. [9](#0-8) [10](#0-9)

    init: &ProposalInit,
) -> Result<starknet_api::block::BlockInfo, StarknetApiError> {
    if init.l1_gas_price_fri.0 == 0
        || init.l1_gas_price_wei.0 == 0
        || init.l1_data_gas_price_fri.0 == 0
        || init.l1_data_gas_price_wei.0 == 0
        || init.l2_gas_price_fri.0 == 0
    {
        warn!("Zero gas price detected in block info: {:?}", init);

    let l1_data_gas_price_fri = NonzeroGasPrice::new(init.l1_data_gas_price_fri)?;
    let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
    let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
