### Title
Linked Signer Can Self-Replace with Attacker-Controlled Address via `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner=true`, which permits the **current linked signer** (not just the subaccount owner) to authorize a new `LinkSigner` transaction. Because the signature check accepts either the subaccount owner or the current linked signer, a malicious or compromised linked signer can atomically replace themselves with an attacker-controlled address, permanently seizing signing authority over the victim's subaccount.

---

### Finding Description

In `EndpointTx.sol`, the `LinkSigner` branch of `processTransactionImpl` is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(...);
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner=true` calls `validateSignature`, which passes `getLinkedSigner(sender)` as the accepted alternate signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` then accepts the signature if the recovered address equals **either** the subaccount owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

There is no additional guard that restricts `LinkSigner` mutations to the subaccount owner only. The `linkedSigners` mapping is written unconditionally after the signature check passes: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

Once a linked signer is set, that signer can:
1. Sign a `LinkSigner{sender: victimSubaccount, signer: attacker2}` transaction.
2. Submit it through `submitTransactionsChecked` → `processTransactionImpl`.
3. Validation passes because `allowLinkedSigner=true` and the attacker's key matches `getLinkedSigner(victimSubaccount)`.
4. `linkedSigners[victimSubaccount]` is overwritten to `attacker2`.
5. `attacker2` can now sign `WithdrawCollateral`, `TransferQuote`, and all other `allowLinkedSigner=true` transactions on behalf of the victim.
6. The original owner retains no privileged recovery path beyond submitting a competing `LinkSigner` — but if the attacker front-runs or the sequencer processes the attacker's transaction first, the owner loses control.

This directly satisfies the Critical scope: **unauthorized transaction execution on behalf of a user without valid consent or signer ownership**, enabling unauthorized withdrawals and trades.

---

### Likelihood Explanation

Any party that has ever been granted linked-signer status (e.g., a trading bot, a delegated key, a third-party service) can execute this attack unilaterally. No leaked private keys, admin access, or social engineering is required — only possession of the currently-linked signer key, which is the normal operational state for many subaccounts.

---

### Recommendation

Restrict `LinkSigner` transactions so that only the **subaccount owner** (i.e., `address(uint160(bytes20(sender)))`) can authorize a signer change. Change the `allowLinkedSigner` argument for the `LinkSigner` branch from `true` to `false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

Apply the same fix to the slow-mode `LinkSigner` path, which currently uses `validateSender` (already correct there since it checks `msg.sender` against the subaccount owner), but audit any future paths for consistency.

---

### Proof of Concept

```solidity
// 1. Owner sets attacker1 as linked signer for victimSubaccount
// linkedSigners[victimSubaccount] = attacker1

// 2. attacker1 constructs and signs:
//    LinkSigner{ sender: victimSubaccount, signer: attacker2, nonce: currentNonce }
// Signature is produced by attacker1's private key.

// 3. Submit via submitTransactionsChecked([encodedLinkSignerTx])
//    -> processTransactionImpl
//    -> validateSignedTx(victimSubaccount, nonce, tx, sig, true)
//       -> validateSignature passes: recovered == attacker1 == getLinkedSigner(victimSubaccount)
//    -> linkedSigners[victimSubaccount] = attacker2   ✓ storage mutated

// 4. Assert: linkedSigners[victimSubaccount] == attacker2
// 5. attacker2 signs WithdrawCollateral for victimSubaccount -> drains funds
// 6. Owner's LinkSigner tx to recover must race the sequencer; if lost, control is gone
```
