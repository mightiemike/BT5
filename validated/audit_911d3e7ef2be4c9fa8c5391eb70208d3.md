### Title
Governance can set `MaxSupply` below current circulating supply, permanently halting the chain - (`File: x/inflation/types/params.go`)

### Summary

The `validateMaxSupply` function in `x/inflation/types/params.go` only checks that the new value is non-negative. It does not verify that the new `MaxSupply` is at or above the current circulating supply. A governance proposal that sets `MaxSupply` to any value below the live circulating supply will be accepted by `SetParams`, but will cause `BeginBlocker` to `panic` on the very next block, permanently halting the chain and locking all user funds.

### Finding Description

`validateMaxSupply` enforces only a lower bound of zero:

```go
// x/inflation/types/params.go
func validateMaxSupply(v any) error {
    maxSupply, ok := v.(sdkmath.Int)
    ...
    if maxSupply.IsNegative() {
        return fmt.Errorf("max supply cannot be negative: %s", maxSupply)
    }
    return nil
}
``` [1](#0-0) 

`SetParams` calls `params.Validate()`, which calls `validateMaxSupply`, and stores the params if it passes:

```go
// x/inflation/keeper/keeper.go
func (k Keeper) SetParams(ctx context.Context, params types.Params) error {
    if err := params.Validate(); err != nil {
        return err
    }
    ...
}
``` [2](#0-1) 

`BeginBlocker` then panics every block if `circulating_supply > MaxSupply`:

```go
// x/inflation/abci.go
if maxsupply.IsPositive() && totalsupply.GT(maxsupply) {
    panic(fmt.Sprintf("the total supply has exceeded the maximum supply: %s > %s", totalsupply, maxsupply))
}
``` [3](#0-2) 

The ADR explicitly documents this behavior: "The chain halt is intentional -- it prevents further minting beyond the cap. Recovery requires a governance upgrade or parameter change." [4](#0-3) 

The missing check is: `SetParams` never reads the current circulating supply from the bank module to verify that `new_MaxSupply >= current_circulating_supply`. This check only exists in `BeginBlocker`, which runs after the parameter is already committed to state.

### Impact Explanation

If `MaxSupply` is set to any positive value below the current circulating supply, `BeginBlocker` panics on the very next block. A Cosmos SDK `BeginBlocker` panic halts the entire chain. No transactions can be processed — including a governance proposal to raise `MaxSupply` back to a safe value. Recovery requires a coordinated emergency validator upgrade or hard fork. All user funds (bank balances, staked tokens, tiered rewards positions) are locked for the duration of the halt.

The corrupted invariant is: `MaxSupply >= current_circulating_supply` at the time of parameter application.

### Likelihood Explanation

There are two realistic trigger paths:

1. **Accidental governance mistake during inflation**: A proposer submits `MaxSupply = current_supply + small_buffer`. During the governance voting period (typically days), the Mint module mints new tokens every block. By the time the proposal executes, the circulating supply has grown past the new cap. The proposal passes validation (new value is positive), is stored, and the chain halts on the next block. This is a realistic, non-malicious scenario given the chain's active inflation.

2. **Governance attack**: An attacker with sufficient voting power submits a proposal setting `MaxSupply = 1`. The proposal passes `validateMaxSupply` (value is positive, non-negative), is stored, and the chain halts immediately.

The integration test `test_begin_blocker_halt_on_excess_supply` confirms this halt is reachable and reproducible via governance. [5](#0-4) 

### Recommendation

Add a live-state check inside `MsgUpdateParams` (or `SetParams`) that reads the current circulating supply and rejects any `MaxSupply` that is already exceeded:

```go
func (k msgServer) UpdateParams(goCtx context.Context, msg *types.MsgUpdateParams) (*types.MsgUpdateParamsResponse, error) {
    ...
    if msg.Params.MaxSupply.IsPositive() {
        totalSupply, denom, err := k.GetSupplyAndDenom(ctx)
        // subtract burned addresses
        ...
        if circulatingSupply.GT(msg.Params.MaxSupply) {
            return nil, fmt.Errorf("new max_supply %s is already exceeded by current circulating supply %s", msg.Params.MaxSupply, circulatingSupply)
        }
    }
    ...
}
```

This mirrors the recommendation in the external report: perform a sanity check for the sane value of the parameter when setting a new value.

### Proof of Concept

1. Query current circulating supply: `chain-maind q bank total --denom basecro` → e.g. `98700000000000000000`
2. Submit governance proposal: `MsgUpdateParams { MaxSupply: 1, BurnedAddresses: [...], DecayRate: 0.068 }`
3. Proposal passes `validateMaxSupply` (value `1` is positive and non-negative — no error)
4. `SetParams` stores the new params
5. Next block: `BeginBlocker` computes `circulating_supply = 98700000000000000000 - burned > 1` → `panic("the total supply has exceeded the maximum supply: ...")`
6. Chain halts. All subsequent governance transactions to fix `MaxSupply` are unprocessable.

The unit test `TestBeginBlocker_SupplyExceedsMax` in `x/inflation/abci_test.go` directly confirms step 5–6. [6](#0-5)

def test_begin_blocker_halt_on_excess_supply(cluster):
    """Test that chain halts when total supply exceeds max supply"""

    total_supply_rsp = query_command(cluster, BANK_MODULE, TOTALSUPPLY, DENOM)
    current_total_supply = int(total_supply_rsp[AMOUNT][AMOUNT])
    print("current_total_supply:", current_total_supply)

    # Get current max supply
    rsp = query_command(cluster, INFLATION_MODULE, PARAMS)["params"]
    assert "max_supply" in rsp

    new_max_supply = current_total_supply + 480000
    rsp["max_supply"] = str(new_max_supply)
    proposal_src = _create_max_supply_proposal(rsp)

    rsp = cluster.gov_propose_since_cosmos_sdk_v0_50(
        "community", "submit-proposal", proposal_src
    )
    assert rsp["code"] == 0, rsp["raw_log"]

    approve_proposal(cluster, rsp, msg=f",{MSG}")

    # Panic will not cause supervisor to stop,
    # so we poll node0's log for the panic string instead.
    log_file = f"{cluster.home(0)}/../node0.log"
    timeout_seconds = 60
    start_time = time.time()
    while True:
        try:
            with open(log_file, "r") as f:
                log_content = f.read()
        except OSError:
            log_content = ""
        if ERROR in log_content:
            print("Chain has been halted — panic detected in log")
            return
        if time.time() - start_time > timeout_seconds:
            assert False, (
                f"Timeout after {timeout_seconds}s waiting for halt panic "
                f"in {log_file}"
            )
        time.sleep(1)

// TestBeginBlocker_SupplyExceedsMax verifies that the chain panics when the
// total supply exceeds the maximum supply.
func TestBeginBlocker_SupplyExceedsMax(t *testing.T) {
	app := testutil.Setup(false, nil)
	ctx := app.BaseApp.NewContext(false).WithBlockHeader(tmproto.Header{ChainID: testutil.ChainID})

	totalSupply, _, err := app.InflationKeeper.GetSupplyAndDenom(ctx)
	require.NoError(t, err)

	// Set max_supply 1 below total supply
	params := types.DefaultParams()
	params.MaxSupply = totalSupply.Sub(math.NewInt(1))
	err = app.InflationKeeper.SetParams(ctx, params)
	require.NoError(t, err)

	require.PanicsWithValue(t,
		"the total supply has exceeded the maximum supply: "+totalSupply.String()+" > "+params.MaxSupply.String(),
		func() {
			err := inflation.BeginBlocker(ctx, app.InflationKeeper)
			require.NoError(t, err)
		},
	)
}
