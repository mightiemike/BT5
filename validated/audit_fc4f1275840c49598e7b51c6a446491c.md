### Title
Silent Zero-Shift in `updateTierFeeRates` Permanently Disables Configured Fee Rates for Tiers ≥ 128 — (`core/contracts/OffchainExchange.sol`)

### Summary
`nonDefaultFeeTierMask` is a `uint128`. The expression `uint128(1) << txn.tier` evaluates to `0` in Solidity 0.8 whenever `txn.tier >= 128` (shift amount ≥ type width). As a result, the mask bit for any tier ≥ 128 is never set, and `getTierFeeRateX18` always falls through to the hardcoded 2 bps default, silently ignoring whatever rate was written into `feeRates[tier][productId]`.

### Finding Description

**State variable declaration** — `nonDefaultFeeTierMask` is `uint128`: [1](#0-0) 

**Write path** — `updateTierFeeRates` correctly stores the rates in `feeRates`, then attempts to mark the tier as non-default: [2](#0-1) 

For `txn.tier >= 128`, Solidity 0.8 evaluates `uint128(1) << txn.tier` as `0` (the EVM shift opcode returns 0 when the shift amount ≥ the operand's bit-width). The `|= 0` is a no-op, so the bit is never set.

**Read path** — `getTierFeeRateX18` gates the stored rate behind the mask: [3](#0-2) 

Because the bit for tier ≥ 128 is never set, the condition is always `false`, and the function always returns `{makerRateX18: 0, takerRateX18: 200_000_000_000_000}` (2 bps) regardless of what was configured.

**Fee application** — `getUserFeeRateWithBuilder` calls `getTierFeeRateX18` and uses its result directly for every matched order: [4](#0-3) 

### Impact Explanation
Every user whose `feeTier` is ≥ 128 pays exactly 2 bps taker / 0 maker, regardless of the rate the operator configured for that tier:

- If the tier was configured with a **higher** rate (e.g., a penalty or high-volume tier), those users pay less than required — direct fee revenue loss for the protocol.
- If the tier was configured with a **lower** rate (e.g., a VIP/market-maker rebate), those users are overcharged — a financial harm to the user.

In both cases the invariant "every tier with a configured fee rate must have that rate applied during order matching" is broken. The stored `feeRates[tier][productId]` values are dead storage that can never be read.

### Likelihood Explanation
The bug is latent but deterministic: it fires for every order matched by any user whose tier is ≥ 128. Tier assignment is admin-controlled (`updateFeeTier` requires `msg.sender == clearinghouse`), so an external attacker cannot self-assign a high tier. However, the protocol itself is the victim — any operator action that configures rates for tiers ≥ 128 silently has no effect, and the misconfiguration is undetectable on-chain without off-chain monitoring.

### Recommendation
Change the mask write to use `uint256` arithmetic and widen `nonDefaultFeeTierMask` to `uint256`, or cap `txn.tier` to `< 128` with a `require`. The minimal fix:

```solidity
// Replace uint128 with uint256
uint256 internal nonDefaultFeeTierMask;

// In updateTierFeeRates:
nonDefaultFeeTierMask |= uint256(1) << txn.tier;

// In getTierFeeRateX18:
if (nonDefaultFeeTierMask & (uint256(1) << tier) != 0) {
```

### Proof of Concept

```solidity
// Hardhat / foundry test (unmodified contract)
uint32 tier = 128;
// Admin configures a 10 bps rate for tier 128
offchainExchange.updateTierFeeRates(IEndpoint.UpdateTierFeeRates({
    tier: tier,
    productId: 0,          // QUOTE_PRODUCT_ID → applies to all products
    makerRateX18: 0,
    takerRateX18: 1_000_000_000_000_000  // 10 bps
}));

// nonDefaultFeeTierMask is still 0 because uint128(1) << 128 == 0
assert(offchainExchange.nonDefaultFeeTierMask() == 0);  // passes

// getTierFeeRateX18 returns default 2 bps, not 10 bps
IOffchainExchange.FeeRates memory r = offchainExchange.getTierFeeRateX18(tier, 1);
assert(r.takerRateX18 == 200_000_000_000_000);  // 2 bps — passes, proving the bug
assert(r.takerRateX18 == 1_000_000_000_000_000); // 10 bps — FAILS
```
