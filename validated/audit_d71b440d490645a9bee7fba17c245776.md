### Title
`verify_transaction_inclusion_v2` Permanently Panics for Valid Single-Transaction Block Proofs Due to Inconsistent Empty-Proof Handling — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` implicitly accepts an empty merkle proof (returning the transaction hash unchanged — semantically correct for a single-transaction block). `verify_transaction_inclusion` explicitly rejects an empty proof with a hard panic. `verify_transaction_inclusion_v2` uses both in sequence, creating a cross-module desynchronization: the coinbase proof step passes for a valid single-tx block, but the subsequent call to `verify_transaction_inclusion` always panics. Any downstream consumer of the recommended v2 API is permanently broken for this class of blocks.

---

### Finding Description

`compute_root_from_merkle_proof` iterates over the proof slice. When the slice is empty the loop body never executes and the function returns `transaction_hash` unchanged:

```rust
// merkle-tools/src/lib.rs:39-51
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;
    for proof_hash in merkle_proof {   // ← never entered when proof is empty
        ...
    }
    current_hash   // returns transaction_hash unchanged
}
```

This is semantically correct: for a block containing exactly one transaction, the merkle root **is** the transaction hash, so an empty proof is the only valid proof.

`verify_transaction_inclusion_v2` relies on this behaviour for its coinbase-proof guard:

```rust
// contract/src/lib.rs:358-365
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
```

For a single-tx block `coinbase_tx_id == merkle_root`, so the guard passes with `coinbase_merkle_proof = []`. The length invariant `merkle_proof.len() == coinbase_merkle_proof.len()` (line 349) then forces `merkle_proof = []` as well.

The function then delegates to the deprecated `verify_transaction_inclusion`:

```rust
// contract/src/lib.rs:367-368
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
```

Inside that function, the first thing checked after the confirmations and block-lookup steps is:

```rust
// contract/src/lib.rs:315
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

This unconditionally panics when `merkle_proof` is empty, even though the empty proof is the only mathematically correct proof for a single-tx block. The call never returns `true` or `false`; it always aborts.

---

### Impact Explanation

Any NEAR contract or off-chain caller that invokes `verify_transaction_inclusion_v2` to verify a transaction (including the coinbase) in a single-transaction Bitcoin block will receive a panic instead of a boolean result. Because `verify_transaction_inclusion` is marked `#[deprecated]` and callers are directed to use v2, the recommended API is permanently broken for this class of blocks. Downstream contracts that gate actions on a `true` result (e.g., cross-chain bridges, payment proofs) cannot process transactions from single-tx blocks at all.

---

### Likelihood Explanation

Single-transaction blocks are common in Bitcoin: every block that contains only the coinbase reward (no user transactions) falls into this category. Early mainnet blocks, low-fee periods, and empty-block mining strategies all produce them. A relayer submitting such a block to the contract creates a permanent verification failure for any proof against that block — no configuration change or re-submission can fix it without a contract upgrade.

---

### Recommendation

1. Remove the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard from `verify_transaction_inclusion`, or replace it with a check that allows an empty proof only when `tx_index == 0` and the computed root equals the transaction hash (i.e., the single-tx case).
2. Alternatively, add the same guard directly in `verify_transaction_inclusion_v2` before the coinbase check, and explicitly handle the single-tx case there, so the two code paths are consistent.
3. Add a test case for a single-transaction block to prevent regression.

---

### Proof of Concept

**Setup**: A Bitcoin block `B` contains exactly one transaction `T` (the coinbase). Its `merkle_root == hash(T)`.

**Attacker/user call** to `verify_transaction_inclusion_v2`:
```
tx_id                = hash(T)
tx_block_blockhash   = hash(B)   // block is in the mainchain
tx_index             = 0
merkle_proof         = []        // only valid proof for a single-tx block
coinbase_tx_id       = hash(T)
coinbase_merkle_proof = []
confirmations        = 1
```

**Execution trace**:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` → `0 == 0` ✓ [1](#0-0) 
2. `compute_root_from_merkle_proof(hash(T), 0, &[])` returns `hash(T)` == `merkle_root` ✓ [2](#0-1) 
3. `verify_transaction_inclusion` is called with `merkle_proof = []` [3](#0-2) 
4. `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` → **PANIC** [4](#0-3) 

The call never returns. Every subsequent attempt with the same block produces the same panic. The inconsistency between `compute_root_from_merkle_proof` (which treats an empty proof as valid) and `verify_transaction_inclusion` (which treats it as always invalid) is the root cause. [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
