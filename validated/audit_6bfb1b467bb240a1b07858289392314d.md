### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Breaking Transaction Hash and Consensus Conversion - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` silently converts an `AllResources` variant (with `l2_gas = 0` and `l1_data_gas = 0`) into the `L1Gas` variant. Because `get_tip_resource_bounds_hash` produces a structurally different hash preimage for the two variants (2 vs. 3 resource elements), and because `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` hard-rejects the `L1Gas` variant, a transaction that the gateway accepts and hashes under `AllResources` semantics cannot be reconstructed by a consensus validator that receives it over P2P. The result is a hash-domain split and a hard conversion failure that breaks consensus for any such transaction.

---

### Finding Description

**Step 1 – Gateway accepts the transaction under `AllResources` semantics.**

`RpcInvokeTransactionV3` and `InternalRpcInvokeTransactionV3` both declare `resource_bounds: AllResourceBounds` (never `ValidResourceBounds`). Any transaction submitted via the RPC gateway is therefore always treated as `AllResources`. [1](#0-0) [2](#0-1) 

**Step 2 – Hash is computed with the 3-element `AllResources` preimage.**

`get_tip_resource_bounds_hash` branches on the `ValidResourceBounds` variant. For `AllResources` it appends a third element (`L1_DATA_GAS`) to the hash chain; for `L1Gas` it does not. The two variants therefore produce different Poseidon digests even when `l2_gas = 0` and `l1_data_gas = 0`. [3](#0-2) 

The hash is committed into `InternalRpcTransaction.tx_hash` at conversion time: [4](#0-3) 

**Step 3 – Protobuf serialization preserves all three fields.**

When `ValidResourceBounds::L1Gas` is serialized to protobuf, `l1_data_gas` is explicitly set to a zero `ResourceLimits` message: [5](#0-4) 

**Step 4 – Protobuf deserialization silently downgrades `AllResources{l2=0, l1_data=0}` to `L1Gas`.**

The deserializer applies a heuristic: if both `l2_gas` and `l1_data_gas` are zero after parsing, it returns `ValidResourceBounds::L1Gas`. A transaction originally submitted as `AllResourceBounds{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` is therefore reconstructed as `L1Gas(X)` on the receiving side. [6](#0-5) 

The TODO comment on line 426 acknowledges this is a known transitional state, but the downgrade is unconditional and permanent for any transaction with zero L2/data-gas bounds.

**Step 5 – `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` hard-rejects `L1Gas`.**

When the consensus validator reconstructs the `RpcInvokeTransactionV3` from the deserialized `InvokeTransactionV3`, it explicitly errors on any non-`AllResources` variant: [7](#0-6) 

This causes `convert_consensus_tx_to_internal_consensus_tx` to return an error for the transaction, which propagates as a `HandledProposalPart::Failed` in the proposal validation loop: [8](#0-7) 

**Step 6 – Hash divergence even if the conversion were patched.**

Even if the `TryFrom` conversion were relaxed, the hash recomputed on the validator side would differ from the hash committed by the proposer. The proposer computed H1 = `poseidon(tip, L1_GAS_felt, L2_GAS_felt, L1_DATA_GAS_felt)` (3 elements); the validator would compute H2 = `poseidon(tip, L1_GAS_felt, L2_GAS_felt)` (2 elements). H1 ≠ H2, so signature verification against H2 would fail for a signature produced over H1. [9](#0-8) 

---

### Impact Explanation

Any unprivileged user can submit a valid-looking `RpcInvokeTransactionV3` with `l2_gas = 0` and `l1_data_gas = 0` (a common pattern for pre-0.13.3 style transactions expressed in the new API). The gateway accepts it, assigns it hash H1, and the proposer includes it in a block. When the block is broadcast to validators via P2P consensus, the protobuf round-trip converts the transaction's resource bounds to `L1Gas`, causing a hard conversion error in `convert_consensus_tx_to_internal_consensus_tx`. The validator marks the proposal as `Failed`, breaking consensus for that block. This matches: **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

---

### Likelihood Explanation

The trigger condition (`l2_gas = 0` and `l1_data_gas = 0`) is the natural default for any user who only cares about L1 gas and leaves the other bounds at zero. The gateway's stateless validator explicitly accepts such transactions (test cases `valid_l1_gas` confirm this). The condition is therefore reachable by any ordinary user without any special privilege. [10](#0-9) 

---

### Recommendation

1. **Remove the `L1Gas` downgrade heuristic from protobuf deserialization.** When deserializing a `protobuf::ResourceBounds` that was produced by a node running the current code (which always serializes `AllResources` for RPC transactions), always reconstruct `AllResources`, even when `l2_gas` and `l1_data_gas` are zero. The TODO comment on line 426 should be resolved by asserting `l1_data_gas` is present rather than defaulting it.

2. **Canonicalize the hash domain.** The `get_tip_resource_bounds_hash` function must produce the same output for semantically equivalent resource bounds regardless of which `ValidResourceBounds` variant is used. Either always include `l1_data_gas` in the hash (even when zero), or gate the 3-element form strictly on a protocol version flag rather than on the runtime variant.

3. **Add a round-trip invariant test** asserting that `AllResourceBounds{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` survives a protobuf serialize/deserialize cycle as `AllResources`, not `L1Gas`.

---

### Proof of Concept

```
1. Construct RpcInvokeTransactionV3 with:
     resource_bounds = AllResourceBounds {
         l1_gas:      ResourceBounds { max_amount: 1000, max_price_per_unit: 1 },
         l2_gas:      ResourceBounds { max_amount: 0,    max_price_per_unit: 0 },
         l1_data_gas: ResourceBounds { max_amount: 0,    max_price_per_unit: 0 },
     }

2. Submit via gateway RPC.
   → Gateway calls InternalRpcInvokeTransactionV3::resource_bounds()
     which returns ValidResourceBounds::AllResources(...)
   → get_tip_resource_bounds_hash produces H1 (3-element chain)
   → Transaction stored in mempool with tx_hash = H1.

3. Proposer includes transaction in block and broadcasts ConsensusTransaction
   via P2P protobuf.

4. Protobuf serialization (From<ValidResourceBounds> for protobuf::ResourceBounds):
     l1_gas      = Some(ResourceLimits { max_amount: 1000, ... })
     l2_gas      = Some(ResourceLimits { max_amount: 0,    ... })   ← zero
     l1_data_gas = Some(ResourceLimits { max_amount: 0,    ... })   ← zero

5. Protobuf deserialization (TryFrom<protobuf::ResourceBounds> for ValidResourceBounds):
     l1_data_gas.is_zero() && l2_gas.is_zero()  →  true
     → returns ValidResourceBounds::L1Gas(ResourceBounds { max_amount: 1000, ... })

6. Validator calls TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3:
     match value.resource_bounds {
         ValidResourceBounds::AllResources(bounds) => bounds,  // NOT taken
         _ => return Err(OutOfRange { "resource_bounds" }),    // TAKEN
     }
   → Error returned.

7. convert_consensus_tx_to_internal_consensus_tx returns Err.
   → validate_proposal returns HandledProposalPart::Failed.
   → Validator rejects the block proposal.
   → Consensus stalls for any block containing such a transaction.

Hash divergence (independent of conversion failure):
   H1 = poseidon(tip, concat(L1_GAS, 1000, 1), concat(L2_GAS, 0, 0), concat(L1_DATA, 0, 0))
   H2 = poseidon(tip, concat(L1_GAS, 1000, 1), concat(L2_GAS, 0, 0))
   H1 ≠ H2  →  signature over H1 fails verification against H2.
```

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
