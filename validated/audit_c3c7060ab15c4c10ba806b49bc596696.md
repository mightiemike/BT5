Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates the immediate caller (router) instead of the economic actor, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap()` binds to `msg.sender` — the immediate caller of the pool. When any user routes through `MetricOmmSimpleRouter`, `sender` equals the router's address. A pool admin who allowlists the router so that legitimate users can access the periphery inadvertently grants every address on-chain the ability to swap against the curated pool, fully nullifying the per-user gate. LP principal in institutional or KYC-gated pools is directly at risk.

## Finding Description

**Root cause — `sender` binding in the pool:**

`MetricOmmPool.swap()` calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient, zeroForOne, amountSpecified, priceLimitX64,
    packedSlot0Initial, bidPriceX64, askPriceX64, extensionData
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as the `sender` argument to every configured extension: [2](#0-1) 

**Root cause — extension checks `sender` not the end user:**

`SwapAllowlistExtension.beforeSwap` evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool (correct) and `sender` is the router when the user routes through `MetricOmmSimpleRouter`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Router call path:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` to the pool:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` ignores the caller entirely and checks `owner` — the explicit economic actor passed as a separate parameter by the pool:

```solidity
function beforeAddLiquidity(address /*caller*/, address owner, ...)
    ...
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
``` [6](#0-5) 

The `beforeSwap` interface has no equivalent `owner` / `originator` field — only `sender` (the immediate caller) and `recipient`. `SwapAllowlistExtension` has no way to recover the end user's identity from the arguments it receives. [7](#0-6) 

**Test confirmation:**

The existing test suite confirms the binding: the allowlist is set for `address(callers[0])` (the `TestCaller` wrapper that directly calls the pool), not for `users[0]` (the human). The swap succeeds only because the wrapper — the immediate pool caller — is allowlisted:

```solidity
swapExtension.setAllowedToSwap(address(pool), address(callers[0]), true);
_swap(0, users[0], false, int128(1000), type(uint128).max);
``` [8](#0-7) 

## Impact Explanation

**High.** A curated pool (KYC-gated, institutional-only) that allowlists the router so that legitimate users can access the standard periphery is silently open to every address on-chain. Any unauthorized user can call `router.exactInput*(...)` targeting the curated pool and the extension will evaluate `allowedSwapper[pool][router] → true`, executing the swap. LP principal is directly at risk from uninvited counterparties that the allowlist was specifically designed to exclude. This matches the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" allowed impacts.

## Likelihood Explanation

**Medium.** Allowlisting the router is the natural, expected administrative action for any pool admin who wants allowlisted users to use the standard periphery. The admin has no obvious signal that this action nullifies the per-user gate. The asymmetry with `DepositAllowlistExtension` (which correctly gates `owner`) makes the flaw non-obvious. Any unprivileged user can exploit it with a single router call once the router is allowlisted.

## Recommendation

Align `SwapAllowlistExtension` with the deposit extension's design. The pool's `swap()` interface currently provides no separate `swapOwner` / `originator` field analogous to `owner` in `addLiquidity`. The fix requires one of:

1. **Protocol-level fix (preferred):** Add an explicit `originator` address to the `beforeSwap` hook signature (and `IMetricOmmExtensions`) that the pool populates with `tx.origin` or that the router populates via `extensionData`. `SwapAllowlistExtension` then checks `originator` instead of `sender`.
2. **Extension-level workaround:** Require that allowlisted users call `pool.swap()` directly (never through the router) and document that the router must never be allowlisted on curated pools — but this breaks the intended periphery UX and is not enforceable on-chain.

## Proof of Concept

1. Pool admin deploys a curated pool with `SwapAllowlistExtension` attached.
2. Admin calls `setAllowedToSwap(pool, userA, true)` for each legitimate user.
3. Admin calls `setAllowedToSwap(pool, router, true)` so that `userA` can use `MetricOmmSimpleRouter`.
4. Unauthorized `userC` (not in the allowlist) calls `router.exactInputSingle({pool: curatedPool, ...})`.
5. Router calls `pool.swap(...)` — `msg.sender` to the pool is the router address.
6. Pool calls `extension.beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router] → true`.
8. Swap executes; `userC` trades against the curated pool's LPs with zero allowlist enforcement.

Minimal Foundry test outline:
```solidity
swapExtension.setAllowedToSwap(address(pool), address(router), true);
// userC is NOT allowlisted
vm.prank(userC);
router.exactInputSingle(ExactInputSingleParams({pool: address(pool), ...}));
// swap succeeds — allowlist bypassed
```
