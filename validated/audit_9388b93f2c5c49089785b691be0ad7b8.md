### Title
Stale `digestToSubaccount` Entry After Isolated Subaccount Close Allows Funds to Be Permanently Sent to `FEES_ACCOUNT` — (`core/contracts/OffchainExchange.sol`)

### Summary

When an isolated subaccount is closed via `_tryCloseIsolatedSubaccount`, `parentSubaccounts[iso]` is zeroed but `digestToSubaccount[digest]` is never cleared. A subsequent `matchOrders` call referencing the same digest redirects the fill to the now-parentless iso subaccount. When that position is later closed, `_tryCloseIsolatedSubaccount` transfers the iso subaccount's remaining balances to `parentSubaccounts[iso] == bytes32(0) == FEES_ACCOUNT`, permanently destroying the trader's funds.

### Finding Description

The lifecycle of an isolated subaccount involves two independent mappings:

- `digestToSubaccount[digest]` — maps an order digest to the iso subaccount it created
- `parentSubaccounts[iso]` — maps the iso subaccount back to its parent

`_tryCloseIsolatedSubaccount` (lines 160–208) zeroes `parentSubaccounts[iso]` at line 204 but **never touches `digestToSubaccount[digest]`**. This creates a dangling pointer.

`matchOrders` (lines 673–678) checks `digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)` and, if true, unconditionally sets `taker.order.sender = digestToSubaccount[...]`. There is no check that `parentSubaccounts[that iso subaccount] != bytes32(0)`, i.e., no check that the iso subaccount is still live.

`_tryCloseIsolatedSubaccount` (lines 172–200) reads `parent = parentSubaccounts[subaccount]` and transfers all `vQuoteBalance` and spot quote balance to `parent` with no guard for `parent == bytes32(0)`. Since `FEES_ACCOUNT = bytes32(0)`, those funds are credited to the protocol fee sink and are unrecoverable by the trader.

### Impact Explanation

Permanent loss of trader collateral. Any quote balance or vQuote balance remaining in the iso subaccount at the time of the second close is transferred to `FEES_ACCOUNT` (`bytes32(0)`) and cannot be recovered. The trader's parent subaccount receives nothing.

### Likelihood Explanation

The scenario requires a specific transaction ordering: `tryCloseIsolatedSubaccount` must be processed before a pending `matchOrders` referencing the same digest. This is realistic when:

- A position is closed via liquidation (which calls `tryCloseIsolatedSubaccount` through `ClearinghouseLiq`) while a `matchOrders` transaction for the same order is still queued in the sequencer.
- The sequencer processes the close transaction first, then the fill.

No sequencer compromise is required — this is normal sequencer operation. The trader does not need to take any special action; the vulnerability can be triggered by the natural race between liquidation and pending fill transactions.

### Recommendation

Two fixes are needed, either independently sufficient:

1. **Clear `digestToSubaccount[digest]` in `_tryCloseIsolatedSubaccount`**: When closing an iso subaccount, iterate over or track the associated digest and set `digestToSubaccount[digest] = bytes32(0)`. This prevents stale redirects in `matchOrders`.

2. **Guard against `parent == bytes32(0)` in `_tryCloseIsolatedSubaccount`**: Add an early return if `parentSubaccounts[subaccount] == bytes32(0)`, preventing funds from being sent to `FEES_ACCOUNT`:
   ```solidity
   bytes32 parent = parentSubaccounts[subaccount];
   if (parent == bytes32(0)) return; // already closed, no parent to return funds to
   ```

3. **Guard in `matchOrders`**: Before redirecting `taker.order.sender` to the iso subaccount, verify `parentSubaccounts[digestToSubaccount[digest]] != bytes32(0)`.

### Proof of Concept

```
1. Alice calls createIsolatedSubaccount for productId P, order digest D.
   State: digestToSubaccount[D] = iso, parentSubaccounts[iso] = alice

2. Alice's position is liquidated (balance.amount → 0).

   _tryCloseIsolatedSubaccount: transfers remaining quote to alice, sets parentSubaccounts[iso] = bytes32(0).

   vQuoteBalance and spot quote balance of iso are transferred to FEES_ACCOUNT.
   Alice's funds are permanently destroyed.
