### Title
`op_mod` Hardcoded to Wrong Cost Constants, Causing Systematic Undercharging — (`File: src/more_ops.rs`)

---

### Summary

`op_mod` (opcode 61) and its Malachite variant `op_mod_malachite` both compute their execution cost using `DIV_BASE_COST` (988) and `DIV_COST_PER_BYTE` (4) — the constants belonging to the `div` operator — instead of `DIVMOD_BASE_COST` (1116) and `DIVMOD_COST_PER_BYTE` (6). Because `mod_floor()` internally performs the same full floor-division as `div_mod_floor()`, the `mod` operator is systematically undercharged by up to ~30% at maximum input sizes. An attacker can craft CLVM programs that use opcode 61 with large integer arguments to consume more real computation than the cost limit is supposed to allow.

---

### Finding Description

In `src/more_ops.rs`, the cost constants are defined as:

```
DIVMOD_BASE_COST: Cost = 1116
DIVMOD_COST_PER_BYTE: Cost = 6

DIV_BASE_COST: Cost = 988
DIV_COST_PER_BYTE: Cost = 4
```

`op_divmod` (opcode 20) correctly uses the `DIVMOD` constants: [1](#0-0) 

`op_div` (opcode 19) correctly uses the `DIV` constants: [2](#0-1) 

But `op_mod` (opcode 61) incorrectly uses the `DIV` constants: [3](#0-2) 

And `op_mod_malachite` makes the same mistake: [4](#0-3) 

The root cause is that `mod_floor()` (called by `op_mod`) internally delegates to `div_mod_floor()` (the same function called by `op_divmod`) and simply discards the quotient. The underlying computation is identical. The `DIVMOD` constants exist precisely because this operation was benchmarked to be more expensive than plain `div_floor()`. Wiring `op_mod` to the cheaper `DIV` constants is the same class of error as the governance report's `votingPeriod = 3`: a hardcoded parameter set to the wrong value, silently breaking the invariant the system depends on.

The operator dispatch table confirms opcode 61 routes to `op_mod`: [5](#0-4) 

---

### Impact Explanation

The CLVM cost model is the sole DoS protection for Chia full nodes. Every operator must charge cost that accurately reflects its real CPU time. When `op_mod` is undercharged, a program can perform more real computation than the declared `max_cost` budget permits.

Concrete undercharge at maximum allowed input sizes (dividend = 256 bytes, divisor = 1024 bytes, total = 1280 bytes):

| | Base | Per-byte (×1280) | Total |
|---|---|---|---|
| Charged (DIV constants) | 988 | 4 × 1280 = 5120 | **6,108** |
| Should be (DIVMOD constants) | 1116 | 6 × 1280 = 7680 | **8,796** |
| Undercharge | | | **2,688 (~30%)** |

A program that loops `op_mod` calls with large inputs can exhaust a node's CPU budget while the cost counter stays below `max_cost`. This is a consensus-layer resource-accounting divergence: a program that should be rejected as too expensive is accepted.

---

### Likelihood Explanation

Opcode 61 (`mod`) is a live, dispatched operator in `ChiaDialect` with no guard flag required. Any attacker who can submit a CLVM program to the Chia mempool — which is the normal, permissionless transaction submission path — can trigger this path. The input is fully attacker-controlled (the two integer atoms). No special privileges, social engineering, or compromised nodes are required.

---

### Recommendation

Replace the two wrong constant references in `op_mod` and `op_mod_malachite` with the correct `DIVMOD` constants:

```rust
// op_mod (line 775)
let cost = DIVMOD_BASE_COST + ((a0_len + a1_len) as Cost) * DIVMOD_COST_PER_BYTE;

// op_mod_malachite (line 800)
let cost = DIVMOD_BASE_COST + ((a0_len + a1_len) as Cost) * DIVMOD_COST_PER_BYTE;
```

This aligns `op_mod`'s cost with the actual computation it performs, consistent with how `op_divmod` is charged.

### Proof of Concept

The following CLVM program invokes opcode 61 (`mod`) with a 256-byte dividend and a 1024-byte divisor. The cost charged will be 6,108 + malloc overhead, but the actual computation is equivalent to a `divmod` call that should cost 8,796 + malloc overhead. Repeating this inside a loop (using `op_apply` / recursion) allows a program to exceed the intended CPU budget while staying within the declared cost limit.

```
; Pseudocode: (mod <256-byte-atom> <1024-byte-atom>)
; Opcode 61 = 0x3d
; Attacker submits this as a coin puzzle or generator program.
; Cost reported by run_program: 6108 + malloc
; Cost that should be reported:  8796 + malloc
; Delta per call: 2688 cost units (~30% free computation)
```

The fuzz harness at `fuzz/fuzz_targets/divmod.rs` already exercises `op_mod` alongside `op_div` and `op_divmod` for correctness, but does not assert cost parity between `op_mod` and `op_divmod`. [6](#0-5)
