Audit Report

## Title
SwapAllowlistExtension Allowlist Bypass via Router Identity Loss — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which resolves to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of `pool.swap()`, so the extension checks the router's allowlist status rather than the actual user's. If the pool admin allowlists the router — a natural operational step — any unpermissioned user can bypass the swap allowlist entirely by routing through the public router.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs its identity check as:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension is called by the pool via `ExtensionCalling`), and `sender` is the first argument — the value the pool passes from its own `msg.sender`. [1](#0-0) 

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly from the router contract, making the router itself `msg.sender` of that call: [3](#0-2) 

The same identity-loss applies to `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165) — all call `pool.swap()` directly from the router contract with no mechanism to forward the original caller's identity. [4](#0-3) 

The result: when `bob` routes through the router, the extension sees `sender = router address` and checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][bob]`. If the router is allowlisted, the check passes regardless of `bob`'s allowlist status.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` in `BEFORE_SWAP_ORDER` is intended to restrict swaps to a specific set of addresses (e.g., KYC-verified users, protocol partners, whitelisted market makers). Allowlisting the router is a natural operational requirement for users to access slippage protection and multi-hop routing. Once the router is allowlisted, the access-control invariant is completely broken: any address can execute swaps in the restricted pool by routing through `MetricOmmSimpleRouter`. This constitutes an admin-boundary break where an unprivileged path bypasses a pool admin's access control, and can result in unauthorized users draining liquidity from a permissioned pool or silently bypassing compliance/KYC controls — a direct loss of fund-protection guarantees for the pool.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. A pool admin who wants users to access slippage protection or multi-hop routing has no choice but to allowlist the router. The bypass is then reachable by any unpermissioned user with no special setup, no privileged access, and no non-standard tokens. The only precondition — the router being allowlisted — is a natural and expected operational action, making this highly likely to be triggered in practice.

## Recommendation
`SwapAllowlistExtension.beforeSwap` should gate on the economically relevant actor, not the immediate caller. Two complementary fixes:

1. **Encode the original user in `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`. The extension decodes and verifies this address against the allowlist when `sender` is a known trusted router.
2. **Trusted router registry**: Maintain a registry of trusted periphery contracts in the extension. When `sender` is a trusted router, require `extensionData` to contain the original user address and check it against `allowedSwapper`.

The simplest safe default is to **not allowlist the router address** and explicitly document that pools with `SwapAllowlistExtension` must be accessed directly, not through the router — but this removes router usability for permissioned pools and must be prominently documented.

## Proof of Concept
```
Setup:
  1. Deploy pool with SwapAllowlistExtension in BEFORE_SWAP_ORDER.
  2. Pool admin: setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  3. Pool admin: setAllowedToSwap(pool, router, true)  // router allowlisted for UX
  4. bob is NOT allowlisted.

Direct call (blocked correctly):
  bob → pool.swap(...)
    → _beforeSwap(sender=bob, ...)
    → allowedSwapper[pool][bob] == false → revert NotAllowedToSwap ✓

Router bypass (allowlist defeated):
  bob → MetricOmmSimpleRouter.exactInputSingle(pool=pool, ...)
    → pool.swap(...)  [msg.sender = router]
    → _beforeSwap(sender=router, ...)
    → allowedSwapper[pool][router] == true → passes ✓
    → swap executes for bob despite bob not being allowlisted ✗
```

Root cause: `sender` in `SwapAllowlistExtension.beforeSwap` is the router address when routing through `MetricOmmSimpleRouter`. The allowlist check passes because the router is allowlisted, not because `bob` is. The extension has no mechanism to receive or verify the true initiator. [5](#0-4) [6](#0-5)
