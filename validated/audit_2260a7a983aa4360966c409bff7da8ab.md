### Title
`verify_transaction_inclusion_v2` coinbase anchor can be forged with an internal Merkle node, bypassing the 64-byte proof-forgery mitigation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to fix the 64-byte transaction Merkle proof forgery vulnerability. Its mitigation relies on a caller-supplied `coinbase_tx_id` being the actual coinbase leaf. However, `coinbase_tx_id` is never validated to be a leaf node — an attacker can supply an internal Merkle tree node as `coinbase_tx_id` with a correspondingly shorter `coinbase_merkle_proof`, satisfying the coinbase check while also shortening the required `merkle_proof` length. This re-enables the exact 64-byte forgery attack that v2 was designed to prevent.

---

### Finding Description

`verify_transaction_inclusion_v2` performs three checks before returning a result:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — enforces equal proof depths.
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == header.block_header.merkle_root` — validates the coinbase proof.
3. Delegates to `verify_transaction_inclusion` which validates the tx proof. [1](#0-0) 

The mitigation is only sound if `coinbase_tx_id` is the actual coinbase transaction hash — a leaf at depth 0 in the Merkle tree. If `coinbase_tx_id` is an internal node at depth `d`, then a valid `coinbase_merkle_proof` of length `tree_depth - d` (not `tree_depth`) will still satisfy check (2), because `compute_root_from_merkle_proof` is a pure path-traversal function with no leaf/node distinction: [2](#0-1) 

Since check (1) enforces `merkle_proof.len() == coinbase_merkle_proof.len()`, the attacker's `merkle_proof` for the fake `tx_id` is also only `tree_depth - d` long. This is exactly the shortened-proof condition that enables the 64-byte internal-node forgery.

`coinbase_tx_id` is a fully caller-controlled field in `ProofArgsV2` with no on-chain validation: [3](#0-2) 

---

### Impact Explanation

An unprivileged NEAR caller can make `verify_transaction_inclusion_v2` return `true` for a transaction that is **not** included in the block. Any downstream contract or bridge that trusts this return value to authorize a payment, unlock funds, or confirm a cross-chain event will be deceived. The corrupted proof result is the direct output of the public verification API.

---

### Likelihood Explanation

The entry point is fully public and permissionless — no staking, no role, no privileged key is required. The attacker only needs knowledge of a real confirmed block's Merkle tree (publicly available from any Bitcoin full node). Constructing the forged `coinbase_tx_id` (an internal node) and its matching shorter proof is straightforward arithmetic on known block data.

---

### Recommendation

The coinbase anchor must be validated to be a leaf, not an internal node. Since the contract cannot store per-block transaction lists, the standard approach is to enforce that the coinbase proof length equals `ceil(log2(tx_count))` — but `tx_count` is not stored either. The practical fix is to require that `coinbase_tx_id` is provided alongside a full-depth proof that starts from the actual coinbase txid, and to reject any `coinbase_merkle_proof` whose length is inconsistent with the block's known transaction count, or alternatively store the expected proof depth per block header at submission time.

A minimal defensive change: reject calls where `coinbase_merkle_proof` is empty (a single-tx block where the coinbase IS the merkle root is a degenerate case that should be handled separately), and document that callers must supply the real coinbase txid retrievable from the Bitcoin node — but note this is a social/off-chain constraint, not an on-chain enforcement. A robust fix requires storing the Merkle tree depth or transaction count in `ExtendedHeader` at `submit_blocks` time and asserting `coinbase_merkle_proof.len() == stored_depth`.

---

### Proof of Concept

Given a real confirmed block with Merkle root `R`, tree depth `L = 3` (8 transactions), and internal node `I = hash(leaf0, leaf1)` at depth 1:

1. Attacker sets `coinbase_tx_id = I` (an internal node, not the real coinbase txid).
2. Attacker computes `coinbase_merkle_proof` of length 2 (not 3) such that `compute_root_from_merkle_proof(I, 0, coinbase_merkle_proof) == R`. This is valid because `I` sits at position 0 in the level-1 subtree.
3. Check (1): `merkle_proof.len() == coinbase_merkle_proof.len() == 2` — passes.
4. Check (2): coinbase proof validates to `R` — passes.
5. Attacker sets `tx_id = J` where `J` is another internal node at depth 1, and provides `merkle_proof` of length 2 proving `J` reaches `R`.
6. `verify_transaction_inclusion` computes `compute_root_from_merkle_proof(J, tx_index, merkle_proof) == R` — passes.
7. Function returns `true` for `tx_id = J`, which is not a real transaction. [1](#0-0) [2](#0-1)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
