### Title
Closed Isolated Subaccounts Permanently Consume `isolatedSubaccountsMask` Slots, Blocking Future Isolated Subaccount Creation — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`createIsolatedSubaccount` enforces a per-address cap of `MAX_ISOLATED_SUBACCOUNTS_PER_ADDRESS = 10` using a bitmask (`isolatedSubaccountsMask`). Bits are set on creation but are never cleared when an isolated subaccount is closed. Because the limit check operates on the mask rather than on the count of currently-active subaccounts, a user who creates and closes 10 isolated subaccounts exhausts all 10 mask slots permanently and can never create a new isolated subaccount again, even though zero are active.

---

### Finding Description

`createIsolatedSubaccount` in `OffchainExchange.sol` tracks slot occupancy with a per-address bitmask:

```solidity
// line 1023
uint256 mask = isolatedSubaccountsMask[senderAddress];
```

The hard cap is enforced by checking whether all bits are set:

```solidity
// lines 1045-1048
require(
    mask != (1 << MAX_ISOLATED_SUBACCOUNTS_PER_ADDRESS) - 1,
    "Too many isolated subaccounts"
);
```

When a new slot is allocated, the corresponding bit is permanently OR-ed in:

```solidity
// line 1063
isolatedSubaccountsMask[senderAddress] |= 1 << id;
```

There is no corresponding bit-clearing operation anywhere in the codebase. The `tryCloseIsolatedSubaccount` path (called from `EndpointTx.sol` line 639) zeroes out the `isolatedSubaccounts[sender][id]` mapping entry, but the mask bit for that slot is left set.

This is confirmed by the design of `isIsolatedSubaccountActive`, which deliberately checks the mapping entry rather than the mask:

```solidity
// lines 1123-1128
for (uint256 id = 0; id < MAX_ISOLATED_SUBACCOUNTS_PER_ADDRESS; id++) {
    if (subaccount == isolatedSubaccounts[parent][id]) {
        return true;
    }
}
```

If the mask were kept in sync with active state, this function would simply test the mask bit. The fact that it iterates the mapping confirms the mask and the active-state mapping can diverge.

The slot-finding loop also relies solely on the mask:

```solidity
// lines 1050-1053
uint8 id = 0;
while (mask & 1 != 0) {
    mask >>= 1;
    id += 1;
}
```

A slot whose mapping entry was zeroed by `tryCloseIsolatedSubaccount` still has its mask bit set, so this loop skips it. Combined with the all-bits-set limit check, once 10 subaccounts have been created and closed, the mask equals `(1 << 10) - 1` and the `require` at line 1046 reverts every subsequent creation attempt.

---

### Impact Explanation

A user who creates and closes 10 isolated subaccounts (the protocol-intended maximum of *concurrent* positions) permanently loses the ability to open any further isolated positions. The `MAX_ISOLATED_SUBACCOUNTS_PER_ADDRESS` constant is intended to cap simultaneous active subaccounts, not lifetime subaccounts. Because the mask is never decremented, the effective lifetime cap is also 10, which is a permanent, irreversible loss of core trading functionality for that address.

---

### Likelihood Explanation

Any active trader using isolated margin will naturally cycle through isolated subaccounts as positions are opened and closed. After 10 such cycles the address is permanently locked out of isolated trading. No special privileges or external conditions are required; the user's own normal trading activity triggers the condition.

---

### Recommendation

In `tryCloseIsolatedSubaccount`, after zeroing `isolatedSubaccounts[sender][id]`, also clear the corresponding mask bit:

```solidity
isolatedSubaccountsMask[senderAddress] &= ~(uint256(1) << id);
```

This keeps the mask in sync with the actual active-subaccount state and makes the limit check reflect concurrent active subaccounts rather than lifetime created subaccounts.

---

### Proof of Concept

1. User calls `CreateIsolatedSubaccount` 10 times for 10 different products. After each call, `isolatedSubaccountsMask[user]` gains one more set bit. After 10 calls, `mask == (1 << 10) - 1 == 0x3FF`.
2. User calls `CloseIsolatedSubaccount` 10 times. `tryCloseIsolatedSubaccount` zeroes each `isolatedSubaccounts[user][id]` entry, but `isolatedSubaccountsMask[user]` remains `0x3FF`.
3. User attempts an 11th `CreateIsolatedSubaccount`. The check at line 1046 evaluates `0x3FF != (1 << 10) - 1` → `false` → revert `"Too many isolated subaccounts"`.
