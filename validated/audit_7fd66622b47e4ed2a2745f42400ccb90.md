### Title
NLP Lock Period Bypass via Unvalidated `SpotTick` Time in Slow Mode — (`File: core/contracts/EndpointTx.sol`, `core/contracts/SpotEngineState.sol`)

---

### Summary

The NLP lock mechanism in `SpotEngineState.sol` uses `getOracleTime()` — a sequencer-supplied timestamp — to both set and evaluate lock expiry. Because `SpotTick` transactions are not restricted to owner-only in the slow mode submission path, and because there is no on-chain validation that the `time` field in a `SpotTick` is bounded by `block.timestamp`, an unprivileged user can submit a `SpotTick` with an arbitrarily advanced future timestamp via slow mode. After the 3-day slow mode delay, force-executing this transaction advances `getOracleTime()` far enough into the future that all existing NLP locked balances satisfy `unlockedAt <= getOracleTime()`, bypassing the intended 4-day lock period entirely.

---

### Finding Description

The NLP lock period is enforced in `handleNlpLockedBalance` by recording `unlockedAt = getOracleTime() + NLP_LOCK_PERIOD` (4 days), and in `tryUnlockNlpBalance` by checking `unlockedAt <= getOracleTime()`.

`getOracleTime()` in `EndpointGated.sol` delegates to `IEndpoint(endpoint).getTime()`, which returns `max(times.spotTime, times.perpTime)` — values written exclusively by `SpotTick` and `PerpTick` transactions processed through `processTransactionImpl`.

In `processTransactionImpl`, the `SpotTick` handler is:

```solidity
uint128 dt = t.spotTime == 0 ? 0 : txn.time - t.spotTime;
spotEngine.updateStates(dt);
t.spotTime = txn.time;
times = t;
```

The only guard is `require(dt < 7 * SECONDS_PER_DAY)` inside `updateStates`. There is **no check** that `txn.time` is close to `block.timestamp`.

In `submitSlowModeTransactionImpl`, the owner-only guard covers a specific list of transaction types. `SpotTick` is **not** in that list, so it falls into the `else` branch — any user can submit it by paying the slow mode fee:

```solidity
} else {
    chargeSlowModeFee(_getQuote(), sender);
    slowModeFees += SLOW_MODE_FEE;
}
```

The slow mode transaction is stored with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (3 days). After this delay, the censorship-resistance design allows force-execution. An attacker submits a `SpotTick` with `txn.time = current_oracle_time + (4 days + 1 second)`. Since `dt = 4 days + 1 second < 7 days`, `updateStates` does not revert. `times.spotTime` is set to the future value. `getOracleTime()` now returns a value that exceeds every existing `unlockedAt` for NLP balances locked within the past 4 days, so `tryUnlockNlpBalance` immediately moves all of them into `unlockedBalanceSum`.

---

### Impact Explanation

The 4-day NLP lock period is a core protocol invariant: liquidity providers cannot withdraw NLP tokens immediately after minting, protecting the protocol from liquidity flight during volatile conditions. By advancing `getOracleTime()` by 4+ days, an attacker causes `tryUnlockNlpBalance` to prematurely release all locked NLP balances into `unlockedBalanceSum`. These balances then count as full collateral in health checks and become available for `burnNlp` (NLP redemption). Any user whose NLP was locked can immediately redeem it, bypassing the lock entirely. This is a direct asset-availability impact: locked liquidity becomes withdrawable on demand.

---

### Likelihood Explanation

The attack requires only: (1) paying the slow mode fee (a fixed dollar amount), (2) waiting 3 days for the slow mode delay, and (3) calling the force-execute path. No privileged access, no leaked keys, no governance capture. The attacker does not need to own any NLP themselves — they advance the oracle time for all users simultaneously. The cost is the slow mode fee plus 3 days of waiting.

---

### Recommendation

Restrict `SpotTick` and `PerpTick` to the sequencer (owner) in the slow mode submission path, the same way admin-only transaction types are guarded:

```solidity
txType == IEndpoint.TransactionType.SpotTick ||
txType == IEndpoint.TransactionType.PerpTick ||
// ... existing owner-only types
) {
    require(sender == owner());
}
```

Additionally, add an on-chain bound in `processTransactionImpl` to ensure `txn.time` does not exceed `block.timestamp` by more than a small tolerance (e.g., 5 minutes), preventing any future path from accepting fabricated timestamps.

---

### Proof of Concept

1. Current state: `times.spotTime = T`, NLP user minted at time `T`, so `unlockedAt = T + 4 days`.
2. Attacker calls `submitSlowModeTransaction` with a `SpotTick` payload where `txn.time = T + 4 days + 1`.
3. `SpotTick` is not in the owner-only list → slow mode fee is charged → transaction queued with `executableAt = block.timestamp + 3 days`.
4. After 3 days, attacker force-executes the slow mode transaction.
5. `processTransactionImpl` handles `SpotTick`: `dt = (T + 4 days + 1) - T = 4 days + 1 < 7 days` → `updateStates` passes → `times.spotTime = T + 4 days + 1`.
6. `getOracleTime()` now returns `T + 4 days + 1`.
7. NLP user calls `tryUnlockNlpBalance`: `unlockedAt (T + 4 days) <= getOracleTime() (T + 4 days + 1)` → balance moved to `unlockedBalanceSum`.
8. NLP user calls `burnNlp` → health check passes using the now-unlocked balance → NLP redeemed 1 second after the 4-day lock was supposed to expire, but only 3 days after minting.
