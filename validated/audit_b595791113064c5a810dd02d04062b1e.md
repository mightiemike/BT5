### Title
Empty Merkle Proof Incorrectly Rejected for Single-Transaction Blocks — (`File: contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2` which delegates to it) contains a strict boundary guard that unconditionally rejects an empty `merkle_proof`. This is incorrect: for a block containing exactly one transaction, the Merkle root equals the transaction hash directly, and the valid proof is the empty list. The guard panics on this valid input, permanently blocking SPV verification for any single-transaction block.

### Finding Description

In `verify_transaction_inclusion`, after confirming the block is on the main chain and has enough confirmations, the function performs:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

It then computes the Merkle root via:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

`compute_root_from_merkle_proof` with an empty proof simply returns `transaction_hash` unchanged (the `for proof_hash in merkle_proof` loop body never executes): [3](#0-2) 

For a single-transaction block, `merkle_root == tx_id` by Bitcoin protocol definition. Submitting `merkle_proof = []` would correctly satisfy the root comparison — but the `require!` guard fires first and panics, making the call unreachable for this valid input.

`verify_transaction_inclusion_v2` is the recommended non-deprecated entry point and delegates unconditionally to `verify_transaction_inclusion`:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [4](#0-3) 

Both public methods are `#[pause]`-gated (callable by any unprivileged account when unpaused). [5](#0-4) [6](#0-5) 

### Impact Explanation

Any downstream NEAR contract (bridge, DeFi protocol, cross-chain oracle consumer) that calls `verify_transaction_inclusion_v2` to gate fund release or state transitions on a Bitcoin transaction that happens to be in a single-transaction block will receive a panic instead of `true`. The verification path is permanently blocked for that transaction — not merely delayed. The corrupted invariant is the proof-verification result: a cryptographically valid SPV proof is rejected as invalid.

### Likelihood Explanation

Single-transaction blocks (coinbase-only) occur on Bitcoin mainnet during periods of low mempool activity and are a normal part of the protocol. A relayer submitting such a block to the light client stores it correctly; the bug only surfaces when a consumer later tries to prove inclusion of the coinbase transaction. The trigger requires no privilege — any NEAR account can call `verify_transaction_inclusion_v2` with a valid empty proof.

### Recommendation

Remove or relax the empty-proof guard. An empty proof is valid precisely when the block contains one transaction. The existing root-equality check already handles this correctly without the guard:

```rust
// Remove this line:
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

If a non-empty proof is desired as a security property (to prevent trivial calls), the check should be conditioned on whether the block's Merkle root equals the `tx_id` directly, not on proof length alone.

### Proof of Concept

1. A Bitcoin block is mined containing only the coinbase transaction. Its `merkle_root` equals `coinbase_txid`.
2. The relayer submits this block header via `submit_blocks`; it is stored correctly in `headers_pool`.
3. A bridge contract calls `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_txid`
   - `tx_block_blockhash = <that block's hash>`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = coinbase_txid`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
4. Inside `verify_transaction_inclusion_v2`, the coinbase proof check passes: `compute_root_from_merkle_proof(coinbase_txid, 0, &[])` returns `coinbase_txid` == `merkle_root`. ✓
5. `verify_transaction_inclusion` is called. The confirmations check passes. ✓
6. `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` **panics**. ✗
7. The bridge call reverts; the valid Bitcoin transaction can never be verified through this contract.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
