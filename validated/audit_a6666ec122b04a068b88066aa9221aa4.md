### Title
Non-Canonical Group Encoding in serde_2026 Deserializer: `length_val=-1, count=1` Accepted as Equivalent to `length_val=1` — (`src/serde_2026/de.rs`)

---

### Summary

The serde_2026 deserializer accepts two structurally distinct atom-table group encodings that produce identical output. The `strict` flag does not close this gap. An attacker can craft a byte stream that differs from the canonical serializer output but deserializes to the same CLVM tree, violating the one-to-one mapping between wire bytes and tree identity.

---

### Finding Description

**Serializer invariant** (`src/serde_2026/ser.rs`, `write_atom_table`):

The serializer unconditionally uses a **positive** `length_val` for singleton groups and a **negative** `length_val` only for multi-atom groups: [1](#0-0) 

```
singleton  → write_varint(+length)          [no count field]
multi-atom → write_varint(-length) + count  [count ≥ 2]
```

**Deserializer gap** (`src/serde_2026/de.rs`, `deserialize_2026_body_from_stream`): [2](#0-1) 

The branch `length_val < 0` reads a `count` varint and accepts **any** `count ≥ 1`. There is no check that rejects `count=1` as non-canonical when `length_val` is negative. The guard at line 62 only rejects `count=0`.

**Concrete collision for a 1-byte atom `[0xAB]`:**

| Encoding | Bytes (after magic prefix + group_count=1) | Meaning |
|---|---|---|
| Canonical | `01 AB` | `length_val=+1`, atom data |
| Non-canonical | `7F 01 AB` | `length_val=-1`, `count=1`, atom data |

Both paths execute `atoms.push(allocator.new_atom(&buf)?)` with identical `buf = [0xAB]`.

**`strict` mode does not help.** The `strict` flag in `read_varint` only rejects overlong varint *encodings* (e.g., `0x80 0x00` for `0`): [3](#0-2) 

It has no awareness of the group-encoding semantics. `length_val=-1` encoded as `0x7F` is a minimal single-byte varint; `strict=true` accepts it without complaint.

**`serialized_length_serde_2026` diverges** (`src/serde_2026/de.rs`, lines 186–204): [4](#0-3) 

The length-counting function mirrors the same branch logic and therefore returns a different byte count for the two encodings of the same atom, confirming the streams are byte-distinct.

---

### Impact Explanation

Two distinct byte streams deserialize to the same CLVM tree (same `NodePtr`, same tree hash). If serde_2026 blobs are used as wire-format identifiers for consensus objects (spend bundles, puzzles, coin announcements), an attacker can present a non-canonical encoding that passes deserialization but has a different byte length and different raw bytes than the canonical form. Any system that:

- compares serialized bytes for deduplication, or
- uses `serialized_length_serde_2026` as a framing/cost signal,

will disagree between the canonical and non-canonical forms while both produce the same tree.

---

### Likelihood Explanation

The attacker needs only to craft a valid serde_2026 blob with a negative `length_val` and `count=1`. No special privileges, no compromised nodes, no social engineering. The format is exposed via the Python wheel bindings (`wheel/src/api.rs`) and the public Rust API. The non-canonical blob passes all current validation including `strict=true`.

---

### Recommendation

Add a canonicality check in the deserializer: when `length_val < 0`, require `count ≥ 2`. A `count=1` with a negative `length_val` is semantically identical to a positive `length_val` and must be rejected:

```rust
if length_val < 0 {
    // ...
    let count = checked_usize(read_varint(reader, strict)?)?;
    if count < 2 {  // ADD THIS GUARD
        return Err(EvalErr::SerializationError);
    }
    (atom_len, count)
}
```

Apply the same guard in `serialized_length_serde_2026` at the corresponding branch. [5](#0-4)

### Proof of Concept

```rust
// Canonical: group_count=1, length_val=+1, atom=[0xAB]
// varint(1)=0x01, varint(1)=0x01 (instruction_count), varint(2)=0x02 (push atom 0)
let canonical: Vec<u8> = vec![
    0xfd, 0xff, b'2', b'0', b'2', b'6',  // magic prefix
    0x01,        // group_count = 1
    0x01,        // length_val = +1 (singleton)
    0xAB,        // atom data
    0x01,        // instruction_count = 1
    0x02,        // push atom[0]
];

// Non-canonical: group_count=1, length_val=-1, count=1, atom=[0xAB]
let non_canonical: Vec<u8> = vec![
    0xfd, 0xff, b'2', b'0', b'2', b'6',  // magic prefix
    0x01,        // group_count = 1
    0x7f,        // length_val = -1 (multi-atom encoding, but count=1)
    0x01,        // count = 1  ← non-canonical: should be ≥ 2
    0xAB,        // atom data
    0x01,        // instruction_count = 1
    0x02,        // push atom[0]
];

let mut alloc = Allocator::new();
let node_a = deserialize_2026(&mut alloc, &canonical, usize::MAX, true).unwrap();
let node_b = deserialize_2026(&mut alloc, &non_canonical, usize::MAX, true).unwrap();
// node_a and node_b hold the same atom [0xAB] — same tree hash, different wire bytes
assert_ne!(canonical.len(), non_canonical.len()); // lengths differ
assert_eq!(
    serialized_length_serde_2026(&canonical, usize::MAX, true).unwrap(),
    canonical.len() as u64
);
assert_eq!(
    serialized_length_serde_2026(&non_canonical, usize::MAX, true).unwrap(),
    non_canonical.len() as u64
); // different lengths, same tree
```
