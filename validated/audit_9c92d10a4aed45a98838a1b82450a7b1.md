### Title
`ContractOwner` Has No `receive()` Function, Permanently Locking ETH Sent to `DirectDepositV1` — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner` has no `receive()` or `fallback()` function, making it incapable of receiving ETH. The `withdrawFromDirectDepositV1` function is the only recovery path for native ETH held in a `DirectDepositV1` contract, but it routes ETH through `ContractOwner` as an intermediate. Because `ContractOwner` cannot accept ETH, this path always reverts, permanently locking any ETH that accumulates in `DirectDepositV1`.

---

### Finding Description

`DirectDepositV1` is deployed with a fixed CREATE2 salt (`bytes32(uint256(1))`), making its address deterministic and pre-computable. It has a `receive()` function that wraps incoming ETH into the wrapped native token. However, ETH can bypass this function and accumulate directly in the contract via two realistic vectors:

1. **Pre-funding the deterministic address**: Because the CREATE2 address is fixed, an attacker or user can send ETH to the address before `createDirectDepositV1` is called. The constructor handles this with a non-reverting path — if the wrap call fails, it only emits `NativeTokenTransferFailed` and continues, leaving ETH stuck.

2. **`selfdestruct` force-send**: Any contract can `selfdestruct` and force ETH into `DirectDepositV1`, bypassing the `receive()` function entirely.

Once ETH is in `DirectDepositV1`, the only intended recovery path is `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`:

```solidity
// ContractOwner.sol lines 628–636
uint256 preBalance = address(this).balance;
DirectDepositV1(directDepositV1).withdrawNative();
uint256 postBalance = address(this).balance;
require(postBalance > preBalance, "empty");
(bool success, ) = msg.sender.call{value: postBalance - preBalance}("");
require(success, "xfer");
```

`DirectDepositV1.withdrawNative()` sends ETH to `msg.sender`, which is `ContractOwner`:

```solidity
// DirectDepositV1.sol lines 108–112
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
```

`ContractOwner` has no `receive()` or `fallback()` function. The ETH transfer to `ContractOwner` fails, `withdrawNative()` reverts with `"Failed to transfer native token to owner"`, and the entire `withdrawFromDirectDepositV1` call reverts. ETH in `DirectDepositV1` is permanently irrecoverable.

---

### Impact Explanation

**High.** Any ETH that accumulates in a `DirectDepositV1` instance is permanently locked. There is no alternative recovery path. The `withdrawNative()` function exists specifically to recover ETH, but it is rendered permanently non-functional by `ContractOwner`'s inability to receive ETH. The asset delta is the full ETH balance of the affected `DirectDepositV1` contract.

---

### Likelihood Explanation

**Low.** Normal ETH deposits via `receive()` are immediately wrapped and do not accumulate. ETH accumulation requires either a `selfdestruct` force-send or pre-funding the deterministic CREATE2 address before deployment — both are non-standard interactions. However, the CREATE2 address is publicly computable, making pre-funding a realistic griefing vector.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH during the `withdrawFromDirectDepositV1` flow:

```solidity
receive() external payable {}
```

Alternatively, refactor `DirectDepositV1.withdrawNative()` to accept a `recipient` address parameter so ETH can be sent directly to the final recipient (the multisig owner) without routing through `ContractOwner`.

---

### Proof of Concept

1. Pre-compute the deterministic `DirectDepositV1` address for a given `subaccount` (salt is always `bytes32(uint256(1))`).
2. Send ETH to that address before `createDirectDepositV1` is called, OR call `selfdestruct` on a helper contract targeting the `DirectDepositV1` address after deployment.
3. ETH is now in `DirectDepositV1` (constructor wrap may fail silently; `selfdestruct` bypasses `receive()`).
4. Call `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` (owner-only, but this is the only recovery path).
5. Internally, `DirectDepositV1.withdrawNative()` executes `msg.sender.call{value: balance}("")` where `msg.sender` is `ContractOwner`.
6. `ContractOwner` has no `receive()` or `fallback()` — the call returns `success = false`.
7. `withdrawNative()` reverts: `"Failed to transfer native token to owner"`.
8. ETH is permanently locked in `DirectDepositV1`.

        external
        onlyOwner

        require(directDepositV1 != address(0), "no dda");
        if (token == address(0)) {
            uint256 preBalance = address(this).balance;
            DirectDepositV1(directDepositV1).withdrawNative();
            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");
