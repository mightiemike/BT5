Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's `msg.sender` — the immediate caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router (required for any user to swap through it), every unprivileged user bypasses the allowlist entirely, rendering the sole access-control mechanism on curated pools ineffective.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- immediate caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value as `sender` to the extension hook:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks this `sender` against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The pool's `msg.sender` is the router. The extension therefore evaluates `allowedSwapper[pool][router_address]`. A pool admin who wants allowlisted users to access the pool via the standard router must allowlist the router address. Once the router is allowlisted, any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle()` and the extension check passes — the attacker's address is never consulted.

The test suite confirms this design: `FullMetricExtensionTest.test_allowedSwapSucceeds` allowlists `address(callers[0])` (the intermediate `TestCaller` contract), not `users[0]` (the EOA), and the swap succeeds because the pool passes the intermediate caller as `sender`.

## Impact Explanation

Any user can bypass a `SwapAllowlistExtension`-protected pool by routing through `MetricOmmSimpleRouter` once the router is allowlisted. The allowlist is the sole per-user access-control mechanism on curated pools (e.g., KYC-gated or institutional pools). Its bypass allows non-allowlisted users to trade on pools explicitly configured to exclude them, causing direct loss of curation policy and enabling unauthorized extraction of LP value from restricted pools. This constitutes broken core pool functionality (access control) causing potential loss of funds to LPs who deployed capital under the assumption that only vetted counterparties could trade.

## Likelihood Explanation

High. `MetricOmmSimpleRouter` is the standard, documented swap entry point for the protocol. The router's allowlist status is publicly observable on-chain via `allowedSwapper[pool][router]`. Any user who observes the router is allowlisted can immediately exploit the bypass with a normal router call. No privileged access, special tokens, flash loans, or malicious setup is required — only a standard `exactInputSingle` call.

## Recommendation

The pool should pass the originating user's address — not `msg.sender` — as `sender` to extension hooks, or the router should implement a trusted-forwarder pattern that appends the real user's address to calldata and the extension reads it from there. Alternatively, the extension could require that callers interact directly with the pool (no router intermediary), but this breaks usability. The cleanest fix is for the pool to expose an `originSender` parameter (e.g., via EIP-2771-style forwarding) that the router populates with `msg.sender` before calling the pool, and for the extension to check that value instead of the pool-level `sender`.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as the `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to allow allowlisted users to use `MetricOmmSimpleRouter`.
3. Non-allowlisted EOA `attacker` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: attacker, ...})`.
4. Router calls `pool.swap(attacker, ...)` — pool's `msg.sender` = `router`.
5. Pool calls `_beforeSwap(router, attacker, ...)` → extension receives `sender = router`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true`.
7. Swap executes. `attacker` was never individually allowlisted; the allowlist is fully bypassed. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
