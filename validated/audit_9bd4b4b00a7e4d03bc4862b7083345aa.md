Audit Report

## Title
`SwapAllowlistExtension::beforeSwap` checks the router address instead of the actual end user, blocking allowlisted users from router-based swaps and enabling allowlist bypass ‚Äî (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension::beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool` binds to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` inside the pool, so the extension checks whether the **router** is allowlisted rather than the actual end user. This produces two symmetric failures: allowlisted users are blocked from using the router (default state, no privileged action required), and if the pool admin allowlists the router to enable router-based swaps, every user‚Äîincluding non-allowlisted ones‚Äîbypasses the per-user gate.

## Finding Description

**Pool ‚Üí Extension argument binding**

In `MetricOmmPool::swap`, the pool passes `msg.sender` as the `sender` argument to every before-swap hook: [1](#0-0) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly: [2](#0-1) 

So `msg.sender` inside the pool is the **router address**, not the end user.

**Extension check**

`SwapAllowlistExtension::beforeSwap` receives that value as `sender` and checks it against the per-pool allowlist: [3](#0-2) 

Here `msg.sender` is the pool (the pool calls the extension), and `sender` is whoever called `pool.swap()`. When the router is the caller, the check becomes `allowedSwapper[pool][router]`, completely ignoring the actual end user.

**Two failure modes**

| Router allowlisted? | Allowlisted user via router | Non-allowlisted user via router |
|---|---|---|
| No (default) | **Reverts** ‚Äî broken core swap flow | Reverts (correct) |
| Yes (admin enables router) | Passes (correct) | **Passes** ‚Äî allowlist bypassed |

## Impact Explanation

**Broken core swap flow (no privileged action required):** Allowlisted users on curated pools cannot use `MetricOmmSimpleRouter`‚Äîthe primary user-facing swap entry point‚Äîeven though they are individually permitted. Their only option is to call `pool.swap()` directly, bypassing all periphery slippage and routing helpers. This constitutes an unusable swap flow for the intended user set, matching the "Broken core pool functionality causing unusable swap flows" criterion.

**Allowlist bypass (semi-trusted trigger):** A pool admin who allowlists the router address to enable router-based swaps unknowingly grants every user the ability to swap on the curated pool. Non-allowlisted users can trade on pools designed to restrict access (e.g., for regulatory or risk-management reasons), constituting a policy bypass with direct fund-flow consequences.

## Likelihood Explanation

The DoS failure mode is the default state for any pool that combines `SwapAllowlistExtension` with the router. No special configuration is needed to trigger it‚Äîit fires the moment an allowlisted user tries to use the router. The bypass failure mode requires the pool admin to allowlist the router, which is a natural and non-malicious configuration choice for a pool that wants to support router-based swaps.

## Recommendation

The extension must check the actual end user, not the immediate caller of `pool.swap()`. Two approaches:

1. **Pass the original user through `extensionData`**: The router encodes the original `msg.sender` into `extensionData`; the extension decodes and checks that address. This requires a convention between the router and the extension.

2. **Add a dedicated `originalSender` field to the hook interface**: The pool could forward both `msg.sender` (the immediate caller) and an optional `originalSender` (set by the router via a transient-storage context), letting extensions choose which identity to gate.

Until fixed, pools that require per-user swap gating should not rely on `SwapAllowlistExtension` for router-mediated flows.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension as a before-swap hook
  - allowedSwapper[pool][alice] = true   (alice is individually allowlisted)
  - allowedSwapper[pool][router] = false (router is NOT allowlisted)

Scenario A ‚Äî DoS for allowlisted user:
  alice calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
    ‚Üí router calls pool.swap(...) with msg.sender = router
    ‚Üí pool calls extension.beforeSwap(router, ...)
    ‚Üí allowedSwapper[pool][router] == false ‚Üí revert NotAllowedToSwap
  alice calls pool.swap(...) directly
    ‚Üí pool calls extension.beforeSwap(alice, ...)
    ‚Üí allowedSwapper[pool][alice] == true ‚Üí succeeds

Scenario B ‚Äî Bypass when router is allowlisted:
  Admin sets allowedSwapper[pool][router] = true
  bob (not individually allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
    ‚Üí router calls pool.swap(...) with msg.sender = router
    ‚Üí pool calls extension.beforeSwap(router, ...)
    ‚Üí allowedSwapper[pool][router] == true ‚Üí succeeds (bypass)
```
