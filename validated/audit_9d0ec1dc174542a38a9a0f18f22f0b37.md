Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual User, Allowing Any User to Bypass Per-User Swap Restrictions - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`MetricOmmPool.swap()` passes `msg.sender` verbatim as the `sender` argument to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. When `MetricOmmSimpleRouter` mediates a swap, the pool observes `msg.sender = router`, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. Any pool admin who allowlists the router to support standard UX simultaneously grants every EOA the ability to bypass per-user curation.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that `sender` verbatim into the call to each configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` enforces the allowlist against that `sender` value, where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without passing the originating user's address, making the router the `msg.sender` the pool observes: [4](#0-3) 

The same pattern holds for `exactInput` (all hops), `exactOutputSingle`, and the outer call of `exactOutput`: [5](#0-4) 

For `exactOutput` multi-hop, intermediate hops are triggered from inside `_exactOutputIterateCallback`, where `msg.sender` is the previous pool — so those pools see a pool address as `sender`, not the user or the router: [6](#0-5) 

The pool's `swap()` interface accepts no explicit `sender` parameter; it always derives sender from `msg.sender`. There is no mechanism in the router to propagate the originating user's address to the extension layer.

## Impact Explanation

A pool admin who deploys `SwapAllowlistExtension` to restrict swaps to a curated set (KYC-verified addresses, specific market makers, regulated participants) and also allowlists the router for standard UX support renders the per-user restriction completely void. Any EOA can call `router.exactInputSingle()` and the extension passes because it sees `sender = router`. Unauthorized users can trade at oracle prices in a pool intended to be restricted, violating the pool's access policy and potentially extracting value from LPs. This is a direct bypass of a live access-control mechanism with fund-impacting consequences, meeting the admin-boundary break criterion.

## Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` for per-user curation and also wants to support `MetricOmmSimpleRouter` will face this issue. Allowlisting the router is the natural and expected configuration step for router support. The bypass requires no special privileges, no malicious setup, and no non-standard tokens — only a standard router call from any EOA. The preconditions are the normal operating configuration for a restricted pool that supports the standard periphery.

## Recommendation

The `sender` argument passed to `beforeSwap` should represent the economic actor, not the intermediary. Two complementary fixes:

1. **In the router**: pass the original `msg.sender` (the user) as an explicit `sender` parameter to `pool.swap()` rather than relying on the pool to use `msg.sender`. This requires adding a `sender` parameter to the pool's `swap` interface, or having the router store the user in transient storage and expose it via a callback the extension can query.

2. **In `SwapAllowlistExtension`**: document clearly that `sender` is the direct caller of `pool.swap()`, not the end user, and that allowlisting the router opens the pool to all router users. Until the interface is changed, pool admins must not simultaneously allowlist the router and rely on per-user curation.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists user1 directly: allowedSwapper[pool][user1] = true
  - Pool admin allowlists router for router support: allowedSwapper[pool][router] = true

Attack (user2, not individually allowlisted):
  1. user2 calls router.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, ...) — pool sees msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → TRUE
  5. Swap executes for user2 despite user2 not being individually allowlisted

Direct call (correctly blocked):
  1. user2 calls pool.swap(...) directly
  2. Pool calls _beforeSwap(sender=user2, ...)
  3. SwapAllowlistExtension checks allowedSwapper[pool][user2] → FALSE → revert NotAllowedToSwap

Conclusion: the router allowlist entry acts as a universal bypass for all users,
defeating the per-user curation the pool admin intended to enforce.
```
