### Title
Unvalidated `tx_id` in `verify_transaction_inclusion` Enables Merkle Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

The publicly callable `verify_transaction_inclusion` function accepts a caller-supplied `tx_id` and passes it directly to the Merkle root computation without any validation that it represents a real leaf-level transaction hash rather than an internal Merkle tree node. This is the structural analog to M-01: unvalidated external data accepted at face value and used for a critical security decision. An unprivileged NEAR caller can supply a crafted internal-node hash as `tx_id`, pair it with a valid-looking proof, and cause the function to return `true` for a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` in `contract/src/lib.rs` accepts `ProofArgs` from any caller and immediately feeds the caller-supplied `tx_id` into `merkle_tools::compute_root_from_merkle_proof`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

No check is performed to confirm that `tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node. The function's own documentation acknowledges this explicitly:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."

This is the well-documented 64-byte transaction Merkle proof forgery attack (https://www.bitmex.com/blog/64-Byte-Transactions). Because Bitcoin's Merkle tree is built by hashing 64-byte concatenations of child hashes, an internal node hash is indistinguishable from a transaction hash at the API boundary. An attacker who knows an internal node value from any real block already in the contract's `headers_pool` can construct a shorter proof path that recomputes the same `merkle_root`, making the function return `true`.

The `#[deprecated]` attribute is a Rust compiler hint only — it does not restrict runtime invocation. The function remains `pub` and is gated only by `#[pause]`, meaning it is live and callable by any NEAR account unless an operator has explicitly paused it.

`verify_transaction_inclusion_v2` was introduced to mitigate this by requiring a coinbase Merkle proof of the same depth, which constrains the attacker's freedom to choose an arbitrary internal node. However, `verify_transaction_inclusion_v2` itself calls the vulnerable `v1` function internally after the coinbase check, and the `v1` function remains independently reachable.

---

### Impact Explanation

Any downstream NEAR contract (e.g., a cross-chain bridge or asset-release contract) that calls `verify_transaction_inclusion` and acts on a `true` result — releasing funds, minting tokens, or updating state — can be deceived into processing a fabricated transaction. The attacker does not need to mine a Bitcoin block or control any privileged role; they only need to know an internal Merkle node hash from a block already accepted by the light client, which is derivable from public Bitcoin chain data.

---

### Likelihood Explanation

The 64-byte forgery technique is publicly documented and the required inputs (internal Merkle node hashes) are derivable from any Bitcoin block explorer. The entry point is permissionless. The only practical barrier is that the attacker must identify a downstream contract that trusts `verify_transaction_inclusion` and controls a valuable action. Given that the light client's stated purpose is to serve as a trust anchor for cross-chain bridges, such consumers are the expected production use case.

---

### Recommendation

1. **Remove or hard-panic `verify_transaction_inclusion`**: Replace its body with `env::panic_str("use verify_transaction_inclusion_v2")` so it cannot be called at runtime regardless of the `#[deprecated]` hint.
2. **Force all callers to `verify_transaction_inclusion_v2`**: The coinbase proof requirement in v2 is the correct mitigation; the v1 path should not remain reachable.
3. **Do not rely on `#[deprecated]` for security boundaries**: Rust's `deprecated` attribute is a compile-time warning to the contract's own crate authors, not an access-control mechanism for external NEAR callers.

---

### Proof of Concept

1. Identify any block already accepted into the contract's `headers_pool` (e.g., via `get_block_hash_by_height`).
2. From public Bitcoin chain data, retrieve the full transaction list for that block and compute its Merkle tree. Select any internal node hash `N` at depth `d` from the root.
3. Construct a `merkle_proof` of length `d` whose path, when applied to `N` at the appropriate `tx_index`, recomputes the block's `merkle_root`.
4. Call `verify_transaction_inclusion` with:
   - `tx_id` = `N` (internal node hash, not a real transaction)
   - `tx_block_blockhash` = the real block hash
   - `tx_index` = the crafted position
   - `merkle_proof` = the crafted proof path
   - `confirmations` = 1
5. The function returns `true`, asserting inclusion of a non-existent transaction.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
