### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Allowing Bypass of Coinbase Merkle Proof Validation — (`File: contract/src/lib.rs`)

---

### Summary

The contract exposes two public entry points for SPV proof verification: the deprecated `verify_transaction_inclusion` (v1) and `verify_transaction_inclusion_v2`. The v2 was introduced specifically to close the 64-byte transaction Merkle forgery vulnerability by requiring a coinbase proof. However, v1 remains fully callable by any unprivileged NEAR account and skips the coinbase proof check entirely. An attacker can bypass the security-critical coinbase validation by simply calling v1 instead of v2, obtaining a `true` proof result for a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces that the coinbase transaction's Merkle proof matches the block's `merkle_root` before proceeding: [1](#0-0) 

It then delegates to v1 via `self.verify_transaction_inclusion(args.into())`.

The deprecated v1 function performs **no coinbase proof check**: [2](#0-1) 

The v1 function is still decorated with `#[pause]` (not `#[deprecated]` at the NEAR ABI level) and is fully reachable by any caller. The `#[deprecated]` Rust attribute only produces a compiler warning for Rust callers; it does not prevent on-chain invocation. Any NEAR account can call `verify_transaction_inclusion` directly via a cross-contract call or direct transaction.

The 64-byte forgery attack works as follows: Bitcoin's Merkle tree hashes pairs of 32-byte values. An internal node hash is itself 32 bytes. If an attacker constructs a 64-byte value whose double-SHA256 equals an internal node hash, they can supply that value as `tx_id` with a valid Merkle path to the root, and `compute_root_from_merkle_proof` will return the correct `merkle_root` — causing v1 to return `true` for a non-existent transaction. [3](#0-2) 

The v1 function's own documentation acknowledges this: [4](#0-3) 

---

### Impact Explanation

Any downstream NEAR contract or bridge application that calls `verify_transaction_inclusion` (v1) to gate a payment, withdrawal, or state transition will accept a forged proof as valid. The corrupted value is the **proof result** (`bool` return), which is the sole output consumed by callers. A false `true` result for a fabricated transaction enables theft or unauthorized state changes in any contract that trusts this verification.

---

### Likelihood Explanation

The entry path requires no privileges: any NEAR account can submit a cross-contract call to `verify_transaction_inclusion`. The attacker only needs a valid block header already accepted by the contract (trivially obtained from the mainchain) and the ability to craft a 64-byte preimage — a well-documented, publicly known technique. The attack is deterministic and requires no brute force.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or make it panic unconditionally with a migration message. Do not rely on the Rust `#[deprecated]` attribute to prevent on-chain invocation. All callers must be migrated to `verify_transaction_inclusion_v2`.

If backward compatibility is required, at minimum add a `require!(false, "use verify_transaction_inclusion_v2")` guard inside the v1 function body so that on-chain calls are rejected.

---

### Proof of Concept

1. Identify any block `B` accepted by the contract with known `merkle_root`.
2. Construct a 64-byte value `fake_tx` such that `double_sha256(fake_tx)` equals an internal Merkle node hash reachable from `merkle_root` via a known sibling path `proof`.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = double_sha256(fake_tx)` (the internal node hash)
   - `tx_block_blockhash = B`
   - `tx_index` and `merkle_proof` matching the internal node's position
   - `confirmations = 1`
4. `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof)` reconstructs `merkle_root` correctly, so the function returns `true`.
5. No coinbase proof is checked. The forged transaction is accepted as included. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
