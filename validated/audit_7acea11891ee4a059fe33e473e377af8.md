### Title
Protobuf `ValidResourceBounds` round-trip silently downgrades `AllResources{l2_gas=0,l1_data_gas=0}` to `L1Gas`, producing a divergent transaction hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserialization of `ValidResourceBounds` uses a zero-value heuristic to decide which variant to reconstruct. When a V3 transaction carries `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the round-trip silently produces `L1Gas(X)`. Because `get_tip_resource_bounds_hash` hashes a different number of elements for each variant (3 vs 4), the transaction hash computed after deserialization differs from the hash computed at admission time. Any component that recomputes the hash from the deserialized `InvokeTransactionV3` — including block-sync hash validation — will observe a mismatch.

### Finding Description

**Hash domain divergence — `get_tip_resource_bounds_hash` (`transaction_hash.rs` lines 188–211):**

```rust
pub fn get_tip_resource_bounds_hash(resource_bounds: &ValidResourceBounds, tip: &Tip)
    -> Result<Felt, StarknetApiError>
{
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,   // always present
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],             // ← 3 elements total
        ValidResourceBounds::AllResources(all) =>
            vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?],  // ← 4 elements total
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
``` [3](#0-2) 

- `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` → Poseidon over `[tip, L1Gas(X), L2Gas(0), L1DataGas(0)]` (4 elements)
- `L1Gas(X)` → Poseidon over `[tip, L1Gas(X), L2Gas(0)]` (3 elements)

These produce **different field elements**.

**Admission path (gateway):** `InternalRpcInvokeTransactionV3` always stores `resource_bounds: AllResourceBounds` and calls `calculate_transaction_hash` via `ValidResourceBounds::AllResources(self.resource_bounds)`, so the stored hash uses the 4-element preimage. [4](#0-3) 

**Block-sync path:** `InvokeTransactionV3` carries `resource_bounds: ValidResourceBounds`. After protobuf round-trip, `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` becomes `L1Gas(X)`. Any call to `validate_transaction_hash` on the deserialized object computes the 3-element hash and mismatches the stored 4-element hash. [5](#0-4) 

### Impact Explanation

A transaction admitted with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` carries a hash computed over 4 elements. After protobuf serialization/deserialization (block-sync p2p), the reconstructed `ValidResourceBounds` is `L1Gas(X)`, and any hash recomputation produces a different value. This breaks the canonicalization invariant: the same logical transaction produces two distinct hashes depending on which code path reconstructs it. Concretely, block-sync hash validation (`validate_transaction_hash`) would reject a legitimately sequenced block, or a receipt/event would carry the wrong transaction hash.

This matches: **High — Transaction conversion or signature/hash logic binds the wrong hash or executable payload.**

### Likelihood Explanation

Any unprivileged user can submit a V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` (the gateway's stateless validator imposes no lower bound on these fields). The gateway accepts it, computes the 4-element hash, and includes it in a block. The divergence is triggered automatically on every p2p block-sync peer that deserializes the transaction.

### Recommendation

1. **Remove the zero-value downgrade heuristic** in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Instead, always reconstruct `AllResources` when all three fields are present in the protobuf message, regardless of their values. Reserve `L1Gas` only for messages that explicitly omit `l1_data_gas` (the pre-0.13.3 wire format).
2. **Add a canonicalization test** asserting that `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` survives a protobuf round-trip with the same `ValidResourceBounds` variant and produces the same transaction hash.
3. Once 0.13.2 support is dropped, enforce the `TODO(Shahak)` assertion so that a missing `l1_data_gas` field is a hard error rather than silently defaulting to zero.

### Proof of Concept

use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds,
};
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
use starknet_api::transaction::fields::Tip;
use starknet_api::block::GasPrice;
use starknet_api::execution_resources::GasAmount;
use apollo_protobuf::protobuf;

// 1. Construct AllResources with zero l2_gas and l1_data_gas.
let all = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas: ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(1) },
    l2_gas: ResourceBounds::default(),
    l1_data_gas: ResourceBounds::default(),
});

// 2. Compute hash before round-trip (4-element preimage).
let hash_before = get_tip_resource_bounds_hash(&all, &Tip(0)).unwrap();

// 3. Protobuf round-trip via ValidResourceBounds converter.
let proto: protobuf::ResourceBounds = all.into();
let after = ValidResourceBounds::try_from(proto).unwrap();
// after == ValidResourceBounds::L1Gas(ResourceBounds { max_amount: 100, ... })

// 4. Compute hash after round-trip (3-element preimage).
let hash_after = get_tip_resource_bounds_hash(&after, &Tip(0)).unwrap();

// 5. Hashes differ — canonicalization invariant violated.
assert_ne!(hash_before, hash_after);

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
