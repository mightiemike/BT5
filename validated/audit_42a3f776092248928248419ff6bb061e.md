Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User Sender, Allowing Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is `msg.sender` of `MetricOmmPool.swap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension evaluates the router's allowlist status rather than the actual trader's. This makes the per-user swap allowlist unenforceable for any router-mediated swap, allowing non-allowlisted users to bypass access control by routing through the public router.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender`:** [1](#0-0) 

**`ExtensionCalling._beforeSwap` forwards `sender` verbatim to every configured extension:** [2](#0-1) 

**`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`** — where `msg.sender` is the pool and `sender` is the router when routing through `MetricOmmSimpleRouter`: [3](#0-2) 

**`MetricOmmSimpleRouter` calls `pool.swap()` directly**, making itself `msg.sender` to the pool: [4](#0-3) [5](#0-4) 

**Identity mismatch table:**

| Call path | `sender` seen by extension | Allowlist check |
|---|---|---|
| User → Pool directly | User address | Correct |
| User → Router → Pool | Router address | Wrong — checks router, not user |

**Two concrete failure modes:**
1. **Bypass (higher impact):** If the pool admin adds the router to the allowlist (necessary for allowlisted users to use the router), every non-allowlisted user can also swap by routing through `MetricOmmSimpleRouter`. The per-user allowlist is completely defeated.
2. **Lockout (lower impact):** If the router is not in the allowlist, every allowlisted user who routes through the router is blocked, making the router unusable for allowlisted pools.

No existing guard in `SwapAllowlistExtension`, `ExtensionCalling`, or `MetricOmmPool` recovers the original end-user identity.

## Impact Explanation

`SwapAllowlistExtension` is the primary mechanism for pool admins to restrict swap access to curated counterparties (e.g., KYC-verified addresses, whitelisted market makers). When the allowlist check evaluates the router's identity instead of the end user's, the access control invariant is broken. A non-allowlisted address can execute swaps in a restricted pool by routing through the public `MetricOmmSimpleRouter`. If the pool is restricted to protect LPs from adversarial counterparties (e.g., informed traders, sanctioned addresses), the bypass allows those counterparties to trade against LP capital, causing LP principal loss. This matches the allowed impact gate: **Admin-boundary break — factory/oracle role checks are bypassed by an unprivileged path** and **Broken core pool functionality causing loss of funds**.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard, publicly deployed user-facing entry point for swaps. Any user interacting through the normal UI or SDK routes through it. No special privileges, flash loans, or unusual token behavior are required — any address can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput`. The bypass is deterministic and repeatable on every block. Pool admins who deploy `SwapAllowlistExtension` have no on-chain mechanism to prevent router-mediated bypass without also blocking all router users.

## Recommendation

`SwapAllowlistExtension.beforeSwap` must check the economically relevant actor — the end user — not the intermediary router. The preferred fix is for the router to encode the actual end-user address in `extensionData`, and for the extension to decode and check it when `sender` is a known router. Alternatively, the extension can require that `sender == recipient` for allowlisted pools (enforcing direct pool interaction). The cleanest production fix is the `extensionData` approach: the router passes `msg.sender` (the real user) in `extensionData`, and the extension decodes it when `sender` is a recognized router address, falling back to `sender` otherwise.

## Proof of Concept

```solidity
function testSwapAllowlistBypassViaRouter() public {
    address alice = makeAddr("alice");
    address bob   = makeAddr("bob");

    // Pool admin allowlists alice and the router (so alice can use the router)
    swapAllowlistExtension.setAllowedToSwap(address(pool), alice, true);
    swapAllowlistExtension.setAllowedToSwap(address(pool), address(router), true);

    // Alice can swap directly — expected
    vm.prank(alice);
    pool.swap(alice, false, 1000, type(uint128).max, "", "");

    // Bob direct swap reverts — expected
    vm.prank(bob);
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    pool.swap(bob, false, 1000, type(uint128).max, "", "");

    // Bob routes through the public router
    // Pool sees msg.sender = router (allowlisted) → extension passes
    // Bob receives tokens despite not being allowlisted
    vm.prank(bob);
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            zeroForOne: false,
            amountIn: 1000,
            amountOutMinimum: 0,
            recipient: bob,
            priceLimitX64: 0,
            deadline: block.timestamp,
            tokenIn: token0,
            extensionData: ""
        })
    );
    // bob received tokens — allowlist bypassed
}
```

The `beforeSwap` hook receives `sender = address(router)`, which is allowlisted, so `allowedSwapper[pool][router] = true` passes. Bob's swap settles successfully despite being explicitly excluded from the allowlist. [6](#0-5) [7](#0-6)
