Audit Report

## Title
SwapAllowlistExtension Allowlist Bypass via Router Address Substitution - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps on the direct caller of `pool.swap()`, which is always `address(MetricOmmSimpleRouter)` when users route through the router. If a pool admin allowlists the router to support router-mediated swaps for any authorized user, every user — including unauthorized ones — can bypass the per-user restriction by calling the router. There is no configuration that permits specific users through the router while blocking others.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

**Step 2 — Extension checks `allowedSwapper[pool][sender]` where `sender` is that direct caller.**

`SwapAllowlistExtension.beforeSwap()` uses `msg.sender` (the pool) as the pool key and `sender` (the direct caller of `pool.swap()`) as the swapper key: [2](#0-1) 

**Step 3 — Router is always the direct caller of `pool.swap()`.**

For `exactInputSingle` and `exactOutputSingle`, the router calls the pool directly with no user identity forwarded: [3](#0-2) 

For multi-hop `exactInput`, every hop is called by the router: [4](#0-3) 

In all cases, `sender` seen by the extension is `address(MetricOmmSimpleRouter)`, not the end user.

**Step 4 — The catch-22 with no valid configuration.**

| Admin action | Effect |
|---|---|
| Allowlist only `alice` | Alice cannot swap through the router (router not allowlisted); must call pool directly |
| Allowlist the router | Every user including unauthorized ones can swap through the router |

There is no configuration that allows specific users to swap through the router while blocking others. The allowlist collapses to an all-or-nothing gate on the router contract itself.

**Step 5 — Exploit path.**

1. Admin deploys pool with `SwapAllowlistExtension`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — intending only alice to trade.
3. Admin calls `setAllowedToSwap(pool, router, true)` — intending to let alice use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` with `msg.sender = router`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true` → Bob's swap succeeds.

The `setAllowedToSwap` setter and `allowedSwapper` mapping confirm the per-user design intent: [5](#0-4) 

## Impact Explanation
The `SwapAllowlistExtension` is the sole mechanism for restricting swap access to specific users in a pool. Its bypass allows any unprivileged user to trade in a pool the admin intended to restrict, breaking the core access-control invariant of the extension. Pools using this extension for compliance, private trading, or whitelist-gated liquidity are fully exposed once the router is allowlisted. This constitutes broken core pool functionality and an admin-boundary break where an unprivileged path circumvents the admin's intended restriction.

## Likelihood Explanation
The bypass requires the admin to have allowlisted the router — a natural and expected action for any admin who wants authorized users to benefit from router features (slippage protection, multi-hop, deadline checks). Once that configuration exists, any unprivileged user can exploit it with a single router call. No special privileges, flash loans, or complex setup are required. The exploit is repeatable on every swap.

## Recommendation
The extension must verify the end-user identity, not the direct caller. Two approaches:

1. **Pass the original user through `extensionData`**: Have the router encode `msg.sender` into `extensionData` and have the extension decode and verify it. This requires a trusted router registry or signature scheme to prevent spoofing.
2. **Check both router and originator**: Extend the extension to maintain a `trustedRouter` mapping and, when `sender` is a trusted router, require the extension to receive and verify the originating user address from `extensionData` (signed or registry-verified).
3. **Disallow router allowlisting**: Document that allowlisting a router grants access to all router users, and provide a separate `RouterAllowlistExtension` that enforces per-user checks via a callback or signed message from the router.

router.exactInputSingle(ExactInputSingleParams({

    recipient: bob,
