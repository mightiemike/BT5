### Title
Missing Zero-Address Validation in `BaseWithdrawPool._initialize()` Permanently Breaks Withdrawal Functionality — (File: `core/contracts/BaseWithdrawPool.sol`)

### Summary
`BaseWithdrawPool._initialize()` assigns `clearinghouse` and `verifier` from input parameters without zero-address checks. No setter functions exist for either variable. If either is initialized to `address(0)`, the `submitWithdrawal()` and `submitFastWithdrawal()` functions are permanently broken, locking user collateral in the pool with no recovery path short of redeployment.

### Finding Description
In `BaseWithdrawPool._initialize()`, both `clearinghouse` and `verifier` are assigned directly from parameters with no validation:

```solidity
function _initialize(address _clearinghouse, address _verifier)
    internal
    initializer
{
    __Ownable_init();
    clearinghouse = _clearinghouse;  // no zero check
    verifier = _verifier;            // no zero check
}
``` [1](#0-0) 

Neither `clearinghouse` nor `verifier` has a setter function anywhere in `BaseWithdrawPool.sol` or `WithdrawPool.sol`. The only owner-accessible function is `removeLiquidity()`, which does not update these addresses. [2](#0-1) 

**Case 1 — `clearinghouse = address(0)`:**
`submitWithdrawal()` enforces `require(msg.sender == clearinghouse)`. If `clearinghouse` is `address(0)`, this check becomes `require(msg.sender == address(0))`, which is permanently impossible. Every sequencer-submitted withdrawal reverts. [3](#0-2) 

**Case 2 — `verifier = address(0)`:**
`submitFastWithdrawal()` calls `Verifier(verifier).requireValidTxSignatures(...)`. A call to `address(0)` with no code reverts unconditionally, permanently breaking all fast withdrawals. [4](#0-3) 

The upstream flow in `Clearinghouse.handleWithdrawTransfer()` transfers tokens to `withdrawPool` and then calls `submitWithdrawal()`. If `submitWithdrawal()` is broken, the tokens are already transferred to the pool but can never be forwarded to users. [5](#0-4) 

### Impact Explanation
User collateral is permanently locked inside `WithdrawPool`. The `Clearinghouse` sends tokens to the pool before calling `submitWithdrawal()`; if that call reverts, the tokens are stranded with no on-chain recovery path. The `removeLiquidity()` owner function exists but requires the owner to manually identify and redistribute every affected user's funds — a process with no protocol-level guarantee of correctness. The corrupted state is the `clearinghouse` slot holding `address(0)`, and the asset delta is the full balance of every token held in the pool becoming unwithdrawable.

### Likelihood Explanation
Low probability — requires a deployment mistake by the deployer. However, the pattern is identical to the referenced external report: a single misconfigured parameter at initialization time produces a permanently broken contract with no setter-based recovery. The `WithdrawPool` is deployed and initialized as part of the core protocol setup, and the absence of any guard means the error would only be discovered after deployment when the first withdrawal attempt fails.

### Recommendation
Add explicit zero-address guards in `BaseWithdrawPool._initialize()`:

```solidity
function _initialize(address _clearinghouse, address _verifier)
    internal
    initializer
{
    require(_clearinghouse != address(0), "clearinghouse is zero address");
    require(_verifier != address(0), "verifier is zero address");
    __Ownable_init();
    clearinghouse = _clearinghouse;
    verifier = _verifier;
}
```

Additionally, consider adding owner-accessible setter functions for both `clearinghouse` and `verifier` to allow recovery without redeployment, mirroring the pattern already used in `Clearinghouse.setWithdrawPool()`. [6](#0-5) 

### Proof of Concept
1. Deploy `WithdrawPool` and call `initialize(address(0), validVerifier)`.
2. The `initializer` modifier allows this — no revert occurs during initialization.
3. A user deposits collateral through `Endpoint.depositCollateral()`, which routes through `Clearinghouse.withdrawCollateral()` → `handleWithdrawTransfer()` → `token.safeTransfer(withdrawPool, amount)` followed by `BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx)`.
4. `submitWithdrawal()` executes `require(msg.sender == clearinghouse)` → `require(msg.sender == address(0))` → reverts.
5. The token transfer to `withdrawPool` already succeeded; the revert in `submitWithdrawal()` propagates up and reverts the entire `withdrawCollateral()` call, but the tokens are already in the pool from the deposit path. Any subsequent withdrawal attempt by any user hits the same permanent revert.
