### Title
Burn Fee Exceeds NLP Value for Sub-$1 Positions, Causing Total Loss of User Funds — (`core/contracts/Clearinghouse.sol`)

---

### Summary

In `Clearinghouse.burnNlp`, the minimum burn fee is enforced as `MathHelper.max(ONE, quoteAmount / 1000)`. When the NLP being burned is worth less than `ONE` (1e18, representing $1 in the fixed-point system), the fee floor of `ONE` exceeds `quoteAmount`, driving the credited amount to zero. The user's NLP balance is fully decremented while they receive no quote in return.

---

### Finding Description

The exact arithmetic in `burnNlp`:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);           // line 502
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);     // line 503
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);       // line 504
``` [1](#0-0) 

When `quoteAmount < ONE`:
- `quoteAmount / 1000 < ONE`, so `MathHelper.max(ONE, quoteAmount / 1000) = ONE`
- `quoteAmount - ONE < 0`, so `MathHelper.max(0, quoteAmount - ONE) = 0`

`MathHelper.max` is a simple comparator with no special behavior: [2](#0-1) 

After this, the NLP balance is unconditionally decremented:

```solidity
spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);  // line 511
spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);    // line 512
``` [3](#0-2) 

But the quote credit is gated on `quoteAmount > 0`:

```solidity
if (quoteAmount > 0) {
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
    _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
}
``` [4](#0-3) 

When `quoteAmount = 0`, this block is skipped entirely. The NLP is returned to `N_ACCOUNT` and no quote leaves the pool. The user's NLP value is destroyed.

There is no minimum NLP amount guard in `burnNlp` — only an unlocked balance check: [5](#0-4) 

---

### Impact Explanation

A user burning an NLP position worth less than $1 (i.e., `nlpAmount * oraclePriceX18 / 1e18 < 1e18`) loses 100% of their NLP value. The NLP tokens are returned to the pool (`N_ACCOUNT`) but no corresponding quote is withdrawn from the pool, meaning value is destroyed rather than redistributed. This breaks the invariant that a burn must return `value_burned - fee` to the user.

---

### Likelihood Explanation

This is reachable through the normal `BurnNlp` slow-mode or sequencer-submitted transaction path. A sub-$1 NLP position arises naturally when:
1. A user mints a small NLP position (no minimum mint amount is enforced in `mintNlp`).
2. NLP price declines after minting, pushing an existing position below $1.
3. A user receives a fractional NLP amount via any accounting path.

The `_validateNlpRebalance` call with `-quoteAmount = 0` simply requires the rebalance array to sum to zero, which the sequencer can trivially satisfy (e.g., empty arrays). The maintenance health check at line 526–529 does not prevent the burn from completing if the user has other collateral. [6](#0-5) 

---

### Recommendation

Add a guard that reverts when the burn fee would equal or exceed the gross quote value:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
require(quoteAmount > burnFee, ERR_BURN_FEE_EXCEEDS_VALUE);
quoteAmount = quoteAmount - burnFee;
```

Alternatively, enforce a minimum NLP burn amount such that `nlpAmount.mul(oraclePriceX18) > ONE` before proceeding.

---

### Proof of Concept

```
oraclePriceX18 = 1e18   // NLP price = $1.00
nlpAmount      = 5e17   // 0.5 NLP tokens (valid unlocked balance)

quoteAmount = 5e17 * 1e18 / 1e18 = 5e17   // $0.50, which is < ONE (1e18)
burnFee     = max(1e18, 5e17 / 1000)
            = max(1e18, 5e14)
            = 1e18                          // fee = $1.00 > quoteAmount
quoteAmount = max(0, 5e17 - 1e18)
            = max(0, -5e17)
            = 0

Result:
  spotEngine.updateBalance(NLP_PRODUCT_ID, sender, -5e17)  // NLP burned
  // if-block skipped: no quote credited
  // User loses $0.50 of NLP value; receives $0.00
```
