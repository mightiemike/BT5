### Title
Unsafe Raw `approve()` in `DirectDepositV1.creditDeposit()` Permanently Bricks DDA for Non-Standard ERC20 Tokens — (`core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` as a raw interface call via `IIERC20Base`, which expects `approve` to return a `bool`. For tokens like USDT that do not return a value from `approve`, this call reverts. Additionally, if the endpoint does not consume the full approved allowance, subsequent calls to `creditDeposit()` will revert for tokens (like USDT) that require the allowance to be zero before a new non-zero approval is set. Either path permanently bricks the DDA for the affected token, locking deposited funds.

---

### Finding Description

`DirectDepositV1.creditDeposit()` is a public, permissionless function that iterates over all spot product tokens and attempts to deposit any token balance held by the DDA contract into the endpoint on behalf of the configured subaccount. [1](#0-0) 

At line 92, the raw call is:

```solidity
token.approve(address(endpoint), balance);
``` [2](#0-1) 

`token` is typed as `IIERC20Base`, whose `approve` signature declares `returns (bool)`: [3](#0-2) 

There are two concrete failure modes:

**Path A — Non-returning `approve` (e.g., USDT):** USDT's `approve` emits no return value. When Solidity ABI-decodes the return data expecting a `bool`, it reverts. Every call to `creditDeposit()` for a USDT-like token will revert unconditionally, permanently preventing the DDA from crediting that token to the subaccount.

**Path B — Residual allowance on subsequent calls:** If `depositCollateralWithReferral` does not consume the full approved amount (e.g., due to a minimum deposit check, a cap, or a `uint128` truncation when `balance > type(uint128).max`), a non-zero allowance remains. On the next call to `creditDeposit()`, the raw `token.approve(address(endpoint), balance)` will revert for tokens that prohibit approving a non-zero amount when the current allowance is already non-zero (USDT's well-known behavior).

By contrast, the rest of the codebase uses the `ERC20Helper` library's `safeTransfer` / `safeTransferFrom` wrappers, which handle non-returning tokens correctly: [4](#0-3) 

`creditDeposit()` is the only token-movement path in the DDA that skips this safe wrapper and uses a raw interface call for `approve`.

A secondary instance exists in `ContractOwner.wrapVaultAsset()` (lines 530–531), which also calls raw `assetToken.approve()` via `IERC20Base` with no access control: [5](#0-4) 

---

### Impact Explanation

When `creditDeposit()` is bricked for a given token, any balance of that token sitting in the DDA contract cannot be forwarded to the endpoint. The funds are effectively locked in the DDA with no recovery path callable by the user (only the `onlyOwner` `withdraw()` function can rescue them). This constitutes a concrete asset-locking impact for any user who sent a non-standard ERC20 token to their DDA address expecting it to be credited to their subaccount.

---

### Likelihood Explanation

`creditDeposit()` has no access control — any address can call it at any time. The trigger requires only that a non-standard ERC20 token (e.g., USDT) is listed as a spot product, or that any listed token leaves a residual allowance after a deposit. Both conditions are realistic in a multi-asset protocol. The function is also called internally by `ContractOwner.creditDepositV1()`, which is itself permissionless: [6](#0-5) 

---

### Recommendation

Replace the raw `token.approve()` call in `DirectDepositV1.creditDeposit()` with a safe pattern that:
1. Resets the allowance to zero before setting a new value (to handle USDT's non-zero allowance restriction).
2. Uses a low-level `call` to handle tokens that do not return a `bool` from `approve` — mirroring the existing `safeTransfer` pattern already implemented in `DirectDepositV1` and `ERC20Helper`.

The fix should follow the same pattern as `DirectDepositV1.safeTransfer()`: [7](#0-6) 

Apply the same approach to a `safeApprove` helper: call `approve(spender, 0)` first, then `approve(spender, amount)`, both via low-level `call` with return-data validation.

Apply the same fix to `ContractOwner.wrapVaultAsset()` lines 530–531.

---

### Proof of Concept

1. A spot product is listed with USDT as its collateral token.
2. A user sends USDT to their DDA address.
3. Anyone calls `ContractOwner.creditDepositV1(subaccount)` → `DirectDepositV1.creditDeposit()`.
4. At line 92, `token.approve(address(endpoint), balance)` is executed. USDT's `approve` returns no value; Solidity ABI-decodes the empty return data as `bool` → **revert**.
5. The USDT balance remains locked in the DDA. The user cannot credit their subaccount. Only the `onlyOwner` `withdraw()` path can recover the funds. [1](#0-0)

        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }
