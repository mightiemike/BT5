### Title
`DirectDepositV1.creditDeposit()` Does Not Process Native ETH Balance Despite `isDirectDepositV1Ready()` Counting It — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`ContractOwner.isDirectDepositV1Ready()` explicitly includes native ETH held by the DDA contract in its deposit-readiness balance calculation when the product token is `wrappedNative`. However, `DirectDepositV1.creditDeposit()` — the function that actually executes the deposit — only iterates over ERC20 token balances and never processes native ETH. If native ETH accumulates in the DDA (via `selfdestruct` force-send or a silent constructor wrapping failure), the readiness check returns `true` but the credit call silently processes zero balance for the wrapped-native product, leaving ETH permanently stuck in the DDA and the subaccount uncredited.

---

### Finding Description

`ContractOwner.isDirectDepositV1Ready()` adds `recipient.balance` (native ETH) to the wrapped-native token balance when computing readiness:

```solidity
if (tokenAddr == wrappedNative) {
    balance += recipient.balance;   // native ETH counted here
}
``` [1](#0-0) 

This signals to off-chain infrastructure that the DDA holds enough value to trigger a deposit for the wrapped-native product.

`DirectDepositV1.creditDeposit()`, however, only queries the ERC20 balance of each token:

```solidity
uint256 balance = token.balanceOf(address(this));
if (balance != 0) { ... }
``` [2](#0-1) 

It never reads `address(this).balance`. Native ETH sitting in the DDA is invisible to this function.

Native ETH can reach the DDA without being wrapped in two ways:

1. **Constructor silent failure** — the constructor attempts to wrap any pre-existing ETH balance but explicitly does not revert on failure, emitting `NativeTokenTransferFailed` instead. The ETH remains in the contract. [3](#0-2) 

2. **`selfdestruct` force-send** — a caller deploys a contract funded with ETH and selfdestructs to the DDA address. The EVM credits the ETH directly to the DDA's balance, bypassing the `receive()` function entirely. The `receive()` wrapper is therefore never invoked, and no WETH is minted. [4](#0-3) 

After either event, `isDirectDepositV1Ready(dda, false)` returns `true` (native ETH counted), the off-chain system calls `ContractOwner.creditDepositV1()` → `DirectDepositV1.creditDeposit()`, but `token.balanceOf(dda)` for the wrapped-native product is zero, so the deposit loop skips it silently. The subaccount is never credited. [5](#0-4) 

---

### Impact Explanation

Native ETH that enters the DDA outside the `receive()` path is permanently unprocessable by `creditDeposit()`. The user's subaccount receives no credit for the ETH. Recovery requires the owner to call `withdrawNative()` (owner-only), meaning the user cannot self-serve the recovery. The off-chain deposit pipeline is also misled: `isDirectDepositV1Ready()` returns `true` indefinitely, causing repeated no-op calls to `creditDepositV1()` without ever crediting the subaccount.

---

### Likelihood Explanation

The `selfdestruct` path is the most realistic trigger: any unprivileged caller can deploy a minimal contract funded with a dust amount of ETH and selfdestruct to a known DDA address (DDA addresses are deterministic via CREATE2 with `salt = bytes32(uint256(1))`). The attacker cost is only the gas and the dust ETH. The constructor silent-failure path is lower likelihood but is explicitly acknowledged in the code comments as a known possibility. [6](#0-5) 

---

### Recommendation

`creditDeposit()` should mirror the balance accounting used in `isDirectDepositV1Ready()`. When iterating over product tokens, if the token address equals `wrappedNative` and `address(this).balance > 0`, the function should first wrap the native ETH (call `wrappedNative.call{value: address(this).balance}("")`) and then proceed with the ERC20 deposit. This closes the gap between what the readiness check reports and what the credit function actually processes.

---

### Proof of Concept

1. Compute the deterministic DDA address for a target `subaccount` using the known CREATE2 parameters (`salt = bytes32(uint256(1))`, deployer = `ContractOwner`).
2. Deploy a contract:
   ```solidity
   contract ForceETH {
       constructor(address target) payable {
           selfdestruct(payable(target));
       }
   }
   ```
   with `value = 1 ether` targeting the DDA address (before or after DDA creation).
3. Call `ContractOwner.isDirectDepositV1Ready(dda, false)` — returns `true` because `dda.balance == 1 ether` and `tokenAddr == wrappedNative`.
4. Call `ContractOwner.creditDepositV1(subaccount)` → `DirectDepositV1.creditDeposit()`.
5. Observe: `token.balanceOf(dda)` for the wrapped-native product is `0`; the deposit loop skips it; the subaccount balance is unchanged; `dda.balance` remains `1 ether`.
6. `isDirectDepositV1Ready` continues to return `true` on every subsequent call, but no deposit ever occurs.
