### Title
`meta_tx_v0` hash domain divergence: `only_query` bit injected into meta-tx hash during fee estimation but not during execution or OS proving — (`File: crates/blockifier/src/execution/syscalls/syscall_base.rs`)

### Summary

The `meta_tx_v0` syscall implementation in the blockifier propagates the outer transaction's `only_query` flag into the meta-transaction hash via `signed_tx_version`. During fee estimation (`only_query = true`) the meta-tx hash is computed over `version = 0 | QUERY_VERSION_BASE`, while during actual execution (`only_query = false`) it is computed over `version = 0`. The Starknet OS always uses `version = 0`. This three-way split means fee estimation and actual execution produce structurally different meta-tx hashes, so any account contract that verifies a signature against `tx_info.transaction_hash` inside a meta-tx will see a different digest under `starknet_estimate_fee` than under `starknet_add_invoke_transaction`, making fee estimation an authoritative-looking but wrong oracle.

### Finding Description

**Blockifier path** (`crates/blockifier/src/execution/syscalls/syscall_base.rs`, lines 314–328):

```rust
let only_query = old_tx_context.tx_info.only_query();   // true during estimate_fee

let transaction_hash = InvokeTransactionV0 { … }
    .calculate_transaction_hash(
        &self.context.tx_context.block_context.chain_info.chain_id,
        &signed_tx_version(
            &TransactionVersion::ZERO,
            &TransactionOptions { only_query },   // ← query bit injected here
        ),
    )?;
```

`signed_tx_version` with `only_query = true` returns `0 + QUERY_VERSION_BASE` (bit 128 set). The resulting hash is therefore:

```
Pedersen("invoke", 0x100000000000000000000000000000000, contract_address, …)
```

**OS path** (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`, lines 295–315):

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(…) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,          // ← always 0, no query bit
        …
    );
    return tx_hash;
}
```

The OS (and the blockifier during actual execution) always produces:

```
Pedersen("invoke", 0, contract_address, …)
```

These are two distinct field elements. The `tx_info.transaction_hash` value exposed to the meta-tx callee via `get_execution_info` therefore differs between fee estimation and execution.

### Impact Explanation

Any account contract that signs and verifies against the meta-tx hash (the standard pattern for `meta_tx_v0`) will behave differently under fee estimation versus execution:

- **Scenario A** – user signs the execution hash (`version=0`): fee estimation reverts (wrong hash → bad signature), execution succeeds. `starknet_estimate_fee` returns a revert, misleading the caller into believing the transaction will fail.
- **Scenario B** – user signs the query hash (`version=0|QUERY_VERSION_BASE`): fee estimation succeeds, execution reverts. `starknet_estimate_fee` returns a fee, the caller submits the transaction, and it fails on-chain.

Both scenarios constitute an authoritative-looking wrong value from the RPC fee-estimation endpoint, matching the **High** impact tier: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

### Likelihood Explanation

`meta_tx_v0` is a production syscall present in both the blockifier and the OS. Any integrator building a session-key, paymaster, or delegated-execution contract on top of `meta_tx_v0` with standard ECDSA signature verification will hit this divergence the first time they call `starknet_estimate_fee`. The trigger is unprivileged: any user can submit a fee-estimation RPC call.

### Recommendation

Remove the `only_query` propagation from the meta-tx hash computation. The meta-tx hash represents a canonical, chain-bound transaction identity and must be independent of the simulation context:

```rust
// Fix: always use version 0, never the query bit
let transaction_hash = InvokeTransactionV0 { … }
    .calculate_transaction_hash(
        &self.context.tx_context.block_context.chain_info.chain_id,
        &TransactionVersion::ZERO,   // no signed_tx_version wrapper
    )?;
```
