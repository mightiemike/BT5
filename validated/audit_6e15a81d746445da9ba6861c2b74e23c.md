### Title
Gateway `run_validate_entry_point` uses stale `starknet_version` from previous block to compute `virtual_os_config_hash`, binding the wrong hash function at the `OsConfigHashVersion` cutover boundary — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`run_validate_entry_point` fetches `block_info` from the latest committed block and increments only `block_number`, leaving `starknet_version` at the previous block's value. `virtual_os_config_hash()` is derived from that stale `starknet_version`. At the `OsConfigHashVersion` cutover (V0_14_2 → V0_14_3), the gateway computes the expected config hash with Pedersen (V3) while the blockifier, which receives the correct next-block `starknet_version`, computes it with Blake (V4). The two sides disagree on the canonical config hash, causing the gateway to accept proof-carrying transactions that the blockifier will reject, and to reject proof-carrying transactions that the blockifier would accept.

---

### Finding Description

**The stale-version assignment:**

In `run_validate_entry_point`, only `block_number` is patched forward; `starknet_version` is left at the previous block's value:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
// starknet_version is NOT updated — it remains the previous block's version
let block_context = BlockContext::new(
    block_info,
    self.chain_info.clone(),
    versioned_constants,
    BouncerConfig::max(),
);
``` [1](#0-0) 

**The version-gated hash function:**

`virtual_os_config_hash()` passes `self.block_info.starknet_version` directly into `compute_virtual_os_config_hash`:

```rust
pub(crate) fn virtual_os_config_hash(&self) -> Felt {
    *self.virtual_os_config_hash.get_or_init(|| {
        OsChainInfo::from(&self.chain_info)
            .compute_virtual_os_config_hash(self.block_info.starknet_version)
            .expect("Failed to compute OS config hash")
    })
}
``` [2](#0-1) 

`compute_os_config_hash` selects the hash algorithm based on `OsConfigHashVersion::from(starknet_version)`:

```rust
Ok(match hash_version {
    OsConfigHashVersion::V3 => Pedersen::hash_array(&data),
    OsConfigHashVersion::V4 => {
        Blake2Felt252::encode_felt252_data_and_calc_blake_hash(&data)
    }
})
``` [3](#0-2) 

The cutover is hard-gated: `V0_14_2` → V3 (Pedersen); `V0_14_3` and above → V4 (Blake): [4](#0-3) 

**The validation check that diverges:**

`validate_proof_facts` compares the transaction's `config_hash` against `block_context.virtual_os_config_hash()`:

```rust
let virtual_os_config_hash = block_context.virtual_os_config_hash();
let proof_config_hash = snos_proof_facts.config_hash;
if virtual_os_config_hash != proof_config_hash {
    return Err(TransactionPreValidationError::InvalidProofFacts(...));
}
``` [5](#0-4) 

At the V0_14_2 → V0_14_3 cutover, the gateway's `block_context` carries `starknet_version = V0_14_2` (Pedersen), while the batcher's `block_context` for the same block carries `starknet_version = V0_14_3` (Blake). The two sides compute different values for `virtual_os_config_hash` for the same `(chain_id, strk_fee_token_address)` pair, because the version string and hash primitive both change:

- V3 preimage: `[STARKNET_OS_CONFIG_HASH_VERSION_V3, chain_id, fee_token]` → Pedersen
- V4 preimage: `[STARKNET_OS_CONFIG_HASH_VERSION_V4, chain_id, fee_token]` → Blake [6](#0-5) 

---

### Impact Explanation

Two divergent outcomes occur for proof-carrying Invoke V3 transactions submitted at the cutover block:

1. **False acceptance (gateway admits, blockifier rejects):** A transaction embedding a Pedersen-based `config_hash` (correct for V0_14_2) passes gateway pre-validation (gateway uses Pedersen), enters the mempool, is included in a block proposal, and is then rejected by the blockifier's `validate_proof_facts` (which uses Blake for V0_14_3). The gateway admitted an invalid transaction before sequencing.

2. **False rejection (gateway rejects, blockifier would accept):** A transaction embedding a Blake-based `config_hash` (correct for V0_14_3) is rejected by the gateway (which uses Pedersen), never reaching the mempool. A valid transaction is denied before sequencing.

Both outcomes match the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The trigger is the single-block window at the `OsConfigHashVersion` V3→V4 cutover (V0_14_2 → V0_14_3). Any user submitting a proof-carrying transaction during that window is affected. The cutover is a scheduled protocol upgrade event, so the window is predictable and narrow but real. No privileged access is required; any unprivileged user submitting an Invoke V3 with non-empty `proof_facts` triggers the divergence.

---

### Recommendation

In `run_validate_entry_point`, after fetching `block_info` from the previous block, also advance `starknet_version` to the version the sequencer intends for the block being built, mirroring the same logic used by the batcher when constructing its `BlockContext`. Concretely, the gateway should obtain the next block's `starknet_version` from the node configuration or the batcher's block-building context and set it on `block_info` before constructing `BlockContext`:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
block_info.starknet_version = self.config.next_block_starknet_version; // <-- add this
```

This ensures `virtual_os_config_hash()` uses the same hash function as the blockifier will use during execution.

---

### Proof of Concept

Assume the latest committed block is at `StarknetVersion::V0_14_2` and the sequencer is about to build the first block at `V0_14_3`.

**Step 1 — Gateway computes wrong config hash:**
```
gateway block_info.starknet_version = V0_14_2
→ OsConfigHashVersion::V3 (Pedersen)
→ expected_config_hash = Pedersen([STARKNET_OS_CONFIG_HASH_VERSION_V3, chain_id, fee_token])
```

**Step 2 — User submits a valid proof-carrying transaction for V0_14_3:**
```
tx.proof_facts.config_hash = Blake([STARKNET_OS_CONFIG_HASH_VERSION_V4, chain_id, fee_token])
```

**Step 3 — Gateway rejects the transaction:**
```
virtual_os_config_hash (Pedersen) ≠ proof_config_hash (Blake)
→ TransactionPreValidationError::InvalidProofFacts("Virtual OS config hash mismatch")
```

The valid transaction never reaches the mempool.

**Converse (false acceptance):**
A user submits a transaction with `config_hash = Pedersen(...)` (the V0_14_2 value). The gateway accepts it. The batcher builds the block at V0_14_3, computes `virtual_os_config_hash` with Blake, finds a mismatch, and rejects the transaction during execution — after the gateway already admitted it.
