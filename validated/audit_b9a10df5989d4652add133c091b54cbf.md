### Title
Linked Signer Can Self-Escalate to Persist Access After Revocation â€” (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type is processed with `allowLinkedSigner = true`, meaning the current linked signer (a session key) can sign a `LinkSigner` transaction to change the linked signer to any address. A compromised linked signer can use this to transfer signing authority to an attacker-controlled address before the subaccount owner can revoke it, maintaining persistent unauthorized access â€” a direct on-chain analog to the "failed to invalidate session after logout" class.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` routes through `validateSignature`, which passes the current linked signer address to the verifier: [2](#0-1) 

The verifier then accepts a valid signature from **either** the subaccount owner **or** the current linked signer. This means the linked signer â€” intended to be a limited session key â€” has the authority to sign a `LinkSigner` transaction and change the `linkedSigners` mapping to any address it chooses: [3](#0-2) 

The `getLinkedSigner` function resolves the active linked signer at call time: [4](#0-3) 

Once a linked signer is set, it has the same signing authority as the subaccount owner for all transactions that pass `allowLinkedSigner = true`, including `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, `MintNlp`, `BurnNlp`, `TransferQuote`, and `LinkSigner` itself. [5](#0-4) 

The slow-mode `LinkSigner` path does **not** share this flaw â€” it only calls `validateSender`, which enforces that `msg.sender` matches the address embedded in the subaccount, so only the owner can use it: [6](#0-5) 

The vulnerability is isolated to the fast (sequencer) path.

---

### Impact Explanation

A compromised linked signer can sign a `LinkSigner` transaction to change the linked signer to an attacker-controlled address **before** the subaccount owner can revoke it. The attacker then holds a new linked signer they fully control. With that new linked signer they can:

- Sign `WithdrawCollateral` / `WithdrawCollateralV2` to drain collateral from the subaccount.
- Sign `LiquidateSubaccount` to liquidate other accounts at a profit, charging fees to the victim.
- Sign `MintNlp` / `BurnNlp` / `TransferQuote` to manipulate the victim's NLP position or quote balance.
- Sign another `LinkSigner` to re-establish control after each revocation attempt, creating a persistent access loop.

The corrupted state is `linkedSigners[subaccount]` and, downstream, any balance delta produced by the above operations.

---

### Likelihood Explanation

Linked signers are session keys â€” typically hot keys stored in a browser or mobile wallet â€” and are materially more likely to be compromised than the subaccount owner's cold key. The attack requires only the linked signer's private key, no admin access, no sequencer compromise, and no governance capture. The attacker can act the moment the key is obtained, before the owner notices. Likelihood is **moderate**.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner (the address embedded in the `sender` bytes32) should be permitted to change the linked signer. The linked signer is a delegated session key and must not have the authority to re-delegate or transfer that authority. [7](#0-6) 

---

### Proof of Concept

1. User calls `LinkSigner(sender=S, signer=A, nonce=N)` signed by the subaccount owner â†’ `linkedSigners[S] = A`.
2. Attacker obtains signer A's private key (e.g., from a compromised browser session).
3. Attacker uses signer A to sign `LinkSigner(sender=S, signer=AttackerAddress, nonce=N+1)` and submits it to the sequencer.
4. Sequencer processes it â†’ `linkedSigners[S] = AttackerAddress`. The attacker now holds a fresh linked signer they fully control.
5. Attacker uses `AttackerAddress` to sign `WithdrawCollateral(sender=S, productId=QUOTE, amount=MAX, nonce=N+2)` and submits it.
6. User notices the compromise and submits `LinkSigner(sender=S, signer=0x0, nonce=N+3)` signed by the subaccount owner.
7. Sequencer processes the revocation â†’ `linkedSigners[S] = address(0)`. But the withdrawal in step 5 was already processed with nonce N+2, draining the subaccount.
8. If the attacker pre-signs `LinkSigner(sender=S, signer=AttackerAddress, nonce=N+3)` and races the user's revocation to the sequencer, the attacker can re-establish control before the revocation lands, repeating the cycle.

The root cause â€” `allowLinkedSigner = true` on the `LinkSigner` branch â€” is the necessary and sufficient condition for this attack path. [1](#0-0)
