### Title
Linked Signer Can Overwrite Its Own Authorization to Permanently Escalate Subaccount Control — (`File: core/contracts/EndpointTx.sol`)

---

### Summary
In the sequencer-path `LinkSigner` handler, `validateSignedTx` is called with `allowLinkedSigner = true`, meaning the **current linked signer** of a subaccount can sign a new `LinkSigner` transaction to replace the linked signer with any address it chooses. This is a direct privilege-escalation analog: just as the original report's `owner` could replace `GOVERNANCE`, a linked signer can replace itself, permanently seizing subaccount signing authority and enabling collateral theft via `TransferQuote`.

---

### Finding Description

`EndpointTx.processTransactionImpl` handles the sequencer-path `LinkSigner` transaction at lines 576–590:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the authorized signer as either the subaccount owner **or** the current linked signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if it recovers to either the subaccount owner address or the linked signer address:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means the **linked signer itself** can produce a valid `SignedLinkSigner` transaction that overwrites `linkedSigners[subaccount]` with any address — including a fresh attacker-controlled key.

**Contrast with the slow-mode path**, where `LinkSigner` is gated by `validateSender`, which requires `msg.sender == address(uint160(bytes20(txSender)))` — only the actual subaccount owner address:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // owner-only check
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
``` [4](#0-3) 

The slow-mode path correctly restricts `LinkSigner` to the subaccount owner. The sequencer path does not.

---

### Impact Explanation

Once the linked signer has overwritten `linkedSigners[subaccount]` to an attacker-controlled address, the attacker holds full signing authority over the subaccount. The sequencer-path `TransferQuote` handler also accepts linked-signer signatures (`allowLinkedSigner = true`):

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true
);
...
clearinghouse.transferQuote(signedTx.tx);
``` [5](#0-4) 

The attacker can sign a `TransferQuote` transaction moving the victim's entire quote balance to an attacker-controlled subaccount, then withdraw from there. The victim's only recourse is the slow-mode `LinkSigner` revocation, which is subject to a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`):

```solidity
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    ...
});
``` [6](#0-5) 

During those 3 days the attacker retains full signing authority and can drain all transferable collateral.

**Impact: High** — direct theft of user collateral is possible.

---

### Likelihood Explanation

**Likelihood: Low.** Exploitation requires that a subaccount owner has previously granted a linked signer (e.g., an API trading key or a third-party bot), and that the linked signer is malicious or has been compromised. This is a realistic scenario for active traders using automated systems, but it is not the default state for all users.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the sequencer-path `LinkSigner` handler, so that only the subaccount owner's key can authorize a linked-signer change:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // linked signer must NOT be able to change the linked signer
);
``` [7](#0-6) 

This aligns the sequencer path with the slow-mode path, which already enforces owner-only authorization for `LinkSigner`.

---

### Proof of Concept

1. **Setup**: Victim subaccount `V` has deposited collateral. Victim grants linked signer `LS` (e.g., a trading bot key) via a legitimate `LinkSigner` transaction. `linkedSigners[V] = LS`.

2. **Attack step 1 — escalate**: Attacker (controlling `LS`) crafts `SignedLinkSigner { tx: { sender: V, signer: ATTACKER, nonce: N }, signature: sig_LS }`. The signature is produced by `LS` over the EIP-712 digest of `(LINK_SIGNER_SIGNATURE, V, ATTACKER, N)`. [8](#0-7) 
   Sequencer includes this transaction. `validateSignedTx` passes because `recovered == LS == getLinkedSigner(V)`. Result: `linkedSigners[V] = ATTACKER`.

3. **Attack step 2 — drain**: Attacker crafts `SignedTransferQuote { tx: { sender: V, recipient: ATTACKER_SUBACCOUNT, amount: ALL, nonce: N+1 }, signature: sig_ATTACKER }`. Sequencer includes it. `validateSignedTx` passes because `recovered == ATTACKER == getLinkedSigner(V)`. `clearinghouse.transferQuote` moves all quote balance from `V` to `ATTACKER_SUBACCOUNT`. [9](#0-8) 

4. **Attack step 3 — withdraw**: Attacker withdraws from `ATTACKER_SUBACCOUNT` normally. Victim's slow-mode revocation (3-day delay) arrives too late.
