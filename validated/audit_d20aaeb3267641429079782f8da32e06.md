### Title
Deprecated `verify_transaction_inclusion` Bypasses Coinbase Merkle Proof Validation, Enabling 64-Byte Transaction Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two public SPV verification entry points. The newer `verify_transaction_inclusion_v2` enforces a coinbase Merkle proof to prevent the 64-byte transaction forgery attack. The original `verify_transaction_inclusion` does not. Because the original function is only marked `#[deprecated]` — not removed, not access-controlled, and not paused by default — any unprivileged NEAR caller can invoke it directly, bypassing the coinbase proof requirement and obtaining a `true` result for a forged transaction inclusion proof.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte Merkle proof forgery vulnerability (documented inline and in the function's doc comment). It does so by requiring the caller to also supply a coinbase Merkle proof, which forces the verifier to confirm that the claimed `tx_id` is a leaf node (a real transaction) rather than an internal Merkle tree node: [1](#0-0) 

The deprecated `verify_transaction_inclusion` performs no such check. It only verifies that the supplied Merkle proof produces the block's `merkle_root`, which is equally true for internal nodes: [2](#0-1) 

The function is still `pub`, carries only `#[pause]` (not paused by default), and has no role-based access control: [3](#0-2) 

The contract's own warning acknowledges the gap but delegates responsibility to callers: [4](#0-3) 

This is structurally identical to the reported vulnerability class: a security invariant (coinbase proof validation) is enforced in one entry point (`verify_transaction_inclusion_v2`) but absent in an alternative entry point (`verify_transaction_inclusion`) that achieves the same observable result — a `bool` proof verdict consumed by downstream contracts.

---

### Impact Explanation

Any downstream NEAR contract or off-chain relayer that calls `verify_transaction_inclusion` to gate a security-critical action (e.g., releasing bridged funds, confirming a deposit) can be deceived. An attacker supplies a `tx_id` equal to an internal Merkle tree node and a valid Merkle path for that node. The function returns `true` for a transaction that does not exist as a leaf in the block. The corrupted value is the **proof result** returned to the caller.

---

### Likelihood Explanation

The attack requires no privileges, no mining, and no brute force. Any block already accepted by the contract provides a valid Merkle tree. The attacker computes an internal node at any depth, constructs the corresponding Merkle path (all data is public on-chain), and calls the deprecated public function. The entry path is fully attacker-controlled and requires only a standard NEAR function call.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or add the same coinbase Merkle proof validation that `verify_transaction_inclusion_v2` performs before delegating to the shared proof logic. Deprecation alone does not prevent the function from being called on-chain.

---

### Proof of Concept

1. Identify any confirmed block `B` in the contract's main chain with at least two transactions (so the Merkle tree has at least one internal node).
2. Compute the internal node `N = SHA256d(tx0_hash || tx1_hash)` — this is the Merkle root itself for a two-transaction block, or any intermediate node for larger blocks.
3. Construct a Merkle proof path for `N` at its position in the tree (all data is public).
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node, not a real transaction)
   - `tx_block_blockhash = B`
   - `tx_index` = position of `N` in the tree
   - `merkle_proof` = the path computed in step 3
   - `confirmations = 1`
5. The function returns `true`. [5](#0-4) 

The check `compute_root_from_merkle_proof(N, index, path) == merkle_root` succeeds because `N` is a genuine node in the tree. No coinbase proof is required, so the forgery is undetected.

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
