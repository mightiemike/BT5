### Title
Fee-on-Transfer Token Deposit Inflates User Balance Beyond Actual Collateral Received - (File: `core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` transfers a caller-specified `amount` from the depositor and queues a slow-mode credit for that same `amount`. `Clearinghouse.depositCollateral` then unconditionally credits the full `amount` to the user's spot balance. For any ERC20 token that deducts a transfer fee (fee-on-transfer token), the clearinghouse receives fewer tokens than it credits, permanently inflating internal accounting above actual holdings and corrupting protocol solvency.

---

### Finding Description

In `Endpoint.depositCollateralWithReferral`, the deposit flow is:

1. `handleDepositTransfer` pulls exactly `uint256(amount)` from `msg.sender` into the clearinghouse.
2. A `SlowModeTx` is enqueued encoding `DepositCollateral{sender, productId, amount}` — the original caller-supplied value.
3. When the sequencer (or slow-mode executor) processes the transaction, `Clearinghouse.depositCollateral` executes:

```solidity
// Clearinghouse.sol lines 204-207
int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
```

`txn.amount` is the original caller-supplied value, not the amount actually received by the clearinghouse. No balance snapshot is taken before or after `handleDepositTransfer` to measure the true received amount.

For a fee-on-transfer token with fee rate `f`, the clearinghouse receives `amount × (1 − f)` but credits `amount`. Every deposit inflates the user's balance by `amount × f` beyond actual holdings.

The `assertUtilization` check in `SpotEngine` (lines 232–241) compares `totalDepositsNormalized × cumulativeDepositsMultiplierX18` against `totalBorrowsNormalized × cumulativeBorrowsMultiplierX18` — both derived from internal accounting, not from `token.balanceOf(clearinghouse)`. It therefore cannot detect the growing shortfall.

The `addOrUpdateProduct` function in `SpotEngine` (line 68) accepts any ERC20 token address via `Config.token`, and no code path enforces that listed tokens must not have transfer fees.

---

### Impact Explanation

**Solvency/accounting corruption.** The invariant `sum(user balances for token T) == clearinghouse.balanceOf(T)` is broken on every deposit of a fee-on-transfer token. The deficit grows linearly with deposit volume. When aggregate withdrawals approach actual holdings, `safeTransfer` calls in `handleWithdrawTransfer` (Clearinghouse.sol line 383) begin reverting for the last withdrawers, permanently locking their credited-but-non-existent collateral. The insurance fund does not cover this class of shortfall because it is denominated in quote (USDC), not in the affected token.

---

### Likelihood Explanation

**Medium.** The protocol's `addOrUpdateProduct` is owner-gated, so exploitation requires a fee-on-transfer token to be listed. However, many real ERC20 tokens (e.g., tokens with configurable fees, rebasing tokens with withdrawal fees, or tokens that add fees post-listing) satisfy this condition. Once such a token is listed, every ordinary user deposit — with no special attacker action — silently inflates accounting. The attacker role is simply any depositor of that token.

---

### Recommendation

Measure the actual received amount using a pre/post balance snapshot around `handleDepositTransfer`, and use the delta as the credited amount:

```solidity
uint256 before = IERC20Base(token).balanceOf(address(clearinghouse));
handleDepositTransfer(token, msg.sender, uint256(amount));
uint256 received = IERC20Base(token).balanceOf(address(clearinghouse)) - before;
// enqueue SlowModeTx with received (cast to uint128) instead of amount
```

Alternatively, explicitly document and enforce (via `require`) that only non-fee-on-transfer tokens may be listed as supported products.

---

### Proof of Concept

1. Owner calls `SpotEngine.addOrUpdateProduct` listing a fee-on-transfer token (1 % fee) as `productId = 5`.
2. Alice calls `Endpoint.depositCollateral(subaccountName, 5, 1000e18)`.
3. `handleDepositTransfer` executes `safeTransferFrom(alice, clearinghouse, 1000e18)`; clearinghouse receives **990e18** (1 % fee deducted by the token).
4. `SlowModeTx` is queued with `amount = 1000e18`.
5. Sequencer processes the tx; `Clearinghouse.depositCollateral` credits Alice **1000e18 × multiplier**.
6. Alice's internal balance: **1000e18**. Clearinghouse actual holding: **990e18**. Deficit: **10e18**.
7. After 100 identical deposits by different users, total credited: **100 000e18**; actual holding: **99 000e18**; deficit: **1 000e18**.

        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        uint8 decimals = _decimals(txn.productId);
