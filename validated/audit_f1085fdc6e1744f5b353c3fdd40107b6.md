Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is the `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` is used, the pool's `msg.sender` is the router contract, not the originating user. A pool admin who allowlists the router to enable routing for permitted users inadvertently grants every unpermitted address the ability to bypass the per-user allowlist by routing through the router.

## Finding Description
**Step 1:** `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, passing its own `msg.sender` as `sender`. [1](#0-0) 

**Step 2:** `ExtensionCalling._beforeSwap` forwards `sender` unchanged as the first argument to every configured extension via `_callExtensionsInOrder`. [2](#0-1) 

**Step 3:** `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When the router is the caller, `sender == address(router)`. [3](#0-2) 

**Step 4:** `MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making the router the `msg.sender` of that pool call. The same holds for `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

**Root cause:** The extension has no mechanism to identify the originating user — it only receives `sender`, which collapses to `address(router)` for all router-mediated swaps. `DepositAllowlistExtension` avoids this by ignoring the `sender` (operator) argument and checking `owner` (the position beneficiary) instead, which is the economically relevant actor. [5](#0-4) 

`SwapAllowlistExtension` has no equivalent "actual user" field to inspect. Once the router is allowlisted, `allowedSwapper[pool][router] == true` satisfies the check for every caller regardless of their individual allowlist status. [6](#0-5) 

## Impact Explanation
A pool admin who wants allowlisted users to use the canonical router must call `setAllowedToSwap(pool, router, true)`. Once set, **any** address — including addresses explicitly not on the allowlist — can call `router.exactInputSingle(...)` and the extension will pass because it only checks `allowedSwapper[pool][router]`. The per-user curation is completely defeated: unauthorized users can execute swaps on a pool designed to restrict access to a curated set of counterparties. This constitutes broken core pool functionality — the allowlist extension fails to enforce its stated invariant, allowing unauthorized fund flows through a restricted pool.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical public swap entrypoint. Any pool admin enabling routing for curated pools will naturally allowlist the router — this is the intended usage pattern, not an edge case. Once the router is allowlisted, the bypass requires zero privilege: any EOA calls `exactInputSingle` with the target pool address. The bypass is silent — no revert, no event distinguishing the bypassing user from a legitimate one. All four router entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) are affected.

## Recommendation
The extension must identify the originating user, not just the immediate caller of `pool.swap()`. Two viable approaches:

1. **Pass the real user in `extensionData`**: The router encodes `msg.sender` into `extensionData` before forwarding to the pool; the extension decodes and verifies it. The extension should also require that `sender` (the router) is itself allowlisted, so the decoded user address cannot be spoofed by an arbitrary caller.

2. **Allowlist at the router level**: Deploy a router wrapper that enforces per-user access before calling the pool, and allowlist only that wrapper. The pool-level extension then only needs to verify the trusted wrapper is the caller.

The core invariant to restore: the identity checked by the allowlist must be the economically relevant actor — the user whose funds are at risk — not the intermediate contract routing the call.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured (beforeSwap order = extension 1).
  - Admin calls setAllowedToSwap(pool, router, true)   // enable routing for permitted users
  - Admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  attacker calls router.exactInputSingle({
    pool:      <curated pool>,
    recipient: attacker,
    zeroForOne: true,
    amountIn:  X,
    ...
  })

Trace:
  router.exactInputSingle()
    → pool.swap(recipient=attacker, ...)   // msg.sender = router
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (no revert)
      → swap executes, attacker receives tokens

Result:
  attacker bypasses the per-user allowlist and swaps on a curated pool.
  allowedSwapper[pool][attacker] was never set to true.
  The wrong value: allowedSwapper[pool][router] is checked instead of
  allowedSwapper[pool][attacker], causing the extension to return
  IMetricOmmExtensions.beforeSwap.selector instead of reverting.
```
