### Title
Linked Signer Can Permanently Hijack Subaccount by Overwriting `linkedSigners` — (`core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the **current linked signer** — not just the subaccount owner — can sign a `LinkSigner` transaction to replace themselves with an attacker-controlled address, permanently seizing signing authority over the victim's subaccount.

---

### Finding Description

The fast-path (sequencer-processed) `LinkSigner` handler in `EndpointTx.processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the acceptable signer as either the subaccount owner **or** the current linked signer: [2](#0-1) 

This means a linked signer can sign a `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with an attacker-controlled address.

By contrast, the slow-mode `LinkSigner` path correctly enforces that only the subaccount owner can change the linked signer, via `validateSender(txn.sender, sender)` which checks `address(uint160(bytes20(txn.sender))) == msg.sender`: [3](#0-2) 

The two paths are inconsistent. The slow-mode path is safe; the fast-mode path is not.

---

### Impact Explanation

Once the attacker controls the linked signer for a victim's subaccount, they can sign any transaction that accepts `allowLinkedSigner = true`, including:

- **`WithdrawCollateral` (V1)** — initiates a withdrawal to the subaccount owner's address (not directly to the attacker, but drains the subaccount)
- **`MatchOrders`** — the attacker can sign orders on behalf of the victim's subaccount and match them against an attacker-controlled counterparty subaccount at maximally unfavorable prices, transferring the victim's quote balance to the attacker's subaccount
- **`MintNlp` / `BurnNlp`** — can be used to manipulate the victim's NLP position [4](#0-3) 

The `MatchOrders` path passes the linked signer directly to `OffchainExchange.matchOrders` without any additional ownership check: [5](#0-4) 

The victim's subaccount can be fully drained through unfavorable trades. The original owner can submit a slow-mode `LinkSigner` to recover, but slow-mode transactions have a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`), during which the attacker can complete the drain.

---

### Likelihood Explanation

Any user who has ever called `LinkSigner` to authorize a hot wallet or trading bot is exposed. The attack requires only that the linked signer be malicious or compromised — a realistic condition for hot wallets used in automated trading. The sequencer processes all valid signed transactions and has no mechanism to distinguish a legitimate `LinkSigner` from a malicious one signed by the current linked signer.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`:

```solidity
// Before (vulnerable):
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // ← allows linked signer to overwrite itself
);

// After (fixed):
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // ← only subaccount owner can change linked signer
);
```

This aligns the fast-mode path with the slow-mode path, which already enforces owner-only control over the linked signer. [1](#0-0) 

---

### Proof of Concept

1. User A owns subaccount `subA` (address `0xAAAA...`) with 10,000 USDC and has linked signer `0xBBBB...` (signer B) for trading.
2. Signer B (malicious) constructs a `SignedLinkSigner` transaction: `{sender: subA, signer: 0xCCCC..., nonce: N}` and signs it with key B.
3. The sequencer receives this transaction. `validateSignedTx(subA, N, tx, sig, true)` passes because `getLinkedSigner(subA)` returns `0xBBBB...` and the signature is valid.
4. `linkedSigners[subA]` is overwritten with `0xCCCC...` (attacker-controlled).
5. Attacker (controlling `0xCCCC...`) creates a sell order for asset X at price P (extremely high) signed by their own subaccount `subAttacker`, and a buy order for asset X at price P signed by `subA` using key `0xCCCC...`.
6. The sequencer processes `MatchOrders`: `subA` buys asset X at price P, transferring 10,000 USDC to `subAttacker`.
7. User A's subaccount is drained. The attacker has 10,000 USDC in `subAttacker`.
8. User A submits a slow-mode `LinkSigner` to recover, but the 3-day delay means the drain is complete before recovery.
