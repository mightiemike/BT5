### Title
Third-party `TransferToGasKey` can permanently lock a contract account's gas key balance above `MAX_BALANCE_TO_BURN`, making the key and account undeletable — (`runtime/runtime/src/access_keys.rs`)

---

### Summary

The `GasKeys` feature (protocol version 85) introduces an asymmetry: any account or contract can fund a gas key on any other account via `TransferToGasKey` (available as both a transaction action and a host function), but `WithdrawFromGasKey` is intentionally restricted to transactions only — no host function exists. When a contract account (one without a full-access key) holds a gas key, any unprivileged third party can push the gas key balance above `GasKeyInfo::MAX_BALANCE_TO_BURN = 1 NEAR`. This permanently blocks both `DeleteKey` and `DeleteAccount` on that account. Because the contract cannot sign a `WithdrawFromGasKey` transaction and no host function exists for it, the NEAR tokens in the gas key balance become permanently irrecoverable — an exact protocol-layer analog to the donated-tokens-lost-on-shelter-deactivation bug.

---

### Finding Description

**Step 1 — Permissionless funding.**
`action_transfer_to_gas_key` performs no ownership check on the caller. Any receipt whose receiver is the victim account and whose action list contains `TransferToGasKey { public_key: K, deposit: X }` will succeed as long as gas key `K` exists on the victim account. [1](#0-0) 

The corresponding host function `promise_batch_action_transfer_to_gas_key` makes this reachable from any deployed contract: [2](#0-1) 

**Step 2 — Deletion blocked above threshold.**
`delete_gas_key` (called from both `action_delete_key` and `action_delete_account`) hard-blocks deletion when `gas_key_info.balance > GasKeyInfo::MAX_BALANCE_TO_BURN`: [3](#0-2) 

The threshold is exactly 1 NEAR: [4](#0-3) 

`action_delete_account` applies the same guard over the aggregate of all gas key balances: [5](#0-4) 

**Step 3 — No host-function escape hatch.**
`WithdrawFromGasKey` is explicitly documented as transaction-only with no host function: [6](#0-5) 

A contract account that has no full-access key cannot sign a `WithdrawFromGasKey` transaction. There is no `promise_batch_action_withdraw_from_gas_key` host function in `VMLogic`. The contract is therefore unable to drain the balance below the threshold by any on-chain path.

---

### Impact Explanation

For any contract account (no full-access key) that holds a gas key:

1. An attacker sends a single transaction or contract call with `TransferToGasKey { public_key: K, deposit: 1 NEAR + 1 yocto }` targeting the victim account.
2. `gas_key_info.balance` now exceeds `MAX_BALANCE_TO_BURN`.
3. Every subsequent `DeleteKey(K)` receipt fails with `GasKeyBalanceTooHigh`.
4. Every subsequent `DeleteAccount` receipt fails with `GasKeyBalanceTooHigh` (aggregate check).
5. The contract has no host function to call `WithdrawFromGasKey`.
6. The NEAR tokens in the gas key balance are permanently irrecoverable — they cannot be burned (deletion blocked), cannot be withdrawn (no host function), and cannot be transferred (gas key balance is not the account balance).

The account is permanently undeletable and the attacker-deposited NEAR is permanently locked in protocol state. The cost to the attacker is slightly more than 1 NEAR per victim gas key.

---

### Likelihood Explanation

- `TransferToGasKey` is a standard action reachable from any transaction or contract call; no special privilege is required.
- The public key of a gas key is visible via `view_access_key_list` RPC, so the attacker can enumerate targets.
- Gas keys are a new feature (protocol version 85, stabilized in 2.13) and are expected to be used by relayer contracts and wallet contracts — exactly the class of accounts that typically operate without a full-access key.
- The cost to trigger the condition is slightly above 1 NEAR, which is economically feasible for a targeted griefing attack.

---

### Recommendation

One or more of the following mitigations should be applied:

1. **Add a `promise_batch_action_withdraw_from_gas_key` host function** so contract accounts can drain their own gas key balance before deletion. This is the most direct fix and mirrors the symmetry that `TransferToGasKey` already has.

2. **Cap third-party funding.** Validate in `action_transfer_to_gas_key` that the receipt predecessor is the receiver account itself (or a designated authorized funder), preventing unprivileged accounts from pushing the balance above the deletion threshold.

3. **Remove the hard block; redirect excess balance.** Instead of returning `GasKeyBalanceTooHigh`, transfer the gas key balance to the account's own balance (or to the beneficiary on `DeleteAccount`) unconditionally. The 1 NEAR cap was designed to limit accidental burns, not to create an irrecoverable state.

---

### Proof of Concept

```
Victim:   contract.near  (no full-access key, has gas key with public key K)
Attacker: attacker.near

1. attacker.near submits:
   Transaction {
     receiver_id: "contract.near",
     actions: [TransferToGasKey { public_key: K, deposit: 1_000_000_000_000_000_000_000_001 }]
   }

2. Runtime executes action_transfer_to_gas_key on contract.near:
   gas_key_info.balance += 1_000_000_000_000_000_000_000_001
   // balance now = 1 NEAR + 1 yocto > MAX_BALANCE_TO_BURN

3. contract.near submits (via its own contract logic):
   Receipt { actions: [DeleteKey { public_key: K }] }
   → delete_gas_key: balance (1 NEAR + 1 yocto) > MAX_BALANCE_TO_BURN → Err(GasKeyBalanceTooHigh)

4. contract.near submits:
   Receipt { actions: [DeleteAccount { beneficiary_id: "..." }] }
   → action_delete_account: aggregate balance > MAX_BALANCE_TO_BURN → Err(GasKeyBalanceTooHigh)

5. contract.near has no full-access key → cannot sign WithdrawFromGasKey transaction.
   No promise_batch_action_withdraw_from_gas_key host function exists.

Result: gas key K on contract.near is permanently undeletable.
        1 NEAR + 1 yocto is permanently locked in the gas key balance.
        contract.near is permanently undeletable.
```

The exact divergent value is `GasKeyInfo::MAX_BALANCE_TO_BURN = Balance::from_near(1)` at [7](#0-6) . Any gas key balance strictly greater than this value, reachable by any unprivileged `TransferToGasKey` sender, triggers the irrecoverable state for contract accounts.

pub(crate) fn action_transfer_to_gas_key(
    state_update: &mut TrieUpdate,
    result: &mut ActionResult,
    account_id: &AccountId,
    action: &TransferToGasKeyAction,
) -> Result<(), RuntimeError> {
    let Some(mut access_key) = get_access_key(state_update, account_id, &action.public_key)? else {
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };
    let Some(gas_key_info) = access_key.gas_key_info_mut() else {
        // Key exists but is not a gas key
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };

    gas_key_info.balance = gas_key_info.balance.checked_add(action.deposit).ok_or_else(|| {
        RuntimeError::StorageError(StorageError::StorageInconsistentState(
            "gas key balance integer overflow".to_string(),
        ))
    })?;
    set_access_key(state_update, account_id.clone(), action.public_key.clone(), &access_key);
    Ok(())
}
