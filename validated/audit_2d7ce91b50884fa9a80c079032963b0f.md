### Title
Consensus Proposal Validation Path Bypasses OS-Enforced Stateless Checks, Allowing OS-Invalid Transactions to Be Sequenced — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

The gateway enforces OS-required field constraints via `StatelessTransactionValidator::validate()` before admitting any transaction. The consensus validation path in `validate_proposal.rs` converts `ConsensusTransaction` objects directly to `InternalConsensusTransaction` via `convert_consensus_tx_to_internal_consensus_tx` without invoking any stateless validation. A malicious proposer can therefore include transactions that violate OS-enforced invariants — non-empty `paymaster_data`, non-empty `account_deployment_data`, or L2 DA modes — causing the Starknet OS to fail when generating a proof for the committed block.

### Finding Description

**Gateway path (stateless validation applied):**

```
add_tx()
  → stateless_tx_validator.validate()   // OS-enforced checks here
  → convert_rpc_tx_to_internal()
  → stateful validation
  → mempool
``` [1](#0-0) 

**Consensus path (stateless validation absent):**

```
validate_proposal()
  → handle_proposal_part()
      → convert_consensus_tx_to_internal_consensus_tx()   // NO stateless validation
      → send_txs_for_proposal()
      → batcher / blockifier
``` [2](#0-1) 

`convert_consensus_tx_to_internal_consensus_tx` delegates to `convert_rpc_tx_to_internal`, which computes the transaction hash and compiles declare classes but never calls `StatelessTransactionValidator::validate()`: [3](#0-2) [4](#0-3) 

The four OS-enforced checks that are bypassed are explicitly documented in the gateway as preventing OS failures:

| Check | Comment in source |
|---|---|
| `validate_empty_paymaster_data` | *"The Starknet OS enforces that the paymaster data is empty."* |
| `validate_empty_account_deployment_data` | *"The Starknet OS enforces that the deployer data is empty."* |
| `validate_nonce_data_availability_mode` | *"The Starknet OS enforces that the nonce data availability mode is L1."* |
| `validate_fee_data_availability_mode` | *"The Starknet OS enforces that the fee data availability mode is L1."* | [5](#0-4) [6](#0-5) 

The blockifier does **not** enforce these constraints; it passes `paymaster_data`, `account_deployment_data`, `nonce_data_availability_mode`, and `fee_data_availability_mode` straight through to `TransactionInfo` and exposes them to contracts via `get_execution_info`: [7](#0-6) 

Additionally, `validate_client_side_proving_allowed` and `validate_proof_facts_and_proof_consistency` are also skipped, meaning a proposer can inject invoke transactions that carry `proof_facts` without a matching `proof` (or vice versa), or carry proof data when the node's config has `allow_client_side_proving = false`: [8](#0-7) 

### Impact Explanation

Transactions with non-empty `paymaster_data`, `account_deployment_data`, or L2 DA modes are accepted by the blockifier and committed to the block. When the Starknet OS subsequently attempts to generate a validity proof for that block, it enforces these same constraints and fails. The block is committed on-chain but unprovable, breaking the ZK-rollup's proof pipeline. This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing"* and potentially **Critical**: *"Wrong state … or revert result from blockifier/syscall/execution logic for accepted input."*

### Likelihood Explanation

Any validator that wins a proposer slot — a normal, rotating role in Tendermint — can craft and broadcast a `ConsensusTransaction` with forbidden field values. No special privilege beyond being a validator is required. The structural gap exists in every proposal-validation round.

### Recommendation

Apply the same stateless checks in the consensus path. Either:

1. Call `StatelessTransactionValidator::validate()` inside `convert_consensus_tx_to_internal_consensus_tx` (or a wrapper called from `handle_proposal_part`) before forwarding transactions to the batcher.
2. Add a dedicated validation step in `validate_proposal.rs` that mirrors the gateway's `stateless_tx_validator.validate()` call for each `ConsensusTransaction::RpcTransaction` received in a `TransactionBatch`.

### Proof of Concept

1. A validator wins a proposer slot for height *H*.
2. The proposer constructs an `RpcInvokeTransactionV3` with `paymaster_data = vec![0x1]` (non-empty) and a valid signature over the correct transaction hash (the hash commits to `paymaster_data`, so the signature is valid).
3. The proposer wraps it in `ConsensusTransaction::RpcTransaction` and streams it as a `ProposalPart::Transactions` batch.
4. Each validating node calls `handle_proposal_part` → `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal`. No stateless validation is invoked; the transaction hash is recomputed correctly and the transaction is accepted.
5. `send_txs_for_proposal` forwards the transaction to the batcher; the blockifier executes it successfully (it does not check `paymaster_data`).
6. The proposal is finalized and the block is committed.
7. The Starknet OS processes the block to generate a proof. It enforces `paymaster_data.is_empty()` and aborts proof generation.
8. The block is on-chain but permanently unprovable.

    /// The Starknet OS enforces that the deployer data is empty. We add this validation here in the
    /// gateway to prevent transactions from failing the OS.
    fn validate_empty_account_deployment_data(
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let account_deployment_data = match tx {
            RpcTransaction::DeployAccount(_) => return Ok(()),
            RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => &tx.account_deployment_data,
            RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => &tx.account_deployment_data,
        };

        if account_deployment_data.is_empty() {
            Ok(())
        } else {
            Err(StatelessTransactionValidatorError::NonEmptyField {
                field_name: "account_deployment_data".to_string(),
            })
        }
    }

    /// The Starknet OS enforces that the paymaster data is empty. We add this validation here in
    /// the gateway to prevent transactions from failing the OS.
    fn validate_empty_paymaster_data(
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let paymaster_data = match tx {

                &tx.paymaster_data

            RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => &tx.paymaster_data,
            RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => &tx.paymaster_data,

            Ok(())
        } else {
            Err(StatelessTransactionValidatorError::NonEmptyField {
                field_name: "paymaster_data".to_string(),
            })
        }
    }

    fn validate_client_side_proving_allowed(
        &self,
        tx: &RpcInvokeTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if self.config.allow_client_side_proving {
            return Ok(());
        }

        // Reject V3 transactions with proofs when client-side proving is disabled.
        let RpcInvokeTransaction::V3(tx) = tx;
        let has_proof_data = !tx.proof_facts.is_empty() || !tx.proof.is_empty();
        if has_proof_data {
            return Err(StatelessTransactionValidatorError::ClientSideProvingNotAllowed);
        }

        Ok(())
    }

    fn validate_proof_facts_and_proof_consistency(
        &self,
        tx: &RpcInvokeTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let RpcInvokeTransaction::V3(tx) = tx;
        let has_proof_facts = !tx.proof_facts.is_empty();
        let has_proof = !tx.proof.is_empty();
        if has_proof_facts != has_proof {
            return Err(StatelessTransactionValidatorError::ProofFactsAndProofConsistency {
                has_proof_facts,
                has_proof,
            });
        }
        Ok(())
    }
