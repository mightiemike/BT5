### Title
Integer Overflow in `op_unknown` Cost Multiplier Allows Undercharged Execution - (File: src/more_ops.rs)

### Summary
The `op_unknown` function computes the cost of unknown opcodes by multiplying a base cost by `(cost_multiplier + 1)`. This multiplication is unchecked in Rust release builds and can wrap around to 0. An attacker who controls CLVM bytes can engineer the pre-multiplication cost to be an exact multiple of `2^32`, causing the multiplication to overflow to 0 and the function to return a cost of 0 — bypassing the execution cost limit entirely.

### Finding Description
In `op_unknown` (`src/more_ops.rs`, lines 258–266), after computing the base cost and checking it against `max_cost`, the code multiplies `cost` by `(cost_multiplier + 1)`:

```rust
assert!(cost > 0);

check_cost(cost, max_cost)?;
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
``` [1](#0-0) 

`cost_multiplier` is derived from `u32_from_u8(&op[0..op.len() - 1])`, which returns at most `u32::MAX = 4294967295`, so `cost_multiplier + 1` is at most `2^32 = 4294967296`. [2](#0-1) 

In Rust release builds, integer overflow wraps (two's complement). If `cost` is a multiple of `2^32` and `cost_multiplier + 1 = 2^32`, then:

- `cost * (cost_multiplier + 1) = 2^64 ≡ 0 (mod 2^64)`
- The post-multiplication guard `if cost > u32::MAX` evaluates to `0 > u32::MAX` → **false**
- The function returns `Reduction(0, nil)` — a cost of **zero**

The attacker controls both the opcode bytes (which set `cost_multiplier` and `cost_function`) and the arguments (which set the pre-multiplication cost). With `cost_function = 1` (ARITH-like):

```
cost = ARITH_BASE_COST + n * ARITH_COST_PER_ARG + byte_count * ARITH_COST_PER_BYTE
     = 99 + n * 320 + byte_count * 3
``` [3](#0-2) 

Setting `n = 13,421,771` and `byte_count = 159` yields:

```
99 + 13421771 × 320 + 159 × 3 = 99 + 4294966720 + 477 = 4294967296 = 2^32
```

This is within the allocator's pair limit of 62,500,000 pairs per argument list node, and the atom limit of 62,500,000. [4](#0-3) 

The `check_cost` call before the multiplication passes (cost ≤ max_cost), the `assert!(cost > 0)` passes (cost = 2^32 > 0), and then the multiplication silently wraps to 0 in release mode.

### Impact Explanation
An attacker submits a CLVM program with a crafted unknown opcode. In lenient mode (used by the Chia mempool and soft-fork guard evaluation), `op_unknown` is invoked and returns a cost of 0. This means:

1. **Cost limit bypass**: A program that should cost ~4.3 × 10^18 units is accepted as costing 0, allowing it to pass any `max_cost` check.
2. **Consensus divergence**: Nodes running lenient mode compute cost 0; nodes running strict mode reject the unknown op entirely. This divergence can cause mempool/block-validation disagreement.
3. **Block space exhaustion**: An attacker can fill blocks with programs that appear cheap but are computationally expensive.

### Likelihood Explanation
The attack is fully attacker-controlled: the opcode bytes (setting `cost_multiplier = u32::MAX` and `cost_function = 1`) and the argument list (setting the pre-multiplication cost to exactly `2^32`) are both embedded in the CLVM program bytes submitted by the attacker. The required ~13.4 million arguments are within the allocator's enforced limits. The attack path is reachable whenever `allow_unknown_ops()` returns true, which is the case for the Chia mempool and soft-fork validation contexts.

### Recommendation
Replace the unchecked multiplication with a checked variant that returns an error on overflow:

```rust
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

This mirrors the pattern already used elsewhere in the codebase (e.g., `saturating_add` in `serialized_length` and `tree_cache`). [5](#0-4) 

### Proof of Concept

Craft a CLVM program containing an unknown opcode where:
- The last byte has bits 7–6 = `01` → `cost_function = 1` (ARITH-like)
- The preceding bytes encode `0xFFFFFFFF` → `cost_multiplier = u32::MAX`
- The argument list contains 13,421,771 atoms, with 159 total bytes distributed among them

Pre-multiplication cost:
```
99 + 13421771 × 320 + 159 × 3 = 4294967296 = 2^32
```

Post-multiplication (release build, wrapping):
```
2^32 × 2^32 = 2^64 ≡ 0 (mod 2^64)
```

`op_unknown` returns `Reduction(0, nil)`. The program executes with a reported cost of 0, bypassing any `max_cost` enforcement.
