### Title
`SwapAllowlistExtension` gates the router address instead of the originating user, making per-user swap allowlisting impossible for router-mediated swaps — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the **router contract**, not the original user. The allowlist therefore checks the router's address, not the individual user's address. A pool admin who allowlists the router to enable router-mediated swaps for approved users simultaneously opens the pool to every user on the network.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← router address when called via MetricOmmSimpleRouter
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension:

```solidity
// ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks the allowlist keyed on `[msg.sender=pool][sender=router]`:

```solidity
// SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

The pool admin faces an inescapable dilemma:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use `MetricOmmSimpleRouter` at all |
| **Allowlist the router** | Every user on the network can bypass the allowlist via the router |

There is no configuration that simultaneously allows approved users to use the router and blocks unapproved users from doing the same.

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) all call `pool.swap()` directly with the router as `msg.sender`, so every public router entry point is affected:

```solidity
// MetricOmmSimpleRouter.sol L104-112
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
    .swap(
        i == last ? params.recipient : address(this),
        zeroForOne,
        amount,
        MetricOmmSwapPath.openLimit(zeroForOne),
        "",
        params.extensionDatas[i]
    );
```

---

### Impact Explanation

Any user can bypass a pool's `SwapAllowlistExtension` by routing through `MetricOmmSimpleRouter` whenever the pool admin has allowlisted the router address. Unauthorized traders gain full swap access to a pool that was explicitly configured to be restricted. They can trade against LP positions, move the pool's internal bin cursor, and extract value from LPs who deposited under the assumption that only approved counterparties would trade. This breaks the admin-boundary invariant: an unprivileged path (the public router) circumvents a guard the pool admin deliberately installed.

---

### Likelihood Explanation

The bypass is reachable whenever a pool admin allowlists the router — a natural and expected configuration step for any pool that wants approved users to benefit from the router's multicall, exact-output, and multi-hop features. The pool admin has no way to know that allowlisting the router also opens the pool to the entire public. The trigger requires no special privilege; any EOA or contract can call the public router functions.

---

### Recommendation

Pass the **original caller's address** through the router to the pool so the extension can gate on the true user identity. One approach is to encode the originating `msg.sender` inside `callbackData` or `extensionData` and have the extension verify it against a router-signed attestation. Alternatively, document clearly that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` and that pools using the allowlist must require direct pool interaction only.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, userA, true)` — only `userA` should be able to swap.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — necessary so `userA` can use the router.
4. Unauthorized `userB` calls `router.exactInputSingle(ExactInputSingleParams{pool: pool, ...})`.
5. Router calls `pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → hook passes.
8. `userB`'s swap executes successfully in the restricted pool, bypassing the allowlist entirely. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
