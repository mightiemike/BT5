### Title
Storage Deposit Permanently Locked After GC Frees Storage — (`File: contract/src/lib.rs`)

---

### Summary

The `submit_blocks` function accepts a NEAR token deposit to pay for on-chain storage. When the garbage-collection routine (`run_mainchain_gc`) later removes those block headers and frees the storage, the corresponding NEAR tokens are released back into the contract's own account balance but cannot be recovered by anyone — there is no withdrawal function in the contract.

---

### Finding Description

`submit_blocks` is marked `#[payable]` and collects a deposit from the caller to cover the cost of storing new block headers: [1](#0-0) 

The function correctly refunds the *excess* deposit to the caller: [2](#0-1) 

However, the portion that is *consumed* (`required_deposit`) stays in the contract to pay for NEAR storage staking. When `run_mainchain_gc` later deletes those same headers, the storage is freed and the staked NEAR is released back into the contract's own account balance: [3](#0-2) 

At that point the freed NEAR tokens sit in the contract account with no path out. A search across all contract source files finds **zero** functions named `withdraw`, `refund`, `reclaim`, or `storage_withdraw`. The full public API consists only of read-only getters, `submit_blocks`, `run_mainchain_gc`, `verify_transaction_inclusion*`, and the privileged `migrate`/`init` — none of which transfer the accumulated balance back to depositors. [4](#0-3) 

---

### Impact Explanation

Every time the relayer submits a batch of headers that later gets garbage-collected, the NEAR tokens paid for that storage are permanently locked in the contract. Over the operational lifetime of the light client — which is designed to run continuously and GC old blocks on every `submit_blocks` call — this accumulates into a growing, irrecoverable balance. The contract has no owner-withdrawal, no `storage_withdraw`, and no DAO-callable drain function.

---

### Likelihood Explanation

This is not theoretical. GC is triggered automatically inside every `submit_blocks` call: [5](#0-4) 

With a `gc_threshold` of 52,704 blocks (one year, as recommended in the constructor comment), the relayer continuously pays storage deposits for new headers while old ones are silently deleted. The locked amount grows with every GC cycle. Any production deployment with a live relayer will exhibit this behavior.

---

### Recommendation

Add a privileged (e.g., `Role::DAO`-gated) withdrawal function that transfers the contract's free balance to a designated treasury or back to the relayer. Alternatively, track per-depositor storage contributions and implement a `storage_withdraw` pattern consistent with the [NEAR Storage Management standard (NEP-145)](https://nomicon.io/Standards/StorageManagement).

---

### Proof of Concept

1. Relayer calls `submit_blocks` with 100 headers and attaches `D` yoctoNEAR (exactly `required_deposit`).
2. Contract stores 100 `ExtendedHeader` entries; `D` is consumed as storage staking.
3. On the next `submit_blocks` call, `run_mainchain_gc` removes the oldest headers (batch_size = num_of_headers), freeing the storage and releasing `D` back into the contract's account balance.
4. `D` is now in the contract's balance. No function exists to send it anywhere.
5. Repeat indefinitely — the locked balance grows monotonically with every GC cycle. [6](#0-5) [7](#0-6)

    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
