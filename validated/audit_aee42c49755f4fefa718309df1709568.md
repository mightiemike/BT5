Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks caller-controlled `owner` instead of `sender`, fully defeating the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual depositor, `msg.sender` of `addLiquidity`) and instead checks `owner` against the allowlist. Because `owner` is a free parameter supplied by the caller in `addLiquidity(address owner, ...)`, any unprivileged address can bypass the allowlist by setting `owner` to any known allowlisted address, causing unauthorized liquidity to enter the pool and forcing an unsolicited LP position onto the allowlisted address.

## Finding Description
In `MetricOmmPool.addLiquidity`, the pool calls `_beforeAddLiquidity(msg.sender, owner, ...)` where `msg.sender` is the actual depositor and `owner` is a free caller-supplied parameter: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly forwards both as `(sender, owner, ...)` to the extension: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as unnamed, discarding it entirely, and checks `owner` against the allowlist: [3](#0-2) 

Since `owner` is freely chosen by the caller, any address can pass the guard by supplying an allowlisted address as `owner`. The `SwapAllowlistExtension` does not share this flaw — it correctly checks `sender`: [4](#0-3) 

The allowlist state is publicly readable: [5](#0-4) 

## Impact Explanation
The pool admin's deposit access control is completely defeated by any unprivileged caller. Unauthorized liquidity enters the pool, violating the documented invariant ("Gates `addLiquidity` by depositor address, per pool"). Additionally, the allowlisted `owner` receives an unsolicited LP position funded by the attacker's tokens; since `removeLiquidity` enforces `msg.sender == owner`, the allowlisted address can drain those tokens, constituting a direct loss of the attacker's principal and a forced transfer. This is a broken core pool functionality / admin-boundary break by an unprivileged path, meeting Sherlock High severity thresholds.

## Likelihood Explanation
Exploitation requires only: (a) reading the public `allowedDepositor[pool][addr]` mapping to identify one allowlisted address, and (b) calling `pool.addLiquidity(owner=allowlisted_addr, ...)`. No special privileges, flash loans, or oracle manipulation are needed. The attack is repeatable and permissionless.

## Recommendation
Replace the `owner` check with a `sender` check:

```solidity
// FIXED
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  Pool P has DepositAllowlistExtension configured.
  allowedDepositor[P][alice] = true
  bob is NOT in the allowlist.

Attack:
  1. bob calls P.addLiquidity(owner=alice, salt=S, deltas=D, callbackData, extensionData)
  2. Pool calls _beforeAddLiquidity(sender=bob, owner=alice, ...)
  3. Extension receives (_, alice, ...) — bob (sender) is silently dropped.
  4. Check: allowedDepositor[P][alice] == true → no revert.
  5. Pool credits LP position to alice at salt S.
  6. Pool calls bob.metricOmmSwapCallback(...) — bob's tokens are pulled into the pool.
  7. alice holds the LP position worth bob's deposited tokens.
  8. alice calls P.removeLiquidity(owner=alice, salt=S, ...) and recovers the tokens.

Result:
  - bob's deposit bypasses the allowlist entirely.
  - alice receives an unsolicited LP position and can drain it.
  - The pool holds unauthorized liquidity, defeating the pool admin's access control.
```
