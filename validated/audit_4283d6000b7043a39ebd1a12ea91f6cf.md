Audit Report

## Title
`ProtectedPriceProviderL2` Missing Sequencer Uptime Check Violates Stated Invariant, Enabling Bad-Price Swaps After Sequencer Restart - (File: `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary

`ProtectedPriceProviderL2._computeBidAsk()` performs staleness, validity, and price-guard checks but contains no sequencer uptime feed query. The protocol README explicitly states as a hard invariant that "swaps revert on stale price (maxTimeDelta/maxRefStaleness), excessive Chainlink deviation, or (L2) sequencer down." The registry ABI confirms a `ChainlinkVerifierL2` contract with `sequencerUptimeFeed` and `GRACE_PERIOD` was designed for this purpose but never integrated. When an L2 sequencer restarts after downtime, the last pre-downtime oracle price remains in storage and passes all existing guards, allowing a trader to execute swaps at a stale price while LPs cannot rebalance.

## Finding Description

`ProtectedPriceProviderL2._computeBidAsk()` executes four guards in sequence:

1. `_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)` — rejects prices older than `MAX_TIME_DELTA` or more than `FUTURE_TOLERANCE` seconds in the future.
2. `price == 0 || spread >= ORACLE_BPS` — rejects zero price or stall marker.
3. `offchainOracle.priceGuard(offchainFeedId)` bounds check.
4. Confidence/marginStep shaping.

None of these steps queries a sequencer uptime feed. The `FUTURE_TOLERANCE` field is explicitly documented as handling clock skew ("L2 sequencer timestamp can lag behind oracle publication time"), not sequencer downtime. When the sequencer is down, off-chain keepers cannot push price updates. When it restarts, the last pre-downtime price is still in oracle storage. If that price is within `MAX_TIME_DELTA` of `block.timestamp`, it passes the staleness check and all subsequent guards, producing a valid bid/ask from a stale price.

The registry ABI at `smart-contracts-poc/contract-registry/versions/registry.json` (lines 5685–5791) confirms `ChainlinkVerifierL2` with `sequencerUptimeFeed()` and `GRACE_PERIOD()` was designed but no source file exists and it is not referenced anywhere in `ProtectedPriceProviderL2`. `PriceProviderL2` is structurally identical — it also has no sequencer check — confirming the check was never implemented in either L2 provider.

The protocol is deployed on Base (an L2 with a sequencer), and the registry shows `PriceProviderFactoryL2` deployed on Arbitrum, Base, Optimism, and other L2s — all chains where sequencer downtime is a real operational risk.

## Impact Explanation

When the sequencer restarts after downtime, a trader can immediately call `pool.swap()`. `ProtectedPriceProviderL2._computeBidAsk()` reads the last pre-downtime oracle price, which passes the staleness check (age ≤ `MAX_TIME_DELTA`), and returns a bid/ask anchored to the stale price. If the true market price moved significantly during the outage (e.g., a sharp ETH drop), the trader buys at the old high price from LPs who cannot rebalance, extracting value directly from LP principal. This is bad-price execution: the pool settles a trade at a price the oracle/bin curve should have rejected per the stated invariant. The loss is bounded by the price movement during downtime and the pool's liquidity depth, but can easily exceed Sherlock Medium thresholds (>0.01% and >$10) for any meaningful price move.

## Likelihood Explanation

L2 sequencer downtime has occurred historically on Arbitrum, Optimism, and Base. The attack requires: (1) sequencer downtime, (2) meaningful price movement during the outage, (3) sequencer restart, (4) a swap executed before oracle keepers push a fresh price. Steps 3 and 4 are trivially achievable by any public trader monitoring the Chainlink sequencer uptime feed. The window is bounded by `MAX_TIME_DELTA` (up to 7 days per the constructor bound), but even a window of minutes is sufficient for a prepared attacker. Likelihood is **medium** — requires an external sequencer event, but the attack is fully permissionless once it occurs.

## Recommendation

Integrate a sequencer uptime check into `ProtectedPriceProviderL2._computeBidAsk()` using the already-designed `ChainlinkVerifierL2` pattern. Before accepting any oracle price, call `sequencerUptimeFeed.latestRoundData()`, verify `answer == 0` (sequencer is up), and enforce that `block.timestamp - startedAt >= GRACE_PERIOD` (e.g., 3600 seconds) to ensure oracle prices have had time to update after a restart. Revert (fail closed) if either condition fails:

```solidity
// In ProtectedPriceProviderL2._computeBidAsk(), before staleness check:
(, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0 || block.timestamp - startedAt < GRACE_PERIOD) {
    return (0, type(uint128).max); // fail closed
}
```

## Proof of Concept

1. Deploy `ProtectedPriceProviderL2` on Base/Arbitrum with `MAX_TIME_DELTA = 3600`.
2. Oracle stores price `P0 = 3000e8` (ETH/USD) at `T0`.
3. L2 sequencer goes down at `T0 + 1`. True market price drops to `P1 = 2000e8`.
4. Sequencer restarts at `T0 + 1800` (30 minutes later). Oracle storage still holds `P0` with `refTime = T0`.
5. Attacker calls `pool.swap()` immediately. `_computeBidAsk(P0, spread, T0)` runs:
   - `_isStale(T0, T0+1800, 3600, futureTol)` → `1800 ≤ 3600` → **not stale, passes**.
   - No sequencer check → **no revert**.
   - Pool quotes bid/ask around `P0 = 3000e8`.
6. Attacker buys ETH from the pool at ~3000 when true price is ~2000, extracting ~$1000 per ETH from LP principal.
7. Oracle keepers push `P1` shortly after; the window closes, but the loss is already realized.

Foundry fork test: fork Base/Arbitrum at a block just after a historical sequencer restart, mock the oracle to return a pre-downtime price within `MAX_TIME_DELTA`, and assert that `getBidAndAskPrice()` succeeds (returns non-stale bid/ask) — demonstrating the missing guard.
