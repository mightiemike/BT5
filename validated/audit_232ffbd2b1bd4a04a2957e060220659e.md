### Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end-user, allowing any unprivileged user to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so `sender = router`. If the pool admin allowlists the router (a necessary step to enable any router-mediated swap for their curated pool), every unprivileged user can bypass the allowlist by routing through the public router.

---

### Finding Description

**Call path:**

```
User → MetricOmmSimpleRouter.exactInput*()
     → MetricOmmPool.swap(recipient, ..., extensionData)
          msg.sender = router
     → _beforeSwap(msg.sender=router, recipient, ...)
     → ExtensionCalling._callExtensionsInOrder(BEFORE_SWAP_ORDER, ...)
     → SwapAllowlistExtension.beforeSwap(sender=router, ...)
```

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether `sender` (the router) is allowlisted on the pool: [3](#0-2) 

The allowlist mapping is keyed `allowedSwapper[pool][sender]`. When the call arrives through the router, `sender` is the router's address, not the actual end-user. The extension has no visibility into who initiated the router call.

**The inescapable dilemma for the pool admin:**

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Router-mediated swaps fail for *everyone*, including legitimately allowlisted users |
| **Allowlist the router** | *Every* user can bypass the allowlist by routing through the public router |

There is no configuration that simultaneously allows router-mediated swaps for specific users while blocking others. The guard is structurally misbound to the wrong actor.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, protocol-owned addresses, or whitelisted market makers) loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The attacker pays no special cost: the router is a public, permissionless contract. The allowlist extension — the sole access-control boundary on the swap path — fails open.

Consequences include:
- Unprivileged users executing swaps on pools explicitly designed to exclude them.
- Front-running or MEV extraction on pools whose liquidity was provisioned under the assumption of a restricted counterparty set.
- Broken core pool functionality: the pool's configured protection hook does not enforce the invariant it was deployed to enforce.

This matches the allowed impact gate: **admin-boundary break — factory/oracle role checks are bypassed by an unprivileged path** and **broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows**.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless periphery contract reachable by any EOA or contract.
- A pool admin who deploys a `SwapAllowlistExtension` and wants router-mediated swaps to work for their allowlisted users has no choice but to allowlist the router — the bypass is a direct consequence of normal, expected admin configuration.
- No special privileges, flash loans, or oracle manipulation are required. A single `exactInputSingle` call through the router suffices.

---

### Recommendation

The `SwapAllowlistExtension` must gate the **economically relevant actor** — the address that initiated the trade — not the immediate pool caller. Two sound approaches:

1. **Check `recipient` as a proxy for the end-user** (only valid if the router always sets `recipient = msg.sender`, which should be enforced by convention or interface).
2. **Require the end-user's address in `extensionData`** and have the router sign or forward it; the extension verifies the embedded address against the allowlist.
3. **Reject router-mediated calls entirely** by checking `sender != trustedRouter` and requiring direct pool calls for allowlisted pools.

The root fix is to ensure the identity checked by the hook is the same identity the pool admin intended to gate, regardless of which supported public entrypoint reaches the pool.

---

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted.
// allowedSwapper[pool][allowedUser] = true
// allowedSwapper[pool][router]      = true  ← admin adds this to enable router swaps

// Attack: attacker (not allowlisted) bypasses the guard via the router
function testBypassAllowlist() public {
    // attacker is NOT in allowedSwapper[pool]
    vm.startPrank(attacker);

    // Direct call → reverts: allowedSwapper[pool][attacker] == false
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    pool.swap(attacker, true, 1000, 0, "", "");

    // Router call → succeeds: allowedSwapper[pool][router] == true
    // sender seen by extension = router, not attacker
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            tokenIn: token0,
            tokenOut: token1,
            pool: address(pool),
            recipient: attacker,
            amountIn: 1000,
            amountOutMinimum: 0,
            priceLimitX64: 0,
            callbackData: "",
            extensionData: ""
        })
    );
    // swap executes; allowlist guard was bypassed
    vm.stopPrank();
}
```

The extension receives `sender = address(router)`, which is allowlisted, so the guard passes. The actual attacker address is never checked. [3](#0-2) [4](#0-3) [2](#0-1)
