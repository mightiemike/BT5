### Title
Third-Party Relayer Fee Charged in Withdrawal Product Token Blocks Fast Withdrawals — (`core/contracts/BaseWithdrawPool.sol`)

### Summary
In `BaseWithdrawPool.submitFastWithdrawal`, when a third-party relayer processes a fast withdrawal (i.e., `sendTo != msg.sender`), the protocol pulls the fee from the relayer in `token` — the withdrawal product token. This is the wrong asset for relayer compensation. If the relayer does not hold that specific product token, the `safeTransferFrom` call reverts and the fast withdrawal cannot be processed by any relayer. For `WithdrawCollateralV2` transactions with a custom `sendTo`, even the user themselves cannot self-serve without holding the withdrawal token externally, locking their funds in the fast-withdrawal queue.

### Finding Description

`submitFastWithdrawal` in `BaseWithdrawPool` handles two cases based on whether the caller is the intended recipient: [1](#0-0) 

```solidity
if (sendTo == msg.sender) {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);
} else {
    safeTransferFrom(token, msg.sender, uint128(fee));
}
```

When `sendTo == msg.sender` (user self-serves), the fee is deducted from the withdrawal amount — no external token holding required. When `sendTo != msg.sender` (third-party relayer, or user withdrawing to a custom address), the protocol calls `safeTransferFrom(token, msg.sender, uint128(fee))`, pulling `fee` of `token` (the withdrawal product token) from the caller's external wallet.

`token` is resolved from `productId` via `getToken`: [2](#0-1) 

This means the relayer must hold the specific withdrawal product token (e.g., a non-USDC spot asset) in their external wallet. Relayers typically hold stablecoins or native gas tokens, not every product token listed on the exchange.

For `WithdrawCollateralV2` with a non-zero `sendTo`: [3](#0-2) 

`sendTo` is set to the custom recipient address. If the user calls `submitFastWithdrawal` themselves, `msg.sender` is the user's wallet but `sendTo` is the custom recipient — so `sendTo != msg.sender` — and the user must also pay the fee from their external wallet in the withdrawal product token. A user who has deposited all their tokens into the protocol and holds none externally cannot pay this fee, and neither can a relayer who doesn't hold that token. The fast withdrawal is permanently stuck.

### Impact Explanation

For `WithdrawCollateralV2` with a custom `sendTo`: both the user and any relayer are blocked from processing the fast withdrawal if neither holds the withdrawal product token externally. The user's funds remain locked in the fast-withdrawal queue indefinitely. The slow-mode path via `submitWithdrawal` requires the clearinghouse (sequencer) to call it: [4](#0-3) 

If the sequencer is unavailable or censoring, there is no permissionless recovery path. For standard `WithdrawCollateral` (V1), the user can always self-serve (since `sendTo` resolves to their own address), but third-party relayers are still blocked for any product token they don't hold.

### Likelihood Explanation

Medium. The `WithdrawCollateralV2` path with a custom `sendTo` is a supported, documented transaction type. Any user who specifies a custom recipient and has no external balance of the withdrawal token triggers this condition. Relayers operating across multiple product tokens are unlikely to hold all of them, making the third-party relayer path unreliable for non-quote tokens.

### Recommendation

When `sendTo != msg.sender`, the fee should be deducted from `transferAmount` (same as the self-serve path) rather than pulled from the caller in the withdrawal product token. Alternatively, the fee should be charged in the quote token (product ID 0), which relayers are far more likely to hold. The corrected branch:

```solidity
} else {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);
    // transfer fee to pool separately if needed
}
```

This mirrors the fix applied in the referenced H-04 patch: use the asset that is guaranteed to be present in the relevant context rather than an asset that may not exist in the caller's wallet.

### Proof of Concept

1. User signs a `WithdrawCollateralV2` transaction with `sendTo = 0xRecipient` (a hardware wallet address) and `productId = 5` (some non-USDC spot token).
2. User has deposited all their product-5 tokens into the protocol; their external wallet holds zero product-5 tokens.
3. User calls `submitFastWithdrawal(idx, transaction, signatures)`.
4. `resolveFastWithdrawal` returns `sendTo = 0xRecipient`, `productId = 5`.
5. `token = getToken(5)` — the product-5 ERC-20.
6. `sendTo (0xRecipient) != msg.sender (user)` → branch: `safeTransferFrom(token, msg.sender, fee)`.
7. User holds 0 of product-5 externally → `safeTransferFrom` reverts.
8. No relayer holds product-5 either → every call to `submitFastWithdrawal` reverts.
9. `submitWithdrawal` requires `msg.sender == clearinghouse` — not callable by the user.
10. Fast withdrawal is permanently stuck; user funds are locked until the sequencer processes the slow-mode path. [5](#0-4)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
