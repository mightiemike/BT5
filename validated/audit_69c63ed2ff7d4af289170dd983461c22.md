### Title
`ResourceBoundsMapping`→`ValidResourceBounds` Conversion in RPC Fee Estimation Produces Wrong Hash Preimage Length for V3 Transactions with Zero L2/Data-Gas Bounds — (`crates/apollo_rpc/src/v0_8/transaction.rs`)

### Summary

The `starknet_estimateFee` and `starknet_simulateTransactions` RPC endpoints convert incoming `BroadcastedTransaction` objects through `ResourceBoundsMapping → ValidResourceBounds`. When `l2_gas` and `l1_data_gas` are both zero, this conversion silently produces `ValidResourceBounds::L1Gas` (a 2-element hash preimage). The gateway's `add_invoke_transaction` path, however, always produces `ValidResourceBounds::AllResources` (a 3-element hash preimage) for the same numeric values. Because `get_tip_resource_bounds_hash` hashes a different number of field elements depending on the variant, the transaction hash computed internally by the fee estimator diverges from the hash the user signed. When `validate=true` (the default), the account's `__validate__` entrypoint is called with the wrong hash, causing it to fail signature verification and returning an authoritative-looking wrong result.

### Finding Description

**Divergent conversion paths for the same wire bytes:**

The gateway submission path (`add_invoke_transaction`) receives a `TypedInvokeTransaction` wrapping `RpcInvokeTransactionV3`, which carries `AllResourceBounds` directly. The `From` impl unconditionally wraps it as `AllResources`:

```rust
// crates/starknet_api/src/rpc_transaction.rs:568-583
impl From<RpcInvokeTransactionV3> for InvokeTransactionV3 {
    fn from(tx: RpcInvokeTransactionV3) -> Self {
        Self {
            resource_bounds: ValidResourceBounds::AllResources(tx.resource_bounds),
            ...
        }
    }
}
```

The fee estimation / simulation path receives a `BroadcastedTransaction` carrying `ResourceBoundsMapping`. Its conversion is conditional:

```rust
// crates/apollo_rpc/src/v0_8/transaction.rs:188-199
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← 2-element hash preimage
        } else {
            Self::AllResources(AllResourceBounds { ... })  // ← 3-element hash preimage
        }
    }
}
```

**Hash preimage length divergence:**

`get_tip_resource_bounds_hash` hashes a different number of field elements depending on the variant:

```rust
// crates/starknet_api/src/transaction_hash.rs:188-211
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,   // zero for L1Gas variant
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],             // stops here: 2 elements
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

For a V3 transaction with `l2_gas=0, l1_data_gas=0`:
- Gateway hash: `Poseidon(tip, L1_GAS_packed, L2_GAS_zero_packed, L1_DATA_GAS_zero_packed)` — 3 elements
- Fee estimator hash: `Poseidon(tip, L1_GAS_packed, L2_GAS_zero_packed)` — 2 elements

These are cryptographically distinct values. The hash is then embedded in the full transaction hash via `get_invoke_transaction_v3_hash` (line 375-376 of `transaction_hash.rs`).

**Hash is computed internally during fee estimation:**

`estimate_fee` calls `execute_transactions` with `tx_hashes=None`, which triggers `calc_tx_hashes`:

```rust
// crates/apollo_rpc_execution/src/lib.rs:469-476
fn calc_tx_hash(self, chain_id: &ChainId) -> ExecutionResult<(Self, TransactionHash)> {
    match self.apply_on_transaction(|tx, only_query| {
        get_transaction_hash(tx, chain_id, &TransactionOptions { only_query })
    }) { ... }
}
```

The internally computed hash (2-element variant) is passed to the blockifier as the authoritative `tx_hash`. When `validate=true`, the account's `__validate__` is called with this wrong hash, and the user's signature (computed over the 3-element hash) fails verification.

### Impact Explanation

`starknet_estimateFee` and `starknet_simulateTransactions` with `validate=true` (the default, no `SKIP_VALIDATE` flag) return a validation failure for any V3 transaction where `l2_gas=0` and `l1_data_gas=0` but the user signed using the `AllResources` (3-element) hash domain — which is the correct domain for any post-0.13.3 SDK. The RPC endpoint returns an authoritative-looking `TRANSACTION_EXECUTION_ERROR` (signature mismatch) instead of a fee estimate. This matches the allowed impact: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

Any user of a post-0.13.3 SDK who submits a V3 transaction for fee estimation with only L1 gas bounds (a common pattern for simple transfers) will trigger this. The `BroadcastedTransaction` wire format uses `ResourceBoundsMapping` for both old and new V3 transactions, so the conversion silently downgrades the variant. The gateway path is unaffected, meaning the same transaction that fails fee estimation will succeed when submitted directly.

### Recommendation

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in `crates/apollo_rpc/src/v0_8/transaction.rs` should not silently downgrade to `L1Gas` based on zero values. The fee estimation and simulation paths should use the same `AllResourceBounds`-preserving conversion that the gateway uses, or the `BroadcastedTransaction` type should carry `AllResourceBounds` directly (matching `RpcInvokeTransactionV3`) so that the hash preimage length is always determined by the transaction type, not by the numeric values of the bounds.

### Proof of Concept

1. User creates a V3 invoke transaction with `l1_gas=X, l2_gas=0, l1_data_gas=0` using a post-0.13.3 SDK.
2. SDK signs with hash `H_3 = Poseidon(INVOKE, 3, sender, Poseidon(tip, L1_GAS_packed, L2_GAS_zero_packed, L1_DATA_GAS_zero_packed), ...)`.
3. User calls `starknet_estimateFee([tx], [], "latest")` (no `SKIP_VALIDATE`).
4. The RPC handler converts `ResourceBoundsMapping{l1_gas=X, l2_gas=0, l1_data_gas=0}` → `ValidResourceBounds::L1Gas(X)`.
5. `calc_tx_hashes` computes `H_2 = Poseidon(INVOKE, 3, sender, Poseidon(tip, L1_GAS_packed, L2_GAS_zero_packed), ...)`.
6. `H_2 ≠ H_3`. The blockifier calls `__validate__` with `H_2`; the account verifies the signature against `H_2` but the signature was made over `H_3` → signature check fails.
7. `starknet_estimateFee` returns `TRANSACTION_EXECUTION_ERROR` instead of a fee estimate.
8. The same transaction submitted via `add_invoke_transaction` succeeds, because the gateway path always uses `AllResources` and computes `H_3`.

```rust
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,

    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,

    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

```rust
pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
```
