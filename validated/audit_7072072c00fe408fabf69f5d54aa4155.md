### Title
Missing Lower-Bound Validation on `gc_threshold` Permanently Breaks SPV Proof Verification - (File: `contract/src/lib.rs`)

### Summary
The `init` function accepts `gc_threshold` from `InitArgs` without any lower-bound check. A deployer who fat-fingers this value (e.g., sets it to `0` or `1`) permanently disables the contract's core SPV proof verification capability, because both the GC logic and the confirmation guard in `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` are gated on this single unchecked value.

### Finding Description
`InitArgs.gc_threshold` is stored directly into contract state with no validation: [1](#0-0) 

`verify_transaction_inclusion` enforces `args.confirmations <= self.gc_threshold` as a hard gate: [2](#0-1) 

With `gc_threshold = 0`, every call with `confirmations >= 1` is permanently rejected. With `gc_threshold = 1`, only zero- or one-confirmation proofs are accepted, which is below any meaningful Bitcoin security threshold (typically 6).

Additionally, `run_mainchain_gc` is called inside `submit_blocks` with `batch_size = num_of_headers`: [3](#0-2) 

When `gc_threshold = 0`, the GC condition `amount_of_headers_we_store > self.gc_threshold` is always true after genesis. The GC removes `amount_of_headers_we_store - gc_threshold` = all stored blocks, then attempts to look up the block at `end_removal_height`: [4](#0-3) 

If `end_removal_height` equals or exceeds the tip height, this lookup panics with `ERR_KEY_NOT_EXIST`, bricking `submit_blocks` permanently.

The `gc_threshold` field in `InitArgs` is a plain `u64` with no documented minimum: [5](#0-4) 

### Impact Explanation
**High.** A `gc_threshold` of `0` causes two simultaneous permanent failures:
1. Every `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` call with any non-zero confirmation count is rejected, making SPV proof verification completely non-functional — the contract's sole purpose.
2. `submit_blocks` panics once the GC tries to set `mainchain_initial_blockhash` to a block that no longer exists, halting all future header ingestion.

Even a small non-zero value (e.g., `1`–`5`) silently degrades security: callers are forced to accept fewer confirmations than Bitcoin's standard 6-confirmation threshold, making double-spend attacks feasible against downstream consumers of `verify_transaction_inclusion`.

### Likelihood Explanation
**Low.** The `init` function is `#[private]`, so only the deployer can trigger this. However, `gc_threshold` is a numeric field with no documented minimum in the struct or the `init` docstring (the comment only recommends `52704`). A fat-finger — e.g., entering `0` instead of `52704`, or omitting the field — produces a silently accepted but permanently broken deployment. The original M-03 report accepts the same likelihood profile for the same reason.

### Recommendation
Add a lower-bound `require!` check inside `init` before storing `gc_threshold`. A sensible minimum is at least `MEDIAN_TIME_SPAN + 1` (12 for Bitcoin/Litecoin/Dogecoin) to guarantee that MTP computation and at least one confirmation are always possible. A practical production minimum matching the documented recommendation would be `2016` (one difficulty epoch):

```rust
require!(
    args.gc_threshold >= btc_types::network::MEDIAN_TIME_SPAN as u64 + 1,
    "gc_threshold must be at least MEDIAN_TIME_SPAN + 1"
);
```

### Proof of Concept
1. Deploy the contract with `InitArgs { gc_threshold: 0, ... }` — accepted without error.
2. Submit any block via `submit_blocks([header])`. Inside `run_mainchain_gc(1)`:
   - `amount_of_headers_we_store = 2 > 0`, so GC fires.
   - Removes the genesis block; tries `mainchain_height_to_header.get(&1)` — if only one block was submitted, this panics with `ERR_KEY_NOT_EXIST`.
3. Alternatively, call `verify_transaction_inclusion` with `confirmations: 1` — immediately panics with `"The required number of confirmations exceeds the number of blocks stored in memory"` because `1 > 0 = gc_threshold`.
4. The contract is now permanently non-functional for its core SPV purpose with no upgrade path short of a full state migration.
