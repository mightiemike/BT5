Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the actual user, allowing any user to bypass the per-user swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. Allowlisting the router — the only way to let allowlisted users access the router — opens the pool to every user on-chain, rendering the per-user swap allowlist unenforceable.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` (the direct caller) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` in `ExtensionCalling` encodes that `sender` and dispatches it to each configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` — making the **router** the `msg.sender` to the pool: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`, all of which call `pool.swap` from the router contract. [5](#0-4) 

The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The pool admin faces an impossible configuration: allowlisting only specific users blocks the router for everyone; allowlisting the router opens the pool to all users.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the `owner` argument (the economic actor), not `sender` (the technical caller): [6](#0-5) 

The `recipient` argument is already passed to `beforeSwap` (second parameter, currently ignored) and could serve the analogous role.

## Impact Explanation

Any pool deploying `SwapAllowlistExtension` to restrict swaps to a curated set (KYC-gated, institutional-only, MEV-protected) is rendered unenforceable once the pool admin allowlists the router. Non-allowlisted users can swap freely through the router, extracting value from LPs who expected a restricted counterparty set. This constitutes a direct loss of LP principal through trades the pool was explicitly configured to block — a broken core pool functionality causing loss of funds.

## Likelihood Explanation

The trigger is a pool admin allowlisting the router, which is the natural and expected action when allowlisted users need to use the standard periphery. The misconfiguration is invisible: `setAllowedToSwap(pool, router, true)` gives no indication it opens the pool to all users. Any pool combining `SwapAllowlistExtension` with the public router is vulnerable once the router is added to the allowlist.

## Recommendation

Gate the actual economic actor, not the technical caller. The simplest fix mirrors the deposit allowlist pattern: check `recipient` (the second argument to `beforeSwap`, currently ignored) instead of `sender`. Since `recipient` is the address receiving output tokens — analogous to `owner` in the deposit path — this correctly identifies the economic actor regardless of whether the user calls the pool directly or through the router.

Alternatively, have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData`, and have `SwapAllowlistExtension` decode and check that address when the caller is a known trusted router.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` as a `beforeSwap` hook.
2. Pool admin allowlists Alice: `setAllowedToSwap(pool, alice, true)`.
3. Pool admin allowlists the router so Alice can use it: `setAllowedToSwap(pool, router, true)`.
4. Non-allowlisted Bob calls `router.exactInputSingle({pool: pool, recipient: bob, ...})`.
5. The router calls `pool.swap(bob, ...)` with `msg.sender = router`.
6. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][router]` → `true` → swap succeeds.
7. Bob, who was never allowlisted, successfully swaps in a pool intended to be restricted.

A Foundry integration test can confirm this by deploying the extension, configuring the allowlist with only Alice and the router, then asserting that a `pool.swap` call from Bob via the router succeeds while a direct call from Bob reverts.
