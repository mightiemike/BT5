Audit Report

## Title
`PriceVelocityGuardExtension` Uses `block.number` Instead of `block.timestamp` for Velocity Window, Making the Guard Inaccurate on Chains with Non-Constant Block Time - (File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol)

## Summary
`PriceVelocityGuardExtension` tracks elapsed time via `block.number` and computes the allowed price-change window as `maxChangePerBlockE18² × (1 + blockDiff)`. On Berachain — a confirmed production deployment target with non-constant block time — the effective per-second rate limit drifts with block cadence, making the guard either too permissive or too restrictive relative to the admin's intended real-world rate. In the permissive direction, a price move that exceeds the intended rate passes the `beforeSwap` hook, enabling bad-price execution against LP principal.

## Finding Description
In `beforeSwap`, the contract reads `prevBlock = s.lastUpdateBlock` (stored as `uint64(block.number)`), computes `blockDiff = block.number - prevBlock`, and evaluates:

```solidity
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
if (actualSq > allowedSq) revert PriceVelocityExceeded(actualSq, allowedSq);
```

`blockDiff` counts blocks, not seconds. When Berachain's block time increases (blocks produced more slowly), the same real-world elapsed time maps to a smaller `blockDiff`, so `allowedSq` is smaller than the admin intended — the guard becomes more restrictive. When block time decreases (blocks produced faster), `blockDiff` grows for the same real-world elapsed time, `allowedSq` grows, and the guard becomes more permissive than intended. The admin sets `maxChangePerBlockE18` calibrated to an assumed block cadence; any sustained deviation silently miscalibrates the guard in either direction. No attacker action is required for the permissive direction — the miscalibration is passive and continuous. The `setLastMidPrice` admin setter also writes `block.number`, so even manual resets do not correct the underlying measurement unit.

## Impact Explanation
In the permissive direction, the velocity guard — the last hook-level defense against rapid oracle price movement reaching a swap — passes a price delta that exceeds the admin's intended per-second rate. The `beforeSwap` hook returns without reverting, the pool executes the swap at the manipulated bid/ask, and LPs absorb the difference between the manipulated price and the true market price. This constitutes bad-price execution (an unclamped oracle-derived quote reaching a pool swap) and direct loss of LP principal, both of which are listed allowed impacts. In the restrictive direction, legitimate swaps revert with `PriceVelocityExceeded` during periods of faster block production, breaking core swap functionality.

## Likelihood Explanation
Berachain is a confirmed production deployment target (present in `registry.json` and all deployment scripts). Berachain's block time is documented as variable (congestion-dependent). No privileged actor or special attacker capability is required for the permissive miscalibration — it activates automatically whenever block time deviates from the admin's assumed cadence. The condition is persistent and worsens monotonically as block time drifts. Any oracle price movement that is legitimate in real-world seconds but exceeds the per-block cap (or vice versa) triggers the incorrect outcome.

## Recommendation
Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension`. Rename `maxChangePerBlockE18` to `maxChangePerSecondE18` and `lastUpdateBlock` to `lastUpdateTimestamp`. Update the velocity formula to:

```solidity
uint256 timeDiff = block.timestamp - prevTimestamp;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + timeDiff);
```

This makes the guard invariant to block-time fluctuations and consistent with the oracle layer, which already uses unix timestamps for staleness and future-drift checks.

## Proof of Concept
1. Deploy a pool on Berachain with `PriceVelocityGuardExtension` configured: `maxChangePerBlockE18 = 0.005e18` (0.5% per block, calibrated at 2 s/block → 0.25%/s intended).
2. Berachain block time speeds up to 0.5 s/block (4 blocks per 2 seconds).
3. After 2 real-world seconds, `blockDiff = 4`, `allowedSq = (0.005e18)² × 5` → allows a 1.118% price move.
4. The admin intended only 0.25%/s × 2 s = 0.5% over those 2 seconds.
5. Submit a swap when the oracle mid-price has moved 1.0% in 2 seconds. `actualSq = (0.01e18)² = 1e32`; `allowedSq = (0.005e18)² × 5 = 1.25e32`. Guard passes (`actualSq < allowedSq`), swap executes at the manipulated price, LPs absorb the 0.5% excess price movement.
