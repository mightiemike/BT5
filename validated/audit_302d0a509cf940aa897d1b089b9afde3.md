### Title
Quadratic Execution Complexity in `op_add`/`op_subtract` Due to Undercharged Accumulator Growth — (File: `src/more_ops.rs`)

---

### Summary

The `op_add` and `op_subtract` functions charge cost proportional only to each incoming argument's byte size (O(N·B) total), but the actual bignum addition work grows quadratically (O(N²·B)) as the running accumulator expands with every iteration. An attacker can craft a CLVM program with many large-integer arguments that consumes far more CPU time than the cost model permits, systematically undercharging execution relative to actual validator work.

---

### Finding Description

In the slow path of `op_add` (lines 452–481 of `src/more_ops.rs`), each argument is charged:

```rust
cost += ARITH_COST_PER_ARG;          // 320
cost += ARITH_COST_PER_BYTE * (buf.len() as Cost);  // 3 per byte
``` [1](#0-0) 

This charges only for the size of the *incoming* argument. However, the actual bignum operation `acc[rng.random_range(0..2)] += val` must process the entire current accumulator, which grows by approximately B bytes per step. After i steps, the accumulator is ~i·B/2 bytes, so step i costs O(i·B/2) in actual CPU work. Summing over N steps gives **O(N²·B/4) total work** while the charged cost is only **O(N·B)**. [2](#0-1)
