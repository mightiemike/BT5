### Title
Stale `digestToSubaccount` After Close Enables Zero-Margin Isolated Position Replay — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`_tryCloseIsolatedSubaccount` clears the live subaccount registries (`parentSubaccounts`, `isolatedSubaccounts`, `isolatedSubaccountsMask`) but never clears `digestToSubaccount[digest]`. The early-return guard in `createIsolatedSubaccount` treats a non-zero `digestToSubaccount[digest]` as "already set up" and returns the stale isolated subaccount address without re-transferring margin. `matchOrders` then routes the same order to that zero-balance subaccount, opening a position with no collateral.

---

### Finding Description

**Root cause — missing cleanup in `_tryCloseIsolatedSubaccount`:** [1](#0-0) 

Lines 202-204 clear `isolatedSubaccountsMask`, `isolatedSubaccounts[parent][id]`, and `parentSubaccounts[subaccount]`. Neither `digestToSubaccount[digest]` nor `digestToMargin[digest]` is touched anywhere in this function. There is no grep match for `digestToSubaccount[digest] = bytes32(0)` anywhere in the codebase.

**Early-return path in `createIsolatedSubaccount` skips margin transfer:** [2](#0-1) 

If `digestToSubaccount[digest] != bytes32(0)` the function returns immediately — no signature re-check, no margin deduction from parent, no re-registration of `parentSubaccounts`. Because the close left `digestToSubaccount[digest]` populated, this branch is taken on replay, returning the now-zero-balance `iso` address.

**`matchOrders` blindly substitutes the stale isolated subaccount as sender:** [3](#0-2) 

No liveness check (`isIsolatedSubaccountActive`, `parentSubaccounts != bytes32(0)`) is performed before the substitution. The order is routed to `iso` regardless of its closed state.

**Health check is a no-op in the base contract:** [4](#0-3) 

`isHealthy` is a `virtual` function that unconditionally returns `true`. The post-fill health check at line 826 therefore passes even when `iso` carries a position with zero quote balance.

---

### Impact Explanation

After the exploit sequence completes, `iso` holds an open perp position with zero USDC collateral. `parentSubaccounts[iso]` is `bytes32(0)` (cleared by close, never restored by the early-return path), so any subsequent liquidation or `_tryCloseIsolatedSubaccount` call will attempt to sweep funds to `bytes32(0)`, effectively burning them. The protocol absorbs the loss as bad debt. This directly matches the Critical scope: *unauthorized release / loss of funds via cross-contract state desynchronization between OffchainExchange, SpotEngine, and PerpEngine*.

---

### Likelihood Explanation

The sequence is reachable through normal sequencer-processed transactions:

1. Trader signs isolated order A (amount X, margin M).
2. Sequencer submits `CreateIsolatedSubaccount(A)` → `iso` created, M transferred from parent.
3. Sequencer submits `MatchOrders(A, counterparty)` → partially fills Y units; `filledAmounts[digestA] = Y`.
4. Trader signs reduce-only order B (amount −Y) to close the position.
5. Sequencer submits `MatchOrders(B, counterparty)` → position zeroed.
6. Sequencer submits `CloseIsolatedSubaccount(iso)` → `parentSubaccounts[iso]` cleared, M returned to parent; `digestToSubaccount[digestA]` **not** cleared.
7. Trader resubmits order A (remaining unfilled amount X − Y > 0 is valid).
8. Sequencer submits `CreateIsolatedSubaccount(A)` → early-return, **no margin transfer**, `iso` balance = 0.
9. Sequencer submits `MatchOrders(A, counterparty)` → fills X − Y units against zero-balance `iso`.

No privileged access, no key compromise, and no impossible token behavior is required. The only prerequisite is a partially-filled isolated order that has been closed before full fill.

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, after confirming `balance.amount == 0`, iterate over all digests that map to the closing subaccount and clear them, or — more practically — add a reverse mapping `subaccountToDigests` populated in `createIsolatedSubaccount` and cleared here:

```solidity
// in _tryCloseIsolatedSubaccount, after line 204:
bytes32[] storage digests = subaccountToDigests[subaccount];
for (uint256 i = 0; i < digests.length; i++) {
    delete digestToSubaccount[digests[i]];
    delete digestToMargin[digests[i]];
}
delete subaccountToDigests[subaccount];
```

Alternatively, add an active-subaccount liveness check in `matchOrders` before the `digestToSubaccount` substitution:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    bytes32 iso = digestToSubaccount[ordersInfo.taker.digest];
    require(parentSubaccounts[iso] != bytes32(0), ERR_INVALID_TAKER);
    taker.order.sender = iso;
}
```

    function _tryCloseIsolatedSubaccount(bytes32 subaccount) internal {
        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
        if (productId == 0) {
            return;
        }
        IPerpEngine.Balance memory balance = perpEngine.getBalance(
            productId,
            subaccount
        );
        if (balance.amount == 0) {
            uint8 id = RiskHelper.getIsolatedId(subaccount);
            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(
                    productId,
                    subaccount,
                    0,
                    -balance.vQuoteBalance
                );
                perpEngine.updateBalance(
                    productId,
                    parent,
                    0,
                    balance.vQuoteBalance
                );
            }
            int128 quoteBalance = spotEngine
                .getBalance(QUOTE_PRODUCT_ID, subaccount)
                .amount;
            if (quoteBalance != 0) {

                    subaccount,

        }
    }
