### Title
`op_mod` Uses Wrong Cost Constants (`DIV_BASE_COST`/`DIV_COST_PER_BYTE` Instead of `DIVMOD_BASE_COST`/`DIVMOD_COST_PER_BYTE`) — (File: `src/more_ops.rs`)

---

### Summary

`op_mod` (opcode 61) and its malachite variant charge execution cost using `DIV_BASE_COST` (988) and `DIV_COST_PER_BYTE` (4) — the same constants as `op_div`. However, computing the floor remainder requires the same underlying bignum floor-division as `op_divmod`, which correctly uses `DIVMOD_BASE_COST` (1116) and `DIVMOD_COST_PER_BYTE` (6). The wrong constants are wired in, causing `op_mod` to be systematically undercharged relative to the computation it actually performs. An attacker supplying crafted CLVM bytes can exploit this to execute more real computation per cost unit than the protocol intends.

---

### Finding Description

In `src/more_ops.rs`, the cost constants are defined as:

```
DIV_BASE_COST:    988   (line 55)
DIV_COST_PER_BYTE:  4   (line 56)

DIVMOD_BASE_COST: 1116  (line 52)
DIVMOD_COST_PER_BYTE: 6 (line 53)
```

`op_divmod` correctly uses its own constants: [1](#0-0) 

`op_mod` and `op_mod_malachite` both use the **wrong** constants — those belonging to `op_div`: [2](#0-1) [3](#0-2) 

The `mod` operator computes the floor remainder of two big integers. In both `num-bigint` and `malachite-bigint`, `mod_floor` is not a cheaper primitive than `div_mod_floor`; the remainder cannot be obtained without performing the full floor division. `op_divmod` calls `div_mod_floor` and `op_mod` calls `mod_floor`, but both require the same CPU work. The cost model already acknowledges this by giving `divmod` higher constants than `div`. Wiring `op_mod` to `DIV_BASE_COST`/`DIV_COST_PER_BYTE` instead of `DIVMOD_BASE_COST`/`DIVMOD_COST_PER_BYTE` is the same class of wrong-variable substitution as the external report: one similar-looking constant is silently used in place of the correct one.

The entry path is direct: any CLVM program byte-sequence that encodes opcode `0x3d` (61) with two atom arguments reaches `op_mod` through `ChiaDialect::op`: [4](#0-3) 

The `MALACHITE` flag path reaches `op_mod_malachite` with the same wrong constants: [5](#0-4)

### Impact Explanation

For maximum-size operands (256-byte dividend, 1024-byte divisor — the hard limits enforced by both `op_mod` and `op_divmod`):

| Operator | Charged cost |
|---|---|
| `mod` (current) | 988 + (256+1024)×4 = **6,108** |
| `divmod` (correct reference) | 1116 + (256+1024)×6 = **8,796** |

`op_mod` is undercharged by ~44% at maximum operand size. An attacker can replace every `divmod` call in a program with a `mod` call (discarding the quotient) and fit ~44% more real bignum divisions within the same `max_cost` budget. This breaks the cost model's DoS-prevention guarantee: nodes will spend more wall-clock CPU time per cost unit than the protocol assumes, allowing a single transaction to consume disproportionate validator resources. Because the cost model is consensus-critical in Chia (nodes reject programs that exceed `max_cost`), a systematic undercharge is a consensus-relevant arithmetic semantic mismatch.

---

### Likelihood Explanation

Likelihood is high. The `mod` opcode is a standard arithmetic primitive reachable by any CLVM program. No special permissions, flags, or softfork guards are required — opcode 61 is dispatched unconditionally in `ChiaDialect::op`. An attacker needs only to submit a transaction whose puzzle uses `mod` with large operands repeatedly. The undercharge is deterministic and reproducible across all nodes running this code.

---

### Recommendation

Replace `DIV_BASE_COST` and `DIV_COST_PER_BYTE` with `DIVMOD_BASE_COST` and `DIVMOD_COST_PER_BYTE` in both `op_mod` and `op_mod_malachite`:

```rust
// op_mod  (line 775)
- let cost = DIV_BASE_COST + ((a0_len + a1_len) as Cost) * DIV_COST_PER_BYTE;
+ let cost = DIVMOD_BASE_COST + ((a0_len + a1_len) as Cost) * DIVMOD_COST_PER_BYTE;

// op_mod_malachite  (line 800)
- let cost = DIV_BASE_COST + ((a0_len + a1_len) as Cost) * DIV_COST_PER_BYTE;
+ let cost = DIVMOD_BASE_COST + ((a0_len + a1_len) as Cost) * DIVMOD_COST_PER_BYTE;
```

Add a test that asserts `op_mod` with maximum-size operands charges the same base cost as `op_divmod`.

---

### Proof of Concept

Construct two CLVM programs with identical operands:

**Program A** — uses `divmod` (opcode 20 = `0x14`):
```
(divmod <256-byte-atom> <1024-byte-atom>)
```
Charged cost: **8,796** (plus malloc).

**Program B** — uses `mod` (opcode 61 = `0x3d`), discards quotient:
```
(mod <256-byte-atom> <1024-byte-atom>)
```
Charged cost: **6,108** (plus malloc).

Both programs perform identical bignum floor-division. Program B is accepted at a `max_cost` of 7,000 while Program A is rejected. An attacker submitting a puzzle built from repeated `mod` calls with large operands will consume ~44% more real CPU per cost unit than the protocol intends, degrading full-node performance without triggering the cost limit.
