Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap()` gates swaps on the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` inside the pool. If the pool admin allowlists the router to enable router-based swaps for legitimate users, every non-allowlisted user can bypass the restriction by routing through the router, completely defeating the access control.

## Finding Description

**Root cause — `MetricOmmPool.swap()` passes `msg.sender` as `sender`:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, recipient, ...)`, so `sender` is whoever called `pool.swap()`. [1](#0-0) 

**`ExtensionCalling._beforeSwap()` forwards that `sender` verbatim to the extension:** [2](#0-1) 

**`SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`:** [3](#0-2) 

**`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, making the router `msg.sender` inside the pool:** [4](#0-3) 

The same misbinding occurs in `exactInput()` (all hops): [5](#0-4) 

And in `exactOutputSingle()`: [6](#0-5) 

And in `_exactOutputIterateCallback()` for multi-hop exact-output: [7](#0-6) 

**Exploit flow:**
1. Pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict trading to specific addresses.
2. Admin calls `setAllowedToSwap(pool, alice, true)` and `setAllowedToSwap(pool, address(router), true)` to let alice use the router.
3. Attacker (not allowlisted) calls `router.exactInputSingle(...)` targeting the pool.
4. Router calls `pool.swap(attacker_recipient, ...)` — `msg.sender` inside the pool is the router.
5. `beforeSwap` receives `sender = address(router)`, finds `allowedSwapper[pool][router] == true`, and permits the swap.
6. Attacker trades on a pool they were never authorized to access.

No existing guard prevents this: the `allowedSwapper` mapping is keyed on the intermediary (router), not the economic actor.

## Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` intends to restrict trading to specific addresses (e.g., KYC-verified counterparties or whitelisted institutions). Once the router is allowlisted — a natural and necessary action to let legitimate users use the standard periphery — the allowlist check `allowedSwapper[pool][router]` returns `true` for every swap routed through it, regardless of who the original caller is. Any non-allowlisted user can trade freely on the curated pool, exposing LP funds to unauthorized counterparties and breaking any compliance or pricing assumption tied to the allowlist. This constitutes a broken core pool access-control mechanism causing unauthorized access to LP funds.

## Likelihood Explanation

The trigger is a natural and expected admin action: allowlisting the router so that allowlisted users can use the standard periphery. There is no warning in the contract or its NatSpec that doing so opens the gate to all users. Any pool that (a) uses `SwapAllowlistExtension` and (b) wants router support will hit this condition. The attacker needs no special privilege — a standard `exactInputSingle` call suffices, and the attack is repeatable indefinitely.

## Recommendation

Gate on the economically relevant actor, not the intermediary. Two sound approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Requires a coordinated convention between router and extension.
2. **Check `recipient` instead of `sender`**: The `recipient` argument to `beforeSwap` is set to the original user by the router (`params.recipient`). Gating on `recipient` would correctly identify the economic beneficiary for single-hop swaps, though the pool admin must decide which actor to gate for multi-hop paths where intermediate recipients are the router itself.
3. **Document and provide a router-aware extension**: Warn that allowlisting the router opens the gate to all users, and provide a separate extension that decodes the original caller from `extensionData`.

## Proof of Concept

```solidity
// Setup: curated pool with SwapAllowlistExtension
// Admin allowlists alice and the router (to let alice use the router)
swapAllowlist.setAllowedToSwap(pool, alice, true);
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attacker (not allowlisted) swaps directly → reverts correctly
vm.prank(attacker);
pool.swap(attacker, true, 1000, type(uint128).max, "", "");
// → NotAllowedToSwap ✓

// Attacker routes through the router → passes the allowlist check
vm.prank(attacker);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// → succeeds; allowlist check saw sender=router which is allowlisted ✗
```

The pool's `beforeSwap` hook receives `sender = address(router)`, finds it in `allowedSwapper[pool]`, and permits the swap. The attacker trades on a pool they were never authorized to access.
