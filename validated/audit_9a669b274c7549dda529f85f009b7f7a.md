The vulnerability is real and concrete. The key logic is in `contract/src/dogecoin.rs`:

- `submit_block_header` (lines 176–188): when `aux_data` is `None`, it falls through to the standalone PoW check with **no guard** verifying that the AuxPoW version flag is absent.
- `check_aux` (lines 58–61): the `BLOCK_VERSION_AUXPOW` flag check only runs when `aux_data` is `Some(...)`.

There is no `require!(block_header.version & BLOCK_VERSION_AUXPOW == 0, ...)` in the `else` branch.

### Title
Missing AuxPoW Version Flag Guard Allows AuxPoW-Flagged Dogecoin Block to Pass Standalone PoW Validation — (`contract/src/dogecoin.rs`)

### Summary
In the Dogecoin build, `submit_block_header` dispatches to either `check_aux` (when `aux_data` is `Some`) or a standalone `block_hash_pow() <= target` check (when `aux_data` is `None`). The AuxPoW version flag (`version & 0x100 != 0`) is only verified inside `check_aux`. No guard in the `else` branch rejects a block whose version field has the AuxPoW flag set. An attacker can submit a block with `version = 0x101` and `aux_data = None`, causing the contract to accept it via standalone PoW validation — bypassing all AuxPoW rules.

### Finding Description
In `submit_block_header`:

```rust
if let Some(ref aux_data) = aux_data {
    self.check_aux(&block_header, aux_data);
} else {
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        format!("block should have correct pow")
    );
}
``` [1](#0-0) 

The `else` branch performs only a hash-vs-target check. It never asserts `block_header.version & BLOCK_VERSION_AUXPOW == 0`. The flag check exists only inside `check_aux`:

```rust
require!(
    block_header.version & BLOCK_VERSION_AUXPOW != 0,
    "Aux POW block does not have AuxPoW flag set in version"
);
``` [2](#0-1) 

This check is one-directional: it rejects AuxPoW data submitted for a non-AuxPoW-flagged block, but it never rejects an AuxPoW-flagged block submitted without AuxPoW data.

Under Dogecoin consensus (post-block 371337), every block with `version & 0x100 != 0` **must** carry and pass AuxPoW validation. A block with the flag set but no AuxPoW data is unconditionally invalid on the Dogecoin network.

### Impact Explanation
An attacker who can call `submit_blocks` (either as a trusted relayer or holding `Role::UnrestrictedSubmitBlocks`) can insert a block that:
- Has `version = 0x101` (AuxPoW flag set)
- Has `aux_data = None`
- Has a `block_hash_pow()` value that satisfies `<= target_from_bits(bits)` (achievable by mining a standalone PoW block, which is far easier than constructing valid AuxPoW)

The block passes `check_target` (bits validation) and the standalone hash check, then is stored in `headers_pool` via `submit_block_header_inner`. [3](#0-2) 

If the block accumulates sufficient chainwork, it becomes the canonical tip. Downstream callers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will then accept Merkle proofs anchored to this invalid block, producing false transaction inclusion results. [4](#0-3) 

### Likelihood Explanation
The `submit_blocks` entrypoint is gated by `#[trusted_relayer]` with `bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks)`. [5](#0-4) 

Any account holding `Role::UnrestrictedSubmitBlocks` — or any staked trusted relayer — can reach this path directly. The standalone PoW work required is far less than constructing a valid AuxPoW chain, making the attack economically feasible for a motivated actor with relayer access.

### Recommendation
Add an explicit rejection of AuxPoW-flagged blocks in the `else` branch of `submit_block_header`:

```rust
} else {
    const BLOCK_VERSION_AUXPOW: i32 = 0x100;
    require!(
        block_header.version & BLOCK_VERSION_AUXPOW == 0,
        "AuxPoW-flagged block must be submitted with aux_data"
    );
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        format!("block should have correct pow")
    );
}
```

This mirrors the invariant already enforced in `check_aux` and closes the bypass.

### Proof of Concept
1. Deploy the Dogecoin build with `skip_pow_verification = false`.
2. Mine a block header with `version = 0x101` (AuxPoW flag set) whose `block_hash_pow()` satisfies `<= target_from_bits(bits)` (standard standalone PoW, no AuxPoW required).
3. Call `submit_blocks([(header_with_version_0x101, None)])`.
4. Observe: `check_aux` is never called; the block passes the standalone hash check and is stored in `headers_pool`.
5. Assert: the block is present in `headers_pool` — a Dogecoin-invalid block has been accepted by the light client.
