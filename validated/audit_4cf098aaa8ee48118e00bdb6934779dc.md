### Title
`AllResources` V3 Transaction with Zero L2/L1-Data Gas Deserialized as `L1Gas` via Protobuf, Producing Wrong Transaction Hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion in the P2P protobuf layer uses a zero-check on `l2_gas` and `l1_data_gas` to decide which `ValidResourceBounds` variant to reconstruct. A V3 transaction originally signed and hashed as `AllResources` (with zero l2_gas and zero l1_data_gas) is serialized to protobuf with those fields set to zero, then deserialized back as `L1Gas`. Because `get_tip_resource_bounds_hash` includes the `l1_data_gas` element in the Poseidon chain for `AllResources` but omits it for `L1Gas`, the reconstructed transaction produces a different hash than the one the signer committed to. This is the direct sequencer analog of the Maple Finance `triggerDefault` bug: control flow is driven by an internal zero-check on a field value rather than the canonical representation, and the discrepancy causes wrong downstream behavior.

---

### Finding Description

**Root cause — protobuf deserialization uses a zero-check to select the variant:** [1](#0-0) 

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The comment on line 426 (`// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.`) confirms this is a known compatibility shim, not an intentional canonical rule.

**Serialization is lossy — both variants produce identical protobuf bytes when l2/l1_data are zero:** [2](#0-1) 

When `ValidResourceBounds::L1Gas` is serialized, `l1_data_gas` is set to `ResourceBounds::default()` (zero). When `AllResources` with zero l2_gas and zero l1_data_gas is serialized, those fields are also zero. The two variants produce **identical protobuf bytes**, so deserialization always reconstructs `L1Gas` — the original `AllResources` variant is permanently lost.

**Hash function is sensitive to the variant — `AllResources` includes `l1_data_gas` in the Poseidon chain, `L1Gas` does not:** [3](#0-2) 

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements
    }
});
```

For `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)`:
```
hash = poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0), concat(L1_DATA_GAS, 0))
```

For `L1Gas(X)` (what deserialization produces):
```
hash = poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0))
```

These are **different Poseidon outputs** because the input length differs.

**The `InternalRpcInvokeTransactionV3` always uses `AllResources` at the gateway:** [4](#0-3) 

```rust
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
```

The gateway computes and stores `tx_hash` using `AllResources`. The P2P sync path reconstructs the transaction as `L1Gas` and recomputes a different hash.

**The same zero-check pattern also exists in the RPC layer:** [5](#0-4) 

```rust
if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
    Self::L1Gas(value.l1_gas)
} else { ... }
```

This affects the RPC read path for stored transactions, causing `starknet_getTransactionByHash` to return a transaction whose locally-recomputed hash does not match the stored hash.

---

### Impact Explanation

A V3 Invoke transaction submitted with `AllResources` bounds where `l2_gas = 0` and `l1_data_gas = 0` is a valid, well-formed transaction. The gateway accepts it, computes its hash using the `AllResources` branch (3-element Poseidon chain), and stores it. When this transaction is propagated via P2P and deserialized by a peer, the peer reconstructs it as `L1Gas` (2-element Poseidon chain). Any subsequent call to `calculate_transaction_hash` on the peer — including `validate_transaction_hash` during block validation — produces a hash that does not match the committed hash. This causes the peer to reject a valid block or to execute the transaction under the wrong `GasVectorComputationMode` (`NoL2Gas` instead of `All`), producing wrong fee accounting and wrong execution receipts.

**Matching impact**: *Wrong state, receipt, event, or revert result from blockifier/execution logic for accepted input* (Critical); *Transaction conversion or signature/hash logic binds the wrong hash or executable payload* (High).

---

### Likelihood Explanation

Any user can submit a V3 Invoke transaction with `AllResourceBounds { l1_gas: nonzero, l2_gas: 0, l1_data_gas: 0 }`. This is a structurally valid transaction accepted by the gateway. No special privilege is required. The trigger is a normal user action. The defect fires on every such transaction that crosses a P2P boundary.

---

### Recommendation

Preserve the original variant through the protobuf round-trip. Add a boolean or enum field to the protobuf `ResourceBounds` message (e.g., `bool use_all_resources`) that is set when the transaction was originally `AllResources`, so deserialization can reconstruct the correct variant regardless of whether the numeric values are zero. Alternatively, always deserialize as `AllResources` when all three fields are present in the protobuf message (which they are, per the serialization code), and only fall back to `L1Gas` when `l1_data_gas` is absent (the pre-0.13.3 wire format). The `rpc_transaction.rs` converter already does this correctly: [6](#0-5) 

```rust
impl TryFrom<protobuf::ResourceBounds> for AllResourceBounds {
    // Always AllResources — no zero-check, no variant ambiguity
    Ok(Self {
        l1_gas: value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?,
        l2_gas: value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?,
        l1_data_gas: value.l1_data_gas.ok_or(missing("ResourceBounds::l1_data_gas"))?.try_into()?,
    })
}
```

Apply the same unconditional `AllResources` reconstruction to `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` when `l1_data_gas` is present in the message, and to `From<ResourceBoundsMapping> for ValidResourceBounds` in the RPC layer.

### Proof of Concept

1. Construct a V3 Invoke transaction with `resource_bounds = AllResourceBounds { l1_gas: ResourceBounds { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: ResourceBounds::default(), l1_data_gas: ResourceBounds::default() }`.
2. Compute its hash via `get_invoke_transaction_v3_hash` — this calls `get_tip_resource_bounds_hash` with `AllResources`, producing a 3-element Poseidon chain. Call this `H_original`.
3. Serialize the transaction to `protobuf::ResourceBounds` via `From<ValidResourceBounds>`. Observe that `l1_data_gas` is set to zero.
4. Deserialize via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Observe the result is `L1Gas(ResourceBounds { max_amount: 1000, max_price_per_unit: 1 })`.
5. Recompute the hash — `get_tip_resource_bounds_hash` now uses a 2-element Poseidon chain. Call this `H_reconstructed`.
6. Assert `H_original != H_reconstructed`. The two hashes differ because the Poseidon input length differs by one element (`concat(L1_DATA_GAS, 0)`).
7. Call `validate_transaction_hash` with the deserialized transaction and `H_original` — it returns `false`, causing the block containing this transaction to be rejected by any syncing peer.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
