The code evidence is clear and the vulnerability is real.

## The Exact Bug

In `crates/blockifier/src/transaction/transactions.rs`, the `run_execute` implementation for `DeclareTransaction` contains this guard:

```rust
if context.tx_context.block_context.versioned_constants.block_casm_hash_v1_declares
    && self.version() >= TransactionVersion::THREE
{
    self.check_compile_class_hash_v2_declaration()?
}
try_declare(self, state, class_hash, Some(*compiled_class_hash))?
``` [1](#0-0) 

The condition requires **both** `block_casm_hash_v1_declares == true` **and** `version >= THREE`. For a `DeclareV2` transaction (`TransactionVersion::TWO`), the second condition is always false, so `check_compile_class_hash_v2_declaration()` is **never called**, regardless of the `block_casm_hash_v1_declares` flag.

## What `check_compile_class_hash_v2_declaration` Does

The function, defined in `crates/starknet_api/src/executable_transaction.rs`, recomputes the CASM hash using `HashVersion::V2` (Blake2Felt252) and asserts it matches the `compiled_class_hash` field in the transaction: [2](#0-1) 

Without this check, the `compiled_class_hash` field is taken verbatim from the transaction and stored directly into state via `try_declare`: [3](#0-2) 

## The Divergent Value

`HashVersion::V1` uses Poseidon; `HashVersion::V2` uses Blake2Felt252. These produce numerically different `CompiledClassHash` values for the same CASM: [4](#0-3) 

An attacker submitting a `DeclareV2` with `compiled_class_hash = casm.hash(HashVersion::V1)` when `block_casm_hash_v1_declares=true` will have the V1 Poseidon hash stored in state — the exact divergent value the flag was introduced to prevent.

### Title
DeclareV2 bypasses `block_casm_hash_v1_declares` V2-hash enforcement, storing V1 Poseidon `CompiledClassHash` in state — (`crates/blockifier/src/transaction/transactions.rs`)

### Summary
When `block_casm_hash_v1_declares=true` (Starknet ≥ 0.14.1), the blockifier is supposed to reject any new class declaration whose `compiled_class_hash` is not a V2 (Blake2Felt252) hash. The enforcement guard is gated on `version >= THREE`, so `DeclareV2` transactions are entirely exempt. An attacker can submit a `DeclareV2` with a V1 (Poseidon) `compiled_class_hash` and it will be accepted and committed to state.

### Finding Description
In `Executable<S> for DeclareTransaction::run_execute`, the branch handling `DeclareTransaction::V2` and `DeclareTransaction::V3` shares a single code path. The call to `check_compile_class_hash_v2_declaration()` is guarded by `self.version() >= TransactionVersion::THREE`, which is always false for V2. The `compiled_class_hash` from the transaction is then passed directly to `try_declare` → `state.set_compiled_class_hash(class_hash, compiled_class_hash)` with no hash-domain validation. [5](#0-4) 

### Impact Explanation
The `CompiledClassHash` stored in state is used in the state commitment trie and is read by the OS during proof generation. If a V1 Poseidon hash is stored for a class declared after the 0.14.1 migration boundary, the OS — which expects all post-migration classes to carry V2 Blake hashes — will encounter an unexpected value. This causes state commitment divergence between the sequencer and the OS/prover, which is a Critical impact (wrong state value for an accepted transaction).

### Likelihood Explanation
`DeclareV2` transactions are still structurally accepted by the blockifier (the match arm exists and executes). The attacker only needs to craft a valid `DeclareV2` with `compiled_class_hash` set to the V1 Poseidon hash of their CASM. No privileged access is required.

### Recommendation
Change the version guard to also cover `TransactionVersion::TWO`:

```rust
if context.tx_context.block_context.versioned_constants.block_casm_hash_v1_declares
    && self.version() >= TransactionVersion::TWO   // was: THREE
{
    self.check_compile_class_hash_v2_declaration()?
}
```

Alternatively, if `DeclareV2` is intended to be fully deprecated in 0.14.1+ blocks, add a gateway-level rejection for `DeclareV2` when `block_casm_hash_v1_declares=true`.

### Proof of Concept
Build a Rust test in `crates/blockifier` with `block_casm_hash_v1_declares=true`, execute a `DeclareV2` transaction whose `compiled_class_hash` equals `casm.hash(&HashVersion::V1)`, and assert that `state.get_compiled_class_hash(class_hash) == V1_hash`. The assertion will pass, confirming the V1 hash is stored in state despite the flag being active.

    tx: &DeclareTransaction,
    state: &mut S,
    class_hash: ClassHash,

    pub fn check_compile_class_hash_v2_declaration(&self) -> Result<(), StarknetApiError> {

                expected: compiled_class_hash_v2,

        }
        Ok(())
