### Title
Slow Mode `LinkSigner` Lacks Nonce Validation, Enabling Irrevocable Linked Signer Backdoor — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The slow mode execution path for `LinkSigner` in `processSlowModeTransactionImpl` authenticates the caller only by checking that `msg.sender` matches the address portion of the subaccount (`validateSender`), with no nonce validation. The fast path for the same operation enforces both an EIP-712 signature and a nonce. Because slow mode transactions cannot be cancelled once queued, an attacker who briefly controls a victim's address can queue a slow mode `LinkSigner` that will reinstate their address as linked signer three days later — surviving any fast-path revocation the victim performs in the interim.

---

### Finding Description

**Fast path** (`processTransactionImpl`, `EndpointTx.sol` lines 576–590):

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true
);
linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
```

`validateSignedTx` calls `validateNonce` (which increments `nonces[address]`) and `validateSignature` (EIP-712 ECDSA). Both checks are mandatory.

**Slow mode path** (`processSlowModeTransactionImpl`, `EndpointTx.sol` lines 232–239):

```solidity
IEndpoint.LinkSigner memory txn = abi.decode(transaction[1:], (IEndpoint.LinkSigner));
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

`validateSender` only checks:

```solidity
address(uint160(bytes20(txSender))) == sender || sender == address(this)
```

No nonce is read or incremented. No signature is verified. The `nonce` field present in the `LinkSigner` struct (as declared in `Verifier.sol`'s `LINK_SIGNER_SIGNATURE` constant) is decoded but silently ignored.

The slow mode submission gate (`submitSlowModeTransactionImpl`, lines 332–385) places `LinkSigner` in the fee-paying `else` branch — any address can submit one by paying the $1 slow mode fee. Once queued, the transaction executes unconditionally after `SLOW_MODE_TX_DELAY` (3 days); there is no cancellation function anywhere in the contract.

---

### Impact Explanation

`linkedSigners[subaccount]` is the sole alternative signer accepted by both `validateSignature` / `validateCompactSignature` (EndpointTx.sol lines 172–198) and `_checkSignature` in OffchainExchange (lines 332–343). A linked signer can sign `WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, `LiquidateSubaccount`, `MintNlp`, `BurnNlp`, and all order types on behalf of the subaccount owner. Reinstating an attacker-controlled address as linked signer therefore grants full asset-draining authority over the victim's subaccount.

**Corrupted state:** `linkedSigners[victim_subaccount]` is set to `attacker_address` after the victim has already revoked it, breaking the invariant that a fast-path revocation is final.

---

### Likelihood Explanation

The attack requires only a brief window of control over the victim's Ethereum address (e.g., a temporarily compromised hot wallet, a phishing-signed transaction, or a session key leak). The attacker does not need sustained access: one `submitSlowModeTransaction` call is sufficient to plant the backdoor. The 3-day delay is publicly visible on-chain but provides no remedy because there is no cancellation path. The victim's fast-path revocation is silently overridden when the queued transaction executes.

---

### Recommendation

1. **Add nonce validation to the slow mode `LinkSigner` handler.** Read `txn.nonce` from the decoded struct and call `validateNonce(txn.sender, txn.nonce)` before writing to `linkedSigners`.
2. **Alternatively, implement a slow mode transaction cancellation function** that allows the original submitter to invalidate a queued transaction before it executes.
3. **Consider requiring a signature even in the slow mode path** for `LinkSigner`, consistent with the fast path, since the operation grants persistent signing authority.

---

### Proof of Concept

1. Attacker briefly controls `victim_address` (e.g., via a compromised session key).
2. Attacker calls `Endpoint.submitSlowModeTransaction` with:
   ```
   transaction = abi.encodePacked(
       uint8(TransactionType.LinkSigner),
       abi.encode(LinkSigner({ sender: victim_subaccount, signer: bytes32(uint256(uint160(attacker_address))), nonce: 0 }))
   )
   ```
   `msg.sender = victim_address` → `validateSender` passes because `address(bytes20(victim_subaccount)) == victim_address`. Slow mode fee is paid. Transaction is queued with `executableAt = block.timestamp + 3 days`.
3. Victim regains control and immediately calls the fast path `LinkSigner` to set `signer = address(0)` (revocation). Nonce is incremented to N+1. `linkedSigners[victim_subaccount] = address(0)`.
4. Three days pass. `executeSlowModeTransaction` is called (by anyone). `processSlowModeTransactionImpl` executes the queued transaction: `linkedSigners[victim_subaccount] = attacker_address`. No nonce check prevents this.
5. Attacker signs a `WithdrawCollateral` or `WithdrawCollateralV2` transaction using `attacker_address` as the linked signer. `validateSignature` accepts it (EndpointTx.sol line 180: `allowLinkedSigner ? getLinkedSigner(sender) : address(0)`).     function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days

            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }
