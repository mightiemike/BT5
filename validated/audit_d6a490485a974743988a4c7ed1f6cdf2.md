### Title
`ValidResourceBounds` Protobuf Deserialization Produces `L1Gas` Variant for Zero-Bounds Transactions, Diverging from the `AllResources` Hash Domain Used at Ingestion — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The P2P block-sync protobuf converter for `ValidResourceBounds` silently narrows any invoke-V3 transaction whose `l2_gas` and `l1_data_gas` fields are both zero into the `ValidResourceBounds::L1Gas` variant. The RPC/gateway ingestion path always stores the same transaction under `ValidResourceBounds::AllResources`. Because `get_tip_resource_bounds_hash` includes the `L1_DATA_GAS` resource felt in the Poseidon preimage only for the `AllResources` arm, the two paths produce structurally different hash preimages for identical wire bytes, breaking the canonicalization invariant that every representation of the same transaction must hash to the same value.

### Finding Description

**Ingestion path (RPC → gateway → mempool → block)**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`. The converter `TransactionConverter::convert_rpc_tx_to_internal` wraps it as `InternalRpcTransactionWithoutTxHash::Invoke(tx.into())`, where `InternalRpcInvokeTransactionV3::resource_bounds()` always returns `ValidResourceBounds::AllResources(self.resource_bounds)`. [1](#0-0) 

`get_invoke_transaction_v3_hash` then calls `get_tip_resource_bounds_hash` with `AllResources`, which appends the `L1_DATA_GAS` resource felt to the Poseidon preimage even when `l1_data_gas = {0, 0}`: [2](#0-1) 

The resulting hash `H_all` is stored in `InternalRpcTransaction.tx_hash` and committed to the block. [3](#0-2) 

**P2P block-sync path**

When a syncing node deserializes the same transaction from protobuf via `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`, it calls `ValidResourceBounds::try_from(protobuf::ResourceBounds)`: [4](#0-3) 

The branch at line 431 produces `ValidResourceBounds::L1Gas(l1_gas)` whenever `l1_data_gas.is_zero() && l2_gas.is_zero()`. For the `L1Gas` arm, `get_tip_resource_bounds_hash` appends **no** `L1_DATA_GAS` felt: [5](#0-4) 

The recomputed hash `H_l1` ≠ `H_all`. The two paths diverge on the same wire bytes.

**Contrast with the consensus/RPC protobuf path**

The `rpc_transaction.rs` converter for `AllResourceBounds` always requires all three fields and never narrows to `L1Gas`: [6](#0-5) 

Only the block-sync `transaction.rs` converter performs the narrowing.

**Gateway admission of the trigger transaction**

The stateless validator accepts a transaction with `l1_gas = {amount>0, price>0}`, `l2_gas = {0, 0}`, `l1_data_gas = {0, 0}` because `max_possible_fee` is non-zero (driven by l1_gas) and the `min_gas_price` check on `l2_gas.max_price_per_unit` passes when `min_gas_price = 0`: [7](#0-6) 

### Impact Explanation

A syncing node that recomputes the transaction hash from the deserialized `InvokeTransactionV3` obtains `H_l1` while the committed block records `H_all`. This breaks the canonicalization invariant that every representation of the same transaction must hash identically. Concretely:

- Block-hash validation that covers transaction hashes will fail on syncing nodes for any block containing such a transaction, causing those nodes to reject a valid sequencer block.
- Transaction-by-hash RPC lookups on syncing nodes will fail or return wrong data because the stored key diverges from the key used by the sequencer.
- The wrong hash binding falls squarely in the **High** impact category: "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."

### Likelihood Explanation

Low-to-moderate. The trigger requires a gateway configuration with `min_gas_price = 0` (or any value that permits `l2_gas.max_price_per_unit = 0`) so that a transaction with all-zero l2/l1_data bounds is admitted. The attacker needs no privilege beyond submitting a standard invoke-V3 transaction. The divergence is deterministic once the transaction is included in a block.

### Recommendation

Replace the narrowing heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with a strict rule that always produces `AllResources` for any transaction deserialized from the block-sync path, matching the invariant enforced by the RPC/consensus path. Alternatively, remove the `L1Gas` variant from the hash preimage logic entirely and always include the `L1_DATA_GAS` felt (as zero when absent), so both paths produce identical preimages regardless of which `ValidResourceBounds` variant is in use.

### Proof of Concept

1. Submit an invoke-V3 transaction to the gateway with:
   - `l1_gas = { max_amount: 1, max_price_per_unit: 1 }`
   - `l2_gas = { max_amount: 0, max_price_per_unit: 0 }`
   - `l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }`
   - `min_gas_price = 0` in gateway config

2. Gateway accepts the transaction. `InternalRpcInvokeTransactionV3::resource_bounds()` returns `ValidResourceBounds::AllResources(...)`. `get_tip_resource_bounds_hash` appends `L1_DATA_GAS = 0` to the Poseidon preimage. Hash `H_all` is stored in the block.

3. A syncing node receives the block via P2P. `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` calls `ValidResourceBounds::try_from`. Since `l2_gas.is_zero() && l1_data_gas.is_zero()`, it returns `ValidResourceBounds::L1Gas(l1_gas)`.

4. The syncing node recomputes the hash. `get_tip_resource_bounds_hash` takes the `L1Gas` arm and does **not** append `L1_DATA_GAS`. Hash `H_l1` ≠ `H_all`.

5. Block or transaction hash validation on the syncing node fails, causing it to reject the valid block or store the transaction under the wrong key. [8](#0-7) [5](#0-4) [1](#0-0)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
