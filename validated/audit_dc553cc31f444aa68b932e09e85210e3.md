### Title
Division by Zero in `PerpEngine.socializeSubaccount` When `openInterest` Is Zero Permanently Blocks Liquidation Finalization — (File: `core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when spreading a residual negative `vQuoteBalance` across market participants. When `openInterest` is zero — a reachable state after all participants have closed their positions — `MathSD21x18.div` reverts with `"DBZ"`. This permanently blocks the `_finalizeSubaccount` liquidation path for any subaccount holding residual negative `vQuoteBalance` in a market with no remaining open interest, leaving insolvent accounts unresolvable and the insurance fund unable to cover the loss.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after insurance is exhausted, the code socializes the remaining negative `vQuoteBalance` across all open-interest holders:

```solidity
// actually socialize if still not enough
if (balance.vQuoteBalance < 0) {
    // socialize across all other participants
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ← reverts with "DBZ" if openInterest == 0
    );
    state.cumulativeFundingLongX18 += fundingPerShare;
    state.cumulativeFundingShortX18 -= fundingPerShare;
    balance.vQuoteBalance = 0;
}
``` [1](#0-0) 

`MathSD21x18.div` explicitly guards against zero denominators:

```solidity
function div(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        require(y != 0, ERR_DIV_BY_ZERO);
``` [2](#0-1) 

Any call with `openInterest == 0` therefore reverts unconditionally.

**Reachable trigger path:**

1. A trader opens a perp position; `openInterest` becomes non-zero.
2. Adverse funding payments accumulate, making `balance.vQuoteBalance < 0`.
3. The trader closes their position (`amount = 0`); `openInterest` decreases by their share.
4. All counterparties (e.g., NLP) also close their positions; `openInterest` reaches `0`.
5. The trader's account is now unhealthy (negative health from `vQuoteBalance < 0`).
6. A liquidator calls `liquidateSubaccountImpl` with `productId = type(uint32).max`.
7. `_finalizeSubaccount` in `ClearinghouseLiq.sol` passes all position checks (all `amount == 0`) and calls `perpEngine.socializeSubaccount`. [3](#0-2) 

8. The division by `openInterest = 0` reverts with `"DBZ"`, blocking finalization entirely.

The violated assumption is: *"there is always at least one other open-interest holder to absorb the socialized loss."* This assumption is not enforced or checked anywhere in the code path.

---

### Impact Explanation

The liquidation finalization path is permanently blocked for any subaccount matching the above conditions. The insolvent account cannot be cleaned up, the insurance fund cannot be applied to cover the residual loss, and the socialization mechanism is broken for that market. This is not a transient revert — the condition (`openInterest == 0` with `vQuoteBalance < 0`) persists until an external participant opens a new position in that market, which may never happen in a winding-down or low-liquidity market. The protocol is left with an unresolvable insolvent account, corrupting its accounting invariants.

---

### Likelihood Explanation

The scenario is realistic in low-liquidity or winding-down perp markets. A single trader who accumulates negative funding payments and then closes their position — with no other participants remaining — can trigger this condition through entirely normal trading activity. No privileged access, sequencer compromise, or social engineering is required. Any unprivileged trader interacting through the standard `matchOrders` → `liquidateSubaccountImpl` path can reach this state.

---

### Recommendation

Before dividing by `state.openInterest`, add an explicit zero-check:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No participants to socialize against; write off the loss entirely
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

This mirrors the intent described in the analogous nord server fix: handle the edge case explicitly rather than assuming the denominator is always non-zero.

---

### Proof of Concept

1. Deploy a perp market (e.g., ETH-PERP).
2. Trader A opens a long position; NLP takes the short. `openInterest > 0`.
3. Funding rate is negative for longs; Trader A accumulates `vQuoteBalance < 0`.
4. Trader A closes their long (`amount = 0`). NLP closes its short. `openInterest = 0`.
5. Trader A's account health is negative due to `vQuoteBalance < 0`.
6. Liquidator calls `liquidateSubaccountImpl(txn)` with `txn.productId = type(uint32).max`.
7. `_finalizeSubaccount` passes all position checks (all `amount == 0`). [4](#0-3) 

8. `perpEngine.socializeSubaccount(txn.liquidatee, v.insurance)` is called.
9. `balance.vQuoteBalance < 0` → enters socialization branch.
10. `MathSD21x18.div(-balance.vQuoteBalance, state.openInterest)` reverts: `"DBZ"`.
11. Finalization is permanently blocked; the insolvent account cannot be resolved.
