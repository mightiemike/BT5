Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates `owner` Instead of `sender`, Allowing Any Unprivileged Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual token-paying caller) and validates only `owner` (the LP position recipient). Because `MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address supplied by the caller, any unprivileged address can bypass the deposit allowlist by passing any already-allowlisted address as `owner`. The pool admin's access control is completely defeated.

## Finding Description
`MetricOmmPool.addLiquidity` (L191) calls `_beforeAddLiquidity(msg.sender, owner, ...)`, forwarding both the actual caller (`msg.sender`) and the arbitrary `owner` address to the extension. Inside `DepositAllowlistExtension.beforeAddLiquidity` (L32–42), the first parameter (`sender`) is unnamed and ignored; only `owner` is checked:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The actual token payment is collected from `msg.sender` (the caller) via the callback in `LiquidityLib.addLiquidity` (L147–148), while LP shares are credited to `owner` via `positionBinShares[posKey] = newUserShares` (L121), where `posKey = keccak256(owner, salt, bin)`. Since `removeLiquidity` enforces `msg.sender == owner` (MetricOmmPool.sol L206), the allowlisted `owner` can later call `removeLiquidity` to recover the tokens deposited by the unprivileged caller.

`SwapAllowlistExtension.beforeSwap` (L31–41) correctly checks `sender` (the first parameter), confirming the asymmetry is a bug in the deposit guard, not an intentional design.

## Impact Explanation
The deposit allowlist provides zero protection against unauthorized deposits. Any unprivileged address can inject liquidity into a restricted pool by routing through any allowlisted `owner`. This is a direct admin-boundary break: the pool admin configured the allowlist to restrict depositors (e.g., for KYC/AML compliance or pool composition control), but the guard is completely bypassed. The wrong value is `allowedDepositor[pool][owner]` being checked instead of `allowedDepositor[pool][sender]`, causing the extension to return the valid selector and permit the deposit unconditionally for any caller who knows an allowlisted address.

## Likelihood Explanation
The bypass requires no special permissions, no flash loan, and no oracle manipulation. Any EOA or contract can call `addLiquidity` with an arbitrary `owner`. Allowlisted addresses are discoverable on-chain via the `AllowedToDepositSet` event or the `allowedDepositor` mapping. The attack is a single transaction, repeatable indefinitely.

## Recommendation
Check `sender` (the actual token payer) instead of `owner` (the LP position recipient), mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  - Pool P has DepositAllowlistExtension configured.
  - allowedDepositor[P][Alice] = true
  - Bob is NOT allowlisted.

Attack:
  1. Bob deploys a contract implementing IMetricOmmModifyLiquidityCallback
     that transfers required tokens to the pool in the callback.
  2. Bob calls pool.addLiquidity(owner=Alice, salt=0, deltas=..., callbackData=..., extensionData=...).
  3. Pool calls _beforeAddLiquidity(sender=Bob, owner=Alice, ...) → extension checks
     allowedDepositor[P][Alice] → true → no revert.
  4. LiquidityLib credits positionBinShares[keccak256(Alice,0,bin)] += shares.
  5. Callback fires to Bob's contract; Bob's contract transfers tokens to the pool.
  6. Alice calls removeLiquidity(owner=Alice, ...) — passes msg.sender==owner check —
     and recovers the tokens.

Result: Bob deposited into a restricted pool without being allowlisted.
        The deposit allowlist is completely bypassed.
