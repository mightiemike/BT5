### Title
SwapAllowlistExtension Gates the Router Address Instead of the Originating User, Allowing Any Caller to Bypass the Swap Allowlist via MetricOmmSimpleRouter — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument supplied by the pool, which is always `msg.sender` at the pool level. When a swap is routed through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the originating user. A pool admin who allowlists the router to enable router-mediated swaps for legitimate users simultaneously opens an unconditional bypass: any non-allowlisted address can call the router and pass the extension check because the router is allowlisted, not the user.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // ← always the immediate caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension:

```solidity
// ExtensionCalling.sol:162-165
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)
)
```

`SwapAllowlistExtension.beforeSwap` then gates on that value:

```solidity
// SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)` directly:

```solidity
// MetricOmmSimpleRouter.sol:72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
);
```

When this call reaches the pool, `msg.sender` is the router. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`. The router carries no per-user identity into the extension.

This creates a binary trap for pool admins:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all — broken core functionality |
| **Allowlist the router** | Every address on the network can bypass the allowlist by calling the router |

The router is a public, permissionless contract. Allowlisting it is the only way to let legitimate users swap through it, but doing so makes the allowlist meaningless.

---

### Impact Explanation

**Direct loss / broken core functionality — High.**

A curated pool deploying `SwapAllowlistExtension` to restrict trading to KYC'd or otherwise vetted addresses loses that restriction entirely the moment the router is allowlisted. Any non-allowlisted address can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) and execute a swap against the pool. The pool receives and pays out real tokens; the allowlist provides zero protection.

---

### Likelihood Explanation

**High.**

The router is the standard, documented periphery path for end-user swaps. Pool admins who want allowlisted users to be able to use the router (the normal UX) must allowlist it. The bypass requires no special privileges, no flash loans, no price manipulation — only a call to a public function on a public contract. Any user who discovers the router is allowlisted can exploit it immediately.

---

### Recommendation

The extension must gate on the **economically relevant actor**, not the immediate pool caller. Two sound approaches:

1. **Pass the original user through the router.** Add an authenticated `originSender` field to the swap call or extension payload that the router populates with `msg.sender` before calling the pool. The extension reads that field instead of the `sender` argument. This requires a protocol-level convention and cannot be spoofed if the pool verifies the payload source.

2. **Check `tx.origin` as a fallback only when `sender` is a known router.** Less clean but immediately deployable: if `sender` is a factory-registered router, substitute `tx.origin` for the allowlist lookup. This is safe only for EOA users and breaks for smart-contract wallets.

3. **Require direct pool calls for allowlisted pools.** Document that pools using `SwapAllowlistExtension` must not allowlist the router, and that allowlisted users must call `pool.swap()` directly. This is a policy fix, not a code fix, and is fragile.

The cleanest long-term fix is option 1: extend the extension interface to carry a verified originator field that the router sets and the pool authenticates.

---

### Proof of Concept

```
Setup
─────
1. Deploy a pool with SwapAllowlistExtension as a beforeSwap hook.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is the only allowed swapper
3. Pool admin calls setAllowedToSwap(pool, router, true)  // required so alice can use the router

Attack
──────
4. Bob (not allowlisted) calls:
       router.exactInputSingle({
           pool:      pool,
           tokenIn:   token0,
           recipient: bob,
           zeroForOne: true,
           amountIn:  X,
           ...
       })

5. router calls pool.swap(bob_recipient, true, X, ...) — msg.sender = router
6. pool calls _beforeSwap(router, ...)
7. extension checks allowedSwapper[pool][router] → true → no revert
8. Swap executes; Bob receives token1 from the curated pool.

Result: Bob, who is not on the allowlist, successfully swaps against a pool
        that was configured to restrict trading to alice only.
