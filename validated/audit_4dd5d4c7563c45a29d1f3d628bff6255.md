### Title
Stale Slow-Mode `LinkSigner` Unconditionally Overwrites Sequencer-Revoked Linked Signer, Restoring Attacker Signing Authority — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

`processSlowModeTransactionImpl` unconditionally writes `linkedSigners[txn.sender]` with whatever address was encoded in the slow-mode payload at submission time, with no check against the current value. An attacker who briefly compromises a user's wallet can queue a slow-mode `LinkSigner` pointing to an attacker-controlled address. Even if the user immediately revokes via the sequencer path, the queued slow-mode transaction fires three days later and silently restores the attacker's signing authority over the subaccount.

---

### Finding Description

There are two code paths that write `linkedSigners`:

**Sequencer path** (`processTransactionImpl`, line 576–590):
```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(...);
    validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, ...);
    linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
}
```

**Slow-mode path** (`processSlowModeTransactionImpl`, line 232–239):
```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
```

Both writes are unconditional. Neither checks whether `linkedSigners[txn.sender]` has been updated since the transaction was submitted. The slow-mode queue has a hardcoded three-day delay (`SLOW_MODE_TX_DELAY`). A slow-mode `LinkSigner` submitted at time T carries a snapshot of the desired signer at T. Any sequencer-path revocation applied between T and T+3 days is silently overwritten when the slow-mode transaction executes.

There is no cancellation mechanism for queued slow-mode transactions. The queue is strictly FIFO (`txUpTo` increments monotonically). `executeSlowModeTransaction()` is callable by any unprivileged address after the delay, so the attacker can trigger execution themselves. ---

### Impact Explanation

`linkedSigners[subaccount]` is the sole authorization gate for all sequencer-path transactions that pass `allowLinkedSigner = true`: `WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, `MintNlp`, `BurnNlp`, `LiquidateSubaccount`, and order matching. Restoring an attacker-controlled address as the linked signer gives the attacker full signing authority to drain the subaccount's collateral via `WithdrawCollateral` or redirect it via `TransferQuote`, with no further on-chain check. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The attacker requires only a brief, temporary compromise of the victim's wallet (phishing, malware, session hijack) to call `submitSlowModeTransaction` once. The slow-mode fee is the only cost. After the wallet is recovered and the user revokes via the sequencer, the user has no on-chain mechanism to cancel the queued slow-mode transaction. After three days, the attacker calls `executeSlowModeTransaction()` themselves (public, permissionless), the stale payload fires, and the attacker's address is restored as linked signer. The attack window between restoration and the user's next revocation is sufficient to submit a `WithdrawCollateral` signed by the attacker's key. [3](#0-2) [6](#0-5) 

---

### Recommendation

Before writing `linkedSigners[txn.sender]` in the slow-mode path, compare the payload's signer against the current value and revert (or no-op) if the current value differs from what was set at submission time. Alternatively, record a `linkedSignerNonce` per subaccount that is incremented on every sequencer-path `LinkSigner` update, embed the expected nonce in the slow-mode payload at submission time, and reject the slow-mode execution if the nonce has advanced. Either approach ensures that an intervening sequencer-path revocation permanently invalidates the in-flight slow-mode payload. [1](#0-0) 

---

### Proof of Concept

1. Attacker briefly compromises victim's wallet (address `V`). Victim has subaccount `V ++ name` with no linked signer.
2. Attacker calls `submitSlowModeTransaction` from `V` with payload `LinkSigner { sender: V++name, signer: AttackerAddress }`. Slow-mode fee is paid. The tx is queued at index `N`, executable at `block.timestamp + 3 days`.
3. Victim recovers wallet, immediately submits sequencer-path `LinkSigner { sender: V++name, signer: address(0) }` (nonce-validated, processed within seconds). `linkedSigners[V++name]` is now `address(0)`.
4. Victim believes they are safe. No on-chain cancellation of slow-mode index `N` exists.
5. After three days, attacker calls `executeSlowModeTransaction()`. `processSlowModeTransactionImpl` runs: `validateSender` passes (stored `sender` is `V`), `requireSubaccount` passes, and `linkedSigners[V++name] = AttackerAddress` is written unconditionally, overwriting the revocation.
6. Attacker signs a `WithdrawCollateral` transaction with `AttackerAddress`'s key, targeting `V++name`. The sequencer path calls `validateSignature` → `getLinkedSigner` returns `AttackerAddress` → signature validates → `clearinghouse.withdrawCollateral` drains the subaccount to the attacker.
