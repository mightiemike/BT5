### Title
Wrong Block Timestamp Used in Difficulty Retarget Calculation (Mainchain Height Lookup Instead of Fork Ancestor) - (`contract/src/dogecoin.rs`)

### Summary

In `contract/src/dogecoin.rs`, the `get_next_work_required` function fetches the retarget boundary block's timestamp using `get_header_by_height`, which reads from the **mainchain height index**. When a fork block crosses a difficulty adjustment boundary, this returns the mainchain's block timestamp at that height rather than the fork chain's actual ancestor block timestamp. The wrong timestamp is then fed into `calculate_next_work_required`, producing an incorrect expected difficulty target for the fork block.

### Finding Description

The Dogecoin difficulty retarget logic in `get_next_work_required` computes how far back to walk (`blocks_to_go_back`) and then retrieves the first block in the retarget window: [1](#0-0) 

```rust
let height_first = prev_block_header
    .block_height
    .checked_sub(blocks_to_go_back)
    .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;

calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
```

The developer's own `TODO` comment acknowledges the concern. `get_header_by_height` reads from `mainchain_height_to_header`, the bidirectional mainchain index: [2](#0-1) 

During fork evaluation, the block at `height_first` on the mainchain is not necessarily the ancestor of the fork block at that height. The fork may have diverged before `height_first`, meaning the mainchain block at that height is a completely different block with a different timestamp. The correct approach is to walk back through the fork's own chain ancestry (via `get_prev_header`) to find the true retarget boundary block.

The correct difficulty calculation requires:

```
actual_time_taken = prev_block_time - first_block_time_of_FORK_ANCESTOR
```

But the contract computes:

```
actual_time_taken = prev_block_time - first_block_time_of_MAINCHAIN_BLOCK_AT_SAME_HEIGHT
```

These two timestamps can differ arbitrarily when the fork diverged before the retarget boundary.

### Impact Explanation

The incorrect `first_block_time` produces a wrong `actual_time_taken`, which is clamped and then used to scale the new difficulty target: [3](#0-2) 

An attacker who controls the timestamps of blocks on a crafted fork can cause the contract to compute a lower difficulty target than the real network requires for that fork. The contract would then accept fork blocks that do not meet the real network's difficulty, and if those fork blocks accumulate enough `chain_work` (computed from the accepted, artificially easy `bits`), the contract would trigger a chain reorganization, replacing the legitimate mainchain tip with the attacker's fork. Downstream consumers of `verify_transaction_inclusion` would then verify transactions against a fraudulent chain.

### Likelihood Explanation

This is triggered whenever a fork block is submitted that crosses a Dogecoin difficulty adjustment boundary (every 1 block after height 145,000 in the new protocol, or every `difficulty_adjustment_interval` blocks before). An unprivileged NEAR caller can submit such fork headers via the public `submit_blocks` entry point with no special permissions. The attacker needs to supply a chain of headers with valid PoW at the manipulated (lower) target, which is feasible if the manipulated target is sufficiently reduced.

### Recommendation

Replace the `get_header_by_height` lookup with an ancestry walk that follows `get_prev_header` from `prev_block_header` backward by `blocks_to_go_back` steps. This ensures the retarget boundary block is the true ancestor on the fork chain, not the mainchain block at the same height. The same audit should be applied to `contract/src/bitcoin.rs` and `contract/src/litecoin.rs` if they use the same pattern to obtain `first_block_time`.

### Proof of Concept

1. The mainchain is at height H, with a retarget boundary at height H - N (where N = `blocks_to_go_back`). The mainchain block at height H - N has timestamp `T_main`.
2. An attacker submits a fork starting at height H - N - 1 (before the retarget boundary), with a fork block at H - N having timestamp `T_fork` where `T_fork >> T_main`.
3. When the attacker submits a fork block at height H + 1 (crossing the retarget boundary), `get_next_work_required` calls `get_header_by_height(H - N)`, which returns the **mainchain** block with timestamp `T_main`, not the fork's block with `T_fork`.
4. `actual_time_taken = prev_block_time - T_main` is computed instead of `prev_block_time - T_fork`.
5. If `T_main` is large (old block), `actual_time_taken` is small, and the new target is scaled down (easier). The contract accepts fork blocks at this artificially easy difficulty.
6. The attacker mines subsequent fork blocks at the reduced difficulty, accumulates `chain_work > mainchain_chain_work`, and triggers `reorg_chain`, replacing the legitimate tip. [4](#0-3)
