### Title
Post-Trade Health Invariant Never Enforced Due to Stub `isHealthy()` — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.matchOrders()` calls `isHealthy()` on both taker and maker after applying balance deltas, but the `isHealthy()` implementation unconditionally returns `true` without performing any actual health evaluation. This is a direct analog to the Aftermath Finance missing invariant check: a post-operation safety assertion exists in code but is never actually evaluated, leaving the protocol's core solvency invariant unenforced after every trade settlement.

---

### Finding Description

After applying balance updates to both taker and maker in `matchOrders()`, the contract calls:

```solidity
require(isHealthy(taker.order.sender), ERR_INVALID_TAKER);
require(isHealthy(maker.order.sender), ERR_INVALID_MAKER);
``` [1](#0-0) 

However, the `isHealthy()` function is defined as:

```solidity
function isHealthy(
    bytes32 /* subaccount */
) internal view virtual returns (bool) {
    return true;
}
``` [2](#0-1) 

The function is `virtual` but no contract in the deployed scope overrides it. The `require` statements at lines 826–827 are structurally present as invariant guards but are permanently vacuous — they can never revert. No other post-trade health check exists in `matchOrders()` after `_updateBalances()` is called for both parties. [3](#0-2) 

The correct post-trade check would call `clearinghouse.getHealth(subaccount, HealthType.INITIAL) >= 0`, as is done in `withdrawCollateral`, `mintNlp`, `burnNlp`, and `transferQuote` in `Clearinghouse.sol`. [4](#0-3) 

---

### Impact Explanation

After `_updateBalances()` applies `amountDelta` and `quoteDelta` to both taker and maker, neither party's resulting health is verified on-chain. A trade that leaves a taker or maker below their initial margin requirement — or even with a deeply negative quote balance — will be accepted and committed to state. The protocol's core solvency invariant (`health >= 0` post-trade) is entirely absent from the settlement path.

Corrupted state: `spotEngine.balances` and `perpEngine.balances` for both taker and maker can be left in an undercollateralized configuration with no on-chain rejection.

---

### Likelihood Explanation

`matchOrders` is `onlyEndpoint`, so the sequencer controls which orders are submitted. However, the `isHealthy` check is specifically the on-chain safety net designed to be enforced regardless of sequencer behavior. Its absence means:

1. Any sequencer error that produces a balance-decreasing trade for a party is not caught.
2. A sequencer that submits a batch where one side of a trade is already near-insolvent will not be rejected by the contract.
3. The protocol's stated defense-in-depth (on-chain health enforcement) is silently absent for every single matched order.

The sequencer is a realistic operational actor, and the stub removes the only on-chain check on post-trade solvency.

---

### Recommendation

Override `isHealthy()` with a real implementation that calls `clearinghouse.getHealth()`, or replace the stub calls with explicit health checks inline in `matchOrders()`, consistent with how other state-mutating operations in `Clearinghouse.sol` enforce health post-update:

```solidity
function isHealthy(bytes32 subaccount) internal virtual returns (bool) {
    return clearinghouse.getHealth(subaccount, IProductEngine.HealthType.INITIAL) >= 0;
}
```

---

### Proof of Concept

1. Taker holds a position near initial margin.
2. Sequencer submits a `MatchOrdersWithSigner` transaction that increases the taker's base position beyond their collateral capacity.
3. `_updateBalances()` applies the delta, leaving taker's health negative.
4. `require(isHealthy(taker.order.sender), ERR_INVALID_TAKER)` evaluates `isHealthy()` which returns `true` unconditionally.
5. The trade is committed. The taker is now undercollateralized with no on-chain rejection. [2](#0-1) [3](#0-2)
