### Title
Linked Signer Can Hijack Subaccount Authority by Self-Replacing via `LinkSigner` Fast Path — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` fast-path transaction in `EndpointTx.processTransactionImpl` passes `allowLinkedSigner = true` to `validateSignedTx`, meaning the **current linked signer** can sign a `LinkSigner` transaction and overwrite itself with an attacker-controlled address. Only the subaccount owner should be permitted to change the linked signer, as the slow-mode path enforces. This is a direct authorization deviation analogous to the reported ERC-721 `approve()` bug: a privileged-but-subordinate role is granted authority over a state mutation that should be reserved for the principal owner.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` fast-path branch is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes `getLinkedSigner(sender)` as the permitted alternate signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

This means the **existing linked signer** is a valid signer for a transaction that **replaces the linked signer**. The linked signer can therefore point `linkedSigners[subaccount]` at any address it chooses.

By contrast, the slow-mode `LinkSigner` path uses `validateSender`, which enforces that only the address embedded in the subaccount bytes (i.e., the owner) can submit the transaction:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

`validateSender` requires `address(uint160(bytes20(txSender))) == sender`, where `sender` is `msg.sender` — the on-chain caller, not a signature-recovered address: [4](#0-3) 

The two paths are therefore **inconsistent**: slow mode restricts `LinkSigner` to the owner; fast mode allows the linked signer to execute it.

---

### Impact Explanation

Once a linked signer is compromised, the attacker can:

1. Sign a fast-path `LinkSigner` transaction pointing `linkedSigners[victim_subaccount]` to an attacker-controlled address.
2. Immediately use that new linked signer to sign a `WithdrawCollateral` (V1) or `WithdrawCollateralV2` (with `sendTo == address(0)`) transaction — both of which also pass `allowLinkedSigner = true`: [5](#0-4) 

3. Drain the subaccount's collateral before the owner can react.

The owner's only recovery path is a slow-mode `LinkSigner` transaction, which is subject to a hardcoded `SLOW_MODE_TX_DELAY` (three days). During that window the attacker retains full signing authority and can execute withdrawals, trades, and liquidations on behalf of the victim.

---

### Likelihood Explanation

Linked signers are by design hot wallets or automated trading bots — classes of keys with materially higher compromise probability than the cold-wallet owner key. The attack requires only a single sequencer-submitted transaction and no special privileges beyond possession of the linked signer's private key. Any user who has ever called `LinkSigner` is exposed.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the fast-path `LinkSigner` branch so that only the subaccount owner's key can authorize a linked-signer change:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the owner may change the linked signer
);
```

This aligns the fast path with the slow-mode path, which already enforces owner-only authorization via `validateSender`.

---

### Proof of Concept

1. Alice owns subaccount `alice_subaccount` and sets her linked signer to `hot_wallet` via a slow-mode `LinkSigner` transaction.
2. `hot_wallet` is compromised by an attacker.
3. Attacker constructs a fast-path `SignedLinkSigner` transaction:
   - `signedTx.tx.sender = alice_subaccount`
   - `signedTx.tx.signer = bytes32(uint256(uint160(attacker_address)))`
   - Signs with `hot_wallet`'s private key.
4. Sequencer submits the transaction; `validateSignedTx(..., true)` recovers `hot_wallet` from the signature, matches it against `getLinkedSigner(alice_subaccount)` → passes.
5. `linkedSigners[alice_subaccount]` is now `attacker_address`.
6. Attacker immediately signs a `WithdrawCollateral` fast-path transaction with `attacker_address`, draining Alice's collateral before the 3-day slow-mode recovery window can close.
