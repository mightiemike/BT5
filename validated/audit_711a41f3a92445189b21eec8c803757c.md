### Title
`get_sortable_validator_online_ratio` ignores `endorsement_cutoff_threshold`, causing incorrect kickout-exemption ordering for chunk-validator-only validators — (File: `chain/epoch-manager/src/validator_stats.rs`)

---

### Summary

`get_sortable_validator_online_ratio` always calls `get_validator_online_ratio` with `endorsement_cutoff_threshold = None`, while the actual kickout decision and reward calculation both use `Some(chunk_validator_only_kickout_threshold)`. This creates a domain-invariant divergence: the sort used to determine which validators are **exempted** from kickout uses the raw endorsement ratio, while the kickout decision and reward gate use a binarized (0 or 1) endorsement ratio. A chunk-validator-only validator with an endorsement ratio just below the threshold can exploit this to rank higher in the exemption sort than validators with a lower raw ratio, avoiding kickout and continuing to receive epoch rewards it should not receive.

---

### Finding Description

**Kickout-exemption sort path** (`chain/epoch-manager/src/lib.rs`, lines 500–506):

```rust
let mut sorted_validators = validator_block_chunk_stats
    .iter()
    .map(|(account, stats)| (get_sortable_validator_online_ratio(stats), account))
    .collect_vec();
sorted_validators.sort_by(validator_comparator);
let accounts_sorted_by_online_ratio = ...;
```

`get_sortable_validator_online_ratio` is defined as:

```rust
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);   // ← always None
    ...
}
```

`None` means `get_endorsement_ratio` returns the raw `(produced, expected)` pair — no binarization.

**Kickout decision path** (`chain/epoch-manager/src/lib.rs`, lines 541–554):

```rust
let chunk_validator_only =
    stats.block_stats.expected == 0 && stats.chunk_stats.expected() == 0;
if chunk_validator_only
    && stats.chunk_stats.endorsement_stats()
        .less_than(chunk_validator_only_kickout_threshold)
{
    validator_kickout.entry(...).or_insert_with(|| NotEnoughChunkEndorsements { ... });
}
```

This uses the raw ratio compared against `chunk_validator_only_kickout_threshold` directly.

**Reward calculation path** (`chain/epoch-manager/src/lib.rs`, lines 898–913):

```rust
let online_thresholds = ValidatorOnlineThresholds {
    ...
    endorsement_cutoff_threshold: Some(
        epoch_config.chunk_validator_only_kickout_threshold,
    ),
};
self.reward_calculator.calculate_reward(
    validator_block_chunk_stats,
    ...
    online_thresholds,
    ...
)
```

Here the endorsement ratio is binarized: below threshold → 0, at or above → 1.

**The divergence**: For a chunk-validator-only validator with endorsement ratio `r` where `r < threshold`:

| Path | Effective endorsement value |
|---|---|
| Exemption sort (`get_sortable_validator_online_ratio`) | `r` (raw, e.g. 0.69) |
| Kickout decision | below threshold → kicked out |
| Reward calculation | 0 (binarized) |

The exemption sort ranks this validator at `r = 0.69`, above a validator with `r = 0.0`, even though both are identically "below threshold" from the kickout and reward perspective.

---

### Impact Explanation

`compute_exempted_kickout` uses `accounts_sorted_by_online_ratio` to protect the top-ranked validators from kickout when `validator_max_kickout_stake_perc` would otherwise be exceeded. Because the sort uses raw endorsement ratios, a validator with endorsement ratio just below the threshold (e.g., 69% when threshold is 70%) ranks above a validator with 0% endorsement. The 69% validator is exempted from kickout and continues to receive epoch rewards; the 0% validator is kicked out. From the binarized perspective used by both the kickout gate and the reward gate, both validators have effective endorsement ratio 0 and should be treated identically in the exemption sort. The wrong validator is protected, and the wrong validator is ejected and loses its staking position.

---

### Likelihood Explanation

Triggered whenever: (a) `chunk_validator_only_kickout_threshold > 0` (always true in production configs), (b) `validator_max_kickout_stake_perc` is set below 100 (always true in production), and (c) multiple chunk-validator-only validators fall below the threshold with differing raw endorsement ratios. A validator can deliberately maintain an endorsement ratio just below the threshold to exploit this ordering advantage. All conditions are reachable in normal mainnet operation.

---

### Recommendation

Pass the active `endorsement_cutoff_threshold` into `get_sortable_validator_online_ratio` so the exemption sort uses the same binarized metric as the kickout decision and reward calculation:

```rust
pub(crate) fn get_sortable_validator_online_ratio(
    stats: &BlockChunkValidatorStats,
    endorsement_cutoff_threshold: Option<u8>,   // add parameter
) -> BigRational {
    let ratio = get_validator_online_ratio(stats, endorsement_cutoff_threshold);
    ...
}
```

Call it in `compute_validators_to_reward_and_kickout` with `Some(chunk_validator_only_kickout_threshold)` to match the kickout and reward paths.

### Proof of Concept

Setup:
- `chunk_validator_only_kickout_threshold = 70`
- `validator_max_kickout_stake_perc = 30` (at most 30% of total stake can be kicked out per epoch)
- Two chunk-validator-only validators: **A** (endorsement 69/100, stake 500) and **B** (endorsement 0/100, stake 500)
- Two block+chunk producers with combined stake 2000 (not below any threshold)

Step-by-step:

1. Both A and B are below the 70% threshold → both are candidates for `NotEnoughChunkEndorsements` kickout.
2. Total stake = 3000. 30% = 900. Kicking out both A+B = 1000 > 900, so one must be exempted.
3. `get_sortable_validator_online_ratio` is called with `None`:
   - A: raw ratio = 69/100 = 0.69
   - B: raw ratio = 0/100 = 0.0
4. A is sorted higher → A is placed in `exempted_validators` → A is **not** kicked out.
5. B is kicked out (`NotEnoughChunkEndorsements { produced: 0, expected: 100 }`).
6. In the next epoch, A continues to receive rewards. From the binarized perspective (both have effective endorsement = 0), A and B are identical; the exemption should have been decided by stake or account_id, not by raw endorsement ratio.

**Exact divergent value**: `get_sortable_validator_online_ratio` returns `69/100` for validator A; the correct binarized value used by every other path is `0/1`. This single `None` vs `Some(threshold)` discrepancy at `chain/epoch-manager/src/validator_stats.rs:111` propagates to an incorrect exemption decision at `chain/epoch-manager/src/lib.rs:510–516`. fn get_endorsement_ratio(stats: &ValidatorStats, cutoff_threshold: Option<u8>) -> (u64, u64) {
```rust
    let (numer, denom) = if stats.expected == 0 {
        debug_assert_eq!(stats.produced, 0);
        (0, 0)
    } else if let Some(threshold) = cutoff_threshold {
        if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
    } else {
        (stats.produced, stats.expected)
    };
    (numer, denom)
}
```
