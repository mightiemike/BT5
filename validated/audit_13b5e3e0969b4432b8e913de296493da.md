### Title
Tier `BonusApy` Update Does Not Checkpoint Existing Positions, Causing Retroactive Reward Miscalculation - (`x/tieredrewards/keeper/tier.go`, `x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

When governance updates a tier's `BonusApy` via `SetTier`, existing positions in that tier are not checkpointed first. Because `computeSegmentBonus` applies the **current** `BonusApy` to the entire elapsed segment since the position's last accrual timestamp, the new rate is applied retroactively to time that accrued under the old rate. This is a direct analog to the reserve-factor desynchronization: a rate parameter is changed without flushing accumulated state, corrupting the rewards accounting.

---

### Finding Description

Bonus rewards for tiered positions are computed lazily at claim time. The core formula is in `computeSegmentBonus`: [1](#0-0) 

The function receives the `tier` object fetched from current storage and multiplies `tier.BonusApy` by the full duration `[segmentStart, segmentEnd]`. `segmentStart` is the position's stored `LastBonusAccrual` timestamp, which is only updated when rewards are explicitly claimed or a position-mutating transaction occurs.

The tier itself is written by `SetTier`, which is documented as the entry point for governance messages, genesis, and chain upgrades: [2](#0-1) 

`SetTier` performs no checkpoint of existing positions before overwriting the tier. Contrast this with `deleteTier`, which at least guards against active positions: [3](#0-2) 

No equivalent guard or flush exists for `SetTier`. After a governance proposal updates `BonusApy`, every position in that tier will have its next `computeSegmentBonus` call apply the new rate to the entire period `[LastBonusAccrual, now]`, including the portion that elapsed under the old rate.

---

### Impact Explanation

**If `BonusApy` is increased:** All existing positions retroactively earn the higher rate for the period before the governance change. The `RewardsPoolName` module account is drained by more than the protocol intended, potentially exhausting the pool faster than planned and shortchanging future stakers.

**If `BonusApy` is decreased:** All existing positions retroactively earn the lower rate for the period before the governance change. Stakers who accrued rewards at the old (higher) rate receive less than they are owed â€” a direct loss of earned rewards.

The corrupted value is the `RewardsPoolName` balance and the per-position bonus payout computed from `computeSegmentBonus`. [4](#0-3) 

---

### Likelihood Explanation

Any governance proposal that adjusts a tier's `BonusApy` â€” a routine protocol parameter change â€” triggers this desynchronization. No attacker capability beyond submitting or voting on a governance proposal is required. The entry path is a standard `MsgUpdateTier` (or equivalent) governance transaction, which is a supported production entrypoint. The effect is automatic and silent: no special timing or race condition is needed.

---

### Recommendation

Before writing the updated tier to storage in `SetTier`, iterate over all positions belonging to that tier and force a bonus accrual checkpoint (update `LastBonusAccrual` to the current block time, accumulating any earned bonus at the old rate). This mirrors the fix applied in the referenced report (`pool.updateState()` before changing the reserve factor). Concretely:

1. In `SetTier`, retrieve the existing tier from storage.
2. If `BonusApy` differs, iterate `PositionsByTier` for that tier ID and call the equivalent of `applyBonusAccrualCheckpoint` on each position, persisting the updated `LastBonusAccrual`.
3. Only then write the new tier. [5](#0-4) 

---

### Proof of Concept

1. Governance creates a tier with `BonusApy = 0.05` (5%).
2. Alice locks funds into that tier at `T=0`. Her `LastBonusAccrual = T0`.
3. 180 days pass. Alice has accrued ~2.5% bonus on her position at the 5% rate.
4. Governance passes a proposal calling `SetTier` with `BonusApy = 0.10` (10%). `SetTier` writes the new tier directly; Alice's `LastBonusAccrual` remains `T0`.
5. Alice calls `ClaimTierRewards` at `T=180d`. `computeSegmentBonus` is called with the current tier (`BonusApy=0.10`) and `segmentStart=T0`, `segmentEnd=T180d`.
6. Alice receives ~5% bonus (180 days at 10%) instead of the correct ~2.5% (180 days at 5%) â€” double the intended payout, draining the rewards pool. [2](#0-1) [4](#0-3)

```go
func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
	accrualEnd := blockTime
	if pos.CompletedExitLockDuration(blockTime) {
		accrualEnd = pos.ExitUnlockAt
	}
	pos.UpdateLastBonusAccrual(accrualEnd)
}
```

```go
func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}
```

```go
func (k Keeper) SetTier(ctx context.Context, tier types.Tier) error {
	if err := tier.Validate(); err != nil {
		return err
	}
	if err := k.Tiers.Set(ctx, tier.Id, tier); err != nil {
		return errorsmod.Wrapf(err, "%s (tier id %d)", types.ErrTierStore.Error(), tier.Id)
	}
	return nil
}
```

```go
func (k Keeper) deleteTier(ctx context.Context, tierId uint32) error {
	hasPositions, err := k.hasPositionsForTier(ctx, tierId)
	if err != nil {
		return err
	}
	if hasPositions {
		return types.ErrTierHasActivePositions
	}
	if err := k.Tiers.Remove(ctx, tierId); err != nil {
		if stderrors.Is(err, collections.ErrNotFound) {
			return errorsmod.Wrapf(types.ErrTierNotFound, "tier id %d", tierId)
		}
		return errorsmod.Wrapf(err, "%s (tier id %d)", types.ErrTierStore.Error(), tierId)
	}
	return nil
}
```
