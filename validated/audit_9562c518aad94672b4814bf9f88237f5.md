### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable Multiple Bonus Reward Claims from the Same Position — (`File: x/tieredrewards/keeper/msg_server.go`)

### Summary
`ClaimTierRewards` loads all positions from storage into an in-memory slice **before** processing any of them. Because no deduplication is enforced on `msg.PositionIds`, an attacker who owns a position can include the same position ID multiple times. Each duplicate entry is loaded with the same initial checkpoint state, causing `processEventsAndClaimBonus` to compute and transfer the same bonus amount from the `RewardsPoolName` module account for every duplicate occurrence.

### Finding Description
In `ClaimTierRewards` (`msg_server.go` lines 429–468), all positions are fetched from storage in a first loop and appended to an in-memory slice:

```go
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)   // reads from storage
    ...
    positions = append(positions, pos)
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
``` [1](#0-0) 

`claimRewardsAndUpdatesPositions` then iterates over the in-memory slice. For each element it:
1. Calls `claimBaseRewards` (withdraws staking rewards — returns 0 on the second call since the distribution module has already paid out, but does **not** error).
2. Calls `processEventsAndClaimBonus`, which reads `pos.LastEventSeq` and `pos.LastBonusAccrual` from the in-memory copy to compute and transfer bonus coins from the `RewardsPoolName` module account.
3. Calls `setPosition` to persist the updated checkpoints. [2](#0-1) 

Because both copies of a duplicated position ID were loaded from storage **before** any `setPosition` call, the second copy retains the original `LastEventSeq` and `LastBonusAccrual` values. `processEventsAndClaimBonus` therefore recomputes the identical bonus amount and issues a second `SendCoinsFromModuleToAccount` transfer from the bonus pool: [3](#0-2) 

The only guard against over-payment is `sufficientBonusPoolBalance`, which checks the pool balance at the moment of each individual transfer. After the first transfer the pool balance is reduced, but if the pool holds at least `N × bonus_per_claim` tokens, an attacker who submits the same position ID `N` times can drain the entire pool in a single transaction. [4](#0-3) 

### Impact Explanation
The `RewardsPoolName` module account holds the bonus reward funds for all tiered-rewards participants. A position owner can drain an arbitrary fraction of this pool proportional to their accrued bonus and the number of duplicate IDs they include. With a sufficiently large accrued bonus or a small pool, the entire pool can be emptied in one transaction. The corrupted value is the `RewardsPoolName` module account balance (bond-denom coins).

### Likelihood Explanation
The entry path is a standard, unprivileged `MsgClaimTierRewards` transaction signed by any position owner. No special role, governance action, or external dependency is required. The only precondition is that the position has accrued non-zero bonus rewards, which is the normal operating state for any active tiered-rewards position.

### Recommendation
Deduplicate `msg.PositionIds` before loading positions, or load and immediately save each position inside the same loop iteration (checks-effects-interactions). The simplest fix is to reject the message if `PositionIds` contains duplicates in `msg.Validate()`. A complementary fix is to restructure `claimRewardsAndUpdatesPositions` so that `setPosition` is called before moving to the next position, and to re-read the position from storage rather than relying on the pre-loaded in-memory slice.

### Proof of Concept
1. Attacker owns position with ID `42` that has accrued `100 BASECRO` in bonus rewards.
2. Attacker submits `MsgClaimTierRewards { owner: attacker, position_ids: [42, 42, 42, ..., 42] }` with `N` copies of `42`.
3. `ClaimTierRewards` loads `N` copies of position `42` from storage, all with the same `LastEventSeq` and `LastBonusAccrual`.
4. `claimRewardsAndUpdatesPositions` iterates: for each copy, `processEventsAndClaimBonus` computes `100 BASECRO` and calls `SendCoinsFromModuleToAccount(RewardsPoolName, attacker, 100 BASECRO)`.
5. After `N` iterations, attacker has received `N × 100 BASECRO` from the bonus pool while only legitimately entitled to `100 BASECRO`.
6. The final `setPosition` call saves the same updated checkpoint state that the first iteration already saved — the position appears "claimed" exactly once on-chain, but `N` transfers occurred.

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
