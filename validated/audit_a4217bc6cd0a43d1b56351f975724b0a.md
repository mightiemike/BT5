### Title
`ClearExit` Discards Unclaimed Bonus Rewards by Advancing `LastBonusAccrual` Past `ExitUnlockAt` — (`File: x/tieredrewards/types/position_state.go`)

### Summary
When a user's tiered-rewards position has its exit cleared (e.g., by re-locking after triggering exit), the `ClearExit` function unconditionally sets `LastBonusAccrual = blockTime`. If the user has not yet claimed their bonus rewards that accrued between the previous `LastBonusAccrual` and `ExitUnlockAt`, those rewards are permanently skipped and lost. The `computeSegmentBonus` formula already caps accrual at `ExitUnlockAt`, so the defensive reset is both unnecessary and destructive to the user's unclaimed yield.

### Finding Description

The `ClearExit` method in `x/tieredrewards/types/position_state.go` resets the position's `LastBonusAccrual` timestamp to the current block time: [1](#0-0) 

The in-code comment reads: *"required so that positions who clear exit after exit lock duration won't have extra bonus accrued."* However, `computeSegmentBonus` already enforces this cap independently — it truncates the segment end to `ExitUnlockAt` when computing the bonus: [2](#0-1) 

Because `computeSegmentBonus` already prevents any bonus from accruing after `ExitUnlockAt`, the reset in `ClearExit` is redundant for preventing over-accrual. But it is **destructive** for unclaimed rewards: it advances `LastBonusAccrual` from its previous value (somewhere before `ExitUnlockAt`) all the way to `blockTime` (which is after `ExitUnlockAt`). The segment `[LastBonusAccrual, ExitUnlockAt]` — which contains legitimate, earned, but unclaimed bonus — is permanently skipped.

The lazy-processing pattern means rewards are only computed when a position is "touched." If the user does not claim before calling the message that triggers `ClearExit`, the reward catch-up loop will start from the new (post-`ExitUnlockAt`) `LastBonusAccrual` and compute zero bonus for all future segments up to `ExitUnlockAt`, because that window is now in the past and was never processed. [1](#0-0) 

### Impact Explanation

A user who:
1. Locks a position in a tier,
2. Triggers exit (setting `ExitUnlockAt`),
3. Waits for `ExitUnlockAt` to pass without claiming bonus rewards,
4. Then re-locks or clears the exit via the supported message path,

will permanently lose all bonus rewards that accrued between their last claim and `ExitUnlockAt`. These rewards are backed by the module's bonus pool and were legitimately earned. The loss is irreversible because `LastBonusAccrual` is a write-once-forward field — once advanced past `ExitUnlockAt`, the skipped segment can never be replayed.

This matches the external report's impact: **permanent freezing of unclaimed yield** for a user who performs a supported state-change operation on their position.

### Likelihood Explanation

The trigger is a normal, unprivileged user action (clearing/re-locking a position after exit). Any user who does not claim rewards before clearing their exit will silently lose them. There is no warning in the protocol, and the lazy-processing design makes it non-obvious that a claim must precede the state change.

### Recommendation

Remove the `LastBonusAccrual = blockTime` assignment from `ClearExit`. The `computeSegmentBonus` function already caps accrual at `ExitUnlockAt`, making the reset unnecessary for correctness. Alternatively, force a reward claim (flush all pending bonus up to `ExitUnlockAt`) before advancing `LastBonusAccrual`, mirroring how `MsgTierRedelegate` claims rewards before moving the delegation. [3](#0-2) 

### Proof of Concept

1. User locks 10,000 CRO in a 1-year tier at time `T=0`. `ExitUnlockAt` is unset; `LastBonusAccrual = T`.
2. At `T=6months`, user triggers exit. `ExitUnlockAt = T+6months+ExitDuration`.
3. User does **not** claim bonus rewards.
4. At `T=6months+ExitDuration+1day` (after `ExitUnlockAt`), user calls the re-lock/clear-exit message.
5. `ClearExit` fires: `LastBonusAccrual` is set to `blockTime ≈ T+6months+ExitDuration+1day`.
6. User calls `MsgClaimTierRewards`. The reward catch-up loop starts from `LastBonusAccrual` (after `ExitUnlockAt`) and computes zero bonus — `computeSegmentBonus` caps at `ExitUnlockAt`, which is now in the past.
7. The 6 months of accrued bonus (segment `[T, ExitUnlockAt]`) is permanently lost. [1](#0-0) [4](#0-3)

func (p *PositionState) ClearExit(blockTime time.Time) {
	p.ExitTriggeredAt = time.Time{}
	p.ExitUnlockAt = time.Time{}
	if p.IsDelegated() {
		// required so that positions who clear exit after exit lock duration won't have extra bonus accrued
		p.LastBonusAccrual = blockTime
	}
}

// TestComputeSegmentBonus_CappedAtExitUnlockAt verifies that when segmentEnd
// exceeds ExitUnlockAt, the bonus is capped at ExitUnlockAt — not the full
// segment duration. This prevents bonus accrual after the exit lock expires.
func (s *KeeperSuite) TestComputeSegmentBonus_CappedAtExitUnlockAt() {
	s.setupTier(1)
	tier, _ := s.keeper.Tiers.Get(s.ctx, 1)
	tier.BonusApy = sdkmath.LegacyNewDecWithPrec(4, 2) // 4%

	now := s.ctx.BlockTime()
	tokensPerShare := sdkmath.LegacyNewDec(1)
	exitUnlockAt := now.Add(10 * 24 * time.Hour) // 10 days from now

	pos := types.PositionState{
		Position: types.Position{
			ExitUnlockAt: exitUnlockAt,
		},
		Delegation: &stakingtypes.Delegation{Shares: sdkmath.LegacyNewDec(1000)},
	}

	// Segment that extends 365 days past ExitUnlockAt.
	segmentEnd := exitUnlockAt.Add(365 * 24 * time.Hour)
	bonusCapped := s.keeper.ComputeSegmentBonus(pos, tier, now, segmentEnd, tokensPerShare)

	// Compare against exact 10-day bonus (capped at ExitUnlockAt).
	bonusExact := s.keeper.ComputeSegmentBonus(pos, tier, now, exitUnlockAt, tokensPerShare)

	s.Require().Equal(bonusExact.String(), bonusCapped.String(),
		"bonus should be capped at ExitUnlockAt regardless of segment end; capped=%s, exact=%s",
		bonusCapped, bonusExact)
	s.Require().True(bonusCapped.IsPositive(), "capped bonus should still be positive")

	// Segment that starts AFTER ExitUnlockAt should yield zero.
	bonusAfter := s.keeper.ComputeSegmentBonus(pos, tier, exitUnlockAt, segmentEnd, tokensPerShare)
	s.Require().True(bonusAfter.IsZero(),
		"segment starting at ExitUnlockAt should yield zero bonus")
}
