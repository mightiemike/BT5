### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Guard — (`contract/src/lib.rs`)

---

### Summary

Two separate public entry points exist for transaction inclusion verification. The newer path (`verify_transaction_inclusion_v2`) enforces a coinbase Merkle proof check that prevents the 64-byte internal-node forgery attack. The older path (`verify_transaction_inclusion`, v1) does not enforce this check and remains fully callable by any unprivileged NEAR account. An attacker can call v1 directly, supply an internal Merkle-tree node as `tx_id`, and receive a `true` result — a forged proof of inclusion.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability (documented inline and in the deprecation notice). It enforces two additional invariants that v1 lacks:

1. **Equal proof depths**: `merkle_proof.len() == coinbase_merkle_proof.len()` — forces the target transaction to sit at the same tree depth as the coinbase (i.e., a leaf node).
2. **Coinbase anchor**: the coinbase transaction must hash to the block's `merkle_root` at position 0, anchoring the depth constraint to a known-valid leaf. [1](#0-0) 

v1 enforces neither constraint. Its only Merkle check is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

The code itself acknowledges the gap:

> *"This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."* [3](#0-2) 

Despite the `#[deprecated]` annotation, v1 carries no `#[private]` guard and is decorated only with `#[pause]`. When the contract is not paused, any NEAR account can invoke it. [4](#0-3) 

The `#[deprecated]` Rust attribute emits a compiler warning only for Rust callers at compile time; it has no effect on NEAR cross-contract calls or direct RPC invocations.

---

### Impact Explanation

Any consumer contract or off-chain verifier that calls `verify_transaction_inclusion` (v1) to gate an action (e.g., releasing bridged funds, confirming a Bitcoin deposit, authorizing a cross-chain message) can be deceived. An attacker selects any confirmed mainchain block, picks an internal Merkle-tree node `N` at depth `d`, constructs the `d`-step proof from `N` to the root, and submits:

- `tx_id = N` (an internal node, not a real transaction)
- `tx_index = index_of_N_at_depth_d`
- `merkle_proof = [sibling hashes from depth d to root]`

`compute_root_from_merkle_proof` will correctly reconstruct the block's `merkle_root`, so v1 returns `true`. The consumer contract is then tricked into treating a non-existent transaction as confirmed.

The corrupted proof result is the boolean return value of `verify_transaction_inclusion`; any authorization assumption downstream that relies on it is broken.

---

### Likelihood Explanation

- No special privilege is required; any NEAR account can call v1 directly.
- No new block needs to be mined; any existing mainchain block with more than one transaction provides a usable internal node.
- The only prerequisite is a consumer contract that calls v1 rather than v2. Because `#[deprecated]` is invisible to non-Rust callers and the function name is stable, this is a realistic deployment scenario.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or annotate it with `#[private]` so it can only be called by the contract itself (as an internal helper for v2). If backward compatibility must be preserved, add the same coinbase-proof length check and coinbase-anchor validation directly inside v1.

---

### Proof of Concept

**Setup**: A mainchain block `B` with hash `bh` is stored. `B` has 4 transactions; its Merkle tree has 3 internal nodes.

**Attack**:
1. Pick internal node `N = hash(tx0 || tx1)` (the left child of the root).
2. Its Merkle proof to the root is `[hash(tx2 || tx3)]` (one sibling), `tx_index = 0` at depth 1.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = N`
   - `tx_block_blockhash = bh`
   - `tx_index = 0`
   - `merkle_proof = [hash(tx2 || tx3)]`
   - `confirmations = 1`
4. `compute_root_from_merkle_proof(N, 0, [hash(tx2||tx3)])` = `hash(N || hash(tx2||tx3))` = `merkle_root` of `B`. ✓
5. Function returns `true` — a forged proof that the non-existent "transaction" `N` is confirmed in block `B`.

Calling `verify_transaction_inclusion_v2` with the same `tx_id` and a coinbase proof of length 2 would fail the length-equality check (coinbase proof has length 2, but the forged proof has length 1), blocking the attack. [5](#0-4)

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
