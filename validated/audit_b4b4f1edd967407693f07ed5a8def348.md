### Title
Unguarded Hard-Fork Flag Acceptance in `run_serialized_chia_program` Enables Consensus-Divergent Dialect Instantiation — (File: `wheel/src/api.rs`)

---

### Summary

`run_serialized_chia_program` accepts a raw `flags: u32` from any Python caller and converts it with `ClvmFlags::from_bits_truncate`, which silently accepts every bit combination including hard-fork-only flags explicitly documented as "Hard-fork; enable only when it activates." There is no allowlist, no epoch check, and no guard preventing a caller from enabling flags such as `RELAXED_BLS` (0x0008), `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` (0x0100), or `MALACHITE` (0x1000) before the corresponding fork has activated on-chain. Any Python caller can therefore instantiate a `ChiaDialect` whose operator semantics diverge from the live consensus rules, using the same official `clvm_rs` API surface.

---

### Finding Description

In `wheel/src/api.rs`, `run_serialized_chia_program` receives `flags: u32` from the Python caller and immediately converts it:

```rust
let flags = ClvmFlags::from_bits_truncate(flags);   // line 47 — silently drops unknown bits, keeps all known ones
…
let dialect = ChiaDialect::new(flags);               // line 57 — no validation
``` [1](#0-0) 

`from_bits_truncate` is defined by the `bitflags` crate to accept any subset of the declared flag bits without error. The Python module exports only seven constants:

```rust
m.add("NO_UNKNOWN_OPS", …)?;
m.add("LIMIT_HEAP",     …)?;
m.add("MEMPOOL_MODE",   …)?;
m.add("ENABLE_SHA256_TREE", …)?;
m.add("ENABLE_SECP_OPS",    …)?;
m.add("DISABLE_OP",         …)?;
m.add("CANONICAL_INTS",     …)?;
``` [2](#0-1) 

The following hard-fork flags are **not** exported but are fully accepted by the API:

| Flag | Bit | Documentation |
|---|---|---|
| `RELAXED_BLS` | 0x0008 | "Hard-fork; enable only when it activates" |
| `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` | 0x0100 | "Hard-fork; enable only when it activates" |
| `ENABLE_SHA256_TREE` | 0x0400 | (exported, but still a hard-fork flag) |
| `MALACHITE` | 0x1000 | Switches bigint backend for div/divmod/mod/modpow | [3](#0-2) 

The most concrete broken invariant is `RELAXED_BLS`. When that bit is set, `op_bls_g1_negate` and `op_bls_g2_negate` skip the on-curve point validation that is mandatory under current consensus:

```rust
// src/bls_ops.rs  op_bls_g1_negate
let strict = !flags.contains(ClvmFlags::RELAXED_BLS);
…
if strict {
    a.validate_g1(point, blob)?;   // skipped when RELAXED_BLS is set
}
``` [4](#0-3) 

```rust
// src/bls_ops.rs  op_bls_g2_negate
let strict = !flags.contains(ClvmFlags::RELAXED_BLS);
…
if strict {
    a.validate_g2(point, blob)?;   // skipped when RELAXED_BLS is set
}
``` [5](#0-4) 

The root cause is structurally identical to the external report: there is no `GlobalConfig`-equivalent allowlist that restricts which flag combinations are legal for a given caller or epoch. `ChiaDialect::new` accepts any `ClvmFlags` value unconditionally:

```rust
pub fn new(flags: ClvmFlags) -> ChiaDialect {
    ChiaDialect { flags }
}
``` [6](#0-5) 

---

### Impact Explanation

A Python caller — wallet, mempool checker, block pre-validator, or any downstream library — can pass `flags = 0x0008` (`RELAXED_BLS`) to `run_serialized_chia_program` before the corresponding hard-fork activates. Under that dialect:

- `bls_g1_negate` and `bls_g2_negate` accept and negate **invalid** G1/G2 points (wrong subgroup, non-canonical encoding, etc.) that the live consensus would reject with an error.
- A CLVM spend that embeds such a point and calls `bls_g1_negate` will **succeed** (return a `NodePtr` result) under the caller's dialect but **fail** under the official consensus dialect.

This is a concrete consensus-divergence: the same attacker-supplied CLVM bytes produce different Boolean outcomes depending solely on which `flags` value the Python caller chose. A wallet that evaluates coin spends with `RELAXED_BLS` will accept spends that full nodes reject, enabling double-spend or fund-loss scenarios. A mempool that uses this flag will relay transactions that are invalid on-chain.

The `MALACHITE` flag (0x1000) introduces a second divergence path: switching the bigint backend for `div`, `divmod`, `mod`, and `modpow` can produce different results for edge-case inputs (e.g., division rounding of negative numbers), again without any guard.

---

### Likelihood Explanation

The integer value `0x0008` is not secret; it is visible in `src/chia_dialect.rs`. Any developer reading the source to understand available flags will discover it. The Python API provides no documentation warning that certain bit values are fork-gated. A developer building a wallet or mempool tool who wants "relaxed" BLS handling (e.g., for testing) can trivially pass the raw integer. The attack surface is the entire Python ecosystem that wraps `clvm_rs`.

---

### Recommendation

Add a flag-validation step in `run_serialized_chia_program` (and any future Python-facing entry points) that rejects flag combinations containing bits not in an explicitly maintained allowlist of currently-active flags. Hard-fork flags should be gated behind a separate, clearly named API or a version/epoch parameter that the caller must explicitly opt into, mirroring the `GlobalConfig` / `authorized_initializer` pattern recommended in the external report.

```rust
// Example guard
const CALLER_ALLOWED_FLAGS: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::LIMIT_SOFTFORK)
    .union(ClvmFlags::ENABLE_SHA256_TREE)
    .union(ClvmFlags::ENABLE_SECP_OPS);

if flags.difference(CALLER_ALLOWED_FLAGS) != ClvmFlags::empty() {
    return Err(PyValueError::new_err("flags contain reserved hard-fork bits"));
}
```

### Proof of Concept

```python
import clvm_rs
from clvm_rs.serde import node_to_bytes  # illustrative

# RELAXED_BLS = 0x0008 — not exported, but silently accepted
RELAXED_BLS = 0x0008

# Serialized CLVM program: (bls_g1_negate <48-byte invalid G1 point>)
# opcode 51 = bls_g1_negate; the point has correct length but is off-curve.
invalid_g1 = b'\xc0' + b'\xff' * 47          # compressed-flag set, garbage body
program_bytes = bytes([...])                  # (51 . (invalid_g1 . ()))  serialized

# Under official consensus flags (no RELAXED_BLS) this raises an error:
try:
    clvm_rs.run_serialized_chia_program(program_bytes, b'\x80', 10**9, flags=0)
except Exception as e:
    print("consensus rejects:", e)            # EvalErr: invalid G1 point

# Under the unguarded dialect this succeeds — consensus divergence:
cost, result = clvm_rs.run_serialized_chia_program(
    program_bytes, b'\x80', 10**9, flags=RELAXED_BLS
)
print("spoofed dialect accepts, cost=", cost) # returns negated bytes, no error
```

The attacker supplies the CLVM bytes; the unguarded `flags` parameter is the missing allowlist check that makes the same bytes produce opposite outcomes across callers.
