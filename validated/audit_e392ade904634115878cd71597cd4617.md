Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the actual caller (`sender`) and gates access only on the `owner` parameter — the LP share recipient — which any caller may set to an arbitrary address. Because `addLiquidity` imposes no restriction on who may supply an arbitrary `owner` value (unlike `removeLiquidity`, which enforces `msg.sender == owner`), any unprivileged address can bypass the deposit allowlist entirely by naming an allowlisted address as `owner`.

## Finding Description

`MetricOmmPool.addLiquidity` invokes the hook with `msg.sender` as the first argument and the caller-supplied `owner` as the second: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` discards the first argument entirely (unnamed `address`) and checks only `owner`: [2](#0-1) 

Inside the hook, `msg.sender` is the pool address (the extension is called by the pool), so `allowedDepositor[msg.sender][owner]` resolves to `allowedDepositor[pool][owner]`. Any caller who names an allowlisted address as `owner` passes this check unconditionally.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly captures and checks the first parameter (`sender`): [3](#0-2) 

`removeLiquidity` enforces `msg.sender == owner`, closing the analogous path there: [4](#0-3) 

`addLiquidity` has no such guard, leaving the `owner` field freely settable by any caller: [5](#0-4) 

## Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting which addresses may inject liquidity (e.g., KYC/AML gating, curated market-maker whitelists, regulatory compliance). With this bug, the allowlist is rendered entirely ineffective: any address can deposit tokens into a restricted pool by naming any allowlisted address as `owner`. This constitutes an admin-boundary break — an unprivileged path bypasses a pool admin access-control gate — and also enables forced LP positions on the named `owner` (unwanted exposure, accounting complications, positions they must actively unwind). This meets the "admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" criterion in the allowed impact gate.

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with a known allowlisted address as `owner`. The `allowedDepositor` mapping is public, so any observer can identify allowlisted addresses. No special privileges, flash loans, oracle manipulation, or unusual conditions are required. The attack is immediately repeatable by any EOA or contract.

## Recommendation

Change `beforeAddLiquidity` to check the first parameter (the actual caller) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
// Fixed:
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Also update `setAllowedToDeposit`, `isAllowedToDeposit`, and the `allowedDepositor` mapping semantics to reflect that the allowlisted entity is the caller, not the LP share recipient.

## Proof of Concept

1. Pool `P` is deployed with `DepositAllowlistExtension`. `allowAllDepositors[P] = false`.
2. Admin calls `setAllowedToDeposit(P, alice, true)`. Alice is the only allowlisted depositor.
3. Bob (not allowlisted) calls `P.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(bob, alice, ...)` → extension's `beforeAddLiquidity(bob, alice, ...)`.
5. Extension discards `bob`; checks `allowedDepositor[P][alice]` → `true` → no revert.
6. Bob's swap callback transfers tokens into the pool; Alice receives LP shares she never requested.
7. Bob has deposited into a restricted pool without being on the allowlist.
