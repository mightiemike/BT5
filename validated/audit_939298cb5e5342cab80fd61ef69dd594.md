### Title
`SignedWithdrawCollateralV2.feeX18` Slippage Check Bypassed in Fast Withdrawal Path — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

The `WithdrawCollateralV2` transaction type includes a `feeX18` field in `SignedWithdrawCollateralV2` that functions as a user-signed maximum-fee slippage guard. In the normal sequencer path (`EndpointTx.processTransactionImpl`), this field is enforced: the contract verifies `signedTx.feeX18 <= currentFeeX18` and charges exactly `signedTx.feeX18`. In the fast withdrawal path (`BaseWithdrawPool.submitFastWithdrawal`), the signed `feeX18` is silently discarded. The fee is instead recomputed at execution time via `fastWithdrawalFeeAmount()`, which always produces a value at least 5× the normal `withdrawFeeX18`. Because the user's signed `feeX18` must be `<= withdrawFeeX18`, the fast withdrawal path always charges more than the user agreed to, with no bound the user can enforce.

---

### Finding Description

`SignedWithdrawCollateralV2` carries a `feeX18` field:

```solidity
struct SignedWithdrawCollateralV2 {
    WithdrawCollateralV2 tx;
    CompactSignature signature;
    int128 feeX18;   // user-signed maximum fee
}
``` [1](#0-0) 

In the sequencer path, `EndpointTx.processTransactionImpl` enforces this field as a slippage check:

```solidity
int128 currentFeeX18 = spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18;
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
chargeFee(signedTx.tx.sender, signedTx.feeX18, signedTx.tx.productId);
``` [2](#0-1) 

The user pays exactly what they signed, bounded by the current protocol fee.

In the fast withdrawal path, `resolveFastWithdrawal` decodes the same `WithdrawCollateralV2` transaction but returns only `(productId, sendTo, amount)` — the `feeX18` field is never read:

```solidity
if (txType == IEndpoint.TransactionType.WithdrawCollateralV2) {
    IEndpoint.SignedWithdrawCollateralV2 memory signedTx = abi.decode(...);
    // V2 appendix is intentionally ignored until fast-withdraw features use it.
    ...
    return (signedTx.tx.productId, resolvedSendTo, signedTx.tx.amount);
    // signedTx.feeX18 is silently dropped
}
``` [3](#0-2) 

`submitFastWithdrawal` then recomputes the fee from scratch at execution time:

```solidity
int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);
``` [4](#0-3) 

`fastWithdrawalFeeAmount` computes:

```solidity
int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);  // 0.1% of amount
int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;
int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
``` [5](#0-4) 

The minimum floor of the fast withdrawal fee is `5 × withdrawFeeX18`. Because the sequencer path enforces `signedTx.feeX18 <= withdrawFeeX18`, the fast withdrawal fee is structurally always at least 5× what the user signed. The user's signed slippage constraint is never consulted.

---

### Impact Explanation

A user signs a `WithdrawCollateralV2` transaction with `feeX18 = F` (where `F <= withdrawFeeX18`). An unprivileged LP (or the user themselves) submits this via `submitFastWithdrawal`. The contract charges `max(amount × 0.1%, 5 × withdrawFeeX18)` instead of `F`. The user receives `transferAmount − fastFee` on-chain, which is strictly less than `transferAmount − F`. The user's signed fee bound — the only protection they have against excessive fee extraction — is bypassed entirely. The asset delta is the difference between the fast fee and the signed fee, which can be substantial for large withdrawals or high `withdrawFeeX18` values. [6](#0-5) 

---

### Likelihood Explanation

Every `WithdrawCollateralV2` fast withdrawal triggers this. The `submitFastWithdrawal` function is `public` with no access control — any address can call it. The signed transaction is already on-chain (submitted to the sequencer or slow-mode queue), so the calldata is available to any observer. The discrepancy between the signed fee and the charged fee is not a corner case; it is the structural difference between the normal and fast withdrawal fee schedules. [7](#0-6) 

---

### Recommendation

In `resolveFastWithdrawal`, return `feeX18` alongside the other fields for `WithdrawCollateralV2` transactions. In `submitFastWithdrawal`, after computing `fastWithdrawalFeeAmount`, assert that the computed fee does not exceed the user's signed `feeX18`:

```solidity
// After resolveFastWithdrawal returns signedFeeX18 for V2 transactions:
require(fee <= uint128(signedFeeX18), "Fee exceeds user-signed maximum");
```

This mirrors the enforcement already present in `EndpointTx.processTransactionImpl` and ensures the fast withdrawal path respects the user's signed slippage constraint. [8](#0-7) 

---

### Proof of Concept

1. User signs `WithdrawCollateralV2` with `amount = 1000 USDC`, `feeX18 = 1e18` (= $1, the current `withdrawFeeX18`).
2. The signature is broadcast; the sequencer would charge exactly $1.
3. An LP calls `submitFastWithdrawal(idx, transaction, signatures)` before the sequencer processes it.
4. `resolveFastWithdrawal` returns `(productId, sendTo, 1000e6)` — `feeX18 = 1e18` is dropped.
5. `fastWithdrawalFeeAmount` computes `max(1000e6 × 0.001, 5 × 1e18 / 1e12)` = `max(1e6, 5e6)` = `5e6` (= $5).
6. The user receives `1000e6 − 5e6 = 995 USDC` instead of the `999 USDC` their signed `feeX18` guaranteed.
7. The user's signed slippage bound of $1 was bypassed; they paid $5 instead. [9](#0-8) [10](#0-9)

    function fastWithdrawalFeeAmount(
        IERC20Base token,
        uint32 productId,
        uint128 amount
    ) public view returns (int128) {
        uint8 decimals = token.decimals();
        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
        int128 amountX18 = int128(amount) * int128(multiplier);

        int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
        int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;

        int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
        return feeX18 / int128(multiplier);
    }
