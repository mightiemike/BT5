### Title
Valid SPV Proofs for Single-Transaction Blocks Are Always Rejected — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2` which delegates to it) unconditionally rejects an empty `merkle_proof` with a hard `require!`. In Bitcoin's merkle tree, an empty proof is the **only correct proof** when a block contains exactly one transaction (the merkle root equals the transaction hash). This is a direct structural analog to the DSClock bug: a sentinel/degenerate value that is semantically valid is rejected by a blanket condition check, making a documented feature permanently unreachable.

---

### Finding Description

In Bitcoin, when a block contains a single transaction, the merkle root is exactly that transaction's hash. No sibling hashes exist, so the valid merkle proof is an empty list. The contract's `verify_transaction_inclusion` function enforces:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This check is unconditional. It fires before the merkle root comparison, so no empty proof can ever reach the actual verification logic, regardless of whether it is mathematically correct.

`verify_transaction_inclusion_v2` does not add its own empty-proof guard — it passes through to `verify_transaction_inclusion` via `args.into()`:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [2](#0-1) 

So both public entry points are affected. The `ProofArgs` and `ProofArgsV2` structs accept `merkle_proof: Vec<H256>` with no minimum-length constraint at the type level: [3](#0-2) 

The final comparison that would succeed for a single-transaction block is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [4](#0-3) 

This comparison would return `true` (since `compute_root_from_merkle_proof(tx_id, 0, &[])` returns `tx_id`, which equals the merkle root for a single-transaction block), but it is never reached because the `require!` at line 315 panics first.

---

### Impact Explanation

Any unprivileged NEAR caller — a bridge contract, an escrow, or a direct user — that submits a structurally valid SPV proof for a Bitcoin transaction in a single-transaction block will always receive a contract panic. The verification result is permanently `false` (via panic) for an entire class of valid Bitcoin blocks. Applications that gate financial operations (cross-chain transfers, conditional payments) on the result of `verify_transaction_inclusion_v2` cannot process transactions from coinbase-only blocks, locking out legitimate users.

---

### Likelihood Explanation

Coinbase-only blocks are a normal occurrence in Bitcoin, particularly during periods of low mempool activity or immediately after a difficulty adjustment. Any relayer or user who attempts to prove inclusion of a coinbase transaction in such a block will trigger this bug deterministically. No special attacker capability is required — the caller only needs to supply a valid block hash and an empty proof list, both of which are publicly available on-chain data.

---

### Recommendation

Before the `is_empty` guard, add a short-circuit for the single-transaction case:

```rust
if args.merkle_proof.is_empty() {
    return args.tx_id == header.block_header.merkle_root;
}
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

Alternatively, remove the `require!` entirely and let `compute_root_from_merkle_proof` handle the empty-proof case (returning the leaf itself), then rely solely on the root comparison. Add a unit test that submits a single-transaction block and verifies the coinbase with an empty proof.

---

### Proof of Concept

1. A Bitcoin block is mined containing only the coinbase transaction with hash `T`. Its merkle root is therefore `T`.
2. The relayer submits this block header via `submit_blocks`; the contract stores it.
3. A user calls `verify_transaction_inclusion_v2` with:
   - `tx_id = T`
   - `tx_block_blockhash = <block hash>`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = T`
   - `coinbase_merkle_proof = []`
   - `confirmations = 0`
4. The length check passes: `0 == 0`.
5. The coinbase proof check passes: `compute_root_from_merkle_proof(T, 0, &[]) == T == merkle_root`.
6. `verify_transaction_inclusion` is called.
7. **Line 315 panics**: `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")`.
8. The valid, mathematically correct proof is rejected. The caller's application cannot proceed.
