Audit Report

## Title
SwapAllowlistExtension Bypassed via Router: Any User Can Swap in Allowlist-Gated Pool — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router` and forwards the router address as `sender` to the extension. If the pool admin allowlists the router to enable router-mediated swaps, every unprivileged user can bypass the per-user allowlist by routing through the public router contract, as the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs its identity check using `msg.sender` (the pool) as the mapping key and `sender` (the first parameter) as the swapper:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap(), not the originating EOA
    ...
);
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no mechanism to forward the originating user's address:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The originating user's address (`msg.sender`) is stored only in transient storage for the payment callback via `_setNextCallbackContext`; it is never surfaced to the pool or the extension. Consequently, when a user routes through the router, the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

The same structural gap exists in `exactInput` (multi-hop, L99-118) and `exactOutputSingle`/`exactOutput` paths.

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension` to gate swaps to specific users.
2. Admin allowlists the router address via `setAllowedToSwap(pool, router, true)` to permit router-mediated swaps.
3. Any unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the allowlist-gated pool.
4. The pool receives `msg.sender = router`, passes `router` as `sender` to the extension.
5. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. The per-user allowlist is fully bypassed; the unprivileged user executes a swap that should have been rejected.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control hook for pools that require restricted swap access (e.g., KYC-gated, institutional, or compliance-restricted pools). When the router is allowlisted, the allowlist provides zero per-user protection: any address can swap by routing through the public router. This breaks the core pool functionality of access-gated swap flows and constitutes an admin-boundary break where an unprivileged path bypasses a security control the pool admin explicitly configured.

## Likelihood Explanation
The precondition — admin allowlisting the router — is a natural and expected configuration for any pool that intends to support router-mediated swaps alongside the allowlist. The admin has no alternative mechanism to allow router-mediated swaps for specific users; the only option is to allowlist the router address itself, which inadvertently grants access to all users. The bypass is trivially repeatable by any unprivileged user with no special capability required beyond calling the public router.

## Recommendation
The extension must verify the originating user, not the intermediate caller. Options include:
1. **Forward the originating user via `extensionData`**: The router encodes `msg.sender` into `extensionData` and the extension decodes and verifies it, with the pool verifying the caller is a trusted router before trusting the encoded address.
2. **Transient storage forwarding**: The router writes the originating user to a known transient storage slot before calling `pool.swap()`, and the extension reads from that slot.
3. **Router-level allowlist**: Maintain a separate allowlist at the router level that gates which users can call the router for allowlist-gated pools.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension, only `allowedUser` is allowlisted
swapAllowlist.setAllowedToSwap(pool, allowedUser, true);
// Admin allowlists router to enable router-mediated swaps
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attack: unprivileged user routes through the router
vm.prank(unprivilegedUser);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: unprivilegedUser,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// Swap succeeds — allowlist bypassed
```
