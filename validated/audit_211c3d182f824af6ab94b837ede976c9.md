### Title
Mempool/Consensus Split via `bls_g1_negate` with Invalid G1 Point When `RELAXED_BLS` Hard-Fork Is Active — (`src/bls_ops.rs`, `src/chia_dialect.rs`)

### Summary
After the `RELAXED_BLS` hard-fork activates, consensus nodes accept `bls_g1_negate` (opcode 51) with any 48-byte atom regardless of curve validity. `MEMPOOL_MODE` does not include `RELAXED_BLS`, so mempool nodes call `validate_g1` and reject the same input. A transaction containing such a program is valid on-chain but unpropagatable through the mempool.

### Finding Description
`op_bls_g1_negate` derives `strict` from `!flags.contains(ClvmFlags::RELAXED_BLS)`. [4](#0-3) 

When `strict` is `false` (consensus with `RELAXED_BLS`), `validate_g1` is never called and the function proceeds to XOR byte 0 and return the result. [5](#0-4) 

`MEMPOOL_MODE` is a fixed constant that does not include `RELAXED_BLS`: [1](#0-0) 

There is no code path that conditionally adds `RELAXED_BLS` to `MEMPOOL_MODE` after hard-fork activation. The split is structural, not conditional on runtime state.

### Impact Explanation
Any CLVM program that calls `bls_g1_negate` with a 48-byte atom that is not a valid G1 curve point will be:
- **Accepted** by consensus nodes running with `RELAXED_BLS`
- **Rejected** by mempool nodes running `MEMPOOL_MODE`

This means such transactions cannot propagate through the peer-to-peer mempool after the hard-fork activates, even though they are fully valid for block inclusion. Farmers who include them directly bypass the mempool, but ordinary users cannot submit them via standard transaction flow.

### Likelihood Explanation
The hard-fork flag `RELAXED_BLS` is already defined and wired into `op_bls_g1_negate`. Once any deployment sets `RELAXED_BLS` in its consensus `ChiaDialect`, the split is immediately active. The attacker input is trivially constructible: any 48-byte sequence with the correct compression flag bits but an invalid field element suffices.

### Recommendation
Add `ClvmFlags::RELAXED_BLS` to `MEMPOOL_MODE` when the hard-fork activates, mirroring the pattern used for other hard-fork flags. Alternatively, document explicitly that `MEMPOOL_MODE` intentionally remains stricter and that transactions relying on `RELAXED_BLS` behavior must be submitted out-of-band — but this would be a protocol-level design decision, not a code fix.

### Proof of Concept
```rust
// Construct a 48-byte atom that has the correct compression flag (0x80 set,
// 0x40 clear = compressed, non-infinity) but is not a valid G1 point.
let mut bad_g1 = [0u8; 48];
bad_g1[0] = 0x80; // compressed, non-infinity, invalid field element

let mut alloc = Allocator::new();
let atom = alloc.new_atom(&bad_g1).unwrap();
let args = alloc.new_pair(atom, alloc.nil()).unwrap();

// Consensus mode (RELAXED_BLS set): succeeds
let result_consensus = op_bls_g1_negate(
    &mut alloc, args, u64::MAX, ClvmFlags::RELAXED_BLS
);
assert!(result_consensus.is_ok(), "consensus should accept");

// Mempool mode (RELAXED_BLS absent): fails
let result_mempool = op_bls_g1_negate(
    &mut alloc, args, u64::MAX, MEMPOOL_MODE
);
assert!(result_mempool.is_err(), "mempool should reject");
// Split confirmed: same program, opposite outcomes.
```
