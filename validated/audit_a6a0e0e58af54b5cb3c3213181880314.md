### Title
`BurnNFT` Requires Simultaneous Owner + Creator — NFT Permanently Unburnable After Transfer — (`x/nft/keeper/keeper.go`)

### Summary

`keeper.BurnNFT` enforces two independent guards in sequence: the caller must be **both** the current NFT owner **and** the denom creator. Once the creator transfers an NFT to any other address, no party can ever satisfy both conditions simultaneously, making the NFT permanently unburnable and the supply counter permanently inflated.

### Finding Description

`keeper.BurnNFT` at lines 141–161 applies two sequential checks: [1](#0-0) 

```go
nft, err := k.IsOwner(ctx, denomID, tokenID, owner)   // caller must own the NFT
if err != nil { return err }

_, err = k.IsDenomCreator(ctx, denomID, owner)         // caller must also be denom creator
if err != nil { return err }
```

After `MsgTransferNFT` moves `tok1` from Alice (creator) to Bob:

| Caller | `IsOwner` | `IsDenomCreator` | Result |
|--------|-----------|-----------------|--------|
| Alice  | **FAIL** (no longer owner) | PASS | Error |
| Bob    | PASS | **FAIL** (not creator) | Error |

Neither party can satisfy both guards. The NFT record persists in the KV store, `decreaseSupply` is never called, and the supply counter is permanently inflated by 1 for every such transfer.

The existing keeper test at lines 172–180 **explicitly confirms** this behavior — it shows `BurnNFT` fails after transfer to a non-creator, and the workaround used in the test is to transfer the NFT *back* to the creator before burning: [2](#0-1) 

This workaround requires Bob's cooperation. If Bob refuses (or is a contract/module account), the NFT is permanently locked.

`BurnNFTUnverified` (used only for IBC) skips the creator check but is not reachable via `MsgBurnNFT`: [3](#0-2) 

`MsgBurnNFT` in `msg_server.go` routes exclusively through `keeper.BurnNFT`: [4](#0-3) 

### Impact Explanation

- Supply counter (`GetTotalSupply`) is permanently inflated by ≥1 for every NFT transferred away from the creator that is not returned.
- The NFT record is never deleted; `decreaseSupply` is never reached.
- The invariant stated in the spec — *"Sender of MsgBurnNFT should be the Owner of the NFT and Creator of the denomination"* — is impossible to satisfy post-transfer without Bob's cooperation. [5](#0-4) 

### Likelihood Explanation

Any user who receives an NFT via `MsgTransferNFT` from the denom creator triggers this condition. It requires no special privileges — just a normal transfer followed by a burn attempt. The scenario is reachable through standard Cosmos SDK transaction flow.

### Recommendation

Change `BurnNFT` to require **either** (a) the caller is the current owner, **or** (b) the caller is the denom creator and the NFT exists. The most natural fix is: allow the current owner to burn unconditionally (matching how `TransferOwner` works), and optionally allow the denom creator to burn regardless of current ownership.

### Proof of Concept

The existing test `TestBurnNFT` at lines 172–180 already proves the invariant break: [6](#0-5) 

A minimal reproducer (no transfer-back step):
1. Issue denom with Alice as creator.
2. `MintNFT(denomID, tok1, ..., sender=Alice, owner=Alice)` — succeeds.
3. `TransferOwner(denomID, tok1, Alice, Bob)` — succeeds.
4. `BurnNFT(denomID, tok1, Alice)` — **fails**: `Alice is not the owner of denomID/tok1`.
5. `BurnNFT(denomID, tok1, Bob)` — **fails**: `Bob is not the creator of denomID`.
6. `GetTotalSupply(denomID)` returns 1 permanently; `tok1` record remains in store.

```go
func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	_, err = k.IsDenomCreator(ctx, denomID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
}
```
