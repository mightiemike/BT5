### Title
Duplicate Position ID in `MsgClaimTierRewards` Enables Double Bonus Reward Drain — (`x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`ClaimTierRewards` loads all requested positions into an in-memory slice before processing any of them. If the same position ID appears twice in `msg.PositionIds`, both slice entries carry identical stale state. The first iteration updates on-chain state via `setPosition`, but the second iteration re-processes the same validator events from the stale in-memory copy and re-sends bonus coins from the `RewardsPoolName` module account — a direct analog to the Checks-Effects-Interactions violation in the dHedge report.

---

### Finding Description

In `ClaimTierRewards` (`x/tieredrewards/keeper/msg_server.go`, lines 429–468), positions are loaded into a slice in a validation loop, then handed to `claimRewardsAndUpdatesPositions`:

```go
// msg_server.go:434-446
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    ...
    if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
        return nil, err
    }
    positions = append(positions, pos)
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
```

If `msg.PositionIds = [X, X]`, `positions[0]` and `positions[1]` are two independent in-memory copies of the same on-chain state (e.g., `LastEventSeq: 5`, `LastBonusAccrual: T1`).

Inside `claimRewardsAndUpdatesPositions` (`x/tieredrewards/keeper/claim_rewards.go`, lines 106–135):

```go
for i := range positions {
    pos := &positions[i]
    base, err := k.claimBaseRewards(ctx, *pos)
    ...
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)  // mutates pos in-place
    ...
    if err := k.setPosition(ctx, pos.Position, nil); err != nil { ... }
}
```

**Iteration 1 (i=0):** `processEventsAndClaimBonus` walks events since `LastEventSeq=5`, computes bonus, calls `bankKeeper.SendCoinsFromModuleToAccount` to pay bonus, advances `pos.LastEventSeq` and `pos.LastBonusAccrual`, then `setPosition` writes the updated state to the store.

**Iteration 2 (i=1):** `positions[1]` still holds the original stale state (`LastEventSeq=5`, `LastBonusAccrual=T1`). `processEventsAndClaimBonus` re-walks the same events, re-computes the same bonus, and calls `bankKeeper.SendCoinsFromModuleToAccount` a second time — paying the bonus again. `setPosition` then overwrites the store with the same updated state.

The broken invariant is: **bonus rewards for a given event range must be paid at most once per position**. The position's `LastEventSeq`/`LastBonusAccrual` checkpoint is the guard, but it is only read from the store once (at load time) and never re-read before the second processing of the same position.

Additionally, `decrementEventRefCount` (`claim_rewards.go`, line 196) is called for each event in each iteration, so the same event's reference count is decremented twice, corrupting the event lifecycle accounting.

---

### Impact Explanation

A position owner submits a single `MsgClaimTierRewards` transaction with `PositionIds = [X, X, X, ...]` (the same position ID repeated N times). Each repetition re-pays the full accrued bonus from the `RewardsPoolName` module account. With enough repetitions in one transaction, the attacker drains the entire bonus rewards pool. The corrupted value is the `RewardsPoolName` module account balance and the per-position `LastEventSeq`/`LastBonusAccrual` checkpoint.

---

### Likelihood Explanation

The entry path is a standard user-signed `MsgClaimTierRewards` transaction — no privilege required, no leaked keys, no social engineering. The only mitigation would be a deduplication check in `msg.Validate()` or at the top of `ClaimTierRewards`. Neither is visible in the code examined. Any position owner with accrued bonus rewards can trigger this immediately.

---

### Recommendation

Deduplicate `msg.PositionIds` in `MsgClaimTierRewards.Validate()` (reject if any ID appears more than once), or re-read each position's state from the store inside `claimRewardsAndUpdatesPositions` rather than relying on the pre-loaded slice. The latter mirrors Option 3 from the dHedge report: ensure state is always current before the external payment call.

---

### Proof of Concept

1. Owner has position ID `42` with `LastEventSeq=5`, `LastBonusAccrual=T1`, and 100 bonus tokens accrued.
2. Owner submits `MsgClaimTierRewards{Owner: owner, PositionIds: [42, 42]}`.
3. `ClaimTierRewards` loads position 42 twice → `positions = [{...seq=5...}, {...seq=5...}]`.
4. Iteration 0: `processEventsAndClaimBonus` pays 100 tokens, advances `positions[0]` to `seq=10`; `setPosition` writes `seq=10` to store.
5. Iteration 1: `positions[1]` still has `seq=5`; `processEventsAndClaimBonus` re-walks the same events, pays another 100 tokens; `setPosition` writes `seq=10` again.
6. Owner receives 200 tokens instead of 100. Repeating with N copies yields N×100 tokens drained from `RewardsPoolName`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
