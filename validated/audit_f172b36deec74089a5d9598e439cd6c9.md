### Title
Accumulated Fast-Withdrawal Fees Permanently Locked When Product Token Address Is Updated — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool` accumulates fast-withdrawal fees per `productId` in the `fees` mapping. All token recovery paths resolve the token address dynamically via `getToken(productId)`, which reads the **current** token from `SpotEngine`. Because `SpotEngine.addOrUpdateProduct` can overwrite the token address for an existing product unconditionally, any fees accumulated in the old token become permanently unrecoverable once the token is changed — there is no arbitrary-token rescue function in `WithdrawPool`.

---

### Finding Description

Fast-withdrawal fees are collected and tracked in `fees[productId]` inside `BaseWithdrawPool`: [1](#0-0) 

The only owner-callable function that can move tokens out of the pool is `removeLiquidity`: [2](#0-1) 

It resolves the token to transfer via `getToken(productId)`, which dynamically reads the current token from `SpotEngine`: [3](#0-2) 

The token address stored in `SpotEngine` for any product can be overwritten at any time by the owner through `addOrUpdateProduct`, because `configs[productId] = config` executes unconditionally for both new and existing products: [4](#0-3) 

Once the token for a `productId` is changed, `getToken(productId)` returns the new token. Any call to `removeLiquidity` for that `productId` will attempt to transfer the **new** token, not the old one. The old token balance — which includes all previously accumulated fast-withdrawal fees — has no extraction path. `WithdrawPool` adds no new functions beyond `BaseWithdrawPool`: [5](#0-4) 

There is no arbitrary-token rescue function anywhere in `BaseWithdrawPool` or `WithdrawPool`.

---

### Impact Explanation

Protocol fast-withdrawal fee revenue denominated in the old token is permanently locked in the `WithdrawPool` contract. The `fees[productId]` accounting also becomes desynchronized from actual recoverable balances. The magnitude equals the total fast-withdrawal fees collected for the affected product since deployment.

**Impact: 3 / 5** — Direct, permanent loss of protocol fee assets with no recovery path.

---

### Likelihood Explanation

The Nado codebase already contains evidence of a real token migration scenario (`replaceUsdcEWithUsdc` in `ContractOwner.sol`), demonstrating that token address changes for existing products are a realistic operational event. Any such migration for a product with accumulated fast-withdrawal fees triggers the lock. [6](#0-5) 

**Likelihood: 3 / 5** — Token migrations are operationally plausible and have precedent in this codebase.

---

### Recommendation

Add an arbitrary-token rescue function to `BaseWithdrawPool` (or `WithdrawPool`) that accepts an explicit token address and recipient, callable only by the owner:

```solidity
function rescueToken(address tokenAddr, address sendTo, uint128 amount) external onlyOwner {
    IERC20Base(tokenAddr).safeTransfer(sendTo, uint256(amount));
}
```

This mirrors the fix applied in the referenced Moonwell commit, which added a token-address-parameterized withdraw function to `TokenSaleDistributor`.

---

### Proof of Concept

1. Product `productId = 1` is configured with `tokenA` via `SpotEngine.addOrUpdateProduct`.
2. Multiple users call `BaseWithdrawPool.submitFastWithdrawal` for `productId = 1`. Each call adds to `fees[1]`, and the corresponding `tokenA` balance accumulates in the `WithdrawPool` contract. [7](#0-6) 
3. The owner calls `SpotEngine.addOrUpdateProduct(1, ..., Config({ token: tokenB, ... }), ...)`. This overwrites `configs[1].token` to `tokenB`. [8](#0-7) 
4. The owner calls `WithdrawPool.removeLiquidity(1, amount, recipient)`. Internally, `getToken(1)` now returns `tokenB`. The call attempts `tokenB.safeTransfer(recipient, amount)` — operating on the wrong token entirely. [9](#0-8) 
5. The `tokenA` fees remain in the contract with no callable path to extract them. `fees[1]` still records the accumulated amount, but it is permanently irrecoverable.

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
