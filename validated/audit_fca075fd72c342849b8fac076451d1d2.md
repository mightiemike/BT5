### Title
Failed Slow-Mode Deposit Silently Discards User Funds With No Recovery Path ā€” (`core/contracts/Endpoint.sol`)

---

### Summary

`depositCollateralWithReferral` transfers a user's ERC20 tokens to the `Clearinghouse` contract before queuing a `SlowModeTx`. If that queued transaction later reverts during execution (e.g., because the product was delisted in the intervening 3-day window), the `_executeSlowModeTransaction` catch block silently discards the failure ā€” explicitly annotated `// try return funds now removed` ā€” with no refund and no recovery mechanism. The user's deposited collateral is permanently locked.

---

### Finding Description

The deposit flow in `Endpoint.sol` is a two-step process:

**Step 1 ā€” Immediate token transfer** (`depositCollateralWithReferral`, lines 144ā€“148):

```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
```

`handleDepositTransfer` in `EndpointStorage.sol` (lines 111ā€“119) performs `safeTransferFrom(token, from, amount)` followed by `safeTransferTo(token, address(clearinghouse), amount)` ā€” the tokens are immediately moved to the `Clearinghouse`. [1](#0-0) 

**Step 2 ā€” Deferred credit** (lines 152ā€“166): A `SlowModeTx` is queued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (3 days). The actual subaccount credit via `clearinghouse.depositCollateral` only happens when this queued tx is executed. [2](#0-1) 

**The failure path** ā€” `_executeSlowModeTransaction` (lines 185ā€“229):

```solidity
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];   // tx deleted before execution

try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed
}
```

The slow mode tx is **deleted from the queue before execution** (line 194). If `processSlowModeTransaction` reverts for any non-OOG reason, the catch block silently continues. The comment `// try return funds now removed` is an explicit acknowledgment that a refund mechanism was deliberately removed, leaving no recovery path. [3](#0-2) 

**Concrete revert trigger** ā€” `clearinghouse.depositCollateral` (lines 193ā€“209 of `Clearinghouse.sol`) calls `_decimals(txn.productId)` which calls `_tokenAddress(productId)` ā†’ `_spotEngine().getConfig(productId).token`. If the product has been delisted between deposit submission and execution, `getConfig` returns a zeroed struct, `token` is `address(0)`, and the subsequent `token.decimals()` call reverts on the zero address. [4](#0-3) 

The `processSlowModeTransactionImpl` path for `DepositCollateral` in `EndpointTx.sol` (lines 209ā€“216) calls `clearinghouse.depositCollateral(txn)` with no guard against a delisted product. [5](#0-4) 

---

### Impact Explanation

User ERC20 collateral is permanently locked in the `Clearinghouse` contract. The tokens were transferred at deposit time, the slow mode tx is deleted from the queue on execution, no subaccount balance is credited, and no refund function exists anywhere in the protocol. The asset delta is: user loses `amount` tokens of `productId` with zero on-chain credit.

---

### Likelihood Explanation

**Medium.** The 3-day `SLOW_MODE_TX_DELAY` window creates a realistic gap during which a product can be delisted. Product delisting is an owner-controlled operation (`DelistProduct` slow mode tx, owner-only per `submitSlowModeTransactionImpl` line 368), so it requires an admin action ā€” but product lifecycle events are normal protocol operations. Any user who deposited into a product that is subsequently delisted before their slow mode tx executes will permanently lose their funds. The `executeSlowModeTransaction` function is callable by anyone (line 231), so the failure can be triggered permissionlessly after the 3-day delay. [6](#0-5) [7](#0-6) 

---

### Recommendation

Restore a refund mechanism in the catch block of `_executeSlowModeTransaction`. When a `DepositCollateral` slow mode tx fails, the contract should transfer the deposited amount back to the original sender from the `Clearinghouse`. Alternatively, before deleting the slow mode tx from the queue, attempt execution and only delete on success ā€” or store a pending-refund mapping that users can claim. The removed `// try return funds` logic should be reinstated specifically for `DepositCollateral` tx types.

---

### Proof of Concept

1. Product `productId = 5` (e.g., WBTC) is live. User calls `depositCollateral("default\x00\x00\x00\x00\x00\x00", 5, 1e8)`. Tokens are transferred to `Clearinghouse`. A `SlowModeTx` is queued at index `N` with `executableAt = T + 3 days`.

2. At time `T + 1 day`, the owner submits a `DelistProduct` slow mode tx for product 5. The sequencer processes it immediately via `ExecuteSlowMode`. Product 5 is removed from `SpotEngine`; `getConfig(5).token` now returns `address(0)`.

3. At time `T + 3 days + 1 second`, anyone calls `executeSlowModeTransaction()`.

4. Inside `_executeSlowModeTransaction`: the tx at index `N` is deleted (line 194). `processSlowModeTransaction` is called, which calls `processSlowModeTransactionImpl` ā†’ `clearinghouse.depositCollateral(txn)` ā†’ `_decimals(5)` ā†’ `IERC20Base(address(0)).decimals()` ā†’ **reverts**.

5. The catch block fires. `gasleft()` is well above 250,000 and above `gasRemaining / 2`. Execution continues past `// try return funds now removed`.

6. User's 1e8 WBTC tokens remain in `Clearinghouse`. No subaccount balance was credited. No refund was issued. Funds are permanently locked. [8](#0-7) [9](#0-8)

```text
    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
```

```text
    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
```
