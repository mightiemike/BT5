### Title
`submitFastWithdrawal` Lacks Balance Reservation Check, Allowing Pool Drain That Blocks Pending Standard Withdrawals — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal()` is a public function callable by any user holding a valid protocol-signed withdrawal transaction. It transfers tokens from the pool with no check that the remaining balance is sufficient to cover pending standard withdrawals queued by the clearinghouse. An unprivileged user with a large valid fast-withdrawal can drain the pool, causing all subsequent `submitWithdrawal()` calls from the clearinghouse to revert, permanently blocking other users' collateral exits.

---

### Finding Description

`BaseWithdrawPool` exposes two withdrawal execution paths:

**Path A — `submitWithdrawal` (clearinghouse-only, lines 116–132):** [1](#0-0) 

This path is restricted to `msg.sender == clearinghouse` and processes sequencer-batched withdrawals. It sets `minIdx = idx` to invalidate superseded fast-withdrawal indices, then calls `handleWithdrawTransfer` directly — with no balance reservation check.

**Path B — `submitFastWithdrawal` (public, lines 81–114):** [2](#0-1) 

This path is callable by **any address** that presents a valid Verifier-signed transaction. After signature validation and fee deduction, it calls `handleWithdrawTransfer` to transfer `transferAmount` tokens to `sendTo` — again with no balance reservation check. [3](#0-2) 

The pool holds a single undifferentiated token balance that funds **both** paths. There is no accounting variable tracking "tokens reserved for pending sequencer withdrawals" analogous to `totalAllocatedOutputToken - totalReleasedOutputToken` in the reference report. Because `submitFastWithdrawal` performs no such check, a single large fast-withdrawal can reduce `token.balanceOf(address(this))` to zero, after which every `submitWithdrawal` call from the clearinghouse will revert inside `handleWithdrawTransfer`: [4](#0-3) 

The `minIdx` replay-prevention mechanism does **not** mitigate this: it only prevents the same `idx` from being processed twice. It does not prevent a fast-withdrawal at `idx = N` from consuming tokens that were deposited to fund standard withdrawals at `idx = N-k … N-1`.

---

### Impact Explanation

When the pool is drained by a fast-withdrawal:

- All pending `submitWithdrawal(token, sendTo, amount, idx)` calls from the clearinghouse revert because `token.safeTransfer` fails on insufficient balance.
- Affected users cannot receive their collateral. Their funds remain locked inside the protocol's internal accounting with no on-chain mechanism to recover them without owner intervention (`removeLiquidity`).
- The attacker receives their own legitimately signed amount; the loss falls entirely on other users whose standard withdrawals are blocked.

This matches the reference report's impact class: one execution path (fast) drains tokens reserved for another path (standard), causing future claimants to be unable to receive their entitled amounts.

---

### Likelihood Explanation

The attack requires only:

1. The attacker is a legitimate protocol user with a non-trivial balance.
2. They request a fast withdrawal from the protocol for their full balance — a normal, supported operation.
3. The protocol's signers countersign the transaction (standard protocol behavior).
4. The attacker calls `submitFastWithdrawal` with the signed transaction.

No key compromise, admin access, or social engineering is required. The attacker uses the normal fast-withdrawal flow. The condition is most easily triggered when the pool's liquidity is thin relative to a single user's balance, which is a realistic operational state.

---

### Recommendation

Before executing the token transfer in `submitFastWithdrawal`, add a check that the pool's remaining balance after the transfer is at least equal to the sum of all amounts owed to pending standard-withdrawal indices that have not yet been processed by the clearinghouse. This requires the pool to track a `totalPendingStandard` accumulator that is incremented when the clearinghouse registers a pending withdrawal and decremented when `submitWithdrawal` executes it. The guard would be:

```solidity
require(
    token.balanceOf(address(this)) - transferAmount >= totalPendingStandard[address(token)],
    "INSUFFICIENT_RESERVED_BALANCE"
);
```

This mirrors the fix applied to `transmuteInstant()` in the reference report.

---

### Proof of Concept

1. Pool holds 1,000 USDC to fund 10 pending standard withdrawals of 100 USDC each (idx 101–110), queued by the clearinghouse but not yet executed.
2. Attacker (a legitimate user) requests a fast withdrawal for 1,000 USDC. Protocol signers countersign it with `idx = 200`.
3. Attacker calls `submitFastWithdrawal(200, signedTx, sigs)`.
   - `markedIdxs[200]` is false → passes.
   - `200 > minIdx` (e.g., `minIdx = 100`) → passes.
   - Verifier validates signatures → passes.
   - `handleWithdrawTransfer` transfers 1,000 USDC to attacker → pool balance = 0.
4. Clearinghouse later calls `submitWithdrawal(USDC, user1, 100, 101)`.
   - `markedIdxs[101]` is false → proceeds.
   - `handleWithdrawTransfer` calls `token.safeTransfer(user1, 100)` → **reverts** (balance = 0).
5. All 10 standard withdrawals (idx 101–110) are permanently blocked. Users cannot retrieve their collateral. [2](#0-1) [1](#0-0) [4](#0-3)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function submitWithdrawal(
        IERC20Base token,
        address sendTo,
        uint128 amount,
        uint64 idx
    ) public {
        require(msg.sender == clearinghouse);

        if (markedIdxs[idx]) {
            return;
        }
        markedIdxs[idx] = true;
        // set minIdx to most recent withdrawal submitted by sequencer
        minIdx = idx;

        handleWithdrawTransfer(token, sendTo, amount);
    }
