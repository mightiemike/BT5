Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Allowing Any User to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is the pool's `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router address, not the end-user. If the pool admin allowlists the router (required for any router-mediated swap to succeed on a restricted pool), every unprivileged user can bypass the per-user swap allowlist by routing through the router. The actual end-user address is never inspected.

## Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// SwapAllowlistExtension.sol L37-38
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the pool calls the extension via `CallExtension.callExtension`), and `sender` is the value the pool received as its own `msg.sender` — i.e., whoever called `pool.swap()`.

`MetricOmmPool.swap` passes `msg.sender` directly as the first argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← router address when called via router
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this unchanged into the ABI-encoded call to the extension:

```solidity
// ExtensionCalling.sol L162-165
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)   // sender = router
)
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The complete call chain:
```
user → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(...)           // msg.sender = router
              → _beforeSwap(sender=router, ...)
                   → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        → checks allowedSwapper[pool][router]  // ← router, not user
```

For any router-mediated swap to succeed on a restricted pool, the pool admin must call `setAllowedToSwap(pool, router, true)`. Once that entry exists, the extension's check passes for **every** user who calls the router — the actual end-user's address is never inspected. The `exactInput` multi-hop path has the same flaw: the router calls `pool.swap()` in a loop, so `sender` is still the router for every hop.

The `exactOutput` multi-hop path also calls `pool.swap()` from within `_exactOutputIterateCallback`, again with `msg.sender = router`.

No existing guard compensates for this: `BaseMetricExtension.beforeSwap` only enforces `onlyPool` (that the caller is a registered pool), not the identity of the end-user.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (e.g., KYC-verified counterparties, institutional LPs, or whitelisted market-makers) loses that restriction entirely for any user who routes through the allowlisted router. The guard the pool admin believed was enforcing per-user access control is silently bypassed. Any unauthorized address can execute swaps against the pool's liquidity, potentially draining value from LPs who deposited under the assumption that only vetted counterparties could trade. This constitutes a broken core pool access-control invariant and an admin-boundary break reachable by an unprivileged path.

## Likelihood Explanation

The router must be allowlisted for it to be usable at all on a restricted pool. A pool admin who wants to support both router-based UX and per-user access control will naturally allowlist the router, unknowingly opening the bypass. The trigger requires no special privilege — any EOA can call `MetricOmmSimpleRouter.exactInputSingle` or `exactInput`. The condition is self-inflicted by the expected operational pattern and is therefore highly likely to be triggered in practice.

## Recommendation

The extension must verify the **end-user** identity, not the intermediate caller. Two viable approaches:

1. **Pass the real user in `extensionData`**: The router encodes `msg.sender` (the end-user) into `extensionData`; the extension decodes and checks it when `sender` is a known router. This requires a coordinated convention between router and extension.
2. **Maintain a trusted-router registry in the extension**: When `sender` is a registered trusted router, require the extension to receive the real user address via `extensionData` and check that address against `allowedSwapper`. When `sender` is not a router, check `sender` directly as today.
3. **Remove router from the allowlist and require direct pool calls**: Simplest but breaks router UX for all users.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` so that router-mediated swaps work for allowlisted users.
3. Pool admin calls `setAllowedToSwap(pool, alice, true)` — Alice is the only intended user.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. The pool calls `extension.beforeSwap(sender=router, ...)`.
7. The extension checks `allowedSwapper[pool][router]` → `true` → no revert.
8. Bob's swap executes successfully despite never being added to the allowlist.

Foundry test outline:
```solidity
function test_bypassAllowlistViaRouter() public {
    // Setup: pool with SwapAllowlistExtension, router allowlisted, only alice allowlisted as user
    vm.prank(poolAdmin);
    ext.setAllowedToSwap(pool, address(router), true);
    vm.prank(poolAdmin);
    ext.setAllowedToSwap(pool, alice, true);

    // Bob (not allowlisted) swaps via router — should revert but does not
    vm.prank(bob);
    router.exactInputSingle(ExactInputSingleParams({pool: pool, ...}));
    // Bob's swap succeeds: allowlist bypassed
}
```
