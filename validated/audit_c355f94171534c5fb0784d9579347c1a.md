### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase-Proof Invariant Enforced by `verify_transaction_inclusion_v2` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is still a live, unpermissioned public entry point. `verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte-transaction Merkle-proof forgery vector by requiring a coinbase proof. Because the old function was never removed or access-gated, any NEAR caller can invoke it directly, skipping the coinbase-proof check entirely and obtaining a `true` return value for a transaction that does not exist in the claimed block.

---

### Finding Description

The contract exposes two SPV verification entry points:

**`verify_transaction_inclusion`** (lines 288–323):
- Checks confirmations and mainchain membership.
- Requires `!args.merkle_proof.is_empty()`.
- Computes `compute_root_from_merkle_proof(tx_id, tx_index, &merkle_proof)` and compares to `header.block_header.merkle_root`.
- **No coinbase proof is required.** [1](#0-0) 

**`verify_transaction_inclusion_v2`** (lines 347–369):
- First validates a coinbase merkle proof against the block's `merkle_root` (lines 358–365), panicking if it does not match.
- Only then delegates to `verify_transaction_inclusion`. [2](#0-1) 

The coinbase-proof step is the critical invariant: it forces the prover to demonstrate knowledge of a real 32-byte transaction at index 0, making it computationally infeasible to substitute a crafted 64-byte internal Merkle node as the "transaction." The contract's own documentation acknowledges this:

> *"This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."* [3](#0-2) 

Despite being marked `#[deprecated]`, `verify_transaction_inclusion` carries only a `#[pause]` attribute — the same gate as v2. It has no role restriction, no removal, and no forwarding to v2. It is a fully reachable public method. [4](#0-3) 

The structural parallel to the reported issue is exact:

| Perps analog | BTC light client |
|---|---|
| `removeMargin()` — strict check | `verify_transaction_inclusion_v2` — coinbase proof required |
| `setPositionLeverage()` — weak check | `verify_transaction_inclusion` — no coinbase proof |
| Attacker increases leverage to extract margin | Attacker calls v1 directly to forge inclusion proof |

---

### Impact Explanation

A recipient smart contract on NEAR that consumes the boolean result of `verify_transaction_inclusion` to gate a cross-chain action (e.g., releasing funds, minting wrapped BTC, confirming a swap) can be deceived into treating a non-existent Bitcoin transaction as confirmed. The attacker supplies a crafted 64-byte value as `tx_id` that is actually an internal Merkle tree node; `compute_root_from_merkle_proof` will reconstruct the correct `merkle_root` from it, and the function returns `true`. The block header itself is valid and on-chain; only the claimed transaction is fabricated. [5](#0-4) 

---

### Likelihood Explanation

The entry point is public, requires no special role, and is callable by any NEAR account whenever the contract is not paused (the normal operating state). The 64-byte transaction forgery technique is well-documented (BitMEX research, CVE-2017-12842). No privileged access, leaked keys, or social engineering is needed. The only prerequisite is knowledge of a valid mainchain block header already stored in the contract, which is public state.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or redirect it unconditionally to `verify_transaction_inclusion_v2` logic (requiring a coinbase proof). If backward compatibility must be preserved for existing callers, add an explicit `require!(false, "use verify_transaction_inclusion_v2")` panic, or gate the function behind a role that no external caller holds. The coinbase-proof check must be an invariant on **every** reachable SPV verification path, not only on the v2 path. [6](#0-5) 

---

### Proof of Concept

1. Identify any valid mainchain block hash `B` stored in the contract with sufficient confirmations.
2. Retrieve the `merkle_root` of block `B` from the contract's `headers_pool`.
3. Construct a 64-byte value `fake_tx_id` that is an internal Merkle node of block `B`'s transaction tree (obtainable from public Bitcoin block data).
4. Build a `merkle_proof` path from `fake_tx_id` upward to `merkle_root` using the sibling hashes of the internal node.
5. Call `verify_transaction_inclusion` with `tx_id = fake_tx_id`, `tx_block_blockhash = B`, `tx_index = <position of the internal node>`, `merkle_proof = <constructed path>`, `confirmations = 1`.
6. `compute_root_from_merkle_proof(fake_tx_id, index, proof)` returns `merkle_root`; the equality check passes; the function returns `true`.
7. Any downstream contract that trusts this result treats the fabricated transaction as confirmed on Bitcoin. [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
