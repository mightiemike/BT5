### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase-Proof Check Enforced by `verify_transaction_inclusion_v2` — (`File: contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is still a live, public, unpause-gated contract method. It omits the coinbase Merkle proof check that `verify_transaction_inclusion_v2` was specifically introduced to enforce. Any unprivileged NEAR caller — or any recipient contract — can invoke the old path directly, supply a crafted internal-node hash as `tx_id`, and receive a `true` inclusion result for a Bitcoin transaction that was never mined.

---

### Finding Description

`verify_transaction_inclusion_v2` was added to close the 64-byte transaction Merkle-proof forgery vulnerability (documented at https://www.bitmex.com/blog/64-Byte-Transactions). It enforces a coinbase proof check before delegating to the old function: [1](#0-0) 

```rust
pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
    require!(
        merkle_tools::compute_root_from_merkle_proof(
            args.coinbase_tx_id.clone(),
            0usize,
            &args.coinbase_merkle_proof,
        ) == header.block_header.merkle_root,
        "Incorrect coinbase merkle proof"
    );
    #[allow(deprecated)]
    self.verify_transaction_inclusion(args.into())   // delegates after the guard
}
```

The old function is marked `#[deprecated]` but **not removed, not access-controlled, and not gated behind any role**: [2](#0-1) 

Its own doc-comment acknowledges the gap explicitly:

> **Warning**: This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. [3](#0-2) 

The only checks performed by the old path are: confirmation count, mainchain membership, and raw Merkle root equality — none of which prevent an internal-node forgery: [4](#0-3) 

The coinbase-proof guard that blocks the forgery lives **exclusively** in `verify_transaction_inclusion_v2` and is never reached when the old entry point is called directly.

---

### Impact Explanation

An attacker who knows the Merkle tree of any confirmed Bitcoin block can:

1. Pick any internal Merkle node `N` at depth `d`.
2. Construct a valid sibling path from `N` up to the `merkle_root` stored in the on-chain header.
3. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index` matching the subtree position, and the sibling path.
4. The function computes `compute_root_from_merkle_proof(N, index, path) == merkle_root` — this is **true by construction** — and returns `true`.

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a fund release, cross-chain claim, or proof-of-payment check will be deceived into accepting a fabricated Bitcoin transaction. The corrupted value is the **SPV proof result** (`bool`) returned to the caller, which is the sole output the light client is designed to produce.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. The method is a standard public NEAR view/call that any account can invoke. The 64-byte forgery technique is well-documented and the required inputs (block Merkle tree, sibling hashes) are publicly available from any Bitcoin full node. Likelihood is **high** for any integrator that calls the old entry point, and **medium** overall given that the v2 path exists and is documented.

---

### Recommendation

Remove `verify_transaction_inclusion` entirely, or gate it with an access-control role that prevents unprivileged callers from reaching it. A `#[deprecated]` attribute in Rust does not restrict call-site access at runtime; it only emits a compiler warning to Rust callers, which is irrelevant for a NEAR contract method invoked via JSON-RPC or cross-contract call.

---

### Proof of Concept

```
# 1. Identify a confirmed Bitcoin block already accepted by the light client,
#    e.g., block at height H with known merkle_root R.
# 2. Retrieve the full transaction list; compute the Merkle tree.
# 3. Select internal node N = hash(tx[0] || tx[1]) at depth 1, tx_index = 0.
# 4. The sibling path from N to R is just [sibling_of_N, ...upper nodes...].
# 5. Call (via near-cli or cross-contract):
near call <contract> verify_transaction_inclusion \
  '{"tx_id": "<N_as_hex>",
    "tx_block_blockhash": "<block_hash>",
    "tx_index": 0,
    "merkle_proof": ["<sibling_hex>", ...],
    "confirmations": 1}'
# 6. Contract returns `true` — no Bitcoin transaction with id N exists.
```

The call bypasses the coinbase-proof guard entirely because it never enters `verify_transaction_inclusion_v2`. The broken invariant is: *a `true` result from the light client must correspond to a real, mined Bitcoin transaction*. That invariant does not hold for the old entry point.

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
