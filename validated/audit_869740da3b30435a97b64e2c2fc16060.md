### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Allowing Any User to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool, which is the pool's `msg.sender` — the **router contract address** when a user enters through `MetricOmmSimpleRouter`. Any user who routes through the public router is checked against the allowlist as the router, not as themselves. If the pool admin allowlists the router to enable router-mediated swaps, the per-user allowlist is completely bypassed for all users.

---

### Finding Description

The call chain is:

```
User → MetricOmmSimpleRouter.exactInputSingle(...)
         → pool.swap(recipient, ...)          // msg.sender in pool = router
             → _beforeSwap(msg.sender, ...)   // sender = router address
                 → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                     → allowedSwapper[pool][router]  // checks router, not user
```

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

When the router calls `pool.swap(...)`, the pool's `msg.sender` is the router: [2](#0-1) 

The extension then checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router: [3](#0-2) 

This creates an irreconcilable dilemma for any pool admin who deploys a curated pool with `SwapAllowlistExtension`:

- **If the router is NOT allowlisted:** All router-mediated swaps revert with `NotAllowedToSwap`, even for users whose individual addresses are allowlisted. The supported periphery path is broken for all allowlisted users.
- **If the router IS allowlisted** (the only way to enable router-mediated swaps): The extension sees `sender = router` and passes for every user, regardless of whether their individual address is on the allowlist. Any non-allowlisted user can bypass the gate by calling `router.exactInputSingle(...)`.

The `DepositAllowlistExtension` does not share this flaw because it checks the `owner` parameter (explicitly supplied by the caller), not `sender`: [4](#0-3) 

---

### Impact Explanation

A curated pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC'd addresses, institutional partners) can be fully bypassed by any unprivileged user routing through the public `MetricOmmSimpleRouter`. The non-allowlisted user executes swaps against the pool's liquidity at oracle prices, extracting value from LP positions that were intended to be protected behind the allowlist. This is a direct loss of LP principal and a broken core pool invariant: the allowlist policy is not enforced on the supported periphery path.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary public swap interface for the protocol. Any user can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` against any pool address. No special privilege, role, or setup is required beyond having the input tokens. The bypass is triggered by the normal, documented usage of the router on a pool that has `SwapAllowlistExtension` configured with the router allowlisted.

---

### Recommendation

The `SwapAllowlistExtension` must gate the **economically relevant actor** — the end user — not the intermediary. Two approaches:

1. **Short term:** Add a `recipient` parameter check or require the pool to pass an additional "original caller" field through `extensionData`. The extension should check the address that will economically benefit from the swap (the `recipient` or a caller identity embedded in `extensionData`).

2. **Long term:** Redesign the `beforeSwap` hook signature to include an explicit `originator` field that the pool populates from a trusted source (e.g., transient storage set at the outermost entry point), so extensions can always gate the true end user regardless of how many intermediary contracts are in the call stack.

---

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted.
// Admin must also allowlist the router for router-mediated swaps to work.
extension.setAllowedToSwap(address(pool), allowedUser, true);
extension.setAllowedToSwap(address(pool), address(router), true); // required for router path

// Attack: bannedUser bypasses the allowlist via the router.
vm.prank(bannedUser);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 10_000,
        amountOutMinimum: 0,
        recipient: bannedUser,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// Extension sees sender = address(router), which is allowlisted → swap succeeds.
// bannedUser has successfully traded on a curated pool that should have blocked them.
```
