### Title
Equihash Solution Validity Result Discarded — Invalid PoW Accepted for Zcash Headers - (File: `contract/src/zcash.rs`)

### Summary
In the Zcash build of the BTC light client, `equihash::is_valid_solution` returns a `Result<bool, _>`. The code only handles the `Err` branch via `.unwrap_or_else`, but the resulting `bool` is never checked. A structurally parseable but cryptographically invalid Equihash solution returns `Ok(false)`, which is silently discarded. Any unprivileged NEAR caller can submit a Zcash block header with a fabricated Equihash solution and have it accepted as valid PoW.

### Finding Description
In `contract/src/zcash.rs`, `check_pow` performs the Equihash solution check as follows: [1](#0-0) 

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
```

`equihash::is_valid_solution` returns `Result<bool, Error>`:
- `Err(e)` → the `.unwrap_or_else` panics (correct).
- `Ok(true)` → solution is valid; the `bool` is returned from `unwrap_or_else` and **immediately dropped** (harmless but wasteful).
- `Ok(false)` → solution is **cryptographically invalid**; the `bool` `false` is returned from `unwrap_or_else` and **immediately dropped** — the contract continues without panicking.

The computed validity result is never asserted. This is the direct analog of the external report's "value calculated but not used" class.

`check_pow` is called from `check_target`, which is called from `submit_block_header` (non-dogecoin path) when `skip_pow_verification` is `false`: [2](#0-1) 

The public entry point is `submit_blocks`, which is callable by any NEAR account (subject only to the `trusted_relayer` and `pause` guards, both of which are open to normal relayers): [3](#0-2) 

### Impact Explanation
An attacker submitting a Zcash block header with a structurally valid but cryptographically invalid Equihash solution will pass all other checks (difficulty `bits` match, timestamp bounds) and have the header stored in `headers_pool`. If the forged header's `chain_work` exceeds the current main chain tip, `submit_block_header_inner` will trigger a chain reorganization, promoting the forged chain to canonical status: [4](#0-3) 

This corrupts the canonical chain state: `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` all reflect a chain backed by zero real proof-of-work. Any downstream consumer calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against this forged chain will receive fraudulent inclusion proofs.

### Likelihood Explanation
The Zcash feature is a named production build target (`make build-zcash`). The `submit_blocks` endpoint is reachable by any registered relayer without privileged access. Constructing a structurally parseable Equihash solution (e.g., a zero-filled or randomly filled byte array of the correct length) that is cryptographically invalid is trivial — no mining hardware is required. The attacker only needs to satisfy the `bits` field check (which is enforced against the expected difficulty) and the timestamp checks, both of which are achievable with real chain data for the `bits` and a crafted fake solution.

### Recommendation
Assert the returned boolean. Replace the current call with:

```rust
require!(
    equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
        .unwrap_or_else(|e| env::panic_str(&format!("Invalid Equihash solution: {e}"))),
    "Invalid Equihash solution"
);
``` [5](#0-4) 

### Proof of Concept

1. Deploy the contract with `feature = "zcash"` and `skip_pow_verification = false`.
2. Obtain a real Zcash mainnet header at a retarget boundary so `bits` matches the expected difficulty.
3. Replace `solution` with any byte array of the correct Equihash (200,9) solution length that is structurally parseable but cryptographically invalid (e.g., all-zero indices, or indices that do not satisfy the XOR/collision constraints).
4. Call `submit_blocks` with this header.
5. Observe: the contract does not panic; the header is stored in `headers_pool` and, if its `chain_work` is sufficient, is promoted to the main chain tip — despite having no valid proof-of-work.
