### Title
`tx_index` High-Bit Aliasing in `compute_root_from_merkle_proof` Allows Index Forgery — (`merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` only consumes the lowest `k` bits of `transaction_position` (where `k = merkle_proof.len()`). Any bits at position `k` and above are silently ignored. As a result, `verify_transaction_inclusion` returns `true` for `tx_index = N + 2^k`, `N + 2^(k+1)`, etc., using the same proof that is valid for index `N`. The light client falsely attests that a transaction is at a position it does not occupy.

---

### Finding Description

The loop in `compute_root_from_merkle_proof` runs exactly `merkle_proof.len()` times. Each iteration reads only `current_position % 2` (the lowest bit) and then right-shifts with `current_position /= 2`. [1](#0-0) 

After `k` iterations the loop ends. Bits at positions `≥ k` in the original `transaction_position` are never examined. Concretely, for a proof of depth `k=3` and valid index `N=1` (binary `001`):

| Index | Binary | Bit sequence consumed | Root |
|-------|--------|-----------------------|------|
| 1 | `001` | odd, even, even | R |
| 9 | `1001` | odd, even, even | R |
| 17 | `10001` | odd, even, even | R |

All produce the identical root `R` because the lower 3 bits are identical.

`verify_transaction_inclusion` passes `args.tx_index` directly to this function with no upper-bound check: [2](#0-1) 

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase check, so it inherits the same flaw: [3](#0-2) 

`tx_index` is a caller-supplied `u64` with no validation: [4](#0-3) 

---

### Impact Explanation

The light client's documented contract is that it verifies `tx_id` is at position `tx_index` in the block's merkle tree. When it returns `true` for `tx_index = N + 2^k`, it is falsely attesting that the transaction occupies a position it does not. Any consuming contract that uses the returned `tx_index` to identify which output, event, or UTXO to process receives a fabricated position while the light client returns `true`. An attacker holding a valid proof for index `N` can replay it with index `N + 2^k` to trigger a different code path in the consumer without constructing any new cryptographic material.

---

### Likelihood Explanation

The attack requires only a valid existing proof and knowledge of the proof depth. No privileged role, key material, or chain manipulation is needed. The attacker supplies a single modified `tx_index` value in a public NEAR view or change call. Both `verify_transaction_inclusion` (public, deprecated but still callable) and `verify_transaction_inclusion_v2` (current production path) are reachable.

---

### Recommendation

Add a bounds check at the start of `compute_root_from_merkle_proof`:

```rust
assert!(
    transaction_position < (1usize << merkle_proof.len()),
    "tx_index exceeds the addressable range of the merkle proof"
);
```

This ensures that every bit of `transaction_position` is actually consumed by the loop, making the index-to-root mapping injective.

---

### Proof of Concept

```rust
#[test]
fn test_index_aliasing() {
    // 4-leaf tree: depth k=2
    let tx0 = H256::from([0u8; 32]);
    let tx1 = H256::from([1u8; 32]);
    let tx2 = H256::from([2u8; 32]);
    let tx3 = H256::from([3u8; 32]);
    let leaves = vec![tx0.clone(), tx1.clone(), tx2.clone(), tx3.clone()];

    // Generate valid proof for index 1
    let proof = merkle_proof_calculator(leaves, 1);
    assert_eq!(proof.len(), 2); // k=2

    // All of these must produce the same root as index 1
    let root_1  = compute_root_from_merkle_proof(tx1.clone(), 1,  &proof);
    let root_5  = compute_root_from_merkle_proof(tx1.clone(), 5,  &proof); // 1 + 2^2
    let root_9  = compute_root_from_merkle_proof(tx1.clone(), 9,  &proof); // 1 + 2^3
    let root_13 = compute_root_from_merkle_proof(tx1.clone(), 13, &proof); // 1 + 2^2 + 2^3

    assert_eq!(root_1, root_5);
    assert_eq!(root_1, root_9);
    assert_eq!(root_1, root_13);
    // All four pass verify_transaction_inclusion with the same proof
}

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
