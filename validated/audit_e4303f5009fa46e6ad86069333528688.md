### Title
Smart Contract Recipients Cannot Claim Airdrop Tokens Due to `msg.sender`-Only Enforcement â€” (`core/contracts/Airdrop.sol`)

---

### Summary

`Airdrop.claim()` hardcodes `msg.sender` as both the Merkle-leaf identity and the token recipient. If a smart contract address is included in the Merkle tree but lacks a function to call `Airdrop.claim()`, its allocated tokens are permanently unclaimable.

---

### Finding Description

In `Airdrop._claim()`, the proof verification and token transfer both use `msg.sender` unconditionally:

```solidity
function _claim(uint32 week, uint256 totalAmount, bytes32[] calldata proof) internal {
    _verifyProof(week, msg.sender, totalAmount, proof);          // leaf = hash(msg.sender, totalAmount)
    SafeERC20.safeTransfer(IERC20(token), msg.sender, totalAmount);
    emit Claim(msg.sender, week, totalAmount);
}
``` [1](#0-0) 

Inside `_verifyProof`, the Merkle leaf is constructed as:

```solidity
bytes32 leaf = keccak256(bytes.concat(keccak256(abi.encode(sender, totalAmount))));
``` [2](#0-1) 

where `sender` is always `msg.sender`. This means the only address that can ever claim a given leaf is the address encoded in that leaf â€” there is no `claimFor(address recipient, ...)` path, no delegated claim, and no owner rescue function.

The public `claim()` entry point provides no alternative:

```solidity
function claim(ClaimProof[] calldata claimProofs) external {
    for (uint32 i = 0; i < claimProofs.length; i++) {
        _claim(claimProofs[i].week, claimProofs[i].totalAmount, claimProofs[i].proof);
    }
}
``` [3](#0-2) 

The `IAirdrop` interface exposes only this single `claim` function with no recipient parameter. [4](#0-3) 

---

### Impact Explanation

Any smart contract address included in a weekly Merkle tree â€” e.g., a DAO treasury, a multisig wallet that lacks a generic `execute()` call, a protocol-owned vault, or any contract deployed without an explicit `Airdrop.claim()` wrapper â€” permanently loses its allocated tokens. The `claimed[week][address]` mapping is never set, but the tokens are also never transferable because no valid `msg.sender` path exists. The airdrop allocation is effectively burned inside the contract.

---

### Likelihood Explanation

Nado is a DEX protocol where protocol-owned subaccounts, DAO treasuries, and smart-contract-based liquidity providers are natural participants. The Merkle tree is constructed off-chain by the owner, and including a contract address (e.g., a multisig or NLP pool address) as a recipient is a realistic operational scenario. No attacker action is required â€” the loss is triggered simply by the contract address being present in the tree and lacking a `claim` wrapper.

---

### Recommendation

Add a `claimFor(address recipient, ClaimProof[] calldata claimProofs)` function that verifies the proof against `recipient` and transfers to `recipient`, callable by anyone. This mirrors the standard pattern used by Uniswap and other protocols for Merkle distributors:

```solidity
function claimFor(address recipient, ClaimProof[] calldata claimProofs) external {
    for (uint32 i = 0; i < claimProofs.length; i++) {
        _claimFor(recipient, claimProofs[i].week, claimProofs[i].totalAmount, claimProofs[i].proof);
    }
}
```

where `_claimFor` replaces the hardcoded `msg.sender` with the `recipient` parameter in both `_verifyProof` and `safeTransfer`.

---

### Proof of Concept

1. Owner registers a Merkle root for week 1 that includes leaf `hash(contractAddress, 1000e18)` where `contractAddress` is a deployed contract with no `claim()` wrapper.
2. Any EOA calls `Airdrop.claim()` â€” reverts because `msg.sender != contractAddress`, so the leaf proof is invalid.
3. `contractAddress` itself cannot call `Airdrop.claim()` because it has no such function.
4. `claimed[1][contractAddress]` remains `0` forever, but the 1000e18 tokens are locked in `Airdrop` with no recovery path â€” no owner rescue function exists in the contract. [1](#0-0)
