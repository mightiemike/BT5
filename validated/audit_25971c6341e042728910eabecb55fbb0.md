Audit Report

## Title
`SwapAllowlistExtension` Per-User Allowlist Bypassed via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking whether the `sender` argument — which is always `msg.sender` of the `pool.swap` call — is allowlisted for the pool. When swaps are routed through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of `pool.swap`, so the extension checks the router's allowlist status rather than the originating user's. A pool admin who allowlists the router to enable router-based swaps for their vetted users inadvertently grants every unprivileged address the ability to swap by routing through the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes and forwards that `sender` value unchanged to every extension in `BEFORE_SWAP_ORDER`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value forwarded above: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the `msg.sender` of that call: [4](#0-3) 

The same pattern applies to `exactOutputSingle` (line 136–137), every hop of `exactInput` (lines 103–112), and `exactOutput` (lines 165–181): [5](#0-4) 

The originating user's address is never visible to the extension. The extension has no mechanism to distinguish a router-mediated swap by an allowlisted user from one by an arbitrary address. No existing guard in the extension or pool checks the true initiator.

## Impact Explanation
A pool protected by `SwapAllowlistExtension` is intended to restrict swaps to a curated set of addresses (e.g., KYC-verified counterparties, specific market makers). Once the router is allowlisted — a natural operational step to enable router-based swaps for vetted users — any address can call `router.exactInputSingle(...)` and the extension passes unconditionally. Unauthorized swappers can consume LP liquidity at oracle-anchored prices, extracting value from LPs who deposited under the assumption that only vetted counterparties could trade. This constitutes a direct loss of LP principal through adversarial flow that the allowlist was designed to block — a High-severity direct loss of user principal.

## Likelihood Explanation
The pool admin must allowlist the router for the bypass to be active. This is a predictable and natural operational step: any pool admin who wants allowlisted users to use the standard periphery router will allowlist the router address. The extension provides no mechanism to distinguish router-mediated swaps by allowlisted users from those by anyone else, creating a forced choice: either block all router swaps or open the bypass. This structural mismatch makes the vulnerable configuration likely in any production deployment combining the allowlist extension with the router.

## Recommendation
1. **Extension-side**: `SwapAllowlistExtension.beforeSwap` should also check the `recipient` argument (the second parameter, currently unnamed/ignored) when `sender` is a known intermediary, or require `sender == recipient` for allowlist enforcement.
2. **Router-side**: `MetricOmmSimpleRouter` should forward the original `msg.sender` through `extensionData` so allowlist extensions can decode and verify the true initiator. The `beforeSwap` interface already accepts `extensionData` as its final parameter.
3. **Documentation**: Until fixed, `SwapAllowlistExtension` NatSpec must warn that allowlisting any intermediary contract (router, multicall, aggregator) grants that contract's entire user base access to the pool.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension in beforeSwap order
  pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
  pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  attacker (0xBad) calls:
    router.exactInputSingle({
      pool: pool,
      recipient: attacker,
      ...
      extensionData: ""
    })

  Execution trace:
    router.exactInputSingle()                          // msg.sender = 0xBad
      → pool.swap(recipient=attacker, ..., extensionData)
          msg.sender = router
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
              allowedSwapper[pool][router] == true  ✓  (passes)
        → swap executes, attacker receives output tokens

  0xBad's address is never checked. The allowlist is fully bypassed.
```
