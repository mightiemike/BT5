### Title
Missing Zero-Address Validation for `_quote` in `Clearinghouse.initialize()` - (File: core/contracts/Clearinghouse.sol)

### Summary
`Clearinghouse.initialize()` accepts a `_quote` address parameter and assigns it directly to the `quote` storage variable without any zero-address guard. If `_quote` is mistakenly set to `address(0)` at deployment, the entire protocol's collateral accounting and token transfer layer is permanently broken with no upgrade path available to recover without redeployment.

### Finding Description
In `Clearinghouse.initialize()`, the `_quote` parameter is assigned directly to `quote` with no zero-address check:

```solidity
function initialize(
    address _endpoint,
    address _quote,
    address _clearinghouseLiq,
    uint256 _spreads,
    address _withdrawPool
) external initializer {
    __Ownable_init();
    setEndpoint(_endpoint);
    quote = _quote;           // ← no require(_quote != address(0))
    clearinghouse = address(this);
    clearinghouseLiq = _clearinghouseLiq;
    spreads = _spreads;
    withdrawPool = _withdrawPool;
    emit ClearinghouseInitialized(_endpoint, _quote);
}
``` [1](#0-0) 

The `quote` variable is the ERC20 collateral token address used throughout the `Clearinghouse` for all deposit, withdrawal, and settlement flows. It is stored in `ClearinghouseStorage` and referenced by downstream logic. [2](#0-1) 

Similarly, `_clearinghouseLiq` and `_withdrawPool` are also stored without zero-address validation: [3](#0-2) 

The same pattern exists in `BaseEngine._initialize()`, where `_clearinghouseAddr`, `_offchainExchangeAddr`, and `_endpointAddr` are all stored without zero-address checks: [4](#0-3) 

And in `BaseWithdrawPool._initialize()`, where `_clearinghouse` and `_verifier` are stored without zero-address checks: [5](#0-4) 

### Impact Explanation
If `_quote` is set to `address(0)` during `Clearinghouse.initialize()`, every ERC20 call routed through the quote token address will revert or silently fail. All user deposits, withdrawals, and collateral settlements become permanently non-functional. Because `initialize()` is protected by the `initializer` modifier, it cannot be called again to correct the value — the proxy would need to be redeployed. The same applies to `_clearinghouseLiq` (liquidation engine address) and `_withdrawPool` (withdrawal pool address): a zero value for either permanently disables liquidations or withdrawals respectively. [6](#0-5) 

### Likelihood Explanation
Deployment misconfiguration is a realistic operational risk, especially across multi-contract systems where addresses are passed as constructor/initializer arguments. The absence of a guard means a typo or script error during deployment silently succeeds and emits a `ClearinghouseInitialized` event with a zero address, providing no on-chain revert signal to catch the mistake. [7](#0-6) 

### Recommendation
Add explicit zero-address guards for all critical address parameters in every `initialize` function:

```solidity
function initialize(
    address _endpoint,
    address _quote,
    address _clearinghouseLiq,
    uint256 _spreads,
    address _withdrawPool
) external initializer {
    require(_endpoint != address(0), "Clearinghouse: zero endpoint");
    require(_quote != address(0), "Clearinghouse: zero quote");
    require(_clearinghouseLiq != address(0), "Clearinghouse: zero liq");
    require(_withdrawPool != address(0), "Clearinghouse: zero withdrawPool");
    __Ownable_init();
    setEndpoint(_endpoint);
    quote = _quote;
    ...
}
```

Apply the same pattern to `BaseEngine._initialize()` and `BaseWithdrawPool._initialize()`. [4](#0-3) [5](#0-4) 

### Proof of Concept
1. Deploy `Clearinghouse` proxy and call `initialize(validEndpoint, address(0), validLiq, spreads, validPool)`.
2. The call succeeds — no revert, `ClearinghouseInitialized` is emitted with `_quote = address(0)`.
3. Any subsequent call that routes through `quote` (e.g., `depositCollateral`) will attempt ERC20 operations on `address(0)`, reverting for all users permanently.
4. `initialize()` cannot be called again due to the `initializer` guard, leaving the contract in a permanently broken state. [1](#0-0)
