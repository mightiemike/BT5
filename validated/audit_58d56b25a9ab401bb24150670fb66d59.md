### Title
`DepositAllowlistExtension` Checks LP Position `owner` Instead of Actual Depositor `sender`, Allowing Full Allowlist Bypass — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual depositor) and checks the `owner` parameter (the LP position owner) against the allowlist. Because `owner` is a free caller-controlled argument to `addLiquidity`, any unprivileged actor can bypass the deposit allowlist by nominating any allowlisted address as `owner`.

---

### Finding Description

The pool calls `_beforeAddLiquidity(msg.sender, owner, ...)` where `msg.sender` is the actual depositor and `owner` is a caller-supplied LP-position recipient. [1](#0-0) 

`ExtensionCalling` faithfully forwards both as the first two positional arguments of the hook: [2](#0-1) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and discarded; only `owner` is tested:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

The allowlist mapping is keyed on `depositor`: [4](#0-3) 

The intent is clearly to gate the **depositor** (the address supplying tokens), not the LP-position owner. The sister extension `SwapAllowlistExtension.beforeSwap` demonstrates the correct pattern — it checks `sender` and discards `recipient`:

```solidity
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
``` [5](#0-4) 

`DepositAllowlistExtension` inverts this: it checks the wrong field.

---

### Impact Explanation

The deposit allowlist is a pool-admin-configured access control gate. Bypassing it allows any unprivileged actor to deposit tokens into a restricted pool. The LP shares are credited to the allowlisted `owner` address (which the attacker does not control), while the attacker's tokens are consumed by the pool. Consequences include:

- **Allowlist bypass**: the pool admin's restriction on who may provide liquidity is rendered ineffective.
- **Forced LP share assignment**: an attacker can credit LP shares to any allowlisted address without that address's consent, potentially complicating their accounting or future removals.
- **Pool composition manipulation**: an attacker can shift bin balances in a restricted pool, affecting oracle-anchored pricing and potentially harming existing LPs through adverse bin selection.

---

### Likelihood Explanation

Exploitation requires no special privilege. Any address can call `addLiquidity` on the pool with `owner` set to any allowlisted address. The allowlist of a live pool is observable on-chain via `allowedDepositor` and `allowAllDepositors`. The attack is trivially repeatable.

---

### Recommendation

Name and check `sender` instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address /*owner*/, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists only `Alice`:
   `allowedDepositor[pool][Alice] = true`
2. `Bob` (not allowlisted) calls:
   `pool.addLiquidity(Alice /*owner*/, salt, deltas, callbackData, extensionData)`
3. The pool calls `_beforeAddLiquidity(Bob /*sender*/, Alice /*owner*/, ...)`.
4. `DepositAllowlistExtension.beforeAddLiquidity` receives `sender = Bob` (discarded) and `owner = Alice`.
5. The check `allowedDepositor[pool][Alice]` is `true` → no revert.
6. Bob's tokens are pulled from his wallet via the liquidity callback and deposited into the pool; LP shares are credited to Alice.
7. Bob has deposited into a pool he was explicitly barred from, bypassing the pool admin's access control entirely.
