### Title
Router-mediated swaps substitute router address for end-user identity in `SwapAllowlistExtension::beforeSwap`, breaking per-user allowlist gating — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

The `SwapAllowlistExtension` is documented as gating `swap` by **swapper address, per pool**. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `swap()` passes `msg.sender` (the router contract) as the `sender` argument to `_beforeSwap`. The extension then checks `allowedSwapper[pool][router]` — not `allowedSwapper[pool][end-user]`. The end-user's identity is permanently lost at the pool boundary.

---

### Finding Description

**Call chain:**

```
User → MetricOmmSimpleRouter::exactInputSingle()
         → pool.swap(recipient, zeroForOne, ..., extensionData)   // msg.sender = router
              → _beforeSwap(msg.sender=router, ...)
                   → ExtensionCalling encodes sender=router
                        → SwapAllowlistExtension::beforeSwap(sender=router, ...)
                             checks: allowedSwapper[pool][router]   ← NOT the user
```

**Step 1 — Pool passes `msg.sender` (router) as `sender`:**

In `MetricOmmPool::swap()`:

```solidity
_beforeSwap(
  msg.sender,   // ← router address when called via router
  recipient,
  ...
);
``` [1](#0-0) 

**Step 2 — `ExtensionCalling` forwards `sender` (router) verbatim to the extension:** [2](#0-1) 

**Step 3 — Extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`:**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool (enforced by `onlyPool`) and `sender` is the router — the actual user is never checked. [4](#0-3) 

**The two broken states this creates:**

| Router allowlisted? | Effect |
|---|---|
| **Yes** | Every user — including those not individually allowlisted — can swap through the router. The per-user allowlist is fully bypassed. |
| **No** | Individually allowlisted users cannot use the router at all, even though they are explicitly permitted. Core router functionality is broken for them. |

A pool admin who wants to allow specific users to swap via the router **must** allowlist the router contract, which simultaneously opens the pool to every user. There is no configuration that simultaneously (a) allows specific users through the router and (b) blocks others.

---

### Impact Explanation

**Broken core pool functionality / unauthorized swap execution.** The `SwapAllowlistExtension` is a core protection mechanism. When the router is allowlisted (the only way to enable router swaps for any user), the allowlist ceases to gate individual swappers. Any unprivileged user can route through `MetricOmmSimpleRouter` and execute swaps on a pool the admin intended to restrict. This is a direct bypass of the configured access control with no additional preconditions.

---

### Likelihood Explanation

The router is the primary public entrypoint for swaps. Any pool that (a) uses `SwapAllowlistExtension` and (b) allowlists the router to enable router-mediated swaps for its permitted users is immediately vulnerable. No special attacker capability is required — only calling `exactInputSingle` / `exactOutputSingle` / `exactInput` / `exactOutput` through the public router.

---

### Note on the "pool paused" framing in the question

The question frames this around a pool "paused for swaps but LP withdrawals remain live." This framing is a **red herring**. `MetricOmmPool::swap()` carries a `whenNotPaused` modifier that reverts unconditionally when `pauseLevel != 0`, **before** any extension hook is reached. [5](#0-4) [6](#0-5) 

A paused pool blocks all swaps entirely; the allowlist hook is never invoked. The "watermark boundary" and "stop-loss threshold" language in the question also has no basis in `SwapAllowlistExtension`, which contains no drawdown or decay logic. The real vulnerability is purely the router identity substitution described above.

---

### Recommendation

Pass the **original caller's identity** through the extension data or a dedicated field, or require the pool to record the true initiator before calling hooks. One concrete fix: have the router pass the end-user address in `extensionData`, and have `SwapAllowlistExtension::beforeSwap` decode and check that address when `sender` is a known router. Alternatively, maintain a router registry and unwrap the payer identity from transient storage before the hook fires.

---

### Proof of Concept

```solidity
// Pool configured with SwapAllowlistExtension:
//   allowAllSwappers[pool] = false
//   allowedSwapper[pool][router] = true   ← admin enables router swaps
//   allowedSwapper[pool][alice] = true    ← alice is individually permitted
//   allowedSwapper[pool][attacker] = false ← attacker is NOT permitted

// Direct swap by attacker → reverts (allowedSwapper[pool][attacker] = false) ✓

// Router swap by attacker:
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    recipient: attacker,
    amountIn: 1e18,
    ...
}));
// pool.swap() called with msg.sender = router
// _beforeSwap(sender=router, ...)
// allowedSwapper[pool][router] = true → PASSES
// Attacker executes swap successfully ← BYPASS
```

The `sender` checked by the hook is the router, not the attacker. The allowlist is bypassed. [7](#0-6) [8](#0-7)
