### Title
Empty Merkle Proof Causes Permanent DoS on `verify_transaction_inclusion` for Single-Transaction Blocks — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` unconditionally panics when the supplied `merkle_proof` is empty. For a block that contains exactly one transaction, the Bitcoin Merkle proof is legitimately empty (the transaction hash *is* the Merkle root). The guard therefore permanently blocks verification of any transaction that lives in a single-transaction block, even when the caller supplies a cryptographically correct proof.

### Finding Description

`verify_transaction_inclusion` contains the following guard before computing the Merkle root:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

`compute_root_from_merkle_proof` is designed to handle an empty proof correctly — with no iterations it simply returns the supplied `transaction_hash` unchanged:

```rust
for proof_hash in merkle_proof {   // zero iterations when proof is empty
    ...
}
current_hash   // returns transaction_hash as-is
``` [2](#0-1) 

For a block whose Merkle tree contains exactly one leaf, the Merkle root equals that leaf's hash. An empty proof is therefore the only correct proof. The `require!` fires before `compute_root_from_merkle_proof` is ever called, so the comparison against `header.block_header.merkle_root` is never reached and the call always panics.

The same defect is inherited by `verify_transaction_inclusion_v2`, which validates the coinbase proof independently and then delegates to `verify_transaction_inclusion`:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

For a single-transaction block the v2 path proceeds as follows:

1. Length check passes — both `merkle_proof` and `coinbase_merkle_proof` are empty, so `0 == 0`.
2. Coinbase proof check passes — `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id`, which equals `merkle_root` for a single-transaction block.
3. The delegated call to `verify_transaction_inclusion` panics with `"Merkle proof is empty"`. [4](#0-3) 

### Impact Explanation

Any NEAR consumer contract that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction included in a single-transaction block will always receive a panic, regardless of proof correctness. The proof result is permanently corrupted for this class of blocks: a valid inclusion proof is treated as an invalid one. Cross-chain applications (atomic swaps, bridged assets, cross-chain state verification) that depend on the verification result are blocked for the affected blocks.

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) are uncommon on Bitcoin mainnet but are valid protocol-level constructs and occur regularly on testnets and on the other supported chains (Dogecoin, Litecoin, Zcash). Because the contract accepts any Uniswap-style open submission model — any relayer can submit any valid chain header — such blocks will inevitably appear in the on-chain header store. No privileged access is required to trigger the failure; any unprivileged NEAR account can call `verify_transaction_inclusion_v2` with a legitimately empty proof and cause the panic.

### Recommendation

Remove the blanket empty-proof guard and instead allow `compute_root_from_merkle_proof` to handle the empty case (which it already does correctly). If a non-empty proof is required for security reasons, add the check only after confirming the block contains more than one transaction:

```diff
-       require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
-
        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
```

### Proof of Concept

1. Relay a block whose coinbase transaction is the only transaction (Merkle root == coinbase tx hash).
2. Call `verify_transaction_inclusion_v2` with:
   - `tx_id` = coinbase tx hash
   - `tx_index` = 0
   - `merkle_proof` = `[]`
   - `coinbase_tx_id` = coinbase tx hash
   - `coinbase_merkle_proof` = `[]`
   - `confirmations` = 1
3. Step 1 of v2 passes (`0 == 0`). Step 2 passes (`coinbase_tx_id == merkle_root`). Step 3 panics: `"Merkle proof is empty"`.
4. The call reverts; the consumer contract receives no usable result despite supplying a cryptographically valid proof. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
