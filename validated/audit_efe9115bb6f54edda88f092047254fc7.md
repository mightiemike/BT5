### Title
Gateway Admits `InvokeV3` Transactions with Non-Empty `account_deployment_data`, Which the OS Always Rejects via Hard-Coded Assertion — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The Starknet OS `compute_invoke_transaction_hash` function contains a hard-coded assertion `assert account_deployment_data_size = 0` that unconditionally aborts for any `InvokeV3` transaction carrying non-empty `account_deployment_data`. No corresponding admission gate exists in the gateway's stateless validator, the blockifier's pre-validation stage, or the mempool. An unprivileged user can submit such a transaction, have it accepted, executed by the blockifier, and committed into a block, after which the OS will always fail to prove that block.

---

### Finding Description

**OS-side hard assertion (always fails for non-empty field)**

In `compute_invoke_transaction_hash`, the very first executable statement is:

```cairo
// TODO(Noa, 01/01/2026): remove the following `assert` once the field is supported.
assert account_deployment_data_size = 0;
``` [1](#0-0) 

This assertion precedes all hash computation. Any `InvokeV3` transaction with `account_deployment_data.len() > 0` causes the OS to abort block proving unconditionally.

**Rust hash computation does not mirror the restriction**

`get_invoke_transaction_v3_hash` always includes `account_deployment_data_hash` in the hash chain, regardless of whether the field is empty:

```rust
let account_deployment_data_hash = HashChain::new()
    .chain_iter(transaction.account_deployment_data().0.iter())
    .get_poseidon_hash();
let mut hash_chain = HashChain::new()
    ...
    .chain(&account_deployment_data_hash)
    .chain(&calldata_hash);
``` [2](#0-1) 

For empty `account_deployment_data`, both the Rust and OS hash computations agree (hash of empty sequence). For non-empty `account_deployment_data`, the Rust hash succeeds and produces a valid-looking `tx_hash`, while the OS assertion fails before any hash is computed.

**No admission gate in the gateway stateless validator**

`StatelessTransactionValidator` validates resource bounds, calldata length, signature length, Sierra version, and proof-facts consistency, but contains no check that `account_deployment_data` is empty for invoke transactions: [3](#0-2) 

**No admission gate in the blockifier pre-validation stage**

`perform_pre_validation_stage` calls `validate_proof_facts`, `check_fee_bounds`, `handle_nonce`, and `verify_can_pay_committed_bounds`. None of these inspect `account_deployment_data`: [4](#0-3) 

**The `InternalRpcInvokeTransactionV3` struct carries the field without restriction**

The internal representation accepts `account_deployment_data` as a plain field with no invariant enforcing it is empty: [5](#0-4) 

**Conversion from RPC to internal preserves the field verbatim** [6](#0-5) 

**Attack path:**
1. Submit `RpcInvokeTransactionV3` with `account_deployment_data: AccountDeploymentData(vec![Felt::ONE])` via `starknet_addInvokeTransaction`.
2. Gateway stateless validator accepts (no `account_deployment_data` check).
3. Blockifier executes the transaction and includes it in a block.
4. OS calls `compute_invoke_transaction_hash` with `account_deployment_data_size = 1`.
5. `assert account_deployment_data_size = 0` fails → entire block proof aborts.

---

### Impact Explanation

A block containing such a transaction cannot be proven by the OS. The sequencer has already committed the block and applied state changes, but L1 finalization is permanently blocked for that block. This matches the Critical impact category: **Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.** The Rust-side hash and the OS-side hash diverge for the same transaction object, breaking the canonicalization invariant between the execution layer and the proving layer.

---

### Likelihood Explanation

Any unprivileged user can craft and submit an `InvokeV3` transaction with non-empty `account_deployment_data` via the public JSON-RPC endpoint. The field is part of the public `RpcInvokeTransactionV3` struct. No special privileges, keys, or insider access are required. The trigger is a single well-formed RPC call.

---

### Recommendation

Add a check in `StatelessTransactionValidator` to reject `InvokeV3` transactions with non-empty `account_deployment_data` until the OS assertion is removed and the field is fully supported. This mirrors the existing `validate_client_side_proving_allowed` pattern used for proof facts. Alternatively, add the check in the blockifier's `perform_pre_validation_stage` as a defense-in-depth measure.

---

    fn validate_client_side_proving_allowed(
        &self,
        tx: &RpcInvokeTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if self.config.allow_client_side_proving {
            return Ok(());
        }

        // Reject V3 transactions with proofs when client-side proving is disabled.
        let RpcInvokeTransaction::V3(tx) = tx;
        let has_proof_data = !tx.proof_facts.is_empty() || !tx.proof.is_empty();
        if has_proof_data {
            return Err(StatelessTransactionValidatorError::ClientSideProvingNotAllowed);
        }

        Ok(())
    }

    fn validate_proof_facts_and_proof_consistency(
        &self,
        tx: &RpcInvokeTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let RpcInvokeTransaction::V3(tx) = tx;
        let has_proof_facts = !tx.proof_facts.is_empty();
        let has_proof = !tx.proof.is_empty();
        if has_proof_facts != has_proof {
            return Err(StatelessTransactionValidatorError::ProofFactsAndProofConsistency {
                has_proof_facts,
                has_proof,
            });
        }
        Ok(())
    }

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt())
        {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof version {} is not allowed under this protocol version.",
                snos_proof_facts.proof_version
            )));
        }

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let virtual_os_config_hash = block_context.virtual_os_config_hash();
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
