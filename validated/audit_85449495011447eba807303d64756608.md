### Title
Accrued Spot Interest Lost on Withdrawal Due to Missing `updateStates` Call Before Balance Reduction — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

The `withdrawCollateral` function in `Clearinghouse.sol` reduces a user's spot balance without first advancing the `cumulativeDepositsMultiplierX18` via `spotEngine.updateStates()`. Because actual balances are computed as `amountNormalized × cumulativeDepositsMultiplierX18`, any interest accrued since the last `SpotTick` transaction is silently lost for the withdrawing user — a direct structural analog to the Sperax `rebase()`-after-burn ordering flaw.

---

### Finding Description

The `SpotEngine` tracks depositor balances in normalized form. A user's real balance is:

```
actualBalance = amountNormalized × cumulativeDepositsMultiplierX18
```

The multiplier grows over time as interest accrues. It is advanced exclusively by `updateStates(dt)`, which is only callable via `SpotTick` transactions submitted by the sequencer. [1](#0-0) 

When a user withdraws, `_updateBalanceNormalized` is called with the **current** (potentially stale) multiplier:

```
newAmount = amountNormalized × stale_multiplier − withdrawAmount
newAmountNormalized = newAmount / stale_multiplier
``` [2](#0-1) 

After `SpotTick` eventually advances the multiplier to `newMultiplier`, the user's remaining balance becomes:

```
remaining = newAmountNormalized × newMultiplier
          = (amountNormalized × stale_multiplier − withdrawAmount) / stale_multiplier × newMultiplier
```

Had `updateStates` been called **before** the withdrawal, the remaining balance would be:

```
remaining_correct = amountNormalized × newMultiplier − withdrawAmount
```

The difference — the interest lost — is:

```
loss = withdrawAmount × (newMultiplier / stale_multiplier − 1)
```

The entire `withdrawCollateral` call chain contains no `updateStates` invocation: [3](#0-2) 

Neither the fast-path (`WithdrawCollateral` / `WithdrawCollateralV2`) nor the slow-mode path in `EndpointTx` calls `updateStates` before reducing the balance: [4](#0-3) [5](#0-4) 

The `SpotTick` handler that does call `updateStates` is a completely separate transaction type with no ordering guarantee relative to withdrawals: [6](#0-5) 

---

### Impact Explanation

Every user who withdraws collateral between two `SpotTick` transactions loses interest proportional to:

- The **withdrawal amount** (larger withdrawals lose more)
- The **deposit interest rate** for the product
- The **elapsed time** since the last `SpotTick`

The corrupted state delta is `amountNormalized` for the withdrawing subaccount: it is reduced by more than the true withdrawal warrants, permanently understating the user's remaining balance. Unlike the Sperax case, the user has **no self-help remedy** — `updateStates` is `onlyEndpoint` and cannot be called by the user directly. [7](#0-6) 

---

### Likelihood Explanation

`SpotTick` is called periodically (not before every withdrawal). In normal sequencer operation, any withdrawal processed in the same batch as or after a `SpotTick` interval boundary will silently lose interest. This is not a sequencer-compromise scenario — it is the ordinary operational cadence. Every withdrawal between two `SpotTick` calls is affected. The slow-mode path (user-submitted after a 3-day delay) is also affected and requires no sequencer involvement at all. [8](#0-7) 

---

### Recommendation

Inside `Clearinghouse.withdrawCollateral`, call `spotEngine.updateStates` (or an equivalent single-product state refresh) for the relevant `productId` before calling `spotEngine.updateBalance`. This mirrors the correct pattern: advance the cumulative multiplier first, then apply the balance delta, so the user's accrued interest is materialized before their normalized balance is reduced. [9](#0-8) 

---

### Proof of Concept

1. At time T₀, `SpotTick` is processed: `cumulativeDepositsMultiplierX18 = M₀`.
2. Alice holds `amountNormalized = N`, so her actual balance = `N × M₀`.
3. At time T₁ (before the next `SpotTick`), Alice submits `WithdrawCollateral` for amount `W`.
4. `withdrawCollateral` calls `spotEngine.updateBalance(productId, alice, -W)`.
5. `_updateBalanceNormalized` computes: `newAmountNormalized = (N × M₀ − W) / M₀`.
6. At time T₂, `SpotTick` is processed: multiplier advances to `M₁ > M₀`.
7. Alice's remaining balance = `(N × M₀ − W) / M₀ × M₁`.
8. Correct remaining balance (had `updateStates` been called at T₁ first) = `N × M₁ − W`.
9. **Loss = `W × (M₁/M₀ − 1)`** — interest on the withdrawn principal for the period `[T₀, T₁]` is permanently lost. [10](#0-9) [11](#0-10)

    function _updateBalanceNormalized(
        State memory state,
        BalanceNormalized memory balance,
        int128 balanceDelta
    ) internal pure {
        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized -= balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized += balance.amountNormalized;
        }

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        int128 newAmount = balance.amountNormalized.mul(
            cumulativeMultiplierX18
        ) + balanceDelta;

        if (newAmount > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);

        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized += balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized -= balance.amountNormalized;
        }
    }

    function updateStates(uint128 dt) external onlyEndpoint {
        State memory quoteState;
        require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
        for (uint32 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            if (productId == NLP_PRODUCT_ID) {
                continue;
            }
            State memory state = states[productId];
            if (productId == QUOTE_PRODUCT_ID) {
                quoteState = state;
            }
            if (state.totalDepositsNormalized == 0) {
                continue;
            }
            _updateState(productId, state, dt);
            _setState(productId, state);
        }
    }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
