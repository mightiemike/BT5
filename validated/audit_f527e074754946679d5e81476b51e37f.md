Audit Report

## Title
Unguarded `refundETH` drains any ETH left on the router to an arbitrary caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH()` forwards the router's entire native ETH balance to `msg.sender` with no per-depositor accounting or access control. Because `multicall` is `payable` and `pay()` wraps only the exact swap-required amount of ETH, any excess `msg.value` remains on the router after the swap completes. A third party who contributed no ETH can call `refundETH()` in a subsequent transaction and receive that balance, causing direct loss of user principal.

## Finding Description
`refundETH()` reads `address(this).balance` and transfers the full amount to `msg.sender` unconditionally: [1](#0-0) 

There is no mapping of depositor → deposited amount, no check that `msg.sender` contributed any ETH, and no reentrancy guard.

`pay()` wraps exactly the swap-required amount when the native balance covers it, leaving any excess ETH on the contract: [2](#0-1) 

`multicall` is `payable` and uses `delegatecall`, so `msg.value` is visible to every sub-call but is not consumed or tracked beyond what `pay()` uses: [3](#0-2) 

The `receive()` guard (only WETH can push ETH in) does not protect against ETH arriving via `msg.value` in a payable call — it only blocks direct ETH transfers from non-WETH addresses: [4](#0-3) 

**Exploit flow:**
1. User A calls `router.multicall{value: N + excess}(...)` with a swap for exactly `N` ETH.
2. `pay()` wraps `N` ETH; `excess` ETH remains on the router.
3. User B (attacker) calls `router.refundETH()` in a subsequent transaction.
4. `refundETH()` sends `excess` ETH to User B. User A's funds are permanently lost.

No privileged role, malicious pool, or non-standard token is required.

## Impact Explanation
Direct loss of user principal. Any ETH sent above the exact swap amount is permanently claimable by any caller of `refundETH()`. This meets the "Critical/High/Medium direct loss of user principal" allowed impact gate. Severity is **Medium**: the loss is bounded to the excess ETH per transaction, and exploitation requires User A to not bundle `refundETH()` atomically in the same multicall.

## Likelihood Explanation
Any user who sends more ETH than the exact swap amount is exposed — a common pattern with `exactOutputSingle` where the exact input is not known ahead of time. The attack window is the time between User A's swap transaction and their `refundETH()` call. A mempool-watching bot can reliably front-run this. The only mitigating factor is that a careful user who bundles `refundETH()` in the same `multicall` is not affected.

## Recommendation
**Preferred fix:** Track per-caller ETH deposits in transient storage at the start of each `multicall` / payable entry point and restrict `refundETH()` to return only the caller's recorded deposit.

**Minimum fix:** Enforce that `refundETH()` can only be called via `delegatecall` (i.e., as part of a `multicall`), ensuring it is always atomic with the swap that created the excess and preventing standalone calls from a different `msg.sender`.

## Proof of Concept
```solidity
function test_refundETH_frontrun() public {
    // User A: swap 1 ETH worth, but sends 2 ETH
    bytes[] memory calls = new bytes[](1);
    calls[0] = abi.encodeCall(router.exactInputSingle, (paramsFor1ETH));
    router.multicall{value: 2 ether}(calls);
    // 1 ETH now sits on the router

    // User B: front-runs refundETH
    vm.prank(userB);
    router.refundETH();

    assertEq(userB.balance, 1 ether);   // User B received User A's excess ETH
    assertEq(address(router).balance, 0);
}
```
The assertion passes because `refundETH()` sends `address(this).balance` to `msg.sender` (User B) with no check that User B contributed any ETH.
