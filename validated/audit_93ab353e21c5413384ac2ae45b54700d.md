### Title
Intermediate `int128` Overflow in Decimal-Normalization Multiplication Causes Deposit and Fast-Withdrawal DoS — (`core/contracts/Clearinghouse.sol`, `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`Clearinghouse.depositCollateral` and `BaseWithdrawPool.fastWithdrawalFeeAmount` both multiply a user-supplied `uint128` amount by a decimal-normalization `multiplier` entirely within `int128` arithmetic. The guard `require(txn.amount <= INT128_MAX)` only validates the raw amount fits in `int128`; it does **not** validate that `amount × multiplier` fits. For any token whose decimals are less than 18, the multiplier is greater than 1, and the product can overflow `int128` even when the raw amount is individually valid. Under Solidity 0.8's checked arithmetic this causes a revert, permanently blocking deposits (and fast withdrawals) for amounts above the true safe threshold.

---

### Finding Description

`MAX_DECIMALS = 18` and `INT128_MAX = uint128(type(int128).max) ≈ 1.7 × 10^38`.

**`Clearinghouse.depositCollateral` (line 199–205):**

```solidity
require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);   // line 199
...
int256 multiplier = int256(10**(MAX_DECIMALS - decimals));     // line 204
int128 amountRealized = int128(txn.amount) * int128(multiplier); // line 205 ← overflow
```

The guard at line 199 only ensures `txn.amount ≤ INT128_MAX`. The multiplication at line 205 is performed in `int128` space. For a token with `decimals = 6` (e.g. USDC), `multiplier = 10^12`. Any `txn.amount > INT128_MAX / 10^12 ≈ 1.7 × 10^26` raw units will overflow `int128` and revert the entire deposit transaction.

The identical pattern is repeated in:
- `depositInsurance` (line 265) [1](#0-0) 
- `withdrawInsurance` (line 282) [2](#0-1) 
- `checkMinDeposit` (line 708) [3](#0-2) 

**`BaseWithdrawPool.fastWithdrawalFeeAmount` (line 141–142):**

```solidity
int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
int128 amountX18 = int128(amount) * int128(multiplier);   // line 142 ← overflow
```

The caller at line 100 guards `transferAmount <= INT128_MAX` but not `transferAmount × multiplier`. For the same reason, a fast-withdrawal of a large amount of a low-decimal token reverts inside `fastWithdrawalFeeAmount`. [4](#0-3) 

The root cause in both files is that `int128(multiplier)` is computed from an `int256` value and then multiplied against another `int128` without first widening to `int256`. [5](#0-4) 

---

### Impact Explanation

- **`depositCollateral`**: Any user depositing a low-decimal token in an amount exceeding `INT128_MAX / multiplier` receives a permanent revert. Because the ERC-20 transfer and the balance update are atomic, no funds are lost — but the deposit is irrevocably blocked at that amount. The user cannot split the deposit into smaller chunks and later consolidate, because the internal balance is stored in X18 units; the overflow occurs at the normalization step, not at the storage step.
- **`fastWithdrawalFeeAmount`**: A user who has accumulated a large balance (via many smaller deposits) and attempts a single large fast withdrawal will have the fee calculation revert, blocking the fast-withdrawal path entirely for that amount. The slow-withdrawal path is unaffected, so this is a partial DoS.
- **`depositInsurance` / `withdrawInsurance`**: Insurance operations on low-decimal quote tokens are similarly blocked above the threshold.

---

### Likelihood Explanation

The overflow threshold for a 6-decimal token is `≈ 1.7 × 10^26` raw units (`≈ 1.7 × 10^20` USDC). For a 0-decimal token the threshold is `≈ 1.7 × 10^20` raw units. These are very large nominal amounts, making accidental triggering unlikely for common tokens. However, the code accepts any amount up to `INT128_MAX` without the correct per-multiplier bound, so the defect is structurally present and reachable by any unprivileged depositor who submits a sufficiently large amount. The likelihood is **low** but the root cause is concrete and the entry path is fully user-controlled.

---

### Recommendation

Replace the in-`int128` multiplication with a widened `int256` intermediate, then range-check before downcasting — exactly as `MathSD21x18` already does for all its own operations:

```solidity
// Before (unsafe):
int128 amountRealized = int128(txn.amount) * int128(multiplier);

// After (safe):
int256 amountRealizedWide = int256(int128(txn.amount)) * multiplier;
require(
    amountRealizedWide >= type(int128).min &&
    amountRealizedWide <= type(int128).max,
    ERR_CONVERSION_OVERFLOW
);
int128 amountRealized = int128(amountRealizedWide);
```

Apply the same fix to `depositInsurance`, `withdrawInsurance`, `checkMinDeposit`, and `fastWithdrawalFeeAmount`.

---

### Proof of Concept

1. Deploy Nado with a 6-decimal ERC-20 token registered as a spot product.
2. Approve and call `depositCollateral` with `txn.amount = 2 × 10^26` (raw units; well below `INT128_MAX ≈ 1.7 × 10^38`, so the existing guard passes).
3. Inside `depositCollateral`, `multiplier = 10^12` and `int128(2 × 10^26) * int128(10^12) = 2 × 10^38 > INT128_MAX`.
4. Solidity 0.8 checked arithmetic panics; the transaction reverts.
5. The user's deposit is permanently blocked at this amount despite the raw value being individually valid. [6](#0-5) [7](#0-6) [8](#0-7)

    function fastWithdrawalFeeAmount(
        IERC20Base token,
        uint32 productId,
        uint128 amount

        uint8 decimals = token.decimals();
        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
        int128 amountX18 = int128(amount) * int128(multiplier);

    }
