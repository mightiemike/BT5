### Title
`MintNlp` and `BurnNlp` Signed Transactions Lack a Deadline, Exposing Users to Unfavorable NLP Price Execution — (File: `core/contracts/interfaces/IEndpoint.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` user-signed structs contain no `deadline` or expiry field. The oracle price used to compute the NLP amount received (on mint) or the quote amount returned (on burn) is supplied by the sequencer at execution time and is **not** committed to by the user's signature. If the sequencer delays execution and the NLP price moves adversely, the user receives materially less value than they expected when they signed, with no on-chain protection.

---

### Finding Description

`MintNlp` and `BurnNlp` are the two price-sensitive user-initiated operations in the Nado protocol. Their signed structs are:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint64  nonce;
    // ← no deadline
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64  nonce;
    // ← no deadline
}
```

The user's ECDSA signature covers only these fields. The oracle price is carried in the outer, **unsigned** wrapper:

```solidity
struct SignedMintNlp {
    MintNlp  tx;
    bytes    signature;
    int128   oraclePriceX18;        // ← NOT signed by user
    int128[] nlpPoolRebalanceX18;
}

struct SignedBurnNlp {
    BurnNlp  tx;
    bytes    signature;
    int128   oraclePriceX18;        // ← NOT signed by user
    int128[] nlpPoolRebalanceX18;
}
```

At execution time in `EndpointTx.processTransactionImpl`, the sequencer-supplied `oraclePriceX18` is passed directly to `Clearinghouse.mintNlp` / `burnNlp` without any check against a user-specified bound:

```solidity
clearinghouse.mintNlp(
    signedTx.tx,
    signedTx.oraclePriceX18,   // sequencer-controlled
    nlpPools,
    signedTx.nlpPoolRebalanceX18
);
```

Inside `Clearinghouse.mintNlp`, the NLP amount is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
```

And inside `Clearinghouse.burnNlp`:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
```

There is no `minAmountOut` guard and no deadline. The user has no on-chain mechanism to say "execute this before timestamp T, otherwise revert."

---

### Impact Explanation

**Impact: High**

- For `MintNlp`: if the NLP price rises between the time the user signs and the time the sequencer executes, `quoteAmount.div(oraclePriceX18)` yields fewer NLP tokens. The user pays the same quote but receives less NLP.
- For `BurnNlp`: if the NLP price falls, `nlpAmount.mul(oraclePriceX18)` yields less quote. The user burns the same NLP but receives less quote.
- There is no slippage floor (`minAmountOut`) in either struct, so the loss is unbounded.
- The corrupted state delta is the user's NLP balance (mint) or quote balance (burn) — a direct, measurable asset loss.

---

### Likelihood Explanation

**Likelihood: Low**

The sequencer must delay execution of the signed transaction while the NLP price moves adversely. This requires either sequencer malice or a period of sequencer unavailability followed by a large NLP price move. The scenario is analogous to the original report's "low gas fee → stale mempool transaction" vector: the signed intent is valid indefinitely, and the price at execution is outside the user's control.

---

### Recommendation

1. Add a `uint64 expiration` field to both `MintNlp` and `BurnNlp` structs (mirroring the `Order.expiration` pattern already used in `OffchainExchange`).
2. In `Clearinghouse.mintNlp` and `Clearinghouse.burnNlp`, revert if `block.timestamp > txn.expiration`.
3. Optionally add a `minAmountOut` field (e.g., `minNlpAmount` for mint, `minQuoteAmount` for burn) to give users slippage protection independent of the deadline.

---

### Proof of Concept

1. Alice signs `MintNlp { sender: alice, quoteAmount: 1000e18, nonce: 1 }` when NLP price is `1.00 USDC`.
   - Expected NLP received: `1000e18 / 1.00e18 = 1000 NLP`.
2. The sequencer queues the transaction but does not execute it immediately.
3. NLP price rises to `2.00 USDC` (e.g., due to protocol profits accruing to the pool).
4. Sequencer executes the transaction with `oraclePriceX18 = 2.00e18`.
5. `Clearinghouse.mintNlp` computes `nlpAmount = 1000e18 / 2.00e18 = 500 NLP`.
6. Alice receives only 500 NLP instead of the 1000 NLP she expected — a 50% loss — with no on-chain recourse because her signed struct contained no deadline or minimum output.
