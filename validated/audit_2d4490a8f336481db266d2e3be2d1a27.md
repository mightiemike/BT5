### Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed from the pool. When a user routes through `MetricOmmSimpleRouter`, the pool's `swap` function sets `sender = msg.sender`, which is the **router address**, not the end user. The extension therefore checks whether the router is allowlisted, not whether the actual economic actor is allowlisted. If the pool admin allowlists the router to enable router-mediated swaps for permitted users, every unpermitted user can bypass the allowlist by routing through the same public router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value received from the pool — the router: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the `msg.sender` of that call: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

The pool's own documentation acknowledges the operator-pattern separation: `msg.sender` pays but need not equal the economic actor. The allowlist is supposed to gate the economic actor, but it gates the intermediary instead. [6](#0-5) 

---

### Impact Explanation

**Allowlist bypass (direct fund impact):** A pool admin who wants to restrict swaps to KYC'd or permissioned addresses allowlists those addresses and also allowlists the router so those users can interact via the standard periphery. Once the router is allowlisted, `allowedSwapper[pool][router] == true`, and the extension's check passes for **any** caller who routes through the router — including addresses that were never permitted. Unauthorized users can execute swaps in a pool that was designed to be access-controlled, receiving pool output tokens they should not be entitled to and draining LP assets at oracle-derived prices.

**Broken access control (secondary impact):** If the admin does not allowlist the router (to avoid the bypass), then all allowlisted users are forced to call `pool.swap` directly and cannot use the standard router. This breaks the intended UX and may cause users to interact with the pool in unintended ways.

---

### Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` and also permits router-mediated swaps (the standard user path) is vulnerable. The router is a public, permissionless contract. No special privilege or flash-loan is required — a plain `exactInputSingle` call suffices. The only precondition is that the pool admin has allowlisted the router address, which is the natural configuration for a pool that wants to restrict users but still support the periphery.

---

### Recommendation

The `SwapAllowlistExtension` must gate on the **originating user**, not the immediate caller of `pool.swap`. Two approaches:

1. **Pass the original user through the router:** Have `MetricOmmSimpleRouter` encode `msg.sender` (the end user) into `extensionData` and have the extension decode and check that value. This requires a convention between the router and the extension.

2. **Check `tx.origin` as a fallback:** Not recommended in general, but acceptable in a controlled extension context where the pool admin controls which extensions are attached.

3. **Preferred — gate on `sender` only for direct pool calls; require router to pass user identity explicitly:** The pool could expose a separate `swapWithSender(address trueSender, ...)` entry point that the router calls, forwarding the real user. The extension then checks `trueSender`.

The simplest safe fix is to have the router pass the real user address as part of `extensionData` and update `SwapAllowlistExtension.beforeSwap` to decode and check it when present.

### Proof of Concept

```
Setup:
  pool = deploy MetricOmmPool with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)       // alice is KYC'd
  admin calls setAllowedToSwap(pool, router, true)      // enable router for alice

Attack:
  eve (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: eve, ...})

Execution trace:
  router.exactInputSingle()
    → pool.swap(recipient=eve, ...)          // msg.sender = router
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (bypass!)
      → swap executes, eve receives tokens

Result:
  Eve, who was never allowlisted, successfully swaps in a restricted pool.
  The allowlist invariant is broken; LP assets flow to an unauthorized recipient.
