Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address as the swapper identity, enabling full allowlist bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `pool.swap()` — the router address when swaps are routed through `MetricOmmSimpleRouter`. If the pool admin allowlists the router (the natural action to enable router-based swaps), every unpermitted user can bypass the per-user restriction by routing through the router. Alternatively, if the router is not allowlisted, even permitted users cannot use the standard periphery swap path.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs the check at L37:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` is the pool (correct namespace key). `sender` is the first argument forwarded by the pool's extension machinery.

In `MetricOmmPool.swap()` at L230–231, the pool calls:
```solidity
_beforeSwap(
    msg.sender,   // ← becomes `sender` in the extension
    ...
);
```

`ExtensionCalling._beforeSwap` at L160–176 forwards `sender` unchanged via `abi.encodeCall`. When `MetricOmmSimpleRouter.exactInputSingle` (L72–80) or `exactOutputSingle` (L136–137) calls `pool.swap(...)`, `msg.sender` inside the pool is the **router contract address**. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` at L38 correctly checks `allowedDepositor[msg.sender][owner]` where `owner` is the explicit position-owner parameter passed to `addLiquidity` — not `sender` (the caller). The `swap()` function has no equivalent explicit "swapper" parameter, so there is no way for the extension to recover the actual initiating user from the call arguments.

**Exploit path (Scenario A — Allowlist bypass):**
1. Pool admin deploys pool with `SwapAllowlistExtension` as `beforeSwap` hook.
2. Admin calls `setAllowAllSwappers(pool, false)` (deny-all default).
3. Admin calls `setAllowedToSwap(pool, alice, true)` (permit alice).
4. Admin calls `setAllowedToSwap(pool, router, true)` (permit router so alice can use periphery).
5. Charlie (not on allowlist) calls `router.exactInputSingle({pool: pool, ...})`.
6. Router calls `pool.swap(recipient=charlie, ...)`.
7. Pool calls `_beforeSwap(msg.sender=router, ...)`.
8. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
9. Charlie's swap executes in a pool that should have blocked him.

**Scenario B — Broken core swap functionality:** If the admin does not allowlist the router, `allowedSwapper[pool][router]` is `false`, and all router-based swaps revert with `NotAllowedToSwap` even for allowlisted users. The only usable path is calling `pool.swap()` directly and implementing `metricOmmSwapCallback`, which is not the supported periphery path.

Existing guards are insufficient: the extension has no access to the original EOA initiating the router call; the `sender` parameter is structurally bound to `msg.sender` of `pool.swap()`.

## Impact Explanation
Scenario A constitutes a direct admin-boundary break: an unprivileged user bypasses a curated pool's swap allowlist, gaining unauthorized access to swap in a pool restricted to KYC'd or otherwise permitted addresses. This defeats the core invariant of the `SwapAllowlistExtension` — that only explicitly permitted addresses may swap. Scenario B constitutes broken core swap functionality: the standard periphery path is rendered unusable for legitimate allowlisted users, forcing them to interact directly with the pool contract and implement the swap callback themselves.

## Likelihood Explanation
Medium-High. `MetricOmmSimpleRouter` is the documented production periphery swap path. Any pool deploying `SwapAllowlistExtension` that also wants users to swap through the router must allowlist the router — which is the natural and expected configuration. Once the router is allowlisted, the bypass requires no special privilege: any address can call the public `exactInputSingle` or `exactOutput` router functions. The attacker needs only a funded wallet and knowledge of the pool address.

## Recommendation
The extension must gate on the actual initiating user, not the intermediary router. Two viable approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`. The extension decodes and verifies it. This requires a trusted encoding convention between the router and the extension.

2. **Add an explicit `swapper` parameter to `pool.swap()`**: The pool's `swap()` signature accepts a `swapper` address (analogous to how `addLiquidity` accepts `owner`), and the pool passes it to `_beforeSwap` instead of `msg.sender`. The router sets `swapper = msg.sender` (the actual user). This mirrors how `DepositAllowlistExtension` correctly checks `owner` rather than `sender`.

## Proof of Concept
```
1. Deploy MetricOmmPool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls setAllowAllSwappers(pool, false).
3. Pool admin calls setAllowedToSwap(pool, alice, true).
4. Pool admin calls setAllowedToSwap(pool, router, true).
5. charlie (not on allowlist) calls:
       router.exactInputSingle(ExactInputSingleParams{pool: pool, recipient: charlie, ...})
6. Router calls pool.swap(recipient=charlie, ...) — msg.sender inside pool = router.
7. Pool calls _beforeSwap(sender=router, ...).
8. Extension evaluates: allowedSwapper[pool][router] == true → passes.
9. charlie's swap executes successfully in a pool that should have blocked him.

Root cause: SwapAllowlistExtension.sol:37 checks `sender` (the router) instead of the actual
initiating user; MetricOmmPool.sol:231 unconditionally forwards msg.sender as `sender` to all
swap hooks with no mechanism to recover the original EOA.
```
