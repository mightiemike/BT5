Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on LP Recipient (`owner`) Instead of Actual Caller (`sender`), Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the real `msg.sender` who called `addLiquidity`) and instead gates on `owner` (the caller-supplied LP position recipient). Because `owner` is an unconstrained input, any address can bypass the allowlist by passing any allowlisted address as `owner`. The sibling `SwapAllowlistExtension` correctly gates on `sender`, confirming the asymmetry is a defect.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` argument as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both and dispatches them to the extension in order `(sender, owner, ...)`: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, ...)` but names the first parameter `address` (unnamed, discarded) and gates exclusively on `owner`: [3](#0-2) 

`SwapAllowlistExtension.beforeSwap` correctly gates on `sender`: [4](#0-3) 

Because `owner` is a free parameter with no on-chain constraint, any caller can supply any allowlisted address as `owner` and pass the check. No existing guard prevents this: `addLiquidity` imposes no restriction on who may specify any `owner` value, and the extension itself performs no cross-check between `sender` and `owner`.

## Impact Explanation

An address not on the deposit allowlist can call `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)`. The hook evaluates `allowedDepositor[pool][allowlistedAddress]` → `true` → no revert. The unauthorized caller supplies tokens via the callback and the LP position is minted to the allowlisted `owner`. Every deposit restriction configured by the pool admin is rendered completely ineffective. Secondary impact: a legitimately allowlisted `sender` who specifies a non-allowlisted `owner` (e.g., a vault or sub-account) is incorrectly blocked, breaking normal LP workflows.

## Likelihood Explanation

Exploitation requires no special privilege. The allowlist state (`allowedDepositor`, `AllowedToDepositSet` events) is fully public on-chain. Any actor can identify a valid `owner` address and call `addLiquidity` with it. The only prerequisite is the ability to satisfy the token-transfer callback, which is a standard requirement for any liquidity provider. The attack is repeatable and permissionless.

## Recommendation

Replace the unnamed first parameter with `sender` and gate on it, mirroring `SwapAllowlistExtension`:

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

1. Pool deployed with `DepositAllowlistExtension`; `allowAllDepositors[pool] = false`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` — only `alice` is permitted.
3. `bob` (not allowlisted) calls `pool.addLiquidity(alice /*owner*/, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`.
5. Extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
6. `bob`'s callback pays the tokens; LP position minted to `alice`.
7. `bob` has deposited into a pool that was supposed to block him — allowlist fully bypassed.
