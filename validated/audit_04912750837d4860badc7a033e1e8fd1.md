### Title
`feeX18` Excluded from EIP-712 Digest Allows Sequencer to Charge Arbitrary Fee on `WithdrawCollateralV2` - (File: `core/contracts/Verifier.sol`)

---

### Summary

The `feeX18` field in `SignedWithdrawCollateralV2` is not included in the EIP-712 type hash used to compute the user's signature digest. The user's signature covers only the `WithdrawCollateralV2` struct fields, but the sequencer-supplied `feeX18` — which directly debits the user's collateral balance — is never committed to by the signer. A malicious sequencer can set `feeX18` to any value from `0` to `currentFeeX18` without invalidating the user's signature.

---

### Finding Description

When a user signs a `WithdrawCollateralV2` transaction, the EIP-712 type string used to compute the digest is:

```
"WithdrawCollateralV2(bytes32 sender,uint32 productId,uint128 amount,uint64 nonce,address sendTo,uint128 appendix)"
``` [1](#0-0) 

The digest computation in `computeDigest()` encodes exactly these six fields: [2](#0-1) 

However, the on-chain struct that is ABI-decoded from the transaction bytes is `SignedWithdrawCollateralV2`, which carries an additional `feeX18` field:

```solidity
struct SignedWithdrawCollateralV2 {
    WithdrawCollateralV2 tx;
    CompactSignature signature;
    int128 feeX18;   // ← NOT in the EIP-712 type hash
}
``` [3](#0-2) 

In `processTransactionImpl`, after signature validation passes, `feeX18` is read directly from the decoded struct and used to debit the user's spot balance: [4](#0-3) 

The only constraint on `feeX18` is:

```solidity
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
``` [5](#0-4) 

Because `feeX18` is absent from the signed digest, the sequencer can freely set it to any value in `[0, currentFeeX18]` without the user's knowledge or consent, and the signature check will still pass.

---

### Impact Explanation

`chargeFee` calls `spotEngine.updateBalance(productId, sender, -fee)`, directly reducing the user's collateral balance by `feeX18` and crediting the sequencer fee pool by the same amount. [6](#0-5) 

A malicious sequencer can set `feeX18 = currentFeeX18` on every `WithdrawCollateralV2` transaction, extracting the maximum possible fee from every user withdrawal without the user having agreed to that fee amount. This is a direct, measurable asset loss: the user's signed intent covers the withdrawal amount and destination, but not the fee deducted from their balance.

---

### Likelihood Explanation

The attack requires the sequencer (or a quorum majority of Schnorr signers) to act maliciously. The sequencer is a semi-trusted entity; it is not a permissionless unprivileged caller. However, the design flaw is structural: the user's signature provides no protection against fee manipulation regardless of sequencer behavior. Any sequencer operator — even one acting within the protocol's intended trust model — can silently maximize fees on all `WithdrawCollateralV2` transactions. No key compromise or external exploit is needed; the sequencer simply sets the field to the maximum allowed value when constructing the batch.

---

### Recommendation

Include `feeX18` in the EIP-712 type string and struct hash for `WithdrawCollateralV2`:

```
"WithdrawCollateralV2(bytes32 sender,uint32 productId,uint128 amount,uint64 nonce,address sendTo,uint128 appendix,int128 feeX18)"
```

This ensures the user's signature commits to the exact fee they will be charged, preventing the sequencer from substituting a higher value unilaterally. Alternatively, remove `feeX18` as a parameter and derive it deterministically from `spotEngine.getConfig(productId).withdrawFeeX18`, as is done in the `WithdrawCollateral` (V1) path. [7](#0-6) 

---

### Proof of Concept

1. User signs a `WithdrawCollateralV2` transaction: `sender=Alice, productId=1, amount=1000e18, nonce=5, sendTo=address(0), appendix=1`.
2. The EIP-712 digest is computed over these six fields only; `feeX18` is absent.
3. The sequencer constructs the batch transaction bytes with `feeX18 = currentFeeX18` (e.g., `50e14` = 0.5%).
4. `validateSignedTx` recovers Alice's address from the signature — validation passes because `feeX18` was never part of the signed data.
5. `chargeFee(Alice, 50e14, 1)` executes: `spotEngine.updateBalance(1, Alice, -50e14)` debits Alice's balance; `sequencerFee[1] += 50e14` credits the sequencer.
6. Alice receives her withdrawal but loses an additional `50e14` units of product-1 collateral she never agreed to pay. [2](#0-1) [8](#0-7)
