### Title
`proof_facts` Preserved but `proof` Silently Dropped in Consensus P2P Serialization, Causing Valid Proposal Rejection — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

When a `ConsensusTransaction` containing a client-side proving `InvokeV3` transaction (with non-empty `proof_facts` and `proof`) is serialized to protobuf for P2P consensus propagation, the `proof` blob is silently dropped because `protobuf::InvokeV3` has no `proof` field. Upon deserialization, `proof_facts` is non-empty but `proof` is `Proof::default()` (empty). The `TransactionConverter` then unconditionally spawns a proof verification task using the empty proof. If the receiving validator's proof manager does not already hold the proof (e.g., the transaction arrived via consensus P2P before or instead of mempool P2P), `verify_proof` is called with an empty proof blob against non-empty `proof_facts`, the task fails, and the valid consensus proposal is rejected.

### Finding Description

**Serialization boundary — `proof` dropped, `proof_facts` kept:**

`protobuf::InvokeV3` (used for consensus P2P) carries `proof_facts` but has no `proof` field:

```protobuf
pub struct InvokeV3 {
    ...
    pub proof_facts: ::prost::alloc::vec::Vec<Felt252>,
    // no `proof` field
}
``` [1](#0-0) 

The serialization path for `ConsensusTransaction`:

```rust
ConsensusTransaction::RpcTransaction(RpcTransaction::Invoke(
    RpcInvokeTransaction::V3(txn),
)) => protobuf::ConsensusTransaction {
    txn: Some(protobuf::consensus_transaction::Txn::InvokeV3(txn.into())),
    ...
}
``` [2](#0-1) 

`txn.into()` calls `From<RpcInvokeTransactionV3> for InvokeTransactionV3` (which drops `proof`) then `From<InvokeTransactionV3> for protobuf::InvokeV3`. The `proof` field is gone. [3](#0-2) 

By contrast, the **mempool** P2P path uses `protobuf::InvokeV3WithProof` which does carry the proof blob: [4](#0-3) 

**Deserialization restores `proof_facts` but leaves `proof` empty:**

```rust

**Converter unconditionally spawns verification when `proof_facts` is non-empty:**

```rust

    let proof_data = if tx.proof_facts.is_empty() {
        None
    } else {

    };
    ...

    self.spawn_verify_and_store_proof(proof_facts, proof)  // verifies empty proof
});
``` [6](#0-5) 

**Verification fails when proof is not already cached:**

```rust
async fn run_proof_verification(...) {
    let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;
    if contains_proof {
        return Ok(false);  // safe path — only if proof was previously stored
    }
    // falls through to verify empty proof against non-empty proof_facts → error
    starknet_proof_verifier::verify_proof(proof_facts, proof)
}
``` [7](#0-6) 

### Impact Explanation

A validator that receives a consensus proposal containing a client-side proving `InvokeV3` transaction via P2P — but whose local proof manager does not yet hold the proof (e.g., the transaction was submitted to a different node's gateway, or the mempool P2P message has not yet arrived) — will attempt to verify an empty proof blob against non-empty `proof_facts`. The verification task fails, and the valid proposal is rejected. In a distributed validator topology where proof manager state is not shared, this can prevent consensus from reaching quorum on any block that includes a client-side proving transaction.

**Impact category:** High — RPC/consensus admission rejects valid transactions before sequencing; valid consensus proposals are rejected.

### Likelihood Explanation

Client-side proving (`proof_facts` + `proof`) is a supported and documented feature. In any multi-validator deployment where validators do not all share the same gateway (the normal production topology), a validator receiving a consensus proposal for the first time will not have the proof in its local proof manager. The race condition between mempool P2P delivery and consensus P2P delivery is inherent to the architecture. Any user who submits a client-side proving transaction can trigger this condition unprivileged.

### Recommendation

Choose one of:

1. **Include the proof in consensus P2P messages**: Use `protobuf::InvokeV3WithProof` (already defined and used for mempool) for the `ConsensusTransaction::InvokeV3` variant, so the proof is preserved across the P2P boundary.

2. **Fetch the proof from the proof manager before spawning verification**: In `convert_rpc_tx_to_internal`, if `proof_facts` is non-empty and `proof` is empty, attempt to fetch the proof from the proof manager before spawning the verification task. If the proof is not found, either reject the transaction or skip verification (trusting the proposer's prior validation).

3. **Skip proof verification in the consensus path**: Consensus-received transactions have already been validated by the proposer's gateway. The consensus converter could skip re-verification and only call `contains_proof`, treating a missing proof as a non-fatal condition.

### Proof of Concept

1. Node A and Node B are validators. Node A is the proposer.
2. User submits an `InvokeV3` transaction with non-empty `proof_facts` and `proof` to Node A's gateway.
3. Node A verifies the proof, stores it in its proof manager, and adds the transaction to its mempool.
4. Node A proposes a block containing this transaction. The proposal is serialized via `From<ConsensusTransaction> for protobuf::ConsensusTransaction`, which calls `txn.into()` → `protobuf::InvokeV3` — the `proof` field is dropped.
5. Node B receives the proposal. It has not yet received the mempool P2P message (or is a consensus-only node).
6. Node B calls `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal`.
7. `tx.proof_facts` is non-empty, `tx.proof` is `Proof::default()` (empty). `proof_data = Some((proof_facts, empty_proof))`.
8. `spawn_verify_and_store_proof(proof_facts, empty_proof)` is called.
9. `contains_proof` returns `false` (proof not in Node B's proof manager).
10. `starknet_proof_verifier::verify_proof(proof_facts, empty_proof)` is called and fails.
11. The `VerifyAndStoreProofTask` returns an error. Node B rejects the valid proposal. [8](#0-7) [9](#0-8) [10](#0-9)

    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,
        proof_manager_client: SharedProofManagerClient,
    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

        if contains_proof {
            return Ok(false);
        }

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)
        })
        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }
