### Title
Insufficient Bonus Pool Blocks All Exit Paths, Locking User Principal - (File: x/tieredrewards/keeper/msg_server.go)

### Summary
The `x/tieredrewards` module atomically calls `claimRewards` inside every exit operation (`TierUndelegate`, `ExitTierWithDelegation`). `claimRewards` calls `sufficientBonusPoolBalance`, which returns `ErrInsufficientBonusPool` if the `RewardsPoolName` module account cannot cover accrued bonus. Because the error propagates unconditionally, both exit paths fail entirely when the pool is underfunded, leaving users unable to recover their locked principal.

### Finding Description
`sufficientBonusPoolBalance` in `x/tieredrewards/keeper/bonus_rewards.go` compares the live pool balance against the computed bonus and returns a hard error when the pool is short: [1](#0-0) 

This check is invoked inside `processEventsAndClaimBonus`, which is called by `claimRewards`. `claimRewards` is called unconditionally at the top of both exit handlers before any delegation transfer occurs:

**`TierUndelegate`** — line 166: [2](#0-1) 

**`ExitTierWithDelegation`** — line 532: [3](#0-2) 

In both cases the error is returned immediately, aborting the transaction before `undelegate` or `transferDelegationFromPosition` is ever reached. The user's principal remains locked in the position's per-position delegator address.

`ClearPosition` (cancel exit) also calls `claimRewards` unconditionally: [4](#0-3) 

So a user who has triggered exit cannot cancel it either, leaving them stuck in an exiting state with no recourse.

The only exit path that skips `claimRewards` is `WithdrawFromTier`: [5](#0-4) 

But `WithdrawFromTier` requires the position to already be undelegated (i.e., `TierUndelegate` must have completed successfully first), so it is unreachable when the pool is empty.

The slash handler explicitly handles this case to prevent chain halt, but the user-facing message handlers do not: [6](#0-5) 

The `BeginBlocker` actively drains the same `RewardsPoolName` account every block to top up base rewards: [7](#0-6) 

This creates a realistic path to pool exhaustion through normal protocol operation.

### Impact Explanation
When the `RewardsPoolName` module account balance falls below the sum of accrued bonus rewards for any position, the position owner cannot:
- Undelegate their principal (`TierUndelegate` fails)
- Exit with delegation (`ExitTierWithDelegation` fails)
- Cancel a triggered exit (`ClearPosition` fails)

The user's locked principal — tokens transferred to the module at `LockTier` or `CommitDelegationToTier` time — is inaccessible for as long as the pool remains underfunded. The corrupted invariant is: **accrued bonus tracked in position state > actual `RewardsPoolName` balance**, which causes the exit operations that should return principal to fail entirely.

### Likelihood Explanation
The `RewardsPoolName` account is drained every block by the `BeginBlocker` base-rewards top-up mechanism and by every successful `ClaimTierRewards` call. The pool has no automatic refill mechanism; it depends on external funding. Under normal chain operation with many active positions, the pool will periodically run low or empty. No privileged access is required to trigger the locked state — any user with a position that has accrued bonus rewards will be affected the moment the pool is insufficient.

### Recommendation
Decouple principal recovery from bonus reward settlement in exit operations. Specifically:

1. In `TierUndelegate` and `ExitTierWithDelegation`, attempt to claim rewards but continue with the exit even if `ErrInsufficientBonusPool` is returned (log the unclaimed bonus and allow the user to claim it separately once the pool is refunded, or forfeit it).
2. Alternatively, record unclaimed bonus as a pending debt on the position and allow exit to proceed, settling the debt when the pool is next funded.
3. Ensure `ClearPosition` similarly does not block on pool insufficiency, since cancelling an exit should not require paying out rewards.

### Proof of Concept
1. User calls `MsgLockTier` with `amount = tier.MinLockAmount`, creating a delegated position.
2. Time passes; bonus rewards accrue against the position.
3. The `RewardsPoolName` account is drained to zero (via `BeginBlocker` top-ups or other users claiming rewards).
4. User calls `MsgTierUndelegate` on their position.
5. Inside `TierUndelegate`, `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`.
6. `TierUndelegate` returns the error; the transaction is rejected.
7. User calls `MsgExitTierWithDelegation`; same failure path at line 532.
8. User calls `MsgClearPosition` to cancel exit; same failure path at line 406.
9. User's principal remains locked in the position's delegator address with no available exit path until an external party refunds the pool.

The existing test `TestMsgClaimTierRewards_FailsWhenBonusPoolInsufficient` confirms the pool-check failure behavior: [8](#0-7)

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```

```go
// TestMsgClaimTierRewards_FailsWhenBonusPoolInsufficient verifies that ClaimTierRewards
// returns ErrInsufficientBonusPool when accrued bonus cannot be paid, so the tx rolls
// back and the user can retry later without losing base rewards to a partial claim.
func (s *KeeperSuite) TestMsgClaimTierRewards_FailsWhenBonusPoolInsufficient() {
	lockAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
	pos := s.setupNewTierPosition(lockAmount, false)
	delAddr := sdk.MustAccAddressFromBech32(pos.Owner)
	valAddr := sdk.MustValAddressFromBech32(pos.Delegation.ValidatorAddress)
	_, bondDenom := s.getStakingData()
	msgServer := keeper.NewMsgServerImpl(s.keeper)

	s.setValidatorCommission(valAddr, sdkmath.LegacyZeroDec())

	// Advance time and allocate base rewards, but intentionally leave bonus pool empty.
	s.ctx = s.ctx.WithBlockHeight(s.ctx.BlockHeight() + 1)
	s.ctx = s.ctx.WithBlockTime(s.ctx.BlockTime().Add(time.Hour * 24 * 365))
	s.allocateRewardsToValidator(valAddr, sdkmath.NewInt(100), bondDenom)

	// Bonus pool remains at 0 — bonus accrued but pool cannot cover it.
	balBefore := s.app.BankKeeper.GetBalance(s.ctx, delAddr, bondDenom)

	// Use a branched context so a failed message does not persist state (matches DeliverTx rollback).
	cacheCtx, _ := s.ctx.CacheContext()
	resp, err := msgServer.ClaimTierRewards(cacheCtx, &types.MsgClaimTierRewards{
		Owner:       delAddr.String(),
		PositionIds: []uint64{pos.Id},
	})
	s.Require().Error(err)
	s.Require().True(errors.Is(err, types.ErrInsufficientBonusPool))
	s.Require().Nil(resp)

	balAfter := s.app.BankKeeper.GetBalance(s.ctx, delAddr, bondDenom)
	s.Require().True(balAfter.Amount.Equal(balBefore.Amount), "failed claim must not transfer rewards")
```
