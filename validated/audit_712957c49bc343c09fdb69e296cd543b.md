### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, making the allowlist either permanently broken or fully bypassable — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. The pool always sets `sender = msg.sender` of the `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks the router's address — not the actual end user's address. This is the direct analog of the external bug: a context value that should be bounded/resolved to the correct actor is instead passed raw, causing the guard to operate on the wrong identity.

---

### Finding Description

**Call chain:**

```
User → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(recipient, ..., extensionData)   // msg.sender = router
              → _beforeSwap(msg.sender=router, ...)
                   → SwapAllowlistExtension.beforeSwap(sender=router, ...)
```

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // ← this is the router, not the end user
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards it verbatim to the extension:

```solidity
// ExtensionCalling.sol line 160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol line 37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (correct key), but `sender` is the router address — not the end user. The allowlist lookup is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Two concrete failure modes:**

**Mode A — Allowlisted users are silently blocked from the router.** A pool admin allowlists specific EOAs (e.g., `alice`, `bob`). Those users call `router.exactInputSingle()`. The extension sees `sender = router`, which is not in the allowlist → `NotAllowedToSwap`. The allowlisted users must call `pool.swap()` directly, which requires them to implement `metricOmmSwapCallback` — an interface not available to plain EOAs. The standard periphery path is permanently broken for every allowlisted user.

**Mode B — Allowlist fully bypassed.** To fix Mode A, the pool admin allowlists the router address (`allowedSwapper[pool][router] = true`). Now any user — including those the admin explicitly never allowlisted — can bypass the gate by routing through `MetricOmmSimpleRouter`. The curated pool's access control is completely defeated.

---

### Impact Explanation

**Mode A** breaks core pool functionality: allowlisted users on a curated pool cannot use the supported periphery swap path. Swaps are unusable through the router for the intended participants.

**Mode B** is a direct allowlist bypass: an unprivileged, non-allowlisted user can execute swaps on a pool that was configured to restrict access. If the pool holds concentrated liquidity at oracle prices, the attacker can drain value from LPs at will, constituting a direct loss of LP principal. This matches the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" and "Broken core pool functionality causing loss of funds" impact categories.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Any pool that deploys `SwapAllowlistExtension` to create a curated pool will encounter Mode A immediately — allowlisted users will be unable to use the router. The natural remediation (allowlisting the router) triggers Mode B. Both modes are reachable by any public user with no special privileges, no malicious setup, and no non-standard tokens.

---

### Recommendation

The extension must resolve the actual end user, not the direct caller of `pool.swap()`. The cleanest fix is to have the router forward the original `msg.sender` inside `extensionData` and have the extension decode it, or to add a `recipient`/`originator` field to the swap hook interface. A simpler short-term fix is to check `recipient` (the address receiving output tokens) when `sender` is a known router, but the robust solution is to pass the originating EOA explicitly through `extensionData` and verify it in the extension:

```solidity
// In SwapAllowlistExtension.beforeSwap:
address effectiveSender = extensionData.length == 20
    ? abi.decode(extensionData, (address))  // router-forwarded originator
    : sender;
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][effectiveSender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

The router would then encode `msg.sender` into `extensionData` before calling the pool.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — Alice is the only allowed swapper.
3. Alice calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ..., extensionData)` — `msg.sender` to pool = router.
5. Pool calls `_beforeSwap(sender=router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `false` → reverts `NotAllowedToSwap`.
7. Alice's swap fails despite being explicitly allowlisted.

**Bypass variant:**

8. Pool admin calls `setAllowedToSwap(pool, router, true)` to fix Alice's access.
9. Bob (never allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
10. Extension checks `allowedSwapper[pool][router]` → `true` → swap succeeds.
11. Bob bypasses the allowlist entirely. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
