Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the actual trader, allowing any user to bypass a curated pool's swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` to the pool. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. A pool admin who allowlists the router (to enable normal routing) inadvertently grants every user bypass access to the per-user allowlist, defeating the extension's entire purpose.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates the allowlist against `sender` (the pool's `msg.sender`): [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly: [4](#0-3) 

At that point `msg.sender` to the pool is the router contract address. The extension therefore evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][actual_user]`. A pool admin who allowlists the router as a trusted periphery (the natural configuration for a pool that supports routing) makes the per-user allowlist trivially bypassable: any address can route through the public router and the check passes because the router is allowlisted, regardless of whether the originating user is on the list.

There is no mechanism in the hook call path to recover the original `msg.sender`. The `extensionData` field is caller-supplied and unauthenticated, so it cannot serve as a trusted identity channel without additional protocol-level enforcement.

## Impact Explanation

This is a direct admin-boundary break. A pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict trading to a specific set of addresses (e.g., KYC'd counterparties). To support normal routing, the admin allowlists the router. Any address not on the per-user allowlist can then bypass the restriction by calling `MetricOmmSimpleRouter` instead of `pool.swap` directly. The access control the admin believed was enforced is silently nullified. The pool cannot simultaneously support routing and enforce a per-user swap allowlist.

## Likelihood Explanation

Exploitation requires no special privileges, no malicious setup, and no non-standard tokens. The router is the canonical, documented entry point for swaps. Any user who knows the pool uses a swap allowlist can trivially route through the public router. The precondition — router being allowlisted — is the expected configuration for any pool that intends to support routing, making this a near-certain misconfiguration in practice.

## Recommendation

The `SwapAllowlistExtension` should not key its check on `sender` (which is `msg.sender` to the pool). Options:

1. **Router-level attestation**: Have the router pass the originating user in `extensionData` in an authenticated way (e.g., signed or verified by the pool), and have the extension verify it.
2. **Dedicated entry point**: Add a `swapOnBehalf(address originator, ...)` entry point to the pool that records the true originator in transient storage before invoking the hook, callable only by allowlisted routers.
3. **Recipient-based gating**: If the economic actor is always the recipient, gate on `recipient` instead of `sender` — though this only works if recipient equals the true trader.
4. **Router prohibition**: Document that pools using `SwapAllowlistExtension` must not allowlist the router, and enforce this at the factory or extension configuration level.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` in `BEFORE_SWAP_ORDER`. Only address `A` is added to `allowedSwapper[pool]`. The router is allowlisted via `setAllowedToSwap(pool, router, true)` to enable routing.
2. Address `B` (not allowlisted) calls `pool.swap(...)` directly → `beforeSwap` receives `sender = B` → `allowedSwapper[pool][B]` is `false` → revert `NotAllowedToSwap`. Guard works.
3. Address `B` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})` → router calls `pool.swap(...)` → pool's `msg.sender` is the router → `_beforeSwap(router, ...)` → `SwapAllowlistExtension.beforeSwap` receives `sender = router` → `allowedSwapper[pool][router]` is `true` → check passes → swap executes → allowlist bypassed.
