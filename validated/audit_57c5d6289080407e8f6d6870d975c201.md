### Title
Migration upgrade halt via `ErrInsufficientBonusPool` in `ForceFullExitWithDelegation` — (`x/tieredrewards/keeper/force_exit.go`, `x/tieredrewards/migrations/v2/migrate.go`)

### Summary

The v8 upgrade migration calls `ForceFullExitWithDelegation` for every vesting-owned tier position. That function unconditionally calls `claimRewards`, which calls `processEventsAndClaimBonus`, which calls `sufficientBonusPoolBalance`. If the rewards pool balance is less than the accrued bonus for any vesting-owned position, `ErrInsufficientBonusPool` is returned and propagates unhandled all the way up through the upgrade handler, halting the chain at the upgrade height.

### Finding Description

**Exact call chain:**

`RunMigrations` → `Migrate1to2` → `v2.Migrate` → `exitVestedAccountsPositions` → `ForceFullExitWithDelegation` → `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → returns `ErrInsufficientBonusPool` → propagates up unhandled → upgrade handler returns error → chain halts.

**Step 1 — Upgrade handler (`app/upgrades.go`):**

The upgrade handler calls `RunMigrations` and propagates any error directly: [1](#0-0) 

**Step 2 — Migration (`x/tieredrewards/migrations/v2/migrate.go`):**

`exitVestedAccountsPositions` calls `ForceFullExitWithDelegation` for each vesting-owned position and propagates any error: [2](#0-1) 

**Step 3 — `ForceFullExitWithDelegation` (`x/tieredrewards/keeper/force_exit.go`):**

`claimRewards` is called unconditionally and any error is wrapped and returned — no special handling for `ErrInsufficientBonusPool`: [3](#0-2) 

**Step 4 — `claimRewards` → `processEventsAndClaimBonus` (`x/tieredrewards/keeper/claim_rewards.go`):**

`processEventsAndClaimBonus` is called and its error propagated: [4](#0-3) 

**Step 5 — `sufficientBonusPoolBalance` (`x/tieredrewards/keeper/bonus_rewards.go`):**

If `poolBalance < bonus`, returns `ErrInsufficientBonusPool`: [5](#0-4) 

**Contrast with the slash hook**, which explicitly swallows `ErrInsufficientBonusPool` to prevent chain halt: [6](#0-5) 

The migration path has no equivalent guard. The ADR-006 design doc only documents the "fail atomically, user retries" behavior for user-driven paths: [7](#0-6) 

The migration path is not mentioned, and no equivalent "forfeit bonus silently" guard exists in `ForceFullExitWithDelegation`.

**The integration test confirms the dependency:** the test setup explicitly funds the rewards pool before the upgrade to avoid this exact failure: [8](#0-7) 

This is a workaround in the test, not a fix in the production migration code.

### Impact Explanation

If the rewards pool is insufficient at upgrade height, the upgrade handler returns an error. In Cosmos SDK, a failing upgrade handler causes the node to panic and halt. All nodes halt at the same height. The chain is stuck in a partially-migrated state: some vesting positions may have been force-exited (those processed before the failing one), while others remain. The chain cannot proceed until a coordinated emergency patch is deployed.

### Likelihood Explanation

The preconditions are realistic:
1. Vesting accounts with tier positions exist — the migration is specifically designed to handle them, and the integration test creates them.
2. The rewards pool can be drained before the upgrade by: (a) the `BeginBlocker` top-up mechanism continuously drawing from the pool for base rewards, (b) normal user bonus claims in the blocks leading up to the upgrade height, or (c) simply never being funded sufficiently relative to accrued bonus.

No attacker action is required — this is a passive state that can arise from normal chain operation.

### Recommendation

Apply the same "forfeit bonus silently on `ErrInsufficientBonusPool`" pattern used in the slash hook to `ForceFullExitWithDelegation`. Specifically, in `force_exit.go`, after calling `claimRewards`, check if the error is `ErrInsufficientBonusPool` and log it without returning, allowing the force-exit to proceed without paying the bonus. This matches the documented chain-halt-avoidance design intent already present in the slash hook.

### Proof of Concept

1. Create a vesting account (e.g. `PermanentLockedAccount`) with a tier position that has accrued bonus rewards (advance block time by 30+ days after position creation).
2. Drain the rewards pool to zero (or leave it unfunded).
3. Call `migrator.Migrate1to2(ctx)`.
4. Observe that it returns an error wrapping `ErrInsufficientBonusPool`, and the migration aborts with some positions force-exited and others not.

The existing keeper test `TestProcessEvents_InsufficientPool_Error` already proves that `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool` when the pool is empty: [9](#0-8) 

Combining this with the migration test setup in `TestMigrate1to2_ExitsVestedOwnerPositions` (which funds the pool via `advanceForRewards`) confirms the failure mode is reachable when the pool is not funded. [10](#0-9)

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```

```go
// TestProcessEvents_InsufficientPool_Error verifies that claiming without
// a funded pool returns ErrInsufficientBonusPool.
func (s *KeeperSuite) TestProcessEvents_InsufficientPool_Error() {
	lockAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
	pos := s.setupNewTierPosition(lockAmount, false)

	// Advance time so bonus would be non-zero. Do NOT fund the pool.
	s.ctx = s.ctx.WithBlockTime(s.ctx.BlockTime().Add(30 * 24 * time.Hour))

	_, err := s.keeper.ProcessEventsAndClaimBonus(s.ctx, &pos)
	s.Require().Error(err, "should fail when bonus pool is insufficient")
	s.Require().ErrorContains(err, "insufficient bonus pool",
		"error should mention insufficient bonus pool")
}
```

```go
func (s *KeeperSuite) TestMigrate1to2_ExitsVestedOwnerPositions() {
	s.setupTier(1)
	vals, bondDenom := s.getStakingData()
	val := vals[0]
	valAddr := sdk.MustValAddressFromBech32(val.GetOperator())
	s.setValidatorCommission(valAddr, sdkmath.LegacyZeroDec())

	amount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())

	// Regular (non-vesting) owner with a tier position; must survive.
	regularOwner := sdk.AccAddress(secp256k1.GenPrivKey().PubKey().Address())
	regularAcc := s.app.AccountKeeper.NewAccountWithAddress(s.ctx, regularOwner)
	s.app.AccountKeeper.SetAccount(s.ctx, regularAcc)
	s.Require().NoError(banktestutil.FundAccount(
		s.ctx, s.app.BankKeeper, regularOwner, sdk.NewCoins(sdk.NewCoin(bondDenom, amount)),
	))
	regularPos := s.createLockTierPositionV1(regularOwner, valAddr, amount)

	// Vesting owner with two tier positions.
	// Both must be deleted by the migration.
	vestingOwner := s.newVestingOwnerWithBalance(bondDenom, amount, amount.MulRaw(3))
	commitPos := s.createCommitPositionV1(vestingOwner, val, valAddr, amount)
	lockPos := s.createLockTierPositionV1(vestingOwner, valAddr, amount)

	s.advanceForRewards(valAddr, bondDenom)

	migrator := keeper.NewMigrator(s.keeper)
	s.Require().NoError(migrator.Migrate1to2(s.ctx))

	// Both vesting-owned positions deleted.
	_, err := s.keeper.Positions.Get(s.ctx, commitPos.Id)
	s.Require().Error(err, "commit-origin vesting position must be deleted")
	_, err = s.keeper.Positions.Get(s.ctx, lockPos.Id)
	s.Require().Error(err, "lock-origin vesting position must be deleted")

	// Regular position survives, with DelegatorAddress equal to the legacy
	// derivation.
	survived, err := s.keeper.Positions.Get(s.ctx, regularPos.Id)
	s.Require().NoError(err, "regular position must survive")
	s.Require().Equal(regularOwner.String(), survived.Owner)
	s.Require().Equal(migration.LegacyDelegatorAddress(regularPos.Id), survived.DelegatorAddress)
}
```
