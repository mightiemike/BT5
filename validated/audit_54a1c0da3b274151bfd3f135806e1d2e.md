### Title
Slow-Mode Deposit Funds Permanently Lost When `processSlowModeTransaction` Reverts — (`core/contracts/Endpoint.sol`)

---

### Summary

When a user deposits collateral via `depositCollateralWithReferral()`, real ERC-20 tokens are transferred to the `Endpoint` contract **before** the slow-mode transaction is queued. If that slow-mode transaction later reverts during execution, the `catch` block in `_executeSlowModeTransaction()` silently discards the failure without refunding the depositor. A code comment explicitly acknowledges the missing refund: `// try return funds now removed`.

---

### Finding Description

`Endpoint.depositCollateralWithReferral()` follows a two-step pattern:

1. **Step 1 — funds taken immediately**: `handleDepositTransfer()` pulls `amount` tokens from `msg.sender` into the contract.
2. **Step 2 — credit deferred**: A `SlowModeTx` of type `DepositCollateral` is enqueued; the user's on-chain balance is only credited when the sequencer (or the user after the timeout) calls `_executeSlowModeTransaction()`. [1](#0-0) 

When the slow-mode transaction is executed, the call to `processSlowModeTransaction()` is wrapped in a `try/catch`: [2](#0-1) 

If the inner call reverts for any reason, the `catch` block performs **no action**. The comment on line 226 — `// try return funds now removed` — is an explicit admission that a refund path existed and was deleted. The user's tokens remain locked in the contract with no recovery mechanism available to them.

The slow-mode processing path for `DepositCollateral` in `EndpointTx.processSlowModeTransactionImpl()` calls `clearinghouse.depositCollateral(txn)`, which can revert under conditions that arise after the deposit was made (e.g., the product being delisted via a `DelistProduct` slow-mode transaction that is processed before the deposit). [3](#0-2) 

---

### Impact Explanation

Any user whose slow-mode `DepositCollateral` transaction fails after funds have already been transferred suffers a **permanent, unrecoverable loss** of the deposited tokens. There is no user-callable recovery function; `removeLiquidity` in `BaseWithdrawPool` is `onlyOwner`. [4](#0-3) 

---

### Likelihood Explanation

The gap between deposit and slow-mode execution is at least `SLOW_MODE_TX_DELAY` (three days, per the inline comment). During that window:

- An admin can submit a `DelistProduct` slow-mode transaction that is processed before the deposit, causing `clearinghouse.depositCollateral()` to revert on an invalid product.
- Any future revert condition added to `clearinghouse.depositCollateral()` (e.g., utilization caps, sanctions checks) would silently swallow the failure and strand funds.
- The `DirectDepositV1.creditDeposit()` path calls `depositCollateralWithReferral` for every registered product, amplifying exposure across multiple assets simultaneously. [5](#0-4) 

---

### Recommendation

Restore the refund logic inside the `catch` block of `_executeSlowModeTransaction()`. When a slow-mode `DepositCollateral` transaction fails, the contract must return the deposited tokens to the original depositor (recoverable from the `SlowModeTx.sender` field and the encoded `DepositCollateral.amount`). This mirrors the fix recommended in the reference report: add the return-funds path to the cancellation/failure flow, analogous to how `AuctionHouse.createBid()` refunds the previous bidder.

---

### Proof of Concept

1. User calls `Endpoint.depositCollateralWithReferral(subaccount, productId, 1000e6, "")`.
2. `handleDepositTransfer()` transfers 1000 USDC from the user to the `Endpoint` contract.
3. A `SlowModeTx` is queued with `executableAt = block.timestamp + 3 days`.
4. Within the 3-day window, an admin submits a `DelistProduct` slow-mode transaction for `productId`; the sequencer processes it immediately.
5. After 3 days, the user calls `executeSlowModeTransaction()`.
6. `processSlowModeTransaction()` calls `clearinghouse.depositCollateral(txn)`, which reverts because the product no longer exists.
7. The `catch` block in `_executeSlowModeTransaction()` executes with no body (`// try return funds now removed`).
8. The user's 1000 USDC is permanently locked in the `Endpoint` contract. [6](#0-5)
