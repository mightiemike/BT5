Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Any User to Bypass Per-User Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is set to the pool's `msg.sender` — the router contract address when a user routes through `MetricOmmSimpleRouter`. If the pool admin allowlists the router (the only way to support router-mediated swaps for any user), every unprivileged user can bypass the per-user allowlist by calling any router entry point. There is no configuration of `SwapAllowlistExtension` that simultaneously allows allowlisted users to swap via the router and blocks non-allowlisted users from doing the same.

## Finding Description
**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**
`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`. When the call originates from `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract address, not the end user.

**Step 2 — Router does not forward the original caller.**
`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)` directly. The original `msg.sender` is stored in transient storage for the payment callback (`_setNextCallbackContext`) but is never encoded into `extensionData` passed to the pool. The extension therefore never sees the end user's address.

**Step 3 — Extension checks the router address.**
`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router. If `allowedSwapper[pool][router] == true`, the check passes unconditionally for any end user routing through the router.

**The inescapable dilemma:**
- Admin allowlists only EOAs, not the router → those EOAs cannot swap via the router (router address fails the check).
- Admin allowlists the router → every user, including those not individually allowlisted, can swap by routing through the router.

No existing guard in `SwapAllowlistExtension` or the router resolves this conflict.

## Impact Explanation
This is an admin-boundary break. The `SwapAllowlistExtension` is a production access-control extension whose stated purpose is to gate `swap` by swapper address per pool. When the router is allowlisted — the only operational configuration that supports router-mediated swaps — the allowlist is rendered completely ineffective. Any unprivileged user can execute a swap against a pool intended to be restricted by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point). The pool admin's configured restriction is bypassed by an unprivileged path through a public periphery contract.

## Likelihood Explanation
Pool admins who deploy `SwapAllowlistExtension` and also want to support the official router will naturally allowlist the router address. This is the expected operational pattern for any pool that restricts direct swaps but still wants to be accessible via the standard periphery. The bypass is reachable in any realistic deployment that uses both the allowlist extension and the router, requires no special privileges, and is repeatable by any address.

## Recommendation
The extension must gate on the ultimate end user, not the immediate pool caller. The preferred fix is to have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` (e.g., as the first 20 bytes or via `abi.encode`) and have `SwapAllowlistExtension.beforeSwap` decode and check that address when the caller is a trusted router. Until this is implemented, `SwapAllowlistExtension` must be documented as incompatible with router-mediated swaps and must only be used with direct pool calls.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension in the beforeSwap extension slot
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (to allow router-mediated swaps for allowlisted users)
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  - attacker (not individually allowlisted) calls:
      MetricOmmSimpleRouter.exactInputSingle({
        pool: restrictedPool,
        recipient: attacker,
        zeroForOne: true,
        amountIn: X,
        ...
      })
  - Router calls pool.swap(...) → pool sets sender = router address (L231)
  - Pool calls SwapAllowlistExtension.beforeSwap(sender=router, ...)
  - Extension checks allowedSwapper[pool][router] → true → passes (L37)
  - Swap executes for attacker despite attacker not being allowlisted

Result:
  - attacker successfully swaps against a pool intended to be restricted
  - SwapAllowlistExtension's per-user gate is completely bypassed
```
