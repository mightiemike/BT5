### Title
u64 Overflow in `op_unknown` Cost Multiplication Enables Free Execution of Unknown Ops — (`File: src/more_ops.rs`)

### Summary

In `src/more_ops.rs`, the `op_unknown` function computes the cost of unknown CLVM opcodes by multiplying a base cost by `(cost_multiplier + 1)`. Both values are `u64`, and in Rust release builds integer overflow wraps silently. An attacker who controls the CLVM bytes can choose a `cost_multiplier` and supply enough arguments to make the pre-multiplication cost land on a value whose product wraps to ≤ `u32::MAX`, causing the operation to return with a cost of 0 (or any small value). This is an undercharged-execution bug: the block validator accepts a block whose true cost exceeds the limit.

---

### Finding Description

`op_unknown` is the handler for all unknown opcodes in lenient/consensus mode. Its cost model is: [1](#0-0) 

The last byte of the opcode encodes a 2-bit `cost_function` (0–3); the preceding bytes encode a `cost_multiplier` (up to `u32::MAX`): [2](#0-1) 

After computing a base cost from the arguments (bounded by `check_cost` against `max_cost`), the code multiplies: [3](#0-2) 

`cost` and `cost_multiplier` are both `u64`. In Rust release mode, `cost *= cost_multiplier + 1` wraps silently on overflow. The only post-multiplication guard is `cost > u32::MAX as u64`; if the wrapped result is ≤ `u32::MAX`, the function returns `Ok(Reduction(cost, nil))` with the wrapped (tiny) cost.

**Concrete trigger:**

Choose opcode bytes `[0x7F, 0xFF, 0xFF, 0xFF, 0xC0]`:
- `op[0] = 0x7F`, `op[1] = 0xFF` → not reserved (reserved requires both `0xFF, 0xFF`)
- `cost_function = (0xC0 >> 6) = 3` (CONCAT-like)
- `cost_multiplier = u32_from_u8(&[0x7F, 0xFF, 0xFF, 0xFF]) = 0x7FFFFFFF = 2147483647`
- `cost_multiplier + 1 = 2^31 = 2147483648` [4](#0-3) 

Now supply arguments so that the pre-multiplication `cost` equals exactly `2^33 = 8,589,934,592`:

```
cost_function = 3 (CONCAT-like):
  CONCAT_BASE_COST    = 142
  CONCAT_COST_PER_ARG = 135
  CONCAT_COST_PER_BYTE = 3
``` [5](#0-4) 

Using a mix of zero-byte and one-byte atoms, the attacker can solve the linear Diophantine equation `142 + 135·n + 138·m = 2^33` for non-negative integers `n, m`. The `check_cost` inside the loop only fires if cost exceeds `max_cost` (e.g., 11 billion); `2^33 ≈ 8.6 billion < 11 billion`, so the loop completes without error.

Then:

```
cost *= cost_multiplier + 1
     = 2^33 * 2^31
     = 2^64
     ≡ 0  (mod 2^64)   ← wraps to 0 in release mode
```

`0 ≤ u32::MAX` → the guard passes → `op_unknown` returns `Reduction(0, nil)`.

The `Cost` type is `u64`: [6](#0-5) 

and `check_cost` only compares against `max_cost`: [7](#0-6) 

There is no saturating or checked arithmetic at the multiplication site.

---

### Impact Explanation

An attacker can include a CLVM program in a Chia coin spend that invokes an unknown opcode with a crafted multiplier and enough arguments to trigger the overflow. The program's reported cost is 0 (or any small value), while its true cost is ~8.6 billion × 2^31 ≈ 18.4 × 10^18. The block validator sums reported costs; the block passes the cost limit check even though the true computational work far exceeds it. This is a **consensus-critical undercharged-execution** bug: nodes that enforce the cost limit correctly would reject the block, while nodes running this code accept it, causing a chain split.

---

### Likelihood Explanation

The attack requires:
1. Knowing the opcode encoding scheme (documented in the source comments).
2. Solving a simple linear Diophantine equation to hit the exact pre-multiplication cost.
3. Submitting a large but valid CLVM program (~63 million nil arguments).

All three are within reach of any attacker who reads the source. The opcode is not reserved (first two bytes are not both `0xFF`), so it is accepted in consensus mode. The Chia block cost limit (11 billion) is well above the required pre-multiplication cost (8.6 billion), making the window reachable.

---

### Recommendation

Replace the bare multiplication with a checked or saturating variant and treat overflow as an error:

```rust
// Before (line 261):
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}

// After:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}
```

This ensures that any product exceeding `u64::MAX` is treated as an invalid op rather than silently wrapping to a small cost.

### Proof of Concept

**Opcode bytes:** `[0x7F, 0xFF, 0xFF, 0xFF, 0xC0]`
- Not reserved: `op[0]=0x7F ≠ 0xFF`
- `cost_multiplier = 0x7FFFFFFF = 2147483647`; `cost_multiplier + 1 = 2^31`
- `cost_function = 3`

**Arguments:** A list of atoms whose total cost (via the CONCAT-like formula) equals `2^33 = 8,589,934,592`. For example, solve `142 + 135n + 138m = 8,589,934,592` → `n = 63,629,885`, `m = 1` (one 1-byte atom, the rest nil).

**Execution trace (release build):**
```
pre-multiplication cost  = 8,589,934,592  (= 2^33)
cost_multiplier + 1      = 2,147,483,648  (= 2^31)
product                  = 2^64           → wraps to 0
guard: 0 > u32::MAX?     → false
return Reduction(0, nil) ← cost reported as 0
```

**Effect:** The block's total reported cost does not include the true cost of this operation. A block containing many such invocations passes the cost limit while performing unbounded computation, causing consensus divergence between patched and unpatched nodes. [3](#0-2)
