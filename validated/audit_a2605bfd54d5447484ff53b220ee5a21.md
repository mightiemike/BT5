Audit Report

## Title
`SwapAllowlistExtension` gates the router address instead of the actual end-user, allowing any user to bypass per-user swap restrictions via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives the `sender` argument from the pool, which is set to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. If the router is allowlisted for a pool, every user on the network can bypass per-user swap restrictions by routing through the router, completely defeating the access-control invariant the pool admin configured.

## Finding Description
**Step 1 — Pool passes `msg.sender` as `sender` to the hook.**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` at lines 230–240. `ExtensionCalling._beforeSwap` encodes that value as the `sender` field forwarded to every configured extension via `_callExtensionsInOrder`.

**Step 2 — Router is the `msg.sender` the pool sees.**

`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` directly at lines 72–80. Inside the pool, `msg.sender` is therefore the router address (`MetricOmmSimpleRouter`), not the originating EOA.

**Step 3 — The allowlist checks the router, not the user.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the caller of the extension) and `sender` is the router address. The allowlist mapping is keyed `allowedSwapper[pool][swapper]`. If the router is in the allowlist for that pool, the check passes unconditionally for any user who calls `exactInputSingle`.

**Why existing guards are insufficient:**

The `onlyPool` modifier on the base class enforces that only a registered pool can call the extension, but this does not help — the pool itself is the caller, and it faithfully passes the router address as `sender`. There is no mechanism in `SwapAllowlistExtension` to unwrap the originating user from behind the router.

## Impact Explanation
Any unprivileged, non-allowlisted user can swap in a pool whose admin configured `SwapAllowlistExtension` to restrict access, provided the router is allowlisted. This is a direct admin-boundary break: a pool admin-configured guard is bypassed by an unprivileged path (`MetricOmmSimpleRouter`). Unauthorized swaps in a restricted pool cause LP-side harm through unintended price impact and liquidity consumption, and violate the access-control invariant the pool admin explicitly set. The symmetric failure mode also breaks core swap functionality: allowlisted users cannot use the router at all if the router itself is not allowlisted.

## Likelihood Explanation
Allowlisting the router is the natural and expected configuration for any restricted pool that still wants to support the standard periphery swap path. A pool admin who adds the router to the allowlist (expecting it to act as a trusted intermediary) does not realize this grants every user on the network unrestricted access. The trigger requires no privileged action from the attacker — a standard `exactInputSingle` call suffices, with no special parameters or setup beyond the router being allowlisted.

## Recommendation
The `beforeSwap` hook must gate the actual end user, not the immediate caller of `pool.swap()`. The cleanest fix is:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. The extension should additionally verify that `sender` (the pool-level caller) is a known trusted router, so that only verified routers can supply the originating-user claim.

2. **Maintain a trusted-router registry** in `SwapAllowlistExtension` so that when `sender` is a trusted router, the extension decodes and checks the real user from `extensionData`; when `sender` is not a router, it checks `sender` directly.

## Proof of Concept
```
1. Pool admin deploys a pool with SwapAllowlistExtension configured.
2. Admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
3. Admin calls setAllowedToSwap(pool, alice, true)    // allowlist alice
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ...})
5. Inside pool.swap(), msg.sender = router.
6. _beforeSwap(sender=router, ...) is called.
7. SwapAllowlistExtension checks allowedSwapper[pool][router] == true → passes.
8. Bob's swap executes successfully despite not being allowlisted.
```

Foundry test plan: deploy `SwapAllowlistExtension`, configure a pool with it, allowlist only the router and alice, then call `exactInputSingle` from an address that is not allowlisted and assert the swap succeeds (no `NotAllowedToSwap` revert).
