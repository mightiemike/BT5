### Title
Unbounded `NlpLockedBalanceQueue` Loop Enables Permanent NLP Burn DoS — (`File: core/contracts/SpotEngineState.sol`)

---

### Summary

`SpotEngine.handleNlpLockedBalance` appends a new `NlpLockedBalance` entry to a per-subaccount queue on every NLP mint that occurs at a distinct oracle timestamp. The `tryUnlockNlpBalance` function, called unconditionally on every subsequent mint or burn, loops over **all** accumulated expired entries in a single transaction. A user who mints NLP many times across different oracle ticks accumulates an unbounded queue; once the lock period elapses, the unlock loop exhausts the block gas limit and permanently prevents the user from burning NLP to recover collateral.

---

### Finding Description

`SpotEngine.handleNlpLockedBalance` is invoked on every `updateBalance` call for `NLP_PRODUCT_ID`:

```solidity
// SpotEngine.sol – handleNlpLockedBalance (positive amountDelta branch)
queue.balances[queue.balanceCount] = NlpLockedBalance({
    balance: Balance({amount: amountDelta}),
    unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
});
queue.balanceCount++;
``` [1](#0-0) 

A new entry is only merged into the previous one when `unlockedAt` matches exactly:

```solidity
if (
    queue.balanceCount > 0 &&
    queue.balances[queue.balanceCount - 1].unlockedAt ==
    getOracleTime() + NLP_LOCK_PERIOD
) { ... } else { queue.balanceCount++; }
``` [2](#0-1) 

Because `getOracleTime()` advances with every sequencer tick, each mint at a different tick creates a distinct entry. `tryUnlockNlpBalance` then iterates over every expired entry in one call:

```solidity
while (
    queue.unlockedUpTo < queue.balanceCount &&
    queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
) {
    queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
    delete queue.balances[queue.unlockedUpTo];
    queue.unlockedUpTo++;
}
``` [3](#0-2)

### Impact Explanation

Once the accumulated expired entries exceed the gas budget for a single transaction, every call to `updateBalance(NLP_PRODUCT_ID, subaccount, ...)` reverts. This blocks:

- **`BurnNlp`** — the only path for a user to redeem NLP tokens back to collateral. The user's NLP balance becomes permanently unburnable, locking the underlying collateral value.
- **`MintNlp`** — further minting for the same subaccount also fails.
- **`getNlpUnlockedBalance`** — view function also reverts, breaking any off-chain tooling that relies on it.

The corrupted state is `nlpLockedBalanceQueues[subaccount].balanceCount` growing without a corresponding bound on the single-transaction processing cost. [6](#0-5) 

---

### Likelihood Explanation

The trigger is a user minting NLP across many distinct oracle ticks. On Arbitrum, oracle ticks are submitted frequently by the sequencer. A user who mints NLP once per tick over an extended period (e.g., automated strategies, repeated small mints) naturally accumulates entries. No adversarial intent is required — normal repeated usage is sufficient. The lock period means all entries created before it expire simultaneously, maximising the loop length in a single call.

---

### Recommendation

Apply the same counter-based approach already used for `unlockedUpTo`, but also cap the number of entries processed per call:

```solidity
function tryUnlockNlpBalance(bytes32 subaccount, uint256 maxIterations)
    public returns (Balance memory)
{
    NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[subaccount];
    uint256 processed = 0;
    while (
        queue.unlockedUpTo < queue.balanceCount &&
        queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime() &&
        processed < maxIterations
    ) {
        queue.unlockedBalanceSum.amount +=
            queue.balances[queue.unlockedUpTo].balance.amount;
        delete queue.balances[queue.unlockedUpTo];
        queue.unlockedUpTo++;
        processed++;
    }
    return queue.unlockedBalanceSum;
}
```

Callers in `handleNlpLockedBalance` and `getNlpUnlockedBalance` should pass a safe `maxIterations` bound. Alternatively, enforce a protocol-level cap on `balanceCount - unlockedUpTo` (i.e., reject new mints if the queue depth exceeds a threshold), preventing unbounded accumulation at the source.

---

### Proof of Concept

1. User calls `MintNlp` (via `Endpoint.submitTransactionsChecked`) N times, each at a different oracle tick. Each call reaches `SpotEngine.updateBalance(NLP_PRODUCT_ID, subaccount, +amount)` → `handleNlpLockedBalance` → appends entry → `queue.balanceCount = N`. [7](#0-6) 

2. User waits for `NLP_LOCK_PERIOD` to elapse. All N entries now satisfy `unlockedAt <= getOracleTime()`.

3. User calls `BurnNlp`. This reaches `SpotEngine.updateBalance(NLP_PRODUCT_ID, subaccount, -amount)` → `handleNlpLockedBalance` → `tryUnlockNlpBalance`. [8](#0-7) 

4. The `while` loop iterates N times, each iteration performing a `delete` (SSTORE) and an addition. For large N (e.g., N ≈ 5,000–10,000 on Arbitrum), the transaction exceeds the gas limit and reverts.

5. Every subsequent `BurnNlp` or `MintNlp` for this subaccount also reverts. The user's NLP collateral is permanently locked with no recovery path.
