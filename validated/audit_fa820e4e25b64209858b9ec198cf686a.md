### Title
`verify_transaction_inclusion_v2` Coinbase Anchor Does Not Validate Leaf-Node Identity, Enabling Merkle Proof Forgery — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` was introduced to fix the 64-byte transaction Merkle forgery vulnerability present in the deprecated `verify_transaction_inclusion`. Its mitigation requires a caller-supplied coinbase proof of equal length to the transaction proof. However, the function never verifies that `coinbase_tx_id` is an actual leaf node (a real transaction). An attacker can supply an internal Merkle tree node as both `coinbase_tx_id` and `tx_id`, satisfying all checks while proving the inclusion of a transaction that does not exist.

### Finding Description

`verify_transaction_inclusion_v2` performs three checks before returning a result:

1. `merkle_proof.len() == coinbase_merkle_proof.len()`
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == header.block_header.merkle_root`
3. (via the deprecated call) `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.block_header.merkle_root` [1](#0-0) 

The coinbase check at step 2 is intended to anchor the proof depth to a real transaction, preventing an attacker from presenting an internal node at a shallower depth. However, `compute_root_from_merkle_proof` is purely positional — it does not distinguish between a leaf hash and an internal node hash. For a block with four transactions `[A, B, C, D]`, the Merkle root is `hash(hash(A,B), hash(C,D))`. The internal node `N = hash(A,B)` satisfies:

```
compute_root_from_merkle_proof(N, 0, [hash(C,D)])
  = hash(N, hash(C,D))
  = hash(hash(A,B), hash(C,D))
  = merkle_root  ✓
``` [2](#0-1) 

An attacker sets `coinbase_tx_id = N` and `coinbase_merkle_proof = [hash(C,D)]`. The coinbase check passes. They then set `tx_id = N`, `tx_index = 0`, `merkle_proof = [hash(C,D)]`. Both proofs have length 1, so the length equality check passes. The transaction inclusion check also passes. The function returns `true` for `N`, which is not a real transaction.

The root cause is that the function blindly accepts the caller-supplied `coinbase_tx_id` as the coinbase without verifying it is a leaf node — directly analogous to the reported vulnerability where a precompile blindly accepted a `Transfer` object without checking its source and target. [3](#0-2) 

### Impact Explanation

Any downstream NEAR contract (e.g., a Bitcoin bridge) that calls `verify_transaction_inclusion_v2` to authorize fund releases can be deceived into releasing funds for a Bitcoin transaction that was never broadcast or confirmed. The attacker needs only a real mainchain block with four or more transactions and the ability to compute `hash(A,B)` from the block's publicly available Merkle tree. No privileged access is required.

### Likelihood Explanation

The attack is trivially executable by any unprivileged NEAR caller. All required inputs — the block hash, the Merkle tree node hashes — are publicly available from the Bitcoin blockchain. The attacker does not need to mine a block or control any key. The only prerequisite is that a downstream contract uses `verify_transaction_inclusion_v2` to gate a fund-releasing action.

### Recommendation

Before performing the coinbase Merkle proof check, independently verify that `coinbase_tx_id` is the actual coinbase transaction of the block. This requires the caller to supply the raw coinbase transaction bytes, deserialize them on-chain, compute their hash, and compare it to `coinbase_tx_id`. Only then does the coinbase anchor meaningfully prevent internal-node forgery. Alternatively, enforce that `coinbase_tx_id != tx_id` and that the coinbase proof path is disjoint from the transaction proof path.

### Proof of Concept

Given a real mainchain Bitcoin block with transactions `[A, B, C, D]` and `merkle_root = hash(hash(A,B), hash(C,D))`:

```
coinbase_tx_id       = hash(A, B)          // internal node, not a real tx
coinbase_merkle_proof = [hash(C, D)]        // length 1
tx_id                = hash(A, B)          // same internal node
tx_index             = 0
merkle_proof         = [hash(C, D)]        // length 1
confirmations        = 1
tx_block_blockhash   = <valid mainchain block hash>
```

Step-by-step evaluation inside `verify_transaction_inclusion_v2`:

1. `merkle_proof.len() (1) == coinbase_merkle_proof.len() (1)` → **passes**
2. `compute_root_from_merkle_proof(hash(A,B), 0, [hash(C,D)]) = hash(hash(A,B), hash(C,D)) = merkle_root` → **passes**
3. (via deprecated call) `compute_root_from_merkle_proof(hash(A,B), 0, [hash(C,D)]) = merkle_root` → **passes**

Result: `verify_transaction_inclusion_v2` returns **`true`** for `tx_id = hash(A,B)`, a hash that corresponds to no real Bitcoin transaction. [1](#0-0) [2](#0-1)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
