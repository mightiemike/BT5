Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP-Position Recipient (`owner`) Instead of Actual Caller (`sender`), Enabling Allowlist Bypass and Breaking Vault-Owner Deposits — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead enforces the allowlist against `owner` — a free, caller-controlled parameter with no on-chain restriction in `addLiquidity`. This creates two concrete failure modes: an unauthorized caller can bypass the allowlist by nominating any allowlisted address as `owner`, and a legitimately authorized caller is blocked whenever they assign the LP position to an address (e.g., a vault or multisig) that is not itself in the allowlist.

## Finding Description

`MetricOmmPool.addLiquidity` accepts a free `owner` parameter and passes `msg.sender` as `sender` and the caller-supplied `owner` to the extension hook: [1](#0-0) 

There is no guard in `addLiquidity` requiring `msg.sender == owner` (contrast with `removeLiquidity` which enforces `if (msg.sender != owner) revert NotPositionOwner()`): [2](#0-1) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and discarded, and the allowlist lookup is performed on `owner`: [3](#0-2) 

The allowlist state is keyed `allowedDepositor[pool][depositor]` and is set via `setAllowedToDeposit`: [4](#0-3) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller), not the recipient: [5](#0-4) 

**Path A — Bypass:** Bob (not allowlisted) calls `pool.addLiquidity(owner=Alice, ...)`. The extension receives `beforeAddLiquidity(sender=Bob, owner=Alice, ...)`, checks `allowedDepositor[pool][Alice]` → `true` → passes. Bob's tokens enter the pool; Alice receives LP shares. The allowlist is defeated.

**Path B — Broken functionality:** Alice (allowlisted) calls `pool.addLiquidity(owner=Vault, ...)`. The extension checks `allowedDepositor[pool][Vault]` → `false` → reverts `NotAllowedToDeposit`. Alice cannot assign LP positions to any vault or multisig not individually listed, making the vault-owner pattern entirely unusable on allowlisted pools.

## Impact Explanation

Path A constitutes an admin-boundary break: any unprivileged address can inject liquidity into a KYC/risk-restricted pool by supplying an allowlisted address as `owner`. The pool admin's intent to restrict who may provide liquidity is violated. Path B constitutes broken core pool functionality: a legitimately authorized depositor cannot use the standard vault-owner pattern, permanently blocking a core liquidity flow. Both impacts are within the allowed impact gate (admin-boundary break; broken core pool functionality causing unusable liquidity flows).

## Likelihood Explanation

Any pool deploying `DepositAllowlistExtension` with `allowAllDepositors == false` is immediately affected. Path A requires only a standard `addLiquidity` call with a crafted `owner` argument — no special permissions, flash loans, or oracle manipulation. Path B is triggered by any allowlisted depositor who attempts the common pattern of depositing into a vault-owned position. Both paths are trivially reachable by any unprivileged caller.

## Recommendation

Replace the `owner` check with a `sender` check in `beforeAddLiquidity`:

```solidity
// Correct — checks actual caller
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Pool admins who need to allow vault-owner patterns should allowlist the vault's caller address (`sender`), not the vault contract itself.

## Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][Alice] = true
  allowedDepositor[pool][Bob]   = false  (Bob is unauthorized)
  allowedDepositor[pool][Vault] = false  (Vault is not listed)

--- Path A: Bypass ---
Bob calls: pool.addLiquidity(owner=Alice, salt=0, deltas=..., ...)
  _beforeAddLiquidity(msg.sender=Bob, owner=Alice, ...)
  Extension: beforeAddLiquidity(sender=Bob [discarded], owner=Alice, ...)
  Check: allowedDepositor[pool][Alice] == true → PASSES
  Result: Bob's tokens enter the pool; Alice receives LP shares.
  Bob has bypassed the allowlist.

--- Path B: Broken functionality ---
Alice calls: pool.addLiquidity(owner=Vault, salt=0, deltas=..., ...)
  _beforeAddLiquidity(msg.sender=Alice, owner=Vault, ...)
  Extension: beforeAddLiquidity(sender=Alice [discarded], owner=Vault, ...)
  Check: allowedDepositor[pool][Vault] == false → REVERTS NotAllowedToDeposit
  Result: Alice (authorized) cannot assign her LP position to a vault.
```

A Foundry test can confirm both paths by deploying a pool with `DepositAllowlistExtension`, setting `allowedDepositor[pool][Alice] = true`, then asserting that Bob's `addLiquidity(owner=Alice, ...)` succeeds and Alice's `addLiquidity(owner=Vault, ...)` reverts.
