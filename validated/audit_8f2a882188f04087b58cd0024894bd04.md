### Title
`StatelessTransactionValidator::validate` Never Calls `validate_proof_size`, Allowing Oversized Proofs to Bypass Gateway Admission - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

The `StatelessTransactionValidator::validate` function performs several checks on incoming invoke transactions — including `validate_client_side_proving_allowed` and `validate_proof_facts_and_proof_consistency` — but never calls the `validate_proof_size` method that is defined in the same struct. As a result, any invoke transaction carrying a proof whose element count exceeds the configured `max_proof_size` passes stateless validation and is admitted to the mempool.

---

### Finding Description

`StatelessTransactionValidator::validate` is the single gateway-level admission gate for all incoming `RpcTransaction`s. For invoke transactions it calls two proof-related checks:

```rust
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;
    self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
}
``` [1](#0-0) 

A third method, `validate_proof_size`, is defined in the same `impl` block and checks that `tx.proof.0.len()` does not exceed `self.config.max_proof_size`:

```rust
fn validate_proof_size(
    &self,
    tx: &RpcInvokeTransaction,
) -> StatelessTransactionValidatorResult<()> {
    let RpcInvokeTransaction::V3(tx) = tx;
    let proof_size = tx.proof.0.len();
    if proof_size > self.config.max_proof_size {
        return Err(StatelessTransactionValidatorError::ProofTooLarge {
            proof_size,
            max_proof_size: self.config.max_proof_size,
        });
    }
    Ok(())
}
``` [2](#0-1)

This is the direct Sequencer-native analog of the LES bug: just as `LesApiBackend::SendTx` was missing the `seqRPCService.CallContext` forwarding step that `EthAPIBackend::SendTx` correctly performed, `validate` is missing the `validate_proof_size` call that should appear alongside the other proof-related validations.

The `convert_rpc_tx_to_internal` path in `TransactionConverter` then spawns a proof-verification task for any invoke transaction whose `proof_facts` is non-empty: [4](#0-3) 

Because `validate_proof_size` is never invoked, a transaction with a proof of arbitrary element count reaches `spawn_proof_verification` and, if the proof cryptographically verifies, is stored in the proof manager and forwarded to the mempool — entirely bypassing the `max_proof_size` admission gate.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An unprivileged user can submit an `RpcInvokeTransaction::V3` with a `proof` array whose length exceeds `max_proof_size`. The gateway accepts it, the proof-verification task runs against the oversized payload (CPU/memory exhaustion), and on success the transaction is stored in the proof manager and inserted into the mempool. The operator-configured `max_proof_size` admission control is completely inoperative.

---

### Likelihood Explanation

The trigger is fully unprivileged: any caller of the HTTP `add_transaction` endpoint or the P2P propagation path can craft an invoke transaction with an arbitrarily large `proof` field. No special account, key, or network position is required. The `allow_client_side_proving` flag must be `true` (otherwise `validate_client_side_proving_allowed` rejects the transaction first), but that flag is the intended production setting for client-side proving deployments.

---

### Recommendation

Add the missing call inside the invoke branch of `validate`:

```rust
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;
    self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
    self.validate_proof_size(invoke_tx)?;   // ← add this
}
``` [1](#0-0) 

---

### Proof of Concept

1. Enable `allow_client_side_proving = true` in the gateway config.
2. Construct an `RpcInvokeTransaction::V3` with non-empty `proof_facts` and a `proof` whose `0` vector contains `max_proof_size + 1` elements (any values).
3. POST it to the gateway's `/add_transaction` endpoint.
4. Observe: `validate_proof_facts_and_proof_consistency` passes (both fields non-empty), `validate_proof_size` is never reached, the transaction is converted to `InternalRpcTransaction`, the proof-verification task is spawned, and — if the proof bytes happen to verify — the transaction is admitted to the mempool.

The `validate_proof_size` function exists, has the correct logic, and is tested in isolation, but the call site in `validate` was simply never added — the same class of oversight as the LES forwarding omission. [2](#0-1) [3](#0-2)
