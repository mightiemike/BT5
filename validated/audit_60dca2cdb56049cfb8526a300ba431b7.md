### Title
Linked Signer Can Unilaterally Replace Itself via `LinkSigner` Transaction — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The sequencer-path handler for `LinkSigner` transactions passes `allowLinkedSigner: true` to `validateSignedTx`, meaning the **current linked signer** — not just the subaccount owner — can sign a `LinkSigner` transaction and replace itself with any arbitrary address. The slow-mode path for the same operation correctly restricts this to the subaccount owner. The asymmetry creates a concrete privilege-escalation path: a compromised or malicious linked signer can permanently seize signing authority over a subaccount and subsequently drain its collateral.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch is:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` calls `validateSignature`, which in turn calls `verifier.validateSignature` with the current linked signer as the second accepted signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if the recovered address equals **either** the subaccount owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

The slow-mode path for the identical operation enforces the opposite: it calls `validateSender(txn.sender, sender)`, which requires `msg.sender` to be the address embedded in the subaccount bytes — i.e., the owner only:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The two paths are therefore inconsistent: the slow-mode path enforces owner-only authorization for `LinkSigner`, while the sequencer path allows the current linked signer to authorize a replacement.

---

### Impact Explanation

Once a linked signer replaces itself with an attacker-controlled address, the attacker inherits all permissions granted to a linked signer across the protocol. Every transaction type processed with `allowLinkedSigner: true` — including `WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, `LiquidateSubaccount`, `MintNlp`, `BurnNlp` — becomes signable by the attacker. The attacker can immediately sign a `WithdrawCollateral` or `WithdrawCollateralV2` transaction to drain the subaccount's collateral to an arbitrary `sendTo` address. The subaccount owner retains the ability to sign a counter-`LinkSigner` transaction, but the attacker can race to drain funds before the owner detects the change, and can also keep re-replacing the linked signer if the owner attempts to recover.

---

### Likelihood Explanation

Linked signers are the standard mechanism for trading bots, API keys, and automated strategies in DEX protocols. Any private-key compromise of a linked signer — through server breach, leaked `.env` file, or supply-chain attack on a trading bot — immediately enables this escalation. The attacker needs only one valid nonce and one signed `LinkSigner` transaction. No admin access, governance capture, or social engineering is required. The sequencer processes the transaction in the normal fast path with no additional checks.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` branch in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may authorize a new linked signer
);
```

This aligns the sequencer path with the slow-mode path, which already enforces owner-only authorization for `LinkSigner` via `validateSender`.

---

### Proof of Concept

1. Alice owns subaccount `aliceSub` and has linked `botAddress` as her trading bot's linked signer via a prior `LinkSigner` transaction.
2. An attacker compromises `botAddress` and obtains its private key.
3. The attacker constructs a `SignedLinkSigner` transaction:
   - `sender = aliceSub`
   - `signer = attackerAddress` (bytes32-encoded)
   - `nonce = nonces[address(aliceSub)]` (current nonce)
4. The attacker signs the EIP-712 digest with `botAddress`'s key and submits the transaction to the sequencer.
5. The sequencer calls `processTransactionImpl`. `validateSignedTx` recovers `botAddress` from the signature, which equals `getLinkedSigner(aliceSub)`, so validation passes.
6. `linkedSigners[aliceSub]` is overwritten with `attackerAddress`.
7. The attacker immediately signs a `WithdrawCollateralV2` transaction (`sender = aliceSub`, `sendTo = attackerEOA`, `allowLinkedSigner = true`) with `attackerAddress`'s key.
8. Alice's collateral is transferred to the attacker's address. The corrupted state is: `linkedSigners[aliceSub] = attackerAddress` and Alice's collateral balance reduced to zero.
