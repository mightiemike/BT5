### Title
Incomplete 64-Byte Transaction Merkle Proof Forgery Mitigation in `verify_transaction_inclusion_v2` — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` accepts a caller-supplied `coinbase_tx_id` without verifying it is the actual coinbase transaction hash. An unprivileged NEAR caller can supply an internal Merkle-tree node as `coinbase_tx_id`, paired with a shortened proof, to satisfy the coinbase anchor check while simultaneously proving a different internal node as a "transaction." The function returns `true` for a transaction that does not exist in the block.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions). The mitigation strategy is to require a coinbase proof of the same length as the target-transaction proof, anchoring the tree depth.

The implementation is:

```rust
require!(
    args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
    "Coinbase merkle proof and transaction merkle proof should have the same length"
);

let header = self.headers_pool.get(&args.tx_block_blockhash)...;

require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);

self.verify_transaction_inclusion(args.into())
``` [1](#0-0) 

The critical flaw: `coinbase_tx_id` is **never verified to be the hash of the actual coinbase transaction**. The check only verifies that `coinbase_tx_id`, combined with `coinbase_merkle_proof`, produces the stored `merkle_root`. Any internal Merkle node satisfies this check if the proof is shortened accordingly.

`compute_root_from_merkle_proof` is a pure path-traversal function with no leaf-vs-internal-node distinction:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;
    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
    current_hash
}
``` [2](#0-1) 

For a block whose Merkle tree has depth N, an internal node at level L (counting from leaves at level 0) has a valid path to the root of length `N − L`. The attacker can:

1. Pick any internal node **C** at level L, position 0 (the ancestor of the coinbase at that level). Supply it as `coinbase_tx_id` with a proof of length `N − L`. The coinbase check passes.
2. Pick any internal node **I** at level L, position P. Supply it as `tx_id` with a proof of length `N − L`. The length equality check passes.
3. The deprecated v1 call inside v2 then checks `compute_root_from_merkle_proof(I, P, merkle_proof) == merkle_root`, which also passes.

The function returns `true` for a transaction that does not exist.

The deprecated v1 function explicitly documents this limitation:

```
/// # Warning
/// This function may return `true` if the provided `tx_id` is a hash of an internal node
/// in the Merkle tree rather than a valid transaction hash.
``` [3](#0-2) 

The v2 function was intended to close this gap but does not.

---

### Impact Explanation

Any NEAR DApp calling `verify_transaction_inclusion_v2` to gate value transfers, unlock funds, or record Bitcoin payment proofs receives a `true` result for a fabricated transaction. The corrupted state is the **proof result** returned to the consuming contract: a non-existent Bitcoin transaction is certified as confirmed in a mainchain block. The attacker controls `tx_id`, `tx_index`, `tx_block_blockhash`, `coinbase_tx_id`, and both proof vectors — all caller-supplied inputs with no privileged-role requirement.

**Impact: 5** — Proof-verification forgery enabling false Bitcoin transaction inclusion claims against any consumer of this API.

---

### Likelihood Explanation

The attack requires only:
- Knowledge of any mainchain Bitcoin block with ≥ 2 transactions (all blocks are public)
- Ability to call `verify_transaction_inclusion_v2` as an unprivileged NEAR account

No privileged role, no leaked key, no social engineering. The function is `#[pause]`-gated but has no caller restriction.

**Likelihood: 5** — Trivially executable by any NEAR caller with access to public Bitcoin block data.

---

### Recommendation

The `coinbase_tx_id` must be verified to be the actual leaf-level coinbase transaction hash, not merely any value that traverses to the merkle root. Two concrete options:

1. **Require the full coinbase raw bytes** in `ProofArgsV2` and compute the coinbase txid on-chain (double-SHA256), then verify the resulting hash against the Merkle proof. This eliminates the ability to substitute an internal node.
2. **Require `coinbase_merkle_proof.len() > merkle_proof.len()`** as an additional constraint. Because the coinbase is always at the deepest leaf level, its proof must be strictly longer than any proof for an internal node at a shallower level. This does not fully close the gap for equal-depth internal nodes but raises the bar significantly.

Option 1 is the correct fix. Option 2 is a partial mitigation only.

### Proof of Concept

```
Level 3 (root):  R
Level 2:         AB        CD
Level 1:         A  B      C  D
Level 0 (leaf):  t0 t1 t2 t3 t4 t5 t6 t7
```

Where `A = hash(t0, t1)`, `B = hash(t2, t3)`, `AB = hash(A, B)`, etc.
