### Title
Fee-on-Transfer Token Deposit Over-Credits Internal Balance, Causing Withdrawal Reverts — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` records the caller-supplied `amount` in the slow-mode transaction queue rather than the actual tokens received by the Clearinghouse. For fee-on-transfer tokens, this inflates `totalDepositsNormalized` beyond the real token balance held on-chain. When a user later attempts to withdraw their full credited amount, `Clearinghouse.handleWithdrawTransfer` calls `token.safeTransfer(withdrawPool, amount)` and reverts because the Clearinghouse holds fewer tokens than the internal ledger promises. The shortfall is permanent and unrecoverable through any normal protocol path.

---

### Finding Description

**Deposit path — user-specified amount is recorded verbatim:**

In `Endpoint.depositCollateralWithReferral`, the function first pulls tokens from the caller, then enqueues a slow-mode transaction that carries the raw `amount` parameter:

```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)          // ← actual received may be less for fee-on-transfer tokens
);
// ...
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    // ...
    tx: abi.encodePacked(
        uint8(TransactionType.DepositCollateral),
        abi.encode(
            DepositCollateral({
                sender: subaccount,
                productId: productId,
                amount: amount   // ← user-supplied, not measured post-transfer
            })
        )
    )
});
``` [1](#0-0) 

When the sequencer processes this slow-mode transaction, `Clearinghouse.depositCollateral` credits the subaccount with the full `amount * multiplier`:

```solidity
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [2](#0-1) 

For a fee-on-transfer token with fee `f`, the Clearinghouse physically holds `amount - f` tokens, but `totalDepositsNormalized` is inflated as if `amount` tokens arrived.

**Withdrawal path — safeTransfer reverts on shortfall:**

`Clearinghouse.withdrawCollateral` calls `handleWithdrawTransfer` *before* decrementing the internal balance:

```solidity
handleWithdrawTransfer(token, sendTo, amount, idx);   // safeTransfer here
// ...
spotEngine.updateBalance(productId, sender, amountRealized);
spotEngine.assertUtilization(productId);
``` [3](#0-2) 

`handleWithdrawTransfer` calls `token.safeTransfer(withdrawPool, uint256(amount))`:

```solidity
function handleWithdrawTransfer(
    IERC20Base token,
    address to,
    uint128 amount,
    uint64 idx
) internal virtual {
    token.safeTransfer(withdrawPool, uint256(amount));
    BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
}
``` [4](#0-3) 

If the Clearinghouse holds `amount - f` but the withdrawal requests `amount`, `safeTransfer` reverts. The entire `withdrawCollateral` call reverts, the user's internal balance is not decremented, and the user cannot retrieve the fee-shortfall portion. The `assertUtilization` check (`totalDeposits >= totalBorrows`) is purely internal accounting and never compares against the actual ERC20 balance, so it provides no protection here. [5](#0-4) 

The discrepancy accumulates with every deposit. In a pool with many depositors of the same fee-on-transfer token, later withdrawers will find the Clearinghouse balance exhausted before their credited amounts are satisfied — a classic bank-run insolvency caused by the accounting gap.

---

### Impact Explanation

- Users who deposit a fee-on-transfer token are credited with more than the Clearinghouse physically holds.
- Any withdrawal request for the full credited amount reverts at `safeTransfer`.
- The fee shortfall per deposit is permanently unwithdrawable; it accumulates across all depositors.
- In aggregate, the last depositors to withdraw will find the Clearinghouse insolvent for that product: their tokens are stuck with no upgrade or recovery path available through the standard protocol.

---

### Likelihood Explanation

The trigger requires a fee-on-transfer token to be registered as a spot product (an admin action). Once registered, the vulnerability is activated by any ordinary user calling `depositCollateralWithReferral` or by `DirectDepositV1.creditDeposit()` (callable by anyone with no access control):

```solidity
uint256 balance = token.balanceOf(address(this));
if (balance != 0) {
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
}
``` [6](#0-5) 

Fee-on-transfer tokens are a well-known token class (e.g., STA, PAXG in certain modes, tokens with protocol fees). The admin listing such a token is not itself the attack; the vulnerability fires automatically on every subsequent deposit by any unprivileged user.

---

### Recommendation

Measure the actual received amount by snapshotting the Clearinghouse token balance before and after `handleDepositTransfer`, and use the delta — not the caller-supplied `amount` — when constructing the slow-mode `DepositCollateral` transaction. This ensures `totalDepositsNormalized` never exceeds the real token balance held on-chain.

---

### Proof of Concept

1. Admin registers a fee-on-transfer token (1% fee) as product `X`.
2. User calls `depositCollateralWithReferral(subaccount, X, 1_000e6, "")`.
3. `handleDepositTransfer` pulls 1_000e6 from the user; the token contract deducts 1% → Clearinghouse receives **990e6**.
4. Slow-mode tx is enqueued with `amount = 1_000e6`.
5. Sequencer processes the slow-mode tx: user's internal balance is credited with `1_000e6 * multiplier`; `totalDepositsNormalized` reflects 1_000e6.
6. User requests withdrawal of 1_000e6 via the sequencer.
7. `Clearinghouse.withdrawCollateral` calls `handleWithdrawTransfer(token, sendTo, 1_000e6, idx)`.
8. `token.safeTransfer(withdrawPool, 1_000e6)` — Clearinghouse holds only 990e6 → **REVERT**.
9. The user's 990e6 tokens are stuck in the Clearinghouse. The 10e6 phantom credit can never be fulfilled. With N depositors the shortfall is `N × fee`, eventually making the product insolvent.
