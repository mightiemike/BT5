Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks `sender` (immediate pool caller) instead of the actual user, enabling full allowlist bypass through `MetricOmmSimpleRouter` â€” (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` enforces its per-pool allowlist against `sender`, which the pool sets to `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates the call, `sender` is the router address, not the end-user. A pool admin who allowlists the router to enable router-based swaps inadvertently grants every address â€” including non-allowlisted ones â€” the ability to swap in the curated pool, completely defeating the access control.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- immediate caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension. `SwapAllowlistExtension.beforeSwap` then checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly, making the router `msg.sender` at the pool level:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,   // actual user is only the recipient
        ...
        params.extensionData
    );
```

So the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. If the admin allowlists the router to make router-based swaps work, every caller of the router passes the check regardless of individual allowlist status.

`DepositAllowlistExtension` avoids this by checking `owner` (the position beneficiary, second argument) rather than `sender` (the payer/caller):

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-38
function beforeAddLiquidity(address, address owner, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

The integration test confirms the design: `test_allowedSwapSucceeds` allowlists `address(callers[0])` â€” the `TestCaller` contract that directly calls the pool â€” not `users[0]` (the human recipient), which is exactly the wrong actor when a router intermediary is present.

## Impact Explanation

A pool admin deploying a curated pool with `SwapAllowlistExtension` faces two bad outcomes:

1. **Router not allowlisted**: Individually allowlisted users who call through `MetricOmmSimpleRouter` are rejected with `NotAllowedToSwap`. The primary user-facing swap path is broken for legitimate users of the curated pool.

2. **Router allowlisted** (the natural admin fix): `allowedSwapper[pool][router] = true` passes the check for every caller of the router, regardless of whether the actual end-user is on the allowlist. Any address can bypass the per-user curation policy by routing through the public router. The allowlist is rendered entirely ineffective for router-based swaps.

This constitutes an admin-boundary break: an unprivileged address executes swaps in a pool explicitly configured to deny them, with zero privileged access required.

## Likelihood Explanation

- `SwapAllowlistExtension` is a production periphery extension explicitly designed for curated pools.
- `MetricOmmSimpleRouter` is the primary user-facing swap entry point; all four swap methods (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) call `pool.swap()` directly, making the router `msg.sender`.
- A pool admin who wants router-based swaps to work will naturally attempt to allowlist the router, triggering the bypass.
- No privileged attacker capability is required beyond calling the public router.
- The asymmetry with `DepositAllowlistExtension` (which correctly checks `owner`) makes the swap extension's behavior non-obvious and likely to be misconfigured.

## Recommendation

The `beforeSwap` hook receives both `sender` (immediate caller) and `recipient` (output beneficiary) as arguments. For router-mediated swaps, the actual user identity must be communicated through `extensionData`. The recommended fix is to have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that address when present, falling back to `sender` for direct pool calls. Alternatively, document and enforce at the factory level that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter`.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Call `swapExtension.setAllowedToSwap(pool, router, true)` â€” admin allowlists the router so router-based swaps work.
3. From `attacker` (not in the allowlist), call `router.exactInputSingle({pool: pool, recipient: attacker, ...})`.
4. The router calls `pool.swap(recipient=attacker, ...)` with `msg.sender = router`.
5. The pool calls `extension.beforeSwap(sender=router, recipient=attacker, ...)`.
6. The extension checks `allowedSwapper[pool][router]` â†’ `true` â†’ passes.
7. The swap executes for `attacker` despite `allowedSwapper[pool][attacker]` being `false`.
