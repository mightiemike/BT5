### Title
`SyncStateReader::get_compiled_class_hash` Panics via `todo!()` on Any Declare V2/V3 Transaction — (`File: crates/apollo_gateway/src/sync_state_reader.rs`)

---

### Summary

`SyncStateReader`, the production state reader used by the gateway's stateful transaction validator, implements `BlockifierStateReader::get_compiled_class_hash` with `todo!()`. Any user submitting a Declare V2/V3 transaction causes the gateway to call this method during stateful validation, triggering an unconditional runtime panic and crashing the sequencer node.

---

### Finding Description

`SyncStateReader` implements the `BlockifierStateReader` trait for the gateway's stateful validation path. All methods are implemented except `get_compiled_class_hash`, which contains `todo!()`: [1](#0-0) 

```rust
fn get_compiled_class_hash(&self, _class_hash: ClassHash) -> StateResult<CompiledClassHash> {
    todo!()
}
```

The `SyncOrGenesisStateReader` enum wraps `SyncStateReader` and unconditionally delegates `get_compiled_class_hash` to it: [2](#0-1) 

The `BlockifierStateReader` trait defines `get_compiled_class_hash` as a required method for all state readers: [3](#0-2) 

During Declare V2/V3 transaction execution, `CachedState::update_initial_values_of_write_only_access` calls `get_compiled_class_hash` on the underlying reader to verify the class is not already declared: [4](#0-3) 

The `GenesisStateReader` (used only when no blocks exist yet) correctly returns `Ok(CompiledClassHash::default())`, but `SyncStateReader` — used for all normal operation — panics: [5](#0-4) 

---

### Impact Explanation

Any unprivileged user can submit a syntactically valid Declare V2/V3 transaction to the gateway RPC endpoint. The gateway's `StatefulTransactionValidator` creates a `SyncStateReader`-backed `CachedState`, and the blockifier calls `get_compiled_class_hash` during execution. This hits `todo!()`, panicking the tokio worker thread and crashing the sequencer node. The node stops processing all transactions and cannot sync with the relay chain until restarted.

This matches: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing** (via node crash).

---

### Likelihood Explanation

Declare V2/V3 transactions are a standard, publicly documented Starknet transaction type. Any user deploying a Cairo 1 contract triggers this path. No special privileges, keys, or knowledge of the bug are required. The trigger is a normal user action.

---

### Recommendation

**Short term**: Implement `get_compiled_class_hash` in `SyncStateReader` by querying the state sync client (analogous to how `get_storage_at`, `get_nonce_at`, and `get_class_hash_at` are implemented), or return `Ok(CompiledClassHash::default())` as a safe stub if the method is not semantically needed for the gateway's validation logic.

**Long term**: Add a compile-time or CI lint that rejects `todo!()` / `unimplemented!()` in any type that implements a public trait used in the production transaction execution path.

---

### Proof of Concept

1. Start the Apollo sequencer node with at least one block finalized (so `SyncStateReader` is used, not `GenesisStateReader`).
2. Submit a valid Declare V2/V3 transaction to the gateway RPC endpoint (e.g., `starknet_addDeclareTransaction` with a Sierra class and compiled class hash).
3. The gateway's `StatefulTransactionValidator` calls `SyncStateReader::get_compiled_class_hash` via `CachedState`.
4. The `todo!()` macro fires, panicking the tokio worker thread.
5. The sequencer node crashes and stops processing all transactions.

The panic message will be: `thread 'tokio-runtime-worker' panicked at 'not yet implemented', crates/apollo_gateway/src/sync_state_reader.rs:198`. [1](#0-0)
