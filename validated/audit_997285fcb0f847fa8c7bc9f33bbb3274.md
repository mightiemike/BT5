### Title
Coinbase Merkle Proof Check Enforced in `verify_transaction_inclusion_v2` But Bypassed via Direct Call to Deprecated `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

---

### Summary

The contract introduced `verify_transaction_inclusion_v2` to enforce a coinbase Merkle proof check that prevents the 64-byte transaction Merkle proof forgery vulnerability. However, the original `verify_transaction_inclusion` (v1) remains publicly callable on-chain and does not enforce this check. Any unprivileged NEAR caller can bypass the coinbase proof validation by calling v1 directly, obtaining a `true` result for a forged transaction inclusion proof.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces a coinbase Merkle proof check before verifying the transaction proof: [1](#0-0) 

This check is absent in v1, which performs only a raw Merkle root comparison: [2](#0-1) 

Despite being marked `#[deprecated]`, v1 carries no runtime restriction — the Rust `#[deprecated]` attribute is a compile-time warning only. Any NEAR account can call it directly: [3](#0-2) 

The contract's own documentation acknowledges the risk explicitly: [4](#0-3) 

The analog to the external report is exact:

| External Report | This Repository |
|---|---|
| Minimum collateral check enforced in `maxBorrowableUSDS` (borrow path) | Coinbase Merkle proof check enforced in `verify_transaction_inclusion_v2` |
| Check **not** enforced in the withdrawal path | Check **not** enforced in `verify_transaction_inclusion` (v1) |
| Withdrawal path still publicly callable | v1 still publicly callable by any NEAR account |
| Attacker creates small loan via withdrawal | Attacker forges tx inclusion proof via v1 |

### Impact Explanation

The 64-byte transaction Merkle proof forgery attack allows an attacker to supply a `tx_id` that is actually an internal Merkle tree node hash (not a real transaction). Because Bitcoin's Merkle tree does not distinguish leaf nodes from internal nodes, a valid Merkle path from that internal node to the block's Merkle root can be constructed. Without the coinbase anchor check, `verify_transaction_inclusion` will return `true` for this forged proof.

Any downstream NEAR contract that calls v1 to gate a cross-chain action (e.g., token minting, bridge unlock, settlement) can be deceived into accepting a fabricated Bitcoin transaction as confirmed.

---

### Likelihood Explanation

Medium. The v1 entry point is reachable by any unprivileged NEAR caller with no deposit or role requirement beyond the `#[pause]` gate (which is not paused in normal operation). The 64-byte transaction attack is a well-documented Bitcoin vulnerability with published exploit techniques. Any integrator that has not yet migrated to v2 is exposed.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI, or add a runtime `panic!` / `require!(false, "use verify_transaction_inclusion_v2")` to make it uncallable. If strict backward compatibility is required, backport the coinbase Merkle proof check into v1 so both paths enforce the same invariant.

---

### Proof of Concept

1. Attacker identifies a downstream NEAR contract that calls `verify_transaction_inclusion` (v1) to authorize an action.
2. Attacker selects any Bitcoin block already accepted by the light client (present in `mainchain_header_to_height`).
3. Attacker picks an internal Merkle tree node from that block and uses it as `tx_id`.
4. Attacker constructs a valid Merkle path from that internal node to the block's `merkle_root` (standard 64-byte attack construction).
5. Attacker calls `verify_transaction_inclusion` with `tx_block_blockhash` = real block hash, `tx_id` = internal node hash, `tx_index` = crafted index, `merkle_proof` = crafted path, `confirmations` = any value ≤ `gc_threshold`.
6. The function computes `compute_root_from_merkle_proof(internal_node, crafted_index, crafted_path) == merkle_root` — this evaluates to `true`.
7. The downstream contract receives `true` and executes its protected action for a transaction that does not exist.

The coinbase anchor check present in `verify_transaction_inclusion_v2` would have blocked step 6 by requiring an independent proof that the coinbase transaction (index 0) also hashes to the same root, which is infeasible to forge simultaneously. v1 has no such anchor, making the bypass trivially reachable.     pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
