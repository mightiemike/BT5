Audit Report

## Title
`DepositAllowlistExtension` gates on LP position `owner` instead of the actual depositing `sender`, allowing any unpermissioned address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the address that actually funds the deposit via the modify-liquidity callback) and instead checks `owner` (the LP share recipient). Because `MetricOmmPool.addLiquidity` imposes no constraint tying `msg.sender` to `owner`, any unpermissioned address can bypass the allowlist entirely by naming any already-allowlisted address as the position owner. The deposit allowlist — the primary access-control mechanism for restricted pools — is rendered completely ineffective.

## Finding Description

`MetricOmmPool.addLiquidity` passes both `msg.sender` (the actual caller/funder) and the caller-supplied `owner` into the extension hook chain:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly forwards both addresses to the extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` leaves the first parameter (`sender`) unnamed and never reads it, checking only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

The actual token transfer is performed by calling back into `msg.sender` of `addLiquidity` (i.e., the `sender`, not `owner`):

```solidity
IMetricOmmModifyLiquidityCallback(msg.sender)
    .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
``` [4](#0-3) 

There is no check in `addLiquidity` that ties `msg.sender` to `owner`: [5](#0-4) 

**Exploit path:**
1. Pool `P` is deployed with `DepositAllowlistExtension`; `allowAllDepositors[P] = false`.
2. Admin calls `setAllowedToDeposit(P, alice, true)`. Bob is **not** allowlisted.
3. Bob calls `P.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `extension.beforeAddLiquidity(bob, alice, ...)`.
5. Extension checks `allowedDepositor[P][alice]` → `true` → no revert.
6. Pool calls `IMetricOmmModifyLiquidityCallback(bob).metricOmmModifyLiquidityCallback(...)` → Bob's tokens are pulled.
7. Alice receives LP shares; Bob has deposited into a restricted pool without being on the allowlist.

The wrong value is `allowedDepositor[msg.sender][owner]` — it should be `allowedDepositor[msg.sender][sender]`.

## Impact Explanation

The deposit allowlist is the sole mechanism for pools that need to restrict who can provide liquidity (e.g., permissioned or regulated pools). Bypassing it means: (1) unauthorized funds enter the pool — the pool admin's restriction is completely ineffective against any caller who knows one allowlisted address; (2) pool state (`curPosInBin`, bin balances, marginal price) is altered by an address the admin never approved; (3) the named `owner` accumulates LP shares they did not initiate. This is a direct admin-boundary break: an unprivileged path bypasses a pool admin access control, qualifying as High severity under the contest rules.

## Likelihood Explanation

No special privilege is required — any EOA or contract can call `pool.addLiquidity`. Allowlisted addresses are publicly readable from the `allowedDepositor` public mapping. The attacker only needs to name one such address as `owner` and implement `metricOmmModifyLiquidityCallback` to supply tokens. The attack is repeatable every block with no cooldown or cost beyond gas and token supply. [6](#0-5) 

## Recommendation

Replace the unnamed first parameter with `sender` and gate on it instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the funder and the position owner, both addresses should be checked independently.

## Proof of Concept

1. Deploy pool `P` with `DepositAllowlistExtension`; `allowAllDepositors[P] = false`.
2. Admin calls `setAllowedToDeposit(P, alice, true)`. Bob is not allowlisted.
3. Bob deploys a contract implementing `metricOmmModifyLiquidityCallback` that transfers tokens from Bob to the pool.
4. Bob calls `P.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
5. Extension checks `allowedDepositor[P][alice]` → `true` → passes without revert.
6. Pool calls Bob's callback; Bob's tokens are transferred to the pool.
7. Alice's position shares increase; Bob has successfully deposited into a restricted pool.

A Foundry test can confirm this by: deploying the extension and pool, allowlisting Alice, calling `addLiquidity` from Bob with `owner = alice`, and asserting the call succeeds and Alice's position shares increase.
