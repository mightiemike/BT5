### Title
`ReservedDataAvailabilityMode::from()` Always Returns `DataAvailabilityMode::L1`, Silently Corrupting DA Mode in FGW-Synced V3 Transactions - (`crates/apollo_starknet_client/src/reader/objects/transaction.rs`)

---

### Summary

The `From<ReservedDataAvailabilityMode> for DataAvailabilityMode` conversion in the Feeder Gateway reader unconditionally returns `DataAvailabilityMode::L1`, ignoring the actual value of the input. This is the direct Sequencer analog of the Maia M-10 bug: a boolean flag is always encoded as the same constant regardless of the user-supplied value. Any V3 transaction carrying `nonce_data_availability_mode = L2` or `fee_data_availability_mode = L2` that is ingested through the FGW state-sync path will have both DA-mode fields silently overwritten to `L1`, producing a divergent internal representation whose recomputed hash will never match the canonical on-chain hash.

---

### Finding Description

**Root cause — the broken conversion:**

```rust
// crates/apollo_starknet_client/src/reader/objects/transaction.rs  L180-L185
impl From<ReservedDataAvailabilityMode> for starknet_api::data_availability::DataAvailabilityMode {
    // TODO(Arni): Fix this. Support L2 data availability mode.
    fn from(_: ReservedDataAvailabilityMode) -> Self {
        starknet_api::data_availability::DataAvailabilityMode::L1   // ← always L1
    }
}
```

The enum it converts from has two live variants:

```rust
// L172-L178
pub enum ReservedDataAvailabilityMode {
    Reserved = 0,   // means L1
    L2       = 1,   // means L2 — silently collapsed to L1 above
}
```

**Call sites that propagate the wrong value into V3 transaction structs:**

```rust
// L321-L334  (DeclareTransactionV3 conversion)
nonce_data_availability_mode: declare_tx
    .nonce_data_availability_mode
    .ok_or(...)?.into(),   // ReservedDataAvailabilityMode::L2 → L1 (wrong)
fee_data_availability_mode: declare_tx
    .fee_data_availability_mode
    .ok_or(...)?.into(),   // same
```

Identical patterns exist for `IntermediateDeployAccountTransaction → DeployAccountTransactionV3` and `IntermediateInvokeTransaction → InvokeTransactionV3`.

**Why the wrong DA mode corrupts the transaction hash:**

The DA mode is a direct input to the V3 hash preimage via `concat_data_availability_mode` in `crates/starknet_api/src/transaction_hash.rs`:

```rust
// L231-L238
fn concat_data_availability_mode(
    nonce_mode: &DataAvailabilityMode,
    fee_mode: &DataAvailabilityMode,
) -> Felt {
    (data_availability_mode_index(fee_mode)
        + (data_availability_mode_index(nonce_mode) << DATA_AVAILABILITY_MODE_BITS))
        .into()
}
```

`data_availability_mode_index` maps `L1 → 0` and `L2 → 1`. Forcing both modes to `L1` produces `concat_data_availability_mode = 0` even when the canonical value is `(1 << 32) + 1 = 0x100000001`. The resulting hash diverges from the on-chain hash stored in `declare_tx.transaction_hash`.

---

### Impact Explanation

When the FGW serves a V3 transaction whose `nonce_data_availability_mode` or `fee_data_availability_mode` is `1` (L2):

1. The FGW reader deserializes the field as `ReservedDataAvailabilityMode::L2` (valid deserialization — the variant exists).
2. `.into()` silently returns `DataAvailabilityMode::L1`.
3. The internal `DeclareTransactionV3` / `InvokeTransactionV3` / `DeployAccountTransactionV3` carries the wrong DA mode.
4. Any subsequent hash recomputation (e.g., during block verification, RPC `get_transaction_by_hash`, or state-diff commitment) produces a hash that does not match the canonical on-chain hash.
5. The sequencer's local view of the transaction is permanently inconsistent with the network's canonical view.

This matches the allowed impact: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."**

---

### Likelihood Explanation

L2 DA mode (volition) is not yet activated on Starknet mainnet. All current V3 transactions carry `Reserved = 0` for both DA-mode fields, so the conversion is accidentally correct today. However:

- The `ReservedDataAvailabilityMode::L2 = 1` variant is already present and deserializable.
- The FGW reader will accept any JSON payload with `"nonce_data_availability_mode": 1` without error.
- No gateway-level rejection of `L2` DA mode exists in this code path.
- The TODO comment explicitly acknowledges the defect and defers the fix.

Once volition is enabled (or if a node syncs from a network that already uses it), every affected transaction will be stored with a wrong hash, breaking block verification and RPC consistency.

---

### Recommendation

Replace the wildcard discard with a proper match:

```rust
impl From<ReservedDataAvailabilityMode> for starknet_api::data_availability::DataAvailabilityMode {
    fn from(mode: ReservedDataAvailabilityMode) -> Self {
        match mode {
            ReservedDataAvailabilityMode::Reserved => DataAvailabilityMode::L1,
            ReservedDataAvailabilityMode::L2       => DataAvailabilityMode::L2,
        }
    }
}
```

### Proof of Concept

```
FGW response (JSON):
  "nonce_data_availability_mode": 1,   // L2
  "fee_data_availability_mode":  1,    // L2

Deserialization:
  ReservedDataAvailabilityMode::L2 (value = 1)  ← correct

Conversion (.into()):
  DataAvailabilityMode::L1              ← WRONG (should be L2)

concat_data_availability_mode(L1, L1):
  data_availability_mode_index(L1) = 0
  result = 0 + (0 << 32) = 0x0         ← WRONG

concat_data_availability_mode(L2, L2) [canonical]:
  data_availability_mode_index(L2) = 1
  result = 1 + (1 << 32) = 0x100000001 ← correct

Hash divergence:
  internal hash ≠ on-chain hash
