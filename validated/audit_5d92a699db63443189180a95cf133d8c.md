### Title
`topUpBaseRewards` Ignores Pending Bonus Obligations When Reading Pool Balance, Draining Funds Needed for Bonus Reward Claims - (File: `x/tieredrewards/keeper/abci.go`)

---

### Summary

`topUpBaseRewards` reads the raw bank balance of the `RewardsPoolName` module account to determine how much it can transfer to the distribution module for base rewards top-up. It does not subtract the total pending bonus rewards already owed to all active positions. Because both base top-up and bonus payouts draw from the same pool, the BeginBlocker can drain tokens that position holders have already earned as bonus rewards, causing all subsequent `MsgClaimTierRewards` calls to fail with `ErrInsufficientBonusPool`.

---

### Finding Description

In `topUpBaseRewards` (`x/tieredrewards/keeper/abci.go`), the available pool balance is read as:

```go
poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
poolBalance := k.bankKeeper.GetBalance(ctx, poolAddr, bondDenom)
``` [1](#0-0) 

This raw balance includes tokens that are already owed as pending (accrued but unclaimed) bonus rewards to every active delegated position. The function treats the entire balance as freely available for base rewards top-up. When the fee collector balance is below the `TargetBaseRewardsRate` target, the function transfers up to the full pool balance to the distribution module:

```go
if poolBalance.Amount.LT(shortFallAmount) {
    topUpAmount = poolBalance.Amount   // drains the entire pool
}
err = k.bankKeeper.SendCoinsFromModuleToModule(ctx, types.RewardsPoolName, distributiontypes.ModuleName, ...)
``` [2](#0-1) 

The same `RewardsPoolName` account is the exclusive source for bonus reward payouts in `processEventsAndClaimBonus`:

```go
k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins)
``` [3](#0-2) 

Before paying out, `sufficientBonusPoolBalance` checks the pool's current bank balance:

```go
poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
if !poolBalance.IsAllGTE(bonus) {
    return errorsmod.Wrapf(types.ErrInsufficientBonusPool, ...)
}
``` [4](#0-3) 

After `topUpBaseRewards` has transferred tokens out of the pool, the pool balance is reduced. If the transferred amount consumed tokens that were needed for pending bonus obligations, `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` and the user's `MsgClaimTierRewards` transaction fails entirely (both base and bonus rewards are rolled back).

The `RewardsPoolBalances` gRPC query has the same blind spot — it returns the raw bank balance without subtracting pending bonus obligations, misleading operators about true available capacity:

```go
balances := q.k.bankKeeper.GetAllBalances(ctx, addr)
``` [5](#0-4) 

---

### Impact Explanation

The corrupted value is the `RewardsPoolName` module account balance as seen by `topUpBaseRewards`. It is overestimated because it includes tokens already committed as pending bonus rewards to all active positions. The BeginBlocker can transfer these tokens to the distribution module, leaving the pool unable to satisfy legitimate bonus claims. Position holders who have accrued bonus rewards over days or weeks are denied those rewards — their `MsgClaimTierRewards` transactions revert with `ErrInsufficientBonusPool`. The base rewards portion of the same transaction is also rolled back, compounding the loss of access.

---

### Likelihood Explanation

The condition is triggered automatically every block by the BeginBlocker whenever the fee collector balance falls below the `TargetBaseRewardsRate` target. This is a normal operating condition during low-fee periods or when `x/mint` inflation is disabled (as in the test configuration `base_rewards.jsonnet`). No privileged access or special transaction is required. Any block production is sufficient to trigger the drain. The more positions exist and the longer they go unclaimed, the larger the pending bonus obligation that is invisible to `topUpBaseRewards`, making the collision more likely as the chain matures.

---

### Recommendation

Track total pending bonus obligations as a module-level state variable (e.g., a `PendingBonusObligations` accumulator updated on every accrual and decremented on every payout). In `topUpBaseRewards`, compute the effective available balance as:

```
effectivePoolBalance = poolBalance - pendingBonusObligations
```

and cap `topUpAmount` against `effectivePoolBalance` rather than the raw `poolBalance`. The `RewardsPoolBalances` query should also expose the net available balance alongside the gross balance.

---

### Proof of Concept

**Setup:** Pool is funded with 1,000 tokens. One position has accrued 800 tokens of pending bonus rewards (not yet claimed). Fee collector is empty (e.g., inflation disabled). `TargetBaseRewardsRate` implies a per-block shortfall of 1,000 tokens.

**Step 1 — BeginBlocker runs (`topUpBaseRewards`):**
- `poolBalance = 1,000` (raw bank balance, includes the 800 owed as bonus)
- `shortFallAmount = 1,000 > poolBalance`, so `topUpAmount = poolBalance = 1,000`
- Transfers 1,000 tokens from `RewardsPoolName` → `distribution`
- Pool balance is now 0

**Step 2 — Position holder submits `MsgClaimTierRewards`:**
- `processEventsAndClaimBonus` computes `totalBonus = 800`
- `sufficientBonusPoolBalance` checks pool balance = 0
- `0 < 800` → returns `ErrInsufficientBonusPool`
- Entire transaction reverts; user receives neither bonus nor base rewards

The 800 tokens the user legitimately earned have been redirected to validators as base rewards top-up. The user's claim will continue to fail every block until the pool is externally refunded. [6](#0-5) [7](#0-6) [8](#0-7)

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
