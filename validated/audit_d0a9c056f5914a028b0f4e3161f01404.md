### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Exposing Consumers to 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two SPV proof endpoints: the deprecated `verify_transaction_inclusion` (v1) and the secure `verify_transaction_inclusion_v2`. The v1 endpoint is still `pub` and reachable by any unprivileged NEAR caller. It omits the coinbase-merkle-proof guard that v2 adds specifically to block the 64-byte internal-node forgery attack. Any caller that routes through v1 — including downstream consumer contracts — receives a weaker security guarantee than the contract already provides, with no indication at the call site that a safer path exists.

---

### Finding Description

`verify_transaction_inclusion` was deprecated at v0.5.0 with an explicit note to use `verify_transaction_inclusion_v2` instead, and its own docstring warns:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

Despite this, the function carries `pub` visibility and the `#[pause]` attribute — meaning it is a live, on-chain callable endpoint, not a dead code path. [2](#0-1) 

The v1 function's entire verification is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [3](#0-2) 

`compute_root_from_merkle_proof` treats `tx_id` as an opaque 32-byte leaf and hashes it up the tree without any check that it represents a real transaction rather than an internal Merkle node. [4](#0-3) 

The v2 function fixes this by first verifying a coinbase proof against the same Merkle root, which constrains the tree depth and prevents an attacker from substituting an internal node as a leaf: [5](#0-4) 

`ProofArgs` (v1 input) has no coinbase fields at all, confirming the protection is structurally absent from the v1 path: [6](#0-5) 

---

### Impact Explanation

A caller — any unprivileged NEAR account or consumer contract — can invoke `verify_transaction_inclusion` with:
- `tx_id` set to the hash of a real internal Merkle node from a confirmed block
- `merkle_proof` set to the sibling path from that internal node up to the Merkle root
- `tx_index` set to the position of that internal node at its level

`compute_root_from_merkle_proof` will reconstruct the correct Merkle root and the function returns `true`, falsely asserting that a non-existent transaction is confirmed in that block. Any bridge, atomic-swap, or lending protocol that gates asset release on this return value can be drained by presenting a forged proof for a transaction that never occurred.

---

### Likelihood Explanation

- The function is `pub` with no access-control gate beyond the global pause; any NEAR account can call it.
- The 64-byte internal-node forgery technique is publicly documented (https://www.bitmex.com/blog/64-Byte-Transactions) and the contract's own docstring acknowledges the exact attack vector.
- Consumer contracts integrated before v0.5.0 — or written against the ABI without reading deprecation notices — will silently use the weaker path. Rust `#[deprecated]` is a compiler lint, not a runtime guard; it produces no on-chain enforcement.
- The attack requires only knowledge of any confirmed block's Merkle tree, which is public Bitcoin data.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or replace its body with `env::panic_str("use verify_transaction_inclusion_v2")` so that any caller receives an immediate, unambiguous runtime error. A Rust `#[deprecated]` attribute is invisible to NEAR callers and provides no protection. Consumer contracts must be migrated to `verify_transaction_inclusion_v2`, which enforces the coinbase-proof guard that closes the forgery path.

---

### Proof of Concept

1. Identify any confirmed mainchain block `B` with at least two transactions. Its Merkle tree has at least one internal node `N` at depth 1 whose 32-byte hash is known from public Bitcoin data.
2. Call `verify_transaction_inclusion` with:
   - `tx_id = hash(N)` — the internal node, not a real txid
   - `tx_block_blockhash = B`
   - `tx_index = 0` (position of `N`'s subtree at depth 1)
   - `merkle_proof = [sibling of N]` — one element, the sibling hash at depth 1
   - `confirmations = 1`
3. `compute_root_from_merkle_proof(hash(N), 0, [sibling])` computes `double_sha256(hash(N) || sibling)` which equals the Merkle root stored in `B`.
4. The function returns `true`.
5. A consumer contract that calls `verify_transaction_inclusion` to gate a cross-chain asset release now believes a fabricated transaction is confirmed, and releases funds to the attacker. [7](#0-6) [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
