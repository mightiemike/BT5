### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Proof Requirement — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to enforce a coinbase Merkle proof check that prevents the 64-byte transaction Merkle proof forgery attack. However, the original `verify_transaction_inclusion` (v1) is only marked `#[deprecated]` — a Rust compile-time annotation that has no on-chain enforcement. Any unprivileged NEAR caller can invoke v1 directly, bypassing the coinbase proof requirement entirely and obtaining a `true` result for a fabricated transaction inclusion claim.

---

### Finding Description

`verify_transaction_inclusion_v2` adds a mandatory coinbase proof check before delegating to v1: [1](#0-0) 

This check is the sole mitigation for the 64-byte transaction Merkle proof forgery vulnerability (referenced in the code's own doc comment at line 271). The v2 function then calls v1 internally: [2](#0-1) 

v1 itself is still a fully public, `#[pause]`-gated (but not access-controlled) on-chain method: [3](#0-2) 

The `#[deprecated]` attribute is a Rust compiler hint only. It emits a warning at compile time for Rust callers but imposes zero restriction on NEAR runtime calls. Any account can invoke `verify_transaction_inclusion` directly via a NEAR transaction, completely skipping the coinbase proof check that v2 enforces.

v1's own documentation explicitly acknowledges the consequence: [4](#0-3) 

The only checks v1 performs are: confirmations within `gc_threshold`, block in mainchain, non-empty proof, and Merkle root match — none of which prevent an internal-node forgery: [5](#0-4) 

---

### Impact Explanation

A NEAR caller can supply a `tx_id` that is the hash of an internal Merkle tree node (not a real transaction). Because v1 only checks that `compute_root_from_merkle_proof(tx_id, tx_index, proof) == merkle_root`, and a 64-byte internal node can be crafted to satisfy this equation, v1 returns `true` for a non-existent transaction. Any downstream contract or bridge that consumes this boolean result is deceived into treating a fabricated transaction as confirmed on Bitcoin.

---

### Likelihood Explanation

The entry path is fully unprivileged: any NEAR account can call `verify_transaction_inclusion` with no role, no stake, and no special permission. The 64-byte Merkle forgery technique is publicly documented (linked in the contract's own comments at line 271). The only barrier is constructing a valid-looking Merkle proof for an internal node, which is straightforward given knowledge of a real block's Merkle tree.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or gate it with an access-control role so it cannot be called by unprivileged accounts. Only `verify_transaction_inclusion_v2` should remain publicly reachable. Alternatively, add the coinbase proof check directly inside v1 and remove the delegation pattern.

---

### Proof of Concept

1. Attacker identifies any block `B` in the current mainchain whose Merkle tree has at least two transactions.
2. Attacker computes the hash of an internal Merkle node `N` (64 bytes of two child hashes concatenated and double-SHA256'd).
3. Attacker constructs a Merkle proof `P` such that `compute_root_from_merkle_proof(N, index, P) == B.merkle_root`. This is valid because `N` is a genuine internal node.
4. Attacker calls `verify_transaction_inclusion` directly (not v2) with `tx_id = N`, `tx_block_blockhash = B.hash`, `tx_index = index`, `merkle_proof = P`, `confirmations = 1`.
5. v1 passes all checks and returns `true`.
6. Any downstream bridge or contract that calls `verify_transaction_inclusion` to gate a withdrawal or state transition accepts the forged proof as a confirmed Bitcoin transaction.
