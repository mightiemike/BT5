### Title
Fast Withdrawal Fees Permanently Locked in `BaseWithdrawPool` with No Claim Mechanism — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal()` collects fees from fast-withdrawal submitters and accumulates them in the `fees[productId]` mapping. No function exists to claim or distribute these accumulated fees to any party. The fee tokens are permanently locked in the contract.

---

### Finding Description

In `BaseWithdrawPool.sol`, the `submitFastWithdrawal()` function computes a fee via `fastWithdrawalFeeAmount()` and records it:

```solidity
fees[productId] += fee;
``` [1](#0-0) 

The `fees` mapping is declared as a public state variable that only ever increases:

```solidity
mapping(uint32 => int128) public fees;
``` [2](#0-1) 

Across the entire contract, there is no function that:
- Decrements `fees[productId]`
- Transfers the accumulated fee tokens to a protocol treasury or any recipient
- Resets the fee counter after distribution

The only owner-callable token-moving function is `removeLiquidity()`:

```solidity
function removeLiquidity(uint32 productId, uint128 amount, address sendTo)
    external onlyOwner {
    handleWithdrawTransfer(getToken(productId), sendTo, amount);
}
``` [3](#0-2) 

`removeLiquidity` moves raw token balances but never touches the `fees` mapping. This means:

1. The `fees` mapping grows indefinitely and is never decremented — it is a dead-end accounting variable.
2. The ERC20 tokens representing collected fees sit in the contract with no dedicated, protocol-sanctioned distribution path.
3. Even if the owner calls `removeLiquidity` to extract tokens, the `fees` accounting is permanently desynchronized from the actual token balance, breaking any off-chain or on-chain invariant that relies on `fees[productId]` reflecting unclaimed fees.

The fee collection path is triggered by any unprivileged caller via `submitFastWithdrawal()`: [4](#0-3) 

---

### Impact Explanation

Fast withdrawal fees paid by users in real ERC20 tokens accumulate in `BaseWithdrawPool` with no mechanism to distribute them. The protocol permanently loses access to these fees through any proper accounting path. The `fees` mapping is a write-only counter — it is incremented on every fast withdrawal but never read for distribution or reset. This is a direct analog to the Ve.sol issue: assets accumulate in a contract that has no claim/distribution function.

---

### Likelihood Explanation

**High.** Every call to `submitFastWithdrawal()` by any user triggers fee accumulation. The issue is structural and affects all fast withdrawals from the moment of deployment. No special conditions or attacker setup are required — normal protocol usage is sufficient to lock fees permanently.

---

### Recommendation

Add a dedicated fee-claim function that transfers accumulated fees to a designated recipient and resets the counter:

```solidity
function claimFees(uint32 productId, address recipient) external onlyOwner {
    int128 amount = fees[productId];
    require(amount > 0, "No fees to claim");
    fees[productId] = 0;
    handleWithdrawTransfer(getToken(productId), recipient, uint128(amount));
}
```

This mirrors the fix applied to Ve.sol (`claimFees()`) and ensures accumulated fees are claimable and the accounting mapping stays consistent.

---

### Proof of Concept

1. User A calls `submitFastWithdrawal(idx, transaction, signatures)` with a valid fast-withdrawal transaction.
2. Inside the function, `fee = fastWithdrawalFeeAmount(...)` is computed (non-zero for any realistic withdrawal amount).
3. `fees[productId] += fee` is executed — fee tokens now sit in the contract, `fees[productId]` is non-zero.
4. Steps 1–3 repeat for every subsequent fast withdrawal. `fees[productId]` grows monotonically.
5. No function in `BaseWithdrawPool` or `WithdrawPool` decrements `fees[productId]` or transfers the fee tokens to any protocol address.
6. The owner can call `removeLiquidity` to extract tokens, but this does not reset `fees[productId]`, permanently desynchronizing the accounting. The fees have no proper claim path. [5](#0-4)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
