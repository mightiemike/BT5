### Title
Fork Retarget Uses Mainchain Ancestor Instead of Fork Lineage — (`contract/src/litecoin.rs`, `contract/src/lib.rs`)

---

### Summary

`get_header_by_height` is hardwired to the mainchain height index. When `get_next_work_required` calls it during a fork submission that crosses a retarget boundary, it reads the **mainchain** block at `first_block_height` instead of the fork's own ancestor at that height. An attacker who controls the fork's timestamps can make the contract compute a lower difficulty than the real Litecoin network would require, causing the contract to accept and store a fork with scrypt-valid but protocol-invalid headers.

---

### Finding Description

**Root cause — `store_fork_header` vs `get_header_by_height`:**

Fork blocks are stored only in `headers_pool`, never in `mainchain_height_to_header`:

```rust
// contract/src/lib.rs:665-667
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [1](#0-0) 

But `get_header_by_height` reads exclusively from `mainchain_height_to_header`:

```rust
// contract/src/lib.rs:677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

**Retarget path that triggers the bug:**

When a fork block lands exactly on a retarget boundary, `get_next_work_required` computes `first_block_height` and calls `get_header_by_height` to get the interval-start timestamp:

```rust
// contract/src/litecoin.rs:86-93
let first_block_height = prev_block_header.block_height - blocks_to_go_back;
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [3](#0-2) 

If the fork diverged before `first_block_height`, the fork has its own block at that height (with an attacker-chosen timestamp), but the contract ignores it and uses the mainchain block's timestamp instead.

**Exploit mechanics:**

Let `I = difficulty_adjustment_interval` (2016 for Litecoin). Let the next retarget be at height `R`.

1. Attacker builds a fork diverging at height `K < R - I`.
2. On the fork, the block at height `first_block_height = R - I - 1` (or `R - I` depending on the off-by-one path) is assigned timestamp `T_fork = T_main + Δ`, where `T_main` is the mainchain block's timestamp at that height and `Δ` is a positive offset (e.g., several days). Timestamps are walked up gradually across fork blocks to pass the MTP check.
3. When the fork's block at height `R` is submitted, the contract computes:
   - `actual_time_taken = T_prev_fork - T_main` (uses mainchain timestamp, not `T_fork`)
   - True value would be `T_prev_fork - T_fork = T_prev_fork - T_main - Δ` (smaller → harder)
4. The inflated `actual_time_taken` produces a lower `bits` target (easier difficulty), capped at 4× reduction.
5. The attacker mines the retarget block at this easier target (scrypt-valid, passes `U256::from_le_bytes(&pow_hash.0) <= target_from_bits(header.bits)`).
6. The contract stores the fork block with the wrong `bits`, accepting it as valid. [4](#0-3) 

**Reorg path to financial impact:**

Once the fork has enough blocks at artificially low difficulty, if its accumulated `chain_work` exceeds the mainchain's, `submit_block_header_inner` triggers `reorg_chain`, promoting the attacker's fork to the mainchain: [5](#0-4) 

A downstream bridge calling `verify_transaction_inclusion` against the now-promoted fork would treat attacker-controlled confirmations as final.

---

### Impact Explanation

The contract stores and potentially promotes a fork whose `bits` field was validated against the wrong ancestor timestamp. The real Litecoin network would reject this fork at the retarget block. Any bridge or application relying on the light client's mainchain as canonical Litecoin would accept fraudulent transaction proofs, enabling theft or loss of bridged funds.

---

### Likelihood Explanation

On Litecoin testnet, scrypt difficulty is low enough that an attacker with modest hardware can mine 2016+ blocks. The timestamp manipulation is bounded by `MAX_FUTURE_BLOCK_TIME_LOCAL` per block but can be accumulated across the entire fork interval, achieving close to the 4× difficulty reduction cap. No privileged access is required — `submit_blocks` is a public NEAR contract call reachable through the standard relayer path.

---

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` with a walk-back through `get_prev_header` starting from `prev_block_header`, traversing exactly `blocks_to_go_back` steps. This traversal uses `headers_pool` (which contains both mainchain and fork blocks) and therefore follows the fork's own lineage regardless of what the mainchain index holds at that height. [6](#0-5) 

---

### Proof of Concept

```
Setup (Litecoin testnet, I=2016, next retarget at height R=4032):

Mainchain:  [0] ... [2015] [2016] ... [4031]  ← mainchain block at 2016 has timestamp T_main
Fork:       [0] ... [2015] [f2016] ... [f4031] ← fork block at 2016 has timestamp T_main + 7 days

Submit fork block at height 4032 (retarget boundary):
  - Contract calls get_header_by_height(2016)
  - Returns mainchain block with timestamp T_main  ← BUG: should return f2016 with T_main+7days
  - actual_time_taken = T_f4031 - T_main          ← inflated by 7 days
  - Computed bits = lower difficulty (up to 4× easier)
  - Attacker mines block at this easier target, submits it
  - Contract accepts it (bits match computed expected_bits)
  - Fork accumulates chainwork → reorg → bridge accepts attacker's chain as canonical
```
