### Title
Bonus reward computed using current delegation shares for all historical segments, inflating payout when a position is expanded before claiming — (File: `x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

`computeSegmentBonus` always multiplies the **current** `pos.Delegation.Shares` by a historical `tokensPerShare` snapshot when computing bonus for past time segments. If a user adds to their position (increasing shares) without first claiming rewards, every historical segment since the last claim is retroactively re-priced at the larger share count, causing the user to drain more from the bonus pool than they are entitled to.

---

### Finding Description

In `computeSegmentBonus`, the bonus formula is:

```
tokens = pos.Delegation.Shares × tokensPerShare
bonus  = tokens × tier.BonusApy × durationSeconds / SecondsPerYear
``` [1](#0-0) 

`pos.Delegation.Shares` is the **live** delegation shares fetched at claim time via `getPositionState` → `getDelegation`. [2](#0-1) 

In `processEventsAndClaimBonus`, this function is called once per historical validator-event segment, each time passing a **snapshot** `evt.TokensPerShare` rate recorded at the event boundary: [3](#0-2) 

And again for the open trailing segment using the current rate: [4](#0-3) 

In both calls, `pos.Delegation.Shares` is never adjusted to reflect what the shares were **during** each segment — it is always the shares at claim time. The `LastBonusAccrual` checkpoint only records the timestamp of the last claim, not the share count at that time. [5](#0-4) 

If a user calls `MsgAddToPosition` (evidenced by `msg_server_add_to_position_test.go`) between two claims, the delegation shares grow. On the next claim, every historical segment since `LastBonusAccrual` is evaluated with the inflated share count, yielding a bonus larger than the position actually earned during those segments.

---

### Impact Explanation

The position owner receives bonus coins from `types.RewardsPoolName` in excess of what their historical stake entitles them to. [6](#0-5) 

The over-payment is proportional to `(sharesAfterAddition / sharesBeforeAddition - 1) × bonusApy × unclaimed_duration`. A user who stakes a small amount, waits a long time, then adds a large amount before claiming can extract a multiple of the legitimate bonus, draining the pool at the expense of other position holders whose future bonus claims may fail the `sufficientBonusPoolBalance` check. [7](#0-6) 

---

### Likelihood Explanation

The trigger is two ordinary, unprivileged transactions — `MsgAddToPosition` followed by a reward claim — with no privileged role required. A rational user who discovers the accounting gap is directly incentivized to delay claiming until after adding to their position. The longer the gap and the larger the addition, the greater the over-payment.

---

### Recommendation

Enforce a reward claim (flush `LastBonusAccrual` to the current block time and settle all pending bonus) as the **first step** inside `MsgAddToPosition`, before the new tokens are delegated and shares increase. This ensures that every historical segment is evaluated with the shares that were actually staked during it, matching the intent of the per-segment snapshot design.

---

### Proof of Concept

1. User creates a position with 100 shares at block time **T0**; `LastBonusAccrual = T0`.
2. A validator bond/unbond event is recorded at **T1**; user does **not** claim rewards.
3. User calls `MsgAddToPosition`, increasing shares to **1 000** at **T2**.
4. User calls the claim-rewards message at **T3**.
5. `processEventsAndClaimBonus` walks the segment `[T0, T1]` with `pos.Delegation.Shares = 1 000` and the snapshot `tokensPerShare` from T1, then the segment `[T1, T3]` with the current rate — both using 1 000 shares.
6. The legitimate bonus for `[T0, T2]` should have been computed on 100 shares; instead it is computed on 1 000 shares — a **10× over-payment** for that period — drawn directly from `types.RewardsPoolName`. [8](#0-7)

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
