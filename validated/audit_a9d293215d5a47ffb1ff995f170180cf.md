Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Full Allowlist Bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]` where `sender` is the pool's `msg.sender` — the immediate caller of `pool.swap`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. If a pool admin allowlists the router (the natural step to enable router-mediated swaps for curated users), every user who calls the router passes the check, completely neutralizing the allowlist.

## Finding Description
**Root cause — wrong actor checked:**

`SwapAllowlistExtension.beforeSwap` (line 37) checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool — the pool's own `msg.sender`.

**Call chain binding `sender` to the router:**

`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`:
```solidity
_beforeSwap(msg.sender, recipient, zeroForOne, ...);
```
`ExtensionCalling._beforeSwap` forwards that value verbatim as `sender` in the `abi.encodeCall` to the extension.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the pool's `msg.sender` the router contract, not the end-user (`msg.sender` of the router call). The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**The irreconcilable dilemma:**

| Admin configuration | Consequence |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot use `MetricOmmSimpleRouter` — the primary user-facing interface is broken for the curated pool |
| Router **allowlisted** | `allowedSwapper[pool][router] = true` → every user routing through the router passes the check; allowlist fully bypassed |

There is no mechanism in the extension to distinguish which end-user is behind a router call. `allowedSwapper[pool][router]` is a single boolean that opens the gate for all users simultaneously.

## Impact Explanation
A curated pool's swap allowlist is completely neutralized. Any unprivileged user can execute swaps on a pool the admin intended to restrict to specific counterparties by routing through `MetricOmmSimpleRouter`. This enables unauthorized traders to drain LP inventory at oracle prices on pools designed for specific market makers, violates KYC/compliance policies enforced via the allowlist, and circumvents any economic protection the admin encoded via the allowlist. The LP providers' principal is at risk because the pool's access-control invariant — "only allowlisted addresses may swap" — is broken for all router-mediated paths, which is the primary production swap path.

## Likelihood Explanation
Medium. The bypass requires the admin to allowlist the router address. This is a natural and expected configuration step: `MetricOmmSimpleRouter` is the primary user-facing swap interface, and an admin setting up a curated pool would naturally want their allowlisted users to be able to use it. The call `setAllowedToSwap(pool, router, true)` looks identical to allowlisting any other address — there is no warning that this collapses the entire allowlist. The protocol's own test infrastructure for `SwapAllowlistExtension` only exercises direct pool calls, never router-mediated calls, so the interaction is untested.

## Recommendation
The extension must check the actual end-user, not the pool's `msg.sender`. Concrete options:
1. **Extension-data forwarding**: The router encodes the real user's address in `extensionData`; the extension decodes and checks it. The router already forwards `extensionData` verbatim to the pool.
2. **Separate router allowlist**: Maintain `allowedRouter[pool][router]` and `allowedSwapper[pool][swapper]`; require both to be set, with the router responsible for passing the real swapper identity.
3. **`sender` vs `origin` split**: Pass both `msg.sender` (pool's caller) and a user-supplied identity field through `extensionData`, and verify the caller is an approved router before trusting the identity field.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  admin: setAllowedToSwap(pool, alice, true)   // alice is the intended curated user
  admin: setAllowedToSwap(pool, router, true)  // admin enables router for alice's convenience

Attack (Bob, not allowlisted):
  Bob calls router.exactInputSingle({
      pool:      pool,
      recipient: bob,
      tokenIn:   token0,
      amountIn:  X,
      ...
  })

Execution trace:
  router.exactInputSingle(...)                          // msg.sender = Bob
    → pool.swap(bob, zeroForOne, X, ...)               // pool.msg.sender = router
        → _beforeSwap(router, bob, ...)                // sender = router
            → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                allowedSwapper[pool][router] == true ✓ // check passes
        → swap executes, bob receives output tokens

Bob's swap succeeds on a pool that should have rejected him.
```
