### Title
No Cancellation Mechanism for Slow Mode Transactions Causes Permanent Fee Loss - (`core/contracts/Endpoint.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

Users who submit slow mode transactions pay a `SLOW_MODE_FEE` upfront in the quote token. If the queued transaction subsequently fails during execution, the fee is silently consumed with no refund path and no cancellation API. A removed refund comment in the code (`// try return funds now removed`) confirms this was an explicit design choice that leaves users with no recourse.

---

### Finding Description

When a user calls `submitSlowModeTransaction` for eligible transaction types (e.g., `WithdrawCollateral`, `LinkSigner`), `EndpointTx.submitSlowModeTransactionImpl` charges a fee via `chargeSlowModeFee` before enqueuing the transaction: [1](#0-0) 

The fee is transferred from the user to the `Endpoint` contract and tracked in `slowModeFees`. The transaction is then stored in the FIFO `slowModeTxs` mapping with a 3-day execution delay: [2](#0-1) 

When the queue is eventually drained, `_executeSlowModeTransaction` deletes the entry and calls `processSlowModeTransaction` inside a `try/catch`: [3](#0-2) 

If `processSlowModeTransaction` reverts for any reason, the catch block silently discards the failure. The comment at line 226 — `// try return funds now removed` — is a direct admission that a fee-refund path previously existed and was deliberately removed. No refund is issued, and the fee is permanently retained by the contract.

There is no `cancelSlowModeTransaction` function anywhere in `Endpoint.sol`, `EndpointTx.sol`, or `IEndpoint.sol`: [4](#0-3) 

The queue is strictly FIFO (`txUpTo` to `txCount`), so a user cannot skip or remove their own pending entry: [5](#0-4) 

---

### Impact Explanation

Any user whose slow mode transaction fails during execution permanently loses the `SLOW_MODE_FEE` paid in the quote token. The fee is a real, non-dust amount (set by `clearinghouse.getSlowModeFee()`). There is no on-chain path to reclaim it — no cancellation, no refund on failure, and no governance mechanism to return it to the original sender. [6](#0-5) 

---

### Likelihood Explanation

The 3-day `SLOW_MODE_TX_DELAY` between submission and execution creates a realistic window for a transaction to become invalid. Concrete failure scenarios include:

- A `WithdrawCollateral` slow mode transaction is submitted, but the subaccount's health deteriorates below maintenance margin during the 3-day window, causing `clearinghouse.withdrawCollateral` to revert.
- A product is delisted between submission and execution, causing the withdrawal to revert.
- A `LinkSigner` transaction is submitted but the subaccount is liquidated before execution, causing `requireSubaccount` to fail.

Any unprivileged user who relies on the slow mode queue for censorship resistance is exposed to this loss. [7](#0-6) 

---

### Recommendation

1. **Add a cancellation function** that allows the original `sender` of a slow mode transaction to cancel it after the `executableAt` timestamp has passed (or after a separate cancellation delay), refunding the `SLOW_MODE_FEE`.
2. **Restore fee refund on failure**: In the `catch` block of `_executeSlowModeTransaction`, refund `SLOW_MODE_FEE` to `txn.sender` instead of silently discarding it.

```solidity
// In _executeSlowModeTransaction catch block:
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // Refund the slow mode fee to the original sender
    _getQuote().safeTransfer(txn.sender, uint256(SLOW_MODE_FEE));
    slowModeFees -= SLOW_MODE_FEE;
}
```

---

### Proof of Concept

1. User calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload.
2. `EndpointTx.submitSlowModeTransactionImpl` charges `SLOW_MODE_FEE` from the user and enqueues the transaction with `executableAt = block.timestamp + 3 days`.
3. During the 3-day window, the user's subaccount health drops below maintenance margin (e.g., due to adverse price movement).
4. After 3 days, the user (or anyone) calls `executeSlowModeTransaction`.
5. `_executeSlowModeTransaction` dequeues the entry and calls `processSlowModeTransaction` inside `try/catch`.
6. `clearinghouse.withdrawCollateral` reverts because the subaccount is unhealthy.
7. The `catch` block silently swallows the revert. The transaction is gone from the queue.
8. The user's `SLOW_MODE_FEE` remains in the `Endpoint` contract with no refund path.
9. There is no `cancelSlowModeTransaction` to call before or after this point. [8](#0-7)

```text
    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
