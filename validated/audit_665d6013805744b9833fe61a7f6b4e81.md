### Title
Secp256k1/r1 4-byte Opcode Path Bypasses `ENABLE_SECP_OPS` Flag Guard — (File: `src/chia_dialect.rs`)

---

### Summary

In `ChiaDialect::op()`, secp256k1 and secp256r1 verify operators are reachable via **two distinct opcode encodings**: a 1-byte form (opcodes 64 and 65) and a 4-byte form (opcodes `0x13d61f00` and `0x1c3a8f00`). The 1-byte path correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte path dispatches directly to the same operator functions **with no flag check at all**, allowing an attacker to invoke secp operations unconditionally regardless of whether the hard-fork flag is active.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` handles 4-byte opcodes in a dedicated branch before the 1-byte dispatch table:

```rust
// 4-byte path — no flag check
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
``` [1](#0-0) 

The 1-byte dispatch table, by contrast, correctly guards the same operators:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

`ENABLE_SECP_OPS` is defined as a hard-fork activation flag:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

The 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` are the cost-formula-compatible encodings of the same secp operators. Before the hard fork, a program using these 4-byte forms would have been treated as an unknown operator (returning nil with a cost). After this code was introduced, they dispatch to the real secp functions — but without the `ENABLE_SECP_OPS` guard that the 1-byte forms require. This is the exact same class of bug as the original report: two authorization paths exist, only one is checked.

---

### Impact Explanation

A CLVM program submitted with the 4-byte secp opcode form will execute `op_secp256k1_verify` or `op_secp256r1_verify` even on a node that does **not** have `ENABLE_SECP_OPS` set (i.e., a pre-hard-fork node). On such a node:

- The 1-byte form (opcode 64/65) falls through to `unknown_operator`, returning nil in consensus mode.
- The 4-byte form (`0x13d61f00`/`0x1c3a8f00`) executes the real secp function, which **raises an error on signature failure** instead of returning nil.

This produces a **consensus divergence**: a program that passes on one node (unknown-op → nil → program continues) fails on another (secp verify → error → program aborts), depending solely on which opcode encoding the attacker chose. The corrupted result is the program's final evaluation outcome and its associated cost. [4](#0-3) 

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM program submitted to the network can use the 4-byte opcode encoding. The attacker only needs to know the two 4-byte opcode values (`0x13d61f00`, `0x1c3a8f00`), which are documented in the source code comment. No special privileges, keys, or social engineering are required. The trigger is a single crafted CLVM byte sequence.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode branch, mirroring the 1-byte branch:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both encoding paths are subject to the same activation guard, eliminating the inconsistency.

---

### Proof of Concept

1. Construct a CLVM program that uses the 4-byte opcode `0x13d61f00` (secp256k1_verify) with an intentionally invalid signature.
2. Run it via `run_program` with a `ChiaDialect` that does **not** have `ENABLE_SECP_OPS` set (pre-hard-fork node, e.g. `ClvmFlags::empty()`).
3. **Expected (correct) behavior**: the 4-byte opcode is treated as an unknown operator, returns nil, program continues.
4. **Actual behavior**: `op_secp256k1_verify` is called, the invalid signature causes `EvalErr::Secp256Failed`, the program aborts.
5. Run the same program on a node with `ENABLE_SECP_OPS` set — same abort. Run with the 1-byte opcode 64 without the flag — nil returned, program continues.

The divergence between the 4-byte path (no flag check → real secp execution) and the 1-byte path (flag check → unknown-op nil) is the concrete, reproducible root cause. [5](#0-4) [6](#0-5)
