### Title
`#[pause]` on `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` with No Exception Roles, While `submit_blocks` Advances Chain and Runs GC During Pause, Permanently Invalidating Pending Proofs - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are gated by `#[pause]` with no exception roles. Meanwhile, `submit_blocks` carries a `#[pause]` exception for `UnrestrictedSubmitBlocks` and internally calls `run_mainchain_gc`, which prunes old block headers from on-chain storage. During a pause, the chain tip advances and old blocks are garbage-collected, but no proof verification can occur. When the contract is unpaused, blocks that were within the confirmation window before the pause may have been GC'd, making their transaction proofs permanently unverifiable.

---

### Finding Description

Three public methods interact to create this vulnerability:

**`submit_blocks`** (line 167–198) carries `#[pause]` but the `near-plugins` `Pausable` macro expands this to allow accounts holding `Role::UnrestrictedSubmitBlocks` to bypass the pause check. A privileged relayer can therefore continue submitting Bitcoin block headers while the contract is paused. At line 181, `submit_blocks` calls `self.run_mainchain_gc(num_of_headers)` as an **internal Rust call** — the `#[pause]` attribute on the public `run_mainchain_gc` method (line 376) only intercepts external NEAR cross-contract calls, not internal method dispatch. GC therefore runs unconditionally inside every `submit_blocks` invocation, regardless of pause state. [1](#0-0) 

**`run_mainchain_gc`** (lines 377–416) removes block headers from `headers_pool` and `mainchain_height_to_header` once the stored chain exceeds `gc_threshold`. It advances `mainchain_initial_blockhash` forward, permanently deleting the pruned entries from NEAR storage. [2](#0-1) 

**`verify_transaction_inclusion`** (line 287) and **`verify_transaction_inclusion_v2`** (line 346) are both decorated with a bare `#[pause]` — no `except(roles(...))` clause. Unlike `submit_blocks` and `run_mainchain_gc`, there is no role that can bypass the pause check for these two functions. They are completely inaccessible to every NEAR account while the contract is paused. [3](#0-2) [4](#0-3) 

The asymmetry is the root cause: `submit_blocks` (with GC) can run during pause; `verify_transaction_inclusion*` cannot. The role definitions confirm no bypass exists for verification: [5](#0-4) 

---

### Impact Explanation

A downstream bridge or DeFi protocol submits a BTC transaction and waits for N confirmations before calling `verify_transaction_inclusion_v2`. If the contract is paused while that transaction's block is within the confirmation window:

1. The privileged relayer continues submitting new Bitcoin headers via `submit_blocks`.
2. Each batch triggers GC internally, advancing `mainchain_initial_blockhash` and deleting old `ExtendedHeader` entries from `headers_pool`.
3. The block containing the pending BTC transaction is pruned from storage.
4. When the contract unpauses, `verify_transaction_inclusion` panics at line 301 (`"block does not belong to the current main chain"`) or line 313 (`"cannot find requested transaction block"`) because the block no longer exists in storage.

The proof is permanently unverifiable. Any downstream protocol action gated on that proof (e.g., minting wrapped BTC, releasing collateral, settling a cross-chain swap) is permanently blocked with no recovery path. The longer the pause and the smaller the `gc_threshold`, the more transactions are affected. [6](#0-5) 

---

### Likelihood Explanation

The `UnrestrictedSubmitBlocks` role exists precisely so that chain synchronization continues during maintenance pauses — the design intent is that the chain keeps advancing. Bitcoin produces ~144 blocks per day. With a `gc_threshold` of 52,704 (one year, as recommended in the `init` docstring), a pause of even a few days is safe. However, with smaller `gc_threshold` values (e.g., 1,000 blocks ≈ 7 days, as used in tests), a pause of a week causes GC to sweep blocks that were pending verification. Any operator who pauses the contract for a security investigation while a bridge transaction is in-flight triggers this condition without any adversarial action required. [7](#0-6) 

---

### Recommendation

Two mitigations, in order of preference:

1. **Add an exception role to the verification functions** (analogous to `UnrestrictedSubmitBlocks`). Define a `Role::UnrestrictedVerify` and apply it:
   ```rust
   #[pause(except(roles(Role::UnrestrictedVerify)))]
   pub fn verify_transaction_inclusion_v2(...) -> bool { ... }
   ```
   This allows trusted downstream contracts or operators to continue verifying proofs during a pause.

2. **Implement a grace period after unpause** before GC resumes pruning blocks that were within the confirmation window at the time of the pause. This mirrors the Aave 3.1 approach cited in the reference report.

At minimum, the `#[pause]` on `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` should be treated with the same care as `submit_blocks` — if chain advancement is allowed during pause, proof verification must also be allowed, or GC must be suspended during pause to prevent the state divergence.

### Proof of Concept

```
1. Contract is initialized with gc_threshold = 1000.
2. Chain tip is at height 1100; mainchain_initial_blockhash is at height 100.
3. A BTC transaction is confirmed at height 1050. A downstream contract
   is waiting to call verify_transaction_inclusion_v2 with confirmations=6.
4. PauseManager pauses the contract.
5. UnrestrictedSubmitBlocks relayer calls submit_blocks([h1101..h1110]).
   - Internally, run_mainchain_gc(10) fires.
   - Blocks at heights 100–109 are removed from headers_pool and
     mainchain_height_to_header.
   - This repeats over many batches until height 1050 is pruned
     (after ~950 more blocks are submitted, i.e., ~6.6 days at 144/day).
6. PauseManager unpauses the contract.
7. Downstream contract calls verify_transaction_inclusion_v2 with
   tx_block_blockhash = hash_of_block_1050.
8. Line 355: headers_pool.get(&args.tx_block_blockhash) → None
   → env::panic_str("cannot find requested transaction block")
9. Proof is permanently unverifiable. Downstream protocol is permanently stuck.
``` [8](#0-7) [9](#0-8)
