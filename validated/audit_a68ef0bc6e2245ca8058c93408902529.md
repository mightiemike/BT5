### Title
SwapAllowlistExtension Gates the Router Address Instead of the Real User, Allowing Any Caller to Bypass the Swap Allowlist via the Public Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` as seen by the pool — i.e., whoever called `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the actual end-user. If the pool admin allowlists the router (which is required for allowlisted users to use the standard periphery interface), the check becomes `allowedSwapper[pool][router] == true`, and **every user** who calls the router passes the gate regardless of whether they are on the allowlist.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on that `sender` value: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap`: [4](#0-3) 

So the allowlist lookup becomes `allowedSwapper[pool][router]`. The pool admin faces an impossible choice:

- **Do not allowlist the router** → allowlisted users cannot use the standard periphery interface at all.
- **Allowlist the router** → the check is satisfied for every caller of the router, including addresses that are not on the allowlist, because the router is a permissionless public contract.

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same.

---

### Impact Explanation

Any user can swap on a pool that is intended to be restricted to a specific set of addresses by routing through `MetricOmmSimpleRouter`. The swap allowlist — the only on-chain mechanism for restricting swap access — is fully bypassed. Depending on the pool's purpose (e.g., institutional-only pricing, KYC-gated access, or pools with favorable spreads for specific counterparties), this can result in direct extraction of value by unauthorized parties or violation of the pool's intended access model. This matches the "admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" criterion.

---

### Likelihood Explanation

Likelihood is high. The router is the primary user-facing swap interface. Any pool admin who wants allowlisted users to be able to use the standard periphery must allowlist the router, which immediately opens the gate to all users. The attack requires no special privileges, no flash loans, and no unusual token behavior — only a standard router call.

---

### Recommendation

The extension should gate on the **economic actor**, not the intermediary. Two sound approaches:

1. **Check `recipient` instead of `sender`** — but `recipient` is user-supplied and can be set to any address, so this is also gameable.
2. **Require the router to forward the real user's address** — add an `extensionData`-encoded caller field that the router populates with `msg.sender`, and have the extension decode and verify it. The extension should also verify that `sender` is a trusted router before trusting the decoded field.
3. **Gate on `sender` only for direct pool calls; require a trusted-router flag for router calls** — the extension checks whether `sender` is a known trusted router; if so, it decodes the real user from `extensionData`; otherwise it checks `sender` directly.

The simplest safe fix is option 2: the router encodes `msg.sender` into `extensionData`, and the extension decodes it only when `sender` is a known trusted router address registered with the extension.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack (executed by bob, who is NOT on the allowlist):
  1. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: bob, ...})
  2. Router calls pool.swap(bob, zeroForOne, amount, limit, "", extensionData)
     → pool's msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. ExtensionCalling encodes sender=router and calls SwapAllowlistExtension.beforeSwap(router, ...)
  5. Extension checks: allowedSwapper[pool][router] == true  → PASSES
  6. bob's swap executes successfully on the restricted pool.

Result: bob, who is not on the allowlist, completes a swap on a pool
        that is supposed to be restricted to alice only.
```
