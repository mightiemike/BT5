### Title
`From<ResourceBoundsMapping> for ValidResourceBounds` silently downgrades `AllResources` V3 transactions to `L1Gas` when `l1_data_gas=0` and `l2_gas=0`, producing a wrong transaction hash - (File: `crates/apollo_rpc/src/v0_8/transaction.rs`)

---

### Summary

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in the RPC v0.8 module silently changes the resource-bounds variant from `AllResources` to `L1Gas` whenever `l1_data_gas` and `l2_gas` are both zero. Because `get_tip_resource_bounds_hash` hashes a different number of resource felts for each variant, the resulting transaction hash diverges from the hash the signer computed. Any RPC path that re-derives the hash from this converted representation — simulation, fee estimation, tracing, or blockifier re-execution — will produce an authoritative-looking but wrong value.

---

### Finding Description

**Root cause — silent variant downgrade**

`crates/apollo_rpc/src/v0_8/transaction.rs` lines 188-200:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← drops AllResources semantics
        } else {
            Self::AllResources(AllResourceBounds {
                l1_gas: value.l1_gas,
                l1_data_gas: value.l1_data_gas,
                l2_gas: value.l2_gas,
            })
        }
    }
}
```

A V3 transaction whose signer set `AllResources` bounds with `l1_data_gas = 0` and `l2_gas = 0` is silently re-classified as `L1Gas` the moment it passes through this conversion.

**Hash divergence — different number of resource felts**

`crates/starknet_api/src/transaction_hash.rs` lines 188-211:

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let mut resource_felts = vec![
        get_concat_resource(&resource_bounds.get_l1_bounds(), L1_GAS)?,
        get_concat_resource(&resource_bounds.get_l2_bounds(), L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts
        ValidResourceBounds::AllResources(all) => {
            vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?]    // 3 felts
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

| Variant | Elements hashed | Result |
|---|---|---|
| `L1Gas` | `tip, l1_gas, l2_gas` | H_l1 |
| `AllResources` (l1_data_gas=0, l2_gas=0) | `tip, l1_gas, l2_gas, l1_data_gas` | H_all ≠ H_l1 |

`poseidon_hash_many` is length-sensitive, so appending a zero-valued fourth element produces a strictly different digest. The signer computed H_all; after the conversion the sequencer computes H_l1.

**Divergent path**

The gateway accepts new transactions via `RpcDeployAccountTransactionV3` / `RpcInvokeTransactionV3`, which carry `AllResourceBounds` directly and never pass through `ResourceBoundsMapping`. The hash is therefore computed correctly at admission time. However, the RPC v0.8 module (`apollo_rpc/src/v0_8/transaction.rs`) represents stored and broadcast V3 transactions with `ResourceBoundsMapping`. Any downstream component that re-derives the hash from this representation — `starknet_simulateTransactions`, `starknet_estimateFee`, `starknet_traceTransaction`, or the blockifier re-execution path — will call `get_tip_resource_bounds_hash` on the downgraded `L1Gas` variant and obtain H_l1 instead of H_all.

---

### Impact Explanation

**High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

For any V3 transaction signed with `AllResources` bounds where `l1_data_gas = 0` and `l2_gas = 0`:

- `starknet_simulateTransactions` / `starknet_estimateFee` re-derive the hash from the downgraded `L1Gas` variant. Signature verification inside the simulated execution checks the account's `__validate__` against H_l1, not H_all. A correctly signed transaction will appear to fail validation; a transaction signed against H_l1 (which no honest signer would produce) would pass.
- `starknet_traceTransaction` and blockifier re-execution reconstruct the executable transaction from the stored RPC representation, recompute the hash, and may diverge from the committed receipt.

---

### Likelihood Explanation

V3 transactions with `l1_data_gas = 0` and `l2_gas = 0` are the natural pre-0.13.3 style of V3 transaction (the comment in the code itself acknowledges this: `// Pre 0.13.3. Only L1 gas. L2 bounds are signed but never used.`). Any wallet or SDK that submits a V3 transaction without explicitly setting data-gas or L2-gas bounds will produce exactly this pattern. The trigger requires only a standard, unprivileged RPC call (`starknet_simulateTransactions` or `starknet_estimateFee`) with such a transaction.

---

### Recommendation

Remove the zero-value heuristic from `From<ResourceBoundsMapping> for ValidResourceBounds`. The variant must be determined by the transaction version or an explicit tag, not by inspecting field values:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        // Always preserve AllResources for V3 transactions; never infer L1Gas from zero values.
        Self::AllResources(AllResourceBounds {
            l1_gas: value.l1_gas,
            l1_data_gas: value.l1_data_gas,
            l2_gas: value.l2_gas,
        })
    }
}
```

Alternatively, pass the transaction version alongside the mapping and select the variant based on version (≥ 0.13.3 → `AllResources`, < 0.13.3 → `L1Gas`).

---

### Proof of Concept

1. Construct a V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`.
2. Compute the canonical hash H_all via `get_tip_resource_bounds_hash` on `ValidResourceBounds::AllResources(...)` — this hashes 4 elements: `[tip, l1_gas_packed, l2_gas_packed(=0), l1_data_gas_packed(=0)]`.
3. Sign the transaction over H_all and submit it to the gateway. The gateway uses `AllResourceBounds` directly; admission succeeds and the stored hash is H_all.
4. Call `starknet_simulateTransactions` with the same transaction body. The RPC v0.8 layer deserializes `resource_bounds` as `ResourceBoundsMapping`, invokes `From<ResourceBoundsMapping> for ValidResourceBounds`, and — because `l2_gas.is_zero() && l1_data_gas.is_zero()` — produces `ValidResourceBounds::L1Gas(X)`.
5. `get_tip_resource_bounds_hash` now hashes only 3 elements: `[tip, l1_gas_packed, l2_gas_packed(=0)]`, yielding H_l1 ≠ H_all.
6. The simulation's `__validate__` call verifies the account signature against H_l1; the signature (over H_all) is rejected, and the simulation returns a spurious validation failure for a legitimately signed transaction.

    tip: &Tip,

    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.

        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,

        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
