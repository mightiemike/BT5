### Title
Stale `lastLiquidationFees` Cross-Subaccount Contamination Causes Incorrect Insurance Accounting During Socialization â€” (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a single global `int128` in `ClearinghouseStorage` with no per-subaccount scoping. Every call to `_handleLiquidationPayment` overwrites it. When `_finalizeSubaccount` is called for subaccount B, it reads the current `lastLiquidationFees`, which may have been set by a completely unrelated liquidation of subaccount A. When the insurance fund is insufficient to cover the full deficit (i.e., socialization actually fires), this produces a concrete, measurable accounting error: the wrong amount of insurance is applied before socialization, causing participants to absorb more loss than they should while the insurance fund retains a surplus.

---

### Finding Description

`lastLiquidationFees` is declared as a single protocol-wide slot: [1](#0-0) 

Every call to `_handleLiquidationPayment` unconditionally overwrites it with the fees from the most recent liquidation step, regardless of which subaccount is being liquidated: [2](#0-1) 

When `_finalizeSubaccount` is triggered (via `productId == type(uint32).max`), it reads this global value to compute the insurance available for socialization: [3](#0-2) 

And adds it back at the end: [4](#0-3) 

The design intent (per the inline comment at lines 581â€“585) is that the fees from the **last liquidation step of the subaccount being finalized** are withheld from socialization to break a circular dependency: those fees were just added to `insurance`, and using them immediately in socialization could falsely make the subaccount appear coverable, only for the add-back to re-expose the deficit.

The bug: if any other liquidation fires between subaccount B's last liquidation step and its finalization call, `lastLiquidationFees` is overwritten with fees from that unrelated liquidation (call it `X_A`). Finalization of B then uses `X_A` instead of `X_B`.

---

### Impact Explanation

The add-back at line 410 uses the same stale value, so in the **non-socialization case** (insurance fully covers the deficit), the final stored `insurance` is identical regardless of which `lastLiquidationFees` value is used â€” the subtract and add-back cancel out.

The error only materializes when **socialization actually fires** â€” i.e., `v.insurance` after the subtraction is insufficient to cover the full perp `vQuoteBalance` deficit. In that case `perpEngine.socializeSubaccount` consumes all of `v.insurance` and returns 0: [5](#0-4) 

Concrete arithmetic (stale `X_A > X_B`, deficit = 1200, `insurance` = 1000):

| | Correct (`X_B = 50`) | Stale (`X_A = 100`) |
|---|---|---|
| Available for socialization | 950 | 900 |
| Amount socialized (spread to participants) | 250 | 300 |
| `v.insurance` after socialization | 0 | 0 |
| Add-back | +50 | +100 |
| Final stored `insurance` | 50 | 100 |

Result: 50 units of loss that should have been covered by insurance are instead spread to all open-interest holders via `cumulativeFundingLongX18` / `cumulativeFundingShortX18` adjustments, while the insurance fund retains 50 units it should have spent. This is a direct, measurable transfer of value from market participants to the insurance fund.

---

### Likelihood Explanation

The attack path is straightforward and requires no special privileges:

1. The sequencer processes `LiquidateSubaccount` transactions in submission order.
2. A liquidator submits: (a) a normal liquidation of subaccount A's last position (setting `lastLiquidationFees = X_A`), then (b) a finalization (`productId = type(uint32).max`) of subaccount B.
3. Both transactions are valid, signed, and pass all existing guards.
4. No admin access, no governance capture, no key compromise required.

The condition that makes the error observable (socialization fires) is precisely the scenario where the insurance fund is under stress â€” the most operationally likely time for multiple concurrent liquidations. [6](#0-5) 

---

### Recommendation

Scope `lastLiquidationFees` to the subaccount being finalized. The simplest fix is to reset it to zero at the start of `_finalizeSubaccount` (or pass the correct per-subaccount fee as a parameter). Alternatively, track `lastLiquidationFees` as a `mapping(bytes32 => int128)` keyed by subaccount, and clear the entry after finalization.

---

//   v.insurance = 1150e18 - 50e18 = 1100e18

//     returns 0
//   v.insurance += 50e18 => insurance = 50e18
