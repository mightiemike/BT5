### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Drain from DirectDepositV1 Accounts — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(...)` without checking its return value. The rest of the codebase consistently uses `ERC20Helper.safeTransferFrom`, which checks both the call success flag and the decoded boolean return. This single function bypasses that pattern. If the USDC token returns `false` instead of reverting on a failed transfer, the function continues execution, withdraws usdcE from the victim DDA, and delivers it to the caller — with no USDC paid.

---

### Finding Description

`ContractOwner.replaceUsdcEWithUsdc` is an `external` function with no `onlyOwner` guard — its only access control is a chain-ID check (`require(block.chainid == 57073)`). Its purpose is a 1-for-1 swap: the caller supplies USDC and receives usdcE from a target `DirectDepositV1` (DDA) account.

The swap sequence is:

1. Read `balance = usdcE.balanceOf(directDepositV1)`.
2. Pull USDC from `msg.sender` into the DDA: `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)`.
3. Withdraw usdcE from the DDA to `ContractOwner`: `DirectDepositV1(directDepositV1).withdraw(usdcE)`.
4. Forward usdcE to `msg.sender`: `IERC20Base(usdcE).safeTransfer(msg.sender, balance)`.

Step 2 uses a raw `.transferFrom()` call whose boolean return value is silently discarded. [1](#0-0) 

Every other transfer in the codebase goes through `ERC20Helper.safeTransferFrom`, which low-level-calls the token and `require`s `success && (data.length == 0 || abi.decode(data, (bool)))`. [2](#0-1) 

`replaceUsdcEWithUsdc` imports `ERC20Helper` via `using ERC20Helper for IERC20Base` (inherited through the file's imports) but calls the raw interface method instead. [3](#0-2) 

If the USDC token at the hardcoded address returns `false` on a failed transfer rather than reverting, steps 3 and 4 still execute: the DDA loses its usdcE and the caller receives it without paying.

---

### Impact Explanation

**Asset delta:** An attacker drains the full usdcE balance of any DDA (`DirectDepositV1`) that holds usdcE, paying nothing. The DDA's usdcE is transferred to `ContractOwner` (step 3) and then forwarded to the attacker (step 4). The DDA subaccount's collateral is permanently reduced by the stolen amount, corrupting the on-chain balance that the sequencer uses for settlement. [4](#0-3) 

**Broken invariant:** The function's atomicity guarantee — "USDC in ↔ usdcE out" — is violated. The usdcE leg executes unconditionally while the USDC leg may silently no-op.

---

### Likelihood Explanation

The function is callable by any unprivileged address on Ink mainnet (chainid 57073) — no role, signature, or sequencer involvement is required. [5](#0-4) 

Exploitability depends on whether the specific USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink Chain returns `false` or reverts on failure. Circle's canonical USDC implementations revert, which would prevent exploitation with that exact token. However:

- The code pattern is unconditionally broken for any non-reverting ERC20 at that slot.
- If the token address is ever updated or the contract is reused in a context with a different token, the vulnerability becomes immediately exploitable.
- The inconsistency with every other transfer site in the codebase (`safeTransfer`/`safeTransferFrom` used everywhere else) indicates this is an unintentional omission rather than a deliberate design choice.

---

### Recommendation

Replace the raw call with the project's own safe wrapper:

```solidity
// Before (vulnerable):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`safeTransferFrom` is already available via `using ERC20Helper for IERC20Base` and is used consistently everywhere else in the codebase. [2](#0-1) 

---

### Proof of Concept

1. A DDA for `subaccount` holds `N` usdcE tokens.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` on Ink mainnet.
3. `balance = N` is read from the DDA.
4. `usdc.transferFrom(attacker, dda, N)` is called. If USDC returns `false` (no revert), execution continues.
5. `DirectDepositV1(dda).withdraw(usdcE)` transfers `N` usdcE from the DDA to `ContractOwner`.
6. `usdcE.safeTransfer(attacker, N)` transfers `N` usdcE to the attacker.
7. Attacker holds `N` usdcE; DDA holds 0 usdcE; attacker paid 0 USDC. [6](#0-5) [7](#0-6)

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
