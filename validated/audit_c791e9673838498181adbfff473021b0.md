### Title
`DirectDepositV1.creditDeposit()` Ignores Native ETH While `isDirectDepositV1Ready()` Counts It, Causing Permanent Fund Lock — (File: `core/contracts/DirectDepositV1.sol`, `core/contracts/ContractOwner.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` only iterates over ERC20 token balances and never deposits native ETH. However, `ContractOwner.isDirectDepositV1Ready()` explicitly adds `recipient.balance` (native ETH) to the `wrappedNative` product balance when computing deposit readiness. If native ETH is present in the DDA contract — via the constructor's silent-failure path or a `selfdestruct` force-send — the readiness check returns `true` but `creditDeposit()` deposits nothing, permanently locking the ETH in the DDA.

---

### Finding Description

`ContractOwner.isDirectDepositV1Ready()` computes the effective balance for the `wrappedNative` product as:

```solidity
uint256 balance = token.balanceOf(recipient);
if (tokenAddr == wrappedNative) {
    balance += recipient.balance;   // native ETH counted here
}
``` [1](#0-0) 

`DirectDepositV1.creditDeposit()` only deposits ERC20 balances and never touches `address(this).balance`:

```solidity
uint256 balance = token.balanceOf(address(this));
if (balance != 0) {
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(...);
}
``` [2](#0-1) 

Native ETH can accumulate in the DDA through two concrete paths:

**Path 1 — Constructor silent failure.** The DDA is deployed via CREATE2 with a fixed salt (`bytes32(uint256(1))`), making its address deterministic. If the address is pre-funded with ETH before deployment, the constructor attempts to wrap it but silently swallows failure:

```solidity
(bool success, ) = wrappedNative.call{value: balance}("");
if (!success) {
    emit NativeTokenTransferFailed(balance);   // no revert, ETH stays
}
``` [3](#0-2) 

The comment acknowledges the contradiction: "shouldn't revert even if the transfer fails, otherwise the funds will be stuck in the DDA forever" — yet a failed wrap leaves the ETH stuck forever regardless.

**Path 2 — `selfdestruct` force-send.** Any caller can force ETH into the DDA by self-destructing a contract targeting the DDA address. This bypasses `receive()` entirely, leaving raw ETH that `creditDeposit()` will never process. [4](#0-3) 

---

### Impact Explanation

Once native ETH is stuck in the DDA:

1. `isDirectDepositV1Ready()` returns `true` (it counts the native ETH balance).
2. The off-chain system calls `creditDeposit()`.
3. `creditDeposit()` finds zero ERC20 balance for `wrappedNative` and deposits nothing.
4. The user's subaccount is never credited.
5. The ETH is permanently locked in the DDA until the owner manually calls `withdrawNative()`.

The user's subaccount balance is the exact corrupted state: it is not incremented despite ETH being present in the DDA and the readiness check reporting the deposit as processable. [5](#0-4) 

---

### Likelihood Explanation

- The DDA address is deterministic (CREATE2, fixed salt `bytes32(uint256(1))`), so pre-funding before deployment is straightforward for any observer watching the mempool for `createDirectDepositV1` calls.
- `selfdestruct` force-sends require only a trivial contract and a small ETH amount.
- Both paths are reachable by an unprivileged external caller with no special permissions. [6](#0-5) 

---

### Recommendation

1. **In `creditDeposit()`**: Add a native ETH check alongside the ERC20 loop. If `address(this).balance > 0` and `wrappedNative` is a registered product token, wrap the ETH first (call `wrappedNative.call{value: address(this).balance}("")`) before iterating over ERC20 balances.

2. **In the constructor**: If `wrappedNative.call` fails, the ETH is already stuck. Either revert (preventing deployment with stuck ETH) or immediately emit and record the amount so `creditDeposit()` can handle it.

3. **In `isDirectDepositV1Ready()`**: Only count `recipient.balance` if the DDA's `receive()` path is guaranteed to wrap it — i.e., do not count native ETH that cannot be deposited by `creditDeposit()` in its current form.

---

### Proof of Concept

1. Compute the deterministic DDA address for a target `subaccount` using CREATE2 with `salt = bytes32(uint256(1))`.
2. Send 1 ETH to that address before `createDirectDepositV1` is called (or use `selfdestruct` after deployment).
3. Call `ContractOwner.isDirectDepositV1Ready(ddaAddress, false)` → returns `true` (native ETH counted).
4. Call `ContractOwner.creditDepositV1(subaccount)` → `creditDeposit()` runs, finds zero ERC20 balance, deposits nothing.
5. The 1 ETH remains in the DDA; the subaccount balance is unchanged.
6. Only `withdrawNative()` (owner-only) can recover the ETH, and the subaccount is never credited. [7](#0-6) [8](#0-7)
