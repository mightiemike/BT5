### Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Mid Price, Systematically Undercounting `metricT0` and Weakening the Token-1-Outflow Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the **arithmetic mean** of bid and ask. The pool itself — and every other component that needs a mid price — uses the **geometric mean** (`SwapMath.midAndSpreadFeeX64FromBidAsk`). Because the arithmetic mean is always ≥ the geometric mean (AM-GM inequality), the stop-loss extension permanently uses a mid price that is too high. This causes `metricT0` (value per share in token-0 terms) to be systematically underestimated, which sets the high-watermark lower than reality and raises the drawdown floor that must be breached before the guard fires. The net effect is that the stop-loss protecting against token-1 outflow (`zeroForOne == true` swaps) is less sensitive than the pool admin configured, allowing LPs to lose more token-1 than the declared drawdown threshold.

---

### Finding Description

**Root cause — line 218 of `OracleValueStopLossExtension.sol`:**

```solidity
// OracleValueStopLossExtension._afterSwapOracleStopLoss
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;   // arithmetic mean
```

**Correct formula used everywhere else in the protocol — `SwapMath.midAndSpreadFeeX64FromBidAsk` line 70:**

```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);   // geometric mean
```

`PriceVelocityGuardExtension.beforeSwap` (line 48) calls `SwapMath.midAndSpreadFeeX64FromBidAsk` and therefore uses the geometric mean. `MetricOmmPool.swap` (line 243) does the same. Only `OracleValueStopLossExtension` diverges.

**How the wrong mid price corrupts the metrics:**

The per-share metric in token-0 terms is:

```
metricT0 = t0/shares  +  (t1 × 2^64 / midPriceX64) / shares
```

Because `midPriceX64_arithmetic ≥ midPriceX64_geometric`, the term `t1 × 2^64 / midPriceX64` is **smaller** with the arithmetic mean, so `metricT0` is **underestimated**.

Symmetrically, `metricT1 = (t0 × midPriceX64 / 2^64) / shares + t1/shares` is **overestimated**.

**How the wrong metric corrupts the watermark and guard:**

`_applyWatermark` ratchets the high-watermark up to the live metric value:

```solidity
if (metric >= hwm) return (metric, false);   // ratchet up
breached = metric < (hwm * floorMultiplier) / E6;
```

Because `metricT0` is always underestimated, `hwm0` is set to a value lower than the true per-share value. The drawdown floor `hwm0 × floorMultiplier / E6` is therefore also lower. The guard fires only when the live metric falls below this floor, so a larger real-world value drop is required before the stop-loss triggers for `zeroForOne == true` swaps (token-1 outflow).

**Quantified error:**

For a spread `s` (ask = bid × (1 + s)):

```
AM  = bid × (1 + s/2)
GM  = bid × √(1 + s)  ≈  bid × (1 + s/2 − s²/8)
AM − GM  ≈  bid × s²/8
```

| Spread | AM overestimates GM by | metricT0 underestimated by | Extra drain allowed (5% drawdown) |
|--------|------------------------|----------------------------|-----------------------------------|
| 1 %    | 0.001 %                | 0.001 %                    | negligible                        |
| 5 %    | 0.031 %                | 0.031 %                    | ~0.6 % relative                  |
| 10 %   | 0.12 %                 | 0.12 %                     | ~2.4 % relative                  |
| 20 %   | 0.46 %                 | 0.46 %                     | ~9 % relative                    |

For pools with wide oracle spreads (RWA pairs, illiquid assets), the guard is materially weaker than configured.

---

### Impact Explanation

The `OracleValueStopLossExtension` is the primary on-chain mechanism protecting LP principal from value drain between oracle updates. Because `metricT0` is systematically underestimated:

1. The high-watermark for token-0-denominated value is set below the true per-share value.
2. The drawdown floor is proportionally lower.
3. Swaps that drain token-1 from the pool (zeroForOne = true) can proceed past the configured drawdown threshold without triggering the stop-loss revert.
4. LPs suffer a larger-than-configured loss of token-1 principal before the guard fires.

This is a direct loss of LP assets. For pools with spreads ≥ 5 % (common for RWA or low-liquidity pairs), the effective protection is materially weaker than the admin intended.

---

### Likelihood Explanation

The miscalibration is present on **every swap** that touches a pool with this extension configured and a non-zero spread. No special attacker action is required; the error is structural. Any swap that moves the pool through bins will underestimate `metricT0` and set a lower watermark, permanently weakening the guard for the lifetime of the pool.

---

### Recommendation

Replace the arithmetic mean with the same geometric-mean helper used by the pool and `PriceVelocityGuardExtension`:

```solidity
// Before (line 218):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After:
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This ensures the stop-loss extension uses the same price reference as the pool's swap engine, eliminating the systematic underestimation of `metricT0`.

### Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension`, `drawdownE6 = 50_000` (5 %), and an oracle with bid = 1e18, ask = 1.1e18 (10 % spread).
2. Execute a swap (`zeroForOne = true`) to set the initial watermark.
   - Extension computes `midPriceX64 = (bid + ask) / 2 = 1.05e18`.
   - Pool computes `midPriceX64 = sqrt(bid × ask) ≈ 1.0488e18`.
   - `metricT0` is underestimated by ≈ 0.12 % relative to the correct value.
   - `hwm0` is set to this underestimated value.
3. Drain token-1 via further `zeroForOne = true` swaps until `metricT0` falls 5 % below `hwm0`.
   - Because `hwm0` was set 0.12 % too low, the actual per-share value has fallen 5.12 % before the guard fires.
4. Observe that the stop-loss reverts only after the true drawdown exceeds the configured 5 % threshold by the AM-GM error margin, confirming LPs lost more than the declared maximum drawdown.
