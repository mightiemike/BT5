### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Forgery Protection - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR contract method despite being deprecated. `verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase proof. Because v1 remains reachable by any unprivileged NEAR caller, the coinbase proof check can be bypassed entirely, allowing an attacker to obtain a `true` verification result for a fabricated transaction ID derived from an internal Merkle tree node.

### Finding Description

`verify_transaction_inclusion_v2` enforces an additional coinbase Merkle proof check before delegating to v1:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);

#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [1](#0-0) 

However, `verify_transaction_inclusion` (v1) is still decorated only with `#[pause]` — not `#[private]` — meaning any NEAR account can call it directly on-chain:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [2](#0-1) 

The Rust `#[deprecated]` attribute only emits compiler warnings for callers within the same Rust codebase. It has no effect on external NEAR transaction callers. The function is fully reachable at the protocol level.

The v1 function itself acknowledges the forgery risk in its own doc comment:

> This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. [3](#0-2) 

The Merkle proof verification in v1 is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [4](#0-3) 

`compute_root_from_merkle_proof` in `merkle-tools` performs no structural validation — it simply hashes pairs up the tree: [5](#0-4) 

An attacker can pick any 64-byte internal node from a real mainchain block's Merkle tree, treat it as a `tx_id`, supply a valid Merkle proof path for that node, and the function returns `true`.

### Impact Explanation

**Impact: Medium**

Any downstream NEAR contract or off-chain consumer that calls `verify_transaction_inclusion` (v1) and trusts its boolean result can be deceived into accepting a fabricated Bitcoin transaction proof. The attacker does not need to forge a block — they only need a real mainchain block and knowledge of its Merkle tree structure to construct a valid-looking proof for a non-existent transaction. This corrupts the core SPV guarantee the contract is designed to provide.

### Likelihood Explanation

**Likelihood: High**

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` directly. The required inputs (a mainchain block hash, a crafted `tx_id` from an internal node, and a Merkle proof path) are all derivable from public Bitcoin blockchain data. The bypass is structural — it requires no key compromise, social engineering, or special configuration.

### Recommendation

Mark `verify_transaction_inclusion` with `#[private]` or remove it entirely from the public ABI. If backward compatibility is required, gate it behind an access-control role (e.g., `Role::DAO`) so unprivileged callers cannot invoke it. All external callers should be directed exclusively to `verify_transaction_inclusion_v2`.

### Proof of Concept

1. Identify any mainchain block `B` stored in the contract (obtainable via `get_block_hash_by_height`).
2. Obtain block `B`'s full transaction list from a Bitcoin node.
3. Compute the Merkle tree for `B`. Select any internal node `N` at depth `d` (a 64-byte value).
4. Construct a Merkle proof path from `N` up to the Merkle root, treating `N` as if it were a leaf transaction at position `p`.
5. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash)
   - `tx_block_blockhash = B`
   - `tx_index = p`
   - `merkle_proof = <path from N to root>`
   - `confirmations = 1`
6. The contract returns `true` for a transaction that does not exist on Bitcoin.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
