### Title
`WithdrawCollateralV2` Allows User-Controlled Zero Fee, Bypassing Protocol Withdrawal Fee Enforcement — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

`WithdrawCollateralV2` introduces a user-controlled `feeX18` field in `SignedWithdrawCollateralV2` that is covered by the user's EIP-712 signature. The contract only enforces an upper bound (`feeX18 <= currentFeeX18`) with no minimum floor. A user can sign a `WithdrawCollateralV2` transaction with `feeX18 = 0`, which the sequencer cannot alter without invalidating the signature, and the contract will charge zero withdrawal fees — completely bypassing the fee mechanism that `WithdrawCollateral` (V1) enforces unconditionally at the contract level.

---

### Finding Description

`WithdrawCollateral` (V1) always charges the full configured fee at the contract level:

```solidity
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,
    signedTx.tx.productId
);
``` [1](#0-0) 

`WithdrawCollateralV2` instead charges the user-supplied `signedTx.feeX18`, with only an upper-bound check:

```solidity
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
chargeFee(signedTx.tx.sender, signedTx.feeX18, signedTx.tx.productId);
``` [2](#0-1) 

The `feeX18` field lives in the outer `SignedWithdrawCollateralV2` wrapper and is included in the bytes passed to `validateSignedTx` as `transaction`, meaning it is covered by the user's compact signature:

```solidity
struct SignedWithdrawCollateralV2 {
    WithdrawCollateralV2 tx;
    CompactSignature signature;
    int128 feeX18;          // user-signed, sequencer cannot alter
}
``` [3](#0-2) 

Because `feeX18` is signed by the user, the sequencer cannot change it to a non-zero value without invalidating the signature. The contract accepts `feeX18 = 0` because `0 >= 0` and `0 <= currentFeeX18` both pass. The result is that `chargeFee` is called with `fee = 0`, and `sequencerFee[productId]` receives nothing. [4](#0-3) 

`WithdrawCollateralV2` is not available in the slow-mode path (`processSlowModeTransactionImpl`), so it is exclusively processed through the sequencer's fast-mode batch submission (`submitTransactionsChecked` / `processTransactionImpl`). [5](#0-4) 

---

### Impact Explanation

Any user who submits a `WithdrawCollateralV2` transaction with `feeX18 = 0` to the sequencer pays zero withdrawal fees. The protocol's `sequencerFee[productId]` accumulator receives nothing for that withdrawal, and the fee is never collected. Since the user signs `feeX18 = 0`, the sequencer cannot substitute a non-zero fee without breaking the signature. The broken invariant is: **all collateral withdrawals must pay at least `withdrawFeeX18` to the protocol**. V1 enforces this; V2 does not. [6](#0-5) 

---

### Likelihood Explanation

The attack requires the sequencer to include the transaction. The sequencer is a trusted entity (`require(msg.sender == sequencer)`). However, the sequencer's role is to relay valid signed user transactions — it cannot modify `feeX18` without invalidating the user's signature. If the sequencer's inclusion policy is "include any transaction with a valid signature," a user can reliably obtain zero-fee withdrawals by always using V2 with `feeX18 = 0`. The contract provides no on-chain backstop. Likelihood is **medium**: it depends on whether the sequencer enforces a minimum fee off-chain, but the contract itself provides no guarantee. [7](#0-6) 

---

### Recommendation

Enforce the minimum fee at the contract level in the `WithdrawCollateralV2` handler, mirroring V1 behavior:

```solidity
require(signedTx.feeX18 >= currentFeeX18, "Fee below minimum");
```

Or, if V2 is intended to support a fee-market model where users bid fees, enforce a protocol-defined floor (e.g., a separate `minWithdrawFeeX18` config field) rather than allowing `feeX18 = 0`. [8](#0-7) 

---

### Proof of Concept

1. User constructs a `WithdrawCollateralV2` struct with `feeX18 = 0` and signs it with their EIP-712 key.
2. User submits the signed transaction to the sequencer off-chain.
3. Sequencer includes it in a batch call to `submitTransactionsChecked`.
4. `processTransactionImpl` decodes the transaction, calls `validateSignedTx` (passes — signature is valid over `feeX18 = 0`), then checks `require(0 >= 0)` and `require(0 <= currentFeeX18)` — both pass.
5. `chargeFee(sender, 0, productId)` is called: `spotEngine.updateBalance(productId, sender, 0)` and `sequencerFee[productId] += 0` — no fee collected.
6. `clearinghouse.withdrawCollateral` executes the full withdrawal amount with zero fee deducted. [6](#0-5) [9](#0-8)
