### Title
Spot Liability Liquidation Reverts on Bad Debt, Liquidator Receives No Fees — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

When a subaccount's negative quote balance exceeds the insurance fund (`quoteBalance.amount + insurance < 0`), the regular spot-liability liquidation path in `_assertLiquidationAmount` computes `maxLiquidatable = 0` and reverts. The only remaining path — finalization via `_finalizeSubaccount` — socializes the bad debt but pays the liquidator **zero fees**. This is a direct analog to the Aloe M-3 bug: a liquidator cannot be compensated for clearing bad-debt spot positions, removing all economic incentive to do so.

---

### Finding Description

In `liquidateSubaccountImpl`, when a liquidator submits `txn.amount < 0` to liquidate a spot liability, the call flow is:

1. `_assertCanLiquidateLiability` — passes (all spot balances ≤ 0, no perp positions outside spread).
2. `_assertLiquidationAmount` — **reverts** when bad debt exceeds insurance.

The revert originates in the spot-liability branch of `_assertLiquidationAmount`:

```solidity
// ClearinghouseLiq.sol lines 167–180
(int128 liquidationPrice, ) = getLiqPriceX18(txn.productId, txn.amount);
ISpotEngine.Balance memory quoteBalance = spotEngine.getBalance(
    QUOTE_PRODUCT_ID,
    txn.liquidatee
);
int128 maxBuy = (quoteBalance.amount + insurance).div(liquidationPrice);
maxBuy = MathHelper.max(maxBuy + 1, 0);
maxLiquidatable = MathHelper.max(spotAmount, -maxBuy);
```

When `quoteBalance.amount + insurance < 0` (bad debt exceeds insurance), `(quoteBalance.amount + insurance).div(liquidationPrice)` is a negative fixed-point value. Adding the raw `+1` epsilon and clamping to zero yields `maxBuy = 0`. Then:

```solidity
maxLiquidatable = MathHelper.max(spotAmount, -0) = 0   // spotAmount < 0
```

The final guard:

```solidity
// lines 189–193
} else {
    require(maxLiquidatable <= txn.amount, ERR_NOT_LIQUIDATABLE_AMT);
}
```

requires `0 <= txn.amount`, but `txn.amount < 0` — so the call **reverts**. [1](#0-0) [2](#0-1) 

The only remaining path is finalization (`txn.productId == type(uint32).max`), handled by `_finalizeSubaccount`. When `canLiquidateMore` is false (i.e., `quoteBalance.amount + insurance ≤ 0`), it skips the spot-balance-zero check and calls `spotEngine.socializeSubaccount` to distribute the bad debt across all depositors. [3](#0-2) 

Critically, `_finalizeSubaccount` never calls `_handleLiquidationPayment`, so no `liquidationFees` are computed or credited to the liquidator. The liquidator's quote balance is only touched by `_settlePnlAgainstLiquidator` for perp PnL — which is irrelevant for a pure spot-liability bad-debt account with no open perp positions. [4](#0-3) [5](#0-4) 

Compare with the normal liquidation path where `_handleLiquidationPayment` explicitly credits `liquidationFees` to the insurance fund and charges the liquidator a discounted price (i.e., the liquidator profits from the spread): [6](#0-5) [7](#0-6) 

---

### Impact Explanation

A liquidator who finalizes a bad-debt spot position:
- Receives **no liquidation fee** (no discounted asset purchase, no fee credit).
- Bears gas costs.
- Triggers socialization that dilutes all depositors' balances via `cumulativeDepositsMultiplierX18` reduction in `socializeSubaccount`. [8](#0-7) 

With no economic incentive, bad-debt spot positions will remain unfinalized. Depositors who need to withdraw are blocked until someone absorbs the gas cost for zero reward. The longer the position sits, the more interest accrues on the liability side, worsening the bad debt.

---

### Likelihood Explanation

Any subaccount that borrows a spot asset and experiences a price move large enough to push `quoteBalance + insurance < 0` triggers this path. This is a realistic scenario during high-volatility events or oracle latency gaps — exactly the conditions that produce bad debt in leveraged spot protocols. The insurance fund (`insurance` in `ClearinghouseStorage`) is a finite shared pool; once it is depleted by prior liquidations, subsequent bad-debt accounts hit this condition immediately. [9](#0-8) 

---

### Recommendation

In `_finalizeSubaccount`, when the subaccount is socialized (bad debt path), grant the liquidator a protocol-defined flat fee or a fraction of the socialized amount as compensation. Alternatively, allow the regular liability liquidation path to proceed even when `quoteBalance.amount + insurance < 0` by clamping `maxLiquidatable` to `spotAmount` (the full liability) and letting `updateQuoteFromInsurance` absorb whatever it can, then socializing the remainder — mirroring the fix suggested for Aloe M-3 of not blocking the swap when bad debt is the cause.

---

### Proof of Concept

**Scenario**:
- Subaccount `alice` has: spot liability = −100 TOKEN, quote balance = −50 USDC, insurance = 30 USDC.
- `quoteBalance.amount + insurance = −20 USDC < 0`.

**Step 1 — Regular liquidation reverts**:
```
liquidator calls liquidateSubaccountImpl({
    liquidatee: alice,
    productId: TOKEN_SPOT_ID,
    amount: -100e18,   // liquidate full liability
    isEncodedSpread: false
})
→ _assertLiquidationAmount:
    maxBuy = (-20e18).div(liquidationPrice) → negative
    maxBuy = max(negative + 1, 0) = 0
    maxLiquidatable = max(-100e18, 0) = 0
    require(0 <= -100e18)  ← REVERTS with ERR_NOT_LIQUIDATABLE_AMT
```

**Step 2 — Finalization pays nothing**:
```
liquidator calls liquidateSubaccountImpl({
    liquidatee: alice,
    productId: type(uint32).max,   // finalize
    amount: 0,
    isEncodedSpread: false
})
→ _finalizeSubaccount:
    canLiquidateMore = (-50 + 30) > 0 = false
    perpEngine.socializeSubaccount(alice, 30) → no perp positions, returns 30
    insuranceCover = min(30, 50) = 30
    quoteBalance becomes -20
    insurance = 0 → spotEngine.socializeSubaccount(alice)
    → alice's spot liability socialized across all depositors
    → liquidator quote balance: UNCHANGED (no fees paid)
```

The liquidator spent gas and received nothing. All depositors silently absorb the loss via `cumulativeDepositsMultiplierX18` reduction. [10](#0-9) [11](#0-10)

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }

    function socializeSubaccount(bytes32 subaccount) external {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];

            State memory state = states[productId];
            Balance memory balance = balanceNormalizedToBalance(
                state,
                balances[productId][subaccount]
            );
            if (balance.amount < 0) {
                int128 totalDeposited = state.totalDepositsNormalized.mul(
                    state.cumulativeDepositsMultiplierX18
                );

                state.cumulativeDepositsMultiplierX18 = (totalDeposited +
                    balance.amount).div(state.totalDepositsNormalized);

                require(state.cumulativeDepositsMultiplierX18 > 0);

                state.totalBorrowsNormalized += balance.amount.div(
                    state.cumulativeBorrowsMultiplierX18
                );

                _setBalanceAndUpdateBitmap(
                    productId,
                    subaccount,
                    BalanceNormalized({amountNormalized: 0})
                );
                _setState(productId, state);
            }
        }
    }
