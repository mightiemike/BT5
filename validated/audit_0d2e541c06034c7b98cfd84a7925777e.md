### Title
Coinbase Proof Check in `verify_transaction_inclusion_v2` Does Not Bind the Coinbase Identity, Allowing Internal-Node Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to fix the 64-byte Merkle proof forgery vulnerability present in the deprecated `verify_transaction_inclusion`. Its mitigation requires a caller to supply a coinbase Merkle proof alongside the transaction proof. However, the function never asserts that `coinbase_tx_id` is the actual coinbase of the block, nor does it enforce any structural relationship between the two proofs. An unprivileged NEAR caller can set `coinbase_tx_id = tx_id` and `coinbase_merkle_proof = merkle_proof`, causing both independent checks to pass with identical inputs. This allows any internal Merkle tree node to be presented as a valid transaction, returning `true` for a forged inclusion proof.

---

### Finding Description

`verify_transaction_inclusion_v2` performs three checks before returning a result:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — equal-length guard.
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof) == header.block_header.merkle_root` — coinbase proof verifies against the block's Merkle root.
3. Delegates to `verify_transaction_inclusion(args.into())`, which verifies `compute_root_from_merkle_proof(tx_id, tx_index, &merkle_proof) == merkle_root`. [1](#0-0) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` iterates over the proof hashes and combines them with the starting hash. It places no constraint on what the starting hash represents — leaf or internal node. [2](#0-1) 

Because the two proofs are verified **independently** against the same Merkle root, and because `coinbase_tx_id` is never checked against any on-chain record of the actual coinbase transaction, an attacker can supply identical values for both proof pairs. Consider a block with four transactions `[coinbase, T1, T2, T3]`:

```
Level 0 (leaves): coinbase, T1, T2, T3
Level 1:          AB = SHA256d(coinbase || T1),  CD = SHA256d(T2 || T3)
Level 2 (root):   R  = SHA256d(AB || CD)
```

The attacker submits:

| Field | Value |
|---|---|
| `tx_id` | `AB` (internal node) |
| `tx_index` | `0` |
| `merkle_proof` | `[CD]` |
| `coinbase_tx_id` | `AB` (identical to `tx_id`) |
| `coinbase_merkle_proof` | `[CD]` (identical to `merkle_proof`) |

- Length check: `len([CD]) == len([CD])` → passes.
- Coinbase check: `compute_root_from_merkle_proof(AB, 0, [CD])` = `SHA256d(AB || CD)` = `R` → passes.
- Transaction check (inside `verify_transaction_inclusion`): `compute_root_from_merkle_proof(AB, 0, [CD])` = `R` → passes.

The function returns `true` for `AB`, which is not a real transaction. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

Any NEAR contract that calls `verify_transaction_inclusion_v2` to gate a privileged action (e.g., releasing bridged funds, minting tokens, confirming a cross-chain payment) can be deceived into accepting a forged proof. The attacker does not need to forge a Bitcoin block or break SHA256d — all required values (`AB`, `CD`, `R`) are public data derivable from any confirmed Bitcoin block. The corrupted state is the **proof result** returned to the consuming contract: a `true` value for a transaction that was never broadcast or confirmed.

---

### Likelihood Explanation

Medium. The attacker needs only:
1. A block already accepted by the light client (public, on-chain state).
2. Knowledge of that block's Merkle tree (computable from public Bitcoin RPC data).
3. The ability to call a public, unpermissioned NEAR function.

No privileged role, no key material, and no computational infeasibility is involved. The only constraint is that the consuming contract must act on the boolean result without additional off-chain validation.

---

### Recommendation

The coinbase proof must be structurally linked to the transaction proof, not merely checked independently against the same root. Specifically:

- Assert `coinbase_tx_id != tx_id` to prevent the trivial self-referential bypass.
- Enforce that the two proof paths are consistent: at each tree level where the paths of position `0` (coinbase) and position `tx_index` diverge, the sibling node in one proof must equal the hash of the sibling pair in the other proof. This is the structural binding that the bitmex mitigation actually requires.
- Alternatively, require the caller to supply the raw coinbase transaction bytes and verify `SHA256d(coinbase_bytes) == coinbase_tx_id`, ensuring the coinbase identity is grounded in real transaction data.

---

### Proof of Concept

Given a block accepted by the contract with four transactions and Merkle root `R`:

```
AB = SHA256d(coinbase_bytes || T1_bytes)
CD = SHA256d(T2_bytes || T3_bytes)
R  = SHA256d(AB || CD)
```

Call (via NEAR CLI or any NEAR SDK client):

```json
{
  "tx_id": "<AB_hex>",
  "tx_block_blockhash": "<block_hash_hex>",
  "tx_index": 0,
  "merkle_proof": ["<CD_hex>"],
  "coinbase_tx_id": "<AB_hex>",
  "coinbase_merkle_proof": ["<CD_hex>"],
  "confirmations": 1
}
```

`verify_transaction_inclusion_v2` returns `true`. The internal node `AB` is accepted as a confirmed transaction, bypassing the 64-byte forgery mitigation that the v2 function was designed to enforce. [1](#0-0) [2](#0-1) [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
