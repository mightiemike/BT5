### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Check Introduced in v2 — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase merkle proof. However, the original `verify_transaction_inclusion` remains a fully public, externally callable NEAR contract method. Any unprivileged caller can invoke it directly, bypassing the coinbase proof check entirely and obtaining a forged `true` SPV result.

---

### Finding Description

The contract exposes two SPV proof entry points:

**`verify_transaction_inclusion`** — the deprecated v1: [1](#0-0) 

It carries only `#[pause]` and performs no coinbase proof validation. Its own NatSpec warns: [2](#0-1) 

**`verify_transaction_inclusion_v2`** — the secure replacement: [3](#0-2) 

v2 first verifies that the coinbase transaction (index 0) is correctly proven against the block's merkle root, then delegates to v1 for the actual tx check. This coinbase anchor prevents the 64-byte internal-node forgery attack.

The critical flaw: Rust's `#[deprecated]` attribute is a **compile-time warning only**. It has zero effect on external NEAR JSON-RPC callers. Any account on NEAR can call `verify_transaction_inclusion` directly as a contract view or change method, completely skipping the coinbase proof gate.

The `ProofArgs` struct accepted by v1 is a strict subset of `ProofArgsV2` — it requires no coinbase fields: [4](#0-3) 

So the attacker simply omits `coinbase_tx_id` and `coinbase_merkle_proof` and calls the old endpoint.

---

### Impact Explanation

The 64-byte transaction forgery attack (https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to craft a 64-byte input whose double-SHA256 hash equals an internal Merkle tree node hash. By supplying this crafted hash as `tx_id` with a valid Merkle path to the block's `merkle_root`, `verify_transaction_inclusion` returns `true` for a transaction that does not exist in the block.

Any downstream NEAR contract or off-chain system that calls `verify_transaction_inclusion` (rather than v2) to gate a high-value action — e.g., a Bitcoin bridge releasing tokens, a cross-chain payment protocol, or an escrow — can be deceived into accepting a forged proof. The corrupted proof result is the concrete state delta: a `true` return value for a non-existent Bitcoin transaction.

---

### Likelihood Explanation

The entry point is unconditionally reachable by any unprivileged NEAR account with no preconditions. The function is `pub`, has no `#[trusted_relayer]` guard, and the only protection (`#[pause]`) is a global operational switch not specific to this vulnerability. The 64-byte forgery technique is well-documented and publicly known. Any attacker aware of the contract's dual-endpoint design can exploit this immediately.

---

### Recommendation

Remove `verify_transaction_inclusion` as a public NEAR contract method. Since `verify_transaction_inclusion_v2` already delegates to it internally via `self.verify_transaction_inclusion(args.into())`, the v1 logic can be refactored into a private helper function (e.g., `fn verify_inclusion_inner(...)`) that is not exposed as a NEAR entry point. This eliminates the bypass while preserving the internal call chain.

---

### Proof of Concept

1. Attacker identifies a block `B` stored in the contract's `headers_pool` with merkle root `R`.
2. Attacker constructs a 64-byte value `X` such that `SHA256d(X) == R` (or an internal node hash that chains to `R`), exploiting the known Bitcoin Merkle second-preimage weakness.
3. Attacker calls `verify_transaction_inclusion` directly (not v2) with:
   - `tx_id = X`
   - `tx_block_blockhash = B`
   - `tx_index` and `merkle_proof` chosen so `compute_root_from_merkle_proof(X, index, proof) == R`
4. The function at line 318–322 computes the root and compares it to `header.block_header.merkle_root`: [5](#0-4) 
5. The comparison succeeds and the function returns `true` — a forged inclusion proof — with no coinbase anchor check ever executed.
6. Had the attacker called `verify_transaction_inclusion_v2`, step 358–365 would have rejected the proof: [6](#0-5)
