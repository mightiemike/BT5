Audit Report

## Title
`OracleValueStopLossExtension._afterSwapOracleStopLoss` Uses Arithmetic Mean Instead of Geometric Mean for Mid-Price, Miscalibrating the Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid-price as `(bid + ask) / 2` (arithmetic mean) at line 218, while the rest of the protocol — `SwapMath.midAndSpreadFeeX64FromBidAsk`, `MetricOmmPool.swap`, and `PriceVelocityGuardExtension.beforeSwap` — all use the geometric mean `sqrt(bid * ask)`. Because AM ≥ GM, the per-bin value metrics fed into the watermark comparison are systematically biased: `metricToken0` (which divides by mid) is understated, weakening the stop-loss guard for `zeroForOne = true` swaps and allowing LPs to lose more than the configured `drawdownE6` threshold before the guard fires.

## Finding Description
In `_afterSwapOracleStopLoss` (line 218), the mid-price is computed as:

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
```

This value is then passed to `_metrics` (lines 254–255), which computes:

```solidity
metricT0 = _clampMetric(t0ps + Math.mulDiv(Math.mulDiv(uint256(t1), Q64, midPriceX64), METRIC_SCALE, shares));
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
```

The correct formula used everywhere else in the protocol is `SwapMath.midAndSpreadFeeX64FromBidAsk`, which computes `sqrt(bid * ask)` (line 70 of `SwapMath.sol`). `OracleValueStopLossExtension.sol` does not import `SwapMath` at all.

Since AM ≥ GM always, the arithmetic mid is inflated by approximately `mid_geo * S²/2` where `S` is the half-spread fraction. This inflated mid causes:
- `metricT0` (divides by mid) to be **lower** than correct → watermark for `zeroForOne = true` is set too low → stop-loss fires later than configured, allowing excess token1 outflow
- `metricT1` (multiplies by mid) to be **higher** than correct → watermark for `zeroForOne = false` is set too high → potential false-positive reverts on legitimate swaps

The watermark comparison at lines 270–278 then uses these miscalibrated metrics to decide whether to revert, making the guard's effective drawdown threshold `drawdownE6 + S²/2` instead of `drawdownE6` for the `zeroForOne = true` direction.

## Impact Explanation
This is a direct bypass of the LP stop-loss protection invariant. An unprivileged trader executing `zeroForOne = true` swaps can extract additional LP value beyond the configured `drawdownE6` threshold before the guard triggers. The excess equals approximately `S²/2` of the metric value: ~0.125% for a 10% oracle spread, ~3.1% for a 50% spread. For pools with volatile or illiquid pairs (wider spreads), the bypass magnitude is material and constitutes a direct loss of LP principal that the extension is specifically designed to prevent.

## Likelihood Explanation
Every swap through a pool configured with `OracleValueStopLossExtension` and a non-zero `drawdownE6` triggers this path. No special privilege is required — any unprivileged trader can execute swaps. The bias is always present and grows with oracle spread width. Pools with wider bid/ask spreads are most affected. The condition is continuously exploitable.

## Recommendation
Import `SwapMath` into `OracleValueStopLossExtension` and replace the arithmetic mean with the geometric mean:

```solidity
// Before (wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This aligns the stop-loss guard's mid-price anchor with the one used by the pool's swap math and `PriceVelocityGuardExtension`.

## Proof of Concept
1. Deploy a pool with `OracleValueStopLossExtension`, `drawdownE6 = 50_000` (5%), and an oracle with a 10% spread (`bid = 0.95 * mid_geo`, `ask = 1.05 * mid_geo`).
2. Execute a first swap to establish the watermark. The arithmetic mid ≈ `mid_geo * 1.0025` (0.25% above geometric mid), so `metricT0` is ~0.25% lower than correct, and the watermark is set ~0.25% too low.
3. The breach threshold is `watermark * 0.95`. Because the watermark is 0.25% low, the effective drawdown allowed before the stop-loss fires is ~5.25% instead of 5%.
4. A trader executing `zeroForOne = true` swaps can extract an additional ~0.25% of LP value beyond the configured drawdown before the guard triggers — a direct bypass of the LP protection invariant. The bypass scales with `S²/2` and is larger for pools with wider oracle spreads.
