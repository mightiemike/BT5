### Title
Uninitialized Implementation Contracts Allow Attacker to Seize Ownership and Brick the Proxy via `delegatecall` - (File: `core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`, `core/contracts/SpotEngine.sol`, `core/contracts/PerpEngine.sol`, `core/contracts/OffchainExchange.sol`)

---

### Summary

Multiple upgradeable implementation contracts in the Nado protocol are missing the `_disableInitializers()` constructor guard recommended by OpenZeppelin. An unprivileged attacker can front-run deployment to call `initialize()` directly on the unprotected implementation contracts, seize ownership, and — in the case of `Endpoint.sol` — escalate to bricking the proxy entirely via its internal `delegatecall` to an attacker-controlled `endpointTx` address.

---

### Finding Description

OpenZeppelin's upgradeable contract documentation explicitly requires that implementation contracts include:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

This prevents any caller from invoking `initialize()` on the bare implementation (as opposed to the proxy). Without this guard, the implementation contract's `initializer` modifier is never locked, leaving it callable by anyone.

### Impact Explanation

**Primary impact — `Endpoint.sol` (most severe):**

`Endpoint.initialize()` calls `__Ownable_init()`, making the first caller the owner of the implementation. The attacker then controls `endpointTx`, `sequencer`, `clearinghouse`, `offchainExchange`, `verifier`, and `sanctions` on the implementation contract.

Critically, `Endpoint.sol` contains an internal `_delegatecallEndpointTx` that performs an unchecked `delegatecall` to the `endpointTx` address stored in implementation storage: [9](#0-8) 

An attacker who owns the implementation sets `endpointTx` to a malicious contract containing `selfdestruct`. When any function that routes through `_delegatecallEndpointTx` is triggered on the implementation, the `selfdestruct` executes in the implementation's context, destroying the implementation contract. With the implementation destroyed, the proxy becomes permanently bricked — all user funds locked with no upgrade path.

**Secondary impact — `Clearinghouse.sol`, `SpotEngine.sol`, `PerpEngine.sol`, `OffchainExchange.sol`:**

An attacker who initializes these implementations becomes their `owner`. While this does not directly corrupt proxy storage, it:
- Allows the attacker to call all `onlyOwner` functions on the implementation (e.g., `addOrUpdateProduct`, `updateRisk`, `removeLiquidity` on `BaseWithdrawPool` derivatives)
- Poisons the implementation's state, which can interfere with future upgrades if the proxy admin upgrades to a new implementation that reads from or delegates to the compromised one
- Satisfies the exact vulnerability class described in the external report

---

### Likelihood Explanation

The attack requires no special privileges. The attacker only needs to:
1. Monitor the mempool for deployment of the implementation contract
2. Call `initialize(...)` on the implementation before the deployer does (front-run), or simply call it any time after deployment if the deployer has not yet initialized the implementation

This is a well-known, mechanically straightforward front-run. The window is open from the moment the implementation is deployed until the deployer calls `initialize()` on it (which happens in a separate transaction). The `Endpoint.sol` case is particularly dangerous because the `delegatecall` escalation path to `selfdestruct` is reachable without any further governance action.

---

### Recommendation

Add the OpenZeppelin-recommended constructor to every affected implementation contract:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

Apply this to: `Endpoint.sol`, `Clearinghouse.sol`, `SpotEngine.sol`, `PerpEngine.sol`, and `OffchainExchange.sol`. This atomically locks the `initializer` at deployment time without requiring any additional transaction, eliminating the front-run window entirely.

---

### Proof of Concept

```
1. Deployer broadcasts tx to deploy Endpoint implementation contract.
2. Attacker observes the pending tx in the mempool.
3. Attacker submits a higher-gas tx calling:
       Endpoint(implementationAddress).initialize(
           attackerSanctions,
           attackerSequencer,
           attackerOffchainExchange,
           attackerClearinghouse,
           attackerVerifier,
           attackerEndpointTx   // <-- malicious contract with selfdestruct
       )
4. Attacker's tx is mined first. Attacker is now owner of the implementation.
   endpointTx on the implementation = attackerEndpointTx.
5. Attacker calls any Endpoint function that internally routes to
   _delegatecallEndpointTx (e.g., submitTransactions or any sequencer-gated
   path that the attacker can satisfy given they control sequencer address).
6. delegatecall executes selfdestruct in the context of the implementation.
7. Implementation contract is destroyed.
8. Proxy now points to a dead address. All user funds are permanently locked.
``` [10](#0-9) [11](#0-10) [8](#0-7)
