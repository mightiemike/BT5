### Title
`burnNlp` Uses Wrong Health Type (`MAINTENANCE` Instead of `INITIAL`), Allowing Subaccounts to Fall Below Initial Health â€” (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse::burnNlp` enforces a `MAINTENANCE` health check after the burn, while every other user-initiated state-changing operation (`withdrawCollateral`, `mintNlp`, `transferQuote`) enforces `INITIAL` health. This inconsistency allows a user to burn NLP and leave their subaccount below the initial health threshold, making it immediately liquidatable.

---

### Finding Description

In `Clearinghouse::burnNlp`, the final health guard is:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) 

Every other user-initiated operation uses `INITIAL`:

- `withdrawCollateral` (line 417): `IProductEngine.HealthType.INITIAL`
- `mintNlp` (line 480): `IProductEngine.HealthType.INITIAL`
- `transferQuote` (line 249): `_isAboveInitial()` â†’ `INITIAL` [2](#0-1) [3](#0-2) [4](#0-3) 

`MAINTENANCE` weights are always higher than `INITIAL` weights for long positions (`longWeightMaintenanceX18 > longWeightInitialX18`), as defined in `RiskHelper._getWeightX18`: [5](#0-4) 

Therefore `MAINTENANCE` health â‰Ą `INITIAL` health for any long position. A user whose `INITIAL` health is slightly above zero can burn NLP, pay the burn fee (which reduces health), and end up with `INITIAL` health < 0 while `MAINTENANCE` health â‰Ą 0. The check passes, leaving the subaccount in a liquidatable state.

The in-code comment reads: *"This check prevents malicious actors from deliberately creating unhealthy subaccounts through NLP burns."* But using `MAINTENANCE` instead of `INITIAL` does **not** prevent subaccounts from falling below initial health â€” it only prevents them from falling below maintenance health. The comment describes the intended behavior, but the wrong health type was used. [6](#0-5) 

The burn fee is computed as:

```solidity
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [7](#0-6) 

At most 0.1% of the NLP value. For large NLP positions this is a meaningful absolute amount that can push `INITIAL` health negative while `MAINTENANCE` health remains comfortably positive.

The `BurnNlp` transaction is user-signed and processed via `processTransactionImpl` with `allowLinkedSigner = true`, so both the account owner and any linked signer can trigger this path: [8](#0-7) 

---

### Impact Explanation

A user (or their linked signer) who burns NLP while their `INITIAL` health is close to zero will end up with `INITIAL` health < 0. Their subaccount is now immediately liquidatable per `ClearinghouseLiq::liquidateSubaccountImpl`, which only requires `isUnderMaintenance` to be false for the liquidatee to be safe â€” but here the subaccount is below `INITIAL`, so a liquidator can act. [9](#0-8) 

The user cannot self-rescue by withdrawing quote tokens received from the burn, because `withdrawCollateral` enforces `INITIAL` health and will revert. The user is trapped in a liquidatable state and loses funds to the liquidation discount.

---

### Likelihood Explanation

Medium. The scenario requires `INITIAL` health to be within the burn fee amount of zero at the time of the burn. For a user with a large leveraged position (e.g., 100 ETH at 80% initial weight), the gap between `INITIAL` and `MAINTENANCE` health is large, but the burn fee only needs to exceed the small positive `INITIAL` health margin. A user who mints NLP to maximize leverage and then burns it is a realistic path. Linked signers further expand the attack surface.

---
