### Title
Coinbase `tx_id` Not Verified to Be a Leaf Node — 64-Byte Merkle Forgery Mitigation Bypassed - (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` is intended to prevent the 64-byte transaction Merkle proof forgery attack by requiring a coinbase proof of equal depth. However, the contract never verifies that `coinbase_tx_id` is actually the hash of the real coinbase transaction (a leaf node). An attacker can supply an internal Merkle tree node as `coinbase_tx_id` with a shorter proof that still reconstructs the root, satisfying the length-equality check with an equally short `merkle_proof` for the forged transaction. The result is that `verify_transaction_inclusion_v2` returns `true` for a `tx_id` that is an internal node, not a real transaction — the exact attack the mitigation was designed to block.

---

### Finding Description

The v2 mitigation relies on the invariant that the coinbase is always a real leaf at depth = tree depth, so its proof length equals the full tree depth. If an attacker tries to forge a transaction at a shallower depth (using an internal node), the coinbase proof would need to be the same length, but the real coinbase proof is longer — lengths would not match.

The flaw: the contract does not enforce that `coinbase_tx_id` is the actual coinbase leaf. It only checks:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` [1](#0-0) 
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == merkle_root` [2](#0-1) 
3. Then delegates to `verify_transaction_inclusion(args.into())`, which checks `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == merkle_root` [3](#0-2) 

`compute_root_from_merkle_proof` is a pure hash-chain computation with no leaf/node distinction: [4](#0-3) 

An internal node at position 0 in the tree satisfies check 2 with a proof one level shorter than the full tree depth. Setting `coinbase_tx_id = tx_id = <internal_node>` and `coinbase_merkle_proof = merkle_proof = <shorter_proof>` satisfies all three checks simultaneously.

`ProofArgsV2` carries `coinbase_tx_id` as a raw `H256` with no semantic constraint: [5](#0-4) 

---

### Impact Explanation

An attacker can call `verify_transaction_inclusion_v2` with a `tx_id` that is an internal Merkle tree node (not a real transaction), and the contract returns `true`. This is the 64-byte forgery attack: a 64-byte blob interpretable as two concatenated 32-byte hashes can be an internal node. Any downstream system that trusts the contract's `true` return to confirm a payment or cross-chain event is deceived into accepting a non-existent transaction.

---

### Likelihood Explanation

The call is public (no `#[private]`, no trusted-relayer access control on the view path). Any caller can submit arbitrary `ProofArgsV2` values. The attacker only needs to know the Merkle tree of a block already stored in the contract's `headers_pool`, which is public on-chain state. No privileged role or key is required.

---

### Recommendation

After the coinbase proof passes, verify that `coinbase_tx_id` equals the actual coinbase transaction hash stored or derivable from the block. At minimum, enforce that the coinbase proof length equals the full tree depth (i.e., `coinbase_merkle_proof.len()` must equal `ceil(log2(tx_count))`), and that `coinbase_tx_id` is not equal to `tx_id` when `tx_index != 0`. The strongest fix is to store the coinbase txid alongside the block header at submission time and compare against it directly in `verify_transaction_inclusion_v2`.

---

### Proof of Concept

For a block with 4 transactions `[A (coinbase), B, C, D]`:

```
        root
       /    \
     AB      CD
    /  \    /  \
   A    B  C    D
```

where `AB = double_sha256(A || B)`, `CD = double_sha256(C || D)`, `root = double_sha256(AB || CD)`.

Submit `ProofArgsV2`:
```
tx_id                = AB          // internal node, not a real tx
tx_block_blockhash   = block_hash
tx_index             = 0
merkle_proof         = [CD]        // length 1
coinbase_tx_id       = AB          // same internal node
coinbase_merkle_proof = [CD]       // length 1
confirmations        = 0
```

- Check 1: `1 == 1` ✓
- Check 2: `compute_root_from_merkle_proof(AB, 0, [CD]) = hash(AB, CD) = root` ✓
- Check 3: `compute_root_from_merkle_proof(AB, 0, [CD]) = hash(AB, CD) = root` ✓

Contract returns `true` for `tx_id = AB`, an internal node — the 64-byte forgery mitigation is fully bypassed.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
