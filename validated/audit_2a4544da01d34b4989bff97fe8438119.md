Audit Report

## Title
`SwapAllowlistExtension::beforeSwap` Checks Router Address Instead of Actual End-User, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which the pool sets to `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. The extension therefore evaluates `allowedSwapper[pool][routerAddress]` instead of `allowedSwapper[pool][actualUser]`, completely nullifying per-user access control. This produces two fund-impacting outcomes: a full allowlist bypass when the router is allowlisted, and permanent denial of service for legitimately allowlisted users who swap through the router.

## Finding Description
`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- pool's msg.sender = whoever called pool.swap()
    recipient,
    ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this `sender` verbatim to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]` — i.e., `allowedSwapper[pool][pool.msg.sender]`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly, making the pool's `msg.sender` the **router contract**: [4](#0-3) 

The same pattern applies to `exactOutputSingle` and `exactInput`/`exactOutput` multi-hop paths: [5](#0-4) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the explicit `owner` parameter (the actual LP), not `sender` (the direct caller): [6](#0-5) 

The `beforeSwap` interface exposes both `sender` and `recipient`; the extension uses `sender` (the router) and ignores `recipient` (the actual user): [7](#0-6) 

The existing unit tests for `SwapAllowlistExtension` call `extension.beforeSwap(swapper, ...)` with `vm.prank(address(pool))` directly — they never exercise the router path, so the mismatch is untested: [8](#0-7) 

## Impact Explanation
**Bypass (Critical):** A pool admin who calls `setAllowedToSwap(pool, routerAddress, true)` to enable router-based swaps inadvertently grants every address — including those never allowlisted — the ability to swap through the router. Any attacker can drain liquidity from a pool intended to be restricted to specific counterparties.

**Broken functionality (High):** A pool admin who allowlists individual user addresses finds those users permanently blocked from the standard swap path (the router is not allowlisted), making the pool's primary swap interface unusable for legitimate participants and locking LP assets in an inaccessible pool.

Both outcomes constitute direct loss of user principal or broken core swap functionality, meeting Sherlock Critical/High thresholds.

## Likelihood Explanation
`SwapAllowlistExtension` is a production periphery contract. Any pool that configures it with `beforeSwap` and expects per-user gating is affected on every router-mediated swap. No special attacker capability is required — a standard `exactInputSingle` call suffices. The router is the primary swap path for end-users, so the mismatch is triggered on every normal interaction.

## Recommendation
The extension must check the actual end-user identity, not the direct caller of `pool.swap()`. Two viable approaches:

1. **Use `recipient`** if the pool's design guarantees `recipient` is the economic beneficiary (minimal change, but semantically fragile if `recipient` can be set to an arbitrary address).

2. **Router encodes real user in `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes it and verifies it against the callback context already stored in transient storage (the router already stores `msg.sender` as the payer via `_setNextCallbackContext`). This is the robust fix.

Minimal diff for approach 1:
```diff
- function beforeSwap(address sender, address, bool, ...)
+ function beforeSwap(address, address recipient, bool, ...)
  {
-   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
+   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
```

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured in `beforeSwap` order.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` — intending to enable router-based swaps.
3. Attacker (address never allowlisted) calls `router.exactInputSingle({recipient: attacker, ...})`.
4. Router calls `pool.swap(recipient=attacker, ...)` with `msg.sender = router`.
5. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. Attacker successfully swaps in a pool intended to restrict them.

Alternatively (broken-functionality path):
1. Pool admin calls `setAllowedToSwap(pool, userAddress, true)` for a legitimate user.
2. Legitimate user calls `router.exactInputSingle(...)`.
3. Extension evaluates `allowedSwapper[pool][router]` → `false` → `NotAllowedToSwap` revert.
4. Legitimate user is permanently blocked from the standard swap path despite being allowlisted.

A Foundry integration test deploying a real pool with `SwapAllowlistExtension`, allowlisting only a specific user, then calling `router.exactInputSingle` from a non-allowlisted address will reproduce the bypass.
