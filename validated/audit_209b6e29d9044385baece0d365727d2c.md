### Title
`forceRebalanceNlpPool` Skips Health Check for `nlpPools[0]` After Balance Reduction — (`File: core/contracts/Clearinghouse.sol`)

### Summary
`forceRebalanceNlpPool` applies quote balance deltas to **all** NLP pools via `_applyNlpRebalance`, but the post-rebalance health validation loop starts at index `1`, permanently skipping `nlpPools[0]` (which is initialized to `N_ACCOUNT`, the protocol's primary liquidity pool). A rebalance that reduces `N_ACCOUNT`'s quote balance below zero passes without any health enforcement, leaving the main pool in an unhealthy state with no on-chain detection.

### Finding Description
`_applyNlpRebalance` iterates from `i = 0` and applies `nlpPoolRebalanceX18[i]` to every pool's `QUOTE_PRODUCT_ID` balance:

```solidity
for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
    spotEngine.updateBalance(
        QUOTE_PRODUCT_ID,
        nlpPools[i].subaccount,
        nlpPoolRebalanceX18[i]
    );
}
``` [1](#0-0) 

Immediately after, `forceRebalanceNlpPool` validates health only for pools at indices `1` through `nlpPools.length - 1`:

```solidity
for (uint128 i = 1; i < nlpPools.length; i++) {
    require(
        getHealth(nlpPools[i].subaccount, IProductEngine.HealthType.INITIAL) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
``` [2](#0-1) 

`nlpPools[0]` is hardcoded to `N_ACCOUNT` at initialization:

```solidity
nlpPools.push(NlpPool({
    poolId: 0,
    subaccount: N_ACCOUNT,
    owner: address(0),
    balanceWeightX18: uint128(ONE)
}));
``` [3](#0-2) 

`_validateNlpRebalance` only enforces that the **sum** of all rebalance amounts equals zero — it does not prevent any individual pool from receiving a large negative delta:

```solidity
require(deltaQuoteAmount == rebalanceAmount, ERR_INVALID_NLP_REBALANCE);
``` [4](#0-3) 

So a rebalance where `nlpPoolRebalanceX18[0]` is a large negative value and the remaining entries are positive passes `_validateNlpRebalance`, reduces `N_ACCOUNT`'s quote balance to an arbitrarily negative level, and then exits without any health check on pool 0.

### Impact Explanation
`N_ACCOUNT` is the protocol's primary NLP pool. It is explicitly excluded from liquidation (`txn.liquidatee != N_ACCOUNT` in `liquidateSubaccountImpl`), so once it becomes unhealthy there is no recovery path through the normal liquidation flow. [5](#0-4) 

A deeply negative quote balance on `N_ACCOUNT` corrupts the protocol's aggregate solvency accounting: `manualAssert` and `assertUtilization` both rely on the spot engine's total deposit/borrow state, which is directly affected by `N_ACCOUNT`'s balance. Downstream, `burnNlp` and `nlpProfitShare` both pay out from NLP pool quote balances; if `N_ACCOUNT` is already insolvent, those payouts deepen the deficit silently. [6](#0-5) 

### Likelihood Explanation
`forceRebalanceNlpPool` is a routine sequencer operation used to redistribute quote liquidity across NLP pools during normal protocol activity. The sequencer calls it legitimately; the bug is in the protocol's own invariant enforcement, not in sequencer behavior. Any `ForceRebalanceNlpPool` transaction where `nlpPoolRebalanceX18[0]` is sufficiently negative triggers the missing check. No malicious actor is required — a miscalculation or an edge-case rebalance scenario is sufficient.

### Recommendation
Change the health-check loop in `forceRebalanceNlpPool` to start at `i = 0` so that `nlpPools[0]` (`N_ACCOUNT`) is also validated:

```solidity
for (uint128 i = 0; i < nlpPools.length; i++) {
    require(
        getHealth(nlpPools[i].subaccount, IProductEngine.HealthType.INITIAL) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
``` [2](#0-1) 

### Proof of Concept
1. Protocol has two NLP pools: `nlpPools[0] = N_ACCOUNT`, `nlpPools[1] = poolB`.
2. Sequencer submits a `ForceRebalanceNlpPool` transaction with:
   - `nlpPoolRebalanceX18[0] = -X` (large negative, drains `N_ACCOUNT`)
   - `nlpPoolRebalanceX18[1] = +X` (offsets sum to zero, passes `_validateNlpRebalance`)
3. `_applyNlpRebalance` reduces `N_ACCOUNT`'s quote balance by `X`, potentially below zero.
4. The health-check loop runs only for `i = 1` (poolB), which received `+X` and is healthy.
5. `N_ACCOUNT` is now insolvent; no revert occurs, no on-chain flag is set.
6. Subsequent `burnNlp` calls draw further quote from the already-insolvent `N_ACCOUNT`, compounding the deficit with no protocol-level detection. [7](#0-6)
