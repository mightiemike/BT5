### Title
Linked Signer Can Unilaterally Replace Itself via `LinkSigner` Transaction, Enabling Persistent Subaccount Takeover — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type in `processTransactionImpl` is validated with `allowLinkedSigner = true`, meaning the **current linked signer** — not just the subaccount owner — can sign a new `LinkSigner` transaction to replace itself with any attacker-controlled address. This gives a compromised or malicious linked signer the ability to permanently re-delegate signing authority over a subaccount, defeating any key-rotation attempt by the legitimate owner.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the sequencer-path `LinkSigner` handler is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes the current linked signer address to `verifier.validateSignature`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

The verifier accepts the signature if the recovered address is **either** the subaccount owner **or** the linked signer:

```solidity
(recovered == address(uint160(bytes20(sender)))) ||
    (recovered == linkedSigner)),
ERR_INVALID_SIGNATURE
``` [3](#0-2) 

Because `LinkSigner` itself is validated with `allowLinkedSigner = true`, the linked signer can sign a `LinkSigner` transaction that writes a **new, attacker-controlled address** into `linkedSigners[subaccount]`. [4](#0-3) 

The slow-mode `LinkSigner` path, by contrast, only allows the subaccount owner (`validateSender` checks `msg.sender` against the first 20 bytes of the subaccount) to set the linked signer — it does not accept a linked-signer signature:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [5](#0-4) 

This asymmetry means the fast (sequencer) path grants the linked signer a capability — self-replacement — that the slow (owner-only) path deliberately withholds.

---

### Impact Explanation

Once a linked signer replaces itself with an attacker-controlled address, the attacker's new key can sign any transaction that accepts `allowLinkedSigner = true`, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` (sends funds to an arbitrary `sendTo` address) [6](#0-5) 
- `TransferQuote` (moves quote balance to an attacker subaccount) [7](#0-6) 
- `LiquidateSubaccount` (can be used to force-liquidate the victim at a disadvantageous price) [8](#0-7) 

The critical additional harm beyond a simple key compromise is **persistence**: if the legitimate owner detects the compromise and attempts to rotate the linked signer by signing a new `LinkSigner` with the old key, the attacker can race to sign another `LinkSigner` first (both are sequencer-ordered), re-establishing attacker control. The owner's only reliable recovery path is the slow-mode queue, which has a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). [9](#0-8) 

**Corrupted state**: `linkedSigners[victimSubaccount]` is overwritten to an attacker address, enabling full signing authority over the subaccount's collateral.

---

### Likelihood Explanation

Linked signers are the standard mechanism for trading bots and delegated order-signing in the Nado protocol. Any user who grants a linked signer (a common, documented use case) is exposed. The attacker entry path requires only:

1. Possession of the current linked signer private key (e.g., a compromised hot wallet, a malicious third-party trading service, or a leaked API key).
2. Submission of a single `LinkSigner` transaction through the sequencer.

No admin access, governance capture, or oracle manipulation is required. The attack is a single sequencer-submitted transaction.

---

### Recommendation

Change the `LinkSigner` fast-path validation to `allowLinkedSigner = false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the slow-mode path's intent: only the subaccount owner (whose key signs the EIP-712 digest) should be permitted to mutate the `linkedSigners` mapping. The linked signer's role is to act on behalf of the owner, not to re-delegate that authority.

---

### Proof of Concept

1. Alice sets `linkedSigners[aliceSubaccount] = botAddress` (normal usage).
2. Attacker compromises `botAddress` private key.
3. Attacker constructs `SignedLinkSigner { tx: { sender: aliceSubaccount, signer: attackerAddress, nonce: N }, signature: sig_by_botKey }`.
4. Sequencer calls `processTransactionImpl` → `validateSignedTx(..., true)` → `verifier.validateSignature` recovers `botAddress` == current linked signer → **passes**.
5. `linkedSigners[aliceSubaccount]` is now `attackerAddress`.
6. Alice signs a new `LinkSigner` with `botAddress` to rotate — but the sequencer sees `linkedSigners[aliceSubaccount] == attackerAddress`, not `botAddress`, so Alice's rotation attempt with `botAddress` is **rejected** (botAddress is no longer the linked signer and is not the subaccount owner).
7. Attacker uses `attackerAddress` to sign `WithdrawCollateralV2` with `sendTo = attackerWallet`, draining Alice's collateral before Alice can execute a slow-mode revocation (3-day delay). [1](#0-0) [10](#0-9)
