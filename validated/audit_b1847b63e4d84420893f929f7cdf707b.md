### Title
Relayer Silently Skips Unconfirmed Block Batches When `final_execution_outcome` Is `None` — (`relayer/src/near_client.rs`)

---

### Summary

`parse_submit_blocks_response` returns `Ok(SubmitResult { gas_burnt: 0 })` when `final_execution_outcome` is `None`. Because `get_tx_status` uses `wait_until: TxExecutionStatus::Executed` (not `Final`), the NEAR RPC can legitimately return a response with `final_execution_outcome = None` for a transaction that has been executed but not yet finalized. The caller in `main.rs` unconditionally advances `first_block_height_to_submit` on any `Ok(Ok(_))` result, so the relayer permanently skips the unsubmitted batch.

---

### Finding Description

**Step 1 — `get_tx_status` waits only for `Executed`, not `Final`.**

`get_tx_status` issues the RPC call with `wait_until: TxExecutionStatus::Executed`. [1](#0-0) 

Per NEAR protocol semantics, `TxExecutionStatus::Executed` means all receipts have been processed, but the block containing the transaction has not necessarily been finalized. The `final_execution_outcome` field in `RpcTransactionResponse` is only guaranteed to be populated when the status reaches `TxExecutionStatus::Final`. Under normal network load, the RPC can and does return a response with `final_execution_outcome = None` for an `Executed` transaction.

**Step 2 — `parse_submit_blocks_response` treats `None` as success.**

The function checks for `final_execution_outcome` with an `if let Some(...)` guard. When the field is absent, execution falls through to the unconditional `Ok(SubmitResult { gas_burnt: 0 })` at line 340 — no error, no retry signal. [2](#0-1) 

**Step 3 — The caller advances `first_block_height_to_submit` unconditionally on `Ok(Ok(_))`.**

In `prepare_and_submit_batches`, the `Ok(Ok(result))` arm stores `tx.last_block_height + 1` into `first_block_height_to_submit` before checking `gas_burnt`. The `gas_burnt == 0` guard only gates the `sizer.on_success` call; the pointer advance happens regardless. [3](#0-2) 

So when `final_execution_outcome` is `None`:
- `parse_submit_blocks_response` returns `Ok(SubmitResult { gas_burnt: 0 })`
- `first_block_height_to_submit` is advanced to `tx.last_block_height + 1`
- The batch is never retried
- The on-chain contract never received those headers

---

### Impact Explanation

The relayer's internal cursor `first_block_height_to_submit` advances past a batch of BTC headers that were never accepted by the contract. On the next iteration the relayer fetches headers starting from `tx.last_block_height + 1`, submitting them against a contract tip that does not include the skipped batch. The contract rejects the submission with `PrevBlockNotFound`, and the relayer enters a permanent error loop. `mainchain_tip_blockhash` stalls at the last successfully confirmed block; the light client becomes unable to verify any BTC transaction whose confirmation window spans the gap.

---

### Likelihood Explanation

This condition does not require an attacker. It occurs naturally whenever the NEAR network is under load and a transaction is executed but not yet finalized at the moment the RPC responds. It can also be triggered deterministically by a relayer operator pointing at a non-archival or lagging RPC node, or by any network partition that delays finality. No privileged access, key compromise, or social engineering is required.

---

### Recommendation

`parse_submit_blocks_response` must treat `final_execution_outcome = None` as an error, not a success. The simplest fix mirrors the pattern already used in `parse_verify_response`:

```rust
// near_client.rs — parse_submit_blocks_response
let final_execution_outcome = response
    .final_execution_outcome
    .ok_or(CustomError::TxExecutionError(
        "missing final_execution_outcome".to_string(),
    ))?;
```

Alternatively, change `get_tx_status` to use `wait_until: TxExecutionStatus::Final`, which guarantees `final_execution_outcome` is populated before the RPC returns. [4](#0-3) 

---

### Proof of Concept

1. Configure the relayer against a NEAR RPC node under load (or a stub that returns `RpcTransactionResponse { final_execution_outcome: None, .. }` for any `tx` status query).
2. Trigger `prepare_and_submit_batches` with a batch of, say, 10 headers at heights `[H, H+9]`.
3. Observe that `get_tx_status` returns `Ok(response)` with `response.final_execution_outcome == None`.
4. `parse_submit_blocks_response` returns `Ok(SubmitResult { gas_burnt: 0 })`.
5. `first_block_height_to_submit` is stored as `H + 10`.
6. Query the contract's `get_last_block_header`: tip is still at `H - 1`.
7. On the next relayer iteration, it submits headers `[H+10, H+19]`; the contract returns `PrevBlockNotFound` because height `H` was never submitted.
8. The relayer is now permanently stuck.
