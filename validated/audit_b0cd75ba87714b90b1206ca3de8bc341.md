Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address, but its `beforeAddLiquidity` hook silently discards the `sender` argument and checks `owner` instead. Because `owner` is a free caller-supplied parameter with no on-chain constraint tying it to `msg.sender`, any unprivileged address can bypass the allowlist by passing any allowlisted address as `owner`, depositing tokens that are then locked under that address's position.

## Finding Description
`MetricOmmPool.addLiquidity` calls the extension hook with `msg.sender` as the first argument (the actual depositor) and the caller-supplied `owner` as the second: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both positionally to the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` leaves the first parameter (`sender`) unnamed and checks only `owner`: [3](#0-2) 

The check `allowedDepositor[msg.sender][owner]` should be `allowedDepositor[msg.sender][sender]`. The correct pattern is already used in `SwapAllowlistExtension.beforeSwap`, which checks `allowedSwapper[msg.sender][sender]`: [4](#0-3) 

After the guard passes, LP shares are minted to `owner`. `removeLiquidity` enforces `msg.sender == owner`: [5](#0-4) 

So the depositor (`bob`) cannot recover the tokens; only `owner` (`alice`) can withdraw them.

## Impact Explanation
Two concrete impacts:

1. **Admin-boundary break**: The pool admin's deposit allowlist — the primary mechanism for permissioned pools (KYC-gated, institutional-only, whitelist-only) — is completely inoperative. Any excluded address can deposit by naming any allowlisted address as `owner`.
2. **Direct loss of user principal**: The non-allowlisted depositor provides tokens via the swap callback but LP shares are minted to `owner`. Since `removeLiquidity` requires `msg.sender == owner`, the depositor has no path to recover their tokens. This constitutes a direct, irrecoverable loss of deposited principal for the attacker-as-depositor scenario, or can be used to forcibly credit tokens to any allowlisted address.

Both impacts — admin-boundary break and direct loss of user principal — are within the contest's allowed impact gate.

## Likelihood Explanation
Exploitation requires only: (1) knowledge of one allowlisted address, readable from the public `allowedDepositor` mapping or emitted `AllowedToDepositSet` events; and (2) the ability to call `addLiquidity`. No privileged role, flash loan, oracle manipulation, or special setup is needed. Any user of the protocol can trigger this on any pool that deploys `DepositAllowlistExtension` in non-allow-all mode.

## Recommendation
Name the first parameter `sender` and check it instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists only `alice`:
   ```solidity
   extension.setAllowedToDeposit(pool, alice, true);
   ```
2. `bob` (not allowlisted) calls:
   ```solidity
   pool.addLiquidity(alice, salt, deltas, callbackData, extensionData);
   ```
3. Inside `beforeAddLiquidity`: `msg.sender == pool`, `owner == alice`. The check `allowedDepositor[pool][alice]` returns `true` → no revert.
4. `bob` satisfies the token transfer callback, depositing his tokens into the pool. LP shares are minted to `alice`.
5. `alice` calls `removeLiquidity(alice, ...)` and withdraws `bob`'s tokens. `bob` has no recourse because `removeLiquidity` enforces `msg.sender == owner`.
6. The pool admin's deposit restriction was never enforced against `bob`.
