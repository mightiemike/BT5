### Title
`validate_resource_bounds()` in `StatelessTransactionValidator` Skips `max_l2_gas_amount` Check for Declare Transactions - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The gateway's stateless validator enforces a `max_l2_gas_amount` upper bound on `l2_gas.max_amount` for `Invoke` and `DeployAccount` transactions, but explicitly skips this check for `Declare` transactions. Any unprivileged user can submit a `Declare` transaction with `l2_gas.max_amount = u64::MAX`, bypassing the configured admission limit entirely.

### Finding Description

In `validate_resource_bounds()`, the `max_l2_gas_amount` guard is wrapped in a type-dispatch that short-circuits for `Declare`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
```

The production default for `max_l2_gas_amount` is `1,210,000,000`. For `Invoke` and `DeployAccount` transactions, any `l2_gas.max_amount` exceeding this value is rejected at the gateway. For `Declare` transactions, the branch is a no-op and the check is never reached.

The test `valid_l2_gas_amount_on_declare` explicitly codifies this as accepted behavior: a Declare transaction with `max_amount: GasAmount(200)` passes when `max_l2_gas_amount: 100` is configured.

The `test_invalid_max_l2_gas_amount` test only covers `TransactionType::DeployAccount` and `TransactionType::Invoke`, confirming `Declare` is intentionally excluded from enforcement.

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

An unprivileged user can submit a `RpcDeclareTransaction::V3` with `l2_gas.max_amount = u64::MAX` (18,446,744,073,709,551,615 gas units — over 15 billion times the configured limit). The gateway's stateless validator will accept it. This breaks the admission invariant that `max_l2_gas_amount` is a hard ceiling on the L2 gas bound any transaction may declare, and allows Declare transactions to enter the mempool with resource bounds that would be rejected for every other transaction type.

### Likelihood Explanation

**High.** The bypass requires no privilege, no special account state, and no knowledge beyond the public Starknet RPC API. Any caller of `starknet_addDeclareTransaction` can set `l2_gas.max_amount` to an arbitrary value. The TODO comment in the source confirms the gap is known but unresolved.

### Recommendation

Remove the `Declare`-specific exemption and apply the same `max_l2_gas_amount` bound uniformly:

```rust
fn validate_resource_bounds(
    &self,
    tx: &RpcTransaction,
) -> StatelessTransactionValidatorResult<()> {
    if !self.config.validate_resource_bounds {
        return Ok(());
    }

    let resource_bounds = *tx.resource_bounds();
    if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
        return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
    }

    if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
        return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { … });
    }

    // Apply max_l2_gas_amount to ALL transaction types, including Declare.
    if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
        return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
    }

    Ok(())
}
```

If Declare transactions legitimately require a higher gas ceiling, introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.

### Proof of Concept

1. Construct a valid `RpcDeclareTransaction::V3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and a valid Sierra contract class.
2. Submit via `starknet_addDeclareTransaction` to the gateway.
3. The stateless validator calls `validate_resource_bounds()`. The `if let RpcTransaction::Declare(_) = tx {}` branch matches and the `max_l2_gas_amount` check is skipped entirely.
4. The transaction passes stateless validation and enters the mempool, despite `l2_gas.max_amount` being `u64::MAX` — over 15 billion times the configured `max_l2_gas_amount` of `1,210,000,000`.

The existing test `valid_l2_gas_amount_on_declare` in `crates/apollo_gateway/src/stateless_transaction_validator_test.rs` (lines 173–201) directly demonstrates this: it asserts `Ok(())` for a Declare transaction with `max_amount: GasAmount(200)` when `max_l2_gas_amount: 100`.
