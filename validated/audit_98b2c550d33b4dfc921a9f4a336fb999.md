### Title
`max_l2_gas_amount` Cap Not Enforced for Declare Transactions in Gateway Stateless Validation — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

The `StatelessTransactionValidatorConfig` defines a `max_l2_gas_amount` cap (default: 1,210,000,000) that is meant to prevent transactions from declaring more L2 gas than a block can accommodate. This cap is enforced for `Invoke` and `DeployAccount` transactions, but is explicitly skipped for `Declare` transactions. Any user can submit a `Declare` transaction with `l2_gas.max_amount` set to an arbitrarily large value (e.g., `u64::MAX`) and it will pass gateway admission.

---

### Finding Description

In `validate_resource_bounds` inside `StatelessTransactionValidator`, the `max_l2_gas_amount` check contains an explicit early-return for `Declare` transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

The `if let RpcTransaction::Declare(_) = tx { }` branch is an empty body — it does nothing and falls through to `Ok(())`, bypassing the cap check entirely. The TODO comment acknowledges this gap. The test `valid_l2_gas_amount_on_declare` explicitly confirms the bypass: a Declare with `max_amount: GasAmount(200)` passes when `max_l2_gas_amount: 100`.

The `max_l2_gas_amount` is configured at `1,210,000,000` by default and is the gateway's primary admission control for L2 gas amount. It is enforced for all other transaction types.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

A Declare transaction with `l2_gas.max_amount = u64::MAX` (or any value exceeding `max_l2_gas_amount`) passes all gateway stateless checks and is admitted to the mempool. The gateway's cap — which exists to prevent transactions from claiming more gas than a block can hold — is silently bypassed for the entire Declare transaction type. This allows an unprivileged attacker to flood the mempool with Declare transactions carrying unbounded gas claims, circumventing the admission control invariant that the gateway is supposed to enforce.

---

### Likelihood Explanation

**Likelihood: High.**

The bypass requires no special privilege. Any user submitting an `RpcDeclareTransaction::V3` with `resource_bounds.l2_gas.max_amount` set above the configured cap will pass gateway validation unconditionally. The code path is reachable through the standard public RPC `starknet_addDeclareTransaction` endpoint. The TODO comment confirms the developers are aware the check is missing.

---

### Recommendation

Apply the same `max_l2_gas_amount` check to `Declare` transactions. Remove the empty `if let RpcTransaction::Declare(_) = tx { }` branch and let the existing check apply uniformly:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher gas ceiling (e.g., due to large Sierra programs), introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.

---

### Proof of Concept

The existing test `valid_l2_gas_amount_on_declare` in `crates/apollo_gateway/src/stateless_transaction_validator_test.rs` already demonstrates the bypass:

```rust
// config: max_l2_gas_amount: 100
// tx: l2_gas.max_amount: GasAmount(200)  -- exceeds cap by 2x
// tx_type: Declare
assert_matches!(tx_validator.validate(&tx), Ok(()));  // passes, cap not enforced
```

An attacker submits:
```json
{
  "type": "DECLARE",
  "resource_bounds": {
    "l2_gas": {
      "max_amount": "0xFFFFFFFFFFFFFFFF",
      "max_price_per_unit": "<valid_price>"
    }
  }
}
```

This transaction passes `StatelessTransactionValidator::validate_resource_bounds` and is admitted to the mempool, despite `l2_gas.max_amount` being `u64::MAX` — orders of magnitude above the configured `max_l2_gas_amount` of `1,210,000,000`.
