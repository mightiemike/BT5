### Title
Difficulty Calculation Anchored to Mainchain Ancestor Instead of Fork Ancestor, Enabling Incorrect PoW Acceptance — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`)

---

### Summary

In `get_next_work_required` for both Dogecoin and Bitcoin, the difficulty calculation fetches the first block of the adjustment interval via `get_header_by_height`, which always reads from the **current mainchain** mapping. When validating fork blocks, this returns the mainchain block at that height instead of the fork's actual ancestor, producing an incorrect `expected_bits` value. For Dogecoin post-height 145,000 (where the adjustment interval is 1), this affects **every fork block** beyond the first. The developer explicitly flagged this with a `TODO` comment in the code.

---

### Finding Description

`get_header_by_height` is implemented in `lib.rs` as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

It unconditionally reads from `mainchain_height_to_header`, which maps only mainchain blocks. It has no awareness of which fork chain is currently being validated.

In `dogecoin.rs`, `get_next_work_required` calls this function to retrieve the first block of the difficulty interval:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [2](#0-1) 

For Dogecoin post-height 145,000, the protocol sets `difficulty_adjustment_interval = 1`:

```rust
let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
let difficulty_adjustment_interval = if new_difficulty_protocol { 1 } else { config.difficulty_adjustment_interval };
``` [3](#0-2) 

With interval = 1, `blocks_to_go_back = 1` and `height_first = prev_block_header.block_height - 1` for every block. This means every fork block's difficulty is computed using the **mainchain block's timestamp** at `prev_height - 1`, not the fork's ancestor's timestamp at that height.

The same structural flaw exists in `bitcoin.rs` at difficulty adjustment boundaries:

```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [4](#0-3) 

The fork block submission path in `submit_block_header` correctly resolves the fork's previous block via `get_prev_header` (which uses `prev_block_hash` to traverse the fork chain), but then passes control to `check_target` → `check_pow` → `get_next_work_required`, which breaks the fork-aware traversal by switching to the mainchain-indexed lookup. [5](#0-4) 

---

### Impact Explanation

The `expected_bits` computed for a fork block is wrong whenever the mainchain block at `height_first` has a different timestamp than the fork's actual ancestor at that height. Two consequences follow:

1. **Legitimate fork blocks are rejected.** A real Dogecoin fork relayed by an honest relayer carries the correct `bits` for the fork's actual difficulty. The contract computes a different `expected_bits` using the mainchain ancestor's timestamp and panics with `"bad-diffbits: incorrect proof of work"`. This breaks fork tracking and chain reorganization entirely for Dogecoin post-145,000.

2. **Adversarial fork blocks with reduced PoW can be accepted.** An attacker who controls the timestamps of submitted fork blocks can exploit the fact that `first_block_time` is taken from the mainchain. If the mainchain block at `height_first` has an earlier timestamp than the fork's ancestor at that height, the computed `modulated_timespan` is inflated, the difficulty target is raised (easier), and the attacker's fork blocks need less PoW to satisfy the check. A successful fraudulent fork promoted via `reorg_chain` corrupts the canonical chain, causing `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to return results against the wrong chain. [6](#0-5) 

---

### Likelihood Explanation

- **Dogecoin post-145,000:** High. The interval is 1, so `get_header_by_height` is called for every fork block beyond the first. Any fork submission triggers the bug.
- **Bitcoin:** Medium. The bug activates only when a fork spans a difficulty adjustment boundary (every 2,016 blocks), but this is a normal occurrence over time.
- **Entry path:** Any unprivileged NEAR caller can invoke `submit_blocks` (subject to the trusted-relayer check, but the relayer role is open to staking participants). The adversarial input is a crafted sequence of `BlockHeader` values with manipulated timestamps.

---

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` with an ancestor traversal that walks back through the fork's own chain using `get_prev_header`. Starting from `prev_block_header`, iterate backwards `blocks_to_go_back` steps using `prev_block_hash` links, which are stored in `headers_pool` for all submitted blocks (both mainchain and fork). This mirrors how `get_median_time_past` correctly traverses the ancestor chain:

```rust
// Instead of:
let first_block_time = blocks_getter.get_header_by_height(height_first).block_header.time;

// Use ancestor traversal:
let mut cursor = prev_block_header.clone();
for _ in 0..blocks_to_go_back {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time = cursor.block_header.time;
``` [7](#0-6) 

---

### Proof of Concept

**Dogecoin post-145,000 scenario:**

1. Initialize the contract with genesis at height 145,001.
2. Submit mainchain blocks M1 (height 145,002, timestamp `T_m1`), M2 (145,003, `T_m2`), M3 (145,004, `T_m3`).
3. Submit fork block F1 (height 145,002, `prev = genesis`, timestamp `T_f1` where `T_f1 >> T_m1`) — accepted as a fork.
4. Submit fork block F2 (height 145,003, `prev = F1`, timestamp `T_f2`).
   - Contract calls `get_next_work_required` with `prev = F1`, `height_first = 145,002`.
   - `get_header_by_height(145,002)` returns **M1** (mainchain), timestamp `T_m1`.
   - Difficulty is computed as `f(T_f2 - T_m1)` instead of the correct `f(T_f2 - T_f1)`.
   - Since `T_f1 >> T_m1`, the timespan `T_f2 - T_m1` is much larger than `T_f2 - T_f1`, inflating the target (lowering difficulty).
5. F2 with reduced PoW satisfies the inflated target and is accepted.
6. Repeat for F3, F4, … until fork chainwork exceeds mainchain, triggering `reorg_chain`.
7. The fraudulent fork becomes the canonical chain; subsequent `verify_transaction_inclusion` calls return results against the attacker-controlled chain. [8](#0-7) [9](#0-8)

    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))

pub fn get_median_time_past(
    block_header: ExtendedHeader,
    prev_block_getter: &impl BlocksGetter,
) -> u32 {
    use btc_types::network::MEDIAN_TIME_SPAN;

    let mut median_time = [0u32; MEDIAN_TIME_SPAN];
    let mut current_header = block_header;

    for slot in &mut median_time {
        *slot = current_header.block_header.time;
        current_header = prev_block_getter.get_prev_header(&current_header.block_header);
    }

    median_time.sort_unstable();
    median_time[median_time.len() / 2]
}
