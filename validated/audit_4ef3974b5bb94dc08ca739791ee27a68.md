Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any User to Bypass Per-Pool Swap Allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract, not the end user. If a pool admin allowlists the router (the natural action to enable router-based swaps), every unprivileged user can bypass the per-pool swap allowlist by routing through the router, as the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`.

## Finding Description
The call chain is as follows:

1. `MetricOmmSimpleRouter.exactInputSingle()` calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` directly, with no mechanism to forward the original `msg.sender` to the pool. [1](#0-0) 

2. `MetricOmmPool.swap()` passes `msg.sender` (the router address) as `sender` to `_beforeSwap`. [2](#0-1) 

3. `ExtensionCalling._beforeSwap()` forwards `sender` (the router) unchanged to every configured extension via `_callExtensionsInOrder`. [3](#0-2) 

4. `SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][router]`. The actual end user is never checked. [4](#0-3) 

The result is a permanent identity mismatch: once the router is allowlisted, the extension's check passes for any caller of the router, regardless of whether that caller is on the allowlist. There are no existing guards in the router, pool, or extension that recover the original end-user identity.

## Impact Explanation
A pool admin who wants to allow router-based swaps for their allowlisted users will naturally call `setAllowedToSwap(pool, address(router), true)`. Once the router is allowlisted, every user — including those explicitly excluded from the allowlist — can bypass the access-control gate by routing through `MetricOmmSimpleRouter`. The allowlist ceases to function as an access-control boundary for any router-mediated swap. Because the pool is oracle-driven, unauthorized swappers can interact at live oracle prices, draining LP-owned liquidity in directions the pool admin intended to restrict (e.g., blocking specific counterparties, complying with regulatory requirements, or protecting against oracle-latency exploitation by known actors). This constitutes broken core pool functionality causing potential loss of funds and an admin-boundary break where an unprivileged path bypasses a pool admin's configured access control.

## Likelihood Explanation
The scenario is reachable by any unprivileged user with no special setup beyond the pool admin having allowlisted the router — a natural and expected administrative action for any pool that intends to support the standard periphery. `MetricOmmSimpleRouter` is the canonical user-facing entry point. No malicious initial setup, non-standard tokens, or privileged access is required on the attacker's side. The condition (router allowlisted) is a prerequisite for the pool to be usable via the router at all, making this a near-universal precondition for any router-enabled allowlisted pool.

## Recommendation
The extension must receive the economic actor (the end user), not the immediate caller of `pool.swap()`. Recommended approaches:

1. **Pass the original user through the router**: Have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it. This requires a protocol-level convention for trusted routers.
2. **Pool-level originator field**: Add an `originator` field to the swap call that the router populates with its own `msg.sender`, and have the allowlist check that field. This mirrors how Uniswap v4 handles hook-visible originators and is the cleanest fix.
3. **Gate on `recipient` instead of `sender`**: If the pool's intent is to restrict who receives output tokens, check `recipient` (the second argument to `beforeSwap`). For exact-input single-hop swaps the user typically sets themselves as recipient.

## Proof of Concept
```solidity
// Pool is configured with SwapAllowlistExtension.
// Admin allowlists the router to support router-based swaps:
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Alice is NOT on the allowlist:
assertFalse(swapAllowlist.isAllowedToSwap(pool, alice));

// Alice bypasses the allowlist by going through the router:
vm.prank(alice);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: alice,
    zeroForOne: true,
    amountIn: 1000e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    tokenIn: token0,
    extensionData: ""
}));
// Swap succeeds — allowlist bypassed.
// The extension checked allowedSwapper[pool][router] == true, not allowedSwapper[pool][alice].
```

The `beforeSwap` hook receives `sender = address(router)`, which is allowlisted, so the check at `allowedSwapper[msg.sender][sender]` passes regardless of who Alice is. [5](#0-4)
