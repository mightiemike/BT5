### Title
`burnNlp` Uses Insufficient `MAINTENANCE` Health Check Instead of `INITIAL`, Allowing Collateral Extraction Below Initial Margin â€” (`core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp` function in `Clearinghouse.sol` enforces only a `MAINTENANCE` health check after burning NLP tokens, while the analogous collateral-exit path (`withdrawCollateral`) correctly enforces the stricter `INITIAL` health check. A user whose subaccount is under initial health but above maintenance health can burn NLP to extract quote collateral, bypassing the initial margin requirement entirely.

---

### Finding Description

In `Clearinghouse.sol`, `withdrawCollateral` enforces the stricter `INITIAL` health type for all regular (non-`X_ACCOUNT`) senders: [1](#0-0) 

```solidity
IProductEngine.HealthType healthType = sender == X_ACCOUNT
    ? IProductEngine.HealthType.PNL
    : IProductEngine.HealthType.INITIAL;
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
```

However, `burnNlp` only enforces the weaker `MAINTENANCE` health check: [2](#0-1) 

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
```

The asymmetry is confirmed by comparing with `mintNlp`, which correctly uses `INITIAL`: [3](#0-2) 

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

The `INITIAL` health type uses tighter risk weights (`longWeightInitialX18`, `shortWeightInitialX18`) than `MAINTENANCE`: [4](#0-3) 

A subaccount can legitimately be in the state `INITIAL < 0` and `MAINTENANCE >= 0` â€” this is the normal "under initial margin but not yet liquidatable" state. In this state:
- `withdrawCollateral` correctly **rejects** the withdrawal.
- `burnNlp` **accepts** the burn and transfers quote tokens back to the sender.

The burn flow:
1. NLP tokens are removed from the sender.
2. `quoteAmount = nlpAmount.mul(oraclePriceX18)` is computed.
3. A burn fee of `max(1e18, quoteAmount / 1000)` is deducted.
4. The remaining `quoteAmount` is credited to the sender's spot quote balance.
5. Only `MAINTENANCE >= 0` is checked â€” not `INITIAL >= 0`. [5](#0-4) 

This is the direct analog to the reported bug: just as `liquidationInitialAsk` was compared to a weaker bound (`params.amount`) instead of the correct bound (`onew = amount + accrued interest`), `burnNlp` is validated against a weaker health threshold (`MAINTENANCE`) instead of the correct threshold (`INITIAL`).

---

### Impact Explanation

A user holding NLP tokens whose subaccount is under initial health (but above maintenance) can call `burnNlp` to receive quote tokens back, effectively extracting collateral from a position that the protocol's risk model explicitly prohibits from withdrawing. This:

- Corrupts the initial margin accounting invariant: positions that should be locked at `INITIAL < 0` can shed collateral.
- Reduces the protocol's collateral buffer, increasing the risk of bad debt if prices move further against the position.
- The burn fee (â‰Ą 0.1%) does not prevent the bypass; it only slightly reduces the extracted amount.

The concrete corrupted state: a subaccount ends with `INITIAL < 0` and a reduced quote balance, a state that `withdrawCollateral` would never permit.

---

### Likelihood Explanation

Medium. The precondition â€” holding NLP tokens while being under initial health â€” is reachable through normal market activity (e.g., a user mints NLP, then their other positions lose value, pushing `INITIAL < 0` while `MAINTENANCE >= 0`). No privileged access, governance capture, or key compromise is required. The entry path is a standard user-submitted transaction through `Endpoint.sol` â†’ `Clearinghouse.burnNlp()`.

---

### Recommendation

Change the health check in `burnNlp` to use `INITIAL` health type, consistent with `withdrawCollateral` and `mintNlp`:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [6](#0-5) 

---

### Proof of Concept

1. User mints NLP via `mintNlp` â€” passes `INITIAL >= 0` check at that time.
2. User opens a leveraged spot borrow or perp position. Market moves adversely.
3. Subaccount reaches state: `INITIAL health = -X` (negative), `MAINTENANCE health = +Y` (positive). `withdrawCollateral` would revert at this point.
4. User submits a `BurnNlp` transaction through `Endpoint`.
5. `Clearinghouse.burnNlp` executes: burns NLP, credits `quoteAmount - burnFee` to the sender's quote balance.
6. Post-burn check: `getHealth(sender, MAINTENANCE) >= 0` â€” passes (assuming the burn fee does not push health below maintenance).
7. User has successfully extracted quote collateral from a subaccount that `withdrawCollateral` would have rejected, leaving the subaccount with `INITIAL < 0` and a lower collateral buffer. [7](#0-6) [6](#0-5)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
