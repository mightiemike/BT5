Audit Report

## Title
`DepositAllowlistExtension` Bypassed via Caller-Supplied `owner` in `addLiquidityExactShares` — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` gates deposits by checking the caller-supplied `owner` parameter rather than the actual initiating address. `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` and `addLiquidityWeighted` (owner-explicit overloads) accept any non-zero `owner` from the caller without verifying it matches `msg.sender`. An unprivileged attacker can supply any whitelisted address as `owner`, causing the allowlist check to pass while the attacker's own tokens fund the deposit, violating the pool admin's admission policy.

## Finding Description
`DepositAllowlistExtension.beforeAddLiquidity` ignores the `sender` parameter (unnamed, discarded) and gates exclusively on `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

`MetricOmmPool.addLiquidity` passes `msg.sender` (the `LiquidityAdder` contract) as `sender` and the caller-supplied `owner` verbatim to the extension hook: [2](#0-1) 

`ExtensionCalling._beforeAddLiquidity` forwards both to the extension unchanged: [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (owner-explicit overload) accepts any non-zero `owner` from the caller and passes it directly to the pool: [4](#0-3) 

`_validateOwner` only rejects `address(0)`, imposing no identity constraint: [5](#0-4) 

The same flaw exists in `addLiquidityWeighted(pool, owner, ...)`: [6](#0-5) 

**Exploit call chain:**
1. Attacker (not on allowlist) calls `LiquidityAdder.addLiquidityExactShares(pool, whitelistedAddr, salt, deltas, max0, max1, extData)`.
2. `LiquidityAdder` calls `pool.addLiquidity(whitelistedAddr, salt, deltas, KIND_PAY, extData)` — `msg.sender` to the pool is `LiquidityAdder`.
3. Pool calls `extension.beforeAddLiquidity(LiquidityAdder, whitelistedAddr, ...)`.
4. Extension evaluates `allowedDepositor[pool][whitelistedAddr]` → `true` → guard passes.
5. Callback pulls tokens from the attacker (`payer = msg.sender` stored in transient context); position is minted under `whitelistedAddr`.

The `sender` field seen by the extension is always the `LiquidityAdder` contract address, never the attacker, so no individual-user check on `sender` is possible either. The only actor the extension can meaningfully gate on is `owner`, and that is fully attacker-controlled.

## Impact Explanation
The pool admin's deposit allowlist is bypassed by an unprivileged path. This constitutes an **admin-boundary break**: factory/oracle role checks are bypassed by an unprivileged path. Concrete consequences include:
- Unauthorized liquidity enters a curated pool, violating the admission policy.
- LP dilution and price-impact griefing: attacker can deposit at any bin composition, shifting the pool cursor and diluting existing LP positions.
- Unsolicited positions minted under whitelisted addresses may cause salt collisions or interfere with those addresses' own position management.

## Likelihood Explanation
- The `addLiquidityExactShares(pool, owner, ...)` overload is a standard public periphery entry point requiring no special role.
- Whitelisted addresses are discoverable from `AllowedToDepositSet` events emitted by `setAllowedToDeposit`.
- The attacker only needs token approval on the `LiquidityAdder` and knowledge of one whitelisted address.
- No flash loan, oracle manipulation, or privileged access is required. The attack is repeatable at will.

## Recommendation
**Option A (minimal — periphery fix):** In `MetricOmmPoolLiquidityAdder._validateOwner`, require `owner == msg.sender` so callers can only create positions for themselves:

```solidity
function _validateOwner(address owner) internal view {
    if (owner == address(0)) revert InvalidPositionOwner();
    if (owner != msg.sender) revert InvalidPositionOwner();
}
```

**Option B (extension-side):** Change `DepositAllowlistExtension.beforeAddLiquidity` to check `sender` instead of `owner`. Pool admins would then allowlist the `LiquidityAdder` contract rather than individual users — a coarser but unspoofable policy.

**Option C (both):** Apply both fixes for defense in depth.

## Proof of Concept
```solidity
// Setup: pool with DepositAllowlistExtension; alice is whitelisted, attacker is not.
address alice = makeAddr("alice");
address attacker = makeAddr("attacker");

extension.setAllowedToDeposit(address(pool), alice, true);
// attacker is NOT on the allowlist

token0.mint(attacker, 1_000_000);
token1.mint(attacker, 1_000_000);
vm.startPrank(attacker);
token0.approve(address(liquidityAdder), type(uint256).max);
token1.approve(address(liquidityAdder), type(uint256).max);

// Attacker supplies alice as owner — bypasses the allowlist check
LiquidityDelta memory delta = LiquidityDelta({binIdxs: ..., shares: ...});
liquidityAdder.addLiquidityExactShares(
    address(pool),
    alice,       // whitelisted owner — guard passes
    0,           // salt
    delta,
    1_000_000,
    1_000_000,
    ""
);
vm.stopPrank();

// Attacker's tokens are now in the pool under alice's position.
// DepositAllowlistExtension did not block the unauthorized deposit.
```
