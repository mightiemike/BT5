### Title
Zero `confirmations` Bypasses Confirmation Security in Transaction Inclusion Verification — (`contract/src/lib.rs`)

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` parameter with no minimum value enforcement. Passing `confirmations = 0` causes the confirmation depth check to trivially pass for any mainchain block, returning `true` for a transaction with zero confirmations. This is a direct analog to the reported pattern: a user-controlled resource/security parameter has only a trivially weak bound (here: none at all, vs. `> 0` in the original), allowing the parameter to be set to a value that defeats the purpose of the check.

### Finding Description

`verify_transaction_inclusion` enforces only an upper bound on `confirmations`:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

No lower bound is enforced. The subsequent depth check is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

When `confirmations = 0`, the right-hand side is `0`, and any `u64` value satisfies `>= 0`. The check always passes regardless of how recently the block was included. The function then proceeds to Merkle proof verification and returns `true` for a transaction with zero confirmations.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase proof check, so it inherits the same flaw:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

The `confirmations` field is part of `ProofArgs` and `ProofArgsV2`, both of which are fully caller-controlled: [4](#0-3) 

### Impact Explanation

The `confirmations` parameter exists specifically to protect downstream consumers against Bitcoin chain reorganizations. A downstream NEAR contract that gates fund releases on a `true` result from `verify_transaction_inclusion_v2` can be called by an attacker who:

1. Submits a Bitcoin transaction and waits for it to appear in one block.
2. Calls `verify_transaction_inclusion_v2` with `confirmations = 0`.
3. Receives `true` immediately, triggering fund release on NEAR.
4. If the Bitcoin block is subsequently reorganized away, the attacker has received funds for a transaction that no longer exists on the canonical chain.

The corrupted protocol invariant is the **proof result**: the function returns `true` for a transaction that has not met the required confirmation depth, making the SPV proof semantically invalid.

### Likelihood Explanation

The entry path requires no privilege. Any unprivileged NEAR account can call `verify_transaction_inclusion_v2` directly with `confirmations = 0`. The function is public and gated only by the `#[pause]` attribute, which is inactive in normal operation. The parameter is a plain `u64` with no on-chain enforcement of a minimum.

### Recommendation

Enforce a protocol-defined minimum confirmation count. At minimum, reject `confirmations = 0`:

```rust
require!(args.confirmations > 0, "confirmations must be at least 1");
```

A stronger fix would enforce a chain-specific minimum (e.g., 6 for Bitcoin mainnet) as a contract constant, preventing callers from accidentally or maliciously bypassing reorganization protection.

### Proof of Concept

```rust
// Attacker calls verify_transaction_inclusion_v2 with confirmations = 0
// after a transaction appears in the most recent block (tip).
// The depth check: (tip_height - tip_height) + 1 = 1 >= 0 → passes.
// Merkle proof is valid → function returns true.
// Downstream contract releases funds.
let args = ProofArgsV2 {
    tx_id: <valid_tx_hash>,
    tx_block_blockhash: <tip_block_hash>,  // most recent block
    tx_index: 0,
    merkle_proof: <valid_proof>,
    coinbase_tx_id: <coinbase_hash>,
    coinbase_merkle_proof: <valid_coinbase_proof>,
    confirmations: 0,  // bypasses all confirmation depth enforcement
};
// Returns true despite zero confirmations.
contract.verify_transaction_inclusion_v2(args);
```
