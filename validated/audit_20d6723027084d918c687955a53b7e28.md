### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass the Swap Allowlist via the Router - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

The `SwapAllowlistExtension.beforeSwap` hook receives `sender` as the pool's `msg.sender` — which is the router contract when a user routes through `MetricOmmSimpleRouter` — rather than the originating EOA. If the pool admin allowlists the router address (a natural step to permit router-mediated swaps for legitimate users), the allowlist is silently bypassed for **every** user, because the hook checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`.

---

### Finding Description

**Hook argument binding in `ExtensionCalling._beforeSwap`:**

The pool passes `msg.sender` as the `sender` argument to every extension:

```solidity
// MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    zeroForOne,
    amountSpecified,
    priceLimitX64,
    packedSlot0Initial,
    bidPriceX64,
    askPriceX64,
    extensionData
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension:

```solidity
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, amountSpecified, ...)
    )
);
``` [2](#0-1) 

**The allowlist check in `SwapAllowlistExtension.beforeSwap`:**

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [3](#0-2) 

Inside this call:
- `msg.sender` = the pool (the extension's caller) — used as the pool key ✓
- `sender` = whoever called `pool.swap()` — **the router**, not the end user, when routing through `MetricOmmSimpleRouter`

So the effective check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_EOA]`.

**The bypass path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict trading to a curated set of addresses.
2. Admin calls `setAllowedToSwap(pool, router, true)` so that allowlisted users can conveniently use the router.
3. Any unprivileged EOA calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting that pool.
4. The router calls `pool.swap(...)` with `msg.sender = router`.
5. The extension sees `sender = router`, finds `allowedSwapper[pool][router] == true`, and passes — the actual caller is never checked.
6. The unauthorized user's swap executes in full.

The allowlist is completely neutralized for any user who routes through the public router.

---

### Impact Explanation

The `SwapAllowlistExtension` is the sole on-chain mechanism for restricting who may trade in a pool. Bypassing it allows any unprivileged address to execute swaps in pools that the pool admin explicitly intended to gate. Depending on the pool's purpose (e.g., institutional-only liquidity, regulatory compliance, risk-controlled market making), this can result in:

- Unauthorized actors draining LP liquidity at oracle-anchored prices.
- LP positions suffering losses from trades the pool was designed to block.
- Protocol-level access control invariant broken by a public, zero-privilege path (the router).

This matches the allowed impact gate: **admin-boundary break — factory/oracle role checks bypassed by an unprivileged path**.

---

### Likelihood Explanation

- The router is a public, permissionless contract that any user can call.
- The bypass requires no special privileges, no malicious setup, and no non-standard tokens.
- Any pool that (a) uses `SwapAllowlistExtension` and (b) allowlists the router — a natural and expected configuration — is fully exposed.
- The attacker needs only to call `MetricOmmSimpleRouter.exactInputSingle` with the target pool address.

---

### Recommendation

The `SwapAllowlistExtension` should gate the **originating user**, not the immediate caller of `pool.swap()`. Two complementary fixes:

1. **Pass the true initiator through `extensionData`**: The router encodes the original `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a trusted router or a signed payload.

2. **Check `tx.origin` as a fallback** (only acceptable if the pool is not used in contract-to-contract flows): replace `sender` with `tx.origin` inside the extension.

3. **Preferred — check both**: Require that `sender` (the direct caller) is either an allowlisted EOA **or** a trusted router, and if it is a trusted router, additionally verify the end-user identity from `extensionData`.

The `DepositAllowlistExtension` should be audited for the same pattern on the `beforeAddLiquidity` path, where `owner`/`sender` separation via `MetricOmmPoolLiquidityAdder` may introduce an analogous bypass.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Pool admin calls setAllowedToSwap(pool, router, true).
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true).
  - Pool admin does NOT call setAllowAllSwappers(pool, true).

Attack:
  attacker (unlisted EOA) calls:
    MetricOmmSimpleRouter.exactInputSingle(ExactInputSingleParams{
        pool:    <restricted pool>,
        tokenIn: token1,
        ...
    });

  Router calls pool.swap(recipient, zeroForOne, amount, priceLimit, bid, ask, extensionData).
  pool.swap() → msg.sender = router.
  _beforeSwap(sender=router, ...) → extension.beforeSwap(sender=router, ...).
  Extension checks: allowedSwapper[pool][router] == true → passes.
  Swap executes for the attacker.

Expected: revert NotAllowedToSwap().
Actual:   swap succeeds; attacker receives tokens from a pool they are not authorized to trade in.
```
