### Title
Sequencer-Supplied `oraclePriceX18` Outside User Signature Scope Enables Unfavorable MintNlp/BurnNlp Execution Without Slippage Protection â€” (File: `core/contracts/interfaces/IEndpoint.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The user-signed `MintNlp` and `BurnNlp` structs contain no minimum output bounds and no expiration deadline. The `oraclePriceX18` that governs how many NLP tokens a user receives (mint) or how much quote a user receives (burn) lives in the outer sequencer-supplied wrapper struct and is not covered by the user's EIP-712 signature. The sequencer can therefore supply any price at execution time, and the transaction cannot revert on the user's behalf because no user-defined floor or ceiling exists in the signed payload.

---

### Finding Description

**Signed structs lack output bounds.**

The inner structs that the user actually signs are:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint64 nonce;
}
``` [1](#0-0) 

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64 nonce;
}
``` [2](#0-1) 

Neither struct carries a `minNlpAmount`, `minQuoteAmount`, or `expiration` field.

**`oraclePriceX18` is outside the user's signature scope.**

The outer wrappers that the sequencer constructs and submits are:

```solidity
struct SignedMintNlp {
    MintNlp tx;
    bytes signature;
    int128 oraclePriceX18;       // sequencer-supplied, not signed by user
    int128[] nlpPoolRebalanceX18;
}
``` [3](#0-2) 

```solidity
struct SignedBurnNlp {
    BurnNlp tx;
    bytes signature;
    int128 oraclePriceX18;       // sequencer-supplied, not signed by user
    int128[] nlpPoolRebalanceX18;
}
``` [4](#0-3) 

`validateSignedTx` computes the EIP-712 digest over `transaction[1:]`, which is the ABI-encoded `SignedMintNlp`/`SignedBurnNlp`. Because the `signature` field is itself part of that encoding, the verifier must extract only the inner `MintNlp`/`BurnNlp` struct to avoid a circular hash â€” meaning `oraclePriceX18` is not covered by the user's signature. [5](#0-4) 

**Execution uses the unchecked price directly.**

In `mintNlp`, the output NLP amount is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [6](#0-5) 

In `burnNlp`, the output quote amount is computed as:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [7](#0-6) 

Neither function checks a user-defined minimum output. The sequencer injects `oraclePriceX18` at submission time: [8](#0-7)

### Impact Explanation

**MintNlp:** A user who signs `quoteAmount = 1000 USDC` when the NLP oracle price is `$1` expects `1000` NLP tokens. If the sequencer submits the transaction with `oraclePriceX18 = $2`, the user receives only `500` NLP tokens â€” a 50% shortfall â€” with no on-chain mechanism to revert.

**BurnNlp:** A user who signs `nlpAmount = 1000` when the NLP oracle price is `$1` expects `~999 USDC` (after the 0.1% burn fee). If the sequencer submits with `oraclePriceX18 = $0.50`, the user receives `~499 USDC` â€” a ~50% shortfall â€” with no on-chain mechanism to revert.

The corrupted state is the user's `NLP_PRODUCT_ID` balance (mint) or `QUOTE_PRODUCT_ID` balance (burn) in the `SpotEngine`. [9](#0-8) [10](#0-9) 

---

### Likelihood Explanation

The sequencer is the sole entity that can submit fast-path transactions and is the sole entity that sets `oraclePriceX18` via `UpdatePrice`. Two realistic scenarios exist:

1. **Legitimate price drift:** The NLP oracle price is updated between the moment the user signs and the moment the sequencer batches the transaction. Because the user's signed payload has no expiration and no minimum output, even an honest sequencer executing at the current oracle price can deliver a materially worse fill than the user intended.

2. **Sequencer acting in self-interest:** Because `oraclePriceX18` is not covered by the user's signature, a sequencer that has any incentive to do so (e.g., it also holds NLP or quote positions) can supply a price that extracts value from the user's mint or burn without triggering any on-chain revert.

---

### Recommendation

**Short term:** Add `minNlpAmount` to `MintNlp` and `minQuoteAmount` to `BurnNlp` as user-signed fields. Enforce them in `mintNlp` and `burnNlp` respectively:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint128 minNlpAmount;   // new: revert if nlpAmount < this
    uint64 nonce;
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint128 minQuoteAmount; // new: revert if quoteAmount < this
    uint64 nonce;
}
```

Add an `expiration` field analogous to the `Order` struct to bound how long a signed intent remains valid.

**Long term:** Add tests that simulate oracle price movement between signing and execution and assert that mint/burn transactions revert when the computed output falls below the user-specified minimum.

---

### Proof of Concept

1. Alice signs `MintNlp{sender=alice, quoteAmount=10_000e18, nonce=5}` when the NLP oracle price is `$1.00`.
2. Before the sequencer batches Alice's transaction, the sequencer submits an `UpdatePrice` transaction setting `oraclePriceX18 = $2.00`.
3. The sequencer then submits Alice's `SignedMintNlp` with `oraclePriceX18 = $2.00`.
4. `mintNlp` computes `nlpAmount = 10_000e18 / 2e18 = 5_000e18`.
5. Alice's `NLP_PRODUCT_ID` balance increases by `5_000` NLP instead of the `10_000` she expected.
6. No check in `mintNlp` compares the computed `nlpAmount` against any user-defined floor; the transaction succeeds and Alice has no recourse.
