Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of end-user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swap access by checking the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap` call. When `MetricOmmSimpleRouter` is used, `msg.sender` to the pool is the router contract address — not the end-user. Any pool admin who allowlists the router (required for allowlisted users to use the standard periphery) inadvertently grants every user the ability to bypass the allowlist by routing through the router.

## Finding Description

**Root cause — wrong actor checked:**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
  }
  return IMetricOmmExtensions.beforeSwap.selector;
}
```

`MetricOmmPool.swap` passes `msg.sender` of the `swap` call as the first argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← router address when routing through MetricOmmSimpleRouter
  recipient, ...
);
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making itself `msg.sender` to the pool. The actual end-user (`msg.sender` of the router call) is only stored in transient storage for payment callback purposes and is never forwarded to the pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData   // end-user identity absent
  );
```

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` correctly ignores `sender` (the periphery contract address) and checks `owner` — the position owner explicitly passed as a dedicated parameter through the call chain:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
```

`swap` has no equivalent explicit `swapper` parameter — only `msg.sender` (the router) is available to the pool and extension.

**Exploit path:**

1. Pool admin deploys a curated pool with `SwapAllowlistExtension`.
2. Pool admin allowlists user A directly: `setAllowedToSwap(pool, userA, true)`.
3. Pool admin allowlists the router so user A can use the standard periphery: `setAllowedToSwap(pool, router, true)`.
4. User C (not allowlisted) calls `router.exactInputSingle(...)` targeting the pool.
5. The router calls `pool.swap(...)` — `msg.sender` to the pool is the router address.
6. `beforeSwap` checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. User C executes a real swap on a pool they were not authorized to access.

Existing guards are insufficient: the extension has no mechanism to distinguish which end-user initiated the router call, and the pool interface provides no dedicated `swapper` parameter to carry that identity.

## Impact Explanation

Any unprivileged user can execute swaps on pools protected by `SwapAllowlistExtension` by routing through `MetricOmmSimpleRouter`, provided the router is in the allowlist. This completely defeats the access control purpose of the extension — KYC gates, institutional-only pools, or any allowlist-gated trading venue built on this extension are rendered ineffective. The bypass results in unauthorized users executing real token swaps, moving funds out of the pool at oracle-derived prices. This is a direct, fund-impacting action meeting the admin-boundary break and broken core pool functionality criteria.

## Likelihood Explanation

Likelihood is high. `MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who deploys a curated pool and wants allowlisted users to use the standard periphery must add the router to the allowlist — this is the obvious and expected operational step. Once done, the bypass is trivially reachable by any user with no special privileges, no malicious setup, and no non-standard tokens. The same issue applies to `exactOutputSingle` and multi-hop `exactInput`/`exactOutput` paths in the router.

## Recommendation

The extension must gate the economically relevant actor — the end-user — not the immediate caller of `pool.swap`. Two approaches:

1. **Pass the end-user through `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Requires coordination between router and extension but preserves the current pool interface.

2. **Add a `swapper` parameter to `swap`:** Analogous to how `addLiquidity` has an explicit `owner` parameter, `swap` could accept an explicit `swapper` address that the pool passes to `_beforeSwap` instead of `msg.sender`. The router would pass `msg.sender` (the end-user) as this parameter. This mirrors the design that makes `DepositAllowlistExtension` correct and is the cleaner fix.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension, router allowlisted, user C not allowlisted
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
ext.setAllowedToSwap(address(pool), address(router), true); // required for periphery use
ext.setAllowedToSwap(address(pool), userA, true);           // userA is allowlisted
// userC is NOT allowlisted

// Direct call by userC — correctly blocked
vm.prank(userC);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(userC, false, 1000, type(uint128).max, "", "");

// Router call by userC — BYPASSES the allowlist
// sender seen by extension = router address = allowlisted
vm.prank(userC);
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: userC,
    zeroForOne: false,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: type(uint128).max,
    deadline: block.timestamp,
    extensionData: ""
}));
// Swap succeeds — userC traded on a pool they were not authorized to access
