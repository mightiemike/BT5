### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension.beforeAddLiquidity` hook ignores the `sender` argument (the actual caller who provides tokens) and instead validates the `owner` argument (the LP-share recipient). Because `owner` is a free caller-controlled parameter in `MetricOmmPool.addLiquidity`, any address can bypass the deposit allowlist by supplying an already-allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` calls the extension hook with `msg.sender` as `sender` and the caller-supplied `owner` as the LP-share recipient: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values to the extension: [2](#0-1) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is explicitly unnamed and silently discarded. The allowlist check is performed only on `owner`: [3](#0-2) 

Because `owner` is a free parameter chosen by the caller, any unauthorized address can call:

```solidity
pool.addLiquidity(
    allowlistedAddress,   // owner — passes the allowlist check
    salt,
    deltas,
    callbackData,
    extensionData
);
```

The extension sees `allowedDepositor[pool][allowlistedAddress] == true` and does not revert. The unauthorized caller provides the tokens via the swap callback; the allowlisted address receives the LP shares. The `sender` who actually transfers funds is never validated.

This is the direct analog of the UpDown boundary-condition bug: just as the `else` branch in `finalizeGame()` incorrectly captured the `price == startingPrice` case and assigned it to down-betters, here the wrong party (`owner` instead of `sender`) is checked in the guard, giving unauthorized depositors a trivial bypass path.

---

### Impact Explanation

The deposit allowlist is a pool-level access-control mechanism set by the pool admin to restrict who may add liquidity (e.g., KYC-gated pools, institutional-only pools, or pools that must limit LP composition for risk reasons). Bypassing it:

1. **Breaks the pool's core access-control invariant** — unauthorized parties become LPs in pools explicitly configured to exclude them.
2. **Enables liquidity manipulation** — an unauthorized actor can add liquidity in specific bins to skew the pool's marginal price curve, then exploit the distortion via a swap (if they are also an allowed swapper, or if swap allowlisting is absent).
3. **Griefs allowlisted addresses** — the allowlisted `owner` receives LP shares they did not request; they bear the position's market risk until they choose to remove it.

---

### Likelihood Explanation

The bypass requires no special privilege, no flash loan, and no complex setup. Any address can call `addLiquidity` with `owner` set to any allowlisted address. The allowlisted address need not cooperate. Likelihood is **High**.

---

### Recommendation

Check `sender` (the actual depositor) instead of `owner` (the LP-share recipient):

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(
    address sender,   // ← name and use this parameter
    address,          // owner — not relevant for access control
    uint80,
    LiquidityDelta calldata,
    bytes calldata
) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

### Proof of Concept

```solidity
// Assume:
//   pool has DepositAllowlistExtension configured
//   allowedDepositor[pool][alice] = true
//   bob is NOT allowlisted

// Bob bypasses the allowlist:
pool.addLiquidity(
    alice,          // owner — allowlisted, passes the check
    0,              // salt
    deltas,         // desired liquidity
    callbackData,   // bob provides tokens here via metricOmmSwapCallback
    extensionData
);
// Result: bob's tokens enter the pool; alice receives LP shares.
// The extension never checked bob's address.
``` [3](#0-2) [4](#0-3)
