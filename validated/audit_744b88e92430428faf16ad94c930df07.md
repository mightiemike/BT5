### Title
Missing Upper-Bound Validation on `min_gas_price_percentage` Enables Gateway DoS for V3 Transactions - (File: `crates/apollo_gateway_config/src/config.rs`)

### Summary
`StatefulTransactionValidatorConfig.min_gas_price_percentage` is a `u8` field (range 0–255) with **no `#[validate]` attribute**. When set above 100, the gateway computes a threshold exceeding the previous block's L2 gas price and rejects every incoming V3 (`AllResources`) transaction with `GAS_PRICE_TOO_LOW`, while V1/V2 (`L1Gas`) transactions are silently unaffected — a split-path DoS that mirrors the external report's pattern exactly.

### Finding Description

`StatefulTransactionValidatorConfig` derives `Validate` but places no constraint on `min_gas_price_percentage`:

```rust
// crates/apollo_gateway_config/src/config.rs  line 276-287
#[derive(Clone, Debug, Serialize, Deserialize, Validate, PartialEq)]
pub struct StatefulTransactionValidatorConfig {
    pub validate_resource_bounds: bool,
    ...
    pub min_gas_price_percentage: u8, // E.g., 80 to require 80% of threshold.
}
```

The default is 100 and the production config ships with 100:

```json
// crates/apollo_deployments/resources/app_configs/gateway_config.json  line 19
"gateway_config.static_config.stateful_tx_validator_config.min_gas_price_percentage": 100
```

The value is consumed in `validate_tx_l2_gas_price_within_threshold`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 367-371
let gas_price_threshold_multiplier =
    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
```

With `min_gas_price_percentage = 200`, `threshold = 2 × previous_block_l2_gas_price`. Every V3 transaction whose `max_price_per_unit` is at or near the current market price is rejected. The V1/V2 branch is entirely skipped:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 385-387
ValidResourceBounds::L1Gas(_) => {
    // No validation required for legacy transactions.
}
```

By contrast, the analogous `fee_escalation_percentage` in `MempoolStaticConfig` is correctly guarded:

```rust
// crates/apollo_mempool_config/src/config.rs  line 65-66
#[validate(range(min = 1, max = 100))]
pub fee_escalation_percentage: u8,
```

and its test explicitly rejects zero:

```rust
// crates/apollo_mempool_config/src/config_test.rs  lines 11-13
fn zero_fee_escalation_percentage_fails_validation() {
    let static_config = MempoolStaticConfig { fee_escalation_percentage: 0, ..Default::default() };
    assert!(static_config.validate().is_err());
}
```

No equivalent test or attribute exists for `min_gas_price_percentage`.

### Impact Explanation

When `min_gas_price_percentage > 100` and `validate_resource_bounds = true` (both the default and the production value), the gateway rejects every V3 (`AllResources`) transaction submitted to the sequencer. V1/V2 transactions continue to be admitted. This is a partial but severe DoS of transaction ingress: all modern Starknet transactions (which use `AllResources`) are blocked at the gateway before reaching the mempool or blockifier.

Impact: **High — Mempool/gateway admission rejects valid transactions before sequencing.**

### Likelihood Explanation

Requires an operator to set `min_gas_price_percentage` to a value above 100 in the node configuration. This is the same class of operator mistake as the external report (owner setting `basisPoints` to 10000). The field is a plain `u8` with no schema-level constraint, so a misconfiguration (e.g., entering `200` intending "200% of minimum" rather than "require 200% of previous price") is plausible. The production config schema entry carries no documented upper bound:

```json
// crates/apollo_node/resources/config_schema.json  lines 3112-3115
"gateway_config.static_config.stateful_tx_validator_config.min_gas_price_percentage": {
    "description": "Minimum gas price as percentage of threshold to accept transactions.",
    "value": 100
}
```

### Recommendation

Add an upper-bound validator to `min_gas_price_percentage` in `StatefulTransactionValidatorConfig`, mirroring the pattern already used for `fee_escalation_percentage`:

```rust
// crates/apollo_gateway_config/src/config.rs
#[validate(range(max = 100))]
pub min_gas_price_percentage: u8,
```

(No `min` constraint is needed because `0` is intentionally valid — it disables the threshold check, as confirmed by the existing test case `gas_price_check_disabled_when_percentage_zero_pass`.)

Add a corresponding config-validation test in `apollo_gateway_config`:

```rust
fn above_100_min_gas_price_percentage_fails_validation() {
    let config = StatefulTransactionValidatorConfig {
        min_gas_price_percentage: 101,
        ..Default::default()
    };
    assert!(config.validate().is_err());
}
```

### Proof of Concept

1. Set `gateway_config.static_config.stateful_tx_validator_config.min_gas_price_percentage` to `200` in the node config (a plausible operator mistake meaning "require 200% of the minimum").
2. Start the sequencer node with `validate_resource_bounds = true` (the default).
3. Submit any V3 (`AllResources`) transaction whose `max_price_per_unit` equals the current L2 gas price (e.g., `100`).
4. The gateway computes `threshold = (200/100) × 100 = 200` and rejects the transaction with `GAS_PRICE_TOO_LOW: Transaction L2 gas price 100 is below the required threshold 200`.
5. Submit a legacy V1 (`L1Gas`) transaction — it is admitted without any gas-price check, confirming the split-path behavior. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
