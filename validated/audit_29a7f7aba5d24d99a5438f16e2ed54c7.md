### Title
Difficulty Retarget Uses Mainchain-Only Lookup for Fork Block Validation, Allowing Incorrect Difficulty Acceptance - (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

The `get_next_work_required` function in all three non-Zcash chain modules (Bitcoin, Litecoin, Dogecoin) fetches the retarget-window boundary block via `blocks_getter.get_header_by_height()`. That helper is implemented exclusively against `mainchain_height_to_header` — it never traverses fork-chain ancestry. When a fork block at or past a retarget boundary is validated, the contract silently substitutes the mainchain block at the boundary height for the fork's true ancestor, producing a wrong expected-`bits` value and accepting fork blocks whose actual difficulty is incorrect.

---

### Finding Description

`get_header_by_height` is the only way the retarget code reaches the boundary block:

```rust
// lib.rs – the sole implementation of BlocksGetter::get_header_by_height
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← mainchain index only
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

Fork blocks are stored via `store_fork_header`, which inserts only into `headers_pool` — **not** into `mainchain_height_to_header`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [2](#0-1) 

Each chain module calls `get_header_by_height` at the retarget boundary:

``` [3](#0-2)

    .get_header_by_height(height_first)
    .block_header
    .time;
``` [5](#0-4) 

Suppose a fork diverges from the mainchain at height `H`. Fork blocks at heights `H+1`, `H+2`, … are stored only in `headers_pool`. When the contract validates a fork block whose retarget window boundary (`first_block_height`) falls at or above `H`, `get_header_by_height(first_block_height)` returns the **mainchain** block at that height — a completely different block from the fork's true ancestor. The resulting `first_block_time` is wrong, the computed timespan is wrong, and therefore the expected `bits` value is wrong. The contract then enforces this wrong expected value against the submitted block's `bits` field.

The desynchronization is structurally identical to the OlympusPrice analog: a cumulative/windowed state value (the retarget boundary block) is fetched from a shared pool that reflects one chain's history, but is used to validate a block on a different chain, causing the computed reference value to be incorrect.

---

### Impact Explanation

An attacker who controls a fork can craft fork blocks whose `bits` field matches the **mainchain-derived** expected difficulty rather than the correct fork-derived difficulty. Depending on the timestamp difference between the mainchain boundary block and the fork's true ancestor at that height, the attacker can:

- Submit fork blocks with **artificially easy difficulty** (if the mainchain boundary block is older than the fork's ancestor, the computed timespan is larger, lowering the required target).
- Accumulate chainwork faster than honest fork blocks would allow.
- Trigger a chain reorganization (`reorg_chain`) that promotes the attacker's fork to the main chain, corrupting `mainchain_tip_blockhash` and all downstream SPV proofs returned by `verify_transaction_inclusion` / `verify_transaction_inclusion_v2`. [6](#0-5) 

---

### Likelihood Explanation

**Dogecoin** is the most immediately exploitable: after block 145,000 the difficulty adjusts every block (`difficulty_adjustment_interval = 1`), so `height_first = prev_block_header.block_height - 1`. The bug activates on the **third** fork block submitted (when `height_first` equals the height of the first fork block, which is not in the mainchain index). Any unprivileged NEAR caller invoking `submit_blocks` with a two-block-deep Dogecoin fork triggers the wrong lookup. [7](#0-6) 

**Bitcoin and Litecoin** require a fork at least 2016 blocks deep before the retarget window boundary falls inside the fork, making exploitation harder but not impossible for a well-resourced attacker. [3](#0-2) [4](#0-3) 

---

### Recommendation

Replace `get_header_by_height` (mainchain-only) with a fork-aware ancestor traversal that walks back through `prev_block_hash` links from `prev_block_header` for exactly `blocks_to_go_back` steps — analogous to how `zcash_get_next_work_required` already correctly uses `prev_block_getter.get_prev_header()` in a loop:

```rust
// zcash.rs – correct pattern: follows prev_block_hash, not mainchain index
current_header = prev_block_getter.get_prev_header(&current_header.block_header);
``` [8](#0-7) 

For Bitcoin/Litecoin/Dogecoin, replace the `get_header_by_height` call with a loop that walks `blocks_to_go_back` steps via `get_prev_header` starting from `prev_block_header`. This ensures the boundary block is always the fork's true ancestor, regardless of what the mainchain index contains at that height.

---

### Proof of Concept

1. Deploy the contract in Dogecoin mode (post-block-145000 state, `skip_pow_verification = false`).
2. Establish a mainchain with blocks at heights `H`, `H+1`, `H+2` where the mainchain block at `H+1` has timestamp `T_main`.
3. Submit a fork block `F1` at height `H+1` (diverges from mainchain at `H`). It is stored via `store_fork_header` — **not** inserted into `mainchain_height_to_header`.
4. Submit a fork block `F2` at height `H+2` with parent `F1`. The contract calls `get_next_work_required` with `height_first = H+1`. `get_header_by_height(H+1)` returns the **mainchain** block at `H+1` (timestamp `T_main`), not `F1` (timestamp `T_fork`).
5. The expected `bits` for `F2` is computed from `T_main` instead of `T_fork`. If `T_main < T_fork`, the computed timespan is larger, the expected target is easier, and `F2` can carry a lower difficulty than the fork's true history requires.
6. `F2` passes `check_target` with the attacker-chosen easy `bits`. If `F2`'s chainwork exceeds the mainchain tip's chainwork, `reorg_chain` is triggered, corrupting the canonical chain state. [9](#0-8) [10](#0-9)

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
```
