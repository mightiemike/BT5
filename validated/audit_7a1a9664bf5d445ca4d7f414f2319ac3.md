### Title
Linked Signer Can Unilaterally Reassign Itself to an Unauthorized Address — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-path `LinkSigner` transaction in `EndpointTx.processTransactionImpl` accepts signatures from the **current linked signer** (via `allowLinkedSigner = true`), allowing the linked signer to overwrite `linkedSigners[sender]` with any arbitrary address — including one the account owner never authorized. There is no check that the new signer value differs from the stored one, and no restriction that only the account owner may perform this state mutation.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch decodes a `SignedLinkSigner` transaction and calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` passes `getLinkedSigner(sender)` as the permitted alternate signer to `verifier.validateSignature`: [2](#0-1) 

This means the signature check succeeds if **either** the account owner **or** the current linked signer signs the transaction. After validation, `linkedSigners[signedTx.tx.sender]` is unconditionally overwritten with the new address: [3](#0-2) 

There is no guard requiring the new signer to differ from the stored one, and no guard requiring the transaction to be signed by the account owner rather than the linked signer. The `linkedSigners` mapping is stored in `EndpointStorage`: [4](#0-3) 

By contrast, the slow-mode `LinkSigner` path correctly enforces that only the account owner can submit this transaction via `validateSender(txn.sender, sender)`: [5](#0-4) 

The asymmetry means the fast path (sequencer-submitted) has a weaker authorization model than the slow path (user-submitted).

---

### Impact Explanation

A linked signer can sign and submit a `LinkSigner` transaction that replaces itself with any address — including a malicious third party. The account owner authorized delegation to a specific address; they did not authorize that address to further delegate to arbitrary parties. After the reassignment:

- The new linked signer can sign `WithdrawCollateral`, `TransferQuote`, `MintNlp`, `BurnNlp`, and all other transactions that accept `allowLinkedSigner = true`.
- The account owner's collateral and positions are exposed to an address they never explicitly trusted.
- The linked signer also consumes the account owner's nonce (`nonces[address(uint160(bytes20(sender)))]`) during the attack, which is the nonce for the account owner's address.

The corrupted state is `linkedSigners[subaccount]` — a mapping that controls which external address may sign on behalf of a subaccount.

---

### Likelihood Explanation

Linked signers are commonly automated bots, third-party trading services, or hot wallets. Any of these can be compromised or act maliciously. The attack requires only that the current linked signer constructs and signs a valid `SignedLinkSigner` transaction — a straightforward on-chain operation requiring no special privileges beyond what the linked signer already holds. The account owner's current nonce is publicly readable via `getNonce`.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the fast-path `LinkSigner` transaction in `processTransactionImpl`. Only the account owner should be permitted to change the linked signer:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only account owner may change linked signer
);
```

Additionally, consider adding a check that the new signer value differs from the currently stored value to prevent no-op nonce consumption.

---

### Proof of Concept

1. Alice (account owner, subaccount `0xAlice...`) sets Bob (`0xBob`) as her linked signer via a slow-mode `LinkSigner` transaction. `linkedSigners[0xAlice...] = 0xBob`.
2. Bob constructs a `SignedLinkSigner` struct: `sender = 0xAlice...`, `signer = 0xCharlie`, `nonce = getNonce(address(0xAlice...))`.
3. Bob signs this transaction with his own private key.
4. Bob submits the signed transaction to the sequencer.
5. The sequencer includes it in a batch submitted to `submitTransactionsChecked`.
6. `processTransactionImpl` reaches the `LinkSigner` branch and calls `validateSignedTx(..., true)`.
7. `validateSignature` checks: is the signature from `0xAlice` or from `getLinkedSigner(0xAlice...)` = `0xBob`? Bob's signature is valid — check passes.
8. `linkedSigners[0xAlice...] = 0xCharlie` is written.
9. Charlie can now sign `WithdrawCollateral`, `TransferQuote`, and all other linked-signer-permitted transactions on behalf of Alice's subaccount — without Alice's knowledge or consent.
