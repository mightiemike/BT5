### Title
`verify_transaction_inclusion` Always Panics for Single-Transaction Blocks — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (and by extension `verify_transaction_inclusion_v2`) unconditionally rejects empty Merkle proofs via a hardcoded guard. For single-transaction blocks, the valid Merkle proof is empty — the Merkle root equals the transaction hash directly — so this guard always causes a panic, making it impossible for any caller to verify a transaction in a single-transaction block.

### Finding Description

In `contract/src/lib.rs` at line 315, `verify_transaction_inclusion` contains:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

For a block containing only one transaction (a coinbase-only block), the Merkle tree has a single leaf. The Merkle root equals the transaction hash directly, and no sibling hashes are needed — the proof is legitimately empty. The `compute_root_from_merkle_proof` function in `merkle-tools/src/lib.rs` correctly handles this case: an empty proof loop returns the transaction hash unchanged. [2](#0-1) 

However, the `require!` guard fires unconditionally before reaching that computation, causing a panic for every valid single-transaction block proof.

`verify_transaction_inclusion_v2` (line 347) calls `verify_transaction_inclusion` internally at line 368, inheriting the same defect: [3](#0-2) 

A caller submitting a valid proof for a single-transaction block with `merkle_proof = []` and `coinbase_merkle_proof = []` will:
1. Pass the length-equality check (`0 == 0`).
2. Pass the coinbase proof check — `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id`, which equals `header.block_header.merkle_root` for a single-transaction block.
3. Then unconditionally panic inside `verify_transaction_inclusion` at the empty-proof guard.

### Impact Explanation

Any downstream NEAR contract or user that needs to verify a transaction included in a single-transaction block cannot do so. The verification function always panics for this valid input class, making the light client permanently non-functional for single-transaction blocks. This is not a transient or configuration-dependent failure — it is an unconditional code path that always panics.

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) are valid and occur on all supported chains (Bitcoin, Litecoin, Dogecoin, Zcash). They are rare on mainnet but are entirely valid protocol outputs. Any legitimate use case requiring SPV proof of a coinbase transaction — cross-chain bridges tracking block rewards, protocols verifying miner payouts — is permanently broken. The entry path is fully unprivileged: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public contract methods callable by any NEAR account. [4](#0-3) [5](#0-4) 

### Recommendation

Remove or relax the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard. The `compute_root_from_merkle_proof` function already handles empty proofs correctly. If a guard is desired, replace it with a consistency check: an empty proof is valid if and only if `tx_index == 0` and the block's Merkle root equals `tx_id` directly.

### Proof of Concept

1. A block with a single transaction `T` has `merkle_root = hash(T)`.
2. Call `verify_transaction_inclusion_v2` with:
   - `tx_id = hash(T)`, `tx_block_blockhash = <block hash>`, `tx_index = 0`
   - `merkle_proof = []`, `coinbase_tx_id = hash(T)`, `coinbase_merkle_proof = []`, `confirmations = 1`
3. `merkle_proof.len() == coinbase_merkle_proof.len()` → `0 == 0` → **passes**.
4. `compute_root_from_merkle_proof(hash(T), 0, &[])` = `hash(T)` == `header.block_header.merkle_root` → **passes**.
5. `verify_transaction_inclusion` is called internally.
6. `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` → **panics unconditionally**.

The function always panics for this valid input, making it impossible to verify any transaction in a single-transaction block through either the v1 or v2 API. [6](#0-5)
