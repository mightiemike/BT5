Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of the pool call) and checks only `owner` (a freely chosen position beneficiary). Because `MetricOmmPool.addLiquidity` imposes no constraint that `msg.sender == owner`, any unlisted address can bypass the allowlist by supplying an allowlisted address as `owner`. This breaks the admin-enforced access boundary for restricted pools.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument and the caller-supplied `owner` as the second to the extension hook: [1](#0-0) 

There is no `msg.sender == owner` guard in `addLiquidity` (contrast with `removeLiquidity` at line 206 which enforces `if (msg.sender != owner) revert NotPositionOwner()`): [2](#0-1) 

The extension hook leaves the first parameter (`sender`) unnamed and never reads it, checking only `owner`: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender`, ignoring the second address argument: [4](#0-3) 

The asymmetry confirms the deposit extension checks the wrong field. An unlisted `bob` calls `pool.addLiquidity(owner=alice, ...)`. The extension sees `allowedDepositor[pool][alice] == true`, passes the check, and the pool then calls `bob.metricOmmSwapCallback(...)` to pull tokens from `bob`, minting the position to `alice`. The allowlist is fully bypassed.

## Impact Explanation

This is a direct admin-boundary break. The deposit allowlist is the primary access-control mechanism for restricted pools (KYC/AML-gated, institutional-only, whitelist-only). Any unlisted address can add liquidity to such a pool, altering bin composition and pool depth without authorization, potentially diluting existing LPs or violating compliance requirements the pool admin intended to enforce. The corrupted value is the `allowedDepositor` extension decision: it returns `beforeAddLiquidity.selector` (allow) when it should revert with `NotAllowedToDeposit`.

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with any known allowlisted address as `owner`. The `allowedDepositor` mapping is public, so any observer can identify a valid `owner` to supply. No privileged role, flash loan, or oracle manipulation is needed. The attack is immediately repeatable by any EOA or contract.

## Recommendation

Check `sender` (the actual depositor/payer) instead of `owner` (the position beneficiary):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the payer and the beneficiary, both `sender` and `owner` should be checked.

## Proof of Concept

```
Setup:
  - Pool P is configured with DepositAllowlistExtension.
  - allowedDepositor[P][alice] = true  (alice is allowlisted)
  - bob is NOT in the allowlist

Attack:
  1. bob calls P.addLiquidity(owner=alice, salt=0, deltas=..., callbackData=..., extensionData=...)
  2. Pool calls _beforeAddLiquidity(msg.sender=bob, owner=alice, ...)
  3. Extension checks allowedDepositor[P][alice] → true → no revert
  4. Pool calls LiquidityLib.addLiquidity(..., owner=alice, ...)
  5. Pool calls bob.metricOmmSwapCallback(...) → bob transfers tokens to pool
  6. Position shares are minted to alice

Result: bob (unlisted) has added liquidity to a restricted pool.
        alice can later call removeLiquidity to recover the tokens.
        The allowlist is fully bypassed.
```
