### Title
Unregulated Lower Bound on `confirmations` Parameter Allows Zero-Confirmation SPV Proof Acceptance - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` with only an upper-bound check (`confirmations <= gc_threshold`) and no lower-bound check. Passing `confirmations = 0` makes the depth guard trivially true for every block in the mainchain, returning `true` for a transaction with zero confirmations. Any unprivileged NEAR caller can exploit this to obtain a valid SPV proof for a transaction that has not yet been buried under any confirmations.

---

### Finding Description

In `verify_transaction_inclusion`, the only guard on `confirmations` is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

The subsequent depth check is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

Because `confirmations` is `u64`, when the caller passes `0`, the right-hand side of `>=` is `0`. Any `u64` value is `>= 0`, so the check is unconditionally satisfied for every block in the mainchain. The function proceeds to verify the Merkle proof and returns `true` for a transaction with zero confirmations.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase-proof check, so it inherits the same flaw:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

The `confirmations` field is a plain `u64` in both `ProofArgs` and `ProofArgsV2` with no validation at the type level:

```rust
pub struct ProofArgs {
    ...
    pub confirmations: u64,
}
``` [4](#0-3) 

The analog to the external report is exact: just as `setJoiningFee` only enforces `fee <= BASIS_PRECISION` (no lower bound), `verify_transaction_inclusion` only enforces `confirmations <= gc_threshold` (no lower bound). Setting the parameter to its extreme low value (0) nullifies the security guarantee the parameter is meant to enforce.

---

### Impact Explanation

A bridge or DeFi protocol on NEAR that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` and allows the caller to supply `confirmations` receives a `true` result for a transaction that has zero on-chain depth. The transaction may still be reorganized away. This enables a double-spend attack: the attacker broadcasts a Bitcoin transaction, gets it included in one block, immediately calls the NEAR bridge with `confirmations = 0`, receives a valid SPV proof result, claims the bridged asset, and then mines or bribes a longer chain to reverse the original Bitcoin transaction.

The corrupted proof result is the concrete broken invariant: the function is documented to require `confirmations` confirmed blocks before returning `true`, but with `confirmations = 0` it returns `true` with no depth requirement at all.

---

### Likelihood Explanation

The entry path requires no privilege. Any NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` directly. The `confirmations` field is a plain integer in the Borsh-serialized argument struct with no on-chain enforcement of a minimum. Any consumer contract that forwards a user-supplied `confirmations` value is immediately exploitable. The attack requires only a valid Merkle proof for a real transaction in any mainchain block, which is straightforward to construct from public Bitcoin data.

---

### Recommendation

Add a minimum confirmation lower-bound check at the start of `verify_transaction_inclusion`:

```rust
const MIN_CONFIRMATIONS: u64 = 1;

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    "Confirmations must be at least 1"
);
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

For production bridge use, the minimum should reflect the expected reorganization depth for the supported chain (e.g., 6 for Bitcoin mainnet). Alternatively, enforce the minimum in the `ProofArgs` / `ProofArgsV2` deserialization layer so the constraint is impossible to bypass regardless of caller.

---

### Proof of Concept

1. Relay a real Bitcoin block header to the contract so it is accepted into the mainchain.
2. Obtain the Merkle proof for any transaction in that block from a Bitcoin node (`gettxoutproof`).
3. Call `verify_transaction_inclusion` with:
   - `tx_id`: the transaction hash
   - `tx_block_blockhash`: the block hash
   - `tx_index`: the transaction index
   - `merkle_proof`: the Merkle path
   - `confirmations`: **0**
4. The upper-bound check `0 <= gc_threshold` passes.
5. The depth check `(tip_height - target_height + 1) >= 0` evaluates to `true` unconditionally.
6. The Merkle proof is verified and the function returns `true`.
7. The transaction is accepted as SPV-proven with zero confirmations, even if the block was submitted one second ago and has no depth. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```
