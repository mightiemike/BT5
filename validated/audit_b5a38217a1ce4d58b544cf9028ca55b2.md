### Title
Zero-Signer State Bypasses `requireValidTxSignatures`, Enabling Unauthorized Fast Withdrawals - (File: `core/contracts/Verifier.sol`)

---

### Summary

`requireValidTxSignatures` in `Verifier.sol` uses an equality check `nSignatures == nSigner` to enforce multi-sig quorum. When `nSigner == 0`, this check trivially passes with zero signatures, allowing any unprivileged caller to invoke `submitFastWithdrawal` on `BaseWithdrawPool` without any valid signature and drain the pool.

---

### Finding Description

`requireValidTxSignatures` counts non-empty entries in the caller-supplied `signatures` array and then enforces:

```solidity
require(nSignatures == nSigner, "not enough signatures");
``` [1](#0-0) 

When `nSigner == 0` — which is a reachable state if `Verifier.initialize` is called with all-zero `Point[8]` (no key is assigned, so `nSigner` is never incremented) — the check becomes `require(0 == 0)`, which always passes. A caller supplying an empty `signatures` array (`[]`) satisfies the check with no cryptographic work at all. [2](#0-1) 

This is structurally identical to the BSB22 zero-commitment flaw: the verifier hardcodes an assumption that the count of required proofs/signers is always nonzero, and the equality check degenerates to a no-op when the count is zero.

Contrast this with `requireValidSignature`, which routes through `checkQuorum`:

```solidity
return nSigned * 2 > nSigner;
``` [3](#0-2) 

`checkQuorum` returns `false` when `nSigner == 0` (since `0 * 2 > 0` is false), so `submitTransactionsChecked` is protected. `requireValidTxSignatures` has no equivalent guard, creating an inconsistency between the two verification paths.

The vulnerable path is `BaseWithdrawPool.submitFastWithdrawal`, which is a public, permissionless function that calls `requireValidTxSignatures` directly:

```solidity
v.requireValidTxSignatures(transaction, idx, signatures);
``` [4](#0-3) 

After the signature check passes, the function decodes the withdrawal transaction and calls `handleWithdrawTransfer`, transferring tokens to the `sendTo` address derived from the transaction body — which the attacker fully controls.

---

### Impact Explanation

An attacker can drain all ERC-20 collateral held in any `WithdrawPool` contract by:
1. Crafting a `WithdrawCollateral` or `WithdrawCollateralV2` transaction body pointing `sendTo` to their own address with an arbitrary `amount`.
2. Calling `submitFastWithdrawal` with an empty `signatures` array.
3. Receiving the full `transferAmount` with no fee deduction (since `sendTo != msg.sender` branch calls `safeTransferFrom` on the caller for the fee, but the attacker can set `sendTo == msg.sender` to subtract only the fee).

The corrupted asset delta is the entire token balance of the withdraw pool for any supported `productId`.

---

### Likelihood Explanation

The trigger condition is `nSigner == 0`. This is reachable in two ways:

1. **Initialization with empty key set**: `Verifier.initialize` does not enforce that at least one non-zero key is provided. If deployed with all-zero `initialSet`, `nSigner` stays at 0 permanently until an owner calls `assignPubKey`. During this window, the pool is fully unprotected.
2. **Owner deletes all keys**: `deletePubkey` (owner-only) decrements `nSigner`. If all keys are removed (e.g., during a key rotation where old keys are deleted before new ones are added), the window is open.

Scenario 1 is a realistic misconfiguration that the contract does not prevent. Scenario 2 requires owner action but is a natural operational risk during key rotation.

---

### Recommendation

Add an explicit guard at the top of `requireValidTxSignatures`:

```solidity
require(nSigner > 0, "no signers registered");
```

This mirrors the implicit protection that `checkQuorum` provides for `requireValidSignature` and closes the zero-signer bypass.

---

### Proof of Concept

1. Deploy `Verifier` via `initialize` with `initialSet` = `[Point(0,0), Point(0,0), ..., Point(0,0)]` (8 zero points). `nSigner` remains `0`.
2. Deploy `WithdrawPool` pointing to this `Verifier`. Fund it with 1,000,000 USDC.
3. Attacker constructs a `WithdrawCollateralV2` transaction body: `sender = attacker_address ++ 0x000000000000000000000000`, `productId = USDC_PRODUCT_ID`, `amount = 1_000_000e6`, `sendTo = attacker_address`.
4. Attacker calls `submitFastWithdrawal(idx=1, transaction=<crafted_bytes>, signatures=[])`.
5. `requireValidTxSignatures` executes: `nSignatures = 0`, `nSigner = 0`, `require(0 == 0)` passes.
6. `resolveFastWithdrawal` decodes the transaction, returns `sendTo = attacker_address`, `amount = 1_000_000e6`.
7. `handleWithdrawTransfer` transfers 1,000,000 USDC to the attacker. [5](#0-4) [4](#0-3)

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }
```
