### Title
Locked NLP Balance Included in Health Calculation While Unavailable for Burning, Causing Health Overstatement and Undercollateralization — (`core/contracts/SpotEngine.sol`, `core/contracts/SpotEngineState.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The `getHealth` function in `Clearinghouse.sol` computes subaccount health using `spotEngine.getHealthContribution`, which internally calls `_getBalance` → `getBalance` → reads from `balances[NLP_PRODUCT_ID][subaccount]`. This mapping stores the **full** NLP balance (locked + unlocked). However, `burnNlp` enforces that only the **unlocked** portion (tracked separately in `nlpLockedBalanceQueues[subaccount].unlockedBalanceSum`) can be burned. The locked portion is inaccessible during `NLP_LOCK_PERIOD`. This is a direct analog to the Alchemix `_harvestAndReport` bug: a balance component is incorrectly handled in a value-calculation function, corrupting solvency accounting.

---

### Finding Description

When a user mints NLP via `Clearinghouse.mintNlp`, two separate accounting paths are updated:

**Path 1 — main balance (used by health):**
`spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, nlpAmount)` → `_updateBalanceNormalized(state, balance, amountDelta)` → writes full amount into `balances[NLP_PRODUCT_ID][subaccount]`. [1](#0-0) 

**Path 2 — locked queue (used by burn gate):**
`handleNlpLockedBalance(subaccount, amountDelta)` → pushes the same amount into `nlpLockedBalanceQueues[subaccount]` with `unlockedAt = getOracleTime() + NLP_LOCK_PERIOD`. [2](#0-1) 

Health is computed via `getHealth` → `spotEngine.getHealthContribution` → `_getBalance`:

```solidity
function _getBalance(uint32 productId, bytes32 subaccount)
    internal view override returns (int128, int128)
{
    return (getBalance(productId, subaccount).amount, 0);
}
``` [3](#0-2) 

`getBalance` reads from `balances[productId][subaccount]` — the **full** balance including locked NLP. [4](#0-3) 

Meanwhile, `burnNlp` enforces:

```solidity
require(
    spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
    ERR_UNLOCKED_NLP_INSUFFICIENT
);
``` [5](#0-4) 

So locked NLP **cannot** be burned to cover liabilities, but it **is** counted at full weight in health. This is the accounting desynchronization.

A second corruption occurs during liquidation. `_assertLiquidationAmount` computes `maxLiquidatable` from the full balance: [6](#0-5) 

Then `_handleLiquidationPayment` calls `spotEngine.updateBalance(NLP_PRODUCT_ID, liquidatee, -txn.amount)`, which routes through `handleNlpLockedBalance(liquidatee, -txn.amount)`. When `amountDelta < 0`, the code unconditionally subtracts from `unlockedBalanceSum`:

```solidity
} else if (amountDelta < 0) {
    Balance memory balanceSum = nlpLockedBalanceQueues[subaccount].unlockedBalanceSum;
    balanceSum.amount += amountDelta;
    nlpLockedBalanceQueues[subaccount].unlockedBalanceSum = balanceSum;
}
``` [7](#0-6) 

There is no check that `unlockedBalanceSum >= |amountDelta|`. If all NLP is locked (`unlockedBalanceSum == 0`), liquidating any amount drives `unlockedBalanceSum` negative, permanently corrupting the locked balance state for that subaccount.

---

### Impact Explanation

1. **Health overstatement**: A user who mints NLP and has it fully locked can borrow against the full NLP value. If NLP price falls during `NLP_LOCK_PERIOD`, the account becomes undercollateralized with no way to self-liquidate (cannot burn locked NLP).
2. **Bad debt accumulation**: The protocol must liquidate the position, but liquidation corrupts `unlockedBalanceSum` (drives it negative), meaning the liquidatee's future NLP unlocks are silently consumed to offset the negative sum before any NLP becomes burnable again.
3. **Cascading state corruption**: A negative `unlockedBalanceSum` persists until enough NLP unlocks to bring it back to zero, during which the user cannot burn any NLP even after the lock period expires for their original deposit.

---

### Likelihood Explanation

Any user who mints NLP and then borrows against it is exposed. NLP price volatility during the lock period is a realistic market condition. The liquidation path through `handleNlpLockedBalance` with no underflow guard is reachable by any liquidator calling `liquidateSubaccountImpl` on an undercollateralized subaccount holding locked NLP. [8](#0-7) 

---

### Recommendation

1. **Exclude locked NLP from health**: In `_getBalance` for `NLP_PRODUCT_ID`, return only `getNlpUnlockedBalance(subaccount).amount` instead of the full `balances` amount. This aligns health with what is actually liquid.
2. **Guard `handleNlpLockedBalance` against underflow**: When `amountDelta < 0`, first consume from `unlockedBalanceSum` down to zero, then consume from the locked queue (earliest-expiring first), rather than blindly subtracting.
3. **Alternatively**, apply a time-weighted discount to locked NLP in health calculations proportional to remaining lock time.

---

### Proof of Concept

```
1. User mints 100 NLP (worth $100) → all 100 NLP locked for NLP_LOCK_PERIOD
   - balances[NLP][user] = 100
   - unlockedBalanceSum = 0
   - lockedQueue = [{amount: 100, unlockedAt: T + NLP_LOCK_PERIOD}]

2. getHealth(user) includes 100 NLP at full weight → user borrows $80 USDC
   (health check passes because locked NLP is counted)

3. NLP price drops 50% → user's health = 100 * $0.50 * weight - $80 < 0
   (maintenance health breached)

4. Liquidator calls liquidateSubaccountImpl(liquidatee=user, productId=NLP_PRODUCT_ID, amount=50)

5. _assertLiquidationAmount:
   spotAmount = spotEngine.getBalance(NLP_PRODUCT_ID, user).amount = 100  ✓
   maxLiquidatable = 100 >= 50  ✓ (passes, even though all NLP is locked)

6. _handleLiquidationPayment:
   spotEngine.updateBalance(NLP_PRODUCT_ID, user, -50)
   → handleNlpLockedBalance(user, -50)
   → unlockedBalanceSum.amount += -50  →  unlockedBalanceSum = -50  ← CORRUPTED

7. After NLP_LOCK_PERIOD expires, tryUnlockNlpBalance adds 100 to unlockedBalanceSum:
   unlockedBalanceSum = -50 + 100 = 50
   But user originally had 100 NLP; 50 NLP worth of value is silently lost.
```
