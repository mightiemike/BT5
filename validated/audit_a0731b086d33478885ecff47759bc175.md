Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates on Router Address Instead of Originating User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the value forwarded by `MetricOmmPool.swap` as `msg.sender` — which is the router contract address when a swap is routed through `MetricOmmSimpleRouter`. Any pool admin who allowlists the router to enable router-mediated access for legitimate users simultaneously opens the pool to every address on the network, completely defeating the per-user allowlist.

## Finding Description

**Root cause — three-step call chain:**

1. `MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` directly. The pool's `msg.sender` at that point is the router contract address, not the originating EOA. [1](#0-0) 

2. `MetricOmmPool.swap` passes `msg.sender` (the router) as the first argument to `_beforeSwap`, which propagates it to every configured extension as `sender`. [2](#0-1) 

3. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` — `msg.sender` here is the pool (correct), but `sender` is the router address, not the originating user. [3](#0-2) 

**Why existing guards are insufficient:**

The `onlyPool` guard (via `msg.sender` being the pool) correctly identifies which pool is calling, but does nothing to verify the identity of the economic actor. The `extensionData` bytes are passed through from the router to the extension, but `beforeSwap` ignores them entirely — it only reads `sender`. There is no mechanism in the current code to recover the originating user's address inside the extension. [3](#0-2) 

**Impossible admin configuration:**

| Admin action | Effect |
|---|---|
| Allowlist only specific EOAs | Allowlisted users **cannot** use the router (router address not in allowlist → revert) |
| Allowlist the router address | **Every** user can bypass the per-user allowlist via the router |

The same structural flaw applies to `exactInput`, `exactOutputSingle`, and `exactOutput` in the router, all of which call `pool.swap(...)` with the router as `msg.sender`. [4](#0-3) 

## Impact Explanation

A pool configured with `SwapAllowlistExtension` is intended to restrict swaps to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs). Once the pool admin allowlists the router — a necessary step to let legitimate users access the pool through the standard periphery — any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle` targeting that pool and swap successfully. This is a complete admin-boundary break: the pool admin's configured access control is rendered inoperative. Unauthorized swappers can execute swaps the admin explicitly intended to block, resulting in direct loss of LP principal through unwanted trades against the pool's liquidity.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard user-facing swap entry point. Any pool admin who deploys a swap-allowlisted pool and wants legitimate users to access it through the router will allowlist the router address. The bypass requires no special privileges, no flash loans, and no unusual token behavior — only a call to the public router with a valid pool address. The attack is repeatable on every block and requires no setup beyond knowing the pool address.

## Recommendation

The extension must gate the originating user, not the intermediary. Two viable approaches:

1. **Trusted forwarder pattern**: The router encodes `msg.sender` into `extensionData` using a well-known ABI layout. The extension decodes and verifies the originating user only when `sender` matches a known trusted router (verified against the factory registry). Non-router direct calls continue to use `sender` directly.

2. **Explicit swapper field in pool.swap**: Add a `swapper` parameter to `pool.swap` (separate from `msg.sender`) that the router populates with `msg.sender` before forwarding. The extension checks `swapper` instead of `sender`. This requires a core interface change.

Either approach must ensure the identity field cannot be spoofed by an untrusted caller.

## Proof of Concept

```solidity
// Setup
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
// Pool deployed with ext as beforeSwap extension

// Admin allowlists Alice and the router (necessary for Alice to use the router)
ext.setAllowedToSwap(pool, alice, true);
ext.setAllowedToSwap(pool, address(router), true);

// Attack: Bob (not allowlisted) calls the router directly
// pool.swap receives msg.sender = address(router)
// _beforeSwap(address(router), ...) is called
// ext.beforeSwap receives sender = address(router)
// allowedSwapper[pool][router] == true → check passes
// Bob's swap executes against the restricted pool

vm.prank(bob); // bob is NOT in the allowlist
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool,
        recipient: bob,
        tokenIn: token0,
        zeroForOne: true,
        amountIn: 1_000e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    })
);
// Bob successfully swaps; allowlist is bypassed
```

The exact wrong value is `allowedSwapper[pool][router]` being evaluated instead of `allowedSwapper[pool][bob]` at `SwapAllowlistExtension.sol` line 37, caused by `MetricOmmPool.sol` line 231 passing `msg.sender` (the router) as `sender` to the extension hook. [5](#0-4) [6](#0-5)

    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
      if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

      amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    }

    if (amount <= 0) revert InvalidSwapDeltas();
    amountOut = MetricOmmSwapInputs.int128ToUint128(amount);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
