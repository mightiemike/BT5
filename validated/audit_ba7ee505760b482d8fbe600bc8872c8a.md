### Title
`MsgUpdateTier` BonusApy Governance Change Creates Self-Reinforcing Pool-Drain Lock That Traps All Tier Positions - (File: x/tieredrewards/keeper/msg_server_auth.go)

### Summary
When governance executes `MsgUpdateTier` with an increased `BonusApy`, the pool drains faster. Once the pool is empty, governance cannot reduce the APY back (the proposal fails atomically), and simultaneously every user-driven exit path fails atomically. Users are permanently locked in their positions until the pool is externally replenished — a self-reinforcing feedback loop with no on-chain escape hatch.

### Finding Description
`UpdateTier` in `msg_server_auth.go` contains two coupled behaviors that together create the loop:

**Step 1 — APY change triggers a full-tier reward sweep:** [1](#0-0) 

When `BonusApy` changes, `claimRewardsAndUpdateTierPositions` iterates every position in the tier and calls `processEventsAndClaimBonus` for each one: [2](#0-1) 

If the pool is insufficient for **any** position in the sweep, the entire governance proposal execution reverts with `ErrInsufficientBonusPool`: [3](#0-2) 

This is confirmed by the keeper test `TestUpdateTier_BonusApyChange_InsufficientPool`: [4](#0-3) 

**Step 2 — Every user-driven exit path also fails when the pool is empty:**

`MsgExitTierWithDelegation`, `MsgTierUndelegate`, `MsgTierRedelegate`, `MsgAddToTierPosition`, and `MsgClearPosition` all call `claimRewards` before mutating position state: [5](#0-4) 

This is the documented design: [6](#0-5) 

**The feedback loop:**
1. Governance increases `BonusApy` (e.g. 5% → 50%). The sweep at execution time pays out old-rate rewards (if pool is sufficient at that moment) and advances `LastBonusAccrual` for all positions.
2. New bonus accrues at 50% — pool drains 10× faster.
3. Pool empties.
4. Governance submits a corrective proposal to reduce APY back to 5%. At execution, `claimRewardsAndUpdateTierPositions` tries to pay out all accrued 50%-rate rewards. Pool is empty → proposal fails → APY stays at 50%.
5. Simultaneously, every user who tries to undelegate, redelegate, add to position, clear position, or exit with delegation hits `ErrInsufficientBonusPool` and is rejected.
6. Users are locked in their positions with no on-chain path out until an external actor funds the pool.

### Impact Explanation
The corrupted invariant is: **users must always be able to exit a tier position they own**. When the pool is empty and any bonus has accrued, the following messages all fail atomically for every position in the affected tier:
- `MsgTierUndelegate` — cannot start unbonding
- `MsgExitTierWithDelegation` — cannot transfer delegation back
- `MsgTierRedelegate` — cannot move to a safer validator
- `MsgClearPosition` — cannot cancel exit
- `MsgClaimTierRewards` — cannot claim

Users' staked tokens are inaccessible. If the locked validator is subsequently jailed or slashed, users suffer staking losses they cannot mitigate. The only escape is an external party funding the `RewardsPoolName` module account — there is no on-chain governance path to unblock users while the APY remains high and the pool is empty.

### Likelihood Explanation
Medium. A governance proposal to increase `BonusApy` is a normal operational action (the integration tests demonstrate exactly this flow). The pool drain rate scales linearly with APY, so a 10× APY increase depletes the pool 10× faster. If the pool is not simultaneously replenished to match the new rate, the feedback loop engages. There is no on-chain validation that prevents a governance proposal from setting an APY that the current pool cannot sustain.

### Recommendation
1. **Pre-execution pool check**: Before executing a `BonusApy` increase, verify that the pool balance is sufficient to sustain the new rate for a minimum period (e.g. 30 days at current total delegated stake).
2. **Emergency exit path**: Provide a separate `MsgEmergencyWithdraw` or allow `MsgTierUndelegate` to forfeit unclaimed bonus when the pool is empty, so users are never permanently locked.
3. **Governance proposal bundling**: Require that any `MsgUpdateTier` increasing `BonusApy` be bundled with a `MsgFundRewardsPool` in the same proposal, ensuring the pool is replenished atomically.
4. **APY cap relative to pool**: Add a keeper-level validation in `SetTier` that rejects a `BonusApy` value whose projected drain rate exceeds the current pool balance divided by a minimum runway constant.

func (ms msgServer) UpdateTier(ctx context.Context, msg *types.MsgUpdateTier) (*types.MsgUpdateTierResponse, error) {
	if err := ms.requireAuthority(msg.Authority); err != nil {
		return nil, err
	}

	oldTier, err := ms.getTier(ctx, msg.Tier.Id)
	if err != nil {
		return nil, err
	}

	if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
		if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
			return nil, err
		}
	}

	if err := ms.SetTier(ctx, msg.Tier); err != nil {
		return nil, err
	}

	if err := ms.emitTierChangedEvent(ctx, types.TierChangeAction_TIER_CHANGE_ACTION_UPDATE, msg.Tier); err != nil {
		return nil, err
	}

	return &types.MsgUpdateTierResponse{}, nil
}

func (k Keeper) claimRewardsAndUpdateTierPositions(ctx context.Context, tierId uint32) error {
	ids, err := k.getPositionsIdsByTier(ctx, tierId)
	if err != nil {
		return err
	}
	if len(ids) == 0 {
		return nil
	}

	for _, id := range ids {
		pos, err := k.getPositionState(ctx, id)
		if err != nil {
			return err
		}
		if !pos.IsDelegated() {
			continue
		}

		if _, err := k.claimBaseRewards(ctx, pos); err != nil {
			return err
		}
		if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
			return err
		}
		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return err
		}
	}

	return nil
}

func (s *KeeperSuite) TestUpdateTier_BonusApyChange_InsufficientPool() {
	lockAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
	pos := s.setupNewTierPosition(lockAmount, false)
	valAddr := sdk.MustValAddressFromBech32(pos.Delegation.ValidatorAddress)

	s.setValidatorCommission(valAddr, sdkmath.LegacyZeroDec())

	// Advance time so bonus accrues.
	s.ctx = s.ctx.WithBlockHeight(s.ctx.BlockHeight() + 1)
	s.ctx = s.ctx.WithBlockTime(s.ctx.BlockTime().Add(30 * 24 * time.Hour))

	// Update tier with new BonusApy — should fail due to insufficient pool.
	tier := newTestTier(1)
	initialBonusApy := tier.BonusApy
	updatedBonusApy := sdkmath.LegacyNewDecWithPrec(8, 2)
	tier.BonusApy = updatedBonusApy
	msgServer := keeper.NewMsgServerImpl(s.keeper)
	_, err := msgServer.UpdateTier(s.ctx, &types.MsgUpdateTier{
		Authority: s.keeper.GetAuthority(),
		Tier:      tier,
	})
	s.Require().ErrorIs(err, types.ErrInsufficientBonusPool)

	// Tier should NOT have been updated (tx failed).
	got, err := s.keeper.GetTier(s.ctx, 1)
	s.Require().NoError(err)
	s.Require().True(initialBonusApy.Equal(got.BonusApy), "tier should still have old APY")
}
