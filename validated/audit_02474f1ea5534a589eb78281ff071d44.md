## Root Cause Verification

**`getCoreRisk` in `BaseEngine.sol` (line 190):**

```solidity
function getCoreRisk(...) external returns (IProductEngine.CoreRisk memory) {
    RiskHelper.Risk memory risk = _risk(productId);
    (int128 amount, ) = _getBalance(productId, subaccount);
    return IProductEngine.CoreRisk(
        amount,
        risk.priceX18,
        RiskHelper._getWeightX18(risk, 1, healthType)  // ← hardcoded 1, not amount
    );
}
```

The `longWeight` field of `CoreRisk` is always computed with `amount=1`, so `_getWeightX18` always takes the `amount >= 0` branch and returns `longWeightMaintenanceX18` / `longWeightInitialX18` — never the short weight — regardless of the actual position direction.

**`_getWeightX18` in `RiskHelper.sol` (lines 44–52):**

```solidity
if (amount >= 0) {
    weight = healthType == IProductEngine.HealthType.INITIAL
        ? risk.longWeightInitialX18
        : risk.longWeightMaintenanceX18;
} else {
    weight = ...shortWeight...;
}
```

With hardcoded `1`, the `else` branch is never reached.

## How This Propagates Into `getHealth`

In `Clearinghouse.getHealth` (lines 125–135), the spread adjustment is:

```solidity
int128 existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2;
int128 spreadWeight = RiskHelper._getSpreadWeightX18(perpCoreRisk, spotCoreRisk, healthType);
health += basisAmount.mul(spotCoreRisk.price + perpCoreRisk.price).mul(spreadWeight - existingWeight);
```

`_getSpreadWeightX18` (lines 66–69) also reads `spotCoreRisk.longWeight` / `perpCoreRisk.longWeight` directly from the struct:

```solidity
if (spotCoreRisk.amount > 0) {
    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
} else {
    spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;  // ← uses wrong weight for short spot
}
```

The design intent is that `CoreRisk.longWeight` holds the **direction-appropriate** weight (long weight for longs, short weight for shorts). The `_getSpreadWeightX18` function branches on `spotCoreRisk.amount` to pick which side is risky, then uses that side's `longWeight` field — expecting it to carry the short weight when the position is short. The hardcoded `1` breaks this.

## Concrete Numerical Proof

**Scenario:** Short spot + long perp basis position.
- `spotAmount = -1000`, `perpAmount = +1000`, price = 1000 for both
- `longWeight_spot = 0.9`, `shortWeight_spot = 1.1`
- `longWeight_perp = 0.9`
- Quote balance = 50,000

**`basisAmount`** (lines 116–121 of `Clearinghouse.sol`):
`basisAmount = -max(-1000, -1000) = 1000` (positive)

**Base health** (computed correctly in `_calculateProductHealth` using actual `amount`):
- Short spot: `-1000 × 1.1 × 1000 = -1,100,000`
- Long perp: `+1000 × 0.9 × 1000 = +900,000`
- Quote: `+50,000`
- **Total base = -150,000**

**Spread adjustment — WITH BUG** (both `longWeight` fields = 0.9):
- `existingWeight = (0.9 + 0.9) / 2 = 0.9`
- `spreadWeight = 1 - (1 - 0.9)/5 = 0.98`
- `spreadWeight - existingWeight = +0.08`
- Adjustment = `1000 × 2000 × 0.08 = +160,000`
- **Total health = -150,000 + 160,000 = +10,000 → NOT liquidatable**

**Spread adjustment — CORRECT** (spot uses `shortWeight = 1.1`):
- `existingWeight = (1.1 + 0.9) / 2 = 1.0`
- `spreadWeight = 1 - (1 - 1.1)/5 = 1.02` → capped at `0.994`
- `spreadWeight - existingWeight = -0.006`
- Adjustment = `1000 × 2000 × (-0.006) = -12,000`
- **Total health = -150,000 + (-12,000) = -162,000 → liquidatable**

The bug flips the sign of `(spreadWeight - existingWeight)` from **−0.006** to **+0.08**, inflating health by **172,000** and turning a −162,000 health into +10,000.

### Title
Hardcoded `amount=1` in `BaseEngine.getCoreRisk` causes spread health to be inflated, blocking liquidation of insolvent basis-trade accounts — (`core/contracts/BaseEngine.sol`)

### Summary
`getCoreRisk` passes the literal `1` instead of the actual position `amount` to `_getWeightX18`, so `CoreRisk.longWeight` always holds the long weight. The spread health adjustment in `Clearinghouse.getHealth` uses this field for both `existingWeight` and `_getSpreadWeightX18`. For short-side positions (short spot or short perp), the correct short weight (> 1) is replaced by the long weight (< 1), reversing the sign of `(spreadWeight − existingWeight)` from negative to positive and inflating health by up to ~8.6% of `basisAmount × (spotPrice + perpPrice)`.

### Finding Description
`BaseEngine.getCoreRisk` (line 190) calls `RiskHelper._getWeightX18(risk, 1, healthType)` with a hardcoded `1` instead of the actual balance `amount`. Because `1 ≥ 0`, `_getWeightX18` always returns the long weight. The `CoreRisk` struct's `longWeight` field is consumed by `Clearinghouse.getHealth` in two places:

1. **`existingWeight`** — the average of spot and perp `longWeight` fields, representing the weight already applied in the base health calculation. For a short spot position, the correct value is `shortWeight_spot` (> 1), but the bug supplies `longWeight_spot` (< 1), making `existingWeight` too small.

2. **`_getSpreadWeightX18`** — branches on `spotCoreRisk.amount` to pick the risky side, then computes `1 − (1 − w)/5` using that side's `longWeight`. For a short spot, the correct `w` is `shortWeight_spot > 1`, giving `spreadWeight > 1` (capped at 0.994). The bug supplies `longWeight_spot < 1`, giving `spreadWeight < 1`.

The combined effect reverses `(spreadWeight − existingWeight)` from approximately −0.006 (correct, penalising the spread) to +0.08 (buggy, rewarding it), inflating the spread health contribution by ~8.6% of `basisAmount × totalPrice`.

### Impact Explanation
An account holding a large short-spot / long-perp (or long-spot / short-perp) basis position can be genuinely insolvent (MAINTENANCE health < 0 under correct weights) yet `getHealth` returns a positive value. `isUnderMaintenance` returns `false`, `liquidateSubaccountImpl` reverts with `ERR_NOT_LIQUIDATABLE`, and the account cannot be liquidated. The protocol accumulates bad debt from these unliquidatable insolvent accounts, violating the core solvency invariant.

### Likelihood Explanation
Basis trades (short spot + long perp) are a standard, incentivised strategy on the protocol. Any account with a sufficiently large spread position and a quote balance that partially offsets the base health deficit will trigger this condition. No special privileges or external conditions are required — the bug is present in every call to `getHealth` that processes a spread pair.

### Recommendation
Replace the hardcoded `1` with the actual `amount` in `getCoreRisk`:

```solidity
// BaseEngine.sol line 190
- RiskHelper._getWeightX18(risk, 1, healthType)
+ RiskHelper._getWeightX18(risk, amount, healthType)
```

This ensures `CoreRisk.longWeight` carries the direction-appropriate weight (long weight for long positions, short weight for short positions), which is what both `existingWeight` and `_getSpreadWeightX18` require.

    function _getSpreadWeightX18(
        IProductEngine.CoreRisk memory perpCoreRisk,
        IProductEngine.CoreRisk memory spotCoreRisk,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }
        int128 spreadWeight;
        if (spotCoreRisk.amount > 0) {
            spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
        } else {
            spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
        }
        int128 maxSpreadWeight;
        if (healthType == IProductEngine.HealthType.INITIAL) {
            maxSpreadWeight = ONE - ONE / 100; // 0.99
        } else {
            maxSpreadWeight = ONE - (ONE * 6) / 1000; // 0.994
        }
        if (spreadWeight > maxSpreadWeight) {
            spreadWeight = maxSpreadWeight;
        }
        return spreadWeight;
    }
