Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End-User, Allowing Any User to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` unconditionally sets to `msg.sender` — the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router (required for any approved user to use it), every non-allowlisted address can bypass the gate by routing through the same public router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool. The real user's address is stored in transient storage via `_setNextCallbackContext` only for the payment callback — it is never forwarded to the pool or any extension as the identity to gate: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) [6](#0-5) 

The bypass is structural: the pool has no mechanism to receive or forward the originating user's address to extensions; the router has no mechanism to inject it.

## Impact Explanation
A pool admin who deploys a restricted pool (e.g., KYC-only, institutional-only) using `SwapAllowlistExtension` must allowlist the router address the moment any approved user needs to use it. Once the router is allowlisted, the access-control invariant is completely broken: any non-approved address calls `exactInputSingle` (or any other router swap function), the extension sees `sender = router`, `allowedSwapper[pool][router] = true` passes, and the attacker receives output tokens. LP providers are exposed to trades the pool admin explicitly intended to block. This is a complete, silent bypass of the configured access-control invariant with direct fund-flow consequences — matching the "admin-boundary break by an unprivileged path" allowed impact.

## Likelihood Explanation
`MetricOmmSimpleRouter` is a public, permissionless periphery contract callable by any address. The only precondition is that the router is allowlisted for the target pool — a configuration the pool admin is forced into the moment any approved user needs router access. No privileged role, no special setup, and no malicious initial configuration is required on the attacker's side. The bypass is repeatable on every block.

## Recommendation
The `sender` argument passed to `beforeSwap` must represent the economic actor (the end user), not the intermediary contract. Two complementary fixes:

1. **In `MetricOmmSimpleRouter`**: pass the original `msg.sender` as a dedicated `originator` field in `extensionData` (or as a new explicit parameter in the pool's `swap` signature), and have `SwapAllowlistExtension` read and check that field instead of `sender`.
2. **In `SwapAllowlistExtension`**: until the core passes the originating user, document clearly that `sender` is the direct caller of `pool.swap()` and that router-mediated swaps will present the router address, making the extension unsafe for end-user identity gating when a router is in the path.

The cleanest fix is to add an explicit `originator` field to the pool's `swap` signature (or to `extensionData`) that the router populates with `msg.sender` before calling the pool, and have `SwapAllowlistExtension` check that field instead of `sender`.

## Proof of Concept
```solidity
// Setup
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
// Pool admin allowlists only `approvedUser` and the router
ext.setAllowedToSwap(pool, approvedUser, true);
ext.setAllowedToSwap(pool, address(router), true); // required for approvedUser to use router

// Attack: attacker (not allowlisted) calls the router directly
vm.prank(attacker); // attacker is NOT in allowedSwapper[pool]
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1_000e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    tokenIn: token0,
    extensionData: "",
    deadline: block.timestamp
}));
// router calls pool.swap() → msg.sender = router → sender = router
// allowedSwapper[pool][router] = true → PASSES
// attacker receives token1 output despite not being on the allowlist
```
