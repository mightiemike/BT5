Looking at the Nado codebase for an analog to the "hardcoded fee calibrated for one token applied to all tokens" vulnerability class.

The key constants and fee-handling code are in `core/contracts/common/Constants.sol`, `core/contracts/Clearinghouse.sol`, and `core/contracts/EndpointTx.sol`.

---

### Title
`slowModeFees` Accumulator Uses Raw Unscaled `SLOW_MODE_FEE` While `chargeSlowModeFee` Charges Token-Decimal-Scaled Amount, Causing Permanent Fee Accounting Corruption — (`core/contracts/EndpointTx.sol`)

### Summary

`SLOW_MODE_FEE = 1000000` is hardcoded assuming a 6-decimal quote token (USDC). `chargeSlowModeFee` correctly scales this via `getSlowModeFee()`, but the `slowModeFees` accumulator always adds the raw unscaled constant. For any quote token with more than 6 decimals, the protocol collects far more tokens than it tracks, permanently corrupting the fee accounting state.

### Finding Description

`SLOW_MODE_FEE` is defined as `1000000` (representing $1 in USDC terms): [1](#0-0) 

`getSlowModeFee()` in `Clearinghouse.sol` scales this constant to the actual quote token's decimals:

```solidity
int256 multiplier = int256(10**(token.decimals() - 6));
return uint256(int256(SLOW_MODE_FEE) * multiplier);
``` [2](#0-1) 

`chargeSlowModeFee` in `EndpointStorage.sol` correctly uses this scaled value when pulling tokens from the user: [3](#0-2) 

However, in `submitSlowModeTransactionImpl`, after charging the scaled fee, the accumulator is incremented with the **raw unscaled constant**:

```solidity
chargeSlowModeFee(_getQuote(), sender);   // charges getSlowModeFee() = SLOW_MODE_FEE * 10^(decimals-6)
slowModeFees += SLOW_MODE_FEE;            // tracks only 1_000_000 regardless of token decimals
``` [4](#0-3) 

For an 18-decimal quote token (e.g., DAI, WETH):
- `chargeSlowModeFee` pulls `1_000_000 × 10^12 = 10^18` tokens from the user
- `slowModeFees` is incremented by only `1_000_000`

This creates a permanent 10^12× discrepancy per slow-mode transaction between actual tokens held and the tracked `slowModeFees` balance. When the protocol later uses `slowModeFees` to determine how much to transfer out (in `DumpFees` or sequencer fee claims), it will transfer only `n × 10^6` tokens instead of `n × 10^18`, leaving the vast majority of collected fees permanently stranded in the contract.

A secondary issue exists in the same `getSlowModeFee()` function: `token.decimals() - 6` is a `uint8` subtraction. If the quote token has fewer than 6 decimals, this underflows to 252, causing `10^252` which overflows `int256`, making every non-admin slow-mode transaction permanently revert. [5](#0-4) 

### Impact Explanation

For any deployment where the quote token has more than 6 decimals, every slow-mode transaction (withdrawals, linked signer changes, etc.) causes the protocol to collect the correct scaled fee in tokens but record only the USDC-equivalent dust amount in `slowModeFees`. The difference — `(10^(decimals-6) - 1) × SLOW_MODE_FEE` tokens per transaction — accumulates in the contract and cannot be claimed through the normal fee-dump path. This is a direct protocol revenue loss proportional to the number of slow-mode transactions submitted.

### Likelihood Explanation

Any deployment using a non-USDC quote token (e.g., DAI, WETH, or any 18-decimal stablecoin) triggers this immediately on the first slow-mode transaction submitted by any unprivileged user. The entry path is `submitSlowModeTransaction` → `submitSlowModeTransactionImpl`, which is publicly callable by any user.

### Recommendation

Replace the raw `slowModeFees += SLOW_MODE_FEE` with the scaled amount:

```solidity
slowModeFees += int128(uint128(clearinghouse.getSlowModeFee()));
```

Additionally, add a `require(token.decimals() >= 6)` guard in `getSlowModeFee()` to prevent the `uint8` underflow for tokens with fewer than 6 decimals.

### Proof of Concept

1. Deploy the protocol with an 18-decimal ERC20 as the quote token.
2. Any user calls `submitSlowModeTransaction` with a `WithdrawCollateral` transaction.
3. `chargeSlowModeFee` pulls `10^18` tokens from the user (1 full token).
4. `slowModeFees` is incremented by `1_000_000` (10^-12 of a token).
5. After `N` such transactions, the contract holds `N × 10^18` tokens in slow-mode fees but `slowModeFees == N × 10^6`.
6. When `DumpFees` is called, it transfers `N × 10^6` tokens — leaving `N × (10^18 - 10^6)` tokens permanently stranded.
