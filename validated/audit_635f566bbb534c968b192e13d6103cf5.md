### Title
`EpochCache::try_resolve_epoch_id` Binds Wrong Epoch ID When Staking Contract Skips an Epoch, Selecting Wrong Proposer - (`crates/apollo_staking/src/staking_manager.rs`)

---

### Summary

`EpochCache::try_resolve_epoch_id` unconditionally returns `current.epoch_id + 1` for block heights that fall within the "next epoch min bounds" window. This hardcoded `+1` assumption mirrors the external bug exactly: just as the vault assumed oracle version N+1 existed when the product had already advanced to N+2, the sequencer assumes the next epoch is always `current + 1`. When the staking contract produces an epoch shorter than `MIN_EPOCH_LENGTH` (explicitly present in test constants as `SHORT_EPOCH_LENGTH = 30`), heights that physically belong to epoch N+2 are resolved to epoch N+1 without ever refreshing the epoch cache. The wrong committee and wrong randomness seed are then used for proposer selection, binding the wrong signer to every block height in that window.

---

### Finding Description

**Root cause — the `+1` assumption in `try_resolve_epoch_id`:**

```rust
fn try_resolve_epoch_id(&self, height: BlockNumber) -> Option<u64> {
    let current = self.current.as_ref()?;
    if current.contains(height) {
        Some(current.epoch_id)
    } else if current.within_next_epoch_min_bounds(height) {
        Some(current.epoch_id + 1)   // ← hardcoded assumption: next epoch = current + 1
    } else {
        None
    }
}
``` [1](#0-0) 

The `within_next_epoch_min_bounds` window is defined as the first `MIN_EPOCH_LENGTH.min(self.epoch_length)` blocks after the current epoch ends:

```rust
fn within_next_epoch_min_bounds(&self, height: BlockNumber) -> bool {
    let next_epoch_start_block = BlockNumber(self.start_block.0 + self.epoch_length);
    let next_epoch_min_length = MIN_EPOCH_LENGTH.min(self.epoch_length);
    range_contains(height, next_epoch_start_block, next_epoch_min_length)
}
``` [2](#0-1) 

The comment states: *"Only heights guaranteed to fall within the next epoch may resolve to `epoch_id + 1`."* This guarantee holds only if every epoch has length ≥ `MIN_EPOCH_LENGTH`. The code never validates this invariant against the staking contract's output.

**The invariant is violated by design — short epochs exist:**

```rust
// Epochs shorter than MIN_EPOCH_LENGTH, matching the config-backed mock contract's epochs.
const SHORT_EPOCH_LENGTH: u64 = 30;
const SHORT_EPOCH_4: Epoch =
    Epoch { epoch_id: 4, start_block: BlockNumber(120), epoch_length: SHORT_EPOCH_LENGTH };
``` [3](#0-2) 

`MIN_EPOCH_LENGTH = 5 * 60 / 2 = 150` blocks. `SHORT_EPOCH_LENGTH = 30 < 150`. When epoch N+1 has length 30, the "next epoch min bounds" window of epoch N spans 150 blocks starting at epoch N+1's start. The last 120 of those 150 blocks physically belong to epoch N+2. For any height H in that tail, `try_resolve_epoch_id` returns N+1 and **returns immediately without calling `update_epochs`**. [4](#0-3) 

**Downstream effect — wrong committee and wrong randomness seed:**

`epoch_at_height` returns the wrong epoch ID (N+1). `committee_at_height` then calls `fetch_and_build_committee(N+1)`, which calls `proposer_randomness_block_hash(N+1)`:

```rust
let previous_epoch_id = epoch - 1;   // = N
let prev_epoch = cache.get_epoch(previous_epoch_id);  // finds epoch N in cache
let block_hash = self.get_block_hash_with_fallback(prev_epoch.start_block).await?;
``` [5](#0-4) 

For the correct epoch N+2, the randomness seed should be the start-block hash of epoch N+1. Instead, epoch N's start-block hash is used. The committee membership (stakers for epoch N+1 vs. N+2) and the randomness seed are both wrong, producing a completely different proposer address.

**Node divergence:** A node whose epoch cache already holds epoch N+2 as `current` will correctly resolve height H to epoch N+2. A node whose cache still holds epoch N will resolve H to epoch N+1. The two nodes select different proposers for the same height, causing consensus to stall or accept a block from an unauthorized proposer.

---

### Impact Explanation

The wrong proposer is bound to every block height that falls in the overlap zone (heights in epoch N+2 that lie within the `MIN_EPOCH_LENGTH` window of epoch N). Nodes with stale epoch caches accept a different proposer than nodes with fresh caches. This maps directly to **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"** — the randomness hash and committee membership together constitute the signer-binding mechanism for block proposals, and both are wrong.

---

### Likelihood Explanation

The staking contract is a Cairo contract whose epoch lengths are not constrained by the sequencer. The test suite explicitly defines `SHORT_EPOCH_LENGTH = 30 < MIN_EPOCH_LENGTH = 150` and uses it in multiple test epochs (`SHORT_EPOCH_4` through `SHORT_EPOCH_7`), confirming the protocol allows sub-`MIN_EPOCH_LENGTH` epochs. The trigger requires: (1) a short epoch N+1 (length < `MIN_EPOCH_LENGTH`), and (2) a node that has not yet called `update_epochs` since epoch N ended. Condition (2) is the normal operating state during the "next epoch min bounds" window, which is precisely the window this code path is designed to serve.

---

### Recommendation

After `try_resolve_epoch_id` returns a value via the `within_next_epoch_min_bounds` branch, do not trust the `+1` result without first refreshing the epoch cache. The simplest fix is to always call `update_epochs` when the height is not contained in the current epoch, and only use the `+1` shortcut as a hint to avoid a contract call when the cache confirms the next epoch has already started. Alternatively, validate that the epoch returned by `get_current_epoch()` is exactly `current_epoch_id + 1` before using the cached value; if it is not sequential, fall through to a full cache refresh.

---

### Proof of Concept

**Setup:**
- Cache: `current = Epoch { epoch_id: 1, start_block: 1, epoch_length: 1000 }`
- Staking contract state: epoch 3 is current (epoch 2 had `epoch_length = 30`, already ended)
- `MIN_EPOCH_LENGTH = 150`

**Step 1 — compute the window for epoch 1:**
- `next_epoch_start = 1 + 1000 = 1001`
- `next_epoch_min_length = min(150, 1000) = 150`
- Window = `[1001, 1151)`

**Step 2 — query height H = 1031 (first block of epoch 3, since epoch 2 ran from 1001–1030):**
- `current.contains(1031)` → false (epoch 1 ends at 1000)
- `current.within_next_epoch_min_bounds(1031)` → true (1031 ∈ [1001, 1151))
- `try_resolve_epoch_id(1031)` → **returns 2** (wrong; correct answer is 3)
- `update_epochs` is **never called**

**Step 3 — committee built for epoch 2:**
- `fetch_and_build_committee(2)` fetches stakers for epoch 2 (different from epoch 3)
- `proposer_randomness_block_hash(2)` uses epoch 1's start-block hash (should use epoch 2's)
- Proposer selected ≠ proposer that nodes with a fresh cache select for height 1031

**Step 4 — consensus divergence:**
- Nodes with `current = epoch 3` in cache resolve height 1031 → epoch 3 → correct proposer P3
- Nodes with `current = epoch 1` in cache resolve height 1031 → epoch 2 → wrong proposer P2
- P2 ≠ P3; consensus stalls or an unauthorized block from P2 is accepted by stale-cache nodes

    async fn epoch_at_height(&self, height: BlockNumber) -> CommitteeProviderResult<u64> {
        // Try to resolve the epoch from cache.
        let mut epoch_cache = self.epoch_cache.lock().await;
        if let Some(epoch_id) = epoch_cache.try_resolve_epoch_id(height) {
            return Ok(epoch_id);
        }

        // Otherwise, sync the epochs from the state.
        info!("Epoch resolving for height={}, fetching latest epochs from contract.", height.0);
        epoch_cache.update_epochs(self.staking_contract.clone()).await?;

        let epoch_id = epoch_cache
            .try_resolve_epoch_id(height)
            .ok_or(CommitteeProviderError::InvalidHeight { height })?;

        info!("Resolved height={height} to epoch_id={epoch_id}.");
        Ok(epoch_id)
    }
