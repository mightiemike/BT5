### Title
Frequent `MsgClaimTierRewards` (and Reward-Settling Messages) Cause Permanent Loss of Bonus Rewards via Per-Segment `TruncateInt()` Without Fractional Carry-Forward — (`File: x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

Every time bonus rewards are settled in `x/tieredrewards`, the function `computeSegmentBonus` applies `TruncateInt()` to the result of `tokens * bonusApy * durationSeconds / SecondsPerYear`. After settlement, `LastBonusAccrual` is advanced to the exact block time with no fractional remainder preserved. Each settlement independently discards up to one full unit of the bond denom. A position owner who claims frequently — or who triggers implicit settlements via `MsgAddToTierPosition`, `MsgTierRedelegate`, `MsgClearPosition`, `MsgTierUndelegate`, or `MsgExitTierWithDelegation` — permanently loses bonus rewards they are entitled to. The lost tokens remain in the `rewards_pool` module account and are never redistributed to the user.

---

### Finding Description

`computeSegmentBonus` computes the bonus for a time segment as:

```
tokens * bonusApy * durationSeconds / SecondsPerYear
```

and truncates the result to an integer: [1](#0-0) 

After each settlement, `applyBonusAccrualCheckpoint` advances `LastBonusAccrual` to the current block time with no fractional carry-forward: [2](#0-1) 

This checkpoint is applied unconditionally at the end of every `processEventsAndClaimBonus` call: [3](#0-2) 

The `SecondsPerYear` constant is 31,557,600: [4](#0-3) 

Because the fractional part of `tokens * bonusApy * durationSeconds / SecondsPerYear` is silently dropped on every settlement and the discarded fraction is never carried into the next segment, splitting one long accrual window into N shorter windows always yields ≤ the single-window result. The deficit is permanent — the position's `LastBonusAccrual` is already advanced past the lost fraction.

Reward settlement is triggered not only by `MsgClaimTierRewards`: [5](#0-4) 

but also implicitly by `MsgAddToTierPosition` (which calls `claimRewards` before delegating new tokens): [6](#0-5) 

and by `MsgTierRedelegate`, `MsgClearPosition`, `MsgTierUndelegate`, and `MsgExitTierWithDelegation`, all of which settle rewards before their state mutation per the ADR: [7](#0-6) 

---

### Impact Explanation

**Corrupted value:** The user's accrued bonus reward balance. Tokens that should be transferred from `rewards_pool` to the owner are instead permanently stranded in the pool.

**Quantified loss for a 1,000,000-token position at 4% APY (40,000 tokens/year):**

| Claim frequency | Tokens per claim | Annual total | Loss |
|---|---|---|---|
| Every block (5 s) | `floor(0.00634) = 0` | 0 | 100% |
| Every hour (3,600 s) | `floor(4.56) = 4` | 35,040 | 12.4% |
| Every day (86,400 s) | `floor(109.5) = 109` | 39,785 | 0.5% |
| Once per year | `floor(40,000) = 40,000` | 40,000 | 0% |

A user who claims every block loses their entire annual bonus. A user who claims daily loses ~0.5%. The loss is permanent and non-recoverable.

---

### Likelihood Explanation

The loss is triggered by any of the six reward-settling messages. Users who actively manage positions — adding tokens, redelegating, or clearing exits — trigger implicit settlements at each operation. The `MsgClaimTierRewards` message itself has no rate limit and accepts up to 500 position IDs per call. Any position owner exercising normal lifecycle operations more than once per year is affected. The more actively a user manages their position, the greater the cumulative loss.

---

### Recommendation

Carry the fractional remainder forward by storing the untruncated accrued bonus in the position state (e.g., an `OwedBonusRemainder sdk.Dec` field), and add it to the next segment's computation before truncating. Alternatively, track `LastBonusAccrual` at sub-second precision and accumulate the Dec value across segments before the final `TruncateInt()`, only resetting the checkpoint to the truncation boundary rather than the exact block time. This mirrors the fix described in the referenced report: floor the accrual timestamp to the nearest unit boundary so that the fractional period is not discarded.

---

### Proof of Concept

**Setup:** Position with 1,000,000 tokens, 4% APY, bonded validator, `tokensPerShare = 1.0`.

**Single annual claim (correct):**
```
durationSeconds = 31,557,600
bonus = floor(1,000,000 * 1.0 * 0.04 * 31,557,600 / 31,557,600) = floor(40,000.0) = 40,000
```

**Two claims of 6 months each (loss):**
```
claim 1: durationSeconds = 15,778,800
  bonus = floor(1,000,000 * 1.0 * 0.04 * 15,778,800 / 31,557,600) = floor(20,000.0) = 20,000
  LastBonusAccrual → T + 6 months

claim 2: durationSeconds = 15,778,800
  bonus = floor(20,000.0) = 20,000
  total = 40,000  (no loss here because the division is exact)
```

**365 daily claims (significant loss):**
```
per claim: durationSeconds = 86,400
  bonus = floor(1,000,000 * 0.04 * 86,400 / 31,557,600) = floor(109.50...) = 109
  total = 109 * 365 = 39,785  (loss = 215 tokens)
```

**6,307,200 per-block claims (total loss):**
```
per claim: durationSeconds = 5
  bonus = floor(1,000,000 * 0.04 * 5 / 31,557,600) = floor(0.00634) = 0
  total = 0  (loss = 40,000 tokens, 100%)
```

The root cause is `TruncateInt()` at line 45 of `x/tieredrewards/keeper/bonus_rewards.go` combined with `UpdateLastBonusAccrual(accrualEnd)` at line 20 of the same file discarding the fractional remainder on every settlement. [8](#0-7) [2](#0-1)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}
