### Title
`max_contract_size` Shared Between Local and Global Contract Deployment Validation ‚Äî (`File: runtime/runtime/src/action_validation.rs`)

### Summary

The single `LimitConfig::max_contract_size` parameter is used to enforce the size ceiling for both `DeployContractAction` (local, per-account contract) and `DeployGlobalContractAction` (global, cross-shard contract). These two actions have fundamentally different storage models, fee structures, and cross-shard blast radii. Any protocol upgrade that changes `max_contract_size` for local-contract reasons simultaneously and silently changes the ceiling for global contracts, which carry a much larger network-wide cost because each global contract is distributed to every shard via `GlobalContractDistributionReceipt`.

### Finding Description

In `runtime/runtime/src/action_validation.rs`, both validation functions read the same field:

```rust
// local contract
fn validate_deploy_contract_action(limit_config: &LimitConfig, action: &DeployContractAction) {
    if action.code.len() as u64 > limit_config.max_contract_size { ... }
}

// global contract
fn validate_deploy_global_contract_action(limit_config: &LimitConfig, action: &DeployGlobalContractAction) {
    if action.code.len() as u64 > limit_config.max_contract_size { ... }
}
```

The same coupling exists in the host-function path: `runtime/near-vm-runner/src/wasmtime_runner/logic.rs` line 2710 reads `ctx.config.limit_config.max_contract_size` for `promise_batch_action_deploy_global_contract`, and the VM logic test at `runtime/near-vm-runner/src/logic/tests/miscs.rs` lines 484‚Äì498 explicitly exercises both `promise_batch_action_deploy_contract` and `promise_batch_action_deploy_global_contract` against the same limit.

The protocol designers already recognized that global contracts have different economics: protocol version 77 introduced a separate `global_contract_storage_amount_per_byte` fee, and separate `action_deploy_global_contract` / `action_deploy_global_contract_per_byte` gas fees. No analogous separation was made for the size limit.

A global contract is not stored per-account. After deployment it is propagated to every shard via a chain of `GlobalContractDistributionReceipt` messages, one per shard. The payload of each such receipt is the full contract bytecode. With the current 4 MiB limit and six shards, a single max-size global contract deployment generates ~24 MiB of cross-shard receipt data. If `max_contract_size` is raised (e.g., to 8 MiB to accommodate larger local contracts), the global contract ceiling rises identically, multiplying the cross-shard overhead without any independent review of whether that is safe for the global-contract path.

### Impact Explanation

**Protocol-upgrade boundary interaction.** `max_contract_size` lives in `LimitConfig` inside `RuntimeConfig`, which is versioned by protocol version. Every future upgrade that adjusts `max_contract_size` for local-contract reasons will silently change the global-contract ceiling. The two ceilings serve different invariants:

- Local contract ceiling: bounds per-account trie storage and wasm compilation cost on one shard.
- Global contract ceiling: bounds the payload of `GlobalContractDistributionReceipt` messages that must be processed on *every* shard, multiplying the network-wide cost by the shard count.

**Unprivileged exploitation.** Once a protocol upgrade raises `max_contract_size`, any account with sufficient NEAR to cover `global_contract_storage_amount_per_byte √ó code_len` can deploy a near-maximum-size global contract. The storage fee does not price the cross-shard distribution overhead (compute and bandwidth on every shard). Multiple concurrent large global-contract deployments can saturate the delayed-receipt queue on receiving shards, degrading throughput for all users.

**Broken domain invariant.** The invariant "the global-contract size ceiling is independently tunable to reflect cross-shard distribution cost" is broken. Operators and protocol engineers cannot tighten the global-contract ceiling without also tightening the local-contract ceiling, and vice versa.

### Likelihood Explanation

Global contracts are live on mainnet (introduced in protocol version 77 / release 2.6.0). The `max_contract_size` parameter has been changed before (from 4 MiB to 1.5 MiB and back in various diffs). Any future upgrade that revisits this limit ‚Äî for performance, security, or feature reasons ‚Äî will trigger the coupling. The exploitation path (deploying a large global contract after the limit is raised) requires no special privilege beyond holding enough NEAR for the storage fee.

### Recommendation

Introduce a separate `max_global_contract_size` field in `LimitConfig` (and a corresponding `Parameter::MaxGlobalContractSize` entry in the parameter table). Initialize it to the current `max_contract_size` value (4,194,304 bytes) so there is no behavior change at stabilization. Use `max_global_contract_size` exclusively in `validate_deploy_global_contract_action` and in the host-function path (`promise_batch_action_deploy_global_contract_impl`). This mirrors the existing pattern where `global_contract_storage_amount_per_byte` is already a separate, independently tunable parameter.

/// Validates `DeployContractAction`. Checks that the given contract size doesn't exceed the limit.
fn validate_deploy_contract_action(
    limit_config: &LimitConfig,
    action: &DeployContractAction,
) -> Result<(), ActionsValidationError> {
    if action.code.len() as u64 > limit_config.max_contract_size {
        return Err(ActionsValidationError::ContractSizeExceeded {
            size: action.code.len() as u64,
            limit: limit_config.max_contract_size,
        });
    }

    Ok(())
}

#[test]
fn test_contract_size_limit() {
    let mut logic_builder = VMLogicBuilder::default();
    let limit = 1024;
    logic_builder.config.limit_config.max_contract_size = limit;
    let mut logic = logic_builder.build();

    let account_id = logic.internal_mem_write(b"alice");

    let promise_id = logic
        .promise_batch_create(account_id.len, account_id.ptr)
        .expect("Number of promises is under the limit");

    logic
        .promise_batch_action_deploy_contract(promise_id, limit, 0)
        .expect("The length of the contract code is under the limit");
    assert_eq!(
        logic.promise_batch_action_deploy_contract(promise_id, limit + 1, 0),
        Err(HostError::ContractSizeExceeded { size: limit + 1, limit }.into())
    );

    logic
        .promise_batch_action_deploy_global_contract(promise_id, limit, 0)
        .expect("The length of the contract code is under the limit");
    assert_eq!(
        logic.promise_batch_action_deploy_global_contract(promise_id, limit + 1, 0),
        Err(HostError::ContractSizeExceeded { size: limit + 1, limit }.into())
    );

pub(crate) fn action_deploy_global_contract(
    state_update: &mut TrieUpdate,
    account: &mut Account,
    account_id: &AccountId,
    apply_state: &ApplyState,
    deploy_contract: &DeployGlobalContractAction,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let _span = tracing::debug_span!(target: "runtime", "action_deploy_global_contract").entered();

    let storage_cost = apply_state
        .config
        .fees
        .storage_usage_config
        .global_contract_storage_amount_per_byte
        .saturating_mul(deploy_contract.code.len() as u128);
    let Some(updated_balance) = account.amount().checked_sub(storage_cost) else {
        result.result = Err(ActionErrorKind::LackBalanceForState {
            account_id: account_id.clone(),
            amount: storage_cost,
        }
        .into());
        return Ok(());
    };
    result.tokens_burnt =
        result.tokens_burnt.checked_add(storage_cost).ok_or(IntegerOverflowError)?;
    account.set_amount(updated_balance);

    initiate_distribution(
        state_update,
        account_id.clone(),
        deploy_contract.code.clone(),
        &deploy_contract.deploy_mode,
        apply_state.shard_id,
        result,
    )?;

    Ok(())
}
