### Title
Dogecoin AuxPoW Enforcement Missing: Block Without AuxPoW Data Accepted After AuxPoW Activation — (`contract/src/dogecoin.rs`)

### Summary

The `submit_block_header` function for the Dogecoin build accepts block headers without AuxPoW data even when the block's version field has the AuxPoW flag set, and even after the Dogecoin AuxPoW activation height. The contract falls back to validating the Dogecoin block's own scrypt hash against the target, which is a weaker check that real Dogecoin nodes would never apply to post-activation blocks. A relayer supplying adversarial chain data can submit a structurally invalid Dogecoin block that passes the contract's PoW check but would be unconditionally rejected by every real Dogecoin node.

### Finding Description

In `contract/src/dogecoin.rs`, `submit_block_header` branches on whether `aux_data` is `Some` or `None`:

```rust
if let Some(ref aux_data) = aux_data {
    self.check_aux(&block_header, aux_data);
} else {
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        format!("block should have correct pow")
    );
}
``` [1](#0-0) 

When `aux_data` is `None`, the contract only checks that the Dogecoin block's own scrypt hash is below the target. There is no check that:

1. **If `version & BLOCK_VERSION_AUXPOW != 0`, then `aux_data` must be `Some`.** The `check_aux` function enforces the forward direction (AuxPoW data present → version flag must be set), but the reverse is never enforced. [2](#0-1) 

2. **After Dogecoin AuxPoW activation (block 371337), all blocks must carry AuxPoW data.** The `check_pow` function validates difficulty, MTP, and future-timestamp rules, but contains no check that the AuxPoW flag is present or that AuxPoW data is required. [3](#0-2) 

The `BlockHeader` type for Dogecoin is `(Header, Option<AuxData>)`, so the caller fully controls whether AuxPoW data is supplied. [4](#0-3) 

### Impact Explanation

A relayer supplying adversarial data can submit a Dogecoin block header with `aux_data = None` (or with `version & 0x100 != 0` but `aux_data = None`) after AuxPoW activation. If the block's own scrypt hash is below the current Dogecoin target, the contract stores it as a valid block. Real Dogecoin nodes would reject this block unconditionally because it lacks the required AuxPoW coinbase commitment. The contract's canonical chain state diverges from the real Dogecoin chain. Once a forged block is stored in `headers_pool` and promoted to the mainchain, it can be used as the anchor for `verify_transaction_inclusion` or `verify_transaction_inclusion_v2`, causing those functions to return `true` for transactions that were never confirmed on the real Dogecoin chain. [5](#0-4) 

The corrupted canonical mapping is: `mainchain_height_to_header[height] → forged_block_hash` and `headers_pool[forged_block_hash] → ExtendedHeader` with an attacker-controlled `merkle_root`, enabling arbitrary SPV proof forgery.

### Likelihood Explanation

The `submit_blocks` entry point is gated by `#[trusted_relayer]`. [6](#0-5) 

If the trusted-relayer system is staking-based and permissionless (any account that stakes can register), the attacker only needs to stake and then find a Dogecoin-difficulty scrypt solution — no Bitcoin-level hash power is required, because the AuxPoW parent-block check is bypassed entirely. This is the direct analog to the original report's finding that an attacker with mining-pool-level hash power can forge blocks the contract accepts but nodes reject. If the relayer system requires DAO approval, the attack requires a compromised or malicious relayer, which reduces likelihood but does not eliminate it given the value of cross-chain bridge contracts.

### Recommendation

1. In `submit_block_header` (Dogecoin build), add an explicit check: if `block_header.version & BLOCK_VERSION_AUXPOW != 0`, then `aux_data` must be `Some`; panic otherwise.
2. In `check_pow` (Dogecoin build), after the AuxPoW activation height, require that `block_header.version & BLOCK_VERSION_AUXPOW != 0` so that non-AuxPoW blocks are rejected at the PoW-check stage regardless of whether `aux_data` is supplied.
3. Consider adding a minimum block-height threshold below which non-AuxPoW blocks are accepted and above which they are always rejected, mirroring the Dogecoin protocol rule.

### Proof of Concept

1. Deploy the Dogecoin build of the contract and initialize it with a real Dogecoin genesis segment.
2. Register as a trusted relayer (or use an account with `Role::UnrestrictedSubmitBlocks`).
3. Construct a `(Header, Option<AuxData>)` where `aux_data = None`, `version = 0x00000100` (AuxPoW flag set), and all other fields are chosen so that `scrypt(header_bytes) < target_from_bits(bits)`.
4. Call `submit_blocks` with this header.
5. Observe that the block is accepted and stored in `headers_pool` and `mainchain_height_to_header` despite having no AuxPoW coinbase commitment — a block that every real Dogecoin node at height > 371337 would reject.
6. Construct a fake Merkle proof against the attacker-controlled `merkle_root` in the forged header and call `verify_transaction_inclusion`; observe it returns `true`. [7](#0-6)

    pub(crate) fn submit_block_header(
        &mut self,
        header: (Header, Option<AuxData>),
        skip_pow_verification: bool,
    ) {
        let (block_header, aux_data) = header;

        let prev_block_header = self.get_prev_header(&block_header);
        let current_block_hash = block_header.block_hash();

        if !skip_pow_verification {
            self.check_target(&block_header, &prev_block_header);

            if let Some(ref aux_data) = aux_data {
                self.check_aux(&block_header, aux_data);
            } else {
                let pow_hash = block_header.block_hash_pow();
                // Check if the block hash is less than or equal to the target
                require!(
                    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
                    format!("block should have correct pow")
                );
            }
        }

        let (current_block_computed_chain_work, overflow) = prev_block_header
            .chain_work
            .overflowing_add(work_from_bits(block_header.bits));
        require!(!overflow, "Addition of U256 values overflowed");

        let current_header = ExtendedHeader {
            block_header: block_header.clone().into_light(),
            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        self.submit_block_header_inner(current_header, &prev_block_header);
    }

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
