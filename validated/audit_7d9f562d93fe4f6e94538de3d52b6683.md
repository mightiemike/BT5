Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any non-allowlisted address to bypass the deposit gate — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual token-providing caller) and checks only whether the LP-position `owner` is allowlisted. Any address can call `pool.addLiquidity(owner=allowlistedAddress, ...)`, pass the guard, and deposit tokens into a restricted pool without ever being allowlisted itself.

## Finding Description
`MetricOmmPool.addLiquidity` captures `msg.sender` and passes it as `sender` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes and forwards `sender` faithfully to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` declares its first parameter unnamed and never reads it. Its guard checks `allowedDepositor[msg.sender][owner]` — i.e., whether the LP-position recipient is allowlisted — not whether the token provider (`sender`) is allowlisted: [3](#0-2) 

After the extension check passes, `LiquidityLib.addLiquidity` calls `IMetricOmmModifyLiquidityCallback(msg.sender).metricOmmModifyLiquidityCallback(...)` — i.e., the callback is invoked on the actual caller (`sender`), who provides the tokens: [4](#0-3) 

LP shares are minted to `owner` (keyed by `_positionBinKey(owner, salt, bin)`), not to `sender`: [5](#0-4) 

**Attack path:**
1. Pool is deployed with `DepositAllowlistExtension` as an extension in `BEFORE_ADD_LIQUIDITY_ORDER`.
2. Pool admin allowlists address A (`allowedDepositor[pool][A] = true`). Attacker B is not allowlisted.
3. Attacker B calls `pool.addLiquidity(owner=A, salt, deltas, callbackData, extensionData)`.
4. Pool calls `extension.beforeAddLiquidity(sender=B, owner=A, ...)`.
5. Extension checks `allowedDepositor[pool][A] == true` → passes. B is never checked.
6. Pool executes `LiquidityLib.addLiquidity(owner=A, ...)`, invoking the callback on B to pull tokens, and minting LP shares to A.
7. B has deposited into a restricted pool without being allowlisted; A receives unwanted LP shares.

No existing guard prevents this. The `allowAllDepositors` short-circuit is irrelevant when it is `false` (the gating case). The `removeLiquidity` path enforces `msg.sender == owner`, but `addLiquidity` has no such check.

## Impact Explanation
The `DepositAllowlistExtension` is the sole mechanism for restricting who may deposit into a pool. The bypass is complete: any address can deposit tokens into an allowlist-gated pool by nominating any allowlisted address as `owner`. The actual token provider (`sender`) is never validated. This breaks the core security invariant of the extension — that only allowlisted addresses may provide liquidity — and allows unrestricted token inflows into pools that pool admins intended to gate. The allowlisted `owner` also receives LP shares they did not request, creating an unwanted position (griefing vector).

## Likelihood Explanation
The entrypoint `addLiquidity` is fully public (`external`). No privileged role, special token, or deployment control is required. The attacker only needs to know one allowlisted address for the target pool, which is trivially discoverable on-chain via `AllowedToDepositSet` events. Exploitation requires a single transaction from any EOA or contract that can implement `metricOmmModifyLiquidityCallback`.

## Recommendation
Check `sender` (the actual token provider), not `owner` (the LP-position recipient):

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```solidity
// Setup: pool with DepositAllowlistExtension, only address A allowlisted
extension.setAllowedToDeposit(address(pool), A, true);
// allowedDepositor[pool][B] == false

// Attack: B calls addLiquidity with owner=A
// B must implement IMetricOmmModifyLiquidityCallback and hold tokens
vm.prank(B);
pool.addLiquidity(
    A,          // owner = allowlisted address
    salt,
    deltas,
    callbackData,
    extensionData
);

// Assert: deposit succeeded, LP shares minted to A, B was never checked
assertGt(positionBinShares[keccak256(abi.encode(A, salt, binIdx))], 0);
assertFalse(extension.allowedDepositor(address(pool), B)); // B still not allowlisted
```
