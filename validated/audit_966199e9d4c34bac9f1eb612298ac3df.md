### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` When `openInterest` Is Zero Permanently Blocks Liquidation Finalization — (`File: core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` divides by `state.openInterest` to compute a per-share funding adjustment when socializing a liquidatee's residual negative `vQuoteBalance`. If no other participants hold open positions in that perp market at the time of finalization, `openInterest` is zero and the division reverts with `"DBZ"`, permanently blocking the finalization of the insolvent subaccount.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after exhausting the insurance fund against a liquidatee's negative `vQuoteBalance`, the protocol attempts to spread the remaining loss across all open-interest holders:

```solidity
if (balance.vQuoteBalance < 0) {
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ← reverts with "DBZ" when == 0
    );
    state.cumulativeFundingLongX18 += fundingPerShare;
    state.cumulativeFundingShortX18 -= fundingPerShare;
    balance.vQuoteBalance = 0;
}
``` [1](#0-0) 

`state.openInterest` is the sum of absolute position sizes across all participants. Before `socializeSubaccount` is entered, `_finalizeSubaccount` in `ClearinghouseLiq.sol` enforces that the liquidatee's own `balance.amount == 0`:

```solidity
require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
``` [2](#0-1) 

`getStateAndBalance` inside `socializeSubaccount` calls `_updateBalance(state, balance, 0, 0)`, which subtracts `balance.amount.abs() == 0` from `openInterest` and adds back 0. Therefore the returned `state.openInterest` reflects only *other* participants' positions — not the liquidatee's. [3](#0-2) 

If no other participant holds an open position in that perp market, `state.openInterest == 0` and `MathSD21x18.div` reverts unconditionally:

```solidity
function div(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        require(y != 0, ERR_DIV_BY_ZERO);
``` [4](#0-3) 

The call chain is:

```
Endpoint → Clearinghouse.liquidateSubaccount
         → ClearinghouseLiq.liquidateSubaccountImpl (delegatecall)
         → _finalizeSubaccount (productId == type(uint32).max)
         → perpEngine.socializeSubaccount        ← reverts here
``` [5](#0-4) [6](#0-5) 

Because the revert propagates all the way up, the finalization transaction always fails. There is no alternative code path to clear the liquidatee's negative `vQuoteBalance` or to mark the subaccount as resolved.

---

### Impact Explanation

- The insolvent subaccount can never be finalized. Its negative `vQuoteBalance` constitutes permanent, unresolvable bad debt in the protocol.
- The `availableSettle` pool for that perp market is never credited, so other participants' positive PnL settlement is permanently impaired.
- The insurance fund has already been partially or fully consumed by the time the revert occurs, so the loss is real and unrecoverable.
- The liquidatee's subaccount remains in an unhealthy state indefinitely; no liquidator can complete the process.

---

### Likelihood Explanation

The condition requires two simultaneous facts:

1. **`openInterest == 0` at finalization time** — the liquidatee was the sole open-position holder in that perp market, or all other participants closed their positions before the finalization call. This is realistic in low-liquidity or newly-listed perp markets, or during a market-wide deleveraging event.
2. **Insurance is insufficient** — the insurance fund cannot fully cover the liquidatee's negative `vQuoteBalance`. This is a documented protocol risk scenario (the entire socialization path exists precisely for this case).

Neither condition requires privileged access. Any unprivileged liquidator calling `liquidateSubaccount` with `productId == type(uint32).max` triggers the path.

---

### Recommendation

Guard the socialization branch against zero `openInterest`. If `openInterest == 0` there are no counterparties to absorb the loss; the remaining bad debt should either be written off against the insurance fund entirely (even if it goes negative) or deferred until new open interest exists. A minimal fix:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No counterparties; absorb into insurance or mark as irrecoverable
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

The exact policy for the zero-`openInterest` case (e.g., deduct from insurance, emit an event, allow negative insurance) should be decided by the protocol team, but the division must not be attempted unconditionally.

---

### Proof of Concept

1. Deploy the protocol. Add a perp product (e.g., `productId = 1`).
2. Alice opens a long position of size `S` at price `P`. `openInterest = S`.
3. Price drops; Alice's position goes deeply underwater (`vQuoteBalance << 0`).
4. Alice closes her position (`amount → 0`). `openInterest = 0`.
5. Alice's `vQuoteBalance` remains negative (unrealized loss not yet settled).
6. Insurance fund is empty (or insufficient to cover Alice's full loss).
7. Liquidator calls `liquidateSubaccount` with `productId = type(uint32).max` to finalize Alice.
8. `_finalizeSubaccount` passes all `require(balance.amount == 0)` checks.
9. `perpEngine.socializeSubaccount(alice, 0)` is called.
10. Inside, `balance.vQuoteBalance < 0` and `state.openInterest == 0`.
11. `MathSD21x18.div` reverts with `"DBZ"`.
12. The entire liquidation transaction reverts. Alice's subaccount is permanently unresolvable. [7](#0-6)
