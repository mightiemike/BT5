### Title
Missing zero-address validation on `recipient` in `unwrapWETH9` causes permanent ETH loss — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.unwrapWETH9()` forwards unwrapped native ETH to a caller-supplied `recipient` without validating `recipient != address(0)`. A low-level `.call{value: ...}("")` to `address(0)` succeeds on the EVM, so the ETH is silently burned with no revert.

---

### Finding Description

`unwrapWETH9` withdraws all WETH held by the router, converts it to native ETH via `IWETH9.withdraw`, and then calls `_transferETH(recipient, balanceWETH)`: [1](#0-0) 

`_transferETH` performs a bare low-level call: [2](#0-1) 

On the EVM, `address(0).call{value: v}("")` returns `(true, "")`. The `ok` check passes, no revert is raised, and the ETH is permanently burned. Neither `unwrapWETH9` nor `_transferETH` contains a `recipient != address(0)` guard.

This is the direct analog of FETH M-09: a withdrawal function that transfers funds to a caller-supplied address without validating it is non-zero.

Note: the parallel `sweepToken` function has the same missing check on `recipient`, but for in-scope ERC-20 tokens (USDC, USDT, WETH) the underlying `transfer` reverts on `address(0)`, so that path does not produce a reachable loss. The ETH path in `unwrapWETH9` has no such safety net.

---

### Impact Explanation

Any ETH unwrapped through this path is permanently destroyed if `recipient == address(0)`. This is a direct, irreversible loss of user principal. The amount lost equals the full WETH balance held by the router at the time of the call, which in a multicall context can equal the full swap output of the preceding step.

---

### Likelihood Explanation

Low. The user must supply `address(0)` explicitly. Realistic triggers include: UI bugs that fail to populate the recipient field, copy-paste errors in scripts, or a misconfigured `multicall` payload where the recipient argument is left as the zero default. The FETH audit committee accepted the same likelihood profile as medium.

---

### Recommendation

Add a zero-address guard at the entry of `unwrapWETH9`:

```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
+   if (recipient == address(0)) revert InvalidRecipient();
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    ...
}
```

Optionally apply the same guard to `sweepToken` for defence-in-depth, even though standard tokens revert independently.

---

### Proof of Concept

1. User constructs a `multicall` on `MetricOmmSimpleRouter`: first a swap that outputs WETH to the router, then `unwrapWETH9(0, address(0))`.
2. After the swap, the router holds `balanceWETH > 0`.
3. `IWETH9(WETH).withdraw(balanceWETH)` converts WETH → ETH; router now holds native ETH.
4. `_transferETH(address(0), balanceWETH)` executes `address(0).call{value: balanceWETH}("")`.
5. EVM returns `(true, "")`. The `if (!ok)` branch is not taken.
6. Function returns normally. The user receives zero ETH; `balanceWETH` wei is permanently burned.
