### Title
Wrong Transaction Hash in RPC Simulation for V3 Transactions with Zero L2/L1DataGas Bounds Due to Incorrect Version Classification in `ResourceBoundsMapping` → `ValidResourceBounds` Conversion - (`File: crates/apollo_rpc/src/v0_8/transaction.rs`)

---

### Summary

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in `crates/apollo_rpc/src/v0_8/transaction.rs` uses a zero-value sentinel check to classify V3 transactions as either pre-0.13.3 (`L1Gas`) or post-0.13.3 (`AllResources`). A V3 transaction with zero `l1_data_gas` and `l2_gas` bounds is silently misclassified as `L1Gas`. Because `get_tip_resource_bounds_hash` produces structurally different hash preimages for the two variants (3 vs 4 Poseidon inputs), the hash computed in the RPC simulation path diverges from the hash the signer committed to, causing `starknet_simulateTransactions` with `skip_validate = false` to reject a cryptographically valid transaction.

---

### Finding Description

**Version-discriminating conversion (the root cause)**

`crates/apollo_rpc/src/v0_8/transaction.rs` defines a flat `ResourceBoundsMapping` struct (always carrying all three fields) and converts it to the internal `ValidResourceBounds` enum using a zero-value sentinel:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← pre-0.13.3 branch
        } else {
            Self::AllResources(AllResourceBounds { … })  // ← post-0.13.3 branch
        }
    }
}
```

Zero bounds are used as the sentinel for "this is a pre-0.13.3 transaction." However, zero bounds are also a perfectly legal value for a post-0.13.3 transaction that simply does not consume L2 gas or L1 data gas.

**Hash divergence (the consequence)**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` produces structurally different Poseidon preimages for the two variants:

```rust
// L1Gas  → 3 inputs: [tip, L1_GAS_concat, L2_GAS_concat]
// AllResources → 4 inputs: [tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
```

`get_concat_resource` encodes the resource *name* bytes into the felt even when the amount and price are zero, so the L1_DATA_GAS element is non-zero even for zero bounds. The two hashes are therefore always distinct.

**The divergent pipeline**

| Path | Struct used | Classification | Hash preimage |
|---|---|---|---|
| Gateway submission (`RpcTransaction`) | `AllResourceBounds` | Always `AllResources` | 4 inputs |
| RPC v0.8 simulation (`BroadcastedInvokeTransactionV3`) | `ResourceBoundsMapping` | `L1Gas` when l2=0 ∧ l1data=0 | 3 inputs |

The user signs the 4-input hash (gateway path). The simulation path computes the 3-input hash. Signature verification fails.

This is the direct analog of the AaveV3 bundler bug: a sentinel value (`type(uint256).max` there; zero bounds here) is forwarded as-is to a downstream system that interprets it differently, producing a wrong result instead of the intended "use the available value" behavior.

---

### Impact Explanation

**High — RPC simulation returns an authoritative-looking wrong value.**

`starknet_simulateTransactions` with `skip_validate = false` will return a signature-verification failure for any valid V3 transaction whose `l2_gas` and `l1_data_gas` bounds are both zero. The response is indistinguishable from a genuinely invalid transaction, misleading callers (wallets, SDKs, dApps) into believing their transaction is malformed when it is not.

Additionally, this falls under **High — Transaction conversion or signature/hash logic binds the wrong hash**: the conversion permanently binds the wrong hash variant to the transaction before it reaches the blockifier, affecting any downstream logic that depends on the hash (tracing, fee estimation with validation enabled).

---

### Likelihood Explanation

**Medium.** Any V3 transaction that sets only `l1_gas` bounds and leaves `l2_gas` and `l1_data_gas` at zero (a common pattern for simple transfers or calls that do not require explicit L2/data-gas budgets) will trigger this misclassification. The gateway test suite already exercises this exact case (`valid_l1_gas` test case with only `l1_gas` non-zero), confirming it is a supported and expected transaction shape.

---

### Recommendation

Remove the zero-value sentinel heuristic from the `From<ResourceBoundsMapping> for ValidResourceBounds` conversion. Since the RPC v0.8 API exclusively handles V3 transactions (post-0.13.3), the conversion should unconditionally produce `AllResources`:

```rust
// Before
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)
        } else {
            Self::AllResources(AllResourceBounds { … })
        }
    }
}

// After
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        Self::AllResources(AllResourceBounds {
            l1_gas: value.l1_gas,
            l1_data_gas: value.l1_data_gas,
            l2_gas: value.l2_gas,
        })
    }
}
```

The same fix should be applied to the protobuf converter in `crates/apollo_protobuf/src/converters/transaction.rs`, which contains an identical zero-value sentinel pattern.

---

### Proof of Concept

1. Construct a V3 invoke transaction:
   - `l1_gas = { max_amount: 1000, max_price_per_unit: 1 }`
   - `l2_gas = { max_amount: 0, max_price_per_unit: 0 }`
   - `l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }`

2. Compute the canonical hash via the gateway path (`AllResourceBounds` → `AllResources`):
   - Poseidon preimage: `[tip, L1_GAS_concat, L2_GAS_concat=0, L1_DATA_GAS_concat≠0]` (4 inputs; L1_DATA_GAS_concat is non-zero because the resource name bytes are included)

3. Sign the transaction with this hash.

4. Submit to `starknet_simulateTransactions` with `skip_validate = false` via the RPC v0.8 API.

5. The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion fires:
   - `l1_data_gas.is_zero() && l2_gas.is_zero()` → `true`
   - Classification: `L1Gas` ← **wrong**

6. `get_tip_resource_bounds_hash` computes the hash with 3 inputs: `[tip, L1_GAS_concat, L2_GAS_concat=0]` — the L1_DATA_GAS element is absent.

7. The computed hash differs from the signed hash → signature verification fails → simulation returns an error for a cryptographically valid transaction.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
