### Title
`DelegateAction` (Meta-Transaction) Signature Domain Lacks Chain ID â€” Cross-Network Replay Attack - (`core/primitives/src/action/delegate.rs`)

---

### Summary

`DelegateAction::get_nep461_hash()` produces the signing payload as `SHA256(borsh(discriminant_u32 || delegate_action_fields))`. Neither the `MessageDiscriminant` (a bare NEP number) nor any field of `DelegateAction` contains a chain ID or genesis hash. A `SignedDelegateAction` produced on testnet is cryptographically valid on mainnet whenever the same account, public key, and a compatible nonce exist on both networks. A malicious relayer can lift a user's off-chain-signed `DelegateAction` from one NEAR network and submit it on another, executing arbitrary inner actions (transfers, key mutations, contract deployments) against the user's mainnet account without their consent.

---

### Finding Description

`DelegateAction` (NEP-366) and `DelegateActionV2` (NEP-611) are the meta-transaction primitives. A user signs the action off-chain; a relayer wraps it in an outer `SignedTransaction` and submits it on-chain. The user's signature covers only the inner `DelegateAction` payload; the outer transaction's `block_hash` is chain-specific, but the inner signature is not.

**Signing path** (`core/primitives/src/action/delegate.rs`):

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
```

`SignableMessage` is:

```rust
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,  // u32 = 2^30 + 366
    pub msg: &'a T,                         // the DelegateAction fields
}
```

The Borsh-serialized payload is therefore:

```
[discriminant: u32 LE] || [sender_id] || [receiver_id] || [actions] || [nonce: u64] || [max_block_height: u64] || [public_key]
```

No chain ID, no genesis hash, no network identifier appears anywhere in this byte string. The same is true for `DelegateActionV2` via `SignableMessageType::DelegateActionV2` (discriminant `2^30 + 611`).

**Verification path** (`runtime/runtime/src/actions.rs`):


None of the four checks (`verify`, `max_block_height`, `sender_id`, `validate_delegate_action_key`) involves a chain ID. The runtime has no mechanism to reject a `SignedDelegateAction` whose signature was produced for a different NEAR network.

### Impact Explanation

**High / Critical.** A malicious relayer who receives a user's `SignedDelegateAction` for testnet can submit it verbatim on mainnet. The inner actions execute with `sender_id` as the predecessor, so the user's mainnet account is the origin of the receipt. Possible consequences include:

- Draining NEAR token balance via `Transfer`
- Adding or deleting access keys (`AddKey` / `DeleteKey`), enabling persistent account takeover
- Deploying or replacing contract code (`DeployContract`)
- Calling arbitrary contract methods (`FunctionCall`) with the user's identity

The outer `SignedTransaction` is signed by the relayer and is chain-specific (it includes a `block_hash`), so the relayer pays gas. The user's loss is the execution of the inner actions on the wrong network.

---

### Likelihood Explanation

**Medium.** The preconditions are:

1. The same named account exists on both mainnet and testnet (extremely common â€” most developers mirror their account names).
2. The same public key is registered on both networks (common for accounts created with the same seed phrase or wallet).
3. The nonce on mainnet for that key must be exactly `nonce_in_action - 1`. This is the binding constraint, but it is satisfied trivially for fresh accounts (nonce = 0 â†’ first `DelegateAction` with nonce = 1 is valid on both networks) and for accounts whose nonces happen to align.
4. `max_block_height` must not have passed on mainnet. Since block heights on mainnet and testnet are independent and both grow at ~1 block/second, a `max_block_height` set to "current height + 100" on testnet is almost certainly still valid on mainnet.

The attack requires a malicious relayer, which is the exact threat model meta-transactions are designed to tolerate for gas payment but not for cross-network execution.

---

### Recommendation

Include the chain ID (or genesis block hash) in the signed payload. The minimal fix is to add it to `SignableMessage` or to `DelegateAction` itself:

```rust
// Option A: add chain_id to SignableMessage
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,
    pub chain_id: String,   // e.g. "mainnet", "testnet"
    pub msg: &'a T,
}
```

```rust
// Option B: add chain_id field to DelegateAction (protocol change)
pub struct DelegateAction {
    pub chain_id: String,
    pub sender_id: AccountId,
    ...
}
```

Option A is less invasive (no struct change to `DelegateAction`) but requires a new `SignableMessageType` variant (new discriminant) to avoid breaking existing signatures. Option B is a clean protocol-level fix but requires a new `DelegateAction` version. Either way, `apply_delegate_action` must verify the chain ID matches the executing network before accepting the signature.

```rust
    pub fn verify(&self) -> bool {
```
