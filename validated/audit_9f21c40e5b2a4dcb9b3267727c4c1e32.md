### Title
`SpiceContractCodeResponse` Accepted Without Sender Signature Verification — (`File: chain/client/src/spice/chunk_validator_actor.rs`)

---

### Summary

The SPICE path's `SpiceContractCodeResponse` message carries no producer signature and the receiver (`handle_spice_contract_code_response`) performs no sender-identity check. Any network peer can send a `SpiceContractCodeResponse` to a chunk validator; the validator will accept and act on it. The non-SPICE path explicitly added `ProtocolFeature::SignedContractCodeResponse` / `ContractCodeResponseV2` to close exactly this gap, but the SPICE path never received the equivalent treatment.

---

### Finding Description

**Non-SPICE path (protected):**

`ContractCodeResponse` has a `V2` variant that carries a validator signature over a Borsh-serialized inner struct. `validate_contract_code_response` checks `ProtocolFeature::SignedContractCodeResponse` and calls `validate_witness_contract_code_response_signature`, which verifies the responder is a chunk producer for the shard and that the signature is valid. [1](#0-0) [2](#0-1) [3](#0-2) 

**SPICE path (unprotected):**

`SpiceContractCodeResponse` is defined as a single unsigned variant `V1(SpiceContractCodeResponseV1)` containing only `chunk_id` and `compressed_contracts`. There is no `signature` field, no `verify_signature` method, and no `responder` field. [4](#0-3) 

`handle_spice_contract_code_response` in `SpiceChunkValidatorActor` decompresses the contracts and resolves them against all pending chunks with zero sender-identity checks: [5](#0-4) 

The `chunk_id` field carried in the response is also never checked against the pending chunk the validator is actually waiting on — contracts are matched purely by hash across all pending chunks.

**Contrast with the request side:** `SpiceContractCodeRequest` IS signed and the distributor actor verifies the signature before processing: [6](#0-5) 

---

### Impact Explanation

Any network peer — not just a legitimate chunk producer — can send a `SpiceContractCodeResponse` to a chunk validator. The validator will:

1. Decompress the payload.
2. Compute `CodeHash(hash(&contract.0))` for each received contract.
3. Remove matching hashes from `trusted.missing` and push the bytes into `entry.contracts`.
4. Trigger `try_assemble_and_validate_chunk` once all missing hashes are resolved.

The content-addressed hash check (`trusted.missing.remove(hash)`) prevents injection of bytes that do not hash to an expected value, so a preimage attack is required to inject wrong contract code. However:

- **Broken authentication invariant**: validators cannot distinguish a legitimate chunk-producer response from one sent by an arbitrary peer. The security model that only chunk producers should supply contract code is violated.
- **`chunk_id` ignored**: the `chunk_id` embedded in the response is never validated against the chunk the validator is waiting on. A response for chunk A can resolve contracts for chunk B if the same contract hash appears in both.
- **Front-running / liveness interference**: a malicious peer that has observed the contract code on-chain can race the legitimate producer and resolve the validator's pending contracts, bypassing the trusted-sender tracking entirely.

---

### Likelihood Explanation

The SPICE feature is gated by `protocol_feature_spice` but is reachable by any peer that can send a `T1MessageBody::SpiceContractCodeResponse` routed message. The routing-layer signature only authenticates the routing hop, not the content. Any validator node running with the Spice feature enabled is exposed. The attack requires only knowledge of the contract bytes (publicly available from chain state) and the ability to send a routed network message.

---

### Recommendation

Add a producer signature to `SpiceContractCodeResponse` following the same pattern as `ContractCodeResponseV2`:

1. Add `responder: AccountId` and `signature: Signature` fields (or an inner/outer struct split) to `SpiceContractCodeResponseV1` (or introduce a `V2`).
2. Sign the response in `handle_spice_contract_code_request` using the node's `ValidatorSigner`.
3. In `handle_spice_contract_code_response`, verify that the responder is a chunk producer for the relevant shard and that the signature is valid before accepting any contract bytes.
4. Validate that the `chunk_id` in the response matches the chunk the validator is waiting on.

    response: &ContractCodeResponse,

    let Some(responder) = response.responder() else {

pub struct ContractCodeResponseV2 {
    inner: ContractCodeResponseV2Inner,
    /// Signature of the inner, signed by the responder.
    signature: Signature,
}

impl ContractCodeResponseV2 {
    pub fn encode(
        next_chunk: ChunkProductionKey,
        contracts: &Vec<CodeBytes>,
        signer: &ValidatorSigner,
    ) -> std::io::Result<Self> {
        let inner =
            ContractCodeResponseV2Inner::encode(next_chunk, contracts, signer.validator_id())?;
        let signature = signer.sign_bytes(&borsh::to_vec(&inner).unwrap());
        Ok(Self { inner, signature })
    }

    fn verify_signature(&self, public_key: &PublicKey) -> bool {
        self.signature.verify(&borsh::to_vec(&self.inner).unwrap(), public_key)
    }
}

#[borsh(use_discriminant = true)]
#[repr(u8)]
pub enum SpiceContractCodeResponse {
    V1(SpiceContractCodeResponseV1) = 0,
}

impl SpiceContractCodeResponse {
    pub fn encode(chunk_id: SpiceChunkId, contracts: &Vec<CodeBytes>) -> std::io::Result<Self> {
        SpiceContractCodeResponseV1::encode(chunk_id, contracts).map(|v1| Self::V1(v1))
    }

    pub fn chunk_id(&self) -> &SpiceChunkId {
        match self {
            Self::V1(v1) => &v1.chunk_id,
        }
    }

    pub fn decompress_contracts(&self) -> std::io::Result<Vec<CodeBytes>> {
        let compressed_contracts = match self {
            Self::V1(v1) => &v1.compressed_contracts,
        };
        compressed_contracts.decode().map(|(data, _size)| data)
    }
}

    /// Handles a contract code response from a chunk producer.
    /// Resolves received contracts across all pending chunks (not just the chunk_id
    /// in the response), since multiple chunks may be waiting on the same contract.
    fn handle_spice_contract_code_response(
        &mut self,
        response: SpiceContractCodeResponse,
    ) -> Result<(), Error> {
        let contracts = response.decompress_contracts()?;

        // Map code_hash -> bytes for the received contracts.
        let mut received: HashMap<CodeHash, CodeBytes> = HashMap::new();
        for contract in contracts {
            let hash = CodeHash(hash(&contract.0));
            received.insert(hash, contract);
        }

        // Resolve across all pending chunks that are waiting on any of these contracts.
        let mut maybe_ready: Vec<SpiceChunkId> = Vec::new();
        for (chunk_id, entry) in &mut self.partial_chunk_data {
            let Some(trusted) = &mut entry.trusted else {
                continue;
            };

            let mut resolved_any = false;
            for (hash, bytes) in &received {
                if trusted.missing.remove(hash) {
                    entry.contracts.push(bytes.clone());
                    resolved_any = true;
                }
            }
            if resolved_any && trusted.missing.is_empty() {
                maybe_ready.push(chunk_id.clone());
            }
        }

        let signer = self
            .validator_signer
            .get()
            .ok_or_else(|| Error::NotAValidator("no signer".to_owned()))?;
        for chunk_id in maybe_ready {
            self.try_assemble_and_validate_chunk(&chunk_id, signer.clone())?;
        }
        Ok(())
    }
