### Title
Wrong Ancestor Lookup in Fork Difficulty Calculation Allows Difficulty Manipulation — (`contract/src/dogecoin.rs`, `get_next_work_required`)

---

### Summary

`get_next_work_required` in the Dogecoin build calls `get_header_by_height(height_first)` to obtain the timestamp of the retarget boundary block. The implementation of `get_header_by_height` unconditionally reads from `mainchain_height_to_header`, which only contains mainchain blocks. When validating a **fork** block, this returns the mainchain block at that height rather than the fork's actual ancestor. A trusted relayer can exploit this by crafting a fork whose ancestor timestamps differ from the mainchain blocks at the same heights, causing `expected_bits` to be computed from the wrong timestamp and allowing fork blocks with incorrect (easier) difficulty to pass `check_pow`.

---

### Finding Description

**Root cause — `get_header_by_height` is mainchain-only:** [1](#0-0) 

`get_header_by_height` reads exclusively from `mainchain_height_to_header`. Fork blocks are stored only in `headers_pool` via `store_fork_header`: [2](#0-1) 

They are never inserted into `mainchain_height_to_header`. So for any fork block, `get_header_by_height(height_first)` returns the **mainchain** block at that height, not the fork ancestor.

**The acknowledged bug site:** [3](#0-2) 

The TODO comment at line 291 explicitly flags this as unverified behavior.

**Why it fires on every Dogecoin mainnet fork block (height ≥ 145,000):**

For `prev_block_header.block_height >= 145_000`, `difficulty_adjustment_interval` is set to `1`: [4](#0-3) 

With interval = 1, the condition `(prev_height + 1) % 1 != 0` is never true, so execution always falls through to the retarget path. `blocks_to_go_back` becomes `1`, so `height_first = prev_height - 1`. This means **every** fork block at height ≥ fork_point + 2 triggers the wrong-ancestor lookup.

**Timestamp divergence enables difficulty manipulation:**

Suppose the fork diverges at height H. The attacker controls the timestamp of fork block F_H (subject only to MTP and the 2-hour future cap). When validating fork block F_H+2:

- `prev_block_header` = F_H+1 (height H+1)
- `height_first` = H
- `get_header_by_height(H)` returns **M_H** (mainchain block at H), not **F_H**

`calculate_next_work_required` then computes:

```
modulated_timespan = T_{F_H+1} - T_{M_H}
```

instead of the correct:

```
modulated_timespan = T_{F_H+1} - T_{F_H}
``` [5](#0-4) 

If the attacker sets `T_{F_H}` significantly later than `T_{M_H}` (e.g., by setting F_H's timestamp near the 2-hour future cap), then `T_{F_H+1} - T_{M_H}` is much larger than `T_{F_H+1} - T_{F_H}`, producing a higher (easier) target. The attacker then sets `F_H+2.bits` to this easier value and finds valid PoW for it.

**`check_pow` accepts the manipulated block:** [6](#0-5) 

`expected_bits == block_header.bits` passes because both sides use the wrong (easier) value. The PoW hash check also passes because the attacker mined against the easier target.

**Chain promotion path:** [7](#0-6) 

If the fork's `chain_work` exceeds the mainchain's, `reorg_chain` is called and `mainchain_tip_blockhash` is updated to the fork tip. Subsequent `verify_transaction_inclusion` calls then operate against the fraudulent fork.

---

### Impact Explanation

The light client accepts a fork chain whose blocks carry difficulty values that are invalid under correct Dogecoin consensus rules. Once promoted via `reorg_chain`, the fraudulent fork becomes the canonical chain. Any downstream consumer of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will verify transactions against a chain that did not satisfy the correct proof-of-work requirement. This breaks the core security invariant of the light client: that only chains satisfying Dogecoin's actual difficulty rules are accepted.

---

### Likelihood Explanation

The entry point is `submit_blocks`, gated by `#[trusted_relayer]`: [8](#0-7) 

A trusted relayer is a participant who has staked and been approved through the normal protocol process — not a DAO or admin role. A malicious or compromised trusted relayer can submit crafted fork headers with manipulated timestamps. The bug fires on every Dogecoin mainnet fork block at height ≥ fork_point + 2, and the TODO comment confirms the developers themselves identified this as an open correctness question. The Digishield modulation caps the per-block difficulty shift to ±25–50%, but the attacker can compound the effect across multiple fork blocks since each retargets independently.

---

### Recommendation

Replace the `get_header_by_height` call with an ancestor walk that follows `prev_block_hash` links from `prev_block_header` backward by `blocks_to_go_back` steps. This traversal uses `get_prev_header`, which reads from `headers_pool` and correctly follows the fork's own ancestry regardless of what the mainchain maps contain. The same fix is needed in `bitcoin.rs` and `litecoin.rs` for consistency, though those chains have longer retarget intervals that reduce the practical exposure.

---

### Proof of Concept

**State setup (Dogecoin mainnet, height ≥ 145,001):**

1. Mainchain has blocks M_{H-1}, M_H, M_{H+1} with timestamps T_{M_{H-1}}, T_{M_H}, T_{M_{H+1}}.
2. Attacker (trusted relayer) submits fork block F_H with `prev_hash = M_{H-1}.hash` and timestamp `T_{F_H} = T_{M_H} + Δ` where Δ is large (e.g., 7,000 seconds, within the 2-hour future cap). F_H passes `check_pow` because `height_first = H-2`, and `get_header_by_height(H-2)` returns the shared ancestor M_{H-2} — correct at this step.
3. Attacker submits F_{H+1} with `prev_hash = F_H.hash`. Difficulty check uses `get_header_by_height(H-1)` = M_{H-1} — still the shared ancestor, still correct.
4. Attacker submits F_{H+2} with `prev_hash = F_{H+1}.hash`. Difficulty check calls `get_header_by_height(H)` = **M_H** (not F_H). The computed timespan is `T_{F_{H+1}} - T_{M_H}` instead of `T_{F_{H+1}} - T_{F_H}`. Since `T_{F_H} = T_{M_H} + Δ`, the correct timespan is `Δ` shorter, meaning the correct difficulty is harder. The wrong `expected_bits` (easier) is accepted.
5. Attacker mines F_{H+2} against the easier target, sets `F_{H+2}.bits` to the wrong `expected_bits`, and submits. `check_pow` passes.
6. If the fork's cumulative `chain_work` exceeds the mainchain's, `reorg_chain` promotes the fork to `mainchain_tip_blockhash`.

**Verification:** In a unit test, construct two `ExtendedHeader` chains sharing an ancestor at H-1 but diverging at H with different timestamps. Call `get_next_work_required` for a block at H+2 on the fork and observe that `expected_bits` differs from what is produced when the correct fork ancestor at H is used as `first_block_time`.
