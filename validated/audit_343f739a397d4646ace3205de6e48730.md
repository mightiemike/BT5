### Title
`burnNlp` Uses Maintenance Health Check Instead of Initial, Allowing Liquidation-Eligible Users to Extract NLP Collateral — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.burnNlp()` enforces only a **maintenance** health check after burning NLP tokens, while `mintNlp()` enforces an **initial** health check. A user whose initial health is negative (eligible for liquidation) but whose maintenance health is still non-negative can burn their unlocked NLP tokens and receive quote in return, reducing the NLP collateral available to liquidators before any liquidation is processed.

---

### Finding Description

`mintNlp` and `burnNlp` apply asymmetric health thresholds:

`mintNlp` (line 479–482) enforces `INITIAL` health:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

`burnNlp` (lines 526–529) enforces only `MAINTENANCE` health:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) [2](#0-1) 

The NLP lock period (`NLP_LOCK_PERIOD`) is the only temporal gate on burning. Once NLP is unlocked, `burnNlp` can be called at any time. The `getNlpUnlockedBalance` / `tryUnlockNlpBalance` mechanism only checks whether the lock period has elapsed since mint — it is backward-looking from the mint time, not forward-looking from when the account entered the initial-maintenance health gap. [3](#0-2) [4](#0-3) 

The `burnNlp` execution path:
1. Burns `nlpAmount` from the user's NLP balance.
2. Credits `quoteAmount` (oracle price × amount, minus a small fee) to the user's quote balance.
3. Checks only `MAINTENANCE` health. [5](#0-4) 

The sequencer dispatches `BurnNlp` via `EndpointTx.processTransactionImpl` (lines 554–573), which validates the user's EIP-712 signature and nonce, then calls `clearinghouse.burnNlp`. No additional health gate exists between the sequencer and the clearinghouse. [6](#0-5) 

---

### Impact Explanation

A user whose initial health is negative (i.e., `getHealth(sender, INITIAL) < 0`) is eligible for liquidation. Because `burnNlp` only checks maintenance health, this user can:

1. Submit a `BurnNlp` transaction while their initial health is negative.
2. Receive quote tokens in exchange for NLP at oracle price.
3. Reduce the NLP collateral that liquidators would otherwise seize.

The quote received stays in the subaccount and cannot be withdrawn while initial health remains negative (since `withdrawCollateral` enforces `INITIAL` health). However, the conversion from NLP to quote changes the composition of the collateral pool: NLP tokens that would have been liquidated at a discount are replaced by quote at full oracle value, potentially improving the user's initial health enough to then withdraw the quote via `withdrawCollateral`. [7](#0-6) 

This is the direct analog to the RPL report: just as a node operator could withdraw RPL before slashing was reported (because the cooldown only checked the last stake block, not a forward-looking delay), a Nado user can burn NLP before liquidation is processed because the health check uses the weaker maintenance threshold, not the initial threshold that governs liquidation eligibility.

---

### Likelihood Explanation

**Medium.** The attack requires:
- The user to hold NLP tokens whose lock period has already elapsed.
- The user's account to be in the initial-maintenance health gap (initial health < 0, maintenance health ≥ 0).

Both conditions are realistic in a live market: a user who minted NLP some time ago and whose position has since deteriorated due to price moves will naturally satisfy both. The user has a clear economic incentive to burn NLP before a liquidator processes their account, since liquidation applies a discount.

---

### Recommendation

Replace the `MAINTENANCE` health check in `burnNlp` with an `INITIAL` health check, consistent with `mintNlp` and `withdrawCollateral`:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

This ensures that a user who is eligible for liquidation cannot reduce their NLP exposure by burning tokens before the liquidation is processed. [8](#0-7) 

---

### Proof of Concept

1. Alice mints NLP tokens at time T. The lock period expires at T + `NLP_LOCK_PERIOD`.
2. At time T + `NLP_LOCK_PERIOD` + ε, a price move causes Alice's initial health to drop below 0 (liquidation eligible), but her maintenance health remains ≥ 0.
3. Before any liquidator submits a `LiquidateSubaccount` transaction, Alice submits a `BurnNlp` transaction for her full unlocked NLP balance.
4. The sequencer processes `BurnNlp`: `getNlpUnlockedBalance` returns Alice's full NLP balance (lock period elapsed), the NLP is burned, and quote is credited at oracle price minus the 0.1% fee.
5. The maintenance health check at line 526–529 passes.
6. Alice's NLP collateral is gone; liquidators now find a smaller or differently-composed position to liquidate, potentially at reduced profit or with the liquidation no longer viable if the quote credit pushed initial health back above 0. [9](#0-8) [10](#0-9)

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,

        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);

            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,

    function handleNlpLockedBalance(bytes32 subaccount, int128 amountDelta)
        internal
    {
        _assertInternal();

        // N_ACCOUNT is not limited by lock period
        if (subaccount == N_ACCOUNT) return;

        tryUnlockNlpBalance(subaccount);
        if (amountDelta > 0) {
            NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
                subaccount
            ];
            if (
                queue.balanceCount > 0 &&
                queue.balances[queue.balanceCount - 1].unlockedAt ==
                getOracleTime() + NLP_LOCK_PERIOD
            ) {
                queue
                    .balances[queue.balanceCount - 1]
                    .balance
                    .amount += amountDelta;
            } else {
                queue.balances[queue.balanceCount] = NlpLockedBalance({
                    balance: Balance({amount: amountDelta}),
                    unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
                });
                queue.balanceCount++;
            }
        } else if (amountDelta < 0) {
            Balance memory balanceSum = nlpLockedBalanceQueues[subaccount]
                .unlockedBalanceSum;
            balanceSum.amount += amountDelta;
            nlpLockedBalanceQueues[subaccount].unlockedBalanceSum = balanceSum;
        }
    }
