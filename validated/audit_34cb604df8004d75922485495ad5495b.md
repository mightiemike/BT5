### Title
`nonDefaultFeeTierMask` Bits Can Only Be Set, Never Cleared — Once a Fee Tier Is Marked Non-Default It Cannot Be Reverted to the Protocol Default Fee Schedule - (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

In `OffchainExchange.sol`, the `nonDefaultFeeTierMask` bitmask is updated with a bitwise-OR assignment (`|=`) inside `updateTierFeeRates`. This means once a tier's bit is set, it can never be cleared. The protocol can never revert a tier to the hardcoded default fee schedule (0 maker / 2 bps taker). Any new product added after a tier is marked non-default will silently serve 0 fees for users in that tier, because `feeRates[tier][newProduct]` is uninitialized (zero) and the default fallback path is permanently bypassed.

---

### Finding Description

`getTierFeeRateX18` uses `nonDefaultFeeTierMask` as a gate to decide whether to return the custom stored rates or the hardcoded default:

```solidity
// OffchainExchange.sol line 938
if (nonDefaultFeeTierMask & (1 << tier) != 0) {
    return feeRates[tier][productId];   // custom path
}
return FeeRates({ makerRateX18: 0, takerRateX18: 200_000_000_000_000 }); // default 2 bps
```

The bit for a tier is set inside `updateTierFeeRates`:

```solidity
// OffchainExchange.sol line 990
nonDefaultFeeTierMask |= uint128(1) << txn.tier;
```

`|=` can only set bits. There is no corresponding `&= ~(...)` path, no reset function, and no other write to `nonDefaultFeeTierMask` anywhere in the codebase. Once bit `N` is set, `getTierFeeRateX18(N, productId)` will always take the custom path and return `feeRates[N][productId]`, even if that mapping entry is zero (uninitialized for a newly listed product). ---

### Impact Explanation

**Fee revenue loss on newly listed products.** The concrete scenario:

1. Admin calls `updateTierFeeRates` for tier `T`, product `A` → bit `T` is set in `nonDefaultFeeTierMask`, `feeRates[T][A]` is written.
2. A new product `B` is later listed on the exchange.
3. `feeRates[T][B]` is never written (zero by default).
4. Every trade on product `B` by a user in tier `T` calls `getUserFeeRateWithBuilder` → `getTierFeeRateX18(T, B)` → bit `T` is set → returns `feeRates[T][B]` = `{0, 0}`.
5. Those users pay **0 taker fees** instead of the intended 2 bps default.

The admin cannot fix this by clearing the bit — the only recourse is to explicitly call `updateTierFeeRates` for every new product for every affected tier, which is an operational burden that is easy to miss and silently wrong until noticed. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

The Nado protocol lists new products over time. Any tier that has ever had custom rates set (a routine admin operation) will silently apply 0 fees to every new product unless the admin remembers to re-run `updateTierFeeRates` for each new product × each affected tier. The `QUOTE_PRODUCT_ID` bulk-update path only covers products that exist at call time; products listed afterward are not covered. The likelihood of at least one new product being listed after a tier is customized is high. [5](#0-4) 

---

### Recommendation

Replace the one-way `|=` with a direct assignment that reflects the intended semantics. If the intent is to mark a tier as non-default whenever any rate is written, the bit should still be set — but a companion function (or an additional parameter) should allow clearing the bit when the admin wants to revert a tier to the default schedule:

```diff
- nonDefaultFeeTierMask |= uint128(1) << txn.tier;
+ // Set or clear the bit based on whether the rates are non-zero / non-default
+ if (txn.makerRateX18 != 0 || txn.takerRateX18 != DEFAULT_TAKER_RATE) {
+     nonDefaultFeeTierMask |= uint128(1) << txn.tier;
+ } else {
+     nonDefaultFeeTierMask &= ~(uint128(1) << txn.tier);
+ }
```

Alternatively, expose a dedicated `resetTierToDefault(uint32 tier)` admin function that clears the bit and deletes the stored rates.

---

### Proof of Concept

1. Deploy `OffchainExchange`. `nonDefaultFeeTierMask` = 0.
2. Call `updateTierFeeRates({tier: 1, productId: 1, makerRateX18: 0, takerRateX18: 100_000_000_000_000})` (1 bps custom rate for tier 1, product 1). `nonDefaultFeeTierMask` = `0b10` (bit 1 set).
3. List new product with `productId = 2`. `feeRates[1][2]` = `{0, 0}` (never written).
4. Call `getTierFeeRateX18(1, 2)`. Bit 1 is set → returns `feeRates[1][2]` = `{0, 0}`.
5. A user in tier 1 trades product 2 as taker → `feeRate` = 0 → pays **zero fees**.
6. Expected: the default 2 bps taker fee should apply since no custom rate was ever set for product 2.
7. There is no function call that can clear bit 1 from `nonDefaultFeeTierMask` to restore the default path. [6](#0-5) [7](#0-6)
