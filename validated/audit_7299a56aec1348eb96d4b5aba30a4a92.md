Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, nullifying per-pool swap allowlists for any router-routed swap — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. A pool admin who allowlists the router so their curated users can swap through it inadvertently grants every user on the network the ability to bypass the allowlist, completely nullifying the restriction.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

When the call originates from `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the router contract address.

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged to the extension.** [2](#0-1) 

**Step 3 — The extension checks `sender` (the router) against the allowlist.** [3](#0-2) 

`msg.sender` here is the pool (correct), but `sender` is the router address for any user who entered through `MetricOmmSimpleRouter`.

**Step 4 — The router is always `msg.sender` for every swap path.**

- `exactInputSingle`: router calls `pool.swap()` directly. [4](#0-3) 

- `exactInput`: every hop is called by the router. [5](#0-4) 

- `exactOutputSingle` and `exactOutput` follow the same pattern. [6](#0-5) 

In all paths, `pool.swap()` sees `msg.sender = router`, so `sender` delivered to `beforeSwap` is always the router address, never the end user.

**Step 5 — The admin faces an impossible choice.**

To allow their allowlisted users to swap through the public router, the admin must call:
```solidity
extension.setAllowedToSwap(pool, address(router), true);
``` [7](#0-6) 

Once the router is allowlisted, `allowedSwapper[pool][router] = true`, and the check at line 37 passes for **every** user who routes through the router — the per-user allowlist is completely nullified.

**Contrast with `DepositAllowlistExtension`.**

The deposit extension correctly checks `owner` (the position holder), not `sender` (the liquidity adder contract): [8](#0-7) 

The swap extension has no equivalent separation — it only receives `sender` (the direct caller) and has no access to the originating user's address.

## Impact Explanation
A curated pool that deploys `SwapAllowlistExtension` to restrict trading to a specific set of addresses loses that protection entirely for any user who routes through `MetricOmmSimpleRouter`. Disallowed users can execute swaps at oracle-derived prices, draining LP value or front-running allowlisted participants on pools designed to be closed to the public. This is a direct loss of LP principal and a broken core pool invariant (the allowlist). The impact meets the "Broken core pool functionality causing loss of funds" and "Admin-boundary break" criteria.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint in the periphery. A pool admin who configures a swap allowlist and also wants their allowlisted users to use the router will naturally allowlist the router address — there is no documentation warning against this, and the deposit extension's correct `owner`-based check creates a false sense of symmetry. The trigger requires only a standard admin configuration action followed by any public user calling the router. No privileged attacker capability is required.

## Recommendation
Pass the originating user's address through the extension mechanism so the allowlist can gate the economically relevant actor:

1. **Preferred**: Have the router encode the originating user in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that address when `sender` is a known router.
2. **Alternative**: Add a `recipient` check in addition to `sender`, since `recipient` is the address that receives output tokens and is set by the user in `exactInputSingle`/`exactOutputSingle`.
3. **Structural**: Mirror the deposit extension's design — expose a separate `originator` field in the swap hook signature so extensions always have access to the true initiating address regardless of routing depth.

## Proof of Concept
```solidity
// Pool admin sets up a curated pool with SwapAllowlistExtension.
// Only `allowedUser` is supposed to be able to swap.
extension.setAllowedToSwap(pool, allowedUser, true);

// Admin also allowlists the router so allowedUser can use it conveniently.
extension.setAllowedToSwap(pool, address(router), true);

// Now ANY user bypasses the allowlist:
// bannedUser calls the router — pool sees msg.sender = router, check passes.
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: bannedUser,
    zeroForOne: true,
    amountIn: 1_000e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    extensionData: "",
    deadline: block.timestamp
}));
// Swap executes successfully despite bannedUser not being on the allowlist.
```
