Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the end user, allowing any caller to bypass the swap allowlist via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which the pool sets to its own `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is the pool's `msg.sender`, so the extension checks whether the **router** is allowlisted rather than the actual end user. If the pool admin allowlists the router (the natural step to let curated users use the standard periphery), every unprivileged user can bypass the swap allowlist by calling the router directly.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to `_beforeSwap`:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)`: [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged to every configured extension:** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`**, where `msg.sender` is the pool and `sender` is the value forwarded above: [3](#0-2) 

**Step 4 — `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly**, making the router the pool's `msg.sender`: [4](#0-3) 

At that point, `sender` forwarded to the extension is the **router address**, not the end user. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. Once the router is allowlisted, the guard passes for every caller of the router regardless of their own allowlist status.

**Contrast with `DepositAllowlistExtension`:** The deposit extension avoids this problem by checking the `owner` argument — the position owner explicitly supplied by the caller — rather than `sender`. The swap extension has no equivalent; it must rely on `sender`, which collapses to the router for all router-mediated swaps. [5](#0-4) 

## Impact Explanation

A pool admin who deploys a `SwapAllowlistExtension` intends to restrict swaps to a named set of addresses. To let those addresses use the standard periphery they must allowlist the router. Once `allowedSwapper[pool][router] == true`, the guard passes for **every** caller of the router regardless of their own allowlist status. Any unprivileged user can execute swaps against the pool's LP liquidity, draining LP value through oracle-priced trades that the pool admin intended to block. This constitutes broken core pool functionality (allowlist bypass) causing direct loss of LP assets — a valid impact under the contest rules.

## Likelihood Explanation

The scenario requires the pool admin to allowlist the router — a natural and expected configuration step whenever the allowlist is meant to permit a curated set of users to trade via the standard periphery. There is no on-chain signal that warns the admin this also opens the gate to all other router users. The bypass requires no special privileges, no malicious setup, and no non-standard tokens; any EOA can call `exactInputSingle` on the public router.

## Recommendation

The extension must gate on the **economic actor**, not the immediate pool caller. Two options:

1. **Check `recipient` instead of `sender`** — the recipient is the address that receives output tokens and is the economically relevant party. The pool already passes `recipient` as the second argument to `beforeSwap`.

2. **Require the router to forward the originating user** — add an `originSender` field to `extensionData` that the router populates with `msg.sender`, and have the extension decode and check that value. This requires a coordinated change to the router and the extension, but preserves the intended "payer" semantics.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)      // alice is the intended grantee
  admin calls setAllowedToSwap(pool, router, true)     // router allowlisted so alice can use periphery

Attack:
  charlie (not in allowlist) calls:
    router.exactInputSingle({pool: pool, ..., recipient: charlie})

Execution trace:
  router.exactInputSingle()
    → pool.swap(recipient=charlie, ...)          // msg.sender to pool = router
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (guard passes)
      → swap executes, charlie receives tokens

Result:
  charlie swaps successfully despite never being allowlisted.
  The SwapAllowlistExtension guard is completely ineffective for
  any user who routes through MetricOmmSimpleRouter.
```
