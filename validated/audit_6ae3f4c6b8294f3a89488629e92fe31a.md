### Title
Gas-key exec-fee host-function path uses borsh wire length instead of on-trie hash length for ML-DSA-65 keys, causing ~59× overcharge — (`runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

When a WASM contract calls `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, or `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 public key, the **exec-fee** is computed using the raw borsh-encoded wire length of the key (1953 bytes) instead of the on-trie hash-form length (33 bytes). The transaction path correctly normalizes to `trie_id_len()` (33 bytes) before calling the same fee helpers. The host-function path does not, producing a ~59× overcharge on the per-byte exec component and breaking the protocol invariant that the same action costs the same gas regardless of invocation path.

---

### Finding Description

ML-DSA-65 access keys are stored in the trie as a SHA3-256 hash (`[tag=3] || 32-byte digest`, 33 bytes total), not as the full 1952-byte pubkey. `PublicKey::trie_id_len()` returns 33 for `MLDSA65`; `PublicKey::len()` returns 1953.

**Transaction path** — `exec_fee` in `runtime/runtime/src/config.rs` correctly normalizes:

```rust
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
```

and `permission_exec_fees` uses `public_key.trie_id_len()` for `AddKey` with gas-key permissions.

**Host-function path** — `promise_batch_action_add_gas_key_with_full_access` in `runtime/near-vm-runner/src/logic/logic.rs` passes the raw caller-supplied `public_key_len` directly:

```rust
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← borsh wire length (1953), not trie_id_len (33)
    num_nonces,
);
```

`gas_key_add_key_exec_fee` in `core/parameters/src/cost.rs` then computes:

```rust
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len) + size_of::<NonceIndex>();
let per_byte = cfg.fee(ActionCosts::gas_key_byte).exec_fee()
    .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
    .unwrap()
    .checked_mul(num_nonces)
    .unwrap();
```

With `public_key_len = 1953` the nonce key length is inflated by 1920 bytes per nonce versus the correct 33-byte hash form. The same mismatch applies to `gas_key_transfer_exec_fee` called from `promise_batch_action_transfer_to_gas_key`.

The decoded `PublicKey` is available in the same function (used later in `public_key.decode()?` when appending the action) but its `trie_id_len()` is never consulted for fee purposes.

The existing parity test `test_gas_key_fee_parity` in `test-loop-tests/src/tests/gas_keys.rs` only exercises ED25519 keys, for which `len() == trie_id_len() == 33`, so the divergence is invisible there.

---

### Impact Explanation

Any WASM contract that adds or funds an ML-DSA-65 gas key via the host-function API is charged ~59× more exec gas than the equivalent transaction. The per-byte exec fee is multiplied by `(account_id_len + 1953 + NonceIndex_size + NONCE_VALUE_LEN) * num_nonces` instead of the correct `(account_id_len + 33 + NonceIndex_size + NONCE_VALUE_LEN) * num_nonces`. This:

1. Breaks the protocol invariant that identical on-chain state changes cost identical gas regardless of invocation path.
2. Makes ML-DSA-65 gas keys effectively unusable from contracts (prohibitively expensive exec gas).
3. Can cause legitimate cross-contract calls to fail with out-of-gas errors when the caller budgets based on the expected (transaction-path) cost.

---

### Likelihood Explanation

`PostQuantumSignatures` is assigned protocol version 85, which is below the current stable version 86, so the feature is live on mainnet. Any deployed contract can call the affected host functions. The attacker-controlled input is the borsh-encoded ML-DSA-65 public key passed to the host function; no privileged role is required.

---

### Recommendation

After decoding the public key in each affected host function, use `decoded_key.trie_id_len()` instead of `public_key_len as usize` when calling `gas_key_add_key_exec_fee` and `gas_key_transfer_exec_fee`. For example, in `promise_batch_action_add_gas_key_with_full_access`:

```rust
let decoded = public_key.decode()?;
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded.trie_id_len(),   // ← normalized to on-trie length
    num_nonces,
);
// reuse `decoded` when appending the action
self.ext.append_action_add_gas_key_with_full_access(receipt_idx, decoded, num_nonces);
```

Apply the same fix to `promise_batch_action_add_gas_key_with_function_call` and `promise_batch_action_transfer_to_gas_key`. Extend `test_gas_key_fee_parity` to cover `KeyType::MLDSA65` to prevent regression.

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }

fn access_key_storage_usage(

    public_key: &PublicKey,
    access_key: &AccessKey,
) -> StorageUsage {
    let storage_usage_config = &fee_config.storage_usage_config;
    // Use the on-trie identifier length, not the borsh-serialized pubkey
