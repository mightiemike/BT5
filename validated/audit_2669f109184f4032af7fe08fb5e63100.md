### Title
Tier `BonusApy` Change Applied Retroactively to All Unclaimed Bonus Periods — (`x/tieredrewards/keeper/claim_rewards.go`, `x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

When governance updates a tier's `BonusApy` via `SetTier`, the new rate is silently applied to the **entire unclaimed bonus history** of every existing position in that tier. Because `processEventsAndClaimBonus` fetches the current tier at claim time and passes it unchanged into every historical segment computation, no checkpoint is recorded at the moment of the parameter change. This is the direct analog of the AMM rate desynchronization: a policy parameter changes, but the dependent per-position accrual state is never snapshotted, so the new rate bleeds backward into periods it should not govern.

---

### Finding Description

`processEventsAndClaimBonus` in `x/tieredrewards/keeper/claim_rewards.go` fetches the tier once at the top of the function and then passes it into `computeSegmentBonus` for every historical validator-event segment:

```go
tier, err := k.getTier(ctx, pos.TierId)   // current tier, fetched at claim time
...
for _, entry := range events {
    evt := entry.Event
    if bonded {
        bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
        // tier.BonusApy is the *current* value, applied to a *historical* segment
    }
    ...
}

`computeSegmentBonus` multiplies the delegation shares by `tier.BonusApy` for the full segment duration:

`SetTier`, which is the write path used by governance messages, genesis, and chain upgrades, stores the new tier directly without first checkpointing accrued rewards for existing positions:

```go
// SetTier writes a tier after validation. Used by governance messages, genesis, and chain upgrades.

    if err := tier.Validate(); err != nil {
        return err
    }
    if err := k.Tiers.Set(ctx, tier.Id, tier); err != nil { ... }
    return nil

``` [3](#0-2) 

The helper `claimRewardsAndUpdateTierPositions` exists but is a separate, explicitly-invoked function — it is not called inside `SetTier`, so no automatic checkpoint occurs at the moment of the parameter change. [4](#0-3) 

---

### Impact Explanation

**If `BonusApy` is lowered by governance:** every position holder who has not claimed since before the change will have the new lower rate applied retroactively to the entire unclaimed window. Rewards that were legitimately earned at the old higher rate are silently reduced. The position holder suffers a direct, unrecoverable loss of bonus tokens proportional to `(oldRate − newRate) × shares × tokensPerShare × unclaimed_duration`.

**If `BonusApy` is raised by governance:** every position holder receives a windfall for the entire unclaimed window at the new higher rate. The `RewardsPoolName` module account is drained faster than the governance proposal intended, potentially exhausting the pool and causing `ErrInsufficientBonusPool` for later claimants.

The corrupted value is the `bonusCoins` amount transferred from `types.RewardsPoolName` to the position owner inside `processEventsAndClaimBonus`. [5](#0-4) 

---

### Likelihood Explanation

Governance proposals to adjust tier incentive rates (`BonusApy`) are a routine protocol operation as market conditions change. Any governance participant can submit such a proposal. The effect is automatic and silent — neither the governance tooling nor the on-chain message handler warns that existing unclaimed positions will be retroactively re-rated. The longer a position holder has gone without claiming, the larger the discrepancy. This is a realistic, high-frequency scenario for an active staking rewards protocol.

---

### Recommendation

Before writing the new tier value in `SetTier`, call `claimRewardsAndUpdateTierPositions` for the affected tier ID. This checkpoints all accrued bonus at the old rate before the new rate takes effect, ensuring each time segment is priced at the rate that was actually in force during that segment. If the number of positions makes an inline sweep impractical, record a `BonusApyChangeEvent` in the validator-event log (analogous to how slash/bond/unbond events are recorded) so that `processEventsAndClaimBonus` can apply the correct rate to each historical segment during the next claim.

---

### Proof of Concept

1. Governance sets tier 1 `BonusApy = 0.20` (20 % APY). Alice locks 1 000 000 basecro into tier 1 at block T₀ and does not claim for 180 days.
2. At day 90, governance passes a proposal lowering tier 1 `BonusApy` to `0.05` (5 % APY). `SetTier` writes the new value; no checkpoint is recorded for Alice's position.
3. At day 180, Alice calls `ClaimTierRewards`. `processEventsAndClaimBonus` fetches the current tier (`BonusApy = 0.05`) and applies it to the full 180-day window.
   - **Expected:** 90 days × 20 % + 90 days × 5 % ≈ 49 315 + 12 329 = **61 644 basecro**
   - **Actual:** 180 days × 5 % ≈ **24 658 basecro**
   - **Alice's loss:** ≈ 36 986 basecro, silently transferred to the rewards pool instead of Alice.

The inverse scenario (rate raised) causes the pool to pay out ≈ 36 986 basecro more than governance intended, draining `RewardsPoolName` ahead of schedule and potentially triggering `ErrInsufficientBonusPool` for other claimants. [6](#0-5) [7](#0-6)

func (k Keeper) claimRewardsAndUpdateTierPositions(ctx context.Context, tierId uint32) error {

		}

			return err
		}
	}

	return nil

func (k Keeper) processEventsAndClaimBonus(ctx context.Context, pos *types.PositionState) (sdk.Coins, error) {
	// Rewards should have been claimed before undelegation
	if !pos.IsDelegated() {
		return sdk.NewCoins(), nil
	}

	valAddr, err := sdk.ValAddressFromBech32(pos.Delegation.ValidatorAddress)
	if err != nil {
		return nil, err
	}

	events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	blockTime := sdkCtx.BlockTime()

	totalBonus := math.ZeroInt()
	// Use the persisted bonded state from the last replay, not a hardcoded default.
	// This prevents overpaying bonus for unbonded gaps between claims.
	bonded := pos.LastKnownBonded
	segmentStart := pos.LastBonusAccrual

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	for _, entry := range events {
		evt := entry.Event

		if bonded {
			// Compute bonus for the bonded segment [segmentStart, eventTime]
			// using the snapshot rate at the event.
			bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
			totalBonus = totalBonus.Add(bonus)

func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}
