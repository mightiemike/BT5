### Title
`limbs_for_int` Omits Sign-Extension Byte, Causing Cost Undercharge and Size-Limit Bypass in `op_multiply` — (`File: src/more_ops.rs`)

---

### Summary

`limbs_for_int` computes the canonical byte-length of a `Number` (BigInt) as `ceil(bits / 8)`. This is correct for negative numbers and for positive numbers whose most-significant bit is 0, but it is **off by one** for positive numbers whose most-significant bit is 1 (e.g. 0x80, 0xFF, 0x8000, 0xFFFF, …). Those values require a leading `0x00` sign-extension byte in CLVM's canonical signed-big-endian encoding, making their actual atom length `ceil(bits / 8) + 1`. `op_multiply` uses the underestimated value as `l0` — the running product size — for both the per-step cost formula and the 1024-byte size-limit guard. The result is deterministic cost undercharging and a one-byte bypass of the product-size cap, both reachable by any attacker who can submit a CLVM program.

---

### Finding Description

**Root cause — `limbs_for_int` (src/more_ops.rs:100-102)**

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

`BigInt::bits()` returns the number of bits required to represent the **absolute value**. For the positive integer 128 (0x80), `bits()` = 8, so `limbs_for_int` returns 1. But the canonical CLVM atom for 128 is `[0x00, 0x80]` — two bytes — because the high bit is set and a leading zero is mandatory to prevent the value from being interpreted as −128. This is confirmed by `new_number` in `src/allocator.rs:704-722`, which strips redundant leading zeros but **preserves** the leading zero when `slice[1] & 0x80 == 0x80`, and by the `test_new_number` cases:

```
0x80  → [0x00, 0x80]   (2 bytes)
0xff  → [0x00, 0xff]   (2 bytes)
0x8000 → [0x00, 0x80, 0x00]  (3 bytes)
```

`len_for_value` (src/allocator.rs:342-356) — the authoritative size function for small positive integers — correctly returns 2 for 0x80 and 5 for 0xFFFFFFFF, matching the canonical encoding. `limbs_for_int` returns values that are 1 less for every positive integer in the ranges [0x80, 0xFF], [0x8000, 0xFFFF], [0x800000, 0xFFFFFF], etc.

**Propagation — `op_multiply` (src/more_ops.rs:649-652)**

After each multiplication step the running product size is updated:

```rust
l0 = limbs_for_int(&total);   // ← underestimates by 1 when high bit of total is set
if l0 > 1024 {
    return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
}
```

`l0` is then used in the **next** iteration's cost formula:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

Two broken invariants follow:

1. **Cost undercharge.** When `total` has its high bit set, `l0` is 1 less than the actual canonical byte length. Every subsequent multiplication step charges `(actual_l0 - 1 + l1)` instead of `(actual_l0 + l1)` for the linear term, and `(actual_l0 - 1) * l1 / divider` instead of `actual_l0 * l1 / divider` for the quadratic term. For large operands near the 1024-byte limit the quadratic term makes this gap significant.

2. **Size-limit bypass.** If `total` is a positive number whose canonical encoding is exactly 1025 bytes (high bit set, `bits()` = 8192), `limbs_for_int` returns 1024, the guard `1024 > 1024` is false, and the oversized product is accepted. The next multiplication then uses `l0 = 1024` while the actual operand is 1025 bytes.

---

### Impact Explanation

- **Consensus divergence / cost-accounting integrity.** Every full node must agree on the cost of every CLVM program. A program whose true cost exceeds `max_cost` but whose computed cost (using the underestimated `l0`) does not will be accepted by nodes running this code and rejected by a reference implementation that uses the correct size. This is a consensus-critical integrity failure.
- **Undercharged execution.** An attacker can craft a chain of multiplications where each intermediate product lands in a high-bit-set range, accumulating a cost discount proportional to the number of such steps and the size of the operands. Near the 1024-byte limit the quadratic term `l0 * l1 / divider` is large, so the per-step discount is non-trivial.
- **Size-limit bypass.** A product of 1025 canonical bytes can be produced and passed to `malloc_cost`, allocating more heap than the 1024-byte invariant permits.

---

### Likelihood Explanation

The trigger is fully attacker-controlled: any CLVM program submitted to the network can invoke `*` (op_multiply) with operands chosen so that intermediate products fall in the high-bit-set ranges. No privileged access, no social engineering, and no dependency on external state is required. The condition (product's high bit set) occurs for roughly half of all large intermediate values, so it is not an edge case.

---

### Recommendation

Replace `limbs_for_int` with a function that computes the canonical CLVM atom length, matching `new_number`'s encoding logic:

```rust
fn limbs_for_int(v: &Number) -> usize {
    // canonical signed-big-endian length: same as new_number() produces
    let bytes = v.to_signed_bytes_be();
    let mut slice = bytes.as_slice();
    while slice.len() > 1 && slice[0] == 0 && (slice[1] & 0x80) == 0 {
        slice = &slice[1..];
    }
    if slice == [0u8] { 0 } else { slice.len() }
}
```

Alternatively, allocate the intermediate result with `new_number` and call `atom_len` on it, as the `generate-modpow-tests.rs` tool already does correctly (tools/src/bin/generate-modpow-tests.rs:77-81).

### Proof of Concept

**Concrete numeric example (single step):**

- First operand: `0x80` (128). `int_atom` returns `l0 = 2` (actual atom `[0x00, 0x80]`).
- Multiply by `0x80` (128). Product = 16384 = `0x4000`. `limbs_for_int(16384) = ceil(15/8) = 2`. Actual atom `[0x40, 0x00]` = 2 bytes. Correct here.
- Multiply by `0x80` again. Product = 2097152 = `0x200000`. `limbs_for_int = ceil(22/8) = 3`. Actual atom `[0x20, 0x00, 0x00]` = 3 bytes. Correct.
- Multiply by `0x80` again. Product = 268435456 = `0x10000000`. `limbs_for_int = ceil(29/8) = 4`. Actual = 4 bytes. Correct.
- Multiply by `0x80` again. Product = 34359738368 = `0x800000000`. `limbs_for_int = ceil(36/8) = 5`. Actual atom = `[0x08, 0x00, 0x00, 0x00, 0x00]` = 5 bytes. Correct.

Now choose a product that lands with its high bit set:

- Start with `0x80` (128). `l0 = 2` (correct).
- Multiply by `0x01`. Product = 128. `limbs_for_int(128) = ceil(8/8) = 1`. **Actual atom = `[0x00, 0x80]` = 2 bytes.** `l0` is now **1** instead of **2**.
- Next multiplication charges `(1 + l1) * MUL_LINEAR_COST_PER_BYTE` instead of `(2 + l1) * MUL_LINEAR_COST_PER_BYTE`. Cost is undercharged by `MUL_LINEAR_COST_PER_BYTE` = 3 per step.

**Size-limit bypass:**

Construct a product `P` such that `P` is a positive integer with exactly 8192 bits and its most-significant bit is 1 (i.e., `P ≥ 2^8191`). Then `limbs_for_int(P) = 1024` but `new_number(P)` produces a 1025-byte atom. The guard at line 650 passes, and the 1025-byte value is handed to `malloc_cost`.
