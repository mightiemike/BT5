### Title
`wrapVaultAsset` Silently Skips Deposit When ERC4626 Vault `totalSupply == 0` Due to `previewDeposit` Zero-Return Gate - (`core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.wrapVaultAsset` uses `IERC4626Base(tokenAddr).previewDeposit(assetBalance) != 0` as the sole gate before executing the actual vault deposit. When the ERC4626 vault is fresh (`totalSupply == 0`), conformant implementations of `previewDeposit` that use the formula `(assets * totalSupply) / totalAssets` return `0`, even though the actual `deposit` call would succeed and mint shares 1:1. The gate evaluates to `false`, the deposit is silently skipped, and the user's underlying assets remain permanently locked in `DirectDepositV1`.

---

### Finding Description

In `ContractOwner.sol`, `wrapVaultAsset` pulls the underlying asset balance held by the `DirectDepositV1` escrow and then calls `previewDeposit` to decide whether to proceed: [1](#0-0) 

```solidity
uint256 assetBalance = IERC20Base(assetTokenAddr).balanceOf(directDepositV1);
if (IERC4626Base(tokenAddr).previewDeposit(assetBalance) != 0) {
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(assetTokenAddr));
    IERC20Base assetToken = IERC20Base(assetTokenAddr);
    assetToken.approve(tokenAddr, 0);
    assetToken.approve(tokenAddr, assetBalance);
    IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
}
```

The `IERC4626Base` interface exposes both `previewDeposit` and `deposit`: [2](#0-1) 

For a vault whose `totalSupply == 0`, the standard share-price formula `(assets × totalSupply) / totalAssets` evaluates to `0`. `previewDeposit` therefore returns `0`. The `if` guard is `false`, so the entire deposit block is bypassed — yet the actual `deposit` function on the same vault would have minted shares equal to `assetBalance` (1:1 bootstrap ratio). This is the exact divergence identified in the reference report: the estimation path and the execution path disagree at the `totalSupply == 0` boundary.

The same broken gate appears in `isWrapVaultAssetReady`: [3](#0-2) 

```solidity
uint256 wrappedBalance = IERC4626Base(tokenAddr).previewDeposit(assetBalance);
if (wrappedBalance != 0) {
    ...
    return _isDepositAmountReady(productId, wrappedBalance, isFirstDeposit);
}
return false;
```

When `totalSupply == 0`, this view function returns `false` even though the deposit would succeed, causing any caller (frontend, keeper, or on-chain contract) that checks readiness before calling `wrapVaultAsset` to conclude the operation is not yet possible.

---

### Impact Explanation

A user who sends underlying assets to their `DirectDepositV1` address for a newly listed ERC4626 vault product will have those assets permanently stranded. `wrapVaultAsset` silently exits without wrapping, so no vault shares are minted, no deposit is credited to the subaccount, and the assets cannot be recovered through the normal flow. The `DirectDepositV1` escrow holds the funds with no further mechanism to retry once the gate has been evaluated as `false`.

---

### Likelihood Explanation

The condition `totalSupply == 0` is the natural state of every newly deployed ERC4626 vault. Any user who attempts to wrap assets into a freshly listed vault product — which is the expected first-depositor scenario — will trigger this path. No special permissions or adversarial setup are required; a normal unprivileged caller invoking `wrapVaultAsset` is sufficient.

---

### Recommendation

Replace the `previewDeposit` gate with a direct balance check, matching the logic that the actual `deposit` call would use:

```diff
- if (IERC4626Base(tokenAddr).previewDeposit(assetBalance) != 0) {
+ if (assetBalance > 0) {
```

Apply the same fix to `isWrapVaultAssetReady`:

```diff
- uint256 wrappedBalance = IERC4626Base(tokenAddr).previewDeposit(assetBalance);
- if (wrappedBalance != 0) {
+ if (assetBalance > 0) {
+     uint256 wrappedBalance = IERC4626Base(tokenAddr).previewDeposit(assetBalance);
+     if (wrappedBalance == 0) wrappedBalance = assetBalance; // handle totalSupply == 0
```

---

### Proof of Concept

1. A new ERC4626 vault product is registered in Nado; its `totalSupply == 0`.
2. A user sends underlying tokens to their `DirectDepositV1` address.
3. Anyone calls `ContractOwner.wrapVaultAsset(subaccount, productId)`.
4. `assetBalance = IERC20Base(assetTokenAddr).balanceOf(directDepositV1)` → non-zero (e.g., `1000e6`).
5. `IERC4626Base(tokenAddr).previewDeposit(1000e6)` → `0` (because `totalSupply == 0` makes `(assets × 0) / totalAssets = 0`).
6. The `if` guard is `false`; the deposit block is skipped entirely.
7. The actual `IERC4626Base(tokenAddr).deposit(1000e6, directDepositV1)` call — had it been reached — would have minted `1000e6` shares (1:1 bootstrap).
8. User's `1000e6` underlying tokens remain locked in `DirectDepositV1` with no recovery path through the normal protocol flow. [4](#0-3) [5](#0-4)
