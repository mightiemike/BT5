### Title
SwapAllowlistExtension gates the router address instead of the end user, allowing any user to bypass the swap allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is `msg.sender` from the pool's perspective — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. The extension therefore checks whether the **router** is allowlisted, not the actual swapper. Any user can bypass the per-pool swap allowlist by routing through the public `MetricOmmSimpleRouter`.

---

### Finding Description

The call chain for a router-mediated swap is:

```
User → MetricOmmSimpleRouter.exactInputSingle()
     → pool.swap(recipient, ..., extensionData)   [msg.sender = router]
     → _beforeSwap(msg.sender=router, recipient, ...)
     → SwapAllowlistExtension.beforeSwap(sender=router, ...)
```

In `MetricOmmPool.swap`, the pool passes `msg.sender` (the router) as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
  msg.sender,   // ← router address, not end user
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to the extension:

```solidity
// ExtensionCalling.sol:160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, ...)   // sender = router
  )
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool (the pool calls the extension), and `sender` = router. The check resolves to `allowedSwapper[pool][router]`. The pool admin's intent is to gate individual users, but the extension gates the router contract.

This creates an irreconcilable conflict:
- If the pool admin **does not** allowlist the router → all router-mediated swaps revert, even for legitimate users.
- If the pool admin **does** allowlist the router → every user on the network can bypass the allowlist by calling `MetricOmmSimpleRouter.exactInputSingle()` (or `exactInput`, `exactOutputSingle`, `exactOutput`).

The same bypass applies to the `exactOutput` multi-hop path, where intermediate hops also call `pool.swap()` with `msg.sender = router`.

---

### Impact Explanation

The `SwapAllowlistExtension` is the production mechanism for restricting which addresses may trade against a pool. Pools that deploy this extension do so to prevent toxic flow, enforce KYC/whitelist requirements, or limit participation to specific counterparties. A complete bypass of this gate means:

- Any unprivileged user can execute swaps against a pool that was configured to deny them.
- LP funds are exposed to the full universe of traders, defeating the purpose of the allowlist.
- If the pool was designed to trade only with trusted counterparties (e.g., to avoid adverse selection), the bypass allows adversarial traders to extract value from LPs at oracle-anchored prices.

This is a direct loss-of-principal risk for LPs in allowlisted pools.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract — any address can call it.
- No special setup is required: the attacker simply calls `exactInputSingle` with the target pool.
- The bypass is unconditional once the router is allowlisted (which is the only way legitimate users can use the router with this pool).
- The attacker needs no privileged role, no special token, and no prior relationship with the pool.

Likelihood is **high**.

---

### Recommendation

The `beforeSwap` hook must receive the **original end user's address**, not the intermediary router's address. Two approaches:

1. **Pass the original caller through `extensionData`**: The router encodes `msg.sender` (the end user) into `extensionData` before calling `pool.swap`. The extension decodes and checks this value. This requires the extension and router to agree on an encoding convention and the extension to trust the pool's forwarding of `extensionData`.

2. **Add a dedicated `swapper` parameter separate from `sender`**: Modify the `beforeSwap` hook interface to carry both the immediate caller (`sender`) and the economic actor (`swapper`). The router would pass `msg.sender` as `swapper` explicitly. This is a breaking interface change but is the cleanest fix.

Until fixed, pools that need per-user swap gating must not rely on `SwapAllowlistExtension` when `MetricOmmSimpleRouter` is in use, or must not allowlist the router (accepting that no router-mediated swaps are possible).

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured.
  - Pool admin allowlists address(router) so that legitimate users can use the router.
  - Pool admin does NOT allowlist attacker EOA.

Attack:
  1. attacker calls MetricOmmSimpleRouter.exactInputSingle({
       pool: targetPool,
       recipient: attacker,
       zeroForOne: true,
       amountIn: X,
       ...
     })
  2. Router calls targetPool.swap(attacker, true, X, ..., extensionData)
     → pool.msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true (router was allowlisted)
  5. Swap executes. Attacker receives output tokens.

Result: attacker, who was never individually allowlisted, successfully swaps against the pool.
