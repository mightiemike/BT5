### Title
`ExitTierWithDelegation` Blocked by Pool Depletion via `topUpBaseRewards` — (`x/tieredrewards/keeper/abci.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`RewardsPoolName` is the single source for both the BeginBlocker base-rewards top-up and bonus reward payouts. When `TargetBaseRewardsRate > 0` and the fee collector is empty (zero-transaction blocks), `topUpBaseRewards` transfers the entire remaining pool balance to the distribution module each block. Once the pool reaches zero, any call to `ExitTierWithDelegation` that would owe non-zero bonus rewards fails with `ErrInsufficientBonusPool`, permanently blocking principal recovery until the pool is externally replenished.

---

### Finding Description

**Pool depletion path — `topUpBaseRewards` (`abci.go` lines 96–116):**

`topUpBaseRewards` computes `shortFallAmount` as the full per-block target when `feeCollectorBalance` is zero:

```
shortFallAmount = targetStakersRewardPerBlock - 0 = targetStakersRewardPerBlock
```

If `poolBalance < shortFallAmount`, the entire pool balance is transferred: [1](#0-0) 

The guard at line 99 only skips when the pool is already zero — it does not prevent the pool from being drained to zero over successive blocks.

**Bonus check in `processEventsAndClaimBonus` (`claim_rewards.go` lines 219–232):**

When `totalBonus > 0`, `sufficientBonusPoolBalance` is called unconditionally: [2](#0-1) 

`sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` if the pool cannot cover the bonus: [3](#0-2) 

**Exit path blocked — `ExitTierWithDelegation` (`msg_server.go` lines 532–535):**

`ExitTierWithDelegation` calls `claimRewards` before transferring the delegation back to the owner. There is no special handling for an empty pool: [4](#0-3) 

`claimRewards` propagates the error from `processEventsAndClaimBonus` without any fallback: [5](#0-4) 

The same blocking pattern applies to `TierUndelegate` (line 166) and `ForceFullExitWithDelegation` (line 37), which also call `claimRewards` unconditionally: [6](#0-5) [7](#0-6) 

**Both pools are the same account:**

`topUpBaseRewards` drains `types.RewardsPoolName` (line 113), and `processEventsAndClaimBonus` pays bonus from `types.RewardsPoolName` (line 239). There is no separation: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

Once `RewardsPoolName` is drained to zero by `topUpBaseRewards`, every position owner with accrued non-zero bonus rewards is unable to exit via `ExitTierWithDelegation`, `TierUndelegate`, or `ForceFullExitWithDelegation`. Their staked principal is locked in the module until the pool is externally replenished. The only exit path that avoids `claimRewards` is `WithdrawFromTier`, but that requires the position to already be undelegated — and `TierUndelegate` itself calls `claimRewards` first, so it is equally blocked.

---

### Likelihood Explanation

The condition is reachable under normal chain operation: a high `TargetBaseRewardsRate` (governance-set), low transaction volume (empty or near-empty blocks), and a finite pool balance are all realistic. No attacker action is required beyond waiting. The ADR explicitly documents that user-driven paths fail atomically when the pool is empty and "retry after pool is replenished": [10](#0-9) 

However, `ExitTierWithDelegation` is **not listed** in the documented set of paths that are expected to fail atomically (ClaimTierRewards, AddToPosition, Undelegate, Redelegate, ClearPosition). The ADR edge-case table also notes `ExitTierWithDelegation` fails when the validator is jailed/unbonding but does not mention pool depletion as an expected failure mode for this path: [11](#0-10) 

This gap between documented intent and implementation is the root cause.

---

### Recommendation

Decouple the two competing uses of `RewardsPoolName`, or handle the insufficient-pool case gracefully in the exit path. Concretely:

1. **Skip bonus on exit when pool is empty**: In `ExitTierWithDelegation` (and `TierUndelegate`), catch `ErrInsufficientBonusPool` from `processEventsAndClaimBonus` and proceed with zero bonus rather than aborting — consistent with the `BeforeRedelegationSlashed` hook which already forfeits bonus silently when the pool is insufficient: [12](#0-11) 

2. **Or** reserve a minimum pool balance for bonus payouts in `topUpBaseRewards`, capping the top-up to `poolBalance - reservedBonusBuffer`.

3. **Or** use two separate module accounts — one for base-rewards top-up and one for bonus rewards — so the BeginBlocker cannot deplete the bonus pool.

---

### Proof of Concept

```
1. Deploy chain with TargetBaseRewardsRate = 1.0, mint inflation disabled.
2. Fund RewardsPoolName with N basecro (finite).
3. Owner creates a position via MsgLockTier; bonus begins accruing.
4. Run N / TOPUP_PER_BLOCK empty blocks (no transactions).
   → topUpBaseRewards drains RewardsPoolName to 0 each block.
5. Assert RewardsPoolName balance == 0.
6. Owner submits MsgExitTierWithDelegation.
   → claimRewards → processEventsAndClaimBonus → sufficientBonusPoolBalance
   → returns ErrInsufficientBonusPool.
   → Exit fails; principal locked.
7. Assert owner's staked balance is unchanged (principal not returned).
```

The integration test scaffold in `integration_tests/test_base_rewards_top_up.py` already demonstrates pool depletion to zero (`test_pool_drains_to_zero`, `test_insufficient_pool_partial_drain`): [13](#0-12) 

An additional test asserting that `ExitTierWithDelegation` still succeeds (with zero bonus) after pool depletion would confirm the invariant break.

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
