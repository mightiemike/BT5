### Title
Sanctioned User Can Complete Collateral Withdrawal via Slow Mode Execution Path — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `submitSlowModeTransactionImpl` function enforces a sanctions check at **submission time**, but `processSlowModeTransactionImpl` — invoked during execution — performs **no sanctions check**. Because `executeSlowModeTransaction()` is callable by anyone after the 3-day delay, a user who submits a `WithdrawCollateral` slow mode transaction before being sanctioned can have their collateral transferred out after the delay, even after being added to the sanctions list.

---

### Finding Description

**Path A — Slow Mode Submission (sanctions enforced):**

In `submitSlowModeTransactionImpl`, `requireUnsanctioned(sender)` is called before the transaction is queued: [1](#0-0) 

This correctly blocks a sanctioned user from submitting new slow mode transactions.

**Path B — Slow Mode Execution (no sanctions check):**

When the queued transaction is later executed, `processSlowModeTransactionImpl` handles `WithdrawCollateral` with no sanctions check: [2](#0-1) 

It calls `clearinghouse.withdrawCollateral(...)` directly. The grep for `requireUnsanctioned`/`isSanctioned` across all production contracts confirms **no sanctions check exists in `Clearinghouse.sol`**, so the call chain proceeds to `withdrawPool.submitWithdrawal(...)` and ultimately transfers tokens to the sanctioned address. [3](#0-2) 

**Permissionless execution trigger:**

`executeSlowModeTransaction()` is `external` with no access control — any caller can trigger it after the 3-day delay: [4](#0-3) 

The `_executeSlowModeTransaction` call with `fromSequencer = false` only requires `txn.executableAt <= block.timestamp`: [5](#0-4) 

This means the sanctioned user themselves (or any third party) can trigger the execution after the delay window.

---

### Impact Explanation

A user who submits a `WithdrawCollateral` slow mode transaction before being sanctioned can have their collateral transferred to their wallet after the 3-day delay, bypassing the protocol's OFAC/sanctions enforcement. The `BaseWithdrawPool.submitWithdrawal` path also has no sanctions check: [6](#0-5) 

The corrupted state delta is: sanctioned address receives ERC-20 collateral tokens that the protocol's sanctions system was intended to freeze.

---

### Likelihood Explanation

The 3-day slow mode delay (`SLOW_MODE_TX_DELAY`) is architecturally designed to give the protocol a window to intervene in exactly this scenario. The gap is therefore not a theoretical edge case — it is a direct failure of the mechanism intended to handle post-submission sanctions. A sanctioned user who anticipates enforcement can pre-submit a withdrawal, or a user sanctioned after submission has their withdrawal automatically executable by anyone after the delay.

---

### Recommendation

Add a `requireUnsanctioned` check inside `processSlowModeTransactionImpl` for the `WithdrawCollateral` (and `WithdrawCollateral`) cases, checking the resolved sender address before calling `clearinghouse.withdrawCollateral`:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
+   requireUnsanctioned(address(uint160(bytes20(txn.sender))));
    clearinghouse.withdrawCollateral(...);
```

Alternatively, enforce the check inside `Clearinghouse.withdrawCollateral` so that all withdrawal paths — slow mode, fast withdrawal, and sequencer-submitted — are uniformly protected.

---

### Proof of Concept

1. User calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload. `requireUnsanctioned` passes — user is not yet sanctioned. Transaction is queued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY`.
2. Protocol adds user to the Chainalysis sanctions list. `requireUnsanctioned` would now revert for this address.
3. After 3 days, anyone calls `executeSlowModeTransaction()`.
4. `_executeSlowModeTransaction` → `processSlowModeTransaction` → `processSlowModeTransactionImpl` → `WithdrawCollateral` branch → `clearinghouse.withdrawCollateral(txn.sender, ...)` — **no sanctions check at any step**.
5. `Clearinghouse.withdrawCollateral` calls `withdrawPool.submitWithdrawal(token, sendTo, amount, idx)`.
6. `BaseWithdrawPool.handleWithdrawTransfer` transfers tokens to the sanctioned address.
7. Sanctioned user receives collateral despite being on the sanctions list.
