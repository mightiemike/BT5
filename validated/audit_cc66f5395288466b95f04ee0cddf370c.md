Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual trader — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` uses `msg.sender` (the pool) as the key and the `sender` argument (the immediate pool caller) as the swapper to check. When a trade is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`. Any unprivileged user who is not individually allowlisted can bypass the per-user gate by going through the router, provided the router address is allowlisted — a configuration the pool admin is naturally expected to make.

## Finding Description

**Call chain:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` (or `exactInput`/`exactOutput`). The router calls `IMetricOmmPoolActions(pool).swap(recipient, ...)` — so from the pool's perspective `msg.sender = router`.

2. Inside `MetricOmmPool.swap()`, `_beforeSwap(msg.sender, ...)` is called, passing the router address as `sender`:
   ```
   _beforeSwap(msg.sender /*= router*/, recipient, ...)
   ```

3. `ExtensionCalling._beforeSwap` encodes and forwards `sender = router` to the extension.

4. `SwapAllowlistExtension.beforeSwap` receives `sender = router` and checks:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
       revert NotAllowedToSwap();
   }
   // msg.sender = pool, sender = router
   // → checks allowedSwapper[pool][router]
   ```

The extension is documented as "Gates `swap` by swapper address, per pool." The actual swapper (the EOA or contract that called the router) is never examined. The check is against the router's address.

**Why existing guards are insufficient:**
- `BaseMetricExtension` has an `onlyPool` modifier on the default `beforeSwap`, but `SwapAllowlistExtension` overrides it without that modifier and without any substitute guard that recovers the original user.
- The router stores the original `msg.sender` only in transient storage for payment purposes (`_setNextCallbackContext`), and never forwards it to the pool's `swap` call as a distinct parameter.
- There is no mechanism in `ExtensionCalling` or `MetricOmmPool` to propagate the original EOA through to extensions.

**Exploit scenario:**
- Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a whitelist of known counterparties.
- Pool admin allowlists the router address (reasonable: they want allowlisted users to be able to use the standard router UI).
- Any unprivileged user — not on the per-user allowlist — calls `router.exactInputSingle(...)`. The pool sees `sender = router`, which is allowlisted, and the swap proceeds.
- The per-user allowlist is completely bypassed.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control primitive. Its stated invariant — "only allowlisted addresses may swap" — is broken for any pool that allowlists the router. An unprivileged trader can execute swaps in a pool that is supposed to be restricted, constituting broken core pool functionality and an admin-boundary break (the pool admin's access control is bypassed by an unprivileged path). Severity: High.

## Likelihood Explanation
The condition is easily met: the pool admin allowlisting the router is the expected and natural configuration for any pool that wants to support the standard periphery. The bypass requires no special privileges, no flash loans, and no unusual token behavior — only a call to the public router. It is repeatable on every swap.

## Recommendation
The router should forward the original caller's identity to the pool in a way that extensions can consume it. One approach: pass the original `msg.sender` as the first element of `extensionData` (with a well-known prefix), and have `SwapAllowlistExtension.beforeSwap` decode and check it when present. Alternatively, add a dedicated `originSender` field to the pool's `swap` signature, or have the router pass the real user address as the `recipient`-equivalent actor for allowlist purposes. The extension must never rely solely on the pool-level `sender` when a router intermediary is in the call path.

## Proof of Concept

```solidity
// Setup:
// 1. Deploy pool with SwapAllowlistExtension
// 2. Pool admin allowlists router: extension.setAllowedToSwap(pool, address(router), true)
// 3. Pool admin does NOT allowlist attacker: allowedSwapper[pool][attacker] == false

// Attack:
vm.prank(attacker); // attacker is NOT individually allowlisted
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: address(token0),
    tokenOut: address(token1),
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: attacker,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// Swap succeeds: extension checked allowedSwapper[pool][router] == true
// attacker bypassed the per-user allowlist
```
