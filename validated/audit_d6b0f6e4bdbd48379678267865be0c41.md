### Title
Missing `ENABLE_SECP_OPS` Flag Validation on 4-Byte Secp Opcode Dispatch Allows Flag-Bypass Execution - (File: `src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` in `src/chia_dialect.rs` dispatches the 4-byte secp opcodes `0x13d61f00` (`secp256k1_verify`) and `0x1c3a8f00` (`secp256r1_verify`) directly to their implementation functions **without checking the `ENABLE_SECP_OPS` flag**. The 1-byte opcode aliases (64 and 65) for the same operators are correctly gated by `ENABLE_SECP_OPS`. This inconsistency allows attacker-controlled CLVM bytes to invoke secp cryptographic verification regardless of whether the softfork flag is active, bypassing the activation mechanism and causing consensus divergence.

---

### Finding Description

`ChiaDialect::op()` has two separate dispatch paths for secp operators:

**Path 1 — 1-byte opcodes (correctly gated):**
```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [1](#0-0) 

**Path 2 — 4-byte opcodes (flag check absent):**
```rust
if op_len == 4 {
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => { return unknown_operator(...); }
    };
    return f(allocator, argument_list, max_cost, flags);
}
``` [2](#0-1) 

The 4-byte path dispatches to `op_secp256k1_verify` / `op_secp256r1_verify` with **no `ENABLE_SECP_OPS` check**. The `ENABLE_SECP_OPS` flag is defined as the exclusive gate for secp operator availability: [3](#0-2) 

Every other recently-added operator follows the same gating pattern — `keccak256` (opcode 62) requires `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, `sha256_tree` (opcode 63) requires `ENABLE_SHA256_TREE`, and the 1-byte secp opcodes require `ENABLE_SECP_OPS`. The 4-byte secp opcodes are the sole exception. [4](#0-3) 

The section comment even labels the 4-byte block as "unknown operators with assigned cost," implying they should fall through to `unknown_operator` for unrecognized opcodes — yet two specific 4-byte values are silently promoted to full secp execution without the required flag. [5](#0-4) 

---

### Impact Explanation

**Corrupted result:** When `ENABLE_SECP_OPS` is not set, a node evaluating a CLVM program that uses the 4-byte secp opcode encoding will execute real secp256k1/r1 signature verification and return a cryptographic boolean result. A node running older code (or any code that treats `0x13d61f00` as a true unknown operator) returns `nil` with a cost-formula-derived value. These two outcomes are irreconcilable.

**Consensus divergence:** Any coin spend that embeds the 4-byte secp opcode will be evaluated differently by nodes depending on whether they carry this code version and which flags are active. A spend that should be rejected (secp not yet activated) passes on nodes with this code; a spend that should pass fails on nodes without it.

**Flag bypass:** The `ENABLE_SECP_OPS` flag is the network's mechanism for controlling softfork activation of secp operators. The 4-byte path renders this mechanism ineffective for any program that uses the 4-byte encoding.

**Impact:** High — consensus-breaking divergence reachable by any coin spend submitting attacker-controlled CLVM bytes.

---

### Likelihood Explanation

**Likelihood:** High. The 4-byte opcodes are fixed, publicly derivable constants (`0x13d61f00`, `0x1c3a8f00`). Any party submitting a coin spend can embed these bytes directly in CLVM. No privileged access, social engineering, or dependency compromise is required. The entry path is the standard `run_program` → `apply_op` → `ChiaDialect::op()` chain, reachable from any externally submitted program. [6](#0-5) 

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte dispatch path, mirroring the 1-byte path:

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
        0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

This ensures both opcode encodings respect the same activation gate, eliminating the bypass path.

### Proof of Concept

Craft a CLVM program using the raw 4-byte secp opcode atom `0x13d61f00` as the operator, with a valid secp256k1 signature triple as arguments. Evaluate it via `run_program` with a `ChiaDialect` constructed **without** `ENABLE_SECP_OPS`:

```rust
let dialect = ChiaDialect::new(ClvmFlags::NO_UNKNOWN_OPS); // ENABLE_SECP_OPS absent
// Build: (0x13d61f00 pubkey msg sig)
let op = allocator.new_atom(&[0x13, 0xd6, 0x1f, 0x00])?;
// ... build args with valid secp256k1 pubkey/msg/sig ...
let result = run_program(&mut allocator, &dialect, program, args, max_cost);
// result: Ok(...) — secp verification executed despite flag being absent
// Expected: Err(Unimplemented) or unknown-op nil, because ENABLE_SECP_OPS is not set
```

The 1-byte opcode 64 with the same dialect and `NO_UNKNOWN_OPS` correctly returns `Err(Unimplemented)` (as confirmed by the existing test cases). [7](#0-6) 

The 4-byte path silently executes the secp function, producing a divergent result that breaks the flag-gated activation invariant. [8](#0-7)
