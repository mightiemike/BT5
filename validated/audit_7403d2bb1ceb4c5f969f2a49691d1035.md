### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables USDC-E Theft from Direct Deposit Accounts — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

The `replaceUsdcEWithUsdc` function in `ContractOwner.sol` does not check the return value of `IERC20Base(usdc).transferFrom`. If the USDC token at the hardcoded address on Ink mainnet silently returns `false` (without reverting), any unprivileged caller can drain USDC-E from a Direct Deposit Account (DDA) without providing any USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is a publicly callable migration helper (no access control beyond a chain ID check) that is intended to let a caller swap USDC into a DDA in exchange for the DDA's USDC-E balance. The critical flaw is that the USDC `transferFrom` call uses the raw interface method and its return value is silently discarded: [1](#0-0) 

```solidity
uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
if (balance > 0) {
    IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← return value ignored
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
    IERC20Base(usdcE).safeTransfer(msg.sender, balance);
}
```

If `transferFrom` returns `false` (i.e., the USDC transfer silently fails), execution does **not** revert. The function proceeds to:

1. Call `DirectDepositV1(directDepositV1).withdraw(usdcE)` — ContractOwner is the owner of the DDA, so this succeeds and moves all USDC-E from the DDA to ContractOwner.
2. Call `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — transfers the USDC-E from ContractOwner to the caller.

The caller receives USDC-E without having transferred any USDC.

Contrast this with the rest of the codebase, which consistently uses `ERC20Helper.safeTransferFrom` that checks the return value and reverts on failure: [2](#0-1) 

The `replaceUsdcEWithUsdc` function is the only place in the production contracts that calls `transferFrom` directly without this safety wrapper.

---

### Impact Explanation

An attacker can steal the entire USDC-E balance of any DDA on chain 57073 (Ink mainnet) that has a non-zero USDC-E balance. The corrupted state delta is: DDA USDC-E balance goes to zero, ContractOwner USDC-E balance goes to zero, attacker's wallet receives the full USDC-E amount — with no USDC deposited in return. This is a direct, irreversible asset theft from protocol users whose funds are held in DDAs.

---

### Likelihood Explanation

The function is callable by any unprivileged address on chain 57073 with no further restrictions: [3](#0-2) 

The trigger condition is that the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink mainnet returns `false` from `transferFrom` rather than reverting. This is structurally identical to the Amphora finding: the Convex CVX contract's `mint` silently returns without minting when the operator changes. Here, if the USDC contract on Ink Chain is a non-standard implementation, a proxy upgrade, or a wrapped/bridged variant that returns `false` on insufficient allowance or balance rather than reverting, the exploit path is immediately open to any caller. The unchecked return value is the Nado-side root cause; the external token's silent-failure behavior is the enabling condition — matching the medium-severity "high impact, special requirement" profile of the reference finding.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's own `safeTransferFrom` wrapper from `ERC20Helper`, which checks the return value and reverts on failure:

```solidity
// Before (vulnerable):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [4](#0-3) 

---

### Proof of Concept

1. Attacker identifies any `subaccount` whose DDA (`directDepositV1Address[subaccount]`) holds a non-zero USDC-E balance on chain 57073.
2. Attacker calls `replaceUsdcEWithUsdc(subaccount)` — no USDC approval or balance required.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, balance)` returns `false` (silent failure — no revert).
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes successfully (ContractOwner is the DDA owner), transferring all USDC-E from the DDA to ContractOwner.
5. `IERC20Base(usdcE).safeTransfer(attacker, balance)` transfers all USDC-E from ContractOwner to the attacker.
6. Attacker holds the full USDC-E balance; the DDA is drained; no USDC was ever deposited. [5](#0-4)

**File:** core/contracts/libraries/ERC20Helper.sol (L23-42)
```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
