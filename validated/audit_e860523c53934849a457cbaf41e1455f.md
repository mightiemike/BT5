### Title
Missing `ghost_heap` Increment in `new_substr` for SmallAtom-to-SmallAtom Substrings — (File: `src/allocator.rs`)

---

### Summary

The `new_substr` function in `src/allocator.rs` fails to increment `ghost_heap` when a substring of a `SmallAtom` produces another `SmallAtom`. Every other code path that creates a `SmallAtom` increments `ghost_heap` by the atom's byte length, but this branch does not. This causes `heap_size()` to be silently under-reported, which allows a CLVM program to bypass the allocator's heap limit by an amount proportional to the number of SmallAtom-to-SmallAtom `substr` calls it makes.

---

### Finding Description

`ghost_heap` is a running accumulator that tracks "virtual" heap bytes — bytes that are logically allocated (and must count against the heap limit for consensus purposes) but are stored inline in a `NodePtr` rather than in `u8_vec`. The public `heap_size()` method returns `u8_vec.len() + ghost_heap`, and every heap-limit guard uses this combined value: [1](#0-0) 

Every code path that creates a `SmallAtom` correctly increments `ghost_heap` by the atom's byte length:

- `new_atom` (small-atom branch): [2](#0-1) 

- `new_small_number`: [3](#0-2) 

However, in `new_substr`, when a `SmallAtom` input produces a `SmallAtom` result, only `ghost_atoms` is incremented — `ghost_heap` is **not** incremented by `(end - start)`: [4](#0-3) 

The `else` branch of the same match arm (SmallAtom → `Bytes`) correctly writes bytes into `u8_vec`, so the heap is properly accounted for there. The omission is exclusive to the SmallAtom → SmallAtom path.

The broken invariant: after any `new_substr` call that returns a `SmallAtom` from a `SmallAtom` input, `heap_size()` is under-reported by exactly `(end - start)` bytes. Repeated calls accumulate the deficit linearly.

---

### Impact Explanation

The heap limit guard in `new_atom`, `new_small_number`, and `new_concat` all read `ghost_heap` directly: [5](#0-4) [6](#0-5) [7](#0-6) 

Because `ghost_heap` is under-counted, these guards allow more bytes to be allocated than `heap_limit` permits. An attacker who issues N SmallAtom-to-SmallAtom `substr` calls, each producing a k-byte result, can inflate the effective heap limit by N×k bytes. For callers using `Allocator::new_limited()` with a specific bound (e.g., a consensus-enforced memory cap), this is a direct bypass of that bound. If different network nodes enforce different limits, or if the limit is tightened in a future consensus rule, the divergence between the tracked and actual heap size becomes a consensus-critical discrepancy.

---

### Likelihood Explanation

The `substr` operator is a standard, attacker-reachable CLVM opcode. `new_substr` is called from `src/more_ops.rs`: [8](#0-7) 

Any CLVM program submitted to the network can call `substr` on small integer atoms (values ≤ `NODE_PTR_IDX_MASK` that fit in a `SmallAtom`). The trigger requires only that the input atom and the resulting slice both satisfy `fits_in_small_atom`. This is trivially achievable with small integer atoms. The default `Allocator::new()` sets `heap_limit = u32::MAX`, so the bypass has no practical effect under the default configuration; however, any caller using `Allocator::new_limited()` — including future consensus-enforced memory caps — is directly affected.

---

### Recommendation

Add the missing `ghost_heap` increment in the SmallAtom-to-SmallAtom branch of `new_substr`, mirroring the pattern used in `new_atom` and `new_small_number`:

```rust
// src/allocator.rs — new_substr, SmallAtom branch
if let Some(new_val) = fits_in_small_atom(substr) {
    self.ghost_atoms += 1;
    self.ghost_heap += (end - start) as usize;  // ← add this line
    Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
}
``` [9](#0-8)

### Proof of Concept

```rust
// Demonstrates ghost_heap under-counting via new_substr SmallAtom → SmallAtom
let mut a = Allocator::new_limited(20);

// Allocate a SmallAtom holding value 0x0102 (2 bytes).
// new_small_number increments ghost_heap by 2.
let atom = a.new_small_number(0x0102).unwrap();

// Take a 1-byte substr [0..1] → value 0x01, still a SmallAtom.
// new_substr increments ghost_atoms but NOT ghost_heap.
let _sub = a.new_substr(atom, 0, 1).unwrap();

// heap_size() is now under-reported by 1 byte.
// Repeating this N times under-reports by N bytes,
// allowing allocation of N bytes beyond heap_limit.
assert!(a.heap_size() < /* expected */ a.heap_size() + 1);
```

Each iteration of this pattern with a k-byte SmallAtom substr reduces the effective heap limit by k bytes, allowing a crafted CLVM program to allocate beyond the configured bound.
