### Title
Silent Bonus Checkpoint Advancement on `ErrInsufficientBonusPool` During Redelegation Slash Permanently Corrupts Position Reward Accounting — (`File: x/tieredrewards/keeper/slash.go`)

---

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is deliberately swallowed to prevent a chain halt. However, `processEventsAndClaimBonus` has already mutated the position's in-memory checkpoints (`LastEventSeq`, `LastBonusAccrual`, `LastKnownBonded`) and decremented event reference counts in the store — all before the pool-balance check fires. The subsequent `setPositionWithState` call then persists those advanced checkpoints. The position's accounting permanently records the bonus period as settled even though no payment was made. The user can never reclaim the lost bonus.

---

### Finding Description

`processEventsAndClaimBonus` (`claim_rewards.go:142–251`) takes a `*types.PositionState` pointer and mutates it in-place during the event-replay loop:

- **Line 193** — `pos.UpdateLastEventSeq(entry.Seq)` advances the event cursor for every processed event.
- **Line 196** — `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` writes the decremented reference count to the store (potentially garbage-collecting the event).
- **Line 215** — `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` advances `LastBonusAccrual` to the current block time.
- **Line 217** — `pos.UpdateLastKnownBonded(bonded)` updates the persisted bonded state.

Only **after** all of these mutations does the function check whether the pool can cover the computed bonus:

```go
// claim_rewards.go:230
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err   // returns ErrInsufficientBonusPool here
}
```

Back in `slashRedelegationPosition` (`slash.go:54–64`), the caller catches exactly this error and swallows it:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // error swallowed — execution continues with mutated pos
    } else {
        return err
    }
}
```

Execution then falls through to `setPositionWithState` (`slash.go:71` or `slash.go:77`), which persists the already-mutated `pos` — including the advanced `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded`. The store-level reference-count decrements from `decrementEventRefCount` are also committed because the hook itself does not return an error, so the Cosmos SDK cached-store write-set is not discarded.

The net result: the position's accounting state permanently reflects that all bonus up to the current block time has been processed, but the user received nothing.

---

### Impact Explanation

The corrupted value is `pos.LastBonusAccrual` / `pos.LastEventSeq` / `pos.LastKnownBonded` — the three fields that gate all future bonus claims for the position. Once advanced, they cannot be rewound by any user-facing message. The user's entire accrued bonus for the period from the previous claim up to the slash block is irrecoverably lost. Additionally, the decremented event reference counts may cause the validator events to be garbage-collected, making the loss unrecoverable even by governance. This is a direct, permanent loss of user funds (bond-denom tokens that should have been transferred from the rewards pool to the owner).

---

### Likelihood Explanation

The trigger requires two concurrent conditions:

1. **A tier position with an active redelegation entry** — any user who called `MsgTierRedelegate` and whose redelegation has not yet completed.
2. **The bonus pool is insufficient** — the pool can be drained by normal `MsgClaimTierRewards` calls from other users, or it may simply never have been funded adequately.

A validator double-sign (or any other slashable infraction) causes the staking module to call `SlashRedelegation`, which fires `BeforeRedelegationSlashed`. Evidence submission (`MsgSubmitEvidence`) is open to any account. Neither condition requires privileged access. The combination is realistic on a live chain with active tiered-rewards users.

---

### Recommendation

Move the `sufficientBonusPoolBalance` check to **before** the event-replay loop and checkpoint mutations in `processEventsAndClaimBonus`. If the pool cannot cover the projected bonus, return early without advancing any checkpoint or decrementing any reference count. Alternatively, if the pool check must remain at the end (e.g., because the exact bonus is unknown until the loop completes), the function must not mutate `pos` or decrement reference counts until after the check passes — or it must roll back those mutations on failure. The `slashRedelegationPosition` caller should also be updated to skip `setPositionWithState` entirely when the bonus settlement was forfeited, rather than persisting the partially-mutated position.

---

### Proof of Concept

1. User calls `MsgTierRedelegate` for position P from validator V1 to V2; a redelegation entry with `unbondingId = U` is created and stored in `RedelegationMappings[U] = P`.
2. Bonus pool is drained to zero by concurrent `MsgClaimTierRewards` calls from other users.
3. Any account submits `MsgSubmitEvidence` for a double-sign by V2 (or V2 is slashed by the slashing module for downtime/double-sign).
4. Staking module calls `SlashRedelegation` → `BeforeRedelegationSlashed(ctx, U, sharesToUnbond)` → `slashRedelegationPosition(ctx, U, sharesToUnbond)`.
5. `processEventsAndClaimBonus(ctx, &pos)` runs the event loop: advances `pos.LastEventSeq`, decrements event ref counts in the store, advances `pos.LastBonusAccrual` to block time, updates `pos.LastKnownBonded`.
6. `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`; the function returns the error with `pos` already mutated.
7. `slashRedelegationPosition` swallows the error and calls `setPositionWithState(ctx, pos, nil)`, persisting the advanced checkpoints.
8. User later calls `MsgClaimTierRewards` for position P. `processEventsAndClaimBonus` starts from the now-advanced `LastEventSeq` and `LastBonusAccrual` — the pre-slash accrual period is gone. The user receives zero bonus for that period despite having earned it.
