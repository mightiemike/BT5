### Title
Flat `SLOW_MODE_FEE` Enables Cheap Slow Mode Queue Stuffing, Indefinitely Delaying User Withdrawals — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`submitSlowModeTransactionImpl` accepts a `bytes calldata transaction` of **arbitrary size** and charges a flat `SLOW_MODE_FEE` of $1 regardless of payload length. There is no upper bound on the transaction bytes stored on-chain. An unprivileged attacker can flood the FIFO slow mode queue with large, cheaply-crafted entries, pushing legitimate user withdrawal transactions arbitrarily far back in the queue and breaking the protocol's censorship-resistance guarantee.

---

### Finding Description

`submitSlowModeTransactionImpl` in `EndpointTx.sol` performs the following steps for any non-admin, non-`DepositCollateral` transaction type:

1. Reads `txType` from `transaction[0]`.
2. Calls `chargeSlowModeFee(_getQuote(), sender)` — a flat transfer of `clearinghouse.getSlowModeFee()` (anchored to `SLOW_MODE_FEE = 1_000_000`, i.e. $1).
3. Stores the **full, unbounded** `transaction` bytes on-chain:

```solidity
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: transaction          // ← no size cap
});
``` [1](#0-0) 

The fee mechanism is defined as:

```solidity
int128 constant SLOW_MODE_FEE = 1000000; // $1
``` [2](#0-1) 

`chargeSlowModeFee` is a flat ERC-20 transfer with no size-proportional component:

```solidity
function chargeSlowModeFee(IERC20Base token, address from) internal virtual {
    require(address(token) != address(0));
    token.safeTransferFrom(from, address(this), clearinghouse.getSlowModeFee());
}
``` [3](#0-2) 

The slow mode queue is strictly FIFO: `_executeSlowModeTransaction` always processes `slowModeTxs[txUpTo]` and increments `txUpTo` by one per call. [4](#0-3) 

There is no maximum queue depth, no per-address submission cap, and no transaction size limit anywhere in the submission path. [5](#0-4) 

**Decoupled resource vs. cost — the exact analog to the external report:**

| Dimension | MonadBFT (external) | Nado (this finding) |
|---|---|---|
| Resource | Block byte space | Slow mode queue slots |
| Cost mechanism | Gas limit | Flat `SLOW_MODE_FEE` |
| Decoupling | Zero-byte calldata fills bytes cheaply | Large calldata fills queue slots cheaply |
| Broken invariant | Gas cost should bound block fill | Fee should bound queue spam |

---

### Impact Explanation

The slow mode queue is Nado's **censorship-resistance escape hatch**: when the sequencer refuses to process a user's withdrawal, the user submits via `submitSlowModeTransaction` and can self-execute after 3 days via `executeSlowModeTransaction()`. This is the only permissionless path to recover funds without sequencer cooperation. [6](#0-5) 

An attacker who stuffs `N` entries ahead of a victim's withdrawal forces the victim to wait for all `N` entries to drain — each requiring a separate call — before their own withdrawal can execute. Because `SLOW_MODE_FEE` is flat, the attacker's cost is linear in `N` (just $1 per slot) while the victim's delay is also linear in `N`. At $1 per slot, stuffing 10,000 queue positions costs $10,000 and can delay a victim's withdrawal by an operationally unbounded period.

The concrete corrupted state is the **slow mode queue position** (`txUpTo` / `txCount` delta), which directly controls when a user's `WithdrawCollateral` or `LinkSigner` slow mode transaction becomes executable. [7](#0-6) 

---

### Likelihood Explanation

- Entry point is fully permissionless: any address with $1 of quote token can call `submitSlowModeTransaction`.
- No signature, no subaccount registration, no privileged role required.
- The attacker does not need to know the victim's queue position in advance; pre-emptive stuffing works.
- The attack is economically rational when the attacker wants to prevent a specific withdrawal (e.g., a competitor liquidator, or an attacker who has already compromised a subaccount and wants to prevent the owner from withdrawing).

---

### Recommendation

1. **Add a maximum transaction size check** at the top of `submitSlowModeTransactionImpl`:
   ```solidity
   uint256 constant MAX_SLOW_MODE_TX_SIZE = 1024; // bytes
   require(transaction.length <= MAX_SLOW_MODE_TX_SIZE, ERR_TX_TOO_LARGE);
   ```
   This is the direct analog to the MonadBFT recommendation of constraining `proposal_byte_limit` relative to gas cost.

2. **Alternatively, make the fee scale with transaction size** so that submitting a 100 KB transaction costs proportionally more than submitting a 100-byte transaction.

3. **Add a per-address queue submission cap** to limit how many pending slow mode transactions a single address can have outstanding at once.

---

### Proof of Concept

```solidity
// Attacker stuffs the queue with 1000 large entries ahead of victim
for (uint i = 0; i < 1000; i++) {
    // WithdrawCollateral = type byte 0x02, followed by 30 KB of zero padding
    bytes memory junk = new bytes(30_001);
    junk[0] = 0x02; // TransactionType.WithdrawCollateral
    // Approve $1 quote token, then:
    endpoint.submitSlowModeTransaction(junk);
    // Cost per iteration: $1 SLOW_MODE_FEE + gas
}

// Victim submits a legitimate withdrawal
endpoint.submitSlowModeTransaction(validWithdrawTx);
// Victim's tx is at queue position 1001.
// After 3 days, victim calls executeSlowModeTransaction() —
// but txUpTo is still pointing at attacker entry #1.
// Victim must wait for all 1000 attacker entries to drain first.
// Each attacker entry silently fails in processSlowModeTransactionImpl
// (abi.decode rejects the oversized payload, caught by the try/catch),
// but txUpTo still advances, so the victim's delay is real.

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }
