Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual user, allowing any user to bypass per-user swap allowlist via router - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the first argument passed by the pool — which is `msg.sender` from the pool's perspective. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual end user. This means the allowlist checks the router's address rather than the individual user's address. Any unprivileged user can bypass a per-user swap allowlist on a curated pool by routing through the public router.

## Finding Description

**Root cause — wrong actor in `sender`:**

In `MetricOmmPool.swap()` (line 231), the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:
```solidity
_beforeSwap(
  msg.sender,   // ← whoever called pool.swap()
  recipient, ...
);
```

In `SwapAllowlistExtension.beforeSwap` (line 37):
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` = pool address (key), `sender` = whoever called `pool.swap()`.

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly (line 72-80 of `MetricOmmSimpleRouter.sol`), making the router the `sender` seen by the extension. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][userEOA]`.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only `alice` (`allowedSwapper[pool][alice] = true`).
2. Pool admin must also allowlist the router address to permit any router-mediated swap (`allowedSwapper[pool][router] = true`), since the router is the direct caller of `pool.swap()`.
3. Attacker (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
4. Router calls `pool.swap(recipient, ...)` with `msg.sender = router`.
5. Pool calls `_beforeSwap(router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. Attacker successfully swaps on a pool that was supposed to block them.

**Existing guards are insufficient:** `BaseMetricExtension.onlyPool` (line 19-24) only verifies the caller is a registered pool — it does not verify the identity of the original user. The `SwapAllowlistExtension` has no mechanism to recover the original EOA from the router call.

## Impact Explanation
Direct allowlist bypass on curated pools. A pool admin who intends to restrict swaps to a specific set of addresses cannot enforce that restriction when the public `MetricOmmSimpleRouter` is available. Any unprivileged user can trade on a pool that was designed to exclude them, causing unauthorized token flows and breaking the core curation invariant. This meets the "broken core pool functionality" and "admin-boundary break" criteria: an unprivileged path (the public router) bypasses a configured access control gate.

## Likelihood Explanation
The `MetricOmmSimpleRouter` is the primary public swap entrypoint. Any user who knows a pool uses `SwapAllowlistExtension` can trivially route through the router. No special privileges, flash loans, or multi-block setup are required. The attack is repeatable every block.

## Recommendation
The extension must check the actual end user, not the immediate caller of `pool.swap()`. Options:
1. Pass the original `msg.sender` (user EOA) through the router as part of `extensionData`, and have the extension decode and verify it — but this requires the router to be trusted to not forge it.
2. Require direct pool interaction (no router) for allowlisted pools — document this constraint clearly.
3. Redesign the allowlist to gate the `recipient` field instead of `sender`, since the recipient is the economically relevant party receiving output tokens and is harder to spoof.
4. Alternatively, add a `trustedForwarder` pattern where the router attests the original caller in a verifiable way (e.g., signed or via transient storage written before the pool call).
