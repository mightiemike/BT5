### Title
Unordered `receipt_proofs` within `ReceiptProofResponse` accepted by `set_state_header`, enabling attacker-controlled receipt application order and state-root divergence — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

An unprivileged peer serving a `ShardStateSyncResponseHeader` can reorder the `Vec<ReceiptProof>` inside any `ReceiptProofResponse` and simultaneously reorder the corresponding `Vec<RootProof>` at the same outer index `i` in `root_proofs`. Because `set_state_header` pairs entries only by position (`root_proofs[i][j]` ↔ `receipt_proofs[j]`), a consistent co-permutation keeps every individual `(root, proof, receipts_hash)` triple cryptographically valid. All four checks in step 4 pass. The stored header is then consumed verbatim by `set_state_finalize`, which flattens receipts in the attacker-chosen order and passes them to `apply_chunk`. Because receipt-processing order is protocol-visible (it determines which receipts are delayed and the resulting state root), the syncing node silently diverges from the canonical chain state.

---

### Finding Description

**Header construction (honest server, `adapter.rs:178–218`):**

The serving node iterates `receipt_proofs` in whatever order they were stored and, for each entry at position `j`, pushes a `RootProof` derived from `from_shard_id` of that entry:

```
root_proofs_cur.push(RootProof(root_proof, block_receipts_proofs[from_shard_index]));
``` [1](#0-0) 

The result is that `root_proofs[i][j]` is semantically tied to `receipt_proofs[j]` only by construction order, not by any field that the verifier checks.

**Verification (`set_state_header`, `adapter.rs:476–503`):**

The verifier pairs entries by index `j`:

```rust
let RootProof(root, block_proof) = &shard_state_header.root_proofs()[i][j];
let receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts));
if !verify_path(*root, proof, &receipts_hash) { … }          // 4e
if !verify_path(…prev_chunk_outgoing_receipts_root(), block_proof, root) { … } // 4f
``` [2](#0-1) 

The only structural constraint on ordering is the `visited_shard_ids` uniqueness check (step 4d): [3](#0-2) 

There is **no check** that `from_shard_id` values appear in any canonical order (e.g., ascending shard index). A co-permutation of `receipt_proofs` and `root_proofs[i]` that preserves the j-pairing passes every check:

- Step 4c: lengths unchanged.
- Step 4d: `from_shard_id` set is unchanged (just reordered).
- Step 4e: `verify_path(root_B, proof_B, hash_B)` still holds for the permuted pair.
- Step 4f: `verify_path(block_root, block_proof_B, root_B)` still holds.

**Downstream consumption (`set_state_finalize`, `chain_update.rs:479–487`):**

The stored header is read back and its `incoming_receipts_proofs` are used directly:

```rust
let receipts = collect_receipts_from_response(&receipt_proof_responses);
// …
let apply_result = self.runtime_adapter.apply_chunk(…, &receipts, transactions)?;
``` [4](#0-3) 

`collect_receipts_from_response` flattens receipts in the order `receipt_proofs` appear:

```rust
receipt_proof_response.iter()
    .flat_map(|ReceiptProofResponse(_, proofs)| proofs.iter())
``` [5](#0-4) 

There is no re-sorting or canonicalization before `apply_chunk`. The runtime processes incoming receipts strictly in the order supplied: [6](#0-5) 

Receipt processing order is protocol-visible: it determines which receipts exhaust the gas limit and are pushed to the delayed queue, and therefore the resulting state root stored in `ChunkExtra`.

### Impact Explanation

A syncing node that accepts the manipulated header will apply the chunk with receipts in the attacker-chosen order. The resulting state root (written to `ChunkExtra` and used as `prev_state_root` for the next chunk) can differ from the canonical value computed by every honest node that processed the same block normally. The syncing node silently holds a divergent shard state. Any chunk it subsequently produces will be rejected by the network, effectively ejecting it from participation without any explicit error.

The divergence is in a protocol-visible, consensus-critical value: the shard state root.

---

### Likelihood Explanation

- Any network peer can serve state sync responses; no validator or operator privilege is required.
- The attacker only needs to intercept or race the honest state sync response. State sync peers are selected from the p2p network.
- Constructing the attack payload requires knowing the real `receipt_proofs` and `root_proofs` (both are public chain data) and permuting them consistently — straightforward for any node that has synced the relevant blocks.
- The attack is silent: the victim node logs no error and believes it has successfully synced.

---

### Recommendation

In `set_state_header`, after the uniqueness check (step 4d), enforce that `from_shard_id` values appear in strictly ascending shard-index order (matching the canonical order used during block processing). Concretely, after inserting into `visited_shard_ids`, assert:

```rust
let from_shard_index = prev_shard_layout.get_shard_index(*from_shard_id)?;
if from_shard_index != j {
    return Err(Error::Other("set_shard_state failed: receipt_proofs not in canonical shard order".into()));
}
```

This makes the accepted ordering unique and matches the order in which `root_proofs` are constructed by the honest server. [7](#0-6)

### Proof of Concept

1. Obtain a valid `ShardStateSyncResponseHeader` for a shard with ≥ 2 shards sending receipts (so `receipt_proofs.len() ≥ 2` for some `ReceiptProofResponse`).
2. Swap `receipt_proofs[0]` ↔ `receipt_proofs[1]` within that `ReceiptProofResponse`.
3. Swap `root_proofs[i][0]` ↔ `root_proofs[i][1]` in the corresponding `root_proofs` entry.
4. Serve the modified header to a syncing node.
5. Observe that `set_state_header` returns `Ok(())` (all checks pass).
6. After `set_state_finalize`, compare the stored `ChunkExtra.state_root` against the value stored by a node that processed the block normally — they will differ whenever the two shards' receipts are not commutative under the gas schedule.
