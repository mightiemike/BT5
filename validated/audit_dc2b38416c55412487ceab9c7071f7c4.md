### Title
Deprecated `verify_transaction_inclusion` Remains Callable and Lacks Coinbase Merkle Proof Validation, Enabling 64-Byte Merkle Forgery - (`contract/src/lib.rs`)

### Summary
The contract exposes a deprecated `verify_transaction_inclusion` method that is still fully callable by any unprivileged NEAR account. Unlike its successor `verify_transaction_inclusion_v2`, it performs no coinbase Merkle proof validation, making it vulnerable to the 64-byte transaction Merkle proof forgery attack. An attacker can supply a crafted 64-byte internal Merkle tree node as `tx_id` and receive a `true` return value for a transaction that does not exist in the block.

### Finding Description
`verify_transaction_inclusion` is marked `#[deprecated]` in Rust source, but this is a compiler-level lint warning only — it does not restrict runtime access. The function remains a live, `pub` NEAR contract method gated only by `#[pause]`, which is inactive by default. [1](#0-0) 

The function computes the Merkle root from the caller-supplied `tx_id` and `merkle_proof` and compares it to the stored `header.block_header.merkle_root`: [2](#0-1) 

There is no coinbase transaction proof requirement. The successor `verify_transaction_inclusion_v2` fixes this by first requiring that a coinbase Merkle proof resolves to the same `merkle_root`, which prevents an attacker from substituting an internal node for a leaf: [3](#0-2) 

The 64-byte forgery works because Bitcoin's Merkle tree does not distinguish between leaf nodes (transactions) and internal nodes. A 64-byte blob can be crafted such that `SHA256d(blob)` equals an internal node hash that, when walked up the tree with a valid sibling path, produces the correct `merkle_root`. The deprecated function accepts this crafted blob as a valid `tx_id`.

### Impact Explanation
Any NEAR consumer contract or bridge that calls `verify_transaction_inclusion` (the deprecated entrypoint) to gate cross-chain actions — token releases, atomic swaps, lending collateral unlocks — can be deceived into accepting a proof for a Bitcoin transaction that was never broadcast or confirmed. The attacker does not need to mine a block or control any privileged role; they only need to construct a valid Merkle sibling path for an internal node and call the deprecated method directly.

### Likelihood Explanation
The deprecated function is reachable by any unprivileged NEAR account with no additional preconditions. The 64-byte Merkle forgery technique is publicly documented and has known tooling. Consumer contracts that integrated before `v0.5.0` may still be calling the deprecated entrypoint. The function is not removed, not access-controlled beyond the global pause, and not guarded by any on-chain deprecation enforcement.

### Recommendation
Remove `verify_transaction_inclusion` entirely from the contract's public interface, or replace its body with an unconditional `env::panic_str("deprecated: use verify_transaction_inclusion_v2")`. Do not rely on the Rust `#[deprecated]` attribute to prevent runtime calls — it has no effect on NEAR contract dispatch.

### Proof of Concept

1. Identify a confirmed Bitcoin block whose `merkle_root` is stored in the light client.
2. Construct a 64-byte blob `B` such that `SHA256d(B)` equals an internal Merkle node `N` in that block's transaction tree.
3. Build a valid Merkle sibling path from `N` up to `merkle_root`.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = SHA256d(B)` (the internal node hash, not a real txid)
   - `tx_block_blockhash` = the target block hash
   - `merkle_proof` = the sibling path from step 3
   - `tx_index` = the index corresponding to `N`'s position
   - `confirmations` = any value ≤ current depth
5. The function returns `true` because `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.block_header.merkle_root` holds, even though `tx_id` is not a real transaction hash. [4](#0-3)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
