Audit Report

## Title
`SwapAllowlistExtension` gates on router address instead of end user, allowlist fully bypassed via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument it receives, which is `msg.sender` of `pool.swap`. When `MetricOmmSimpleRouter` mediates a swap, it is `msg.sender` of `pool.swap`, so the extension always sees the router address. A pool admin who allowlists the router to enable normal user-facing swap flows inadvertently opens the gate to every caller, completely defeating the curated-pool invariant.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router `msg.sender` of that call: [4](#0-3) 

When any user calls `router.exactInputSingle(...)`, the router becomes `msg.sender` of `pool.swap`, so `sender` = router address. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. Once the router is allowlisted — the only way to enable normal user-facing swap flows — every caller passes the check regardless of their individual allowlist status.

The `DepositAllowlistExtension` has a symmetric problem: it checks `owner` (a caller-supplied argument), not `msg.sender`. Any address can call `pool.addLiquidity(allowlistedAddress, ...)` and the extension approves, while the actual caller is never checked: [5](#0-4) [6](#0-5) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` is intended to restrict swaps to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs, or protocol-controlled addresses). Once the router is allowlisted, the guard fails open for every caller. Any unprivileged user can execute swaps on a pool designed to be closed, draining LP value at oracle prices without authorization. This is broken core pool functionality causing direct loss of the curation guarantee and potential loss of LP funds.

## Likelihood Explanation
The router is the primary public swap entrypoint. Any pool admin who deploys a curated pool and also wants users to access it through the standard periphery must allowlist the router. This is the expected operational pattern, making the bypass reachable on every curated pool that uses the router. No special permissions or malicious setup are required — a standard `exactInputSingle` call through the router is sufficient.

## Recommendation
The extension must gate on the actual end user, not the intermediary. Two complementary fixes:

1. **Pass the real user through the router**: `MetricOmmSimpleRouter` should forward the actual `msg.sender` (the end user) to the pool via `extensionData` or a dedicated field. The extension then reads the real user from that field after verifying the router's trusted status.
2. **Restrict direct-call-only for allowlisted pools**: The extension can require that `sender == tx.origin` or that the pool only accepts direct calls (no router intermediary) for allowlisted pools, and the router must be redesigned to pass user identity in a verifiable way.

The invariant that must hold: every guard must key authorization to the same actor that the economic action is actually attributed to.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin allowlists only `trustedUser`:
      setAllowedToSwap(pool, trustedUser, true)
  - Pool admin also allowlists the router (required for normal UI access):
      setAllowedToSwap(pool, address(router), true)

Attack:
  - `attacker` (not in allowlist) calls router.exactInputSingle(pool, ...)
  - Router calls pool.swap(recipient, ...) — msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - Extension checks allowedSwapper[pool][router] → true → passes
  - attacker's swap executes at oracle price on a pool intended to be closed to them

Result:
  - SwapAllowlistExtension is completely bypassed
  - attacker drains LP value on a curated pool without authorization
```
