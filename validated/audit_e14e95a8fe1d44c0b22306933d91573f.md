### Title
Unrestricted `run_mainchain_gc` Allows Any Caller to Permanently Lock Out Pending Proof Verifications — (`contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` carries only a `#[pause]` gate, not a `#[trusted_relayer]` or role-based access-control guard. When the contract is unpaused (normal production state), **any NEAR account** can call it. Because `remove_block_header` deletes entries from both `mainchain_header_to_height` and `headers_pool`, any block removed by GC can never be verified again. The guard inside `verify_transaction_inclusion` (`args.confirmations <= gc_threshold`) validates only the *requested* confirmation count, not the actual age of the target block, so a proof consumer can legitimately request `confirmations = 1` for a block that is older than `gc_threshold` — and an external caller can race-remove that block first.

---

### Finding Description

**Access control on `run_mainchain_gc`** [1](#0-0) 

The only decorator is `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. Per `near-plugins` semantics, this decorator blocks the call *only when the contract is paused*; when unpaused it is a no-op guard. There is no `#[trusted_relayer]` attribute (compare `submit_blocks` at line 168). Any NEAR account can invoke `run_mainchain_gc(batch_size)` in normal operation.

**What GC deletes** [2](#0-1) 

`remove_block_header` removes the hash from both `mainchain_header_to_height` and `headers_pool`. The deletion is permanent; there is no archive or recovery path.

**The insufficient guard in `verify_transaction_inclusion`** [3](#0-2) 

Line 290 checks `args.confirmations <= self.gc_threshold` — this validates only the *caller-supplied* confirmation count, not the actual depth of the target block. A proof consumer may legitimately pass `confirmations = 1` for a block whose real depth is `gc_threshold + N`. That call passes the guard. But if GC has already removed that block, line 300–301 panics with `"block does not belong to the current main chain"`.

**Concrete attack sequence**

1. Chain grows to `gc_threshold + N` stored blocks (routine after ~1 year with default `gc_threshold = 52704`).
2. Block B is the oldest stored block; its actual confirmation depth is `gc_threshold + N`.
3. Proof consumer submits `verify_transaction_inclusion` for block B with `confirmations = 1`.
4. Before that call lands, an external account calls `run_mainchain_gc(N)`.
5. GC removes N blocks including block B from both maps.
6. The proof consumer's call hits line 300, gets `None`, and panics permanently.

The `run_mainchain_gc` call at step 4 succeeds because: [4](#0-3) 

`amount_of_headers_we_store > gc_threshold` is satisfied, so the loop executes and removes block B.

---

### Impact Explanation

Permanent proof lockout. Once a block is removed from `headers_pool` and `mainchain_header_to_height`, no SPV proof for any transaction in that block can ever succeed. Downstream contracts or applications that depend on `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` returning `true` for those transactions are permanently broken. The attacker needs no stake, no role, and no privileged key — only a NEAR account and gas.

---

### Likelihood Explanation

- The chain exceeds `gc_threshold` in normal production operation (default ~52 704 blocks ≈ 1 year of Bitcoin blocks).
- `run_mainchain_gc` is a public, permissionless call when unpaused.
- The attacker only needs to front-run a proof consumer, which is straightforward on NEAR where transaction ordering within a block is observable.
- No special setup, no leaked keys, no social engineering.

---

### Recommendation

Add `#[trusted_relayer]` (or an explicit role check such as `Role::UnrestrictedRunGC` as a *positive* access-control requirement, not just a pause bypass) to `run_mainchain_gc` so that only authorized accounts can trigger GC externally.

Additionally, strengthen `verify_transaction_inclusion` to reject blocks whose actual depth exceeds `gc_threshold`:

```rust
// After resolving target_block_height:
require!(
    heaviest_block_header.block_height.saturating_sub(target_block_height) + 1
        <= self.gc_threshold,
    "Block is outside the GC retention window and may have been pruned"
);
```

This makes the stored-window invariant explicit and prevents proof consumers from requesting verification of blocks that are outside the retention window regardless of their `confirmations` argument.

---

### Proof of Concept

```rust
// NEAR SDK sandbox test (pseudocode)
// 1. Init contract with gc_threshold = 10
// 2. Submit 15 blocks via trusted relayer → mainchain has 15 blocks
// 3. Any account calls run_mainchain_gc(5) → removes 5 oldest blocks
// 4. Proof consumer calls verify_transaction_inclusion for one of the 5 removed blocks
//    with confirmations = 1
// Expected: panics with "block does not belong to the current main chain"
// Actual:   same — confirms permanent lockout

let attacker = worker.dev_create_account().await?;  // no special role
attacker
    .call(contract.id(), "run_mainchain_gc")
    .args_json(json!({"batch_size": 5}))
    .max_gas()
    .transact()
    .await?
    .unwrap();  // succeeds — no access control

let result = user_account
    .call(contract.id(), "verify_transaction_inclusion")
    .args_borsh(ProofArgs {
        tx_id: removed_block_tx,
        tx_block_blockhash: removed_block_hash,
        tx_index: 0,
        merkle_proof: valid_proof,
        confirmations: 1,   // passes args.confirmations <= gc_threshold
    })
    .transact()
    .await?;
assert!(result.is_failure());  // panics: "block does not belong to the current main chain"
```
