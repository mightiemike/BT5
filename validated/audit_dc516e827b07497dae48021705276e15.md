Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End-User, Allowing Allowlist Bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is the direct caller, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. If the router is allowlisted (required for any router-mediated swap to work on the pool), every unpermissioned user bypasses the per-user allowlist entirely.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` verbatim to the extension via `abi.encodeCall`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The same applies to `exactInput` (all hops), `exactOutputSingle`, and `exactOutput` — the router is always the direct caller of each pool's `swap()`: [5](#0-4) 

The extension never sees the end user's address. The `DepositAllowlistExtension` does **not** share this bug because it checks `owner`, an explicit parameter passed by the caller rather than derived from `msg.sender`: [6](#0-5) 

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension` as `beforeSwap` hook to restrict swaps to KYC'd addresses.
2. Admin calls `setAllowedToSwap(pool, router, true)` — required for any router-mediated swap to function.
3. Attacker (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` — pool's `msg.sender` = router.
5. Pool calls `_beforeSwap(router, recipient, ...)` — `sender` = router.
6. Extension checks `allowedSwapper[pool][router]` == `true` → passes.
7. Swap executes for attacker despite attacker not being on the allowlist.

Conversely, if the admin does **not** allowlist the router, legitimately allowlisted users cannot swap through the router at all — the extension sees `sender = router`, which is not in the allowlist, and reverts.

## Impact Explanation
This is a broken core pool functionality issue with direct fund impact. A pool admin configures `SwapAllowlistExtension` to restrict swaps to curated/KYC'd addresses (e.g., to protect LPs from trading with sanctioned counterparties or to enforce regulatory compliance). Once the router is allowlisted (the only way to make router-mediated swaps work), every unpermissioned user can bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`. The per-user curation is completely defeated, allowing unauthorized users to drain a curated pool. Alternatively, if the router is not allowlisted, the pool's primary public swap path is broken for all legitimately allowlisted users. Both outcomes are fund-impacting: either unauthorized users access a restricted pool, or legitimate LPs cannot exit positions.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical public swap entry-point for all pools. Any pool that enables `SwapAllowlistExtension` and also wants router support will hit this immediately upon deployment. The trigger requires only a standard `exactInputSingle` call — no special permissions, no flash loans, no price manipulation. The precondition (router allowlisted) is a natural operational step that any pool admin would take.

## Recommendation
Pass the originating user through the swap call so the extension can gate on the correct actor. Options:

1. **Add an `originator` field to `pool.swap()`** that the router populates with `msg.sender` before delegating to the pool. The extension then checks this field instead of `sender`.
2. **Enforce the allowlist at the router level** — the router reads the pool's configured allowlist provider and checks `msg.sender` before forwarding. This keeps the pool interface unchanged but requires the router to be allowlist-aware.
3. **Gate on `recipient` instead of `sender`** if the pool's security model is output-side (less common and does not solve the general case).

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured as beforeSwap hook
  - Pool admin calls setAllowedToSwap(pool, router, true)
    // router must be allowed for any router swap to work
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  - attacker calls router.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(recipient, ...)  →  pool.msg.sender = router
  - Pool calls _beforeSwap(router, recipient, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true  →  PASSES
  - Swap executes for attacker despite attacker not being on the allowlist

Result:
  - attacker successfully swaps on a curated pool
  - SwapAllowlistExtension provides zero per-user protection for any router-mediated swap
```

Foundry test outline:
1. Deploy `SwapAllowlistExtension`, pool with extension as `beforeSwap` hook.
2. Call `setAllowedToSwap(pool, router, true)`.
3. From `attacker` address (not allowlisted), call `router.exactInputSingle(...)`.
4. Assert swap succeeds (no `NotAllowedToSwap` revert).
5. Confirm `allowedSwapper[pool][attacker]` is `false`, proving bypass.
