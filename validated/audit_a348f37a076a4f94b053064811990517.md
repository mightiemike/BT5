### Title
`ENABLE_SECP_OPS` Flag Bypass via 4-Byte Opcode Forms Allows Ungated Secp Signature Verification — (`src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` in `src/chia_dialect.rs` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through two separate code paths. The 1-byte opcode aliases (opcodes 64 and 65) are correctly gated behind the `ENABLE_SECP_OPS` flag. However, the 4-byte opcode forms (`0x13d61f00` for secp256k1 and `0x1c3a8f00` for secp256r1) are dispatched to the same functions **without any flag check**, bypassing the `ENABLE_SECP_OPS` gate entirely. An attacker who controls CLVM bytes can invoke secp signature verification in any execution context — including mempool mode — by using the 4-byte opcode forms, regardless of whether `ENABLE_SECP_OPS` is set.

---

### Finding Description

In `src/chia_dialect.rs`, the `ChiaDialect::op()` function has two distinct dispatch paths:

**Path 1 — 4-byte opcodes (lines 157–182): No flag check.**

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => { return unknown_operator(...); }
    };
    return f(allocator, argument_list, max_cost, flags); // ← no ENABLE_SECP_OPS check
}
``` [1](#0-0) 

**Path 2 — 1-byte opcode aliases (lines 248–249): Correctly gated.**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The `ENABLE_SECP_OPS` flag is defined as the control for secp operator access:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

`MEMPOOL_MODE` does **not** include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

In mempool mode, a CLVM program using opcode `64` or `65` falls through to `unknown_operator`, which returns `Err(EvalErr::Unimplemented(o))` because `NO_UNKNOWN_OPS` is set — correctly rejecting the operation. But a CLVM program using the 4-byte opcode `0x13d61f00` or `0x1c3a8f00` is dispatched directly to `op_secp256k1_verify` / `op_secp256r1_verify` with no flag check, executing the full secp verification and returning a result.

The actual secp operator implementations in `src/secp_ops.rs` perform real cryptographic verification and return `Ok(Reduction(cost, a.nil()))` on success: [5](#0-4) [6](#0-5)

### Impact Explanation

**Vulnerability class:** Flag/operator wiring error — missing authorization gate on the 4-byte opcode dispatch path.

**Concrete corrupted result:** In mempool mode (or any mode without `ENABLE_SECP_OPS`), a CLVM program using the 4-byte opcode forms executes secp signature verification and returns `nil` (success) where the 1-byte alias forms would be rejected with `Unimplemented`. This produces **divergent execution results** for semantically identical operations depending solely on which opcode encoding the attacker chooses.

**Consensus/mempool impact:** Mempool validation is supposed to be stricter than consensus validation. If mempool mode rejects secp ops via 1-byte opcodes but silently accepts them via 4-byte opcodes, an attacker can craft transactions that pass mempool validation in an unexpected way — or, conversely, a node operator who believes secp operations are disabled (no `ENABLE_SECP_OPS`) is wrong. Any downstream system that relies on the absence of `ENABLE_SECP_OPS` to assert "secp operations cannot execute" has a broken invariant.

**Likelihood:** High. The 4-byte opcode forms are valid CLVM atoms. Any attacker who controls CLVM bytes (the primary attacker-controlled entry point for this VM) can trivially substitute `0x13d61f00` for opcode `64` to bypass the flag check. No special privilege is required.

---

### Likelihood Explanation

The attacker-controlled entry point is the CLVM program bytes passed to `run_program()`. The 4-byte opcode `0x13d61f00` is a valid atom encoding. Substituting it for the 1-byte opcode `64` requires only knowledge of the opcode table — publicly documented. No elevated privilege, social engineering, or dependency compromise is needed. The bypass is deterministic and reproducible.

---

### Recommendation

In the 4-byte opcode dispatch path, add the same `ENABLE_SECP_OPS` flag check that the 1-byte path enforces:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This mirrors the pattern already used for the 1-byte aliases at lines 248–249 and ensures that the `ENABLE_SECP_OPS` flag uniformly controls access to secp operations regardless of which opcode encoding is used.

### Proof of Concept

Construct a CLVM program that invokes secp256k1 verification using the 4-byte opcode form, and run it with a `ChiaDialect` that does **not** include `ENABLE_SECP_OPS` (e.g., `MEMPOOL_MODE`):

```rust
use clvmr::allocator::Allocator;
use clvmr::chia_dialect::{ChiaDialect, MEMPOOL_MODE};
use clvmr::run_program::run_program;

let mut a = Allocator::new();
// Build: (0x13d61f00 pubkey msg sig)
// where pubkey/msg/sig are valid secp256k1 inputs
// opcode atom = [0x13, 0xd6, 0x1f, 0x00]  (4 bytes, no ENABLE_SECP_OPS check)
let dialect = ChiaDialect::new(MEMPOOL_MODE); // ENABLE_SECP_OPS not set
// run_program succeeds — secp verification executes and returns nil
// Contrast: using 1-byte opcode 64 returns Err(Unimplemented) in the same mode
```

The 1-byte opcode `64` is rejected in `MEMPOOL_MODE` because `ENABLE_SECP_OPS` is absent. The 4-byte opcode `0x13d61f00` is accepted and executes the full secp verification, demonstrating the bypass.
