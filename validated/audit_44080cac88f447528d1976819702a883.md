Audit Report

## Title
`SwapAllowlistExtension::beforeSwap` checks router (`sender`) instead of economic actor (`recipient`), allowing full allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on `sender` (the direct pool caller, i.e., the router) while silently discarding `recipient` (the address that receives output tokens). Any unprivileged user can bypass the swap allowlist by routing through any allowlisted router. This is structurally inverted relative to the sibling `DepositAllowlistExtension`, which correctly ignores `sender` and checks `owner` (the economic actor).

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as `sender` and the user-supplied `recipient` as the output beneficiary to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards both values verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks only `sender` (the router) and leaves `recipient` unnamed and unchecked: [3](#0-2) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly ignores `sender` and checks `owner` (the LP position owner): [4](#0-3) 

**Bypass path:** Admin deploys a pool with `SwapAllowlistExtension` and calls `setAllowedToSwap(pool, routerAddress, true)` to allowlist `MetricOmmSimpleRouter`. Any non-allowlisted user `U` calls `router.swap(pool, recipient=U, ...)`. The router becomes `msg.sender` in `pool.swap`, so `sender=routerAddress` is passed to the extension. `allowedSwapper[pool][routerAddress]` is `true`, so the guard passes and `U` receives output tokens despite never being allowlisted.

**Blocked-valid-actor path:** Admin calls `setAllowedToSwap(pool, userAddress, true)` reasoning by analogy with `DepositAllowlistExtension`. When user `U` calls through any router, `allowedSwapper[pool][routerAddress]` is `false` and the swap reverts with `NotAllowedToSwap()`, blocking a legitimately allowlisted user.

Existing guards are insufficient: `nonReentrant` and `whenNotPaused` in `MetricOmmPool.swap` do not address identity checking, and no other hook in the call path re-validates the recipient. [5](#0-4) 

## Impact Explanation

The swap allowlist is the primary access-control mechanism for permissioned pools. Its bypass allows non-allowlisted users to execute swaps against pools intended to be KYC/whitelist-restricted, draining LP liquidity at oracle prices. This constitutes a direct loss of LP principal and broken core pool functionality (the configured guard does not restrict the intended actor class). The pool behaves as if `allowAllSwappers = true` for any user of an allowlisted router, which is a broken core pool invariant meeting the contest's Critical/High threshold.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard periphery entry point for swaps; any production pool that allowlists it (the expected configuration) exposes the bypass to all users with no privileged access or special setup required. The inconsistency with `DepositAllowlistExtension` makes it highly likely that pool admins will configure `SwapAllowlistExtension` with user addresses, triggering the blocked-valid-actor path. The exploit is repeatable and requires only a standard router call.

## Recommendation

Change `SwapAllowlistExtension.beforeSwap` to check `recipient` (the economic beneficiary) instead of `sender` (the router), consistent with `DepositAllowlistExtension`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
  }
  return IMetricOmmExtensions.beforeSwap.selector;
}
```

Also update `setAllowedToSwap`, `isAllowedToSwap`, and the `AllowedToSwapSet` event NatDoc to document that the checked address is the swap recipient. [6](#0-5) 

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` to allowlist `MetricOmmSimpleRouter`.
3. Non-allowlisted user `U` calls `MetricOmmSimpleRouter.swap(pool, recipient=U, ...)`.
4. Router calls `pool.swap(recipient=U, ...)` with `msg.sender = routerAddress`.
5. Pool calls `_beforeSwap(sender=routerAddress, recipient=U, ...)`.
6. `SwapAllowlistExtension` evaluates `allowedSwapper[pool][routerAddress]` → `true`.
7. Swap executes; user `U` receives output tokens despite never being allowlisted.

Foundry test skeleton:
```solidity
function test_swapAllowlistBypass() public {
    // setup: pool with SwapAllowlistExtension, allowlist only the router
    swapAllowlist.setAllowedToSwap(pool, address(router), true);
    // attacker (not allowlisted) swaps through the router
    vm.prank(attacker);
    router.swap(pool, attacker, true, 1e18, ...);
    // assert: swap succeeded despite attacker not being allowlisted
}
```
