Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of Actual Swapper, Enabling Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is the immediate `msg.sender` to the pool — the router contract — not the originating user. When a pool admin allowlists the router address to permit legitimate users to swap via `MetricOmmSimpleRouter`, every non-allowlisted user can bypass the restriction by routing through the same public router. This breaks the pool admin's access-control boundary without any privileged action.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- immediate caller, not originating user
  recipient,
  ...
  extensionData
);
```

`ExtensionCalling._beforeSwap` forwards this `sender` verbatim to every configured extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` then gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the immediate pool caller:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` (and all `exact*` variants) calls `pool.swap(...)` directly, making the router the `msg.sender` to the pool:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
  );
```

The extension therefore checks whether the **router** is allowlisted, not whether the **user** is allowlisted. No existing guard in the extension, pool, or router corrects this misbinding.

## Impact Explanation

This is an admin-boundary break. A pool admin who deploys a `SwapAllowlistExtension` (e.g., for a KYC-gated, RWA, or restricted-counterparty pool) and allowlists the router address — a necessary step for any legitimate user to swap via the router — inadvertently opens the gate to every non-allowlisted user who routes through `MetricOmmSimpleRouter`. The exact corrupted value is the `allowedSwapper[pool][sender]` authorization decision: it evaluates to `true` for the router when it should evaluate against the originating user. This allows unauthorized principals to trade on curated pools, breaking the admin's intended access-control boundary with direct fund-impacting consequences.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool that deploys a `SwapAllowlistExtension` and expects users to swap via the router is affected. The router is a public, permissionless contract. No special setup, privileged access, or unusual conditions are required — a standard router swap call is sufficient to trigger the bypass. The precondition (router allowlisted by admin) is the natural operational state for any allowlisted pool that supports router-mediated swaps.

## Recommendation

The `SwapAllowlistExtension` must not rely on `sender` (the immediate pool caller) as the authorization subject. Two approaches:

1. Have the router forward the original `msg.sender` as part of `extensionData`, and have the extension decode and check that value — with the pool or factory enforcing that only trusted periphery contracts may supply this field.
2. Pass the true originating user through a trusted forwarder pattern with signature verification.

Additionally, the core pool should document that `sender` in extension hooks is the immediate `msg.sender` to the pool, not the end user, so extension authors do not conflate the two.

## Proof of Concept

```
1. Pool admin deploys a pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, alice, true)       // Alice is allowlisted
3. Pool admin calls setAllowedToSwap(pool, router, true)      // Router allowlisted so Alice can use it
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ..., recipient: bob})
5. Router calls pool.swap(recipient=bob, ...) — msg.sender to pool = router address.
6. MetricOmmPool._beforeSwap(sender=router, ...) is invoked.
7. SwapAllowlistExtension.beforeSwap checks:
       allowedSwapper[pool][router] → true  (admin added router in step 3)
8. Hook passes. Bob's swap executes successfully despite not being allowlisted.
9. Repeat for exactInput, exactOutputSingle, exactOutput — all router entry points
   call pool.swap() directly, all produce sender=router to the pool.
```
