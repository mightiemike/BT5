### Title
Fork Retarget Difficulty Computed from Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/lib.rs`, `contract/src/bitcoin.rs`)

---

### Summary

`get_header_by_height` unconditionally reads from `mainchain_height_to_header`. When `get_next_work_required` is called for a fork block that falls on a retarget boundary, it fetches the retarget-interval start block from the **mainchain** map rather than walking the fork's own ancestor chain. An attacker who controls a trusted-relayer account can craft a fork whose block at `first_block_height` carries a later timestamp than the mainchain's block at the same height, causing `calculate_next_work_required` to see a larger `actual_time_taken` and therefore produce an easier `bits` value. The fork's retarget block is then accepted with a difficulty lower than the fork's own chain history would mandate.

---

### Finding Description

**Entrypoint** — `submit_blocks` (gated by `#[trusted_relayer]`): [1](#0-0) 

`submit_blocks` iterates headers and calls `submit_block_header`, which calls `check_target` → `check_pow` → `get_next_work_required`: [2](#0-1) 

**Retarget path in `get_next_work_required`** — when `(prev_block_height + 1) % 2016 == 0`, the function computes `first_block_height` and calls `get_header_by_height`: [3](#0-2) 

**`get_header_by_height` always reads the mainchain map:** [4](#0-3) 

`mainchain_height_to_header` is only written by `store_block_header` (mainchain path) and `reorg_chain`. Fork blocks are stored exclusively in `headers_pool` via `store_fork_header`: [5](#0-4) 

Therefore, when a fork block sits at a retarget boundary and the fork diverged **before** `first_block_height`, `get_header_by_height(first_block_height)` returns the **mainchain's** block at that height, not the fork's ancestor.

**`calculate_next_work_required` uses the returned timestamp directly:** [6](#0-5) 

`actual_time_taken = prev_block_time − first_block_time`. If the mainchain's block at `first_block_height` has an earlier timestamp than the fork's block at the same height, `actual_time_taken` is inflated, the clamped value is larger, and the resulting `bits` encodes an easier target.

---

### Impact Explanation

A fork block at a retarget boundary is accepted with a `bits` value derived from the mainchain's retarget-interval start timestamp rather than the fork's own ancestor's timestamp. This directly violates the invariant that difficulty for a fork block must be computed from the fork's own chain history. Concretely:

- The fork block's `bits` field encodes an easier target than the fork's chain history requires.
- `check_pow` accepts the block because `expected_bits == block_header.bits` passes (both sides use the wrong, easier value).
- The PoW hash check then only requires the hash to be below the easier target.
- The attacker mines the retarget block with less work than the network would enforce, admitting a header that is invalid under Bitcoin consensus rules.

---

### Likelihood Explanation

Exploitation requires holding a trusted-relayer account (staking/registration gated by `RelayerManager`). The question explicitly lists the trusted-relayer submission path as a valid attack surface. Once a relayer account is obtained, the attacker:

1. Submits a fork chain that diverges before `first_block_height` of the next retarget interval.
2. Sets the fork's block at `first_block_height` to a timestamp later than the mainchain's block at that height (within MTP and future-time constraints).
3. Submits fork blocks up to the retarget boundary; the contract computes an easier `bits`.
4. Mines the retarget block against the easier target and submits it.

No DAO compromise, key leak, or social engineering is required beyond holding a relayer account.

---

### Recommendation

`get_header_by_height` must be replaced with a fork-aware ancestor lookup when validating fork blocks. Instead of indexing `mainchain_height_to_header`, the retarget-interval start block should be found by walking `prev_block_hash` links backward from `prev_block_header` through `headers_pool` until the target height is reached. This ensures the timestamp used in `calculate_next_work_required` always comes from the fork's own ancestor chain.

---

### Proof of Concept

```
State setup (Bitcoin mainnet, difficulty_adjustment_interval = 2016):
  Genesis at height 0; mainchain built to height 4031 (two full intervals).
  mainchain_height_to_header[2016] → block M2016 with time T_main.

Attack:
  1. Fork diverges at height 2010 (before first_block_height = 2016).
  2. Attacker submits fork blocks 2011..4031, each stored only in headers_pool.
     Fork block at height 2016 carries time T_fork > T_main (e.g., T_main + 3600).
  3. Attacker submits fork block at height 4032 (retarget boundary):
       prev_block_height = 4031
       first_block_height = 4031 - 2015 = 2016
       get_header_by_height(2016) → M2016 (mainchain), time = T_main
       actual_time_taken = fork_prev_time − T_main   (larger than correct value)
       → easier bits B_wrong
  4. Contract checks: expected_bits == B_wrong == block_header.bits  ✓
     Attacker mines hash below target(B_wrong) — less work than target(B_correct).
  5. Fork block at height 4032 accepted with incorrect, easier difficulty.

Assert: bits accepted by contract ≠ bits that would result from using
        fork's own block at height 2016 (time T_fork).
```

fn calculate_next_work_required(
    config: &NetworkConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let prev_block_time: i64 = prev_block_header.block_header.time.into();

    let mut actual_time_taken: i64 = prev_block_time - first_block_time;
    if actual_time_taken < config.pow_target_timespan / 4 {
        actual_time_taken = config.pow_target_timespan / 4;
    }
    if actual_time_taken > config.pow_target_timespan * 4 {
        actual_time_taken = config.pow_target_timespan * 4;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(actual_time_taken).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target = new_target
        / U256::from(<i64 as TryInto<u64>>::try_into(config.pow_target_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
