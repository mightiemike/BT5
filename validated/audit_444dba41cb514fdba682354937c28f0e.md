### Title
Attacker-Controlled `coinbase_tx_id` Bypasses 64-Byte Transaction Merkle Proof Forgery Mitigation - (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability. However, the mitigation is incomplete: the caller fully controls `coinbase_tx_id` and can set it equal to `tx_id` (an internal Merkle node hash), making the coinbase guard a no-op. An unprivileged NEAR caller can obtain a `true` return for a fake, non-existent Bitcoin transaction against any real mainchain block.

---

### Finding Description

`verify_transaction_inclusion_v2` performs two checks before returning a result:

1. **Length equality**: `args.merkle_proof.len() == args.coinbase_merkle_proof.len()`
2. **Coinbase guard**: `compute_root_from_merkle_proof(args.coinbase_tx_id, 0usize, &args.coinbase_merkle_proof) == header.block_header.merkle_root` [1](#0-0) 

The coinbase guard is intended to anchor the Merkle tree to a known-real transaction (the coinbase at index 0), preventing an attacker from substituting an internal node hash as a leaf. However, the contract **never validates that `coinbase_tx_id` is distinct from `tx_id`**, nor that it corresponds to the actual coinbase transaction. The field is a plain `H256` supplied entirely by the caller. [2](#0-1) 

`compute_root_from_merkle_proof` is a pure arithmetic function: given any starting hash and any proof vector, it walks up the tree using the position parity rule. [3](#0-2) 

Because the coinbase position is hardcoded to `0usize`, the guard only checks that the provided `coinbase_tx_id`, treated as a node at position 0, reaches the merkle root via `coinbase_merkle_proof`. An attacker can satisfy this with any internal node that sits at an even position in its level — including the leftmost internal node at depth 1 (parent of the real coinbase and T1), which is always at position 0 in its level.

---

### Impact Explanation

Any downstream NEAR contract (bridge, escrow, cross-chain settlement) that calls `verify_transaction_inclusion_v2` and releases funds or changes state on a `true` result can be exploited. The attacker obtains a `true` return for a fake 64-byte "transaction" (whose hash equals an internal Merkle node) against a real, confirmed Bitcoin block. The contract's stated purpose — preventing the 64-byte forgery — is entirely defeated.

---

### Likelihood Explanation

The entry point is fully public (`#[pause]`, no role restriction for callers). The attacker needs only:
- A real Bitcoin block already tracked in the light client's mainchain (trivially available)
- Knowledge of that block's Merkle tree (computable from public Bitcoin data)
- The ability to call `verify_transaction_inclusion_v2` on NEAR

No privileged role, leaked key, or social engineering is required.

---

### Recommendation

The coinbase guard must enforce that `coinbase_tx_id` is actually the coinbase transaction, not merely some hash that arithmetically reaches the root. Concretely:

- Require `coinbase_tx_id != tx_id` as a minimum guard.
- Better: require that the coinbase proof path is **disjoint** from the transaction proof path (i.e., at every level the sibling hashes differ), so the two proofs cannot be identical copies of each other.
- Best: follow the BIP-0098 / CVE-2017-12842 mitigation precisely: verify that the coinbase proof and the target-tx proof share no common prefix, ensuring the coinbase anchors a genuinely different leaf.

---

### Proof of Concept

Consider a real mainchain block with 4 transactions `[T0, T1, T2, T3]`:

```
Level 0 (leaves): T0   T1   T2   T3
Level 1:          H01=(T0‖T1)  H23=(T2‖T3)
Level 2 (root):   R = H(H01‖H23)
```

The attacker submits `ProofArgsV2` with:

```
tx_id               = H01          // internal node hash = hash of fake 64-byte tx
tx_block_blockhash  = <real block>
tx_index            = 0
merkle_proof        = [H23]        // length 1
coinbase_tx_id      = H01          // ← same as tx_id; the bypass
coinbase_merkle_proof = [H23]      // ← same as merkle_proof; length check passes
confirmations       = 1
```

**Coinbase guard evaluation** (`position = 0usize`):
```
compute_root_from_merkle_proof(H01, 0, [H23])
  → position 0 is even → compute_hash(H01, H23) = R  ✓
```

**Transaction proof evaluation** (inside `verify_transaction_inclusion`, `tx_index = 0`):
```
compute_root_from_merkle_proof(H01, 0, [H23])
  → R == header.block_header.merkle_root  ✓
```

`verify_transaction_inclusion_v2` returns `true` for the fake transaction `H01`. [4](#0-3) [5](#0-4) [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
