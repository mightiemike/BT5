### Title
CEI Violation in `withdrawCollateral` Enables Cross-Function Reentrancy via ERC-777 Token Callback — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs an external token transfer before updating the spot engine balance. With no reentrancy guard anywhere in the codebase, a recipient contract using an ERC-777 `tokensReceived` hook can re-enter `Endpoint.depositCollateral` during the transfer, crediting a deposit against a stale (pre-withdrawal) balance. When `withdrawCollateral` resumes and decrements the balance, the attacker retains both the withdrawn tokens and an inflated on-chain balance.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` — triggers `token.safeTransfer` to `withdrawPool`, which immediately calls `BaseWithdrawPool.submitWithdrawal`, which calls `token.safeTransfer(sendTo, amount)`. This is the external call that hands tokens to the recipient.
2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)` — only now is the balance decremented.
3. **Line 419** — `require(getHealth(sender, healthType) >= 0)` — health check runs after the balance update. [1](#0-0) 

The state update (`spotEngine.updateBalance`) must happen **before** the external transfer to follow CEI. It does not.

A grep across the entire repository confirms there is no `nonReentrant` modifier or `ReentrancyGuard` import anywhere: [2](#0-1) 

The `handleWithdrawTransfer` helper in `Clearinghouse` routes through `BaseWithdrawPool.submitWithdrawal`, which calls `token.safeTransfer(to, amount)` — the final hop that delivers tokens to the attacker-controlled address: [3](#0-2) [4](#0-3) 

---

### Impact Explanation

An attacker controlling a recipient contract with an ERC-777 `tokensReceived` hook can, during the token delivery inside `withdrawCollateral`, call `Endpoint.depositCollateral` with the just-received tokens. Because `spotEngine.updateBalance` has not yet been called, the spot engine still reflects the pre-withdrawal balance. The deposit increments the balance. When `withdrawCollateral` resumes and decrements the balance, the net result is:

- Attacker has physically received the withdrawn tokens.
- Attacker's spot engine balance is inflated by the deposit amount, effectively double-counting the collateral.

This corrupts the `SpotEngine` balance invariant for the attacker's subaccount, allowing them to borrow against or withdraw phantom collateral, directly threatening protocol solvency. [5](#0-4) 

---

### Likelihood Explanation

The trigger requires the collateral token to support ERC-777 or a similar callback mechanism (e.g., a token with a `transfer` hook). ERC-777 tokens are a supported token standard and are not exotic. The `withdrawCollateral` function is reachable by any user who submits a `WithdrawCollateral` transaction through `Endpoint`, which is an unprivileged, publicly callable entry point: [6](#0-5) 

No admin or privileged role is required. The attacker only needs to control the `sendTo` address (their own wallet/contract) and use a supported ERC-777 collateral token.

---

### Recommendation

1. Move `spotEngine.updateBalance` and `spotEngine.assertUtilization` to execute **before** `handleWithdrawTransfer`.
2. Add OpenZeppelin's `ReentrancyGuardUpgradeable` to `Clearinghouse` and apply `nonReentrant` to `withdrawCollateral` and any other function that makes external calls before completing state updates.

Corrected order:

```solidity
// CEI-compliant withdrawCollateral
int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);   // Effects first
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
handleWithdrawTransfer(token, sendTo, amount, idx);             // Interaction last
emit ModifyCollateral(amountRealized, sender, productId);
```

### Proof of Concept

```
1. Attacker deploys MaliciousRecipient implementing ERC-777 tokensReceived hook.
2. Attacker deposits 1000 USDC (ERC-777) into their subaccount via Endpoint.depositCollateral.
   → spotEngine balance[attacker] = 1000
3. Attacker submits WithdrawCollateral(amount=1000, sendTo=MaliciousRecipient) via Endpoint.
4. Endpoint calls Clearinghouse.withdrawCollateral.
5. withdrawCollateral calls handleWithdrawTransfer → token.safeTransfer(MaliciousRecipient, 1000).
6. ERC-777 hook fires on MaliciousRecipient.tokensReceived:
   a. MaliciousRecipient approves Endpoint for 1000 USDC.
   b. MaliciousRecipient calls Endpoint.depositCollateral(attacker_subaccount, 1000).
   c. Endpoint calls Clearinghouse.depositCollateral → spotEngine.updateBalance(+1000).
   d. spotEngine balance[attacker] = 1000 + 1000 = 2000  ← stale, pre-withdrawal value used
7. Hook returns; withdrawCollateral resumes.
8. spotEngine.updateBalance(productId, sender, -1000) executes.
   → spotEngine balance[attacker] = 2000 - 1000 = 1000
9. Health check passes (balance = 1000, no debt).

Result: Attacker withdrew 1000 USDC tokens AND retains 1000 USDC of spot engine balance.
        Attacker can now borrow or withdraw an additional 1000 USDC of phantom collateral.

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
