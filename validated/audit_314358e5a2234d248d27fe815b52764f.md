### Title
Rebasable Collateral Token Causes Accounting-to-Balance Desynchronization, Enabling Insolvency — (`core/contracts/Clearinghouse.sol`, `core/contracts/SpotEngine.sol`)

---

### Summary

Nado's collateral accounting records a strict nominal deposit amount and enforces only an internal accounting invariant (`totalDeposits >= totalBorrows`) on withdrawal. It never verifies that the Clearinghouse's actual ERC20 token balance covers the sum of all credited depositor claims. If a rebasable token (e.g., stETH, aUSDC) is registered as a spot collateral product and undergoes a negative rebase, the contract becomes underfunded while all accounting checks pass, allowing early withdrawers to drain the pool and permanently blocking later withdrawers from recovering their credited balance.

---

### Finding Description

**Deposit path — strict nominal amount credited:**

In `Endpoint.depositCollateralWithReferral()`, the user-specified `amount` is transferred from the caller and the same `amount` is encoded into a slow-mode `DepositCollateral` transaction: [1](#0-0) 

When the sequencer processes that transaction, `Clearinghouse.depositCollateral()` scales `txn.amount` by a decimal multiplier and credits it directly to the subaccount's balance in `SpotEngine`: [2](#0-1) 

No balance-before/after snapshot is taken. The accounting record is permanently anchored to the nominal `txn.amount`, not to what the contract actually holds.

**Withdrawal path — accounting invariant only, no real-balance check:**

`Clearinghouse.withdrawCollateral()` transfers `amount` tokens to the `withdrawPool` and then calls `spotEngine.assertUtilization(productId)`: [3](#0-2) 

`assertUtilization` enforces only the internal accounting invariant: [4](#0-3) 

This check computes `totalDeposits` and `totalBorrows` entirely from normalized accounting state (`totalDepositsNormalized * cumulativeDepositsMultiplierX18`). It never queries `token.balanceOf(address(this))`. The invariant `totalDeposits >= totalBorrows` can pass even when the actual token balance is far below `totalDeposits - totalBorrows`.

**The floating balance gap:**

For a rebasable token, the contract's actual ERC20 balance drifts independently of the accounting state after every rebase event. The protocol has no mechanism to reconcile the two. The invariant that must hold — `actual_token_balance(Clearinghouse) >= totalDeposits - totalBorrows` — is never asserted anywhere in the codebase.

---

### Impact Explanation

**Negative rebase scenario (solvency corruption):**

1. 10 users each deposit 100 units of a rebasable token. Accounting records 1000 units total deposited; contract holds 1000 tokens.
2. A negative rebase event (e.g., validator slashing for stETH) reduces every holder's balance by 10%. The contract now holds 900 tokens, but accounting still records 1000 units of depositor claims.
3. The first 9 users to withdraw 100 units each succeed — `assertUtilization` passes (accounting invariant holds), `safeTransfer` succeeds while tokens remain.
4. The 10th user's withdrawal of 100 units causes `safeTransfer` to revert (only 0 tokens remain), permanently locking their credited balance. Their accounting credit is valid but unrecoverable.

This is a direct solvency/accounting corruption: credited balances become unbacked, and the loss is socialized onto the last withdrawers rather than distributed proportionally.

**Positive rebase scenario (stuck funds):**

A positive rebase silently inflates the contract's token balance above what accounting tracks. The surplus tokens are permanently inaccessible to depositors and can only be extracted via privileged `rebalanceXWithdraw` to `X_ACCOUNT`, representing a loss of yield for all depositors.

---

### Likelihood Explanation

Likelihood is **medium**. The trigger requires a rebasable token to be registered as a spot collateral product via `SpotEngine.addOrUpdateProduct()`. This is an owner-controlled action, but the protocol's `IERC20Base` interface imposes no restriction on token type — any ERC20 with a `decimals()` function is accepted. Rebasable tokens (stETH, aTokens, rebasing stablecoins) are common DeFi primitives and plausible collateral choices. Once such a token is listed, any negative rebase event — which is externally triggered and unprivileged — activates the vulnerability. No attacker action is required beyond normal withdrawal calls.

---

### Recommendation

1. **Warn against rebasable tokens as collateral** in protocol documentation and `addOrUpdateProduct` NatSpec, mirroring the 1inch recommendation.
2. **Add a real-balance solvency check** in `assertUtilization`: compare `token.balanceOf(address(clearinghouse))` against `(totalDeposits - totalBorrows)` scaled to native decimals, and revert if the contract is underfunded.
3. **Use balance-before/after accounting on deposit**: in `Clearinghouse.depositCollateral`, credit only the delta `balanceAfter - balanceBefore` rather than the nominal `txn.amount`, so the accounting always reflects actual received tokens.
