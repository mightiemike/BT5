### Title
Fork Difficulty Calculated Using Stale Mainchain Block Timestamps Instead of Fork Ancestor Timestamps — (`contract/src/dogecoin.rs`)

### Summary

In the Dogecoin build, `get_next_work_required` fetches the start-of-interval block's timestamp via `get_header_by_height`, which unconditionally reads from the **mainchain** height-to-hash mapping. When this function is called during fork block validation, it returns the mainchain block at that height rather than the fork's actual ancestor, producing an incorrect difficulty target. The code itself contains a `TODO` comment acknowledging this uncertainty.

### Finding Description

`get_header_by_height` is implemented in `contract/src/lib.rs`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // always mainchain
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

In `contract/src/dogecoin.rs`, `get_next_work_required` calls this function to obtain the timestamp of the first block in the difficulty window:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [2](#0-1) 

The result feeds directly into `calculate_next_work_required`:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
``` [3](#0-2) 

For Dogecoin post-height 145,000, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `height_first = prev_block_height - 1`. [4](#0-3) 

When a fork block at height H is being validated:
- `prev_block_header` = fork's parent at H-1 (correctly fetched via `prev_block_hash`)
- `first_block_time` = **mainchain** block at H-2's timestamp (wrong — should be the fork's ancestor at H-2)

If the fork diverged at or before height H-2, `mainchain[H-2]` and `fork_ancestor[H-2]` are **different blocks** with potentially different timestamps. The difficulty calculation therefore uses the wrong timespan.

The same structural issue exists in the Bitcoin and Litecoin builds, but only becomes reachable when a fork spans a full 2016-block difficulty period, which is practically infeasible without majority hashrate. For Dogecoin the window is 1 block, so any fork that has diverged ≥ 2 blocks triggers the bug. [5](#0-4) 

### Impact Explanation

The computed `modulated_timespan` is wrong:

- If `mainchain[H-2].time < fork_ancestor[H-2].time`: the timespan is inflated → difficulty target is **easier** than correct → the contract accepts fork blocks whose PoW does not meet the true required difficulty. A chain reorg can be triggered with less accumulated work than the protocol requires.
- If `mainchain[H-2].time > fork_ancestor[H-2].time`: the timespan is deflated → difficulty is **harder** than correct → valid fork blocks are rejected, breaking liveness for fork tracking.

The security-critical case is the first: an attacker can craft a fork where the mainchain's block at `height_first` has an older timestamp than the fork's actual ancestor, causing the contract to accept under-difficulty fork blocks and potentially corrupt the canonical chain mapping (`mainchain_tip_blockhash`, `mainchain_height_to_header`, `mainchain_header_to_height`). [6](#0-5) 

### Likelihood Explanation

For Dogecoin (the only chain where `difficulty_adjustment_interval = 1` post-145,000), the bug is reachable for any fork that has diverged at least 2 blocks from the mainchain. Forks of this length are routine in normal Dogecoin operation. The entry path is `submit_blocks`, which carries a `#[trusted_relayer]` guard; however, the `trusted_relayer` macro implements a staking-based registration system (managed by `RelayerManager`), meaning any account that stakes can become a relayer and submit adversarial fork headers. [7](#0-6) 

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` (all chain variants) with a loop that walks the fork's actual ancestor chain via `get_prev_header` until reaching `height_first`. This mirrors how Bitcoin Core resolves the interval-start block: by traversing `prev_block_hash` links rather than a height index that is only valid for the canonical chain.

### Proof of Concept

1. Deploy the contract in Dogecoin mode (mainnet, `skip_pow_verification = false`).
2. Submit a valid mainchain sequence up to height 1002. Mainchain block at height 1000 has timestamp `T_main`.
3. Craft a fork starting at height 1000: fork block at 1000 has timestamp `T_fork` where `T_fork > T_main` (e.g., `T_fork = T_main + 3600`).
4. Submit fork blocks at heights 1001 and 1002. When validating fork block at 1002:
   - `prev_block_header` = fork[1001] (correct)
   - `height_first = 1000`
   - `get_header_by_height(1000)` returns **mainchain[1000]** with timestamp `T_main`
   - `modulated_timespan` = `fork[1001].time - T_main` (inflated by 3600 s)
   - Computed target is easier than the correct target derived from `fork[1001].time - fork[1000].time`
5. The contract accepts fork block 1002 with PoW that does not satisfy the correct difficulty, and if fork chainwork exceeds mainchain chainwork, triggers a reorg that corrupts the canonical chain state. [8](#0-7) [1](#0-0)

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
