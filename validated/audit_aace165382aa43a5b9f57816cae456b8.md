### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable Without Coinbase Proof Prerequisite, Enabling 64-Byte Transaction Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is still a live, publicly callable NEAR function that lacks the coinbase Merkle proof prerequisite introduced in `verify_transaction_inclusion_v2`. Any unprivileged NEAR caller — including a malicious recipient contract — can invoke the deprecated function directly, bypassing the 64-byte transaction forgery mitigation entirely. A forged proof returns `true`, causing any downstream contract that trusts the result to act on a Bitcoin transaction that never existed.

---

### Finding Description

The contract exposes two SPV proof verification entry points:

**`verify_transaction_inclusion` (deprecated v1)** — accepts `ProofArgs` and verifies a single Merkle path against the stored block header's `merkle_root`. It carries an explicit `#[deprecated]` annotation but remains `pub` and is guarded only by `#[pause]` (active by default). [1](#0-0) 

**`verify_transaction_inclusion_v2` (secure v2)** — accepts `ProofArgsV2` and adds the coinbase proof prerequisite: it first verifies a coinbase Merkle proof against the same `merkle_root`, enforcing that the proof depth is consistent with a real leaf node and not an internal tree node. [2](#0-1) 

The coinbase proof check is the **prerequisite** that prevents the 64-byte transaction forgery attack. In the H-01 analog, "signup" was the prerequisite before "purchase." Here, the coinbase proof validation is the prerequisite before accepting any SPV proof as valid. The v1 function omits this prerequisite entirely:

```
verify_transaction_inclusion:
  ✓ confirmations check
  ✓ block in main chain
  ✓ merkle_proof not empty
  ✗ NO coinbase proof prerequisite  ← missing check
```

Because `verify_transaction_inclusion` is `pub` with no access-control gate beyond `#[pause]`, any NEAR account can call it directly, skipping the coinbase proof step that v2 enforces.

**Root cause of the 64-byte forgery:** `compute_root_from_merkle_proof` uses `double_sha256(left_child || right_child)` to build internal nodes — the same operation used to hash a 64-byte raw transaction. There is no domain separation between leaf and internal nodes. [3](#0-2) 

An attacker who knows a real block's Merkle tree can identify two 32-byte internal child hashes `L` and `R` such that `double_sha256(L || R)` equals some internal node `N`. They then supply `tx_id = N` with a valid Merkle path from `N` to the block's `merkle_root`. The v1 function computes the root, finds it matches, and returns `true` — for a transaction that does not exist on Bitcoin. [4](#0-3) 

---

### Impact Explanation

A recipient NEAR DApp (e.g., a Bitcoin bridge, a cross-chain DEX, or a payment verifier) that calls `verify_transaction_inclusion` and acts on a `true` result will:

- Release bridged assets or tokens for a Bitcoin deposit that never occurred.
- Mark a payment as settled when no on-chain Bitcoin payment was made.
- Update its own state (balances, order books, escrow releases) based on a forged proof.

This is a direct theft / unauthorized movement of assets impact. The contract itself does not hold funds, but its verification result is the trust anchor for every downstream contract that integrates it. A false positive from `verify_transaction_inclusion` corrupts that trust anchor.

---

### Likelihood Explanation

- **No privileged access required.** Any NEAR account can call `verify_transaction_inclusion` directly.
- **The attack is documented and known.** The 64-byte transaction vulnerability (CVE-2012-2459 / BitMEX writeup) is well-understood; tooling to construct forged proofs exists.
- **The deprecated function is not removed.** It is still compiled into the deployed WASM and callable on-chain. Rust `#[deprecated]` is a compiler lint, not a runtime guard.
- **Integrators may not know to avoid v1.** The function appears in the public ABI. A new integrator reading the contract ABI without reading the deprecation notice will call v1.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or add an unconditional `env::panic_str("use verify_transaction_inclusion_v2")` at the top of the function body so it is unreachable at runtime. A Rust `#[deprecated]` attribute does not prevent on-chain invocation.

```rust
pub fn verify_transaction_inclusion(&self, ...) -> bool {
    env::panic_str("Removed: use verify_transaction_inclusion_v2 to prevent 64-byte tx forgery");
}
```

Alternatively, gate the function behind an access-control role so only trusted internal callers (i.e., `verify_transaction_inclusion_v2` itself) can reach it, and remove the `pub` visibility.

---

### Proof of Concept

**Setup:** A deployed `BtcLightClient` contract with a known Bitcoin block at height H, confirmed beyond the required threshold. The block has Merkle root `R` and contains at least two transactions, producing internal node `N = double_sha256(txid_0 || txid_1)`.

**Attacker steps:**

1. Observe the real block's Merkle tree. Identify internal node `N` at depth `d-1` (one level above the leaves), where `N = double_sha256(txid_0 || txid_1)`.
2. Construct a Merkle proof path `P` of length `d-1` from `N` up to `R` (this is a valid path through the real tree).
3. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (an internal node, not a real transaction)
   - `tx_block_blockhash` = the real block hash at height H
   - `tx_index` = the index corresponding to `N`'s position
   - `merkle_proof = P`
   - `confirmations` = any value ≤ current depth
4. The function computes `compute_root_from_merkle_proof(N, index, P)` → `R`, compares to `header.merkle_root` → `R`, and returns **`true`**.
5. The attacker presents this `true` result to a recipient bridge contract, which releases bridged BTC-equivalent tokens for a deposit that never happened.

**Why v2 blocks this:** `verify_transaction_inclusion_v2` requires a coinbase proof of the same depth `d`. The forged proof at depth `d-1` would fail the `merkle_proof.len() == coinbase_merkle_proof.len()` check if the attacker uses the real coinbase proof (depth `d`), or fail the coinbase root check if the attacker fabricates a shorter coinbase proof. [5](#0-4)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
