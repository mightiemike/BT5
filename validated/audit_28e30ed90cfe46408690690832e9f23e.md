Audit Report

## Title
Wrong-actor binding in `DepositAllowlistExtension.beforeAddLiquidity` allows unauthorized depositors to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` receives both `sender` (the actual `msg.sender` of `addLiquidity`) and `owner` (the position recipient) but silently discards `sender` and gates access solely on `owner`. Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint, any unauthorized actor can pass an allowlisted address as `owner`, satisfy the check, and have their tokens deposited into the pool — completely bypassing the allowlist's intended restriction.

## Finding Description
**Root cause — wrong parameter consumed:**

In `DepositAllowlistExtension.beforeAddLiquidity` the first parameter (`sender`) is unnamed and ignored:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`msg.sender` here is the pool (the caller of the extension). The check is therefore `allowedDepositor[pool][owner]` — it validates the *position recipient*, not the *actual depositor*.

**Call path that delivers both values:**

`ExtensionCalling._beforeAddLiquidity` correctly encodes both actors and passes them to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L88-99
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`sender` is `msg.sender` of `addLiquidity` (the real depositor). The extension receives it but throws it away.

**No `msg.sender == owner` guard in the pool:**

```solidity
// metric-core/contracts/MetricOmmPool.sol L182-196
function addLiquidity(address owner, uint80 salt, ...) external nonReentrant(...) {
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(..., owner, ...);
    ...
}
```

There is no check that `msg.sender == owner`. Compare with `removeLiquidity` which does enforce `if (msg.sender != owner) revert NotPositionOwner()`.

**Exploit flow:**
1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists only `alice`.
2. Unauthorized actor `eve` calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `extension.beforeAddLiquidity(eve, alice, ...)`.
4. Extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
5. `LiquidityLib.addLiquidity` credits the position to `alice`; tokens are pulled from `eve` via the callback.
6. `eve`'s tokens enter the pool; the allowlist is bypassed.

**Existing guards are insufficient:** The only guard is the allowlist check itself, which is bound to the wrong actor. No other mechanism in `addLiquidity` verifies the caller's allowlist status.

## Impact Explanation
The deposit allowlist is a core access-control mechanism. Its bypass means:
- Unauthorized actors can inject liquidity into pools that are explicitly restricted (e.g., KYC-gated, regulatory-compliant, or curated pools).
- The unauthorized actor can grief an allowlisted address by forcing an unwanted position onto it (in arbitrary bins, with arbitrary `salt`), which the allowlisted address cannot prevent.
- Pool invariants around "only approved depositors hold positions" are broken, which can have downstream compliance or protocol-design consequences.

This maps to **broken core pool functionality** (the allowlist hook is the designated enforcement point and it fails to enforce). Severity: **Medium** — no direct theft of existing LP funds, but a named access-control invariant is fully defeated by any unprivileged caller.

## Likelihood Explanation
The attack requires no special privileges, no flash loan, and no oracle manipulation. Any EOA or contract can call `addLiquidity` with an allowlisted `owner`. It is repeatable at will, costs only gas plus the deposited tokens (which go to the `owner`'s position), and works on every pool that uses this extension. Likelihood: **High**.

## Recommendation
In `beforeAddLiquidity`, check `sender` (the actual depositor) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Additionally, consider whether `addLiquidity` should enforce `msg.sender == owner` (as `removeLiquidity` already does) to prevent unsolicited position creation on behalf of third parties.

## Proof of Concept
```solidity
// Foundry test sketch
function test_allowlistBypass() public {
    // Setup: pool with DepositAllowlistExtension; only alice is allowlisted
    address alice = makeAddr("alice");
    address eve   = makeAddr("eve");
    extension.setAllowedToDeposit(address(pool), alice, true);

    // eve is NOT allowlisted
    assertFalse(extension.isAllowedToDeposit(address(pool), eve));

    // eve calls addLiquidity with owner = alice
    vm.startPrank(eve);
    token0.approve(address(router), type(uint256).max);
    token1.approve(address(router), type(uint256).max);
    // This should revert but does NOT — eve bypasses the allowlist
    pool.addLiquidity(alice, 0, deltas, callbackData, "");
    vm.stopPrank();

    // Position is credited to alice; eve's tokens entered the pool
    // allowlist invariant violated
}
```
