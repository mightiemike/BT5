Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension` is documented to gate `addLiquidity` by depositor address. Its `beforeAddLiquidity` hook silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and instead checks `owner` (the LP-position recipient). Any unprivileged address can bypass the allowlist by calling `pool.addLiquidity` with an allowlisted address as `owner`, while the unauthorized caller pays the tokens and the extension passes without reverting.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the LP-position recipient to the extension hook: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` silently drops `sender` (unnamed first parameter) and checks only `owner`: [2](#0-1) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller) and discards the second parameter (`recipient`): [3](#0-2) 

The exploit path: an attacker calls `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)`. The pool passes `msg.sender = attacker` as `sender` (discarded by the extension) and `allowlistedAddress` as `owner` (checked and passes). The extension does not revert. The attacker pays tokens via the `metricOmmModifyLiquidityCallback` and the LP position is minted to `allowlistedAddress`. No existing guard in `addLiquidity` checks that `msg.sender` is allowlisted.

## Impact Explanation

The pool admin deploys a pool with `DepositAllowlistExtension` to restrict deposits to approved addresses (e.g., KYC'd users). The admin-configured access boundary is broken: any unprivileged address can deposit into the restricted pool by supplying an allowlisted address as `owner`. This constitutes an admin-boundary break — an unprivileged path bypasses a pool-admin-configured access control invariant. Additionally, the allowlisted `owner` receives LP shares they did not request, enabling griefing.

## Likelihood Explanation

High. The bypass requires no special privileges, no flash loan, and no complex setup. Any address can call `pool.addLiquidity` directly with an allowlisted `owner`. The only cost to the attacker is the token payment for the deposit, which is transferred to the pool and credited as LP shares to the allowlisted address.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual caller/payer) instead of `owner`, consistent with how `SwapAllowlistExtension` handles `beforeSwap`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

## Proof of Concept

```solidity
// Pool configured with DepositAllowlistExtension
// Only `alice` is allowlisted as a depositor

address attacker = makeAddr("attacker");
// attacker is NOT allowlisted

// Attacker calls addLiquidity with alice as owner
vm.prank(attacker);
pool.addLiquidity(
    alice,       // owner = allowlisted address → extension checks this → passes
    salt,
    deltas,
    callbackData,
    extensionData
);
// Extension receives: sender = attacker (discarded), owner = alice (checked)
// allowedDepositor[pool][alice] == true → no revert
// Attacker pays tokens via callback
// LP position minted to alice
// Deposit allowlist bypassed
```

The call path is: `pool.addLiquidity` → `_beforeAddLiquidity(msg.sender=attacker, owner=alice, ...)` → `DepositAllowlistExtension.beforeAddLiquidity(attacker, alice, ...)` → checks `allowedDepositor[pool][alice]` → `true` → passes.
