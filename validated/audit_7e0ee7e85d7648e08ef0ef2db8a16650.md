### Title
Silently Failed `DepositCollateral` Slow-Mode Transactions Permanently Lock User Funds — (`core/contracts/Endpoint.sol`)

---

### Summary

`depositCollateralWithReferral` in `Endpoint.sol` takes custody of a user's ERC-20 tokens immediately, then queues a slow-mode `DepositCollateral` transaction for deferred on-chain credit. If that slow-mode transaction later fails during execution, the `try/catch` block in `_executeSlowModeTransaction` silently discards the error. The funds already transferred to the `Endpoint` contract are permanently unrecoverable — the refund path was explicitly removed (evidenced by the comment `// try return funds now removed`).

---

### Finding Description

**Step 1 — Funds taken upfront.**
`depositCollateralWithReferral` calls `handleDepositTransfer` to pull tokens from `msg.sender` into the `Endpoint` contract before any slow-mode processing occurs:

```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
```

It then enqueues a `SlowModeTx` with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (hardcoded 3 days). [1](#0-0) 

**Step 2 — Silent failure on execution.**
When the slow-mode transaction is eventually executed (by the sequencer or by anyone after the delay), `_executeSlowModeTransaction` wraps the call in a `try/catch`. If `processSlowModeTransaction` reverts for any reason, the catch block does nothing — and the comment at line 226 explicitly acknowledges the missing refund:

```solidity
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    // ...gas heuristic...
    // try return funds now removed
}
``` [2](#0-1) 

**Step 3 — No recovery path.**
After the `SlowModeTx` entry is deleted (`delete slowModeTxs[_slowModeConfig.txUpTo++]`), there is no on-chain mechanism for the user to reclaim the tokens that were already transferred. The `withdrawFromDirectDepositV1` function in `ContractOwner.sol` is `onlyOwner` and only covers the `DirectDepositV1` sub-contract, not the `Endpoint` itself. [3](#0-2) 

**Failure conditions that cause `processSlowModeTransactionImpl` to revert:**
- The subaccount's address is added to the sanctions list between deposit and execution (the `requireUnsanctioned` check in `submitSlowModeTransactionImpl` runs at submission time, but `clearinghouse.depositCollateral` may enforce its own checks).
- The product is delisted between deposit and execution.
- Any revert inside `clearinghouse.depositCollateral` (e.g., overflow, zero-address token, or future validation logic). [4](#0-3) 

The same risk applies to `DirectDepositV1.creditDeposit()`, which calls `endpoint.depositCollateralWithReferral` on behalf of a subaccount — if the resulting slow-mode tx fails, the tokens already approved and transferred are lost. [5](#0-4) 

---

### Impact Explanation

Any user who calls `depositCollateralWithReferral` (directly or via `DirectDepositV1.creditDeposit`) and whose slow-mode `DepositCollateral` transaction subsequently fails will permanently lose the deposited tokens. The tokens sit in the `Endpoint` contract with no user-accessible withdrawal path. The asset delta is the full `amount` passed to the deposit call.

---

### Likelihood Explanation

The 3-day delay between token custody and credit creates a meaningful window for state changes (sanctions, product delisting, future validation additions). The comment `// try return funds now removed` confirms the refund path was deliberately deleted, meaning the protocol is aware of the gap but has not replaced it with an alternative recovery mechanism. Any user depositing via the slow-mode path is exposed.

---

### Recommendation

- **Short term:** Restore a fund-return mechanism in the `catch` block of `_executeSlowModeTransaction`. When a `DepositCollateral` slow-mode transaction fails, the contract should refund the original `amount` to the depositor's address.
- **Long term:** Maintain an explicit mapping of `(subaccount, productId) → pendingDepositAmount` so that users can self-serve reclaim funds for any slow-mode deposit that was consumed but never credited. Document all slow-mode transaction states and their transitions, including the failed/unprocessed state.

---

### Proof of Concept

1. User calls `depositCollateralWithReferral(subaccount, productId, 1000e6, "")`.
2. `handleDepositTransfer` pulls 1000 USDC from the user into `Endpoint`. A `SlowModeTx` is queued with `executableAt = now + 3 days`.
3. During the 3-day window, the user's address is added to the sanctions list (or the product is delisted).
4. After 3 days, anyone calls `executeSlowModeTransaction()`.
5. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(...)` inside `try/catch`. The inner call reverts (sanctions check or product check).
6. The `catch` block executes: `// try return funds now removed` — no refund issued, `SlowModeTx` entry already deleted.
7. User's 1000 USDC remains in `Endpoint` with no on-chain path to recover it. [6](#0-5) [7](#0-6)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,

                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
```
