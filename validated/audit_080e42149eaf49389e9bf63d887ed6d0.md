### Title
Gas-key exec-fee host-function path uses wire length instead of trie-ID length for ML-DSA-65 keys, causing ~59x per-byte overcharge — (`runtime/near-vm-runner/src/logic/logic.rs`)

### Summary

The host-function implementations of `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` pass the raw wire/borsh length of the public key (`public_key_len as usize`, 1953 bytes for ML-DSA-65) to `gas_key_transfer_exec_fee` and `gas_key_add_key_exec_fee`. Both fee helpers use that value to compute the trie key length via `access_key_key_len(account_id_len, public_key_len)`. For ML-DSA-65 keys the trie stores a 33-byte SHA3-256 hash, not the 1953-byte full key. The transaction-path caller in `permission_exec_fees` correctly passes `public_key.trie_id_len()` (33 bytes), but the host-function path passes the wire length (1953 bytes), producing a ~59× overcharge on the per-byte exec-fee component for every gas-key operation involving an ML-DSA-65 key.

### Finding Description

`PublicKey::len()` returns the borsh-encoded wire length (1 + 1952 = 1953 bytes for ML-DSA-65). `PublicKey::trie_id_len()` returns the on-trie identifier length (1 + 32 = 33 bytes for ML-DSA-65, because the trie stores the SHA3-256 hash of the key, not the key itself). [1](#0-0) [2](#0-1) 

The exec-fee helpers are documented to operate on the **trie key length**: [3](#0-2) [4](#0-3) 

The transaction-path caller in `permission_exec_fees` correctly passes `public_key.trie_id_len()`: [5](#0-4) 

But both host-function implementations pass `public_key_len as usize` — the raw wire length supplied by the contract: [6](#0-5) [7](#0-6) [8](#0-7) 

The wasmtime path has the identical defect: [9](#0-8) [10](#0-9) 

For an ML-DSA-65 key and a 10-character account ID:

| Path | `public_key_len` used | `trie_key_len` computed | Correct value |
|---|---|---|---|
| Transaction (`permission_exec_fees`) | 33 (trie_id_len) | 45 bytes | 45 bytes |
| Host function | 1953 (wire len) | 1965 bytes | 45 bytes |

The per-byte exec-fee rate is `gas_key_byte.exec_fee` = 101,435,400 gas/byte. The overcharge per `TransferToGasKey` call is `101,435,400 × (1965 − 45)` ≈ **194 Ggas**, compared to a base exec fee of ≈ 235 Ggas — an 82% surcharge on top of the base. For `AddGasKey` with 4 nonces the per-byte overcharge is ≈ **776 Ggas**, roughly 3× the nonce-write base fee.

The existing fee-parity test only exercises ED25519 keys and therefore does not catch this divergence: [11](#0-10) 

The design document explicitly flags this class of mistake as a known risk: [12](#0-11) 

### Impact Explanation

Any unprivileged contract that calls `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key will have its `gas_used` (prepaid exec gas) inflated by ~194–776 Ggas per call. Contracts that budget gas based on the documented fee schedule or on parity with the transaction path will fail with out-of-gas errors. Because the overcharge is deterministic and protocol-level, it cannot be worked around without knowing the exact divergence; it silently breaks any contract that uses ML-DSA-65 gas keys via host functions.

### Likelihood Explanation

`PostQuantumSignatures` (protocol version 85) and `GasKeys` are both stable and active on mainnet. Any contract author who follows the documented fee model or the transaction-path behavior will encounter this. The trigger requires only a valid ML-DSA-65 public key passed to one of the three host functions — no privileged role is needed.

### Recommendation

Replace `public_key_len as usize` with `public_key.trie_id_len()` in all three host-function call sites for the exec-fee helpers. The decoded `public_key` is already available at each call site (it is used immediately after to call `public_key.decode()?`), so the fix requires only substituting the length source:

```rust
// Before (wrong — wire length):
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,
);

// After (correct — trie-ID length):
let decoded_pk = public_key.decode()?;
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_pk.trie_id_len(),
);
```

Apply the same substitution to the `gas_key_add_key_exec_fee` call sites in `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`, and add a fee-parity test that exercises ML-DSA-65 keys alongside ED25519 keys.

### Proof of Concept

The divergence is directly observable by comparing the two code paths for the same ML-DSA-65 key:

1. **Transaction path** — `permission_exec_fees` in `runtime/runtime/src/config.rs` calls `gas_key_add_key_exec_fee(fees, account_id.len(), public_key.trie_id_len(), ...)` → `public_key.trie_id_len()` = 33 for ML-DSA-65 → `nonce_key_len` = 2 + account_id_len + 33 + 2.

2. **Host-function path** — `promise_batch_action_add_gas_key_with_full_access` in `logic.rs` calls `gas_key_add_key_exec_fee(&self.fees_config, receiver_id.len(), public_key_len as usize, ...)` → `public_key_len` = 1953 for ML-DSA-65 → `nonce_key_len` = 2 + account_id_len + 1953 + 2.

For a 10-character account ID and 4 nonces, the per-byte exec-gas charged via host function is `101_435_400 × (1967 − 47) × 4 ≈ 776 Ggas` versus the correct `101_435_400 × (47 − 47 + 47) × 4 ≈ 19 Ggas` — a ~40× total overcharge on the per-byte component alone. A contract that attaches the gas budget computed from the transaction-path formula will fail with `GasExceeded` when the same operation is performed via host function with an ML-DSA-65 key.

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

/// Exec fee when adding a gas key with `num_nonces` nonces, split into base
/// and per-byte components. Each nonce writes a trie entry of
/// (access_key_key_len + NonceIndex) key bytes and NONCE_VALUE_LEN value bytes.
pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
}
