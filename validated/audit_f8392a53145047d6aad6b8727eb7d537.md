Now I have all the information needed. Let me check one more thing — whether `BaseEngine` or `PerpEngineState` has a `_disableInitializers()` constructor guard.

The absence of `_disableInitializers()` in `BaseEngine`/`PerpEngine` is confirmed, while other contracts in the same repo (`Verifier`, `BaseWithdrawPool`, `BaseProxyManager`) all have it. This is a real vulnerability.

---

### Title
Uninitialized Proxy Race Allows Attacker to Hijack `canApplyDeltas` and Write Arbitrary Perp Balances — (`core/contracts/PerpEngine.sol`)

---

### Summary
`PerpEngine.initialize()` is an unrestricted `external` function with no caller access control. Neither `PerpEngine` nor its parent `BaseEngine` defines a `constructor()` that calls `_disableInitializers()`. An attacker who calls `initialize()` before the legitimate deployer does so will set attacker-controlled addresses into `canApplyDeltas`, then call `updateBalance()` from one of those addresses to write arbitrary `amount` and `vQuoteBalance` into any subaccount's `Balance` storage.

---

### Finding Description

`PerpEngine.initialize()` delegates directly to `BaseEngine._initialize()`, which is guarded only by OpenZeppelin's `initializer` modifier: [1](#0-0) 

The `initializer` modifier prevents a *second* call, but it does not restrict *who* makes the first call. There is no `onlyOwner`, no factory-only check, and no `_disableInitializers()` in a constructor to lock the implementation slot.

`_initialize()` unconditionally writes attacker-supplied addresses into `canApplyDeltas`: [2](#0-1) 

`_assertInternal()` — the sole guard on `updateBalance()` and `settlePnl()` — passes for any address in that map: [3](#0-2) 

`updateBalance()` then writes directly to `balances[productId][subaccount]` with no further validation: [4](#0-3) 

By contrast, every other upgradeable contract in the same repo that carries this risk already applies the fix: [5](#0-4) [6](#0-5) [7](#0-6) 

`BaseEngine` and `PerpEngine` are the only upgradeable contracts in scope that omit this guard.

---

### Impact Explanation
An attacker who wins the initialization race (or exploits a forgotten initialization):

1. Becomes `owner` of `PerpEngine` via `transferOwnership(_admin)` where `_admin` is attacker-controlled.
2. Gets three attacker-controlled addresses inserted into `canApplyDeltas`.
3. Can call `updateBalance(productId, victimSubaccount, largeAmount, largeVQuote)` from any of those addresses, writing arbitrary `Balance.amount` and `Balance.vQuoteBalance` for any subaccount.
4. Can call `settlePnl()` to drain `state.availableSettle` for any product.
5. The legitimate clearinghouse and offchain exchange are excluded from `canApplyDeltas`, so all real protocol operations on this engine revert — effectively bricking the PerpEngine for the protocol while the attacker retains full write access.

---

### Likelihood Explanation
The race window exists between proxy deployment and the `initialize()` call whenever those are separate transactions (the common pattern for upgradeable proxy deployments). A mempool-watching attacker can front-run the initialization transaction. The precondition requires no special privilege — only the ability to submit a transaction before the deployer's `initialize()` is mined.

---

### Recommendation
Add a constructor to `BaseEngine` (or `PerpEngineState`) that calls `_disableInitializers()`, matching the pattern already used by `Verifier`, `BaseWithdrawPool`, and `BaseProxyManager`:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

This locks the implementation contract's initializer slot at deployment time, making it impossible for any caller — including a front-runner — to invoke `initialize()` on either the implementation or an uninitialized proxy before the deployer does.

---

### Proof of Concept

```solidity
// Hardhat test — unmodified production contracts
it("attacker hijacks uninitialized PerpEngine proxy", async () => {
    // 1. Deploy proxy (no initialize call yet)
    const PerpEngine = await ethers.getContractFactory("PerpEngine");
    const proxy = await upgrades.deployProxy(PerpEngine, { initializer: false });

    // 2. Attacker calls initialize with their own addresses
    const [, attacker] = await ethers.getSigners();
    const attackerCH = attacker.address;
    await proxy.connect(attacker).initialize(
        attackerCH, attackerCH, ethers.constants.AddressZero, attackerCH, attackerCH
    );

    // 3. Attacker calls updateBalance from attackerCH (canApplyDeltas[attackerCH] == true)
    const victimSubaccount = ethers.utils.formatBytes32String("victim");
    await proxy.connect(attacker).updateBalance(0, victimSubaccount, 1_000_000e18, 1_000_000e18);

    // 4. Assert balance was mutated
    const [, balance] = await proxy.getStateAndBalance(0, victimSubaccount);
    expect(balance.amount).to.equal(1_000_000e18); // arbitrary value written
});
```
