### Title
Difficulty Retargeting Uses Mainchain Block Timestamp Instead of Fork Ancestor's Timestamp — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`)

### Summary

`get_next_work_required` in both `dogecoin.rs` and `bitcoin.rs` fetches the first block of the difficulty period via `get_header_by_height`, which is a **mainchain-only lookup**. When validating a fork block, this returns the mainchain block at that height rather than the fork's actual ancestor. The resulting difficulty calculation uses a stale, wrong timestamp, directly analogous to the external report's `transform()` always setting `timeRemainingSeconds` to the full duration instead of the current remaining value.

---

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes the first block of the retarget window via: [1](#0-0) 

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

The developer's own TODO comment acknowledges the uncertainty. `get_header_by_height` resolves exclusively from `mainchain_height_to_header`: [2](#0-1) 

During fork block validation, the mainchain mapping has **not yet been updated** (the reorg happens only after `submit_block_header_inner` completes). So for a fork that diverges at height H, when validating the fork block at height H+2:

- `prev_block_header` = fork block at H+1
- `height_first` = H
- `get_header_by_height(H)` → **mainchain block at H**, not the fork block at H

The same pattern exists in `contract/src/bitcoin.rs`: [3](#0-2) 

---

### Impact Explanation

The difficulty calculation for Dogecoin (after height 145,000, where `difficulty_adjustment_interval = 1`) computes: [4](#0-3) 

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
```

If the attacker's fork block at height H carries a timestamp `T_fork` that is later than the mainchain block's timestamp `T_main` at the same height, the contract substitutes `T_main` for `T_fork`. This inflates `modulated_timespan` by up to `MAX_FUTURE_BLOCK_TIME_LOCAL` (7200 s), which after Digishield damping and clamping can reduce the required difficulty by up to ~1.5× (from 60 s target to the 90 s maximum). An attacker can therefore submit fork blocks that satisfy a lower proof-of-work threshold than the fork's own timestamps actually require, accumulating chainwork with less real computation and potentially triggering a fraudulent chain reorganization.

For Bitcoin the same root cause applies but requires a fork at least 2016 blocks deep (one full retarget window), making it harder to exploit in practice. [5](#0-4) 

---

### Likelihood Explanation

For Dogecoin (post-height 145,000), the issue manifests at the **third fork block** (H+2). An unprivileged NEAR caller can submit fork blocks via the public `submit_blocks` entry point with no special role required. The attacker only needs to:

1. Submit one fork block at height H with a timestamp up to 2 hours ahead of the mainchain block at H (within `MAX_FUTURE_BLOCK_TIME_LOCAL`).
2. Submit a valid fork block at H+1.
3. Submit fork block H+2 with bits encoding a lower difficulty than the fork's own timestamps require. [6](#0-5) 

---

### Recommendation

Replace the `get_header_by_height` call with an ancestor traversal that walks the fork's own `prev_block_hash` chain back to `height_first`. This mirrors what the reference Dogecoin implementation does and is what the TODO comment already flags as the correct approach. For Bitcoin, apply the same fix to `get_next_work_required` in `bitcoin.rs`.

---

### Proof of Concept

1. Contract is initialized with Dogecoin mainnet at height ≥ 145,001.
2. Mainchain block at height H has timestamp `T_main`.
3. Attacker calls `submit_blocks` with a fork block at height H whose timestamp is `T_main + 7200` (2 hours ahead, within `MAX_FUTURE_BLOCK_TIME_LOCAL`). This passes MTP and future-time checks.
4. Attacker submits a fork block at H+1 with any valid timestamp.
5. Attacker submits a fork block at H+2. The contract calls `get_next_work_required`, which calls `get_header_by_height(H)` and receives the **mainchain** block with timestamp `T_main`. The computed `modulated_timespan` is `fork_H+1_time − T_main`, which is ~7200 s larger than the correct `fork_H+1_time − (T_main + 7200)`. After Digishield clamping this yields the minimum required difficulty (maximum target). The fork block at H+2 is accepted with ~1.5× less proof-of-work than the fork's own timestamps demand.                 .into()

        }
