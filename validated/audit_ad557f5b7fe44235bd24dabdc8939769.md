Audit Report

## Title
`SwapAllowlistExtension` gates the router address instead of the originating user, allowing any caller to bypass the per-user allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every `beforeSwap` hook. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. `SwapAllowlistExtension.beforeSwap` checks only this `sender` argument against the allowlist, so allowlisting the router — a prerequisite for any approved user to use the router — simultaneously opens the gate for every unapproved user who routes through the same contract.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` checks whether that `sender` argument is allowlisted for the calling pool: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly with no mechanism to forward the originating `msg.sender` to the pool: [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

The pool therefore always sees `msg.sender = router`. The check `allowedSwapper[pool][router]` passes for every caller — allowlisted or not — as long as the router is in the allowlist. There is no existing guard that recovers the true originator; the `extensionData` field is passed through unchanged but the extension never reads it for identity purposes.

## Impact Explanation

Any unapproved user can call `MetricOmmSimpleRouter.exactInputSingle` (or the other router entry points) targeting a pool configured with `SwapAllowlistExtension` in its `beforeSwap` order. Provided the router address appears in `allowedSwapper[pool][router]` — a necessary condition for any approved user to use the router — the hook passes unconditionally for the unapproved caller. The unapproved user executes a live swap against LP liquidity in a pool explicitly configured to restrict access, constituting a direct policy bypass. This is an admin-boundary break: the pool admin's access-control configuration is bypassed by an unprivileged path, with direct fund-impacting consequences (LP exposure to counterparties the admin intended to exclude).

## Likelihood Explanation

Both required conditions — deploying `SwapAllowlistExtension` in `beforeSwap` and allowlisting the router — are normal operational choices for any pool that wants to restrict swaps to approved users while still supporting the standard periphery router. No privileged attacker capability, malicious setup, or non-standard token is required. Any EOA can call the public router functions. The scenario is reachable on any production pool meeting these two conditions.

## Recommendation

The `sender` argument passed to `beforeSwap` must represent the economic actor, not the immediate caller. Two complementary fixes:

1. **In the router**: store the originating `msg.sender` in transient storage and expose it via a standard interface (e.g., `ISwapOriginator`) so the extension can read the true initiator during the hook call.
2. **In `SwapAllowlistExtension`**: when the immediate `sender` is a known router, read the originating user from the router's transient storage (or from `extensionData` if the router encodes it there) and check that address against `allowedSwapper` instead.

Alternatively, document explicitly that `SwapAllowlistExtension` gates the immediate pool caller (the router) and that pool admins must not allowlist the router if per-user restrictions are intended — though this effectively makes the extension incompatible with router-mediated swaps for restricted pools.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension in beforeSwap order
  pool admin: setAllowedToSwap(pool, alice, true)
  pool admin: setAllowedToSwap(pool, router, true)  // required for alice to use the router

Attack:
  bob (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      recipient: bob,
      zeroForOne: true,
      amountIn: X,
      ...
    })

Execution trace:
  router.exactInputSingle()
    → pool.swap(recipient=bob, ...)        // msg.sender = router
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (passes)
      → swap executes, bob receives tokens

Result: bob swaps successfully in a pool that was supposed to block him.
```

A Foundry integration test can confirm this by deploying a pool with `SwapAllowlistExtension`, allowlisting only `alice` and the router, then calling `exactInputSingle` from `bob` and asserting the swap succeeds.

  function exactInput(ExactInputParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
      if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

      amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    }

    if (amount <= 0) revert InvalidSwapDeltas();
    amountOut = MetricOmmSwapInputs.int128ToUint128(amount);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
