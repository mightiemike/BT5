### Title
`deser_auto` silently ignores `max_atom_len` and `strict` parameters for classic/backrefs-format blobs — (File: `wheel/src/api.rs`)

### Summary

The Python-exposed `deser_auto` function accepts `max_atom_len` and `strict` parameters but only forwards them to the underlying decoder when the input blob is in 2026 format. For classic or backrefs-format blobs, both parameters are silently dropped and the raw `node_from_bytes_backrefs` path is taken with no atom-length cap and no strictness enforcement. This is a direct structural analog to the Hats `balanceOf`/`balanceOfBatch` divergence: one entry point applies the guard, the sibling entry point bypasses it entirely.

### Finding Description

In `wheel/src/api.rs` lines 147–157, `deser_auto` is declared with two keyword-only parameters:

```rust
#[pyfunction]
#[pyo3(signature = (blob, *, max_atom_len=PY_DEFAULT_MAX_ATOM_LEN, strict=true))]
fn deser_auto(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
        deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)
            .map_err(eval_to_py)?
    } else {
        node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?
        // ^^^ max_atom_len and strict are silently discarded here
    };
    Ok(LazyNode::new(Rc::new(a), node))
}
``` [1](#0-0) 

When the blob carries the 2026 magic prefix, `deserialize_2026_body_from_stream` receives both `max_atom_len` and `strict`. When the blob is in classic or backrefs format, `node_from_bytes_backrefs` is called with neither. The `Allocator` is created with `Allocator::new()` (unbounded heap), so there is no fallback cap. [2](#0-1) 

By contrast, `deser_2026` — the single-format counterpart — always enforces both parameters:

```rust

    let mut a = Allocator::new();
    let node = deserialize_2026(&mut a, blob, max_atom_len, strict)...
``` [3](#0-2) 

The default `PY_DEFAULT_MAX_ATOM_LEN` is 1 MiB: [4](#0-3) 

### Impact Explanation

**`max_atom_len` bypass — memory exhaustion**: A caller that passes `deser_auto(blob, max_atom_len=4096)` to bound memory use receives no protection when `blob` is classic-format. An attacker-supplied classic blob encoding a single 512 MiB atom is accepted without error, allocating that memory in the unbounded `Allocator`. Any service that processes untrusted CLVM bytes through `deser_auto` and relies on `max_atom_len` as its OOM guard is fully exposed.

**`strict` bypass — non-canonical data accepted**: The `strict` flag controls enforcement of canonical encoding rules (e.g., no leading zero bytes on integers). A caller passing `strict=True` to `deser_auto` expects non-canonical atoms to be rejected regardless of format. A classic-format blob containing non-canonical integers bypasses this check entirely. If the resulting `LazyNode` is subsequently hashed, re-serialized, or compared against a canonically-encoded counterpart, the results diverge — a direct consensus or integrity hazard for any downstream code that assumes `deser_auto(..., strict=True)` guarantees canonical form.

### Likelihood Explanation

The attacker-controlled entry path is straightforward: supply a classic-format CLVM blob (no magic prefix required) to any Python caller that uses `deser_auto`. Classic format is the default and most widely used CLVM serialization. No privileges are required. The function is exported as a public Python API (`m.add_function(wrap_pyfunction!(deser_auto, m)?)`), making it reachable from any Python consumer of the wheel. [5](#0-4) 

### Recommendation

Forward `max_atom_len` and `strict` to the classic/backrefs path. Since `node_from_bytes_backrefs` does not currently accept these parameters, either:

1. Add `max_atom_len` and `strict` parameters to `node_from_bytes_backrefs` (and `node_from_bytes`) and thread them through, or
2. Apply a post-parse atom-length walk before returning the `LazyNode` from the classic path, or
3. Document explicitly — with a deprecation warning at call time — that `max_atom_len` and `strict` are **ignored** for non-2026 blobs, so callers are not silently misled.

Option 1 is the correct fix and mirrors the fix suggested in the Hats report (replace the raw storage read with the guarded single-item call).

### Proof of Concept

```python
from clvm_rs import deser_auto

# Build a classic-format CLVM blob encoding a single 2 MiB atom.
# Classic encoding: length prefix 0xfe <2-byte big-endian length> <data>
size = 2 * 1024 * 1024          # 2 MiB
header = bytes([0xFE, (size >> 8) & 0xFF, size & 0xFF])
blob = header + b'\xAB' * size  # no 2026 magic prefix → classic path

# Caller expects max_atom_len=1024 to reject this blob.
node = deser_auto(blob, max_atom_len=1024, strict=True)
# No exception is raised. The 2 MiB atom is loaded into the allocator.
# max_atom_len and strict were silently ignored.
print(node.atom[:8])  # b'\xab\xab\xab\xab\xab\xab\xab\xab'
```

The same blob passed to `deser_2026` (after prepending the magic prefix) would raise a `ValueError` because `max_atom_len=1024` would be enforced. The two sibling functions return inconsistent results for equivalent data, exactly mirroring the `balanceOf`/`balanceOfBatch` divergence in the reference report.
