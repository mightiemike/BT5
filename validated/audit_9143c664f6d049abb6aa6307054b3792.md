Audit Report

## Title
Swap Allowlist Guard Checks Router Address Instead of End User, Enabling Full Bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router address, not the end user. Any non-allowlisted user can bypass a curated pool's swap allowlist by routing through the router, completely neutralizing the access-control guard.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` verbatim to every configured extension via `_callExtensionsInOrder`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly, making the router the `msg.sender` to the pool: [4](#0-3) 

At that point, `sender` forwarded to the extension is the router address. The extension evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][end_user]`. The same structural flaw exists in `exactInput` (multi-hop) and `exactOutputSingle`/`exactOutput`, all of which call `pool.swap()` with `msg.sender = router`: [5](#0-4) 

This creates an inescapable dilemma for the pool admin:
- **Router not allowlisted**: every router-mediated swap reverts for all users, breaking normal UX.
- **Router allowlisted**: the check degenerates to `allowedSwapper[pool][router] == true`, so every user — including those explicitly blocked — can swap by routing through the router.

There are no existing guards in the extension or the pool that recover the original initiator's address. The `extensionData` field is passed through but the extension does not decode it, and the router passes the user-supplied `params.extensionData` directly without injecting the real caller.

## Impact Explanation

A pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict trading to KYC'd or whitelisted counterparties. Any non-allowlisted user calls `router.exactInputSingle()` targeting the curated pool. Because the router must be allowlisted for any router-mediated swap to work for legitimate users, the extension passes for all callers, and the non-allowlisted user executes a full swap against the pool. The allowlist guard is completely neutralized — this is a direct, unprivileged bypass of a configured access-control guard, with fund-impacting consequences: unauthorized trading on a curated pool, potential regulatory or LP-policy violations, and loss of the protection the pool admin configured.

## Likelihood Explanation

Likelihood is **High**. `MetricOmmSimpleRouter` is the standard periphery swap entrypoint. Any user who knows the pool address can call it without any special privilege, key, or token. No setup beyond a normal swap call is required. The bypass is repeatable and unconditional whenever the router is allowlisted.

## Recommendation

The extension must gate the economically relevant actor — the end user — not the intermediate contract. The cleanest fix is for the router to encode `msg.sender` (the end user) into `extensionData` for allowlist-aware pools, and for the extension to decode and check that address when present, falling back to `sender` for direct pool calls. Concretely:

1. In `MetricOmmSimpleRouter`, prepend `abi.encode(msg.sender)` to `params.extensionData` before passing it to `pool.swap()`.
2. In `SwapAllowlistExtension.beforeSwap`, decode the first 32 bytes of `extensionData` as an address (if non-empty) and use that as the effective swapper; otherwise fall back to `sender`.

This keeps the check inside the extension without changing the core pool interface and does not require `tx.origin`.
