Looking at the exact code path in question:

### Title
`vm_resources_to_execution_resources` hardcodes `gas_consumed = default()`, zeroing per-call Sierra gas in every `FunctionInvocation` simulation trace — (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`FunctionInvocation::try_from` converts a blockifier `CallInfo` into the RPC trace object. It calls `vm_resources_to_execution_resources` but never passes `call_info.execution.gas_consumed` (the actual Sierra gas consumed by the call). The function unconditionally writes `gas_consumed: StarknetApiGasVector::default()` — all zeros — into every `FunctionInvocation.execution_resources` regardless of actual Sierra gas used. Any caller of `starknet_simulateTransactions` (or `starknet_traceTransaction`) against a Sierra ≥ 1.7 contract receives a trace where every per-call `gas_consumed` is `{l1_gas: 0, l2_gas: 0, l1_data_gas: 0}`.

### Finding Description

**Where the Sierra gas is stored in blockifier:**

`CallExecution.gas_consumed` (a `u64`) is set to the actual Sierra gas consumed per call in SierraGas mode:

- CASM path: `gas_consumed = initial_gas - remaining_gas` when `tracked_resource == SierraGas` [1](#0-0) 
- Native path: `gas_consumed = initial_gas - remaining_gas` [2](#0-1) 

Concrete values from tests confirm this is nonzero: e.g., `413070` for a Cairo1 fee-transfer call, `11690` for a typical inner call. [3](#0-2) [4](#0-3) 

**Where the conversion drops it:**

`FunctionInvocation::try_from` at line 352–355 calls:

```rust
execution_resources: vm_resources_to_execution_resources(
    call_info.resources.vm_resources,  // VM resources only
    gas_vector,                         // transaction-level DA gas
)?,
```

`call_info.execution.gas_consumed` — the per-call Sierra gas — is **never passed** to `vm_resources_to_execution_resources`. [5](#0-4) 

**The function hardcodes zero:**

```rust
Ok(ExecutionResources {
    steps: ...,
    builtin_instance_counter,
    memory_holes: ...,
    da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
    gas_consumed: StarknetApiGasVector::default(),   // ← always {0, 0, 0}
})
``` [6](#0-5) 

**The same wrong `gas_vector` (transaction DA gas) is propagated to all inner calls:**

```rust
calls: call_info.inner_calls.into_iter()
    .map(|call_info| (call_info, gas_vector))  // same DA gas for every inner call
    .map(Self::try_from)
    ...
``` [7](#0-6) 

This means `da_gas_consumed` is also wrong (it copies the transaction-level DA gas into every inner call), but the `gas_consumed` zeroing is the primary issue.

**The callers all pass `receipt.da_gas` as the gas vector:** [8](#0-7) [9](#0-8) [10](#0-9) 

### Impact Explanation

Every `FunctionInvocation.execution_resources.gas_consumed` in the simulation/trace response is `{l1_gas: 0, l2_gas: 0, l1_data_gas: 0}` for all Sierra ≥ 1.7 contracts, regardless of actual gas consumed. The RPC response is authoritative-looking but carries a concretely wrong value. Tooling, wallets, and developers relying on per-call gas breakdown from `starknet_simulateTransactions` or `starknet_traceTransaction` receive systematically incorrect data.

This fits: **High — RPC simulation/tracing returns an authoritative-looking wrong value.**

### Likelihood Explanation

- Reachable by any unprivileged caller of the public `starknet_simulateTransactions` or `starknet_traceTransaction` RPC endpoints.
- Triggered by any Sierra ≥ 1.7 contract (the default for all modern Cairo 1 contracts under `min_sierra_version_for_sierra_gas = 1.7.0`).
- The bug is unconditional — there is no code path that sets `gas_consumed` to anything other than `default()`.

### Recommendation

Pass `call_info.execution.gas_consumed` into `vm_resources_to_execution_resources` and use it to populate `gas_consumed.l2_gas` when `call_info.tracked_resource == TrackedResource::SierraGas`. The signature should become:

```rust
fn vm_resources_to_execution_resources(
    vm_resources: VmExecutionResources,
    da_gas: GasVector,
    sierra_gas_consumed: u64,
    tracked_resource: TrackedResource,
) -> ExecutionResult<ExecutionResources>
```

and the output field should be:

```rust
gas_consumed: match tracked_resource {
    TrackedResource::SierraGas => StarknetApiGasVector::from_l2_gas(GasAmount(sierra_gas_consumed)),
    TrackedResource::CairoSteps => StarknetApiGasVector::default(),
},
```

The inner-call recursion at line 332 should also pass each child's own `call_info.execution.gas_consumed` rather than the parent's `gas_vector`.

### Proof of Concept

1. Deploy a Cairo 1 contract compiled with Sierra ≥ 1.7.0 (any modern contract).
2. Call `starknet_simulateTransactions` with a transaction invoking that contract.
3. Inspect any `FunctionInvocation.execution_resources.gas_consumed` in the returned trace.
4. Observe `{l1_gas: 0, l2_gas: 0, l1_data_gas: 0}` for every call node.
5. Compare against `call_info.execution.gas_consumed` logged from blockifier (e.g., `413070` for a fee-transfer call, `11690` for a typical inner call) — the divergence is concrete and universal.
