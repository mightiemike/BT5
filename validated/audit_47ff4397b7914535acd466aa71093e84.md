### Title
`starknet_estimateFee` / `starknet_simulateTransactions` `FeeEstimation` omits `l2_gas_consumed` while `overall_fee` includes L2 gas cost — (`File: crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`tx_execution_output_to_fee_estimation` populates `overall_fee` from `receipt.fee`, which includes L2 gas costs, but the breakdown fields only expose `gas_consumed` (L1 gas) and `data_gas_consumed` (L1 data gas). The `l2_gas_consumed` field is entirely absent from the response. The OpenRPC spec and the Rust doc-comment both describe `overall_fee` as equalling `gas_consumed * gas_price + data_gas_consumed * data_gas_price`, which is false for any V3 transaction that consumes L2 gas. This is the sequencer-native analog of the ERC-4626 `maxWithdraw()` bug: a "view" function returns an authoritative-looking value that is inconsistent with the actual constrained resource, causing callers that rely on the breakdown to construct or verify transactions to fail.

### Finding Description

In `crates/apollo_rpc_execution/src/objects.rs`, `tx_execution_output_to_fee_estimation` builds the `FeeEstimation` struct as follows:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),       // L1 gas only
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // L1 data gas only
    l1_data_gas_price,
    l2_gas_price,                                   // price present …
    overall_fee: tx_execution_output.execution_info.receipt.fee, // … but l2_gas_consumed absent
    unit: tx_execution_output.price_unit,
})
``` [1](#0-0) 

The `receipt.fee` is computed by `get_fee_by_gas_vector`, which sums all three resources:

```
fee = l1_gas * l1_gas_price
    + l1_data_gas * l1_data_gas_price
    + l2_gas * (l2_gas_price + tip)
``` [2](#0-1) 

The `FeeEstimation` struct's own doc-comment and the OpenRPC spec both state that `overall_fee` equals `gas_consumed * gas_price + data_gas_consumed * data_gas_price`:

```rust
/// The total amount of fee. This is equal to:
/// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
pub overall_fee: Fee,
``` [3](#0-2) 

```json
"overall_fee": {
    "description": "The estimated fee for the transaction … equals to gas_consumed*gas_price + data_gas_consumed*data_gas_price",
``` [4](#0-3) 

The team is aware of the missing field but has not resolved it:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
``` [5](#0-4) 

**Concrete divergence from test data** (`test_simulate_and_get_initial_reads.json`):

| Field | Value |
|---|---|
| `l1_gas_consumed` | `0x0` |
| `l1_data_gas_consumed` | `0x80` (128) |
| `l2_gas_consumed` | `0xb56b6` (743,094) — present in the prover's RPC record but absent from the sequencer's `FeeEstimation` |
| `l2_gas_price` | `0x1dcd65000` (8,000,000,000) |
| `overall_fee` | `0x151eb86f3ed400` (5,944,752,000,000,000 FRI) | [6](#0-5) 

A caller computing `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` from the sequencer's response gets `0 + 128 * 1000 = 128,000 FRI` — roughly **46 million times smaller** than the returned `overall_fee`. The L2 gas component (`743,094 * 8,000,000,000 ≈ 5.94 × 10¹⁵ FRI`) is entirely invisible in the breakdown.

### Impact Explanation

This matches the **High** impact category: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

1. **Incorrect fee reconstruction**: Any caller that verifies `overall_fee` by summing the breakdown fields will observe a massive discrepancy, making the response appear fraudulent or buggy.
2. **Broken resource-bound construction**: A caller that uses `starknet_estimateFee` to derive the `l2_gas.max_amount` for a subsequent V3 transaction has no basis for setting it (the field is absent). Setting it to zero causes the transaction to fail with out-of-gas during execution — the exact `maxWithdraw()` analog: the estimation says the transaction is executable, but execution fails because the constrained resource (`l2_gas`) was not reported.
3. **Wrong spec contract**: The OpenRPC `FEE_ESTIMATE` schema and the Rust doc-comment both assert an equality that is false for all V3 transactions with non-zero L2 gas, which is the dominant transaction type in the current protocol.

### Likelihood Explanation

Every V3 (`AllResources`) transaction consumes L2 gas. The `starknet_estimateFee` and `starknet_simulateTransactions` endpoints are the primary interfaces used by wallets, SDKs, and dApps to determine fees and construct resource bounds. The discrepancy is present on every such call and is not gated by any version flag or configuration.

### Recommendation

In `tx_execution_output_to_fee_estimation`, add `l2_gas_consumed: gas_vector.l2_gas.0.into()` to the `FeeEstimation` struct and expose it in the response. Update the OpenRPC schema to include `l2_gas_consumed` and `l2_gas_price` as required fields, and correct the `overall_fee` description to reflect that it equals `gas_consumed*l1_gas_price + data_gas_consumed*l1_data_gas_price + l2_gas_consumed*l2_gas_price`.

### Proof of Concept

1. Submit a V3 invoke transaction to `starknet_estimateFee`.
2. Observe the response: `gas_consumed = X`, `data_gas_consumed = Y`, `l2_gas_price = P`, `overall_fee = F`.
3. Compute `X * l1_gas_price + Y * l1_data_gas_price`. For any transaction with non-trivial L2 execution, this value is far smaller than `F`.
4. Attempt to construct a V3 transaction using `l2_gas.max_amount = 0` (the only value derivable from the response): the transaction fails with `Out of gas` during execution, despite the fee estimation having succeeded.
5. The prover's own RPC records (`test_simulate_and_get_initial_reads.json`) confirm that `l2_gas_consumed = 0xb56b6` is the dominant cost component, yet it is absent from the sequencer's `FeeEstimation` output. [7](#0-6) [1](#0-0)

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
