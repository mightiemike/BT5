### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Free usdcE Drain - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` directly without checking its return value. The rest of the codebase uses `ERC20Helper.safeTransferFrom` for this purpose. If the USDC token on the target chain returns `false` on a failed transfer rather than reverting, the function continues execution and sends usdcE to the caller without receiving any USDC in return.

---

### Finding Description

`replaceUsdcEWithUsdc` is an unguarded public function (no `onlyOwner` or `onlyDeployer` modifier) that is callable by any address on chain 57073 (Ink). Its logic is:

1. Compute `balance` = usdcE held by `directDepositV1`.
2. Pull `balance` USDC from `msg.sender` into `directDepositV1` via raw `transferFrom`.
3. Withdraw all usdcE from `directDepositV1` to `ContractOwner` via `DirectDepositV1.withdraw`.
4. Send that usdcE to `msg.sender` via `safeTransfer`.

Step 2 uses the raw `IERC20Base.transferFrom` call and **does not check the boolean return value**:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [1](#0-0) 

The contract declares `using ERC20Helper for IERC20Base` and uses `safeTransfer` on lines 618 and 642, but this single `transferFrom` call bypasses the safe wrapper entirely. [2](#0-1) 

`ERC20Helper.safeTransferFrom` performs a low-level `.call` and requires both `success == true` and a truthy decoded return value, which is the correct pattern for non-standard tokens: [3](#0-2) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink (chain 57073) returns `false` on a failed transfer (e.g., caller has zero allowance or zero balance) rather than reverting, the function silently skips the payment and still:

- Drains all usdcE from the `directDepositV1` contract (step 3).
- Transfers that usdcE to the attacker (step 4).

The attacker receives real usdcE tokens at zero cost, directly stealing assets held in `directDepositV1` contracts.

---

### Likelihood Explanation

- The function has **no access control** — any EOA or contract on Ink can call it.
- The only gate is `block.chainid == 57073`, which is trivially satisfied by any on-chain caller.
- Bridged or wrapped USDC variants on newer chains (like Ink) may not follow the standard revert-on-failure behavior.
- The attacker only needs to call the function with a `subaccount` whose `directDepositV1` holds a non-zero usdcE balance. [4](#0-3) 

---

### Recommendation

Replace the raw `transferFrom` call with `ERC20Helper.safeTransferFrom`, consistent with the rest of the codebase:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [3](#0-2) 

---

### Proof of Concept

1. A `directDepositV1` for some `subaccount` holds 1000 usdcE.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, 1000)` returns `false` (no revert).
4. Execution continues: `DirectDepositV1(directDepositV1).withdraw(usdcE)` pulls 1000 usdcE into `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, 1000)` sends 1000 usdcE to the attacker.
6. Attacker has gained 1000 usdcE; the `directDepositV1` vault is drained; no USDC was paid. [5](#0-4)

        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
