Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Ultimate User, Allowing Any Router Caller to Bypass Per-User Swap Restrictions — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps on `allowedSwapper[pool][sender]`, where `sender` is the direct `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates, `sender` is the router address, not the end user. A pool admin who allowlists the router so that approved users can trade via it simultaneously opens the pool to every caller of the router, collapsing the per-user allowlist to a per-router allowlist.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs the check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the contract calling the extension) and `sender` is the first argument forwarded from the pool. `MetricOmmPool.swap` always passes its own `msg.sender` as that argument:

```solidity
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    recipient,
    ...
)
``` [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as `sender` to the extension: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router `msg.sender` inside the pool: [4](#0-3) 

The router passes the user-supplied `params.extensionData` unmodified and has no mechanism to encode the original caller identity. The extension therefore evaluates `allowedSwapper[pool][router]`. Once the router is allowlisted, this check passes for every caller of the router regardless of individual approval status.

## Impact Explanation
LP providers on KYC-gated or institutionally curated pools are exposed to swaps from counterparties who were never individually approved. The `SwapAllowlistExtension`'s core access-control invariant — that only explicitly approved addresses may swap — is broken for any pool whose admin allowlists the router. This constitutes broken core pool functionality (the allowlist hook) and an admin-boundary break where an unprivileged path (calling the router) bypasses a required per-user gate.

## Likelihood Explanation
The router is the standard periphery entry point for end users. Any pool admin who wants approved users to trade via the router must add the router to the allowlist — there is no finer-grained mechanism (`setAllowedToSwap` is binary per address). The misconfiguration is therefore a predictable and near-inevitable operational outcome for any pool using `SwapAllowlistExtension` with router-based access. [5](#0-4) 

## Recommendation
The extension must verify the ultimate user identity rather than the direct pool caller. The most robust fix is to have the router encode the original `msg.sender` into `extensionData` using a trusted convention, and have the extension decode and check that value when the caller is a known router. Alternatively, document that pools using `SwapAllowlistExtension` must not allowlist the router address and that allowlisted users must call the pool directly — but this eliminates router usability for curated pools.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls setAllowedToSwap(pool, userA, true).
3. Pool admin calls setAllowedToSwap(pool, router, true)
   — required so userA can trade via the router.
4. Non-allowlisted userC calls:
     router.exactInputSingle({pool: pool, tokenIn: ..., tokenOut: ..., amountIn: ..., recipient: userC, ...})
   The router calls pool.swap(recipient=userC, ...) with msg.sender=router.
   Pool passes msg.sender (router) as sender to _beforeSwap → extension.
   Extension checks: allowedSwapper[pool][router] == true → passes.
5. userC's swap executes on the curated pool despite never being individually approved.
```
