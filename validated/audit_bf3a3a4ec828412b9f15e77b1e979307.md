### Title
`delistProduct` Gated by `onlyDeployer` Instead of `onlyOwner`, Allowing Hot-Wallet Deployer to Force-Settle All Positions — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.delistProduct()` is protected by `onlyDeployer` rather than `onlyOwner`. The `deployer` is a hot-wallet address fixed at initialization, while `owner` is a multisig. Every other critical administrative action in `ContractOwner` that submits a slow-mode transaction is gated by `onlyOwner`. The inconsistency lets the deployer unilaterally force-settle any product at an arbitrary price, bypassing multisig governance.

---

### Finding Description

In `ContractOwner.initialize()`, ownership is immediately transferred to a multisig, while `deployer` is stored as the raw deployment EOA: [1](#0-0) 

The `onlyDeployer` modifier enforces only that `msg.sender == deployer`: [2](#0-1) 

`delistProduct` is the only slow-mode-submitting function in `ContractOwner` that uses this weaker guard: [3](#0-2) 

Every other slow-mode-submitting function — `withdrawInsurance`, `depositInsurance`, `updateBuilders`, `addNlpPool`, `rebalanceXWithdraw`, `updateTierFeeRates`, etc. — is `onlyOwner`: [4](#0-3) 

`EndpointTx.submitSlowModeTransactionImpl` enforces that the caller of `submitSlowModeTransaction` must be `owner()` for `DelistProduct` transactions: [5](#0-4) 

Because `ContractOwner` is the registered owner of `Endpoint` (evidenced by all other `onlyOwner` slow-mode paths routing through it), `ContractOwner` satisfies that check. The deployer therefore has a complete, unobstructed path to submit a `DelistProduct` slow-mode transaction with attacker-controlled `pricesX18`.

---

### Impact Explanation

`DelistProduct` forces settlement of every open position in the targeted product at the caller-supplied `pricesX18`. A deployer who submits this with manipulated prices can:

- Settle long positions at near-zero prices, wiping out user collateral.
- Settle short positions at inflated prices, draining the insurance fund and socializing losses across all remaining subaccounts.

The corrupted state delta is: `spotEngine` and `perpEngine` balances for every subaccount holding the delisted product, plus the `insurance` reserve in `ClearinghouseStorage`. These are permanent on-chain mutations with no recovery path.

---

### Likelihood Explanation

The `deployer` is a hot-wallet EOA (lower operational security than a multisig). It is a fixed, non-rotatable address. A compromised or malicious deployer can execute this attack at any time without any additional on-chain precondition — no governance vote, no timelock, no co-signer. The slow-mode delay is the only friction, and the deployer controls when to trigger it.

---

### Recommendation

Change the access control on `delistProduct` from `onlyDeployer` to `onlyOwner`, consistent with every other critical slow-mode-submitting function in `ContractOwner`:

```solidity
// Before
function delistProduct(...) external onlyDeployer { ... }

// After
function delistProduct(...) external onlyOwner { ... }
``` [6](#0-5) 

---

### Proof of Concept

1. At deployment, `deployer` is set to the hot-wallet EOA; `owner` is transferred to the multisig. [1](#0-0) 
2. Deployer EOA calls `ContractOwner.delistProduct([productId], [1], [])` — passes `onlyDeployer`. [3](#0-2) 
3. `_submitSlowModeTransaction` calls `endpoint.submitSlowModeTransaction(...)` with `msg.sender = address(ContractOwner)`. [7](#0-6) 
4. `EndpointTx.submitSlowModeTransactionImpl` checks `sender == owner()` — passes because `ContractOwner` is `Endpoint`'s owner. [5](#0-4) 
5. After the slow-mode delay, the sequencer (or anyone) executes the queued transaction; `clearinghouse.delistProduct(transaction)` settles all positions at price `1`, zeroing user balances. [8](#0-7)
