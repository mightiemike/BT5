Audit Report

## Title
`allowPushers` consent signature replayable after `revokePusher` within deadline window — (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

## Summary
`CompressedOracleV1.allowPushers` verifies a pusher's EIP-191 consent signature but neither tracks consumed signatures nor includes a per-pusher nonce. The sole replay barrier is the deadline check in `_ensureDeadline`. After a pusher self-revokes via `revokePusher`, the namespace creator can call `allowPushers` again with the identical `(deadline, pushers, signatures)` arguments — provided the deadline has not expired — silently re-establishing the delegation and redirecting all subsequent pushes to the creator's namespace without the pusher's ongoing consent.

## Finding Description
`allowPushers` constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

There is no nonce, no per-pusher sequence counter, and no `usedSignatures` mapping. The only protection against replay is `_ensureDeadline`:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [2](#0-1) 

`revokePusher` sets `namespaceRemapping[msg.sender] = address(0)`: [3](#0-2) 

After revocation, the creator still holds the original signature bytes. If `block.timestamp <= deadline`, calling `allowPushers` again with the same arguments passes both the `_ensureDeadline` check and the ECDSA recovery check, and writes `namespaceRemapping[pusher] = msg.sender` again — exactly reversing the revocation. The code's own comment acknowledges the deadline is the sole replay barrier but only contemplates undated signatures, not a dated signature with a future deadline being replayed post-revocation: [4](#0-3) 

## Impact Explanation
After re-delegation, the `fallback` push path routes the pusher's writes to the creator's namespace: [5](#0-4) 

The pusher's own pool then reads a feed with `timestamp = 0` (never updated in the pusher's own namespace), which every consumer rejects as stale — causing bad-price execution or a complete halt of swaps on that pool. Simultaneously, the creator's pool receives oracle data it was not supposed to receive after the pusher's revocation, constituting oracle data injection into a live pool's price feed without the pusher's ongoing consent. This meets the bad-price execution and broken core pool functionality impact criteria.

## Likelihood Explanation
The creator retains the pusher's consent signature after the initial `allowPushers` call. Deadlines are set by the caller and can be arbitrarily far in the future. The replay requires only the creator's EOA and the stored signature bytes — no privileged keys. The pusher has no on-chain mechanism to invalidate the old signature short of waiting for the deadline to expire. Any creator who wishes to prevent a pusher from operating independently after revocation can replay the signature at any time before the deadline.

## Recommendation
Add a per-pusher nonce to the signed message and increment it on each successful delegation:

```solidity
mapping(address pusher => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]++
    ))
);
```

Alternatively, maintain a `mapping(bytes32 => bool) usedSignatures` and mark each digest as consumed after first use, preventing any signature from being accepted more than once regardless of deadline.

## Proof of Concept
```solidity
function test_allowPushers_replayAfterRevoke() public {
    uint256 deadline = block.timestamp + 365 days;

    // 1. Pusher signs consent for creator
    bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
    );
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
    bytes memory sig = abi.encodePacked(r, s, v);

    address[] memory pushers_ = new address[](1);
    pushers_[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // 2. Creator delegates pusher
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers_, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // 3. Pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // 4. Creator replays the SAME signature — revocation silently undone
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers_, sigs); // succeeds, no revert
    assertEq(oracle.namespaceRemapping(pusher), creator); // delegation restored

    // 5. Pusher's pushes land in creator's namespace; pusher's own pool reads stale data
}
```
