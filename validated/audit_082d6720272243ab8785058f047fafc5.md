### Title
Silent Bit-Shift Truncation in `updateTierFeeRates` Causes Permanent Fee Miscalculation for Tiers ≥ 128 — (`core/contracts/OffchainExchange.sol`)

### Summary

`nonDefaultFeeTierMask` is a `uint128`. The expression `uint128(1) << txn.tier` silently evaluates to `0` whenever `txn.tier >= 128`, because the EVM performs the shift in 256-bit space and the result (`2^tier`) is then truncated back to 128 bits. The mask is never updated, so `getTierFeeRateX18` always falls through to the hardcoded 2 bps default for any tier ≥ 128, even though the fee rates themselves are correctly written into `feeRates[txn.tier][productId]`.

### Finding Description

**`nonDefaultFeeTierMask` declaration** — `uint128 internal nonDefaultFeeTierMask;` [1](#0-0) 

**Mask write in `updateTierFeeRates`:**
```solidity
nonDefaultFeeTierMask |= uint128(1) << txn.tier;
```
`txn.tier` is `uint32`. When `txn.tier >= 128`, `uint128(1) << txn.tier` computes `2^tier` in 256-bit EVM arithmetic, which exceeds `2^128 - 1` and is truncated to `0` on assignment back to `uint128`. The `|= 0` is a no-op; the mask bit is never set. No revert occurs — Solidity 0.8.x does not apply overflow checks to shift operators. [2](#0-1) 

**Mask read in `getTierFeeRateX18`:**
```solidity
if (nonDefaultFeeTierMask & (1 << tier) != 0) {
    return feeRates[tier][productId];
}
return FeeRates({ makerRateX18: 0, takerRateX18: 200_000_000_000_000 }); // 2 bps
```
For `tier >= 128`, `(1 << tier)` is again `0` (same truncation), so the condition is always `false`. The stored `feeRates[tier][productId]` is unreachable; the default 2 bps is always returned. [3](#0-2) 

The fee rates are correctly written to storage (lines 973–988) but are permanently inaccessible for any tier ≥ 128. [4](#0-3) 

The fee rate is consumed in `getUserFeeRateWithBuilder`, which calls `getTierFeeRateX18(feeTier, productId)` and feeds the result directly into `applyFee`. [5](#0-4) 

### Impact Explanation

Any trader whose `feeTiers[address]` is set to a value ≥ 128 will always be charged the default 2 bps taker rate and 0 maker rate, regardless of the configured rate. Concretely:

- A VIP tier configured with a negative maker rebate (e.g., −5 bps) will never receive that rebate; the protocol silently overcharges them.
- A high-volume tier configured with a higher taker rate will be undercharged; the protocol loses fee revenue.
- `collectedFees` in `marketInfo[productId]` accumulates the wrong amounts, permanently desynchronizing the fee accounting from the intended fee obligations.

This is a direct, concrete desynchronization of `collectedFees` state from actual fee obligations across `OffchainExchange`, `SpotEngine`, and `PerpEngine` (which receive the accumulated `collectedFees` on settlement). [6](#0-5) 

### Likelihood Explanation

The entry point is `ContractOwner.updateTierFeeRates`, which is `onlyOwner` and submits a slow-mode transaction. The `tier` field is `uint32` with no upper-bound validation, so values ≥ 128 are accepted without error. [7](#0-6) 

The protocol operator acting in good faith — expanding fee tiers beyond 127 as the user base grows — would trigger this silently. There is no on-chain revert, no event anomaly, and no off-chain signal that the mask update failed. The misconfiguration is permanent until the contract is upgraded.

### Recommendation

1. Replace `uint128 nonDefaultFeeTierMask` with `uint256 nonDefaultFeeTierMask` to support up to 256 tiers without truncation.
2. Alternatively, add an explicit upper-bound check in `updateTierFeeRates`: `require(txn.tier < 128, "tier out of range");` (or `< 256` after widening the mask).
3. Add a corresponding check in `updateFeeTier` to prevent assigning users to tiers that cannot be represented in the mask.
