### Title
`DepositAllowlistExtension.beforeAddLiquidity` ignores the actual caller (`sender`) and only gates on `owner`, allowing any unprivileged user to bypass the deposit allowlist - (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` by "depositor address, per pool." However, its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual `msg.sender` of the pool's `addLiquidity` call) and only checks the `owner` (position holder). Because the pool's `addLiquidity` explicitly supports an operator pattern where `msg.sender ≠ owner`, any unprivileged user can bypass the allowlist by supplying any allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct identities to the extension hook:

```solidity
// MetricOmmPool.sol:191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the actual caller (the depositor/payer); `owner` is the position holder. The pool's own NatDoc acknowledges the split:

> "If `DEPOSIT_ALLOWLIST_PROVIDER` is set, `owner` must pass allowlist. `msg.sender` pays but need not equal `owner` (operator pattern)."

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both to the extension:

```solidity
// ExtensionCalling.sol:95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

But `DepositAllowlistExtension.beforeAddLiquidity` silently drops `sender` (first argument is unnamed) and only checks `owner`:

```solidity
// DepositAllowlistExtension.sol:32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`msg.sender` inside the extension is the pool (enforced by `onlyPool`), so the check reduces to: "is `owner` on the allowlist for this pool?" The actual caller who triggers the deposit and pays the tokens is never examined.

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (with explicit `owner`) makes the split trivially reachable:

```solidity
// MetricOmmPoolLiquidityAdder.sol:56-68
function addLiquidityExactShares(
    address pool, address owner, uint80 salt, LiquidityDelta calldata deltas,
    uint256 maxAmountToken0, uint256 maxAmountToken1, bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _validateOwner(owner);   // only checks owner != address(0)
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
```

`_validateOwner` only rejects `address(0)`. There is no check that `owner == msg.sender`.

---

### Impact Explanation

A pool admin deploys a pool with `DepositAllowlistExtension` to restrict deposits to a curated set of addresses. Any unprivileged user (Bob) can:

1. Call `addLiquidityExactShares(pool, alice_allowlisted, salt, deltas, ...)` via the `MetricOmmPoolLiquidityAdder`.
2. The pool calls `_beforeAddLiquidity(sender=LiquidityAdder, owner=alice, ...)`.
3. The extension checks `allowedDepositor[pool][alice]` → passes.
4. Bob's tokens are pulled (Bob is the payer stored in transient context).
5. Alice receives the LP position.

Bob has injected liquidity into a restricted pool without being on the allowlist. The pool admin's access control boundary is fully bypassed. Bob can:
- Deposit into any bin to manipulate the pool's liquidity distribution and affect pricing for legitimate LPs.
- Force liquidity into bins the pool admin intended to keep empty, disrupting the pool's intended depth profile.
- Repeat across many bins, affecting gas costs and pool state for all participants.

This is a direct admin-boundary break: an unprivileged path bypasses a configured pool guard.

---

### Likelihood Explanation

- The bypass requires no special privileges, no malicious token, and no admin cooperation.
- `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` is a public, documented periphery entry point that explicitly supports `owner ≠ msg.sender`.
- Any user who knows an allowlisted address (which may be publicly visible from past transactions or events) can execute the bypass immediately.
- The pool's own test suite (`test_exactShares_canAddOnBehalfOfAnotherOwner`) demonstrates the operator pattern works without restriction, confirming the path is live.

---

### Recommendation

The `beforeAddLiquidity` hook receives `sender` (the actual `msg.sender` of `addLiquidity`) as its first argument. The allowlist check should gate on `sender`, not `owner`, to enforce who is actually depositing:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    // Gate the actual depositor (caller), not just the position owner.
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intended semantics are to gate the position owner (not the caller), the contract's NatDoc and name should be updated to reflect that, and pool admins must be warned that any user can deposit on behalf of an allowlisted owner.

---

### Proof of Concept

```solidity
// Setup: pool with DepositAllowlistExtension; only `alice` is allowlisted.
// Bob is NOT allowlisted.

address alice = makeAddr("alice");
address bob   = makeAddr("bob");

// Pool admin allowlists only alice.
vm.prank(poolAdmin);
depositAllowlist.setAllowedToDeposit(address(pool), alice, true);

// Confirm bob is blocked when he is the owner.
vm.prank(bob);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToDeposit.selector);
pool.addLiquidity(bob, 0, deltas, callbackData, "");

// Bob bypasses the allowlist by using alice as owner via the LiquidityAdder.
// Bob has approved the adder for his tokens.
vm.prank(bob);
// owner = alice (allowlisted), payer = bob (not allowlisted) — passes the hook.
liquidityAdder.addLiquidityExactShares(
    address(pool), alice, 0, deltas, type(uint256).max, type(uint256).max, ""
);

// Alice now holds LP shares funded by Bob's tokens — allowlist bypassed.
uint256 aliceShares = stateView.positionBinShares(address(pool), alice, 0, binIdx);
assertGt(aliceShares, 0); // Bob deposited into a restricted pool without being allowlisted.
```
