### Title
`LIMIT_HEAP` Flag Declared in `MEMPOOL_MODE` but Never Enforced in Execution Engine - (File: src/chia_dialect.rs, src/run_program.rs)

### Summary
`ClvmFlags::LIMIT_HEAP` is defined with an explicit documentation promise ("limits the number of atom-bytes allowed to be allocated, as well as the number of pairs"), included in the security-critical `MEMPOOL_MODE` constant, but is never consulted anywhere in `run_program.rs` or the core execution path. The flag is wired into the dialect but has no enforcement site, creating a false sense of security for any Rust caller relying on `MEMPOOL_MODE` to bound heap usage.

### Finding Description

`ClvmFlags::LIMIT_HEAP` is defined in `src/chia_dialect.rs`:

```rust
/// When set, limits the number of atom-bytes allowed to be allocated,
/// as well as the number of pairs.
const LIMIT_HEAP = 0x0004;
``` [1](#0-0) 

It is unconditionally included in `MEMPOOL_MODE`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [2](#0-1) 

Contrast this with `LIMIT_SOFTFORK`, which is actually checked inside `apply_op` in `run_program.rs`:

```rust
if self.dialect.flags().contains(ClvmFlags::LIMIT_SOFTFORK)
    && self.softfork_stack.len() >= 20
{
    return Err(EvalErr::SoftforkStackDepthExceeded);
}
``` [3](#0-2) 

`LIMIT_HEAP` has no equivalent enforcement block anywhere in `run_program.rs`. The `Allocator` is constructed externally and passed into `run_program` — the execution engine never reads `LIMIT_HEAP` from the dialect flags and never instructs the allocator to enforce atom-byte or pair-count limits based on it. [4](#0-3) 

The other flags in `MEMPOOL_MODE` are all actively enforced:
- `NO_UNKNOWN_OPS` — checked in `unknown_operator()` [5](#0-4) 
- `CANONICAL_INTS` — passed to operator functions via `flags` and checked in `op_utils`
- `DISABLE_OP` — checked in `op_modpow`, `op_div`, `op_divmod`, `op_mod` [6](#0-5) 
- `LIMIT_SOFTFORK` — enforced in `run_program.rs` as shown above

`LIMIT_HEAP` alone has no enforcement site in the core library.

### Impact Explanation

**Impact: Low**

Any Rust caller that constructs `ChiaDialect::new(MEMPOOL_MODE)` and calls `run_program` believes heap allocation is bounded by the flag's documented semantics. It is not. A crafted CLVM program can allocate atom-bytes and pairs without the flag-implied ceiling being applied. The cost accounting still runs, so unbounded allocation is bounded by `max_cost`, but the separate heap-byte and pair-count limits that `LIMIT_HEAP` promises are absent. This is a false security invariant, not a direct exploit path — hence Low impact, matching the original report's classification.

### Likelihood Explanation

**Likelihood: High**

`LIMIT_HEAP` is present in `MEMPOOL_MODE`, which is the standard mode for mempool validation. Every Rust caller using `MEMPOOL_MODE` is affected. The flag's documentation is unambiguous about what it should do. The gap between declaration and enforcement is certain and unconditional — there is no code path that enforces it.

### Recommendation

Either:
1. Remove `LIMIT_HEAP` from `ClvmFlags` and from `MEMPOOL_MODE` if heap limiting is handled externally (e.g., by the caller constructing the `Allocator` with limits), and update documentation accordingly; or
2. Add enforcement inside `run_program` that reads `dialect.flags().contains(ClvmFlags::LIMIT_HEAP)` and, when set, checks the allocator's atom-byte and pair counts against defined ceilings at each allocation step, returning `EvalErr` on violation.

### Proof of Concept

```rust
use clvmr::allocator::Allocator;
use clvmr::chia_dialect::{ChiaDialect, MEMPOOL_MODE};
use clvmr::run_program::run_program;
use clvmr::serde::node_from_bytes;

let mut allocator = Allocator::new(); // no heap limits set
let dialect = ChiaDialect::new(MEMPOOL_MODE); // LIMIT_HEAP is set in flags

// A CLVM program that allocates many large atoms via concat/sha256 chains
// will succeed even though LIMIT_HEAP implies it should be bounded.
// The flag is never read by run_program, so no heap limit is applied.
let program = node_from_bytes(&mut allocator, &[...]).unwrap();
let env = allocator.nil();
let result = run_program(&mut allocator, &dialect, program, env, 10_000_000_000);
// result: Ok(...) — heap grew without any LIMIT_HEAP enforcement
```

The attacker entry point is any caller-supplied CLVM byte sequence evaluated under `MEMPOOL_MODE`. The necessary vulnerable step is `run_program` in `src/run_program.rs`, which never reads `LIMIT_HEAP` from `dialect.flags()`.
