### Title
Dogecoin Fork-Block Retarget Uses Mainchain Ancestor Instead of Actual Fork Ancestor — (`contract/src/dogecoin.rs`)

---

### Summary

`get_next_work_required` resolves the retarget window's first-block timestamp via `get_header_by_height`, which reads from `mainchain_height_to_header`. When the block being validated is a **fork block** whose retarget window spans the fork divergence point, the mainchain block at `height_first` is a **different block** than the fork's actual ancestor at that height. The two blocks carry different timestamps, so `calculate_next_work_required` receives a wrong `first_block_time`, producing an incorrect `expected_bits`. The contract then enforces that wrong target, accepting fork blocks whose actual PoW is insufficient relative to the correct difficulty.

---

### Finding Description

**Entrypoint**: `submit_blocks` (Dogecoin build, `contract/src/lib.rs` line 169) → `submit_block_header` (`contract/src/dogecoin.rs` line 166) → `check_target` → `check_pow` → `get_next_work_required`.

**The flaw** is at `contract/src/dogecoin.rs` lines 291–295:

```rust
// TODO: check if it is correct to get block header by height from mainchain
//       without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

This always returns the **current canonical mainchain** block at `height`, not the block that is an ancestor of the fork block being validated.

**Triggering condition** (Dogecoin post-height 145,000): [3](#0-2) 

`difficulty_adjustment_interval = 1`, so **every block** is a retarget block and `blocks_to_go_back = 1`, meaning `height_first = prev_height − 1`. [4](#0-3) 

**State divergence**: Suppose the mainchain has block `AH` at height `H` (timestamp `T_AH`). The attacker builds a fork that diverges at height `H`, placing fork block `BH` (timestamp `T_BH`) at height `H`. When the attacker submits fork block `B(H+2)`:

- `prev_block_header` = `B(H+1)` at height `H+1`
- `height_first` = `H`
- `get_header_by_height(H)` → returns `AH` (mainchain block), **not** `BH` (fork ancestor)
- `first_block_time` = `T_AH` ≠ `T_BH`

`check_pow` is called **before** `submit_block_header_inner`, so the mainchain still reflects `AH` at height `H` at the time of validation: [5](#0-4) 

The attacker sets `T_BH` to just above the MTP (minimum valid timestamp), while `T_AH` is the real mainchain timestamp (typically much larger). The resulting `modulated_timespan` in `calculate_next_work_required` is computed from `T_AH` instead of `T_BH`: [6](#0-5) 

With Digishield clamping (`min_timespan = retarget_timespan − retarget_timespan/4`), the attacker can shift the computed target up to **25% easier** per block. The attacker predicts the wrong `expected_bits` the contract will compute (since `T_AH` is public on-chain state), sets `block_header.bits` to that value, and mines a block satisfying the easier target. The `require!(expected_bits == block_header.bits, ...)` check passes with the incorrect target. [7](#0-6) 

---

### Impact Explanation

Fork blocks are accepted with `bits` encoding a lower difficulty than the correct retarget would require. Each such block contributes less chainwork (`work_from_bits(block_header.bits)`) than a legitimately-mined block at the correct difficulty. If the fork accumulates enough chainwork to trigger `reorg_chain`, the light client promotes the fork to canonical, and `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` will confirm transactions in the fraudulent fork while rejecting those in the honest chain. This enables double-spend proofs against downstream consumers of the light client.

---

### Likelihood Explanation

The `submit_blocks` entry point is gated by `#[trusted_relayer]`. The `trusted_relayer` macro (from `omni_utils`) implements a staking-based registration system; `manager_roles(Role::DAO, Role::RelayerManager)` can reject relayers but registration itself is permissionless via staking. A well-resourced attacker can register as a trusted relayer, then submit a crafted fork. The TODO comment at line 291 confirms the developers are aware the invariant is unverified. The bug is deterministically reproducible on any Dogecoin deployment at height ≥ 145,001.

---

### Recommendation

Replace the `get_header_by_height` call with an ancestor walk starting from `prev_block_header`, following `prev_block_hash` links through `headers_pool` until reaching `height_first`. This mirrors how `get_prev_header` already traverses the fork-aware pool: [8](#0-7) 

The corrected logic should walk `blocks_to_go_back` steps back through `get_prev_header` from `prev_block_header`, guaranteeing the returned block is the true ancestor regardless of mainchain state.

---

### Proof of Concept

State-machine test (Dogecoin, height ≥ 145,001, `skip_pow_verification = false`):

1. Initialize contract with genesis at height `H−1`. Mainchain tip = `AH` at height `H` with timestamp `T_AH = 1_700_000_100`.
2. Submit fork block `BH` at height `H` (prev = same parent as `AH`), timestamp `T_BH = 1_700_000_001` (just above MTP). Fork is stored in `headers_pool` but does not yet surpass mainchain chainwork.
3. Submit fork block `B(H+1)` at height `H+1`, prev = `BH`.
4. Attempt to submit fork block `B(H+2)` at height `H+2`, prev = `B(H+1)`.
   - Contract calls `get_next_work_required`: `height_first = H`, `get_header_by_height(H)` returns `AH` (timestamp `T_AH = 1_700_000_100`).
   - Correct call should use `BH` (timestamp `T_BH = 1_700_000_001`).
   - `modulated_timespan` computed from `T_AH` is larger → lower difficulty → `expected_bits` encodes an easier target.
5. Set `B(H+2).bits` to the incorrectly computed `expected_bits`. Mine `B(H+2)` satisfying only the easier target.
6. Assert: contract accepts `B(H+2)` despite its PoW being insufficient for the correct difficulty.
7. Assert: `get_header_by_height(H)` returns `AH.block_hash`, not `BH.block_hash`, confirming the wrong ancestor was used.
