### Title
SPV Proof Replay — Stateless `verify_transaction_inclusion` Records No Used-Proof State, Enabling Unlimited Replay by Any Caller - (File: `contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are pure read functions that write no on-chain state. The contract stores no record of which `(tx_id, tx_block_blockhash)` pairs have already been verified and acted upon. Any unprivileged NEAR caller can submit the identical SPV proof an unlimited number of times and receive `true` each time. A consuming contract that uses these results to authorize an irreversible cross-chain action (token mint, fund release) without implementing its own replay guard is therefore vulnerable to double-spend via proof replay.

---

### Finding Description

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are the two public entry points that downstream bridge contracts use to confirm a Bitcoin transaction was included in a confirmed block. [1](#0-0) [2](#0-1) 

Neither function writes any state. There is no `used_proofs: LookupSet<(H256, H256)>`, no nonce map, and no other mechanism that would cause a second call with the same arguments to behave differently from the first. The contract's `BtcLightClient` struct contains only header maps and chain-tip pointers. [3](#0-2) 

`ProofArgs` and `ProofArgsV2` carry no caller-supplied nonce or expiry field. [4](#0-3) 

The result of `compute_root_from_merkle_proof` compared against the stored `merkle_root` is deterministic and will always return the same value for the same inputs. [5](#0-4) 

---

### Impact Explanation

A consuming bridge contract that calls `verify_transaction_inclusion_v2` and, on `true`, mints wrapped BTC or releases escrowed funds has no contract-level guarantee that the same proof has not already been used. An attacker who controls the Bitcoin transaction (or simply observes a legitimate user's proof) can replay the identical call repeatedly. Each call returns `true`, and each call triggers another fund release. This is a direct double-spend against the consuming contract's asset pool.

The impact is loss of funds proportional to the value of the bridged transaction, repeatable without limit until the consuming contract's reserves are drained.

---

### Likelihood Explanation

The entry point is fully public — `verify_transaction_inclusion_v2` carries only a `#[pause]` guard, not an access-control role restriction. [6](#0-5) 

Any NEAR account can call it with a previously observed valid proof. No privileged key, social engineering, or off-chain coordination is required. The attack is executable in a single NEAR transaction after the first legitimate proof is submitted.

---

### Recommendation

Add a `used_proofs` set (e.g., `LookupSet<H256>` keyed on `keccak/sha256(tx_id || tx_block_blockhash || tx_index)`) to `BtcLightClient`. Before returning `true`, record the proof key and reject any future call that presents the same key. This mirrors the `randUsed[rand] = true` pattern that zAuction uses for bid nonces. Alternatively, document explicitly that replay protection is the sole responsibility of the consuming contract and provide a reference implementation, so integrators are not left to discover this gap independently.

---

### Proof of Concept

1. A legitimate user submits a valid SPV proof for Bitcoin transaction `T` in block `B` to a consuming bridge contract `C`.
2. `C` calls `verify_transaction_inclusion_v2({tx_id: T, tx_block_blockhash: B, ...})` → returns `true`.
3. `C` mints 1 WBTC to the user.
4. The attacker (or the same user) immediately calls `C` again with the identical proof struct.
5. `C` calls `verify_transaction_inclusion_v2` again with the same arguments.
6. The light client has recorded no state from step 2; `mainchain_header_to_height` still maps `B` to its height, `headers_pool` still holds the header with the same `merkle_root`, and `compute_root_from_merkle_proof` returns the same value.
7. `verify_transaction_inclusion_v2` returns `true` again.
8. `C` mints a second 1 WBTC — a double-spend with no Bitcoin transaction required beyond the original one.

Steps 4–8 can be repeated until `C`'s reserves are exhausted. [7](#0-6)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
