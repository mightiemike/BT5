### Title
Linked Signer Can Hijack Subaccount Authority by Self-Replacing via `LinkSigner` Transaction — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the **current linked signer** — a session key with delegated authority — can sign a `LinkSigner` transaction to replace itself with any arbitrary address. The subaccount owner's signature is not required. This is a direct access control analog to H-11: an approved delegate can mutate the authorization state of the account it was granted access to, without the owner's consent.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch reads:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` delegates to `validateSignature`, which accepts a signature from either `address(bytes20(sender))` (the subaccount owner) **or** `getLinkedSigner(sender)` (the current linked signer):

```solidity
function validateSignature(
    bytes32 sender,
    bytes32 digest,
    bytes memory signature,
    bool allowLinkedSigner
) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

And `Verifier.validateSignature` accepts either the owner or the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means: if Alice has set Bob as her linked signer, Bob can sign a `LinkSigner` transaction naming Charlie as the new `signer`. The sequencer includes it in a batch, `processTransactionImpl` validates Bob's signature as sufficient, and `linkedSigners[Alice's subaccount]` is overwritten with Charlie's address.

By contrast, the **slow-mode path** for `LinkSigner` correctly uses `validateSender`, which enforces that `address(bytes20(txn.sender)) == msg.sender` — i.e., only the subaccount owner can submit it:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The inconsistency between the two paths is the root cause: the sequencer path grants the linked signer the ability to mutate the very authorization mapping that defines its own scope of authority.

---

### Impact Explanation

Once Charlie is installed as the new linked signer, he can sign any transaction type that accepts `allowLinkedSigner = true` on Alice's subaccount:

- **`WithdrawCollateral` (V1)**: Linked signer allowed unconditionally. Charlie can drain Alice's collateral. Funds go to Alice's address, but forced withdrawal of collateral while Alice has open perp positions will trigger liquidation of those positions, causing direct financial loss. [5](#0-4) 
- **`MintNlp` / `BurnNlp`**: Linked signer allowed. Charlie can burn Alice's NLP tokens; `burnNlp` deducts a mandatory burn fee (`max(1e18, quoteAmount/1000)`), causing a direct loss. [6](#0-5) 
- **`LiquidateSubaccount`**: Linked signer allowed. Charlie can use Alice's subaccount as the liquidator to extract liquidation fees from third parties, with the fee charged to Alice's subaccount. [7](#0-6) 
- **`TransferQuote`**: Linked signer allowed. Charlie can transfer quote between Alice's own subaccounts (same `bytes20` prefix), enabling position manipulation. [8](#0-7) 

The corrupted state is `linkedSigners[Alice's subaccount]` — a mapping that controls who can authorize all high-value operations on the subaccount.

---

### Likelihood Explanation

Linked signers are explicitly designed as session keys for automated trading bots and delegated execution. Bot keys are operationally more exposed than cold-storage owner keys (they are hot, often server-side, and may be shared across infrastructure). A compromised bot key — through server breach, key leakage, or a malicious third-party integration — is a realistic threat. The attacker needs only one valid linked-signer signature to permanently escalate to full subaccount control. No admin access, governance capture, or sequencer compromise is required.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` branch in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [9](#0-8) 

Only the subaccount owner (the address encoded in `bytes20(sender)`) should be permitted to install or rotate a linked signer. This matches the invariant already enforced in the slow-mode path.

---

### Proof of Concept

1. Alice calls `submitSlowModeTransaction` with a `LinkSigner` transaction naming Bob (a trading bot) as her linked signer. `validateSender` passes because Alice is `msg.sender`.
2. An attacker obtains Bob's private key (e.g., via server compromise).
3. The attacker signs a `LinkSigner` transaction: `sender = Alice's subaccount`, `signer = Attacker's address`, `nonce = current nonce`.
4. The attacker submits this signed transaction to the sequencer's off-chain API.
5. The sequencer includes it in the next batch submitted to `submitTransactionsChecked`.
6. `processTransactionImpl` reaches the `LinkSigner` branch, calls `validateSignedTx(..., true)`, recovers Bob's address from the signature, accepts it as the linked signer, and writes `linkedSigners[Alice's subaccount] = Attacker`.
7. The attacker now signs a `BurnNlp` transaction for Alice's full NLP balance. The burn fee is deducted from Alice's quote balance. Alice suffers a direct financial loss with no recourse.
