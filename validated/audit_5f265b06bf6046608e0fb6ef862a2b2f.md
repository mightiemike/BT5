### Title
`CREATE2` Address Collision via User-Controlled `subaccount` in `createDirectDepositV1` Allows Theft of Deposited Funds — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.createDirectDepositV1` deploys a `DirectDepositV1` contract using `CREATE2` with a **fixed salt** but a **user-controlled constructor argument** (`subaccount`). Because the `subaccount` value is a free `bytes32` parameter supplied by any caller, it fully determines the `init_code` hash and therefore the deployed address. An attacker can brute-force `subaccount` values to pre-compute a large set of future DDA addresses and find a collision with an attacker-controlled wallet, enabling theft of any tokens deposited to that address.

---

### Finding Description

The CREATE2 address formula is:

```
keccak256(0xff ++ deployer ++ salt ++ keccak256(init_code))
```

In `createDirectDepositV1`:

- `deployer` = `ContractOwner` address — **fixed**
- `salt` = `bytes32(uint256(1))` — **fixed**
- `init_code` = `DirectDepositV1` bytecode + ABI-encoded `(endpoint, spotEngine, subaccount, wrappedNative)`

`endpoint`, `spotEngine`, and `wrappedNative` are fixed protocol addresses. The **only variable** is `subaccount`, which is a raw `bytes32` accepted from any caller with no restriction. [1](#0-0) 

This means an attacker can enumerate `subaccount` values freely to pre-compute ~2^80 distinct future DDA addresses, enabling a meet-in-the-middle collision search against an attacker-controlled CREATE2 wallet — identical in structure to the Arcadia vulnerability.

The function is `public` with no access control, so any unprivileged caller can invoke it. [2](#0-1) 

The `creditDepositV1` entry point is also `external` with no access control, meaning the attacker can trigger DDA deployment for any `subaccount` they choose. [3](#0-2) 

---

### Impact Explanation

Once a collision address `0xCOLLIDED` is found:

**Tx 1:**
- Attacker deploys an attack contract to `0xCOLLIDED` via their own CREATE2 factory.
- Attack contract sets infinite ERC-20 allowances from `0xCOLLIDED` to the attacker wallet for all tokens supported by the protocol.
- Attack contract calls `selfdestruct` in the same transaction (valid post-Dencun per EIP-6780 — same-tx `selfdestruct` still clears code).

**Tx 2:**
- Attacker calls `creditDepositV1(subaccount_x)`, deploying `DirectDepositV1` to `0xCOLLIDED`.
- Any tokens sent to `0xCOLLIDED` (the DDA) can now be drained by the attacker using the pre-set allowances, bypassing the `onlyOwner` `withdraw()` guard entirely. [4](#0-3) 

The `creditDeposit()` function on `DirectDepositV1` is callable by anyone and deposits all token balances to `subaccount_x` — which the attacker controls — further allowing the attacker to credit stolen funds into their own protocol subaccount. [5](#0-4) 

The `onlyOwner` `withdraw()` guard is irrelevant once the allowance is pre-set, since `transferFrom` bypasses it entirely.

---

### Likelihood Explanation

The attack is identical in feasibility to the Arcadia finding. The Bitcoin network hashrate (~6×10²⁰ hashes/sec as of the referenced report) achieves 2^80 hashes in ~33 minutes. A fraction of that power finds a collision in a practical timeframe. The cost has been estimated at a few million USD, which is profitable against any DDA holding meaningful user deposits. The `subaccount` space is a full 32-byte input, giving the attacker complete control over one side of the collision search.

---

### Recommendation

1. **Remove user control over the deployed address.** Replace `CREATE2` with vanilla `CREATE` (no `salt`). The address will then be determined by `ContractOwner`'s nonce, which the attacker cannot pre-compute.
2. **If `CREATE2` must be retained**, do not allow the full 32-byte `subaccount` to influence `init_code`. Truncate the attacker-influenceable input to ≤64 bits (e.g., use only the lower 8 bytes of `subaccount` as the salt, keeping the rest fixed), which raises the collision cost by a factor of 2^32 or more.
3. **Alternatively**, include a protocol-controlled entropy source (e.g., a monotonically incrementing nonce stored in `ContractOwner`) as part of the salt or constructor args, so the attacker cannot pre-compute the address before the deployment transaction is mined.

---

### Proof of Concept

```
1. Attacker enumerates subaccount values s_0, s_1, ..., s_{2^80}
   and pre-computes DDA addresses:
     addr_i = keccak256(0xff ++ ContractOwner ++ bytes32(1) ++ keccak256(DirectDepositV1_initcode(s_i)))

2. Attacker independently brute-forces CREATE2 salts for their own factory
   to find salt_j such that their factory deploys to some addr_i = 0xCOLLIDED.

3. Tx 1 (attacker):
   - Deploy attack contract to 0xCOLLIDED via attacker's CREATE2 factory with salt_j.
   - Inside attack contract constructor:
       IERC20(token).approve(attacker_wallet, type(uint256).max);
   - selfdestruct(attacker_wallet);  // same-tx, valid post-Dencun

4. Tx 2 (attacker):
   - Call ContractOwner.creditDepositV1(s_i)
     → deploys DirectDepositV1 to 0xCOLLIDED
   - Any user tokens sent to 0xCOLLIDED are now drainable via transferFrom.
   - Attacker calls creditDeposit() to credit tokens to their own subaccount s_i,
     then withdraws from the protocol.
```

The `getDirectDepositV1BytecodeHash()` integrity check does not mitigate this — it only verifies the `DirectDepositV1` bytecode has not been tampered with, not that the deployment address is safe. [6](#0-5)
