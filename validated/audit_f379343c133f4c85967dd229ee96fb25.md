### Title
Silent Receipt-Size Clamping in `try_forward` Breaks Congestion-Control Size-Limit Accounting — (`runtime/runtime/src/congestion_control.rs`)

### Summary

`ReceiptSinkV2::try_forward` silently clamps an oversized receipt's `size` to `max_receipt_size` before both the admission check and the limit-decrement step. The receipt is then forwarded at its **actual** (larger) byte count, but the outgoing size budget is only reduced by the **clamped** value. This is the nearcore analog of M-11: instead of rejecting the over-limit value, the code silently substitutes the maximum, allowing the real value to pass through while the accounting tracks a smaller number.

### Finding Description

In `ReceiptSinkV2::try_forward`, when a receipt's serialized size exceeds `max_receipt_size`, the local variable `size` is overwritten with `max_receipt_size`:

```rust
// runtime/runtime/src/congestion_control.rs  lines 418-426
if size > max_receipt_size {
    tracing::debug!(...);
    size = max_receipt_size;   // ← silent clamp
}
```

The clamped `size` is then used for two purposes:

1. **Admission check** (line 451): `forward_limit.size >= size` — the receipt passes because the clamped value is smaller than the actual byte count.
2. **Budget decrement** (line 455): `forward_limit.size -= size` — the budget is reduced by the clamped value, not the actual receipt size.

The receipt itself is pushed to `outgoing_receipts` at its full, unclamped byte count (line 453). The divergent pair is:

| Variable | Value |
|---|---|
| Actual receipt bytes forwarded | `actual_size` (e.g. 4 MB + N) |
| Bytes subtracted from `forward_limit.size` | `max_receipt_size` (4 MB) |
| Phantom capacity created | `actual_size − max_receipt_size` = N bytes |

Subsequent receipts in the same chunk can consume this phantom capacity, causing the total bytes forwarded to a shard to exceed `outgoing_receipts_usual_size_limit` (100 KB) or `outgoing_receipts_big_size_limit` (~4.5 MB) by up to `actual_size − max_receipt_size` per oversized receipt.

Oversized receipts are reachable today via two user-controlled paths documented in the test suite: `promise_return` chaining and large-value returns (issue #12606). Both paths are exercised by unprivileged accounts with deployed contracts.

### Impact Explanation

The congestion-control size limit is a protocol-level invariant: each shard's outgoing budget must accurately reflect the bytes actually forwarded so that the receiving shard's memory pressure is bounded. When the budget is decremented by less than the actual bytes sent, the invariant is broken. A shard can forward more bytes per chunk than the protocol intends, undermining the backpressure mechanism that prevents receiving shards from running out of memory. The effect compounds across multiple oversized receipts in the same chunk.

**Severity: High** — breaks a load-bearing protocol invariant; reachable by any user with a deployed contract; no privileged role required.

### Likelihood Explanation

Oversized receipts already exist on mainnet (the workaround was added precisely because they were observed). Any contract that uses `promise_return` with a near-maximum-size receipt, or returns a value close to `max_receipt_size`, can produce an oversized receipt. The clamping path is therefore reachable in normal production operation, not only under adversarial conditions.

### Recommendation

The admission check and the budget decrement must use the **actual** receipt size, not the clamped value. Two options:

1. **Reject oversized receipts at the forwarding boundary** — return `ReceiptForwarding::NotForwarded` (buffer the receipt) when `actual_size > max_receipt_size`, and fix the upstream creation bug (#12606) so such receipts are never produced.
2. **Deduct the actual size** — if the workaround must remain, deduct `actual_size` from `forward_limit.size` (not the clamped value) so the budget accurately reflects bytes forwarded. The admission check should also use `actual_size` to prevent a receipt that would exceed the budget from being admitted.

### Proof of Concept

**Divergent bytes (exact values, mainnet protocol version ≥ 70):**

- `max_receipt_size` = 4,194,304 bytes
- `outgoing_receipts_big_size_limit` = 4,718,592 bytes

A contract using `promise_return` can produce a receipt of size 4,194,304 + 200,000 = 4,394,304 bytes (as demonstrated in `test_max_receipt_size_promise_return`). When `try_forward` processes it:

- `size` is clamped to 4,194,304
- Admission check: 4,718,592 ≥ 4,194,304 → **passes**
- Receipt forwarded at actual 4,394,304 bytes
- Budget decremented by 4,194,304 → remaining = 524,288 bytes (phantom)
- A second receipt of 400,000 bytes: 524,288 ≥ 400,000 → **passes**
- Total forwarded: 4,394,304 + 400,000 = **4,794,304 bytes > 4,718,592 limit**
