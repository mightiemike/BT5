### Title
Linked Signer Can Overwrite Its Own Authorization via `LinkSigner` Transaction — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type is processed with `allowLinkedSigner = true`, meaning the currently registered linked signer of a subaccount can sign and submit a new `LinkSigner` transaction to replace itself with any arbitrary address. This allows a compromised or malicious linked signer to permanently escalate its own access or lock out the original subaccount owner from their own signing authority.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes the current linked signer as the accepted signer when `allowLinkedSigner` is true:

```solidity
// EndpointTx.sol lines 172–184
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`verifier.validateSignature` accepts the signature if it recovers to either the subaccount owner address **or** the linked signer:

```solidity
// Verifier.sol lines 291–304
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

The `linkedSigners` mapping is the sensitive access-control state:

```solidity
// EndpointStorage.sol line 50
mapping(bytes32 => address) internal linkedSigners;
``` [4](#0-3) 

Because the linked signer is accepted as a valid signer for the `LinkSigner` transaction type itself, it can unilaterally rewrite `linkedSigners[subaccount]` to any address it chooses — including a fresh address it controls — without any participation from the original subaccount owner.

The protocol demonstrates awareness that linked signers should not be trusted for all sensitive operations: `WithdrawCollateralV2` restricts linked signer acceptance when `sendTo != address(0)`:

```solidity
// EndpointTx.sol line 447
signedTx.tx.sendTo == address(0)   // allowLinkedSigner only when sendTo is zero
``` [5](#0-4) 

No equivalent restriction exists for `LinkSigner`, which is the most privileged mutation of all.

---

### Impact Explanation

The corrupted state is `linkedSigners[subaccount]`. Once overwritten by the linked signer, the new address has full signing authority over the subaccount, including the ability to:

- Sign `WithdrawCollateral` / `WithdrawCollateralV2` to drain all collateral
- Sign `TransferQuote` to move quote balances to attacker-controlled subaccounts
- Sign `LiquidateSubaccount` to liquidate positions at will
- Sign further `LinkSigner` transactions to chain the takeover indefinitely

The original owner loses the ability to revoke the linked signer via the normal `LinkSigner` path because the attacker's new linked signer can immediately re-link to another address, racing any revocation attempt. The only recovery path would be a slow-mode `LinkSigner` (3-day delay), during which the attacker can drain the account.

---

### Likelihood Explanation

The linked signer role is explicitly designed for delegation to hot wallets, bots, or third-party trading services — entities that are not fully trusted by the subaccount owner. Any such entity that turns malicious (or is compromised) can execute this attack in a single sequencer batch. No special on-chain conditions are required. The attacker-controlled entry path is: craft a valid `SignedLinkSigner` struct, sign it with the current linked signer key, and submit it to the sequencer for inclusion in `submitTransactions` / `submitTransactionsChecked`.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Modifying the linked signer is a privileged ownership operation and must require a signature from the subaccount owner address directly, not from the currently delegated signer:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

Apply the same restriction to the slow-mode `LinkSigner` path in `processSlowModeTransactionImpl` (line 232–239), which currently bypasses signature validation entirely and only checks `validateSender` — confirm that `validateSender` enforces the owner address and not the linked signer. [6](#0-5) 

---

### Proof of Concept

1. Alice owns subaccount `0xAlice000...000` and has linked Bob (`0xBob`) as her trading bot via a prior `LinkSigner` transaction.
2. Bob constructs `SignedLinkSigner { tx: { sender: 0xAlice000...000, signer: 0xBobNew, nonce: N }, signature: sig_bob }` where `sig_bob` is Bob's ECDSA signature over the EIP-712 digest of the `LinkSigner` struct.
3. Bob submits this to the sequencer (or waits for it to be included in a batch).
4. `processTransactionImpl` decodes the transaction, calls `validateSignedTx(..., true)`.
5. `validateSignature` calls `verifier.validateSignature(0xAlice000...000, 0xBob, digest, sig_bob)`.
6. `ECDSA.recover` returns `0xBob`, which equals `linkedSigner` → validation passes.
7. `linkedSigners[0xAlice000...000]` is set to `0xBobNew`.
8. Bob now controls `0xBobNew` and uses it to sign a `WithdrawCollateral` transaction draining Alice's collateral before she can react. [1](#0-0) [3](#0-2)
