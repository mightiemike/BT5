### Title
Unchecked Boolean Return Value of `equihash::is_valid_solution` Allows Invalid PoW Acceptance - (File: `contract/src/zcash.rs`)

### Summary
`check_pow` in the Zcash build calls `equihash::is_valid_solution`, which returns `Result<bool, Error>`. The code handles only the `Err` variant via `.unwrap_or_else`; the inner `bool` is silently discarded. If the library returns `Ok(false)` for an invalid solution, the block passes PoW verification and is accepted into the light client's canonical chain.

### Finding Description
In `contract/src/zcash.rs`, the Equihash solution check reads:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
``` [1](#0-0) 

`equihash::is_valid_solution` has the signature `fn is_valid_solution(...) -> Result<bool, Error>`. The `.unwrap_or_else` call converts the `Result` into a bare `bool` by panicking on `Err(_)` and returning the inner value on `Ok(_)`. The trailing `;` then discards that `bool` entirely. Consequently:

- `Ok(true)` → discarded, block accepted ✓ (correct)
- `Err(e)` → `env::panic_str`, block rejected ✓ (correct)
- `Ok(false)` → `false` returned from closure, **discarded**, block accepted ✗ (incorrect)

The `equihash` crate documents `Ok(false)` as the normal path for a well-formed but cryptographically invalid solution (i.e., the solution bytes parse correctly but do not satisfy the Equihash constraints). This is precisely the case an attacker would craft.

### Impact Explanation
A relayer or any unprivileged NEAR caller can submit a Zcash block header whose `solution` field is syntactically valid (correct byte length, parseable) but cryptographically invalid. The block passes `check_pow` without error, is stored in `headers_pool`, and can become the canonical chain tip if its chain-work exceeds the current tip. Downstream callers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will then receive `true` for transactions in a block that was never actually mined under valid Zcash consensus rules, breaking the core SPV guarantee of the light client. [2](#0-1) 

### Likelihood Explanation
The Zcash build is a supported production target (built via `make build-zcash`). The `submit_blocks` entry point is `#[payable]` and `#[pause]`-gated but otherwise open to any NEAR account. Crafting a syntactically valid but semantically invalid Equihash solution requires only knowledge of the expected byte layout (200/9 parameters, 1344-byte solution), which is fully public. No privileged role, leaked key, or social engineering is required. [3](#0-2) 

### Recommendation
Replace the discarded-result pattern with an explicit check on the returned boolean:

```rust
let valid = equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
require!(valid, "Invalid Equihash solution");
```

This mirrors the `require!` pattern used for every other PoW check in the same function and throughout the contract.

### Proof of Concept
1. Build the contract with `make build-zcash`.
2. Initialize the contract with a valid Zcash genesis block.
3. Construct a `BlockHeader` for the next height whose `solution` field has the correct byte length (1344 bytes for n=200, k=9) but contains arbitrary bytes that do not satisfy the Equihash constraints (e.g., all zeros).
4. Call `submit_blocks([crafted_header])` from any NEAR account with sufficient deposit.
5. Observe that the call succeeds and `get_last_block_header()` returns the crafted header as the new chain tip.
6. Call `verify_transaction_inclusion_v2` with any `tx_id` and a valid Merkle proof against the crafted block's `merkle_root` (which the attacker controls); observe it returns `true`. [4](#0-3)

        // This retrospectively activates at block height 2 for mainnet and regtest,
        // and 6 blocks after Blossom activation for testnet.
        //
        // MAX_FUTURE_BLOCK_TIME_MTP is typically 129600 seconds (36 hours) in Zcash

        // Check timestamp

    #[payable]
    #[pause]
    #[trusted_relayer]
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
