### Title
`DepositAllowlistExtension` Checks LP Owner Instead of Actual Depositor, Allowing Complete Allowlist Bypass — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` validates the `owner` (LP position recipient) against the allowlist instead of the `sender` (the actual caller of `addLiquidity`). Because `addLiquidity` accepts a caller-supplied `owner` with no requirement that `msg.sender == owner`, any unprivileged actor can bypass the deposit allowlist entirely by specifying an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address and passes both `msg.sender` (as `sender`) and `owner` to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both values and forwards them to the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first argument (`sender`) and checks only `owner`: [3](#0-2) 

Because `addLiquidity` has no `msg.sender == owner` guard, any caller can supply any `owner`. If that `owner` is on the allowlist, the check passes regardless of who the actual depositor is.

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (the actual swapper) and ignores `recipient`: [4](#0-3) 

The asymmetry is the root cause: the swap guard checks the right actor; the deposit guard checks the wrong one.

---

### Impact Explanation

The deposit allowlist is completely ineffective. Any unprivileged actor can:

1. Identify any allowlisted address (e.g., from past `AllowedToDepositSet` events).
2. Call `pool.addLiquidity(owner = allowlisted_address, ...)` directly.
3. Provide tokens via the callback (tokens come from `msg.sender`, not `owner`).
4. The extension passes because `allowedDepositor[pool][allowlisted_address]` is `true`.

The allowlisted address receives LP shares it did not request. The pool admin's intent to restrict depositors is entirely defeated. This is an admin-boundary break: an unprivileged path bypasses a pool-admin-configured access control guard with direct fund-flow consequences (unauthorized liquidity enters the pool, LP positions are created without the position owner's consent). [5](#0-4) 

---

### Likelihood Explanation

Exploitation requires no special privileges, no flash loans, and no complex setup. Any EOA or contract can call `addLiquidity` with a known allowlisted address as `owner`. Allowlisted addresses are discoverable on-chain from emitted events. Likelihood is **High**.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor) instead of `owner`, consistent with how `SwapAllowlistExtension` handles `beforeSwap`:

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

---

### Proof of Concept

```
Setup:
  - Pool P deployed with DepositAllowlistExtension E.
  - Admin calls E.setAllowedToDeposit(P, Alice, true).
  - Bob is NOT on the allowlist.

Attack:
  1. Bob calls P.addLiquidity(owner=Alice, salt=0, deltas=..., callbackData=..., extensionData=...).
  2. Pool calls E.beforeAddLiquidity(sender=Bob, owner=Alice, ...).
  3. Extension checks allowedDepositor[P][Alice] == true → passes.
  4. Bob's callback provides tokens; Alice receives LP shares at (Alice, salt=0).
  5. Bob has bypassed the allowlist and deposited into a restricted pool.

Result:
  - Unauthorized liquidity enters the pool.
  - Alice holds an unsolicited LP position.
  - The deposit allowlist provides zero protection against any actor who knows an allowlisted address.
```
