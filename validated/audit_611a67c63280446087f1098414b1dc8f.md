### Title
Missing Zero-Address Validation in `updateSubmitter` Can Lock Protocol Upgrade Pipeline - (File: `core/contracts/BaseProxyManager.sol`)

---

### Summary

`BaseProxyManager.updateSubmitter` assigns a new `submitter` address with no validation whatsoever — not even a zero-address check. If the owner mistakenly sets `submitter` to `address(0)` or an uncontrolled address, the `onlySubmitter`-gated upgrade functions (`submitImpl`, `refreshCodeHash`) become permanently inaccessible, freezing the protocol's entire upgrade pipeline.

---

### Finding Description

In `BaseProxyManager.sol`, the `updateSubmitter` function is:

```solidity
function updateSubmitter(address newSubmitter) external onlyOwner {
    submitter = newSubmitter;
}
``` [1](#0-0) 

There is no `require(newSubmitter != address(0), ...)` guard, no code-existence check, and no two-step confirmation pattern. The `submitter` state variable is the sole address authorized by the `onlySubmitter` modifier:

```solidity
modifier onlySubmitter() {
    require(
        msg.sender == submitter,
        "only submitter can submit new impls."
    );
    _;
}
``` [2](#0-1) 

This modifier gates both `submitImpl` and `refreshCodeHash` in `ProxyManager.sol`:

```solidity
function submitImpl(string memory name, address impl)
    external
    override
    onlySubmitter
{ ... }
``` [3](#0-2) 

`submitImpl` is the **only** entry point to queue a new implementation address into `pendingImpls[name]`, which is a prerequisite for `migrateAll` to execute any upgrade. If `submitter` is set to `address(0)`, no EOA can satisfy `msg.sender == submitter`, and the upgrade pipeline is bricked.

---

### Impact Explanation

Setting `submitter` to `address(0)` or a lost/invalid address causes:

- `submitImpl` to revert for every caller — no new implementation can ever be queued.
- `refreshCodeHash` to revert — code hash state cannot be refreshed.
- `migrateAll` to be permanently blocked, since `pendingImpls` can never be updated.

The protocol's entire on-chain upgrade mechanism is frozen. The owner retains the ability to call `updateSubmitter` again to recover, **but only if the owner key is still valid**. If the same operational mistake that corrupted `submitter` also affects the owner (e.g., a multisig misconfiguration), recovery is impossible.

**Impact score: 4** — Upgrade pipeline fully locked; recovery possible only if owner remains valid.

---

### Likelihood Explanation

Requires the owner (multisig) to pass an incorrect address to `updateSubmitter`. This is a low-probability operational error, consistent with the original report's likelihood score.

**Likelihood score: 1**

---

### Recommendation

Add a zero-address guard and optionally a two-step transfer pattern (pending → accept) mirroring the `transferOwnership` pattern:

```solidity
function updateSubmitter(address newSubmitter) external onlyOwner {
    require(newSubmitter != address(0), "BaseProxyManager: new submitter is zero address");
    submitter = newSubmitter;
}
```

For stronger safety, adopt a pending/accept pattern so the new submitter must confirm acceptance before the role is transferred, eliminating the risk of assigning to an address whose private key is unavailable.

---

### Proof of Concept

1. Owner calls `updateSubmitter(address(0))` — no revert occurs.
2. `submitter` is now `address(0)`.
3. Any call to `submitImpl(name, impl)` reverts with `"only submitter can submit new impls."` because `msg.sender` can never equal `address(0)`.
4. `pendingImpls[name]` can never be updated.
5. `migrateAll(newImpls)` will always revert at the `hasPending()` check or silently fail to apply upgrades, since `pendingImpls` is stale.
6. The entire upgrade pipeline is frozen until the owner calls `updateSubmitter` with a valid address — assuming the owner key is still accessible. [1](#0-0) [2](#0-1) [3](#0-2)
