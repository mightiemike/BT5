**The off-by-one that makes the exploit real:**

Suppose `gc_threshold = N`. Chain has exactly N blocks at heights H…H+N-1. Tip = H+N-1.

- Block at height H has exactly N confirmations: `(H+N-1) - H + 1 = N`. ✓
- User constructs proof for block H with `confirmations = N`.
- A new block arrives at height H+N (via `submit_blocks` by a relayer, or any caller triggers `run_mainchain_gc`).
- Now `amount_of_headers_we_store = N+1 > N` → GC removes block at height H.
- User's `verify_transaction_inclusion` executes:
  - Line 290: `N <= N` → **passes**.
  - Line 300: `mainchain_header_to_height.get(&block_H_hash)` → **None** → panic `"block does not belong to the current main chain"`.

The block was canonically valid and had the required confirmations at proof-construction time. The check is supposed to guarantee verifiability but is off-by-one: it allows `confirmations = gc_threshold` yet GC removes exactly the block that would have that many confirmations once the next block arrives.

### Title
Off-by-one in `verify_transaction_inclusion` confirmation guard allows GC to permanently invalidate proofs for the oldest retained block — (`contract/src/lib.rs`, `btc-types/src/contract_args.rs`)

### Summary
`verify_transaction_inclusion` permits `confirmations == gc_threshold`, but GC removes the oldest block the moment a new block is appended, making the block with exactly `gc_threshold` confirmations permanently unverifiable. Because `run_mainchain_gc` is publicly callable (no access control beyond the pause gate), any unprivileged caller can trigger this deletion before a pending verification executes.

### Finding Description
`verify_transaction_inclusion` enforces:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

This is intended to guarantee the target block is still in storage. However, `run_mainchain_gc` removes the oldest block whenever `amount_of_headers_we_store > gc_threshold`:

```rust

    let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
    ...
    self.remove_block_header(blockhash);          // removes from headers_pool
    self.mainchain_height_to_header.remove(&height);
``` [2](#0-1) 

`remove_block_header` deletes from both `mainchain_header_to_height` and `headers_pool`:

```rust

    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [3](#0-2) 

After GC fires (triggered by a new block submission or a direct call to `run_mainchain_gc`), the lookup at line 300 returns `None` for the removed block:

```rust

    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
``` [4](#0-3) 

`run_mainchain_gc` carries no privileged-caller restriction in normal (unpaused) operation:

```rust

``` [5](#0-4) 

GC also fires automatically inside `submit_blocks`:

```rust

``` [6](#0-5)

### Impact Explanation
Once a block is removed by GC it is gone permanently. Any transaction whose SPV proof targets that block hash can never be verified again. If the light client is used as a bridge oracle (the primary use case), this constitutes a permanent, irrecoverable proof lockout for every transaction confirmed in that block. The victim cannot retry with a different block — the transaction is in exactly one block.

### Likelihood Explanation
The condition is reached in normal operation: every time a relayer submits a new block when the chain is at exactly `gc_threshold` stored blocks, GC fires and removes the oldest block. No attacker is required; the natural relay cadence produces this state continuously. An adversary can also call `run_mainchain_gc(large_batch)` directly at any time to aggressively prune blocks and race any in-flight verification transaction.

### Recommendation
Change the confirmation guard from `<=` to `<`:

```rust
require!(
    args.confirmations < self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

This ensures that a block with `confirmations = gc_threshold - 1` is always at least one block inside the retained window, so a single new block arrival cannot evict it before verification completes. Alternatively, enforce that GC never removes a block whose height satisfies `tip_height - block_height + 1 <= gc_threshold`, keeping the oldest retained block strictly inside the window.

### Proof of Concept
State-machine test (no privileged roles needed):

1. Initialize contract with `gc_threshold = N`, submit genesis + N-1 additional blocks → chain has exactly N blocks at heights H…H+N-1.
2. Record `oldest_hash = mainchain_height_to_header[H]`.
3. Submit one more block (height H+N). `submit_blocks` internally calls `run_mainchain_gc(1)`, removing block H. Alternatively, call `run_mainchain_gc(1)` directly as any unprivileged account.
4. Call `verify_transaction_inclusion` with `tx_block_blockhash = oldest_hash`, `confirmations = N` (≤ gc_threshold, so the guard passes).
5. Assert the call panics with `"block does not belong to the current main chain"` — confirming permanent proof lockout for a block that was canonically valid and within the declared confirmation window at proof-construction time.
