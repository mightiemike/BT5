### Title
Lossy Protobuf `ValidResourceBounds` Round-Trip Silently Downgrades `AllResources` to `L1Gas`, Producing a Divergent Transaction Hash in P2P Block Sync — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf serialization of `ValidResourceBounds` is lossy: both the `L1Gas` variant and the `AllResources` variant with zero `l2_gas` and `l1_data_gas` produce identical protobuf bytes. On deserialization, the converter uses value-based logic to pick the variant, silently reconstructing an `AllResources` transaction as `L1Gas`. Because `get_tip_resource_bounds_hash` hashes a different number of resource elements depending on the variant (2 for `L1Gas`, 3 for `AllResources`), the recomputed transaction hash diverges from the hash that was computed at gateway admission and stored in the block. Any node that receives such a transaction via P2P block sync stores a structurally wrong transaction object whose hash cannot be reproduced from its own fields.

---

### Finding Description

**Step 1 – Serialization is lossy.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` serializes both variants with all three fields present:

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),   // always zero
    l1_data_gas: Some(ResourceBounds::default().into()), // always zero
},
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(l2_gas.into()),
    l1_data_gas: Some(l1_data_gas.into()),
},
```

When `AllResources` has `l2_gas = 0` and `l1_data_gas = 0`, the two serialized forms are byte-for-byte identical. The variant tag is not encoded. [1](#0-0) 

**Step 2 – Deserialization picks the wrong variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` reconstructs the variant by inspecting values:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

A transaction originally submitted as `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` is reconstructed as `L1Gas(l1_gas=X)`. [2](#0-1) 

**Step 3 – The hash preimage differs between variants.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` hashes a different number of resource elements depending on the variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements total
    }
});
```

`L1Gas` → Poseidon over `[tip, L1_GAS_concat, L2_GAS_concat]`  
`AllResources` (even with `l1_data_gas=0`) → Poseidon over `[tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat(zero)]`

These produce different field elements and therefore different transaction hashes. [3](#0-2) 

**Step 4 – The gateway always uses `AllResources` for RPC transactions.**

`RpcInvokeTransactionV3`, `RpcDeclareTransactionV3`, and `RpcDeployAccountTransactionV3` all carry `AllResourceBounds` (not `ValidResourceBounds`). Every conversion to `InvokeTransactionV3` / `DeclareTransactionV3` wraps it as `ValidResourceBounds::AllResources(...)`. The hash stored at admission time is therefore always computed with the 3-element preimage. [4](#0-3) [5](#0-4) 

**Step 5 – The P2P block sync path uses the lossy converter.**

The P2P sync server reads `Transaction` objects (which carry `ValidResourceBounds`) from storage and serializes them to protobuf `FullTransaction`. The P2P sync client deserializes them back using `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The reconstructed `Transaction` has `L1Gas` instead of `AllResources`. The stored `transaction_hash` field is passed through unchanged (it is the original `AllResources` hash), so the database now holds a transaction whose fields cannot reproduce its own stored hash. [6](#0-5) 

**Step 6 – The OS hint `AssertTransactionHash` enforces hash consistency.**

During block re-execution (proof generation / verification), the OS hint `assert_transaction_hash` computes the transaction hash from the transaction fields and compares it to the stored hash. A node that re-executes a block using the P2P-synced transaction data would compute the `L1Gas` hash, which differs from the stored `AllResources` hash, causing the assertion to fail. [7](#0-6) 

---

### Impact Explanation

A V3 transaction with `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` is admitted by the gateway with a hash `H_all` (3-element Poseidon). After P2P block sync, the receiving node stores the transaction as `L1Gas(l1_gas=X)`. Any subsequent hash recomputation from stored fields yields `H_l1 ≠ H_all`. This breaks:

1. **RPC wrong value** (`starknet_getTransactionByHash`, `starknet_getTransactionReceipt`): the returned resource bounds show `L1Gas` instead of `AllResources`, an authoritative-looking wrong value.
2. **Block re-execution / proof generation**: `assert_transaction_hash` fails, making the node unable to re-execute or prove any block containing such a transaction.
3. **`validate_transaction_hash`**: returns `false` for a valid, included transaction, breaking any downstream logic that relies on hash verification.

---

### Likelihood Explanation

Any unprivileged user can submit a V3 invoke/declare/deploy-account transaction with `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0`, and `l1_gas.max_amount > 0`, `l1_gas.max_price_per_unit > 0`. The stateless validator only rejects transactions where `max_possible_fee(Tip::ZERO) == 0`; with non-zero L1 gas bounds this check passes. The stateful validator only checks the L2 gas price threshold; with `l2_gas.max_price_per_unit = 0` this passes when the threshold is zero (bootstrap / testing configurations) or when the check is disabled. The transaction is included in a block, and every P2P-syncing node is affected. [8](#0-7) 

---

### Recommendation

1. **Encode the variant explicitly in protobuf.** Add a boolean or enum field (e.g., `is_all_resources`) to the `ResourceBounds` protobuf message so the variant can be round-tripped faithfully without inspecting values.

2. **Alternatively, treat all protobuf-deserialized V3 transactions as `AllResources`.** Since the sequencer only admits V3 transactions as `AllResources` at the RPC layer, the P2P block sync deserializer for V3 transactions should unconditionally produce `AllResources`, matching the admission-time invariant.

3. **Add a round-trip test.** Assert that `ValidResourceBounds::AllResources(AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 })` survives a protobuf round-trip with the same variant and produces the same transaction hash.

---

### Proof of Concept

```
1. Construct a V3 invoke transaction with:
     resource_bounds = AllResourceBounds {
         l1_gas:      ResourceBounds { max_amount: 1000, max_price_per_unit: 1 },
         l2_gas:      ResourceBounds { max_amount: 0,    max_price_per_unit: 0 },
         l1_data_gas: ResourceBounds { max_amount: 0,    max_price_per_unit: 0 },
     }

2. Compute hash H_all = get_invoke_transaction_v3_hash(tx, chain_id, version)
   → get_tip_resource_bounds_hash uses AllResources → 3 resource elements
   → H_all = Poseidon([tip, L1_GAS_concat, L2_GAS_concat(0), L1_DATA_GAS_concat(0)])

3. Serialize to protobuf::ResourceBounds:
     l1_gas      = Some(1000, 1)
     l2_gas      = Some(0, 0)
     l1_data_gas = Some(0, 0)

4. Deserialize via TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
   → l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → returns ValidResourceBounds::L1Gas(ResourceBounds { max_amount: 1000, max_price_per_unit: 1 })

5. Recompute hash H_l1 from the reconstructed transaction:
   → get_tip_resource_bounds_hash uses L1Gas → 2 resource elements
   → H_l1 = Poseidon([tip, L1_GAS_concat, L2_GAS_concat(0)])

6. H_all ≠ H_l1  (different number of elements in the Poseidon chain)

7. assert_transaction_hash(stored=H_all, computed=H_l1) → FAIL
```

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl FetchBlockData for FullTransaction {
    async fn fetch_block_data(
        block_number: BlockNumber,
        txn: &StorageTxn<'_, db::RO>,
        _class_manager_client: &mut SharedClassManagerClient,
    ) -> Result<Vec<Self>, P2pSyncServerError> {
        let transactions =
            txn.get_block_transactions(block_number)?.ok_or(P2pSyncServerError::BlockNotFound {
                block_hash_or_number: BlockHashOrNumber::Number(block_number),
            })?;
        let transaction_outputs = txn.get_block_transaction_outputs(block_number)?.ok_or(
            P2pSyncServerError::BlockNotFound {
                block_hash_or_number: BlockHashOrNumber::Number(block_number),
            },
        )?;
        let transaction_hashes = txn.get_block_transaction_hashes(block_number)?.ok_or(
            P2pSyncServerError::BlockNotFound {
                block_hash_or_number: BlockHashOrNumber::Number(block_number),
            },
        )?;
        let mut result: Vec<FullTransaction> = Vec::new();
        for (transaction, transaction_output, transaction_hash) in transactions
            .into_iter()
            .zip(transaction_outputs)
            .zip(transaction_hashes)
            .map(|((a, b), c)| (a, b, c))
        {
            result.push(FullTransaction { transaction, transaction_output, transaction_hash });
        }
        Ok(result)
    }

pub(crate) fn assert_transaction_hash<S: StateReader>(
    hint_processor: &mut SnosHintProcessor<'_, S>,
    ctx: HintContext<'_>,
) -> OsHintResult {
    let stored_transaction_hash = ctx.get_integer(Ids::TransactionHash)?;
    let calculated_tx_hash =
        hint_processor.get_current_execution_helper()?.tx_tracker.get_tx()?.tx_hash().0;

    if calculated_tx_hash == stored_transaction_hash {
        Ok(())
    } else {
        Err(OsHintError::AssertionFailed {
            message: format!(
                "Computed transaction_hash is inconsistent with the hash in the transaction. \
                 Computed hash = {stored_transaction_hash:#x}, Expected hash = \
                 {calculated_tx_hash:#x}."
            ),
        })
    }
}
