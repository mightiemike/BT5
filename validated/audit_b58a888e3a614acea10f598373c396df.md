### Title
Unchecked `transferFrom` Return Value Enables Theft of usdcE from DirectDepositV1 Accounts - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value. If the USDC token returns `false` on failure rather than reverting, the subsequent unconditional withdrawal of usdcE from the DDA and transfer to the caller still executes, allowing any unprivileged caller to drain usdcE from any `DirectDepositV1` account without providing USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is an unguarded `external` function (only gated by `block.chainid == 57073`) that performs a three-step token swap:

1. Pull USDC from `msg.sender` into the target DDA — **return value unchecked**
2. Withdraw usdcE from the DDA to `ContractOwner`
3. Forward usdcE from `ContractOwner` to `msg.sender` [1](#0-0) 

Steps 2 and 3 execute unconditionally regardless of whether step 1 succeeded. The rest of the codebase consistently uses the `ERC20Helper.safeTransfer` / `safeTransferFrom` wrappers, which low-level-call the token and `require` a truthy return value. [2](#0-1) 

The single raw `.transferFrom(...)` call at line 616 is the only place in the production contract set that bypasses this pattern. [3](#0-2) 

`DirectDepositV1.withdraw` is `onlyOwner`, and `ContractOwner` is the owner of every DDA it deploys, so the withdrawal in step 2 always succeeds from `ContractOwner`'s perspective. [4](#0-3) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (Ink chain) returns `false` on a failed `transferFrom` (e.g., insufficient allowance or balance) rather than reverting, an attacker with zero USDC can:

- Call `replaceUsdcEWithUsdc(subaccount)` for any subaccount whose DDA holds usdcE.
- The `transferFrom` silently fails; no USDC moves.
- `DirectDepositV1.withdraw(usdcE)` transfers the full usdcE balance from the DDA to `ContractOwner`.
- `safeTransfer(msg.sender, balance)` forwards that usdcE to the attacker.

**Corrupted asset delta:** the DDA loses its entire usdcE balance; the attacker gains it; no USDC is deposited in exchange. This is a direct, complete theft of user-deposited collateral held in DDAs.

---

### Likelihood Explanation

The function is callable by any address on chain 57073 with no privilege requirement. The exploitability depends on whether the specific USDC deployment at that address returns `false` on failure rather than reverting. Many USDC deployments (Circle's canonical contracts) do revert, which reduces immediate likelihood. However:

- The address is a hardcoded constant; if it is a proxy that is upgraded to a non-reverting implementation, the vulnerability becomes immediately exploitable.
- The code pattern is unconditionally wrong regardless of current token behavior, and the Nado codebase itself treats this as a known risk class (hence the `ERC20Helper` library used everywhere else).

Likelihood: **Medium** (token behavior is the gating factor; the on-chain entry path is fully open to any unprivileged caller).

---

### Recommendation

Replace the bare `.transferFrom` call with the project's own `safeTransferFrom` wrapper from `ERC20Helper`:

```solidity
// Before (ContractOwner.sol L616)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ContractOwner` already imports and uses `ERC20Helper` via `using ERC20Helper for IERC20Base` at line 24, so no additional imports are needed. [5](#0-4) 

---

### Proof of Concept

```
Precondition:
  - Chain ID = 57073 (Ink)
  - subaccount S has a deployed DDA with usdcE balance B > 0
  - USDC token at 0x2D270... returns false (not revert) on transferFrom failure

Attack:
  1. Attacker calls ContractOwner.replaceUsdcEWithUsdc(S)
     with zero USDC allowance / balance.

  2. L616: IERC20Base(usdc).transferFrom(attacker, dda, B)
     → returns false, no USDC moves, no revert.

  3. L617: DirectDepositV1(dda).withdraw(usdcE)
     → ContractOwner (owner of dda) pulls B usdcE from dda to itself.

  4. L618: IERC20Base(usdcE).safeTransfer(attacker, B)
     → B usdcE sent to attacker.

Result: Attacker holds B usdcE; DDA holds 0 usdcE; 0 USDC deposited.
```

```text
    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
