### Title
Silent Swallow of Slow-Mode `DepositCollateral` Failure Permanently Locks User Tokens — (`core/contracts/Endpoint.sol`)

---

### Summary

`_executeSlowModeTransaction` in `Endpoint.sol` permanently deletes a slow-mode transaction from the queue **before** calling `processSlowModeTransaction`. If the inner call reverts for any non-OOG reason, the `catch` block silently discards the failure. For `DepositCollateral` transactions enqueued via `depositCollateralWithReferral`, the user's ERC-20 tokens have already been transferred into the `Endpoint` contract in a prior, separate transaction. If the slow-mode execution later fails silently, those tokens are permanently locked with no recovery path — a direct analog to the Optimism `revert`-vs-`return` desynchronization bug.

---

### Finding Description

**Step 1 — Tokens transferred before the slow-mode entry is created**

`depositCollateralWithReferral` first pulls tokens from the caller into the contract, then appends a `DepositCollateral` entry to the slow-mode queue. These are two separate, independent state changes. [1](#0-0) 

**Step 2 — Queue entry is deleted before execution**

`_executeSlowModeTransaction` deletes the entry from `slowModeTxs` and increments `txUpTo` on line 194, **before** `processSlowModeTransaction` is ever called. This is the committed state change that cannot be undone if the subsequent call fails. [2](#0-1) 

**Step 3 — Silent catch with no recovery**

The `try/catch` block on lines 207–227 catches any non-OOG revert from `processSlowModeTransaction` and does nothing. The comment on line 226 — `// try return funds now removed` — is an explicit admission that a prior fund-return mechanism was deliberately deleted, leaving no fallback. [3](#0-2) 

**Step 4 — `processSlowModeTransactionImpl` can revert for `DepositCollateral`**

Inside `processSlowModeTransactionImpl`, the `DepositCollateral` branch calls `clearinghouse.depositCollateral(txn)`. The analogous `withdrawCollateral` path in `Clearinghouse.sol` shows that the clearinghouse enforces `require(address(token) != address(0))` — a check that would revert if the product was delisted between the time the user deposited and the time the slow-mode transaction is executed (the delay is hardcoded to three days). [4](#0-3) [5](#0-4) 

**Desynchronization summary**

| Contract | State change | Reversible? |
|---|---|---|
| `Endpoint` | Tokens transferred in (prior tx) | No |
| `Endpoint` | `slowModeTxs[txUpTo]` deleted, `txUpTo++` | No (catch swallows revert) |
| `Clearinghouse` | `depositCollateral` — user balance credited | Never reached |

The user's on-chain token balance is decremented, the slow-mode entry is gone, and the engine balance is never credited. There is no replay mechanism.

---

### Impact Explanation

A user who calls `depositCollateral` / `depositCollateralWithReferral` loses their deposited ERC-20 tokens permanently if the corresponding slow-mode `DepositCollateral` transaction fails silently during execution. The tokens sit in the `Endpoint` contract with no owner-accessible withdrawal path for the affected user.

---

### Likelihood Explanation

The slow-mode delay is hardcoded to three days. [6](#0-5) 

During that window, a `DelistProduct` admin transaction can remove a product, zeroing its token address in the spot engine. Any pending `DepositCollateral` slow-mode entry for that product will then revert inside `clearinghouse.depositCollateral` and be silently swallowed. The `DelistProduct` transaction type is explicitly handled in `processSlowModeTransactionImpl`, confirming it is a supported in-protocol operation. [7](#0-6) 

Additionally, `DirectDepositV1.creditDeposit` calls `depositCollateralWithReferral` for every product returned by `spotEngine.getProductIds()`, widening the attack surface to any product that could be delisted. [8](#0-7) 

---

### Recommendation

Restore a fund-return path in the `catch` block. When `processSlowModeTransaction` fails for a `DepositCollateral` transaction, the `Endpoint` contract holds the tokens and knows the original depositor (`txn.sender`). The catch block should decode the transaction type and, for `DepositCollateral`, transfer the tokens back to the depositor before discarding the entry. Alternatively, re-queue the failed entry so it can be retried, mirroring the `failedMessages` replay mechanism used in Optimism's `CrossDomainMessenger`.

---

### Proof of Concept

1. Alice calls `depositCollateral(subaccountName, productId, amount)` on `Endpoint`. Tokens are pulled from Alice into the contract; a `DepositCollateral` slow-mode entry is appended at index `txCount`. [9](#0-8) 

2. Within the three-day window, the protocol owner submits a `DelistProduct` slow-mode transaction for `productId`. It is processed by the sequencer, zeroing the token address in the spot engine.

3. After the delay, anyone calls `executeSlowModeTransaction()`. `_executeSlowModeTransaction` runs:
   - Line 194: `delete slowModeTxs[txUpTo++]` — Alice's entry is gone.
   - Line 207: `try this.processSlowModeTransaction(...)` — `clearinghouse.depositCollateral` hits `require(address(token) != address(0))` and reverts.
   - Lines 219–227: `gasleft()` is well above the OOG threshold; the `catch` block falls through to `// try return funds now removed` and does nothing. [10](#0-9) 

4. Alice's tokens remain in the `Endpoint` contract. Her engine balance was never credited. The slow-mode entry no longer exists. There is no replay or refund mechanism. Alice's funds are permanently lost.
