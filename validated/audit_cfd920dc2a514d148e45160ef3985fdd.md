### Title
Unchecked Boolean Return Value of `equihash::is_valid_solution` Allows Invalid PoW Blocks to Be Accepted — (`contract/src/zcash.rs`)

---

### Summary

In `contract/src/zcash.rs`, the `check_pow` function calls `equihash::is_valid_solution(...)`, which returns `Result<bool, Error>`. The code handles the `Err` variant by panicking, but **silently discards the `Ok(false)` case** — meaning a structurally valid but cryptographically invalid Equihash solution passes the check without triggering any rejection. This is a direct analog to the Remora bug: a verification function returns a boolean indicating failure, but the caller never checks it.

---

### Finding Description

In `contract/src/zcash.rs`, `check_pow` performs the Equihash solution check as follows:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
``` [1](#0-0) 

`equihash::is_valid_solution` has the signature `fn(...) -> Result<bool, Error>`:
- `Err(e)` → format/parse error → the closure panics (correctly handled)
- `Ok(true)` → valid solution → accepted (correct)
- `Ok(false)` → **invalid solution, but structurally parseable** → the `bool` is returned by `unwrap_or_else` and then **immediately discarded as an unused expression statement**

The `Ok(false)` path is never enforced. No `require!`, no assertion, no check. The function returns normally, and the block proceeds to be stored in the chain.

This `check_pow` is invoked via `check_target`:

```rust
fn check_target(&self, block_header: &Header, prev_block_header: &ExtendedHeader) {
    self.check_pow(block_header, prev_block_header);
}
``` [2](#0-1) 

Which is called from `submit_block_header` when `skip_pow_verification` is `false`:

```rust
if !skip_pow_verification {
    self.check_target(&header, &prev_block_header);
    ...
}
``` [3](#0-2) 

And `submit_block_header` is called for every header in the public `submit_blocks` entry point:

```rust
for header in headers {
    self.submit_block_header(header, self.skip_pow_verification);
}
``` [4](#0-3) 

---

### Impact Explanation

An attacker (or compromised relayer) can craft a Zcash block header with a structurally valid but cryptographically invalid Equihash solution. When submitted via `submit_blocks`, the `check_pow` call returns without panicking (because `Ok(false)` is silently discarded), and the block is accepted into `headers_pool` and potentially promoted to the mainchain tip. This corrupts the canonical chain state: the contract stores a block that does not represent real Bitcoin-family PoW, breaking the core SPV security guarantee. Downstream callers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` may then receive `true` for transactions in fabricated blocks. [5](#0-4) 

---

### Likelihood Explanation

The `submit_blocks` function is gated by the `#[trusted_relayer]` macro, which restricts callers to registered/staked relayers. However, the entire security model of a light client is that PoW verification is the trustless, on-chain guarantee — even a registered relayer must not be able to inject invalid blocks. A compromised or malicious relayer can trivially construct a Zcash header with a well-formed but invalid Equihash solution (e.g., a solution vector of the correct length with wrong values) and submit it. The `equihash` crate will return `Ok(false)`, which is silently ignored.

---

### Recommendation

Enforce the boolean return value with a `require!`:

```rust
let is_valid = equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
require!(is_valid, "Invalid Equihash solution: solution check returned false");
```

Alternatively, map `Ok(false)` directly to a panic:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| env::panic_str(&format!("Invalid Equihash solution: {e}")))
    .then(|| ())
    .unwrap_or_else(|| env::panic_str("Invalid Equihash solution"));
```

---

### Proof of Concept

1. Build a Zcash block header with valid `bits`, `time`, `version >= 4`, and a correctly-sized `solution` byte vector that does not satisfy the Equihash(200,9) constraints.
2. Submit it via `submit_blocks` as a registered relayer (or with `Role::UnrestrictedSubmitBlocks`).
3. `check_pow` is called; `equihash::is_valid_solution` returns `Ok(false)`; the return value is discarded.
4. The block passes all checks and is stored via `store_block_header` / `store_fork_header`.
5. If its `chain_work` exceeds the current tip, `reorg_chain` promotes it to the mainchain.
6. `verify_transaction_inclusion_v2` will now return `true` for any transaction whose Merkle proof matches the fabricated block's `merkle_root`. [6](#0-5) [7](#0-6)

        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }
