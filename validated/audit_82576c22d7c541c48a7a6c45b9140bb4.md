### Title
Linked Signer Can Authorize Its Own Replacement via `LinkSigner` Fast-Path Transaction — (`core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` fast-path transaction is validated with `allowLinkedSigner = true`. This means the currently linked signer — a delegated hot-wallet key — can sign and submit a `LinkSigner` transaction to replace itself with an attacker-controlled address. The slow-mode path correctly restricts this operation to the subaccount owner only, but the fast path does not enforce the same restriction, creating an asymmetric and exploitable inconsistency.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` fast-path case passes `allowLinkedSigner = true` to `validateSignedTx`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` calls `validateSignature`, which passes the current linked signer as an accepted signer when `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 172–184
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`Verifier.validateSignature` accepts a signature from either the subaccount owner address or the linked signer:

```solidity
// Verifier.sol lines 297–303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

By contrast, the **slow-mode** `LinkSigner` path enforces that only the subaccount owner can submit this operation:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // ← msg.sender must be the owner address
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

`validateSender` requires `address(uint160(bytes20(txn.sender))) == sender` (i.e., the actual `msg.sender` must be the wallet address embedded in the subaccount). The fast path has no equivalent restriction.

The protocol is demonstrably aware of linked-signer privilege risks: `WithdrawCollateralV2` sets `allowLinkedSigner = signedTx.tx.sendTo == address(0)` to prevent a linked signer from withdrawing to an arbitrary address, and `Clearinghouse.transferQuote` includes the comment *"require the sender address to be the same as the recipient address, otherwise linked signers can transfer out."* The `LinkSigner` fast path is the one sensitive operation where this guard was not applied. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

A compromised linked signer (e.g., an API key or hot wallet used for automated trading) can:

1. Sign a `LinkSigner` transaction targeting the victim's subaccount, setting the new linked signer to an attacker-controlled address.
2. The sequencer processes this as a valid fast-path transaction.
3. The attacker's address is now the permanent linked signer for the victim's subaccount.
4. The attacker can sign `MatchOrders`, `WithdrawCollateral` (V1, to the owner's address), `MintNlp`, `BurnNlp`, `TransferQuote`, and `LiquidateSubaccount` transactions on behalf of the victim indefinitely.
5. The attacker can place adversarial trades (e.g., as a counterparty) that drain the victim's collateral through losses, or trigger liquidations.

The victim's only recovery path is to submit a slow-mode `LinkSigner` transaction (which requires a 3-day delay) or a new fast-path `LinkSigner` signed with their own key — but the attacker can front-run the nonce by submitting another `LinkSigner` via the sequencer before the victim's slow-mode transaction executes.

---

### Likelihood Explanation

Linked signers are explicitly designed for automated trading bots and API integrations — hot wallets with elevated exposure. Key compromise of such wallets is a realistic threat (phishing, server breach, leaked `.env` files). Once a linked signer is compromised, the attacker can immediately escalate to persistent subaccount control without any on-chain signal until trades begin. The nonce required for the attack is publicly readable via `Endpoint.getNonce()`. [7](#0-6) 

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` fast-path transaction, so that only the subaccount owner's key can authorize a linked-signer change:

```solidity
// EndpointTx.sol — LinkSigner fast-path
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← only the subaccount owner may change the linked signer
);
```

This aligns the fast path with the slow-mode path, which already enforces owner-only authorization via `validateSender`.

---

### Proof of Concept

1. Alice has subaccount `alice_sub` (`bytes32`) with linked signer `hot_wallet`.
2. Attacker compromises `hot_wallet`.
3. Attacker reads `Endpoint.getNonce(alice_address)` → `nonce = N`.
4. Attacker crafts `SignedLinkSigner { tx: { sender: alice_sub, nonce: N, signer: attacker_wallet }, signature: sig_by_hot_wallet }`.
5. Attacker submits this to the sequencer (off-chain RFQ / sequencer API).
6. Sequencer calls `submitTransactionsChecked` → `processTransaction` → `processTransactionImpl`.
7. `validateSignedTx(alice_sub, N, ..., sig_by_hot_wallet, true)` passes because `hot_wallet == getLinkedSigner(alice_sub)`.
8. `linkedSigners[alice_sub] = attacker_wallet` is written.
9. Attacker now signs `MatchOrders` and other transactions for `alice_sub` indefinitely. [1](#0-0) [8](#0-7)
