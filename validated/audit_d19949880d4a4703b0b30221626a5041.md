### Title
Missing `ENABLE_SECP_OPS` Flag Check in 4-Byte Opcode Path Allows Secp Operations to Execute Unconditionally — (`File: src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through two separate code paths: a 1-byte opcode path (opcodes 64 and 65) and a 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`). The 1-byte path correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte path does not check this flag at all, allowing an attacker-controlled CLVM program to invoke secp signature verification unconditionally, regardless of whether the hard-fork flag has been activated.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` handles operators of length 4 bytes separately from 1-byte operators:

**4-byte path (lines 157–183) — no flag check:**
```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // ← no ENABLE_SECP_OPS check
        0x1c3a8f00 => op_secp256r1_verify,   // ← no ENABLE_SECP_OPS check
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

**1-byte path (lines 248–249) — flag correctly checked:**
```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The `ENABLE_SECP_OPS` flag is defined as a hard-fork activation gate:

> "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)."

When this flag is absent (e.g., pre-fork consensus mode, or `MEMPOOL_MODE` which does not include `ENABLE_SECP_OPS`), the 1-byte opcodes 64 and 65 fall through to `unknown_operator()` and are either silently no-oped (consensus) or rejected (mempool). However, the 4-byte encodings `0x13d61f00` and `0x1c3a8f00` bypass this gate entirely and execute the real secp operations.

An attacker-controlled CLVM program need only encode the operator as a 4-byte atom to trigger this path.

---

### Impact Explanation

**Consensus divergence / hard-fork bypass:** The `ENABLE_SECP_OPS` flag is a hard-fork control. Before the fork activates, secp operations should be unavailable. A CLVM program using 4-byte secp opcodes executes them regardless of the flag, meaning:

- Pre-fork nodes running this code accept secp-verified transactions that should be invalid.
- If any nodes correctly enforce the flag for 4-byte opcodes (e.g., a patched version), consensus splits: one set of nodes accepts the transaction, the other rejects it.
- In mempool mode (`MEMPOOL_MODE`), the 1-byte secp opcodes are rejected as unimplemented, but the 4-byte path executes them, creating a mempool/consensus inconsistency for the same logical operation.

The corrupted result is a `Reduction` from `op_secp256k1_verify` or `op_secp256r1_verify` being returned when the correct result should be an `EvalErr::Unimplemented` or a no-op nil.

---

### Likelihood Explanation

The 4-byte opcode encoding is a documented, reachable part of the CLVM wire format. Any caller that can submit attacker-controlled CLVM bytes (e.g., via transaction submission to a full node) can craft a program with a 4-byte operator atom. No special privileges are required. The encoding is straightforward: serialize the operator as a 4-byte atom `[0x13, 0xd6, 0x1f, 0x00]` for secp256k1 or `[0x1c, 0x3a, 0x8f, 0x00]` for secp256r1.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the guard already present in the 1-byte path:

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
        0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

This ensures both encoding paths are governed by the same flag, eliminating the bypass.

---

### Proof of Concept

1. Construct a CLVM program where the operator atom is the 4-byte sequence `[0x13, 0xd6, 0x1f, 0x00]` (secp256k1_verify) with valid secp256k1 arguments.
2. Run it via `run_program` with a `ChiaDialect` that does **not** have `ENABLE_SECP_OPS` set (e.g., `ChiaDialect::new(ClvmFlags::empty())`).
3. Observe that `op_secp256k1_verify` executes and returns a `Reduction` instead of failing or returning nil.
4. Run the same logical operation using 1-byte opcode 64 under the same flags.
5. Observe that opcode 64 falls through to `unknown_operator` and returns nil (consensus mode) or `EvalErr::Unimplemented` (mempool mode).

The two paths produce different results for the same logical operation under the same flags, confirming the flag/operator wiring error. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
