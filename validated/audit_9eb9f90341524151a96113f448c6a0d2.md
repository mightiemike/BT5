### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Validation Added in `verify_transaction_inclusion_v2` — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof. However, the original `verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR contract method. Any unprivileged caller can invoke v1 directly, skipping the coinbase proof check entirely. The `#[deprecated]` Rust attribute is a compile-time hint only; it does not remove the function from the on-chain ABI.

---

### Finding Description

`verify_transaction_inclusion_v2` performs two steps: (1) it validates a coinbase Merkle proof against the block's `merkle_root`, and (2) it delegates to `verify_transaction_inclusion` for the actual transaction proof check. [1](#0-0) 

The coinbase check is the only guard against the 64-byte forgery attack. It exists exclusively inside `verify_transaction_inclusion_v2`; the inner function `verify_transaction_inclusion` contains no such guard. [2](#0-1) 

Because `verify_transaction_inclusion` carries `pub` visibility and the `#[pause]` attribute (not `#[private]`), it is part of the contract's exported NEAR ABI and is callable by any account via a NEAR transaction or cross-contract call. The `#[deprecated]` Rust attribute only produces a compiler warning for Rust callers; it has no effect on the on-chain interface. [3](#0-2) 

---

### Impact Explanation

The 64-byte transaction forgery (documented at https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to craft a 64-byte blob that is simultaneously a valid serialized internal Merkle tree node and a plausible "transaction." By supplying this blob as `tx_id` with a matching Merkle path, an attacker can make `verify_transaction_inclusion` return `true` for a transaction that was never included in the block. Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a bridge withdrawal, cross-chain settlement, or asset release will accept a forged proof and release funds or update state incorrectly.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. Any NEAR account can submit a transaction calling `verify_transaction_inclusion` with a crafted `ProofArgs`. The function is not paused by default. The only prerequisite is knowledge of a valid block header already stored in the contract's `headers_pool`, which is public state.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI by either:
- Marking it `#[private]` (restricts callers to the contract itself), or
- Deleting the function body and replacing it with `env::panic_str("removed: use verify_transaction_inclusion_v2")`.

The internal delegation from `verify_transaction_inclusion_v2` should then call the shared proof logic directly rather than going through the deprecated public method.

---

### Proof of Concept

1. Identify any block hash stored in `mainchain_header_to_height` (readable via `get_block_hash_by_height`).
2. Construct a 64-byte payload that, when double-SHA256'd, equals an internal Merkle node of the target block's Merkle tree. Supply this as `tx_id` with a one-element `merkle_proof` containing the sibling node, and `tx_index = 0`.
3. Call `verify_transaction_inclusion` directly (not `_v2`) with these arguments and `confirmations = 1`.
4. The function computes `compute_root_from_merkle_proof(tx_id, 0, &merkle_proof)` and compares it to `header.block_header.merkle_root`. [4](#0-3) 

Because no coinbase proof is checked, the forged `tx_id` (which hashes to a real internal node) produces the correct Merkle root, and the function returns `true` for a transaction that does not exist. A downstream contract consuming this result would treat the forgery as a confirmed Bitcoin transaction.

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
