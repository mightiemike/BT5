### Title
Governance `MsgUpdateTier` applies updated `BonusApy` retroactively to all historical accrual segments, wiping unclaimed bonus rewards for existing position holders — (`x/tieredrewards/keeper/claim_rewards.go`)

### Summary

`processEventsAndClaimBonus` fetches the tier's `BonusApy` **at claim time** and applies it uniformly to every historical bonded segment since the position was created. A governance-approved `MsgUpdateTier` that reduces `BonusApy` (including to zero) therefore retroactively zeroes out all bonus rewards that existing position holders have already earned but not yet claimed.

### Finding Description

In `processEventsAndClaimBonus`, the tier is loaded once from the current store state: [1](#0-0) 

The same `tier` object — carrying the **current** `BonusApy` — is then passed into `computeSegmentBonus` for every historical bonded segment in the event replay loop: [2](#0-1) 

and again for the final open segment: [3](#0-2) 

The formula applied to every segment is `shares × tokensPerShare × bonusApy × durationSeconds / SecondsPerYear`. [4](#0-3) 

`BonusApy` is a mutable field on the `Tier` struct, changeable via the governance-gated `MsgUpdateTier`: [5](#0-4) 

The `Tier` proto carries `BonusApy` as a live governance-mutable field with no per-position snapshot: [6](#0-5) 

The Architecture Decision Record explicitly lists which messages settle rewards before mutations — `MsgTierUndelegate`, `MsgTierRedelegate`, `MsgAddToTierPosition`, `MsgClearPosition`, and `MsgExitTierWithDelegation` — and **`MsgUpdateTier` is absent from that list**: [7](#0-6) 

A helper `claimRewardsAndUpdateTierPositions` exists that could settle all positions in a tier before a mutation: [8](#0-7) 

However, the ADR's omission of `MsgUpdateTier` from the reward-settlement list, combined with the fact that `processEventsAndClaimBonus` re-fetches the tier from the store (meaning it would use the **post-update** APY if called after the update), indicates that no pre-settlement guard is in place for tier APY changes. I was unable to read `x/tieredrewards/keeper/msg_server_auth.go` directly to confirm the exact call order; if `claimRewardsAndUpdateTierPositions` is called **before** the tier write in that handler, the vulnerability is mitigated. If it is absent or called after, the vulnerability is fully present.

### Impact Explanation

Every position holder who has accrued bonus rewards since their last claim has those rewards computed using the **current** `BonusApy` at claim time. A governance proposal that sets `BonusApy` to `0` causes `computeSegmentBonus` to return `0` for every historical segment, regardless of how long the position has been bonded. The user receives zero bonus tokens from `RewardsPoolName` for the entire accrual period. The corrupted value is the `totalBonus` integer accumulated in `processEventsAndClaimBonus` across all historical segments. [9](#0-8) 

### Likelihood Explanation

The trigger path is a standard governance proposal (`MsgUpdateTier`) submitted and approved by the chain's governance. This is a higher bar than the single-owner case in the original report, but it is a fully supported, documented, and tested on-chain operation: [10](#0-9) 

Any governance participant can submit such a proposal. The impact is proportional to the total unclaimed bonus rewards across all positions in the affected tier at the time the proposal executes.

### Recommendation

Snapshot `BonusApy` into the `Position` record at lock time (analogous to the Maia fix of snapshotting `protocolFee` per deposit). Alternatively, ensure `MsgUpdateTier` calls `claimRewardsAndUpdateTierPositions` **before** writing the new tier to the store, so all historical accrual is settled at the old rate before the new rate takes effect. The `claimRewardsAndUpdateTierPositions` helper already exists for exactly this purpose. [8](#0-7) 

### Proof of Concept

1. User locks tokens into Tier 1 (e.g., `BonusApy = 0.10`, 10%) via `MsgLockTier`. Position is created; `LastBonusAccrual` is set to block time T0.
2. 180 days pass. The user does not claim. Accrued bonus ≈ `shares × tokensPerShare × 0.10 × (180×86400) / 31557600`.
3. Governance submits and passes `MsgUpdateTier` setting `BonusApy = 0` for Tier 1.
4. User calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` fetches the tier and reads `BonusApy = 0`. `computeSegmentBonus` returns `0` for every segment from T0 to now.
5. User receives **zero** bonus tokens despite 180 days of bonded stake at the originally advertised 10% APY. [11](#0-10)

// claimRewardsAndUpdateTierPositions claims base and bonus rewards for all
// delegated positions in the given tier.
func (k Keeper) claimRewardsAndUpdateTierPositions(ctx context.Context, tierId uint32) error {
	ids, err := k.getPositionsIdsByTier(ctx, tierId)
	if err != nil {
		return err
	}
	if len(ids) == 0 {
		return nil
	}

	for _, id := range ids {
		pos, err := k.getPositionState(ctx, id)
		if err != nil {
			return err
		}
		if !pos.IsDelegated() {
			continue
		}

		if _, err := k.claimBaseRewards(ctx, pos); err != nil {
			return err
		}
		if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
			return err
		}
		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return err
		}
	}

	return nil
}
