### Title
No Way to Update or Remove a Product Engine Once Assigned to an EngineType Key — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.addEngine()` enforces a one-time-only assignment of a `SpotEngine` or `PerpEngine` to its `EngineType` key via a hard `require` that the slot is empty. No `removeEngine()` or `updateEngine()` function exists anywhere in the protocol. If either engine is exploited or a critical bug is discovered, the protocol has no on-chain mechanism to replace it, permanently locking all user collateral and positions to the compromised engine.

---

### Finding Description

`Clearinghouse.addEngine()` contains the following guard:

```solidity
// Clearinghouse.sol L161
require(address(engineByType[engineType]) == address(0));
```

Once a `SpotEngine` or `PerpEngine` address is written into `engineByType[EngineType.SPOT]` or `engineByType[EngineType.PERP]`, the slot can never be overwritten or cleared. A grep across all production Solidity files confirms there is no `removeEngine`, `updateEngine`, or any function that writes to `engineByType` after initialization — only `addEngine` does so, and only when the slot is empty.

`ClearinghouseStorage._spotEngine()` and `_perpEngine()` always resolve the live engine address directly from this mapping:

```solidity
// ClearinghouseStorage.sol L31-38
function _spotEngine() internal view returns (ISpotEngine) {
    return ISpotEngine(address(engineByType[IProductEngine.EngineType.SPOT]));
}
function _perpEngine() internal view returns (IPerpEngine) {
    return IPerpEngine(address(engineByType[IProductEngine.EngineType.PERP]));
}
```

Every critical protocol operation — `depositCollateral`, `withdrawCollateral`, `getHealth`, `liquidateSubaccount`, `settlePnl`, `claimSequencerFees`, `mintNlp`, `burnNlp` — routes through one or both of these helpers. The `productToEngine` mapping is also seeded from `engineByType` at registration time (line 170), so all per-product routing is equally locked.

`ContractOwner.addEngine()` (the owner-facing entry point) simply delegates to `clearinghouse.addEngine()` with no additional override path.

---

### Impact Explanation

If either the `SpotEngine` or `PerpEngine` is exploited (e.g., a reentrancy or accounting bug drains balances), the protocol owner has no on-chain path to swap in a patched engine. All user collateral tracked in `SpotEngine.balances` and all perpetual positions tracked in `PerpEngine` remain permanently routed through the compromised contract. Deposits, withdrawals, health checks, and liquidations all continue to call into the exploited engine. The entire user fund base — every subaccount balance across every product — is at permanent risk with no recovery mechanism.

---

### Likelihood Explanation

Medium. The trigger requires an exploitable bug in `SpotEngine` or `PerpEngine` to materialize. However, the absence of a removal function is a concrete, unconditional protocol design flaw: the moment such a bug is found, the protocol is structurally unable to respond on-chain. Given that both engines are upgradeable proxies managed by `ProxyManager`, the intent to support upgrades is clear — the missing `removeEngine` path is an oversight, not a deliberate design choice.

---

### Recommendation

Add a privileged `removeEngine(IProductEngine.EngineType engineType)` function to `Clearinghouse.sol`, callable only by the owner, that clears `engineByType[engineType]` and removes the corresponding entry from `supportedEngines`. In an emergency, the owner can remove the compromised engine and call `addEngine` again with a patched replacement. Ensure that `productToEngine` entries pointing to the removed engine are also cleared or remapped to prevent stale routing.

---

### Proof of Concept

1. Owner calls `ContractOwner.addEngine(spotEngineAddr, offchainExchange, EngineType.SPOT)`.
2. `Clearinghouse.addEngine` writes `engineByType[SPOT] = spotEngineAddr` and returns. [1](#0-0) 
3. A critical vulnerability is discovered in `SpotEngine` (e.g., `updateBalance` can be called by a crafted contract that passes `canApplyDeltas` check).
4. Owner attempts to call `addEngine` again with a patched `SpotEngine` address — transaction reverts at `require(address(engineByType[engineType]) == address(0))` because the slot is already occupied. [2](#0-1) 
5. No `removeEngine` function exists in `Clearinghouse.sol`, `ContractOwner.sol`, or any other production contract. [3](#0-2) 
6. `_spotEngine()` continues to resolve to the compromised address for every subsequent `depositCollateral`, `withdrawCollateral`, `getHealth`, and `liquidateSubaccount` call. [4](#0-3) 
7. All user collateral and positions remain permanently routed through the exploited engine with no on-chain recovery path.

    function addEngine(
        address engine,
        address offchainExchange,
        IProductEngine.EngineType engineType
    ) external onlyOwner {
        require(address(engineByType[engineType]) == address(0));
        require(engine != address(0));
        IProductEngine productEngine = IProductEngine(engine);
        // Register
        supportedEngines.push(engineType);
        engineByType[engineType] = productEngine;

        // add quote to product mapping
        if (engineType == IProductEngine.EngineType.SPOT) {
            productToEngine[QUOTE_PRODUCT_ID] = productEngine;
        }

        // Initialize engine
        productEngine.initialize(
            address(this),
            offchainExchange,
            quote,
            getEndpoint(),
            owner()
        );
    }
