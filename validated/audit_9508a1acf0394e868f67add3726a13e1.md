### Title
Missing Sanctions Check on `sendTo` Recipient in `WithdrawCollateralV2` Allows Funds to Reach Sanctioned Addresses — (`File: core/contracts/EndpointTx.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`depositCollateralWithReferral` enforces sanctions checks on both the depositor (`msg.sender`) and the subaccount owner (`sender`). However, `WithdrawCollateralV2` — which introduces a user-controlled `sendTo` field — never checks whether `sendTo` is a sanctioned address. Funds can therefore be routed to a sanctioned recipient through a normal, unprivileged withdrawal flow.

---

### Finding Description

Nado's deposit path applies two sanctions checks before crediting any subaccount:

```solidity
// Endpoint.sol lines 133-135
requireUnsanctioned(msg.sender);
requireUnsanctioned(sender);
``` [1](#0-0) 

The `WithdrawCollateralV2` struct adds an explicit `sendTo` field that overrides the default behaviour of sending funds back to the subaccount owner:

```solidity
// IEndpoint.sol lines 97-104
struct WithdrawCollateralV2 {
    bytes32 sender;
    uint32 productId;
    uint128 amount;
    uint64 nonce;
    address sendTo;
    uint128 appendix;
}
``` [2](#0-1) 

When the sequencer processes a `WithdrawCollateralV2` transaction in `processTransactionImpl`, it validates the sender's signature and then passes `signedTx.tx.sendTo` directly to `clearinghouse.withdrawCollateral` — with no `requireUnsanctioned` call on `sendTo`:

```solidity
// EndpointTx.sol lines 437-465
} else if (txType == IEndpoint.TransactionType.WithdrawCollateralV2) {
    ...
    validateSignedTx(...);
    ...
    clearinghouse.withdrawCollateral(
        signedTx.tx.sender,
        signedTx.tx.productId,
        signedTx.tx.amount,
        signedTx.tx.sendTo,   // ← never sanction-checked
        nSubmissions
    );
``` [3](#0-2) 

`Clearinghouse.withdrawCollateral` resolves the final recipient and immediately routes the transfer:

```solidity
// Clearinghouse.sol lines 404-408
if (sendTo == address(0)) {
    sendTo = address(uint160(bytes20(sender)));
}
handleWithdrawTransfer(token, sendTo, amount, idx);
``` [4](#0-3) 

`handleWithdrawTransfer` forwards the call to `BaseWithdrawPool.submitWithdrawal`, which also performs no sanctions check before executing the ERC-20 transfer:

```solidity
// BaseWithdrawPool.sol lines 116-132
function submitWithdrawal(IERC20Base token, address sendTo, uint128 amount, uint64 idx) public {
    require(msg.sender == clearinghouse);
    ...
    handleWithdrawTransfer(token, sendTo, amount);
}
``` [5](#0-4) 

The same gap exists in the fast-withdrawal path (`submitFastWithdrawal`), which resolves `sendTo` from the signed transaction and transfers without any sanctions check:

```solidity
// BaseWithdrawPool.sol lines 67-77
address resolvedSendTo = signedTx.tx.sendTo == address(0)
    ? address(uint160(bytes20(signedTx.tx.sender)))
    : signedTx.tx.sendTo;
return (signedTx.tx.productId, resolvedSendTo, signedTx.tx.amount);
``` [6](#0-5) 

---

### Impact Explanation

Any user with a non-zero balance can sign a `WithdrawCollateralV2` transaction naming a sanctioned address as `sendTo`. The sequencer validates only the sender's signature; it has no on-chain mechanism to reject a sanctioned `sendTo`. The result is a direct ERC-20 transfer to a sanctioned address, violating OFAC compliance obligations and the protocol's own sanctions enforcement model. The corrupted state is the token balance of the sanctioned address, which receives funds it should never be able to receive.

**Impact: Medium**

---

### Likelihood Explanation

The `sendTo` field is a standard, documented feature of `WithdrawCollateralV2`. Any user who wishes to route funds to a sanctioned address needs only to set `sendTo` to that address in their signed withdrawal. No privileged access, no governance capture, and no external dependency is required. The sequencer cannot distinguish a sanctioned `sendTo` from a legitimate one without an on-chain check.

**Likelihood: Medium**

---

### Recommendation

Add a `requireUnsanctioned` check on the resolved `sendTo` address before executing the transfer in `Clearinghouse.withdrawCollateral`. Since `Clearinghouse` does not hold a direct reference to the sanctions list, the check can be delegated back through the `Endpoint` (which already owns the `sanctions` reference), or the sanctions list address can be stored in `ClearinghouseStorage`. Symmetrically, apply the same check in `BaseWithdrawPool.submitFastWithdrawal` after `resolveFastWithdrawal` returns the `sendTo` address.

```solidity
// Clearinghouse.withdrawCollateral — after resolving sendTo
if (sendTo == address(0)) {
    sendTo = address(uint160(bytes20(sender)));
}
requireUnsanctioned(sendTo); // ← add this
handleWithdrawTransfer(token, sendTo, amount, idx);
```

---

### Proof of Concept

1. Alice (non-sanctioned) holds 10,000 USDC in her Nado subaccount.
2. Bob's address is on the Chainalysis sanctions list.
3. Alice signs a `WithdrawCollateralV2` transaction: `sender = Alice's subaccount`, `amount = 10000e6`, `sendTo = Bob`.
4. Alice submits the signed transaction to the sequencer.
5. The sequencer calls `submitTransactionsChecked` → `processTransactionImpl`.
6. `processTransactionImpl` validates Alice's signature (passes) and calls `clearinghouse.withdrawCollateral(..., Bob, ...)`.
7. `Clearinghouse.withdrawCollateral` resolves `sendTo = Bob` (non-zero, so no override), calls `handleWithdrawTransfer(token, Bob, 10000e6, idx)`.
8. `BaseWithdrawPool.submitWithdrawal` calls `token.safeTransfer(Bob, 10000e6)`.
9. Bob (sanctioned) receives 10,000 USDC with no on-chain rejection at any step. [7](#0-6) [3](#0-2)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
