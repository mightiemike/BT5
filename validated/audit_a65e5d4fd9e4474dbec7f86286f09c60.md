### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Silent Failure and Potential usdcE Drain — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` directly via the typed interface without checking its return value and without using the protocol's own `ERC20Helper.safeTransferFrom` wrapper. If the USDC token returns `false` silently (without reverting), execution continues, withdrawing usdcE from the target DDA and sending it to the caller — without the caller having actually transferred any USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is a publicly callable migration function (no access modifier) that is supposed to atomically swap usdcE held in a `DirectDepositV1` address for USDC provided by the caller:

```solidity
// ContractOwner.sol line 616–619
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← raw, unchecked
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
IERC20Base(usdcE).safeTransfer(msg.sender, balance);                 // ← safe wrapper used here
``` [1](#0-0) 

The protocol already defines `ERC20Helper.safeTransferFrom`, which uses a low-level `.call()` and explicitly checks both the call success and the decoded return value:

```solidity
// ERC20Helper.sol lines 23–42
function safeTransferFrom(IERC20Base self, address from, address to, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(...);
    require(success && (data.length == 0 || abi.decode(data, (bool))), ERR_TRANSFER_FAILED);
}
``` [2](#0-1) 

Line 616 bypasses this wrapper entirely. The `IERC20Base` interface declares `transferFrom` as returning `bool`, but the return value is never read or asserted. [3](#0-2) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (Ink chain, `chainid == 57073`) returns `false` on a failed transfer without reverting, the three-step sequence executes as follows:

1. `transferFrom` returns `false` — no USDC moves from caller to DDA. Return value is silently ignored.
2. `DirectDepositV1(directDepositV1).withdraw(usdcE)` — usdcE is pulled from the DDA into `ContractOwner`.
3. `safeTransfer(msg.sender, balance)` — usdcE is sent to the caller.

The caller receives usdcE without having provided any USDC. Any DDA with a nonzero usdcE balance is drainable by any unprivileged caller. The corrupted asset delta is the full usdcE balance of the targeted DDA. [4](#0-3) 

---

### Likelihood Explanation

The function has no access control — it is callable by any address on chain 57073. The specific USDC token is likely a standard implementation that reverts on insufficient allowance rather than returning `false`, which reduces the probability of silent failure in practice. However, the protocol's own `safeTransferFrom` was written precisely to guard against this class of token behavior, and its omission here is a concrete code defect. Any future token upgrade or redeployment that returns `false` on failure would immediately expose the drain path.

---

### Recommendation

Replace the raw `transferFrom` call with `ERC20Helper.safeTransferFrom`, consistent with how every other token transfer in the protocol is handled:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ContractOwner` already imports and uses `ERC20Helper` via `using ERC20Helper for IERC20Base` at line 24, so no additional imports are needed. [5](#0-4) 

---

### Proof of Concept

1. Attacker identifies a `directDepositV1` address (via `directDepositV1Address[subaccount]`) holding a nonzero usdcE balance.
2. Attacker calls `replaceUsdcEWithUsdc(subaccount)` on chain 57073 **without** approving any USDC to `ContractOwner`.
3. If the USDC token returns `false` silently on the `transferFrom` (insufficient allowance path), the call does not revert.
4. `DirectDepositV1.withdraw(usdcE)` transfers the full usdcE balance to `ContractOwner`.
5. `safeTransfer(msg.sender, balance)` sends the full usdcE balance to the attacker.
6. Attacker has drained the DDA's usdcE at zero cost. [4](#0-3) [6](#0-5)
