### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is marked `#[deprecated]` but is still a fully public NEAR contract method. The `#[deprecated]` attribute is a Rust compile-time hint only — it does not restrict external NEAR calls. Any unprivileged NEAR account can invoke v1 directly, bypassing the coinbase Merkle proof check that `verify_transaction_inclusion_v2` adds specifically to prevent the 64-byte transaction Merkle proof forgery attack. This is a direct structural analog to the Rubic blacklist-bypass: a security control exists in one callable context (v2) but is absent in a second callable context (v1) that produces the same proof result.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to close the 64-byte Merkle forgery vulnerability. Its own doc comment explains the threat:

> *"This is needed to mitigate the 64-byte transaction Merkle proof forgery vulnerability: https://www.bitmex.com/blog/64-Byte-Transactions"*

The protection it adds is a coinbase Merkle proof check: [1](#0-0) 

Only after that check passes does it delegate to v1: [2](#0-1) 

The deprecated v1 function, however, performs **no coinbase proof check**. It only verifies that the supplied `tx_id` hashes to the block's `merkle_root` via the provided `merkle_proof`: [3](#0-2) 

Critically, v1 carries an explicit warning that this is exploitable: [4](#0-3) 

Despite this, v1 is declared `pub` with no `#[private]` guard: [5](#0-4) 

`#[deprecated]` is a Rust compiler annotation. It emits a warning to Rust callers at compile time. It has zero effect on the NEAR runtime's method dispatch. The method remains registered as a public contract entry point and is callable by any NEAR account over the network.

---

### Impact Explanation

The 64-byte Merkle forgery attack works as follows: Bitcoin's Merkle tree does not distinguish leaf nodes from internal nodes. A 64-byte internal node hash can be presented as a `tx_id`. Without a coinbase proof anchor, `compute_root_from_merkle_proof` will return the correct `merkle_root` for a crafted path, causing `verify_transaction_inclusion` to return `true` for a transaction that does not exist in the block.

Any downstream NEAR contract or off-chain consumer that calls `verify_transaction_inclusion` (v1) and trusts its `true` result as proof of Bitcoin transaction inclusion can be deceived into accepting a forged proof. This corrupts the core security invariant of the light client: that a `true` result guarantees the transaction was included in a confirmed Bitcoin block.

---

### Likelihood Explanation

The entry path is fully unprivileged. Any NEAR account can call `verify_transaction_inclusion` with a crafted `ProofArgs`. The 64-byte forgery technique is publicly documented and well understood. The only prerequisite is that a downstream contract or user relies on v1's result — which is realistic given that v1 predates v2 and may still be integrated in existing consumers.

---

### Recommendation

Apply `#[private]` to `verify_transaction_inclusion` to prevent external NEAR calls, or remove it from the public ABI entirely. If backward compatibility is required, gate v1 behind an access-control role so that only trusted internal callers (i.e., v2's internal delegation) can invoke it. The `#[deprecated]` attribute alone provides no runtime protection.

---

### Proof of Concept

1. Identify a Bitcoin block whose Merkle tree has at least one internal node whose serialized 64-byte representation can be treated as a valid `tx_id` input (this is the standard 64-byte forgery setup).
2. Construct a `merkle_proof` path such that `compute_root_from_merkle_proof(forged_tx_id, forged_index, &forged_proof)` equals the real `merkle_root` of a block already accepted by the light client.
3. Call `verify_transaction_inclusion` directly (not v2) with `ProofArgs { tx_id: forged_tx_id, tx_block_blockhash: real_block_hash, tx_index: forged_index, merkle_proof: forged_proof, confirmations: 1 }`.
4. The function returns `true` because:
   - The block is in the mainchain (`mainchain_header_to_height` lookup succeeds). [6](#0-5) 
   - The forged proof computes to the correct `merkle_root`. [7](#0-6) 
   - No coinbase proof check is performed anywhere in v1's execution path.
5. A downstream contract consuming this `true` result accepts the forged transaction as confirmed on Bitcoin.
