### Title
EIP-712 `computeDigest` Returns Raw `hashStruct` Without Domain Separator, Enabling Cross-Chain Signature Replay â€” (`core/contracts/Verifier.sol`)

---

### Summary

`Verifier.computeDigest` computes only the `hashStruct` component of an EIP-712 digest and returns it directly, omitting the mandatory `keccak256(abi.encodePacked(0x1901, domainSeparator, hashStruct))` wrapping. This means every user-signed transaction type processed through this path â€” `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, `LinkSigner`, `TransferQuote`, `MintNlp`, `BurnNlp` â€” produces a digest that is not bound to any specific chain ID or contract address. On a multi-chain deployment, a valid signature from one chain is replayable on any other chain where the protocol is deployed and the user's nonce matches.

---

### Finding Description

EIP-712 specifies that the final digest must be:

```
keccak256(abi.encodePacked(0x1901, domainSeparator, hashStruct))
```

`Verifier.computeDigest` computes only the inner `hashStruct` and returns it as the digest:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(WITHDRAW_COLLATERAL_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.productId,
        signedTx.tx.amount,
        signedTx.tx.nonce
    )
);
``` [1](#0-0) 

No domain separator is applied anywhere in `computeDigest`. The function is declared `public pure`, which structurally prevents it from ever calling `_domainSeparatorV4()` (a `view` function). Despite `Verifier` inheriting `EIP712Upgradeable`, the inherited domain-separator machinery is never used in this path. [2](#0-1) 

This is in direct contrast to `OffchainExchange.getDigest`, which correctly wraps the `structHash` with a domain separator via `ECDSAUpgradeable.toTypedDataHash(domainSeparator, structHash)`: [3](#0-2) 

The inconsistency confirms the omission in `Verifier.computeDigest` is a defect, not a deliberate design choice.

---

### Impact Explanation

The digest passed to `validateSignature` / `validateCompactSignature` contains no chain ID and no verifying contract address: [4](#0-3) 

A signature produced by a user on Chain A is byte-for-byte valid on Chain B, provided the user's nonce on Chain B matches the nonce embedded in the signed message. Because nonces start at zero for every new deployment and increment independently per chain, a fresh deployment on a second chain will share nonce state with the original chain for any user who has not yet transacted there. An attacker can:

- Replay a `WithdrawCollateral` signature to drain a user's collateral on the second chain.
- Replay a `LinkSigner` signature to hijack subaccount signing authority on the second chain.
- Replay a `LiquidateSubaccount` signature to force a liquidation on the second chain.

The corrupted state delta is: user collateral balance decremented, or linked signer overwritten, on a chain the user never intended to interact with.

---

### Likelihood Explanation

The protocol README explicitly targets "EVM-compatible chains" (plural), making multi-chain deployment the stated operational model. Any user who signs a transaction on the first deployed chain and has not yet used the same nonce on a second chain is immediately vulnerable upon that second deployment going live. No privileged access is required â€” any unprivileged attacker who observes a signed transaction on one chain can submit it on another.

---

### Recommendation

Replace the bare `hashStruct` return in every branch of `computeDigest` with a proper EIP-712 digest. Since `Verifier` already inherits `EIP712Upgradeable`, use `_hashTypedDataV4`:

```solidity
// Instead of returning hashStruct directly:
return _hashTypedDataV4(digest);
```

Or equivalently:

```solidity
return keccak256(
    abi.encodePacked(bytes2(0x1901), _domainSeparatorV4(), digest)
);
```

This requires changing `computeDigest` from `pure` to `view`. Off-chain signing code must be updated to use `eth_signTypedData_v4` with the matching domain (`name: "Nado"`, `version: "0.0.1"`, `chainId`, `verifyingContract`), consistent with how `OffchainExchange` already handles order signatures. [5](#0-4) 

---

### Proof of Concept

1. Protocol is deployed on Chain A (e.g., Arbitrum) and Chain B (e.g., Base). Both deployments share the same contract addresses via deterministic deployment.
2. Alice deposits collateral on Chain A and signs `WithdrawCollateral(sender=alice_subaccount, productId=0, amount=1000e18, nonce=0)`. The digest is `keccak256(abi.encode(TYPEHASH, alice_subaccount, 0, 1000e18, 0))` â€” no chain ID, no contract address.
3. Alice has never transacted on Chain B, so her nonce there is also `0`.
4. Attacker observes Alice's signed transaction on Chain A and submits the identical `(tx, signature)` bytes to `Endpoint` on Chain B.
5. `Verifier.computeDigest` on Chain B produces the identical digest (same inputs, no chain binding). `validateSignature` recovers Alice's address and accepts the signature.
6. Alice's collateral on Chain B is withdrawn to the `sendTo` address the attacker controls (or to Alice's address, but without her intent â€” and in `WithdrawCollateralV2` the `sendTo` field is attacker-chosen within the original signed message). [6](#0-5) [4](#0-3)

    function getDigest(uint32 productId, IEndpoint.Order memory order)
        public
        view
        returns (bytes32)
    {
        string
            memory structType = "Order(bytes32 sender,int128 priceX18,int128 amount,uint64 expiration,uint64 nonce,uint128 appendix)";

        bytes32 structHash = keccak256(
            abi.encode(
                keccak256(bytes(structType)),
                order.sender,
                order.priceX18,
                order.amount,
                order.expiration,
                order.nonce,
                order.appendix
            )
        );

        bytes32 domainSeparator = keccak256(
            abi.encode(
                _TYPE_HASH,
                _EIP712NameHash(),
                _EIP712VersionHash(),
                block.chainid,
                address(uint160(productId))
            )
        );

        return ECDSAUpgradeable.toTypedDataHash(domainSeparator, structHash);
    }
