### Title
`PriceVelocityGuardExtension` Per-Swap Reference-Price Update Lets Cumulative Block-Level Oracle Movement Exceed the Configured Cap — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` overwrites `lastMidPriceX64` with the current oracle mid on **every** swap before performing the velocity check. When multiple swaps execute in the same block the reference price is the previous swap's price, not the block-open price. Each individual swap satisfies the cap, but the cumulative oracle movement within the block is unbounded, defeating the guard's stated purpose.

---

### Finding Description

The NatSpec states the extension "Caps how fast the provided price can move **between blocks**" with allowed deviation `maxChangePerBlockE18 * sqrt(1 + blockDifference)`.

The implementation, however, unconditionally writes the new mid-price to storage before the check:

```solidity
// PriceVelocityGuardExtension.sol lines 57-58
s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);
``` [1](#0-0) 

When a second swap arrives in the same block, `blockDiff = block.number - prevBlock = 0`, so:

```
allowedSq = maxChange² × (1 + 0) = maxChange²
```

The check compares the current price only to the **previous swap's price** (already updated), not to the block-open price. Each hop of `maxChange` passes individually, but N swaps in the same block can move the oracle-derived mid by approximately `N × maxChange` in total — far beyond the intended single-block cap.

The pool fetches a fresh oracle quote on every `swap()` call:

```solidity
// MetricOmmPool.sol lines 228-240
(uint128 bidPriceX64, uint128 askPriceX64) = _getBidAndAskPriceX64();
_beforeSwap(msg.sender, recipient, zeroForOne, amountSpecified,
            priceLimitX64, packedSlot0Initial, bidPriceX64, askPriceX64, extensionData);
``` [2](#0-1) 

For push-based oracles (Pyth Lazer, Chainlink Data Streams) a new signed price report can be submitted on-chain between transactions in the same block, so the oracle mid genuinely changes between swaps. Each submission + swap pair satisfies the per-hop cap while the aggregate movement is unconstrained.

---

### Impact Explanation

The velocity guard is the LP-protection layer against rapid oracle price manipulation. If bypassed, a sophisticated actor can walk the oracle mid across many small steps within one block, executing swaps at each step at prices that are increasingly unfavorable to LPs. Because the pool settles every swap at the live oracle price, LPs bear the loss on each leg. The cumulative drain is proportional to the number of hops and the per-hop cap, and is limited only by available liquidity and oracle update throughput — not by the configured `maxChangePerBlockE18`.

---

### Likelihood Explanation

Pyth Lazer and Chainlink Data Streams are push-based: any party holding a valid signed price report can submit it on-chain. A block builder or MEV searcher can bundle multiple (price-update, swap) pairs in a single block. No privileged role is required beyond holding valid oracle reports, which are publicly available from the respective off-chain infrastructure. The pool's reentrancy guard uses transient storage and resets between transactions, so multiple swaps per block are explicitly supported.

---

### Recommendation

Track the block-open price separately from the per-swap reference. Only update `lastMidPriceX64` when `block.number > lastUpdateBlock` (i.e., on the first swap of a new block). For all subsequent swaps within the same block, compare against the block-open price and leave `lastMidPriceX64` unchanged until the next block.

```solidity
if (block.number > prevBlock) {
    // first swap of this block — update reference
    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);
}
// velocity check always uses the block-open reference (prevMid)
```

This ensures the guard enforces a true per-block cap regardless of how many swaps occur within the block.

---

### Proof of Concept

Setup: pool with `PriceVelocityGuardExtension`, `maxChangePerBlockE18 = 0.01e18` (1 % per block).

1. Oracle mid starts at `P₀ = 1.000`.
2. Attacker submits Pyth report → oracle mid = `P₁ = 1.010` (+1 %).
3. Attacker calls `pool.swap(...)` — `beforeSwap` checks `|P₁−P₀|/P₀ = 1 % ≤ 1 %` ✓, writes `lastMidPriceX64 = P₁`.
4. Attacker submits next Pyth report → oracle mid = `P₂ = 1.0201` (+1 % from `P₁`).
5. Attacker calls `pool.swap(...)` — `beforeSwap` checks `|P₂−P₁|/P₁ = 1 % ≤ 1 %` ✓, writes `lastMidPriceX64 = P₂`.
6. Repeat 10 times in the same block.

Result: oracle mid has moved from 1.000 to `(1.01)^10 ≈ 1.105` (+10.5 %) within a single block, while the guard allowed each individual hop. LPs traded against a price that drifted 10× the configured cap, with the excess loss extracted by the attacker. [3](#0-2)

    // casting to `uint128` is safe: geometric mid of two uint128 bid/ask quotes fits uint128 (same bound as pool)

    PriceVelocityState storage s = priceVelocityState[pool_];
    uint128 prevMid = s.lastMidPriceX64;

    s.lastMidPriceX64 = midPrice;

    if (prevMid != 0) {
      uint64 maxChange = s.maxChangePerBlockE18;
      if (maxChange != 0) {
        uint256 blockDiff = block.number - prevBlock;

        uint256 delta = midPrice > prevMid ? uint256(midPrice - prevMid) : uint256(prevMid - midPrice);

        uint256 changeE18 = (delta * 1e18) / uint256(prevMid);

        uint256 actualSq = changeE18 * changeE18;
        uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);

        if (actualSq > allowedSq) {
          revert PriceVelocityExceeded(actualSq, allowedSq);
        }
      }

  }
