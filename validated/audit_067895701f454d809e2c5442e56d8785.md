### Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates on `sender`, which resolves to `msg.sender` of the pool's `swap` call. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the actual user. If the pool admin allowlists the router to enable router-mediated swaps for legitimate users, any unprivileged user can bypass the swap allowlist by routing through the router.

---

### Finding Description

In `SwapAllowlistExtension.beforeSwap`:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (the extension is called by the pool via `ExtensionCalling._callExtensionsInOrder`). `sender` is the first argument forwarded by the pool — specifically `msg.sender` of the pool's own `swap` call. [1](#0-0) 

The pool's `swap` function passes `msg.sender` as `sender` to `_beforeSwap`: [2](#0-1) 

`ExtensionCalling._beforeSwap` then encodes that `sender` and forwards it to the extension: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)`, the pool sees `msg.sender = router`. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. [4](#0-3) 

This is structurally asymmetric with `DepositAllowlistExtension`, which correctly gates on `owner` — the economic actor explicitly passed by the liquidity adder — rather than `sender` (the direct caller). The deposit allowlist correctly identifies the beneficiary regardless of which contract calls the pool: [5](#0-4) 

The swap allowlist has no equivalent mechanism. There is no way for the router to forward the actual user's identity to the pool's `swap` function, because the pool's `swap` signature has no explicit `swapper` parameter — it always uses `msg.sender`.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC'd users, institutional partners, or whitelisted market makers) can be fully bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. The non-allowlisted user executes swaps against LP funds that were intended to be protected by the allowlist. This is a direct broken-invariant impact: LP assets are exposed to unauthorized counterparties, and the pool's core access-control functionality is rendered ineffective for any pool that supports router-mediated swaps.

---

### Likelihood Explanation

The bypass requires the pool admin to have allowlisted the router address. This is a natural and expected configuration for any pool that wants to support router-mediated swaps for legitimate users — without it, no user can swap through the router at all. Once the router is allowlisted (a common, reasonable admin action), the bypass is reachable by any unprivileged user with no special permissions or capital requirements beyond the swap amount. Likelihood is **Medium**: conditional on the router being allowlisted, but that condition is a standard operational requirement.

---

### Recommendation

Gate on the actual user identity rather than the direct caller. Two viable approaches:

1. **Extension-data forwarding**: The router encodes the original `msg.sender` into `extensionData` for each hop, and `SwapAllowlistExtension` decodes and checks that address instead of `sender`. The extension must verify the forwarded address is authentic (e.g., by trusting only known router contracts).
2. **Explicit swapper parameter**: Add an explicit `swapper` parameter to the pool's `swap` interface that the router populates with the actual user, and forward it to extensions as the gated identity.

As a minimum, document clearly that allowlisting the router grants unrestricted access to all router users, and advise pool admins not to combine per-user allowlisting with router allowlisting.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Admin allowlists user A: `setAllowedToSwap(pool, userA, true)`.
3. Admin allowlists the router to enable router-mediated swaps for user A: `setAllowedToSwap(pool, router, true)`.
4. User B (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, ...)` — pool sees `msg.sender = router`.
6. Pool calls `_beforeSwap(router, ...)` — extension checks `allowedSwapper[pool][router]` → **passes**.
7. User B successfully swaps on the curated pool, bypassing the allowlist entirely. [1](#0-0) [6](#0-5) [3](#0-2)
