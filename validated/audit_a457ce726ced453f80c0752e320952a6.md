### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Allowing Merkle Proof Forgery That Bypasses the Coinbase-Proof Protection of `verify_transaction_inclusion_v2` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to close the 64-byte Merkle-proof forgery vector documented at https://www.bitmex.com/blog/64-Byte-Transactions. Its two guards — a coinbase-proof check and a proof-length equality check — are both absent from the original `verify_transaction_inclusion` (v1), which remains a live, unpermissioned public entry point. Any NEAR caller can invoke v1 directly, supply an internal Merkle-tree node hash as `tx_id`, and receive `true` — a forged proof of transaction inclusion — without ever touching v2's protections. This is the direct analog of the Frankencoin double-entrypoint pattern: a "legacy" entry point that delegates to the same underlying resource while bypassing the guard that was added to the "new" entry point.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces two invariants before delegating to v1:

1. **Length equality**: `args.merkle_proof.len() == args.coinbase_merkle_proof.len()` — this forces the attacker's proof to span the full tree depth, preventing a shorter proof for an internal node.
2. **Coinbase proof**: `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof) == header.block_header.merkle_root` — this anchors the tree depth to a real transaction at index 0. [1](#0-0) 

Neither guard exists in v1: [2](#0-1) 

The contract's own documentation acknowledges the gap:

> This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. [3](#0-2) 

The `#[deprecated]` Rust attribute is a compile-time warning only; it imposes no runtime restriction. The function carries `#[pause]` but is not paused by default, so it is reachable by any unprivileged NEAR account.

The broken invariant: `verify_transaction_inclusion` is supposed to verify that `tx_id` is a real leaf transaction in the block's Merkle tree. Because it only checks `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.block_header.merkle_root`, it cannot distinguish a leaf hash from an internal-node hash. An attacker who supplies an internal node at depth D needs only a proof of length `(tree_depth − D)`, which is shorter than the full-depth proof v2 would require — but v1 imposes no depth constraint at all.

The `merkle_tools::compute_root_from_merkle_proof` function is a pure hash-chain computation with no leaf/node distinction: [4](#0-3) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) — or that accepts a caller-chosen function selector — can be made to accept a forged proof. The corrupted value is the **proof result**: the function returns `true` for a `tx_id` that is an internal Merkle-tree node, not a real Bitcoin transaction. In cross-chain bridge or SPV-payment scenarios this allows an attacker to falsely prove that an arbitrary "transaction" was confirmed in a real Bitcoin block, enabling theft or double-spend fraud at the application layer.

---

### Likelihood Explanation

The attack requires only public information: the Merkle tree of any mainchain Bitcoin block (available from any block explorer or full node). No privileged role, private key, or social engineering is needed. The attacker calls a public NEAR function with crafted arguments. Likelihood is **medium-high** for any application that has not independently enforced use of v2.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI, or gate it behind a role that no external caller holds. If backward compatibility must be preserved, replicate v2's coinbase-proof and length-equality checks inside v1 so both entry points enforce the same invariant.

---

### Proof of Concept

Given a mainchain Bitcoin block `B` with four transactions `[T1, T2, T3, T4]` and Merkle root `R = H(H(T1,T2), H(T3,T4))`:

1. Compute the internal node `N = H(T1, T2)`.
2. Observe that `compute_root_from_merkle_proof(N, 0, [H(T3,T4)]) == R` — a one-step proof from the internal node reaches the root.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (internal node, not a real transaction)
   - `tx_block_blockhash = B`'s hash (a real mainchain block)
   - `tx_index = 0`
   - `merkle_proof = [H(T3,T4)]` (length 1, one level shorter than the full depth-2 proof)
   - `confirmations = 1`
4. The mainchain check passes because `B` is on the mainchain. [5](#0-4) 
5. The Merkle check passes because `H(N, H(T3,T4)) == R`. [6](#0-5) 
6. The function returns `true` — a forged proof that the fake "transaction" `N` was confirmed in block `B`.

Calling `verify_transaction_inclusion_v2` with the same inputs would fail at step 1 (length mismatch: `merkle_proof.len()=1 ≠ coinbase_merkle_proof.len()=2`) or step 2 (coinbase proof would not match the root with a depth-1 proof). The legacy entry point bypasses both guards, exactly mirroring the Frankencoin double-entrypoint pattern where the legacy token address bypassed the collateral-withdrawal guard.

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
