### Title
Zero-Signer Bypass in `requireValidTxSignatures` Allows Sequencer Impersonation Before Any Signers Are Registered — (`core/contracts/Verifier.sol`)

---

### Summary

`Verifier.requireValidTxSignatures` enforces sequencer authorization by checking that the number of valid signatures equals `nSigner`. When `nSigner == 0` (no sequencer public keys registered, or all deleted), the check `require(nSignatures == nSigner)` passes trivially with an empty `signatures` array, allowing any caller to bypass sequencer authentication entirely.

---

### Finding Description

In `Verifier.sol`, the function `requireValidTxSignatures` validates that a submitted transaction batch was signed by the required number of registered sequencer signers: [1](#0-0) 

The quorum enforcement is:

```solidity
require(nSignatures == nSigner, "not enough signatures");
``` [2](#0-1) 

`nSigner` is a storage variable that starts at `0` and is incremented only when a non-zero pubkey is assigned via `_assignPubkey`: [3](#0-2) 

The `initialize` function conditionally assigns pubkeys — only if `!isPointNone(initialSet[i])`. If the contract is initialized with all-zero points, `nSigner` remains `0`: [4](#0-3) 

Similarly, `deletePubkey` can reduce `nSigner` back to `0` if all signers are removed: [5](#0-4) 

When `nSigner == 0`, an attacker passes `signatures = []` (empty array). The loop body never executes, `nSignatures` stays `0`, and `require(0 == 0)` passes unconditionally. The attacker has fully bypassed sequencer authentication.

This is structurally identical to the Olympus H-02 pattern: a threshold check of the form `actual == total` passes trivially when `total == 0`, enabling unauthorized protocol actions before the system is properly initialized.

Note: the parallel `checkQuorum` path used by `requireValidSignature` is **not** vulnerable to this — it returns `nSigned * 2 > nSigner`, which evaluates to `0 > 0 = false` when both are zero, and also short-circuits to `false` if any bit in the bitmask maps to a zero pubkey: [6](#0-5) 

The vulnerability is isolated to the `requireValidTxSignatures` path.

---

### Impact Explanation

`requireValidTxSignatures` is the authorization gate for sequencer-submitted transaction batches processed by the Endpoint. Bypassing it allows an attacker to submit arbitrary sequencer transactions. Transactions that do not require individual user signatures — such as `UpdatePrice`, `SpotTick`, `PerpTick`, and `SettlePnl` — can be submitted freely. An attacker can:

1. Submit a manipulated `UpdatePrice` transaction to set oracle prices to extreme values.
2. Trigger liquidations of healthy subaccounts at the manipulated price, stealing collateral.
3. Corrupt protocol accounting state (PnL settlement, funding rates) irreversibly.

This constitutes a direct path to loss of user funds held in the protocol.

---

### Likelihood Explanation

The window exists during two realistic scenarios:

1. **Deployment race**: The Verifier is deployed and initialized with all-zero pubkeys (e.g., keys are assigned in a separate transaction). An attacker monitoring the mempool can front-run the `assignPubKey` call.
2. **Key rotation**: An operator calls `deletePubkey` for all registered signers before assigning new ones, transiently setting `nSigner = 0`.

Both scenarios are realistic operational states, not theoretical-only conditions.

---

### Recommendation

Add an explicit guard in `requireValidTxSignatures` to reject calls when no signers are registered:

```solidity
require(nSigner > 0, "no signers registered");
require(nSignatures == nSigner, "not enough signatures");
```

Additionally, the `initialize` function should enforce that at least one valid (non-zero) pubkey is provided:

```solidity
require(nSigner > 0, "must initialize with at least one signer");
```

---

### Proof of Concept

```solidity
// Before any pubkeys are assigned (nSigner == 0):
// 1. Deploy Verifier with all-zero initialSet
// 2. Call requireValidTxSignatures with empty signatures array

bytes[] memory emptySigs = new bytes[](0);
bytes memory txData = abi.encode(/* UpdatePrice transaction */);

// This call succeeds — nSignatures (0) == nSigner (0)
verifier.requireValidTxSignatures(txData, 0, emptySigs);

// Attacker can now submit arbitrary sequencer transactions to Endpoint,
// including UpdatePrice to manipulate oracle prices and trigger liquidations.
``` [7](#0-6)

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }
