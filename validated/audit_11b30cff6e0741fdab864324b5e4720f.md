### Title
Unchecked u64 Multiplication Overflow in `op_unknown` Cost Multiplier Produces Falsely-Small Returned Cost — (`File: src/more_ops.rs`)

---

### Summary

`op_unknown` in `src/more_ops.rs` applies a final cost multiplier via `cost *= cost_multiplier + 1` without overflow protection. In Rust release builds, u64 overflow wraps silently. When an attacker crafts an unknown opcode with `cost_multiplier = u32::MAX` and supplies arguments that drive the pre-multiplication cost above `u32::MAX`, the product wraps to a value ≤ `u32::MAX`, bypasses the post-multiplication rejection guard, and the function returns a falsely-small (potentially zero) cost to the interpreter. This breaks the cost-accounting invariant that `run_program` relies on to enforce the block cost limit.

---

### Finding Description

In `op_unknown`, the cost is computed in three phases:

**Phase 1 — argument-dependent cost (cost_function 1/2/3):** [1](#0-0) 

Each branch accumulates `cost` and calls `check_cost(cost, max_cost)?` inside the loop, so `cost` is bounded by `max_cost` at every step.

**Phase 2 — final pre-multiplication guard:** [2](#0-1) 

`check_cost(cost, max_cost)?` passes because `cost ≤ max_cost`. The `assert!(cost > 0)` also passes because the cost is at least the base constant.

**Phase 3 — unchecked multiplication and post-multiplication guard:** [3](#0-2) 

```rust
cost *= cost_multiplier + 1;          // ← no overflow check
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is derived from the opcode prefix bytes via `u32_from_u8`, so its maximum value is `u32::MAX = 4 294 967 295`: [4](#0-3) 

`cost_multiplier + 1` therefore reaches `2^32`. In Rust release mode, `u64` arithmetic wraps on overflow (two's complement). When `cost > u32::MAX` (achievable because Chia's block cost limit is ~11 billion, well above `u32::MAX`), the product `cost × 2^32` overflows `u64`. If the wrapped result is ≤ `u32::MAX`, the rejection guard is bypassed and the function returns `Ok(Reduction(wrapped_cost, nil))` — a cost far smaller than the true cost, or even zero.

`Cost` is defined as `u64`: [5](#0-4) 

The run_program loop accumulates the returned cost directly: [6](#0-5) 

A returned cost of 0 does not advance the cost counter, allowing the attacker to include arbitrarily many such unknown-op invocations within the block cost limit.

**Concrete overflow condition:**

For `cost_multiplier = u32::MAX` (opcode prefix bytes all `0xff`, last byte with cost_function bits set):

```
cost × 2^32  mod  2^64  =  (cost mod 2^32) × 2^32
```

For the result to be ≤ `u32::MAX`, the attacker needs `cost mod 2^32 = 0`, i.e., `cost` must be a multiple of `2^32`. With `cost_function = 2` (MUL-like), cost grows quadratically with the number of arguments (`l0 += l1` accumulates across iterations). With ~37 800 one-byte arguments, cost reaches ~`4.3 × 10^9 > u32::MAX`. By tuning argument count and sizes, the attacker can align `cost` to a multiple of `2^32`.

---

### Impact Explanation

The cost model is the sole DoS defence in the CLVM interpreter. `run_program` enforces `max_cost` by accumulating the costs returned by each operator: [7](#0-6) 

If `op_unknown` returns 0 (or any value far below the true cost), the attacker can embed many unknown-op invocations in a single program without exhausting the cost budget. Because `op_unknown` is a no-op (always returns `nil`), the attacker gains no computational result, but the node is forced to execute the full argument-cost loop for each invocation while the cost counter barely advances. This is a **cost-undercharge / resource-exhaustion** vulnerability: the node performs real work (iterating over thousands of arguments) that is not reflected in the charged cost, enabling a block that should be rejected as too expensive to be accepted and replayed across all validating nodes.

---

### Likelihood Explanation

- **Lenient mode is the production consensus path.** `op_unknown` is only reachable when `allow_unknown_ops()` returns true, which is the case for Chia's mempool and block validation.
- **Chia's block cost limit (~11 billion) is well above `u32::MAX` (~4.3 billion).** Phase-2 `check_cost` therefore permits `cost` values that cause overflow in Phase 3.
- **The attacker controls both the opcode bytes and the argument list.** The opcode bytes set `cost_multiplier = u32::MAX` and `cost_function = 2`; the argument list tunes `cost` to a multiple of `2^32`. Both are fully attacker-controlled inputs in any CLVM program submitted to the network.
- **No special privilege is required.** Any user who can submit a transaction can trigger this path.

---

### Recommendation

Replace the bare multiplication with a checked or saturating variant and treat overflow as an invalid opcode:

```rust
// Before (vulnerable):
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}

// After (safe):
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}
```

This mirrors the pattern already used in `serialized_length_serde_2026`, which uses `checked_mul` and `checked_add` throughout: [8](#0-7)

### Proof of Concept

```
Opcode bytes: [0xff, 0xff, 0xff, 0xff, 0x80]
  → cost_function  = (0x80 >> 6) & 0b11 = 2   (MUL-like)
  → cost_multiplier = u32_from_u8([0xff,0xff,0xff,0xff]) = u32::MAX = 4294967295
  → cost_multiplier + 1 = 2^32

Arguments: ~37 800 one-byte atoms
  → cost_function=2 loop accumulates cost ≈ 4 294 967 296 = 2^32
  → check_cost(2^32, 11_000_000_000) passes (2^32 < 11e9)

Multiplication (release mode, wrapping):
  cost = 2^32
  cost *= 2^32
  cost = 2^64 mod 2^64 = 0

Post-multiplication guard:
  0 > u32::MAX  →  false
  → returns Ok(Reduction(0, nil))

Effect in run_program:
  total_cost += 0   // cost budget not consumed
  // attacker repeats this in a loop; each iteration costs 0
  // node performs O(37 800) work per invocation for free
```
