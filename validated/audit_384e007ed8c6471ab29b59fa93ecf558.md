### Title
`ValidResourceBounds` variant silently mutated by protobuf round-trip causes transaction hash divergence across P2P sync boundary — (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a **value-based heuristic** to reconstruct the enum variant. An `AllResources` V3 transaction whose `l2_gas` and `l1_data_gas` bounds are both zero is silently reconstructed as `L1Gas` after a protobuf round-trip. Because `get_tip_resource_bounds_hash` produces a structurally different hash preimage for `L1Gas` (2 resource elements) versus `AllResources` (3 resource elements), the transaction hash computed on the originating node diverges from the hash that would be computed on any node that receives the transaction via P2P block sync.

---

### Finding Description

**Step 1 — The value-based heuristic in the protobuf deserializer**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` decides the variant purely from the wire values:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

There is no explicit type tag in the protobuf message. The variant is inferred from whether the numeric fields are zero. The serializer for `ValidResourceBounds::AllResources` emits all three bounds including zero-valued ones:

```rust
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),
    },
``` [2](#0-1) 

When `l2_gas = {0, 0}` and `l1_data_gas = {0, 0}`, the serialized bytes are indistinguishable from a `L1Gas` transaction. The deserializer therefore reconstructs `L1Gas` instead of `AllResources`.

**Step 2 — The hash function branches on the variant**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` produces a different number of hash elements depending on the variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements total
    }
});
``` [3](#0-2) 

For `AllResources` with zero `l2_gas` and `l1_data_gas`, the hash preimage is:

```
Poseidon([tip, concat(L1_GAS, A, P), concat(L2_GAS, 0, 0), concat(L1_DATA_GAS, 0, 0)])
```

After the protobuf round-trip reconstructs `L1Gas`, the hash preimage becomes:

```
Poseidon([tip, concat(L1_GAS, A, P), concat(L2_GAS, 0, 0)])
```

These are structurally different Poseidon inputs and produce different digests H1 ≠ H2, even though all numeric field values are identical.

**Step 3 — The P2P block sync path carries the mutated variant into storage**

In `crates/apollo_p2p_sync/src/client/transaction.rs`, the P2P sync client deserializes `FullTransaction` objects from `protobuf::TransactionWithReceipt`. The `Transaction` object is reconstructed via `TryFrom<protobuf::TransactionInBlock>`, which calls `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` for V3 transactions. The transaction hash is taken directly from the peer without recomputation:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [4](#0-3) 

The stored `Transaction` object therefore carries `ValidResourceBounds::L1Gas` while the stored `transaction_hash` is H1 (the `AllResources` hash). These two are permanently inconsistent in storage.

**Step 4 — Downstream hash recomputation produces the wrong value**

`validate_transaction_hash` recomputes the hash from the stored `Transaction`:

```rust
possible_hashes.push(get_transaction_hash(transaction, chain_id, transaction_options)?);
Ok(possible_hashes.contains(&expected_hash))
``` [5](#0-4) 

Because the stored `Transaction` has `L1Gas` variant, `get_transaction_hash` computes H2. The expected hash H1 is not in `possible_hashes`, so `validate_transaction_hash` returns `false` for a valid transaction.

The same divergence affects any RPC path that recomputes the hash from the stored `Transaction`, and any consensus validation that calls `Transaction::calculate_transaction_hash`.

---

### Impact Explanation

**Impact: High — Transaction conversion or signature/hash logic binds the wrong hash type or executable payload.**

A valid `AllResources` V3 transaction with zero `l2_gas` and `l1_data_gas` bounds is permanently misclassified as a pre-0.13.3 `L1Gas` transaction on every node that receives it via P2P block sync. The stored `Transaction` object has the wrong variant, causing:

1. `validate_transaction_hash` to return `false` for a valid transaction — any node that later validates the stored transaction against its stored hash will reject it.
2. Any RPC endpoint that recomputes the hash from the stored `Transaction` (e.g., for tracing or simulation) returns an authoritative-looking wrong hash H2 instead of H1.
3. The `resource_bounds` type reported by `starknet_getTransactionByHash` is `L1Gas` instead of `AllResources`, misrepresenting the transaction's protocol version to clients.

---

### Likelihood Explanation

The trigger requires submitting a V3 `AllResources` transaction with `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0`. This is a valid and accepted transaction — the gateway accepts `AllResourceBounds` with zero values. Any user who submits such a transaction (e.g., a transaction that only consumes L1 gas and sets the other bounds to zero) triggers the bug on every syncing node. No privileged access is required.

---

### Recommendation

Replace the value-based heuristic with an explicit type tag in the protobuf schema, or preserve the variant information through a separate boolean/enum field. At minimum, the deserializer should reconstruct `AllResources` whenever `l1_data_gas` is present in the wire message (even if zero), reserving `L1Gas` only for messages where `l1_data_gas` is absent (the pre-0.13.3 wire format):

```rust
// Current (broken):
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else { ... })

// Fixed: use presence of l1_data_gas field, not its value
let l1_data_gas_present = value.l1_data_gas.is_some();
let l1_data_gas = value.l1_data_gas.unwrap_or_default().try_into()?;
Ok(if !l1_data_gas_present && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [6](#0-5) 

Additionally, the TODO comment in the P2P sync client should be resolved — transaction hashes received from peers should be validated against the locally recomputed hash before storage. [4](#0-3)

### Proof of Concept

```
1. Construct a V3 Invoke transaction with:
   - resource_bounds = AllResources {
       l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
     }
   - tip = 0

2. Submit via RPC. The gateway calls convert_rpc_tx_to_internal(), which calls
   InternalRpcTransactionWithoutTxHash::calculate_transaction_hash() using
   AllResourceBounds. get_tip_resource_bounds_hash() hashes:
     Poseidon([0, concat(L1_GAS,1000,1), concat(L2_GAS,0,0), concat(L1_DATA_GAS,0,0)])
   → H1 (4-element Poseidon input)

3. Transaction is included in a block and served via P2P sync as
   protobuf::TransactionInBlock with resource_bounds:
     { l1_gas: {1000,1}, l2_gas: {0,0}, l1_data_gas: {0,0} }

4. Receiving node deserializes via TryFrom<protobuf::ResourceBounds>:
   l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → reconstructs ValidResourceBounds::L1Gas({1000,1})

5. If the receiving node calls Transaction::calculate_transaction_hash():
   get_tip_resource_bounds_hash() hashes:
     Poseidon([0, concat(L1_GAS,1000,1), concat(L2_GAS,0,0)])
   → H2 (3-element Poseidon input)

6. H1 ≠ H2. validate_transaction_hash(tx, block_number, chain_id, H1) returns false.
   The valid transaction is permanently misclassified on the receiving node.
```
