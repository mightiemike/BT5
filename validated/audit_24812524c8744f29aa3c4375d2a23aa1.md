### Title
`AllResourceBounds` with zero L2/L1DataGas silently coerced to `L1Gas` variant during protobuf round-trip, causing consensus transaction rejection ŌĆö (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion uses a **value-based heuristic** to reconstruct the `ValidResourceBounds` variant. When a transaction carries `AllResourceBounds` with zero `l2_gas` and `l1_data_gas`, the protobuf round-trip silently coerces the type to `ValidResourceBounds::L1Gas`. The subsequent conversion from `InvokeTransactionV3` back to `RpcInvokeTransactionV3` then fails with `DEPRECATED_RESOURCE_BOUNDS_ERROR`, causing every validator to reject a transaction the proposer accepted ŌĆö a proposer/validator split that permanently drops the transaction from the block.

---

### Finding Description

The protobuf `ResourceBounds` message carries no discriminant field for the `ValidResourceBounds` Rust enum variant. The deserializer at line 431 infers the variant purely from the values:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs:417-436
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // ...
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // silently defaults to zero
        // ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)                      // ŌåÉ wrong variant
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
``` [1](#0-0) 

An `RpcInvokeTransactionV3` always carries `AllResourceBounds` (not `ValidResourceBounds`):

```rust

When a `ConsensusTransaction` containing such a transaction (with `l2_gas = 0, l1_data_gas = 0`) is serialized to protobuf and deserialized by a validator, the path is:

```
protobuf::InvokeV3WithProof
  ŌåÆ InvokeTransactionV3  (resource_bounds = ValidResourceBounds::L1Gas ŌåÉ wrong)
  ŌåÆ RpcInvokeTransactionV3::try_from(snapi_invoke)  ŌåÉ FAILS
```

The conversion from `InvokeTransactionV3` to `RpcInvokeTransactionV3` explicitly rejects non-`AllResources` bounds:

```rust
// crates/apollo_protobuf/src/converters/rpc_transaction.rs:115-131
impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        let snapi_invoke: InvokeTransactionV3 = value.invoke...?.try_into()?;
        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
}
``` [3](#0-2) 

The proposer computes the transaction hash from `InternalRpcTransactionWithoutTxHash` (which always uses `AllResourceBounds`) and includes the transaction in the proposal. Every validator deserializes the same protobuf bytes, hits `DEPRECATED_RESOURCE_BOUNDS_ERROR`, and rejects the transaction ŌĆö a permanent proposer/validator split on that transaction.

The `get_tip_resource_bounds_hash` function also produces a **different hash** for `L1Gas` (2 resource entries) vs `AllResources` (3 resource entries), so even if the conversion did not fail, the hash computed by the validator would diverge from the one stored in `InternalRpcTransaction::tx_hash`:

```rust

---

### Impact Explanation

A transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is accepted by the proposer's gateway and mempool, included in a block proposal, and serialized to protobuf. Every validator deserializes it, reconstructs the wrong `ValidResourceBounds::L1Gas` variant, and fails the `RpcInvokeTransactionV3` conversion. The transaction is permanently dropped. If the proposer's `ProposalFin` commitment includes this transaction's hash, validators will reject the entire proposal, causing a consensus round failure.

This matches: **High ŌĆö Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

---

### Likelihood Explanation

The trigger requires a transaction with `AllResourceBounds` where both `l2_gas.is_zero()` and `l1_data_gas.is_zero()` are true (both `max_amount = 0` and `max_price_per_unit = 0`). The stateful validator's `validate_resource_bounds` check would normally reject such a transaction because `l2_gas.max_price_per_unit = 0 < threshold`. However, the check is skipped when:

1. `validate_resource_bounds = false` (explicitly supported bootstrap/testing configuration), or
2. `min_gas_price_percentage = 0` (making `threshold = 0`, so `0 < 0` is false and the check passes). [5](#0-4) [6](#0-5) 

Both configurations are reachable in production deployments (bootstrap phase, or a node operator setting `min_gas_price_percentage = 0`).

---

### Recommendation

Add an explicit discriminant to the protobuf `ResourceBounds` message (e.g., a boolean `is_all_resources` field or a `oneof` wrapper) so the `ValidResourceBounds` variant is preserved across serialization. Alternatively, reject `AllResourceBounds` with zero `l2_gas` and `l1_data_gas` at the gateway stateless validator unconditionally, independent of the `validate_resource_bounds` config flag, so the ambiguous case never reaches consensus.

---

### Proof of Concept

1. Configure the gateway with `validate_resource_bounds = false` (bootstrap mode).
2. Submit an invoke V3 transaction with:
   ```
   AllResourceBounds {
     l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
     l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },  // is_zero() = true
     l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },  // is_zero() = true
   }
   ```
3. Gateway accepts it; mempool queues it; proposer includes it in a block proposal.
4. Proposer serializes `ConsensusTransaction` to protobuf: `l2_gas = {0,0}`, `l1_data_gas = absent/zero`.
5. Validator deserializes: `l1_data_gas.is_zero() && l2_gas.is_zero()` ŌåÆ `ValidResourceBounds::L1Gas`.
6. `InvokeTransactionV3 ŌåÆ RpcInvokeTransactionV3` conversion fails: `DEPRECATED_RESOURCE_BOUNDS_ERROR`.
7. Validator rejects the transaction; proposer/validator split on the block content; consensus round fails. [7](#0-6) [8](#0-7)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
