### Title
Missing Zero-Address Check for `_quote` Token in `SpotEngine.initialize()` — (File: core/contracts/SpotEngine.sol)

---

### Summary

`SpotEngine.initialize()` assigns the `_quote` parameter directly to `configs[QUOTE_PRODUCT_ID].token` with no zero-address guard. If the deployer passes `address(0)` for `_quote`, the quote token slot is permanently set to zero for the lifetime of that initialization, making all protocol deposits and withdrawals non-functional until an owner-level corrective action is taken.

---

### Finding Description

In `SpotEngine.initialize()`, the `_quote` address is written unconditionally into the `Config` struct for `QUOTE_PRODUCT_ID`:

```solidity
// core/contracts/SpotEngine.sol L23-31
configs[QUOTE_PRODUCT_ID] = Config({
    token: _quote,          // ← no require(_quote != address(0))
    interestInflectionUtilX18: 8e17,
    interestFloorX18: 1e16,
    interestSmallCapX18: 4e16,
    interestLargeCapX18: ONE,
    withdrawFeeX18: ONE,
    minDepositRateX18: 0
});
```

The `initializer` guard lives inside `BaseEngine._initialize()`, which is called on the same transaction. Once that flag is set, `SpotEngine.initialize()` cannot be called again. There is no `require(_quote != address(0))` anywhere in the call path, and `_quote` is not validated in `BaseEngine._initialize()` either (which does not receive `_quote` at all).

The `_quote` token is the sole quote-collateral token (USDC) for the entire spot engine. It is the only token address set at initialization time for `QUOTE_PRODUCT_ID`; every subsequent spot product references it as the settlement asset.

---

### Impact Explanation

If `_quote == address(0)` at initialization time:

1. **Deposits fail.** `Endpoint.depositCollateralWithReferral()` calls `spotEngine.getToken(productId)`, which returns `address(0)` for `QUOTE_PRODUCT_ID`. The subsequent `handleDepositTransfer(IERC20Base(address(0)), msg.sender, amount)` call targets the zero address and reverts. No user can deposit collateral.

2. **Withdrawals fail.** `BaseWithdrawPool.getToken()` contains `require(address(token) != address(0))`, which reverts immediately when the stored token is zero. No user can withdraw collateral.

3. **Insurance operations fail.** `ContractOwner.depositInsurance()` calls `spotEngine.getToken(QUOTE_PRODUCT_ID)`, receives `address(0)`, and the subsequent `approve` call reverts.

The entire protocol is non-functional for collateral movement. Recovery requires the owner to call `SpotEngine.addOrUpdateProduct(QUOTE_PRODUCT_ID, ...)` with the correct token address (this path is available because `QUOTE_PRODUCT_ID` is already registered, so `isNewProduct == false` and the `require(productId != QUOTE_PRODUCT_ID)` guard inside the `if (isNewProduct)` block is not reached). This means full redeployment is not strictly required, but the protocol is completely halted until that corrective owner transaction is mined.

---

### Likelihood Explanation

The trigger is a deployment-time mistake: the deployer passes `address(0)` for `_quote`. This is a low-probability event in a careful deployment, but the absence of any on-chain guard means there is no safety net. Given that `Clearinghouse.initialize()` also accepts a `_quote` parameter (set as `quote = _quote` with no zero check), a copy-paste or scripting error during a multi-contract deployment sequence could plausibly produce this state. Likelihood is **Low**, but the consequence is a fully halted protocol.

---

### Recommendation

Add an explicit zero-address guard at the top of `SpotEngine.initialize()` before the `_initialize` call:

```solidity
function initialize(
    address _clearinghouse,
    address _offchainExchange,
    address _quote,
    address _endpoint,
    address _admin
) external {
    require(_quote != address(0), "SpotEngine: zero quote token");
    _initialize(_clearinghouse, _offchainExchange, _endpoint, _admin);
    configs[QUOTE_PRODUCT_ID] = Config({ token: _quote, ... });
    ...
}
```

The same pattern should be applied to `Clearinghouse.initialize()` for its `_quote` parameter.

---

### Proof of Concept

1. Deploy `SpotEngine` behind a proxy.
2. Call `SpotEngine.initialize(clearinghouse, offchainExchange, address(0), endpoint, admin)`.
3. Attempt `Endpoint.depositCollateral(subaccountName, QUOTE_PRODUCT_ID, amount)`.
4. The call reaches `handleDepositTransfer(IERC20Base(address(0)), msg.sender, amount)` and reverts — no user can deposit.
5. Attempt `BaseWithdrawPool.submitWithdrawal(...)` — `getToken(QUOTE_PRODUCT_ID)` triggers `require(address(token) != address(0))` and reverts — no user can withdraw.
6. The protocol remains halted until the owner calls `spotEngine.addOrUpdateProduct(QUOTE_PRODUCT_ID, quoteId, sizeIncrement, minSize, correctConfig, riskStore)` to overwrite the zero token address. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
