### Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the **router address**, not the end user. If the pool admin allowlists the router (the natural configuration for a curated pool that still wants to support router-mediated swaps), every unprivileged user can bypass the allowlist by routing through the router.

---

### Finding Description

**Pool → Extension argument binding**

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← router address when called via MetricOmmSimpleRouter
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension:

```solidity
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, ...)
)
```

**SwapAllowlistExtension check**

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When the path is `user → router → pool`, `sender` equals the router address. The check therefore asks: *"Is the router allowlisted for this pool?"* — not *"Is the end user allowlisted?"*

**Router call path**

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly with no identity forwarding:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData   // ← end-user identity never placed here
    );
```

The original `msg.sender` is stored only in transient callback context for payment purposes; it is never forwarded to the pool as the swapper identity.

**Resulting bypass**

| Configuration | Outcome |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot use the router; direct pool call required. Breaks intended UX. |
| Router **allowlisted** | Every user — allowlisted or not — can swap by routing through the router. Allowlist is fully bypassed. |

Both outcomes are broken. The second is the fund-impacting path.

---

### Impact Explanation

A curated pool (e.g., KYC-only, institutional-only, or regulatory-restricted) relies on `SwapAllowlistExtension` to enforce that only approved counterparties trade against its LPs. Once the router is allowlisted, any unprivileged user can trade in the pool. LPs are exposed to unintended counterparties, adverse selection, and potential regulatory liability. The pool admin's curation boundary is completely nullified by a public, permissionless contract path. This is a direct admin-boundary break with LP-principal impact above contest thresholds.

---

### Likelihood Explanation

The bypass requires only that the pool admin allowlists the router — a configuration any reasonable admin would make to let their approved users access the standard periphery. The exploit itself requires no special privileges: any user calls `MetricOmmSimpleRouter.exactInputSingle` targeting the curated pool. No setup beyond a standard router call is needed.

---

### Recommendation

The extension must gate by the **actual end user**, not the immediate caller of `pool.swap()`. Two viable approaches:

1. **Pass original caller in `extensionData`**: The router encodes `msg.sender` into `extensionData` for each hop; the extension decodes and checks that address. Requires a convention between router and extension.

2. **Check `sender` AND require `sender == msg.sender` at the pool level**: The pool could enforce that `sender` (the checked identity) must equal the actual `msg.sender`, preventing routers from substituting their own address. This would require direct-pool-only access for allowlisted pools.

The simplest safe fix is option 1: the router places the original user address in `extensionData`, and `SwapAllowlistExtension` decodes and checks it when `extensionData` is non-empty, falling back to `sender` for direct pool calls.
