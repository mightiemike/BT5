### Title
Swap Allowlist Gates Router Address Instead of Ultimate User, Enabling Any User to Bypass Per-User Swap Restrictions — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` keys its allowlist check on `sender`, which the pool sets to `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates a swap, `msg.sender` inside the pool is the router, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. A pool admin who allowlists the router to let approved users trade via the router simultaneously opens the pool to every user of that router, defeating the per-user curation the extension is meant to enforce.

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (the contract that calls the extension), and `sender` is whatever the pool forwarded. The pool always forwards its own `msg.sender`:

```solidity
// MetricOmmPool.sol – _beforeSwap call site
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap(), not the ultimate user
    recipient,
    ...
)
```

When `MetricOmmSimpleRouter` executes a swap on behalf of a user, it calls `pool.swap(...)` directly, making the router the `msg.sender` inside the pool. The extension therefore evaluates `allowedSwapper[pool][router]`.

A pool admin who wants allowlisted users to be able to trade via the router must add the router to the allowlist. Once the router is allowlisted, the check `allowedSwapper[pool][router]` passes for **every** caller of the router, including users who were never individually approved. The per-user gate collapses to a per-router gate.

### Impact Explanation

Any user who routes through `MetricOmmSimpleRouter` can swap on a curated pool whose admin has allowlisted the router. LP providers on institutional or KYC-gated pools are exposed to swaps from non-approved counterparties. Because the pool prices swaps from the oracle, an adversarial non-allowlisted user can execute swaps that extract value from LP positions at oracle-fair prices, causing direct loss of LP principal on pools that were designed to restrict counterparty exposure.

### Likelihood Explanation

The router is the standard periphery entry point. Any pool admin who wants approved users to trade via the router must allowlist it. The admin has no mechanism to allowlist the router for specific users only — the allowlist is binary per address. The misconfiguration is therefore a predictable operational outcome, not a rare edge case.

### Recommendation

The extension should verify the ultimate user identity rather than the direct pool caller. Two approaches:

1. **Pass the original user through `extensionData`**: The router encodes the original `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a trusted encoding convention.
2. **Check `recipient` instead of `sender` for swap allowlisting**: If the pool's design intent is to gate who *receives* output tokens, `recipient` is the correct field. If the intent is to gate who *initiates* the swap, the router must relay the original caller identity.
3. **Require direct pool interaction for allowlisted pools**: Document that pools using `SwapAllowlistExtension` must not allowlist the router address, and that allowlisted users must call the pool directly.

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls setAllowedToSwap(pool, userA, true).
3. Pool admin calls setAllowedToSwap(pool, router, true)
   — necessary so userA can trade via the router.
4. Non-allowlisted userC calls:
     router.exactInputSingle(pool, tokenIn, tokenOut, amountIn, ...)
   The router calls pool.swap(recipient=userC, ...).
   Inside the pool, msg.sender = router.
   Extension checks: allowedSwapper[pool][router] == true → passes.
5. userC's swap executes on the curated pool despite never being individually approved.
