### Title
Fork Retarget Difficulty Computed from Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

`get_header_by_height` is implemented exclusively against `mainchain_height_to_header`. When `get_next_work_required` is called for a fork block that sits at a retarget boundary, it fetches the interval-start timestamp from the mainchain map rather than from the fork's own ancestor at that height. If the fork diverged before `first_block_height`, the mainchain block and the fork's ancestor at that height are different objects with different timestamps, so the difficulty target computed for the fork block is wrong — it reflects the mainchain's retarget history, not the fork's.

The developer already flagged this in `dogecoin.rs`:

```
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
```

---

### Finding Description

**`BlocksGetter::get_header_by_height` always reads the mainchain index.** [1](#0-0) 

There is no fork-aware path. The function unconditionally resolves height → hash via `mainchain_height_to_header`, then fetches the header from `headers_pool`.

**`get_next_work_required` (Bitcoin) calls this at every retarget boundary.** [2](#0-1) 

`first_block_height = prev_block_header.block_height - (difficulty_adjustment_interval - 1)`. The timestamp of the block at that height is passed directly to `calculate_next_work_required` as `first_block_time`.

**Same pattern in Litecoin and Dogecoin.** [3](#0-2) [4](#0-3) 

**The invariant that is broken:** for a fork block at height `R` (a retarget boundary), the correct `first_block_time` is the timestamp of the fork's own ancestor at height `R - (interval - 1)`. If the fork diverged at any height ≤ `R - (interval - 1)`, the fork's ancestor at that height is a different block than the mainchain block at the same height. The contract uses the mainchain block's timestamp, not the fork's.

**`calculate_next_work_required` is sensitive to `first_block_time`.** [5](#0-4) 

`actual_time_taken = prev_block_time - first_block_time`. A smaller `first_block_time` (earlier mainchain anchor) inflates `actual_time_taken`, which — after the ×4 cap — produces a higher (easier) target. The attacker controls the fork's `prev_block_time` (within the MTP and future-time guards) and benefits from the mainchain's fixed, potentially earlier `first_block_time`.

---

### Impact Explanation

A fork block at a retarget boundary is accepted by `check_pow` / `check_target` with a `bits` value that satisfies the mainchain-derived difficulty formula rather than the fork-derived one. Concretely:

- If the mainchain's block at `first_block_height` has an earlier timestamp than the fork's ancestor at that height, `actual_time_taken` is inflated, the new target is higher, and the fork block is accepted at a lower difficulty than the network rules require.
- The accepted `bits` field is then stored in `headers_pool` and propagated as the base for subsequent fork blocks' difficulty checks, compounding the deviation across the next interval.
- On a successful reorg, these incorrectly-difficulty-checked fork blocks become the new mainchain, permanently recording headers that would be invalid under correct Bitcoin consensus rules. [6](#0-5) 

---

### Likelihood Explanation

**Barriers:**

1. `submit_blocks` is gated by `#[trusted_relayer]`. An attacker must either stake as a relayer or hold `Role::UnrestrictedSubmitBlocks` / `Role::DAO`. [7](#0-6) 

2. For **Bitcoin mainnet** (`pow_allow_min_difficulty_blocks = false`), every fork block must carry valid SHA-256d PoW at the current network difficulty. Mining 2016+ fork blocks requires a majority of the global hash rate — this is a 51%-class precondition and is not practically reachable by a single attacker.

3. For **Bitcoin testnet** (`pow_allow_min_difficulty_blocks = true`), the min-difficulty shortcut at lines 57–62 fires when `block_header.time > prev_time + 2 * target_spacing`, bypassing `get_header_by_height` entirely for non-retarget blocks. At the retarget block itself the normal path is taken, but testnet PoW is trivially satisfiable, so a trusted relayer on a testnet deployment can reach the retarget path with crafted timestamps.

4. The developer's own TODO comment in `dogecoin.rs` line 291 confirms awareness of the issue but no fix has been applied to any of the three affected modules.

**Net assessment:** On mainnet the precondition (mining 2016 blocks) is a 51%-class requirement and is rejected by the scope rules. On testnet the path is reachable by any trusted relayer. The bug is real and the invariant is provably broken; practical severity on mainnet is low due to the PoW barrier, but the logic error is present in production code for all three chains.

---

### Recommendation

Replace the height-based lookup in `get_next_work_required` with an ancestor walk that follows `prev_block_hash` links through `headers_pool` (the same store used by `get_prev_header`). This is chain-agnostic and correct for both mainchain and fork contexts:

```rust
// Walk back (difficulty_adjustment_interval - 1) steps from prev_block_header
// using get_prev_header, which reads headers_pool regardless of chain membership.
let mut cursor = prev_block_header.clone();
for _ in 0..(config.difficulty_adjustment_interval - 1) {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time = cursor.block_header.time;
```

This mirrors how Bitcoin Core's `GetAncestor` traverses the block index and is the correct fix for all three affected modules (`bitcoin.rs`, `litecoin.rs`, `dogecoin.rs`).

### Proof of Concept

State-machine sketch (no external dependencies, uses `skip_pow_verification = true` to isolate the logic bug):

```
Setup:
  genesis at height 0 (mainchain)
  mainchain blocks 0..2015 with timestamps T_main[h]
  T_main[0] = 1_000_000   (the anchor the mainchain retarget will use)

Fork:
  diverge at height 0 (same genesis)
  fork blocks 0..2014 with timestamps T_fork[h]
  T_fork[0] = 1_500_000   (later than T_main[0] by 500_000 s)
  fork block at height 2015 (retarget boundary):
    prev_time = T_fork[2014]

Expected bits at fork retarget:
  actual_time = T_fork[2014] - T_fork[0]   (uses fork's own anchor)

Actual bits computed by contract:
  get_header_by_height(0) → mainchain block → time = T_main[0] = 1_000_000
  actual_time = T_fork[2014] - 1_000_000   (inflated by 500_000 s)
  → higher target → easier bits

Assert:
  bits_from_contract != bits_expected_from_fork_history
  bits_from_contract encodes an easier target
  check_pow accepts the fork block with the easier bits
```

The fork block at height 2015 is stored with `bits` that no correct Bitcoin node would accept, violating the consensus-rule enforcement invariant of the light client. [8](#0-7) [1](#0-0) [9](#0-8)

fn get_next_work_required(
    config: &NetworkConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    if (prev_block_header.block_height + 1) % config.difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            if block_header.time
                > prev_block_header.block_header.time + 2 * config.pow_target_spacing
            {
                return config.proof_of_work_limit_bits;
            }

            let mut current_block_header = prev_block_header.clone();
            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            let last_bits = current_block_header.block_header.bits;
            return last_bits;
        }
        return prev_block_header.block_header.bits;
    }

    let first_block_height =
        prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

    let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
    calculate_next_work_required(
        config,
        prev_block_header,
        interval_tail_extend_header.block_header.time.into(),
    )
}
