### Title
Silent Bonus Reward Forfeiture with Permanent Checkpoint Advancement During Redelegation Slash - (File: x/tieredrewards/keeper/slash.go)

---

### Summary

In `slashRedelegationPosition`, when the rewards pool is insufficient to pay accrued bonus rewards during a `BeforeRedelegationSlashed` hook, the `ErrInsufficientBonusPool` error is silently swallowed. However, `processEventsAndClaimBonus` has already advanced the position's `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded` checkpoints in-memory, and decremented event reference counts on-chain, before returning the error. The caller then persists the updated position with advanced checkpoints via `setPositionWithState`. The user's accrued bonus rewards are permanently and irrecoverably lost — there is no retry path and no minimum-amount protection.

---

### Finding Description

`slashRedelegationPosition` in `x/tieredrewards/keeper/slash.go` is invoked by the `BeforeRedelegationSlashed` staking hook whenever a redelegating tier position is slashed. It calls `processEventsAndClaimBonus` to settle bonus rewards before the slash modifies shares.

Inside `processEventsAndClaimBonus` (`x/tieredrewards/keeper/claim_rewards.go`), the function:

1. Walks all pending validator events since `pos.LastEventSeq`, updating `pos.LastEventSeq` and decrementing each event's on-chain reference count for each processed event.
2. Calls `applyBonusAccrualCheckpoint` to advance `pos.LastBonusAccrual` to the current block time.
3. Calls `pos.UpdateLastKnownBonded` to persist the new bonded state.
4. **Only then** checks `sufficientBonusPoolBalance` — if the pool is short, returns `ErrInsufficientBonusPool`.

Back in `slashRedelegationPosition`, the error is caught and explicitly swallowed:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // error is dropped — execution continues
    } else {
        return err
    }
}
```

Execution then falls through to `setPositionWithState(ctx, pos, ...)`, which persists the `pos` struct that already has its checkpoints advanced past the accrual window. No bonus coins are ever sent to the owner.

The event reference counts were also decremented on-chain during the walk. When they reach zero, those events are garbage-collected and can never be replayed. The accrual window is permanently closed.

---

### Impact Explanation

A tier position owner who has a redelegating position (created via `MsgTierRedelegate`) permanently loses all accrued bonus rewards for the period since their last claim if:

- The destination validator is slashed during the redelegation unbonding window, AND
- The `RewardsPoolName` module account balance is insufficient to cover the computed bonus at that moment.

The corrupted value is the `bonus_rewards` balance that should have been transferred from `types.RewardsPoolName` to the position owner. The position's `LastBonusAccrual` and `LastEventSeq` are advanced to the slash block, so the lost rewards can never be reclaimed — not even after the pool is replenished. This is a direct, permanent loss of earned rewards for the position owner.

---

### Likelihood Explanation

The rewards pool (`RewardsPoolName`) is a shared resource drained by two concurrent paths:

1. **`topUpBaseRewards` (BeginBlocker)**: Every block, if the fee collector is short, the BeginBlocker drains the pool by `min(pool_balance, shortfall)`. Under a high `TargetBaseRewardsRate` with low fee activity, this can drain the pool to zero within a few blocks.
2. **User bonus claims**: Any user claiming bonus rewards reduces the pool balance.

A redelegation slash can occur at any time during the 21-day unbonding window. If the pool happens to be empty or insufficient at that block — a realistic steady-state condition — the hook silently forfeits the bonus. No attacker action is required; the pool draining is a normal protocol operation. A sophisticated attacker could also deliberately drain the pool (by timing their own large bonus claims) immediately before a known slash event to guarantee the victim's forfeiture.

---

### Recommendation

The checkpoint advancement and event reference-count decrements must not be committed when the bonus payment fails. Two options:

1. **Atomic rollback**: Do not advance checkpoints or decrement reference counts until after the pool-balance check succeeds. Restructure `processEventsAndClaimBonus` so all state mutations occur only after the payment is confirmed.

2. **Partial payout with preserved checkpoints**: If the pool is insufficient, pay whatever the pool holds (`min(pool_balance, bonus)`) and advance checkpoints only up to the point covered by the partial payment, leaving the remainder claimable after the pool is replenished.

Either way, the `slashRedelegationPosition` caller must not persist a `pos` with advanced checkpoints when no bonus was paid.

---

### Proof of Concept

1. User calls `MsgTierRedelegate` on position P, creating a redelegation entry with `unbondingId = U`.
2. The destination validator is slashed during the 21-day unbonding window.
3. The staking module fires `BeforeRedelegationSlashed(ctx, U, sharesToUnbond)`.
4. `slashRedelegationPosition` is called; it calls `processEventsAndClaimBonus(&pos)`.
5. Inside `processEventsAndClaimBonus`: events are walked, `pos.LastEventSeq` is updated, event reference counts are decremented on-chain, `pos.LastBonusAccrual` is advanced to `blockTime`, `pos.LastKnownBonded` is updated.
6. `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` (pool is empty from BeginBlocker draining).
7. `processEventsAndClaimBonus` returns the error without sending any coins.
8. `slashRedelegationPosition` swallows the error and calls `setPositionWithState(ctx, pos, ...)` with the checkpoint-advanced `pos`.
9. The position is now persisted with `LastBonusAccrual = blockTime` and `LastEventSeq` past all processed events. No bonus was paid.
10. After the pool is replenished, the user calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` starts from the already-advanced `LastEventSeq` and `LastBonusAccrual` — the forfeited accrual window is gone. The user receives zero bonus for that period.
