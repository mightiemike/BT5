### Title
Unvalidated `tx_id` Enables Merkle Internal-Node Substitution in `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

---

### Summary

The public, unpermissioned `verify_transaction_inclusion` function accepts a caller-supplied `tx_id` without validating that it is a leaf-level transaction hash. An unprivileged NEAR caller can substitute any internal Merkle tree node hash as `tx_id`, supply a shortened proof derived entirely from public Bitcoin block data, and receive a `true` verification result for a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` in `contract/src/lib.rs` accepts `ProofArgs` and performs the following checks before computing the proof: [1](#0-0) 

The only guard on `merkle_proof` is: [2](#0-1) 

There is no check that `tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node. The function's own documentation acknowledges this:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

The verification delegates to `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs`: [4](#0-3) 

This function takes `transaction_hash` and `transaction_position` and iteratively hashes upward. It has no knowledge of whether the starting hash is a leaf or an internal node — it simply applies the proof steps mechanically.

**Concrete attack path** (all inputs derived from public Bitcoin block data):

1. Take any Bitcoin block already stored in the contract's `headers_pool` (e.g., block 685452 used in the test suite).
2. Compute the block's Merkle tree. Pick an internal node at depth `d`, position `p`, with hash `H`. Its sibling at depth `d` is `S`.
3. The proof from `H` to the root is the `D − d` sibling hashes above it, where `D` is the full tree depth.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = H` (the internal node hash)
   - `tx_index = p`
   - `merkle_proof = [S, sibling_{d+1}, ..., sibling_{D-1}]` (length `D − d`)
   - `tx_block_blockhash` = the real block hash
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(H, p, proof)` faithfully reconstructs the Merkle root, because `H` is a genuine node in the tree.
6. The comparison `== header.block_header.merkle_root` succeeds.
7. The function returns `true` for a transaction that does not exist.

The function is `pub` with no caller restriction (`#[pause]` only allows pausing, not caller filtering): [5](#0-4) 

The `#[deprecated]` attribute is a Rust compile-time warning only; it imposes no runtime restriction on NEAR RPC callers.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate fund release (e.g., a bridge contract releasing wrapped BTC upon proof of a Bitcoin deposit) can be deceived into releasing funds for a fabricated transaction. The returned `bool` is treated as authoritative proof of Bitcoin transaction inclusion. A false positive causes an irreversible, incorrect state transition in the downstream contract — directly analogous to the external report's irreversible fund loss from an unvalidated parameter.

---

### Likelihood Explanation

The attack requires only:
- Knowledge of any Bitcoin block already accepted by the contract (all block data is public).
- The ability to compute Merkle tree internal node hashes (trivial arithmetic on public data).
- A single unpermissioned NEAR function call.

No mining power, privileged role, leaked key, or social engineering is required. The function remains callable on any non-paused deployment.

---

### Recommendation

1. **Remove or hard-disable `verify_transaction_inclusion` (v1)** at the contract level. Replace its body with `env::panic_str("deprecated: use verify_transaction_inclusion_v2")` so NEAR RPC callers cannot invoke it.
2. **Enforce migration to `verify_transaction_inclusion_v2`**, which adds the coinbase proof length check that prevents internal-node substitution: [6](#0-5) 

3. As a defense-in-depth measure, add a check in `verify_transaction_inclusion_v2` that `args.tx_index < (1u64 << args.merkle_proof.len())` to reject indices that are structurally inconsistent with the declared proof depth.

---

### Proof of Concept

Using the real block headers already present in the test suite (`real_block_headers_685440_to_685451`, block 685452): [7](#0-6) 

1. Fetch block 685452's full transaction list from any Bitcoin node or block explorer.
2. Build the Merkle tree. For a block with `N` transactions, the root's left child `H = SHA256d(txid_0 || txid_1)` is an internal node at depth 1, position 0.
3. The right child `S = SHA256d(txid_2 || txid_3)` is the sibling.
4. Call `verify_transaction_inclusion` with `tx_id = H`, `tx_index = 0`, `merkle_proof = [S]`, `tx_block_blockhash = hash_of_685452`, `confirmations = 1`.
5. `compute_root_from_merkle_proof(H, 0, [S])` computes `SHA256d(H || S) = Merkle root of 685452`.
6. The function returns `true`. No such transaction exists in block 685452.

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
