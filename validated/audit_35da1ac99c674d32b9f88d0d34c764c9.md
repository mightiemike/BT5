### Title
Hardcoded `LastKnownBonded=true` in `createDelegatedPosition` allows bonus accrual during unbonded validator periods — (`x/tieredrewards/keeper/position.go`)

### Summary
`createDelegatedPosition` unconditionally sets `LastKnownBonded=true` when creating a new position. If the target validator is currently unbonded, this false initial state causes `processEventsAndClaimBonus` to treat the entire gap between position creation and the validator's next re-bond as a bonded segment, paying out bonus rewards the attacker is not entitled to.

### Finding Description
In `createDelegatedPosition` (`position.go:56`), `NewPosition` is called with the literal `true` for `lastKnownBonded`. No check is made against the validator's actual bond status. `validateNewPosition` imposes no bond-status restriction, so `LockTier` and `CommitDelegationToTier` both succeed on unbonded validators.

When `processEventsAndClaimBonus` later runs, it initialises `bonded := pos.LastKnownBonded` (`claim_rewards.go:164`). Because `lastEventSeq` was snapshotted at creation time, the UNBOND event that preceded the position is invisible. The only event in the replay window is the subsequent BOND event. With `bonded=true`, the loop computes `computeSegmentBonus` for `[LastBonusAccrual, BOND_event_time]` — the full unbonded gap — before processing the BOND event.

### Impact Explanation
The attacker receives bonus tokens from the `RewardsPoolName` module account for a time window during which the validator was unbonded. This directly drains the shared bonus pool, reducing rewards available to legitimate position holders. The exact delta is `computeSegmentBonus(pos, tier, creationTime, reBondTime, tokensPerShareAtBond)`.

### Likelihood Explanation
Validators are routinely jailed for downtime. An attacker can monitor the chain, create a position on a jailed validator (which is cheap — only the min-lock amount is required), wait for the validator to unjail, and immediately claim. No privileged access is required; the entire path is reachable via standard `MsgLockTier` + `MsgClaimTierRewards` transactions.

### Recommendation
In `createDelegatedPosition`, query the validator's actual bond status and pass it as `lastKnownBonded`:

```go
val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
if err != nil {
    return types.Position{}, err
}
pos := types.NewPosition(id, owner, tier.Id, delAddr.String(),
    blockHeight, lastEventSeq, blockTime, val.IsBonded(), blockTime)
```

### Proof of Concept
```
1. Jail validator V → AfterValidatorBeginUnbonding fires, UNBOND event seq=N recorded.
2. Attacker calls MsgLockTier targeting V.
   - createDelegatedPosition: lastEventSeq = N, LastKnownBonded = true (bug).
3. Advance time by T days (V remains unbonded).
4. Unjail V → AfterValidatorBonded fires, BOND event seq=N+1 recorded.
5. Attacker calls MsgClaimTierRewards.
   - processEventsAndClaimBonus: bonded=true, segmentStart=creationTime.
   - Loop: event seq=N+1 (BOND), bonded==true → bonus computed for [creationTime, bondTime].
   - Attacker receives T days of bonus for an unbonded period.
6. Assert: bonus > 0 despite validator being unbonded for the entire accrual window.
```

func (h Hooks) AfterValidatorBonded(ctx context.Context, _ sdk.ConsAddress, valAddr sdk.ValAddress) error {
	count, err := h.k.getPositionCountForValidator(ctx, valAddr)
	if err != nil {
		return err
	}
	if count == 0 {
		return nil
	}

	tokensPerShare, err := h.k.getTokensPerShare(ctx, valAddr)
	if err != nil {
		return err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	_, err = h.k.appendValidatorEvent(ctx, valAddr, types.ValidatorEvent{
		Height:         sdkCtx.BlockHeight(),
		Timestamp:      sdkCtx.BlockTime(),
		EventType:      types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND,
		TokensPerShare: tokensPerShare,
		ReferenceCount: count,
	})
	return err
