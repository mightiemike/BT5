### Title
`validate_proposal` uses `VersionedConstants::latest_constants()` for `fee_proposal_margin_ppt` while the version gate uses the block's `starknet_version` — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The fee-proposal bounds check in `validate_proposal` always reads `fee_proposal_margin_ppt` from `VersionedConstants::latest_constants()`, while the version gate that decides whether a fee proposal is *required at all* correctly reads the block's own `starknet_version`. The two checks therefore operate under different protocol-version assumptions. If `fee_proposal_margin_ppt` changes between the block's declared version and the latest version, validators will apply the wrong margin: either rejecting valid proposals (consensus stall) or accepting out-of-range proposals (incorrect gas-price committed to state).

---

### Finding Description

**Version gate — uses block's version (correct):**

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:371
let fee_proposal_required = init_proposed.starknet_version >= StarknetVersion::V0_14_3;
```

**Margin check — always uses latest constants (wrong):**

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:401-415
let (lower_bound, upper_bound) = fee_proposal_bounds(
    fee_actual,
    VersionedConstants::latest_constants().fee_proposal_margin_ppt,
);
if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
```

The `apollo_versioned_constants` crate defines five distinct versioned-constants files (V0_14_0 … V0_14_4), each of which can carry a different `fee_proposal_margin_ppt`:

```rust
// crates/apollo_versioned_constants/src/lib.rs:33-43
define_versioned_constants!(
    VersionedConstants,
    VersionedConstantsError,
    StarknetVersion::V0_14_0,
    "resources/versioned_constants_diff_regression",
    (V0_14_0, "../resources/orchestrator_versioned_constants_0_14_0.json"),
    ...
    (V0_14_4, "../resources/orchestrator_versioned_constants_0_14_4.json"),
);
```

The `VersionedConstants` struct exposes `fee_proposal_margin_ppt` as a first-class versioned field:

```rust
// crates/apollo_versioned_constants/src/lib.rs:30
pub fee_proposal_margin_ppt: u128,
```

**Analog to the LMPVault bug:** `previewDeposit` used `totalAssets_cached` while `_withdraw` used `totalAssets_actual` — two code paths used different views of the same quantity. Here, the version gate uses the block's declared version (the "actual" view) while the margin check uses `latest_constants()` (a potentially divergent "cached" view). The divergence is exploitable whenever the two versions carry different `fee_proposal_margin_ppt` values.

---

### Impact Explanation

`fee_proposal` is the proposer's declared gas price for the next block. It is committed to consensus state and drives the EIP-1559 base-fee update. Two divergence scenarios:

| `fee_proposal_margin_ppt` in latest vs block version | Effect |
|---|---|
| **Tighter in latest** | Validator rejects a fee proposal that is valid under the block's version → consensus stall; no block can be finalized until the proposer is replaced |
| **Looser in latest** | Validator accepts a fee proposal that is invalid under the block's version → wrong gas price committed to state → economic impact on all fee-paying transactions in subsequent blocks |

Both scenarios fall under **"Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."**

---

### Likelihood Explanation

The trigger requires only that:
1. A block is proposed under a `starknet_version` that is not the latest (normal during any protocol upgrade window), and
2. `fee_proposal_margin_ppt` differs between that version and the latest.

Condition 1 is routine during upgrades. Condition 2 is structurally possible because the parameter is versioned and the diff-regression test infrastructure exists precisely to track such changes. No privileged access is required; any block proposer running the older protocol version triggers the discrepancy automatically.

---

### Recommendation

Replace `VersionedConstants::latest_constants()` with a lookup keyed on the block's declared version:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs
use apollo_versioned_constants::VersionedConstants;
use starknet_api::versioned_constants_logic::VersionedConstantsTrait;

let vc = VersionedConstants::get(&init_proposed.starknet_version)
    .map_err(|e| ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!("Unknown starknet version: {e}"),
    ))?;

let (lower_bound, upper_bound) = fee_proposal_bounds(
    fee_actual,
    vc.fee_proposal_margin_ppt,
);
```

This mirrors the pattern already used correctly for the version gate and ensures both checks operate under the same protocol-version assumption.

### Proof of Concept

**Setup:** Two consecutive protocol versions where `fee_proposal_margin_ppt` differs.
- V0_14_3: `fee_proposal_margin_ppt = 20` (2 %)
- V0_14_4 (latest): `fee_proposal_margin_ppt = 10` (1 %)

**Attack path (consensus stall):**

1. Proposer runs V0_14_3 software; produces a block with `starknet_version = V0_14_3`.
2. `fee_actual` from the sliding window = 1 000 000 fri.
3. Proposer sets `fee_proposal = 1 015 000` fri (1.5 % above `fee_actual`, within the V0_14_3 2 % margin).
4. Validator runs V0_14_4 software; calls `validate_proposal`.
5. Version gate passes: `V0_14_3 >= V0_14_3` → `fee_proposal_required = true`.
6. Margin check uses `latest_constants().fee_proposal_margin_ppt = 10` → allowed range = [990 000, 1 010 000].
7. `1 015 000 > 1 010 000` → `ValidateProposalError::InvalidProposalInit` returned.
8. All V0_14_4 validators reject the proposal; consensus stalls.

**Attack path (wrong gas price committed):**

Reverse the margin values (latest is looser). Proposer sets `fee_proposal` outside the V0_14_3 margin but inside the V0_14_4 margin. Validators accept it; the wrong gas price is written to state, affecting all subsequent transactions. [1](#0-0) [2](#0-1) [3](#0-2)

#[derive(Clone, Debug, Deserialize)]
pub struct VersionedConstants {
    ///  This is used to calculate the base gas price for the next block according to EIP-1559 and
    /// serves as a sensitivity parameter that limits the maximum rate of change of the gas price
    /// between consecutive blocks.
    pub gas_price_max_change_denominator: u128,
    /// The minimum gas price in fri.
    pub min_gas_price: GasPrice,
    /// The maximum block size in gas units.
    // NOTE: Must stay in sync with BouncerWeights receipt_l2_gas.
    // NOTE: When max_block_size is changed, update `gas_target` accordingly to maintain the ratio.
    pub max_block_size: GasAmount,
    /// The target gas usage per block. Used by EIP-1559 to calculate the next block's gas price.
    // Target is 60% of max_block_size, making price adjustment more responsive to congestion.
    pub gas_target: GasAmount,
    /// The margin for the eth to fri rate disagreement, expressed as a percentage (parts per
    /// hundred).
    pub l1_gas_price_margin_percent: u32,
    /// Number of `fee_proposal` values used to compute `fee_actual` (sliding window).
    pub fee_proposal_window_size: u64,
    /// Maximum `fee_proposal` change per block in parts per thousand (e.g., `2` = 0.2%).
    pub fee_proposal_margin_ppt: u128,
}
