### Title
Unchecked `u64` Multiplication Overflow in `op_unknown` Cost Calculation Enables Undercharged Execution — (File: `src/more_ops.rs`)

---

### Summary

`op_unknown` in `src/more_ops.rs` computes a final cost via `cost *= cost_multiplier + 1` without overflow protection. In Rust release mode, `u64` multiplication wraps silently (two's complement). An attacker who controls the opcode bytes and argument atoms can craft inputs where the product wraps to a value below `u32::MAX`, bypassing the only post-multiplication guard (`if cost > u32::MAX as u64`), and causing the interpreter to return a near-zero cost for an operation that was supposed to be expensive. This is the direct analog of the external report's `int64` fee-calculation overflow: both involve fixed-width integer arithmetic on an attacker-controlled multiplier applied to an attacker-controlled base cost, with no checked-arithmetic primitive.

---

### Finding Description

In `op_unknown` (`src/more_ops.rs`): [1](#0-0) 

`cost_multiplier` is decoded from the opcode atom via `u32_from_u8`, giving a value in `[0, u32::MAX]`. As a `u64`, `cost_multiplier + 1` therefore ranges up to `2^32 = 4_294_967_296`. [2](#0-1) 

For `cost_function = 2` (mul-like), the inner loop accumulates a quadratic term `(l0 * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER` and calls `check_cost(cost, max_cost)?` each iteration, so `cost` is bounded by `max_cost` — which in consensus mode (no `LIMIT_HEAP`, no `NO_UNKNOWN_OPS`) can be the full block cost limit (~11 billion). [3](#0-2) 

The critical sequence is:
1. `check_cost(cost, max_cost)?` — passes because `cost ≤ max_cost`
2. `cost *= cost_multiplier + 1;` — **no overflow check**; wraps in release mode
3. `if cost > u32::MAX as u64` — the only guard, which the wrapped value can bypass

`Cost` is `u64` (`src/cost.rs` line 3). Rust's release-mode semantics define wrapping on overflow. There is no `checked_mul`, `saturating_mul`, or `wrapping_mul` call here. [4](#0-3) 

---

### Impact Explanation

`op_unknown` is reachable in **consensus mode** (default `ClvmFlags::empty()`). `MEMPOOL_MODE` sets `NO_UNKNOWN_OPS`, which blocks `op_unknown` there, but consensus validation does not set that flag: [5](#0-4) 

When the wrapped cost is returned (e.g., 0), it is added to the running `cost` accumulator in `run_program`. The program's total cost is therefore underreported relative to what the opcode's declared multiplier intended. A consensus node accepts the block as within the cost limit while the true intended cost would have exceeded it. This is an **undercharged execution** vulnerability: the cost model is corrupted for any block containing a crafted unknown opcode, breaking the DoS-prevention invariant of the cost system.

---

### Likelihood Explanation

- The attacker controls the opcode bytes (setting `cost_multiplier = u32::MAX`, so `cost_multiplier + 1 = 2^32`) and the argument atoms (setting their sizes to satisfy `cost ≡ 0 (mod 2^32)` with `cost > u32::MAX`).
- For `cost_function = 2`, with two atoms of sizes `l0` and `l1`, the pre-multiplication cost is `92 + 885 + (l0+l1)*6 + (l0*l1)/128`. The attacker solves `cost ≡ 0 (mod 2^32)` by choosing `l0`, `l1` appropriately (e.g., near 1 MB each). The resulting `cost * 2^32 mod 2^64 = 0`, so the function returns cost = 0.
- In consensus mode, `LIMIT_HEAP` is absent, so large atoms are not blocked by the allocator.
- The overflow is deterministic and reproducible across all nodes running the same release binary, making it consensus-divergence-safe from the attacker's perspective (all nodes are equally affected).

---

### Recommendation

Replace the bare multiplication with a checked or saturating variant:

```rust
// Instead of:
cost *= cost_multiplier + 1;

// Use:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::CostExceeded)?;
```

This mirrors the fix pattern used in the external report (switching from unchecked fixed-width arithmetic to an overflow-safe alternative) and is consistent with how `op_add`'s fast path already uses `checked_add`: [6](#0-5)

### Proof of Concept

Craft a CLVM program containing a single unknown opcode call:

- **Opcode bytes**: 4 bytes, e.g. `[0xFF, 0xFF, 0xFF, 0x80]`
  - Last byte `0x80`: `cost_function = (0x80 >> 6) & 0x3 = 2` (mul-like)
  - Prefix `[0xFF, 0xFF, 0xFF]`: `cost_multiplier = u32_from_u8([0xFF,0xFF,0xFF]) = 16777215`... 

  More precisely, use a 4-byte opcode not in the known dispatch table (e.g., `[0x00, 0xFF, 0xFF, 0x80]`) with prefix bytes chosen so `cost_multiplier + 1 = 2^32` (i.e., prefix = `[0xFF, 0xFF, 0xFF]`, giving `cost_multiplier = 0xFFFFFF = 16777215`). Adjust to `cost_multiplier = 0xFFFFFFFF` by using a 5-byte opcode prefix.

- **Arguments**: Two atoms each of size `L` bytes, where `L` is chosen so that `(92 + 885 + 2L*6 + L^2/128) mod 2^32 = 0` and the total > `u32::MAX`. For `L ≈ 1_048_576` (1 MB), the quadratic term alone is `≈ 8.6 × 10^9 > u32::MAX`.

- **Expected result**: `op_unknown` returns `Reduction(0, nil)` instead of the correct multi-billion cost. The program's total cost is underreported, and `run_program` returns success within the cost limit. [3](#0-2)

use crate::error::{EvalErr, Result};

pub type Cost = u64;

pub fn check_cost(cost: Cost, max_cost: Cost) -> Result<()> {
    if cost > max_cost {
        Err(EvalErr::CostExceeded)
    } else {
        Ok(())
    }
}
