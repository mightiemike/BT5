Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the originating user, enabling complete allowlist bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` as the `msg.sender` of the pool's `swap` call. When users enter through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the originating EOA. If the pool admin allowlists the router address (required for any allowlisted user to swap via the router), every unprivileged user can bypass the allowlist by routing through the same router, defeating the curated-pool protection entirely.

## Finding Description

**Call path:**

1. Unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle(params)`.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, params.zeroForOne, ...)` — `msg.sender` at the pool is the router address.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` where `msg.sender` = router.
4. `ExtensionCalling._beforeSwap` encodes and dispatches `IMetricOmmExtensions.beforeSwap(sender=router, ...)` to the configured extension.
5. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` = `allowedSwapper[pool][router]`.
6. If the router is allowlisted for that pool, the check passes for **any** originating user.

**Root cause:** `MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as `sender` to the extension hook. The router is the immediate caller for all router-mediated swaps. The extension has no access to the original EOA; the router stores the original user only in transient storage for payment settlement, not for extension arguments.

**Why existing guards are insufficient:**
- `BaseMetricExtension.onlyPool` verifies only that the extension caller is a registered pool — it does not validate the identity of the originating user.
- There is no forwarding mechanism in the router or pool to pass the original `msg.sender` through to the extension.
- The pool admin cannot simultaneously (a) allowlist the router so that specific users can swap via it, and (b) block non-allowlisted users from using the same router path.

**Exact wrong value:** `allowedSwapper[pool][router]` is evaluated instead of `allowedSwapper[pool][originalEOA]`.

## Impact Explanation
Any unprivileged user can swap on a pool protected by `SwapAllowlistExtension` by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutputSingle` / `exactOutput`), provided the router address is in the allowlist. Since the router is the canonical public entry point, pool admins who intend to allow their allowlisted users to use the router must add the router to the allowlist, which simultaneously opens the pool to all users. This is a direct, complete bypass of a core access-control mechanism, constituting broken core pool functionality and an admin-boundary break reachable by any unprivileged caller.

## Likelihood Explanation
The router is the primary user-facing entry point documented and deployed for the protocol. Any pool admin who configures a swap allowlist and also wants their permitted users to access the pool via the router must allowlist the router. This is a natural and expected operational step. Once the router is allowlisted, the bypass is unconditional and repeatable by any address with no special privileges, capital, or timing requirements.

## Recommendation
Pass the originating user's address through the router to the pool, and have the pool forward it to extensions as a distinct `originator` field, separate from `sender`. Concretely:

- Add an `originator` parameter to `IMetricOmmExtensions.beforeSwap` (and the pool's internal `_beforeSwap`).
- In `MetricOmmSimpleRouter`, store `msg.sender` in transient storage before calling `pool.swap`, and expose it via a callback or pass it as part of `callbackData` / a dedicated field so the pool can retrieve and forward it.
- `SwapAllowlistExtension.beforeSwap` should check `allowedSwapper[pool][originator]` rather than `allowedSwapper[pool][sender]`.

Alternatively, restrict the allowlist check to `sender` only when `sender` is not a known periphery contract, and require periphery contracts to attest the originating user via a signed or transient-storage mechanism verified by the extension.
