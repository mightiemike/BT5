### Title
Linked Signer Cannot Be Immediately Revoked, Enabling a Compromised Signer to Drain a Subaccount During the 3-Day Slow-Mode Window — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `linkedSigners` mapping in Nado is the protocol-level analog of an ERC20 allowance. When a subaccount owner wants to change or revoke their linked signer, the only user-controlled path is a slow-mode `LinkSigner` transaction, which is subject to a hardcoded 3-day delay. During this window the old signer remains fully authorized. A compromised or malicious linked signer that observes the pending revocation on-chain can frontrun it by submitting signed `WithdrawCollateral` or `TransferQuote` transactions through the sequencer's fast path, draining the subaccount before the revocation takes effect.

---

### Finding Description

`EndpointStorage.sol` stores a per-subaccount authorized signer:

```solidity
mapping(bytes32 => address) internal linkedSigners;   // line 50
``` [1](#0-0) 

`Verifier.validateSignature` accepts a transaction as valid if it is signed by either the subaccount owner address **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [2](#0-1) 

The only way to change `linkedSigners[subaccount]` via the user-controlled (non-sequencer) path is to call `submitSlowModeTransaction` with a `LinkSigner` payload. Inside `submitSlowModeTransactionImpl`, `LinkSigner` falls into the generic `else` branch, which enqueues the transaction with a **3-day delay**:

```solidity
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
    ...
});
``` [3](#0-2) 

`SLOW_MODE_TX_DELAY` is 259,200 seconds (3 days):

```solidity
uint64 constant SLOW_MODE_TX_DELAY = 3 * 24 * 60 * 60; // 3 days
``` [4](#0-3) 

When the slow-mode `LinkSigner` is eventually processed, it simply overwrites the mapping:

```solidity
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [5](#0-4) 

There is no separate, immediately-effective revocation function. The fast-path `LinkSigner` (processed by the sequencer via `processTransactionImpl`) also writes to the same mapping:

```solidity
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [6](#0-5) 

but that path is sequencer-controlled and the sequencer may interleave the old signer's transactions before the `LinkSigner` is processed.

---

### Impact Explanation

During the 3-day slow-mode window, the old linked signer retains full authority to sign any transaction on behalf of the subaccount. Concretely, it can submit:

- **`WithdrawCollateral` / `WithdrawCollateralV2`** — transfers all collateral out of the subaccount to an external address.
- **`TransferQuote`** — moves quote balance to an attacker-controlled subaccount.
- **`LiquidateSubaccount`** — initiates a self-liquidation to extract value.

All of these are validated through `validateSignedTx` → `validateSignature`, which accepts the linked signer's signature as sufficient:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [7](#0-6) 

The corrupted state delta is the subaccount's collateral balances in `SpotEngine` and `PerpEngine`, which can be reduced to zero. The subaccount owner has no on-chain mechanism to stop this during the 3-day window.

---

### Likelihood Explanation

The scenario requires that a linked signer is compromised or turns malicious — a realistic operational risk for any user who delegates signing to a hot wallet, bot, or third-party service. The 3-day window is long enough for an attacker to act. The pending slow-mode transaction is publicly visible on-chain, giving the attacker a clear trigger. No privileged access (owner, sequencer key, governance) is required beyond possession of the old linked signer's private key.

---

### Recommendation

Add an immediately-effective revocation function that can be called directly by the subaccount owner (the address embedded in the first 20 bytes of the `bytes32` subaccount identifier) without going through the slow-mode queue:

```solidity
function revokeSigner(bytes12 subaccountName) external {
    bytes32 subaccount = bytes32(abi.encodePacked(msg.sender, subaccountName));
    requireSubaccount(subaccount);
    linkedSigners[subaccount] = address(0);
}
```

This mirrors the ERC20 mitigation of providing a `decreaseAllowance`-style path that does not require the two-step race window. Alternatively, require the current linked signer to be included in the `LinkSigner` transaction as an "old signer" field and verify it matches before updating, so a pending revocation cannot be raced against.

---

### Proof of Concept

1. Alice links signer `A` to her subaccount via a sequencer-processed `LinkSigner` transaction. `linkedSigners[alice_subaccount] = A`.
2. Alice discovers `A` is compromised. She calls `submitSlowModeTransaction` with a `LinkSigner` payload setting `signer = bytes32(0)` (revoke). The transaction is queued with `executableAt = block.timestamp + 259200`.
3. The attacker controlling `A` observes the pending slow-mode transaction on-chain.
4. The attacker signs a `WithdrawCollateralV2` transaction with `A`'s key, specifying `sendTo = attacker_address` and `amount = full_balance`. This is submitted to the sequencer.
5. The sequencer processes the withdrawal. `validateSignedTx` calls `validateSignature` → `Verifier.validateSignature`, which recovers `A` and accepts it as the linked signer. The collateral is transferred out.
6. Three days later, the slow-mode `LinkSigner` executes and sets `linkedSigners[alice_subaccount] = address(0)`. The account is now empty. [8](#0-7) [9](#0-8)
