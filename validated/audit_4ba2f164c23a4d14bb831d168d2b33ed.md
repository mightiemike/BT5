### Title
`ContractOwner` Cannot Receive Native ETH, Permanently Locking Funds in `DirectDepositV1` — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.withdrawFromDirectDepositV1()` is the designated recovery path for native ETH stuck in a `DirectDepositV1` (DDA) contract. However, `ContractOwner` declares no `receive()` or `fallback()` function, so any attempt by `DirectDepositV1.withdrawNative()` to push ETH to it will revert. Native ETH that becomes stranded in a DDA is permanently unrecoverable.

---

### Finding Description

`DirectDepositV1` is deployed via CREATE2 with a deterministic salt, meaning its address is predictable before deployment. The constructor explicitly handles the case where ETH has been pre-sent to that address: [1](#0-0) 

If the `wrappedNative.call` fails (e.g., the wrapped-native contract is paused, not yet deployed, or rejects the call), the constructor does **not** revert — it emits `NativeTokenTransferFailed` and leaves the ETH sitting in the DDA. The comment on line 54–55 even acknowledges this: *"shouldn't revert even if the transfer fails, otherwise the funds will be stuck in the DDA forever."* The irony is that the funds are still stuck — the constructor just avoids reverting.

The intended recovery path is `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`: [2](#0-1) 

This calls `DirectDepositV1.withdrawNative()`: [3](#0-2) 

`withdrawNative()` does `msg.sender.call{value: balance}("")` where `msg.sender` is `ContractOwner`. Because `ContractOwner` has no `receive()` or `fallback()` function, the low-level call returns `success = false`, and `require(success, "Failed to transfer native token to owner")` reverts the entire transaction.

`ContractOwner` inherits only from `EIP712Upgradeable` and `OwnableUpgradeable`, neither of which defines a `receive()` function. The `preBalance`/`postBalance` check at lines 629–632 is therefore never reached. [4](#0-3) 

---

### Impact Explanation

**Medium.** Any native ETH that becomes stranded in a `DirectDepositV1` contract — specifically via the constructor's silent-failure path — is permanently locked. There is no alternative recovery function. The `withdrawNative()` path in `DirectDepositV1` is the only mechanism to extract native ETH, and it unconditionally pushes to `msg.sender` (i.e., `ContractOwner`), which cannot accept it.

---

### Likelihood Explanation

**Medium.** `DirectDepositV1` is deployed with a fixed CREATE2 salt (`bytes32(uint256(1))`), making its address fully predictable before deployment. [5](#0-4) 

Any party can pre-fund the deterministic address with native ETH before `createDirectDepositV1` is called. If the `wrappedNative` contract rejects the deposit at constructor time (e.g., due to a temporary failure), the ETH is silently retained and becomes permanently irrecoverable. This is not a theoretical path — the constructor was explicitly written to tolerate this failure mode, and the recovery function was written to handle it, but the missing `receive()` breaks the recovery.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept native ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

This mirrors the fix recommended for the Infrared contract and is the minimal change needed to make the `withdrawFromDirectDepositV1(subaccount, address(0))` path functional.

---

### Proof of Concept

1. Compute the deterministic address of the `DirectDepositV1` for a given `subaccount` using the CREATE2 formula (salt = `bytes32(uint256(1))`, deployer = `ContractOwner`).
2. Send native ETH to that address before `createDirectDepositV1` is called.
3. Call `ContractOwner.creditDepositV1(subaccount)` (or `createDirectDepositV1` directly). The constructor runs, attempts `wrappedNative.call{value: balance}("")`, which fails (simulate by using a `wrappedNative` that reverts on receive). `NativeTokenTransferFailed` is emitted; ETH remains in the DDA.
4. The owner calls `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`.
5. Internally, `DirectDepositV1.withdrawNative()` executes `ContractOwner.call{value: balance}("")` — this returns `false` because `ContractOwner` has no `receive()`.
6. `require(success, "Failed to transfer native token to owner")` reverts.
7. The ETH is permanently locked in the DDA with no recovery path. [6](#0-5) [7](#0-6)
