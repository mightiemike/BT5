### Title
`ResourceBoundsMapping` → `ValidResourceBounds` Conversion Silently Collapses `AllResources` to `L1Gas`, Producing a Different Transaction Hash Preimage - (File: crates/apollo_rpc/src/v0_8/transaction.rs)

### Summary

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in the RPC layer silently maps `AllResources{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` to `L1Gas(X)`. Because `get_tip_resource_bounds_hash` branches on this enum variant to decide whether to include `L1_DATA_GAS` in the Poseidon hash preimage, the same economic intent produces two distinct transaction hashes depending on which representation is used. A V3 transaction submitted through the gateway is hashed under `AllResources` (H1, includes `L1_DATA_GAS = 0`), but when the same transaction data is read back through the RPC or fed into `starknet_estimateFee` / `starknet_simulateTransactions` via `BroadcastedTransaction`, the conversion produces `L1Gas` (H2, omits `L1_DATA_GAS`). H1 ≠ H2.

### Finding Description

`ValidResourceBounds` has two variants:

```
L1Gas(ResourceBounds)          // pre-0.13.3: only L1 gas; L1_DATA_GAS absent from hash
AllResources(AllResourceBounds) // post-0.13.3: L1_DATA_GAS present in hash
```

`get_tip_resource_bounds_hash` gates inclusion of `L1_DATA_GAS` on this variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // L1_DATA_GAS omitted
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // included
    }
});
``` [1](#0-0) 

The RPC-layer conversion collapses the two variants:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← AllResources silently becomes L1Gas
        } else {
            Self::AllResources(AllResourceBounds { ... })
        }
    }
}
``` [2](#0-1) 

The gateway submission path uses `RpcDeclareTransactionV3` / `RpcInvokeTransactionV3` / `RpcDeployAccountTransactionV3`, all of which store `resource_bounds: AllResourceBounds` and whose `DeclareTransactionV3Trait` / `InvokeTransactionV3Trait` implementations unconditionally return `ValidResourceBounds::AllResources(self.resource_bounds)`:

```rust
impl DeclareTransactionV3Trait for InternalRpcDeclareTransactionV3 {
    fn resource_bounds(&self) -> ValidResourceBounds {
        ValidResourceBounds::AllResources(self.resource_bounds)  // always AllResources
    }
``` [3](#0-2) 

So a transaction submitted with `AllResourceBounds{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` is hashed as `AllResources` (H1, preimage includes `L1_DATA_GAS = 0`). When the same transaction data is later round-tripped through `ValidResourceBounds → ResourceBoundsMapping → ValidResourceBounds` (which happens in the RPC response path and in `BroadcastedTransaction` handling for fee estimation / simulation), the result is `L1Gas(X)`, and the hash is recomputed as H2 (preimage omits `L1_DATA_GAS`). H1 ≠ H2.

The `ValidResourceBounds → ResourceBoundsMapping` direction is lossless for `AllResources` with non-zero fields, but lossy when `l2_gas = 0` and `l1_data_gas = 0`:

```rust
impl From<ValidResourceBounds> for ResourceBoundsMapping {
    fn from(value: ValidResourceBounds) -> Self {
        match value {
            ValidResourceBounds::L1Gas(l1_gas) => Self {
                l1_gas, l1_data_gas: ResourceBounds::default(), l2_gas: ResourceBounds::default(),
            },
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l1_data_gas, l2_gas })
                => Self { l1_gas, l1_data_gas, l2_gas },
        }
    }
}
``` [4](#0-3) 

Both `L1Gas` and `AllResources{l2_gas:0, l1_data_gas:0}` serialize to the same `ResourceBoundsMapping`, so the round-trip is not injective.

### Impact Explanation

**High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

For a V3 transaction with `AllResourceBounds{l1_gas: X, l2_gas: 0, l1_data_gas: 0}`:

- The gateway computes and stores H1 (with `L1_DATA_GAS = 0` in the Poseidon preimage).
- `starknet_getTransactionByHash` returns the transaction body with `ResourceBoundsMapping{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` alongside the stored hash H1. Any client or tool that reconstructs the hash from the returned body (following the same `ResourceBoundsMapping → ValidResourceBounds` logic) computes H2 ≠ H1 and concludes the hash is invalid.
- `starknet_estimateFee` / `starknet_simulateTransactions` accept `BroadcastedTransaction` types that carry `ResourceBoundsMapping`. After conversion to `ValidResourceBounds::L1Gas(X)`, the simulation computes H2 as the transaction hash for account validation. The user's signature was computed over H1. The simulation therefore fails signature verification and returns a wrong (failed) result for a transaction that would actually succeed on-chain, or vice versa.

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` (a valid configuration — the gateway's `validate_resource_bounds` only requires at least one non-zero resource bound, and `l1_gas > 0` satisfies that). The condition is reachable on every V3 transaction type (Invoke, Declare, DeployAccount). [5](#0-4) 

### Recommendation

Remove the lossy `ResourceBoundsMapping → ValidResourceBounds` conversion. Instead, preserve the original variant by either:

1. Adding a discriminant field (e.g., `"version": "0x3"` already present on V3 transactions) to distinguish old `L1Gas` transactions from new `AllResources` transactions with zero data/L2 gas, and using that to select the correct variant; or
2. Storing `ValidResourceBounds` directly in the RPC response types instead of `ResourceBoundsMapping`, eliminating the round-trip lossy conversion entirely.

The `From<ResourceBoundsMapping> for ValidResourceBounds` impl at `apollo_rpc/src/v0_8/transaction.rs:188–199` should be audited at every call site to ensure it is never used on data that originated as `AllResources`.

### Proof of Concept

1. Submit a V3 invoke transaction with `resource_bounds = AllResourceBounds{l1_gas: 1000, l2_gas: 0, l1_data_gas: 0}` via `starknet_addInvokeTransaction`. The gateway accepts it and returns hash H1.

2. Call `starknet_getTransactionByHash(H1)`. The response body contains `resource_bounds = {L1_GAS: 1000, L2_GAS: 0, L1_DATA: 0}`.

3. Reconstruct the hash from the response body using the same `ResourceBoundsMapping → ValidResourceBounds` logic: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `L1Gas(1000)`. Feed into `get_tip_resource_bounds_hash` → preimage omits `L1_DATA_GAS`. Result: H2.

4. Assert H1 ≠ H2. The `L1_DATA_GAS` element `get_concat_resource(&ResourceBounds{max_amount:0, max_price:0}, b"L1_DATA")` is a non-zero Felt (the resource name bytes are non-zero), so its presence or absence changes the Poseidon hash. [6](#0-5) 

5. Call `starknet_estimateFee` with the same transaction body. The simulation computes H2 for account validation, but the user's ECDSA signature covers H1. Signature verification fails, and the fee estimation returns an error for a transaction that would succeed on-chain.

fn get_concat_resource(
    resource_bounds: &ResourceBounds,
    resource_name: &ResourceName,
) -> Result<Felt, StarknetApiError> {
    let max_amount = resource_bounds.max_amount.0.to_be_bytes();
    let max_price = resource_bounds.max_price_per_unit.0.to_be_bytes();
    let concat_bytes =
        [[0_u8].as_slice(), resource_name.as_slice(), max_amount.as_slice(), max_price.as_slice()]
            .concat();
    Ok(Felt::from_bytes_be(&concat_bytes.try_into().expect("Expect 32 bytes")))
}
