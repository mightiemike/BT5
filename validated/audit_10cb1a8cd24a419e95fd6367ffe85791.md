### Title
`FeeEstimation.overall_fee` includes unreported L2 gas cost, violating the documented invariant — (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary
`estimate_fee` and `simulate_transactions` return a `FeeEstimation` response where `overall_fee` includes `l2_gas * (l2_gas_price + tip)`, but the struct exposes no `l2_gas_consumed` field. The documented invariant (`overall_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`) is violated for any v3 transaction with nonzero L2 gas consumption.

### Finding Description
In `tx_execution_output_to_fee_estimation`:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),       // L1 gas only
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // L1 data gas only
    l2_gas_price,                                   // price exposed, but no l2_gas_consumed
    overall_fee: tx_execution_output.execution_info.receipt.fee, // includes l2_gas cost!
    ...
})
```

`receipt.fee` is computed by `GasVector::cost()` as:
```
l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * (l2_gas_price + tip)
```

For a v3 `InvokeTransaction` with `AllResources` bounds (the format used since Starknet 0.14.0), `gas_mode = GasVectorComputationMode::All`, so `gas_vector.l2_gas > 0` for Sierra contracts. The `overall_fee` therefore exceeds `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` by exactly `l2_gas * (l2_gas_price + tip)`.

The TODO comment at line 104 explicitly acknowledges the missing field: *"TODO(Tzahi): Add l2_gas_consumed."*

### Impact Explanation
Any client (wallet, SDK, explorer) that reconstructs the fee from the documented formula `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` will compute a value lower than `overall_fee`. The gap is `l2_gas * (l2_gas_price + tip)`, which for real Sierra transactions on Starknet 0.14.x can be the dominant cost component (as shown in the test fixture where `l2_gas_consumed = 0xb56b6` at `l2_gas_price = 0x1dcd65000`). The RPC response is internally inconsistent: `overall_fee` cannot be reconstructed from the other returned fields.

### Likelihood Explanation
This is triggered by any v3 `InvokeTransaction` (or `Declare`/`DeployAccount` v3) targeting a Sierra contract on Starknet >= 0.14.0. This is the standard transaction type for all modern Starknet usage. No special attacker capability is required — any unprivileged user calling `starknet_estimateFee` or `starknet_simulateTransactions` triggers this.

### Recommendation
Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0.into()` in `tx_execution_output_to_fee_estimation`. Update the `overall_fee` doc comment to reflect the three-component formula. Update the OpenRPC spec accordingly.

### Proof of Concept
1. Submit a v3 `InvokeTransaction` with `AllResources` bounds targeting any Sierra contract to `starknet_estimateFee`.
2. Receive `FeeEstimation { gas_consumed, l1_gas_price, data_gas_consumed, l1_data_gas_price, l2_gas_price, overall_fee }`.
3. Compute `reconstructed = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`.
4. Assert `overall_fee > reconstructed`.
5. The difference equals `l2_gas * (l2_gas_price + tip)` — a value that cannot be derived from the returned fields because `l2_gas_consumed` is absent from the response.
