### Title
Legacy `tieredrewards` Module Account Unblocked for External Sends With No Withdrawal Path — (`app/app.go`)

### Summary

`moduleAccsAllowedToReceiveExternalFunds` explicitly sets `tieredrewardstypes.ModuleName: true`, causing `BlockedAddrs()` to return `false` for `authtypes.NewModuleAddress("tieredrewards")`. Any user can therefore submit a standard `MsgSend` that deposits `basecro` into this address. The `tieredrewards` keeper has no code path that ever reads from or spends the balance of the `tieredrewards` module account — all active keeper logic uses either the `rewards_pool` module account or per-position delegator addresses. Funds deposited into the legacy module account are permanently unrecoverable.

---

### Finding Description

`BlockedAddrs()` iterates `maccPerms` and sets each entry to `!moduleAccsAllowedToReceiveExternalFunds[acc]`: [1](#0-0) 

`moduleAccsAllowedToReceiveExternalFunds` contains: [2](#0-1) 

So `BlockedAddrs()["cosmos1...tieredrewards"] = false`, meaning the Cosmos SDK bank module's `MsgSend` handler will not reject a send to that address.

The `tieredrewards` keeper's entire active surface uses only two fund pools:

- **`types.RewardsPoolName`** ("rewards_pool") — for bonus reward payouts and BeginBlocker top-ups: [3](#0-2) [4](#0-3) 

- **Per-position delegator addresses** (derived as `authtypes.NewModuleAddress("tieredrewards/position/<id>")`) — for staking and withdrawal: [5](#0-4) [6](#0-5) 

No keeper method ever calls `SendCoinsFromModuleToAccount(ctx, types.ModuleName, ...)`, `SendCoinsFromModuleToModule(ctx, types.ModuleName, ...)`, `GetBalance(ctx, moduleAddr, ...)`, or any equivalent for the `tieredrewards` module account itself. The `maccPerms` entry for it carries `{authtypes.Staking}` — a permission from the old implementation — but no current code exercises it: [7](#0-6) 

`InitGenesis` only asserts the account exists; it does not fund or drain it: [8](#0-7) 

---

### Impact Explanation

Any `basecro` (or any denom) sent to `authtypes.NewModuleAddress("tieredrewards")` via `MsgSend` is permanently locked. There is no keeper message, governance spend path, or upgrade handler that reads from this account's balance. The funds cannot be recovered without a coordinated chain upgrade that adds an explicit drain path — which is a privileged, out-of-band action.

---

### Likelihood Explanation

Low. A user must deliberately address a `MsgSend` to the module account's bech32 address. There is no UI flow or protocol message that routes user funds there. The risk is highest in scenarios where the module account address is mistakenly published as a deposit address (e.g., for "funding the rewards pool"), since `RewardsPoolName` is the intended deposit target but both addresses are unblocked.

---

### Recommendation

Remove `tieredrewardstypes.ModuleName` from `moduleAccsAllowedToReceiveExternalFunds`. The legacy module account has no active spending path; it should be blocked like every other non-operational module account. Only `tieredrewardstypes.RewardsPoolName` needs to remain unblocked, as it is the intended external-funding target. [9](#0-8) 

---

### Proof of Concept

```go
// In a keeper integration test (unmodified Go/Cosmos test setup):

// Fund a user account with 1000 basecro, then:

    sdk.NewCoins(sdk.NewCoin("basecro", math.NewInt(1000))))
// assert: err == nil  (send succeeds — address is not blocked)

// Attempt any keeper withdrawal path — none exist for this account:
// app.TieredRewardsKeeper has no method referencing types.ModuleName balance.
// assert: legacyModuleAddr balance remains 1000 basecro forever.
```

func (app *ChainApp) BlockedAddrs() map[string]bool {
	blockedAddrs := make(map[string]bool)
	for acc := range maccPerms {
		blockedAddrs[authtypes.NewModuleAddress(acc).String()] = !moduleAccsAllowedToReceiveExternalFunds[acc]
	}

	return blockedAddrs
}
