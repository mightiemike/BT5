### Title
Accumulated Fast Withdrawal Fees Are Permanently Locked With No Claim Method — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool` accumulates ERC20 fast-withdrawal fees into the `fees` mapping on every `submitFastWithdrawal` call, but neither `BaseWithdrawPool` nor its concrete implementation `WithdrawPool` provides any function to withdraw or claim those accumulated fees. The tokens are permanently locked in the contract.

---

### Finding Description

In `submitFastWithdrawal`, a fee is computed and added to `fees[productId]`: [1](#0-0) [2](#0-1) 

The fee is either deducted from the withdrawal amount (when `sendTo == msg.sender`) or pulled from the caller via `safeTransferFrom`. In both cases the fee amount is credited to `fees[productId]` and the corresponding ERC20 tokens remain held by the `WithdrawPool` contract.

Searching the entire contract and its concrete child `WithdrawPool.sol` reveals no function that reads `fees[productId]` and transfers those tokens anywhere: [3](#0-2) 

`removeLiquidity` is the only owner-gated transfer function, but it operates on arbitrary `amount` from the pool's general token balance — it has no awareness of the `fees` mapping and does not zero it out. `WithdrawPool` adds only an `initialize` function: [4](#0-3) 

This is a direct structural analog to the BazaarManagerBlast issue: a setup/accumulation step is implemented (`configureClaimableGas` ↔ `fees[productId] += fee`) but the corresponding claim/withdraw action is entirely absent.

---

### Impact Explanation

Every successful `submitFastWithdrawal` call permanently locks ERC20 fee tokens inside `WithdrawPool`. The `fees` mapping grows monotonically and can never be decremented. The protocol loses all fast-withdrawal fee revenue; those tokens are irrecoverable without a contract upgrade.

---

### Likelihood Explanation

`submitFastWithdrawal` is a public, permissionless function callable by any address that holds the required tokens and a valid sequencer-signed withdrawal transaction. Fee accumulation occurs on every such call. The path is live and continuously exercised in normal protocol operation.

---

### Recommendation

Add an `onlyOwner`-gated function to `BaseWithdrawPool` that transfers the accumulated `fees[productId]` balance to a designated recipient and resets the mapping entry to zero, mirroring the pattern used in `removeLiquidity`. Example:

```solidity
function claimFees(uint32 productId, address recipient) external onlyOwner {
    int128 amount = fees[productId];
    require(amount > 0, "no fees");
    fees[productId] = 0;
    getToken(productId).safeTransfer(recipient, uint256(uint128(amount)));
}
```

---

### Proof of Concept

1. Any caller invokes `submitFastWithdrawal` with a valid sequencer-signed `WithdrawCollateral` transaction.
2. `fastWithdrawalFeeAmount` returns a non-zero `fee`.
3. `fees[productId] += fee` is executed — ERC20 tokens are now held by `WithdrawPool` and credited to the mapping.
4. No function in `BaseWithdrawPool` or `WithdrawPool` reads `fees[productId]` and transfers tokens out.
5. Repeat indefinitely; the balance grows and is never recoverable without an upgrade. [5](#0-4)
