### Title
Missing `validate_proof_size` call in `StatelessTransactionValidator::validate` allows oversized client-side proofs to bypass gateway admission - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate` invokes several proof-related checks for invoke transactions but never calls the defined `validate_proof_size` method. A transaction carrying a `proof` field larger than `max_proof_size` (480,000 bytes) is accepted by the gateway and forwarded to the mempool, bypassing the only size gate that exists for this field.

### Finding Description

`StatelessTransactionValidator::validate` dispatches the following checks for `RpcTransaction::Invoke`:

```rust
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;
    self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
}
``` [1](#0-0) 

`validate_proof_size` is defined in the same `impl` block and has its own dedicated config field and error variant:

```rust
fn validate_proof_size(
    &self,
    tx: &RpcInvokeTransaction,
) -> StatelessTransactionValidatorResult<()> {
    let RpcInvokeTransaction::V3(tx) = tx;
    let proof_size = tx.proof.0.len();
    if proof_size > self.config.max_proof_size {
        return Err(StatelessTransactionValidatorError::ProofTooLarge {
            proof_size,
            max_proof_size: self.config.max_proof_size,
        });
    }
    Ok(())
}
``` [2](#0-1) 

The error variant `ProofTooLarge` is declared: [3](#0-2) 

The config field `max_proof_size` is set to 480,000 in the default config and is serialized into the node config schema: [4](#0-3) [5](#0-4) 

`validate_tx_size` (which is called) checks `max_calldata_length` and `max_signature_length` — it does not cover the `proof` field, which is a separate byte-array field on `RpcInvokeTransactionV3`. The `proof` field is stripped during `RpcTransaction → InternalRpcTransaction` conversion: [6](#0-5) 

The proof is only verified cryptographically (not size-checked) in the asynchronous `spawn_proof_verification` task spawned during conversion: [7](#0-6) 

The gateway flow admits the transaction to the mempool before or concurrently with proof verification: [8](#0-7) 

### Impact Explanation

An unprivileged external caller can submit an `RpcInvokeTransactionV3` with `proof_facts` and `proof` both non-empty (satisfying `validate_proof_facts_and_proof_consistency`) and with `allow_client_side_proving: true` (the production default). The `proof` byte-array can be arbitrarily large — up to whatever the HTTP body limit allows — and will pass all stateless checks. The transaction is forwarded to the mempool. The asynchronous proof verifier then receives an oversized payload, consuming unbounded CPU and memory during `privacy_circuit_verify_v1::verify_recursive_circuit`. This constitutes gateway admission of an invalid transaction and enables a resource-exhaustion vector against the proof verification worker.

Impact: **High** — Mempool/gateway admission accepts invalid transactions before sequencing.

### Likelihood Explanation

The `allow_client_side_proving` flag defaults to `true` in production config: [9](#0-8) 

Any external caller with network access to the gateway HTTP endpoint can trigger this path. No privileged access, special account state, or prior on-chain setup is required. The attacker only needs to craft a valid-looking `RpcInvokeTransactionV3` with a large `proof` blob.

### Recommendation

Add `self.validate_proof_size(invoke_tx)?;` inside the `if let RpcTransaction::Invoke` branch in `StatelessTransactionValidator::validate`, immediately after `validate_proof_facts_and_proof_consistency`:

```rust
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;
    self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
    self.validate_proof_size(invoke_tx)?;   // <-- add this
}
``` [1](#0-0) 

### Proof of Concept

1. Start a node with default config (`allow_client_side_proving: true`, `max_proof_size: 480000`).
2. Construct an `RpcInvokeTransactionV3` with:
   - `proof_facts`: any valid non-empty `ProofFacts` (e.g., `create_valid_proof_facts_for_testing()`)
   - `proof`: a `Proof` whose inner byte-array length is 480,001 or more
   - All other fields valid (resource bounds, nonce, sender address, etc.)
3. Submit via `starknet_addInvokeTransaction` or the gateway's `add_tx` endpoint.
4. Observe: `validate` returns `Ok(())` — `ProofTooLarge` is never raised.
5. The transaction reaches `convert_rpc_tx_to_internal_and_executable_txs`, which spawns `spawn_proof_verification` with the oversized payload.
6. The proof verifier (`privacy_circuit_verify_v1::verify_recursive_circuit`) processes the oversized blob, consuming disproportionate resources.
7. The transaction is added to the mempool before the verification task completes.

The existing test `test_client_side_proving_flag` with `enabled_accepts_both` confirms that a transaction with non-empty proof data passes `validate` without any size check: [10](#0-9)

    let tx_validator = StatelessTransactionValidator { config };

    // Check for proof data before moving values.
    let has_proof_data = proof_facts.is_some() || proof.is_some();

    let rpc_tx_args = RpcTransactionArgs {
        proof_facts: proof_facts.unwrap_or_default(),
        proof: proof.unwrap_or_default(),
        ..Default::default()
    };

    let tx = rpc_tx_for_testing(TransactionType::Invoke, rpc_tx_args);

    // Disabled ⇒ reject txs with proof data.
    // Enabled  ⇒ always accept.

        assert_eq!(

            StatelessTransactionValidatorError::ClientSideProvingNotAllowed
        );
    } else {
        assert_matches!(tx_validator.validate(&tx), Ok(()));
