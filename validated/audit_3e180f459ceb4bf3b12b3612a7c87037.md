### Title
OracleValueStopLossExtension Uses Arithmetic Mean Instead of Geometric Mean for Mid Price, Miscalibrating the Stop-Loss Guard - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the arithmetic mean of bid and ask, while every other component in the protocol — the pool's swap engine and `PriceVelocityGuardExtension` — uses the geometric mean via `SwapMath.midAndSpreadFeeX64FromBidAsk`. Because AM ≥ GM always holds for bid ≠ ask, the stop-loss metrics are computed against a systematically biased mid price, causing the token0 value metric to be understated and the token1 value metric to be overstated. The resulting watermarks and drawdown floors are miscalibrated relative to the configured drawdown, allowing more LP value to leak than the pool admin intended before the guard triggers.

### Finding Description

In `_afterSwapOracleStopLoss`:

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

The pool's swap engine and `PriceVelocityGuardExtension` both use the geometric mean:

```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
``` [2](#0-1) [3](#0-2) 

The per-bin value metrics are:

```
metricToken0 = t0/shares + t1 * Q64 / (mid * shares)
metricToken1 = t0 * mid / (Q64 * shares) + t1/shares
``` [4](#0-3) 

Since `AM > GM` for any non-zero spread:
- `metricToken0` is **understated** (dividing by a larger mid reduces the t1 component)
- `metricToken1` is **overstated** (multiplying by a larger mid inflates the t0 component)

The watermarks ratchet up to these biased values. The drawdown floor is `hwm * (1 - drawdownE6)`. When the spread changes between the watermark-setting swap and a later swap, the AM/GM ratio shifts, causing the effective protection to diverge from the configured drawdown:

- **Spread narrows after watermark is set**: AM/GM decreases, so the current `metricToken0_am` rises relative to the floor set at the wider spread. The stop-loss for `zeroForOne` swaps becomes **less sensitive** — it fails to trigger when the true GM-based metric has already breached the configured floor. This is the dangerous direction: token0 value leaks beyond the configured drawdown without the guard firing.

- **Spread widens after watermark is set**: AM/GM increases, so `metricToken0_am` falls relative to the floor. The stop-loss becomes **more sensitive**, blocking legitimate swaps prematurely.

The test suite does not catch this because `_exposeStopLoss` always passes `priceX64, priceX64` as both bid and ask, making AM == GM in every test:

```solidity
extension.afterSwap(
    address(0), address(0), zeroForOne, 0, 0,
    _packSlot0(loBin), _packSlot0(hiBin),
    priceX64, priceX64,   // bid == ask → AM == GM
    0, 0, 0, ""
);
``` [5](#0-4) 

In production the pool enforces `bid < ask`: [6](#0-5) 

so the discrepancy is always present.

### Impact Explanation

The stop-loss guard is the primary on-chain protection for LP principal. When the guard is miscalibrated, LPs can lose more than the configured drawdown before the guard fires. The magnitude of the extra leakage is:

| Oracle spread | AM/GM − 1 | Extra leakage beyond 5% drawdown |
|---|---|---|
| 1% | ~0.0025% | ~0.0025% of bin value |
| 10% | ~0.6% | ~0.57% of bin value |
| 50% | ~3.3% | ~3.15% of bin value |

For pools with wide oracle spreads (e.g., illiquid RWA pairs), the miscalibration is material. The false-negative direction (stop-loss fails to fire) directly exposes LP principal to losses beyond the configured threshold.

### Likelihood Explanation

Every pool using `OracleValueStopLossExtension` with a non-zero oracle spread is affected. The spread is always non-zero in production (the pool reverts on `bid >= ask`). Any public swap that crosses a bin boundary triggers `afterSwap`, which recomputes the metric with the wrong mid. The effect accumulates whenever the oracle spread changes between swaps, which is normal oracle behavior.

### Recommendation

Replace the arithmetic mean with the geometric mean, consistent with the rest of the protocol:

```solidity
// Before (wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64),
    uint256(askPriceX64)
);
```

Update the test helper `_exposeStopLoss` to pass distinct bid and ask values so the test suite exercises the real production path.

### Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension`, drawdown = 5%, oracle spread = 10% (bid = 0.95·Q64, ask = 1.05·Q64).
2. Execute a swap. The extension computes:
   - AM = (0.95 + 1.05)/2 · Q64 = 1.00 · Q64
   - GM = sqrt(0.95 · 1.05) · Q64 ≈ 0.99875 · Q64
   - `metricToken0_am` is understated by ~0.125% relative to `metricToken0_gm`.
   - Watermark `hwm0` is set to the understated value.
3. Oracle spread narrows to 1% (bid = 0.995·Q64, ask = 1.005·Q64). A subsequent swap drains token0 until the true GM-based metric has fallen exactly 5% below `hwm0_gm`.
4. The AM-based metric is now above the AM-based floor (because the floor was set lower due to the wider spread at step 2), so the stop-loss does **not** revert.
5. LP principal has leaked beyond the configured 5% drawdown without the guard triggering. [7](#0-6)
