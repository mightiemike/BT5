### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Unpermissioned Caller to Bypass the Deposit Allowlist ŌĆö (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension` is designed to gate `addLiquidity` by the depositor's address. However, the `beforeAddLiquidity` hook silently discards the `sender` parameter (the actual caller) and validates `owner` (the position recipient) instead. Because `owner` is a free caller-supplied argument, any unpermissioned address can bypass the allowlist by passing any already-allowlisted address as `owner`.

### Finding Description

`MetricOmmPool.addLiquidity` calls the extension hook with two distinct addresses:

```
_beforeAddLiquidity(msg.sender /*sender*/, owner /*owner*/, ...)
``` [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both to the extension:

```
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and ignored. The guard is evaluated only against `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

`owner` is a free parameter supplied by the caller of `addLiquidity`. Any unpermissioned address can therefore pass an already-allowlisted address as `owner`, satisfy the check, and execute the deposit. The position is minted to the allowlisted `owner`, but the actual depositor (`sender`) is never validated.

This is directly inconsistent with `SwapAllowlistExtension.beforeSwap`, which correctly validates `sender` (the actual caller) and ignores `recipient`:

```solidity
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [4](#0-3) 

The admin-facing setter also names the parameter `depositor`, confirming the intent is to gate the caller, not the position owner:

```solidity
function setAllowedToDeposit(address pool_, address depositor, bool allowed) external onlyPoolAdmin(pool_) {
    allowedDepositor[pool_][depositor] = allowed;
``` [5](#0-4) 

### Impact Explanation

The deposit allowlist is rendered entirely inoperative. Any address ŌĆö regardless of allowlist status ŌĆö can call `addLiquidity` on a restricted pool by supplying any allowlisted address as `owner`. The pool admin's configured access control is bypassed by an unprivileged path, satisfying the **admin-boundary break** impact gate. Concretely:

- Pools intended to accept liquidity only from vetted/KYC'd participants accept deposits from arbitrary callers.
- Unallowlisted actors can manipulate bin state (shift `curBinIdx`, `curPosInBin`, bin balances) in pools where the admin explicitly restricted such access, potentially harming existing LPs through adverse price-position interactions.
- The `OracleValueStopLossExtension` watermarks and `PriceVelocityGuardExtension` state can be perturbed by deposits from actors the admin never intended to admit.

### Likelihood Explanation

Exploitation requires no special privilege, no flash loan, and no complex setup. Any EOA or contract can call `pool.addLiquidity(allowlistedAddress, ...)`. The allowlisted address is publicly readable from `allowedDepositor` or `allowAllDepositors`. Likelihood is **high**.

### Recommendation

Replace the unnamed first parameter with `sender` and validate it instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

### Proof of Concept

1. Pool P is deployed with `DepositAllowlistExtension` configured; `allowAllDepositors[P] = false`.
2. Admin calls `setAllowedToDeposit(P, alice, true)`. Alice is the only allowlisted depositor.
3. Attacker (Bob, not allowlisted) calls `P.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. `beforeAddLiquidity` is invoked with `sender = Bob`, `owner = alice`.
5. The guard evaluates `allowedDepositor[P][alice]` ŌåÆ `true` ŌåÆ no revert.
6. Bob's callback provides tokens; the position is minted to `alice`.
7. Bob has successfully deposited into a pool he was explicitly excluded from. The allowlist is fully bypassed.
