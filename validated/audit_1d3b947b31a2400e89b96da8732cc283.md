### Title
Sanctions Guard Checked Only at Slow Mode Submission, Not at Execution — Sanctioned Users Can Withdraw Funds via Queued Slow Mode Transactions — (File: `core/contracts/EndpointTx.sol`)

### Summary
`submitSlowModeTransactionImpl` enforces `requireUnsanctioned(sender)` at queue time, but `processSlowModeTransactionImpl` never re-validates this guard when the queued transaction is actually executed. Because `executeSlowModeTransaction()` is permissionlessly callable by anyone after the 3-day delay, a user who queues a `WithdrawCollateral` or `LinkSigner` slow mode transaction while unsanctioned, then becomes sanctioned within the delay window, can still have their transaction executed — bypassing the sanctions enforcement entirely. The state mutation (fund withdrawal or linked signer registration) persists with no on-chain mechanism to block it.

### Finding Description

**Submission guard (present):**

In `submitSlowModeTransactionImpl`, line 375:
```solidity
requireUnsanctioned(sender);
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({...});
``` [1](#0-0) 

The sanctions check gates queuing. Once the transaction is in `slowModeTxs`, no further guard is applied.

**Execution path — no re-check (root cause):**

In `processSlowModeTransactionImpl`, the `WithdrawCollateral` branch (lines 217–229) calls `clearinghouse.withdrawCollateral` with no `requireUnsanctioned`:
```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(...);
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(
        txn.sender, txn.productId, txn.amount, address(0), nSubmissions
    );
``` [2](#0-1) 

The `LinkSigner` branch (lines 232–239) similarly sets `linkedSigners` with no sanctions re-check:
```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

**Permissionless execution trigger:**

`executeSlowModeTransaction()` in `Endpoint.sol` is callable by any external address after `SLOW_MODE_TX_DELAY`:
```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);
    ...
}
``` [4](#0-3) 

The `_executeSlowModeTransaction` path only checks `txn.executableAt <= block.timestamp` — no sanctions re-check: [5](#0-4) 

**Exploit flow — `WithdrawCollateral`:**
1. User A (unsanctioned) calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload → passes `requireUnsanctioned` → queued in `slowModeTxs`.
2. User A is added to the Chainalysis sanctions list (e.g., after a regulatory action).
3. After 3 days, any address calls `executeSlowModeTransaction()`.
4. `processSlowModeTransactionImpl` executes `clearinghouse.withdrawCollateral` for User A with no sanctions re-check.
5. User A's collateral is transferred out of the protocol despite being sanctioned.

**Exploit flow — `LinkSigner` (persistent authorization bypass):**
1. User A (unsanctioned) queues a `LinkSigner` slow mode tx pointing to address B.
2. User A is sanctioned.
3. After 3 days, the tx executes → `linkedSigners[A's subaccount] = B`.
4. B can now sign sequencer-submitted transactions for A's subaccount indefinitely.
5. The sequencer's on-chain path (`processTransactionImpl`) has no sanctions check for `WithdrawCollateral` or other transaction types — it relies on `validateSignature` which accepts B as a valid signer for A. [6](#0-5) 

The `requireUnsanctioned` function is defined in `EndpointStorage.sol` and is only called at deposit and slow mode submission boundaries — never at execution: [7](#0-6) 

### Impact Explanation
A sanctioned user can withdraw collateral from the protocol by exploiting the 3-day slow mode window. The sanctions mechanism — intended to freeze sanctioned users' ability to move funds — is bypassed because the guard is only checked at queue time, not at execution time. For the `LinkSigner` variant, the sanctioned user gains a persistent authorized signer that can act on their behalf indefinitely, since `linkedSigners` state is written without re-checking sanctions and there is no automatic expiry or revocation path triggered by sanctions status.

### Likelihood Explanation
Realistic. Chainalysis sanctions list additions are not instantaneous — there is a known window between when a user is identified and when they are listed. A user who anticipates being sanctioned (e.g., after a public regulatory announcement) can pre-submit slow mode transactions. The 3-day delay is a fixed, known constant. `executeSlowModeTransaction()` is permissionless, so the attacker or any third party can trigger execution after the delay without sequencer cooperation.

### Recommendation
Re-validate `requireUnsanctioned` in `processSlowModeTransactionImpl` at execution time for all transaction types that move funds or mutate authorization state. Specifically, add a sanctions check against the subaccount owner's address for `WithdrawCollateral` and `LinkSigner` branches before executing the state change. The check should use `address(uint160(bytes20(txn.sender)))` to extract the owner address from the subaccount identifier.

### Proof of Concept
1. Deploy Nado on a local fork with a mock `ISanctionsList` that allows toggling sanctions status.
2. User A (address not sanctioned) calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload. Passes `requireUnsanctioned` at line 375. Transaction queued.
3. Add User A's address to the mock sanctions list.
4. Advance block timestamp by `SLOW_MODE_TX_DELAY` (3 days).
5. Any address calls `executeSlowModeTransaction()`.
6. `processSlowModeTransactionImpl` executes `clearinghouse.withdrawCollateral` for User A — no `requireUnsanctioned` call occurs.
7. User A's funds are withdrawn despite being sanctioned. Confirm by checking User A's collateral balance before and after step 6.
