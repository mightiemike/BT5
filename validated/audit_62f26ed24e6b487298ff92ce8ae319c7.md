Audit Report

## Title
Missing Zero-Address Check on `owner` in `addLiquidity` Permanently Locks LP Funds - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address and passes it directly to `LiquidityLib.addLiquidity`, which stores minted shares under `_positionBinShares` keyed by `(owner, salt, binIdx)`, without any zero-address validation. Because `removeLiquidity` enforces `msg.sender == owner` and `msg.sender` can never be `address(0)` in the EVM, any position minted with `owner = address(0)` is permanently irrecoverable, locking the deposited tokens inside the pool forever.

## Finding Description
`MetricOmmPool.addLiquidity` (L182–196) is a public, permissionless `external` function with no access control beyond `nonReentrant`. It accepts `owner` as a raw `address` parameter and passes it unvalidated into `LiquidityLib.addLiquidity` (L192–194), which writes shares to `_positionBinShares[_positionBinKey(owner, salt, binIdx)]` (LiquidityLib.sol L72–74) and increments `binTotals.scaledToken0`/`scaledToken1` after the callback settles tokens.

`removeLiquidity` (L199–212) enforces `if (msg.sender != owner) revert NotPositionOwner();` at L206. Since the EVM never sets `msg.sender` to `address(0)`, a position keyed under `address(0)` can never satisfy this check. The deposited tokens are absorbed into `binTotals` and credited to an unreachable position key.

The periphery router `MetricOmmPoolLiquidityAdder` does guard against this via `_validateOwner` (L247–249), which reverts with `InvalidPositionOwner` when `owner == address(0)`. However, this guard exists only in the periphery and is not enforced by the core pool. Any caller that bypasses the router — including direct pool calls, custom integrations, or buggy routers — can trigger the vulnerability. The optional `DepositAllowlistExtension` gates by `owner` identity but does not reject `address(0)` specifically, and pools without any extension have no protection at all.

The factory's `_validatePoolParameters` (MetricOmmPoolFactory.sol L548–554) and `setPoolAdminFeeDestination` (L444) both guard their respective address parameters against zero, establishing the pattern — but no equivalent guard exists in `addLiquidity`.

## Impact Explanation
Any tokens transferred into the pool via the `IMetricOmmModifyLiquidityCallback` during an `addLiquidity` call with `owner = address(0)` are permanently locked. `binTotals` is updated to include the deposited amounts (pool accounting remains consistent), but the LP's principal is credited to an unreachable position and can never be withdrawn. This is a direct, irreversible loss of user principal above Sherlock thresholds, matching the "Critical/High direct loss of user principal" allowed impact.

## Likelihood Explanation
The pool is a low-level contract explicitly designed to be called by routers and integrators. Any router or integration that fails to validate its `owner` argument — or that passes a computed/default address that resolves to zero — will silently lock user funds with no recovery path. A direct call to the pool with `owner = address(0)` requires no special privilege. Likelihood is medium: it requires an integration error or a deliberate direct call, but the consequence is irreversible and the pool provides no safety net.

## Recommendation
Add a zero-address check at the top of `addLiquidity` in `MetricOmmPool.sol`, mirroring the pattern already applied in `_validatePoolParameters`:

```solidity
if (owner == address(0)) revert InvalidOwner();
```

This should be placed before the `_beforeAddLiquidity` hook call (L191) so that no extension logic executes for an invalid owner.

## Proof of Concept
1. Deploy a pool without a `DepositAllowlistExtension` (or with `allowAllDepositors = true`).
2. Call `pool.addLiquidity(address(0), salt, deltas, callbackData, "")` directly, where `deltas` specifies non-zero shares in one or more bins.
3. The callback fires; the caller transfers `amount0Added` and `amount1Added` tokens to the pool.
4. `binTotals.scaledToken0`/`scaledToken1` are incremented; shares are stored under `keccak256(abi.encode(address(0), salt, binIdx))`.
5. Any subsequent call to `pool.removeLiquidity(address(0), salt, deltas, "")` reverts with `NotPositionOwner` because `msg.sender != address(0)` is always true.
6. The deposited tokens remain in the pool forever, credited to an unreachable position.

Foundry test plan: fork the pool, call `addLiquidity` with `owner = address(0)` and valid `deltas`, assert tokens are transferred in, then assert `removeLiquidity(address(0), ...)` reverts with `NotPositionOwner` and pool token balances are unchanged.
