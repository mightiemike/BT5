### Title
Premature Socialization via `lastLiquidationFees` Subtraction in `_finalizeSubaccount` — (`core/contracts/ClearinghouseLiq.sol`)

### Summary
`_finalizeSubaccount` subtracts `lastLiquidationFees` from `insurance` before computing whether the insurance fund can cover the liquidatee's remaining debt. When the full insurance is sufficient but the reduced value is not, spot socialization is triggered prematurely, spreading losses to innocent users even though the protocol had enough funds to absorb them.

### Finding Description
In `_handleLiquidationPayment`, after every regular liquidation: [1](#0-0) 

`insurance` is incremented by `v.liquidationFees` and `lastLiquidationFees` is set to the same value.

In `_finalizeSubaccount`, the effective insurance used for all socialization decisions is: [2](#0-1) 

This reduces `v.insurance` to `I_old` (pre-fee level). The spot socialization gate is: [3](#0-2) 

The fees are only added back **after** socialization has already been triggered: [4](#0-3) 

The design intent (per the comment at lines 581–585) is to prevent fees from *blocking* socialization when insurance is genuinely insufficient. However, the subtraction also fires when insurance is genuinely *sufficient*, causing false socialization. [5](#0-4) 

### Impact Explanation
When `I_old < D ≤ I_old + fees` (where `D` is the liquidatee's remaining quote debt), the insurance fund has enough to cover the full debt but the subtraction makes it appear insufficient. `spotEngine.socializeSubaccount(liquidatee)` is called, spreading `D - I_old` of losses across all other users' balances. The insurance fund retains the fees (line 410 adds them back), but the socialization is irreversible — innocent users have already absorbed bad debt that the protocol could have paid.

### Likelihood Explanation
A liquidator can deliberately engineer this by:
1. Closing all of a liquidatee's positions via regular liquidations until only a negative USDC balance remains.
2. Submitting a final regular liquidation that generates `fees > D - I_old` (achievable by choosing a large enough position or a product with a high liquidation penalty).
3. Immediately submitting a finalization transaction in the same sequencer batch.

The endpoint processes both in order via `processTransactionImpl`: [6](#0-5) 

The `liquidateSubaccount` call is `onlyEndpoint` and uses `delegatecall` to `ClearinghouseLiq`: [7](#0-6) 

Both `insurance` and `lastLiquidationFees` are shared storage variables: [8](#0-7) 

The liquidator does not need any special privileges — only the ability to submit signed liquidation transactions, which is a standard production path.

### Recommendation
The `lastLiquidationFees` subtraction should only reduce `v.insurance` to a floor of zero (i.e., never below zero), and the socialization gate should compare against the **full** `insurance` value, not the reduced one. Alternatively, the subtraction should be applied only when `v.insurance` after subtraction is still negative (i.e., only when insurance is genuinely insufficient), preserving the original intent without causing false socialization:

```solidity
v.insurance = insurance;
// Only exclude lastLiquidationFees if insurance is already insufficient
int128 effectiveInsurance = v.insurance - lastLiquidationFees;
v.canLiquidateMore = (quoteBalance.amount + effectiveInsurance) > 0;
// But use full v.insurance for actual coverage decisions
```

Or more simply: use `insurance` (not `insurance - lastLiquidationFees`) for the `insuranceCover` and socialization gate, and only apply the subtraction to `canLiquidateMore`.
