Audit Report

## Title
`SwapAllowlistExtension` Allowlist Bypass via Router: Any User Can Swap on Allowlisted Pools — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to its own `msg.sender` — the immediate caller of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates a swap, the pool's `msg.sender` is the router, not the end user. A pool admin who allowlists the router to enable legitimate users to access multi-hop routing simultaneously opens the gate to every unprivileged user, because the router is a public, permissionless contract. The allowlist invariant is completely broken for all router-mediated swaps.

## Finding Description

`SwapAllowlistExtension.beforeSwap` performs its identity check as:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct) and `sender` is the first argument forwarded from `MetricOmmPool.swap`:

```solidity
_beforeSwap(
  msg.sender,   // ← whoever called pool.swap()
  recipient,
  ...
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly without forwarding the original `msg.sender`:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
  );
``` [3](#0-2) 

The pool therefore passes `address(router)` as `sender` to `beforeSwap`. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The same path exists for `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165). [4](#0-3) 

## Impact Explanation

The `SwapAllowlistExtension` is documented as "Gates `swap` by swapper address, per pool." [5](#0-4) 

Once the router is allowlisted, the gate is completely ineffective: any address can execute swaps against a pool intended to be restricted. Pools relying on the allowlist for compliance, rate-limiting, or LP protection lose that guarantee entirely for every router-mediated swap. This constitutes broken core pool functionality — the allowlist invariant is violated for all router-mediated swaps, which is the primary user-facing swap path.

## Likelihood Explanation

The pool admin must allowlist the router to let allowlisted users benefit from multi-hop routing, slippage protection, and deadline enforcement. There is no mechanism to say "allow the router only for specific end-users." The admin is forced to choose between blocking the router entirely (degraded UX for legitimate users) or allowlisting it (full bypass for everyone). A pool admin who chooses usability inadvertently enables the bypass. This is a realistic, semi-trusted trigger with no malicious setup required.

## Recommendation

The extension must verify the economic actor, not the immediate caller. Two concrete approaches:

1. **Pass the original user through the router.** The router forwards the real `msg.sender` in `extensionData`; the extension decodes and checks it. Requires coordinated change to router and extension.
2. **Adopt the same pattern as `DepositAllowlistExtension`**, which gates on `owner` (the economic beneficiary). For swaps, the analogous field would be the `recipient` or a user-supplied identity in `extensionData`.
3. **Detect known router addresses in the extension** and apply a secondary check requiring the real user to be encoded in `extensionData` when `sender` is a known router.

## Proof of Concept

```solidity
// Pool admin sets up allowlist: only Alice is allowed to swap
swapAllowlist.setAllowedToSwap(pool, alice, true);

// Admin also allowlists the router so Alice can use it
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Mallory (not allowlisted) calls the router directly
// pool.swap() sees msg.sender = router
// extension checks allowedSwapper[pool][router] → true
// Mallory's swap succeeds despite not being on the allowlist
vm.prank(mallory);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    recipient: mallory,
    amountIn: 1000e18,
    amountOutMinimum: 0,
    zeroForOne: true,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// → succeeds; allowlist completely bypassed
``` [6](#0-5) [7](#0-6)
