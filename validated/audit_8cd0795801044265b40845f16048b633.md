### Title
Unchecked ERC20 `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Draining usdcE Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` performs a raw `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` call without checking its return value. If the USDC token returns `false` on failure instead of reverting, an unprivileged caller can drain all usdcE held in a `DirectDepositV1` address without providing any USDC.

---

### Finding Description

In `ContractOwner.replaceUsdcEWithUsdc`, the function is designed to atomically swap usdcE held in a `DirectDepositV1` contract for USDC from the caller:

```solidity
// core/contracts/ContractOwner.sol, lines 608–620
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    address payable directDepositV1 = directDepositV1Address[subaccount];
    require(directDepositV1 != address(0), "no dda");
    address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
    address usdc  = 0x2D270e6886d130D724215A266106e6832161EAEd;
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← unchecked
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);
    }
}
``` [1](#0-0) 

The `transferFrom` call at line 616 is a direct call on `IERC20Base` — not routed through `ERC20Helper.safeTransferFrom`, which is the safe wrapper used everywhere else in the protocol. [2](#0-1) 

The `ERC20Helper` library is already imported and `using ERC20Helper for IERC20Base` is declared in `ContractOwner`, making the safe path available but unused here. [3](#0-2) 

The subsequent two calls — `DirectDepositV1.withdraw` and `safeTransfer` — both execute unconditionally after the unchecked `transferFrom`. `DirectDepositV1.withdraw` transfers usdcE from the DDA to `ContractOwner` (which is the DDA's owner), and `safeTransfer` then forwards that usdcE to `msg.sender`. [4](#0-3) 

---

### Impact Explanation

**Impact: High**

If the USDC token at the hardcoded address (`0x2D270e6886d130D724215A266106e6832161EAEd` on Ink chain, chainid 57073) returns `false` on a failed `transferFrom` (e.g., insufficient allowance or balance) rather than reverting, the execution flow continues:

1. USDC is **not** transferred from the attacker to `directDepositV1` (silent failure).
2. All usdcE held in `directDepositV1` is transferred to `ContractOwner`.
3. All usdcE is forwarded to the attacker via `safeTransfer`.

The attacker receives the full usdcE balance of any targeted `DirectDepositV1` contract at zero cost. This is a direct, irreversible asset loss for the users whose funds are held in those DDA contracts.

---

### Likelihood Explanation

**Likelihood: Low**

The function is restricted to `block.chainid == 57073` (Ink chain). The exploitability depends on whether the specific USDC deployment at the hardcoded address returns `false` on failure rather than reverting. Circle's canonical USDC implementations typically revert on failure. However, the code pattern is unconditionally incorrect — the return value is never checked — and the protocol's own `ERC20Helper.safeTransferFrom` exists precisely to handle this class of token. Any non-standard USDC deployment or future upgrade that returns `false` on failure would make this immediately exploitable by any unprivileged caller.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's existing `ERC20Helper.safeTransferFrom`, which is already imported and bound via `using ERC20Helper for IERC20Base`:

```solidity
// Before (vulnerable):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [5](#0-4) 

Similarly, the unchecked `approve` calls in `depositInsurance` (line 254) and `wrapVaultAsset` (lines 530–531) should be reviewed, though their impact is limited to DoS rather than fund loss. [6](#0-5) 

---

### Proof of Concept

**Precondition:** A `DirectDepositV1` contract exists for some `subaccount` and holds a non-zero usdcE balance. The USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on a failed `transferFrom`.

**Attack steps (on Ink chain, chainid 57073):**

1. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance granted to `ContractOwner`.
2. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, balance)` returns `false` (no allowance). Return value is not checked; execution continues.
3. `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes: usdcE balance is transferred from `directDepositV1` to `ContractOwner`.
4. `IERC20Base(usdcE).safeTransfer(attacker, balance)` executes: usdcE is transferred from `ContractOwner` to the attacker.

**Result:** Attacker receives the full usdcE balance of `directDepositV1` without providing any USDC. [7](#0-6)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
