### Title
SPV Verification Window Permanently Closes for Transactions Near the GC Boundary After Protocol Resumes from Pause - (File: `contract/src/lib.rs`)

---

### Summary

The `BtcLightClient` contract exposes a time-bounded SPV verification window defined by `gc_threshold`. Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are gated by `#[pause]`, blocking all verification during a pause. When the contract resumes, the relayer submits the backlog of Bitcoin blocks, triggering GC that permanently removes old block headers. Transactions in blocks that were verifiable before the pause may become permanently unverifiable after the pause ends, because the pause duration is not accounted for in the GC window.

---

### Finding Description

The contract stores only the most recent `gc_threshold` Bitcoin block headers. The SPV verification functions enforce this window: a transaction can only be verified if its containing block is still present in `headers_pool` and indexed in `mainchain_header_to_height`.

Three interacting mechanisms create the vulnerability:

**1. Verification is blocked during pause.**
Both SPV entry points carry `#[pause]`: [1](#0-0) [2](#0-1) 

**2. Block submission is also blocked during pause.**
`submit_blocks` carries `#[pause]`, so the relayer cannot advance the on-chain chain tip while the contract is paused: [3](#0-2) 

**3. GC runs automatically when submission resumes.**
Every call to `submit_blocks` unconditionally calls `run_mainchain_gc` with the number of newly submitted headers as the batch size: [4](#0-3) 

`run_mainchain_gc` removes the oldest stored block headers whenever the stored count exceeds `gc_threshold`: [5](#0-4) 

The pause duration is never subtracted from the GC window. When the relayer catches up after a pause by submitting K backlog blocks, GC removes up to K of the oldest stored headers. Any block that was within the `gc_threshold` window before the pause but falls outside it after the catch-up is permanently deleted from `headers_pool` and `mainchain_height_to_header`.

---

### Impact Explanation

A downstream contract (e.g., a BTC bridge) that needs to call `verify_transaction_inclusion_v2` for a BTC deposit transaction confirmed near the tail of the `gc_threshold` window will find the call permanently reverting after the pause ends. The block is no longer in `headers_pool`, so the lookup panics: [6](#0-5) 

The user cannot recover: the block is gone from contract storage and cannot be re-submitted (it would be treated as a fork block below the GC floor). The SPV proof is permanently invalidated, which can translate to a stuck or lost bridge deposit in any consuming protocol.

---

### Likelihood Explanation

**Low.** The scenario requires:
- A pause event of non-trivial duration (enough real Bitcoin blocks mined during the pause to push the target block outside `gc_threshold`).
- The target transaction's block must be near the oldest end of the stored window at pause time.

Bitcoin mines ~144 blocks/day. With the recommended `gc_threshold = 52704` (~1 year of blocks), a pause would need to last many months to affect most users. However, for transactions confirmed near the GC tail at pause time, even a pause of a few days (hundreds of blocks) is sufficient to cause permanent loss of verifiability.

---

### Recommendation

Account for the paused duration within the GC window, analogous to the original report's recommendation. Concretely:

1. Record the block height at which the contract was paused (`paused_at_height`).
2. When the contract is unpaused and the relayer resumes, extend the effective `gc_threshold` by the number of Bitcoin blocks produced during the pause before running GC.
3. Alternatively, document clearly that any transaction whose block is within `K` blocks of the GC tail at pause time risks permanent loss of verifiability, where `K` is the number of Bitcoin blocks expected during the pause window, and advise downstream integrators accordingly.

---

### Proof of Concept

**Setup**: `gc_threshold = 1000`. Mainchain tip is at height 10000. Oldest stored block is at height 9001. A user's BTC deposit transaction is confirmed at height 9010 (near the GC tail, 9 blocks above the floor).

**Step 1**: `PauseManager` pauses the contract. `verify_transaction_inclusion_v2` is now blocked.

**Step 2**: Bitcoin mines 50 real blocks (heights 10001–10050) during the pause. The NEAR contract does not advance.

**Step 3**: Contract is unpaused. The relayer calls `submit_blocks` with the 50 backlog headers.

**Step 4**: `submit_blocks` calls `run_mainchain_gc(50)`. GC removes heights 9001–9050 from `headers_pool` and `mainchain_height_to_header`. The new oldest stored block is at height 9051.

**Step 5**: The user's block at height 9010 has been deleted. Calling `verify_transaction_inclusion_v2` now panics at: [7](#0-6) 

The SPV proof is permanently unverifiable. The pause duration was not accounted for in the GC window.
