### Title
Raw `IIERC20Base.approve` Call in `DirectDepositV1.creditDeposit` Permanently Blocks USDT Collateral Deposits - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` using a high-level Solidity interface call that expects a `bool` return value. USDT on Ethereum mainnet does not return a value from `approve`. The ABI decoder reverts when it finds no return data, permanently blocking any USDT collateral deposit routed through the `DirectDepositV1` mechanism. A secondary instance exists in `ContractOwner.wrapVaultAsset()`.

---

### Finding Description

`DirectDepositV1` is a per-subaccount deposit address contract. When a user sends tokens to their DDA and calls `creditDeposit()`, the function iterates over all registered spot products, checks the DDA's balance of each token, and attempts to approve the `endpoint` to spend that balance before calling `depositCollateralWithReferral`.

The approval is performed via a high-level Solidity call through the `IIERC20Base` interface:

```solidity
// DirectDepositV1.sol line 11
function approve(address spender, uint256 amount) external returns (bool);
```

```solidity
// DirectDepositV1.sol lines 89-92
IIERC20Base token = IIERC20Base(tokenAddr);
uint256 balance = token.balanceOf(address(this));
if (balance != 0) {
    token.approve(address(endpoint), balance);   // <-- reverts for USDT
```

Because the interface declares `approve` as returning `bool`, the Solidity compiler emits ABI-decoding logic that reads 32 bytes from the return data. USDT's `approve` on Ethereum mainnet returns nothing (zero-length return data). The ABI decoder reverts, and the entire `creditDeposit` call fails.

`ERC20Helper` — the library used elsewhere in the codebase for safe token interactions — implements `safeTransfer` and `safeTransferFrom` using low-level `.call()` with the pattern `data.length == 0 || abi.decode(data, (bool))`, which correctly handles non-returning tokens. However, **no equivalent `safeApprove` exists** in `ERC20Helper`, and `creditDeposit` does not use `ERC20Helper` at all.

A secondary instance exists in `ContractOwner.wrapVaultAsset()` at lines 530–531, where `assetToken.approve(tokenAddr, 0)` and `assetToken.approve(tokenAddr, assetBalance)` are called via the `IERC20Base` interface (also declared as returning `bool`) on the underlying asset of an ERC4626 vault. If that underlying asset is USDT, both calls revert.

---

### Impact Explanation

Any user who sends USDT to their `DirectDepositV1` address and calls `creditDeposit()` will have the call revert. The USDT remains stuck in the DDA contract. The user cannot deposit USDT collateral into their subaccount through this path. Since `creditDeposit` iterates all products and reverts on the first non-zero USDT balance, it also blocks deposits of any other tokens held in the same DDA in the same transaction. This is a complete denial of the deposit flow for USDT, a core supported collateral token.

---

### Likelihood Explanation

`creditDeposit()` is an `external` function with no access control — any caller can invoke it. The `ContractOwner.creditDepositV1()` wrapper also calls it with no restriction. Any user who follows the standard DDA deposit flow for USDT will trigger this revert immediately. Likelihood is high given USDT is a primary collateral asset.

---

### Recommendation

Add a `safeApprove` function to `ERC20Helper` using the same low-level `.call()` pattern already used for `safeTransfer` and `safeTransferFrom`:

```solidity
function safeApprove(IERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_APPROVE_FAILED
    );
}
```

Replace the raw `token.approve(...)` calls in `DirectDepositV1.creditDeposit` and `ContractOwner.wrapVaultAsset` with this safe variant. For USDT compatibility, also reset the allowance to zero before setting a new value (USDT reverts if the current allowance is non-zero).

---

### Proof of Concept

**Root cause — `DirectDepositV1.sol`:** [1](#0-0) 

The interface declares `approve` as returning `bool`. [2](#0-1) 

The high-level call at line 92 causes the Solidity ABI decoder to expect 32 bytes of return data. USDT returns nothing, so the decoder reverts. [3](#0-2) 

**Contrast with safe transfer pattern in the same codebase:**

`ERC20Helper.safeTransfer` uses `data.length == 0 || abi.decode(data, (bool))` to tolerate missing return values, but no equivalent exists for `approve`. [4](#0-3) 

**Secondary instance — `ContractOwner.wrapVaultAsset`:** [5](#0-4) 

Both `approve` calls on `IERC20Base` (which also declares `approve` as returning `bool`) will revert if the underlying vault asset is USDT. [6](#0-5)
