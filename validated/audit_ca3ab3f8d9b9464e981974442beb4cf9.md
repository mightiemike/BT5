### Title
NFT Permanently Locked After Transfer: `EditNFT`/`BurnNFT` Require Simultaneous Owner AND Creator — (`x/nft/keeper/keeper.go`)

### Summary
`EditNFT` and `BurnNFT` both enforce two sequential authorization checks against the **same** address: `IsOwner` (lines 93, 146) and `IsDenomCreator` (lines 98, 151). After a legitimate `TransferNFT`, no single address can satisfy both conditions simultaneously, permanently locking the NFT in an uneditable and unburnable state.

### Finding Description

`EditNFT` enforces: [1](#0-0) 

`BurnNFT` enforces the identical pattern: [2](#0-1) 

`IsOwner` checks that the caller is the current NFT owner: [3](#0-2) 

`IsDenomCreator` checks that the caller is the denom creator (immutable after `IssueDenom`): [4](#0-3) 

`TransferOwner`, by contrast, only requires `IsOwner` — no creator check — so transfers succeed freely: [5](#0-4) 

This creates a permanent deadlock after any transfer:

| Caller | `IsOwner` | `IsDenomCreator` | Result |
|---|---|---|---|
| addrA (creator, ex-owner) | FAIL | PASS | `ErrUnauthorized` |
| addrB (current owner, non-creator) | PASS | FAIL | `ErrUnauthorized` |

### Impact Explanation
After `addrA` issues a denom, mints an NFT, and transfers it to `addrB`:
- `addrB` (owner) cannot call `MsgBurnNFT` — fails `IsDenomCreator`
- `addrA` (creator) cannot call `MsgBurnNFT` — fails `IsOwner`
- Same deadlock applies to `MsgEditNFT`

The only bypass is `BurnNFTUnverified`, which is exclusively called from the IBC module path and is not user-accessible: [6](#0-5) 

The NFT is permanently locked: uneditable and unburnable, with no on-chain recovery path for any normal user.

### Likelihood Explanation
Any `MsgTransferNFT` on a non-IBC NFT triggers this condition. This is a normal, expected user action (gifting, selling, marketplace transfer). The deadlock is deterministic and immediate — 100% reproducible on any transfer.

### Recommendation
Separate the authorization logic. `BurnNFT` should require **either** `IsOwner` (current owner can burn their own NFT) **or** `IsDenomCreator` (creator can burn from any holder). `EditNFT` should similarly be relaxed to allow the current owner to edit metadata, or define a clear policy and enforce only one check.

### Proof of Concept

```go
// 1. addrA issues denom → addrA is creator
k.IssueDenom(ctx, "testdenom", "Test", "", "", addrA)

// 2. addrA mints NFT to addrA
k.MintNFT(ctx, "testdenom", "token1", "name", "uri", "data", addrA, addrA)

// 3. addrA transfers NFT to addrB
k.TransferOwner(ctx, "testdenom", "token1", addrA, addrB)
// → succeeds (only IsOwner required)

// 4. addrA tries to EditNFT → fails IsOwner (no longer owner)
err := k.EditNFT(ctx, "testdenom", "token1", "newname", "", "", addrA)
// → ErrUnauthorized: "addrA is not the owner of testdenom/token1"

// 5. addrB tries to EditNFT → fails IsDenomCreator (not creator)
err = k.EditNFT(ctx, "testdenom", "token1", "newname", "", "", addrB)
// → ErrUnauthorized: "addrB is not the creator of testdenom"

// 6. addrB tries to BurnNFT → fails IsDenomCreator
err = k.BurnNFT(ctx, "testdenom", "token1", addrB)
// → ErrUnauthorized: "addrB is not the creator of testdenom"
// NFT is now permanently locked.
```

```go
// BurnNFTUnverified deletes a specified NFT without verifying if the owner is the creator of denom
// Needed for IBC transfer of NFT
func (k Keeper) BurnNFTUnverified(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
```
