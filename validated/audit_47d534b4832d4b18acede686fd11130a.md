Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual user, enabling allowlist bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed from the pool, which is always `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. If the pool admin allowlists the router address (the only way to permit router-based swaps on a curated pool), every unprivileged user can bypass the per-user allowlist by routing through the router.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` (or any `exact*` variant).
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — `msg.sender` to the pool is the **router address**.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, forwarding the **router address** as `sender`.
4. `ExtensionCalling._beforeSwap` encodes that `sender` and dispatches to the configured extension.
5. `SwapAllowlistExtension.beforeSwap` executes the check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool (correct), but `sender` = **router address**, not the actual user.

**Root cause:** The pool passes its own `msg.sender` (the immediate caller) as the `sender` argument to extensions. There is no mechanism for the router to forward the original user's address into this argument.

**Exploit flow:**
- Pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists specific user addresses.
- To allow those users to also swap via the router, the admin must allowlist the router address: `setAllowedToSwap(pool, router, true)`.
- Once the router is allowlisted, any unprivileged user calls `router.exactInputSingle(...)` targeting the curated pool. The extension sees `sender = router`, which is allowlisted, and the check passes — the non-allowlisted user's swap executes.

**Why existing guards fail:** The `SwapAllowlistExtension.beforeSwap` check is structurally correct but operates on the wrong identity. The pool's `_beforeSwap` always passes `msg.sender` (the immediate caller) as `sender`; there is no separate "originator" field. The router has no way to inject the real user's address into that slot.

## Impact Explanation
Any unprivileged user can trade on a pool that the pool admin intended to restrict to a specific set of allowlisted addresses. This breaks the curation invariant of the `SwapAllowlistExtension` and constitutes a direct policy bypass on curated pools. Depending on the pool's purpose (e.g., institutional-only, KYC-gated, or whitelist-only liquidity), this allows unauthorized parties to extract value from LP positions that were only meant to serve approved counterparties — a direct loss of curation integrity and potential fund impact for LPs.

## Likelihood Explanation
The precondition — router being allowlisted — is the natural and expected admin action when a curated pool is meant to be accessible via the standard periphery. There is no documentation warning against this, and the system provides no alternative mechanism to allowlist individual users through the router. Any pool admin who configures a swap allowlist and also wants router support will trigger this condition. The exploit is then trivially repeatable by any unprivileged caller with no special capability.

## Recommendation
Pass the original user's address through the swap path so extensions can gate on the true economic actor. One approach: add an `originator` parameter to `pool.swap()` that the router populates with `msg.sender` before calling the pool, and have `_beforeSwap` forward that value as `sender` to extensions. Alternatively, `SwapAllowlistExtension` could accept a signed proof of the real user's identity via `extensionData`. At minimum, document clearly that allowlisting the router grants access to all router users, and provide a separate per-user mechanism for router-mediated swaps.

## Proof of Concept

```solidity
// Setup: curated pool with SwapAllowlistExtension
// Admin allowlists alice directly AND allowlists the router (to let alice use it)
swapExtension.setAllowedToSwap(pool, alice, true);
swapExtension.setAllowedToSwap(pool, address(router), true);

// Attack: bob (not allowlisted) calls through the router
vm.startPrank(bob);
token0.approve(address(router), type(uint256).max);
// Extension sees sender = router (allowlisted) → passes
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: address(token0),
    recipient: bob,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// bob's swap succeeds despite not being on the allowlist
vm.stopPrank();
```
