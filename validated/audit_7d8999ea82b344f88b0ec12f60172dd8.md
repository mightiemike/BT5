### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Allows Draining usdcE Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` at line 616 without checking the return value. If the USDC token at the hardcoded address returns `false` on failure instead of reverting, the function continues execution, withdrawing usdcE from the `DirectDepositV1` contract and sending it to the caller — without the caller ever having provided USDC.

---

### Finding Description

The function `replaceUsdcEWithUsdc` is designed as a token-swap helper: the caller provides USDC, which is deposited into a `DirectDepositV1` contract, and in return the caller receives the usdcE that was previously held there.

The critical sequence is:

```solidity
// ContractOwner.sol line 614–619
uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
if (balance > 0) {
    IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← unchecked
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
    IERC20Base(usdcE).safeTransfer(msg.sender, balance);
}
```

The `transferFrom` on line 616 returns a `bool` per the `IERC20Base` interface, but the return value is silently discarded. If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (Ink chain) returns `false` on failure (e.g., insufficient allowance, paused state, or any non-reverting failure mode), execution falls through to lines 617–618, which unconditionally withdraw usdcE from `directDepositV1` and send it to `msg.sender`.

The codebase already provides `ERC20Helper.safeTransferFrom` — used correctly in `EndpointStorage.chargeSlowModeFee` and `EndpointStorage.safeTransferFrom` — which wraps the call and reverts on a false return. The inconsistency is that line 616 bypasses this safe wrapper entirely.

The function has **no access control** (`onlyOwner` or similar is absent), so any unprivileged caller can invoke it for any subaccount that has a non-zero usdcE balance in its `DirectDepositV1` contract.

---

### Impact Explanation

If `transferFrom` fails silently:
- `directDepositV1` never receives USDC.
- `DirectDepositV1(directDepositV1).withdraw(usdcE)` still executes, pulling usdcE out of the deposit contract.
- `safeTransfer(msg.sender, balance)` still executes, sending that usdcE to the attacker.

Net result: the attacker receives usdcE for free. The `DirectDepositV1` contract loses its entire usdcE balance with no USDC compensation. This is a direct, permanent asset loss for the subaccount owner whose deposit contract is drained.

---

### Likelihood Explanation

The function is publicly callable by any address on Ink chain (chainid 57073). The only precondition is that a `DirectDepositV1` contract for some subaccount holds a non-zero usdcE balance — a normal operational state. The trigger depends on whether the specific USDC token at the hardcoded address can return `false` rather than revert (e.g., under a paused or non-standard state). Given that the codebase explicitly anticipates this class of token behavior (hence `ERC20Helper.safeTransferFrom` exists), the risk is concrete and the missing check is an oversight rather than an intentional design choice.

---

### Recommendation

Replace the raw `transferFrom` call at line 616 with the `safeTransferFrom` helper already available via `ERC20Helper`, consistent with how all other token transfers in the protocol are handled:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

This requires `ContractOwner` to use the `ERC20Helper` library for `IERC20Base`, which is already imported and applied via `using ERC20Helper for IERC20Base` at line 24.

---

### Proof of Concept

1. A `DirectDepositV1` contract exists for some `subaccount` and holds `N` usdcE tokens.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, N)` is called. If the USDC token returns `false` (no revert), execution continues.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` transfers `N` usdcE to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(msg.sender, N)` sends `N` usdcE to the attacker.
6. Attacker has received `N` usdcE; `directDepositV1` has lost `N` usdcE and received 0 USDC.

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
