### Title
Unbounded Gas Cost in `reorg_chain()` Due to Uncapped Fork Depth Permanently Freezes Canonical Chain Tracking — (`contract/src/lib.rs`)

---

### Summary

`reorg_chain()` contains two unbounded loops that iterate over all fork blocks accumulated in `headers_pool` and all displaced mainchain blocks. Because fork blocks are never garbage-collected and no contract-level cap on fork depth exists, a sufficiently deep fork triggers a reorg transaction that exceeds NEAR's per-transaction gas limit. The reorg never completes, `mainchain_tip_blockhash` is never updated, and the contract permanently tracks a stale canonical chain.

---

### Finding Description

`reorg_chain()` is called from `submit_block_header_inner()` whenever a fork's cumulative `chain_work` exceeds the mainchain's:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [1](#0-0) 

Inside `reorg_chain()` there are two unbounded loops:

**Loop 1** — removes mainchain blocks above the fork tip height:
```rust
for height in (fork_tip_height + 1)..=last_main_chain_block_height {
    ...
    self.remove_block_header(&current_main_chain_blockhash);
    self.mainchain_height_to_header.remove(&height);
}
``` [2](#0-1) 

**Loop 2** — walks every fork block from tip back to the common ancestor, performing 3–5 storage operations per iteration:
```rust
while !self.mainchain_header_to_height.contains_key(&fork_header_cursor.block_hash) {
    ...
    self.mainchain_height_to_header.insert(...);
    self.mainchain_header_to_height.insert(...);
    // conditional remove_block_header (2 more ops)
    fork_header_cursor = self.headers_pool.get(&prev_block_hash)...;
}
``` [3](#0-2) 

The critical structural fact is that **fork blocks are never garbage-collected**. GC (`run_mainchain_gc`) only removes mainchain blocks:

> "Only mainchain blocks are deleted; fork/sidechain blocks are not affected" [4](#0-3) 

`store_fork_header()` inserts into `headers_pool` with no eviction path:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [5](#0-4) 

There is **no contract-level limit on fork depth**. The `max_fork_len = 500` value exists only in the off-chain relayer config and is not enforced by the contract: [6](#0-5) 

A relayer (or any account with `Role::UnrestrictedSubmitBlocks`) can submit fork blocks across many separate `submit_blocks` transactions, accumulating an arbitrarily deep fork in `headers_pool`. When the final block tips the fork's `chain_work` over the mainchain, `reorg_chain` is triggered and must walk the entire accumulated fork depth atomically within that single transaction.

---

### Impact Explanation

NEAR's per-transaction gas limit is 300 TGas. Each iteration of the while loop in `reorg_chain` performs at minimum 3 storage reads/writes (`contains_key`, `insert` × 2, `get`) and up to 5 (`remove_block_header` adds 2 more). NEAR storage writes cost on the order of tens of TGas each. At ~100 TGas per iteration, a fork depth of ~50–100 blocks is sufficient to exhaust the gas budget.

When the gas limit is exceeded:
- The `submit_blocks` transaction reverts entirely.
- The fork blocks remain in `headers_pool` (they were written in prior transactions).
- `mainchain_tip_blockhash` is never updated to the fork tip.
- Every subsequent attempt to submit the reorg-triggering block will re-enter `reorg_chain` with the same unbounded walk and fail identically.
- The contract is permanently stuck: it tracks a stale canonical chain, and all `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` calls will evaluate inclusion against the wrong mainchain state, returning incorrect results for any block on the true chain beyond the frozen tip. [7](#0-6) 

---

### Likelihood Explanation

The contract's own documentation flags `reorg_chain` as "The most expensive operation": [8](#0-7) 

For **Dogecoin** (AuxPoW merged mining), reorgs of 50–100 blocks are plausible protocol events. For **Zcash** and **Litecoin**, deep forks are rarer but not impossible. Even without adversarial intent, a legitimate relayer submitting a real long reorg (built up over many `submit_blocks` calls) will trigger this path. An adversarial relayer (one that has staked and been admitted) can deliberately build a deep fork chain to permanently freeze the contract's canonical tip.

---

### Recommendation

Enforce a contract-level cap on the maximum fork depth that `reorg_chain` will process in a single call. Reject fork blocks whose depth from the current mainchain exceeds this cap at submission time (inside `submit_block_header_inner`), or split the reorg into a batched, multi-call operation similar to how `run_mainchain_gc` uses `batch_size`. The off-chain `max_fork_len` config is not a substitute because it is not enforced on-chain.

---

### Proof of Concept

1. Contract is deployed with `gc_threshold = 52704` (default). Mainchain has N blocks.
2. Attacker (or legitimate relayer) submits 200 fork blocks anchored at mainchain block M, each with valid PoW, across 14 separate `submit_blocks` calls of 15 blocks each. Each call succeeds individually (fork blocks are stored via `store_fork_header`, no reorg triggered yet because fork `chain_work` < mainchain `chain_work`).
3. Attacker submits the 201st fork block whose cumulative `chain_work` exceeds the mainchain tip's `chain_work`.
4. `submit_block_header_inner` calls `reorg_chain(fork_tip, last_main_chain_block_height)`.
5. The while loop at line 616 must walk 200 fork blocks, each requiring 3–5 storage operations (~300–1000 TGas total), exceeding NEAR's 300 TGas limit.
6. Transaction reverts with "Exceeded the maximum amount of gas". The 200 fork blocks remain in `headers_pool`. `mainchain_tip_blockhash` is unchanged.
7. Any future attempt to submit the same reorg-triggering block repeats step 4–6 identitely. The mainchain tip is permanently frozen. [9](#0-8) [3](#0-2)
