### Title
Fee-on-Transfer Token Accounting Discrepancy in Deposit Flow Causes Protocol Insolvency - (File: `core/contracts/EndpointStorage.sol`)

---

### Summary

`handleDepositTransfer` in `EndpointStorage.sol` performs two sequential ERC20 transfers of the same nominal `amount`: first from the depositor to the Endpoint, then from the Endpoint to the Clearinghouse. For fee-on-transfer tokens, the Clearinghouse receives less than `amount`, but `clearinghouse.depositCollateral` unconditionally credits the full `amount` (scaled) to the user's on-chain balance. This creates a permanent solvency gap between the Clearinghouse's actual token holdings and the sum of all credited subaccount balances.

---

### Finding Description

The deposit entry point `depositCollateralWithReferral` in `Endpoint.sol` calls `handleDepositTransfer` with the caller-supplied `amount`: [1](#0-0) 

`handleDepositTransfer` performs two transfers of the identical `amount` value: [2](#0-1) 

Both calls delegate to `ERC20Helper.safeTransferFrom` / `safeTransfer`, which only verify that the low-level call succeeded and the return value (if any) is `true`: [3](#0-2) 

Neither helper checks the contract's balance before and after the transfer to confirm the actual amount received. For a fee-on-transfer token with rate `r`:

- **Transfer 1** (user → Endpoint): Endpoint receives `amount × (1 − r)`.
- **Transfer 2** (Endpoint → Clearinghouse): Clearinghouse receives `amount × (1 − r)` (a second fee is deducted from the `amount` the Endpoint attempts to forward). For this transfer to not revert, the Endpoint must hold a prior balance of at least `amount × r` in that token (e.g., accumulated from slow-mode fees paid in the quote token via `chargeSlowModeFee`). [4](#0-3) 

After both transfers succeed, the slow-mode queue records the original `amount`: [5](#0-4) 

When the sequencer processes the queued transaction, `clearinghouse.depositCollateral` credits `amountRealized = int128(txn.amount) * int128(multiplier)` — the full, undeducted `amount` — to the subaccount: [6](#0-5) 

The Clearinghouse's actual token balance is `amount × (1 − r)`, but the credited balance is `amount`. The difference `amount × r` is a permanent deficit that accumulates with every deposit of a fee-on-transfer token.

---

### Impact Explanation

The Clearinghouse's real token holdings fall short of the aggregate of all credited subaccount balances by `fee_rate × deposited_amount` per deposit. As more users deposit and later attempt to withdraw, the Clearinghouse will be unable to satisfy all withdrawal requests, resulting in **protocol insolvency**. Users who withdraw last receive nothing or less than their credited balance. The `manualAssert` invariant check (`spotStates` balance reconciliation) will also diverge from on-chain reality, masking the deficit until it is too large to recover. [7](#0-6) 

---

### Likelihood Explanation

The Endpoint accumulates a quote-token balance from slow-mode fees (`chargeSlowModeFee`), which provides the prior balance needed for the second transfer to succeed without reverting. If the quote token (e.g., USDT) activates its built-in fee mechanism — a known, documented possibility — every subsequent deposit of that token through `depositCollateralWithReferral` or the slow-mode `DepositInsurance` path silently under-credits the Clearinghouse. The trigger requires no privileged access: any ordinary depositor calling the public `depositCollateral` or `depositCollateralWithReferral` entrypoints is sufficient.

---

### Recommendation

Record the Clearinghouse's token balance before and after `safeTransferFrom` / `safeTransfer` and use the measured delta — not the caller-supplied `amount` — as the value passed to `clearinghouse.depositCollateral` and stored in the slow-mode queue. Concretely, in `handleDepositTransfer`:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // use actualReceived instead of amount when building the slow-mode DepositCollateral tx
}
```

Additionally, vet every token listed as a supported collateral product to confirm it does not charge transfer fees, and document this as an explicit listing requirement.

---

### Proof of Concept

1. USDT activates its transfer fee at rate `r = 0.1%`.
2. The Endpoint holds 10 USDT of accumulated slow-mode fees.
3. Alice calls `depositCollateral("alice", QUOTE_PRODUCT_ID, 1000e6)`.
4. `handleDepositTransfer` runs:
   - `safeTransferFrom(USDT, Alice, 1000e6)` → Endpoint receives `999e6` (fee = `1e6`).
   - `safeTransferTo(USDT, Clearinghouse, 1000e6)` → Endpoint sends `1000e6`; Clearinghouse receives `999e6` (second fee = `1e6`); Endpoint's prior balance drops by `1e6`.
5. Slow-mode queue records `amount = 1000e6`.
6. Sequencer processes the queue; `clearinghouse.depositCollateral` credits `1000e6 × multiplier` to Alice's subaccount.
7. Clearinghouse holds `999e6` USDT but has credited `1000e6` — a `1e6` deficit per deposit.
8. Repeated deposits widen the gap until the Clearinghouse cannot honour all withdrawals. [8](#0-7) [9](#0-8)

    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
