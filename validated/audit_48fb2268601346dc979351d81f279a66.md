### Title
Missing Blocked-Address Guard in IBC NFT Receive Path Allows Minting to Module Accounts — (`x/nft-transfer/keeper/packet.go`)

---

### Summary

The IBC NFT transfer module's receive path (`processReceivedPacket`) decodes `data.Receiver` and passes it directly to `MintNFT` without checking whether the address is a blocked module account. The standard ICS-20 fungible transfer module performs this check via `bankKeeper.BlockedAddr(receiver)`, but the ICS-721 implementation has no equivalent guard anywhere in the stack.

---

### Finding Description

**Entry point**: `OnRecvPacket` → `keeper.OnRecvPacket` → `processReceivedPacket`

In `keeper/relay.go`, `OnRecvPacket` calls `data.ValidateBasic()` and then `processReceivedPacket`: [1](#0-0) 

`ValidateBasic` in `types/packet.go` only verifies that `Receiver` is a non-empty, valid bech32 string: [2](#0-1) 

It does **not** check whether the decoded address is a blocked module account.

`processReceivedPacket` then decodes the receiver and passes it straight to `MintNFT`: [3](#0-2) 

`MintNFT` in the NFT keeper only verifies denom creator status — no blocked-address check: [4](#0-3) 

The `nft-transfer` `Keeper` struct holds no bank keeper reference and no blocked-addresses map, so there is no mechanism to perform this check: [5](#0-4) 

The `NFTKeeper` interface also exposes no blocked-address method: [6](#0-5) 

A grep across both `x/nft-transfer` and `x/nft` confirms zero occurrences of `BlockedAddr`, `blockedAddr`, or `BlockedAddresses`.

---

### Impact Explanation

When an NFT is minted to a module account address:

1. **Permanent asset lock**: Module accounts have no private key; the NFT owner record is set to the module account address via `setOwner` / `setNFT`. No transaction can ever transfer or burn it.
2. **Supply invariant violation**: `increaseSupply` is called but the corresponding `decreaseSupply` (via burn) can never be called, permanently inflating the tracked supply for that IBC class.
3. **Escrow/burn on source chain is irreversible**: The source chain has already escrowed or burned the original NFT. The acknowledgement will be a success (no error is returned), so no refund occurs.

---

### Likelihood Explanation

- Any user on a counterparty chain can set `Receiver` to a known module account address (e.g., the `nft-transfer` module account, `distribution`, `bonded_tokens_pool`, etc.) on the destination chain. These addresses are deterministic and publicly known.
- A malicious counterparty chain can craft arbitrary packet data with any valid bech32 receiver.
- No special privilege is required; this is reachable through the standard IBC packet relay path.

---

### Recommendation

Add a blocked-address check in `processReceivedPacket` before calling `MintNFT`, mirroring the ICS-20 pattern. This requires injecting a `BankKeeper` (or a dedicated `BlockedAddr(addr sdk.AccAddress) bool` interface) into the `nft-transfer` Keeper and calling it after decoding the receiver:

```go
receiver, err := sdk.AccAddressFromBech32(data.Receiver)
if err != nil {
    return err
}
if k.bankKeeper.BlockedAddr(receiver) {
    return errorsmod.Wrapf(sdkerrors.ErrUnauthorized,
        "%s is not allowed to receive IBC NFTs", data.Receiver)
}
```

Apply the same guard in `refundPacketToken` for the mint-on-refund path. [7](#0-6) 

---

### Proof of Concept

1. On chain B, obtain the bech32 address of the `nft-transfer` module account (deterministic from `authtypes.NewModuleAddress("nft-transfer")`).
2. On chain A, call `MsgTransfer` with `Receiver` set to that address.
3. The IBC packet is relayed; `OnRecvPacket` is called on chain B.
4. `ValidateBasic` passes (valid bech32).
5. `processReceivedPacket` mints the NFT to the module account — no error, success ACK returned.
6. Query NFT ownership on chain B: owner is the module account.
7. Attempt any transfer or burn from the module account: impossible — no signer.
8. Query denom supply: permanently incremented with no path to decrement.

func (k Keeper) processReceivedPacket(ctx sdk.Context, packet channeltypes.Packet,
	data types.NonFungibleTokenPacketData,
) error {
	receiver, err := sdk.AccAddressFromBech32(data.Receiver)
	if err != nil {
		return err
	}

	isAwayFromOrigin := types.IsAwayFromOrigin(packet.GetSourcePort(), packet.GetSourceChannel(), data.ClassId)

	// create the escrow address for creating denom and minting nft
	escrowAddress := types.GetEscrowAddress(packet.GetDestPort(), packet.GetDestChannel())

	if isAwayFromOrigin {
		// since SendPacket did not prefix the classID, we must prefix classID here
		classPrefix := types.GetClassPrefix(packet.GetDestPort(), packet.GetDestChannel())
		// NOTE: sourcePrefix contains the trailing "/"
		prefixedClassID := classPrefix + data.ClassId

		// construct the class trace from the full raw classID
		classTrace := types.ParseClassTrace(prefixedClassID)
		if !k.HasClassTrace(ctx, classTrace.Hash()) {
			k.SetClassTrace(ctx, classTrace)
		}

		voucherClassID := classTrace.IBCClassID()

		if !k.nftKeeper.HasDenomID(ctx, voucherClassID) {
			if err := k.nftKeeper.IssueDenom(ctx, voucherClassID, voucherClassID, "", data.ClassUri, escrowAddress); err != nil {
				return err
			}
		}
		sdkCtx := sdk.UnwrapSDKContext(ctx)
		sdkCtx.EventManager().EmitEvent(
			sdk.NewEvent(
				types.EventTypeClassTrace,
				sdk.NewAttribute(types.AttributeKeyTraceHash, classTrace.Hash().String()),
				sdk.NewAttribute(types.AttributeKeyClassID, voucherClassID),
			),
		)

		for i, tokenID := range data.TokenIds {
			if err := k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, receiver); err != nil {
				return err
			}
		}
