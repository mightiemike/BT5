### Title
`RewardCalculator::calculate_reward` Silently Overrides Configured `protocol_reward_rate` with Hardcoded `1/10` for Mainnet/Testnet Genesis Chains — (File: `chain/epoch-manager/src/reward_calculator.rs`)

---

### Summary

`RewardCalculator::calculate_reward` unconditionally substitutes the configured `protocol_reward_rate` with a hardcoded `Rational32::new_raw(1, 10)` whenever `self.genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION` (29). The function accepts a `_protocol_version: ProtocolVersion` parameter that is never consulted (prefixed `_`), meaning no protocol upgrade can ever change the protocol treasury reward rate for mainnet/testnet chains. This is the direct nearcore analog of the Uniswap V3 AMO bug: a configured parameter is silently replaced by a hardcoded default, and the mechanism that should allow the value to evolve (oracle / protocol version) is accepted as a parameter but never used.

---

### Finding Description

In `chain/epoch-manager/src/reward_calculator.rs` lines 63–68:

```rust
let use_hardcoded_value = self.genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION;
let protocol_reward_rate = if use_hardcoded_value {
    Rational32::new_raw(1, 10)
} else {
    self.protocol_reward_rate
};
``` [1](#0-0) 

The function signature accepts `_protocol_version: ProtocolVersion` but the leading underscore signals it is intentionally unused: [2](#0-1) 

`PROD_GENESIS_PROTOCOL_VERSION` is `29`, the genesis protocol version for both mainnet and testnet: [3](#0-2) 

`RewardCalculator::new` reads `protocol_reward_rate` from `GenesisConfig` and stores it in `self.protocol_reward_rate`: [4](#0-3) 

But for any chain whose genesis protocol version is 29 (mainnet, testnet), `self.protocol_reward_rate` is dead code — the hardcoded `1/10` is always used instead. The test `test_adjust_max_inflation` explicitly documents this with the comment `// Unused, would only be used for genesis_protocol_version` while setting `protocol_reward_rate: Ratio::new(0, 1)` and `genesis_protocol_version: PROD_GENESIS_PROTOCOL_VERSION`: [5](#0-4) 

The `epoch_protocol_treasury` — the amount minted to the protocol treasury each epoch — is computed directly from `protocol_reward_rate`: [6](#0-5) 

---

### Impact Explanation

Every epoch on mainnet and testnet, `epoch_protocol_treasury` is computed using the hardcoded `1/10` fraction regardless of:

1. What `protocol_reward_rate` is set to in the genesis config.
2. What the current `_protocol_version` is — the parameter is accepted but never read.

This breaks the protocol upgrade boundary: if a future protocol upgrade intends to change the protocol treasury reward rate (e.g., reduce it from 10% to 5% as part of an economics change), the change would be silently ignored for mainnet/testnet. The `max_inflation_rate` IS correctly passed through and used (it comes from `EpochConfig` and varies by protocol version), but `protocol_reward_rate` is frozen at `1/10` forever for production chains. The divergent exact value is `Rational32::new_raw(1, 10)` vs `self.protocol_reward_rate`.

**Scope**: All epoch reward distributions on mainnet and testnet. The protocol treasury share and the validator share are both derived from this rate, so any deviation silently mis-allocates newly minted NEAR tokens every epoch.

---

### Likelihood Explanation

This condition is always true for mainnet and testnet (`genesis_protocol_version == 29`). It fires on every single call to `calculate_reward`, which is called once per epoch finalization in `EpochManager::finalize_epoch`. There is no configuration knob, feature flag, or protocol version gate that can disable the override for production chains.

---

### Recommendation

- **Short term**: Either use `self.protocol_reward_rate` unconditionally (removing the hardcoded override), or rename the `_protocol_version` parameter to `protocol_version` and use it to gate a versioned lookup of the rate from `EpochConfig`, consistent with how `max_inflation_rate` is already handled.
- **Long term**: Audit all fields in `RewardCalculator` for similar silent overrides keyed on `genesis_protocol_version`. Ensure that the protocol upgrade path for economic parameters (treasury rate, inflation rate) is exercised end-to-end in integration tests that use `PROD_GENESIS_PROTOCOL_VERSION` as the genesis version.

---

### Proof of Concept

The existing test `test_adjust_max_inflation` in `chain/epoch-manager/src/reward_calculator.rs` (lines 590–646) is the proof of concept. It constructs a `RewardCalculator` with `protocol_reward_rate: Ratio::new(0, 1)` (zero) and `genesis_protocol_version: PROD_GENESIS_PROTOCOL_VERSION`, then asserts that the protocol treasury receives exactly `expected_total / 10` — i.e., 10% — confirming that the configured zero rate is silently discarded and the hardcoded `1/10` is used instead: [7](#0-6)
