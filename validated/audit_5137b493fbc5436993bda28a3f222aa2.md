### Title
Unbounded `transaction` bytes in `submitSlowModeTransaction` permanently DoS the slow-mode queue via storage-delete OOG — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The Nado `Endpoint` slow-mode queue processes entries in strict FIFO order. An unprivileged attacker can submit a slow-mode transaction with an arbitrarily large `transaction` bytes payload. When the sequencer or any user later calls `executeSlowModeTransaction`, the EVM must load and then `delete` the oversized `bytes tx` field from storage — an operation whose gas cost scales linearly with the payload size. If the payload is large enough, the `delete` alone exceeds the block gas limit, causing every attempt to advance the queue to revert with OOG. Because there is no mechanism to skip a stuck entry, the entire slow-mode queue is permanently frozen, blocking all pending deposits and withdrawals.

---

### Finding Description

`submitSlowModeTransactionImpl` in `EndpointTx.sol` accepts an arbitrary-length `bytes calldata transaction` and stores it verbatim in the `slowModeTxs` mapping with no size restriction:

```solidity
// EndpointTx.sol lines 374-384
IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
requireUnsanctioned(sender);
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: transaction          // ← no size cap
});
slowModeConfig = _slowModeConfig;
``` [1](#0-0) 

The `SlowModeTx.tx` field is a Solidity `bytes` (dynamic), so its storage footprint grows with the payload. A 100 KB payload occupies ≈ 3 200 storage slots.

When `_executeSlowModeTransaction` is called, it unconditionally loads the full struct into memory and then deletes it **before** the try/catch guard:

```solidity
// Endpoint.sol lines 193-194
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];
``` [2](#0-1) 

Zeroing 3 200 previously-written storage slots costs ≈ 3 200 × 5 000 = **16 M gas** (EIP-2929 warm SSTORE). Loading them first adds another ≈ 3 200 × 100 = 320 K gas. On an OP-Stack chain with a 30 M block gas limit, a payload of ≈ 180 KB is sufficient to make the `delete` alone exceed the block gas limit, causing every call to `executeSlowModeTransaction` to revert with OOG before the try/catch is even reached.

Because the OOG revert unwinds the entire transaction, `slowModeConfig.txUpTo` is never incremented and the poisoned entry remains at the head of the queue forever. The try/catch OOG heuristic at lines 219–224 only applies to the **inner** `processSlowModeTransaction` call; the `delete` happens in the **outer** frame and is not protected:

```solidity
// Endpoint.sol lines 205-227
uint256 gasRemaining = gasleft();
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
}
``` [3](#0-2) 

The sequencer path (`processTransaction` with `ExecuteSlowMode`) calls the same `_executeSlowModeTransaction` and is equally affected. [4](#0-3) 

---

### Impact Explanation

The slow-mode queue is the fallback settlement layer for:

- **Collateral deposits** — `depositCollateral` and `depositCollateralWithReferral` both enqueue a `DepositCollateral` slow-mode entry.
- **Collateral withdrawals** — users submit `WithdrawCollateral` via `submitSlowModeTransaction` after the 3-day delay.
- **Signer links, builder fee claims**, and other user-initiated operations. [5](#0-4) 

Freezing the queue means:
1. All pending `WithdrawCollateral` entries (funds already deducted from subaccount balances) can never be executed — user funds are permanently locked in the contract.
2. All pending `DepositCollateral` entries are never credited — deposited tokens are held by the clearinghouse with no corresponding balance update.
3. No recovery path exists on-chain; there is no `skipSlowModeTransaction` or admin override.

---

### Likelihood Explanation

The attack costs the attacker only the $1 `SLOW_MODE_FEE` plus the one-time gas cost to write a large payload to storage (paid once). Any unprivileged address can call `submitSlowModeTransaction` with a `WithdrawCollateral` (or any non-owner, non-`DepositCollateral`) transaction type padded to the required size. The attacker does not need any special role, existing balance, or coordination. [6](#0-5) 

---

### Recommendation

1. **Cap the size of `transaction` in `submitSlowModeTransactionImpl`**: add a `require(transaction.length <= MAX_SLOW_MODE_TX_SIZE)` guard (e.g., 1 KB) at the top of the function.
2. **Move the `delete` inside the try/catch** (or defer it to after successful processing) so that an OOG in the inner call does not leave the queue permanently stuck.
3. **Add an owner-callable `skipSlowModeTransaction`** that can advance `txUpTo` past a poisoned entry as an emergency escape hatch.

---

### Proof of Concept

```solidity
// Attacker pays $1 SLOW_MODE_FEE and submits a WithdrawCollateral
// slow-mode transaction padded to ~180 KB.
bytes memory largePayload = new bytes(180_000);
largePayload[0] = uint8(IEndpoint.TransactionType.WithdrawCollateral);
// Fill the rest with arbitrary bytes — abi.decode ignores trailing data
// for fixed-size structs, but the storage cost is paid regardless.

endpoint.submitSlowModeTransaction(largePayload);

// After 3 days, any call to executeSlowModeTransaction reverts with OOG:
// The `delete slowModeTxs[txUpTo]` must zero ~5 600 storage slots,
// consuming > 28 M gas — exceeding the block gas limit.
// txUpTo is never incremented; the queue is permanently frozen.
endpoint.executeSlowModeTransaction(); // always reverts

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }

    function executeSlowModeTransaction() external {
        SlowModeConfig memory _slowModeConfig = slowModeConfig;
        _executeSlowModeTransaction(_slowModeConfig, false);
        nSubmissions += 1;
        slowModeConfig = _slowModeConfig;
    }
