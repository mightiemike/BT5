### Title
Spot Mainchain State Used for Difficulty Calculation Enables Intra-Batch Manipulation — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary

The difficulty calculation functions in all three chain modules (`get_next_work_required`) derive the required PoW target by calling `get_header_by_height`, which reads the **current/spot mainchain state** (`mainchain_height_to_header`). Within a single `submit_blocks` call, earlier headers in the batch can trigger a chain reorganization that mutates this mapping. When a later header in the same batch is validated, `get_header_by_height` returns a different block than the actual ancestor of the submitted header, producing a wrong expected difficulty. The developers explicitly flagged this in a TODO comment in `dogecoin.rs`.

---

### Finding Description

**Root cause — spot state read in difficulty calculation**

`get_header_by_height` is implemented as:

```rust
// contract/src/lib.rs  lines 677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

It reads `mainchain_height_to_header`, which is the **live, mutable mainchain index**. [1](#0-0) 

This function is called during difficulty validation for every chain module. For Dogecoin (`difficulty_adjustment_interval = 1`, `pow_target_timespan = 60`):

```rust
// contract/src/dogecoin.rs  lines 286-295
let height_first = prev_block_header
    .block_height
    .checked_sub(blocks_to_go_back)
    ...;

// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [2](#0-1) 

The TODO comment is the developers' own acknowledgment that this lookup is incorrect: the correct approach is to traverse the ancestor chain via `prev_block_hash` links, not to look up by height in the live mainchain index.

The same pattern appears in Bitcoin and Litecoin:

```rust
// contract/src/bitcoin.rs  line 81
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [3](#0-2) 

```rust
// contract/src/litecoin.rs  line 88
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [4](#0-3) 

**How the mainchain state is mutated mid-batch**

`submit_blocks` iterates over all submitted headers sequentially in a single call:

```rust
// contract/src/lib.rs  lines 177-179
for header in headers {
    self.submit_block_header(header, self.skip_pow_verification);
}
``` [5](#0-4) 

Each call to `submit_block_header` can trigger `reorg_chain`, which rewrites `mainchain_height_to_header` for every height from the fork point to the old tip:

```rust
// contract/src/lib.rs  lines 628-630
let main_chain_block = self
    .mainchain_height_to_header
    .insert(&current_height, &current_block_hash);
``` [6](#0-5) 

After the reorg, `mainchain_height_to_header[H]` points to the fork block `F_H`, not the original mainchain block `M_H`. Any subsequent header in the same batch that calls `get_header_by_height(H)` receives `F_H`.

**Concrete attack path for Dogecoin (most impactful)**

Dogecoin mainnet has `difficulty_adjustment_interval = 1`, meaning every block is a retarget boundary. The retarget formula is:

```rust
// contract/src/dogecoin.rs  lines 307-309
let modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
``` [7](#0-6) 

where `modulated_timespan = prev_block_header.time - first_block_time`. With `pow_target_timespan = 60` seconds, a timestamp difference of only a few minutes shifts the required difficulty by the maximum allowed factor (±12.5% per block, clamped). [8](#0-7) 

**Step-by-step exploit:**

1. Attacker registers as a trusted relayer (staking-based open role in the `omni_utils` trusted_relayer system).
2. Attacker crafts a batch `[F_H, B_{H+1}]` submitted in one `submit_blocks` call:
   - `F_H`: a fork block at height H with a timestamp crafted to be far from the real chain's block at H. `F_H` must carry enough cumulative chainwork to trigger a reorg (achievable by building on a fork that already has high chainwork, or by mining `F_H` with a lower `bits` value).
   - `B_{H+1}`: a block at height H+1 whose parent is the original mainchain block `M_H`. Its `bits` field encodes the difficulty the attacker wants to pass — lower than what the correct ancestor-based calculation would require.
3. Processing `F_H` triggers `reorg_chain`; `mainchain_height_to_header[H]` is overwritten with `F_H`'s hash.
4. Processing `B_{H+1}`: `check_pow` → `get_next_work_required` → `get_header_by_height(H)` returns `F_H` (not `M_H`). `first_block_time = F_H.time`. The computed `modulated_timespan` is wrong, yielding an `expected_bits` that matches `B_{H+1}.bits` even though `B_{H+1}` does not satisfy the correct difficulty.
5. `B_{H+1}` passes difficulty validation and is stored in `headers_pool`.
6. Attacker extends the chain from `B_{H+1}` with further low-difficulty blocks, eventually accumulating enough chainwork to promote this chain to the mainchain.

---

### Impact Explanation

- **Difficulty validation bypass**: A block with a `bits` value encoding lower difficulty than the protocol requires is accepted into `headers_pool` and can later become the canonical mainchain tip.
- **Canonical chain corruption**: Once the low-difficulty chain is promoted via `reorg_chain`, `mainchain_tip_blockhash` and `mainchain_height_to_header` reflect a chain that did not satisfy the real PoW rules. All subsequent `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` calls operate against this corrupted canonical state.
- **False transaction proofs**: A consumer contract calling `verify_transaction_inclusion` against the corrupted mainchain can be made to accept proofs for transactions that were never confirmed on the real chain, enabling cross-chain fraud (e.g., double-spend proofs, fake bridge deposits).

---

### Likelihood Explanation

- **Dogecoin** is the highest-risk target: `difficulty_adjustment_interval = 1` means every block is a retarget, and `pow_target_timespan = 60` s means small timestamp deltas produce large difficulty swings. The attacker needs to mine only one fork block with sufficient chainwork, which is feasible for any miner with non-trivial hash rate.
- **Bitcoin/Litecoin**: The 2016-block retarget interval means the attack requires a reorg that reaches back to the interval boundary — a much higher bar, but not impossible for a well-resourced attacker targeting testnet (`pow_allow_min_difficulty_blocks = true`).
- The `trusted_relayer` gate on `submit_blocks` requires staking NEAR tokens, which is an economic barrier but not a privileged-key barrier; any party can self-register.

---

### Recommendation

Replace all `get_header_by_height` calls inside difficulty calculation with ancestor traversal via `get_prev_header` links. The correct block at the start of the retarget interval must be reached by walking `prev_block_hash` from the block being validated, not by indexing the live mainchain map. This is exactly what the TODO comment in `dogecoin.rs` identifies as the open question — the answer is that the current approach is incorrect and must be replaced with ancestor traversal.

```rust
// Correct pattern: walk prev_block_hash links
let mut cursor = prev_block_header.clone();
for _ in 0..blocks_to_go_back {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time = cursor.block_header.time;
```

This ensures the difficulty calculation is always grounded in the actual ancestor chain of the block being validated, regardless of any mainchain mutations that occurred earlier in the same batch.

### Proof of Concept

**Setup (Dogecoin mainnet, `difficulty_adjustment_interval = 1`):**

Assume the current mainchain tip is `M_H` at height H with timestamp `T_M` and `bits = B_correct`.

**Batch submitted in one `submit_blocks` call:**

```
headers = [F_H, B_{H+1}]
```

- `F_H`: fork block at height H, parent = `M_{H-1}` (same parent as `M_H`), timestamp `T_F = T_M + 3600` (1 hour later), `bits` low enough to give `F_H` more chainwork than `M_H`.
- `B_{H+1}`: block at height H+1, parent = `M_H`, timestamp `T_B`, `bits = B_low` where `B_low` encodes a target computed using `T_F` as `first_block_time` (i.e., `modulated_timespan = T_B - T_F`, which is small, yielding a lower difficulty).

**Execution trace:**

1. `submit_block_header(F_H)` → `F_H.chain_work > M_H.chain_work` → `reorg_chain` → `mainchain_height_to_header[H] = hash(F_H)`.
2. `submit_block_header(B_{H+1})` → `check_pow` → `get_next_work_required` → `get_header_by_height(H)` returns `F_H` → `first_block_time = T_F` → `modulated_timespan = T_B - T_F` (small) → `expected_bits = B_low` → matches `B_{H+1}.bits` → **validation passes**.
3. `B_{H+1}` is stored in `headers_pool` with difficulty lower than the protocol requires.

**Expected correct behavior:** Step 2 should use `M_H.time` as `first_block_time` (the actual ancestor), yielding `expected_bits = B_correct ≠ B_low`, and `B_{H+1}` should be rejected.
