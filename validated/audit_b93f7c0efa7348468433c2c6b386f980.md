### Title
Missing Minimum Confirmation Enforcement in `verify_transaction_inclusion` Allows Zero-Confirmation Proof Acceptance — (`contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` (and by delegation, `verify_transaction_inclusion_v2`) function accepts a caller-supplied `confirmations` value with no lower-bound enforcement. When a caller passes `confirmations = 0`, the confirmation-depth check is trivially satisfied for any block in the mainchain, because the left-hand side of the comparison is a `u64` that is always `>= 0`. A consuming NEAR contract that calls this function with `confirmations = 0` will receive `true` for a transaction that sits in a block with zero depth — a block that can still be reorganized away — and may release funds or trigger irreversible state changes based on that false guarantee.

---

### Finding Description

In `contract/src/lib.rs`, `verify_transaction_inclusion` performs the following confirmation-depth check:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

The only upper-bound guard on `confirmations` is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [2](#0-1) 

There is **no lower-bound guard**. When `args.confirmations = 0`, the expression `(any_u64) >= 0` is always `true` in Rust's type system (u64 is unsigned), so the check is a no-op. The function then proceeds to verify the Merkle proof against the stored header and returns `true` for a transaction whose block may be at the very tip of the chain — depth zero — with no finality guarantee whatsoever.

`verify_transaction_inclusion_v2` delegates directly to `verify_transaction_inclusion` after its coinbase-proof check, passing `confirmations` through unchanged via `args.into()`:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

So both public entry points share the same root cause.

The `ProofArgs` struct makes `confirmations` a plain caller-supplied field with no documented minimum:

```rust
pub struct ProofArgs {
    pub tx_id: H256,
    pub tx_block_blockhash: H256,
    pub tx_index: u64,
    pub merkle_proof: Vec<H256>,
    pub confirmations: u64,
}
``` [4](#0-3) 

The contract never enforces `confirmations >= 1` anywhere in the verification path.

---

### Impact Explanation

A consuming NEAR contract (e.g., a cross-chain bridge or payment gateway) that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with `confirmations = 0` will receive `true` for a transaction whose containing block has zero depth on the tracked chain. That block is still subject to natural reorganization. If the consuming contract releases funds, mints tokens, or records an irreversible state change on the strength of this `true` result, and the block is subsequently reorganized away, the consuming contract has been deceived into acting on a transaction that no longer exists in the canonical chain. The BTC light client contract itself will correctly update its state after the reorg (the `reorg_chain` path in `submit_block_header_inner` handles this), but the consuming contract's already-executed action cannot be undone. [5](#0-4) 

---

### Likelihood Explanation

Any unprivileged NEAR account can call `verify_transaction_inclusion` directly — it is a public, non-gated function. A consuming contract that omits a meaningful `confirmations` argument (or passes `0` as a "no-op" default) is immediately vulnerable. Chain reorganizations of 1–2 blocks occur naturally on all supported chains (Bitcoin, Litecoin, Dogecoin, Zcash) without any attacker mining power; an adversary who controls the submitted transaction simply needs to wait for a natural shallow reorg, or on smaller/testnet chains can induce one with modest hashrate. The attack surface is proportional to how many consuming contracts call these functions without enforcing their own minimum.

---

### Recommendation

Add an explicit lower-bound guard at the top of `verify_transaction_inclusion`, before any other logic:

```rust
require!(args.confirmations >= 1, "confirmations must be at least 1");
```

Alternatively, document a recommended minimum (e.g., 6 for Bitcoin mainnet) and enforce it as a contract-level constant, or expose a `get_min_confirmations()` view so consuming contracts can query the safe floor. The same guard must be present in `verify_transaction_inclusion_v2` (or inherited via the shared path) to cover both entry points.

---

### Proof of Concept

1. A relayer (trusted) submits block **B** at height **H** (the current tip) containing transaction **T** via `submit_blocks`. The contract records **B** in `mainchain_height_to_header` and `headers_pool`.
2. An unprivileged NEAR caller (or a consuming bridge contract) immediately calls `verify_transaction_inclusion` with `tx_block_blockhash = B`, a valid Merkle proof for **T**, and `confirmations = 0`.
3. The check `(H - H) + 1 >= 0` → `1 >= 0` → `true`. The Merkle proof is valid. The function returns `true`.
4. The consuming contract releases funds to the beneficiary.
5. A competing chain of equal or greater chainwork is submitted (natural reorg or attacker-induced on a smaller chain). `reorg_chain` executes, removing **B** from the mainchain maps. Transaction **T** is no longer in the canonical chain.
6. The consuming contract's funds have already been released; the action is irreversible. [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
