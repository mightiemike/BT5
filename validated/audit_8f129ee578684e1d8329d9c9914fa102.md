### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Allows Theft of DDA USDC-E Funds — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` is a publicly callable migration function that swaps USDC-E held in a `DirectDepositV1` (DDA) contract for USDC. It calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` using the raw interface method — not the `safeTransferFrom` wrapper defined in `ERC20Helper`. The return value is silently discarded. If the USDC token at the hardcoded address returns `false` instead of reverting on a failed transfer, the function continues execution, withdraws the DDA's USDC-E to the caller, and sends it out — without the caller ever providing USDC. Any user's pending USDC-E deposit sitting in their DDA can be drained.

---

### Finding Description

`replaceUsdcEWithUsdc` is defined in `ContractOwner.sol` with no access-control modifier beyond a chain ID check:

```solidity
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    ...
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← unchecked
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);
    }
}
``` [1](#0-0) 

The protocol already defines a safe wrapper for exactly this purpose in `ERC20Helper`:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [2](#0-1) 

Every other token transfer in the codebase uses `safeTransfer` / `safeTransferFrom` from this library. The raw call on line 616 is the sole exception.

The `IERC20Base` interface exposes `transferFrom` as a function returning `bool`. The interface does not mandate that the implementation reverts on failure — it may legally return `false`. This is the same ambiguity the external report identifies for `mintCountTo`: the interface specification does not require a revert, so a compliant (or simply non-standard) token implementation can return `false` silently. [3](#0-2) 

---

### Impact Explanation

If `IERC20Base(usdc).transferFrom` returns `false` (no revert):

1. No USDC is transferred from the attacker to the DDA.
2. `DirectDepositV1(directDepositV1).withdraw(usdcE)` still executes — it transfers all USDC-E from the DDA to `ContractOwner`.
3. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` still executes — it sends all USDC-E to the attacker.

The attacker receives the victim's USDC-E for free. The DDA is drained. The user whose subaccount the DDA belongs to loses their pending deposit entirely, with no on-chain recourse.

---

### Likelihood Explanation

- `replaceUsdcEWithUsdc` has no `onlyOwner` or `onlyDeployer` guard — any EOA or contract can call it on chain 57073.
- The attacker needs only to identify a DDA (`directDepositV1Address[subaccount] != address(0)`) with a non-zero USDC-E balance, which is observable on-chain.
- The attack requires USDC at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink chain to return `false` rather than revert. Non-standard ERC-20 behavior (returning `false` on failure) is a documented real-world pattern (e.g., ZRX, BNB). The protocol's own `ERC20Helper` was written specifically to handle this class of token, confirming the developers are aware of the risk — yet this one call site was left unprotected.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's own `safeTransferFrom` wrapper:

```diff
- IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
+ IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

This mirrors every other transfer in the codebase and ensures the transaction reverts if the USDC transfer does not succeed, preventing the subsequent USDC-E withdrawal from proceeding. [4](#0-3) 

---

### Proof of Concept

1. Alice sends USDC-E to her DDA (e.g., by transferring tokens directly to the DDA address before `creditDeposit` is called). The DDA now holds `N` USDC-E.
2. Bob (attacker) calls `replaceUsdcEWithUsdc(alice_subaccount)` on chain 57073.
3. `IERC20Base(usdc).transferFrom(bob, dda, N)` returns `false` — Bob has no USDC, or USDC is a non-reverting token. The return value is discarded.
4. `DirectDepositV1(dda).withdraw(usdcE)` executes: `N` USDC-E moves from the DDA to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(bob, N)` executes: `N` USDC-E moves from `ContractOwner` to Bob.
6. Alice's DDA is empty. Bob holds `N` USDC-E at zero cost.
