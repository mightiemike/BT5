### Title
Difficulty Retarget Uses Mainchain Ancestor Instead of Fork Ancestor, Enabling Invalid Fork Block Acceptance — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

All three chain implementations (`dogecoin.rs`, `bitcoin.rs`, `litecoin.rs`) compute the required difficulty for an incoming block by calling `blocks_getter.get_header_by_height(height_first)`. This helper always resolves the block from the **mainchain** height index, not from the submitted block's actual ancestor chain. When a fork block is submitted that is two or more blocks deep into a fork, the reference block used for difficulty calculation is the mainchain block at that height — which is a **different block** from the fork's true ancestor. The resulting `expected_bits` is computed from the wrong timestamp, so the difficulty check can accept fork blocks whose `bits` field would be rejected if the correct ancestor were used. The Dogecoin module even contains an explicit `TODO` comment acknowledging this exact concern.

---

### Finding Description

**Root cause — Dogecoin (per-block Digishield retarget):** [1](#0-0) 

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

`get_header_by_height` is implemented as: [2](#0-1) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

It unconditionally resolves the block from `mainchain_height_to_header` — the mainchain index — regardless of whether the block being validated is a fork block.

**Same pattern in Bitcoin and Litecoin:** [3](#0-2) [4](#0-3) 

**When the desynchronization occurs:**

For Dogecoin with post-145k per-block retarget, `difficulty_adjustment_interval = 1` and `blocks_to_go_back = 1`, so `height_first = prev_block_header.block_height − 1`.

- Fork block at height H+1 (fork diverges here): `height_first = H`. Mainchain block at H **is** the fork's ancestor — correct.
- Fork block at height H+2: `height_first = H+1`. Mainchain block at H+1 **is not** the fork's ancestor; the fork block at H+1 is. If their timestamps differ, the difficulty calculation is wrong.
- Every subsequent fork block compounds the error.

For Bitcoin/Litecoin the same desynchronization occurs whenever the fork diverges before the retarget boundary block (`first_block_height = prev_block_header.block_height − 2015`).

**Effect on difficulty calculation:** [5](#0-4) 

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
```

`first_block_time` is the mainchain block's timestamp. If the mainchain block at `height_first` has an **earlier** timestamp than the fork's true ancestor at that height, `modulated_timespan` is inflated, Digishield computes a lower difficulty (higher target), and the contract accepts a `bits` value that is easier to satisfy than the fork's actual chain history warrants. The PoW check then passes against this inflated target. [6](#0-5) 

---

### Impact Explanation

Fork blocks with an incorrect `bits` field pass `check_target` and are stored in `headers_pool`. Because `chain_work` is derived directly from `bits`: [7](#0-6) 

an attacker who crafts fork block timestamps to produce an inflated target can mine fork blocks with less PoW than the protocol requires while still accumulating chainwork sufficient to trigger a reorg. After a reorg, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` operate against a mainchain that was promoted via incorrect difficulty validation, undermining the integrity of every SPV proof the contract issues. [8](#0-7) 

---

### Likelihood Explanation

The bug is triggered by any fork submission that is two or more blocks deep — a routine occurrence during normal chain operation. For Dogecoin, with per-block retargeting, every fork block after the first one is affected. The attacker controls fork block timestamps (within MTP and `MAX_FUTURE_BLOCK_TIME_LOCAL` bounds), giving them a concrete lever to steer the wrong difficulty calculation in a favorable direction. The `#[trusted_relayer]` gate on `submit_blocks` limits the entry path, but the `bypass_roles` include `Role::UnrestrictedSubmitBlocks`, and the system is architecturally designed to accept submissions from external relayers. [9](#0-8) 

---

### Recommendation

Replace `get_header_by_height` in all three retarget functions with a chain-walk that follows `get_prev_header` from `prev_block_header` back to `height_first`. This ensures the reference block is always the submitted block's true ancestor, regardless of whether it is a mainchain or fork block:

```rust
let mut cursor = prev_block_header.clone();
while cursor.block_height > height_first {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time = cursor.block_header.time;
```

The same fix must be applied in `bitcoin.rs` and `litecoin.rs`.

---

### Proof of Concept

1. Deploy the Dogecoin build. Initialize with a genesis chain at height ≥ 145 001 (post-Digishield).
2. Submit mainchain blocks M[H], M[H+1], M[H+2] with timestamps T, T+60, T+120.
3. Submit fork block F[H+1] (prev = M[H]) with timestamp T+3600 (1 hour later than M[H+1]).
4. Submit fork block F[H+2] (prev = F[H+1]) with timestamp T+3660.
5. For F[H+3] (prev = F[H+2]), `height_first = H+1`. The contract fetches M[H+1] (timestamp T+60) instead of F[H+1] (timestamp T+3600). `modulated_timespan = (T+3660) − (T+60) = 3600 s` instead of the correct `(T+3660) − (T+3600) = 60 s`. Digishield computes a much lower difficulty (higher target) for F[H+3].
6. Submit F[H+3] with `bits` matching the inflated target. `check_target` passes. The block is stored with inflated `chain_work`.
7. Repeat until fork `chain_work` exceeds mainchain, triggering a reorg via `reorg_chain`. [10](#0-9) [11](#0-10)

    block_header: &Header,
    prev_block_header: &ExtendedHeader,

    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;

    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
