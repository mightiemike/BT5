### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas` When L2/Data Gas Are Zero, Producing a Different Transaction Hash and Rejecting Valid Consensus Proposals - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to distinguish old V3 transactions (pre-0.13.3, `L1Gas` only) from new V3 transactions (`AllResources`). When a new V3 invoke transaction carries zero L2 gas and zero L1 data gas bounds — a valid configuration — the heuristic silently produces `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. This causes two cascading failures: (1) the subsequent conversion to `RpcInvokeTransactionV3` immediately returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`, so validator nodes reject any consensus proposal containing such a transaction; (2) even if that error were bypassed, the transaction hash computed by the validator would omit the L1_DATA_GAS field that the proposer included, producing a divergent hash for the same transaction.

---

### Finding Description

**Root cause — the heuristic at the protobuf boundary:**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` decides which variant to produce based solely on whether the deserialized values are zero:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← chosen for ANY tx with zero L2/data gas
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The comment above this block acknowledges the ambiguity: `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.` [2](#0-1) 

**How a valid transaction triggers the bug:**

A user submits an invoke V3 transaction via RPC with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The gateway accepts it. The `RpcInvokeTransactionV3` struct always stores `resource_bounds: AllResourceBounds` (never `ValidResourceBounds`): [3](#0-2) 

The `InvokeTransactionV3Trait` implementation for `InternalRpcInvokeTransactionV3` always wraps this as `ValidResourceBounds::AllResources(...)`: [4](#0-3) 

So the hash is computed with `AllResources`, which causes `get_tip_resource_bounds_hash` to include the L1_DATA_GAS field: [5](#0-4) 

**Failure 1 — conversion error on the validator:**

When the proposer serializes the transaction to protobuf for consensus, `From<RpcInvokeTransactionV3> for protobuf::InvokeV3WithProof` converts it through `InvokeTransactionV3` (which uses `ValidResourceBounds::AllResources`) and then to protobuf bytes with `l1_data_gas = 0`: [6](#0-5) 

The validator deserializes the protobuf. `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` calls the heuristic, which sees `l1_data_gas.is_zero() && l2_gas.is_zero()` and produces `ValidResourceBounds::L1Gas(l1_gas)`: [7](#0-6) 

Then `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` calls `snapi_invoke.try_into()` which is `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`. That conversion explicitly rejects `L1Gas`:

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => {
        return Err(StarknetApiError::OutOfRange { string: "resource_bounds".to_string() });
    }
},
``` [8](#0-7) 

The error is mapped to `DEPRECATED_RESOURCE_BOUNDS_ERROR` and propagates through `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction`: [9](#0-8) [10](#0-9) 

The validator rejects the entire proposal.

**Failure 2 — hash domain divergence (if conversion were patched):**

`get_tip_resource_bounds_hash` produces different output for the two variants. For `AllResources` it chains three resource felts (L1_GAS, L2_GAS, L1_DATA_GAS); for `L1Gas` it chains only two (L1_GAS, L2_GAS): [11](#0-10) 

The proposer's hash (computed from `AllResources`) and the validator's hash (computed from `L1Gas`) are therefore different for the same transaction bytes. The validator's `convert_rpc_tx_to_internal` recomputes the hash and stores a different value than the proposer committed to: [12](#0-11) 

---

### Impact Explanation

**High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

Any invoke V3 transaction with zero L2 gas and zero L1 data gas bounds (a valid and common configuration for transactions that only consume L1 gas) will be accepted by the gateway, included in a block proposal by the proposer, and then cause every validator to fail to deserialize the proposal. This breaks consensus liveness: the proposer cannot produce a valid proposal that validators will accept for any block containing such a transaction.

---

### Likelihood Explanation

The trigger condition — a V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` — is reachable by any unprivileged user submitting a transaction via the public RPC endpoint. The gateway's stateless validator accepts transactions with any combination of resource bounds as long as at least one is non-zero. The bug is latent in every deployment and activates the moment such a transaction is included in a proposal.

---

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, remove the value-based heuristic for consensus transactions. Since consensus only carries V3 transactions (post-0.13.3), always produce `ValidResourceBounds::AllResources` when deserializing from a `protobuf::ConsensusTransaction`. The `L1Gas` variant should only be produced when deserializing historical state-sync transactions where the protobuf message genuinely lacks an `l1_data_gas` field (i.e., `value.l1_data_gas` is `None`), not when the field is present but zero:

```rust
// Proposed fix: distinguish absent field from zero-valued field
let l1_data_gas_opt = value.l1_data_gas;
Ok(match l1_data_gas_opt {
    None => ValidResourceBounds::L1Gas(l1_gas),   // truly old tx: field absent
    Some(l1_data_gas_proto) => {
        let l1_data_gas: ResourceBounds = l1_data_gas_proto.try_into()?;
        ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
    }
})
```

Additionally, add a round-trip test asserting that `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` survives protobuf serialization and deserialization as `AllResources`, and that the transaction hash is identical before and after the round-trip.

### Proof of Concept

1. Construct an `RpcInvokeTransactionV3` with `resource_bounds = AllResourceBounds { l1_gas: ResourceBounds { max_amount: 100, max_price_per_unit: 1 }, l2_gas: ResourceBounds::default(), l1_data_gas: ResourceBounds::default() }`.
2. Submit it to the gateway via RPC. It is accepted; the hash is computed with `AllResources` (L1_DATA_GAS = 0 included in the Poseidon preimage).
3. The proposer wraps it in a `ConsensusTransaction` and serializes to `protobuf::ConsensusTransaction` bytes via `From<ConsensusTransaction> for protobuf::ConsensusTransaction` → `From<RpcInvokeTransactionV3> for protobuf::InvokeV3WithProof`. The protobuf `ResourceBounds` message has `l1_data_gas` present with value 0.
4. A validator deserializes the bytes via `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction`. Inside, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` sees `l1_data_gas.is_zero() && l2_gas.is_zero()` and returns `ValidResourceBounds::L1Gas(l1_gas)`.
5. `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` receives `ValidResourceBounds::L1Gas` and returns `Err(StarknetApiError::OutOfRange { string: "resource_bounds" })`, mapped to `DEPRECATED_RESOURCE_BOUNDS_ERROR`.
6. The validator returns an error from `TryFrom<protobuf::ConsensusTransaction>`, rejects the proposal, and consensus stalls.
