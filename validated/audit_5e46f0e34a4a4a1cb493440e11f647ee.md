### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Protection Against 64-Byte Forgery - (File: `contract/src/lib.rs`)

### Summary
The deprecated `verify_transaction_inclusion` function is still publicly callable by any unprivileged NEAR account. The v2 replacement (`verify_transaction_inclusion_v2`) added a coinbase merkle proof check specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability. Because the old function remains reachable at runtime, any caller can bypass that protection entirely by invoking it directly, forging a transaction inclusion proof using an internal merkle tree node.

### Finding Description

`verify_transaction_inclusion_v2` was introduced to close the 64-byte merkle forgery attack. It does so by requiring the caller to also prove the coinbase transaction (index 0) with a proof of the same length as the target tx proof, ensuring the target tx is a genuine leaf node and not an internal node of the tree. [1](#0-0) 

However, the original `verify_transaction_inclusion` is still decorated only with `#[deprecated]` and `#[pause]`, but **not** with `#[private]` or any access-control guard. In Rust/NEAR, `#[deprecated]` is a compile-time lint that produces a warning for callers in the same codebase; it has zero effect on the on-chain ABI. The function remains a fully public NEAR contract method. [2](#0-1) 

Any NEAR account can call `verify_transaction_inclusion` directly, supplying a `ProofArgs` struct with an internal merkle tree node as `tx_id` and a shortened proof path. The function computes the merkle root from the supplied proof and compares it to the stored block header's `merkle_root`; it does not verify that `tx_id` is a leaf. [3](#0-2) 

`compute_root_from_merkle_proof` accepts any starting hash and any proof length; it will produce the correct root whether the starting hash is a leaf or an internal node. [4](#0-3) 

### Impact Explanation

Any protocol on NEAR that gates privileged actions (fund release, cross-chain settlement, etc.) on a `true` result from `verify_transaction_inclusion` can be deceived into accepting a forged Bitcoin transaction inclusion proof. An attacker identifies an internal merkle node `N = SHA256d(left || right)` in a real Bitcoin block already tracked by the light client, constructs a valid (but shortened) proof path from `N` to the root, and calls `verify_transaction_inclusion` with `tx_id = N`. The function returns `true` for a transaction that was never broadcast or mined.

**Impact: High** — forged proof acceptance corrupts the core security guarantee of the light client (SPV verification integrity).

### Likelihood Explanation

**Likelihood: Low** — the attacker must identify a suitable internal node in a tracked block and understand the merkle tree structure, but no privileged access, key material, or social engineering is required. The entry point is a standard public NEAR contract call available to any account.

### Recommendation

Remove the public NEAR method exposure of `verify_transaction_inclusion`. The simplest fix is to change it from a `pub` `#[near]` method to a private helper, or add `#[private]` so only the contract itself can call it (as it does internally from `verify_transaction_inclusion_v2`). Alternatively, delete the function body and have `verify_transaction_inclusion_v2` call its own internal logic directly. [5](#0-4) 

### Proof of Concept

Given a Bitcoin block already in the light client's `headers_pool` with at least 4 transactions (txA, txB, txC, txD):

```
Merkle tree:
  Level 0 (leaves): H(txA), H(txB), H(txC), H(txD)
  Level 1:          N_AB = SHA256d(H(txA)||H(txB)),  N_CD = SHA256d(H(txC)||H(txD))
  Level 2 (root):   root = SHA256d(N_AB || N_CD)
```

1. The attacker sets `tx_id = N_AB` (an internal node, not a real txid).
2. The attacker sets `tx_index = 0` and `merkle_proof = [N_CD]` (length 1, not 2).
3. `compute_root_from_merkle_proof(N_AB, 0, [N_CD])` = `SHA256d(N_AB || N_CD)` = `root` ✓
4. The block is on the mainchain and has enough confirmations.
5. `verify_transaction_inclusion` returns `true`.

The coinbase check in `verify_transaction_inclusion_v2` would have caught this (the coinbase proof would have length 2, but the tx proof has length 1, failing the length-equality require). By calling the deprecated function directly, the attacker skips that check entirely. [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
