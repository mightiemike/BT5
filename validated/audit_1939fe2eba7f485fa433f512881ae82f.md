Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which the pool sets to `msg.sender` — the immediate caller of `pool.swap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. The extension therefore checks whether the router is allowlisted rather than the actual user, nullifying the curated-pool access-control policy for all router users.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs the check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct, used as the per-pool namespace key), and `sender` is the first argument forwarded by the pool's `_beforeSwap` dispatcher.

In `MetricOmmPool.swap`, the pool passes `msg.sender` as `sender` to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap — the router, not the user
    recipient,
    ...
);
``` [2](#0-1) 

`ExtensionCalling._beforeSwap` then encodes this `sender` value and forwards it to the extension: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` (or any other swap entry point) calls `pool.swap(params.recipient, ...)`, the pool sees `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. [4](#0-3) 

This is structurally inconsistent with `DepositAllowlistExtension`, which correctly gates by `owner` — an explicit parameter that `MetricOmmPoolLiquidityAdder` sets to the actual user's address (`msg.sender`), not its own address: [5](#0-4) 

The deposit path correctly threads the real user through `owner`; the swap path has no equivalent — the real user is invisible to the extension.

## Impact Explanation
**Allowlist bypass (admin-boundary break):** A pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict swaps to a specific set of addresses (e.g., KYC'd counterparties, whitelisted market makers). The admin allowlists the router so that approved users can reach the pool through the standard periphery path. Because the extension checks the router address rather than the user, any non-allowlisted user can call `router.swap(pool, ...)` and the extension will pass — the router is allowlisted, so the check succeeds regardless of who the actual user is. The curated pool's entire access-control policy is nullified for every user who routes through the router.

**Inverse DoS:** If the admin does not allowlist the router (to avoid the above), then legitimately allowlisted users who call through the router are blocked, making the standard periphery path unusable for them.

Both outcomes break the core invariant that a curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it. This qualifies as an admin-boundary break: the pool admin's access-control configuration is bypassed by an unprivileged path available to any user.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary production entry point for swaps. Allowlisting the router is the natural operational step a pool admin would take to allow approved users to use the standard periphery. The bypass therefore activates under a routine, expected configuration — not an exotic or adversarial one. Any user aware of the router address can exploit it without any special privilege. The router's `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` entry points all call `pool.swap` with `msg.sender = router`, so all swap paths through the router are affected. [6](#0-5) 

## Recommendation
The pool must thread the originating user's identity through to the extension in a way that cannot be spoofed by an intermediate router. The cleanest fix mirrors the deposit path: add an explicit `payer` (or `originator`) address to `pool.swap` that the router sets to `msg.sender` (the actual user). The extension checks this field instead of `sender`. This mirrors how `addLiquidity` passes `owner` explicitly, allowing `DepositAllowlistExtension` to gate on the correct actor.

## Proof of Concept
```
1. Pool admin deploys pool with SwapAllowlistExtension as beforeSwap hook.
2. Admin allowlists the router:
       extension.setAllowedToSwap(pool, address(router), true)
   (natural step: "let router users reach my pool")
3. Non-allowlisted user Alice calls:
       router.exactInputSingle(pool, recipient=Alice, zeroForOne=true, amountIn=X, ...)
4. Router calls pool.swap(recipient=Alice, ...) — pool sees msg.sender = router.
5. Pool calls _beforeSwap(sender=router, recipient=Alice, ...).
6. Extension evaluates:
       allowedSwapper[pool][router] == true  →  check passes, no revert.
7. Alice's swap executes on the curated pool despite never being allowlisted.
8. Pool admin's curation policy is completely bypassed for all router users.
```
