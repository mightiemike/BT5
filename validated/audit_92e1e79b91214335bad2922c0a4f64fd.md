### Title
Protobuf `ValidResourceBounds` deserialization silently downcasts `AllResources` to `L1Gas` when l2_gas and l1_data_gas are zero, causing consensus proposal rejection — (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` silently converts an `AllResources` variant into `L1Gas` whenever both `l2_gas` and `l1_data_gas` are zero. Because the gateway freely admits invoke transactions whose `AllResourceBounds` has only `l1_gas` set, a proposer can include such a transaction in a block. When validators receive the proposal over the wire, the protobuf round-trip produces the `L1Gas` variant, which the subsequent `RpcInvokeTransactionV3` conversion rejects as a deprecated resource-bounds type. The validator marks the entire proposal as `Failed`, breaking consensus for that round.

**Hash domain divergence (secondary)**

Even if the conversion were made non-fatal, the hash computed by the proposer and the hash that would be computed by the validator differ. `get_tip_resource_bounds_hash` hashes three resource felts for `AllResources` (L1_GAS, L2_GAS, L1_DATA_GAS) but only two for `L1Gas` (L1_GAS, L2_GAS): [7](#0-6) 

The proposer always uses `AllResources` (three felts); a validator that successfully deserialized the same bytes as `L1Gas` would compute a two-felt hash, producing a different `tx_hash` and breaking the commitment.

### Impact Explanation

**High — Transaction conversion logic binds the wrong type, causing valid proposals to be rejected.**

An unprivileged attacker submits a single invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The gateway admits it. The proposer includes it in a block. Every validator that receives the proposal over protobuf fails to deserialize the transaction and marks the proposal `Failed`. The proposer loses its round; repeated submissions cause sustained consensus liveness failures.

---

### Likelihood Explanation

The gateway explicitly accepts transactions with only `l1_gas` set (test case `valid_l1_gas`). No additional privilege is required. The attacker only needs to submit a standard invoke transaction with zero l2 and l1_data gas bounds, which is a normal pattern for pre-0.13.3-style transactions. The trigger is deterministic and reproducible.

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```
