Looking at the actual code to validate the claim.

Audit Report

## Title
`PriceVelocityGuardExtension` First-Swap Velocity Check Bypass Due to Uninitialized `lastMidPriceX64` — (File: `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary
`PriceVelocityGuardExtension.beforeSwap` gates its entire velocity enforcement block behind `if (prevMid != 0)`, where `prevMid` is read from `lastMidPriceX64` — a Solidity storage mapping that zero-initialises. On the first `beforeSwap` call for any pool, `prevMid` is always `0`, so the velocity check is unconditionally skipped. Any unprivileged trader can execute the first swap at whatever extreme oracle price is currently reported, bypassing the cap the extension was configured to enforce.

## Finding Description
In `PriceVelocityGuardExtension.beforeSwap` (lines 53–76), the function reads `prevMid = s.lastMidPriceX64`, immediately overwrites storage with the current oracle mid-price, then gates the velocity check:

```solidity
uint128 prevMid = s.lastMidPriceX64;   // zero on first call
...
s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {                     // always false on first swap
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        // squared-envelope velocity check — never reached
    }
}
```

Because `priceVelocityState` is a `mapping(address => PriceVelocityState)`, `lastMidPriceX64` is zero-initialised for every new pool. The `if (prevMid != 0)` guard is always false on the first `beforeSwap` call, so the velocity cap is never applied.

The pool admin can pre-seed via `setLastMidPrice` (lines 29–34), but this is not enforced on-chain. Unlike `OracleValueStopLossExtension`, which has a factory-gated `initialize` function (lines 46–68) that atomically sets all config at pool creation and is enforced by `onlyFactory`, `PriceVelocityGuardExtension` overrides nothing from `BaseMetricExtension.initialize` (line 41–43 of `BaseMetricExtension.sol`) — it simply returns the selector without seeding `lastMidPriceX64`. There is no on-chain mechanism that prevents the first swap from executing before the admin calls `setLastMidPrice`.

The call path is confirmed: `MetricOmmPool.swap` calls `_beforeSwap` (lines 230–240 of `MetricOmmPool.sol`) before settlement, and `ExtensionCalling._beforeSwap` (lines 149–177 of `ExtensionCalling.sol`) dispatches to the extension with the live oracle bid/ask prices. The extension derives `midPrice` from those prices and proceeds to the skipped check.

## Impact Explanation
The velocity guard is specifically designed to protect LPs from rapid oracle price movements. If the oracle price has moved dramatically before the first swap — due to a market event or oracle feed anomaly — the first swap executes at the current extreme price. LPs receive far less than the velocity-guarded price would have permitted, losing principal on settlement. After this first swap, `lastMidPriceX64` is set to the extreme price and all subsequent swaps are correctly checked — the guard is permanently active only after the damage is done. For pools with significant liquidity, the LP loss on the first swap can be material. This constitutes direct loss of LP principal, matching the "bad-price execution" allowed impact.

## Likelihood Explanation
This affects every pool using `PriceVelocityGuardExtension` where the admin has not manually called `setLastMidPrice` before the first swap. The extension provides no `initialize` hook and no on-chain enforcement of pre-seeding. The bypass window is open from pool deployment until the first swap executes — the highest-risk period (pool launch, no prior price history). Any unprivileged trader can trigger the first swap and exploit the open window.

## Recommendation
Add a factory-gated `initialize` override to `PriceVelocityGuardExtension` (mirroring `OracleValueStopLossExtension.initialize`) that accepts an initial mid-price and sets `lastMidPriceX64` and `lastUpdateBlock` atomically at pool creation, protected by `onlyFactory`. This ensures the velocity guard is active from the very first swap. Alternatively, if the first-swap skip is intentional by design, remove `setLastMidPrice` to avoid the false impression that pre-seeding activates the guard from block zero, and document the open first-swap window explicitly in the NatSpec.

## Proof of Concept
1. Deploy a pool with `PriceVelocityGuardExtension` configured with `maxChangePerBlockE18 = 1e16` (1% per block cap).
2. Do **not** call `setLastMidPrice` — `lastMidPriceX64` remains `0`.
3. Allow the oracle price to move 50% (well beyond the 1% cap) before any swap occurs.
4. Execute the first swap on the pool as an unprivileged trader.
5. `beforeSwap` reads `prevMid = 0`; the `if (prevMid != 0)` block is skipped entirely; the swap executes at the 50%-moved oracle price.
6. LPs receive far less than the velocity-guarded price would have permitted, losing principal on settlement.
7. After this first swap, `lastMidPriceX64` is set to the 50%-moved price, and all subsequent swaps are correctly checked — the guard is permanently active only after the damage is done.
