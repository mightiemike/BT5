### Title
`DepositAllowlistExtension` Checks LP Position `owner` Instead of the Actual Depositor/Sender, Allowing Unauthorized Liquidity Deposits — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` is documented as gating `addLiquidity` **by depositor address**, but the guard reads the second hook parameter (`owner`, the LP-position recipient) instead of the first parameter (the actual caller/sender). Any address not on the allowlist can bypass the gate by naming any allowlisted address as the `owner` of the new LP position.

---

### Finding Description

The `beforeAddLiquidity` hook signature (mirroring `SwapAllowlistExtension.beforeSwap`) passes the **actual caller** as the first `address` argument and the **LP-position owner** as the second:

```solidity
// SwapAllowlistExtension — correctly checks the first param (sender)
function beforeSwap(address sender, address /*recipient*/, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

`DepositAllowlistExtension` silently drops the first parameter and checks the second (`owner`) instead:

```solidity
// DepositAllowlistExtension — WRONG: checks owner (2nd param), ignores sender (1st param)
function beforeAddLiquidity(address /*sender — ignored*/, address owner, ...)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [1](#0-0) 

The NatSpec and the storage mapping name both confirm the intended subject is the **depositor**, not the owner: [2](#0-1) [3](#0-2) 

The parallel `SwapAllowlistExtension` correctly checks `sender` (first parameter): [4](#0-3) 

---

### Impact Explanation

An unauthorized address `Alice` can call `pool.addLiquidity(...)` with `owner = Bob` where `Bob` is any allowlisted address. The hook receives `(alice, bob, ...)`, evaluates `allowedDepositor[pool][bob]` → `true`, and passes. Alice's tokens enter the pool and the LP position is minted to Bob. The deposit allowlist — the pool admin's primary mechanism for restricting who can supply liquidity — is completely bypassed. This is an admin-boundary break: an unprivileged path circumvents a configured pool-level access control, allowing unauthorized parties to inject liquidity into restricted pools and affect bin-level balances, pool dynamics, and LP dilution.

---

### Likelihood Explanation

Exploitation requires no special privilege. Any address can call `addLiquidity` on a pool using this extension, supply any allowlisted address as `owner`, and the guard will pass unconditionally. The allowlisted `owner` need not cooperate; the attacker simply loses the deposited tokens (which are credited to the named owner's LP position). The bypass is therefore trivially reachable by any external actor.

---

### Recommendation

Replace `owner` with the actual sender (first parameter) in the allowlist check, mirroring `SwapAllowlistExtension`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; `allowAllDepositors[pool] = false`.
2. Admin calls `setAllowedToDeposit(pool, bob, true)`. Alice is **not** allowlisted.
3. Alice calls `pool.addLiquidity(...)` with `owner = bob`.
4. Pool calls `extension.beforeAddLiquidity(alice, bob, ...)`.
5. Guard evaluates `allowedDepositor[pool][bob]` → `true` → no revert.
6. Alice's tokens are deposited; Bob receives the LP position.
7. Alice has successfully added liquidity to a pool she is not authorized to touch, bypassing the deposit allowlist entirely.
