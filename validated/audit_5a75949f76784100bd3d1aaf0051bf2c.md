Audit Report

## Title
Router Address Substitution Bypasses Per-User Swap Allowlist in `SwapAllowlistExtension` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the allowlist, but `sender` is `msg.sender` of `MetricOmmPool.swap()` — which is the router contract, not the originating EOA. Any user who routes through an allowlisted router bypasses the per-user swap allowlist entirely, breaking the core invariant of the extension ("Gates `swap` by swapper address, per pool").

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument to `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the caller of the extension) and `sender` is the router address: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, the router is `msg.sender` to the pool: [4](#0-3) 

The allowlist check therefore resolves to `allowedSwapper[pool][router]`. If the router is allowlisted, every user of that router passes the check regardless of their own allowlist status. The `allowedSwapper` mapping is keyed by `(pool, swapper)` and is intended to gate individual swapper addresses, but the router substitution means the individual EOA identity is never evaluated. [5](#0-4) 

## Impact Explanation
Pools using `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., KYC'd addresses, institutional partners) are fully bypassed. Any EOA can route through the allowlisted router and execute swaps against LP assets. This breaks the core invariant of the extension and constitutes direct loss of LP principal: unauthorized traders can drain pool assets at will, above Sherlock thresholds.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical swap interface for the protocol. Pool admins who allowlist the router — a natural and expected configuration for any pool that wants to support standard swap flows — unknowingly open the pool to all users. No special privileges or unusual conditions are required; any EOA can exploit this immediately by calling `exactInputSingle` or `exactInput` through the router.

## Recommendation
`SwapAllowlistExtension.beforeSwap` should check the originating user rather than the direct caller of `pool.swap()`. Options include: (1) require routers to forward the end-user address via `extensionData` and have the extension decode and check it; (2) have the pool pass `tx.origin` as an additional parameter (with documented trust assumptions); or (3) document explicitly that the extension only gates the direct caller of `pool.swap()` and is incompatible with router-mediated flows, and enforce that the router address must never be added to the allowlist.

## Proof of Concept
```solidity
// Foundry integration test sketch
function test_routerBypassesAllowlist() public {
    // Setup: pool with SwapAllowlistExtension
    // Pool admin sets: allowedSwapper[pool][router] = true
    // allowedSwapper[pool][attacker] is never set (false by default)

    vm.prank(attacker); // attacker is NOT individually allowlisted
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(token0),
            // ...
        })
    );
    // Swap succeeds: pool calls _beforeSwap(msg.sender=router, ...)
    // Extension checks allowedSwapper[pool][router] = true → no revert
    // Attacker receives output tokens despite not being individually authorized
}
```
The `beforeSwap` check `allowedSwapper[msg.sender][sender]` evaluates as `allowedSwapper[pool][router] = true`, so the revert at line 38 of `SwapAllowlistExtension.sol` is never reached. [6](#0-5)
