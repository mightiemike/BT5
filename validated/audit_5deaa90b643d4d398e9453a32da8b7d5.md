Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on `owner` instead of `sender`, allowing full allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the direct caller of `pool.addLiquidity`) and gates only on `owner` (the freely chosen position beneficiary). Because `MetricOmmPoolLiquidityAdder` accepts an arbitrary `owner` from any caller, any unprivileged address can bypass the deposit allowlist by supplying an already-allowlisted address as `owner` while paying for the deposit themselves. The allowlist provides zero protection against this path.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to the `_beforeAddLiquidity` dispatcher: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both arguments faithfully to the extension: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` names the first parameter with `address,` (unnamed, discarded) and gates only on `owner`: [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a caller-supplied `owner` with no restriction beyond `owner != address(0)`, stores `msg.sender` (the attacker) as `payer` in transient storage, and calls `pool.addLiquidity(positionOwner, ...)`: [4](#0-3) [5](#0-4) 

The hook sees `owner = allowlisted address` → passes; tokens are pulled from the attacker (payer) via the callback; LP shares are minted to the allowlisted `owner`. The actual depositor (`sender = LiquidityAdder`, representing the attacker) is never checked.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly gates on `sender`: [6](#0-5) 

The two extensions are architecturally inconsistent: the swap guard checks the economically relevant actor; the deposit guard does not.

The `addLiquidityWeighted` path compounds this: the probe call also passes through `beforeAddLiquidity` with the attacker-supplied `owner`, leaking pool state (token needs per bin) to the unauthorized caller before the paying deposit executes: [7](#0-6) 

## Impact Explanation

The pool admin's intent to restrict depositors is fully defeated. Any unprivileged caller with capital can:
- Add liquidity to any pool protected by `DepositAllowlistExtension`, concentrating or diluting liquidity in specific bins and shifting the pool cursor, altering the effective bid/ask spread for all subsequent swaps.
- Dilute existing LPs' share of spread and notional fees, constituting a direct reduction in owed LP assets.

This is a broken core access-control mechanism with direct impact on LP principal and pool liquidity distribution, meeting the "Admin-boundary break" and "direct loss of owed LP assets" criteria.

## Likelihood Explanation

- `MetricOmmPoolLiquidityAdder` is a public, permissionless periphery contract with no access control.
- The attacker only needs one allowlisted address, observable on-chain from `AllowedToDepositSet` events or direct `allowedDepositor` reads.
- No privileged access, no special token behavior, and no malicious pool setup is required.
- The attack is executable in a single transaction and is repeatable indefinitely.

## Recommendation

Gate on `sender` (the direct caller of `pool.addLiquidity`) rather than `owner` (the freely chosen position beneficiary), mirroring the swap guard pattern:

```solidity

## Proof of Concept

```
Setup:
  - Pool P deployed with DepositAllowlistExtension E.
  - Pool admin allowlists Alice: allowedDepositor[P][Alice] = true.
  - Bob (unauthorized) holds token0/token1 and has approved LiquidityAdder.

Attack (single tx, Bob as msg.sender):
  1. Bob calls:
       LiquidityAdder.addLiquidityExactShares(
           pool  = P,
           owner = Alice,   // allowlisted address observed on-chain
           salt  = 0,
           deltas = { binIdxs: [0], shares: [1_000_000] },
           maxAmountToken0 = X,
           maxAmountToken1 = Y,
           extensionData = ""
       )

  2. LiquidityAdder stores Bob as payer in transient storage, calls pool.addLiquidity(Alice, 0, deltas, KIND_PAY, "").

  3. Pool calls E.beforeAddLiquidity(LiquidityAdder, Alice, 0, deltas, "").
     Hook checks allowedDepositor[P][Alice] → true → passes.

  4. Pool mints LP shares to Alice; callback pulls tokens from Bob.

Result:
  - Bob (unauthorized) has deposited into the restricted pool.
  - Alice receives LP shares she did not request.
  - The deposit allowlist provided zero protection against Bob.

Foundry test plan:
  - Deploy pool with DepositAllowlistExtension.
  - Allowlist Alice, not Bob.
  - Call LiquidityAdder.addLiquidityExactShares from Bob with owner=Alice.
  - Assert the call succeeds (no revert), Bob's token balance decreases, Alice's LP shares increase.
```
