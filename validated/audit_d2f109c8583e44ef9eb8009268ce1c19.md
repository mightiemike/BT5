### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument it receives from the pool. When a user swaps directly, `sender` equals the user's address. When a user swaps through `MetricOmmSimpleRouter`, `sender` equals the **router's address**. If the pool admin allowlists the router (a natural action to enable router-mediated swaps for curated users), every unpermissioned user can bypass the per-user allowlist by routing through the public router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs its identity check as follows:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (correct), and `sender` is the first argument forwarded by the pool from its own `swap()` call:

```solidity
// MetricOmmPool.sol â€” swap() calls _beforeSwap with msg.sender
_beforeSwap(
    msg.sender,   // <-- whoever called pool.swap()
    recipient,
    ...
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
```

The router is `msg.sender` of `pool.swap()`, so the extension receives `sender = address(router)`. The allowlist lookup becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

The pool admin who wants to allow router-mediated swaps for their curated users must allowlist the router address. Once the router is allowlisted, **every user** â€” including those explicitly excluded from the per-user allowlist â€” can bypass the gate by calling any of the router's public entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`).

---

### Impact Explanation

**Direct loss of allowlist enforcement on curated pools.** A pool configured with `SwapAllowlistExtension` to restrict trading to KYC'd, whitelisted, or otherwise vetted counterparties loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The attacker needs no special privilege â€” the router is a public, permissionless contract. Any swap that should have been blocked by the allowlist executes at the live oracle price, draining pool liquidity to unauthorized parties. This matches the "allowlist bypass" impact class: disallowed users can trade on a curated pool, defeating the pool admin's access-control intent.

---

### Likelihood Explanation

**Medium-High.** The pool admin must allowlist the router to enable router-mediated swaps for their permitted users. This is a natural and expected configuration step â€” the router is the primary user-facing periphery contract. Any pool that (a) uses `SwapAllowlistExtension` and (b) allowlists the router (or sets `allowAllSwappers = true` to simplify setup) is immediately exploitable by any public caller. The trigger requires no privileged access, no special token, and no multi-step setup beyond a standard router call.

---

### Recommendation

The extension must recover the **original user** rather than the immediate caller of `pool.swap()`. Two approaches:

1. **Pass the real user through `extensionData`**: Require the router to encode the originating user in `extensionData` and have the extension decode and verify it. This requires a coordinated change to the router and extension.

2. **Check `recipient` instead of `sender` for swap allowlists**: If the intent is to gate who *receives* output, `recipient` is the correct field. If the intent is to gate who *initiates* the swap, the extension must receive the original EOA through a trusted channel (e.g., a signed payload or a dedicated router that forwards the real caller).

3. **Require direct pool calls for allowlisted pools**: Document that pools using `SwapAllowlistExtension` must not allowlist the router address, and instead require allowlisted users to call `pool.swap()` directly. This is a policy fix, not a code fix, and is fragile.

The cleanest fix is option 1: the router encodes `msg.sender` into `extensionData`, and the extension verifies it against the allowlist instead of the raw `sender` argument.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists Alice (address A) as a permitted swapper
  - Pool admin allowlists the router address to enable router-mediated swaps for Alice

Attack:
  - Bob (address B, not allowlisted) calls:
      MetricOmmSimpleRouter.exactInputSingle({pool: pool, ..., extensionData: ""})
  - Router calls pool.swap() with msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] â†’ TRUE (router is allowlisted)
  - Bob's swap executes at live oracle price

Result:
  - Bob, who is not on the allowlist, successfully swaps on a curated pool
  - The per-user allowlist is completely bypassed
  - Any user can repeat this for any swap direction or amount
```
