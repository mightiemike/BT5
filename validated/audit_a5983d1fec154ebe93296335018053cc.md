Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the actual trader, allowing any user to bypass the per-pool swap allowlist via the router - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is always `msg.sender` of `pool.swap()`. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original EOA. The allowlist therefore checks whether the router is allowlisted, not whether the individual trader is allowlisted. Any non-allowlisted user can bypass the swap gate by routing through the public router if the router address is allowlisted, or the pool admin is forced into an all-or-nothing choice: allowlist the router (opening the gate to everyone) or block it (making the router unusable for all users).

## Finding Description

**Root cause — `sender` is the immediate caller of `pool.swap()`, not the originating user.**

In `MetricOmmPool.swap()`:
```solidity
_beforeSwap(
    msg.sender,   // always the router when called via router
    recipient, ...
);
```
`msg.sender` here is `MetricOmmSimpleRouter`, because the router calls `pool.swap()` directly with no mechanism to forward the original user's address.

In `SwapAllowlistExtension.beforeSwap`:
```solidity
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```
`msg.sender` = pool, `sender` = router. The check is `allowedSwapper[pool][router]`.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only specific KYC'd EOAs.
2. Pool admin also allowlists the router address so that allowlisted users can use the router (a natural operational requirement).
3. Any non-allowlisted EOA calls `MetricOmmSimpleRouter.exactInputSingle()`.
4. The router calls `pool.swap()` with `msg.sender = router`.
5. `beforeSwap` receives `sender = router`, checks `allowedSwapper[pool][router] == true` → passes.
6. The non-allowlisted user's swap executes successfully, bypassing the allowlist entirely.

**Existing guards are insufficient:** The `onlyPool` modifier on `beforeSwap` only verifies the caller is the pool — it does not verify that `sender` is the actual originating user. The router's `_setNextCallbackContext` stores the original `msg.sender` only for the payment callback, not for the extension hook. There is no mechanism in `pool.swap()` to accept or forward an originating user address.

## Impact Explanation
The swap allowlist is a core access-control feature intended to restrict which addresses can trade on a given pool (e.g., KYC/AML compliance, permissioned pools). When the router is allowlisted (a necessary operational step for any pool that wants to support router-based swaps), the allowlist is completely nullified: any unprivileged user can swap by going through the public router. This constitutes a broken core pool functionality — the allowlist gate fails to enforce its intended restriction, allowing unauthorized traders to execute swaps and receive output tokens from a pool that was explicitly configured to deny them.

## Likelihood Explanation
The bypass is trivially reachable by any unprivileged user with no special setup: call `MetricOmmSimpleRouter.exactInputSingle()` targeting an allowlisted pool. The only precondition is that the router address is allowlisted on the pool, which is a natural and expected operational configuration for any pool that intends to support router-based trading. The attack is repeatable every block with no cost beyond gas.

## Recommendation
The pool's `swap()` function should accept an explicit `originator` parameter that extensions receive as `sender`, or the router should pass the original `msg.sender` through `extensionData` and the `SwapAllowlistExtension` should decode and verify it. The cleanest fix is to add an `originator` field to `pool.swap()` that the pool passes as `sender` to all extension hooks, with the router forwarding `msg.sender` in that field. Alternatively, `SwapAllowlistExtension` can be redesigned to check the payer stored in transient storage, but this couples the extension to the router's internal layout.
