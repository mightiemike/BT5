### Title
`ENABLE_SECP_OPS`-gated 1-byte secp opcodes silently return nil in consensus mode due to missing flag activation path — (`File: src/chia_dialect.rs`)

---

### Summary

`ClvmFlags::ENABLE_SECP_OPS` (0x0800) is defined and checked in the operator dispatch for 1-byte opcodes 64 (`secp256k1_verify`) and 65 (`secp256r1_verify`). However, this flag is never set by any production code path: it is absent from `MEMPOOL_MODE`, and no `softfork_extension` arm injects it. When the flag is absent, opcodes 64 and 65 fall through to `unknown_operator`, which in consensus mode (no `NO_UNKNOWN_OPS`) silently returns nil — indistinguishable from a successful signature verification. This is structurally identical to the reported bug: a security-critical dispatch table entry is defined and read, but the mechanism to activate it is missing, so the security check silently does nothing.

---

### Finding Description

In `src/chia_dialect.rs`, the 1-byte operator dispatch contains:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
_ => {
    return unknown_operator(allocator, o, argument_list, flags, max_cost);
}
``` [1](#0-0) 

When `ENABLE_SECP_OPS` is not set, both opcodes fall through to `unknown_operator`. In consensus mode (no `NO_UNKNOWN_OPS`), `unknown_operator` calls `op_unknown`, which returns nil with a formula-derived cost — the same return value as a *successful* secp verification:

```rust
fn unknown_operator(...) -> Response {
    if flags.contains(ClvmFlags::NO_UNKNOWN_OPS) {
        Err(EvalErr::Unimplemented(o))?
    } else {
        op_unknown(allocator, o, args, max_cost)
    }
}
``` [2](#0-1) 

`MEMPOOL_MODE` — the constant used for mempool validation — does not include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [3](#0-2) 

The `softfork_extension` function maps extension 0 → `Bls` and extension 1 → `Keccak`, but has no arm that injects `ENABLE_SECP_OPS`:

```rust
fn softfork_extension(&self, ext: u32) -> OperatorSet {
    match ext {
        0 => OperatorSet::Bls,
        1 => OperatorSet::Keccak,
        _ => OperatorSet::Default,
    }
}
``` [4](#0-3) 

The `op` function's extension-to-flag merge also has no arm for `ENABLE_SECP_OPS`:

```rust
let flags = self.flags
    | match extension {
        OperatorSet::Default => ClvmFlags::empty(),
        OperatorSet::Bls => ClvmFlags::empty(),
        OperatorSet::Keccak => ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD,
    };
``` [5](#0-4) 

The flag is exported to Python callers but only as a constant — there is no production code path that sets it automatically:

```rust
m.add("ENABLE_SECP_OPS", ClvmFlags::ENABLE_SECP_OPS.bits())?;
``` [6](#0-5) 

The 4-byte secp opcodes (0x13d61f00, 0x1c3a8f00) are dispatched unconditionally and are unaffected. Only the 1-byte aliases 64 and 65 are broken. [7](#0-6) 

---

### Impact Explanation

`op_secp256k1_verify` and `op_secp256r1_verify` are assert-style operators: they return nil on success and raise on failure. When opcode 64 or 65 (1-byte) is executed in consensus mode without `ENABLE_SECP_OPS`, `op_unknown` returns nil — identical to a successful verification. A puzzle that uses 1-byte opcode 64 to guard a coin spend would have its signature check silently bypassed in consensus mode, allowing unauthorized coin spending. **Impact: 4/10.**

---

### Likelihood Explanation

The Chia toolchain generates 4-byte secp opcodes (0x13d61f00 / 0x1c3a8f00), not 1-byte aliases. Exploitation requires a deliberately crafted puzzle using 1-byte opcode 64 or 65. Additionally, the mempool rejects such transactions (unknown op under `NO_UNKNOWN_OPS`), so a miner would need to include the transaction directly, bypassing the mempool. **Likelihood: 2/10.**

---

### Recommendation

Either:
1. Add a `softfork_extension` arm and a corresponding `OperatorSet::Secp` variant that injects `ENABLE_SECP_OPS` into the flags merge inside `op()`, mirroring the existing `Keccak` pattern; or
2. Remove opcodes 64 and 65 from the 1-byte dispatch entirely until the soft-fork activates, to prevent the silent nil-return path.

Additionally, include `ENABLE_SECP_OPS` in `MEMPOOL_MODE` once the soft-fork is intended to be active, so mempool and consensus agree on the operator's behavior.

---

### Proof of Concept

```python
# Craft a CLVM program: (64 pubkey msg sig)  -- 1-byte opcode 64
# In consensus mode (flags=0), this returns nil (success) without verifying.
# In mempool mode (MEMPOOL_MODE), this raises Unimplemented.

import clvm_rs

program = bytes.fromhex("ff40ff...")  # opcode 64 + dummy args
args    = bytes.fromhex("80")         # nil env

# Consensus mode: no flags → ENABLE_SECP_OPS absent → op_unknown → nil (success)
cost, result = clvm_rs.run_serialized_chia_program(program, args, 10**9, flags=0)
assert result.atom == b""  # nil — looks like successful verification

# Mempool mode: NO_UNKNOWN_OPS set → Unimplemented raised
try:
    clvm_rs.run_serialized_chia_program(program, args, 10**9, flags=clvm_rs.MEMPOOL_MODE)
    assert False, "should have raised"
except Exception:
    pass  # correctly rejected
```

The divergence between consensus (silent nil) and mempool (rejection) is the root cause. A puzzle relying on 1-byte opcode 64 for signature enforcement provides no actual security in consensus mode.
