Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Addresses to Bypass the Deposit Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the address actually calling `addLiquidity` and providing tokens) and instead checks `owner` (the LP-position recipient). Because any caller can freely specify an arbitrary `owner`, any non-allowlisted address can bypass the deposit guard by naming any allowlisted address as the position recipient. The pool admin's per-depositor access control is rendered completely ineffective.

## Finding Description

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both `sender` (the actual depositor) and `owner` (the position recipient) as distinct arguments to the extension: [1](#0-0) 

The interface explicitly separates these two roles: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` leaves `sender` unnamed (discarded) and checks `owner` instead: [3](#0-2) 

The admin setter `setAllowedToDeposit` stores permissions keyed by `depositor`, making the semantic intent clear — the guard is meant to restrict who provides funds, not who receives the LP position: [4](#0-3) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender`: [5](#0-4) 

**Exploit path:**
1. Pool is deployed with `DepositAllowlistExtension`; admin calls `setAllowedToDeposit(pool, alice, true)`.
2. Non-allowlisted `bob` calls `pool.addLiquidity(owner=alice, ...)`.
3. Pool calls `extension.beforeAddLiquidity(sender=bob, owner=alice, ...)`.
4. The check evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. Bob's tokens are transferred into the pool; the LP position is minted to `alice`.
6. Bob has deposited into a restricted pool without being allowlisted.

No existing guard prevents this. `allowedDepositor` is a public mapping, so `bob` can trivially read any allowlisted address on-chain to use as `owner`.

## Impact Explanation

The deposit allowlist guard is completely bypassed. Any non-allowlisted address can deposit tokens into a restricted pool by specifying any allowlisted address as `owner`. The non-allowlisted address provides the funds; the allowlisted address receives the LP position. The pool admin's access-control configuration is silently circumvented with no revert. This breaks the core invariant of the extension — that only allowlisted addresses may add liquidity — and constitutes broken core pool functionality causing direct loss of the pool's access-control integrity and potential fund impact (unauthorized liquidity provision into pools intended to be restricted).

## Likelihood Explanation

Exploitation requires only: (1) knowing one allowlisted address, which is publicly readable from the `allowedDepositor` mapping, and (2) calling `addLiquidity` directly on the pool with that address as `owner`. No privileged access, special tokens, flash loans, or complex setup is needed. Any non-allowlisted address can trigger this at will, repeatedly, on any pool using this extension.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor) instead of `owner`, consistent with how `SwapAllowlistExtension` handles `beforeSwap`:

```solidity
// Correct — checks actual caller providing funds
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```solidity
// 1. Deploy pool with DepositAllowlistExtension
// 2. Pool admin allowlists alice only
extension.setAllowedToDeposit(pool, alice, true);

// 3. Bob (not allowlisted) calls addLiquidity with owner=alice
// pool.addLiquidity calls extension.beforeAddLiquidity(sender=bob, owner=alice, ...)
// Check: allowedDepositor[pool][alice] == true → passes, no revert
// Result: Bob's tokens deposited, LP position minted to alice
// Bob has bypassed the deposit allowlist entirely
vm.prank(bob);
pool.addLiquidity(alice /*owner*/, salt, deltas, extensionData);
// Succeeds — bob is not allowlisted but deposit goes through
```
