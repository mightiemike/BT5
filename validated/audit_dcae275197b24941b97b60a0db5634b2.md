### Title
Fork Block Difficulty Computed Against Mainchain Ancestor Instead of Fork's Actual Ancestor — (`contract/src/dogecoin.rs`)

---

### Summary

In the Dogecoin build, `get_next_work_required` resolves the boundary block for difficulty calculation via `get_header_by_height`, which always returns the **current mainchain** block at a given height. When the block under validation is a fork block whose chain diverged before that height, the function silently uses the mainchain block's timestamp instead of the fork's actual ancestor's timestamp. This is structurally identical to the external report's root cause: a lookup keyed by a shared identifier (block height) returns the current live-state entry instead of the historically correct entry for the chain being evaluated.

The developers themselves flagged this with an unresolved TODO:

```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
```

---

### Finding Description

`get_next_work_required` in `dogecoin.rs` computes the expected difficulty for a new block using the timestamp difference between the previous block and the block at `height_first`. [1](#0-0) 

For Dogecoin after block 145,000, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `height_first = prev_block_header.block_height - 1`. [2](#0-1) 

The block at `height_first` is fetched via:

```rust
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [3](#0-2) 

This unconditionally resolves through `mainchain_height_to_header`, returning the **mainchain** block at `height_first` regardless of which chain the block under validation belongs to.

`check_pow` is called for both mainchain and fork blocks inside `submit_block_header`: [4](#0-3) 

For a mainchain block, the ancestor at `height_first` is always the mainchain block — correct. For a fork block whose chain diverged at or before `height_first`, the fork's actual ancestor at that height is a **fork block** stored only in `headers_pool`, not in `mainchain_height_to_header`. The lookup silently returns the wrong block.

The analog to the external report is exact:

| External Report | This Repository |
|---|---|
| `operatorNodes[A]` stores `nodeId_X` permanently | Fork block's chain history includes ancestor at `height_first` |
| `registeredValidators(P_X)` returns current live `validationID_B2` | `mainchain_height_to_header[height_first]` returns current mainchain block |
| Operator B's stake attributed to Operator A | Mainchain block's timestamp used for fork block's difficulty |

---

### Impact Explanation

An attacker who submits a fork that diverges at or before `height_first = prev_block_height - 1` can craft the fork ancestor at `height_first` (call it `F_{N-2}`) with a timestamp **earlier** than the mainchain block at the same height (`M_{N-2}.time`), while still satisfying the MTP constraint against the common ancestor.

The difficulty for the next fork block `F_N` is then computed as:

```
actual_timespan = F_{N-1}.time − M_{N-2}.time   ← wrong (uses mainchain block)
```

instead of the correct:

```
actual_timespan = F_{N-1}.time − F_{N-2}.time   ← correct (uses fork ancestor)
```

Since `F_{N-2}.time < M_{N-2}.time` (attacker-controlled), the incorrect timespan is **smaller** than the correct one, yielding an easier difficulty target for `F_N`. The Digishield clamping bounds the maximum reduction to approximately 25% (clamped to `min_timespan = retarget_timespan − retarget_timespan/4`), but this is a real reduction in the PoW security of fork block validation. Fork blocks can be accepted with less cumulative work than the protocol requires, weakening the chain-selection security guarantee.

---

### Likelihood Explanation

The attack requires only a 2-block-deep fork (so that the block at `height_first` is a fork block, not a shared ancestor). Two-block forks are a normal occurrence in any PoW chain. The attacker controls the timestamps of their fork blocks within the MTP and future-time constraints, and the manipulation is deterministic and requires no privileged access.
