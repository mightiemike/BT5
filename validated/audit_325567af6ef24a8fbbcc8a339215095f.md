Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Unpermissioned Depositors to Bypass the Deposit Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual token depositor, i.e. `msg.sender` of `addLiquidity`) and instead validates `owner` (the LP-share recipient). Any address not on the allowlist can bypass the guard by supplying an allowlisted address as `owner`, causing the pool to accept tokens from an unauthorized depositor while crediting LP shares to the allowlisted address. This renders the deposit allowlist entirely ineffective.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-controlled `owner` parameter as `owner` to `_beforeAddLiquidity`:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` encodes both and forwards them to the extension in order — `sender` first, `owner` second:

```solidity
// ExtensionCalling.sol L95-L98
_callExtensionsInOrder(
    BEFORE_ADD_LIQUIDITY_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

`DepositAllowlistExtension.beforeAddLiquidity` leaves the first parameter (`sender`) unnamed and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol L32-L42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The contract's own NatSpec states: *"Gates `addLiquidity` by depositor address, per pool."* The depositor is `sender` (`msg.sender` of `addLiquidity`), not `owner`. The asymmetry is confirmed by `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender`:

```solidity
// SwapAllowlistExtension.sol L31-L37
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
```

**Exploit path:**
1. Pool admin deploys a pool with `DepositAllowlistExtension`; sets `allowedDepositor[pool][Bob] = true`, Alice is not allowlisted.
2. Alice calls `pool.addLiquidity(owner = Bob, salt, deltas, callbackData, extensionData)`.
3. Extension receives `sender = Alice` (ignored), `owner = Bob` (checked). `allowedDepositor[pool][Bob] == true` → no revert.
4. Alice's tokens are transferred into the pool via the liquidity callback; Bob is credited LP shares.
5. Alice (unpermissioned) has successfully deposited into a restricted pool.

No existing guard prevents this: `removeLiquidity` enforces `msg.sender == owner`, but `addLiquidity` has no such constraint, and the extension check is the sole access-control gate for deposits.

## Impact Explanation
The deposit allowlist guard is rendered completely ineffective. Any unpermissioned address can inject tokens into a pool whose admin explicitly restricted deposits, violating the admin-configured access-control invariant. In KYC-gated or risk-managed pools, this is a direct bypass of the admin boundary. The allowlisted `owner` receives LP shares they did not request; the pool's liquidity composition is altered without admin consent. This constitutes a broken core pool functionality (access control) causing unauthorized fund flows into the pool, matching the "Admin-boundary break: pool admin exceeds caps, bypasses timelocks, or factory/oracle role checks are bypassed by an unprivileged path" allowed impact.

## Likelihood Explanation
Exploitation requires only a standard `addLiquidity` call with a known allowlisted address as `owner`. Allowlisted addresses are publicly discoverable via `allowedDepositor` mapping reads or emitted `AllowedToDepositSet` events. No privileged access, flash loans, or special setup is needed. The bypass is deterministic and repeatable by any unpermissioned address.

## Recommendation
Replace the unnamed first parameter with `sender` and gate on it instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][Bob] = true
  allowedDepositor[pool][Alice] = false  (Alice is NOT allowed)

Attack:
  Alice calls pool.addLiquidity(owner = Bob, salt, deltas, callbackData, extensionData)

Extension check (DepositAllowlistExtension.beforeAddLiquidity):
  msg.sender = pool
  owner      = Bob   ← checked against allowedDepositor[pool][Bob] == true → passes
  sender     = Alice ← silently ignored

Result:
  Alice's tokens transferred into pool via liquidity callback
  Bob credited with LP shares
  Alice (unpermissioned) has successfully deposited into a restricted pool
  Deposit allowlist is fully bypassed
