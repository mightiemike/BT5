Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router address, not the end user. If a pool admin allowlists the router address to enable standard-UX access, the allowlist is completely bypassed for every user on the network.

## Finding Description

**Root cause — allowlist check in `SwapAllowlistExtension.beforeSwap`:** [1](#0-0) 

The extension receives `sender` from the pool and checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

**Pool passes `msg.sender` as `sender` to `_beforeSwap`:** [2](#0-1) 

**Router calls `pool.swap()` directly — the original user's address is only stored in transient storage for the payment callback, never forwarded to the pool as the swap initiator:** [3](#0-2) 

`_setNextCallbackContext` stores `msg.sender` (the end user) in transient storage solely for token-pull purposes in the callback. The actual `pool.swap()` call has `msg.sender = router`. The pool then passes `router` as `sender` to every `beforeSwap` extension hook.

**Result:** `allowedSwapper[pool][router]` is checked, not `allowedSwapper[pool][end_user]`. The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) [5](#0-4) 

## Impact Explanation

Two fund-impacting outcomes arise:

1. **Allowlist bypass (High):** A pool admin who allowlists the router address — a natural step to enable the standard periphery UX for their curated users — inadvertently opens the pool to every user on the network. Any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle/exactInput/exactOutput/exactOutputSingle` and trade against a pool that was supposed to be restricted. LPs in a private/institutional pool are exposed to unauthorized counterparties, enabling adverse selection and direct LP principal loss.

2. **Allowlisted users blocked from router (Medium):** If the pool admin allowlists individual user addresses (not the router), those users cannot swap through the router at all — the extension sees the router address and reverts `NotAllowedToSwap`. The standard periphery path is broken for every legitimately allowlisted user.

This matches the allowed impact gate: an unprivileged path (`MetricOmmSimpleRouter`) bypasses an admin-configured access control boundary (`SwapAllowlistExtension`), with direct LP fund exposure as the consequence.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Any pool admin who deploys a `SwapAllowlistExtension` and wants their allowlisted users to use the standard router will naturally allowlist the router address. The bypass is then immediately available to every user on the network. No special knowledge or privileged access is required — any address can call the public router functions.

## Recommendation

The `SwapAllowlistExtension` must gate the economic actor (the end user), not the intermediary. Two options:

1. **Pass the original initiator through `extensionData`:** The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`. The extension decodes and checks that address instead of `sender`. This requires a convention between the router and the extension.

2. **Standardized `extensionData` prefix:** Add a dedicated field or a standardized `extensionData` prefix that the router always populates with the originating user, and have the extension decode it when present, falling back to `sender` for direct pool calls.

The core invariant to restore: the allowlist must check the same identity regardless of whether the swap arrives via direct pool call or via the router.

## Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — intending to let allowlisted users use the standard UX.
3. Attacker (not in any allowlist) calls:
     MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
4. pool.swap() is called with msg.sender = router.
5. _beforeSwap passes sender = router to SwapAllowlistExtension.
6. Extension checks allowedSwapper[pool][router] == true → passes.
7. Attacker's swap executes on the restricted pool.
   LP funds are exposed to an unauthorized counterparty.
```

The bypass requires zero privileged access and is reachable through all four public `MetricOmmSimpleRouter` entrypoints: `exactInputSingle` (L67), `exactInput` (L92), `exactOutputSingle` (L130), and `exactOutput` (L154). [6](#0-5) [7](#0-6)
