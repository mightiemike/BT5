### Title
Cross-Subaccount `lastLiquidationFees` Contamination Causes Premature Socialization — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a single global storage slot in `ClearinghouseStorage` with no per-subaccount scoping. `_finalizeSubaccount` unconditionally subtracts it from the working insurance value before calling `perpEngine.socializeSubaccount`, but never verifies that the stored value belongs to the subaccount currently being finalized. A liquidator who processes a large-position liquidation on subaccount A immediately before finalizing subaccount B (with `productId = type(uint32).max`) causes B's finalization to use artificially deflated insurance, triggering premature socialization that spreads losses to all depositors when the insurance fund could have absorbed them.

---

### Finding Description

`ClearinghouseStorage` declares `lastLiquidationFees` as a single protocol-wide `int128`: [1](#0-0) 

At the end of every `_handleLiquidationPayment` call — for **any** subaccount — this slot is overwritten: [2](#0-1) 

The design intent (documented in the comment at lines 581–585) is that when `_finalizeSubaccount` is called, `lastLiquidationFees` holds the fees from the **most recent liquidation step of the same subaccount**, so that the insurance fund is not "refilled" by those fees and then used to block socialization.

However, `_finalizeSubaccount` reads and uses `lastLiquidationFees` with no check that it belongs to `txn.liquidatee`: [3](#0-2) 

The reduced `v.insurance` is then passed directly into `perpEngine.socializeSubaccount`: [4](#0-3) 

and into the `insuranceCover` calculation: [5](#0-4) 

Only after all socialization and cover decisions are made does the code add `lastLiquidationFees` back: [6](#0-5) 

This add-back restores the stored `insurance` slot to a numerically plausible value, but it cannot undo the socialization that `perpEngine.socializeSubaccount` and `spotEngine.socializeSubaccount` already executed against the deflated figure.

---

### Impact Explanation

Let `I` = current insurance, `F_A` = fees from the last liquidation step of subaccount A.

When subaccount B is finalized immediately after:

| Step | With contamination | Correct behavior |
|---|---|---|
| Working insurance | `I − F_A` | `I` |
| Passed to `perpEngine.socializeSubaccount` | `I − F_A` | `I` |
| `insuranceCover` cap | `min(I − F_A − Δperp, −quoteB)` | `min(I − Δperp, −quoteB)` |
| `spotEngine.socializeSubaccount` triggered if | `v.insurance ≤ 0` (easier to reach) | harder to reach |

The socialization calls distribute `F_A` worth of losses to all depositors that the insurance fund could have absorbed. The stored `insurance` value after the call is also incorrect because `S(I − F_A) + F_A ≠ S(I)` in general (the socialization function is not linear in the insurance argument).

Additionally, `v.canLiquidateMore` (line 370) may be forced to `false`, bypassing the stricter `balance.amount == 0` spot-liability check at lines 372–383, allowing finalization to proceed when it should revert. [7](#0-6) 

---

### Likelihood Explanation

The preconditions are low-friction:

1. **Subaccount A** — any liquidatable position with non-trivial size; the liquidator earns fees, so this step is economically motivated regardless.
2. **Subaccount B** — must be under maintenance with all perp positions at zero and all spot assets ≤ 0 (i.e., already fully stripped, only a negative quote balance remains). Such subaccounts exist naturally at the tail end of multi-step liquidations.
3. **Ordering** — the liquidator submits both transactions to the sequencer back-to-back. No special privilege is required; any liquidator can do this.

No admin access, key compromise, or governance capture is needed. The contamination can also occur non-maliciously whenever two subaccounts happen to be finalized in close succession.

---

### Recommendation

Scope `lastLiquidationFees` to the subaccount being finalized. The simplest fix is to reset it to zero at the entry of `_finalizeSubaccount` (since no liquidation step for the target subaccount has occurred in this call):

```solidity
function _finalizeSubaccount(...) internal returns (bool) {
    if (txn.productId != type(uint32).max) {
        return false;
    }
    // Reset: no liquidation step was executed for txn.liquidatee in this call.
    lastLiquidationFees = 0;
    ...
}
```

Alternatively, track `lastLiquidationFees` as a `mapping(bytes32 => int128)` keyed by subaccount, so that each subaccount's finalization only deducts its own most-recent fees.

### Proof of Concept

```
State: insurance = 1_000_000e18, lastLiquidationFees = 0

Step 1 — liquidateSubaccountImpl(sender=attacker, liquidatee=subaccountA, productId=perpX, amount=large)
  → _handleLiquidationPayment executes
  → v.liquidationFees = 500_000e18  (large position)
  → insurance = 1_500_000e18
  → lastLiquidationFees = 500_000e18   ← global slot overwritten

Step 2 — liquidateSubaccountImpl(sender=attacker, liquidatee=subaccountB, productId=type(uint32).max)
  → _finalizeSubaccount called
  → v.insurance = 1_500_000e18
  → v.insurance -= lastLiquidationFees  →  v.insurance = 1_000_000e18
  → quoteBalance(B) = -1_200_000e18
  → v.canLiquidateMore = (−1_200_000 + 1_000_000) > 0  →  false  ← WRONG (should be true with full insurance)
  → perpEngine.socializeSubaccount(B, 1_000_000e18)  ← 500_000e18 less than available
  → insuranceCover = min(1_000_000, 1_200_000) = 1_000_000e18
  → v.insurance = 0  →  spotEngine.socializeSubaccount(B) called  ← WRONG
  → v.insurance += 500_000e18  →  insurance = 500_000e18

Correct behavior (lastLiquidationFees = 0 for B):
  → v.insurance = 1_500_000e18
  → v.canLiquidateMore = (−1_200_000 + 1_500_000) > 0  →  true
  → perpEngine.socializeSubaccount(B, 1_500_000e18)
  → insuranceCover = min(1_500_000, 1_200_000) = 1_200_000e18
  → v.insurance = 300_000e18  →  spotEngine.socializeSubaccount NOT called
  → insurance = 300_000e18

Delta: 200_000e18 of losses incorrectly socialized to depositors; spotEngine.socializeSubaccount
       triggered when insurance was solvent.
```

Fuzz assertion: after two sequential liquidations on distinct subaccounts, assert that `lastLiquidationFees` read inside `_finalizeSubaccount` equals zero (no prior liquidation step for the finalized subaccount in the same call).
