Audit Report

## Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing Factory-Enforced Spread Fee Ceiling - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`MetricOmmPoolFactory` enforces a hard ceiling on the pool admin's global spread fee via `maxAdminSpreadFeeE6`, but the `setPoolBinAdditionalFees` function applies no analogous cap on `addFeeBuyE6` / `addFeeSellE6`. A pool admin can set per-bin additional fees up to the `uint16` maximum (65,535 E6 ≈ 6.55%) on any bin, causing swappers to pay fees exceeding the factory-enforced ceiling on every swap through that bin.

## Finding Description

The factory defines `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) as the absolute hard ceiling. [1](#0-0) 

The factory owner sets `maxAdminSpreadFeeE6` (bounded by the hard ceiling), and every call to `setPoolAdminFees` is gated: [2](#0-1) 

However, the second admin fee lever — `setPoolBinAdditionalFees` — performs no cap check: [3](#0-2) 

The pool-side `setBinAdditionalFees` only validates the bin index range, not the fee values: [4](#0-3) 

`addFeeBuyE6` and `addFeeSellE6` are stored as `uint16` in `BinState`, so the maximum settable value is 65,535 E6 ≈ 6.55%: [5](#0-4) 

During every swap, `SwapMath` adds the per-bin fee directly on top of the base spread fee for each bin traversed: [6](#0-5) [7](#0-6) 

The same additive pattern applies in `getSellAndBuyPrices`: [8](#0-7) 

The documentation acknowledges the interaction with the global spread fee but provides no on-chain enforcement: [9](#0-8) 

## Impact Explanation

This is a confirmed admin-boundary break: the pool admin can exceed the factory-enforced fee cap via the per-bin path. Every swapper crossing a bin with `addFeeBuyE6 = 65535` or `addFeeSellE6 = 65535` pays up to 6.55% additional fee on top of the global spread fee, directly reducing the output token amount received. This is a direct, quantifiable loss of user principal on every swap through the affected bin(s). The active bin (the one most swaps traverse) is the highest-value target. The impact satisfies the "Admin-boundary break: pool admin exceeds caps" and "direct loss of user principal" criteria.

## Likelihood Explanation

The pool admin is explicitly semi-trusted: the factory caps their global fee precisely because they are not fully trusted with fee extraction. The per-bin path is reachable by any current `poolAdmin[pool]` in a single transaction with no timelock or delay. Any pool whose admin is compromised, acts adversarially, or has misaligned incentives can exploit this immediately and repeatedly.

## Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the global fee guard:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxBinAdditionalFeeE6` constant or state variable for finer-grained control.

## Proof of Concept

1. Factory owner deploys a pool with `maxAdminSpreadFeeE6 = 200_000` (20%) and `adminSpreadFeeE6 = 10_000` (1%).
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert occurs because `setPoolBinAdditionalFees` has no cap check and `setBinAdditionalFees` only validates the bin index.
3. A swapper calls `pool.swap(...)` with the active bin at index 0.
4. `SwapMath.buyToken0InBinSpecifiedIn` (or `buyToken0InBinSpecifiedOut`) computes the effective fee as `params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)`, adding 6.5535% on top of the global spread fee.
5. The swapper receives fewer output tokens than the protocol's fee cap system was designed to guarantee — a direct loss of principal on every swap through the affected bin.
