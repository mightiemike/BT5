### Title
`AllResources` V3 Transaction with Zero L2/L1-Data Gas Bounds Misclassified as `L1Gas` After Protobuf Round-Trip, Producing Wrong Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses the zero-ness of `l2_gas` and `l1_data_gas` as the sole discriminant between the `L1Gas` (pre-0.13.3) and `AllResources` (post-0.13.3) variants. Because `AllResources` with both fields set to zero is a valid, accepted transaction form, the discriminant is ambiguous: a legitimate `AllResources` transaction is silently downgraded to `L1Gas` after any P2P protobuf round-trip. The `get_tip_resource_bounds_hash` function hashes a different number of elements for the two variants, so the recomputed hash diverges from the hash the submitter signed, breaking hash/signature binding for those transactions.

### Finding Description

In `crates/apollo_protobuf/src/converters/transaction.rs` lines 417–436, the `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation decides which variant to reconstruct:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // defaults to zero
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← pre-0.13.3 path
} else {
    ValidResourceBounds::AllResources(...)       // ← post-0.13.3 path
})
```

The serializer (`From<ValidResourceBounds> for protobuf::ResourceBounds`, lines 471–490) emits explicit zero `ResourceLimits` for both `l2_gas` and `l1_data_gas` when the variant is `L1Gas`:

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),          // zero
    l1_data_gas: Some(ResourceBounds::default().into()), // zero
},
```

An `AllResources` transaction whose `l2_gas` and `l1_data_gas` are both zero produces an **identical** protobuf wire encoding. On deserialization the condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is true, so the receiver reconstructs `L1Gas` instead of `AllResources`.

This is the direct analog of the Saddle Finance bug: `tokenIndexes[token] = 0` (a valid assigned index) is indistinguishable from the mapping's default uninitialized value `0`. Here, `AllResources` with zero bounds (a valid assigned variant) is indistinguishable from the default `L1Gas` sentinel.

The divergence matters because `get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` lines 188–211 hashes a **different number of elements** for the two variants:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements total
    }
});
```

`AllResources({l1_gas, l2_gas:0, l1_data_gas:0})` hashes `[tip, l1_gas_packed, l2_gas_packed(=0), l1_data_gas_packed(=0)]` (3 resource elements), while `L1Gas(l1_gas)` hashes `[tip, l1_gas_packed, l2_gas_packed(=0)]` (2 resource elements). These produce different Poseidon digests even when all numeric values are identical.

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter is invoked in the P2P block-sync path for all three V3 transaction types:

- `DeclareTransactionV3Common::try_from(protobuf::DeclareV3Common)` — `crates/apollo_protobuf/src/transaction.rs` line 38
- `DeployAccountTransactionV3::try_from(protobuf::DeployAccountV3)` — `crates/apollo_protobuf/src/converters/transaction.rs` line 328
- `InvokeTransactionV3::try_from(protobuf::InvokeV3)` — (same file, analogous path)

### Impact Explanation

Any V3 transaction committed to a block with `AllResources` bounds where `l2_gas.max_amount == 0`, `l2_gas.max_price_per_unit == 0`, `l1_data_gas.max_amount == 0`, and `l1_data_gas.max_price_per_unit == 0` will be reconstructed as `L1Gas` by a syncing peer. The syncing peer recomputes the transaction hash using the 2-element preimage and obtains a value that differs from the hash stored in the block header. This constitutes:

- **Wrong transaction hash / wrong executable payload** bound to the transaction after P2P sync — the hash the signer committed to is not the hash the executing node uses.
- **Wrong gas computation mode**: `L1Gas` → `GasVectorComputationMode::NoL2Gas`; `AllResources` → `GasVectorComputationMode::All`. Fee validation, balance checks, and post-execution gas accounting all branch on this mode, producing different execution outcomes for the same transaction.

### Likelihood Explanation

The trigger condition — a V3 transaction with zero `l2_gas` and `l1_data_gas` — is reachable today. The gateway accepts all V3 transactions as `AllResourceBounds` (the RPC layer always uses `AllResourceBounds`). A user submitting a transaction that only consumes L1 gas and sets both L2 and L1-data gas bounds to zero produces exactly this case. The transaction is valid, accepted by the gateway, and committed to a block. Every peer that syncs the block via P2P will misclassify it.

### Recommendation

Replace the zero-value heuristic with an explicit discriminant. The protobuf `ResourceBounds` message already has `l1_data_gas` as `optional`; the intended fix is noted in the TODO comment on line 426. Once support for 0.13.2 (which omits `l1_data_gas`) is dropped, treat `l1_data_gas == None` as `L1Gas` and `l1_data_gas == Some(...)` (even if zero) as `AllResources`. Until then, add a boolean discriminant field to the protobuf message (e.g., `bool is_all_resources = 4;`) so the variant survives the round-trip without relying on value semantics.

### Proof of Concept

1. Construct a V3 invoke transaction with `AllResourceBounds { l1_gas: {max_amount:100, max_price_per_unit:1}, l2_gas: {0,0}, l1_data_gas: {0,0} }`.
2. Compute its hash via `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(...)` → hash includes 3 resource elements.
3. Serialize to `protobuf::ResourceBounds` via `From<ValidResourceBounds>` → wire bytes are `{l1_gas: {100,1}, l2_gas: {0,0}, l1_data_gas: {0,0}}`.
4. Deserialize via `ValidResourceBounds::try_from(protobuf::ResourceBounds)` → `l1_data_gas.is_zero() && l2_gas.is_zero()` is true → result is `ValidResourceBounds::L1Gas({100,1})`.
5. Recompute hash with `L1Gas` → hash includes only 2 resource elements → **different Poseidon digest**.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
