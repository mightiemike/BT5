### Title
Stale `BonusApy` Applied to Historical Reward Segments After Governance Tier Update — (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` fetches the **current** tier (including its `BonusApy`) at claim time and applies it uniformly to all historical bonded segments, including those that accrued before a governance-driven `BonusApy` change. Because `TokensPerShare` is correctly snapshotted per validator event but `BonusApy` is not, any governance update to a tier's `BonusApy` retroactively reprices every unclaimed historical segment for all positions in that tier.

---

### Finding Description

In `processEventsAndClaimBonus`, the tier is loaded once from the current store state: [1](#0-0) 

That single `tier` object — carrying the **current** `BonusApy` — is then passed into `computeSegmentBonus` for every historical event segment: [2](#0-1) 

And for the live trailing segment: [3](#0-2) 

`computeSegmentBonus` multiplies delegation tokens by `tier.BonusApy` directly: [4](#0-3) 

The design clearly intends historical accuracy: `evt.TokensPerShare` is a **snapshot** stored at the time of each validator event, so each segment uses the exchange rate that was in effect then. `BonusApy`, however, is never snapshotted — it is always the live governance value. This asymmetry means a governance `SetTier` call that changes `BonusApy` silently reprices all unclaimed historical segments for every position in that tier.

`SetTier` is explicitly documented as a governance entrypoint: [5](#0-4) 

---

### Impact Explanation

**If `BonusApy` is raised by governance:** every position holder who has not yet claimed receives the higher rate retroactively for all past bonded time. The `RewardsPoolName` module account is drained at a rate higher than the pool was funded to sustain, potentially exhausting it ahead of schedule and denying future claimants their rightful bonus.

**If `BonusApy` is lowered by governance:** every position holder who has not yet claimed receives less than the rate that was in effect when their delegation accrued. Users who claimed before the governance change receive the full historical rate; users who claimed after receive a reduced rate for the same historical period — a direct, concrete underpayment of user funds.

The corrupted value is the `sdk.Coins` amount transferred from `types.RewardsPoolName` to the position owner in: [6](#0-5) 

---

### Likelihood Explanation

Governance proposals to adjust tier `BonusApy` are a normal, expected protocol operation (the comment on `SetTier` explicitly lists "governance messages" as a caller). Any such proposal — even a well-intentioned one — immediately creates the inconsistency for all positions with unclaimed rewards. No attacker capability beyond submitting or voting on a governance proposal is required. The window of exposure is the entire period between the governance change and each position's next claim.

---

### Recommendation

Snapshot `BonusApy` alongside `TokensPerShare` in each `ValidatorEvent` (or in a separate per-tier history log), and use the snapshotted value when computing bonus for historical segments. The current trailing segment (from `segmentStart` to `blockTime`) may legitimately use the current `BonusApy`. This mirrors the existing pattern for `TokensPerShare` and ensures that a governance change to `BonusApy` only affects rewards accruing **after** the change takes effect.

---

### Proof of Concept

1. Alice creates a position in Tier 1 (`BonusApy = 10%`) and delegates 1,000,000 tokens. She does not claim for 6 months.
2. A governance proposal passes that raises Tier 1 `BonusApy` to 20%; `SetTier` is called.
3. Alice calls `MsgClaimRewards`. `processEventsAndClaimBonus` loads the current tier with `BonusApy = 20%` and computes all 6 months of historical segments at 20%.
4. Alice receives ~2× the bonus the pool was funded to pay for her historical period.
5. Bob, who held an identical position but claimed one block before the governance change, received the correct 10% rate for the same 6-month period.
6. The `RewardsPoolName` account is debited the inflated amount; if many positions are in the same tier, the pool can be exhausted before all legitimate claimants are paid.

func (k Keeper) SetTier(ctx context.Context, tier types.Tier) error {
	if err := tier.Validate(); err != nil {
		return err
	}
	if err := k.Tiers.Set(ctx, tier.Id, tier); err != nil {
		return errorsmod.Wrapf(err, "%s (tier id %d)", types.ErrTierStore.Error(), tier.Id)
	}
	return nil
}
