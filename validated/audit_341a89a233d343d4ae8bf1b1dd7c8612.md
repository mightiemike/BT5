### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable Double-Claiming of Bonus Rewards — (File: `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`ClaimTierRewards` loads all requested positions into a slice without deduplicating position IDs. It then delegates to `claimRewardsAndUpdatesPositions`, which processes each slice element independently and only persists the updated position state to the store **after** sending rewards. If the same position ID appears more than once in `msg.PositionIds`, the second iteration operates on a stale in-memory copy of the position (with the original `LastEventSeq`) and re-sends bonus rewards from the module pool.

---

### Finding Description

In `ClaimTierRewards`, positions are loaded into a slice in a loop with no duplicate check: [1](#0-0) 

The slice is then passed to `claimRewardsAndUpdatesPositions`: [2](#0-1) 

Inside `claimRewardsAndUpdatesPositions`, for each element the function:
1. Calls `claimBaseRewards` (withdraws from distribution module),
2. Calls `processEventsAndClaimBonus` (sends coins from `RewardsPoolName` to owner),
3. **Only then** calls `setPosition` to persist the updated `LastEventSeq` and bonus accrual checkpoint. [3](#0-2) 

If position ID `X` appears twice in `msg.PositionIds`, both `positions[0]` and `positions[1]` are loaded with the same initial state before the loop begins. After the first iteration processes `positions[0]` and calls `setPosition`, the store is updated — but `positions[1]` already holds the stale pre-update copy. The second iteration calls `processEventsAndClaimBonus` with the original `LastEventSeq`, re-fetches the same validator events from the store (their reference counts were only decremented from N to N-1, not to zero, as long as N > 1), recomputes the same bonus, and calls `bankKeeper.SendCoinsFromModuleToAccount` a second time: [4](#0-3) [5](#0-4) 

The `claimBaseRewards` path does not double-pay because `distributionKeeper.WithdrawDelegationRewards` returns zero on the second call and is handled gracefully: [6](#0-5) 

The bonus path, however, re-sends because the stale `LastEventSeq` causes `getValidatorEventsSince` to return the same events again, and `sufficientBonusPoolBalance` is the only guard — it only prevents the send if the pool is already empty.

---

### Impact Explanation

An attacker who owns any position for a validator that has at least two tiered-rewards positions (so validator event reference counts are ≥ 2) can submit a single `MsgClaimTierRewards` transaction with the same position ID repeated K times and receive K× the legitimate bonus reward. This directly drains the `RewardsPoolName` module account, which holds CRO earmarked for all other participants' bonus rewards. The corrupted value is the `RewardsPoolName` module account balance and every other position owner's future bonus entitlement.

---

### Likelihood Explanation

The entry path is a standard, unprivileged `MsgClaimTierRewards` transaction signed by any position owner. No special role, governance action, or social engineering is required. The only precondition — that the target validator has ≥ 2 tiered-rewards positions — is trivially satisfied by the attacker creating a second position themselves before executing the attack.

---

### Recommendation

Deduplicate `msg.PositionIds` before loading positions, for example:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", posId)
    }
    seen[posId] = struct{}{}
    // ... load and validate
}
```

Alternatively, follow the checks-effects-interactions pattern by persisting the updated position state to the store (calling `setPosition`) **before** sending any coins, so that a re-read of the same position ID in a subsequent iteration would observe the already-advanced `LastEventSeq`.

---

### Proof of Concept

1. Attacker creates two positions (`posA`, `posB`) both delegated to validator `V`. Validator events for `V` now have `ReferenceCount = 2`.
2. Attacker submits `MsgClaimTierRewards{ Owner: attacker, PositionIds: [posA, posA] }`.
3. First iteration: loads `posA` (state S0), calls `processEventsAndClaimBonus` → events have ref count decremented 2→1, bonus B sent to attacker, `setPosition` persists state S1 (advanced `LastEventSeq`).
4. Second iteration: uses the already-loaded stale copy of `posA` (state S0, original `LastEventSeq`), calls `processEventsAndClaimBonus` → events still exist (ref count 1 > 0), same bonus B computed and sent again, `setPosition` overwrites store with S1 again.
5. Attacker receives 2B instead of B. Repeating with `[posA, posA, posA, …]` (up to the pool balance) drains the `RewardsPoolName` account. [7](#0-6) [8](#0-7)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
