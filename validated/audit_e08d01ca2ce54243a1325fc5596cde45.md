### Title
Global `lastLiquidationFees` Overwritten by Interleaved Liquidations Causes Premature Socialization — (`core/contracts/ClearinghouseStorage.sol` / `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a single global `int128` storage slot shared across all subaccount liquidations. `_finalizeSubaccount` reads it to determine how much insurance to withhold from socialization. If any other subaccount's liquidation step executes between the last non-finalization step for subaccount A and the finalization call for A, the slot is overwritten with the other subaccount's fees. When `feesB > feesA`, the finalization of A sees a smaller effective insurance pool than it should, triggering `spotEngine.socializeSubaccount` when the insurance fund could have fully covered the bad debt.

---

### Finding Description

`lastLiquidationFees` is declared as a plain global in `ClearinghouseStorage`: [1](#0-0) 

Every call to `_handleLiquidationPayment` — for **any** subaccount — unconditionally overwrites it: [2](#0-1) 

`_finalizeSubaccount` then reads the slot to compute the insurance available for socialization: [3](#0-2) 

and adds it back at the very end before writing `insurance`: [4](#0-3) 

The design intent (stated in the comment at lines 581–585) is to hold back the fees from the **last liquidation step of the subaccount being finalized**, so that the insurance refill from that step does not block socialization. The mechanism only works correctly when no other liquidation executes between the last step for A and the finalization of A.

**Concrete state trace** (all values in quote units):

| Step | Action | `insurance` | `lastLiquidationFees` |
|------|--------|-------------|----------------------|
| 0 | initial | `I` | `0` |
| 1 | liquidate A (last step, `feesA = 10`) | `I + 10` | `10` |
| 2 | liquidate B (`feesB = 500`) | `I + 510` | `500` ← overwrites |
| 3 | finalize A | `v.insurance = I + 510 − 500 = I + 10` | — |

**Correct behavior** (no interleaving):

| Step | Action | `insurance` | `lastLiquidationFees` |
|------|--------|-------------|----------------------|
| 0 | initial | `I` | `0` |
| 1 | liquidate A (last step, `feesA = 10`) | `I + 10` | `10` |
| 2 | finalize A | `v.insurance = I + 10 − 10 = I` | — |

With interleaving, `v.insurance = I + feesA` instead of the correct `I + feesB`. When `feesB > feesA`, the finalization of A has **less** effective insurance (`I + feesA < I + feesB`), so `insuranceCover` is smaller and `spotEngine.socializeSubaccount` is called when it should not be: [5](#0-4) 

---

### Impact Explanation

Depositors bear losses via `spotEngine.socializeSubaccount` that the insurance fund could have absorbed. The insurance fund retains the excess (`feesB − feesA`) that should have been applied to the bad debt. This is a direct, measurable value transfer from depositors to the insurance fund, violating the invariant that socialization only occurs when insurance is genuinely exhausted.

---

### Likelihood Explanation

In any volatile market episode, multiple subaccounts become liquidatable simultaneously. The sequencer processes transactions in submission order. A liquidator who submits `[liquidate A (last step), liquidate B (large position), finalize A]` in sequence will reliably trigger this path. No privileged access, no oracle manipulation, and no governance capture is required — only two concurrent liquidatable subaccounts and normal sequencer FIFO ordering.

---

### Recommendation

Replace the global `lastLiquidationFees` with a per-subaccount mapping:

```solidity
mapping(bytes32 => int128) internal lastLiquidationFees;
```

Write `lastLiquidationFees[txn.liquidatee] = v.liquidationFees` in `_handleLiquidationPayment`, and read/clear `lastLiquidationFees[txn.liquidatee]` in `_finalizeSubaccount`. Alternatively, reset `lastLiquidationFees` to zero at the start of `_finalizeSubaccount` and pass the correct per-subaccount fee through a local variable or a dedicated storage slot keyed by subaccount.

---

### Proof of Concept

```solidity
// Setup: insurance = 1000, two liquidatable subaccounts A and B
// A has bad debt of 1100 (needs 1100 to cover, insurance = 1000 → should NOT socialize)
// B is a large position with feesB = 500

// Step 1: liquidate A last step → feesA = 10
//   insurance = 1010, lastLiquidationFees = 10

// Step 2: liquidate B → feesB = 500
//   insurance = 1510, lastLiquidationFees = 500  ← overwrites

// Step 3: finalize A
//   v.insurance = 1510 - 500 = 1010
//   insuranceCover = min(1010, 1100) = 1010  → covers only 1010 of 1100 bad debt
//   v.insurance = 0 → socializeSubaccount(A) called  ← WRONG

// Correct behavior (no interleaving):
//   v.insurance = 1510 - 10 = 1500
//   insuranceCover = min(1500, 1100) = 1100  → fully covers bad debt
//   v.insurance = 400 > 0 → NO socialization  ← CORRECT

// Assert: after finalization with interleaving, socializeSubaccount was called
// Assert: after finalization without interleaving, socializeSubaccount was NOT called
```

The sequence is directly reproducible in an unmodified Hardhat test by deploying the contracts, seeding the insurance fund, creating two liquidatable subaccounts, and submitting the three `liquidateSubaccount` transactions in the order above.
