### Title
Fork Difficulty Computed from Mainchain Ancestor Instead of Fork Ancestor, Allowing Underwork Fork Promotion — (`contract/src/dogecoin.rs`, `get_next_work_required`)

---

### Summary

`get_next_work_required` fetches the retarget boundary block via `get_header_by_height`, which unconditionally reads from `mainchain_height_to_header`. When the block being validated is a fork block, this returns the **mainchain** ancestor at that height rather than the **fork** ancestor. A malicious trusted relayer can craft fork block timestamps so that the wrong ancestor's timestamp produces an artificially easy `expected_bits`, allowing fork blocks to pass `check_pow` with less actual proof-of-work than required, and ultimately promoting the fork to `mainchain_tip_blockhash` with incorrect `chain_work`.

---

### Finding Description

**Buggy lookup — `dogecoin.rs:291-295`** [1](#0-0) 

The `TODO` comment is the developer's own acknowledgement of the flaw. `blocks_getter.get_header_by_height(height_first)` resolves to: [2](#0-1) 

This always reads `mainchain_height_to_header`, a map that only contains mainchain blocks. Fork blocks are stored only in `headers_pool` via `store_fork_header`: [3](#0-2) 

**When `check_pow` is called for a fork block**, the fork block has not yet been stored anywhere. The mainchain at `height_first` still holds the original mainchain block. The fork ancestor at that height is a different block with a different timestamp, but the contract uses the mainchain block's timestamp.

**Post-145,000 Dogecoin retarget arithmetic** [4](#0-3) 

`difficulty_adjustment_interval = 1`, so every block is a retarget block. `blocks_to_go_back` becomes `1` (line 282), making `height_first = prev_block_header.block_height - 1`. For a fork block at height H (prev at H-1), the contract fetches `mainchain[H-2]` instead of `fork[H-2]`.

**Difficulty formula sensitivity** [5](#0-4) 

`modulated_timespan = F_{H-1}.time − first_block_time`. If the attacker sets `F_{H-2}.time` (fork ancestor) close to `F_{H-1}.time`, the correct timespan is small → correct difficulty is hard. But if `M_{H-2}.time` (mainchain ancestor) is much earlier, the wrong timespan is large → wrong difficulty is easy. The contract enforces the wrong (easy) `expected_bits`.

**Entry point** [6](#0-5) 

`submit_blocks` is the production entry point. It is gated by `#[trusted_relayer]`, which is an externally reachable, staking-based role explicitly listed as a valid attacker entry path in the scope rules ("trusted-relayer submission path with attacker-relevant data"). The attacker registers as a trusted relayer through the normal staking process and submits crafted fork headers.

**Reorg trigger** [7](#0-6) 

Once `current_header.chain_work > total_main_chain_chainwork`, `reorg_chain` is called and `mainchain_tip_blockhash` is updated to the fork tip.

---

### Impact Explanation

`mainchain_tip_blockhash` is set to a fork tip whose constituent blocks were validated against an incorrect (mainchain-derived) difficulty. The fork's blocks carry less actual proof-of-work than the difficulty rules of the fork chain require. Any downstream consumer of `mainchain_tip_blockhash` (e.g., `verify_transaction_inclusion`, `verify_transaction_inclusion_v2`) now operates against a canonicalized chain that was promoted with underwork, breaking the fundamental security invariant of the light client.

---

### Likelihood Explanation

The precondition is that the attacker is a registered trusted relayer. The staking/registration mechanism is the only barrier. Once registered, the attacker can submit any sequence of fork headers with crafted timestamps. The timestamp fields are attacker-controlled inputs with no cross-chain oracle validation. The discrepancy between `M_{H-2}.time` and `F_{H-2}.time` can be made arbitrarily large by the attacker when constructing the fork, making the difficulty gap as wide as desired. The `TODO` comment confirms the developers are aware the lookup is wrong.

---

### Recommendation

Replace the height-based mainchain lookup with an ancestor walk along the fork chain. `get_prev_header` already traverses `headers_pool` by hash and correctly follows the fork lineage. `get_next_work_required` should walk back `blocks_to_go_back` steps from `prev_block_header` using `get_prev_header` in a loop, rather than calling `get_header_by_height`. This mirrors how the reference Dogecoin node resolves the retarget boundary block.

---

### Proof of Concept

State-machine test (Dogecoin feature, `skip_pow_verification = true` to isolate the difficulty logic):

1. Initialize with genesis at height 145,001 (post-new-difficulty-protocol). Submit mainchain blocks M1…M5. `M3.time = 1000`, `M4.time = 1060`, `M5.time = 1120`.

2. Submit fork block `F2` (height 145,003, prev = M1). Set `F2.time = 1059` (just under M3.time).

3. Submit fork block `F3` (height 145,004, prev = F2). Set `F3.time = 1060` (same as M4.time, so `F3.time - F2.time = 1`, a tiny timespan → correct difficulty for F4 should be very hard).

4. Submit fork block `F4` (height 145,005, prev = F3). The contract calls `get_header_by_height(145,003)` and gets `M3` (time=1000), not `F2` (time=1059). `modulated_timespan = F3.time − M3.time = 1060 − 1000 = 60` (large) vs. correct `1060 − 1059 = 1` (tiny). The contract computes easy `expected_bits` from the large timespan.

5. Construct `F4` with `bits = expected_bits_wrong` (easy). With `skip_pow_verification = false`, mine `F4` to the easy target. Verify that `F4.bits` would **fail** `check_pow` if `get_header_by_height` returned `F2` (time=1059) instead of `M3` (time=1000).

6. Submit `F4`. It passes. If `F4.chain_work > mainchain_tip.chain_work`, `reorg_chain` fires and `mainchain_tip_blockhash` is set to `F4.block_hash`, confirming the invariant break.
