### Title
`get_split_parent_shard_uids` stamps parent-shard DB keys with the child layout's `ShardVersion`, producing wrong `ShardUId` values at every resharding boundary — (File: `core/primitives/src/shard_layout/mod.rs`)

### Summary

`ShardLayout::get_split_parent_shard_uids` is documented to return the `ShardUId` values of shards that existed in the **previous** layout and were split in the **current** layout. However, it constructs those UIDs using `self.version()` — the version of the **current** (child) layout — instead of the version of the **previous** (parent) layout. Because `ShardUId` is the literal 8-byte DB key prefix used for every trie, flat-state, and memtrie lookup, every caller that relies on the returned set to locate parent-shard data silently addresses the wrong DB namespace.

### Finding Description

`ShardUId` encodes `(version: u32, shard_id: u32)` as an 8-byte little-endian pair and is used verbatim as the prefix for all on-disk state (trie nodes, flat state, chunk extras, etc.). The invariant is: a shard's UID must carry the version of the layout in which that shard was **active**. A parent shard that lived in layout `V_prev` must be addressed as `ShardUId { version: V_prev, shard_id: parent_id }`.

`get_split_parent_shard_uids` violates this invariant:

```rust
// core/primitives/src/shard_layout/mod.rs  L426-L434
/// Returns all the shards from the previous shard layout that were
/// split into multiple shards in this shard layout.
pub fn get_split_parent_shard_uids(&self) -> BTreeSet<ShardUId> {
    let parent_shard_ids = self.get_split_parent_shard_ids();
    parent_shard_ids
        .into_iter()
        .map(|shard_id| ShardUId::new(self.version(), shard_id))  // ← wrong version
        .collect()
}
```

`self` is the **new** layout (version `V_new`). The parent shard IDs returned by `get_split_parent_shard_ids` belong to the **previous** layout (version `V_prev = V_new − 1`). The function stamps them with `V_new`, producing UIDs that do not exist in the database.

The `ShardUId` byte layout confirms the mismatch:

```rust
// core/primitives/src/shard_layout/mod.rs  L497-L502
pub fn to_bytes(&self) -> [u8; 8] {
    let mut res = [0; 8];
    res[0..4].copy_from_slice(&u32::to_le_bytes(self.version));
    res[4..].copy_from_slice(&u32::to_le_bytes(self.shard_id));
    res
}
```

The first four bytes are the version. A lookup with `version=V_new` will never find data written under `version=V_prev`.

The same wrong-version pattern appears in `ShardLayoutV3::ancestor_uids`, which hard-codes `VERSION` (= 3) for all ancestor UIDs regardless of which layout generation each ancestor actually belonged to:

```rust
// core/primitives/src/shard_layout/v3.rs  L304-L309
pub fn ancestor_uids(&self, shard_id: ShardId) -> Vec<ShardUId> {
    self.shards_ancestor_map
        .get(&shard_id)
        .map(|ancestor_ids| ancestor_ids.iter().map(|id| ShardUId::new(VERSION, *id)).collect())
        .unwrap_or_default()
}
```

### Impact Explanation

Every caller of `get_split_parent_shard_uids` receives wrong UIDs:

**1. `get_resharding_parent_shard_uid` (`chain/epoch-manager/src/adapter.rs` L908–L923)**

```rust
let split_parent_shard_uids = next_layout.get_split_parent_shard_uids();
Ok(split_parent_shard_uids.into_iter().next())
```

This returns the wrong UID to `chain/chain/src/chain.rs`, which uses it to locate the parent shard's memtrie for background preloading before resharding executes. The memtrie map is keyed by the correct UID (`V_prev`); the wrong UID (`V_new`) produces a miss, so preloading silently does nothing. When resharding executes, the parent memtrie is absent, causing the resharding to fail or fall back to a slow disk-based path — a liveness-breaking outcome at the epoch boundary.

**2. `chain/chain/src/garbage_collection.rs`**

Uses the returned UIDs to identify which parent-shard DB entries to GC after resharding. Wrong UIDs cause the GC to skip the parent shard's data entirely, leaking storage indefinitely.

**3. `core/store/src/archive/cold_storage.rs`**

Uses the returned UIDs to archive parent-shard state. Wrong UIDs cause the archival to miss the parent shard's data.

### Likelihood Explanation

The bug fires at every resharding epoch boundary. `get_resharding_parent_shard_uid` is called from `chain/chain/src/chain.rs` (2 call sites) and `chain/client/src/sync/state/shard.rs` (1 call site) on every block that crosses an epoch boundary where the shard layout changes. Dynamic resharding (`ProtocolFeature::DynamicResharding`) is the primary activation path for the V2→V3 layout transition, which is the first case where this code path is exercised with a version increment under the new memtrie-preloading machinery. The GC and cold-storage callers are also active on any resharding epoch.

### Recommendation

In `get_split_parent_shard_uids`, the parent shard IDs must be paired with the **previous** layout's version. The function signature should accept the previous layout (or its version) as a parameter:

```rust
pub fn get_split_parent_shard_uids(&self, prev_layout: &ShardLayout) -> BTreeSet<ShardUId> {
    let parent_shard_ids = self.get_split_parent_shard_ids();
    parent_shard_ids
        .into_iter()
        .map(|shard_id| ShardUId::new(prev_layout.version(), shard_id))
        .collect()
}
```

`get_resharding_parent_shard_uid` already has `current_layout` in scope and must pass it:

```rust
let split_parent_shard_uids = next_layout.get_split_parent_shard_uids(&current_layout);
```

`ShardLayoutV3::ancestor_uids` must similarly derive each ancestor's UID from the layout version in which that ancestor was active, not from the constant `VERSION = 3`.

### Proof of Concept

Consider a V1→V2 resharding where shard `S1` (stored under `version=1`) is split into `S2` and `S3` (version=2):

1. `next_layout` = V2 layout (`version()` returns 2).
2. `next_layout.get_split_parent_shard_uids()` returns `{ShardUId { version: 2, shard_id: S1 }}`.
3. `S1`'s trie data is stored under the 8-byte key prefix `[1,0,0,0, S1_bytes]` (version=1).
4. The returned UID produces key prefix `[2,0,0,0, S1_bytes]` — absent from the DB.
5. `get_resharding_parent_shard_uid` returns this wrong UID to `chain.rs`.
6. The memtrie map lookup for `ShardUId { version: 2, shard_id: S1 }` returns `None`.
7. Background memtrie preloading is skipped; resharding proceeds without the parent memtrie loaded, causing failure or severe performance degradation at the epoch boundary. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    fn get_resharding_parent_shard_uid(
        &self,
        epoch_id: &EpochId,
        last_block_hash: &CryptoHash,
    ) -> Result<Option<ShardUId>, EpochError> {
        let next_epoch_id = self.get_next_epoch_id(last_block_hash)?;
        let current_layout = self.get_shard_layout(epoch_id)?;
        let next_layout = self.get_shard_layout(&next_epoch_id)?;
        if current_layout == next_layout {
            return Ok(None);
        }
        let split_parent_shard_uids = next_layout.get_split_parent_shard_uids();
        // There should be exactly one shard split when layout changes
        debug_assert!(split_parent_shard_uids.len() == 1);
        Ok(split_parent_shard_uids.into_iter().next())
    }
