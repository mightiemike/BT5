### Title
Deprecated `verify_transaction_inclusion` Accepts Internal Merkle-Tree Node Hashes as `tx_id`, Enabling Proof-Verification Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` remains a live, unpermissioned public entry point despite being deprecated. It accepts any 32-byte value as `tx_id` — including hashes of internal Merkle-tree nodes — without filtering for leaf-level transaction hashes. An unprivileged NEAR caller can supply a crafted `tx_id` equal to an internal node of a real block's Merkle tree, pair it with a valid sibling proof, and receive `true`, falsely certifying a non-existent Bitcoin transaction.

---

### Finding Description

The contract exposes two transaction-inclusion verification methods. The v2 method (`verify_transaction_inclusion_v2`) was introduced specifically to close the 64-byte Merkle-proof forgery path by requiring a coinbase proof at index 0. The v1 method (`verify_transaction_inclusion`) was deprecated at the same time. However, `#[deprecated]` in Rust is a compile-time lint only; it does not restrict runtime call access. The function remains fully reachable by any NEAR account. [1](#0-0) 

The function's own documentation acknowledges the flaw: [2](#0-1) 

The verification logic in `compute_root_from_merkle_proof` is position-driven and mathematically correct for any 32-byte starting value — it does not distinguish a leaf transaction hash from an internal node hash: [3](#0-2) 

The only guard inside `verify_transaction_inclusion` is a non-empty proof check: [4](#0-3) 

This check does not filter the *type* of the supplied `tx_id`. An internal node hash at height `h` in the Merkle tree has a valid sibling path of length `h` that correctly reconstructs the Merkle root, so the comparison on line 322 succeeds.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a privileged action (releasing bridged funds, minting wrapped tokens, confirming a cross-chain event) can be deceived into treating a fabricated internal-node hash as a confirmed Bitcoin transaction. The corrupted value is the **proof result** returned to the caller: `true` for a transaction that was never broadcast or mined.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. The attacker needs only:
1. A block hash already stored in `headers_pool` (publicly queryable via `get_block_hash_by_height`).
2. Knowledge of that block's Merkle tree structure (public Bitcoin data).
3. The ability to call a public NEAR contract method.

The 64-byte forgery technique is well-documented (referenced in the contract's own comments at line 268) and tooling exists to construct such proofs. [5](#0-4) 

---

### Recommendation

Remove `verify_transaction_inclusion` entirely from the contract. It is already deprecated and superseded by `verify_transaction_inclusion_v2`. Retaining a callable but insecure public method creates an unfiltered input path that downstream integrators may unknowingly rely on. If removal is not immediately possible, add an `#[access_control]` guard or an unconditional `env::panic_str` body so the method cannot be invoked at runtime.

---

### Proof of Concept

1. Query `get_block_hash_by_height(H)` to obtain a known mainchain block hash `B`.
2. Fetch block `B`'s full transaction list from a Bitcoin node. Compute its Merkle tree. Select any internal node `N` at height `h ≥ 1` with sibling path `[s_1, …, s_h]` and tree-position index `p`.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (internal node hash, not a real transaction)
   - `tx_block_blockhash = B`
   - `tx_index = p`
   - `merkle_proof = [s_1, …, s_h]`
   - `confirmations = 1`
4. `compute_root_from_merkle_proof(N, p, [s_1,…,s_h])` reconstructs the correct Merkle root; the comparison at line 322 returns `true`.
5. The caller receives `true` — a forged proof of inclusion for a transaction that does not exist. [6](#0-5) [3](#0-2)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
