### Title
Silent Failure in `_executeSlowModeTransaction` Permanently Locks User Deposit Funds When `DepositCollateral` Slow Mode Transaction Reverts — (`core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint._executeSlowModeTransaction()` wraps `processSlowModeTransaction()` in a try/catch that silently discards all non-OOG failures. Because `depositCollateralWithReferral()` transfers user tokens **before** queuing the slow mode transaction, any failure during execution permanently locks those tokens in the contract. The slow mode entry is deleted before the try/catch runs, so no retry is possible. The comment `// try return funds now removed` in the catch block explicitly confirms that the fund-recovery path was deliberately removed, leaving no remediation path for the user.

---

### Finding Description

`depositCollateralWithReferral()` follows a two-phase pattern:

**Phase 1 — token custody (immediate):** [1](#0-0) 

Tokens are pulled from the caller into the contract via `handleDepositTransfer`.

**Phase 2 — accounting credit (deferred, up to 3 days later):** [2](#0-1) 

A `DepositCollateral` slow mode transaction is queued with a `SLOW_MODE_TX_DELAY` delay.

When the slow mode transaction is eventually executed, `_executeSlowModeTransaction()` **deletes the queue entry first**, then wraps execution in a try/catch: [3](#0-2) 

The critical lines are:
- Line 194: `delete slowModeTxs[_slowModeConfig.txUpTo++]` — the entry is gone before execution is attempted.
- Lines 207–227: the try/catch swallows any revert from `processSlowModeTransaction`.
- Line 226: `// try return funds now removed` — the fund-recovery branch was explicitly deleted.

If `processSlowModeTransaction` → `EndpointTx.processSlowModeTransactionImpl` → `clearinghouse.depositCollateral` reverts for any reason (e.g., the product was delisted during the 3-day window via a `DelistProduct` slow mode transaction), the catch block runs, does nothing, and the user's tokens remain in the `Endpoint` contract with no recovery path. [4](#0-3) 

---

### Impact Explanation

User tokens transferred in `handleDepositTransfer` are permanently locked in the `Endpoint` contract. Because the slow mode queue entry is deleted before the try/catch, there is no retry mechanism. The user cannot re-submit the deposit (the tokens are already held by the contract) and cannot recover them (no withdrawal path exists for a failed deposit). This is a direct, irreversible loss of user funds.

---

### Likelihood Explanation

The trigger requires `clearinghouse.depositCollateral` to revert after the deposit was queued. The most realistic scenario is a product being delisted (via `DelistProduct`) during the 3-day slow mode window. While `DelistProduct` is an admin-gated action, it is a normal protocol operation (products can be retired), and the 3-day window makes the race condition plausible. Any user who deposited to a product that is subsequently delisted before their slow mode transaction executes will permanently lose their funds. The `depositCollateralWithReferral` entry point is public and callable by any user. [5](#0-4) 

---

### Recommendation

Restore the fund-recovery branch in the catch block of `_executeSlowModeTransaction`. When a `DepositCollateral` slow mode transaction fails, the contract should transfer the deposited tokens back to the original sender. Alternatively, instead of deleting the queue entry before execution, only delete it on success and allow re-execution (or explicit cancellation with refund) on failure.

---

### Proof of Concept

1. User calls `depositCollateralWithReferral(subaccount, productId, 100e18, "")`.
   - `handleDepositTransfer` pulls 100e18 tokens from the user into `Endpoint`.
   - A `DepositCollateral` slow mode tx is queued at index `N`, executable in 3 days.

2. Admin submits and executes a `DelistProduct` slow mode transaction for `productId` before the user's tx executes.

3. After 3 days, anyone calls `executeSlowModeTransaction()`.
   - `_executeSlowModeTransaction` deletes `slowModeTxs[N]`.
   - `try this.processSlowModeTransaction(...)` calls `clearinghouse.depositCollateral`, which reverts because the product no longer exists.
   - The catch block runs; `gasleft()` is well above the OOG threshold; execution continues silently.

4. Result: 100e18 tokens remain in `Endpoint`. The slow mode entry is gone. The user has no recourse — no retry, no refund.
