### Title
Hardcoded `MAX_DAILY_FUNDING_RATE` Cannot Be Updated Per-Product, Causing Permanent Funding Rate Miscalibration — (`File: core/contracts/PerpEngineState.sol`)

---

### Summary

`MAX_DAILY_FUNDING_RATE` is a file-level Solidity constant hardcoded at `0.02` (2%/day) in `PerpEngineState.sol`. It is applied as a global cap on the funding rate for **every** perpetual product. Because it is a compile-time constant, it cannot be adjusted without a full contract upgrade and liquidity migration. The protocol's own comment acknowledges this is a temporary design choice. When market conditions change — such as listing a highly volatile asset or during periods of extreme market stress — the cap will either be too strict (perp price diverges from index, funding mechanism fails to balance longs/shorts) or too permissive (funding rate is unconstrained relative to the product's risk profile), causing systematic economic flaws for all traders in the affected market.

---

### Finding Description

In `PerpEngineState.sol`, three file-level constants govern the funding rate calculation:

```solidity
int128 constant EMA_TIME_CONSTANT_X18 = 998334721450938752;
int128 constant ONE_DAY_X18 = 86400_000000000000000000; // 24 hours

// we will want to config this later, but for now this is global and a percentage
int128 constant MAX_DAILY_FUNDING_RATE = 20000000000000000; // 0.02
``` [1](#0-0) 

The `updateStates()` function — callable only by the Endpoint (i.e., the sequencer) — applies this cap to the sequencer-supplied `avgPriceDiffs` for every product:

```solidity
int128 maxPriceDiff = MAX_DAILY_FUNDING_RATE.mul(indexPriceX18);

if (priceDiffX18.abs() > maxPriceDiff) {
    priceDiffX18 = (priceDiffX18 > 0)
        ? maxPriceDiff
        : -maxPriceDiff;
}

int128 paymentAmount = priceDiffX18.mul(dtX18).div(ONE_DAY_X18);
state.cumulativeFundingLongX18 += paymentAmount;
state.cumulativeFundingShortX18 += paymentAmount;
``` [2](#0-1) 

The cap is applied uniformly across all perp products regardless of their individual volatility profiles. Because `MAX_DAILY_FUNDING_RATE` is a compile-time constant (not a per-product storage variable), there is no admin function, governance path, or upgrade mechanism that can change it without redeploying the entire `PerpEngine` contract and migrating all open positions and state.

The protocol's own inline comment — *"we will want to config this later, but for now this is global and a percentage"* — explicitly acknowledges this is a known design debt. [3](#0-2) 

---

### Impact Explanation

**Too-strict cap scenario (most likely):** If a listed perp product experiences real market funding rates above 2%/day (common for volatile altcoins or during market stress), the on-chain cap prevents `cumulativeFundingLongX18` / `cumulativeFundingShortX18` from accumulating at the rate needed to close the mark-index spread. The funding mechanism fails to incentivize arbitrageurs to balance longs and shorts. The perp mark price can sustain a persistent premium or discount relative to the index price. Any trader holding a position in the direction of the divergence pays less (or receives less) funding than the true market rate, representing a direct transfer of value away from the counterparty side.

**Too-permissive cap scenario:** For a low-volatility product (e.g., a stablecoin perp), a 2%/day cap is far too high, allowing the sequencer to apply funding payments that are economically unjustified, draining one side of the book.

The corrupted on-chain state is `state.cumulativeFundingLongX18` and `state.cumulativeFundingShortX18` for every affected product, which directly determines the `vQuoteBalance` delta applied to every trader's position on the next balance update. [4](#0-3) 

---

### Likelihood Explanation

Medium-to-high. The protocol is deployed on Ink Chain (an L2), which — like Optimism Bedrock — is subject to chain-level parameter changes. More importantly, the protocol is designed to list multiple perp products with different volatility profiles. A single global 2%/day cap is structurally mismatched to a multi-product exchange. The comment in the source code confirms the developers already identified this as a problem requiring future remediation. Any new volatile product listing immediately triggers the too-strict scenario.

---

### Recommendation

Replace the file-level constant with a per-product configurable parameter stored in the `State` struct or a separate mapping. Expose an admin/governance function (with appropriate access control and a freeze period for LPs/traders to exit if they disagree with the change) to update the cap per product. This mirrors the Basin fix: rename the constant to make its purpose clear, and make it updatable through a governed process.

```solidity
// In State struct or separate mapping:
mapping(uint32 => int128) public maxDailyFundingRateX18;

// In updateStates():
int128 maxPriceDiff = maxDailyFundingRateX18[productId].mul(indexPriceX18);
```

---

### Proof of Concept

1. Protocol lists a volatile perp product (e.g., a memecoin perpetual) where real market funding rates routinely reach 5%/day.
2. The sequencer calls `updateStates(dt, avgPriceDiffs)` with `avgPriceDiffs[i]` reflecting the true 5%/day mark-index spread.
3. On-chain, `maxPriceDiff = MAX_DAILY_FUNDING_RATE.mul(indexPriceX18)` caps the effective rate at 2%/day.
4. `paymentAmount` is computed using the capped value, so `cumulativeFundingLongX18` accumulates at only 2%/day instead of 5%/day.
5. The mark price remains ~3%/day above the index price in persistent equilibrium — the funding mechanism cannot close the gap.
6. A trader opens a large long position, pays only 2%/day funding instead of the market-clearing 5%/day, and profits from the sustained premium at the expense of shorts and the protocol's risk model.
7. There is no on-chain mechanism to raise the cap without a full `PerpEngine` redeployment and state migration. [5](#0-4)

    function updateStates(uint128 dt, int128[] calldata avgPriceDiffs)
        external
        onlyEndpoint
    {
        int128 dtX18 = int128(dt).fromInt();
        for (uint32 i = 0; i < avgPriceDiffs.length; i++) {
            uint32 productId = productIds[i];
            State memory state = states[productId];
            if (state.openInterest == 0) {
                continue;
            }
            require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
            {
                int128 indexPriceX18 = _risk(productId).priceX18;

                // cap this price diff
                int128 priceDiffX18 = avgPriceDiffs[i];

                int128 maxPriceDiff = MAX_DAILY_FUNDING_RATE.mul(indexPriceX18);

                if (priceDiffX18.abs() > maxPriceDiff) {
                    // Proper sign
                    priceDiffX18 = (priceDiffX18 > 0)
                        ? maxPriceDiff
                        : -maxPriceDiff;
                }

                int128 paymentAmount = priceDiffX18.mul(dtX18).div(ONE_DAY_X18);
                state.cumulativeFundingLongX18 += paymentAmount;
                state.cumulativeFundingShortX18 += paymentAmount;

                emit FundingPayment(
                    productId,
                    dt,
                    state.openInterest,
                    paymentAmount
                );
            }
            _setState(productId, state);
            emit PriceQuery(productId);
        }
    }
