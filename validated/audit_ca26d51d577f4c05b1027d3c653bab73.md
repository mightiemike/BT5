Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates Position Owner Instead of Actual Depositor, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and validates only the `owner` (position recipient). Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` allows any caller to specify an arbitrary `owner`, an address not on the allowlist can deposit tokens into a restricted pool by routing through any allowlisted owner address, completely defeating the access-control invariant the pool admin configured.

## Finding Description
`MetricOmmPool.addLiquidity` calls the extension hook with `msg.sender` as the first argument:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

When called through `MetricOmmPoolLiquidityAdder`, `msg.sender` to the pool is the LiquidityAdder contract address — not the end user. The LiquidityAdder stores the real payer (`msg.sender` to the adder) only in transient context for callback settlement:

```solidity
return _addLiquidity(pool, owner, salt, deltas, msg.sender, maxAmountToken0, maxAmountToken1, extensionData);
``` [2](#0-1) 

```solidity
_setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
``` [3](#0-2) 

This payer identity is never forwarded to the extension. The extension discards the first parameter entirely and only checks `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [4](#0-3) 

`_validateOwner` in the LiquidityAdder only rejects `address(0)`, placing no restriction on which allowlisted address can be named as `owner`: [5](#0-4) 

The exploit path is: unprivileged `bob` calls `addLiquidityExactShares(pool, alice, ...)` where `alice` is allowlisted. The pool calls `beforeAddLiquidity(liquidityAdder, alice, ...)`. The extension checks `allowedDepositor[pool][alice]` → `true` → no revert. Bob pays tokens via transient callback context; alice receives the position. If bob controls alice (two wallets), alice calls `removeLiquidity` (which requires `msg.sender == owner`) and returns the tokens to bob — net effect: bob deposited into a restricted pool with zero restriction. [6](#0-5) 

## Impact Explanation
The deposit allowlist is the pool admin's primary mechanism to restrict who may provide liquidity (e.g., KYC/compliance gating, curated LP sets). The check on `owner` rather than the actual depositor constitutes an admin-boundary break: an unprivileged path bypasses a pool admin access-control check. Even without recovering funds, unauthorized capital enters the pool, violating the invariant the admin configured. With a second controlled wallet, the bypass is complete and lossless.

## Likelihood Explanation
No special privilege is required — any EOA or contract can call `addLiquidityExactShares` with an arbitrary `owner`. Allowlisted addresses are discoverable on-chain from `AllowedToDepositSet` events. The `MetricOmmPoolLiquidityAdder` is the standard periphery entry point, so the bypass path is the normal user flow. Likelihood is high: trivially reachable with no preconditions beyond knowing one allowlisted address.

## Recommendation
Check the actual depositor (`sender`, the first parameter) rather than — or in addition to — `owner`. Note that when `addLiquidity` is called through `MetricOmmPoolLiquidityAdder`, `sender` will be the LiquidityAdder contract address, not the end user. The pool should either (a) pass the real payer through `extensionData`, or (b) the allowlist should gate on `owner` only when the intent is to restrict position ownership rather than token origin. At minimum, the extension should not silently discard the `sender` argument:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    address pool = msg.sender;
    if (!allowAllDepositors[pool]
        && !allowedDepositor[pool][sender]
        && !allowedDepositor[pool][owner])
    {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true   // alice is KYC'd (bob's second wallet)
  allowedDepositor[pool][bob]   = false  // bob is NOT on allowlist

Attack:
  1. bob calls:
       liquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "")

  2. LiquidityAdder calls pool.addLiquidity(alice, salt, deltas, abi.encode(KIND_PAY), "")
     Pool calls beforeAddLiquidity(liquidityAdder_address, alice, ...).
     Extension checks allowedDepositor[pool][alice] → true → no revert.

  3. Pool triggers metricOmmModifyLiquidityCallback; LiquidityAdder pulls tokens from bob (payer).
     alice receives the position shares.

  4. alice (bob's second wallet) calls pool.removeLiquidity(alice, salt, deltas, "")
     msg.sender == owner check passes; tokens returned to alice/bob.

Result: bob deposited into a pool he is not authorized to access.
        DepositAllowlistExtension did not revert.
```
