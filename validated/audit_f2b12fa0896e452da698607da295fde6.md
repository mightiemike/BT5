### Title
Sequencer Can Freely Set `feeX18` in `WithdrawCollateralV2` Outside the User-Signed EIP-712 Digest ŌĆö (File: `core/contracts/EndpointTx.sol`)

---

### Summary
`SignedWithdrawCollateralV2` carries a `feeX18` field that sits **outside** the `WithdrawCollateralV2` inner struct the user actually signs. Because `feeX18` is never committed to in the EIP-712 digest, the sequencer can freely set it to any value in `[0, currentFeeX18]` without invalidating the user's signature, allowing it to silently charge the maximum configured withdrawal fee on every `WithdrawCollateralV2` transaction.

---

### Finding Description

The two relevant structs are defined in `IEndpoint.sol`:

```solidity
struct WithdrawCollateralV2 {          // ŌåÉ user signs this
    bytes32 sender;
    uint32  productId;
    uint128 amount;
    uint64  nonce;
    address sendTo;
    uint128 appendix;
}

struct SignedWithdrawCollateralV2 {
    WithdrawCollateralV2 tx;
    CompactSignature     signature;
    int128               feeX18;       // ŌåÉ NOT in the signed struct
}
``` [1](#0-0) 

In `EndpointTx.processTransactionImpl`, after the signature is validated against the digest of `WithdrawCollateralV2 tx` only, the contract charges the fee using the sequencer-supplied `feeX18`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    signedTx.tx.sendTo == address(0)
);
int128 currentFeeX18 = spotEngine
    .getConfig(signedTx.tx.productId)
    .withdrawFeeX18;
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
chargeFee(
    signedTx.tx.sender,
    signedTx.feeX18,          // ŌåÉ sequencer-controlled, not signed
    signedTx.tx.productId
);
``` [2](#0-1) 

The only guard is `0 Ōēż feeX18 Ōēż currentFeeX18`. The user's signature covers `sender`, `productId`, `amount`, `nonce`, `sendTo`, and `appendix` ŌĆö but **not** `feeX18`. The sequencer can therefore pin `feeX18` at `currentFeeX18` for every user on every withdrawal, regardless of any fee discount the user expected.

This is structurally identical to the Hinkal `encryptedOutputs` flaw: a parameter that materially affects the user's asset outcome is passed alongside a signed transaction but is excluded from the integrity check, so the sequencer can substitute any value it chooses.

---

### Impact Explanation

The sequencer can charge every `WithdrawCollateralV2` caller the maximum configured withdrawal fee (`currentFeeX18`) even when the user expected a lower fee ŌĆö for example, after a fee reduction, or for users who negotiated a discounted rate. The user's on-chain balance is debited by `feeX18 ├Ś amount` more than they consented to, constituting a direct, irreversible loss of collateral. Because the signature check passes regardless of the `feeX18` value chosen, the user has no on-chain recourse. [3](#0-2) 

---

### Likelihood Explanation

The sequencer is the sole entity authorised to call `submitTransactions` and therefore the only party that can inject a `WithdrawCollateralV2` transaction. No external attacker is required; a compromised or malicious sequencer can silently maximise fees on every withdrawal. The attack requires no special on-chain state, no additional privileges, and leaves no distinguishable trace beyond the normal fee deduction event.

---

### Recommendation

Bind the fee to the user's signed intent by one of two approaches:

1. **Add `feeX18` to `WithdrawCollateralV2`** so it is covered by the EIP-712 digest. The user commits to an exact fee at signing time; any sequencer substitution invalidates the signature.
2. **Encode the fee in `appendix`** (already signed and reserved for forward-compatible withdrawal features per the inline comment), and derive `feeX18` exclusively from `appendix` inside the contract rather than accepting it as a separate unsigned field. [1](#0-0) 

---

### Proof of Concept

1. User signs `WithdrawCollateralV2 { sender, productId, amount=1 000 USDC, nonce, sendTo=0xŌĆ”, appendix=0 }` expecting a 0 % fee (fee was recently set to zero by governance).
2. Sequencer reads `currentFeeX18 = 50_000_000_000_000` (0.5 bps, the previous value still stored on-chain due to a delayed update), sets `feeX18 = currentFeeX18`, and submits the transaction.
3. `validateSignedTx` succeeds ŌĆö the digest covers only `WithdrawCollateralV2 tx`, which is unchanged.
4. `require(signedTx.feeX18 <= currentFeeX18)` passes.
5. `chargeFee(sender, 50_000_000_000_000, productId)` deducts the fee from the user's spot balance.
6. User receives `1 000 USDC ŌłÆ fee` instead of the full `1 000 USDC` they signed for, with no ability to detect or prevent the overcharge. [4](#0-3)
