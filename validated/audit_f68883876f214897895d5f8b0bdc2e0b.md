### Title
Unsafe Wrapping Multiplication in `op_unknown` Cost Scaling Bypasses Cost Metering - (File: `src/more_ops.rs`)

### Summary

`op_unknown` in `src/more_ops.rs` performs a raw `*=` multiplication on `u64` cost values at line 261 without any overflow guard. In Rust release builds, integer overflow wraps silently. A crafted unknown opcode with a maximum `cost_multiplier` and arguments sized to push the pre-multiplication cost to a specific multiple of `2^32` causes the product to wrap to zero, which then passes the subsequent `u32::MAX` guard and returns `Ok(Reduction(0, nil))` — charging zero cost for an arbitrarily expensive operation.

### Finding Description

In `op_unknown` (`src/more_ops.rs`), after computing a base cost via one of four cost functions, the code scales it by the opcode-embedded multiplier:

```rust
// line 260
check_cost(cost, max_cost)?;
// line 261 — raw multiplication, no overflow check
cost *= cost_multiplier + 1;
// line 262
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is decoded from the opcode bytes as a `u32` cast to `u64`, so `cost_multiplier + 1` reaches at most `2^32 = 4_294_967_296`. The `check_cost` call at line 260 only validates the pre-multiplication value; the multiplication itself is unchecked. In Rust release mode, `u64` overflow wraps modulo `2^64`.

**Concrete overflow path using `cost_function = 2` (mul-like cost):**

The cost_function=2 branch accumulates cost quadratically in atom sizes:

```
cost ≈ MUL_BASE_COST + MUL_COST_PER_OP
     + (l0 + l1) * MUL_LINEAR_COST_PER_BYTE
     + (l0 * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER
```

With `MUL_SQUARE_COST_PER_BYTE_DIVIDER = 128`, two atoms of approximately `s ≈ 1_048_576` bytes (1 MiB) each yield:

```
cost ≈ s² / 128 ≈ (2^20)² / 2^7 = 2^40 / 2^7 = 2^33 = 8_589_934_592
```

This value is within Chia's standard `max_cost` of `11_000_000_000`, so `check_cost` passes. Then:

```
cost *= cost_multiplier + 1
     = 2 × 2^32 × 2^32
     = 2^65
     ≡ 0  (mod 2^64)   ← wraps to zero
```

The guard `if cost > u32::MAX` evaluates `0 > 4_294_967_295` → false, so the function returns `Ok(Reduction(0, allocator.nil()))` — zero cost charged for a program that consumed significant resources.

Critically, the `op_unknown` cost_function=2 branch has **no per-operand size cap** (unlike `op_multiply`, which enforces `l1 > 256` → reject). The only size constraint comes from the allocator itself.

### Impact Explanation

The cost returned by `op_unknown` is added directly to the running `cost` accumulator in `run_program`. A wrapped-to-zero cost means:

1. **Cost metering bypass**: The program's true computational cost is not reflected in the charged cost. An attacker can include programs whose actual resource consumption far exceeds `max_cost` while the metered cost remains near zero.
2. **Consensus divergence risk**: Any future implementation that correctly computes the cost (e.g., a reference implementation or a patched node) would reject the same program that the vulnerable node accepts, causing a chain split.
3. **DoS vector**: Blocks can be filled with programs that appear cheap but are computationally expensive, degrading node performance without triggering cost limits.

The corrupted result is the `Cost` field of the returned `Reduction` — it is `0` instead of the true scaled cost (which would be `> u32::MAX` and should have been rejected as `EvalErr::Invalid`).

### Likelihood Explanation

The trigger requires:
- An unknown opcode with `cost_multiplier = u32::MAX` (first 4 opcode bytes = `0xFF 0xFF 0xFF 0xFF`) and `cost_function = 2` (bits 7–6 of last byte = `0b10`). This is fully attacker-controlled via the CLVM program bytes.
- Two atom arguments large enough to push the pre-multiplication cost to a multiple of `2^32`. With `cost_function = 2`, this requires atoms of ~1 MiB. Whether the allocator enforces a hard cap below this size determines practical reachability. The `op_unknown` cost_function=2 branch imposes no per-atom size limit of its own.
- The program must be executed in a context where `allow_unknown_ops()` returns true (lenient/consensus mode with softfork extensions), which is a standard Chia execution path.

The attacker controls all three inputs (opcode bytes, argument atoms, execution flags) via externally submitted CLVM bytes.

### Recommendation

Replace the raw multiplication at line 261 with a checked or saturating variant:

```rust
// Replace:
cost *= cost_multiplier + 1;

// With:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::CostExceeded)?;
```

This ensures that any overflow is treated as a cost-exceeded error rather than silently wrapping to a small value. Alternatively, validate that `cost <= u64::MAX / (cost_multiplier + 1)` before multiplying.
