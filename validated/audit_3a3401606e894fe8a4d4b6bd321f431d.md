### Title
Compromised Linked Signer Can Perpetuate Its Own Authorization via Fast-Mode `LinkSigner` Accepting `allowLinkedSigner = true` — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction path in `EndpointTx.sol` is validated with `allowLinkedSigner = true`, meaning the currently registered linked signer can authorize a new `LinkSigner` transaction — including one that re-links itself or links another attacker-controlled address. Because the only owner-exclusive revocation path is the slow-mode queue (3-day delay), a compromised linked signer can perpetuate its own authorization indefinitely and drain the subaccount's collateral before revocation takes effect.

---

### Finding Description

`EndpointTx.processTransactionImpl` handles the fast-mode `LinkSigner` transaction type at lines 576–590:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:], (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which in turn passes `getLinkedSigner(sender)` to `Verifier.validateSignature`:

```solidity
function validateSignature(...) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if it recovers to either the subaccount owner address **or** the linked signer address:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means the **currently registered linked signer** can sign and submit a `LinkSigner` transaction to overwrite the `linkedSigners[subaccount]` mapping — including re-linking itself or linking a fresh attacker-controlled address.

By contrast, the slow-mode `LinkSigner` path enforces that only the wallet owner (`msg.sender`) can submit the transaction:

```solidity
validateSender(txn.sender, sender);   // sender == msg.sender, must match wallet in subaccount bytes
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The slow-mode path enqueues the transaction with a hardcoded 3-day delay before execution:

```solidity
executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
``` [5](#0-4) 

During that 3-day window, the compromised linked signer can:

1. **Re-link itself** — submit a fast-mode `LinkSigner` (signed with the compromised key) pointing back to itself or to a fresh attacker address, consuming the next nonce and overwriting any pending state.
2. **Drain collateral** — `WithdrawCollateral` and `WithdrawCollateralV2` are also validated with `allowLinkedSigner = true`:

```solidity
validateSignedTx(
    signedTx.tx.sender, signedTx.tx.nonce,
    transaction, signedTx.signature,
    true   // ← allowLinkedSigner = true
);
``` [6](#0-5) 

The `linkedSigners` mapping is the sole on-chain state governing which external address may authorize high-value operations on a subaccount:

```solidity
mapping(bytes32 => address) internal linkedSigners;
``` [7](#0-6) 

---

### Impact Explanation

A compromised linked signer can:

- **Withdraw all collateral** from the victim subaccount via fast-mode `WithdrawCollateral` / `WithdrawCollateralV2` before the 3-day slow-mode revocation executes.
- **Prevent revocation** by continuously re-linking itself (or a fresh address it controls) via fast-mode `LinkSigner`, racing against the account owner's slow-mode revocation. Each re-link consumes a nonce and re-establishes attacker authority.
- **Affect isolated subaccounts** — `getLinkedSigner` for any isolated subaccount resolves to the parent's `linkedSigners` entry, so a single compromised linked signer can authorize operations across all isolated subaccounts of the parent. [8](#0-7) 

The corrupted state is `linkedSigners[subaccount]` and the resulting asset delta is the full collateral balance of the subaccount (and its isolated children).

---

### Likelihood Explanation

Linked signers are explicitly designed for hot-wallet or programmatic trading use cases — they are expected to be keys held in less-secure environments (bots, servers, browser wallets). Key compromise of a hot signer is a realistic, well-documented threat. The attacker needs only the compromised linked signer's private key and the ability to submit transactions through the sequencer — no privileged access, no governance capture, no admin keys required.

---

### Recommendation

Remove `allowLinkedSigner = true` from the fast-mode `LinkSigner` validation. Signer delegation is a credential-management operation that must be authorized exclusively by the subaccount owner (the wallet address embedded in the subaccount `bytes32`), not by the delegate itself. The corrected call should be:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the access control already enforced in the slow-mode `LinkSigner` path (`validateSender`), and eliminates the ability of a compromised linked signer to perpetuate its own authorization.

---

### Proof of Concept

1. Alice owns subaccount `S` and links hot-wallet `B` as her linked signer: `linkedSigners[S] = B`.
2. Attacker Mallory compromises `B`'s private key.
3. Alice detects the compromise and submits a slow-mode `LinkSigner` transaction (signed by her wallet, `msg.sender == Alice`) to set `linkedSigners[S] = address(0)`. This is queued with `executableAt = now + 3 days`.
4. Mallory, using `B`'s key, immediately constructs a fast-mode `LinkSigner` transaction: `{sender: S, signer: B, nonce: currentNonce}`, signs it with `B`, and submits it to the sequencer.
5. The sequencer processes Mallory's fast-mode `LinkSigner`. `validateSignedTx` passes because `recovered == B == getLinkedSigner(S)`. `linkedSigners[S]` is set back to `B` (or to a fresh Mallory-controlled address).
6. Mallory then submits a fast-mode `WithdrawCollateral` signed by `B`, draining Alice's collateral to an external address. This also passes because `allowLinkedSigner = true`.
7. Alice's slow-mode revocation executes after 3 days — but the collateral is already gone.

The root cause is exclusively in `EndpointTx.processTransactionImpl` at the `LinkSigner` branch, line 586 (`allowLinkedSigner = true`). [9](#0-8)
