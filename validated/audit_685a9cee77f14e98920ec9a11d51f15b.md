### Title
Spice Chunk Header (V6) Round-Trips Through `ChunkHeaderView` as V3, Producing Wrong `ChunkHash` and Invalid Signature — (`File: core/primitives/src/views.rs`)

### Summary

`From<ShardChunkHeader> for ChunkHeaderView` unconditionally emits `congestion_info: Some(CongestionInfo::default())` for a V6 spice chunk (which has no real `congestion_info` field). The reverse conversion `From<ChunkHeaderView> for ShardChunkHeader` then reconstructs the header as `ShardChunkHeaderInner::V3` instead of `V6`, producing different Borsh bytes, a different `ChunkHash`, and an invalid producer signature. This is the direct nearcore analog of the GMX "object created without `touch()`" bug: a struct is serialized with a fabricated field value that causes the deserialized version to carry the wrong version tag, making the hash/signature domain permanently inconsistent.

### Finding Description

**Serialization side** (`From<ShardChunkHeader> for ChunkHeaderView`, line 1160):

```rust
congestion_info: Some(inner.congestion_info().into()),
```

For `ShardChunkHeaderInner::V6`, `congestion_info()` returns `CongestionInfo::default()` — a fabricated zero value, not a real field of the struct:

```rust
Self::V6(_) => CongestionInfo::default(),   // line 219
```

`bandwidth_requests()` returns `None` for V6, and `has_proposed_split_field()` returns `false`. So the emitted `ChunkHeaderView` has `(proposed_split=None, bandwidth_requests=None, congestion_info=Some(...))`.

**Deserialization side** (`From<ChunkHeaderView> for ShardChunkHeader`, line 1176):

```rust
let inner = match (view.proposed_split, view.bandwidth_requests, view.congestion_info) {
    (None, None, Some(congestion_info)) => {
        ShardChunkHeaderInner::V3(ShardChunkHeaderInnerV3 { ... })  // line 1217
    }
    ...
};
```

The `(None, None, Some(...))` arm is hit, so the V6 chunk is reconstructed as `ShardChunkHeaderInner::V3` (Borsh discriminant `2` instead of `5`). The code even acknowledges this:

```
// TODO: store the version in ChunkHeaderView instead of guessing it
// from which fields are populated.
```

After reconstruction, `header.init()` recomputes the `ChunkHash` from the V3 Borsh bytes. The original signature was computed over the V6 hash. The two hashes differ because:
- V3 includes `prev_state_root`, `prev_outcome_root`, `prev_gas_used`, `gas_limit`, `prev_balance_burnt`, `prev_validator_proposals`, and `congestion_info` — none of which exist in V6.
- V6 omits all of those fields.

The resulting `ShardChunkHeader` carries a `ChunkHash` that does not match the producer's signature.

### Impact Explanation

Any consumer that round-trips a spice chunk header through `ChunkHeaderView` — including light-client verification, state-sync reconstruction, or the Rosetta RPC adapter — receives a header whose `ChunkHash` and signature are permanently inconsistent. Signature verification against the reconstructed header will always fail. A light client or external verifier that trusts the RPC-reconstructed header will either reject valid spice chunks or be unable to verify them at all, breaking the verifiability invariant for the entire spice epoch.

### Likelihood Explanation

The `ProtocolFeature::Spice` gate controls when V6 headers appear on-chain. Once spice activates, every chunk header produced is a V6 header. Every call to `From<ShardChunkHeader> for ChunkHeaderView` on a spice chunk triggers the fabrication, and every call to `From<ChunkHeaderView> for ShardChunkHeader` on the resulting view triggers the wrong-version reconstruction. Both conversions are in the public RPC and Rosetta code paths.

### Recommendation

1. Add a `is_spice_chunk` flag (or store the inner version number) to `ChunkHeaderView` so the reverse conversion can reconstruct the correct variant without guessing.
2. In `From<ShardChunkHeader> for ChunkHeaderView`, set `congestion_info: None` (or a dedicated `Option<CongestionInfoView>` with a spice-aware sentinel) when `inner.is_spice_chunk()` is true, so the fabricated default is never emitted.
3. Add a round-trip test asserting that `ShardChunkHeader::V3(ShardChunkHeaderV3::new_for_spice(...))` survives `ChunkHeaderView` serialization and deserialization with the same `ChunkHash` and valid signature.

### Proof of Concept

```
// 1. Produce a V6 spice chunk header
let header_v6 = ShardChunkHeader::V3(ShardChunkHeaderV3::new_for_spice(
    prev_block_hash, merkle_root, encoded_length, height, shard_id,
    receipts_root, tx_root, &signer,
));
let original_hash = *header_v6.chunk_hash();

// 2. Serialize to ChunkHeaderView (public RPC path)
let view: ChunkHeaderView = header_v6.into();
// view.congestion_info == Some(CongestionInfo::default())  ← fabricated
// view.bandwidth_requests == None
// view.proposed_split == None

// 3. Deserialize back (Rosetta / light-client path)
let reconstructed: ShardChunkHeader = view.into();
// reconstructed inner is ShardChunkHeaderInner::V3, NOT V6

let reconstructed_hash = *reconstructed.chunk_hash();
assert_ne!(original_hash, reconstructed_hash);  // FAILS: hashes differ

// 4. Signature verification fails
let producer_pubkey = signer.public_key();
assert!(reconstructed.verify_signature(&producer_pubkey));  // FAILS
```

**Exact divergent Borsh bytes:** the original header serializes with inner discriminant byte `0x05` (V6); the reconstructed header serializes with inner discriminant byte `0x02` (V3), plus additional fields (`prev_state_root`, `prev_outcome_root`, `prev_gas_used`, `gas_limit`, `prev_balance_burnt`, `prev_validator_proposals`) that are absent in V6 but present in V3, all set to zero/empty defaults from the view fields.
