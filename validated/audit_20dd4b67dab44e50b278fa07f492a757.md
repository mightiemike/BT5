Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of actual user, allowing allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router`, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actualUser]`. Any non-allowlisted user can bypass the per-user swap gate by calling the router if the router address is itself allowlisted.

## Finding Description
`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and uses it as the identity to gate:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (the extension's caller), and `sender` is whatever the pool forwarded. The pool always forwards its own `msg.sender` as `sender`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool therefore sees `msg.sender = router`, and the extension checks `allowedSwapper[pool][router]`. The actual user (`msg.sender` of `exactInputSingle`) is stored only in transient callback context via `_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)` at line 71, but is never forwarded to the extension.

**Exploit path:**
1. Pool admin deploys a curated pool with `SwapAllowlistExtension`, allowlists Alice: `setAllowedToSwap(pool, alice, true)`.
2. Admin also allowlists the router so Alice can use the standard periphery: `setAllowedToSwap(pool, router, true)`.
3. Mallory (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` with `msg.sender = router`.
5. Pool calls `extension.beforeSwap(sender=router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → passes.
7. Mallory's swap executes on the curated pool.

No existing guard prevents this. The extension has no mechanism to distinguish between different users arriving through the same router address.

## Impact Explanation
`SwapAllowlistExtension` is the sole mechanism for curated pools to restrict trading to approved counterparties. Once the router is allowlisted — a necessary step for approved users to access the standard periphery — the curation boundary collapses entirely: any address can trade on the pool. Pools configured for restricted audiences (e.g., priced or structured for specific counterparties) are exposed to unauthorized swap access, enabling direct extraction of value from pools whose pricing or configuration assumed a restricted trading set. This constitutes broken core pool functionality causing loss of funds.

## Likelihood Explanation
The scenario is realistic and requires only normal admin behavior. A pool admin who wants approved users to use the standard router will naturally allowlist the router — the periphery is the canonical swap entry point. The admin has no indication that allowlisting the router grants universal access. The trigger requires one admin configuration step (allowlisting the router) followed by any non-allowlisted user calling the router, both of which are expected actions.

## Recommendation
The extension must check the actual economic actor, not the intermediary. The correct fix is to have the router encode `msg.sender` (the actual user) into `extensionData` and have the extension decode and verify it:

```solidity
// In MetricOmmSimpleRouter.exactInputSingle, replace "" with abi.encode(msg.sender)
// In SwapAllowlistExtension.beforeSwap, decode extensionData to get the real user
address realUser = abi.decode(extensionData, (address));
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][realUser]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

The router already stores the real payer in transient context at line 71 of `MetricOmmSimpleRouter.sol`; the same identity should be forwarded in `extensionData`.

## Proof of Concept
```
Setup:
  pool = curated pool with SwapAllowlistExtension
  alice = allowlisted user
  mallory = non-allowlisted user
  router = MetricOmmSimpleRouter

Admin actions:
  extension.setAllowedToSwap(pool, alice, true)
  extension.setAllowedToSwap(pool, router, true)  // to let alice use the router

Attack:
  mallory calls router.exactInputSingle({pool: pool, recipient: mallory, ...})
  → router calls pool.swap(...) [MetricOmmSimpleRouter.sol L72]
  → pool calls _beforeSwap(msg.sender=router, ...) [MetricOmmPool.sol L230]
  → extension checks allowedSwapper[pool][router] == true → PASSES [SwapAllowlistExtension.sol L37]
  → mallory's swap executes on the curated pool

Result:
  mallory, not in the allowlist, successfully swaps on the curated pool.
