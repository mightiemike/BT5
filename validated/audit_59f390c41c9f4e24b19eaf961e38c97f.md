### Title
No-Maximum-Bound on `Tier.ExitDuration` Allows Governance to Permanently Lock User Tokens - (File: `x/tieredrewards/types/tier.go`)

### Summary
`Tier.Validate()` enforces only a lower bound (`ExitDuration > 0`) on the exit-commitment duration but imposes no upper bound. A governance proposal (`MsgAddTier` or `MsgUpdateTier`) that sets `ExitDuration` to an astronomically large value (e.g. `math.MaxInt64` nanoseconds ≈ 292 years) passes validation and is stored. Any user who subsequently calls `MsgTriggerExitFromTier` on a position in that tier receives an `ExitUnlockAt` timestamp centuries in the future, blocking every exit path (`MsgTierUndelegate`, `MsgWithdrawFromTier`, `MsgExitTierWithDelegation`) indefinitely. If the tier is simultaneously or subsequently marked `CloseOnly`, `MsgClearPosition` is also blocked, making the lock permanent with no recovery path.

---

### Finding Description

`Tier.Validate()` in `x/tieredrewards/types/tier.go` checks:

```go
if t.ExitDuration <= 0 {
    return fmt.Errorf("exit duration must be positive")
}
```

There is no upper-bound check. [1](#0-0) 

`SetTier` in `x/tieredrewards/keeper/tier.go` calls `tier.Validate()` before writing to the store — this is the only validation gate. [2](#0-1) 

Both `AddTier` and `UpdateTier` in `msg_server_auth.go` reach `SetTier` after only checking authority and tier existence — no additional duration check is performed. [3](#0-2) 

When `MsgTriggerExitFromTier` is processed, the handler reads the live `tier.ExitDuration` and calls `pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)`. [4](#0-3) 

`TriggerExit` sets `ExitUnlockAt = blockTime.Add(duration)` — if `duration` is `math.MaxInt64` nanoseconds, `ExitUnlockAt` overflows or resolves to year ~2262. [5](#0-4) 

All three exit paths (`MsgTierUndelegate`, `MsgWithdrawFromTier`, `MsgExitTierWithDelegation`) gate on `CompletedExitLockDuration(blockTime)`, which requires `block_time >= ExitUnlockAt`. [6](#0-5) 

`MsgClearPosition` (the only cancel path) is blocked when the tier is `CloseOnly`. Per the ADR, `CloseOnly` blocks: `LockTier`, `CommitDelegation`, `AddToPosition`, `TierRedelegate`, **`ClearPosition`**. [7](#0-6) 

---

### Impact Explanation

**High.** Any user who calls `MsgTriggerExitFromTier` on a position in the misconfigured tier has their `ExitUnlockAt` set to a timestamp centuries in the future. All three exit paths are gated on that timestamp. If the tier is also `CloseOnly` (set in the same proposal or a subsequent one), `MsgClearPosition` is also blocked, making the lock permanent. The corrupted value is `Position.ExitUnlockAt`, which gates every exit path for the position's principal (delegated tokens held by the per-position delegator address). This constitutes a permanent loss of access to user funds.

Even without `CloseOnly`, a user who cancels via `MsgClearPosition` can never successfully exit the tier (re-triggering exit produces the same unbounded `ExitUnlockAt`), so their tokens remain permanently locked in the module with no viable exit.

---

### Likelihood Explanation

**Low.** Requires a governance proposal to pass with an erroneously large `ExitDuration` — a fat-finger (e.g. specifying `"87600000h"` instead of `"8760h"`) or a deliberate misconfiguration error. Governance requires a quorum of validators/delegators to approve, which provides a social check. However, duration strings are easy to mistype by several orders of magnitude, and the protocol provides no guardrail. This matches the external report's characterization: low likelihood, high impact.

---

### Recommendation

Add a maximum bound for `ExitDuration` in `Tier.Validate()` in `x/tieredrewards/types/tier.go`. A reasonable upper bound (e.g. 10 years = `10 * 365 * 24 * time.Hour`) should be enforced:

```go
const MaxExitDuration = 10 * 365 * 24 * time.Hour

if t.ExitDuration > MaxExitDuration {
    return fmt.Errorf("exit duration must not exceed %s: got %s", MaxExitDuration, t.ExitDuration)
}
```

This mirrors the existing upper-bound pattern already applied to `BonusApy`. [8](#0-7) 

---

### Proof of Concept

1. Submit a governance proposal `MsgAddTier` with:
   ```json
   {
     "id": 99,
     "exit_duration": "9223372036854775807ns",
     "bonus_apy": "0.04",
     "min_lock_amount": "1000000",
     "close_only": false
   }
   ```
2. Proposal passes governance vote. `AddTier` → `SetTier` → `tier.Validate()` passes (duration > 0). Tier 99 is stored. [9](#0-8) 
3. User calls `MsgLockTier` with `tier_id=99`. Position is created normally.
4. User calls `MsgTriggerExitFromTier` on their position. `TriggerExit(blockTime, math.MaxInt64 ns)` sets `ExitUnlockAt` to year ~2262 (or overflows). [5](#0-4) 
5. User attempts `MsgTierUndelegate` / `MsgExitTierWithDelegation` / `MsgWithdrawFromTier` — all fail because `CompletedExitLockDuration` returns false. [6](#0-5) 
6. Governance passes `MsgUpdateTier` setting `close_only=true` for tier 99. `MsgClearPosition` is now also blocked. User's tokens are permanently locked with no exit path. [7](#0-6)

### CloseOnly Tiers

When a tier is marked `CloseOnly` by governance, these messages are **blocked**: LockTier, CommitDelegation, AddToPosition, TierRedelegate, ClearPosition. **Allowed**: TriggerExit, TierUndelegate, WithdrawFromTier, ClaimTierRewards, ExitTierWithDelegation. This lets governance sunset a tier while existing positions close out.
