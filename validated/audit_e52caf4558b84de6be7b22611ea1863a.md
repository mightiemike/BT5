### Title
Accrued Bonus Rewards Permanently Lost When Bonus Pool Is Insufficient During Redelegation Slash — (`File: x/tieredrewards/keeper/slash.go`)

### Summary

In `x/tieredrewards/keeper/slash.go`, the `slashRedelegationPosition` function deliberately swallows `ErrInsufficientBonusPool` from `processEventsAndClaimBonus` to prevent a chain halt. However, `processEventsAndClaimBonus` has already mutated the position's reward checkpoints in memory (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) and decremented validator event reference counts in the store before returning the error. The caller then persists the mutated position via `setPositionWithState`, permanently advancing the checkpoints without ever paying the bonus. The accrued rewards are irrecoverably lost.

### Finding Description

`slashRedelegationPosition` is invoked by the `BeforeRedelegationSlashed` staking hook whenever a redelegation entry is slashed. It calls `processEventsAndClaimBonus` to settle bonus rewards against pre-slash shares:

```go
// x/tieredrewards/keeper/slash.go:54-64
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    // Deliberately forgo bonus rewards if pool is insufficient to prevent chain halt.
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
    } else {
        return err
    }
}
```

Inside `processEventsAndClaimBonus`, before the pool-balance check, the function:

1. Walks all pending validator events, calling `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` for each — a direct store write that is not rolled back.
2. Mutates `pos.LastEventSeq` in memory via `pos.UpdateLastEventSeq(entry.Seq)`.
3. Calls `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` — advancing `pos.LastBonusAccrual` to the current block time in memory.
4. Calls `pos.UpdateLastKnownBonded(bonded)` in memory.

Only after all of the above does it check the pool:

```go
// x/tieredrewards/keeper/claim_rewards.go:230-232
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err  // returns ErrInsufficientBonusPool here
}
```

Back in `slashRedelegationPosition`, the error is swallowed and execution continues with the already-mutated `pos`:

```go
// x/tieredrewards/keeper/slash.go:76-77
pos.Delegation.Shares = pos.Delegation.Shares.Sub(sharesToUnbond)
return k.setPositionWithState(ctx, pos, nil)
```

`setPositionWithState` persists the mutated position — with `LastBonusAccrual` advanced to the slash block time and `LastEventSeq` advanced past all pending events — even though no bonus was paid. The accrual window is permanently closed.

### Impact Explanation

A position owner loses all bonus rewards that accrued between the previous `LastBonusAccrual` checkpoint and the slash block time. The corrupted values are:

- `pos.LastBonusAccrual` — advanced to `blockTime`, closing the accrual window permanently.
- `pos.LastEventSeq` — advanced past all pending events; those events' reference counts are also decremented, potentially garbage-collecting them.
- `pos.LastKnownBonded` — updated to reflect post-event bonded state.

On the next `ClaimTierRewards` or any reward-settling message, `processEventsAndClaimBonus` will start from the new (advanced) `LastBonusAccrual` and `LastEventSeq`, finding no pending accrual. The bonus for the lost period is unrecoverable. The magnitude equals `shares × tokensPerShare × bonusApy × duration / SecondsPerYear` for the gap period, which can be substantial for large positions or long gaps.

### Likelihood Explanation

The trigger requires two concurrent conditions:

1. A position is in an active redelegation (21-day unbonding window after `MsgTierRedelegate`).
2. The source validator is slashed during that window, causing `BeforeRedelegationSlashed` to fire.

Validator slashes (downtime or double-signing) are normal protocol events. The bonus pool can be depleted by concurrent bonus claims from many positions, or by the `BeginBlocker` top-up mechanism draining it for base rewards. The pool is a shared resource with no per-position reservation. A large validator slash affecting many positions simultaneously could drain the pool and trigger this path for multiple positions at once. This is a realistic scenario on a live chain.

### Recommendation

The fix must ensure that when `ErrInsufficientBonusPool` is encountered inside `processEventsAndClaimBonus` during a hook path, the position's checkpoints are **not** advanced and the event reference counts are **not** decremented. Two approaches:

1. **Preferred**: Use a cached/branched context for the `processEventsAndClaimBonus` call inside `slashRedelegationPosition`. If the pool is insufficient, discard the cache (rolling back the ref-count decrements) and do not persist the mutated `pos` checkpoints. Only advance checkpoints when the bonus is actually paid.

2. **Alternative**: Move the `sufficientBonusPoolBalance` check to before the event-processing loop and checkpoint mutations, so that if the pool is insufficient, no state is mutated at all.

The ADR-006 security invariant "Pool balance is never exceeded. User paths fail atomically; the `BeforeRedelegationSlashed` hook forfeits bonus silently if the pool is short, to avoid chain halt" should be updated to clarify that forfeiting bonus must not advance checkpoints.

### Proof of Concept

1. Alice creates a tier position with a large delegation on validator V1.
2. Alice calls `MsgTierRedelegate` to move to validator V2. A redelegation entry is created with an `unbondingId`.
3. The bonus pool is drained to near-zero by other claimants or the BeginBlocker.
4. V1 is slashed (downtime). The staking module calls `SlashRedelegation`, which fires `BeforeRedelegationSlashed(ctx, unbondingId, sharesToUnbond)`.
5. `slashRedelegationPosition` is called. It calls `processEventsAndClaimBonus(&pos)`.
6. Inside `processEventsAndClaimBonus`: event ref counts are decremented in the store; `pos.LastBonusAccrual` is advanced to the slash block time; `pos.LastEventSeq` is advanced. Then `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`.
7. `slashRedelegationPosition` logs the error and continues. It calls `setPositionWithState(ctx, pos, nil)`, persisting the advanced checkpoints.
8. Alice later calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` starts from the new `LastBonusAccrual` (slash block time) and finds no accrual since then. Alice receives zero bonus for the entire period from her original `LastBonusAccrual` to the slash block time. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
