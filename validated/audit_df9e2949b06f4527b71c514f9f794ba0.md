### Title
`Endpoint.sol` Uses Deprecated `draft-EIP712Upgradeable` While Current `EIP712Upgradeable` Exists ‚Äî (`File: core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.sol` imports the deprecated `draft-EIP712Upgradeable` from OpenZeppelin, while `ContractOwner.sol` in the same codebase already imports the finalized, non-draft `EIP712Upgradeable`. The draft version caches the EIP-712 domain separator at initialization time and never recomputes it if the chain ID changes. This creates a stale domain separator in the protocol's primary transaction entry point, enabling cross-chain signature replay of all EIP-712-signed user transactions if Ink Chain forks or the contract is deployed on a second chain.

---

### Finding Description

`Endpoint.sol` line 6 imports:

```solidity
import "@openzeppelin/contracts-upgradeable/utils/cryptography/draft-EIP712Upgradeable.sol";
``` [1](#0-0) 

`ContractOwner.sol` line 4 imports the current, non-deprecated version:

```solidity
import "@openzeppelin/contracts-upgradeable/utils/cryptography/EIP712Upgradeable.sol";
``` [2](#0-1) 

In OpenZeppelin v4.8.0, `draft-EIP712Upgradeable` was finalized and the `draft-` prefix was removed. The draft version's `_domainSeparatorV4()` caches the domain separator once at `__EIP712_init` time and never recomputes it. The non-draft version added a guard: if `block.chainid` differs from the cached chain ID at initialization, it recomputes the separator dynamically.

`Endpoint.sol` calls `__EIP712_init("Nado", "0.0.1")` during initialization, locking in the domain separator with the chain ID at deployment time. [3](#0-2) 

All EIP-712 digest construction for user-facing operations ‚Äî including `submitTransactionsChecked`, slow-mode transactions, and linked-signer operations ‚Äî flows through this stale domain separator.

The codebase itself demonstrates awareness of the non-draft version: `ContractOwner.sol` already uses `EIP712Upgradeable` (non-draft), making the continued use of the draft version in `Endpoint.sol` an inconsistency directly analogous to M-02's use of a deprecated contract when the correct version was already known.

---

### Impact Explanation

If Ink Chain undergoes a hard fork that changes the chain ID, or if the `Endpoint` contract is deployed on a second EVM-compatible chain (e.g., for expansion), the domain separator stored in `Endpoint.sol` remains anchored to the original chain ID. An attacker can take any EIP-712-signed transaction (deposit, withdrawal, linked-signer assignment, slow-mode submission) that was valid on chain A and replay it verbatim on chain B, because both chains share the same stale domain separator. This corrupts subaccount balances, signer state, and collateral ownership on the replayed chain without any new authorization from the user.

---

### Likelihood Explanation

Ink Chain is a live EVM-compatible network. Hard forks that change chain IDs are uncommon but not unprecedented (cf. Ethereum Classic split). Multi-chain deployment of a DEX is a standard growth path. The inconsistency between `Endpoint.sol` (draft) and `ContractOwner.sol` (non-draft) in the same repo confirms the team has already migrated some contracts and overlooked `Endpoint.sol`. The entry path requires no privileged access: any user transaction signed under the original chain ID is the replay payload.

---

### Recommendation

Replace the deprecated import in `Endpoint.sol`:

```solidity
// Remove:
import "@openzeppelin/contracts-upgradeable/utils/cryptography/draft-EIP712Upgradeable.sol";

// Replace with:
import "@openzeppelin/contracts-upgradeable/utils/cryptography/EIP712Upgradeable.sol";
```

This aligns `Endpoint.sol` with `ContractOwner.sol`, which already uses the finalized version, and ensures the domain separator is recomputed dynamically whenever `block.chainid` differs from the cached value.

---

### Proof of Concept

1. Deploy `Endpoint.sol` on Ink Chain (chain ID = X). Domain separator is cached with chain ID X.
2. Ink Chain forks; the fork chain has chain ID Y. `Endpoint.sol` on the fork still returns domain separator computed with chain ID X (draft version never recomputes).
3. A user signs a `DepositCollateral` or `LinkSigner` transaction on chain X. The EIP-712 digest uses domain separator with chain ID X.
4. An attacker submits the identical signed transaction on chain Y via `submitSlowModeTransaction` or `submitTransactionsChecked`. Because the domain separator on chain Y is also anchored to chain ID X (stale), the signature verifies successfully.
5. The attacker replays the user's transaction on chain Y without the user's knowledge, corrupting the user's subaccount state on the fork chain.

The root cause ‚Äî `draft-EIP712Upgradeable` in `Endpoint.sol` ‚Äî is the necessary vulnerable step; the non-draft version used in `ContractOwner.sol` would have prevented this by recomputing the separator with the current `block.chainid`. [1](#0-0) [2](#0-1) [4](#0-3)
