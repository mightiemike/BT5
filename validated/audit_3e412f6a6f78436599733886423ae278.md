### Title
Fast Withdrawal Transfers Tokens Without Balance Validation or SpotEngine State Update, Enabling Pool Drainage — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`submitFastWithdrawal` in `BaseWithdrawPool.sol` is a public function that transfers tokens from the `WithdrawPool` to a recipient without (1) checking the subaccount's current balance in `SpotEngine`, and (2) updating the subaccount's balance in `SpotEngine` after the transfer. The balance decrement and health check are deferred entirely to a subsequent sequencer-submitted transaction via `clearinghouse.withdrawCollateral`. If that deferred transaction reverts — which it will whenever the subaccount's health has deteriorated between sequencer signing and execution — the pool is permanently drained of the transferred amount while the subaccount balance in `SpotEngine` is never decremented.

---

### Finding Description

`submitFastWithdrawal` (lines 81–114 of `BaseWithdrawPool.sol`) is callable by any unprivileged address. Its execution path is:

1. Validates that `idx` has not been used and is above `minIdx`.
2. Verifies sequencer signatures via `v.requireValidTxSignatures`.
3. Decodes the withdrawal amount and recipient from the signed transaction.
4. Calls `handleWithdrawTransfer` — which calls `token.safeTransfer(sendTo, amount)` — transferring real ERC-20 tokens out of the pool. [1](#0-0) 

Critically, **no call to `spotEngine.updateBalance` or `clearinghouse.withdrawCollateral` is made here**. The subaccount's balance in `SpotEngine` is left unchanged. The design intent is that the sequencer will later submit the same withdrawal through the normal path (`EndpointTx` → `clearinghouse.withdrawCollateral`), which will:

- Skip the token transfer (because `markedIdxs[idx]` is already `true`),
- Call `spotEngine.updateBalance(productId, sender, amountRealized)` to decrement the balance,
- Call `getHealth(sender, healthType) >= 0` to enforce the health invariant. [2](#0-1) 

The broken invariant: **if the sequencer's deferred transaction reverts** (because the health check at line 419 fails), `markedIdxs[idx]` remains `true`, `submitWithdrawal` returns early on any retry, and `spotEngine.updateBalance` is never applied. The pool has already paid out the tokens; the subaccount balance is never decremented. [3](#0-2) 

The normal `withdrawCollateral` path enforces the health check atomically with the balance update and token transfer — if health fails, the entire transaction reverts including the transfer. The fast withdrawal path breaks this atomicity by separating the transfer (immediate, in `submitFastWithdrawal`) from the balance update and health check (deferred, in the sequencer's subsequent transaction). [4](#0-3) 

---

### Impact Explanation

- The `WithdrawPool` contract loses ERC-20 tokens permanently with no on-chain recourse.
- The subaccount's balance in `SpotEngine` is never decremented, corrupting the protocol's accounting invariant (`totalDeposits >= totalBorrows` enforced by `assertUtilization`).
- Other users' deposited collateral held in the pool is at risk of drainage.
- The corrupted `SpotEngine` balance allows the subaccount to continue trading, borrowing, or opening positions as if the withdrawal never occurred.

---

### Likelihood Explanation

The attack requires a valid sequencer-signed withdrawal transaction. However, the sequencer signs based on the subaccount's state at signing time. In a volatile market, the following realistic sequence is sufficient:

1. Subaccount holds 100 USDC; sequencer signs a withdrawal for 100 USDC.
2. Subaccount opens a leveraged perp position; market moves adversely, reducing net health below zero.
3. Any caller invokes `submitFastWithdrawal` with the previously signed transaction — 100 USDC leaves the pool.
4. Sequencer submits the deferred `withdrawCollateral` transaction; `getHealth` returns negative; transaction reverts.
5. `markedIdxs[idx]` is `true`; no further token transfer or balance update is possible.

The subaccount has received 100 USDC and retains its full (now-stale) `SpotEngine` balance. The pool is short 100 USDC. This scenario does not require sequencer compromise — only a signed withdrawal and subsequent balance deterioration, which is a normal market condition.

---

### Recommendation

- Perform the `spotEngine.updateBalance` and health check **inside `submitFastWithdrawal`** before or atomically with the token transfer, mirroring the logic in `withdrawCollateral`.
- Alternatively, have `submitFastWithdrawal` call `clearinghouse.withdrawCollateral` directly (with a flag indicating the LP is fronting the transfer), so the balance decrement and health check are enforced at fast-withdrawal time rather than deferred.
- At minimum, add a pre-transfer check that the subaccount's current `SpotEngine` balance is sufficient to cover the withdrawal amount.

---

### Proof of Concept

```
1. Alice has spotEngine balance = 100 USDC (productId = QUOTE).
2. Sequencer signs WithdrawCollateral: sender=Alice, amount=100 USDC, idx=42.
3. Alice opens a 10x leveraged BTC-PERP long; BTC drops 15%; Alice's net health < 0.
4. Bob (unprivileged) calls WithdrawPool.submitFastWithdrawal(42, signedTx, seqSigs).
   - markedIdxs[42] = true
   - pool.safeTransfer(Alice, 100 USDC)  ← tokens leave pool
   - spotEngine balance: unchanged (still 100 USDC)
5. Sequencer submits the same withdrawal via Endpoint → Clearinghouse.withdrawCollateral.
   - submitWithdrawal: markedIdxs[42] == true → return (no token transfer)
   - spotEngine.updateBalance(QUOTE, Alice, -100e18)
   - getHealth(Alice, INITIAL) → negative → REVERT
6. Transaction in step 5 reverts entirely.
   - spotEngine balance for Alice: still 100 USDC (never decremented)
   - Pool: short 100 USDC permanently
   - Alice: received 100 USDC + retains full SpotEngine balance
``` [5](#0-4) [2](#0-1)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function submitWithdrawal(
        IERC20Base token,
        address sendTo,
        uint128 amount,
        uint64 idx
    ) public {
        require(msg.sender == clearinghouse);

        if (markedIdxs[idx]) {
            return;
        }
        markedIdxs[idx] = true;
        // set minIdx to most recent withdrawal submitted by sequencer
        minIdx = idx;

        handleWithdrawTransfer(token, sendTo, amount);
    }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
