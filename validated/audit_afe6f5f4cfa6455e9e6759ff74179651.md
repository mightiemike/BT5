### Title
Permanent Bonus Loss via `LastBonusAccrual` Checkpoint Advance Before Pool Check in `slashRedelegationPosition` — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary
`processEventsAndClaimBonus` advances `pos.LastBonusAccrual` to `blockTime` (and advances `LastEventSeq`) **before** checking pool sufficiency. When called from `slashRedelegationPosition`, the `ErrInsufficientBonusPool` error is silently swallowed and the mutated position is persisted. The accrued bonus for the pre-slash period is permanently forfeited and irrecoverable after pool replenishment.

The `pos` pointer passed to `processEventsAndClaimBonus` is mutated at line 215 before the pool check at line 230. When the pool is insufficient, the error is returned but `pos.LastBonusAccrual` has already been advanced. `slashRedelegationPosition` swallows the error and persists the mutated position. Additionally, inside the event loop at line 193/196, `pos.LastEventSeq` is advanced and event reference counts are decremented — these side effects also survive the error path.

### Impact Explanation
For any **partial** redelegation slash where the bonus pool is insufficient at slash time:
- The position owner's accrued bonus for `[old_LastBonusAccrual, slash_blockTime]` is permanently lost.
- After pool replenishment, `ClaimTierRewards` returns zero bonus for that period because `LastBonusAccrual` already points past it.
- This is a direct, irreversible fund loss proportional to the position size, APY, and the duration since the last claim.

### Likelihood Explanation
The pool being insufficient at slash time is a realistic production scenario:
1. The pool is funded by governance and drained by legitimate bonus claims.
2. The BeginBlocker top-up mechanism also draws from the pool for base reward shortfalls.
3. A redelegation slash can fire at any block via the staking module's `SlashRedelegation` path, which is triggered by evidence submission (double-sign) — a standard Cosmos SDK transaction path.

No privileged access or social engineering is required. The bug fires whenever the pool balance is below the owed bonus at the moment a `BeforeRedelegationSlashed` hook fires for a tier position.

### Recommendation
Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the successful `SendCoinsFromModuleToAccount` call, so checkpoints are only advanced when the bonus is actually paid. Alternatively, in `slashRedelegationPosition`, re-read the position from the store after a swallowed `ErrInsufficientBonusPool` error (discarding the in-memory mutations) before calling `setPositionWithState`.
