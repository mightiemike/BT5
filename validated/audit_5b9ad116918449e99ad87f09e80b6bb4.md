Audit Report

## Title
`SwapAllowlistExtension` Allowlist Bypass via `MetricOmmSimpleRouter`: Any Caller Can Swap in Restricted Pools - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender`. When `MetricOmmSimpleRouter` is used, the pool sees `msg.sender = router`, so the extension checks whether the **router** is allowlisted rather than the actual end user. Any pool admin who allowlists the router (required for allowlisted users to route through it) silently opens the allowlist to every caller of the public router, completely defeating the access control.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` verbatim as `sender` to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever address called `pool.swap()`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no mechanism to forward the original caller — the pool therefore sees `msg.sender = router`: [3](#0-2) 

The same identity-collapse occurs in `exactInput` (all hops), `exactOutputSingle`, and `exactOutput`: [4](#0-3) [5](#0-4) 

There are no existing guards that recover the original caller. The `extensionData` field is passed through but `SwapAllowlistExtension.beforeSwap` ignores it entirely (the `bytes calldata` parameter is unnamed and unused). [6](#0-5) 

## Impact Explanation
This is an admin-boundary break. `SwapAllowlistExtension` is the sole on-chain mechanism for restricting which addresses may trade in a pool (e.g., KYC-gated, institutional, or counterparty-limited pools). When the router is allowlisted — the necessary configuration for any allowlisted user to use the router — the guard becomes completely ineffective: any unprivileged address can call `router.exactInputSingle()` and execute a swap in a pool that was intended to be restricted. Unauthorized users gain full swap access, directly violating the admin-configured access boundary.

## Likelihood Explanation
The bypass is trivially reachable with a single transaction. `MetricOmmSimpleRouter` is a public, permissionless contract requiring no special privileges. The precondition — router being allowlisted — is the natural and expected configuration for any pool that wants allowlisted users to be able to use the router. No flash loans, multi-step setup, or special knowledge is required.

## Recommendation
Pass the original caller through the swap path so the extension can gate on the economically relevant actor. Two concrete options:

1. **Extend the pool's `swap` interface** to accept an explicit `swapper` address (validated against `msg.sender` or a trusted forwarder list), and pass that address to `_beforeSwap` instead of raw `msg.sender`.

2. **Have the router encode the real caller in `extensionData`** and update `SwapAllowlistExtension.beforeSwap` to decode and check the real caller when the immediate sender is a known, trusted router address.

Until fixed, pool admins must **not** allowlist the router address on pools that use `SwapAllowlistExtension` with a restricted swapper set.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension (allowAllSwappers[pool] = false)
  - Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  1. attacker calls router.exactInputSingle({pool: pool, ...})
  2. router calls pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)
  3. pool calls _beforeSwap(msg.sender=router, ...)
  4. SwapAllowlistExtension.beforeSwap receives sender=router
  5. allowedSwapper[pool][router] == true  →  check passes
  6. Swap executes for attacker despite attacker not being allowlisted

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
```
