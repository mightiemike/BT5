### Title
Slow Mode Delay Enforcement Broken by zkSync Frozen `block.timestamp`, Locking User Funds — (`Endpoint.sol`, `EndpointTx.sol`)

---

### Summary

Nado's slow mode mechanism enforces a 3-day delay using `block.timestamp` before users can self-execute queued transactions. On zkSync, `block.timestamp` can be frozen (identical across consecutive blocks), causing the `executableAt <= block.timestamp` guard to remain permanently unsatisfiable during frozen periods. This breaks the user's only sequencer-bypass path and locks deposited collateral.

---

### Finding Description

Two locations stamp slow mode entries with `uint64(block.timestamp) + SLOW_MODE_TX_DELAY`:

`Endpoint.sol` line 153 (inside `depositCollateralWithReferral`): [1](#0-0) 

`EndpointTx.sol` line 377 (inside `submitSlowModeTransactionImpl`): [2](#0-1) 

`SLOW_MODE_TX_DELAY` is defined as exactly 3 days (259 200 seconds): [3](#0-2) 

The user-callable execution path enforces the timestamp guard:

```solidity
require(
    fromSequencer || (txn.executableAt <= block.timestamp),
    ERR_SLOW_TX_TOO_RECENT
);
``` [4](#0-3) 

`executeSlowModeTransaction()` always passes `fromSequencer = false`, so the timestamp check is mandatory for every user-initiated call: [5](#0-4) 

On zkSync, `block.timestamp` is not guaranteed to advance between blocks — consecutive blocks may carry the same timestamp. If a user submits a slow mode transaction at timestamp `T`, `executableAt` is set to `T + 259200`. If `block.timestamp` is frozen at `T` (or advances slower than wall-clock time), the condition `T + 259200 <= block.timestamp` cannot be satisfied, and every call to `executeSlowModeTransaction()` reverts with `ERR_SLOW_TX_TOO_RECENT`.

The sequencer path (`processTransaction` → `_executeSlowModeTransaction` with `fromSequencer = true`) bypasses the timestamp check entirely: [6](#0-5) 

This means only the sequencer can drain the slow mode queue during a frozen-timestamp period. The user has no alternative path.

---

### Impact Explanation

The slow mode queue is the protocol's only censorship-resistance mechanism. Transactions queued there include `WithdrawCollateral`, `DepositCollateral`, and `LinkSigner`. If the timestamp guard cannot be satisfied:

- Users cannot withdraw deposited collateral via `executeSlowModeTransaction()`.
- Funds already transferred to the contract during `depositCollateralWithReferral` (the ERC-20 transfer happens before the slow mode entry is written) are held with no user-accessible release path.
- The sequencer retains exclusive control over execution, defeating the purpose of slow mode entirely.

Concrete corrupted state: `slowModeTxs[idx]` entries remain unexecutable by the depositor/withdrawer; the corresponding collateral balances in `Clearinghouse` are inaccessible to the user.

---

### Likelihood Explanation

zkSync's `SystemContext.sol` explicitly permits new blocks to carry the same `block.timestamp` as the previous block. While the freeze cannot persist indefinitely (block timestamps are validated against a boundary at commit time), the window can be long enough to prevent users from satisfying a 3-day delay. Any period where `block.timestamp` advances slower than wall-clock time directly extends the effective lockup beyond the intended 3 days, with no upper bound visible to the Nado contracts.

---

### Recommendation

Replace the raw `block.timestamp` stamping with an off-chain sequencer-supplied time (already tracked in `times.spotTime` / `times.perpTime` via `SpotTick`/`PerpTick`), or document explicitly that the protocol is not compatible with zkSync's block timing model. If zkSync deployment is intended, the slow mode delay should be enforced using a sequencer-attested timestamp rather than `block.timestamp`, consistent with how `SpotTick` and `PerpTick` already inject time into the engines. [7](#0-6) 

---

### Proof of Concept

1. User calls `depositCollateralWithReferral(subaccount, productId, amount, referral)` on zkSync at block timestamp `T`. ERC-20 tokens are transferred to `Clearinghouse`. A `SlowModeTx` entry is written with `executableAt = T + 259200`.
2. zkSync produces several blocks all carrying timestamp `T` (frozen).
3. User calls `executeSlowModeTransaction()` after 3 real-world days. `block.timestamp` is still `T` (or has advanced by less than 259200 seconds). The check `T + 259200 <= T` fails → revert `ERR_SLOW_TX_TOO_RECENT`.
4. The sequencer is the only party that can process the entry (via `submitTransactionsChecked` → `ExecuteSlowMode` → `fromSequencer = true`). If the sequencer is unresponsive, the user's funds are locked with no on-chain recourse.
