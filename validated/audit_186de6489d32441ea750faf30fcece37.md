### Title
`ENABLE_SECP_OPS` Flag Bypass via 4-Byte Opcode Aliases — (`File: src/chia_dialect.rs`)

---

### Summary

The `ENABLE_SECP_OPS` flag in `ChiaDialect` is intended to gate access to the secp256k1 and secp256r1 signature-verification operators. However, the flag is only checked for the 1-byte opcode aliases (opcodes 64 and 65). The identical underlying functions `op_secp256k1_verify` and `op_secp256r1_verify` are also reachable via 4-byte opcodes `0x13d61f00` and `0x1c3a8f00`, which are dispatched unconditionally — with no `ENABLE_SECP_OPS` check — in the same `op()` method. An attacker-controlled CLVM program can invoke full secp signature verification in any dialect configuration that does not set `ENABLE_SECP_OPS` (including `MEMPOOL_MODE`) simply by using the 4-byte opcode form.

---

### Finding Description

In `src/chia_dialect.rs`, the `ChiaDialect::op()` method handles two opcode-length branches:

**4-byte branch (lines 157–183):** Dispatches `0x13d61f00` → `op_secp256k1_verify` and `0x1c3a8f00` → `op_secp256r1_verify` with **no flag check**:

```rust
// src/chia_dialect.rs lines 175-182
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

**1-byte branch (lines 248–249):** Dispatches opcodes 64 and 65 to the same functions, but **only when `ENABLE_SECP_OPS` is set**:

```rust
// src/chia_dialect.rs lines 248-249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The `ENABLE_SECP_OPS` flag is defined as:

```rust
// src/chia_dialect.rs lines 62-63
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

`MEMPOOL_MODE` does **not** include `ENABLE_SECP_OPS`:

```rust
// src/chia_dialect.rs lines 72-76
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

In mempool mode, a CLVM program using 1-byte opcode 64 falls through to `unknown_operator`, which immediately returns `EvalErr::Unimplemented` because `NO_UNKNOWN_OPS` is set. The same program using 4-byte opcode `0x13d61f00` bypasses both the `ENABLE_SECP_OPS` check and the `unknown_operator` path, and executes `op_secp256k1_verify` directly and successfully.

The underlying functions perform no flag check of their own:

```rust
// src/secp_ops.rs lines 61-65
pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,   // flags are ignored
) -> Response {
```

### Impact Explanation

The broken invariant is: **"secp signature verification is unavailable when `ENABLE_SECP_OPS` is not set."** The 4-byte opcode form violates this invariant unconditionally.

Concrete consequences:

1. **Mempool/consensus divergence:** A transaction submitted to a mempool node using opcode `0x13d61f00` executes secp verification and is accepted. The same logical operation using opcode 64 is rejected. Two nodes running different dialect configurations (one with `ENABLE_SECP_OPS`, one without) will reach different accept/reject decisions for the same economic transaction, depending solely on which opcode encoding the submitter chose.

2. **Flag-gating intent is broken:** Any caller that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` to restrict secp operations (e.g., before a hard-fork activation height) cannot actually enforce that restriction. An attacker who knows the 4-byte encoding can always invoke secp verification regardless of the flag.

3. **Operator availability inconsistency:** The cost of both paths is identical (1,300,000 for k1, 1,850,000 for r1), so there is no cost-accounting divergence, but the accept/reject behavior diverges, which is the exact class of inconsistency flagged in the reference report.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM program submitted to a Chia node is attacker-controlled bytes. The 4-byte opcode encoding is documented in the cost formula comment in the source and is derivable from the cost constants. No special privilege, social engineering, or configuration access is required. The bypass requires only substituting the 4-byte opcode for the 1-byte opcode in a CLVM program.

---

### Recommendation

Add an `ENABLE_SECP_OPS` guard to the 4-byte dispatch branch, mirroring the guard on the 1-byte branch:

```rust
// src/chia_dialect.rs, 4-byte branch
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

If the 4-byte form is intentionally always available (as a legacy path), the `ENABLE_SECP_OPS` flag documentation and the 1-byte guard should be updated to make this asymmetry explicit and intentional, and the mempool-mode behavior should be audited to confirm the inconsistency is acceptable.

### Proof of Concept

Given a `ChiaDialect` constructed with `MEMPOOL_MODE` (which excludes `ENABLE_SECP_OPS`):

**Rejected (1-byte opcode 64):**
```
CLVM program: (64 pubkey msg sig)
Result: EvalErr::Unimplemented  ← blocked by NO_UNKNOWN_OPS
```

**Accepted (4-byte opcode 0x13d61f00):**
```
CLVM program: (0x13d61f00 pubkey msg sig)
Result: op_secp256k1_verify executes, returns nil on success
Cost charged: 1,300,000
```

Both programs invoke `op_secp256k1_verify` from `src/secp_ops.rs`. Only the 1-byte form is blocked by the `ENABLE_SECP_OPS` guard at `src/chia_dialect.rs` lines 248–249.
