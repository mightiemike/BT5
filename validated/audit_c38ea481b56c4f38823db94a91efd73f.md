### Title
NEAR Storage Deposits for GC'd Block Headers Are Permanently Locked in the Contract — (`File: contract/src/lib.rs`)

---

### Summary

`submit_blocks` is the only `#[payable]` entry point in `BtcLightClient`. It collects NEAR deposits to cover on-chain storage costs. When `run_mainchain_gc` later removes old block headers, the storage those headers occupied is freed, but the NEAR that was deposited in **prior calls** to cover that storage has no withdrawal path and is permanently locked in the contract.

---

### Finding Description

`submit_blocks` measures the **net** storage delta of the current call (after GC has already run) and refunds only the surplus from that single call:

```
let initial_storage = env::storage_usage();          // snapshot before
// ... add new headers ...
self.run_mainchain_gc(num_of_headers);               // may remove old headers
let diff_storage_usage = env::storage_usage()
    .saturating_sub(initial_storage);               // net delta, floor 0
let required_deposit = env::storage_byte_cost()
    .saturating_mul(diff_storage_usage.into());
// refund only the excess from THIS call
let refund = amount.saturating_sub(required_deposit);
``` [1](#0-0) 

`run_mainchain_gc` removes entries from `mainchain_height_to_header`, `mainchain_header_to_height`, and `headers_pool` once the stored chain exceeds `gc_threshold`: [2](#0-1) 

`remove_block_header` deletes the two index entries and the pool entry, freeing their storage bytes: [3](#0-2) 

The freed storage reduces the contract's storage obligation, making the corresponding NEAR available in the contract's balance. However, there is **no function** in the contract — no `withdraw`, no `refund_storage`, no admin drain — that can move that balance back to any caller. A search of the entire `contract/src/lib.rs` confirms `#[payable]` appears exactly once (on `submit_blocks`) and no transfer-out path exists for the contract's own balance. [4](#0-3) 

**Concrete scenario:**

| Step | Action | Storage | Contract NEAR balance |
|------|--------|---------|----------------------|
| Call A | Submit 100 headers, deposit 100 × `storage_byte_cost` | +100 units | +100 × cost |
| Call B | Submit 100 headers; GC removes the 100 from Call A | net Δ = 0 → `required_deposit = 0`; full deposit B refunded | Call A's NEAR still locked |
| … | Repeat indefinitely | GC keeps freeing | Balance grows monotonically |

---

### Impact Explanation

**High.** Every NEAR token deposited by a relayer to cover storage for headers that are subsequently garbage-collected is permanently locked in the contract. Over the operational lifetime of the light client (configured with `gc_threshold = 52704` blocks ≈ one year of Bitcoin blocks), the cumulative locked amount grows proportionally to the total number of headers ever submitted and GC'd. There is no privileged or unprivileged path to recover these funds.

---

### Likelihood Explanation

**High.** GC is not an edge case — it is the normal steady-state operation of the contract. `run_mainchain_gc` is called automatically inside every `submit_blocks` invocation once the stored chain exceeds `gc_threshold`. The relayer is designed to run continuously, so GC will trigger on every batch submission after the threshold is reached. The locking of prior-call deposits is therefore a guaranteed, recurring consequence of normal operation, not an accidental or rare event.

---

### Recommendation

Add a privileged withdrawal function (callable by `Role::DAO` or a dedicated treasurer role) that transfers the contract's surplus balance — i.e., `env::account_balance() - env::storage_byte_cost() * env::storage_usage()` — to a designated recipient. Alternatively, track per-call depositors and freed storage in a side map so that freed deposits can be refunded to the original depositor when GC removes their headers.

---

### Proof of Concept

1. Deploy the contract with `gc_threshold = 10`.
2. Relayer (trusted) calls `submit_blocks` with 10 headers and a deposit of `10 × storage_byte_cost`. Deposit is retained; contract balance increases.
3. Relayer calls `submit_blocks` with 10 more headers and the same deposit. Inside the call, `run_mainchain_gc` removes the first 10 headers. `env::storage_usage()` after GC equals `initial_storage` (net Δ = 0), so `required_deposit = 0` and the current call's deposit is fully refunded.
4. The NEAR from step 2 remains in the contract's balance. No function exists to retrieve it.
5. Repeat steps 2–4 indefinitely; the locked balance grows by `10 × storage_byte_cost` per cycle with no recovery path. [5](#0-4)

    #[payable]
    #[pause]
    #[trusted_relayer]
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
