Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end user, enabling allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension evaluates the router's allowlist status rather than the end user's. Any non-allowlisted user can bypass the swap gate by routing through an allowlisted router, breaking the pool's access-control invariant.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every registered extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called the pool: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly; the pool therefore sees the router as `msg.sender`, not the originating user: [4](#0-3) 

The extension consequently evaluates `allowedSwapper[pool][router]`. A pool admin who allowlists the router (the natural action to permit standard UI/SDK access) inadvertently grants every downstream caller of the router the ability to swap, regardless of individual allowlist status. No existing guard in the extension or the pool checks the originating user.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of addresses (e.g., KYC'd counterparties, whitelisted market makers) loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The non-allowlisted user receives the full swap output at the oracle-anchored price; the pool's LP assets are exposed to unrestricted trading. This is a direct, fund-impacting policy bypass: the allowlist invariant — "only approved addresses may swap" — is broken for the router-mediated path.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical periphery entry point for swaps. Pool admins are expected to allowlist it so their users can access the pool through the standard UI/SDK. The `setAllowedToSwap` and `setAllowAllSwappers` interfaces give no indication that allowlisting the router grants blanket access to all downstream callers. The mistake is a natural and foreseeable admin action. The bypass is repeatable by any user with access to the router.

## Recommendation

Pass the economically relevant actor to the extension. The preferred fix is for `MetricOmmSimpleRouter` to encode the original `msg.sender` in `extensionData` and for `SwapAllowlistExtension.beforeSwap` to decode and check it when the immediate caller is a known router. Alternatively, the pool can be extended to carry both the immediate caller and the originating user so extensions can choose which to gate. At minimum, the extension documentation must warn that allowlisting a router grants access to all its callers.

## Proof of Concept

```
Setup:
  1. Deploy pool with SwapAllowlistExtension.
  2. Admin calls setAllowedToSwap(pool, alice, true)   // alice is individually allowed
  3. Admin calls setAllowedToSwap(pool, router, true)  // router allowlisted for standard access
  4. bob is NOT individually allowlisted.

Attack:
  5. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: bob, ...})
  6. Router calls pool.swap(bob, ...) — pool's msg.sender = router.
  7. _beforeSwap passes sender = router to SwapAllowlistExtension.
  8. Extension checks allowedSwapper[pool][router] == true → passes.
  9. bob's swap executes at oracle price; allowlist is bypassed.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds, bob receives token output.
```
