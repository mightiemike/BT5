### Title
Unconstrained `VersionedConstantsOverrides` Allows Gateway to Accept Transactions That Blockifier/Batcher Will Reject (or Vice Versa) — (`File: crates/apollo_gateway_config/src/config.rs`, `crates/blockifier/src/blockifier_versioned_constants.rs`)

---

### Summary

`VersionedConstantsOverrides` (`validate_max_n_steps`, `invoke_tx_max_n_steps`, `max_recursion_depth`, `max_n_events`) is a fully operator-configurable struct with **no range checks or upper-bound enforcement**. It is applied independently in two separate components — the gateway's `StatefulTransactionValidator` and the batcher's `BlockBuilder` — each reading from its own config subtree. There is no enforcement that the two components use the same values. A misconfigured node (or a deliberately crafted deployment config) can set the gateway's `validate_max_n_steps` higher than the batcher's, causing the gateway to admit transactions whose `__validate__` entry point will exceed the batcher's step limit and be rejected or reverted during block building. This is the direct sequencer analog of the AssetProxyOwner timelock mismatch: a configurable protocol parameter has no range check, so the "specification" (gateway admission) and the "implementation" (blockifier execution) can diverge.

---

### Finding Description

`VersionedConstantsOverrides` is defined in `crates/blockifier/src/blockifier_versioned_constants.rs`:

```rust
pub struct VersionedConstantsOverrides {
    pub validate_max_n_steps: u32,
    pub max_recursion_depth: usize,
    pub invoke_tx_max_n_steps: u32,
    pub max_n_events: usize,
}
```

It carries no `#[validate(range(...))]` attributes and no upper-bound check anywhere in its construction or application path. [1](#0-0) 

`VersionedConstants::get_versioned_constants` blindly substitutes the override values with no clamping: [2](#0-1) 

The gateway's `StatefulTransactionValidatorConfig` holds an `Option<VersionedConstantsOverrides>` and passes it directly to `get_versioned_constants` inside `run_validate_entry_point`: [3](#0-2) 

The batcher's `BlockBuilderConfig` has its own independent `versioned_constants_overrides` pointer in the config schema, pointing to the same shared pointer target but **independently overridable** per deployment: [4](#0-3) [5](#0-4) 

Both config subtrees (`batcher_config.static_config.block_builder_config.versioned_constants_overrides.*` and `gateway_config.static_config.stateful_tx_validator_config.versioned_constants_overrides.*`) are independently settable. The `StatefulTransactionValidatorConfig` struct has no `#[validate(nested)]` on the `versioned_constants_overrides` field and `VersionedConstantsOverrides` itself has no `Validate` derive: [6](#0-5) 

---

### Impact Explanation

**Impact: High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

If `gateway.versioned_constants_overrides.validate_max_n_steps` is set to a value larger than `batcher.versioned_constants_overrides.validate_max_n_steps`, the gateway will admit a transaction whose `__validate__` function runs more steps than the batcher allows. The batcher will then fail or revert the transaction during block building, wasting block space and causing the user's transaction to be silently dropped or reverted after admission. Conversely, if the gateway's limit is set lower than the batcher's, valid transactions are rejected at the gateway that would have succeeded in the block.

The same divergence applies to `invoke_tx_max_n_steps`, `max_recursion_depth`, and `max_n_events`.

---

### Likelihood Explanation

**Medium.** In the default consolidated deployment, both components share the same pointer target (`versioned_constants_overrides.*`) so they are identical. However, in the distributed/hybrid deployment topology — which is the production topology — the gateway and batcher run as separate services with separate config files (`gateway_config.json`, `batcher_config.json`). An operator who tunes one without the other, or a deployment that uses the `replacer_general_config.json` template with `versioned_constants_overrides.#is_none` set to different values per service, will silently create a divergence. There is no startup validation that cross-checks the two values. [7](#0-6) 

---

### Recommendation

1. **Short term:** Add `#[validate(range(max = PROTOCOL_VALIDATE_MAX_N_STEPS))]` (and equivalent for `invoke_tx_max_n_steps`, `max_recursion_depth`, `max_n_events`) to `VersionedConstantsOverrides`, where the upper bound is the value from the canonical versioned constants JSON for the current protocol version. Alternatively, add a cross-component consistency check at node startup that asserts `gateway.versioned_constants_overrides == batcher.versioned_constants_overrides`.

2. **Long term:** Enforce that `VersionedConstantsOverrides` values can only be set **at or below** the protocol-canonical values from the versioned constants JSON files. The override mechanism should be a downward-only relaxation (e.g., for testing), not an upward extension that can exceed what the blockifier enforces during execution.

---

### Proof of Concept

1. Deploy the sequencer in distributed mode.
2. Set in `gateway_config.json`:
   ```json
   "gateway_config.static_config.stateful_tx_validator_config.versioned_constants_overrides.validate_max_n_steps": 2000000
   ```
3. Set in `batcher_config.json` (or leave at default):
   ```json
   "batcher_config.static_config.block_builder_config.versioned_constants_overrides.validate_max_n_steps": 1000000
   ```
4. Submit an invoke transaction whose `__validate__` function consumes between 1,000,001 and 2,000,000 steps.
5. **Observed:** The gateway's `run_validate_entry_point` calls `VersionedConstants::get_versioned_constants(Some(overrides))` with `validate_max_n_steps = 2_000_000` and admits the transaction to the mempool.
6. **Observed:** The batcher builds a block using `validate_max_n_steps = 1_000_000`; the transaction's `__validate__` exceeds the limit and the transaction is rejected/reverted during block building.
7. The transaction was admitted by the gateway (RPC returns success) but never included in a valid block — a gateway admission/sequencing divergence.

The exact divergence point is `VersionedConstants::get_versioned_constants` called with two different `VersionedConstantsOverrides` instances, one per component, with no cross-check: [2](#0-1) [8](#0-7)

    pub fn get_versioned_constants(
        versioned_constants_overrides: Option<VersionedConstantsOverrides>,
    ) -> Self {
        let latest_constants = Self::latest_constants().clone();
        match versioned_constants_overrides {
            None => latest_constants,
            Some(VersionedConstantsOverrides {
                validate_max_n_steps,
                max_recursion_depth,
                invoke_tx_max_n_steps,
                max_n_events,
            }) => {
                let tx_event_limits = EventLimits {
                    max_n_emitted_events: max_n_events,
                    ..latest_constants.tx_event_limits
                };
                Self {
                    validate_max_n_steps,
                    max_recursion_depth,
                    invoke_tx_max_n_steps,
                    tx_event_limits,
                    ..latest_constants
                }
            }
        }
    }
