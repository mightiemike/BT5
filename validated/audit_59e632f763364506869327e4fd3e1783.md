Audit Report

## Title
Pool Admin Can Reduce Timelock to Zero and Atomically Disable All Stop-Loss Protections Without LP Reaction Window — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`proposeOracleStopLossTimelock` accepts `newTimelock = 0` with no minimum floor validation. Once executed, `_afterTimelock` returns `block.timestamp`, and `_requireElapsed(block.timestamp)` passes immediately because `block.timestamp < block.timestamp` is false. The pool admin can then propose and execute `drawdownE6 = 0` in the same block, causing `_afterSwapOracleStopLoss` to return early unconditionally and disabling the entire stop-loss guard with zero LP reaction window — a direct bypass of the timelock mechanism that was explicitly designed to constrain the pool admin.

## Finding Description

**Root cause — no minimum timelock validation:**

`proposeOracleStopLossTimelock` at L78–84 accepts any `uint32 newTimelock`, including `0`, without calling any validation function:

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);   // uses current timelock, not newTimelock
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    ...
}
``` [1](#0-0) 

After the current timelock T elapses, `executeOracleStopLossTimelock` writes `oracleStopLossConfig[pool_].timelock = 0`: [2](#0-1) 

**Zero timelock collapses the propose/execute guard:**

With `timelock = 0`, `_afterTimelock` returns `uint32(block.timestamp + 0)` = `block.timestamp`: [3](#0-2) 

`_requireElapsed(block.timestamp)` evaluates `block.timestamp < block.timestamp` → **false** → does not revert: [4](#0-3) 

So `proposeOracleStopLossDrawdown(pool_, 0)` followed immediately by `executeOracleStopLossDrawdown(pool_)` in the **same block** succeeds. `_validateDrawdown(0)` only checks `0 > E6` which is false, so `drawdownE6 = 0` is accepted: [5](#0-4) 

**Stop-loss guard fully disabled:**

`_afterSwapOracleStopLoss` short-circuits at L217 when `drawdown == 0`, skipping all watermark checks for every subsequent swap: [6](#0-5) 

The same zero-delay pattern applies identically to `proposeOracleStopLossDecay`/`executeOracleStopLossDecay` and `proposeOracleStopLossHighWatermarks`/`executeOracleStopLossHighWatermarks`, so all three parameter axes can be atomically corrupted in the same block.

**Existing guards are insufficient:** `initialize` calls `_validateDrawdown` and `_validateDecay` but has no `_validateTimelock`, so a zero timelock can be set both at initialization and post-deployment: [7](#0-6) 

## Impact Explanation

Once `drawdownE6 = 0`, the `afterSwap` hook returns early for every swap. The stop-loss that prevents per-share value from falling below the drawdown floor is completely inoperative. LPs are exposed to unlimited value leakage through swaps with no on-chain protection. This is a direct admin-boundary break: the pool admin exceeds the cap imposed by the timelock mechanism that the NatDoc explicitly states exists so "LPs can react." The corrupted value is `oracleStopLossConfig[pool_].drawdownE6` written to `0`, causing the extension decision in `_afterSwapOracleStopLoss` to unconditionally return without checking watermarks. **Severity: High** — broken core pool functionality (stop-loss guard) causing potential loss of LP principal.

## Likelihood Explanation

The pool admin is a single EOA or contract returned by `IMetricOmmPoolFactory(FACTORY).poolAdmin(pool_)` and is explicitly semi-trusted — the timelock exists precisely to limit their power. The attack requires only two phases: (1) wait the current timelock (visible on-chain, but LPs may not monitor), then (2) atomically disable protections in one block. No external oracle manipulation, no non-standard tokens, and no factory owner involvement is needed. The attack is repeatable and leaves no on-chain trace distinguishable from a legitimate admin action.

## Recommendation

1. **Enforce a minimum timelock floor** in `proposeOracleStopLossTimelock`: add `require(newTimelock >= MIN_TIMELOCK, ...)` where `MIN_TIMELOCK` is a protocol constant (e.g., 1 hour or 1 day).
2. **Use strict inequality** in `_requireElapsed`: change `block.timestamp < executeAfter` to `block.timestamp <= executeAfter` so same-block execution is always rejected regardless of timelock value.
3. **Validate timelock at initialization**: add a `_validateTimelock` call in `initialize` alongside the existing `_validateDrawdown` and `_validateDecay` calls to prevent a zero timelock from being set at pool creation.

## Proof of Concept

```solidity
// Foundry integration test sketch
function test_timelockZeroDisablesStopLoss() public {
    // Assume pool deployed with timelock = 1 days, drawdownE6 = 50_000 (5%)

    // Phase 1: propose timelock = 0 (must wait current timelock)
    vm.prank(poolAdmin);
    ext.proposeOracleStopLossTimelock(pool, 0);

    // Warp past current timelock
    vm.warp(block.timestamp + 1 days);

    // Execute timelock reduction — now oracleStopLossConfig[pool].timelock == 0
    vm.prank(poolAdmin);
    ext.executeOracleStopLossTimelock(pool);

    // Phase 2: same block — propose drawdown = 0
    vm.prank(poolAdmin);
    ext.proposeOracleStopLossDrawdown(pool, 0);
    // executeAfter = block.timestamp + 0 = block.timestamp

    // Same block — execute drawdown = 0 (block.timestamp < block.timestamp == false, passes)
    vm.prank(poolAdmin);
    ext.executeOracleStopLossDrawdown(pool);

    // Assert drawdown is now 0
    (uint32 drawdown,,,) = ext.oracleStopLossConfig(pool);
    assertEq(drawdown, 0);

    // Phase 3: swap that would have triggered stop-loss now succeeds
    // _afterSwapOracleStopLoss returns early at `if (drawdown == 0) return;`
    // No OracleStopLossTriggered revert — stop-loss is fully disabled
    _executeValueDrainingSwap(pool); // succeeds, no revert
}
```
