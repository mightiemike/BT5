### Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any non-allowlisted address to bypass the deposit gate — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual token-providing caller) and instead checks whether the LP-position `owner` is allowlisted. Any non-allowlisted address can call `pool.addLiquidity(owner=allowlistedAddress)`, pass the guard, and deposit tokens into the pool.

### Finding Description
`MetricOmmPool.addLiquidity` captures `msg.sender` and passes it as `sender` to `_beforeAddLiquidity`, which forwards it to every configured extension via `abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, ...))`.

`DepositAllowlistExtension.beforeAddLiquidity` declares its first parameter unnamed (`address,`) and never reads it. Its guard is:

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner])
```

`msg.sender` here is the pool (correct for the pool-keyed mapping), but `owner` is the LP-position recipient, not the token provider. The token provider (`sender`) is never checked.

Attack path:
1. Pool is deployed with `DepositAllowlistExtension` as `extension1`.
2. Pool admin allowlists address A (`allowedDepositor[pool][A] = true`). Attacker B is not allowlisted.
3. Attacker B calls `pool.addLiquidity(owner=A, ...)`.
4. Pool calls `extension.beforeAddLiquidity(sender=B, owner=A, ...)`.
5. Extension checks `allowedDepositor[pool][A] == true` → passes.
6. Pool executes `LiquidityLib.addLiquidity(owner=A, ...)`, pulling tokens from B via callback and minting LP shares to A.
7. B has deposited into a restricted pool without ever being allowlisted.

### Impact Explanation
The `DepositAllowlistExtension` is the sole mechanism for restricting who may deposit into a pool. The bypass is complete: any address can deposit tokens into an allowlist-gated pool by nominating any allowlisted address as `owner`. The depositing address (the actual token provider) is never validated. This breaks the core security invariant of the extension — that only allowlisted addresses may provide liquidity — and allows unrestricted token inflows into pools that pool admins intended to gate.

Additionally, the allowlisted `owner` receives LP shares they did not request, creating an unwanted position they must actively manage (griefing vector).

### Likelihood Explanation
The entrypoint is fully public (`addLiquidity` is `external`). No privileged role, special token, or deployment control is required. The attacker only needs to know one allowlisted address for the target pool (trivially discoverable on-chain via `AllowedToDepositSet` events). Exploitation requires a single transaction.

### Recommendation
Check `sender` (the actual token provider), not `owner` (the LP-position recipient):

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```
