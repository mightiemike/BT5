### Title
Integer Overflow in `op_unknown` Cost Multiplier Allows Undercharged Execution - (File: src/more_ops.rs)

### Summary

`op_unknown` in `src/more_ops.rs` applies a cost multiplier to the accumulated base cost after the `max_cost` guard has already passed. In Rust release mode, the `u64 *= u64` multiplication wraps silently on overflow. An attacker-controlled unknown opcode with a large `cost_multiplier` and arguments sized to push the pre-multiplication cost above `u64::MAX / (cost_multiplier + 1)` causes the returned cost to wrap to a tiny value, undercharging the execution and allowing programs to consume more resources than the cost budget permits.

### Finding Description

`op_unknown` computes cost in three phases:

1. Accumulate a base cost proportional to argument sizes (cost_function 0–3), with incremental `check_cost` guards inside the loop.
2. Call `check_cost(cost, max_cost)` once more after the loop — this is the last guard before multiplication.
3. Multiply: `cost *= cost_multiplier + 1`. [1](#0-0) 

The post-multiplication result is only checked against `u32::MAX`, **not** against `max_cost`: [2](#0-1) 

`cost_multiplier` is decoded from the opcode bytes as a `u32` (up to `u32::MAX = 4 294 967 295`), cast to `u64`: [3](#0-2) 

For cost_function `2` (mul-like), the base cost grows quadratically with argument size: [4](#0-3) 

With two arguments of length L bytes, the base cost is approximately `L² / 128`. For L ≈ 742 000 bytes (≈ 742 KB), the base cost reaches ≈ 4.3 billion, which is above `u64::MAX / 2³² ≈ 4 294 967 296`. When multiplied by `cost_multiplier + 1 = 2³²`, the product overflows `u64` and wraps to a small value (e.g., 2). The `if cost > u32::MAX` guard does not fire, and `Reduction(2, nil)` is returned instead of the correct multi-billion cost.

In Rust release builds, `u64 *= u64` wraps silently on overflow (two's complement). This is the necessary vulnerable step.

### Impact Explanation

The returned cost is added to the running total in `run_program`: [5](#0-4) 

Because the wrapped cost is tiny, the program's running total is barely incremented. All subsequent operators in the same program can consume up to nearly the full `max_cost` budget again, effectively granting the program a second (or larger) cost budget. A block generator can include programs whose true resource consumption far exceeds the declared cost limit, allowing more computation per block than consensus rules intend. All nodes running the same release binary compute the same wrapped cost, so there is no consensus split — the undercharging is universal and silent.

`op_unknown` is reachable in consensus mode (without `NO_UNKNOWN_OPS`): [6](#0-5) 

`MEMPOOL_MODE` sets `NO_UNKNOWN_OPS`, blocking this path in mempool validation, but consensus-mode block execution does not: [7](#0-6) 

### Likelihood Explanation

An attacker must craft a CLVM program containing an unknown opcode byte sequence that encodes `cost_function = 2` and `cost_multiplier = u32::MAX`, paired with two atom arguments of ≈ 742 KB each. The total serialized program is ≈ 1.5 MB, well within Chia's block size limits. No privileged access is required; any coin puzzle submitted to the network can trigger this path. The construction is deterministic and requires no brute-force search.

### Recommendation

Replace the bare multiplication with a checked or saturating variant and enforce `max_cost` after multiplication:

```rust
// Instead of:
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 { ... }

// Use:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::CostExceeded)?;
check_cost(cost, max_cost)?;
if cost > u32::MAX as u64 { ... }
```

This ensures the post-multiplication cost is both overflow-safe and bounded by `max_cost` before the result is returned.

### Proof of Concept

```
Opcode bytes (5 bytes, cost_function=2, cost_multiplier=u32::MAX):
  [0xFF, 0xFF, 0xFF, 0xFF, 0x80]
  ^--- 4-byte multiplier = 0xFFFFFFFF = u32::MAX
                            ^--- last byte: bits[7:6] = 10 → cost_function = 2

Arguments: two atoms, each 742 000 bytes of arbitrary data.

Pre-multiplication cost (cost_function 2, two args of length L=742000):
  cost ≈ MUL_BASE_COST + MUL_COST_PER_OP
       + (L + L) * MUL_LINEAR_COST_PER_BYTE
       + (L * L) / MUL_SQUARE_COST_PER_BYTE_DIVIDER
  ≈ 92 + 885 + 1484000*6 + (742000²/128)
  ≈ 92 + 885 + 8 904 000 + 4 303 281 250
  ≈ 4 312 186 227   (> u64::MAX / 2^32 = 4 294 967 296 ✓)

check_cost(4_312_186_227, 11_000_000_000) → passes

cost *= (u32::MAX + 1) = 4_294_967_296
  4_312_186_227 * 4_294_967_296 = 18_521_... > u64::MAX → wraps to small value

if wrapped_cost > u32::MAX → false
→ returns Reduction(wrapped_cost /* e.g. ~2 */, nil)
```

The running cost in `run_program` increases by only the wrapped value, leaving nearly the full `max_cost` budget available for subsequent operators.
