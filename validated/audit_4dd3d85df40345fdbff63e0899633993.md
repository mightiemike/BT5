### Title
Fork Retarget Uses Mainchain Ancestor Timestamp Instead of Fork's Own Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

`get_header_by_height` always resolves through `mainchain_height_to_header`. When `get_next_work_required` is called for a fork block at a retarget boundary, it fetches the **mainchain** block at `first_block_height` rather than the fork's actual ancestor at that height. If the fork diverged before `first_block_height`, the two blocks are distinct and may carry different timestamps, causing `calculate_next_work_required` to compute a wrong `expected_bits` for the fork. The developers themselves flagged this in a TODO comment in `dogecoin.rs`.

---

### Finding Description

**Root cause — `BlocksGetter::get_header_by_height` is mainchain-only:** [1](#0-0) 

The implementation unconditionally reads from `mainchain_height_to_header`. There is no fork-aware path.

**Bitcoin retarget path:** [2](#0-1) 

`first_block_height = prev_block_header.block_height − (difficulty_adjustment_interval − 1)`. At the second retarget (height 4032), `first_block_height = 2016`. If the fork diverged at height 2000, the fork's ancestor at height 2016 is a different block than the mainchain block at height 2016. The code fetches the mainchain block and uses its timestamp.

**Litecoin retarget path (identical pattern):** [3](#0-2) 

**Dogecoin retarget path — developer-acknowledged TODO:** [4](#0-3) 

The comment "TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor" confirms the developers recognized the issue but left it unresolved.

**Difficulty calculation using the wrong timestamp:** [5](#0-4) 

`actual_time_taken = prev_block_time − first_block_time`. If the mainchain block at `first_block_height` has an earlier timestamp than the fork's actual ancestor at that height, `actual_time_taken` is inflated, producing an easier (higher) target. The attacker controls the fork's ancestor timestamps (within MTP and future-time bounds), so they can engineer this condition deliberately.

**Validation accepts the wrong `bits`:** [6](#0-5) 

`check_pow` enforces `expected_bits == block_header.bits`. Because `expected_bits` was computed from the wrong (mainchain) ancestor, a fork block carrying the attacker-chosen easier `bits` passes this check.

**Entry point:** [7](#0-6) 

`submit_blocks` is the public entry point. It is gated by `#[trusted_relayer]`, an economic staking barrier, not a privileged-role restriction. Any account that stakes the required amount can submit fork headers.

---

### Impact Explanation

A fork block at a retarget boundary is accepted with a `bits` value that does not correspond to the fork's true required difficulty. This is a direct consensus-rule violation: the block is invalid under the intended network rules but passes `check_pow`. Concretely:

- The attacker mines the retarget block against an easier target (up to 4× easier, bounded by the `pow_target_timespan` clamp), reducing PoW cost for that block.
- The accepted block carries the attacker-chosen `bits`, so `work_from_bits` computes less chainwork for it. This partially offsets the mining advantage, but the attacker still benefits from reduced mining expenditure at the boundary.
- If the fork accumulates sufficient total chainwork (possible with a long enough fork), `reorg_chain` promotes it to `mainchain_tip_blockhash` with at least one block whose `bits` violates the fork's true consensus rules. [8](#0-7) 

---

### Likelihood Explanation

**Barriers:**
1. The attacker must be a trusted relayer (economic staking cost).
2. All fork blocks before the retarget boundary must carry valid PoW hashes.
3. The fork must diverge before `first_block_height`, which for Bitcoin means before the start of the current 2016-block interval — requiring the attacker to have mined or obtained valid fork blocks spanning that range.

**Facilitating factors:**
1. The bug is unconditional in the code — no configuration disables it.
2. The TODO comment confirms the developers are aware and have not fixed it.
3. On testnet or lower-difficulty networks the PoW cost is negligible.
4. The attacker fully controls fork ancestor timestamps within the MTP/future-time window, making the timestamp manipulation deterministic.

Likelihood is **low on mainnet** (PoW cost) and **medium-to-high on testnet or low-difficulty deployments**.

---

### Recommendation

Replace the `get_header_by_height` call in all three retarget functions with an ancestor walk that follows `get_prev_header` from `prev_block_header` backward by `blocks_to_go_back` steps. This mirrors how Bitcoin Core actually traverses the chain in `GetAncestor`. The `get_header_by_height` shortcut is only correct when the fork and mainchain share the same block at `first_block_height`, which is not guaranteed when the fork diverges before that height.

---

### Proof of Concept

**Setup (Bitcoin feature, NEAR SDK test):**

1. Initialize the contract at height 2016 (a retarget boundary), supplying 12+ real mainchain headers.
2. Submit mainchain blocks up to height 4031 (end of second interval). Record the mainchain block at height 2016 with timestamp `T_main`.
3. Build a fork starting at height 2000 (before `first_block_height = 2016`). Give the fork's block at height 2016 timestamp `T_fork = T_main + 3600` (one hour later, within future-time bounds).
4. Submit fork blocks 2001–4031 with valid PoW (use `skip_pow_verification = true` in the test to isolate the difficulty logic).
5. Submit fork block 4032 (the retarget block). Observe that `get_next_work_required` calls `get_header_by_height(2016)`, returns the mainchain block with `T_main`, and computes `expected_bits_mainchain`.
6. Compute what `expected_bits` should be using `T_fork`: `expected_bits_fork ≠ expected_bits_mainchain`.
7. Assert that the contract accepts a fork block 4032 carrying `bits = expected_bits_mainchain` (easier), which would be rejected by a correct implementation that uses `T_fork`.

The assertion at step 7 demonstrates that a fork header invalid under the fork's own consensus rules is accepted by the contract. [1](#0-0) [2](#0-1) [4](#0-3)

fn calculate_next_work_required(
    config: &NetworkConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let prev_block_time: i64 = prev_block_header.block_header.time.into();

    let mut actual_time_taken: i64 = prev_block_time - first_block_time;
    if actual_time_taken < config.pow_target_timespan / 4 {
        actual_time_taken = config.pow_target_timespan / 4;
    }
    if actual_time_taken > config.pow_target_timespan * 4 {
        actual_time_taken = config.pow_target_timespan * 4;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(actual_time_taken).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target = new_target
        / U256::from(<i64 as TryInto<u64>>::try_into(config.pow_target_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
