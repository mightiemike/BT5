### Title
`build_precommit_vote_message_digest` Omits `height`, `round`, `chain_id`, and `vote_type` from Signature Preimage, Enabling Cross-Context Replay — (`crates/apollo_signature_manager/src/signature_manager.rs`)

---

### Summary

`build_precommit_vote_message_digest` constructs the signed payload for a consensus precommit vote using only the `block_hash` (plus a static domain tag). It omits `height`, `round`, `chain_id`, and `vote_type`. This is the direct sequencer analog of the `RentPayload` replay bug: just as `RentPayload` lacked `orderHash` and could be replayed across orders sharing the same metadata, a precommit signature here can be replayed across any consensus context that presents the same `block_hash` — different heights, different rounds, or a different chain.

---

### Finding Description

`sign_precommit_vote` calls `build_precommit_vote_message_digest(block_hash)`, which builds the signed message as:

```
PRECOMMIT_VOTE || block_hash.to_bytes_be()
``` [1](#0-0) 

The `Vote` struct that carries this signature contains `height`, `round`, `vote_type`, and `voter` as plaintext fields that are **not** committed to by the signature: [2](#0-1) 

The corresponding verification function reconstructs the same incomplete digest: [3](#0-2) 

The consensus state machine currently defers signature verification with a `TODO`: [4](#0-3) 

And self-votes are emitted with a default (empty) signature: [5](#0-4) 

When the TODO is resolved and `verify_precommit_vote_signature` is wired into `handle_vote`, the broken preimage will be in production. The missing fields create three distinct replay surfaces:

1. **Cross-round replay**: The state machine supports reproposals (`SMRequest::Repropose`). If a proposer reproposed the same block in round R+1, a precommit signature from round R passes verification unchanged.
2. **Cross-height replay**: If the same `block_hash` value appears at two different heights (possible if block hash derivation is not strictly height-bound at the commitment layer), a signature from height H is valid at height H'.
3. **Cross-chain replay**: No `chain_id` is included. A precommit signature produced on a testnet node is cryptographically valid on mainnet for the same `block_hash`.

Additionally, because `vote_type` is absent, if prevote signing is later added using the same scheme, a prevote signature becomes a valid precommit signature for the same `block_hash`.

---

### Impact Explanation

When signature verification is enabled, an attacker who observes a valid precommit `(height=H, round=R, block_hash=X, voter=V, sig=S)` can construct a forged vote `(height=H', round=R', block_hash=X, voter=V, sig=S)` that passes `verify_precommit_vote_signature`. This allows injecting fraudulent precommit votes into the consensus state machine, potentially driving a false quorum or blocking a legitimate one. This maps to: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."**

---

### Likelihood Explanation

The signature infrastructure is actively under construction (two `TODO(Asmaa)` markers). The `sign_precommit_vote` / `verify_precommit_vote_signature` pair is already wired through the `SignatureManager` component and its communication layer. The broken preimage will become load-bearing as soon as the TODOs are resolved. Reproposals (same `block_hash`, different round) are an explicitly supported code path in the state machine, making the cross-round replay immediately triggerable.

---

### Recommendation

Include all context-binding fields in the signed preimage:

```rust
fn build_precommit_vote_message_digest(
    block_hash: BlockHash,
    height: BlockNumber,
    round: Round,
    chain_id: &ChainId,
) -> MessageDigest {
    let mut message = Vec::new();
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash.to_bytes_be());
    message.extend_from_slice(&height.0.to_be_bytes());
    message.extend_from_slice(&round.to_be_bytes());
    message.extend_from_slice(chain_id.as_bytes());
    MessageDigest(blake2s_to_felt(&message))
}
```

This mirrors the fix applied to `RentPayload` (adding `orderHash`) and the pattern already used in `build_peer_identity_message_digest`, which binds both `peer_id` and `challenge` to prevent reuse across identity challenges. [6](#0-5) 

---

### Proof of Concept

1. Validator V signs a precommit for `block_hash=X` at `height=5, round=0`. The digest is `blake2s("PRECOMMIT_VOTE" || X)`.
2. The proposer reproposals the same block at `round=1` (via `SMRequest::Repropose`).
3. An attacker replays the vote `(height=5, round=1, block_hash=X, voter=V, sig=S)` to the network.
4. `verify_precommit_vote_signature(X, S, V.public_key)` recomputes `blake2s("PRECOMMIT_VOTE" || X)` — identical to step 1 — and returns `true`.
5. The state machine in `handle_vote` accepts the replayed vote as a fresh precommit for round 1, potentially contributing to a false quorum. [7](#0-6) [1](#0-0)
