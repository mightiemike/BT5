### Title
Missing Contract Existence Check in `ERC20Helper` Enables Silent Transfer Failure on Destroyed Token — (`File: core/contracts/libraries/ERC20Helper.sol`)

---

### Summary

`ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` issue low-level `.call()` to a token address without first verifying that the address contains contract code. If a supported collateral token is destroyed (via `selfdestruct`), all calls to these functions silently return `success = true` with empty return data, passing the `require` check. This causes withdrawals and fast-withdrawal fee collections to appear to succeed while no tokens are actually moved, permanently corrupting user balances in the SpotEngine.

---

### Finding Description

`ERC20Helper.safeTransfer` and `safeTransferFrom` both perform a raw `.call()` to `address(self)`: [1](#0-0) [2](#0-1) 

Neither function checks `address(self).code.length > 0` before issuing the call. On the EVM, a low-level `call` to an address with no bytecode (e.g., a contract destroyed via `selfdestruct`, or an EOA) returns `(true, "")`. The `require(success && (data.length == 0 || abi.decode(data, (bool))), ERR_TRANSFER_FAILED)` guard passes because `success == true` and `data.length == 0`. No revert occurs, and no tokens are transferred.

This library is used in two critical withdrawal paths:

**Path 1 — Regular withdrawal** (`Clearinghouse.handleWithdrawTransfer`): [3](#0-2) 

`token.safeTransfer(withdrawPool, amount)` silently succeeds. `BaseWithdrawPool.submitWithdrawal` then calls `handleWithdrawTransfer` again: [4](#0-3) 

Both calls silently succeed. The SpotEngine balance is then decremented: [5](#0-4) 

**Path 2 — Fast withdrawal** (`BaseWithdrawPool.submitFastWithdrawal`): [6](#0-5) 

`safeTransferFrom` (fee collection from provider) and `handleWithdrawTransfer` (payout to user) both silently succeed. The `markedIdxs[idx]` flag is set to `true`, so when the sequencer later calls `submitWithdrawal`, it returns early without attempting a second transfer: [7](#0-6) 

---

### Impact Explanation

**Regular withdrawal**: A user's SpotEngine balance is decremented and the withdrawal is permanently marked as processed (`markedIdxs[idx] = true`), but no tokens are transferred to the user. The user's on-chain collateral record is destroyed with no recourse. This is an irreversible accounting corruption — the internal ledger diverges from actual token holdings.

**Fast withdrawal**: The fast-withdrawal provider pays no fee (the `safeTransferFrom` silently succeeds), the user receives no tokens, and the withdrawal idx is permanently consumed. The sequencer's subsequent `submitWithdrawal` call is a no-op. The user loses their internal balance with no token payout.

In both cases, the corrupted state is permanent: the idx is marked, the SpotEngine balance is gone, and no retry path exists.

---

### Likelihood Explanation

The trigger requires a supported collateral token to be destroyed via `selfdestruct`. While `selfdestruct` is increasingly restricted post-EIP-6780 (only effective in the same transaction as deployment), tokens deployed with `selfdestruct` in their constructor or via a proxy that can be bricked remain viable targets. On Ink Chain (an EVM-compatible L2), the opcode behavior depends on the chain's EVM version. If any listed collateral token is a proxy or contains a `selfdestruct` path, this becomes directly exploitable. The likelihood is **medium**: the precondition is non-trivial but realistic for a protocol supporting multiple collateral tokens over time.

---

### Recommendation

Add a contract existence check before each low-level call in `ERC20Helper`:

```solidity
function safeTransfer(IERC20Base self, address to, uint256 amount) internal {
    require(address(self).code.length > 0, "ERC20Helper: token has no code");
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Apply the same guard to `safeTransferFrom`. Consider replacing `ERC20Helper` with OpenZeppelin's `SafeERC20`, which includes this check via `Address.isContract`.

### Proof of Concept

1. A collateral token `T` supported by the Nado SpotEngine is destroyed (e.g., via `selfdestruct` in a proxy upgrade, or by a token with a built-in kill switch).
2. User `A` has an internal SpotEngine balance of 1000 `T`.
3. User `A` submits a `WithdrawCollateral` transaction for 1000 `T`.
4. The sequencer calls `Clearinghouse.withdrawCollateral`.
5. `handleWithdrawTransfer(token, sendTo, 1000, idx)` is called.
6. `token.safeTransfer(withdrawPool, 1000)` → `address(T).call(...)` → `T` has no code → returns `(true, "")` → `require` passes.
7. `BaseWithdrawPool.submitWithdrawal` → `token.safeTransfer(sendTo, 1000)` → same silent success.
8. `spotEngine.updateBalance(productId, sender, -1000e18)` executes — user's balance is zeroed.
9. User `A` receives 0 tokens. Their internal balance is permanently gone. The withdrawal idx is consumed and cannot be replayed. [8](#0-7) [9](#0-8) [10](#0-9)
