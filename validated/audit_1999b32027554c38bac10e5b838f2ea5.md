### Title
`DepositAllowlistExtension` Checks LP Position `owner` Instead of Transaction `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` parameter and gates on `owner` instead. Because `MetricOmmPool.addLiquidity` lets `msg.sender` differ from `owner`, any address not on the allowlist can deposit into a permissioned pool by simply naming an allowlisted address as the LP position owner.

### Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` for the LP position and passes both `msg.sender` (as `sender`) and `owner` to the extension hook: [1](#0-0) 

`removeLiquidity` enforces `msg.sender == owner`, but `addLiquidity` has no such constraint — any caller may credit shares to any owner address. [2](#0-1) 

The extension hook receives `(sender, owner, …)` in that order. `DepositAllowlistExtension.beforeAddLiquidity` names the first parameter `address` (unnamed, i.e. discarded) and checks only `owner`: [3](#0-2) 

The contract's own NatSpec says it "Gates `addLiquidity` by depositor address", and `setAllowedToDeposit` / `isAllowedToDeposit` are framed around the depositor concept: [4](#0-3) 

The actual depositor (`sender`) is never checked.

### Impact Explanation

An address not on the allowlist calls:

```
pool.addLiquidity(allowlistedVictim, salt, deltas, callbackData, extensionData)
```

The extension evaluates `allowedDepositor[pool][allowlistedVictim]` → `true`, so the guard passes. The unauthorized caller pays the tokens; the LP shares are credited to `allowlistedVictim`. Consequences:

1. **Allowlist completely defeated.** The pool admin's intent to restrict who may deposit is nullified. Any address — including sanctioned or otherwise excluded actors — can inject tokens and alter bin state.
2. **Forced LP position on victim.** The allowlisted address receives shares in bins it never chose, potentially in bins that are about to lose value through a swap. While the victim can remove liquidity, the attacker can repeat the deposit to grief continuously.
3. **Pool-state manipulation for downstream extensions.** If the pool also runs `OracleValueStopLossExtension`, the attacker can shift `t0`/`t1` bin balances to manipulate per-share metrics and watermark comparisons, potentially suppressing a legitimate stop-loss trigger and allowing further LP value drain. [5](#0-4) 

### Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with `owner` set to any allowlisted address. No privileged access, no special token, no flash loan. The attacker must supply tokens (which they lose to the pool), making the attack economically rational only for griefing or state manipulation, but the barrier is trivially low.

### Recommendation

Replace the unnamed first parameter with `sender` and check it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the design intent is to allow a whitelisted router to deposit on behalf of any owner, both `sender` and `owner` should be checked with explicit policy (e.g., `sender` must be allowlisted, or `sender == owner`).

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  bob is NOT on the allowlist

Attack:
  bob calls pool.addLiquidity(
      owner        = alice,   // allowlisted — passes the guard
      salt         = 0,
      deltas       = { binIdxs: [0], shares: [1e18] },
      callbackData = ...,     // bob pays tokens in the callback
      extensionData = ""
  )

Result:
  - beforeAddLiquidity checks allowedDepositor[pool][alice] → true → no revert
  - bob's tokens are transferred to the pool
  - alice receives 1e18 LP shares in bin 0
  - bob has bypassed the deposit allowlist entirely
``` [3](#0-2) [6](#0-5)
