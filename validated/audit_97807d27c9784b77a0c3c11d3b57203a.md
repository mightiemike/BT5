### Title
Zero-Confirmation Verification Accepted — No Minimum Confirmation Floor Enforced - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` with no lower-bound enforcement. Any unprivileged NEAR caller may pass `confirmations: 0`, causing the contract to return `true` for a transaction in the current chain-tip block — a block that has not yet been buried under any subsequent work and is therefore fully vulnerable to a Bitcoin re-org.

---

### Finding Description

`ProofArgs` and `ProofArgsV2` both carry a `confirmations: u64` field that is entirely caller-controlled. [1](#0-0) 

Inside `verify_transaction_inclusion`, the contract applies exactly one bound on this value — an **upper** bound against `gc_threshold`: [2](#0-1) 

The depth check that follows is: [3](#0-2) 

When `args.confirmations == 0`, the expression `(tip_height - target_height + 1) >= 0` is trivially true for every block in the chain, including the block at the current tip. The contract therefore returns `true` for a transaction whose block has received zero subsequent confirmations.

`verify_transaction_inclusion_v2` delegates directly to `verify_transaction_inclusion` after its coinbase-proof check, so it inherits the same flaw: [4](#0-3) 

No minimum confirmation constant, no chain-specific floor, and no warning path exist anywhere in the contract source.

---

### Impact Explanation

A recipient NEAR contract that calls `verify_transaction_inclusion_v2` and acts on a `true` result (e.g., minting wrapped BTC, releasing escrowed funds, crediting a bridge balance) will do so for a transaction that may be erased by a Bitcoin re-org. Bitcoin re-orgs of depth 1–2 occur in normal network operation; deeper re-orgs have occurred on minority-hashrate chains (ETC, BCH) supported by the same codebase via feature flags. Because the contract returns a definitive boolean and provides no "depth at time of verification" metadata, the consuming contract has no way to detect that the result was produced with zero confirmation depth. The corrupted state is the consuming contract's ledger entry — funds released or tokens minted against a transaction that no longer exists on the canonical chain.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account may call `verify_transaction_inclusion_v2` with `confirmations: 0`. A malicious actor who controls a Bitcoin miner (or can time a submission to coincide with a natural re-org) can submit a block, immediately call the verifier with `confirmations: 0`, trigger the consuming contract's release logic, and then allow the re-org to erase the original transaction. On chains with lower hashrate (Dogecoin, Litecoin, Zcash — all supported via feature flags), the cost of engineering a short re-org is substantially lower than on Bitcoin mainnet.

---

### Recommendation

1. Enforce a **minimum confirmation floor** inside `verify_transaction_inclusion`. Reject any call where `args.confirmations < MIN_SAFE_CONFIRMATIONS`, where `MIN_SAFE_CONFIRMATIONS` is a chain-specific constant (e.g., 6 for Bitcoin mainnet, higher for chains with lower hashrate).
2. Store the chain identity (`Network`) in contract state (it is already present in `InitArgs`) and derive the per-chain floor from it, avoiding hardcoded values scattered across call sites.
3. Do not allow callers to bypass the floor; if a consuming application genuinely needs a lower depth, that is a risk the application must accept explicitly — but the light-client contract should not silently permit `0`.
4. Emit a structured error (not just a panic) that includes the required minimum, so consuming contracts can distinguish "not yet confirmed" from "proof invalid."

---
