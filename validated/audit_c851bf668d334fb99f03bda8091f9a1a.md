Audit Report

## Title
Router-mediated swaps pass the router address as `sender` to `SwapAllowlistExtension.beforeSwap`, making per-user allowlisting unenforceable for any pool that permits router access — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]` where `sender` is the immediate caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` is the router contract, not the originating user. A pool admin who allowlists the router to enable router usage inadvertently grants every unprivileged user the ability to bypass the per-user allowlist.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap` (metric-core/contracts/MetricOmmPool.sol L230–231), which forwards it unchanged via `ExtensionCalling._beforeSwap` to every configured extension (metric-core/contracts/ExtensionCalling.sol L160–176). `SwapAllowlistExtension.beforeSwap` uses this value as the identity to check against `allowedSwapper[pool][sender]` (metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37–39). When `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` calls `pool.swap`, `msg.sender` at the pool is the router address. The extension therefore checks whether the router is allowlisted, not whether the originating user is allowlisted. The router stores the originating user only as the payer in transient callback storage via `_setNextCallbackContext` (metric-periphery/contracts/MetricOmmSimpleRouter.sol L71, L103), which the extension never reads. Since the router is a single shared contract, allowlisting it opens the gate for all users simultaneously. There is no configuration that allows allowlisted users to use the router while blocking non-allowlisted users.

## Impact Explanation
Any pool that configures `SwapAllowlistExtension` to restrict swaps to a specific set of addresses and allowlists the router so those addresses can use it is fully open to any unprivileged user via the router. The protection boundary the pool admin believed they were enforcing does not exist for router-mediated swaps. Depending on pool economics (favorable rates, restricted liquidity, etc.), this can result in unauthorized extraction of value from the pool by bypassing a core access-control mechanism.

## Likelihood Explanation
The router is the standard user-facing entry point. Any pool admin who wants their allowlisted users to be able to use the router must allowlist it. The bypass requires no special privileges, no flash loans, and no unusual token behavior — only a standard `exactInputSingle` or `exactInput` call through the router. The precondition (router allowlisted) is the natural and expected configuration for any pool that intends to support router usage.

## Recommendation
Pass the originating user's address through the swap path so extensions can check it. One approach: add an `originSender` field to the `extensionData` that the router populates with `msg.sender` before calling `pool.swap`, and have `SwapAllowlistExtension` read and verify it (with the pool authenticating that the field was set by a trusted router). Alternatively, document clearly that `SwapAllowlistExtension` only gates direct callers of `pool.swap` and cannot enforce per-user restrictions for router-mediated swaps.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` as `EXTENSION_1`, `BEFORE_SWAP_ORDER = 1`.
2. Call `setAllowedToSwap(pool, router, true)` — allowlist the router so users can use it.
3. As an address `attacker` that is **not** in `allowedSwapper[pool]`, call `router.exactInputSingle(...)` targeting the pool.
4. The pool calls `_beforeSwap(msg.sender=router, ...)`. The extension checks `allowedSwapper[pool][router] == true` and returns `beforeSwap.selector` without reverting.
5. The swap executes successfully for the non-allowlisted attacker.
6. Confirm: calling `pool.swap(...)` directly from `attacker` reverts with `NotAllowedToSwap`, proving the allowlist is only bypassed via the router.
