### Title
`Liquidation` Event Emits Stale `amountQuote` for Spread Liquidations — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary
In `ClearinghouseLiq._handleLiquidationPayment`, the `Liquidation` event's `amountQuote` parameter is populated from `v.liquidationPayment`, which is **reassigned mid-function** during spread liquidations. The event is emitted after the reassignment, so it reports the perp-leg oracle price payment rather than the spot-leg liquidation payment that was actually transferred to the liquidatee in quote. Off-chain systems consuming this event for spread liquidations receive a structurally incorrect `amountQuote`.

---

### Finding Description

`_handleLiquidationPayment` in `ClearinghouseLiq.sol` handles three liquidation modes. In the spread branch, `v.liquidationPayment` is used for two distinct purposes and is **overwritten** between them:

```solidity
// First assignment — spot-leg liquidation payment
v.liquidationPayment = v.liquidationPriceX18.mul(txn.amount);
...
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.liquidatee, v.liquidationPayment);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender,
    -v.liquidationPayment - v.liquidationFees);

// Second assignment — perp-leg oracle price payment (OVERWRITES first)
v.liquidationPayment = v.oraclePriceX18Perp.mul(txn.amount);
perpEngine.updateBalance(perpId, txn.liquidatee, txn.amount, -v.liquidationPayment);
perpEngine.updateBalance(perpId, txn.sender,   -txn.amount,  v.liquidationPayment);
``` [1](#0-0) 

The `Liquidation` event is then emitted **after** the overwrite, using the second (perp-leg) value:

```solidity
emit Liquidation(
    txn.sender,
    txn.liquidatee,
    txn.productId,
    txn.isEncodedSpread,
    txn.amount,
    v.liquidationPayment   // ← perp oracle price, NOT spot liquidation payment
);
``` [2](#0-1) 

For non-spread spot and perp liquidations, `v.liquidationPayment` is set once and never overwritten, so the event is correct in those paths. The inconsistency is exclusive to `isEncodedSpread == true`.

The `Liquidation` event is defined as:

```solidity
event Liquidation(
    bytes32 indexed liquidatorSubaccount,
    bytes32 indexed liquidateeSubaccount,
    uint32 productId,
    bool isEncodedSpread,
    int128 amount,
    int128 amountQuote
);
``` [3](#0-2) 

For spread liquidations, `amountQuote` should represent the quote transferred to the liquidatee for the spot leg (`v.liquidationPriceX18.mul(txn.amount)`), consistent with how non-spread paths populate it. Instead it emits `v.oraclePriceX18Perp.mul(txn.amount)`.

---

### Impact Explanation
Any off-chain system (indexer, risk dashboard, insurance-fund tracker, alert bot) that reads the `Liquidation` event and uses `amountQuote` to reconstruct the quote flow of a spread liquidation will receive a value that does not match the actual quote transferred on-chain.
