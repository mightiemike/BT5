Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the actual swapper, allowing any user to bypass the allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is always `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, the router is `msg.sender` inside the pool, so the allowlist check resolves to `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. Any pool admin who allowlists the router to enable standard periphery UX inadvertently grants every address on the network the ability to bypass the allowlist gate.

## Finding Description
**Root cause — `pool.swap()` passes `msg.sender` (the router) as `sender` to extensions:**

In `MetricOmmPool.swap` (lines 230–240), `_beforeSwap` is called with `msg.sender` as the first argument:
```solidity
_beforeSwap(
  msg.sender,   // whoever called pool.swap() — the router, not the user
  recipient,
  ...
);
```
When a user calls any of the four public entry points on `MetricOmmSimpleRouter` (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`), the router itself calls `IMetricOmmPoolActions(pool).swap(...)` directly (e.g., lines 72–80, 104–112, 136–137, 165–181). Therefore `msg.sender` inside the pool is always the router address.

**`SwapAllowlistExtension.beforeSwap` checks `sender`, which is the router:**

```solidity
// SwapAllowlistExtension.sol lines 31–41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
      revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```
`msg.sender` is the pool (correct). `sender` is the router (wrong — should be the original user). The check resolves to `allowedSwapper[pool][router]`.

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension`.
2. Admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for legitimate users.
3. Admin calls `setAllowedToSwap(pool, alice, true)` and intentionally does NOT allowlist `bob`.
4. `bob` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, ...)` — `msg.sender` inside pool = router.
6. Pool calls `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
7. Extension checks `allowedSwapper[pool][router] == true` → passes.
8. Bob's swap executes despite not being on the allowlist.

**Existing guards are insufficient:** There is no secondary check on the original caller. The `extensionData` field is passed through but `SwapAllowlistExtension` ignores it entirely (the `bytes calldata` parameter is unnamed and unused). The `recipient` field is not checked. No on-chain mechanism recovers the original `msg.sender` of the router call.

## Impact Explanation
`SwapAllowlistExtension` is the sole mechanism for restricting which addresses may trade on a pool. Once the router is allowlisted — the only operational configuration that allows legitimate allowlisted users to use the standard periphery — the restriction is completely void for all four swap paths. Any unprivileged address can swap by routing through `MetricOmmSimpleRouter`, exposing LP assets to trades from counterparties the admin explicitly intended to exclude. For institutional-only, KYC-gated, or whitelist-only pools, this constitutes a direct breach of the pool's intended access-control invariant and can result in direct loss of LP principal through adversarial swaps from excluded addresses.

## Likelihood Explanation
The trigger requires no special privilege. Any user who knows the pool uses `SwapAllowlistExtension` and that the router is allowlisted can exploit this immediately. The pool admin is virtually certain to allowlist the router because without it, even legitimately allowlisted users cannot use the standard periphery. The misconfiguration is therefore the expected operational state, not an edge case. The attack is repeatable on every swap.

## Recommendation
The pool must receive the original user's address, not the router's. Two complementary fixes:

1. **Pass the real caller through `extensionData`**: The router encodes the original `msg.sender` into `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap` decodes and checks that address instead of `sender`. A known prefix/tag should be used so a malicious user cannot spoof the field by crafting their own `extensionData`.

2. **Alternatively, check `recipient` instead of `sender`**: Only viable for single-hop swaps where `recipient` is always the economic beneficiary; not safe for multi-hop paths where intermediate recipients are the router itself.

The cleanest fix is option 1.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, router, true)   // to enable router UX
  - Pool admin calls setAllowedToSwap(pool, alice, true)    // alice is the only intended user
  - Pool admin does NOT allowlist bob

Attack:
  - bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(recipient, ...) — msg.sender in pool = router
  - Pool calls SwapAllowlistExtension.beforeSwap(sender=router, ...)
  - Extension checks allowedSwapper[pool][router] == true  ✓
  - Bob's swap executes successfully despite not being on the allowlist

Result: Bob receives pool output tokens; LP providers bear swap exposure
        from an address the admin intended to exclude.
```

Foundry test outline:
1. Deploy `SwapAllowlistExtension`, pool, and `MetricOmmSimpleRouter`.
2. Configure pool with the extension.
3. `setAllowedToSwap(pool, router, true)` and `setAllowedToSwap(pool, alice, true)`.
4. `vm.prank(bob); router.exactInputSingle(...)` — assert it succeeds (no `NotAllowedToSwap` revert).
5. `vm.prank(bob); pool.swap(...)` directly — assert it reverts with `NotAllowedToSwap`.
6. Step 4 succeeding while step 5 reverts demonstrates the bypass.
