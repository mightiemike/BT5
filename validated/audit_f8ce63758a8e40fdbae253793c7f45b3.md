### Title
Hash Domain Divergence in `ResourceBoundsMapping → ValidResourceBounds` Conversion Causes Wrong Simulation/Fee-Estimation Hash for V3 Transactions with Zero L1DataGas/L2Gas - (File: crates/apollo_rpc/src/v0_8/transaction.rs)

### Summary

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in the RPC layer applies a value-based heuristic that silently selects a different `ValidResourceBounds` variant — and therefore a different transaction hash preimage — than the gateway admission path uses for the same V3 transaction. A V3 transaction with `l1_data_gas = 0` and `l2_gas = 0` is accepted by the gateway under the `AllResources` hash domain (which includes `L1_DATA_GAS` in the Poseidon preimage), but the same transaction submitted to `starknet_estimateFee` or `starknet_simulateTransactions` is hashed under the `L1Gas` domain (which omits `L1_DATA_GAS`). The two hashes are structurally different, so signature validation inside the simulation will fail for a transaction that the sequencer itself accepted.

### Finding Description

**Gateway admission path** (`starknet_addInvokeTransaction`):

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds` directly. [1](#0-0) 

The `InvokeTransactionV3Trait` implementation for `InternalRpcInvokeTransactionV3` unconditionally wraps this as `ValidResourceBounds::AllResources(self.resource_bounds)`. [2](#0-1) 

**RPC execution path** (`starknet_estimateFee` / `starknet_simulateTransactions`):

Both endpoints receive `Vec<BroadcastedTransaction>` and immediately convert each entry to an `ExecutableTransactionInput`. [3](#0-2) 

`BroadcastedDeclareV3Transaction` (and the analogous invoke/deploy-account broadcasted structs) carry `resource_bounds: ResourceBoundsMapping`. [4](#0-3) 

The conversion from `ResourceBoundsMapping` to `ValidResourceBounds` uses a value-based heuristic:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← L1_DATA_GAS omitted from hash
        } else {
            Self::AllResources(AllResourceBounds { ... })  // ← L1_DATA_GAS included
        }
    }
}
``` [5](#0-4) 

**Hash preimage divergence**:

`get_tip_resource_bounds_hash` conditionally appends `L1_DATA_GAS` to the Poseidon preimage only for `AllResources`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // no L1_DATA_GAS
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [6](#0-5) 

For a transaction where `l1_data_gas = 0` and `l2_gas = 0`:

| Path | `ValidResourceBounds` variant | `L1_DATA_GAS` in hash preimage | Resulting hash |
|---|---|---|---|
| `starknet_addInvokeTransaction` | `AllResources` | **yes** (value = 0) | H₁ |
| `starknet_simulateTransactions` | `L1Gas` | **no** | H₂ ≠ H₁ |

The user signs H₁. The simulation computes H₂ and passes it to `__validate__`. The account contract rejects the signature.

### Impact Explanation

`starknet_simulateTransactions` and `starknet_estimateFee` return an authoritative-looking `TRANSACTION_EXECUTION_ERROR` (signature validation failure) for a transaction that the sequencer's own gateway accepted and will execute correctly. This matches the allowed impact: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

A developer relying on simulation to pre-flight a transaction before submission will observe a spurious failure and may incorrectly conclude their transaction or account contract is broken. Conversely, a transaction that would actually fail on-chain could pass simulation if the hash collision happens to satisfy a different validation path.

### Likelihood Explanation

The trigger condition — a V3 transaction with `l1_data_gas = 0` and `l2_gas = 0` — is reachable by any unprivileged user. Wallets and SDKs that set only `l1_gas` (leaving the other two bounds at zero) will hit this path. No special privileges, malformed bytes, or peer-level access are required; a normal JSON-RPC call suffices.

### Recommendation

Remove the value-based heuristic from `From<ResourceBoundsMapping> for ValidResourceBounds`. For V3 transactions, always produce `ValidResourceBounds::AllResources`, regardless of whether `l1_data_gas` and `l2_gas` are zero. The `L1Gas` variant should only be constructed for transactions that were originally serialized as pre-0.13.3 V3 transactions (identified by a version tag or explicit schema field, not by inspecting runtime values).

Alternatively, align the `BroadcastedTransaction` structs to use `AllResourceBounds` directly (as `RpcInvokeTransactionV3` already does), eliminating the `ResourceBoundsMapping → ValidResourceBounds` conversion from the simulation path entirely.
