Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the originating user, allowing any unprivileged user to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` mediates a swap, `sender` resolves to the router's address, not the originating EOA. A pool admin who allowlists the router to enable router-mediated access for their allowlisted users simultaneously grants every on-chain user the ability to swap through that pool, completely defeating the allowlist.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` (the direct caller) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension via `_callExtensionsInOrder`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the extension's caller) and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` of that call: [4](#0-3) 

The same pattern applies to `exactInput` and `exactOutputSingle`: [5](#0-4) [6](#0-5) 

The allowlist check therefore resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. The extension's own NatSpec states it "Gates `swap` by swapper address, per pool," but the implementation gates by the immediate caller of `pool.swap()`, which is the router for all router-mediated flows: [7](#0-6) 

No existing guard in the extension inspects the originating EOA or decodes it from `extensionData`. The `extensionData` field is passed through but never read by `SwapAllowlistExtension`.

## Impact Explanation

Any user can bypass a pool's swap allowlist by calling `MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, or `exactOutputSingle` on a pool that has `SwapAllowlistExtension` configured and the router address allowlisted. Unauthorized swappers gain full access to the pool's liquidity, enabling them to extract value from LPs through swaps the pool admin explicitly intended to block. This is a direct Admin-boundary break: an unprivileged path (the public router) bypasses a configured access-control guard, causing direct loss of LP assets through unauthorized swaps.

## Likelihood Explanation

The scenario requires two normal operational steps: (1) deploying a pool with `SwapAllowlistExtension` as a configured extension â€” a deliberate, documented feature â€” and (2) the pool admin adding the router to the allowlist, which is the natural and expected action to let their allowlisted users access the pool via the periphery router. No malicious setup, non-standard tokens, or privileged attacker capability is required. Any user who observes the on-chain allowlist state can immediately exploit it by calling the public router.

## Recommendation

The extension must gate the originating user, not the intermediate contract. Two sound approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. The pool's `nonReentrant` guard prevents the router from being re-entered to forge this value.

2. **Router-aware check**: When `sender` is a known router address, the extension reads the real user from a router-provided field in `extensionData` and checks that address against `allowedSwapper` instead of the router address.

In either case, `allowedSwapper[pool][router]` must never be the terminal check when the router is acting on behalf of an arbitrary user.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` and calls `setAllowedToSwap(pool, userA, true)` to restrict swaps to `userA`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` so that `userA` can use the router.
3. `userB` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. The router calls `pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)` â€” `msg.sender` of this call is the router address.
5. The pool calls `_beforeSwap(router, ...)`, which calls `SwapAllowlistExtension.beforeSwap(router, ...)`.
6. The extension evaluates `allowedSwapper[pool][router]` â†’ `true` â†’ no revert.
7. `userB`'s swap executes successfully despite never being allowlisted, draining LP funds the pool admin intended to protect.
