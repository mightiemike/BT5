### Title
Unbounded `while` Loop in `reorg_chain` Can Permanently Strand the Contract on a Lower-Work Chain — (`File: contract/src/lib.rs`)

### Summary
The `reorg_chain` function contains an unbounded `while` loop that walks every fork block from the fork tip back to the common ancestor. Because fork blocks are never garbage-collected, a sufficiently long fork causes the loop to exhaust NEAR's per-transaction gas limit. The transaction is rolled back, the reorg never completes, and the contract is permanently stuck tracking the lower-work chain. Any subsequent call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` then verifies transactions against a chain that Bitcoin's fork-choice rule has already invalidated.

### Finding Description

`submit_block_header_inner` stores a fork header and, when the fork's accumulated `chain_work` exceeds the main chain's, immediately calls `reorg_chain`: [1](#0-0) 

`reorg_chain` contains a `while` loop that iterates once per fork block between the fork tip and the common ancestor: [2](#0-1) 

Each iteration performs at minimum five storage operations (`contains_key`, two `insert`s, one `get`, and conditionally a `remove`). NEAR storage I/O is the dominant gas consumer; with a 300 TGas per-transaction cap, the loop can exhaust gas after a relatively small number of iterations.

The critical asymmetry is that **fork blocks are never garbage-collected**. The GC mechanism (`run_mainchain_gc`) explicitly targets only mainchain blocks: [3](#0-2) 

Fork blocks accumulate in `headers_pool` without bound across many separate `submit_blocks` transactions. There is no cap on fork depth anywhere in the contract.

When the triggering `submit_blocks` call runs out of gas, NEAR rolls back the entire transaction. `store_fork_header` for the tip block is undone, but all previously submitted fork blocks remain in `headers_pool`. Every future attempt to submit the same (or any equally long) fork tip will re-enter the same unbounded loop and fail identically. The contract is permanently unable to reorg to the heavier chain.

### Impact Explanation

The contract's stated purpose is to track the canonical (highest-work) Bitcoin-family chain and provide SPV proofs against it. If `reorg_chain` can never complete, `mainchain_tip_blockhash` remains pinned to the lower-work chain: [4](#0-3) 

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` resolve the target block exclusively through `mainchain_header_to_height`, which is never updated: [5](#0-4) 

Transactions confirmed only on the reorganized-away (lower-work) chain will continue to return `true`. Downstream contracts or bridges that consume these proofs will accept double-spent or reversed transactions as valid. This is a direct corruption of the contract's canonical-chain mapping and proof-result integrity.

### Likelihood Explanation

**Low.** The attacker must be a registered trusted relayer (the `#[trusted_relayer]` gate on `submit_blocks`): [6](#0-5) 

With PoW verification enabled (production default), building a fork whose accumulated work exceeds the main chain requires substantial hash power — approaching a 51%-attack level for Bitcoin. For lower-difficulty chains (Dogecoin, Litecoin, Zcash) the bar is lower. The fork blocks can be submitted incrementally across many transactions; only the final tip submission triggers the unbounded loop.

### Recommendation

Introduce a maximum fork depth constant and enforce it at fork-block submission time. If the fork depth from the common ancestor exceeds the limit, reject the block rather than storing it. Alternatively, restructure `reorg_chain` to accept a `batch_size` parameter (mirroring `run_mainchain_gc`) so the reorg can be completed across multiple transactions, each bounded in gas.

### Proof of Concept

1. Trusted relayer submits N fork blocks (each extending the previous fork block, all branching from some historical main-chain block). Each call to `submit_blocks` succeeds; each fork block is stored via `store_fork_header` into `headers_pool`.
2. Relayer submits the (N+1)-th fork block whose cumulative `chain_work` exceeds `total_main_chain_chainwork`.
3. `submit_block_header_inner` calls `store_fork_header` then `reorg_chain`.
4. The `while` loop at line 616 iterates N+1 times, each performing ≥5 storage operations. For sufficiently large N, gas is exhausted before the loop terminates.
5. NEAR rolls back the transaction; the fork tip is not stored, but the N prior fork blocks remain in `headers_pool`.
6. Steps 2–5 repeat identically for every future attempt; the reorg can never complete.
7. `mainchain_tip_blockhash` remains on the lower-work chain; `verify_transaction_inclusion` returns `true` for transactions on the invalidated chain.
