Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Enabling Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` of the pool call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. If the router is allowlisted, every user bypasses the curation policy; if it is not allowlisted, individually allowlisted users cannot use the router at all.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` verbatim as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← always the immediate pool caller
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` is allowlisted:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly, making the router `msg.sender` to the pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,   // recipient = user-supplied
        ...                 // msg.sender to pool = router contract
    );
```

So when alice routes through the router: `sender = router`, and the check becomes `allowedSwapper[pool][router]`. If the router is allowlisted (a natural admin action for supporting the standard periphery), alice's swap passes regardless of her individual allowlist status.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly uses `owner` (the economically relevant actor) rather than `sender`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

The swap allowlist lacks this equivalent pattern. The existing integration test `FullMetricExtension.t.sol` exercises the allowlist only via direct pool calls through `TestCaller`, so the router-mediated bypass path is entirely untested.

## Impact Explanation
**Scenario A — Router allowlisted (bypass):** A pool admin allowlists the router as trusted periphery. Every user, including those not individually allowlisted, can trade on the curated pool by routing through `MetricOmmSimpleRouter`. The per-user allowlist is completely nullified — disallowed users (e.g., non-KYC'd addresses) trade on a pool designed to restrict access. This is a direct curation failure and broken core pool functionality.

**Scenario B — Router not allowlisted (broken functionality):** Individually allowlisted users cannot use the router because the extension sees `allowedSwapper[pool][router] == false` and reverts `NotAllowedToSwap`. Allowlisted users are forced to call the pool directly, breaking the standard swap UX and any multi-hop routing path.

Both scenarios represent a broken invariant: the allowlist guard does not enforce the same policy regardless of which supported public entrypoint reaches the pool.

## Likelihood Explanation
- Allowlisting the router is a natural and expected admin action for any pool that wants to support the standard periphery while restricting direct pool access.
- `MetricOmmSimpleRouter` is a deployed, public, permissionless contract — any user can call it.
- No special privileges, no malicious setup, and no non-standard tokens are required.
- The bypass is triggered in a single transaction by any unprivileged user with a valid swap amount.
- The `generate_scanned_questions.py` audit target explicitly identifies this exact vector: "swap allowlist gate" with entrypoint `MetricOmmSimpleRouter.exact*` and validation focus on whether "router-mediated swaps preserve that identity."

## Recommendation
The `beforeSwap` hook must gate the end user, not the immediate pool caller. Two sound approaches:

1. **Pass the original initiator through `extensionData`.** The router encodes `msg.sender` into `extensionData`; the extension decodes it. `_beforeSwap` already forwards `extensionData` unchanged, so no core changes are needed.

2. **Check `recipient` instead of `sender`.** When the swap output goes to the user, `recipient` is the economically relevant actor. This requires no encoding changes but is less clean for cases where `recipient` differs from the initiator.

The `DepositAllowlistExtension` already demonstrates the correct pattern by checking `owner` rather than `sender` for the deposit path.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][router] = true   // admin allowlists router as trusted periphery
  allowedSwapper[pool][alice]  = false  // alice is NOT individually allowlisted

Attack (alice bypasses the allowlist):
  alice calls MetricOmmSimpleRouter.exactInputSingle({pool, recipient: alice, ...})
    → router calls pool.swap(recipient=alice, ...)   // msg.sender to pool = router
    → pool calls _beforeSwap(sender=router, ...)
    → SwapAllowlistExtension.beforeSwap(sender=router, ...)
    → checks allowedSwapper[pool][router] == true    ✓ PASSES
    → alice's swap executes on the curated pool

Expected: revert NotAllowedToSwap
Actual:   swap succeeds — allowlist bypassed

Foundry test plan:
  1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  2. Deploy MetricOmmSimpleRouter.
  3. setAllowedToSwap(pool, router, true); setAllowedToSwap(pool, alice, false).
  4. vm.prank(alice); router.exactInputSingle({pool, ...}).
  5. Assert swap succeeds (no revert) — demonstrating bypass.
  6. Assert direct pool call from alice reverts NotAllowedToSwap — confirming the allowlist works for direct calls but not router-mediated ones.
```
