### Title
Reorg Vulnerability in `DirectDepositV1` Deployment Allows Tokens to Be Credited to Wrong Subaccount — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1` (DDA) contracts are deployed via the `new` keyword (CREATE opcode), making their on-chain address a function of the deployer's address and nonce. When multiple DDAs are deployed in the same block and a chain reorg reorders those transactions, the address-to-subaccount binding emitted in `DirectDepositV1Created` becomes stale. Any tokens subsequently sent to the original address are credited to the wrong subaccount via `creditDeposit()`, causing irreversible fund loss.

---

### Finding Description

`DirectDepositV1` is deployed with a plain `new` constructor call: [1](#0-0) 

The constructor hardcodes `subaccount` and immediately emits:

```solidity
emit DirectDepositV1Created(version(), subaccount, address(this));
``` [2](#0-1) 

Off-chain clients and integrators rely on this event to learn which DDA address maps to which subaccount. The DDA address is determined solely by `deployer_address` and `deployer_nonce` (CREATE semantics). No CREATE2 salt encoding the subaccount is used.

When `creditDeposit()` is later called, it unconditionally deposits to the `subaccount` baked into the constructor:

```solidity
endpoint.depositCollateralWithReferral(
    subaccount,   // hardcoded — cannot be overridden
    productId,
    uint128(balance),
    "-1"
);
``` [3](#0-2) 

**Reorg scenario:**

1. Block N: DDA-A (for `subaccount-A`) is deployed at `address1`; DDA-B (for `subaccount-B`) is deployed at `address2`. Events are emitted accordingly.
2. A reorg drops block N and replaces it with block N′ where the two deployment transactions are ordered in reverse.
3. In block N′: DDA-B lands at `address1` (nonce N), DDA-A lands at `address2` (nonce N+1).
4. A user who observed the original event sends tokens to `address1` expecting DDA-A.
5. `creditDeposit()` is called on `address1`; it deposits to `subaccount-B` — not `subaccount-A`.
6. The user's funds are permanently credited to the wrong subaccount.

---

### Impact Explanation

Tokens transferred to a DDA address are irrevocably credited to whichever `subaccount` is hardcoded in the DDA that occupies that address post-reorg. The original depositor loses their funds with no on-chain recourse, because `depositCollateralWithReferral` accepts any `bytes32 subaccount` and the DDA's `subaccount` field is immutable after construction. [4](#0-3) 

---

### Likelihood Explanation

The Nado protocol targets EVM-compatible chains. Chains with probabilistic finality (e.g., those without instant finality) are subject to short reorgs. The preconditions are:

- Two or more DDA deployments originating from the same deployer address (e.g., a factory or the protocol deployer) land in the same block.
- A reorg of depth ≥ 1 reorders those transactions.
- At least one user acts on the pre-reorg event before the reorg is detected.

All three conditions are realistic in production. The `ContractOwner.sol` file (19 references to `DirectDepositV1`) suggests a factory-style deployment pattern where multiple DDAs may be created in a single block, increasing exposure.

---

### Recommendation

Replace the plain `new` deployment with CREATE2 using a salt derived from the target `subaccount`. This makes the DDA address deterministic and independent of deployment ordering:

```solidity
bytes32 salt = keccak256(abi.encodePacked(_subaccount));
DirectDepositV1 dda = new DirectDepositV1{salt: salt}(
    _endpoint, _spotEngine, _subaccount, _wrappedNative
);
```

With CREATE2, the address is `f(deployer, salt, initcode_hash)`. Because `salt` encodes `_subaccount`, no reorg can place a different subaccount's DDA at the same address.

### Proof of Concept

```
Block N (observed by user):
  tx1: deploy DDA-A(subaccount=Alice) → address1  [nonce=5]
  tx2: deploy DDA-B(subaccount=Attacker) → address2 [nonce=6]
  Event: DirectDepositV1Created(1, Alice, address1)

Alice reads event → sends 10,000 USDC to address1

Reorg: block N replaced by block N′:
  tx2 first: deploy DDA-B(subaccount=Attacker) → address1 [nonce=5]
  tx1 second: deploy DDA-A(subaccount=Alice) → address2 [nonce=6]

Block N′ confirmed. address1 now holds DDA-B (Attacker's subaccount).

Attacker calls creditDeposit() on address1:
  → endpoint.depositCollateralWithReferral(Attacker_subaccount, USDC_productId, 10000e6, "-1")
  → Alice's 10,000 USDC credited to Attacker's subaccount

Attacker submits WithdrawCollateral signed tx → receives 10,000 USDC.
Alice's funds are gone.
``` [1](#0-0) [4](#0-3)
