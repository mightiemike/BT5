### Title
Division by Zero in `PerpEngine.socializeSubaccount` Blocks Liquidation Finalization When `openInterest` Is Zero — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when attempting to spread a residual negative `vQuoteBalance` across remaining participants. If `openInterest` is zero at the time of the call — a reachable state when all perp positions in a product have been closed — the `MathSD21x18.div` call reverts with `"DBZ"`, permanently blocking liquidation finalization for the affected subaccount and leaving unresolvable bad debt in the protocol.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after insurance is applied to a negative `vQuoteBalance`, any remaining deficit is socialized via:

```solidity
int128 fundingPerShare = -balance.vQuoteBalance.div(
    state.openInterest
);
``` [1](#0-0) 

`MathSD21x18.div` enforces `require(y != 0, ERR_DIV_BY_ZERO)`: [2](#0-1) 

There is no guard checking `state.openInterest != 0` before this division.

`state.openInterest` is the sum of absolute values of all open positions across all participants in a given perp product. It is zero when every participant has `balance.amount == 0`. A subaccount can simultaneously hold `amount == 0` and `vQuoteBalance < 0` — the normal result of closing a losing position — because `vQuoteBalance` is only updated when an explicit `vQuoteDelta` is passed to `_updateBalance`: [3](#0-2) 

`socializeSubaccount` is called unconditionally inside `_finalizeSubaccount`: [4](#0-3) 

`_finalizeSubaccount` is the only code path that can resolve a subaccount whose perp positions are all zero but whose `vQuoteBalance` is negative — the exact precondition for finalization requires `balance.amount == 0` for every perp product: [5](#0-4) 

---

### Impact Explanation

When the revert fires, `liquidateSubaccountImpl` (via `delegatecall` to `ClearinghouseLiq`) reverts entirely: [6](#0-5) 

The affected subaccount cannot be finalized. Because all perp positions are already zero, no other liquidation step can reduce the negative `vQuoteBalance` — the only resolution path is finalization. The bad debt is permanently frozen: the insurance fund is never debited, the loss is never socialized, and the subaccount remains in an unhealthy state indefinitely. This is a concrete solvency accounting corruption, not merely a gas-only DoS.

---

### Likelihood Explanation

The trigger requires three simultaneous conditions:

1. The liquidatee has `amount == 0` and `vQuoteBalance < 0` in at least one perp product (a normal post-trade state after closing a losing position).
2. `openInterest == 0` for that product (all other participants have also closed their positions — realistic in thin or newly listed markets).
3. Insurance is insufficient to fully cover the negative `vQuoteBalance` (so the socialization branch is reached).

An attacker can deliberately engineer this: open a position in a thin perp market as the sole or dominant participant, close it at a realized loss (paying fees to the sequencer), and allow the subaccount to become unhealthy. Because the attacker controls when they close and can ensure they are the only open-interest holder, all three conditions are satisfiable with a single funded subaccount. The cost to the attacker is the realized loss plus fees, which may be small relative to the bad debt left unresolvable.

---

### Recommendation

Add a zero-check for `state.openInterest` before the division in `PerpEngine.socializeSubaccount`. If `openInterest` is zero, there are no other participants to absorb the loss; the remaining deficit should be treated as fully covered by insurance (or flagged for protocol-level handling), rather than causing a revert:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; treat as unrecoverable
        // or handle via insurance top-up / protocol reserve.
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

---

### Proof of Concept

1. A thin perp market (e.g., product ID `2`) has no open positions (`openInterest == 0`).
2. Attacker opens a long position of size `S` via `matchOrders` — `openInterest` becomes `S`.
3. Attacker closes the position at a loss via `matchOrders` — `amount` returns to `0`, `vQuoteBalance` is now `-L` (negative), `openInterest` returns to `0`.
4. Attacker's subaccount health drops below maintenance threshold (negative `vQuoteBalance` contributes directly to health via `_calculateProductHealth`). [7](#0-6) 
5. A liquidator calls `liquidateSubaccount` with `productId = type(uint32).max` to finalize the subaccount.
6. `_finalizeSubaccount` passes the `amount == 0` check for all perps, then calls `perpEngine.socializeSubaccount`.
7. Inside `socializeSubaccount`, insurance covers part of `-L` but not all (insurance < `L`). `balance.vQuoteBalance` remains negative.
8. Execution reaches `fundingPerShare = -balance.vQuoteBalance.div(state.openInterest)` with `state.openInterest == 0`.
9. `MathSD21x18.div` reverts with `"DBZ"`. The entire liquidation transaction reverts. The subaccount is permanently unliquidatable. Bad debt is frozen.
