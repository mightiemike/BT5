Audit Report

## Title
`SwapAllowlistExtension` receives router address as `sender`, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[pool][sender]`. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `swap` function receives `msg.sender = router`, so the extension checks `allowedSwapper[pool][router]` instead of the originating user's address. Any pool admin who allowlists the router (required for legitimate users to use it) simultaneously grants every address in existence the ability to swap through that pool, defeating the allowlist entirely.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension via `_callExtensionsInOrder`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `sender` (the first parameter) against `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly without forwarding the originating user's address: [4](#0-3) 

At that point `msg.sender` inside the pool is the router contract, so the extension receives `sender = address(MetricOmmSimpleRouter)`. The router does store the originating `msg.sender` in transient storage via `_setNextCallbackContext`, but this is only used for the payment callback — it is never forwarded to extensions. The `extensionData` passed to the pool comes from the user's params unchanged; the extension does not decode any real-sender field from it.

The same structural mismatch applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

This creates an irresolvable dilemma for any pool admin deploying `SwapAllowlistExtension`:

| Router allowlist state | Effect |
|---|---|
| Router **not** allowlisted | All router-mediated swaps revert — even for users on the allowlist |
| Router **allowlisted** | Every address can swap through the router, defeating the allowlist entirely |

No existing guard in `SwapAllowlistExtension` or `MetricOmmSimpleRouter` resolves this mismatch. The `allowAllSwappers` escape hatch only makes the bypass unconditional. [6](#0-5) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., institutional market makers, KYC'd addresses, or whitelisted LPs) can be fully bypassed by any unprivileged user calling `MetricOmmSimpleRouter`. The unauthorized user executes swaps at live oracle prices against LP capital deposited under the assumption that only vetted counterparties would trade. This constitutes direct loss of LP principal through unauthorized extraction at oracle-fair prices on a pool whose access policy was supposed to prevent it — a broken core pool functionality causing loss of funds.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool that (a) deploys `SwapAllowlistExtension` and (b) needs to support router-mediated swaps for its allowed users is immediately vulnerable the moment the admin allowlists the router. The attacker requires no special role, no privileged setup, and no tokens beyond the swap input — a single `exactInputSingle` call suffices. The condition (router allowlisted) is a necessary operational step for any pool that intends to support router users, making exploitation automatic in any realistic deployment.

## Recommendation
The router must forward the originating user's address so the extension can gate on the correct identity. Two viable approaches:

1. **Encode the real sender in `extensionData`**: The router appends `abi.encode(msg.sender)` to `extensionData` before forwarding it to the pool. `SwapAllowlistExtension.beforeSwap` decodes and checks that address instead of `sender`. The extension must verify `msg.sender` (the pool) is calling from a trusted router to prevent spoofing.

2. **Add a `realSender` field to the swap interface**: The pool passes an explicit `realSender` to extensions, populated by the router from its own `msg.sender` and by direct callers from their own `msg.sender`. This requires a protocol-level interface change.

Either fix must be authenticated to prevent an attacker from spoofing the real-sender field.

## Proof of Concept
```
Setup:
  Pool P has SwapAllowlistExtension E configured.
  Admin sets allowedSwapper[P][alice] = true   // alice is the only allowed swapper
  Admin sets allowedSwapper[P][router] = true  // necessary so alice can use the router

Attack:
  bob (not on allowlist) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})

Execution path:
  router.exactInputSingle()                          // msg.sender = bob
    -> pool.swap(recipient, ...)                     // msg.sender = router inside pool
      -> _beforeSwap(sender=router, ...)
        -> SwapAllowlistExtension.beforeSwap(sender=router, ...)
          -> allowedSwapper[pool][router] == true ✓  // passes — bob is never checked
      -> swap executes against LP funds

Result: bob swaps successfully on a pool that was supposed to block him.
```

If the admin does NOT allowlist the router to prevent this, alice also cannot use the router — the allowlist becomes router-incompatible, breaking core swap functionality for all legitimate users.
