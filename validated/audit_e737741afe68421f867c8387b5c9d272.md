### Title
Linked Signer Can Unilaterally Reassign Signing Authority to Any Arbitrary Address — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` fast-path transaction in `EndpointTx.processTransactionImpl` passes `allowLinkedSigner = true` to `validateSignedTx`. This means the **currently linked signer** can sign a new `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address — including an attacker-controlled one — without the subaccount owner's knowledge or consent. The subaccount owner can grant signing authority (link a signer), but has no protection against that signer silently transferring the authority to a third party.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which in turn calls `verifier.validateSignature` passing `getLinkedSigner(sender)` as the permitted alternate signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts a signature from either the subaccount owner address **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

Therefore, the currently linked signer can produce a valid `SignedLinkSigner` transaction that sets `linkedSigners[victim_subaccount]` to any address of their choosing. The subaccount owner is not consulted.

By contrast, the **slow-mode** `LinkSigner` path uses `validateSender`, which enforces `msg.sender == address(uint160(bytes20(txn.sender)))` — only the subaccount owner's EOA can submit it:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The fast path has no equivalent owner-only guard, creating an asymmetric permission: the owner can grant authority, but the grantee can silently reassign it.

---

### Impact Explanation

Once a subaccount owner links a signer, the linked signer can:

1. Sign a `LinkSigner` transaction naming any attacker-controlled address as the new signer.
2. After the sequencer processes it, `linkedSigners[victim_subaccount]` points to the attacker.
3. The attacker can now sign any `allowLinkedSigner = true` transaction on behalf of the victim, including `WithdrawCollateral` and `WithdrawCollateralV2`. [5](#0-4) 

This results in **complete, permanent loss of collateral** for the victim subaccount. The victim has no on-chain mechanism to detect or prevent the reassignment before the attacker drains funds.

---

### Likelihood Explanation

Any user who has ever linked a signer is exposed. The attack requires only:
- Knowledge of the victim's current nonce (publicly readable via `Endpoint.getNonce`).
- A valid ECDSA signature from the current linked signer key.

A malicious linked signer (e.g., a compromised API key, a rogue trading bot, or a third-party integration) can execute this silently in a single sequencer batch. No privileged access, governance capture, or admin keys are required. [6](#0-5) 

---

### Recommendation

Change the `LinkSigner` fast-path handler to pass `allowLinkedSigner = false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the slow-mode path's `validateSender` restriction and enforces the invariant that only the subaccount owner can modify their own linked signer.

---

### Proof of Concept

1. Alice owns subaccount `alice_sub` and links Bob (`bob_addr`) as her linked signer via a slow-mode `LinkSigner` tx. `linkedSigners[alice_sub] = bob_addr`.
2. Bob (or an attacker who has compromised Bob's key) reads `Endpoint.getNonce(alice_addr)` to get the current nonce `N`.
3. Bob constructs `SignedLinkSigner { tx: { sender: alice_sub, signer: attacker_addr, nonce: N }, signature: bob_sig }`.
4. Bob submits this to the sequencer.
5. The sequencer calls `processTransactionImpl` → `LinkSigner` branch → `validateSignedTx(..., true)`.
6. `verifier.validateSignature` recovers `bob_addr` from the signature, which equals `getLinkedSigner(alice_sub)` → validation passes.
7. `linkedSigners[alice_sub]` is overwritten with `attacker_addr`.
8. Attacker signs `WithdrawCollateral { sender: alice_sub, productId: QUOTE, amount: MAX }` and submits it.
9. `validateSignedTx(..., true)` recovers `attacker_addr` == `getLinkedSigner(alice_sub)` → passes.
10. Alice's entire collateral balance is withdrawn to the attacker. [1](#0-0) [7](#0-6)
