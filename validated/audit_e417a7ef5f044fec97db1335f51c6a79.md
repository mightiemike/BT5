### Title
Stale `EpochInfo` Fallback in `finalize_epoch` Silently Discards Computed `shard_layout` and `last_resharding` on Validator-Threshold Failure During Dynamic Resharding — (`File: chain/epoch-manager/src/lib.rs`)

---

### Summary

When `finalize_epoch` encounters a `ThresholdError` or `NotEnoughValidators` error from `proposals_to_epoch_info`, it falls back to cloning the **previous** epoch's `EpochInfo` (`next_epoch_info`) and incrementing only the epoch height. This stale clone carries the old `shard_layout` and old `last_resharding` from epoch T+1, silently discarding the freshly derived `next_next_shard_layout` (a new `ShardLayoutV3` with the split) and the newly computed `last_resharding` value. The saved epoch T+2 `EpochInfo` then permanently misrepresents both the shard layout and the resharding cooldown state, while the committed block header for the last block of epoch T still carries `shard_split = Some(...)`.

---

### Finding Description

In `finalize_epoch` (`chain/epoch-manager/src/lib.rs`), the function first correctly derives the new shard layout and resharding metadata:

```rust
let next_next_shard_layout = self.next_next_shard_layout(..., block_info)?;

let has_same_shard_layout = next_next_shard_layout == next_shard_layout;
let last_resharding = (!has_same_shard_layout)
    .then(|| next_epoch_info.epoch_height() + 1)
    .or_else(|| next_epoch_info.last_resharding());
``` [1](#0-0) 

These values are then passed to `proposals_to_epoch_info`:

```rust
let next_next_epoch_info = match proposals_to_epoch_info(
    &next_next_epoch_config, rng_seed, &next_epoch_info,
    all_proposals, validator_kickout, validator_reward, minted_amount,
    next_next_epoch_version,
    next_next_shard_layout.clone(),   // ← new V3 split layout
    &strategy,
    last_resharding,                  // ← Some(epoch_height + 1)
) {
    Ok(next_next_epoch_info) => next_next_epoch_info,
    Err(EpochError::ThresholdError { .. }) => {
        let mut epoch_info = EpochInfo::clone(&next_epoch_info); // ← STALE COPY
        *epoch_info.epoch_height_mut() += 1;
        epoch_info  // ← shard_layout and last_resharding NOT updated
    }
    Err(EpochError::NotEnoughValidators { .. }) => {
        let mut epoch_info = EpochInfo::clone(&next_epoch_info); // ← STALE COPY
        *epoch_info.epoch_height_mut() += 1;
        epoch_info  // ← shard_layout and last_resharding NOT updated
    }
    ...
};
``` [2](#0-1) 

The fallback `EpochInfo` is a clone of `next_epoch_info` (epoch T+1), which carries:
- `shard_layout` = the **old** layout from epoch T+1 (not `next_next_shard_layout`)
- `last_resharding` = the **old** value from epoch T+1 (not `Some(next_epoch_info.epoch_height() + 1)`)

This stale clone is then saved as the authoritative epoch T+2 `EpochInfo`: [3](#0-2) 

The `EpochInfoV5` struct holds both fields as first-class members: [4](#0-3) 

`get_shard_layout` for epoch T+2 reads `EpochInfo::shard_layout()` first (V5+), so it will return the stale layout: [5](#0-4) 

`can_reshard` reads `last_resharding` from the epoch T+2 `EpochInfo`: [6](#0-5) 

With the stale `last_resharding` (either `None` or an old epoch height), `can_reshard` returns `true` when it should return `false`, bypassing the cooldown.

---

### Impact Explanation

**Broken invariant 1 — shard layout mismatch**: The committed block header for the last block of epoch T carries `shard_split = Some((shard_id, boundary_account))` in `BlockHeaderInnerRestV6`. The `BlockInfo` for that block also carries `shard_split`. But the epoch T+2 `EpochInfo.shard_layout` is the old layout (no split). `get_shard_layout(epoch_T+2)` returns the wrong layout. `start_resharding` checks `will_shard_layout_change` by comparing current and next epoch layouts; with the stale layout, no resharding executes. The child shards are never created, but the block header permanently says they should be.

**Broken invariant 2 — cooldown bypass**: `can_reshard` uses `EpochInfoV5::last_resharding` for an O(1) cooldown check. With the stale `last_resharding` (None or old), the cooldown is bypassed. The documentation explicitly states: "allowing back-to-back reshardings is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`." [7](#0-6) 

**Broken invariant 3 — chunk header validation failure**: `validate_chunk_with_chunk_extra_and_receipts_root` compares `chunk_header.proposed_split()` against `prev_chunk_extra.proposed_split()`. With the wrong shard layout in epoch T+2, the child shards' chunk extras are never created, causing `InvalidChunkHeaderShardSplit` for every chunk in epoch T+2. This can halt block production.

**Severity: High.** The chain can stall at the epoch T+2 boundary with `InvalidChunkHeaderShardSplit` errors, or proceed with a permanently inconsistent shard layout that corrupts all subsequent dynamic resharding decisions.

---

### Likelihood Explanation

The trigger requires two conditions to coincide:
1. Dynamic resharding is active (`ProtocolFeature::DynamicResharding`, protocol version 85) and a shard split is scheduled for epoch T+2.
2. `proposals_to_epoch_info` fails with `ThresholdError` or `NotEnoughValidators` at the same epoch boundary.

`NotEnoughValidators` is returned when `num_chunk_producers < minimum_validators_per_shard`: [8](#0-7) 

A shard split increases `num_shards`, which can push the validator-to-shard ratio below the minimum, making `NotEnoughValidators` more likely precisely at a resharding epoch boundary. The combination is low-to-medium likelihood but the impact is high.

---

### Recommendation

In the `ThresholdError` and `NotEnoughValidators` fallback arms of `finalize_epoch`, after cloning `next_epoch_info`, explicitly apply the computed `next_next_shard_layout` and `last_resharding` to the fallback `EpochInfo` before saving it. Specifically, if the fallback `EpochInfo` is `V5`, update its `shard_layout` field to `next_next_shard_layout` and its `last_resharding` field to the computed `last_resharding` value. This mirrors the fix in the Fluid DEX report: use the already-updated state rather than a stale snapshot.

```rust
Err(EpochError::ThresholdError { stake_sum, num_seats }) => {
    tracing::warn!(...);
    let mut epoch_info = EpochInfo::clone(&next_epoch_info);
    *epoch_info.epoch_height_mut() += 1;
    // FIX: propagate the computed shard layout and cooldown state
    if let EpochInfo::V5(ref mut v5) = epoch_info {
        v5.shard_layout = next_next_shard_layout.clone();
        v5.last_resharding = last_resharding;
    }
    epoch_info
}
```

### Proof of Concept

**Sequence:**

1. Enable dynamic resharding (protocol version ≥ 85). Configure `min_epochs_between_resharding = 2`.
2. Drive a shard above the memory threshold so `compute_proposed_split` returns `Some(TrieSplit)` near the end of epoch N. The last block of epoch N embeds `shard_split = Some((shard_id, boundary))` in `BlockHeaderInnerRestV6`.
3. At the epoch N boundary, `finalize_epoch` is called. `next_next_shard_layout` is correctly derived as a new `ShardLayoutV3`. `last_resharding = Some(next_epoch_info.epoch_height() + 1)`.
4. Arrange for all validators to submit unstake proposals simultaneously (or reduce the validator set below `minimum_validators_per_shard`). `proposals_to_epoch_info` returns `Err(EpochError::NotEnoughValidators)`.
5. The fallback path saves `EpochInfo::clone(&next_epoch_info)` (with only `epoch_height += 1`) as epoch N+2's info.
6. **Observe**: `get_shard_layout(epoch_N+2)` returns the old layout (no split). `can_reshard` returns `true` immediately (cooldown bypassed). `start_resharding` skips execution (no layout change detected). The block header for the last block of epoch N permanently carries `shard_split = Some(...)` while epoch N+2's `EpochInfo.shard_layout` has no split.
7. In epoch N+2, chunk producers for the (non-existent) child shards cannot be found. `InvalidChunkHeaderShardSplit` is produced for every chunk, halting block production. [9](#0-8) [10](#0-9) [11](#0-10)
