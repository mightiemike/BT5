### Title
P2P Protobuf `ValidResourceBounds` Deserialization Misclassifies `AllResources` as `L1Gas` When `l2_gas` and `l1_data_gas` Are Zero, Causing Valid Transaction Rejection and Peer Reporting — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion uses a zero-value heuristic to select between `ValidResourceBounds::L1Gas` and `ValidResourceBounds::AllResources`. A valid V3 invoke transaction with `AllResourceBounds` where `l2_gas = 0` and `l1_data_gas = 0` (but `l1_gas` non-zero) is accepted by the gateway, stored in the mempool, and propagated via P2P. On the receiving peer, the protobuf deserializer classifies the resource bounds as `L1Gas`. The subsequent conversion to `RpcInvokeTransactionV3` unconditionally rejects any non-`AllResources` variant, causing the transaction to be dropped and the sending peer to be reported as faulty.

---

### Finding Description

**Step 1 — Valid transaction accepted at gateway.**

`RpcInvokeTransactionV3` always carries `AllResourceBounds` (a struct, not an enum). A user may legally set `l2_gas` and `l1_data_gas` to zero while keeping `l1_gas` non-zero; the stateless validator accepts this: [1](#0-0) 

The gateway converts it to `InternalRpcInvokeTransactionV3` (which also stores `AllResourceBounds`) and computes the hash using `ValidResourceBounds::AllResources`, which includes `L1_DATA_GAS` in the Poseidon preimage: [2](#0-1) 

**Step 2 — P2P serialization loses the variant tag.**

When the mempool propagates the transaction, `MempoolP2pPropagator` converts `InternalRpcTransaction → RpcTransaction → InvokeTransactionV3 → protobuf::InvokeV3`. The `InvokeTransactionV3` carries `ValidResourceBounds::AllResources(AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 })`. The protobuf wire format has no field that encodes the enum variant; it only carries the three numeric resource-limit pairs. [3](#0-2) 

**Step 3 — Receiving peer's deserializer applies the wrong variant.**

On the receiving peer, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` reconstructs the variant by checking whether both `l2_gas` and `l1_data_gas` are zero:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← wrong for AllResources with zero l2/l1_data
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [4](#0-3) 

Because both fields are zero, the deserializer produces `L1Gas` instead of `AllResources`.

**Step 4 — Conversion to `RpcInvokeTransactionV3` fails.**

`TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` calls `snapi_invoke.try_into()` which is `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`. That conversion hard-rejects any non-`AllResources` variant:

```rust
// This conversion can fail only if the resource_bounds are not AllResources.
Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
``` [5](#0-4) 

`DEPRECATED_RESOURCE_BOUNDS_ERROR` is `MissingField { field_description: "ResourceBounds::l1_data_gas" }`. [6](#0-5) 

**Step 5 — Receiving peer reports the sending peer as faulty.**

`MempoolP2pRunner` treats any deserialization error as a "faulty transaction from network" and calls `report_peer`: [7](#0-6) 

---

### Impact Explanation

A valid V3 invoke transaction with `AllResourceBounds { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` is accepted by the originating sequencer's gateway and mempool but is silently dropped by every peer that receives it via P2P gossip. If the originating node is not the current block proposer, the transaction never reaches the proposer's mempool and is never sequenced. Additionally, the originating node is incorrectly reported as a faulty peer, which can trigger peer-scoring penalties or banning across the network.

This matches: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

---

### Likelihood Explanation

Any user who submits a V3 invoke transaction specifying only `l1_gas` (leaving `l2_gas` and `l1_data_gas` at their zero defaults) triggers this path. The stateless validator explicitly permits this combination. The condition is therefore reachable by any unprivileged user with no special knowledge of the network internals.

---

### Recommendation

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion must not use a zero-value heuristic to infer the enum variant. Two concrete fixes:

1. **Add an explicit discriminant field** to `protobuf::ResourceBounds` (e.g., a boolean `is_all_resources`) so the variant is encoded unambiguously on the wire.

2. **Alternatively**, since the mempool P2P protocol only carries V3 transactions (which always use `AllResources`), change the deserialization to unconditionally produce `AllResources` and reject the message if `l1_gas` is absent, removing the ambiguous zero-check entirely.

The `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion should also emit a clear, distinct error rather than reusing `DEPRECATED_RESOURCE_BOUNDS_ERROR`, to avoid misattributing the failure to a deprecated-format issue.

---

### Proof of Concept

1. Construct a valid `RpcInvokeTransactionV3` with:
   ```
   resource_bounds = AllResourceBounds {
       l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
   }
   ```
2. Submit it to sequencer A's gateway via `POST /add_transaction`. It is accepted; the gateway returns a `tx_hash` computed with `AllResources` (including `L1_DATA_GAS` in the Poseidon preimage).
3. Sequencer A's mempool propagates the transaction to sequencer B via the gossipsub mempool topic.
4. Sequencer B's `MempoolP2pRunner` deserializes the protobuf message. `ValidResourceBounds::try_from` sees `l2_gas.is_zero() && l1_data_gas.is_zero()` → produces `L1Gas`.
5. `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` returns `Err(DEPRECATED_RESOURCE_BOUNDS_ERROR)`.
6. `MempoolP2pRunner` logs "Received a faulty transaction from network" and calls `report_peer` on sequencer A.
7. The transaction is absent from sequencer B's mempool. If B is the block proposer, the transaction is never sequenced. [8](#0-7) [9](#0-8) [10](#0-9)

            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
