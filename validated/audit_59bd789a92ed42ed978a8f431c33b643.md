### Title
GC Off-by-One Permanently Locks Out Proofs for Blocks with Exactly `gc_threshold + 1` Confirmations — (`contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` removes the oldest mainchain block whenever `amount_of_headers_we_store > gc_threshold`. The removed block has exactly `gc_threshold + 1` actual confirmations. `verify_transaction_inclusion` guards only that `args.confirmations <= gc_threshold`, which passes for any request up to `gc_threshold` confirmations — including requests targeting the block that was just deleted. The result is a permanent, irrecoverable panic for any proof consumer targeting that block.

---

### Finding Description

**GC removal arithmetic** (`contract/src/lib.rs`, `run_mainchain_gc`):

```
amount_of_headers_we_store = tip_height − initial_height + 1
trigger condition:          amount_of_headers_we_store > gc_threshold
                            → fires when count = gc_threshold + 1
total_amount_to_remove    = amount_of_headers_we_store − gc_threshold = 1
selected_amount_to_remove = min(1, batch_size) = 1
```

Block at `initial_height` (call it H) is removed. Its actual confirmation count at removal time is:

```
(tip_height − H) + 1 = (H + gc_threshold) − H + 1 = gc_threshold + 1
``` [1](#0-0) 

**Guard in `verify_transaction_inclusion`:**

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [2](#0-1) 

A proof consumer requesting `confirmations = gc_threshold` for block H passes this guard (`gc_threshold <= gc_threshold` is `true`). The function then immediately attempts:

```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
``` [3](#0-2) 

`remove_block_header` deletes the entry from both `mainchain_header_to_height` and `headers_pool`:

```rust
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [4](#0-3) 

Because block H is no longer in `mainchain_header_to_height`, the `unwrap_or_else` panics unconditionally. The same path is hit by `verify_transaction_inclusion_v2`, which delegates to `verify_transaction_inclusion` after its coinbase check. [5](#0-4) 

---

### Impact Explanation

Any proof consumer that observed block H confirmed with `>= gc_threshold` confirmations and then calls `verify_transaction_inclusion` (or `_v2`) for block H after the GC fires receives a permanent panic. The block cannot be re-submitted (it would be treated as a fork or duplicate), and the proof is permanently unverifiable. This is a **permanent proof lockout** for a legitimately confirmed block, matching the stated critical scope.

---

### Likelihood Explanation

The trigger is a normal, expected relayer operation: submitting the `(gc_threshold + 1)`-th block. With `gc_threshold = 144` (the documented minimum recommended value), this fires after just one day of Bitcoin blocks. The relayer is a trusted role, but the GC fires automatically inside every `submit_blocks` call — no special intent is required. Any proof consumer whose proof window straddles the GC boundary is affected.

---

### Recommendation

Change the GC trigger and removal logic so that the oldest block retained always has at least `gc_threshold` confirmations, i.e., keep `gc_threshold + 1` blocks rather than `gc_threshold`:

```rust
// Current (buggy):
if amount_of_headers_we_store > self.gc_threshold {
    let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;

// Fixed: retain gc_threshold + 1 blocks so the oldest has gc_threshold confirmations
if amount_of_headers_we_store > self.gc_threshold + 1 {
    let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold - 1;
```

Alternatively, strengthen the guard in `verify_transaction_inclusion` to check that the specific block hash is present in `mainchain_header_to_height` before relying on the confirmation arithmetic.

---

### Proof of Concept

Concrete state trace with `gc_threshold = 144`, genesis at height H:

| Step | Action | Chain stored | Oldest block | Oldest confirmations |
|------|--------|-------------|--------------|----------------------|
| 0 | init | H | H | 1 |
| 1 | submit 144 blocks (H+1…H+144) | H…H+144 (145 blocks) | H | 145 |
| 2 | `run_mainchain_gc(150)` fires inside `submit_blocks` | H+1…H+144 (144 blocks) | H+1 | 144 |
| 3 | `verify_transaction_inclusion(block=H, confirmations=144)` | — | — | panics |

Step 3 arithmetic:
- Guard: `144 <= 144` → **passes**
- `mainchain_header_to_height.get(H)` → `None` (removed in step 2)
- `unwrap_or_else` → **panic**: `"block does not belong to the current main chain"` [6](#0-5) [7](#0-6)
