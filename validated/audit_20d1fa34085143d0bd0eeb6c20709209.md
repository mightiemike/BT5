### Title
SwapAllowlistExtension gates `sender` (router address) instead of the actual swapper, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

The `SwapAllowlistExtension` is configured to restrict which addresses may swap on a pool. However, the extension receives and checks `sender`, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router address — not the actual user. If the router is allowlisted (or the pool admin adds it to enable router-mediated swaps), any unprivileged user can bypass the allowlist entirely by routing through the public router contract.

### Finding Description

The pool's `swap()` function passes `msg.sender` as `sender` to `_beforeSwap()`: [1](#0-0) 

`_beforeSwap` encodes `sender` and forwards it to every configured extension in order: [2](#0-1) 

The `SwapAllowlistExtension.beforeSwap` hook receives this `sender` value and checks it against an `allowedSwapper` mapping keyed by `(pool, sender)`. When a user calls `MetricOmmSimpleRouter.exactInput*()`, the router calls `pool.swap(...)` directly, so `msg.sender` of the pool is the **router address**, not the user. The extension therefore checks whether the **router** is allowlisted, not the actual swapper.

The `addLiquidity` path has the same structural issue in the opposite direction: `_beforeAddLiquidity` passes `msg.sender` as `sender` and the user-supplied `owner` as a separate argument: [3](#0-2) 

The `DepositAllowlistExtension` checks `owner` (keyed by `(pool, owner)`), meaning a non-allowlisted caller can supply an allowlisted `owner` address and pass the gate while being the actual payer through the callback. [4](#0-3) 

### Impact Explanation

A pool admin deploys a pool with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., a private institutional pool or a KYC-gated pool). The admin allowlists the `MetricOmmSimpleRouter` to allow allowlisted users to trade via the standard router. Because the extension checks `sender = router`, **any unprivileged user** can now swap on the pool by calling the public router. This breaks the core access-control invariant and allows unauthorized parties to:

- Execute swaps that drain pool reserves at oracle-quoted prices
- Cause bad-price execution against LPs who deposited under the assumption of a restricted counterparty set
- Render the allowlist configuration completely ineffective

This matches the **Admin-boundary break** and **Broken core pool functionality causing loss of funds** impact categories.

### Likelihood Explanation

The trigger is fully unprivileged: any user with tokens can call `MetricOmmSimpleRouter.exactInputSingle()` or `exactInput()` targeting the allowlist-protected pool. No special role, malicious setup, or non-standard token is required. The only precondition is that the pool admin has configured `SwapAllowlistExtension` and allowlisted the router (a natural operational step to support normal UX). This is a reachable, one-transaction bypass.

### Recommendation

The `SwapAllowlistExtension.beforeSwap` hook must check the **original user identity**, not the intermediary. Two options:

1. **Check `recipient` instead of `sender`** if the pool's intent is to gate who receives swap output — but this is semantically wrong for an "allowed swapper" gate.
2. **Require the router to forward the original caller** via `extensionData`, and have the extension decode and verify that identity. The router would encode `msg.sender` into `extensionData` before calling the pool, and the extension would verify the decoded address against the allowlist.
3. **Gate `sender` AND require that `sender == recipient`** so the router cannot be used as a pass-through.

For the `DepositAllowlistExtension`, the hook should check `sender` (the actual payer/caller) rather than `owner` (the position recipient), or check both, to prevent a non-allowlisted caller from depositing by supplying an allowlisted `owner`.

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured.
2. Pool admin allowlists the MetricOmmSimpleRouter address for the pool.
3. Pool admin does NOT allowlist attacker's EOA.
4. Attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
5. Router calls pool.swap(...) — msg.sender = router.
6. _beforeSwap passes sender = router to SwapAllowlistExtension.beforeSwap.
7. Extension checks allowedSwapper[pool][router] == true → passes.
8. Attacker's swap executes on the allowlist-protected pool.
9. Attacker receives tokens; pool LPs suffer unauthorized price impact.
```

The root cause mirrors the LendingPool analog exactly: a configured guard (`SwapAllowlistExtension`) operates on the wrong identity (`sender = router`) rather than the economically relevant actor (the actual user), making the guard ineffective whenever the standard periphery router is used.
