### Title
Symmetric Funding Accumulator Update Creates Unbounded Protocol Deficit When Open Interest Is Imbalanced — (`core/contracts/PerpEngineState.sol`)

### Summary
`updateStates` increments both `cumulativeFundingLongX18` and `cumulativeFundingShortX18` by the same `paymentAmount`. Because funding PnL is computed as `diffX18 * balance.amount`, the net vQuoteBalance change across all users equals `paymentAmount * (TotalShortOI − TotalLongOI)`. When OI is imbalanced, the protocol either creates vQuoteBalance out of thin air (deficit) or destroys it (surplus) with no corresponding asset movement, breaking the core accounting invariant.

### Finding Description
In `updateStates`: [1](#0-0) 

Both accumulators are incremented identically. In `_updateBalance`, the funding charge applied to each position is `diffX18 * balance.amount`: [2](#0-1) 

For a long of size `L`: charge = `+paymentAmount * L` (deducted from vQuoteBalance).
For a short of size `S` (stored as `-S`): charge = `-paymentAmount * (-S) = +paymentAmount * S` (added to vQuoteBalance).

Net change = `paymentAmount * (TS − TL)`. When `TS ≠ TL`, this is non-zero. No mechanism in `updateStates` adjusts `availableSettle` or any insurance pool to absorb this imbalance. [3](#0-2) 

`availableSettle` is initialized to 0 at product creation and never touched by `updateStates`: [4](#0-3) 

### Impact Explanation
**Direct cash-out via `settlePnl` is gated by `availableSettle = 0`**, so the surplus vQuoteBalance cannot be immediately withdrawn: [5](#0-4) 

However, two concrete impact paths remain:

1. **Health inflation**: `calculatePositionPnl` = `price * amount + vQuoteBalance` is used directly in health/margin checks. Shorts with inflated vQuoteBalance appear over-collateralized, allowing them to open larger positions or avoid liquidation, compounding the deficit. [6](#0-5) 

2. **Deficit realization via socialization**: When any subaccount with negative vQuoteBalance is liquidated, `socializeSubaccount` sets `state.availableSettle += insuranceCover`. Once `availableSettle > 0`, the previously accumulated vQuoteBalance surplus in shorts becomes settleable, directly draining protocol assets. [7](#0-6) 

The deficit magnitude is `paymentAmount * |TS − TL|` per `updateStates` call, compounding over time at up to `MAX_DAILY_FUNDING_RATE` (2% per day). [8](#0-7) 

### Likelihood Explanation
OI imbalance is the normal state of any perp market — it is never perfectly balanced. The endpoint calls `updateStates` regularly as part of normal protocol operation (`onlyEndpoint`). No special attacker action is required beyond opening a position on the dominant side. The deficit accumulates passively on every funding epoch. [9](#0-8) 

### Recommendation
Replace the symmetric update with an asymmetric one that transfers funding directly from longs to shorts (or vice versa), weighted by their respective OI:

```solidity
// When paymentAmount > 0: longs pay, shorts receive
// longPayment = paymentAmount (per unit of long OI)
// shortReceive = paymentAmount * TotalLongOI / TotalShortOI (if TS > 0)
state.cumulativeFundingLongX18 += paymentAmount;
if (state.openInterestShort > 0) {
    state.cumulativeFundingShortX18 += paymentAmount.mul(state.openInterestLong).div(state.openInterestShort);
}
```

Alternatively, track `openInterestLong` and `openInterestShort` separately and compute the per-unit payment for each side so that `totalLongPayment == totalShortReceive` exactly.
