### Title
Linked Signer Can Overwrite `linkedSigners` Mapping to Hijack Subaccount Signing Authority — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction handler in `EndpointTx.sol` passes `allowLinkedSigner = true` to `validateSignedTx`, meaning the **current linked signer** can sign and submit a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with an arbitrary address. There is no check that the signer of a `LinkSigner` transaction is the actual subaccount owner rather than the existing linked signer. A compromised or malicious linked signer can permanently replace itself with an attacker-controlled address, seizing full fast-mode signing authority over the subaccount.

---

### Finding Description

In `EndpointTx.sol`, the fast-mode `LinkSigner` branch decodes the transaction and calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` delegates to `validateSignature`, which accepts a valid signature from **either** the subaccount owner **or** the current linked signer: [2](#0-1) 

After validation, the handler unconditionally overwrites the `linkedSigners` mapping: [3](#0-2) 

The `linkedSigners` mapping is declared in `EndpointStorage.sol`: [4](#0-3) 

There is no guard requiring that the signer of a `LinkSigner` transaction be the actual subaccount owner (i.e., the address embedded in the `sender` bytes32). The `validateSignedTx` function applies no per-transaction-type restriction on who may sign: [5](#0-4) 

This is the direct analog to the external report: just as `registerAsset()` overwrites `_assetClass` without checking if already set, `LinkSigner` overwrites `linkedSigners[subaccount]` without checking that the signer is the actual owner rather than the existing linked signer.

---

### Impact Explanation

The `linkedSigners` mapping is the authority source for all fast-mode transaction validation. `getLinkedSigner` is consulted for every fast-mode operation including `WithdrawCollateral`, `LiquidateSubaccount`, `TransferQuote`, `MintNlp`, `BurnNlp`, and `CreateIsolatedSubaccount`: [6](#0-5) 

If an attacker overwrites `linkedSigners[victim_subaccount]` to an attacker-controlled address, they gain the ability to sign any of these operations on behalf of the victim. Concretely: the attacker can drain the subaccount via `WithdrawCollateral`, force unfavorable liquidations, or transfer quote balances — all without the subaccount owner's knowledge or consent.

---

### Likelihood Explanation

Linked signers are by design hot wallets (API keys, automated trading bots). They are materially more likely to be compromised than the cold-wallet subaccount owner. Once a linked signer is compromised, the attacker can immediately escalate to full account control by submitting a single `LinkSigner` fast-mode transaction. The subaccount owner's only recovery path is a slow-mode `LinkSigner` transaction, which may be delayed by the sequencer queue, giving the attacker a window to drain funds.

---

### Recommendation

The `LinkSigner` fast-mode handler should pass `allowLinkedSigner = false` to `validateSignedTx`, ensuring only the actual subaccount owner (the address embedded in `sender`) can authorize a linked-signer change:

```solidity
// In the LinkSigner fast-mode branch of EndpointTx.sol
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // <-- must be false; only the owner may change the linked signer
);
```

This mirrors the mitigation in the external report (`require(_assetClass[_asset] == 0, "Already exists")`): add a check that prevents an already-authorized-but-limited party from overwriting the state that granted them their authority.

---

### Proof of Concept

1. Alice owns `alice_subaccount` and has set `linkedSigners[alice_subaccount] = hot_wallet`.
2. Attacker compromises `hot_wallet`.
3. Attacker constructs a `SignedLinkSigner` struct with `sender = alice_subaccount`, `signer = attacker_address`, and the correct nonce.
4. Attacker signs the EIP-712 digest using `hot_wallet`'s private key.
5. Attacker submits the transaction to the sequencer (fast mode).
6. `validateSignedTx(..., true)` accepts the `hot_wallet` signature as valid.
7. `linkedSigners[alice_subaccount]` is overwritten to `attacker_address`.
8. Attacker now signs a `WithdrawCollateral` transaction using `attacker_address`, draining Alice's collateral. [1](#0-0)
