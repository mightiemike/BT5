### Title
Attacker-Controlled `tx_id` Accepted as Trusted Transaction Identity in `verify_transaction_inclusion`, Enabling Merkle Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` accepts a caller-supplied `tx_id` and uses it directly as the leaf of a Merkle proof computation without validating that it represents an actual transaction rather than an internal Merkle tree node. An unprivileged NEAR caller can supply an internal node hash as `tx_id`, construct a valid sibling proof for it, and cause the function to return `true` for a Bitcoin transaction that never existed. Any downstream contract that gates asset release or state transitions on this return value is directly exploitable.

---

### Finding Description

The public method `verify_transaction_inclusion` computes the Merkle root from the caller-supplied `tx_id` and `merkle_proof`, then compares it against the stored block header's `merkle_root`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

The function performs no validation that `args.tx_id` is a leaf-level transaction hash rather than an intermediate node in the Merkle tree. The contract's own documentation acknowledges this explicitly:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification." [2](#0-1) 

The function is marked `#[deprecated]` since v0.5.0, but `#[deprecated]` in Rust is a compiler lint only — it does not restrict on-chain invocation. The function remains `pub`, carries only a `#[pause]` guard (which is inactive by default), and is reachable by any unprivileged NEAR account. [3](#0-2) 

The `verify_transaction_inclusion_v2` replacement mitigates this by requiring a coinbase proof of equal depth, which anchors the tree depth and prevents shorter internal-node proofs from passing. However, `v1` remains independently callable and unprotected. [4](#0-3) 

---

### Impact Explanation

Any NEAR contract that calls `verify_transaction_inclusion` to authorize a cross-chain action (e.g., minting wrapped BTC, releasing escrowed NEAR, or updating bridge state) can be deceived into treating a fabricated `tx_id` as a confirmed Bitcoin transaction. The attacker does not need to find a hash preimage — they only need to:

1. Select any confirmed mainchain block whose Merkle tree has depth ≥ 2.
2. Compute an internal node hash at any depth D ≥ 1 (these are deterministic from public block data).
3. Construct the sibling proof path from depth D up to the root (all values are public).
4. Call `verify_transaction_inclusion` with the internal node as `tx_id`, the correct `tx_index` for that node's position, and the sibling path as `merkle_proof`.

The function returns `true`. The corrupted proof result is indistinguishable from a legitimate one to any consumer of the return value.

---

### Likelihood Explanation

- The entry point is public and requires no privileged role, staked deposit beyond storage, or special account.
- All inputs needed for the attack (block Merkle trees, internal node hashes, sibling paths) are derived entirely from public Bitcoin chain data.
- The vulnerability is acknowledged in the contract's own NatSpec, confirming the developers are aware the invariant is not enforced at this layer — but the function remains callable.
- Any deployed bridge or escrow contract that has not migrated to `verify_transaction_inclusion_v2` is immediately at risk.

---

### Recommendation

1. **Remove or gate `verify_transaction_inclusion` at the contract level.** Add a runtime `env::panic_str` at the top of the function body so it is unreachable on-chain, rather than relying solely on a compiler-level `#[deprecated]` attribute that has no runtime effect.
2. **Enforce leaf-level validation in `verify_transaction_inclusion_v2`.** Require that `args.tx_index < (1 << args.merkle_proof.len())` to ensure the claimed position is a valid leaf slot for the proven tree depth.
3. **Document the migration requirement** in the contract's public interface so all downstream consumers know `v1` must not be used for authorization decisions.

---

### Proof of Concept

Given a confirmed mainchain block `B` with Merkle root `R` and at least 4 transactions `[T0, T1, T2, T3]`:

```
        R
       / \
      N01  N23
     / \   / \
    T0  T1 T2  T3
```

The attacker computes `N01 = dSHA256(T0 || T1)` (public data). They then call:

```json
{
  "tx_id":            "<N01 bytes>",
  "tx_block_blockhash": "<hash of B>",
  "tx_index":         0,
  "merkle_proof":     ["<N23 bytes>"],
  "confirmations":    1
}
```

`compute_root_from_merkle_proof(N01, 0, [N23])` computes `dSHA256(N01 || N23) = R`, which equals `header.block_header.merkle_root`. [5](#0-4) 

The function returns `true` for a transaction `N01` that does not exist in the Bitcoin blockchain. A downstream contract consuming this result would authorize the corresponding cross-chain action for a fabricated transaction.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
