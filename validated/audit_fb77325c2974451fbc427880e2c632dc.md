### Title
MintNlp/BurnNlp `oraclePriceX18` Excluded from User Signature Digest — No Slippage Protection — (`core/contracts/Verifier.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

When a user signs a `MintNlp` or `BurnNlp` transaction, the EIP-712 digest covers only `{sender, quoteAmount/nlpAmount, nonce}`. The `oraclePriceX18` field — which directly determines how many NLP tokens the user receives (mint) or how much USDC the user receives (burn) — is **not committed to by the user's signature**. The sequencer supplies `oraclePriceX18` at submission time. The user has no on-chain mechanism to enforce a minimum acceptable exchange rate, making them fully exposed to price changes between signing and execution.

---

### Finding Description

`SignedMintNlp` and `SignedBurnNlp` each carry an `oraclePriceX18` field alongside the user-signed inner struct:

```solidity
// IEndpoint.sol
struct SignedMintNlp {
    MintNlp tx;          // {sender, quoteAmount, nonce} — user signs this
    bytes signature;
    int128 oraclePriceX18;        // NOT signed by user
    int128[] nlpPoolRebalanceX18; // NOT signed by user
}
``` [1](#0-0) 

In `Verifier.sol`, the digest for `MintNlp` is computed as:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(MINT_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.quoteAmount,
        signedTx.tx.nonce
        // oraclePriceX18 is absent
    )
);
``` [2](#0-1) 

The same omission applies to `BurnNlp`: [3](#0-2) 

In `EndpointTx.sol`, the sequencer-supplied `signedTx.oraclePriceX18` is passed directly to `clearinghouse.mintNlp()` / `clearinghouse.burnNlp()`: [4](#0-3) [5](#0-4) 

In `Clearinghouse.sol`, the NLP amount minted is `quoteAmount / oraclePriceX18`, and the USDC amount returned on burn is `nlpAmount * oraclePriceX18 - burnFee`: [6](#0-5) [7](#0-6) 

Because `oraclePriceX18` is not part of the signed digest, the user's signature is valid for **any** price the sequencer chooses to supply.

---

### Impact Explanation

**MintNlp:** A user signs a transaction to spend `quoteAmount` USDC. The NLP tokens received = `quoteAmount / oraclePriceX18`. If `oraclePriceX18` is higher than the price the user observed when signing, the user receives fewer NLP tokens than expected — a direct asset shortfall with no on-chain recourse.

**BurnNlp:** A user signs a transaction to burn `nlpAmount` NLP. The USDC received = `nlpAmount * oraclePriceX18 - burnFee`. If `oraclePriceX18` is lower than the price the user observed when signing, the user receives less USDC than expected.

In both cases the corrupted state delta is the user's quote or NLP balance in `SpotEngine`. There is no minimum-output guard anywhere in the call path.

---

### Likelihood Explanation

The NLP price (`oraclePriceX18`) is a live oracle value that changes continuously as the NLP pool's underlying assets fluctuate. Even with a fully honest sequencer, the price at the moment the sequencer batches and submits the transaction can differ materially from the price the user saw when signing. The gap is bounded only by sequencer latency, not by any protocol invariant. The user has no way to express a price bound in their signed message.

---

### Recommendation

Include `oraclePriceX18` (or a user-specified `minNlpAmount` / `minQuoteAmount` bound) in the EIP-712 digest for both `MintNlp` and `BurnNlp`. For `MintNlp`, the user should commit to a `minNlpAmount`; for `BurnNlp`, a `minQuoteAmount`. The on-chain execution should revert if the actual output falls below the user-committed bound.

Concretely, add a `minAmountOut` field to `MintNlp` and `BurnNlp` structs, include it in the Verifier digest, and add a `require(actualOut >= txn.minAmountOut)` check in `Clearinghouse.mintNlp` / `Clearinghouse.burnNlp`.

---

### Proof of Concept

1. NLP oracle price is `1.00 USDC/NLP`. User signs `MintNlp{quoteAmount=1000e18, nonce=N}` expecting to receive 1000 NLP.
2. Before the sequencer batches the transaction, the NLP pool rebalances and the oracle price moves to `1.10 USDC/NLP`.
3. Sequencer submits `SignedMintNlp{tx=..., oraclePriceX18=1.10e18, ...}`. Signature validation passes because `oraclePriceX18` is not in the digest.
4. `Clearinghouse.mintNlp` computes `nlpAmount = 1000e18 / 1.10e18 ≈ 909 NLP`.
5. User loses ~91 NLP (~91 USDC of value) with no on-chain protection.

    function mintNlp(
        IEndpoint.MintNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.quoteAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 quoteAmount = int128(txn.quoteAmount);
        int128 nlpAmount = quoteAmount.div(oraclePriceX18);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] >= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, -nlpAmount);

        spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -quoteAmount);
        _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);

        require(
            getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
