### Title
`limbs_for_int` Underestimates Byte Length of Negative Numbers, Causing Cost Undercharge in `op_multiply` — (File: `src/more_ops.rs`)

### Summary

`limbs_for_int` computes the canonical byte length of a `Number` (BigInt) using `v.bits().div_ceil(8)`. For negative numbers whose absolute value has its high bit set in the minimal byte representation (e.g., `-129`, `-255`, `-32769`), this formula returns a value that is 1 less than the actual canonical two's-complement byte length. In `op_multiply`, this underestimated `l0` is used both for per-step cost charging and for the intermediate-size limit check (`l0 > 1024`), causing cost undercharging and a 1-byte bypass of the size guard.

### Finding Description

`limbs_for_int` is defined as: [1](#0-0) 

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

`BigInt::bits()` returns the number of bits in the **absolute value**. For a negative number `-n`, the canonical two's-complement encoding needs an extra byte whenever the high bit of the `ceil(bits(n)/8)`-byte representation of `n` is set (to avoid the sign bit being misread as positive). Specifically, for any `-n` where `2^(8k-1) < n < 2^(8k)` (e.g., `n ∈ {129..255}` for `k=1`), the actual canonical length is `k+1` bytes, but `limbs_for_int` returns `k`.

Concrete example: `-129` is encoded as `[0xff, 0x7f]` (2 bytes), but `limbs_for_int(-129) = bits(129).div_ceil(8) = 8/8 = 1`.

In `op_multiply`, after each multiplication step, `l0` is updated with this underestimated value: [2](#0-1) 

This underestimated `l0` is then used in the **next** iteration's cost formula: [3](#0-2) 

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

And in the size-limit guard: [4](#0-3) 

```rust
if l0 > 1024 {
    return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
}
```

The existing unit tests for `limbs_for_int` do not cover the problematic range: [5](#0-4) 

The test helper only checks values like `[0x80]` (= -128, which is exactly `2^7` and does **not** trigger the bug) and `[0x81]` (= -127, which also does not trigger it). The range `{-129, ..., -255}` is untested.

### Impact Explanation

**Impact: Low.**

For each multiplication step where the intermediate `total` is a negative number with high-bit-set absolute value, the cost charged is underestimated by:

```
MUL_LINEAR_COST_PER_BYTE × 1  +  (l1 / MUL_SQUARE_COST_PER_BYTE_DIVIDER)
= 6  +  (l1 / 128)
```

For the maximum `l1 = 256`, this is `6 + 2 = 8` cost units per affected step. The size-limit guard can be bypassed by 1 byte (a 1025-byte intermediate result passes as 1024 bytes). The final `malloc_cost` call uses `a.atom_len(ptr)` (the actual length), so the output allocation cost is correct; only the intermediate per-step cost is undercharged.

### Likelihood Explanation

**Likelihood: High.**

Any attacker-controlled CLVM program that invokes `op_multiply` with a first argument in the range `(-255, -129)` (or analogous ranges for larger byte widths) and at least one subsequent argument will trigger the underestimate. The entry path is direct: attacker supplies serialized CLVM bytes → `run_program` dispatches to `op_multiply` → `limbs_for_int` is called on the intermediate result.

### Recommendation

Replace `limbs_for_int` with a function that accounts for the sign-extension byte required by two's-complement encoding for negative numbers:

```rust
fn limbs_for_int(v: &Number) -> usize {
    if v.sign() == Sign::NoSign {
        return 0;
    }
    let bits = v.bits() as usize;
    let byte_len = bits.div_ceil(8);
    // For negative numbers, if the high bit of the abs value fills the top byte,
    // the two's-complement encoding needs an extra byte for sign extension.
    if v.sign() == Sign::Minus {
        let abs_val = v.magnitude(); // BigUint
        // Check if the high bit of the minimal representation is set
        // i.e., abs is NOT a power of 2 and bits % 8 == 0
        if bits % 8 == 0 {
            return byte_len + 1;
        }
    }
    byte_len
}
```

Alternatively, convert the number to its canonical signed bytes and measure the length directly, matching what `new_number` / `to_signed_bytes_be` actually produces.

### Proof of Concept

```rust
use clvmr::allocator::Allocator;
use clvmr::chia_dialect::ClvmFlags;
use clvmr::more_ops::op_multiply;
use clvmr::number::Number;

fn main() {
    let mut alloc = Allocator::new();

    // Build args: (* -129 1)
    // -129 canonical = [0xff, 0x7f] (2 bytes)
    let neg129 = alloc.new_atom(&[0xff, 0x7f]).unwrap(); // -129
    let one    = alloc.new_atom(&[0x01]).unwrap();        // 1
    let nil    = alloc.nil();
    let tail   = alloc.new_pair(one, nil).unwrap();
    let args   = alloc.new_pair(neg129, tail).unwrap();

    // limbs_for_int(-129) returns 1, but actual byte length is 2.
    // Cost for the second step uses l0=1 instead of l0=2:
    //   charged:  (1+1)*6 + (1*1)/128 = 12
    //   correct:  (2+1)*6 + (2*1)/128 = 18
    // Undercharge = 6 cost units.
    let result = op_multiply(&mut alloc, args, 1_000_000, ClvmFlags::empty()).unwrap();
    println!("cost charged: {}", result.0);
    // Actual atom length of result (-129) is 2 bytes, malloc_cost adds 20.
    // Total cost = MUL_BASE_COST(92) + MUL_COST_PER_OP(885) + 12 (undercharged) + 20 = 1009
    // Correct total would be: 92 + 885 + 18 + 20 = 1015
}
```
