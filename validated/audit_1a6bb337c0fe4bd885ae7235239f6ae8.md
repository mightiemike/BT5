### Title
Slow Mode Transaction Permanently Deleted Without Processing on Failure, Locking Deposited User Funds — (`core/contracts/Endpoint.sol`)

---

### Summary

In `Endpoint._executeSlowModeTransaction`, the slow mode transaction entry is deleted from storage and the queue pointer is incremented **before** the `try/catch` call to `processSlowModeTransaction`. If `processSlowModeTransaction` reverts, the catch block silently discards the error without restoring the deleted transaction or rolling back the counter. For `DepositCollateral` slow mode transactions, tokens are transferred to the clearinghouse at submission time; if the corresponding slow mode transaction later fails during execution, those tokens are permanently locked in the clearinghouse with no balance credit to the user. The comment `// try return funds now removed` at line 226 explicitly acknowledges that a prior cleanup mechanism was removed, leaving the inconsistency unaddressed.

---

### Finding Description

In `Endpoint._executeSlowModeTransaction` (Endpoint.sol:185–229), the execution order is:

1. **L194**: `delete slowModeTxs[_slowModeConfig.txUpTo++]` — the transaction is permanently erased from storage and the in-memory `txUpTo` counter is incremented.
2. **L207**: `try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch { ... }` — if this reverts, the catch block runs but performs no state restoration.
3. Back in `executeSlowModeTransaction` (L231–236): `nSubmissions += 1` and `slowModeConfig = _slowModeConfig` are executed unconditionally, persisting the incremented `txUpTo` and the incremented submission counter regardless of whether the inner call succeeded. [1](#0-0) 

The state delta on failure is:
- `slowModeTxs[N]` is permanently deleted (the transaction is gone forever).
- `slowModeConfig.txUpTo` is permanently incremented to `N+1` (the slot is skipped).
- `nSubmissions` is incremented as if the transaction was successfully processed.

The fund-loss path is rooted in `depositCollateralWithReferral` (Endpoint.sol:123–167), which transfers tokens to the clearinghouse **before** queuing the slow mode transaction: [2](#0-1) 

If the corresponding slow mode `DepositCollateral` transaction later fails during `processSlowModeTransactionImpl` (EndpointTx.sol:209–216), the tokens already held by the clearinghouse have no corresponding balance credit and no recovery path exists. [3](#0-2) 

The `nSubmissions` counter is also used as the `idx` argument to `clearinghouse.withdrawCollateral` and `clearinghouse.rebalanceXWithdraw`, which feeds into `markedIdxs` in the withdraw pool. Incrementing it for a failed transaction desynchronizes the withdrawal deduplication index from the actual processed state. [4](#0-3) 

---

### Impact Explanation

**Direct fund loss**: A user who calls `depositCollateralWithReferral` has their tokens transferred to the clearinghouse at submission time. If the slow mode `DepositCollateral` transaction fails during execution (e.g., due to a storage migration bug during a contract upgrade, a product state change, or any revert in `clearinghouse.depositCollateral`), the tokens are permanently locked in the clearinghouse with no balance credit. There is no protocol mechanism to recover them.

**Queue accounting corruption**: `slowModeConfig.txUpTo` and `nSubmissions` are permanently incremented for a transaction that was never processed. This desynchronizes the slow mode queue's processed-up-to pointer and the withdrawal deduplication index (`markedIdxs`), creating a persistent accounting inconsistency.

---

### Likelihood Explanation

The protocol uses upgradeable contracts via `ProxyManager`. The slow mode queue has a mandatory 3-day delay between submission and execution, creating a window during which contract upgrades can occur. The external report's cited scenario — storage values becoming unparsable due to incorrect migrations — applies directly here. The comment `// try return funds now removed` at Endpoint.sol:226 confirms that the developers were aware a cleanup mechanism was needed but removed it, leaving the inconsistency live. [5](#0-4) 

Any revert inside `processSlowModeTransactionImpl` — whether from a migration bug, a product state change between submission and execution, or an arithmetic overflow in the engine — will trigger this path. The 3-day window makes it realistic that at least one upgrade occurs while slow mode transactions are pending.

---

### Recommendation

Move the `delete` and counter increment to **after** a successful `try`, and restore the transaction on failure:

```solidity
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
// Do NOT delete here

try this.processSlowModeTransaction(txn.sender, txn.tx) {
    delete slowModeTxs[_slowModeConfig.txUpTo++]; // only on success
} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // txUpTo is NOT incremented; transaction remains in queue
    return; // or handle appropriately
}
```

Additionally, `nSubmissions += 1` in `executeSlowModeTransaction` should only be incremented when the inner call succeeds, to keep the withdrawal deduplication index consistent.

---

### Proof of Concept

1. User calls `Endpoint.depositCollateralWithReferral(subaccount, productId, amount)`.
   - Tokens are transferred from the user to the clearinghouse (Endpoint.sol:144–148).
   - A `DepositCollateral` slow mode transaction is queued at index `N` with `executableAt = now + 3 days` (Endpoint.sol:152–166).

2. During the 3-day window, the protocol owner upgrades the `SpotEngine` or `Clearinghouse` contract with a storage layout change that causes `spotEngine.updateBalance` to revert for existing product IDs.

3. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`.
   - `_executeSlowModeTransaction` runs: `delete slowModeTxs[N]`, `txUpTo` becomes `N+1` (Endpoint.sol:194).
   - `try this.processSlowModeTransaction(...)` reverts due to the storage migration bug.
   - The catch block runs; no state is restored (Endpoint.sol:207–227).
   - `nSubmissions += 1`, `slowModeConfig = _slowModeConfig` (Endpoint.sol:234–235).

4. Result:
   - `slowModeTxs[N]` is permanently deleted.
   - `txUpTo` is permanently `N+1`.
   - `nSubmissions` is incremented.
   - The user's tokens remain in the clearinghouse with no balance credit.
   - The user has no protocol-level recourse to recover their funds.
