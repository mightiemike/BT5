### Title
Nil-Response IBC Acknowledgement Bypasses NFT Refund, Causing Permanent Escrow Lock — (`x/nft-transfer/keeper/relay.go`)

### Summary
`Keeper.OnAcknowledgementPacket` uses a type-switch that only triggers `refundPacketToken` for `*channeltypes.Acknowledgement_Error`. A `channeltypes.Acknowledgement` with a nil `Response` field (produced by unmarshaling `{}`) falls to the `default` branch and returns `nil` with no refund, leaving the NFT permanently escrowed.

### Finding Description
The entry point is `IBCModule.OnAcknowledgementPacket`:

```
acknowledgement []byte  →  UnmarshalJSON  →  ack.Response == nil  →  keeper.OnAcknowledgementPacket
``` [1](#0-0) 

After unmarshaling, no `ack.ValidateBasic()` is called. The keeper then executes:

```go
switch ack.Response.(type) {
case *channeltypes.Acknowledgement_Error:
    return k.refundPacketToken(ctx, packet, data)
default:
    return nil   // nil Response lands here — no refund
}
``` [2](#0-1) 

A malicious counterparty chain writes acknowledgement bytes `{}` (valid JSON, valid Merkle commitment). The relayer submits `MsgAcknowledgement` with a valid proof. IBC core verifies the proof and calls the app module. `UnmarshalJSON("{}", &ack)` succeeds with `ack.Response == nil`. The switch falls to `default: return nil`. `refundPacketToken` is never called. [3](#0-2) 

The NFT remains owned by the escrow address indefinitely with no recovery path.

### Impact Explanation
The sender's NFT is transferred to the escrow address during `createOutgoingPacket`: [4](#0-3) 

With a nil-Response ack, `refundPacketToken` (which calls `TransferOwner` back to sender) is never invoked. The NFT is permanently locked in the escrow account. This is an irreversible fund loss for the sender.

### Likelihood Explanation
Requires a malicious or compromised counterparty chain that deliberately writes a nil-Response acknowledgement. This is within the IBC threat model: IBC core only verifies Merkle proofs, not acknowledgement content. Any counterparty chain operator can craft such an ack. The standard ibc-go fungible token transfer module avoids this by treating unknown/nil ack types as errors; this custom nft-transfer module does not.

### Recommendation
In `Keeper.OnAcknowledgementPacket`, explicitly handle `*channeltypes.Acknowledgement_Result` as success and treat the `default` case (nil or unknown Response) as an error requiring refund:

```go
switch ack.Response.(type) {
case *channeltypes.Acknowledgement_Result:
    return nil
case *channeltypes.Acknowledgement_Error:
    return k.refundPacketToken(ctx, packet, data)
default:
    // nil or unknown response — treat as failure and refund
    return k.refundPacketToken(ctx, packet, data)
}
```

Additionally, add `ack.ValidateBasic()` in `IBCModule.OnAcknowledgementPacket` immediately after unmarshaling: [5](#0-4) 

### Proof of Concept
1. User on source chain sends NFT via `MsgTransfer` → NFT escrowed at `GetEscrowAddress(sourcePort, sourceChannel)`.
2. Malicious counterparty chain commits acknowledgement bytes `{}` for the packet sequence.
3. Relayer submits `MsgAcknowledgement` with `Acknowledgement: []byte("{}")` and a valid Merkle proof.
4. IBC core verifies proof → calls `IBCModule.OnAcknowledgementPacket`.
5. `types.ModuleCdc.UnmarshalJSON([]byte("{}"), &ack)` succeeds; `ack.Response == nil`.
6. `keeper.OnAcknowledgementPacket` switch: nil Response → `default: return nil`.
7. `refundPacketToken` is never called; NFT remains in escrow permanently.

A keeper unit test can reproduce this by calling `keeper.OnAcknowledgementPacket` directly with `channeltypes.Acknowledgement{Response: nil}` and asserting that `refundPacketToken` was invoked (it will not be, demonstrating the bug). [6](#0-5)

	if err != nil {
		return err
	}

	classTrace := types.ParseClassTrace(data.ClassId)
	voucherClassID := classTrace.IBCClassID()

	isAwayFromOrigin := types.IsAwayFromOrigin(packet.GetSourcePort(),
		packet.GetSourceChannel(), data.ClassId)

	escrowAddress := types.GetEscrowAddress(packet.GetSourcePort(), packet.GetSourceChannel())

	if isAwayFromOrigin {
		// unescrow tokens back to the sender

				return err
			}
		}
	}

	return nil
