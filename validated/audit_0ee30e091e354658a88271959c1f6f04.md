Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the actual trader, enabling full allowlist bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the pool's `msg.sender` — the router contract — not the originating user. Any unprivileged trader can bypass a curated pool's swap allowlist by routing through the public `MetricOmmSimpleRouter`, because the router's address is what the extension sees as the swapper identity.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` (line 67–86).
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — at this point `msg.sender` inside the pool is the **router address**.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` (lines 230–240), forwarding the router address as `sender`.
4. `ExtensionCalling._beforeSwap` encodes `sender` (= router) and dispatches to the extension (lines 160–176).
5. `SwapAllowlistExtension.beforeSwap` receives `sender = router` and evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The check is `allowedSwapper[pool][router]`, **not** `allowedSwapper[pool][actualUser]`.

**Root cause:** The pool correctly passes `msg.sender` (its direct caller) as `sender` to the extension. When the direct caller is the router rather than the end user, the extension's allowlist lookup keys on the router address. The extension has no mechanism to recover the originating EOA.

**Why existing guards fail:** The `onlyPool` modifier in `BaseMetricExtension` only verifies the extension is called by a registered pool — it does not validate that the `sender` argument represents the economic actor. The router performs no allowlist pre-check before calling the pool.

## Impact Explanation

A pool admin who configures `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties) loses that protection entirely for any user who routes through `MetricOmmSimpleRouter`. If the router is allowlisted (the natural production configuration for a pool that wants to support router-based trading), every unprivileged address can trade on the restricted pool. This constitutes a direct admin-boundary break: the pool admin's allowlist policy is bypassed by an unprivileged public path, and the pool's curation guarantee — a core security property — is nullified.

## Likelihood Explanation

The bypass requires only that the attacker call `MetricOmmSimpleRouter.exactInputSingle` (or any multi-hop variant) targeting the restricted pool. No special privileges, flash loans, or multi-transaction setup are needed. The router is a public, permissionless contract. The condition is trivially reachable on every block and is repeatable indefinitely.

## Recommendation

The extension must recover the originating user identity rather than trusting the `sender` argument when the direct caller is a known intermediary. Two sound approaches:

1. **Pass the payer through extensionData:** The router already stores the payer in transient storage (`_getPayer()`). Encode the real payer into `extensionData` and have the extension verify it against the allowlist, with the pool or extension validating the encoding.
2. **Check at the router level:** Add an `isAllowedToSwap` pre-check in the router before calling the pool, and revert if the originating `msg.sender` is not on the allowlist for that pool.

Either fix must ensure the checked identity is the economically relevant actor (the originating EOA or delegated account), not the routing intermediary.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted
// allowedSwapper[pool][router] = true  (router is allowlisted so router-based swaps work)
// allowedSwapper[pool][attacker] = false

// Attacker (not allowlisted) calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: restrictedPool,
    tokenIn: token0,
    recipient: attacker,
    amountIn: 1000e18,
    ...
}));
// Pool.swap is called with msg.sender = router
// _beforeSwap(router, ...) → extension checks allowedSwapper[pool][router] = true → passes
// Attacker receives token1 output despite not being on the allowlist
```

A Foundry integration test deploying `SwapAllowlistExtension`, allowlisting only the router (not the attacker EOA), and asserting the attacker's `exactInputSingle` call succeeds will reproduce this unconditionally.
