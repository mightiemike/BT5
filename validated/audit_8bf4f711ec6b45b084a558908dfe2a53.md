### Title
`op_sha256_tree` (opcode 63) Permanently Unreachable Inside Softfork Guard Due to Missing Extension Wiring — (`File: src/chia_dialect.rs`)

---

### Summary

`op_sha256_tree` (opcode 63) is gated behind the `ENABLE_SHA256_TREE` flag in `ChiaDialect::op()`, but no `softfork_extension()` arm ever injects that flag. As a result, any CLVM program that invokes opcode 63 inside a softfork guard silently receives `nil` (consensus mode) or an `unimplemented operator` error (mempool mode) instead of the sha256tree hash. The operator is permanently unreachable through the softfork mechanism, mirroring the report's "handler restricted to a caller that never invokes it" pattern.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method computes a local `flags` value by OR-ing the dialect's base flags with whatever the active `OperatorSet` extension contributes:

```rust
let flags = self.flags
    | match extension {
        OperatorSet::Default => ClvmFlags::empty(),
        OperatorSet::Bls    => ClvmFlags::empty(),
        OperatorSet::Keccak => ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD,
    };
``` [1](#0-0) 

Opcode 63 is then dispatched only when `ENABLE_SHA256_TREE` is present in those flags:

```rust
63 if flags.contains(ClvmFlags::ENABLE_SHA256_TREE) => op_sha256_tree,
``` [2](#0-1) 

The only `OperatorSet` variant that injects a new flag is `Keccak`, which adds `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` — enabling `op_keccak256` (opcode 62) inside the softfork guard. No variant adds `ENABLE_SHA256_TREE`. The `softfork_extension()` function maps extension integers to `OperatorSet` values:

```rust
fn softfork_extension(&self, ext: u32) -> OperatorSet {
    match ext {
        0 => OperatorSet::Bls,
        1 => OperatorSet::Keccak,
        _ => OperatorSet::Default,
    }
}
``` [3](#0-2) 

There is no extension value that produces an `OperatorSet` variant carrying `ENABLE_SHA256_TREE`. Consequently, `ENABLE_SHA256_TREE` can only reach `op()` if the caller hard-codes it into the initial `ChiaDialect::new(flags)` call — it can never be injected by a softfork guard at runtime.

Compare the parallel for keccak256: extension 1 → `OperatorSet::Keccak` → `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` → opcode 62 dispatches. The sha256tree path has no equivalent chain. The `OperatorSet` enum itself has no `Sha256Tree` variant: [4](#0-3) 

---

### Impact Explanation

When a CLVM program executes opcode 63 inside a softfork guard (any extension value), the guard's `operator_set` is one of `Default`, `Bls`, or `Keccak`. None of these cause `ENABLE_SHA256_TREE` to be set. The match arm `63 if flags.contains(ClvmFlags::ENABLE_SHA256_TREE)` is never satisfied, so control falls to the wildcard:

```rust
_ => {
    return unknown_operator(allocator, o, argument_list, flags, max_cost);
}
``` [5](#0-4) 

- **Consensus mode** (`allow_unknown_ops = true`): `op_unknown` returns `nil` with a cost computed from the opcode bytes. The program receives `nil` where it expected a 32-byte sha256 tree hash. Any downstream condition that checks this value (e.g., a coin-ID assertion, a signature message) silently evaluates against `nil`, producing incorrect spend validation.
- **Mempool mode** (`NO_UNKNOWN_OPS`): `unknown_operator` returns `EvalErr::Unimplemented`, rejecting the transaction. A transaction that would be valid under the intended semantics is permanently rejected by the mempool.

This is a consensus/mempool divergence: the operator is wired into the dispatch table but the activation path through the softfork mechanism is broken, making sha256tree functionality permanently unavailable inside a softfork guard.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: an adversary submits CLVM bytes containing opcode 63 inside a `softfork` guard. No special privileges are required. The `run_serialized_chia_program` Python API accepts arbitrary program bytes and flags from callers: [6](#0-5) 

The flag `ENABLE_SHA256_TREE` is exported to Python, so callers are aware of it, but passing it at the top level does not help a program that needs sha256tree *inside* a softfork guard — the guard's extension mechanism is the only runtime path, and it is broken.

---

### Recommendation

Add a new `OperatorSet::Sha256Tree` variant to `src/dialect.rs` and a corresponding softfork extension integer (e.g., extension 2) in `softfork_extension()`. In the `op()` match, map `OperatorSet::Sha256Tree` to inject `ClvmFlags::ENABLE_SHA256_TREE`, mirroring the existing `OperatorSet::Keccak` → `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` pattern. This closes the wiring gap and makes opcode 63 reachable inside a softfork guard.

---

### Proof of Concept

```
; Softfork guard with extension 1 (Keccak), using opcode 63 (sha256tree)
; In consensus mode this silently returns nil instead of a hash.
(softfork
  (q . 1000)   ; declared cost
  (q . 1)      ; extension = Keccak (no ENABLE_SHA256_TREE injected)
  (q . (63 (q . 0xdeadbeef)))  ; opcode 63 = sha256tree
  (q . ()))
```

With `flags = ClvmFlags::empty()` (consensus mode), `run_program` returns `nil` and charges the unknown-op cost rather than executing `op_sha256_tree`. With `flags = ClvmFlags::NO_UNKNOWN_OPS` (mempool mode), it returns `EvalErr::Unimplemented`. In neither case does `op_sha256_tree` execute, because `ENABLE_SHA256_TREE` is never set by any `OperatorSet` variant reachable through `softfork_extension()`. [7](#0-6) [3](#0-2)
