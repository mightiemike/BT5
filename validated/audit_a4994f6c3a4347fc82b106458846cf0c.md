### Title
`py_l1_handler` Trusts Python-Supplied `hash_value` Instead of Computing Canonical L1 Handler Transaction Hash - (File: crates/native_blockifier/src/py_l1_handler.rs)

### Summary
The `py_l1_handler` function in `crates/native_blockifier/src/py_l1_handler.rs` accepts the transaction hash directly from the Python caller's `hash_value` attribute without recomputing it from the transaction fields. This is the direct sequencer analog of the external bug: the system trusts a return value from a semi-trusted external source (the Python sequencer layer) instead of computing the canonical value independently. The Rust-native path uses `L1HandlerTransaction::create()` which always computes the hash, but the Python-bridge path bypasses this invariant entirely.

### Finding Description
In `py_l1_handler`, the `tx_hash` is taken verbatim from the Python object:

```rust
// crates/native_blockifier/src/py_l1_handler.rs:36
let tx_hash = TransactionHash(py_attr::<PyFelt>(py_tx, "hash_value")?.0);
```

The transaction fields (`nonce`, `contract_address`, `entry_point_selector`, `calldata`) are correctly extracted and validated, but the hash is never recomputed from them. The function then stores the caller-supplied hash directly into the executable struct:

```rust
Ok(L1HandlerTransaction { tx, tx_hash, paid_fee_on_l1 })
```

The canonical Rust path, used by every other code path in the codebase, computes the hash from the fields:

```rust
// crates/starknet_api/src/executable_transaction.rs:395
pub fn create(raw_tx: ..., chain_id: &ChainId, paid_fee_on_l1: Fee) -> ... {
    let tx_hash = raw_tx.calculate_transaction_hash(chain_id, &raw_tx.version)?;
    Ok(Self { tx: raw_tx, tx_hash, paid_fee_on_l1 })
}
```

The same canonical computation is used in `convert_consensus_l1_handler_to_internal_l1_handler` (the Rust consensus path) and in `apollo_l1_events_types` when scraping L1 events. Only the Python bridge deviates.

The hash computation itself (`get_l1_handler_transaction_hash`) chains `L1_HANDLER` prefix, version, contract address, entry point selector, calldata hash, zero fee, chain id, and nonce via Pedersen. A Python caller supplying a mismatched `hash_value` produces an `L1HandlerTransaction` whose `tx_hash` field is inconsistent with its `tx` fields.

### Impact Explanation
The stored `tx_hash` flows into `create_tx_info()`:

```rust
// crates/blockifier/src/transaction/l1_handler_transaction.rs:43
transaction_hash: self.tx_hash,
```

This `TransactionInfo` is the authoritative source for the `get_tx_info` syscall. Any contract executing under this L1 handler will observe the Python-supplied hash as the canonical transaction hash. Contracts that use `get_tx_info().transaction_hash` for replay protection, event emission, or cross-contract authentication will receive a wrong value. The transaction receipt and emitted events will also record the wrong hash. This matches: **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

The Starknet OS does recompute and assert the hash (`%{ AssertTransactionHash %}` in `transaction_impls.cairo`), but that check runs in the prover path after blockifier execution has already committed state and emitted events under the wrong hash. The blockifier execution result — including syscall return values and emitted events — is produced before the OS assertion fires.

### Likelihood Explanation
The Python sequencer is a semi-trusted actor analogous to the periphery manager in the external report: it controls the `hash_value` field passed to the blockifier. A bug in the Python hash computation (e.g., using a deprecated hash variant, wrong chain id, or off-by-one in calldata encoding) silently produces a divergent hash that the Rust blockifier accepts without verification. The Python sequencer is still the active execution driver for the `native_blockifier` path; the crate is production code, not a test utility.

### Recommendation
Remove the trust in the Python-supplied `hash_value` and recompute the hash from the transaction fields, mirroring `L1HandlerTransaction::create()`. The `chain_id` must be threaded into `py_l1_handler` (it is already available in the `NativeBlockifier` context):

```rust
pub fn py_l1_handler(
    py_tx: &PyAny,
    chain_id: &ChainId,
) -> NativeBlockifierResult<L1HandlerTransaction> {
    let py_l1_handler_tx: PyL1HandlerTransaction = py_tx.extract()?;
    let tx = starknet_api::transaction::L1HandlerTransaction::try_from(py_l1_handler_tx)?;
    let paid_fee_on_l1 = Fee(py_attr::<u128>(py_tx, "paid_fee_on_l1")?);
    // Compute canonically; do not trust the Python-supplied hash_value.
    Ok(L1HandlerTransaction::create(tx, chain_id, paid_fee_on_l1)?)
}
```

Optionally, assert the computed hash equals the Python-supplied `hash_value` and return an error on mismatch, to surface Python-side bugs early rather than silently diverging.

### Proof of Concept
1. Python sequencer constructs a Python `L1HandlerTransaction` object with correct `nonce`, `contract_address`, `entry_point_selector`, `calldata`, but sets `hash_value` to an arbitrary felt (e.g., `0xdeadbeef`).
2. `py_l1_handler` is called; it extracts the fields correctly but stores `tx_hash = TransactionHash(0xdeadbeef)` without verification.
3. The blockifier executes the L1 handler. Inside the executing contract, `get_tx_info()` returns `transaction_hash = 0xdeadbeef` instead of the canonical Pedersen hash of the transaction fields.
4. Any event emitted by the contract that includes `get_tx_info().transaction_hash` records `0xdeadbeef` in the event data.
5. The transaction receipt stored by the sequencer carries `tx_hash = 0xdeadbeef`, diverging from what the Starknet OS would compute for the same fields — a canonicalization invariant break between the blockifier execution layer and the OS/prover layer. fn get_common_l1_handler_transaction_hash(
    transaction: &L1HandlerTransaction,
    chain_id: &ChainId,
    version: L1HandlerVersions,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    Ok(TransactionHash(
        HashChain::new()
        .chain_if_fn(|| {
            if version == L1HandlerVersions::AsInvoke {
                Some(*INVOKE)
            } else {
                Some(*L1_HANDLER)
            }
        })
        .chain_if_fn(|| {
            if version > L1HandlerVersions::V0Deprecated {
                Some(transaction_version.0)
            } else {
                None
            }
        })
        .chain(transaction.contract_address.0.key())
        .chain(&transaction.entry_point_selector.0)
        .chain(&HashChain::new().chain_iter(transaction.calldata.0.iter()).get_pedersen_hash())
        // No fee in l1 handler transaction.
        .chain_if_fn(|| {
            if version > L1HandlerVersions::V0Deprecated {
                Some(Felt::ZERO)
            } else {
                None
            }
        })
        .chain(&Felt::try_from(chain_id)?)
        .chain_if_fn(|| {
            if version > L1HandlerVersions::AsInvoke {
                Some(transaction.nonce.0)
            } else {
                None
            }
        })
        .get_pedersen_hash(),
    ))
}
