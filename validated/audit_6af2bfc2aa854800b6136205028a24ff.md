### Title
SwapAllowlistExtension gates the router's address instead of the end-user, allowing any user to bypass the swap allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. When a swap is routed through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router` and forwards `sender = router` to the extension. A pool admin who allowlists the router to support router-based swaps inadvertently opens the allowlist to every user on the network, because the extension cannot distinguish which end-user is behind the router call.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured extension: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value as the first argument of `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks that argument against the allowlist keyed by `msg.sender` (the pool) and `sender` (whoever called `pool.swap()`): [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant), the router is the direct caller of `pool.swap()`: [4](#0-3) 

So `sender` arriving at the extension is the **router address**, not the end-user. The extension has no visibility into the original `msg.sender` of the router call.

A pool admin who wants allowlisted users to be able to trade through the router must add the router to `allowedSwapper[pool]`. The moment they do, the check `allowedSwapper[pool][router]` is `true` for every call that arrives through the router, regardless of who the actual end-user is. Any non-allowlisted address can bypass the gate by routing through `MetricOmmSimpleRouter`.

The same structural problem exists for multi-hop `exactInput` and `exactOutput` paths: [5](#0-4) 

### Impact Explanation

A pool configured with `SwapAllowlistExtension` is intended to be a restricted venue — only explicitly approved addresses may trade. Once the router is allowlisted (the only way to let approved users trade through the standard periphery), the restriction is silently voided for the entire public. Any address can call `exactInputSingle` on the router targeting the restricted pool and the extension will pass. This is a direct loss of the access-control invariant the pool admin paid to enforce, and it exposes LP funds to trades from counterparties the pool was designed to exclude (e.g., non-KYC'd users, competitors, or addresses subject to sanctions screening).

### Likelihood Explanation

Every pool that (a) deploys `SwapAllowlistExtension` and (b) wants its approved users to trade through the standard router must allowlist the router. This is the normal operational path for any production restricted pool that does not require every user to call the pool contract directly. The misconfiguration is therefore not an edge case — it is the expected configuration for any pool that uses both the allowlist extension and the periphery router together.

### Recommendation

The extension must gate the **economic actor**, not the immediate caller. Two viable approaches:

1. **Router-forwarded identity via `extensionData`**: The router encodes `msg.sender` (the end-user) into `extensionData` before calling `pool.swap`. The extension decodes and checks that address when `sender` is a recognized router. This requires a convention between the router and the extension.

2. **Dedicated `tx.origin`-free forwarding field**: Add an `originator` field to the swap call path so the pool can pass the true initiator to extensions without relying on `tx.origin`. The extension then checks `originator` instead of `sender`.

Until fixed, pool admins should not allowlist the router on pools with `SwapAllowlistExtension` active; approved users must call `pool.swap` directly.

### Proof of Concept

```
Setup:
  pool P configured with SwapAllowlistExtension E
  pool admin calls E.setAllowedToSwap(P, alice, true)   // alice is approved
  pool admin calls E.setAllowedToSwap(P, router, true)  // router allowlisted so alice can use it

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: P, recipient: bob, ...})

  Execution trace:
    router.exactInputSingle  (msg.sender = bob)
      pool.swap(sender = router, ...)   // pool sees msg.sender = router
        E.beforeSwap(sender = router, ...)
          allowedSwapper[pool][router] == true  → passes, no revert

  Result: bob swaps on a pool he was never approved for.
``` [6](#0-5) [7](#0-6) [4](#0-3)
