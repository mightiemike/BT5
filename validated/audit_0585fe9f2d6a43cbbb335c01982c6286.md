### Title
`UnrestrictedSubmitBlocks` Role Is Defined as a Pause Bypass but Has No Effect on the `#[pause]` Check in `submit_blocks` — (`contract/src/lib.rs`)

---

### Summary

The `UnrestrictedSubmitBlocks` role is explicitly documented as allowing block submission even when the contract is paused, but the `#[pause]` macro applied to `submit_blocks` contains no `except(roles(...))` clause for this role. The role is therefore dead for its stated purpose: when the contract is paused, `submit_blocks` panics unconditionally for every caller, including accounts that hold `UnrestrictedSubmitBlocks`. This is a direct structural analog to the external report, where a state variable (`stopped`) was set but never checked, rendering the emergency mechanism non-functional.

---

### Finding Description

The `UnrestrictedSubmitBlocks` role is defined with the comment:

> "Allows to use contract API even after contract is paused" [1](#0-0) 

The `submit_blocks` function is decorated with `#[pause]` and `#[trusted_relayer]`: [2](#0-1) 

The `#[trusted_relayer]` macro is configured with `bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks)`: [3](#0-2) 

However, `bypass_roles` in the `#[trusted_relayer]` macro controls only who may bypass the **trusted-relayer identity check** — it has no effect on the **pause check**. The `#[pause]` macro on `submit_blocks` has no `except(roles(...))` clause, so it enforces an unconditional block when the contract is paused.

Compare this with `run_mainchain_gc`, which correctly wires its bypass role into the pause macro: [4](#0-3) 

The pattern that works for `UnrestrictedRunGC` was never applied to `UnrestrictedSubmitBlocks`. The execution path when the contract is paused is:

1. External caller (holding `UnrestrictedSubmitBlocks`) calls `submit_blocks`.
2. The `#[pause]` macro fires first; it finds no `except` clause and no bypass roles.
3. The call panics unconditionally — the `#[trusted_relayer]` bypass is never reached.
4. No block headers are submitted; the on-chain chain state freezes.

---

### Impact Explanation

When a `PauseManager` pauses the contract (e.g., in response to a suspected relayer compromise or upstream Bitcoin anomaly), the intended design is that accounts holding `UnrestrictedSubmitBlocks` can continue advancing the light-client chain state. Because the bypass is missing, this is impossible. The `mainchain_tip_blockhash` and `headers_pool` freeze at the block height at which the pause was triggered. Any downstream NEAR contract calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for transactions in blocks mined after the pause will receive a panic (`"cannot find requested transaction block"` or `"block does not belong to the current main chain"`), making SPV verification unavailable for the entire pause window. The `UnrestrictedSubmitBlocks` role can be granted to accounts but confers no actual capability — it is a dead role. [5](#0-4) 

---

### Likelihood Explanation

The contract is designed to be paused in emergency scenarios. The `PauseManager` role exists precisely for this purpose, and the `UnrestrictedSubmitBlocks` role exists to allow continued operation during such pauses. Any real pause event will immediately expose the broken bypass: the relayer holding `UnrestrictedSubmitBlocks` will attempt to submit blocks and fail. The trigger is a normal, expected operational action (pausing), not an exotic edge case.

---

### Recommendation

Add the missing `except` clause to the `#[pause]` macro on `submit_blocks`, mirroring the pattern already used for `run_mainchain_gc`:

```rust
// Before (broken):
#[pause]
#[trusted_relayer]
pub fn submit_blocks(...)

// After (correct):
#[pause(except(roles(Role::UnrestrictedSubmitBlocks)))]
#[trusted_relayer]
pub fn submit_blocks(...)
```

This makes the `UnrestrictedSubmitBlocks` role functional for its documented purpose and aligns `submit_blocks` with the bypass pattern already correctly implemented for `run_mainchain_gc`. [4](#0-3) 

---

### Proof of Concept

1. Deploy the contract with a `PauseManager` account `pm.near` and a trusted relayer `relayer.near`. Grant `relayer.near` the `UnrestrictedSubmitBlocks` role.
2. `pm.near` calls `pa_pause_feature("submit_blocks")` (or the global pause). The contract is now paused.
3. `relayer.near` calls `submit_blocks([header])`. Expected (per role documentation): succeeds and advances the chain tip. Actual: panics with the near-plugins pause error before the `#[trusted_relayer]` bypass is evaluated.
4. The chain tip remains frozen. Any call to `verify_transaction_inclusion_v2` for a block mined after step 2 panics with `"cannot find requested transaction block"`.
5. Granting `relayer.near` the `UnrestrictedRunGC` role instead of `UnrestrictedSubmitBlocks` has no effect on `submit_blocks`, confirming the roles are not interchangeable and the bypass for `submit_blocks` is simply absent. [6](#0-5) [7](#0-6)
