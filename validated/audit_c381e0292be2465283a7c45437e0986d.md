### Impact Explanation

A farmer can craft a spend bundle whose CLVM puzzle uses `(softfork (q . 0x0001) ...)` — where `0x0001` is the two-byte atom `[0x00, 0x01]` (non-canonical encoding of 1) — and include it directly in a block. Full nodes validate blocks in consensus mode and accept it. The same transaction is rejected by every mempool node running `MEMPOOL_MODE`, so it cannot be relayed or fee-filtered through normal mempool paths. This is a confirmed consensus/mempool split with direct block-acceptance impact.

### Likelihood Explanation

The split is mechanically guaranteed by the two divergent branches in `uint_atom`. Any atom of the form `[0x00, X]` where `X & 0x80 == 0` (i.e., `X` in `0x00–0x7F`) triggers it. The canonical encoding of any value 1–127 is a single byte, so the non-canonical two-byte form is trivially constructable. No special privileges or compromised nodes are required — only the ability to submit a block (farmer role) or to craft a spend bundle.

### Recommendation

The cost argument to `softfork` should be validated with the same canonical-integer rules in both modes, or the cost atom should be normalized (leading zeros stripped) unconditionally before the `CANONICAL_INTS` check, so that consensus and mempool agree on validity. The simplest fix is to always strip leading zeros from the cost atom before checking size, regardless of `CANONICAL_INTS`, since the cost is a semantic unsigned integer and non-canonical encodings carry no additional meaning.
