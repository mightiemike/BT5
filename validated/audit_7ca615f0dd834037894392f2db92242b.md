Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates the Router (`sender`) Instead of the Economic Beneficiary (`recipient`), Breaking Per-User Swap Access Control — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which `MetricOmmPool.swap()` binds to `msg.sender` — the router address when called via `MetricOmmSimpleRouter`. This means the allowlist never evaluates the actual end user (`recipient`), producing two fund-impacting failure modes: allowlisted users are blocked from swapping through the router, and if the router is allowlisted, every user bypasses the per-user gate entirely. The sibling `DepositAllowlistExtension` correctly checks `owner` (the user), confirming the asymmetry is a bug.

## Finding Description
`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, recipient, ...)`, binding `sender` to whoever called `pool.swap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller (the router): [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` with `msg.sender = router`, so the extension always sees `sender = router`, never the end user: [4](#0-3) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` ignores the first parameter (the payer/intermediary) and checks `owner` — the economic beneficiary: [5](#0-4) 

The `recipient` parameter is available as the second argument to `beforeSwap` but is silently discarded (`address,`), while `sender` (the router) is used for the allowlist check. No existing guard compensates for this misbinding.

## Impact Explanation
**Failure mode A (broken core functionality):** A pool admin allowlists specific user EOAs. Those users call `router.exactInputSingle(...)`. The extension evaluates `allowedSwapper[pool][router]` → `false` → `NotAllowedToSwap`. The standard periphery swap flow is completely unusable for the intended audience, constituting broken core pool functionality causing loss of access to funds.

**Failure mode B (allowlist bypass, fund loss):** A pool admin allowlists the router as a trusted intermediary. Any non-allowlisted user calls `router.exactInputSingle(pool, recipient: attacker, ...)`. The extension evaluates `allowedSwapper[pool][router]` → `true` → swap executes. On a KYC-gated or restricted-counterparty pool, unauthorized parties can drain LP liquidity at oracle-derived prices, constituting direct loss of LP principal.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the documented standard periphery entry point. Any pool deploying `SwapAllowlistExtension` to gate individual users immediately hits failure mode A the first time an allowlisted user uses the router. Failure mode B is triggered by the natural admin configuration of allowlisting the router address to support both direct and router-based access. Both paths require no malicious setup and are reachable by ordinary unprivileged users.

## Recommendation
Change `beforeSwap` to gate on `recipient` (the second parameter, the economic beneficiary) instead of `sender`:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

This mirrors `DepositAllowlistExtension`, which checks `owner` (the user) rather than the payer/intermediary, enforcing the same policy regardless of whether the user calls the pool directly or through the router.

## Proof of Concept
**Failure mode B (allowlist bypass):**
1. Deploy a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Admin calls `swapExtension.setAllowedToSwap(pool, router, true)` — allowlisting the router.
3. Non-allowlisted attacker calls `router.exactInputSingle({pool, recipient: attacker, ...})`.
4. Router calls `pool.swap(recipient=attacker, ...)` with `msg.sender = router`.
5. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap executes.
6. Attacker receives output tokens from the curated pool without being individually allowlisted.

**Failure mode A (broken access for allowlisted users):**
1. Admin calls `swapExtension.setAllowedToSwap(pool, alice, true)`.
2. Alice calls `router.exactInputSingle({pool, recipient: alice, ...})`.
3. Extension evaluates `allowedSwapper[pool][router]` → `false` → `NotAllowedToSwap`.
4. Alice's swap reverts despite being explicitly allowlisted.
