Audit Report

## Title
`PriceVelocityGuardExtension.beforeSwap` resets `lastMidPriceX64` before the velocity check, enabling intra-block oracle stepping to bypass the per-block cap - (`File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap` unconditionally overwrites `lastMidPriceX64` and `lastUpdateBlock` in storage at lines 57‚Äď58, before the velocity check at lines 60‚Äď75. For same-block swaps `blockDiff = 0`, so the allowed budget is `maxChange¬≤ √ó 1` per swap. Because the `PythOracle` accepts permissionless signed price pushes via its public `fallback()`, an attacker can interleave Pyth price updates and pool swaps within a single block, stepping the reference anchor by at most `maxChange` on each iteration and achieving a cumulative intra-block price movement of N √ó `maxChange` while every individual check passes. LPs suffer direct adverse-selection losses equal to the full uncapped price movement.

## Finding Description

**Root cause ‚ÄĒ premature state write:**

In `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol` lines 57‚Äď58, the new mid-price and block number are written to storage before the guard check:

```solidity
s.lastMidPriceX64 = midPrice;          // ‚Üź anchor reset BEFORE check
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {
    ...
    uint256 blockDiff = block.number - prevBlock;   // = 0 for same-block swaps
    uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
    // allowedSq = maxChange¬≤ for every same-block swap
    if (actualSq > allowedSq) revert PriceVelocityExceeded(...);
}
```

Each swap checks only against the **previous swap's** mid-price, not the block-start price.

**Oracle update path ‚ÄĒ permissionless Pyth push:**

`smart-contracts-poc/contracts/oracles/providers/PythOracle.sol` exposes a public `fallback()` that accepts any Lazer-signed payload and stores it if the per-feed timestamp is newer than the stored one. No privileged role is required to push an update ‚ÄĒ the Lazer signature is the only trust anchor. Multiple valid signed updates with strictly increasing microsecond timestamps can exist for a single block window.

**Attack sequence (single block N):**

1. Push Pyth update P‚āÄ‚ÜíP‚āĀ (+1%) via `PythOracle.fallback()` ‚ÄĒ permissionless.
2. Call `pool.swap()` ‚Üí `_getBidAndAskPriceX64()` reads P‚āĀ ‚Üí `beforeSwap` checks |P‚āĀ‚ąíP‚āÄ|/P‚āÄ = 1% ‚Č§ 1% ‚úď, then writes `lastMidPriceX64 = P‚āĀ`.
3. Push Pyth update P‚āĀ‚ÜíP‚āā (+1%).
4. Call `pool.swap()` ‚Üí checks |P‚āā‚ąíP‚āĀ|/P‚āĀ = 1% ‚Č§ 1% ‚úď, writes `lastMidPriceX64 = P‚āā`.
5. Repeat N times.

Total movement: N √ó 1%. A direct single-step swap from P‚āÄ to P_N would revert with `PriceVelocityExceeded` (N¬≤ > 1 √ó 1).

**Existing guards are insufficient:**

- The reentrancy guard (`nonReentrant`) does not prevent sequential transactions in the same block.
- The staleness check in the price provider only rejects updates older than `MAX_TIME_DELTA`; multiple fresh Pyth updates within one block all pass.
- The `blockDiff` multiplier in `allowedSq` is designed to relax the cap across multiple blocks, but provides no protection within a single block.

## Impact Explanation

The velocity guard is the LP's primary on-chain protection against oracle-speed manipulation and adverse selection. By stepping through intermediate Pyth price updates in a single block, an attacker executes swaps at a final price N √ó `maxChangePerBlockE18` away from the block-start price ‚ÄĒ a movement the guard was explicitly configured to block. LPs are forced to trade at prices far outside the configured safety envelope with no opportunity to react or withdraw, resulting in direct principal loss. This matches the "bad-price execution" and "direct loss of LP principal" impact categories.

**Severity: High** ‚ÄĒ direct, repeatable loss of LP principal through a bypassed core protection hook, requiring no privileged role.

## Likelihood Explanation

- `PythOracle.fallback()` is a public, permissionless entry point; any actor holding a valid Lazer-signed payload (freely obtainable from Pyth's public price service) can push it.
- On Ethereum (~12 s blocks) and especially on L2s (Arbitrum/Base, ~250 ms blocks), multiple valid Pyth updates with distinct microsecond timestamps accumulate within a single block window.
- The attacker needs no privileged role, no flash loan, and no special token ‚ÄĒ only the ability to call `pool.swap()` and push Pyth updates in sequence, achievable via a single multicall or purpose-built contract.
- The attack is repeatable every block and scales linearly with the number of available intermediate Pyth updates.

## Recommendation

Track the block-start reference price separately from the per-swap reference price. Only advance `lastMidPriceX64` once per block (when `block.number > lastUpdateBlock`). All velocity checks within the same block must compare against the price recorded at the start of that block:

```solidity
// Only update the reference when entering a new block
if (block.number > prevBlock) {
    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);
}
// Always check against the block-start reference (prevMid)
```

This ensures that no matter how many swaps occur in a single block, the velocity check always measures the total movement from the block-start price.

## Proof of Concept

**Setup:**
- Pool with `PriceVelocityGuardExtension`, `maxChangePerBlockE18 = 1e16` (1%).
- `PythOracle` as the oracle backend (production configuration).
- Block N‚ąí1: oracle mid-price P‚āÄ = 100. `lastMidPriceX64 = P‚āÄ`, `lastUpdateBlock = N‚ąí1`.
- Real market moves to P‚āĀ‚āÄ ‚Čą 110 within the block window. Attacker collects 10 valid Lazer-signed updates: P‚āÄ‚ÜíP‚āĀ (1%), ‚Ä¶, P‚āČ‚ÜíP‚āĀ‚āÄ (1%), each with a strictly increasing microsecond timestamp.

**Execution (all in block N):**

**Result:** 10% total price movement in one block. A single swap from P‚āÄ to P‚āĀ‚āÄ would have triggered `PriceVelocityExceeded` (100 > 1). LPs sold at prices P‚āĀ through P‚āĀ‚āÄ while the market had already moved to P‚āĀ‚āÄ, suffering the full adverse-selection loss the guard was designed to prevent.
