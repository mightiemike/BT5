### Title
Mainchain Block Timestamp Used Instead of Fork Ancestor in Difficulty Calculation, Enabling Fork Blocks to Be Accepted at Incorrect Difficulty — (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`)

---

### Summary

The difficulty calculation for fork block submissions uses `get_header_by_height`, which always resolves to the **mainchain** block at a given height, not the fork's actual ancestor at that height. This is the same vulnerability class as the external report: a wrong/static value is substituted where a dynamic, chain-specific value is required. The result is that the expected PoW target for fork blocks is computed from incorrect timestamps, allowing an attacker to submit fork blocks that satisfy a lower difficulty than the protocol requires.

---

### Finding Description

When a fork block is submitted via `submit_blocks`, the contract calls `check_pow` → `check_target` → `get_next_work_required`. Inside `get_next_work_required` (both for Bitcoin and Dogecoin), the retarget boundary block's timestamp is fetched via:

**Dogecoin** (`contract/src/dogecoin.rs`, lines 291–295):
```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

**Bitcoin** (`contract/src/bitcoin.rs`, lines 78–86):
```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [2](#0-1) 

`get_header_by_height` is implemented as:
```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [3](#0-2) 

This always returns the **mainchain** block at the requested height. When validating a fork block whose ancestor chain diverged before the retarget boundary, the fork's actual ancestor at `height_first` is a different block with a different timestamp than the mainchain block at the same height. The difficulty calculation therefore uses the wrong `first_block_time`.

The developers themselves flagged this with a `TODO` comment acknowledging the issue: [4](#0-3) 

For Dogecoin (post-height 145,000, `difficulty_adjustment_interval = 1`), `blocks_to_go_back = 1`, so `height_first = prev_block_header.block_height - 1`. The timespan is computed as:

```
modulated_timespan = fork_prev_block.time − mainchain_block_at(fork_prev_height − 1).time
``` [5](#0-4) 

The correct computation should be:

```
modulated_timespan = fork_prev_block.time − fork_ancestor_at(fork_prev_height − 1).time
```

These two values diverge whenever the fork's ancestor at `height_first` has a different timestamp than the mainchain block at the same height — which is the normal case for any non-trivial fork.

---

### Impact Explanation

An attacker who submits a fork chain can craft the fork's block timestamps so that `fork_prev_block.time − mainchain_block_at(height_first).time` produces a large timespan. A large timespan causes `calculate_next_work_required` to return a lower difficulty (higher target), clamped at `max_timespan = retarget_timespan + retarget_timespan / 2`: [6](#0-5) 

The contract then enforces `expected_bits == block_header.bits`: [7](#0-6) 

The attacker sets `block_header.bits` to match the incorrectly computed (easier) target and mines a block satisfying it. The contract accepts the block. Repeating this across a fork chain allows the attacker to accumulate chainwork cheaply. Once the fork's `chain_work` exceeds the mainchain's, `reorg_chain` is triggered: [8](#0-7) 

This corrupts the canonical chain state: `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` are all rewritten to the attacker's fork. Any downstream SPV proof verified via `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against the corrupted mainchain will return incorrect results.

---

### Likelihood Explanation

Dogecoin is the most exposed path because every block recalculates difficulty (`difficulty_adjustment_interval = 1`), so the wrong-ancestor bug fires on every fork block submission. The attacker only needs to submit a fork that diverges one block before the target, which is a minimal setup. The `submit_blocks` entry point is payable and open to any NEAR account that meets the deposit requirement or registers as a trusted relayer (a low-barrier action). No privileged key or social engineering is required.

---

### Recommendation

Replace `get_header_by_height` in difficulty calculation with a function that walks the `prev_block_hash` chain from `prev_block_header` backward by `blocks_to_go_back` steps, staying within the fork's own ancestor chain rather than the mainchain height index. This mirrors how Bitcoin Core resolves the retarget boundary block via `GetAncestor`.

---

### Proof of Concept

1. Mainchain has blocks at heights 0…N. Mainchain block at height N−1 has timestamp `T_main`.
2. Attacker submits a fork block at height N−1 (diverging from height N−2) with timestamp `T_fork = T_main + 3600` (1 hour later, within the 2-hour future-time window).
3. Attacker submits a fork block at height N. The contract calls `get_next_work_required`:
   - `height_first = (N−1) − 1 = N−2`
   - `first_block_time = mainchain_block_at(N−2).time = T_main_N2`
   - `modulated_timespan = T_fork − T_main_N2` (uses fork block at N−1 minus mainchain block at N−2)
   - Correct would be: `T_fork − fork_ancestor_at(N−2).time`
4. If `T_main_N2` is much earlier than the fork's block at N−2 would be, `modulated_timespan` is inflated, producing a lower `expected_bits` (easier target).
5. Attacker sets `block_header.bits` to this easier value and mines a valid PoW hash satisfying it.
6. Contract accepts the block. Repeating across enough blocks triggers `reorg_chain`, corrupting the canonical chain.
