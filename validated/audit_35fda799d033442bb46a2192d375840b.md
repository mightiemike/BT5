Audit Report

## Title
Deposit Allowlist Checks `owner` Instead of `sender`, Allowing Any Non-Allowlisted Actor to Deposit into Curated Pools — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual caller/payer) and gates only on the caller-supplied `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` accepts any caller-supplied `owner` with no restriction beyond rejecting `address(0)`, a non-allowlisted attacker can set `owner` to any allowlisted address and unconditionally bypass the deposit guard.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

The extension receives `(sender, owner, ...)` but the first parameter is unnamed and entirely discarded: [2](#0-1) 

The guard evaluates `allowedDepositor[msg.sender][owner]` where `msg.sender` is the pool (correct) and `owner` is the attacker-controlled position recipient. If the attacker sets `owner = allowlisted_address`, the check resolves to `allowedDepositor[pool][allowlisted_address] = true` and the guard passes regardless of who is actually calling and paying.

`_validateOwner` in the liquidity adder only rejects `address(0)`, imposing no restriction on what `owner` a caller may supply: [3](#0-2) 

The pool itself imposes no restriction on who may call `addLiquidity` or what `owner` they supply, so the attacker can also call the pool directly, bypassing the adder entirely.

## Impact Explanation

`DepositAllowlistExtension` is the sole mechanism for pool curation — restricting which addresses may add liquidity. With this bypass, any non-allowlisted actor can inject liquidity into any bin of a curated pool by supplying an allowlisted address as `owner`. The pool admin's invariant that only allowlisted depositors can add liquidity is completely broken. The attacker's tokens are pulled and LP shares are minted to the allowlisted address; the attacker loses funds but the pool's intended composition and price curve can be disrupted by any external actor. This constitutes broken core pool functionality with direct on-chain effect.

## Likelihood Explanation

The attack requires only knowing one allowlisted address (publicly readable from `allowedDepositor`) and having tokens to spend plus a callback implementation. No privileged access, no special setup, and no oracle manipulation is required. `addLiquidity` is fully public. The attack is repeatable at any time.

## Recommendation

Check `sender` (the actual caller/payer) instead of `owner` in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If paying on behalf of an allowlisted owner is an intended use case, both `sender` and `owner` should be checked (e.g., require at least one to be allowlisted, or require both).

## Proof of Concept

```solidity
// attacker is NOT in allowedDepositor[pool]
// allowlisted_address IS in allowedDepositor[pool]

vm.prank(attacker); // attacker has tokens and implements IMetricOmmCallback
pool.addLiquidity(
    allowlisted_address, // owner = allowlisted → guard passes
    salt,
    deltas,
    callbackData,
    extensionData
);

// Assert: LP shares minted to allowlisted_address
// Assert: pool token balances increased (attacker's tokens pulled via callback)
// Assert: attacker was never in allowedDepositor[pool]
// Assert: allowlist provided zero protection
```
