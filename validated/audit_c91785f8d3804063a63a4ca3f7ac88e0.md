### Title
Unbounded Multi-Loop Iteration Over `productIds` Storage Array in `_finalizeSubaccount` Permanently Blocks Liquidation Finalization — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`_finalizeSubaccount` in `ClearinghouseLiq.sol` executes **five separate loops** over the unbounded `productIds` storage arrays returned by `spotEngine.getProductIds()` and `perpEngine.getProductIds()`. Because `BaseEngine._addOrUpdateProduct` appends to `productIds` with no maximum-size guard, the cumulative gas cost of these loops grows without bound. Once the product count is large enough, every call to `liquidateSubaccountImpl` with `txn.productId == type(uint32).max` (the finalization path) will revert, permanently preventing bad-debt socialization and protocol insolvency resolution.

---

### Finding Description

`BaseEngine._addOrUpdateProduct` unconditionally pushes a new entry onto the `productIds` storage array whenever a new product is registered, with no upper-bound check: [1](#0-0) 

There is no `MAX_PRODUCTS` constant or `require(productIds.length < MAX, ...)` guard anywhere in the codebase.

`_finalizeSubaccount` is entered whenever a liquidator submits a `LiquidateSubaccount` transaction with `txn.productId == type(uint32).max`. Inside, it performs **five full iterations** over the product arrays, each body making one or more external `STATICCALL`/`CALL` operations to the engine contracts:

- **Loop 1** (lines 301–311): iterates `spotIds`, calling `spotEngine.getRisk` + `spotEngine.getBalance` per entry.
- **Loop 2** (lines 313–320): iterates `perpIds`, calling `perpEngine.getBalance` per entry.
- **Loop 3** (lines 323–338): iterates `perpIds` again, calling `perpEngine.getBalance` + conditionally `_settlePnlAgainstLiquidator` (4 more balance updates) per entry.
- **Loop 4** (lines 346–366): iterates `perpIds` a third time, same pattern.
- **Loop 5** (lines 373–383): iterates `spotIds` again, calling `spotEngine.getBalance` + `spotEngine.getRisk` per entry. [2](#0-1) 

Each external call costs a minimum of ~2,100 gas (cold SLOAD inside the callee) plus the call overhead (~700 gas). With N spot products and M perp products, the worst-case gas for the five loops is roughly:

```
(2 × N × ~5,000) + (3 × M × ~5,000) + conditional settlement writes
```

At 30M block gas limit, this saturates at roughly 1,000–2,000 total products depending on storage warmth — a realistic scale for a growing derivatives protocol.

---

### Impact Explanation

When `_finalizeSubaccount` reverts due to gas exhaustion:

1. **Bad-debt socialization is permanently blocked.** The only path to call `perpEngine.socializeSubaccount` and `spotEngine.socializeSubaccount` for an underwater account runs through `_finalizeSubaccount`. [3](#0-2) 

2. **Insurance fund cannot cover negative quote balances.** The `insuranceCover` logic at lines 395–406 is also inside `_finalizeSubaccount` and becomes unreachable.

3. **Underwater subaccounts accumulate indefinitely.** Because finalization always reverts, insolvent positions are never closed, and their losses are never mutualized across depositors. This is a direct solvency/accounting corruption impact — not merely a gas-performance issue.

---

### Likelihood Explanation

**Medium.** The `productIds` array grows through normal, legitimate protocol operation (adding new spot and perp markets). No administrative malice is required; the protocol simply needs to reach a sufficient product count. There is no maximum-product-count guard anywhere in `BaseEngine._addOrUpdateProduct`, confirmed by the absence of any `MAX_PRODUCTS` constant or length check in the codebase. [4](#0-3) 

---

### Recommendation

1. **Add a maximum product count** in `BaseEngine._addOrUpdateProduct`:
   ```solidity
   uint256 constant MAX_PRODUCTS = 256; // tune to gas budget
   require(productIds.length < MAX_PRODUCTS, ERR_TOO_MANY_PRODUCTS);
   ```

2. **Alternatively, add a product-removal mechanism** so delisted products are pruned from `productIds`, keeping the array bounded.

3. **Refactor `_finalizeSubaccount`** to avoid redundant full-array passes — e.g., merge the three `perpIds` loops into one, reducing the per-product external-call count from ~9 to ~3.

---

### Proof of Concept

**Entry path (unprivileged liquidator):**

```
Endpoint.submitTransactions(...)
  → EndpointTx.processTransactionImpl(LiquidateSubaccount)   // EndpointTx.sol:391
      → Clearinghouse.liquidateSubaccount(txn)
          → ClearinghouseLiq.liquidateSubaccountImpl(txn)    // ClearinghouseLiq.sol:598
              → _finalizeSubaccount(txn, ...)                // triggered when txn.productId == type(uint32).max
                  → Loop 1: spotIds (N iterations × ~5k gas)
                  → Loop 2: perpIds (M iterations × ~5k gas)
                  → Loop 3: perpIds (M iterations × ~5k gas)
                  → Loop 4: perpIds (M iterations × ~5k gas)
                  → Loop 5: spotIds (N iterations × ~5k gas)
                  → OUT OF GAS → revert
``` [5](#0-4) 

**Concrete trigger:** Once the protocol lists enough markets (e.g., 500 spot + 500 perp products — plausible for a mature derivatives exchange), the five loops collectively exceed the 30M block gas limit. Every subsequent finalization attempt reverts. Underwater subaccounts can never be socialized, and the insurance fund can never cover their negative balances, resulting in permanent bad-debt accumulation.

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }
