### Title
Active Redelegation on Position Delegator Blocks v8 Migration Force-Exit, Locking Vesting Owner's Delegation — (`x/tieredrewards/keeper/force_exit.go`, `x/tieredrewards/keeper/transfer_delegation.go`)

---

### Summary

A vesting account owner who calls `TierRedelegate` before the v8 upgrade, leaving an active (still-unbonding) redelegation on the position's dedicated delegator account at migration time, causes `ForceFullExitWithDelegation` → `transferDelegationFromPosition` to return `ErrActiveRedelegation`. Because `exitVestedAccountsPositions` propagates that error without recovery, the entire `Migrate1to2` upgrade handler aborts, halting the chain and permanently locking the owner's delegation inside the position delegator account.

---

### Finding Description

**Step 1 — `TierRedelegate` creates a redelegation on `pos.DelegatorAddress`.**

`TierRedelegate` calls `ms.redelegate(ctx, delAddr, srcValAddr, dstValAddr, pos.Delegation.Shares)` where `delAddr` is `pos.DelegatorAddress`. [1](#0-0) 

`redelegate` calls `k.stakingKeeper.BeginRedelegation(ctx, delAddr, srcValAddr, dstValAddr, shares)`, which registers a redelegation entry in the staking module keyed to `pos.DelegatorAddress`. [2](#0-1) 

**Step 2 — `isRedelegating` reads that entry back.**

`isRedelegating` calls `k.stakingKeeper.GetRedelegations(ctx, delAddr, 1)` where `delAddr` is `pos.DelegatorAddress`. If the redelegation from Step 1 is still within its unbonding period, `len(reds) > 0` is `true`. [3](#0-2) 

**Step 3 — `transferDelegationFromPosition` hard-blocks on that result.**

The "Defensive" guard at lines 117–123 returns `ErrActiveRedelegation` unconditionally when `isRedelegating` is `true`, regardless of whether the subsequent operation is a `BeginRedelegation` (which the SDK restricts) or a plain `Unbond` (which the SDK does **not** restrict on active redelegations). [4](#0-3) 

**Step 4 — `ForceFullExitWithDelegation` propagates the error.** [5](#0-4) 

**Step 5 — The migration aborts with no recovery path.**

`exitVestedAccountsPositions` returns the error immediately; there is no skip-and-continue logic. [6](#0-5) 

`Migrate` propagates it to `Migrate1to2`, which propagates it to the upgrade handler. In Cosmos SDK, an upgrade handler that returns an error causes a consensus panic and chain halt. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

- The owner's delegation tokens remain inside the position's dedicated delegator account (`tieredrewards/position/<id>` module address) with no user-accessible path to recover them, because the position record still exists but the migration that would have returned the delegation has failed.
- The v8 upgrade handler aborts, halting the chain for all users until a patched binary is deployed.
- The `isRedelegating` guard is unnecessary for `Unbond`: the Cosmos SDK staking module does not prohibit unbonding from a delegator that has an active outgoing redelegation; only `BeginRedelegation` carries that restriction. The defensive check is therefore both incorrect and harmful in the force-exit context.

---

### Likelihood Explanation

- `TierRedelegate` is a standard, unprivileged `MsgServer` transaction available to any position owner.
- The Cosmos SDK default redelegation unbonding period is 21 days. Any vesting account owner who redelegated within 21 days before the v8 upgrade block will trigger this condition.
- No special timing, governance access, or key compromise is required.

---

### Recommendation

Remove or bypass the `isRedelegating` check inside `transferDelegationFromPosition` for the force-exit code path. Since the subsequent operation is `Unbond` (not `BeginRedelegation`), the SDK imposes no such restriction. The simplest fix is to add a `skipRedelegationCheck bool` parameter, or to extract a separate `forceTransferDelegationFromPosition` helper used only by `ForceFullExitWithDelegation` that omits the guard.

Alternatively, `exitVestedAccountsPositions` could skip positions with active redelegations and re-attempt them after the redelegation completes, but this requires additional migration state tracking.

---

### Proof of Concept

```
1. Create a vesting account V.
2. V calls CommitDelegationToTier → position P created with pos.DelegatorAddress = D.
3. V calls TierRedelegate on P (srcVal → dstVal).
   → BeginRedelegation(D, srcVal, dstVal) recorded in staking module.
4. Immediately (within the 21-day unbonding window) trigger the v8 upgrade.
5. Migrate1to2 → v2.Migrate → exitVestedAccountsPositions → ForceFullExitWithDelegation(P).
6. ForceFullExitWithDelegation → transferDelegationFromPosition:
     isRedelegating(D) → GetRedelegations(D, 1) → len > 0 → true
     → return ErrActiveRedelegation
7. ForceFullExitWithDelegation returns error.
8. exitVestedAccountsPositions returns error.
9. Migrate1to2 returns error → upgrade handler panics → chain halt.
10. V's delegation remains locked in D; position P is not deleted.
```

}

func (k Keeper) isRedelegating(ctx context.Context, delegatorAddress string) (bool, error) {
	delAddr, err := sdk.AccAddressFromBech32(delegatorAddress)
	if err != nil {
		return false, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}
	reds, err := k.stakingKeeper.GetRedelegations(ctx, delAddr, 1)
	if err != nil {
		return false, err
	}
	return len(reds) > 0, nil
}
