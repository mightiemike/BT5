### Title
`forceRebalanceNlpPool` Skips Health Check for `nlpPools[0]`, Allowing First NLP Pool to Become Insolvent — (`File: core/contracts/Clearinghouse.sol`)

### Summary

`Clearinghouse.forceRebalanceNlpPool` applies quote rebalances to **all** NLP pools via `_applyNlpRebalance` (starting at index 0), but the post-rebalance health check loop starts at index `i = 1`, permanently skipping the health check for `nlpPools[0]`. This is a direct analog to H-24: a validation step that fails to account for one of the affected entities, allowing a solvency invariant to be violated for that entity.

### Finding Description

`_applyNlpRebalance` iterates over all pools from index 0:

```solidity
for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
    spotEngine.updateBalance(
        QUOTE_PRODUCT_ID,
        nlpPools[i].subaccount,
        nlpPoolRebalanceX18[i]
    );
}
```

Immediately after, the health check loop starts at `i = 1`:

```solidity
for (uint128 i = 1; i < nlpPools.length; i++) {
    require(
        getHealth(nlpPools[i].subaccount, IProductEngine.HealthType.INITIAL) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
```

`nlpPools[0]` receives its rebalance delta (which can be negative — draining its quote balance) but is **never health-checked**. The `_validateNlpRebalance` call only enforces that the sum of all rebalance amounts equals zero; it does not prevent any individual pool from receiving an arbitrarily large negative delta. [1](#0-0) [2](#0-1) 

### Impact Explanation

A `ForceRebalanceNlpPool` transaction can be crafted with `nlpPoolRebalanceX18[0] = -X` and `nlpPoolRebalanceX18[1] = +X` (satisfying the zero-sum constraint). After execution:

- `nlpPools[1]` gains `X` quote → health check passes.
- `nlpPools[0]` loses `X` quote → **no health check is performed**.

`nlpPools[0]`'s subaccount can be driven to negative initial health. Because NLP pool subaccounts hold LP depositor funds, an insolvent pool 0 means LP funds backing that pool are at risk — directly matching the H-24 impact of vault insolvency and LP fund loss. [3](#0-2) 

### Likelihood Explanation

`forceRebalanceNlpPool` is submitted as a slow-mode transaction via `ContractOwner.forceRebalanceNlpPool`, which carries an `onlyOwner` guard on the `ContractOwner` helper contract. However, the Nado protocol's slow-mode path exists specifically for censorship resistance — allowing any party to submit transactions directly to the `Endpoint` without going through the sequencer. If the `Endpoint` does not independently restrict `TransactionType.ForceRebalanceNlpPool` to the owner, any user can encode and submit this transaction type directly, bypassing the `ContractOwner` guard. The `EndpointTx.sol` processing path for this transaction type contains no caller restriction beyond `onlyEndpoint` on the `Clearinghouse` side. [4](#0-3) [5](#0-4) 

### Recommendation

Change the health check loop to start at `i = 0` so that all pools — including `nlpPools[0]` — are validated after the rebalance is applied:

```solidity
for (uint128 i = 0; i < nlpPools.length; i++) {
    require(
        getHealth(nlpPools[i].subaccount, IProductEngine.HealthType.INITIAL) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
``` [3](#0-2) 

### Proof of Concept

1. Protocol has two NLP pools: `nlpPools[0]` (subaccount `P0`) and `nlpPools[1]` (subaccount `P1`). Both are healthy with 100 USDC quote each.
2. Attacker submits a `ForceRebalanceNlpPool` slow-mode transaction with `nlpPoolRebalanceX18 = [-150e18, +150e18]`.
3. `_validateNlpRebalance` passes: sum = 0.
4. `_applyNlpRebalance` executes: `P0` quote → -50 USDC (negative balance), `P1` quote → +250 USDC.
5. Health check loop runs for `i = 1` only: `P1` health passes.
6. `P0` health is **never checked** — `P0` now has negative quote balance and negative initial health.
7. LP funds deposited into pool 0 are now backed by an insolvent subaccount. [1](#0-0) [6](#0-5)
