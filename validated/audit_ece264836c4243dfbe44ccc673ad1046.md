### Title
Linked Signer Can Replace Itself via `LinkSigner` Transaction, Hijacking Subaccount Signing Authority - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.sol`, the sequencer-path handler for `TransactionType.LinkSigner` calls `validateSignedTx` with `allowLinkedSigner = true`. This permits the currently registered linked signer — a lower-privileged role — to sign and submit a `LinkSigner` transaction that overwrites the `linkedSigners[subaccount]` mapping entry with any arbitrary address. The subaccount owner (the higher-privileged role whose address is embedded in the `sender` bytes32) never consented to this change. This is a direct role-separation failure: a delegated signer can unilaterally mutate the very credential that governs who may act on behalf of the subaccount.

---

### Finding Description

The sequencer-path `LinkSigner` handler in `EndpointTx.sol` is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature(sender, getLinkedSigner(sender), digest, signature)`:

```solidity
function validateSignature(...) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if the recovered address equals **either** the subaccount owner **or** the linked signer:

```solidity
function validateSignature(...) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),
        ERR_INVALID_SIGNATURE
    );
}
``` [3](#0-2) 

Therefore, the currently registered linked signer can produce a valid `LinkSigner` transaction that sets `linkedSigners[subaccount]` to any address — including an attacker-controlled one — without the subaccount owner's knowledge or consent.

The `linkedSigners` mapping is the sole credential store that determines who may sign privileged sequencer-path transactions on behalf of a subaccount:

```solidity
mapping(bytes32 => address) internal linkedSigners;
``` [4](#0-3) 

---

### Impact Explanation

Once the linked signer replaces itself with an attacker-controlled address, that address inherits full signing authority over the subaccount for every transaction type that passes `allowLinkedSigner = true` to `validateSignedTx`. This includes `WithdrawCollateral`, `TransferQuote`, `LiquidateSubaccount`, `MintNlp`, `BurnNlp`, and `CreateIsolatedSubaccount`. The attacker can drain the subaccount's collateral, transfer quote balances to themselves, or force unfavorable liquidations. The subaccount owner loses control of their own account without having performed any action.

The corrupted state is: `linkedSigners[victim_subaccount]` is overwritten from the legitimate linked signer to an attacker-controlled address, granting that address the ability to sign any user-level transaction on behalf of the victim.

---

### Likelihood Explanation

Any address that has ever been granted linked-signer status for a subaccount can execute this attack unilaterally. No additional privileges, leaked keys, or governance capture are required. The attacker only needs to be the current linked signer and submit a sequencer-path `LinkSigner` transaction signed with their own key. This is a realistic scenario: linked signers are commonly used by trading bots, market makers, and automated strategies that operate with delegated authority.

---

### Recommendation

The `LinkSigner` transaction must only be authorized by the subaccount owner — the address whose 20-byte prefix matches `sender`. Change the `allowLinkedSigner` flag to `false` for the `LinkSigner` handler in the sequencer path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the correct separation of roles: the owner controls who is delegated, and the delegate cannot self-escalate or reassign that delegation.

---

### Proof of Concept

1. Alice owns subaccount `alice_sub` (address `0xAlice...` embedded in the bytes32 sender).
2. Alice links Bob (`0xBob`) as her linked signer: `linkedSigners[alice_sub] = 0xBob`.
3. Bob constructs a `SignedLinkSigner` transaction:
   - `tx.sender = alice_sub`
   - `tx.signer = bytes32(uint256(uint160(0xAttacker)))` (attacker-controlled address)
   - `tx.nonce = nonces[0xBob]` (current nonce for Alice's subaccount)
4. Bob signs the EIP-712 digest of this transaction with his own key (`0xBob`).
5. Bob (or the sequencer) submits this transaction via the sequencer path.
6. `validateSignedTx` recovers `0xBob`, checks it against `getLinkedSigner(alice_sub)` which returns `0xBob` — validation passes.
7. `linkedSigners[alice_sub]` is overwritten to `0xAttacker`.
8. The attacker now signs a `WithdrawCollateral` transaction for `alice_sub`, draining Alice's collateral to an arbitrary address. [5](#0-4) [1](#0-0) [3](#0-2)
