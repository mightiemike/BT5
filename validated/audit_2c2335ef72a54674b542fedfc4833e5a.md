### Title
Linked Signer Can Unilaterally Replace Itself via `LinkSigner` Transaction, Enabling Permanent Subaccount Takeover — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The sequencer-path handler for `LinkSigner` transactions in `EndpointTx.sol` calls `validateSignedTx` with `allowLinkedSigner = true`. This permits the currently-linked signer to sign a new `LinkSigner` transaction and replace itself with any attacker-controlled address. Because `WithdrawCollateral` also accepts linked-signer signatures, the attacker can immediately drain all collateral from the subaccount.

---

### Finding Description

In `EndpointTx.sol`, two separate code paths handle `LinkSigner` transactions:

**Slow-mode path** (censorship-resistance escape hatch):

```solidity
// EndpointTx.sol:232-239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(transaction[1:], (IEndpoint.LinkSigner));
    validateSender(txn.sender, sender);   // msg.sender must equal subaccount owner address
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
```

`validateSender` enforces that `msg.sender == address(uint160(bytes20(txn.sender)))`, so only the actual subaccount owner can set the linked signer via slow mode.

**Sequencer path** (normal operation):

```solidity
// EndpointTx.sol:576-590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(transaction[1:], (IEndpoint.SignedLinkSigner));
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true              // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
}
```

`validateSignedTx` with `allowLinkedSigner = true` resolves to:

```solidity
// EndpointTx.sol:172-184
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
```

And `Verifier.validateSignature` accepts the signature if the recovered address equals **either** the subaccount owner **or** the linked signer:

```solidity
// Verifier.sol:297-303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
    ((recovered == address(uint160(bytes20(sender)))) ||
     (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

This means the currently-linked signer can sign a `LinkSigner` transaction that writes a new address into `linkedSigners[subaccount]`. The linked signer is supposed to be a limited delegate (e.g., a hot wallet or API key for trading), not an entity with authority to modify its own delegation. The slow-mode path correctly restricts this to the account owner; the sequencer path does not.

Once the linked signer has replaced itself with an attacker address, the attacker can immediately sign `WithdrawCollateral` transactions — which also accept linked-signer signatures (`allowLinkedSigner = true` at line 423) — to drain all collateral. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

A compromised or malicious linked signer can:

1. Sign a `LinkSigner` transaction replacing itself with an attacker-controlled address.
2. Use the new linked signer to sign `WithdrawCollateral` (or `WithdrawCollateralV2`) transactions to drain all collateral from the subaccount.

The victim's only recovery path is the slow-mode escape hatch, which has a built-in delay (`slowModeConfig`). By the time the victim submits a slow-mode `LinkSigner` to revoke the attacker's address, the funds may already be withdrawn.

**Corrupted state:** `linkedSigners[subaccount]` is permanently overwritten to an attacker address, and the subaccount's collateral balances are drained to zero. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

Linked signers are the standard mechanism for API-key-style trading in Nado. Any user who has set a linked signer (the expected majority of active traders) is exposed. Linked signers are hot wallets with elevated operational exposure — they are more likely to be compromised than cold-storage owner keys. A malicious contract set as a linked signer (directly analogous to the original report's "contract which can change implementation") can execute this atomically with no user interaction required beyond the initial `LinkSigner` setup. [7](#0-6) 

---

### Recommendation

Change `allowLinkedSigner` to `false` for the sequencer-path `LinkSigner` handler, so that only the subaccount owner's key can authorize a linked-signer change — consistent with the slow-mode path:

```solidity
// EndpointTx.sol ~line 581
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the invariant already enforced in the slow-mode path via `validateSender`. [8](#0-7) 

---

### Proof of Concept

1. Alice owns `alice_subaccount` (address `0xAlice...`) and has set `linkedSigners[alice_subaccount] = 0xHotWallet`.
2. `0xHotWallet` is compromised by an attacker.
3. Attacker constructs a `SignedLinkSigner` transaction:
   - `tx.sender = alice_subaccount`
   - `tx.signer = bytes32(uint256(uint160(0xAttacker)))`
   - `tx.nonce = nonces[0xAlice]` (current nonce)
   - `signature` = ECDSA signature over the EIP-712 digest, signed by `0xHotWallet`
4. Attacker submits this to the sequencer (or directly via `submitTransactionsChecked`).
5. `validateSignedTx` passes because `recovered == getLinkedSigner(alice_subaccount) == 0xHotWallet`.
6. `linkedSigners[alice_subaccount]` is now set to `0xAttacker`.
7. Attacker constructs a `SignedWithdrawCollateral` transaction signed by `0xAttacker`, withdrawing all collateral to `0xAttacker`'s address.
8. `validateSignedTx` passes again because `recovered == getLinkedSigner(alice_subaccount) == 0xAttacker`.
9. Alice's collateral is fully drained. [9](#0-8) [1](#0-0)
