### Title
P2P Sync Client Accepts Attacker-Supplied `transaction_hash` Without Recomputation or Commitment Verification — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

A malicious p2p peer can send a `TransactionInBlock` protobuf message with a valid transaction body but an arbitrary `transaction_hash` field. The deserialization path takes the hash directly from the wire without recomputing it from the body, and the sync client stores it verbatim with no check against the block header's `transaction_commitment`. The result is that the node permanently stores and serves a `FullTransaction` whose `transaction_hash` is attacker-controlled, breaking every downstream lookup keyed on that hash.

---

### Finding Description

**Step 1 — Protobuf deserialization takes hash from wire verbatim.**

In `TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)`, the `tx_hash` is extracted directly from the `transaction_hash` protobuf field with no recomputation: [1](#0-0) 

There is no call to `calculate_transaction_hash` / `get_transaction_hash` / `validate_transaction_hash` anywhere in this converter.

**Step 2 — The p2p sync client explicitly skips hash validation.**

In `TransactionStreamFactory::parse_data_for_block`, the only check performed is a count against `n_transactions` from the stored block header. The hash is pushed to `block_body.transaction_hashes` with a developer-acknowledged TODO: [2](#0-1) 

The comment `// TODO(eitan): Validate transaction hash from untrusted sources` confirms the gap is known and unfixed.

**Step 3 — No `transaction_commitment` cross-check.**

The block header contains a `transaction_commitment` (a Poseidon Patricia root over `H(tx_hash, tx_signature)` for each transaction): [3](#0-2) 

The `parse_data_for_block` function reads `n_transactions` from the header but never reads or verifies `transaction_commitment` against the received hashes. The header sync also does not verify the block signature against a known sequencer public key — it only checks signature count: [4](#0-3) 

**Step 4 — Wrong hash is committed to storage.**

`write_transactions` stores the attacker-supplied hash in two tables without any guard: [5](#0-4) 

- `transaction_hash_to_idx_table`: maps `fake_hash → TransactionIndex`
- `TransactionMetadata.tx_hash`: stores `fake_hash` alongside the transaction body

The canonical hash is never inserted, so `get_transaction_by_hash(canonical_hash)` returns `None`, while `get_transaction_by_hash(fake_hash)` returns the transaction.

---

### Impact Explanation

This falls under **High — Transaction conversion or signature/hash logic binds the wrong hash to the executable payload**, and **High — RPC returns an authoritative-looking wrong value**.

Concrete consequences:
- `starknet_getTransactionByHash` with the real hash returns "not found"; with the fake hash returns the transaction.
- Events stored under `TransactionIndex` are served with the wrong hash when queried via `starknet_getTransactionReceipt` or `starknet_getEvents`.
- The stored `transaction_commitment` in the block header diverges from the hashes actually stored, making the node's view of the block internally inconsistent.
- Any downstream component (e.g., receipt commitment verification, event indexing) that relies on the stored hash being canonical is silently corrupted.

---

### Likelihood Explanation

Any peer the syncing node connects to can exploit this. The node actively queries peers for transaction data; the attacker only needs to be one of those peers. No special privileges are required. The attack is silent — the node does not detect or log the mismatch.

---

### Recommendation

In `parse_data_for_block` (or in the `TryFrom` converter), after deserializing each `FullTransaction`:

1. Recompute the canonical hash using `get_transaction_hash(transaction, chain_id, &TransactionOptions::default())`.
2. Compare it against the wire-supplied `transaction_hash`; treat a mismatch as `BadPeerError`.
3. Additionally, after collecting all transactions for a block, recompute the `transaction_commitment` (Poseidon Patricia root over `H(tx_hash, tx_signature)`) and compare it against the value stored in the block header. Treat a mismatch as `BadPeerError`.

The function `validate_transaction_hash` in `crates/starknet_api/src/transaction_hash.rs` already handles historical hash variants and is the correct entry point: [6](#0-5)

### Proof of Concept

```rust
// In crates/apollo_protobuf/src/converters/transaction_test.rs (illustrative)
use starknet_types_core::felt::Felt;
use crate::protobuf;
use starknet_api::transaction::{FullTransaction, TransactionHash};

#[test]
fn attacker_can_inject_arbitrary_transaction_hash() {
    // Build a valid InvokeV3 body protobuf
    let invoke_v3 = protobuf::transaction_in_block::InvokeV3 { /* valid fields */ };
    let fake_hash = Felt::ONE; // attacker-chosen value

    let tx_in_block = protobuf::TransactionInBlock {
        txn: Some(protobuf::transaction_in_block::Txn::InvokeV3(invoke_v3)),
        transaction_hash: Some(fake_hash.into()),
    };

    let (_, stored_hash) =
        <(Transaction, TransactionHash)>::try_from(tx_in_block).unwrap();

    // Passes: the stored hash is Felt::ONE, not the canonical hash
    assert_eq!(stored_hash.0, Felt::ONE);
    // The canonical hash would differ — no check is performed
}
```

The test passes today because `TryFrom<protobuf::TransactionInBlock>` performs no hash recomputation, and `parse_data_for_block` contains only the acknowledged-but-unimplemented TODO at line 88.
