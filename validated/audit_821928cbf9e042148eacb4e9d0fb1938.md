### Title
`ContractOwner` Cannot Receive ETH, Making `withdrawFromDirectDepositV1` Always Revert and Permanently Locking Native ETH — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.withdrawFromDirectDepositV1()` calls `DirectDepositV1.withdrawNative()`, which attempts to send ETH back to `msg.sender` (i.e., `ContractOwner`). Because `ContractOwner` defines no `receive()` or `fallback()` function, the ETH transfer always fails, the call reverts, and any native ETH held in a `DirectDepositV1` contract is permanently unrecoverable.

---

### Finding Description

`ContractOwner.withdrawFromDirectDepositV1` handles the native-ETH branch (`token == address(0)`) as follows: [1](#0-0) 

It calls `DirectDepositV1.withdrawNative()`, which is: [2](#0-1) 

Inside `withdrawNative()`, `msg.sender` is `ContractOwner`. The low-level call `payable(msg.sender).call{value: balance}("")` attempts to push ETH into `ContractOwner`. A codebase-wide search confirms that **only** `DirectDepositV1` and `MockWrappedNative` define a `receive()` function: [3](#0-2) 

`ContractOwner` inherits solely from `EIP712Upgradeable` and `OwnableUpgradeable`, neither of which define `receive()` or `fallback()`. [4](#0-3) 

Therefore the `call` returns `success = false`, `withdrawNative()` reverts with `"Failed to transfer native token to owner"`, and the entire `withdrawFromDirectDepositV1` call reverts.

ETH can legitimately accumulate inside a `DirectDepositV1` instance in two ways:

1. **Constructor failure path** — if the initial wrapping call to `wrappedNative` fails, the constructor emits `NativeTokenTransferFailed` and continues without reverting, leaving ETH stranded: [5](#0-4) 

2. **Force-send via `selfdestruct`** — ETH pushed this way bypasses `receive()` entirely and accumulates silently.

---

### Impact Explanation

Any native ETH that ends up in a `DirectDepositV1` contract is permanently unrecoverable. The only designated recovery path — `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` — unconditionally reverts because `ContractOwner` cannot accept ETH. The ETH is locked with no alternative withdrawal mechanism.

---

### Likelihood Explanation

The constructor wrapping failure path is realistic: if `wrappedNative` is temporarily non-functional or misconfigured at deployment time, ETH is silently retained. Force-sends via `selfdestruct` are always possible. Once ETH is in the contract, the recovery function is permanently broken regardless of how many times it is called.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept the ETH forwarded by `DirectDepositV1.withdrawNative()`. The existing logic in `withdrawFromDirectDepositV1` already measures the pre/post balance delta and forwards the received ETH onward to `msg.sender`, so only the missing `receive()` hook needs to be added:

```solidity
// In ContractOwner.sol
receive() external payable {}
```

This mirrors the fix applied in the referenced Wormhole report (adding `receive()` to `SpokeVault`).

---

### Proof of Concept

1. Deploy `ContractOwner` (no `receive()` function present).
2. Deploy a `DirectDepositV1` for some `subaccount`; ensure it holds ETH (e.g., by making the `wrappedNative` call fail in the constructor, or by force-sending ETH via `selfdestruct`).
3. Call `contractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` as the owner.
4. Execution path: `ContractOwner` → `DirectDepositV1.withdrawNative()` → `ContractOwner.call{value: balance}("")` → `success = false` → revert `"Failed to transfer native token to owner"`.
5. ETH remains permanently locked in `DirectDepositV1`.
