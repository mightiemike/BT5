### Title
Hardcoded Dogecoin Mainnet Protocol-Switch Heights Applied to All Networks — (`File: contract/src/dogecoin.rs`)

### Summary

Two Dogecoin mainnet-specific block-height constants (`145_000` for DigiShield activation and `157_500` for the min-difficulty reset) are hardcoded directly in the difficulty-calculation functions in `contract/src/dogecoin.rs`. They are not part of `DogecoinConfig` and are never branched on `self.network`. When the contract is deployed for `Network::Testnet`, the wrong difficulty algorithm is applied, allowing an attacker to submit blocks whose PoW does not satisfy the correct testnet difficulty rules, corrupting the canonical chain and causing `verify_transaction_inclusion` to return false positives.

### Finding Description

`DogecoinConfig` in `btc-types/src/network.rs` correctly differentiates mainnet from testnet for several parameters (`pow_allow_min_difficulty_blocks`, `pow_limit`, etc.): [1](#0-0) 

However, two critical protocol-switch heights that govern which difficulty algorithm is used are hardcoded as bare integer literals inside `contract/src/dogecoin.rs` and are never conditioned on the stored `network` field:

**DigiShield activation height** — switches the difficulty-adjustment interval from the legacy value (`config.difficulty_adjustment_interval`) to `1`: [2](#0-1) 

**Min-difficulty reset height** — gates whether a testnet block is allowed to use the minimum-difficulty target: [3](#0-2) 

A `grep` across the entire repository confirms these two literals appear **only** in `contract/src/dogecoin.rs` and are absent from `DogecoinConfig` or any network-keyed lookup. The comment on line 218 even labels `157_500` a "Magic number at which reset protocol switches," acknowledging it is a chain-specific constant — yet it is never parameterised.

On Dogecoin testnet the DigiShield activation and the min-difficulty reset occur at different block heights than on mainnet. Because the contract reads `self.network` to build a `DogecoinConfig` but then ignores it for these two height thresholds, a testnet deployment silently applies mainnet rules.

### Impact Explanation

When the contract is initialised with `Network::Testnet`:

1. **Wrong difficulty algorithm selected.** For testnet blocks in the height range where mainnet and testnet disagree on DigiShield activation, `get_next_work_required` computes `expected_bits` using the wrong interval (either `1` or `config.difficulty_adjustment_interval`). The computed target diverges from the real testnet consensus rule.

2. **Invalid blocks accepted / valid blocks rejected.** `submit_block_header` compares the submitted header's `bits` field against `expected_bits`. With the wrong expected value, a relayer (or attacker) can submit headers whose `bits` satisfy the miscalculated target but not the real testnet rule. These headers pass `check_pow` and are stored in `headers_pool` as part of the canonical chain.

3. **`verify_transaction_inclusion` returns false positives.** Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` on a block that was accepted under the wrong difficulty rule receive `true` for a transaction that was never confirmed under valid consensus, enabling fraudulent cross-chain proofs. [4](#0-3) 

### Likelihood Explanation

The contract explicitly supports `Network::Testnet` as a first-class deployment option (it is a named variant of the `Network` enum stored in contract state and passed through `InitArgs`). Any operator deploying the Dogecoin build against testnet triggers the bug automatically — no special attacker capability is required beyond being a relayer or any account that can call `submit_blocks`. [5](#0-4) [6](#0-5) 

### Recommendation

Add the two protocol-switch heights as fields in `DogecoinConfig` and populate them with the correct per-network values in `get_dogecoin_config`:

```rust
pub struct DogecoinConfig {
    // ... existing fields ...
    pub digishield_activation_height: u64,   // mainnet: 145_000; testnet: <testnet value>
    pub min_difficulty_reset_height: u64,    // mainnet: 157_500; testnet: <testnet value>
}
```

Then replace the hardcoded literals in `contract/src/dogecoin.rs`:

```rust
// get_next_work_required
let new_difficulty_protocol =
    prev_block_header.block_height >= config.digishield_activation_height;

// allow_min_difficulty_for_block
if prev_block_header.block_height < config.min_difficulty_reset_height {
    return false;
}
```

This mirrors the fix recommended in the reference report: move the chain-specific constant out of a hardcoded position and into the per-deployment configuration struct.

### Proof of Concept

1. Deploy the Dogecoin-feature WASM with `InitArgs { network: Network::Testnet, … }`.
2. Craft a sequence of block headers whose `bits` field satisfies the **mainnet** DigiShield difficulty rule for heights around 145,000 but does **not** satisfy the real testnet rule.
3. Call `submit_blocks` with these headers. Because `get_next_work_required` uses the hardcoded `145_000` threshold, `expected_bits` matches the attacker-supplied `bits`, and `check_pow` passes.
4. Call `verify_transaction_inclusion_v2` referencing a transaction in one of these accepted blocks. The function returns `true`, producing a fraudulent SPV proof backed by a block that was never valid under real testnet consensus. [7](#0-6) [8](#0-7)

// source https://github.com/dogecoin/dogecoin/blob/2c513d0172e8bc86fe9a337693b26f2fdf68a013/src/pow.cpp#L17
fn allow_min_difficulty_for_block(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
) -> bool {
    // check if the chain allows minimum difficulty blocks
    if !config.pow_allow_min_difficulty_blocks {
        return false;
    }

    // Dogecoin: Magic number at which reset protocol switches
    // check if we allow minimum difficulty at this block-height
    if prev_block_header.block_height < 157_500 {
        return false;
    }

    // Allow for a minimum block time if the elapsed time > 2*nTargetSpacing
    block_header.time > prev_block_header.block_header.time + config.pow_target_spacing * 2
}
