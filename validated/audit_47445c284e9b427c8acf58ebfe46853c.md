### Title
Protobuf `ValidResourceBounds` Deserialization Collapses `AllResources` to `L1Gas` Variant, Producing Wrong Transaction Hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation silently downgrades a V3 `AllResources` transaction to the `L1Gas` variant whenever `l2_gas` and `l1_data_gas` are both zero. Because `get_tip_resource_bounds_hash` branches on this variant to decide how many resource felts to include in the Poseidon preimage, the hash computed after a protobuf round-trip differs from the hash the signer originally committed to. Any node that recomputes the transaction hash from the deserialized body (e.g., during P2P block sync validation) will obtain a wrong value, causing valid transactions to be rejected or stored with an inconsistent hash/body pair.

---

### Finding Description

**Step 1 – Origination at the gateway.**
Every `RpcTransaction` carries `resource_bounds: AllResourceBounds`. The gateway converts it to `InternalRpcTransactionWithoutTxHash` and immediately computes the canonical hash:

```
tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)
``` [1](#0-0) 

For an invoke V3 transaction this calls `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(self.resource_bounds)`: [2](#0-1) 

**Step 2 – Hash preimage for `AllResources`.**
`get_tip_resource_bounds_hash` always appends the `l1_data_gas` felt when the variant is `AllResources`, even when `l1_data_gas` is all-zero:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [3](#0-2) 

So the signed hash is `Poseidon(tip, l1_gas_felt, l2_gas_felt(0), l1_data_gas_felt(0))` — **three** resource felts.

**Step 3 – Protobuf serialization.**
`From<ValidResourceBounds> for protobuf::ResourceBounds` faithfully encodes all three fields for `AllResources`: [4](#0-3) 

**Step 4 – Protobuf deserialization — the bug.**
`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` (used in the P2P sync path for `Transaction` objects) applies a heuristic: if `l2_gas` and `l1_data_gas` are both zero, it returns `L1Gas` instead of `AllResources`:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← wrong variant for a V3 AllResources tx
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [5](#0-4) 

**Step 5 – Wrong hash on recomputation.**
After deserialization, any call to `get_tip_resource_bounds_hash` on the recovered `L1Gas` variant produces `Poseidon(tip, l1_gas_felt, l2_gas_felt(0))` — only **two** resource felts. This is a different Poseidon output from the original three-felt preimage, so the recomputed hash diverges from the hash the signer committed to.

The `validate_transaction_hash` function, which is called during central sync and potentially during P2P block validation, would then reject the transaction as having an invalid hash: [6](#0-5) 

**Contrast with the RPC/mempool path.**
The mempool propagation path uses `InternalRpcTransaction`, which stores `resource_bounds: AllResourceBounds` directly and uses a separate protobuf converter (`TryFrom<protobuf::ResourceBounds> for AllResourceBounds`) that always produces `AllResources`. That path is unaffected. The bug is isolated to the `Transaction`-level protobuf converter used in the P2P block-sync path. [7](#0-6) 

---

### Impact Explanation

A V3 transaction with non-zero `l1_gas` but zero `l2_gas` and `l1_data_gas` is valid and accepted by the gateway (the stateless validator only rejects transactions where *all* resource bounds are zero). When such a transaction is included in a block and that block is later synced via P2P, the receiving node deserializes the transaction with the wrong `ValidResourceBounds` variant. Any subsequent hash recomputation from the stored body produces a value that does not match the block's transaction commitment, causing the block to be rejected or the stored transaction to have an inconsistent hash/body pair. This maps to:

- **High**: P2P sync rejects valid blocks containing affected transactions.
- **High**: RPC execution / tracing / simulation that recomputes the transaction hash from the stored body returns an authoritative-looking wrong value.

---

### Likelihood Explanation

The trigger condition — a V3 transaction with non-zero `l1_gas` and zero `l2_gas`/`l1_data_gas` — is reachable by any unprivileged user submitting a transaction to the gateway. The gateway's `validate_resource_bounds` check only rejects transactions where `max_possible_fee(Tip::ZERO) == 0`, which passes when `l1_gas` is non-zero. The protobuf round-trip then silently changes the variant, and no downstream check catches the discrepancy before hash recomputation.

---

### Recommendation

Remove the heuristic variant-downgrade from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Instead, preserve the original variant by encoding it explicitly in the protobuf message (e.g., a boolean or enum field indicating whether the transaction is pre-0.13.3 `L1Gas` or post-0.13.3 `AllResources`). Alternatively, always deserialize as `AllResources` when all three fields are present in the protobuf message, regardless of whether `l2_gas` and `l1_data_gas` are zero, and reserve `L1Gas` only for messages that explicitly omit those fields (i.e., `l1_data_gas` is `None`).

---

### Proof of Concept

1. Construct a V3 invoke transaction with:
   - `l1_gas = ResourceBounds { max_amount: 1000, max_price_per_unit: 1 }`
   - `l2_gas = ResourceBounds::default()` (zeros)
   - `l1_data_gas = ResourceBounds::default()` (zeros)

2. Submit to the gateway. It is accepted. The stored `tx_hash` is computed via `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `AllResources` → Poseidon over **three** resource felts.

3. Serialize the corresponding `Transaction` object to protobuf via `From<ValidResourceBounds> for protobuf::ResourceBounds`. All three fields are present.

4. Deserialize via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Because `l2_gas.is_zero() && l1_data_gas.is_zero()`, the result is `ValidResourceBounds::L1Gas(l1_gas)`.

5. Call `get_tip_resource_bounds_hash` on the deserialized value. It now hashes over **two** resource felts (no `l1_data_gas`). The output differs from the original `tx_hash`.

6. `validate_transaction_hash` returns `false` for the deserialized transaction, causing the block containing it to be rejected by any node that validates transaction hashes after P2P deserialization.
