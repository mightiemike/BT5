### Title
Linked Signer Can Self-Escalate Authorization State via `LinkSigner` with `allowLinkedSigner = true` — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In the sequencer path, `LinkSigner` transactions are validated with `allowLinkedSigner = true`. This means the current linked signer — not just the subaccount owner — can sign a `LinkSigner` transaction to overwrite the `linkedSigners` mapping with a new address it controls. The authorization state governing who may act on behalf of a subaccount is stored without the access restriction that only the subaccount owner should be able to modify it.

---

### Finding Description

The `linkedSigners` mapping in `EndpointStorage.sol` is the on-chain authorization record that determines which address (beyond the subaccount owner) may sign sequencer-path transactions for a given subaccount. [1](#0-0) 

In the sequencer path, when a `LinkSigner` transaction is processed, `validateSignedTx` is called with `allowLinkedSigner = true`: [2](#0-1) 

`validateSignedTx` with `allowLinkedSigner = true` passes the current linked signer to `validateSignature`, which accepts a signature from either the subaccount owner **or** the linked signer: [3](#0-2) [4](#0-3) 

This means the current linked signer can sign a `LinkSigner` transaction that replaces itself with any new address. There is no restriction preventing the linked signer from mutating its own authorization record.

By contrast, the slow-mode path for `LinkSigner` correctly restricts this operation to the subaccount owner only via `validateSender`, which checks `msg.sender == address(uint160(bytes20(txn.sender)))`: [5](#0-4) [6](#0-5) 

The sequencer path lacks this restriction, creating an asymmetry: the slow-mode path enforces owner-only modification of the linked signer, but the sequencer path does not.

---

### Impact Explanation

A compromised or malicious linked signer can:

1. Sign a `LinkSigner` transaction replacing itself with an attacker-controlled address.
2. The attacker's address becomes the new linked signer and retains full sequencer-path signing authority over the subaccount.
3. If the original owner attempts to revoke the linked signer via the sequencer path (by submitting a `LinkSigner` transaction signed by themselves), the attacker can front-run or race this by submitting another `LinkSigner` transaction signed by the current (attacker-controlled) linked signer, re-installing their address.
4. The original owner is forced to use the slow-mode path to recover, which introduces latency and censorship-resistance concerns.

Additionally, the linked signer already has `allowLinkedSigner = true` for `WithdrawCollateral` (v1): [7](#0-6) 

This means the attacker-installed linked signer can immediately initiate withdrawals to the subaccount owner's address (the `sendTo` is hardcoded to `address(0)`) and continue trading, liquidating, minting/burning NLP, and transferring quote — all without the original owner's consent — until slow-mode recovery completes.

The corrupted state is: `linkedSigners[subaccount]` — the on-chain authorization record for the subaccount — is set to an attacker-controlled address without the subaccount owner's authorization.

---

### Likelihood Explanation

Linked signers are typically hot wallets or API keys used for automated trading. These are high-value targets for compromise. A leaked or phished linked signer key is a realistic scenario. Once compromised, the attacker can immediately execute this self-escalation to persist access, making the attack practical and requiring no privileged protocol access.

---

### Recommendation

**Short term:** Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in the sequencer path. Only the subaccount owner (the address embedded in the subaccount `bytes32`) should be permitted to modify the linked signer. This mirrors the restriction already enforced in the slow-mode path.

```solidity
// EndpointTx.sol, sequencer path for LinkSigner
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // was: true — linked signer must NOT be able to change itself
);
```

**Long term:** Empirically validate that no sequencer-path transaction type that mutates authorization state (`LinkSigner`) accepts a linked signer signature. Audit all `validateSignedTx` call sites and confirm that `allowLinkedSigner = true` is only used for transaction types where linked signer authority is intentional and scoped (e.g., trading, not authorization mutation).

---

### Proof of Concept

1. Alice owns subaccount `S` and sets linked signer `LS_alice` (a hot wallet) via slow mode.
2. Mallory compromises `LS_alice` (e.g., via a leaked API key).
3. Mallory uses `LS_alice` to sign a `SignedLinkSigner` transaction: `{ sender: S, signer: LS_mallory, nonce: N }`.
4. Mallory submits this to the sequencer. The sequencer path processes it: `validateSignedTx(..., allowLinkedSigner = true)` accepts the signature from `LS_alice` (the current linked signer). `linkedSigners[S]` is overwritten with `LS_mallory`.
5. Alice notices and submits a `LinkSigner` transaction signed by herself (the subaccount owner) to revoke the linked signer. Mallory, watching the mempool/sequencer queue, immediately submits another `LinkSigner` signed by `LS_mallory` to re-install her address.
6. Alice is forced to use the slow-mode path to override `linkedSigners[S]`, which requires waiting for slow-mode processing and paying the slow-mode fee.
7. During the window, `LS_mallory` retains full sequencer-path signing authority over `S`, including the ability to sign `WithdrawCollateral` transactions. [2](#0-1) [6](#0-5)

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
