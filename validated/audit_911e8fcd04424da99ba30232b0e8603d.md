Audit Report

## Title
Single-slot `pendingHighWatermark` per pool silently overwrites prior bin proposals, permanently losing stop-loss watermark updates — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`pendingHighWatermark` is declared as `mapping(address pool => PendingHighWatermarks)`, storing only one pending watermark proposal per pool regardless of `binIdx`. A second call to `proposeOracleStopLossHighWatermarks` for a different `binIdx` unconditionally overwrites the first proposal with no revert, no guard, and no warning event. When `executeOracleStopLossHighWatermarks` is called, only the second bin's watermark is applied; the first bin's update is permanently discarded, leaving it with a stale watermark that may fail to block value-draining swaps.

## Finding Description

The root cause is a key-dimension mismatch between the pending and live watermark mappings:

```solidity
// Line 39 — single slot per pool
mapping(address pool => PendingHighWatermarks) public pendingHighWatermark;

// Line 40 — per (pool, binIdx)
mapping(address pool => mapping(int8 binIdx => BinHighWatermarks)) public highWatermarks;
```

`proposeOracleStopLossHighWatermarks` writes unconditionally to `pendingHighWatermark[pool_]` with no existence check:

```solidity
// Lines 163–164
pendingHighWatermark[pool_] =
  PendingHighWatermarks({token0: newHwmToken0, token1: newHwmToken1, binIdx: binIdx, executeAfter: executeAfter});
```

A second call with a different `binIdx` silently replaces the first entry. `executeOracleStopLossHighWatermarks` then reads the single slot and applies it only to the second bin:

```solidity
// Lines 170–175
PendingHighWatermarks memory pending = pendingHighWatermark[pool_];
if (pending.executeAfter == 0) revert OracleStopLossNoPendingHighWatermark(pool_);
_requireElapsed(pending.executeAfter);
highWatermarks[pool_][pending.binIdx] =
  BinHighWatermarks({token0: pending.token0, token1: pending.token1, lastDecayTs: uint32(block.timestamp)});
delete pendingHighWatermark[pool_];
```

`cancelOracleStopLossHighWatermarks` compounds the issue: it deletes the single slot without inspecting `binIdx`, so a cancel after an overwrite silently discards the second proposal while the admin believes they cancelled the first:

```solidity
// Lines 179–183
function cancelOracleStopLossHighWatermarks(address pool_) external onlyPoolAdmin(pool_) {
    if (pendingHighWatermark[pool_].executeAfter == 0) revert OracleStopLossNoPendingHighWatermark(pool_);
    delete pendingHighWatermark[pool_];
    emit OracleStopLossHighWatermarkCancelled(pool_);
}
```

No existing guard prevents the overwrite. The `_checkAndUpdateWatermarks` path at lines 258–285 compares current oracle-derived metrics against `highWatermarks[pool_][binIdx]`; if that entry was never updated (because its proposal was overwritten), the comparison uses the stale stored value, causing the stop-loss to fire at the wrong level or not at all.

## Impact Explanation

The stop-loss extension is the primary on-chain mechanism capping LP value drawdown. When a bin's watermark update is silently lost, the stale (looser) watermark remains in `highWatermarks[pool_][binIdx]`. Swaps traversing that bin are evaluated against the stale floor: if the intended tighter watermark would have blocked the swap, it proceeds, draining LP principal beyond the configured stop-loss threshold. This is a direct, fund-impacting cap failure — the stop-loss guard that LPs rely on to bound drawdown is silently misconfigured by a routine admin operation, meeting Sherlock's threshold for a High-severity loss of user principal.

## Likelihood Explanation

The trigger is the pool admin performing routine multi-bin watermark management — proposing updates for two different bins within the same timelock window. No malicious intent is required; the overwrite occurs automatically on the second legitimate admin call. When the timelock is zero (permitted by the contract), back-to-back proposals for different bins are a normal operational pattern. The pool admin is semi-trusted and this constitutes a fund-impacting cap failure, satisfying the contest's semi-trusted admin validity criterion.

## Recommendation

Change `pendingHighWatermark` to include `binIdx` as a second key dimension, matching the live `highWatermarks` layout:

```solidity
// Before
mapping(address pool => PendingHighWatermarks) public pendingHighWatermark;

// After
mapping(address pool => mapping(int8 binIdx => PendingHighWatermarks)) public pendingHighWatermark;
```

Update `proposeOracleStopLossHighWatermarks`, `executeOracleStopLossHighWatermarks`, and `cancelOracleStopLossHighWatermarks` to pass `binIdx` as the second key. Optionally add a guard in `propose` that reverts if a non-zero `executeAfter` already exists for the `(pool, binIdx)` pair, preventing silent overwrites.
