### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades V3 Transactions to `L1Gas` Variant, Producing a Divergent Transaction Hash - (File: crates/apollo_protobuf/src/converters/transaction.rs)

### Summary

The protobuf round-trip for `ValidResourceBounds` silently converts `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` into `L1Gas(X)`. Because `get_tip_resource_bounds_hash` hashes a different number of Poseidon elements for each variant (4 vs 3), the transaction hash recomputed from a protobuf-deserialized `InvokeTransactionV3` differs from the hash computed at gateway admission. Any code path that recomputes the hash from the deserialized `Transaction` object — including `validate_transaction_hash` used during block sync — will produce a wrong value and bind the wrong hash to the transaction.

### Finding Description

**Root cause — protobuf deserialization (`transaction.rs`)**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies a heuristic: if both `l2_gas` and `l1_data_gas` are zero it returns `ValidResourceBounds::L1Gas`, otherwise `ValidResourceBounds::AllResources`. [1](#0-0) 

A V3 transaction submitted with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is perfectly valid (zero L2/data-gas bounds are common in client-side proving transactions, as confirmed by production test data). [2](#0-1) 

**Hash preimage divergence (`transaction_hash.rs`)**

`get_tip_resource_bounds_hash` hashes a different number of elements depending on the variant:

- `AllResources(X, 0, 0)` → `Poseidon(tip, L1_packed(X), L2_packed(0), L1_DATA_packed(0))` — **4 elements**
- `L1Gas(X)` → `Poseidon(tip, L1_packed(X), L2_packed(0))` — **3 elements** [3](#0-2) 

These two Poseidon hashes are distinct for any non-trivial `X`.

**Gateway admission always uses `AllResources`**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`. `convert_rpc_tx_to_internal` wraps it in `ValidResourceBounds::AllResources` via `From<InternalRpcInvokeTransactionV3> for InvokeTransactionV3`, then calls `calculate_transaction_hash` → `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash`. The stored `tx_hash` is therefore the 4-element hash H₁. [4](#0-3) [5](#0-4) 

**Protobuf serialization preserves zero values**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes `AllResources(X, 0, 0)` with `l2_gas = 0` and `l1_data_gas = 0` on the wire. [6](#0-5) 

**Deserialization produces `L1Gas` variant**

On the receiving side, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` sees `l2_gas.is_zero() && l1_data_gas.is_zero()` and returns `L1Gas(X)`. The `InvokeTransactionV3` stored in the deserialized `FullTransaction` now carries the wrong variant. [7](#0-6) 

**`validate_transaction_hash` recomputes and mismatches**

`validate_transaction_hash` calls `get_transaction_hash` on the deserialized `Transaction`. For the `L1Gas` variant it computes H₂ (3-element Poseidon). H₂ ≠ H₁, so the function returns `false` — a valid, sequenced transaction is declared to have an invalid hash. [8](#0-7) 

### Impact Explanation

Any component that calls `validate_transaction_hash` on a `Transaction` object obtained via protobuf deserialization will compute the wrong hash for every V3 invoke with zero L2/data-gas bounds. This matches the allowed impact: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."**

Concretely, a syncing node that validates transaction hashes after P2P block sync will reject valid blocks containing such transactions, causing a chain-split between the proposer and syncing nodes. Additionally, any RPC or tracing component that re-derives the hash from the stored `Transaction` object will return an authoritative-looking wrong value.

### Likelihood Explanation

The trigger is unprivileged: any user submitting a V3 invoke with zero L2 and data-gas bounds (a normal pattern for client-side proving transactions, as seen in production block data) produces the affected transaction type. No special permissions or malicious peer are required. The protobuf path is exercised on every P2P block sync.

### Recommendation

Remove the `L1Gas`-vs-`AllResources` heuristic from the protobuf deserialization of `ValidResourceBounds` for V3 transactions. V3 transactions must always deserialize to `AllResources`, regardless of whether the individual gas bounds are zero. The `L1Gas` variant should only be produced when deserializing pre-V3 (V1/V2) transactions that structurally cannot carry three resource bounds.

Concretely, in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, always return `AllResources` when all three fields are present (even if zero), and only fall back to `L1Gas` when `l1_data_gas` is absent (the pre-0.13.3 wire format).

### Proof of Concept

```
1. Submit RpcInvokeTransactionV3 with:
     resource_bounds = AllResourceBounds { l1_gas: (amount=100, price=1), l2_gas: (0,0), l1_data_gas: (0,0) }
     tip = 0

2. Gateway calls get_tip_resource_bounds_hash(AllResources(...)):
     resource_felts = [L1_packed(100,1), L2_packed(0,0), L1_DATA_packed(0,0)]   // 3 felts
     H1 = Poseidon(0 /*tip*/, L1_packed, L2_packed, L1_DATA_packed)              // 4-element hash

3. tx_hash = H1 is stored in InternalRpcTransaction and committed to the block.

4. Block is served to a syncing peer via P2P protobuf:
     protobuf::ResourceBounds { l1_gas: (100,1), l2_gas: (0,0), l1_data_gas: (0,0) }

5. Peer deserializes via TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
     l1_data_gas.is_zero() && l2_gas.is_zero()  →  ValidResourceBounds::L1Gas((100,1))

6. Peer calls validate_transaction_hash on the deserialized Transaction:
     get_tip_resource_bounds_hash(L1Gas((100,1))):
       resource_felts = [L1_packed(100,1), L2_packed(0,0)]                       // 2 felts
       H2 = Poseidon(0 /*tip*/, L1_packed, L2_packed)                            // 3-element hash

7. H2 ≠ H1  →  validate_transaction_hash returns false
   Peer rejects the block as containing a transaction with an invalid hash.
```

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
