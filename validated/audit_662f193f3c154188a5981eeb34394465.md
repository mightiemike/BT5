### Title
Missing Minimum Confirmations Enforcement Allows Zero-Confirmation SPV Proof Acceptance - (File: `contract/src/lib.rs`)

### Summary
The `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` functions accept a caller-supplied `confirmations` value with no lower-bound enforcement. Passing `confirmations = 0` trivially satisfies the confirmation depth check, causing the function to return `true` for any transaction in any mainchain block regardless of how recently it was submitted. Downstream NEAR contracts that consume this result as a security gate can be exploited.

### Finding Description

`ProofArgs.confirmations` is a `u64` with only an upper-bound check:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

The actual depth check is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

When `args.confirmations = 0`, the right-hand side is `0`. Because `block_height` values are `u64`, the left-hand side is always `>= 0`, so the `require!` never panics. The function returns `true` for any transaction whose block hash exists in the mainchain, even one submitted in the same NEAR transaction.

The same flaw is inherited by `verify_transaction_inclusion_v2`, which calls `verify_transaction_inclusion` via `self.verify_transaction_inclusion(args.into())` after only validating the coinbase proof length and root match. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation

Any NEAR smart contract that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` and gates an action (e.g., releasing bridged funds, minting tokens, settling a bet) on a `true` result can be bypassed. An attacker who controls a Bitcoin transaction that has been relayed into the light client (even with 1 confirmation) can immediately call the verification function with `confirmations = 0` and receive `true`, then trigger the downstream action before the Bitcoin transaction has reached the protocol's intended safety depth. This enables double-spend attacks against any protocol built on top of this light client. [5](#0-4) 

### Likelihood Explanation

The entry path requires no privilege: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, pausable-only functions callable by any NEAR account. The attacker only needs a transaction already present in the light client's mainchain (which the relayer continuously populates). Supplying `confirmations = 0` in the `ProofArgs` struct is trivial. Any downstream protocol that does not independently enforce a minimum confirmation count is immediately exploitable. [6](#0-5) [7](#0-6) 

### Recommendation

Add a minimum confirmation guard at the start of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "Confirmations must be at least 1");
```

Alternatively, enforce a protocol-level minimum (e.g., 6 for Bitcoin mainnet) and expose it as a contract constant so downstream consumers can rely on it. The `ProofArgsV2` path inherits the fix automatically since it delegates to `verify_transaction_inclusion`. [8](#0-7) 

### Proof of Concept

1. Relayer submits block `B` at height `H` to the light client (normal operation).
2. Attacker immediately calls:
   ```json
   verify_transaction_inclusion({
     "tx_id": "<tx hash in block B>",
     "tx_block_blockhash": "<hash of B>",
     "tx_index": 0,
     "merkle_proof": [...],
     "confirmations": 0
   })
   ```
3. The upper-bound check passes: `0 <= gc_threshold`.
4. The depth check evaluates: `(H - H) + 1 = 1 >= 0` → `true`.
5. The function returns `true` with zero confirmations.
6. A downstream contract that gates fund release on this result releases funds immediately, before the Bitcoin transaction has any meaningful confirmation depth, enabling a double-spend on the Bitcoin side. [9](#0-8)

```rust
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```
