### Title
Incomplete Mutual Exclusivity Guard on `_isReduceOnly` + `_isIsolated` Allows Reduce-Only Semantics Bypass and Unchecked Margin Drain — (`core/contracts/OffchainExchange.sol`)

---

### Summary

The `createIsolatedSubaccount` function in `OffchainExchange.sol` contains a partial guard against the mutually exclusive `_isReduceOnly` and `_isIsolated` order flags. The guard only fires when creating a **brand-new** isolated subaccount. When an existing isolated subaccount is found for the same product, the guard is silently skipped, allowing a reduce-only order to register a digest-to-subaccount mapping and drain margin from the parent subaccount without a health check.

---

### Finding Description

The `appendix` field of an `Order` struct encodes multiple operational flags. Two of them are semantically incompatible:

- `_isReduceOnly` (bit 11): constrains the order to only reduce an existing position; no new position setup, no margin movement.
- `_isIsolated` (bit 8): routes the order to an isolated subaccount and may transfer margin from the parent.

The code explicitly acknowledges this incompatibility with a guard: [1](#0-0) 

```solidity
if (newIsolatedSubaccount == bytes32(0)) {
    require(
        !_isReduceOnly(txn.order.appendix),
        "Reduce-only order cannot create isolated subaccount"
    );
```

However, this guard is wrapped inside `if (newIsolatedSubaccount == bytes32(0))`. The outer loop (lines 1025–1038) searches for an existing isolated subaccount for the same product: [2](#0-1) 

When one is found, `newIsolatedSubaccount` is set to the existing subaccount and the `if` block — including the `!_isReduceOnly` check — is **entirely skipped**. Execution falls through unconditionally to: [3](#0-2) 

```solidity
digestToSubaccount[digest] = newIsolatedSubaccount;

int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
}
```

Two consequences follow:

1. **Digest registration**: `digestToSubaccount[digest]` is set, so subsequent `matchOrders` calls remap `order.sender` to the isolated subaccount. The reduce-only constraint in `_validateOrder` then reads the **isolated subaccount's** balance instead of the parent's: [4](#0-3) 

This breaks the invariant that a reduce-only order reduces the **sender's** (parent's) position.

2. **Unchecked margin transfer**: `spotEngine.updateBalance` is called directly with no health check on the parent subaccount. There is no call to `clearinghouse` or any margin-adequacy check after the deduction.

---

### Impact Explanation

A trader who holds a leveraged position on their parent subaccount can:

- Submit a `CreateIsolatedSubaccount` transaction with `_isReduceOnly = true`, `_isIsolated = true`, and `_isolatedMargin = X` after an isolated subaccount for the product already exists.
- The reduce-only guard is bypassed; `X` quote tokens are transferred from the parent subaccount to the isolated subaccount with no health check.
- The parent subaccount's collateral drops by `X`. If the parent was near its initial margin requirement, it becomes immediately undercollateralized and liquidatable.
- The reduce-only constraint is subsequently evaluated against the isolated subaccount's balance (wrong account), so the parent's position is never reduced as the user intended.

Corrupted state: `spotEngine` balance of the parent subaccount (reduced by `_isolatedMargin`) and `digestToSubaccount` mapping (incorrectly populated for a reduce-only order).

---

### Likelihood Explanation

The entry path is fully user-controlled. A trader signs an `Order` struct with both `_isReduceOnly` (bit 11) and `_isIsolated` (bit 8) set in `appendix`, plus a non-zero `_isolatedMargin` (bits 64+). They submit this as a `CreateIsolatedSubaccount` transaction to the sequencer after having already established an isolated subaccount for the same product. No privileged access is required. The sequencer processes the transaction as normal. [5](#0-4) 

---

### Recommendation

Move the `!_isReduceOnly` check **outside** the `if (newIsolatedSubaccount == bytes32(0))` block so it applies unconditionally whenever a `CreateIsolatedSubaccount` transaction is processed:

```solidity
require(
    !_isReduceOnly(txn.order.appendix),
    "Reduce-only order cannot create isolated subaccount"
);

if (newIsolatedSubaccount == bytes32(0)) {
    // ... create new isolated subaccount
}
```

Alternatively, enforce mutual exclusivity of `_isReduceOnly` and `_isIsolated` at the `_validateOrder` level so no order carrying both flags can ever pass validation.

---

### Proof of Concept

1. User calls `CreateIsolatedSubaccount` (normal, non-reduce-only) for product P → isolated subaccount `ISO` is created.
2. User signs a second `Order` for product P with `_isReduceOnly = true`, `_isIsolated = true`, `_isolatedMargin = 1000e6`.
3. User submits this as a second `CreateIsolatedSubaccount` transaction to the sequencer.
4. Inside `createIsolatedSubaccount`:
   - Loop at lines 1025–1038 finds `ISO` for product P → `newIsolatedSubaccount = ISO`.
   - `if (newIsolatedSubaccount == bytes32(0))` is **false** → reduce-only check is skipped.
   - `digestToSubaccount[digest] = ISO` is written.
   - `spotEngine.updateBalance(QUOTE_PRODUCT_ID, parent, -1000e6)` executes with no health check.
5. Parent subaccount loses 1000 USDC of collateral. If parent was at 1.01× initial margin, it is now below 1× and immediately liquidatable.
6. Any subsequent `matchOrders` call using this digest routes to `ISO`; the reduce-only check reads `ISO`'s balance, not the parent's — the parent's position is never reduced.
