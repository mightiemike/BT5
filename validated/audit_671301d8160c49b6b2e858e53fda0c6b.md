### Title
Complete Absence of Emergency Pausability on All Critical Fund-Transfer Entry Points - (File: `core/contracts/Endpoint.sol`, `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

The Nado protocol contains zero pausability infrastructure across its entire smart contract codebase. Unlike the Blex analog where at least four functions retained a pause modifier, Nado's `Endpoint.sol` and `BaseWithdrawPool.sol` expose multiple unprivileged fund-transfer entry points with no on-chain mechanism to halt them. In the event of an active exploit, protocol operators have no ability to stop deposits, withdrawals, or fast-withdrawal payouts while a fix is developed and deployed.

---

### Finding Description

A codebase-wide search for `pause`, `whenNotPaused`, and `Pausable` returns zero matches across all Solidity files in the repository. No contract in the Nado protocol inherits from OpenZeppelin's `PausableUpgradeable` or implements any equivalent emergency-stop primitive.

The following unprivileged, externally reachable functions perform direct ERC-20 token transfers or queue/execute collateral movements with no guard that could be toggled by the protocol team:

**`Endpoint.sol`**

- `depositCollateral()` — callable by any user; pulls ERC-20 tokens from `msg.sender` and forwards them to `Clearinghouse` via `handleDepositTransfer`. [1](#0-0) 

- `depositCollateralWithReferral()` — `public`, callable by any address; same token-transfer path. [2](#0-1) 

- `submitSlowModeTransaction()` — callable by any user; queues `WithdrawCollateral` and other fund-affecting transactions. [3](#0-2) 

- `executeSlowModeTransaction()` — callable by any user after the delay; executes queued withdrawals, triggering `Clearinghouse.withdrawCollateral` → `handleWithdrawTransfer`. [4](#0-3) 

**`BaseWithdrawPool.sol`**

- `submitFastWithdrawal()` — callable by any address; verifies sequencer signatures and immediately transfers tokens to the recipient via `handleWithdrawTransfer`. [5](#0-4) 

The token transfer path for deposits flows through `EndpointStorage.handleDepositTransfer`, which calls `safeTransferFrom` and `safeTransferTo` with no interruptible state: [6](#0-5) 

The withdrawal path in `Clearinghouse.withdrawCollateral` calls `handleWithdrawTransfer` → `BaseWithdrawPool.submitWithdrawal` → `handleWithdrawTransfer` → `token.safeTransfer`, also with no interruptible state: [7](#0-6) [8](#0-7) 

Neither `Endpoint`, `EndpointTx`, `Clearinghouse`, `ClearinghouseLiq`, `SpotEngine`, `PerpEngine`, nor `BaseWithdrawPool` inherits from any pausable base: [9](#0-8) [10](#0-9) [11](#0-10) 

---

### Impact Explanation

If a vulnerability is discovered in any component that affects fund accounting — for example, a flaw in the health-check logic in `Clearinghouse.getHealth`, a signature-verification bypass in `Verifier`, or an accounting error in `SpotEngine`/`PerpEngine` — the protocol team has no on-chain lever to halt deposits, withdrawals, or fast-withdrawal payouts while a patch is prepared and deployed. An attacker can continue to drain funds through any of the five entry points listed above for the entire duration of the incident response window. All ERC-20 collateral held in `Clearinghouse` and all liquidity held in `WithdrawPool` is at risk. [12](#0-11) 

---

### Likelihood Explanation

The Nado protocol is a multi-component system with a large attack surface: `Endpoint`, `EndpointTx` (delegatecall target), `Clearinghouse`, `ClearinghouseLiq` (delegatecall target), `OffchainExchange`, `SpotEngine`, `PerpEngine`, and `Verifier` all interact. The use of `delegatecall` for `EndpointTx` and `ClearinghouseLiq` introduces additional upgrade-path risk. Any post-deployment vulnerability discovery in any of these components that touches fund flows would leave the protocol with no on-chain mitigation path. The likelihood of needing an emergency pause at some point in the protocol's lifetime is non-trivial for a system of this complexity. [13](#0-12) [14](#0-13) 

---

### Recommendation

1. Inherit from OpenZeppelin's `PausableUpgradeable` in `Endpoint.sol` and `BaseWithdrawPool.sol`.
2. Apply `whenNotPaused` to at minimum: `depositCollateral`, `depositCollateralWithReferral`, `submitSlowModeTransaction`, `executeSlowModeTransaction`, and `submitFastWithdrawal`.
3. Expose a `pause()` function restricted to the protocol owner or a designated guardian multisig.
4. Consider also gating `processTransactionImpl` in `EndpointTx.sol` for the `WithdrawCollateral` and `WithdrawCollateralV2` transaction types.

---

### Proof of Concept

1. Deploy the protocol normally.
2. Assume a vulnerability is discovered in `Clearinghouse.getHealth` that allows a subaccount to pass health checks with insufficient collateral.
3. The protocol team has no on-chain call they can make to stop `depositCollateral`, `executeSlowModeTransaction`, or `submitFastWithdrawal`.
4. An attacker calls `Endpoint.depositCollateral()` with minimal collateral, exploits the health-check flaw to withdraw more than deposited via `submitSlowModeTransaction` + `executeSlowModeTransaction` (after the 3-day delay) or immediately via `BaseWithdrawPool.submitFastWithdrawal` with valid sequencer signatures.
5. The attack continues unimpeded until an upgrade is deployed, which itself requires governance time.

The absence of any `pause`/`whenNotPaused` symbol anywhere in the codebase is confirmed by a full-repository grep returning zero matches. [1](#0-0) [5](#0-4)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
