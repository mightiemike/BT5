Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address as Swapper, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` always sets to `msg.sender` of the pool call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` of `pool.swap`, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][realUser]`. A pool admin who allowlists the router to permit router-based swaps inadvertently grants every user on the network access to the curated pool.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes its own `msg.sender` as `sender`:** [1](#0-0) 

`msg.sender` here is whoever called `pool.swap()`. When the router is the caller, the router's address is forwarded as `sender` to every extension.

**`SwapAllowlistExtension.beforeSwap` checks that forwarded `sender`:** [2](#0-1) 

`msg.sender` inside the extension is the pool; `sender` is the argument — i.e., the router's address, not the end-user.

**`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly:** [3](#0-2) 

The router is `msg.sender` of `pool.swap()`. The pool forwards the router's address as `sender` to `_beforeSwap`. The extension therefore evaluates `allowedSwapper[pool][router]`.

**The dilemma for pool admins:**

| Admin action | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all |
| **Allowlist the router** | Every user on the network bypasses the allowlist |

No configuration simultaneously allows allowlisted users to use the router and blocks non-allowlisted users. The same issue applies to `exactInput` and `exactOutputSingle`. [4](#0-3) [5](#0-4) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` for access control (KYC compliance, institutional-only access, whitelist-gated liquidity) provides zero protection against any user routing through `MetricOmmSimpleRouter`. Any non-allowlisted user can call `exactInputSingle`, `exactInput`, or `exactOutputSingle` on the router and execute swaps on the curated pool. Non-permitted actors can drain LP value from a pool designed to restrict access — a direct, fund-impacting policy bypass meeting the "broken core pool functionality causing loss of funds" threshold.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint in the periphery. Any user who discovers the allowlist on a curated pool can trivially route through the router instead of calling the pool directly. No special privileges, flash loans, or multi-step setup are required — a single `exactInputSingle` call suffices. The attack is repeatable indefinitely.

## Recommendation
The extension must gate on the economically relevant actor — the end-user — not the immediate caller of `pool.swap`. Two viable approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap`; the extension decodes and checks it. Requires a trusted encoding convention between router and extension.
2. **Preferred — dedicated `originalSender` field**: The pool exposes a separate parameter (or transient storage slot) that the router populates with `msg.sender` before calling `pool.swap`, and the extension reads that field. This is analogous to Uniswap v4's `hookData` pattern with a trusted router.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack:
  - bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({
        pool: curated_pool,
        recipient: bob,
        ...
    })
  - Router calls pool.swap(...) with msg.sender = router
  - Pool passes router address as `sender` to _beforeSwap
  - Extension checks allowedSwapper[pool][router] → true
  - Bob's swap executes on the curated pool despite not being allowlisted
```

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist only `alice` and the router, then call `exactInputSingle` from `bob` and assert the swap succeeds — demonstrating the bypass.
