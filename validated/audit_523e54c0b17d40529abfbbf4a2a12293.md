### Title
Equihash Solution Validity Return Value Silently Discarded, Allowing Invalid PoW Block Acceptance — (`contract/src/zcash.rs`)

### Summary

In the Zcash PoW verification path, `equihash::is_valid_solution` returns `Result<bool, Error>`. The contract code unwraps the `Result` to handle the `Err` variant but **discards the inner `bool`**. An invalid Equihash solution that is syntactically well-formed returns `Ok(false)`, not `Err`. Because `false` is silently dropped, the block passes PoW verification and is committed to the mainchain.

### Finding Description

In `contract/src/zcash.rs`, `check_pow` performs Equihash validation as follows: [1](#0-0) 

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
```

The `equihash::is_valid_solution` API returns:
- `Ok(true)` — solution is valid
- `Ok(false)` — solution is **invalid** (wrong PoW, but well-formed bytes)
- `Err(e)` — malformed/unparseable solution

The `.unwrap_or_else(|e| { env::panic_str(...) })` call handles only the `Err` branch (panics). For the `Ok(false)` branch, `unwrap_or_else` returns `false` — which is immediately dropped because the expression result is unused. No `require!`, no assertion, no binding. The function returns normally, and `check_pow` returns without rejecting the block.

The call chain is: [2](#0-1) 

`submit_blocks` → `submit_block_header` → `check_target` → `check_pow` → discarded `Ok(false)` [3](#0-2) 

After `check_pow` returns without panicking, `submit_block_header_inner` is called and the invalid block is stored in `headers_pool` and potentially promoted to `mainchain_tip_blockhash`. [4](#0-3) 

### Impact Explanation

A block with a fabricated or zeroed Equihash solution — requiring zero real PoW — is accepted as a valid mainchain block. The attacker can:

1. Advance `mainchain_tip_blockhash` to an attacker-controlled block with no real work.
2. Trigger a chain reorg via `reorg_chain` by submitting a sequence of zero-work blocks whose cumulative `chain_work` (computed from `bits`, not from the Equihash solution) exceeds the honest chain.
3. Cause `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to return `true` for transactions that were never confirmed on the real Bitcoin/Zcash chain.

The broken invariant is: **only blocks whose Equihash solution satisfies the PoW target are accepted**. The corrupted canonical mapping is `mainchain_tip_blockhash` and the `mainchain_height_to_header` / `mainchain_header_to_height` maps. [5](#0-4) 

### Likelihood Explanation

Any account that qualifies as a trusted relayer (via the staking mechanism governed by `RelayerManager`) can call `submit_blocks`. The `trusted_relayer` macro is a staking-based permissioning layer, not a cryptographic guarantee. A relayer who has staked can immediately exploit this by submitting a single Zcash header with an all-zero `solution` field. No leaked keys or social engineering are required beyond obtaining relayer status. [6](#0-5) 

### Recommendation

Assert the returned `bool` with `require!`:

```rust
let is_valid = equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
require!(is_valid, "Invalid Equihash solution: solution does not meet PoW target");
```

This mirrors the pattern used for the hash-vs-target check in the Bitcoin/Litecoin/Dogecoin paths: [7](#0-6) 

### Proof of Concept

1. Construct a syntactically valid Zcash block header (`ZcashHeader`) with `solution` set to 1344 zero bytes (a well-formed but cryptographically invalid Equihash solution).
2. Set `bits` to any value (e.g., `proof_of_work_limit_bits`) so `check_pow`'s `expected_bits == block_header.bits` check passes.
3. Call `submit_blocks` as a trusted relayer with `skip_pow_verification = false`.
4. `equihash::is_valid_solution` returns `Ok(false)`; the `false` is discarded; `check_pow` returns without panicking.
5. The block is stored via `store_block_header` or `store_fork_header` and, if its `chain_work` exceeds the current tip, becomes the new `mainchain_tip_blockhash`.
6. Subsequent calls to `verify_transaction_inclusion` against this block return `true` for attacker-crafted Merkle proofs. [8](#0-7)

    fn submit_block_header_inner(
        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }
