### Title
Linked Signer Can Unilaterally Mutate Shared `linkedSigners` State, Atomically Breaking All Dependent Isolated Subaccounts — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` fast-path transaction handler in `EndpointTx.sol` accepts a signature from the **current linked signer** (`allowLinkedSigner = true`). Because `linkedSigners[parentSubaccount]` is shared state consumed by both the parent subaccount and every isolated subaccount that inherits from it, a linked signer can unilaterally overwrite this slot — simultaneously removing or hijacking signing authority for the parent **and** all of its isolated subaccounts. The subaccount owner never authorizes this mutation.

---

### Finding Description

**Shared state and two roles**

`EndpointStorage.sol` holds a single mapping:

```solidity
mapping(bytes32 => address) internal linkedSigners;   // line 50
```

This slot serves two distinct roles:

1. **Parent subaccount role** — `getLinkedSigner` returns `linkedSigners[subaccount]` directly for non-isolated subaccounts.
2. **Isolated subaccount role** — `getLinkedSigner` returns `linkedSigners[parentSubaccount]` for every isolated subaccount, meaning all isolated children share the parent's slot. [1](#0-0) [2](#0-1) 

**The `LinkSigner` handler allows the linked signer to self-modify**

The fast-path `LinkSigner` handler calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(...);
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← linked signer is accepted as valid signer
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [3](#0-2) 

`validateSignature` then passes `getLinkedSigner(sender)` to the verifier, so the verifier accepts a signature from either the subaccount owner **or** the current linked signer: [4](#0-3) 

This means the linked signer can craft and submit a valid `LinkSigner` transaction that writes any value — including `address(0)` or an attacker-controlled address — into `linkedSigners[parentSubaccount]`.

**Isolated subaccounts inherit from the same slot**

`createIsolatedSubaccount` records `parentSubaccounts[newIsolatedSubaccount] = txn.order.sender` and all subsequent signature checks for the isolated subaccount read `linkedSigners[parentSubaccount]` dynamically: [5](#0-4) [6](#0-5) 

There is no per-isolated-subaccount signer slot; all isolated children share the parent's `linkedSigners` entry. A single write to `linkedSigners[parent]` therefore atomically affects every isolated subaccount.

---

### Impact Explanation

A linked signer (e.g., a hot wallet or automated trading bot) can:

1. **Zero out the slot** — set `linkedSigners[parent] = address(0)`. Every isolated subaccount immediately loses its linked signer. Any in-flight or future order signed by the linked signer on behalf of an isolated subaccount will fail signature validation. Open isolated positions can no longer be managed by the linked signer, potentially leaving them unable to be closed or adjusted in time.

2. **Replace the slot with an attacker address** — set `linkedSigners[parent] = attackerAddress`. The attacker now holds signing authority over the parent subaccount **and** every isolated subaccount simultaneously. The attacker can submit `WithdrawCollateral`, `TransferQuote`, or any other `allowLinkedSigner = true` transaction on behalf of all of them.

The corrupted state is `linkedSigners[parentSubaccount]` — a single slot that controls signing for an unbounded number of isolated subaccounts.

---

### Likelihood Explanation

The linked signer is an externally-controlled address (hot wallet, keeper, or third-party service). Any of the following realistic conditions trigger the issue:

- The linked signer key is compromised.
- A third-party keeper service turns adversarial.
- A user accidentally submits a `LinkSigner` transaction signed by the linked signer (e.g., via a front-end bug or SDK misuse) intending to update only the parent, unaware it also breaks all isolated subaccounts.

The entry path requires no special privilege beyond possessing the linked signer's private key, which is the normal operational assumption for this role. Likelihood is **medium**.

---

### Recommendation

1. **Restrict `LinkSigner` to the subaccount owner only** — change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type. The linked signer should not be able to modify its own authorization slot.

2. **Scope the mutation** — if the linked signer must be allowed to rotate itself, introduce a separate per-isolated-subaccount signer slot so that modifying the parent's slot does not atomically affect all children.

3. **Emit an event** on every `linkedSigners` write so that off-chain monitoring can detect unauthorized mutations.

---

### Proof of Concept

```
1. Alice owns parent subaccount P and has set linkedSigners[P] = BOT.
2. Alice creates isolated subaccounts I1, I2, I3 under P.
   - getLinkedSigner(I1/I2/I3) all return linkedSigners[P] = BOT.
3. BOT (the linked signer) constructs a LinkSigner transaction:
     sender = P, signer = ATTACKER, nonce = current_nonce
   and signs it with BOT's private key.
4. BOT submits the transaction through the sequencer.
5. EndpointTx.sol:581-590 validates the signature:
     validateSignedTx(..., allowLinkedSigner = true)
   → verifier accepts BOT's signature because BOT == getLinkedSigner(P).
6. linkedSigners[P] is overwritten with ATTACKER.
7. getLinkedSigner(I1), getLinkedSigner(I2), getLinkedSigner(I3)
   all now return ATTACKER.
8. ATTACKER can sign WithdrawCollateral / TransferQuote for P, I1, I2, I3.
   Alice's funds across all isolated subaccounts are at risk.
``` [7](#0-6) [1](#0-0) [8](#0-7)
