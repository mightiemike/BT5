### Title
`ContractOwner` Missing `receive()` Function Permanently Breaks Native ETH Recovery from `DirectDepositV1` — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner` is the sole owner of every `DirectDepositV1` contract it deploys. The only admin path to recover native ETH stuck in a `DirectDepositV1` is `ContractOwner.withdrawFromDirectDepositV1(..., address(0))`, which internally calls `DirectDepositV1.withdrawNative()`. That function pushes ETH to `msg.sender` (i.e., `ContractOwner`) via a low-level `call`. Because `ContractOwner` declares no `receive()` or payable `fallback()` function, the ETH transfer always reverts, making any native ETH that becomes stuck in a `DirectDepositV1` permanently unrecoverable.

---

### Finding Description

`DirectDepositV1` is deployed by `ContractOwner.createDirectDepositV1()` and inherits OpenZeppelin `Ownable`, so `ContractOwner` is its owner. [1](#0-0) 

The constructor of `DirectDepositV1` silently handles a pre-existing ETH balance: if the `wrappedNative.call` fails, it emits `NativeTokenTransferFailed` and continues — leaving ETH stranded in the contract with no revert. [2](#0-1) 

The designated recovery path is `DirectDepositV1.withdrawNative()`, which pushes the full ETH balance to `msg.sender`: [3](#0-2) 

`ContractOwner.withdrawFromDirectDepositV1` is the only caller that can invoke this (owner-gated) function. It calls `withdrawNative()` and expects `ContractOwner` itself to receive the ETH: [4](#0-3) 

However, `ContractOwner` declares **no `receive()` function and no payable `fallback()`**. A grep across the entire file returns zero matches for either. Because `ContractOwner` cannot accept a plain ETH transfer, the `call{value: balance}("")` inside `withdrawNative()` will always return `success = false`, causing `require(success, "Failed to transfer native token to owner")` to revert. [5](#0-4) 

---

### Impact Explanation

Any native ETH that ends up in a `DirectDepositV1` contract — specifically via the silent constructor failure path — is permanently locked. The only withdrawal function (`withdrawNative`) is owner-gated to `ContractOwner`, and `ContractOwner` structurally cannot receive ETH. There is no alternative recovery path. The ETH balance is irrecoverable.

---

### Likelihood Explanation

The trigger condition is: a `DirectDepositV1` is deployed at an address that already holds an ETH balance (e.g., pre-funded via `CREATE2` with a known salt), and the `wrappedNative.call` in the constructor fails (e.g., wrapped-native contract not yet live, or temporarily broken). The `CREATE2` salt is hardcoded to `bytes32(uint256(1))`, making the deployment address deterministic and predictable. This is a realistic scenario on chains where wrapped-native contracts may not be deployed at the expected address at the time of `DirectDepositV1` deployment. [6](#0-5) 

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

Alternatively, refactor `withdrawFromDirectDepositV1` to pull ETH directly to `msg.sender` (the multisig) rather than routing it through `ContractOwner`:

```solidity
// In DirectDepositV1.withdrawNative, accept a `to` parameter:
function withdrawNative(address payable to) external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = to.call{value: balance}("");
    require(success, "Failed to transfer native token");
}
```

---

### Proof of Concept

1. `ContractOwner.createDirectDepositV1(subaccount)` is called. The `CREATE2` salt is `bytes32(uint256(1))`, so the `DirectDepositV1` address is deterministic.
2. Before or at deployment, the `DirectDepositV1` address holds an ETH balance (e.g., sent by anyone knowing the deterministic address).
3. In the constructor, `wrappedNative.call{value: balance}("")` fails (returns `false`). The constructor emits `NativeTokenTransferFailed` and completes — ETH remains in `DirectDepositV1`.
4. Admin calls `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`.
5. This calls `DirectDepositV1(directDepositV1).withdrawNative()` where `msg.sender == address(ContractOwner)`.
6. Inside `withdrawNative`: `(bool success, ) = msg.sender.call{value: balance}("")` — `ContractOwner` has no `receive()`, so `success = false`.
7. `require(success, "Failed to transfer native token to owner")` reverts.
8. ETH is permanently stuck in `DirectDepositV1` with no recovery path. [2](#0-1) [3](#0-2) [4](#0-3)

        external
        onlyOwner
    {
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        if (token == address(0)) {
            uint256 preBalance = address(this).balance;
            DirectDepositV1(directDepositV1).withdrawNative();
            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");
            (bool success, ) = msg.sender.call{value: postBalance - preBalance}(
                ""
            );
            require(success, "xfer");
        } else {
            uint256 preBalance = IERC20Base(token).balanceOf(address(this));
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));
            require(postBalance > preBalance, "empty");
            IERC20Base(token).safeTransfer(
                msg.sender,
                postBalance - preBalance
            );
