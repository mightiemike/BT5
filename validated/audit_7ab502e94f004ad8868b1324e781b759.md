### Title
Silent Liquidation Bypass via Missing Code Existence Check on `clearinghouseLiq` Delegatecall — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.liquidateSubaccount` performs a `delegatecall` to `clearinghouseLiq` without verifying that the target address contains deployed bytecode. Per EVM design, a `delegatecall` to a codeless address returns `success = true` with empty return data. The subsequent `require(success, string(result))` guard passes, making the liquidation appear successful while no liquidation logic was actually executed.

---

### Finding Description

In `Clearinghouse.sol`, the `liquidateSubaccount` function delegates all liquidation logic to the mutable `clearinghouseLiq` address:

```solidity
(bool success, bytes memory result) = clearinghouseLiq.delegatecall(
    liquidateSubaccountCall
);
require(success, string(result));
```

There is no check that `clearinghouseLiq` contains code before the `delegatecall` is issued. If `clearinghouseLiq` holds an address with no deployed bytecode — whether due to a deployment error during an upgrade, a pre-deployment address being set, or a self-destructed contract — the EVM returns `(true, "")`. The `require(success, string(result))` guard evaluates to `require(true, "")` and passes unconditionally. No liquidation logic runs, no state is mutated, and the call returns as if the liquidation succeeded.

This is structurally identical to the external report's `functionDelegateCallUnverified` pattern: the only post-call guard is a success flag check, which is insufficient because the EVM always returns `success = true` for calls to codeless addresses.

The `clearinghouseLiq` address is a mutable storage variable initialized in `Clearinghouse.initialize` and updatable via `upgradeClearinghouseLiq`, which is gated to the ProxyManagerHelper. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

If `clearinghouseLiq` has no code at call time, every invocation of `liquidateSubaccount` silently returns success without executing any liquidation logic. Insolvent subaccounts are never actually liquidated. Bad debt accumulates in the clearinghouse without any on-chain signal of failure. This directly corrupts the core solvency invariant of the protocol: the assumption that a successful `liquidateSubaccount` call has reduced or eliminated an insolvent position.

The corrupted state delta is: `clearinghouseLiq` storage balances and insurance fund are never updated, while the sequencer's submission index advances and the liquidation is recorded as processed. [4](#0-3) 

---

### Likelihood Explanation

The `clearinghouseLiq` address is set at initialization and can be updated by the ProxyManagerHelper via `upgradeClearinghouseLiq`. A deployment error during an upgrade — setting `clearinghouseLiq` to a pre-deployment address or an incorrect address — would silently break all liquidations with no revert or on-chain error. There is no on-chain safety net (no `extcodesize` check, no interface probe) to catch this class of misconfiguration. The missing check means the protocol has no defense-in-depth against this failure mode. [3](#0-2) 

---

### Recommendation

**Short term**: Add an explicit code existence check before the `delegatecall` in `liquidateSubaccount`:

```solidity
require(clearinghouseLiq.code.length > 0, "clearinghouseLiq has no code");
(bool success, bytes memory result) = clearinghouseLiq.delegatecall(
    liquidateSubaccountCall
);
require(success, string(result));
```

**Long term**: Apply the same guard to every `delegatecall` target in the protocol (including `endpointTx` in `Endpoint._delegatecallEndpointTx`, which only checks `!= address(0)` but not code existence). Document the invariant that all delegatecall targets must have deployed code, and enforce it at the upgrade boundary in `upgradeClearinghouseLiq`. [5](#0-4) 

---

### Proof of Concept

1. During a protocol upgrade, `upgradeClearinghouseLiq` is called with an address that has no deployed bytecode (e.g., a pre-deployment address or a misconfigured address).
2. `clearinghouseLiq` is now set to a codeless address.
3. A subaccount becomes insolvent (maintenance health < 0).
4. The sequencer submits a `LiquidateSubaccount` transaction via `Endpoint`, which calls `Clearinghouse.liquidateSubaccount`.
5. `clearinghouseLiq.delegatecall(liquidateSubaccountCall)` is issued to a codeless address.
6. The EVM returns `(true, "")` — no code executed, no state changed.
7. `require(true, "")` passes.
8. The function returns normally. The sequencer advances. The insolvent position is never liquidated.
9. Bad debt accumulates; the insurance fund is never drawn; the protocol becomes insolvent. [1](#0-0) [6](#0-5)
