### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Transaction Forgery Protection Enforced by v2 - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` was deprecated in favour of `verify_transaction_inclusion_v2`, which adds a mandatory coinbase Merkle proof check to close the well-known 64-byte transaction forgery vector. However, the deprecated function is still a fully public NEAR entry point. Any unprivileged caller can invoke it directly, skipping the coinbase proof check entirely and obtaining a `true` return value for a forged `tx_id` that is actually an internal Merkle-tree node rather than a real transaction hash.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability documented at https://www.bitmex.com/blog/64-Byte-Transactions. It does so by first verifying a coinbase Merkle proof against the block's `merkle_root`, then delegating to the deprecated v1 function:

```rust
// contract/src/lib.rs  lines 347-369
pub fn verify_transaction_inclusion_v2(&self, args: ProofArgsV2) -> bool {
    require!(
        args.merkle_proof.len() == args.coinbase_merkle_proof.len(), ...
    );
    // coinbase proof must reach the same merkle_root
    require!(
        merkle_tools::compute_root_from_merkle_proof(
            args.coinbase_tx_id.clone(), 0usize, &args.coinbase_merkle_proof,
        ) == header.block_header.merkle_root,
        "Incorrect coinbase merkle proof"
    );
    #[allow(deprecated)]
    self.verify_transaction_inclusion(args.into())   // ← delegates to v1
}
```

The deprecated v1 function, however, is still declared `pub` and carries only a Rust-level `#[deprecated]` attribute, which is a compiler hint and **not an access restriction on NEAR**:

```rust
// contract/src/lib.rs  lines 283-323
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, args: ProofArgs) -> bool {
    // ... confirmations check, mainchain membership check ...
    require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root   // ← no coinbase proof check
}
```

The function's own documentation acknowledges the gap:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` (lines 34-52) is a pure path-traversal function: it accepts any 32-byte value as the starting hash and walks up the tree. It has no awareness of whether the starting value is a leaf (real transaction) or an internal node.

---

### Impact Explanation

An attacker who calls `verify_transaction_inclusion` directly can supply:
- `tx_id` = the hash of an internal Merkle-tree node (a 64-byte preimage of a parent node)
- A valid `merkle_proof` path from that internal node to the block's `merkle_root`

The function will return `true`. Any on-chain or off-chain consumer that calls `verify_transaction_inclusion` and acts on a `true` result (e.g., releasing funds, minting tokens, updating state) can be deceived into accepting a non-existent transaction as confirmed. The corrupted proof result is the boolean `true` returned for a `tx_id` that does not correspond to any real transaction in the block.

---

### Likelihood Explanation

The entry point is unconditionally reachable by any NEAR account with no staking, role, or deposit requirement beyond the standard NEAR gas fee. The 64-byte transaction forgery technique is publicly documented and has known tooling. The only prerequisite is knowledge of an internal Merkle-tree node hash for a block already stored in the contract's `headers_pool`, which is derivable from public Bitcoin chain data. Likelihood is **high** for any deployment whose downstream consumers have not independently re-implemented the coinbase proof check.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or gate it behind an access-control role so it is no longer callable as a NEAR transaction entry point. Alternatively, add the same coinbase proof check to `verify_transaction_inclusion` itself so that both paths enforce the same invariant. The `#[deprecated]` Rust attribute alone provides zero on-chain access restriction.

---

### Proof of Concept

1. Identify any block `B` stored in the contract's `mainchain_height_to_header` with at least two transactions.
2. Compute the internal Merkle-tree node `N = SHA256d(tx0 || tx1)` from public chain data.
3. Construct a `merkle_proof` path from `N` to `B.merkle_root` (one level shorter than a leaf proof).
4. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B.hash`, `tx_index = 0`, `merkle_proof = <path from N to root>`, `confirmations = 1`.
5. The function returns `true` because `compute_root_from_merkle_proof(N, 0, proof) == B.merkle_root`, even though `N` is not a transaction hash.
6. A consumer contract that trusts this result will treat the forged `tx_id` as a confirmed Bitcoin transaction.

**Root cause**: `verify_transaction_inclusion` at `contract/src/lib.rs` lines 283-323 is a public NEAR entry point with no coinbase proof guard, while the system's security invariant (established by `verify_transaction_inclusion_v2` at lines 347-369) requires that invariant to hold for all callers. pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
