### Title
`max_nonce_for_validation_skip` Config Field Ignored in Gateway — `__validate__` Bypass Is Hardcoded and Unconfigurable - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

`StatefulTransactionValidatorConfig` declares a `max_nonce_for_validation_skip` field that is serialized into the node config schema and is intended to bound the nonce range for which the gateway skips the account's `__validate__` entry point. However, the actual gateway function `skip_stateful_validations` never reads this field — it hardcodes the nonce-1 check. The config parameter is therefore dead code in the gateway path, meaning the skip cannot be disabled or widened by operators, and any transaction with nonce=1 from an undeployed account that has a `deploy_account` in the mempool will always bypass `__validate__` at admission time regardless of configuration.

### Finding Description

`StatefulTransactionValidatorConfig` in `crates/apollo_gateway_config/src/config.rs` declares:

```rust
pub struct StatefulTransactionValidatorConfig {
    pub max_nonce_for_validation_skip: Nonce,
    ...
}
```

with default value `Nonce(Felt::ONE)` and a config-schema entry `"max_nonce_for_validation_skip"` with description *"Maximum nonce for which the validation is skipped."* [1](#0-0) 

The gateway's `skip_stateful_validations` function, called from `run_pre_validation_checks`, does **not** accept `self.config` as a parameter and does not read `max_nonce_for_validation_skip`. Instead it hardcodes the nonce-1 check:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [2](#0-1) 

The call site in `run_pre_validation_checks` passes no config to the function:

```rust
let skip_validate =
    skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
``` [3](#0-2) 

By contrast, the `PyValidator` (native-blockifier Python wrapper) correctly reads `self.max_nonce_for_validation_skip` and uses it to gate the skip:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
``` [4](#0-3) 

The two code paths therefore implement different policies: the Python wrapper respects the config; the Rust gateway ignores it.

### Impact Explanation

When `skip_validate = true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false`, which means the blockifier's `StatefulValidator` does **not** call the account's `__validate__` entry point:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [5](#0-4) 

`__validate__` is the account's signature-verification entry point. Skipping it at the gateway means the gateway admits an invoke transaction without verifying the ECDSA/Stark signature. The transaction is then forwarded to the mempool and eventually included in a block, where it will revert during execution (because the OS does call `__validate__`). This enables:

1. **Spam / block-space exhaustion**: An attacker submits a `deploy_account` for a fresh address (valid), then submits an invoke with nonce=1 and a garbage signature. The gateway accepts the invoke unconditionally. The transaction occupies block space and reverts, wasting sequencer resources.
2. **Config control plane is broken**: An operator who sets `max_nonce_for_validation_skip = 0x0` to harden the node (disable the skip entirely) gets no effect. The gateway continues to skip `__validate__` for nonce-1 transactions. This is the direct analog to the Yearn bug: a security check that is advertised as configurable is silently bypassed.

This matches the impact category: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The trigger is fully unprivileged:
1. Submit any valid `deploy_account` for a fresh address (no special key needed).
2. Submit an invoke with `nonce=1`, `sender_address` = that address, and an arbitrary/invalid signature.
3. `skip_stateful_validations` returns `true`; the gateway admits the transaction without signature verification.

The condition `mempool_client.account_tx_in_pool_or_recent_block` is satisfied by step 1. No privileged access, no special network position, and no race condition is required.

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` (or make it a method on `StatefulTransactionValidator`) and replace the hardcoded `Nonce(Felt::ONE)` upper-bound with the configured value:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    max_nonce_for_validation_skip: Nonce,   // ← add parameter
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        let tx_nonce = tx.nonce();
        if tx_nonce >= Nonce(Felt::ONE)
            && tx_nonce <= max_nonce_for_validation_skip   // ← use config
            && account_nonce == Nonce(Felt::ZERO)
        {
            ...
        }
    }
    Ok(false)
}
```

And update the call site in `run_pre_validation_checks` to pass `self.config.max_nonce_for_validation_skip`.

### Proof of Concept

1. Start a gateway node with default config (`max_nonce_for_validation_skip = 0x1`).
2. Submit a `deploy_account` V3 transaction for address `A` — this enters the mempool.
3. Construct an invoke V3 transaction: `sender_address = A`, `nonce = 1`, `signature = [0x0, 0x0]` (invalid).
4. POST to the gateway's `add_transaction` endpoint.
5. Observe: the gateway calls `skip_stateful_validations`, which returns `true` (nonce=1, account_nonce=0, deploy_account in mempool). `run_validate_entry_point` is called with `skip_validate=true`, so `__validate__` is never invoked. The gateway returns a success response and the transaction enters the mempool.
6. Now set `max_nonce_for_validation_skip = 0x0` in the config and restart. Repeat steps 2–5. The gateway **still** accepts the invalid-signature invoke, demonstrating that the config field has no effect. [6](#0-5) [7](#0-6)

```rust
/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
