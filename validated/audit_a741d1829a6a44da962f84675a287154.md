### Title
P2P Protobuf Receipt Converter Silently Drops AddMod, MulMod, RangeCheck96 Builtin Counts — (`crates/apollo_protobuf/src/converters/receipt.rs`)

### Summary

The `BuiltinCounter` message in the protobuf receipt schema does not define fields for `add_mod`, `mul_mod`, or `range_check96`. The Rust `From<HashMap<Builtin, u64>> for ProtobufBuiltinCounter` converter silently discards those three counts (treating them as 0), and the inverse `TryFrom` never reconstructs them. Any node that receives block data via P2P sync stores zeroed counts for these builtins and subsequently serves wrong `execution_resources` in RPC receipts.

### Finding Description

**Proto schema — no fields for the three builtins:** [1](#0-0) 

The `BuiltinCounter` message contains only `bitwise`, `ecdsa`, `ec_op`, `pedersen`, `range_check`, `poseidon`, `keccak`, `output`. A developer TODO comment acknowledges the gap: `// TODO(alonl): add the missing builtins`.

**Serialization — three builtins are not written:** [2](#0-1) 

`From<HashMap<Builtin, u64>> for ProtobufBuiltinCounter` maps `RangeCheck`, `Pedersen`, `Poseidon`, `EcOp`, `Ecdsa`, `Bitwise`, `Keccak`, and hard-codes `output: 0`. `Builtin::AddMod`, `Builtin::MulMod`, and `Builtin::RangeCheck96` are never read from the map; their counts are silently discarded.

**Deserialization — three builtins are not read back:** [3](#0-2) 

`TryFrom<ProtobufBuiltinCounter> for HashMap<Builtin, u64>` inserts only the same seven builtins plus `SegmentArena: 0`. `AddMod`, `MulMod`, and `RangeCheck96` are never inserted; they are absent from the reconstructed map (equivalent to 0).

**The three builtins are valid, production-used variants in `starknet_api`:** [4](#0-3) 

**P2P sync server serializes `TransactionOutput` (which contains `ExecutionResources`) directly through this converter:** [5](#0-4) 

The server reads `transaction_outputs` from storage and sends them as protobuf `Receipt` messages. The receiving node deserializes and stores the result — with zeroed `AddMod`/`MulMod`/`RangeCheck96` counts.

### Impact Explanation

A syncing node that receives block data via P2P will store `ExecutionResources` with `AddMod=0`, `MulMod=0`, `RangeCheck96=0` for any transaction that actually used those builtins. When that node serves `starknet_getTransactionReceipt` (or any receipt-bearing RPC method), it returns an authoritative-looking but wrong `execution_resources` object. This falls squarely under:

> **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

Contracts using circuit evaluation (`EvalCircuit` hint, which drives `add_mod` and `mul_mod`) are the primary affected class. Any tooling or protocol participant that relies on the reported builtin counts for gas/resource analysis will receive incorrect data from a P2P-synced node.

### Likelihood Explanation

- The P2P sync path is the standard block propagation mechanism between sequencer nodes and full nodes.
- Contracts using `add_mod`/`mul_mod` (circuit evaluation) are already deployed on Starknet mainnet.
- No guard exists: the converter compiles and runs without error; the loss is completely silent.
- The TODO comment in the `.proto` file confirms the omission is known but unresolved.

### Recommendation

1. Add `add_mod`, `mul_mod`, and `range_check96` fields to `BuiltinCounter` in `receipt.proto`.
2. Update `From<HashMap<Builtin, u64>> for ProtobufBuiltinCounter` to write those three fields.
3. Update `TryFrom<ProtobufBuiltinCounter> for HashMap<Builtin, u64>` to read them back.
4. Add a round-trip test that constructs `ExecutionResources` with non-zero counts for all three builtins, converts to protobuf and back, and asserts the counts are preserved.

### Proof of Concept

```rust
// Pseudocode round-trip test (confirms the assertion fails today)

counter.insert(Builtin::AddMod, 5u64);
counter.insert(Builtin::MulMod, 3u64);
```

```rust
#[async_trait]
impl FetchBlockData for FullTransaction {
    async fn fetch_block_data(
        block_number: BlockNumber,
        txn: &StorageTxn<'_, db::RO>,
        _class_manager_client: &mut SharedClassManagerClient,
    ) -> Result<Vec<Self>, P2pSyncServerError> {
        let transactions =
            txn.get_block_transactions(block_number)?.ok_or(P2pSyncServerError::BlockNotFound {
                block_hash_or_number: BlockHashOrNumber::Number(block_number),
            })?;
        let transaction_outputs = txn.get_block_transaction_outputs(block_number)?.ok_or(
            P2pSyncServerError::BlockNotFound {
                block_hash_or_number: BlockHashOrNumber::Number(block_number),
            },
        )?;
        let transaction_hashes = txn.get_block_transaction_hashes(block_number)?.ok_or(
            P2pSyncServerError::BlockNotFound {
                block_hash_or_number: BlockHashOrNumber::Number(block_number),
            },
        )?;
        let mut result: Vec<FullTransaction> = Vec::new();
        for (transaction, transaction_output, transaction_hash) in transactions
            .into_iter()
            .zip(transaction_outputs)
            .zip(transaction_hashes)
            .map(|((a, b), c)| (a, b, c))
        {
            result.push(FullTransaction { transaction, transaction_output, transaction_hash });
        }
        Ok(result)
    }
```
