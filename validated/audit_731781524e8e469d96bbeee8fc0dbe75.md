### Title
Slow-Mode `BurnNlp` Accepts User-Controlled `oraclePriceX18` Without On-Chain Validation, Enabling NLP Pool Drain — (`core/contracts/Clearinghouse.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

Nado's `BurnNlp` transaction type embeds `oraclePriceX18` directly in the user-signed payload. In normal sequencer operation the sequencer validates this price against real market data before batching. However, the slow-mode path (`submitSlowModeTransaction` → `executeSlowModeTransaction`) accepts and executes `BurnNlp` transactions after a 3-day delay with no on-chain price validation. A user can sign a `BurnNlp` with an arbitrarily inflated oracle price, submit it via slow-mode, and after the delay executes it to drain quote tokens from the NLP pool at a price far above market.

---

### Finding Description

`BurnNlp` processing in `EndpointTx.sol` reads the oracle price directly from the signed transaction struct:

```solidity
// EndpointTx.sol ~L554-570
} else if (txType == IEndpoint.TransactionType.BurnNlp) {
    IEndpoint.SignedBurnNlp memory signedTx = abi.decode(
        transaction[1:], (IEndpoint.SignedBurnNlp)
    );
    validateSignedTx(...);
    chargeFee(signedTx.tx.sender, HEALTHCHECK_FEE);
    priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;   // ← user-supplied
    clearinghouse.burnNlp(
        signedTx.tx,
        signedTx.oraclePriceX18,                           // ← user-supplied
        ...
    );
``` [1](#0-0) 

`Clearinghouse.burnNlp` then computes the quote payout as:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
``` [2](#0-1) 

There is no comparison of `oraclePriceX18` against any on-chain Chainlink feed, TWAP, or stored `priceX18` value. The only check is that the EIP-712 signature is valid — which it will be, because the user signed the inflated price themselves.

The slow-mode submission path in `submitSlowModeTransactionImpl` does **not** block `BurnNlp`:

```solidity
if (txType == IEndpoint.TransactionType.DepositCollateral) {
    revert();
} else if (txType == ... WithdrawInsurance || ... UpdateBuilder) {
    require(sender == owner());   // owner-only types
} else {
    chargeSlowModeFee(_getQuote(), sender);   // BurnNlp lands here
    slowModeFees += SLOW_MODE_FEE;
}
``` [3](#0-2) 

The slow-mode delay is hardcoded to 3 days:

```solidity
uint64 constant SLOW_MODE_TX_DELAY = 3 * 24 * 60 * 60; // 3 days
``` [4](#0-3) 

After 3 days any caller can invoke `executeSlowModeTransaction()`, which calls `processSlowModeTransaction` → `processSlowModeTransactionImpl` with the stored payload, executing the `BurnNlp` at the attacker-chosen price. [5](#0-4) 

The analog to the Dutch auction report is direct: in the Index Protocol, sequencer downtime lets the auction price drift to an exploitable level over time; in Nado, the slow-mode queue is a time-delayed execution path where the "price" is entirely user-controlled and never validated on-chain, so the attacker simply picks the most favorable price at signing time.

---

### Impact Explanation

An attacker holding any amount of NLP can:

1. Sign a `BurnNlp` with `oraclePriceX18` set to, e.g., 100× the real NLP NAV.
2. Submit it via `submitSlowModeTransaction` (cost: `SLOW_MODE_FEE = $1`).
3. After 3 days call `executeSlowModeTransaction`.
4. Receive `nlpAmount × inflatedPrice` in quote tokens from the NLP pool subaccount (`N_ACCOUNT`).

The NLP pool (`N_ACCOUNT`) is debited the full inflated quote amount:

```solidity
spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);   // NLP returned
// quoteAmount = nlpAmount * inflatedOraclePrice
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount); // quote paid out
_applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
``` [6](#0-5) 

The only guard is the maintenance health check on the *sender*, not on the NLP pool itself. The pool can be drained to insolvency, socialising losses across all remaining NLP holders.

---

### Likelihood Explanation

- No special privilege is required; any address that holds NLP (or can acquire a small amount) can execute this.
- The attack is most potent during sequencer downtime (when the normal sequencer path that would reject the inflated price is unavailable), but the slow-mode path is always open, making this exploitable even when the sequencer is live — the sequencer simply cannot censor a slow-mode transaction once submitted.
- The $1 slow-mode fee is negligible relative to the potential profit.
- The 3-day delay is the only friction; it is publicly known and does not prevent the attack.

---

### Recommendation

1. **Validate `oraclePriceX18` on-chain** in `burnNlp` and `mintNlp`: compare the submitted price against the last sequencer-written `priceX18[NLP_PRODUCT_ID]` and revert if the deviation exceeds a configurable tolerance (e.g., ±5%).
2. **Block `MintNlp`/`BurnNlp` in the slow-mode submission path** (add them to the owner-only list in `submitSlowModeTransactionImpl`), forcing these operations to go through the sequencer where price validation occurs.
3. As a defence-in-depth measure, add a sequencer uptime check (e.g., Chainlink L2 sequencer feed) and invalidate or pause NLP mint/burn operations if the sequencer has been offline, mirroring the recommendation from the referenced Index Protocol report.

---

### Proof of Concept

```
1. Attacker mints 1 NLP token via normal sequencer path at real price P.
2. Attacker signs SignedBurnNlp{nlpAmount: 1, ...} with oraclePriceX18 = 1000 * P.
3. Attacker calls endpoint.submitSlowModeTransaction(encodedBurnNlp), paying $1.
4. 3 days pass (SLOW_MODE_TX_DELAY).
5. Anyone calls endpoint.executeSlowModeTransaction().
6. processSlowModeTransactionImpl executes BurnNlp:
     quoteAmount = 1 * (1000 * P) = 1000P
     N_ACCOUNT is debited 1000P in quote tokens.
     Attacker receives 1000P quote tokens for NLP worth P.
7. Net profit ≈ 999P, cost = $1 + HEALTHCHECK_FEE ($1).
``` [7](#0-6) [8](#0-7)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
