### Title
ML-DSA-65 Gas-Key Host Functions Charge Fees on Wire Length (1953 B) Instead of Trie-Key Length (33 B) — (File: `runtime/near-vm-runner/src/logic/logic.rs`)

### Summary

The `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` host functions compute gas fees using the raw borsh wire length of the caller-supplied public key (`public_key_len as usize`). For an ML-DSA-65 key this is **1953 bytes**. The transaction-path fee helpers in `runtime/runtime/src/config.rs` correctly call `action.public_key.trie_id_len()`, which returns **33 bytes** for ML-DSA-65 (the SHA3-256 hash form stored in the trie). The two paths diverge by a factor of ~59×, breaking the invariant that the same logical operation costs the same gas regardless of how it is invoked.

### Finding Description

`PublicKey::len()` returns the borsh-encoded wire length (1 + 1952 = **1953** for ML-DSA-65), while `PublicKey::trie_id_len()` returns the on-trie identifier length (1 + 32 = **33** for ML-DSA-65, the SHA3-256 hash form). [1](#0-0) [2](#0-1) 

The transaction-path fee computation in `total_send_fees` and `exec_fee` correctly uses `trie_id_len()`: [3](#0-2) [4](#0-3) 

The host-function path in `promise_batch_action_transfer_to_gas_key` passes the raw caller-supplied `public_key_len` (the borsh wire length) directly to both fee helpers: [5](#0-4) 

The same pattern appears in `promise_batch_action_add_gas_key_with_full_access`: [6](#0-5) 

`gas_key_transfer_exec_fee` uses `public_key_len` to compute `access_key_key_len`, which is supposed to model the trie key length the receiver must read/write: [7](#0-6) 

With `public_key_len = 1953` instead of 33, `trie_key_len` is ~1920 bytes larger than the actual trie key, and `gas_key_add_key_exec_fee` multiplies this error by `num_nonces`: [8](#0-7) 

The design document explicitly states that every storage-stake and trie-byte-priced fee path must use `trie_id_len()`: [9](#0-8) 

### Impact Explanation

Any contract that calls `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key is charged approximately **59× more exec gas** than the equivalent transaction. With `num_nonces > 1` the overcharge compounds linearly. Contracts that budget gas based on the transaction-path cost will run out of gas mid-execution, causing silent action failures. The same logical state transition costs different gas depending on the invocation path, breaking determinism guarantees that validators rely on for consensus.

Both `GasKeys` and `PostQuantumSignatures` stabilized at protocol version 85: [10](#0-9) 

The current stable protocol version is 86, so both features are active on mainnet. [11](#0-10) 

### Likelihood Explanation

Any unprivileged user who deploys a contract and calls these host functions with an ML-DSA-65 key triggers the overcharge. No special role is required. The `PostQuantumSignatures` gate permits ML-DSA-65 keys in `AddKey` actions and transactions at v85+, so ML-DSA-65 gas keys are a supported, documented feature. The fee parity test (`test_gas_key_fee_parity`) only exercises ED25519 keys and does not catch the ML-DSA-65 divergence: [12](#0-11) 

### Recommendation

Replace `public_key_len as usize` with `public_key.decode()?.trie_id_len()` (or compute `trie_id_len` from the decoded key before charging fees) in all three host functions: `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call`, in both the interpreter path (`logic/logic.rs`) and the wasmtime path (`wasmtime_runner/logic.rs`). Add a fee-parity test that exercises ML-DSA-65 keys alongside ED25519 keys, analogous to the existing `test_gas_key_fee_parity` test.

### Proof of Concept

1. Deploy a contract that calls `promise_batch_action_transfer_to_gas_key` with a borsh-encoded ML-DSA-65 public key (1953 bytes).
2. Submit the same `TransferToGasKey` action as a direct transaction with the same ML-DSA-65 key.
3. Compare `gas_burnt` on the action execution receipt for both paths.
4. The host-function path burns approximately `1920 × gas_key_byte_exec_fee` more gas than the transaction path for the exec component, and the same divergence appears in the send component.

The exact divergent values are:
- Host function path: `public_key_len = 1953` → `trie_key_len ≈ account_id_len + 1953 + separators`
- Transaction path: `trie_id_len() = 33` → `trie_key_len ≈ account_id_len + 33 + separators`
- Ratio: ~59× overcharge on the per-byte exec fee component [13](#0-12) [14](#0-13)

```rust
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
```

```rust
pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
```
