### Title
Secp Operator Registered at Two Opcode Paths with Divergent Activation Conditions — (`File: src/chia_dialect.rs`)

### Summary
`op_secp256k1_verify` and `op_secp256r1_verify` are each wired into `ChiaDialect::op()` at **two independent opcode paths** with different flag requirements. The 4-byte opcode path (`0x13d61f00` / `0x1c3a8f00`) is unconditionally active, while the 1-byte opcode path (opcodes `64` / `65`) is gated by `ClvmFlags::ENABLE_SECP_OPS`. Because the same underlying function is reachable through both paths, the flag that is supposed to control secp operator availability is silently bypassed whenever a program uses the 4-byte encoding.

### Finding Description

In `src/chia_dialect.rs`, `ChiaDialect::op()` dispatches operators in two separate branches.

**Branch 1 — 4-byte opcodes (lines 157–182):** No flag check. Any 4-byte opcode whose value matches `0x13d61f00` or `0x1c3a8f00` unconditionally invokes the secp functions. [1](#0-0) 

**Branch 2 — 1-byte opcodes (lines 248–249):** Gated by `ClvmFlags::ENABLE_SECP_OPS`. Opcodes `64` and `65` only reach the same functions when the flag is present. [2](#0-1) 

The flag is defined as: [3](#0-2) 

And is exported to Python callers as a first-class API constant: [4](#0-3) 

The `flags` variable used in both branches is computed once from `self.flags | extension_flags`: [5](#0-4) 

The 4-byte branch returns early before the 1-byte match is reached, so the flag check in the 1-byte branch is never consulted when the 4-byte opcode is used. The two registrations of the same operator are therefore governed by two separate, independent conditions that can diverge — exactly the parallel-governor pattern from the reference report.

### Impact Explanation

A caller (Python or Rust) who constructs a `ChiaDialect` **without** `ENABLE_SECP_OPS` — intending to run a program in a mode where secp signature verification is unavailable — receives no error and no indication that the secp operators are still fully reachable via the 4-byte opcode encoding. Any attacker-supplied CLVM program that encodes the operator as the 4-byte atom `\x13\xd6\x1f\x00` or `\x1c\x3a\x8f\x00` will execute `op_secp256k1_verify` or `op_secp256r1_verify` respectively, regardless of the flag. This breaks the API contract of `ENABLE_SECP_OPS` and can cause consensus divergence: a node that believes secp ops are disabled (because it did not set the flag) will still accept and execute secp-bearing programs submitted via the 4-byte opcode path. [6](#0-5) 

### Likelihood Explanation

The 4-byte opcode values are deterministic and publicly derivable from the cost formula documented in the source. Any attacker who reads the source or the Chia protocol documentation can craft a CLVM program that uses the 4-byte encoding. The entry path is the standard `run_serialized_chia_program` Python API or `run_program` Rust API — both are externally reachable with attacker-controlled bytes. [7](#0-6) 

### Recommendation

The 4-byte opcode branch should apply the same `ENABLE_SECP_OPS` guard as the 1-byte branch. Concretely, after matching `0x13d61f00` or `0x1c3a8f00`, the code should check `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` and fall through to `unknown_operator` if the flag is absent — mirroring the guard on lines 248–249. Alternatively, if the 4-byte opcodes are intentionally always active (as a hard-fork baseline), the flag comment and name should be updated to clarify that `ENABLE_SECP_OPS` only enables the 1-byte aliases, not secp operations in general, so callers are not misled.

### Proof of Concept

```python
# Python caller believes ENABLE_SECP_OPS is not set, so secp ops are "disabled"
import clvm_rs

# flags WITHOUT ENABLE_SECP_OPS (0x0800)
flags = clvm_rs.NO_UNKNOWN_OPS  # 0x0002

# Craft a CLVM program that calls secp256k1_verify via the 4-byte opcode 0x13d61f00
# (pubkey msg sig) — the 4-byte opcode bypasses the ENABLE_SECP_OPS flag entirely.
# The program will execute op_secp256k1_verify despite the flag being absent.
program_bytes = bytes.fromhex("...")  # serialized (0x13d61f00 pubkey msg sig)
args_bytes    = bytes.fromhex("ff80")  # ()

cost, result = clvm_rs.run_serialized_chia_program(
    program_bytes, args_bytes, 11_000_000_000, flags
)
# op_secp256k1_verify runs and returns nil — secp "disabled" flag had no effect.
```

The root cause is in `src/chia_dialect.rs` lines 175–182 (4-byte path, no flag check) versus lines 248–249 (1-byte path, flag-gated) — two parallel registrations of the same operator with divergent activation conditions. [8](#0-7)
