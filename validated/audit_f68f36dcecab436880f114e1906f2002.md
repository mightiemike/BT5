### Title
CEI Violation in `withdrawCollateral` — External Token Transfer Precedes Balance Update and Health Check - (File: `core/contracts/Clearinghouse.sol`)

### Summary
`Clearinghouse.withdrawCollateral` performs two external token transfers via `handleWithdrawTransfer` before updating the sender's balance in `spotEngine` and executing the health check. This is a direct violation of the Checks-Effects-Interactions pattern. If a non-standard ERC20 token with transfer hooks (e.g., ERC777) is ever listed as collateral, a recipient contract can re-enter the protocol during the token transfer while the sender's on-chain balance has not yet been decremented, making the sender appear more solvent than they are.

---

### Finding Description

In `Clearinghouse.withdrawCollateral` (lines 391–421), the execution order is:

1. **External call 1:** `token.safeTransfer(withdrawPool, amount)` — tokens leave the Clearinghouse.
2. **External call 2:** `withdrawPool.submitWithdrawal(token, sendTo, amount, idx)` — `WithdrawPool` calls `token.safeTransfer(sendTo, amount)`, delivering tokens to the recipient.
3. **State update (deferred):** `spotEngine.updateBalance(productId, sender, amountRealized)` — balance is decremented only after both external calls complete.
4. **Health check (deferred):** `getHealth(sender, healthType) >= 0` — solvency is verified only after both external calls complete. [1](#0-0) 

The internal helper that performs the two external calls: [2](#0-1) 

`BaseWithdrawPool.submitWithdrawal` then calls `handleWithdrawTransfer`, which calls `token.safeTransfer(sendTo, amount)`: [3](#0-2) [4](#0-3) 

The correct CEI order — used in `BaseWithdrawPool.submitFastWithdrawal` — marks state (`markedIdxs[idx] = true`) **before** calling `handleWithdrawTransfer`. `withdrawCollateral` inverts this ordering for the `spotEngine` balance and health check. [5](#0-4) 

---

### Impact Explanation

During the window between external call 2 (tokens delivered to `sendTo`) and step 3 (balance decremented), the sender's `spotEngine` balance is still at its pre-withdrawal value. Any protocol function that reads this balance during re-entry will observe an inflated, stale value.

Concrete re-entry path via `Endpoint.executeSlowModeTransaction` (publicly callable, no access control):

1. Attacker holds balance `X` in `spotEngine` and has a previously queued slow-mode `WithdrawCollateral` for amount `A`.
2. Attacker submits a `WithdrawCollateralV2` for amount `B` with `sendTo` = malicious contract (ERC777 `tokensReceived` recipient).
3. During `token.safeTransfer(maliciousContract, B)`, the `tokensReceived` hook fires.
4. Hook calls `Endpoint.executeSlowModeTransaction`, processing the queued withdrawal for `A`.
5. At this point, `spotEngine` balance is still `X` (not yet decremented by the outer call). The queued withdrawal's health check sees `X - A`, which passes.
6. Balance is decremented to `X - A`. Hook returns.
7. Outer `withdrawCollateral` decrements balance to `X - A - B` and checks health.
8. If the attacker holds other collateral sufficient to cover `X - A - B < 0`, the health check passes — the attacker has withdrawn `A + B > X` in net token value while the protocol's accounting only reflects a debt backed by other collateral.

The broken invariant: **a subaccount's on-chain token balance in `spotEngine` must be decremented before tokens are transferred out**, so that any re-entrant health-dependent operation observes the correct post-withdrawal state. [6](#0-5) 

---

### Likelihood Explanation

- **Trigger condition:** A non-standard ERC20 token with transfer hooks (ERC777, ERC677, or a custom callback token) must be listed as collateral. The protocol's `IERC20Base` interface does not restrict this.
- **Attacker-controlled entry:** `WithdrawCollateralV2` allows a caller-specified `sendTo` address. A user can set `sendTo` to any contract they control.
- **Re-entry vector:** `Endpoint.executeSlowModeTransaction` is publicly callable with no access control, making it a valid re-entry target from within a token hook.
- **No privileged access required:** The entire path is reachable by an unprivileged user who submits a signed `WithdrawCollateralV2` transaction processed by the sequencer. [7](#0-6) 

---

### Recommendation

Reorder `withdrawCollateral` to follow CEI: perform `spotEngine.updateBalance` and `getHealth` **before** calling `handleWithdrawTransfer`. This matches the pattern already used in `BaseWithdrawPool.submitFastWithdrawal`, where `markedIdxs[idx] = true` is set before any external transfer.

```solidity
// Correct CEI order:
int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);   // Effect first
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH); // Check second
handleWithdrawTransfer(token, sendTo, amount, idx);             // Interaction last
```

---

### Proof of Concept

1. Deploy an ERC777 token and register it as a collateral product in the protocol.
2. Deploy a malicious recipient contract implementing `tokensReceived`; inside the hook, call `Endpoint.executeSlowModeTransaction()`.
3. Fund a subaccount with `X` units of the ERC777 token plus sufficient other collateral to cover a negative ERC777 balance.
4. Queue a slow-mode `WithdrawCollateral` for `X` tokens via `Endpoint.submitSlowModeTransaction`.
5. Submit a sequencer-processed `WithdrawCollateralV2` for `X` tokens with `sendTo` = malicious contract.
6. When the sequencer processes step 5, `handleWithdrawTransfer` delivers `X` tokens to the malicious contract; the `tokensReceived` hook fires and calls `executeSlowModeTransaction`, processing the queued withdrawal for `X` tokens while the `spotEngine` balance is still `X`.
7. Both withdrawals complete; the attacker has received `2X` tokens while `spotEngine` records a `-X` debt covered by the other collateral — net asset extraction of `X` tokens beyond the ERC777 balance. [2](#0-1) [3](#0-2) [8](#0-7)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
