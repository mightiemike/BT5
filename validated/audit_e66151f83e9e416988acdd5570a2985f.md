### Title
Caller-Controlled `bidPriceX64`/`askPriceX64` Fed to Extension Guards via `simulateSwapAndRevert`, Persisting Extension State Changes After Pool Revert — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary
`simulateSwapAndRevert` accepts caller-supplied `bidPriceX64` and `askPriceX64` and passes them directly into `_beforeSwap` and `_afterSwap` extension hooks. Because extensions are external contracts, any state they write during those hooks is **not** rolled back when the pool reverts with `SimulateSwap`. An unprivileged caller can therefore feed arbitrary prices to guard extensions (e.g., `OracleValueStopLossExtension`, `PriceVelocityGuardExtension`), corrupting their internal observations and causing them to misclassify a subsequent real swap as safe.

### Finding Description

The real `swap` path fetches prices exclusively from the oracle:

```solidity
// MetricOmmPool.sol – swap()
(uint128 bidPriceX64, uint128 askPriceX64) = _getBidAndAskPriceX64();
``` [1](#0-0) 

`simulateSwapAndRevert` instead accepts these values as caller parameters and skips the oracle call entirely:

```solidity
function simulateSwapAndRevert(
    address recipient,
    bool zeroForOne,
    int128 amountSpecified,
    uint128 priceLimitX64,
    uint128 bidPriceX64,   // ← caller-controlled
    uint128 askPriceX64,   // ← caller-controlled
    bytes calldata extensionData
) external nonReentrant(PoolActions.SIMULATE_SWAP_AND_REVERT) ...
``` [2](#0-1) 

The only validation applied is:

```solidity
if (bidPriceX64 >= askPriceX64) revert BidGreaterThanAsk();
if (bidPriceX64 == 0) revert BidIsZero();
``` [3](#0-2) 

These caller-controlled values are then forwarded verbatim into `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender, recipient, zeroForOne, amountSpecified,
    priceLimitX64, packedSlot0Initial,
    bidPriceX64, askPriceX64,   // ← attacker values
    extensionData
);
``` [4](#0-3) 

and into `_afterSwap`:

```solidity
_afterSwap(
    msg.sender, recipient, zeroForOne, amountSpecified,
    priceLimitX64, packedSlot0Initial, packedSlot0Final,
    bidPriceX64, askPriceX64,   // ← attacker values
    ...
);
``` [5](#0-4) 

`_beforeSwap` and `_afterSwap` dispatch to every configured extension in order:

```solidity
function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;
    while (true) {
        uint256 extensionIndex = order & 0x7;
        if (extensionIndex == 0) break;
        address extension = _extensionAddress(extensionIndex);
        if (extension == address(0)) revert PanicEmptyExtension();
        CallExtension.callExtension(extension, data);
        order >>= 3;
    }
}
``` [6](#0-5) 

Extensions are external contracts. Solidity's revert only rolls back the **calling contract's** storage. When the pool finally executes:

```solidity
_nonReentrantAfter();
revert SimulateSwap(amount0Delta, amount1Delta);
``` [7](#0-6) 

the pool's own storage is restored, but **every storage write made by extension contracts during those hook calls is permanently committed**. Guard extensions such as `OracleValueStopLossExtension` and `PriceVelocityGuardExtension` that record price observations or update "last-seen" state in `beforeSwap`/`afterSwap` will retain the attacker-supplied values.

Additionally, `inSwap()` returns `address(0)` during `simulateSwapAndRevert` (it checks for `PoolActions.SWAP`, not `SIMULATE_SWAP_AND_REVERT`), so extensions cannot distinguish a simulation call from a real one through the pool's own API:

```solidity
function inSwap() external view returns (address priceProvider_) {
    if (_currentAction() == PoolActions.SWAP) {
        return _resolvedPriceProvider();
    }
    return address(0);
}
``` [8](#0-7) 

### Impact Explanation

A guard extension that stores a price observation in `beforeSwap` (e.g., to detect velocity or stop-loss conditions) will record the attacker-supplied `bidPriceX64`/`askPriceX64` as a legitimate oracle reading. On the next real `swap` call, the guard compares the true oracle price against this poisoned baseline. Depending on the guard's logic, this can:

- **Suppress a stop-loss trigger**: attacker pre-seeds a fake "safe" price so the guard does not halt a subsequent swap that would otherwise be blocked, allowing a swap at a price that drains LP reserves.
- **Suppress a velocity trigger**: attacker pre-seeds a fake price close to the current oracle price, making a large real price move appear within the configured velocity threshold, bypassing the guard entirely.

Both outcomes allow swaps to execute at prices the guard was specifically configured to block, resulting in direct loss of LP principal.

### Likelihood Explanation

- `simulateSwapAndRevert` is a public, permissionless function with no access control.
- The only input constraints (`bid < ask`, `bid != 0`) are trivially satisfied by any attacker.
- The attack requires a single transaction and no special privileges or capital.
- Any pool that configures a price-observation guard extension is affected.

### Recommendation

1. **Do not call extension hooks inside `simulateSwapAndRevert`**, or pass a simulation flag so extensions can skip state-writing logic.
2. Alternatively, fetch prices from the oracle inside `simulateSwapAndRevert` (as `swap` does) rather than accepting them as parameters, so extensions always receive authentic oracle data.
3. Add a `isSimulation` parameter to the extension interface so guard extensions can distinguish simulation calls and skip persistent state updates.

### Proof of Concept

```
1. Pool is configured with OracleValueStopLossExtension as EXTENSION_1,
   with BEFORE_SWAP_ORDER = 0x1 (call extension 1 before swap).

2. Oracle currently reports bid=100, ask=101.
   Stop-loss guard is configured to halt swaps if price drops >5% from
   last recorded observation (stored in extension storage).

3. Attacker calls:
     pool.simulateSwapAndRevert(
         recipient, zeroForOne=true, amountSpecified=1,
         priceLimitX64=0,
         bidPriceX64=200,   // fake high price
         askPriceX64=201,
         ""
     )
   → _beforeSwap fires → OracleValueStopLossExtension.beforeSwap()
     records lastObservedPrice = 200 in extension storage.
   → Pool reverts with SimulateSwap; pool storage restored.
   → Extension storage retains lastObservedPrice = 200.

4. Oracle price then drops to bid=90 (a real 10% drop that should
   trigger the stop-loss).

5. Attacker (or anyone) calls pool.swap(...).
   → _beforeSwap fires → guard compares 90 against lastObservedPrice=200
     → 55% drop → should trigger stop-loss.
   BUT because the baseline was poisoned to 200, the guard's threshold
   arithmetic may now compute a different reference, or the attacker
   can repeat step 3 with bidPriceX64=95 to narrow the apparent drop
   to <5%, bypassing the guard entirely.

6. Swap executes at the depressed oracle price, draining token0 from
   LP bins at a rate the stop-loss was meant to prevent.
```
