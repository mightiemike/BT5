### Title
Checks-Effects-Interactions Violation in `withdrawCollateral` Enables Re-Entrancy via `submitFastWithdrawal` - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` transfers tokens to the recipient **before** updating the SpotEngine balance. If the collateral token supports transfer callbacks (e.g., ERC777), the recipient contract can re-enter the publicly callable `BaseWithdrawPool.submitFastWithdrawal` during the transfer, consuming additional WithdrawPool liquidity against a balance that has not yet been decremented on-chain. The subsequent health check passes because it only accounts for the regular withdrawal, leaving the subaccount under-collateralized.

---

### Finding Description

`Clearinghouse.withdrawCollateral` violates the checks-effects-interactions pattern. The token transfer to `sendTo` is executed at line 408 via `handleWithdrawTransfer`, which internally calls `token.safeTransfer(withdrawPool, amount)` followed by `BaseWithdrawPool.submitWithdrawal` → `token.safeTransfer(sendTo, amount)`. Only after this full transfer chain completes does the function decrement the SpotEngine balance at line 412 and perform the health check at line 419. [1](#0-0) 

The internal `handleWithdrawTransfer` first moves tokens from the Clearinghouse to the WithdrawPool, then calls `submitWithdrawal` which immediately forwards them to `sendTo`: [2](#0-1) 

`BaseWithdrawPool.submitWithdrawal` marks `markedIdxs[idx] = true` and updates `minIdx = idx` before the transfer, then calls `handleWithdrawTransfer(token, sendTo, amount)`: [3](#0-2) 

`handleWithdrawTransfer` in `BaseWithdrawPool` calls `token.safeTransfer(to, amount)` directly: [4](#0-3) 

At the moment `sendTo` receives tokens, the SpotEngine balance for `sender` is still at its pre-withdrawal value. `submitFastWithdrawal` is publicly callable with no access control on `msg.sender`: [5](#0-4) 

`submitFastWithdrawal` marks `markedIdxs[idx] = true` before its own transfer (line 88), so the same fast-withdrawal `idx` cannot be replayed. However, a different `idx` (greater than the `minIdx` just set by `submitWithdrawal`) is accepted. The WithdrawPool holds the freshly transferred tokens from the Clearinghouse, so it has liquidity to pay out the fast withdrawal. `submitFastWithdrawal` does **not** update the SpotEngine balance on-chain — that decrement is deferred to the sequencer off-chain.

When control returns to `withdrawCollateral`, `spotEngine.updateBalance` decrements the balance only by the regular withdrawal amount, and the health check at line 419 passes because it does not yet reflect the fast withdrawal. The sequencer later applies the fast withdrawal's balance decrement off-chain, leaving the subaccount insolvent.

---

### Impact Explanation

An attacker with a subaccount of balance `B` can receive `R` (regular withdrawal) + `F` (fast withdrawal) tokens in a single transaction while the on-chain health check only validates against `B - R`. If `R + F > B`, the protocol absorbs the shortfall from WithdrawPool liquidity providers. The corrupted state is the SpotEngine balance for `sender`: it is decremented by `R` on-chain but the health invariant was evaluated before `F` was also removed from the pool, enabling insolvency. [6](#0-5) 

---

### Likelihood Explanation

Exploitability requires the collateral token to support transfer callbacks (ERC777 `tokensReceived` or equivalent hook). The attacker must also hold a sequencer-signed fast withdrawal transaction for the same subaccount, which is obtainable through the normal fast-withdrawal request flow. The timing constraint (regular withdrawal and fast withdrawal signed for the same subaccount) is realistic: an attacker can request a fast withdrawal, receive the signed transaction, and then wait for the sequencer to also process a pending regular withdrawal for the same subaccount. The `submitFastWithdrawal` entry point is permissionless.

---

### Recommendation

Apply the checks-effects-interactions pattern in `Clearinghouse.withdrawCollateral`: update the SpotEngine balance and perform the health check **before** calling `handleWithdrawTransfer`. Move lines 410–419 to execute before line 408:

```solidity
// 1. Compute and apply balance delta first
int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);
spotEngine.assertUtilization(productId);

IProductEngine.HealthType healthType = sender == X_ACCOUNT
    ? IProductEngine.HealthType.PNL
    : IProductEngine.HealthType.INITIAL;
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

// 2. Then perform the external token transfer
handleWithdrawTransfer(token, sendTo, amount, idx);
```

Additionally, consider adding a `ReentrancyGuard` (OpenZeppelin `nonReentrant`) to `submitFastWithdrawal` and `withdrawCollateral` as a defense-in-depth measure.

---

### Proof of Concept

1. Attacker deploys contract `Exploit` that implements ERC777 `tokensReceived`.
2. Attacker registers `Exploit` as the subaccount address and deposits collateral `B` in an ERC777-backed product.
3. Attacker requests a fast withdrawal of amount `F` from the sequencer; receives a signed `WithdrawCollateralV2` transaction with `idx = N+1`.
4. Sequencer submits a regular `WithdrawCollateral` for amount `R` (idx = `N`) to the Endpoint.
5. Endpoint → `Clearinghouse.withdrawCollateral` → `handleWithdrawTransfer` → `submitWithdrawal` → `token.safeTransfer(Exploit, R)`.
6. ERC777 fires `tokensReceived` on `Exploit`. Inside the hook, `Exploit` calls `WithdrawPool.submitFastWithdrawal(N+1, signedTx, sigs)`.
   - `markedIdxs[N+1]` is false, `N+1 > minIdx (N)` → passes.
   - WithdrawPool holds `R` tokens (just received from Clearinghouse) → transfer of `F` to `Exploit` succeeds.
7. `tokensReceived` returns. Control resumes in `withdrawCollateral`.
8. `spotEngine.updateBalance` decrements balance by `R` → balance is `B - R`.
9. `getHealth` passes (assuming `B - R >= 0`).
10. Off-chain sequencer processes the fast withdrawal, decrementing balance by `F` → final balance `B - R - F`.
11. If `B - R - F < 0`, the subaccount is insolvent; the deficit is borne by WithdrawPool liquidity providers. [7](#0-6) [8](#0-7)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
