### Title
ML-DSA-65 Gas Key Host Functions Use Borsh-Encoded Key Length Instead of Trie-ID Length for Exec Fee Calculation — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The host functions `promise_batch_action_transfer_to_gas_key` and `promise_batch_action_add_gas_key_with_full_access` (and the function-call variant) pass the raw borsh-encoded `public_key_len` from WASM memory directly to `gas_key_transfer_exec_fee` and `gas_key_add_key_exec_fee`. Those fee helpers use the value as the on-trie key length. For ed25519 and secp256k1 the two lengths are identical, so the mismatch is invisible. For ML-DSA-65 they diverge by 1920 bytes (borsh = 1953, trie-id = 33), causing the exec fee to be computed against the wrong representation — the same "wrong layer's encoding used downstream" pattern as the external double-decode bug.

---

### Finding Description

**Two representations of an ML-DSA-65 public key exist in the protocol:**

| Representation | Value | Used by |
|---|---|---|
| `PublicKey::len()` — borsh-encoded | 1 + 1952 = **1953 bytes** | wire format, WASM memory |
| `PublicKey::trie_id_len()` — on-trie hash | 1 + 32 = **33 bytes** | trie key, storage accounting |

The runtime's `access_key_storage_usage` correctly uses `trie_id_len()`: [1](#0-0) 

The host function `promise_batch_action_transfer_to_gas_key` receives `public_key_len: u64` — the byte length of the borsh-encoded key as laid out in WASM linear memory — and passes it unchanged to both the send-fee and exec-fee helpers: [2](#0-1) 

The exec-fee helper `gas_key_transfer_exec_fee` uses that value as the on-trie key length: [3](#0-2) 

Its own doc comment states the intent: *"Based on the access key trie key length + estimated value length (what the receiver needs to read/write in the trie)"*. For ML-DSA-65 the trie key holds a 33-byte SHA3-256 hash, not the 1953-byte borsh key. Passing 1953 instead of 33 inflates the per-byte exec component by **1920 × `gas_key_byte` exec fee** per call.

The identical mismatch exists in the wasmtime runner: [4](#0-3) 

And in `promise_batch_action_add_gas_key_with_full_access` / `promise_batch_action_add_gas_key_with_function_call`, which pass `public_key_len as usize` to `gas_key_add_key_exec_fee`: [5](#0-4) 

`gas_key_add_key_exec_fee` uses the value to compute the nonce trie key length: [6](#0-5) 

The correct trie-id length for ML-DSA-65 is 33, not 1953: [7](#0-6) 

---

### Impact Explanation

Once `ProtocolFeature::PostQuantumSignatures` activates, any on-chain contract that calls `promise_batch_action_transfer_to_gas_key` or `promise_batch_action_add_gas_key_with_full_access` with an ML-DSA-65 key will have its exec gas charged against a 1953-byte trie key instead of the actual 33-byte trie key. The exec fee is inflated by ~58× for the per-byte component. Contracts that budget gas based on the correct trie cost will hit out-of-gas failures. The send fee (charged to the sender) uses the borsh length intentionally (the sender physically transmits the 1953-byte key), so only the exec side is wrong. The divergence is deterministic across all nodes, so it does not break consensus, but it breaks the protocol invariant that exec fees reflect actual trie I/O cost and causes ML-DSA-65 gas key operations to be economically broken for contract callers.

---

### Likelihood Explanation

The bug is latent until `PostQuantumSignatures` activates. After activation, any unprivileged contract that holds or receives an ML-DSA-65 public key and calls the gas-key host functions triggers the mismatch. No privileged role is required. The test contract already exercises this path: [8](#0-7) 

---

### Recommendation

In `promise_batch_action_transfer_to_gas_key` and the gas-key `AddKey` host functions, decode the public key first, then derive the trie-id length from the decoded key before passing it to the exec-fee helpers:

```rust
let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_key = public_key.decode()?;
let trie_id_len = decoded_key.trie_id_len();

let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize); // borsh len: correct for send
let exec = gas_key_transfer_exec_fee(&self.fees_config, receiver_id.len(), trie_id_len); // trie len: correct for exec
```

This mirrors how `access_key_storage_usage` already uses `public_key.trie_id_len()` on the runtime side, making the two layers consistent.

---

### Proof of Concept

For an ML-DSA-65 key:
- `public_key_len` passed by WASM contract = **1953** (borsh: 1 tag + 1952 raw bytes)
- `trie_id_len()` = **33** (1 tag + 32-byte SHA3-256 hash)

`gas_key_transfer_exec_fee` computes:
```
trie_key_len = access_key_key_len(account_id_len, 1953)
             = 1 + account_id_len + 1 + 1953
```
Actual trie key length:
```
             = 1 + account_id_len + 1 + 33
```
Excess bytes charged: **1920** × `gas_key_byte` exec fee per `TransferToGasKey` call.

For `gas_key_add_key_exec_fee` with `num_nonces` nonces, the excess is **1920 × num_nonces** × `gas_key_byte` exec fee, compounding with each nonce slot written.

pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
```

pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
```
