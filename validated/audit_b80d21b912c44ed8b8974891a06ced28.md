Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on `owner` instead of `sender`, allowing full allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual depositor / direct caller of `pool.addLiquidity`) and checks only `owner` (the freely chosen position beneficiary). Because `MetricOmmPoolLiquidityAdder` accepts an arbitrary `owner` from any caller, any unprivileged address can bypass the deposit allowlist by supplying an already-allowlisted address as `owner` while paying for the deposit themselves, fully nullifying the pool admin's access-control intent.

## Finding Description

`MetricOmmPool.addLiquidity` calls the hook as `_beforeAddLiquidity(msg.sender, owner, ...)`, where `msg.sender` is the direct caller (e.g., `MetricOmmPoolLiquidityAdder`) and `owner` is the position beneficiary passed in by that caller: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives both but names only `owner`, leaving `sender` as an unnamed (discarded) parameter, and gates exclusively on `owner`: [2](#0-1) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a caller-supplied `owner` with no restriction beyond `owner != address(0)`, then passes `msg.sender` (the attacker) as `payer` and the attacker-supplied `owner` as the position beneficiary: [3](#0-2) 

The hook sees `allowedDepositor[pool][allowlistedAddress]` → `true` and passes. Tokens are pulled from the unauthorized payer; LP shares are minted to the allowlisted `owner`. The actual economically relevant actor (the payer) is never checked.

`addLiquidityWeighted` compounds this: the probe call also passes through `beforeAddLiquidity` with the attacker-supplied `owner`, leaking pool state (token needs per bin) before the paying deposit executes: [4](#0-3) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly gates on `sender` (the first, named parameter), not on `recipient`: [5](#0-4) 

The two guards are architecturally inconsistent: the swap guard checks the right identity; the deposit guard does not.

## Impact Explanation

The pool admin's deposit allowlist is fully nullified. Any unprivileged caller with capital can add liquidity to a restricted pool by supplying one allowlisted address as `owner`. Consequences include: unauthorized liquidity injection into restricted pools, manipulation of bin liquidity distribution (shifting the pool cursor and altering effective bid/ask spread for all subsequent swaps), and dilution of existing LPs' share of spread and notional fees — constituting a direct reduction in owed LP assets. This is a broken core pool functionality / admin-boundary bypass causing loss of owed LP assets and corrupted pool state, meeting Critical/High Sherlock thresholds.

## Likelihood Explanation

- `MetricOmmPoolLiquidityAdder` is a public, permissionless periphery contract with no access control on callers.
- The attacker only needs one allowlisted address, observable on-chain via `AllowedToDepositSet` events or direct `allowedDepositor` reads.
- No privileged access, no special token behavior, and no malicious pool setup is required.
- The attack is executable in a single transaction and is trivially repeatable across any bin configuration.

## Recommendation

Gate on `sender` (the direct caller of `pool.addLiquidity`) rather than `owner` (the freely chosen position beneficiary), mirroring `SwapAllowlistExtension`:

```solidity
// Before (wrong):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// After (correct):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {

## Proof of Concept

```
Setup:
  - Pool P deployed with DepositAllowlistExtension E.
  - Pool admin calls setAllowedToDeposit(P, Alice, true).
  - Bob (unauthorized) holds token0/token1 and has approved LiquidityAdder.

Attack (single tx, Bob as msg.sender):
  1. Bob calls:
       LiquidityAdder.addLiquidityExactShares(
           pool  = P,
           owner = Alice,   // allowlisted address observed on-chain
           salt  = 0,
           deltas = { binIdxs: [0], shares: [1_000_000] },
           maxAmountToken0 = X,
           maxAmountToken1 = Y,
           extensionData = ""
       )

  2. LiquidityAdder calls pool.addLiquidity(Alice, 0, deltas, KIND_PAY, "").

  3. Pool calls E.beforeAddLiquidity(LiquidityAdder, Alice, 0, deltas, "").
     Hook checks allowedDepositor[P][Alice] → true → passes.

  4. Pool mints LP shares to Alice; callback pulls tokens from Bob (payer).

Result:
  - Bob (unauthorized) has deposited into the restricted pool.
  - Alice receives LP shares she did not request.
  - The deposit allowlist provided zero protection against Bob.
  - Bob can repeat across different bins to manipulate pool liquidity distribution.
```
