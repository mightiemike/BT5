### Title
Secp256k1/r1 Operators Bypass `ENABLE_SECP_OPS` Flag and `NO_UNKNOWN_OPS` Guard via 4-Byte Opcode Path — (File: `src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches secp256k1 and secp256r1 verification operators through two separate code paths. The 1-byte opcode path (opcodes 64 and 65) correctly enforces the `ENABLE_SECP_OPS` flag. The 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) dispatches to the same underlying functions with **no flag check at all**, and also bypasses the `NO_UNKNOWN_OPS` guard that `MEMPOOL_MODE` relies on to reject non-standard operators. An attacker-controlled CLVM program can invoke secp cryptographic operations in any execution context — including strict mempool mode — simply by encoding the operator as a 4-byte atom.

---

### Finding Description

`ChiaDialect::op()` branches first on operator atom length:

**4-byte path** — no `ENABLE_SECP_OPS` check, no `NO_UNKNOWN_OPS` check:
```rust
// src/chia_dialect.rs lines 157–182
if op_len == 4 {
    let opcode = u32::from_be_bytes(...);
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // dispatched unconditionally
        0x1c3a8f00 => op_secp256r1_verify,   // dispatched unconditionally
        _ => { return unknown_operator(...); }
    };
    return f(allocator, argument_list, max_cost, flags); // no flag gate
}
```

**1-byte path** — `ENABLE_SECP_OPS` enforced:
```rust
// src/chia_dialect.rs lines 248–249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

`MEMPOOL_MODE` is defined as:
```rust
// src/chia_dialect.rs lines 72–76
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
// ENABLE_SECP_OPS is absent
```

Two invariants are simultaneously broken for the 4-byte secp opcodes in `MEMPOOL_MODE`:

1. **`ENABLE_SECP_OPS` is not checked.** The flag's documented purpose is "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)." The 4-byte aliases invoke the identical functions without consulting this flag.

2. **`NO_UNKNOWN_OPS` is not enforced.** `MEMPOOL_MODE` sets `NO_UNKNOWN_OPS` to reject non-standard operators. All other unrecognized 4-byte operators fall through to `unknown_operator()`, which checks this flag and returns `Err(EvalErr::Unimplemented)`. The two secp opcodes are explicitly matched and returned early, before `unknown_operator()` is ever reached, so `NO_UNKNOWN_OPS` has no effect on them.

The result: in `MEMPOOL_MODE`, a CLVM program using 1-byte opcode `64` is rejected, but the same program rewritten to use 4-byte opcode `0x13d61f00` is accepted and executes `op_secp256k1_verify` at full cost (1,300,000). The access control layer that `ENABLE_SECP_OPS` and `NO_UNKNOWN_OPS` are supposed to provide is completely absent on the 4-byte path.

---

### Impact Explanation

**Operator-level access control bypass.** The analog to the RBAC finding is direct: `ENABLE_SECP_OPS` is the "role" that authorizes secp operations, but the authorization check is only wired to one of the two opcode encodings. Any caller that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` — including the standard `MEMPOOL_MODE` — and expects secp operations to be unavailable is wrong. The 4-byte encoding bypasses the role check entirely.

**Mempool/consensus policy divergence.** The mempool is supposed to be strictly more restrictive than consensus. A transaction using 4-byte secp opcodes is accepted by the mempool (4-byte path, no flag check) but would be evaluated identically in consensus. A transaction using 1-byte opcode 64 is rejected by the mempool. This creates an inconsistency: the mempool's operator access policy depends on which encoding the submitter chose, not on whether the operation is permitted.

**Corrupted invariant.** The concrete broken invariant is: `flags.contains(NO_UNKNOWN_OPS)` should cause any operator not in the hardcoded baseline set to be rejected. The 4-byte secp opcodes are not in the baseline set (they are described in comments as "unknown operators with assigned cost"), yet they are dispatched successfully even when `NO_UNKNOWN_OPS` is set.

**Impact score: 4** — mempool access control is bypassed for a class of cryptographic operators; the same operation produces different accept/reject outcomes depending solely on opcode encoding.

---

### Likelihood Explanation

**Likelihood score: 4** — The trigger requires only crafting a CLVM program that uses a 4-byte atom (`0x13d61f00` or `0x1c3a8f00`) as the operator. No special privileges, no compromised infrastructure, and no social engineering are needed. The 4-byte encoding is a valid CLVM atom and is accepted by all serialization paths. Any party that can submit a CLVM program to a node running in mempool mode can exercise this path.

---

### Recommendation

The 4-byte secp opcode dispatch block should apply the same flag gates as the 1-byte path. Specifically:

- Check `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` before dispatching `0x13d61f00` and `0x1c3a8f00`, and fall through to `unknown_operator()` if the flag is absent. This restores the `NO_UNKNOWN_OPS` enforcement as well, since `unknown_operator()` already checks that flag.

```rust
// proposed fix
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => { return unknown_operator(allocator, o, argument_list, flags, max_cost); }
};
```

If the 4-byte opcodes are intentionally always available (pre-dating the flag), the flag should be renamed to `ENABLE_SECP_OPS_SHORT_ALIASES` and its documentation updated to make clear that the 4-byte forms are unconditionally active, so callers do not incorrectly rely on the flag to disable secp operations.

### Proof of Concept

**Entry path:** Attacker submits a CLVM program to a node running in `MEMPOOL_MODE` (i.e., `ChiaDialect::new(MEMPOOL_MODE)`).

**Program (pseudocode):**
```
; operator is the 4-byte atom 0x13d61f00 (secp256k1_verify)
; arguments: attacker-supplied pubkey, msg, sig
(0x13d61f00 pubkey msg sig)
```

**Expected behavior under `MEMPOOL_MODE`:** Rejected, because `ENABLE_SECP_OPS` is absent from `MEMPOOL_MODE` and `NO_UNKNOWN_OPS` is set.

**Actual behavior:** `op()` enters the `op_len == 4` branch at line 157, matches `0x13d61f00` at line 176, and calls `op_secp256k1_verify` at line 182 — returning either `Ok(Reduction(1_300_000, nil))` on valid signature or `Err(Secp256Failed)` on invalid signature. Neither `ENABLE_SECP_OPS` nor `NO_UNKNOWN_OPS` is consulted.

**Contrast:** The same operation encoded as 1-byte opcode `64` falls to line 248, the `if flags.contains(ClvmFlags::ENABLE_SECP_OPS)` guard fails, execution falls to the `_` arm at line 250, and `unknown_operator()` is called, which returns `Err(EvalErr::Unimplemented)` because `NO_UNKNOWN_OPS` is set.

The two encodings of the same cryptographic operator produce opposite accept/reject outcomes under identical dialect flags. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
