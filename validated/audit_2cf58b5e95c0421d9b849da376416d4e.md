### Title
Lack of Slippage Protection in `burnNlp` / `mintNlp` — (`core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp` and `mintNlp` functions in `Clearinghouse.sol` execute NLP redemptions and mints at a sequencer-supplied `oraclePriceX18` that is **not covered by the user's EIP-712 signature**. Because the user's signed payload contains no minimum-output or maximum-input bound, the user has no on-chain protection against receiving fewer quote tokens (on burn) or fewer NLP tokens (on mint) than they expected when they signed the transaction.

---

### Finding Description

When a user wants to burn NLP tokens and receive quote, they sign a `BurnNlp` struct:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;   // NLP to burn — user-controlled
    uint64  nonce;
}
```

The `Verifier.sol` digest for `BurnNlp` covers only these three fields:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(BURN_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.nlpAmount,
        signedTx.tx.nonce
    )
);
``` [1](#0-0) 

The `oraclePriceX18` field lives in `SignedBurnNlp` (the sequencer-assembled wrapper), not in `BurnNlp` (the user-signed payload):

```solidity
struct SignedBurnNlp {
    BurnNlp tx;
    bytes   signature;
    int128  oraclePriceX18;      // sequencer-supplied, NOT signed by user
    int128[] nlpPoolRebalanceX18;
}
``` [2](#0-1) 

Inside `Clearinghouse.burnNlp`, the quote the user receives is computed entirely from this unsigned price:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee     = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount        = MathHelper.max(0, quoteAmount - burnFee);
``` [3](#0-2) 

There is no `minQuoteAmount` field in `BurnNlp`, and no on-chain check that `quoteAmount` meets any user-specified floor. The same structural gap exists in `mintNlp`: the user signs a fixed `quoteAmount` but receives `nlpAmount = quoteAmount.div(oraclePriceX18)` with no `minNlpAmount` guard. [4](#0-3) 

---

### Impact Explanation

A user who signs a `BurnNlp` for `N` NLP tokens expecting to receive `N × P_signed` quote (where `P_signed` is the NLP price at signing time) will receive `N × P_exec − fee` quote at execution time, where `P_exec` is whatever `oraclePriceX18` the sequencer supplies. If the NLP pool has suffered losses between signing and execution (e.g., traders profited against the pool, reducing NLP NAV), `P_exec < P_signed` and the user receives fewer quote tokens than expected with no ability to revert. The corrupted state delta is a direct reduction in the user's `QUOTE_PRODUCT_ID` balance credit with no recourse.

---

### Likelihood Explanation

NLP price is a function of the pool's net asset value, which fluctuates continuously as traders open and close positions against the pool. The sequencer batches transactions with a delay; any NLP price decline between a user signing `BurnNlp` and the sequencer including it in a batch silently reduces the user's payout. This is a normal operating condition, not an edge case.

---

### Recommendation

Add a `minQuoteAmount` field to the `BurnNlp` struct (and a `minNlpAmount` field to `MintNlp`), include them in the EIP-712 digest in `Verifier.sol`, and enforce them in `Clearinghouse.burnNlp` / `mintNlp`:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64  nonce;
    int128  minQuoteAmount;  // new: user-specified floor
}
```

In `burnNlp`:
```solidity
require(quoteAmount >= txn.minQuoteAmount, "STH"); // slippage too high
```

This mirrors the pattern already used elsewhere in the codebase (`ERR_SLIPPAGE_TOO_HIGH` already exists as a defined error string). [5](#0-4) 

---

### Proof of Concept

1. Alice signs `BurnNlp { sender: alice, nlpAmount: 1000e18, nonce: 5 }` when NLP oracle price is `1.00 USDC/NLP`. She expects ≈ `999 USDC` (after 0.1% fee).
2. Before the sequencer includes her transaction, a large trader profits against the NLP pool, dropping the NLP NAV. The sequencer now uses `oraclePriceX18 = 0.90e18`.
3. `burnNlp` executes: `quoteAmount = 1000 × 0.90 = 900`, fee = `max(1, 0.9) = 1`, net = `899 USDC`.
4. Alice receives `899 USDC` instead of `999 USDC` — a loss of `100 USDC` — with no on-chain mechanism to have prevented it.

The entry path is: user signs `BurnNlp` → sequencer calls `submitTransactionsChecked` → `EndpointTx` decodes and calls `clearinghouse.burnNlp(signedTx.tx, signedTx.oraclePriceX18, ...)` → quote credited at sequencer-chosen price. [6](#0-5) [7](#0-6)
