### Title
`getSubaccountId()` Returns Zero for Non-Existent Subaccounts Without Reverting — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.getSubaccountId()` silently returns `0` for any subaccount that has never been registered, instead of reverting. Because the protocol assigns IDs starting from `1`, the value `0` is the sentinel for "does not exist," yet the function exposes this ambiguous default to any external caller with no error signal.

---

### Finding Description

`_recordSubaccount` assigns IDs via `++numSubaccounts`, so the first registered subaccount receives ID `1` and `0` is permanently reserved as the "not-found" sentinel. [1](#0-0) 

The public getter, however, performs no existence check:

```solidity
function getSubaccountId(bytes32 subaccount)
    external
    view
    returns (uint64)
{
    return subaccountIds[subaccount];   // returns 0 for any unknown subaccount
}
``` [2](#0-1) 

Any caller passing an unregistered `subaccount` receives `0` — the same value that would be returned if the mapping slot simply defaulted — with no revert and no indication that the query was invalid. Compare this with `requireSubaccount`, which explicitly checks `subaccountIds[subaccount] != 0` before proceeding: [3](#0-2) 

The public getter provides none of that guard.

---

### Impact Explanation

**Impact: Low.**  
Off-chain clients, integrators, and smart-contract callers that rely on `getSubaccountId` to verify whether a subaccount is registered receive a silent `0` for any arbitrary input. They cannot distinguish "valid subaccount whose ID happens to be 0" from "subaccount that was never deposited." This can cause integrators to treat unregistered subaccounts as registered (ID = 0 is truthy in some off-chain contexts), or to silently skip error handling that should have been triggered.

---

### Likelihood Explanation

**Likelihood: High.**  
The function is `external view` with no access control. Any unprivileged caller — including off-chain tooling, aggregators, or other contracts — can query it with an arbitrary `bytes32`. Every query for a non-existent subaccount silently returns `0`.

---

### Recommendation

Add an existence check and revert for unregistered subaccounts:

```solidity
function getSubaccountId(bytes32 subaccount)
    external
    view
    returns (uint64)
{
    uint64 id = subaccountIds[subaccount];
    require(id != 0, ERR_SUBACCOUNT_NOT_FOUND);
    return id;
}
```

`ERR_SUBACCOUNT_NOT_FOUND` (`"SNF"`) is already defined in `core/contracts/common/Errors.sol`. [4](#0-3) 

---

### Proof of Concept

1. Deploy or connect to `Endpoint`.
2. Call `getSubaccountId(bytes32(uint256(0xdeadbeef)))` — an address that has never deposited.
3. Observe return value `0` with no revert.
4. Call `getSubaccountId` for a legitimately registered subaccount — also returns a `uint64`, but callers have no on-chain way to distinguish the two cases without separately calling `requireSubaccount` (which is `internal`).
