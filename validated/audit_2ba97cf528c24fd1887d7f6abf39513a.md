### Title
Incorrect Block Timestamp Used in Fork Difficulty Calculation — (`contract/src/dogecoin.rs`)

### Summary

In the Dogecoin difficulty adjustment path, `get_next_work_required` fetches the boundary block's timestamp via `get_header_by_height`, which always resolves against the **mainchain** height-to-header mapping. When a fork block is being validated and the fork diverged before the difficulty-window boundary, the mainchain block at that height is a **different block** than the fork's true ancestor. The wrong timestamp is therefore fed into `calculate_next_work_required`, producing an incorrect difficulty target — an exact structural analog to Perennial's incorrect price used during liquidation.

The developers themselves flagged this with a `TODO` comment at the exact call site.

---

### Finding Description

In `get_next_work_required` (Dogecoin path), the difficulty boundary block is retrieved as:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

This unconditionally resolves the block hash from `mainchain_height_to_header` — the **current mainchain** mapping — not from the fork's ancestry chain.

For post-145,000 Dogecoin blocks, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `height_first = prev_block_header.block_height - 1`. [3](#0-2) 

When a fork block at height `H+2` is submitted (fork diverged at `H+1`), `height_first = H+1`. The mainchain block at `H+1` is a **different block** than the fork's block at `H+1`. The timestamp of the mainchain block — not the fork ancestor — is passed to `calculate_next_work_required`, producing a corrupted difficulty target. [4](#0-3) 

The same structural flaw exists in the Bitcoin path:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [5](#0-4) 

For Bitcoin the window is 2016 blocks, making exploitation practically infeasible. For Dogecoin post-145,000 the window is 1 block, making any fork ≥ 2 blocks deep a trigger.

---

### Impact Explanation

The corrupted `first_block_time` value is the sole time-span input to `calculate_next_work_required`. A mainchain block at `height_first` with a **later** timestamp than the true fork ancestor produces a **larger** `modulated_timespan`, which inflates `new_target`, yielding a **lower (easier) difficulty**. An attacker who controls the fork chain can therefore have fork blocks accepted with less proof-of-work than the protocol requires. Conversely, a mainchain block with an earlier timestamp produces a harder target, causing valid fork blocks to be incorrectly rejected (denial of fork promotion). Either outcome corrupts the canonical-chain determination and the `chain_work` comparison used for reorg decisions. [6](#0-5) 

---

### Likelihood Explanation

For Dogecoin (post-height 145,000, which is the current live state of the chain), every fork block submitted at depth ≥ 2 from the divergence point triggers the wrong lookup. Any relayer or unprivileged NEAR caller invoking `submit_blocks` with a sequence of fork headers exercises this path. No special privileges, leaked keys, or social engineering are required. The entry point is the public `submit_blocks` method. [7](#0-6) 

---

### Recommendation

Replace the height-based mainchain lookup with an ancestor walk starting from `prev_block_header`. The correct approach is to traverse `get_prev_header` backwards `blocks_to_go_back` times from `prev_block_header`, which always follows the fork's own ancestry regardless of what the mainchain mapping contains. This is exactly what the `TODO` comment at line 291 of `dogecoin.rs` anticipates.

---

### Proof of Concept

1. Deploy the contract in Dogecoin mode with a genesis at height ≥ 145,001.
2. Submit a valid mainchain sequence: blocks at heights H, H+1, H+2 with timestamps T0, T1, T2.
3. Now submit a fork block at height H+1 (different hash, same parent as H). This is stored as a fork header.
4. Submit a fork block at height H+2 whose parent is the fork block at H+1.
   - `check_pow` → `get_next_work_required` → `height_first = H+1`
   - `get_header_by_height(H+1)` returns the **mainchain** block at H+1 with timestamp T1.
   - The fork block at H+1 has timestamp T1' ≠ T1.
   - `calculate_next_work_required` uses T1 (wrong) instead of T1' (correct).
   - If T1 > T1', the computed target is inflated; the fork block passes PoW with less work than required.
5. Submit a fork block at height H+3 with chain_work exceeding the mainchain tip — the reorg is accepted, and the canonical chain is now a chain whose blocks were validated against incorrect difficulty targets. [8](#0-7) [9](#0-8)

// source https://github.com/dogecoin/dogecoin/blob/2c513d0172e8bc86fe9a337693b26f2fdf68a013/src/dogecoin.cpp#L41
fn calculate_next_work_required(
    config: &DogecoinConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let retarget_timespan = config.pow_target_timespan;
    let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;

    let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;

    let min_timespan = retarget_timespan - (retarget_timespan / 4);
    let max_timespan = retarget_timespan + (retarget_timespan / 2);

    if modulated_timespan < min_timespan {
        modulated_timespan = min_timespan;
    } else if modulated_timespan > max_timespan {
        modulated_timespan = max_timespan;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(modulated_timespan).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target =
        new_target / U256::from(<i64 as TryInto<u64>>::try_into(retarget_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
