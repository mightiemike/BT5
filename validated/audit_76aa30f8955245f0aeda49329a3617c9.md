### Title
Deterministic Account ID Commits Only to `AccountId` String, Not to Actual Code Bytes — (`core/primitives/src/utils.rs`, `core/primitives-core/src/deterministic_account_id.rs`)

### Summary

`derive_near_deterministic_account_id` hashes the full `DeterministicAccountStateInit`, but when the `code` field is `GlobalContractIdentifier::AccountId(account_id)`, the hash covers only the deployer's account-ID string — not the actual Wasm bytes stored under that identifier. Because `GlobalContractDeployMode::AccountId` explicitly allows the owner to replace the code at any time, the same `DeterministicAccountStateInit` always produces the same account ID yet can deploy entirely different code depending on when initialization occurs. Any user who pre-computes the deterministic account ID and sends funds to it before calling `DeterministicStateInitAction` is exposed to a code-substitution attack by the global-contract owner.

### Finding Description

`derive_near_deterministic_account_id` in `core/primitives/src/utils.rs` computes:

```
account_id = "0s" + keccak256(borsh(DeterministicAccountStateInit))[12:32].hex()
``` [1](#0-0) 

`DeterministicAccountStateInit::V1` carries two fields: `code: GlobalContractIdentifier` and `data: BTreeMap<Vec<u8>, Vec<u8>>`. [2](#0-1) 

`GlobalContractIdentifier` has two variants:

```rust
pub enum GlobalContractIdentifier {
    CodeHash(CryptoHash) = 0,   // 32 immutable bytes
    AccountId(AccountId)  = 1,  // mutable pointer
}
``` [3](#0-2) 

When `CodeHash` is used, the hash input contains the exact 32-byte code digest — the account ID is fully bound to a specific, immutable Wasm binary. When `AccountId` is used, the hash input contains only the UTF-8 account-ID string. The actual Wasm bytes are resolved at initialization time by `use_global_contract`, which performs a live trie lookup:

```rust
let key = TrieKey::GlobalContractCode { identifier: contract_identifier.clone().into() };
if !state_update.contains_key(&key, ...) { ... }
``` [4](#0-3) 

The `AccountId` deploy mode is explicitly designed to be mutable:

> "Contract is deployed under the owner account id. This allows the owner to update the contract for all its users." [5](#0-4) 

The validation gate `validate_deterministic_state_init` only checks that `derived_id == receiver_id`; it does not verify that the code currently stored under the `AccountId` matches any expected hash: [6](#0-5) 

**Attack path (unprivileged, reachable):**

1. Attacker deploys a benign global contract under `AccountId` mode at `attacker.near`.
2. Victim (or a cross-contract factory) pre-computes `derive_near_deterministic_account_id({code: AccountId("attacker.near"), data: {owner: "victim"}})` and sends NEAR to that address via `Transfer` (explicitly supported before `DeterministicStateInitAction`).
3. Attacker calls `DeployGlobalContractAction` with `AccountId` mode again, replacing the code with a malicious Wasm that drains the account balance.
4. Attacker (or anyone) submits `DeterministicStateInitAction` targeting the pre-funded address. The runtime calls `use_global_contract` which resolves the *current* (malicious) code at `attacker.near` and sets `AccountContract::GlobalByAccount("attacker.near")` on the new account.
5. Attacker calls the malicious method; victim's pre-deposited funds are stolen.

The `DeterministicStateInitAction` is idempotent (does nothing if the account already has a contract), so the attacker must race to initialize before the victim does — but the attacker controls the timing by choosing when to update the code.

### Impact Explanation

**High.** Any NEAR balance sent to a pre-computed deterministic account ID that uses `GlobalContractIdentifier::AccountId` is at risk. The invariant stated in the protocol documentation — "deterministic accounts have a fixed code and fixed initial state associated with it" — is violated for the `AccountId` variant. Cross-contract factories that pre-fund child accounts (a common pattern, mirroring the sharded-contract test contract) are directly exposed.

### Likelihood Explanation

**Medium.** The `AccountId` mode is the primary upgrade-friendly deployment path and is actively used in tests and the sharded-contract reference implementation. Any integrator who pre-computes a deterministic account ID and pre-funds it (the documented use-case for `Transfer` before `DeterministicStateInitAction`) is vulnerable to a malicious or compromised global-contract owner. No validator or privileged protocol role is required — any account holder can deploy and update a global contract.

### Recommendation

Restrict `DeterministicAccountStateInit` to `GlobalContractIdentifier::CodeHash` only. The `AccountId` variant introduces a mutable indirection that breaks the determinism guarantee the feature is built on. If mutable global contracts must be supported, the account ID should be derived from the code hash at the time of pre-computation (snapshotted into the `DeterministicAccountStateInit`), not from a live pointer. Concretely:

- In `validate_deterministic_state_init`, reject `GlobalContractIdentifier::AccountId` with a new `ActionsValidationError::DeterministicStateInitMutableCode` variant.
- Or, resolve the current code hash at validation time and embed it in the hash input, making the account ID commit to a specific binary.

### Proof of Concept

```
// state_init_v1 at T=0: code = AccountId("attacker.near"), data = {owner: "victim"}
// derived_id = "0sXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX"

// Step 1: victim sends 100 NEAR to derived_id via Transfer
// Step 2: attacker calls DeployGlobalContractAction(malicious_wasm, AccountId)
//         → TrieKey::GlobalContractCode{AccountId("attacker.near")} now holds malicious_wasm
// Step 3: attacker sends DeterministicStateInitAction(state_init_v1, deposit=0) to derived_id
//         → deploy_deterministic_account calls use_global_contract(AccountId("attacker.near"))
//         → account.set_contract(GlobalByAccount("attacker.near"))  ← malicious code
// Step 4: attacker calls malicious method on derived_id → drains 100 NEAR
```

The exact divergent Borsh bytes: the `DeterministicAccountStateInit` serializes `[0x00, 0x01, <account_id_len_le32>, <account_id_bytes>, 0x00_00_00_00]` (V1 discriminant, AccountId discriminant, empty data map). The hash is identical at T=0 and T=2, but the code resolved at T=2 is the attacker's malicious Wasm — not the benign Wasm present at T=0.

pub fn derive_near_deterministic_account_id(
    state_init: &DeterministicAccountStateInit,
) -> AccountId {
    use sha3::Digest;
    let data = borsh::to_vec(&state_init).expect("borsh must not fail");
    let hash = sha3::Keccak256::digest(&data);
    format!("0s{}", hex::encode(&hash[12..32])).parse().unwrap()
}
