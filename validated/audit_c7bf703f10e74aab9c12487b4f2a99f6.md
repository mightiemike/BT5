### Title
Missing Domain Separator in `computeDigest` Enables Cross-Chain Signature Replay — (`File: core/contracts/Verifier.sol`)

---

### Summary

`Verifier.computeDigest` computes raw EIP-712 struct hashes for all user-signed transaction types without wrapping them in a domain separator. Because no chain ID or verifying contract address is bound into the digest, a valid user signature produced on one chain deployment is cryptographically identical on any other deployment of the same protocol, enabling cross-chain replay of withdrawals, signer-linking, and NLP operations.

---

### Finding Description

`Verifier.computeDigest` is declared `public pure` and produces digests for every user-signed transaction type: `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, `MintNlp`, `BurnNlp`, `LinkSigner`, and `TransferQuote`. [1](#0-0) 

Each branch hashes only the type-string hash and the transaction fields: [2](#0-1) 

No call to `_hashTypedDataV4` is made, and no domain separator (chain ID, contract address, name, version) is included. The function is `pure`, so it cannot read `block.chainid` or `address(this)`.

Contrast this with `OffchainExchange.getDigest`, which correctly constructs a full domain separator: [3](#0-2) 

`OffchainExchange.getDigest` binds the digest to `block.chainid` and a verifying contract address, while `Verifier.computeDigest` does not.

Although `Verifier` inherits `EIP712Upgradeable` and `Endpoint` calls `__EIP712_init("Nado", "0.0.1")`, the domain separator is never applied inside `computeDigest`: [4](#0-3) 

The resulting digest is then passed directly to `validateSignature` or `validateCompactSignature`: [5](#0-4) 

---

### Impact Explanation

A user signature for `WithdrawCollateral` (or any other covered type) produced on chain A is byte-for-byte valid on chain B, provided the nonce field matches the victim's current nonce on chain B. Because nonces start at zero on every fresh deployment, a user's very first signed transaction on any chain is immediately replayable on every other chain where the protocol is deployed and the user has not yet consumed nonce 0.

Concrete asset deltas:
- **WithdrawCollateral / WithdrawCollateralV2**: attacker drains victim's collateral on the target chain.
- **LinkSigner**: attacker installs an attacker-controlled linked signer on the victim's subaccount, gaining permanent signing authority.
- **MintNlp / BurnNlp**: attacker forces unwanted NLP minting or burning against the victim's quote balance.
- **TransferQuote**: attacker replays a quote transfer to redirect funds.

---

### Likelihood Explanation

The Nado protocol targets EVM chains and the README references Ink Chain environments alongside local nodes, indicating multi-chain deployment is an explicit design goal. Nonce alignment is the only barrier: a user whose nonce on chain B equals any nonce they have previously signed on chain A is immediately vulnerable. For a user's first transaction (nonce = 0) on any chain, the replay window is open from the moment the signature is broadcast on chain A. Signed transactions are observable on-chain (emitted in events or visible in sequencer submission calldata), so an attacker has no privileged access requirement.

---

### Recommendation

Replace the raw struct hash in every branch of `computeDigest` with a call to `_hashTypedDataV4`, which prepends the EIP-712 domain separator (including `chainId` and `verifyingContract`):

```solidity
// Before (no domain binding):
digest = keccak256(abi.encode(
    keccak256(bytes(WITHDRAW_COLLATERAL_SIGNATURE)),
    signedTx.tx.sender, ...
));

// After (domain-bound):
digest = _hashTypedDataV4(keccak256(abi.encode(
    keccak256(bytes(WITHDRAW_COLLATERAL_SIGNATURE)),
    signedTx.tx.sender, ...
)));
```

Because `_hashTypedDataV4` reads `block.chainid` and `address(this)`, `computeDigest` must be changed from `pure` to `view`. The `IVerifier` interface and any callers must be updated accordingly. Ensure `Verifier` is initialized with `__EIP712_init` using the same name and version as `Endpoint` and `OffchainExchange`.

---

### Proof of Concept

1. User signs a `WithdrawCollateral` transaction on chain A (e.g., testnet, chain ID 57073) with nonce = 0. The signature `sig_A` is broadcast and visible on-chain.
2. The same user has never interacted with the Nado deployment on chain B (mainnet, chain ID 763374). Their nonce on chain B is also 0.
3. Attacker calls `submitSlowModeTransaction` on chain B, submitting the `WithdrawCollateral` transaction body with `sig_A` attached.
4. `Verifier.computeDigest(WithdrawCollateral, txBody)` produces the identical `bytes32` digest on chain B as on chain A, because chain ID is absent.
5. `validateSignature` recovers the user's address from `sig_A` against this digest and accepts it.
6. The withdrawal executes on chain B, transferring the user's collateral to the attacker-specified address. [6](#0-5) [7](#0-6) [8](#0-7)
