### Title
Linked Signer Can Use Delegated Signing Authority to Modify Its Own Assignment — (`core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction in `EndpointTx.processTransactionImpl()` is validated with `allowLinkedSigner = true`, meaning the current linked signer of a subaccount can sign and submit a `LinkSigner` transaction to overwrite the linked signer assignment. This allows a linked signer — who holds only delegated authority — to use that authority to perpetuate or transfer their own access without the subaccount owner's consent. The slow-mode path correctly restricts this to the owner only, making the asymmetry the root cause.

---

### Finding Description

`linkedSigners` is a mapping from subaccount (`bytes32`) to a delegated signer address. A linked signer is granted the ability to sign certain transactions on behalf of a subaccount owner. [1](#0-0) 

In `processTransactionImpl()`, when a `LinkSigner` transaction is processed via the fast (sequencer) path, `validateSignedTx` is called with the fifth argument hardcoded to `true`: [2](#0-1) 

`allowLinkedSigner = true` causes `validateSignature` to pass the current linked signer address to the verifier: [3](#0-2) 

The verifier then accepts a signature from either the subaccount owner address **or** the linked signer: [4](#0-3) 

This means the current linked signer can craft and submit a `LinkSigner` transaction — signed by themselves — to overwrite `linkedSigners[subaccount]` with any address they choose, including another address they control.

By contrast, the slow-mode `LinkSigner` path correctly uses `validateSender`, which enforces that only the actual owner address (`msg.sender`) can submit the transaction: [5](#0-4) 

The asymmetry between the two paths is the root cause. The fast-mode path has no equivalent ownership check for this sensitive operation.

---

### Impact Explanation

A linked signer can use their delegated signing authority to:

1. **Perpetuate their own access**: Replace themselves with another address they control before the owner's revocation transaction is processed.
2. **Transfer delegated authority**: Assign the linked signer role to any arbitrary address without the owner's knowledge or consent.
3. **Undermine revocation**: Since both the owner and the linked signer share the same nonce counter (`nonces[address(uint160(bytes20(sender)))]`), the linked signer can race the owner's revocation transaction through the sequencer, consuming the nonce and causing the owner's transaction to fail.

This is unauthorized mutation of the `linkedSigners` state — a privilege escalation where a delegated authority can extend or transfer its own delegation.

---

### Likelihood Explanation

Any user who has been granted linked signer access can exploit this immediately. No special preconditions beyond holding a valid linked signer role are required. The attacker submits a signed `LinkSigner` transaction through the normal sequencer API — a fully supported, unprivileged entry point. The attack is realistic and low-cost.

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only the subaccount owner may change the linked signer
);
```

Only the address embedded in `signedTx.tx.sender` (the subaccount owner) should be permitted to modify the linked signer assignment. This mirrors the correct restriction already enforced in the slow-mode path via `validateSender`.

---

### Proof of Concept

1. Alice owns subaccount `alice_sub` and sets Bob as the linked signer via a legitimate `LinkSigner` transaction.
2. Bob (linked signer) crafts a new `LinkSigner` transaction: `sender = alice_sub`, `signer = charlie` (an address Bob controls), signed by Bob's key.
3. Bob submits this transaction through the sequencer's fast-mode path.
4. `validateSignedTx` is called with `allowLinkedSigner = true`; the verifier recovers Bob's address from the signature and accepts it because `recovered == linkedSigner`.
5. `linkedSigners[alice_sub]` is overwritten to Charlie.
6. Bob has transferred the linked signer role to Charlie without Alice's consent.
7. If Alice simultaneously submits a revocation `LinkSigner` transaction (nonce N), Bob can front-run it with his own transaction at nonce N, causing Alice's transaction to fail with `ERR_WRONG_NONCE`.

        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
