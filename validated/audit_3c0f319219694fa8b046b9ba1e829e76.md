### Title
Missing Minimum Confirmation Bound Allows Zero-Confirmation Transaction Verification - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` enforce only an upper bound on the caller-supplied `confirmations` parameter (`<= gc_threshold`) but impose no lower bound. Any unprivileged NEAR caller may pass `confirmations = 0`, causing the contract to return `true` for a transaction in the current chain tip ‚Äî a block with no finality guarantee that may be reorganized away.

### Finding Description

In `verify_transaction_inclusion`, the only guard on `confirmations` is: [1](#0-0) 

No minimum is enforced. The subsequent confirmation-depth check: [2](#0-1) 

evaluates as `(tip_height - target_height + 1) >= 0` when `confirmations = 0`, which is trivially true for all `u64` arithmetic. The function then proceeds to Merkle-proof verification and returns `true` if the proof is valid ‚Äî regardless of whether the block has any confirmations at all.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase-proof check, so it inherits the same flaw: [3](#0-2) 

The `confirmations` field in `ProofArgs` and `ProofArgsV2` is a plain `u64` with no contract-level minimum: [4](#0-3) 

### Impact Explanation

Consumer contracts (bridges, atomic swaps, cross-chain lending protocols) that call `verify_transaction_inclusion` as a finality oracle can be exploited. An attacker submits a Bitcoin transaction, waits for it to appear in the tip block, immediately calls the verification function with `confirmations = 0`, receives `true`, and triggers the consumer contract's settlement logic ‚Äî before the block has any finality. If the Bitcoin chain reorganizes that block away, the consumer contract has already acted on a transaction that no longer exists on the canonical chain.

**Impact: Medium** ‚Äî Downstream consumer contracts that do not independently enforce a minimum confirmation count are vulnerable to accepting unconfirmed (reorg-able) transactions as finalized.

### Likelihood Explanation

**Likelihood: Medium** ‚Äî The entry point is fully unprivileged; any NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with `confirmations = 0`. The attack requires only a valid Merkle proof for a transaction in the current tip block, which is trivially obtainable from any Bitcoin full node. Consumer contracts that delegate confirmation-threshold enforcement to the light client (a reasonable assumption given the parameter exists) are directly at risk.

### Recommendation

Enforce a minimum confirmation count inside the contract itself, rather than relying on callers to supply a safe value. Add a configurable `min_confirmations` field to `BtcLightClient` (set at `init` time) and reject any call where `args.confirmations < self.min_confirmations`. This mirrors the recommendation in the reference report: place the protective bound in the vault/contract itself, not in the caller.

```rust
require!(
    args.confirmations >= self.min_confirmations,
    "Confirmations below the minimum required threshold"
);
```

### Proof of Concept

1. Deploy the contract with any valid genesis and a small `gc_threshold`.
2. Submit one or more block headers so the chain tip is at height `H`.
3. Call `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = hash of the current tip block (height `H`)
   - `confirmations = 0`
   - A valid Merkle proof for any transaction in that block
4. The contract returns `true`.
5. The tip block is subsequently orphaned by a Bitcoin reorg; the consumer contract has already settled based on a transaction that no longer exists on the canonical chain.

The check at [2](#0-1)  reduces to `(H - H + 1) >= 0` ‚Üí `1 >= 0` ‚Üí always passes, confirming the bypass.
