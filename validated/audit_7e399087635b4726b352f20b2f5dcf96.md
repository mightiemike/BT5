### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Validation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) was deprecated in favour of `verify_transaction_inclusion_v2` specifically to close the 64-byte transaction Merkle proof forgery vulnerability. However, v1 is still declared `pub` and is still registered as a callable NEAR contract method. Any unprivileged caller can invoke v1 directly, skipping the coinbase Merkle proof check that v2 enforces, and obtain a `true` verification result for a forged transaction.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to mitigate the well-known 64-byte transaction Merkle proof forgery attack (https://www.bitmex.com/blog/64-Byte-Transactions). It does so by first verifying a coinbase Merkle proof against the block's `merkle_root`, then delegating to v1 for the actual transaction check. [1](#0-0) 

The coinbase check is the only gate that prevents an internal Merkle tree node from being passed as a `tx_id`. The contract's own documentation acknowledges this: [2](#0-1) 

Despite being marked `#[deprecated]`, v1 is still `pub` and still compiled into the contract's WASM ABI as a callable method: [3](#0-2) 

Rust's `#[deprecated]` attribute emits a **compiler warning only**. It does not remove the function from the contract's exported interface. Any NEAR account — with no special role or permission — can call `verify_transaction_inclusion` directly, supplying only `ProofArgs` (which carries no `coinbase_tx_id` or `coinbase_merkle_proof`): [4](#0-3) 

The v2 path that enforces the coinbase check is entirely bypassed.

---

### Impact Explanation

A recipient contract or any on-chain protocol that consumes the verification result — or that was written against the v1 API for backward compatibility — will receive `true` for a transaction that does not exist. The 64-byte forgery technique allows an attacker to craft a `tx_id` that is actually an internal Merkle tree node, construct a valid Merkle path to the block's `merkle_root`, and pass v1's check: [5](#0-4) 

Any downstream protocol (e.g., a bridge, a peg, a payment channel) that calls `verify_transaction_inclusion` and acts on a `true` result can be defrauded: the attacker proves inclusion of a Bitcoin transaction that never existed.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` with a crafted `ProofArgs`. The 64-byte forgery technique is publicly documented and has known tooling. The only precondition is that the attacker knows a valid mainchain block hash stored in the contract, which is public state. Likelihood is **high** for any deployment where a downstream contract calls v1 (e.g., for backward compatibility or because the caller was written before v2 existed).

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract ABI entirely. Because NEAR's `#[near]` macro exports every `pub` method, the function must be made non-`pub` (e.g., `pub(crate)`) or removed. If backward compatibility is required, the method should be replaced with a wrapper that internally enforces the coinbase check before delegating, so there is no callable path that skips it.

---

### Proof of Concept

1. Attacker identifies a valid mainchain block hash `B` stored in the contract (readable via `get_block_hash_by_height`).
2. Using the 64-byte forgery technique, attacker constructs a `tx_id` that is an internal Merkle node of block `B`, along with a Merkle path that hashes to `B`'s `merkle_root`.
3. Attacker calls `verify_transaction_inclusion` directly (not v2) with:
   - `tx_id` = forged internal node hash
   - `tx_block_blockhash` = `B`
   - `tx_index` = crafted index
   - `merkle_proof` = crafted path
   - `confirmations` = any valid value
4. `verify_transaction_inclusion` computes `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof)` and compares it to `header.block_header.merkle_root`. The forged path matches, so the function returns `true`.
5. No coinbase proof is ever checked. The downstream protocol accepts the forged proof as valid. [6](#0-5) [7](#0-6)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
