### Title
Slow-Mode `WithdrawCollateral` Skips Nonce Validation, Enabling Double-Withdrawal via Forced Borrow — (`core/contracts/EndpointTx.sol`)

---

### Summary

The slow-mode `WithdrawCollateral` path in `processSlowModeTransactionImpl` does not consume or validate a nonce. A user can simultaneously queue a slow-mode withdrawal and execute a sequencer-path `SignedWithdrawCollateral` for the same amount. After the 3-day delay, the slow-mode transaction executes against an already-drained balance, creating an unauthorized borrow position and extracting tokens beyond the user's actual deposit.

---

### Finding Description

**Slow-mode path — no nonce consumed:**

In `EndpointTx.sol`, `processSlowModeTransactionImpl` handles `WithdrawCollateral` by decoding the plain `IEndpoint.WithdrawCollateral` struct and calling only `validateSender`, then directly invoking `clearinghouse.withdrawCollateral`. No nonce is read, incremented, or validated. [1](#0-0) 

The `WithdrawCollateral` struct does carry a `nonce` field, but it is entirely ignored in this path. [2](#0-1) 

The developer comment above the function even acknowledges this: `// TODO: these do not need senders or nonces`. [3](#0-2) 

**Sequencer path — nonce IS consumed:**

`processTransactionImpl` for `WithdrawCollateral` calls `validateSignedTx`, which calls `validateNonce`, atomically incrementing `nonces[sender]`. [4](#0-3) 

`validateNonce` enforces strict sequential nonce consumption: [5](#0-4) 

**`clearinghouse.withdrawCollateral` — health check is the only guard:**

The clearinghouse transfers tokens first, then updates the balance, then checks health: [6](#0-5) 

The health check at line 419 is the sole barrier. If the subaccount holds sufficient other collateral (e.g., ETH), health remains ≥ 0 even after the USDC balance goes negative, and the withdrawal succeeds.

**`assertUtilization` is not a meaningful barrier:**

`assertUtilization` checks that protocol-wide `totalDeposits >= totalBorrows`. In any live system with multiple depositors, this passes trivially when one subaccount borrows a small amount. [7](#0-6) 

**Slow-mode execution silently swallows reverts (non-hardhat):**

On production chains, `_executeSlowModeTransaction` wraps execution in a `try/catch`. If the slow-mode tx reverts (e.g., health check fails), it is silently discarded. But if health passes, the withdrawal succeeds and tokens leave the clearinghouse. [8](#0-7) 

---

### Impact Explanation

An attacker with subaccount S holding collateral C (e.g., 100 USDC + 200 ETH):

1. Submits slow-mode `WithdrawCollateral` for 100 USDC — queued, no nonce consumed.
2. Submits sequencer-path `SignedWithdrawCollateral` for 100 USDC — nonce N consumed, S's USDC balance → 0.
3. After 3 days, slow-mode tx executes: clearinghouse transfers another 100 USDC out, S's USDC balance → −100 USDC. Health check passes because ETH covers the borrow.
4. Attacker has received 200 USDC total against a 100 USDC deposit — net extraction of 100 USDC.

The protocol is left holding an unauthorized borrow position for S. If ETH price drops below the maintenance threshold, the protocol absorbs the loss. The clearinghouse's token reserves are depleted by the amount of the unauthorized borrow, at the expense of other depositors.

---

### Likelihood Explanation

- The slow-mode path is publicly callable by any EOA — no special privilege required.
- The attacker only needs to hold mixed collateral (any non-USDC asset with positive initial weight).
- The 3-day delay is a known, predictable constraint that does not prevent the exploit.
- `assertUtilization` is a protocol-wide check that passes in any normally operating system.
- The attack is repeatable across multiple subaccounts or multiple assets.

---

### Recommendation

Consume the nonce in the slow-mode `WithdrawCollateral` path, mirroring the sequencer path. Specifically, in `processSlowModeTransactionImpl`, after `validateSender`, call `validateNonce(txn.sender, txn.nonce)` before invoking `clearinghouse.withdrawCollateral`. This ensures that a slow-mode withdrawal and a sequencer-path withdrawal cannot both execute for the same authorization.

Alternatively, record a per-subaccount "pending slow-mode withdrawal" flag that blocks sequencer-path withdrawals while a slow-mode withdrawal is queued, or snapshot the balance at submission time and reject execution if the balance has since been reduced below the requested amount.

---

### Proof of Concept

```solidity
// Hardhat test (chainId 31337 — reverts are NOT silently swallowed)
// Deploy Endpoint, Clearinghouse, SpotEngine with USDC and ETH products.

// 1. Attacker deposits 100 USDC + 200 ETH into subaccount S.
endpoint.depositCollateral(S, USDC_ID, 100e6);
endpoint.depositCollateral(S, ETH_ID, 200e18);

// 2. Submit slow-mode WithdrawCollateral for 100 USDC (no nonce consumed).
bytes memory slowTx = abi.encodePacked(
    uint8(TransactionType.WithdrawCollateral),
    abi.encode(WithdrawCollateral({sender: S, productId: USDC_ID, amount: 100e6, nonce: 0}))
);
endpoint.submitSlowModeTransaction(slowTx);

// 3. Sequencer processes SignedWithdrawCollateral for 100 USDC (nonce 0 consumed).
//    S's USDC balance → 0.
endpoint.processTransaction(encodeSignedWithdraw(S, USDC_ID, 100e6, nonce=0, sig));

// 4. Advance time by 3 days.
vm.warp(block.timestamp + 3 days + 1);

// 5. Execute slow-mode tx.
//    clearinghouse transfers 100 USDC out again.
//    S's USDC balance → -100 USDC.
//    Health check: ETH collateral covers borrow → passes.
endpoint.executeSlowModeTransaction();

// 6. Assert attacker received 200 USDC total.
assertEq(usdc.balanceOf(attacker), 200e6); // double withdrawal confirmed
// S's on-chain USDC balance is now -100 USDC (unauthorized borrow).
assertEq(spotEngine.getBalance(USDC_ID, S).amount, -100e18 /* normalized */);
```
