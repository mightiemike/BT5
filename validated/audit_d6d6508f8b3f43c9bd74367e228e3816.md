### Title
Router-Mediated Swap Bypasses `SwapAllowlistExtension` When Router Is Allowlisted — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[pool][sender]`. The pool passes `msg.sender` (the direct caller of `pool.swap()`) as `sender`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` on the pool call, so the extension sees `sender = router`. If the pool admin has allowlisted the router address, every non-allowlisted EOA can execute swaps through the router, completely defeating the per-user gate.

### Finding Description

**Call chain:**

```
EOA (non-allowlisted)
  → MetricOmmSimpleRouter.exactInputSingle()          [msg.sender = EOA]
      → IMetricOmmPoolActions(pool).swap(...)          [msg.sender = router]
          → MetricOmmPool._beforeSwap(msg.sender=router, ...)
              → ExtensionCalling._callExtensionsInOrder(...)
                  → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                      check: allowedSwapper[pool][router]  ← TRUE → passes
```

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router: [3](#0-2) 

In `exactInputSingle`, the router calls `pool.swap()` directly — the original EOA identity is never forwarded: [4](#0-3) 

A pool admin who wants to permit router-mediated swaps for their allowlisted users will naturally call `setAllowedToSwap(pool, routerAddress, true)`. Once that entry exists, `allowedSwapper[pool][router] == true` and the `beforeSwap` check passes for **every** caller of `exactInputSingle`, regardless of whether that EOA is individually allowlisted.

### Impact Explanation

Any non-allowlisted EOA can drain tokens from a pool that was configured to restrict swappers. The pool admin's intent — "only these addresses may swap" — is silently nullified. Token outflow is unrestricted for the full liquidity depth of the pool, constituting direct loss of LP principal.

### Likelihood Explanation

Allowlisting the official router is the natural, non-malicious configuration for a pool that wants to support router-mediated swaps for its permitted users. There is no warning in the contract or interface that doing so opens the gate to all callers. The misconfiguration is easy to make and hard to detect without auditing the transitive call chain.

### Recommendation

The router should forward the originating EOA as the `sender` to the pool (e.g., via a dedicated parameter or a trusted-forwarder pattern), or `SwapAllowlistExtension.beforeSwap` should reject calls where `sender` is a known router/contract and require the original EOA to be passed explicitly in `extensionData`. Alternatively, document clearly that allowlisting any contract address (including the router) grants blanket access to all callers of that contract.
