### Title
Missing Cross-Binding Between `inner` and `metadata` in `ChunkEndorsementV2` Enables Endorsement Suppression via Signature Splicing - (`core/primitives/src/stateless_validation/chunk_endorsement.rs`, `chain/client/src/stateless_validation/chunk_endorsement.rs`)

---

### Summary

`ChunkEndorsementV2` uses two independently-signed fields — `inner` (containing `chunk_hash`) and `metadata` (containing `shard_id`, `epoch_id`, `height_created`) — with no cryptographic cross-binding between them. An unprivileged network observer can splice the `inner`+`signature` from validator V's endorsement of chunk C1 with the `metadata`+`metadata_signature` from V's endorsement of chunk C2 to produce a structurally valid cross-bound endorsement. This passes `validate_chunk_endorsement`, is stored in the cache under C2's `ChunkProductionKey` with C1's `chunk_hash`, blocks V's legitimate C2 endorsement via the deduplication guard, and is then filtered out by `collect_chunk_endorsements` (wrong `chunk_hash`), effectively erasing V's endorsement for C2.

---

### Finding Description

**The two-signature scheme and its missing cross-binding**

`ChunkEndorsementV2` is defined as:

```rust
pub struct ChunkEndorsementV2 {
    inner: ChunkEndorsementInnerV1,       // contains chunk_hash
    signature: Signature,                 // signs inner only
    metadata: ChunkEndorsementMetadata,   // contains shard_id, epoch_id, height_created
    metadata_signature: Signature,        // signs metadata only
}
``` [1](#0-0) 

The `verify` method checks both signatures independently against the same public key, with no field that binds `inner.chunk_hash` to `metadata.{shard_id, epoch_id, height_created}`:

```rust
fn verify(&self, public_key: &PublicKey) -> bool {
    let inner = borsh::to_vec(&self.inner).unwrap();
    let metadata = borsh::to_vec(&self.metadata).unwrap();
    self.signature.verify(&inner, public_key)
        && self.metadata_signature.verify(&metadata, public_key)
}
``` [2](#0-1) 

`ChunkEndorsementInnerV1` contains only `chunk_hash` and a static `signature_differentiator` string `"ChunkEndorsement"` — no epoch, shard, or height fields: [3](#0-2) 

**The splice attack**

An unprivileged network observer collects two legitimately-produced endorsements from the same validator V:
- Endorsement for C1: `(inner_C1, sig_C1, metadata_C1, meta_sig_C1)`
- Endorsement for C2: `(inner_C2, sig_C2, metadata_C2, meta_sig_C2)`

They construct a cross-bound message: `ChunkEndorsementV2 { inner: inner_C1, signature: sig_C1, metadata: metadata_C2, metadata_signature: meta_sig_C2 }`. No key material is required — only recombination of observed wire messages.

**Why it passes `validate_chunk_endorsement`**

`validate_chunk_endorsement` calls `validate_chunk_relevant_as_validator` using `endorsement.chunk_production_key()`, which reads from `metadata` (C2's key), and then calls `validate_chunk_endorsement_signature` which calls `endorsement.verify(public_key)` — the independent two-sig check above. Both checks pass: [4](#0-3) 

**Cache storage and deduplication**

`process_chunk_endorsement` stores the cross-bound endorsement under C2's `ChunkProductionKey` (from metadata) with C1's `chunk_hash` (from inner): [5](#0-4) 

The deduplication guard then rejects V's legitimate C2 endorsement when it arrives: [6](#0-5) 

**Filtering in `collect_chunk_endorsements`**

When the block producer calls `collect_chunk_endorsements` for C2's header, the stored entry is filtered out because `chunk_hash_C1 ≠ chunk_header.chunk_hash()` for C2: [7](#0-6) 

V's endorsement is absent from the result, as if V never endorsed C2.

### Impact Explanation

An attacker who can observe P2P endorsement messages and send messages to a block producer node can selectively suppress individual validators' endorsements for targeted chunks. If enough validators' endorsements are suppressed, the chunk fails to reach the endorsement threshold, causing a liveness failure for that chunk. The attack is targeted (specific validator × specific chunk) and repeatable across blocks.

---

### Likelihood Explanation

Chunk endorsements are broadcast over the P2P network and are observable by any network participant. Validators are assigned as chunk validators across all shards in an epoch, so the same validator V will routinely produce endorsements for multiple chunks (different shards, same or adjacent heights) within the same epoch, giving the attacker the two required endorsements. The attacker must win a network race to deliver the cross-bound message before the legitimate one, but since they observe the legitimate message in transit, they can immediately construct and forward the splice. The race window is narrow but repeatable.

---

### Recommendation

Add a cryptographic cross-binding between `inner` and `metadata`. The simplest fix is to include the `metadata` fields (or a hash of them) inside `ChunkEndorsementInnerV1`, so that a single signature covers both `chunk_hash` and `{shard_id, epoch_id, height_created, account_id}`. Alternatively, sign a single combined struct that contains all fields. The `metadata_signature` field can then be removed entirely, collapsing the two-signature scheme into one.

//   endorsement_C1: V's endorsement for chunk C1 (shard 0, height 10)
//   endorsement_C2: V's endorsement for chunk C2 (shard 1, height 10)
//
// Both are ChunkEndorsement::V2(_) and are broadcast in plaintext.

// Splice: take inner+signature from C1, metadata+metadata_signature from C2
let ChunkEndorsement::V2(v2_c1) = endorsement_c1;
let ChunkEndorsement::V2(v2_c2) = endorsement_c2;

let spliced = ChunkEndorsement::V2(ChunkEndorsementV2 {
    inner: v2_c1.inner,               // chunk_hash = C1's hash
    signature: v2_c1.signature,       // valid sig over C1's inner
    metadata: v2_c2.metadata,         // shard_id/epoch_id/height = C2's key

// Send spliced to block producer before V's legitimate C2 endorsement arrives.
// validate_chunk_endorsement passes (both sigs valid for V's pubkey).
// Stored under C2's ChunkProductionKey with C1's chunk_hash.
// V's real C2 endorsement is rejected as duplicate.
// collect_chunk_endorsements for C2 filters out the spliced entry (wrong chunk_hash).
// V's endorsement is missing from C2's endorsement set.
