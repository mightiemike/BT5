### Title
Missing `validate_proof_size` call in `StatelessTransactionValidator::validate()` allows oversized client-side proofs to bypass gateway admission — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate()` defines a `validate_proof_size` helper that enforces `max_proof_size` on invoke transactions carrying client-side proofs, but **never calls it**. When `allow_client_side_proving` is enabled, any peer can submit an invoke transaction whose `proof` field is arbitrarily large; the gateway accepts it, propagates it over P2P, and forwards it to the proof manager — all without enforcing the configured size bound.

---

### Finding Description

`StatelessTransactionValidator::validate()` dispatches a fixed set of sub-validators for invoke transactions:

```rust
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;
    self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
    // validate_proof_size is NOT called here
}
``` [1](#0-0) 

The missing call is to `validate_proof_size`, which is fully implemented in the same file:

```rust
fn validate_proof_size(
    &self,
    tx: &RpcInvokeTransaction,
) -> StatelessTransactionValidatorResult<()> {
    let RpcInvokeTransaction::V3(tx) = tx;
    let proof_size = tx.proof.0.len();
    if proof_size > self.config.max_proof_size {
        return Err(StatelessTransactionValidatorError::ProofTooLarge {
            proof_size,
            max_proof_size: self.config.max_proof_size,
        });
    }
    Ok(())
}
``` [2](#0-1) 

The corresponding error variant and config field exist and are wired up correctly — the only missing piece is the call site in `validate()`. [3](#0-2) 

The `allow_client_side_proving` guard (`validate_client_side_proving_allowed`) correctly rejects proof-bearing transactions when the feature is disabled, so the missing check only matters when `allow_client_side_proving = true`. In that configuration, `validate_proof_size` is the sole enforcement point for `max_proof_size`, and it is never reached. [4](#0-3) 

---

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

When `allow_client_side_proving` is enabled, an unprivileged external caller can submit an `RpcInvokeTransaction::V3` whose `proof` field contains an arbitrarily large byte vector. The gateway accepts it, the `MempoolP2pPropagator` serializes and broadcasts the full proof over the P2P network, and the `TransactionConverter` forwards it to the proof manager for verification. There is no upper bound enforced anywhere in this path. The operator-configured `max_proof_size` is silently ignored. [5](#0-4) 

---

### Likelihood Explanation

Likelihood is **medium**. The `allow_client_side_proving` flag must be set to `true` in the gateway config (it is `false` by default). Once enabled, any external caller submitting to the gateway's `add_transaction` endpoint can trigger the issue with a single crafted transaction — no privileged access, no special account, no prior state required.

---

### Recommendation

Add the missing call to `validate_proof_size` inside the `RpcTransaction::Invoke` branch of `validate()`, immediately after `validate_proof_facts_and_proof_consistency`:

```rust
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;
    self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
    self.validate_proof_size(invoke_tx)?;   // <-- add this
}
``` [6](#0-5) 

---

### Proof of Concept

1. Deploy a gateway node with `allow_client_side_proving = true` and `max_proof_size = 1000` (or any finite value).
2. Construct an `RpcInvokeTransactionV3` with `proof.0` containing 100 000 `Felt` elements (well above `max_proof_size`) and valid `proof_facts`.
3. POST the transaction to the gateway's `add_transaction` endpoint.
4. Observe: the gateway returns a success response (transaction hash), the oversized proof is propagated to peers via `MempoolP2pPropagator`, and the proof manager receives a verification request for the 100 000-element proof — all without any size-limit error.

The `ProofTooLarge` error variant is never emitted because `validate_proof_size` is never called. [7](#0-6)

    fn validate_client_side_proving_allowed(
        &self,
        tx: &RpcInvokeTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if self.config.allow_client_side_proving {
            return Ok(());
        }

        // Reject V3 transactions with proofs when client-side proving is disabled.
        let RpcInvokeTransaction::V3(tx) = tx;
        let has_proof_data = !tx.proof_facts.is_empty() || !tx.proof.is_empty();
        if has_proof_data {
            return Err(StatelessTransactionValidatorError::ClientSideProvingNotAllowed);
        }

        Ok(())
    }
