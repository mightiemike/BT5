### Title
`MintNlp` Lacks Minimum NLP Return Check — Oracle Price Is Sequencer-Supplied and Unsigned - (File: `core/contracts/Clearinghouse.sol`, `core/contracts/Verifier.sol`)

---

### Summary

The `mintNlp` flow in Nado allows a user to commit to spending a fixed `quoteAmount` of collateral, but the number of NLP tokens they receive is determined entirely by an `oraclePriceX18` value that is **supplied by the sequencer at execution time and is not part of the user's signed digest**. There is no `minNlpAmount` field in the user-signed `MintNlp` struct, and no on-chain check that the minted NLP amount meets any user-defined floor. A user can receive arbitrarily fewer NLP tokens than they expected for their quote spend, with no recourse.

---

### Finding Description

The user signs a `MintNlp` transaction containing only three fields:

```
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint64 nonce;
}
``` [1](#0-0) 

The EIP-712 digest computed in `Verifier.computeDigest` covers only `sender`, `quoteAmount`, and `nonce`:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(MINT_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.quoteAmount,
        signedTx.tx.nonce
    )
);
``` [2](#0-1) 

The `oraclePriceX18` lives in `SignedMintNlp` **outside** the inner `tx` struct and is therefore **not committed to by the user's signature**:

```
struct SignedMintNlp {
    MintNlp tx;
    bytes signature;
    int128 oraclePriceX18;       // ← not signed
    int128[] nlpPoolRebalanceX18; // ← not signed
}
``` [3](#0-2) 

In `EndpointTx`, the sequencer decodes the full `SignedMintNlp` and passes the unsigned `oraclePriceX18` directly to `clearinghouse.mintNlp`:

```solidity
clearinghouse.mintNlp(
    signedTx.tx,
    signedTx.oraclePriceX18,   // sequencer-controlled
    nlpPools,
    signedTx.nlpPoolRebalanceX18
);
``` [4](#0-3) 

Inside `Clearinghouse.mintNlp`, the NLP amount minted is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [5](#0-4) 

There is no check that `nlpAmount` meets any minimum the user intended. The entire output is a function of the sequencer-supplied price.

---

### Impact Explanation

A user who signs `MintNlp { quoteAmount: 1000e18 }` expecting to receive NLP tokens at a fair price of, say, `1.00` per NLP (i.e., 1000 NLP) will receive only 500 NLP if the sequencer submits `oraclePriceX18 = 2e18`. The user's full `quoteAmount` is debited regardless:

```solidity
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -quoteAmount);
``` [6](#0-5) 

The user loses the full quote amount but receives a fraction of the NLP they expected, with no on-chain protection. The same structural flaw exists in `burnNlp`: the user signs `nlpAmount` to burn but has no `minQuoteAmount` check — the quote returned is `nlpAmount * oraclePriceX18 - burnFee`, entirely at the sequencer's discretion. [7](#0-6) 

---

### Likelihood Explanation

The sequencer does not need to be malicious. The NLP oracle price can legitimately move between the moment a user signs their `MintNlp` transaction and the moment the sequencer processes it. Because the protocol is off-chain-matched, there is no mempool ordering guarantee from the user's perspective — the sequencer batches and submits transactions, and the price used is whatever the sequencer includes in the batch at that time. Even under honest sequencer behavior, a user who signed during a low-price window can have their transaction executed at a significantly higher price, receiving far fewer NLP tokens than expected. The absence of any user-controlled slippage parameter makes this unavoidable.

---

### Recommendation

1. Add a `minNlpAmount` field to the `MintNlp` struct and include it in the EIP-712 digest so the user commits to a minimum acceptable output.
2. In `Clearinghouse.mintNlp`, assert `nlpAmount >= txn.minNlpAmount` after computing the minted amount.
3. Symmetrically, add a `minQuoteAmount` field to `BurnNlp` and assert the net quote returned meets it.

---

### Proof of Concept

1. User signs `MintNlp { sender: alice, quoteAmount: 1000e18, nonce: 0 }` when NLP oracle price is `1e18` (1 USDC per NLP), expecting to receive 1000 NLP.
2. Before the sequencer processes the transaction, the NLP oracle price rises to `2e18`.
3. Sequencer submits the batch with `oraclePriceX18 = 2e18`.
4. `Clearinghouse.mintNlp` computes `nlpAmount = 1000e18 / 2e18 = 500`.
5. Alice's quote balance is debited by `1000e18` but she receives only `500` NLP — half of what she expected.
6. No on-chain check prevents this outcome. Alice had no way to express a minimum acceptable NLP amount in her signed transaction. [8](#0-7)
