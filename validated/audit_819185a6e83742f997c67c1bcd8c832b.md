### Title
`From<ReservedDataAvailabilityMode>` Unconditionally Returns `L1`, Silently Discarding Any L2 DA Mode During Feeder-Gateway Sync — (`crates/apollo_starknet_client/src/reader/objects/transaction.rs`)

### Summary
The `From<ReservedDataAvailabilityMode>` implementation in the feeder-gateway reader uses a wildcard `_` pattern and unconditionally returns `DataAvailabilityMode::L1`, discarding the actual value. This is the direct sequencer analog of `receiveFlashLoan` ignoring `feeAmounts`: a parameter is received but silently replaced with a hardcoded constant. The `ReservedDataAvailabilityMode` enum already defines `L2 = 1`, so the deserializer can produce it, but the conversion throws it away. If L2 DA mode is ever activated on Starknet, every transaction synced from the feeder gateway with `nonce_data_availability_mode = 1` or `fee_data_availability_mode = 1` would be stored with the wrong DA mode, causing the sequencer to compute a different transaction hash than the feeder gateway, breaking hash canonicalization and state consistency.

### Finding Description
In `crates/apollo_starknet_client/src/reader/objects/transaction.rs`, the `ReservedDataAvailabilityMode` enum has two variants:

```rust
pub enum ReservedDataAvailabilityMode {
    Reserved = 0,
    L2 = 1,
}
```

The `From` implementation ignores the actual variant with a wildcard:

```rust
impl From<ReservedDataAvailabilityMode> for starknet_api::data_availability::DataAvailabilityMode {
    // TODO(Arni): Fix this. Support L2 data availability mode.
    fn from(_: ReservedDataAvailabilityMode) -> Self {
        starknet_api::data_availability::DataAvailabilityMode::L1
    }
}
```

This conversion is called via `.into()` in both `TryFrom<IntermediateDeclareTransaction>` and `TryFrom<IntermediateDeployAccountTransaction>` (and the equivalent invoke path) when syncing blocks from the feeder gateway:

```rust
nonce_data_availability_mode: deploy_account_tx
    .nonce_data_availability_mode
    .ok_or(...)?
    .into(),   // <-- calls the broken From impl
fee_data_availability_mode: deploy_account_tx
    .fee_data_availability_mode
    .ok_or(...)?
    .into(),   // <-- calls the broken From impl
```

The DA mode is a direct input to the transaction hash via `concat_data_availability_mode`:

```rust
fn concat_data_availability_mode(
    nonce_mode: &DataAvailabilityMode,
    fee_mode: &DataAvailabilityMode,
) -> Felt {
    (data_availability_mode_index(fee_mode)
        + (data_availability_mode_index(nonce_mode) << DATA_AVAILABILITY_MODE_BITS))
        .into()
}
```

and is chained into the Poseidon hash in `get_invoke_transaction_v3_hash` (and the declare/deploy-account equivalents):

```rust
.chain(&data_availability_mode)
```

A transaction with `nonce_data_availability_mode = L2` (integer 1) produces a different `data_availability_modes` felt than one with `L1` (integer 0). The feeder gateway computes and stores the hash using the real value; the sequencer recomputes it using the silently-substituted `L1`, producing a divergent hash.

The gateway's stateless validator currently rejects any transaction whose DA mode is not `L1`:

```rust
fn validate_nonce_data_availability_mode(...) {
    let expected_da_mode = DataAvailabilityMode::L1;
    if da_mode != expected_da_mode {
        return Err(...);
    }
}
```

This gate prevents new transactions from entering the mempool with L2 DA mode, but it does not protect the **central-sync / feeder-gateway reader path**, which bypasses the gateway entirely and calls the broken `From` conversion directly.

### Impact Explanation
If Starknet governance activates L2 DA mode (the enum variant already exists and the TODO comment explicitly anticipates this), the central-sync path would silently corrupt every synced transaction that uses it:

- The stored `nonce_data_availability_mode` / `fee_data_availability_mode` fields would be `L1` instead of `L2`.
- Any subsequent hash recomputation (e.g., `validate_transaction_hash`, block-hash derivation, or re-execution) would produce a hash that diverges from the canonical feeder-gateway hash.
- This constitutes **wrong state and wrong hash** for accepted input — matching the "High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload" impact criterion.

### Likelihood Explanation
Currently low: L2 DA mode is not live on Starknet, and the gateway rejects any transaction that carries it. The risk is identical in structure to the Balancer flash-loan fee issue — the ignored parameter is zero today, but the code path that would make it non-zero already exists (`L2 = 1` in the enum, the TODO comment, and the gateway's own `DataAvailabilityMode::L2` variant). The moment L2 DA mode is activated, the bug becomes immediately reachable through the central-sync path without any privileged action beyond submitting a valid L2-DA-mode transaction to the feeder gateway.

### Recommendation
Replace the wildcard `_` with a proper match so the actual variant is preserved:

```rust
impl From<ReservedDataAvailabilityMode> for starknet_api::data_availability::DataAvailabilityMode {
    fn from(mode: ReservedDataAvailabilityMode) -> Self {
        match mode {
            ReservedDataAvailabilityMode::Reserved => DataAvailabilityMode::L1,
            ReservedDataAvailabilityMode::L2 => DataAvailabilityMode::L2,
        }
    }
}
```

Additionally, remove the `// TODO(Arni): Fix this.` comment and the corresponding TODO in the enum definition once the fix is applied, and add a round-trip test that verifies `ReservedDataAvailabilityMode::L2` converts to `DataAvailabilityMode::L2` and that the resulting transaction hash matches the feeder-gateway-computed hash.

### Proof of Concept

1. Starknet governance activates L2 DA mode.
2. A user submits an Invoke V3 transaction with `nonce_data_availability_mode = 1` (`L2`) to the feeder gateway.
3. The feeder gateway computes and stores `tx_hash_FGW = poseidon(... || concat_da_modes(L2, L1) || ...)`.
4. The sequencer's central-sync path reads the block, deserializes `nonce_data_availability_mode = 1` as `ReservedDataAvailabilityMode::L2`, then calls `.into()`.
5. The broken `From` impl returns `DataAvailabilityMode::L1` regardless of the `L2` input.
6. The sequencer stores the transaction with `nonce_data_availability_mode = L1`.
7. Any hash recomputation produces `tx_hash_SEQ = poseidon(... || concat_da_modes(L1, L1) || ...)`.
8. `tx_hash_SEQ ≠ tx_hash_FGW` — hash validation fails, or the sequencer silently stores a transaction under the wrong hash, corrupting state. [1](#0-0) [2](#0-1) [3](#0-2)
