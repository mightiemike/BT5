### Title
Unpermissioned `creditDeposit()` Allows Any Caller to Force DDA Token Deposits and Corrupt Slow-Mode Queue State — (`File: core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` is an `external` function with no access control modifier. Any unprivileged caller can invoke it to force the Direct Deposit Address (DDA) to sweep all of its token balances into the Endpoint's slow-mode queue, targeting the DDA's fixed `subaccount`, without the DDA owner's consent.

---

### Finding Description

`DirectDepositV1` is a contract that acts as a deposit relay: tokens sent to it are meant to be credited to a fixed `subaccount` via `creditDeposit()`. The `withdraw()` and `withdrawNative()` functions are correctly gated with `onlyOwner`, but `creditDeposit()` carries no such restriction. [1](#0-0) 

When called, `creditDeposit()` iterates over every product ID registered in `spotEngine`, reads the DDA's balance for each token, approves the Endpoint, and calls `endpoint.depositCollateralWithReferral()` for each non-zero balance. [1](#0-0) 

`depositCollateralWithReferral()` in `Endpoint.sol` transfers the tokens from the DDA into the Endpoint contract and appends a new `SlowModeTx` entry to `slowModeTxs`, incrementing `slowModeConfig.txCount`. [2](#0-1) 

The slow-mode queue is a shared, ordered data structure consumed by the sequencer. Unauthorized entries corrupt its state. [3](#0-2) 

---

### Impact Explanation

- Any caller can force the DDA to sweep its entire token balance into the slow-mode queue at any time, removing the DDA owner's control over deposit timing.
- Each forced call inserts one `SlowModeTx` entry per product with a non-zero balance into `slowModeTxs`, incrementing `slowModeConfig.txCount` without the owner's authorization — this is a concrete queue-state corruption.
- The DDA owner cannot batch or time deposits strategically; an attacker can front-run any intended deposit window.
- The referral code is hardcoded to `"-1"` regardless of the owner's preference (though the parameter is currently ignored in the Endpoint implementation).
- Impact class: griefing — no profit motive for the attacker, but the DDA owner loses operational control and the slow-mode queue accumulates unauthorized entries.

---

### Likelihood Explanation

- The entry path is a single direct call to `creditDeposit()` on any deployed `DirectDepositV1` instance — no special role, signature, or precondition is required.
- The DDA is a production contract deployed per-subaccount; any deployed instance is permanently exposed.
- The attack is trivially repeatable whenever the DDA holds a non-zero token balance.

---

### Recommendation

Add an `onlyOwner` modifier to `creditDeposit()`, consistent with the access control already applied to `withdraw()` and `withdrawNative()`:

```solidity
function creditDeposit() external onlyOwner {
    ...
}
``` [4](#0-3) 

---

### Proof of Concept

```solidity
// Attacker has no special role.
// Assume DDA holds 100 USDC for productId=1.
DirectDepositV1 dda = DirectDepositV1(ddaAddress);

// Any EOA can call this:
dda.creditDeposit();
// Result:
// - 100 USDC transferred from DDA to Endpoint
// - slowModeTxs[txCount] populated with a DepositCollateral entry for the DDA's subaccount
// - slowModeConfig.txCount incremented
// - DDA owner had no opportunity to batch, time, or prevent this deposit
``` [1](#0-0) [5](#0-4)

    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });
        slowModeConfig = _slowModeConfig;
    }
