### Title
Caller-Controlled Zero-Confirmation Verification Enables Block Reorg Double Spending — (File: `contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a fully caller-controlled `confirmations` field with **no enforced minimum**, including `0`. Because the confirmation arithmetic uses `saturating_sub`, passing `confirmations = 0` always satisfies the check, and `confirmations = 1` passes for any block that is currently the chain tip. A relayer-submitted reorg that replaces the verified block's position in the canonical chain invalidates the proof result after the fact, enabling a double-spend against any downstream contract (e.g., Fast Bridge) that gates fund release on the returned `bool`.

---

### Finding Description

In `verify_transaction_inclusion` the only upper-bound guard is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

There is no lower-bound guard. The confirmation depth check that follows is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

When `args.confirmations = 0`, the left-hand side evaluates to at least `1` (because `saturating_sub` floors at `0` and `+1` is always applied), so `1 >= 0` is unconditionally true. When `args.confirmations = 1`, any block that is currently on the main chain — including the very tip — satisfies the check.

`verify_transaction_inclusion_v2`, the production-recommended path, delegates directly to the deprecated function after the coinbase proof check, inheriting the same flaw:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

Neither function is access-controlled beyond the `#[pause]` gate; any unprivileged NEAR account can call them. [4](#0-3) [5](#0-4) 

The contract's reorg mechanism is correct by design: when a fork accumulates more chain work than the current tip, `reorg_chain` is triggered and the canonical mapping (`mainchain_height_to_header`, `mainchain_header_to_height`) is rewritten to reflect the new best chain:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [6](#0-5) 

After a reorg, the previously canonical block is removed from `mainchain_header_to_height` and `headers_pool` via `remove_block_header`. A subsequent call to `verify_transaction_inclusion` for that block would panic. But the damage is already done: the earlier call already returned `true` and any fund release it gated has already executed. [7](#0-6) 

---

### Impact Explanation

A downstream contract (Fast Bridge or any SPV consumer) that calls `verify_transaction_inclusion_v2` with `confirmations = 1` will receive `true` the moment the target block appears at the chain tip. If it releases funds on that result, an attacker who subsequently submits a valid competing chain (one block of equal or greater work) causes the contract to reorg, removing the original block from the canonical chain. The attacker retains both the released funds and the original Bitcoin UTXO — a classic double spend. The corrupted state is the `mainchain_header_to_height` canonical mapping, which is rewritten by `reorg_chain` to no longer contain the verified block.

---

### Likelihood Explanation

- **Bitcoin mainnet**: A 1-block reorg requires briefly outpacing the network, which is feasible for a miner with ~10–20% hash rate over a short window, or via selfish-mining strategies. Historically, 1-block reorgs occur organically several times per year.
- **Litecoin / Dogecoin**: Smaller total hash rate makes deliberate 1-block reorgs significantly cheaper and more realistic for a motivated attacker.
- The `confirmations = 0` path requires zero mining effort at all — the block need not even be buried.

---

### Recommendation

1. **Enforce a contract-level minimum**: Add a constant (e.g., `MIN_CONFIRMATIONS: u64 = 6` for Bitcoin, higher for merge-mined chains) and `require!(args.confirmations >= MIN_CONFIRMATIONS, ...)` at the top of `verify_transaction_inclusion`.
2. **Reject `confirmations = 0` explicitly**: The current arithmetic silently accepts it; it should be an explicit error.
3. **Document the safe floor per chain**: Since the contract is multi-chain, the minimum should be configurable per `Network` variant stored in contract state.

---

### Proof of Concept

1. Relayer submits block `B` (height `H`) to the light client; it becomes the chain tip. `mainchain_header_to_height[B] = H`.
2. Attacker (or any NEAR account) calls `verify_transaction_inclusion_v2` with `tx_block_blockhash = B`, `confirmations = 1`. The check `(H - H) + 1 = 1 >= 1` passes. Function returns `true`.
3. Fast Bridge observes `true`, releases bridged funds to the attacker.
4. Attacker (who controls sufficient hash power) mines an alternative block `B'` at height `H` with higher or equal chain work, and submits it via `submit_blocks`.
5. `submit_block_header_inner` detects `B'.chain_work > tip.chain_work`, calls `reorg_chain`. `remove_block_header(B)` is called: `B` is evicted from `mainchain_header_to_height` and `headers_pool`.
6. The canonical chain no longer contains the transaction. The attacker double-spends: funds were released on step 3, and the Bitcoin UTXO is unspent on the reorged chain.

Relevant state transition in `reorg_chain`:

```rust
if let Some(current_main_chain_blockhash) = main_chain_block {
    self.remove_block_header(&current_main_chain_blockhash); // B evicted here
}
``` [8](#0-7)
