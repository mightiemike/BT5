Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address as Swapper, Enabling Complete Allowlist Bypass for Router-Mediated Swaps — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender`. When users swap through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. If a pool admin allowlists the router (a natural action to enable router usage), every address on the network can bypass the per-user allowlist by routing through the router.

## Finding Description

**Step 1 — Pool passes its own `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

`_beforeSwap` in `ExtensionCalling.sol` forwards that value unchanged as the `sender` argument to every configured extension: [2](#0-1) 

**Step 2 — Router calls `pool.swap` without exposing the actual user.**

`MetricOmmSimpleRouter.exactInputSingle` stores the actual user (`msg.sender`) only in transient storage for the payment callback via `_setNextCallbackContext`, and never passes it to the pool: [3](#0-2) 

So when the pool invokes the extension, `sender = router address`.

**Step 3 — Extension checks the router address, not the actual user.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [4](#0-3) 

The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. Individual user identities are invisible to the extension.

**Existing guards are insufficient.** There is no mechanism in the pool, extension calling path, or `SwapAllowlistExtension` itself to recover the originating user from transient storage or any other source. The `extensionData` field is caller-supplied and unauthenticated, so it cannot be trusted to carry user identity.

## Impact Explanation

When a pool admin allowlists the router to enable standard periphery usage, `allowedSwapper[pool][router] = true`. Any unprivileged EOA can then call `router.exactInputSingle` targeting that pool and pass the extension check, because the extension sees the router address — which is allowlisted — not the caller. The per-user allowlist is completely bypassed. This matches the "admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" impact gate: a non-allowlisted user gains access to a curated pool (e.g., restricted to institutional counterparties or KYC-verified addresses) through a routine, non-malicious admin action.

## Likelihood Explanation

The trigger is a pool admin calling `setAllowedToSwap(pool, router, true)` — a routine, expected action for any operator who wants their pool usable through the standard periphery. The admin has no indication this opens the pool to all callers. The attacker requires no special privilege: any EOA can call `router.exactInputSingle` with the target pool address. The attack is repeatable and requires no front-running or timing.

## Recommendation

The extension must check the identity of the economic actor, not the intermediary contract. Two approaches:

1. **Pass the originating user through the pool.** Add an `originator` field to the swap call or encode it in `extensionData` in a tamper-evident way (e.g., signed by the router). The extension then checks `originator` instead of `sender`.
2. **Maintain a trusted-router registry in the extension.** When `sender` is a known router, revert unconditionally (since the real user cannot be identified). This forces curated pools to be accessed only via direct pool calls, until a proper originator-forwarding mechanism is in place.

## Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, alice, true)
   and setAllowedToSwap(pool, bob, true).
3. Pool admin calls setAllowedToSwap(pool, router, true)
   so that alice and bob can use the standard router.
4. charlie (not allowlisted) calls:
     router.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(recipient, ...) → pool's msg.sender = router.
6. Pool calls _beforeSwap(router, ...) → extension.beforeSwap(router, ...).
7. Extension checks allowedSwapper[pool][router] → true.
8. charlie's swap executes on the curated pool.
9. The allowlist is completely bypassed.
```
