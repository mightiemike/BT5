### Title
SwapAllowlistExtension Checks Router Address Instead of Originating EOA, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, that `sender` is the **router address**, not the originating EOA. If the router is allowlisted, any non-allowlisted EOA can execute swaps by calling the router, completely defeating the allowlist.

### Finding Description

The call chain is:

```
attacker (non-allowlisted EOA)
  → MetricOmmSimpleRouter.exactInputSingle()
      → pool.swap(recipient, ...)          // msg.sender = router
          → _beforeSwap(msg.sender=router, ...)
              → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                  → allowedSwapper[pool][router] == true  ✓  (passes!)
```

In `MetricOmmPool.swap()`, the pool passes its own `msg.sender` (the router) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

In `SwapAllowlistExtension.beforeSwap`, the guard checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router — not the originating EOA: [2](#0-1) 

Since `allowedSwapper[pool][router] = true`, the check passes for every user who calls through the router, regardless of whether the EOA is allowlisted.

### Impact Explanation

The swap allowlist is the primary curation mechanism for restricting who may trade in a pool. A pool admin who allowlists only the router address (a common and expected configuration) inadvertently grants swap access to every EOA on the network. Non-allowlisted users can execute real swaps, moving pool reserves and extracting output tokens. This is a direct, fund-impacting bypass of a core access-control feature.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the canonical swap entry point for the protocol. Any pool that uses `SwapAllowlistExtension` and allowlists the router (the natural configuration) is immediately vulnerable to any EOA. No special privileges, malicious setup, or non-standard tokens are required — only a public call to `exactInputSingle`. [3](#0-2) 

### Recommendation

The extension must check the **originating actor**, not the immediate pool caller. Two options:

1. **Pass the original payer through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a convention between router and extension.
2. **Check `recipient` or use `tx.origin` as a last resort**: Neither is clean. The correct fix is to redesign the `sender` semantics so the pool passes the true initiator, or to have the router encode the real payer in `extensionData` and have the extension verify it.
