### Title
TWAP Order Overfill via Stale `taker.order.amount` in `matchOrders` — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`_validateOrder` subtracts `filledAmounts[digest]` from a **local copy** of `order.amount` only. The caller's `taker.order.amount` is never updated. When `matchOrders` subsequently bounds `txn.takerAmountDelta` against `taker.order.amount`, it compares against the **original signed amount**, not the remaining unfilled amount. A sequencer can therefore submit a `takerAmountDelta` equal to the full original amount on a partially-filled TWAP order, causing `filledAmounts[digest]` to accumulate beyond the signer's authorized total.

---

### Finding Description

**`_validateOrder` — local-copy-only adjustment (lines 424, 435–436):**

`_validateOrder` creates a memory copy of `signedOrder.order` and subtracts `filledAmounts[orderDigest]` from that copy's `amount` field. [1](#0-0) 

This mutation is **never written back** to `signedOrder.order.amount`. After `_validateOrder` returns, `taker.order.amount` in `matchOrders` still holds the original signed value `A`.

**`matchOrders` — TWAP cap check uses stale original amount (lines 709–714):**

```
if (taker.order.amount > 0) {
    require(
        taker.order.amount >= txn.takerAmountDelta &&   // ← A >= delta, not (A-F) >= delta
            maker.order.amount <= -txn.takerAmountDelta,
        ERR_INVALID_TAKER
    );
}
``` [2](#0-1) 

With `filledAmounts[digest] = F` and original amount `A`, the remaining authorized amount is `A - F`. The check permits `txn.takerAmountDelta` up to `A`, not `A - F`.

**`matchOrders` — amount override and fill accumulation (lines 723, 832–834):**

```
taker.order.amount = txn.takerAmountDelta;   // set to A
...
filledAmounts[ordersInfo.taker.digest] += ordersInfo.taker.amountDelta;  // += A → total = F + A
``` [3](#0-2) [4](#0-3) 

After execution: `filledAmounts[digest] = F + A > A`, violating the invariant that total filled ≤ signed amount.

---

### Impact Explanation

The taker's account receives `A` units of base (and corresponding quote) in a single fill, even though only `A - F` remained authorized. The excess `F` units of base and their quote equivalent are credited without any corresponding user authorization. This constitutes unauthorized balance mutation beyond the signer's intent. If the taker's account is leveraged, the inflated position can create bad debt for the protocol when the position is liquidated.

---

### Likelihood Explanation

Exploitable by the sequencer (the entity that submits `MatchOrdersWithSigner` transactions via the endpoint) acting maliciously. No external key compromise is required — the sequencer simply submits a `takerAmountDelta` equal to the original order amount on any partially-filled TWAP order. The precondition (a partially-filled TWAP order) is a normal protocol state that arises in routine TWAP execution.

---

### Recommendation

In `matchOrders`, before the TWAP cap check, subtract `filledAmounts[ordersInfo.taker.digest]` from `taker.order.amount` to obtain the true remaining amount:

```solidity
if (txn.takerAmountDelta != 0) {
    require(_isTWAP(taker.order.appendix), ERR_INVALID_TAKER);
    int128 takerRemaining = taker.order.amount
        - filledAmounts[ordersInfo.taker.digest];   // ← add this
    require(
        (txn.takerAmountDelta > 0) == (takerRemaining > 0),
        ERR_INVALID_TAKER
    );
    if (takerRemaining > 0) {
        require(
            takerRemaining >= txn.takerAmountDelta &&
                maker.order.amount <= -txn.takerAmountDelta,
            ERR_INVALID_TAKER
        );
    } else { ... }
    taker.order.amount = txn.takerAmountDelta;
    ...
}
```

---

### Proof of Concept

1. User signs a TWAP taker order: `amount = 100`, `appendix` has trigger bits ≥ 2.
2. Sequencer submits a first `MatchOrdersWithAmount` with `takerAmountDelta = 60`. After execution: `filledAmounts[digest] = 60`.
3. `_validateOrder` is called again: local `order.amount = 100 - 60 = 40 ≠ 0` → returns `true`. `taker.order.amount` in `matchOrders` remains `100`.
4. Sequencer submits a second `MatchOrdersWithAmount` with `takerAmountDelta = 100`. Check at line 711: `100 >= 100` → passes.
5. `taker.order.amount = 100`; `ordersInfo.taker.amountDelta = 100`; `_updateBalances` credits 100 units.
6. `filledAmounts[digest] += 100` → `filledAmounts[digest] = 160 > 100`.

Assert `filledAmounts[digest] <= 100` fails. The taker received 160 units of base against a 100-unit signed order. [5](#0-4)
