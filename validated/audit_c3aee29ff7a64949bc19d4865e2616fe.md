### Title
v8 Migration Halts Chain Upgrade When Vesting-Owned Position Is Delegated to a Jailed Validator - (`File: x/tieredrewards/keeper/force_exit.go`, `x/tieredrewards/migrations/v2/migrate.go`)

---

### Summary

The v8 upgrade migration unconditionally calls `ForceFullExitWithDelegation()` for every vesting-owned tier position. That function internally calls `transferDelegationFromPosition()`, which hard-rejects non-bonded validators with `ErrValidatorNotBonded`. If any vesting-owned position is delegated to a jailed or unbonding validator at the upgrade block, the error propagates all the way up through `Migrate1to2` → `RunMigrations` → the upgrade handler, causing the entire chain upgrade to fail and the chain to halt.

---

### Finding Description

The v8 upgrade handler calls `app.ModuleManager.RunMigrations(...)`, which triggers `Migrate1to2` for the `tieredrewards` module. [1](#0-0) 

`Migrate1to2` delegates to `v2.Migrate`, which calls `exitVestedAccountsPositions`. That function iterates all positions, collects those owned by vesting accounts, and calls `pk.ForceFullExitWithDelegation(ctx, posID)` for each one. [2](#0-1) 

`ForceFullExitWithDelegation` checks whether the position is delegated. If it is, it proceeds to call `transferDelegationFromPosition`: [3](#0-2) 

`transferDelegationFromPosition` contains a hard bonded-validator guard: [4](#0-3) 

If the validator is jailed (status = Unbonding or Unbonded), this returns `ErrValidatorNotBonded`. The error is wrapped and returned at every level:

- `ForceFullExitWithDelegation` → `fmt.Errorf("transfer delegation back to owner for position %d: %w", ...)`
- `exitVestedAccountsPositions` → `fmt.Errorf("force-exit position %d: %w", ...)`
- `Migrate` → `fmt.Errorf("exit vested accounts positions: %w", ...)`
- `Migrate1to2` → returned to `RunMigrations`
- `RunMigrations` → returned to the upgrade handler
- Upgrade handler → returns `(nil, err)`, causing the chain to halt at the upgrade height

The ADR-006 edge-case table explicitly documents that `ExitTierWithDelegation` fails when the validator is jailed/unbonding, but the migration path has no equivalent fallback. [5](#0-4) 

---

### Impact Explanation

**Corrupted invariant**: The chain upgrade state machine. If the upgrade handler returns an error, the chain halts at the upgrade height and cannot produce new blocks until the binary is patched and the upgrade re-attempted.

**Concrete corrupted value**: The `RunMigrations` return value becomes a non-nil error, which BaseApp treats as a fatal upgrade failure.

---

### Likelihood Explanation

Validators are jailed routinely for downtime (missing blocks) or double-signing. The v8 migration was specifically designed to handle vesting-owned positions that existed before the vesting-account block was added. Any such position delegated to a validator that happens to be jailed at the upgrade block triggers the halt. The window is the entire period between the upgrade proposal passing and the upgrade height firing — during which any validator can be jailed. This is a realistic, non-negligible probability.

---

### Recommendation

In `ForceFullExitWithDelegation`, detect the non-bonded validator case before calling `transferDelegationFromPosition` and fall back to a safe alternative:

1. **Preferred**: If the validator is not bonded, call `stakingKeeper.Unbond` directly (without re-delegating to the owner) to start the standard unbonding period. The owner can later withdraw via normal staking unbonding. Delete the position record.
2. **Minimal**: Skip the delegation transfer entirely and only delete the position record, leaving the delegation at the position's delegator address. The owner can recover it via standard staking unbonding after the upgrade.

Either approach prevents the migration from returning an error for a jailed-validator position, eliminating the chain-halt risk.

---

### Proof of Concept

1. Before the v8 upgrade height, a vesting account (`PermanentLockedAccount`) holds a tier position delegated to validator `V`.
2. Validator `V` is jailed for downtime at block `N`, transitioning to `Unbonding` status.
3. The v8 upgrade fires at block `N+k`.
4. `exitVestedAccountsPositions` walks positions, finds the vesting-owned position, and calls `ForceFullExitWithDelegation`.
5. `ForceFullExitWithDelegation` calls `transferDelegationFromPosition` with the jailed validator's address.
6. `transferDelegationFromPosition` fetches the validator, finds `!validator.IsBonded()`, and returns `ErrValidatorNotBonded`.
7. The error propagates through `Migrate1to2` → `RunMigrations` → upgrade handler.
8. The upgrade handler returns `(nil, err)`.
9. The chain halts at the upgrade height. [6](#0-5) [7](#0-6) [2](#0-1)
