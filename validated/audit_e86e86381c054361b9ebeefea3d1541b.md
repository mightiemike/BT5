### Title
`ContractOwner` Lacks `receive()` Function, Permanently Bricking Native ETH Recovery from `DirectDepositV1` — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner` has no `receive()` function. Its own `withdrawFromDirectDepositV1` function attempts to pull native ETH out of a `DirectDepositV1` contract by calling `DirectDepositV1.withdrawNative()`, which sends ETH back to `msg.sender` (i.e., `ContractOwner`). Because `ContractOwner` cannot receive ETH, the call reverts unconditionally, permanently locking any native ETH that accumulates in a `DirectDepositV1` contract.

---

### Finding Description

`DirectDepositV1.withdrawNative()` sends the contract's full ETH balance to `msg.sender` via a low-level call with a hard `require`:

```solidity
// core/contracts/DirectDepositV1.sol lines 108–112
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
```

The owner of every `DirectDepositV1` is `ContractOwner`, because `DirectDepositV1` inherits OpenZeppelin `Ownable` and is deployed via `new DirectDepositV1{salt: ...}(...)` inside `ContractOwner.createDirectDepositV1()`, making `ContractOwner` the `msg.sender` at construction time and therefore the `owner`. [1](#0-0) 

The intended recovery path is `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`:

```solidity
// core/contracts/ContractOwner.sol lines 628–636
uint256 preBalance = address(this).balance;
DirectDepositV1(directDepositV1).withdrawNative();   // ← sends ETH to ContractOwner
uint256 postBalance = address(this).balance;
require(postBalance > preBalance, "empty");
(bool success, ) = msg.sender.call{value: postBalance - preBalance}("");
require(success, "xfer");
``` [2](#0-1) 

When `withdrawNative()` executes, `msg.sender` is `ContractOwner`. The `call{value: balance}("")` targets `ContractOwner`, which has no `receive()` or `fallback()` function anywhere in its implementation. The call returns `success = false`, causing `withdrawNative()` to revert with `"Failed to transfer native token to owner"`, which bubbles up and reverts `withdrawFromDirectDepositV1` entirely. [3](#0-2) 

**How ETH accumulates in `DirectDepositV1`:**

1. **Constructor path**: If `wrappedNative.call{value: balance}("")` fails at deployment time (e.g., the wrapped-native contract is not yet live or is paused), the constructor emits `NativeTokenTransferFailed` and continues without reverting, leaving the pre-existing ETH balance permanently in the contract. [4](#0-3) 

2. **Forceful ETH send**: ETH sent via `selfdestruct` bypasses the `receive()` function entirely, depositing ETH directly into `DirectDepositV1`'s balance without triggering any wrapping logic.

In both cases, the only designed recovery mechanism — `withdrawFromDirectDepositV1(..., address(0))` — is permanently broken.

---

### Impact Explanation

Any native ETH that accumulates in a `DirectDepositV1` contract (via the constructor's silent failure path or a forceful `selfdestruct` send) is permanently unrecoverable. The `withdrawFromDirectDepositV1` function, which is the sole on-chain path to extract native ETH from these contracts, will always revert because `ContractOwner` cannot receive ETH. The asset delta is the full ETH balance of the affected `DirectDepositV1` contract, permanently locked.

---

### Likelihood Explanation

The constructor's silent-failure path (`NativeTokenTransferFailed` event + continue) is a deliberate design choice to avoid bricking the deployment if wrapping fails. This means the scenario is explicitly anticipated by the developers. Any deployment where `wrappedNative` is temporarily non-functional or misconfigured at the moment `createDirectDepositV1` is called will silently strand ETH. Additionally, any party can forcefully send ETH to a `DirectDepositV1` address via `selfdestruct`. The `DirectDepositV1` address is deterministic (uses `salt: bytes32(uint256(1))`), making it predictable and targetable. [5](#0-4) [6](#0-5) 

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

This mirrors the fix applied to the analogous `ForwarderBase` and `WithdrawalHelperBase` contracts in the referenced upstream report. The ETH received by `ContractOwner` is immediately forwarded to `msg.sender` on line 633, so no ETH will be permanently retained by `ContractOwner`. [7](#0-6) 

---

### Proof of Concept

1. Deploy the system normally. `ContractOwner` creates a `DirectDepositV1` for a subaccount via `createDirectDepositV1`.
2. Force ETH into the `DirectDepositV1` contract via `selfdestruct` (or observe the constructor's `NativeTokenTransferFailed` event if `wrappedNative` was non-functional at deploy time).
3. Call `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` as the owner (multisig).
4. Execution flow: `ContractOwner` → `DirectDepositV1.withdrawNative()` → `ContractOwner.call{value: balance}("")` → **reverts** (`success = false`) → `require(success, "Failed to transfer native token to owner")` fires → entire call reverts.
5. ETH remains permanently locked in `DirectDepositV1` with no on-chain recovery path. [8](#0-7) [3](#0-2)
