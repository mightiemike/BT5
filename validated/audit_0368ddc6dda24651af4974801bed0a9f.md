### Title
Attacker Can Force Liquidations by Inflating Spot Utilization Rate to Spike Borrow Multiplier ‚ÄĒ (`core/contracts/SpotEngineState.sol`)

---

### Summary

An unprivileged attacker can borrow a large amount of a spot asset to push `utilizationRatioX18` toward 100%, causing the kinked interest rate model to apply a dramatically elevated `borrowRateMultiplierX18`. When the sequencer calls `updateStates`, the inflated multiplier is permanently compounded into `cumulativeBorrowsMultiplierX18`. Every existing borrower's actual debt ‚ÄĒ computed as `amountNormalized √ó cumulativeBorrowsMultiplierX18` ‚ÄĒ grows proportionally, reducing their health and potentially pushing them below the maintenance threshold. The attacker can then liquidate those accounts at a discount.

---

### Finding Description

**Interest rate model ‚ÄĒ `_updateState`**

`SpotEngineState._updateState` computes the annualised borrow rate from the current utilisation ratio: [1](#0-0) 

At the inflection point (80 % utilisation) the rate is `interestFloorX18 + interestSmallCapX18 = 1 % + 4 % = 5 %` annualised. Above the inflection point the large-cap term kicks in: [2](#0-1) 

With the default QUOTE product configuration (`interestLargeCapX18 = ONE = 100 %`), at 100 % utilisation the annualised rate reaches **105 %**: [3](#0-2) 

This rate is converted to a per-second multiplier and compounded over `dt` seconds: [4](#0-3) 

The result is permanently written into `cumulativeBorrowsMultiplierX18`: [5](#0-4) 

**`updateStates` allows up to 7 days of accrual in one call** [6](#0-5) 

At 100 % utilisation over the maximum `dt = 7 √ó 86 400 = 604 800` seconds, the borrow multiplier increases by ‚Čą **1.98 %** in a single transaction.

**Health calculation uses the actual (denormalised) balance**

`BaseEngine._calculateProductHealth` converts the stored normalised balance back to an actual amount before computing health: [7](#0-6) 

`_getBalance` for the spot engine returns `amountNormalized √ó cumulativeBorrowsMultiplierX18` (negative for borrowers): [8](#0-7) 

As `cumulativeBorrowsMultiplierX18` grows, every borrower's `amount` becomes more negative, directly reducing `health += amount.mul(weight).mul(risk.priceX18)`.

**No per-transaction utilisation cap**

`assertUtilization` only enforces `totalDeposits >= totalBorrows` (‚Č§ 100 % utilisation) and is only called on withdrawals, not on borrows: [9](#0-8) 

There is no mechanism that prevents a single actor from pushing utilisation to 99 %+ in one transaction.

---

### Impact Explanation

After the attacker's large borrow is processed and `updateStates(604800)` is called, every existing borrower's debt increases by up to ‚Čą 2 %. Any account whose maintenance health margin was smaller than that delta becomes eligible for liquidation. The attacker liquidates those accounts at the protocol's liquidation discount (typically 5‚Äď10 %), netting the spread over their own interest cost. The corrupted state is the `cumulativeBorrowsMultiplierX18` for the targeted product, which is permanently elevated and cannot be reversed.

---

### Likelihood Explanation

- The attacker needs sufficient collateral to borrow enough to push utilisation near 100 %; this is capital-intensive but not impossible.
- The attacker must hold the position for up to 7 days (the maximum `dt`), paying elevated interest on their own borrow ‚ÄĒ but the liquidation discount exceeds this cost.
- Target accounts must be within ‚Čą 2 % of the maintenance threshold; in a mature market with leveraged borrowers this is a realistic condition.
- The sequencer controls `updateStates` timing, which is a partial mitigant, but the sequencer is expected to call it with the true elapsed time; refusing to do so would itself be a liveness failure.

Overall likelihood: **low-medium** ‚ÄĒ requires capital and patience, but the conditions are plausible in a live market.

---

### Recommendation

1. **Utilisation cap on borrows**: Reject new borrows that would push utilisation above a configurable ceiling (e.g., 95 %) to prevent a single actor from reaching the steep part of the rate curve.
2. **Time-weighted utilisation**: Compute the interest rate from a time-weighted average utilisation rather than the instantaneous snapshot, analogous to Notional's `timeWindow` fix. This prevents a single large borrow from immediately spiking the rate applied to all existing borrowers.
3. **Per-product borrow limits**: Enforce a maximum single-subaccount borrow fraction of total deposits to limit the leverage any one actor can apply to the utilisation ratio.

---

### Proof of Concept

1. Attacker deposits $10 M USDC as collateral into their subaccount.
2. Attacker submits a borrow transaction for $9.9 M USDC (the maximum allowed by `assertUtilization`), pushing `utilizationRatioX18` from, say, 50 % to ‚Čą 99 %.
3. The sequencer processes the borrow; `totalBorrowsNormalized` spikes.
4. Seven days later the sequencer calls `updateStates(604800)`.
5. `_updateState` computes `borrowerRateX18 ‚Čą 1.05 / 31536000` per second; `borrowRateMultiplierX18 = (1 + 3.33e-8)^604800 ‚Čą 1.0198`.
6. `cumulativeBorrowsMultiplierX18` is multiplied by 1.0198 ‚ÄĒ a permanent 1.98 % increase.
7. All existing borrowers' actual debt amounts grow by 1.98 %; any account whose maintenance health was within that margin is now below zero.
8. Attacker submits `LiquidateSubaccount` transactions against those accounts, receiving their collateral at a 5‚Äď10 % discount and profiting net of their own interest cost. [10](#0-9) [6](#0-5) [7](#0-6)

    function updateStates(uint128 dt) external onlyEndpoint {
        State memory quoteState;
        require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
        for (uint32 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            if (productId == NLP_PRODUCT_ID) {
                continue;
            }
            State memory state = states[productId];
            if (productId == QUOTE_PRODUCT_ID) {
                quoteState = state;
            }
            if (state.totalDepositsNormalized == 0) {
                continue;
            }
            _updateState(productId, state, dt);
            _setState(productId, state);
        }
    }
