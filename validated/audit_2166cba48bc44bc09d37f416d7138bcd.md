Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, allowing any unprivileged caller to bypass the swap allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension checks whether the **router** is allowlisted rather than the actual end user. Any unprivileged user can bypass a curated pool's swap allowlist by calling the public router, completely defeating the KYC/access-control purpose of the extension.

## Finding Description
The call chain is fully confirmed in production code:

1. `MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — here `msg.sender` to the pool is the **router contract address**. [1](#0-0) 

2. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, passing the router as `sender`. [2](#0-1) 

3. `ExtensionCalling._beforeSwap` encodes `sender` (the router) and dispatches to the configured extension. [3](#0-2) 

4. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` — i.e., `allowedSwapper[pool][router]` — not `allowedSwapper[pool][actual_user]`. [4](#0-3) 

**Bypass path:** A pool admin allowlists individual KYC'd addresses and also allowlists the router (the necessary step to allow those users to trade via the standard periphery). Any non-KYC'd user then calls `MetricOmmSimpleRouter.exactInputSingle`; the extension checks `allowedSwapper[pool][router] == true` and passes, executing the swap against LP assets despite the user not being allowlisted.

**Inverse breakage:** If the admin does *not* allowlist the router, then even individually-allowlisted users cannot trade through the router, breaking the expected user flow. There is no mechanism in the hook interface to recover the original caller's identity — the router passes an empty string `""` as `callbackData` and the user-supplied `params.extensionData` as `extensionData`, but the extension does not decode any identity from `extensionData`. [5](#0-4) 

The root cause is at line 37 of `SwapAllowlistExtension.sol`: `sender` is the direct caller of `pool.swap()` (the router), not the originating user. [6](#0-5) 

## Impact Explanation
Any user can bypass a curated pool's swap allowlist by routing through `MetricOmmSimpleRouter`. This defeats the core purpose of `SwapAllowlistExtension`: gating which addresses may trade against the pool. Pools using this extension for KYC, compliance, or access-control purposes are fully open to any public user via the router. Unauthorized parties execute swaps against LP assets — a direct admin-boundary break with fund-impacting consequences, meeting the contest's "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" criterion.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entry point. Any pool that deploys `SwapAllowlistExtension` and expects users to interact via the router (the normal periphery path) is immediately affected. The attacker requires no special privileges — only the ability to call the public router. The precondition (router being allowlisted) is the natural and expected admin action when deploying this extension for router-mediated pools.

## Recommendation
The extension must gate the economically relevant actor, not the intermediary. Two options:

1. **Forward real user identity via `extensionData`:** Modify `MetricOmmSimpleRouter` to encode the original `msg.sender` in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when the direct `sender` is a known trusted router.

2. **Trusted-forwarder model:** Make the extension aware of trusted forwarders (routers). When `sender` is a known router, require the real user identity to be ABI-encoded in `extensionData` and check that address against `allowedSwapper[pool][realUser]`.

The simplest safe fix is to require the router to always forward the real user identity in `extensionData`, and have the extension consume it when the direct caller is a recognized intermediary.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured as beforeSwap hook.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (to allow router-mediated swaps for allowlisted users).
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  - attacker (not allowlisted) calls:
      MetricOmmSimpleRouter.exactInputSingle({
          pool: pool,
          ...
      })
  - Router calls pool.swap(...) with msg.sender = router.
  - Pool calls _beforeSwap(router, ...).
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true → passes.
  - Attacker's swap executes against LP assets despite not being allowlisted.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
```
