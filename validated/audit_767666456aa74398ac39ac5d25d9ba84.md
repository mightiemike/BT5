Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original Trader — Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist, but `MetricOmmPool.swap` passes `msg.sender` as `sender` — which is the router contract address, not the original end-user, whenever a swap is routed through `MetricOmmSimpleRouter`. If a pool admin allowlists the router (the natural action to enable router-based swaps), every unprivileged trader can bypass the per-user allowlist by calling the router instead of the pool directly.

## Finding Description

**Root cause — wrong identity checked:**

`MetricOmmPool.swap` calls the extension hook with its own `msg.sender` as the `sender` argument:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // ← always the immediate caller of pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that value against the allowlist:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

**Router path — sender is always the router:**

Every public swap entry point in `MetricOmmSimpleRouter` calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol line 72-80 (exactInputSingle)
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ...
);
// msg.sender inside pool.swap() == address(router)
```

The same holds for `exactInput` (line 104), `exactOutputSingle` (line 136), and every recursive hop in `exactOutput` / `_exactOutputIterateCallback` (line 220). In all cases the pool sees `msg.sender == router`, so `sender` forwarded to `beforeSwap` is the router address, never the original trader.

**Exploit flow:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to KYC'd addresses.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` so that users can reach the pool through the standard router (the only practical UX path).
3. Any unprivileged trader calls `router.exactInputSingle(...)` — the pool's `beforeSwap` receives `sender = router`, finds the router on the allowlist, and allows the swap.
4. The per-user allowlist is completely bypassed; the pool behaves as if `setAllowAllSwappers` were set.

Alternatively, if the pool admin does *not* allowlist the router, legitimately allowlisted users are blocked from using the router at all — broken core swap functionality.

**Existing guards are insufficient:**

The `onlyPool` modifier on `beforeSwap` only verifies that the caller is a valid pool; it does not recover the original trader identity. There is no mechanism in the current call path to propagate the original `msg.sender` from the router through the pool to the extension.

## Impact Explanation
An unprivileged trader can execute swaps in a pool that is intended to be restricted to allowlisted addresses, by routing through `MetricOmmSimpleRouter`. This constitutes an admin-boundary break: the pool admin's access-control configuration is rendered ineffective by an unprivileged path. Pools designed for permissioned trading (e.g., RWA pools, KYC-gated pools) are exposed to unrestricted swap access, which can cause unauthorized asset flows and undermine LP risk assumptions.

## Likelihood Explanation
The router is the primary and expected user-facing entry point. Any pool admin who enables router-based swaps (the normal operational choice) must allowlist the router, at which point the bypass is trivially reachable by any address with no special capability. The condition is realistic and repeatable.

## Recommendation
Pass the original caller identity through the call chain. One approach: add an `originator` field to the extension hook arguments that the pool populates from a trusted transient-storage slot written by the router before calling `pool.swap()`. The extension should then check `allowedSwapper[pool][originator]` rather than `allowedSwapper[pool][sender]`. Alternatively, document that `SwapAllowlistExtension` is incompatible with the router and enforce this at the factory level.
