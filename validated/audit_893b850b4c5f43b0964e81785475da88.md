Audit Report

## Title
SwapAllowlistExtension checks router address instead of originating user, allowing full allowlist bypass via MetricOmmSimpleRouter ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` calls `pool.swap()`, `msg.sender` is the router contract, not the originating user. A pool admin who allowlists the router to support router-mediated swaps for curated users inadvertently grants every user the ability to bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`.

## Finding Description
**Root cause ŌĆö three confirmed code locations:**

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` value verbatim to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` ŌĆö i.e., `allowedSwapper[pool][router]` ŌĆö when the router is the caller: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` stores the real user (`msg.sender`) only in transient storage for the payment callback, and calls `pool.swap()` directly ŌĆö so the pool never sees the originating user: [4](#0-3) 

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension` to restrict swaps to KYC-verified addresses.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` ŌĆö the natural action to support the standard periphery.
3. Attacker (not individually allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` with `msg.sender = router`.
5. Pool calls `_beforeSwap(sender=router, ...)`.
6. Extension checks `allowedSwapper[pool][router] == true` ŌåÆ passes.
7. Swap executes against LP funds without the attacker being individually allowlisted.

**Why existing guards are insufficient:** The extension has no mechanism to distinguish individual users behind the router. The `extensionData` field passed by the router (`params.extensionData`) is user-controlled and not authenticated, so the extension cannot safely decode a real user from it without a trusted-forwarder pattern.

## Impact Explanation
A pool deploying `SwapAllowlistExtension` intends to restrict swaps to a curated set of addresses (e.g., KYC-verified counterparties or institutional LPs). Once the router is allowlisted, the per-user allowlist is fully bypassed for all router-mediated swaps. Unauthorized users can trade against LP funds at oracle-derived prices. This constitutes broken core pool functionality with direct exposure of LP assets to unintended counterparties ŌĆö matching the "Broken core pool functionality causing loss of funds" allowed impact.

## Likelihood Explanation
Allowlisting the router is the natural and expected action for any pool admin who wants their curated pool usable through the standard periphery (`MetricOmmSimpleRouter` is the documented swap entry point). The restriction that allowlisting the router opens the pool to all users is non-obvious and undocumented. The bypass is reachable on any curated pool that supports router-mediated swaps, which is the common deployment pattern. No special attacker capability is required beyond calling the public router.

## Recommendation
The extension must gate the originating user, not the immediate caller of `pool.swap()`. The most robust fix is a trusted-forwarder pattern: add a `trustedForwarder` mapping to `SwapAllowlistExtension`. When `sender` is a registered trusted forwarder (e.g., the router), decode the real user from `extensionData` and apply the allowlist check against that address. The router must be updated to encode `msg.sender` into `extensionData` before calling the pool. Since the router is a known periphery contract, trusting it to supply the correct value is acceptable.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls: swapExtension.setAllowedToSwap(pool, router, true)
    // natural action to support router-mediated swaps for allowlisted users

Attack:
  attacker (not in allowedSwapper[pool]) calls:
    MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      recipient: attacker,
      ...
    })

  Router executes:
    _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, msg.sender, tokenIn);
    pool.swap(attacker, zeroForOne, amount, limit, "", extensionData)
      // msg.sender = router

  Pool executes:
    _beforeSwap(sender=router, ...)

  SwapAllowlistExtension.beforeSwap:
    allowedSwapper[pool][router] == true  ŌåÉ passes
    swap executes against LP funds

Result:
  Attacker swaps on a curated pool without being individually allowlisted.
  Per-user allowlist is fully bypassed for all router-mediated swaps.
```

Foundry test outline: deploy pool with `SwapAllowlistExtension`, allowlist only the router (not the attacker), call `exactInputSingle` from the attacker address, assert the swap succeeds and `allowedSwapper[pool][attacker]` is `false`.
