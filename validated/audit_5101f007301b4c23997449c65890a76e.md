### Title
Linked Signer Can Override Its Own `linkedSigners` Mapping Entry to Permanently Hijack Subaccount Control — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-path `LinkSigner` transaction is validated with `allowLinkedSigner = true`, meaning the currently registered linked signer can sign a new `LinkSigner` transaction and unconditionally overwrite `linkedSigners[sender]` with any attacker-controlled address. This permanently hijacks the subaccount without the wallet owner's knowledge or consent.

---

### Finding Description

In `EndpointTx.sol`, the fast-path `LinkSigner` handler (lines 576–590) calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true                   // ← linked signer is accepted as valid signer
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
```

`validateSignedTx` calls `validateSignature`, which passes `getLinkedSigner(sender)` to the verifier:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
```

The verifier accepts a signature from either the subaccount owner **or** the linked signer. Because `allowLinkedSigner = true`, the existing linked signer `S` can produce a valid signature for a `LinkSigner` transaction that sets `linkedSigners[sender] = attacker`. The mapping is then written unconditionally with no existence check.

By contrast, the slow-path `LinkSigner` handler (lines 232–239) calls `validateSender`, which enforces that only the actual wallet address (`address(uint160(bytes20(txn.sender)))`) can submit the transaction — correctly preventing the linked signer from changing itself:

```solidity
validateSender(txn.sender, sender);   // wallet address must match msg.sender
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

The asymmetry between the two paths is the root cause: the slow path correctly restricts `LinkSigner` to the wallet owner, but the fast path does not.

---

### Impact Explanation

Once the linked signer overwrites `linkedSigners[sender]` with an attacker address `A`, `A` becomes the authorized signer for the subaccount. `A` can then:

- Sign `WithdrawCollateral` / `WithdrawCollateralV2` transactions to drain all collateral.
- Sign `TransferQuote` transactions to move funds to attacker-controlled subaccounts.
- Sign further `LinkSigner` transactions to lock out the original owner permanently.

The original wallet owner cannot recover control through the fast path (their nonce has advanced), and must rely on the slow-mode path — but by then the collateral may already be drained. The corrupted state is `linkedSigners[victimSubaccount]`, which directly controls who can authorize withdrawals and trades.

---

### Likelihood Explanation

Linked signers are the standard mechanism for automated trading bots, API keys, and keeper integrations in Nado. Any of the following realistic scenarios triggers the exploit:

- A trading bot's private key is leaked (e.g., via a compromised server or environment variable).
- A malicious third-party integration is granted linked signer status.
- A linked signer contract has a bug that allows an attacker to produce signatures.

The attacker only needs to know the victim's current nonce (publicly readable from `nonces[address]`) and submit one valid signed transaction to the sequencer. No admin access, governance capture, or social engineering is required.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the fast-path `LinkSigner` handler so that only the subaccount owner's key can authorize a linked signer change:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← only the wallet owner may change the linked signer
);
```

This aligns the fast path with the slow path's security model.

---

### Proof of Concept

1. UserA deposits collateral and sets linked signer to address `S` via the slow-mode `LinkSigner` path (requires UserA's wallet signature — correct).
2. `S` (the linked signer) constructs a fast-path `LinkSigner` transaction: `sender = UserA_subaccount`, `signer = attacker_address_A`, `nonce = current_nonce`.
3. `S` signs the EIP-712 digest of this transaction.
4. The sequencer includes the transaction and calls `processTransactionImpl`.
5. `validateSignedTx(..., true)` is called; the verifier recovers `S` from the signature and accepts it because `getLinkedSigner(UserA_subaccount) == S`. [1](#0-0) 
6. `linkedSigners[UserA_subaccount] = A` is written unconditionally. [2](#0-1) 
7. `A` signs a `WithdrawCollateral` transaction; `getLinkedSigner(UserA_subaccount)` now returns `A`; the withdrawal is authorized and UserA's collateral is drained. [3](#0-2) 

The slow-mode path that correctly restricts this operation to the wallet owner is at: [4](#0-3) 

The `linkedSigners` storage slot that is unconditionally overwritten: [5](#0-4)
