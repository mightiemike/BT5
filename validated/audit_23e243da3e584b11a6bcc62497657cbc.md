### Title
Silent Bonus Reward Forfeiture with Permanent Checkpoint Advance During Underfunded Redelegation Slash - (`File: x/tieredrewards/keeper/slash.go`)

### Summary

In `slashRedelegationPosition`, when the bonus pool is insufficient to pay accrued bonus rewards during a `BeforeRedelegationSlashed` hook, the error is deliberately swallowed to avoid a chain halt. However, `processEventsAndClaimBonus` advances the position's bonus checkpoints (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) and decrements event reference counts **before** the pool balance check. The modified position is then persisted with advanced checkpoints. The accrued bonus rewards are permanently lost with no mechanism for the owner to reclaim them.

### Finding Description

`slashRedelegationPosition` in `x/tieredrewards/keeper/slash.go` calls `processEventsAndClaimBonus` and explicitly swallows `ErrInsufficientBonusPool`: [1](#0-0) 

Inside `processEventsAndClaimBonus`, the position's checkpoints are advanced and event reference counts are decremented **before** the pool balance check: [2](#0-1) 

The pool check only happens after all state mutations to `pos` and event ref counts: [3](#0-2) 

When `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`, the function returns an error with `pos` already mutated in memory (`LastBonusAccrual` advanced to `blockTime`, `LastEventSeq` advanced past all processed events, event ref counts decremented in the store). Back in `slashRedelegationPosition`, the error is swallowed and `setPositionWithState` is called with the mutated `pos`: [4](#0-3) 

The persisted position now has checkpoints advanced past the period for which bonus was owed but never paid. The owner has no path to reclaim the forfeited bonus — `MsgClaimTierRewards` will compute zero bonus for the already-advanced window.

This contrasts with user-driven paths, which fail atomically (the entire transaction rolls back, so no checkpoint is advanced): [5](#0-4) 

### Impact Explanation

A tier position owner permanently loses all bonus rewards accrued between `pos.LastBonusAccrual` and the slash block time. The corrupted value is the owner's bonus reward balance: the `RewardsPoolName` module account retains the tokens that should have been transferred, but the position's checkpoints are advanced so those tokens can never be claimed by the rightful owner. The loss is proportional to `shares × tokensPerShare × bonusApy × duration / SecondsPerYear` for the unclaimed window.

### Likelihood Explanation

The trigger requires two concurrent conditions: (1) a validator slash that affects a redelegating tier position, and (2) the bonus pool being underfunded at that block. The bonus pool can be drained by the `BeginBlocker` top-up mechanism (which transfers pool funds to the distribution module every block when `TargetBaseRewardsRate > 0`): [6](#0-5) 

It can also be drained by concurrent `MsgClaimTierRewards` calls from many stakers. Validator slashing is a normal protocol event (downtime, double-sign). The combination is realistic in production, especially when the pool is near-empty.

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` (and the event reference count decrements) to **after** the `sufficientBonusPoolBalance` check, so that if the pool is insufficient, the position's checkpoints are not advanced and the owner can retry the claim once the pool is replenished. Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, explicitly restore the original checkpoint values on `pos` before calling `setPositionWithState`, so the position is saved with its pre-call checkpoints intact.

### Proof of Concept

1. Alice creates a tier position and redelegates to validator B via `MsgTierRedelegate`. A `RedelegationMapping` entry is created.
2. Time passes; Alice accrues significant bonus rewards.
3. The bonus pool is drained (e.g., by BeginBlocker top-ups or concurrent claims from other stakers).
4. Validator B is slashed. The staking module calls `SlashRedelegation` → `BeforeRedelegationSlashed` → `slashRedelegationPosition`.
5. Inside `slashRedelegationPosition`, `processEventsAndClaimBonus` runs: it walks events, decrements their ref counts in the store, advances `pos.LastBonusAccrual` to the current block time, advances `pos.LastEventSeq`, then hits `sufficientBonusPoolBalance` — which fails because the pool is empty.
6. The error is caught and swallowed at `slash.go:56`. `setPositionWithState` is called with the mutated `pos`.
7. Alice's position is now saved with `LastBonusAccrual = slashBlockTime` and `LastEventSeq` advanced past all events.
8. Alice calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` computes zero bonus (no events since `LastEventSeq`, and `LastBonusAccrual == blockTime`). Alice receives nothing for the forfeited period.
9. Even after the pool is replenished, Alice cannot recover the lost bonus.
