Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Recipient `owner` Instead of Actual Depositor `sender`, Allowing Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` who satisfies the token callback) and gates access solely on `owner` (the LP position recipient). Any unprivileged address can bypass the deposit allowlist by calling `addLiquidity` with an allowlisted address as `owner`, injecting unauthorized liquidity into a permissioned pool while the allowlisted address receives the LP shares.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the LP recipient to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly encodes both into the hook call: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as anonymous and checks only `owner`: [3](#0-2) 

**Exploit path:**
1. Pool admin configures `DepositAllowlistExtension` on a permissioned pool, allowlisting only trusted addresses (e.g., `trustedLP`).
2. Attacker (not allowlisted) calls `pool.addLiquidity(owner=trustedLP, salt, deltas, callbackData, extensionData)`.
3. The hook fires: `sender` = attacker address (discarded), `owner` = `trustedLP` (checked and passes).
4. The allowlist check passes; attacker's callback provides tokens; LP shares are minted to `trustedLP`.
5. Unauthorized liquidity is injected into the permissioned pool with no revert.

The existing guard `allowedDepositor[msg.sender][owner]` uses `msg.sender` as the pool address (correct) but `owner` as the depositor identity (wrong). There is no check anywhere on the actual `sender`.

## Impact Explanation
The deposit allowlist — the sole access control mechanism restricting who may add liquidity to a permissioned pool — is completely ineffective. Any unprivileged address can inject liquidity into a pool intended to be restricted, violating the admin-enforced boundary. This constitutes an admin-boundary break where an unprivileged path bypasses a pool admin's access control, meeting the contest's allowed impact gate.

## Likelihood Explanation
No special privileges are required. Any externally owned account or contract can call `addLiquidity` on any pool using `DepositAllowlistExtension`. The attacker only needs to know one allowlisted address to use as `owner`. The attack is trivially repeatable and requires no setup beyond a standard `addLiquidity` call.

## Recommendation
Replace the anonymous first parameter with a named `sender` and check it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```solidity
// Foundry test sketch
function test_allowlistBypass() public {
    address trustedLP = makeAddr("trustedLP");
    address attacker  = makeAddr("attacker");

    // Admin allowlists only trustedLP
    vm.prank(poolAdmin);
    depositAllowlist.setAllowedToDeposit(address(pool), trustedLP, true);

    // Attacker is NOT allowlisted
    assertFalse(depositAllowlist.allowedDepositor(address(pool), attacker));

    // Attacker calls addLiquidity with owner=trustedLP
    vm.prank(attacker);
    // Provide tokens via callback from attacker's address
    pool.addLiquidity(trustedLP, salt, deltas, callbackData, extensionData);
    // Expect: no revert, liquidity added, allowlist bypassed
}
```
