Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Originating Caller, Enabling Full Per-User Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the immediate caller of the pool. When swaps are routed through `MetricOmmSimpleRouter`, the router becomes `msg.sender` at the pool boundary. If the pool admin allowlists the router (a natural operational step), every unprivileged address can bypass the per-user allowlist by routing through the router, completely defeating the access-control invariant.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- whoever called pool.swap()
  recipient,
  ...
  extensionData
);
```

`_beforeSwap` forwards that value unchanged through `ExtensionCalling` to the registered extensions. `SwapAllowlistExtension.beforeSwap` then checks whether `sender` (the pool's `msg.sender`) is on the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension caller) and `sender` is whoever called `pool.swap()`.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no mechanism to forward the originating user's identity:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

The router is `msg.sender` at the pool boundary, so the extension sees `sender = router`. The actual user who called the router is invisible to the extension. The same identity-loss occurs in `exactInput` (L104-112), `exactOutputSingle` (L136-137), and `exactOutput` (L165-181).

There is no existing guard that recovers the original caller: `extensionData` is passed through from the user's call parameters and is not populated by the router with any caller identity.

## Impact Explanation

A pool admin who deploys a pool with `SwapAllowlistExtension` to restrict swaps to specific addresses will:
1. Allowlist specific users: `setAllowedToSwap(pool, user1, true)`.
2. Allowlist the router so permitted users can use it: `setAllowedToSwap(pool, router, true)`.

Step 2 is a routine, non-malicious operational action. Its unintended consequence is that **any address** can call `router.exactInputSingle({pool: pool, ...})`. The extension sees `sender = router`, which is allowlisted, and the swap proceeds regardless of who the actual caller is. The per-user allowlist is completely defeated. The admin has no way to selectively enable router-based swaps for only allowlisted users: either the router is not allowlisted (blocking all router paths, including for permitted users) or it is allowlisted (opening the pool to everyone).

This matches the "admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" impact class. The corrupted value is the `allowedSwapper[pool][sender]` extension decision — it evaluates to `true` for the router when it should evaluate against the originating user.

## Likelihood Explanation

The trigger is the pool admin allowlisting the router — a routine operational step any operator would take to make the pool usable through the standard periphery. No malicious admin intent is required. Once the router is allowlisted, any unprivileged address can exploit the bypass immediately and repeatedly with no special capability or capital requirement beyond the swap input tokens.

## Recommendation

The extension must gate the **original user**, not the intermediary. The router should encode `msg.sender` into `extensionData` before calling `pool.swap()`, and `SwapAllowlistExtension.beforeSwap` should decode and verify that field when `sender` is a known router (or always, if a trusted-router convention is adopted). Alternatively, the extension can require a pool-admin-signed allowlist credential in `extensionData` when `sender` is not itself on the per-address allowlist.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `permittedUser` is allowlisted.
// Admin also allowlists the router so permittedUser can use it.
extension.setAllowedToSwap(address(pool), address(router), true);
extension.setAllowedToSwap(address(pool), permittedUser, true);

// Attacker is NOT allowlisted.
// Direct swap reverts:
vm.prank(attacker);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(attacker, false, 1000, type(uint128).max, "", "");

// Router-mediated swap succeeds — allowlist bypassed:
vm.prank(attacker);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        recipient: attacker,
        tokenIn: address(token1),
        zeroForOne: false,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: type(uint128).max,
        deadline: block.timestamp,
        extensionData: ""
    })
);
// Attacker receives token0 — allowlist was not enforced.
// Root cause: extension checks sender=router (allowlisted) not attacker (not allowlisted).
```
