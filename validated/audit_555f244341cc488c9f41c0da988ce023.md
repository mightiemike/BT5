Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original Caller, Allowing Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` validates the `sender` argument, which is `msg.sender` of the `pool.swap()` call. When `MetricOmmSimpleRouter` executes a swap, it becomes `msg.sender` of `pool.swap()`, so the extension checks the router address against the allowlist rather than the original user. Any pool admin who allowlists the router to enable legitimate users to trade also inadvertently opens the pool to every caller of the router.

## Finding Description
In `MetricOmmPool.swap`, the pool passes `msg.sender` directly as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
  msg.sender,   // router address when called via MetricOmmSimpleRouter
  recipient, ...
);
``` [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that `sender` against the per-pool allowlist:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

In `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly, making itself `msg.sender` of that call. The original user's address is stored only in transient callback context (`_setNextCallbackContext`) for payment purposes and is never forwarded to the extension via `extensionData`:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., "", params.extensionData);
``` [3](#0-2) 

The same pattern applies to `exactInput` (multi-hop), `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension` and allowlists users A, B, C.
2. Admin also calls `setAllowedToSwap(pool, router, true)` so A, B, C can use the router.
3. Non-allowlisted user D calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` — `msg.sender` = router.
5. Pool calls `extension.beforeSwap(sender=router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `true`.
7. D's swap executes successfully despite not being allowlisted.

The `params.extensionData` passed by the router is user-supplied and unverified, so the extension cannot safely decode an original-caller claim from it without a trusted attestation mechanism that does not exist.

## Impact Explanation
Any unprivileged user can bypass a pool's swap allowlist by routing through the public `MetricOmmSimpleRouter`. This breaks the core access-control invariant of `SwapAllowlistExtension`: pools configured for restricted trading (regulatory compliance, whitelisted market makers, private pools) become fully open to any caller via the router. Unauthorized swaps drain pool liquidity at oracle prices, directly harming LP principal — a direct loss of user/LP assets meeting Sherlock High/Critical thresholds.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard public swap interface. The precondition — router being allowlisted — is the normal operational path whenever any allowlisted user needs to use the router. Any user who observes the allowlist configuration can trivially route around it with a single router call. No special privileges, flash loans, or complex setup are required.

## Recommendation
The extension must verify the original user, not the immediate pool caller. The cleanest fix is for `MetricOmmSimpleRouter` to ABI-encode `msg.sender` into `extensionData` before calling `pool.swap()`, and for `SwapAllowlistExtension.beforeSwap` to decode and check that field when `sender` is a known trusted router. Alternatively, the pool could expose a mechanism for the router to attest the original caller in a tamper-proof way (e.g., via a dedicated router registry checked by the extension). A simpler but more restrictive option is to never allowlist the router address and instead require users to call the pool directly for allowlisted pools.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension (beforeSwapOrder = extension1)
  - setAllowedToSwap(pool, userA, true)
  - setAllowedToSwap(pool, router, true)  // admin adds so userA can use router

Attack:
  - userD (not allowlisted) calls:
      router.exactInputSingle({pool: pool, tokenIn: T0, tokenOut: T1,
                               recipient: userD, amountIn: X, ...})
  - Router executes: _setNextCallbackContext(pool, JUST_PAY, userD, T0)
  - Router calls: pool.swap(userD, true, X, limit, "", extensionData)
    → msg.sender = router
  - Pool calls: extension.beforeSwap(sender=router, ...)
  - Extension checks: allowedSwapper[pool][router] → true → passes
  - Swap executes; userD receives token1 output

Result:
  - userD receives tokens from a restricted pool
  - SwapAllowlistExtension invariant broken: non-allowlisted user traded
  - Foundry test: assert swap succeeds for userD via router when router is allowlisted
```
