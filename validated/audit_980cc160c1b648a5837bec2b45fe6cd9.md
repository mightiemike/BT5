### Title
Phantom Duplicate Index Accepted in Odd-Length Merkle Tree — (`merkle-tools/src/lib.rs`)

### Summary

`compute_root_from_merkle_proof` accepts a phantom `tx_index = N` (one past the last real leaf) for any odd-length block, producing the same root as the legitimate `tx_index = N-1`. The contract returns `true` for an inclusion claim at a position that does not exist in the Bitcoin block.

### Finding Description

**Root cause — `merkle_proof_calculator` (lines 10–11):**

When the tree has an odd number of leaves, the last leaf is duplicated:

```rust
if current_hashes.len() % 2 == 1 {
    current_hashes.push(current_hashes[current_hashes.len() - 1].clone());
}
``` [1](#0-0) 

For a 5-leaf tree `[T0, T1, T2, T3, T4]`, the proof for position 4 is:

```
proof_for_4 = [T4, H(T4,T4), H(H(T0,T1), H(T2,T3))]
```

**Root cause — `compute_root_from_merkle_proof` (lines 42–48):**

```rust
for proof_hash in merkle_proof {
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
``` [2](#0-1) 

**Trace for `tx_index = 4` (legitimate):**
- pos=4 (even): `H(T4, proof[0]=T4)` → `H(T4,T4)`, pos→2
- pos=2 (even): `H(H(T4,T4), proof[1]=H(T4,T4))` → `H(H(T4,T4),H(T4,T4))`, pos→1
- pos=1 (odd): `H(proof[2], ...)` → ROOT ✓

**Trace for `tx_index = 5` (phantom, same proof):**
- pos=5 (odd): `H(proof[0]=T4, T4)` → `H(T4,T4)`, pos→2
- pos=2 (even): `H(H(T4,T4), proof[1]=H(T4,T4))` → `H(H(T4,T4),H(T4,T4))`, pos→1
- pos=1 (odd): `H(proof[2], ...)` → ROOT ✓

Both produce the identical root. The symmetry arises because `proof[0] == leaf` (the duplicated leaf is its own sibling), so `H(leaf, proof[0])` and `H(proof[0], leaf)` are identical — `H(T4, T4) == H(T4, T4)` regardless of argument order.

**The contract entrypoint passes `tx_index` directly to this function without bounds-checking:**

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [3](#0-2) 

There is no check that `tx_index < actual_tx_count`. The block header stores only the Merkle root, not the transaction count, so the contract has no way to enforce this bound.

**`verify_transaction_inclusion_v2` does not fix this.** It adds a coinbase proof check (index 0 is always even, never affected by this bug), then delegates to `verify_transaction_inclusion` with the attacker-supplied `tx_index` unchanged: [4](#0-3) 

### Impact Explanation

The contract returns `true` for an inclusion claim at a phantom position that does not exist in the Bitcoin block. Any downstream NEAR contract that uses `(block_hash, tx_index)` as a uniqueness key to prevent double-processing is vulnerable: an attacker can present the same real transaction twice — once at index `N-1` and once at the phantom index `N` — and both calls return `true`. This enables double-spending or double-redemption of any asset gated on SPV proof verification.

The bug affects every odd-length Bitcoin block (roughly half of all blocks), making it broadly applicable.

### Likelihood Explanation

The attack requires only knowledge of a real transaction in an odd-length block and the ability to call the public NEAR contract method. No privileged role, leaked key, or social engineering is needed. The phantom index is trivially computed as `actual_last_index + 1`.

### Recommendation

Reject any `tx_index` that is out of range. Since the block header does not contain the transaction count, the proof length encodes the tree depth. The maximum valid index for a proof of length `L` is `2^L - 1`. Enforce:

```rust
require!(
    args.tx_index < (1u64 << args.merkle_proof.len()),
    "tx_index out of range for proof length"
);
```

This is a necessary but not sufficient bound. A stronger fix is to require callers to supply the transaction count and validate `tx_index < tx_count`, or to require that the proof not contain a self-sibling (i.e., `proof[0] != tx_id` when `tx_index` is even, which would catch the phantom duplicate case directly).

### Proof of Concept

```rust
#[test]
fn test_phantom_duplicate_index() {
    // 5-leaf tree: T4 is the last real leaf
    let t4 = decode_hex("048f3897c16bdc59ec1187aa080a4b4aa5ec1afcb4b776cf8b8a214b01990a7b");
    let tx_hashes = vec![
        decode_hex("18afbf37d136ff62644b231fcde72f1fb8edd04a798fb00cb06360da635da275"),
        decode_hex("30b19832a5f4b952e151de77d96139987492becc8b6e1e914c4103cfbb06c01e"),
        decode_hex("b94ed12902e35b29dd53cf25e665b4d0bc92f22adbc383ad90566584902b061d"),
        decode_hex("1920e5d8a10018dc65308bb4d1f11d30b5406c6499688443bfcd1ef364206b14"),
        t4.clone(),
    ];

    // Generate legitimate proof for position 4
    let proof = merkle_proof_calculator(tx_hashes, 4);

    let root_at_4 = compute_root_from_merkle_proof(t4.clone(), 4, &proof);
    // Phantom position 5 — does not exist in the block
    let root_at_5 = compute_root_from_merkle_proof(t4.clone(), 5, &proof);

    // Both return the same root → contract returns true for the phantom index
    assert_eq!(root_at_4, root_at_5,
        "VULNERABILITY: phantom index 5 produces same root as real index 4");
}
```

This test passes on the unmodified codebase, confirming the invariant is broken: `verify_transaction_inclusion` returns `true` for `tx_index = 5` in a 5-transaction block, an index that corresponds to no real transaction.
