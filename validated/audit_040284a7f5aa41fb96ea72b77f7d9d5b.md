### Title
Linked Signer Can Unilaterally Replace Itself via `LinkSigner` Transaction, Taking Over Subaccount Signing Authority ŌĆö (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.sol`, the fast-path handler for `TransactionType.LinkSigner` calls `validateSignedTx` with `allowLinkedSigner = true`. This means the **current linked signer** is accepted as a valid signer for a transaction that **replaces the linked signer**. A compromised or malicious linked signer can therefore replace itself with an attacker-controlled address, permanently taking over signing authority for the subaccount ŌĆö including the ability to sign `WithdrawCollateral` transactions to drain all collateral.

---

### Finding Description

The `linkedSigners` mapping in `EndpointStorage.sol` stores one authorized delegate signer per subaccount. This linked signer is permitted to sign any transaction on behalf of the subaccount owner. [1](#0-0) 

In `processTransactionImpl`, the `LinkSigner` fast-path handler validates the transaction signature with `allowLinkedSigner = true`: [2](#0-1) 

`validateSignedTx` with `allowLinkedSigner = true` delegates to `validateSignature`, which passes the current linked signer as an accepted signer: [3](#0-2) 

`Verifier.validateSignature` accepts the signature if it recovers to **either** the subaccount owner address **or** the linked signer: [4](#0-3) 

This means the current linked signer can produce a valid `LinkSigner` transaction that writes a new address into `linkedSigners[subaccount]`. There is no restriction preventing the linked signer from modifying its own slot.

By contrast, the **slow-mode** path for `LinkSigner` uses `validateSender`, which enforces that `msg.sender` must be the subaccount owner address ŌĆö it is not vulnerable: [5](#0-4) 

The fast path has no equivalent owner-only guard.

Additionally, `getLinkedSigner` propagates the parent subaccount's linked signer to all isolated subaccounts, so a single compromised linked signer affects the entire isolated subaccount tree: [6](#0-5) 

---

### Impact Explanation

Once the attacker replaces the linked signer with their own address, they can sign any subsequent transaction for the subaccount ŌĆö including `WithdrawCollateral` and `WithdrawCollateralV2` ŌĆö to drain all deposited collateral. The original subaccount owner cannot reclaim signing authority through the fast path (their nonce has advanced), and must rely on the slow-mode path with a 3-day delay, during which funds can be fully extracted.

---

### Likelihood Explanation

Linked signers are intended for automated trading bots or API keys. These are high-value targets for key compromise (server breaches, leaked environment variables, etc.). The attack requires only a single valid signature from the compromised linked signer key ŌĆö no admin access, no governance capture, and no sequencer compromise. The sequencer will process the transaction as valid because the signature check passes.

---

### Recommendation

Pass `allowLinkedSigner = false` when validating `LinkSigner` transactions in `processTransactionImpl`. Only the subaccount owner (the address embedded in the subaccount `bytes32`) should be permitted to change the linked signer. The slow-mode path already enforces this correctly via `validateSender`.

```solidity
// In processTransactionImpl, change:
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // <-- was true; linked signer must NOT authorize its own replacement
);
```

---

### Proof of Concept

1. Alice sets `linkedSigners[aliceSubaccount] = tradingBot` via a legitimate `LinkSigner` transaction.
2. Attacker compromises `tradingBot`'s private key.
3. Attacker constructs a `SignedLinkSigner` transaction: `{ sender: aliceSubaccount, signer: attackerAddress, nonce: currentNonce }`, signed by `tradingBot`.
4. Sequencer receives the transaction and calls `processTransactionImpl`.
5. `validateSignedTx(..., allowLinkedSigner = true)` ŌåÆ `validateSignature` ŌåÆ `Verifier.validateSignature` recovers `tradingBot` == `linkedSigners[aliceSubaccount]` ŌåÆ **passes**.
6. `linkedSigners[aliceSubaccount]` is overwritten with `attackerAddress`.
7. Attacker signs a `WithdrawCollateral` transaction for `aliceSubaccount` using `attackerAddress`'s key.
8. Signature validation passes (attacker is now the linked signer); all collateral is withdrawn to attacker's address.

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
