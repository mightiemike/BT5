Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on Caller-Controlled `owner` Instead of `sender`, Allowing Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of the pool call) and instead checks `owner`, which is a free parameter supplied by the caller. Any unprivileged address can bypass the deposit allowlist by naming an already-allowlisted address as `owner` when calling `pool.addLiquidity`. This breaks the core permissioning invariant of pools configured with this extension.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as two distinct arguments into the extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
_callExtensionsInOrder(
  BEFORE_ADD_LIQUIDITY_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

However, `DepositAllowlistExtension.beforeAddLiquidity` ignores the first parameter entirely and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`owner` is the address that will hold the resulting LP position — a value the caller chooses freely in the `addLiquidity` call signature. It is not the address supplying tokens. By contrast, `SwapAllowlistExtension.beforeSwap` correctly gates on `sender`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
  if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
  }
  ...
}
```

The exploit path is direct: an unprivileged attacker calls `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)`. The extension checks `allowedDepositor[pool][allowlistedAddress] == true` → guard passes. The attacker's callback supplies the tokens; the LP position is minted to the allowlisted address.

## Impact Explanation

A pool configured with `DepositAllowlistExtension` is explicitly permissioned — only allowlisted addresses should be able to supply liquidity. The broken check allows any unprivileged address to alter pool bin totals and per-share balances without authorization. This directly affects extensions that derive their state from pool bin data (e.g., `OracleValueStopLossExtension` watermarks), enabling watermark manipulation that can suppress or spuriously trigger stop-loss guards. Additionally, LP positions are minted to addresses that did not consent to the deposit, forcing unwanted exposure on allowlisted owners. This constitutes broken core pool functionality causing potential loss of funds and unusable/corrupted pool state.

## Likelihood Explanation

Exploitation requires no special privilege, no flash loan, and no oracle manipulation. Allowlisted addresses are discoverable on-chain via `AllowedToDepositSet` events. Any EOA or contract can call `pool.addLiquidity` with an allowlisted address as `owner`. The attack is repeatable at will.

## Recommendation

Gate on `sender` (the actual depositor who supplies tokens via callback), consistent with `SwapAllowlistExtension`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
  {
-   if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+   if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
      revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

## Proof of Concept

```solidity
// Assume pool has DepositAllowlistExtension; only `alice` is allowlisted.
address alice = makeAddr("alice");
extension.setAllowedToDeposit(address(pool), alice, true);

// Attacker (bob) is NOT allowlisted.
address bob = makeAddr("bob");
vm.startPrank(bob);

// Bob calls addLiquidity naming alice as owner.
// Extension checks allowedDepositor[pool][alice] == true → guard passes.
// Bob's callback supplies the tokens; alice receives the LP shares.
pool.addLiquidity(
    alice,        // owner — allowlisted, so guard passes
    0,            // salt
    deltas,
    callbackData,
    extensionData
);
// Bob has bypassed the deposit allowlist without being allowlisted.
vm.stopPrank();
