### Title
Fee-on-Transfer Token Deposit Overcredits Subaccount Balance and Drains Clearinghouse Reserves — (`File: core/contracts/EndpointStorage.sol`)

### Summary

`handleDepositTransfer` in `EndpointStorage.sol` does not verify the actual amount received after `safeTransferFrom`. It blindly forwards the nominal `amount` to the clearinghouse and records that same `amount` in the slow-mode queue for on-chain crediting. With a fee-on-transfer token (e.g., USDT with fees enabled), the Endpoint receives less than `amount`, yet the clearinghouse credits the full `amount` to the depositor's subaccount — corrupting spot balance accounting and potentially draining other users' funds held at the Endpoint.

---

### Finding Description

The deposit flow is:

1. `Endpoint.depositCollateralWithReferral` calls `handleDepositTransfer(token, msg.sender, uint256(amount))`.
2. `handleDepositTransfer` executes two sequential transfers using the same nominal `amount`:

```solidity
// EndpointStorage.sol lines 111-119
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);          // pulls `amount` from user
    safeTransferTo(token, address(clearinghouse), amount); // pushes `amount` to clearinghouse
}
``` [1](#0-0) 

3. After the transfer, a `SlowModeTx` is enqueued with the original `amount`:

```solidity
// Endpoint.sol lines 152-165
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    ...
    tx: abi.encodePacked(
        uint8(TransactionType.DepositCollateral),
        abi.encode(DepositCollateral({ sender: subaccount, productId: productId, amount: amount }))
    )
});
``` [2](#0-1) 

4. When the sequencer processes the slow-mode tx, `Clearinghouse.depositCollateral` credits `txn.amount` (the original nominal value) to the subaccount via `spotEngine.updateBalance`:

```solidity
// Clearinghouse.sol lines 205-208
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [3](#0-2) 

No balance snapshot is taken before/after `safeTransferFrom` to compute the actual received amount. The `ERC20Helper.safeTransferFrom` wrapper only checks for call success and a truthy return value — it does not validate that `balanceOf(recipient)` increased by `amount`. [4](#0-3) 

---

### Impact Explanation

Two distinct asset-level corruptions occur simultaneously:

**1. Endpoint drains other depositors' funds.**
`safeTransferFrom` pulls `amount - fee` from the depositor into the Endpoint. `safeTransferTo` then immediately pushes the full `amount` to the clearinghouse. The shortfall (`fee`) is silently covered by tokens already sitting in the Endpoint from prior deposits or slow-mode fee accumulation, stealing from other users.

**2. Clearinghouse spot balance is overcredited.**
The slow-mode queue records `amount`, so `spotEngine.updateBalance` credits the depositor with `amount` worth of collateral even though the clearinghouse only received `amount - fee`. Every such deposit inflates the depositor's subaccount balance beyond the real token backing, making the system fractionally insolvent. Repeated deposits compound the deficit until withdrawals fail or liquidations are undercollateralized.

---

### Likelihood Explanation

USDT — one of the most commonly listed spot assets on any DEX/CEX — has a fee switch in its contract that has historically been set to 0 but can be enabled by the USDT issuer at any time. If USDT is listed as a spot product and its fee is enabled, every subsequent `depositCollateral` call by any user triggers this bug. No special attacker capability is required beyond being a normal depositor.

---

### Recommendation

Measure the actual received amount using a balance snapshot pattern in `handleDepositTransfer`:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    require(actualReceived == amount, "Fee-on-transfer token not supported");
}
```

Alternatively, propagate `actualReceived` back to the caller so the slow-mode tx records the real credited amount rather than the nominal input. Additionally, maintain an explicit token allowlist that rejects known fee-on-transfer and rebasing tokens at product listing time.

---

### Proof of Concept

1. USDT is listed as a spot product (productId = X) in the Nado protocol.
2. USDT issuer enables a 0.5% transfer fee.
3. Attacker (or any user) calls `Endpoint.depositCollateral("default", X, 1_000_000e6)`.
4. `handleDepositTransfer` pulls `995_000e6` USDT from the caller into the Endpoint (fee deducted), then pushes `1_000_000e6` from the Endpoint to the clearinghouse — the 5_000e6 shortfall is taken from other users' tokens already in the Endpoint.
5. The slow-mode queue records `amount = 1_000_000e6`.
6. Sequencer processes the tx; `Clearinghouse.depositCollateral` calls `spotEngine.updateBalance(X, subaccount, 1_000_000e6 * multiplier)`.
7. The attacker's subaccount now shows `1_000_000e6` of collateral, but the clearinghouse only holds `995_000e6` net new tokens.
8. Repeated deposits compound the deficit; eventual withdrawals by honest users fail due to insufficient clearinghouse token balance.

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
