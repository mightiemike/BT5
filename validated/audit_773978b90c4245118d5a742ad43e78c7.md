### Title
Pausing `verify_transaction_inclusion_v2` Does Not Block the Deprecated `verify_transaction_inclusion` Entry Point — (`contract/src/lib.rs`)

---

### Summary

The `near_plugins` `#[pause]` macro creates an **independent** pausable feature per decorated function, keyed by function name. `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are therefore two separate pausable features. Pausing the secure v2 entry point does not block the deprecated v1 entry point. Because v1 is still `pub` and lacks the coinbase-proof depth check, any unprivileged NEAR caller can invoke it directly while v2 is paused, exploiting the known 64-byte transaction Merkle-proof forgery to obtain a `true` SPV result for a non-existent Bitcoin transaction.

---

### Finding Description

`verify_transaction_inclusion` is decorated with `#[pause]` and is `pub`:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    // ...
    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root
}
```

`verify_transaction_inclusion_v2` is also decorated with `#[pause]` and delegates to v1 internally:

```rust
#[pause]
pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
    // coinbase depth check ...
    #[allow(deprecated)]
    self.verify_transaction_inclusion(args.into())   // internal call — v1's own #[pause] check fires here too
}
```

In `near_plugins`, `#[pause]` expands to a guard that calls `self.pa_is_paused("<function_name>")` at the top of the function body. The feature name is the function's own identifier. Consequently:

- Pausing `"verify_transaction_inclusion_v2"` blocks only that entry point.
- `"verify_transaction_inclusion"` is a **distinct** feature; it remains unpaused and callable by any NEAR account.

There is also a structural asymmetry: because v2 calls v1 internally, pausing `"verify_transaction_inclusion"` silently breaks v2 as well (the internal call panics). But the reverse is not true — pausing v2 leaves v1 fully reachable.

The v1 function performs no coinbase-proof depth validation. It accepts any 32-byte value as `tx_id`, including an internal Merkle-tree node hash, and returns `true` whenever the recomputed root matches the stored block header's `merkle_root`. This is the documented 64-byte transaction forgery path.

---

### Impact Explanation

An attacker who calls `verify_transaction_inclusion` directly while `verify_transaction_inclusion_v2` is paused can supply a crafted `tx_id` that is actually an internal Merkle-tree node. The function returns `true`, producing a false SPV proof of inclusion for a Bitcoin transaction that does not exist. Any dependent NEAR contract that gates fund releases or cross-chain actions on this boolean result will incorrectly authorize the attacker's claim, leading to direct asset loss in those contracts.

---

### Likelihood Explanation

The trigger requires the `PauseManager` to pause `verify_transaction_inclusion_v2` without also pausing `verify_transaction_inclusion`. This is plausible in at least two realistic scenarios:

1. The operator discovers a bug in the coinbase-proof check added in v2 and pauses v2 to stop its use, not realising v1 is still a live public entry point.
2. The operator intends to pause "the verification feature" and pauses only the current recommended function, unaware that the deprecated function is independently pausable.

Once the operator has paused v2, the exploit itself requires no privileges — any NEAR account can call `verify_transaction_inclusion` directly.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` so it becomes a private helper callable only by `verify_transaction_inclusion_v2`. Its `#[pause]` decorator is then irrelevant for external access and the asymmetry disappears:

```rust
// Change `pub fn` → `fn` (private)
fn verify_transaction_inclusion(&self, args: ProofArgs) -> bool { ... }
```

If keeping it public is required for backward compatibility, add an explicit cross-feature pause check inside v1 that also checks whether `verify_transaction_inclusion_v2` is paused, or apply a shared named feature to both functions via `#[pause(name = "spv_verification")]`.

---

### Proof of Concept

1. `PauseManager` calls `pa_pause_feature("verify_transaction_inclusion_v2")` to pause the secure entry point (e.g., in response to a reported bug in the coinbase-proof logic).
2. Attacker identifies a Bitcoin block already stored in `headers_pool` whose Merkle tree has at least two levels. They select two sibling internal-node hashes `L` and `R` such that `double_sha256(L ‖ R)` equals a known parent node `P` that lies on the path to the Merkle root.
3. Attacker calls `verify_transaction_inclusion` with:
   - `tx_id = P` (the internal node hash, 32 bytes — not a real transaction)
   - `tx_block_blockhash` = the stored block's hash
   - `tx_index` = the index corresponding to `P`'s position in the tree
   - `merkle_proof` = the remaining path from `P` to the Merkle root
   - `confirmations = 1`
4. `verify_transaction_inclusion` is not paused (only v2 is). The function computes `compute_root_from_merkle_proof(P, index, proof)`, which equals the block's `merkle_root`. The function returns `true`.
5. Any dependent NEAR contract that called `verify_transaction_inclusion` to authorise a withdrawal now releases funds to the attacker for a Bitcoin transaction that never existed. [1](#0-0) [2](#0-1) [3](#0-2)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
