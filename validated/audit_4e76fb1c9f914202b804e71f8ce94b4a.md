### Title
Missing Sanctions Check on Linked Signer Registration Allows Sanctioned Address to Bypass Enforcement - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `EndpointTx.sol` registers an arbitrary address as an authorized signer for a subaccount without verifying whether that address is sanctioned. Because `Verifier.validateSignature` accepts signatures from linked signers without any sanctions check, a sanctioned address can be registered as a linked signer and subsequently sign and execute protocol transactions on behalf of a non-sanctioned subaccount, fully bypassing the sanctions enforcement that exists elsewhere in the protocol.

---

### Finding Description

The Nado protocol enforces a sanctions list via `ISanctionsList.isSanctioned()`, exposed through `EndpointStorage.requireUnsanctioned()`. This check is applied to depositors in `depositCollateralWithReferral` and to slow-mode transaction submitters in `submitSlowModeTransactionImpl`. However, the `LinkSigner` transaction type â€” which registers a new authorized signer for a subaccount â€” performs no sanctions check on the signer address being registered.

`Verifier.validateSignature` only checks that the recovered address matches the subaccount owner or the linked signer â€” no sanctions enforcement:

```solidity

The sanctions check infrastructure exists and is used in other paths:

```solidity

    require(!sanctions.isSanctioned(sender), ERR_WALLET_SANCTIONED);
}
``` [5](#0-4) 

But it is never applied to the `signer` field in either `LinkSigner` code path.

---

### Impact Explanation

A sanctioned address can be registered as a linked signer for any non-sanctioned subaccount. Once registered, it can sign and submit all transaction types that accept linked signer signatures (`allowLinkedSigner = true`), including trades, withdrawals, transfers, and NLP operations. This completely circumvents the protocol's sanctions enforcement for the sanctioned address, allowing it to interact with the protocol indirectly through a proxy subaccount. The corrupted state is `linkedSigners[subaccount]` being set to a sanctioned address, which persists until explicitly overwritten.

---

### Likelihood Explanation

The attack requires only that a non-sanctioned subaccount owner (or the sanctioned address itself, if it controls a non-sanctioned wallet) submits a `LinkSigner` slow-mode transaction pointing to the sanctioned address. This is a standard user-accessible entry point with no privilege requirements. The slow-mode path (`submitSlowModeTransactionImpl`) does call `requireUnsanctioned(msg.sender)` on the submitter, but the submitter is the non-sanctioned subaccount owner â€” not the sanctioned signer being registered. The sanctioned address never needs to directly call any function to be registered.

---

### Recommendation

Add a `requireUnsanctioned` check on the signer address in both `LinkSigner` processing paths:

```diff
// Slow-mode path
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender);
+   requireUnsanctioned(address(uint160(bytes20(txn.signer))));
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));

// Sequencer-processed path
    validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, true);
+   requireUnsanctioned(address(uint160(bytes20(signedTx.tx.signer))));
    linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
```

Additionally, consider checking sanctions on the linked signer at signature validation time in `getLinkedSigner` or `validateSignature`, so that a signer who becomes sanctioned after registration is also blocked.

---

### Proof of Concept

1. Address `SANCTIONED` is added to the Chainalysis sanctions list.
2. Address `CLEAN` (non-sanctioned) owns a Nado subaccount `CLEAN_SUB`.
3. `CLEAN` submits a slow-mode `LinkSigner` transaction: `{ sender: CLEAN_SUB, signer: bytes32(uint256(uint160(SANCTIONED))) }`.
4. `submitSlowModeTransactionImpl` checks `requireUnsanctioned(msg.sender)` â€” passes because `msg.sender` is `CLEAN`.
5. No check is performed on `SANCTIONED`. `linkedSigners[CLEAN_SUB] = SANCTIONED` is written.
6. The sequencer processes the slow-mode transaction.
7. `SANCTIONED` now signs a `WithdrawCollateral` transaction for `CLEAN_SUB`.
8. `validateSignature` recovers `SANCTIONED`, matches it against `getLinkedSigner(CLEAN_SUB)`, and accepts it.
9. The withdrawal executes successfully â€” `SANCTIONED` has bypassed sanctions enforcement.
