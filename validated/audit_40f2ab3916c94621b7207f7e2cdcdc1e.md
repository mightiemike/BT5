### Title
`ValidResourceBounds` Protobuf Round-Trip Collapses `AllResources(l1_gas, 0, 0)` into `L1Gas`, Producing a Different Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf serialization of `ValidResourceBounds` is ambiguous: both `ValidResourceBounds::L1Gas(l1_gas)` and `ValidResourceBounds::AllResources(l1_gas, l2_gas=0, l1_data_gas=0)` serialize to identical protobuf bytes. On deserialization, both are decoded as `L1Gas`. However, `get_tip_resource_bounds_hash` computes **structurally different hashes** for these two variants — `AllResources` includes a third `L1_DATA_GAS` field element in the Poseidon chain while `L1Gas` does not. A V3 transaction submitted via RPC with `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` is assigned hash H1 at the gateway. After a P2P protobuf round-trip, the same transaction is decoded as `L1Gas(l1_gas=X)` and its hash recomputes to H2 ≠ H1. This is a direct analog to the external report's aliasing invariant: two distinct domain representations collapse to one wire encoding, but diverge in the hash/signature preimage.

### Finding Description

**Serialization collapse (same wire bytes, different semantic type):**

`ValidResourceBounds::L1Gas(l1_gas)` serializes as:
```
protobuf::ResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }
```
`ValidResourceBounds::AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` serializes as:
```
protobuf::ResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }
```

Both produce **identical protobuf bytes**. [1](#0-0) 

**Deserialization collapses both to `L1Gas`:**

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← AllResources(X,0,0) becomes L1Gas(X)
} else {
    ValidResourceBounds::AllResources(...)
})
``` [2](#0-1) 

**Hash divergence in `get_tip_resource_bounds_hash`:**

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts total
    }
});
Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
``` [3](#0-2) 

- `L1Gas(X)` → `poseidon(tip, concat(L1_GAS,X), concat(L2_GAS,0))` — **2 resource felts**
- `AllResources(X,0,0)` → `poseidon(tip, concat(L1_GAS,X), concat(L2_GAS,0), concat(L1_DATA_GAS,0))` — **3 resource felts**

These are provably different Poseidon outputs because the input lengths differ.

### Impact Explanation

A user submits a V3 transaction via RPC with `AllResourceBounds(l1_gas=X, l2_gas=0, l1_data_gas=0)`. The gateway converts this to `ValidResourceBounds::AllResources(X,0,0)` and computes hash H1 (3-felt chain). The transaction is stored and propagated over P2P as a protobuf message. A peer node deserializes the `ResourceBounds` proto and, because `l2_gas=0 && l1_data_gas=0`, reconstructs `ValidResourceBounds::L1Gas(X)`. When the peer recomputes the transaction hash it gets H2 (2-felt chain), H2 ≠ H1. The peer either:

1. **Rejects the transaction** — the recomputed hash does not match the transmitted hash, so the transaction is treated as invalid and dropped from the mempool/sync pipeline (valid transaction rejected before sequencing).
2. **Stores it under the wrong hash** — if the peer trusts the transmitted hash but later validates the signature against the recomputed hash, signature verification fails, binding the wrong hash to the executable payload.

Both outcomes match the allowed impact: *"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

### Likelihood Explanation

Any V3 transaction with non-zero L1 gas but zero L2 gas and zero L1 data gas triggers this. This is a valid and reachable input: the gateway's `validate_resource_bounds` check only requires at least one non-zero bound, and L1-gas-only V3 transactions are economically meaningful. No special privilege is required — any unprivileged user can submit such a transaction via the public RPC endpoint.

### Recommendation

The protobuf `ResourceBounds` message must carry an explicit discriminant field (e.g., a `bounds_type` enum: `L1_GAS_ONLY` vs `ALL_RESOURCES`) so that the `L1Gas` vs `AllResources` distinction survives the wire encoding. Alternatively, the deserialization logic must not silently coerce `AllResources(X,0,0)` to `L1Gas(X)` — it should preserve the original variant. The hash function `get_tip_resource_bounds_hash` must then be called with the correctly preserved variant.

### Proof of Concept

```
1. Craft a V3 InvokeTransaction with:
     resource_bounds = AllResourceBounds { l1_gas: {max_amount:1, max_price:1}, l2_gas: {0,0}, l1_data_gas: {0,0} }

2. Submit via RPC → gateway computes:
     H1 = poseidon("invoke", version, sender, poseidon(tip, L1_GAS||1||1, L2_GAS||0||0, L1_DATA_GAS||0||0), ...)

3. Serialize to protobuf::ResourceBounds → { l1_gas:{1,1}, l2_gas:{0,0}, l1_data_gas:{0,0} }

4. Peer deserializes: l2_gas.is_zero() && l1_data_gas.is_zero() → ValidResourceBounds::L1Gas({1,1})

5. Peer computes:
     H2 = poseidon("invoke", version, sender, poseidon(tip, L1_GAS||1||1, L2_GAS||0||0), ...)
     // L1_DATA_GAS field is absent

6. H1 ≠ H2 → peer rejects or misidentifies the transaction.
```

The divergence is in `get_tip_resource_bounds_hash` at [3](#0-2)  triggered by the type collapse at [2](#0-1) .
