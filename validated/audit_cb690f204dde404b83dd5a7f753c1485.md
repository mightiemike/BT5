Audit Report

## Title
Wrong-actor binding in `SwapAllowlistExtension.beforeSwap`: router address checked instead of end user — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap()` call. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the hook checks the router's address rather than the end user's address. This breaks the allowlist invariant in both directions: either all router-mediated swaps revert for legitimately allowlisted users, or any user can bypass the allowlist entirely by routing through the public router.

## Finding Description
In `MetricOmmPool.swap()`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

When `MetricOmmSimpleRouter.exactInputSingle()` calls the pool, the router is `msg.sender`: [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards this value verbatim to the extension as `sender`. `SwapAllowlistExtension.beforeSwap` then checks: [3](#0-2) 

The `sender` argument is the router address, not the end user. The allowlist is keyed by end-user addresses (`setAllowedToSwap(pool, alice, true)`), so:

- **Path A (router not allowlisted):** Every router-mediated swap reverts with `NotAllowedToSwap`, even for users the pool admin explicitly allowlisted.
- **Path B (router allowlisted to fix Path A):** Any arbitrary user bypasses the allowlist entirely by routing through `MetricOmmSimpleRouter`, since the hook only sees the router address and passes.

No existing guard in the extension or pool prevents this misbinding — the extension has no mechanism to distinguish the originating user from the immediate caller of `swap()`. [4](#0-3) 

## Impact Explanation
The allowlist is the pool designer's primary access-control mechanism for curated/permissioned pools. The wrong-actor binding means either (A) the allowlist breaks all router usage for legitimate users (broken core swap functionality), or (B) the allowlist is trivially bypassed by any user who routes through the public router (unauthorized swaps on permissioned pools). In either case, the invariant "only allowlisted addresses may swap" is violated. For pools restricting trading to KYC'd counterparties or specific market makers, this constitutes a policy bypass enabling unauthorized swaps — broken core functionality.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary public entrypoint for swaps. Any user calling `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` triggers this bug. No special conditions, privileged access, or unusual token behavior is required — the attacker simply calls the public router. [5](#0-4) 

## Recommendation
The pool cannot recover the original end user once the router is `msg.sender`. The fix must be applied at the router level: encode `msg.sender` (the end user) into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that value when present, falling back to `sender` for direct pool calls. Alternatively, adopt a trusted-forwarder pattern where the router signs or encodes the originating user and the extension verifies the router's identity before trusting the embedded address.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` active.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is allowlisted.
3. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. The pool calls `_beforeSwap(router_address, ...)`.
5. The extension checks `allowedSwapper[pool][router_address]`.
6. If the router was allowlisted (to allow Alice to use it), this returns `true` and Bob's swap succeeds — allowlist bypassed.
7. Conversely, if the router is not allowlisted, Alice's swap via the router reverts at step 5 even though she is explicitly allowlisted — legitimate users locked out.
