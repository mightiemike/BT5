Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and instead checks `owner` — the position beneficiary, which is a free caller-supplied parameter with no on-chain constraint tying it to `msg.sender`. Any address not on the allowlist can bypass the guard by naming any allowlisted address as `owner`, completely defeating the access-control intent of the extension.

## Finding Description

`ExtensionCalling._beforeAddLiquidity` encodes and forwards both `sender` (the pool's `msg.sender`) and `owner` (the caller-supplied beneficiary) to the extension hook: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives both parameters but leaves `sender` unnamed and unused, evaluating `allowedDepositor[msg.sender][owner]` — the pool-keyed allowlist entry for the beneficiary address — rather than for the actual depositor: [2](#0-1) 

Because `owner` is a free parameter supplied by the caller with no restriction, any non-allowlisted address can pass an allowlisted address as `owner` and the guard evaluates `true`. The actual token provider (`sender`) is never checked.

`SwapAllowlistExtension.beforeSwap` demonstrates the correct pattern — it names and checks `sender`: [3](#0-2) 

The asymmetry between the two extensions confirms the deposit extension checks the wrong address.

## Impact Explanation

Any address not on the deposit allowlist can call `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, "")`. The extension receives `owner = allowlistedAddress`, evaluates `allowedDepositor[pool][allowlistedAddress] = true`, and does not revert. The caller's tokens are pulled via the liquidity callback and the LP position is minted to the allowlisted address. Pools configured as restricted (KYC-gated, institutional-only, or exploit-mitigation allowlists) are fully open to any depositor. The pool admin's core security invariant — restricting who may supply tokens — is completely defeated.

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with a known allowlisted address as `owner`. The `allowedDepositor` mapping is public, so any observer can enumerate allowlisted addresses on-chain. No privileged access, flash loan, or special token behavior is required. The bypass is unconditional and repeatable by any address.

## Recommendation

Replace the unnamed first parameter with `sender` and check it instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

## Proof of Concept

1. Deploy a pool with `DepositAllowlistExtension` configured in `beforeAddLiquidity` order.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is **not** allowlisted.
3. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
4. `ExtensionCalling._beforeAddLiquidity` encodes `(sender=bob, owner=alice, ...)` and calls the extension.
5. Extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
6. Bob's tokens are pulled via the liquidity callback; LP position is minted to Alice.
7. Bob has deposited into a restricted pool without being on the allowlist. [4](#0-3)
