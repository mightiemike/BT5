### Title
No Return Of Funds If `DepositCollateral` Slow Mode Transaction Fails - (File: `core/contracts/Endpoint.sol`)

### Summary

`depositCollateralWithReferral` in `Endpoint.sol` transfers ERC20 tokens from the user to the clearinghouse before queuing a `DepositCollateral` slow mode transaction. If `processSlowModeTransaction` subsequently fails during `_executeSlowModeTransaction`, the catch block silently discards the failure and the user's funds are permanently locked in the clearinghouse with no subaccount credit and no refund path. The comment `// try return funds now removed` at line 226 of `Endpoint.sol` explicitly confirms the fund-return logic was stripped from this version.

### Finding Description

`depositCollateralWithReferral` calls `handleDepositTransfer` (defined in `EndpointStorage.sol`) which performs two sequential transfers: user → endpoint (`safeTransferFrom`) and endpoint → clearinghouse (`safeTransferTo`). Only after both transfers succeed does it enqueue a `SlowModeTx` of type `DepositCollateral`. [1](#0-0) [2](#0-1) 

When the slow mode queue is drained — either by the sequencer via `ExecuteSlowMode` or by any caller via `executeSlowModeTransaction` — `_executeSlowModeTransaction` wraps `processSlowModeTransaction` in a try/catch. The catch block only checks for out-of-gas conditions; it does not return the deposited funds to the user. The removed fund-return logic is explicitly acknowledged in the source: [3](#0-2) 

`processSlowModeTransactionImpl` handles `DepositCollateral` by calling `clearinghouse.depositCollateral(txn)`, which can revert for a variety of reasons (e.g., product delisted between deposit and execution, clearinghouse invariant violation, or any future revert condition in the clearinghouse). [4](#0-3) 

When that revert is caught, the user's tokens remain in the clearinghouse with no accounting entry crediting their subaccount, and no mechanism to recover them.

### Impact Explanation

A user who calls `depositCollateralWithReferral` loses their deposited ERC20 tokens permanently if the corresponding slow mode `DepositCollateral` transaction fails at execution time. The tokens are held by the clearinghouse but are never credited to the user's subaccount, and there is no refund path in the catch block. This is a direct, irreversible asset loss for the depositing user.

### Likelihood Explanation

The window between deposit and slow mode execution is hardcoded to three days (`SLOW_MODE_TX_DELAY`). [5](#0-4) 

During that window, any state change that causes `clearinghouse.depositCollateral` to revert (product delisting, clearinghouse upgrade, or any new validation added to the clearinghouse) will silently discard the transaction and strand the funds. The likelihood is low under normal operation but non-zero given the three-day delay and the fact that the catch block is explicitly designed to swallow reverts.

### Recommendation

Restore the `tryReturnFunds` logic in the catch block of `_executeSlowModeTransaction`. When the failed transaction is of type `DepositCollateral`, decode the transaction to recover the `sender`, `productId`, and `amount`, then transfer the tokens back from the clearinghouse (or endpoint) to the original depositor. This mirrors the fix described in the referenced patch (PR #116). [6](#0-5) 

### Proof of Concept

1. User calls `depositCollateralWithReferral(subaccount, productId, amount)`.
2. `handleDepositTransfer` moves `amount` tokens: user → endpoint → clearinghouse.
3. A `SlowModeTx` of type `DepositCollateral` is enqueued with a 3-day delay.
4. Before execution, the product is delisted or any other condition causes `clearinghouse.depositCollateral` to revert.
5. The sequencer (or any caller) triggers `_executeSlowModeTransaction`.
6. `try this.processSlowModeTransaction(...)` reverts; the catch block checks gas thresholds and then falls through with `// try return funds now removed` — no refund is issued.
7. The user's `amount` tokens remain in the clearinghouse, unaccounted, permanently inaccessible. [7](#0-6) [4](#0-3)
