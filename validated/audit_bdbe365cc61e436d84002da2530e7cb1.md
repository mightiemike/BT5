Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks position `owner` instead of actual caller/payer, allowing any non-allowlisted address to bypass the deposit gate — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards its first parameter (the actual caller/payer) and gates only the `owner` (position recipient) against the allowlist. Because `MetricOmmPool.addLiquidity` and `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` both accept an arbitrary `owner` address, any non-allowlisted address can bypass the deposit guard by naming an allowlisted address as the position owner. The allowlisted owner receives a funded LP position they did not pay for, and can withdraw the full token value via `removeLiquidity`.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter unnamed and never reads it, then checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` accepts an arbitrary `owner`, validates only that it is non-zero, and stores `msg.sender` as the payer in transient context:

```solidity
function addLiquidityExactShares(address pool, address owner, uint80 salt, ...) external payable override {
    _validateOwner(owner);  // only checks owner != address(0)
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
``` [3](#0-2) [4](#0-3) 

Inside `_addLiquidity`, the pool is called with `positionOwner = owner` (the supplied address), so the extension sees the allowlisted owner and passes: [5](#0-4) 

Token payment is pulled from the stored `payer` (the actual `msg.sender`/Bob), not from `owner`: [6](#0-5) 

`removeLiquidity` enforces `msg.sender == owner`, so the allowlisted owner (Alice) can withdraw the full token value: [7](#0-6) 

## Impact Explanation
The deposit allowlist is the pool admin's primary mechanism for restricting who may provide liquidity (KYC/AML compliance, curated LP sets, regulatory perimeters). With this bug, any non-allowlisted address can act as the economic depositor in a restricted pool without being on the allowlist. The allowlisted owner receives a funded LP position they did not pay for and can withdraw the full token value. The pool admin's access-control invariant — that only allowlisted addresses may deposit — is completely broken. This constitutes a broken core pool functionality causing loss of funds (the allowlisted owner receives unearned LP assets) and an admin-boundary break (unprivileged caller bypasses the deposit gate).

## Likelihood Explanation
No special privilege is required; any EOA or contract can call `pool.addLiquidity` or `liquidityAdder.addLiquidityExactShares` directly. Allowlisted addresses are publicly discoverable from on-chain `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. The `MetricOmmPoolLiquidityAdder` provides a clean, public entry point that explicitly separates `owner` from `msg.sender`. The bypass is reachable whenever a pool is deployed with `DepositAllowlistExtension` in a non-`allowAllDepositors` configuration, which is the entire purpose of the extension.

## Recommendation
Check the actual caller (first parameter) instead of `owner`:

```solidity
function beforeAddLiquidity(address caller, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][caller]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This ensures the address that actually pays tokens — not the address that receives the position — is the one gated by the allowlist.

## Proof of Concept
1. Deploy a pool with `DepositAllowlistExtension` configured (`allowAllDepositors[pool] = false`).
2. Pool admin calls `setAllowedToDeposit(pool, Alice, true)`. Bob is not allowlisted.
3. Bob calls `pool.addLiquidity(Alice, salt, deltas, callbackData, extensionData)` directly.
4. Pool calls `_beforeAddLiquidity(Bob, Alice, ...)` → extension checks `allowedDepositor[pool][Alice]` → `true` → no revert.
5. Pool calls `Bob.metricOmmModifyLiquidityCallback(amount0, amount1, ...)` → Bob pays tokens.
6. Alice's position is credited with the deposited shares.
7. Alice calls `pool.removeLiquidity(Alice, salt, deltas, ...)` and withdraws the full token value.
8. Bob has deposited into a restricted pool without being allowlisted; Alice received a funded position she did not pay for. The same bypass is reachable via `liquidityAdder.addLiquidityExactShares(pool, Alice, ...)` with Bob as `msg.sender`.
