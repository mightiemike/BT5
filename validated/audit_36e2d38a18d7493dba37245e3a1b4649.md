Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates the Router Address Instead of the Original User, Allowing Any User to Bypass a Curated Pool's Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `sender`, which is `msg.sender` of the pool's `swap()` call — the immediate caller. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the original user. If a pool admin allowlists the router to enable router-mediated swaps for their curated users, every unprivileged user can bypass the per-user gate by routing through the router, completely defeating the allowlist policy.

## Finding Description
`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct) and `sender` is the first argument forwarded by the pool — which is `msg.sender` of the pool's own `swap()` call.

`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient, ...
);
``` [2](#0-1) 

`ExtensionCalling._beforeSwap` then encodes and forwards this as the `sender` argument to the extension: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly, making itself `msg.sender` at the pool:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [4](#0-3) 

So the extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The original user's identity is lost.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly ignores `sender` and gates by `owner` (the economic actor), demonstrating the intended design pattern:

```solidity
function beforeAddLiquidity(address, address owner, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
``` [5](#0-4) 

The swap extension has no equivalent originator field — its `beforeSwap` signature exposes only `sender` (immediate caller) and an unnamed `recipient` (second argument, ignored), with no dedicated payer/originator identity. [6](#0-5) 

## Impact Explanation
A pool admin who wants to allow router-mediated swaps for their allowlisted users has only one option: `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, every user — including those not individually allowlisted — can call `router.exactInputSingle(pool, ...)` and the extension passes, because it sees `sender = router` which is allowlisted. The per-user curation policy is completely defeated. Any non-allowlisted user can trade on a pool designed to be permissioned (e.g., KYC-gated, institutional, or regulatory-compliant pools). This constitutes broken core pool functionality: the allowlist mechanism, a primary pool feature for permissioned pools, is rendered entirely ineffective.

## Likelihood Explanation
The trigger is a single, natural admin action: allowlisting the router so that allowlisted users can use the standard periphery. This is the expected operational pattern for any pool that wants to support the official router. The attacker needs no special access — any EOA can call `MetricOmmSimpleRouter.exactInputSingle` with the target pool. The router is a factory-validated contract, so `onlyPool` guards on the extension are satisfied. The bypass is fully unprivileged once the router is allowlisted, and is repeatable indefinitely.

## Recommendation
Gate the swap allowlist by the original user rather than the immediate caller. The correct fix mirrors `DepositAllowlistExtension`: the pool should pass a dedicated payer/originator identity to `beforeSwap`. The router already stores the original `msg.sender` in transient storage via `_setNextCallbackContext` — this originator could be forwarded as an authenticated field in `extensionData`, and the extension should decode and verify it. Alternatively, the `beforeSwap` hook interface should be extended with an `originator` parameter analogous to `owner` in `beforeAddLiquidity`, populated by the pool from a trusted source rather than raw `msg.sender`.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for allowlisted users.
3. Non-allowlisted attacker calls `router.exactInputSingle({pool: pool, recipient: attacker, ...})`.
4. Router calls `pool.swap(attacker, ...)` — `msg.sender` at the pool is the router.
5. Pool calls `_beforeSwap(msg.sender=router, recipient=attacker, ...)`.
6. `SwapAllowlistExtension.beforeSwap(sender=router, ...)` checks `allowedSwapper[pool][router]` → `true` → passes.
7. Attacker's swap executes on a pool that should have blocked them.

The attacker is never individually allowlisted; the bypass is entirely through the router intermediary. The exact wrong value is the extension's boolean decision: `allowedSwapper[pool][router]` is evaluated instead of `allowedSwapper[pool][attacker]`, causing the gate to return `true` when it should revert with `NotAllowedToSwap`.
