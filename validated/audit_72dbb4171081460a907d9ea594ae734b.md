### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Unauthorized Depositor to Bypass the Allowlist Guard — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` parameter (the actual token-providing caller) and performs its allowlist check on `owner` (the LP-share recipient). Because `owner` is caller-controlled, any unprivileged address can bypass the deposit restriction by naming an allowlisted address as `owner`.

---

### Finding Description

In `MetricOmmPool.addLiquidity`, the pool calls the extension with:

```
_beforeAddLiquidity(msg.sender /*sender*/, owner /*owner*/, salt, deltas, extensionData)
```

`sender` = `msg.sender` — the address that will provide tokens via the `IMetricOmmSwapCallback`. `owner` = an arbitrary address supplied by the caller that receives the LP shares. [1](#0-0) 

Inside the extension, the `sender` slot is unnamed and ignored. The guard reads only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

Because `owner` is a free parameter chosen by the caller, any address can pass the guard by supplying an allowlisted address as `owner`. The actual depositor (`sender`) is never validated.

This is the direct analog of the external `calculateMaxAllocation` bug: there, the wrong variable (`userMaxAlloc`) was returned instead of the cap (`maxAllocation`); here, the wrong actor (`owner`) is checked instead of the actual depositor (`sender`).

The `ExtensionCalling._beforeAddLiquidity` wrapper confirms that `sender` is always `msg.sender` of the pool call and is passed as the first argument — it is structurally available to the extension but discarded. [3](#0-2) 

---

### Impact Explanation

The deposit allowlist is the sole access-control gate for `addLiquidity` on pools that configure it. Bypassing it allows any unprivileged address to:

1. **Add liquidity to a permissioned pool**, altering bin token balances (`t0`, `t1`) and total shares.
2. **Manipulate the inputs to `OracleValueStopLossExtension`** — per-share metrics are computed from live bin balances; injecting liquidity can suppress a breach or manufacture one, causing the stop-loss to fire or fail to fire on the next swap.
3. **Dilute existing LP returns** in pools where the admin intended to cap participation.

The LP shares are credited to the attacker-chosen `owner`, so the attacker does not profit directly, but the pool's access-control invariant is fully broken and pool state can be manipulated with fund-impacting consequences for existing LPs.

---

### Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with `owner` set to any address that appears in `allowedDepositor`. Allowlisted addresses are observable on-chain (via `AllowedToDepositSet` events or direct storage reads). No special role, flash loan, or multi-step setup is needed. Any unprivileged EOA or contract can trigger this immediately after pool deployment.

---

### Recommendation

Name and check `sender` (the actual token provider) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

Note that `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the first named parameter); `DepositAllowlistExtension` should follow the same pattern. [4](#0-3) 

---

### Proof of Concept

```
Setup:
  - Pool deployed with DepositAllowlistExtension.
  - Admin calls setAllowedToDeposit(pool, alice, true).
  - bob is NOT allowlisted.

Attack:
  1. bob calls pool.addLiquidity(
         owner      = alice,   // allowlisted address
         salt       = 0,
         deltas     = <valid deltas>,
         callbackData = ...,
         extensionData = ""
     );

  2. Pool calls extension.beforeAddLiquidity(
         sender = bob,    // ignored — unnamed parameter
         owner  = alice   // checked — passes because alice is allowlisted
     );

  3. Guard passes. bob's callback provides tokens.
     LP shares are minted to alice.

Result:
  bob (unauthorized) has deposited into the restricted pool.
  Pool bin balances are altered without admin consent.
  The deposit allowlist guard is fully bypassed.
```
