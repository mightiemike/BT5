### Title
Declare Transactions Bypass `max_l2_gas_amount` Gateway Admission Check via Empty `if-else` Branch — (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

In `StatelessTransactionValidator::validate_resource_bounds`, an `if-else` construct with an intentionally empty `if` body for `RpcTransaction::Declare` causes the `max_l2_gas_amount` upper-bound check to be silently skipped for every Declare transaction. Any unprivileged user can submit a Declare transaction with `l2_gas.max_amount` set to an arbitrarily large value (e.g., `u64::MAX`) and have it accepted by the gateway, bypassing the admission-control invariant that is enforced for Invoke and DeployAccount transactions.

### Finding Description

`validate_resource_bounds` enforces two resource-bound checks when `validate_resource_bounds` config is enabled:

1. Total fee must be non-zero.
2. `l2_gas.max_price_per_unit` must meet the minimum gas price.
3. `l2_gas.max_amount` must not exceed `config.max_l2_gas_amount`.

Check 3 is implemented as:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // empty — check is skipped
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The `if` branch for `Declare` is empty; the `else if` that enforces the ceiling is therefore unreachable for Declare transactions. The TODO comment confirms this is a known gap. The structural pattern is identical to the external report: an `if-else` where satisfying the first condition (being a Declare transaction) silently bypasses a check that should apply independently.

The `validate` entry point calls `validate_resource_bounds` unconditionally for all transaction types: [2](#0-1) 

So the bypass is reachable on every Declare submission through the public RPC endpoint (`add_declare_transaction`).

### Impact Explanation

A Declare transaction with `l2_gas.max_amount = u64::MAX` passes all stateless gateway checks and is forwarded to the mempool. The gateway's admission-control invariant — that no transaction may claim more L2 gas than `max_l2_gas_amount` — is violated for the Declare type. This matches the allowed impact:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

Downstream effects include: the mempool holds a transaction whose declared resource bound is orders of magnitude above any block limit; the batcher's bouncer must evaluate and discard it; and an attacker can flood the mempool with such transactions at low cost (Declare fees are still charged, but the gas-amount ceiling that limits per-transaction resource claims is absent).

### Likelihood Explanation

Any unprivileged user with RPC access can craft a `BROADCASTED_DECLARE_TXN_V3` with `resource_bounds.l2_gas.max_amount` set to `0xFFFFFFFFFFFFFFFF`. No special privilege, key material, or prior state is required. The gateway's stateless validator is the only admission gate before the mempool; the stateful validator does not re-check this field.

### Recommendation

Replace the empty `if-else` with two independent `if` statements so that the `max_l2_gas_amount` check applies to Declare transactions as well:

```rust
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(…MaxGasPriceTooLow…);
}

// Apply to all tx types once the TODO is resolved:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(…MaxGasAmountTooHigh…);
}
```

This mirrors the fix recommended in the external report: replace `if-else` with independent `if` statements so that each check fires based solely on its own condition, not on whether a prior branch was taken.

### Proof of Concept

1. Construct a valid `RpcDeclareTransaction::V3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and all other fields valid (correct Sierra class, valid signature, nonce, etc.).
2. Submit via `starknet_addDeclareTransaction` to a node running with `validate_resource_bounds: true`.
3. Observe: the gateway returns a transaction hash (success) rather than `MaxGasAmountTooHigh`.
4. Confirm: submitting the same transaction as `RpcInvokeTransaction::V3` with the same `l2_gas.max_amount` is rejected with `StatelessTransactionValidatorError::MaxGasAmountTooHigh`.

The divergence between Declare and Invoke handling at lines 79–85 is the sole root cause. [3](#0-2)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
