Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`, who provides the tokens) and instead validates `owner` (the LP-position beneficiary). Because `owner` is caller-controlled, any address not on the allowlist can bypass the guard by passing an allowlisted address as `owner`, voiding the pool admin's access-control restriction entirely.

## Finding Description
`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)` at line 191, forwarding the actual depositor as `sender` and the LP-position beneficiary as `owner`. [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes the call as `(sender, owner, salt, deltas, extensionData)`, so the extension receives both values correctly. [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` declares its first parameter unnamed (`address,`), discarding `sender`, and then checks `allowedDepositor[msg.sender][owner]` — where `msg.sender` is the pool and `owner` is the attacker-supplied beneficiary address. [3](#0-2) 

The contract's own NatSpec states its purpose is to gate `addLiquidity` **by depositor address**, confirming `sender` is the intended subject of the check. [4](#0-3) 

There are no other guards in the `addLiquidity` path that enforce the allowlist on `sender`. The `removeLiquidity` path enforces `msg.sender == owner` but `addLiquidity` has no such constraint, making `owner` freely specifiable by any caller. [5](#0-4) 

## Impact Explanation
The deposit allowlist is the pool admin's primary access-control mechanism for restricting which addresses may provide liquidity (e.g., for KYC/compliance or curated LP sets). Because the check is on the caller-controlled `owner` rather than the actual depositor `sender`, the restriction is unconditionally bypassable. A non-allowlisted address can deposit tokens into a restricted pool, with LP shares credited to an allowlisted `owner`. This constitutes a broken core pool functionality / admin-boundary break: the pool admin's restriction is silently voided, unauthorized liquidity enters the pool, and the non-allowlisted depositor's tokens are accepted in violation of the allowlist invariant.

## Likelihood Explanation
The bypass requires no special privilege. Any EOA or contract can call `addLiquidity` on a pool configured with `DepositAllowlistExtension`. The attacker only needs to know one allowlisted address (e.g., the pool admin, readable from the factory). The bypass is unconditional whenever `sender != owner` and `allowedDepositor[pool][owner] == true`. It is repeatable at will with zero cost beyond gas.

## Recommendation
Replace the unnamed first parameter with `sender` and validate it instead of `owner`:

```solidity
// Before
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }

// After
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
```

## Proof of Concept
1. Deploy a pool with `DepositAllowlistExtension` configured in `BEFORE_ADD_LIQUIDITY_ORDER`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` — `alice` is allowlisted; `bob` is not.
3. `bob` calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. `_beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)` is invoked; the extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
5. `LiquidityLib.addLiquidity` executes: `bob`'s tokens are pulled via the swap callback; LP shares are credited to `alice`.
6. `bob` (non-allowlisted) has successfully deposited into a restricted pool, violating the allowlist invariant. [3](#0-2) [6](#0-5)
