### Title
`ERC20Helper.safeTransfer` / `safeTransferFrom` Do Not Verify Token Address Has Contract Code, Enabling Phantom Deposits — (File: `core/contracts/libraries/ERC20Helper.sol`)

---

### Summary

Nado's `ERC20Helper` library implements `safeTransfer` and `safeTransferFrom` using low-level `.call()` without verifying that the target token address contains deployed contract code. If a product is registered with a token address that has no code at call time, both functions silently succeed, allowing an attacker to receive a credited spot balance in the clearinghouse without transferring any real tokens.

---

### Finding Description

`ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` perform a low-level `.call()` to the token address and check only `success && (data.length == 0 || abi.decode(data, (bool)))`: [1](#0-0) [2](#0-1) 

When the target address has no deployed code, the EVM low-level call returns `(true, "")`. The require condition evaluates as `true && (true || ...)` → passes. No revert occurs. No tokens move.

This flaw propagates directly into `EndpointStorage.handleDepositTransfer`, which is the core deposit path: [3](#0-2) 

`handleDepositTransfer` is invoked from `Endpoint.depositCollateralWithReferral` for every user deposit: [4](#0-3) 

The token address used is fetched from `SpotEngine.getToken(productId)`, which returns `configs[productId].token` — a value set by the owner via `addOrUpdateProduct`. If this address has no code at deposit time (e.g., the owner pre-registered a product before the token contract was deployed, or the token self-destructed), both the `safeTransferFrom` (user → endpoint) and `safeTransferTo` (endpoint → clearinghouse) calls silently succeed. A slow-mode `DepositCollateral` transaction is then queued and, when processed by the sequencer, `Clearinghouse.depositCollateral` credits the attacker's subaccount balance in `SpotEngine`: [5](#0-4) 

The same missing check exists in `DirectDepositV1.safeTransfer`: [6](#0-5) 

---

### Impact Explanation

**Impact: High.**

An attacker who deposits against a product whose token address has no code receives a positive `SpotEngine` balance without transferring any real tokens. Once the real token is deployed and legitimate users deposit actual funds into the clearinghouse, the attacker can submit a withdrawal and drain those real tokens. Honest depositors lose their funds with no recourse. The corrupted state is the `SpotEngine` balance mapping and the clearinghouse's actual token holdings — the on-chain balance diverges from the sum of credited deposits. [7](#0-6) 

---

### Likelihood Explanation

**Likelihood: Low.**

The precondition is that a product must be registered with a token address that has no code at the time of deposit. This can occur if:
1. The protocol owner pre-registers a product before the token contract is deployed (a common operational pattern when using CREATE2 or deterministic deployers).
2. A registered token self-destructs (rare but possible on some chains).

An attacker monitoring the mempool or on-chain product registrations can detect this window and act before the token is deployed.

---

### Recommendation

Add a contract-code-length check inside both `ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom`, mirroring OpenZeppelin's `Address.verifyCallResultFromTarget`:

```solidity
function safeTransfer(IERC20Base self, address to, uint256 amount) internal {
    require(address(self).code.length > 0, "ERC20Helper: token is not a contract");
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}

function safeTransferFrom(IERC20Base self, address from, address to, uint256 amount) internal {
    require(address(self).code.length > 0, "ERC20Helper: token is not a contract");
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.transferFrom.selector, from, to, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Apply the same fix to `DirectDepositV1.safeTransfer`. [8](#0-7) [9](#0-8) 

---

### Proof of Concept

**Setup:**
- Owner calls `SpotEngine.addOrUpdateProduct(productId, ..., Config({ token: predictedTokenAddr, ... }))` where `predictedTokenAddr` is a CREATE2-deterministic address not yet deployed.
- Attacker observes the registration on-chain.

**Attack:**
1. Attacker calls `Endpoint.depositCollateralWithReferral(attackerSubaccount, productId, largeAmount, "")`.
2. `handleDepositTransfer(IERC20Base(predictedTokenAddr), attacker, largeAmount)` is called.
3. `safeTransferFrom` low-level calls `predictedTokenAddr` (no code) → returns `(true, "")` → require passes. No tokens move.
4. `safeTransferTo` low-level calls `predictedTokenAddr` (no code) → returns `(true, "")` → require passes. No tokens move.
5. A slow-mode `DepositCollateral` transaction is queued for `attackerSubaccount`.
6. Sequencer processes it: `Clearinghouse.depositCollateral` credits `largeAmount` (scaled) to `attackerSubaccount` in `SpotEngine`.
7. Owner deploys the real token at `predictedTokenAddr`. Legitimate users deposit real tokens.
8. Attacker submits a withdrawal for `largeAmount`, draining the real tokens deposited by legitimate users. [3](#0-2) [10](#0-9)
