Audit Report

## Title
Permanently Broken Extension Irreversibly Locks LP Funds and Bricks All Pool Operations — (`metric-core/contracts/ExtensionCalling.sol`)

## Summary

`CallExtension.callExtension` performs a raw low-level call and unconditionally reverts on failure with no try/catch isolation. All seven extension slot addresses and their order bitmaps are stored as `immutable` constructor arguments with no setter, meaning a permanently reverting extension in `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` will permanently block `removeLiquidity`. Because `removeLiquidity` carries no `whenNotPaused` guard, even a fully paused pool cannot allow LPs to exit.

## Finding Description

`CallExtension.callExtension` (lines 8–17) performs a raw `.call` and on failure either bubbles the revert reason or throws `ExtensionCallFailed()` — no isolation:

```solidity
(bool success, bytes memory result) = extension.call(data);
if (!success) {
    if (result.length > 0) {
        assembly ("memory-safe") { revert(add(result, 32), mload(result)) }
    }
    revert ExtensionCallFailed();
}
```

`_callExtensionsInOrder` (lines 75–86 of `ExtensionCalling.sol`) loops over every packed 3-bit slot and calls `callExtension` for each non-zero index with no per-iteration isolation:

```solidity
while (true) {
    uint256 extensionIndex = order & 0x7;
    if (extensionIndex == 0) break;
    address extension = _extensionAddress(extensionIndex);
    if (extension == address(0)) revert PanicEmptyExtension();
    CallExtension.callExtension(extension, data);
    order >>= 3;
}
```

`_beforeRemoveLiquidity` and `_afterRemoveLiquidity` delegate unconditionally to `_callExtensionsInOrder` (lines 118–147 of `ExtensionCalling.sol`).

`removeLiquidity` (lines 199–212 of `MetricOmmPool.sol`) calls both hooks but carries only `nonReentrant` — no `whenNotPaused`:

```solidity
function removeLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)
    external
    nonReentrant(PoolActions.REMOVE_LIQUIDITY)   // ← no whenNotPaused
    returns (uint256 amount0Removed, uint256 amount1Removed)
```

By contrast, `swap` (line 224) carries `whenNotPaused`. `setPause` (lines 454–461) only updates `pauseLevel`, which is only checked by `_checkNotPaused` inside `whenNotPaused`, a modifier not applied to `removeLiquidity`.

All seven extension addresses (`EXTENSION_1`…`EXTENSION_7`) and all six order bitmaps are `immutable` (lines 17–35 of `ExtensionCalling.sol`). The only factory-callable setters on the pool are `collectFees`, `setPoolFees`, `setPause`, `setBinAdditionalFees`, and `setPriceProvider` — none can replace or nullify an extension slot.

**Exploit path:**
1. Pool is deployed with `OracleValueStopLossExtension` (or any oracle-dependent extension) registered in `BEFORE_REMOVE_LIQUIDITY_ORDER`.
2. LPs add liquidity normally.
3. The oracle contract used by the extension is deprecated/upgraded and now reverts on every call.
4. Any LP calls `removeLiquidity(owner, salt, deltas, extensionData)`.
5. `_beforeRemoveLiquidity` → `_callExtensionsInOrder` → `CallExtension.callExtension(brokenExtension, data)` → extension reverts → `callExtension` reverts → `removeLiquidity` reverts.
6. Pool admin calls `setPause(1)` — no effect on `removeLiquidity` (no `whenNotPaused` guard).
7. No other admin function can replace `EXTENSION_1`…`EXTENSION_7` or the order bitmaps.
8. LP funds are permanently locked.

## Impact Explanation

If any extension registered in a remove-liquidity hook permanently reverts, LP principal is permanently inaccessible. The same failure in swap hooks permanently bricks trading. Because extensions are immutable and `setPause` does not bypass extension calls on `removeLiquidity`, there is no recovery path. This directly matches the allowed impact: *broken core pool functionality causing loss of funds or unusable withdraw/liquidity flows*.

| Broken hook order | Blocked operation | Fund impact |
|---|---|---|
| `BEFORE_REMOVE_LIQUIDITY_ORDER` | `removeLiquidity` | LP principal permanently locked |
| `AFTER_REMOVE_LIQUIDITY_ORDER` | `removeLiquidity` | LP principal permanently locked |
| `BEFORE_SWAP_ORDER` / `AFTER_SWAP_ORDER` | `swap` | Pool unusable |
| `BEFORE_ADD_LIQUIDITY_ORDER` | `addLiquidity` | New deposits blocked |

## Likelihood Explanation

Extensions are external contracts that may depend on oracles (`OracleValueStopLossExtension`, `PriceVelocityGuardExtension`), allowlist registries (`DepositAllowlistExtension`, `SwapAllowlistExtension`), or other mutable state. Realistic triggers include: (1) an upgradeable extension contract receives a breaking upgrade; (2) an oracle dependency becomes permanently stale or its contract is deprecated/self-destructed; (3) an allowlist registry contract is paused or decommissioned by its own admin; (4) the extension contract is hacked and self-destructed. None of these require any action by the pool admin or an attacker against the pool itself — they are ordinary lifecycle events for external contracts. The pool has no circuit-breaker for this scenario.

## Recommendation

1. **Wrap each extension call in try/catch** inside `_callExtensionsInOrder`. On failure, either revert with a clear error (preserving current strict behavior) or allow governance to disable the broken slot.
2. **Add a mutable emergency bypass** — a factory-controlled flag per extension slot that skips that slot's call, allowing the pool admin to neutralize a broken extension without redeploying the pool.
3. **Extend `whenNotPaused` (or a separate emergency mode) to skip extension calls on `removeLiquidity`**, so LPs can always exit even if an extension is broken.

## Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension` registered in `BEFORE_REMOVE_LIQUIDITY_ORDER`.
2. LPs call `addLiquidity` — succeeds normally.
3. Deprecate/upgrade the oracle contract so it reverts on every call.
4. Any LP calls `removeLiquidity(owner, salt, deltas, extensionData)`.
5. Execution path: `removeLiquidity` → `_beforeRemoveLiquidity` → `_callExtensionsInOrder` → `CallExtension.callExtension(brokenExtension, data)` → extension reverts → `callExtension` reverts → `removeLiquidity` reverts.
6. Pool admin calls `setPause(1)` — `removeLiquidity` has no `whenNotPaused` guard, so pause has no effect.
7. No admin function exists to replace `EXTENSION_1`…`EXTENSION_7` or the order bitmaps.
8. LP funds are permanently locked with no on-chain recovery path.
