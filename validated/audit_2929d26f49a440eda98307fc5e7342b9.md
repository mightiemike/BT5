### Title
Checks-Effects-Interactions Violation in `withdrawCollateral()` Enables Reentrancy Double-Withdrawal via `executeSlowModeTransaction()` — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral()` performs the token transfer (external call) **before** updating the on-chain balance and running the health check. No `nonReentrant` guard exists anywhere in the codebase. A reentrant call through the publicly accessible `Endpoint.executeSlowModeTransaction()` can re-enter `withdrawCollateral()` while the balance is still at its pre-withdrawal value, allowing a double withdrawal against a single balance.

---

### Finding Description

`Clearinghouse.withdrawCollateral()` follows this order:

1. **Transfer tokens** (external call) — `handleWithdrawTransfer(token, sendTo, amount, idx)` [1](#0-0) 
2. **Update balance** — `spotEngine.updateBalance(productId, sender, amountRealized)` [2](#0-1) 
3. **Health check** — `require(getHealth(sender, healthType) >= 0, ...)` [3](#0-2) 

This is a textbook CEI (Checks-Effects-Interactions) violation. The interaction (token transfer) precedes the effect (balance decrement) and the final check (health gate).

`Clearinghouse.handleWithdrawTransfer()` first sends tokens to `withdrawPool`, then calls `withdrawPool.submitWithdrawal()`, which calls `token.safeTransfer(sendTo, amount)` directly to the recipient: [4](#0-3) [5](#0-4) 

If `sendTo` is a contract holding an ERC-777 `tokensReceived` hook (or any token with a transfer callback), the hook fires **while the sender's balance is still at its original value**.

The reentrant entry point is `Endpoint.executeSlowModeTransaction()`, which is **publicly callable with no access control**: [6](#0-5) 

This function reads `slowModeConfig` into memory, executes the queued transaction, and only writes the updated config back **after** the external call returns. If the attacker has a pending slow-mode `WithdrawCollateral` queued, the reentrant call processes it while the outer call's balance decrement has not yet been committed to storage.

The reentrant path is:

```
tokensReceived hook
  → endpoint.executeSlowModeTransaction()
    → _executeSlowModeTransaction()
      → this.processSlowModeTransaction()          // external call to self
        → _delegatecallEndpointTx(processSlowModeTransactionImpl)
          → clearinghouse.withdrawCollateral()     // onlyEndpoint passes (msg.sender == Endpoint)
``` [7](#0-6) [8](#0-7) 

The `onlyEndpoint` modifier on `withdrawCollateral` is satisfied because `processSlowModeTransaction` is called via `this.processSlowModeTransaction(...)`, making `msg.sender` the Endpoint contract itself. [9](#0-8) 

There is **zero reentrancy protection** anywhere in the codebase — no `nonReentrant` modifier, no `ReentrancyGuard` import, no mutex storage variable. [10](#0-9) 

---

### Impact Explanation

**Concrete state delta corrupted:** `spotEngine` balance for the victim subaccount is decremented twice (once by the reentrant slow-mode call, once by the outer call), while the protocol's token reserve is drained by `2 × amount`. The attacker receives `2 × amount` tokens but their on-chain balance ends at `balance − 2 × amount`.

If the attacker holds sufficient other collateral (e.g., ETH or perp positions) such that `getHealth(sender, INITIAL) >= 0` still holds after the double decrement, **both health checks pass** and the transaction completes successfully. The attacker has effectively borrowed `amount` tokens against their other collateral through an unintended reentrancy path — bypassing any normal borrow/leverage flow and draining the protocol's token reserve beyond what the subaccount's token balance permits.

---

### Likelihood Explanation

**Required conditions:**

1. A supported collateral token that implements ERC-777 or any transfer-hook pattern (e.g., USDC on some chains, or any future listed token with hooks). The protocol uses a generic `IERC20Base` interface with no hook-exclusion guard. [11](#0-10) 
2. The attacker must have a pending slow-mode `WithdrawCollateral` transaction in the queue (trivially achievable by calling `submitSlowModeTransaction` in advance).
3. The attacker's `sendTo` address must be a contract that fires the reentrant call in its receive hook.
4. The attacker must hold enough other collateral to pass the double-decremented health check.

Conditions 2–4 are fully attacker-controlled. Condition 1 depends on the token list. Likelihood is **medium** given the protocol's intent to support multiple assets.

---

### Recommendation

Apply the standard CEI pattern in `Clearinghouse.withdrawCollateral()`: update the balance and run the health check **before** calling `handleWithdrawTransfer`. Additionally, add a `nonReentrant` modifier (OpenZeppelin `ReentrancyGuardUpgradeable`) to `withdrawCollateral`, `executeSlowModeTransaction`, and `depositCollateralWithReferral` to eliminate the entire class of reentrancy risk.

---

### Proof of Concept

```
Setup:
  - Token T is ERC-777.
  - Attacker subaccount S has: 100 T balance + 500 USDC worth of ETH collateral.
  - Attacker pre-queues a slow-mode WithdrawCollateral(S, T, 100) → slowModeTxs[k].
  - Attacker sets sendTo = AttackerContract (has tokensReceived hook).

Attack:
  1. Sequencer submits WithdrawCollateral(S, T, 100, AttackerContract) via submitTransactionsChecked.
  2. Clearinghouse.withdrawCollateral() is entered.
  3. handleWithdrawTransfer → token.safeTransfer(withdrawPool, 100)
                            → withdrawPool.submitWithdrawal → token.safeTransfer(AttackerContract, 100)
  4. ERC-777 tokensReceived fires on AttackerContract.
  5. AttackerContract calls endpoint.executeSlowModeTransaction().
  6. Slow-mode WithdrawCollateral(S, T, 100) is processed.
  7. clearinghouse.withdrawCollateral() re-entered:
       - balance[S][T] still = 100 (outer call hasn't decremented yet)
       - transfers 100 T to attacker's EOA
       - balance[S][T] -= 100  →  balance = 0
       - getHealth(S, INITIAL) = 0 + ETH_value >= 0  ✓
  8. Control returns to outer withdrawCollateral (step 2):
       - balance[S][T] -= 100  →  balance = -100
       - getHealth(S, INITIAL) = -100*price_T + ETH_value >= 0  ✓  (ETH covers it)
  9. Transaction succeeds.

Result: Attacker received 200 T, balance[S][T] = -100 (unauthorized debt).
        Protocol token reserve drained by 200 T instead of 100 T.
```

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
