### Title
Deprecated `verify_transaction_inclusion()` Remains Publicly Callable, Bypassing Coinbase Proof Validation Added in v2 - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion()` was deprecated in favour of `verify_transaction_inclusion_v2()`, which adds a coinbase Merkle proof check specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability. However, `verify_transaction_inclusion()` remains a fully public NEAR contract method with no caller restriction. Any unprivileged NEAR account can invoke it directly, completely bypassing the coinbase proof validation that `verify_transaction_inclusion_v2()` enforces.

### Finding Description

`verify_transaction_inclusion_v2()` was introduced to close a known proof-forgery vector. Before delegating to the inner function, it requires the caller to supply a valid coinbase Merkle proof that independently anchors the tree depth:

```
verify_transaction_inclusion_v2()
  ├─ [guard] coinbase merkle proof must match block's merkle_root   ← lines 358-365
  └─ calls verify_transaction_inclusion()                           ← line 368
```

`verify_transaction_inclusion()` itself performs no such depth-anchoring check. It only recomputes the Merkle root from the supplied `tx_id` and `merkle_proof` and compares it to the stored `merkle_root`. Because the function is `pub fn` with only a `#[pause]` attribute (no caller restriction), any NEAR account can call it directly, skipping the coinbase guard entirely. [1](#0-0) [2](#0-1) 

The `#[deprecated]` Rust attribute is a **compiler lint only**; it has no effect on NEAR contract method visibility or on-chain accessibility. The method is still exported as a callable NEAR contract function.

### Impact Explanation

The 64-byte transaction attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions, referenced in the contract's own deprecation notice) allows an attacker to craft a 64-byte blob whose double-SHA256 hash equals an internal Merkle tree node. By presenting this blob as a `tx_id` with a carefully constructed `merkle_proof`, the attacker can make `verify_transaction_inclusion()` return `true` for a transaction that was never included in any Bitcoin block.

Any downstream NEAR contract that calls `verify_transaction_inclusion()` directly (or that a user routes through this entry point) will receive a forged `true` result. This corrupts the SPV proof result — the core security guarantee of the light client — and can be used to deceive bridge contracts, payment verifiers, or any other consumer that trusts the return value. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

The entry path requires no privileges whatsoever. Any NEAR account can call `verify_transaction_inclusion` directly as a view or change call. The attack technique is publicly documented and the contract's own deprecation notice explicitly names the vulnerability class. The `#[deprecated]` attribute provides zero on-chain enforcement. [1](#0-0) 

### Recommendation

Remove `verify_transaction_inclusion()` from the public NEAR contract API entirely. Since `verify_transaction_inclusion_v2()` already calls it internally via `#[allow(deprecated)]`, the inner logic can be refactored into a private helper function. If backward compatibility must be preserved, the public `verify_transaction_inclusion()` method should at minimum be gated so it is not callable by external accounts (e.g., `#[private]` or an explicit `assert_eq!(env::predecessor_account_id(), env::current_account_id())`).

### Proof of Concept

1. Deploy the contract with a known set of Bitcoin headers.
2. Identify a block `B` in the mainchain with a Merkle tree of depth ≥ 2.
3. Craft a 64-byte payload `P` such that `SHA256d(P)` equals an internal Merkle node `N` at depth 1.
4. Construct a `merkle_proof` that, starting from `N`, correctly reconstructs the Merkle root of block `B`.
5. Call `verify_transaction_inclusion` directly (not `_v2`) with `tx_id = SHA256d(P)`, `tx_block_blockhash = B`, and the crafted `merkle_proof`.
6. The function returns `true` — a forged proof of inclusion for a transaction that does not exist.

Step 5 is the bypass: calling `verify_transaction_inclusion_v2` with the same inputs would fail at the coinbase proof guard (lines 358–365), but the direct call to `verify_transaction_inclusion` skips that guard entirely. [5](#0-4) [6](#0-5)
