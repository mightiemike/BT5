### Title
`only_query` Flag Hardcoded to `false` in `BroadcastedTransaction`→`ExecutableTransactionInput` Conversion Causes `starknet_simulateTransactions` to Apply Real-Execution Validation Rules Instead of Query-Mode Rules - (File: crates/apollo_rpc/src/v0_8/api/mod.rs)

### Summary

The `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` implementation unconditionally sets `only_query = false` for every transaction type submitted to `starknet_simulateTransactions` and `starknet_estimateFee`. The Starknet protocol requires that simulation/estimation calls set the query bit (bit 128) in the transaction version so that account contracts can distinguish simulation from real execution and, for example, skip signature verification. Because `only_query` is always `false`, the account contract's `__validate__` entry point receives the plain version (e.g. `0x3`) instead of the query version (e.g. `0x100000000000000000000000000000003`), causing simulation to apply real-execution validation rules and return authoritative-looking wrong results.

### Finding Description

`signed_tx_version` sets bit 128 of the transaction version when `only_query = true`: [1](#0-0) 

This signed version is what the account contract observes through the `get_execution_info` syscall during `__validate__`. Account contracts that check whether bit 128 is set to allow simulation without a valid signature (a standard Starknet pattern) will see the plain version and therefore require a real signature.

The conversion that feeds `starknet_simulateTransactions` hardcodes `only_query = false` for every variant, with an explicit TODO acknowledging the value is wrong: [2](#0-1) 

This `TryFrom` impl is the sole conversion path used by `simulate_transactions`: [3](#0-2) 

The resulting `ExecutableTransactionInput` carries `only_query = false` all the way into `to_blockifier_tx`, which builds the `ExecutionFlags` passed to the blockifier: [4](#0-3) 

The `ExecutionFlags::only_query` field is then read by `TransactionInfoCreatorInner::create_tx_info` and propagated into `CommonAccountFields::only_query`, which `TransactionInfo::signed_version()` uses to decide whether to set bit 128: [5](#0-4) 

### Impact Explanation

Any account contract that inspects the transaction version inside `__validate__` to detect query mode (the standard way to allow fee-free or signature-free simulation) will observe the plain version and apply real-execution rules. For such contracts, `starknet_simulateTransactions` with `validate = true` (i.e., without `SkipValidate`) returns a validation failure even though the identical transaction with a valid signature would succeed on-chain. The RPC response is authoritative-looking and will mislead callers into believing their transaction is invalid. The same path is used by `starknet_estimateFee`, so fee estimates for these accounts are also wrong (the call errors out instead of returning a fee).

This matches the allowed impact: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

The query-bit pattern is documented in the Starknet specification and is used by account contracts that want to support gasless simulation. Any integrator or wallet that calls `starknet_simulateTransactions` without `SkipValidate` against such an account will observe the wrong result. The trigger requires no privilege: a normal RPC call with a well-formed transaction is sufficient.

### Recommendation

Remove the hardcoded `false` and derive `only_query` from the transaction version field that the caller supplies. For `BroadcastedTransaction` variants, the Starknet spec encodes the query bit in the `version` field (bit 128 set). Extract it before constructing `ExecutableTransactionInput`:

```rust
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        match value {
            BroadcastedTransaction::DeployAccount(tx) => {
                let only_query = tx.version().has_query_bit(); // derive from version
                Ok(Self::DeployAccount(tx.into(), only_query))
            }
            BroadcastedTransaction::Invoke(tx) => {
                let only_query = tx.version().has_query_bit();
                Ok(Self::Invoke(tx.into(), only_query))
            }
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
        }
    }
}
```

The `BroadcastedDeclareTransaction` path should be updated similarly.

### Proof of Concept

1. Deploy an account contract whose `__validate__` checks `get_execution_info().tx_info.version >= 2^128`; if the query bit is set it returns `VALID` unconditionally, otherwise it verifies the ECDSA signature.
2. Call `starknet_V0_8_simulateTransactions` with a V3 invoke from that account, providing an **invalid** (or empty) signature, and **without** the `SKIP_VALIDATE` flag.
3. **Expected (spec-compliant) result:** The simulation succeeds because `only_query = true` causes the account to see version `0x100000000000000000000000000000003`, skip signature verification, and return `VALID`.
4. **Actual result:** The simulation fails with a validate error because `only_query = false` causes the account to see version `0x3`, attempt ECDSA verification against the invalid signature, and revert — an authoritative-looking wrong value returned by the RPC.

pub fn signed_tx_version(
    tx_version: &TransactionVersion,
    transaction_options: &TransactionOptions,
) -> TransactionVersion {
    // If only_query is true, set the 128-th bit.
    let query_only_bit = *QUERY_VERSION_BASE;
    assert_eq!(
        tx_version.0.to_biguint() & query_only_bit.to_biguint(),
        BigUint::from(0_u8),
        "Requested signed tx version with version that already has query bit set: {tx_version:?}."
    );
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}
