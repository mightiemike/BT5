Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end user, allowing any unpermissioned user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which resolves to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][end_user]`. Any pool admin who allowlists the router (required for legitimate users to use it) simultaneously opens the gate to every unpermissioned user who calls the same public router.

## Finding Description

`SwapAllowlistExtension.beforeSwap` enforces the allowlist as:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension is called by the pool), and `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`:

```solidity
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
``` [2](#0-1) 

`MetricOmmPool.swap` passes `msg.sender` as that `sender`:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
``` [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [4](#0-3) 

The real end-user address is stored only in transient storage via `_setNextCallbackContext` for payment purposes and is never forwarded to the pool or extension. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. This creates an irresolvable dilemma: if the admin does not allowlist the router, allowlisted users cannot use it; if the admin does allowlist the router, every unpermissioned user can bypass the gate through the same public router.

`DepositAllowlistExtension` does not share this flaw because it checks `owner` (the position owner explicitly passed to `pool.addLiquidity`), which is correctly preserved through the liquidity adder path. [5](#0-4) 

## Impact Explanation

Any pool deploying `SwapAllowlistExtension` to restrict trading to a curated set of addresses has its access control completely nullified for any user routing through `MetricOmmSimpleRouter`. The router is a public, permissionless contract. An unpermissioned user can execute swaps against the pool, trading against oracle prices the pool admin intended to expose only to vetted counterparties. This constitutes a direct loss of LP principal and a broken core pool invariant — matching the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" allowed impacts.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical periphery entry point for swaps. Any pool operator who configures `SwapAllowlistExtension` and expects users to use the router (the normal UX path) will either inadvertently open the gate to all users or block legitimate users from using the router. No special attacker setup is required beyond calling a public function on the deployed router.

## Recommendation

The extension must check the economically relevant actor — the end user — not the intermediary. Two complementary approaches:

1. **Pass the original initiator through the router.** The router already stores `msg.sender` as the payer in transient storage via `_setNextCallbackContext`. The router should populate `extensionData` with the real initiator address, and the extension should decode and gate on that address when `sender` is a known router.

2. **Require router-attested identity in `extensionData`.** The extension's `beforeSwap` should require that callers going through the router include the real user address in `extensionData`, and gate on `allowedSwapper[pool][realUser]` instead of `allowedSwapper[pool][router]`.

The simplest safe fix is to have the router always populate `extensionData` with the real user address and have the extension verify `allowedSwapper[pool][realUser]` when `sender` is a router address.

## Proof of Concept

```solidity
// Pool is configured with SwapAllowlistExtension.
// Admin allowlists userA and the router (so userA can use the router).
swapExtension.setAllowedToSwap(address(pool), address(router), true);
swapExtension.setAllowedToSwap(address(pool), userA, true);
// userB is NOT allowlisted.

// userB calls the router directly.
// Extension checks allowedSwapper[pool][router] == true → passes.
vm.prank(userB);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        recipient: userB,
        deadline: block.timestamp + 1,
        amountIn: 1000,
        amountOutMinimum: 0,
        zeroForOne: true,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// userB successfully swaps despite not being on the allowlist.
```
