### Title
Odd-Width Merkle Tree Ghost-Duplicate Index Bypass — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`, `contract/src/lib.rs::verify_transaction_inclusion`)

---

### Summary

`compute_root_from_merkle_proof` never validates that `transaction_position` is within the actual transaction count of the block. Bitcoin's Merkle tree construction duplicates the last leaf when the leaf count is odd. This means the proof for the last real transaction (index `N-1`) is byte-for-byte identical to a proof for the non-existent ghost duplicate (index `N`). Both calls to `verify_transaction_inclusion` return `true` against the same block header.

---

### Finding Description

**Root cause — `compute_root_from_merkle_proof`** [1](#0-0) 

The function accepts any `transaction_position` value. It uses only `position % 2` to decide left/right ordering and `position /= 2` to walk up the tree. No upper-bound check exists.

**Bitcoin's odd-leaf duplication**

`merkle_proof_calculator` explicitly duplicates the last leaf when the count is odd: [2](#0-1) 

For a 3-transaction block `[T0, T1, T2]`, the padded leaf array is `[T0, T1, T2, T2]`. The canonical proof for `T2` at index 2 is `[T2, hash(T0,T1)]`.

**Trace through `compute_root_from_merkle_proof` for index 2 vs index 3 with the same proof `[T2, hash(T0,T1)]`:**

| Step | index 2 | index 3 |
|------|---------|---------|
| pos=2 (even) | `hash(T2, T2)` | — |
| pos=3 (odd) | — | `hash(T2, T2)` |
| pos=1 (odd) | `hash(hash(T0,T1), hash(T2,T2))` = root ✓ | `hash(hash(T0,T1), hash(T2,T2))` = root ✓ |

Both paths produce the same root. Index 3 does not correspond to any real transaction.

**`verify_transaction_inclusion` performs no index-range check** [3](#0-2) 

The only guards are: block is on the main chain, confirmations are sufficient, and the proof is non-empty. There is no check that `tx_index < tx_count`. The block header stores only the Merkle root, not the transaction count, so the verifier has no way to enforce the bound without additional caller-supplied data.

**`ProofArgs` exposes `tx_index` as a fully attacker-controlled `u64`** [4](#0-3) 

**`verify_transaction_inclusion_v2` is also affected**

The v2 function adds a coinbase proof check but then delegates to the deprecated v1 function: [5](#0-4) 

The coinbase proof validates index 0 independently; it does not constrain the `tx_index` of the target transaction.

**GC timing is irrelevant to this bug**

`run_mainchain_gc` removes headers from storage but the oldest retained block remains fully accessible via `mainchain_header_to_height`. The ghost-duplicate bypass works on any canonical block regardless of GC state.

---

### Impact Explanation

`verify_transaction_inclusion` returns `true` for a transaction position that does not exist in the block. A downstream bridge, mint, or withdrawal contract that keys its replay-protection map on `(tx_id, tx_block_blockhash, tx_index)` — rather than `tx_id` alone — will accept two distinct "proof events" for the same Bitcoin transaction, enabling double-minting or double-withdrawal.

The severity is bounded by downstream replay-key design: contracts that deduplicate solely on `tx_id` are not directly exploitable. However, the light client itself is the authoritative source of truth for inclusion; returning `true` for a non-existent index is an incorrect attestation regardless of how callers consume it.

---

### Likelihood Explanation

- Any Bitcoin block with an odd transaction count (the majority of mainnet blocks) is a valid target.
- The attacker needs only the real Merkle proof for the last transaction, which is public on-chain data.
- No privileged role, relayer compromise, or special timing is required. The function is `pub` with only a `#[pause]` gate.
- The ghost index (`N`) is trivially derived: it equals the real last index plus one.

---

### Recommendation

1. **Require the caller to supply `tx_count`** and add a `require!(tx_index < tx_count)` guard before calling `compute_root_from_merkle_proof`. The Merkle proof length constrains the tree depth (`2^proof_len` is the maximum leaf count), so `tx_count <= 2^proof_len` can be enforced without trusting the caller blindly.
2. **Alternatively**, reject any `tx_index` that is `>= 2^(proof_len - 1)` when the last proof element equals `tx_id` itself (the canonical signal of an odd-width duplicate), though this heuristic is less robust.
3. Downstream contracts should always deduplicate on `tx_id` alone, never on `(tx_id, tx_index)`.

---

### Proof of Concept

```rust
// 3-transaction block: [T0, T1, T2]
// Merkle tree (padded): [T0, T1, T2, T2]
// Level 1:              [hash(T0,T1), hash(T2,T2)]
// Root:                 hash(hash(T0,T1), hash(T2,T2))

let proof = vec![T2.clone(), hash(T0, T1)];

// Real call — index 2 exists
assert!(verify_transaction_inclusion(ProofArgs {
    tx_id: T2.clone(), tx_index: 2, merkle_proof: proof.clone(), ..
}));

// Ghost call — index 3 does NOT exist in the block
assert!(verify_transaction_inclusion(ProofArgs {
    tx_id: T2.clone(), tx_index: 3, merkle_proof: proof.clone(), ..
}));
// Both return true. A bridge keyed on (tx_id, tx_index) mints twice.
```

The identical arithmetic path through `compute_root_from_merkle_proof` is shown in the table above: both index 2 (even→odd) and index 3 (odd→odd) produce `hash(T2,T2)` at the first step, then the same root at the second step. [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
