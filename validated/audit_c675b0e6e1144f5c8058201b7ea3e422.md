### Title
Taker Fee `alreadyMatched` Computed With Current Maker Price Instead of Actual Historical Fill Prices — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

In `OffchainExchange.sol`, the `alreadyMatched` quote parameter passed to `applyFee` for taker orders is computed by multiplying the current maker's price by the total base amount previously filled. When a taker order is filled across multiple makers at different prices, this produces an incorrect quote-denominated "already matched" value, causing the `minSize`-threshold fee logic to over- or under-charge fees.

---

### Finding Description

`matchOrders` calls `applyFee` for the taker with:

```solidity
applyFee(
    callState.productId,
    ordersInfo.taker,
    market,
    -maker.order.priceX18.mul(filledAmounts[ordersInfo.taker.digest]),
    taker.order.appendix,
    true
);
``` [1](#0-0) 

`filledAmounts[ordersInfo.taker.digest]` stores the cumulative **base** amount filled across all prior `matchOrders` calls for this taker digest. [2](#0-1) 

To convert that base amount into a quote amount, the code multiplies by `maker.order.priceX18` — the **current** maker's price. But prior fills may have executed at entirely different prices. The correct `alreadyMatched` value is the sum of `(base_i × price_i)` across all previous fills, not `current_price × Σbase_i`.

Inside `applyFee`, the corrupted value propagates directly into the fee-metering logic:

```solidity
int128 feeApplied = MathHelper.abs(alreadyMatched + matchQuote) -
    market.minSize;
feeApplied = MathHelper.min(feeApplied, matchQuote.abs());
``` [3](#0-2) 

`feeApplied` determines how much of the current fill is above the `minSize` threshold and therefore subject to the percentage fee rate. An incorrect `alreadyMatched` shifts this threshold incorrectly.

---

### Impact Explanation

**Under-charging (leak of value):** When a taker is selling a large order and the price is declining across fills, the current maker's price is *lower* than the prices at which previous fills occurred. This makes `alreadyMatched` smaller than the true already-matched quote, so `abs(alreadyMatched + matchQuote)` is smaller than it should be, `feeApplied` is understated, and the protocol collects less fee than it should.

**Over-charging:** Conversely, when a taker is buying and the price is rising, `alreadyMatched` is overstated, and the taker is charged more fee than the protocol intends.

The magnitude of the error scales with `|current_price − average_prior_fill_price| × prior_base_filled`. For large orders with meaningful price impact, this is a non-trivial amount. The under-charging direction is a direct "leak of value" from the protocol's fee revenue, analogous to the external report's finding.

---

### Likelihood Explanation

Any taker order that is partially filled across multiple `matchOrders` calls at different prices triggers this bug. This is the normal operating mode of the sequencer-driven matching engine — large taker orders routinely sweep through multiple maker levels. No special attacker setup is required; the bug fires on ordinary trading activity. The sequencer submits each taker-maker pair as a separate `matchOrders` transaction, so multi-fill taker orders are the common case.

---

### Recommendation

Track the cumulative **quote** amount filled for each taker digest, not just the base amount. Replace `filledAmounts` (base-denominated) with a parallel `filledQuoteAmounts` mapping that accumulates `ordersInfo.taker.quoteDelta` (before fee deduction) on each fill. Pass this stored quote total as `alreadyMatched` instead of recomputing it from the current maker's price:

```solidity
// Before applyFee:
int128 alreadyMatchedQuote = filledQuoteAmounts[ordersInfo.taker.digest];

applyFee(
    callState.productId,
    ordersInfo.taker,
    market,
    alreadyMatchedQuote,   // actual historical quote, not current_price * prev_base
    taker.order.appendix,
    true
);

// After applyFee, update the quote tracker:
filledQuoteAmounts[ordersInfo.taker.digest] += ordersInfo.taker.quoteDelta + ordersInfo.taker.fee;
```

This mirrors how swap fees in the referenced Algebra/Uniswap V3 report should have been accumulated inside the loop rather than recomputed from the final state.

---

### Proof of Concept

1. Taker places a sell order for 100 units at limit price $10.
2. First `matchOrders` call: fills 50 units at maker price $10. `filledAmounts[digest] = -50`. `alreadyMatched = -(-50 × 10) = 500`. Correct.
3. Second `matchOrders` call: fills 50 units at maker price $9 (price has dropped). `filledAmounts[digest] = -100`. `alreadyMatched = -(-100 × 9) = 900`. But the actual already-matched quote is `50 × 10 = 500`. The code overstates `alreadyMatched` by 400.
4. Inside `applyFee`: `abs(alreadyMatched + matchQuote) = abs(900 + 450) = 1350`. Actual should be `abs(500 + 450) = 950`. `feeApplied = 1350 − minSize` vs correct `950 − minSize`. The taker is over-charged on this fill.
5. Reverse scenario (buy order, price rising): `alreadyMatched` is understated → under-charging → protocol fee revenue is reduced. [1](#0-0) [4](#0-3) [5](#0-4)
