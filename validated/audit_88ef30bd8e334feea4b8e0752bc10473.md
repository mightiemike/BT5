Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual user — allowlist fully bypassed via router (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every extension hook. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the router contract, not the originating user. `SwapAllowlistExtension.beforeSwap` keys its allowlist check on this `sender` value, so it checks whether the **router** is allowlisted rather than whether the **user** is allowlisted. Any user can bypass a curated pool's swap allowlist by routing through the public router.

## Finding Description

**Root cause — pool passes `msg.sender` as `sender`:**

`MetricOmmPool.swap` (line 230–240) calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
_beforeSwap(
    msg.sender,   // ← always the immediate caller, i.e. the router
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` (lines 149–177) forwards this value verbatim to every configured extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

**Allowlist check uses the wrong actor:**

`SwapAllowlistExtension.beforeSwap` (lines 31–41) checks:

```solidity
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

Here `msg.sender` is the pool (correct) and `sender` is whoever called the pool. When the user goes through `MetricOmmSimpleRouter`, `sender` = router address.

**Exploit path:**

1. Pool admin deploys a curated pool with `SwapAllowlistExtension` configured; only specific addresses are allowlisted.
2. Pool admin (or factory) allowlists the public `MetricOmmSimpleRouter` so that legitimate users can trade through it — a necessary operational step.
3. Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInput/exactOutput` targeting the curated pool.
4. The router calls `pool.swap(...)` with `msg.sender = router`.
5. `SwapAllowlistExtension.beforeSwap` receives `sender = router`, finds `allowedSwapper[pool][router] == true`, and passes.
6. The non-allowlisted user's swap executes successfully, bypassing the curation policy entirely.

**Existing guards are insufficient:** The extension has no mechanism to recover the original user's address; it only receives the `sender` argument forwarded by the pool, which is always the immediate caller.

## Impact Explanation
A curated pool's swap allowlist — the primary access-control mechanism for permissioned pools — is completely bypassed for any user who routes through the public router. This constitutes broken core pool functionality and a direct admin-boundary break: the pool admin's curation policy is rendered ineffective by an unprivileged actor using a supported public entrypoint. Any non-allowlisted address can trade on a pool that was explicitly restricted to approved counterparties, causing direct loss of the curation guarantee and potential fund-impacting toxic flow on pools that rely on allowlisting for protection.

## Likelihood Explanation
The condition is trivially reachable: the public `MetricOmmSimpleRouter` is the standard user-facing entrypoint. For the router to be usable at all on an allowlisted pool, the pool admin must allowlist the router address. Once that operational step is taken (which is the only way to let any user trade through the router), the bypass is open to every address. No special privileges, flash loans, or multi-transaction setup are required — a single `exactInput` or `exactOutput` call suffices.

## Recommendation
The pool must pass the **originating user** to extensions, not the immediate `msg.sender`. Two standard approaches:

1. **Trusted forwarder pattern:** Require the router to append the original `msg.sender` to `callbackData` or a dedicated field; the pool reads and forwards it as `sender` to extensions only when `msg.sender` is a factory-registered router.
2. **Transient-storage sender:** The router writes the original caller into a transient storage slot before calling the pool; the pool reads it as the canonical `sender` for extension dispatch.

The `SwapAllowlistExtension` itself should also validate that `sender` is not a known router/intermediary unless the design explicitly intends to gate at the router level.

// Foundry integration test sketch
// 1. Deploy factory, pool with SwapAllowlistExtension, and MetricOmmSimpleRouter.
// 2. Pool admin allowlists the router (required for any router-mediated swap).
// 3. Pool admin does NOT allowlist `attacker`.
// 4. Attacker calls router.exactInput(...) targeting the curated pool.
// 5. Assert: swap succeeds (no NotAllowedToSwap revert).
// 6. Assert: attacker's balance of output token increased.
// Expected: step 5 should revert but does not — allowlist bypassed.
function test_swapAllowlistBypassViaRouter() public {
    // setup: pool with SwapAllowlistExtension, router allowlisted, attacker not allowlisted
    vm.prank(poolAdmin);
    swapAllowlist.setAllowedToSwap(address(pool), address(router), true);
    // attacker is NOT allowlisted

    uint256 balBefore = token1.balanceOf(attacker);
    vm.prank(attacker);
    router.exactInput(...); // routes through the allowlisted pool
    uint256 balAfter = token1.balanceOf(attacker);

    assertGt(balAfter, balBefore); // attacker received tokens despite not being allowlisted
}
