### Title
Fee-on-Transfer Token Deposit Credits Full Nominal Amount, Inflating Subaccount Collateral — (`core/contracts/EndpointStorage.sol`)

---

### Summary

`handleDepositTransfer` in `EndpointStorage.sol` performs two sequential `safeTransfer` calls using the caller-supplied `amount` with no pre/post balance check. If a fee-on-transfer ERC20 is listed as a spot product, the Clearinghouse receives fewer tokens than `amount`, but the slow-mode transaction records the full nominal `amount`, which `Clearinghouse.depositCollateral` then credits 1:1 to the subaccount. The depositor ends up with more collateral credit than tokens actually delivered, enabling them to withdraw the surplus at the expense of other users' reserves.

---

### Finding Description

`handleDepositTransfer` executes two transfers in sequence:

```
safeTransferFrom(token, from, address(this), amount);   // Endpoint receives ≤ amount
safeTransferTo(token, address(clearinghouse), amount);  // Clearinghouse receives ≤ amount
``` [1](#0-0) 

Neither call snapshots the balance before or after. The original `amount` is then embedded verbatim into the slow-mode transaction payload: [2](#0-1) 

When the sequencer (or anyone after the delay) executes the slow-mode transaction, `Clearinghouse.depositCollateral` reads `txn.amount` — the original nominal value — and calls `spotEngine.updateBalance` with it: [3](#0-2) 

Because the fee was silently deducted during transfer, the on-chain token balance of the Clearinghouse is lower than the sum of all credited subaccount balances. The gap is the cumulative fee taken across all such deposits.

---

### Impact Explanation

Every deposit of a fee-on-transfer token inflates the depositor's subaccount balance by the fee amount. The depositor can subsequently call `withdrawCollateral`, which debits the inflated balance and transfers the full credited amount out of the Clearinghouse: [4](#0-3) 

The shortfall is socialized across all other depositors of the same product, degrading the protocol's solvency for that token. Repeated deposits and withdrawals drain the pool incrementally.

**Impact: 5 / 10** — Direct asset loss from the Clearinghouse token reserve; bounded per-deposit by the fee rate but unbounded in aggregate.

---

### Likelihood Explanation

Exploitation requires a fee-on-transfer ERC20 to be registered as a spot product. Product registration is owner-controlled (`addEngine` / product initialization), so the attack surface only opens if such a token is ever listed. Once listed, any unprivileged user calling the public `depositCollateralWithReferral` entrypoint can trigger the bug with no further preconditions. [5](#0-4) 

**Likelihood: 2 / 10** — Requires a fee-on-transfer token to be listed; the current quote token (USDC) is not one, but the code imposes no guard against listing such tokens.

---

### Recommendation

Capture the Clearinghouse's token balance before and after the transfer pair and use the measured delta — not the caller-supplied `amount` — as the value recorded in the slow-mode transaction and credited to the subaccount:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 preBal = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 postBal = token.balanceOf(address(clearinghouse));
    require(postBal - preBal == amount, ERR_FEE_ON_TRANSFER_TOKEN);
}
```

Alternatively, whitelist only non-fee-on-transfer tokens at the product registration layer and enforce this invariant on-chain.

---

### Proof of Concept

1. Owner lists a fee-on-transfer ERC20 (e.g., 1% fee) as spot `productId = 5`.
2. Attacker calls `Endpoint.depositCollateral("default", 5, 1_000e18)`.
3. `handleDepositTransfer` pulls 1 000 tokens from attacker; Endpoint receives 990; Clearinghouse receives 980.
4. Slow-mode transaction is queued with `amount = 1_000e18`.
5. After the delay, sequencer (or attacker via `executeSlowModeTransaction`) processes the tx; `Clearinghouse.depositCollateral` credits attacker's subaccount with `1_000e18 * multiplier`.
6. Attacker calls `withdrawCollateral` for the full credited amount; Clearinghouse transfers 1 000 tokens out, but only holds 980 — the 20-token deficit is absorbed from other depositors' reserves.
7. Repeat to drain the pool. [1](#0-0) [6](#0-5)
