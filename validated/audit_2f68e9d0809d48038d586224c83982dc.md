Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged actor to bypass the deposit allowlist - (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter and checks `owner` instead, despite the contract's documented intent to gate deposits by depositor address. Any unprivileged actor can bypass the allowlist by calling `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with an allowlisted address as `owner`, causing their own tokens to be permanently credited to the allowlisted owner's position with no recovery path.

## Finding Description
`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)` where `msg.sender` is the actual depositor/router and `owner` is the position recipient supplied by the caller. [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, ...)` but leaves `sender` unnamed (discarded) and evaluates `allowedDepositor[msg.sender][owner]`: [2](#0-1) 

This contradicts the NatSpec and storage variable name, both of which declare the intent is to gate by depositor: [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a caller-supplied `owner` and stores `msg.sender` as `payer` in transient context for the callback: [4](#0-3) 

The callback pulls tokens from `payer` (the actual `msg.sender` of `addLiquidityExactShares`, i.e., the unauthorized actor): [5](#0-4) 

`removeLiquidity` enforces `msg.sender == owner`, so the unauthorized depositor cannot recover their tokens from the allowlisted owner's position: [6](#0-5) 

**Exploit path:**
1. Pool uses `DepositAllowlistExtension` with `allowAllDepositors[pool] = false`; only `ALICE` is in `allowedDepositor[pool]`.
2. Unauthorized `BOB` calls `liquidityAdder.addLiquidityExactShares(pool, ALICE, salt, deltas, max0, max1, data)`.
3. `LiquidityAdder` stores `BOB` as `payer` in transient context, then calls `pool.addLiquidity(ALICE, ...)`.
4. Pool calls `_beforeAddLiquidity(LiquidityAdder, ALICE, ...)`.
5. Extension evaluates `allowedDepositor[pool][ALICE]` → `true` → no revert.
6. Liquidity is added; callback pulls tokens from `BOB`.
7. `BOB`'s tokens are credited to `ALICE`'s position. `BOB` cannot call `removeLiquidity` (not the owner). Allowlist bypassed; `BOB`'s principal is permanently lost.

## Impact Explanation
Direct, irreversible loss of user principal. The unauthorized depositor's tokens are transferred into the pool and credited to an allowlisted owner's position. Since `removeLiquidity` requires `msg.sender == owner`, the depositor has no recovery path. Pools relying on this extension for KYC/regulatory gating or access control have that control fully negated by any unprivileged actor. This constitutes a Critical/High direct loss of user funds and a complete bypass of a core access-control mechanism.

## Likelihood Explanation
The bypass requires no privileged access, flash loan, or special setup. The only precondition is knowledge of one allowlisted address, which is publicly readable from `allowedDepositor`. Any actor can trigger this against any pool using `DepositAllowlistExtension` with `allowAllDepositors = false` by calling the deployed `MetricOmmPoolLiquidityAdder`. The attack is repeatable and permissionless.

## Recommendation
Replace the `owner` check with a `sender` check in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns the implementation with the documented intent ("Gates `addLiquidity` by depositor address") and the storage variable name `allowedDepositor`.

## Proof of Concept
1. Deploy pool with `DepositAllowlistExtension`; set `allowAllDepositors[pool] = false`; add only `ALICE` to `allowedDepositor[pool]`.
2. As `BOB` (not allowlisted), call:
   ```solidity
   liquidityAdder.addLiquidityExactShares(pool, ALICE, salt, deltas, maxAmount0, maxAmount1, extensionData);
   ```
3. Observe: `LiquidityAdder` calls `pool.addLiquidity(ALICE, ...)` → extension checks `allowedDepositor[pool][ALICE]` → `true` → passes.
4. Callback pulls tokens from `BOB` (stored as `payer`).
5. Assert: `BOB`'s token balance decreased; shares credited to `ALICE`'s position.
6. `BOB` attempts `pool.removeLiquidity(ALICE, ...)` → reverts with `NotPositionOwner`.
7. Confirm: allowlist bypassed, `BOB`'s principal permanently lost to `ALICE`'s position.
