Audit Report

## Title
Retrospective Decay-Rate Application in `executeOracleStopLossDecay` Retroactively Zeros Stop-Loss Watermarks, Disabling LP Principal Protection — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`executeOracleStopLossDecay` writes a new (higher) `decayPerSecondE8` into `oracleStopLossConfig` without resetting `BinHighWatermarks.lastDecayTs` for any bin. The next swap after the rate change calls `_checkAndUpdateWatermarks`, which computes `dt = block.timestamp − lastDecayTs` and applies the new rate over the entire elapsed interval. For any pool idle longer than `1e8 / newRate` seconds before execution, `_decayed` returns 0 for every touched bin, permanently zeroing all watermarks and disabling the stop-loss guard for those bins.

## Finding Description
`executeOracleStopLossDecay` (lines 139–147) updates only `oracleStopLossConfig[pool_].decayPerSecondE8`; it does not touch `highWatermarks[pool_][binIdx].lastDecayTs` for any bin. On the next swap, `_afterSwapOracleStopLoss` reads `cfg.decayPerSecondE8` (now the new rate) and passes it to `_checkAndUpdateWatermarks` (lines 258–285). That function computes `dt = block.timestamp - hwmS.lastDecayTs` using the stale timestamp and calls `_decayed(hwmS.token0, decayRate, dt)`. Inside `_decayed` (lines 319–324), `factor = ratePerSecondE8 * dt`; if `factor >= E8` (i.e., `dt >= 1e8 / newRate` seconds), the function returns 0. With watermarks zeroed, `_applyWatermark` (lines 328–336) can never report a breach (`metric >= 0 = hwm` always), so `OracleStopLossTriggered` is never emitted regardless of how far per-share value falls. The timelock does not prevent this because it only delays the rate write; it does not snapshot or apply the old-rate decay before the new rate takes effect.

## Impact Explanation
The stop-loss extension's sole purpose is to cap LP value loss per bin. Once watermarks are zeroed, the guard is permanently disabled for those bins: any subsequent swap — including one that extracts all remaining token0 or token1 — passes `afterSwap` without triggering `OracleStopLossTriggered`. LPs suffer direct loss of principal bounded only by the bin's total balance, which can be the entire LP deposit. This is a direct loss of user principal and a broken core pool functionality (stop-loss guard), meeting Critical/High impact thresholds.

## Likelihood Explanation
The trigger is the pool admin executing a timelocked decay-rate increase — a routine administrative action explicitly supported by the contract. No malicious setup is required. The retroactive zeroing is most severe when the pool has been swap-idle for `> 1e8 / newRate` seconds before execution. At `newRate = 1000` (0.001%/s), this threshold is ~27.8 hours — easily exceeded in low-volume pools or during market downtime. The pool admin is semi-trusted; the timelock is the only LP protection mechanism and is bypassed by this path. The condition is naturally occurring and repeatable.

## Recommendation
In `executeOracleStopLossDecay`, apply the old-rate decay to every bin's watermarks and reset `lastDecayTs` to `block.timestamp` before writing the new rate. Since iterating all bins is not feasible on-chain, the minimal fix is to store a rate-change checkpoint per bin (rate + timestamp) so `_decayed` can split the elapsed interval at the rate-change boundary and apply each segment at its respective rate. Alternatively, store the current rate alongside `lastDecayTs` in `BinHighWatermarks` and update it atomically with `lastDecayTs` on every watermark touch, so the stored rate always matches the rate used for the last decay computation.
