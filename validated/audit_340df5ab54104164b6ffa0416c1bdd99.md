### Title
Deprecated `verify_transaction_inclusion` Bypasses Coinbase Proof Check, Enabling 64-Byte Merkle Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction Merkle proof forgery vulnerability. It enforces a coinbase anchor check before delegating to the deprecated `verify_transaction_inclusion`. However, the deprecated function remains a fully public, unrestricted on-chain entry point. Any unprivileged NEAR caller can invoke it directly, skipping the coinbase check entirely and obtaining a `true` proof result for a fabricated transaction ID.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces two invariants before calling the deprecated function:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` (same tree depth)
2. The coinbase transaction's Merkle proof must reconstruct the block's `merkle_root` at index 0. [1](#0-0) 

The deprecated `verify_transaction_inclusion` performs no such coinbase anchor check. It fetches the stored header and directly compares `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof)` against `header.block_header.merkle_root`. [2](#0-1) 

Rust's `#[deprecated]` attribute emits only a compiler warning; it does not restrict on-chain visibility. The function carries no role guard, no `#[private]`, and no `#[pause]` bypass requirement beyond the standard pause check. It is callable by any NEAR account. [3](#0-2) 

The 64-byte forgery works because Bitcoin's Merkle tree uses the same double-SHA256 hash for both leaf nodes (real transactions) and internal nodes. A 64-byte value that is itself the concatenation of two 32-byte hashes can be submitted as `tx_id`; the Merkle proof then "proves" that this internal node exists in the tree, and `compute_root_from_merkle_proof` returns the correct root. Without the coinbase anchor, the contract cannot distinguish a real leaf from a forged internal node. [4](#0-3) 

The contract's own NatSpec on `verify_transaction_inclusion` explicitly warns of this:

> *"This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."* [5](#0-4) 

The analog to the external report is direct: just as the `Claimer::claimPrizes()` function was open to anyone and could be called from within a hook to bypass fees, `verify_transaction_inclusion` is open to anyone and can be called directly to bypass the coinbase proof check that `verify_transaction_inclusion_v2` enforces. In both cases, a "secure" path exists but an unrestricted alternative entry point undermines it.

---

### Impact Explanation

Any downstream consumer — a cross-chain bridge, a token-unlock contract, or any NEAR contract gating fund releases on a Bitcoin transaction proof — that calls `verify_transaction_inclusion` will receive `true` for a fabricated transaction. The attacker never needs to broadcast a real Bitcoin transaction. They only need to craft a 64-byte value whose double-SHA256 Merkle path matches a stored block's `merkle_root`. This enables theft of bridged assets proportional to the bridge's TVL, with the proof result itself being the corrupted invariant.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. The deprecated function is callable by any NEAR account. The 64-byte forgery technique is publicly documented (referenced in the contract's own NatSpec via the BitMEX research link). The only prerequisite is that a target block's header is already stored in the contract — the normal operating state of any live deployment.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it with `#[private]` to prevent external calls. If backward compatibility is required, replicate the coinbase proof validation from `verify_transaction_inclusion_v2` inside `verify_transaction_inclusion` so both entry points enforce the same security invariant.

---

### Proof of Concept

1. Identify any block hash `B` stored in `mainchain_header_to_height` with a known `merkle_root R`.
2. Construct a 64-byte value `F` such that `double_sha256(F[0..32] || F[32..64])` participates in a Merkle path that reconstructs `R` (standard 64-byte forgery construction, publicly documented).
3. Call `verify_transaction_inclusion` with `tx_id = F`, `tx_block_blockhash = B`, `tx_index = <forged index>`, `merkle_proof = <forged path>`, `confirmations = 1`.
4. `compute_root_from_merkle_proof(F, forged_index, forged_path)` returns `R`, which equals `header.block_header.merkle_root`.
5. The function returns `true` despite `F` not being a real transaction.
6. A bridge contract gated on this call releases funds to the attacker. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
