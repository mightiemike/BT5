### Title
`ValidResourceBounds` Protobuf Deserialization Silently Downgrades `AllResources` to `L1Gas` Variant, Producing a Wrong Transaction Hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` silently reconstructs an `AllResources`-variant transaction as the `L1Gas` variant whenever `l2_gas` and `l1_data_gas` are both zero. Because `get_tip_resource_bounds_hash` produces a structurally different hash for each variant (2 vs. 3 resource elements), the hash recomputed after deserialization diverges from the hash the user originally signed. A V3 transaction accepted by the gateway with an `AllResources` signature is therefore rejected by every syncing peer that recomputes the hash from the deserialized wire form.

### Finding Description

`ValidResourceBounds` is a two-variant enum:

```
L1Gas(ResourceBounds)          // pre-0.13.3: only L1 gas
AllResources(AllResourceBounds) // post-0.13.3: L1 + L2 + L1_DATA gas
```

The transaction hash function `get_tip_resource_bounds_hash` branches on this variant:

- `L1Gas` → hashes `[tip, L1_GAS_packed, L2_GAS_packed]` (2 resource elements)
- `AllResources` → hashes `[tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed]` (3 resource elements) [1](#0-0) 

These produce **different Poseidon digests** even when the extra element is zero, because the hash input length differs.

The protobuf deserializer reconstructs the variant using a zero-check:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(...)
})
``` [2](#0-1) 

A user who submits a V3 transaction with `AllResources` where `l2_gas = 0` and `l1_data_gas = 0` (a valid post-0.13.3 transaction that happens to use only L1 gas) has their transaction hash computed with the 3-element path. The gateway accepts it via `RpcInvokeTransactionV3`, which stores `resource_bounds: AllResourceBounds` and always produces the `AllResources` hash. [3](#0-2) 

When the block is synced to peers via P2P, the transaction is serialized as `protobuf::ResourceBounds` with `l2_gas = 0` and `l1_data_gas = 0`. On deserialization, the receiving node reconstructs `ValidResourceBounds::L1Gas`, computes the 2-element hash, and calls `validate_transaction_hash`, which finds no match against the stored 3-element hash. [4](#0-3) 

The codebase itself acknowledges this hazard in a test comment:

> "If all the fields of `AllResources` are 0 upon serialization, then the deserialized value will be interpreted as the `L1Gas` variant." [5](#0-4) 

The test workaround sets `l2_gas.max_amount = GasAmount(1)` to avoid the collision, confirming the divergence is real and known.

### Impact Explanation

A block containing a V3 transaction with `AllResources{l2_gas=0, l1_data_gas=0}` is accepted by the sequencer but cannot be validated by any syncing peer. The peer recomputes a different hash from the deserialized transaction and rejects it, causing a permanent sync failure for that block. This maps to: **Wrong state or revert result from execution logic for accepted input** and **Transaction conversion or signature/hash logic binds the wrong hash**.

### Likelihood Explanation

Any user submitting a V3 transaction that only consumes L1 gas (e.g., a simple transfer on a network where L2 gas is not yet priced) will naturally produce `l2_gas = 0` and `l1_data_gas = 0`. The gateway accepts it; every syncing node rejects it. No privileged access is required; a normal user transaction is sufficient to trigger the condition.

### Recommendation

**Short term**: In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not infer the variant from zero-checks on the wire values. Instead, carry an explicit variant tag in the protobuf message, or always reconstruct as `AllResources` when all three fields are present on the wire (which they are, since the serializer always emits all three).

**Long term**: Add a round-trip test that serializes an `AllResources` transaction with `l2_gas = 0` and `l1_data_gas = 0`, deserializes it, and asserts that `validate_transaction_hash` still returns `true`. The existing test helper `add_gas_values_to_transaction` masks the bug rather than fixing it.

### Proof of Concept

1. Construct a V3 invoke transaction with:
   ```
   resource_bounds = AllResources { l1_gas: {max_amount: 1000, max_price: 1}, l2_gas: {0,0}, l1_data_gas: {0,0} }
   ```
2. Compute the transaction hash via `get_tip_resource_bounds_hash` with `AllResources` → 3-element Poseidon hash. Sign it.
3. Submit to the gateway. It stores `AllResourceBounds` and recomputes the same 3-element hash → accepted, included in a block.
4. Serialize the block's `InvokeTransactionV3` to protobuf bytes via `From<ValidResourceBounds> for protobuf::ResourceBounds`.
5. Deserialize on a peer via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`: `l1_data_gas.is_zero() && l2_gas.is_zero()` → reconstructed as `L1Gas`.
6. Call `validate_transaction_hash` on the peer: computes 2-element hash ≠ stored 3-element hash → returns `false` → block rejected. [6](#0-5) [7](#0-6)

        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));

        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;

            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
