### Title
Bonus Pool Depletion Blocks All Position Exit and Undelegation Paths â€” (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

The `claimRewards` helper is called unconditionally as a prerequisite inside every position-exit message handler (`TierUndelegate`, `TierRedelegate`, `ExitTierWithDelegation`, `ClearPosition`, `AddToTierPosition`). Inside `claimRewards`, `processEventsAndClaimBonus` performs a hard `sufficientBonusPoolBalance` check and returns an error if the bonus pool cannot cover the owed bonus. When the bonus pool is depleted and a user has accrued non-zero bonus rewards, every exit path fails atomically, permanently locking the user's staked principal in the position delegator account with no alternative retrieval path.

---

### Finding Description

`claimRewards` (line 87) calls two sub-routines sequentially:

1. `claimBaseRewards` â€” withdraws standard staking rewards via `distributionKeeper.WithdrawDelegationRewards`.
2. `processEventsAndClaimBonus` â€” computes bonus owed and, at line 230, calls `sufficientBonusPoolBalance`; if the pool balance is insufficient it returns a non-nil error. [1](#0-0) [2](#0-1) 

Because the check is only skipped when `totalBonus.IsZero()` (line 219), any user who has accrued even one unit of bonus reward is affected the moment the pool runs dry.

This `claimRewards` call is embedded as a mandatory step in every message handler that modifies or exits a position:

When `processEventsAndClaimBonus` returns an error, the entire message handler returns an error, and Cosmos SDK reverts all state changes atomically. The user's staked tokens remain locked inside the position's delegator sub-account (`pos.DelegatorAddress`). The only remaining exit path, `WithdrawFromTier`, only transfers spendable coins from the delegator account and does not call `claimRewards`, but it requires the undelegation to have already completed â€” which is itself blocked. The corrupted invariant: a user's staked principal locked in a tier position must always be retrievable via undelegation. When the bonus pool is empty, this invariant is broken â€” the principal is inaccessible regardless of the user's own actions.

### Impact Explanation

A user's staked tokens (the bond denom, e.g. `basecro`) are held in the position's dedicated delegator sub-account. When the bonus pool is depleted:

- `TierUndelegate` fails â†’ the unbonding period never starts â†’ `WithdrawFromTier` can never be reached.
- `ExitTierWithDelegation` fails â†’ the delegation cannot be transferred back to the owner.
- `TierRedelegate` fails â†’ the user cannot move to a different validator.

The user's principal is locked with no protocol-level escape hatch until governance refills the bonus pool. The exact value at risk is the full staked amount of every position that has accrued non-zero bonus rewards at the time the pool reaches zero.

---

### Likelihood Explanation

The bonus pool is a finite module account (`types.RewardsPoolName`) funded by governance or inflation. As the user base grows and bonus rewards are continuously paid out, the pool balance trends toward zero. No automatic refill mechanism is visible in the keeper code. A pool depletion event is therefore a foreseeable operational condition, not a theoretical edge case. Any unprivileged user who holds a tier position and has accrued bonus rewards is affected the moment the pool balance falls below their owed bonus.

---

### Recommendation

Decouple the bonus reward payment from the exit/undelegation critical path. Two safe approaches:

1. **Skip-on-empty**: In `processEventsAndClaimBonus`, if `sufficientBonusPoolBalance` fails, emit an event recording the unpaid bonus and return `sdk.NewCoins(), nil` instead of an error. Record the owed amount in state for later settlement.
2. **Defer bonus on exit**: Allow `TierUndelegate` and `ExitTierWithDelegation` to proceed without calling `claimRewards` when the pool is insufficient; store the owed bonus as a claimable debt redeemable once the pool is refilled.

Either approach ensures the user's principal is never held hostage by the bonus pool balance.

func (k Keeper) claimRewards(ctx context.Context, pos types.PositionState) (types.PositionState, sdk.Coins, sdk.Coins, error) {
	if !pos.IsDelegated() {
		return pos, sdk.NewCoins(), sdk.NewCoins(), nil
	}

	base, err := k.claimBaseRewards(ctx, pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	return pos, base, bonus, nil
}

func (ms msgServer) TierUndelegate(ctx context.Context, msg *types.MsgTierUndelegate) (*types.MsgTierUndelegateResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateUndelegatePosition(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	pos, _, _, err = ms.claimRewards(ctx, pos)
	if err != nil {
		return nil, err
	}

	srcValidator := pos.Delegation.ValidatorAddress
	valAddr, err := sdk.ValAddressFromBech32(srcValidator)
	if err != nil {
		return nil, err
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	completionTime, _, err := ms.undelegate(ctx, delAddr, valAddr, pos.Delegation.Shares)
	if err != nil {
		return nil, err
	}

	pos.ClearBonusCheckpoints()

	if err := ms.setPosition(ctx, pos.Position, &ValidatorTransition{PreviousAddress: srcValidator}); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionUndelegated{
		PositionId:     pos.Id,
		TierId:         pos.TierId,
		Owner:          pos.Owner,
		Validator:      srcValidator,
		CompletionTime: completionTime,
	}); err != nil {
		return nil, err
	}

	return &types.MsgTierUndelegateResponse{
		CompletionTime: completionTime,
		PositionId:     pos.Id,
	}, nil
}
