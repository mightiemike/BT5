Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates Position `owner` Instead of Token-Paying `sender`, Enabling Complete Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is designed to gate which addresses may deposit liquidity into a restricted pool. Its `beforeAddLiquidity` hook discards the `sender` parameter (the address that calls `addLiquidity` and pays tokens via callback) and instead validates `owner` (the position recipient). Any non-allowlisted address can bypass the guard by naming any allowlisted address as `owner`, depositing tokens into the pool while the allowlisted address receives the position.

## Finding Description
`MetricOmmPool.addLiquidity` invokes the extension hook as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The first argument is `msg.sender` (the caller/token payer); the second is `owner` (the position recipient, freely chosen by the caller).

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the first parameter and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [2](#0-1) 

After the hook passes, `LiquidityLib.addLiquidity` issues the token-pull callback to `msg.sender` (the original `addLiquidity` caller, confirmed by the library's delegatecall semantics):

```solidity
IMetricOmmModifyLiquidityCallback(msg.sender)
    .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
``` [3](#0-2) 

The entity actually paying tokens is never checked. `SwapAllowlistExtension.beforeSwap` demonstrates the correct pattern — it validates `sender` (first parameter), not the recipient:

```solidity
function beforeSwap(address sender, address, bool, ...)
    ...
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
``` [4](#0-3) 

The inconsistency between the two sibling extensions confirms `DepositAllowlistExtension` is checking the wrong actor.

## Impact Explanation
The pool admin's deposit allowlist is completely ineffective. Any non-allowlisted address can deposit tokens into a restricted pool by supplying any allowlisted address as `owner`. The non-allowlisted address pays the tokens; the allowlisted address receives the position. This is a direct admin-boundary break: an unprivileged caller bypasses an access-control invariant the extension is explicitly designed to enforce, with no fund loss to the attacker and no special privileges required.

## Likelihood Explanation
The bypass requires no special privileges, no flash loans, and no external state manipulation. Any caller can supply an arbitrary `owner`. Allowlisted addresses are typically known (e.g., whitelisted LPs), making the bypass trivially constructable by any non-allowlisted actor on every call to `addLiquidity` on a restricted pool.

## Recommendation
Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
```

Also update `setAllowedToDeposit` / `isAllowedToDeposit` documentation to reflect that the allowlist governs the caller of `addLiquidity` (the token payer), not the position owner.

## Proof of Concept
1. Pool is deployed with `DepositAllowlistExtension`; `allowAllDepositors[pool] = false`.
2. Admin calls `setAllowedToDeposit(pool, alice, true)`. Alice is the only allowlisted depositor.
3. Bob (non-allowlisted, implements `IMetricOmmModifyLiquidityCallback`) calls `pool.addLiquidity(owner=alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(bob, alice, ...)` → extension checks `allowedDepositor[pool][alice]` → `true` → hook returns success.
5. `LiquidityLib.addLiquidity` proceeds; callback `metricOmmModifyLiquidityCallback` is issued to Bob (`msg.sender` in delegatecall context).
6. Bob transfers tokens to the pool; Alice's position shares are credited.
7. Bob has deposited into the restricted pool; the allowlist is fully bypassed.
