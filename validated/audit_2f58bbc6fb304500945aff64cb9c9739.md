### Title
`CloseOnly` Tier Blocks Redelegate While Slashing Remains Active, Trapping Users on Jailing Validators - (`File: x/tieredrewards/keeper/msg_validate.go`)

### Summary

When governance marks a tier as `CloseOnly`, `MsgTierRedelegate` is unconditionally blocked by `ErrTierIsCloseOnly`. Slashing via staking hooks is not blocked. A user whose validator begins misbehaving after the `CloseOnly` flag is set cannot move their delegation to a healthy validator, and their position tokens are slashed with no recourse.

### Finding Description

The `CloseOnly` flag on a `Tier` is intended to sunset a tier by blocking new entries while allowing existing positions to exit. The ADR documents the blocked messages as: `LockTier`, `CommitDelegation`, `AddToPosition`, `TierRedelegate`, and `ClearPosition`. [1](#0-0) 

The enforcement in `validateRedelegatePosition` unconditionally rejects any redelegate when the tier is `CloseOnly`: [2](#0-1) 

Similarly, `validateClearPosition` blocks `MsgClearPosition` on a `CloseOnly` tier: [3](#0-2) 

Slashing, however, is applied through staking hooks (`BeforeValidatorSlashed`, `BeforeRedelegationSlashed`) that are entirely independent of the `CloseOnly` flag: [4](#0-3) 

This creates an asymmetric window: the protocol can reduce a user's position tokens via slashing, but the user cannot move their delegation away from the misbehaving validator.

The only remaining exit paths after `CloseOnly` is set are `MsgTriggerExitFromTier` + `MsgTierUndelegate`. But `MsgTierUndelegate` requires both exit to be triggered **and** `ExitDuration` to have elapsed: [5](#0-4) 

`ExitDuration` can be months or years (e.g., a 1-year tier). During the entire wait, the user is trapped on the slashing validator. `MsgExitTierWithDelegation` also requires exit elapsed and the validator to be bonded — which fails if the validator is jailed. [6](#0-5) 

### Impact Explanation

A user's `pos.Delegation.Shares` — and therefore their recoverable token amount — is reduced by the slash fraction applied to the validator. Because redelegate is blocked, the user cannot escape to a healthy validator before or during the slash. The corrupted value is the position's token amount, which is read live from staking via `TokensFromShares`. The user suffers a direct, irreversible token loss with no protective action available.

### Likelihood Explanation

Governance can mark any tier `CloseOnly` at any time via `MsgUpdateTier`. Validator downtime is a routine, unprivileged event — any validator can miss blocks and be jailed. The combination is realistic: governance sunsets a tier, and one or more validators holding positions in that tier subsequently get jailed and slashed. No attacker control is required; the harm is a natural consequence of the two independent mechanisms interacting.

### Recommendation

`MsgTierRedelegate` should not be blocked by `CloseOnly` when the source validator is jailed or unbonding. The `CloseOnly` intent is to prevent new capital from entering the tier, not to trap existing capital on failing validators. One approach: in `validateRedelegatePosition`, skip the `CloseOnly` check when the source validator is not bonded, allowing users to escape a jailing validator even in a `CloseOnly` tier.

```go
// In validateRedelegatePosition:
srcVal, err := k.stakingKeeper.GetValidator(ctx, srcValAddr)
if err == nil && srcVal.IsBonded() {
    // Only enforce CloseOnly when src validator is healthy
    if tier.IsCloseOnly() {
        return types.ErrTierIsCloseOnly
    }
}
```

Alternatively, allow `MsgTierRedelegate` unconditionally in `CloseOnly` tiers (since it does not add new capital, only moves existing capital).

### Proof of Concept

1. Governance creates tier 1 with `exit_duration = 365 days`, `bonus_apy = 4%`.
2. User calls `MsgLockTier` on tier 1, delegating to validator V.
3. Governance calls `MsgUpdateTier` setting `close_only = true` on tier 1.
4. Validator V misses blocks and is jailed (downtime slash).
5. User calls `MsgTierRedelegate` to move to a healthy validator V2.
6. Transaction fails with `ErrTierIsCloseOnly` — enforced at `msg_validate.go:90`.
7. Validator V is slashed; user's position token amount is reduced.
8. User calls `MsgTriggerExitFromTier` (allowed), but must wait 365 days for `ExitDuration` before `MsgTierUndelegate` is accepted.
9. During the 365-day wait, further slashes on V continue to reduce the user's position with no recourse. [7](#0-6) [8](#0-7) [1](#0-0)
