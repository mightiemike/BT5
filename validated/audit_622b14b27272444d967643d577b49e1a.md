### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual caller of `addLiquidity`) and instead checks the caller-supplied `owner` (LP position owner) against the allowlist. Because `owner` is freely chosen by the caller, any unauthorized address can bypass the allowlist by naming any authorized address as `owner`, causing the guard to pass on a value the attacker controls rather than on the value the guard is meant to protect.

---

### Finding Description

In `MetricOmmPool.addLiquidity`, the pool passes `msg.sender` as `sender` and the caller-supplied `owner` as the LP position owner to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values to every registered extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, ...)` but drops `sender` (unnamed first parameter) and checks only `owner`: [3](#0-2) 

Because `owner` is an arbitrary address supplied by the caller of `addLiquidity`, any address not in the allowlist can pass `owner = <any_authorized_address>` and the check `allowedDepositor[pool][owner]` returns `true`, letting the deposit proceed. The LP shares are then credited to the authorized address's position while the unauthorized caller funds the deposit via the swap callback.

The parallel `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller), confirming the asymmetry is unintentional: [4](#0-3) 

The contract's own NatSpec ("Gates `addLiquidity` by depositor address") and the mapping name `allowedDepositor` both confirm the intent was to gate on the actual depositor (`sender`), not the LP position owner. [5](#0-4) 

---

### Impact Explanation

The deposit allowlist is rendered completely ineffective. Any unprivileged address can:

1. Bypass the allowlist and inject liquidity into any bin of a restricted pool.
2. Alter the pool's bin composition and `curPosInBin` state, shifting the effective bid/ask spread seen by subsequent swappers.
3. If the pool has no `SwapAllowlistExtension` (or the attacker is separately allowlisted to swap), the attacker can follow the deposit with a swap that benefits from the manipulated bin state, extracting value from existing LPs.

The pool admin's core access-control invariant — that only authorized addresses may add liquidity — is broken by an unprivileged path with no special role required.

---

### Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with `owner` set to any address already in the allowlist. No privileged role, flash loan, or oracle manipulation is needed. The attacker's only cost is the tokens transferred via the callback, which are credited to the authorized address's LP position (not recoverable by the attacker), making this a griefing or pool-manipulation vector rather than a direct theft, but the allowlist bypass itself is trivially reachable by any caller.

---

### Recommendation

Replace the unnamed first parameter with `sender` and check it instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(
    address sender,   // was: address (unnamed/discarded)
    address,          // owner — not the depositor
    uint80,
    LiquidityDelta calldata,
    bytes calldata
) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

---

### Proof of Concept

Setup: pool deployed with `DepositAllowlistExtension`; only `Alice` is in `allowedDepositor[pool]`.

```
// Bob (not allowlisted) calls:
pool.addLiquidity(
    owner        = Alice,   // authorized address — passes the check
    salt         = 0,
    deltas       = <any valid LiquidityDelta>,
    callbackData = <Bob funds the deposit in the callback>,
    extensionData = ""
);

// Extension evaluates: allowedDepositor[pool][Alice] == true  → no revert
// LiquidityLib credits LP shares to Alice's position
// Bob's callback transfers tokens to the pool
// Result: Bob bypassed the allowlist; unauthorized liquidity entered the pool
```

Calling `pool.addLiquidity(owner=Bob, ...)` from Bob's address would correctly revert (`allowedDepositor[pool][Bob] == false`), confirming the guard fires on `owner`, not on the actual depositor.
