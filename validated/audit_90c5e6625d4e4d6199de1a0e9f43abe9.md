### Title
Fee-on-Transfer Token Deposit Inflates Subaccount Credit Beyond Actual Clearinghouse Receipt — (`core/contracts/EndpointStorage.sol`)

---

### Summary

In `Endpoint.depositCollateralWithReferral`, the `handleDepositTransfer` helper transfers tokens to the Clearinghouse but the slow-mode transaction queued for later processing records the **caller-supplied `amount`**, not the actual tokens received. For fee-on-transfer tokens, the Clearinghouse receives less than `amount` while the subaccount is credited the full `amount`, corrupting the protocol's solvency invariant.

---

### Finding Description

`depositCollateralWithReferral` is a `public` function callable by any unprivileged user:

```solidity
// Endpoint.sol lines 123-167
function depositCollateralWithReferral(
    bytes32 subaccount,
    uint32 productId,
    uint128 amount,   // ← caller-controlled
    string memory
) public {
    ...
    handleDepositTransfer(
        IERC20Base(spotEngine.getToken(productId)),
        msg.sender,
        uint256(amount)   // ← passed verbatim
    );
    slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
        ...
        tx: abi.encodePacked(
            uint8(TransactionType.DepositCollateral),
            abi.encode(DepositCollateral({
                sender: subaccount,
                productId: productId,
                amount: amount   // ← original param, NOT actual received
            }))
        )
    });
``` [1](#0-0) 

`handleDepositTransfer` performs two sequential transfers:

```solidity
// EndpointStorage.sol lines 111-119
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);              // Endpoint ← depositor
    safeTransferTo(token, address(clearinghouse), amount); // Clearinghouse ← Endpoint
}
``` [2](#0-1) 

`safeTransferFrom` and `safeTransfer` in `ERC20Helper` only verify the boolean success flag — they do **not** return or validate the actual amount received:

```solidity
// ERC20Helper.sol lines 23-42
function safeTransferFrom(IERC20Base self, address from, address to, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(...);
    require(success && (data.length == 0 || abi.decode(data, (bool))), ERR_TRANSFER_FAILED);
}
``` [3](#0-2) 

For a fee-on-transfer token where the fee is charged on `transfer` (not `transferFrom`):

| Step | Operation | Endpoint balance | Clearinghouse receives |
|---|---|---|---|
| 1 | `safeTransferFrom(token, depositor, amount)` | `+amount` | — |
| 2 | `safeTransferTo(token, clearinghouse, amount)` | `-amount` | `amount − fee` |

The slow-mode tx records `amount`. When the sequencer processes it, `Clearinghouse.depositCollateral` credits the full `amount` to the subaccount:

```solidity
// Clearinghouse.sol lines 205-208
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [4](#0-3) 

The Clearinghouse holds `amount − fee` tokens but credits `amount` — a `fee`-sized discrepancy per deposit.

---

### Impact Explanation

The Clearinghouse's actual token balance becomes less than the sum of all credited subaccount balances. As the discrepancy accumulates across multiple deposits, the protocol becomes insolvent for that token. Users who withdraw later cannot receive their full credited balance because the Clearinghouse lacks sufficient tokens to cover all withdrawals. This is a direct, concrete accounting corruption with measurable asset impact: the shortfall equals the cumulative transfer fees across all deposits of that token.

---

### Likelihood Explanation

The trigger requires a fee-on-transfer token to be listed as supported collateral (specifically one where the fee applies on `transfer` but not `transferFrom`). If such a token is listed — either intentionally or inadvertently — any depositor calling the `public` `depositCollateralWithReferral` function triggers the discrepancy on every deposit. No special permissions or privileged access are required. The entry path is fully attacker-controlled.

---

### Recommendation

In `handleDepositTransfer`, measure the Clearinghouse's actual balance change after the transfer and pass that realized amount into the slow-mode transaction, rather than the caller-supplied `amount`:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // return actualReceived and use it in the slow-mode tx amount field
}
```

Alternatively, restrict supported collateral tokens to those without fee-on-transfer mechanics and enforce this at token listing time.

---

### Proof of Concept

1. A fee-on-transfer token (1% fee on `transfer`, no fee on `transferFrom`) is listed as collateral for `productId = 1`.
2. Attacker calls `depositCollateralWithReferral(attackerSubaccount, 1, 1000e6, "")`.
3. `handleDepositTransfer` pulls `1000e6` from attacker (no fee), then transfers `1000e6` to Clearinghouse — Clearinghouse receives `990e6` (1% fee deducted).
4. Slow-mode tx records `amount = 1000e6`.
5. Sequencer processes the tx; `Clearinghouse.depositCollateral` credits `1000e6` to attacker's subaccount.
6. Attacker's subaccount shows `1000e6` credit; Clearinghouse holds only `990e6`.
7. Repeated across many depositors: Clearinghouse's actual balance falls progressively below total credited balances.
8. Last withdrawers find the Clearinghouse cannot fulfill their full withdrawal, resulting in permanent loss of funds.
