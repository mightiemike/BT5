### Title
Non-Canonical Single-Atom Encoding Accepted by Deserializer — (`src/serde_2026/de.rs`)

### Summary

The `deserialize_2026_body_from_stream` function accepts a multi-atom group encoding (`negative_length_varint + count=1 + atom_bytes`) for a single atom, even though the canonical serializer always emits a singleton encoding (`positive_length_varint + atom_bytes`) for single atoms. There is no guard rejecting `count == 1` in the multi-atom branch. This means two structurally distinct byte streams deserialize to the same `NodePtr` and produce the same tree hash, violating the one-stream-per-consensus-object invariant.

---

### Finding Description

**Serializer (`write_atom_table`, `ser.rs` lines 164–175):**

The serializer enforces a strict rule: a group of exactly one atom always uses the positive-length singleton encoding. [1](#0-0) 

**Deserializer (`deserialize_2026_body_from_stream`, `de.rs` lines 50–64):**

The deserializer branches on the sign of `length_val`. In the negative branch (multi-atom), it reads a `count` varint. The only guard is: [2](#0-1) 

The check `if length == 0 || count == 0` rejects `count == 0` but **does not reject `count == 1`**. A blob with `negative_length_varint + count=1 + atom_bytes` passes this guard and is processed identically to the canonical `positive_length_varint + atom_bytes` form.

**`strict` mode is not a defense.** The `strict` flag in `read_varint` only rejects overlong varint encodings (e.g., encoding `0` in two bytes instead of one). [3](#0-2) 

It has no awareness of the semantic constraint that `count == 1` is non-canonical in the multi-atom branch. Even with `strict=true`, the non-canonical blob is accepted.

The same missing guard exists in `serialized_length_serde_2026`: [4](#0-3) 

Notably, the test suite itself asserts `count > 1` as an invariant of multi-atom groups in `tests.rs` line 377, but this is only a test-side assertion — it is never enforced in the production deserializer: [5](#0-4) 

---

### Impact Explanation

Two distinct byte streams deserialize to the same `NodePtr` and therefore the same tree hash:

- **Canonical:** `group_count=1 | +length=1 | 0x41` → atom `b"A"`
- **Non-canonical:** `group_count=1 | -length=-1 | count=1 | 0x41` → atom `b"A"` (same `NodePtr`, same tree hash)

Any system that uses serde_2026 for consensus-critical serialization (puzzle reveals, solution blobs, coin spends) and relies on the property that a given tree has exactly one valid wire encoding is broken. An attacker can present a non-canonical blob that:
1. Deserializes to the same tree as the canonical blob.
2. Produces the same tree hash.
3. Is byte-for-byte distinct from the canonical encoding.

This breaks blob-level deduplication, any signature-over-wire-bytes scheme, and any cache or mempool keyed on raw blob bytes rather than tree hash.

---

### Likelihood Explanation

The attacker-controlled entrypoint is `deserialize_2026_body_from_stream` / `deserialize_2026`, which are public production APIs exported from `mod.rs`: [6](#0-5) 

Any caller that accepts attacker-supplied serde_2026 blobs is reachable. No special privileges are required — only the ability to submit a crafted blob.

---

### Recommendation

Add a `count == 1` rejection in the multi-atom (negative `length_val`) branch of both `deserialize_2026_body_from_stream` and `serialized_length_serde_2026`:

```rust
// de.rs, inside the `length_val < 0` branch, after reading count:
if length == 0 || count == 0 || count == 1 {
    return Err(EvalErr::SerializationError);
}
```

This enforces the invariant that the multi-atom encoding is only used when `count >= 2`, matching the serializer's behavior in `write_atom_table`.

---

### Proof of Concept

Construct the following two body blobs (no magic prefix) and call `deserialize_2026_body_from_stream` on each:

```
Canonical (singleton encoding of atom b"A"):
  group_count = 1  → varint 0x01
  length      = 1  → varint 0x01  (positive → singleton)
  atom bytes       → 0x41
  instr_count = 1  → varint 0x01
  instruction = 2  → varint 0x02  (push atom[0])
  Body: [0x01, 0x01, 0x41, 0x01, 0x02]

Non-canonical (multi-atom encoding with count=1):
  group_count = 1  → varint 0x01
  length      = -1 → varint 0x7f  (negative → multi-atom branch)
  count       = 1  → varint 0x01  ← NOT rejected by de.rs line 62
  atom bytes       → 0x41
  instr_count = 1  → varint 0x01
  instruction = 2  → varint 0x02
  Body: [0x01, 0x7f, 0x01, 0x41, 0x01, 0x02]
```

Both calls succeed and return a `NodePtr` for atom `b"A"`. Re-serializing both with `serialize_2026` produces identical bytes. The tree hashes are identical. The wire blobs are not. [7](#0-6)
