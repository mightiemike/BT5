Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, enabling allowlist bypass or permanent swap breakage — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is always `msg.sender` of the `pool.swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the originating user. The extension therefore gates on the router's allowlist entry rather than the individual user's, producing either a complete allowlist bypass (if the router is allowlisted) or permanent breakage of router-based swaps for all users (if it is not).

## Finding Description

`MetricOmmPool.swap` unconditionally passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← always the direct caller of pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to every registered extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` then uses that `sender` argument to look up the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When the call originates from `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`), the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
  );
```

The router never passes the originating user address into `swap()`; it is not a parameter of the function signature. Consequently, `sender` arriving at the extension is always the router address, and the check resolves to `allowedSwapper[pool][router]` regardless of which user initiated the transaction.

The `addLiquidity` path avoids this problem by accepting an explicit `owner` parameter separate from `msg.sender`, which extensions receive and can gate on independently. No equivalent `swapper` parameter exists in `swap()`.

**Outcome A — Allowlist bypass:** A pool admin who wants allowlisted users to be able to use the router must add the router to the allowlist. Once `allowedSwapper[pool][router] = true`, any address — including non-allowlisted users — can call `exactInputSingle` and the check passes, silently defeating the per-user gate for the entire pool.

**Outcome B — Broken swap functionality:** If the admin does not allowlist the router, every call through the router reverts with `NotAllowedToSwap`, even for users whose individual entries are `true`. The primary swap interface is permanently unusable for all users of that pool.

## Impact Explanation

Both outcomes constitute broken core pool functionality causing loss of funds or an unusable swap flow, matching the allowed impact gate. Outcome A is a direct fund-loss path: an unprivileged, non-KYC'd user can drain pool liquidity at oracle price on a pool explicitly configured to restrict access. Outcome B renders the documented primary swap interface permanently non-functional for all users of any pool that deploys `SwapAllowlistExtension` with the `beforeSwap` hook enabled. Severity: Critical (Outcome A) / High (Outcome B).

## Likelihood Explanation

`MetricOmmSimpleRouter` is the documented primary swap interface. Any pool deployer who configures `SwapAllowlistExtension` and expects users to interact via the router immediately encounters one of the two failure modes. Outcome A requires only the natural operational step of adding the router to the allowlist to unblock legitimate users, after which the bypass is silently active for all callers. No special privileges, flash loans, or unusual conditions are required; a standard `exactInputSingle` call suffices.

## Recommendation

The root cause is that `pool.swap()` has no explicit `swapper` parameter analogous to the `owner` parameter in `addLiquidity`. Two complementary fixes:

1. **Protocol-level (preferred):** Add an explicit `swapper` parameter to `IMetricOmmPoolActions.swap()` and pass it through `_beforeSwap` to extensions. `MetricOmmSimpleRouter` would populate this with `msg.sender` (the originating user). This mirrors the `addLiquidity` / `DepositAllowlistExtension` pattern exactly.

2. **Extension-level workaround:** Have `SwapAllowlistExtension.beforeSwap` decode a trusted `swapper` address from `extensionData` when `sender` is a known router, and gate on that decoded address. This is less robust because it requires router cooperation and correct encoding by callers.

Fix 1 is strongly preferred as it closes the gap at the protocol level for all current and future extensions.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension, beforeSwap hook enabled
  - allowedSwapper[pool][alice] = true   (alice is KYC'd)
  - allowedSwapper[pool][bob]   = false  (bob is not KYC'd)
  - allowedSwapper[pool][router] = true  (admin adds router to unblock alice)

Attack (Outcome A — bypass):
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → pool.swap(...) with msg.sender = router
    → _beforeSwap(sender=router, ...)
    → SwapAllowlistExtension.beforeSwap(sender=router)
    → allowedSwapper[pool][router] == true → PASSES
  bob successfully swaps on an allowlisted pool he was never permitted to access.

Broken functionality (Outcome B — if router not allowlisted):
  alice calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → pool.swap(...) with msg.sender = router
    → _beforeSwap(sender=router, ...)
    → SwapAllowlistExtension.beforeSwap(sender=router)
    → allowedSwapper[pool][router] == false → REVERTS NotAllowedToSwap
  alice cannot use the router despite being explicitly allowlisted.
```

A Foundry integration test can reproduce both paths by deploying a pool with the extension, configuring the allowlist as above, and calling `exactInputSingle` from `bob`'s address after the router is allowlisted.
