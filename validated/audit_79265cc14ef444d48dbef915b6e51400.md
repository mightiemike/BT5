Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual caller/payer) and checks only `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` imposes no constraint that `msg.sender == owner`, any unprivileged address can specify an allowlisted address as `owner`, pay the tokens themselves via the liquidity callback, and pass the allowlist check — fully circumventing a pool guard that is supposed to restrict who may provide liquidity.

## Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives two identity arguments: `sender` (the address that called `pool.addLiquidity`) and `owner` (the intended position holder). The implementation discards `sender` entirely (unnamed parameter) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner`, with no requirement that they match: [2](#0-1) 

The pool then fires the liquidity callback on `msg.sender` (the actual caller) to collect payment, not on `owner`: [3](#0-2) 

`removeLiquidity` enforces `msg.sender == owner`, so only `owner` can withdraw: [4](#0-3) 

**Direct pool exploit path:**
1. Attacker (not allowlisted) implements `IMetricOmmModifyLiquidityCallback` and calls `pool.addLiquidity(allowlistedUser, salt, deltas, callbackData, extensionData)`.
2. Pool calls `_beforeAddLiquidity(attacker, allowlistedUser, ...)`.
3. Extension checks `allowedDepositor[pool][allowlistedUser]` → `true` → no revert.
4. `LiquidityLib.addLiquidity` mints shares to `allowlistedUser`; callback fires on attacker, attacker pays tokens.
5. With `allowlistedUser`'s cooperation, they call `removeLiquidity` and split proceeds.

**Periphery exploit path via `MetricOmmPoolLiquidityAdder`:**

`addLiquidityExactShares(pool, owner, ...)` accepts an arbitrary `owner` and uses `msg.sender` as the payer: [5](#0-4) 

`_validateOwner` only rejects `address(0)`, not non-`msg.sender` addresses: [6](#0-5) 

When `MetricOmmPoolLiquidityAdder` calls `pool.addLiquidity(allowlistedUser, ...)`, the pool's `msg.sender` is the adder contract, and the extension sees `owner = allowlistedUser` → passes. The adder's callback pulls tokens from the attacker (stored payer). This provides a second, router-assisted entry point requiring no custom callback contract.

## Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting who may provide liquidity (e.g., institutional-only pools, KYC-gated pools). The bypass allows any unprivileged address to inject liquidity into a restricted pool by routing through an allowlisted address as `owner`. With collusion, the attacker recovers their tokens and the allowlist provides zero protection. Even without collusion, the attacker can force unauthorized liquidity into the pool, altering its depth and price dynamics in ways the admin did not sanction. This is a direct admin-boundary break: an unprivileged path defeats a configured pool guard.

## Likelihood Explanation

High. The bypass requires no special role or privilege — only knowledge of one allowlisted address (which is on-chain public state via `allowedDepositor`) and willingness to call `pool.addLiquidity` directly or use `MetricOmmPoolLiquidityAdder.addLiquidityExactShares`. No existing guard in the pool or extension prevents this. The `MetricOmmPoolLiquidityAdder` provides a second entry point that requires no custom callback implementation, lowering the technical barrier further.

## Recommendation

Check `sender` (the actual payer/caller) instead of `owner` in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the payer and the position owner, both should be checked. Additionally, `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` should enforce `owner == msg.sender` or be explicitly documented as incompatible with deposit allowlist pools.

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension`; only `allowlistedUser` is in `allowedDepositor[pool]`.
2. Attacker implements `IMetricOmmModifyLiquidityCallback` and calls:
   ```solidity
   pool.addLiquidity(allowlistedUser, salt, deltas, callbackData, extensionData);
   ```
3. Pool calls `_beforeAddLiquidity(attacker, allowlistedUser, ...)` via `ExtensionCalling._beforeAddLiquidity`.
4. Extension checks `allowedDepositor[pool][allowlistedUser]` → `true` → no revert.
5. `LiquidityLib.addLiquidity` mints shares to `allowlistedUser`; callback fires on attacker, attacker pays tokens.
6. `allowlistedUser` calls `pool.removeLiquidity(allowlistedUser, ...)`, receives tokens, shares proceeds with attacker.
7. Net result: attacker provided liquidity to a restricted pool; the deposit allowlist was never enforced against the actual payer.
