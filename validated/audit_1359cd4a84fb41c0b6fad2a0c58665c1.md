### Title
Missing Zero-Address Validation for `wrappedNative`, `endpoint`, and `spotEngine` in `DirectDepositV1` Constructor тАФ (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.sol`'s constructor accepts `_endpoint`, `_spotEngine`, and `_wrappedNative` as address parameters without any zero-address guard. `ContractOwner.initialize()` similarly stores `_wrappedNative` without validation. Because `createDirectDepositV1()` and `creditDepositV1()` are both publicly callable with no access control, any user can trigger DDA deployment. If `wrappedNative` is `address(0)` in `ContractOwner`, every DDA deployed will silently burn all ETH sent to it.

---

### Finding Description

`DirectDepositV1` constructor assigns all three address parameters directly with no `require(addr != address(0))` guard: [1](#0-0) 

`ContractOwner.initialize()` stores `_wrappedNative` into state without validation: [2](#0-1) 

`createDirectDepositV1()` is declared `public` with no `onlyOwner` or `onlyDeployer` modifier, and passes `ContractOwner`'s stored `wrappedNative` directly into the DDA constructor: [3](#0-2) 

`creditDepositV1()` is `external` with no access control and lazily calls `createDirectDepositV1()` on first use: [4](#0-3) 

Once a DDA is deployed with `wrappedNative = address(0)`, its `receive()` function forwards all incoming ETH to `address(0)`: [5](#0-4) 

A low-level `.call{value: X}("")` to `address(0)` returns `success = true` in the EVM (address(0) is treated as an empty account, not a revert). The `require(success, ...)` check therefore passes, and the ETH is permanently burned with no error surfaced to the caller.

---

### Impact Explanation

If `ContractOwner.wrappedNative` is `address(0)` (due to the missing validation in `initialize()`), any user who sends native ETH to a DDA address will have their funds permanently burned. The `receive()` path silently succeeds, the user receives no error, and the ETH is unrecoverable. Additionally, `creditDeposit()` calls `spotEngine.getProductIds()` on `address(0)`, which will revert, making the DDA permanently non-functional for ERC-20 deposits as well. [5](#0-4) 

---

### Likelihood Explanation

The prerequisite is that `ContractOwner.wrappedNative` is `address(0)`, which requires the deployer to have passed `address(0)` during `initialize()`. The missing validation in `initialize()` makes this a realistic misconfiguration risk. Once the misconfiguration exists, the impact is triggered by any unprivileged user calling the publicly accessible `creditDepositV1()` or `createDirectDepositV1()`, or by any user sending ETH to an already-deployed DDA. [3](#0-2) 

---

### Recommendation

Add zero-address guards in both places:

1. In `DirectDepositV1` constructor, add:
   ```solidity
   require(_endpoint != address(0), "endpoint is zero");
   require(_spotEngine != address(0), "spotEngine is zero");
   require(_wrappedNative != address(0), "wrappedNative is zero");
   ```

2. In `ContractOwner.initialize()`, add:
   ```solidity
   require(_wrappedNative != address(0), "wrappedNative is zero");
   require(_endpoint != address(0), "endpoint is zero");
   require(_spotEngine != address(0), "spotEngine is zero");
   ``` [1](#0-0) [6](#0-5) 

---

### Proof of Concept

1. Deploy `ContractOwner` and call `initialize()` with `_wrappedNative = address(0)` тАФ no revert occurs.
2. Any user calls `creditDepositV1(someSubaccount)`. Since no DDA exists yet, `createDirectDepositV1()` is called, deploying a `DirectDepositV1` with `wrappedNative = address(0)`.
3. A user sends 1 ETH to the deployed DDA address. `receive()` executes `address(0).call{value: 1 ether}("")`, which returns `(true, "")`. `require(success)` passes. The 1 ETH is permanently burned.
4. The user has no recourse; the ETH is gone with no on-chain error. [5](#0-4) [4](#0-3)
