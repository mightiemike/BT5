### Title
Unbounded `fee_proposal_fri` During Startup/Transition Window Permanently Corrupts `fee_actual` and Future L2 Gas Prices — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The SNIP-35 fee-proposal validation in `is_proposal_init_valid` intentionally skips the `fee_proposal_fri` bounds check when `proposal_init_validation.fee_actual` is `None`. This `None` state persists for the entire `window_size`-block period after V0_14_3 activation (or after any node restart at a height below `window_size`), because `compute_fee_actual` returns `None` whenever any block in the sliding window carries `fee_proposal_fri = None` (pre-V0_14_3 blocks). During this window, a proposer can set `fee_proposal_fri` to any value — including `u128::MAX` or `0` — and every validator will accept it. These unchecked values are stored in `BlockHeaderWithoutHash.fee_proposal_fri` and in the in-memory `fee_proposals_window`. Once the window fills with V0_14_3 blocks, `compute_fee_actual` computes the median of the stored values, which becomes the `fee_actual` that bounds all future proposals and drives `calculate_next_l2_gas_price_for_fin`. A proposer controlling a majority of blocks in the transition window can permanently anchor `fee_actual` at an extreme value.

---

### Finding Description

**Step 1 — The bypass.**

In `is_proposal_init_valid`, the bounds check is guarded by a pattern match that fires only when both `fee_actual` and `fee_proposal` are `Some`:

```rust
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    // fee_proposal_bounds check
}
``` [1](#0-0) 

**Step 2 — When `fee_actual` is `None`.**

`compute_fee_actual` returns `None` whenever any block in the `[height - window_size, height)` range is absent from the window map or carries `fee_proposal_fri = None`. Pre-V0_14_3 blocks always carry `None`:

```rust
Some(None) | None => {
    warn!("... fee_proposals_window has no recorded fee_proposal for height {source_height}");
    return None;
}
``` [2](#0-1) 

This means `fee_actual` is `None` for the entire `window_size` (10) blocks after V0_14_3 activation, because the window still contains pre-V0_14_3 `None` entries.

**Step 3 — The extreme value is accepted and stored.**

A malicious proposer sets `fee_proposal_fri = Some(u128::MAX)` in `ProposalInit`. The validator skips the bounds check (Step 1). The commitment `Poseidon(partial_block_hash, u128::MAX)` is computed and matches `ProposalFin.proposal_commitment`, so the proposal is accepted. [3](#0-2) 

The value is then stored in two places:

1. In `BlockHeaderWithoutHash.fee_proposal_fri` via `update_state_sync_with_new_block`: [4](#0-3) 

2. In the in-memory `fee_proposals_window` via `record_fee_proposal` called from `finalize_decision`: [5](#0-4) 

**Step 4 — Corruption of `fee_actual` after the window fills.**

Once the window contains only V0_14_3 blocks, `compute_fee_actual` computes the median of the stored values. If the attacker controlled ≥ 6 of the 10 transition blocks and set each to `u128::MAX`, the median is `u128::MAX`. This becomes the `fee_actual` passed to `calculate_next_l2_gas_price_for_fin`: [6](#0-5) 

**Step 5 — Downstream effects.**

All future `fee_proposal` values are bounded within `±0.2%` of `u128::MAX` (the `fee_proposal_margin_ppt = 2` constraint). The L2 gas price is set to an astronomically high value, making all transactions fail or charge extreme fees. The `proposal_commitment` for every future block is computed from this corrupted `fee_proposal_fri`, producing wrong commitments that are forwarded to the centralized cende pipeline: [7](#0-6) 

---

### Impact Explanation

- **Wrong state committed**: `BlockHeaderWithoutHash.fee_proposal_fri` stores an extreme value, which is persisted to `StorageBlockHeader` and served by the RPC layer. [8](#0-7) 

- **Wrong `proposal_commitment`**: The consensus commitment `Poseidon(partial_block_hash, fee_proposal_fri)` is computed from the extreme value and signed by validators, binding the wrong hash into the chain. [9](#0-8) 

- **Wrong L2 gas price for all future blocks**: `fee_actual = u128::MAX` drives `calculate_next_l2_gas_price_for_fin` to an extreme value, causing all transactions to fail or pay extreme fees — a direct economic impact on every user.

- **Wrong fee estimation**: RPC fee estimation reads the corrupted `fee_actual` and returns authoritative-looking wrong values.

Matches: *Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact* and *High — RPC execution, fee estimation returns an authoritative-looking wrong value*.

---

### Likelihood Explanation

The V0_14_3 activation is a one-time network upgrade event. The transition window is exactly `fee_proposal_window_size = 10` blocks. [10](#0-9) 

A validator controlling a simple majority of the 10 transition blocks (≥ 6) can set the median to any desired value. In a small or concentrated validator set, this is achievable by a single large validator or a small coalition. The attack requires no special privilege beyond being selected as proposer during the transition window — the same role any validator legitimately holds.

---

### Recommendation

Replace the unconditional skip with a fallback bound. When `fee_actual` is `None`, enforce that `fee_proposal_fri` lies within a reasonable range anchored to the current `l2_gas_price` (the same fallback the proposer uses in `compute_proposer_fee_proposal`):

```rust
// When fee_actual is None (startup/transition), bound fee_proposal to
// [l2_gas_price / (1 + margin), l2_gas_price * (1 + margin)] as a fallback.
let effective_fee_actual = proposal_init_validation.fee_actual
    .unwrap_or(proposal_init_validation.l2_gas_price_fri);
if let Some(fee_proposal) = init_proposed.fee_proposal_fri {
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        effective_fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(...);
    }
}
```

This preserves the intended behavior (no oracle-derived `fee_actual` during startup) while preventing extreme values from being injected into the window.

---

### Proof of Concept

1. Network activates V0_14_3 at block height `H`. The `fee_proposals_window` contains blocks `[H - 10, H)`, all with `fee_proposal_fri = None` (pre-V0_14_3).

2. A malicious validator is selected as proposer for blocks `H, H+1, ..., H+5` (6 consecutive blocks).

3. For each of these blocks, the proposer sets `fee_proposal_fri = Some(GasPrice(u128::MAX))` in `ProposalInit`.

4. Each validator calls `is_proposal_init_valid`. `proposal_init_validation.fee_actual` is `None` (because the window still contains `None` entries from pre-V0_14_3 blocks). The bounds check at lines 398–416 is skipped. The proposal is accepted.

5. `record_fee_proposal(height, Some(GasPrice(u128::MAX)))` is called for each of the 6 blocks.

6. At block `H + 10`, the window `[H, H+10)` contains 6 entries with `u128::MAX` and 4 entries with normal values. `compute_fee_actual` sorts the window and returns the median = `u128::MAX`.

7. `calculate_next_l2_gas_price_for_fin` receives `fee_actual = u128::MAX`. The next L2 gas price is set to `u128::MAX`.

8. All subsequent transactions submitted to the gateway fail fee validation or are charged `u128::MAX` fees. RPC fee estimation returns `u128::MAX`. The chain is effectively halted for normal users.

/// Bind `fee_proposal_fri` to the proposal commitment hash.
///
/// Pre-V0_14_3 blocks have `fee_proposal = None` and the commitment is just `partial.0`,
/// preserving on-chain behavior. From V0_14_3 onward, the commitment is
/// `Poseidon(partial.0, fee_proposal_fri)`, so a proposer cannot equivocate on
/// `fee_proposal_fri` without changing the commitment that consensus signs over.
///
/// This computation lives in the orchestrator (not in `starknet_api`) because the
/// fee_proposal is a consensus-time signal — `starknet_api`'s block-hash primitives
/// remain unaware of it, and `BlockHash` is unaffected.
pub(crate) fn proposal_commitment_from(
    partial: PartialBlockHash,
    fee_proposal: Option<GasPrice>,
) -> ProposalCommitment {
    let Some(fee_proposal) = fee_proposal else {
        return ProposalCommitment(partial.0);
    };
    ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
}
```
