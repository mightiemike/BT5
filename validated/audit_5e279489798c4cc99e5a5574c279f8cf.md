### Title
Spread Health Overestimation via Hardcoded `longWeight` in `getCoreRisk` for Short Perp Positions — (`core/contracts/BaseEngine.sol`, `core/contracts/libraries/RiskHelper.sol`)

---

### Summary

`getCoreRisk` unconditionally returns `longWeight` for every position by passing a hardcoded `1` to `_getWeightX18`. For a long-spot / short-perp spread, both `existingWeight` and `spreadWeight` are therefore computed with `perpLongWeight` instead of `perpShortWeight`. Because `perpLongWeight < perpShortWeight`, the term `(spreadWeight − existingWeight)` is inflated, and the spread health adjustment is more positive than the protocol's own accounting warrants. A subaccount can borrow beyond what its real collateral supports.

---

### Finding Description

**Root cause — `getCoreRisk` always returns `longWeight`:** [1](#0-0) 

The third field of `CoreRisk` is populated by `_getWeightX18(risk, 1, healthType)`. The literal `1` is always `>= 0`, so `_getWeightX18` always takes the `longWeight` branch: [2](#0-1) 

The actual position direction (`amount`) is read on line 185 and stored in `CoreRisk.amount`, but it is **never forwarded** to `_getWeightX18`. `CoreRisk.longWeight` is therefore always the long weight, regardless of whether the subaccount is long or short.

**How `getHealth` uses this field:** [3](#0-2) 

`existingWeight` averages `spotCoreRisk.longWeight` and `perpCoreRisk.longWeight`. For a long-spot / short-perp spread, the perp engine's `_calculateProductHealth` correctly applied `perpShortWeight` (> 1) to the negative perp balance, but `existingWeight` uses `perpLongWeight` (< 1) — an underestimate.

**`_getSpreadWeightX18` compounds the error:** [4](#0-3) 

When `spotCoreRisk.amount > 0`, `spreadWeight` is derived from `perpCoreRisk.longWeight`. With `perpLongWeight = 0.9` this gives `spreadWeight = 0.98`; the correct derivation from `perpShortWeight = 1.1` gives `1.02`, capped at `0.99`.

**Net overestimation (concrete numbers):**

| | Buggy | Correct |
|---|---|---|
| `existingWeight` | `(0.9 + 0.9)/2 = 0.90` | `(0.9 + 1.1)/2 = 1.00` |
| `spreadWeight` | `1 − 0.1/5 = 0.98` | `min(1.02, 0.99) = 0.99` |
| `spreadWeight − existingWeight` | **+0.08** | **−0.01** |

Spread health adjustment = `basisAmount × (spotPrice + perpPrice) × Δ`

With `basisAmount = 1`, `price = 100`: buggy adds **+16**, correct subtracts **−2** — a **+18 unit overestimation** per unit of spread size.

---

### Impact Explanation

The inflated health value allows a subaccount holding a long-spot / short-perp spread to pass `INITIAL` health checks with less real collateral than required. The subaccount can withdraw collateral or open additional positions that would be rejected under correct accounting, leaving the protocol undercollateralized. This is a direct solvency failure matching the Critical scope.

---

### Likelihood Explanation

The path is fully permissionless: any user can open a spot long and a perp short on a registered spread pair through the normal order/deposit flow, then call any action that triggers `getHealth`. No special privileges are required. The overestimation scales linearly with position size, so the impact grows with capital.

---

### Recommendation

In `getCoreRisk`, pass the actual `amount` (not the literal `1`) to `_getWeightX18` so that `CoreRisk.longWeight` reflects the direction-correct weight:

```solidity
// BaseEngine.sol — getCoreRisk
return IProductEngine.CoreRisk(
    amount,
    risk.priceX18,
-   RiskHelper._getWeightX18(risk, 1, healthType)
+   RiskHelper._getWeightX18(risk, amount, healthType)
);
```

Rename the field to `weight` (not `longWeight`) in `CoreRisk` and update all consumers accordingly to avoid future confusion.

---

### Proof of Concept

```solidity
// Invariant: getHealth(INITIAL) must not exceed the manually computed health
// for a (spotAmount > 0, perpAmount < 0) spread position.

// Setup:
//   perpLongWeightInitial  = 0.9e18
//   perpShortWeightInitial = 1.1e18
//   spotLongWeightInitial  = 0.9e18
//   spotPrice = perpPrice  = 100e18
//   spotAmount = 1e18, perpAmount = -1e18

// Individual health already applied by engines:
//   spot:  1e18 * 100e18 * 0.9  =  90e18
//   perp: -1e18 * 100e18 * 1.1  = -110e18
//   subtotal = -20e18

// Buggy spread adjustment:
//   existingWeight = (0.9 + 0.9) / 2 = 0.9
//   spreadWeight   = 1 - (1 - 0.9)/5 = 0.98
//   delta          = 0.98 - 0.9 = 0.08
//   adjustment     = 1 * 200 * 0.08 = +16
//   getHealth()    = -20 + 16 = -4   ← passes INITIAL check if threshold is -5

// Correct spread adjustment:
//   existingWeight = (0.9 + 1.1) / 2 = 1.0
//   spreadWeight   = min(1.02, 0.99) = 0.99
//   delta          = 0.99 - 1.0 = -0.01
//   adjustment     = 1 * 200 * (-0.01) = -2
//   correct health = -20 + (-2) = -22  ← fails INITIAL check

// assert(getHealth(subaccount, INITIAL) <= -22e18);  // will fail with current code
```
