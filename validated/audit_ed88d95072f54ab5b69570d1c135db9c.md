### Title
`ValidResourceBounds` Variant Selection in `ResourceBoundsMapping::from` Causes Simulation to Compute Wrong Transaction Hash for V3 Transactions with Zero `l2_gas`/`l1_data_gas` — (`File: crates/apollo_rpc/src/v0_8/transaction.rs`)

---

### Summary

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in the RPC layer silently selects the `L1Gas` variant when both `l2_gas` and `l1_data_gas` are zero. Because `get_tip_resource_bounds_hash` produces a structurally different hash for `L1Gas` (two resources) versus `AllResources` (three resources), the transaction hash computed during `starknet_simulateTransactions` / `starknet_estimateFee` diverges from the hash the sequencer computes at actual admission for the same V3 transaction. The `get_execution_info` syscall inside the simulation therefore returns an authoritative-looking but wrong transaction hash.

---

### Finding Description

`ValidResourceBounds` has two variants:

```
L1Gas(ResourceBounds)          // pre-0.13.3, omits L1_DATA_GAS from hash
AllResources(AllResourceBounds) // 0.13.3+, includes L1_DATA_GAS in hash
```

The conversion used in the simulation path is:

```rust
// crates/apollo_rpc/src/v0_8/transaction.rs:188-199
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← selects L1Gas silently
        } else {
            Self::AllResources(AllResourceBounds { ... })
        }
    }
}
```

The hash function branches on this variant:

```rust
// crates/starknet_api/src/transaction_hash.rs:188-211
pub fn get_tip_resource_bounds_hash(...) -> Result<Felt, ...> {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                    // ← 2 elements
        ValidResourceBounds::AllResources(all) => {
            vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?]  // ← 3 elements
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

Poseidon hash of `[tip, L1_GAS, L2_GAS]` ≠ Poseidon hash of `[tip, L1_GAS, L2_GAS, L1_DATA_GAS(0)]`, so the two variants produce **different transaction hashes** even when the numeric values are identical.

The gateway admission path never uses this conversion. `InternalRpcInvokeTransactionV3` stores `AllResourceBounds` directly and its `InvokeTransactionV3Trait::resource_bounds()` always returns `ValidResourceBounds::AllResources(self.resource_bounds)`:

```rust
// crates/starknet_api/src/rpc_transaction.rs:636-639
impl InvokeTransactionV3Trait for InternalRpcInvokeTransactionV3 {
    fn resource_bounds(&self) -> ValidResourceBounds {
        ValidResourceBounds::AllResources(self.resource_bounds)  // always AllResources
    }
```

So for any V3 transaction with `l2_gas = 0` and `l1_data_gas = 0`:

| Path | Variant chosen | Hash includes L1_DATA_GAS? |
|---|---|---|
| `starknet_simulateTransactions` | `L1Gas` | No |
| Gateway admission / actual execution | `AllResources` | Yes (zero value) |

The hashes diverge.

---

### Impact Explanation

The `get_execution_info` syscall exposes the transaction hash to the executing contract. When a developer or wallet calls `starknet_simulateTransactions` or `starknet_estimateFee` for a V3 transaction with `l2_gas = 0` and `l1_data_gas = 0`, the simulation returns a transaction hash that differs from the hash the sequencer will assign at actual execution. Any contract that reads `get_execution_info().tx_info.transaction_hash` and uses it for replay protection, unique-ID storage, or signature verification will observe a different value in simulation than in execution. This makes simulation an unreliable oracle for such contracts and can cause developers to incorrectly conclude their contract logic is correct when it will actually fail (or vice versa).

---

### Likelihood Explanation

V3 transactions with `l2_gas = 0` and `l1_data_gas = 0` are valid and common: any wallet or SDK that has not yet adopted the full three-resource model will submit exactly this shape. The divergence is silent — no error is returned, and the simulation completes successfully with a plausible-looking but wrong hash. The attacker-controlled input is simply the `resource_bounds` field of a broadcasted transaction.

---

### Recommendation

Remove the `L1Gas`/`AllResources` branching from the simulation-path conversion. For V3 transactions, always use `ValidResourceBounds::AllResources`, even when `l2_gas` and `l1_data_gas` are zero:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        // Always AllResources for V3; L1Gas is only for pre-0.13.3 stored transactions.
        Self::AllResources(AllResourceBounds {
            l1_gas: value.l1_gas,
            l1_data_gas: value.l1_data_gas,
            l2_gas: value.l2_gas,
        })
    }
}
```

Alternatively, the `BroadcastedTransaction` → `ExecutableTransactionInput` conversion should bypass `ResourceBoundsMapping` entirely and construct `ValidResourceBounds::AllResources` directly, matching the gateway admission path.

---

### Proof of Concept

1. Construct a V3 invoke transaction with `l2_gas = { max_amount: 0, max_price_per_unit: 0 }` and `l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }`.
2. Call `starknet_simulateTransactions` with this transaction against a contract that reads `get_execution_info().tx_info.transaction_hash` and emits it as an event.
3. Observe the emitted hash H_sim.
4. Submit the same transaction through the gateway and observe the actual transaction hash H_real from the receipt.
5. H_sim ≠ H_real because:
   - Simulation: `ResourceBoundsMapping::from` selects `L1Gas` → `get_tip_resource_bounds_hash` hashes `[tip, L1_GAS, L2_GAS]`
   - Actual: `InternalRpcInvokeTransactionV3::resource_bounds()` returns `AllResources` → `get_tip_resource_bounds_hash` hashes `[tip, L1_GAS, L2_GAS, L1_DATA_GAS(0)]`

    tip: &Tip,

    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.

        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,

    // For new V3 txs, need to also hash the data gas bounds.

        ValidResourceBounds::L1Gas(_) => vec![],

            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
