Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual swapper, making allowlist bypassable via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. Any pool admin who allowlists the router to enable legitimate users to reach the pool via the standard UI simultaneously opens the pool to every address on the network, rendering the allowlist completely inoperative.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to `_beforeSwap`:**

`MetricOmmPool.swap` at line 231 calls `_beforeSwap(msg.sender, ...)`. When the call originates from `MetricOmmSimpleRouter`, `msg.sender` is the router contract address. [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged:**

The value is passed verbatim into `abi.encodeCall` and dispatched to every configured extension. [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`:**

`msg.sender` here is the pool (correct), but `sender` is the router address when the swap originates from the router. The check `allowedSwapper[pool][router]` is evaluated, never `allowedSwapper[pool][actualUser]`. [3](#0-2) 

**Step 4 — All four router entry points call `pool.swap()` with `msg.sender = router`:**

`exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` all call `IMetricOmmPoolActions(pool).swap(...)` directly, so the pool always sees the router as `msg.sender`. [4](#0-3) [5](#0-4) 

**Irresolvable dilemma for pool admins:**

| Router allowlisted? | Allowlisted user via router | Non-allowlisted user via router |
|---|---|---|
| No | Blocked (broken UX) | Blocked |
| Yes | Allowed | **Allowed (bypass)** |

There is no configuration that simultaneously permits allowlisted users to use the router and blocks non-allowlisted users. The exact wrong value checked is `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`.

## Impact Explanation

When the pool admin allowlists the router (the only way to let legitimate users reach the pool through the standard periphery), every address on the network can swap in the restricted pool. Any pool relying on `SwapAllowlistExtension` for regulatory compliance, KYC gating, or exclusive market-making access is fully bypassed by any caller who routes through `MetricOmmSimpleRouter`. This is a direct admin-boundary break: an unprivileged public path nullifies a pool-level access control the admin explicitly configured.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Pool admins who want to restrict swaps will naturally allowlist the router so their permitted users can interact normally. The bypass requires no special privileges, no flash loans, and no unusual token behavior — any EOA can call `exactInputSingle` on the router pointing at the restricted pool. The attack is repeatable and costless beyond gas.

## Recommendation

The extension must check the original user's identity, not the intermediary. The simplest correct fix is for the router to encode the original `msg.sender` into `extensionData`, and for `SwapAllowlistExtension.beforeSwap` to decode and check that value when present. Alternatively, the pool admin must never allowlist the router and instead require users to call the pool directly — but this breaks the standard UX and is not enforced by any protocol mechanism.

## Proof of Concept

```
Setup
─────
1. Pool admin deploys pool with SwapAllowlistExtension (extension2, beforeSwap order = 2).
2. Pool admin allowlists userA (KYC'd trader):
       swapExtension.setAllowedToSwap(pool, userA, true)
3. Pool admin allowlists the router so userA can use the standard UI:
       swapExtension.setAllowedToSwap(pool, address(router), true)

Attack
──────
4. userB (not allowlisted) calls:
       router.exactInputSingle(ExactInputSingleParams{
           pool:          pool,
           tokenIn:       token0,
           tokenOut:      token1,
           zeroForOne:    true,
           amountIn:      1_000e18,
           amountOutMinimum: 0,
           recipient:     userB,
           ...
       })

5. Router calls pool.swap(...) — pool sees msg.sender = router.
6. Pool calls _beforeSwap(sender=router, ...).
7. SwapAllowlistExtension checks allowedSwapper[pool][router] → true → no revert.
8. userB receives token1 output. Allowlist completely bypassed.

Foundry test skeleton:
  vm.prank(userB);
  router.exactInputSingle(params);
  // assert userB received tokens despite not being in allowedSwapper
```
