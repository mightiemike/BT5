### Title
Unprivileged Caller Can Directly Invoke `run_mainchain_gc` to Prematurely Prune Mainchain Headers, Permanently Breaking SPV Verification - (File: contract/src/lib.rs)

### Summary

`run_mainchain_gc` is a public, state-mutating function with no caller access control. Any unprivileged NEAR account can call it directly with an arbitrarily large `batch_size`, bypassing the relayer-gated `submit_blocks` flow that is the intended and controlled entry point for GC. This allows an attacker to prune all eligible mainchain headers in a single call, permanently destroying the on-chain data needed for SPV proof verification of transactions in those blocks.

### Finding Description

The intended execution flow for garbage collection is:

1. A trusted relayer calls `submit_blocks` (protected by `#[trusted_relayer]`)
2. `submit_blocks` internally calls `self.run_mainchain_gc(num_of_headers)` with a bounded `batch_size` equal to the number of headers just submitted — an incremental, controlled pruning [1](#0-0) 

However, `run_mainchain_gc` is also exposed as a standalone public method: [2](#0-1) 

The only decorator on it is `#[pause(except(roles(Role::UnrestrictedRunGC)))]`, which only controls access when the contract is **paused**. When the contract is running normally (unpaused), there is no `#[trusted_relayer]`, `#[private]`, or any other caller restriction. Any NEAR account can call it directly.

The GC logic removes the oldest mainchain headers when `amount_of_headers_we_store > gc_threshold`: [3](#0-2) 

An attacker calls `run_mainchain_gc(u64::MAX)`, which sets `selected_amount_to_remove = total_amount_to_remove` (all eligible blocks), and removes every header older than `gc_threshold` from the tip in a single transaction. In the normal incremental flow, the relayer would remove only `num_of_headers` per submission, leaving eligible-but-not-yet-pruned blocks available for verification for an extended period. The attacker collapses this window to zero.

### Impact Explanation

After the attacker's call, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will permanently panic for any transaction whose containing block was pruned: [4](#0-3) 

The block hash is no longer in `mainchain_header_to_height`, so the lookup panics with `"block does not belong to the current main chain"`. The pruning is irreversible — the headers are deleted from `headers_pool` and `mainchain_height_to_header`. Any downstream contract or user that relied on verifying a transaction in a pruned block can never do so again.

Additionally, as documented in `contract/CLAUDE.md`:

> "If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound` when it cannot walk the chain back to the common ancestor." [5](#0-4) 

An attacker can therefore also permanently prevent a pending chain reorganization by pruning the blocks needed to walk back to the common ancestor.

### Likelihood Explanation

The entry point requires no privileges, no staking, no special role, and no knowledge beyond the contract's account ID. Any NEAR account can call `run_mainchain_gc` at any time the contract is unpaused. The attack is a single transaction with a trivially constructed argument (`batch_size = u64::MAX`). The window of exploitability exists whenever the mainchain size exceeds `gc_threshold`, which is the normal steady-state of a live deployment.

### Recommendation

Add `#[trusted_relayer]` (or an equivalent role-based guard such as `Role::UnrestrictedRunGC`) to `run_mainchain_gc` so that only authorized relayers or privileged roles can invoke it directly. The internal call from `submit_blocks` already passes through the `#[trusted_relayer]` gate on that function, so the internal call path is unaffected. Alternatively, make `run_mainchain_gc` a private method (`pub(crate)`) and remove its external exposure entirely, since its only legitimate caller is `submit_blocks`. [2](#0-1) 

### Proof of Concept

1. Deploy the contract with `gc_threshold = 52704` and allow the relayer to sync the chain until `mainchain_size > gc_threshold` (e.g., 60,000 blocks stored, 7,296 eligible for GC).
2. From any unprivileged NEAR account (no role granted), call:
   ```
   run_mainchain_gc({ "batch_size": 18446744073709551615 })
   ```
3. The contract removes all 7,296 eligible headers in a single call. `mainchain_initial_blockhash` is now advanced by 7,296 blocks.
4. Any user who now calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction in one of those 7,296 pruned blocks receives a panic: `"block does not belong to the current main chain"`.
5. The pruning is permanent — no recovery path exists without redeploying the contract. [6](#0-5)
