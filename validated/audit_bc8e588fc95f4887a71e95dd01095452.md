### Title
Sequencer-Injected `oraclePriceX18` Is Excluded from User's EIP-712 Digest, Enabling Stale-Price NLP Mint/Burn — (`core/contracts/Verifier.sol`, `core/contracts/EndpointTx.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The EIP-712 digest computed for `MintNlp` and `BurnNlp` transactions does not commit to `oraclePriceX18`. This field is the sole determinant of how many NLP tokens a user receives on mint, or how much quote they receive on burn. Because it is outside the signed data, the sequencer can inject any price — including a stale one — without invalidating the user's signature. Users have no on-chain slippage protection, and the NLP valuation used at execution time is not cryptographically bound to the user's intent.

---

### Impact Explanation

- **Mint with stale low price:** `nlpAmount = quoteAmount / oraclePriceX18`. If `oraclePriceX18` is below the true NLP NAV, the user receives more NLP tokens than the current NAV justifies, diluting existing NLP holders.
- **Burn with stale high price:** `quoteAmount = nlpAmount × oraclePriceX18`. If `oraclePriceX18` is above the true NLP NAV, the user extracts more quote than their NLP tokens are worth, draining the NLP pool.
- The global `priceX18[NLP_PRODUCT_ID]` is overwritten with the injected value, corrupting subsequent health calculations for all subaccounts holding NLP collateral.
- Users have zero on-chain slippage protection — they cannot specify a minimum acceptable NLP amount or a maximum acceptable price.

---

### Likelihood Explanation

The NLP NAV is a function of the portfolio value of all NLP pool subaccounts, which fluctuates continuously with market prices. Even without malicious intent, if the sequencer's `oraclePriceX18` lags the true NAV by even a few percent during volatile periods, users minting or burning NLP will receive materially incorrect amounts. The gap between signing time and sequencer inclusion time creates a natural window for stale-price execution. A user who observes a favorable stale price in the pending queue can time their mint or burn to exploit the discrepancy.

---

### Recommendation

Include `oraclePriceX18` in the EIP-712 digest for both `MintNlp` and `BurnNlp`. Update the signature type strings and `computeDigest` in `Verifier.sol`:

```solidity
string internal constant MINT_NLP_SIGNATURE =
    "MintNlp(bytes32 sender,uint128 quoteAmount,uint64 nonce,int128 oraclePriceX18)";

string internal constant BURN_NLP_SIGNATURE =
    "BurnNlp(bytes32 sender,uint128 nlpAmount,uint64 nonce,int128 oraclePriceX18)";
```

And include `signedTx.oraclePriceX18` in the `abi.encode` call inside `computeDigest` for both transaction types. This cryptographically binds the user to a specific price, providing slippage protection equivalent to a limit-price commitment.

---

### Proof of Concept

1. NLP NAV is $1.10 (sequencer has updated product prices but not yet submitted a fresh `oraclePriceX18` for NLP).
2. User signs a `MintNlp` for `quoteAmount = 1000e18` USDC. The signature covers only `{sender, 1000e18, nonce}`.
3. Sequencer batches the transaction with `oraclePriceX18 = 1.00e18` (stale, one hour old).
4. On-chain: `nlpAmount = 1000e18 / 1.00e18 = 1000 NLP`. At the true NAV of $1.10, the user should have received `≈909 NLP`.
5. User holds 91 excess NLP tokens minted at the expense of existing NLP pool holders.
6. User immediately signs a `BurnNlp` for 1000 NLP. Sequencer processes it with the now-current `oraclePriceX18 = 1.10e18`.
7. On-chain: `quoteAmount = 1000 × 1.10e18 = 1100 USDC` returned to user.
8. Net profit: $100 extracted from the NLP pool with no capital at risk beyond the round-trip fee. [9](#0-8) [10](#0-9) [11](#0-10)
