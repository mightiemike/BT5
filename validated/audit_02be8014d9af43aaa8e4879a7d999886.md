### Title
Unhandled Return Value of `transferFrom` in `replaceUsdcEWithUsdc` Enables USDC.e Theft from DDAs — (`core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` at line 616 without checking the return value. If the USDC token returns `false` on failure instead of reverting, execution continues: USDC.e is withdrawn from the DDA and sent to the caller — without the caller ever providing USDC. The same function uses `safeTransfer` on line 618, making the inconsistency explicit.

---

### Finding Description

`replaceUsdcEWithUsdc` is an `external` function with no caller access control (only a chain ID guard). Its intended flow is:

1. Read `balance` = USDC.e held by the target DDA.
2. Pull `balance` USDC from `msg.sender` into the DDA via `transferFrom`.
3. Withdraw USDC.e from the DDA to `ContractOwner` via `DirectDepositV1.withdraw`.
4. Forward the USDC.e to `msg.sender` via `safeTransfer`.

Step 2 uses a raw, unchecked `transferFrom`: [1](#0-0) 

If the USDC token returns `false` (e.g., caller has no allowance or insufficient balance), the call does not revert. Steps 3 and 4 still execute: USDC.e is drained from the DDA and delivered to the attacker for free.

The protocol already defines `ERC20Helper.safeTransferFrom` and imports it via `using ERC20Helper for IERC20Base`: [2](#0-1) [3](#0-2) 

The outbound transfer on line 618 correctly uses `safeTransfer`, but the inbound `transferFrom` on line 616 does not use the safe wrapper — a direct inconsistency within the same function.

---

### Impact Explanation

An attacker who calls `replaceUsdcEWithUsdc(subaccount)` for any DDA that holds USDC.e can receive the full USDC.e balance of that DDA without depositing any USDC, provided the USDC token returns `false` on a failed transfer rather than reverting. The DDA's USDC.e balance is permanently drained. This is a direct asset theft from user-controlled deposit accounts.

---

### Likelihood Explanation

The function is `external` with no access control beyond `block.chainid == 57073`. Any unprivileged caller can invoke it against any subaccount whose DDA holds USDC.e. The exploitability depends on whether the specific USDC deployment on chain 57073 (Ink) returns `false` on failure. Even if the current deployment reverts, the code is structurally unsafe: a token upgrade, migration, or future listing of a non-reverting ERC20 would immediately enable the exploit. The inconsistency with `safeTransfer` on line 618 in the same function confirms this was an oversight.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's own `safeTransferFrom` wrapper, consistent with how outbound transfers are handled in the same function:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`safeTransferFrom` is already available via `using ERC20Helper for IERC20Base` and checks both the call success and the decoded boolean return value. [4](#0-3) 

---

### Proof of Concept

1. A DDA for `subaccount` holds 1000 USDC.e.
2. Attacker calls `replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance.
3. `IERC20Base(usdc).transferFrom(attacker, dda, 1000)` returns `false` — no revert.
4. `DirectDepositV1(dda).withdraw(usdcE)` executes: 1000 USDC.e moves from DDA → `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, 1000)` executes: 1000 USDC.e moves from `ContractOwner` → attacker.
6. Attacker receives 1000 USDC.e having provided 0 USDC. DDA balance is zeroed. [5](#0-4)

```text
    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
