### Title
No Minimum Confirmation Threshold Enforced in SPV Proof Verification Allows Zero-Confirmation Acceptance — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: 0`. The contract enforces only an upper bound (`confirmations <= gc_threshold`) but no lower bound. When `confirmations` is `0`, the finality check is trivially bypassed, and the function returns `true` for any transaction in any main-chain block — including the chain tip — regardless of how recently it was added. Any recipient NEAR contract that gates an action on this `true` result is exposed to a reorg-based double-spend.

---

### Finding Description

In `verify_transaction_inclusion`, the only guard on the caller-supplied `confirmations` field is an upper-bound check:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

No lower bound is enforced. When `args.confirmations = 0`, the subsequent finality check:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

reduces to `u64_value >= 0`, which is unconditionally `true` in Rust. The function proceeds to verify the Merkle proof and returns `true` for any transaction in any main-chain block, including the chain tip (zero confirmations). `verify_transaction_inclusion_v2` delegates directly to `verify_transaction_inclusion` after its coinbase-proof check, inheriting the same flaw:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

The `confirmations` field is part of the public `ProofArgs` / `ProofArgsV2` structs, freely set by any NEAR caller: [4](#0-3) 

---

### Impact Explanation

The corrupted value is the **proof result**: the contract returns `true` for a transaction that has achieved zero on-chain finality. Any recipient NEAR contract that calls `verify_transaction_inclusion_v2` and gates an action (fund release, token mint, state update) on the returned `true` can be exploited:

1. Attacker submits a block containing a target transaction (or uses a real one already on-chain).
2. Attacker immediately calls `verify_transaction_inclusion_v2` with `confirmations: 0`.
3. Contract returns `true` — the Merkle proof is valid, the block is on the main chain, and the confirmation check is bypassed.
4. Recipient contract executes the gated action.
5. Attacker (or natural network variance) causes a reorg that removes the block, reversing the transaction while the action on NEAR has already been committed.

This is a direct analog to the GLP liquidation-threshold issue: just as 10% leftover collateral was too narrow a window to liquidate before bad debt accumulated, `confirmations: 0` is too narrow a finality window to act on before a reorg reverses the underlying Bitcoin transaction.

---

### Likelihood Explanation

**Medium.** For Bitcoin mainnet, causing a reorg requires significant hash power. However:
- The contract also supports Litecoin, Dogecoin, and Zcash, where reorg costs are substantially lower.
- On Dogecoin testnet or Zcash testnet, a single attacker can trivially reorg recent blocks.
- Even on mainnet, natural 1-block reorgs occur regularly; a recipient contract accepting `confirmations: 0` or `confirmations: 1` is exposed to these without any attacker effort.
- The entry path requires no privileged role: any NEAR account can call `verify_transaction_inclusion_v2`.

---

### Recommendation

Enforce a minimum confirmation threshold inside `verify_transaction_inclusion`. Reject `confirmations: 0` explicitly, and document a chain-specific recommended minimum (e.g., 6 for Bitcoin mainnet, higher for lower-hashrate chains):

```rust
require!(args.confirmations >= 1, "confirmations must be at least 1");
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

Consider exposing a `min_confirmations` field in contract state (set at `init`) so the deployer can tune the floor per chain, mirroring the `gc_threshold` pattern already in use.

---

### Proof of Concept

```
1. Deploy BtcLightClient (any chain variant, skip_pow_verification=true for PoC).
2. Submit genesis + 1 block containing transaction T at index 0.
3. Call verify_transaction_inclusion_v2 with:
     tx_id                 = hash(T)
     tx_block_blockhash    = hash of the submitted block (chain tip)
     tx_index              = 0
     merkle_proof          = []          // single-tx block: empty proof
     coinbase_tx_id        = hash(T)     // T is the coinbase
     coinbase_merkle_proof = []
     confirmations         = 0           // ← zero-confirmation bypass
4. Contract returns `true`.
5. Recipient contract releases funds.
6. Relayer submits a competing fork block at the same height with higher chain_work → reorg.
7. The original block (and T) is removed from the main chain; NEAR-side action is irreversible.
```

The `confirmations: 0` path is reachable by any unprivileged NEAR caller through the public `verify_transaction_inclusion_v2` entry point with no additional preconditions.
