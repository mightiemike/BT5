Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Original User, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps on `allowedSwapper[pool][sender]`, where `sender` is the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating EOA. This means the extension checks the router's allowlist status rather than the actual swapper's, completely voiding the per-user curation invariant: if the router is allowlisted, any address can bypass the restriction; if it is not, all allowlisted users are blocked from using the protocol's own standard swap interface.

## Finding Description
`MetricOmmPool.swap()` passes its immediate `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged into every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` the pool sees: [4](#0-3) 

The same pattern holds for `exactInput` (multi-hop), `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

There is no mechanism in the pool, the extension calling path, or the router to propagate the originating EOA to the extension. The `extensionData` field is passed through but the router does not encode the original `msg.sender` into it, and the extension does not decode it. No transient-storage context carries the original caller to the extension. The existing `allowAllSwappers` escape hatch does not fix the per-user check. All guards are therefore insufficient.

## Impact Explanation
**Bypass (High):** A pool admin who allowlists the router — the natural configuration to permit standard periphery access — inadvertently grants every on-chain address the ability to swap on a curated pool. Any EOA calls `MetricOmmSimpleRouter.exactInputSingle` targeting the pool; the extension checks `allowedSwapper[pool][router] == true` and passes regardless of whether the originating EOA is on the allowlist. The curation policy is completely voided, allowing unauthorized traders to drain liquidity at oracle-derived prices from a pool intended to be restricted. This is a direct loss of user principal and LP assets above Sherlock thresholds.

**DoS (Medium):** If the router is not allowlisted, every allowlisted user who attempts to swap through the router is blocked because the extension checks `allowedSwapper[pool][router] == false`. The protocol's own canonical swap interface becomes unusable for all curated pools, breaking core swap functionality.

## Likelihood Explanation
The router is the canonical user-facing entry point. Pool admins configuring a curated pool will naturally either (a) allowlist the router to permit standard access — triggering the bypass — or (b) not allowlist it — triggering the DoS. Both outcomes are reachable by any unprivileged user with no special preconditions beyond a pool having `SwapAllowlistExtension` in its `BEFORE_SWAP_ORDER`. No privileged role, special token, or off-chain data is required. The attack is repeatable on every router-mediated swap.

## Recommendation
The extension must gate on the **original user**, not the immediate pool caller. Two sound approaches:

1. **Extension-data forwarding:** Require the router to ABI-encode the originating `msg.sender` into `extensionData` for each hop, and have `SwapAllowlistExtension.beforeSwap` decode and check that address instead of `sender`. This requires no interface changes but relies on router cooperation.
2. **Dedicated sender field:** Add an `originalSender` parameter to `IMetricOmmExtensions.beforeSwap` that the pool populates from a transient-storage context set by the router before calling `pool.swap()`, analogous to how the router already stores payer context for callbacks via `_setNextCallbackContext`. This is the more robust approach as it does not depend on callers correctly populating `extensionData`.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension in BEFORE_SWAP_ORDER
  pool admin calls swapExtension.setAllowedToSwap(pool, address(router), true)
    (admin allowlists the router to permit standard periphery access)
  pool admin does NOT allowlist attacker EOA

Attack:
  attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)
      msg.sender to pool = address(router)
    → pool calls _beforeSwap(address(router), ...)
    → SwapAllowlistExtension.beforeSwap(address(router), ...) called
      checks allowedSwapper[pool][address(router)] == true  ✓
    → swap executes for attacker despite attacker not being allowlisted

Result:
  attacker swaps on a curated pool that was intended to restrict access,
  bypassing the per-user allowlist entirely.
```
