### Title
`env::block_timestamp_ms()` Used as Real-World Clock in Bitcoin Header Timestamp Validation — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`, `contract/src/zcash.rs`)

---

### Summary

All four chain-specific `check_pow` implementations substitute NEAR's `env::block_timestamp_ms()` for the real-world local clock when enforcing the `MAX_FUTURE_BLOCK_TIME_LOCAL` (2-hour) upper bound on submitted Bitcoin-family block header timestamps. NEAR's block timestamp is set by validators and is not guaranteed to track wall-clock time. When NEAR's block timestamp lags real time, valid Bitcoin headers are incorrectly rejected. When it runs ahead, headers with timestamps beyond the 2-hour window are incorrectly accepted. Either direction breaks the light client's fidelity to the reference Bitcoin validation rule.

---

### Finding Description

Bitcoin Core's `CheckBlockHeader` enforces that a submitted block's `nTime` field must not exceed the node's local system clock by more than 2 hours (`MAX_FUTURE_BLOCK_TIME = 7200s`). This is a real-world-time check, not a chain-time check.

In this contract, all four chain modules implement the same guard using `env::block_timestamp_ms()`:

**`contract/src/bitcoin.rs` lines 35–39:**
```rust
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
    "time-too-new: block timestamp too far in the future"
);
```

The identical pattern appears in `litecoin.rs` (lines 36–40), `dogecoin.rs` (lines 42–46), and `zcash.rs` (lines 48–52).

`env::block_timestamp_ms()` returns the timestamp embedded in the current NEAR block by its block producer. NEAR's protocol allows block timestamps to drift from wall-clock time: they must be monotonically increasing and within protocol-defined bounds relative to the previous block, but they are not required to track real-world time precisely. During validator-set transitions, network congestion, or deliberate manipulation within protocol bounds, NEAR block timestamps can lag or lead real time by a meaningful margin.

`MAX_FUTURE_BLOCK_TIME_LOCAL` is defined as `2 * 60 * 60 = 7200` seconds.

---

### Impact Explanation

**Direction 1 — NEAR timestamp lags real time (stale):**
A relayer submitting a freshly mined Bitcoin block (whose `nTime` is close to real-world now) will have it rejected with `"time-too-new"` because `env::block_timestamp_ms() / 1000` is behind real time. The block is valid by Bitcoin consensus rules but the contract refuses it. The light client's canonical chain stalls; SPV proofs for transactions in that block and all subsequent blocks cannot be produced until NEAR's timestamp catches up or the relayer retries in a later NEAR block.

**Direction 2 — NEAR timestamp leads real time:**
A proof submitter can craft or relay a Bitcoin block whose `nTime` is more than 2 hours ahead of real-world time (but within `env::block_timestamp_ms() / 1000 + 7200`). Such a block would be rejected by every honest Bitcoin node but accepted by this contract. If the crafted block also satisfies PoW (e.g., on testnet where difficulty is low, or via a 51%-attack scenario on a minority chain), it is stored in the contract's header pool and can influence chain-work comparisons and reorg decisions, corrupting the canonical chain mapping.

---

### Likelihood Explanation

NEAR block timestamps are set by the block producer and are validated only to be greater than the previous block's timestamp and within a loose future bound. Empirically, NEAR block timestamps can diverge from wall-clock time by tens of seconds to minutes during normal operation. The 2-hour window means Direction 1 (stale rejection) is a realistic operational hazard for any relayer. Direction 2 (future-block acceptance) requires NEAR's timestamp to be ahead of real time, which is bounded by NEAR's own protocol but is non-zero in adversarial validator scenarios. On testnet deployments (where `pow_allow_min_difficulty_blocks: true` and PoW difficulty is minimal), Direction 2 is directly exploitable by any unprivileged caller of `submit_blocks()`.

---

### Recommendation

Replace `env::block_timestamp_ms()` with a caller-supplied or oracle-supplied timestamp that is validated against the submitted block's own chain context (e.g., median-time-past of the preceding headers), or remove the `MAX_FUTURE_BLOCK_TIME_LOCAL` check entirely and rely solely on the `time-too-old` (MTP) check, which is self-contained within the submitted header chain and does not depend on any external clock. If the local-time check is retained, document explicitly that NEAR's block timestamp is used as a proxy for wall-clock time and that this proxy can diverge.

---

### Proof of Concept

1. Deploy the contract on NEAR testnet (Bitcoin testnet variant, `pow_allow_min_difficulty_blocks: true`).
2. Observe that `env::block_timestamp_ms()` in the executing NEAR block is, say, `T_near`.
3. Construct a Bitcoin testnet block header with `nTime = T_near / 1000 + 7199` (just within the window) and valid PoW at minimum difficulty.
4. Call `submit_blocks([header])`. The contract accepts it.
5. Now construct a second header with `nTime = T_near / 1000 + 7201` (just outside the window by real-world time but within the window if NEAR's timestamp is 2 seconds ahead of real time).
6. During any NEAR block where `env::block_timestamp_ms()` leads real time by ≥2 seconds, this second header also passes the check, demonstrating that the guard is not anchored to real-world time.

Conversely, during any NEAR block where `env::block_timestamp_ms()` lags real time by more than a few seconds, a legitimately mined Bitcoin block with `nTime` equal to real-world now will be rejected, stalling the relayer.
