### Title
Silent `l1_data_gas` Zeroing in Protobuf Block-Sync Deserialization Produces Hash/Field Inconsistency — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` used in the block-sync path silently defaults the `l1_data_gas` field to zero when it is absent from the wire message. Because the transaction hash is taken verbatim from the peer (not recomputed), the node stores a transaction whose hash was computed over non-zero `l1_data_gas` bounds but whose in-memory/DB representation has `l1_data_gas = 0`. Any subsequent RPC call that reads or re-executes that transaction returns authoritative-looking wrong resource bounds, and any component that recomputes the hash from the stored fields will produce a divergent value.

---

### Finding Description

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` (the converter used by the block-sync `TransactionInBlock` path), the `l1_data_gas` field is handled with:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
``` [1](#0-0) 

When `l1_data_gas` is absent, it becomes `ResourceLimits { max_amount: 0, max_price_per_unit: 0 }`. The subsequent variant-selection logic is:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [2](#0-1) 

For a post-0.13.3 V3 transaction where `l2_gas > 0` and `l1_data_gas > 0`, a peer that omits `l1_data_gas` causes the node to reconstruct `AllResources` with `l1_data_gas = 0`.

The block-sync deserializer (`TryFrom<protobuf::TransactionInBlock>`) takes the transaction hash directly from the wire without recomputing it:

```rust
let tx_hash = value
    .transaction_hash
    .clone()
    .ok_or(missing("Transaction::transaction_hash"))?
    .try_into()
    .map(TransactionHash)?;
``` [3](#0-2) 

The stored `(Transaction, TransactionHash)` pair therefore has a hash that was computed over the original non-zero `l1_data_gas`, but the stored `ValidResourceBounds` has `l1_data_gas = 0`.

The canonical hash function `get_tip_resource_bounds_hash` includes `l1_data_gas` in the Poseidon preimage for `AllResources` transactions:

```rust
ValidResourceBounds::AllResources(all_resources) => {
    vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
}
``` [4](#0-3) 

So recomputing the hash from the stored fields yields a different value than the stored hash — a permanent hash/field inconsistency in the node's DB.

By contrast, the **mempool/gateway path** (`TryFrom<protobuf::ResourceBounds> for AllResourceBounds` in `rpc_transaction.rs`) correctly requires `l1_data_gas`:

```rust
l1_data_gas: value
    .l1_data_gas
    .ok_or(missing("ResourceBounds::l1_data_gas"))?
    .try_into()?,
``` [5](#0-4) 

The two converters for the same protobuf type diverge: the block-sync converter silently accepts a missing field while the mempool converter rejects it.

---

### Impact Explanation

**High — RPC returns authoritative-looking wrong values; hash/field binding is broken.**

1. `starknet_getTransactionByHash` returns `l1_data_gas = 0` for a transaction that was actually signed and executed with non-zero `l1_data_gas` bounds.
2. `starknet_simulateTransactions`, `starknet_estimateFee`, and `starknet_traceTransaction` re-execute the stored transaction with wrong resource bounds, producing wrong fee/gas results.
3. The OS hint `assert_transaction_hash` (which recomputes the hash from stored fields) will produce a divergent hash, potentially causing proof generation to fail or to accept a wrong preimage. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The trigger requires a peer to send a `TransactionInBlock` protobuf message for a post-0.13.3 V3 transaction with the `l1_data_gas` field omitted. This is reachable from any connected P2P peer (no privileged access required). The TODO comment explicitly acknowledges the field is intentionally not enforced: `"Assert data gas is not none once we remove support for 0.13.2."` A malicious peer or a peer running a 0.13.2-era implementation can trigger this silently on every synced block containing such transactions.

---

### Recommendation

Replace the silent default with an explicit version-aware check. Once 0.13.2 support is dropped, enforce the field:

```rust
let l1_data_gas = value.l1_data_gas
    .ok_or(missing("ResourceBounds::l1_data_gas"))?;
```

Until then, gate the enforcement on the Starknet version carried in the block header, or at minimum add a hash-recomputation validation step after deserialization (analogous to `validate_transaction_hash`) to detect and reject any transaction whose stored fields do not reproduce the wire hash.

---

### Proof of Concept

1. Attacker controls or emulates a P2P peer.
2. Peer sends a `TransactionsResponse` containing a `TransactionWithReceipt` for a V3 Invoke transaction where `resource_bounds.l1_data_gas` is omitted from the protobuf.
3. The receiving node calls `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` → `l1_data_gas.unwrap_or_default()` → `ResourceBounds { max_amount: 0, max_price_per_unit: 0 }`.
4. Since `l2_gas > 0`, the variant is `AllResources` with `l1_data_gas = 0`.
5. The transaction hash from the wire (computed over the original non-zero `l1_data_gas`) is stored as-is.
6. Call `starknet_getTransactionByHash` → response shows `l1_data_gas: { max_amount: "0x0", max_price_per_unit: "0x0" }` — wrong.
7. Call `starknet_estimateFee` re-executing the stored transaction → fee computed without `l1_data_gas` cost → wrong (lower) fee returned.
8. Any component recomputing `get_tip_resource_bounds_hash` from the stored `ValidResourceBounds` produces a hash that diverges from the stored `TransactionHash`, breaking hash-integrity invariants relied upon by the OS prover.

pub(crate) fn assert_transaction_hash<S: StateReader>(
    hint_processor: &mut SnosHintProcessor<'_, S>,
    ctx: HintContext<'_>,
) -> OsHintResult {
    let stored_transaction_hash = ctx.get_integer(Ids::TransactionHash)?;
    let calculated_tx_hash =
        hint_processor.get_current_execution_helper()?.tx_tracker.get_tx()?.tx_hash().0;

    if calculated_tx_hash == stored_transaction_hash {
        Ok(())
    } else {
        Err(OsHintError::AssertionFailed {
            message: format!(
                "Computed transaction_hash is inconsistent with the hash in the transaction. \
                 Computed hash = {stored_transaction_hash:#x}, Expected hash = \
                 {calculated_tx_hash:#x}."
            ),
        })
    }
}
