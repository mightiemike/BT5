### Title
ERC4626 Vault Inflation Attack Bypasses `wrapVaultAsset` Wrapping Guard, Leaving Raw Underlying Assets in DDA — (`core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.wrapVaultAsset` gates the entire wrapping step on a single `previewDeposit != 0` check. Because both `wrapVaultAsset` and `creditDepositV1` are permissionless (`external`, no `onlyOwner`), an attacker can donate underlying tokens directly to the ERC4626 vault to inflate `totalAssets` without minting shares, forcing `previewDeposit(assetBalance) == 0` via integer truncation, causing the wrapping branch to be silently skipped. A subsequent `creditDepositV1` call then either credits the wrong product or permanently strands the underlying assets in the DDA.

---

### Finding Description

The vulnerable guard is at `ContractOwner.sol` line 525:

```solidity
if (IERC4626Base(tokenAddr).previewDeposit(assetBalance) != 0) {
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(assetTokenAddr));
    ...
    IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
}
``` [1](#0-0) 

For a standard ERC4626 vault, `previewDeposit` computes:

```
shares = assets * totalSupply / totalAssets
```

An attacker transfers underlying tokens directly to the vault contract address (not through `deposit`), inflating `totalAssets` without minting any new shares. Once `totalAssets` is large enough that `assetBalance * totalSupply < totalAssets`, integer division truncates to 0, the `if` branch is never entered, and the underlying assets remain in the DDA untouched.

Neither `wrapVaultAsset` nor `creditDepositV1` carry an `onlyOwner` modifier: [2](#0-1) [3](#0-2) 

So the attacker can call both functions themselves in the same transaction after the donation.

`creditDeposit()` in `DirectDepositV1` iterates over all registered product IDs and deposits any non-zero token balance it finds: [4](#0-3) 

If the underlying asset (e.g., USDC) is itself a registered spot product, `creditDeposit` credits it under the underlying-asset product ID instead of the vault-share product ID — wrong product, zero vault-share balance. If the underlying asset is not a registered product, the balance is silently skipped and the funds are stranded in the DDA with no protocol-level recovery path (the only `withdraw` on the DDA is `onlyOwner` and is only invoked internally by `wrapVaultAsset`). [5](#0-4) 

---

### Impact Explanation

- **Wrong-product crediting**: If the underlying asset is a registered spot product, the user's subaccount is credited with the underlying token product instead of vault shares. The vault-share product balance remains zero, violating the deposit invariant.
- **Permanent fund stranding**: If the underlying asset is not a registered product, the raw underlying tokens remain in the DDA indefinitely. There is no permissionless recovery path; the only `withdraw` call is gated by `onlyOwner` on the DDA and is only reachable through the `wrapVaultAsset` branch that was just bypassed.

---

### Likelihood Explanation

The attack is cheapest against newly initialized vaults (low `totalSupply`). For a vault with `totalSupply = S` shares and `totalAssets = A` tokens, the attacker needs to donate `D` such that `assetBalance * S < A + D`. For small `S` (e.g., first depositor scenario), `D` can be as small as `assetBalance` tokens. The attacker loses the donated tokens to existing shareholders, making this a griefing/sabotage attack rather than a profit-seeking one, but the user's funds are still at risk. Both trigger functions are permissionless, so no privileged access is required.

---

### Recommendation

1. **Remove the `previewDeposit` gate entirely** or replace it with a check on `assetBalance > 0`. The wrapping should always be attempted if there is a non-zero underlying balance; a zero-share result from the vault should revert, not silently skip.
2. **Add `onlyOwner` to `wrapVaultAsset` and `creditDepositV1`** so only the trusted protocol operator can trigger the deposit flow, eliminating the permissionless front-run surface.
3. **Revert instead of silently skipping**: if `previewDeposit(assetBalance) == 0` when `assetBalance > 0`, revert with an explicit error rather than proceeding to `creditDepositV1` with unwrapped assets.

---

### Proof of Concept

```solidity
// 1. User sends 100e6 USDC (underlying) to their DDA.
// 2. Attacker donates D USDC directly to the vault contract:
//    vault_totalAssets becomes 1 + D (vault has 1 share, 1 token initially)
//    previewDeposit(100e6) = 100e6 * 1 / (1 + D)
//    Choose D = 100e6: result = 100e6 * 1 / 100e6+1 = 0 (truncated)
underlyingToken.transfer(address(vault), D);

// 3. Attacker calls wrapVaultAsset — previewDeposit returns 0, branch skipped.
contractOwner.wrapVaultAsset(subaccount, vaultProductId);

// 4. Attacker calls creditDepositV1 — creditDeposit() iterates products.
//    If USDC is a registered product: credits USDC product, not vault shares.
//    If USDC is not registered: 100e6 USDC stranded in DDA forever.
contractOwner.creditDepositV1(subaccount);

// Assert: vault-share product balance for subaccount == 0  ✓ (invariant broken)
```
