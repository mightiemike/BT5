### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling Merkle Internal-Node Forgery to Produce False SPV Proof Results - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` is still a live, unpermissioned public entry point on the contract despite being deprecated. It accepts a raw `tx_id` field without validating that the value is a leaf node in the Merkle tree, allowing any NEAR caller to supply an internal Merkle node hash as `tx_id` with a shortened proof path and receive a `true` return value for a transaction that was never included in the block.

### Finding Description

`verify_transaction_inclusion` carries an explicit code-level warning that it may return `true` when `tx_id` is an internal Merkle tree node rather than a real transaction hash, and defers that validation to "a higher level." [1](#0-0) 

Despite this, the function remains `pub` and is decorated only with `#[pause]` (not paused by default) and `#[deprecated]` (a Rust compiler hint that has no effect on external NEAR RPC callers). [2](#0-1) 

The core verification step is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [3](#0-2) 

`compute_root_from_merkle_proof` applies each proof hash iteratively from the supplied starting value up to the root, with no awareness of tree depth. [4](#0-3) 

The only guard on the proof is `require!(!args.merkle_proof.is_empty(), ...)`. [5](#0-4) 

There is no check that `merkle_proof.len()` equals the full tree depth, and no check that `tx_id` is a leaf. This means: if an attacker supplies an internal node `N` at depth `D` (where `0 < D < tree_depth`) as `tx_id`, and supplies the `D`-length sibling path from `N` to the root as `merkle_proof`, `compute_root_from_merkle_proof` will correctly reconstruct the stored Merkle root and the function returns `true`.

`verify_transaction_inclusion_v2` fixes this by requiring `merkle_proof.len() == coinbase_merkle_proof.len()` and verifying the coinbase at position 0, which anchors the expected tree depth. [6](#0-5) 

But v2's mitigation is entirely bypassed when a caller invokes v1 directly.

The structural parallel to `dangerouslySetInnerHTML` is exact:

### Impact Explanation

Any NEAR account (no role required) can call `verify_transaction_inclusion` and receive `true` for a fabricated `tx_id` that corresponds to an internal Merkle node in a real, mainchain block. Downstream contracts or off-chain systems that call this function to gate cross-chain asset releases, bridge settlements, or other trust-sensitive operations will accept a forged proof as valid. The canonical SPV guarantee of the light client — that `true` means a real transaction was mined in a confirmed block — is broken for all consumers of this entry point.

### Likelihood Explanation

The attack requires no cryptographic break. Internal Merkle node hashes are computable from publicly available Bitcoin block data. The attacker needs only to: (1) pick any confirmed mainchain block stored in the contract, (2) compute any internal node hash and its sibling path to the root, and (3) call `verify_transaction_inclusion` with those values. The function is callable by any NEAR account with no deposit or role requirement beyond the contract not being paused.

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely. The function's own deprecation notice and the existence of `verify_transaction_inclusion_v2` confirm it is no longer needed as a public entry point. If internal reuse is required (as it currently is by v2 via `#[allow(deprecated)] self.verify_transaction_inclusion(args.into())`), refactor the shared logic into a private helper and expose only `verify_transaction_inclusion_v2` externally. This mirrors the recommendation in the reference report: remove the unsafe mechanism altogether rather than leaving it reachable.

### Proof of Concept

1. Identify any block `B` on the contract's mainchain (e.g., via `get_block_hash_by_height`).
2. Obtain `B`'s full transaction list from a Bitcoin node and compute its Merkle tree.
3. Select any internal node `N` at depth `D` (e.g., the left child of the root, `D=1`). Record its position `P` and its sibling `S` (the right child of the root).
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N`
   - `tx_block_blockhash = B`
   - `tx_index = P` (even, so `N` is placed on the left)
   - `merkle_proof = [S]` (length 1, the single sibling needed to reach the root)
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, P, [S])` computes `double_sha256(N || S)` = Merkle root of `B`.
6. The function returns `true`, falsely asserting that `N` (a non-existent transaction) is included in block `B`. [7](#0-6) [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
