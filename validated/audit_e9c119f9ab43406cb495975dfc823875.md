### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Safety Check — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` enforces a coinbase Merkle proof check to prevent the 64-byte transaction Merkle proof forgery attack. The older `verify_transaction_inclusion` function, which lacks this check, is still publicly callable by any unprivileged NEAR caller. An attacker can invoke the deprecated function directly to bypass the coinbase proof validation and obtain a `true` inclusion result for a crafted (non-existent) transaction.

### Finding Description

The contract exposes two public transaction-inclusion verification functions:

**`verify_transaction_inclusion`** (deprecated since v0.5.0): [1](#0-0) 

This function computes the Merkle root from the supplied `tx_id` and `merkle_proof` and compares it to the block's `merkle_root`. It contains an explicit warning that it may return `true` if `tx_id` is an internal Merkle tree node rather than a real transaction hash. It performs **no coinbase proof validation**.

**`verify_transaction_inclusion_v2`** (the safe replacement): [2](#0-1) 

This function adds the coinbase Merkle proof check — it requires the caller to also supply `coinbase_tx_id` and `coinbase_merkle_proof`, verifies the coinbase proof against the block's `merkle_root`, and only then delegates to `verify_transaction_inclusion`. This coinbase check is the mitigation for the [64-byte transaction Merkle proof forgery vulnerability](https://www.bitmex.com/blog/64-Byte-Transactions).

The analog to M-04 is exact:

| M-04 (LatentSwapLEX) | This repository |
|---|---|
| `redeem()` enforces `_checkRedeemCap()` | `verify_transaction_inclusion_v2` enforces coinbase proof check |
| `swap()` performs equivalent operation, skips the check | `verify_transaction_inclusion` performs equivalent operation, skips the check |
| Both are publicly callable | Both are publicly callable (`verify_transaction_inclusion` is `pub`, not `pub(crate)`) |

The deprecation annotation is a Rust compiler hint only — it does not restrict on-chain access. Any NEAR account can call `verify_transaction_inclusion` directly via a cross-contract call or direct invocation, completely bypassing the coinbase proof safety mechanism.

### Impact Explanation

The 64-byte forgery attack works because Bitcoin's Merkle tree does not distinguish between leaf nodes (transactions) and internal nodes. An internal node is 64 bytes, which is also a valid serialized transaction length. An attacker who knows the Merkle tree structure of a real block can craft a `tx_id` equal to an internal node hash and supply a shortened `merkle_proof` that still produces the correct `merkle_root`. Without the coinbase proof check, `verify_transaction_inclusion` returns `true` for this forged proof.

Any downstream NEAR contract that calls `verify_transaction_inclusion` (rather than `verify_transaction_inclusion_v2`) to gate asset releases, bridge withdrawals, or other state transitions can be deceived into accepting a proof for a Bitcoin transaction that never existed.

**Impact: Medium** — Requires knowledge of the target block's Merkle tree structure and ability to craft a valid 64-byte internal-node forgery, but the attack is well-documented and the entry path is fully open.

### Likelihood Explanation

**Likelihood: Medium** — The function is public and callable by any unprivileged NEAR account. The 64-byte Merkle forgery technique is publicly documented. Any integrating contract that calls the deprecated function (which is still the simpler API, requiring fewer arguments) is immediately vulnerable without any privileged access required.

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or gate it with `#[private]` so it can only be called internally (as it is by `verify_transaction_inclusion_v2`). Alternatively, add the coinbase proof check directly inside `verify_transaction_inclusion` so both code paths enforce the same safety invariant, matching the fix pattern recommended in M-04: integrate the missing check into the bypassed execution path.

### Proof of Concept

1. Attacker identifies a real Bitcoin block `B` with known Merkle tree structure.
2. Attacker computes an internal Merkle node hash `N` (64 bytes) and a shortened proof `P` such that `compute_root_from_merkle_proof(N, index, P) == B.merkle_root`.
3. Attacker calls `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B.hash`, `merkle_proof = P`, and sufficient `confirmations`.
4. The function at [3](#0-2)  computes the root, finds it matches, and returns `true` — with no coinbase proof check ever executed.
5. The same call to `verify_transaction_inclusion_v2` would fail at [4](#0-3)  because the attacker cannot supply a valid `coinbase_merkle_proof` for the forged node.

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
