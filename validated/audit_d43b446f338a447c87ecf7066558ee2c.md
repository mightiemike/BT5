### Title
`finalize_epoch()` fallback paths drop computed `shard_layout` and `last_resharding` when validator selection fails under DynamicResharding — (`chain/epoch-manager/src/lib.rs`)

### Summary

`EpochManager::finalize_epoch()` computes a new `ShardLayoutV3` and a new `last_resharding` epoch-height for epoch N+2 whenever a dynamic shard split is scheduled. If `proposals_to_epoch_info()` then fails with `ThresholdError` or `NotEnoughValidators`, the two fallback arms clone `next_epoch_info` (epoch N+1) and only increment `epoch_height`, silently discarding both the new shard layout and the updated `last_resharding`. The stored `EpochInfoV5` for epoch N+2 therefore carries the old layout and the old cooldown timestamp, breaking the two invariants that the dynamic-resharding subsystem depends on.

### Finding Description

In `finalize_epoch()`, the happy path is:

```rust

    .or_else(|| next_epoch_info.last_resharding());

    Ok(next_next_epoch_info) => next_next_epoch_info,

The two fallback arms are:

`EpochInfoV5` carries both `shard_layout` and `last_resharding` as first-class fields:

```rust
pub struct EpochInfoV5 {
    pub shard_layout: ShardLayout,
    pub last_resharding: Option<EpochHeight>,
    ...
}
``` [2](#0-1) 

When the fallback fires, the `EpochInfoV5` written to the DB for epoch N+2 has:
- `shard_layout` = the layout of epoch N+1 (the split is silently dropped)
- `last_resharding` = whatever epoch N+1 recorded (the cooldown timestamp is not advanced)

### Impact Explanation

**Broken shard-layout invariant.** `get_shard_layout()` for epoch N+2 reads `EpochInfo::shard_layout()` for V5 instances. With the wrong layout stored, every consumer of the epoch N+2 layout — chunk producer assignment, state-witness validation, flat-storage resharding, trie-split routing — operates on the pre-split layout even though the last block of epoch N has `shard_split` embedded in its header. This is a permanent, consensus-consistent but semantically incorrect state.

**Broken resharding cooldown.** `can_reshard()` reads `last_resharding` from the epoch info:

```rust
let can_reshard = next_epoch_info.last_resharding().is_none_or(|last_resharding| {
    next_epoch_info.epoch_height() - last_resharding >= min_epochs_between_resharding.get()
});
``` [3](#0-2) 

Because the fallback does not advance `last_resharding`, the cooldown check will return `true` too soon. The documentation explicitly states this is unsafe: *"allowing back-to-back reshardings is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`."* [4](#0-3) 

### Likelihood Explanation

`ThresholdError` fires when `stake_sum < num_seats` (all validators unstake simultaneously). `NotEnoughValidators` fires when the validator count falls below `num_shards`. Both are documented as real production edge cases, not hypothetical ones — the warn-level log messages confirm they are expected to occur. Dynamic resharding is a live protocol feature gated by `ProtocolFeature::DynamicResharding`. Any epoch boundary where a shard split is scheduled and validator churn simultaneously triggers either error will silently corrupt the epoch N+2 `EpochInfoV5`.

### Recommendation

In both fallback arms, after cloning `next_epoch_info`, update the V5-specific fields to match what the successful path would have stored:

```rust
Err(EpochError::ThresholdError { .. }) | Err(EpochError::NotEnoughValidators { .. }) => {
    let mut epoch_info = EpochInfo::clone(&next_epoch_info);
    *epoch_info.epoch_height_mut() += 1;
    // Preserve the computed shard layout and cooldown timestamp
    if let EpochInfo::V5(ref mut v5) = epoch_info {
        v5.shard_layout = next_next_shard_layout.clone();
        v5.last_resharding = last_resharding;
    }
    epoch_info
}
```

This mirrors the pattern used in `proposals_to_epoch_info` itself, which always stores both fields when constructing `EpochInfoV5`. [5](#0-4) 

### Proof of Concept

1. Enable `ProtocolFeature::DynamicResharding`.
2. Configure `min_epochs_between_resharding = 2`.
3. At the last block of epoch N, arrange for a shard split to be selected (`shard_split` embedded in block header).
4. Simultaneously arrange for all validators to unstake, so `proposals_to_epoch_info` returns `ThresholdError`.
5. Observe that `finalize_epoch()` stores epoch N+2's `EpochInfoV5` with the old `shard_layout` (no split) and the old `last_resharding`.
6. At the last block of epoch N+1, call `can_reshard()`: it returns `true` because `last_resharding` was not advanced, bypassing the cooldown.
7. A second split is scheduled back-to-back, triggering `InvalidChunkHeaderShardSplit` on the freshly-created child shard's first chunk, as documented. [6](#0-5) [7](#0-6)

    fn can_reshard(
        &self,
        block_hash: &CryptoHash,
        min_epochs_between_resharding: NonZeroEpochHeight,
    ) -> Result<bool, EpochError> {
        let block_info = self.get_block_info(block_hash)?;
        let next_epoch_id = self.get_next_epoch_id_from_info(&block_info)?;
        let next_epoch_info = self.get_epoch_info(&next_epoch_id)?;

        // last_resharding() returns `None` if no resharding happened since dynamic resharding
        // has been enabled. It is theoretically possible that a static resharding was scheduled

        let can_reshard = next_epoch_info.last_resharding().is_none_or(|last_resharding| {
            next_epoch_info.epoch_height() - last_resharding >= min_epochs_between_resharding.get()
        });
        Ok(can_reshard)
    }
