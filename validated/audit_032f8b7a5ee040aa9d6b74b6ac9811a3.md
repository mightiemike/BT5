### Title
Missing Minimum Confirmation Bound Allows Zero-Confirmation SPV Proof Acceptance — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` enforce only an **upper** bound on the caller-supplied `confirmations` parameter. No lower bound is enforced, so any unprivileged NEAR caller may pass `confirmations = 0`. The confirmation-depth check then reduces to `X >= 0`, which is trivially true for every `u64`, meaning the contract certifies a transaction as "confirmed" with zero reorg protection. This is the direct analog of setting `amount0Min = 0` in AMM interactions: the parameter that is supposed to provide a safety floor is left unconstrained, silently eliminating the protection it is meant to supply.

---

### Finding Description

In `contract/src/lib.rs`, `verify_transaction_inclusion` applies two guards on `confirmations`:

```
// upper-bound only
require!(
    args.confirmations <= self.gc_threshold,   // line 290
    "The required number of confirmations exceeds the number of blocks stored in memory"
);

// depth check
require!(
    (heaviest_block_header.block_height)
        .saturating_sub(target_block_height) + 1
        >= args.confirmations,                 // line 305-307
    "Not enough blocks confirmed"
);
```

When `args.confirmations == 0`:

- The first `require!` passes because `0 <= gc_threshold` is always true.
- The second `require!` evaluates to `(any u64 value) >= 0`, which is always true because the left-hand side is a `u64` and can never be negative.

The function then proceeds to verify the Merkle proof and returns `true` for a transaction that may sit in the very tip block — a block with no finality guarantee whatsoever.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase-proof check, so it inherits the same flaw identically. [1](#0-0) 

---

### Impact Explanation

**Impact: High**

The BTC light client is explicitly designed as a foundational layer for bridges, atomic swaps, and cross-chain lending protocols. Consumer contracts that call `verify_transaction_inclusion` with `confirmations = 0` — whether by mistake, by misunderstanding the parameter, or because they delegate the value to an untrusted caller — receive a `true` result for a transaction that has no confirmation depth. An attacker who controls a Bitcoin miner (or can bribe one) can:

1. Broadcast a transaction T in block B at the chain tip.
2. Have block B submitted to the light client.
3. Trigger the consumer contract's verification path with `confirmations = 0`; the light client returns `true`.
4. The consumer contract releases funds, mints tokens, or extends credit.
5. The attacker mines a competing chain that excludes block B, causing a reorg that invalidates T.

The corrupted invariant is the canonical-chain finality guarantee that the light client is supposed to provide: a transaction certified by `verify_transaction_inclusion` is treated as final by downstream contracts, but with `confirmations = 0` it is not final at all. [2](#0-1) 

---

### Likelihood Explanation

**Likelihood: Medium**

- `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, pausable functions callable by any NEAR account without any role restriction.
- Consumer contracts that integrate the light client must supply `confirmations` themselves. Passing `0` is a natural default or off-by-one mistake, especially for developers unfamiliar with Bitcoin finality semantics.
- The contract provides no documentation warning that `0` is a dangerous value, and no runtime rejection to catch it.
- The deprecated `verify_transaction_inclusion` is still live and is internally called by `verify_transaction_inclusion_v2`, doubling the exposure surface. [3](#0-2) 

---

### Recommendation

Add a lower-bound guard immediately after the existing upper-bound check in `verify_transaction_inclusion`:

```rust
require!(
    args.confirmations >= 1,
    "Confirmations must be at least 1"
);
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

Alternatively, introduce a configurable `min_confirmations` state variable (analogous to the slippage-percentage state variable recommended in the external report) so the contract owner can raise the floor during periods of elevated reorg risk. [4](#0-3) 

---

### Proof of Concept

**Setup**: A bridge contract B calls `verify_transaction_inclusion_v2` and releases wrapped BTC when it returns `true`.

1. Attacker submits Bitcoin transaction T paying themselves on chain C1. T is mined into block `B_tip` at height H.
2. Relayer submits `B_tip` to the NEAR light client via `submit_blocks`; `B_tip` becomes `mainchain_tip_blockhash`.
3. Attacker calls bridge contract B, supplying `confirmations = 0` (or bridge contract B hardcodes 0).
4. Bridge calls `verify_transaction_inclusion_v2` → internally calls `verify_transaction_inclusion`:
   - `0 <= gc_threshold` ✓
   - `mainchain_header_to_height.get(B_tip)` returns `H` ✓
   - `(H - H) + 1 = 1 >= 0` ✓
   - Merkle proof for T is valid ✓
   - Returns `true`.
5. Bridge mints wrapped BTC to attacker.
6. Attacker mines a competing chain C2 that omits T (double-spend). If attacker has sufficient hashrate, C2 overtakes C1 and the light client reorgs; T is no longer on the canonical chain.
7. Attacker has both the wrapped BTC (from step 5) and the original BTC (from the double-spend). [5](#0-4) [6](#0-5)

        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,

    }
