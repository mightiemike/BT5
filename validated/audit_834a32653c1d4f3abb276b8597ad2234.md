### Title
No Staleness Threshold on Chain Tip Allows Stale SPV Proofs to Be Accepted - (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept SPV proofs without any check on how recently the contract's chain tip was updated. If the relayer stops submitting blocks (e.g., due to high NEAR gas costs — the exact same economic pressure described in the reference report), the contract's view of the Bitcoin chain becomes arbitrarily stale. During that window, proofs for transactions in blocks that have since been reorganized out of the real Bitcoin chain will still return `true`.

---

### Finding Description

Both SPV verification entry points fetch the chain tip and compute confirmations relative to it:

```rust
// contract/src/lib.rs:294-308
let heaviest_block_header = self
    .headers_pool
    .get(&self.mainchain_tip_blockhash)
    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

Neither function checks whether `heaviest_block_header.block_header.time` (the Bitcoin timestamp of the stored tip) is within any acceptable window of `env::block_timestamp_ms() / 1000` (the current NEAR wall-clock time). The `time` field is a `u32` Unix timestamp stored in every `ExtendedHeader` and is already used elsewhere in the contract (e.g., `contract/src/bitcoin.rs` lines 35–38, `contract/src/utils.rs` line 20).

The relayer is the only entity that can advance the tip via `submit_blocks`, which is gated by `#[trusted_relayer]`. If the relayer halts — due to high NEAR gas costs, operational failure, or any other reason — the tip freezes. The contract has no mechanism to detect or reject this condition at verification time.

---

### Impact Explanation

A consumer contract (e.g., a Bitcoin bridge) calls `verify_transaction_inclusion_v2` to confirm that a Bitcoin payment was made before releasing funds. If the contract's tip is stale by N blocks:

1. A Bitcoin transaction TX was included in block B, which was on the main chain at the time of the last relayer update.
2. Bitcoin subsequently reorganizes: block B is no longer canonical.
3. The relayer has stopped, so the contract never learns of the reorg.
4. An attacker submits a valid merkle proof for TX against block B.
5. `verify_transaction_inclusion_v2` returns `true` — block B is still in `mainchain_header_to_height`, the confirmation count passes against the frozen tip, and the merkle proof is cryptographically valid.
6. The bridge releases funds for a Bitcoin transaction that no longer exists on the canonical chain.

The corrupted value is the SPV proof result: a `true` return from `verify_transaction_inclusion_v2` for a transaction that is not in the real Bitcoin canonical chain.

---

### Likelihood Explanation

The reference report explicitly identifies high gas costs as the reason oracle updates are skipped. The same pressure applies here: `submit_blocks` requires a NEAR deposit to cover storage costs (lines 173–188 of `lib.rs`). During periods of high NEAR gas prices, the relayer may slow or halt submissions. Bitcoin reorganizations of 1–6 blocks occur regularly; a stale window of even a few hours is sufficient for a shallow reorg to go undetected. Any unprivileged NEAR account can call `verify_transaction_inclusion_v2` at any time the contract is not paused.

---

### Recommendation

Add a configurable staleness threshold (analogous to the fix in the reference report) checked inside both verification functions. Before computing confirmations, verify that the stored tip's Bitcoin timestamp is within an acceptable window of the current NEAR block time:

```rust
let tip_timestamp = heaviest_block_header.block_header.time;
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    current_timestamp <= tip_timestamp + MAX_TIP_STALENESS_SECONDS,
    "Chain tip is stale: relayer has not submitted recent blocks"
);
```

`MAX_TIP_STALENESS_SECONDS` should be a configurable contract parameter (e.g., defaulting to 2 hours, matching `MAX_FUTURE_BLOCK_TIME_LOCAL` defined in `btc-types/src/network.rs` line 17) settable by a privileged role, so it can be tuned per deployment without a contract upgrade.

---

### Proof of Concept

1. Deploy the contract initialized at Bitcoin mainnet height H.
2. Relayer submits blocks up to height H+100; tip is at H+100 with timestamp T.
3. Relayer stops (simulating high NEAR gas costs).
4. On real Bitcoin, a 3-block reorg occurs: blocks H+98, H+99, H+100 are replaced.
5. Transaction TX was in block H+98 (now orphaned). Its merkle proof is still valid against the stored header for H+98.
6. Attacker calls `verify_transaction_inclusion_v2` with TX's proof, `tx_block_blockhash` = hash of H+98, `confirmations = 3`.
7. Contract evaluates: `(H+100) - (H+98) + 1 = 3 >= 3` ✓; merkle proof valid ✓; returns `true`.
8. No staleness check fires. The bridge releases funds.

The root cause is the absence of any freshness guard in `verify_transaction_inclusion` [1](#0-0)  and `verify_transaction_inclusion_v2` [2](#0-1) , neither of which compares `heaviest_block_header.block_header.time` against the current NEAR timestamp. The `time` field is available on every stored header as demonstrated by its use in MTP computation [3](#0-2)  and the future-block check in `check_pow` [4](#0-3) . The `MAX_FUTURE_BLOCK_TIME_LOCAL` constant in `btc-types/src/network.rs` [5](#0-4)  provides a natural reference point for what the codebase already treats as an acceptable time bound.
