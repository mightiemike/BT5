### Title
Missing Health Check After Margin Deduction in `createIsolatedSubaccount()` - (File: `core/contracts/OffchainExchange.sol`)

### Summary

The `createIsolatedSubaccount()` function deducts user-controlled margin from the parent subaccount's quote balance when creating an isolated subaccount, but performs no health check on the parent subaccount afterward. Every other balance-reducing operation in the protocol enforces a post-mutation health check. This omission allows a user to deliberately leave their parent subaccount below initial health, breaking the protocol's core solvency invariant.

### Finding Description

The protocol consistently enforces that after any balance-reducing operation, the affected subaccount must remain at or above initial health. For example:

- `transferQuote()` in `Clearinghouse.sol` transfers quote between subaccounts and then checks `require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH)`. [1](#0-0) 

- `withdrawCollateral()` reduces a spot balance and then checks `require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH)`. [2](#0-1) 

- `mintNlp()` deducts quote and checks `require(getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0, ERR_SUBACCT_HEALTH)`. [3](#0-2) 

However, `createIsolatedSubaccount()` in `OffchainExchange.sol` deducts margin from the parent subaccount's quote balance with **no subsequent health check**:

```solidity
int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
}
// ← no health check here
``` [4](#0-3) 

The margin amount is entirely user-controlled: it is decoded from the `appendix` field of the user-signed order via `_isolatedMargin(txn.order.appendix)`. [5](#0-4) 

The function is gated by `onlyEndpoint`, meaning it is invoked by the Endpoint contract when processing a user-submitted `CreateIsolatedSubaccount` transaction. The user signs the order and controls the `appendix` encoding, including the margin value. There is no cap or validation on the margin amount relative to the parent subaccount's health.

Additionally, the `isHealthy()` stub in `OffchainExchange.sol` unconditionally returns `true`, so the health check inside `matchOrders()` provides no coverage either. [6](#0-5) 

### Impact Explanation

A user can set the margin to an amount that, after deduction, leaves the parent subaccount below initial health (but above maintenance health, avoiding immediate liquidation). This breaks the invariant that all non-liquidatable subaccounts must satisfy initial health. The parent subaccount becomes undercollateralized without being eligible for liquidation, corrupting the protocol's risk accounting. If the parent subaccount subsequently takes on more risk or prices move adversely, the protocol may face bad debt that would otherwise have been prevented by the initial health gate.

### Likelihood Explanation

Any user with an active subaccount can trigger this by submitting a `CreateIsolatedSubaccount` transaction with a margin amount calibrated to bring their parent subaccount just below initial health. No special privileges are required. The user fully controls the margin value via the signed `appendix` field.

### Recommendation

Add a health check on the parent subaccount immediately after the margin deduction in `createIsolatedSubaccount()`:

```solidity
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
    require(
        IClearinghouse(clearinghouse).getHealth(
            txn.order.sender,
            IProductEngine.HealthType.INITIAL
        ) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
```

This mirrors the pattern used in `transferQuote()` and `withdrawCollateral()`. [7](#0-6) 

### Proof of Concept

1. Parent subaccount holds 100 USDC quote balance and has open perp positions requiring 80 USDC initial margin → initial health = +20 USDC.
2. User crafts an isolated order with `_isolatedMargin(appendix)` encoding 25 USDC (i.e., `25e18` in 18-decimal fixed-point).
3. Sequencer calls `createIsolatedSubaccount()` with this signed order.
4. `spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -25e18)` executes — parent quote balance drops to 75 USDC.
5. Parent subaccount initial health = 75 − 80 = **−5 USDC** (below initial health).
6. No health check is performed; the transaction succeeds.
7. Parent subaccount is now below initial health. If maintenance health remains positive, it is not liquidatable, leaving the protocol holding an undercollateralized position.
