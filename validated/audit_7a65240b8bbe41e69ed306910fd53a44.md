### Title
`DepositAllowlistExtension.beforeAddLiquidity` Silently Drops `sender`, Allowing Any Unpermissioned Caller to Bypass the Deposit Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and validates only `owner` (the position holder). Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender ≠ owner`, any unpermissioned address can bypass the deposit allowlist by nominating an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses into the extension hook:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the actual caller (the operator who pays tokens); `owner` is the position holder chosen by that caller. The pool explicitly permits `msg.sender ≠ owner` — there is no `require(msg.sender == owner)` guard in `addLiquidity`.

`DepositAllowlistExtension.beforeAddLiquidity` receives both values but silently discards `sender`:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The first positional parameter — the actual caller — is unnamed and never read. Only `owner` is checked against the allowlist.

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` accepts a caller-supplied `owner` with no restriction beyond `owner != address(0)`:

```solidity
// MetricOmmPoolLiquidityAdder.sol lines 56-68
function addLiquidityExactShares(address pool, address owner, ...) external payable override {
    _validateOwner(owner);   // only checks != address(0)
    ...
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
```

**Attack path:**

1. Pool P has `DepositAllowlistExtension` configured; only address `A` is allowlisted.
2. Unpermissioned address `B` calls `LiquidityAdder.addLiquidityExactShares(pool=P, owner=A, ...)`.
3. The adder calls `P.addLiquidity(owner=A, ...)` with `msg.sender = LiquidityAdder`.
4. The pool calls `extension.beforeAddLiquidity(sender=LiquidityAdder, owner=A, ...)`.
5. The extension checks `allowedDepositor[P][A]` → `true` → passes.
6. `B` (via the adder) successfully deposits tokens into pool `P` — a pool it is not authorized to interact with.

The same path works when `B` calls `pool.addLiquidity(owner=A, ...)` directly.

---

### Impact Explanation

The deposit allowlist is the primary access-control mechanism for restricted pools. Its bypass means:

- Any unpermissioned address can add liquidity to a pool that is supposed to be gated (e.g., a private institutional pool, a KYC-gated pool, or a pool with regulatory restrictions on depositors).
- The unpermissioned caller pays tokens and the allowlisted `owner` receives LP shares they did not request. The allowlisted owner can then call `removeLiquidity` to extract those tokens, effectively letting the unpermissioned party use the allowlisted party as a pass-through to interact with the pool.
- The core invariant — "only approved addresses may deposit" — is broken for the actual payer/operator dimension.

---

### Likelihood Explanation

- The `MetricOmmPoolLiquidityAdder` is the standard periphery entry point for liquidity addition and explicitly exposes an `owner` parameter that any caller can set to any non-zero address.
- No additional privilege or special setup is required beyond knowing one allowlisted address (which may be observable on-chain from prior `AllowedToDepositSet` events).
- The bypass is reachable by any EOA or contract in a single transaction.

---

### Recommendation

Check `sender` (the actual caller) in addition to — or instead of — `owner`, depending on the intended semantics:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    bool poolAllowsAll = allowAllDepositors[msg.sender];
    // Gate the actual operator/payer, not just the position owner
    if (!poolAllowsAll && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate the position holder (owner), the current check on `owner` is correct but the `sender` bypass should be documented explicitly. If the intent is to gate the depositing operator, `sender` must be checked.

// `attacker` is NOT allowlisted.

    deltas,
    maxAmount0,
    maxAmount1,
    ""
