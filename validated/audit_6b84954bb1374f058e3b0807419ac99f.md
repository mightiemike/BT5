### Title
Legacy `verify_transaction_inclusion` Endpoint Remains Callable, Bypassing the Coinbase-Proof Security Upgrade - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction Merkle-proof forgery vulnerability by requiring a coinbase merkle proof. However, the original `verify_transaction_inclusion` function remains a fully public, on-chain callable method. Any unprivileged NEAR caller can invoke it directly, bypassing the coinbase-proof check entirely and obtaining a `true` result for a forged transaction inclusion.

### Finding Description

Version 0.5.0 introduced `verify_transaction_inclusion_v2`, which adds a mandatory coinbase merkle proof validation step before delegating to the original function: [1](#0-0) 

The coinbase proof check is the security-critical gate: [2](#0-1) 

The old function, however, is still exported as a public `#[near]` method with only a Rust `#[deprecated]` attribute: [3](#0-2) 

`#[deprecated]` in Rust is a **compile-time lint only**. It emits a compiler warning to callers in the same codebase but has zero effect on the deployed WASM binary. The NEAR runtime exposes every `pub fn` in a `#[near] impl` block as a callable contract method. There is no `#[private]`, role guard, or `#[pause]` restriction that would prevent an external account from calling `verify_transaction_inclusion` directly.

The function itself performs only a standard Merkle root comparison with no coinbase anchor: [4](#0-3) 

The code's own documentation acknowledges the risk: [5](#0-4) 

This is structurally identical to M-04: a new security configuration was introduced for new callers, but the legacy entry point was left reachable, allowing any caller to permanently opt out of the new protection.

### Impact Explanation

An attacker can craft a 64-byte internal Merkle tree node whose hash collides with a valid transaction hash (the documented BitMEX attack). By calling `verify_transaction_inclusion` directly with this forged `tx_id` and a valid Merkle path to an internal node, the function returns `true`. Any downstream recipient contract that trusts this result—e.g., a bridge, escrow, or settlement contract—will accept a fabricated transaction as confirmed on Bitcoin, enabling theft or unauthorized state transitions.

**Impact: Medium** — requires constructing a valid 64-byte collision preimage, which is computationally feasible for a motivated attacker given the known attack technique.

### Likelihood Explanation

The entry point is unconditionally reachable by any unprivileged NEAR account. No staking, role, or deposit is required. The 64-byte Merkle forgery technique is publicly documented and has known tooling. Any integrator that migrated to `verify_transaction_inclusion_v2` is safe, but any integrator still calling v1—or any new integrator who discovers the v1 method via on-chain introspection—is exposed.

**Likelihood: Medium** — the attack requires knowledge of the forgery technique and a target integrator still using v1, but the method is permanently on-chain and the technique is public.

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI. The simplest safe approach is to add `#[private]` to the function so it is only callable by the contract itself (as it already is via `verify_transaction_inclusion_v2`'s internal `self.verify_transaction_inclusion(args.into())` call). Alternatively, inline its logic directly into `verify_transaction_inclusion_v2` and delete the standalone public method entirely.

### Proof of Concept

1. Identify a Bitcoin block already accepted by the light client (any `tx_block_blockhash` present in `mainchain_header_to_height`).
2. Construct a forged `tx_id` that is the hash of a 64-byte internal Merkle node in that block, along with a valid `merkle_proof` path to the block's `merkle_root` (standard 64-byte attack).
3. Call `verify_transaction_inclusion` directly (not v2) from any NEAR account:
   ```
   near call <contract> verify_transaction_inclusion \
     '{"tx_id": "<forged_node_hash>", "tx_block_blockhash": "<valid_hash>",
       "tx_index": <crafted_index>, "merkle_proof": [...], "confirmations": 1}'
   ```
4. The function returns `true` because it only checks `compute_root_from_merkle_proof(...) == header.block_header.merkle_root` with no coinbase anchor, and the forged internal node satisfies this equality.
5. A recipient contract consuming this `true` result treats the fabricated transaction as confirmed. [6](#0-5)

        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
