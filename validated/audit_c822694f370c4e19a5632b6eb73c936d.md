### Title
Duplicate-Leaf Merkle Padding Allows Phantom Transaction Inclusion Proof — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` returns `true` for a `tx_index` that points to a phantom duplicate leaf position that does not correspond to any real transaction in the block. For any odd-length block, the last transaction hash (`tx[N-1]`) produces an identical Merkle proof for both its real index `N-1` and the phantom index `N`. Because Bitcoin block headers contain no transaction count, the contract cannot bound-check `tx_index`, and the duplicate-leaf padding in `merkle_proof_calculator` makes both positions cryptographically indistinguishable.

---

### Finding Description

`merkle_proof_calculator` pads odd-length transaction lists by duplicating the last element: [1](#0-0) 

This is standard Bitcoin Merkle tree behavior. The problem is that `compute_root_from_merkle_proof` is a pure positional computation with no awareness of the actual transaction count: [2](#0-1) 

And `verify_transaction_inclusion` performs no bounds check on `tx_index` — it only checks that the computed root matches the stored header's `merkle_root`: [3](#0-2) 

`verify_transaction_inclusion_v2` adds a coinbase proof check but does not add any `tx_index` bounds check either: [4](#0-3) 

**Mathematical proof for a 3-tx block `[tx0, tx1, tx2]`:**

After padding: `[tx0, tx1, tx2, tx2_dup]`

Tree:
```
Level 0:  tx0(0)   tx1(1)   tx2(2)   tx2(3)
Level 1:  H01=H(tx0,tx1)    H22=H(tx2,tx2)
Root:     R = H(H01, H22)
```

The Merkle proof for `tx2` at real index `2` is `[tx2, H01]`:
- pos 2 is even → sibling is pos 3 = `tx2`
- pos 1 is odd → sibling is pos 0 = `H01`

The Merkle proof for phantom `tx2` at index `3` is also `[tx2, H01]`:
- pos 3 is odd → sibling is pos 2 = `tx2`
- pos 1 is odd → sibling is pos 0 = `H01`

Both positions produce **identical proofs**. Verifying `compute_root_from_merkle_proof(tx2, 3, [tx2, H01])`:
- Step 1: pos 3 odd → `H(tx2, tx2)` = `H22`, pos → 1
- Step 2: pos 1 odd → `H(H01, H22)` = `R` ✓

The coinbase proof for `tx0` at index `0` is `[tx1, H22]` (length 2, matching the phantom proof length), which also validates correctly against `R`. All guards in `verify_transaction_inclusion_v2` pass.

---

### Impact Explanation

Any downstream system (bridge, payment verifier, cross-chain protocol) that calls `verify_transaction_inclusion_v2` to confirm a Bitcoin transaction was included in a block can be deceived into accepting a phantom transaction that does not exist. The attacker supplies `tx_id = tx[N-1]` (the real last transaction of any odd-length block) at `tx_index = N` (the phantom duplicate position), and the contract returns `true`. This constitutes **false transaction inclusion verification** — the core invariant of an SPV light client. [5](#0-4) 

---

### Likelihood Explanation

- The attack requires only a publicly callable NEAR view/transaction on `verify_transaction_inclusion_v2` — no privileged role, no relayer compromise, no key leak.
- Odd-length transaction blocks are common in Bitcoin (any block with an odd number of transactions).
- The attacker only needs the block hash (already on-chain), the last transaction hash (publicly available from any Bitcoin node), and the ability to compute standard Merkle proofs.
- The proof is fully constructible offline from public data.

---

### Recommendation

Bitcoin block headers do not encode the transaction count, so an on-chain bounds check is not directly possible without additional data. Mitigations include:

1. **Require the caller to supply the transaction count** as part of `ProofArgsV2`, and validate that `tx_index < tx_count` and that `ceil(log2(tx_count))` matches `merkle_proof.len()`. The supplied count must itself be authenticated — e.g., by requiring a proof that the coinbase transaction's `merkle_proof` depth is consistent with the claimed count.
2. **Reject proofs where `tx_index >= 2^(merkle_proof.len() - 1)`** when the last proof element is the same as `tx_id` (a heuristic signal of duplicate-leaf padding), though this is incomplete.
3. The most robust fix is to require callers to provide the raw transaction count and validate that `tx_index < tx_count`, with the count committed to via a separate authenticated channel or by verifying the coinbase's `vin[0].sequence` / coinbase data that encodes block height. [6](#0-5) 

---

### Proof of Concept

For a 3-transaction block with transactions `[tx0, tx1, tx2]` and Merkle root `R = H(H(tx0,tx1), H(tx2,tx2))`:

```
// Attacker constructs:
ProofArgsV2 {
    tx_id:                 tx2,           // real last tx hash
    tx_block_blockhash:    block_hash,    // real on-chain block
    tx_index:              3,             // phantom position (block only has 3 txs, indices 0-2)
    merkle_proof:          [tx2, H(tx0,tx1)],   // valid for phantom index 3
    coinbase_tx_id:        tx0,
    coinbase_merkle_proof: [tx1, H(tx2,tx2)],   // valid coinbase proof, length 2 == merkle_proof.len()
    confirmations:         0,
}
```

Step-by-step through `verify_transaction_inclusion_v2`:
1. `merkle_proof.len() == coinbase_merkle_proof.len()` → `2 == 2` ✓
2. Coinbase check: `compute_root_from_merkle_proof(tx0, 0, [tx1, H22])` = `H(H(tx0,tx1), H(tx2,tx2))` = `R` ✓
3. Delegates to `verify_transaction_inclusion`: `compute_root_from_merkle_proof(tx2, 3, [tx2, H01])` = `H(H01, H22)` = `R` ✓
4. **Returns `true`** — phantom transaction at non-existent index 3 is "verified." [7](#0-6) [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
