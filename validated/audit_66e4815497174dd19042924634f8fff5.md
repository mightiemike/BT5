Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first parameter (`sender`, the actual external caller) and validates only `owner` (the LP position holder) against the allowlist. Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint, any unpermissioned address can bypass the deposit guard by calling `addLiquidity` with `owner` set to any allowlisted address, depositing tokens into a permissioned pool they are not authorized to access.

## Finding Description

`MetricOmmPool.addLiquidity` invokes the extension hook as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The first argument is the real caller (`msg.sender`); the second is the position `owner`. `ExtensionCalling._beforeAddLiquidity` encodes both and forwards them faithfully:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives these in the same order but discards the first (the actual depositor) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [3](#0-2) 

The extension's own NatSpec and admin setter name the gated entity "depositor": [4](#0-3) 

The sibling `SwapAllowlistExtension` correctly reads the first parameter as `sender` and checks it:

```solidity
function beforeSwap(address sender, address, ...)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
``` [5](#0-4) 

`addLiquidity` has no `msg.sender == owner` guard (only `removeLiquidity` enforces that): [6](#0-5) 

**Exploit path:**
1. Pool is configured with `DepositAllowlistExtension`; `allowedDepositor[pool][Bob] = true`; Alice is not allowlisted.
2. Alice calls `pool.addLiquidity(owner=Bob, salt=0, deltas=..., callbackData=..., extensionData="")`.
3. `beforeAddLiquidity(sender=Alice, owner=Bob, ...)` is called; the check evaluates `allowedDepositor[pool][Bob] == true` → no revert.
4. Alice's callback transfers tokens to the pool; Bob's position is credited with shares.
5. Alice has deposited into a permissioned pool she is not authorized to access.

Note: The interface NatSpec at line 147 states "If `DEPOSIT_ALLOWLIST_PROVIDER` is set, `owner` must pass allowlist," which could suggest intentional design. However, the extension's own NatSpec ("Gates `addLiquidity` by depositor address"), the `allowedDepositor` mapping name, the `setAllowedToDeposit(depositor)` setter, and the asymmetry with `SwapAllowlistExtension` all confirm the intent is to gate the actual depositor (`sender`), not the position holder. [7](#0-6) 

## Impact Explanation

**Allowlist bypass / unauthorized pool participation.** A pool admin deploys a permissioned pool (e.g., KYC-only LPs). Any non-allowlisted address can deposit by setting `owner` to any allowlisted address. The attacker provides tokens via the callback; the position is credited to the allowlisted `owner`. The attacker has injected liquidity into a private pool, bypassing the core access-control mechanism.

**Griefing allowlisted LPs.** An attacker can force unwanted liquidity positions onto an allowlisted address. The victim holds the shares but did not initiate the deposit and cannot easily unwind positions they did not choose. Since `removeLiquidity` requires `msg.sender == owner`, only the allowlisted address can remove the forced position.

**Bin-balance manipulation in permissioned pools.** Unauthorized deposits shift `token0BalanceScaled`/`token1BalanceScaled` and `binTotals`, altering the effective price curve and LP share values for all existing LPs.

This constitutes a broken core pool functionality (access control bypass) and admin-boundary break (pool admin's allowlist is circumvented by an unprivileged path), meeting Sherlock contest thresholds.

## Likelihood Explanation

**High.** Preconditions are minimal: the pool must have `DepositAllowlistExtension` configured, and at least one allowlisted address must exist (required for the pool to be usable). The attacker needs only to know any allowlisted address and call `addLiquidity` with `owner` set to it. No privileged access, flash loan, or oracle manipulation is required.

## Recommendation

Rename the first parameter and check `sender` instead of `owner`, matching the intent stated in the NatSpec and the pattern used by `SwapAllowlistExtension`:

```diff
-function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
     external view override returns (bytes4)
 {
-    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
         revert IMetricOmmPoolActions.NotAllowedToDeposit();
     }
     return IMetricOmmExtensions.beforeAddLiquidity.selector;
 }
```

Also update the interface NatSpec in `IMetricOmmPoolActions.sol` line 147 to reflect that `sender` (not `owner`) must pass the allowlist.

## Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][Bob] = true
  Alice is NOT allowlisted

Attack:
  Alice calls pool.addLiquidity(
      owner    = Bob,       // allowlisted → check passes
      salt     = 0,
      deltas   = { binIdxs: [0], shares: [1e18] },
      callbackData = ...,   // Alice transfers tokens in callback
      extensionData = ""
  )

Result:
  beforeAddLiquidity(sender=Alice, owner=Bob) is called
  allowedDepositor[pool][Bob] == true → no revert
  Alice's callback transfers tokens to the pool
  Bob's position is credited with 1e18 shares in bin 0
  Alice has deposited into a pool she is not authorized to access
  Bob holds an unwanted position he cannot easily identify or unwind
```

Foundry test: deploy pool with `DepositAllowlistExtension`, allowlist only Bob, call `addLiquidity` from Alice with `owner=Bob`, assert the call succeeds and Bob's position shares are non-zero.
