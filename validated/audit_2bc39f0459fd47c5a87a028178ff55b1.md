### Title
Unprivileged User Can Bypass `__validate_declare__` Signature Verification by Impersonating the BOOTSTRAP Address During Bootstrap Mode - (File: `crates/starknet_api/src/executable_transaction.rs`, `crates/blockifier/src/transaction/account_transaction.rs`, `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The Sequencer contains a direct analog to the external "skipped signature verification for privileged addresses" bug. The hardcoded `BOOTSTRAP` address (`0x424f4f545354524150`) acts as a privileged identity: any declare transaction whose `sender_address` equals this value, with `nonce = 0` and zero resource bounds, causes the blockifier to skip `__validate_declare__` entirely — no account contract is invoked, no signature is checked, and no nonce is incremented. When the node is in bootstrap mode (`validate_resource_bounds = false`), the gateway's stateless and stateful resource-bound checks are both disabled, so any unprivileged user can craft and submit such a transaction. The BOOTSTRAP address is publicly known and carries no cryptographic protection.

---

### Finding Description

**Root cause — `is_bootstrap_declare` checks only attacker-controlled fields**

`DeclareTransaction::is_bootstrap_declare` in `crates/starknet_api/src/executable_transaction.rs` returns `true` when three conditions hold:

```rust
pub fn is_bootstrap_declare(&self, charge_fee: bool) -> bool {
    if let crate::transaction::DeclareTransaction::V3(tx) = &self.tx {
        return tx.sender_address == Self::bootstrap_address()
            && tx.nonce == Nonce(Felt::ZERO)
            && !charge_fee;
    }
    false
}
```

All three conditions (`sender_address`, `nonce`, `charge_fee`) are fully attacker-controlled fields in the submitted transaction. There is no cryptographic proof that the submitter controls the BOOTSTRAP address.

**Execution bypass in `execute_raw`**

When `is_bootstrap_declare` returns `true`, `execute_raw` in `crates/blockifier/src/transaction/account_transaction.rs` returns immediately after running only the class-storage write, skipping `perform_pre_validation_stage` (nonce check), `run_or_revert` (which calls `__validate_declare__`), and fee charging:

```rust
if let Transaction::Declare(tx) = &self.tx {
    if tx.is_bootstrap_declare(self.execution_flags.charge_fee) {
        // ... run_execute only (class storage write)
        return Ok(TransactionExecutionInfo::default());
    }
}
// Nonce and fee check should be done before running user code.
self.perform_pre_validation_stage(state, &tx_context).map_err(Box::new)?;
// Run validation and execution.
... self.run_or_revert(...) ...
```

The same skip exists in the Cairo OS at `transaction_impls.cairo` line 764:

```cairo

    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        // Declare the class hash and skip the rest of the transaction.
        dict_update{dict_ptr=contract_class_changes}(...);
        %{ SkipTx %}
        return ();
    }
}
```

**Gateway admission path in bootstrap mode**

The `validate_resource_bounds` flag is a shared pointer target that simultaneously disables three independent checks:

1. **Stateless validator** (`crates/apollo_gateway/src/stateless_transaction_validator.rs`):
   ```rust
   fn validate_resource_bounds(&self, tx: &RpcTransaction) -> ... {
       if !self.config.validate_resource_bounds {
           return Ok(());
       }
       // zero-bounds check skipped
   }
   ```

2. **Stateful validator** (`crates/apollo_gateway/src/stateful_transaction_validator.rs`):
   ```rust
   async fn validate_resource_bounds(&self, executable_tx: &ExecutableTransaction) -> ... {
       // Skip this validation during the systems bootstrap phase.
       if self.config.validate_resource_bounds {
           // threshold check skipped
       }
       Ok(())
   }
   ```

3. **Mempool** (`mempool_config.static_config.validate_resource_bounds`) — also disabled.

The node config schema documents this flag explicitly:

> `"validate_resource_bounds"`: "Indicates that validations related to resource bounds are applied. It should be set to false during a system bootstrap."

When `validate_resource_bounds = false`, a transaction with `sender_address = BOOTSTRAP`, `nonce = 0`, and `AllResourceBounds` all zero passes every gateway check. `enforce_fee` returns `false` for zero resource bounds, making `charge_fee = false`, which satisfies the third condition of `is_bootstrap_declare`.

**Nonce check also passes**

The BOOTSTRAP address is never deployed as a contract, so its on-chain nonce is always `Nonce(Felt::ZERO)`. The stateful nonce check for declare transactions requires `incoming_tx_nonce == account_nonce`, which is `0 == 0` — it passes.

**`authorized_declarer_accounts` does not protect BOOTSTRAP**

`check_declare_permissions` checks `is_authorized_declarer`. The production default is `authorized_declarer_accounts.#is_none = true`, meaning no restriction — all sender addresses, including BOOTSTRAP, are authorized.

---

### Impact Explanation

In bootstrap mode, any unprivileged user can submit a `DeclareTransactionV3` with `sender_address = 0x424f4f545354524150` ("BOOTSTRAP"), `nonce = 0`, and zero resource bounds. The blockifier accepts it, skips `__validate_declare__`, and writes the attacker-chosen `(class_hash → compiled_class_hash)` mapping into the class registry. Because the Cairo OS enforces `prev_value = 0` (a class may be declared only once), an attacker who front-runs the operator's bootstrap declares can:

- Permanently poison class hash slots with wrong `compiled_class_hash` values, causing the operator's subsequent legitimate bootstrap declares to fail with a storage collision.
- Declare arbitrary contract classes (including malicious ones) under class hashes the operator intended to use for system contracts.
- Prevent the system from bootstrapping correctly, since the operator cannot overwrite already-declared classes.

This matches **Critical — Invalid or unauthorized Starknet transaction accepted through account validation / signature logic**, because `__validate_declare__` (the account's signature-verification entry point) is bypassed entirely for an attacker-controlled transaction.

---

### Likelihood Explanation

Bootstrap mode is a real, documented operational mode used when initializing a new Starknet system. The `validate_resource_bounds` flag is a single shared pointer target that disables all resource-bound enforcement simultaneously. The BOOTSTRAP address value (`0x424f4f545354524150`) is publicly documented in the codebase and test utilities. An attacker monitoring the mempool or gateway during bootstrap can trivially craft and submit competing bootstrap declares before the operator's transactions are sequenced.

---

### Recommendation

Add an explicit on-chain authorization check for the BOOTSTRAP address. The simplest fix is to require that bootstrap declares carry a valid signature from a known operator public key, or to gate the bootstrap path behind a sequencer-controlled flag that is checked inside `is_bootstrap_declare` rather than relying solely on the external `validate_resource_bounds` config. Alternatively, restrict the BOOTSTRAP address in `check_declare_permissions` to a configurable allowlist of authorized submitters, separate from the general `authorized_declarer_accounts` list, so that even in bootstrap mode only the operator can submit bootstrap declares.

---

### Proof of Concept

**Preconditions:** Node is running in bootstrap mode (`validate_resource_bounds = false`). `authorized_declarer_accounts` is `None` (default).

**Steps:**

1. Choose any Sierra class and compute its `class_hash` and `compiled_class_hash` (V2).

2. Craft an `RpcDeclareTransactionV3`:
   ```
   sender_address  = 0x424f4f545354524150  // "BOOTSTRAP"
   nonce           = 0x0
   resource_bounds = AllResourceBounds { l1_gas: 0, l2_gas: 0, l1_data_gas: 0 }
   signature       = []  // empty — not checked
   class_hash      = <target class hash>
   compiled_class_hash = <attacker-chosen value>
   ```

3. Submit to the gateway's `add_tx` endpoint.

4. **Gateway path:**
   - `check_declare_permissions`: passes (no allowlist restriction).
   - Stateless `validate_resource_bounds`: skipped (`validate_resource_bounds = false`).
   - `convert_rpc_tx_to_internal`: succeeds, computes tx hash.
   - Stateful `validate_resource_bounds`: skipped.
   - `validate_nonce`: `0 == 0`, passes.
   - `run_validate_entry_point`: calls blockifier with `validate = true`.

5. **Blockifier path:**
   - `enforce_fee` returns `false` (zero resource bounds → `charge_fee = false`).
   - `is_bootstrap_declare`: `sender_address == BOOTSTRAP && nonce == 0 && !charge_fee` → `true`.
   - `execute_raw` returns immediately after the class storage write, skipping `__validate_declare__`.

6. **Result:** The class is declared under the attacker-chosen `compiled_class_hash`. The operator's subsequent bootstrap declare for the same `class_hash` fails with a storage collision (`prev_value != 0`). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
