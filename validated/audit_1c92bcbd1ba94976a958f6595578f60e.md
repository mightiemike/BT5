### Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the end user, allowing any user to bypass per-user swap restrictions via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is designed to restrict which addresses may swap on a curated pool. However, the `sender` argument it receives is always `msg.sender` of the `pool.swap()` call — which is the **router contract**, not the end user. If the pool admin allowlists the router (the natural action to enable router-mediated swaps), every user in the system can bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the hook.**

In `MetricOmmPool.swap()`, the `_beforeSwap` dispatcher is called with `msg.sender` as the `sender` argument:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
``` [1](#0-0) 

**Step 2 — Router calls `pool.swap()` directly; `msg.sender` to the pool is the router.**

`MetricOmmSimpleRouter.exactInputSingle` (and all other `exact*` entry points) calls `pool.swap()` without forwarding the original caller's identity:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [2](#0-1) 

The original `msg.sender` (the end user) is stored only in the **callback context** for payment purposes; it is never forwarded to the pool's `swap()` call as the `sender` argument.

**Step 3 — Extension checks the router's address, not the end user's.**

`SwapAllowlistExtension.beforeSwap` receives `sender` (the router) and checks it against the per-pool allowlist:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [3](#0-2) 

Here `msg.sender` is the pool (correct key), and `sender` is the router address. The extension has no visibility into the original end user.

**Step 4 — The bypass.**

A pool admin who wants to allow router-mediated swaps must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, **any** user — including those the admin never intended to allow — can swap on the curated pool simply by calling `router.exactInputSingle(...)`. The per-user allowlist is completely ineffective for the router path.

Conversely, if the admin allowlists individual users but not the router, those users cannot use the standard periphery path at all — their swaps revert because the extension sees `sender = router` (not allowlisted).

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` for compliance, KYC, or counterparty-restriction purposes loses its access control entirely for the router path. Any unprivileged user can trade on the curated pool by routing through `MetricOmmSimpleRouter`, draining LP funds at oracle prices that were only intended for trusted counterparties. This is a direct loss of LP principal and a curation failure.

---

### Likelihood Explanation

The trigger is a standard, publicly accessible periphery call (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`). No special privileges are required. The only precondition is that the pool admin has allowlisted the router — a natural and expected action for any pool that intends to support router-mediated swaps. The bypass is reachable by any user on every such pool.

---

### Recommendation

The `sender` forwarded to the `beforeSwap` hook must represent the **economically relevant actor** — the end user — not the intermediary router. Two viable fixes:

1. **Router forwards original caller in `extensionData`**: The router encodes `msg.sender` into `extensionData` and the extension decodes and verifies it. This requires a coordinated convention between router and extension.

2. **Pool exposes a `swapWithSender` entry point**: Add a variant of `swap` that accepts an explicit `sender` address, callable only by factory-registered routers, so the pool can pass the true originator to hooks.

The `DepositAllowlistExtension` correctly avoids this problem by gating on `owner` (the position owner, explicitly passed by the adder) rather than `sender` (the adder contract address): [4](#0-3) 

The swap allowlist should adopt the same pattern — gate on the identity that the pool's economic accounting attributes the action to.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as the `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps.
3. Pool admin calls `setAllowedToSwap(pool, userA, true)` intending to restrict swaps to `userA` only.
4. `userB` (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. The pool calls `_beforeSwap(msg.sender=router, ...)`.
6. The extension evaluates `allowedSwapper[pool][router] == true` → passes.
7. `userB`'s swap executes successfully, bypassing the per-user allowlist.

The allowlist check that should have blocked `userB` at step 6 instead passes because the router — not `userB` — is the `sender` the extension sees. [5](#0-4) [6](#0-5)
