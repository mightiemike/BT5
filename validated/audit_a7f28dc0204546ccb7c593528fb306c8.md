### Title
Linked Signer Can Self-Replace via `LinkSigner` in Fast Mode, Enabling Subaccount Takeover — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction is validated with `allowLinkedSigner = true`. This permits the **current linked signer** (a session key) to sign and submit a `LinkSigner` transaction that replaces itself with any attacker-controlled address. The subaccount owner's primary key is never required. Once replaced, the attacker's address becomes the authorized linked signer and can sign withdrawals, liquidations, and transfers against the victim's subaccount.

---

### Finding Description

In fast mode, `processTransactionImpl` handles `LinkSigner` as follows:

```solidity
// EndpointTx.sol:576-590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the authorized signer as either the subaccount owner **or** the current linked signer:

```solidity
// EndpointTx.sol:172-184
function validateSignature(
    bytes32 sender,
    bytes32 digest,
    bytes memory signature,
    bool allowLinkedSigner
) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

And in `Verifier.validateSignature`, the check passes if the recovered signer is either the subaccount owner address **or** the `linkedSigner`:

```solidity
// Verifier.sol:297-303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means the current linked signer can produce a valid `LinkSigner` signature that overwrites `linkedSigners[subaccount]` with an attacker-controlled address — without any involvement from the subaccount owner.

**Contrast with slow mode**, where `LinkSigner` correctly enforces that only the subaccount owner (`msg.sender`) can change the linked signer:

```solidity
// EndpointTx.sol:232-239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.LinkSigner)
    );
    validateSender(txn.sender, sender);   // ← requires msg.sender == subaccount owner
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

`validateSender` enforces `address(uint160(bytes20(txSender))) == sender`, binding the change to the actual EOA owner. [5](#0-4) 

The fast-mode path has no equivalent ownership check, creating a direct inconsistency.

---

### Impact Explanation

Once a linked signer self-replaces with an attacker address, the attacker's key becomes the authorized linked signer. All subsequent fast-mode transactions that accept `allowLinkedSigner = true` — `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, `TransferQuote`, `MintNlp`, `BurnNlp` — can be signed by the attacker on behalf of the victim subaccount. [6](#0-5) 

The corrupted state is `linkedSigners[victim_subaccount]` in `EndpointStorage`, which directly controls who can authorize asset-moving transactions. [7](#0-6) 

The result is complete subaccount takeover and full collateral theft.

---

### Likelihood Explanation

Linked signers are session keys — typically hot wallets or browser-extension keys stored in less secure environments than the primary subaccount owner key. They are materially more likely to be compromised. Any party that obtains the linked signer's private key (via phishing, malware, or key leakage) can immediately escalate to full subaccount control without the owner ever being aware, since no on-chain event distinguishes a legitimate `LinkSigner` from a malicious one.

The sequencer is the only gatekeeper for fast-mode transactions, and it has no basis to reject a validly signed `LinkSigner` transaction.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` handler in `processTransactionImpl`. A linked signer is a delegated session key and must not be permitted to modify the authorization state of the subaccount it was granted access to. Only the subaccount owner (primary key) should be able to change the linked signer, consistent with the slow-mode implementation.

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← only the subaccount owner may change the linked signer
);
```

---

### Proof of Concept

1. Alice owns `alice_subaccount` (bytes32 with her EOA address in the first 20 bytes) and has linked `session_key` as her linked signer via a prior `LinkSigner` transaction.
2. Attacker compromises `session_key`.
3. Attacker constructs `LinkSigner { sender: alice_subaccount, signer: attacker_address, nonce: current_nonce }` and signs it with `session_key`.
4. Attacker submits this to the sequencer, which calls `processTransactionImpl`.
5. `validateSignedTx(..., allowLinkedSigner = true)` passes: `getLinkedSigner(alice_subaccount)` returns `session_key`, and the recovered signer matches. [8](#0-7) 
6. `linkedSigners[alice_subaccount]` is overwritten with `attacker_address`. [9](#0-8) 
7. Attacker signs `WithdrawCollateral { sender: alice_subaccount, productId: QUOTE_PRODUCT_ID, amount: full_balance, nonce: next_nonce }` with `attacker_address`.
8. `validateSignedTx(..., allowLinkedSigner = true)` passes: `getLinkedSigner(alice_subaccount)` now returns `attacker_address`. Alice's collateral is drained.
