### Title
SwapAllowlistExtension gates the router address instead of the end user, allowing any caller to bypass the curated-pool swap allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the **direct caller of `pool.swap()`** (the `sender` argument forwarded by the pool), not the originating end user. When `MetricOmmSimpleRouter` is allowlisted as a swapper — the only way to enable router-mediated swaps on an allowlisted pool — any unprivileged user can bypass the curated-pool swap gate by routing through the public router.

---

### Finding Description

**Root cause — wrong actor checked by the allowlist hook**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every configured extension: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every hook: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`: [3](#0-2) 

**The bypass path**

`MetricOmmSimpleRouter.exactInputSingle` is a public, permissionless function — no access control of any kind: [4](#0-3) 

When the router calls `pool.swap(...)`, the pool receives `msg.sender = router`. The extension therefore checks `allowedSwapper[pool][router]`, not the originating EOA.

**The unavoidable trap for pool admins**

If a pool has `SwapAllowlistExtension` configured and the admin wants allowlisted users to be able to swap through the router (the standard periphery path), the admin **must** add the router to the allowlist. But the router is a public contract — once it is allowlisted, every user on the network can call `exactInputSingle` and the extension will pass, because the pool always sees the router as `sender`. There is no way to simultaneously:

1. Allow router-mediated swaps for allowlisted users, and  
2. Block non-allowlisted users from using the same router.

This is the direct analog of the Centrifuge finding: the router is added as a trusted intermediary, but because it is a public contract with no caller validation, any user can exploit that trust to bypass the guard.

---

### Impact Explanation

Any user — regardless of whether they appear in the allowlist — can execute swaps on a curated pool by routing through `MetricOmmSimpleRouter`, provided the pool admin has allowlisted the router (the natural operational step). The swap allowlist, which is the pool's primary curation mechanism, is rendered ineffective. Depending on the pool's purpose (e.g., KYC-gated, institutional-only, or rate-limited liquidity), this constitutes a direct policy bypass with fund-impacting consequences: unauthorized parties can drain arbitrage value, front-run allowlisted LPs, or interact with pools they were explicitly excluded from.

---

### Likelihood Explanation

Medium-to-high. Any pool admin who deploys a `SwapAllowlistExtension`-protected pool and also wants to support the standard router path will allowlist the router. The design gives no indication that doing so opens the gate to all users. The trap is structural, not a configuration error that requires malicious intent.

---

### Recommendation

The allowlist hook must check the **originating user**, not the intermediary. Two viable approaches:

1. **Pass the real caller through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap`; the extension decodes and checks it. This requires the router to be trusted to supply honest data, which is acceptable for a canonical periphery contract.

2. **Check `sender` only when it is not the router**: The extension maintains a registry of trusted forwarders; when `sender` is a forwarder, it reads the real caller from `extensionData`; otherwise it checks `sender` directly.

Additionally, document clearly that allowlisting the router address is equivalent to `allowAllSwappers = true` for any user who can reach the router.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, address(router), true)
    → intent: allow router-mediated swaps for allowlisted users
    → actual effect: any user can now swap

Attack:
  attacker = address not in allowedSwapper[pool]
  attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...) with msg.sender = router
    → pool calls _beforeSwap(router, ...)
    → SwapAllowlistExtension.beforeSwap receives sender = router
    → checks allowedSwapper[pool][router] → true → passes
    → swap executes; attacker receives output tokens
```

The allowlist check never sees the attacker's address. The guard is fully bypassed. [5](#0-4) [4](#0-3) [1](#0-0)
