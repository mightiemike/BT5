### Title
`verify_transaction_inclusion_v2` 64-byte Forgery Mitigation Bypassed via Internal Merkle Node as `coinbase_tx_id` — (`contract/src/lib.rs`, `btc-types/src/contract_args.rs`)

---

### Summary

The v2 coinbase-anchor check in `verify_transaction_inclusion_v2` can be satisfied by supplying an **internal merkle node** as `coinbase_tx_id`. Because `compute_root_from_merkle_proof` is a pure arithmetic function with no leaf-vs-internal-node distinction, an attacker can construct a valid-looking coinbase proof that anchors at a depth-1 (or any intermediate) node rather than at a real leaf transaction. This completely bypasses the 64-byte transaction forgery mitigation the function was designed to provide, allowing `verify_transaction_inclusion_v2` to return `true` for a `tx_id` that is not a real transaction in the block.

---

### Finding Description

`ProofArgsV2` carries a `coinbase_tx_id: H256` field intended to be the hash of the coinbase transaction (always at index 0). [1](#0-0) 

`verify_transaction_inclusion_v2` validates it with a single check:

```
compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof) == merkle_root
``` [2](#0-1) 

`compute_root_from_merkle_proof` is a pure iterative hash-chain function. It accepts any 32-byte value as the starting hash and has no mechanism to distinguish a leaf transaction hash from an internal node hash: [3](#0-2) 

For a 4-transaction block `[tx0, tx1, tx2, tx3]`:

```
node_01 = SHA256d(tx0 || tx1)   // depth-1 internal node
node_23 = SHA256d(tx2 || tx3)   // depth-1 internal node
root    = SHA256d(node_01 || node_23)
```

An attacker supplies:

| Field | Value |
|---|---|
| `coinbase_tx_id` | `node_01` (internal node, NOT a real tx) |
| `coinbase_merkle_proof` | `[node_23]` |
| `tx_id` | `node_23` (hash of a crafted 64-byte fake tx) |
| `tx_index` | `1` |
| `merkle_proof` | `[node_01]` |

**Step-by-step evaluation:**

1. Length check: `1 == 1` ✓
2. Coinbase proof: `compute_root_from_merkle_proof(node_01, 0, [node_23])` → `SHA256d(node_01 || node_23)` = `root` ✓
3. `args.into()` drops coinbase fields, producing `ProofArgs { tx_id: node_23, tx_index: 1, merkle_proof: [node_01], … }` [4](#0-3) 
4. Tx proof: `compute_root_from_merkle_proof(node_23, 1, [node_01])` → `SHA256d(node_01 || node_23)` = `root` ✓ [5](#0-4) 

All checks pass. The function returns `true` for `tx_id = node_23`, which is the hash of a crafted 64-byte fake transaction `T = tx2_id_bytes || tx3_id_bytes` that is **not in the block**.

The function is public (no `#[private]`, no access-control guard beyond `#[pause]`), so any NEAR account can call it. [6](#0-5) 

---

### Impact Explanation

Any consumer of `verify_transaction_inclusion_v2` that relies on its `true` return value to confirm a Bitcoin payment or cross-chain event can be deceived into accepting a fabricated transaction. The entire purpose of the v2 upgrade — preventing the [64-byte transaction forgery](https://www.bitmex.com/blog/64-Byte-Transactions) — is nullified. The attacker needs only a real block already in the contract's `headers_pool`; all required values (`node_01`, `node_23`) are computable from public block data.

---

### Likelihood Explanation

The attack requires no privileged role, no key compromise, and no social engineering. The attacker needs:
1. A block accepted into the contract's `headers_pool` (normal operation).
2. The ability to compute internal merkle nodes from the block's transaction list (public data).
3. A crafted 64-byte transaction whose `txid` equals the chosen internal node (constructible by definition: `T = left_child_bytes || right_child_bytes`).

This is fully constructible offline and submittable by any NEAR account.

---

### Recommendation

The coinbase proof must be anchored to a **leaf-level** position. The fix requires that the depth of `coinbase_merkle_proof` matches the actual tree depth for the block (i.e., `ceil(log2(tx_count))`), which means the contract must know or verify the transaction count. The most robust fix is to require the caller to supply the raw coinbase transaction bytes, hash them on-chain to produce `coinbase_tx_id`, and then verify the proof — making it impossible to substitute an internal node. Alternatively, enforce that `coinbase_merkle_proof.len()` equals the proof depth of the main `merkle_proof` AND that both proofs share the same sibling at every level above the leaf, which structurally prevents a shorter/shallower coinbase proof from passing.

---

### Proof of Concept

```rust
#[test]
fn test_v2_bypass_with_internal_node_as_coinbase() {
    use merkle_tools::{compute_root_from_merkle_proof, H256};
    use btc_types::hash::double_sha256;

    // Simulate 4 transactions: tx0 (coinbase), tx1, tx2, tx3
    let tx0 = H256([0x01u8; 32]);
    let tx1 = H256([0x02u8; 32]);
    let tx2 = H256([0x03u8; 32]);
    let tx3 = H256([0x04u8; 32]);

    // Build internal nodes
    let node_01 = {
        let mut v = Vec::with_capacity(64);
        v.extend(tx0.0); v.extend(tx1.0);
        H256(double_sha256(&v).try_into().unwrap())
    };
    let node_23 = {
        let mut v = Vec::with_capacity(64);
        v.extend(tx2.0); v.extend(tx3.0);
        H256(double_sha256(&v).try_into().unwrap())
    };
    let root = {
        let mut v = Vec::with_capacity(64);
        v.extend(node_01.0); v.extend(node_23.0);
        H256(double_sha256(&v).try_into().unwrap())
    };

    // Attacker uses node_01 as coinbase_tx_id with a 1-element proof
    let coinbase_check = compute_root_from_merkle_proof(
        node_01.clone(), 0, &vec![node_23.clone()]
    );
    assert_eq!(coinbase_check, root, "coinbase check passes with internal node");

    // Attacker proves node_23 (fake tx) is at index 1
    let tx_check = compute_root_from_merkle_proof(
        node_23.clone(), 1, &vec![node_01.clone()]
    );
    assert_eq!(tx_check, root, "tx proof passes for internal node — BYPASS CONFIRMED");

    // node_23 is NOT tx2 or tx3; it is a fabricated txid
    assert_ne!(node_23, tx2);
    assert_ne!(node_23, tx3);
}
```

This test passes on the unmodified codebase, confirming that `verify_transaction_inclusion_v2` returns `true` for a `tx_id` (`node_23`) that is not a real transaction in the block.

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
