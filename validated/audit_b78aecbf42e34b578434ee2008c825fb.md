Audit Report

## Title
`SwapAllowlistExtension` allowlist bypass via router address substitution — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap()` sets to its own `msg.sender`. When users route through `MetricOmmSimpleRouter`, `msg.sender` inside `pool.swap()` is the router address, not the actual user. If the pool admin allowlists the router — a required operational step for any router-mediated swap to function — the allowlist check degenerates to "is the router allowed?", which is always true, granting every user unrestricted access to the curated pool.

## Finding Description

`SwapAllowlistExtension.beforeSwap` performs the access check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension is called by the pool via `_callExtensionsInOrder`) and `sender` is the first argument forwarded from the pool. [1](#0-0) 

`MetricOmmPool.swap()` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards this `sender` value unchanged into the ABI-encoded extension call: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, making the router the `msg.sender` inside the pool: [4](#0-3) 

For multi-hop `exactInput`, intermediate hops explicitly use `address(this)` (the router itself) as the payer context, and the router is again `msg.sender` of each `pool.swap()` call: [5](#0-4) 

The pool admin faces an impossible choice: not allowlisting the router breaks all router-mediated swaps for legitimate users; allowlisting the router makes the check trivially true for every user, since `allowedSwapper[pool][router] == true` for all callers. There is no mechanism in the current code to thread the original EOA caller through to the extension.

## Impact Explanation

The `SwapAllowlistExtension` is the sole on-chain mechanism for curated pools to restrict who may trade. A complete bypass means any unprivileged user can execute swaps on a pool explicitly configured to be permissioned (e.g., KYC-gated, institutional-only, or regulatory-restricted). Every swap executed by a non-allowlisted user represents a direct policy violation: LP providers deposited under the assumption that only vetted counterparties would trade against them. This constitutes broken core pool functionality causing real economic harm to LPs and constitutes an admin-boundary break where an unprivileged path circumvents the pool admin's access control configuration. Severity: **High**. [6](#0-5) 

## Likelihood Explanation

The bypass requires the pool admin to have allowlisted the router address, which is a natural and expected operational step: the router is the protocol's own supported periphery contract, and pool admins who want their allowlisted users to benefit from slippage protection, multi-hop routing, and deadline checks must allowlist it. The attack requires no special privileges, no flash loans, and no oracle manipulation — any address can call `MetricOmmSimpleRouter.exactInputSingle()`. The condition is likely to be met in any production deployment of a curated pool that supports router-based trading. Likelihood: **Medium**. [7](#0-6) 

## Recommendation

The extension must gate on the actual end-user, not the intermediary. The simplest correct fix is for the router to encode the originating user (`msg.sender`) into `extensionData` and for `SwapAllowlistExtension.beforeSwap` to decode and prefer that value over the raw `sender` when present. This requires a trust assumption that the router is honest, which is acceptable since the router is a known, audited protocol contract. Concretely:

1. In `MetricOmmSimpleRouter`, ABI-encode `msg.sender` into `extensionData` before calling `pool.swap()` (or append it to the caller-supplied `extensionData`).
2. In `SwapAllowlistExtension.beforeSwap`, decode the originating user from `extensionData` when present and use that address for the allowlist check instead of `sender`.

Alternatively, the extension can detect router-mediated calls (`sender == KNOWN_ROUTER`) and require a signed or encoded user identity in `extensionData` for those cases. [8](#0-7) 

## Proof of Concept

```
Setup:
  - Pool deployed with SwapAllowlistExtension as beforeSwap hook
  - Pool admin calls setAllowedToSwap(pool, allowedUser, true)
  - Pool admin calls setAllowedToSwap(pool, router, true)  ← required for router UX

Attack:
  1. nonAllowedUser calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)
     → msg.sender inside pool.swap() = router address
  3. Pool calls _beforeSwap(sender=router, ...)
  4. ExtensionCalling dispatches to SwapAllowlistExtension.beforeSwap(sender=router, ...)
  5. Extension evaluates: allowedSwapper[pool][router] == true → passes
  6. nonAllowedUser's swap executes successfully on the curated pool

Result:
  - nonAllowedUser trades on a pool they were never authorized to access
  - The allowlist invariant is broken for every user of the router
  - Reproducible as a Foundry integration test: deploy pool + extension + router,
    configure allowlist with router but not attacker, call exactInputSingle from
    attacker address, assert swap succeeds
``` [9](#0-8) [10](#0-9) [1](#0-0)
