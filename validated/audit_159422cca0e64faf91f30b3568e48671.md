### Title
NLP Mint and Burn Lack Slippage Protection ŌĆö Users Cannot Bound Oracle Price at Execution ŌĆö (`core/contracts/Clearinghouse.sol`, `core/contracts/interfaces/IEndpoint.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` user-signed structs contain no minimum-output parameters. The `oraclePriceX18` that governs the quote-to-NLP and NLP-to-quote conversion is supplied by the sequencer at execution time and is structurally excluded from the user-signed payload. Users therefore have no on-chain mechanism to reject an execution that uses an unfavorable price.

---

### Finding Description

**Struct layout ŌĆö what the user signs vs. what the sequencer appends**

The user signs only the inner `MintNlp` / `BurnNlp` struct:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;   // user specifies quote in
    uint64  nonce;
    // ŌåÉ no minNlpAmount
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;     // user specifies NLP in
    uint64  nonce;
    // ŌåÉ no minQuoteAmount
}
``` [1](#0-0) 

The outer `SignedMintNlp` / `SignedBurnNlp` wrappers carry `oraclePriceX18` and `nlpPoolRebalanceX18`, which are sequencer-provided fields appended after the user's `signature` field:

```solidity
struct SignedMintNlp {
    MintNlp  tx;
    bytes    signature;          // user's sig covers only MintNlp above
    int128   oraclePriceX18;     // sequencer-supplied at execution time
    int128[] nlpPoolRebalanceX18;
}
``` [2](#0-1) 

**Execution path ŌĆö price is applied without a user-specified bound**

In `EndpointTx.processTransactionImpl`, the sequencer decodes `SignedMintNlp`, validates the user's signature over the inner `MintNlp` struct, then immediately forwards the sequencer-chosen `oraclePriceX18` to `Clearinghouse.mintNlp`:

```solidity
priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;
clearinghouse.mintNlp(
    signedTx.tx,
    signedTx.oraclePriceX18,   // sequencer-controlled
    nlpPools,
    signedTx.nlpPoolRebalanceX18
);
``` [3](#0-2) 

Inside `Clearinghouse.mintNlp`, the NLP amount minted is derived purely from that price:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [4](#0-3) 

No floor on `nlpAmount` is checked against any user-specified minimum. The symmetric path for `burnNlp` computes the quote returned as:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [5](#0-4) 

No floor on `quoteAmount` is checked against any user-specified minimum.

---

### Impact Explanation

**Minting:** A user who signs a `MintNlp` for `quoteAmount = 10,000 USDC` expecting the current NLP price of `$1.00` to yield `10,000 NLP` will receive only `5,000 NLP` if the sequencer executes the transaction with `oraclePriceX18 = 2e18`. The user has already committed the quote; there is no on-chain check that rejects the execution.

**Burning:** A user who signs a `BurnNlp` for `nlpAmount = 10,000 NLP` expecting `$10,000` back will receive only `$5,000` (minus fee) if the sequencer executes with a halved oracle price. The 4-day lock period means the user cannot simply cancel and retry; they must accept the execution or wait for the lock to expire on a future deposit. [6](#0-5) 

---

### Likelihood Explanation

The NLP oracle price is updated via sequencer-submitted `UpdatePrice` transactions and can change between the moment a user signs a `MintNlp`/`BurnNlp` and the moment the sequencer includes it in a batch. Even without sequencer misbehavior, normal oracle volatility during high-activity periods creates a window where the realized conversion rate diverges materially from the user's expectation. The 4-day lock amplifies the impact because users cannot immediately reverse an unfavorable mint.

---

### Recommendation

1. Add `minNlpAmount` to `MintNlp` and `minQuoteAmount` to `BurnNlp` as user-signed fields:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint128 minNlpAmount;   // slippage guard
    uint64  nonce;
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint128 minQuoteAmount; // slippage guard
    uint64  nonce;
}
```

2. In `Clearinghouse.mintNlp`, assert `nlpAmount >= txn.minNlpAmount` after computing `nlpAmount`.
3. In `Clearinghouse.burnNlp`, assert `quoteAmount >= txn.minQuoteAmount` after computing `quoteAmount` (post-fee).
4. Optionally add a `deadline` field to both structs so stale signed transactions cannot be executed under significantly different market conditions.

---

### Proof of Concept

1. NLP oracle price is `$1.00`. User signs `MintNlp { quoteAmount: 10_000e18, nonce: N }`, expecting Ōēł10,000 NLP.
2. Before the sequencer includes the transaction, the NLP oracle price moves to `$2.00` (e.g., due to a large profitable trading period for the pool).
3. Sequencer submits `SignedMintNlp` with `oraclePriceX18 = 2e18`.
4. `Clearinghouse.mintNlp` computes `nlpAmount = 10_000e18 / 2e18 = 5_000e18`.
5. User receives 5,000 NLP instead of 10,000 NLP. The 5,000 NLP are immediately locked for 4 days; the user cannot exit.
6. No revert occurs because there is no `minNlpAmount` check anywhere in the execution path. [7](#0-6) [8](#0-7)

```text
    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
```
