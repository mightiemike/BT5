### Title
Unnecessary Non-Empty Merkle Proof Precondition Inherited via Shared Logic Blocks Valid Single-Transaction Block Proofs — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` delegates to the deprecated `verify_transaction_inclusion` (v1) for its final proof check. v1 contains a hard `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard that is unnecessary in the v2 call path. This guard fires and panics for any caller attempting to verify a transaction in a single-transaction Bitcoin block, even though the proof is mathematically valid and the coinbase anchor check in v2 has already confirmed the block structure. The result is that any unprivileged NEAR caller or recipient contract is permanently blocked from obtaining a valid `true` result for a legitimate proof.

### Finding Description

`verify_transaction_inclusion_v2` (lib.rs:347–369) performs two steps before delegating:

1. It enforces `merkle_proof.len() == coinbase_merkle_proof.len()` (line 349).
2. It verifies the coinbase anchor: `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof) == header.block_header.merkle_root` (lines 359–364).

It then calls `self.verify_transaction_inclusion(args.into())` (line 368), which is the v1 path.

Inside v1 (lib.rs:288–323), after re-fetching the same header from storage, the function executes:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

followed by:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

For a single-transaction block, the correct and complete Merkle proof for the coinbase transaction is the empty vector `[]`. `compute_root_from_merkle_proof` with an empty proof simply returns the `transaction_hash` argument unchanged:

```rust
for proof_hash in merkle_proof {   // loop body never executes
    ...
}
current_hash   // == transaction_hash
``` [3](#0-2) 

So `compute_root_from_merkle_proof(tx_id, 0, &[]) == merkle_root` is `true` whenever `tx_id == merkle_root`, which is exactly the invariant for a single-transaction block. The guard at line 315 fires before this comparison is ever reached, causing an unconditional panic.

The shared-logic root cause mirrors the original report exactly: v2 reuses v1's body wholesale via `self.verify_transaction_inclusion(args.into())` rather than extracting only the logic it needs. [4](#0-3) 

### Impact Explanation

Any NEAR contract or user that calls `verify_transaction_inclusion_v2` with a valid empty-proof argument for a single-transaction block will receive a panic instead of `true`. Downstream contracts that gate asset releases, cross-chain settlements, or state transitions on a `true` result from this function are permanently unable to process those Bitcoin transactions. The caller cannot work around this: the length-equality check at line 349 forces `coinbase_merkle_proof` to be empty whenever `merkle_proof` is empty, so there is no way to supply a non-empty `merkle_proof` that is still correct for a single-transaction block. [5](#0-4) 

### Likelihood Explanation

Single-transaction blocks (coinbase-only) occur on mainnet during periods of low mempool activity and are common on testnets and in integration environments. Any relayer that submits such a block header to the contract creates a permanent, reproducible failure for every caller that subsequently tries to prove the coinbase transaction. No privileged access, key compromise, or social engineering is required; the caller only needs to supply a valid `ProofArgsV2` with both proof vectors empty and a `tx_block_blockhash` pointing to a single-transaction block already stored in the contract.

### Recommendation

Remove the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard from `verify_transaction_inclusion` (v1), or — preferably — do not call v1 from v2 at all. Instead, inline only the confirmation-count check and the final `compute_root_from_merkle_proof` comparison directly in `verify_transaction_inclusion_v2`, eliminating the shared-logic coupling that imports the unnecessary precondition.

### Proof of Concept

1. Submit a single-transaction block header to the contract (merkle_root == coinbase_tx_id).
2. Call `verify_transaction_inclusion_v2` with:
   - `tx_id` = coinbase tx hash
   - `tx_block_blockhash` = that block's hash
   - `tx_index` = 0
   - `merkle_proof` = `[]`
   - `coinbase_tx_id` = coinbase tx hash
   - `coinbase_merkle_proof` = `[]`
   - `confirmations` = 1
3. Step 1 of v2 passes (both lengths are 0, equal).
4. Step 2 of v2 passes: `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id` == `merkle_root`. ✓
5. v1 is called. The guard `require!(!args.merkle_proof.is_empty())` fires immediately and the transaction panics with `"Merkle proof is empty"`, despite the proof being mathematically correct.
