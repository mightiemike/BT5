### Title
NLP Price Not Updated Before Liquidation Causes Liquidatee to Lose Accrued Yield — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

The NLP token's on-chain price (`oraclePriceX18`) is only updated when the sequencer processes a `MintNlp` or `BurnNlp` transaction. Between those updates, the NLP pool continues to earn trading fees and PnL, increasing the true NAV per token. When a subaccount holding NLP is liquidated, the liquidation path uses the stale on-chain price without first refreshing it, causing the liquidatee to lose all yield that accrued since the last price update.

---

### Finding Description

In `Clearinghouse.sol`, both `mintNlp` and `burnNlp` call `spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18)` before any balance changes, ensuring the on-chain price reflects the current NAV:

```solidity
// Clearinghouse.sol::mintNlp
spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);   // price refreshed
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
```

```solidity
// Clearinghouse.sol::burnNlp
spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);   // price refreshed
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
```

However, the liquidation path in `ClearinghouseLiq.sol::_handleLiquidationPayment` calls `getLiqPriceX18(txn.productId, txn.amount)` to value the NLP being seized, which reads the last stored on-chain price — it never calls `updatePrice` for `NLP_PRODUCT_ID` first:

```solidity
// ClearinghouseLiq.sol::_handleLiquidationPayment (spot branch)
(v.liquidationPriceX18, v.oraclePriceX18) = getLiqPriceX18(
    txn.productId,   // could be NLP_PRODUCT_ID
    txn.amount
);
// No spotEngine.updatePrice(NLP_PRODUCT_ID, ...) call here
spotEngine.updateBalance(txn.productId, txn.liquidatee, -txn.amount);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.liquidatee, v.liquidationPayment);
```

The NLP price stored on-chain is therefore whatever was last set by a `mintNlp`/`burnNlp` transaction. During any period with no NLP mint/burn activity, the pool continues to earn fees and PnL, but the on-chain price does not advance. A liquidator can call `liquidateSubaccount` at any time; the liquidatee's NLP is valued at the stale price, and the yield accrued since the last price update is permanently lost to the liquidatee.

This is structurally identical to the referenced report: a secondary mechanism (liquidation here; shelter drain there) that moves user funds without first triggering the accounting update (price refresh here; `_checkpoint` there), causing the user to forfeit accrued yield.

---

### Impact Explanation

A liquidatee holding NLP tokens receives quote tokens valued at the last on-chain NLP price rather than the current NAV. The difference — `(currentNAV − staledPrice) × nlpAmount` — is effectively stolen from the liquidatee. Because the NLP pool earns continuous fees from all trading activity, the gap between the true NAV and the stale on-chain price grows monotonically between `mintNlp`/`burnNlp` events. In a high-fee environment or after a prolonged period without NLP activity, this loss can be material.

---

### Likelihood Explanation

Any liquidator (unprivileged caller) can trigger `liquidateSubaccount` whenever a subaccount's maintenance health is negative. The precondition — a stale NLP price — is a normal operating condition: the price is only refreshed when the sequencer processes a `MintNlp` or `BurnNlp` transaction. During any gap between such transactions (which can be hours or days in low-activity periods), the vulnerability is live. A sophisticated liquidator can deliberately time the liquidation to maximise the staleness gap.

---

### Recommendation

Mirror the pattern used in `burnNlp`: before computing the liquidation price for `NLP_PRODUCT_ID`, require the sequencer to supply and commit the current NLP oracle price as part of the `LiquidateSubaccount` transaction payload, and call `spotEngine.updatePrice(NLP_PRODUCT_ID, currentOraclePriceX18)` at the start of `_handleLiquidationPayment` when the product being liquidated is NLP. Alternatively, enforce that the stored NLP price is no older than a configurable staleness threshold before any liquidation of NLP proceeds.

---

### Proof of Concept

1. Alice holds 1 000 NLP tokens. The last `mintNlp`/`burnNlp` set the on-chain NLP price to `1.00 USDC`.
2. Over the next 48 hours the pool earns fees; the true NAV rises to `1.05 USDC`. No `mintNlp`/`burnNlp` is processed, so the on-chain price remains `1.00`.
3. Alice's perp position loses value; her maintenance health turns negative.
4. Bob (liquidator) calls `liquidateSubaccount` targeting Alice's NLP balance.
5. `_handleLiquidationPayment` calls `getLiqPriceX18(NLP_PRODUCT_ID, amount)`, which returns a price derived from the stale `1.00` on-chain price.
6. Alice receives `1 000 × 1.00 = 1 000 USDC` instead of `1 000 × 1.05 = 1 050 USDC`.
7. The 50 USDC of accrued yield is lost to Alice; it remains in the pool, effectively redistributed to remaining NLP holders. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }
