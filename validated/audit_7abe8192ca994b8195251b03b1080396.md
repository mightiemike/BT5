Audit Report

## Title
`SwapAllowlistExtension` checks the router address as `sender` instead of the end-user, enabling full allowlist bypass via `MetricOmmSimpleRouter` â€” (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as the `sender` argument to `_beforeSwap`, which forwards it verbatim to `SwapAllowlistExtension.beforeSwap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. Any pool admin who allowlists the router to enable router-mediated swaps simultaneously opens a bypass for every blocked user, since the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actualUser]`.

## Finding Description
`MetricOmmPool.swap` captures `msg.sender` and passes it as `sender` to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // pool's msg.sender â€” the router when routed
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` encodes that value and dispatches it to each configured extension:

```solidity
// ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, amountSpecified, priceLimitX64,
         packedSlot0Initial, bidPriceX64, askPriceX64, extensionData))
);
```

`SwapAllowlistExtension.beforeSwap` then gates on that `sender` value:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` (and all other entry points) calls `pool.swap(...)` directly, making the router the pool's `msg.sender`:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The router passes no user identity in `extensionData`. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. A pool admin who allowlists the router (the only way to make router-mediated swaps work) simultaneously grants every user â€” including explicitly blocked ones â€” the ability to bypass the gate.

## Impact Explanation
Any user blocked by the swap allowlist can trivially bypass it by calling `MetricOmmSimpleRouter` instead of the pool directly. The allowlist â€” the pool admin's primary mechanism for restricting who may trade â€” is rendered ineffective the moment the router is permitted. Pools deployed for regulatory compliance, KYC gating, or risk-controlled access lose their enforcement guarantee entirely. Blocked users can execute unauthorized swaps against LP inventory at oracle prices, constituting a broken core pool functionality / admin-boundary break with direct fund-impact potential.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical public entry point for swaps. Any pool admin who configures a `SwapAllowlistExtension` and also wants users to be able to use the router will naturally allowlist the router, unknowingly opening the bypass. The trigger requires no privileged action beyond the pool admin's own expected setup steps. Any blocked user can exploit this immediately and repeatedly.

## Recommendation
The pool must propagate the originating user through the hook call chain, not just `msg.sender`. Two viable approaches:

1. **Explicit originator parameter on `swap`**: Add an `originator` parameter to `IMetricOmmPoolActions.swap` so the router can pass `msg.sender` (the actual user) explicitly. The pool validates `msg.sender == originator` for direct calls and trusts the router to supply the real user for routed calls.

2. **Extension-data convention**: The router encodes the real user address in `extensionData`; `SwapAllowlistExtension.beforeSwap` decodes and checks it when present, falling back to `sender` for direct calls. This requires a documented convention between the router and the extension.

Either way, the extension must gate the economically relevant actor (the end user), not the intermediary contract.
