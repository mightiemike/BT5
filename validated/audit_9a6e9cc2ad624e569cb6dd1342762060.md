### Title
Zero-Confirmations Bypass Defeats Finality Guarantee in SPV Proof Verification — (`contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2`, which delegates to it) accepts a caller-supplied `confirmations` value of `0`. When `confirmations = 0`, the confirmation-depth check reduces to a tautology (`u64 >= 0` is always true), so the function returns `true` for any transaction in any main-chain block — including the chain tip — with no finality guarantee. This is the direct analog of the `maxTraversal = 0` case in the external report: a zero-valued caller-controlled parameter causes a security-critical check to be silently skipped.

### Finding Description

`verify_transaction_inclusion` is a public, unpermissioned NEAR method. Its `ProofArgs` struct carries a `confirmations: u64` field that is entirely caller-controlled. [1](#0-0) 

Inside the function, the contract enforces only an **upper** bound on `confirmations`: [2](#0-1) 

It then performs the confirmation-depth check: [3](#0-2) 

When `args.confirmations = 0`, the expression `(heaviest_block_header.block_height).saturating_sub(target_block_height) + 1 >= 0` is a `u64 >= 0` comparison, which is **unconditionally true** in Rust. The check never rejects the call. The function then proceeds to verify the Merkle proof and returns `true` for a transaction that may be in the very tip block — a block with zero depth and no finality.

`verify_transaction_inclusion_v2` is equally affected because it converts its `ProofArgsV2` into `ProofArgs` and delegates directly: [4](#0-3) 

Neither function enforces a minimum `confirmations` value, and neither is access-controlled beyond the `#[pause]` gate.

### Impact Explanation

The confirmation depth is the sole on-chain finality guarantee the light client provides to consuming DApps. A DApp that calls `verify_transaction_inclusion` to gate an asset release (e.g., a bridge unlock) assumes the returned `true` means the transaction is buried under at least `N` blocks and is therefore resistant to reorganization. With `confirmations = 0`, an attacker can:

1. Broadcast a Bitcoin transaction.
2. Wait for it to appear in the next block (no mining required by the attacker).
3. Call `verify_transaction_inclusion` with `confirmations = 0` the moment the relayer submits that block.
4. Receive `true` and trigger the DApp's asset-release logic.
5. Simultaneously attempt a double-spend on Bitcoin; if a reorg succeeds, the Bitcoin transaction is reversed while the NEAR-side asset has already been released.

The corrupted invariant is the canonical confirmation-depth mapping: the contract certifies finality it has not verified.

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` with a crafted `ProofArgs`. The only prerequisite is that the target transaction appears in a main-chain block already submitted by the relayer, which is a normal operational state. The attacker does not need to control the relayer, any keys, or any admin role.

### Recommendation

Enforce a minimum confirmation depth inside `verify_transaction_inclusion`. The simplest fix is to require `args.confirmations >= 1` (or a protocol-defined minimum such as 6 for Bitcoin) before any other check:

```rust
require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    "Confirmations must be at least MIN_CONFIRMATIONS"
);
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

This mirrors the fix recommended in the external report: validate that the caller-supplied parameter satisfies the security invariant before allowing the check to proceed.

### Proof of Concept

1. Deploy the contract with any valid genesis and `gc_threshold = 1000`.
2. Submit one Bitcoin block header via `submit_blocks`.
3. Call `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = hash of the just-submitted block (tip)
   - `tx_id` = any transaction hash present in that block
   - `tx_index` = correct index
   - `merkle_proof` = valid Merkle path for that transaction
   - `confirmations = 0`
4. The upper-bound check passes (`0 <= 1000`). The depth check evaluates `(tip_height - tip_height + 1) >= 0` → `1 >= 0` → `true`. The Merkle proof is verified. The function returns `true` for a transaction with zero confirmations.
5. A consuming DApp that trusts this result would release assets for an unconfirmed, potentially reversible Bitcoin transaction. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
