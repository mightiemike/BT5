Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end-user, allowing any unprivileged swapper to bypass a curated pool's allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is set to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so the extension checks whether the **router** is allowlisted rather than the actual end-user. If the router is allowlisted (the only way to let legitimate users trade through it), every non-allowlisted address can bypass the curated pool's swap gate by calling any router swap function.

## Finding Description

**Root cause — actor mismatch in the extension check:**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    recipient,
    ...
);
``` [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` uses that `sender` argument as the identity to gate:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

Here `msg.sender` is the pool (correct pool key), and `sender` is the direct caller of `pool.swap()` — not the originating user.

**Router path — router becomes `sender`:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
``` [3](#0-2) 

The pool therefore passes `sender = address(router)` to the extension. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

The same wrong-actor binding applies to `exactInput` (multi-hop, line 104), `exactOutputSingle` (line 136), and `exactOutput` recursive callback path (line 220) — all call `pool.swap()` from within the router, making the router the `sender` in every case. [4](#0-3) 

**Why existing guards fail:**

The extension has no mechanism to distinguish the originating user from the direct pool caller. There is no trusted-router registry, no `extensionData` decoding of a delegated identity, and no fallback check on `tx.origin`. The guard is structurally blind to the router indirection.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of addresses (e.g., KYC'd counterparties, whitelisted market makers, or protocol-controlled addresses) loses that protection entirely for any user who routes through the public `MetricOmmSimpleRouter`. The non-allowlisted user receives real token output from the pool; the pool's LP providers are exposed to trades from actors the pool admin explicitly intended to exclude. This constitutes broken core pool functionality for allowlisted pools and a direct loss of the curation guarantee — a High-severity impact under the allowed impact gate (broken core pool functionality causing loss of funds or unusable swap flows; admin-boundary break bypassed by an unprivileged path).

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary supported public swap entrypoint. The bypass requires no privileged access, no special token behavior, and no unusual setup — a standard `exactInputSingle` call from any EOA suffices. Any user who observes that a pool uses `SwapAllowlistExtension` can trivially route through the router. The condition is always reachable once the router is allowlisted (which is required for any legitimate user to use it).

## Recommendation

The extension must recover the true end-user identity rather than trusting the `sender` argument. The recommended fix:

**Pass the originating user through `extensionData`**: require the router to encode `msg.sender` (the end-user) into `extensionData` and have the extension decode and verify it. The extension must also verify that `sender` (the direct pool caller) is a registered trusted periphery contract before accepting the delegated identity. For direct pool calls (no router), `sender` is checked directly as today. This requires no architectural changes to the pool or router beyond encoding and decoding the user address.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is allowed.
3. Admin calls `setAllowedToSwap(pool, address(router), true)` — necessary so Alice can use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient=bob, ...)` — pool's `msg.sender` is the router.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Bob receives token output from the curated pool despite never being allowlisted.

If the admin does **not** allowlist the router (step 3 omitted), Alice's router calls also revert at step 7 (`allowedSwapper[pool][router]` → `false`), breaking the supported swap path for legitimate users — making the dilemma inescapable with the current design.
