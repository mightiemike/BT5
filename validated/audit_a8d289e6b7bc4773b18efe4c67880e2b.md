### Title
Deprecated `verify_transaction_inclusion` Remains an Unrestricted Public Endpoint, Enabling Merkle Proof Forgery via Internal-Node `tx_id` — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR function with no access control. It accepts an arbitrary `tx_id` without filtering out internal Merkle tree node hashes. The contract's own documentation warns this causes the function to return `true` for forged proofs. The analog to the external report is exact: a powerful function that should filter dangerous inputs does not, and the dangerous variant remains reachable even though a safer replacement exists.

---

### Finding Description

`verify_transaction_inclusion` is decorated `#[deprecated]` and `#[pause]` but carries no `#[private]` guard and no role-based access control. Any unprivileged NEAR account can call it directly. [1](#0-0) 

The function's own docstring states the broken invariant explicitly:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [2](#0-1) 

The root cause is in `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs`. It treats `transaction_hash` as an opaque 32-byte value and hashes it upward through the proof path with no check that it is a leaf node rather than an internal node: [3](#0-2) 

The final comparison is simply: [4](#0-3) 

`ProofArgs.tx_id` is a plain `H256` with no type-level or runtime distinction between leaf and internal hashes: [5](#0-4) 

The safer replacement, `verify_transaction_inclusion_v2`, adds a coinbase-proof check but then delegates back to v1 via `args.into()`, meaning v1's unfiltered path is still the final arbiter: [6](#0-5) 

---

### Impact Explanation

Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion` (v1) to gate a privileged action (e.g., releasing bridged funds, minting wrapped tokens, confirming a cross-chain payment) can be deceived into accepting a forged proof. The corrupted value is the boolean proof result returned to the caller — the core security output of the light client. A `true` result for a non-existent transaction breaks the SPV guarantee the system is built on.

---

### Likelihood Explanation

Bitcoin Merkle trees are fully public. An attacker can inspect any confirmed block, identify an internal node hash at a known tree position, construct the corresponding sibling path to the root, and submit it as a valid `ProofArgs`. No privileged access, leaked keys, or social engineering is required. The entry point is a standard NEAR RPC call open to any account.

---

### Recommendation

**Option A (preferred):** Remove `verify_transaction_inclusion` entirely. It has been deprecated since v0.5.0 and `verify_transaction_inclusion_v2` is the intended replacement.

**Option B:** Add a `#[private]` attribute or a role guard so only trusted callers (e.g., `Role::DAO`) can invoke v1, eliminating the unprivileged entry path.

**Option C:** Add an input filter inside `verify_transaction_inclusion` that rejects any `tx_id` that is structurally consistent with an internal Merkle node (e.g., by requiring the caller to also supply a raw 64-byte serialized transaction whose double-SHA256 equals `tx_id`), analogous to how the external report recommended filtering dangerous function selectors in `call()`.

---

### Proof of Concept

1. Select any confirmed Bitcoin block whose Merkle tree has depth ≥ 2.
2. Compute the hash of any internal node `H` at tree depth `d` and position `p` (both publicly derivable from block data).
3. Build the sibling path from `H` up to the Merkle root (length `d`, all values public).
4. Call `verify_transaction_inclusion` on the NEAR contract:
   ```json
   {
     "tx_id": "<H as hex>",
     "tx_block_blockhash": "<block hash>",
     "tx_index": <p>,
     "merkle_proof": ["<sibling_0>", "...", "<sibling_d-1>"],
     "confirmations": 1
   }
   ```
5. The function returns `true` — confirming inclusion of a transaction that does not exist — because `compute_root_from_merkle_proof(H, p, siblings)` equals the block's real Merkle root, and no filtering rejects `H` as an internal node. [7](#0-6) [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
