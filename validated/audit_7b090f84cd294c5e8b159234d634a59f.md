### Title
`SwapAllowlistExtension` gates the router address instead of the actual end-user, making the per-user allowlist bypassable via the router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which is `msg.sender` inside the pool's `swap` function. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. The extension therefore gates the router's address, not the individual user's address, making the allowlist either permanently broken for router-mediated swaps or trivially bypassable.

---

### Finding Description

**Pool `swap` passes its own `msg.sender` as `sender` to the extension:** [1](#0-0) 

```solidity
_beforeSwap(
  msg.sender,   // ← this is the router when called via MetricOmmSimpleRouter
  recipient,
  ...
  extensionData
);
```

**`SwapAllowlistExtension.beforeSwap` checks that `sender` value against the allowlist:** [2](#0-1) 

```solidity
function beforeSwap(address sender, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

Here `msg.sender` is the pool (correct key) and `sender` is the pool's `msg.sender` — the router — not the actual end user.

**The router never forwards the real user's address to the pool:** [3](#0-2) 

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
```

The real user's address (`msg.sender`) is stored only in transient storage for payment purposes; it is never surfaced to the pool or the extension as the swap initiator.

This creates two mutually exclusive failure modes:

| Router allowlist state | Effect |
|---|---|
| Router **not** allowlisted | All router-mediated swaps revert even for allowlisted users — core swap path broken |
| Router **allowlisted** | Any user can bypass the per-user allowlist by routing through the router |

The NatSpec of `SwapAllowlistExtension` explicitly states it "Gates `swap` by swapper address, per pool." [4](#0-3) 
The actual swapper (end user) is never the value checked.

---

### Impact Explanation

**Bypass path (router allowlisted):** A pool admin who allowlists the router to enable router-mediated swaps for their approved users inadvertently opens the pool to every user on the network. Any address can call `exactInputSingle` on the router and the extension will approve the swap because `allowedSwapper[pool][router] == true`. This directly violates the LP's expectation of a restricted counterparty set and can expose LP funds to unauthorized traders.

**DoS path (router not allowlisted):** Allowlisted users who attempt to swap through the router are permanently blocked. The only usable path is a direct pool call, which requires the caller to implement `IMetricOmmSwapCallback` themselves — effectively making the public router unusable for any allowlisted pool.

Both outcomes break a core invariant: the allowlist cannot simultaneously permit the router and restrict individual users.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary public entry point for swaps. [5](#0-4) 
Any pool that deploys `SwapAllowlistExtension` and expects per-user gating will encounter this immediately upon the first router-mediated swap attempt. No special privileges or unusual conditions are required — a standard `exactInputSingle` call is sufficient to trigger either failure mode.

---

### Recommendation

The extension must check the actual end-user's address, not the immediate pool caller. Two viable approaches:

1. **Router forwards the real user via `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires a trusted encoding convention between the router and the extension.

2. **Pool exposes the original initiator:** Add a separate `initiator` field to the `beforeSwap` hook signature that the pool populates from a transient-storage context set at the top of `swap`, analogous to how the router already stores the payer in transient storage.

The `DepositAllowlistExtension` avoids this problem by checking `owner` (the LP position owner, explicitly passed as a separate parameter) rather than the caller: [6](#0-5) 
A similar design — separating the "economically relevant actor" from the "immediate caller" — is needed for the swap allowlist.

---

### Proof of Concept

```
Setup:
  pool P with SwapAllowlistExtension E
  allowedSwapper[P][userA] = true   (intended: only userA may swap)
  allowedSwapper[P][router] = true  (needed so userA can use the router)

Attack:
  userB (not allowlisted) calls:
    router.exactInputSingle({pool: P, ...})

  Execution trace:
    router.exactInputSingle()
      → pool.swap(msg.sender = router, ...)
        → extension.beforeSwap(sender = router, ...)
          → allowedSwapper[P][router] == true  ✓ (no revert)
        → swap executes for userB

Result:
  userB successfully swaps in a pool they were never authorized to access.
  The per-user allowlist is completely bypassed.
```

The root cause is structurally identical to the seeded bug: a guard reads the wrong value from its call context — `block.number` instead of `block.number - 1` in the original; the router's address instead of the end user's address here — causing the guard to either always fail or always pass for the wrong actor.
