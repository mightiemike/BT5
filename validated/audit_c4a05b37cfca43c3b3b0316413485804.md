### Title
IBC NFT Packet Receiver Bypasses `ValidateTokenID` Format Guard, Minting Permanently Unmanageable NFTs — (File: `x/nft-transfer/keeper/packet.go`)

### Summary

The `processReceivedPacket` function in the IBC NFT-transfer keeper mints NFTs using token IDs taken directly from the incoming IBC packet without calling `ValidateTokenID`. Every direct on-chain path (`MsgMintNFT`, `MsgTransferNFT`, `MsgBurnNFT`) enforces the format guard at `ValidateBasic()`, but the IBC receive path skips it entirely. This is the exact analog of the external report's bug class: a guard that is supposed to restrict inputs is present and correct for one entry path but is silently absent for another, allowing out-of-spec values to enter the state machine.

---

### Finding Description

`x/nft/types/validation.go` defines `ValidateTokenID`, which enforces that every NFT token ID must be 3–64 characters, start with a lowercase ASCII letter, and contain only `[a-z0-9]`: [1](#0-0) 

All three on-chain message types call this guard inside `ValidateBasic()`: [2](#0-1) [3](#0-2) 

The IBC receive path in `processReceivedPacket` calls `k.nftKeeper.MintNFT` with the raw `tokenID` string from the packet without any format check: [4](#0-3) 

`OnRecvPacket` only calls `data.ValidateBasic()` before delegating to `processReceivedPacket`: [5](#0-4) 

`NonFungibleTokenPacketData.ValidateBasic()` checks only that `TokenIds` is non-empty and that its length matches `TokenUris`; it performs no per-element format validation: [6](#0-5) 

The keeper's `MintNFT` and `MintNFTUnverified` functions contain no tokenID format check either: [7](#0-6) 

---

### Impact Explanation

An IBC packet carrying a token ID that violates `ValidateTokenID` (e.g., starts with a digit, contains uppercase letters, contains `/`, or exceeds 64 bytes) is accepted and stored. Once stored, the NFT is permanently unmanageable on chain-main:

- `MsgTransferNFT.ValidateBasic()` calls `ValidateTokenID` and rejects the ID → owner cannot transfer on-chain.
- `MsgBurnNFT.ValidateBasic()` calls `ValidateTokenID` and rejects the ID → owner cannot burn on-chain.
- `MsgEditNFT.ValidateBasic()` calls `ValidateTokenID` and rejects the ID → owner cannot edit on-chain.

The NFT supply counter is incremented for an NFT that can never be decremented through normal on-chain operations, corrupting the collection's supply invariant. The receiver's ownership rights are hollow: they hold title to an asset they cannot exercise.

If the token ID contains the KV-store key separator (likely `/`), the stored key `KeyNFT(denomID, tokenID)` becomes ambiguous and could collide with a legitimately-issued NFT in a different denom, potentially corrupting that NFT's state. [8](#0-7) 

---

### Likelihood Explanation

ICS-721 imposes no character-set restriction on token IDs; counterparty chains routinely use IDs that are valid on their own chain but violate chain-main's stricter `[a-z0-9]` rule (e.g., uppercase hex IDs, numeric-prefixed IDs, UUIDs). Any relayer can relay a packet from such a chain. No privileged key or social engineering is required — a normal IBC channel open and a single `RecvPacket` relay suffice.

---

### Recommendation

Call `ValidateTokenID` (or an IBC-aware equivalent that also permits the `ibc/HASH` format) on each element of `data.TokenIds` inside `OnRecvPacket` / `processReceivedPacket` before minting. Return an error acknowledgement for packets that fail validation, consistent with how `data.ValidateBasic()` failures are already handled:

```go
for _, tokenID := range data.TokenIds {
    if err := nfttypes.ValidateTokenID(tokenID); err != nil {
        return err
    }
}
```

---

### Proof of Concept

1. On a counterparty chain, issue an NFT with token ID `"1BadToken"` (starts with digit, contains uppercase — valid on that chain, invalid on chain-main).
2. Open an IBC channel between the counterparty and chain-main.
3. Send the NFT to a chain-main address via `MsgTransfer`.
4. The relayer submits `RecvPacket`; `OnRecvPacket` calls `data.ValidateBasic()` (passes — non-empty token ID), then `processReceivedPacket` calls `k.nftKeeper.MintNFT(ctx, voucherClassID, "1BadToken", ...)`.
5. `MintNFT` → `MintNFTUnverified` stores the NFT without format validation. Supply is incremented.
6. The receiver now holds the NFT. Attempting `MsgTransferNFT{Id: "1BadToken", ...}` returns `ErrInvalidTokenID` from `ValidateBasic()`. Attempting `MsgBurnNFT{Id: "1BadToken", ...}` likewise fails. The NFT is permanently unmanageable on-chain. [9](#0-8) [10](#0-9)

func ValidateTokenID(tokenID string) error {
	if len(tokenID) < MinDenomLen || len(tokenID) > MaxDenomLen {
		return sdkerrors.Wrapf(ErrInvalidTokenID, "the length of nft id(%s) only accepts value [%d, %d]", tokenID, MinDenomLen, MaxDenomLen)
	}
	if !IsBeginWithAlpha(tokenID) || !IsAlphaNumeric(tokenID) {
		return sdkerrors.Wrapf(ErrInvalidTokenID, "nft id(%s) only accepts lowercase alphanumeric characters, and begin with an english letter", tokenID)
	}
	return nil
}

func (nftpd NonFungibleTokenPacketData) ValidateBasic() error {
	if strings.TrimSpace(nftpd.ClassId) == "" {
		return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
	}

	if len(nftpd.TokenIds) == 0 {
		return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
	}

	if len(nftpd.TokenIds) != len(nftpd.TokenUris) {
		return newsdkerrors.Wrap(ErrInvalidPacket, "tokenIds and tokenUris lengths do not match")
	}

	if strings.TrimSpace(nftpd.Sender) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "sender address cannot be blank")
	}

	// decode the sender address
	if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
	}

	if strings.TrimSpace(nftpd.Receiver) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "receiver address cannot be blank")
	}

	// decode the receiver address
	if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
	}
	return nil
}

// MintNFTUnverified mints an NFT without verifying if the owner is the creator of denom
// Needed during genesis initialization
func (k Keeper) MintNFTUnverified(ctx sdk.Context, denomID, tokenID, tokenNm, tokenURI, tokenData string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	if k.HasNFT(ctx, denomID, tokenID) {
		return sdkerrors.Wrapf(types.ErrNFTAlreadyExists, "NFT %s already exists in collection %s", tokenID, denomID)
	}

	k.setNFT(
		ctx, denomID,
		types.NewBaseNFT(
			tokenID,
			tokenNm,
			owner,
			tokenURI,
			tokenData,
		),
	)
	k.setOwner(ctx, denomID, tokenID, owner)
	k.increaseSupply(ctx, denomID)

	return nil
}
