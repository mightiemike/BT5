### Title
SwapAllowlistExtension checks the router address instead of the end user, allowing any caller to bypass per-pool swap access control via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps on `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap()` call. When a swap is routed through `MetricOmmSimpleRouter`, that `sender` is the **router address**, not the end user. If the router is allowlisted, every non-allowlisted EOA can execute swaps on the pool by calling the router.

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

**Step 2 — Extension receives the router address as `sender`:**

When `MetricOmmSimpleRouter.exactInput()` (or `exactInputSingle`) calls `pool.swap(...)`, the pool's `msg.sender` is the router: [2](#0-1) 

**Step 3 — Allowlist check is against the router address, not the end user:**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` = pool and `sender` = router: [3](#0-2) 

If `allowedSwapper[pool][router] = true`, the check passes for **any** end user who calls the router, regardless of whether that end user is individually allowlisted.

### Impact Explanation

The `SwapAllowlistExtension` is the sole mechanism for restricting who may swap on a pool. A pool admin who allowlists only the router (a common operational pattern — allowlist the router, not every individual user) inadvertently opens the pool to all users. Any non-allowlisted address can call `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` and execute swaps that the pool admin intended to block. This breaks the core access-control invariant of the extension.

**Impact gate match:** "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" and "Broken core pool functionality."

### Likelihood Explanation

- The router is a first-party periphery contract; pool admins are expected to allowlist it.
- No special setup is required by the attacker — any EOA can call the public router functions.
- The bypass is unconditional once the router is allowlisted.

### Recommendation

`SwapAllowlistExtension.beforeSwap` should check the **end user** rather than (or in addition to) the immediate pool caller. One approach: add an `originSender` field to `extensionData` that the router populates with `msg.sender` before calling the pool, and have the extension verify that field. Alternatively, the extension can check `tx.origin` as a secondary gate (with the usual caveats), or the router can be modified to pass the original caller through `extensionData` in a standardized way.

            amountOutMinimum: 0,
            recipient: attacker,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""

    );
