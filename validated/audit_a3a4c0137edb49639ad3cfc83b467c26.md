Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass the Swap Allowlist via the Router - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` as the pool's direct `msg.sender`, which is the `MetricOmmSimpleRouter` contract when a user routes through it. If the pool admin allowlists the router address to permit router-mediated swaps for legitimate users, the check `allowedSwapper[pool][router]` passes for every caller of the router, completely neutralizing the allowlist for any user who routes through the public, permissionless router contract.

## Finding Description
**Root cause — `sender` binding in the swap call chain:**

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension: [2](#0-1) 

**The allowlist check:**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (used as the pool key) and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**The router as the direct pool caller:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` to the pool: [4](#0-3) 

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict trading.
2. Admin calls `setAllowedToSwap(pool, router, true)` so allowlisted users can use the router.
3. Any unprivileged EOA calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the restricted pool.
4. The router calls `pool.swap(...)` — `msg.sender` to the pool is the router.
5. `_beforeSwap(sender=router, ...)` → extension checks `allowedSwapper[pool][router] == true` → passes.
6. The unauthorized user's swap executes in full; the actual EOA is never checked.

The wrong value is the `sender` argument passed to `beforeSwap`: it is the router address rather than the originating EOA, causing `allowedSwapper[pool][router]` to be evaluated instead of `allowedSwapper[pool][actual_EOA]`.

## Impact Explanation
The `SwapAllowlistExtension` is the sole on-chain mechanism for restricting who may trade in a pool. Bypassing it allows any unprivileged address to execute swaps in pools the admin explicitly intended to gate. This constitutes an admin-boundary break: factory/oracle role checks bypassed by an unprivileged path (the public router). Depending on the pool's purpose (institutional-only liquidity, regulatory compliance, risk-controlled market making), unauthorized actors can drain LP liquidity at oracle-anchored prices, causing direct loss of LP principal.

## Likelihood Explanation
The bypass requires no special privileges, no malicious setup, and no non-standard tokens. Any pool that (a) uses `SwapAllowlistExtension` and (b) allowlists the router — a natural and expected configuration for any pool that wants to support router-mediated swaps — is fully exposed. The attacker needs only to call `MetricOmmSimpleRouter.exactInputSingle` with the target pool address.

## Recommendation
The extension must gate the **originating user**, not the immediate caller of `pool.swap()`. Two viable approaches:

1. **Pass the true initiator through `extensionData`**: The router encodes the original `msg.sender` into `extensionData`; the extension decodes and checks it. This requires the extension to trust only known routers as the `sender`.
2. **Check both `sender` and a decoded initiator**: Require that `sender` is either an allowlisted EOA **or** a trusted router, and if it is a trusted router, additionally verify the end-user identity decoded from `extensionData`.

The `DepositAllowlistExtension` should also be audited: it checks `owner` (not `sender`) in `beforeAddLiquidity`, which is the position owner passed by the caller — a `MetricOmmPoolLiquidityAdder` intermediary could introduce an analogous bypass on the liquidity path. [5](#0-4) 

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Pool admin calls setAllowedToSwap(pool, router, true).
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true).
  - Pool admin does NOT call setAllowAllSwappers(pool, true).

Attack:
  attacker (unlisted EOA) calls:
    MetricOmmSimpleRouter.exactInputSingle(ExactInputSingleParams{
        pool:    <restricted pool>,
        tokenIn: token0,
        zeroForOne: true,
        ...
    });

  Router calls pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData).
  pool.swap() → msg.sender = router → _beforeSwap(sender=router, ...).
  Extension checks: allowedSwapper[pool][router] == true → passes.
  Swap executes for the attacker.

Expected: revert NotAllowedToSwap().
Actual:   swap succeeds; attacker receives tokens from a pool they are not authorized to trade in.
```
