### Title
Trivial Signature Bypass in `submitFastWithdrawal` When `nSigner == 0` Allows Unauthorized Draining of WithdrawPool — (`File: core/contracts/Verifier.sol`, `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`Verifier.requireValidTxSignatures` enforces `require(nSignatures == nSigner)`. When no pubkeys are registered (`nSigner == 0`), this check passes trivially with an empty signatures array. Any unprivileged caller can invoke `BaseWithdrawPool.submitFastWithdrawal` with a crafted withdrawal transaction and zero signatures, bypassing all cryptographic verification and draining the WithdrawPool.

---

### Finding Description

`Verifier.initialize` accepts an `initialSet` of eight `Point` values and only registers a pubkey if the point is non-zero:

```solidity
for (uint256 i = 0; i < 8; ++i) {
    if (!isPointNone(initialSet[i])) {
        _assignPubkey(i, initialSet[i].x, initialSet[i].y);
    }
}
``` [1](#0-0) 

There is no `require` enforcing that at least one pubkey is assigned. If the contract is deployed with all-zero points, `nSigner` remains `0`.

`requireValidTxSignatures` counts non-empty signatures and then asserts:

```solidity
require(nSignatures == nSigner, "not enough signatures");
``` [2](#0-1) 

When `nSigner == 0`, passing an empty `signatures` array yields `nSignatures = 0`, satisfying `0 == 0`. The loop body never executes, so no individual signature is verified.

`BaseWithdrawPool.submitFastWithdrawal` calls this function as its sole authentication gate before transferring tokens:

```solidity
Verifier v = Verifier(verifier);
v.requireValidTxSignatures(transaction, idx, signatures);
``` [3](#0-2) 

After the check passes, `resolveFastWithdrawal` decodes the caller-supplied `transaction` bytes to extract `productId`, `sendTo`, and `amount` — none of which are independently validated against on-chain state:

```solidity
return (
    signedTx.tx.productId,
    address(uint160(bytes20(signedTx.tx.sender))),
    signedTx.tx.amount
);
``` [4](#0-3) 

The attacker controls all three fields. The pool then transfers `transferAmount - fee` to `sendTo`. [5](#0-4) 

---

### Impact Explanation

An attacker can craft a `WithdrawCollateral` transaction naming themselves as `sender` (and therefore `sendTo`) with `amount` equal to the full pool balance of any supported token. With `nSigner == 0`, calling `submitFastWithdrawal(idx, crafted_tx, [])` passes all checks and transfers the tokens to the attacker. Every token held in the `WithdrawPool` — representing pending user withdrawals — can be stolen in a single transaction per product. The corrupted state is the `WithdrawPool` token balance and the `markedIdxs` mapping, which permanently blocks legitimate users from receiving their withdrawals for the consumed `idx` values.

---

### Likelihood Explanation

The `Verifier` contract has no deployment-time enforcement requiring at least one pubkey. The `initialize` function silently accepts an all-zero `initialSet`, leaving `nSigner = 0`. This mirrors the external report's pattern exactly: the protocol permits deployment without the privileged configuration, and the absence of that configuration silently disables the security check. Additionally, `deletePubkey` (callable by the owner) can reduce `nSigner` to zero post-deployment with no floor check, creating a second path to this state. [6](#0-5) 

---

### Recommendation

Add a minimum-signer invariant in `requireValidTxSignatures`:

```solidity
require(nSigner > 0, "no signers registered");
require(nSignatures == nSigner, "not enough signatures");
```

Additionally, enforce in `initialize` that at least one non-zero pubkey is provided, and add a floor check in `deletePubkey` preventing `nSigner` from reaching zero.

---

### Proof of Concept

1. Deploy `Verifier` with `initialize([Point(0,0), Point(0,0), ..., Point(0,0)])`. `nSigner == 0`.
2. Attacker encodes a `WithdrawCollateral` transaction:
   - `sender = bytes32(uint256(uint160(attacker)))`
   - `productId = USDC_PRODUCT_ID`
   - `amount = pool.balanceOf(USDC)`
   - `nonce = 0` (arbitrary, not checked here)
   - `signature = ""` (arbitrary, not checked here)
3. Attacker calls `WithdrawPool.submitFastWithdrawal(1, crafted_tx, [])`.
4. `requireValidTxSignatures` executes: loop body skipped (empty array), `require(0 == 0)` passes.
5. `resolveFastWithdrawal` returns `(USDC_PRODUCT_ID, attacker, pool_balance)`.
6. Since `sendTo == msg.sender`, fee is deducted from `transferAmount`.
7. `handleWithdrawTransfer` sends `pool_balance - fee` USDC to the attacker.
8. All legitimate pending withdrawals backed by that USDC are now unserviceable. [7](#0-6) [8](#0-7)

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }
```

        require(idx > minIdx, "idx too small");

        (

        IERC20Base token = getToken(productId);
