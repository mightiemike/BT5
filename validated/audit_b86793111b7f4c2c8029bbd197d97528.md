### Title
Unprivileged Attacker Can Invalidate Any Pending Meta-Transaction by Front-Running the Relayer's `SignedDelegateAction` — (`File: runtime/runtime/src/actions.rs`)

### Summary

`DelegateAction` (NEP-366 meta-transactions) lets a user sign an action off-chain and hand it to a relayer, which wraps it in an outer transaction and pays the gas. Because the `SignedDelegateAction` is fully self-contained and publicly visible in the mempool, any unprivileged third party can extract it, wrap it in their own outer transaction addressed to the same sender account, and submit it with higher priority. When that transaction executes first, it advances the sender's access-key nonce to the value embedded in the `DelegateAction`, making the relayer's copy fail with `DelegateActionInvalidNonce`. This is the exact nearcore analog of the ERC20Permit front-running bug: a signed, nonce-bearing artifact is consumed in a different submission context, invalidating the original intended submission.

---

### Finding Description

`DelegateAction` (V1) carries a plain `Nonce` field that is validated and updated against the sender's `AccessKey.nonce` — the same counter used for regular transactions. [1](#0-0) 

`validate_delegate_action_key` reads `access_key.nonce` as the current nonce, rejects the delegate nonce if `<= current_nonce`, and on success writes the delegate nonce back into `access_key.nonce`: [2](#0-1) [3](#0-2) 

The outer transaction that carries the `SignedDelegateAction` is signed by the **relayer** (or any other account), not by the sender. The protocol only requires that the outer transaction's `receiver_id` equals `DelegateAction.sender_id`; the inner signature is verified independently: [4](#0-3) 

Because the `SignedDelegateAction` is fully self-contained (it carries its own signature, nonce, sender, receiver, and actions), any account that observes the relayer's pending transaction in the mempool can:

1. Extract the `SignedDelegateAction` bytes verbatim.
2. Construct a new outer transaction: `attacker → alice [same SignedDelegateAction]`.
3. Submit it with a higher gas price so it is included first.
4. The runtime executes `validate_delegate_action_key`, advances `alice.access_key.nonce` to the delegate nonce value, and succeeds.
5. The relayer's original transaction arrives and fails: `DelegateActionInvalidNonce { delegate_nonce: N+1, ak_nonce: N+1 }`.

The attacker pays only the gas for the outer transaction (which is cheap — the relayer's gas budget for the inner actions is not charged to the attacker because the inner actions are not re-executed; the receipt is simply forwarded). The relayer loses its gas and Alice's intended action is delayed until she re-signs with a fresh nonce.

The `DelegateActionV2` / `VersionedSignedDelegateAction` path uses a `TransactionNonce` (which may carry a `nonce_index` for gas keys), but the same structural problem applies: the signed payload is publicly extractable and re-submittable by any account. [5](#0-4) 

---

### Impact Explanation

- **Relayer griefing**: Any relayer's pending meta-transaction can be invalidated at will. The relayer burns gas with no execution of the inner actions.
- **User action delay**: Alice must re-sign a new `DelegateAction` with a higher nonce and re-coordinate with the relayer. If `max_block_height` has passed by then, the signed payload is permanently expired.
- **Protocol reputation**: Meta-transactions are advertised as a gasless UX primitive. Systematic front-running makes relayers unreliable.
- **No financial gain for attacker**: Pure griefing, identical in character to the ERC20Permit report.

Severity: **High** (griefing with protocol-level impact on a core feature; no attacker profit required).

---

### Likelihood Explanation

- Any unprivileged account can perform this attack — no special role, key, or stake required.
- The `SignedDelegateAction` is fully visible in the mempool inside the outer transaction's action list.
- The attacker's cost is one cheap outer transaction (no inner-action gas charged to them).
- The attack window is bounded by `max_block_height`, but that window is typically set to tens or hundreds of blocks, giving ample time.
- Automated bots can monitor the mempool and front-run systematically.

---

### Recommendation

Apply the same pattern recommended in the external report: before consuming the nonce, check whether the desired effect is already in place. For `DelegateAction` the direct analog is:

> If `access_key.nonce >= delegate_action.nonce`, the nonce has already been consumed. Instead of failing with `DelegateActionInvalidNonce`, check whether the nonce was consumed by an execution of *this exact* `SignedDelegateAction` (e.g., by recording a receipt-ID or action-hash in state). If it was, treat the submission as a no-op (idempotent success) rather than an error, so the relayer is not penalized for a front-run.

A simpler near-term mitigation is to give `DelegateAction` its own independent nonce counter (separate from `AccessKey.nonce`) so that a front-run does not also invalidate any subsequent regular transactions the sender may have queued, and so that the nonce space cannot be exhausted by regular-transaction activity.

The root change would be in `validate_delegate_action_key` in `runtime/runtime/src/actions.rs` and the `DelegateAction` / `DelegateActionV2` structs in `core/primitives/src/action/delegate.rs`.

### Proof of Concept

```
Precondition: alice.access_key.nonce = N

Step 1 – Alice signs a DelegateAction:
  DelegateAction {
    sender_id: "alice",
    receiver_id: "bob",
    actions: [Transfer(100 NEAR)],
    nonce: N+1,
    max_block_height: current_height + 100,
    public_key: alice_pub_key,
  }
  signature = sign(get_nep461_hash(delegate_action), alice_priv_key)

Step 2 – Alice hands SignedDelegateAction to relayer (off-chain).

Step 3 – Relayer broadcasts:
  Transaction {
    signer_id: "relayer",
    receiver_id: "alice",   // must equal sender_id
    nonce: relayer_nonce,
    actions: [Delegate(SignedDelegateAction)],
    block_hash: ...,
  }

Step 4 – Attacker observes the mempool, extracts SignedDelegateAction, broadcasts:
  Transaction {
    signer_id: "attacker",
    receiver_id: "alice",   // same requirement
    nonce: attacker_nonce,
    actions: [Delegate(same SignedDelegateAction)],  // identical bytes
    block_hash: ...,
    // higher gas price → included first
  }

Step 5 – Attacker's tx executes:
  validate_delegate_action_key: N+1 > N  ✓
  access_key.nonce set to N+1
  Inner receipt forwarded to "bob" (attacker pays this gas)

Step 6 – Relayer's tx executes:
  validate_delegate_action_key: N+1 <= N+1  ✗
  → DelegateActionInvalidNonce { delegate_nonce: N+1, ak_nonce: N+1 }
  Relayer's gas is burned; Alice's action was executed by the attacker
  (or attacker can choose NOT to forward the inner receipt by using a
   different receiver_id in the outer tx — but the nonce is still consumed,
   so the relayer's tx still fails regardless)
``` [6](#0-5) [3](#0-2) [1](#0-0)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
}
