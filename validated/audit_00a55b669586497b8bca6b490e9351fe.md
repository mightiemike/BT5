### Title
Stale `digestToSubaccount` Mapping After Isolated Subaccount Closure Enables Undercollateralized Trading — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

When an isolated subaccount is closed via `_tryCloseIsolatedSubaccount`, the `digestToSubaccount[digest]` and `digestToMargin[digest]` mappings are not cleared. A trader can close their isolated subaccount (recovering margin), then reuse the same signed order to trade through the now-zero-balance isolated subaccount, bypassing the margin requirement entirely.

---

### Finding Description

In `OffchainExchange.createIsolatedSubaccount`, when a new isolated subaccount is created, two persistent storage entries are written:

```solidity
digestToSubaccount[digest] = newIsolatedSubaccount;   // line 1072
// ...
if (margin > 0) {
    digestToMargin[digest] = margin;                   // line 1076
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
}
``` [1](#0-0) 

When the isolated subaccount is later closed via `_tryCloseIsolatedSubaccount`, the function correctly returns margin to the parent and clears `isolatedSubaccountsMask`, `isolatedSubaccounts[parent][id]`, and `parentSubaccounts[subaccount]`. However, it does **not** clear `digestToSubaccount[digest]` or `digestToMargin[digest]`:

```solidity
isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
isolatedSubaccounts[parent][id] = bytes32(0);
parentSubaccounts[subaccount] = bytes32(0);
// digestToSubaccount[digest] is NOT cleared
// digestToMargin[digest] is NOT cleared
emit CloseIsolatedSubaccount(subaccount, parent);
``` [2](#0-1) 

The stale `digestToSubaccount[digest]` entry is then consumed in two places:

**1. `createIsolatedSubaccount` early-return path** — when the same order is resubmitted, the function returns the old (closed) subaccount immediately without re-transferring margin:

```solidity
if (digestToSubaccount[digest] != bytes32(0)) {
    return digestToSubaccount[digest];   // early return, no margin transferred
}
``` [3](#0-2) 

**2. `matchOrders` sender redirect** — the stale mapping silently redirects the order to the closed, zero-balance isolated subaccount:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
}
``` [4](#0-3) 

The `isHealthy` check in `matchOrders` is a no-op (always returns `true`), so no health gate prevents the trade from settling against the zero-balance subaccount:

```solidity
function isHealthy(bytes32 /* subaccount */) internal view virtual returns (bool) {
    return true;
}
``` [5](#0-4) 

The signature check in `_validateOrder` passes because `address(uint160(bytes20(isolatedSubaccount)))` resolves to the parent's address (the first 20 bytes of the isolated subaccount encoding), so the parent's original signature remains valid:

```solidity
newIsolatedSubaccount = bytes32(
    (uint256(uint160(senderAddress)) << 96) | ...
);
``` [6](#0-5) 

---

### Impact Explanation

A trader can:
1. Create an isolated subaccount with order `O`, depositing margin `M` into it.
2. Close the isolated subaccount — margin `M` is returned to the parent; `digestToSubaccount[digest(O)]` remains set.
3. Resubmit order `O` to the sequencer. The sequencer calls `CreateIsolatedSubaccount` → returns the old (closed) subaccount with no margin re-transfer.
4. The sequencer calls `matchOrders` with order `O` → the order is matched against the zero-balance isolated subaccount.
5. The trade settles, opening a perp position with zero collateral backing it.

The protocol accumulates bad debt: the isolated subaccount holds an uncollateralized perp position, and the parent has already recovered the margin. Any loss on the position is socialized across other participants via `socializeSubaccount`.

---

### Likelihood Explanation

The attack requires the sequencer to process the resubmitted order. The sequencer is a trusted off-chain component, but the on-chain code imposes no guard against this scenario. A user who controls their own order submission can craft a long-expiry, unfilled (or partially filled) isolated order, close the subaccount, and resubmit the order.
