### Title
Incomplete Merkle Proof Validation Causes `verify_transaction_inclusion` to Always Panic for Single-Transaction Blocks — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` contains an overly restrictive guard that rejects empty merkle proofs unconditionally. For single-transaction Bitcoin blocks, an empty proof is the mathematically correct proof (the transaction hash *is* the merkle root). The underlying `compute_root_from_merkle_proof` utility handles this case correctly, but the guard fires before it is reached, causing the function to always panic for this valid input class. The recommended replacement, `verify_transaction_inclusion_v2`, delegates to the same broken function and inherits the same failure.

---

### Finding Description

In `contract/src/lib.rs` at line 315, `verify_transaction_inclusion` unconditionally rejects empty merkle proofs:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

For a Bitcoin block containing exactly one transaction (a coinbase-only block), the merkle tree has a single leaf. The merkle root equals the transaction hash directly; no sibling hashes are needed. The correct proof is an empty vector.

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` already handles this correctly: when `merkle_proof` is empty, the `for` loop does not execute and the function returns `transaction_hash` unchanged.

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {   // ← loop body never runs for empty proof
        ...
    }
    current_hash   // returns transaction_hash unchanged
}
``` [2](#0-1) 

For a single-tx block, `compute_root_from_merkle_proof(tx_id, 0, &[])` would return `tx_id`, which equals `header.block_header.merkle_root`. The verification would succeed — but the `require!` guard fires first and panics unconditionally.

`verify_transaction_inclusion_v2`, the function explicitly recommended as the secure replacement for the deprecated `verify_transaction_inclusion`, delegates to the same broken path:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

The length-equality pre-check in `verify_transaction_inclusion_v2` passes for two empty proofs (`0 == 0`), the coinbase merkle proof verification also passes (same empty-proof logic), and then the call to `verify_transaction_inclusion` panics. [4](#0-3) 

---

### Impact Explanation

Any NEAR caller supplying a mathematically valid proof for a transaction in a single-transaction block will always receive a contract panic. The proof result is corrupted: instead of returning `true`, the contract aborts. Downstream contracts or bridge applications that gate security-sensitive operations (e.g., releasing bridged assets after confirming a Bitcoin transaction) on the return value of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` are permanently broken for this class of blocks. The broken invariant is: *a valid merkle proof must produce a correct boolean result*; instead it produces an unconditional panic.

---

### Likelihood Explanation

Single-transaction (coinbase-only) blocks are a normal part of the Bitcoin protocol and appear during low-activity periods and in early chain history. The entry path is fully unprivileged: any NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with a valid single-tx block proof. No special role, key, or configuration is required to trigger the panic.

---

### Recommendation

Remove the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard from `verify_transaction_inclusion`. [5](#0-4) 

`compute_root_from_merkle_proof` already handles the empty-proof case correctly and returns the transaction hash unchanged, which is the correct merkle root for a single-transaction block. No guard is needed; the mathematical check at the end of the function is sufficient.

---

### Proof of Concept

1. Deploy the contract with a Bitcoin genesis block.
2. Submit a single-transaction block where `merkle_root == coinbase_tx_id`.
3. Call `verify_transaction_inclusion` (or `verify_transaction_inclusion_v2`) with:
   - `tx_id = coinbase_tx_id`
   - `tx_block_blockhash = <that block's hash>`
   - `tx_index = 0`
   - `merkle_proof = []` (empty — the correct proof for a single-tx block)
   - `confirmations = 1`
4. The call panics with `"Merkle proof is empty"` despite the proof being valid.
5. Manually evaluating `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id`, which equals `merkle_root` — confirming the proof is correct and the guard is the sole cause of failure.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
