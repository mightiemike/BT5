### Title
Unchecked Boolean Return Value from `equihash::is_valid_solution` Allows Invalid Equihash Proof Acceptance - (File: `contract/src/zcash.rs`)

---

### Summary

`check_pow` in the Zcash build of the BTC light client calls `equihash::is_valid_solution`, which returns `Result<bool, Error>`. The code handles the `Err` variant by panicking, but **completely discards the `bool` payload of the `Ok` variant**. When the library returns `Ok(false)` — a structurally valid call that signals an invalid solution — the contract silently continues and accepts the block header.

---

### Finding Description

In `contract/src/zcash.rs`, the Equihash solution check is written as:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
``` [1](#0-0) 

`equihash::is_valid_solution` has the signature `fn is_valid_solution(...) -> Result<bool, Error>`. The two distinct failure modes are:

| Return value | Meaning | Contract reaction |
|---|---|---|
| `Err(e)` | Library-level error (e.g., bad parameter) | Panics — correct |
| `Ok(false)` | Solution is structurally parseable but cryptographically invalid | **Silently ignored — bug** |
| `Ok(true)` | Solution is valid | Silently ignored — happens to be correct |

Because `.unwrap_or_else` only intercepts the `Err` arm and the resulting `bool` is never bound or asserted, a block carrying an `Ok(false)` Equihash solution passes `check_pow` without rejection.

Every other PoW check in the same function uses `require!` to assert a boolean condition. The Equihash check is the sole exception. [2](#0-1) 

---

### Impact Explanation

An attacker can submit a Zcash block header whose `solution` field is syntactically well-formed (so the equihash library does not return `Err`) but cryptographically invalid (so the library returns `Ok(false)`). The contract will:

1. Accept the header into `headers_pool`.
2. Potentially update `mainchain_tip_blockhash` if the header's chain work exceeds the current tip.
3. Return `true` from `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` for any SPV proof anchored to that fraudulent block.

This breaks the core security invariant of the light client: downstream NEAR contracts that call `verify_transaction_inclusion` to gate asset releases or cross-chain actions will receive a forged `true` result for transactions that never existed on the real Zcash chain. [3](#0-2) 

---

### Likelihood Explanation

The Zcash equihash crate's `is_valid_solution` is documented to return `Ok(false)` for solutions that fail the cryptographic validity check (wrong indices, duplicate indices, etc.). Crafting a header with a syntactically valid but semantically invalid solution is straightforward — no mining hardware or privileged access is required. Any unprivileged NEAR account can call `submit_blocks` with such a header. [4](#0-3) 

---

### Recommendation

Assert the boolean return value explicitly:

```rust
let valid = equihash::is_valid_solution(
    n, k, &input, &block_header.nonce.0, &block_header.solution,
)
.unwrap_or_else(|e| env::panic_str(&format!("Invalid Equihash solution: {e}")));

require!(valid, "Invalid Equihash solution: proof check returned false");
```

This mirrors the pattern used for every other PoW assertion in the same function and closes the unchecked-return-value gap. [1](#0-0) 

---

### Proof of Concept

1. Construct a Zcash block header whose `solution` field contains the correct number of bytes for an Equihash(200,9) solution but with indices that do not satisfy the algorithm (e.g., all-zero indices).
2. Call `submit_blocks` on the deployed Zcash-feature contract with this header (no stake or role required).
3. Observe that `equihash::is_valid_solution` returns `Ok(false)`, `.unwrap_or_else` does not fire, and the function returns `false` which is discarded.
4. The header is stored in `headers_pool` and, if its chain work exceeds the current tip, becomes the new `mainchain_tip_blockhash`.
5. Call `verify_transaction_inclusion_v2` with a fabricated SPV proof anchored to this block; the contract returns `true`. [5](#0-4)

    #[pause]
    #[trusted_relayer]
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
