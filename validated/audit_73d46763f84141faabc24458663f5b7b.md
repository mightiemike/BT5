### Title
`build_precommit_vote_message_digest` Omits `height` and `round` from Signed Preimage, Enabling Cross-Round Signature Replay — (`crates/apollo_signature_manager/src/signature_manager.rs`)

---

### Summary

The `build_precommit_vote_message_digest` function constructs the ECDSA signing preimage for consensus precommit votes using only the domain separator `PRECOMMIT_VOTE` and the `block_hash`. It omits `height` (block number) and `round` from the signed bytes. Because the `Vote` struct carries `height`, `round`, `vote_type`, `voter`, `proposal_commitment`, and `signature` as separate fields, a signature produced over `PRECOMMIT_VOTE || block_hash` is valid for any `(height, round)` pair that shares the same `block_hash`. Since `block_hash` already commits to `block_number` (via `calculate_block_hash`), cross-height replay is practically impossible, but **cross-round replay within the same height is fully possible**: a precommit signature produced at round 0 is cryptographically indistinguishable from one produced at round 1 for the same block.

---

### Finding Description

In `crates/apollo_signature_manager/src/signature_manager.rs`, the preimage for a precommit vote is built as:

```rust
fn build_precommit_vote_message_digest(block_hash: BlockHash) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let mut message = Vec::with_capacity(PRECOMMIT_VOTE.len() + block_hash.len());
    message.extend_from_slice(PRECOMMIT_VOTE);   // domain tag
    message.extend_from_slice(&block_hash);       // only field
    MessageDigest(blake2s_to_felt(&message))
}
``` [1](#0-0) 

The `Vote` struct that carries this signature contains `height`, `round`, `vote_type`, `voter`, `proposal_commitment`, and `signature`: [2](#0-1) 

The signed message binds only to `block_hash`. Neither `height` nor `round` is included. The `block_hash` is computed by `calculate_block_hash`, which does include `block_number`: [3](#0-2) 

So cross-height replay is practically blocked. However, `round` is **not** part of the block hash computation and is **not** included in the signed preimage. A precommit signature for `(height=H, round=0, block_hash=X)` is byte-for-byte identical to a valid signature for `(height=H, round=1, block_hash=X)`.

The Tendermint-style consensus engine used here explicitly supports reproposing the same block at a later round via `SMRequest::Repropose`: [4](#0-3) 

This means the same `block_hash` can legitimately appear at multiple rounds within the same height, making cross-round replay a reachable attack path.

The developers have already noted a related concern in a TODO comment immediately above the digest-building utilities: [5](#0-4) 

The verification counterpart `verify_precommit_vote_signature` uses the same incomplete preimage: [6](#0-5) 

---

### Impact Explanation

An attacker who observes a validator's precommit vote at `(height=H, round=0)` for `block_hash=X` can rebroadcast that vote with the `round` field changed to `1` (or any later round). The embedded ECDSA signature remains valid because `round` is not part of the signed preimage. If the same block is reproposed at round 1 (a normal Tendermint scenario), the replayed vote is indistinguishable from a genuine round-1 precommit from that validator. Accumulating enough such replayed votes can manufacture a false quorum, causing the sequencer to commit a block that did not receive genuine 2/3 precommit support at the claimed round. This maps to: **High — signature/hash logic binds the wrong hash/type, binding the wrong executable payload context.**

---

### Likelihood Explanation

The Tendermint repropose path (`SMRequest::Repropose`) is a normal code path exercised whenever round 0 times out. Any network-level observer (including other validators or a malicious peer) can collect precommit votes from round 0 and replay them at round 1 with a modified `round` field. No privileged access is required; the attacker only needs to be a participant in the P2P gossip layer.

---

### Recommendation

Include `height` and `round` in the signed preimage, mirroring how the peer-identity digest includes all context fields:

```rust
fn build_precommit_vote_message_digest(
    block_hash: BlockHash,
    height: BlockNumber,
    round: Round,
) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let height_bytes = height.0.to_be_bytes();
    let round_bytes = round.to_be_bytes();
    let mut message = Vec::with_capacity(
        PRECOMMIT_VOTE.len() + block_hash.len() + height_bytes.len() + round_bytes.len()
    );
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash);
    message.extend_from_slice(&height_bytes);
    message.extend_from_slice(&round_bytes);
    MessageDigest(blake2s_to_felt(&message))
}
```

Update `sign_precommit_vote` and `verify_precommit_vote_signature` to accept and pass `height` and `round`. Callers in the consensus orchestrator already have both values available when constructing or verifying a `Vote`.

---

### Proof of Concept

1. Validator V signs a precommit for `block_hash=X` at `height=5, round=0`. The signed bytes are `blake2s(b"PRECOMMIT_VOTE" || X.to_bytes_be())`.
2. Round 0 fails (no quorum). The consensus engine reproposals the same block at round 1.
3. An attacker takes V's `Vote { height=5, round=0, proposal_commitment=X, signature=S }` and constructs `Vote { height=5, round=1, proposal_commitment=X, signature=S }`.
4. `verify_precommit_vote_signature(X, S, V.public_key)` returns `true` because the preimage is identical — `round` is not part of it.
5. The forged round-1 vote is accepted by the consensus engine as a genuine precommit from V, contributing to a false quorum at round 1. [7](#0-6) [1](#0-0) [6](#0-5)
