### Title
Difficulty Retarget Uses Stale Main-Chain Ancestor Instead of Fork Ancestor - (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary

All three chain-specific `get_next_work_required` implementations resolve the first block of a difficulty period by calling `blocks_getter.get_header_by_height(first_block_height)`, which unconditionally reads from `mainchain_height_to_header`. When the block being validated belongs to a fork that diverged before `first_block_height`, the function silently uses the main-chain block's timestamp instead of the fork's actual ancestor's timestamp. This is the same class of bug as H-8: a value that must be computed from the current, relevant chain state is instead taken from a stale, mismatched reference.

### Finding Description

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

It always returns the block that the **main chain** recorded at that height. The difficulty retarget logic in all three chain modules calls this function to obtain the timestamp of the first block in the adjustment window:

**Bitcoin** (`bitcoin.rs`):
```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header,
    interval_tail_extend_header.block_header.time.into())
``` [2](#0-1) 

**Dogecoin** (`dogecoin.rs`) — the developers themselves flagged this with a TODO:
```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [3](#0-2) 

**Litecoin** (`litecoin.rs`):
```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header,
    interval_tail_extend_header.block_header.time.into())
``` [4](#0-3) 

When a fork block is submitted at height `D+k` and the fork diverged at height `D`, any `height_first` value that falls at or after `D` will resolve to the **main-chain** block at that height, not the fork's ancestor. The two blocks have different timestamps, so the computed `actual_time_taken` / `modulated_timespan` is wrong, yielding an incorrect expected `bits` value.

The `check_pow` / `check_target` call happens **before** `submit_block_header_inner`, so the fork block is accepted or rejected based on this incorrect difficulty:

```rust
if !skip_pow_verification {
    self.check_target(&header, &prev_block_header);   // uses wrong first_block_time
    ...
}
self.submit_block_header_inner(current_header, &prev_block_header);
``` [5](#0-4) 

### Impact Explanation

An attacker who controls a fork can set the timestamp of the fork's divergence block (`T_fork_D`) to be later than the main-chain block at the same height (`T_main_D`). For every subsequent fork block at a retarget boundary, the contract computes:

```
modulated_timespan = T_fork_prev - T_main_D   (wrong)
```

instead of:

```
modulated_timespan = T_fork_prev - T_fork_D   (correct)
```

Because `T_fork_D > T_main_D`, the contract overestimates the elapsed time, which lowers the required difficulty. The attacker mines fork blocks at this artificially reduced difficulty. If the fork's accumulated `chain_work` exceeds the main chain's, `reorg_chain` is triggered and the contract's canonical chain is replaced by a chain containing blocks that do not satisfy the protocol's actual difficulty rules. This corrupts the canonical-chain mapping and invalidates all subsequent SPV proofs.

For **Dogecoin** (post-block 145 000, `difficulty_adjustment_interval = 1`), the retarget fires on every single block, so the bug is reachable starting from the **second** fork block. The `calculate_next_work_required` clamps the timespan to `[45 s, 90 s]`, giving up to a 2× difficulty reduction per block. [6](#0-5) 

For **Bitcoin / Litecoin**, the fork must diverge before the start of the current 2016-block window, which requires more hashpower but is still within scope for a well-resourced attacker.

### Likelihood Explanation

Any unprivileged NEAR caller can invoke `submit_blocks` with adversarially crafted headers. The attacker needs only to:
1. Build a fork that diverges before a retarget boundary.
2. Set the divergence block's timestamp later than the corresponding main-chain block (within `MAX_FUTURE_BLOCK_TIME_LOCAL`).
3. Mine subsequent fork blocks at the reduced difficulty.

No privileged role, leaked key, or social engineering is required. For Dogecoin the window opens after just two fork blocks.

### Recommendation

Replace the `get_header_by_height` call with a backward walk along `prev_block_hash` links starting from `prev_block_header`, so the first-period block is always the fork's true ancestor:

```rust
// Walk back `blocks_to_go_back` steps through prev_block_hash links
let mut cursor = prev_block_header.clone();
for _ in 0..blocks_to_go_back {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time = cursor.block_header.time;
```

This mirrors how Bitcoin Core resolves the ancestor (`GetAncestor`) and ensures the difficulty calculation is always relative to the chain being validated, not the current main chain.

### Proof of Concept

1. Main chain has blocks at heights 0…`D` with `T_main_D = 1_000_000`.
2. Attacker submits a fork block at height `D` with `T_fork_D = 1_000_060` (60 s later, within `MAX_FUTURE_BLOCK_TIME_LOCAL`).
3. Attacker submits fork block at height `D+1` with `T_fork_D1 = 1_000_120`.
4. For the fork block at height `D+2` (a retarget boundary in Dogecoin), the contract calls `get_header_by_height(D)` and gets `T_main_D = 1_000_000`.
5. `modulated_timespan = 1_000_120 - 1_000_000 = 120 s` → clamped to `max_timespan = 90 s`.
6. Correct value would be `1_000_120 - 1_000_060 = 60 s` → `modulated_timespan = 60 s`.
7. The contract accepts a fork block at height `D+2` mined to the difficulty corresponding to 90 s spacing instead of the correct 60 s spacing — a ~33% easier target.
8. Repeated over many blocks, the fork accumulates enough chainwork to trigger `reorg_chain`, replacing the canonical chain with one containing under-difficulty blocks.
