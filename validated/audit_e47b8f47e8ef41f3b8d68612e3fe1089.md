### Title
Deprecated `verify_transaction_inclusion` Accepts Fake Transaction IDs (Internal Merkle Nodes), Returning `true` for Non-Existent Transactions - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` is a publicly callable NEAR function that accepts a caller-supplied `tx_id` without verifying it is a real Bitcoin transaction hash rather than an internal Merkle tree node. Any unprivileged NEAR caller can supply a crafted internal-node hash as `tx_id`, causing the function to return `true` and falsely certifying that a non-existent transaction was included in a confirmed Bitcoin block. Downstream NEAR contracts that consume this result are deceived.

### Finding Description
`verify_transaction_inclusion` is marked `#[deprecated]` but remains a live, publicly callable contract method — the Rust `#[deprecated]` attribute emits only a compiler warning; it imposes no runtime access restriction. The function is gated only by `#[pause]`, which is inactive by default. [1](#0-0) 

The function accepts a `ProofArgs` struct containing a caller-controlled `tx_id` and a `merkle_proof`. It verifies that the supplied `tx_id`, walked up the Merkle path, reaches the block's `merkle_root`. It performs no check that `tx_id` is a leaf-level transaction hash rather than an internal 64-byte node hash. [2](#0-1) 

The code itself documents this broken invariant:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

The 64-byte transaction forgery attack (https://www.bitmex.com/blog/64-Byte-Transactions) exploits exactly this: an internal Merkle node is 64 bytes (two concatenated 32-byte hashes). A crafted 64-byte byte string can be interpreted as both a valid internal node and a plausible "transaction hash." An attacker who knows the Merkle tree of any confirmed block can compute an internal node hash, construct a valid Merkle path from that node to the root, and call `verify_transaction_inclusion` with that node hash as `tx_id`. The function will compute the same root and return `true`.

`verify_transaction_inclusion_v2` was introduced to fix this by requiring a coinbase proof of equal depth, but it does not remove or restrict `verify_transaction_inclusion`. [4](#0-3) 

### Impact Explanation
Any NEAR contract that calls `verify_transaction_inclusion` to gate a privileged action (e.g., releasing bridged BTC, minting wrapped tokens, settling a payment) can be deceived into accepting a proof for a Bitcoin transaction that never existed. The corrupted proof result is `true` for a fabricated `tx_id`, which is the exact canonical value this function is trusted to produce correctly. The broken invariant is: *the function must return `true` only for leaf-level transaction hashes that genuinely appear in the block's Merkle tree*.

### Likelihood Explanation
The attack requires only:
1. Knowledge of any confirmed Bitcoin block hash stored in the contract's `mainchain_header_to_height` map (publicly readable on-chain).
2. The ability to compute an internal Merkle node hash from that block's transaction set (public Bitcoin data).
3. A single NEAR transaction calling `verify_transaction_inclusion` — no role, no stake, no privileged key required.

This is a low-skill, zero-cost attack for any party with access to Bitcoin block data.

### Recommendation
1. **Remove or gate `verify_transaction_inclusion`**: Since `verify_transaction_inclusion_v2` is the safe replacement, `verify_transaction_inclusion` should be made `#[private]` or removed entirely to prevent direct external calls. Marking it `#[deprecated]` is insufficient.
2. **Add a coinbase-depth check inside `verify_transaction_inclusion`**: If the function must remain public, add the same coinbase proof requirement that `verify_transaction_inclusion_v2` enforces, so the internal-node forgery path is closed at the leaf level.
3. **Document the trust boundary**: Any downstream contract consuming either SPV function must be warned that `tx_id` legitimacy (leaf vs. internal node) is not enforced by the contract itself.

### Proof of Concept
```
1. Observe any confirmed block hash B stored in mainchain_header_to_height (public state).
2. Fetch block B's full transaction list from a Bitcoin node.
3. Compute the Merkle tree. Pick any internal node N at depth D with siblings S_1..S_D.
4. Construct merkle_proof = [S_1, ..., S_D-1] (the path from N up to the root, excluding N itself and the root).
5. Call verify_transaction_inclusion({
       tx_id: N,                  // internal node hash, not a real tx
       tx_block_blockhash: B,
       tx_index: <position of N at depth D>,
       merkle_proof: [S_1..S_D-1],
       confirmations: 1
   })
6. The function computes compute_root_from_merkle_proof(N, index, proof) == merkle_root of B → true.
7. The function returns true, falsely certifying that N is an included transaction.
``` [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
