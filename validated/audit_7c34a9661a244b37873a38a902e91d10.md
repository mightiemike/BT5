Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of actual user, allowing allowlist bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates pool swaps by checking the `sender` argument, which is always `msg.sender` of the `pool.swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router's address. If a pool admin allowlists the router to enable router-mediated swaps for permitted users, any unprivileged user can bypass the allowlist entirely by calling through the router, as the extension sees the allowlisted router address and permits the swap.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs the check `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the extension is called by the pool via `_callExtensionsInOrder`) and `sender` is the first argument passed by the pool. [1](#0-0) 

In `MetricOmmPool.swap()`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`: [2](#0-1) 

When `MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()`, `msg.sender` at the pool is the router contract, not the original user. The router never forwards the original caller's identity: [3](#0-2) 

The extension therefore evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][actual_user]`. The same issue applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the `owner` parameter (the economic actor explicitly passed by the pool), not `sender`: [4](#0-3) 

The swap hook has no equivalent "economic actor" parameter — only `sender` (the immediate caller) and `recipient` (the output destination) — so there is no correct address to check when the swap is router-mediated.

## Impact Explanation
A pool admin who deploys a pool with `SwapAllowlistExtension` and allowlists the router (a natural action to enable router-mediated swaps for permitted users) inadvertently opens the pool to all users. Any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle()` targeting the restricted pool; the extension sees `sender = router`, finds the router allowlisted, and permits the swap. The allowlist guard is completely bypassed, breaking the core access-control invariant of the extension. Swaps that should be blocked execute and settle real token transfers, constituting a broken core pool functionality causing loss of funds for the pool admin's intended access-control model.

## Likelihood Explanation
The trigger requires the pool admin to allowlist the router address. This is a reasonable and expected action: a pool admin who wants their allowlisted users to use the standard router will allowlist it. The admin has no on-chain signal that doing so opens the pool to all users, because `isAllowedToSwap` only checks a single address and does not expose the router-identity conflation. The bypass is then reachable by any unprivileged user with no special permissions, no capital requirements beyond the swap input, and is repeatable indefinitely.

## Recommendation
The router should embed the original user's address in `extensionData` on every hop (e.g., `abi.encode(msg.sender)`), and `SwapAllowlistExtension.beforeSwap` should decode and check that address when `extensionData.length >= 32`, falling back to `sender` otherwise. Alternatively, the pool could expose a separate `originalSender` slot in the hook arguments (analogous to how `DepositAllowlistExtension` correctly checks `owner` rather than `sender`), ensuring the extension always gates the economic actor rather than the immediate caller.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only Alice should be able to swap.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — to let Alice use the router.
4. Charlie (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(recipient, ...)` with `msg.sender = router`.
6. The pool calls `_beforeSwap(router, ...)`, which calls `SwapAllowlistExtension.beforeSwap(router, ...)`.
7. The extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
8. Charlie's swap executes and settles real token transfers, bypassing the allowlist entirely.

A Foundry integration test can confirm this by: deploying the pool with the extension, setting Alice and the router as allowed, then pranking as Charlie to call `exactInputSingle` and asserting the swap succeeds without revert.
