### Title
Taker Fee Bypass via Uninitialized Fee Rates for New Products When Tier Is in `nonDefaultFeeTierMask` — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`getTierFeeRateX18` uses a single bitmask (`nonDefaultFeeTierMask`) to decide whether to return stored fee rates or the hardcoded default. The mask is set at the **tier level**, but fee rates are stored at the **(tier, productId)** level. When a new product is added after a tier has been marked non-default, `feeRates[tier][newProductId]` is uninitialized (`FeeRates(0, 0)`). Any user who routes an order through a registered builder whose `defaultFeeTier` is in `nonDefaultFeeTierMask` will pay **zero taker fees** on the new product instead of the default 2 bps.

---

### Finding Description

`getTierFeeRateX18` branches on whether the tier is in `nonDefaultFeeTierMask`:

```solidity
// OffchainExchange.sol lines 933–946
function getTierFeeRateX18(uint32 tier, uint32 productId)
    public view returns (FeeRates memory)
{
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        return feeRates[tier][productId];   // ← returns (0,0) if never set
    }
    return FeeRates({
        makerRateX18: 0,
        takerRateX18: 200_000_000_000_000   // 2 bps default
    });
}
```

`nonDefaultFeeTierMask` is set whenever `updateTierFeeRates` is called for **any** product under a tier:

```solidity
// OffchainExchange.sol line 990
nonDefaultFeeTierMask |= uint128(1) << txn.tier;
```

This means once tier `T` is marked non-default (even for a single product), **all** products under tier `T` will return stored rates — including products added later whose rates were never initialized.

In `getUserFeeRateWithBuilder`, a user can elevate their effective tier to a builder's `defaultFeeTier`:

```solidity
// OffchainExchange.sol lines 498–506
uint32 feeTier = feeTiers[address(uint160(bytes20(sender)))];
if (feeTier < builder.defaultFeeTier) {
    feeTier = builder.defaultFeeTier;
}
FeeRates memory userFeeRates = getTierFeeRateX18(feeTier, productId);
```

A user controls the `builderId` field in their order's `appendix`. By pointing to any registered builder whose `defaultFeeTier` is in `nonDefaultFeeTierMask` and whose rates for the new product are uninitialized, the user receives `FeeRates(0, 0)`.

In `applyFee`, a zero `feeRate` means `keepRateX18 = ONE`, so `fee = 0`:

```solidity
// OffchainExchange.sol lines 556–560
int128 keepRateX18 = ONE - feeInfo.feeRate;
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(keepRateX18)
    : meteredQuote.div(keepRateX18);
orderInfo.fee = meteredQuote - newMeteredQuote;  // = 0 when feeRate = 0
```

---

### Impact Explanation

A user trading on a newly added product can pay **zero taker fees** (instead of the default 2 bps) by routing their order through a registered builder whose `defaultFeeTier` is in `nonDefaultFeeTierMask` but has no rates configured for the new product. The protocol loses all taker fee revenue on that product for those orders. For high-volume traders, 2 bps per trade is material. The corrupted state is `marketInfo[productId].collectedFees`, which accumulates less than it should, and the `X_ACCOUNT` (protocol treasury) receives fewer fees via `dumpFees`.

---

### Likelihood Explanation

The scenario is realistic in a growing protocol:
1. Admin registers a builder with `defaultFeeTier = T` (e.g., a VIP tier for a major integrator).
2. Admin calls `updateTierFeeRates` for tier `T` on existing products → `nonDefaultFeeTierMask |= (1 << T)`.
3. A new trading product is added to the protocol.
4. Admin forgets to call `updateTierFeeRates` for tier `T` on the new product.
5. Any user can now set `builderId` in their order appendix to point to the builder with `defaultFeeTier = T` and trade the new product with zero taker fees.

The user controls `builderId` in the signed order appendix. The sequencer processes the order as signed. No admin cooperation is needed beyond the initial (normal) builder registration.

---

### Recommendation

In `getTierFeeRateX18`, treat a stored `takerRateX18 == 0` for a non-default tier as uninitialized and fall back to the default rate:

```solidity
function getTierFeeRateX18(uint32 tier, uint32 productId)
    public view returns (FeeRates memory)
{
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        FeeRates memory stored = feeRates[tier][productId];
        if (stored.takerRateX18 != 0 || stored.makerRateX18 != 0) {
            return stored;
        }
    }
    return FeeRates({
        makerRateX18: 0,
        takerRateX18: 200_000_000_000_000
    });
}
```

Alternatively, when a new product is registered, iterate over all non-default tiers and initialize their fee rates to the default.

---

### Proof of Concept

1. Admin calls `updateTierFeeRates({tier: 5, productId: 1, makerRateX18: 0, takerRateX18: 100_000_000_000_000})` → `nonDefaultFeeTierMask |= (1 << 5)`, `feeRates[5][1]` is set.
2. Admin registers builder 7 with `defaultFeeTier = 5`, `lowestFeeRate = 0`, `highestFeeRate = 0`.
3. Admin adds new product with `productId = 99`. `feeRates[5][99]` is never set → remains `FeeRates(0, 0)`.
4. User constructs an order for product 99 with `appendix` encoding `builderId = 7`, `builderFeeRate = 0`.
5. Sequencer calls `matchOrders`. `getUserFeeRateWithBuilder` computes `feeTier = max(userTier=0, builder.defaultFeeTier=5) = 5`.
6. `getTierFeeRateX18(5, 99)` → `nonDefaultFeeTierMask & (1<<5) != 0` → returns `feeRates[5][99]` = `FeeRates(0, 0)`.
7. `applyFee` computes `keepRateX18 = ONE - 0 = ONE`, `fee = 0`. User pays zero taker fees.

        Builder memory builder;

            if (

                builderFeeRate < builder.lowestFeeRate
