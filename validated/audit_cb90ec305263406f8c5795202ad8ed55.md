### Title
`topUpBaseRewards` Drains `RewardsPoolName` Without Reserving Outstanding Bonus Obligations — (`x/tieredrewards/keeper/abci.go`)

### Summary

`topUpBaseRewards`, called every block in `BeginBlocker`, can drain the entire `RewardsPoolName` balance to the distribution module with no floor that reserves funds for accrued-but-unclaimed bonus obligations. When the pool is emptied, all subsequent `ClaimTierRewards` calls that compute a non-zero bonus revert with `ErrInsufficientBonusPool`, making earned bonus permanently unclaimable until the pool is externally replenished. The funds that should have backed those obligations are instead paid to validators.

### Finding Description

`RewardsPoolName` is the single module account that backs two distinct liabilities:

1. **Base-rewards top-up** — transferred to `distributiontypes.ModuleName` every block by `topUpBaseRewards`.
2. **Bonus rewards** — paid directly to position owners by `processEventsAndClaimBonus` via `SendCoinsFromModuleToAccount`.

`topUpBaseRewards` computes a `shortFallAmount` and caps it at the full pool balance:

```go
// abci.go:105-111
if poolBalance.Amount.LT(shortFallAmount) {
    topUpAmount = poolBalance.Amount   // entire pool drained
}
// abci.go:113
k.bankKeeper.SendCoinsFromModuleToModule(ctx, types.RewardsPoolName, distributiontypes.ModuleName, ...)
``` [1](#0-0) 

There is no on-chain accounting of total outstanding bonus obligations (no "reserved bonus" state variable), and no floor that prevents the transfer from reducing the pool below the sum of all accrued-but-unclaimed bonus across all active positions.

`sufficientBonusPoolBalance` is a point-in-time spot check at claim time:

```go
// bonus_rewards.go:48-61
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
    poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
    if !poolBalance.IsAllGTE(bonus) {
        return errorsmod.Wrapf(types.ErrInsufficientBonusPool, ...)
    }
    return nil
}
``` [2](#0-1) 

It cannot protect against a pool that was already drained by `topUpBaseRewards` in the same or a prior block.

### Impact Explanation

When `topUpBaseRewards` drains the pool:

- Every call to `processEventsAndClaimBonus` that computes `totalBonus > 0` hits `sufficientBonusPoolBalance` and returns `ErrInsufficientBonusPool`.
- The bonus accrual is still recorded in position state (shares, tier BonusApy, elapsed time), so the obligation exists on-chain, but the backing funds have been transferred to validators via the distribution module.
- Value leaves the `RewardsPoolName` boundary and flows to validators rather than to position owners who earned it. [3](#0-2) 

### Likelihood Explanation

No governance manipulation is required. The precondition is:

- `TargetBaseRewardsRate > 0` (any non-zero value, set at genesis or by governance in normal operation).
- Fee-collector balance is low relative to `totalBonded * TargetBaseRewardsRate / blocksPerYear` — a routine condition during low-activity periods.
- At least one position has accrued bonus since its last claim.

Under these conditions, `topUpBaseRewards` fires every block and will drain the pool to zero over time. The larger `TargetBaseRewardsRate` is relative to actual fee income, the faster the drain.

### Recommendation

Introduce an on-chain running total of outstanding bonus obligations (incremented as bonus accrues, decremented as it is paid). In `topUpBaseRewards`, cap `topUpAmount` so that `poolBalance - topUpAmount >= totalOutstandingBonus`:

```go
reservedForBonus := k.getTotalOutstandingBonus(ctx)
available := poolBalance.Amount.Sub(reservedForBonus)
if available.IsNegative() {
    available = math.ZeroInt()
}
if topUpAmount.GT(available) {
    topUpAmount = available
}
```

Alternatively, split `RewardsPoolName` into two separate module accounts — one for base-rewards top-up and one for bonus payments — so that `topUpBaseRewards` cannot touch bonus funds.

### Proof of Concept

1. Deploy chain with `TargetBaseRewardsRate = 0.20` (20% APR), low fee volume.
2. Create N positions in a tier with `BonusApy > 0`; let them accrue bonus for several blocks without claiming.
3. Observe `BeginBlocker → topUpBaseRewards` draining `RewardsPoolName` to zero each block (fee shortfall > pool balance).
4. Submit `MsgClaimTierRewards` for any position with accrued bonus.
5. Observe the transaction revert with `ErrInsufficientBonusPool` even though the position legitimately earned bonus.
6. Confirm via bank query that `RewardsPoolName` balance is 0 and the distribution module received the drained funds.

The invariant `RewardsPoolName.balance >= Σ computeSegmentBonus(pos)` for all active positions is broken after step 3. [1](#0-0) [2](#0-1) [3](#0-2)
