### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Vulnerable to 64-Byte Merkle Proof Forgery - (File: contract/src/lib.rs)

### Summary
The contract exposes a deprecated `verify_transaction_inclusion` function that is still publicly reachable by any unprivileged NEAR caller. The function lacks coinbase Merkle proof validation, making it vulnerable to the well-documented 64-byte transaction Merkle proof forgery attack. An attacker can supply an internal Merkle tree node as the `tx_id` and construct a valid proof path to the block's Merkle root, causing the function to return `true` for a transaction that does not exist in the block. Consumer contracts (bridges, atomic swaps, cross-chain lending) that call this endpoint can be deceived into accepting fraudulent Bitcoin transaction inclusion proofs.

### Finding Description
`verify_transaction_inclusion` is marked `#[deprecated(since = "0.5.0")]` and carries an explicit warning that it "may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." Despite this, the function remains a live, public, `#[pause]`-gated contract method callable by any NEAR account. [1](#0-0) 

The verification logic reduces to a single Merkle root comparison with no constraint that `tx_id` is a leaf: [2](#0-1) 

Bitcoin's Merkle tree hashes pairs of 32-byte child hashes into a 64-byte input. An internal node's hash is therefore indistinguishable from a leaf hash at the API level. An attacker who knows the Merkle tree of any confirmed block can pick any internal node `N` at depth `d`, set `tx_id = N`, set `tx_index` to the position that `N` occupies at depth `d`, and supply the `d`-element sibling path. `compute_root_from_merkle_proof` will traverse the path and arrive at the correct Merkle root, so the function returns `true`. [3](#0-2) 

The v2 function fixes this by requiring a coinbase proof at index 0, which pins the tree depth and prevents internal-node substitution: [4](#0-3) 

### Impact Explanation
Any consumer contract or off-chain service that calls `verify_transaction_inclusion` to gate a high-value action (e.g., releasing bridged BTC, settling an atomic swap, unlocking collateral in a cross-chain lending protocol) can be made to accept a proof for a Bitcoin transaction that was never broadcast or confirmed. The attacker does not need to mine a block or control any privileged role; they only need to know the Merkle tree of any already-confirmed block, which is public data.

### Likelihood Explanation
The 64-byte Merkle forgery attack is publicly documented (https://www.bitmex.com/blog/64-Byte-Transactions) and the exploit inputs are trivially derivable from any Bitcoin block explorer. The deprecated function is still live and callable. Any consumer that has not migrated to `verify_transaction_inclusion_v2` is immediately exploitable. Likelihood is **High** given the function is reachable with no privilege requirement and the attack inputs are public.

### Recommendation
Remove `verify_transaction_inclusion` entirely from the contract, or gate it so it always panics (breaking change with a migration notice). All callers must be migrated to `verify_transaction_inclusion_v2`, which enforces coinbase proof validation and is not susceptible to the internal-node substitution attack. [5](#0-4) 

### Proof of Concept

1. Identify any confirmed Bitcoin block `B` whose Merkle tree has at least two transactions.
2. Compute the Merkle tree of `B`. Pick any internal node `N` at depth `d` (counting from the root). Let `pos` be the position of `N` among nodes at depth `d`.
3. Collect the `d` sibling hashes that form the Merkle path from `N` to the Merkle root. This is the `merkle_proof`.
4. Submit `B`'s header to the contract via `submit_blocks` (or use an already-stored block).
5. Call `verify_transaction_inclusion` with:
   - `tx_id` = `N` (the internal node hash, not a real transaction)
   - `tx_block_blockhash` = hash of block `B`
   - `tx_index` = `pos`
   - `merkle_proof` = the `d` sibling hashes collected in step 3
   - `confirmations` = 1
6. The function returns `true`, falsely asserting that the non-existent "transaction" `N` is included in block `B`. [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
