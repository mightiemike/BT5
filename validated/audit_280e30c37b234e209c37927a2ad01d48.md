### Title
No Slippage Protection for `burnNlp` / `mintNlp` May Execute at an Unfavorable Oracle Price for the User — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`burnNlp` and `mintNlp` in `Clearinghouse.sol` compute the user's output amount using an `oraclePriceX18` parameter supplied by the sequencer at execution time. Neither the `BurnNlp` nor the `MintNlp` transaction struct carries a user-specified minimum output amount. A user who signs a burn or mint transaction has no on-chain protection against receiving fewer tokens than expected if the oracle price moves unfavorably between signing and sequencer execution — a direct analog to `minOut: 0` in the GMX finding.

---

### Finding Description

**`mintNlp`** computes the NLP minted as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [1](#0-0) 

There is no `require(nlpAmount >= txn.minNlpAmount)` guard. The user spends a fixed `quoteAmount` and receives however many NLP tokens the sequencer-supplied price dictates.

**`burnNlp`** computes the quote returned as:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [2](#0-1) 

There is no `require(quoteAmount >= txn.minQuoteAmount)` guard. The user burns a fixed `nlpAmount` and receives however much quote the sequencer-supplied price dictates, minus the burn fee.

In both cases `oraclePriceX18` is **not** part of the user-signed struct — it is a separate argument injected by the sequencer at call time:

```solidity
function mintNlp(
    IEndpoint.MintNlp calldata txn,
    int128 oraclePriceX18,          // sequencer-supplied, not user-signed
    IEndpoint.NlpPool[] calldata nlpPools,
    int128[] calldata nlpPoolRebalanceX18
) external onlyEndpoint {
``` [3](#0-2) 

```solidity
function burnNlp(
    IEndpoint.BurnNlp calldata txn,
    int128 oraclePriceX18,          // sequencer-supplied, not user-signed
    IEndpoint.NlpPool[] calldata nlpPools,
    int128[] calldata nlpPoolRebalanceX18
) external onlyEndpoint {
``` [4](#0-3) 

The oracle price for the NLP product is also written inside these same calls (`spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18)`), meaning the sequencer sets the price and executes the conversion atomically in the same transaction batch, with no user-side floor. [5](#0-4) 

---

### Impact Explanation

A user burning NLP tokens may receive materially fewer quote tokens than expected if the oracle price drops between signing and execution. A user minting NLP may receive materially fewer NLP tokens if the oracle price rises. In both cases the user's signed transaction contains no on-chain bound on the output, so the loss is unbounded from the user's perspective. This is a direct asset loss to the user — the corrupted state is the user's quote or NLP balance in `SpotEngine`.

---

### Likelihood Explanation

Medium. Oracle prices change continuously with market conditions. Nado's off-chain sequencer model introduces an inherent delay between when a user signs a transaction and when the sequencer includes it in a batch. During that window the NLP oracle price can move. Even with an honest sequencer, a user who signed a `BurnNlp` at price $1.00 may have it executed at $0.80 with no recourse. The entry path is fully unprivileged: any user can submit a `MintNlp` or `BurnNlp` transaction through the normal `Endpoint` flow.

---

### Recommendation

Add a `minNlpAmount` field to the `MintNlp` struct and a `minQuoteAmount` field to the `BurnNlp` struct (both user-signed). Enforce them in `Clearinghouse.sol`:

```solidity
// mintNlp
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
require(nlpAmount >= txn.minNlpAmount, ERR_SLIPPAGE);

// burnNlp
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
require(quoteAmount >= txn.minQuoteAmount, ERR_SLIPPAGE);
```

This mirrors the fix recommended in the GMX report: add slippage protection to the output-side of the conversion, the same way it already exists for other user-facing operations.

---

### Proof of Concept

1. User signs a `BurnNlp` transaction with `nlpAmount = 100e18` when the NLP oracle price is $1.00; user expects ≈ $99.90 in quote (after 0.1 % burn fee).
2. Before the sequencer includes the transaction, the NLP oracle price drops to $0.70 due to market movement.
3. Sequencer calls `burnNlp(..., oraclePriceX18 = 0.70e18, ...)`.
4. `quoteAmount = 100e18 * 0.70 = 70e18`; after burn fee the user receives ≈ $69.93.
5. The user suffers a ~30 % loss relative to their expectation at signing time.
6. No field in the signed `BurnNlp` struct could have prevented this execution.

The root cause — absence of a user-controlled minimum output bound on an oracle-priced conversion — is identical to `minOut: 0` in the GMX `createDecreasePosition` / `closePosition` finding. [6](#0-5)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
