### Title
`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` silently downgrades `AllResources` to `L1Gas` when `l1_data_gas` and `l2_gas` are zero, producing a different transaction hash and wrong gas computation mode after P2P deserialization ŌĆö (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserialization of `InvokeTransactionV3` silently changes `ValidResourceBounds::AllResources` to `ValidResourceBounds::L1Gas` whenever `l1_data_gas` and `l2_gas` are both zero. Because `get_tip_resource_bounds_hash` hashes **3 resource felts** for `AllResources` but only **2 resource felts** for `L1Gas`, the transaction hash computed at submission time diverges from the hash that any component recomputes after a protobuf round-trip. The same type change also switches the gas computation mode from `GasVectorComputationMode::All` to `GasVectorComputationMode::NoL2Gas`, altering fee accounting during re-execution on syncing nodes.

---

### Impact Explanation

**Wrong state / receipt / fee from blockifier execution logic (Critical):**
A syncing node re-executes the deserialized transaction under `GasVectorComputationMode::NoL2Gas`. The sequencer executed it under `GasVectorComputationMode::All`. Even with zero l2_gas and l1_data_gas *bounds*, the actual execution may consume non-zero L2 gas. Under `All` mode the fee includes that L2 gas cost; under `NoL2Gas` mode it is folded into an L1-equivalent, producing a different charged fee and a different post-execution account balance. The syncing node's state root diverges from the sequencer's.

**Wrong hash / transaction conversion binds wrong payload (High):**
Any component that recomputes the transaction hash from the deserialized `InvokeTransactionV3` (e.g., block-hash verification, receipt generation, or the OS re-execution path) will compute HŌéé instead of HŌéü. The Cairo OS `hash_fee_fields` function unconditionally asserts `n_resource_bounds = 3`; feeding it a `L1Gas`-typed transaction (2 bounds) would cause the OS to reject or mis-hash the transaction. [7](#0-6) 

---

### Likelihood Explanation

Any V3 invoke transaction submitted with `AllResources` where `l2_gas = 0` and `l1_data_gas = 0` triggers the downgrade. This is a realistic and common configuration: wallets that do not yet use L2 gas or L1 data gas will submit exactly this pattern. The condition is unprivileged and requires no special access ŌĆö any user submitting a standard V3 invoke transaction can trigger it. The P2P sync path is exercised by every non-sequencer node in the network.

---

### Recommendation

Remove the heuristic type-downgrade from the protobuf deserializer. The `ValidResourceBounds` variant must be preserved across serialization boundaries. One approach: add an explicit discriminant field to the protobuf `ResourceBounds` message (e.g., `bool is_all_resources`) so the deserializer can reconstruct the exact variant without inspecting field values. Alternatively, always deserialize as `AllResources` for V3 transactions (matching the invariant enforced by `RpcInvokeTransactionV3`, `InternalRpcInvokeTransactionV3`, and the Cairo OS), and reserve `L1Gas` only for explicitly versioned pre-0.13.3 transaction types.

The same fix must be applied to `From<ResourceBoundsMapping> for ValidResourceBounds` in `crates/apollo_rpc/src/v0_8/transaction.rs`. [8](#0-7) [6](#0-5)

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
