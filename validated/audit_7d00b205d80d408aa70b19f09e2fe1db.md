### Title
Missing Access Control on `run_mainchain_gc` Allows Any Caller to Prematurely Prune Mainchain Headers — (`contract/src/lib.rs`)

### Summary

`run_mainchain_gc` is a state-mutating public function that permanently deletes mainchain block headers from contract storage. It carries only a `#[pause]` decorator, which restricts callers only when the contract is **paused**. When the contract is running normally, any unprivileged NEAR account can call it with an arbitrary `batch_size`, triggering immediate deletion of all GC-eligible mainchain headers. The analogous protected path — `submit_blocks` — is gated behind `#[trusted_relayer]`. `run_mainchain_gc` has no equivalent caller restriction.

### Finding Description

`run_mainchain_gc` is declared as:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [1](#0-0) 

The `#[pause(except(roles(...)))]` macro only controls access when the contract is in a paused state — the `except(roles(...))` clause allows `UnrestrictedRunGC` holders to bypass the pause. When the contract is **not paused**, the decorator imposes no caller check at all. Any NEAR account can invoke this function.

By contrast, `submit_blocks` — which also calls `run_mainchain_gc` internally — is protected by `#[trusted_relayer]`, requiring the caller to be a registered, active relayer:

```rust
#[payable]
#[pause]
#[trusted_relayer]
pub fn submit_blocks(...)
``` [2](#0-1) 

Inside `run_mainchain_gc`, the function permanently removes headers from three maps and advances `mainchain_initial_blockhash`:

```rust
self.remove_block_header(blockhash);
self.mainchain_height_to_header.remove(&height);
// ...
self.mainchain_initial_blockhash = self.mainchain_height_to_header.get(&end_removal_height)...;
``` [3](#0-2) 

`remove_block_header` deletes the entry from both `mainchain_header_to_height` and `headers_pool`:

```rust
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [4](#0-3) 

### Impact Explanation

The concrete corrupted invariant is the **fork-choice mechanism**. When `reorg_chain` is triggered by a fork with higher chainwork, it walks backward through the fork chain in `headers_pool` until it finds a block whose hash is present in `mainchain_header_to_height` (the common ancestor):

```rust
while !self.mainchain_header_to_height.contains_key(&fork_header_cursor.block_hash) {
    // ...
    fork_header_cursor = self.headers_pool.get(&prev_block_hash)
        .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
}
``` [5](#0-4) 

If an attacker has called `run_mainchain_gc` to remove the common ancestor, that block is gone from both `mainchain_header_to_height` and `headers_pool`. The termination condition is never satisfied, the cursor walks past the common ancestor, and the next `headers_pool.get` panics. The reorg transaction aborts, the fork is never promoted, and the contract remains permanently stuck on the weaker chain — a fork-choice corruption. The project documentation explicitly acknowledges this failure mode:

> "If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound` when it cannot walk the chain back to the common ancestor." [6](#0-5) 

A secondary impact: `verify_transaction_inclusion` checks `mainchain_header_to_height` to confirm a block belongs to the main chain. Blocks removed by premature GC will cause this check to panic, invalidating SPV proofs for transactions in those blocks. [7](#0-6) 

### Likelihood Explanation

The entry path requires no privileges — any NEAR account can call `run_mainchain_gc`. The only precondition is that `amount_of_headers_we_store > gc_threshold`, meaning the chain has grown beyond the configured threshold. Once that condition holds, the attacker can call the function with `batch_size = u64::MAX` to remove all GC-eligible headers in a single transaction. For chains with faster block times (Dogecoin: ~1 min/block, so `gc_threshold=52704` ≈ 36 days) or deployments with smaller thresholds, this condition is routinely satisfied during normal operation.

### Recommendation

Add a role-based caller restriction to `run_mainchain_gc` equivalent to what `submit_blocks` uses. The simplest fix is to add an ACL check at the top of the function body, or introduce a dedicated `GCManager` role and gate the function behind it:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    require!(
        self.acl_has_role(Role::RelayerManager, &env::predecessor_account_id())
            || self.acl_has_role(Role::DAO, &env::predecessor_account_id()),
        "Caller is not authorized to run GC"
    );
    // ...
}
```

Alternatively, mark the function `#[private]` and expose GC only through `submit_blocks`, which is already trusted-relayer-gated.

### Proof of Concept

1. Deploy the contract with `gc_threshold = 100`.
2. Have the trusted relayer submit 200 mainchain blocks. The chain now has 200 headers stored, 100 of which are GC-eligible.
3. Separately, have a relayer submit a fork block at height 50 (branching from a block that is now GC-eligible). The fork has higher chainwork than the main chain.
4. Before the relayer submits the fork tip, an attacker calls:
   ```
   run_mainchain_gc(batch_size: 18446744073709551615)
   ```
   from any unprivileged NEAR account. This removes all 100 GC-eligible headers, including the common ancestor at height 50.
5. The relayer now submits the fork tip. `submit_block_header_inner` detects higher chainwork and calls `reorg_chain`. The reorg loop walks back through the fork chain, fails to find the common ancestor in `mainchain_header_to_height` (it was deleted), and panics.
6. The contract remains on the weaker main chain. The heavier fork is never promoted. All downstream SPV consumers receive incorrect canonical-chain answers. [8](#0-7)
