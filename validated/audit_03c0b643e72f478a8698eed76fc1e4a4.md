### Title
`FeeEstimation.overall_fee` silently includes L2 gas cost not reflected in any advertised consumed-gas field — (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`tx_execution_output_to_fee_estimation` populates `overall_fee` from `receipt.fee`, which is computed by `GasVector::cost` and includes L2 gas cost (`l2_gas * l2_gas_price`). However, the `FeeEstimation` struct exposes no `l2_gas_consumed` field, and the documented invariant — both in the Rust comment and in the RPC OpenAPI spec — states `overall_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`. For any V3 `InvokeTransaction` with `AllResources` bounds that consumes nonzero L2 gas, this invariant is concretely violated.

### Finding Description

`tx_execution_output_to_fee_estimation` sets:

```rust
gas_consumed:      gas_vector.l1_gas.0.into(),
data_gas_consumed: gas_vector.l1_data_gas.0.into(),
overall_fee:       tx_execution_output.execution_info.receipt.fee,
``` [1](#0-0) 

`receipt.fee` is produced by `GasVector::cost`, which sums **all three** gas components:

```rust
for (gas, price, resource) in [
    (self.l1_gas,      gas_prices.l1_gas_price,      Resource::L1Gas),
    (self.l1_data_gas, gas_prices.l1_data_gas_price,  Resource::L1DataGas),
    (self.l2_gas,      tipped_l2_gas_price,            Resource::L2Gas),
] { ... }
``` [2](#0-1) 

So `overall_fee = l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * (l2_gas_price + tip)`.

The `FeeEstimation` struct has no `l2_gas_consumed` field. The comment at line 108 and the RPC OpenAPI spec both document the invariant as only covering L1 and data gas: [3](#0-2) [4](#0-3) 

The developers are aware of the gap — a TODO comment explicitly notes `l2_gas_consumed` is missing and that the approximation may not hold: [5](#0-4) 

### Impact Explanation

Any caller of `starknet_estimateFee` or `starknet_simulateTransactions` on a V3 transaction with `AllResources` bounds that consumes nonzero L2 gas receives a response where:

- `overall_fee` is the correct total fee (includes L2 gas cost)
- `gas_consumed` and `data_gas_consumed` are correct for their respective components
- But `overall_fee ≠ gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`

The divergence is exactly `l2_gas * (l2_gas_price + tip)` — a concrete, nonzero, protocol-visible value. A client that reconstructs the fee from the advertised fields (as the spec says it can) will compute a lower value than `overall_fee`. The response is authoritative-looking but internally inconsistent.

This fits the **High** impact category: *RPC fee estimation returns an authoritative-looking wrong value.*

### Likelihood Explanation

V3 transactions with `AllResources` resource bounds are the current standard transaction format for STRK-fee transactions. Any such transaction that executes Sierra code (i.e., virtually all modern contract calls) will consume nonzero L2 gas under `GasVectorComputationMode::All`. The path is fully unprivileged.

### Recommendation

Add an `l2_gas_consumed: Felt` field to `FeeEstimation` and populate it from `gas_vector.l2_gas.0` in `tx_execution_output_to_fee_estimation`. Update the documented invariant to `overall_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price + l2_gas_consumed * l2_gas_price`. Align the RPC OpenAPI `FEE_ESTIMATE` schema accordingly.

### Proof of Concept

The divergence is directly observable from the existing `GasVector::cost` regression test, which shows that with `l2_gas = 30` and `l2_gas_price = 5`, the fee includes `5 * 30 = 150` that is not captured by `l1_gas` or `l1_data_gas` fields: [6](#0-5) 

A concrete RPC-level demonstration: submit a V3 `InvokeTransaction` with `AllResources` bounds to `starknet_estimateFee`. Assert `overall_fee == gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`. The assertion will fail by exactly `l2_gas * l2_gas_price` whenever the transaction executes Sierra code.
