### Title
Authz `GenericAuthorization` for `MsgTransfer` allows grantee to redirect granter's NFT to a grantee-controlled receiver on the destination chain — (`x/nft-transfer/keeper/packet.go`, `x/nft-transfer/types/msgs.go`)

### Summary
An authz grantee holding a `GenericAuthorization` for `MsgTransfer` can submit a transaction where `msg.Sender` is the granter (NFT owner) and `msg.Receiver` is any address the grantee controls. Because `GetSigners` returns `msg.Sender`, authz authorizes the tx. Because `createOutgoingPacket` checks `sender.Equals(owner)` using `msg.Sender`, the ownership check passes. The IBC packet is sent with the grantee's address as receiver, and the destination chain delivers the NFT to the grantee.

### Finding Description
`MsgTransfer.GetSigners` returns `msg.Sender`, which is the field the authz framework uses to match against the granter. This means any holder of a `GenericAuthorization` for `MsgTransfer` from a granter can craft a `MsgTransfer` with:
- `Sender` = granter's address (NFT owner)
- `Receiver` = any address the grantee controls

The keeper's `createOutgoingPacket` validates ownership by comparing `sender` (derived from `msg.Sender`, i.e., the granter) against the NFT's on-chain owner. This check passes legitimately. The `receiver` field is never validated against the signer or the granter — it flows directly into the IBC packet data and is used verbatim by `processReceivedPacket` on the destination chain to deliver the NFT.

There is no `TransferAuthorization` (analogous to ibc-go's fungible token fix) that would restrict which receiver addresses are permitted under an authz grant.

### Impact Explanation
A grantee can permanently redirect a granter's NFT to themselves or any third party on the destination chain. The granter loses the NFT with no recourse: the escrow on the source chain holds the NFT under the module account, and the destination chain mints/unescrows it to the grantee's address. This is a direct, irreversible theft of an NFT asset.

### Likelihood Explanation
Any user who has ever granted a `GenericAuthorization` for `MsgTransfer` (e.g., to allow a relayer or automation script to initiate transfers on their behalf) is exposed. The attack requires no special privileges beyond holding such a grant. It is executable in a single transaction.

### Recommendation
Implement a custom `TransferAuthorization` type (mirroring ibc-go's `TransferAuthorization` for ICS-20) that:
- Restricts which `(sourcePort, sourceChannel, receiver)` combinations are permitted under a grant.
- Is the only accepted authorization type for `MsgTransfer` (reject `GenericAuthorization` for this message type).

Register it in `codec.go` and enforce it in the authz accept logic so that a grantee cannot freely set `msg.Receiver`.

### Proof of Concept
```
1. Granter owns NFT (classID="myClass", tokenID="token1") on chain A.
2. Granter submits MsgGrant{
       Granter: granterAddr,
       Grantee: granteeAddr,
       Grant: GenericAuthorization{MsgTypeURL: "/chainmain.nft.v1.MsgTransfer"},
   }
3. Grantee submits (via authz MsgExec) MsgTransfer{
       Sender:    granterAddr,   // granter = NFT owner → passes ownership check
       Receiver:  granteeAddr,   // grantee-controlled address on chain B
       ClassId:   "myClass",
       TokenIds:  ["token1"],
       SourcePort/Channel: valid IBC channel,
   }
4. createOutgoingPacket: sender == owner → NFT escrowed to module account.
5. IBC packet sent with Receiver = granteeAddr.
6. Chain B: processReceivedPacket mints/unescrows NFT to granteeAddr.
7. Granter's NFT is now owned by grantee on chain B.
```
