### Title
`SwapAllowlistExtension` Checks Router Address Instead of End-User, Allowing Any User to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

The `SwapAllowlistExtension` gates swaps by checking the `sender` argument passed from the pool's `beforeSwap` hook. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router address, not the original user. This means the allowlist either blocks all router-mediated swaps for legitimately allowlisted users, or — if the pool admin adds the router to the allowlist to fix that — any unprivileged user can bypass the restriction entirely by routing through the router.

---

### Finding Description

In `MetricOmmPool.swap()`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the immediate caller of `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly with no mechanism to forward the original user's address: [4](#0-3) 

The pool therefore passes the **router address** as `sender` to the extension. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

This creates an irresolvable dilemma for the pool admin:

- **Router not allowlisted**: All allowlisted users are blocked from using the router. Core swap functionality is broken for the standard periphery path.
- **Router allowlisted** (to restore router access for legitimate users): The check becomes `allowedSwapper[pool][router] == true`, which passes for **every caller** regardless of their individual allowlist status. Any unprivileged user can bypass the restriction by calling `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` on the router.

The `SwapAllowlistExtension` documentation states it "Gates `swap` by swapper address, per pool," but the implementation gates the immediate caller of `pool.swap()`, not the economic actor initiating the trade.

---

### Impact Explanation

A pool deployer using `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC-verified market makers, whitelisted counterparties, or protocol-controlled addresses) loses that protection entirely once the router is allowlisted. Any user can execute swaps against the restricted pool, extracting value from LPs at prices or in volumes the pool was not designed to expose to the general public. This constitutes a direct loss of LP principal and a broken core pool invariant (allowlist-enforced access control).

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the standard, documented periphery swap entrypoint. Any user who discovers the bypass needs only to call the router with a valid swap path. No privileged access, special tokens, or unusual setup is required. The bypass is reachable on every pool that has `SwapAllowlistExtension` configured and the router allowlisted.

---

### Recommendation

The `sender` forwarded to extension hooks must represent the **economic actor** (the original EOA or contract initiating the trade), not the immediate caller of `pool.swap()`. Two complementary fixes:

1. **Router-level**: Have the router encode the original `msg.sender` into `extensionData` for each hop, so extensions can decode and check the true initiator.
2. **Extension-level**: Update `SwapAllowlistExtension.beforeSwap` to decode and verify the original user from `extensionData` when the immediate `sender` is a known router, or require that the pool is only accessed directly (not via router) when the allowlist is active.

The cleanest long-term fix is for the pool to accept an explicit `originator` field in the swap call that the router populates with `msg.sender`, and for the extension interface to expose this field rather than relying on the immediate `sender`.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists only `alice`
  - Pool admin also allowlists `router` (required for alice to use the router)

Attack:
  - `bob` (not allowlisted) calls:
      router.exactInputSingle({
          pool: restrictedPool,
          tokenIn: token0,
          ...
      })
  - Pool calls _beforeSwap(msg.sender=router, ...)
  - Extension checks: allowedSwapper[pool][router] == true  ✓
  - Bob's swap executes successfully despite not being allowlisted

Result:
  - Bob trades in a pool restricted to alice only
  - If the pool offers favorable pricing or thin liquidity reserved for alice,
    bob extracts value from LPs
