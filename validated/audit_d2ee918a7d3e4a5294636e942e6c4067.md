### Title
Phantom Duplicate-Index Accepted in Odd-Width Merkle Trees Due to Missing `tx_index` Bounds Check — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`, `contract/src/lib.rs::verify_transaction_inclusion_v2`)

---

### Summary

`verify_transaction_inclusion_v2` returns `true` for a phantom transaction position that does not correspond to any real transaction in the block. In any Bitcoin block whose transaction count is odd at any Merkle level, the last real transaction at index `N-1` and a ghost position at index `N` produce identical Merkle roots when the same sibling hash (the duplicate of the last leaf) is supplied. Neither the contract nor the proof library checks that `tx_index` is within the actual transaction count of the block, so the ghost position passes all guards.

---

### Finding Description

**Root cause — `compute_root_from_merkle_proof` has no position-bounds check** [1](#0-0) 

The function accepts any `transaction_position` value and blindly uses its parity at each level to decide left/right placement. It never validates that the position is within the range of real leaves.

**How Bitcoin's odd-width duplication creates the ambiguity**

When a Merkle level has an odd number of nodes, the last node is duplicated: [2](#0-1) 

For a 3-transaction block `[A, B, C]`:
- Level 0 (padded): `[A, B, C, C]`
- Level 1: `[hash(A,B), hash(C,C)]`
- Root: `hash(hash(A,B), hash(C,C))`

**Proof trace for real index 2 (even) vs. phantom index 3 (odd), same inputs:**

| Step | Real: `tx_id=C, idx=2` | Phantom: `tx_id=C, idx=3` |
|------|------------------------|---------------------------|
| 1 | pos=2 (even) → `hash(C, C)`, pos→1 | pos=3 (odd) → `hash(C, C)`, pos→1 |
| 2 | pos=1 (odd) → `hash(hash(A,B), hash(C,C))`, pos→0 | pos=1 (odd) → `hash(hash(A,B), hash(C,C))`, pos→0 |
| Result | **= root ✓** | **= root ✓** |

Both calls with `merkle_proof = [C, hash(A,B)]` produce the same root. The phantom index 3 passes the Merkle check.

**`verify_transaction_inclusion` has no tx_index bounds guard** [3](#0-2) 

The only check is that the proof is non-empty and that the computed root matches the stored `merkle_root`. There is no check that `tx_index < block_tx_count`. The contract does not store transaction counts per block — only the header (which contains only the Merkle root).

**The coinbase proof in v2 does not close this gap** [4](#0-3) 

The coinbase check verifies that `coinbase_tx_id` at index 0 is in the same tree. It is a separate, independent check. It places no constraint on the `tx_index` of the claimed transaction, so the phantom index still passes.

---

### Impact Explanation

The light client returns `true` for a position that does not correspond to any real transaction in the block. Any downstream bridge, unlock, mint, or withdrawal contract that:

1. Calls `verify_transaction_inclusion_v2` to gate an action, **and**
2. Uses `(tx_id, tx_index, block_hash)` as its replay-prevention key (rather than just `tx_id`)

can be triggered twice for the same underlying Bitcoin transaction — once at the real index `N-1` and once at the phantom index `N`. This constitutes a light-client verification bypass enabling economic replay.

Even for applications that key only on `tx_id`, the contract is still making a false attestation: it asserts that a transaction exists at a position that is outside the real transaction list of the block.

---

### Likelihood Explanation

- The majority of Bitcoin blocks have an odd transaction count at at least one Merkle level, making the precondition trivially satisfied for almost any real block.
- The attacker needs no special privileges: `verify_transaction_inclusion_v2` is a public, unpermissioned view/call.
- The attacker only needs to know the last transaction in a block (publicly available on-chain) and supply `tx_index = actual_count` with the same proof as the real last transaction.
- No relayer compromise, key leak, or social engineering is required.

---

### Recommendation

Add a `tx_index` upper-bound check. Since the contract does not store per-block transaction counts, the bound can be derived from the proof length: a proof of depth `d` can only address indices `0 .. 2^d - 1`, but the tighter constraint is that the index must be strictly less than the number of real leaves. The practical fix is one of:

1. **Store the transaction count** in the `ExtendedHeader` when a block is submitted, and reject any `tx_index >= tx_count` in `verify_transaction_inclusion`.
2. **Reject equal-sibling proofs at the last level**: if `merkle_proof[0] == tx_id` and `tx_index` is odd, the caller is claiming the phantom duplicate slot — reject it.
3. **Require `tx_index` to be even when the sibling equals `tx_id`** at any level, since an odd position with a self-equal sibling is always a phantom duplicate.

### Proof of Concept

```rust
// Block with 3 transactions: [coinbase=A, B, C]
// Real Merkle tree (padded): [A, B, C, C]
// Root = hash(hash(A,B), hash(C,C))

// Legitimate call — index 2 (real position of C)
verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id:                 C,
    tx_block_blockhash:    block_hash,
    tx_index:              2,                        // real last tx
    merkle_proof:          vec![C, hash(A,B)],       // sibling=C (duplicate), then hash(A,B)
    coinbase_tx_id:        A,
    coinbase_merkle_proof: vec![B, hash(C,C)],
    confirmations:         0,
}) // → true  (correct)

// Phantom call — index 3 (does not exist in the block)
verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id:                 C,
    tx_block_blockhash:    block_hash,
    tx_index:              3,                        // phantom position
    merkle_proof:          vec![C, hash(A,B)],       // identical proof
    coinbase_tx_id:        A,
    coinbase_merkle_proof: vec![B, hash(C,C)],
    confirmations:         0,
}) // → true  (WRONG — position 3 is not a real transaction)
```

Step-by-step for the phantom call through `compute_root_from_merkle_proof(C, 3, [C, hash(A,B)])`:
- Iteration 1: `pos=3` (odd) → `hash(C, C)`, `pos→1`
- Iteration 2: `pos=1` (odd) → `hash(hash(A,B), hash(C,C))`, `pos→0`
- Returns the real Merkle root → equality check passes → function returns `true`. [5](#0-4) [6](#0-5)

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
