### Title
Base Rewards Top-Up Drains Shared Pool Without Accounting for Pending Bonus Obligations, Causing Permanent Bonus Loss on Redelegation Slash — (File: `x/tieredrewards/keeper/abci.go`)

---

### Summary

The `topUpBaseRewards` BeginBlocker drains the `RewardsPoolName` module account every block to subsidize base staking rewards. Its shortfall calculation does not account for pending (accrued but unclaimed) bonus rewards obligations. When the pool is drained to zero by `topUpBaseRewards` and a redelegation slash subsequently fires, `slashRedelegationPosition` silently and permanently forfeits the position owner's accrued bonus rewards — with no retry path.

---

### Finding Description

**Root cause — `topUpBaseRewards` ignores pending bonus obligations:**

In `x/tieredrewards/keeper/abci.go`, the shortfall is computed as:

```go
targetStakersRewardPerBlock := math.LegacyNewDecFromInt(totalBonded).
    Mul(targetBaseRewardsRate).
    Quo(math.LegacyNewDec(int64(blocksPerYear)))

defaultStakersRewardPerBlock := math.LegacyNewDecFromInt(feeCollectorBalance.Amount).
    MulTruncate(math.LegacyOneDec().Sub(communityTax))

shortFallAmount := targetStakersRewardPerBlock.Sub(defaultStakersRewardPerBlock).TruncateInt()
``` [1](#0-0) 

The calculation considers only the fee-collector balance vs. the target base rewards rate. It does not check or reserve the total pending bonus rewards obligations (accrued but unclaimed bonus for all active positions). When the pool balance is less than the shortfall, the function drains the entire pool:

```go
if poolBalance.Amount.LT(shortFallAmount) {
    topUpAmount = poolBalance.Amount  // drain everything
}
``` [2](#0-1) 

The same `RewardsPoolName` module account is the source for both the base-rewards top-up and bonus rewards payments:

- Top-up: `SendCoinsFromModuleToModule(ctx, types.RewardsPoolName, distributiontypes.ModuleName, ...)`
- Bonus: `SendCoinsFromModule(ctx, types.RewardsPoolName, ownerAddr, bonus)` [3](#0-2) [4](#0-3) 

**Permanent loss path — `slashRedelegationPosition` silently forfeits bonus:**

When a redelegation slash fires (`BeforeRedelegationSlashed` hook → `slashRedelegationPosition`), the hook calls `processEventsAndClaimBonus`. If the pool is empty (drained by `topUpBaseRewards`), the error is swallowed to avoid a chain halt:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // bonus forfeited — no error returned, no retry
    } else {
        return err
    }
}
``` [5](#0-4) 

This is a **permanent** forfeiture. Unlike all user-driven paths (`ClaimTierRewards`, `TierRedelegate`, `AddToPosition`, `ClearPosition`) which fail atomically and allow the user to retry after the pool is replenished, the `BeforeRedelegationSlashed` hook cannot return an error. The accrued bonus is gone. [6](#0-5) 

---

### Impact Explanation

The corrupted value is the position owner's accrued bonus rewards (denominated in the bond denom, drawn from `RewardsPoolName`). These rewards are permanently forfeited — not deferred, not retryable — because the hook path has no failure mode. The pool balance invariant "Pool balance is never exceeded" is upheld, but the complementary invariant that accrued bonus rewards are always paid is broken when the pool is drained by the BeginBlocker before a redelegation slash. [7](#0-6) 

---

### Likelihood Explanation

- `topUpBaseRewards` runs unconditionally every block whenever `TargetBaseRewardsRate > 0` and there is a shortfall. It drains the entire pool if the pool balance is less than the shortfall.
- Redelegation slashes occur when a validator double-signs while a tier position has an active redelegation entry (i.e., within the staking redelegation completion window, typically 21 days).
- If the pool is funded at the bare minimum — just enough to cover pending bonus rewards but not the per-block base-rewards shortfall — the BeginBlocker will drain it on the very next block, and any subsequent redelegation slash will permanently forfeit the position owner's bonus.
- This is a realistic production scenario: pool funders cannot know the exact per-block drain rate vs. pending bonus obligations without off-chain tooling, and the protocol provides no on-chain reservation mechanism.

---

### Recommendation

`topUpBaseRewards` should compute the total pending bonus rewards obligations across all active positions and subtract that reserved amount from the available pool balance before computing `topUpAmount`. Alternatively, separate the pool into two distinct module accounts: one for base-rewards top-up and one for bonus rewards, so the BeginBlocker cannot inadvertently drain the bonus pool.

---

### Proof of Concept

1. Create a tier position with a large accrued bonus (advance time by several months without claiming).
2. Initiate a `MsgTierRedelegate` to create an active redelegation entry (this registers a `RedelegationMapping`).
3. Fund the rewards pool with exactly the pending bonus amount — no excess.
4. Wait one block: `topUpBaseRewards` fires, sees a shortfall larger than the pool balance, and drains the entire pool (`topUpAmount = poolBalance.Amount`).
5. Trigger a double-sign slash on the destination validator while the redelegation is still pending.
6. `BeforeRedelegationSlashed` → `slashRedelegationPosition` → `processEventsAndClaimBonus` → `ErrInsufficientBonusPool` → error swallowed.
7. Observe: the position owner's accrued bonus rewards are permanently forfeited; the pool is empty; no retry is possible. [8](#0-7) [5](#0-4)

## 9. Security Invariants

- Only governance can manage tiers and params.
- Rewards accrue only when delegated to a bonded validator.
- No undelegation until exit commitment has elapsed.
- `MsgClearPosition` settles rewards before clearing exit to prevent bonus manipulation past `ExitUnlockAt`.
- Pool balance is never exceeded. User paths fail atomically; the `BeforeRedelegationSlashed` hook forfeits bonus silently if the pool is short, to avoid chain halt.
- Base-reward routing: `SetWithdrawAddr(posDelAddr, owner)` is installed at position creation and cleared on deletion, so every base-reward withdrawal lands on the owner.

- CloseOnly tiers block new positions while allowing exits.
- Bonus cannot be double-claimed: `LastEventSeq` prevents event replay, `LastBonusAccrual` prevents segment replay, `LastKnownBonded` prevents unbonded gap overpay.
- All delegation paths reject non-bonded validators (delegate, redelegate, transferDelegationToPosition, transferDelegationFromPosition).
- Vesting accounts are not allowed to create a new tier position.
