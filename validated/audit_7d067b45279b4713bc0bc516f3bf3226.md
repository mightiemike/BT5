### Title
Interest Rate Config Update Without Prior State Settlement Causes Retroactive Rate Misapplication — (`core/contracts/SpotEngineState.sol`)

### Summary

`SpotEngineState._updateState()` reads `configs[productId]` at the moment `updateStates` is called by the sequencer. If the owner updates `configs[productId]` (changing interest curve parameters such as `interestFloorX18`, `interestSmallCapX18`, `interestLargeCapX18`, or `minDepositRateX18`) between two sequencer-driven `updateStates` calls, the new parameters are applied retroactively to the entire elapsed time period `dt`, including the portion that accrued under the old config. This corrupts `cumulativeBorrowsMultiplierX18` and `cumulativeDepositsMultiplierX18` for all depositors and borrowers of that product.

---

### Finding Description

`SpotEngineState._updateState()` computes the borrow rate multiplier and deposit rate multiplier for a given time delta `dt`:

```solidity
Config memory config = configs[productId];

int128 borrowerRateX18 = config.interestFloorX18;
// ... uses config.interestSmallCapX18, config.interestLargeCapX18,
//     config.interestInflectionUtilX18 to compute borrowerRateX18
borrowRateMultiplierX18 = (ONE + borrowerRateX18).pow(int128(dt));
``` [1](#0-0) 

These multipliers are then applied to the cumulative state:

```solidity
state.cumulativeBorrowsMultiplierX18 = state
    .cumulativeBorrowsMultiplierX18
    .mul(borrowRateMultiplierX18);

state.cumulativeDepositsMultiplierX18 = state
    .cumulativeDepositsMultiplierX18
    .mul(depositRateMultiplierX18);
``` [2](#0-1) 

`updateStates` is called by the Endpoint (sequencer) with a `dt` representing elapsed time. The owner can update `configs[productId]` independently via `updateConfig` (or equivalent admin function in `SpotEngine.sol`). There is no enforcement that `updateStates` must be called before a config change to settle interest accrued under the old parameters.

**Desynchronization scenario:**

1. At time T₀, `updateStates` is called. `cumulativeBorrowsMultiplierX18` is current.
2. At time T₁ (T₀ + Δt₁), the owner calls `updateConfig` and raises `interestFloorX18` significantly.
3. At time T₂ (T₁ + Δt₂), `updateStates` is called with `dt = T₂ − T₀`.
4. The entire period `dt` is computed using the **new** (higher) `interestFloorX18`, even though only `Δt₂` of that period was under the new config.
5. Borrowers are overcharged interest for the `Δt₁` period; depositors receive a windfall. The reverse holds if rates are lowered. [3](#0-2) 

---

### Impact Explanation

`cumulativeBorrowsMultiplierX18` and `cumulativeDepositsMultiplierX18` are the core accounting primitives for all spot product balances. Every user's effective balance is computed as `amountNormalized × cumulativeMultiplier`. A corrupted multiplier directly misstates every borrower's debt and every depositor's credit for the affected product, causing real asset misallocation between users and the protocol fee account. [4](#0-3) 

---

### Likelihood Explanation

Interest rate curve parameters are expected to be tuned over the protocol's lifetime as market conditions change. Each such tuning event, without a prior `updateStates` call, silently corrupts the multipliers. The sequencer and owner are separate actors with no on-chain coordination requirement, making the unsettled-state window a realistic operational occurrence.

---

### Recommendation

In the `updateConfig` function (or whichever function mutates `configs[productId]`), call `updateStates` for the affected product with the elapsed time before writing the new config:

```diff
function updateConfig(uint32 productId, Config memory newConfig) external onlyOwner {
+   // Settle interest accrued under the old config first
+   uint128 dt = uint128(block.timestamp - lastUpdateTimestamp[productId]);
+   if (dt > 0) {
+       State memory state = states[productId];
+       _updateState(productId, state, dt);
+       _setState(productId, state);
+       lastUpdateTimestamp[productId] = uint128(block.timestamp);
+   }
    configs[productId] = newConfig;
}
```

This mirrors the fix applied in BendDAO: settle the accumulated state under the old parameters before the new parameters take effect.

---

### Proof of Concept

1. Deploy SpotEngine with product `P` having `interestFloorX18 = 0.05e18` (5% floor).
2. Sequencer calls `updateStates(86400, [...])` — 1 day of interest accrues at 5% floor.
3. Owner calls `updateConfig(P, {interestFloorX18: 0.20e18, ...})` — raises floor to 20%.
4. Two days pass. Sequencer calls `updateStates(172800, [...])` — `dt = 2 days`.
5. `_updateState` computes `borrowerRateX18` using `interestFloorX18 = 0.20e18` for the full 2-day `dt`.
6. Borrowers are charged 20% annualized for 2 days, but only 1 day was under the 20% regime. The extra ~15% annualized × 1 day of interest is incorrectly extracted from borrowers and credited to depositors/fees. [5](#0-4)

    function _updateState(
        uint32 productId,
        State memory state,
        uint128 dt
    ) internal {
        int128 borrowRateMultiplierX18;
        int128 totalDeposits = state.totalDepositsNormalized.mul(
            state.cumulativeDepositsMultiplierX18
        );
        int128 totalBorrows = state.totalBorrowsNormalized.mul(
            state.cumulativeBorrowsMultiplierX18
        );
        int128 utilizationRatioX18 = totalBorrows.div(totalDeposits);
        int128 minDepositRateX18;
        {
            Config memory config = configs[productId];

            // annualized borrower rate
            int128 borrowerRateX18 = config.interestFloorX18;
            if (utilizationRatioX18 == 0) {
                // setting borrowerRateX18 to 0 here has the property that
                // adding a product at the beginning of time and not using it until time T
                // results in the same state as adding the product at time T
                borrowerRateX18 = 0;
            } else if (utilizationRatioX18 < config.interestInflectionUtilX18) {
                borrowerRateX18 += config
                    .interestSmallCapX18
                    .mul(utilizationRatioX18)
                    .div(config.interestInflectionUtilX18);
            } else {
                borrowerRateX18 +=
                    config.interestSmallCapX18 +
                    config.interestLargeCapX18.mul(
                        (
                            (utilizationRatioX18 -
                                config.interestInflectionUtilX18).div(
                                    ONE - config.interestInflectionUtilX18
                                )
                        )
                    );
            }

            // convert to per second
            borrowerRateX18 = borrowerRateX18.div(
                MathSD21x18.fromInt(31536000)
            );
            borrowRateMultiplierX18 = (ONE + borrowerRateX18).pow(int128(dt));
            minDepositRateX18 = config.minDepositRateX18;
        }
