### Title
Zero-Signature Fast Withdrawal Authorization Bypass When `nSigner == 0` — (`core/contracts/Verifier.sol`, `core/contracts/BaseWithdrawPool.sol`)

### Summary
`requireValidTxSignatures` enforces unanimity (`nSignatures == nSigner`) rather than a quorum. When `nSigner` is 0 — reachable during a key-rotation window via sequential `deletePubkey` calls — the check trivially passes with an empty signatures array, allowing any caller to drain `BaseWithdrawPool` via `submitFastWithdrawal` with zero authorization.

### Finding Description
`Verifier.requireValidTxSignatures` counts non-empty entries in the supplied `signatures` array and requires the count to equal `nSigner`: [1](#0-0) 

When `nSigner == 0` (all pubkeys deleted), `nSignatures` is also 0 for an empty array, so `0 == 0` passes unconditionally.

By contrast, `checkQuorum` correctly rejects a zero-signer state: [2](#0-1) 

`nSigned * 2 > nSigner` evaluates to `0 > 0 = false` for any empty bitmask, regardless of `nSigner`.

`nSigner` reaches 0 through the owner's `deletePubkey`, which is a normal key-rotation operation: [3](#0-2) 

`submitFastWithdrawal` is `public` and its only authorization gate is `requireValidTxSignatures`: [4](#0-3) 

### Impact Explanation
An attacker who observes `nSigner == 0` on-chain can call `submitFastWithdrawal` with an attacker-crafted `WithdrawCollateral` or `WithdrawCollateralV2` transaction (arbitrary `sendTo`, arbitrary `amount`) and an empty `signatures` array. The call passes all checks and triggers `handleWithdrawTransfer`, transferring pool tokens to the attacker. The pool can be fully drained in a single transaction per product. This is a direct, unauthorized asset transfer — a Critical solvency/balance impact. [5](#0-4) 

### Likelihood Explanation
Key rotation is a documented operational procedure. The window between the last `deletePubkey` and the first new `assignPubKey` is observable on-chain in real time. A bot watching for `DeletePubkey` events and a resulting `nSigner == 0` state can front-run or immediately follow the deletion transaction. No privileged access is required by the attacker; only the owner's routine maintenance creates the window.

### Recommendation
1. Add a `require(nSigner > 0, "no signers")` guard at the top of `requireValidTxSignatures`.
2. Replace the unanimity check with a quorum check consistent with `checkQuorum`: `require(nSignatures * 2 > nSigner, "insufficient signatures")`.
3. Consider an atomic key-rotation function that adds new keys before removing old ones, so `nSigner` never reaches 0.
