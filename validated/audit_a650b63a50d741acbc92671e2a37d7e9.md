Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Enabling Allowlist Bypass and Breaking Router Deposit Path — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and evaluates the allowlist against `owner`, a user-controlled address. This breaks the access-control guard in two directions: any unprivileged caller can bypass the allowlist by supplying an allowlisted address as `owner`, and the canonical `MetricOmmPoolLiquidityAdder` router becomes permanently unusable for pools that allowlist the router contract rather than individual users.

## Finding Description
`MetricOmmPool.addLiquidity` calls the extension hook as `_beforeAddLiquidity(msg.sender, owner, ...)`, forwarding the real caller as `sender` and the user-supplied position owner as `owner`. `ExtensionCalling._beforeAddLiquidity` encodes both and dispatches to the extension. However, `DepositAllowlistExtension.beforeAddLiquidity` declares its first parameter unnamed (`address,`) and checks only `allowedDepositor[msg.sender][owner]`, where `msg.sender` is the pool and `owner` is the attacker-controlled argument. The correct actor — `sender` — is never read. By contrast, `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender`. The asymmetry is structural and confirmed in production code.

**Bypass path:** Bob (not allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")` where `allowedDepositor[pool][alice] == true`. The extension checks `allowedDepositor[pool][alice]` → passes. The pool invokes Bob's `metricOmmModifyLiquidityCallback`; Bob pays the tokens and the position is credited to Alice. The pool admin's deposit restriction is fully circumvented.

**Router breakage path:** Pool admin sets `allowedDepositor[pool][LiquidityAdder] = true`. A user calls `LiquidityAdder.addLiquidityExactShares(pool, user, ...)`. The router calls `pool.addLiquidity(user, ...)` with `msg.sender = LiquidityAdder`. The extension checks `allowedDepositor[pool][user]` → false → `NotAllowedToDeposit()`. Every router-mediated deposit reverts despite the router being allowlisted.

## Impact Explanation
The router breakage path renders the canonical periphery liquidity-addition flow permanently unusable for any pool that allowlists the router contract — a natural and expected admin configuration. This is broken core pool functionality. The bypass path allows any unprivileged caller to deposit into a restricted pool by supplying a known allowlisted address as `owner`, defeating the pool admin's access control entirely.

## Likelihood Explanation
**Bypass:** Zero privilege required. The `allowedDepositor` mapping is public; any caller can read allowlisted addresses and immediately exploit the check. The attacker pays their own tokens, so no victim approval is needed.

**Router breakage:** Triggered automatically whenever a pool admin allowlists the router contract rather than enumerating every individual user — the natural usage pattern for a periphery router. No attacker action required; the misconfiguration is structural.

## Recommendation
Change `beforeAddLiquidity` to name and check `sender` (the first argument), discarding `owner`, exactly mirroring `SwapAllowlistExtension.beforeSwap`:

```solidity
// Correct:
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```
// Bypass:
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  bob is NOT on the allowlist

Attack:
  bob deploys a contract implementing metricOmmModifyLiquidityCallback that pays tokens
  bob calls pool.addLiquidity(alice, salt, deltas, callbackData, "")
  → _beforeAddLiquidity(bob, alice, ...) fires
  → extension checks allowedDepositor[pool][alice] == true → passes
  → pool calls bob's callback → bob pays tokens
  → position credited to alice; allowlist bypassed

// Router breakage:
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][LiquidityAdder] = true
  user is NOT individually listed

Attack (automatic):
  user calls LiquidityAdder.addLiquidityExactShares(pool, user, ...)
  → LiquidityAdder calls pool.addLiquidity(user, ...)
  → _beforeAddLiquidity(LiquidityAdder, user, ...) fires
  → extension checks allowedDepositor[pool][user] == false → NotAllowedToDeposit()
  → all router-mediated deposits revert permanently
```
