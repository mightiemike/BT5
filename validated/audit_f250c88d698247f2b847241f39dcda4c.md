### Title
Unprotected `creditDeposit()` Allows Anyone to Force-Deposit Tokens into Any Subaccount via DDA, Enabling Liquidation Griefing — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` is an `external` function with no access control that reads `token.balanceOf(address(this))` to determine how much to deposit into the associated subaccount. Because anyone can send tokens to a DDA address and then call `creditDeposit()`, an adversary (including the subaccount owner) can force-deposit collateral into a subaccount at will. This directly mirrors the Astaria pattern: a public settlement-adjacent function uses a raw `balanceOf` check instead of tracking authorized transfers, and can be triggered by any caller who first airdrops tokens to the contract.

---

### Finding Description

`DirectDepositV1.creditDeposit()` iterates over all supported product IDs, reads the DDA's token balance, and calls `endpoint.depositCollateralWithReferral` for any non-zero balance: [1](#0-0) 

The function is declared `external` with no `onlyOwner`, no `msg.sender` check, and no authorization guard of any kind. The deposit amount is determined entirely by `token.balanceOf(address(this))` — the live ERC-20 balance of the DDA — not by any tracked or authorized transfer amount. [1](#0-0) 

The DDA address for any subaccount is deterministic and publicly discoverable via `ContractOwner.directDepositV1Address[subaccount]`. [2](#0-1) 

---

### Impact Explanation

A subaccount owner whose account has fallen below the maintenance margin threshold (and is therefore liquidatable) can:

1. Send any amount of a supported collateral token directly to their own DDA address.
2. Call `creditDeposit()` (permissionlessly) to immediately credit that collateral to their subaccount.
3. The subaccount's health improves, causing any in-flight liquidation call to revert with `ERR_NOT_LIQUIDATABLE`. [3](#0-2) 

The liquidator loses gas and their liquidation fee opportunity. If the subaccount was deeply insolvent and the forced deposit is insufficient to fully cover the debt, the protocol's insurance fund may still be exposed — but the liquidation that would have socialized or covered the loss is now blocked. Repeated griefing can delay the protocol's ability to close out bad positions before they worsen.

---

### Likelihood Explanation

- The DDA address is publicly known and deterministic.
- `creditDeposit()` requires no special role or permission.
- The cost to the attacker is only the token amount needed to push the subaccount above the maintenance threshold — which for a marginally unhealthy account can be very small (dust).
- The subaccount owner has a direct economic incentive to execute this attack to avoid liquidation penalties.

---

### Recommendation

Add an access control guard to `creditDeposit()` so that only the subaccount owner (or an authorized operator) can trigger it:

```solidity
function creditDeposit() external {
    require(
        msg.sender == address(uint160(bytes20(subaccount))),
        "Unauthorized"
    );
    // ... existing logic
}
```

Alternatively, track authorized deposit amounts via an explicit `deposit(uint32 productId, uint128 amount)` pattern rather than relying on `balanceOf(address(this))`, so that unsolicited token transfers cannot be weaponized to trigger state changes.

---

### Proof of Concept

1. Subaccount `0xVICTIM000...` falls below maintenance margin; a liquidator prepares a `liquidateSubaccountImpl` call.
2. The subaccount owner identifies their DDA address via `ContractOwner.directDepositV1Address[subaccount]`.
3. The owner transfers a small amount of USDC (e.g., 100 USDC) directly to the DDA address.
4. The owner calls `DirectDepositV1(dda).creditDeposit()` — no permission required.
5. `creditDeposit()` reads `token.balanceOf(address(this)) = 100e6`, approves the Endpoint, and calls `endpoint.depositCollateralWithReferral(subaccount, productId, 100e6, "-1")`.
6. The subaccount's USDC balance increases; `isUnderMaintenance(subaccount)` now returns `false`.
7. The liquidator's `liquidateSubaccountImpl` reverts: `require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE)`. [1](#0-0) [4](#0-3)

    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }
