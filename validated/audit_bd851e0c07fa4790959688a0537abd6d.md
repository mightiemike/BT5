### Title
`SwapAllowlistExtension` checks the router address as the swapper identity, not the originating user — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `swap()` call. When users route through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the **router contract**, not the originating user. A pool admin who allowlists the router address to enable the standard UX path inadvertently opens the pool to every user, completely defeating the allowlist.

---

### Finding Description

**Actor binding in the extension**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist: [1](#0-0) 

`msg.sender` here is the pool (correct key for the mapping), and `sender` is the address the pool forwarded as the swapper identity.

**What the pool forwards as `sender`**

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [2](#0-1) 

`ExtensionCalling._beforeSwap` then encodes that value as the first argument to every extension: [3](#0-2) 

**What the router passes as `msg.sender` to the pool**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly. The pool's `msg.sender` is therefore the **router contract address**, not the originating user: [4](#0-3) 

The same holds for `exactInput` (all hops), `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**The broken invariant**

The pool documentation explicitly states that `msg.sender` pays but need not equal `owner`, and the allowlist is supposed to gate the economically relevant actor: [6](#0-5) 

When the router is the intermediary, the extension sees the router as the swapper. The allowlist therefore gates the router address, not the user.

---

### Impact Explanation

Two fund-impacting outcomes follow directly:

**Bypass (High):** A pool admin who wants to allow router-mediated swaps allowlists the router address. Because every user reaches the pool through the same router, `allowedSwapper[pool][router] == true` passes for every caller. The allowlist is completely bypassed: any address, including those the admin explicitly never allowlisted, can drain LP liquidity at oracle prices.

**Broken core flow (Medium):** A pool admin who allowlists specific user addresses (the intended design) makes those users unable to swap through the router, because the router address is not allowlisted. The only reachable path is a direct `pool.swap()` call, which is not the supported periphery UX and may not be accessible to end users. LP liquidity becomes effectively illiquid for the intended counterparties.

---

### Likelihood Explanation

The router is the canonical, documented swap entrypoint for the protocol. A pool admin configuring a curated pool will naturally test swaps through the router and, finding them blocked, will allowlist the router address to restore functionality — unknowingly opening the pool to all users. The trigger requires no privileged escalation beyond the pool admin's own configuration action, which is within the semi-trusted scope.

---

### Recommendation

The extension must check the **originating user**, not the intermediary. Two sound approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` (the originating user) into `extensionData` before calling `pool.swap`. The `SwapAllowlistExtension` decodes and checks that address. The pool admin documents that the extension requires this encoding.

2. **Check `recipient` instead of `sender`**: For single-hop swaps the recipient is often the originating user. This is fragile for multi-hop paths where intermediate recipients are the router itself.

The cleanest fix is option 1, combined with a note in the extension that direct pool calls (where `sender` equals the real user) remain correctly gated without any encoding.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, address(router), true)` to enable router-mediated swaps.
3. Unprivileged user `attacker` (never allowlisted) calls `router.exactInputSingle(...)` targeting the pool.
4. Inside `pool.swap`, `msg.sender == router`. `_beforeSwap` forwards `sender = router`.
5. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router] == true` → passes.
6. `attacker` completes the swap and receives output tokens from LP reserves, despite never being allowlisted. [1](#0-0) [7](#0-6) [8](#0-7)
