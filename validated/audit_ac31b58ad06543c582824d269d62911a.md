### Title
Fork Difficulty Calculation Uses Mainchain Boundary Block Instead of Fork Ancestor, Enabling Difficulty Manipulation - (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

When computing the expected difficulty for a block on a fork at a difficulty-adjustment boundary, `get_next_work_required` retrieves the boundary block's timestamp via `get_header_by_height`. That helper always reads from `mainchain_height_to_header`, returning the **mainchain** block at that height rather than the fork's actual ancestor. An attacker who controls the fork's boundary block timestamp can therefore cause the difficulty calculation to use a wrong time-span, producing an easier target than the protocol requires, and potentially trigger a chain reorganization that corrupts the canonical chain.

---

### Finding Description

At every difficulty-adjustment boundary, `get_next_work_required` must compute the elapsed time between the first and last block of the retarget window. It does so by calling `blocks_getter.get_header_by_height(height_first)`:

**`contract/src/dogecoin.rs` lines 286–297** (the TODO comment is the developers' own acknowledgement of the problem):

```rust
let height_first = prev_block_header
    .block_height
    .checked_sub(blocks_to_go_back)
    ...;

// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

The same pattern appears in `bitcoin.rs`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [2](#0-1) 

And in `litecoin.rs`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [3](#0-2) 

The implementation of `get_header_by_height` in `lib.rs` unconditionally queries `mainchain_height_to_header`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [4](#0-3) 

Fork blocks are stored only in `headers_pool` via `store_fork_header`, never in `mainchain_height_to_header`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [5](#0-4) 

**The desynchronization**: when `check_pow` → `get_next_work_required` is called for a fork block whose height falls on a retarget boundary, `get_header_by_height(height_first)` returns the **mainchain** block at `height_first`, not the fork's actual ancestor at that height. If the fork diverged before `height_first`, the fork's ancestor at that height is a different block with a potentially different `time` field. The time-span used in `calculate_next_work_required` is therefore computed from the wrong block, producing an incorrect difficulty target.

This is structurally identical to the M-09 class: one state field (`time` of the boundary block) is read from a different module's view (mainchain index) than the one being validated (fork chain), causing the comparison/calculation to operate on desynchronized state.

---

### Impact Explanation

An attacker who controls a fork can set the fork's boundary block (`height_first`) to a very early timestamp (subject only to the MTP constraint enforced at submission time). When the fork later reaches the next retarget height, `get_header_by_height` returns the mainchain's boundary block, which has a later timestamp. The computed time-span is therefore **longer** than the fork's actual time-span, and `calculate_next_work_required` produces a **lower (easier) difficulty target** than the protocol requires for the fork.

With an artificially easy target, the attacker can mine subsequent fork blocks with less PoW. If the fork's accumulated `chain_work` eventually exceeds the mainchain tip's `chain_work`, `submit_block_header_inner` triggers `reorg_chain`:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [6](#0-5) 

After the reorg, `mainchain_tip_blockhash` points to the attacker's fork. Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` verify against the canonical chain derived from `mainchain_tip_blockhash` and `mainchain_header_to_height`, so they will now confirm transactions in the attacker's fabricated chain and reject transactions in the legitimate chain. Any NEAR contract that relies on these proofs for cross-chain asset transfers or settlement is directly affected.

---

### Likelihood Explanation

- The entry point is the permissionless `submit_blocks` call (subject only to the `trusted_relayer` gate, which is bypassable via staking per the contract's own design).
- The attacker needs to mine a fork that diverges before a retarget boundary and accumulate chainwork exceeding the mainchain. The difficulty reduction from the desynchronization directly reduces the mining cost of this attack.
- The TODO comment in `dogecoin.rs` confirms the developers identified this concern but left it unresolved.
- All three chain variants (Bitcoin, Litecoin, Dogecoin) are affected by the same root cause.

---

### Recommendation

Replace `get_header_by_height` in `get_next_work_required` with an ancestor traversal that walks back through `get_prev_header` from the fork's tip until reaching `height_first`. This mirrors how `reorg_chain` traverses the fork chain and ensures the boundary block's timestamp is taken from the fork's actual ancestor, not the mainchain's block at that height.

---

### Proof of Concept

1. Mainchain has blocks at heights `H` (retarget boundary) through `H+N`, with the mainchain block at `H` having timestamp `T_main`.
2. Attacker submits a fork starting from height `H-1` (before the boundary). The fork's block at height `H` is given timestamp `T_fork` where `T_fork << T_main` (set as early as MTP allows).
3. Attacker extends the fork to height `H + difficulty_adjustment_interval` (the next retarget point).
4. `check_pow` → `get_next_work_required` is called for the fork block at `H + difficulty_adjustment_interval`. It calls `get_header_by_height(H)`, which returns the **mainchain** block with timestamp `T_main`.
5. The time-span is computed as `T_fork_tip - T_main` (using the mainchain boundary), which is longer than the fork's actual span `T_fork_tip - T_fork`.
6. `calculate_next_work_required` produces a lower difficulty target than the fork's actual elapsed time warrants.
7. The attacker mines subsequent fork blocks against this easier target, accumulates chainwork, and triggers `reorg_chain`, corrupting `mainchain_tip_blockhash` and all downstream transaction inclusion proofs.
