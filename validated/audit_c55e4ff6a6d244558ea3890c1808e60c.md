### Title
SwapAllowlistExtension Checks the Router Address Instead of the End User, Allowing Any User to Bypass the Swap Allowlist on Curated Pools - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router address, not the end user. If the pool admin allowlists the router to enable router-based swaps for legitimate users, every user—including non-allowlisted ones—can bypass the allowlist by routing through the public router.

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to every configured extension: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that `sender` verbatim into the hook call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called the pool: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly, making the pool see `msg.sender = router`: [4](#0-3) 

The original end user's address is stored only in transient callback context for payment settlement and is never forwarded to the pool as the swap initiator: [5](#0-4) 

**Bypass path:**
1. Pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict swaps to KYC'd addresses.
2. Pool admin allowlists specific user addresses and also allowlists the router so those users can swap via the standard periphery path.
3. A non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle()`.
4. The router calls `pool.swap(...)` — pool sees `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. The non-allowlisted user successfully swaps on a pool that was supposed to block them.

The pool admin has no way to simultaneously allow router-based swaps for legitimate users and block non-allowlisted users from using the same router, because the extension cannot distinguish which end user is behind the router call.

### Impact Explanation

Any user can bypass the swap allowlist on a curated pool by routing through the public `MetricOmmSimpleRouter`. This completely defeats the purpose of the allowlist (e.g., KYC enforcement, institutional-only pools, regulatory compliance). Non-allowlisted users can drain liquidity, front-run allowlisted traders, or execute swaps that the pool admin explicitly intended to block. This is a direct loss of the pool's curation guarantee and constitutes broken core pool functionality.

### Likelihood Explanation

The trigger requires the pool admin to have allowlisted the router address. This is a natural and expected configuration step: any pool admin who wants to support the standard periphery path for their allowlisted users must allowlist the router. The router is a public, permissionless contract — any user can call it. The pool admin has no documentation warning that allowlisting the router opens the allowlist to all users. Likelihood is **Medium-High** for any curated pool that supports router-based swaps.

### Recommendation

The extension must check the actual end user, not the intermediary. Two approaches:

1. **Pass the original caller through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling the pool, and the extension decodes and checks it. This requires a protocol-level convention.

2. **Check `recipient` instead of `sender` for swap allowlisting**: If the pool's intent is to gate who receives output (not who initiates), `recipient` is already the end user in single-hop router calls. However, this does not cover multi-hop paths where intermediate recipients are the router itself.

3. **Preferred fix**: Add a dedicated `originalSender` field to the swap interface or extension data that the router always populates with `msg.sender`, and have the extension check that field when present.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin: setAllowedToSwap(pool, alice, true)       // allowlist alice
  - Pool admin: setAllowedToSwap(pool, router, true)      // allowlist router for periphery use

Attack:
  - bob (not allowlisted) calls:
      router.exactInputSingle({pool: pool, recipient: bob, ...})
  - Router calls: pool.swap(bob, ...)  [msg.sender = router]
  - Extension checks: allowedSwapper[pool][router] == true  → PASS
  - Bob's swap executes on the curated pool despite not being allowlisted

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
