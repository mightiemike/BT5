Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, allowing any caller to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` is intended to restrict which addresses may swap against a pool by checking the `sender` argument passed to its `beforeSwap` hook. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is always the router contract address, not the end user. A pool admin who allowlists the router — the only way to enable router-mediated swaps on an allowlisted pool — inadvertently grants every user of the router unrestricted swap access, defeating the per-user allowlist entirely.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` (the direct caller of `pool.swap()`) as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap()
    recipient,
    ...
    extensionData
);
```

`ExtensionCalling._beforeSwap` forwards `sender` verbatim to every configured extension via `_callExtensionsInOrder`:

```solidity
// metric-core/contracts/ExtensionCalling.sol L149-177
function _beforeSwap(address sender, ...) internal {
    _callExtensionsInOrder(
        BEFORE_SWAP_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
    );
}
```

`SwapAllowlistExtension.beforeSwap` then checks that identity against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter` calls `pool.swap()` directly in all four entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`), making the router itself `msg.sender` in the pool context:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData   // user-controlled, not authenticated
    );
```

The router does not encode the originating `msg.sender` into `extensionData`. For any router-mediated swap to succeed on an allowlisted pool, the pool admin must add the router address to `allowedSwapper[pool][router]`. Once that entry is set, `allowedSwapper[pool][router] == true` for every user who routes through the router, collapsing the per-user gate to a per-contract gate.

## Impact Explanation
Any user explicitly blocked from a pool (`allowedSwapper[pool][attacker] == false`) can bypass the restriction by calling `MetricOmmSimpleRouter.exactInputSingle` or any other router entry point targeting that pool. The extension sees `sender = router`, which passes the allowlist check, and the swap executes. The attacker receives output tokens while the pool's LP capital is consumed without the access restriction the admin configured. This is a direct, fund-impacting break of the pool's intended access-control invariant: unauthorized swap settlement against restricted LP capital.

## Likelihood Explanation
The scenario requires no special setup beyond the pool admin performing the expected operational action of enabling router access. Any unprivileged user can trigger it:
1. Pool is deployed with `SwapAllowlistExtension` in the `beforeSwap` order.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps.
3. Any user (including explicitly blocked addresses) calls any router swap function targeting that pool.
4. The extension sees `sender = router`, passes the check, and the swap executes.

No malicious initial setup, non-standard tokens, or privileged cooperation is required.

## Recommendation
The `sender` identity passed to extension hooks must reflect the economic actor, not the intermediary contract. Two complementary approaches:

1. **Router-level**: `MetricOmmSimpleRouter` should encode the originating `msg.sender` into `extensionData` using an authenticated, tamper-evident field (e.g., ABI-encoded with a known prefix), and `SwapAllowlistExtension` should decode and verify it when `sender` is a known trusted forwarder.
2. **Extension-level**: `SwapAllowlistExtension` should maintain a separate registry of trusted forwarders and, when `sender` is a known forwarder, decode the real user from `extensionData` before performing the allowlist check.

The simplest safe default is to document that the router must never be allowlisted directly, and that all allowlisted swaps must arrive via direct `pool.swap()` calls — but this is a usability constraint that is not enforced on-chain and will be violated in practice.

## Proof of Concept
```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension (beforeSwap order set)
  allowedSwapper[pool][router] = true    ← admin enables router access
  allowedSwapper[pool][attacker] = false ← attacker is explicitly blocked

Attack:
  attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: attacker, ...})
    → router calls pool.swap(recipient=attacker, ...)   [msg.sender = router]
    → pool calls _beforeSwap(sender=router, ...)
    → ExtensionCalling passes sender=router to SwapAllowlistExtension.beforeSwap
    → allowedSwapper[pool][router] == true  ✓ passes
    → swap executes, attacker receives output tokens

Result:
  Attacker swaps against a pool they are explicitly blocked from,
  receiving output tokens while the pool's LP capital is consumed
  without the access restriction the admin configured.
```

```solidity
