### Title
Gateway `validate_proof_facts` binds `virtual_os_config_hash` to the current block's `starknet_version` while the batcher executes under the next block's version, causing a hash-domain mismatch at the `V0_14_3` cutover - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful validator computes `virtual_os_config_hash` using the **current** (latest committed) block's `starknet_version`, while the batcher executes transactions under the **next** block's `starknet_version`. At the `V0_14_3` version cutover, `OsConfigHashVersion` switches the hash function from Pedersen to Blake, producing a different `virtual_os_config_hash`. A transaction whose `proof_facts.config_hash` is correct for the next block (Blake-based) is rejected by the gateway; a transaction whose `config_hash` is correct for the current block (Pedersen-based) passes the gateway but reverts during execution.

---

### Finding Description

`run_validate_entry_point` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` builds its `BlockContext` as follows:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // ← only number is bumped
let block_context = BlockContext::new(
    block_info,          // starknet_version is still the CURRENT block's version
    self.chain_info.clone(),
    versioned_constants,
    BouncerConfig::max(),
);
```

Only `block_number` is incremented; `starknet_version` is left as the current block's value. [1](#0-0) 

`BlockContext::virtual_os_config_hash()` then derives the hash using that `starknet_version`:

```rust
OsChainInfo::from(&self.chain_info)
    .compute_virtual_os_config_hash(self.block_info.starknet_version)
``` [2](#0-1) 

`compute_virtual_os_config_hash` delegates to `OsConfigHashVersion::from(starknet_version)`, which selects **Pedersen** for versions `< V0_14_3` and **Blake** for versions `>= V0_14_3`:

```rust
const OS_CONFIG_HASH_V4_CUTOVER: StarknetVersion = StarknetVersion::V0_14_3;

impl From<StarknetVersion> for OsConfigHashVersion {
    fn from(starknet_version: StarknetVersion) -> Self {
        if starknet_version < OS_CONFIG_HASH_V4_CUTOVER { Self::V3 } else { Self::V4 }
    }
}
``` [3](#0-2) 

The two hash functions produce provably different outputs for the same chain config:

```rust
fn test_os_config_hash_pedersen_and_blake_differ() {
    let pedersen_hash = chain_info.compute_os_config_hash(None, StarknetVersion::V0_14_2).unwrap();
    let blake_hash   = chain_info.compute_os_config_hash(None, StarknetVersion::V0_14_3).unwrap();
    assert_ne!(pedersen_hash, blake_hash);
}
``` [4](#0-3) 

During stateful validation the gateway calls `validate_proof_facts`, which enforces:

```rust
let virtual_os_config_hash = block_context.virtual_os_config_hash();
let proof_config_hash = snos_proof_facts.config_hash;
if virtual_os_config_hash != proof_config_hash {
    return Err(TransactionPreValidationError::InvalidProofFacts(...));
}
``` [5](#0-4) 

The batcher builds block N+1 with `starknet_version = V0_14_3`, so its `virtual_os_config_hash` is Blake-based. The gateway validates against block N's `starknet_version = V0_14_2`, so its `virtual_os_config_hash` is Pedersen-based. The two domains are disjoint.

---

### Impact Explanation

At the single-block version-upgrade boundary (block N = `V0_14_2` → block N+1 = `V0_14_3`):

**Scenario A – valid transaction rejected (High):** A user constructs a client-side-proving Invoke V3 transaction with `proof_facts.config_hash = H_blake` (correct for block N+1). The gateway rejects it with `InvalidProofFacts` because it computes `H_pedersen ≠ H_blake`. The transaction never reaches the mempool or batcher, even though it would execute successfully.

**Scenario B – invalid transaction accepted (High):** A user submits `proof_facts.config_hash = H_pedersen` (correct for block N). The gateway accepts it. The batcher executes it under `H_blake`, the check fails inside the Cairo VM, and the transaction reverts on-chain. The gateway has admitted a transaction that produces a wrong (reverted) execution result.

Both match **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing"** and Scenario B additionally touches **"Critical. Wrong state, receipt, event … or revert result from blockifier/syscall/execution logic for accepted input."**

---

### Likelihood Explanation

The mismatch is deterministic and occurs exactly once per version upgrade that crosses the `V0_14_3` cutover. It requires no special attacker capability: any user submitting a client-side-proving transaction during the upgrade window is affected. The `allow_client_side_proving: true` flag is set in the production gateway config, confirming the feature is live. [6](#0-5) 

---

### Recommendation

In `run_validate_entry_point`, set `block_info.starknet_version` to the version that the batcher will use for block N+1, not the version of block N. The simplest fix is to expose the next block's `starknet_version` through `GatewayFixedBlockStateReader::get_block_info` (or a separate method), so the gateway and batcher agree on the domain constant used to compute `virtual_os_config_hash`. Alternatively, derive `virtual_os_config_hash` from a configuration value that is explicitly shared between the gateway and batcher rather than inferred from the committed block header.

---

### Proof of Concept

1. Block N is committed with `starknet_version = V0_14_2`.
2. The sequencer operator upgrades; block N+1 will be produced with `starknet_version = V0_14_3`.
3. Alice computes `H_blake = OsChainInfo::compute_virtual_os_config_hash(V0_14_3)` and embeds it in her Invoke V3 `proof_facts`.
4. Alice submits the transaction to the gateway.
5. Gateway calls `run_validate_entry_point`; `block_info.starknet_version` is still `V0_14_2`; it computes `H_pedersen ≠ H_blake` and returns `InvalidProofFacts` — Alice's valid transaction is rejected.
6. Bob instead embeds `H_pedersen` in his `proof_facts`.
7. Gateway accepts Bob's transaction (hash matches).
8. Batcher executes block N+1 with `starknet_version = V0_14_3`; `virtual_os_config_hash = H_blake ≠ H_pedersen`; Bob's transaction reverts — an invalid transaction was admitted and produces a wrong execution result.

/// The first Starknet version at which the OS config hash is computed with Blake (V4) instead of
/// Pedersen (V3).
const OS_CONFIG_HASH_V4_CUTOVER: StarknetVersion = StarknetVersion::V0_14_3;

/// Selects the hash function and version short-string used to compute the OS config hash.
/// Gated by the block's Starknet version so that blocks produced before the cutover remain
/// re-executable / re-provable with their original (Pedersen) hash.
#[derive(Clone, Copy, Debug, Eq, PartialEq)]
pub enum OsConfigHashVersion {
    /// Pedersen hash, prefixed with 'StarknetOsConfig3'.
    V3,
    /// Blake hash, prefixed with 'StarknetOsConfig4'.
    V4,
}

impl From<StarknetVersion> for OsConfigHashVersion {
    fn from(starknet_version: StarknetVersion) -> Self {
        if starknet_version < OS_CONFIG_HASH_V4_CUTOVER { Self::V3 } else { Self::V4 }
    }
```
