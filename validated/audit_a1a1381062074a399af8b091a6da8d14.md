### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

The contract exposes two versions of its SPV proof endpoint: the deprecated `verify_transaction_inclusion` (v1) and the current `verify_transaction_inclusion_v2` (v2). The v2 was introduced specifically to close the 64-byte Merkle-node forgery vulnerability. However, v1 was never removed from the public ABI and remains callable by any unprivileged NEAR account. Any caller can bypass the coinbase-proof guard entirely by invoking v1 directly, obtaining a `true` proof result for a fake transaction that is actually an internal Merkle tree node.

---

### Finding Description

The contract struct carries a `skip_pow_verification: bool` flag and exposes two distinct proof-verification methods, creating the same "multiple optional behaviors controlled by flags/versioning" complexity described in the external report.

The critical path is `verify_transaction_inclusion` (v1): [1](#0-0) 

It is marked `#[deprecated]` and carries an explicit warning that it may return `true` for internal Merkle tree nodes, but it remains a `pub` method gated only by `#[pause]`. There is no access-control restriction preventing an unprivileged NEAR account from calling it.

The v2 fix adds a coinbase-proof check before delegating back to v1: [2](#0-1) 

Because v1 is still reachable directly, the coinbase guard in v2 is trivially bypassed.

The underlying Merkle computation in `merkle-tools` treats leaf hashes and internal-node hashes identically: [3](#0-2) 

`compute_root_from_merkle_proof` takes any 32-byte `transaction_hash` as its starting point and walks up the tree. It does not distinguish between a real transaction hash (leaf) and an internal node hash.

---

### Impact Explanation

An attacker who knows the Merkle tree of any confirmed block (all data is public on-chain) can:

1. Pick an internal node `N` at depth `d` in that block's Merkle tree.
2. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = <real block>`, `tx_index = <position of N>`, and `merkle_proof = <siblings from N up to root>`.
3. `compute_root_from_merkle_proof(N, position, proof)` produces the block's real `merkle_root`.
4. The function returns `true`.

Any consumer contract (bridge, atomic swap, cross-chain lending protocol) that calls `verify_transaction_inclusion` to gate a fund release will accept this forged proof as valid, allowing the attacker to claim funds for a transaction that never existed. [4](#0-3) 

---

### Likelihood Explanation

**High.** The attack requires no cryptographic computation, no privileged role, and no special tooling. The attacker only needs:
- A block hash present in the mainchain (publicly readable via `get_block_hash_by_height`).
- The Merkle tree of that block (publicly available from any Bitcoin node or block explorer).
- The ability to call a public NEAR contract method.

The deprecated function has been present since before v0.5.0 and is still in the live ABI. Any integrator that did not migrate to v2 is already exposed, and any attacker aware of the 64-byte forgery class can exploit v1 directly even against integrators that believe they are protected.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely. It should not be `pub`; if internal reuse by v2 is needed, make it `fn` (private). The complexity of maintaining two callable proof endpoints with different security properties is the root cause: settling on a single, secure behavior (v2 only) eliminates the attack surface. [5](#0-4) 

---

### Proof of Concept

```
# 1. Read a confirmed mainchain block hash
near view <contract> get_block_hash_by_height '{"height": 685452}'
# → returns real_block_hash

# 2. Fetch the Merkle tree of that block from any Bitcoin explorer.
#    Identify an internal node N at depth 1:
#      N = double_sha256(tx0_hash || tx1_hash)
#    N's position in the proof is 0 (left child of root).

# 3. Call the deprecated v1 endpoint directly (no role required):
near call <contract> verify_transaction_inclusion \
  --args-borsh <borsh-encoded ProofArgs with:
      tx_id            = N          (internal node hash, 32 bytes)
      tx_block_blockhash = real_block_hash
      tx_index         = 0
      merkle_proof     = [sibling_of_N]   (one element: the right child of root)
      confirmations    = 1
  >

# 4. Contract returns `true`.
#    compute_root_from_merkle_proof(N, 0, [sibling]) = double_sha256(N || sibling)
#                                                    = merkle_root  ✓
```

The result `true` is indistinguishable from a legitimate SPV proof to any consumer contract, enabling fraudulent fund releases. [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
