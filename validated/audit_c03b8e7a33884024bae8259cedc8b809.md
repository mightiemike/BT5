### Title
Migration-Induced Chain Halt via Bonus Pool Exhaustion in `exitVestedAccountsPositions` — (`x/tieredrewards/migrations/v2/migrate.go`)

---

### Summary

The v8 upgrade migration calls `ForceFullExitWithDelegation` on every vesting-owned position. That function unconditionally calls `claimRewards`, which calls `processEventsAndClaimBonus`, which calls `sufficientBonusPoolBalance`. If the `RewardsPoolName` module account balance is below the accrued bonus for any vesting-owned position, `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`. This error propagates unhandled through the entire call stack back to the upgrade handler, causing the upgrade to fail and the chain to halt at the upgrade height.

Any account holding positions with accrued bonus rewards can drain the pool to a level below the vesting-owned position's accrued bonus by submitting `MsgClaimRewards` transactions in the blocks immediately before the upgrade height.

---

### Finding Description

**Call chain (all unguarded):**

1. `v2.Migrate` → `exitVestedAccountsPositions` [1](#0-0) 

2. `exitVestedAccountsPositions` → `pk.ForceFullExitWithDelegation(ctx, posID)` — error returned verbatim [2](#0-1) 

3. `ForceFullExitWithDelegation` → `k.claimRewards(ctx, posState)` — error returned verbatim [3](#0-2) 

4. `claimRewards` → `k.processEventsAndClaimBonus(ctx, &pos)` — error returned verbatim [4](#0-3) 

5. `processEventsAndClaimBonus` → `k.sufficientBonusPoolBalance(ctx, bonusCoins)` — returns `ErrInsufficientBonusPool` when pool balance < bonus [5](#0-4) 

6. `sufficientBonusPoolBalance` hard-errors when `poolBalance < bonus` [6](#0-5) 

At no point in this chain is `ErrInsufficientBonusPool` caught, suppressed, or handled with a fallback. The error defined at: [7](#0-6) 

...is only referenced in production code inside `sufficientBonusPoolBalance` — there is no `errors.Is(err, ErrInsufficientBonusPool)` recovery anywhere in the migration path.

**Attacker-controlled drain path:**

The `RewardsPoolName` module account is drained by legitimate `MsgClaimRewards` / `processEventsAndClaimBonus` calls. Any account with positions that have accrued bonus rewards can submit claim transactions in the blocks immediately before the upgrade height. The attacker does not need governance access, operator keys, or any privileged role — only a standard signed transaction.

**Exact state delta required:**

| Variable | Required value |
|---|---|
| `RewardsPoolName` balance at upgrade block | `< accrued_bonus(vesting_position)` |
| At least one vesting-owned position | `totalBonus > 0` (i.e., `LastBonusAccrual < blockTime`, validator bonded) |

Both conditions are achievable: vesting accounts holding tiered-rewards positions are an expected pre-upgrade state (the migration exists precisely to handle them), and the pool balance can be reduced to any target level by claiming rewards from non-vesting positions.

---

### Impact Explanation

A failed upgrade handler in Cosmos SDK causes `EndBlock` to panic/return an error, which prevents the block from being finalized. The chain halts at the upgrade height and cannot make progress until the binary is patched and nodes are restarted with a corrected state or a new binary that handles the error. This is a complete, persistent chain halt — not a transient revert.

---

### Likelihood Explanation

- Vesting-owned positions with accrued bonus are the exact target of the migration; their existence is assumed.
- The pool balance is observable on-chain. An attacker can compute the exact claim amount needed to push the pool below the vesting position's accrued bonus.
- The attack requires only a standard `MsgClaimRewards` transaction submitted before the upgrade block — no special privileges.
- The window is the entire period between upgrade announcement and upgrade height, which is typically days to weeks.

---

### Recommendation

In `ForceFullExitWithDelegation` (or in `exitVestedAccountsPositions`), catch `ErrInsufficientBonusPool` and handle it gracefully — either skip the bonus payout for the vesting position (forfeiting it or deferring it), or cap the bonus to `min(accrued, poolBalance)`. The migration must never return an error for a recoverable pool-balance condition. A simple fix:

```go
// In ForceFullExitWithDelegation, after claimRewards:
if err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        // Log and continue without bonus; pool is exhausted.
        logger.Warn("force-exit: bonus pool insufficient, skipping bonus claim", ...)
    } else {
        return fmt.Errorf("claim rewards for position %d: %w", posID, err)
    }
}
```

### Proof of Concept

```
1. Create a vesting account V and open a tiered-rewards position P owned by V.
2. Let P accrue bonus rewards (advance block time, keep validator bonded).
3. Create a non-vesting account A with positions; claim all of A's bonus rewards
   until RewardsPoolName balance < P's accrued bonus.
4. Trigger the v8 upgrade (run Migrate1to2).
5. Observe: Migrate1to2 returns an error wrapping ErrInsufficientBonusPool;
   the upgrade handler fails; the chain halts.
```

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
