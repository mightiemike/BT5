### Title
`ClampOutgoingGasAdmission` Admission Check Uses Clamped `admission_gas` But Limit Deduction Uses Original `gas` — (`File: runtime/runtime/src/congestion_control.rs`)

### Summary

In `ReceiptSinkV2::try_forward`, the `ClampOutgoingGasAdmission` feature (protocol version 85) computes a clamped `admission_gas = gas.min(allowed_shard_outgoing_gas)` and uses it for the forwarding gate check, but then deducts the original unclamped `gas` from `forward_limit.gas`. This is the exact nearcore analog of the FertilizerFacet bug: a bounded variable is used for the guard, but the unbounded original is used for the state-mutating operation.

### Finding Description

In `runtime/runtime/src/congestion_control.rs`, the `try_forward` function:

```rust
let admission_gas = if ProtocolFeature::ClampOutgoingGasAdmission
    .enabled(apply_state.current_protocol_version)
{
    gas.min(apply_state.config.congestion_control_config.allowed_shard_outgoing_gas)
} else {
    gas
};

if forward_limit.gas >= admission_gas && forward_limit.size >= size {
    outgoing_receipts.push(receipt);
    forward_limit.gas = forward_limit.gas.saturating_sub(gas);  // ← uses original `gas`, not `admission_gas`
    forward_limit.size -= size;
    ...
}
``` [1](#0-0) 

The gate check on line 451 uses `admission_gas` (clamped to `allowed_shard_outgoing_gas` = 1 PGAS), but the deduction on line 454 uses the original `gas`, which can be up to `max_total_prepaid_gas` = 1 PetaGas (1000× larger).

The `allowed_shard_outgoing_gas` is set to **1 PGAS** (1,000,000,000,000,000) in production: [2](#0-1) 

The `ClampOutgoingGasAdmission` feature activates at protocol version 85 (currently live on mainnet at version 86): [3](#0-2) 

The `CongestionControlConfig::allowed_shard_outgoing_gas` field is documented as the absolute minimum workload a congested shard must accept per round to prevent deadlocks: [4](#0-3) 

### Impact Explanation

When `gas > allowed_shard_outgoing_gas` (e.g., a receipt with 300 PGAS attached):

1. `admission_gas = 1 PGAS` (clamped)
2. Gate check passes: `forward_limit.gas >= 1 PGAS` ✓
3. Deduction: `forward_limit.gas = forward_limit.gas.saturating_sub(300 PGAS)` → saturates to **0**
4. All subsequent receipts in the same chunk see `forward_limit.gas = 0` and are buffered instead of forwarded

The `ClampOutgoingGasAdmission` invariant — that no single receipt can consume more than `allowed_shard_outgoing_gas` from the per-chunk outgoing gas budget — is completely defeated. A single large-gas receipt exhausts the entire outgoing gas budget for a destination shard in one chunk, causing all other buffered receipts destined for that shard to remain buffered unnecessarily. This degrades cross-shard throughput and can be triggered by any unprivileged user who submits a function call with high attached gas.

**Impact: 3 / Likelihood: 3**

### Likelihood Explanation

Any user can attach up to `max_total_prepaid_gas` (1 PetaGas) to a function call, which is 1000× `allowed_shard_outgoing_gas`. The receipt is created deterministically from the transaction and flows through the buffered-receipt forwarding path. No special privilege is required. The feature is live on mainnet (protocol version 85+, stable version 86).

### Recommendation

Replace the deduction of `gas` with `admission_gas` so the limit is consumed by the same clamped value used for the gate check:

```rust
if forward_limit.gas >= admission_gas && forward_limit.size >= size {
    outgoing_receipts.push(receipt);
    forward_limit.gas = forward_limit.gas.saturating_sub(admission_gas); // use admission_gas
    forward_limit.size -= size;
    stats.forwarded_receipts.entry(shard).or_default().add_receipt(size, gas);
    Ok(ReceiptForwarding::Forwarded)
}
```

This ensures the outgoing gas budget is decremented by at most `allowed_shard_outgoing_gas` per receipt, which is the invariant `ClampOutgoingGasAdmission` was designed to enforce.

### Proof of Concept

1. Deploy a contract on shard A that creates a cross-shard receipt to shard B with `gas = max_total_prepaid_gas` (1 PetaGas).
2. The receipt enters shard A's outgoing buffer for shard B.
3. On the next chunk, `try_forward` is called for this receipt. With `allowed_shard_outgoing_gas = 1 PGAS` and `forward_limit.gas = max_outgoing_gas = 300 PGAS`:
   - `admission_gas = min(1_000 PGAS, 1 PGAS) = 1 PGAS`
   - Gate check: `300 PGAS >= 1 PGAS` → passes
   - Deduction: `forward_limit.gas = 300 PGAS - 1_000 PGAS` → saturates to **0**
4. All other buffered receipts destined for shard B in this chunk see `forward_limit.gas = 0` and are not forwarded, even though the intended budget was 300 PGAS.
5. The exact divergent value: `forward_limit.gas` is `0` instead of `300 PGAS - 1 PGAS = 299 PGAS`. [5](#0-4)
