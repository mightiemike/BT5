### Title
IBC-received NFTs permanently non-transferable/non-burnable on-chain due to `ValidateDenomID` rejecting `ibc/` prefix in `MsgTransferNFT`, `MsgEditNFT`, and `MsgBurnNFT` — (`x/nft/types/msgs.go`)

### Summary
`MsgTransferNFT.ValidateBasic`, `MsgEditNFT.ValidateBasic`, and `MsgBurnNFT.ValidateBasic` all call `ValidateDenomID`, which enforces a strict `^[a-z0-9]+$` regex. The IBC NFT-transfer module mints voucher NFTs under denom IDs of the form `ibc/<64-hex-chars>` (containing a `/`). Any user who receives an NFT via IBC is permanently unable to transfer, edit, or burn it on-chain via the `x/nft` message server, because every such transaction is rejected at `ValidateBasic` before it reaches the keeper.

### Finding Description
When an IBC NFT packet arrives at the destination chain, `processReceivedPacket` computes `voucherClassID = classTrace.IBCClassID()` (e.g. `ibc/A3C...`) and mints the NFT under that denom ID. The recipient then owns an NFT whose denom ID contains a `/`. If they submit `MsgTransferNFT{DenomId: "ibc/A3C..."}`, the SDK's `AnteHandler` calls `ValidateBasic`, which calls `ValidateDenomID`. That function checks `IsAlphaNumeric` (`^[a-z0-9]+$`), which rejects `/`, and returns `ErrInvalidDenom`. The transaction is rejected before it ever reaches the keeper. The same applies to `MsgEditNFT` and `MsgBurnNFT`. `ValidateDenomIDWithIBC` already exists in the same file and handles the `ibc/` prefix correctly, but is never invoked from any of these three message types.

### Impact Explanation
Every NFT received via IBC is permanently frozen for on-chain operations: the owner cannot transfer it to another on-chain address, edit its metadata, or burn it. The only remaining action is to send it back via IBC (using `x/nft-transfer MsgTransfer`, which does not go through `MsgTransferNFT.ValidateBasic`). This violates the ownership invariant that any NFT owner can exercise full control over their token on-chain, and effectively seizes IBC-received NFTs in place for all on-chain purposes.

### Likelihood Explanation
This is triggered by the normal IBC NFT receive flow — any user who receives an NFT from another chain via IBC is affected. No special privileges or attacker interaction are required. The path is fully reachable through standard production transactions.

### Recommendation
Replace `ValidateDenomID` with `ValidateDenomIDWithIBC` in the `ValidateBasic` methods of `MsgTransferNFT`, `MsgEditNFT`, and `MsgBurnNFT` in `x/nft/types/msgs.go`.

### Proof of Concept

denomID := "ibc/" + strings.Repeat("a", 64) // valid ibc/HASH format
msg := types.NewMsgTransferNFT(
    "tokenid",
    denomID,
    "cosmos1sender...",
    "cosmos1recipient...",
)
err := msg.ValidateBasic()
// err != nil: "the denom(ibc/aaa...aaa) only accepts lowercase alphanumeric characters"

func ValidateDenomIDWithIBC(denomID string) error {
	if strings.HasPrefix(denomID, IBCPrefix) {
		if len(denomID) != IBCDenomLen {
			return sdkerrors.Wrapf(ErrInvalidDenom, "the length of ibc denom(%s) only accepts value [%d]", denomID, IBCDenomLen)
		}
		if _, err := hex.DecodeString(denomID[len(IBCPrefix):]); err != nil {
			return sdkerrors.Wrapf(ErrInvalidDenom, "the hash of ibc denom(%s) must be valid hex", denomID)
		}

		return nil
	}

	return ValidateDenomID(denomID)
}
