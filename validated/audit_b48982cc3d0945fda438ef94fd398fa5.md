Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the originating user, enabling allowlist bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's own `msg.sender` — the `MetricOmmSimpleRouter` contract when users route through it, not the end user. A pool admin who allowlists the router address to enable router-mediated swaps inadvertently grants every user of that router access to the pool, fully nullifying the allowlist policy. Conversely, if the admin does not allowlist the router, all allowlisted users are blocked from using the standard periphery path.

## Finding Description
The call chain is:

1. `MetricOmmSimpleRouter.exactInputSingle` (line 72) calls `IMetricOmmPoolActions(params.pool).swap(...)` — the pool's `msg.sender` is the router address.
2. `MetricOmmPool.swap` (line 230) calls `_beforeSwap(msg.sender, ...)`, forwarding the router address as `sender`.
3. `ExtensionCalling._beforeSwap` (line 160–176) encodes `sender` (the router) and calls the extension.
4. `SwapAllowlistExtension.beforeSwap` (line 37) evaluates `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][router]`.

The extension never sees the originating user's address. There is no mechanism in the pool's `swap` signature or the extension call encoding to propagate the true end-user identity. The admin API (`setAllowedToSwap`, `setAllowAllSwappers`) is keyed by the address the extension will actually observe as `sender`, which is the router, not the user.

**Scenario A — Allowlist bypass:** Admin allowlists the router so curated users can swap through the standard periphery. Because the check is `allowedSwapper[pool][router]`, every user who calls the router passes the guard regardless of whether their own address is on the allowlist.

**Scenario B — Broken core functionality:** Admin does not allowlist the router. Allowlisted users who attempt to swap through the router receive `NotAllowedToSwap` even though their address is on the allowlist. The only usable path is a direct `pool.swap` call.

## Impact Explanation
Scenario A constitutes a direct policy bypass enabling unauthorized trading on pools intended to be restricted — a broken core allowlist invariant with fund-impacting consequences (unauthorized parties execute swaps on curated pools). Scenario B renders the standard router unusable for any pool deploying this extension, breaking the core swap flow for legitimate users. Both outcomes meet Sherlock contest thresholds: Scenario A is High (allowlist fully nullified, any unprivileged user can trade), Scenario B is Medium (core swap path broken for legitimate users).

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool deploying `SwapAllowlistExtension` and expecting users to interact through the router will encounter one of the two failure modes. The trigger requires no privileged access — any user can call `exactInputSingle` or `exactInput`. The pool admin's natural action of allowlisting the router to "enable router swaps" is precisely what opens Scenario A. No special preconditions beyond a normally configured pool are required.

## Recommendation
The pool's `swap` function should propagate the originating user's address through the extension call chain rather than its own `msg.sender`. One approach: add an explicit `originator` field to the swap parameters that the router populates with `msg.sender` (the end user), and have `_beforeSwap` forward that field as `sender` to extensions. Alternatively, the router can expose the current payer via a transient-storage accessor (it already stores the payer in transient storage via `_setNextCallbackContext`), and the extension can read it. Until resolved, `SwapAllowlistExtension` cannot be safely combined with `MetricOmmSimpleRouter` on any pool where the allowlist is intended to restrict access.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `beforeSwap` order.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice should be able to swap.
3. Admin calls `setAllowedToSwap(pool, router, true)` — necessary so Alice can use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
5. The router calls `pool.swap(...)` with `msg.sender = router` (line 72–80, `MetricOmmSimpleRouter.sol`).
6. The pool calls `_beforeSwap(msg.sender=router, ...)` (line 230–240, `MetricOmmPool.sol`).
7. `ExtensionCalling._beforeSwap` encodes `sender=router` and calls the extension (line 160–176, `ExtensionCalling.sol`).
8. The extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds (line 37, `SwapAllowlistExtension.sol`).
9. Bob successfully swaps on a pool he was never authorized to access.
