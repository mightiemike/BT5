### Title
Deprecated `verify_transaction_inclusion` Accepts Unvalidated Caller-Supplied `tx_id`, Enabling 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` remains a live, callable public entry point on the contract. It accepts a fully attacker-controlled `tx_id` with no validation that the value is an actual leaf-level transaction hash rather than an internal Merkle tree node. Any unprivileged NEAR caller can supply a crafted `ProofArgs` whose `tx_id` is an internal node of a real Bitcoin block's Merkle tree and receive a `true` return value, forging proof of a non-existent transaction's inclusion.

---

### Finding Description

The function `verify_transaction_inclusion` at `contract/src/lib.rs:288–323` is marked `#[deprecated]` but remains a fully reachable public method. The `#[deprecated]` attribute in Rust is a compile-time lint only; it imposes no runtime restriction. Any NEAR account can call it directly.

The function accepts `ProofArgs`, whose fields `tx_id`, `tx_index`, and `merkle_proof` are entirely caller-supplied: [1](#0-0) 

The sole validation performed before returning is: [2](#0-1) 

The function checks only that the proof is non-empty and that `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof)` equals the stored `merkle_root` of the target block. It does **not** validate that `tx_id` is a leaf-level transaction hash. The code itself documents this gap: [3](#0-2) 

The 64-byte attack exploits this directly. A Bitcoin Merkle tree internal node is the `double_sha256` of two concatenated 32-byte child hashes — a 64-byte input. A crafted Bitcoin transaction can also be exactly 64 bytes, making its hash indistinguishable from an internal node hash at the contract level. An attacker who knows the Merkle tree of any block already stored in the contract's mainchain can:

1. Pick any internal node `N` at depth `d` in that block's Merkle tree.
2. Compute the sibling path from `N` up to the root (length `D − d`, where `D` is the full tree depth).
3. Submit `tx_id = N`, `tx_index = (position of N at depth d)`, `merkle_proof = (sibling path)`.

`compute_root_from_merkle_proof` will correctly reconstruct the root from `N` and the sibling path, the equality check passes, and the function returns `true`.

The v2 function mitigates this by requiring a coinbase proof of equal length, anchoring the claimed tree depth: [4](#0-3) 

But v1 has no such guard and remains independently callable.

---

### Impact Explanation

Any recipient contract or off-chain consumer that calls `verify_transaction_inclusion` (v1) and acts on a `true` result — releasing funds, minting tokens, unlocking a bridge withdrawal, or updating cross-chain state — can be deceived into treating a non-existent Bitcoin transaction as confirmed. The forged proof is indistinguishable from a legitimate one at the contract interface level. The corrupted output is the boolean proof result itself, which downstream logic treats as authoritative.

---

### Likelihood Explanation

The entry point is public and requires no privileged role. The only prerequisite is that the target block is already in the contract's mainchain (a normal operational state). The Merkle tree structure of every Bitcoin block is public data, so constructing the crafted `ProofArgs` requires only standard Bitcoin RPC access. Any NEAR account can execute the call. Contracts that integrated before v2 was introduced and have not migrated remain directly exposed.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or gate it behind a role that prevents unprivileged access. Do not rely on the `#[deprecated]` attribute as a security boundary — it is invisible at runtime. Callers must be migrated to `verify_transaction_inclusion_v2`, which enforces tree-depth consistency via the coinbase proof and closes the internal-node substitution path.

---

### Proof of Concept

**Setup:** Block `B` is in the contract's mainchain. Its Merkle tree has depth `D = 3` (8 transactions). Internal node `N` at depth `1` is `double_sha256(leaf_4 || leaf_5)`.

**Attacker constructs:**
```
tx_id             = N   (internal node hash, not a real tx)
tx_block_blockhash = hash(B)
tx_index          = 2   (position of N at depth 1, 0-indexed among pairs)
merkle_proof      = [sibling_of_N_at_depth_1, root_sibling_at_depth_0]
                    (length = D - 1 = 2, non-empty → passes require!)
confirmations     = 1
```

**Execution path in `verify_transaction_inclusion`:**

- `args.merkle_proof.is_empty()` → false, passes [5](#0-4) 
- `compute_root_from_merkle_proof(N, 2, [sib1, sib2])` reconstructs the correct Merkle root of `B` [6](#0-5) 
- Equality check passes → function returns `true`

A recipient contract receives `true` and acts as if a real Bitcoin transaction was confirmed in block `B`.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
