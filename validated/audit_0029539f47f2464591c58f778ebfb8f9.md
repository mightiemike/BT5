### Title
`DAO` Excluded from `Pausable` Manager Roles Allows `PauseManager` to Permanently Halt the Protocol Without Override - (`contract/src/lib.rs`)

---

### Summary

The `BtcLightClient` contract configures its `Pausable` plugin with only `Role::PauseManager` as the authorized manager. `Role::DAO`, the highest-privilege role in the system, is **not** included in `manager_roles` for the pause mechanism. This means a `PauseManager` can pause the entire protocol — blocking all block submissions and transaction verifications — and the `DAO` has no on-chain path to override or unpause it without also separately holding the `PauseManager` role.

---

### Finding Description

The contract struct is decorated with:

```rust
#[pausable(manager_roles(Role::PauseManager))]
``` [1](#0-0) 

This means the `near-plugins` `Pausable` macro generates `pa_pause_feature` and `pa_unpause_feature` methods that are gated exclusively to accounts holding `Role::PauseManager`. `Role::DAO` is not listed.

Contrast this with the `Upgradable` configuration immediately below, where `Role::DAO` is **explicitly** included in every role list:

```rust
#[upgradable(access_control_roles(
    code_stagers(Role::CodeStager, Role::DAO),
    code_deployers(Role::CodeDeployer, Role::DAO),
    duration_initializers(Role::DurationManager, Role::DAO),
    ...
))]
``` [2](#0-1) 

The omission of `Role::DAO` from `manager_roles` in the `pausable` attribute is inconsistent with the rest of the access control design and creates a privilege inversion: the `PauseManager` (a lower-privilege operational role) can halt the protocol, but the `DAO` (the highest-privilege governance role) cannot lift that halt.

The methods blocked by a pause are:

- `submit_blocks` — the sole entry point for relayers to advance the canonical chain
- `verify_transaction_inclusion` — the deprecated proof verification endpoint
- `verify_transaction_inclusion_v2` — the current proof verification endpoint [3](#0-2) [4](#0-3) [5](#0-4) 

---

### Impact Explanation

A `PauseManager` account — whether malicious, compromised, or acting in bad faith — can call `pa_pause_feature` to block `submit_blocks`. Once paused:

1. No relayer can advance the canonical Bitcoin chain state on NEAR (even those with `UnrestrictedSubmitBlocks` bypass the `#[pause]` check, but the `DAO` cannot call `pa_unpause_feature` to restore normal operation for all callers).
2. All downstream contracts consuming `verify_transaction_inclusion_v2` receive no valid proofs.
3. The `DAO` has no on-chain method to unpause unless it was separately granted `PauseManager` — a separate administrative action that may not have been taken.

The result is a permanent, irrecoverable protocol halt from the perspective of the `DAO`'s on-chain authority, matching the exact impact class of M-14.

---

### Likelihood Explanation

The `PauseManager` role is an operational role intended to be granted to a hot key or a multisig with lower security requirements than the `DAO`. This makes it a realistic target for compromise. Additionally, a governance dispute (e.g., the `DAO` attempting to revoke the `PauseManager` role) gives the `PauseManager` a window to pause the contract before the revocation takes effect — directly mirroring the 7-day window scenario in M-14.

---

### Recommendation

Add `Role::DAO` to the `manager_roles` list in the `pausable` attribute, consistent with how `DAO` is included in every `upgradable` role list:

```rust
#[pausable(manager_roles(Role::PauseManager, Role::DAO))]
``` [1](#0-0) 

This ensures the highest-privilege role can always override a lower-privilege halt decision, matching the design intent expressed in the `upgradable` configuration.

---

### Proof of Concept

1. `DAO` grants `Role::PauseManager` to account `pause_manager.near`.
2. `DAO` later initiates revocation of `pause_manager.near`'s role (or attempts to replace it).
3. Before revocation is finalized, `pause_manager.near` calls `pa_pause_feature("submit_blocks")`.
4. All calls to `submit_blocks` now revert. The canonical Bitcoin chain state on NEAR is frozen.
5. `DAO` attempts to call `pa_unpause_feature("submit_blocks")` — the call **reverts** because `DAO` does not hold `Role::PauseManager`.
6. The protocol remains permanently halted. The only recovery path is a full contract upgrade (requiring `CodeStager` + `CodeDeployer` + staging duration), which is a slow, disruptive process. [6](#0-5) [7](#0-6)
