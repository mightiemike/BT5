### Title
Incorrect Difficulty Calculation for Fork Blocks Due to Mainchain-Indexed Timestamp Lookup — (`contract/src/dogecoin.rs`)

### Summary

The `get_next_work_required` function in `dogecoin.rs` (and analogously in `bitcoin.rs` and `litecoin.rs`) retrieves the first block of the difficulty adjustment interval using `get_header_by_height`, which resolves blocks exclusively from the **mainchain height index** rather than walking the submitted block's actual ancestor chain. When a fork block is submitted, this causes the difficulty calculation to use timestamps from mainchain blocks instead of the fork's true ancestors, producing an incorrect `expected_bits`. The contract may then accept fork blocks whose `bits` field satisfies the incorrectly computed difficulty but violates the actual protocol-required difficulty, causing the contract's canonical chain view to diverge from network consensus.

### Finding Description

In `get_next_work_required` (`dogecoin.rs`), the timestamp of the first block in the difficulty adjustment interval is fetched as:

```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`get_header_by_height` resolves blocks from `mainchain_height_to_header`, the mainchain-only bidirectional index. Fork blocks are stored only in `headers_pool`, not in `mainchain_height_to_header`. For Dogecoin after block 145,000, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `height_first = prev_block_header.block_height - 1`. [2](#0-1) 

When a fork block at height `H` is submitted whose fork point is at height `H-2` or earlier, the fork's ancestor at `height_first = H-2` is a fork block (not on the mainchain). The contract silently falls back to the mainchain's block at `H-2`, which carries a different timestamp. The resulting `modulated_timespan` — and therefore `expected_bits` — is computed against the wrong reference point. [3](#0-2) 

The same structural flaw exists in `bitcoin.rs` and `litecoin.rs`, where `get_header_by_height` is used to retrieve the first block of the 2016-block retarget interval: [4](#0-3) [5](#0-4) 

For Bitcoin/Litecoin the fork must diverge more than 2015 blocks deep for the ancestor at `height_first` to differ from the mainchain's block, making exploitation impractical. For Dogecoin's per-block adjustment, a fork only 2 blocks deep is sufficient.

The `BtcLightClient` struct confirms the separation between the mainchain index and the full pool: [6](#0-5) 

### Impact Explanation

An attacker crafts a Dogecoin fork where the fork's block at `height_first` carries a recent timestamp, while the mainchain's block at the same height carries an older timestamp. The contract computes `actual_timespan = time(fork's block at H-1) - time(mainchain's block at H-2)`, which is larger than the correct `time(fork's block at H-1) - time(fork's block at H-2)`. A larger timespan yields a higher target (lower difficulty) after Digishield modulation. The attacker sets the fork block's `bits` to match this incorrectly lowered difficulty, satisfying `check_pow`'s `expected_bits == block_header.bits` check while violating the actual Dogecoin protocol's difficulty requirement. [7](#0-6) 

Accepted fork blocks accumulate chainwork via `work_from_bits(block_header.bits)`. If the attacker consistently obtains lower difficulty on their fork, they can mine blocks faster than the honest rate. The contract's reorg logic triggers when `current_header.chain_work > total_main_chain_chainwork`, so a sustained stream of under-difficulty fork blocks can eventually displace the mainchain tip. Downstream callers of `verify_transaction_inclusion` would then verify SPV proofs against a fraudulent chain. [8](#0-7) 

### Likelihood Explanation

For Dogecoin, the trigger condition — a fork at least 2 blocks deep — is routine. The public `submit_blocks` entry point is reachable by any NEAR caller without any privileged role. The attacker controls fork block timestamps within the MTP lower bound and the 2-hour future-time upper bound, providing sufficient freedom to engineer a favorable `modulated_timespan`. The TODO comment in the code is an explicit developer acknowledgment of the uncertainty: [9](#0-8) 

### Recommendation

Replace `get_header_by_height` with a chain-walking approach that traverses the submitted block's actual ancestor chain. Starting from `prev_block_header`, call `get_prev_header` iteratively `blocks_to_go_back` times to reach the true ancestor at `height_first`. This mirrors the reference Dogecoin implementation's intent and ensures the difficulty calculation uses timestamps from the fork's own chain rather than the mainchain index.

### Proof of Concept

1. Mainchain has blocks at heights `0…H`. Mainchain block at `H-2` has timestamp `T_main = 1000`.
2. Attacker submits a fork block at height `H-2` with timestamp `T_fork = 1050` (recent, within MTP/future-time bounds). This block is stored in `headers_pool` only.
3. Attacker submits fork block at height `H-1` with timestamp `T_fork + 10 = 1060`.
4. Attacker submits fork block at height `H`. `get_next_work_required` computes:
   - `height_first = H-2`
   - `first_block_time = T_main = 1000` (mainchain block, not fork ancestor at `T_fork = 1050`)
   - `actual_timespan = 1060 - 1000 = 60` (correct would be `1060 - 1050 = 10`)
   - `modulated_timespan = 60 + (60 - 60)/8 = 60` (at target, no adjustment)
   - Correct: `modulated_timespan = 60 + (10 - 60)/8 = 60 - 6.25 ≈ 54` → higher difficulty
5. The contract computes `expected_bits` from the inflated timespan, yielding a lower difficulty than the actual protocol requires.
6. Attacker sets the fork block's `bits` to match this lower difficulty, mines the block with reduced PoW, and the contract accepts it — a block the actual Dogecoin network would reject.

// source https://github.com/dogecoin/dogecoin/blob/2c513d0172e8bc86fe9a337693b26f2fdf68a013/src/dogecoin.cpp#L41
fn calculate_next_work_required(
    config: &DogecoinConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let retarget_timespan = config.pow_target_timespan;
    let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;

    let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;

    let min_timespan = retarget_timespan - (retarget_timespan / 4);
    let max_timespan = retarget_timespan + (retarget_timespan / 2);

    if modulated_timespan < min_timespan {
        modulated_timespan = min_timespan;
    } else if modulated_timespan > max_timespan {
        modulated_timespan = max_timespan;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(modulated_timespan).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target =
        new_target / U256::from(<i64 as TryInto<u64>>::try_into(retarget_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
}
