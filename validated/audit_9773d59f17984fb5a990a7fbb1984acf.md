### Title
CEI Violation in `withdrawCollateral()`: External Token Transfer Executes Before Subaccount Balance Is Decremented — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary
`Clearinghouse.withdrawCollateral()` violates the Checks-Effects-Interactions pattern. The external token transfer via `handleWithdrawTransfer()` is dispatched at line 408, before `spotEngine.updateBalance()` decrements the subaccount balance at line 412. During the external call window, the subaccount's on-chain balance in `SpotEngine` still reflects the pre-withdrawal amount. If a supported token implements transfer hooks (e.g., ERC777 `tokensReceived`), the recipient contract at `sendTo` receives execution control while the protocol's accounting state is stale.

---

### Finding Description

`Clearinghouse.withdrawCollateral()` executes in this order:

```
1. handleWithdrawTransfer(token, sendTo, amount, idx)   ← external call (line 408)
     └─ token.safeTransfer(withdrawPool, amount)
     └─ BaseWithdrawPool.submitWithdrawal(...)
          └─ token.safeTransfer(sendTo, amount)         ← sendTo receives tokens HERE
2. spotEngine.updateBalance(productId, sender, amountRealized)  ← state change (line 412)
3. spotEngine.assertUtilization(productId)              (line 413)
4. require(getHealth(sender, healthType) >= 0, ...)     (line 419)
```

The subaccount balance deduction in `SpotEngine` (step 2) happens **after** the tokens have already been transferred to `sendTo` (step 1). This is the same structural flaw as the reported `CreditUSDMinter.mint()` bug: an external call precedes all state changes.

The `sendTo` address is user-controlled — it is either derived from `sender` or explicitly provided in the `WithdrawCollateralV2` transaction type (resolved in `BaseWithdrawPool.resolveFastWithdrawal()` at line 73). If `sendTo` is a contract backed by a token with transfer hooks, it receives a callback while `spotEngine` still records the full pre-withdrawal balance for `sender`.

During this callback window, the attacker's contract can call `BaseWithdrawPool.submitFastWithdrawal()` — which is **publicly callable with no access restriction** — using a pre-signed fast-withdrawal transaction for the same subaccount. `submitFastWithdrawal` pays out from the pool's own liquidity and marks the idx, but does **not** update `spotEngine`. The sequencer later processes the fast-withdrawal's corresponding `withdrawCollateral` call, which will again decrement the balance. Combined with the in-flight regular withdrawal whose balance decrement is still pending, the subaccount ends up double-paid: once from the pool liquidity via `submitFastWithdrawal`, and once from the regular withdrawal path — while the health check at line 419 only fires after the state is finally written, too late to catch the double-spend.

---

### Impact Explanation

An attacker who controls `sendTo` and holds a valid pre-signed fast-withdrawal transaction can extract funds twice for a single collateral balance:

- **Asset delta**: The pool's token reserve is drained by `submitFastWithdrawal` during the callback, and the regular withdrawal also completes. Net effect: two payouts for one balance unit.
- **Broken invariant**: `SpotEngine` balance for `sender` is decremented only once (by the regular withdrawal), but two token transfers have occurred. The pool's liquidity is permanently reduced by the fast-withdrawal amount without a corresponding balance deduction.
- **Scope**: Any product whose token implements transfer hooks (ERC777 or custom `onTokenReceived`-style callbacks).

---

### Likelihood Explanation

**Medium.** Two conditions must hold simultaneously:

1. A supported collateral token implements transfer hooks (ERC777 `tokensReceived` or equivalent). This is a realistic deployment scenario — USDC on some chains and many DeFi tokens use ERC777 or custom hook patterns.
2. The attacker must hold a valid pre-signed fast-withdrawal transaction for the same subaccount with `idx > minIdx`. This is achievable by any user who has previously signed such a transaction off-chain.

The `onlyEndpoint` modifier on `withdrawCollateral` prevents direct re-entry into that function, but it does **not** protect against re-entry into the publicly accessible `submitFastWithdrawal` on `BaseWithdrawPool`.

---

### Recommendation

Apply the CEI pattern: move all `spotEngine` state mutations **before** the external transfer call.

```solidity
function withdrawCollateral(...) public virtual onlyEndpoint {
    ...
    // EFFECTS first
    int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
    int128 amountRealized = -int128(amount) * int128(multiplier);
    spotEngine.updateBalance(productId, sender, amountRealized);
    spotEngine.assertUtilization(productId);
    require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

    // INTERACTIONS last
    handleWithdrawTransfer(token, sendTo, amount, idx);
    emit ModifyCollateral(amountRealized, sender, productId);
}
```

Alternatively, add OpenZeppelin's `ReentrancyGuard` to `Clearinghouse` and apply `nonReentrant` to `withdrawCollateral`, and also to `BaseWithdrawPool.submitFastWithdrawal`.

### Proof of Concept

**Setup**: Token `T` is an ERC777 token registered as a supported collateral. Attacker controls address `A` (subaccount `A_sub`) with balance `X` in `SpotEngine`. Attacker also holds a valid pre-signed `WithdrawCollateral` fast-withdrawal transaction for `A_sub` with `idx = N+1` (where `N` is current `minIdx`).

**Attack steps**:

1. Attacker submits a regular withdrawal of `X` via the sequencer. Sequencer calls `Endpoint.submitTransactions()` → `Clearinghouse.withdrawCollateral(A_sub, productId, X, A_contract, N)`.

2. Inside `withdrawCollateral`, line 408 calls `handleWithdrawTransfer(T, A_contract, X, N)`.
   - `T.safeTransfer(withdrawPool, X)` moves tokens to `WithdrawPool`.
   - `BaseWithdrawPool.submitWithdrawal(T, A_contract, X, N)` sets `markedIdxs[N] = true`, `minIdx = N`, then calls `T.safeTransfer(A_contract, X)`.

3. ERC777 `tokensReceived` hook fires on `A_contract`. At this moment, `spotEngine.getBalance(productId, A_sub)` still returns `X` (line 412 has not executed).

4. `A_contract` calls `BaseWithdrawPool.submitFastWithdrawal(N+1, signedTx, sigs)`.
   - `markedIdxs[N+1]` is false, `N+1 > minIdx (N)` — passes.
   - `markedIdxs[N+1] = true`.
   - `sendTo == msg.sender` branch: `transferAmount -= fee`, pool pays `A_contract` `X - fee` tokens from its own reserve.

5. Control returns to `withdrawCollateral`. Line 412 executes: `spotEngine.updateBalance(productId, A_sub, -X)`. Health check passes (balance is now 0, which is ≥ 0).

6. Later, sequencer processes the fast-withdrawal transaction (idx `N+1`) via `withdrawCollateral` again — but `markedIdxs[N+1]` is already set, so `submitWithdrawal` returns early (line 124–126). No second balance deduction occurs.

**Result**: Attacker received `X` tokens from the regular withdrawal path and `X - fee` tokens from the fast-withdrawal pool, while `SpotEngine` only decremented the balance once. The `WithdrawPool` reserve is short by `X - fee`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

        require(idx > minIdx, "idx too small");

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

            require(transferAmount > uint128(fee), "Fee larger than balance");

            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;
