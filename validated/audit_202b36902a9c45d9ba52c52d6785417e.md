### Title
`isHealthy()` Stub Always Returns `true` in `OffchainExchange.sol`, Permanently Bypassing Post-Trade Health Enforcement

---

### Summary

`OffchainExchange.isHealthy()` is a stub function that unconditionally returns `true` without performing any actual health check. It is called after every order match to enforce that both taker and maker subaccounts remain solvent post-trade. Because it always returns `true`, this on-chain safety invariant is permanently non-functional, allowing order matching to produce undercollateralized subaccounts with no on-chain rejection.

---

### Finding Description

In `OffchainExchange.sol`, the function `isHealthy()` is declared as:

```solidity
function isHealthy(
    bytes32 /* subaccount */
) internal view virtual returns (bool) {
    return true;
}
```

The subaccount parameter is commented out and the body unconditionally returns `true`. This function is called at the conclusion of `matchOrders()` as the final on-chain health gate:

```solidity
require(isHealthy(taker.order.sender), ERR_INVALID_TAKER);
require(isHealthy(maker.order.sender), ERR_INVALID_MAKER);
```

Because `isHealthy()` always returns `true`, these two `require` statements are dead code. No actual health computation (via `getHealth()`, `getHealthContribution()`, or any engine query) is ever performed after balance mutations in `matchOrders()`.

This is structurally identical to the reference bug: a check function whose controlling state is never initialized (or in this case, never implemented), causing it to always pass.

---

### Impact Explanation

The post-trade health check in `matchOrders()` is the on-chain enforcement layer that is supposed to guarantee subaccounts cannot be driven below their initial health threshold through trading. Without it:

- A taker or maker subaccount can be left with negative initial health after a match, creating an undercollateralized position.
- The protocol's stated "on-chain settlement with cryptographic guarantees" is violated: the sequencer can match orders that produce bad debt with no on-chain rejection.
- Undercollateralized positions accumulate silently until liquidation, potentially exceeding the insurance fund and causing socialized losses across the protocol.

The `withdrawCollateral` path does enforce `getHealth() >= 0`, but that check only fires at withdrawal time, not at trade time. Positions can be built up through repeated unhealthy matches before any withdrawal is attempted.

---

### Likelihood Explanation

Every call to `matchOrders()` — the primary settlement path for all spot and perpetual trades — passes through the broken health check. The sequencer submits `MatchOrders` and `MatchOrdersWithAmount` transactions in every trading batch. Any order pair that, after execution, leaves a subaccount below initial health will silently pass the on-chain check. This is reachable by any user who submits a signed order that the sequencer includes in a batch.

---

### Recommendation

Replace the stub with a real health check that calls the clearinghouse:

```solidity
function isHealthy(bytes32 subaccount) internal virtual returns (bool) {
    return clearinghouse.getHealth(
        subaccount,
        IProductEngine.HealthType.INITIAL
    ) >= 0;
}
```

Since `getHealth()` is non-view (it emits `PriceQuery` events), `isHealthy()` must also be non-view, which requires updating the `matchOrders()` signature accordingly.

---

### Proof of Concept

1. User A signs a taker order for product X that, if fully matched, would reduce their quote balance below their initial health threshold.
2. Sequencer calls `submitTransactionsChecked()` → `processTransactionImpl()` → `matchOrders()`.
3. `_updateBalances()` applies the base and quote deltas to both subaccounts.
4. `require(isHealthy(taker.order.sender), ERR_INVALID_TAKER)` is reached.
5. `isHealthy()` returns `true` unconditionally — no health computation occurs.
6. The transaction succeeds. User A's subaccount now has negative initial health with an open position, creating latent bad debt.
