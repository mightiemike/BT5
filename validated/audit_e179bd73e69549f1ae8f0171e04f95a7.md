### Title
`minDepositRateX18` Creates Structural Insolvency When Utilization Is Below 100% — (`core/contracts/SpotEngineState.sol`)

---

### Summary

`SpotEngineState._updateState` applies `minDepositRateX18` to **both** `cumulativeDepositsMultiplierX18` and `cumulativeBorrowsMultiplierX18`. This guarantees depositors a minimum yield, but the real token inflows that fund it come only from borrowers, scaled by utilization. When utilization is below 100%, the accounting obligations to depositors grow faster than actual token inflows, creating a structural deficit that compounds over time and eventually prevents later withdrawers from recovering their full balance.

---

### Finding Description

In `SpotEngineState._updateState`, after computing the normal utilization-based interest, the minimum deposit rate is applied unconditionally to both cumulative multipliers:

```solidity
// core/contracts/SpotEngineState.sol lines 155–161
state.cumulativeBorrowsMultiplierX18 = state
    .cumulativeBorrowsMultiplierX18
    .mul(minDepositRateMultiplierX18);

state.cumulativeDepositsMultiplierX18 = state
    .cumulativeDepositsMultiplierX18
    .mul(minDepositRateMultiplierX18);
``` [1](#0-0) 

The intent is that borrowers fund the minimum rate. However, the actual token inflow from borrowers paying `minDepositRateX18` is proportional to utilization. At utilization `u`:

- Depositors' accounting balances grow by `minDepositRateX18` × `totalDeposits` per year (full amount, regardless of `u`)
- Borrowers pay `minDepositRateX18` × `totalBorrows` = `minDepositRateX18` × `u` × `totalDeposits` per year (only on borrowed portion)

The gap — `(1 − u) × minDepositRateX18 × totalDeposits` per year — is phantom yield: it exists in accounting but is not backed by any real token inflow. No mechanism in the codebase injects tokens from the insurance fund or protocol fees to cover this gap.

The `assertUtilization` guard in `SpotEngine.sol` only checks `totalDeposits >= totalBorrows`, which remains satisfied because both multipliers grow proportionally. It does not verify that the Clearinghouse's actual ERC20 token balance covers the net accounting obligations. [2](#0-1) 

The `QUOTE_PRODUCT_ID` is initialized with `minDepositRateX18: 0`, so the quote token is not directly affected. However, any non-quote spot product added via `addOrUpdateProduct` with a non-zero `minDepositRateX18` is vulnerable. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

The Clearinghouse holds real ERC20 tokens. As `cumulativeDepositsMultiplierX18` grows from `minDepositRateX18` without a corresponding token inflow, the sum of all depositor accounting balances exceeds the actual token balance held. Withdrawals are served first-come-first-served via `withdrawCollateral` → `handleWithdrawTransfer` → `token.safeTransfer`. Early withdrawers receive their full balance; later withdrawers find the contract underfunded and their `safeTransfer` reverts. The deficit compounds annually at `(1 − utilization) × minDepositRateX18 × totalDeposits`. [5](#0-4) 

---

### Likelihood Explanation

Medium. The condition requires: (1) a non-quote spot product configured with `minDepositRateX18 > 0`, and (2) utilization below 100% — the normal operating state for any lending market. The deficit is silent and accumulates continuously with every `updateStates` call, making it inevitable over a long enough time horizon without an external funding mechanism. [6](#0-5) 

---

### Recommendation

One of the following mitigations should be applied:

1. **Fund the gap from the insurance fund**: When applying `minDepositRateX18`, compute the unfunded portion `(1 − utilization) × minDepositRateX18 × totalDeposits` and deduct it from `insurance` in `Clearinghouse`. Revert or cap the rate if insurance is insufficient.
2. **Only apply `minDepositRateX18` to `cumulativeDepositsMultiplierX18`** and separately charge borrowers for it only when they exist, ensuring the rate is always fully funded.
3. **Enforce a minimum utilization** before `minDepositRateX18` takes effect, or scale `minDepositRateX18` by the current utilization ratio so depositors only earn what borrowers actually pay.

---

### Proof of Concept

1. A non-quote spot product is added with `minDepositRateX18 = 5e16` (5% APR).
2. User A deposits 100M tokens; no borrowers exist (`utilizationRatio = 0`).
3. `updateStates(dt)` is called every day for one year.
4. Each call: `cumulativeDepositsMultiplierX18 *= (1 + 5%/year)^dt`; `cumulativeBorrowsMultiplierX18` also grows but `totalBorrowsNormalized = 0`, so no real tokens flow in.
5. After one year, User A's accounting balance = 105M tokens.
6. The Clearinghouse holds 100M tokens.
7. User A calls `withdrawCollateral` for 105M → `token.safeTransfer` reverts; User A can only withdraw 100M.
8. The 5M deficit is permanent and grows each subsequent year. [7](#0-6) [1](#0-0)

        uint32 productId,
        State memory state,
        uint128 dt

        int128 borrowRateMultiplierX18;
        int128 totalDeposits = state.totalDepositsNormalized.mul(
            state.cumulativeDepositsMultiplierX18
        );
        int128 totalBorrows = state.totalBorrowsNormalized.mul(
            state.cumulativeBorrowsMultiplierX18
        );
        int128 utilizationRatioX18 = totalBorrows.div(totalDeposits);
        int128 minDepositRateX18;
        {
            Config memory config = configs[productId];

            // annualized borrower rate
            int128 borrowerRateX18 = config.interestFloorX18;
            if (utilizationRatioX18 == 0) {
                // setting borrowerRateX18 to 0 here has the property that
                // adding a product at the beginning of time and not using it until time T
                // results in the same state as adding the product at time T
                borrowerRateX18 = 0;
            } else if (utilizationRatioX18 < config.interestInflectionUtilX18) {
                borrowerRateX18 += config
                    .interestSmallCapX18
                    .mul(utilizationRatioX18)
                    .div(config.interestInflectionUtilX18);
            } else {
                borrowerRateX18 +=
                    config.interestSmallCapX18 +
                    config.interestLargeCapX18.mul(
                        (
                            (utilizationRatioX18 -
                                config.interestInflectionUtilX18).div(
                                    ONE - config.interestInflectionUtilX18
                                )
                        )

            );

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
