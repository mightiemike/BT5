### Title
Unvalidated Peer-Supplied Transaction Hashes Stored Directly into Block Body — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

The P2P sync client accepts `transaction_hash` values from untrusted peers and writes them directly into storage without recomputing or verifying them against the transaction body. A malicious peer can supply an arbitrary hash for any transaction, causing the node to store and serve a wrong transaction hash through its RPC layer.

---

### Finding Description

In `TransactionStreamFactory::parse_data_for_block`, each `FullTransaction` received from a P2P peer is destructured into `(transaction, transaction_output, transaction_hash)`. The `transaction_hash` field is pushed directly into `block_body.transaction_hashes` with no recomputation or cross-check:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [1](#0-0) 

The codebase has a well-defined path for computing the canonical hash of a transaction: `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash(&self, chain_id)` and the family of `get_*_transaction_hash` functions in `transaction_hash.rs`. These are used correctly in the gateway and consensus converter paths — `convert_rpc_tx_to_internal` always recomputes the hash from the transaction body using the node's own `chain_id`:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
``` [2](#0-1) 

The P2P sync path skips this recomputation entirely. The `FullTransaction` protobuf carries a `transaction_hash` field that is serialized from storage on the serving peer and deserialized on the receiving peer with no integrity check: [3](#0-2) 

The block body (including the peer-supplied hashes) is then written to storage via `write_to_storage`, making the wrong hashes permanent.

The analog to the CRL finding is exact: just as `_getPcsAttestationData` returns CRL bytes and the caller performs no validation beyond non-emptiness, `parse_data_for_block` receives a hash from a peer and performs no validation beyond its presence in the protobuf message.

---

### Impact Explanation

Once the wrong hash is persisted, the node's RPC layer serves it authoritatively. A client calling `starknet_getTransactionByHash` with the correct canonical hash will receive "not found." A client using the attacker-supplied hash will receive the transaction, binding the wrong hash to that transaction in all downstream consumers (explorers, wallets, provers). The block body's `transaction_hashes` vector is also used to compute the transaction commitment for block hash verification; a mismatch between the stored hashes and the commitment in the validated block header goes undetected.

This matches the allowed impact: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**, and **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload**.

---

### Likelihood Explanation

Any peer that the node connects to for P2P sync can trigger this. No special privilege is required — the attacker only needs to be a reachable P2P peer that the syncing node queries for block transactions. The attack is passive from the victim's perspective (the node initiates the sync query). The TODO comment in the source confirms the developers are aware the input is untrusted and the validation is missing.

---

### Recommendation

After receiving each `FullTransaction` from a peer, recompute the transaction hash locally using the node's `chain_id` and the `validate_transaction_hash` function (or the appropriate `get_*_transaction_hash` variant), and reject the peer with a `BadPeerError` if the recomputed hash does not match the peer-supplied one. This mirrors the validation already performed in `convert_rpc_tx_to_internal`.

Additionally, after all transactions for a block are collected, recompute the transaction commitment from the locally-verified hashes and assert it matches the commitment in the already-validated block header.

---

### Proof of Concept

1. A malicious P2P peer receives a `TransactionsRequest` query from the syncing node for block N.
2. The peer responds with valid `FullTransaction` messages for each transaction in block N, but substitutes an arbitrary `transaction_hash` (e.g., all-zeros) for each one.
3. `parse_data_for_block` pushes each attacker-supplied hash into `block_body.transaction_hashes` without recomputation.
4. The block body is written to storage.
5. A client calls `starknet_getTransactionByHash(canonical_hash)` — the node returns "not found."
6. A client calls `starknet_getTransactionByHash(0x000...0)` — the node returns the transaction, binding the wrong hash to it authoritatively. [4](#0-3)

    fn parse_data_for_block<'a>(
        transactions_response_manager: &'a mut ClientResponsesManager<DataOrFin<FullTransaction>>,
        block_number: BlockNumber,
        storage_reader: &'a StorageReader,
    ) -> BoxFuture<'a, Result<Option<Self::Output>, ParseDataError>> {
        async move {
            let mut block_body = BlockBody::default();
            let mut current_transaction_len = 0;
            let target_transaction_len = storage_reader
                .begin_ro_txn()?
                .get_block_header(block_number)?
                .expect("A header with number lower than the header marker is missing")
                .n_transactions;
            while current_transaction_len < target_transaction_len {
                let maybe_transaction = transactions_response_manager.next().await.ok_or(
                    ParseDataError::BadPeer(BadPeerError::SessionEndedWithoutFin {
                        type_description: Self::TYPE_DESCRIPTION,
                    }),
                )?;
                let Some(FullTransaction { transaction, transaction_output, transaction_hash }) =
                    maybe_transaction?.0
                else {
                    if current_transaction_len == 0 {
                        return Ok(None);
                    } else {
                        return Err(ParseDataError::BadPeer(BadPeerError::NotEnoughTransactions {
                            expected: target_transaction_len,
                            actual: current_transaction_len,
                            block_number: block_number.0,
                        }));
                    }
                };
                block_body.transactions.push(transaction);
                block_body.transaction_outputs.push(transaction_output);
                // TODO(eitan): Validate transaction hash from untrusted sources
                block_body.transaction_hashes.push(transaction_hash);
                current_transaction_len += 1;
            }
            Ok(Some((block_body, block_number)))
        }
        .boxed()
    }
