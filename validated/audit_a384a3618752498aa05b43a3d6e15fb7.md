### Title
`socializeSubaccount` Permanently Reverts When Bad Debt Equals or Exceeds Total Deposits, Locking Depositor Funds — (`core/contracts/SpotEngine.sol`)

### Summary

When a single insolvent subaccount's liability equals or exceeds the total deposited amount for a spot product, the `socializeSubaccount` function computes a `cumulativeDepositsMultiplierX18` of zero (or negative), then unconditionally reverts on `require(state.cumulativeDepositsMultiplierX18 > 0)`. Because `_finalizeSubaccount` has no error-handling around this call, the entire liquidation finalization path is permanently bricked for that product, trapping depositor funds.

### Finding Description

In `SpotEngine.socializeSubaccount` (lines 255–263):

```solidity
if (balance.amount < 0) {
    int128 totalDeposited = state.totalDepositsNormalized.mul(
        state.cumulativeDepositsMultiplierX18
    );
    state.cumulativeDepositsMultiplierX18 = (totalDeposited +
        balance.amount).div(state.totalDepositsNormalized);  // line 260-261
    require(state.cumulativeDepositsMultiplierX18 > 0);      // line 263
``` [1](#0-0) 

The new multiplier simplifies algebraically to:

```
new_multiplier = old_multiplier × (totalDeposited + balance.amount) / totalDeposited
```

When `balance.amount` (the bad debt, negative) satisfies `|balance.amount| >= totalDeposited`, the numerator `(totalDeposited + balance.amount)` is `<= 0`. `MathSD21x18.div` performs integer division `(x * 1e18) / y`, so a zero or negative numerator yields `<= 0`, and the `require` reverts. [2](#0-1) 

The critical enabler is `assertUtilization`, which uses `>=` (not `>`):

```solidity
require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
``` [3](#0-2) 

This explicitly permits 100% utilization (`totalDeposits == totalBorrows`). A single subaccount can therefore borrow the entire deposit pool. When that subaccount becomes insolvent and insurance is exhausted, `_finalizeSubaccount` calls `spotEngine.socializeSubaccount` with no try/catch:

```solidity
if (v.insurance <= 0) {
    spotEngine.socializeSubaccount(txn.liquidatee);  // line 408 — no error handling
}
``` [4](#0-3) 

The revert propagates up through `liquidateSubaccountImpl`, making every future finalization attempt fail.

### Impact Explanation

- Depositors cannot withdraw: `assertUtilization` blocks withdrawals at 100% utilization.
- The insolvent subaccount cannot be finalized: every call to `liquidateSubaccountImpl` with `productId == type(uint32).max` reverts.
- The bad debt is never cleared from the books.
- All depositor funds in that product are effectively locked until someone deposits additional capital (which is economically irrational given the known bad debt).

### Likelihood Explanation

The path is reachable through normal protocol operation:
1. A product has total deposits of `D`.
2. One subaccount borrows `D` (100% utilization — permitted by `assertUtilization`).
3. The borrower's collateral value collapses (e.g., oracle price crash).
4. Liquidators drain the insurance fund to zero.
5. `_finalizeSubaccount` is called → `socializeSubaccount` reverts → permanent DoS.

No admin keys, governance capture, or impossible token behavior is required. The scenario is reachable via the standard `liquidateSubaccountImpl` → `_finalizeSubaccount` → `socializeSubaccount` call chain.

### Recommendation

Replace the hard revert with a floor-clamp. If `totalDeposited + balance.amount <= 0`, the multiplier should be set to the minimum representable positive value (1 wei in X18), and the residual bad debt should be absorbed (depositors lose everything in that product, but the protocol state remains consistent and unblocked):

```solidity
int128 remaining = totalDeposited + balance.amount;
if (remaining <= 0) {
    // Total loss: set multiplier to minimum positive, wipe deposits
    state.cumulativeDepositsMultiplierX18 = 1;
} else {
    state.cumulativeDepositsMultiplierX18 = remaining.div(state.totalDepositsNormalized);
    require(state.cumulativeDepositsMultiplierX18 > 0);
}
```

Alternatively, remove the `require` entirely and allow the multiplier to reach zero (treating it as a total-loss event), then gate downstream operations on a zero-multiplier check.
