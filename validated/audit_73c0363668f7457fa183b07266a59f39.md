### Title
Difficulty Retarget Uses Mainchain Ancestor Instead of Fork Ancestor, Corrupting Fork Block Validation — (`contract/src/dogecoin.rs`, `contract/src/litecoin.rs`, `contract/src/bitcoin.rs`)

### Summary
When validating a fork block that crosses a difficulty-adjustment boundary, all three chain modules call `get_header_by_height` to obtain the timestamp of the period-start block. That helper always resolves against the **mainchain** height-to-hash map, not the fork's actual ancestor chain. If the fork diverged before the difficulty-period boundary, the timestamp used belongs to a different block than the fork's true ancestor, producing an incorrect expected-`bits` value and corrupting the difficulty gate for every fork block at or after that boundary.

### Finding Description

In `get_next_work_required` (Bitcoin, Litecoin) and the equivalent Dogecoin function, when the incoming block height falls on a retarget boundary the code does:

```rust
// dogecoin.rs  line 292-295
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

`get_header_by_height` is implemented as:

```rust
// lib.rs  lines 677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

It unconditionally reads from `mainchain_height_to_header`, which maps only **canonical-chain** blocks. When the block being validated belongs to a fork that branched off before `height_first`, the mainchain block at `height_first` is a different block (with a potentially different timestamp) than the fork's true ancestor at that height. The codebase itself flags this with an explicit TODO:

```rust
// dogecoin.rs  line 291
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
```

The same structural flaw exists in `litecoin.rs` and `bitcoin.rs` at the equivalent call sites. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation

`check_pow` computes `expected_bits` from the wrong timestamp and then enforces:

```rust
require!(
    expected_bits == block_header.bits,
    "bad-diffbits: incorrect proof of work"
);
``` [5](#0-4) 

Two concrete outcomes follow:

1. **Wrong expected difficulty is easier than correct**: A fork block whose `bits` field encodes a lower difficulty (higher target, less work) than the fork's true retarget would require passes the `expected_bits` check. The block is stored and its (reduced) chainwork is accumulated. Over many blocks this can allow a fork built with less total PoW to be accepted as valid.

2. **Wrong expected difficulty is harder than correct**: A legitimately mined fork block whose `bits` matches the fork's true retarget is rejected because the mainchain-derived expected value differs. This silently blocks valid chain reorganizations, freezing the light client on a stale canonical chain.

Both outcomes corrupt the fork-choice invariant that the contract is designed to enforce.

### Likelihood Explanation

For **Dogecoin** (every-block retargeting after height 145,000), `height_first = prev_block_header.block_height - 1`. Any fork block at depth ≥ 3 from the fork point triggers the wrong lookup, because the fork's grandparent at that height differs from the mainchain block stored under that height. This is the common case for any non-trivial fork. [6](#0-5) 

For **Bitcoin** and **Litecoin** (2016-block intervals), the bug fires whenever a fork crosses a retarget boundary, which is realistic for forks that persist across an epoch boundary — a scenario the contract explicitly supports via its fork-storage and reorg logic. [7](#0-6) 

The entry path is the public `submit_blocks` call, reachable by any trusted relayer (or by the DAO/`UnrestrictedSubmitBlocks` role), with no additional privilege required beyond submitting headers. [8](#0-7) 

### Recommendation

Replace `get_header_by_height` in the retarget path with an ancestor-traversal that walks the fork's own `prev_block_hash` chain back to `height_first`. Store the result of that traversal (or cache it in `ExtendedHeader`) so the correct period-start timestamp is used regardless of whether the block being validated is on the mainchain or a fork.

### Proof of Concept

**Dogecoin scenario (every-block retarget, height > 145,000):**

1. Mainchain tip is at height H. Mainchain block at H−1 has timestamp `T_main`.
2. A relayer submits a fork block `F1` at height H (diverging from the mainchain at H−1). `F1` has timestamp `T_fork ≠ T_main`.
3. The relayer submits fork block `F2` at height H+1, with `prev = F1`.
4. `get_next_work_required` is called for `F2`. `height_first = H−1`. `get_header_by_height(H−1)` returns the **mainchain** block with timestamp `T_main`.
5. `calculate_next_work_required` uses `T_main` instead of `T_fork`, producing `expected_bits_wrong`.
6. The relayer crafts `F2.bits` to match `expected_bits_wrong` (not the correct retarget). The `expected_bits == block_header.bits` check passes.
7. The PoW check `hash ≤ target_from_bits(F2.bits)` is satisfied against the wrong target.
8. `F2` is stored with chainwork derived from the wrong difficulty, corrupting the fork-choice accounting. [9](#0-8) [10](#0-9)

// source https://github.com/dogecoin/dogecoin/blob/2c513d0172e8bc86fe9a337693b26f2fdf68a013/src/dogecoin.cpp#L41
fn calculate_next_work_required(
    config: &DogecoinConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let retarget_timespan = config.pow_target_timespan;
    let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;

    let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;

    let min_timespan = retarget_timespan - (retarget_timespan / 4);
    let max_timespan = retarget_timespan + (retarget_timespan / 2);

    if modulated_timespan < min_timespan {
        modulated_timespan = min_timespan;
    } else if modulated_timespan > max_timespan {
        modulated_timespan = max_timespan;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(modulated_timespan).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target =
        new_target / U256::from(<i64 as TryInto<u64>>::try_into(retarget_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
