Audit Report

## Title
Pool Admin Can Reduce Timelock to Zero and Atomically Disable Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`proposeOracleStopLossTimelock` accepts `newTimelock = 0` with no minimum validation. Once executed, `_afterTimelock` returns `block.timestamp`, and `_requireElapsed(block.timestamp)` passes immediately because `block.timestamp < block.timestamp` is false. The pool admin can then propose and execute `drawdownE6 = 0` in the same block, causing `_afterSwapOracleStopLoss` to return early unconditionally and disabling the entire stop-loss guard with zero LP reaction window.

## Finding Description

**Root cause — no minimum timelock floor:**

`proposeOracleStopLossTimelock` at [1](#0-0)  accepts any `uint32` value including 0 with no validation, unlike `proposeOracleStopLossDrawdown` and `proposeOracleStopLossDecay` which call `_validateDrawdown`/`_validateDecay`.

**Step 1 — Reduce timelock to zero (must wait current timelock T):**

`proposeOracleStopLossTimelock(pool_, 0)` is accepted. After T seconds, `executeOracleStopLossTimelock` writes `oracleStopLossConfig[pool_].timelock = 0`. [2](#0-1) 

**Step 2 — Atomically propose + execute `drawdownE6 = 0`:**

With `timelock = 0`, `_afterTimelock` returns `uint32(block.timestamp + 0)` = `block.timestamp`. [3](#0-2) 

`_requireElapsed(block.timestamp)` evaluates `block.timestamp < block.timestamp` → false → does not revert. [4](#0-3) 

`_validateDrawdown(0)` checks `0 > E6` → false → does not revert, so `drawdownE6 = 0` is a valid proposal value. [5](#0-4) 

**Step 3 — Stop-loss guard fully disabled:**

`_afterSwapOracleStopLoss` reads `drawdownE6` and returns early when it is 0, skipping all watermark checks for every subsequent swap. [6](#0-5) 

The same zero-delay pattern applies to `proposeOracleStopLossDecay`/`executeOracleStopLossDecay` and `proposeOracleStopLossHighWatermarks`/`executeOracleStopLossHighWatermarks`, so all three parameter axes can be atomically corrupted after the timelock is zeroed. [7](#0-6) 

**Existing guards are insufficient:** `initialize` calls `_validateDrawdown` and `_validateDecay` but has no `_validateTimelock`, so a zero timelock can be set both at initialization and via the propose/execute path. [8](#0-7) 

## Impact Explanation

Once `drawdownE6 = 0`, the `afterSwap` hook returns early for every swap. The stop-loss that prevents per-share value from falling below the drawdown floor is completely inoperative. LPs are exposed to unlimited value leakage through swaps with no on-chain protection and no reaction window. This is a direct admin-boundary break: the pool admin exceeds the constraint imposed by the timelock mechanism that the NatDoc explicitly states exists so "LPs can react." [9](#0-8) 

**Severity: High** — broken core pool functionality (stop-loss guard) causing potential loss of LP principal with zero LP reaction window after the timelock is zeroed.

## Likelihood Explanation

The pool admin is a single EOA or contract returned by `IMetricOmmPoolFactory(FACTORY).poolAdmin(pool_)` and is explicitly semi-trusted — the timelock exists precisely to limit their power. The attack requires only two phases: wait the current timelock (visible on-chain), then atomically disable protections. No external oracle manipulation, no non-standard tokens, and no factory owner involvement is needed. The attack is repeatable and fully on-chain.

## Recommendation

1. **Enforce a minimum timelock floor** in `proposeOracleStopLossTimelock` (e.g., `require(newTimelock >= MIN_TIMELOCK)`), preventing reduction to zero.
2. **Use strict inequality** in `_requireElapsed`: change `block.timestamp < executeAfter` to `block.timestamp <= executeAfter` so that same-block execution is always rejected regardless of timelock value. [4](#0-3) 
3. **Validate timelock at initialization** — add a `_validateTimelock` check in `initialize` mirroring `_validateDrawdown` and `_validateDecay`. [10](#0-9) 

## Proof of Concept

```solidity
// Foundry integration test sketch
function test_timelockZeroDisablesStopLoss() public {
    // Assume pool deployed with timelock = 1 days, drawdownE6 = 50_000 (5%)

    // Phase 1: propose timelock = 0 (must wait current timelock)
    vm.prank(poolAdmin);
    ext.proposeOracleStopLossTimelock(pool, 0);

    // Warp past current timelock
    vm.warp(block.timestamp + 1 days);

    // Execute timelock reduction — now oracleStopLossConfig[pool].timelock == 0
    vm.prank(poolAdmin);
    ext.executeOracleStopLossTimelock(pool);

    // Phase 2: same block — propose drawdown = 0
    // _afterTimelock returns block.timestamp; _requireElapsed passes immediately
    vm.prank(poolAdmin);
    ext.proposeOracleStopLossDrawdown(pool, 0);

    // Same block — execute drawdown = 0
    vm.prank(poolAdmin);
    ext.executeOracleStopLossDrawdown(pool);

    // Assert drawdown is now 0
    (uint32 drawdown,,,) = ext.oracleStopLossConfig(pool);
    assertEq(drawdown, 0);

    // Phase 3: swap that would have triggered stop-loss now succeeds
    // _afterSwapOracleStopLoss returns early at `if (drawdown == 0) return;`
    // No OracleStopLossTriggered revert — stop-loss is fully disabled
    _executeValueDrainingSwap(pool); // succeeds, no revert
}
```
