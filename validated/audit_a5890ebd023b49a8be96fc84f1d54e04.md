### Title
Zero-Fee Fast Withdrawal via Integer Truncation for Low-Decimal Tokens — (`core/contracts/BaseWithdrawPool.sol`)

### Summary
`fastWithdrawalFeeAmount` divides `feeX18` by `multiplier = 10^(18 - decimals)` using integer division with no minimum-fee floor enforced in native token units. For a 6-decimal token with `withdrawFeeX18 = 0`, any withdrawal of `amount ≤ 999` (< $0.001) produces `fee = 0`. `submitFastWithdrawal` has no `require(fee > 0)` guard, so the withdrawal executes with zero fee collected.

### Finding Description
In `fastWithdrawalFeeAmount`:

```solidity
int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));  // 1e12 for 6-dec
int128 amountX18  = int128(amount) * int128(multiplier);
int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18); // amount * 1e9
int128 minFeeX18  = 5 * spotEngine().getConfig(productId).withdrawFeeX18; // 0 if fee=0
int128 feeX18     = MathHelper.max(proportionalFeeX18, minFeeX18);
return feeX18 / int128(multiplier);  // truncates to 0 for amount < 1000
``` [1](#0-0) 

`FAST_WITHDRAWAL_FEE_RATE = 1e15` and `MathSD21x18.mul` divides by `1e18`, so `proportionalFeeX18 = amount * 1e9`. Dividing by `multiplier = 1e12` gives `amount / 1000`, which is 0 for `amount ≤ 999`. When `withdrawFeeX18 = 0`, `minFeeX18 = 0` provides no floor. [2](#0-1) 

`submitFastWithdrawal` applies the fee without any `require(fee > 0)` guard: [3](#0-2) 

### Impact Explanation
The `fees[productId]` accounting invariant is broken: every fast withdrawal is supposed to pay a positive fee to compensate the pool for liquidity risk, but for small amounts with `withdrawFeeX18 = 0`, `fees[productId] += 0`. The pool transfers the full `transferAmount` to the user and earns nothing.

**Important caveat on the "drain" claim:** The pool's *principal* is not at risk. The pool is repaid the full withdrawal amount when the sequencer later calls `submitWithdrawal`. The loss is purely fee revenue — the pool provides liquidity for free. This is a fee accounting failure, not an insolvency or undercollateralization event. [4](#0-3) 

### Likelihood Explanation
**Constrained by verifier signatures.** Each call to `submitFastWithdrawal` requires `v.requireValidTxSignatures(transaction, idx, signatures)` — the attacker cannot forge these. [5](#0-4) 

Additionally, each `idx` must be unique and `> minIdx`. The attacker cannot replay the same signed transaction. Exploiting this at scale requires the verifier to sign many small-amount withdrawal transactions, which is a normal user flow but limits the practical throughput of the attack.

The precondition `withdrawFeeX18 = 0` is also required. The quote product is initialized with `withdrawFeeX18 = ONE` ($1), but other products can be configured with zero. [6](#0-5) 

### Recommendation
Add a minimum native-unit fee floor in `fastWithdrawalFeeAmount`, or add a `require(fee > 0, "Fee must be positive")` guard in `submitFastWithdrawal`. A simple fix:

```solidity
int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);
require(fee > 0, "Fast withdrawal fee rounds to zero");
```

Alternatively, enforce a minimum fee of 1 native token unit inside `fastWithdrawalFeeAmount`:

```solidity
return MathHelper.max(feeX18 / int128(multiplier), 1);
```
