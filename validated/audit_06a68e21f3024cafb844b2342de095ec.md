### Title
Repeated `(pk, msg)` Pairs in `op_bls_verify` Allow Signature Forgery Without the Secret Key - (File: `src/bls_ops.rs`)

---

### Summary

`op_bls_verify` in `src/bls_ops.rs` iterates over an attacker-controlled flat list of `(G1 pubkey, message)` pairs without any deduplication check. Because BLS aggregate verification is linear in the pairing product, repeating the same `(pk, msg)` pair N times while supplying `N * sig` as the signature produces a mathematically valid verification result. An attacker who has observed any valid signature for `(pk, msg)` can compute `N * sig` using the in-CLVM `g2_multiply` operator and pass `bls_verify` with a signature that was never produced by the key holder.

---

### Finding Description

`op_bls_verify` collects `(pk, msg)` pairs into a `Vec` and passes them directly to `aggregate_verify`:

```rust
// src/bls_ops.rs lines 381-396
let mut items = Vec::<(PublicKey, Atom)>::new();
while !nilp(a, args) {
    let pk = a.g1(first(a, args)?)?;
    args = rest(a, args)?;
    let msg = atom(a, first(a, args)?, "bls_verify message")?;
    args = rest(a, args)?;
    // ← no uniqueness check on (pk, msg)
    cost += BLS_PAIRING_COST_PER_ARG;
    ...
    items.push((pk, msg));
}
if !aggregate_verify(&signature, items) {
    Err(EvalErr::BLSVerifyFailed(input))?
```

`aggregate_verify` (from `chia_bls`) implements the AUG-scheme check:

```
e(sig, G) == Π e(pk_i, H(pk_i ‖ msg_i))
```

When the same `(pk, msg)` pair appears N times, the right-hand side becomes `e(pk, H(pk‖msg))^N`. Because the GT group has prime order, this equals `e(N·sig, G)` when `sig` is a valid signature for `(pk, msg)`. Therefore:

```
aggregate_verify(N·sig, [(pk,msg), (pk,msg), …, (pk,msg)]) == true
```

whenever `sig` is a valid BLS signature for `(pk, msg)`. The attacker computes `N·sig` trivially using the `g2_multiply` CLVM operator, which is available in the same execution environment.

The root cause is the absence of any duplicate-entry guard in the loop at `src/bls_ops.rs` lines 382–393. The external report's analog is the absence of a uniqueness check on the `delegates` list in `tallyProposalResult`, where the same address's weight is summed multiple times.

---

### Impact Explanation

An attacker who has observed a valid signature `sig` for `(pk, msg)` (e.g., from any prior on-chain transaction) can:

1. Compute `N·sig` using `(g2_multiply sig N)` inside a CLVM program.
2. Craft a CLVM program that calls `(bls_verify N·sig pk msg pk msg … pk msg)` with N repetitions of `(pk, msg)`.
3. `op_bls_verify` returns success (cost `3_000_000 + N × 1_200_000`, well within block limits).

The corrupted result is a **false positive from `aggregate_verify`**: a signature `N·sig` that was never produced by the holder of `sk` is accepted as valid. Any coin puzzle that uses `bls_verify` to gate spending—and whose solution allows the caller to supply the signature and/or the `(pk, msg)` list—can be bypassed. This constitutes **authentication bypass / signature forgery** at the CLVM layer.

---

### Likelihood Explanation

- The attacker needs only one legitimately-signed `(pk, msg)` pair, which is available from any prior transaction on the blockchain.
- Computing `N·sig` requires only the `g2_multiply` operator, which is a standard CLVM primitive.
- The attack is executable entirely within a single CLVM program; no off-chain cryptographic capability beyond scalar multiplication is required.
- Cost is proportional to N, so the attack is bounded by the block cost limit, but N = 2 is sufficient to demonstrate the forgery.

Likelihood: **Medium-High**. The precondition (a prior valid signature for the target key) is routinely satisfied in production.

---

### Recommendation

Before appending each `(pk, msg)` pair to `items`, check that the pair has not already been seen. A `HashSet<([u8;48], Vec<u8>)>` keyed on the serialized pubkey bytes and message bytes is sufficient. If a duplicate is detected, return `EvalErr::InvalidOpArg` (consistent with other argument-validation errors in the operator). The same guard should be applied to `op_bls_pairing_identity` for defense-in-depth, even though the repeated-pair attack there does not produce a useful bypass due to the prime-order GT group.

### Proof of Concept

Let `pk` be any BLS public key and `msg` any message. Let `sig` be a valid BLS signature for `(pk, msg)` (obtainable from any prior transaction). Let `sig2 = g2_multiply(sig, 2)`.

CLVM invocation (pseudocode):
```
(bls_verify sig2 pk msg pk msg)
```

Internally, `op_bls_verify` builds `items = [(pk, msg), (pk, msg)]` and calls:
```
aggregate_verify(sig2, [(pk, msg), (pk, msg)])
```

The check evaluates:
```
e(sig2, G) == e(pk, H(pk‖msg))^2
e(2·sig, G) == e(pk, H(pk‖msg))^2
e(sig, G)^2 == e(pk, H(pk‖msg))^2   ← true because sig is valid
```

`aggregate_verify` returns `true`; `op_bls_verify` returns `Ok(Reduction(cost, nil))`. The signature `sig2 = 2·sig` was never produced by the holder of the secret key corresponding to `pk`. [1](#0-0) [2](#0-1) [3](#0-2)

```rust
pub fn op_bls_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = BLS_PAIRING_BASE_COST;
    check_cost(cost, max_cost)?;

    let mut args = input;

    // the first argument is the signature
    let signature = a.g2(first(a, args)?)?;

    // followed by a variable number of (G1, msg)-pairs (as a flat list)
    args = rest(a, args)?;

    let mut items = Vec::<(PublicKey, Atom)>::new();
    while !nilp(a, args) {
        let pk = a.g1(first(a, args)?)?;
        args = rest(a, args)?;
        let msg = atom(a, first(a, args)?, "bls_verify message")?;
        args = rest(a, args)?;

        cost += BLS_PAIRING_COST_PER_ARG;
        cost += msg.as_ref().len() as Cost * BLS_MAP_TO_G2_COST_PER_BYTE;
        cost += DST_G2.len() as Cost * BLS_MAP_TO_G2_COST_PER_DST_BYTE;
        check_cost(cost, max_cost)?;

        items.push((pk, msg));
    }

    if !aggregate_verify(&signature, items) {
        Err(EvalErr::BLSVerifyFailed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
}
```
