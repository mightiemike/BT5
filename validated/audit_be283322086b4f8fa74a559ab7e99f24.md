Audit Report

## Title
Router-Mediated Swap Bypasses `SwapAllowlistExtension` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the value passed by the pool as `msg.sender` of the `pool.swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. Consequently, the allowlist checks the router's address rather than the real swapper's address. Any unprivileged user can bypass the per-pool swap allowlist by routing through the public router.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)` where `params.recipient` is the output recipient and `msg.sender` is the actual user.

2. The router calls `pool.swap(params.recipient, ...)`:
   ```solidity
   // MetricOmmSimpleRouter.sol:72-80
   _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
   (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
     .swap(
       params.recipient,   // recipient
       params.zeroForOne,
       ...
     );
   ```

3. Inside `MetricOmmPool.swap`, `msg.sender` is the **router address**. The pool passes `msg.sender` (the router) as `sender` to `_beforeSwap`:
   ```solidity
   // MetricOmmPool.sol:230-240
   _beforeSwap(
     msg.sender,   // <-- router address, NOT the actual user
     recipient,
     ...
   );
   ```

4. `ExtensionCalling._beforeSwap` forwards this `sender` (router address) to the extension's `beforeSwap(sender, ...)`.

5. `SwapAllowlistExtension.beforeSwap` checks:
   ```solidity
   // SwapAllowlistExtension.sol:37-39
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
     revert IMetricOmmPoolActions.NotAllowedToSwap();
   }
   ```
   Here `msg.sender` is the pool and `sender` is the **router address**. The actual user's address is never checked.

**Root cause:** The allowlist checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`. The actual user's identity (`msg.sender` of the router call) is stored only in the router's transient callback context and is never forwarded to the pool or extension.

**Why existing guards fail:** The `onlyPool` modifier in `BaseMetricExtension` correctly ensures only a registered pool can call the extension, but it does not validate that the `sender` argument represents the true initiating user. The pool correctly passes its own `msg.sender` as `sender`, but that is the router, not the user.

**Exploit scenario:**
- Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a whitelist of addresses.
- Pool admin allowlists the router address (`setAllowedToSwap(pool, router, true)`) so that allowlisted users can swap via the router.
- Any unprivileged user (not on the allowlist) calls `router.exactInputSingle(...)` and successfully swaps, because the extension only sees the router address (which is allowlisted) as the sender.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control mechanism. Its bypass allows any unprivileged user to swap on pools intended to be restricted. This breaks the pool's access-control invariant and can result in unauthorized trading activity on pools designed for permissioned participants (e.g., KYC-gated pools, institutional pools). This constitutes broken core pool functionality and an admin-boundary break reachable by an unprivileged trader.

## Likelihood Explanation
The exploit requires only that the pool admin has allowlisted the router address — a natural and expected configuration for any pool that intends to support router-mediated swaps. Any unprivileged user can then call the public router with no special privileges. The attack is repeatable and requires no special setup beyond a standard router call.

## Recommendation
The router should forward the actual user's address to the pool so the extension can check it. One approach: add a `swapper` field to the pool's `swap` call signature, or pass the actual user address in `callbackData` and have the extension decode it. Alternatively, `SwapAllowlistExtension.beforeSwap` should check the `recipient` argument (second parameter) or a dedicated swapper field rather than `sender` when the caller is a known router. The cleanest fix is for the pool to accept an explicit `swapper` address parameter distinct from `msg.sender`, so extensions always receive the true initiating user.

## Proof of Concept

```solidity
// 1. Deploy pool with SwapAllowlistExtension
// 2. Pool admin allowlists the router:
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// 3. Unprivileged user (not individually allowlisted) calls router:
vm.prank(unprivilegedUser); // unprivilegedUser NOT in allowedSwapper[pool]
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: address(token0),
    tokenOut: address(token1),
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: unprivilegedUser,
    deadline: type(uint256).max,
    priceLimitX64: 0,
    extensionData: ""
}));
// Swap succeeds — allowlist bypassed because extension sees sender=router (allowlisted)
// instead of sender=unprivilegedUser (not allowlisted)
```
