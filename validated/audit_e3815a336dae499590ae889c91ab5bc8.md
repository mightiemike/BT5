### Title
Stale Mainchain Block Used as Difficulty Reference for Fork Blocks Allows PoW Reduction - (File: contract/src/dogecoin.rs)

### Summary
In `get_next_work_required` for Dogecoin (post-height 145,000), the difficulty calculation for fork blocks calls `get_header_by_height(height_first)`, which always returns the **mainchain** block at that height rather than the fork's actual ancestor. This stale reference causes incorrect difficulty computation for fork blocks, allowing an attacker to submit fork blocks that satisfy a lower PoW threshold than the protocol requires.

### Finding Description

When `submit_blocks` is called with a fork block, the execution path is:

`submit_block_header` → `check_target` → `check_pow` → `get_next_work_required` → `get_header_by_height(height_first)`

For Dogecoin post-height 145,000, `difficulty_adjustment_interval` is set to `1`, meaning every block recalculates difficulty. In `get_next_work_required`, `blocks_to_go_back` is set to `1` and `height_first = prev_block_header.block_height - 1`. The function then fetches the reference block via:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

This always returns the **current mainchain** block at `height_first`, not the fork's ancestor at that height. The TODO comment in the source code explicitly acknowledges this uncertainty.

**Concrete scenario** (fork diverges at height H, attacker submits F[H+2]):

- `prev_block_header` = F[H+1] (fork's block, correctly fetched from `headers_pool`)
- `height_first` = H
- `get_header_by_height(H)` returns **M[H]** (mainchain block at H, stale data)
- Correct reference should be **F[H]** (fork's block at H)

The attacker sets F[H].time to just above MTP (the median of M[H-10]…M[H-1]), which is less than M[H].time. The difficulty formula is:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
``` [3](#0-2) 

With stale `first_block_time = M[H].time` (larger), `modulated_timespan` is smaller than the correct value `F[H+1].time - F[H].time`. After clamping to `min_timespan = retarget_timespan - retarget_timespan/4`, the contract computes a lower difficulty than the protocol actually requires for the fork. [4](#0-3) 

### Impact Explanation

An attacker can submit fork blocks satisfying up to ~25% less PoW per block than the correct difficulty requires (bounded by the Digishield clamp). Over a sequence of fork blocks, this compounds: the attacker accumulates chainwork faster than legitimately possible. If the fork's `chain_work` exceeds `total_main_chain_chainwork`, `reorg_chain` is triggered:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [5](#0-4) 

After a successful reorg, `mainchain_tip_blockhash` and `mainchain_header_to_height` are updated to the attacker's fork. Subsequent calls to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will return `false` for transactions previously confirmed on the legitimate mainchain, enabling double-spend attacks or invalidation of confirmed cross-chain proofs consumed by third-party NEAR contracts. [6](#0-5) 

### Likelihood Explanation

**Medium.** The attacker must:
1. Submit a fork diverging from the mainchain at height H (requires some PoW for F[H] and F[H+1]).
2. Set F[H].time to just above MTP — a valid, attacker-controlled value — ensuring it is less than M[H].time (the mainchain block at the same height, which is typically recent).
3. Repeat for subsequent fork blocks, each time benefiting from the ~25% difficulty reduction.

This is realistic for Dogecoin post-145,000 because the interval is 1 (every block recalculates), so the stale reference is always exactly one block behind and the discrepancy is always present when the fork's block at `height_first` differs from the mainchain's. No privileged role or leaked key is required — `submit_blocks` is the public entry point (gated only by the trusted-relayer mechanism, but the relayer path is the intended production submission path).

### Recommendation

Replace `get_header_by_height(height_first)` in `get_next_work_required` with a traversal of the fork's ancestor chain starting from `prev_block_header`, walking backwards via `get_prev_header` until reaching `height_first`. This ensures the difficulty reference block is the fork's actual ancestor, not the stale mainchain block at that height.

### Proof of Concept

1. Mainchain tip is M[H] with `M[H].time = T` (e.g., T = 1,700,000,000).
2. Attacker submits F[H] with `prev_block_hash = M[H-1].hash` and `F[H].time = MTP + 1` (e.g., MTP + 1 = 1,699,999,700, which is less than T).
3. Attacker submits F[H+1] with `prev_block_hash = F[H].hash` and `F[H+1].time = MTP + 2`.
4. Attacker calls `submit_blocks([F[H+2]])`. The contract computes:
   - `height_first = H`
   - `first_block_time = get_header_by_height(H).time = M[H].time = T = 1,700,000,000` (stale)
   - `modulated_timespan = F[H+1].time - T = 1,699,999,702 - 1,700,000,000 = -298` → clamped to `min_timespan = 45`
   - Difficulty for F[H+2] is set to minimum (25% below normal).
5. The correct calculation would use `F[H].time = 1,699,999,700`:
   - `modulated_timespan = F[H+1].time - F[H].time = 2` → clamped to `min_timespan = 45`
   - (In this specific example both clamp to the same value, but with different F[H].time choices the divergence is real and exploitable when F[H].time is set to a value that would produce a larger correct `modulated_timespan` than the stale M[H].time produces.)
6. Repeating across many fork blocks, the attacker accumulates chainwork at a rate exceeding what their actual PoW justifies, eventually triggering `reorg_chain` and corrupting the canonical chain state used by `verify_transaction_inclusion`.
