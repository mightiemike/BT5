### Title
Router-Mediated Swaps Bypass `SwapAllowlistExtension` Per-EOA Gate — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap`. When a swap is routed through `MetricOmmSimpleRouter`, that value is the **router contract address**, not the originating EOA. A pool admin who allowlists the router (the only way to permit router-mediated swaps) inadvertently grants every EOA on the network the ability to trade on a pool that is supposed to be restricted to a curated set of addresses.

### Finding Description

The identity check in `SwapAllowlistExtension.beforeSwap` is:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (correct — the pool calls the extension). `sender` is whatever was passed as the first argument to `pool.swap()`, which is `msg.sender` inside `MetricOmmPool.swap()`.

When the call originates from `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`), the pool's `msg.sender` is the **router contract**, so `sender = address(router)`.

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][eoa]`.

This creates an irreconcilable design conflict:

| Pool admin intent | What they must configure | Actual result |
|---|---|---|
| Allow specific EOAs to swap directly | `allowedSwapper[pool][eoa] = true` | Those EOAs can only call `pool.swap()` directly; router path is blocked for them |
| Allow specific EOAs to swap via router | Must allowlist the router address | **Every EOA** can now bypass the allowlist via the router |

There is no configuration that achieves "specific EOAs may swap through the router."

### Impact Explanation

Any non-allowlisted EOA can trade on a pool that is supposed to be restricted to a curated set of swappers, simply by routing through the public `MetricOmmSimpleRouter`. The allowlist — the sole access-control mechanism for curated pools — is rendered ineffective for all router-mediated swap paths. This is broken core functionality: the pool's access-control invariant is violated by an unprivileged public call.

### Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical swap interface for end users. Any pool admin who wants to allow router-mediated swaps for their allowlisted users must allowlist the router address, which is the natural and expected configuration. The bypass is therefore reachable in any realistic curated-pool deployment that uses the router.

### Recommendation

Pass the originating EOA through the call stack rather than the immediate `msg.sender`. One approach: require callers to supply the end-user address as an explicit parameter in `extensionData`, and have the extension decode and verify it against a signature or a trusted forwarder registry. Alternatively, redesign `SwapAllowlistExtension` to treat the router as a transparent forwarder by reading the payer from transient storage (which the router already sets via `_setNextCallbackContext`) and exposing it to extensions.
