Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of original swapper, allowing allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded from the pool, which is the pool's `msg.sender`. When `MetricOmmSimpleRouter` is used, the pool's `msg.sender` is the router contract, not the original user. A pool admin who allowlists the router to enable router-based swaps inadvertently grants swap access to every address on-chain, completely defeating the per-pool allowlist.

## Finding Description
`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` receives this value as `sender` and checks it against the per-pool allowlist: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` directly, making the pool's `msg.sender` the router contract: [3](#0-2) 

The original user's address is stored only in transient callback context for payment settlement and is never forwarded to the pool or any extension: [4](#0-3) 

The allowlist check therefore evaluates `allowedSwapper[pool][router]` — the router address — rather than the original user. Once the router is allowlisted (the natural configuration step for any curated pool whose users are expected to use the standard periphery), `allowedSwapper[pool][router]` is `true` for every call, so any user can bypass the allowlist by routing through `MetricOmmSimpleRouter`. The same mismatch applies to `exactInput` (intermediate hops use `address(this)` as payer), `exactOutputSingle`, and `exactOutput`.

`DepositAllowlistExtension` does not share this flaw because it checks the `owner` argument (the LP position beneficiary), which is passed through correctly and represents the economically relevant actor: [5](#0-4) 

## Impact Explanation
`SwapAllowlistExtension` is the sole on-chain mechanism for restricting swap access to a curated set of addresses (e.g., KYC-verified users, institutional counterparties, or whitelisted strategies). Once the router is allowlisted, the gate is open to every address on-chain. Unauthorized traders can execute swaps at the pool's oracle-anchored prices, causing adverse selection against LPs and extracting value from a pool designed to serve only a restricted set of counterparties. This constitutes a direct loss of LP principal and a complete break of the pool's intended access-control invariant — a High-severity impact under Sherlock criteria.

## Likelihood Explanation
The scenario requires the pool admin to allowlist the router. This is the natural and expected configuration step for any curated pool whose allowlisted users are expected to interact through the standard periphery. The protocol ships `MetricOmmSimpleRouter` as the primary user-facing swap interface; a pool admin who does not allowlist the router effectively makes the router unusable for their pool, which is an undocumented and non-obvious restriction. The probability that at least one production pool admin takes this step is high, making exploitation straightforward and repeatable by any on-chain address.

## Recommendation
The extension must check the economically relevant actor — the address that initiated the transaction — not the immediate `msg.sender` of `pool.swap`. Two viable approaches:

1. **Router-forwarded identity**: Add an optional `swapper` field to `extensionData` that the router encodes as `msg.sender` before forwarding. `SwapAllowlistExtension.beforeSwap` decodes and checks this field when present, falling back to `sender` for direct pool calls.

2. **Pool-level sender override**: Extend the pool's `swap` interface with an explicit `swapper` parameter (separate from `recipient`) that the router populates with `msg.sender`. Extensions receive this value instead of the pool's `msg.sender`.

Either fix must be applied consistently across `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput`.

## Proof of Concept
```
Setup
─────
1. Deploy MetricOmmPool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is the curated user
3. Pool admin calls setAllowedToSwap(pool, router, true)  // needed so alice can use the router

Attack
──────
4. bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool:      <restricted pool>,
           recipient: bob,
           ...
       })

5. Router calls pool.swap(bob, ...) — msg.sender inside pool = router.

6. Pool calls _beforeSwap(sender=router, ...).

7. SwapAllowlistExtension checks allowedSwapper[pool][router] → true (step 3).

8. Swap executes. bob receives tokens from the restricted pool.

Result: bob, who was never allowlisted, successfully swaps in a pool
        intended to be restricted to alice only.
```
