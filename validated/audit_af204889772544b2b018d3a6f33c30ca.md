### Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Mid-Price, Systematically Inflating `metricT1` and Weakening the Stop-Loss Guard for `!zeroForOne` Swaps â€” (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss()` computes the oracle mid-price as the arithmetic mean `(bid + ask) / 2`, while every other price-sensitive path in the protocol (including `PriceVelocityGuardExtension`) uses the geometric mean via `SwapMath.midAndSpreadFeeX64FromBidAsk`. Because the arithmetic mean is always â‰¥ the geometric mean (AM-GM inequality), `metricT1` â€” the per-share value denominated in token1 â€” is systematically overestimated. An overestimated `metricT1` makes the pool appear healthier than it is, causing the stop-loss guard to fail to trigger for `!zeroForOne` swaps (token1 in, token0 out) that should have been blocked.

---

### Finding Description

In `_afterSwapOracleStopLoss`, the mid-price used for all per-bin metric calculations is:

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

This is the arithmetic mean. The rest of the protocol uses the geometric mean. `PriceVelocityGuardExtension.beforeSwap()` calls:

```solidity
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
``` [2](#0-1) 

The two metrics computed from `midPriceX64` are:

```solidity
metricT0 = _clampMetric(t0ps + Math.mulDiv(Math.mulDiv(uint256(t1), Q64, midPriceX64), METRIC_SCALE, shares));
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
``` [3](#0-2) 

Because `midPriceX64_arithmetic â‰¥ midPriceX64_geometric` (strictly greater whenever `bid â‰  ask`):

- **`metricT1`** = `t0 * midPrice / Q64 + t1` â†’ **overestimated** (inflated by the excess mid price).
- **`metricT0`** = `t0 + t1 * Q64 / midPrice` â†’ **underestimated** (deflated by the excess mid price).

The stop-loss check for `!zeroForOne` swaps compares `metricT1` against the decayed watermark floor:

```solidity
(uint256 hwm1, bool breach1) = _applyWatermark(metricT1, _decayed(hwmS.token1, decayRate, dt), floorMultiplier);
if (breach1 && !zeroForOne) {
  revert OracleStopLossTriggered(binIdx, false, metricT1, (hwm1 * floorMultiplier) / E6);
}
``` [4](#0-3) 

`_applyWatermark` sets `breached = metric < (hwm * floorMultiplier) / E6`. [5](#0-4) 

An inflated `metricT1` keeps the metric above the floor even when the true geometric-mid value has already breached it. The guard silently passes, and the value-leaking `!zeroForOne` swap executes.

The watermark ratchet then records the inflated `metricT1` as the new high watermark:

```solidity
hwmS.token1 = uint104(hwm1);
``` [6](#0-5) 

This compounds the error: each swap that passes sets a watermark that is too high, making future breaches even harder to detect.

---

### Impact Explanation

The stop-loss guard for `!zeroForOne` swaps (token1 in, token0 out) is systematically weakened. When the pool's true per-share value in token1 terms has fallen below the configured drawdown floor, the arithmetic-mean inflation of `metricT1` can keep the computed metric above the floor, allowing the swap to proceed. This is a direct bypass of the configured loss-prevention control, permitting continued value leakage from LP positions that the extension was deployed to protect. The magnitude of the bypass scales with `(spread/2)Â²` â€” for a 5% spread it is ~0.03%, for a 20% spread it is ~0.5% â€” but the error is systematic and accumulates across every swap in the affected direction.

---

### Likelihood Explanation

Every pool that deploys `OracleValueStopLossExtension` with a non-zero `drawdownE6` is affected on every `!zeroForOne` swap. No special attacker setup is required: any public user executing a `!zeroForOne` swap on such a pool triggers the weakened guard. The effect is larger when the oracle spread is wider (e.g., volatile assets or low-liquidity feeds), which is precisely when the stop-loss protection is most needed.

---

### Recommendation

Replace the arithmetic mean with the same geometric-mean helper used by `PriceVelocityGuardExtension`:

```diff
- uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
+ (uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This aligns the stop-loss metric computation with the protocol-wide convention for mid-price derivation and eliminates the systematic overestimation of `metricT1`.

### Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension`, `drawdownE6 = 50_000` (5%), `decayPerSecondE8 = 0`.
2. Set the oracle to return `bid = 0.9e18` (in Q64.64 units), `ask = 1.1e18`. Arithmetic mid = `1.0e18`; geometric mid = `sqrt(0.9 * 1.1) * 1e18 â‰ˆ 0.9950e18`.
3. Seed the bin with `t0 = 1000`, `t1 = 1000`, `shares = 1000`. Set the watermark `hwm1` to the geometric-mid metric value.
4. Execute a `!zeroForOne` swap. The extension computes `metricT1` using the arithmetic mid (`1.0e18`), yielding a value ~0.5% higher than the true geometric-mid metric. The inflated metric clears the floor check; the swap executes and the watermark ratchets up to the inflated value.
5. Repeat: each swap sets a higher-than-true watermark, progressively widening the gap between the guard's perceived floor and the actual value floor, allowing sustained value leakage that the stop-loss was configured to prevent.
