### Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter ã (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, not the end user. If the pool admin allowlists the router to enable router-mediated swaps for approved users, every user ã including those not individually approved ã can bypass the allowlist by routing through the router.

### Finding Description

**Step 1 ã Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap()` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim as the `sender` parameter in the ABI-encoded call to the extension: [2](#0-1) 

**Step 2 ã The router is always `msg.sender` of `pool.swap()`.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly. The router is the immediate caller; the end user's address is stored only in transient storage for the payment callback and is never forwarded to the pool as `sender`: [3](#0-2) 

The same pattern holds for `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Step 3 ã The extension checks the router's address, not the end user's.**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router address: [4](#0-3) 

**Step 4 ã The bypass.**

A pool admin who wants approved users to be able to swap through the router must allowlist the router:

```
swapExtension.setAllowedToSwap(pool, address(router), true);
```

Once the router is allowlisted, `allowedSwapper[pool][router]` is `true` for every call that arrives through the router, regardless of who the end user is. Any unapproved address can call `router.exactInputSingle(...)` and the extension passes unconditionally.

### Impact Explanation

A curated pool that deploys `SwapAllowlistExtension` to restrict trading to a specific set of counterparties loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. Unauthorized users can execute swaps against the pool's liquidity, defeating the pool admin's access-control policy. This is a direct policy bypass on a curated pool ã the allowlist provides no protection once the router is allowlisted.

### Likelihood Explanation

The scenario is reachable through normal, non-malicious pool administration. A pool admin who wants approved users to be able to use the standard periphery router must allowlist the router; there is no other supported mechanism to pass the end user's identity through the router to the extension. The mistake is structurally induced by the mismatch between the extension's identity model and the router's call pattern.

### Recommendation

1. **Short-term:** Document that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` and must not be used together unless `allowAllSwappers` is set.
2. **Long-term:** Redesign the allowlist check to recover the real user identity. Two options:
   - Have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension` decode and check it when `sender` is a known router.
   - Add a dedicated `realSender` field to the `beforeSwap` hook signature so the pool can propagate the original initiator separately from the immediate caller.
