### Title
Missing Parent Subaccount Health Check After Margin Deduction in `createIsolatedSubaccount` — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

`createIsolatedSubaccount` deducts a user-controlled margin amount from the parent subaccount's quote balance with no subsequent health check. This is structurally identical to the Hyperdrive bug class: an operation that increases a liability (negative quote balance / borrow) on one account without verifying the protocol's solvency invariant is preserved.

---

### Finding Description

When a user creates an isolated subaccount, the margin encoded in the order's `appendix` field is transferred from the parent subaccount to the new isolated subaccount: [1](#0-0) 

```solidity
int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(
        QUOTE_PRODUCT_ID,
        txn.order.sender,
        -margin          // ← parent quote balance decremented
    );
    spotEngine.updateBalance(
        QUOTE_PRODUCT_ID,
        newIsolatedSubaccount,
        margin
    );
}
// ← no getHealth() check on txn.order.sender here
```

The `_isolatedMargin` helper extracts the margin from bits 64–127 of the `appendix`: [2](#0-1) 

The user signs the order and controls this field entirely. The `SpotEngine.updateBalance` call does not enforce a floor on the resulting balance — it permits the balance to go negative (treated as a borrow): [3](#0-2) 

Every other balance-reducing operation in the Clearinghouse enforces a health check immediately after the deduction:

- `withdrawCollateral`: [4](#0-3) 
- `transferQuote`: [5](#0-4) 
- `mintNlp`: [6](#0-5) 
- `burnNlp`: [7](#0-6) 

`createIsolatedSubaccount` is the sole balance-reducing path that omits this check.

Additionally, `matchOrders` — the function that executes the isolated trade after the subaccount is created — uses an `isHealthy` stub that unconditionally returns `true`, providing no compensating check: [8](#0-7) 

---

### Impact Explanation

A user can submit a `CreateIsolatedSubaccount` transaction with a margin value exceeding their available quote balance. The on-chain code will:

1. Reduce the parent's quote balance below zero (creating an uncollateralized borrow).
2. Fund the isolated subaccount with the full margin.
3. Allow the isolated subaccount to trade with that margin.

If the isolated trade is profitable, the user withdraws the profit. If it is unprofitable, the isolated subaccount is liquidated, but the parent is already insolvent — its negative quote balance represents a loss socialized to other depositors via `SpotEngine.socializeSubaccount`. This is a direct asset delta: other depositors' `cumulativeDepositsMultiplierX18` is reduced, permanently diluting their balances. [9](#0-8) 

---

### Likelihood Explanation

The user controls the margin amount by crafting the `appendix` field of a signed order. The sequencer is expected to process valid user-submitted `CreateIsolatedSubaccount` transactions. The on-chain code is the final enforcement layer; the sequencer's off-chain validation is not a substitute for an on-chain health check. Any sequencer-side validation gap, bug, or omission directly exposes this path. All other balance-reducing entrypoints enforce the invariant on-chain, making this omission a clear inconsistency rather than an intentional design choice.

---

### Recommendation

Add an initial health check on the parent subaccount immediately after the margin deduction in `createIsolatedSubaccount`:

```solidity
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
    // ADD:
    require(
        IClearinghouse(clearinghouse).getHealth(
            txn.order.sender,
            IProductEngine.HealthType.INITIAL
        ) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
```

This is consistent with every other balance-reducing operation in the protocol.

### Proof of Concept

```solidity
function test_isolated_margin_no_health_check() external {
    // Parent subaccount has 100 USDC quote balance
    bytes32 parent = ...; // subaccount with 100e18 quote
    
    // User crafts an order with margin = 1000e18 (10x their balance)
    // appendix encodes margin = 1000e18 in bits 64-127
    uint128 appendix = (uint128(1000e6) << 64) | (1 << 8) | 1; // isolated + version
    
    IEndpoint.Order memory order = IEndpoint.Order({
        sender: parent,
        priceX18: ...,
        amount: ...,
        expiration: ...,
        nonce: ...,
        appendix: appendix
    });
    
    // Sequencer processes CreateIsolatedSubaccount
    // On-chain: parent quote goes from +100e18 to -900e18
    // No health check reverts this
    
    // Parent is now insolvent: health < 0
    // Isolated subaccount has 1000e18 to trade with
    // If trade profits, user extracts value; loss is socialized to depositors
    int128 parentHealth = clearinghouse.getHealth(parent, IProductEngine.HealthType.INITIAL);
    assertLt(parentHealth, 0); // passes — parent is insolvent, no revert occurred
}
```
