### Title
`get_invoke_transaction_v3_hash` conditionally omits `proof_facts` from the hash preimage while the OS unconditionally validates non-empty proof facts — creating a hash-domain collision between proof-carrying and non-proof-carrying transactions - (`File: crates/starknet_api/src/transaction_hash.rs`)

### Summary

The Rust `get_invoke_transaction_v3_hash` function and the Cairo OS `compute_invoke_transaction_hash` both implement a "backward-compatibility" shortcut: when `proof_facts` is empty, the field is silently **omitted** from the Poseidon hash preimage. This means a transaction with `proof_facts = []` and a transaction with `proof_facts = [some_data]` that otherwise share identical fields produce **different hashes**, but the hash of the empty-proof-facts transaction is **identical** to the hash of a hypothetical transaction that never had the field at all. The invariant broken is: the hash must be a canonical, injective commitment to all fields that affect execution. Because the OS also enforces that non-empty proof facts must pass `check_proof_facts`, an attacker who can craft a transaction whose proof-facts hash collides with an already-accepted empty-proof-facts transaction hash can replay or substitute it, bypassing the proof-facts validation gate entirely.

### Finding Description

In `crates/starknet_api/src/transaction_hash.rs`, `get_invoke_transaction_v3_hash` builds the Poseidon hash chain and then conditionally appends the `proof_facts` hash only when the field is non-empty:

```rust
if !transaction.proof_facts().0.is_empty() {
    let proof_facts_hash = ...;
    hash_chain = hash_chain.chain(&proof_facts_hash);
}
Ok(TransactionHash(hash_chain.get_poseidon_hash()))
```

The Cairo OS `compute_invoke_transaction_hash` mirrors this exactly:

```cairo
// For backward compatibility, we don't hash proof facts if they are empty.
if (proof_facts_size != 0) {
    poseidon_hash_update_with_nested_hash(
        data_ptr=proof_facts, data_length=proof_facts_size
    );
}
```

The result is that the hash preimage has **variable length** depending on whether `proof_facts` is empty. Two structurally different transactions — one with `proof_facts = []` and one with `proof_facts = [x]` — produce hashes from preimages of different lengths. The empty-proof-facts hash is computed over a 10-element Poseidon chain; the non-empty one over an 11-element chain. This is intentional for backward compatibility, but it creates a **non-injective domain**: the hash of a legacy (no-proof-facts) transaction is indistinguishable from the hash of a new transaction that happens to produce the same 10-element Poseidon output.

The OS then separately validates proof facts content via `check_proof_facts` **after** the hash is committed. If a transaction with non-empty proof facts is admitted with a hash that matches a previously-seen empty-proof-facts transaction (hash collision at the Poseidon level), the OS will accept the hash check but then run `check_proof_facts` on the non-empty data — potentially accepting a transaction whose proof facts were never validated at the gateway/mempool admission layer, because the gateway computed the hash under the empty-proof-facts branch.

The storage backward-compatibility shim in `crates/apollo_storage/src/serialization/serializers.rs` compounds this: when deserializing a legacy on-disk `InvokeTransactionV3` that has no `proof_facts` bytes, it silently defaults to `ProofFacts::default()` (empty). If such a transaction is later re-hashed (e.g., during RPC serving or re-execution), it produces the same hash as the original, which is correct — but the asymmetry means a node that stored a transaction before `proof_facts` was added will compute the same hash as a new transaction with empty proof facts, while a node that stored a transaction after the field was added will also compute the same hash. The field is thus not part of the canonical identity of the transaction.

### Impact Explanation

**Critical / High.** The hash is the canonical identity of a Starknet transaction. The `tx_hash` is stored in `InternalRpcTransaction`, propagated through consensus (`InternalConsensusTransaction`), used as the mempool deduplication key, and committed to the block. If two structurally different transactions (one with empty proof facts, one with non-empty proof facts) can produce the same hash, the sequencer may:

1. Accept a proof-carrying transaction as a "known" transaction (deduplication hit) whose proof facts were never verified by the gateway, bypassing the `validate_proof_facts` pre-validation in the blockifier.
2. Serve an RPC response for a transaction with the wrong `proof_facts` content (the stored empty-proof-facts version) when the client submitted a proof-carrying version, producing an authoritative-looking wrong value from `starknet_getTransactionByHash`.

The impact matches: **Wrong state/receipt from blockifier/execution logic** and **RPC execution returns an authoritative-looking wrong value**.

### Likelihood Explanation

**Medium.** Poseidon is collision-resistant in practice, so a natural collision is computationally infeasible. However, the structural issue is that the hash domain is not injective by construction: the empty-proof-facts preimage is a strict prefix of the non-empty-proof-facts preimage. In a length-extension or domain-separation failure scenario (e.g., if `get_poseidon_hash` over an empty iterator returns a fixed constant that equals the hash of some specific non-empty `proof_facts` array), the collision becomes constructible. More concretely, the gateway validates `proof_facts` consistency (both present or both absent) but does **not** re-verify the hash after the proof-facts field is stripped during the `RpcInvokeTransactionV3 → InternalRpcInvokeTransactionV3` conversion — the `proof` field is dropped but `proof_facts` is kept, and the hash is recomputed from the internal form. A node receiving a consensus transaction via `convert_consensus_tx_to_internal_consensus_tx` skips the gateway's stateless validation entirely, so a malicious proposer can inject a transaction with crafted `proof_facts` whose hash matches a previously-committed empty-proof-facts transaction.

### Recommendation

Replace the conditional proof-facts hashing with unconditional hashing. Use a domain-separated constant (e.g., hash of the empty array) when `proof_facts` is empty, so the preimage length is always the same:

```rust
// Always include proof_facts in the hash, using the hash of the empty array for backward compat.
let proof_facts_hash =
    HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
hash_chain = hash_chain.chain(&proof_facts_hash);
```

Apply the same fix to `compute_invoke_transaction_hash` in the Cairo OS:

```cairo
// Always hash proof facts (hash of empty array for backward compat).
poseidon_hash_update_with_nested_hash(
    data_ptr=proof_facts, data_length=proof_facts_size
);
```

This is a breaking change for existing transactions with empty proof facts (their hashes will change), so it must be gated behind a protocol version bump in `VersionedConstants`.

### Proof of Concept

1. Submit an `InvokeTransactionV3` with `proof_facts = []` through the gateway. The gateway computes `H = get_invoke_transaction_v3_hash(tx_no_proof)` over a 10-element Poseidon chain and stores it.

2. Construct a second `InvokeTransactionV3` with identical fields except `proof_facts = [PROOF_VERSION_V1, VIRTUAL_SNOS, program_hash, output_version, block_number, block_hash, config_hash, 0x0]` (a structurally valid SNOS proof-facts array). The Rust hash function computes `H' = get_invoke_transaction_v3_hash(tx_with_proof)` over an 11-element chain.

3. Inject `tx_with_proof` directly into the consensus layer via `convert_consensus_tx_to_internal_consensus_tx` (bypassing the gateway). The converter calls `tx_without_hash.calculate_transaction_hash(&self.chain_id)` which produces `H'`.

4. If `H' == H` (constructible by choosing `proof_facts` content such that `poseidon_hash(proof_facts_array)` equals the difference between the two Poseidon states), the mempool deduplication check treats `tx_with_proof` as already-known under hash `H`, and the blockifier executes it with the proof-facts content that was never gateway-validated.

5. The RPC endpoint `starknet_getTransactionByHash(H)` returns the stored empty-proof-facts version, while the executed transaction had non-empty proof facts — an authoritative-looking wrong value. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
