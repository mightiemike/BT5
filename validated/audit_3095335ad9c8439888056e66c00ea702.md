Audit Report

## Title
Pool admin sets uncapped per-bin additional fees via `setPoolBinAdditionalFees`, bypassing the global admin fee cap — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` directly to the pool with no upper-bound validation and no timelock, while the analogous global setter `setPoolAdminFees` enforces `maxAdminSpreadFeeE6`/`maxAdminNotionalFeeE8` caps. A pool admin can raise per-bin fees to `type(uint16).max` (65,535 E6 ≈ 6.55%) on the active bin immediately before a user's swap, causing the user to pay materially more input than the global fee cap system is designed to allow, then reset the fee in the same or next block.

## Finding Description

`MetricOmmPoolFactory.setPoolBinAdditionalFees` passes the fee values straight through with no cap check:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool-side handler validates only the bin index:

```solidity
// metric-core/contracts/MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
```

Contrast with `setPoolAdminFees`, which enforces explicit caps before writing:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

The per-bin fee is consumed directly in the swap math as an additive component of the total bin fee:

```solidity
// metric-core/contracts/MetricOmmPool.sol L910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
```

and in `SwapMath.buyToken0InBinSpecifiedOut`/`buyToken0InBinSpecifiedIn`, the total fee (base + per-bin additional) is charged on the user's input:

```solidity
// metric-core/contracts/libraries/SwapMath.sol L409-411
uint256 feeAmountScaled = Math.ceilDiv(amountInScaled * currBinBuyFeeX64, ONE_X64);
amountInScaled += feeAmountScaled;
uint256 protocolFeeAmountScaled = (feeAmountScaled * spreadFeeE6) / 1e6;
```

The `spreadFeeE6` fraction of the elevated fee accrues as protocol/admin surplus (collected via `collectFees`); the remainder stays in the bin as LP fee. Either way, the user pays more input than the global cap system intends to allow.

The hard caps for the global fee system are:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol L44-45
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000;   // 20%
uint24 internal constant HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000; // 1%
```

Per-bin fees are stored as `uint16` with no documented or enforced ceiling:

```solidity
// metric-core/contracts/types/PoolStorage.sol L22-24
uint16 addFeeBuyE6;   // Additional fee for buying token0; 1e6 = 100%
uint16 addFeeSellE6;  // Additional fee for buying token1; 1e6 = 100%
```

`uint16` max = 65,535 → 65,535 / 1,000,000 ≈ **6.55%** additional fee per bin, additive on top of the global spread fee, with no timelock.

## Impact Explanation

This is an admin-boundary break: the pool admin can exceed the intended fee cap enforced by `setPoolAdminFees` by using the uncapped `setPoolBinAdditionalFees` path. The total effective fee on a swap traversing the active bin becomes `spreadFeeE6 + 65535` (in E6 terms), which can exceed the `maxAdminSpreadFeeE6` cap. The user receives materially less output (or pays more input) than the global fee cap system is designed to guarantee. The excess fee is not refunded — it is either retained in the bin (LP fee) or collected as protocol/admin surplus. This constitutes a direct loss of user principal relative to the fee boundary the protocol enforces on the admin role.

## Likelihood Explanation

The pool admin is a semi-trusted role with no timelock on `setPoolBinAdditionalFees`. On chains with a public mempool (e.g., Ethereum L1, non-private L2s), an EOA admin can observe a pending swap and frontrun it with `setPoolBinAdditionalFees(pool, curBin, 65535, 65535)` in a higher-gas transaction, then reset to 0 in the same or next block. No special preconditions are required beyond the admin role. Users who set `amountOutMinimum: 0` (as seen in test fixtures) receive no revert protection. The attack is repeatable and requires no capital.

## Recommendation

1. **Add a cap** in `setPoolBinAdditionalFees` analogous to the global fee cap:
   ```solidity
   uint16 internal constant MAX_BIN_ADDITIONAL_FEE_E6 = <sane_value>; // e.g. 10_000 (1%)
   if (addFeeBuyE6 > MAX_BIN_ADDITIONAL_FEE_E6 || addFeeSellE6 > MAX_BIN_ADDITIONAL_FEE_E6)
       revert BinFeeTooHigh();
   ```
2. **Add a timelock** on `setPoolBinAdditionalFees` consistent with the oracle rotation timelock already present for `proposePoolPriceProvider`, to prevent same-block frontrunning.
3. **Collect accrued fees** before changing per-bin fees (as `setPoolAdminFees` already does) to prevent retroactive fee manipulation of accrued surplus.

## Proof of Concept

```solidity
// Pool admin frontrun scenario
address pool = ...; // pool with active liquidity in bin 0
int8 activeBin = 0;

// Step 1: Admin observes user's exactInputSingle tx in mempool
// Step 2: Admin frontruns with max per-bin fee (no cap, no timelock)
vm.prank(poolAdmin);
factory.setPoolBinAdditionalFees(pool, activeBin, 65535, 65535); // ~6.55% extra

// Step 3: User's swap executes — pays ~6.55% more input than global fee cap implies
uint256 amountOut = router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool, amountIn: 1e18, amountOutMinimum: 0, ...
    })
);
// amountOut is materially lower than oracle price and global spread fee would imply

// Step 4: Admin resets fee
vm.prank(poolAdmin);
factory.setPoolBinAdditionalFees(pool, activeBin, 0, 0);
```
