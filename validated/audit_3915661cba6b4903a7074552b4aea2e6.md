### Title
`WithdrawCollateralV2` Silently Fails in Slow Mode, Breaking Censorship-Resistance Invariant — (`File: core/contracts/EndpointTx.sol`)

### Summary

The Nado slow mode mechanism is the protocol's censorship-resistance escape hatch: when the sequencer is malicious or offline, users submit transactions via `submitSlowModeTransaction()`, wait 3 days, and self-execute via `executeSlowModeTransaction()`. The `WithdrawCollateralV2` transaction type — which supports withdrawals to a custom `sendTo` address — is accepted and fee-charged during submission but silently discarded during execution, because `processSlowModeTransactionImpl` has no handler for it and falls through to `revert()`. This breaks the invariant that users can always recover their collateral through the slow mode path.

### Finding Description

`submitSlowModeTransactionImpl` in `EndpointTx.sol` has no special case for `WithdrawCollateralV2`, so it falls into the generic `else` branch: [1](#0-0) 

This charges the user a `SLOW_MODE_FEE` ($1 USDC) from their wallet and enqueues the transaction. Three days later, when `executeSlowModeTransaction()` fires `processSlowModeTransactionImpl`, the function handles `WithdrawCollateral` (V1) at line 217 but has no branch for `WithdrawCollateralV2`. It falls to the terminal: [2](#0-1) 

The outer `try/catch` in `_executeSlowModeTransaction` silently swallows this revert: [3](#0-2) 

The comment `// try return funds now removed` confirms that a prior refund mechanism was deliberately deleted. The slow mode entry is consumed (`delete slowModeTxs[_slowModeConfig.txUpTo++]`), the user's $1 fee is gone, and no withdrawal occurs.

`WithdrawCollateralV2` is the only withdrawal path that allows a custom `sendTo` address: [4](#0-3) 

V1 (`WithdrawCollateral`) always resolves `sendTo` to `address(uint160(bytes20(sender)))`: [5](#0-4) 

So users who need to withdraw to a cold wallet or any address other than the one embedded in their subaccount bytes32 have no functional slow mode path.

### Impact Explanation

Two concrete impacts:

1. **Direct fund loss**: Any user who submits a `WithdrawCollateralV2` slow mode transaction loses the $1 `SLOW_MODE_FEE` with no withdrawal and no refund. The transaction slot is consumed and cannot be retried with the same queue index.

2. **Broken censorship-resistance invariant**: If the sequencer is malicious and ignores regular sequencer-path transactions, users who need to withdraw to a non-subaccount address (e.g., a hardware wallet, a multisig, a contract) cannot do so through the only permissionless escape hatch. The `sendTo` field in V2 is the sole mechanism for this. The invariant "users can always recover collateral through slow mode" is violated for this class of withdrawals.

### Likelihood Explanation

The trigger path is fully permissionless: any user can call `submitSlowModeTransaction` with a `WithdrawCollateralV2` payload. The fee loss occurs unconditionally whenever a user submits such a transaction. The censorship-resistance failure requires a malicious sequencer, which is the same precondition as M-09 and is the exact scenario slow mode is designed to protect against.

### Recommendation

Add a `WithdrawCollateralV2` handler in `processSlowModeTransactionImpl` mirroring the V1 handler, decoding `SignedWithdrawCollateralV2`, calling `validateSender` on `txn.sender`, and calling `clearinghouse.withdrawCollateral` with the resolved `sendTo` (defaulting to `address(uint160(bytes20(txn.sender)))` when `sendTo == address(0)`):

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateralV2) {
    IEndpoint.WithdrawCollateralV2 memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateralV2)
    );
    validateSender(txn.sender, sender);
    address resolvedSendTo = txn.sendTo == address(0)
        ? address(uint160(bytes20(txn.sender)))
        : txn.sendTo;
    clearinghouse.withdrawCollateral(
        txn.sender,
        txn.productId,
        txn.amount,
        resolvedSendTo,
        nSubmissions
    );
}
```

### Proof of Concept

1. User calls `Endpoint.submitSlowModeTransaction(abi.encodePacked(uint8(TransactionType.WithdrawCollateralV2), abi.encode(signedV2Tx)))`.
2. `submitSlowModeTransactionImpl` charges `SLOW_MODE_FEE` ($1) from the user's wallet and enqueues the transaction. [6](#0-5) 
3. After `SLOW_MODE_TX_DELAY` (3 days), anyone calls `Endpoint.executeSlowModeTransaction()`. [7](#0-6) 
4. `processSlowModeTransactionImpl` is invoked. `WithdrawCollateralV2` matches no branch and hits `revert()`. [2](#0-1) 
5. The `try/catch` swallows the revert. The queue entry is deleted. No withdrawal is issued. The $1 fee is permanently lost. [3](#0-2) 
6. The user has no remaining slow mode path to withdraw to a custom address.
