### Title
Missing Upper-Bound Validation on `l2_gas.max_amount` for Declare Transactions Allows Unbounded Gas Commitment - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator::validate_resource_bounds` function explicitly skips the `max_l2_gas_amount` upper-bound check for `RpcTransaction::Declare` transactions. A user can submit a Declare V3 transaction with `l2_gas.max_amount` set to `u64::MAX` (18,446,744,073,709,551,615), which is ~15,000× the configured block-wide `max_l2_gas_amount` limit of 1,210,000,000. This unbounded value is committed into the transaction hash, propagated through the mempool, and used directly as the initial Sierra gas budget (`initial_sierra_gas()`) during blockifier execution, causing the blockifier to allocate an execution gas budget far exceeding the block's bouncer capacity.

### Finding Description

In `validate_resource_bounds`, the check for `max_l2_gas_amount` is explicitly bypassed for Declare transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The `max_l2_gas_amount` config is set to `1_210_000_000` by default: [2](#0-1) 

For Invoke and DeployAccount transactions, the check is enforced. For Declare, it is not. The test `valid_l2_gas_amount_on_declare` explicitly confirms this bypass is intentional and tested: [3](#0-2) 

The `l2_gas.max_amount` field from the transaction is used directly as the initial Sierra gas budget in `TransactionContext::initial_sierra_gas()`:

```rust
TransactionInfo::Current(CurrentTransactionInfo {
    resource_bounds: ValidResourceBounds::AllResources(AllResourceBounds { l2_gas, .. }),
    ..
}) => {
    l2_gas.max_amount  // <-- directly from the user-supplied field
}
``` [4](#0-3) 

This value is then used to set the execution gas limit in `EntryPointExecutionContext::max_steps`, which derives the Cairo step limit from `max_amount.0 / step_gas_cost`: [5](#0-4) 

The OS Cairo code also reads `resource_bounds[L2_GAS_INDEX].max_amount` directly as the initial gas bound: [6](#0-5) 

The block bouncer's `receipt_l2_gas` capacity is `5,800,000,000`: [7](#0-6) 

A Declare transaction with `l2_gas.max_amount = u64::MAX` would present an initial gas budget of `18,446,744,073,709,551,615` to the execution engine — orders of magnitude above the block bouncer limit.

### Impact Explanation

**Critical — Incorrect fee/gas/bouncer/resource accounting with economic impact.**

A single Declare V3 transaction with `l2_gas.max_amount = u64::MAX` passes all gateway validation (stateless and stateful), enters the mempool, and is handed to the blockifier. The blockifier computes `initial_sierra_gas()` as `u64::MAX`, allocating a step budget of `u64::MAX / step_gas_cost` Cairo steps. The bouncer's `receipt_l2_gas` check operates on *actual* gas consumed (not the user-declared bound), so the transaction may execute normally — but the fee charged is computed from the actual gas used against the user's `max_price_per_unit`, not the inflated bound. The economic distortion is: the sequencer commits to executing a transaction whose declared resource bound is `u64::MAX` gas, which can cause the block builder to miscompute available block capacity, potentially allowing the block to be overfilled or causing the bouncer to make incorrect admission decisions based on the declared (not actual) gas bound.

Additionally, the unbounded `l2_gas.max_amount` is committed into the transaction hash via `calculate_transaction_hash`, meaning the hash is computed over an unconstrained field value that other transaction types would have rejected at the gateway boundary. [8](#0-7) 

### Likelihood Explanation

**High.** Any unprivileged user can submit a Declare V3 transaction via the public RPC gateway. The bypass is explicitly coded and tested. No privileged access is required. The only prerequisite is a valid Sierra contract class (which is already validated for size/version) and a funded account.

### Recommendation

Remove the Declare-specific exemption and apply the same `max_l2_gas_amount` upper-bound check to Declare transactions:

```rust
fn validate_resource_bounds(&self, tx: &RpcTransaction) -> StatelessTransactionValidatorResult<()> {
    if !self.config.validate_resource_bounds {
        return Ok(());
    }
    let resource_bounds = *tx.resource_bounds();
    if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
        return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
    }
    if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
        return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
    }
    // Apply to ALL transaction types, including Declare:
    if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
        return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
    }
    Ok(())
}
``` [9](#0-8) 

If Declare transactions legitimately require a higher L2 gas bound (e.g., for compilation), introduce a separate `max_l2_gas_amount_declare` config parameter with an appropriate upper limit rather than removing the check entirely.

### Proof of Concept

1. Construct a valid Declare V3 transaction with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and any nonzero `max_price_per_unit >= min_gas_price`.
2. Submit to the gateway's `starknet_addDeclareTransaction` endpoint.
3. Observe that `StatelessTransactionValidator::validate_resource_bounds` returns `Ok(())` — the `if let RpcTransaction::Declare(_) = tx {}` branch is taken, skipping the `MaxGasAmountTooHigh` check.
4. The transaction passes stateful validation (balance check uses `max_possible_fee = max_amount * max_price_per_unit`, which saturates to `Fee(u128::MAX)` — this would fail the balance check unless `max_price_per_unit = 0`, but `max_price_per_unit = 0` would fail the `min_gas_price` check; however with `max_price_per_unit = 1` and `max_amount = u64::MAX`, `max_possible_fee = u64::MAX` as a `u128`, which may exceed balance — the practical exploit uses a moderate `max_price_per_unit` with `max_amount` still far above `max_l2_gas_amount`).
5. Concretely: `max_amount = 10 * max_l2_gas_amount = 12_100_000_000` (fits in `u64`, exceeds the 1,210,000,000 limit by 10×) with `max_price_per_unit = min_gas_price = 8_000_000_000`. This passes all checks for Declare but would be rejected for Invoke/DeployAccount.
6. The blockifier receives `initial_sierra_gas() = GasAmount(12_100_000_000)`, which is 2× the block's `receipt_l2_gas` bouncer limit of `5,800,000,000`, causing incorrect block capacity accounting. [1](#0-0) [10](#0-9) [11](#0-10)
