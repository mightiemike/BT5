### Title
ML-DSA-65 Gas Key Host Functions Use Wire Length Instead of Trie-ID Length for Exec Fee Computation — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The gas-key host functions `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, and `promise_batch_action_transfer_to_gas_key` pass the raw `public_key_len` parameter (the borsh wire length of the key as laid out in contract memory) to the exec-fee helpers `gas_key_add_key_exec_fee` / `gas_key_transfer_exec_fee`. For ML-DSA-65 keys this value is **1953 bytes** (1 tag + 1952-byte full pubkey). The exec fee is supposed to price the trie work the receiver must perform, which uses the **on-trie identifier length** of **33 bytes** (1 tag + 32-byte SHA3-256 hash). The transaction path (`permission_exec_fees` / `exec_fee`) correctly calls `public_key.trie_id_len()`. The host-function path never decodes the key before computing the fee, so it uses the wrong representation — the wire form instead of the trie form — producing a ~59× overcharge in the per-byte exec-fee component for every ML-DSA-65 gas key operation issued from a contract.

---

### Finding Description

`PublicKey::len()` returns the borsh-encoded wire size; `PublicKey::trie_id_len()` returns the on-trie identifier size. For ED25519 and SECP256K1 the two values are identical, so the distinction is invisible. For ML-DSA-65 they diverge sharply:

| Method | ED25519 | SECP256K1 | ML-DSA-65 |
|---|---|---|---|
| `len()` | 33 | 65 | **1953** |
| `trie_id_len()` | 33 | 65 | **33** |

The transaction path in `runtime/runtime/src/config.rs::permission_exec_fees` correctly uses `trie_id_len()`:

```rust
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // ← correct: 33 for ML-DSA-65
    gas_key_info.num_nonces,
);
```

The host-function path in `runtime/near-vm-runner/src/logic/logic.rs::promise_batch_action_add_gas_key_with_full_access` uses the raw `public_key_len` parameter instead:

```rust
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // ← wrong: 1953 for ML-DSA-65 borsh bytes
    num_nonces,
);
```

The same pattern appears in `promise_batch_action_add_gas_key_with_function_call` and `promise_batch_action_transfer_to_gas_key` in both `logic/logic.rs` and `wasmtime_runner/logic.rs`.

`gas_key_add_key_exec_fee` feeds `public_key_len` directly into `access_key_key_len(account_id_len, public_key_len)`, which computes the trie key length. With `public_key_len = 1953` instead of `33`, the per-nonce exec fee is inflated by `(1953 − 33) × gas_key_byte_exec_fee` per nonce — approximately 59× the correct amount.

The analogous `gas_key_transfer_exec_fee` call in `promise_batch_action_transfer_to_gas_key` has the same defect:

```rust
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // ← wrong: wire length, not trie-id length
);
```

while the transaction path in `exec_fee` uses:

```rust
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
```

The existing fee-parity integration test (`test_gas_key_fee_parity_full_access` / `test_gas_key_fee_parity_function_call`) only exercises ED25519 keys, where `len() == trie_id_len()`, so the divergence is not caught.

---

### Impact Explanation

**Protocol fee-domain invariant broken.** The exec fee is a protocol-level value that every node must compute identically. The invariant is: exec fee for a gas-key action via host function must equal exec fee for the same action via transaction. This invariant holds for ED25519/SECP256K1 but is violated for ML-DSA-65 once `PostQuantumSignatures` (protocol version 85) activates. A contract adding an ML-DSA-65 gas key with, say, 4 nonces would be charged ~7680 extra bytes of `gas_key_byte` exec gas per call compared to the transaction path. This makes ML-DSA-65 gas keys prohibitively expensive to manage from contracts, silently breaking the host-function API for the new key scheme. Because all nodes run the same code, the overcharge is deterministic and does not cause a consensus split, but it constitutes a broken protocol guarantee: the fee model advertised for ML-DSA-65 gas keys is wrong for the host-function path.

**Impact: Medium** — fee overcharge, broken parity, no fund loss or consensus failure.

---

### Likelihood Explanation

**Likelihood: High** — `PostQuantumSignatures` is stabilized at protocol version 85. Any contract that manages ML-DSA-65 gas keys via host functions after that version activates will trigger the overcharge. The bug is structural (wrong length source), not conditional on any unusual input. The only mitigation is that ML-DSA-65 gas keys are a new feature and contracts using them are not yet widespread.

---

### Recommendation

After `get_public_key` decodes the key, use `public_key.trie_id_len()` (not `public_key_len as usize`) when calling the exec-fee helpers. Apply the fix to all three host functions in both `logic/logic.rs` and `wasmtime_runner/logic.rs`:

```rust
// promise_batch_action_add_gas_key_with_full_access
let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_key = public_key.decode()?;
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_key.trie_id_len(),   // ← use trie-id length
    num_nonces,
);
// then pass decoded_key to append_action_add_gas_key_with_full_access
```

The same substitution applies to `promise_batch_action_add_gas_key_with_function_call` and `promise_batch_action_transfer_to_gas_key`. Extend `test_gas_key_fee_parity_full_access` and `test_gas_key_fee_parity_function_call` to also exercise ML-DSA-65 keys to prevent regression.

### Proof of Concept

**Divergent value:** For an ML-DSA-65 gas key with `num_nonces = 4` and a 10-byte account ID:

- Correct trie key length: `access_key_key_len(10, 33)` = 1 + 10 + 1 + 33 = 45 bytes
- Actual trie key length used by host fn: `access_key_key_len(10, 1953)` = 1 + 10 + 1 + 1953 = 1965 bytes
- Per-nonce exec fee inflated by: `(1965 − 45 + NONCE_VALUE_LEN) × gas_key_byte_exec_fee × 4 nonces`

```rust
pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
```
