### Title
Caller-Controlled `confirmations` Parameter Can Be Set to Zero in `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are permissionless public functions that accept a caller-supplied `confirmations` parameter. Any unprivileged NEAR caller can set `confirmations = 0`, completely bypassing the confirmation-depth protection and causing the function to return `true` for any transaction in any block regardless of how recently it was submitted — including blocks that could still be reorganized away.

### Finding Description

Both SPV proof functions accept `ProofArgs` / `ProofArgsV2`, which include a `confirmations: u64` field supplied entirely by the caller:

```rust
// btc-types/src/contract_args.rs
pub struct ProofArgs {
    pub tx_id: H256,
    pub tx_block_blockhash: H256,
    pub tx_index: u64,
    pub merkle_proof: Vec<H256>,
    pub confirmations: u64,   // ← fully caller-controlled, no minimum enforced
}
```

The only guard in `verify_transaction_inclusion` is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
...
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

When `confirmations = 0`:
- `0 <= gc_threshold` is always true — first guard passes.
- `(any height difference) + 1 >= 0` is always true — second guard passes.
- The function proceeds to Merkle proof verification and returns `true` for any transaction in any block, including one submitted seconds ago with zero depth.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after the coinbase proof check, so it inherits the same flaw:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
```

### Impact Explanation

**Impact: High.** Any downstream NEAR contract that calls `verify_transaction_inclusion[_v2]` to gate a fund release, bridge unlock, or other irreversible action can be exploited. An attacker submits a transaction in a freshly submitted block, immediately calls the verifier with `confirmations = 0`, receives `true`, and triggers the downstream action before the block can be reorganized away. This enables double-spend attacks against any protocol that relies on this light client for finality guarantees.

### Likelihood Explanation

**Likelihood: Medium.** The entry path requires no privilege — any NEAR account can call the public function with `confirmations = 0`. The only prerequisite is that a downstream consumer exists that trusts the return value. Given the contract is explicitly designed as an SPV oracle for bridges and cross-chain applications, such consumers are the intended use case.

### Recommendation

Enforce a protocol-level minimum confirmation depth inside the contract rather than relying on the caller to supply a safe value. For example:

```rust
const MIN_CONFIRMATIONS: u64 = 6; // or a configurable admin-set floor

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    "Confirmations below minimum safe threshold"
);
```

Alternatively, remove `confirmations` from the public API entirely and hard-code or admin-configure the required depth in contract state, so no unprivileged caller can weaken the guarantee.

### Proof of Concept

1. Deploy the contract and submit a single block header via `submit_blocks`.
2. Call `verify_transaction_inclusion` with a valid Merkle proof for a transaction in that block and `confirmations: 0`.
3. Both `require!` guards pass; the function returns `true` despite the block having zero confirmation depth.
4. A downstream bridge contract that calls this function and trusts the result would unlock funds for a transaction that could still be reorganized away.

```rust
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );
```
