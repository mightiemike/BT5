### Title
Deprecated `verify_transaction_inclusion` Bypasses Coinbase Merkle Proof Validation, Enabling 64-Byte Transaction Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two SPV verification entry points: `verify_transaction_inclusion` (v1, deprecated) and `verify_transaction_inclusion_v2`. The v2 function adds a coinbase Merkle proof check specifically to prevent the 64-byte transaction Merkle proof forgery attack. However, v1 remains unrestricted and publicly callable on-chain. Any unprivileged NEAR caller can invoke v1 directly, bypassing the coinbase proof gate entirely and obtaining a `true` verification result for a forged transaction inclusion proof.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces a coinbase Merkle proof validation step before delegating to v1: [1](#0-0) 

It then calls v1 internally: [2](#0-1) 

`verify_transaction_inclusion` (v1) is still a public, unpermissioned `#[pause]`-only function. It performs no coinbase proof validation whatsoever: [3](#0-2) 

The `#[deprecated]` Rust attribute is a **compiler-only warning**. It imposes no runtime restriction. Any NEAR account can call `verify_transaction_inclusion` directly on-chain, completely skipping the coinbase proof gate that v2 adds.

The contract's own warning acknowledges the risk: [4](#0-3) 

The asymmetry is exact: the coinbase proof check is enforced on the v2 path but absent on the v1 path, and both paths are reachable by any unprivileged caller.

---

### Impact Explanation

The 64-byte transaction Merkle proof forgery (documented at https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to craft a 64-byte blob whose double-SHA256 hash collides with an internal Merkle tree node. By supplying this blob as `tx_id` and a valid Merkle path to that internal node, the attacker can make `verify_transaction_inclusion` return `true` for a Bitcoin transaction that does not exist.

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) — or that accepts a caller-chosen method name — will receive a forged `true` result. This corrupts the SPV proof result, which is the core security guarantee of the light client. Concrete downstream impact includes: falsely unlocking bridged assets, falsely confirming payments, or falsely triggering any on-chain action gated on Bitcoin transaction inclusion.

---

### Likelihood Explanation

The 64-byte forgery technique is publicly documented and well-understood. The attacker needs only to:

1. Identify a target downstream contract that calls `verify_transaction_inclusion` (v1).
2. Construct a 64-byte fake transaction whose hash matches an internal Merkle node in any real Bitcoin block already accepted by the light client.
3. Call `verify_transaction_inclusion` directly with the forged `tx_id`, a valid Merkle path to that internal node, and a `tx_block_blockhash` for the real block.

No privileged role, leaked key, or social engineering is required. The entry point is fully open to any NEAR account.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or gate it behind an access-control role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts. The coinbase proof validation must be enforced on every externally reachable SPV verification path, not only on v2.

---

### Proof of Concept

1. Identify a real Bitcoin block already accepted by the light client (its hash is in `mainchain_header_to_height`).
2. Construct a 64-byte fake transaction `fake_tx` such that `SHA256d(fake_tx)` equals an internal Merkle tree node `N` in that block.
3. Build a Merkle path from `N` up to the block's `merkle_root` (all intermediate hashes are real and publicly available from the Bitcoin block).
4. Call on NEAR:
   ```
   verify_transaction_inclusion({
     tx_id: SHA256d(fake_tx),   // hash of the fake 64-byte blob
     tx_block_blockhash: <real accepted block hash>,
     tx_index: <index matching the path>,
     merkle_proof: <real path from N to merkle_root>,
     confirmations: 1
   })
   ```
5. The function returns `true` because `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.block_header.merkle_root` holds — the coinbase proof check that would have caught this forgery is never executed. [5](#0-4)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
