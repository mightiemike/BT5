### Title
`DepositAllowlistExtension` gates on LP position `owner` instead of the actual depositor `sender`, allowing any address to bypass the deposit allowlist — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool." However, its `beforeAddLiquidity` hook silently drops the `sender` parameter and checks the `owner` (LP position recipient) instead. Any unprivileged address can bypass the allowlist by calling `pool.addLiquidity(allowedOwner, …)` with an allowed owner address, depositing tokens into a restricted pool without being on the allowlist.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

- `sender` = `msg.sender` — the actual caller who provides tokens via the swap callback
- `owner` — the LP position recipient, a free parameter chosen by the caller [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` discards `sender` (unnamed first parameter) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

Because `owner` is a free caller-controlled parameter, any address can pass the guard by supplying any address that the pool admin has already allowlisted as `owner`. The actual token depositor (`sender`) is never validated.

For contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper) and ignores `recipient`:

```solidity
function beforeSwap(address sender, address, …) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
``` [4](#0-3) 

The asymmetry confirms the deposit extension has the wrong parameter.

---

### Impact Explanation

- Any address not on the allowlist can deposit tokens into a pool that the admin intended to restrict, violating the pool admin's access-control invariant.
- LP shares are credited to the supplied `owner` (who did not initiate the deposit), while the unauthorized `sender` provides the tokens. The unauthorized depositor loses their tokens into the pool; the named `owner` receives LP shares they did not request and can later `removeLiquidity` to extract them (since `removeLiquidity` enforces `msg.sender == owner`).
- This constitutes an admin-boundary break: an unprivileged path bypasses a configured pool guard, and unauthorized liquidity enters a restricted pool, corrupting the pool admin's intended liquidity composition. [5](#0-4) 

---

### Likelihood Explanation

- The bypass requires no special privilege: any EOA or contract can call `pool.addLiquidity` with a known allowed owner address.
- Allowed owner addresses are discoverable on-chain via `AllowedToDepositSet` events or by reading `allowedDepositor` storage.
- The attacker only needs to implement `IMetricOmmAddLiquidityCallback` to provide tokens in the callback.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension`. [3](#0-2) 

---

### Proof of Concept

```solidity
// Assume pool has DepositAllowlistExtension configured.
// allowedDepositor[pool][alice] = true; bob is NOT on the allowlist.

// Bob calls addLiquidity with owner = alice (an allowed address).
// The extension checks allowedDepositor[pool][alice] → true → passes.
// Bob provides tokens via callback. LP shares go to alice.
// Bob has bypassed the deposit allowlist entirely.

pool.addLiquidity(
    alice,          // owner = allowed address (bypass key)
    0,              // salt
    deltas,
    callbackData,
    extensionData
);
// Result: bob (unauthorized) deposited tokens; alice received LP shares.
// alice can now call removeLiquidity to extract bob's tokens.
``` [6](#0-5)

  mapping(address pool => mapping(address depositor => bool)) public allowedDepositor;

  function setAllowedToDeposit(address pool_, address depositor, bool allowed) external onlyPoolAdmin(pool_) {
    allowedDepositor[pool_][depositor] = allowed;
    emit AllowedToDepositSet(pool_, depositor, allowed);
  }

  function setAllowAllDepositors(address pool_, bool allowed) external onlyPoolAdmin(pool_) {
