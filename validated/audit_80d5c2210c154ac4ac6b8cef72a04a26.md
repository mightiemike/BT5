### Title
Linked Signer Can Force-Mint NLP Tokens to Lock Subaccount Owner's Funds for 4 Days — (`EndpointTx.sol`)

---

### Summary

The `MintNlp` sequencer-path transaction in `EndpointTx.sol` accepts a signature from the **linked signer** (`allowLinkedSigner = true`). A malicious linked signer can therefore sign a `MintNlp` transaction on behalf of the subaccount owner, converting the owner's quote balance into NLP tokens that are subject to a mandatory 4-day lock (`NLP_LOCK_PERIOD`). The owner cannot burn or use those tokens until the lock expires. Compounding this, the `LinkSigner` sequencer-path transaction also accepts a linked-signer signature, allowing the linked signer to first rotate itself to an attacker-controlled address before executing the NLP mint — extending the effective censorship window to 3 days (slow-mode recovery delay) plus 4 days (NLP lock), mirroring the 1-week fund-freeze described in EIG-17.

---

### Finding Description

**Root cause 1 — `MintNlp` allows linked-signer execution**

In `EndpointTx.sol` the sequencer-path handler for `MintNlp` calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the signer via `getLinkedSigner`, meaning the currently registered linked signer's ECDSA signature is accepted as a valid authorisation for the subaccount owner's `MintNlp` action: [2](#0-1) 

When `mintNlp` executes in `Clearinghouse.sol`, it calls `spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, nlpAmount)`, which internally calls `handleNlpLockedBalance`: [3](#0-2) 

`handleNlpLockedBalance` enqueues the minted NLP amount with `unlockedAt = getOracleTime() + NLP_LOCK_PERIOD`. `NLP_LOCK_PERIOD` is hardcoded to 4 days: [4](#0-3) [5](#0-4) 

The `burnNlp` path in `Clearinghouse.sol` enforces that only the **unlocked** balance can be burned: [6](#0-5) 

So once the linked signer mints NLP on behalf of the owner, the converted quote balance is frozen for 4 days.

---

**Root cause 2 — `LinkSigner` also allows linked-signer execution (compounding)**

The sequencer-path `LinkSigner` handler also uses `allowLinkedSigner = true`: [7](#0-6) 

This means the current linked signer can rotate itself to any attacker-controlled address. The slow-mode path for `LinkSigner` correctly restricts execution to the subaccount owner via `validateSender`: [8](#0-7) [9](#0-8) 

But the slow-mode path carries a mandatory 3-day delay before the owner can execute it: [10](#0-9) 

During those 3 days the attacker-controlled replacement signer can repeatedly call `MintNlp`, re-locking any newly unlocked or deposited quote balance.

---

### Impact Explanation

A malicious linked signer (e.g., a compromised API key or a rogue third-party trading bot) can:

1. Optionally rotate the linked signer to a fresh attacker address (via `LinkSigner`, `allowLinkedSigner = true`).
2. Sign a `MintNlp` transaction converting the victim's entire quote balance into NLP tokens.
3. The NLP tokens are locked for 4 days; the victim cannot trade, withdraw, or use that collateral.
4. The victim must submit a slow-mode `LinkSigner` to revoke the signer, but the slow-mode delay is 3 days, during which the attacker can re-mint any newly available balance.

Net effect: the victim's funds are rendered unusable for up to 7 days (3-day slow-mode recovery + 4-day NLP lock), directly mirroring the 1-week fund freeze in EIG-17. The corrupted state is the `nlpLockedBalanceQueues[subaccount]` entry in `SpotEngine`.

---

### Likelihood Explanation

Medium. The precondition is that the subaccount owner has linked a signer — a common operational pattern for programmatic trading. Any linked signer (compromised key, malicious service, or attacker who socially engineers the owner into linking them) can trigger this without any privileged protocol access. The sequencer merely relays the signed transaction; no sequencer compromise is required.

---

### Recommendation

1. **`MintNlp`**: Change `allowLinkedSigner` to `false` in the `validateSignedTx` call for `MintNlp`. NLP minting is a capital-allocation decision that should require the subaccount owner's direct signature, not a delegated signer's.

2. **`LinkSigner`**: Change `allowLinkedSigner` to `false` in the `validateSignedTx` call for `LinkSigner`. Allowing the linked signer to rotate itself creates a privilege-escalation path; only the subaccount owner should be able to change who is linked. [1](#0-0) [7](#0-6) 

---

### Proof of Concept

```
1. Alice owns subaccount 0xAlice000...0001 and links signer address B
   (via slow-mode LinkSigner or sequencer LinkSigner signed by Alice).

2. B (malicious) constructs:
     SignedMintNlp {
       tx: MintNlp { sender: 0xAlice000...0001, quoteAmount: Alice's full balance, nonce: N },
       signature: B's ECDSA sig over the EIP-712 digest,
       oraclePriceX18: current NLP price,
       nlpPoolRebalanceX18: [quoteAmount]
     }
   and submits it to the sequencer.

3. EndpointTx.processTransactionImpl → validateSignedTx(..., allowLinkedSigner=true)
   → getLinkedSigner(0xAlice000...0001) == B  → signature accepted.

4. clearinghouse.mintNlp executes:
   - spotEngine.updateBalance(NLP_PRODUCT_ID, Alice, nlpAmount)
     → handleNlpLockedBalance enqueues { amount: nlpAmount, unlockedAt: now + 4 days }
   - spotEngine.updateBalance(QUOTE_PRODUCT_ID, Alice, -quoteAmount)

5. Alice's quote balance is now 0; her NLP balance is locked until T+4 days.
   She cannot trade, withdraw, or use the collateral.

6. Alice submits slow-mode LinkSigner to revoke B, but it executes only after 3 days.
   During those 3 days B can repeat step 2 for any new deposits.
```
