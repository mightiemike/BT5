Audit Report

## Title
`SwapAllowlistExtension` Per-User Allowlist Bypassed via Router Identity Substitution — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the pool's `msg.sender` — the router contract address when a user routes through `MetricOmmSimpleRouter`. If the router is allowlisted to support standard periphery usage, every unprivileged user can bypass the per-user restriction by calling the public router, because the allowlist check evaluates the router's identity rather than the end user's identity. The pool admin has no configuration that achieves selective per-user allowlisting for router-mediated swaps.

## Finding Description
**Root cause — wrong identity checked:**
`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks `allowedSwapper[msg.sender][sender]`. [1](#0-0) 

**Pool wires `msg.sender` as `sender`:**
`MetricOmmPool.swap` passes `msg.sender` of the `swap()` call as the `sender` argument to `_beforeSwap`. When the user routes through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the end user. [2](#0-1) 

**ExtensionCalling forwards it unchanged:**
`_beforeSwap` in `ExtensionCalling` passes `sender` directly into the encoded call to every registered extension with no mechanism to recover the original initiator. [3](#0-2) 

**Router calls pool directly:**
`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool. No end-user identity is injected into `extensionData`. [4](#0-3) 

**Contrast with `DepositAllowlistExtension` (correct pattern):**
The deposit extension ignores `sender` and checks `owner` — the actual position owner passed explicitly — so it correctly gates the economic actor even through the `MetricOmmPoolLiquidityAdder`. The swap extension has no equivalent end-user argument. [5](#0-4) 

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension` as `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, userA, true)` — only `userA` is intended to swap.
3. Admin calls `setAllowedToSwap(pool, router, true)` — required for `userA` to use the standard periphery.
4. `userB` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. Router calls `pool.swap(recipient, ...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. `SwapAllowlistExtension` checks `allowedSwapper[pool][router] == true` → **PASSES**.
8. `userB`'s swap executes against the LP pool despite never being added to the allowlist.

The wrong value is the `sender` identity evaluated in the allowlist check: it is the router address, not the end user's address, so the intended protection is bypassed entirely.

## Impact Explanation
A pool admin who deploys `SwapAllowlistExtension` to restrict trading to specific counterparties (KYC-verified market makers, institutional partners, closed beta) must allowlist the router for those users to access the pool via the standard periphery. Once the router is allowlisted, any unprivileged EOA or contract can call `MetricOmmSimpleRouter` and execute swaps against the pool, bypassing the per-user gate entirely. LPs in such a pool are exposed to unrestricted swap flow from actors the pool was explicitly designed to exclude, enabling adverse-selection losses against LP principal. The allowlist extension's core invariant — that only explicitly permitted addresses may swap — is broken for all router-mediated swaps.

## Likelihood Explanation
Both required conditions (swap allowlist configured for access control, router allowlisted to support standard periphery usage) are the natural, expected configuration for any restricted pool that still wants to support the standard router UX. The bypass requires no special privilege — any EOA or contract can call `MetricOmmSimpleRouter`. The vulnerability is immediately exploitable upon deployment and repeatable without limit.

## Recommendation
The `SwapAllowlistExtension` must gate the economic initiator, not the intermediary. Two viable approaches:

1. **Pass initiator through `extensionData`**: Require the router to encode the original `msg.sender` in `extensionData`; the extension decodes and checks that address. This requires a convention between router and extension.
2. **Document and enforce direct-call-only**: Document that the allowlist only works for direct pool calls (no router), and have the router expose a separate entry point that pools can gate differently.

The deposit extension's pattern — checking `owner` rather than `sender` — is the correct model. The swap extension needs an equivalent end-user identity that survives router intermediation.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Admin: setAllowedToSwap(pool, userA, true)   // only userA allowed
3. Admin: setAllowedToSwap(pool, router, true)   // router must be allowlisted for userA to use it
4. userB (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle(pool, ...)
   → router calls pool.swap(recipient, ...) with msg.sender = router
   → pool calls _beforeSwap(sender=router, ...)
   → SwapAllowlistExtension: allowedSwapper[pool][router] == true → PASSES
5. userB's swap executes successfully against the LP pool,
   despite never being added to the allowlist.
```

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist only `userA` and the router, then call `exactInputSingle` from `userB` and assert it succeeds (demonstrating the bypass).
