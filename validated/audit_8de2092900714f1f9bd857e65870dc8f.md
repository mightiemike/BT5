### Title
Missing Event Emission in `BaseEngine.updateRisk` for Critical Risk Parameter Changes - (File: `core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine.updateRisk` silently overwrites all per-product risk parameters — `longWeightInitial`, `shortWeightMaintenance`, `shortWeightInitial`, `longWeightMaintenance`, and `priceX18` — without emitting any event. These parameters are the direct arithmetic inputs to every health calculation in the protocol. Traders, liquidators, and off-chain monitoring systems have no on-chain signal that the risk model for any product has changed.

---

### Finding Description

`BaseEngine.updateRisk` is the privileged function through which the owner updates the risk weights and oracle price for any listed product: [1](#0-0) 

It writes directly to the packed `RiskStore` slot: [2](#0-1) 

No event is emitted. Compare this with `_addOrUpdateProduct`, which does emit `AddOrUpdateProduct` when a product is first listed or updated through the governance path: [3](#0-2) 

`updateRisk` is a separate, post-listing path that bypasses that emission entirely. The owner-facing entry points are `ContractOwner.spotUpdateRisk` and `ContractOwner.perpUpdateRisk`, both of which call directly into `updateRisk`: [4](#0-3) 

The risk values written here feed every health computation. `_calculateProductHealth` reads `_risk(productId)` and applies `longWeight`/`shortWeight` to compute the weighted position value: [5](#0-4) 

`Clearinghouse.getHealth` then aggregates these contributions and applies the `spreads` spread-weight bonus: [6](#0-5) 

A second missing-event instance exists in `Clearinghouse.setSpreads`, which silently replaces the packed `spreads` bitmask that controls which (spot, perp) pairs receive the 5× spread-leverage bonus in `getHealth`: [7](#0-6) 

A third instance is `OffchainExchange.updateTierFeeRates`, which rewrites maker/taker fee rates for every product in a tier without emitting an event: [8](#0-7) 

---

### Impact Explanation

- **Traders and liquidators** rely on off-chain indexers to monitor risk parameters. Without events, any tightening of `longWeightInitial` / `shortWeightMaintenance` is invisible until a position is already unhealthy or liquidated. A trader who sized a position against the old weights has no on-chain signal to reduce exposure.
- **NLP liquidity providers** who model protocol risk off-chain cannot detect that the spread-weight configuration (`spreads`) has changed, altering the effective leverage of basis positions they are exposed to.
- **Fee-sensitive market makers** cannot detect tier fee-rate changes without polling every product's fee table, making automated quoting strategies unreliable.
- The `updateRisk` path is the most critical: it can silently change `priceX18` (the oracle seed price for a product) in addition to weights, which directly affects the dollar value assigned to every open position in health checks.

---

### Likelihood Explanation

The owner (`ContractOwner` multisig) is expected to call `spotUpdateRisk` / `perpUpdateRisk` whenever market conditions require risk parameter adjustments — this is a routine operational action, not an exceptional one. The absence of events is a structural gap that is present from deployment and affects every future risk update.

---

### Recommendation

1. Define a `RiskUpdate` event in `BaseEngine` (or `IProductEngine`):
   ```solidity
   event RiskUpdate(uint32 indexed productId, RiskHelper.RiskStore riskStore);
   ```
2. Emit it at the end of `updateRisk`:
   ```solidity
   emit RiskUpdate(productId, riskStore);
   ```
3. Define and emit a `SpreadsUpdate(uint256 spreads)` event in `Clearinghouse.setSpreads`.
4. Define and emit a `TierFeeRatesUpdate(uint32 tier, uint32 productId, int128 makerRateX18, int128 takerRateX18)` event in `OffchainExchange.updateTierFeeRates`.

---

### Proof of Concept

1. Owner calls `ContractOwner.spotUpdateRisk(productId, newRiskStore)`.
2. This calls `SpotEngine.updateRisk(productId, newRiskStore)` → `BaseEngine.updateRisk`.
3. `_risk().value[productId]` is overwritten with tighter weights.
4. No event is emitted on-chain.
5. An off-chain indexer watching for `AddOrUpdateProduct` events sees nothing.
6. A trader whose position was healthy under the old weights is now below initial margin, but receives no on-chain notification. The next sequencer batch that calls `getHealth` on their subaccount will find them unhealthy, potentially triggering liquidation without any observable prior warning.             productId,
            subaccount
        );
        int128 weight = RiskHelper._getWeightX18(risk, amount, healthType);
        health += quoteAmount;

        if (amount != 0) {
            if (weight == 2 * ONE) {
                return -INF;
            }
            health += amount.mul(weight).mul(risk.priceX18);
            emit PriceQuery(productId);
        }
    }

    function getHealth(bytes32 subaccount, IProductEngine.HealthType healthType)
        public
        returns (int128 health)
    {
        ISpotEngine spotEngine = _spotEngine();
        IPerpEngine perpEngine = _perpEngine();

        health = spotEngine.getHealthContribution(subaccount, healthType);
        // min health means that it is attempting to borrow a spot that exists outside
        // of the risk system -- return min health to error out this action
        if (health == -INF) {
            return health;
        }
        health += perpEngine.getHealthContribution(subaccount, healthType);

        uint256 _spreads = spreads;
        while (_spreads != 0) {
            uint32 _spotId = uint32(_spreads & 0xFF);
            _spreads >>= 8;
            uint32 _perpId = uint32(_spreads & 0xFF);
            _spreads >>= 8;

            IProductEngine.CoreRisk memory perpCoreRisk = perpEngine
                .getCoreRisk(subaccount, _perpId, healthType);

            if (perpCoreRisk.amount == 0) {
                continue;
            }

            IProductEngine.CoreRisk memory spotCoreRisk = spotEngine
                .getCoreRisk(subaccount, _spotId, healthType);

            if (
                (spotCoreRisk.amount == 0) ||
                ((spotCoreRisk.amount > 0) == (perpCoreRisk.amount > 0))
            ) {
                continue;
            }

            int128 basisAmount;
            if (spotCoreRisk.amount > 0) {
                basisAmount = MathHelper.min(
                    spotCoreRisk.amount,
                    -perpCoreRisk.amount
                );
            } else {
                basisAmount = -MathHelper.max(
                    spotCoreRisk.amount,
                    -perpCoreRisk.amount
                );
            }

            // spreads have 5x higher leverage than the underlying products.
            // but it's capped at 100x leverage at most.
            int128 existingWeight = (spotCoreRisk.longWeight +
                perpCoreRisk.longWeight) / 2;
            int128 spreadWeight = RiskHelper._getSpreadWeightX18(
                perpCoreRisk,
                spotCoreRisk,
                healthType
            );

            health += basisAmount
                .mul(spotCoreRisk.price + perpCoreRisk.price)
                .mul(spreadWeight - existingWeight);
            emit PriceQuery(_spotId);
            emit PriceQuery(_perpId);
        }
    }
