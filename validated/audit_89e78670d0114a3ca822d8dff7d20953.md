### Title
Bonus checkpoint advancement without payment in `slashRedelegationPosition` causes permanent loss of accrued bonus rewards - (File: x/tieredrewards/keeper/slash.go)

### Summary
In `slashRedelegationPosition`, when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is silently swallowed. Because `processEventsAndClaimBonus` advances the position's bonus checkpoints (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) and decrements validator event reference counts **before** the pool sufficiency check, the subsequent call to `setPositionWithState` persists those advanced checkpoints even though no bonus was paid. The position owner permanently loses all accrued bonus rewards with no recovery path.

### Finding Description
`processEventsAndClaimBonus` in `x/tieredrewards/keeper/claim_rewards.go` performs two categories of state mutation before it checks whether the pool can cover the computed bonus:

1. **Per-event loop (lines 172–199):** For each pending validator event, `pos.UpdateLastEventSeq(entry.Seq)` advances the in-memory sequence pointer and `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` writes a reference-count decrement directly to the store. Both happen unconditionally before the pool check.

2. **Post-loop checkpoints (lines 215–217):** `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` sets `pos.LastBonusAccrual = blockTime` and `pos.UpdateLastKnownBonded(bonded)` updates the bonded-state flag, both in memory, before `sufficientBonusPoolBalance` is called at line 230.

When the pool is insufficient, `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` at line 230–232, and `processEventsAndClaimBonus` returns that error without paying the bonus. At this point the in-memory `pos` already carries fully advanced checkpoints, and the store already carries decremented event reference counts.

Back in `slashRedelegationPosition` (`x/tieredrewards/keeper/slash.go` lines 54–64):

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // error swallowed — execution continues with modified pos
    } else {
        return err
    }
}
```

The function then falls through to `setPositionWithState` (line 71 or 77), which persists the modified `pos` — including the advanced `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded` — to the store. The next call to `processEventsAndClaimBonus` for this position will start from the advanced checkpoints, so the bonus accrual window that was silently skipped is permanently gone.

### Impact Explanation
The corrupted value is `pos.LastBonusAccrual` / `pos.LastEventSeq` / `pos.LastKnownBonded` stored in the `Positions` collection. After the silent swallow, these fields reflect a state in which the bonus was paid, but the owner's bank balance was never credited. The owner can never reclaim the forfeited bonus: the checkpoints that guard against double-claiming (`LastEventSeq` prevents event replay, `LastBonusAccrual` prevents segment replay) now permanently exclude the accrual window. Additionally, the decremented event reference counts may cause validator events to be garbage-collected prematurely, corrupting the event log for any other position that has not yet processed those events.

### Likelihood Explanation
The trigger requires three concurrent conditions: (1) a position owner has executed `MsgTierRedelegate`, creating a `RedelegationMappings` entry; (2) the destination validator is slashed (a normal governance/slashing event); (3) the bonus rewards pool is insufficient at slash time. Condition (3) is realistic whenever the pool is depleted or nearly depleted — a normal operational state. No attacker control is required; the slash is a standard protocol event. Any position owner who has redelegated is exposed whenever the pool runs low.

### Recommendation
Move the checkpoint-advancement calls (`applyBonusAccrualCheckpoint`, `UpdateLastKnownBonded`, and the per-event `UpdateLastEventSeq` / `decrementEventRefCount`) to **after** the successful `SendCoinsFromModuleToAccount` call, or restructure `processEventsAndClaimBonus` so that it returns the computed bonus amount without mutating `pos` or the store, leaving all mutations to the caller only on success. In `slashRedelegationPosition`, if the pool is insufficient and the bonus must be forfeited to avoid chain halt, the position's checkpoints should be left at their pre-call values so the owner can claim the bonus once the pool is replenished — consistent with the atomic-failure guarantee documented for all user-driven paths.

### Proof of Concept
1. Owner calls `MsgTierRedelegate` for position P to validator V2. A `RedelegationMappings[unbondingId → P.Id]` entry is created.
2. The bonus rewards pool is drained (e.g., by other claimants or simply not replenished).
3. V2 is slashed. The staking module fires `BeforeRedelegationSlashed(ctx, unbondingId, sharesToUnbond)`.
4. `slashRedelegationPosition` is called. It reads position P and calls `processEventsAndClaimBonus(ctx, &pos)`.
5. Inside `processEventsAndClaimBonus`: the event loop runs, `pos.LastEventSeq` is advanced, event reference counts are decremented in the store. Then `applyBonusAccrualCheckpoint` sets `pos.LastBonusAccrual = blockTime`. `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`.
6. Back in `slashRedelegationPosition`: the error is matched as `ErrInsufficientBonusPool`, logged, and swallowed. `setPositionWithState(ctx, pos, ...)` is called, persisting the advanced checkpoints.
7. The pool is later replenished. Owner calls `MsgClaimTierRewards` for position P. `processEventsAndClaimBonus` starts from the already-advanced `LastBonusAccrual` and `LastEventSeq` — the entire accrual window from before the slash is permanently excluded.
