Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Complete Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the immediate `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. Any pool admin who allowlists the router to enable router-mediated access inadvertently grants unrestricted swap access to every user of the router, completely defeating the allowlist.

## Finding Description
**Step 1 ŌĆö Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap()` calls `_beforeSwap` with its own `msg.sender` as the `sender` argument:

```solidity
_beforeSwap(
  msg.sender,   // whoever called pool.swap()
  recipient,
  ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value verbatim and forwards it to the extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
``` [2](#0-1) 

**Step 2 ŌĆö SwapAllowlistExtension checks the wrong actor.**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()`. When a user goes through `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ..., params.extensionData
);
``` [4](#0-3) 

The original caller (`msg.sender` of `exactInputSingle`) is stored only in transient callback context for payment purposes and is never forwarded to `pool.swap()` as the `sender`. The pool therefore receives `msg.sender = router`, and the extension evaluates `allowedSwapper[pool][router]` ŌĆö not `allowedSwapper[pool][user]`.

**Step 3 ŌĆö The operational trap.**

A pool admin deploying `SwapAllowlistExtension` to restrict trading faces a dilemma: allowlisted users cannot use the router (the check fails because the router is not allowlisted), so the admin must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, `allowedSwapper[pool][router] == true` passes for every call arriving through the router, regardless of who the end user is. The individual user allowlist is completely bypassed.

No existing guard in `SwapAllowlistExtension`, `ExtensionCalling`, or `MetricOmmPool` recovers the original EOA from the router call. The `recipient` field is also attacker-controlled and cannot serve as a reliable substitute. [5](#0-4) 

## Impact Explanation
A curated pool using `SwapAllowlistExtension` to restrict trading to a known set of addresses is fully bypassed. Any non-allowlisted user can execute swaps against the pool by routing through `MetricOmmSimpleRouter`. This defeats the purpose of the allowlist entirely, allowing unauthorized parties to trade against pool liquidity, extract value at oracle-anchored prices, or interact with pools intended to be private or permissioned. This constitutes a broken core pool functionality (admin-boundary break: pool admin's access control is bypassed by an unprivileged path through the supported periphery router), with direct impact on pool liquidity and fund security. **Severity: High.**

## Likelihood Explanation
`MetricOmmSimpleRouter` is the production-grade entry point for swaps. Any pool admin who deploys `SwapAllowlistExtension` and wants legitimate allowlisted users to be able to use the router must allowlist the router address ŌĆö this is the natural and expected operational configuration. The bypass is therefore reachable on any real curated pool that supports router-based access. No special privileges or unusual conditions are required; any unprivileged user can call `exactInputSingle` on the router. **Likelihood: High.**

## Recommendation
The extension must gate on the economically relevant actor ŌĆö the end user ŌĆö not the intermediary router.

1. **Preferred ŌĆö extend the pool interface to forward the original caller**: The pool (or router) should pass the originating EOA/contract explicitly as `sender`, analogous to how `addLiquidity` separates `sender` (payer) from `owner` (position holder). The router would pass `msg.sender` as an explicit parameter, and the pool would forward it to extensions.

2. **Alternative ŌĆö check `recipient` instead of `sender`**: For swaps, the recipient receives output tokens and is more likely to represent the intended gated party. However, `recipient` is also caller-controlled and can be set to an arbitrary address, so this is a partial mitigation only.

3. **Immediate mitigation**: Document and enforce that `SwapAllowlistExtension` is incompatible with router-mediated swaps and must only be used with direct pool calls, or add a factory-level guard that prevents pools with `SwapAllowlistExtension` from being used with the router.

## Proof of Concept
```
Setup:
  - Pool deployed with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router so users can swap
  - Pool admin does NOT call setAllowedToSwap(pool, alice, true)

Attack:
  1. Alice (non-allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({pool, recipient=alice, ...})
  2. Router calls pool.swap(recipient=alice, ...)  [msg.sender of pool.swap() = router]
  3. Pool calls _beforeSwap(sender=router, recipient=alice, ...)
  4. ExtensionCalling encodes sender=router and calls SwapAllowlistExtension.beforeSwap
  5. Extension evaluates: allowedSwapper[pool][router] == true  ŌåÆ passes
  6. Swap executes; Alice receives output tokens

Result:
  allowedSwapper[pool][alice] was never checked.
  Alice bypasses the curated allowlist entirely via the standard router flow.
```
