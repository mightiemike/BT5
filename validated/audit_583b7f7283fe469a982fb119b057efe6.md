### Title
Fee-on-Transfer Token Deposit Inflates Subaccount Collateral Credit, Draining Endpoint's Accumulated Fee Balance — (`File: core/contracts/EndpointStorage.sol`)

---

### Summary

`EndpointStorage.handleDepositTransfer()` performs two sequential ERC-20 transfers using the same nominal `amount`. For fee-on-transfer tokens, the Endpoint receives less than `amount` from the depositor but unconditionally forwards the full `amount` to the Clearinghouse, consuming the deficit from the Endpoint's own pre-existing token balance. The slow-mode deposit transaction queued on-chain records the original `amount`, so the sequencer later credits the depositor's subaccount with the full nominal collateral — more than was actually paid.

---

### Finding Description

The deposit flow in Nado proceeds as follows:

**Step 1 — `Endpoint.depositCollateralWithReferral()`** calls `handleDepositTransfer` with the caller-supplied `amount`, then enqueues a `SlowModeTx` encoding `DepositCollateral { sender, productId, amount }` using that same original `amount`. [1](#0-0) 

**Step 2 — `EndpointStorage.handleDepositTransfer()`** executes two transfers back-to-back, both using the same `amount`:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);          // pull from user
    safeTransferTo(token, address(clearinghouse), amount); // forward to clearinghouse
}
``` [2](#0-1) 

For a fee-on-transfer token, `safeTransferFrom` delivers only `amount - fee` to the Endpoint. The subsequent `safeTransferTo` then attempts to forward the full `amount` to the Clearinghouse. The Endpoint covers the `fee`-sized shortfall from its own existing token balance — which accumulates from slow-mode fees charged to other users via `chargeSlowModeFee`. [3](#0-2) 

**Step 3 — `Clearinghouse.depositCollateral()`** is eventually called by the sequencer with the original `txn.amount` and credits the depositor's subaccount with the full scaled amount:

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [4](#0-3) 

No balance-before/after measurement is ever taken. The actual received amount is never computed. [5](#0-4) 

---

### Impact Explanation

The invariant broken is: **sum of all subaccount spot balances in SpotEngine ≤ actual token holdings of Clearinghouse**.

A depositor of a fee-on-transfer token receives `amountRealized` collateral credit in the SpotEngine, but the Clearinghouse only received `amount - fee` net new tokens (the `fee` portion was taken from the Endpoint's accumulated slow-mode fee balance). The Endpoint's quote-token reserve is silently drained. Repeated deposits amplify the discrepancy. When other users attempt to withdraw, the Clearinghouse will be short of tokens, causing withdrawals to fail or be underfunded.

**Impact: 3 / 5** — Solvency/accounting corruption; requires a fee-on-transfer token to be listed as a supported collateral product.

---

### Likelihood Explanation

The Endpoint accumulates a quote-token balance from every slow-mode transaction submitted by any user via `chargeSlowModeFee` / `slowModeFees`. This balance is always present in a live deployment. The only additional precondition is that a fee-on-transfer token is listed as a supported spot product. If such a token is ever listed, any unprivileged user calling `depositCollateral` can trigger the bug with no special access.

**Likelihood: 3 / 5** — Conditional on token listing; entry path is fully permissionless once the condition is met.

---

### Recommendation

Measure the actual received amount using a balance snapshot before and after `safeTransferFrom`, and use that measured amount for both the `safeTransferTo` call and the `SlowModeTx` payload. Analogous to the Beanstalk fix:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal returns (uint256 received) {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(this));
    safeTransferFrom(token, from, amount);
    received = token.balanceOf(address(this)) - before;
    safeTransferTo(token, address(clearinghouse), received);
}
```

The caller (`depositCollateralWithReferral`) must then use the returned `received` value when constructing the `DepositCollateral` slow-mode transaction, so the sequencer credits only the actual deposited amount. [1](#0-0) 

---

### Proof of Concept

1. A fee-on-transfer token `T` (1% fee) is listed as a supported spot product in Nado.
2. The Endpoint holds 100 units of `T` accumulated from slow-mode fees.
3. Attacker calls `Endpoint.depositCollateral(subaccountName, productId, 1000)`.
4. `handleDepositTransfer` pulls 1000 `T` from attacker → Endpoint receives 990 `T` (10 taken as fee by token).
5. `handleDepositTransfer` forwards 1000 `T` to Clearinghouse → succeeds by consuming 10 `T` from Endpoint's existing balance.
6. Slow-mode tx is queued with `amount = 1000`.
7. Sequencer processes the tx; `Clearinghouse.depositCollateral` credits attacker's subaccount with `1000 * multiplier` collateral.
8. Attacker paid 990 `T` net but received credit for 1000 `T`. The 10 `T` deficit was stolen from the Endpoint's fee reserve.
9. Repeating this drains the Endpoint's reserve and leaves the Clearinghouse undercollateralized relative to total subaccount balances. [2](#0-1) [6](#0-5)
