### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Merkle Proof Forgery Mitigation — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof. However, the original `verify_transaction_inclusion` (v1) is still a fully public, on-chain-callable function. Rust's `#[deprecated]` attribute is a compile-time warning only — it imposes no runtime restriction. Any unprivileged NEAR caller can invoke v1 directly, completely bypassing the coinbase proof check that is the entire security purpose of v2.

### Finding Description

`verify_transaction_inclusion_v2` guards against the 64-byte Merkle proof forgery attack by first validating a coinbase Merkle proof before delegating to v1: [1](#0-0) 

The security invariant is: *every call that verifies a transaction inclusion must first prove the coinbase is at index 0, which constrains the tree structure and prevents internal-node forgery.*

`verify_transaction_inclusion` (v1) is still `pub` and decorated only with `#[deprecated]` and `#[pause]`: [2](#0-1) 

`#[deprecated]` emits a Rust compiler warning; it does not prevent the function from being called at runtime. The function is exported in the NEAR contract ABI and is callable by any account. It performs no coinbase proof validation: [3](#0-2) 

The structural parallel to the external report is exact:
- **External report**: `SOFT_RESTRICTED_STAKER_ROLE` restriction was added to `deposit` but the same restriction was missing from `withdraw`/`redeem`, which share the same underlying `_withdraw` path.
- **This repo**: The coinbase-proof security check was added to `verify_transaction_inclusion_v2` but is absent from `verify_transaction_inclusion` (v1), which remains a live, unrestricted entry point performing the same verification outcome.

### Impact Explanation

The 64-byte Merkle proof forgery attack (https://www.bitmex.com/blog/64-Byte-Transactions) works because `compute_root_from_merkle_proof` does not distinguish leaf nodes from internal nodes: [4](#0-3) 

An attacker can craft a 64-byte value that is a valid internal Merkle tree node, supply it as `tx_id` with a valid Merkle path, and `verify_transaction_inclusion` will return `true` for a transaction that never existed. Any downstream NEAR contract that calls v1 to gate a bridge withdrawal, asset release, or cross-chain action can be deceived into accepting a forged proof.

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` directly. The function is part of the public contract ABI. Downstream integrators who read the contract interface (rather than the Rust source) will see a callable public method and may use it. The forgery technique is publicly documented and well-understood.

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or add the same coinbase proof validation to it. The cleanest fix is to make v1 private (change `pub fn` to `fn`) since v2 already calls it internally:

```rust
// contract/src/lib.rs
fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    // ... existing body unchanged
}
```

This preserves the internal delegation from v2 while eliminating the unguarded public entry point.

### Proof of Concept

1. Identify a Bitcoin block `B` stored in the contract's mainchain with Merkle root `R` and at least two transactions.
2. Compute an internal Merkle node `N` (64 bytes = two child hashes concatenated) that lies one level above the leaves.
3. Construct a valid Merkle path from `N` to `R` (one step shorter than a leaf proof).
4. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B.hash`, `tx_index` matching `N`'s position, and the constructed path.
5. `compute_root_from_merkle_proof(N, index, path)` returns `R`, which equals `header.block_header.merkle_root`, so the function returns `true` — confirming a transaction that does not exist.

The same call to `verify_transaction_inclusion_v2` would fail at the coinbase proof check (line 358–365), demonstrating that the protection exists in v2 but is fully absent from the still-public v1 endpoint. [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
