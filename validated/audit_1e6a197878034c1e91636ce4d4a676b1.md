### Title
`filledAmounts` Lacks Idempotency Guard and Upper-Bound Cap, Enabling Replay-Inflated Fill Accounting - (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.sol` maintains a `filledAmounts` mapping keyed by order digest to track how much of each order has been matched. The increment `filledAmounts[digest] += amountDelta` is unconditional and unbounded. The only guard — `_validateOrder`'s `order.amount != 0` check — verifies that remaining capacity is non-zero but does not enforce that the sequencer-supplied `takerAmountDelta` fits within that remaining capacity. A sequencer replay of a partial-fill `matchOrders` transaction (e.g., during a restart or rolling deployment) causes `filledAmounts[digest]` to be incremented a second time, and `_updateBalances` is called again for the same fill, corrupting both the user's on-chain balance and the protocol's fee accounting.

---

### Finding Description

`filledAmounts` is declared as a public mapping at line 30: [1](#0-0) 

It is incremented unconditionally after every successful match at lines 832–839: [2](#0-1) 

The only guard against overfilling is inside `_validateOrder`, which subtracts the current `filledAmounts[digest]` from the order's original amount and checks the result is non-zero: [3](#0-2) 

However, immediately after `_validateOrder` returns `true`, the `matchOrders` function **overrides** the validated remaining amount with the sequencer-supplied `txn.takerAmountDelta`: [4](#0-3) 

There is no assertion that `txn.takerAmountDelta <= (order.amount - filledAmounts[digest])`. The `_validateOrder` check is therefore decoupled from the actual fill size applied. The following defenses are all absent:

1. **No cap**: `filledAmounts[digest] + amountDelta` is never bounded by `order.amount`. No `require(filledAmounts[digest] + amountDelta <= order.amount)` exists.
2. **No replay deduplication**: There is no per-match-transaction identifier or deduplication table. `nSubmissions` is a monotonic counter on the `Endpoint` but does not prevent the sequencer from re-submitting the same `matchOrders` payload.
3. **No source-state guard**: The increment fires regardless of whether `filledAmounts[digest]` already equals `order.amount`.

**Concrete replay scenario (partial fill):**

| Step | State |
|---|---|
| Order A: `amount = 100`, `filledAmounts[digestA] = 0` | Initial |
| Sequencer submits `matchOrders` with `takerAmountDelta = 50` | First fill |
| `_updateBalances` credits/debits 50 units; `filledAmounts[digestA] = 50` | After first fill |
| Sequencer restarts and replays the same `matchOrders` payload | Replay |
| `_validateOrder`: `order.amount = 100 - 50 = 50 ≠ 0` → passes | Replay validation |
| `_updateBalances` credits/debits 50 units again; `filledAmounts[digestA] = 100` | After replay |

The user's balance has been updated for 100 units of base and quote, but only 50 units were legitimately matched. The protocol's fee accounting (`collectedFees`) is also incremented twice for the same economic event. [5](#0-4) 

---

### Impact Explanation

**Direct asset corruption**: `_updateBalances` is called twice for the same fill. A taker buying 50 units of base receives 100 units of base and pays 100 units of quote. A maker selling 50 units loses 100 units of base and receives 100 units of quote. The net effect is a double-transfer of assets between counterparties for a single on-chain economic event.

**Fee double-collection**: `updateCollectedFees` is called twice, so `market.collectedFees` accumulates fees for 100 units of notional when only 50 units were traded. The protocol treasury collects fees on phantom volume.

**`filledAmounts` inflation**: Once `filledAmounts[digestA] = 100 = order.amount`, the order is correctly rejected on the next attempt. But the damage — double balance update and double fee — has already occurred.

**Impact: Medium** (balance corruption and fee inflation per replayed fill; bounded per order by `order.amount`).

---

### Likelihood Explanation

The sequencer submits `matchOrders` transactions via `submitTransactions`. Sequencer restarts, rolling deployments, and retry-on-transient-error logic are standard operational behaviors that cause the same transaction batch to be re-submitted. This is the exact analog of the scanner re-scanning a block range described in the reference report. No malicious intent is required — only normal operational replay. The `Endpoint` has no per-`matchOrders` deduplication mechanism.

**Likelihood: Medium** — triggered by sequencer operational replay, not by an unprivileged external attacker directly, but realistic in any production deployment with retry logic.

---

### Recommendation

**Option 1 (preferred — cap + source-state guard):** Add an upper-bound cap and a guard that prevents incrementing a fully-filled order:

```solidity
// In the fill commit block, replace the unconditional increment:
if (taker.order.sender != X_ACCOUNT) {
    int128 originalTakerAmount = signedTaker.order.amount; // pass through
    require(
        filledAmounts[ordersInfo.taker.digest] + ordersInfo.taker.amountDelta
            <= originalTakerAmount,
        ERR_ORDER_OVERFILL
    );
    filledAmounts[ordersInfo.taker.digest] += ordersInfo.taker.amountDelta;
}
```

Also add the missing bound check in `matchOrders` before the override:
```solidity
int128 remainingTaker = signedTaker.order.amount - filledAmounts[ordersInfo.taker.digest];
require(txn.takerAmountDelta <= remainingTaker.abs(), ERR_ORDER_OVERFILL);
```

**Option 2 — match-transaction deduplication:** Introduce a `bytes32 matchId` field in the `MatchOrders` transaction struct (e.g., `keccak256(taker.digest, maker.digest, takerAmountDelta, blockNumber)`). Store processed `matchId`s in a `mapping(bytes32 => bool) processedMatches` and revert on duplicate.

**Option 3 — CAS-style fill:** Change the fill commit to `filledAmounts[digest] = expectedFilled + amountDelta` and require the caller to pass `expectedFilled`, reverting if the current value differs.

Additionally, the `_validateOrder` function should enforce `order.amount - filledAmount > 0` (strictly positive, not just non-zero) to prevent the edge case where `filledAmount > order.amount` produces a negative remaining amount that passes the `!= 0` check.

---

### Proof of Concept

1. Alice places a limit buy order: `amount = 100e18`, `digest = D`.
2. Sequencer submits `matchOrders(taker=Alice, maker=Bob, takerAmountDelta=50e18)`.
3. Contract executes: `_updateBalances` credits Alice +50e18 base, debits -50e18 quote. `filledAmounts[D] = 50e18`. Fees collected on 50e18 notional.
4. Sequencer process crashes and restarts; the same transaction batch is re-submitted.
5. `_validateOrder` for Alice: `order.amount = 100e18 - 50e18 = 50e18 ≠ 0` → **passes**.
6. `taker.order.amount = 50e18` (override). `_updateBalances` credits Alice +50e18 base again, debits -50e18 quote again. `filledAmounts[D] = 100e18`. Fees collected again on 50e18 notional.
7. **Result**: Alice has received 100e18 base and paid 100e18 quote for a 50e18 fill. Bob has delivered 100e18 base and received 100e18 quote. Protocol has collected double fees. All from a single legitimate 50e18 on-chain match. [2](#0-1) [6](#0-5) [7](#0-6)
