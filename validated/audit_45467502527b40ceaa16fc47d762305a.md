### Title
Missing Zero-Address Validation for `builder_address` in `ContextStaticConfig` Allows Silent Fee Burn and Wrong Block Hash — (File: `crates/apollo_consensus_orchestrator_config/src/config.rs`)

### Summary

`ContextStaticConfig::default()` initializes `builder_address` to `ContractAddress::default()` (address `0x0`), and `validate_node_config()` contains no check that rejects a zero `builder_address`. This address flows directly into every proposed block's `ProposalInit.builder` field, which becomes the `sequencer` field in `BlockHeaderWithoutHash`. A node started with the default config (or with `builder_address` left at `0x0`) will silently produce blocks whose sequencer address is `0x0`, causing all fee transfers to be sent to address `0x0` (burned) and the block hash to commit a wrong sequencer value.

### Finding Description

`ContextStaticConfig` is the static configuration for the consensus orchestrator context. Its `Default` implementation sets `builder_address` to `ContractAddress::default()`:

```rust
// crates/apollo_consensus_orchestrator_config/src/config.rs:248-261
impl Default for ContextStaticConfig {
    fn default() -> Self {
        Self {
            ...
            builder_address: ContractAddress::default(),  // = 0x0
            ...
        }
    }
}
```

The config schema also ships with `"0x0"` as the default value:

```json
// crates/apollo_node/resources/config_schema.json:2797-2800
"consensus_manager_config.context_config.static_config.builder_address": {
    "description": "The address of the contract that builds the block.",
    "value": "0x0"
}
```

`validate_node_config()` in `crates/apollo_node_config/src/node_config.rs` performs cross-member validations (component config presence, timeout ordering, validation-only mode) but contains **no check** that `builder_address != 0x0` when the consensus manager is running locally.

At proposal-build time, `builder_address` is read from `self.config.static_config.builder_address` and placed into `ProposalBuildArguments`:

```rust
// crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs:807
builder_address: self.config.static_config.builder_address,
```

This value is then placed into `ProposalInit.builder` (the consensus P2P message) and, upon decision, becomes the `sequencer` field of `BlockHeaderWithoutHash`:

```rust
// crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs:397
let sequencer = SequencerContractAddress(init.builder);
```

The `sequencer` field is used in two critical downstream paths:

1. **Fee transfer recipient** — `execute_fee_transfer` sends fees to `block_context.block_info.sequencer_address`. If this is `0x0`, all transaction fees are transferred to address `0x0` (burned).

2. **Block hash** — The Cairo OS `calculate_block_hash` function hashes `block_info.sequencer_address` into the canonical block hash. A zero sequencer address produces a different (wrong) block hash than the intended one.

### Impact Explanation

Any node operator who starts the sequencer without explicitly overriding `builder_address` (e.g., using the default config or a config file that omits this field) will silently produce blocks with `sequencer = 0x0`. The consequences are:

- **Economic**: All transaction fees for every block are transferred to address `0x0` and permanently burned. The sequencer earns nothing.
- **Wrong block hash**: The block hash committed to consensus and stored on-chain encodes `sequencer_address = 0x0`, diverging from the intended value. Any verifier or L1 contract that checks the block hash against the expected sequencer address will see a mismatch.
- **Wrong RPC state**: `starknet_getBlockWithTxHashes` and similar RPC calls will return `sequencer_address = 0x0` for all affected blocks, which is an authoritative-looking wrong value.

### Likelihood Explanation

The default value is explicitly `0x0` in both the Rust `Default` impl and the JSON config schema. An operator who deploys using the schema defaults without consulting deployment-specific config files (e.g., `consensus_manager_config.json`) will hit this. The production deployment configs do set a real address, but there is no code-level guard preventing a zero value from reaching execution. The analog to the FortaStaking bug is exact: a critical address field has no zero-check at initialization or validation time.

### Recommendation

Add a non-zero validation for `builder_address` in `ContextStaticConfig` using the `validator` crate (already used elsewhere in the codebase), or add an explicit check in `cross_member_validations`:

```rust
// In ContextStaticConfig, add:
#[validate(custom(function = "validate_nonzero_contract_address"))]
pub builder_address: ContractAddress,
```

Or in `cross_member_validations` in `node_config.rs`:

```rust
if let Some(consensus_config) = &self.consensus_manager_config {
    if consensus_config.context_config.static_config.builder_address
        == ContractAddress::default()
    {
        return Err(ConfigError::ComponentConfigMismatch {
            component_config_mismatch:
                "builder_address must not be zero".to_string(),
        });
    }
}
```

Also change the default in `ContextStaticConfig::default()` to a sentinel that forces explicit configuration, or remove the `Default` impl entirely.

### Proof of Concept

1. Start the sequencer node using the default config schema without overriding `consensus_manager_config.context_config.static_config.builder_address`.
2. The node loads `builder_address = 0x0` (from the schema default `"0x0"`).
3. `validate_node_config()` passes — no zero-address check exists.
4. The node proposes a block. `build_proposal` reads `self.config.static_config.builder_address` = `0x0` and sets `ProposalBuildArguments.builder_address = 0x0`.
5. `ProposalInit.builder = 0x0` is broadcast to peers.
6. On decision, `BlockHeaderWithoutHash.sequencer = SequencerContractAddress(0x0)`.
7. `execute_fee_transfer` calls the fee token's `transfer(recipient=0x0, ...)` — fees are sent to address `0x0`.

            | ReactiveComponentExecutionMode::LocalExecutionWithRemoteEnabled => {

        // Validate proposer_idle_detection_delay < batcher_deadline.

        // If idle_delay >= batcher_deadline, idle detection never triggers (hard deadline fires

        if let (Some(batcher_config), Some(consensus_manager_config)) =
