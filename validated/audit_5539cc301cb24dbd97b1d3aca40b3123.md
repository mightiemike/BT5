### Title
Missing Event Emission on `linkedSigners` Mutation via `LinkSigner` Transaction — (`File: core/contracts/EndpointTx.sol`)

### Summary
Both the fast-mode and slow-mode `LinkSigner` transaction handlers in `EndpointTx.sol` mutate the `linkedSigners` mapping without emitting any event. Because `linkedSigners` controls which address may sign on behalf of a subaccount, this is a security-critical state change that off-chain clients and monitoring infrastructure have no on-chain signal to observe.

### Finding Description
`EndpointTx.processTransactionImpl` handles the fast-mode `LinkSigner` path:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(...);
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`EndpointTx.processSlowModeTransactionImpl` handles the slow-mode `LinkSigner` path:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [2](#0-1) 

In both paths, the write to `linkedSigners` completes with no `emit` statement. No `LinkedSignerSet` (or equivalent) event is defined anywhere in the codebase. [3](#0-2) 

The `linkedSigners` mapping is stored in `EndpointStorage` and is the sole on-chain record of delegated signing authority for each subaccount. [4](#0-3) 

Signature validation in `EndpointTx.validateSignature` passes when the recovered address equals either the subaccount owner or the stored `linkedSigner`, meaning the linked signer has full transaction-signing authority over the subaccount. [5](#0-4) 

### Impact Explanation
Any change to `linkedSigners` — including setting a new signer, rotating to a different signer, or clearing the signer — produces no on-chain log entry. Off-chain monitoring systems, user-facing dashboards, and security alerting infrastructure that rely on event logs to track subaccount authorization state receive no signal when this critical state changes. If a subaccount's private key is compromised and used to install a malicious linked signer, there is no event for automated watchers or users to detect the change before the attacker acts under the new signer authority.

### Likelihood Explanation
`LinkSigner` is a standard user-facing operation reachable by any subaccount holder through both the sequencer fast path and the permissionless slow-mode path. Every legitimate linked-signer rotation silently mutates state, making the absence of events a persistent gap rather than an edge case.

### Recommendation
Define and emit a `LinkedSignerSet(bytes32 indexed sender, address indexed signer)` event in both the fast-mode and slow-mode `LinkSigner` handlers immediately after the `linkedSigners` write. This mirrors the fix applied to the `Governed._initialize` analog: emit the relevant event at the point of every sensitive state mutation so that off-chain clients can reliably track authorization changes.

### Proof of Concept
1. User calls `Endpoint.submitSlowModeTransaction` with a `LinkSigner` payload, or the sequencer includes a `LinkSigner` transaction in `submitTransactionsChecked`.
2. `EndpointTx.processSlowModeTransactionImpl` (line 239) or `processTransactionImpl` (line 588) writes `linkedSigners[sender] = newSigner`.
3. The transaction succeeds and the new signer can immediately sign on behalf of the subaccount.
4. Querying the chain's event logs for any `LinkSigner`-related event returns nothing — the state change is invisible to any log-based observer. [6](#0-5) [7](#0-6)
