### Title
Unchecked Promise Transfer Return Value in `submit_blocks` Traps Refund NEAR Tokens — (File: `contract/src/lib.rs`)

---

### Summary

`submit_blocks` issues a `Promise::transfer(refund)` to return overpaid NEAR to the caller but attaches no `.then()` callback to verify the transfer receipt succeeded. In NEAR Protocol, promise receipts execute asynchronously in a subsequent block. If the transfer receipt fails — because the predecessor account no longer exists at execution time — the refund tokens are silently returned to the contract's own balance with no recovery path.

---

### Finding Description

In `submit_blocks`, after computing the storage cost and the excess deposit, the contract issues a bare transfer promise:

```rust
let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id())
        .transfer(refund)
        .into()
} else {
    PromiseOrValue::Value(())
}
``` [1](#0-0) 

The `Promise::transfer()` call creates an asynchronous receipt that executes in the next block. No `.then()` callback is chained, so the contract never inspects the promise result. If the transfer receipt fails, NEAR's runtime returns the tokens to the contract's own balance — but the contract has no handler for this case and no admin withdrawal function anywhere in the codebase. [2](#0-1) 

---

### Impact Explanation

When the transfer receipt fails, the refund amount is credited back to the contract's balance and permanently trapped. The contract's storage-cost accounting uses `env::attached_deposit()` (the current call's deposit), not the contract's total balance, so accumulated trapped tokens are never redistributed to future callers. There is no privileged withdrawal or sweep function in the contract. The broken invariant is: *every yoctoNEAR overpaid by a caller must be returned to that caller*. Violation permanently destroys the caller's funds.

---

### Likelihood Explanation

NEAR promise receipts execute one block after the originating function call. A NEAR account can be deleted by a `DeleteAccount` action in a separate transaction that lands in that intervening block. Concretely:

1. Trusted relayer R calls `submit_blocks` with an overpayment; the function call receipt executes and schedules a transfer receipt back to R.
2. Before the transfer receipt executes (next block), account R is deleted via a `DeleteAccount` transaction.
3. The transfer receipt targets a non-existent account, fails, and the runtime credits the tokens back to the contract.
4. No callback fires; the tokens are permanently trapped.

The `#[trusted_relayer]` gate means the caller must be a registered relayer (or hold `Role::DAO` / `Role::UnrestrictedSubmitBlocks`), which narrows the population but does not eliminate the risk — relayer key rotation, account migration, or operational error can all produce this state. The financial impact scales with the size of the overpayment, which the relayer controls. [3](#0-2) 

---

### Recommendation

Chain a `.then()` callback on the transfer promise and implement a `#[private]` callback method that checks `env::promise_result(0)`. If the result is `PromiseResult::Failed`, either re-attempt the transfer, store the owed amount in contract state for later claim, or panic to roll back the entire call:

```rust
// In submit_blocks:
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id())
        .transfer(refund)
        .then(
            Self::ext(env::current_account_id())
                .on_refund_complete(env::predecessor_account_id(), refund)
        )
        .into()
} else {
    PromiseOrValue::Value(())
}

// New callback:
#[private]
pub fn on_refund_complete(&mut self, recipient: AccountId, amount: NearToken) {
    match env::promise_result(0) {
        PromiseResult::Successful(_) => {}
        _ => env::panic_str("Refund transfer failed"),
    }
}
```

Alternatively, use `env::predecessor_account_id()` synchronously and assert the account exists before issuing the promise, though the callback approach is more robust.

---

### Proof of Concept

1. Deploy the contract (bitcoin feature) and register account `relayer.near` as a trusted relayer.
2. Call `submit_blocks` from `relayer.near` with a deposit 1 NEAR above the required storage cost. The function schedules `Promise::transfer(1 NEAR)` back to `relayer.near` and returns.
3. In the same block (before the transfer receipt executes), submit a `DeleteAccount` transaction from `relayer.near`, redirecting its balance to a beneficiary.
4. The transfer receipt executes in the next block, finds `relayer.near` does not exist, fails silently, and credits 1 NEAR back to the contract's balance.
5. Inspect the contract's balance: it has grown by 1 NEAR. No callback fired. No recovery path exists. [1](#0-0)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );
