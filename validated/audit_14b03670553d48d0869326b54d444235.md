### Title
Pool Admin Bypasses Factory Fee Cap via Uncapped `setPoolBinAdditionalFees` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`MetricOmmPoolFactory` enforces `maxAdminSpreadFeeE6` on the pool admin's global spread fee through `setPoolAdminFees`, but `setPoolBinAdditionalFees` forwards per-bin additional fees directly to the pool with **no cap check**. A pool admin can set `addFeeBuyE6` / `addFeeSellE6` to `uint16.max` (65,535 E6 ≈ 6.55%) on any bin, stacking that amount on top of the already-capped global spread fee and exceeding the factory owner's intended ceiling. This is the direct analog to the external report's wrong-modifier pattern: the factory owner's cap on admin fee power is enforced on one setter but silently absent on another.

### Finding Description

`setPoolAdminFees` correctly enforces the factory owner's cap:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` passes values straight through with no cap:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also applies no cap — it only validates the bin index:

```solidity
// metric-core/contracts/MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The `BinState` stores `addFeeBuyE6` and `addFeeSellE6` as `uint16` fields that are added on top of the global `spreadFeeE6` during swap math: [4](#0-3) 

The factory owner's cap system (`maxAdminSpreadFeeE6`, `maxAdminNotionalFeeE8`) is documented as the mechanism to bound pool admin fee power: [5](#0-4) 

### Impact Explanation

`addFeeBuyE6` and `addFeeSellE6` are `uint16`, so the pool admin can set them to `65,535` (E6 units ≈ 6.55%) per bin with a single call. These values are added on top of the global `spreadFeeE6`, which itself can be at the hard cap of 200,000 E6 (20%). The total effective fee for a targeted bin reaches ≈ 26.55%, directly overcharging every trader whose swap crosses that bin. The factory owner's `maxAdminSpreadFeeE6` cap — the only mechanism intended to bound admin fee extraction — is completely bypassed. Excess fees are collected and split between protocol and admin via `collectPoolFees`, so the pool admin captures a larger-than-authorized share of trader principal on every affected swap. [6](#0-5) 

### Likelihood Explanation

The pool admin is a semi-trusted role explicitly bounded by factory-owner-set caps. The bypass requires only a single permissioned call (`setPoolBinAdditionalFees`) that is already available to the pool admin. No special timing, flash loan, or oracle manipulation is needed. A malicious or compromised pool admin can execute this immediately after pool deployment.

### Recommendation

Add the same cap guard to `setPoolBinAdditionalFees` that `setPoolAdminFees` already applies:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap that the factory owner can tune independently.

### Proof of Concept

1. Factory owner deploys factory with `maxAdminSpreadFeeE6 = 200_000` (20% hard cap).
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` — admin spread fee is now at the cap; any higher value reverts `AdminFeeTooHigh`.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert; per-bin additional fees are written directly to `BinState.addFeeBuyE6 = 65535` and `addFeeSellE6 = 65535`.
4. A trader swaps through bin 0: effective spread fee = `spreadFeeE6 (200,000)` + `addFeeBuyE6 (65,535)` = 265,535 E6 ≈ **26.55%**, exceeding the factory owner's intended 20% ceiling by 6.55 percentage points.
5. The overcharged fee is collected via `collectPoolFees` and flows to the admin fee destination, giving the pool admin unauthorized fee revenue at traders' expense. [7](#0-6)
