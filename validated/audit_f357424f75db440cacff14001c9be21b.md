### Title
Unprotected Initializer on Proxy Implementation Contracts Allows Attacker to Seize Ownership and Destroy Implementation — (`core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`, `core/contracts/SpotEngine.sol`, `core/contracts/PerpEngine.sol`)

---

### Summary

`Endpoint`, `Clearinghouse`, `SpotEngine`, and `PerpEngine` are deployed as `TransparentUpgradeableProxy` instances but their implementation contracts do not call `_disableInitializers()` in a constructor. An unprivileged attacker can directly call `initialize()` on any of these bare implementation addresses, seize ownership, and — if `SELFDESTRUCT` is available on Ink Chain — destroy the implementation contract. Every proxy pointing at that implementation immediately becomes non-functional and all user funds become inaccessible.

---

### Finding Description

OpenZeppelin's upgradeable pattern requires that every implementation contract call `_disableInitializers()` inside a constructor annotated with `/// @custom:oz-upgrades-unsafe-allow constructor`. This permanently locks the `_initialized` version counter on the implementation itself, preventing anyone from calling `initialize()` directly on it.

Several Nado contracts that are deployed behind `TransparentUpgradeableProxy` follow this pattern correctly: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

However, `Endpoint`, `Clearinghouse`, `SpotEngine`, and `PerpEngine` have no such constructor. Their `initialize()` functions are reachable on the bare implementation address: [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) 

For `SpotEngine` and `PerpEngine`, the `initializer` modifier lives inside `BaseEngine._initialize()`, not on the outer `initialize()` wrapper: [10](#0-9) 

This still protects the proxy (which already has `_initialized = 1` after deployment), but it does **not** protect the bare implementation address, whose `_initialized` slot starts at `0` and can be set by the first caller.

---

### Impact Explanation

An attacker who calls `initialize()` directly on the `Endpoint` implementation becomes its `owner`. With ownership of the implementation they can:

1. Call `SELFDESTRUCT` (directly or via a crafted `onlyOwner` path) on the implementation contract. Every `TransparentUpgradeableProxy` pointing at that address then has no code to delegate to. All `submitTransactions`, `depositCollateral`, `withdrawCollateral`, and slow-mode queue operations revert permanently.
2. All user collateral held in `SpotEngine` / `Clearinghouse` becomes inaccessible until the `ProxyManager` owner (multisig) executes `migrateAll` with a new implementation.

The worst case mirrors the EnsoWallet scenario exactly: if the `ProxyManager` owner is slow to respond or governance is delayed, user funds are frozen for an extended period.

`SpotEngine` is the highest-value target because it holds all spot collateral balances for every subaccount. [11](#0-10) 

---

### Likelihood Explanation

The implementation addresses are publicly readable from the proxy's admin storage slot. Any EOA can call `initialize()` on the implementation with arbitrary constructor arguments at zero cost. No privileged access, leaked key, or social engineering is required. The attack is a single transaction.

---

### Recommendation

Add the standard OpenZeppelin guard to every upgradeable implementation contract that is currently missing it:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

This must be added to `Endpoint.sol`, `Clearinghouse.sol`, `SpotEngine.sol`, and `PerpEngine.sol`. Also verify `OffchainExchange.sol`, which similarly shows no constructor in its source.

---

### Proof of Concept

```
1. Deploy Nado as normal. Record the implementation address of Endpoint
   (readable from proxy admin storage slot 0x360894...).

2. From any EOA, call:
     Endpoint(implAddress).initialize(
         address(0),   // _sanctions
         attacker,     // _sequencer  → attacker becomes sequencer
         address(0),   // _offchainExchange
         IClearinghouse(address(0)),
         address(0),   // _verifier
         address(0)    // _endpointTx
     );
   This succeeds because _initialized == 0 on the implementation.
   __Ownable_init() sets owner = attacker.

3. Attacker now owns the Endpoint implementation.
   If SELFDESTRUCT is available on Ink Chain:
     - Deploy a helper contract that calls SELFDESTRUCT on the implementation.
     - Call it via any onlyOwner path or directly.
     - The implementation bytecode is erased.

4. Every call through the Endpoint proxy now reverts (no code at impl address).
   submitTransactions(), depositCollateral(), withdrawCollateral() all fail.
   All user funds are frozen until ProxyManager.migrateAll() is executed.
```
