### Title
Collateral Withdrawal Permanently Blocked at 100% Utilization via `assertUtilization` Hard Cap — (`core/contracts/SpotEngine.sol`)

---

### Summary

`SpotEngine.assertUtilization` enforces a hard 100% utilization cap on every collateral withdrawal. When protocol-wide `totalBorrows` reaches `totalDeposits` for any spot product, all user withdrawals of that asset revert unconditionally, locking collateral with no escape path.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, after reducing the withdrawing user's spot balance, the code calls `spotEngine.assertUtilization(productId)`: [1](#0-0) 

```solidity
spotEngine.updateBalance(productId, sender, amountRealized);
spotEngine.assertUtilization(productId);
```

`assertUtilization` reads the **global** product state and enforces: [2](#0-1) 

```solidity
function assertUtilization(uint32 productId) external view {
    (State memory _state, ) = getStateAndBalance(productId, X_ACCOUNT);
    int128 totalDeposits = _state.totalDepositsNormalized.mul(
        _state.cumulativeDepositsMultiplierX18
    );
    int128 totalBorrows = _state.totalBorrowsNormalized.mul(
        _state.cumulativeBorrowsMultiplierX18
    );
    require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
}
```

The check is evaluated **after** the caller's balance has already been reduced. If the withdrawal pushes `totalDeposits` below `totalBorrows`, the entire transaction reverts. There is no partial-withdrawal path, no emergency bypass, and no owner-only override in `withdrawCollateral`.

**How utilization reaches 100%:**

The spot engine allows negative balances (borrows). `_updateBalanceNormalized` tracks both sides globally: [3](#0-2) 

As users borrow quote (USDC) to fund leveraged spot positions, `totalBorrowsNormalized` grows. Additionally, `socializeSubaccount` — called during bad-debt liquidations — directly reduces `cumulativeDepositsMultiplierX18`, compressing `totalDeposits` without touching `totalBorrows`: [4](#0-3) 

A single large liquidation with bad debt can therefore push utilization from, say, 95% to 100%, after which **every** subsequent withdrawal for that product reverts.

An adversarial path also exists: a well-capitalised attacker deposits collateral, borrows the maximum allowed across many subaccounts to push utilization to 100%, then all other users' withdrawal attempts revert. The attacker pays interest but causes a protocol-wide liquidity freeze.

---

### Impact Explanation

All users holding a positive balance in the affected spot product are unable to withdraw. Because `withdrawCollateral` is the only on-chain exit path for collateral (fast withdrawals in `BaseWithdrawPool.submitFastWithdrawal` pull from the `WithdrawPool`'s own token balance, not the clearinghouse), users whose funds sit in the clearinghouse have no alternative route. A sustained high-utilization state — whether organic or adversarially induced — permanently locks collateral until borrowers repay, which they have no on-chain incentive to do during a crisis. [5](#0-4) 

---

### Likelihood Explanation

Medium. The interest rate model in `_updateState` applies a steep `interestLargeCapX18` above the inflection point to discourage high utilization: [6](#0-5) 

However, the rate model does not prevent utilization from reaching 100% — it only makes it expensive. A bad-debt socialization event (which is a normal part of the liquidation lifecycle) can push utilization past 100% in a single transaction regardless of interest rates. No circuit-breaker or deposit cap prevents this.

---

### Recommendation

1. **Introduce a withdrawal utilization buffer**: Revert withdrawals that would push utilization above a configurable threshold (e.g., 90%), not at 100%. This gives the protocol headroom to respond before a full lock occurs.

2. **Decouple socialization from the utilization check**: When `socializeSubaccount` reduces `totalDeposits`, it should trigger a rebalancing mechanism (e.g., drawing from the insurance fund) to keep utilization below the threshold, rather than silently compressing the deposit side.

3. **Maintain a liquidity reserve**: Keep a fraction of each product's deposits in an unborrowable reserve so that a minimum withdrawal capacity always exists, analogous to the recommendation in H-3 to keep 10% of USDC outside the vault.

---

### Proof of Concept

1. Product `QUOTE_PRODUCT_ID` (USDC) has `totalDeposits = 1,000,000` and `totalBorrows = 950,000` (95% utilization).
2. A liquidation event calls `socializeSubaccount` on an insolvent account with a `−60,000` USDC balance. `cumulativeDepositsMultiplierX18` is reduced, compressing `totalDeposits` to `940,000`. Utilization is now `950,000 / 940,000 > 100%`.
3. Alice calls `withdrawCollateral` for `1 USDC`. `updateBalance` reduces `totalDeposits` to `939,999`. `assertUtilization` checks `939,999 >= 950,000` → **false** → reverts with `ERR_MAX_UTILIZATION`.
4. Alice's collateral is locked. Every other depositor faces the same revert. The only resolution is for borrowers to voluntarily repay, which has no on-chain enforcement. [7](#0-6) [8](#0-7)

    function _updateBalanceNormalized(
        State memory state,
        BalanceNormalized memory balance,
        int128 balanceDelta
    ) internal pure {
        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized -= balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized += balance.amountNormalized;
        }

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        int128 newAmount = balance.amountNormalized.mul(
            cumulativeMultiplierX18
        ) + balanceDelta;

        if (newAmount > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);

        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized += balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized -= balance.amountNormalized;
        }
    }
