Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of real user, allowing allowlist bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the real user. `SwapAllowlistExtension.beforeSwap` therefore checks whether the router is allowlisted rather than whether the actual user is allowlisted. Any unpermissioned user can bypass a pool's swap allowlist by calling the public router, provided the pool admin has allowlisted the router (the expected configuration for router-mediated trading).

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // immediate caller of pool.swap
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` uses that value to look up the allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap`. When `MetricOmmSimpleRouter.exactInputSingle` is used, the router calls `pool.swap` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The real user's address is stored in transient storage via `_setNextCallbackContext` for payment purposes only — it is never forwarded to the pool as the swap originator. The pool sees `msg.sender == router`, so `sender == router` reaches the extension. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][real_user]`.

**Bypass path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists specific users.
2. To let those users trade through the public router, the admin calls `setAllowedToSwap(pool, router, true)`.
3. Any unpermissioned user calls `router.exactInputSingle({pool: pool, ...})`. The extension sees `sender == router`, finds it allowlisted, and passes. The swap executes.

The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all router entry points call `pool.swap` with the router as `msg.sender`. [1](#0-0) [2](#0-1) [3](#0-2) 

## Impact Explanation
The swap allowlist is the primary access-control mechanism for permissioned pools. Bypassing it lets any unpermissioned address trade against a pool that was explicitly restricted. This constitutes a direct admin-boundary break: an unprivileged caller bypasses a factory/pool-level role check via a public router path. Depending on pool configuration this can drain LP value by allowing adversarial or high-frequency traders the pool admin explicitly excluded, violate regulatory or compliance constraints, and allow sandwich or front-running actors the admin had blocked to re-enter. This is a direct loss-of-access-control impact on LP principal and pool integrity.

## Likelihood Explanation
`MetricOmmSimpleRouter` is a public, permissionless contract — any EOA or contract can call it. The bypass requires only that the pool admin has allowlisted the router, which is the natural and expected configuration for any pool that wants its allowlisted users to trade through the standard router. No privileged access, special tokens, flash loans, or oracle manipulation is required. The attacker needs only to call `exactInputSingle` with the target pool address.

## Recommendation
The pool must communicate the original user's address to the extension, not the immediate `msg.sender`. The preferred fix is to add an `originator` parameter to `MetricOmmPool.swap` (defaulting to `msg.sender` for direct calls) and pass it as `sender` to `_beforeSwap`. The router already stores the real payer in transient storage via `_setNextCallbackContext(..., msg.sender, ...)`; it should supply `msg.sender` as the explicit `originator` argument when calling `pool.swap`. This ensures `SwapAllowlistExtension.beforeSwap` receives and checks the real user's address rather than the router's address.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension (beforeSwap order = 1).
  - Pool admin calls setAllowedToSwap(pool, router, true)   // allow router
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  attacker calls router.exactInputSingle({
      pool: pool,
      tokenIn: token0,
      zeroForOne: true,
      amountIn: X,
      ...
  });

Trace:
  router.exactInputSingle
    → pool.swap(recipient, ...) [msg.sender = router]
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router)
          → allowedSwapper[pool][router] == true  ✓ (passes)
      → swap executes, attacker receives output tokens

Result: attacker bypasses the allowlist and executes a swap in a pool
        they were explicitly excluded from.
```
