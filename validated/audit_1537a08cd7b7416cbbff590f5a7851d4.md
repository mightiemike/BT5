### Title
Unrestricted `subaccount` Parameter in `depositCollateralWithReferral` Allows Deposits to Reserved System Accounts — (File: `core/contracts/Endpoint.sol`)

---

### Summary
`depositCollateralWithReferral` is a `public` function that accepts a raw `bytes32 subaccount` from any caller. It validates only the isolated-subaccount marker (`isIsolatedSubaccount`) but performs **no range check** against the protocol's reserved system accounts (`FEES_ACCOUNT`, `X_ACCOUNT`, `N_ACCOUNT`, `NLP_POOL_ACCOUNT_START`). An unprivileged attacker can pass any of these reserved identifiers as the `subaccount` argument, crediting those system accounts with arbitrary collateral balances and corrupting the protocol's internal accounting state.

---

### Finding Description

The function signature and guard logic are:

```solidity
// core/contracts/Endpoint.sol
function depositCollateralWithReferral(
    bytes32 subaccount,
    uint32 productId,
    uint128 amount,
    string memory
) public {
    require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED); // only guard
    address sender = address(bytes20(subaccount));
    requireUnsanctioned(msg.sender);
    requireUnsanctioned(sender);
    ...
    handleDepositTransfer(...);
``` [1](#0-0) 

The sole structural guard is `isIsolatedSubaccount`, which checks only the 3-byte `iso` magic suffix:

```solidity
// core/contracts/libraries/RiskHelper.sol
function isIsolatedSubaccount(bytes32 subaccount) internal pure returns (bool) {
    return uint256(subaccount) & 0xFFFFFF == 6910831; // 0x696F73 = "iso"
}
``` [2](#0-1) 

The protocol defines the following reserved system accounts, none of which carry the `iso` suffix and therefore none of which are blocked by the existing guard:

```solidity
// core/contracts/common/Constants.sol
bytes32 constant FEES_ACCOUNT          = bytes32(0);
bytes32 constant X_ACCOUNT             = 0x...0001;
bytes32 constant N_ACCOUNT             = 0x...0002;
bytes32 constant NLP_POOL_ACCOUNT_START = 0x...0000000100000000;
``` [3](#0-2) 

`N_ACCOUNT` is the NLP counterparty used in every `mintNlp` call:

```solidity
// core/contracts/Clearinghouse.sol
spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, -nlpAmount);
``` [4](#0-3) 

NLP pool accounts (derived from `NLP_POOL_ACCOUNT_START`) are credited/debited during every rebalance:

```solidity
// core/contracts/Clearinghouse.sol
spotEngine.updateBalance(QUOTE_PRODUCT_ID, nlpPools[i].subaccount, nlpPoolRebalanceX18[i]);
``` [5](#0-4) 

Because `address(bytes20(N_ACCOUNT)) == address(0)` and `address(0)` is not on any sanctions list, the `requireUnsanctioned` check also passes, leaving no on-chain barrier.

---

### Impact Explanation

An attacker who calls `depositCollateralWithReferral(N_ACCOUNT, QUOTE_PRODUCT_ID, amount, "")` injects an arbitrary quote balance into `N_ACCOUNT`. Because `N_ACCOUNT` is the sole counterparty that absorbs NLP supply during minting, an artificially inflated quote balance in `N_ACCOUNT` corrupts the accounting invariant that the protocol relies on to track NLP issuance against real collateral. Similarly, depositing directly into NLP pool subaccounts inflates their quote balances ahead of sequencer-driven rebalancing, causing the rebalance delta applied by `_applyNlpRebalance` to diverge from the true pool state. This is the direct analog of the ELF vaddr issue: just as an unchecked `vaddr` allows a segment to overwrite the `SYSTEM` memory region, an unchecked `subaccount` allows a deposit to overwrite reserved system-account state that the protocol treats as internally controlled. [6](#0-5) 

---

### Likelihood Explanation

The entry point is `public`, requires no privilege, and is reachable by any EOA or contract. The attacker only needs to hold the deposited token and call the function with a crafted `bytes32` argument. No sequencer cooperation, governance action, or leaked key is required. The attack is repeatable and cheap relative to the accounting disruption it causes. [7](#0-6) 

---

### Recommendation

Add an explicit allowlist or denylist check in `depositCollateralWithReferral` (and in the slow-mode deposit handler in `EndpointTx.sol`) that rejects any `subaccount` whose value matches a reserved system account:

```solidity
require(
    subaccount != FEES_ACCOUNT &&
    subaccount != X_ACCOUNT    &&
    subaccount != N_ACCOUNT    &&
    uint256(subaccount) < uint256(NLP_POOL_ACCOUNT_START),
    ERR_UNAUTHORIZED
);
```

Additionally, enforce that `address(bytes20(subaccount)) == msg.sender` for all direct user-facing deposit paths, restricting callers to depositing only into their own subaccounts. [3](#0-2) 

---

### Proof of Concept

1. Attacker holds USDC (the quote token).
2. Attacker calls:
   ```solidity
   endpoint.depositCollateralWithReferral(
       N_ACCOUNT,          // 0x...0002 — NLP counterparty
       QUOTE_PRODUCT_ID,   // 0
       1_000_000e6,        // large USDC amount
       ""
   );
   ```
3. `isIsolatedSubaccount(N_ACCOUNT)` → `false` (no `iso` suffix). [2](#0-1) 
4. `address(bytes20(N_ACCOUNT)) == address(0)` — sanctions check passes.
5. `handleDepositTransfer` pulls USDC from the attacker into the clearinghouse.
6. A slow-mode transaction is queued crediting `N_ACCOUNT` with the deposit.
7. The sequencer processes the slow-mode transaction; `N_ACCOUNT` now carries an attacker-injected quote balance that was never part of any legitimate NLP mint/burn cycle, corrupting the NLP counterparty accounting state. [8](#0-7)
