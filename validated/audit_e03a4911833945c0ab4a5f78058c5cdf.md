### Title
`limbs_for_int` Omits Sign Byte in `op_multiply` Cost Accumulator, Causing Undercharged Execution Cost — (`File: src/more_ops.rs`)

### Summary

In `op_multiply`, the running size variable `l0` is updated after each multiplication using `limbs_for_int(&total)`, which computes only the magnitude byte count (`bits().div_ceil(8)`). This omits the mandatory sign byte that CLVM's signed big-endian encoding requires whenever the most significant bit of a positive result is set. The subsequent multiplication step then uses an underestimated `l0` in the quadratic cost formula, causing the VM to charge less cost than the actual byte-size of the intermediate product warrants. This is a direct arithmetic semantic mismatch: an unscaled (magnitude-only) length is used where the true serialized length is required.

---

### Finding Description

`op_multiply` in `src/more_ops.rs` maintains a running byte-length `l0` of the current product to compute the cost of each successive multiplication:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

After each multiplication, `l0` is refreshed:

```rust
l0 = limbs_for_int(&total);
```

`limbs_for_int` is defined as:

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

This returns the number of bytes needed to represent the **magnitude** of the number — it does not account for the sign byte. In CLVM's signed big-endian encoding, any positive integer whose most significant bit is 1 requires an extra leading `0x00` byte. For example:

| Value | `limbs_for_int` | Actual CLVM atom length |
|-------|----------------|------------------------|
| 127   | 1              | 1                      |
| 128   | 1              | 2 (`0x00 0x80`)        |
| 254   | 1              | 2 (`0x00 0xfe`)        |
| 255   | 1              | 2 (`0x00 0xff`)        |
| 256   | 2              | 2                      |
| 32767 | 2              | 2                      |
| 32768 | 2              | 3 (`0x00 0x80 0x00`)   |

The `limb_test_helper` test in the same file explicitly confirms this: `limb_test_helper(&[0, 0xff])` expects `limbs_for_int` to return `1` (stripping the leading zero), while the actual atom length is `2`.

By contrast, the **first** argument's length is obtained via `int_atom(a, arg, "*")?` which calls `a.atom_len(args)` — the true serialized byte length including any sign byte. So `l0` starts correct but becomes wrong after the first multiplication whenever the product lands in a range where the top bit is set.

---

### Impact Explanation

The cost model in CLVM is consensus-critical: every full node must agree on the cost of every program. An attacker who crafts a `*` expression whose intermediate products repeatedly land in the "top-bit-set" range (e.g., 128–255, 32768–65535, 8388608–16777215, …) causes `l0` to be underestimated by 1 byte at each such step. The cost formula charges:

```
(l0_wrong + l1) * 6  +  (l0_wrong * l1) / 128
```

instead of:

```
(l0_correct + l1) * 6  +  (l0_correct * l1) / 128
```

For a chain of multiplications with `l0_correct = N+1` and `l0_wrong = N`, the per-step undercharge is `6 + N/128`. For large intermediate values (e.g., N = 256), this is `6 + 2 = 8` cost units per step. Over many steps, this allows a program to consume more CPU than its declared cost budget permits, potentially enabling resource exhaustion or consensus divergence between nodes running different implementations.

---

### Likelihood Explanation

The trigger is straightforward: any CLVM program invoking `*` with three or more arguments where the first product has its most significant bit set. This is trivially constructible by an attacker (e.g., `(* 0x7f 0x02 ...)`) and requires no special privileges. The fast path (enabled by default, i.e., without `no-fastpath` feature) is the affected code path.

---

### Recommendation

Replace `limbs_for_int` with a function that accounts for the sign byte, matching the actual CLVM serialized atom length. Concretely, after each multiplication, compute `l0` as the number of bytes `new_number(total)` would occupy in the allocator — i.e., use `a.atom_len(a.new_number(total.clone())?)` — or define a corrected helper:

```rust
fn serialized_len_for_int(v: &Number) -> usize {
    let bits = v.bits() as usize;
    if bits == 0 { return 0; }
    let bytes = bits.div_ceil(8);
    // positive numbers whose top bit is set need a sign byte
    if v.sign() == Sign::Plus && bits % 8 == 0 {
        bytes + 1
    } else {
        bytes
    }
}
```

Then replace line 649:

```rust
// Before (wrong):
l0 = limbs_for_int(&total);

// After (correct):
l0 = serialized_len_for_int(&total);
```

### Proof of Concept

**Attacker-controlled CLVM bytes** (pseudocode):
```
(* 0x7f 0x02 0x02 0x02 ... )   ; many factors of 2
```

Step-by-step:
1. First arg `0x7f` (127): `l0 = int_atom → 1` (correct, 1 byte).
2. Multiply by `0x02`: product = 254. `limbs_for_int(254) = 1` (8 bits → 1 byte). But actual CLVM atom length of 254 is **2** bytes (`0x00 0xfe`). `l0` is set to `1` instead of `2`.
3. Next step uses `l0 = 1`, `l1 = 1`: charges `(1+1)*6 + (1*1)/128 = 12`. Correct charge would be `(2+1)*6 + (2*1)/128 = 18`. **Undercharge: 6 cost units.**
4. Product = 508 (`0x01 0xfc`, 2 bytes). `limbs_for_int(508) = 2` (9 bits). Correct. `l0 = 2`.
5. Multiply by `0x02`: product = 1016 (`0x03 0xf8`, 2 bytes). `limbs_for_int(1016) = 2`. Correct.
6. Multiply by `0x02`: product = 2032 (`0x07 0xf0`, 2 bytes). Correct.
7. Multiply by `0x02`: product = 4064 (`0x0f 0xe0`, 2 bytes). Correct.
8. Multiply by `0x02`: product = 8128 (`0x1f 0xc0`, 2 bytes). Correct.
9. Multiply by `0x02`: product = 16256 (`0x3f 0x80`, 2 bytes). Correct.
10. Multiply by `0x02`: product = 32512 (`0x7f 0x00`, 2 bytes). Correct.
11. Multiply by `0x02`: product = 65024 (`0xfe 0x00`... wait, 65024 = 0xfe00, top bit set → needs sign byte → 3 bytes `0x00 0xfe 0x00`). `limbs_for_int(65024) = 2` (16 bits). But actual length = **3**. `l0` set to `2` instead of `3`. **Undercharge again.**

The pattern repeats every time the product crosses a power-of-2 boundary where the top bit becomes set. An attacker can engineer a long chain of such crossings to accumulate a meaningful cost undercharge. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

pub fn op_multiply(
    a: &mut Allocator,
    mut input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost: Cost = MUL_BASE_COST;
    let mut first_iter: bool = true;
    let mut total: Number = 1.into();
    let mut l0: usize = 0;
    while let Some((arg, rest)) = a.next(input) {
        input = rest;
        if first_iter {
            (total, l0) = int_atom(a, arg, "*")?;
            if l0 > 256 {
                return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
            }
            first_iter = false;
            continue;
        }

        cost += MUL_COST_PER_OP;
        #[cfg(not(feature = "no-fastpath"))]
        match a.node(arg) {
            NodeVisitor::Buffer(buf) => {
                let l1 = buf.len() as u64;
                if l1 > 256 {
                    return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
                }
                cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
                cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
                check_cost(cost, max_cost)?;

                total *= number_from_u8(buf);
            }
            NodeVisitor::U32(val) => {
                let l1 = len_for_value(val) as u64;
                cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
                cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
                check_cost(cost, max_cost)?;

                total *= val;
            }
            NodeVisitor::Pair(_, _) => {
                Err(EvalErr::InvalidOpArg(
                    arg,
                    "Requires Int Argument: *".to_string(),
                ))?;
            }
        }
        #[cfg(feature = "no-fastpath")]
        {
            let (n1, l1) = int_atom(a, arg, "*")?;
            let l1 = l1 as u64;
            if l1 > 256 {
                return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
            }
            cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
            cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
            check_cost(cost, max_cost)?;

            total *= n1;
        }
        l0 = limbs_for_int(&total);
        if l0 > 1024 {
            return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
        }
    }
    let total = a.new_number(total)?;
    Ok(malloc_cost(a, cost, total))
}
