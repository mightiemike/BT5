### Title
Silent Checkpoint Advancement on Insufficient Bonus Pool During Redelegation Slash — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary

`slashRedelegationPosition` calls `processEventsAndClaimBonus` with a pointer to the position. That function mutates `pos.LastEventSeq`, `pos.LastBonusAccrual`, and `pos.LastKnownBonded` **before** checking pool balance. When the pool is empty, `ErrInsufficientBonusPool` is returned with the position already mutated. `slashRedelegationPosition` swallows that error (logs only) and then calls `setPositionWithState` with the mutated position, permanently persisting the advanced checkpoints without having paid any bonus.

### Finding Description

**Step 1 — Mutation before pool check in `processEventsAndClaimBonus`:**

Inside the event loop, `pos.UpdateLastEventSeq(entry.Seq)` is called on every iteration, and `k.decrementEventRefCount` is called in the store: [1](#0-0) 

After the loop, `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` are called unconditionally: [2](#0-1) 

Only **after** all these mutations does the function check pool sufficiency: [3](#0-2) 

Because `processEventsAndClaimBonus` takes `pos *types.PositionState` (a pointer), all mutations to `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded` are visible to the caller when the function returns an error. [4](#0-3) 

**Step 2 — Error swallowed in `slashRedelegationPosition`:** [5](#0-4) 

When `ErrInsufficientBonusPool` is returned, execution continues with `pos` already carrying the advanced checkpoints.

**Step 3 — Mutated state persisted:** [6](#0-5) 

`setPositionWithState` writes the mutated `pos` (with advanced `LastEventSeq` and `LastBonusAccrual`) to the store: [7](#0-6) 

**Step 4 — Event ref counts already decremented:**

`decrementEventRefCount` is called inside the loop before the pool check, so the validator events may be garbage-collected even though bonus was never paid: [8](#0-7) 

### Impact Explanation

The position owner permanently loses all bonus rewards accrued during the pre-slash bonded segment. After the hook fires:
- `LastEventSeq` is advanced past the events that covered the bonded period.
- `LastBonusAccrual` is set to the current block time.
- Event ref counts are decremented; events may be pruned.

Any subsequent call to `ClaimTierRewards` or `processEventsAndClaimBonus` will find no pending events and no accrual gap to pay, so the bonus is irrecoverably lost. This is a direct, permanent loss of funds owed to the position owner.

### Likelihood Explanation

The `RewardsPoolName` module account can reach zero balance through normal operation (pool not topped up, or drained by prior claims). A redelegation slash is a standard Cosmos SDK staking event triggered by evidence submission — no special privileges are required. The combination is realistic in production.

### Recommendation

Snapshot the position's checkpoint fields before calling `processEventsAndClaimBonus`, and restore them if `ErrInsufficientBonusPool` is returned, so that the position's `LastEventSeq` and `LastBonusAccrual` are not advanced when no bonus is paid. Alternatively, restructure `processEventsAndClaimBonus` to perform the pool balance check before mutating any checkpoint fields on the position.

### Proof of Concept

1. Create a tier position and commit a delegation to a bonded validator.
2. Begin a redelegation (so a redelegation mapping entry exists for the unbonding ID).
3. Drain `RewardsPoolName` to zero.
4. Trigger `BeforeRedelegationSlashed` hook with the unbonding ID.
5. Assert: `pos.LastEventSeq` has advanced and `pos.LastBonusAccrual` equals the current block time, but no bonus coins were transferred to the owner.
6. Fund the pool and call `ClaimTierRewards` for the position.
7. Assert: no additional bonus is owed for the skipped period — the rewards are permanently lost.
