Audit Report

## Title
SwapAllowlistExtension Checks Immediate Caller Instead of End User, Enabling Allowlist Bypass via Public Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which the pool binds to `msg.sender` of `pool.swap()` — the immediate caller. When swaps are routed through `MetricOmmSimpleRouter`, the router is the immediate caller, so the extension checks the router's address rather than the end user's. Any pool admin who allowlists the router to support router-mediated swaps for permitted users simultaneously grants every unpermitted user the ability to bypass the individual allowlist by calling the same public router.

## Finding Description

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, passing its own `msg.sender` as the `sender` argument to the extension: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

Here `msg.sender` is the pool and `sender` is the pool's `msg.sender` — the immediate caller. When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)` directly, the router contract is the pool's `msg.sender`: [3](#0-2) 

So the extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. The `FullMetricExtensionTest` confirms this binding: the test allowlists `address(callers[0])` (the `TestCaller` wrapper contract that calls the pool), not `users[0]` (the human address): [4](#0-3) 

`DepositAllowlistExtension.beforeAddLiquidity`, by contrast, ignores the `sender` parameter and checks `owner` — the LP position owner — correctly identifying the economic beneficiary regardless of who the immediate caller is: [5](#0-4) 

This creates a structural asymmetry: the deposit guard checks the right actor; the swap guard checks the wrong one for router-mediated flows.

## Impact Explanation

A pool admin who configures `SwapAllowlistExtension` to restrict swaps to a curated set of users faces an impossible configuration dilemma:

- **Do not allowlist the router** → all router-mediated swaps revert, even for permitted users. The primary public swap entrypoint is unusable on this pool.
- **Allowlist the router** → the extension passes for any `sender = router`, regardless of who called the router. Every non-permitted user bypasses the individual allowlist by calling `MetricOmmSimpleRouter.exactInput/exactOutput`, which is a public, permissionless contract.

There is no configuration that simultaneously allows permitted users to swap through the router and blocks non-permitted users from doing the same. Consequence: unauthorized users trade in a curated pool, extracting fees, causing price impact, or violating regulatory/compliance restrictions the pool admin intended to enforce. This is a direct policy bypass with fund-level consequences (unauthorized fee extraction, unauthorized price impact on LP positions). This meets the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" and "Broken core pool functionality" impact criteria.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary public swap entrypoint in the periphery layer and is permissionless.
- Any pool that configures `SwapAllowlistExtension` and also wants to support router swaps (the common case) must allowlist the router, triggering the bypass.
- The attacker needs no special privileges: they call the public router with a valid swap, and the extension passes because the router is allowlisted.
- The bypass is repeatable and requires no setup beyond calling the public router.

## Recommendation

The extension must check the address that economically benefits from the swap, not the intermediate relay contract. Two approaches:

1. **Extension-level fix**: Mirror the deposit allowlist design — accept a user-supplied identity from `extensionData` that the router populates with `msg.sender` (the actual end user) before forwarding. The extension then checks that identity instead of `sender`.

2. **Pool-level fix**: Have `pool.swap()` accept an explicit originator parameter (the end user) rather than using `msg.sender`, and have the router forward its own `msg.sender` as that parameter. The extension then checks the true originator.

Either way, the invariant must be: the identity checked by `SwapAllowlistExtension` is the address that economically benefits from the swap, not the intermediate contract that relays it.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` as `extension2` and `beforeSwap` order set to `2`.
2. Pool admin calls `swapExtension.setAllowedToSwap(pool, address(router), true)` — necessary to allow any router-mediated swap.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
5. Router calls `pool.swap(...)` → pool passes `sender = address(router)` to `_beforeSwap` → extension checks `allowedSwapper[pool][router]` → `true` → swap succeeds.
6. Attacker has swapped on a pool they were explicitly not permitted to access.

The `DepositAllowlistExtension` does not share this flaw because it checks `owner` (the LP beneficiary) rather than the immediate caller, confirming the swap extension's actor binding is the root cause. [6](#0-5) [1](#0-0) [7](#0-6)
