### Title
Fork Retarget Difficulty Computed from Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

`get_header_by_height` unconditionally reads from `mainchain_height_to_header`. When a fork block falls on a retarget boundary and the fork diverged *before* the start of that retarget interval (`first_block_height`), `get_next_work_required` fetches the mainchain block at `first_block_height` rather than the fork's own ancestor at that height. The resulting `bits` check is performed against a timestamp that does not belong to the fork's chain history, violating the invariant that difficulty for a fork block must be derived from the fork's own ancestors.

---

### Finding Description

**`store_fork_header` never writes to `mainchain_height_to_header`:** [1](#0-0) 

Fork blocks are stored only in `headers_pool`. The height-indexed map is mainchain-only.

**`get_header_by_height` always reads from `mainchain_height_to_header`:** [2](#0-1) 

There is no fork-aware ancestor walk. Any call to `get_header_by_height` during fork validation returns the *mainchain* block at that height.

**`get_next_work_required` (Bitcoin) calls `get_header_by_height` for the retarget start block:** [3](#0-2) 

`first_block_height = prev_block_header.block_height - (difficulty_adjustment_interval - 1)`. If the fork diverged at any height ≤ `first_block_height`, the block returned here is the mainchain block, not the fork's ancestor.

The same pattern exists in Litecoin: [4](#0-3) 

And in Dogecoin, where the developers left an explicit TODO acknowledging the issue: [5](#0-4) 

**`check_pow` enforces `expected_bits == block_header.bits`:** [6](#0-5) 

The `expected_bits` value is derived from the wrong timestamp, so the contract enforces the wrong difficulty for the fork block at the retarget boundary.

---

### Impact Explanation

If an attacker submits a fork chain that diverges at height H_fork ≤ `first_block_height` for a retarget at height H_retarget:

- The contract computes `expected_bits` for the fork's retarget block using `T_mainchain[first_block_height]` instead of `T_fork[first_block_height]`.
- The attacker controls the fork block timestamps. By setting `T_fork[first_block_height]` to be very close to `T_fork[H_retarget - 1]` (short apparent interval → harder difficulty), the mainchain's earlier timestamp makes the contract compute a *longer* apparent interval → *easier* difficulty.
- The contract accepts the fork block at H_retarget with bits that the real Bitcoin network would reject.
- If this fork accumulates enough chainwork to trigger a reorg (`submit_block_header_inner` line 563), the contract's canonical chain diverges from Bitcoin consensus. SPV proofs (`verify_transaction_inclusion`) issued against this chain are invalid. [7](#0-6) 

---

### Likelihood Explanation

For **Bitcoin/Litecoin mainnet** (2016-block intervals): the attacker must mine 2016+ blocks at mainnet difficulty to diverge before `first_block_height`. This is computationally infeasible without 51% hash power.

For **Bitcoin/Litecoin testnet** (`pow_allow_min_difficulty_blocks = true`): min-difficulty blocks are permitted when block time exceeds `2 * pow_target_spacing`. An attacker can submit min-difficulty fork blocks with spaced timestamps, making the full 2016-block fork chain trivially constructable. The retarget manipulation then applies at the next boundary.

For **Dogecoin** (per-block Digishield after height 145,000): `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `first_block_height = prev_height - 1`. The fork only needs to diverge one block before the retarget, which is every block. The window is always open, but the per-block adjustment limits the magnitude of manipulation per block.

The `skip_pow_verification` flag, if set `true` in any deployment, makes this trivially exploitable with no mining at all. [8](#0-7) 

---

### Recommendation

Replace `get_header_by_height` (mainchain-only lookup) with a fork-aware ancestor walk in `get_next_work_required`. Starting from `prev_block_header`, walk backwards via `get_prev_header` (which follows `prev_block_hash` through `headers_pool` and is fork-correct) exactly `difficulty_adjustment_interval - 1` steps to reach the true fork ancestor at `first_block_height`. This is O(interval) but is already the correct algorithm used by Bitcoin Core's `GetAncestor`. Alternatively, store the fork ancestor hash at `first_block_height` in `ExtendedHeader` at submission time.

---

### Proof of Concept

```
State setup (Bitcoin testnet, skip_pow_verification = false, pow_allow_min_difficulty_blocks = true):
  - Initialize contract at height 0 with genesis block.
  - Submit mainchain blocks 1..4031 (normal cadence, retargets at 2016 and 4032).
    - Mainchain block at height 2016 has timestamp T_main_2016.

Fork construction:
  - Fork diverges at height 2000 (before first retarget at 2016).
  - Submit fork blocks 2001..2015 with timestamps spaced > 2*pow_target_spacing
    → each accepted as min-difficulty (proof_of_work_limit_bits).
  - Submit fork block at height 2016 (retarget boundary):
    → first_block_height = 2016 - 2015 = 1
    → get_header_by_height(1) returns mainchain block at 1 (same ancestor, correct).
    → Fork retarget at 2016 is computed correctly.
  - Submit fork blocks 2017..4031 with timestamps spaced > 2*pow_target_spacing
    → accepted as min-difficulty.
  - Submit fork block at height 4032 (second retarget boundary):
    → first_block_height = 4031 - 2015 = 2016
    → get_header_by_height(2016) returns MAINCHAIN block at 2016 (timestamp T_main_2016)
    → BUT fork's ancestor at 2016 is a FORK block (timestamp T_fork_2016 >> T_main_2016)
    → actual_time_taken = T_fork_4031 - T_main_2016  (inflated)
    → contract computes easier bits than fork's own history requires
    → fork block at 4032 is accepted with bits B_wrong < B_correct

Assert:
  - Contract accepted fork block at 4032 with bits derived from T_main_2016.
  - Real Bitcoin network would require bits derived from T_fork_2016.
  - B_wrong encodes an easier target than B_correct.
  - check_target passed with the wrong difficulty.
```
