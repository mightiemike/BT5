### Title
No Mechanism to Withdraw Freed Storage Deposits Accumulated by GC — (`contract/src/lib.rs`)

### Summary
`submit_blocks` is `#[payable]` and requires callers to attach NEAR tokens to cover on-chain storage costs for new block headers. A garbage-collection routine (`run_mainchain_gc`) continuously removes old headers, freeing that storage. In NEAR Protocol, freed storage stake is automatically returned to the contract account's free balance. Because the contract exposes no withdrawal function, these freed NEAR tokens accumulate permanently in the contract's balance with no way for the owner or DAO to recover them.

### Finding Description
`submit_blocks` collects a deposit, stores new headers, runs GC, then computes the **net** storage delta and refunds the excess to the caller: [1](#0-0) 

The net-delta calculation uses `saturating_sub`, so it only captures the storage increase of the current call: [2](#0-1) 

`run_mainchain_gc` removes headers from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`: [3](#0-2) 

When those entries are removed, the NEAR runtime automatically unlocks the corresponding storage stake back into the contract's free balance. That freed balance is never forwarded to any party — there is no `withdraw`, `drain`, or owner-transfer function anywhere in the contract. [4](#0-3) 

`run_mainchain_gc` is also callable as a standalone public method (not just from within `submit_blocks`), meaning GC can free storage without any accompanying deposit refund path at all: [5](#0-4) 

### Impact Explanation
Every header removed by GC was originally paid for by a relayer in a prior `submit_blocks` call. Once removed, the storage stake for that header is unlocked into the contract's free balance and is irrecoverable. With `gc_threshold = 52_704` (≈ 1 year of Bitcoin blocks) and each 80-byte header costing roughly `80 × storage_byte_cost` yoctoNEAR, the steady-state freed balance grows continuously. The DAO role and all privileged roles have no path to retrieve these tokens. Funds are permanently locked.

### Likelihood Explanation
This is certain in normal operation. The relayer submits headers continuously; GC runs on every `submit_blocks` call and also via direct `run_mainchain_gc` calls. Every GC cycle that removes headers paid for in a prior call silently increases the irrecoverable balance. No adversarial action is required — routine protocol operation is sufficient.

### Recommendation
Add a privileged withdrawal function restricted to the DAO or owner role, for example:

```rust
pub fn withdraw_balance(&mut self, amount: NearToken) {
    // Only DAO may call
    require!(
        self.acl_has_role(Role::DAO, env::predecessor_account_id()),
        "Unauthorized"
    );
    Promise::new(env::predecessor_account_id()).transfer(amount);
}
```

Alternatively, track the total storage freed by GC and return it to the original depositor, though this requires per-call accounting.

### Proof of Concept
1. Relayer calls `submit_blocks` with 100 headers and a deposit of `D` NEAR covering their storage cost. Contract stores 100 headers; `D` NEAR is locked as storage stake.
2. On the next `submit_blocks` call, `run_mainchain_gc` removes 50 of those headers. The NEAR runtime unlocks the storage stake for those 50 headers into the contract's free balance.
3. The `diff_storage_usage = env::storage_usage().saturating_sub(initial_storage)` calculation reflects only the net change of the current call. The freed tokens from step 2 remain in the contract's free balance.
4. No function exists to transfer this balance out. Repeat indefinitely — the contract's irrecoverable free balance grows with every GC cycle. [6](#0-5)
