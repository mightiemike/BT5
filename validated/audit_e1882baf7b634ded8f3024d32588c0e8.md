### Title
Order Signature Replay Across Contract Redeployments via Incorrect `verifyingContract` in EIP-712 Domain Separator — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.getDigest` constructs an EIP-712 domain separator that uses `address(uint160(productId))` — a product-ID-derived address — as the `verifyingContract` field instead of `address(this)`. This means the domain separator does not bind to the actual `OffchainExchange` contract deployment. Order signatures are therefore replayable across any redeployment of `OffchainExchange` that uses the same `productId` and `chainId`, bypassing the `filledAmounts` cancellation/fill-tracking state that is reset on redeployment.

---

### Finding Description

`OffchainExchange.getDigest` builds the EIP-712 domain separator as follows:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(uint160(productId))   // ← product ID cast to address, NOT address(this)
    )
);
``` [1](#0-0) 

The EIP-712 standard requires the `verifyingContract` field to be the address of the signing contract so that signatures are bound to a specific deployment. Here, `address(uint160(productId))` is a deterministic, deployment-independent value. For any given `productId` and `chainId`, the domain separator is identical regardless of which `OffchainExchange` instance is deployed.

The `_checkSignature` function that consumes this digest performs no additional deployment-binding check — it only recovers the signer and compares against the subaccount owner or linked signer:

```solidity
function _checkSignature(
    bytes32 subaccount,
    bytes32 digest,
    address linkedSigner,
    bytes memory signature
) internal view virtual returns (bool) {
    address signer = ECDSA.recover(digest, signature);
    return
        (signer != address(0)) &&
        (signer == address(uint160(bytes20(subaccount))) ||
            signer == linkedSigner);
}
``` [2](#0-1) 

Fill-tracking state lives in `filledAmounts`, a mapping keyed by digest:

```solidity
mapping(bytes32 => int128) public filledAmounts;
``` [3](#0-2) 

On redeployment, this mapping is empty. Because the domain separator is identical across deployments, a digest produced on deployment A is identical to the digest produced on deployment B for the same order parameters. Any previously signed order — including ones that were fully filled, cancelled, or expired — can be replayed on the new deployment.

---

### Impact Explanation

An attacker who observed a user's signed order on a prior deployment can submit that same `(order, signature)` pair to a freshly deployed `OffchainExchange`. The `filledAmounts[digest]` entry is zero on the new deployment, so the order passes fill-amount checks. The signature is cryptographically valid because the domain separator is unchanged. The result is an unauthorized trade executed against the user's account, causing direct asset loss (incorrect base/quote balance deltas applied via `_updateBalances`). [4](#0-3) 

The `matchOrders` path is the concrete entry point — it is called by the sequencer via `EndpointTx.processTransactionImpl` and reaches `_checkSignature` through `getDigest`: [5](#0-4) 

---

### Likelihood Explanation

Redeployment of `OffchainExchange` is a realistic operational event for an upgradeable protocol (bug fixes, feature upgrades, emergency migrations). The attacker requires only: (1) a previously broadcast signed order (observable on-chain or off-chain via the sequencer), and (2) the ability to submit a `MatchOrders` transaction to the sequencer after redeployment. No privileged access is needed. The sequencer processes `MatchOrders` transactions submitted by any party.

---

### Recommendation

Replace `address(uint160(productId))` with `address(this)` in the domain separator construction inside `getDigest`:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(this)   // bind to this specific contract deployment
    )
);
``` [1](#0-0) 

Alternatively, use the inherited `_hashTypedDataV4` from `EIP712Upgradeable` (which already uses `address(this)` and `block.chainid` correctly) instead of manually constructing the domain separator. The contract already calls `__EIP712_init("Nado", "0.0.1")` in `initialize`, so `_hashTypedDataV4` is available and correct. [6](#0-5) 

---

### Proof of Concept

1. User Alice signs an order: `Order(sender=Alice, priceX18=3000e18, amount=100e18, expiration=T+1h, nonce=5, appendix=0)` for `productId=2` on deployment A of `OffchainExchange`. The domain separator is `keccak256(TYPE_HASH, nameHash, versionHash, chainId, address(uint160(2)))`.

2. The order is fully matched on deployment A. `filledAmounts[digest] = 100e18`.

3. The protocol redeploys `OffchainExchange` as deployment B (same `productId=2`, same `chainId`). `filledAmounts` is empty.

4. Attacker submits a `MatchOrders` transaction to the sequencer pairing Alice's old `(order, signature)` as taker against a fresh maker order.

5. `getDigest(2, aliceOrder)` on deployment B produces the **identical** digest as on deployment A (same `address(uint160(2))`, same `chainId`, same struct fields).

6. `_checkSignature` recovers Alice's address from the signature — valid. `filledAmounts[digest] == 0` — passes fill check.

7. Alice's position is modified without her consent, resulting in unauthorized asset transfer. [7](#0-6)
