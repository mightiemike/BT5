Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of end user, enabling full allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `sender`, which is `msg.sender` at the pool call boundary. When `MetricOmmSimpleRouter` intermediates a swap, `msg.sender` at the pool is the router contract, not the end user. A pool admin who allowlists the router to enable periphery access simultaneously grants every address the ability to swap, fully neutralizing the curated pool's access control.

## Finding Description
`MetricOmmPool.swap` captures `msg.sender` and passes it as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value as the first positional argument of `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the immediate pool caller: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the `msg.sender` at the pool boundary: [4](#0-3) 

The result: when the admin allowlists the router (`allowedSwapper[pool][router] = true`), the extension evaluates `allowedSwapper[pool][router]` for every caller who routes through `MetricOmmSimpleRouter`, regardless of who that caller is. The asymmetry with `DepositAllowlistExtension` is structural — `beforeAddLiquidity` receives `owner` as an explicit parameter distinct from `msg.sender`, so the real depositor identity is preserved through the router: [5](#0-4) 

No analogous `swapper` parameter exists in the swap path, making the bypass unavoidable when the router is allowlisted.

## Impact Explanation
A pool admin configuring a curated pool with `SwapAllowlistExtension` and allowlisting the router (the expected action to enable approved users to use the standard periphery) inadvertently grants every address swap access. Unauthorized traders can extract LP value at oracle-derived prices, causing direct loss of LP principal. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break via unprivileged path" impact criteria.

## Likelihood Explanation
The trigger is a standard, expected admin action: allowlisting the router so that approved users can use the supported periphery path. No malicious setup is required. Any non-allowlisted user who routes through `MetricOmmSimpleRouter` instead of calling the pool directly bypasses the check immediately. The periphery is the primary user-facing entry point, making this path highly reachable and repeatable.

## Recommendation
Add a `swapper` parameter to `IMetricOmmPoolActions.swap` (mirroring `owner` in `addLiquidity`) that the pool passes through to the extension. `MetricOmmSimpleRouter` sets this to `msg.sender` before calling the pool. `SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[pool][swapper]` against the real user identity rather than the immediate caller.

## Proof of Concept
```solidity
// Pool admin sets up a curated pool with SwapAllowlistExtension.
swapAllowlist.setAllowedToSwap(address(pool), approvedUser, true);
// Admin allowlists the router so approvedUser can use the periphery.
swapAllowlist.setAllowedToSwap(address(pool), address(router), true);

// attacker is NOT allowlisted.
// Direct pool call reverts:
vm.prank(attacker);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(attacker, false, int128(1000), type(uint128).max, callbackData, "");

// Routing through MetricOmmSimpleRouter succeeds:
// pool.swap() sees msg.sender = router → allowedSwapper[pool][router] = true → passes.
vm.prank(attacker);
router.exactInputSingle(/* pool, attacker, zeroForOne, amount, ... */);
// attacker successfully swaps in the curated pool, bypassing the allowlist.
```
