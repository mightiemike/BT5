### Title
Integer Multiplication Overflow in `op_unknown` Wraps Cost to Zero, Enabling Undercharged Execution — (`File: src/more_ops.rs`)

---

### Summary

`op_unknown` in `src/more_ops.rs` computes a final cost by multiplying a pre-checked cost by an attacker-controlled multiplier derived from opcode bytes. The multiplication is performed on `u64` with no overflow guard. In Rust release builds, `u64` overflow wraps silently. An attacker can craft opcode bytes and arguments so that the product wraps to zero (or a value ≤ `u32::MAX`), causing the operation to be credited with zero cost. The post-multiplication guard `if cost > u32::MAX` is bypassed because the wrapped value is small. This is the direct analog of the external report's rounding-to-zero: both involve attacker-controlled arithmetic inputs that cause a security-critical integer calculation to produce an incorrectly small result, bypassing a downstream guard.

---

### Finding Description

In `op_unknown` (`src/more_ops.rs`, lines 160–267), the cost of an unknown opcode is computed in two stages:

**Stage 1 — pre-multiplication cost** (lines 209–256):

```rust
let mut cost = match cost_function {
    0 => 1,
    1 => { /* ARITH_BASE_COST + per-arg + per-byte */ }
    2 => { /* MUL_BASE_COST + per-op + linear + quadratic */ }
    3 => { /* CONCAT_BASE_COST + per-arg + per-byte */ }
    _ => 1,
};
```

**Stage 2 — multiplier application** (lines 258–266):

```rust
assert!(cost > 0);
check_cost(cost, max_cost)?;          // ← bounds cost ≤ max_cost BEFORE multiply
cost *= cost_multiplier + 1;          // ← u64 × u64, NO overflow check
if cost > u32::MAX as u64 {           // ← guard checks AFTER potential wrap
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is decoded from the opcode prefix bytes via `u32_from_u8`, so it is fully attacker-controlled and can be up to `u32::MAX = 4,294,967,295`, making `cost_multiplier + 1 = 2^32`.

`check_cost(cost, max_cost)` is called **before** the multiplication. In Chia's block-validation context, `max_cost` is `11,000,000,000`, which is larger than `2^32 = 4,294,967,296`. Therefore, `cost` can legally be `2^32` after the pre-multiplication stage.

When `cost = 2^32` and `cost_multiplier + 1 = 2^32`:

```
cost × (cost_multiplier + 1) = 2^32 × 2^32 = 2^64
```

In Rust release mode, `u64` arithmetic wraps modulo `2^64`, so the result is `0`. The guard `if cost > u32::MAX` evaluates `0 > 4,294,967,295 = false`, so the function returns `Ok(Reduction(0, nil))` — **zero cost**.

---

### Impact Explanation

The returned `Reduction(0, nil)` cost is added to the running total in `run_program` (`src/run_program.rs`, line 522):

```rust
cost += match op {
    Operation::Apply => self.apply_op(cost, effective_max_cost - cost)?,
    ...
};
```

A zero-cost operation does not advance the running total. An attacker can embed many such unknown opcodes in a single CLVM program, each executing for zero cost, while the running total never approaches `max_cost`. This allows the attacker to execute an unbounded number of operations within a single block's cost budget.

**Concrete impact:**
- **Undercharged execution / consensus DoS**: Nodes validating the block spend CPU time proportional to the actual work done, but the cost meter never trips. A single block can contain arbitrarily expensive computation.
- **Consensus divergence**: Nodes running debug builds (where overflow panics) reject the block; nodes running release builds accept it. This is a consensus split.

---

### Likelihood Explanation

- `op_unknown` is reachable in consensus mode (block validation) whenever `ClvmFlags::NO_UNKNOWN_OPS` is **not** set. `MEMPOOL_MODE` sets `NO_UNKNOWN_OPS`, so mempool nodes are protected, but full-node block validators are not.
- The attacker controls both `cost_multiplier` (via opcode bytes) and the pre-multiplication `cost` (via argument count and sizes). Both are embedded in the CLVM program bytes submitted on-chain.
- Achieving `cost = k × 2^32` for integer `k` requires solving `ARITH_BASE_COST + n × ARITH_COST_PER_ARG + b × ARITH_COST_PER_BYTE = k × 2^32`. Since `gcd(ARITH_COST_PER_ARG, ARITH_COST_PER_BYTE) = gcd(320, 3) = 1`, solutions exist by Bezout's identity. The attacker can find concrete `(n, b)` pairs offline and embed them in the program.
- No privileged access, social engineering, or compromised infrastructure is required.

---

### Recommendation

Replace the unchecked multiplication with an explicit overflow check:

```rust
// Instead of:
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 { ... }

// Use:
let Some(new_cost) = cost.checked_mul(cost_multiplier + 1) else {
    return Err(EvalErr::Invalid(o));
};
if new_cost > u32::MAX as u64 {
    return Err(EvalErr::Invalid(o));
}
Ok(Reduction(new_cost as Cost, allocator.nil()))
```

This ensures that any multiplication that would overflow is treated as an invalid opcode rather than silently wrapping to a small value.

### Proof of Concept

**Opcode construction:**

- Choose `cost_function = 1` (add-like, bits `01` in the top 2 bits of the last byte).
- Set the multiplier prefix bytes to `[0xFF, 0xFF, 0xFF]` so `cost_multiplier = 0xFFFFFF = 16,777,215` and `cost_multiplier + 1 = 16,777,216 = 2^24`.
- Craft arguments so that the pre-multiplication cost equals `2^24 × k` for some `k` such that `k × 2^24 × 2^24 = k × 2^48` wraps to 0 mod `2^64` — i.e., `k` is a multiple of `2^16 = 65,536`.
- Alternatively, use `cost_multiplier = u32::MAX` (prefix `[0xFF, 0xFF, 0xFF, 0xFF]` — but `u32_from_u8` requires ≤ 4 bytes and the last byte encodes `cost_function`, so the prefix is `op[0..op.len()-1]`; a 5-byte opcode with prefix `[0xFF, 0xFF, 0xFF, 0xFF]` gives `cost_multiplier = 0xFFFFFFFF`).

**Concrete numeric example:**

```
cost_multiplier = 0xFFFFFFFF  →  cost_multiplier + 1 = 2^32
target pre-multiplication cost = 2^32 = 4,294,967,296

ARITH_BASE_COST = 99
ARITH_COST_PER_ARG = 320
ARITH_COST_PER_BYTE = 3

Solve: 99 + 320·n + 3·b = 4,294,967,296
       320·n + 3·b = 4,294,967,197

One solution (by extended Euclidean): n = 13,421,772, b = 9
(verify: 320·13,421,772 + 3·9 = 4,294,967,040 + 27 = 4,294,967,067 ≠ 4,294,967,197)

Adjust: n = 13,421,772, b = 43 (320·13421772 + 3·43 = 4294967040 + 129 = 4294967169 ≠ target)
... iterate; since gcd(320,3)=1 a solution exists within a small search.

After multiplication: cost = 0 (wraps from 2^64)
Guard: 0 > u32::MAX → false → returns Ok(Reduction(0, nil))
```

The attacker embeds this opcode (with the crafted argument list) inside a CLVM program submitted in a Chia block. Each invocation costs 0 in the cost meter. The attacker repeats it thousands of times within a single block, forcing validators to perform unbounded computation while the cost counter stays near zero.
