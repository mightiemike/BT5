### Title
Hardcoded `LastKnownBonded = true` in `MsgTierRedelegate` Without Checking Destination Validator Bond Status Allows Overpayment of Bonus Rewards — (`x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`MsgTierRedelegate` unconditionally sets `LastKnownBonded = true` after moving a position to a destination validator, without verifying that the destination validator is actually bonded at that moment. This is the direct analog to the Chainlink stale-price bug: a time-sensitive state value (`LastKnownBonded`) is recorded without a freshness/validity check, and is later used as the starting bonded state for bonus reward computation, causing the protocol to pay bonus rewards for a period when the validator was not bonded.

---

### Finding Description

In `MsgTierRedelegate`, after rewards are claimed and the staking redelegation is executed, the position's bonus checkpoints are updated:

```go
// x/tieredrewards/keeper/msg_server.go
latestSeq, err := ms.getValidatorEventLatestSeq(ctx, dstValAddr)
...
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), true)  // ← hardcoded true
``` [1](#0-0) 

The third argument `true` hardcodes `LastKnownBonded = true` for the position on the destination validator, regardless of whether that validator is currently bonded or unbonding.

`LastKnownBonded` is the persisted starting bonded state used by `processEventsAndClaimBonus` on the next claim:

```go
// x/tieredrewards/keeper/claim_rewards.go
bonded := pos.LastKnownBonded          // ← starting state from persisted field
segmentStart := pos.LastBonusAccrual

for _, entry := range events {
    evt := entry.Event
    if bonded {
        bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
        totalBonus = totalBonus.Add(bonus)
    }
    switch evt.EventType {
    case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND:
        bonded = false
    case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND:
        bonded = true
    ...
    }
    segmentStart = evt.Timestamp
}
``` [2](#0-1) 

**Attack path:**

1. Validator V is bonded, then begins unbonding. An `UNBOND` event is recorded at `seq = N`.
2. A user calls `MsgTierRedelegate` targeting validator V (which is now unbonding). The Cosmos SDK staking module does not require the destination validator to be bonded for `Redelegate` to succeed.
3. `getValidatorEventLatestSeq(ctx, dstValAddr)` returns `N` (the UNBOND event seq). The position is stored with `LastEventSeq = N` and `LastKnownBonded = true`.
4. Validator V re-bonds. A `BOND` event is recorded at `seq = N+1`.
5. The user calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` starts with `bonded = true` (from `LastKnownBonded`). It processes the `BOND` event at `seq = N+1` and, because `bonded = true`, computes bonus for the segment `[LastBonusAccrual, BOND_event_time]` — the entire unbonding period — using `bonded = true`. [3](#0-2) 

The defensive check at the end of `processEventsAndClaimBonus` (`bonded && val.IsBonded()`) only guards the **final open segment** after all events are processed. It does not protect the **intermediate segment** computed when the `BOND` event is encountered. [4](#0-3) 

The bonus formula is:

```
shares × tokensPerShare × bonusApy × durationSeconds / SecondsPerYear
``` [5](#0-4) 

The corrupted value is the `totalBonus` sent from the `RewardsPoolName` module account to the position owner. [6](#0-5) 

---

### Impact Explanation

The rewards pool (`types.RewardsPoolName`) is drained by bonus payments that were not legitimately earned. The position owner receives bonus tokens for the entire unbonding period of the destination validator (typically 21 days in Cosmos SDK), during which no bonus should have accrued. For a large position (e.g., 1,000,000 tokens, 4% APY, 21-day unbonding), the overpayment is approximately 2,300 tokens per exploit cycle. The attack is repeatable: the user can redelegate back and forth to a validator that cycles between bonded and unbonding states.

The exact corrupted invariant: `RewardsPoolName` module account balance is reduced by more than the legitimately accrued bonus for the position.

---

### Likelihood Explanation

The Cosmos SDK `Redelegate` function does not require the destination validator to be bonded. The tieredrewards module's ADR-documented validations for `MsgTierRedelegate` ("Exit not elapsed; tier not close-only; Amount > 0; dst != src") do not include a check that the destination validator is bonded. [7](#0-6) 

Any unprivileged user with a delegated tier position can trigger this by redelegating to a validator that is currently in the unbonding set. Validators enter the unbonding set regularly through normal network operations (jailing, voluntary unbonding, governance slashing). The attacker needs no privileged access — only a valid `MsgTierRedelegate` transaction.

---

### Recommendation

Before calling `UpdateBonusCheckpoints`, read the destination validator's live bond status and use it as the initial `LastKnownBonded` value:

```go
// In MsgTierRedelegate, replace:
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), true)

// With:
dstVal, err := ms.stakingKeeper.GetValidator(ctx, dstValAddr)
if err != nil {
    return nil, err
}
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), dstVal.IsBonded())
``` [8](#0-7) 

This mirrors the fix pattern from the Chainlink report: read the freshness/validity of the value at the point of use, not assume a fixed state.

---

### Proof of Concept

1. Governance creates Tier 1 with `BonusApy = 0.04`, `ExitDuration = 24h`.
2. Alice locks 1,000,000 basecro into Tier 1 on validator V (bonded). Position P is created with `LastKnownBonded = true`, `LastEventSeq = 0`.
3. Validator V begins unbonding. Hook fires: `UNBOND` event recorded at `seq = 1` for validator V.
4. Alice calls `MsgTierRedelegate(positionId=P, dstValidator=V)` (same validator, but now unbonding — or any other unbonding validator). After `claimRewards` settles the pre-redelegate period, `UpdateBonusCheckpoints(1, blockTime, true)` is called. Position P now has `LastEventSeq = 1`, `LastKnownBonded = true`, `LastBonusAccrual = T0`.
5. 21 days pass. Validator V re-bonds. Hook fires: `BOND` event recorded at `seq = 2` for validator V, with timestamp `T1 = T0 + 21 days`.
6. Alice calls `MsgClaimTierRewards(positionIds=[P])`.
   - `bonded = true` (from `LastKnownBonded`)
   - Process event seq=2 (BOND): since `bonded = true`, compute bonus for `[T0, T1]` (21 days) → `1,000,000 × rate × 0.04 × (21×86400) / 31,557,600 ≈ 2,301 basecro`
   - This bonus is paid from the rewards pool, but the validator was unbonding the entire time — no bonus should have accrued. [3](#0-2) [1](#0-0)

```go
func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}
