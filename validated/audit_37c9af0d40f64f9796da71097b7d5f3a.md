### Title
Spot Asset Liquidation Increases Initial Health, Permanently Blocking All Liquidation Paths for Positions Between Initial and Maintenance Thresholds — (`core/contracts/ClearinghouseLiq.sol`)

### Summary
When a subaccount's maintenance health is negative but initial health is positive, the `ERR_LIQUIDATED_TOO_MUCH` guard in `_handleLiquidationPayment` permanently blocks every liquidation path, leaving the position unliquidatable indefinitely.

### Finding Description

**Root cause — `getLiqPriceX18` in `ClearinghouseStorage.sol`:** [1](#0-0) 

The liquidation price for a positive-balance spot asset is:

```
liquidationPrice = oraclePrice * (1 + (longWeightMaintenance - 1) / 5)
```

Because `longWeightMaintenance < 1`, the discount is `(1 - longWeightMaintenance) / 5`. For `longWeightMaintenance = 0.9` this is only **2%**, giving `liquidationPrice = 0.98 * oraclePrice`.

The initial-health contribution of that same spot asset is `longWeightInitial * oraclePrice` per unit (e.g., `0.8 * oraclePrice`). Because `0.98 > 0.80`, the quote the liquidatee receives exceeds the health-weighted value of the asset they lose. Every unit liquidated **increases** initial health.

**The blocking guard — `_handleLiquidationPayment` in `ClearinghouseLiq.sol`:** [2](#0-1) 

After applying the payment, the protocol checks:
```solidity
require(!isAboveInitial(txn.liquidatee), ERR_LIQUIDATED_TOO_MUCH);
```

If initial health was already `> 0` before the call, and liquidating the spot asset increases it further, this check **always reverts** regardless of the liquidation amount chosen.

**Reachable prerequisite — position between initial and maintenance thresholds:**

A subaccount can have maintenance health `< 0` while initial health `> 0` whenever the liability's weight spread `(shortWeightMaintenance − shortWeightInitial)` exceeds the asset's weight spread `(longWeightMaintenance − longWeightInitial)`. Concrete example with `price = 1`:

| | balance | longWeightInitial | longWeightMaintenance | shortWeightInitial | shortWeightMaintenance |
|---|---|---|---|---|---|
| Spot asset A | +100 | 0.80 | 0.90 | — | — |
| Spot liability B | −110 | — | — | 0.70 | 0.85 |

```
Initial health     = 100×0.80 − 110×0.70 = 80 − 77 = +3   (> 0, above initial)
Maintenance health = 100×0.90 − 110×0.85 = 90 − 93.5 = −3.5 (< 0, liquidatable)
```

**All three liquidation paths are blocked:**

1. **Spot asset path** (`txn.amount > 0` on asset A): `ERR_LIQUIDATED_TOO_MUCH` reverts because initial health increases. [2](#0-1) 

2. **Spot liability path** (`txn.amount < 0` on liability B): `_assertCanLiquidateLiability` requires all spot balances `<= 0`, but asset A has a positive balance. [3](#0-2) 

3. **Finalization path** (`txn.productId == type(uint32).max`): `_finalizeSubaccount` requires all spot asset balances `<= 0` before proceeding. [4](#0-3) 

The position is permanently stuck.

### Impact Explanation
Subaccounts with maintenance health `< 0` but initial health `> 0` — a naturally reachable state when liability weight spreads exceed asset weight spreads — cannot be liquidated through any on-chain path. The protocol accumulates unliquidatable underwater positions. Unlike Nado's bad-debt socialization path (which is also blocked here), there is no fallback. Protocol solvency degrades as these positions grow.

### Likelihood Explanation
Moderate. The condition arises organically when a user borrows a spot asset whose `shortWeightMaintenance − shortWeightInitial` is larger than the collateral's `longWeightMaintenance − longWeightInitial`. As oracle prices drift, a healthy position can slide into this zone. No attacker action is required; normal market movement suffices. A malicious liquidator could also deliberately perform a small partial liquidation on a healthy position to push it into this zone (if the partial liquidation itself increases initial health past 0), then abandon it.

### Recommendation
1. **Fix the liquidation price formula** so that `liquidationPrice ≤ longWeightInitial * oraclePrice`, ensuring every spot asset liquidation decreases (never increases) initial health. This requires a larger discount: `penalty ≥ (1 − longWeightInitial)`.
2. **Alternatively**, relax `_assertCanLiquidateLiability` to permit liability liquidation when the normal spot-asset path is provably blocked (i.e., when liquidating the asset would increase initial health).
3. **Or**, change the `ERR_LIQUIDATED_TOO_MUCH` guard to check maintenance health instead of initial health, so that a liquidation is only blocked when it would make the position maintenance-healthy, not merely initial-healthy.

### Proof of Concept
```
Setup:
  Spot asset A:     balance = +100, longWeightInitial = 0.80, longWeightMaintenance = 0.90, price = 1
  Spot liability B: balance = −110, shortWeightInitial = 0.70, shortWeightMaintenance = 0.85, price = 1

Verify precondition:
  initialHealth     = 100×0.80 − 110×0.70 = +3   → isAboveInitial = true
  maintenanceHealth = 100×0.90 − 110×0.85 = −3.5  → isUnderMaintenance = true

Step 1 — liquidate spot asset A (any amount, e.g. 50):
  liquidationPrice = 1 × (1 + (0.90−1)/5) = 0.98
  Δinitial health  = (0.98 − 0.80) × 50 = +9  → initialHealth becomes +12
  → require(!isAboveInitial) REVERTS with ERR_LIQUIDATED_TOO_MUCH

Step 2 — liquidate spot liability B:
  _assertCanLiquidateLiability: asset A balance = +100 > 0
  → REVERTS with ERR_NOT_LIQUIDATABLE_LIABILITIES

Step 3 — finalize (productId = type(uint32).max):
  _finalizeSubaccount: asset A balance = +100 > 0
  → REVERTS with ERR_NOT_FINALIZABLE_SUBACCOUNT

Result: position is permanently unliquidatable.
```
