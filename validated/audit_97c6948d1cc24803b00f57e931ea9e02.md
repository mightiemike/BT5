### Title
Unrestricted `run_mainchain_gc` Allows Any Caller to Maximally Prune Mainchain Headers, Breaking SPV Proof Verification — (`contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public NEAR method with no caller restriction. Any unprivileged NEAR account can invoke it with `batch_size = u64::MAX`, forcing the mainchain to be pruned down to exactly `gc_threshold` blocks in a single call. This is the direct analog of the `bid()` vulnerability: just as anyone could call `bid()` to drive the crowdfund to `maximumBid`, anyone can call `run_mainchain_gc(u64::MAX)` to drive the stored mainchain to its hard floor (`gc_threshold`). A cross-chain bridge or SPV client whose target block falls in the pruned range will have its `verify_transaction_inclusion` call panic, permanently blocking that verification.

---

### Finding Description

`submit_blocks` — the primary header-ingestion entry point — is gated by `#[trusted_relayer]`, which restricts callers to staked, approved relayers: [1](#0-0) 

`submit_blocks` internally calls `run_mainchain_gc` with a controlled batch size equal to the number of headers just submitted: [2](#0-1) 

However, `run_mainchain_gc` is also exposed as a standalone public method with **no** `#[trusted_relayer]` guard and no role check beyond the pause gate: [3](#0-2) 

The `batch_size` parameter is fully attacker-controlled. The function caps removal at `total_amount_to_remove = amount_of_headers_we_store - gc_threshold`, then takes `min(total_amount_to_remove, batch_size)`: [4](#0-3) 

Passing `u64::MAX` as `batch_size` therefore always selects `total_amount_to_remove`, pruning every header beyond `gc_threshold` in one transaction. This is structurally identical to the `bid()` pattern: an unrestricted caller drives a bounded value to its extreme (here, the minimum mainchain size rather than the maximum bid).

---

### Impact Explanation

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2`) look up the target block in `mainchain_header_to_height`. If the block has been removed by GC, the call panics: [5](#0-4) 

A cross-chain bridge or any consumer of the SPV API that holds a valid Merkle proof for a transaction in a block that falls in the pruned range will receive a hard panic — not a graceful `false` — permanently blocking that verification path. Because `mainchain_header_to_height` is also cleared for pruned blocks: [6](#0-5) 

there is no recovery: once pruned, the block hash is gone from both index maps and the headers pool.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `run_mainchain_gc` with an arbitrary `u64` argument at zero cost (no deposit required, no staking). The attacker only needs to observe that the mainchain has grown beyond `gc_threshold` (which happens continuously during normal relayer operation) and time the call to precede a bridge's `verify_transaction_inclusion` call for a block in the excess zone. This is observable on-chain via `get_last_block_height` and `get_block_hash_by_height`.

---

### Recommendation

Apply the same `#[trusted_relayer]` guard used on `submit_blocks` to `run_mainchain_gc`, or add an explicit role check (e.g., `Role::UnrestrictedRunGC` or `Role::DAO`) so that only authorized accounts can trigger external GC. The internal call from `submit_blocks` is unaffected because it bypasses the macro guard as a private method call.

---

### Proof of Concept

1. Relayer submits headers normally; mainchain grows to `gc_threshold + N` blocks (N ≥ 1).
2. A cross-chain bridge prepares a `verify_transaction_inclusion` call for a transaction in block at absolute height `initial_height + k`, where `k < N` (the block is in the excess zone).
3. Attacker calls `run_mainchain_gc(u64::MAX)` from any NEAR account.
4. Inside `run_mainchain_gc`: `total_amount_to_remove = N`, `selected_amount_to_remove = min(N, u64::MAX) = N`. All N excess blocks are removed; `mainchain_initial_blockhash` advances by N. [7](#0-6) 
5. Bridge calls `verify_transaction_inclusion` with `tx_block_blockhash` = hash of the pruned block.
6. `mainchain_header_to_height.get(&args.tx_block_blockhash)` returns `None`; contract panics with `"block does not belong to the current main chain"`. [8](#0-7) 
7. The bridge cannot verify the transaction; any dependent cross-chain action (token unlock, proof settlement) is permanently blocked for that block.
