### Title
`ValidResourceBounds` Variant Lost in Protobuf Deserialization Produces Wrong Transaction Hash Preimage — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf-to-`ValidResourceBounds` conversion silently collapses `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` into `L1Gas(X)`. Because `get_tip_resource_bounds_hash` hashes a **different number of resource felts** depending on the variant, the transaction hash computed after a P2P sync round-trip differs from the hash that was originally signed and accepted by the gateway. This is a direct analog of the external bug: a structural field that controls the hash preimage is not preserved through a conversion boundary, so the signed hash and the post-conversion hash diverge.

---

### Finding Description

`ValidResourceBounds` has two variants with **different hash preimages**:

```
L1Gas(X)          → poseidon([tip, L1_GAS_packed, L2_GAS_packed(0)])          // 2 resource felts
AllResources(X,0,0) → poseidon([tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)])  // 3 resource felts
``` [1](#0-0) 

The protobuf deserializer reconstructs the variant purely from the numeric values:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [2](#0-1) 

A transaction submitted via RPC with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is always stored internally as `ValidResourceBounds::AllResources(X, 0, 0)` and its hash H₁ is computed over **3 resource felts**. When this transaction is serialized to protobuf for P2P block sync and deserialized on a peer, the condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is true, so the peer reconstructs `ValidResourceBounds::L1Gas(X)`. Any subsequent call to `calculate_transaction_hash` on the peer's copy produces H₂ over **2 resource felts**, where H₁ ≠ H₂.

The `ValidResourceBounds` enum definition confirms the two variants are semantically distinct protocol versions: [3](#0-2) 

The hash function that branches on the variant: [4](#0-3) 

The serializer for `ValidResourceBounds` to JSON (storage path) correctly preserves the variant by including or omitting the `L1DataGas` key, so the storage round-trip is safe. Only the protobuf path (P2P sync) loses the variant: [5](#0-4) 

The TODO comment in the protobuf converter acknowledges this is a known compatibility gap: [6](#0-5) 

---

### Impact Explanation

After a P2P block sync, a peer node holds `ValidResourceBounds::L1Gas(X)` for a transaction that was originally signed and committed with `ValidResourceBounds::AllResources(X, 0, 0)`. Any component that calls `calculate_transaction_hash` on the deserialized `InvokeTransactionV3` (e.g., RPC `starknet_getTransactionByHash` hash verification, block hash recomputation, or re-execution) will compute H₂ ≠ H₁. This binds the wrong hash to the transaction on the receiving node, matching the "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload" impact category. [7](#0-6) 

---

### Likelihood Explanation

Any user can submit an `InvokeV3` transaction via the JSON-RPC gateway with `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas` non-zero). The gateway accepts it, computes H₁ from `AllResources`, and commits it to a block. Every peer that syncs this block via P2P will silently reconstruct `L1Gas` and diverge on hash recomputation. No special privilege is required; the transaction is structurally valid and passes all gateway checks. [8](#0-7) 

---

### Recommendation

Preserve the `ValidResourceBounds` variant through the protobuf wire format. The simplest fix is to add a boolean or enum field to the protobuf `ResourceBounds` message that explicitly encodes whether the transaction uses the `L1Gas` (pre-0.13.3) or `AllResources` (0.13.3+) variant, and use that field—not the zero-value heuristic—to reconstruct the Rust enum on deserialization. Alternatively, remove the `L1Gas` variant from the P2P sync path entirely (since all new transactions use `AllResources`) and only apply the heuristic when reading legacy storage. [9](#0-8) 

---

### Proof of Concept

1. Submit an `InvokeV3` transaction to the gateway with `resource_bounds = { L1_GAS: { max_amount: 1000, max_price_per_unit: 1 }, L2_GAS: { 0, 0 }, L1_DATA_GAS: { 0, 0 } }`. The gateway computes H₁ using `get_tip_resource_bounds_hash` with `AllResources`, hashing 3 resource felts.

2. The transaction is committed to a block. The block is propagated via P2P sync. The `InvokeTransactionV3` is serialized to `protobuf::ResourceBounds` with `l2_gas = 0` and `l1_data_gas = 0`.

3. A peer deserializes via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is true, so the peer stores `ValidResourceBounds::L1Gas(1000/1)`.

4. The peer calls `calculate_transaction_hash` on the stored `InvokeTransactionV3`. `get_tip_resource_bounds_hash` now hashes only 2 resource felts (no `L1_DATA_GAS`), producing H₂.

5. H₁ ≠ H₂. The peer's hash for this transaction diverges from the canonical hash stored in the block header and known to the original submitter. [10](#0-9) [1](#0-0)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
