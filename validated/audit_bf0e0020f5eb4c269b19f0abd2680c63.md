### Title
`verify_transaction_inclusion_v2` unconditionally panics for valid single-transaction block proofs due to inherited empty-proof guard — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` calls the deprecated `verify_transaction_inclusion` unconditionally as its final step. That inner function contains an unconditional guard `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")`. For a Bitcoin block containing only the coinbase transaction, the correct and complete merkle proof is an empty vector — yet this guard causes the call to panic, permanently blocking SPV verification for any such block.

---

### Finding Description

`verify_transaction_inclusion_v2` is the recommended, non-deprecated SPV entry point. It first validates the coinbase merkle proof to mitigate the 64-byte forgery attack, then delegates to `verify_transaction_inclusion` unconditionally:

```rust
// contract/src/lib.rs line 367-368
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
```

Inside `verify_transaction_inclusion`, after all other checks, there is an unconditional guard:

```rust
// contract/src/lib.rs line 315
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

For a block that contains exactly one transaction (the coinbase), the Merkle tree has depth zero: `merkle_root == coinbase_tx_id == tx_id`, and the correct proof for that transaction is an empty vector. The `compute_root_from_merkle_proof` function correctly handles this — when called with an empty proof it returns the transaction hash unchanged:

```rust
// merkle-tools/src/lib.rs lines 39-51
let mut current_hash = transaction_hash;
for proof_hash in merkle_proof {   // loop body never executes for empty vec
    ...
}
current_hash   // returns transaction_hash unchanged
```

So in `verify_transaction_inclusion_v2`, the coinbase proof check at lines 358–365 passes correctly (empty proof → `coinbase_tx_id` == `merkle_root`). The length-equality check at lines 348–351 also passes (0 == 0). But the subsequent unconditional call to `verify_transaction_inclusion` then hits the `require!(!args.merkle_proof.is_empty())` guard and panics, even though the proof is mathematically complete and correct.

---

### Impact Explanation

Any unprivileged NEAR caller invoking `verify_transaction_inclusion_v2` for a transaction in a single-transaction Bitcoin block will always receive a panic. The SPV verification API — the primary consumer-facing output of the entire light client — permanently returns an error for a class of valid, on-chain Bitcoin blocks. Downstream contracts or applications that rely on this verification to release funds or authorize cross-chain actions are blocked for those blocks.

---

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) are valid and occur on mainnet, particularly during periods of low mempool activity. The condition is not adversarially constructed; it arises from normal Bitcoin network operation. Any relayer that has submitted such a block header to the contract creates a state where `verify_transaction_inclusion_v2` is permanently broken for that block.

---

### Recommendation

Remove the unconditional `require!(!args.merkle_proof.is_empty())` guard from `verify_transaction_inclusion`, or replace it with a conditional that allows an empty proof when `tx_id == merkle_root`. The security concern that guard was meant to address (preventing a forged empty-proof claim) is already fully handled by the coinbase proof check in `verify_transaction_inclusion_v2`. Alternatively, `verify_transaction_inclusion_v2` can perform its own proof computation directly rather than delegating to the deprecated function, avoiding the inherited guard entirely.

---

### Proof of Concept

1. A Bitcoin block at some height `H` contains only the coinbase transaction with hash `C`. Its `merkle_root == C`.
2. The relayer submits this block header; it is stored in `headers_pool`.
3. A NEAR caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id = C`, `tx_index = 0`, `tx_block_blockhash = hash(H)`
   - `merkle_proof = []`
   - `coinbase_tx_id = C`, `coinbase_merkle_proof = []`
   - `confirmations = 1`
4. **Line 348–351**: `merkle_proof.len() == coinbase_merkle_proof.len()` → `0 == 0` ✓
5. **Lines 358–365**: `compute_root_from_merkle_proof(C, 0, &[])` returns `C`; `C == merkle_root` ✓
6. **Line 368**: `self.verify_transaction_inclusion(args.into())` is called.
7. **Line 315**: `require!(!args.merkle_proof.is_empty())` → **PANIC: "Merkle proof is empty"**

The call reverts. The valid transaction cannot be verified. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
