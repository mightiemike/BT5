### Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension` is designed to gate `addLiquidity` by depositor address. Its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual caller of `addLiquidity`) and instead checks the position `owner`. Because `MetricOmmPoolLiquidityAdder` lets any caller supply an arbitrary `owner`, any unprivileged address can bypass the allowlist by naming an already-allowed address as the position owner.

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives two address arguments: the first (unnamed, discarded) is the address that called `addLiquidity` on the pool (`sender`), and the second is the position `owner`. [1](#0-0) 

The check reads:

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
```

`msg.sender` here is the pool; `owner` is the position owner — **not** the address that is actually depositing tokens. The mapping name `allowedDepositor` and the setter `setAllowedToDeposit` make the intent clear: the guard is supposed to restrict who can deposit, i.e., the `sender`.

The sister extension `SwapAllowlistExtension.beforeSwap` correctly checks the first parameter (`sender`): [2](#0-1) 

The inconsistency confirms the bug in `DepositAllowlistExtension`.

The pool calls `_beforeAddLiquidity(msg.sender, owner, ...)`, passing the actual caller as the first argument: [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (the explicit-owner overload) lets any caller supply an arbitrary `owner`: [4](#0-3) 

Tokens are pulled from `msg.sender` (the attacker) in the callback, while the position is recorded under the supplied `owner`: [5](#0-4) 

### Impact Explanation

The deposit allowlist — the pool admin's primary mechanism for restricting LP participation to approved addresses — is rendered ineffective. Any unprivileged address can deposit into a restricted pool by naming an approved address as `owner`. The pool receives tokens from an unauthorized depositor, violating the admin-configured access boundary. This qualifies as an admin-boundary break: a factory/pool-admin access control is bypassed by an unprivileged path.

### Likelihood Explanation

Exploitation requires only a standard call to `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with `owner` set to any address already on the allowlist. No special privileges, flash loans, or oracle manipulation are needed. Any on-chain observer can read the allowlist state and identify an approved address to use as `owner`.

### Recommendation

Change the `beforeAddLiquidity` hook to check the first parameter (the actual depositor/caller) instead of `owner`, mirroring the pattern used in `SwapAllowlistExtension`:

```solidity
// Before (wrong):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}

// After (correct):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` configured; adds Alice (`0xAlice`) to the allowlist via `setAllowedToDeposit(pool, alice, true)`.
2. Bob (`0xBob`, not on the allowlist) calls `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, data)`.
3. The pool calls `_beforeAddLiquidity(LiquidityAdder, alice, ...)`.
4. The extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. Bob's tokens are pulled in the callback; a position is recorded under Alice.
6. Bob has deposited into a pool he is not authorized to access; the allowlist is bypassed.
