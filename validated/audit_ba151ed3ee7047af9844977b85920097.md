### Title
Missing `max_l2_gas_amount` Enforcement for Declare Transactions at Gateway Admission - (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary
The `StatelessTransactionValidator` enforces an upper bound on `l2_gas.max_amount` for Invoke and DeployAccount transactions but explicitly skips this check for Declare transactions. A user can submit a Declare transaction with an arbitrarily large `l2_gas.max_amount` (up to `u64::MAX`) and the gateway will admit it, bypassing the configured `max_l2_gas_amount` limit entirely.

### Finding Description

In `validate_resource_bounds`, the gateway enforces `max_l2_gas_amount` for all transaction types except Declare:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The configured limit is `max_l2_gas_amount = 1_210_000_000` (from both the default config and the deployed `config_schema.json`): [2](#0-1) [3](#0-2) 

The test `valid_l2_gas_amount_on_declare` explicitly documents and confirms this gap — a Declare transaction with `max_amount: GasAmount(200)` passes validation even when `max_l2_gas_amount: 100`: [4](#0-3) 

The stateful validator's `validate_resource_bounds` only checks the L2 gas *price* threshold (not the amount), so it provides no compensating control for Declare transactions either: [5](#0-4) 

### Impact Explanation

A user can craft a `RpcDeclareTransaction::V3` with `l2_gas.max_amount = u64::MAX` (or any value exceeding `max_l2_gas_amount`). The gateway's `StatelessTransactionValidator` will accept it without error. The transaction then proceeds through stateful validation and into the mempool. This matches the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The `max_l2_gas_amount` limit exists precisely to prevent oversized resource-bound declarations from entering the system. Bypassing it for Declare transactions means the gateway's admission invariant — that no accepted transaction may claim more than `max_l2_gas_amount` of L2 gas — is broken for an entire transaction type.

### Likelihood Explanation

The trigger is fully unprivileged: any user submitting a Declare transaction via the public RPC endpoint can set `l2_gas.max_amount` to any `u64` value. The bypass requires no special knowledge beyond reading the source code (the TODO comment is public). The `min_gas_price` check still applies, so `max_price_per_unit` must be ≥ `8_000_000_000`, but `max_amount` is unconstrained.

### Recommendation

Apply the same `max_l2_gas_amount` upper-bound check to Declare transactions. Remove the early-return branch for `RpcTransaction::Declare` and let the existing check apply uniformly:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher bound (e.g., because class compilation consumes more L2 gas), introduce a separate `max_l2_gas_amount_declare` config parameter rather than removing the check entirely.

### Proof of Concept

1. Construct an `RpcDeclareTransaction::V3` with:
   - `l2_gas.max_amount = GasAmount(u64::MAX)`
   - `l2_gas.max_price_per_unit = GasPrice(8_000_000_000)` (satisfies `min_gas_price`)
   - Any valid Sierra contract class
2. Submit to the gateway's `add_declare_transaction` endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never triggered for Declare transactions, regardless of the `max_l2_gas_amount` config value.

The existing test already encodes this behavior as an expected pass: [4](#0-3)
