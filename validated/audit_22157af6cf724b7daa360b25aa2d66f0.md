### Title
Unprivileged Caller Can Prematurely Garbage-Collect Mainchain Headers via `run_mainchain_gc` — (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public, state-mutating function with no caller authorization check. Any unprivileged NEAR account can call it with an arbitrary `batch_size`, prematurely deleting mainchain block headers from on-chain storage and advancing `mainchain_initial_blockhash`. This permanently corrupts the verifiable window of the light client: once headers are removed, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will panic for any transaction in a removed block, and those blocks cannot be re-submitted.

---

### Finding Description

`submit_blocks` — the primary chain-advancing entry point — is protected by `#[trusted_relayer]`, which requires the caller to be a registered, active relayer with a staked role. [1](#0-0) 

`run_mainchain_gc`, however, carries only `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. That decorator restricts calls **only when the contract is paused**. When the contract is running normally (unpaused), the function is reachable by any NEAR account with no role, no stake, and no deposit requirement. [2](#0-1) 

Inside the function, the attacker-controlled `batch_size` is capped by `total_amount_to_remove = amount_of_headers_we_store - gc_threshold`, so the attacker cannot remove blocks that are still within the threshold window. However, passing `u64::MAX` as `batch_size` causes the function to remove every header that is currently eligible for GC in a single call — the maximum possible damage in one transaction. [3](#0-2) 

The removal loop deletes entries from `mainchain_height_to_header`, calls `remove_block_header` (which also removes from `mainchain_header_to_height` and `headers_pool`), and then advances `mainchain_initial_blockhash` to the new oldest block. [4](#0-3) 

---

### Impact Explanation

After the attack:

1. `verify_transaction_inclusion` looks up `args.tx_block_blockhash` in `mainchain_header_to_height`. For any removed block, this lookup panics with `"block does not belong to the current main chain"`.
2. `verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` and inherits the same failure.
3. Removed headers are **permanently lost**. The relayer cannot re-submit them because the contract only accepts blocks that extend the current chain tip; it has no mechanism to backfill deleted historical headers.

Downstream NEAR contracts that gate financial operations on `verify_transaction_inclusion` (cross-chain bridges, atomic swaps, peg-in/peg-out protocols) will be unable to verify legitimate Bitcoin transactions for any block that was garbage-collected by the attacker, potentially locking user funds indefinitely.

**Impact: High** — permanent corruption of the canonical verifiable window; downstream proof verification fails irreversibly for affected blocks.

---

### Likelihood Explanation

The attack requires no special role, no staked deposit, and no privileged key. Any NEAR account can call `run_mainchain_gc` at any time the contract is unpaused. The call is cheap (gas only) and the damage is immediate and irreversible. A griefing attacker or a competitor of a bridge protocol has clear economic motivation.

**Likelihood: High**

---

### Recommendation

Add a role guard to `run_mainchain_gc` matching the pattern used for `submit_blocks`. Only trusted relayers (or DAO-role holders) should be permitted to invoke GC directly. The internal call from `submit_blocks` (line 181) can bypass the guard via the existing `bypass_roles` mechanism already wired into the `#[trusted_relayer]` macro.

```rust
// Before (no caller restriction when unpaused):
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { … }

// After (restrict to trusted relayers / DAO when unpaused):
#[pause(except(roles(Role::UnrestrictedRunGC)))]
#[trusted_relayer]          // or an explicit acl_require! for Role::DAO / RelayerManager
pub fn run_mainchain_gc(&mut self, batch_size: u64) { … }
```

---

### Proof of Concept

1. Deploy `BtcLightClient` with `gc_threshold = 100` and submit 250 blocks via a trusted relayer. The contract now stores 250 headers; 150 are eligible for GC.
2. Create a fresh NEAR account with no roles and no stake.
3. From that account, call `run_mainchain_gc(u64::MAX)`. The function computes `total_amount_to_remove = 150`, caps `selected_amount_to_remove = min(u64::MAX, 150) = 150`, and deletes headers at heights `[genesis .. genesis+149]`.
4. `mainchain_initial_blockhash` now points to height `genesis+150`.
5. Call `verify_transaction_inclusion` with a `tx_block_blockhash` from any of the 150 deleted blocks. The call panics: `"block does not belong to the current main chain"`.
6. Confirm the deleted headers are absent from `headers_pool` and cannot be re-submitted. [5](#0-4)
