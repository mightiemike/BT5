### Title
Zero-Amount Claim Does Not Mark Week as Claimed, Enabling Double-Claim via Batch — (`core/contracts/Airdrop.sol`)

---

### Summary

`_verifyProof` uses `claimed[week][sender] == 0` as the "not yet claimed" sentinel and writes `claimed[week][sender] = totalAmount` as the "mark claimed" action. When `totalAmount = 0`, the write is a no-op — the storage slot remains `0`, indistinguishable from the initial unclaimed state. A batch `claim()` call with two entries for the same week (`totalAmount=0` first, `totalAmount=X` second) therefore passes the guard twice, transferring `X` tokens on the second pass.

---

### Finding Description

The guard and the state-write in `_verifyProof` share the same sentinel value (`0`):

```
Line 48:  require(claimed[week][sender] == 0, "Already claimed.");
Line 62:  claimed[week][sender] = totalAmount;
``` [1](#0-0) 

When `totalAmount = 0`:
- Line 48 passes (slot is `0` — initial state).
- Line 62 writes `0` back — **the slot is unchanged**.

The public `claim()` iterates over a caller-supplied array without any deduplication check on `week`: [2](#0-1) 

So submitting `[{week:W, totalAmount:0, proof0}, {week:W, totalAmount:X, proofX}]`:

| Iteration | `claimed[W][attacker]` before | Guard result | Write | Transfer |
|-----------|-------------------------------|--------------|-------|----------|
| 1st (amount=0) | `0` | PASS | `0` (no change) | 0 tokens |
| 2nd (amount=X) | `0` (still!) | **PASS** | `X` | **X tokens** |

The precondition is that the Merkle tree for week W contains **both** `(attacker, 0)` and `(attacker, X)` leaves. The code imposes no uniqueness constraint on address entries within a tree, and the owner constructs trees off-chain. A tree with both leaves is structurally valid and verifiable: [3](#0-2) 

---

### Impact Explanation

An attacker holding valid proofs for both a zero-amount and a non-zero-amount leaf for the same week can drain `X` tokens from the contract for that week. If multiple weeks have such trees, the attack is repeatable per week. This is direct, unauthorized token extraction — a Critical asset-seizure impact.

---

### Likelihood Explanation

The precondition (Merkle tree containing both `(attacker, 0)` and `(attacker, X)` for the same address/week) is not the normal case, but:
- The code enforces **no uniqueness** on address entries within a tree.
- Protocols sometimes include zero-amount entries for eligibility tracking or snapshot completeness.
- Any off-chain tooling error that emits two rows for the same address (one zeroed, one real) produces a vulnerable tree.
- Once such a tree is registered, the exploit is fully permissionless and requires no privileged access.

Likelihood is **Medium** (depends on tree construction practice) with **Critical** impact when triggered.

### Recommendation

Replace the `uint256` claimed amount as the sole "claimed" sentinel with an explicit boolean flag, or require `totalAmount > 0` before processing:

```solidity
// Option A: reject zero-amount claims
require(totalAmount > 0, "Zero amount.");
require(claimed[week][sender] == 0, "Already claimed.");
claimed[week][sender] = totalAmount;

// Option B: separate claimed flag
mapping(uint32 => mapping(address => bool)) internal hasClaimed;
require(!hasClaimed[week][sender], "Already claimed.");
hasClaimed[week][sender] = true;
claimed[week][sender] = totalAmount;
```

Also add a duplicate-week guard in `claim()`:

```solidity
// track seen weeks within the batch
mapping(uint32 => bool) memory seenWeeks; // or use a local array
for (uint32 i = 0; i < claimProofs.length; i++) {
    require(!seenWeeks[claimProofs[i].week], "Duplicate week.");
    seenWeeks[claimProofs[i].week] = true;
    _claim(...);
}
```
