### Title
Unsocialized Bad Debt from Incomplete Liquidation Finalization Enables Bank Run — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

Nado's bad debt socialization is gated behind an explicit "finalize" liquidation call (`productId == type(uint32).max`). If no liquidator ever submits this finalization, the insolvent subaccount's negative `vQuoteBalance` (perp) and negative spot quote balance are never socialized. The protocol's tracked liabilities then exceed its actual token holdings, enabling a bank run where early withdrawers are made whole but late withdrawers lose funds.

---

### Finding Description

Nado's liquidation flow is split into two phases:

**Phase 1 — Position liquidation** (any `productId` except `type(uint32).max`): The liquidator closes the insolvent subaccount's positions one by one, collecting a liquidation discount. After all positions are closed, the subaccount may still hold a large negative `vQuoteBalance` in the `PerpEngine` and/or a negative spot quote balance in the `SpotEngine`.

**Phase 2 — Finalization** (`productId == type(uint32).max`): `_finalizeSubaccount` is the *only* code path that socializes bad debt. It calls `perpEngine.socializeSubaccount` and, if insurance is exhausted, `spotEngine.socializeSubaccount`. [1](#0-0) 

```solidity
function _finalizeSubaccount(...) internal returns (bool) {
    if (txn.productId != type(uint32).max) {
        return false;   // ← silently skips socialization for every normal liquidation call
    }
    ...
``` [2](#0-1) 

The socialization logic inside `_finalizeSubaccount` — `perpEngine.socializeSubaccount` and `spotEngine.socializeSubaccount` — is never reached unless a liquidator explicitly submits a transaction with `productId = type(uint32).max`. [3](#0-2) [4](#0-3) 

**Why finalization may never happen**: The only economic incentive to call finalize is to claim the liquidatee's *positive* `vQuoteBalance` (lines 322–337 of `_finalizeSubaccount`). When the subaccount has only negative `vQuoteBalance` (the bad-debt case), calling finalize yields no profit for the liquidator — it merely triggers socialization that dilutes the liquidator's own positions. There is no keeper reward, no protocol-enforced deadline, and no fallback path.

**Why the quote product cannot be directly liquidated**: The protocol explicitly blocks direct liquidation of the quote product: [5](#0-4) 

```solidity
require(txn.productId != QUOTE_PRODUCT_ID, ERR_INVALID_LIQUIDATION_PARAMS);
```

So a negative spot quote balance (bad debt) has *no* liquidation path other than finalization.

**`assertUtilization` does not protect against this**: The withdrawal guard only checks `totalDeposits >= totalBorrows` for the spot product in isolation. It does not account for perp `vQuoteBalance` that has been settled into spot quote by other users but whose corresponding bad debt was never socialized. [6](#0-5) 

---

### Impact Explanation

After any bad-debt liquidation where finalization is not called:

1. The insolvent subaccount holds a negative `vQuoteBalance` (e.g., −$100,000) that is not yet reflected in the spot engine's `totalDepositsNormalized`.
2. Counterparties with positive `vQuoteBalance` settle their PnL via `settlePnl`, which moves real tokens from `state.availableSettle` into their spot quote balance and then out via withdrawal.
3. The protocol pays out real tokens for the positive PnL, but the offsetting bad debt was never socialized — so the protocol's actual token balance is lower than the sum of all user spot balances.
4. The first users to withdraw succeed; the last users find the protocol insolvent.

This is a direct solvency / bank-run impact, identical in class to the Perennial M-4 finding.

---

### Likelihood Explanation

- Liquidators are rational profit-seekers. After extracting the liquidation discount in Phase 1, calling finalize on a subaccount with only negative PnL costs gas and dilutes the liquidator's own perp positions through socialization.
- There is no on-chain incentive (no keeper fee, no reward) attached to the finalization call.
- Any sufficiently large adverse price move that creates bad debt (i.e., collateral < loss) triggers this scenario. Such events are not rare in volatile markets.
- The abandoned subaccount remains under maintenance health (negative quote balance → negative health), but no liquidation path exists for the quote product, so the bad debt is permanently stranded unless finalize is voluntarily called.

---

### Recommendation

1. **Attach a keeper reward to finalization**: When `_finalizeSubaccount` is called and bad debt is socialized, pay the caller a fixed fee from the insurance fund (similar to Perennial's liquidation fee model). This creates an economic incentive to finalize promptly.

2. **Alternatively, socialize bad debt eagerly**: After each Phase-1 liquidation call that leaves the subaccount with zero positions but negative `vQuoteBalance`, automatically trigger socialization without requiring a separate finalization transaction.

3. **Guard `settlePnl` against unsocialized bad debt**: Before allowing positive PnL settlement, verify that no subaccount with zero positions holds unsocialized negative `vQuoteBalance` that would make the protocol insolvent.

---

### Proof of Concept

**Actors**: UserA (maker/LP), UserB (trader), Liquidator.

1. UserA deposits $1,000 USDC and provides liquidity (positive spot quote balance = $1,000).
2. UserB opens a large long perp position with $10 collateral. Price crashes; UserB's `vQuoteBalance` becomes −$110 (bad debt after collateral is consumed).
3. Liquidator calls `liquidateSubaccountImpl` with UserB's perp `productId`. UserB's perp `amount` becomes 0; `vQuoteBalance` remains −$110. Liquidator collects the liquidation discount and stops — no incentive to call finalize.
4. UserA calls `settlePnl` for their offsetting short position. Their positive `vQuoteBalance` (+$110) is settled: `state.availableSettle` decreases by $110, and UserA's spot quote balance increases to $1,110.
5. UserA withdraws $1,110. `assertUtilization` passes because `totalDeposits >= totalBorrows` in the spot engine (the $110 came from `availableSettle`, not a new deposit).
6. Protocol now holds $0 in tokens. UserB's −$110 bad debt was never socialized. Any other user attempting to withdraw finds the protocol empty.

The root cause is `_finalizeSubaccount` returning early for all non-`type(uint32).max` calls: [7](#0-6) 

combined with the absence of any on-chain mechanism that forces or incentivizes the finalization call when the liquidatee has only negative PnL.

    function socializeSubaccount(bytes32 subaccount, int128 insurance)
        external
        returns (int128)
    {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];
            (State memory state, Balance memory balance) = getStateAndBalance(
                productId,
                subaccount
            );
            if (balance.vQuoteBalance < 0) {
                int128 insuranceCover = MathHelper.min(
                    insurance,
                    -balance.vQuoteBalance
                );
                insurance -= insuranceCover;
                balance.vQuoteBalance += insuranceCover;
                state.availableSettle += insuranceCover;

                // actually socialize if still not enough
                if (balance.vQuoteBalance < 0) {
                    // socialize across all other participants
                    int128 fundingPerShare = -balance.vQuoteBalance.div(
                        state.openInterest
                    );
                    state.cumulativeFundingLongX18 += fundingPerShare;
                    state.cumulativeFundingShortX18 -= fundingPerShare;
                    balance.vQuoteBalance = 0;
                }
                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
        }
        return insurance;
    }

    function socializeSubaccount(bytes32 subaccount) external {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];

            State memory state = states[productId];
            Balance memory balance = balanceNormalizedToBalance(
                state,
                balances[productId][subaccount]
            );
            if (balance.amount < 0) {
                int128 totalDeposited = state.totalDepositsNormalized.mul(
                    state.cumulativeDepositsMultiplierX18
                );

                state.cumulativeDepositsMultiplierX18 = (totalDeposited +
                    balance.amount).div(state.totalDepositsNormalized);

                require(state.cumulativeDepositsMultiplierX18 > 0);

                state.totalBorrowsNormalized += balance.amount.div(
                    state.cumulativeBorrowsMultiplierX18
                );

                _setBalanceAndUpdateBitmap(
                    productId,
                    subaccount,
                    BalanceNormalized({amountNormalized: 0})
                );
                _setState(productId, state);
            }
        }
    }
