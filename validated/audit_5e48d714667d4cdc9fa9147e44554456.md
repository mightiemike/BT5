Looking at the external bug class — a threshold/guard formula that uses pre-action state but the action itself changes that state, causing the threshold to be exceeded post-action — I need to find an analog in the Metric OMM extension guards.

**Systematic analysis of each guard:**

**`PriceVelocityGuardExtension.beforeSwap`** [1](#0-0) 

State is written (`lastMidPriceX64 = midPrice`) before the check, but the check uses `prevMid` (captured before the write). If the check reverts, the write is also reverted. The oracle price is external and not changed by the swap itself, so there is no "action changes the measured quantity" feedback loop. The squared-change formula `changeE18^2 <= maxChange^2 * (1 + blockDiff)` is mathematically equivalent to the documented invariant. No threshold-exceeded analog here.

**`OracleValueStopLossExtension.afterSwap / _afterSwapOracleStopLoss`** [2](#0-1) 

The stop-loss reads post-swap bin balances (pool storage already updated before `afterSwap` fires) and computes per-share metrics against decayed watermarks. The check correctly operates on post-action state. The arithmetic-mean mid price (`(bid + ask) / 2`) differs from the geometric mean the pool uses for settlement, but this is a metric-accuracy issue, not a threshold-exceeded feedback loop. No analog.
