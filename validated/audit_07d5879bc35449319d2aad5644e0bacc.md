### Title
Non-Standard ERC20 `approve` Return Value Causes Collateral Lock in `DirectDepositV1.creditDeposit` - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` via the `IIERC20Base` interface, which declares `approve` as returning `bool`. For non-standard ERC20 tokens (e.g., mainnet USDT) that emit no return value from `approve`, the Solidity ABI decoder checks that the return data is exactly 1 byte and reverts when it is 0 bytes. This permanently blocks collateral from being credited to a subaccount, locking user funds in the `DirectDepositV1` contract.

---

### Finding Description

`IIERC20Base` in `DirectDepositV1.sol` declares:

```solidity
function approve(address spender, uint256 amount) external returns (bool);
``` [1](#0-0) 

When `creditDeposit()` executes the direct interface call at line 92:

```solidity
token.approve(address(endpoint), balance);
``` [2](#0-1) 

the compiler emits ABI-decoding logic that requires exactly 1 byte of return data. Tokens like mainnet USDT return nothing from `approve`, so `data.length == 0`, and the implicit ABI check reverts the transaction.

This is structurally inconsistent with the `safeTransfer` helper defined in the same contract, which explicitly tolerates a zero-length return:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    "Transfer failed"
);
``` [3](#0-2) 

`ERC20Helper` — the protocol's safe-transfer library — provides `safeTransfer` and `safeTransferFrom` with the same tolerance pattern, but **no `safeApprove` equivalent** exists anywhere in the codebase. [4](#0-3) 

`creditDeposit()` carries no access modifier, so it is callable by any unprivileged address directly, or via `ContractOwner.creditDepositV1()`, which is also unrestricted: [5](#0-4) 

---

### Impact Explanation

A user who sends a non-standard ERC20 token to their `DirectDepositV1` address and then calls `creditDeposit()` (or `ContractOwner.creditDepositV1()`) will have every invocation revert at the `approve` line. The tokens remain in the `DirectDepositV1` contract and cannot be credited to the subaccount. The only recovery path is the owner calling `DirectDepositV1.withdraw()`, which is `onlyOwner` — the user has no self-service recovery. This constitutes a concrete collateral lock: the user's asset balance is stranded and inaccessible to them.

A secondary instance exists in `ContractOwner.wrapVaultAsset()`, which first pulls asset tokens out of `DirectDepositV1` into `ContractOwner` via `withdraw()`, then calls:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [6](#0-5) 

If `assetToken` is non-standard, the approve reverts after the tokens have already left `DirectDepositV1`, stranding them in `ContractOwner` with no existing recovery function.

---

### Likelihood Explanation

`creditDeposit()` is the primary deposit path for the `DirectDepositV1` flow and is callable by any user. Any protocol deployment that lists a non-standard ERC20 (e.g., mainnet USDT, BNB, or similar tokens with no return value on `approve`) as a supported spot product will trigger this revert for every user attempting to credit that token. The likelihood is high on any chain where such tokens are listed.

---

### Recommendation

Replace the direct interface `approve` call with a low-level safe-approve pattern, consistent with how `safeTransfer` is already implemented in `DirectDepositV1` and `ERC20Helper`:

```solidity
// In DirectDepositV1.creditDeposit():
// Replace:
token.approve(address(endpoint), balance);

// With a safe variant:
(bool success, bytes memory data) = address(token).call(
    abi.encodeWithSelector(IIERC20Base.approve.selector, address(endpoint), balance)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    "Approve failed"
);
```

Add an equivalent `safeApprove` to `ERC20Helper` and use it in `ContractOwner.wrapVaultAsset()` for the `assetToken.approve` calls at lines 530–531. [7](#0-6) 

---

### Proof of Concept

1. Protocol lists mainnet USDT as a supported spot product.
2. User sends USDT to their `DirectDepositV1` address.
3. User (or anyone) calls `DirectDepositV1.creditDeposit()` or `ContractOwner.creditDepositV1(subaccount)`.
4. Execution reaches `token.approve(address(endpoint), balance)` at line 92.
5. USDT's `approve` executes successfully on-chain but returns no data (`data.length == 0`).
6. The Solidity ABI decoder for the `returns (bool)` interface expects 1 byte; finds 0 bytes; reverts.
7. The user's USDT remains in `DirectDepositV1`, uncredited, inaccessible without owner intervention. [8](#0-7)

library ERC20Helper {
    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }

    function safeTransferFrom(
        IERC20Base self,

                amount
