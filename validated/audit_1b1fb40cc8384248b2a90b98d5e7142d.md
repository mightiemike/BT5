### Title
Precommit Vote Signature Preimage Omits Height, Round, and Chain ID, Enabling Cross-Height and Cross-Chain Replay — (`crates/apollo_signature_manager/src/signature_manager.rs`)

---

### Summary

`build_precommit_vote_message_digest` constructs the signed message as `PRECOMMIT_VOTE || block_hash` only. The `height`, `round`, `vote_type`, and `chain_id` fields present in every `Vote` struct are excluded from the preimage. When `verify_precommit_vote_signature` is wired into `handle_vote` (currently deferred by a `TODO`), a single valid precommit signature over a block hash will be accepted as valid for that same block hash at any height, any round, and on any chain — a direct analog to the EIP-7683 nonce-replay issue where a stateless, context-free signature can be reused across distinct protocol instances.

---

### Finding Description

`build_precommit_vote_message_digest` in `crates/apollo_signature_manager/src/signature_manager.rs` constructs the signed digest as:

```
blake2s( b"PRECOMMIT_VOTE" || block_hash.to_bytes_be() )
``` [1](#0-0) 

The `Vote` wire type carries `height`, `round`, `vote_type`, and `voter` alongside the `proposal_commitment` (block hash) and `signature`: [2](#0-1) 

None of `height`, `round`, `vote_type`, or `chain_id` are committed into the signed message. The signing entry point confirms only `block_hash` is passed: [3](#0-2) 

The public verification function mirrors this incomplete preimage: [4](#0-3) 

`handle_vote` in `SingleHeightConsensus` currently skips signature verification entirely with an acknowledged TODO: [5](#0-4) 

When that TODO is resolved and `verify_precommit_vote_signature` is called, the verification will accept any `Vote` message whose `proposal_commitment` matches the signed block hash, regardless of the `height`, `round`, or `vote_type` fields in the message. The self-vote path also defers signing with the same TODO: [6](#0-5) 

The existing comment in `signature_manager.rs` acknowledges the risk of delimiter ambiguity but does not address the missing context fields: [7](#0-6) 

---

### Impact Explanation

**Impact: High.** The signature/hash logic binds the wrong payload: it commits only to the block hash and omits the height, round, vote type, and chain ID. When verification is activated:

1. **Cross-round replay**: A valid precommit for `(height=H, round=0, block_hash=B)` is cryptographically identical to a precommit for `(height=H, round=5, block_hash=B)`. An attacker who captures a validator's precommit from round 0 can replay it in a later round for the same block hash, potentially forcing a premature or incorrect quorum.

2. **Cross-height replay**: If the same block hash ever appears at two different heights (e.g., due to a reorg scenario or a crafted hash collision), the signature is valid at both heights.

3. **Cross-chain replay**: No chain ID is bound into the preimage. A precommit signature produced on a testnet or staging deployment is valid on mainnet for the same block hash.

4. **Vote-type confusion**: Because `vote_type` is not in the preimage, a precommit signature is structurally identical to a prevote signature over the same block hash. If a `sign_prevote` path is later added without its own domain separator, the two vote types become interchangeable.

This matches the allowed impact: *"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

**Likelihood: Medium.** The vulnerability is latent today because `handle_vote` does not yet call `verify_precommit_vote_signature`. However, the TODO is an explicit planned work item. Once wired in, the incomplete preimage becomes immediately exploitable by any network peer who can observe and replay a validator's precommit message at a different round or on a different chain. No privileged access is required; consensus vote messages are broadcast to all peers.

---

### Recommendation

Extend `build_precommit_vote_message_digest` (and any future `build_prevote_message_digest`) to commit to all context fields that distinguish one vote from another:

```rust
fn build_precommit_vote_message_digest(
    block_hash: BlockHash,
    height: BlockNumber,
    round: Round,
    chain_id: &ChainId,
) -> MessageDigest {
    let mut message = Vec::new();
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash.to_bytes_be());
    message.extend_from_slice(&height.0.to_be_bytes());
    message.extend_from_slice(&round.to_be_bytes());
    message.extend_from_slice(chain_id.as_bytes());
    MessageDigest(blake2s_to_felt(&message))
}
```

Correspondingly, `sign_precommit_vote` and `verify_precommit_vote_signature` must accept and pass through these additional fields. The `make_self_vote` path in `StateMachine` must call the signing function with the correct `height`, `round`, and `chain_id` when the TODO is resolved.

---

### Proof of Concept

1. Validator V signs a precommit for `(height=10, round=0, block_hash=B)`. The signed digest is `blake2s(b"PRECOMMIT_VOTE" || B)`.
2. Consensus advances to `(height=10, round=3)` with the same proposal commitment `B` (valid under Tendermint repropose rules).
3. An adversarial peer replays V's round-0 precommit message with `round=3` in the wire fields.
4. When `verify_precommit_vote_signature` is called, it recomputes `blake2s(b"PRECOMMIT_VOTE" || B)` — identical to step 1 — and returns `true`.
5. The replayed vote is accepted as a fresh round-3 precommit from V, contributing to quorum without V's knowledge or intent.

The same replay works across heights and across chains by substituting the appropriate `height` or `chain_id` in the wire message while keeping `block_hash` constant. [1](#0-0) [5](#0-4)
