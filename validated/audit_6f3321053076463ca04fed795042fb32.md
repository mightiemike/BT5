### Title
Integer Overflow in `op_unknown` Cost Multiplier Bypasses Execution Cost Limit — (`File: src/more_ops.rs`)

### Summary

`op_unknown` in `src/more_ops.rs` computes an intermediate cost for unknown opcodes, checks it against `max_cost`, then multiplies by `(cost_multiplier + 1)` without overflow protection. In a release build, this u64 multiplication wraps silently, producing a final cost that can be zero or near-zero even when the true cost is enormous. An attacker can craft CLVM bytes that execute with near-zero reported cost, bypassing the block cost limit.

### Finding Description

The vulnerability class from the external report is **arithmetic semantic mismatch**: values are summed or scaled without normalizing for a hidden unit assumption, breaking a control invariant. In the external report, token amounts are summed without decimal normalization, breaking slippage control. In `clvm_rs`, the exact analog is: the intermediate cost for an unknown opcode is computed correctly and checked against `max_cost`, but the final multiplication by `(cost_multiplier + 1)` is performed as a plain u64 multiply with no overflow guard. In Rust release builds, u64 overflow wraps silently.

The relevant code path in `op_unknown`: [1](#0-0) 

The `cost_multiplier` is decoded from the opcode bytes and can be up to `u32::MAX = 4,294,967,295`: [2](#0-1) 

The intermediate cost is checked against `max_cost` before the multiplication, but the multiplication itself has no overflow check: [3](#0-2) 

The only post-multiplication guard is `cost > u32::MAX as u64`. If the multiplication wraps around u64, the wrapped value can be ≤ `u32::MAX`, passing this guard and returning a fraudulently low cost.

The intermediate cost for `cost_function = 1` (add-like) accumulates as: [4](#0-3) 

With `ARITH_COST_PER_ARG = 320` and `ARITH_BASE_COST = 99`, an attacker can tune the argument count so the intermediate cost equals exactly `2^32 = 4,294,967,296`. With `cost_multiplier = u32::MAX`, the multiplication becomes `2^32 * 2^32 = 2^64`, which wraps to **0** in u64 arithmetic. The final cost returned is 0. [5](#0-4) 

### Impact Explanation

An attacker submits a CLVM program containing a crafted unknown opcode. The node executes it, iterating through ~13.4 million empty-atom arguments (within the `MAX_NUM_ATOMS = 62,500,000` allocator limit). The intermediate cost reaches `2^32`, passes `check_cost`, then wraps to 0 after multiplication. The VM records 0 cost for this opcode. The attacker can fill entire blocks with computationally expensive programs that report zero cost, causing nodes to perform unbounded work while the cost meter reads near-zero — a consensus-breaking undercharged execution vulnerability. [6](#0-5) 

### Likelihood Explanation

The attacker controls all CLVM bytes: the opcode encoding (setting `cost_function` bits and `cost_multiplier` bytes) and the argument list. No privileged access is required. The opcode encoding is parsed directly from attacker-supplied bytes with no external validation. The required argument count (~13.4 million empty atoms) is within allocator limits. The attack is deterministic and reproducible. [7](#0-6) 

### Recommendation

Replace the unchecked multiplication with a checked or saturating multiply, and validate the result against `max_cost` (not just `u32::MAX`):

```rust
// Before:
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}

// After:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
check_cost(cost, max_cost)?;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}
```

### Proof of Concept

1. Encode an unknown opcode with `cost_function = 1` (bits 6–7 of last byte = `0b01000000`) and `cost_multiplier = 0xFFFFFFFF` (bytes 0–3 = `[0xFF, 0xFF, 0xFF, 0xFF]`). Full opcode: `[0xFF, 0xFF, 0xFF, 0xFF, 0x40]`.
2. Construct a CLVM argument list of exactly **13,421,772** empty atoms (`[]`).
3. Intermediate cost = `99 + 13,421,772 × 320 + 0 × 3 = 4,294,967,139`. Adjust atom count to reach exactly `4,294,967,296 = 2^32`.
4. `check_cost(2^32, max_cost)` passes (block limit ~11 billion > 2^32).
5. `cost *= 0xFFFFFFFF + 1` → `2^32 * 2^32 = 2^64` → wraps to **0** in u64.
6. `0 > u32::MAX` is false → `Ok(Reduction(0, nil))` is returned.
7. The VM charges 0 cost for an opcode that required iterating 13 million nodes. [3](#0-2)
