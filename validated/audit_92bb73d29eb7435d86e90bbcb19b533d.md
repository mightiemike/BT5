### Title
Unprotected `initialize()` in `SpotEngine` Allows Front-Running to Permanently Misconfigure the Engine — (`core/contracts/SpotEngine.sol`)

---

### Summary

`SpotEngine.initialize()` is declared `external` with no access control and no `initializer` modifier on the outer function. Any unprivileged caller can front-run the legitimate deployment call and invoke it with attacker-controlled parameters, permanently seizing ownership of the engine and corrupting the `canApplyDeltas` authorization map before the protocol can do so.

---

### Finding Description

`SpotEngine.initialize()` delegates to the internal `BaseEngine._initialize()`, which carries the OpenZeppelin `initializer` modifier. The outer function itself has no guard:

```solidity
// SpotEngine.sol L14-20
function initialize(
    address _clearinghouse,
    address _offchainExchange,
    address _quote,
    address _endpoint,
    address _admin
) external {                          // ← no `initializer`, no access control
    _initialize(_clearinghouse, _offchainExchange, _endpoint, _admin);
    ...
}
``` [1](#0-0) 

The internal `_initialize()` in `BaseEngine` does carry `initializer`, but that only means it runs once — it does not prevent an arbitrary caller from being the one who triggers it first:

```solidity
// BaseEngine.sol L203-218
function _initialize(...) internal initializer {
    __Ownable_init();
    setEndpoint(_endpointAddr);
    transferOwnership(_admin);          // ← attacker becomes owner
    _clearinghouse = IClearinghouse(_clearinghouseAddr);
    canApplyDeltas[_endpointAddr]       = true;
    canApplyDeltas[_clearinghouseAddr]  = true;
    canApplyDeltas[_offchainExchangeAddr] = true;  // ← attacker-controlled addresses
}
``` [2](#0-1) 

Contrast this with `Clearinghouse.initialize()`, `Endpoint.initialize()`, and `OffchainExchange.initialize()`, all of which place the `initializer` modifier directly on the public/external function: [3](#0-2) [4](#0-3) 

`SpotEngine` is the only engine whose outer `initialize()` is unguarded.

The legitimate initialization path is `Clearinghouse.addEngine()` (owner-only), which calls `productEngine.initialize(...)`: [5](#0-4) 

Because `SpotEngine.initialize()` is `external` with no guard, an attacker can call it directly before `addEngine()` is ever invoked.

---

### Impact Explanation

If an attacker front-runs `SpotEngine.initialize()`:

1. **Ownership seizure**: `transferOwnership(_admin)` runs with the attacker's address, making the attacker the owner of `SpotEngine`. The attacker can then call `addOrUpdateProduct()` (which is `onlyOwner`) to register arbitrary products with manipulated risk parameters.
2. **`canApplyDeltas` corruption**: The attacker populates `canApplyDeltas` with their own malicious contract addresses. These addresses can then call `updateBalance()` on any subaccount without restriction, directly minting or burning spot balances.
3. **Permanent misconfiguration**: Because `initializer` runs exactly once, the legitimate `Clearinghouse.addEngine()` call will revert when it tries to call `productEngine.initialize()`, leaving the entire spot engine permanently under attacker control or requiring a full redeployment.

The corrupted state delta is: `canApplyDeltas[attacker_address] = true` and `owner = attacker`, enabling arbitrary `updateBalance()` calls on all spot products including the quote token.

---

### Likelihood Explanation

The attack requires only watching the mempool for the proxy deployment transaction and submitting a front-running call to `SpotEngine.initialize()` with a higher gas price. No privileged access, leaked keys, or social engineering is needed. The window exists between proxy deployment and the owner's `addEngine()` call. This is a realistic, low-effort attack on any EVM chain including Ink Chain.

---

### Recommendation

Add the `initializer` modifier directly to `SpotEngine.initialize()` (and `PerpEngine.initialize()` if it follows the same pattern), matching the pattern used by `Clearinghouse`, `Endpoint`, and `OffchainExchange`:

```solidity
function initialize(
    address _clearinghouse,
    address _offchainExchange,
    address _quote,
    address _endpoint,
    address _admin
) external initializer {   // ← add initializer here
    _initialize(_clearinghouse, _offchainExchange, _endpoint, _admin);
    ...
}
```

Alternatively, add an `onlyOwner` or factory-only access check, or deploy and initialize atomically in a single transaction.

---

### Proof of Concept

1. Attacker monitors the mempool for the `SpotEngine` proxy deployment.
2. Before `Clearinghouse.addEngine()` is called, attacker calls:
   ```solidity
   SpotEngine(proxyAddress).initialize(
       attackerClearinghouse,
       attackerOffchainExchange,
       quoteToken,
       attackerEndpoint,
       attackerAddress   // becomes owner
   );
   ```
3. `BaseEngine._initialize()` runs: `transferOwnership(attackerAddress)`, `canApplyDeltas[attackerOffchainExchange] = true`.
4. Legitimate `Clearinghouse.addEngine()` calls `productEngine.initialize(...)` → reverts because `initializer` already ran.
5. Attacker, now owner, calls `addOrUpdateProduct()` with zero-weight risk stores, or calls `updateBalance()` via their `canApplyDeltas`-whitelisted contract to mint arbitrary spot balances. [6](#0-5) [7](#0-6)
