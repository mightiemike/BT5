Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual user, allowing allowlist bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist, but when a swap is routed through `MetricOmmSimpleRouter`, `sender` is the router contract address — not the originating user. A pool admin who allowlists the router to support legitimate router-mediated swaps inadvertently grants every user, including non-allowlisted ones, the ability to bypass the guard entirely.

## Finding Description
The call chain is:

1. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(recipient, ...)` with no mechanism to forward the original `msg.sender` (the real user) into the pool's extension path. [1](#0-0) 

2. `MetricOmmPool.swap` passes `msg.sender` — which is the router — as the `sender` argument to `_beforeSwap`. [2](#0-1) 

3. `ExtensionCalling._beforeSwap` forwards that value verbatim to the extension. [3](#0-2) 

4. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. [4](#0-3) 

There is no existing guard that recovers the original initiator. The router stores the real caller only in its internal callback context (`_setNextCallbackContext`) for payment purposes, and never surfaces it to the pool or extension layer.

## Impact Explanation
A pool deploying `SwapAllowlistExtension` is explicitly designed to restrict who may trade. Once the router is allowlisted (the only operational path to let legitimate users swap through the supported periphery), the allowlist becomes a no-op for all router-mediated swaps. Any user can trade on a pool intended to be restricted, executing swaps against LPs who deposited under the assumption that only vetted counterparties could trade. This is a broken core pool invariant: the configured access-control guard does not enforce the intended policy on the primary public entrypoint.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who deploys a curated pool and wants legitimate allowlisted users to use the router must allowlist the router address — this is the expected operational step, not an edge case. The bypass is therefore reachable by any user on any curated pool that supports router access, which is the common deployment pattern. No special privileges or unusual conditions are required.

## Recommendation
The extension must check the original user, not the intermediary. The router already knows `msg.sender` (the real user) at entry. It should encode and forward this address in `extensionData` (or a dedicated parameter if the interface is extended), and `SwapAllowlistExtension.beforeSwap` should decode and check that value when `sender` is a known router. The `DepositAllowlistExtension` pattern — checking `owner`, the economically relevant actor explicitly set by the caller — is the correct model to follow.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Admin calls setAllowedToSwap(pool, router, true)   // allowlist router for legitimate users
  - Admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  - attacker (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(...) → msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  - Swap executes for the non-allowlisted attacker

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds — allowlist bypassed
```
