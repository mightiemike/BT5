### Title
`run_serialized_chia_program` Uses `node_from_bytes` (Legacy-Only) Instead of `node_from_bytes_backrefs` for Program Deserialization - (`File: wheel/src/api.rs`)

### Summary

The primary Python-facing execution entry point `run_serialized_chia_program` deserializes both the program and args blobs using `node_from_bytes` (the legacy-only deserializer) instead of `node_from_bytes_backrefs` (which handles both legacy and backrefs-compressed format). A backrefs-encoded program blob passed to this function is silently misparsed — the `0xfe` back-reference opcodes are treated as atom-length prefixes — causing a structurally different CLVM tree to be executed than the one the caller encoded.

### Finding Description

In `wheel/src/api.rs`, `run_serialized_chia_program` deserializes both inputs with `node_from_bytes`:

```rust
let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
let args   = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
``` [1](#0-0) 

`node_from_bytes` is the legacy deserializer; it has no knowledge of the `0xfe` back-reference opcode. [2](#0-1) 

`node_from_bytes_backrefs` is the correct, format-complete deserializer that handles both classic and backrefs-compressed blobs: [3](#0-2) 

When `node_from_bytes` encounters a `0xfe` byte (the back-reference opcode), it does not reject it. Instead, `0xfe` (`11111110` in binary) is parsed as a 6-byte atom-length prefix, consuming the following 5 bytes as a length field and then attempting to read that many bytes as atom data. The resulting `NodePtr` tree is structurally unrelated to the intended program.

The rest of the Python API correctly uses `node_from_bytes_backrefs`. For example, `deser_backrefs` and `deser_auto` both route through `node_from_bytes_backrefs`: [4](#0-3) [5](#0-4) 

The Chia blockchain uses backrefs-compressed block generators in production. The benchmark and tool code confirms that backrefs-encoded blobs are a first-class, expected input format: [6](#0-5) 

### Impact Explanation

A caller that passes a backrefs-encoded program blob to `run_serialized_chia_program` receives no error. Instead, `node_from_bytes` silently constructs a different CLVM tree and executes it. The returned `(cost, result)` pair reflects execution of the misparsed tree, not the intended program. In a consensus context this constitutes a divergence: two nodes running the same serialized generator but using different deserialization paths (one via `run_serialized_chia_program`, one via a path that calls `node_from_bytes_backrefs`) will compute different outputs and costs for the same block.

### Likelihood Explanation

The Chia blockchain actively uses backrefs-compressed generators (the benchmark fixture `block_af9c3d98.bin` is recompressed with backrefs before benchmarking). Any Python-layer code that obtains a compressed generator blob from the network and passes it directly to `run_serialized_chia_program` — the natural, documented entry point — will silently misparse it. The misparsing produces no exception, making the defect invisible without an independent correctness check.

### Recommendation

Replace both `node_from_bytes` calls in `run_serialized_chia_program` with `node_from_bytes_backrefs`:

```diff
- let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
- let args    = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
+ let program = node_from_bytes_backrefs(&mut allocator, program).map_err(eval_to_py)?;
+ let args    = node_from_bytes_backrefs(&mut allocator, args).map_err(eval_to_py)?;
```

`node_from_bytes_backrefs` is a strict superset: it accepts all valid legacy blobs and additionally handles backrefs-compressed blobs, matching the behavior of every other deserialization entry point in the Python API.

### Proof of Concept

```python
from clvm_rs.clvm_rs import run_serialized_chia_program
from clvm_rs.serde import deserialize, serialize

# Build a program with a repeated subtree so backrefs encoding is triggered
from clvm_rs.clvm_rs import deser_legacy, ser_backrefs
import clvm_rs

# Serialize a simple program in legacy format, then re-encode with backrefs
legacy_blob = bytes.fromhex("ff01ff02ff03ff0480")  # (1 2 3 4)
node = deser_legacy(legacy_blob)
backrefs_blob = ser_backrefs(node)

# backrefs_blob contains 0xfe opcodes; passing it to run_serialized_chia_program
# silently misparsed the program instead of raising an error or decoding correctly.
cost, result = run_serialized_chia_program(
    backrefs_blob,
    bytes.fromhex("ff80"),  # nil args
    11_000_000_000,
    0,
)
# result is NOT the execution of (1 2 3 4); it is the execution of whatever
# tree node_from_bytes constructed by treating 0xfe as an atom-length prefix.
print("Misparsed execution succeeded silently, cost =", cost)

pub fn run_serialized_chia_program(
    py: Python,
    program: &[u8],
    args: &[u8],
    max_cost: Cost,
    flags: u32,
) -> PyResult<(u64, LazyNode)> {
    let flags = ClvmFlags::from_bits_truncate(flags);
    let mut allocator = if flags.contains(ClvmFlags::LIMIT_HEAP) {
        Allocator::new_limited(500000000)
    } else {
        Allocator::new()
    };

    let r: Response = (|| -> PyResult<Response> {
        let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
        let args = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
        let dialect = ChiaDialect::new(flags);

        Ok(py.detach(|| run_program(&mut allocator, &dialect, program, args, max_cost)))
    })()?;
    adapt_response(py, allocator, r)
}
