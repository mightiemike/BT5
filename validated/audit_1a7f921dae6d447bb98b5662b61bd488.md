### Title
Python Object Pointer Aliasing in `identity_map` Cache Causes Wrong `NodePtr` Reuse — (`File: wheel/src/api.rs`)

---

### Summary

`clvm_tree_to_lazy_node` caches visited Python CLVM nodes in `identity_map: HashMap<usize, NodePtr>` keyed by raw Python object pointer (`pyobj.as_ptr() as usize`). Python object pointers are only unique for the *lifetime* of the object. After a visited node is dropped and garbage-collected, CPython can immediately reuse that memory address for a new, structurally different Python object. When that new object is later encountered during tree traversal, the stale cache entry is hit and the wrong `NodePtr` is silently returned, producing a corrupted CLVM tree.

---

### Finding Description

In `clvm_tree_to_lazy_node` (`wheel/src/api.rs`, lines 192–301), three caches are maintained:

- `identity_map: HashMap<usize, NodePtr>` — keyed by `pyobj.as_ptr() as usize` (raw CPython object address)
- `atom_map: HashMap<Vec<u8>, NodePtr>` — keyed by atom byte content (safe)
- `pair_map: HashMap<(NodePtr, NodePtr), NodePtr>` — keyed by child `NodePtr` pair (safe) [1](#0-0) 

The `identity_map` key is the raw CPython heap pointer, equivalent to Python's `id()`. CPython's `id()` contract guarantees uniqueness only while the object is alive. Once a `Bound<'py, PyAny>` is dropped at the end of a `Visit` match arm, the Python object's reference count is decremented. If no other Python-side reference holds the object alive (e.g., in a lazy CLVM tree where `.pair` constructs children on-the-fly), the object is immediately deallocated and its memory slot returned to CPython's small-object allocator. [2](#0-1) 

CPython's small-object allocator (`obmalloc`) aggressively reuses freed slots. A subsequent `allocator.new_atom(...)` or any Python-level object construction can land at the identical address. When the traversal later encounters this new object Y at address `X_ptr`, the guard:

```rust
if identity_map.contains_key(&id) {
    continue;
}
``` [3](#0-2) 

fires a false positive. The traversal skips Y entirely and the `BuildPair` work item later resolves `identity_map[&Y_ptr]` to `NodePtr_X` — the `NodePtr` of the previously visited, now-dead object X. [4](#0-3) 

The result is a `LazyNode` whose internal tree structure silently substitutes node X's content wherever node Y should appear.

---

### Impact Explanation

The corrupted `LazyNode` is returned to the Python caller with no error. Any downstream use — `run_serialized_chia_program`, `ser_legacy`, `ser_backrefs`, `ser_2026`, or direct tree-hash computation — operates on the wrong tree. Concretely:

- **Consensus divergence**: a puzzle or solution serialized from the corrupted `LazyNode` produces a different SHA-256 tree hash than the original Python tree, causing full-node validation to disagree.
- **Silent wrong-program execution**: `run_program` executes the substituted subtree, producing a different cost and return value with no error raised.
- **Incorrect serialized bytes**: `node_to_bytes` / `node_to_bytes_backrefs` emit bytes that decode to a different CLVM object than the caller intended.

**Impact: Medium** — the corruption is silent, produces no exception, and propagates into all downstream operations on the returned `LazyNode`.

---

### Likelihood Explanation

**Likelihood: Medium.**

The trigger requires a Python CLVM tree where child nodes are not strongly held by their parents — i.e., a lazy or generator-backed implementation where `.pair` constructs fresh Python objects on each access. This is a realistic pattern: the `clvm` Python library's `CLVMObject` can be subclassed or wrapped to produce children lazily. CPython's `obmalloc` pool reuse is deterministic and fast; in practice, the freed slot is the first candidate for the next same-size allocation, making address collision highly probable in tight traversal loops. An attacker who controls the Python object passed to `clvm_tree_to_lazy_node` can craft such a tree.

---

### Recommendation

Hold strong references to all visited Python objects for the duration of the traversal, preventing garbage collection and address reuse:

```rust
// Add alongside identity_map:
let mut live_objects: Vec<Bound<'py, PyAny>> = Vec::new();

// In the Visit arm, after resolving the node, before dropping pyobj:
live_objects.push(pyobj);
```

Alternatively, key `identity_map` by a stable content-derived identifier (e.g., atom bytes for atoms; recursive structural hash for pairs) rather than the ephemeral CPython pointer. The existing `atom_map` and `pair_map` already do this correctly and can serve as the sole deduplication mechanism, eliminating `identity_map` entirely.

### Proof of Concept

```python
import gc, clvm_rs

class LazyAtom:
    """Each access to .atom creates a fresh bytes object."""
    def __init__(self, val: bytes):
        self._val = val
    @property
    def atom(self):
        return bytes(self._val)   # new object every call
    @property
    def pair(self):
        return None

class LazyPair:
    def __init__(self, left, right):
        self._left = left
        self._right = right
    @property
    def atom(self):
        return None
    @property
    def pair(self):
        # Returns brand-new wrapper objects each time
        return (LazyAtom(self._left), LazyAtom(self._right))

# Build a tree: (b"AAA" . b"BBB")
root = LazyPair(b"AAA", b"BBB")

# Force a GC cycle between visits to maximise address reuse
gc.collect()

node = clvm_rs.clvm_tree_to_lazy_node(root)
result = clvm_rs.ser_legacy(node)

# Verify: if aliasing fires, result encodes the wrong subtree
# (e.g., both sides become b"AAA" because the second LazyAtom
#  lands at the same address as the first and hits the stale cache entry)
print(result.hex())
```
