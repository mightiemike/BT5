### Title
Fee-on-Transfer Token Accounting Corruption in Deposit Flow — (`File: core/contracts/EndpointStorage.sol`)

---

### Summary

`EndpointStorage.handleDepositTransfer` performs a two-hop token transfer (user → Endpoint → Clearinghouse) using the caller-supplied `amount` for both legs. If a supported spot product token later introduces a fee-on-transfer, the Clearinghouse receives less than `amount` but the queued slow-mode transaction records the full `amount`. When the slow-mode transaction is processed, `Clearinghouse.depositCollateral` credits the full `amount` to the subaccount, inflating balances beyond actual holdings. No mechanism exists to remove or disable a product token once registered.

---

### Finding Description

The deposit entry path is:

1. A user calls `Endpoint.depositCollateral()` or `depositCollateralWithReferral()`.
2. `handleDepositTransfer` is called with the user-supplied `amount`: [1](#0-0) 

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);              // user → Endpoint
    safeTransferTo(token, address(clearinghouse), amount); // Endpoint → Clearinghouse
}
```

3. Immediately after, a slow-mode transaction is queued encoding the original `amount`: [2](#0-1) 

4. When the slow-mode transaction is executed, `Clearinghouse.depositCollateral` credits `amountRealized` derived directly from `txn.amount` — the original `amount`, not the actual received amount: [3](#0-2) 

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
```

**With a fee-on-transfer token:**
- Leg 1 (`safeTransferFrom`): Endpoint receives `amount - fee1`, not `amount`.
- Leg 2 (`safeTransferTo`): Endpoint attempts to forward `amount` to Clearinghouse, but only holds `amount - fee1`. If the Endpoint has a pre-existing token balance (from prior deposits or dust), this transfer succeeds and Clearinghouse receives `amount - fee2`. If not, it reverts — but the slow-mode queue entry is already written before the revert, creating a desync.
- `Clearinghouse.depositCollateral` credits the full `amount` regardless.

**No removal mechanism exists.** `SpotEngine.addOrUpdateProduct` only adds or updates entries; `BaseEngine._addOrUpdateProduct` only ever pushes to the `productIds` array: [4](#0-3) 

There is no `removeProduct`, `disableProduct`, or equivalent function anywhere in the codebase. Once a token is registered as a spot product, deposits using it cannot be halted at the contract level.

---

### Impact Explanation

The protocol invariant `sum(subaccount balances for productId) == actual token balance held by Clearinghouse` is broken. Subaccount balances are inflated relative to actual holdings. Users can withdraw amounts that exceed what the Clearinghouse actually holds for that token, draining funds belonging to other depositors and causing protocol insolvency for that product.

---

### Likelihood Explanation

Moderate. Tokens like USDT have fee-on-transfer mechanisms that are currently disabled but can be activated by the token issuer at any time. Once activated, every deposit call through `depositCollateral` or `depositCollateralWithReferral` by any unprivileged user corrupts the accounting. The entry points are public and require no special privileges. [5](#0-4) 

---

### Recommendation

1. **Measure actual received amount** in `handleDepositTransfer` using a before/after `balanceOf` check, and pass the delta (not the input `amount`) into the slow-mode transaction payload.
2. **Add a product disablement mechanism** to `SpotEngine` (e.g., a `disableProduct(uint32 productId)` flag checked in `depositCollateral`) so that if a token's behavior changes, new deposits can be halted while existing balances are wound down.

---

### Proof of Concept

1. Token `T` is registered as a spot product with `productId = 5` (fee currently 0%).
2. Token issuer activates a 1% fee-on-transfer on `T`.
3. Alice calls `Endpoint.depositCollateral("default", 5, 1000e6)`.
4. `handleDepositTransfer` runs:
   - `safeTransferFrom(T, Alice, 1000e6)` → Endpoint receives `990e6` (1% fee deducted).
   - `safeTransferTo(T, clearinghouse, 1000e6)` → if Endpoint has ≥ `1000e6` pre-existing balance, Clearinghouse receives `990e6`; Endpoint's balance decreases by `1000e6`.
5. Slow-mode tx is queued with `amount = 1000e6`.
6. After the 3-day delay, `Clearinghouse.depositCollateral` executes: `amountRealized = 1000e6 * multiplier`. Alice's subaccount is credited `1000e6` worth of balance.
7. Alice withdraws `1000e6`, but Clearinghouse only received `990e6` from her deposit — the `10e6` shortfall is taken from other depositors' funds.
8. Repeated deposits continuously widen the gap between recorded balances and actual holdings. [1](#0-0) [6](#0-5)

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
