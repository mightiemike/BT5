### Title
`get_split_parent_shard_uids` Stamps Parent Shard UIDs with the Child Layout's Version, Producing Wrong DB Keys at Every Resharding Boundary — (`core/primitives/src/shard_layout/mod.rs`)

### Summary

`ShardLayout::get_split_parent_shard_uids()` constructs `ShardUId` values for parent shards using `self.version()` — the version of the **new** (post-split) layout — instead of the version of the **previous** layout where those parent shards actually live. Because `ShardUId` is the 8-byte DB key prefix under which all shard state is stored, every consumer of this function receives a UID that does not match any real DB entry, silently misdirecting memtrie preloading and resharding state lookups at every shard-layout upgrade boundary.

### Finding Description

`ShardUId` encodes `(version: u32, shard_id: u32)` as an 8-byte little-endian key prefix. All trie nodes, flat-storage entries, and `ChunkExtra` records for a shard are stored under the UID that was current **when that shard was created** — i.e., the previous layout's version.

`get_split_parent_shard_uids()` is documented as returning "all the shards from the **previous** shard layout that were split into multiple shards in this shard layout." However, the implementation stamps those parent shard IDs with `self.version()`, the version of the **new** layout:

```rust
// core/primitives/src/shard_layout/mod.rs  lines 428-434
pub fn get_split_parent_shard_uids(&self) -> BTreeSet<ShardUId> {
    let parent_shard_ids = self.get_split_parent_shard_ids();
    parent_shard_ids
        .into_iter()
        .map(|shard_id| ShardUId::new(self.version(), shard_id))  // ← wrong version
        .collect()
}
``` [1](#0-0) 

The correct construction would use the **previous** layout's version (i.e., `prev_layout.version()`), because the parent shard's state is stored under that version. The function has no access to the previous layout and silently substitutes the wrong version.

The divergent value is exact and deterministic: for a V2→V3 resharding transition, `self.version()` returns the V3 constant `VERSION` (defined in `v3.rs`), while the parent shard's actual DB key uses the V2 layout's `version` field — a different integer. The test in `shard_tracker.rs` explicitly asserts that versions change across a resharding boundary:

```rust
// chain/epoch-manager/src/shard_tracker.rs  line 849
assert_ne!(non_parent_shard_new_uid.version, not_parent_shard_uid.version);
``` [2](#0-1) 

The incorrect UID propagates directly into `get_resharding_parent_shard_uid()` in `adapter.rs`, which is the sole source of truth for which shard's memtrie to preload before resharding executes:

```rust
// chain/epoch-manager/src/adapter.rs  lines 919-922
let split_parent_shard_uids = next_layout.get_split_parent_shard_uids();
debug_assert!(split_parent_shard_uids.len() == 1);
Ok(split_parent_shard_uids.into_iter().next())
``` [3](#0-2) 

`get_resharding_parent_shard_uid()` is called from two production paths documented in `dynamic_resharding.md`: (1) `maybe_start_memtrie_preload_for_resharding()` at the epoch boundary to start background memtrie loading, and (2) `Chain::new()` at startup to detect and synchronously load a pending resharding parent. Both paths receive the wrong UID and therefore cannot locate the parent shard's memtrie in the active memtrie map (which is keyed by the correct, previous-layout UID).

### Impact Explanation

When the shard layout version changes at a resharding boundary (e.g., V2→V3 under dynamic resharding), the parent shard's memtrie is never found and never preloaded. The resharding manager's `split_shard()` path then either panics (if `allow_resharding_without_memtries` is false) or proceeds without the parent memtrie, producing empty or incorrect child shard state roots. Incorrect state roots committed to `ChunkExtra` and block headers propagate as consensus-level state corruption: validators computing the correct state root will disagree with the block producer, causing chain stalls or forks at every dynamic resharding event.

### Likelihood Explanation

The bug is triggered unconditionally at every resharding event where the layout version changes. Dynamic resharding (`ProtocolFeature::DynamicResharding`) is the production path for all future shard splits; the first V2→V3 transition activates the discrepancy. No special attacker action is required — the protocol's own epoch-boundary logic calls `get_resharding_parent_shard_uid()` automatically. The only mitigation would be if `allow_resharding_without_memtries` is set to `true` in production, which would silently produce wrong child state rather than panicking.

### Recommendation

`get_split_parent_shard_uids()` must be given access to the previous layout's version. The simplest fix is to add a `prev_version: ShardVersion` parameter and use it instead of `self.version()`:

```rust
pub fn get_split_parent_shard_uids(&self, prev_version: ShardVersion) -> BTreeSet<ShardUId> {
    let parent_shard_ids = self.get_split_parent_shard_ids();
    parent_shard_ids
        .into_iter()
        .map(|shard_id| ShardUId::new(prev_version, shard_id))
        .collect()
}
```

All call sites — especially `get_resharding_parent_shard_uid()` — must pass `current_layout.version()` (the layout **before** the split) as `prev_version`. A regression test should assert that the returned UID matches the UID under which the parent shard's `ChunkExtra` is actually stored in the DB.

### Proof of Concept

Consider a network undergoing its first dynamic resharding (V2→V3):

- **Current layout**: `ShardLayoutV2` with `version = 2`, containing shard `S` (stored in DB under `ShardUId { version: 2, shard_id: S }`)
- **Next layout**: `ShardLayoutV3` with constant `VERSION = 3`, splitting `S` into children `C1`, `C2`

Call `next_layout.get_split_parent_shard_uids()`:
- `get_split_parent_shard_ids()` correctly returns `{S}`
- `ShardUId::new(self.version(), S)` = `ShardUId { version: 3, shard_id: S }` ← **wrong**

`get_resharding_parent_shard_uid()` returns `ShardUId { version: 3, shard_id: S }`.

`maybe_start_memtrie_preload_for_resharding()` calls `spawn_background_memtrie_loading_for_shard(ShardUId { version: 3, shard_id: S })`. The memtrie map contains `ShardUId { version: 2, shard_id: S }` — the lookup returns `None`. No preloading occurs. At the resharding block, `split_shard()` finds no parent memtrie and either panics or produces empty child state, causing all validators to compute a different post-state root than the block producer, halting the chain.
