### Title
`op_modpow` (Opcode 60) Completely Blocked in Mempool Mode While Valid in Consensus Creates Coin-Locking Divergence — (`File: src/chia_dialect.rs`)

---

### Summary

The `DISABLE_OP` flag, included in `MEMPOOL_MODE`, completely disables `op_modpow` (opcode 60) during mempool validation. However, `op_modpow` is a fully valid consensus operator with no flag requirement. This creates a mempool/consensus divergence: any coin whose puzzle uses `modpow` is accepted on-chain but permanently rejected by the mempool, locking the coin's funds from normal spend paths.

---

### Finding Description

In `src/chia_dialect.rs`, the operator dispatch for opcode 60 unconditionally returns `Unimplemented` whenever `DISABLE_OP` is set:

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
``` [1](#0-0) 

`DISABLE_OP` is unconditionally included in `MEMPOOL_MODE`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [2](#0-1) 

In contrast, the other arithmetic operators that also check `DISABLE_OP` — `op_div`, `op_divmod`, and `op_mod` — are only **restricted** for large inputs (> 2048 bytes), not completely disabled:

```rust
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
``` [3](#0-2) [4](#0-3) [5](#0-4) 

`op_modpow` itself already enforces a 256-byte operand size limit internally:

```rust
if bsize > 256 || esize > 256 || msize > 256 {
    return Err(EvalErr::InvalidOpArg(input, "modpow".to_string()));
}
``` [6](#0-5) 

So the complete disable in mempool mode is not justified by any missing size guard — `op_modpow` already self-limits. The asymmetry is the bug: `div/divmod/mod` are restricted for large inputs; `modpow` is unconditionally killed.

---

### Impact Explanation

A coin puzzle that uses `modpow` (opcode 60) with any operand size is:

- **Accepted by consensus** — no flag is required; the operator is in the main dispatch table unconditionally when `DISABLE_OP` is absent.
- **Rejected by the mempool** — `MEMPOOL_MODE` sets `DISABLE_OP`, causing `Unimplemented` to be returned for every invocation of opcode 60, regardless of operand size.

The result is a permanent mempool/consensus divergence. Any coin locked behind a puzzle that calls `modpow` cannot be spent through the normal mempool path. Honest full nodes running mempool validation will reject the spend transaction. The coin's value is effectively frozen: it exists on-chain and is spendable in principle, but no standard node will relay or include the spend.

This is the direct analog of M-36: the `DISABLE_OP` guard plays the role of `onlyWhenVaultIsOn`, and `op_modpow` plays the role of `withdrawalRequest` — a legitimate operation that is blocked by an overly broad state check even though the underlying state (consensus) permits it.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: an adversary crafts a coin puzzle that embeds a `modpow` call (e.g., as part of a puzzle hash commitment or a signature-checking scheme). The coin is funded normally. Once funded, the coin cannot be spent through any standard mempool path. The attacker does not need any special privilege — only the ability to create a coin, which is the normal user action. The `MEMPOOL_MODE` flag is the standard mode used by Chia full nodes for transaction validation, as confirmed by its exposure in the Python wheel API. [7](#0-6) 

---

### Recommendation

Apply the same pattern used for `op_div`, `op_divmod`, and `op_mod`: when `DISABLE_OP` is set, restrict `op_modpow` only for inputs exceeding a threshold (e.g., the existing 256-byte internal limit), rather than completely disabling the operator. Alternatively, remove `op_modpow` from the `DISABLE_OP` check entirely, since it already enforces its own operand size bounds.

---

### Proof of Concept

The following CLVM program uses `modpow` with small, valid inputs:

```
(modpow (q . 2) (q . 10) (q . 1000))
```

- **Consensus mode** (`ClvmFlags::empty()`): executes successfully, returns `24` (2^10 mod 1000).
- **Mempool mode** (`MEMPOOL_MODE` / `DISABLE_OP` set): returns `EvalErr::Unimplemented(op)` at the dispatch point in `src/chia_dialect.rs` line 241, before `op_modpow` is ever called.

Any coin whose puzzle reduces to a call of opcode 60 — regardless of operand size — will be permanently unspendable through the mempool, while remaining valid on-chain. [1](#0-0) [2](#0-1)
