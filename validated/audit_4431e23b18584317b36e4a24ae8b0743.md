Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates on Router Address Instead of End User, Enabling Allowlist Bypass - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` enforces the allowlist against `sender`, which resolves to the direct caller of `pool.swap()` (e.g., `MetricOmmSimpleRouter`), not the end user initiating the trade. When a pool admin allowlists the router so that allowlisted users can swap through it, every non-allowlisted address gains unrestricted swap access by routing through the same router. The analogous `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the economically relevant actor), making the swap extension inconsistent and broken.

## Finding Description
In `MetricOmmPool.swap`, `_beforeSwap` is called with `msg.sender` as the first argument: [1](#0-0) 

This `msg.sender` becomes `sender` in the extension. `SwapAllowlistExtension.beforeSwap` then checks: [2](#0-1) 

When a user routes through `MetricOmmSimpleRouter`, the router calls `pool.swap(...)`, so `msg.sender` to the pool is the router address. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. If the admin adds the router to the allowlist (a natural action to support router-based swaps for allowlisted users), the check passes for **any** caller of the router, including non-allowlisted users.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` explicitly ignores the direct caller (unnamed first parameter) and gates on `owner`: [3](#0-2) 

The deposit extension correctly identifies the economically relevant actor. The swap extension does not — it identifies the intermediary contract instead of the user who initiated the trade.

## Impact Explanation
Once the router is allowlisted, the curated pool's swap access control is silently nullified for all router-routed traffic. Non-allowlisted users gain full swap access to a pool designed to restrict trading, enabling unauthorized extraction of LP value from a pool whose LPs deposited under the assumption that only vetted counterparties could trade against them. This constitutes broken core pool functionality (access control) with direct potential for LP fund loss.

## Likelihood Explanation
The router is the primary user-facing swap interface. Any pool admin who deploys a `SwapAllowlistExtension`-protected pool and wants their allowlisted users to use the router will naturally call `setAllowedToSwap(pool, router, true)`. There is no documentation warning that this grants unrestricted swap access to all users. The precondition is a routine, well-intentioned admin action with no indication of risk.

## Recommendation
Gate the allowlist on the economically relevant actor, mirroring `DepositAllowlistExtension`. The second parameter of `beforeSwap` is `recipient`; using `recipient` as the gated identity is the minimal fix consistent with the deposit extension pattern. A more robust solution requires the actual user identity to be passed through `extensionData` and verified with a signature or trusted forwarder pattern. At minimum, document clearly that allowlisting the router grants unrestricted swap access to all users and provide a separate router-allowlisting path distinct from per-user allowlisting.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, userA, true)` and `setAllowedToSwap(pool, userB, true)`.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` so that `userA` and `userB` can use `MetricOmmSimpleRouter`.
4. Non-allowlisted `userC` calls `MetricOmmSimpleRouter.exactInput(...)` targeting the pool.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. `beforeSwap` evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. `userC` successfully swaps on a pool that was intended to restrict them, bypassing the allowlist entirely.
