### Title
Linked Signer Can Unilaterally Replace Itself via Fast-Mode `LinkSigner`, Permanently Hijacking Subaccount Signing Authority ‚Äî (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` handler in `EndpointTx.sol` accepts a signature from the **current linked signer** to overwrite `linkedSigners[subaccount]` with any new address. Because `validateSignedTx` is called with `allowLinkedSigner=true`, the subaccount owner's key is never required. A compromised or malicious linked signer can silently replace itself with an attacker-controlled address, permanently seizing signing authority over the subaccount without the owner's knowledge or consent.

---

### Finding Description

In `EndpointTx.sol`, the fast-mode `LinkSigner` branch decodes the transaction and calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner=true` resolves the permitted signer as either the subaccount owner address OR the current linked signer: [2](#0-1) 

`Verifier.validateSignature` then accepts either the owner or the linked signer as a valid signer: [3](#0-2) 

This means the **current linked signer** can produce a valid `LinkSigner` transaction that writes a new address into `linkedSigners[subaccount]`: [4](#0-3) 

**Contrast with slow-mode:** The slow-mode `LinkSigner` path enforces `validateSender(txn.sender, sender)`, which requires `msg.sender == address(uint160(bytes20(txn.sender)))` ‚Äî only the subaccount owner can call it: [5](#0-4) 

The fast-mode path has no equivalent restriction. The linked signer is a delegate, not an owner, yet it can mutate the very mapping that defines who the delegate is.

---

### Impact Explanation

Once the linked signer replaces itself with an attacker-controlled address, the attacker holds full signing authority over the subaccount. They can:

- Sign `WithdrawCollateral` / `WithdrawCollateralV2` transactions to drain all collateral to an external address.
- Sign `TransferQuote` transactions to move quote balances to attacker-controlled subaccounts.
- Sign arbitrary orders through `OffchainExchange`, opening or closing positions to manipulate the subaccount's health.
- Sign a further `LinkSigner` to lock out the original owner permanently (the owner's slow-mode revocation attempt will consume a nonce, but the attacker can race ahead in fast-mode).

The corrupted state is `linkedSigners[subaccount]` ‚Äî a single storage slot whose mutation grants unbounded authority over all subsequent signed operations for that subaccount. [6](#0-5) 

---

### Likelihood Explanation

The trigger requires a compromised linked signer key. Linked signers are explicitly designed to be hot-wallet API keys used by trading bots and automated systems ‚Äî they are materially more likely to be compromised than a cold-wallet subaccount owner key. Any user who has ever set a linked signer is exposed. The attack requires no privileged protocol access, no governance capture, and no sequencer compromise ‚Äî only the ability to submit a sequencer batch containing a well-formed `LinkSigner` transaction signed by the current linked signer key.

---

### Recommendation

Restrict the fast-mode `LinkSigner` handler to accept only a signature from the **subaccount owner** (i.e., call `validateSignedTx` with `allowLinkedSigner = false`):

```solidity
// EndpointTx.sol, fast-mode LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // <-- was true; linked signer must NOT be able to replace itself
);
```

This aligns the fast-mode path with the invariant already enforced by the slow-mode path: only the subaccount owner address may modify the linked signer mapping. [7](#0-6) 

---

### Proof of Concept

1. Alice owns subaccount `alice_sub` (first 20 bytes = `alice_addr`). She sets `linkedSigners[alice_sub] = bot_addr` via slow-mode.
2. Attacker compromises `bot_addr`'s private key.
3. Attacker constructs a `SignedLinkSigner` struct:
   ```
   tx.sender = alice_sub
   tx.signer = bytes32(uint256(uint160(attacker_addr)))
   tx.nonce  = nonces[alice_addr]   // current nonce
   signature = sign(digest, bot_key) // signed by the linked signer
   ```
4. Attacker submits this as a fast-mode sequencer transaction.
5. `validateSignedTx(..., allowLinkedSigner=true)` resolves `getLinkedSigner(alice_sub) = bot_addr`, recovers `bot_addr` from the signature ‚Äî check passes.
6. `linkedSigners[alice_sub]` is overwritten with `attacker_addr`.
7. Attacker now signs a `WithdrawCollateralV2` transaction with `attacker_key`, specifying `sendTo = attacker_eoa`, draining Alice's collateral. [8](#0-7) [9](#0-8)

        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
