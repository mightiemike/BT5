### Title
Missing Zero Address Checks in `Endpoint.initialize()` for Critical Protocol Addresses - (File: core/contracts/Endpoint.sol)

### Summary
`Endpoint.initialize()` sets six critical protocol addresses — `sequencer`, `clearinghouse`, `offchainExchange`, `verifier`, `sanctions`, and `endpointTx` — without any zero address validation. Because the function is guarded by the `initializer` modifier, these assignments are effectively permanent (equivalent to immutable variables). A misconfigured deployment with any of these set to `address(0)` permanently bricks the corresponding protocol path with no recovery mechanism.

### Finding Description
In `Endpoint.initialize()`, all six address parameters are assigned directly to storage without zero address guards: [1](#0-0) 

Specifically:
- `sequencer` — controls which address may submit sequenced transactions via `onlySequencer`
- `clearinghouse` — receives all deposited collateral via `handleDepositTransfer` in `EndpointStorage`
- `offchainExchange` — routes all off-chain matched orders
- `verifier` — validates all sequencer and fast-withdrawal signatures
- `sanctions` — called on every deposit/withdrawal via `requireUnsanctioned`
- `endpointTx` — target of every `delegatecall` for transaction processing

The `initializer` modifier (OpenZeppelin upgradeable pattern) ensures `initialize` can only be called once, making these assignments permanent — directly analogous to `immutable` variables in the reported bug.

An identical pattern exists in `BaseWithdrawPool._initialize()`, which sets `clearinghouse` and `verifier` without zero checks: [2](#0-1) 

And in `Clearinghouse.initialize()`, which sets `_endpoint`, `_quote`, `_clearinghouseLiq`, and `_withdrawPool` without zero checks: [3](#0-2) 

### Impact Explanation
- If `clearinghouse` is `address(0)` in `Endpoint`, `handleDepositTransfer` in `EndpointStorage` transfers all deposited collateral to `address(0)`, permanently destroying user funds: [4](#0-3) 
- If `sequencer` is `address(0)`, the `onlySequencer` modifier blocks all sequenced transaction processing, halting the entire protocol.
- If `verifier` is `address(0)` in `BaseWithdrawPool`, `submitFastWithdrawal` calls `Verifier(address(0)).requireValidTxSignatures(...)`, reverting all fast withdrawals permanently: [5](#0-4) 
- If `quote` is `address(0)` in `Clearinghouse`, all collateral accounting is broken at the quote token level.
- No upgrade or admin function exists to correct these values post-initialization.

### Likelihood Explanation
Likelihood is low-to-medium. The trigger is a deployment-time misconfiguration rather than a runtime attacker action. However, the absence of guards means a scripting error, copy-paste mistake, or misconfigured deployment pipeline silently produces a permanently broken protocol with no on-chain recovery path. The severity of the consequence (permanent fund loss or full protocol halt) elevates the overall risk to medium, matching the original report's classification.

### Recommendation
Add explicit zero address checks for all critical address parameters in each initializer:

```solidity
// Endpoint.initialize()
require(_sequencer != address(0), "zero sequencer");
require(address(_clearinghouse) != address(0), "zero clearinghouse");
require(_offchainExchange != address(0), "zero offchainExchange");
require(_verifier != address(0), "zero verifier");
require(_sanctions != address(0), "zero sanctions");
require(_endpointTx != address(0), "zero endpointTx");

// BaseWithdrawPool._initialize()
require(_clearinghouse != address(0), "zero clearinghouse");
require(_verifier != address(0), "zero verifier");

// Clearinghouse.initialize()
require(_endpoint != address(0), "zero endpoint");
require(_quote != address(0), "zero quote");
require(_clearinghouseLiq != address(0), "zero clearinghouseLiq");
require(_withdrawPool != address(0), "zero withdrawPool");
```

### Proof of Concept

1. Deploy the `Endpoint` proxy contract.
2. Call `Endpoint.initialize(address(0), address(0), validOffchainExchange, validClearinghouse, validVerifier, validEndpointTx)` — passing `address(0)` for `_sanctions` and `_sequencer`.
3. The `initializer` modifier marks the contract as initialized; no second call is possible.
4. Any subsequent call to `depositCollateral` triggers `requireUnsanctioned`, which calls `sanctions.isSanctioned(sender)` on `address(0)`, reverting with a low-level call failure — permanently blocking all deposits.
5. Any sequenced transaction submission fails the `onlySequencer` check since `sequencer == address(0)` and `msg.sender` can never equal `address(0)`.
6. The protocol is permanently non-functional with no on-chain remediation path. [6](#0-5) [7](#0-6)
