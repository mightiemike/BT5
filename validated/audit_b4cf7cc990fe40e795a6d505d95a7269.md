### Title
Small `quoteAmount` in `mintNlp` Truncates to Zero NLP Tokens Due to Integer Division — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

In `Clearinghouse.sol`, the `mintNlp` function computes the NLP token amount via integer division. When `quoteAmount` is sufficiently small relative to `oraclePriceX18`, the result truncates to zero. The user's quote tokens are still fully deducted and distributed to NLP pools, but the user receives zero NLP tokens in return — a direct accounting corruption where deposited value is not reflected in the minted balance.

---

### Finding Description

In `mintNlp`, the NLP amount is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [1](#0-0) 

The `div` function in `MathSD21x18` is:

```solidity
int256 result = (int256(x) * ONE_X18) / y;
``` [2](#0-1) 

This computes `(quoteAmount × 1e18) / oraclePriceX18`. Since `oraclePriceX18` is itself in X18 format (i.e., `price × 1e18`), truncation to zero occurs when:

```
quoteAmount × 1e18 < oraclePriceX18
⟹ quoteAmount < price  (in X18 units)
```

For example, with NLP price = $1000 (`oraclePriceX18 = 1e21`), any `quoteAmount < 1000` (in X18 units) yields `nlpAmount = 0`.

After `nlpAmount = 0`, the execution continues:

```solidity
spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, nlpAmount);   // credits 0 NLP
spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, -nlpAmount);   // debits 0 from N_ACCOUNT
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -quoteAmount); // deducts full quote
_applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);     // distributes quote to pools
``` [3](#0-2) 

The user's quote tokens are fully deducted and distributed to NLP pools, but their NLP balance is not updated. There is no minimum-amount guard anywhere in this path.

The final health check does not prevent this:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [4](#0-3) 

A user with sufficient quote balance will pass this check even after losing the dust `quoteAmount`.

The user-signed digest covers only `sender`, `quoteAmount`, and `nonce` — not `oraclePriceX18`:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(MINT_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.quoteAmount,
        signedTx.tx.nonce
    )
);
``` [5](#0-4) 

The user controls `quoteAmount` directly. No admin or sequencer compromise is required to trigger the truncation — the user simply submits a small `quoteAmount` that is below the truncation threshold for the current oracle price.

---

### Impact Explanation

The broken invariant is: a user who deposits `quoteAmount > 0` quote tokens into `mintNlp` must receive a proportional positive NLP balance. When `nlpAmount` truncates to zero, the user's quote tokens are permanently transferred to NLP pools with no NLP credit issued. This is a direct, unrecoverable loss of the deposited quote tokens for the user, and a corresponding unearned gain for NLP pool holders. The accounting state is corrupted: `totalDepositsNormalized` for the quote product decreases (user's quote is removed) while `totalDepositsNormalized` for the NLP product does not increase (no NLP is minted to the user).

The amounts involved are dust-level (sub-`price`-unit in X18 terms), matching the external report's characterization of "minuscule" impact. However, unlike the VotingEscrow analog where funds are only temporarily frozen, here the user's quote tokens are permanently redistributed to NLP pools with no recourse.

---

### Likelihood Explanation

A user submitting a `MintNlp` transaction with a small `quoteAmount` is the direct trigger. This is unlikely to occur accidentally at meaningful scale, but requires no special privilege — any user with a valid subaccount and a signed `MintNlp` transaction can trigger it. As NLP price appreciates over time, the truncation threshold rises, making accidental triggering more plausible for users testing with minimal amounts.

---

### Recommendation

Add a post-division guard in `mintNlp` before proceeding with balance updates:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
require(nlpAmount > 0, "ERR_INSUFFICIENT_QUOTE_AMOUNT");
```

This mirrors the recommendation in the external report to establish a minimum deposit requirement, and ensures the invariant that a non-zero deposit always produces a non-zero minted balance.

---

### Proof of Concept

Assume NLP oracle price = $1000, so `oraclePriceX18 = 1000 × 1e18 = 1e21`.

1. User signs and submits `MintNlp` with `quoteAmount = 999` (in X18 units, ≈ 999 × 10⁻¹⁸ of the quote token).
2. `nlpAmount = (999 × 1e18) / 1e21 = 0` — truncated by integer division.
3. `spotEngine.updateBalance(NLP_PRODUCT_ID, sender, 0)` — user NLP balance unchanged.
4. `spotEngine.updateBalance(QUOTE_PRODUCT_ID, sender, -999)` — user quote balance decremented by 999.
5. `_applyNlpRebalance(...)` — 999 units of quote distributed to NLP pools.
6. Health check passes (user had sufficient quote balance).

Result: user's quote balance is reduced by 999 X18 units; user's NLP balance is unchanged; NLP pool holders receive the 999 units. The contract accepted the deposit but delivered zero NLP tokens.
