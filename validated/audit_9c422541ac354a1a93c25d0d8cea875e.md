### Title
Unbacked `minDepositRateX18` Accrual Inflates Deposit Balances Beyond Actual Token Holdings, Causing Withdrawal Revert — (`core/contracts/SpotEngineState.sol`)

---

### Summary

`SpotEngineState._updateState()` unconditionally applies a minimum deposit rate (`minDepositRateX18`) to `cumulativeDepositsMultiplierX18` even when utilization is zero and no borrower interest is being collected. This inflates protocol-tracked deposit balances beyond the actual token holdings of the Clearinghouse. Because there is no cap on total deposits and no emergency withdrawal path, users whose accrued balance exceeds the contract's real token balance will have their withdrawal permanently reverted.

---

### Finding Description

In `_updateState()`, after the standard borrow/deposit rate update, the following block executes unconditionally whenever `minDepositRateX18 != 0`: [1](#0-0) 

Both `cumulativeBorrowsMultiplierX18` and `cumulativeDepositsMultiplierX18` are multiplied by `minDepositRateMultiplierX18`. When utilization is zero (no borrowers), `totalBorrowsNormalized = 0`, so the borrow multiplier growing has no accounting effect. However, `totalDepositsNormalized > 0`, so the deposit multiplier growing **directly inflates every depositor's tracked balance**:

```
Alice's tracked balance = totalDepositsNormalized * cumulativeDepositsMultiplierX18
```

No new tokens enter the Clearinghouse to back this growth. The standard interest path already handles the zero-utilization case correctly (sets `borrowerRateX18 = 0` at line 75, yielding `depositRateMultiplierX18 = ONE`), but the `minDepositRateX18` block bypasses this and applies growth regardless. [2](#0-1) 

The `assertUtilization` guard only checks `totalDeposits >= totalBorrows`; it does not verify that the contract's actual token balance covers `totalDeposits - totalBorrows`: [3](#0-2) 

There is no cap on total deposits in `depositCollateral`, so the gap between tracked balances and real holdings is unbounded: [4](#0-3) 

---

### Impact Explanation

When a user requests withdrawal of their full balance (which includes accrued `minDepositRateX18` yield), the sequencer calls `withdrawCollateral`, which calls `handleWithdrawTransfer`: [5](#0-4) 

`token.safeTransfer(withdrawPool, uint256(amount))` will revert with an ERC-20 insufficient-balance error because the Clearinghouse holds fewer tokens than the protocol-tracked balance. There is no emergency withdrawal path that skips the transfer or pays only available funds. The withdrawal transaction is permanently stuck until the admin manually deposits additional tokens to cover the shortfall — an unpredictable and unbounded amount that grows with total deposits and elapsed time.

---

### Likelihood Explanation

`minDepositRateX18` is a first-class protocol parameter set via `addOrUpdateProduct` and is already initialized to a non-zero value for the quote product in `SpotEngine.initialize` (`minDepositRateX18: 0` for quote, but configurable for other products). Any product configured with `minDepositRateX18 > 0` and experiencing a period of low utilization (common at launch or during market downturns) will accumulate unbacked deposit credits. The gap grows continuously with time and total deposit volume, both of which are unbounded. No attacker action is required beyond normal depositing and waiting.

---

### Recommendation

1. **Track the minimum-rate liability separately**: When applying `minDepositRateX18`, credit the difference between the minimum rate and the actual earned rate to a dedicated reserve account that must be funded by the protocol treasury, rather than silently inflating `cumulativeDepositsMultiplierX18` without a corresponding token source.
2. **Add a maximum total deposit cap per product**: Bound the total deposits so the protocol can pre-fund the worst-case minimum-rate liability.
3. **Add an emergency withdrawal function**: Allow users to withdraw up to the contract's actual available token balance, bypassing the full-balance requirement, analogous to the `emergencyWithdraw()` fix in the referenced report.

---

### Proof of Concept

**Setup**: Product configured with `minDepositRateX18 = 0.05e18` (5% annual), zero utilization.

**Step 1** — Alice deposits 100 USDC. Clearinghouse holds 100 USDC. Protocol records `totalDepositsNormalized = 100e18`, `cumulativeDepositsMultiplierX18 = 1e18`.

**Step 2** — One year passes. `updateStates(dt=31536000)` is called by the sequencer. Because utilization = 0, the standard path yields `depositRateMultiplierX18 = ONE`. But the `minDepositRateX18` block then executes:

```
minDepositRateMultiplierX18 ≈ 1.05e18
cumulativeDepositsMultiplierX18 ≈ 1.05e18
```

Alice's tracked balance = `100e18 * 1.05e18 / 1e18 = 105e18` → 105 USDC. Clearinghouse still holds 100 USDC. [6](#0-5) 

**Step 3** — Alice submits a withdrawal for 105 USDC. Sequencer calls `withdrawCollateral(alice, productId, 105e6, alice, idx)`.

**Step 4** — `handleWithdrawTransfer` calls `token.safeTransfer(withdrawPool, 105e6)`. The Clearinghouse only holds 100e6 USDC → **revert**. Alice cannot withdraw her funds. [7](#0-6)
