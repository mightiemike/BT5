### Title
Stale `tier.BonusApy` Applied Retroactively to All Historical Bonus Segments in `processEventsAndClaimBonus` — (File: x/tieredrewards/keeper/claim_rewards.go)

---

### Summary

When a user claims bonus rewards, `processEventsAndClaimBonus` fetches the **current** `tier.BonusApy` once and applies it uniformly to every historical time segment, including segments that accrued before a governance-approved change to that rate. The `tokensPerShare` rate is correctly snapshotted per-segment inside validator events, but `tier.BonusApy` is never snapshotted. After a governance parameter update, all unclaimed historical bonus is retroactively recalculated at the new rate, causing overpayment or underpayment from the `RewardsPoolName` module account.

---

### Finding Description

In `processEventsAndClaimBonus`, the tier is loaded once from current on-chain state:

```go
tier, err := k.getTier(ctx, pos.TierId)
``` [1](#0-0) 

This single `tier` object — carrying the **current** `BonusApy` — is then passed into every call to `computeSegmentBonus` for every historical segment:

```go
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
``` [2](#0-1) 

and again for the final live segment:

```go
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
``` [3](#0-2) 

Inside `computeSegmentBonus`, the bonus formula is:

```go
tokens := pos.Delegation.Shares.Mul(tokensPerShare)
return tokens.Mul(tier.BonusApy).MulInt64(durationSeconds).QuoInt64(types.SecondsPerYear).TruncateInt()
``` [4](#0-3) 

The asymmetry is clear: `tokensPerShare` **is** snapshotted at each validator event boundary (stored in `ValidatorEvent.TokensPerShare` by the staking hooks), but `tier.BonusApy` is **never** snapshotted. There is no tier-parameter-change event type, no per-segment `BonusApy` record, and no checkpoint flush triggered when governance updates a tier.

The `SetTier` function — explicitly described as used by governance messages, genesis, and chain upgrades — writes the new `BonusApy` directly to the store with no side-effect on outstanding positions:

```go
func (k Keeper) SetTier(ctx context.Context, tier types.Tier) error {
    if err := tier.Validate(); err != nil { return err }
    if err := k.Tiers.Set(ctx, tier.Id, tier); err != nil { ... }
    return nil
}
``` [5](#0-4) 

Contrast this with how `AddToTierPosition` and `TierUndelegate` both call `claimRewards` **before** any state change, properly settling accrued bonus at the old share count before the position changes: [6](#0-5) [7](#0-6) 

No equivalent flush is performed before a governance tier update.

---

### Impact Explanation

The corrupted value is the bonus coins sent from the `RewardsPoolName` module account to position owners via:

```go
k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins)
``` [8](#0-7) 

- **If `BonusApy` is increased by governance**: every position that has not claimed since before the change receives the higher rate for the entire unclaimed history, draining the rewards pool faster than intended.
- **If `BonusApy` is decreased by governance**: every position that has not claimed since before the change receives the lower rate for the entire unclaimed history, underpaying users for rewards they legitimately earned.

Both directions represent a broken accounting invariant: the rewards pool balance and the bonus owed to users are desynchronized from the rate that was actually in effect during each accrual period.

---

### Likelihood Explanation

**Low.** The trigger requires a governance vote to change `tier.BonusApy`, which is a normal and expected protocol operation (tier parameters are designed to be adjustable). Once a governance change passes, every position owner who claims rewards without having claimed immediately before the change is affected. The user-side action (`MsgClaimTierRewards`) is fully unprivileged and reachable by any position owner.

---

### Recommendation

Snapshot `tier.BonusApy` at each segment boundary, mirroring the existing pattern for `tokensPerShare`. Concretely:

1. Introduce a `TIER_RATE_CHANGE` validator-style event (or a separate per-tier event log) that records the old `BonusApy` and a timestamp whenever `SetTier` is called with a changed `BonusApy`.
2. In `processEventsAndClaimBonus`, replay tier-rate-change events alongside validator events, using the snapshotted `BonusApy` for each segment rather than the current value.
3. Alternatively, force-flush all positions in a tier (call `claimRewardsAndUpdateTierPositions`) inside `SetTier` before persisting the new `BonusApy`, so no historical accrual remains at the old rate.

---

### Proof of Concept

1. Governance sets tier 1 with `BonusApy = 0.10` (10 %).
2. Alice calls `MsgLockTier` at block time **T₁**, creating position P with 1 000 000 shares. No validator events occur.
3. At block time **T₂** (e.g., 6 months later), governance passes a proposal calling `SetTier` with `BonusApy = 0.20` (20 %). The store is updated; no rewards are flushed.
4. At block time **T₃** (1 day after T₂), Alice calls `MsgClaimTierRewards`.
5. `processEventsAndClaimBonus` fetches `tier.BonusApy = 0.20`.
6. The single segment `[T₁, T₃]` (~183 days) is computed entirely at 20 % APY.
7. Alice receives ≈ 2× the bonus she legitimately earned for `[T₁, T₂]` (the 6-month pre-change period), draining the `RewardsPoolName` account by the excess.

The root cause — `tier` fetched once at current state before iterating historical segments — is a direct analog to the external report's pattern of using un-refreshed rates/weights when computing token amounts during a withdrawal operation. [9](#0-8)

func (k Keeper) processEventsAndClaimBonus(ctx context.Context, pos *types.PositionState) (sdk.Coins, error) {
	// Rewards should have been claimed before undelegation
	if !pos.IsDelegated() {
		return sdk.NewCoins(), nil
	}

	valAddr, err := sdk.ValAddressFromBech32(pos.Delegation.ValidatorAddress)
	if err != nil {
		return nil, err
	}

	events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	blockTime := sdkCtx.BlockTime()

	totalBonus := math.ZeroInt()
	// Use the persisted bonded state from the last replay, not a hardcoded default.
	// This prevents overpaying bonus for unbonded gaps between claims.
	bonded := pos.LastKnownBonded
	segmentStart := pos.LastBonusAccrual

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	for _, entry := range events {
		evt := entry.Event

		if bonded {
			// Compute bonus for the bonded segment [segmentStart, eventTime]
			// using the snapshot rate at the event.
			bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
			totalBonus = totalBonus.Add(bonus)
		}

		// Update bonded state based on event type.
		switch evt.EventType {
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND:
			bonded = false
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND:
			bonded = true
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_SLASH:
			// Slash doesn't change bonded state.
		}

		segmentStart = evt.Timestamp
		pos.UpdateLastEventSeq(entry.Seq)

		// Decrement reference count.
		if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
			return nil, err
		}
	}

	val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
	if err != nil {
		return nil, err
	}
	// Defensive: validator bond status check
	if bonded && val.IsBonded() {
		currentRate, err := k.getTokensPerShare(ctx, valAddr)
		if err != nil {
			return nil, err
		}
		bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
		totalBonus = totalBonus.Add(bonus)
	}

	applyBonusAccrualCheckpoint(&pos.Position, blockTime)
