Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the direct pool caller (router) instead of the originating user, allowing full allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed from the pool, which is always `msg.sender` at the pool level — the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router becomes `msg.sender` at the pool, so the extension checks the router's allowlist status rather than the originating user's. This renders the allowlist ineffective for any router-mediated swap.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool — the originating user's address is only stored in transient storage for the payment callback and is never forwarded to the pool as `sender`: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. There is no mechanism in `pool.swap()` to supply a separate originator address — the function signature only accepts `recipient`, not a `sender` override. [5](#0-4) 

**Mode A (full bypass):** Pool admin allowlists the router address (natural choice for "standard" access). Every user — including those explicitly not allowlisted — can swap freely through `exactInputSingle`. The extension never sees the attacker's address; it sees the router's, which passes.

**Mode B (broken legitimate access):** Pool admin allowlists specific user EOAs. Those users are rejected when calling through the router because the extension sees the router address, which is not on the allowlist. Allowlisted users are forced to call `pool.swap()` directly, bypassing slippage, deadline, and multi-hop protections.

## Impact Explanation
Mode A constitutes a direct, complete bypass of a configured access-control guard on a curated pool. Any unprivileged user can trade on a pool designed to restrict swaps to specific addresses (e.g., KYC'd counterparties, whitelisted market makers, protocol-owned addresses). This is a broken core pool functionality and an admin-boundary break: the pool admin's allowlist configuration is rendered entirely ineffective by the supported public periphery path. The wrong value is the extension's authorization decision — `allowedSwapper[pool][router] == true` is evaluated instead of `allowedSwapper[pool][attacker]`, which would be `false`.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any user wanting deadline protection, slippage checks, or multi-hop routing will naturally use it. A pool admin who configures `SwapAllowlistExtension` and adds the router to the allowlist (Mode A) or adds user EOAs (Mode B) triggers the vulnerability through entirely normal, expected usage. No special permissions, malicious setup, or non-standard tokens are required. The exploit is repeatable by any unprivileged address.

## Recommendation
The `SwapAllowlistExtension` must gate on the originating user rather than the direct pool caller. Two complementary approaches:

1. **Pool-side (cleanest):** Add an optional `originator` parameter to `pool.swap()`, mirroring how `addLiquidity` separates `sender` (caller) from `owner` (economic actor). Extensions can then check whichever identity is appropriate for their access-control semantics.

2. **Extension-side (short-term):** Until the pool interface is extended, document that `SwapAllowlistExtension` must allowlist the router address (not individual users) when router access is intended, and that per-user gating requires direct pool calls. Alternatively, consider gating on `recipient` if the pool admin controls who receives output — though this is semantically different from gating on the swapper.

## Proof of Concept
```
Setup:
  1. Deploy pool with SwapAllowlistExtension configured.
  2. Pool admin calls setAllowedToSwap(pool, router, true)
     — intending to allow all "standard" router users.
  3. Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack (Mode A — bypass):
  4. Attacker (not on allowlist) calls:
       router.exactInputSingle({pool: pool, ..., extensionData: ""})
  5. Router calls pool.swap(recipient, zeroForOne, amount, limit, "", "").
  6. Pool calls _beforeSwap(msg.sender=router, ...).
  7. SwapAllowlistExtension checks allowedSwapper[pool][router] == true → passes.
  8. Swap executes. Attacker receives output tokens.
     The allowlist check never evaluated the attacker's address.

Expected: revert NotAllowedToSwap.
Actual:   swap succeeds.
```
