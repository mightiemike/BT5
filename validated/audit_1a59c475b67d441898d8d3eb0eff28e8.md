### Title
Unchecked `transferFrom` Return Value Enables USDC.e Drain from Direct Deposit Accounts - (File: core/contracts/ContractOwner.sol)

### Summary
`ContractOwner.replaceUsdcEWithUsdc` is a permissionless external function that swaps USDC.e held in a Direct Deposit Account (DDA) for USDC. It calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the boolean return value. If this call returns `false` instead of reverting, execution continues: the DDA's USDC.e is withdrawn to `ContractOwner` and then forwarded to the caller via `safeTransfer`, draining USDC.e from the DDA with no USDC provided in exchange.

### Finding Description
`replaceUsdcEWithUsdc` is callable by any address on chain 57073 (Ink). Its three-step swap logic is:

1. Pull USDC from caller into the DDA — **raw `transferFrom`, return value ignored**
2. Pull USDC.e from the DDA into `ContractOwner` — via `DirectDepositV1.withdraw`, which uses `safeTransfer`
3. Push USDC.e from `ContractOwner` to caller — via `IERC20Base.safeTransfer` [1](#0-0) 

Step 1 uses a raw interface call whose `bool` return is silently discarded:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

Steps 2 and 3 both use `safeTransfer` / `safeTransferFrom` wrappers that enforce success. The inconsistency means a failed step 1 is invisible to the contract, while steps 2 and 3 still execute and move real USDC.e to the caller. [2](#0-1) 

`DirectDepositV1.withdraw` transfers the full USDC.e balance of the DDA to `msg.sender` (which is `ContractOwner`), and `ContractOwner` then forwards it to the attacker: [3](#0-2) 

The `ERC20Helper.safeTransfer` used for the outbound leg correctly checks success, confirming the protocol is aware of the pattern — making the omission on the inbound leg a concrete inconsistency, not a style choice: [4](#0-3) 

### Impact Explanation
Any DDA on chain 57073 that holds a USDC.e balance is drainable. An attacker calls `replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance or balance. If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on a failed `transferFrom` rather than reverting, the attacker receives the full USDC.e balance of the targeted DDA for free. The corrupted asset delta is: DDA loses `balance` USDC.e, attacker gains `balance` USDC.e, attacker provides 0 USDC. [5](#0-4) 

### Likelihood Explanation
The function has no access control beyond a chain ID check, making it reachable by any unprivileged caller. The exploitability depends on whether the specific USDC deployment at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink (chain 57073) returns `false` on failure rather than reverting. Non-standard or bridged stablecoin deployments on newer chains frequently deviate from the canonical Circle USDC behavior. Even if the current deployment reverts, the code is structurally broken: a future token upgrade or redeployment to a return-false variant would immediately open the drain path with no contract change required.

### Recommendation
Replace the raw `transferFrom` call with `ERC20Helper.safeTransferFrom`, consistent with how all other token movements in the codebase are handled:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [6](#0-5) 

### Proof of Concept
1. Identify a DDA (`directDepositV1`) with a non-zero USDC.e balance on chain 57073.
2. Call `ContractOwner.replaceUsdcEWithUsdc(subaccount)` from an EOA with zero USDC balance/allowance.
3. If USDC's `transferFrom` returns `false` (no revert): the `if (balance > 0)` branch executes, `DirectDepositV1.withdraw(usdcE)` moves the full USDC.e balance to `ContractOwner`, and `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` delivers it to the attacker.
4. Result: attacker holds `balance` USDC.e; DDA holds 0 USDC.e; DDA received 0 USDC. [7](#0-6)

        require(block.chainid == 57073, ERR_UNAUTHORIZED);

        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
