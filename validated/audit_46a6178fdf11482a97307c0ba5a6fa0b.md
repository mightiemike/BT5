### Title
Linked Signer Can Overwrite Its Own Authorization on Any Subaccount â€” (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type in the fast (sequencer-submitted) path is processed with `allowLinkedSigner = true`. This means the **existing linked signer** â€” a delegated session key â€” can sign and submit a `LinkSigner` transaction to replace itself (or set any address) as the new linked signer for a subaccount it does not own. This is an unauthorized mutation of the subaccount's authorization configuration, directly analogous to the external report's root cause: a credential that should only grant scoped access is instead accepted as proof of ownership to modify the authorization record itself.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch decodes the transaction and calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` calls `validateSignature`, which passes the result of `getLinkedSigner(sender)` as the accepted signer when `allowLinkedSigner` is `true`: [2](#0-1) 

`Verifier.validateSignature` then accepts the signature if it recovers to **either** the subaccount owner address **or** the linked signer: [3](#0-2) 

After validation passes, the linked signer mapping is unconditionally overwritten with the attacker-controlled `signedTx.tx.signer` value: [4](#0-3) 

The `linkedSigners` mapping is the on-chain authorization record that determines who may sign sensitive operations (withdrawals, liquidations, NLP mint/burn) on behalf of a subaccount: [5](#0-4) 

The slow-mode path for `LinkSigner` correctly restricts this to the owner via `validateSender`, which checks `msg.sender` against the address embedded in the subaccount bytes: [6](#0-5) [7](#0-6) 

The fast path has no equivalent ownership check â€” it accepts the linked signer's signature as sufficient authority to rewrite the linked signer record.

---

### Impact Explanation

A malicious or compromised linked signer can:

1. **Persist unauthorized access**: Sign a `LinkSigner` transaction pointing to a new address they control. The owner's only recourse is the slow-mode path, which enforces a hardcoded `SLOW_MODE_TX_DELAY` (three days).
2. **Prevent revocation via the fast path**: Because the nonce is shared per owner address, the malicious linked signer can keep submitting `LinkSigner` transactions with incrementing nonces, racing any fast-path revocation the owner submits to the sequencer.
3. **Exploit the window**: During the three-day slow-mode delay, the newly set linked signer retains the ability to sign `WithdrawCollateral` (draining collateral to the owner's own address, forcing liquidation of open positions), `MintNlp`/`BurnNlp` (at oracle prices the sequencer provides), and `LiquidateSubaccount` (consuming fees from the victim subaccount).

The corrupted state is `linkedSigners[subaccount]` â€” the authorization record that gates all high-value operations on the subaccount.

The protocol itself acknowledges the power of linked signers and adds explicit protections elsewhere. For example, `transferQuote` includes the comment *"otherwise linked signers can transfer out"* and enforces `bytes20(txn.sender) == bytes20(txn.recipient)`: [8](#0-7) 

No analogous protection exists for `LinkSigner`.

---

### Likelihood Explanation

Any user who has ever linked a signer (e.g., for high-frequency trading via a session key) is exposed. The attack requires only that the linked signer is malicious or compromised â€” no sequencer compromise, no admin access, no governance capture. The attacker-controlled entry path is: sign a valid EIP-712 `LinkSigner` struct with the existing linked signer key â†’ submit to the sequencer â†’ sequencer includes it in a batch â†’ `processTransactionImpl` executes â†’ `linkedSigners[victim_subaccount]` is overwritten. This is a realistic scenario for any protocol that encourages session-key delegation for automated trading.

---

### Recommendation

Remove `allowLinkedSigner = true` from the `LinkSigner` branch in `processTransactionImpl`. The ability to change the linked signer is an ownership-level operation and must only be authorized by the subaccount owner's key (i.e., `allowLinkedSigner = false`), consistent with how the slow-mode path already enforces this via `validateSender`.

---

### Proof of Concept

1. Alice owns subaccount `alice_default` (`bytes32(abi.encodePacked(alice_address, "default\0\0\0\0\0"))`).
2. Alice links Bob's address as a session key via a legitimate `LinkSigner` transaction.
3. Bob constructs `LinkSigner { sender: alice_default, signer: bytes32(charlie_address), nonce: current_nonce }`.
4. Bob signs the EIP-712 digest of this struct with his own key.
5. Bob submits the signed transaction to the sequencer.
6. The sequencer includes it; `processTransactionImpl` reaches the `LinkSigner` branch.
7. `validateSignedTx(..., allowLinkedSigner=true)` recovers Bob's address, matches `linkedSigners[alice_default]`, and passes.
8. `linkedSigners[alice_default]` is set to Charlie (Bob's new address).
9. Alice submits a fast-path `LinkSigner` revocation signed by her own key; Bob immediately submits another `LinkSigner` with the next nonce, racing Alice in the sequencer queue.
10. Alice falls back to slow mode; the three-day delay begins. During this window, Charlie signs `WithdrawCollateral` transactions draining Alice's collateral to her own address, forcing her leveraged positions into liquidation.
