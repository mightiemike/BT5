Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Allowing Allowlist Bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps using the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender`. When `MetricOmmSimpleRouter` calls `pool.swap`, `msg.sender` inside the pool is the router's address, not the originating user. Any non-allowlisted user can bypass the restriction by routing through the public router if the router address is itself allowlisted.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` verbatim as `sender` to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value forwarded from the pool — the router address when the call originates from `MetricOmmSimpleRouter`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly with no mechanism to encode the originating user into `extensionData`: [3](#0-2) 

This produces two broken states:
1. **Router not allowlisted** — every router-mediated swap on an allowlisted pool reverts, making the public routing path completely unusable.
2. **Router allowlisted** (required to allow any router swap) — every user, including those explicitly excluded, bypasses the allowlist by routing through `MetricOmmSimpleRouter`.

No existing guard in `SwapAllowlistExtension` or the router prevents this. The `extensionData` field is passed through from the caller unchanged: [4](#0-3) 

## Impact Explanation
A pool operator deploys `SwapAllowlistExtension` to restrict trading to a curated set (e.g., KYC'd addresses). Any excluded user routes through `MetricOmmSimpleRouter`; the allowlist check passes against the router address. The unauthorized user receives output tokens from LP assets the pool was explicitly designed to protect. This is a direct loss of LP-controlled assets to an unauthorized actor, satisfying Sherlock Medium/High thresholds for broken core pool functionality and unauthorized fund access.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical public entry point for swaps. Any user aware of the allowlist restriction has an immediate, zero-cost bypass: call the router instead of the pool directly. No privileged access, special setup, or non-standard token behavior is required. The bypass is reachable by any EOA or contract on every allowlisted pool that also permits router-mediated swaps.

## Recommendation
`SwapAllowlistExtension.beforeSwap` must gate the economically relevant actor. The cleanest fix: require the router to encode the originating caller into `extensionData` before calling `pool.swap`, and have the extension decode and check that address. Concretely, `MetricOmmSimpleRouter` encodes `msg.sender` into `extensionData` for each hop, and `SwapAllowlistExtension.beforeSwap` decodes and validates that address against `allowedSwapper[pool][originatingUser]` instead of `allowedSwapper[pool][sender]`. [5](#0-4) 

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension in BEFORE_SWAP_ORDER.
  - allowedSwapper[pool][allowedUser] = true
  - allowedSwapper[pool][attackerUser] = false
  - allowedSwapper[pool][routerAddress] = true  ← required for any router swap to work

Attack:
  1. attackerUser calls MetricOmmSimpleRouter.exactInputSingle(tokenIn, tokenOut, pool, ...)
  2. Router calls pool.swap(recipient=attackerUser, ...) → msg.sender inside pool = routerAddress
  3. Pool calls _beforeSwap(sender=routerAddress, ...)
  4. SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][routerAddress] → true → passes
  5. Swap executes; attackerUser receives output tokens from the LP-restricted pool.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds; attacker bypasses the allowlist entirely.
``` [6](#0-5) [7](#0-6)

    view
    override
