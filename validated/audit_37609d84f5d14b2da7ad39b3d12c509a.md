Let me analyze the external bug pattern and search for analogs in nearcore. The core issue is a **stale snapshot value** computed with a global multiplier at time T, stored, then used at time T' when the multiplier has changed — causing incorrect downstream calculations.

### Title
Stale `congestion_gas` Snapshot in `StateStoredReceiptMetadata` Becomes Incorrect After Protocol Fee Upgrade — (`runtime/runtime/src/congestion_control.rs`, `core/primitives/src/receipt.rs`)

### Summary

When a receipt cannot be forwarded due to congestion, it is stored in the trie as a `StateStoredReceipt` with a `StateStoredReceiptMetadata` snapshot containing `congestion_gas` — computed once at buffering time using the then-current `RuntimeConfig` fee parameters. When the receipt is later retrieved, `receipt_congestion_gas` returns the stored snapshot value directly, ignoring the current `config`. A protocol upgrade that changes action fee parameters (e.g., `ProtocolFeature::AccountCostIncrease` at version 85, which nearly doubles `action_create_account.exec_fee`) causes all receipts buffered before the upgrade to carry stale `congestion_gas` values. The congestion control system then tracks incorrect gas totals in `CongestionInfo`, causing shards to appear less congested than they are and admitting more work than they can safely handle.

### Finding Description

**Snapshot creation (push path):**

When a receipt is buffered, `compute_receipt_congestion_gas` is called with the current `config` and the result is frozen into `StateStoredReceiptMetadata`: [1](#0-0) 

```rust
let gas = compute_receipt_congestion_gas(&receipt, &config)?;
...
let metadata = StateStoredReceiptMetadata { congestion_gas: gas, congestion_size: size };
```

`compute_receipt_congestion_gas` calls `action_receipt_congestion_gas`, which uses `total_prepaid_exec_fees(config, ...)` and `total_prepaid_send_fees(config, ...)` — both of which depend on versioned fee parameters: [2](#0-1) 

**Stale retrieval (pop path):**

When the receipt is later retrieved, `receipt_congestion_gas` returns the stored snapshot value and **completely ignores the `config` argument**: [3](#0-2) 

```rust
ReceiptOrStateStoredReceipt::StateStoredReceipt(receipt) => {
    Ok(receipt.metadata().congestion_gas)   // config is ignored
}
```

**The fee change at the upgrade boundary:**

`ProtocolFeature::AccountCostIncrease` (protocol version 85) changes `action_create_account.exec_fee` from `3_850_000_000_000` to `7_200_000_000_000` — an ~87% increase: [4](#0-3) 

Any receipt containing a `CreateAccount` action that was buffered before version 85 carries a `congestion_gas` value computed with the old, lower fee. After the upgrade, every pop of such a receipt decrements `CongestionInfo.buffered_receipts_gas` (or `delayed_receipts_gas`) by the stale lower value, causing the running total to understate the true gas load by `3_350_000_000_000` gas per affected receipt.

**The `StateStoredReceipt` format** was introduced at protocol version 72 (`_DeprecatedStateStoredReceipt`), so any receipt buffered between v72 and v85 is affected: [5](#0-4) 

**The `CongestionInfo` is maintained incrementally** — the same stale value is added on push and subtracted on pop, so the running total is internally self-consistent but diverges from the true gas cost under the new config. The `bootstrap_congestion_info` path also calls `receipt_congestion_gas` and inherits the same stale values: [6](#0-5) 

### Impact Explanation

The `CongestionInfo.buffered_receipts_gas` and `delayed_receipts_gas` fields are used to compute the congestion level of a shard, which controls:
- Transaction admission (whether new transactions targeting the shard are accepted)
- Outgoing gas limits (how many receipts can be forwarded to the shard per chunk)
- Bandwidth requests generated by the bandwidth scheduler

With understated gas totals, the shard appears less congested than it is. The congestion control system allows more receipts to be forwarded and more transactions to be admitted than the shard can safely process. During a protocol upgrade that increases action fees (like v85), every shard that had buffered `CreateAccount`-containing receipts before the upgrade will have its congestion level understated for the duration those receipts remain in the queue. In a heavily loaded network, this can cause chunk production delays or missed chunks — a liveness impact.

### Likelihood Explanation

The `StateStoredReceipt` format has been active since protocol version 72. Protocol version 85 (`AccountCostIncrease`) changes action fees significantly. Any shard that experienced congestion between v72 and v85 and buffered receipts with `CreateAccount` or `DeterministicStateInit` actions will exhibit this divergence immediately after the v85 upgrade. No privileged access is required — any user can submit transactions that create such receipts when a shard is congested.

### Recommendation

When a `StateStoredReceipt` is dequeued, recompute `congestion_gas` from the live receipt body using the current `config` rather than returning the stored snapshot. Alternatively, on protocol upgrade, migrate existing `StateStoredReceipt` entries in the trie to recompute their metadata under the new config. A lighter mitigation is to add a protocol-version tag to `StateStoredReceiptMetadata` and recompute on retrieval when the stored version differs from the current one.

### Proof of Concept

1. At protocol version 84, shard S is congested. A user submits a transaction with a `CreateAccount` action targeting shard S. The receipt is buffered; `congestion_gas` is stored as `G_old` (computed with `action_create_account.exec_fee = 3_850_000_000_000`).

2. The network upgrades to protocol version 85 (`AccountCostIncrease`). `action_create_account.exec_fee` is now `7_200_000_000_000`. The true `congestion_gas` for this receipt under the new config would be `G_new = G_old + 3_350_000_000_000`.

3. When the receipt is popped from the buffer, `receipt_congestion_gas` returns `G_old` (the stale snapshot). `CongestionInfo.buffered_receipts_gas` is decremented by `G_old` instead of `G_new`.

4. The shard's congestion level is understated by `3_350_000_000_000` gas per affected receipt. The congestion control system forwards additional receipts to the shard that it would have blocked under correct accounting, potentially overloading it.

The exact divergent Borsh bytes are the `congestion_gas: u64` field of `StateStoredReceiptMetadata` stored in `DBCol::State` under the `TrieKey::BufferedReceipt` and `TrieKey::DelayedReceipt` trie keys — frozen at the old fee schedule and never updated on protocol upgrade. /// Get the receipt gas from the receipt that was retrieved from the state.
/// If it is a [Receipt], the gas will be computed.
/// If it s the [StateStoredReceipt], the size will be read from the metadata.
pub(crate) fn receipt_congestion_gas(
    receipt: &ReceiptOrStateStoredReceipt,
    config: &RuntimeConfig,
) -> Result<Gas, IntegerOverflowError> {
    match receipt {
        ReceiptOrStateStoredReceipt::Receipt(receipt) => {
            compute_receipt_congestion_gas(receipt, config)
        }
        ReceiptOrStateStoredReceipt::StateStoredReceipt(receipt) => {
            Ok(receipt.metadata().congestion_gas)
        }
    }
}

fn action_receipt_congestion_gas(
    receipt: &Receipt,
    config: &RuntimeConfig,
    action_receipt: VersionedActionReceipt,
) -> Result<Gas, IntegerOverflowError> {
    let prepaid_exec_gas =
        total_prepaid_exec_fees(config, &action_receipt.actions(), receipt.receiver_id())?
            .gas
            .checked_add(config.fees.fee(ActionCosts::new_action_receipt).exec_fee().gas)
            .ok_or(IntegerOverflowError)?;
    // account for gas guaranteed to be used for creating new receipts
    let prepaid_send_cost = total_prepaid_send_fees(config, &action_receipt.actions())?;
    let prepaid_gas = prepaid_exec_gas.checked_add_result(prepaid_send_cost.gas)?;

    // account for gas potentially used for dynamic execution
    let gas_attached_to_fns = total_prepaid_gas(&action_receipt.actions())?;
    let gas = gas_attached_to_fns.checked_add_result(prepaid_gas)?;

    Ok(gas)
}

pub fn bootstrap_congestion_info(
    trie: &dyn near_store::TrieAccess,
    config: &RuntimeConfig,
    shard_id: ShardId,
) -> Result<CongestionInfo, StorageError> {
    let mut receipt_bytes: u64 = 0;
    let mut delayed_receipts_gas: u128 = 0;
    let mut buffered_receipts_gas: u128 = 0;

    let delayed_receipt_queue = &DelayedReceiptQueue::load(trie)?;
    for receipt_result in delayed_receipt_queue.iter(trie, true) {
        let receipt = receipt_result?;
        let gas =
            receipt_congestion_gas(&receipt, config).map_err(int_overflow_to_storage_err)?.as_gas();
        delayed_receipts_gas = safe_add_gas_to_u128(delayed_receipts_gas, Gas::from_gas(gas))
            .map_err(int_overflow_to_storage_err)?;

        let memory = receipt_size(&receipt).map_err(int_overflow_to_storage_err)? as u64;
        receipt_bytes = receipt_bytes.checked_add(memory).ok_or_else(overflow_storage_err)?;
    }

    let mut outgoing_buffers = ShardsOutgoingReceiptBuffer::load(trie)?;
    for shard in outgoing_buffers.shards() {
        for receipt_result in outgoing_buffers.to_shard(shard).iter(trie, true) {
            let receipt = receipt_result?;
            let gas = receipt_congestion_gas(&receipt, config)
                .map_err(int_overflow_to_storage_err)?
                .as_gas();
            buffered_receipts_gas = safe_add_gas_to_u128(buffered_receipts_gas, Gas::from_gas(gas))
                .map_err(int_overflow_to_storage_err)?;
            let memory = receipt_size(&receipt).map_err(int_overflow_to_storage_err)? as u64;
            receipt_bytes = receipt_bytes.checked_add(memory).ok_or_else(overflow_storage_err)?;
        }
    }

    Ok(CongestionInfo::V1(CongestionInfoV1 {
        delayed_receipts_gas: delayed_receipts_gas as u128,
        buffered_receipts_gas: buffered_receipts_gas as u128,
        receipt_bytes,
        // For the first chunk, set this to the own id.
        // This allows bootstrapping without knowing all other shards.
        // It is also irrelevant, since the bootstrapped value is only used at
        // the start of applying a chunk on this shard. Other shards will only
        // see and act on the first congestion info after that.
        allowed_shard: shard_id.into(),
    }))
}
