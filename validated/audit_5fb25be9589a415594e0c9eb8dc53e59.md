Audit Report

## Title
SwapAllowlistExtension checks the router address instead of the end user, allowing any caller to bypass the per-pool swap allowlist via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender` — the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. Any pool admin who allowlists the router so that approved users can access it simultaneously opens the gate to every unpermissioned caller, fully defeating the allowlist.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `sender` against the per-pool allowlist: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the pool's `msg.sender`. The original end-user address is stored only in transient storage for the payment callback and is never forwarded to the pool or any extension: [4](#0-3) 

The pool therefore receives `sender = router`. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end-user]`. If the admin has allowlisted the router (the only way to let approved users access the router's UX), every unpermissioned caller who routes through the same public router contract passes the check.

## Impact Explanation
The swap allowlist is the primary access-control mechanism for pools restricting trading to approved counterparties (KYC/AML-gated pools, institutional venues). Bypassing it allows any address to execute swaps against the pool's LP reserves. Every unauthorized swap extracts value from LPs at the oracle-anchored price; a non-allowlisted actor can drain LP principal by repeatedly swapping in the direction that empties the pool's bins. This constitutes a direct loss of LP assets and breaks the core pool functionality the allowlist was deployed to protect — matching the "broken core pool functionality causing loss of funds" and "direct loss of user principal or owed LP assets" impact categories.

## Likelihood Explanation
The trigger is a routine, well-motivated admin action: allowlisting the public router so that approved users can access the router's UX. This is the **only** mechanism available to let allowlisted users use the router, so any admin who deploys a `SwapAllowlistExtension` pool and wants router access for their users will inevitably create the bypass condition. The router is a public, permissionless contract requiring no special capability from the attacker. The bypass is reachable on every pool that has both `SwapAllowlistExtension` configured and the router allowlisted.

## Recommendation
The extension must check the economically relevant actor, not the intermediary:

1. **Router-side fix**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into a standardised field of `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap` can then decode and check that address when the direct `sender` is a known router.

2. **Documentation/operational fix (short-term)**: Clearly document that the allowlist gates the direct caller of `pool.swap()` and that the router must **not** be allowlisted; approved users must call the pool directly. This is a breaking UX constraint and not a long-term solution.

The cleanest fix is option 1: the router forwards the original caller's address in a standardised `extensionData` field so allowlist extensions can always check the true end user.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice]  = true   // approved user
  allowedSwapper[pool][router] = true   // admin adds router so alice can use it

Attack (Carol, not allowlisted):
  carol → router.exactInputSingle({pool: pool, ...})
       → _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, carol, tokenIn)  // carol stored in transient storage only
       → pool.swap(msg.sender=router, ...)
       → _beforeSwap(sender=router, ...)
       → SwapAllowlistExtension.beforeSwap(sender=router)
            allowedSwapper[pool][router] == true  ✓  (no revert)
       → swap executes, carol extracts tokens from LP bins

Result:
  Carol bypasses the allowlist and drains LP funds.
  Alice's individual allowlist entry is irrelevant; the router entry is the effective gate.
```

A Foundry integration test can confirm this by: (1) deploying a pool with `SwapAllowlistExtension`, (2) allowlisting only Alice and the router, (3) calling `router.exactInputSingle` from an address that is not Alice, and (4) asserting the swap succeeds rather than reverting with `NotAllowedToSwap`.
