### Title
`RuntimeDialect` Diverges from `ChiaDialect` Interface: Broken Softfork Extension Dispatch and Missing Operators — (File: `src/runtime_dialect.rs`, `src/f_table.rs`)

---

### Summary

`RuntimeDialect` is the configurable Rust-level alternative to `ChiaDialect` for running CLVM programs. It diverges from `ChiaDialect` in two concrete, attacker-reachable ways: (1) `softfork_extension()` unconditionally returns `OperatorSet::Default`, silently suppressing all softfork guard execution in consensus mode and hard-failing in mempool mode; and (2) the operator name-to-function table in `f_table.rs` omits `op_coinid`, `op_keccak256`, and `op_sha256_tree`, which `ChiaDialect` dispatches correctly. Any downstream Rust consumer that substitutes `RuntimeDialect` for `ChiaDialect` in a consensus-critical path will produce divergent results for the same attacker-supplied CLVM bytes.

---

### Finding Description

**Root cause 1 — `softfork_extension()` always returns `OperatorSet::Default`**

`ChiaDialect::softfork_extension()` maps extension `0` → `OperatorSet::Bls` and extension `1` → `OperatorSet::Keccak`: [1](#0-0) 

`RuntimeDialect::softfork_extension()` ignores the argument entirely and always returns `OperatorSet::Default`: [2](#0-1) 

In `run_program.rs`, `parse_softfork_arguments` calls `self.dialect.softfork_extension()` and immediately errors if the result is `OperatorSet::Default`: [3](#0-2) 

The error is then handled in `apply_op`: [4](#0-3) 

- **Consensus mode** (`allow_unknown_ops() == true`): the error is swallowed, nil is pushed, and the softfork guard body is never executed. A program that should run a keccak256 verification inside the guard silently returns nil instead.
- **Mempool mode** (`NO_UNKNOWN_OPS` set): the error propagates as `UnknownSoftforkExtension`, rejecting a transaction that `ChiaDialect` would accept.

**Root cause 2 — Missing operators in `f_table.rs`**

`opcode_by_name` in `f_table.rs` is the sole source of operator-name-to-function mappings used by `RuntimeDialect` via `f_lookup_for_hashmap`. It contains 44 entries but omits three operators that `ChiaDialect` dispatches at opcodes 48, 62, and 63: [5](#0-4) 

`ChiaDialect` dispatches all three: [6](#0-5) 

If a caller attempts to register any of these operators by name in the `op_map` passed to `RuntimeDialect::new()`, `f_lookup_for_hashmap` panics unconditionally: [7](#0-6) 

If the caller simply omits them, programs using opcodes 48, 62, or 63 are treated as unknown operators — silently no-oped in consensus mode or hard-rejected in mempool mode — diverging from `ChiaDialect` behavior.

---

### Impact Explanation

A downstream Rust consumer that uses `RuntimeDialect` as a drop-in replacement for `ChiaDialect` (the documented configurable alternative) will:

1. **Silently bypass softfork guard execution** in consensus mode for any CLVM program that uses `(softfork cost 1 keccak-program env)`. The keccak verification body is never run; the guard returns nil unconditionally. An attacker can submit a transaction whose puzzle solution is only valid under keccak verification — it will be accepted by a `RuntimeDialect` node and rejected by a `ChiaDialect` node, producing a **consensus split**.

2. **Incorrectly reject valid transactions** in mempool mode for any program using softfork extension 0 or 1, causing a **denial of service** for legitimate users.

3. **Panic or silently misroute** programs using `op_coinid` (48), `op_keccak256` (62), or `op_sha256_tree` (63), producing wrong results or crashes for attacker-supplied CLVM bytes that exercise these opcodes.

---

### Likelihood Explanation

`RuntimeDialect` is a public Rust API in the `clvmr` crate, exported alongside `ChiaDialect`. Its constructor signature (`op_map`, `quote_kw`, `apply_kw`, `flags`) makes it appear as a fully equivalent, configurable replacement. The divergence in `softfork_extension()` is not documented and is not caught by the type system. Any Chia tooling author or downstream node implementation that reaches for `RuntimeDialect` for its configurability will silently inherit the broken softfork dispatch. The missing operators in `f_table.rs` compound the risk for any caller that attempts to build a complete operator set. [8](#0-7) 

---

### Recommendation

1. **Fix `softfork_extension()` in `RuntimeDialect`**: Either delegate to a stored `ChiaDialect` instance, or accept a `softfork_extension` closure/map in `RuntimeDialect::new()` so callers can replicate `ChiaDialect` behavior. At minimum, document that `RuntimeDialect` does not support softfork extensions and must not be used in consensus-critical paths.

2. **Add missing operators to `opcode_by_name` in `f_table.rs`**: Add `op_coinid`, `op_keccak256`, and `op_sha256_tree` to the lookup table so `RuntimeDialect` can dispatch them correctly when callers include them in their `op_map`.

3. **Add a compile-time or runtime assertion** that `RuntimeDialect` and `ChiaDialect` agree on softfork extension dispatch for all known extensions, to prevent future divergence.

---

### Proof of Concept

Craft a CLVM program that uses softfork extension 1 (Keccak):

```
(softfork (q . <cost>) (q . 1) <keccak-verification-program> <env>)
```

Run it under `ChiaDialect` (consensus mode, no `NO_UNKNOWN_OPS`): the keccak program executes inside the guard; cost is validated; the guard returns nil on success or raises on failure.

Run the identical bytes under `RuntimeDialect` (same flags): `softfork_extension(1)` returns `OperatorSet::Default`; `parse_softfork_arguments` returns `Err(UnknownSoftforkExtension)`; `allow_unknown_ops()` is true; nil is pushed without executing the keccak body; the program succeeds unconditionally regardless of whether the keccak verification would have passed.

The same attacker-supplied bytes produce `Ok(nil)` under `RuntimeDialect` and either `Ok(nil)` (valid) or `Err(...)` (invalid) under `ChiaDialect` — a concrete consensus divergence triggered entirely by the choice of dialect on the receiving node. [2](#0-1) [4](#0-3)

    fn softfork_extension(&self, ext: u32) -> OperatorSet {
        match ext {
            // Extension 0 is for the BLS operators, and is still valid.
            // However, the extension doesn't add any addition opcodes,
            // because the BLS operators were hardforked into the main set.
            0 => OperatorSet::Bls,

            // Extension 1 is for the keccak256 operator.
            1 => OperatorSet::Keccak,

            // Extensions 2 and beyond are considered invalid by the mempool.
            // However, all future extensions are valid in consensus mode and reserved for future softforks.
            _ => OperatorSet::Default,
        }
    }

pub fn f_lookup_for_hashmap(opcode_lookup_by_name: HashMap<String, Vec<u8>>) -> FLookup {
    let mut f_lookup = [None; 256];
    for (name, idx) in opcode_lookup_by_name.iter() {
        if idx.len() == 1 {
            let index = idx[0];
            let op = opcode_by_name(name);
            assert!(op.is_some(), "can't find native operator {name}");
            f_lookup[index as usize] = op;
        }
    }
    f_lookup
}
