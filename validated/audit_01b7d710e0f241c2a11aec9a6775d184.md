### Title
`SwapAllowlistExtension` gates the router address instead of the actual end-user, allowing any unprivileged caller to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates a swap, `sender` is the router address, not the actual end-user. A pool admin who allowlists the router (necessary for any allowlisted user to trade through the router) inadvertently opens the pool to every caller of the router. The guard check passes for the router, but the intended per-user restriction is silently unenforced — a structural analog to the Optimism H-2 pattern where a check passes but the actual protection fails.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every before-swap extension hook: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that `sender` is allowlisted for the calling pool (`msg.sender` inside the extension = the pool): [2](#0-1) 

When `MetricOmmSimpleRouter.exactInputSingle` (or any multi-hop variant) calls `pool.swap`, the pool receives `msg.sender = router`: [3](#0-2) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. The router never forwards the real caller's identity to the pool.

This creates an inescapable dilemma for any pool admin who configures a swap allowlist:

| Admin choice | Effect on allowlisted users | Effect on non-allowlisted users |
|---|---|---|
| Do **not** allowlist the router | Cannot use the router at all (broken flow) | Correctly blocked |
| **Allowlist the router** | Can use the router | **Also pass the check — full bypass** |

The allowlist mapping and admin setter are correct in isolation: [4](#0-3) 

The defect is that the hook's identity check is structurally bound to the wrong actor the moment any supported periphery contract intermediates the call.

---

### Impact Explanation

A curated pool whose admin has deployed `SwapAllowlistExtension` to restrict trading to a known set of addresses can be fully bypassed by any unprivileged user calling `MetricOmmSimpleRouter.exactInputSingle` (or the multi-hop variants). The bypass allows:

- Unauthorized swaps against LP liquidity at oracle-derived prices, draining token balances from bins the admin intended to protect.
- Complete nullification of the curation policy without any admin action or privileged access.

This satisfies the allowed-impact gate: **admin-boundary break** (an admin-configured allowlist guard is bypassed by an unprivileged path) and **broken core pool functionality** (allowlisted users cannot use the router without opening the pool to everyone).

---

### Likelihood Explanation

- The trigger requires only a standard call to the public `MetricOmmSimpleRouter` — no special role, no flash loan, no multi-block setup.
- The bypass is deterministic: every router-mediated swap on an allowlisted pool passes the guard with the router's address.
- The only precondition is that the pool admin has allowlisted the router (a natural operational step once any allowlisted user needs router access).

---

### Recommendation

The `beforeSwap` hook should check the **actual end-user** rather than the immediate caller. Two complementary fixes:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData` and the extension decodes and checks it. This requires a coordinated change to the router and the extension.

2. **Check `recipient` instead of `sender` when `sender` is a known router**: The pool already passes `recipient` as a separate argument to `beforeSwap`. For swap-allowlist purposes, gating on `recipient` is closer to the economic actor, though it still does not perfectly identify the payer.

3. **Preferred — gate on the callback payer**: Restructure the allowlist to check the address that will settle the swap callback (the actual token payer). The router stores this as `msg.sender` in its callback context (`_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)`), but this is not surfaced to the pool or extension today.

The minimal safe fix is to reject `sender` values that are known periphery contracts unless the extension has an explicit "trusted forwarder" registry that maps router → real caller.

### Proof of Concept

```solidity
// Pool is deployed with SwapAllowlistExtension.
// Admin allowlists only `trustedUser` for direct swaps,
// and also allowlists the router so trustedUser can use it.

swapAllowlist.setAllowedToSwap(pool, trustedUser, true);
swapAllowlist.setAllowedToSwap(pool, address(router), true); // required for trustedUser to use router

// Attacker (not in allowlist) calls the router directly.
// The extension sees sender = router, which IS allowlisted → check passes.
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token1,
    recipient: attacker,
    amountIn: largeAmount,
    amountOutMinimum: 0,
    zeroForOne: false,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// Swap executes. Allowlist guard was bypassed. LP funds drained at oracle price.
