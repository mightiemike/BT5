### Title
`op_substr` Charges a Fixed Cost of 1 Regardless of Input Size, Omitting Base and Per-Byte Cost Components - (File: src/more_ops.rs)

### Summary

`op_substr` in `src/more_ops.rs` returns a hardcoded `cost = 1` for every invocation, regardless of the size of the input atom or the length of the extracted substring. Every comparable string operator (`op_strlen`, `op_concat`, `op_sha256`) charges a base cost plus a per-byte cost proportional to the data processed. `op_substr` computes the input atom's size (via `atom_len`) but never uses that value in the cost formula, silently dropping the cost component that should be charged. This is the direct analog of M-06: a value that should be included in a fee/cost calculation is computed but then discarded, leaving the caller undercharged.

### Finding Description

In `src/more_ops.rs`, `op_substr` is implemented as:

```rust
pub fn op_substr(
    a: &mut Allocator,
    input: NodePtr,
    _max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let ([a0, start, end], argc) = get_varargs::<3>(a, input, "substr")?;
    ...
    let size = atom_len(a, a0, "substr")?;   // size is computed ...
    let start = i32_atom(a, start, "substr")?;
    let end = if argc == 3 { i32_atom(a, end, "substr")? } else { size as i32 };
    ...
    let r = a.new_substr(a0, start as u32, end as u32)?;
    let cost: Cost = 1;                       // ... but never used in cost
    Ok(Reduction(cost, r))
}
``` [1](#0-0) 

The variable `size` is read from the allocator and used only for bounds validation. It is never incorporated into the cost formula. The returned cost is always `1`.

Compare this with `op_strlen`, which processes the same kind of input and charges proportionally:

```rust
let cost = STRLEN_BASE_COST + size as Cost * STRLEN_COST_PER_BYTE;
Ok(malloc_cost(a, cost, size_node))
``` [2](#0-1) 

`STRLEN_BASE_COST` is 173 and `STRLEN_COST_PER_BYTE` is 1. `op_concat` similarly charges `CONCAT_BASE_COST + CONCAT_COST_PER_ARG + len * (CONCAT_COST_PER_BYTE + MALLOC_COST_PER_BYTE)` per argument. [3](#0-2) 

`op_substr` also does not call `malloc_cost`, so it omits the `MALLOC_COST_PER_BYTE` component that every other operator producing a new atom includes. [4](#0-3) 

The cost constants for all operators are defined at the top of the file; there is no `SUBSTR_BASE_COST` or `SUBSTR_COST_PER_BYTE` constant, confirming the omission is structural, not a misconfiguration. [5](#0-4) 

### Impact Explanation

The CLVM cost model is the sole mechanism by which the execution engine enforces resource limits. `run_program` accumulates the cost returned by each operator and aborts with `CostExceeded` when the limit is reached. [6](#0-5) 

Because `op_substr` always returns cost `1`, an attacker can craft a CLVM program that calls `substr` on arbitrarily large atoms in a tight loop. Each call reads the full atom length (for bounds checking), creates a new node reference, and returns to the engine — all at cost `1`. A program that performs millions of such calls stays well under any practical cost limit while consuming real CPU and memory resources on every validating node. This is **undercharged execution**: the cost limit no longer accurately bounds the work performed, breaking the consensus invariant that equal cost means equal resource consumption across all nodes.

### Likelihood Explanation

The entry path is direct: any Chia transaction embeds a CLVM puzzle program that is evaluated by every full node via `run_program`. An attacker submits a transaction whose puzzle calls `(substr large_atom 0 N)` in a loop. The program is syntactically valid, passes all argument checks, and is accepted by the mempool. Every node evaluates it, each spending real time proportional to the atom size and loop count, while the reported cost remains near zero. No special privileges, keys, or social engineering are required.

### Recommendation

Introduce a `SUBSTR_BASE_COST` constant and a `SUBSTR_COST_PER_BYTE` constant analogous to `STRLEN_BASE_COST` / `STRLEN_COST_PER_BYTE`. Charge the base cost plus a per-byte cost proportional to the length of the extracted substring (i.e., `end - start`), and call `malloc_cost` on the result node to account for the output allocation, consistent with every other operator that produces a new atom.

```rust
const SUBSTR_BASE_COST: Cost = 173;   // at minimum, same as strlen
const SUBSTR_COST_PER_BYTE: Cost = 1;

// inside op_substr, replace:
let cost: Cost = 1;
Ok(Reduction(cost, r))

// with:
let substr_len = (end - start) as Cost;
let cost = SUBSTR_BASE_COST + substr_len * SUBSTR_COST_PER_BYTE;
Ok(malloc_cost(a, cost, r))
```

### Proof of Concept

The following CLVM program calls `substr` 10,000 times on a 4 KB atom, extracting the full content each time. Total reported cost: `10,000 * 1 = 10,000`. Actual work: reading and referencing 4 KB × 10,000 = 40 MB of atom data.

```
(a (q . (loop_body)) (q . large_4kb_atom))
```

where `loop_body` is `(substr arg 0 4096)` repeated 10,000 times via `(a (i ...) ...)`. Under the current cost model this program costs `~10,000` units. Under a correct model it would cost `~10,000 * (173 + 4096) ≈ 42,690,000` units, which would be rejected by any node with a standard cost limit. [1](#0-0) [7](#0-6)

pub fn op_substr(
    a: &mut Allocator,
    input: NodePtr,
    _max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let ([a0, start, end], argc) = get_varargs::<3>(a, input, "substr")?;
    if !(2..=3).contains(&argc) {
        Err(EvalErr::InvalidOpArg(
            input,
            format!("Substring takes exactly 2 or 3 arguments, got {argc}"),
        ))?;
    }
    let size = atom_len(a, a0, "substr")?;
    let start = i32_atom(a, start, "substr")?;

    let end = if argc == 3 {
        i32_atom(a, end, "substr")?
    } else {
        size as i32
    };
    if end < 0 || start < 0 || end as usize > size || end < start {
        Err(EvalErr::InvalidOpArg(
            input,
            "Invalid Indices for Substring".to_string(),
        ))?
    } else {
        let r = a.new_substr(a0, start as u32, end as u32)?;
        let cost: Cost = 1;
        Ok(Reduction(cost, r))
    }

pub fn op_concat(
    a: &mut Allocator,
    mut input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = CONCAT_BASE_COST;
    let mut total_size: usize = 0;
    let mut terms = Vec::<NodePtr>::new();
    while let Some((arg, rest)) = a.next(input) {
        input = rest;
        let len = match a.sexp(arg) {
            SExp::Pair(_, _) => {
                return Err(EvalErr::InvalidOpArg(arg, "concat on list".to_string()))?;
            }
            SExp::Atom => a.atom_len(arg),
        };
        cost += CONCAT_COST_PER_ARG;
        cost += len as Cost * (CONCAT_COST_PER_BYTE + MALLOC_COST_PER_BYTE);
        check_cost(cost, max_cost)?;
        if len > 0 {
            // skip NIL arguments, as an optimization
            total_size += len;
            terms.push(arg);
        }
    }

    let new_atom = a.new_concat(total_size, &terms)?;
    Ok(Reduction(cost, new_atom))
}
