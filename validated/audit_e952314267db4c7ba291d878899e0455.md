Looking at the external bug's invariant class ŌĆö **a shared intermediary causes state from one actor to bleed into another actor's position** ŌĆö and mapping it to the Metric OMM swap allowlist path:

### Title
`SwapAllowlistExtension` gates the router address instead of the actual swapper, allowing any user to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` ŌĆö (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to its own `msg.sender` (i.e., whoever called `pool.swap()`). When users route through `MetricOmmSimpleRouter`, that caller is the router contract, not the actual user. A pool admin who allowlists the router ŌĆö the natural step to let their allowlisted users use the router ŌĆö inadvertently opens the pool to every user, because the extension cannot distinguish allowlisted from non-allowlisted callers behind the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs its identity check as follows: [1](#0-0) 

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` is the pool; `sender` is the first argument forwarded by the pool from its own `msg.sender`: [2](#0-1) 

```solidity
_beforeSwap(
    msg.sender,   // ŌåÉ whoever called pool.swap()
    recipient,
    ...
);
```

When `MetricOmmSimpleRouter.exactInputSingle` (or any router entry point) calls `pool.swap(...)`, the pool's `msg.sender` is the router. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. [3](#0-2) 

The pool admin faces an impossible choice:

- **Option A** ŌĆö Do not allowlist the router ŌåÆ allowlisted users cannot use the router at all (broken core swap flow for every allowlisted user).
- **Option B** ŌĆö Allowlist the router ŌåÆ every user, including non-allowlisted ones, can bypass the guard by routing through `MetricOmmSimpleRouter`.

Under Option B the bypass is fully unprivileged from the attacker's perspective: any address calls `router.exactInputSingle(restrictedPool, ...)` and the extension passes because `allowedSwapper[pool][router] == true`.

The `SwapAllowlistExtension` provides no mechanism to gate by actual user identity behind the router. The `setAllowedToSwap` setter only records `(pool, address) ŌåÆ bool`: [4](#0-3) 

There is no way to express "allow Alice through the router but block Bob through the router" ŌĆö the only granularity is the direct caller of `pool.swap()`.

---

### Impact Explanation

A pool protected by `SwapAllowlistExtension` (e.g., a KYC-gated pool, a private pool with favorable oracle pricing, or a pool with subsidized spreads) can be freely accessed by any user once the pool admin allowlists the router. Unauthorized swappers can drain favorable pricing, extract subsidized output tokens, or interact with a pool that was contractually restricted to specific counterparties, causing direct loss of value to the pool's LPs and the pool admin.

---

### Likelihood Explanation

The pool admin's natural configuration step to let their allowlisted users use the router is to add the router to the allowlist. The extension provides no correct alternative ŌĆö there is no way to gate by actual user identity behind the router. Any pool that uses `SwapAllowlistExtension` and also wants router support is vulnerable. The trigger is a semi-trusted pool admin action taken in good faith.

---

### Recommendation

Gate by the actual user's identity, not the intermediary's address. Options:

1. Have the router forward the actual user's address in `extensionData`; have the extension decode and verify it (requires coordination between router and extension).
2. Add a trusted-forwarder pattern: if `sender` is a known router, decode the actual user from `extensionData` and check that address against the allowlist.
3. Document and enforce at the factory level that `SwapAllowlistExtension` is incompatible with router-mediated swaps, and revert pool creation if both are configured together.
