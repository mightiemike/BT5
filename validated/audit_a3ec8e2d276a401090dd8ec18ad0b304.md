### Title
`verify_transaction_inclusion` Incorrectly Panics for Single-Transaction Blocks Due to Unconditional Empty-Proof Guard — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` contains an unconditional guard at line 315 that panics whenever the caller supplies an empty Merkle proof. For a block that contains exactly one transaction, the Merkle proof is legitimately empty — the transaction hash *is* the Merkle root — so the guard incorrectly rejects a valid, verifiable inclusion claim. The underlying `compute_root_from_merkle_proof` already handles the empty-proof case correctly by returning the input hash unchanged, but the guard fires before that computation is ever reached.

---

### Finding Description

In `verify_transaction_inclusion` (`contract/src/lib.rs`, line 315):

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

This guard is unconditional. It fires regardless of whether the block in question has one transaction or many. For a single-transaction block the Merkle tree has exactly one leaf; the Merkle root equals the transaction hash; and the correct proof is the empty vector `[]`.

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` (lines 34–52) already handles this correctly:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    // loop body never executes when merkle_proof is empty
    for proof_hash in merkle_proof { … }
    current_hash          // returns transaction_hash unchanged
}
```

So `compute_root_from_merkle_proof(tx_id, 0, &[]) == merkle_root` evaluates to `true` for a single-transaction block, and the function *should* return `true`. Instead, the guard at line 315 panics before the computation is reached.

The same defect propagates into `verify_transaction_inclusion_v2` (line 347–368), which calls `verify_transaction_inclusion` internally after its own coinbase-proof check passes. For a single-transaction block both `merkle_proof` and `coinbase_merkle_proof` are empty; the length-equality check at line 348–351 passes (`0 == 0`); the coinbase-proof check at lines 358–365 passes (`coinbase_tx_id == merkle_root`); and then the call to `verify_transaction_inclusion` panics at line 315.

This is the direct analog of the `l1mntSuccess` bug: a guard/flag that defaults to a restrictive state causes the function to abort even when the underlying operation would have succeeded.

---

### Impact Explanation

Any downstream NEAR contract or user that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with a legitimately empty Merkle proof receives a contract panic instead of `true`. The proof result is corrupted: a valid inclusion claim is treated as an error. If a bridge or escrow contract gates fund release on a `true` return value from these functions, funds targeting a single-transaction block become permanently unreleasable through the normal verification path.

---

### Likelihood Explanation

Low but non-zero. Single-transaction blocks (containing only the coinbase) exist in Bitcoin (the genesis block and several early mainnet blocks), Litecoin, and Dogecoin. The contract is initialized with a genesis block that is stored in `mainchain_header_to_height` and `headers_pool`; that block is reachable by any caller until it is garbage-collected. Any unprivileged NEAR account can trigger the panic immediately after contract initialization without any special role or key.

---

### Recommendation

Remove the unconditional guard at line 315. `compute_root_from_merkle_proof` already handles the empty-proof case correctly and will return the transaction hash unchanged, which is then compared against the stored Merkle root. No additional guard is needed.

```rust
// Remove this line:
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

If a non-empty proof is desired as a policy constraint (e.g., to avoid the 64-byte forgery surface), the check should be applied only when the block is known to contain more than one transaction — but that information is not stored in the light client, so the guard cannot be made conditional in its current form and should simply be removed.

---

### Proof of Concept

1. The contract is initialized with the Bitcoin genesis block at height 0. The genesis block contains exactly one transaction (the coinbase). Its Merkle root equals the coinbase transaction hash: `4a5e1e4baab89f3a32518a88c31bc87f618f76673e2cc77ab2127b7afdeda33b` (visible in the test fixtures at `contract/src/lib.rs` lines 773–778).

2. An unprivileged NEAR caller invokes `verify_transaction_inclusion` with:
   - `tx_id = 4a5e1e4baab89f3a32518a88c31bc87f618f76673e2cc77ab2127b7afdeda33b`
   - `tx_block_blockhash = <genesis block hash>`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `confirmations = 1`

3. All checks before line 315 pass: the block is in the main chain, confirmations are satisfied, and the header is found in the pool.

4. Execution reaches line 315: `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` — the contract panics.

5. Had the guard been absent, `compute_root_from_merkle_proof(tx_id, 0, &[])` would return `tx_id`, which equals `header.block_header.merkle_root`, and the function would correctly return `true`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
