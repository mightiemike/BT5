### Title
Gateway Converter Unconditionally Enforces V2 Compiled Class Hash While Blockifier Gates Enforcement on `block_casm_hash_v1_declares` Versioned Constant — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

The `TransactionConverter::convert_rpc_tx_to_internal` always validates a DeclareV3 transaction's `compiled_class_hash` against the V2 (Blake2) hash returned by the class manager, with no version gate. The blockifier's `DeclareTransaction::run_execute` enforces the same V2 constraint only when the `block_casm_hash_v1_declares` versioned constant is `true` **and** `version >= THREE`. On Starknet 0.14.0 — where `block_casm_hash_v1_declares = false` — the two components disagree on what constitutes a valid compiled class hash: the gateway rejects V1 (Poseidon)-hashed DeclareV3 transactions that the blockifier would accept, causing valid transactions to be dropped before sequencing.

---

### Finding Description

**Gateway converter side** — `crates/apollo_transaction_converter/src/transaction_converter.rs`, lines 347–376:

```rust
RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => {
    let ClassHashes { class_hash, executable_class_hash_v2 } =
        self.class_manager_client.add_class(tx.contract_class).await?;
    // TODO(Aviv): Ensure that we do not want to
    // allow declare with compiled class hash v1.
    if tx.compiled_class_hash != executable_class_hash_v2 {
        return Err(TransactionConverterError::ValidateCompiledClassHashError(
            ValidateCompiledClassHashError::CompiledClassHashMismatch { ... }
        ));
    }
    ...
}
```

`executable_class_hash_v2` is always the Blake2 (V2) hash. There is no check against `block_casm_hash_v1_declares`. The developer TODO comment itself acknowledges the uncertainty.

**Blockifier side** — `crates/blockifier/src/transaction/transactions.rs`, lines 184–188:

```rust
if context.tx_context.block_context.versioned_constants.block_casm_hash_v1_declares
    && self.version() >= TransactionVersion::THREE
{
    self.check_compile_class_hash_v2_declaration()?
}
try_declare(self, state, class_hash, Some(*compiled_class_hash))?
```

The blockifier only enforces V2 hash when `block_casm_hash_v1_declares` is `true`. On 0.14.0 this flag is `false`:

```
crates/blockifier/resources/versioned_constants_diff_regression/0.14.0_0.14.1.txt:
~ /block_casm_hash_v1_declares: true
~ /enable_casm_hash_migration: true
```

So on 0.14.0 the blockifier accepts a DeclareV3 carrying a V1 (Poseidon) compiled class hash, while the gateway converter rejects it unconditionally.

The hash algorithm divergence is defined in `crates/starknet_api/src/contract_class/compiled_class_hash.rs`:

```rust
pub enum HashVersion {
    V1,  // Poseidon
    V2,  // Blake2Felt252
}
fn hash(&self, hash_version: &HashVersion) -> CompiledClassHash {
    match hash_version {
        HashVersion::V1 => hash_inner::<Poseidon, EH, NL>(self),
        HashVersion::V2 => hash_inner::<Blake2Felt252, EH, NL>(self),
    }
}
```

The two hash values are numerically distinct for every real contract class (confirmed by the test constants: V1 = `1086536622945160114536053561878005579687531094896980931390443771221568164185`, V2 = `3325824766528769720608805422968547980476991657768956291033663350476760978279`).

---

### Impact Explanation

On a sequencer node configured for Starknet 0.14.0 (`block_casm_hash_v1_declares = false`), any DeclareV3 transaction whose `compiled_class_hash` field carries the V1 (Poseidon) value is rejected at the gateway with `COMPILED_CLASS_HASH_MISMATCH` (RPC error code 60). The blockifier would have accepted and committed the same transaction. This is a **valid transaction rejected before sequencing**, matching the High impact: *"Mempool/gateway/RPC admission rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The incompatibility is structural and version-dependent. Any operator running the Apollo sequencer against a Starknet 0.14.0 state (or any pre-0.14.1 state where `block_casm_hash_v1_declares = false`) will silently drop all DeclareV3 transactions whose signers computed the compiled class hash using the Poseidon algorithm. No special attacker capability is required; any ordinary user submitting such a transaction triggers the rejection.

---

### Recommendation

Gate the compiled class hash validation inside `TransactionConverter::convert_rpc_tx_to_internal` on the same `block_casm_hash_v1_declares` versioned constant that the blockifier uses. Specifically, only reject a V1-hashed `compiled_class_hash` when the active `VersionedConstants` for the current block has `block_casm_hash_v1_declares = true`, mirroring the blockifier's guard in `DeclareTransaction::run_execute`. This aligns the admission boundary with the execution boundary and eliminates the version-dependent split.

---

### Proof of Concept

1. Start the Apollo sequencer with a block context whose `VersionedConstants` has `block_casm_hash_v1_declares = false` (Starknet 0.14.0 JSON).
2. Compile a Cairo 1 contract and compute its compiled class hash using `HashVersion::V1` (Poseidon).
3. Submit a DeclareV3 RPC transaction with `compiled_class_hash` = V1 hash.
4. **Observed**: `TransactionConverter::convert_rpc_tx_to_internal` calls `class_manager_client.add_class`, receives `executable_class_hash_v2` (Blake2 value), finds `tx.compiled_class_hash != executable_class_hash_v2`, and returns `ValidateCompiledClassHashError::CompiledClassHashMismatch`. The gateway returns RPC error 60.
5. **Expected**: The blockifier's `run_execute` path skips `check_compile_class_hash_v2_declaration` (because `block_casm_hash_v1_declares = false`) and calls `try_declare` with the V1 hash, successfully declaring the class.
6. The divergence is the direct analog of the external report's tree-depth mismatch: one component (gateway converter) hard-codes the "new" constant (V2 hash), while the other (blockifier) reads it from a version-gated parameter.
