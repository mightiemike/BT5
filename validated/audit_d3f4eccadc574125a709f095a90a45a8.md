### Title
Missing Input Validation in `op_bls_pairing_identity` Allows Trivial Bypass of Pairing Check — (File: `src/bls_ops.rs`)

---

### Summary

`op_bls_pairing_identity` performs no validation that its argument list is non-empty. When called with zero (G1, G2) pairs — i.e., a nil argument list — it invokes `aggregate_pairing([])` on an empty vector. The BLS12-381 pairing product of zero pairs is the GT identity element, so the identity check trivially passes and the operator returns success (`nil`) without performing any cryptographic work. Any CLVM smart coin that relies on `bls_pairing_identity` to enforce a cryptographic constraint can be bypassed by an attacker who supplies zero pairs.

---

### Finding Description

`op_bls_pairing_identity` is documented as performing a low-level pairing operation on a flat list of (G1, G2) pairs and asserting that the resulting GT point is the identity. The implementation iterates over the argument list and collects pairs into a `Vec`:

```rust
// src/bls_ops.rs lines 332–358
pub fn op_bls_pairing_identity(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = BLS_PAIRING_BASE_COST;
    check_cost(cost, max_cost)?;
    let mut items = Vec::<(G1Element, G2Element)>::new();

    let mut args = input;
    while !nilp(a, args) {          // ← loop body never executes when input is nil
        cost += BLS_PAIRING_COST_PER_ARG;
        check_cost(cost, max_cost)?;
        let g1 = a.g1(first(a, args)?)?;
        args = rest(a, args)?;
        let g2 = a.g2(first(a, args)?)?;
        args = rest(a, args)?;
        items.push((g1, g2));
    }

    if !aggregate_pairing(items) {  // ← called with empty Vec when input is nil
        Err(EvalErr::BLSPairingIdentityFailed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))  // ← always reached for empty input
    }
}
```

When `input` is nil (the CLVM empty list), `nilp(a, args)` is immediately `true`, the loop body never executes, `items` remains an empty `Vec`, and `aggregate_pairing(items)` is called with zero pairs. In BLS12-381 the multi-pairing product of zero pairs is defined as the GT identity element, so `aggregate_pairing([])` returns `true`. The operator returns `Ok(Reduction(BLS_PAIRING_BASE_COST, nil))` — success — without performing any actual pairing computation.

The same structural issue exists in `op_bls_verify` (lines 364–401): after consuming the mandatory G2 signature argument, it accepts zero (pubkey, msg) pairs and calls `aggregate_verify(&signature, [])`. For an empty item list, aggregate verification succeeds when the signature is the G2 identity point (compressed infinity: first byte `0xc0`), which is a valid, publicly known 96-byte atom. An attacker can therefore pass the G2 identity point as the signature with zero pairs and `bls_verify` returns success.

---

### Impact Explanation

A CLVM smart coin that uses `bls_pairing_identity` to enforce a cryptographic invariant — for example, verifying that a set of BLS signatures satisfies a pairing equation — can be completely bypassed. An attacker submits a spend bundle whose puzzle solution causes the operator to be called with a nil argument list. The operator succeeds, the spend is accepted by consensus, and the intended cryptographic gate is never evaluated.

The corrupted result is concrete: the operator returns `Reduction(BLS_PAIRING_BASE_COST, nil)` (success / nil) instead of `Err(EvalErr::BLSPairingIdentityFailed(...))`. The broken invariant is that the operator is supposed to assert a non-trivial pairing product equals the identity, but with zero pairs the assertion is vacuously true.

For `op_bls_verify`, the corrupted result is `Reduction(BLS_PAIRING_BASE_COST, nil)` (verification success) for any spend that supplies the G2 identity point as the signature and zero (pubkey, msg) pairs, bypassing signature verification entirely.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: the attacker writes a CLVM program (arbitrary bytes) that calls opcode `58` (`bls_pairing_identity`) with a nil argument list, or opcode `59` (`bls_verify`) with the G2 identity point and no pairs. No special privileges, compromised nodes, or social engineering are required. The attack is deterministic and reproducible. Likelihood is **medium** — it requires a smart coin author to have used `bls_pairing_identity` or `bls_verify` as a security gate, which is the intended use case for these operators.

---

### Recommendation

Add a non-empty check at the start of `op_bls_pairing_identity` before entering the loop:

```rust
if nilp(a, input) {
    return Err(EvalErr::InvalidOpArg(
        input,
        "bls_pairing_identity requires at least one (G1, G2) pair".to_string(),
    ));
}
```

Similarly, in `op_bls_verify`, after consuming the signature, verify that at least one (pubkey, msg) pair follows:

```rust
if nilp(a, args) {
    return Err(EvalErr::InvalidOpArg(
        input,
        "bls_verify requires at least one (pubkey, message) pair".to_string(),
    ));
}
```

pub fn op_bls_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = BLS_PAIRING_BASE_COST;
    check_cost(cost, max_cost)?;

    let mut args = input;

    // the first argument is the signature
    let signature = a.g2(first(a, args)?)?;

    // followed by a variable number of (G1, msg)-pairs (as a flat list)
    args = rest(a, args)?;

    let mut items = Vec::<(PublicKey, Atom)>::new();
    while !nilp(a, args) {
        let pk = a.g1(first(a, args)?)?;
        args = rest(a, args)?;
        let msg = atom(a, first(a, args)?, "bls_verify message")?;
        args = rest(a, args)?;

        cost += BLS_PAIRING_COST_PER_ARG;
        cost += msg.as_ref().len() as Cost * BLS_MAP_TO_G2_COST_PER_BYTE;
        cost += DST_G2.len() as Cost * BLS_MAP_TO_G2_COST_PER_DST_BYTE;
        check_cost(cost, max_cost)?;

        items.push((pk, msg));
    }

    if !aggregate_verify(&signature, items) {
        Err(EvalErr::BLSVerifyFailed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
}
