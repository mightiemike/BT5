### Title
Admission-only `compiled_class_hash` validation in `convert_rpc_tx_to_internal` is re-applied on the consensus path, causing valid Declare proposals to be rejected when compiler versions diverge — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

`convert_rpc_tx_to_internal` is a shared helper called from both the gateway admission path and the consensus proposal-validation path. For `Declare` transactions it calls `add_class` and then asserts that the user-supplied `compiled_class_hash` equals the hash the local compiler just produced. This assertion is an admission-only invariant: it is correct at the gateway (where the user's claim must be verified) but wrong on the consensus path (where the transaction was already admitted by the proposer's gateway). When a validator node's compiler produces a different `executable_class_hash_v2` than the proposer's compiler did — a realistic condition during rolling upgrades — the assertion fires, the conversion returns `ValidateCompiledClassHashError`, and the entire proposal is dropped as failed.

---

### Finding Description

`convert_rpc_tx_to_internal` is a private helper that is the single shared implementation for two public trait methods:

**Gateway admission path** — `convert_rpc_tx_to_internal_rpc_tx`: [1](#0-0) 

**Consensus proposal-validation path** — `convert_consensus_tx_to_internal_consensus_tx`: [2](#0-1) 

Inside the shared helper, for every `Declare` transaction the code calls `add_class` (which compiles the Sierra class with the local compiler) and then immediately validates the user-supplied hash against the freshly computed one: [3](#0-2) 

`ClassManager::add_class` is idempotent — if the class is already stored it returns the cached `executable_class_hash_v2` immediately: [4](#0-3) 

If the class is **not** yet in the validator's class manager (e.g., the validator is a separate process from the proposer's gateway, or it is a different sequencer node in the BFT committee), `add_class` compiles the Sierra class with the validator's local compiler: [5](#0-4) 

The resulting `executable_class_hash_v2` is then compared against `tx.compiled_class_hash`. The field `tx.compiled_class_hash` was set by the **user** and was already validated by the **proposer's** gateway using the proposer's compiler. If the validator's compiler version differs from the proposer's, the two hashes diverge and the check fails.

The failure propagates up through `convert_consensus_tx_to_internal_consensus_tx` and is caught in `validate_proposal.rs`, which marks the entire proposal as failed: [6](#0-5) 

The analog to the external bug is exact:

| External (Opus) | Sequencer (Apollo) |
|---|---|
| `convert_to_yang` used for both enter and exit | `convert_rpc_tx_to_internal` used for both gateway admission and consensus processing |
| Calls `assert_can_enter` (admission check) | Calls `compiled_class_hash` validation (admission check) |
| Blocks withdrawal of suspended yang | Blocks consensus processing of valid Declare proposals |

---

### Impact Explanation

A validator node that has a different compiler version than the proposer will reject every `Declare` transaction in a proposed block, returning `HandledProposalPart::Failed`. This is a **High** impact: transaction conversion logic rejects valid transactions on the consensus path, matching the scope item *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."* In practice it causes a liveness failure — the validator cannot agree on any block containing a `Declare` transaction — and in a multi-validator BFT setup it can prevent finality entirely.

---

### Likelihood Explanation

The condition is triggered whenever the validator's Sierra-to-CASM compiler produces a different `executable_class_hash_v2` than the proposer's compiler for the same Sierra class. This is a realistic scenario during rolling upgrades (nodes temporarily running different compiler versions), or in any distributed topology where the consensus node's class manager is not pre-populated by the gateway. The `compiler_backward_compatibility_marker` mechanism in the central sync code confirms that compiler-version divergence is an acknowledged operational concern in this codebase. [7](#0-6) 

---

### Recommendation

Split `convert_rpc_tx_to_internal` into two paths:

1. **Gateway admission path** — keep the `compiled_class_hash` validation. This is the correct place to reject user-supplied transactions whose hash claim does not match the compiled output.

2. **Consensus processing path** — call `add_class` for its side-effect (compile and cache the class) but **skip** the `compiled_class_hash != executable_class_hash_v2` assertion. The transaction was already admitted by the proposer's gateway; re-validating the hash against a potentially different local compiler is both redundant and harmful.

A minimal fix mirrors the recommendation in the external report: replace the shared helper with two separate implementations, or add a boolean parameter `validate_compiled_class_hash: bool` that is `true` on the gateway path and `false` on the consensus path.

---

### Proof of Concept

```
1. Node A (proposer, compiler v1.0) admits a Declare tx:
      tx.compiled_class_hash = H1  (= compiler_v1.0(sierra_class))
   Gateway validates: H1 == add_class(sierra_class).executable_class_hash_v2  ✓
   Tx enters mempool, proposer builds a block proposal containing it.

2. Node B (validator, compiler v1.1) receives the proposal.
   validate_proposal calls convert_consensus_tx_to_internal_consensus_tx(tx).
   This calls convert_rpc_tx_to_internal(tx).
   add_class(sierra_class) compiles fresh → H2 = compiler_v1.1(sierra_class), H2 ≠ H1.
   Check: tx.compiled_class_hash (H1) != executable_class_hash_v2 (H2)  → ERROR.
   Returns ValidateCompiledClassHashError::CompiledClassHashMismatch.

3. validate_proposal receives Err, returns HandledProposalPart::Failed.
   Node B rejects the proposal; consensus stalls on any block containing a Declare tx.
```
