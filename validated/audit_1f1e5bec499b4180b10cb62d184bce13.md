### Title
Negative `canSettle` Not Clamped in `settlePnl` Inflates Global Settlement Pool and Corrupts User vQuoteBalance — (`File: core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.settlePnl` calls `getSettlementState`, which can return a **negative** `availableSettle` when a user's position PnL is negative. Because neither `getSettlementState` nor `settlePnl` clamps this value to zero, the negative `canSettle` is applied directly to protocol state, incorrectly **inflating** `state.availableSettle` (phantom settlement capacity) and **increasing** `balance.vQuoteBalance` (reducing the user's virtual loss without payment). This is the direct Nado analog of the external report's pattern: a negative contribution-tracking value is not clamped, corrupting accounting in the user's favor at the protocol's expense.

---

### Finding Description

`getSettlementState` computes `availableSettle` as:

```solidity
availableSettle = MathHelper.min(
    calculatePositionPnl(balance, productId),
    state.availableSettle
);
``` [1](#0-0) 

`calculatePositionPnl` is:

```solidity
positionPnl = priceX18.mul(balance.amount) + balance.vQuoteBalance;
``` [2](#0-1) 

When a user holds a losing position, `calculatePositionPnl` returns a **negative** value. Because `state.availableSettle` is non-negative under normal operation, `MathHelper.min(negative, non-negative)` returns the **negative** position PnL. There is no `max(0, ...)` clamp applied.

`settlePnl` then consumes this unclamped value directly:

```solidity
state.availableSettle -= canSettle;   // subtracts negative → INCREASES pool
balance.vQuoteBalance -= canSettle;   // subtracts negative → INCREASES vQuote
totalSettled += canSettle;            // adds negative → totalSettled goes negative
``` [3](#0-2) 

Three simultaneous corruptions occur:

1. **`state.availableSettle` is inflated** — phantom settlement capacity is created in the global perp state. Other users can now settle more PnL than is backed by real deposited tokens.
2. **`balance.vQuoteBalance` is increased** — the user's virtual loss is reduced without any real payment, improving their position health for free.
3. **`totalSettled` is negative** — the Clearinghouse applies this to the user's spot quote balance, decreasing it, partially masking the benefit but not eliminating the protocol-level accounting corruption.

The root cause is the absence of a `max(0, availableSettle)` clamp in `getSettlementState` or `settlePnl`, directly mirroring the external report's missing clamp on negative `soc_loss_funds`.

---

### Impact Explanation

The inflated `state.availableSettle` is the critical impact. `state.availableSettle` is the global pool from which all users in a given perp market settle their positive PnL. When it is artificially inflated, the protocol allows more cumulative settlement than is backed by deposited collateral. Repeated exploitation across multiple subaccounts or products drains the settlement pool, leaving legitimate profitable traders unable to settle their PnL and rendering the perp engine insolvent. Additionally, the user whose `vQuoteBalance` is incorrectly increased gains improved health, potentially avoiding liquidation they should have faced.

**Corrupted state delta**: `state.availableSettle` (per-product global), `balance.vQuoteBalance` (per-user per-product).

---

### Likelihood Explanation

`settlePnl` is reachable through the standard transaction submission path: `Endpoint.sol → Clearinghouse.sol → PerpEngine.settlePnl`. [4](#0-3)  Any trader with an open losing perp position can trigger this by submitting a `SettlePnl` transaction. No special privileges are required. The condition (negative position PnL) is a routine market state, not an edge case.

---

### Recommendation

Clamp `availableSettle` to zero in `getSettlementState` before returning:

```solidity
availableSettle = MathHelper.max(
    0,
    MathHelper.min(
        calculatePositionPnl(balance, productId),
        state.availableSettle
    )
);
``` [5](#0-4) 

This ensures `settlePnl` only processes non-negative settlement amounts, matching the intended semantics: only positive PnL is settled through this path. Negative PnL is handled separately via `socializeSubaccount` during liquidation. [6](#0-5) 

---

### Proof of Concept

**Setup**: Product `P` has `state.availableSettle = 100e18`. Attacker opens a long position and the price drops such that `calculatePositionPnl` returns `-50e18`.

**Step 1**: Attacker submits a `SettlePnl` transaction through `Endpoint.sol`.

**Step 2**: `getSettlementState` computes:
```
availableSettle = min(-50e18, 100e18) = -50e18
``` [1](#0-0) 

**Step 3**: `settlePnl` applies:
```
state.availableSettle -= (-50e18)  →  state.availableSettle = 150e18  (was 100e18)
balance.vQuoteBalance -= (-50e18)  →  vQuoteBalance increases by 50e18
totalSettled = -50e18
``` [3](#0-2) 

**Step 4**: The Clearinghouse applies `totalSettled = -50e18` to the attacker's spot quote (decreasing it by 50e18). However, `state.availableSettle` is now 150e18 — 50e18 more than the real backing.

**Step 5**: A legitimate profitable trader settles 150e18 of PnL against the inflated pool. The protocol pays out 50e18 more than it has, draining the settlement pool and leaving other traders unable to settle.

**Net protocol loss**: `50e18` per exploit iteration, unbounded with repetition across multiple products or subaccounts.
