Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the end user, allowing any caller to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` resolves to the router contract address, not the actual end user. A pool admin who allowlists the router to enable router-based swaps for permitted users inadvertently grants every unpermissioned user the ability to bypass the allowlist entirely by routing through the public router.

## Finding Description

**Root cause — identity collapse at the pool boundary:**

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` (the immediate caller) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to the registered extension via `_callExtensionsInOrder`.

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly, making the router the `msg.sender` of that call: [3](#0-2) 

The allowlist check therefore resolves to `allowedSwapper[pool][router]`. Once the router is allowlisted (the natural operational step for a pool admin who wants to support router-based swaps for their permitted users), every caller — including those explicitly excluded — can execute swaps by routing through the public router. The same identity collapse occurs in `exactInput`, `exactOutputSingle`, and `exactOutput`: [4](#0-3) 

No existing guard in the extension or the pool checks the original transaction initiator (`tx.origin` or an explicit user field in `extensionData`). The extension has no router-awareness and no mechanism to decode the real user from `extensionData`.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to KYC'd or otherwise vetted addresses loses that restriction entirely for any user who routes through the public `MetricOmmSimpleRouter`. Unauthorized users can execute swaps against the pool's liquidity, draining LP value or executing trades the pool admin explicitly intended to block. This constitutes broken core pool access-control functionality with direct exposure of LP assets to unauthorized traders — a high-severity broken invariant under the contest's allowed impact gate.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entry point. Any pool admin who deploys a swap-allowlisted pool and also wants to support router-based swaps for their permitted users will naturally add the router to the allowlist — this is the expected operational pattern. No special privileges, unusual inputs, or flash loans are required. Any EOA can call `exactInputSingle` with a standard swap payload. The bypass is repeatable on every block.

## Recommendation

The extension must gate the **original end user**, not the intermediary router. Two complementary approaches:

1. **Pass the original initiator through the router.** The router should encode `msg.sender` (the actual user) as an explicit field in `extensionData`, and `SwapAllowlistExtension` should decode and check that address when `sender` is a known/trusted router address.

2. **Maintain a router registry in the extension.** The extension can keep a set of trusted router addresses; when `sender` is a trusted router, it decodes the real user from `extensionData` before performing the allowlist lookup.

The cleanest invariant is: the allowlist always checks the address that economically initiates the swap (the entity that controls the payer), not the contract that mechanically forwards the call.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension (BEFORE_SWAP_ORDER = extension 1)
  - Pool admin calls setAllowedToSwap(pool, userA, true)   // legitimate user
  - Pool admin calls setAllowedToSwap(pool, router, true)  // to enable router swaps for userA
  - Pool admin does NOT allowlist userB (attacker)

Attack:
  1. userB calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(...) — msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  5. Swap executes successfully for userB despite being explicitly excluded

Result:
  userB swaps against the pool's liquidity with no allowlist enforcement.
  The allowlist invariant is broken; LP assets are exposed to unauthorized traders.
```

Foundry test outline:
- Deploy `SwapAllowlistExtension`, configure a pool with it as `BEFORE_SWAP_ORDER = 1`.
- `setAllowedToSwap(pool, router, true)`, leave `userB` unlisted.
- Prank as `userB`, call `router.exactInputSingle(...)`.
- Assert the call succeeds (no `NotAllowedToSwap` revert) and token balances change, confirming the bypass.
