Audit Report

## Title
SwapAllowlistExtension Gates on Router Address Instead of End User, Allowing Any User to Bypass Swap Allowlist via Router — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which is the pool's own `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. A pool admin who allowlists the router to support periphery routing inadvertently grants every unprivileged user the ability to bypass the swap allowlist entirely.

## Finding Description
In `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, recipient, ...)` at line 230, passing its own `msg.sender` as the `sender` argument. `ExtensionCalling._beforeSwap` forwards this value verbatim as the first argument to the extension's `beforeSwap` function (line 165). `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant), the router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` directly. The pool therefore sees `msg.sender == router`, and that is the value forwarded to the extension as `sender`. The actual end-user address is never visible to the extension.

A pool admin who wants to allow router-mediated swaps on a curated pool must call `setAllowedToSwap(pool, router, true)`. Once the router is in `allowedSwapper[pool][router]`, the check passes for every call arriving through the router — regardless of who the actual end user is. The allowlist is silently reduced to a single entry (the router) that any user can exploit as a pass-through.

## Impact Explanation
Broken core pool functionality. A curated pool's entire swap-access policy is defeated. Any unprivileged address can trade on a pool explicitly configured to restrict swappers, bypassing the allowlist with zero cost beyond gas. This enables unauthorized extraction of LP value from pools designed for private or institutional use, and circumvention of compliance or KYC-gated pools. The invariant that `allowedSwapper` represents the set of addresses permitted to trade is broken.

## Likelihood Explanation
High. `MetricOmmSimpleRouter` is the canonical periphery entry point for swaps. Pool admins who configure `SwapAllowlistExtension` but still want to support router-mediated swaps for their allowlisted users will naturally add the router to the allowlist. The bypass requires no special privileges, no flash loans, and no contract deployment — any EOA can call the router directly.

## Recommendation
The extension must gate on the economic actor — the end user — not the immediate caller of `pool.swap()`. Options:

1. **Pass the originating user explicitly.** Modify `ExtensionCalling._beforeSwap` to forward the `recipient` or a separately supplied `originatingUser` field, and have the extension check that field instead of `sender`.
2. **Check `tx.origin` as a fallback.** For EOA-only curated pools, `tx.origin` correctly identifies the end user even through the router. Acceptable when the pool admin's threat model excludes contract-based callers.
3. **Document the constraint.** If the design intent is that `sender` is always the immediate caller, the extension's NatSpec must warn that allowlisting any intermediary (router, aggregator) opens the pool to all users of that intermediary.

## Proof of Concept
```
Setup:
  pool = deploy MetricOmmPool with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, router, true)   // router allowlisted so users can route
  admin does NOT allowlist attacker (0xDEAD)

Attack:
  attacker (0xDEAD) calls MetricOmmSimpleRouter.exactInputSingle(...)
    → router calls pool.swap(recipient=attacker, ...)
    → pool passes msg.sender=router as `sender` to _beforeSwap
    → SwapAllowlistExtension checks allowedSwapper[pool][router] == true  ✓
    → swap executes for attacker despite attacker not being allowlisted

Result:
  attacker successfully swaps on a pool they were never authorized to access.
```

Foundry test plan: deploy `SwapAllowlistExtension`, configure a pool with it, allowlist only the router, then call `MetricOmmSimpleRouter.exactInputSingle` from an un-allowlisted EOA and assert the swap succeeds (no `NotAllowedToSwap` revert).
