### Title
Unprivileged Caller Can Directly Invoke `processTransactionImpl` to Manipulate Any User's Fee Tier Without Signature Validation — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

`processTransactionImpl` is declared `public` in `EndpointTx.sol` with no caller access guard. Within it, the `UpdateFeeTier` transaction branch invokes `clearinghouse.updateFeeTier` with zero signature validation. Because `Endpoint` delegates execution to `EndpointTx` (running in `Endpoint`'s storage context), any unprivileged caller can invoke `processTransactionImpl` directly on `Endpoint`, craft an `UpdateFeeTier` payload, and overwrite any address's fee tier — including their own — to an arbitrary value, bypassing protocol trading fees.

---

### Finding Description

`EndpointTx.processTransactionImpl` is the internal sequencer dispatch loop. It is declared `public` with no `onlySequencer` or equivalent modifier:

```solidity
function processTransactionImpl(bytes calldata transaction) public {
``` [1](#0-0) 

Every other sensitive transaction branch inside this function — `LiquidateSubaccount`, `WithdrawCollateral`, `LinkSigner`, `TransferQuote`, etc. — calls `validateSignedTx` or `validateCompactSignature` before acting. The `UpdateFeeTier` branch does neither:

```solidity
} else if (txType == IEndpoint.TransactionType.UpdateFeeTier) {
    clearinghouse.updateFeeTier(transaction);
}
``` [2](#0-1) 

`clearinghouse.updateFeeTier` is gated only by `onlyEndpoint` (requires `msg.sender == Endpoint`):

```solidity
function updateFeeTier(bytes calldata transaction) external onlyEndpoint {
``` [3](#0-2) 

Because `Endpoint` delegates to `EndpointTx` via `delegatecall` (running in `Endpoint`'s execution context), a direct call to `processTransactionImpl` on `Endpoint` satisfies the `onlyEndpoint` guard in `Clearinghouse`. The call chain is:

```
Attacker → Endpoint.processTransactionImpl(UpdateFeeTier{user: attacker, newTier: 0})
         → [delegatecall context = Endpoint]
         → Clearinghouse.updateFeeTier(...)   // msg.sender == Endpoint ✓
         → OffchainExchange.updateFeeTier(attacker, 0)
```

`OffchainExchange.updateFeeTier` writes the new tier unconditionally:

```solidity
feeTiers[user] = newTier;
``` [4](#0-3) 

The fee tier is then consumed by every order match to determine the maker/taker rate applied to the subaccount:

```solidity
function getTierFeeRateX18(uint32 tier, uint32 productId) public view returns (FeeRates memory) {
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        return feeRates[tier][productId];
    }
    return FeeRates({ makerRateX18: 0, takerRateX18: 200_000_000_000_000 });
}
``` [5](#0-4) 

An attacker who sets their tier to a custom tier configured with `takerRateX18 = 0` (or a negative rebate tier, given `TAKER_FEE_ACCRUAL_RATE_X18 = -300_000_000_000_000`) pays zero or negative fees on every trade. [6](#0-5) 

---

### Impact Explanation

- **Fee theft / protocol revenue loss**: The attacker sets their own address to a zero-fee or rebate tier, paying no taker fees on all subsequent trades. At scale (high-frequency trading), this drains the fee pool that accrues to `X_ACCOUNT` and ultimately to the protocol.
- **Griefing**: The attacker can also set *other* users' addresses to higher-fee tiers, increasing their trading costs without consent.
- **Accounting corruption**: `feeTiers[user]` is a persistent mapping; the corruption persists until an admin issues a corrective `UpdateFeeTier` through the sequencer.

---

### Likelihood Explanation

The entry path requires only a direct call to a `public` function with a crafted ABI-encoded payload — no special role, no leaked key, no governance capture. The `UpdateFeeTier` struct is simple (`address user`, `uint32 newTier`) and fully attacker-controlled. Any on-chain actor who can send a transaction can exploit this. [7](#0-6) 

---

### Recommendation

1. **Add a sequencer-only guard to `processTransactionImpl`**: The function should only be callable by the trusted sequencer (or internally via `submitTransactions`). Add `require(msg.sender == sequencer || msg.sender == address(this))` or an `onlySequencer` modifier.
2. **Add signature validation for `UpdateFeeTier`**: Even if the sequencer guard is added, the `UpdateFeeTier` branch should validate a signed payload from an authorized admin, consistent with how other privileged transaction types are handled.
3. **Audit all branches of `processTransactionImpl` and `processSlowModeTransactionImpl` for missing signature checks**: `ManualAssert`, `AssertCode`, `AssertProduct`, and `DumpFees` branches similarly lack per-call signature validation and should be reviewed.

---
