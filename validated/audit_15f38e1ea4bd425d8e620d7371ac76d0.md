Audit Report

## Title
Per-Block Price Velocity Cap Bypassed by Rolling Reference Price in Same-Block Swaps — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap` writes `lastMidPriceX64` and `lastUpdateBlock` to storage **before** performing the velocity check. When multiple swaps execute in the same block — each preceded by a permissionless oracle price push with a strictly increasing millisecond timestamp — every subsequent swap computes `blockDiff = 0` against the **previous swap's price** as its reference, not the block-start price. This allows each swap to move the oracle mid-price by the full `maxChangePerBlockE18` relative to the prior swap, compounding to N × `maxChangePerBlockE18` total intra-block movement without ever triggering `PriceVelocityExceeded`.

## Finding Description

In `beforeSwap`, state is written unconditionally before the guard check: [1](#0-0) 

The check then uses the just-captured `prevMid` and `prevBlock`: [2](#0-1) 

Each call to `MetricOmmPool.swap` fetches a fresh oracle price via `_getBidAndAskPriceX64()` before invoking `_beforeSwap`: [3](#0-2) 

The oracle price is keyed on off-chain millisecond timestamps, not `block.number`. Both `CompressedOracle` (via permissionless `fallback()`) and `PythOracle` (via permissionless `fallback()`) accept updates from any caller as long as the new timestamp strictly exceeds the stored one: [4](#0-3) 

Within a single 12-second Ethereum block, many legitimate off-chain price updates with distinct millisecond timestamps exist. An attacker can interleave oracle pushes and swaps in one transaction or across sequential transactions in the same block:

1. Push oracle to P1 (within `maxChange` of P0, timestamp T1). Call `swap` → guard records `lastMidPriceX64 = P1`, `lastUpdateBlock = N`.
2. Push oracle to P2 (within `maxChange` of P1, timestamp T2 > T1). Call `swap` → `prevMid = P1`, `prevBlock = N`, `blockDiff = 0`, `allowedSq = maxChange²`. Guard passes because `|P2 − P1| ≤ maxChange`.

After two swaps: total movement from P0 is ≈ 2 × `maxChange`. After N swaps: ≈ N × `maxChange`. The guard's stated invariant (`changeE18² ≤ maxChange² × (1 + blockDiff)`) is satisfied per-swap but violated in aggregate. [5](#0-4) 

## Impact Explanation

The velocity guard's sole purpose is to cap intra-block oracle mid-price movement. Its bypass allows the oracle-derived mid-price used for swap settlement to move by an unbounded multiple of `maxChangePerBlockE18` within a single block. Swaps execute against this manipulated mid-price, constituting **bad-price execution** and direct LP fund loss. This matches the allowed impact gate: bad-price execution where an unclamped oracle quote reaches a pool swap.

## Likelihood Explanation

The attack requires only: (1) access to legitimate off-chain oracle price updates with distinct millisecond timestamps (freely available from Pyth/Lazer feeds), and (2) the ability to call the permissionless oracle `fallback()` and `pool.swap()`. No privileged role, no malicious pool setup, and no non-standard token is needed. Any EOA or contract can execute this within a single block.

## Recommendation

Track a separate `blockStartMidPriceX64` that is only updated when `block.number > lastUpdateBlock`. For same-block swaps (`blockDiff == 0`), compare the current mid-price against `blockStartMidPriceX64` (the price at the start of the block), not against the last swap's rolling price:

```solidity
if (block.number > prevBlock) {
    s.blockStartMidPriceX64 = prevMid; // anchor for this new block
}
uint128 blockRef = (blockDiff == 0) ? s.blockStartMidPriceX64 : prevMid;
// use blockRef for delta and changeE18 computation
```

This ensures the full-block envelope is enforced regardless of how many swaps or oracle updates occur within the block.

// Foundry test sketch
function test_sameBlockVelocityBypass() public {
    // Setup: pool with PriceVelocityGuard, maxChangePerBlockE18 = 0.05e18 (5%)
    // Oracle at P0, lastMidPriceX64 = P0, lastUpdateBlock = N-1

    vm.roll(N);

    // Step 1: push oracle to P1 = P0 * 1.05 (exactly at cap), call swap
    oracle.fallback(encodeUpdate(P1, T1)); // T1 > T0, T1 <= block.timestamp
    pool.swap(...); // blockDiff=1, allowedSq=M²*2, passes; lastMidPriceX64=P1

    // Step 2: push oracle to P2 = P1 * 1.05, call swap (same block N)
    oracle.fallback(encodeUpdate(P2, T2)); // T2 > T1, T2 <= block.timestamp
    pool.swap(...); // blockDiff=0, allowedSq=M²*1, passes; lastMidPriceX64=P2

    // Total movement: P2 = P0 * 1.1025 (+10.25%) in one block
    // Guard cap was 5%; PriceVelocityExceeded was never thrown
    assertGt(P2, P0 * 110 / 100);
}
