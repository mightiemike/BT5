### Title
SwapAllowlistExtension Gates Router Address Instead of Actual User, Allowing Any Caller to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` to the pool. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. If the pool admin allowlists the router (which is necessary for any allowlisted user to use the router), every unpermissioned address can bypass the per-user swap gate by calling through the router.

### Finding Description

**Call chain for a router-mediated swap:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)`.
2. Router calls `IMetricOmmPoolActions(pool).swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)` — the pool's `msg.sender` is the router.
3. Pool calls `_beforeSwap(msg.sender /*= router*/, recipient, …)`.
4. `ExtensionCalling._beforeSwap` encodes `sender = router` and dispatches to the extension.
5. `SwapAllowlistExtension.beforeSwap` receives `sender = router` and evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool; `sender` is the router. The check resolves to `allowedSwapper[pool][router]`.

**The dilemma this creates:**

- If the router is **not** allowlisted → every allowlisted user is also blocked from using the router; they must call the pool directly, making the router unusable for curated pools.
- If the router **is** allowlisted → the per-user gate is completely open; any address can call `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` through the router and the extension passes unconditionally.

The same wrong-actor binding applies to every router entry point. In `exactInput` multi-hop, the pool's `msg.sender` is the router for every hop. In `exactOutput` recursive callbacks, the pool is called from inside `metricOmmSwapCallback`, where `msg.sender` to the pool is again the router.

**Contrast with `DepositAllowlistExtension`:** that extension ignores the `sender` parameter entirely and gates on `owner` — the economically relevant actor for deposits. `SwapAllowlistExtension` has no equivalent mechanism to recover the real user identity from the router call.

**No existing test covers this path.** `FullMetricExtension.t.sol` allowlists a `TestCaller` contract and calls the pool directly through it; the router path is never exercised against the allowlist.

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs, or whitelisted market makers) loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The attacker receives real token output from the pool's LP reserves without being on the allowlist. This is a direct loss of the curation invariant and constitutes unauthorized access to pool liquidity, which can drain LP value if the pool's pricing is calibrated for a specific counterparty set.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface. Pool admins who deploy a curated pool and want their allowlisted users to have a normal UX will allowlist the router. The moment they do, the gate is open to everyone. The trigger requires no special privilege — any EOA or contract can call the router's public `exact*` functions.

### Recommendation

The extension must recover the real user identity rather than trusting the `sender` argument, which is the router when the router is used. Options:

1. **Decode the real swapper from `extensionData`**: require the router to embed the original `msg.sender` in `extensionData` and have the extension verify and decode it (requires a signed or trusted encoding scheme).
2. **Gate on `recipient` instead of `sender`** if the pool's curation intent is to restrict who receives output tokens (not who initiates the call).
3. **Document that `SwapAllowlistExtension` is incompatible with the router** and enforce this at the factory level by preventing pools that configure this extension from being reachable through the router.

### Proof of Concept

```solidity
// Pool is configured with SwapAllowlistExtension.
// Admin allowlists the router so that allowlisted users can use it.
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// attacker is NOT on the per-user allowlist:
// swapExtension.allowedSwapper[pool][attacker] == false

// But attacker calls through the router:
vm.startPrank(attacker);
token0.approve(address(router), type(uint256).max);
// This succeeds: the extension sees sender=router, which IS allowlisted.
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool:            address(pool),
    recipient:       attacker,
    tokenIn:         address(token0),
    zeroForOne:      true,
    amountIn:        1000,
    amountOutMinimum: 0,
    priceLimitX64:   0,
    extensionData:   "",
    deadline:        block.timestamp + 1
}));
vm.stopPrank();
// attacker receives token1 output despite not being on the allowlist.
```
