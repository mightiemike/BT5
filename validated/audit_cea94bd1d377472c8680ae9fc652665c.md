### Title
GC-Pruned Block Causes Unguarded Panic in `verify_transaction_inclusion_v2`, Permanently Blocking SPV Proof Verification - (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion_v2` function performs a direct `headers_pool` lookup for the caller-supplied `tx_block_blockhash` before any guard is applied. If that block has been removed by the garbage-collector (`run_mainchain_gc`), the contract panics unconditionally. The guard that is supposed to prevent this (`args.confirmations <= self.gc_threshold`) lives only in the deprecated `verify_transaction_inclusion` v1 path, is checked *after* the panicking lookup in v2, and checks the wrong invariant regardless: it validates the *number of confirmations requested*, not whether the *target block itself* is still within the GC window.

---

### Finding Description

`run_mainchain_gc` calls `remove_block_header` for every pruned block:

```rust
// contract/src/lib.rs  lines 659-662
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [1](#0-0) 

After GC the block hash is absent from **both** `mainchain_header_to_height` and `headers_pool`.

`verify_transaction_inclusion_v2` (the current, non-deprecated entry point) immediately looks up the block in `headers_pool` with no preceding guard:

```rust
// contract/src/lib.rs  lines 353-356
let header = self
    .headers_pool
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));
``` [2](#0-1) 

If the block has been GC'd this line panics. The `confirmations <= gc_threshold` guard that exists in v1 is only reached *after* this lookup (via the tail call on line 368), so it never fires for a GC'd block. [3](#0-2) 

Even in the v1 path the guard is checking the wrong invariant:

```rust
// contract/src/lib.rs  lines 289-292
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [4](#0-3) 

This allows `confirmations = 1` to pass while the target block is arbitrarily old and already pruned. The correct invariant is `tip_height − target_block_height < gc_threshold`, which is never checked.

GC runs automatically on every `submit_blocks` call:

```rust
// contract/src/lib.rs  lines 181
self.run_mainchain_gc(num_of_headers);
``` [5](#0-4) 

So the pruning window advances with every relayer batch, silently invalidating older block hashes.

---

### Impact Explanation

Any NEAR contract that calls `verify_transaction_inclusion_v2` to gate a state transition (e.g., a bridge releasing funds after a Bitcoin deposit confirmation) will receive an unrecoverable panic for any target block that has been GC'd. Because NEAR cross-contract calls propagate panics as transaction failures, the downstream contract's action is permanently blocked — the proof can never be re-submitted successfully for that block. This is a permanent loss of verifiability for legitimate SPV proofs, not merely a transient rejection.

---

### Likelihood Explanation

The default recommended `gc_threshold` is 52 704 blocks (~1 year of Bitcoin blocks per the init comment at line 131). However: [6](#0-5) 

- `gc_threshold` is a freely configurable init parameter with no enforced minimum.
- Any deployment with a smaller threshold (e.g., a few thousand blocks, ~weeks) makes the window practically reachable.
- An unprivileged caller needs only to supply a `tx_block_blockhash` that is older than the current GC horizon — no special role, no key material, no social engineering.
- The relayer submits blocks continuously, so the GC horizon advances automatically without any attacker action.

---

### Recommendation

1. **In `verify_transaction_inclusion_v2`**: add a guard before the `headers_pool` lookup that checks whether the target block is still within the GC window:
   ```rust
   let target_height = self.mainchain_header_to_height
       .get(&args.tx_block_blockhash)
       .unwrap_or_else(|| env::panic_str("block not on main chain"));
   let tip_height = self.get_last_block_height();
   require!(
       tip_height.saturating_sub(target_height) < self.gc_threshold,
       "Target block has been garbage collected"
   );
   ```
2. **Replace the v1 guard** (`confirmations <= gc_threshold`) with the correct invariant: `tip_height − target_block_height < gc_threshold`.
3. **Document** the GC window as a hard deadline for proof submission, analogous to the 256-block limitation described in the external report.

---

### Proof of Concept

1. Deploy the contract with `gc_threshold = 100`.
2. Submit 201 blocks via `submit_blocks`. GC fires automatically and prunes blocks at heights 0–100, leaving heights 101–200 in storage.
3. As an unprivileged NEAR account, call `verify_transaction_inclusion_v2` with:
   - `tx_block_blockhash` = hash of block at height 50 (GC'd)
   - `confirmations = 1`
   - valid-looking (or empty) merkle proofs
4. Execution reaches `contract/src/lib.rs` line 353–356; `headers_pool.get(...)` returns `None`; `env::panic_str("cannot find requested transaction block")` fires.
5. The guard at line 289–292 (`confirmations (1) <= gc_threshold (100)`) is never reached.
6. The transaction reverts. Any downstream contract gating on this call is permanently blocked for block 50. [7](#0-6) [8](#0-7)
