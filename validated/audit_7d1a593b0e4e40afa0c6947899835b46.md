Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the end user, enabling full allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router is `msg.sender` of `pool.swap()`, so the allowlist check becomes `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][endUser]`. Any pool that allowlists the router to support normal periphery usage inadvertently grants every address — including explicitly blocked ones — the ability to swap freely.

## Finding Description
The exploit chain is confirmed by the production code at each step:

**Step 1:** `MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

**Step 2:** `ExtensionCalling._beforeSwap` forwards `sender` unchanged into the ABI-encoded call to every configured extension: [2](#0-1) 

**Step 3:** `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Step 4:** `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no forwarding of the original `msg.sender` (the end user). The router stores the end user only in transient storage for the payment callback, not in the swap call itself: [4](#0-3) 

Therefore, `sender` arriving at the extension is always the router address. The check `allowedSwapper[pool][router] == true` passes for every caller of the router, regardless of whether the end user is individually blocked. There is no existing guard in the extension, the pool, or the router that recovers the original initiator and passes it to the extension.

## Impact Explanation
A curated pool (KYC-gated, institution-only, etc.) that relies on `SwapAllowlistExtension` to restrict traders is completely voided the moment the router is allowlisted. Any unpermissioned address calls `router.exactInputSingle()` and swaps freely at oracle-derived prices. LPs suffer direct principal loss as unauthorized users drain one side of the pool. This meets the **Critical/High direct loss of user principal** threshold: the allowlist invariant — the sole access-control mechanism for curated pools — is fully bypassed with no special preconditions.

## Likelihood Explanation
High. The router is the canonical, documented user-facing entrypoint. Any pool that enables `SwapAllowlistExtension` and also wants users to use the router must allowlist the router, which automatically triggers the bypass. No privileged access, no non-standard tokens, and no unusual conditions are required. The attacker only needs to call `router.exactInputSingle()`.

## Recommendation
The extension must verify the economic actor, not the immediate caller. Concrete options:

1. **Pass the original initiator via `extensionData`:** Have the router encode `msg.sender` in `extensionData` and have the extension decode and verify it, with a trust check that `msg.sender` (the pool's caller) is a known router.
2. **Expose the payer from transient storage:** The router already stores the end user in transient storage via `_setNextCallbackContext` (the `payer` field). Expose it as a verified field passed through the swap call or readable by the extension.
3. **Document and enforce incompatibility:** Add a check in `ValidateExtensionsConfig` (or equivalent factory validation) that rejects pool configurations combining `SwapAllowlistExtension` with any router-callable path, and provide a separate router-aware allowlist extension that verifies both the router address and a caller-supplied identity (e.g., via EIP-712 signature).

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension configured.
2. Pool admin: setAllowedToSwap(pool, userA, true)   // userA is KYC'd
3. Pool admin: setAllowedToSwap(pool, router, true)  // required for router support
   // userB is NOT allowlisted
4. userB calls router.exactInputSingle({pool: pool, ...})
   → router calls pool.swap(recipient, ...) — msg.sender = router
   → pool calls _beforeSwap(sender=router, ...)
   → extension checks allowedSwapper[pool][router] == true  ✓
   → swap executes for userB
5. userB, who should be blocked, has swapped on the curated pool.
```
