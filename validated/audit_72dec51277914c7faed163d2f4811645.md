### Title
Protobuf `ValidResourceBounds` round-trip silently downgrades `AllResources` to `L1Gas` when l2_gas and l1_data_gas are zero, causing valid post-0.13.3 transactions to be rejected by peers — (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to reconstruct the enum variant: if both `l2_gas` and `l1_data_gas` are zero it silently returns `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. A valid post-0.13.3 `RpcTransaction` whose user deliberately set only `l1_gas` (leaving `l2_gas` and `l1_data_gas` at zero) is accepted by the gateway, but after any protobuf round-trip the variant is downgraded to `L1Gas`. The downstream `RpcDeployAccountTransactionV3` (and analogous Invoke/Declare) converters then fail with `DEPRECATED_RESOURCE_BOUNDS_ERROR`, causing the transaction to be rejected by every receiving peer — mempool P2P and consensus validators alike.

---

### Finding Description

**Step 1 – The lossy protobuf deserializer.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` reconstructs the variant purely from the numeric values of the deserialized fields:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← pre-0.13.3 path
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

There is no discriminator bit in the wire format; the variant is inferred entirely from whether the numeric fields are zero.

**Step 2 – A valid gateway transaction that triggers the downgrade.**

`RpcTransaction` (and all three V3 sub-types) stores `AllResourceBounds`, not `ValidResourceBounds`. The stateless gateway validator explicitly accepts transactions where only `l1_gas` is non-zero and both `l2_gas` and `l1_data_gas` are zero: [2](#0-1) 

Such a transaction is accepted, assigned a hash computed with `ValidResourceBounds::AllResources` (which includes the `L1_DATA_GAS` term in `get_tip_resource_bounds_hash`), and forwarded to the mempool.

**Step 3 – The hash domain divergence.**

`get_tip_resource_bounds_hash` produces structurally different output for the two variants:

- `L1Gas` → hashes `[tip, L1_GAS_concat, L2_GAS_concat]` (2 resource felts)
- `AllResources` with zero l2/l1_data → hashes `[tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]` (3 resource felts) [3](#0-2) 

The gateway stores the hash computed under `AllResources`. After protobuf round-trip the variant becomes `L1Gas`, so any peer that recomputes the hash gets a different value.

**Step 4 – Hard rejection at the receiving peer.**

The `RpcDeployAccountTransactionV3` (and analogous Invoke/Declare) protobuf converters route through the `snapi` (`ValidResourceBounds`) layer and then attempt to narrow to `AllResourceBounds`:

```rust
impl TryFrom<protobuf::DeployAccountV3> for RpcDeployAccountTransactionV3 {
    fn try_from(value: protobuf::DeployAccountV3) -> Result<Self, Self::Error> {
        let snapi_deploy_account: DeployAccountTransactionV3 = value.try_into()?;
        // This conversion can fail only if the resource_bounds are not AllResources.
        snapi_deploy_account.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)
    }
}
``` [4](#0-3) 

When the deserialized variant is `L1Gas`, the narrowing fails and the peer returns `DEPRECATED_RESOURCE_BOUNDS_ERROR` (`MissingField { field_description: "ResourceBounds::l1_data_gas" }`), dropping the transaction entirely.

**Step 5 – The developers already know, but only patched the test.**

The consensus round-trip test explicitly works around the issue by injecting a non-zero `l2_gas.max_amount` before serialization:

```rust
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    ...
    resource_bounds.l2_gas.max_amount = GasAmount(1);
    ...
}
``` [5](#0-4) 

No equivalent guard exists in production code.

---

### Impact Explanation

A valid, gateway-accepted `RpcTransaction` (Invoke, Declare, or DeployAccount) whose user set only `l1_gas` and left `l2_gas` and `l1_data_gas` at zero is silently dropped by every peer that receives it over P2P (mempool propagation or consensus proposal). The transaction can never be sequenced. In the consensus path, a proposer that includes such a transaction will have its proposal rejected by validators, stalling the round.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The stateless validator explicitly permits `AllResourceBounds` with only `l1_gas` non-zero. Any user who submits such a transaction (e.g., a legacy-style V3 invoke that only cares about L1 gas) will hit this silently. The trigger requires no special privilege and no malformed bytes — only a standard, well-formed transaction.

---

### Recommendation

Add a discriminator field to the `ResourceBounds` protobuf message (e.g., a boolean `is_all_resources`) so the variant can be reconstructed faithfully without relying on whether numeric fields are zero. Alternatively, always serialize and deserialize as `AllResources` for post-0.13.3 transactions and reserve the `L1Gas` path exclusively for the legacy sync path where the original wire format is preserved verbatim.

---

### Proof of Concept

1. Construct a valid `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: { max_amount: 100, max_price_per_unit: 1 }, l2_gas: default(), l1_data_gas: default() }`.
2. Submit it to the gateway — it passes stateless and stateful validation.
3. The gateway computes `tx_hash` using `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources`, which includes the `L1_DATA_GAS` concat felt (zero value) in the hash preimage.
4. The transaction is forwarded to the mempool and then serialized as a `MempoolTransaction` protobuf for P2P propagation.
5. A receiving peer deserializes the `ResourceBounds` protobuf: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas(l1_gas)`.
6. The peer attempts `RpcInvokeTransactionV3::try_from(protobuf::InvokeV3)` → routes through `InvokeTransactionV3` (with `ValidResourceBounds::L1Gas`) → narrowing to `AllResourceBounds` fails → `DEPRECATED_RESOURCE_BOUNDS_ERROR` is returned.
7. The transaction is dropped. It never reaches the peer's mempool and cannot be sequenced.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    let transaction = &mut transactions[0];
    match transaction {
        ConsensusTransaction::RpcTransaction(rpc_transaction) => match rpc_transaction {
            RpcTransaction::Declare(RpcDeclareTransaction::V3(RpcDeclareTransactionV3 {
                resource_bounds,
                ..
            }))
            | RpcTransaction::Invoke(RpcInvokeTransaction::V3(RpcInvokeTransactionV3 {
                resource_bounds,
                ..
            }))
            | RpcTransaction::DeployAccount(RpcDeployAccountTransaction::V3(
                RpcDeployAccountTransactionV3 { resource_bounds, .. },
            )) => {
                resource_bounds.l2_gas.max_amount = GasAmount(1);
            }
        },
        ConsensusTransaction::L1Handler(_) => {}
    }
}
