### Title
Stale `digestToSubaccount` Mapping Not Cleared on Isolated Subaccount Close Allows Re-Opening Positions on Parentless Subaccounts, Causing Fund Loss — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

When an isolated subaccount is closed via `_tryCloseIsolatedSubaccount`, the `digestToSubaccount[digest]` mapping that was set during `createIsolatedSubaccount` is never cleared. If the original isolated order was not fully filled before the subaccount was closed (e.g., the position was closed via liquidation or a reduce-only order), the stale digest mapping persists. A subsequent `matchOrders` call will redirect the order sender to the now-closed isolated subaccount — one whose `parentSubaccounts` entry has been zeroed — causing any resulting PnL or quote balance to be transferred to `bytes32(0)` (the zero subaccount) when the subaccount is closed again, permanently destroying those funds.

---

### Finding Description

**Setup path — `createIsolatedSubaccount`:**
When a user submits a `CreateIsolatedSubaccount` transaction, `OffchainExchange.createIsolatedSubaccount` is called. It sets `digestToSubaccount[orderDigest] = isolatedSubaccount` and `digestToMargin[orderDigest] = margin`. The new isolated subaccount is registered via `_recordSubaccount` and `parentSubaccounts[isolatedSubaccount] = parentSubaccount` is written.

**Teardown path — `_tryCloseIsolatedSubaccount`:**
When the isolated subaccount's perp position reaches zero, `_tryCloseIsolatedSubaccount` is called. It clears three pieces of state:

```solidity
isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
isolatedSubaccounts[parent][id] = bytes32(0);
parentSubaccounts[subaccount] = bytes32(0);
``` [1](#0-0) 

It does **not** clear `digestToSubaccount[digest]` or `digestToMargin[digest]`.

**Stale redirect in `matchOrders`:**
Every call to `matchOrders` checks whether the taker or maker digest maps to an isolated subaccount and, if so, silently replaces the order sender:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
}
``` [2](#0-1) 

This redirect happens **after** the `isIsolatedSubaccount` guard that blocks isolated subaccounts from being used as direct senders:

```solidity
require(!RiskHelper.isIsolatedSubaccount(taker.order.sender), ERR_INVALID_TAKER);
``` [3](#0-2) 

So the guard is bypassed by the stale mapping.

**Broken parent lookup after close:**
After `_tryCloseIsolatedSubaccount` runs, `parentSubaccounts[isolatedSubaccount] == bytes32(0)`. When `_tryCloseIsolatedSubaccount` is called a second time (after the re-opened position is closed), it reads:

```solidity
bytes32 parent = parentSubaccounts[subaccount]; // == bytes32(0)
``` [4](#0-3) 

and then transfers any remaining `vQuoteBalance` and `quoteBalance` to `bytes32(0)`:

```solidity
perpEngine.updateBalance(productId, parent, 0, balance.vQuoteBalance);
...
spotEngine.updateBalance(QUOTE_PRODUCT_ID, parent, quoteBalance);
``` [5](#0-4) 

Funds credited to `bytes32(0)` are unrecoverable — no user controls that subaccount.

**Health check does not protect:**
The base `isHealthy` implementation always returns `true`, so no health gate prevents the re-opened position from being accepted:

```solidity
function isHealthy(bytes32 /* subaccount */) internal view virtual returns (bool) {
    return true;
}
``` [6](#0-5) 

---

### Impact Explanation

A user who partially fills an isolated order, then closes the isolated subaccount's position through a separate mechanism (liquidation, reduce-only order, or direct settlement), ends up with a stale `digestToSubaccount` entry. Any subsequent sequencer-submitted match against the original order digest will:

1. Redirect the trade to the closed, parentless isolated subaccount.
2. Credit margin and PnL to that subaccount.
3. When the position is eventually closed, transfer all funds to `bytes32(0)`, permanently destroying them.

The corrupted state is: `digestToSubaccount[digest]` pointing to a subaccount whose `parentSubaccounts` entry is `bytes32(0)`, causing irreversible fund loss for the user whose margin was deposited into the re-opened position.

---

### Likelihood Explanation

The trigger requires:
1. An isolated order that is **not fully filled** before the isolated subaccount's position reaches zero by other means (liquidation is the most realistic path — a partially filled isolated position that goes underwater).
2. The original order must still be unexpired.
3. The sequencer must submit a match against the stale digest.

Liquidation of a partially-filled isolated subaccount is a realistic market event, making this a medium-likelihood issue with high asset-loss impact.

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, after clearing the parent/mask/slot state, also clear the digest mappings associated with the isolated subaccount. This requires either storing the digest on the isolated subaccount at creation time, or iterating over known digests. The simplest fix is to store the originating digest in a `subaccountToDigest` reverse mapping during `createIsolatedSubaccount` and delete both `digestToSubaccount[digest]` and `digestToMargin[digest]` inside `_tryCloseIsolatedSubaccount`:

```solidity
bytes32 digest = subaccountToDigest[subaccount];
if (digest != bytes32(0)) {
    delete digestToSubaccount[digest];
    delete digestToMargin[digest];
    delete subaccountToDigest[subaccount];
}
```

### Proof of Concept

1. User submits `CreateIsolatedSubaccount` with order D (100 units, not expired). `digestToSubaccount[D] = ISO1`, `parentSubaccounts[ISO1] = PARENT`.
2. Sequencer matches 50 units of order D. `filledAmounts[D] = 50`. ISO1 holds a 50-unit perp position.
3. ISO1 goes underwater; liquidator calls `liquidateSubaccountImpl` → ISO1's position is closed to 0 via `_handleLiquidationPayment`. [7](#0-6) 
4. `tryCloseIsolatedSubaccount(ISO1)` is called. `parentSubaccounts[ISO1]` is zeroed. `isolatedSubaccountsMask` bit is cleared. **`digestToSubaccount[D]` is NOT cleared.**
5. Order D still has 50 units remaining and has not expired. Sequencer submits `MatchOrders` with order D as taker.
6. `matchOrders` executes: `taker.order.sender = digestToSubaccount[D] = ISO1`. Signature validation passes (uses original sender's linked signer, resolved before the redirect). [8](#0-7) 
7. `_updateBalances` credits ISO1 with a new 50-unit position and associated margin. `isHealthy(ISO1)` returns `true`.
8. ISO1's position is later closed (amount = 0). `_tryCloseIsolatedSubaccount(ISO1)` runs again. `parent = parentSubaccounts[ISO1] = bytes32(0)`. All `vQuoteBalance` and `quoteBalance` are transferred to `bytes32(0)`. Margin is permanently lost. [9](#0-8)
