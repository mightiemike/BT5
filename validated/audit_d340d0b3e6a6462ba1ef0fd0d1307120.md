### Title
`ValidResourceBounds` Protobuf Deserialization Silently Collapses `AllResources` to `L1Gas` When l2_gas=0 and l1_data_gas=0, Producing a Divergent Transaction Hash Preimage in the P2P Sync Path - (File: crates/apollo_protobuf/src/converters/transaction.rs)

### Summary

The protobuf deserializer for `ValidResourceBounds` in the P2P sync path silently converts an `AllResources` variant (with zero l2_gas and l1_data_gas) into the `L1Gas` variant. Because `get_tip_resource_bounds_hash` hashes a different number of resource-bound felts depending on the variant, the transaction hash computed from the deserialized object diverges from the hash computed at the gateway. A user-submitted Declare/Invoke/DeployAccount V3 transaction with only l1_gas non-zero is accepted by the gateway under the `AllResources` variant and assigned hash H1. After protobuf round-trip in the sync path, the same transaction is reconstructed as `L1Gas`, and any hash recomputation yields H2 ≠ H1.

### Finding Description

**Step 1 – Gateway assigns hash under `AllResources`.**

`RpcDeclareTransactionV3` carries `resource_bounds: AllResourceBounds` (never `L1Gas`). During `convert_rpc_tx_to_internal`, `InternalRpcDeclareTransactionV3::resource_bounds()` always returns `ValidResourceBounds::AllResources(self.resource_bounds)`: [1](#0-0) 

The hash is then computed via `get_declare_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash` with the `AllResources` variant. For `AllResources`, the function appends a third felt for `L1_DATA_GAS`: [2](#0-1) 

So for `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the hash preimage is:
`Poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0), concat(L1_DATA_GAS, 0))` → **H1** (4 elements).

**Step 2 – Protobuf serialization is ambiguous.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes both `L1Gas` and `AllResources{l1, 0, 0}` identically: all three protobuf fields are present with l2_gas=0 and l1_data_gas=0: [3](#0-2) 

**Step 3 – Deserialization silently collapses to `L1Gas`.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` checks whether both l2_gas and l1_data_gas are zero. If so, it returns `L1Gas` regardless of whether the original was `AllResources`: [4](#0-3) 

The TODO comment acknowledges this is a known backward-compat workaround for 0.13.2 transactions, but it incorrectly collapses post-0.13.3 `AllResources` transactions that happen to have zero l2 and l1_data bounds: [5](#0-4) 

**Step 4 – Hash recomputation from deserialized object yields H2 ≠ H1.**

After deserialization, `DeclareTransactionV3.resource_bounds` is `L1Gas(X)`. `get_tip_resource_bounds_hash` for `L1Gas` hashes only two resource felts (no L1_DATA_GAS): [6](#0-5) 

Preimage becomes: `Poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0))` → **H2** (3 elements). H1 ≠ H2.

### Impact Explanation

Any node that receives a block via P2P sync containing a V3 transaction with zero l2_gas and l1_data_gas will reconstruct the transaction with the wrong `ValidResourceBounds` variant. If `validate_transaction_hash` is invoked on the deserialized transaction (as it is during block validation), the recomputed hash H2 will not match the transmitted hash H1, causing the block to be rejected. If hash validation is skipped, the stored transaction carries the wrong variant; any subsequent re-execution (e.g., for SNOS proving) recomputes H2, binding the wrong hash to the executable payload. This matches the allowed impact: **"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"** and **"Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution"** (via wrong re-execution hash).

### Likelihood Explanation

The trigger is a standard, gateway-accepted V3 transaction with only l1_gas non-zero. The gateway's stateless validator explicitly permits transactions with only one resource bound non-zero (e.g., `l1_gas: NON_EMPTY, l2_gas: 0, l1_data_gas: 0`). No special privilege is required. Any user can submit such a transaction. The conversion bug is on the unconditional deserialization path for every syncing node.

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, distinguish between an absent `l1_data_gas` field (backward compat for 0.13.2, → `L1Gas`) and a present-but-zero `l1_data_gas` field (post-0.13.3, → `AllResources`). The fix is to check `value.l1_data_gas.is_none()` rather than `l1_data_gas.is_zero()` as the discriminant:

```rust
Ok(if value.l1_data_gas.is_none() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The TODO comment at line 426 should be resolved by enforcing `l1_data_gas` is non-None for all post-0.13.3 transactions once 0.13.2 support is dropped.

### Proof of Concept

1. Submit a Declare V3 transaction to the gateway with `AllResourceBounds { l1_gas: {max_amount: 1, max_price_per_unit: 1}, l2_gas: {0,0}, l1_data_gas: {0,0} }`. The gateway accepts it and computes H1 = `get_declare_transaction_v3_hash` with `AllResources`.

2. Serialize the resulting `DeclareTransactionV3` to `protobuf::ResourceBounds` via `From<ValidResourceBounds>`. Observe that l2_gas=0 and l1_data_gas=0 in the protobuf message.

3. Deserialize via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Observe the result is `L1Gas(l1_gas)` — not `AllResources`.

4. Recompute the hash from the deserialized `DeclareTransactionV3`. Observe H2 ≠ H1 because `get_tip_resource_bounds_hash` hashes 3 felts for `AllResources` but only 2 for `L1Gas`: [7](#0-6) 

5. Call `validate_transaction_hash` with the deserialized transaction and H1. It returns `false`, demonstrating that a syncing node would reject a valid block containing this transaction.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
