### Title
Caller-Supplied `amount` Used Directly for Collateral Accounting Without Verifying Actual Received Tokens — (`File: core/contracts/EndpointStorage.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` records the caller-supplied `amount` into the slow-mode queue for later crediting to the subaccount, but `handleDepositTransfer` never verifies the actual token balance received. For fee-on-transfer collateral tokens, the Clearinghouse receives fewer tokens than `amount` but credits the full `amount` to the subaccount — inflating collateral balances and creating protocol insolvency.

---

### Finding Description

The deposit flow in Nado is:

1. `Endpoint.depositCollateral` / `depositCollateralWithReferral` is called by an unprivileged user with a caller-controlled `amount`.
2. `handleDepositTransfer` (EndpointStorage.sol:111–119) executes two transfers using the nominal `amount`:
   - `safeTransferFrom(token, from, amount)` — pulls from user to Endpoint
   - `safeTransferTo(token, address(clearinghouse), amount)` — forwards to Clearinghouse
3. A slow-mode transaction is enqueued (Endpoint.sol:152–165) encoding the original `amount` verbatim.
4. When the sequencer processes the slow-mode tx, `Clearinghouse.depositCollateral` (Clearinghouse.sol:193–208) calls `spotEngine.updateBalance(txn.productId, txn.sender, amountRealized)` where `amountRealized` is derived directly from `txn.amount` — the original caller-supplied value.

At no point is a balance-before / balance-after check performed. The protocol trusts the nominal `amount` end-to-end.

For a fee-on-transfer collateral token:
- `safeTransferTo(token, clearinghouse, amount)` causes the Clearinghouse to receive `amount − fee`, not `amount`.
- The slow-mode queue still records `amount`.
- `Clearinghouse.depositCollateral` credits the full `amount` (scaled) to the subaccount.

The Clearinghouse's real token balance is therefore systematically less than the sum of all credited subaccount balances, creating a solvency gap that grows with every deposit.

---

### Impact Explanation

- **Collateral inflation**: Every deposit of a fee-on-transfer token inflates the depositor's subaccount balance by the fee amount relative to actual holdings.
- **Protocol insolvency**: The Clearinghouse's real token balance diverges from the aggregate of credited balances. Later withdrawers cannot be fully paid out.
- **Theft of other users' funds**: A user who deposited a fee-on-transfer token can withdraw the full credited amount, consuming tokens that belong to other depositors.
- **Health-check bypass**: Inflated collateral improves the subaccount's health score, allowing the user to open larger positions than their real collateral supports.

---

### Likelihood Explanation

- Any ERC-20 token with a transfer fee (e.g., tokens with built-in tax mechanisms) listed as a supported collateral product triggers this path.
- The entry point (`depositCollateral`, `depositCollateralWithReferral`) is fully permissionless — any user can call it.
- No privileged access, governance capture, or sequencer compromise is required.
- The only precondition is that a fee-on-transfer token is listed as a supported product, which is a realistic deployment scenario.

---

### Recommendation

Replace the nominal-`amount` accounting with a balance-delta measurement in `handleDepositTransfer`:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal returns (uint256 actualReceived) {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    actualReceived = token.balanceOf(address(clearinghouse)) - before;
}
```

Then propagate `actualReceived` into the slow-mode `DepositCollateral` transaction instead of the raw `amount`. This mirrors the recommendation in the reference report: measure the real output via `balanceOf` rather than trusting the return value or input parameter.

### Proof of Concept

**Root cause chain:**

1. User calls `depositCollateral("default", feeTokenProductId, 1000e6)` on `Endpoint.sol`. [1](#0-0) 

2. `depositCollateralWithReferral` calls `handleDepositTransfer(token, msg.sender, 1000e6)` and then enqueues a slow-mode tx with `amount = 1000e6`. [2](#0-1) 

3. `handleDepositTransfer` performs both transfers using the nominal `1000e6` — no balance check. [3](#0-2) 

4. For a 1% fee-on-transfer token, `safeTransferTo(token, clearinghouse, 1000e6)` causes the Clearinghouse to receive `990e6`. The Endpoint must have ≥ `10e6` pre-existing balance (e.g., from accumulated slow-mode fees in the same token) for this transfer to succeed; otherwise it reverts. Either way, the slow-mode record holds `1000e6`.

5. When the sequencer executes the slow-mode tx, `Clearinghouse.depositCollateral` computes `amountRealized = int128(txn.amount) * int128(multiplier)` from the original `1000e6` and credits the full scaled amount to the subaccount. [4](#0-3) 

6. The subaccount now holds collateral worth `1000e6` (scaled) but the Clearinghouse only received `990e6`. The `10e6` gap is a permanent solvency deficit. Repeated deposits compound the deficit until the Clearinghouse cannot satisfy all withdrawals.

        virtual

        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        uint8 decimals = _decimals(txn.productId);
