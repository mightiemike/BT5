Audit Report

## Title
SwapAllowlistExtension Checks Router Address as `sender` Instead of Actual User, Enabling Allowlist Bypass or Broken Swap Flow — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates access using the `sender` parameter, which the pool sets to its own `msg.sender` — the `MetricOmmSimpleRouter` contract address when users route through it. This means the allowlist checks the router's address rather than the actual end-user's address, producing two concrete failure modes: (1) if the router is allowlisted, any non-allowlisted user bypasses the restriction entirely; (2) if the router is not allowlisted, legitimately allowlisted users are blocked from swapping through the router, breaking the core swap flow.

## Finding Description

**Root cause — pool passes `msg.sender` (the router) as `sender` to extensions:**

In `MetricOmmPool.swap` (line 230–231), the pool calls:
```solidity
_beforeSwap(msg.sender, recipient, ...)
```
When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `IMetricOmmPoolActions(params.pool).swap(...)` (lines 72–80), making the router the pool's `msg.sender`. The pool therefore passes the **router's address** as `sender` to `_beforeSwap`.

`ExtensionCalling._beforeSwap` (lines 160–176) forwards this value unchanged to the extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` (line 37) then evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is the router — not the originating user.

**Failure mode 1 — bypass:** Admin allowlists the router (the only way to permit router-based swaps). Any non-allowlisted user calls `MetricOmmSimpleRouter.swap()` → router is `sender` → `allowedSwapper[pool][router] == true` → allowlist passes → swap executes for an unauthorized user.

**Failure mode 2 — broken functionality:** Admin allowlists specific user addresses but not the router. Those users call through the router → `allowedSwapper[pool][router] == false` → `NotAllowedToSwap` revert → legitimate users cannot use the primary swap interface.

The `DepositAllowlistExtension` is **not** affected by the same issue: its `beforeAddLiquidity` ignores the `sender` parameter and checks `owner` (line 37 of `DepositAllowlistExtension.sol`), which is the actual LP address passed by the caller, not the intermediary.

## Impact Explanation

- **Admin-boundary break:** The pool admin's intended per-address swap restriction is circumvented by any unprivileged user routing through `MetricOmmSimpleRouter` when the router is allowlisted.
- **Broken core swap flow:** Allowlisted users are unable to use `MetricOmmSimpleRouter` — the primary user-facing swap interface — when the router itself is not allowlisted. This renders the allowlist extension incompatible with the router, breaking the core swap path for legitimate users.

Both impacts are direct and require no special privileges beyond calling a public function.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool deploying `SwapAllowlistExtension` and expecting users to interact via the router will be affected. The two failure modes are mutually exclusive but both reachable: either the router is allowlisted (enabling bypass) or it is not (blocking legitimate users). No malicious setup, privileged access, or non-standard token behavior is required.

## Recommendation

The `sender` identity forwarded to `beforeSwap` must reflect the originating user, not the intermediary router. Two approaches:

1. **Router pass-through:** `MetricOmmSimpleRouter` passes the actual caller (`msg.sender` of the router call) as an explicit identity field in `extensionData`, and `SwapAllowlistExtension` reads and verifies this field instead of the `sender` parameter.
2. **Pool-level originator tracking:** The pool's `swap()` function accepts an explicit `originator` parameter that the router populates with its own `msg.sender`, and this value is forwarded to extensions as the identity to check.

## Proof of Concept

1. Deploy pool with `SwapAllowlistExtension`. Admin sets `allowAllSwappers[pool] = false`.
2. Admin calls `setAllowedToSwap(pool, router, true)` to allowlist `MetricOmmSimpleRouter` (required for any router-based swap to succeed).
3. Attacker (address not in `allowedSwapper`) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
4. Router calls `pool.swap(...)` — pool sets `sender = address(router)` and passes it to `_beforeSwap`.
5. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router] == true` → returns selector.
6. Swap executes. Attacker bypasses the per-user allowlist with no special privileges.
