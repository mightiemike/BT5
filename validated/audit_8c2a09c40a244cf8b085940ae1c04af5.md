### Title
`ShardLayout::resolve_to_current_shard` single-generation limit causes chain-halting panic on `GlobalContractDistribution` receipt delayed across two static resharding events — (`File: core/primitives/src/shard_layout/mod.rs`)

---

### Summary

`ShardLayout::resolve_to_current_shard` for V0/V1/V2 layouts performs only a **single-generation** shard-ID lookup. A `GlobalContractDistributionReceipt` carries a `target_shard: ShardId` stamped at creation time. If that receipt is held in the delayed queue across two consecutive static (V2) resharding events, its `target_shard` is two generations old and `resolve_to_current_shard` returns `None`. `receiver_shard_id` then returns `Err(EpochError::ShardingError)`, which `receipt_filter_fn` immediately `.unwrap()`s, panicking the chunk-processing thread and halting the chain. The fix (full cumulative split history) exists only for V3 / dynamic-resharding layouts; V2 static-resharding layouts remain unpatched.

---

### Finding Description

**Root cause — single-generation resolution in V0/V1/V2:**

```rust
// core/primitives/src/shard_layout/mod.rs:230-237
pub fn resolve_to_current_shard(&self, shard_id: ShardId) -> Option<ShardId> {
    match self {
        Self::V0(_) | Self::V1(_) | Self::V2(_) => {
            self.get_children_shards_ids(shard_id).map(|c| c[0])  // single generation only
        }
        Self::V3(v3) => v3.resolve_to_current_shard(shard_id),    // full history
    }
}
```

For a V2 layout, `get_children_shards_ids` only stores the **most recent** split. After two resharding events:

- Layout L1: `{S}` (original shard)
- Layout L2 (V2): `{S1, S2}`, `split_map = {S → [S1, S2]}`
- Layout L3 (V2): `{S3, S4, S2}`, `split_map = {S1 → [S3, S4]}`

On L3, `get_children_shards_ids(S)` returns `None` (S is absent from L3's split map). So `resolve_to_current_shard(S)` returns `None`.

**Stale `target_shard` in `GlobalContractDistributionReceipt`:**

`initiate_distribution` stamps `target_shard = current_shard_id` at deploy time. [1](#0-0) 

The receipt's `target_shard` is never updated as shard layouts evolve. [2](#0-1) 

**`receiver_shard_id` returns `Err` when resolution fails:**

```rust

**`receipt_filter_fn` panics on `Err`:**

```rust

``` [4](#0-3) 

The `.unwrap()` on line 876 panics when `receiver_shard_id` returns `Err`. This function is called from `pop()` and `peek_iter()` inside `process_delayed_receipts`. [5](#0-4) 

**The fix exists only for V3:**

`ShardLayoutV3::resolve_to_current_shard` recursively walks the full cumulative `shards_split_map`, resolving arbitrarily old shard IDs. [6](#0-5) 

The test `test_stale_global_contract_distribution_after_double_resharding` explicitly documents the gap:

> "The fix only works with V3 shard layouts (dynamic resharding). With static resharding, the shard layout doesn't maintain a full split history."
> "If the vulnerability exists, processing the stale GlobalContractDistribution receipt will panic in receipt_filter_fn() when receiver_shard_id() fails to remap the old target_shard after two resharding generations." [7](#0-6) [8](#0-7) 

The test returns early when `DynamicResharding` is not enabled, meaning the V2 static-resharding path is **never exercised** by this test.

---

### Impact Explanation

Any node running with a V2 (static) shard layout that processes a `GlobalContractDistribution` receipt whose `target_shard` was split two or more epochs ago will **panic** inside `receipt_filter_fn`. Because this occurs during `apply_transactions` / `process_delayed_receipts`, the chunk application fails, the block cannot be finalized, and the chain halts. All validators on the affected shard hit the same panic deterministically, making recovery impossible without a hotfix.

---

### Likelihood Explanation

The trigger requires:
1. `GlobalContractDistribution` feature active (it is, as a production protocol feature).
2. A user deploys a global contract — unprivileged, costs only storage fees.
3. The resulting receipt is pushed to the delayed queue (achievable by saturating the shard's compute budget, as the test demonstrates).
4. Two consecutive static resharding events occur while the receipt remains delayed.

On mainnet, static resharding events have occurred at protocol versions 75, 76, and 78. If `GlobalContractDistribution` was active during any two consecutive such events and a receipt was delayed across both, the panic is deterministic. With dynamic resharding (V3) enabled, the code path is fixed; but V2 code remains in the binary and is reachable on any chain still using static layouts.

---

### Recommendation

**Short-term:** Guard `receipt_filter_fn` against `Err` from `receiver_shard_id` — return `true` (retain) or log-and-retain rather than `.unwrap()`, preventing the panic.

**Long-term:** Extend `resolve_to_current_shard` for V1/V2 layouts to perform multi-generation resolution by iterating through the epoch-config shard-layout history (analogous to `ShardLayoutV3::resolve_to_current_shard`), or migrate all active chains to V3 layouts before any further resharding events.

---

### Proof of Concept

The codebase contains a self-documenting regression test that exactly reproduces the scenario:

1. Start with a V3-layout chain (the test skips on V2 precisely because V2 is broken).
2. Deploy a global contract from `user0` (shard S_A).
3. Saturate S_A's compute every block so the `GlobalContractDistribution` receipt is pushed to the delayed queue.
4. Force-split S_A → two children (first resharding).
5. Force-split one child → two grandchildren (second resharding).
6. Stop saturating; let the delayed queue drain.
7. On V2, step 6 panics in `receipt_filter_fn` because `resolve_to_current_shard(S_A)` returns `None` on the grandchild layout. [9](#0-8) 

The divergent Borsh value is the `target_shard` field inside `GlobalContractDistributionReceiptV1`/`V2` — a `ShardId` that is two generations stale relative to the current V2 `ShardLayout`, causing `resolve_to_current_shard` to return `None` instead of a valid current `ShardId`.     pub(crate) fn pop(
        &mut self,
        trie_update: &mut TrieUpdate,
        config: &RuntimeConfig,
    ) -> Result<Option<ReceiptOrStateStoredReceipt<'_>>, RuntimeError> {
        // While processing receipts, we need to keep track of the gas and bytes
        // even for receipts that may be filtered out due to a resharding event
        loop {
            // Check proof size limit before each receipt is popped.
            if trie_update.trie.check_proof_size_limit_exceed() {
                break;
            }
            let Some(receipt) = self.queue.pop_front(trie_update)? else {
                break;
            };
            let delayed_gas = receipt_congestion_gas(&receipt, &config)?;
            let delayed_bytes = receipt_size(&receipt)? as u64;
            self.removed_delayed_gas =
                self.removed_delayed_gas.checked_add(delayed_gas).ok_or(IntegerOverflowError)?;
            self.removed_delayed_bytes = self
                .removed_delayed_bytes
                .checked_add(delayed_bytes)
                .ok_or(IntegerOverflowError)?;

            // Track gas and bytes for receipt above and return only receipt that belong to the shard.
            if self.receipt_filter_fn(&receipt) {
                return Ok(Some(receipt));
            }
        }
        Ok(None)
    }

    /// Resolve any historical shard ID to a current shard by walking the full
    /// split history in `shards_split_map`. Returns the shard itself if it is
    /// current, or follows the first child at each generation until a current
    /// shard is reached. Returns `None` only if the shard ID is absent from
    /// both the current layout and the split history.
    pub fn resolve_to_current_shard(&self, shard_id: ShardId) -> Option<ShardId> {
        if self.shard_ids.contains(&shard_id) {
            return Some(shard_id);
        }
        let children = self.shards_split_map.get(&shard_id)?;
        self.resolve_to_current_shard(children[0])
    }

        node.client().chain.chain_store().head().unwrap().height
    };

        head_height >= drain_end,
        "chain stalled at height {}; expected >= {} (likely panicked processing stale receipt)",
        head_height,
