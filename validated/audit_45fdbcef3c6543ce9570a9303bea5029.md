### Title
Odd-Leaf Merkle Duplicate Allows Phantom-Index Proof Acceptance — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`)

---

### Summary

`verify_transaction_inclusion_v2` returns `true` for a caller-supplied `tx_index` that does not exist in the real block. In any Bitcoin block with an odd transaction count, the last leaf `T_last` is duplicated by the protocol. Because `compute_root_from_merkle_proof` never validates that `transaction_position` is within the actual tree bounds, the same `merkle_proof` that proves `T_last` at its real index `N` also proves `T_last` at the phantom index `N+1`. The coinbase-proof guard added in v2 does not close this gap.

---

### Finding Description

**Root cause — no index-bounds check in `compute_root_from_merkle_proof`** [1](#0-0) 

The function accepts any `transaction_position` value and blindly follows the parity-based path. It never checks that the position is less than the actual number of leaves.

**How Bitcoin's odd-leaf duplication creates two valid paths for the same proof**

For a block with 3 transactions `[T0, T1, T2]`, the protocol builds:

```
Level 0 (padded): [T0,  T1,  T2,  T2]   ← T2 duplicated
Level 1:          [H(T0,T1), H(T2,T2)]
Root:             H(H(T0,T1), H(T2,T2))
```

The Merkle proof for `T2` at real index `2` is `[T2, H(T0,T1)]`.

Verification at **real index 2** (even):
```
hash(T2, T2)          → H(T2,T2),  pos → 1
hash(H(T0,T1), H(T2,T2)) → Root    ✓
```

Verification at **phantom index 3** (odd) with the **identical proof**:
```
hash(T2, T2)          → H(T2,T2),  pos → 1
hash(H(T0,T1), H(T2,T2)) → Root    ✓
```

Both calls return the same root and therefore both pass the equality check in `verify_transaction_inclusion`. [2](#0-1) 

**Why the v2 coinbase guard does not help**

`verify_transaction_inclusion_v2` only requires:
1. `merkle_proof.len() == coinbase_merkle_proof.len()` — trivially satisfied; both proofs have the same depth for any given block.
2. The coinbase proof reconstructs the correct root — the attacker supplies the real coinbase proof, which is public on-chain data. [3](#0-2) 

Neither check constrains `tx_index` to `[0, actual_tx_count)`.

---

### Impact Explanation

A downstream bridge, mint, or withdrawal contract that calls `verify_transaction_inclusion_v2` and uses `(tx_id, tx_index)` as its replay-prevention key can be tricked into processing the same economic event twice:

1. **First call** — attacker (or legitimate user) submits `(T_last, real_index=N, real_proof)` → accepted, event processed, key `(T_last, N)` marked used.
2. **Second call** — attacker submits `(T_last, phantom_index=N+1, same_proof)` → `verify_transaction_inclusion_v2` returns `true`; key `(T_last, N+1)` has never been seen, so the downstream contract processes the event again.

The light client itself is the authoritative oracle; once it returns `true`, downstream contracts have no independent way to detect the phantom index.

---

### Likelihood Explanation

- Odd transaction counts are the norm in Bitcoin (roughly half of all blocks).
- All inputs (`tx_id`, `tx_index`, `merkle_proof`, `coinbase_tx_id`, `coinbase_merkle_proof`) are caller-controlled with no privileged role required.
- The real proof is publicly derivable from any Bitcoin block explorer.
- No special chain state or timing is needed beyond the block being within the confirmation window.

---

### Recommendation

Add an explicit upper-bound check on `tx_index` before calling `compute_root_from_merkle_proof`. The proof depth `merkle_proof.len()` implies a maximum tree width of `2^depth`. Reject any `tx_index >= 2^(merkle_proof.len() - 1) * 2 - (tree_is_odd ? 1 : 0)`, or more simply, require that the caller also supplies the total transaction count and verify `tx_index < tx_count`. Alternatively, inside `compute_root_from_merkle_proof`, assert that `transaction_position < (1 << merkle_proof.len())` and that the position is consistent with the proof length.

---

### Proof of Concept

```rust
// 3-tx block: [coinbase, T1, T_last]
// Real proof for T_last at index 2: proof = [T_last, H(coinbase, T1)]
// Phantom call:
let args = ProofArgsV2 {
    tx_id:                 t_last_hash,
    tx_block_blockhash:    block_hash,
    tx_index:              3,          // phantom — does not exist
    merkle_proof:          vec![t_last_hash, h_coinbase_t1],
    coinbase_tx_id:        coinbase_hash,
    coinbase_merkle_proof: vec![t1_hash, h_t_last_t_last],  // real coinbase proof
    confirmations:         1,
};
// verify_transaction_inclusion_v2(args) → true
// Downstream bridge sees (t_last_hash, index=3) as a fresh, unused key → double-spend
```

`compute_root_from_merkle_proof` reconstructs the correct root for both index 2 and index 3 because the odd-leaf duplicate makes the two paths arithmetically identical, and no code in the call chain bounds-checks `tx_index` against the actual transaction count. [1](#0-0) [3](#0-2) [4](#0-3)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

    }
