### Title
Tier Bonus Rate Applied Retroactively to Historical Segments After Governance Update — (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` fetches the **current** tier at claim time and applies its bonus rate to **all** historical bonded segments, including those that accrued before any governance-driven tier parameter change. If governance updates a tier's bonus rate between position creation and reward claim, every historical segment is repriced at the new rate rather than the rate in effect when the bonus was earned.

---

### Finding Description

In `processEventsAndClaimBonus`, the tier is resolved once from live state at the moment of the claim:

```go
// x/tieredrewards/keeper/claim_rewards.go:167
tier, err := k.getTier(ctx, pos.TierId)
```

That single `tier` object is then passed unchanged into every historical segment computation:

```go
// x/tieredrewards/keeper/claim_rewards.go:178
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
```

and into the live trailing segment:

```go
// x/tieredrewards/keeper/claim_rewards.go:211
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
```

`SetTier` is explicitly documented as the write path for **governance messages, genesis, and chain upgrades**:

```go
// x/tieredrewards/keeper/tier.go:27
// SetTier writes a tier after validation. Used by governance messages, genesis, and chain upgrades.
func (k Keeper) SetTier(ctx context.Context, tier types.Tier) error {
```

Because `getTier` always returns the live tier, any governance proposal that modifies a tier's bonus multiplier takes effect retroactively on all unclaimed historical segments the next time a position owner calls `ClaimTierRewards`, `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, or `ExitTierWithDelegation` — all of which internally invoke `claimRewards` / `processEventsAndClaimBonus`.

The inconsistency is structurally identical to the reported bug: a rate is implicitly fixed at one point in time (position creation / accrual period) but re-read from mutable state at a later point (claim execution), with no snapshot bridging the two.

---

### Impact Explanation

| Governance change | Effect on claimant |
|---|---|
| Bonus rate **increased** | User receives more bonus than earned; excess is drained from `RewardsPoolName` module account |
| Bonus rate **decreased** | User receives less bonus than earned; the shortfall is silently lost |

The corrupted value is the `bonusCoins` amount sent from `types.RewardsPoolName` to the position owner:

```go
// x/tieredrewards/keeper/claim_rewards.go:239
if err := k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins); err != nil {
```

In the underpayment case, users permanently lose earned rewards with no recourse. In the overpayment case, the rewards pool is drained faster than intended, potentially starving later claimants.

**Impact: Medium** — direct token loss for position owners or the rewards pool, proportional to the magnitude of the rate change and the length of unclaimed history.

---

### Likelihood Explanation

Governance proposals to adjust tier parameters (e.g., to rebalance incentives or respond to market conditions) are a routine, expected operation. Any such proposal that touches a tier's bonus rate while positions with unclaimed rewards exist will silently corrupt those rewards at the next claim. No attacker capability is required beyond submitting or voting on a governance proposal — a normal, unprivileged on-chain action.

**Likelihood: Medium** — governance parameter changes are expected; the window between a change and the next claim can span many blocks.

---

### Recommendation

Snapshot the effective bonus rate at the time each bonded segment begins (e.g., store it alongside the validator event or in the position's checkpoint) and pass the snapshotted rate into `computeSegmentBonus` for historical segments. Only the live trailing segment (from `pos.LastBonusAccrual` to `blockTime`) should use the current tier rate. This mirrors the fix suggested in the external report: fix the relevant parameters at the earlier phase so both phases use the same value.

---

### Proof of Concept

1. Alice creates a position in tier `T` whose bonus rate is `R1`. Bonus accrues over 100 blocks.
2. A governance proposal passes that lowers tier `T`'s bonus rate to `R2 < R1` (via `SetTier`).
3. Alice calls `ClaimTierRewards` (or any operation that triggers `claimRewards`).
4. `processEventsAndClaimBonus` calls `getTier(ctx, pos.TierId)` and receives the updated tier with rate `R2`.
5. All 100 blocks of historical segments are computed with `R2` instead of `R1`.
6. Alice receives `bonus * (R2/R1)` of what she earned — a permanent, unrecoverable shortfall sent from `RewardsPoolName`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
