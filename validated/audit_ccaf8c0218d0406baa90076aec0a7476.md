### Title
Gateway Stateful Validator Uses Latest `allowed_proof_versions` While Batcher Uses Block-Version Constants, Causing Proof-Transaction Admission/Rejection Mismatch — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `run_validate_entry_point` always constructs its `BlockContext` from `VersionedConstants::get_versioned_constants()` — the **latest** protocol constants — while the batcher constructs its `BlockContext` from the versioned constants keyed to the **current block's Starknet version**. The `OsConstants::allowed_proof_versions` field differs across versions: V0_14_2 permits only `PROOF_VERSION_V0`; V0_14_3+ permits only `PROOF_VERSION_V1`. During a version-transition window, the gateway admits proof-bearing Invoke V3 transactions that the batcher's `validate_proof_facts` will reject in pre-validation, leaving those transactions permanently stuck in the mempool with no path to sequencing.

---

### Finding Description

**Step 1 — Gateway always uses latest VC.**

In `run_validate_entry_point`, the gateway builds its block context with:

```rust
let mut versioned_constants = VersionedConstants::get_versioned_constants(
    self.config.versioned_constants_overrides.clone(),
);
versioned_constants.disable_casm_hash_migration();
```

`get_versioned_constants` delegates to `Self::latest_constants()`, which resolves to the highest-numbered variant registered in the `define_versioned_constants!` macro — currently V0_14_4. [1](#0-0) [2](#0-1) 

**Step 2 — `allowed_proof_versions` differs between versions.**

V0_14_2 `os_constants.allowed_proof_versions` contains only `PROOF_VERSION_V0` (`0x50524f4f4630`): [3](#0-2) 

V0_14_3 and V0_14_4 `os_constants.allowed_proof_versions` contains only `PROOF_VERSION_V1` (`0x50524f4f4631`): [4](#0-3) 

**Step 3 — `validate_proof_facts` enforces the version-specific allowlist.**

The blockifier's pre-validation stage checks the proof version against `os_constants.allowed_proof_versions` drawn from whichever `BlockContext` was constructed:

```rust
if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt()) {
    return Err(TransactionPreValidationError::InvalidProofFacts(...));
}
``` [5](#0-4) 

**Step 4 — `verify_proof` permanently removes V0 verifiability.**

The gateway's proof verifier hard-rejects V0 proofs regardless of `allowed_proof_versions`:

```rust
ProofVersion::V0 => {
    return Err(VerifyProofError::InvalidProofVersion { actual: proof_version_felt });
}
``` [6](#0-5) 

**Step 5 — The divergence.**

During the V0_14_2 → V0_14_3 transition window (node software upgraded, network not yet):

| Transaction | Gateway (latest = V0_14_4) | Batcher (block = V0_14_2) |
|---|---|---|
| Invoke V3 with V1 proof facts | `verify_proof` ✓; `allowed_proof_versions` contains V1 → **ADMITTED** | `allowed_proof_versions` = `[V0]`; V1 absent → **PRE-VALIDATION REJECTED** |
| Invoke V3 with V0 proof facts | `verify_proof` hard-rejects V0 → **REJECTED** | `allowed_proof_versions` = `[V0]`; V0 present → **WOULD PASS** |

The first row is the critical path: the gateway returns success to the user, the transaction enters the mempool, but the batcher's `validate_proof_facts` rejects it every time it is polled. The transaction is never sequenced.

**Step 6 — Attacker-controlled trigger.**

Any unprivileged user can submit an Invoke V3 transaction with non-empty `proof_facts` containing `PROOF_VERSION_V1` during the V0_14_2 era. The gateway's `convert_rpc_tx_to_internal` extracts the proof facts and spawns `verify_proof` (which passes for V1), then `run_validate_entry_point` passes because the latest VC's `allowed_proof_versions` contains V1. The transaction is forwarded to the mempool. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

This matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**. The gateway returns a success response and a transaction hash to the submitter. The transaction occupies a mempool slot keyed to the sender's address and nonce. Because the batcher rejects it on every poll, the sender's nonce is effectively frozen: no subsequent transaction with the same or higher nonce can be sequenced until the stuck transaction is evicted or the network upgrades. A targeted attacker can freeze any account's nonce at zero cost beyond the gas required to pass gateway stateful validation (which succeeds).

---

### Likelihood Explanation

The window is bounded to the period between a node operator upgrading their software to a version that includes V0_14_3+ constants and the network-wide Starknet version upgrade. In practice this window can span hours to days. The trigger requires only a standard Invoke V3 transaction with a syntactically valid V1 proof facts payload; no privileged access, no malicious peer, and no invalid bytes are needed.

---

### Recommendation

The gateway's `run_validate_entry_point` should select versioned constants using the **current block's Starknet version** (the same version the batcher will use) rather than always using the latest. Concretely, replace:

```rust
let mut versioned_constants = VersionedConstants::get_versioned_constants(
    self.config.versioned_constants_overrides.clone(),
);
```

with a lookup keyed to `block_info.starknet_version` (already available from `get_block_info()`), applying overrides on top. This ensures `allowed_proof_versions`, `allowed_virtual_os_program_hashes`, and any other version-gated fields are evaluated against the same protocol version the batcher will enforce.

Alternatively, add an explicit cross-check in the gateway: after stateful validation succeeds, verify that the transaction's proof version is present in the versioned constants for the **current** block's Starknet version, not only the latest.

---

### Proof of Concept

```
Precondition:
  - Network Starknet version = V0_14_2
    (batcher VC: allowed_proof_versions = ["0x50524f4f4630"]  // V0 only)
  - Node software = latest
    (gateway VC: allowed_proof_versions = ["0x50524f4f4631"]  // V1 only)

1. Attacker constructs RpcInvokeTransactionV3 with:
     proof_facts = [PROOF_VERSION_V1, VIRTUAL_SNOS, <valid_program_hash>, ...]
     proof       = <valid V1 proof blob>

2. POST /gateway/add_transaction
   → convert_rpc_tx_to_internal: proof_data extracted, verify_proof(V1) → Ok
   → run_validate_entry_point (latest VC):
       validate_proof_facts: V1 ∈ allowed_proof_versions → Ok
   → mempool.add_tx → returns tx_hash to caller  ✓

3. Batcher polls mempool, attempts to include transaction:
   → perform_pre_validation_stage (V0_14_2 VC):
       validate_proof_facts: V1 ∉ allowed_proof_versions → Err(InvalidProofFacts)
   → transaction rejected, NOT included in block

4. Repeat step 3 indefinitely: transaction never sequenced.
   Sender's nonce is frozen.
```

    pub fn get_versioned_constants(
        versioned_constants_overrides: Option<VersionedConstantsOverrides>,
    ) -> Self {
        let latest_constants = Self::latest_constants().clone();
        match versioned_constants_overrides {
            None => latest_constants,
            Some(VersionedConstantsOverrides {
                validate_max_n_steps,
                max_recursion_depth,
                invoke_tx_max_n_steps,
                max_n_events,
            }) => {
                let tx_event_limits = EventLimits {
                    max_n_emitted_events: max_n_events,
                    ..latest_constants.tx_event_limits
                };
                Self {
                    validate_max_n_steps,
                    max_recursion_depth,
                    invoke_tx_max_n_steps,
                    tx_event_limits,
                    ..latest_constants
                }
            }
        }
    }
