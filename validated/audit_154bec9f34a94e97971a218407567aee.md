Audit Report

## Title
Router-Mediated Swaps Bypass `SwapAllowlistExtension` Allowlist Check — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool::swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`, so when `MetricOmmSimpleRouter` calls the pool, the extension sees the router address rather than the original user. `SwapAllowlistExtension::beforeSwap` checks `allowedSwapper[pool][sender]`, meaning any unprivileged user can bypass a restricted pool's allowlist by routing through the router if the router is allowlisted.

## Finding Description
**Step 1 — Pool passes `msg.sender` (the router) as `sender` to the hook.**

`MetricOmmPool::swap` calls:
```solidity
_beforeSwap(
  msg.sender,   // ← router address when called via router
  recipient, ...
);
``` [1](#0-0) 

**Step 2 — `ExtensionCalling::_beforeSwap` forwards that `sender` to every configured extension.**

The `sender` argument is ABI-encoded and dispatched to all extensions in order. [2](#0-1) 

**Step 3 — `SwapAllowlistExtension::beforeSwap` checks `allowedSwapper[pool][sender]`.**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
`msg.sender` here is the pool; `sender` is the router — not the original user. [3](#0-2) 

**Step 4 — The router never forwards the original caller's identity to the pool.**

`exactInputSingle` stores `msg.sender` only in transient callback context for payment, then calls `pool.swap` directly with no `sender` override:
```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ...);
``` [4](#0-3) 

The same pattern applies to `exactOutputSingle` and `exactInput`. [5](#0-4) 

## Impact Explanation
The core invariant of `SwapAllowlistExtension` — that only approved addresses may swap on a restricted pool — is broken. A pool admin has no safe configuration: allowlisting the router (required for router-mediated UX) opens the bypass to every unprivileged user; not allowlisting it makes the router unusable for all users on that pool. This constitutes a broken core pool functionality and an admin-boundary break reachable by any unprivileged caller.

## Likelihood Explanation
Any pool using `SwapAllowlistExtension` that expects users to interact via `MetricOmmSimpleRouter` is affected. No special attacker capability is required beyond calling the public router entry points. The vulnerability is structural and repeatable with zero preconditions beyond the router being allowlisted.

## Recommendation
The router must forward the original caller's identity so the extension can check it. Two approaches:

1. **Pass original caller via `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap`. The extension reads and verifies it, with a pool-level trust policy for the router address.
2. **Dedicated `swapper` parameter**: Add an explicit `swapper` field to the pool's `swap` interface that the router populates with `msg.sender`; the pool passes it as `sender` to hooks instead of its own `msg.sender`.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension configured.
2. Pool admin: setAllowedToSwap(pool, alice, true)   // only alice is allowed
3. Pool admin: setAllowedToSwap(pool, router, true)  // router allowlisted for UX
4. Bob (not allowlisted) calls:
     MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
5. Pool calls _beforeSwap(msg.sender=router, ...)
6. Extension checks allowedSwapper[pool][router] → true → passes
7. Bob's swap executes despite not being in the allowlist.

Assert: Bob received output tokens. The allowlist invariant is violated.
```
