Audit Report

## Title
`SwapAllowlistExtension` gates the router address instead of the actual swapper, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender` — the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router contract is that direct caller, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actual_user]`. Any pool admin who allowlists the router to permit legitimate users to reach the pool through the supported periphery simultaneously grants every unpermissioned user the same access, with no configuration that avoids this dilemma.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender`:**
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)` at lines 230–240. When the router calls `pool.swap(...)`, `msg.sender` inside the pool is the router address.

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged:**
`ExtensionCalling._beforeSwap` (lines 149–177) encodes `sender` directly into the `abi.encodeCall` payload passed to every configured extension. No transformation occurs.

**Step 3 — Extension checks the wrong actor:**
`SwapAllowlistExtension.beforeSwap` (line 37) evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender])
```
Here `msg.sender` is the pool and `sender` is the router. The actual originating user is the `recipient` parameter (second positional argument), which is silently discarded (`address,` unnamed) in the function signature at line 31.

**Step 4 — Router calls `pool.swap` directly:**
`MetricOmmSimpleRouter.exactInputSingle` (lines 72–80) and `exactInput` (lines 104–112) call `IMetricOmmPoolActions(pool).swap(params.recipient, ...)` with the router as `msg.sender`. The router is a public, permissionless contract — any address can invoke it.

**Exploit flow:**
1. Admin deploys pool with `SwapAllowlistExtension`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — allowlists alice.
3. Admin calls `setAllowedToSwap(pool, router, true)` — necessary for alice to use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. Router calls `pool.swap(bob_recipient, ...)` → pool calls `extension.beforeSwap(router, bob_recipient, ...)`.
6. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
7. Bob receives output tokens; the allowlist is fully bypassed.

**Existing guards are insufficient:** The only guard is `allowedSwapper[msg.sender][sender]` and `allowAllSwappers[msg.sender]`. Neither captures the originating user when an intermediary router is involved.

## Impact Explanation

A curated pool (KYC-only, institutional, or regulatory-restricted) deploying `SwapAllowlistExtension` to enforce per-user access control has that control completely nullified. Any unpermissioned user who routes through the public `MetricOmmSimpleRouter` receives pool output tokens and the pool receives their input tokens — a direct, fund-impacting policy violation on every such swap. This constitutes broken core pool functionality (the allowlist extension) causing a direct loss of policy enforcement with real token transfers to unauthorized parties, meeting the "Admin-boundary break" and "Broken core pool functionality causing loss of funds" impact criteria.

## Likelihood Explanation

The bypass requires only that the pool admin allowlists the router, which is the natural and expected operational step for any admin deploying a curated pool alongside the supported periphery. The router is a public, permissionless contract deployed at a fixed address. Once the router is allowlisted, the bypass is available to any address with no further preconditions, no special tokens, no privileged access, and no front-running required. It is repeatable on every swap.

## Recommendation

The extension must gate the economically relevant actor, not the intermediary. The `recipient` parameter is already passed to `beforeSwap` as the second positional argument but is currently unnamed and ignored. The minimal fix is to replace the `sender` check with `recipient`:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

The output-token recipient is always explicitly set by the originating user and cannot be spoofed by an intermediary router, making it the correct identity to gate.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin: setAllowedToSwap(pool, alice, true)   // allowlist alice
  admin: setAllowedToSwap(pool, router, true)  // allow alice to use the router

Attack:
  bob (NOT allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: bob, ...})
      → router calls pool.swap(bob, ...)           [MetricOmmPool.sol L72-80]
      → pool calls _beforeSwap(router, bob, ...)   [MetricOmmPool.sol L230-240]
      → extension checks allowedSwapper[pool][router] == true  ✓  [SwapAllowlistExtension.sol L37]
      → swap executes; bob receives output tokens

Result:
  bob, a non-allowlisted user, successfully swaps on a curated pool.
  The allowlist is completely bypassed for every user routing through the public router.
```

Foundry test outline:
1. Deploy pool with `SwapAllowlistExtension` in `beforeSwap` order.
2. `setAllowedToSwap(pool, alice, true)` and `setAllowedToSwap(pool, router, true)`.
3. As `bob` (not allowlisted), call `router.exactInputSingle(...)` targeting the pool.
4. Assert the swap succeeds and bob receives output tokens — confirming the bypass.
