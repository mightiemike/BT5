### Title
Receipt Size Validated Before `output_data_receivers` Are Appended, Allowing Receipts to Silently Exceed `max_receipt_size` - (File: `runtime/runtime/src/lib.rs`)

### Summary

The nearcore runtime validates a new receipt's Borsh-serialized size against `max_receipt_size` at creation time, but then unconditionally appends `output_data_receivers` from the parent receipt to that same receipt object after validation passes. Any contract that uses `promise_return` can craft a receipt that is exactly at the size limit, passes the check, and then grows beyond the limit when the runtime mutates it — producing an oversized receipt that is stored in chain state without triggering any error.

### Finding Description

In `runtime/runtime/src/verifier.rs`, `validate_receipt` in `NewReceipt` mode measures the Borsh-serialized size of the receipt and rejects it if it exceeds `limit_config.max_receipt_size`: [1](#0-0) 

This check is performed on the receipt as it exists at the moment of creation — before the runtime has had a chance to mutate it.

Later, in `runtime/runtime/src/lib.rs`, when a contract executes `promise_return` (returning a receipt index instead of a value), the runtime locates the returned receipt in `result.new_receipts` and **extends its `output_data_receivers` field** with the parent receipt's data receivers: [2](#0-1) 

This mutation happens entirely after `validate_receipt` has already approved the receipt. Each `DataReceiver` entry added to `output_data_receivers` contributes additional Borsh bytes (a `CryptoHash` + an `AccountId`), so a receipt crafted to be exactly at `max_receipt_size` will silently exceed the limit after the append. No re-validation occurs.

The nearcore codebase itself documents this as a known production bug: [3](#0-2) 

The `ValidateReceiptMode::ExistingReceipt` variant was introduced specifically as a workaround — it skips the size check so that the runtime can process these already-oversized receipts without crashing — but the root cause is unresolved: [4](#0-3) 

### Impact Explanation

An oversized receipt stored in chain state inflates the state witness for the shard that processes it. Because stateless validation requires validators to reproduce the same state witness, every chunk validator must download and re-execute the oversized receipt. This increases state witness size beyond the protocol's intended bound, potentially causing:

- State witness size violations that stress the network bandwidth and memory limits of chunk validators.
- Divergence between the `max_receipt_size` invariant that the protocol advertises and the actual receipts that exist on-chain, breaking the protocol's serialization domain guarantee.

The `ExistingReceipt` bypass means the runtime will not crash, but the invariant is permanently broken for any receipt that was created this way.

### Likelihood Explanation

Any unprivileged contract can trigger this by:
1. Deploying a contract that calls `promise_batch_then` to set up a callback (creating a parent receipt with one or more `output_data_receivers`).
2. Inside the first promise, calling `promise_return` on a newly created receipt whose Borsh-serialized size is exactly `max_receipt_size` (currently 4,194,304 bytes, achievable by padding `FunctionCall` arguments).
3. The runtime appends the parent's `output_data_receivers` to the returned receipt, pushing it over the limit.

The test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` demonstrates this path end-to-end and confirms the oversized receipt reaches chain state. [5](#0-4) 

### Recommendation

Move the receipt size validation to occur **after** `output_data_receivers` have been appended, not before. Concretely, in `runtime/runtime/src/lib.rs`, after the `extend_from_slice` call that appends `output_data_receivers` to the returned receipt, re-measure the receipt's Borsh size and return an action error (`NewReceiptValidationError::ReceiptSizeExceeded`) if it now exceeds `max_receipt_size`. This mirrors the fix described in the external report: validate the final value, not the pre-mutation value.

### Proof of Concept

The existing integration test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` (lines 129–208) already demonstrates the bug end-to-end:

1. It computes `args_size = max_receipt_size - base_receipt_size` to craft a receipt that is exactly at the limit.
2. It calls `max_receipt_size_promise_return_method1`, which uses `promise_return` to redirect the callback to the crafted receipt.
3. The test asserts via `assert_oversized_receipt_occurred` that an oversized receipt was indeed stored in chain state — confirming the check-before-mutation ordering flaw. [6](#0-5)

#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
