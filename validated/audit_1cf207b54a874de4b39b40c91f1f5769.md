Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end user, allowing any trader to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which the pool sets to `msg.sender` of the `swap` call. When `MetricOmmSimpleRouter` is used, `msg.sender` at the pool is the router contract, not the end user. A pool admin who allowlists the router to enable periphery-mediated swaps for legitimate users simultaneously opens the allowlist to every address on the network, because the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][endUser]`.

## Finding Description

**Root cause — `SwapAllowlistExtension.beforeSwap`:**

The extension checks `sender` (first argument) as the swapper identity: [1](#0-0) 

**The pool passes `msg.sender` of the `swap` call as `sender`:**

`ExtensionCalling._beforeSwap` encodes `sender` directly from the pool's `msg.sender`: [2](#0-1) 

**The router calls `pool.swap(...)` directly**, making itself `msg.sender` at the pool: [3](#0-2) 

The end user's address (`msg.sender` of the router call) is stored only in transient callback context for payment purposes and is **never forwarded** to the pool or extension. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][endUser]`.

**Contrast with `DepositAllowlistExtension`**, which correctly checks `owner` (second argument). The pool always passes the actual position owner as `owner` regardless of who calls `addLiquidity`, so the deposit allowlist correctly gates the economically relevant actor: [4](#0-3) 

The swap allowlist has no equivalent correct identity to check — the end user is simply absent from the hook arguments.

## Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC'd counterparties) faces an irreconcilable dilemma:

1. **Do not allowlist the router** → all router-mediated swaps revert for every user, including allowlisted ones. The standard periphery path is broken.
2. **Allowlist the router** → the allowlist is bypassed for every user on the network. Any address can call `MetricOmmSimpleRouter.exactInputSingle` and trade on the restricted pool.

In case 2, the pool receives tokens from and sends tokens to any counterparty, defeating the curation policy entirely. This is a direct loss of the access-control invariant the pool admin configured — a broken core pool functionality causing loss of the intended access restriction. This meets the "Admin-boundary break" and "Broken core pool functionality" allowed impact criteria.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is the documented, supported periphery path for swaps.
- Any user who discovers the pool is allowlist-restricted can trivially route through the router instead of calling the pool directly.
- No privileged access, no special tokens, no malicious setup required — a single `exactInputSingle` call suffices.
- The pool admin has no on-chain mechanism to fix this without disabling the router entirely (which breaks all legitimate router users).

## Recommendation

The `beforeSwap` hook must gate the end user, not the immediate pool caller. The preferred fix mirrors the deposit pattern: introduce a dedicated `swapper` parameter (analogous to `owner` in `addLiquidity`) that the pool always sets to the economically relevant actor, independent of who calls `swap`. The router would pass `msg.sender` explicitly as this parameter. Alternatively, the router can encode `msg.sender` into `extensionData` and the extension decodes and checks it, though this is trust-dependent on router cooperation.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (to allow router-mediated swaps for allowlisted users).
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  1. attacker (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool: restrictedPool,
           recipient: attacker,
           ...
       })
  2. Router calls pool.swap(attacker, ...) with msg.sender = router.
  3. Pool calls _beforeSwap(router, attacker, ...).
  4. Extension checks allowedSwapper[pool][router] == true → passes.
  5. Swap executes. attacker receives output tokens.

Result:
  attacker successfully swaps on a pool they were never allowlisted for.
  The allowlist check passed because it evaluated the router's address,
  not the attacker's address.
```
