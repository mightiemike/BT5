### Title
Checkpoint Advancement Without Bonus Payment in `slashRedelegationPosition` When Pool Is Drained — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` advances all three bonus checkpoints (`LastEventSeq`, `LastBonusAccrual`, `LastKnownBonded`) in the `pos` struct **before** it checks whether the rewards pool has sufficient balance. When the pool is empty, it returns `ErrInsufficientBonusPool` with the checkpoints already mutated in memory. `slashRedelegationPosition` swallows that specific error and then calls `setPositionWithState` with the checkpoint-advanced `pos`, permanently committing the advanced state to the store without ever transferring the bonus coins. Any subsequent `ClaimTierRewards` call starts from the already-advanced checkpoints and cannot recover the lost period.

---

### Finding Description

**Step 1 — Checkpoint mutation before pool check in `processEventsAndClaimBonus`** [1](#0-0) 

`pos.UpdateLastEventSeq(entry.Seq)` is called inside the event loop, advancing `LastEventSeq` in-place on the pointer-receiver `pos`. [2](#0-1) 

`applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` advance `LastBonusAccrual` and `LastKnownBonded` in-place — both happen **before** the pool balance check. [3](#0-2) 

Only after all three checkpoints are already mutated does `sufficientBonusPoolBalance` run. If the pool is empty it returns `nil, ErrInsufficientBonusPool` — but `pos` is already fully advanced.

**Step 2 — Error swallow in `slashRedelegationPosition`** [4](#0-3) 

`processEventsAndClaimBonus` is called with `&pos` (pointer). When it returns `ErrInsufficientBonusPool`, the branch logs the error and falls through — it does **not** restore `pos` to its pre-call state.

**Step 3 — Persisting the checkpoint-advanced position** [5](#0-4) 

For a partial slash, `setPositionWithState(ctx, pos, nil)` is called unconditionally with the already-advanced `pos`. For a full slash, `ClearBonusCheckpoints()` is called first (line 70), which zeroes the checkpoints — so the full-slash path does not suffer from this bug. The partial-slash path at line 77 persists the advanced checkpoints with no bonus paid.

**Step 4 — Permanent loss on subsequent claim**

When the pool is later refilled and the owner calls `MsgClaimTierRewards`, `processEventsAndClaimBonus` reads `pos.LastEventSeq` and `pos.LastBonusAccrual` from the store. Both already point past the drained period, so the function computes zero bonus for that interval. The owner has no recourse.

---

### Impact Explanation

Position owners permanently lose all bonus rewards accrued during the period covered by the checkpoint advancement. The coins are never transferred; the accounting window is closed. This is a direct, irrecoverable fund loss (not mere dilution) for any position that is subject to a redelegation slash while the pool balance is zero.

---

### Likelihood Explanation

The pool can reach zero through normal, legitimate reward claims by other position owners — no privileged access is required to deplete it. Redelegation slashes are uncommon but are a standard staking event triggered by double-sign evidence on the source validator of a redelegation. The two conditions (pool empty + redelegation slash) can coincide without any attacker involvement, making this a realistic operational risk rather than a purely theoretical one.

The ADR explicitly acknowledges this trade-off at section 6 and section 9: [6](#0-5) [7](#0-6) 

However, documenting a design decision does not eliminate the fund-loss impact on position owners.

---

### Recommendation

Move the pool balance check **before** any checkpoint mutation in `processEventsAndClaimBonus`, or — if the silent-forfeit behavior must be preserved for chain-halt avoidance — restore `pos` to its pre-call snapshot when `ErrInsufficientBonusPool` is caught in `slashRedelegationPosition`, so that `setPositionWithState` persists the original (un-advanced) checkpoints. The owner can then reclaim the bonus once the pool is refilled.

---

### Proof of Concept

1. Create a tier position and redelegate it (creating a `RedelegationMapping` entry).
2. Advance block time so bonus accrues on the destination validator.
3. Drain `RewardsPoolName` to zero by having other positions claim all available bonus.
4. Trigger `BeforeRedelegationSlashed` for the position's `unbondingId` with a partial `sharesToUnbond`.
5. Assert: `slashRedelegationPosition` returns no error (pool-empty branch is swallowed).
6. Read the position from the store: `LastBonusAccrual` equals the slash block time; `LastEventSeq` is advanced.
7. Refill the pool.
8. Call `MsgClaimTierRewards` for the position owner.
9. Assert: owner receives **zero** bonus for the period between the original `LastBonusAccrual` and the slash block time, confirming permanent loss.
