### Title
Socialization with Tiny `openInterest` Inflates Cumulative Funding, Bricking `_updateBalance` for Large-Position Holders — (`core/contracts/PerpEngine.sol`)

---

### Summary

In `PerpEngine.socializeSubaccount`, when `state.openInterest` is very small at the time of socialization, the computed `fundingPerShare` can be extremely large. This inflated value is permanently added to `cumulativeFundingLongX18` (or subtracted from `cumulativeFundingShortX18`). Any subsequent call to `_updateBalance` for a user with a sufficiently large position will then revert due to int128 overflow in `diffX18.mul(balance.amount)`, bricking trading, liquidation, and settlement for those users.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, when insurance is insufficient to cover a subaccount's negative `vQuoteBalance`, the remaining loss is spread across all open-interest holders via:

```solidity
int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
state.cumulativeFundingLongX18 += fundingPerShare;
state.cumulativeFundingShortX18 -= fundingPerShare;
``` [1](#0-0) 

The `div` function in `MathSD21x18` computes `(int256(x) * ONE_X18) / y`. If `openInterest = 1` (1 wei in X18 format) and `balance.vQuoteBalance = -5e17` (50 cents of loss), then:

```
fundingPerShare = 5e17 * 1e18 / 1 = 5e35
```

This value is added to `cumulativeFundingLongX18`. [2](#0-1) 

Subsequently, in `_updateBalance`, every call for a user with a long position computes:

```solidity
int128 diffX18 = cumulativeFundingAmountX18 - balance.lastCumulativeFundingX18;
int128 deltaQuote = vQuoteDelta - diffX18.mul(balance.amount);
``` [3](#0-2) 

The `mul` function in `MathSD21x18` checks: `require(result >= MIN_X18 && result <= MAX_X18, ERR_OVERFLOW)`. With `diffX18 = 5e35` and `balance.amount = 4e20` (400 units in X18):

```
(5e35 * 4e20) / 1e18 = 2e38 > int128.max (~1.7e38) → reverts with "OF"
``` [4](#0-3) 

Since `_updateBalance` is called from `getBalance`, `getStateAndBalance`, `updateBalance`, and `settlePnl`, all of these revert for any user whose `balance.amount` exceeds the threshold. [5](#0-4) 

---

### Impact Explanation

Once `cumulativeFundingLongX18` is inflated, every operation that calls `_updateBalance` for a large-position holder reverts permanently:

- **Trading** (`PerpEngine.updateBalance`) — users cannot open or close positions
- **Health checks** (`getBalance` → `getHealthContribution`) — health cannot be computed
- **Liquidation** — liquidators cannot process underwater accounts
- **PnL settlement** (`settlePnl`) — users cannot settle realized PnL

The protocol is effectively bricked for all large-position holders in the affected perp market. Funds are locked with no recovery path short of redeployment.

---

### Likelihood Explanation

**Low.** The attack requires:

1. A perp market where `openInterest` is tiny (e.g., 1 wei) at the moment socialization is triggered — possible in a new or low-liquidity market where one participant holds a 1-wei position.
2. A subaccount with `balance.amount = 0` (closed position) and a negative `vQuoteBalance` (unrealized loss from a prior trade) — achievable naturally when a user closes a losing position without settling PnL.
3. Protocol insurance insufficient to cover the loss — possible at protocol launch or after insurance depletion.

These conditions can arise naturally in a low-liquidity market or can be partially engineered by an attacker willing to absorb a small trading loss. The scenario is analogous to the Gamma staking report: low probability but catastrophic if triggered.

---

### Recommendation

1. **Add a minimum `openInterest` guard** before computing `fundingPerShare` in `socializeSubaccount` — if `openInterest` is below a safe threshold (e.g., `1e18`), skip socialization or cap `fundingPerShare`.
2. **Cap `fundingPerShare`** to a maximum safe value (e.g., `MAX_DAILY_FUNDING_RATE * priceX18`) to prevent extreme spikes.
3. **Seed initial open interest** when a new perp product is added (analogous to the Gamma staking recommendation to pre-stake a minimum amount before enabling rewards).

---

### Proof of Concept

1. A new perp market is deployed. Attacker A opens a 1-wei long position → `openInterest = 1`.
2. Attacker B opens a 1-unit long position at price $1, price drops to $0.50, B closes → `balance.amount = 0`, `vQuoteBalance = -5e17`.
3. Protocol insurance is 0 (or < 5e17).
4. Anyone triggers liquidation of B → `socializeSubaccount` is called.
5. `fundingPerShare = 5e17 * 1e18 / 1 = 5e35` is added to `cumulativeFundingLongX18`.
6. User C with a 400-unit long position (`balance.amount = 4e20`) attempts any operation:
   - `diffX18.mul(balance.amount)` = `(5e35 * 4e20) / 1e18 = 2e38 > int128.max`
   - Reverts with `"OF"` — C is permanently locked out of trading, liquidation, and settlement.
