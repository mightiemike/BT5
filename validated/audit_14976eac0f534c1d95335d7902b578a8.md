### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Causing Validator Rejection of Proposer-Accepted Transactions — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

When an `InvokeV3` transaction carrying `AllResourceBounds { l1_gas = X, l2_gas = 0, l1_data_gas = 0 }` is serialized to protobuf and deserialized on a peer node, the `ValidResourceBounds` variant is silently downgraded from `AllResources` to `L1Gas`. The subsequent conversion from `InvokeTransactionV3` back to `RpcInvokeTransactionV3` then hard-fails with `DEPRECATED_RESOURCE_BOUNDS_ERROR` because it requires `AllResources`. The proposer accepted and hashed the transaction under the `AllResources` branch (3-element resource hash), while every validator rejects the proposal outright when it tries to deserialize the same transaction from the P2P wire.

---

### Finding Description

**Step 1 — Gateway accepts the transaction.**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`. A user may submit `AllResourceBounds { l1_gas = X, l2_gas = 0, l1_data_gas = 0 }`. The stateless validator accepts this (tests confirm `valid_l1_gas` with zero l2/l1_data_gas passes). The gateway converts it to `InternalRpcInvokeTransactionV3` (which also stores `AllResourceBounds`) and computes the transaction hash via `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash`. [1](#0-0) 

Inside `get_tip_resource_bounds_hash`, the `AllResources` branch appends `L1_DATA_GAS` (even if zero) as a third element to the hash chain: [2](#0-1) 

So the stored `tx_hash` H₁ commits to **three** resource-bound felts.

**Step 2 — Proposer serializes the transaction to protobuf.**

When the proposer broadcasts the block proposal, `InternalConsensusTransaction` is converted back to `ConsensusTransaction` via `convert_internal_rpc_tx_to_rpc_tx`, which reconstructs `RpcInvokeTransactionV3 { resource_bounds: AllResourceBounds{...} }`. This is then serialized:

`RpcInvokeTransactionV3 → InvokeTransactionV3 (ValidResourceBounds::AllResources) → protobuf::InvokeV3` [3](#0-2) 

The protobuf wire bytes carry `l1_gas=X, l2_gas=0, l1_data_gas=0`.

**Step 3 — Validator deserializes and the variant is silently downgraded.**

On the receiving node, `protobuf::InvokeV3 → InvokeTransactionV3` calls: [4](#0-3) 

Because `l1_data_gas.is_zero() && l2_gas.is_zero()`, the result is `ValidResourceBounds::L1Gas(X)` — the `AllResources` variant is lost.

**Step 4 — Conversion to `RpcInvokeTransactionV3` hard-fails.**

`TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` then calls `snapi_invoke.try_into()`: [5](#0-4) 

`TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` rejects anything that is not `AllResources`: [6](#0-5) 

The conversion returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`.

**Step 5 — Validator rejects the entire proposal.**

In `validate_proposal.rs`, a conversion error causes the proposal to be marked `Failed`: [7](#0-6) 

The proposer's block is rejected by all validators.

---

### Impact Explanation

A single unprivileged user can submit a valid-looking `InvokeV3` transaction with `AllResourceBounds { l1_gas = X, l2_gas = 0, l1_data_gas = 0 }` to the gateway. The proposer accepts it, computes a hash under the `AllResources` branch, and includes it in a block proposal. Every validator node that receives the proposal via P2P will fail to deserialize the transaction and reject the proposal. This matches:

- **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing**: validators reject a transaction the gateway accepted.
- **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload**: the hash H₁ committed by the proposer (3-element resource hash) is irreconcilable with the `L1Gas` variant the validator would reconstruct (2-element resource hash), so the two nodes can never agree on the canonical transaction identity.

---

### Likelihood Explanation

The trigger requires only a standard RPC call with `l2_gas = 0` and `l1_data_gas = 0` inside `AllResourceBounds`. This is a normal, accepted input (the gateway's stateless validator explicitly allows it). No privileged access, no malformed bytes, and no special network position are required. Any user who wants to stall consensus can do so deterministically.

---

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not use the zero-value of `l2_gas` and `l1_data_gas` as a signal to select the `L1Gas` variant. Instead, use an explicit discriminant field in the protobuf schema (e.g., a `bounds_type` enum), or always deserialize into `AllResources` when all three fields are present in the wire message. The `L1Gas` variant should only be produced when the sender explicitly signals a pre-0.13.3 transaction type.

```rust
// Current (broken):
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})

// Fixed: always AllResources when l1_data_gas field is present in the message
Ok(if value.l1_data_gas.is_none() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [4](#0-3)

### Proof of Concept

1. Submit to the gateway an `InvokeV3` with `resource_bounds = { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: { max_amount: 0, max_price_per_unit: 0 }, l1_data_gas: { max_amount: 0, max_price_per_unit: 0 } }`.
2. The gateway accepts it; `get_tip_resource_bounds_hash` hashes three resource felts (L1_GAS, L2_GAS=0, L1_DATA_GAS=0).
3. The proposer includes the transaction in a block proposal and broadcasts it via P2P.
4. A validator receives the protobuf `InvokeV3WithProof`; `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` produces `L1Gas` because `l2_gas.is_zero() && l1_data_gas.is_zero()`.
5. `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`.
6. `validate_proposal.rs` returns `HandledProposalPart::Failed(...)` and the proposal is rejected.
7. The proposer cannot get its block finalized; consensus stalls for that round.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,

    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,

    // For new V3 txs, need to also hash the data gas bounds.

        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
}
