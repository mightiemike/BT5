### Title
Fork Retarget Uses Main-Chain Ancestor Timestamp Instead of Fork Ancestor Timestamp — (File: `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When a fork block falls on a difficulty-retarget boundary, `get_next_work_required` fetches the start-of-period block via `get_header_by_height`, which always reads from `mainchain_height_to_header`. If the fork diverged at or before that start height, the main-chain block at that height has a different timestamp than the fork's own ancestor. The contract therefore computes the expected difficulty from the wrong timestamp, accepting fork blocks whose `bits` field would be invalid under the fork's true history.

---

### Finding Description

In all three chain modules the retarget logic calls:

**`contract/src/bitcoin.rs` lines 78–86:**
```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
```

`get_header_by_height` is implemented as:

**`contract/src/lib.rs` lines 677–682:**
```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

This unconditionally returns the **main-chain** block at `first_block_height`. When the block being verified is on a fork that diverged at or before `first_block_height`, the fork's ancestor at that height is a different block with a different timestamp. The contract uses the main-chain timestamp (wrong) instead of the fork-ancestor timestamp (correct).

The Dogecoin module carries an explicit developer acknowledgement of this uncertainty:

**`contract/src/dogecoin.rs` lines 291–295:**
```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

The same structural flaw is present in `litecoin.rs` lines 88–93 and `bitcoin.rs` lines 81–86.

---

### Impact Explanation

`calculate_next_work_required` computes:

```
actual_time_taken = prev_block_time − first_block_time   (wrong first_block_time)
new_target        = target_from_bits(prev_bits) × actual_time_taken / target_timespan
```

If the main-chain block at `first_block_height` has an earlier timestamp than the fork's own block at that height, `actual_time_taken` is inflated, the computed target is easier, and the contract accepts a fork block whose `bits` encodes a lower difficulty than the fork's true history warrants. The accepted block's `chain_work` is then computed from those easier `bits` via `work_from_bits`, which is lower per block — but the fork can compensate by mining more blocks at the easier target. Conversely, a manipulated timestamp in the other direction causes the contract to reject a legitimately valid fork block, enabling censorship of competing chains.

The corrupted state is `mainchain_tip_blockhash` and the entire `mainchain_height_to_header` / `mainchain_header_to_height` mapping after a reorg, which downstream consumers of `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` rely on for SPV proof correctness.

---

### Likelihood Explanation

The attacker is an unprivileged `submit_blocks` caller. No privileged role is required. The attacker must:
1. Submit a fork that diverges at or before the start of a difficulty period.
2. Mine enough fork blocks to reach the retarget boundary (2016 blocks for Bitcoin/Litecoin, 1 block for Dogecoin post-block 145000).
3. Craft the fork's divergence-point block with a timestamp that shifts `actual_time_taken` in the desired direction.

For Dogecoin (retarget every block after height 145,000), the window is trivially small — every single block is a retarget — making this continuously exploitable with minimal mining power. For Bitcoin and Litecoin the attacker needs to sustain a fork across a 2016-block window, which requires significant but not nation-state-level hashrate.

---

### Recommendation

Replace the `get_header_by_height` lookup with an ancestor traversal that walks the fork's own `prev_block_hash` chain back to `first_block_height`. This mirrors Bitcoin Core's `pindexFirst` walk and ensures the timestamp used for retargeting belongs to the fork's actual history, not the main chain's.

---

### Proof of Concept

**Bitcoin scenario (second retarget, height 4032):**

1. Main chain contains blocks 0–4031. Block at height 2016 has timestamp `T_main`.
2. Attacker submits a fork block at height 2016 with timestamp `T_fork` where `T_fork > T_main` (attacker controls this timestamp within MTP bounds).
3. Attacker mines fork blocks 2017–4031 with valid PoW at the current difficulty.
4. Attacker submits fork block at height 4032 (retarget boundary):
   - `first_block_height = 4031 − 2015 = 2016`
   - `get_header_by_height(2016)` → main-chain block → timestamp `T_main` (**wrong**)
   - `actual_time_taken = T_fork_4031 − T_main` (inflated because `T_main < T_fork`)
   - Contract computes an easier `expected_bits` than the fork's true history warrants.
5. Attacker's fork block at height 4032 carries these easier `bits`; the contract's `check_pow` passes because `expected_bits == block_header.bits`.
6. The block hash satisfies the easier target; the block is accepted.
7. If the fork's accumulated `chain_work` exceeds the main chain's, `reorg_chain` is triggered, replacing `mainchain_tip_blockhash` and all height-to-hash mappings with the attacker's fork — corrupting every subsequent SPV proof result. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```rust
fn calculate_next_work_required(
    config: &NetworkConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let prev_block_time: i64 = prev_block_header.block_header.time.into();

    let mut actual_time_taken: i64 = prev_block_time - first_block_time;
    if actual_time_taken < config.pow_target_timespan / 4 {
        actual_time_taken = config.pow_target_timespan / 4;
    }
    if actual_time_taken > config.pow_target_timespan * 4 {
        actual_time_taken = config.pow_target_timespan * 4;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(actual_time_taken).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target = new_target
        / U256::from(<i64 as TryInto<u64>>::try_into(config.pow_target_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
```
