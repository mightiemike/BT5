Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which equals the pool's `msg.sender` — the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router address, not the originating user. Any pool admin who allowlists the router to enable router-based swaps for their users inadvertently grants swap access to all users, completely bypassing the allowlist invariant.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
  msg.sender,   // ← pool's msg.sender, i.e. the router
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to the extension hook. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The actual end user's address is stored only in transient callback context inside the router for payment purposes (`_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)`) and is never surfaced to the extension hook. The router passes `""` as `extensionData` to `pool.swap`, so no user identity is forwarded:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(
  params.recipient, params.zeroForOne,
  MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
  priceLimitX64,
  "",
  params.extensionData   // ← user-supplied, not router-injected
);
```

This creates two broken scenarios:
- **Scenario A:** Router not allowlisted → allowlisted users cannot use the router at all.
- **Scenario B:** Router allowlisted → any unprivileged user can call `router.exactInputSingle(...)` and the allowlist check passes because `allowedSwapper[pool][router] == true`, regardless of who the caller is.

No existing guard prevents Scenario B. The extension has no mechanism to distinguish which user initiated the router call.

## Impact Explanation
The `SwapAllowlistExtension` is the sole access-control mechanism for curated swap pools. In Scenario B, any unprivileged user bypasses the allowlist on a curated pool by routing through `MetricOmmSimpleRouter`, executing trades against restricted LP positions. This is a direct policy bypass with fund-impacting consequences: unauthorized counterparties trade against pools designed for controlled access, potentially draining value from LP positions. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact criteria.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entry point. Curated pools with `SwapAllowlistExtension` are a documented use case. Any pool admin who wants allowlisted users to use the router must add the router to the allowlist, directly and unavoidably triggering Scenario B. The attack requires no special privileges — any address can call the router. The condition is self-inflicted by correct admin behavior, making exploitation near-certain once a curated pool enables router access.

## Recommendation
The extension must identify the actual end user, not the intermediary router. The most robust fix is to have the router encode the caller in `extensionData` before forwarding to the pool:

```solidity
// In MetricOmmSimpleRouter.exactInputSingle (and other swap entry points):
bytes memory routerExtensionData = abi.encode(msg.sender, params.extensionData);
IMetricOmmPoolActions(params.pool).swap(..., routerExtensionData);
```

`SwapAllowlistExtension.beforeSwap` would then decode the caller from `extensionData` when `sender` is a known trusted router, and check `allowedSwapper[pool][decodedCaller]` instead. This is safe since `MetricOmmSimpleRouter` is a protocol-controlled contract. Alternatively, explicitly document that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` and enforce direct `pool.swap()` calls on allowlisted pools.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin: allowedSwapper[pool][userA] = true   (userA is the intended gated user)
  - Pool admin: allowedSwapper[pool][router] = true  (added so userA can use the router)

Attack (userB, not allowlisted):
  1. userB calls router.exactInputSingle({pool: pool, recipient: userB, ...})
  2. Router calls pool.swap(userB, ..., "", params.extensionData)
     — pool sees msg.sender = router
  3. Pool calls _beforeSwap(sender = router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  5. Swap executes successfully for userB

Result:
  userB, who is not in the allowlist, successfully trades on the curated pool.
  The allowlist is completely bypassed.

Foundry test outline:
  - deployPool(extensions=[swapAllowlist])
  - swapAllowlist.setAllowedToSwap(pool, userA, true)
  - swapAllowlist.setAllowedToSwap(pool, router, true)
  - vm.prank(userB); router.exactInputSingle(...)
  - Assert: swap succeeds (no NotAllowedToSwap revert)
```
