### Title
`cumulativeDepositsMultiplierX18` Accrues Unbacked Interest When No Borrows Exist — (`File: core/contracts/SpotEngineState.sol`)

### Summary

In `SpotEngineState._updateState()`, the `minDepositRate` section unconditionally applies `minDepositRateMultiplierX18` to `cumulativeDepositsMultiplierX18` regardless of whether any borrows exist. When `totalBorrowsNormalized == 0`, depositors earn compounding interest that is funded by nobody, inflating the deposit index beyond what the protocol's token reserves can cover and creating a structural insolvency.

### Finding Description

`_updateState()` correctly zeroes out the utilization-based borrow rate when there are no borrows: [1](#0-0) 

This means `borrowRateMultiplierX18 = ONE`, so the main-path update to `cumulativeBorrowsMultiplierX18` (line 129–131) is a no-op. However, the `minDepositRate` block that follows has no such guard: [2](#0-1) 

When `minDepositRateX18 != 0` and `totalBorrowsNormalized == 0`:

- `cumulativeDepositsMultiplierX18` is multiplied by `minDepositRateMultiplierX18` — depositors' balances grow in absolute terms.
- `cumulativeBorrowsMultiplierX18` is also multiplied, but since `totalBorrowsNormalized == 0`, the absolute borrow total remains zero — **no one pays for the depositor yield**.

The fee collection path confirms no revenue is generated in this state: [3](#0-2) 

When `utilizationRatioX18 == 0`, `totalDepositRateX18 == 0`, `feesAmt == 0`. The protocol collects nothing, yet `cumulativeDepositsMultiplierX18` still grows.

`updateStates()` only skips products with zero deposits, not products with zero borrows: [4](#0-3) 

So every call to `updateStates` during a zero-borrow period silently inflates the deposit index.

### Impact Explanation

Each `updateStates` call during a zero-borrow period increases `cumulativeDepositsMultiplierX18` without any corresponding token inflow. The absolute value of all depositor balances (`totalDepositsNormalized * cumulativeDepositsMultiplierX18`) grows beyond the actual tokens held by the clearinghouse. Early withdrawers receive more than they deposited; later withdrawers face a shortfall. The `assertUtilization` check (`totalDeposits >= totalBorrows`) does not catch this because `totalBorrows` is zero throughout. [5](#0-4) 

The corrupted state is permanent — once `cumulativeDepositsMultiplierX18` is inflated, it cannot be corrected without a manual state override.

### Likelihood Explanation

The `Config` struct exposes `minDepositRateX18` as a first-class parameter: [6](#0-5) 

Any product added via `addOrUpdateProduct` can carry a non-zero `minDepositRateX18`. A zero-borrow period is the normal state for any newly listed product before its first borrower, or after all borrows are repaid. No privileged attacker action is required — the sequencer's routine `updateStates` call is the trigger.

### Recommendation

Before applying `minDepositRateMultiplierX18` to `cumulativeDepositsMultiplierX18`, check that borrows actually exist:

```solidity
if (minDepositRateX18 != 0 && state.totalBorrowsNormalized != 0) {
    // apply minDepositRateMultiplierX18 to both multipliers
}
```

This mirrors the fix implied by the referenced report and matches the AAVE pattern of only accruing indexes when debt is present.

### Proof of Concept

1. Admin calls `addOrUpdateProduct` with `minDepositRateX18 = 5e16` (5% annual).
2. Alice deposits 1000 USDC. `totalDepositsNormalized = 1000`, `cumulativeDepositsMultiplierX18 = 1e18`.
3. No borrower exists. `totalBorrowsNormalized = 0`.
4. One year passes; sequencer calls `updateStates` periodically.
5. After one year: `cumulativeDepositsMultiplierX18 ≈ 1.05e18`. Alice's balance reads `1050 USDC`.
6. Alice withdraws 1050 USDC — the clearinghouse only holds 1000 USDC.
7. The 50 USDC shortfall is borne by subsequent depositors, or the withdrawal reverts, demonstrating insolvency. [7](#0-6)
