### Title
Unsigned `oraclePriceX18` in `SignedMintNlp`/`SignedBurnNlp` Allows Attacker to Manipulate NLP Token Amount via Slow-Mode Submission — (`core/contracts/Verifier.sol`, `core/contracts/EndpointTx.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The `oraclePriceX18` field in `SignedMintNlp` and `SignedBurnNlp` is **not included in the EIP-712 digest** that the user signs. Because `MintNlp` and `BurnNlp` transactions can be submitted by any unprivileged user via the slow-mode path (`submitSlowModeTransaction`), an attacker can craft a transaction with a valid user signature but an arbitrary `oraclePriceX18` value. Since this price directly determines how many NLP tokens are minted or how much quote is returned on burn, the attacker can extract value from the NLP pool.

---

### Finding Description

In `Verifier.sol`, `computeDigest` for `MintNlp` hashes only `sender`, `quoteAmount`, and `nonce`:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(MINT_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.quoteAmount,
        signedTx.tx.nonce
    )
);
``` [1](#0-0) 

The `oraclePriceX18` field — which lives in `SignedMintNlp` outside the inner `MintNlp` struct — is entirely absent from the digest. The same omission applies to `BurnNlp`: [2](#0-1) 

The struct definitions confirm the split: the user signs only the inner `MintNlp`/`BurnNlp` struct, while `oraclePriceX18` and `nlpPoolRebalanceX18` are outer fields that the submitter can freely set: [3](#0-2) 

In `EndpointTx.sol`, the unsigned `oraclePriceX18` is passed directly to the clearinghouse after signature validation passes: [4](#0-3) 

In `Clearinghouse.mintNlp`, the NLP token amount is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [5](#0-4) 

In `Clearinghouse.burnNlp`, the quote returned is:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
``` [6](#0-5) 

The slow-mode submission path allows any unprivileged user to queue a `MintNlp` or `BurnNlp` transaction with arbitrary bytes. `MintNlp` and `BurnNlp` are not in the admin-only gate: [7](#0-6) 

The stored transaction bytes — including the attacker-chosen `oraclePriceX18` — are replayed verbatim when the slow-mode transaction is later executed.

---

### Impact Explanation

**For `MintNlp`:** An attacker sets `oraclePriceX18` to an artificially low value. Because `nlpAmount = quoteAmount / oraclePriceX18`, a lower price yields more NLP tokens for the same quote deposit. The attacker then burns the excess NLP at the real (higher) oracle price, extracting quote from the NLP pool that belongs to other LPs.

**For `BurnNlp`:** An attacker sets `oraclePriceX18` to an artificially high value. Because `quoteAmount = nlpAmount × oraclePriceX18`, a higher price yields more quote for the same NLP burn. The attacker receives more quote than the NLP is actually worth, draining the pool.

In both cases the NLP pool's accounting is corrupted: the `N_ACCOUNT` NLP balance and the quote balances of NLP pool subaccounts are updated based on a fabricated price, permanently misrepresenting the pool's net asset value and harming honest LPs.

---

### Likelihood Explanation

The slow-mode path is a publicly callable, censorship-resistance escape hatch. Any registered subaccount can call `submitSlowModeTransaction` with a `MintNlp` or `BurnNlp` payload. The attacker only needs to:
1. Hold a valid subaccount with sufficient quote (for mint) or NLP (for burn).
2. Construct a `SignedMintNlp`/`SignedBurnNlp` with a valid inner-struct signature and a manipulated `oraclePriceX18`.
3. Wait out the slow-mode delay (3 days).

No privileged access, leaked keys, or governance capture is required. The 3-day delay is a practical friction but not a security barrier, especially for large positions where the profit exceeds the cost of capital.

---

### Recommendation

Include `oraclePriceX18` and `nlpPoolRebalanceX18` in the EIP-712 digest for both `MintNlp` and `BurnNlp` in `Verifier.computeDigest`. This forces the user to commit to a specific oracle price at signing time, preventing post-signing manipulation. Alternatively, the protocol should derive `oraclePriceX18` exclusively from an on-chain oracle at execution time rather than accepting it as a caller-supplied parameter.

---

### Proof of Concept

1. Attacker registers subaccount `A` and deposits 1000 USDC quote.
2. Attacker signs `MintNlp { sender: A, quoteAmount: 1000e18, nonce: N }` — this signature is valid.
3. Attacker wraps it in `SignedMintNlp { tx: ..., signature: ..., oraclePriceX18: 1e15, nlpPoolRebalanceX18: [...] }` — setting the oracle price to 0.001 (1000× below real price of 1.0).
4. Attacker calls `submitSlowModeTransaction(encodedMintNlp)`.
5. After 3 days, the transaction executes. `validateSignedTx` passes (digest excludes `oraclePriceX18`). `nlpAmount = 1000e18 / 1e15 = 1_000_000e18` NLP tokens are minted instead of the correct `1000e18`.
6. Attacker calls `submitSlowModeTransaction(encodedBurnNlp)` with the real oracle price (1.0), burning 1_000_000 NLP and receiving ~1_000_000 USDC, draining the pool. [8](#0-7) [9](#0-8) [10](#0-9)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
