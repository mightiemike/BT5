Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first `address` parameter (the actual `msg.sender` of `addLiquidity`) and instead validates `owner` (the position-recipient supplied by the caller). Because `owner` is caller-controlled, any address excluded from the allowlist can bypass the guard by passing any allowlisted address as `owner`. The pool admin's access-control boundary is broken: actors the admin explicitly excluded can add liquidity freely.

## Finding Description
`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, …)` at line 191, where `msg.sender` is the actual depositor and `owner` is the caller-supplied position recipient. [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both arguments to the extension: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` leaves the first `address` parameter unnamed and never reads it, then checks `allowedDepositor[msg.sender][owner]` (where `msg.sender` is the pool): [3](#0-2) 

Since `owner` is freely chosen by the caller, any non-allowlisted address `B` can pass an allowlisted address `A` as `owner`. The check evaluates `allowedDepositor[pool][A]` → `true` → no revert. `B` pays tokens via the modify-liquidity callback; `A` receives the LP shares. The allowlist is entirely ineffective.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender`: [4](#0-3) 

A secondary effect: an allowlisted `sender` who specifies a non-allowlisted `owner` (e.g., to create a position on behalf of a third party) is incorrectly rejected, breaking legitimate use.

## Impact Explanation
This is an admin-boundary break. The pool admin configures `DepositAllowlistExtension` to restrict which addresses may deposit into the pool. The bug renders that restriction completely ineffective: any unprivileged EOA or contract can add liquidity to a supposedly gated pool by supplying any allowlisted address as `owner`. The wrong value is the extension decision (`allowedDepositor` lookup key is `owner` instead of `sender`), which causes the hook to return `beforeAddLiquidity.selector` (allow) when it should revert.

## Likelihood Explanation
No special privilege is required. Any externally-owned account can call `pool.addLiquidity` directly with an allowlisted address as `owner`. Allowlisted addresses are discoverable on-chain via `AllowedToDepositSet` events or direct storage reads of `allowedDepositor`. The bypass is unconditional, repeatable, and requires no cooperation from the allowlisted address.

## Recommendation
Replace the unnamed first parameter with `sender` and check it instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate on both the depositor and the position owner, both should be checked explicitly.

## Proof of Concept
1. Deploy a pool with `DepositAllowlistExtension`; set `allowedDepositor[pool][Alice] = true`. Bob is not allowlisted.
2. Bob calls `pool.addLiquidity(owner = Alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `extension.beforeAddLiquidity(Bob /*sender*/, Alice /*owner*/, …)`.
4. Extension ignores `Bob`; checks `allowedDepositor[pool][Alice]` → `true` → no revert.
5. Bob pays tokens via the `IMetricOmmAddLiquidityCallback`; Alice receives the LP shares.
6. The deposit allowlist has been bypassed by an unprivileged actor with zero preconditions.
