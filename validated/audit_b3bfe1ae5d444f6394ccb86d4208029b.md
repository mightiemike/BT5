Audit Report

## Title
`DepositAllowlistExtension` checks position `owner` instead of actual token payer, allowing non-allowlisted addresses to deposit into curated pools — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` gates deposits by checking whether the position `owner` is allowlisted, but the actual token payer is a separate address stored in transient storage inside `MetricOmmPoolLiquidityAdder` and never surfaced to the extension. Any non-allowlisted address can bypass the allowlist by calling `addLiquidityExactShares` with an allowlisted address as `owner`, paying tokens themselves while the extension sees only the allowlisted `owner`.

## Finding Description
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (first `address`, unnamed) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
``` [1](#0-0) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` explicitly accepts an arbitrary `owner` while always using `msg.sender` as the payer, passed to `_addLiquidity` as a separate argument: [2](#0-1) 

The payer is stored in transient storage via `_setPayContext` and only used in the callback to pull tokens — it is never forwarded to the extension: [3](#0-2) 

`MetricOmmPool.addLiquidity` passes `msg.sender` (the adder contract) as `sender` to `_beforeAddLiquidity`, not the human caller: [4](#0-3) 

Even if the extension checked `sender`, it would see the adder's address, not the human payer. The complete call trace is:

1. Attacker (non-allowlisted) calls `addLiquidityExactShares(pool, allowlistedOwner, ...)`
2. Adder stores `payer = attacker` in transient storage, calls `pool.addLiquidity(owner=allowlistedOwner, ...)`
3. Pool calls `_beforeAddLiquidity(msg.sender=adder, owner=allowlistedOwner, ...)`
4. Extension checks `allowedDepositor[pool][allowlistedOwner]` → **true → no revert**
5. Pool mints position to `allowlistedOwner`; callback pulls tokens from attacker

The extension's NatSpec states it "Gates `addLiquidity` by depositor address" — but the depositor (payer) is never checked. [5](#0-4) 

## Impact Explanation
The deposit allowlist — the pool admin's primary curation mechanism — is rendered completely ineffective. Non-allowlisted addresses can inject tokens into curated pools, violating the `DEPOSITOR_IS_ALLOWLISTED` invariant. The position is minted to the named allowlisted `owner`, but tokens are pulled from the non-allowlisted payer. This constitutes broken core pool functionality (deposit gating) with direct fund-flow impact: non-allowlisted funds enter the curated pool, and the pool admin has no on-chain mechanism to prevent it.

## Likelihood Explanation
The attack requires only: (1) a pool configured with `DepositAllowlistExtension`, (2) knowledge of any one allowlisted address for that pool (publicly readable from `allowedDepositor`), and (3) token approval to the adder. No privileged access, flash loan, or oracle manipulation is needed. The `addLiquidityExactShares(pool, owner, ...)` overload is a standard public entrypoint explicitly designed to allow `owner != msg.sender`. [6](#0-5) 

## Recommendation
The extension must check the actual depositor, not the position owner. Since the payer is not propagated through the extension call chain, the fix requires one of:

1. **Enforce `owner == sender` in the extension** — require that the position owner and the direct caller of `pool.addLiquidity` are the same address, blocking router-mediated owner-splitting. This is the most robust option without protocol-wide changes.
2. **Propagate the real payer** — pass the actual payer through `extensionData` or a dedicated field and have the extension verify it (requires trust in the router to set it correctly).
3. **Require the router to enforce `owner == msg.sender`** when a deposit allowlist extension is active.

Option 1 is immediately actionable within the extension itself: add `require(sender == owner)` (using the currently discarded first parameter) to `beforeAddLiquidity`.

## Proof of Concept
```solidity
// Setup: pool configured with DepositAllowlistExtension
// allowedDepositor[pool][alice] = true
// bob is NOT allowlisted

vm.prank(bob);
adder.addLiquidityExactShares(
    pool,
    alice,       // allowlisted owner — extension checks this and passes
    salt,
    deltas,
    type(uint256).max,
    type(uint256).max,
    ""
);

// Result:
// - Extension saw owner=alice (allowlisted) → no revert
// - Position minted to alice
// - Tokens pulled from bob (non-allowlisted payer)
assertGt(pool.positionShares(alice, salt, bin), 0);   // alice got position
assertLt(token0.balanceOf(bob), bobBalanceBefore);     // bob paid
```
