Audit Report

## Title
Cross-Leg Timestamp Discrepancy in Synthetic Ratio Mode Allows Stale Price to Reach Pool Swaps - (File: smart-contracts-poc/contracts/AnchoredPriceProvider.sol)

## Summary
`AnchoredPriceProvider._getBidAndAskPrice()` computes a synthetic ratio from two independently-fetched oracle legs, each validated only against `MAX_REF_STALENESS` individually. The `refTime` values returned by `_readLeg` are silently discarded at both call sites, and no cross-leg timestamp alignment check exists. This allows a synthetic ratio computed from prices up to `MAX_REF_STALENESS` apart in time to reach live pool swaps, producing a materially inaccurate bid/ask.

## Finding Description
In `_getBidAndAskPrice()`, both calls to `_readLeg` discard the third return value (`refTime`) via a blank slot in tuple destructuring:

```solidity
(uint256 mid,  uint256 spreadBps,  , bool ok)  = _readLeg(baseFeedId);
(uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
``` [1](#0-0) 

`_readLeg` does return `refTime` and validates it per-leg against `MAX_REF_STALENESS`, but only individually:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
``` [2](#0-1) 

There is no comparison between `refTime1` and `refTime2`. The base feed can be fresh (`refTime = block.timestamp`) while the quote feed is at the staleness boundary (`refTime = block.timestamp - MAX_REF_STALENESS`), both passing individually. The ratio `mid1 / mid2` is then computed from prices up to `MAX_REF_STALENESS` apart. [3](#0-2) 

`_computeBidAsk` provides no remedy — it only checks `MAX_SPREAD_BPS` and band-width invariants, not timestamps. [4](#0-3) 

The resulting (bid, ask) pair is returned directly to `MetricOmmPool._getBidAndAskPriceX64()`, which only checks `bid >= ask` and `bid == 0` before using the price to execute swaps. [5](#0-4) 

## Impact Explanation
During volatile market conditions, a `MAX_REF_STALENESS`-second discrepancy between the two feeds produces a synthetic ratio materially off from the true market price. If the quote asset (e.g., ETH/USD) is stale and has moved 10% in the interim, the computed ratio is off by ~10%. Traders can exploit the inflated or deflated ratio to extract value directly from LP reserves — a direct loss of LP principal matching the "swap conservation failure / bad-price execution" impact class. Severity is High given the direct, quantifiable LP fund loss with no privileged access required.

## Likelihood Explanation
Reachable by any unprivileged trader whenever: (1) a pool uses `AnchoredPriceProvider` in synthetic ratio mode (`quoteFeedId != bytes32(0)`), (2) the two underlying feeds update at different rates or one is temporarily delayed (common during network congestion, oracle keeper delays, or RWA market-hours gaps), and (3) a public swap is executed during this window. No special role or setup is required beyond a standard pool configuration.

## Recommendation
Capture both `refTime` values and add a cross-leg alignment check before computing the ratio:

```solidity
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spreadBps, uint256 refTime1, bool ok) = _readLeg(baseFeedId);
    if (!ok) return (0, type(uint128).max);

    bytes32 _quote = quoteFeedId;
    if (_quote != bytes32(0)) {
        (uint256 mid2, uint256 spreadBps2, uint256 refTime2, bool ok2) = _readLeg(_quote);
        if (!ok2 || mid2 == 0) return (0, type(uint128).max);

        uint256 timeDelta = refTime1 > refTime2 ? refTime1 - refTime2 : refTime2 - refTime1;
        if (timeDelta > MAX_LEG_TIME_DELTA) return (0, type(uint128).max);

        mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
        spreadBps += spreadBps2;
    }
    return _computeBidAsk(mid, spreadBps);
}
```

`MAX_LEG_TIME_DELTA` should be set significantly smaller than `MAX_REF_STALENESS` — for example, the expected maximum update interval of the slower feed.

## Proof of Concept
1. Deploy a pool with `AnchoredPriceProvider` configured with `baseFeedId = BTC/USD`, `quoteFeedId = ETH/USD`, and `MAX_REF_STALENESS = 3600`.
2. At `T = block.timestamp`, update BTC/USD to `$100,000` with `refTime = T`.
3. At `T`, the ETH/USD feed has `refTime = T - 3600` (at the staleness boundary) with price `$2,000`. True ETH price at `T` is `$2,500`.
4. Call `getBidAndAskPrice()`. Both legs pass `_isStale` individually. `refTime` values are discarded at L259 and L264.
5. Computed synthetic ratio: `100,000 / 2,000 = 50 BTC/ETH`. True ratio: `100,000 / 2,500 = 40 BTC/ETH` — inflated by 25%.
6. The pool quotes bid/ask around 50 BTC/ETH. A trader sells BTC into the pool at this inflated price, receiving 25% more ETH than the true market rate, draining LP reserves by the difference.
