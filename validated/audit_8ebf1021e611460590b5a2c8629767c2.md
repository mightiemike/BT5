### Title
Unbounded `NlpLockedBalanceQueue` Growth Causes Permanent NLP Token Lock — (`core/contracts/SpotEngineState.sol`)

---

### Summary

A user who mints NLP tokens repeatedly at distinct oracle timestamps accumulates an unbounded number of entries in their `NlpLockedBalanceQueue`. Once those entries all mature past the lock period, the `tryUnlockNlpBalance` while-loop must iterate over every entry in a single call. With enough entries, this exhausts the block gas limit, permanently preventing the user from burning their NLP tokens and recovering the underlying collateral.

---

### Finding Description

`SpotEngine.handleNlpLockedBalance` appends a new `NlpLockedBalance` entry to the per-subaccount queue whenever `amountDelta > 0` **and** the current oracle timestamp differs from the last entry's `unlockedAt` value: [1](#0-0) 

Because oracle time advances with every sequencer heartbeat, each `mintNlp` call at a new oracle timestamp creates a distinct queue slot. There is no cap on `queue.balanceCount`.

`tryUnlockNlpBalance` processes all matured entries in a single unbounded `while` loop: [2](#0-1) 

Every iteration performs multiple cold-storage reads and writes (the queue is a `mapping(uint256 => NlpLockedBalance)` inside a `mapping(bytes32 => NlpLockedBalanceQueue)`). At roughly 20,000–30,000 gas per iteration, ~1,000–1,500 entries saturate the 30 M gas block limit.

`burnNlp` in `Clearinghouse.sol` unconditionally calls `spotEngine.getNlpUnlockedBalance`, which calls `tryUnlockNlpBalance`, before any burn logic executes: [3](#0-2) [4](#0-3) 

There is no partial-processing path. `tryUnlockNlpBalance` is `public` but still processes the entire unlocked window in one call; calling it directly also OOGs and reverts, making no forward progress. `handleNlpLockedBalance` (called on any subsequent `mintNlp`) also invokes `tryUnlockNlpBalance` first, so minting more NLP is equally blocked.

---

### Impact Explanation

Once the queue exceeds the gas-iteration threshold, the affected subaccount is permanently unable to:

1. **Burn NLP** — `burnNlp` → `getNlpUnlockedBalance` → `tryUnlockNlpBalance` → OOG → revert.
2. **Mint additional NLP** — `mintNlp` → `handleNlpLockedBalance` → `tryUnlockNlpBalance` → OOG → revert.
3. **Drain the queue incrementally** — no batch-processing entry point exists.

The user's NLP balance and the underlying collateral it represents are permanently inaccessible. This is a concrete asset-lock, not a transient denial of service.

---

### Likelihood Explanation

Any user or automated strategy that mints NLP at a rate of one transaction per oracle-time tick (e.g., a bot rebalancing liquidity every block) will accumulate one queue entry per mint. With a multi-day `NLP_LOCK_PERIOD`, hundreds to thousands of entries can accumulate before the first batch unlocks. The threshold for OOG is on the order of ~1,000 entries — reachable by any active liquidity provider over weeks of normal operation. No privileged access is required; only repeated `mintNlp` calls through the standard `Endpoint` entry point.

---

### Recommendation

Add a `maxIterations` parameter to `tryUnlockNlpBalance` so callers can drain the queue in bounded batches:

```solidity
function tryUnlockNlpBalance(bytes32 subaccount, uint256 maxIterations)
    public
    returns (Balance memory)
{
    NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[subaccount];
    uint256 processed = 0;
    while (
        processed < maxIterations &&
        queue.unlockedUpTo < queue.balanceCount &&
        queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
    ) {
        queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
        delete queue.balances[queue.unlockedUpTo];
        queue.unlockedUpTo++;
        processed++;
    }
    return queue.unlockedBalanceSum;
}
```

`burnNlp` should require the queue to be fully drained (i.e., no more matured entries remain) before proceeding, and expose a standalone `drainNlpQueue(bytes32 subaccount, uint256 maxIterations)` function so users can pre-drain in multiple transactions.

---

### Proof of Concept

1. Attacker (or active LP bot) calls `mintNlp` N times, each at a distinct oracle timestamp, accumulating N entries in `nlpLockedBalanceQueues[subaccount]` where N ≈ 1,200.
2. Time advances past `NLP_LOCK_PERIOD`; all N entries become eligible for processing.
3. User submits a `BurnNlp` slow-mode transaction.
4. Sequencer processes it → `Clearinghouse.burnNlp` → `spotEngine.getNlpUnlockedBalance(txn.sender)` → `tryUnlockNlpBalance` iterates ~1,200 times → block gas exhausted → transaction reverts.
5. Direct call to `tryUnlockNlpBalance(subaccount)` also OOGs and reverts — no forward progress is possible.
6. User's NLP balance is permanently locked; underlying collateral is unrecoverable. [5](#0-4) [6](#0-5) [7](#0-6)
