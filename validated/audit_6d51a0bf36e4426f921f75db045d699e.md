Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Enabling Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` always sets to `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router (required for any allowlisted user to trade via the router), every unpermissioned user on the network can bypass the curated allowlist by calling the public router, completely nullifying the pool's access control policy.

## Finding Description
**Root cause — pool passes `msg.sender` as `sender` to hooks:**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` verbatim to every configured extension via `abi.encodeCall`: [2](#0-1) 

**Extension check — gates on the wrong actor:**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Router path — pool sees router, not user:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the pool's `msg.sender` the router contract address: [4](#0-3) 

The same applies to `exactInput` (every hop), `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. No existing guard in the extension or the pool corrects this misbinding — `extensionData` is passed through unchanged and the extension never inspects it for a real-user identity.

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension` as extension1.
2. Admin calls `setAllowedToSwap(pool, router, true)` (required so allowlisted users can use the router).
3. Admin calls `setAllowedToSwap(pool, alice, true)` but does NOT allowlist Eve.
4. Eve calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` — pool sees `msg.sender = router`.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Eve successfully trades on a pool she was never authorized to access.

## Impact Explanation
**Scenario A (Critical — allowlist bypass):** Any unpermissioned user can trade on a KYC-gated or institution-only pool by routing through the public `MetricOmmSimpleRouter`. The curation policy is completely nullified; unauthorized users can drain LP value from a pool intended to be restricted. This is a direct loss of LP principal and a broken core access-control invariant.

**Scenario B (High — broken core functionality):** If the admin does not allowlist the router, allowlisted users cannot use the router at all — their swaps revert with `NotAllowedToSwap` even though they are individually permitted. The primary supported periphery entrypoint is unusable for any allowlisted pool.

## Likelihood Explanation
`SwapAllowlistExtension` is a production periphery contract explicitly designed for curated pools, and `MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Any pool combining both — the natural and expected configuration — is immediately affected. No special privileges, malicious setup, or non-standard tokens are required. Any unpermissioned user can trigger Scenario A by calling the public router. Likelihood is **High**.

## Recommendation
The extension must gate the original end user, not the intermediate router. Two viable fixes:

1. **Router-aware allowlist in the extension:** Maintain a mapping of trusted routers. When `sender` is a trusted router, require the real user identity to be ABI-encoded in `extensionData` and verify it against `allowedSwapper`.
2. **Gate at the router level:** The router maintains its own per-pool user allowlist and reverts before calling `pool.swap()` for non-allowlisted callers, keeping the extension simple but moving trust to the router.

Option 1 is preferable as it keeps enforcement in the extension without requiring a core interface change.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension as extension1.
  - Pool admin: setAllowedToSwap(pool, router, true)
  - Pool admin: setAllowedToSwap(pool, alice, true)
  - Pool admin does NOT call setAllowedToSwap(pool, eve, true)

Attack:
  1. Eve calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, ...) — pool sees msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. ExtensionCalling encodes and calls SwapAllowlistExtension.beforeSwap(sender=router, ...)
  5. Extension checks allowedSwapper[pool][router] → true
  6. Hook returns success selector; swap executes
  7. Eve successfully trades on a pool she was never authorized to access

Verification:
  - Direct call: eve calls pool.swap() → sender=eve → allowedSwapper[pool][eve]=false → revert NotAllowedToSwap ✓ (blocked)
  - Router call: eve calls router.exactInputSingle() → sender=router → allowedSwapper[pool][router]=true → swap succeeds ✗ (bypass)

Foundry test: deploy pool + extension, configure allowlist as above, impersonate Eve,
call exactInputSingle, assert swap succeeds and Eve received output tokens.
```
