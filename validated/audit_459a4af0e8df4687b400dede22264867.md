Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates on the direct pool caller (`sender = router`) instead of the end-user, making the allowlist unenforceable for router-mediated swaps — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is `msg.sender` of `MetricOmmPool.swap()`. When a user swaps through `MetricOmmSimpleRouter`, `sender` is the router address, not the actual user. This produces two fund-impacting outcomes: (1) allowlisted users on curated pools cannot use the router at all, breaking core swap flows; and (2) if the pool admin allowlists the router to restore router access, every unprivileged user can bypass the per-user allowlist entirely.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on that `sender` value: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(recipient, ...)` with `msg.sender = router`: [4](#0-3) 

So the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The actual end-user's identity is never checked.

Contrast with `DepositAllowlistExtension.beforeAddLiquidity`, which correctly gates on `owner` (the position owner, not the caller), because `addLiquidity` passes both `sender` and `owner` as distinct parameters and the extension ignores `sender`: [5](#0-4) 

The `swap()` function has no analogous separate "initiating user" parameter — only `recipient` (output receiver) and `sender` (direct caller). The extension uses `sender`, which collapses to the router address for all router-mediated swaps.

## Impact Explanation

**Path A — Broken core swap flow (no privileged action required):** Any allowlisted user on a curated pool who attempts to swap through the router will have their swap reverted with `NotAllowedToSwap` because the router is not in the allowlist. The router-mediated swap path (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) is completely unusable for every allowlisted user on every curated pool — a broken core pool functionality impact.

**Path B — Complete allowlist bypass (semi-trusted trigger):** A pool admin who wants to restore router access for their allowlisted users has no correct way to do so other than calling `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, `allowedSwapper[pool][router]` is `true` for every call that arrives through the router, regardless of who the actual user is. Any unprivileged user can bypass the per-user allowlist entirely by routing through `MetricOmmSimpleRouter`.

Both paths meet the allowed impact gate: Path A is "broken core pool functionality causing unusable swap flows"; Path B is "factory/oracle role checks are bypassed by an unprivileged path" (allowlist access control bypassed by unprivileged users).

## Likelihood Explanation

Path A requires only that a pool is configured with `SwapAllowlistExtension` and that an allowlisted user attempts to use the router — a normal, expected usage pattern. No attacker capability is required; it is triggered by ordinary users. Path B requires the pool admin to allowlist the router (a semi-trusted action taken in good faith to fix Path A), after which any unprivileged user can exploit it repeatably and permissionlessly. Both paths are highly feasible in production.

## Recommendation

The `swap()` interface does not carry a separate "initiating user" parameter analogous to `owner` in `addLiquidity`. The fix requires one of:

1. **Gate on `recipient` instead of `sender`:** Change `beforeSwap` to check `allowedSwapper[msg.sender][recipient]` (the second parameter, currently unnamed/ignored). This is semantically correct when the recipient is the end-user, which is the common case for `exactInputSingle` and `exactOutputSingle`. However, for multi-hop `exactInput`, intermediate recipients are the router itself, so this is only a partial fix.

2. **Pass the initiating user through `extensionData`:** Have the router encode `msg.sender` into `extensionData` and have the extension decode and verify it. This requires a trusted encoding convention.

3. **Redesign the swap hook signature** to include a separate `initiator` parameter (analogous to `owner` in the liquidity hooks), populated by the pool from a trusted source (e.g., a transient-storage context set by the router before calling `pool.swap`).

Option 1 is the minimal fix for the single-hop case; option 3 is the correct architectural fix.

## Proof of Concept

```solidity
// Setup:
// 1. Deploy pool with SwapAllowlistExtension configured.
// 2. Pool admin calls setAllowedToSwap(pool, alice, true).
//    alice is allowlisted; router is NOT allowlisted.

// Path A — alice cannot swap through router:
vm.prank(alice);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: alice,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    tokenIn: token0,
    deadline: block.timestamp,
    extensionData: ""
}));
// Reverts with NotAllowedToSwap — sender = router, not allowlisted.

// Path B — admin "fixes" by allowlisting router:
vm.prank(poolAdmin);
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Now bob (not allowlisted) can bypass the allowlist:
vm.prank(bob);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: bob,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    tokenIn: token0,
    deadline: block.timestamp,
    extensionData: ""
}));
// Succeeds — allowlist completely bypassed.
```
