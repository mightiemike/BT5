### Title
`submitFastWithdrawal()` Can Be Forced to Revert via Front-Running — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal()` is a public, permissionless function that reverts with `"Withdrawal already submitted"` if `markedIdxs[idx]` is already set. An attacker can copy a pending `submitFastWithdrawal` call from the mempool and front-run it with identical parameters, causing the legitimate provider's transaction to revert. This is structurally identical to the `safePermit()` front-running pattern: a check that hard-reverts on a "already consumed" state, rather than silently succeeding or returning.

---

### Finding Description

`submitFastWithdrawal` is callable by any address with no access control:

```solidity
function submitFastWithdrawal(
    uint64 idx,
    bytes calldata transaction,
    bytes[] calldata signatures
) public {
    require(!markedIdxs[idx], "Withdrawal already submitted");
    require(idx > minIdx, "idx too small");
    markedIdxs[idx] = true;
    ...
``` [1](#0-0) 

All parameters (`idx`, `transaction`, `signatures`) are fully visible in the mempool when a fast withdrawal provider submits the call. An attacker can replay the exact same calldata with a higher gas price. The attacker's transaction executes first, sets `markedIdxs[idx] = true`, and sends `transferAmount` to `sendTo`. When the legitimate provider's transaction executes, it hits the `require(!markedIdxs[idx], ...)` guard and reverts.

By contrast, the sequencer-side path `submitWithdrawal` handles the same "already marked" case with a silent early return:

```solidity
if (markedIdxs[idx]) {
    return;
}
``` [2](#0-1) 

This asymmetry is the root cause: `submitFastWithdrawal` hard-reverts on a state that `submitWithdrawal` treats as idempotent.

The fee branch further enables the attack. When `sendTo != msg.sender` (the normal case: provider calls on behalf of user), the attacker pays the fee from their own wallet:

```solidity
} else {
    safeTransferFrom(token, msg.sender, uint128(fee));
}
``` [3](#0-2) 

The attacker pays a small fee to grief the provider's call.

---

### Impact Explanation

The legitimate fast withdrawal provider's transaction reverts. Any smart contract or automated integration that calls `submitFastWithdrawal` and does not expect a revert (e.g., a batching contract, a keeper bot, or a composable DeFi integration) will have its execution path broken. The user's funds are still delivered (by the attacker's front-run), but the provider's call fails, which can corrupt state in calling contracts that assume success. This is the same class of impact as the reference report: "could cause issues for external integrations that do not expect the call to revert."

---

### Likelihood Explanation

Low-to-medium. The attack requires mempool visibility (standard on most EVM chains), the ability to copy calldata verbatim, and willingness to pay the fast withdrawal fee. The attacker gains nothing financially — they pay the fee and deliver the user's funds correctly. The motivation is pure griefing of a specific provider or integration. On chains with a public mempool (e.g., Ethereum L1 or any non-private-mempool L2), this is trivially executable.

---

### Recommendation

Replace the hard `require` with an idempotent early return, matching the behavior of `submitWithdrawal`:

```solidity
function submitFastWithdrawal(
    uint64 idx,
    bytes calldata transaction,
    bytes[] calldata signatures
) public {
-   require(!markedIdxs[idx], "Withdrawal already submitted");
+   if (markedIdxs[idx]) return;
    require(idx > minIdx, "idx too small");
    markedIdxs[idx] = true;
    ...
```

This ensures that if the withdrawal was already executed (by any caller), subsequent calls silently succeed rather than reverting, eliminating the griefing vector for integrations.

---

### Proof of Concept

1. Provider P constructs a valid `submitFastWithdrawal(idx, transaction, signatures)` call and broadcasts it.
2. Attacker A observes the pending transaction in the mempool and copies `idx`, `transaction`, and `signatures` verbatim.
3. A submits the identical call with a higher gas price.
4. A's transaction is mined first: `markedIdxs[idx]` is set to `true`, `transferAmount` is sent to `sendTo`, A pays `fee` via `safeTransferFrom`.
5. P's transaction is mined next: hits `require(!markedIdxs[idx], "Withdrawal already submitted")` → **reverts**.
6. Any smart contract that called P's function expecting success now has a broken execution path. [4](#0-3)
