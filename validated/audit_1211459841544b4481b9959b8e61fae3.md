Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` gates pool swaps by checking whether the `sender` argument passed to `beforeSwap` is allowlisted. Because `MetricOmmPool.swap` sets `sender = msg.sender` (the immediate caller), routing through the public `MetricOmmSimpleRouter` substitutes the router's address for the user's address. If the pool admin allowlists the router to enable router-mediated swaps for permitted users, every unpermitted user can trivially bypass the per-pool allowlist by calling the router instead of the pool directly.

## Finding Description

**Root cause — `sender` is the immediate caller, not the originating user.**

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged into the extension call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Exploit path via `MetricOmmSimpleRouter`.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool: [4](#0-3) 

The router stores no per-user identity that the pool or extension can inspect. There is no mechanism to propagate the originating EOA's address through the call chain.

**Concrete bypass scenario:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists a set of KYC'd addresses.
2. Admin also calls `setAllowedToSwap(pool, router, true)` so that allowlisted users can also swap via the router.
3. A non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. The pool receives `msg.sender = router`, which is allowlisted → `beforeSwap` passes → the swap executes.
5. The non-allowlisted user has successfully bypassed the access control.

**Existing guards are insufficient.** The only guard is `allowedSwapper[pool][sender]` and `allowAllSwappers[pool]`. Neither captures the originating user when an intermediary router is involved. The router performs no allowlist check of its own. [5](#0-4) 

## Impact Explanation
The `SwapAllowlistExtension` is the production mechanism for restricting swap access on a per-pool basis. Its bypass allows any unprivileged user to execute swaps on a pool the admin intended to restrict, constituting a broken core pool functionality and an admin-boundary break where an unprivileged path circumvents the pool admin's explicit access control. Pools configured for compliance gating (e.g., KYC-only trading) are rendered fully open to any caller of the public router.

## Likelihood Explanation
The attack requires only that the pool admin has allowlisted the router (a natural operational step to support router-mediated swaps for permitted users). Once that condition holds, any unpermitted user can exploit it with a single public call to `MetricOmmSimpleRouter`. No special privileges, flash loans, or complex setup are needed. The condition is realistic and the exploit is trivially repeatable.

## Recommendation
The extension must check the originating user, not the immediate caller. Two viable approaches:

1. **Pass the originating user explicitly:** Have the router store the originating `msg.sender` in transient storage before calling `pool.swap()`, and expose a `getSwapOriginator()` view. The pool (or extension) can then read this value when the immediate caller is a known router. This requires a protocol-level convention.
2. **Allowlist at the router level:** Add a user-level allowlist check inside the router before forwarding to the pool, so the router itself enforces the same restriction. This is simpler but requires the router to be aware of pool-level allowlists.
3. **Document and disallow router allowlisting:** If the design intent is that the router should never be allowlisted as a single entity, enforce this in `setAllowedToSwap` by reverting if `swapper` is a known router address.

## Proof of Concept

```solidity
// Foundry test sketch
function test_allowlistBypass_viaRouter() public {
    // Setup: pool with SwapAllowlistExtension, only `allowedUser` is permitted
    address allowedUser = makeAddr("allowed");
    address blockedUser = makeAddr("blocked");

    vm.prank(poolAdmin);
    swapAllowlist.setAllowedToSwap(pool, allowedUser, true);

    // Admin also allowlists the router so allowedUser can use it
    vm.prank(poolAdmin);
    swapAllowlist.setAllowedToSwap(pool, address(router), true);

    // blockedUser is NOT on the allowlist — direct swap should revert
    vm.prank(blockedUser);
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    pool.swap(blockedUser, true, 1e18, 0, "", "");

    // blockedUser routes through the public router — bypass succeeds
    vm.prank(blockedUser);
    // This should revert but does NOT — allowlist is bypassed
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: pool,
            recipient: blockedUser,
            zeroForOne: true,
            amountIn: 1e18,
            amountOutMinimum: 0,
            priceLimitX64: 0,
            deadline: block.timestamp + 1,
            extensionData: ""
        })
    );
}
```
