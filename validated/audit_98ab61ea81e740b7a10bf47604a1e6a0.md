Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the originating user, enabling allowlist bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is `msg.sender` of the pool's `swap()` call — the router contract — not the original user. When a pool admin allowlists the router to support router-mediated swaps for permitted users, every unprivileged user can bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`, because the extension sees the router address (which is allowlisted) rather than the actual trader.

## Finding Description

**Call path for a direct swap (correct):**
```
User → pool.swap()
  msg.sender = User
  pool calls _beforeSwap(msg.sender=User, ...)
  extension.beforeSwap(sender=User, ...)
  checks allowedSwapper[pool][User]  ← correct actor
```

**Call path for a router swap (broken):**
```
User → MetricOmmSimpleRouter.exactInputSingle(params)
  router calls pool.swap(recipient, ...)   [MetricOmmSimpleRouter.sol:72-80]
  msg.sender inside pool.swap() = Router
  pool calls _beforeSwap(msg.sender=Router, ...)  [MetricOmmPool.sol:230-240]
  ExtensionCalling._beforeSwap passes sender=Router  [ExtensionCalling.sol:160-176]
  extension.beforeSwap(sender=Router, ...)  [SwapAllowlistExtension.sol:31-41]
  checks allowedSwapper[pool][Router]  ← wrong actor
```

Root cause: `MetricOmmPool.swap()` passes `msg.sender` (the router) as the `sender` argument to `_beforeSwap`, and `SwapAllowlistExtension.beforeSwap` uses that argument as the identity to check against the allowlist. The extension has no way to recover the original user.

For the pool to be usable via the router at all, the pool admin must add the router to `allowedSwapper[pool][router]`. Once the router is allowlisted, `allowedSwapper[pool][router]` is `true` for every call that arrives through the router, regardless of who the originating user is. Any non-allowlisted user can then call `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` on the router and the extension check passes.

Existing guards are insufficient: `BaseMetricExtension.onlyPool` (line 19-24) only verifies that `msg.sender` is a registered pool, which is satisfied. The `allowedSwapper` mapping is keyed by `[pool][sender]` where `sender` is the router, so the check passes for all users once the router is allowlisted.

The `DepositAllowlistExtension` does not share this flaw: it ignores the `sender` argument (unnamed first parameter, line 32) and checks `owner` instead, which is the position owner passed explicitly by the caller and preserved correctly through the liquidity adder path.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties is fully bypassed for any user who routes through `MetricOmmSimpleRouter`. Non-allowlisted users can execute swaps against the pool, violating the curation policy. This constitutes a direct admin-boundary break and policy bypass on curated pools, matching the "High direct loss or policy bypass on curated pools" allowed impact. Severity: **High**.

## Likelihood Explanation
The bypass requires only that the pool admin has allowlisted the router (a natural operational step to let permitted users use the standard periphery). Any unprivileged user can then call the public router functions with no special privileges, no front-running, and no preconditions beyond having the input token. The attack is repeatable every block.

## Recommendation
Pass the original user identity through the call stack rather than `msg.sender`. One approach: add a `msgSender` field to the swap callback data or a transient storage slot (mirroring the pattern already used in `MetricOmmSwapRouterBase` for payer tracking) so the extension can read the true originator. Alternatively, `SwapAllowlistExtension.beforeSwap` should check the `recipient` or require the pool to expose the original caller via a dedicated transient slot set before the pool's `swap()` is entered. The fix must ensure the checked identity is the economic actor, not the intermediary contract.

## Proof of Concept

```solidity
// Setup
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
// Admin allowlists only Alice and the router (so Alice can use the router)
ext.setAllowedToSwap(pool, alice, true);
ext.setAllowedToSwap(pool, address(router), true);

// Attack: Bob (not allowlisted) swaps via router
vm.startPrank(bob);
token0.approve(address(router), type(uint256).max);
// Router calls pool.swap() with msg.sender=router
// Extension checks allowedSwapper[pool][router] == true → passes
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    recipient: bob,
    amountIn: 1000,
    amountOutMinimum: 0,
    zeroForOne: true,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// Bob successfully swaps despite not being on the allowlist
```
