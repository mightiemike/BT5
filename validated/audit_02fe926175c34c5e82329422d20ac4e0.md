### Title
Uninitialized Implementation Contracts Allow Attacker Takeover — (`Endpoint.sol`, `Clearinghouse.sol`, `OffchainExchange.sol`, `SpotEngine.sol`, `PerpEngine.sol`)

---

### Summary

Five core upgradeable contracts in the Nado protocol have public `initialize` functions but no constructor calling `_disableInitializers()`. This leaves their implementation contracts unprotected: any unprivileged caller can invoke `initialize` directly on the implementation, seizing ownership before the protocol team does.

---

### Finding Description

The following contracts are deployed behind proxies, inherit `OwnableUpgradeable` (directly or via `EndpointGated`), and expose a public `initialize` function — but define **no constructor** that calls `_disableInitializers()`:

| Contract | Inherits | Has `initialize` | Has `_disableInitializers` |
|---|---|---|---|
| `Endpoint.sol` | `OwnableUpgradeable`, `EIP712Upgradeable` | Yes | **No** |
| `Clearinghouse.sol` | `EndpointGated` → `OwnableUpgradeable` | Yes | **No** |
| `OffchainExchange.sol` | `EndpointGated`, `EIP712Upgradeable` | Yes | **No** |
| `SpotEngine.sol` | `SpotEngineState` → `BaseEngine` → `EndpointGated` → `OwnableUpgradeable` | Yes | **No** |
| `PerpEngine.sol` | `PerpEngineState` → `BaseEngine` → `EndpointGated` → `OwnableUpgradeable` | Yes | **No** |

By contrast, `BaseProxyManager`, `ContractOwner`, `Verifier`, `BaseWithdrawPool`, and `Airdrop` all correctly call `_disableInitializers()` in their constructors.

### Impact Explanation

An attacker who calls `initialize` on the bare implementation contract (not the proxy) becomes its `owner`. Consequences:

1. **Ownership of the implementation**: The attacker can call all `onlyOwner`-gated functions directly on the implementation address. For `Endpoint`, this includes sequencer configuration, slow-mode management, and NLP pool parameters. For `Clearinghouse`, this includes engine registration and insurance management.

2. **If UUPS upgrade logic is present or later introduced**: The attacker-owned implementation can be upgraded to a self-destructing contract, bricking the proxy for all users. Even under a transparent proxy model, the `ProxyManager` upgrade flow reads the current implementation address — a corrupted implementation state can interfere with upgrade hash verification. [8](#0-7) 

3. **`EndpointGated` ownership on engines**: `SpotEngine` and `PerpEngine` use `onlyOwner` to gate `addOrUpdateProduct`. An attacker owning the implementation can manipulate product risk parameters on the implementation contract directly. [9](#0-8) 

---

### Likelihood Explanation

The implementation contract addresses are publicly readable from the ERC-1967 implementation slot of each proxy. Any unprivileged attacker can query the slot, find the uninitialized implementation, and call `initialize` in a single transaction. No special access, leaked keys, or governance capture is required. This is a well-known, trivially executable attack.

---

### Recommendation

Add a constructor calling `_disableInitializers()` to each affected contract, following the pattern already used in `Verifier.sol`, `BaseProxyManager.sol`, `ContractOwner.sol`, `BaseWithdrawPool.sol`, and `Airdrop.sol`:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

Apply this to: `Endpoint.sol`, `Clearinghouse.sol`, `OffchainExchange.sol`, `SpotEngine.sol`, and `PerpEngine.sol`.

---

### Proof of Concept

```solidity
// Attacker script (no special privileges needed)
address endpointImpl = IErc1967(endpointProxy).implementation(); // read slot
Endpoint(endpointImpl).initialize(
    attackerAddress,   // _sanctions (attacker-controlled)
    attackerAddress,   // _sequencer
    attackerAddress,   // _offchainExchange
    IClearinghouse(attackerAddress), // _clearinghouse
    attackerAddress,   // _verifier
    attackerAddress    // _endpointTx
);
// Attacker is now owner of the Endpoint implementation contract
// All onlyOwner functions on the implementation are now attacker-controlled
```

The same pattern applies to `Clearinghouse.initialize`, `OffchainExchange.initialize`, `SpotEngine.initialize`, and `PerpEngine.initialize`. [10](#0-9) [11](#0-10)
