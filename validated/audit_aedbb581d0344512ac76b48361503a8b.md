### Title
`DepositAllowlistExtension` Gates Intermediary `sender` Instead of LP `owner`, Enabling Full Allowlist Bypass via `MetricOmmPoolLiquidityAdder` — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
The `DepositAllowlistExtension` is intended to restrict which addresses may add liquidity to a pool. When a user routes through the public `MetricOmmPoolLiquidityAdder`, the pool's `_beforeAddLiquidity` hook receives `sender = address(liquidityAdder)` and `owner = user`. If the extension gates on `sender` rather than `owner`, any non-allowlisted user can bypass the restriction by routing through the adder, because the adder's address—not the user's—is what the allowlist evaluates.

### Finding Description
`ExtensionCalling._beforeAddLiquidity` forwards both `sender` and `owner` to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol
_callExtensionsInOrder(
    BEFORE_ADD_LIQUIDITY_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

`sender` is `msg.sender` of `pool.addLiquidity()`. When the call originates from `MetricOmmPoolLiquidityAdder`, `sender = address(liquidityAdder)` and `owner = the actual user`. If `DepositAllowlistExtension.beforeAddLiquidity` evaluates `allowedDepositor[pool][sender]` instead of `allowedDepositor[pool][owner]`, the guard is applied to the wrong identity.

Attack path:
1. Pool admin configures `DepositAllowlistExtension` to gate deposits to a curated set of LPs.
2. Pool admin allowlists `MetricOmmPoolLiquidityAdder` (necessary for normal periphery operation).
3. Non-allowlisted user calls `MetricOmmPoolLiquidityAdder.addLiquidity(pool, owner = self, …)`.
4. Adder calls `pool.addLiquidity(owner = user, …)` — pool sees `msg.sender = liquidityAdder`.
5. Pool calls `_beforeAddLiquidity(sender = liquidityAdder, owner = user, …)`.
6. Extension evaluates `allowedDepositor[pool][liquidityAdder]` → `true` → deposit proceeds.
7. User's LP shares are minted under `owner = user` despite never being allowlisted.

The same structural flaw applies to `SwapAllowlistExtension`: `_beforeSwap` passes `sender = address(router)` to the hook, so the extension gates the router rather than the actual trader. Any user can swap through `MetricOmmSimpleRouter` if the router is allowlisted, regardless of whether the user is on the swap allowlist.

### Impact Explanation
- **Deposit allowlist**: Unauthorized users mint LP shares in access-controlled pools (KYC pools, institutional pools, capacity-limited pools). The pool admin's intended restriction is completely nullified. LP dilution and regulatory/compliance violations are direct consequences.
- **Swap allowlist**: Unauthorized traders execute swaps in restricted pools, affecting pool pricing, LP returns, and any compliance intent behind the allowlist.

Both impacts are direct loss of intended access-control invariants with fund-level consequences (unauthorized LP minting, unauthorized swap execution against pool reserves).

### Likelihood Explanation
High likelihood for the deposit path: `MetricOmmPoolLiquidityAdder` is the standard periphery contract for adding liquidity. Any pool admin who enables the allowlist extension and also wants normal periphery operation must allowlist the adder, which simultaneously opens the bypass to all users. The attacker needs no special privileges—only knowledge of the public adder address.

### Recommendation
`DepositAllowlistExtension.beforeAddLiquidity` must gate on `owner`, not `sender`:

```solidity
// Incorrect (current):
if (!allowAll[pool] && !allowedDepositor[pool][sender]) revert NotAllowedToDeposit();

// Correct:
if (!allowAll[pool] && !allowedDepositor[pool][owner]) revert NotAllowedToDeposit();
```

For `SwapAllowlistExtension`, since the actual initiating user is not forwarded through the router (only `sender = router` reaches the hook), the fix requires either (a) passing the real user address in `extensionData` and verifying it in the extension, or (b) having the router inject the original `msg.sender` into `extensionData` so the extension can decode and gate on it.

### Proof of Concept

```solidity
// 1. Pool admin sets up allowlist extension, allowlists only "alice"

// 2. Pool admin also allowlists the liquidity adder (required for normal operation)

// 3. Bob (not allowlisted) routes through the public liquidity adder
