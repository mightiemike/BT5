The discrepancy is real and concrete. Let me verify the exact boundary values and how `PathBuilder::serialized_length()` feeds into the final backref rejection check.

### Title
`PathBuilder::serialized_length()` Uses Wrong Boundary (0x200 vs 0x2000), Causing `find_path()` to Incorrectly Reject Valid Backrefs for Paths 512–8191 Bytes Long — (`src/serde/path_builder.rs`, `src/serde/tree_cache.rs`)

### Summary

`PathBuilder::serialized_length()` switches from `2+len` to `3+len` at `in_use = 0x200` (512), while `serialized_length_atom()` makes the same transition at `lb = 0x2000` (8192). For any path whose encoded form is 512–8191 bytes long, `PathBuilder::serialized_length()` overestimates by exactly 1. The final guard in `find_path()` uses this overestimate to reject a valid backref, causing `node_to_bytes_backrefs` to emit the full subtree inline. Two serializers — one using the correct boundary, one using the buggy one — produce different byte streams for the same CLVM tree.

### Finding Description

**Boundary mismatch — `PathBuilder::serialized_length()`:** [1](#0-0) 

```
0x40..=0x1ff  => 2 + len,   // up to 511 bytes
0x200..=0xfffff => 3 + len, // from 512 bytes  ← wrong threshold
```

**Correct boundary — `serialized_length_atom()`:** [2](#0-1) 

```
else if lb < 0x2000 { 2 + lb }    // up to 8191 bytes ← correct threshold
else if lb < 0x100000 { 3 + lb }
```

For `in_use` ∈ [512, 8191]:
- `PathBuilder::serialized_length()` → `3 + len` (off by +1)
- `serialized_length_atom(&path.done())` → `2 + len` (correct)

**Where the overestimate causes the rejection:**

After the BFS in `find_path()` finds the shortest path, the final guard is: [3](#0-2) 

```rust
let backref_len = ret.serialized_length();   // ← uses buggy method
if u64::from(backref_len) + 1 > entry.serialized_length {
    None   // ← valid backref incorrectly dropped here
} else {
    Some(ret.done())
}
```

**Concrete numeric proof:**

| Value | `PathBuilder::serialized_length()` | `serialized_length_atom()` |
|---|---|---|
| `in_use = 511` | `2 + 511 = 513` | `2 + 511 = 513` ✓ |
| `in_use = 512` | `3 + 512 = 515` | `2 + 512 = 514` ✗ (+1) |
| `in_use = 8191` | `3 + 8191 = 8194` | `2 + 8191 = 8193` ✗ (+1) |
| `in_use = 8192` | `3 + 8192 = 8195` | `3 + 8192 = 8195` ✓ |

**Scenario where a valid backref is dropped:**

- Node has `serialized_length = 515` (e.g., a 513-byte atom: `2 + 513 = 515`).
- Shortest path to this node encodes to exactly 512 bytes (`in_use = 512`).
- `PathBuilder::serialized_length()` returns `3 + 512 = 515`.
- Guard: `515 + 1 = 516 > 515` → returns `None`. Backref rejected.
- Actual path serialized length: `2 + 512 = 514`.
- Correct guard: `514 + 1 = 515 ≤ 515` → should return the path.

**Test gap:**

The existing `test_serialized_length` test iterates over `num_bits` values (bits, not bytes): [4](#0-3) 

`#[case(512)]` tests 512 *bits* = 64 bytes (`in_use = 64`), which is in the `0x40..=0x1ff` range where both functions agree. `#[case(0xfff9)]` = 65529 bits → `in_use = 8192`, which is at the boundary where both functions agree again. The entire range `in_use = 512..=8191` (requiring `num_bits ≈ 4089..=65528`) is untested.

### Impact Explanation

`node_to_bytes_backrefs` calls `find_path()` for every node. When `find_path()` incorrectly returns `None`, the serializer emits the full subtree inline instead of a `0xfe`-prefixed backref. The resulting byte stream is larger and differs from what a correct implementation would produce. `serialized_length_from_bytes` called on the two streams returns different values for the same logical CLVM tree, breaking API-equivalence between serializer implementations.

### Likelihood Explanation

For a path to require 512 bytes to encode, the tree must be approximately 512 × 8 − 1 = 4095 levels deep. CLVM programs in production are not typically this deep. The bug is real and locally provable, but the practical trigger condition is extreme. An attacker who can submit attacker-controlled CLVM data with deeply nested structure (e.g., via a puzzle reveal) can reach this path, but the depth required makes it unlikely in normal operation.

### Recommendation

Fix `PathBuilder::serialized_length()` to use the same boundaries as `serialized_length_atom()`:

```rust
// Replace:
0x40..=0x1ff => 2 + len,
0x200..=0xfffff => 3 + len,

// With:
0x40..=0x1fff => 2 + len,
0x2000..=0xfffff => 3 + len,
```

Add a `test_serialized_length` case for `num_bits` in the range 4096–65528 (e.g., `#[case(4096)]`, `#[case(32768)]`, `#[case(65528)]`) to cover `in_use` values 512–8191.

### Proof of Concept

```rust
// Property test: for in_use = 512..=8191, PathBuilder::serialized_length()
// must equal serialized_length_atom(&path.done())
use bumpalo::Bump;
use clvm_rs::serde::{ChildPos, PathBuilder, serialized_length_atom};

let a = Bump::new();
for num_bits in (512 * 8)..=(8191 * 8) {
    let mut path = PathBuilder::default();
    for _ in 0..num_bits {
        path.push(&a, ChildPos::Right);
    }
    let buggy = path.serialized_length();
    let vec = path.done();
    let correct = serialized_length_atom(&vec);
    assert_eq!(buggy, correct,
        "mismatch at in_use={}: PathBuilder says {}, atom says {}",
        vec.len(), buggy, correct);
}
// This assertion fails for in_use = 512..=8191
```
