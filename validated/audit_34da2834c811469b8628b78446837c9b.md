Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual trader — allowlist fully bypassable via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. The extension therefore checks whether the **router** is allowlisted, not the actual trader. If the router is added to the allowlist (the natural operational step to enable router-based swaps), every unprivileged address can bypass the per-pool swap allowlist entirely.

## Finding Description

**Call chain:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` (or `exactInput`, `exactOutputSingle`, `exactOutput`).
2. The router calls `IMetricOmmPoolActions(params.pool).swap(recipient, zeroForOne, …, extensionData)` — `msg.sender` to the pool is the **router address**.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, …)` — passing the router as `sender`.
4. `ExtensionCalling._beforeSwap` forwards that `sender` to `SwapAllowlistExtension.beforeSwap(sender, …)`.
5. Inside the extension the guard is:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is the router (wrong — should be the EOA/user).

**Root cause:** `MetricOmmPool.swap` passes its own `msg.sender` (the immediate caller) as the `sender` argument to the extension hook. There is no mechanism to propagate the original EOA through the router.

**Exploit path:**
- Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to KYC'd addresses.
- Pool admin calls `setAllowedToSwap(pool, router, true)` to allow router-based swaps (or the router is allowlisted as part of standard setup).
- Any unprivileged EOA calls `MetricOmmSimpleRouter.exactInputSingle(…)` targeting that pool.
- The extension sees `sender = router`, which is allowlisted → swap succeeds.
- The allowlist is completely defeated; every non-allowlisted address can swap.

**Existing guards reviewed:**
- `_requireExpectedCallbackCaller` in the router only validates the callback caller, not the swap initiator.
- The pool's reentrancy guard and `whenNotPaused` modifier do not address identity propagation.
- `allowAllSwappers` is a separate flag; the per-address mapping is the broken path.

**Exact wrong value:** `allowedSwapper[pool][router]` is evaluated instead of `allowedSwapper[pool][user_EOA]`.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC'd addresses, whitelisted market makers) can be freely accessed by any unprivileged address through the public router. This breaks the core allowlist invariant and constitutes an admin-boundary break: the pool admin's access control is bypassed by an unprivileged path. Depending on pool configuration, this can expose restricted liquidity to unauthorized traders, causing direct loss of LP assets or enabling unauthorized price impact.

## Likelihood Explanation
The router is the primary user-facing entry point for swaps. Any pool that uses `SwapAllowlistExtension` and also permits router-based swaps (the expected production configuration) is immediately vulnerable. No special attacker capability is required — a standard `exactInputSingle` call suffices. The condition is trivially repeatable.

## Recommendation
Propagate the originating user through the router to the pool, and have the pool pass it to the extension. Two concrete approaches:

1. **Pass the real sender via `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires the extension to trust the router as the pool-level `msg.sender` and verify the embedded address.
2. **Add a `swapFor(address onBehalfOf, …)` entry point on the pool:** The pool accepts an explicit `onBehalfOf` address (validated by the router's transient context) and passes it as `sender` to extensions.

Either way, `SwapAllowlistExtension.beforeSwap` must check the actual initiating user, not the intermediate contract.

## Proof of Concept

```solidity
// 1. Deploy pool with SwapAllowlistExtension; allowlist only address(0xALICE).
// 2. Pool admin calls: swapAllowlist.setAllowedToSwap(pool, router, true)
//    (to enable router usage — standard operational step)
// 3. Unprivileged address BOB calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: restrictedPool,
    recipient: BOB,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// 4. Pool calls _beforeSwap(router, …); extension checks allowedSwapper[pool][router] == true → passes.
// 5. BOB receives tokens despite not being on the allowlist.
```
