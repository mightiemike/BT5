Audit Report

## Title
`setPoolAdminFeeDestination` Missing Pre-Change Fee Flush Allows Pool Admin to Redirect All Accumulated Admin Fees — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`MetricOmmPoolFactory.setPoolAdminFeeDestination` overwrites `poolAdminFeeDestination[pool]` without first calling `collectFees` with the old destination. Every other fee-config mutator in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) explicitly flushes accumulated fees to the old destination before overwriting. The missing flush lets the pool admin silently redirect all previously accumulated admin fees — both notional and spread — to an arbitrary new address, causing direct loss to the original `adminFeeDestination`.

## Finding Description

**Flush-then-overwrite pattern in `setPoolAdminFees` (L417–425):**

`setPoolAdminFees` reads the old config, calls `collectFees` with the old `poolAdminFeeDestination[pool]`, then overwrites the config: [1](#0-0) 

`setPoolProtocolFee` (L327–335) follows the identical pattern: [2](#0-1) 

**Missing flush in `setPoolAdminFeeDestination` (L438–447):**

`setPoolAdminFeeDestination` directly overwrites `poolAdminFeeDestination[pool]` with no prior `collectFees` call: [3](#0-2) 

**`collectPoolFees` reads the current (post-overwrite) destination:** [4](#0-3) 

**`MetricOmmPool.collectFees` sweeps the entire accumulated balance to whichever address is current:**

All notional fees (`notionalFeeToken0Scaled`, `notionalFeeToken1Scaled`) and the full spread-fee surplus are transferred to `adminFeeDestination_` in a single sweep, then the accumulators are zeroed: [5](#0-4) 

There is no per-epoch or per-destination accounting; the entire balance is swept to whichever address is stored at collection time.

**Exploit path:**
1. Pool is live; fees accumulate for the original `adminFeeDestination` (e.g., a revenue-sharing contract or third-party LP treasury).
2. Pool admin calls `setPoolAdminFeeDestination(pool, attackerAddress)` — no flush occurs, `poolAdminFeeDestination[pool]` is overwritten.
3. Anyone calls `collectPoolFees(pool)` (permissionless).
4. `collectFees` is called with `attackerAddress`; all accumulated admin fees (token0 + token1, notional + spread) are transferred to `attackerAddress`.
5. The original `adminFeeDestination` receives nothing for the entire accumulation period.

## Impact Explanation

Direct loss of all accumulated admin fees (both token0 and token1, both notional and spread components) for the original `adminFeeDestination`. The pool admin can time the call to maximize the redirected amount, since `collectPoolFees` is permissionless and fees grow monotonically with swap volume. Both token balances are affected simultaneously. This constitutes a direct loss of owed fees above Sherlock thresholds for any pool with meaningful volume.

## Likelihood Explanation

The pool admin is a single EOA or multisig with no timelock on `setPoolAdminFeeDestination`. The function is callable at any time with no external preconditions. The only prerequisite — fees having accumulated since the last collection — is the normal operating state of any active pool. No oracle manipulation, flash loans, or external conditions are required.

## Recommendation

Apply the same flush-then-overwrite pattern used by `setPoolAdminFees` and `setPoolProtocolFee`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accumulated fees to the OLD destination before overwriting
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

## Proof of Concept

```
State:
  poolAdminFeeDestination[pool] = ORIGINAL_DEST
  notionalFeeToken0Scaled       = 1000e18
  notionalFeeToken1Scaled       = 500e18
  spread surplus (token0)       = 200e18

Step 1 — pool admin calls:
  factory.setPoolAdminFeeDestination(pool, ATTACKER)
  → poolAdminFeeDestination[pool] = ATTACKER
  → no collectFees called

Step 2 — anyone calls:
  factory.collectPoolFees(pool)
  → pool.collectFees(..., ATTACKER)
  → ATTACKER receives all admin share of 1000e18 token0 + 500e18 token1 notional fees
     plus admin share of 200e18 token0 spread surplus
  → ORIGINAL_DEST receives 0
  → notionalFeeToken0Scaled = 0, notionalFeeToken1Scaled = 0
```

A Foundry integration test can confirm this by: deploying the factory and pool, executing swaps to accumulate fees, calling `setPoolAdminFeeDestination` to a fresh address, calling `collectPoolFees`, and asserting that the original destination's token balance remains zero while the new address receives the full admin fee share.
