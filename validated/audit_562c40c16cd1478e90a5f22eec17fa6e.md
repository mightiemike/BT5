### Title
`getSlowModeFee()` Reverts for Quote Tokens with Fewer Than 6 Decimals Due to Hardcoded Decimal Floor — (File: `Clearinghouse.sol`)

---

### Summary
`Clearinghouse.getSlowModeFee()` hardcodes `6` as the minimum decimal assumption for the quote token. When the quote token has fewer than 6 decimals, the `uint8` subtraction `token.decimals() - 6` underflows, producing a massive exponent that causes the function to revert. This is a direct analog to the external report's bug class: a hardcoded decimal assumption that breaks protocol functionality for smaller-decimal tokens.

---

### Finding Description

In `Clearinghouse.sol`, the `getSlowModeFee()` function computes the slow-mode fee in the quote token's native decimals:

```solidity
function getSlowModeFee() external view returns (uint256) {
    ISpotEngine spotEngine = _spotEngine();
    IERC20Base token = IERC20Base(
        spotEngine.getConfig(QUOTE_PRODUCT_ID).token
    );
    int256 multiplier = int256(10**(token.decimals() - 6));
    return uint256(int256(SLOW_MODE_FEE) * multiplier);
}
``` [1](#0-0) 

The expression `token.decimals() - 6` is evaluated where `token.decimals()` returns `uint8`. In Solidity `^0.8.0`, arithmetic on `uint8` is checked, so if `token.decimals() < 6`, this subtraction reverts with an underflow panic. Even if it did not revert, the resulting exponent would be astronomically large (e.g., `10**252` for a 4-decimal token), making the function unusable.

The constant `SLOW_MODE_FEE = 1000000` is defined as "$1" in 6-decimal terms: [2](#0-1) 

The function scales it up by `10**(decimals - 6)` to convert to the token's native representation. This design only works for tokens with `decimals >= 6`. Tokens such as GUSD (2 decimals) or any other sub-6-decimal asset used as the quote token would permanently break this function.

The rest of the deposit path does handle variable decimals correctly — `depositCollateral` and `checkMinDeposit` both use `10**(MAX_DECIMALS - decimals)` scaling: [3](#0-2) [4](#0-3) 

This makes `getSlowModeFee()` the isolated location where the decimal floor assumption is hardcoded without a guard.

---

### Impact Explanation

**Medium.** `getSlowModeFee()` is an externally callable function used to determine the fee a user must pay to submit a slow-mode transaction — the protocol's censorship-resistance escape hatch. If this function reverts:

1. Users cannot query the required fee amount, making it impossible to correctly approve and submit a slow-mode transaction.
2. Any on-chain or off-chain component that calls `getSlowModeFee()` to gate or validate slow-mode submissions will also revert, effectively disabling the slow-mode path entirely for that quote token.

The slow-mode path is the only mechanism by which users can bypass a censoring sequencer. Disabling it removes a critical safety guarantee of the protocol.

---

### Likelihood Explanation

**Medium.** Tokens with fewer than 6 decimals exist and are in active use (e.g., GUSD at 2 decimals, some wrapped or synthetic assets). The protocol's architecture supports configurable quote tokens. If a sub-6-decimal token is ever configured as the quote token — either at deployment or via an upgrade — `getSlowModeFee()` will permanently revert, with no on-chain guard preventing this misconfiguration.

---

### Recommendation

Replace the hardcoded `6` with a safe, decimal-aware computation. Two options:

1. **Guard against underflow**: Add `require(decimals >= 6)` before the subtraction, and document that the protocol only supports quote tokens with `>= 6` decimals.

2. **Generalize the scaling**: Define `SLOW_MODE_FEE` in the protocol's internal 18-decimal representation (like `LIQUIDATION_FEE = 1e18`) and convert to native decimals using the same `10**(MAX_DECIMALS - decimals)` pattern used elsewhere:

```solidity
int256 multiplier = int256(10**(MAX_DECIMALS - token.decimals()));
return uint256(int256(SLOW_MODE_FEE_X18) / multiplier);
```

This is consistent with how `depositCollateral` and `checkMinDeposit` already handle decimal normalization.

---

### Proof of Concept

1. Deploy the protocol with a quote token that has 4 decimals (e.g., a custom ERC-20 with `decimals() = 4`).
2. Call `Clearinghouse.getSlowModeFee()`.
3. Execution reaches `int256(10**(token.decimals() - 6))` → `uint8(4) - 6` → Solidity 0.8 checked arithmetic panics with underflow.
4. The function reverts unconditionally.
5. Any user attempting to query the slow-mode fee — or any contract that calls `getSlowModeFee()` as part of the slow-mode submission flow — receives a revert, making the slow-mode escape hatch inaccessible. [1](#0-0) [5](#0-4)
