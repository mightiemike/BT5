Audit Report

## Title
SwapAllowlistExtension Checks Router Address as Swapper Identity, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router, not the end-user. If the router is allowlisted — the only way to permit allowlisted users to trade via the router — every address on-chain can bypass the guard entirely by calling through the router.

## Finding Description
**Exact call chain confirmed in production code:**

1. `MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — the pool's `msg.sender` is the router address. [1](#0-0) 

2. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)`, forwarding the router as `sender`. [2](#0-1) 

3. `ExtensionCalling._beforeSwap` ABI-encodes `sender` (the router) and dispatches it unchanged to the extension. [3](#0-2) 

4. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` — i.e., `allowedSwapper[pool][router]`. If the router is allowlisted, the check passes for every caller regardless of their individual allowlist status. [4](#0-3) 

**Root cause:** The extension checks the immediate pool caller (`sender` = router) rather than the economic actor (the user who initiated the transaction). There is no existing guard that recovers the original `msg.sender` from the router context.

The same substitution applies to all router entry points: `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165). [5](#0-4) 

**Contrast with `DepositAllowlistExtension`:** The deposit guard correctly checks `owner` (the economic beneficiary of the deposit) rather than `sender` (the immediate caller), making it robust to intermediary contracts. [6](#0-5) 

## Impact Explanation
A pool deploying `SwapAllowlistExtension` to restrict trading to a curated set of addresses (KYC-verified counterparties, internal market makers, whitelisted LPs) loses that protection entirely for any user routing through `MetricOmmSimpleRouter`. The router is a public, permissionless contract. Once the router is allowlisted, every address on-chain can execute swaps against the pool at live oracle prices, exposing LP principal to unauthorized directional trading and arbitrage. This constitutes broken core pool functionality causing direct loss of LP assets — a qualifying impact under the contest rules.

## Likelihood Explanation
The trigger requires the pool admin to allowlist the router address. This is not a misconfiguration edge case: it is the only way to permit allowlisted users to trade through the router. Any pool that (a) uses `SwapAllowlistExtension` and (b) wants to support router-mediated swaps must allowlist the router, which simultaneously opens the gate to all users. The router is a deployed, immutable, public contract requiring no special access to call. The condition is therefore a natural and expected operational state, not an unlikely edge case.

## Recommendation
The extension must check the economic actor, not the immediate pool caller. Two sound approaches:

1. **Check `recipient`** — the address receiving swap output is set by the originating user even through the router. The `beforeSwap` signature already receives `recipient` as the second parameter (currently ignored with `address`). Change the check to `allowedSwapper[msg.sender][recipient]`.

2. **Pass the real initiator in `extensionData`** — the router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.

Option 1 mirrors the pattern already used correctly in `DepositAllowlistExtension`, which checks `owner` (the economic beneficiary) rather than `sender`.

## Proof of Concept
```
Setup:
  Pool P configured with SwapAllowlistExtension E
  allowedSwapper[P][alice]  = true   // alice is the only intended swapper
  allowedSwapper[P][router] = true   // required so alice can use the router
  allowedSwapper[P][eve]    = false  // eve is explicitly excluded

Attack:
  eve calls MetricOmmSimpleRouter.exactInputSingle({
      pool:      P,
      recipient: eve,
      zeroForOne: true,
      amountIn:  X,
      ...
  })

  Router calls P.swap(eve, true, X, ...)
    pool.msg.sender = router
    _beforeSwap(router, eve, ...)
    SwapAllowlistExtension.beforeSwap(router, eve, ...)
      checks allowedSwapper[P][router] == true  ← passes
    swap executes at live oracle price
    eve receives output tokens

Result:
  eve, an explicitly excluded address, completes a swap in a restricted pool.
  The allowlist guard is fully bypassed.
  LP funds are exposed to unauthorized directional trading at oracle prices.
```

Foundry test plan: deploy pool with `SwapAllowlistExtension`, set `allowedSwapper[pool][router] = true` and `allowedSwapper[pool][eve] = false`, call `exactInputSingle` from `eve`'s address, assert the swap succeeds (no `NotAllowedToSwap` revert).
