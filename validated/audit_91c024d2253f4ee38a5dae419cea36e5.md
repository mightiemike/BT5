### Title
Undercharged Execution in `op_substr` Due to Hardcoded Cost of 1 — (`File: src/more_ops.rs`)

---

### Summary

`op_substr` (opcode 12) in `src/more_ops.rs` charges a fixed cost of `1` for every invocation, regardless of the size of the substring extracted. The `max_cost` parameter is explicitly ignored (prefixed `_`). No `SUBSTR_BASE_COST` or `SUBSTR_COST_PER_BYTE` constants exist anywhere in the codebase. An attacker can craft CLVM programs that call `substr` on arbitrarily large atoms, performing O(n) allocator work at O(1) declared cost, bypassing the cost limit that is the primary DoS defense of the CLVM execution engine.

---

### Finding Description

Every other size-sensitive operator in `more_ops.rs` charges a cost proportional to the byte length of its inputs and outputs. For example, `op_strlen` charges `STRLEN_BASE_COST + size * STRLEN_COST_PER_BYTE`, `op_concat` charges `CONCAT_BASE_COST + args * CONCAT_COST_PER_ARG + total_len * CONCAT_COST_PER_BYTE`, and `op_sha256` charges `SHA256_BASE_COST + args * SHA256_COST_PER_ARG + total_len * SHA256_COST_PER_BYTE`.

`op_substr` breaks this invariant entirely:

```rust
// src/more_ops.rs  (lines 854–885)
pub fn op_substr(
    a: &mut Allocator,
    input: NodePtr,
    _max_cost: Cost,      // ← max_cost is IGNORED
    _flags: ClvmFlags,
) -> Response {
    let ([a0, start, end], argc) = get_varargs::<3>(a, input, "substr")?;
    ...
    let r = a.new_substr(a0, start as u32, end as u32)?;
    let cost: Cost = 1;   // ← hardcoded, no size-based formula
    Ok(Reduction(cost, r))
}
``` [1](#0-0) 

The `_max_cost` parameter is silently discarded (the leading underscore is the Rust convention for intentionally unused variables), and the returned cost is always `1`. There are no `SUBSTR_BASE_COST` or `SUBSTR_COST_PER_BYTE` constants defined anywhere in the file, confirming this is not an accidental omission of a lookup — the formula was never implemented. [2](#0-1) 

`new_substr` in the allocator creates a new atom backed by a copy of the byte range `[start, end)` from the source atom. This is O(end − start) work in both CPU and heap allocation. The heap allocation itself incurs `MALLOC_COST_PER_BYTE` in every other operator that allocates, but `op_substr` skips even that. [3](#0-2) 

Compare with `op_strlen`, which correctly charges per byte:

```rust
// src/more_ops.rs (lines 841–852)
pub fn op_strlen(...) -> Response {
    let [n] = get_args::<1>(a, input, "strlen")?;
    let size = atom_len(a, n, "strlen")?;
    let size_node = a.new_number(size.into())?;
    let cost = STRLEN_BASE_COST + size as Cost * STRLEN_COST_PER_BYTE;
    Ok(malloc_cost(a, cost, size_node))
}
``` [4](#0-3) 

---

### Impact Explanation

The CLVM cost system is the sole mechanism preventing a single coin spend from consuming unbounded node CPU and memory. The `max_cost` passed to `run_program` is the budget enforced by the main loop:

```rust
// src/run_program.rs (lines 514–516)
if cost > effective_max_cost {
    return Err(EvalErr::CostExceeded);
}
``` [5](#0-4) 

Because `op_substr` always returns cost `1`, an attacker can call it in a tight loop against a large atom (e.g., a 1 MB atom produced by `concat`) and perform megabytes of allocator work while consuming only 1 cost unit per call. Within a budget of, say, 11 billion cost units (the Chia block cost limit), an attacker can invoke `substr` approximately 11 billion times, each time allocating and copying up to the full atom size. This constitutes a consensus-layer DoS: a transaction that should be rejected as too expensive is accepted, and every full node must execute it.

Additionally, because the `max_cost` parameter is ignored inside `op_substr`, even a caller that passes a tight budget cannot prevent the function from returning successfully — the function never checks whether its own cost of `1` is within budget (it always is), and it never checks whether the allocation it performs is proportional to the budget.

---

### Likelihood Explanation

`substr` (opcode `0x0c` = 12) is a standard, always-enabled CLVM operator dispatched unconditionally by `ChiaDialect::op` for any program submitted to the network. No special flags, softfork guards, or mempool-mode restrictions gate it. Any attacker who can submit a coin spend — which requires no special permissions on Chia — can trigger this path. The attacker-controlled inputs are the CLVM program bytes and the environment, both of which are fully attacker-controlled in a standard coin spend. [6](#0-5) 

---

### Recommendation

Implement a proper cost formula for `op_substr` consistent with the rest of the operator set. A minimal fix:

```rust
const SUBSTR_BASE_COST: Cost = 240;
const SUBSTR_COST_PER_BYTE: Cost = 1;

pub fn op_substr(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,   // remove the underscore — use this
    _flags: ClvmFlags,
) -> Response {
    ...
    let len = (end - start) as usize;
    let cost = SUBSTR_BASE_COST + len as Cost * SUBSTR_COST_PER_BYTE;
    check_cost(cost, max_cost)?;
    let r = a.new_substr(a0, start as u32, end as u32)?;
    Ok(malloc_cost(a, cost, r))
}
```

The malloc cost for the newly allocated atom should also be included via `malloc_cost`, as done in `op_strlen` and `op_concat`.

---

### Proof of Concept

A CLVM program that demonstrates the undercharging (pseudocode in CLVM assembly):

```
; Allocate a large atom via concat, then extract it via substr at cost 1
(a
  (q . (substr
          (concat <1MB_atom>)   ; cost: ~3,000,000
          (q . 0)
          (q . 1048576)))       ; cost: 1  ← should be ~1,048,816
  1)
```

Concretely, an attacker submits a spend bundle whose puzzle solution contains a CLVM program that:
1. Builds a large atom (e.g., 1 MB) using `concat` — this costs approximately `142 + 135 + 3*1048576 ≈ 3,145,853` units.
2. Calls `substr` on the full range of that atom in a loop — each call costs `1` unit but allocates and copies 1 MB.
3. Within the standard block cost budget of ~11 billion, the loop can run ~3.6 billion times after the initial `concat`, each time performing 1 MB of allocator work at cost `1`.

The program is accepted by the cost checker but forces every validating full node to perform terabytes of memory allocation work.

pub fn op_substr(
    a: &mut Allocator,
    input: NodePtr,
    _max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let ([a0, start, end], argc) = get_varargs::<3>(a, input, "substr")?;
    if !(2..=3).contains(&argc) {
        Err(EvalErr::InvalidOpArg(
            input,
            format!("Substring takes exactly 2 or 3 arguments, got {argc}"),
        ))?;
    }
    let size = atom_len(a, a0, "substr")?;
    let start = i32_atom(a, start, "substr")?;

    let end = if argc == 3 {
        i32_atom(a, end, "substr")?
    } else {
        size as i32
    };
    if end < 0 || start < 0 || end as usize > size || end < start {
        Err(EvalErr::InvalidOpArg(
            input,
            "Invalid Indices for Substring".to_string(),
        ))?
    } else {
        let r = a.new_substr(a0, start as u32, end as u32)?;
        let cost: Cost = 1;
        Ok(Reduction(cost, r))
    }
}
