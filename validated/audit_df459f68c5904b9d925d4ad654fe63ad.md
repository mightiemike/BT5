### Title
`Endpoint::submitTransactionsChecked` and `EndpointTx::submitSlowModeTransactionImpl` Do Not Validate Aliased Sender Addresses, Blocking All Forced-Inclusion Transactions When the Ink L2 Sequencer Is Down — (`File: core/contracts/Endpoint.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

Nado is deployed on Ink (OP Stack L2, `block.chainid == 57073`). Both the Nado sequencer's primary submission path (`submitTransactionsChecked`) and the owner's admin slow-mode submission path (`submitSlowModeTransactionImpl`) check `msg.sender` against a stored address without also accepting the L1-aliased equivalent. When the Ink L2 sequencer is down, any forced-inclusion transaction sent from L1 arrives with `msg.sender = storedAddress + 0x1111000000000000000000000000000000001111`. Both checks reject this aliased address, making it impossible for the Nado sequencer to process any transactions and impossible for the owner to submit emergency admin operations via forced inclusion.

---

### Finding Description

**Root cause 1 — `Endpoint::submitTransactionsChecked`** [1](#0-0) 

```solidity
require(msg.sender == sequencer);
```

`sequencer` is a plain EOA/contract address stored at initialization. On OP Stack chains, when a contract on L1 sends a message to L2 via the `CrossDomainMessenger` or the deposit portal (forced inclusion), the EVM sets `msg.sender` to `address(uint160(uint256(keccak256(abi.encodePacked(originalAddress))) + 0x1111000000000000000000000000000000001111))` — the aliased address. The check above will always revert for a forced-inclusion call from the Nado sequencer's L1 address, because the aliased address ≠ `sequencer`.

**Root cause 2 — `EndpointTx::submitSlowModeTransactionImpl`** [2](#0-1) 

```solidity
address sender = msg.sender;
...
require(sender == owner());
```

For the following privileged slow-mode transaction types — `WithdrawInsurance`, `DelistProduct`, `DumpFees`, `RebalanceXWithdraw`, `UpdateTierFeeRates`, `AddNlpPool`, `UpdateNlpPool`, `DeleteNlpPool`, `ForceRebalanceNlpPool`, `NlpProfitShare`, `UpdateBuilder` — the function requires `msg.sender == owner()`. If the owner (a multisig on L1) attempts to force-include one of these transactions during L2 sequencer downtime, `msg.sender` is the aliased owner address, and the check reverts.

---

### Impact Explanation

When the Ink L2 sequencer is down:

1. **All normal transaction processing is frozen.** `submitTransactionsChecked` is the only path through which the Nado sequencer can batch and process trades, liquidations, withdrawals, and price updates. With forced inclusion blocked, no `processTransaction` call can succeed via the sequencer path.
2. **Emergency admin operations are blocked.** The owner cannot force-include any of the privileged slow-mode transaction types (e.g., `DelistProduct`, `WithdrawInsurance`, `RebalanceXWithdraw`) because `require(sender == owner())` rejects the aliased address.
3. **User withdrawals via slow mode are unaffected** (they do not require `msg.sender == owner()`), but they cannot be executed by the sequencer either, since `executeSlowModeTransaction` depends on `nSubmissions` being advanced by `submitTransactionsChecked`.

Net result: the entire protocol is operationally frozen for the duration of the L2 sequencer outage, and the owner has no on-chain recourse to respond to any concurrent exploit or risk event.

---

### Likelihood Explanation

Ink is an OP Stack L2. OP Stack sequencer outages have occurred historically (Optimism mainnet experienced multi-hour outages in 2023). During such an event, forced inclusion via the L1 deposit portal is the only available transaction path. The probability of a sequencer outage coinciding with a need for emergency admin action is low but non-zero, and the consequence is a complete protocol freeze with no admin override capability.

---

### Recommendation

In `submitTransactionsChecked`, accept both the canonical sequencer address and its L1-aliased equivalent:

```solidity
address constant ADDRESS_ALIAS_OFFSET =
    0x1111000000000000000000000000000000001111;

function applyAlias(address a) internal pure returns (address) {
    return address(uint160(a) + uint160(ADDRESS_ALIAS_OFFSET));
}

require(
    msg.sender == sequencer || msg.sender == applyAlias(sequencer),
    ERR_UNAUTHORIZED
);
```

Apply the same pattern in `submitSlowModeTransactionImpl`:

```solidity
require(
    sender == owner() || sender == applyAlias(owner()),
    ERR_UNAUTHORIZED
);
```

And in `BaseProxyManager::onlySubmitter`:

```solidity
require(
    msg.sender == submitter || msg.sender == applyAlias(submitter),
    "only submitter can submit new impls."
);
```

### Proof of Concept

1. Nado is live on Ink (OP Stack L2). The Nado sequencer address is `SEQ` and the owner multisig is `OWNER` (both controlled from L1).
2. The Ink L2 sequencer goes down. Forced inclusion via the L1 deposit portal is the only available transaction path.
3. The Nado sequencer attempts to force-include a batch via `submitTransactionsChecked`. On L2, `msg.sender == applyAlias(SEQ) != SEQ`. The call reverts at `require(msg.sender == sequencer)`. [4](#0-3) 
4. Simultaneously, an attacker exploits a vulnerability in the protocol. The owner attempts to force-include a `DelistProduct` or `WithdrawInsurance` slow-mode transaction via `submitSlowModeTransaction`. On L2, `msg.sender == applyAlias(OWNER) != OWNER`. The call reverts at `require(sender == owner())`. [5](#0-4) 
5. The sequencer outage persists. No transactions are processed, no emergency admin action is possible, and the attacker's exploit proceeds unchecked until the L2 sequencer recovers.
