### Title
`submit_blocks` Unconditionally Calls `run_mainchain_gc` Which Has a Mismatched Pause-Bypass Role, Breaking `UnrestrictedSubmitBlocks` When Contract Is Paused - (`contract/src/lib.rs`)

---

### Summary

`submit_blocks` is decorated with `#[pause]` and grants bypass access to `Role::UnrestrictedSubmitBlocks`. However, it unconditionally calls `self.run_mainchain_gc(num_of_headers)` at line 181, which carries its own independent `#[pause(except(roles(Role::UnrestrictedRunGC)))]` guard. Because these are two distinct roles, a caller holding `UnrestrictedSubmitBlocks` but not `UnrestrictedRunGC` will pass the outer pause check on `submit_blocks` but then panic inside `run_mainchain_gc`, reverting the entire transaction. The `UnrestrictedSubmitBlocks` role is therefore broken whenever the contract is paused.

---

### Finding Description

`submit_blocks` is the primary block-ingestion entry point:

```rust
#[payable]
#[pause]
#[trusted_relayer]
pub fn submit_blocks(&mut self, ...) -> PromiseOrValue<()> {
    ...
    for header in headers { self.submit_block_header(header, ...); }
    self.run_mainchain_gc(num_of_headers);   // ← unconditional internal call
    ...
}
``` [1](#0-0) 

The `#[trusted_relayer]` macro grants bypass to `Role::DAO` and `Role::UnrestrictedSubmitBlocks`, so those accounts can call `submit_blocks` even while the contract is paused.

`run_mainchain_gc` is a separate public method with its own, different bypass role:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }
``` [2](#0-1) 

The `near-plugins` `#[pause]` macro injects a pause check at the top of the function body. Because `run_mainchain_gc` is called as a plain Rust method (`self.run_mainchain_gc(...)`), not a cross-contract call, the injected check still executes. A caller that holds `UnrestrictedSubmitBlocks` but not `UnrestrictedRunGC` will:

1. Pass the pause guard on `submit_blocks`.
2. Process all submitted headers successfully.
3. Panic inside `run_mainchain_gc` because the caller lacks `UnrestrictedRunGC`.
4. Have the entire transaction reverted — no headers are stored.

The two roles are defined independently:

```rust
UnrestrictedSubmitBlocks,   // line 44
UnrestrictedRunGC,          // line 46
``` [3](#0-2) 

There is no code path in `submit_blocks` that skips or guards the `run_mainchain_gc` call based on the contract's pause state or the caller's role set.

---

### Impact Explanation

The `UnrestrictedSubmitBlocks` role exists precisely to keep the chain synchronized during a pause (e.g., a security incident where the contract is paused but the relayer must continue ingesting headers). Because `submit_blocks` unconditionally calls `run_mainchain_gc`, which enforces a different bypass role, the `UnrestrictedSubmitBlocks` role is entirely non-functional when the contract is paused. Block submissions are frozen for all callers except those holding both `UnrestrictedSubmitBlocks` and `UnrestrictedRunGC` (or `DAO`). The mainchain tip falls behind, and downstream SPV proof verification (`verify_transaction_inclusion_v2`) will fail for any block submitted after the pause began, because those blocks are never stored.

---

### Likelihood Explanation

The scenario requires the contract to be paused — an admin action — and a relayer to hold `UnrestrictedSubmitBlocks` but not `UnrestrictedRunGC`. This is the exact operational pattern the role separation implies: different operators may be granted different bypass rights. The bug is latent and will manifest the first time the contract is paused in production with a relayer that has only `UnrestrictedSubmitBlocks`.

---

### Recommendation

Inside `submit_blocks`, replace the unconditional call to `run_mainchain_gc` with a direct call to the underlying GC logic (extracted into a private helper), bypassing the public method's pause guard. Alternatively, grant `UnrestrictedSubmitBlocks` callers the same pause bypass as `UnrestrictedRunGC` for the internal GC step, or document that both roles must always be granted together and enforce this in the role-granting logic.

---

### Proof of Concept

1. Deploy the contract and pause it (`PauseManager` calls `pa_pause_feature`).
2. Grant account `relayer.near` the `UnrestrictedSubmitBlocks` role but **not** `UnrestrictedRunGC`.
3. Have `relayer.near` call `submit_blocks` with a valid batch of headers.
4. Observe: the call panics inside `run_mainchain_gc` with the pause error, reverting all header storage despite the caller holding the intended bypass role.
5. Grant `relayer.near` also `UnrestrictedRunGC` and repeat — the call succeeds, confirming the root cause is the role mismatch between the two `#[pause]` guards.
