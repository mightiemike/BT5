### Title
Deposit Allowlist Checks `owner` Instead of `sender`, Allowing Any Operator to Bypass the Gate - (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and gates on `owner` (the LP-position recipient) instead. Because `owner` is a caller-controlled parameter, any address not on the allowlist can bypass the restriction by supplying an allowlisted address as `owner`.

### Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` argument as `owner` to the extension hook: [1](#0-0) 

The extension hook receives both values, but the first parameter (`sender`) is unnamed and discarded: [2](#0-1) 

The guard at line 38 checks `allowedDepositor[msg.sender][owner]` â€” where `msg.sender` is the pool (correct key) and `owner` is the LP-position recipient (wrong identity). The actual depositor (`sender`) is never consulted.

Because `owner` is a free parameter in the public `addLiquidity` signature: [3](#0-2) 

any caller can pass `owner = allowlistedAddress` and the check resolves to `allowedDepositor[pool][allowlistedAddress] == true`, passing unconditionally regardless of who the actual caller is.

The periphery `MetricOmmPoolLiquidityAdder` makes this even more accessible â€” it has an explicit overload that accepts a separate `owner` argument: [4](#0-3) 

### Impact Explanation

The deposit allowlist is the sole access-control mechanism for restricting who may add liquidity to a pool. Bypassing it means:

- Any address can inject liquidity into a pool configured as restricted (e.g., KYC-gated, protocol-only, or composition-controlled).
- Allowlisted owners receive LP shares they did not request, which can be used to grief them (forced tax events, dilution of their position key, or locking them into unwanted exposure).
- The pool admin's security invariant â€” that only approved depositors can mint shares â€” is completely broken.

This is broken core pool functionality under the contest rules.

### Likelihood Explanation

The attack requires zero privileges. Any EOA or contract can call `pool.addLiquidity(owner=knownAllowlistedAddress, ...)` directly, or use `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, knownAllowlistedAddress, ...)`. The allowlisted addresses are discoverable from on-chain `AllowedToDepositSet` events. No special setup, flash loan, or oracle manipulation is needed.

### Recommendation

Replace the unnamed first parameter with a named `sender` and gate on it instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern used by `SwapAllowlistExtension`, which gates on `sender` (the actual swap initiator).
