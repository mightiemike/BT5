### Title
`DepositAllowlistExtension` Checks Position Owner Instead of Actual Depositor, Allowing Unauthorized Liquidity Additions - (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` parameter and gates on `owner` (the position beneficiary) instead. Because `MetricOmmPoolLiquidityAdder` lets any caller supply an arbitrary `owner`, an address that is not on the allowlist can add liquidity to a restricted pool by naming an allowlisted address as the position owner.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses into the extension hook:

- `sender` = `msg.sender` of the pool call (the actual depositor / token payer)
- `owner` = the position beneficiary supplied by the caller [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` ignores `sender` entirely (the first parameter is unnamed) and checks only `owner`: [2](#0-1) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (the two-argument overload) accepts an explicit `owner` address from any caller and forwards it to the pool, while `msg.sender` (the payer) is stored only in the transient pay-context: [3](#0-2) 

The pool then calls the extension with `sender = LiquidityAdder` and `owner = alice` (the allowlisted address supplied by the unauthorized caller). The extension evaluates `allowedDepositor[pool][alice]` → `true` and the hook passes, even though the actual payer (`bob`) is not on the allowlist.

This is the direct analog of the AuraLocker skew: `getPastTotalSupply` counted all locked balances while `getPastVotes` counted only delegated ones — two measures of the same concept that diverge because the wrong set is used for one side. Here, the allowlist is named "depositor allowlist" and its NatSpec states *"Gates `addLiquidity` by depositor address"*, but the implementation gates on the position owner (a different entity), creating the same kind of mismatch between the intended guard subject and the actual guard subject.

---

### Impact Explanation

Any address not on the deposit allowlist can add liquidity to a restricted pool by specifying any allowlisted address as `owner`. The unauthorized actor pays the tokens; the position is recorded under the allowlisted address. The pool admin's ability to control who participates as a liquidity provider is completely nullified. In pools where the allowlist enforces regulatory, risk, or counterparty constraints, this allows unrestricted participation, breaking the core access-control invariant of the extension.

---

### Likelihood Explanation

The attack requires no special privileges. Any EOA or contract can call `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, allowlistedAddress, ...)` with a publicly known allowlisted address (e.g., the pool admin, an existing LP, or any address visible on-chain). No flash loan, oracle manipulation, or privileged role is needed. The only cost is the token amount deposited.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor / pool caller) rather than `owner`:

```solidity
function beforeAddLiquidity(address sender, address /*owner*/, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to allowlist end-users rather than intermediary contracts (routers, liquidity adders), the periphery contracts should also forward the originating user address as `sender`, or the allowlist should be applied at the periphery layer where the true payer is known.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` configured; `allowAllDepositors[pool] = false`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is **not** allowlisted.
3. Bob calls:
   ```solidity
   liquidityAdder.addLiquidityExactShares(
       pool,
       alice,   // owner — allowlisted
       salt,
       deltas,
       max0, max1,
       extensionData
   );
   ```
4. `MetricOmmPoolLiquidityAdder._addLiquidity` calls `pool.addLiquidity(alice, salt, deltas, abi.encode(KIND_PAY), extensionData)`.
5. Pool calls `extension.beforeAddLiquidity(sender=LiquidityAdder, owner=alice, ...)`.
6. Extension evaluates `allowedDepositor[pool][alice]` → `true`. Hook passes.
7. Bob's tokens are pulled in the callback; the position is minted under `alice`.
8. Bob has successfully added liquidity to a pool he is not authorized to deposit into, bypassing the allowlist entirely. [2](#0-1) [3](#0-2) [4](#0-3)
