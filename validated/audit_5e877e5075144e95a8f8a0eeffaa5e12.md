### Title
Non-Refundable Slow Mode Fee Is Permanently Lost When Execution Fails Due to Liquidation During the 3-Day Delay Window — (`core/contracts/Endpoint.sol`, `core/contracts/EndpointStorage.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

The Nado protocol charges a non-refundable `SLOW_MODE_FEE` ($1 USDC) at the time a user submits a slow mode transaction. If the queued transaction fails during execution — which can be forced by an unprivileged liquidator draining the user's balance during the mandatory 3-day delay — the fee is permanently lost with no refund path. A comment in the code explicitly confirms the refund mechanism was removed.

---

### Finding Description

When a user calls `submitSlowModeTransaction()`, the protocol immediately collects `SLOW_MODE_FEE` ($1 USDC) from the user's external wallet via `chargeSlowModeFee`: [1](#0-0) 

This fee is transferred to `address(this)` (the Endpoint contract) and the `slowModeFees` accumulator is incremented: [2](#0-1) 

The transaction is queued with a hardcoded 3-day execution delay: [3](#0-2) 

When the slow mode transaction is eventually executed, it is wrapped in a `try/catch`. If the inner call reverts, the catch block does nothing. The comment `// try return funds now removed` explicitly confirms that a prior refund mechanism was intentionally deleted: [4](#0-3) 

There is no code path anywhere in the protocol that refunds `SLOW_MODE_FEE` to the original submitter on failure.

**Attacker-controlled trigger:** A user submits a slow mode `WithdrawCollateral` transaction, paying $1 upfront. During the 3-day window, an unprivileged liquidator submits a signed `LiquidateSubaccount` transaction to the sequencer, which drains the user's on-chain balance: [5](#0-4) 

When the slow mode withdrawal is later executed (callable by anyone via `executeSlowModeTransaction()`), `clearinghouse.withdrawCollateral()` fails because the balance is now zero. The catch block silently swallows the revert, and the $1 fee is permanently locked in the Endpoint contract. [6](#0-5) 

The `slowModeFees` state variable accumulates these lost fees but is never used to issue refunds: [7](#0-6) 

---

### Impact Explanation

A user who submits a slow mode `WithdrawCollateral` and is subsequently liquidated during the 3-day window loses their $1 `SLOW_MODE_FEE` with no recourse. The fee is collected from the user's external ERC20 balance at submission time and is irrecoverable once the queued transaction fails. The `SLOW_MODE_FEE` constant is $1 USDC: [8](#0-7) 

This is a direct asset loss for the user. The impact is **Medium**: the per-transaction loss is bounded at $1, but it is a guaranteed, irreversible loss whenever the failure condition is met, and it can be deliberately triggered by a liquidator.

---

### Likelihood Explanation

The 3-day slow mode delay (`SLOW_MODE_TX_DELAY`) creates a wide window during which market conditions can change and a position can become liquidatable: [9](#0-8) 

Any unprivileged liquidator who observes a pending slow mode withdrawal can monitor the submitter's health and liquidate them before the withdrawal executes. The liquidator is incentivized by the `LIQUIDATION_FEE` profit. This is a realistic scenario in volatile markets and requires no privileged access.

---

### Recommendation

Restore a refund mechanism in the `catch` block of `_executeSlowModeTransaction`. When a slow mode transaction fails, the protocol should return `SLOW_MODE_FEE` to `txn.sender` by transferring the USDC back from the Endpoint contract and decrementing `slowModeFees`. Alternatively, charge the slow mode fee only upon successful execution rather than at submission time.

---

### Proof of Concept

1. User Alice submits a slow mode `WithdrawCollateral` for product 0 (USDC). `chargeSlowModeFee` transfers $1 USDC from Alice's wallet to the Endpoint contract. `slowModeFees += 1e6`.
2. Alice's subaccount balance is $10. During the 3-day window, the USDC price of her collateral drops. Liquidator Bob signs a `LiquidateSubaccount` transaction targeting Alice and submits it to the sequencer. The sequencer includes it; Alice's balance is reduced to 0.
3. After 3 days, anyone calls `executeSlowModeTransaction()`. `_executeSlowModeTransaction` dequeues Alice's withdrawal and calls `this.processSlowModeTransaction(alice, withdrawTx)`.
4. Inside `processSlowModeTransactionImpl`, `clearinghouse.withdrawCollateral(alice, 0, amount, ...)` reverts because Alice's balance is 0.
5. The `catch` block in `_executeSlowModeTransaction` fires. No refund is issued. Alice's $1 SLOW_MODE_FEE remains locked in the Endpoint contract permanently.
