Audit Report

## Title
SwapAllowlistExtension Sender Identity Mismatch: Router-Mediated Swaps Bypass Per-User Allowlist on Curated Pools — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps on `allowedSwapper[pool][sender]` where `sender` is the `msg.sender` of the `pool.swap()` call. When any user routes through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the originating EOA. A pool admin who allowlists the router to enable router-mediated swaps on a curated pool inadvertently opens the pool to every EOA, completely defeating the per-user allowlist.

## Finding Description
`MetricOmmPool.swap()` unconditionally passes `msg.sender` as the `sender` argument to `_beforeSwap()`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- always the immediate caller of pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap()` forwards that value verbatim to the extension via `abi.encodeCall`:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap()` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly with no forwarding of the original `msg.sender`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

The result is a structural identity mismatch: the extension always sees `sender = router` for every user who goes through the router. The admin faces an impossible choice: allowlist individual EOAs only (those users cannot use the router, since the extension sees `sender = router` and reverts), or allowlist the router (any EOA can bypass the per-user gate by routing through the router). There is no configuration that simultaneously allows router-mediated swaps and enforces per-user identity checks. The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` since all router entry points call `pool.swap()` with `msg.sender = router`.

## Impact Explanation
A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router address loses all per-user access control. Any non-allowlisted EOA can execute swaps on the curated pool by calling `MetricOmmSimpleRouter.exactInputSingle()`. The pool curation invariant — that only individually approved addresses may trade — is permanently broken for all router-mediated paths. This constitutes broken core pool functionality: the primary purpose of the extension (restricting who may swap) is nullified.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical swap interface for the protocol. Pool admins who want to support normal user flows will naturally allowlist the router. The mismatch is not documented in the extension or the router, so the admin has no signal that allowlisting the router is equivalent to `setAllowAllSwappers(pool, true)`. The likelihood of this misconfiguration is high for any curated pool that intends to support router usage.

## Recommendation
`SwapAllowlistExtension.beforeSwap()` should gate on the originating user, not the immediate `pool.swap()` caller. Two viable approaches:

1. **Pass `tx.origin` as an additional parameter** — the pool could expose the original initiator alongside `sender`, or the extension could read `tx.origin` directly (acceptable for allowlist gating since `tx.origin` cannot be spoofed by an intermediate contract).
2. **Require direct pool interaction for allowlisted pools** — document and enforce that pools using `SwapAllowlistExtension` must not allowlist the router; users must call `pool.swap()` directly. This requires clear documentation and ideally an on-chain guard.

The cleanest fix is for the pool to pass both `msg.sender` (the immediate caller) and the original initiator (e.g., stored in transient storage by the router and readable by extensions), so the extension can check the correct identity.

## Proof of Concept

```solidity
// 1. Deploy pool with SwapAllowlistExtension
// 2. Allowlist only the router (admin intends to allow router-mediated swaps)
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// 3. Non-allowlisted EOA swaps directly → reverts (correct)
vm.prank(nonAllowlistedEOA);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(nonAllowlistedEOA, false, int128(1000), type(uint128).max, "", "");

// 4. Same non-allowlisted EOA swaps via router → SUCCEEDS (bypass)
vm.prank(nonAllowlistedEOA);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: address(token0),
    tokenOut: address(token1),
    zeroForOne: false,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: nonAllowlistedEOA,
    deadline: block.timestamp + 1,
    priceLimitX64: type(uint128).max,
    extensionData: ""
}));
// swap succeeds — allowlist bypassed
// Extension sees sender = router (allowlisted), passes, even though actual trader is non-allowlisted EOA
```
