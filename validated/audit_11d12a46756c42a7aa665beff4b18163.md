### Title
`skip_stateful_validations` Bypasses Account Signature Verification (`__validate__`) for Any Account With a Pending Mempool Transaction, Not Just Deploy-Account - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function returns `true` (skip `__validate__`) whenever an Invoke transaction has `nonce == 1` and the on-chain account nonce is `0`, provided `account_tx_in_pool_or_recent_block` returns `true`. The guard is too broad: it fires for **any** transaction from the account address in the mempool, not exclusively a `DeployAccount`. An unprivileged attacker who observes a legitimate `DeployAccount` in the mempool can submit an Invoke with `nonce = 1` and a completely invalid signature; the gateway skips `__validate__` and admits the transaction.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` implements a UX shortcut: when a user sends `deploy_account + invoke` simultaneously, the invoke's `__validate__` would fail because the account contract is not yet deployed. The function therefore skips the blockifier-level signature check when the following three conditions hold simultaneously:

```
tx is Invoke
tx.nonce() == Nonce(Felt::ONE)
account_nonce == Nonce(Felt::ZERO)
mempool_client.account_tx_in_pool_or_recent_block(sender) == true
``` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`, which causes `StatefulValidator::perform_validations` to return `Ok(())` immediately after `perform_pre_validation_stage` without ever calling the account contract's `__validate__` entry point. [2](#0-1) 

The guard `account_tx_in_pool_or_recent_block` checks only whether the account address appears anywhere in the mempool pool or state — it does **not** verify that the matching transaction is a `DeployAccount`: [3](#0-2) 

The comment in `skip_stateful_validations` acknowledges this: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is incorrect for the attacker scenario below. [4](#0-3) 

### Impact Explanation

An attacker who observes a legitimate `DeployAccount` for address `A` in the public mempool can submit an `InvokeTransaction` with `nonce = 1` and an **arbitrary/invalid signature** for address `A`. The gateway will:

1. Pass `validate_nonce` — nonce `1` is within `max_allowed_nonce_gap` of account nonce `0`.
2. Pass `validate_by_mempool` — no duplicate, nonce ordering is acceptable.
3. Evaluate `skip_stateful_validations` → `true` because `account_tx_in_pool_or_recent_block(A)` is `true` (the legitimate `DeployAccount` is present).
4. Call `run_validate_entry_point` with `validate = false`, skipping `__validate__` entirely.
5. Admit the invalid transaction to the mempool.

This matches the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.** [5](#0-4) 

### Likelihood Explanation

The mempool is observable (transactions are broadcast over P2P). Any `DeployAccount` for a fresh account is visible. The attacker needs only the target address and the knowledge that a `DeployAccount` is pending — no private key or privileged access is required. The attack is cheap to execute repeatedly for every new account deployment, enabling sustained mempool spam with signature-invalid transactions.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` guard with a check that specifically verifies a `DeployAccount` transaction exists for the sender address in the mempool. Alternatively, restrict the skip to the case where the mempool contains a `DeployAccount` for the exact sender, rather than any transaction type. The nonce check (`validate_nonce`) already runs before `skip_stateful_validations` and is unaffected; only the `__validate__` bypass needs tightening. [6](#0-5) 

### Proof of Concept

1. Legitimate user submits `DeployAccount` for address `A` (account nonce in state = 0). It enters the mempool.
2. Attacker calls the gateway's `add_transaction` with:
   - Type: `InvokeV3`
   - `sender_address`: `A`
   - `nonce`: `1`
   - `signature`: `[0xdeadbeef]` (invalid)
   - Valid resource bounds, tip, etc.
3. Gateway flow:
   - Stateless validation passes (no signature format check beyond length).
   - `validate_nonce`: nonce `1` ≤ `0 + max_allowed_nonce_gap` → passes.
   - `validate_by_mempool`: no duplicate → passes.
   - `skip_stateful_validations`: `nonce == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(A) == true` → returns `true`.
   - `run_validate_entry_point(skip_validate=true)`: sets `validate=false`, skips `__validate__`.
4. Transaction is admitted to the mempool with an invalid signature.
5. When the batcher later executes it, `__validate__` runs and the transaction fails/reverts — but it has already consumed mempool capacity and batcher resources. [7](#0-6) [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
