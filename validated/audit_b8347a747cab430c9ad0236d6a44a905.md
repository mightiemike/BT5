Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of End User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the immediate caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` equals the router address, not the end user. Any pool admin who allowlists the router to permit allowlisted users to trade through it simultaneously opens the gate to every non-allowlisted address that calls the same public router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()`. When the entry point is `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly: [4](#0-3) 

So `sender` = router address. The extension evaluates the router's allowlist entry, not the end user's. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

The `onlyPoolAdmin` modifier correctly gates the setter, but the runtime check evaluates the wrong identity: [6](#0-5) 

## Impact Explanation
This is a direct admin-boundary break. The pool admin's sole swap access-control mechanism — the allowlist — is rendered ineffective for any pool that allowlists the router. LPs who deposited under the assumption that only vetted counterparties could trade against them are exposed to unrestricted swap flow from any address, including adversarial MEV bots and unauthorized volume. The exact corrupted value is the `allowedSwapper[pool][sender]` extension decision: it returns `true` for the router when it should return `false` for the non-allowlisted end user.

## Likelihood Explanation
Medium. The bypass requires the pool admin to have allowlisted the router address. Any pool that wants allowlisted users to access the router must do exactly this, making the precondition a routine operational step. Once the router is allowlisted, the bypass is trivially reachable by any address with no special privileges, no capital requirement beyond the swap amount, and is repeatable indefinitely.

## Recommendation
`SwapAllowlistExtension` must gate on the original end user, not the immediate caller. The cleanest fix: the router encodes `msg.sender` into `extensionData`; `SwapAllowlistExtension.beforeSwap` decodes and checks that address when `sender` is a registered intermediary. Alternatively, the pool could pass an additional `originator` field through the hook interface, populated by the router before calling `pool.swap()`.

## Proof of Concept
```
Setup:
  pool deployed with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, userA, true)
  pool admin calls setAllowedToSwap(pool, router, true)   ← required for userA to use the router

Attack:
  userB (not allowlisted) calls:
    router.exactInputSingle({ pool: pool, ... })

Execution trace:
  router.exactInputSingle()
    → pool.swap(recipient, ...)          // msg.sender in pool = router
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓
          → returns selector (no revert)
      → swap executes for userB

userB's swap succeeds despite never being added to the allowlist.
```
