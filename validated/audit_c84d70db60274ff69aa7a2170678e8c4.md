### Title
`deser_auto` silently drops `max_atom_len` for classic/backrefs-format blobs — (`wheel/src/api.rs`)

---

### Summary

`deser_auto` accepts a `max_atom_len` parameter but only forwards it to the serde_2026 decoder. When the input blob does not start with `SERDE_2026_MAGIC_PREFIX`, the function falls through to `node_from_bytes_backrefs`, which has no `max_atom_len` parameter and no configurable atom-size cap. The cap is silently discarded, and any atom up to the hard-coded ~17 GiB ceiling is accepted.

---

### Finding Description

In `wheel/src/api.rs`, `deser_auto` is defined as:

```rust
#[pyo3(signature = (blob, *, max_atom_len=PY_DEFAULT_MAX_ATOM_LEN, strict=true))]
fn deser_auto(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
        deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)
            .map_err(eval_to_py)?
    } else {
        node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?   // ← max_atom_len dropped
    };
    Ok(LazyNode::new(Rc::new(a), node))
}
``` [1](#0-0) 

The serde_2026 branch correctly passes `max_atom_len` to `deserialize_2026_body_from_stream`, which enforces it via `checked_bounded_usize`. [2](#0-1) 

The classic/backrefs branch calls `node_from_bytes_backrefs`, which has no `max_atom_len` parameter at all. It delegates to `parse_atom` → `decode_size`, whose only hard cap is `0x400000000` (~17 GiB). [3](#0-2) 

The Python `serde.py` `deserialize()` wrapper explicitly documents that `max_atom_len` is "applied to '2026' and 'auto' paths" and passes it through to `deser_auto` — but the Rust function silently ignores it for the classic-format branch. [4](#0-3) 

---

### Impact Explanation

Any Python caller that passes `max_atom_len` to `deser_auto` (or to `deserialize(..., fmt="auto", max_atom_len=N)`) to bound memory allocation receives no protection when the input is a classic or backrefs-format blob. An attacker who controls the serialized bytes can:

1. Craft a classic CLVM blob containing a single large atom (e.g., 100 MiB).
2. Submit it to a path that calls `deser_auto` with a small `max_atom_len` (e.g., 10 bytes).
3. The atom is accepted and allocated in full, bypassing the caller's cap.

This breaks the documented API contract, can cause OOM in memory-constrained environments (mempool, wallet parsing), and can produce consensus divergence if different nodes apply the cap differently (e.g., one node rejects via serde_2026 path, another accepts via classic path).

---

### Likelihood Explanation

The attacker-controlled entrypoint is the public Python wheel API (`deser_auto`, `deserialize`). Classic CLVM format is the dominant wire format for Chia transactions. Any caller that trusts `max_atom_len` to be enforced uniformly — a reasonable expectation given the documented API contract and the default value of `PY_DEFAULT_MAX_ATOM_LEN = 1 << 20` — is silently unprotected for classic-format inputs. [5](#0-4) 

---

### Recommendation

Pass `max_atom_len` into the classic/backrefs branch. This requires adding a `max_atom_len: usize` parameter to `node_from_stream_backrefs` (and its callers), threading it through to `parse_atom`, and enforcing it in `parse_atom_ptr` after `decode_size` returns the atom length — rejecting atoms whose decoded size exceeds the cap before any allocation is attempted.

---

### Proof of Concept

```python
from clvm_rs.serde import deserialize

# Classic CLVM encoding of a single 1000-byte atom:
# length prefix for 1000 bytes = 0x82 0x03 0xe8, then 1000 zero bytes
atom_len = 1000
length_prefix = bytes([0x82, (atom_len >> 8) & 0xFF, atom_len & 0xFF])
classic_blob = length_prefix + b'\x00' * atom_len

# This should raise an error (max_atom_len=10), but it does NOT for classic format:
node = deserialize(classic_blob, fmt="auto", max_atom_len=10)
print("BUG: large atom accepted despite max_atom_len=10")

# Contrast: serde_2026 format correctly rejects the same atom size:
from clvm_rs.serde import serialize, clvm_tree_to_lazy_node
blob_2026 = serialize(node, "2026")
try:
    deserialize(blob_2026, fmt="auto", max_atom_len=10)
    print("BUG: also accepted in 2026 path (unexpected)")
except Exception as e:
    print(f"Correctly rejected in 2026 path: {e}")
```

The first `deserialize` call succeeds (no exception), allocating 1000 bytes despite `max_atom_len=10`. The second call correctly raises an error. This demonstrates the format-dependent inconsistency.
