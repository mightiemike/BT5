### Title
`SwapAllowlistExtension` gates the router address instead of the end user, making per-user swap allowlisting impossible via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is the **direct caller of `pool.swap()`**. When users route through the public `MetricOmmSimpleRouter`, the direct caller is the router contract, not the end user. This creates an irreconcilable mismatch: allowlisting individual user EOAs blocks them from using the router, while allowlisting the router opens the pool to every user on-chain, defeating the purpose of the allowlist entirely.

---

### Finding Description

`ExtensionCalling._beforeSwap` passes `msg.sender` of the pool's `swap()` call as the `sender` argument to every configured extension: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the extension's caller) and `sender` is that forwarded direct-pool-caller address: [2](#0-1) 

`MetricOmmSimpleRouter` is a public, permissionless periphery contract. Any user can call it, and it in turn calls `pool.swap(...)`. The pool therefore sees `msg.sender = router`, so the extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end-user]`. [3](#0-2) 

The pool admin has exactly two choices, both broken:

| Admin action | Effect |
|---|---|
| Allowlist individual user EOAs | Those users cannot swap via the router (router not allowlisted → revert) |
| Allowlist the router address | Every user on-chain can swap via the router, bypassing per-user gating |

There is no configuration that achieves per-user gating for router-mediated swaps.

Note the asymmetry with `DepositAllowlistExtension`, which correctly checks the `owner` (position beneficiary) and ignores `sender`, making the deposit gate identity-stable regardless of who the intermediary is: [4](#0-3) 

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., institutional-only, KYC-gated, or MEV-protected pools) cannot enforce that restriction for router-mediated swaps. If the pool admin allowlists the router to support normal UX, every unpermissioned address can execute swaps against the pool. LP providers who deposited under the assumption that only allowlisted counterparties could trade against them are exposed to unrestricted adversarial flow, including oracle-lag exploitation and directional pressure from non-allowlisted actors. This constitutes a broken core pool functionality with direct LP principal exposure.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap entry point described in the project architecture. Any pool that (a) configures `SwapAllowlistExtension` for access control and (b) allowlists the router to support normal UX triggers the bypass automatically. No special privileges, flash loans, or exotic tokens are required — a standard `exactInput` or `exactOutput` call through the router suffices. [5](#0-4) 

---

### Recommendation

Change `SwapAllowlistExtension.beforeSwap` to check the `recipient` or, better, add an end-user identity field to `extensionData` that the router populates and the extension verifies (with the pool enforcing that the router signs the payload). Alternatively, mirror the deposit-allowlist design: check the `recipient` argument (the economic beneficiary of the swap output) rather than `sender` (the intermediary). At minimum, document clearly that allowlisting the router is equivalent to `allowAll = true` and that per-user gating requires direct pool calls only.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension as beforeSwap hook
  pool admin calls setAllowedToSwap(pool, router, true)   // to support router UX
  pool admin does NOT call setAllowedToSwap(pool, Eve, true)

Attack:
  Eve calls MetricOmmSimpleRouter.exactInput(pool, ...)
  Router calls pool.swap(recipient=Eve, ...)
  Pool calls _beforeSwap(sender=Router, ...)
  SwapAllowlistExtension checks allowedSwapper[pool][Router] → true  ✓
  Eve's swap executes against the restricted pool

Result:
  Eve, a non-allowlisted address, successfully swaps against a pool
  whose admin believed only explicitly allowlisted counterparties could trade.
  LP providers are exposed to unrestricted adversarial swap flow.
``` [6](#0-5) [7](#0-6)
