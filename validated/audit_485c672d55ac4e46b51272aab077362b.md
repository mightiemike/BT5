### Title
Owner Can Accept Unexpected Risk Parameters in `addOrUpdateProducts` Due to Missing Full-Parameter Verification ‚Äî (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.sol` implements a two-step product-update flow: the `deployer` queues pending calls, and the `owner` (multisig) executes them via `addOrUpdateProducts()`. The execution step only verifies product IDs, not the full encoded parameters. The deployer can clear and re-queue calls with the same product IDs but different (potentially dangerous) risk parameters between when the owner prepares their transaction and when it executes, causing the owner to unknowingly apply unexpected `riskStore` or `config` values.

---

### Finding Description

`ContractOwner.sol` separates the roles of `deployer` (schedules updates) and `owner` (executes them). The deployer calls `submitSpotAddOrUpdateProductCall()` or `submitPerpAddOrUpdateProductCall()` to push encoded structs into `rawSpotAddOrUpdateProductCalls` / `rawPerpAddOrUpdateProductCalls`. The owner then calls `addOrUpdateProducts(spotIds, perpIds)` to apply them. [1](#0-0) 

The only validation inside `addOrUpdateProducts` is:

```solidity
require(spotIds[i] == call.productId, "spot mismatch");   // line 156
require(perpIds[i] == call.productId, "perp mismatch");   // line 173
``` [2](#0-1) 

This checks only the product ID. The full struct ‚Äî including `riskStore` (weight parameters, price oracle), `config`, `sizeIncrement`, `minSize`, and `quoteId` ‚Äî is decoded directly from the on-chain queue and applied without any owner-side verification.

The deployer can atomically clear the queue and re-submit with the same product IDs but different parameters:

```solidity
function clearSpotAddOrUpdateProductCalls() external onlyDeployer {
    delete rawSpotAddOrUpdateProductCalls;   // line 140
}
``` [3](#0-2) 

Because `submitSpotAddOrUpdateProductCall` only deduplicates by product ID (not by full parameter hash), the deployer can re-queue the same product ID with a different `riskStore` after the owner has already prepared their `addOrUpdateProducts` calldata. [4](#0-3) 

---

### Impact Explanation

The `riskStore` struct controls `longWeightInitial`, `shortWeightInitial`, `longWeightMaintenance`, `shortWeightMaintenance`, and `priceX18` ‚Äî the core parameters that determine how much leverage a subaccount can take and how the clearinghouse evaluates health. If the deployer substitutes a `riskStore` with inflated weight values, the protocol could allow excessive leverage, leading to undercollateralized positions and solvency risk for the insurance fund and other users.

The `config` field for spot products controls the token address and interest rate model. Substituting a malicious config could redirect collateral accounting to an unexpected token.

---

### Likelihood Explanation

The deployer and owner are explicitly separate roles in `ContractOwner.sol`. The deployer is a hot key used for operational submissions; the owner is a multisig used for final approval. The window between the multisig signers reviewing the pending queue and the `addOrUpdateProducts` transaction being mined is realistic on any network with non-instant finality. The deployer does not need to front-run in the mempool sense ‚Äî they only need to act before the owner's transaction is included.

---

### Recommendation

Modify `addOrUpdateProducts` to accept a hash (or full encoding) of the expected pending calls and compare it against a hash of the on-chain queue before executing. If they differ, revert. For example:

```solidity
function addOrUpdateProducts(
    uint32[] memory spotIds,
    uint32[] memory perpIds,
    bytes32 expectedSpotHash,   // keccak256 of rawSpotAddOrUpdateProductCalls
    bytes32 expectedPerpHash
) external onlyOwner {
    require(keccak256(abi.encode(rawSpotAddOrUpdateProductCalls)) == expectedSpotHash, "spot params changed");
    require(keccak256(abi.encode(rawPerpAddOrUpdateProductCalls)) == expectedPerpHash, "perp params changed");
    // ... existing logic
}
```

This mirrors the fix recommended for `acceptPendingOracle()`: the caller supplies what they expect, and the function reverts if the on-chain state has diverged.

---

### Proof of Concept

1. Deployer calls `submitSpotAddOrUpdateProductCall(productId=1, ..., riskStore=safeParams)`.
2. Owner (multisig) reads the pending queue, sees `productId=1` with `safeParams`, and prepares `addOrUpdateProducts([1], [])`.
3. Before the multisig transaction is mined, deployer calls `clearSpotAddOrUpdateProductCalls()` then `submitSpotAddOrUpdateProductCall(productId=1, ..., riskStore=dangerousParams)`.
4. Owner's `addOrUpdateProducts([1], [])` executes. `require(spotIds[0] == call.productId)` passes (both are `1`). `dangerousParams` are applied to the spot engine instead of `safeParams`.
5. The spot product now has inflated collateral weights, enabling undercollateralized positions. [5](#0-4)
