### Title
Inconsistent Flag-Gated Dispatch: 4-Byte secp Opcodes Bypass `ENABLE_SECP_OPS` Guard — (File: src/chia_dialect.rs)

---

### Summary

`ChiaDialect::op()` in `src/chia_dialect.rs` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through **two separate code paths** with **inconsistent flag validation**. The 1-byte opcode aliases (64, 65) require `ClvmFlags::ENABLE_SECP_OPS` to be set, but the 4-byte opcode forms (`0x13d61f00`, `0x1c3a8f00`) are dispatched **unconditionally** with no flag check. An attacker supplying attacker-controlled CLVM bytes can invoke secp signature verification via the 4-byte opcode path regardless of whether `ENABLE_SECP_OPS` is set, bypassing the intended flag-based access control.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` handles operator dispatch in two distinct branches:

**Branch 1 — 4-byte opcode path (lines 157–182): No flag check.**

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => { return unknown_operator(...); }
    };
    return f(allocator, argument_list, max_cost, flags); // dispatched unconditionally
}
``` [1](#0-0) 

**Branch 2 — 1-byte opcode path (lines 248–249): `ENABLE_SECP_OPS` required.**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The flag `ENABLE_SECP_OPS` is defined as:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

The same underlying functions `op_secp256k1_verify` and `op_secp256r1_verify` (defined in `src/secp_ops.rs`) are reachable through both paths, but only the 1-byte path enforces the flag guard. [4](#0-3) 

`MEMPOOL_MODE` — the stricter mode used for mempool validation — does **not** include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [5](#0-4) 

This means that in `MEMPOOL_MODE`, secp operations are **still reachable** via the 4-byte opcode forms, while the 1-byte aliases (64, 65) are blocked. The two paths for the same operation have divergent validation requirements.

---

### Impact Explanation

Any caller that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` — including callers using `MEMPOOL_MODE` — will find that secp signature verification is still reachable via the 4-byte opcode encoding. This produces two concrete broken invariants:

1. **Flag-based access control bypass**: A node or validator that omits `ENABLE_SECP_OPS` to restrict secp operations cannot actually prevent them. An attacker-controlled CLVM program using opcode `0x13d61f00` or `0x1c3a8f00` will execute `op_secp256k1_verify` / `op_secp256r1_verify` unconditionally.

2. **Consensus / mempool divergence**: In `MEMPOOL_MODE`, the 1-byte secp aliases (64, 65) are rejected as unimplemented (because `NO_UNKNOWN_OPS` is set and `ENABLE_SECP_OPS` is absent), but the 4-byte secp opcodes succeed. A transaction using 4-byte secp opcodes passes mempool validation while a semantically identical transaction using 1-byte aliases is rejected — two programs with identical cryptographic semantics receive different treatment from the same dialect configuration.

---

### Likelihood Explanation

The 4-byte secp opcode forms (`0x13d61f00`, `0x1c3a8f00`) are valid, serializable CLVM atoms. Any attacker who can submit attacker-controlled CLVM bytes — the primary attack surface of this library — can trivially encode these opcodes. No special privileges, compromised nodes, or social engineering are required. The entry path is direct: `run_program()` → `apply_op()` → `ChiaDialect::op()` → 4-byte branch → unconditional secp dispatch. [6](#0-5) 

---

### Recommendation

**Short term**: Add the `ENABLE_SECP_OPS` flag check to the 4-byte secp opcode dispatch path, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => { return unknown_operator(allocator, o, argument_list, flags, max_cost); }
};
```

**Long term**: Adopt a single, unified dispatch table for all secp opcode encodings so that flag validation is applied consistently regardless of opcode byte length. Variation between the 4-byte and 1-byte dispatch paths for the same operation is a structural source of developer error and misconfiguration, directly analogous to the whitelist inconsistency described in the reference report.

---

The same cryptographic operation — secp256k1 signature verification — succeeds via the 4-byte opcode and fails via the 1-byte alias under equivalent flag configurations, confirming the inconsistent validation.

```rust
pub fn op_secp256r1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256R1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256r1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256r1_verify pubkey")?;
    let verifier = P1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256r1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is sha256 hash of message
    let msg = atom(a, msg, "secp256r1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256r1_verify sig")?;
    let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
