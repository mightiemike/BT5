### Title
Unchecked Cost Budget in `op_ash` / `op_lsh` Allows Forced Computation Beyond Remaining Budget - (File: `src/more_ops.rs`)

---

### Summary

`op_ash` and `op_lsh` in `src/more_ops.rs` accept a `max_cost` parameter but never call `check_cost` against it. Both operators perform the full bignum shift and heap allocation unconditionally before computing and returning the cost. An attacker-controlled CLVM program can force a node to execute expensive shift operations even when the remaining cost budget is effectively zero, obtaining computation without paying for it in advance.

---

### Finding Description

Every operator in `clvm_rs` receives a `max_cost: Cost` argument representing the remaining execution budget. Well-behaved operators call `check_cost` before or during expensive work so they can abort early. For example, `op_modpow` calls `check_cost` twice before the modular exponentiation runs, `op_sha256` calls it inside its hashing loop, and `op_concat` calls it per argument.

`op_ash` and `op_lsh` do neither:

```rust
// src/more_ops.rs line 918
pub fn op_ash(a: &mut Allocator, input: NodePtr, _max_cost: Cost, _flags: ClvmFlags) -> Response {
    let [n0, n1] = get_args::<2>(a, input, "ash")?;
    let (i0, l0) = int_atom(a, n0, "ash")?;
    let a1 = i32_atom(a, n1, "ash")?;
    if !(-65535..=65535).contains(&a1) {
        return Err(EvalErr::ShiftTooLarge(n1));
    }
    let v: Number = if a1 > 0 { i0 << a1 } else { i0 >> -a1 };  // shift runs unconditionally
    let l1 = limbs_for_int(&v);
    let r = a.new_number(v)?;                                      // allocation runs unconditionally
    let cost = ASHIFT_BASE_COST + ((l0 + l1) as Cost) * ASHIFT_COST_PER_BYTE;
    Ok(malloc_cost(a, cost, r))                                    // cost reported only after the fact
}
```

The parameter is named `_max_cost` (underscore prefix), confirming it is intentionally unused. `op_lsh` is identical in structure. [1](#0-0) [2](#0-1) 

The `run_program` loop does check cost after each operator returns, but only after the operator has already completed its work and returned:

```rust

The `max_cost` passed into `apply_op` and then forwarded to `dialect.op()` is `effective_max_cost - cost`, which is the remaining budget. `op_ash`/`op_lsh` receive this value and discard it entirely. [4](#0-3) 

Contrast with `op_modpow`, which calls `check_cost` before the expensive computation runs:

```rust

### Impact Explanation

An attacker submits a CLVM program containing `(ash <large_atom> 65535)` or `(lsh <large_atom> 65535)`. The shift amount is bounded to ±65535 bits by the range check, and the input atom is bounded by the allocator, but the combination still forces a bignum left-shift that produces a result atom up to ~8 KB in size. The node must:

1. Perform the full bignum shift (`i0 << 65535`) in the `num-bigint` library.
2. Allocate the result atom in the `Allocator` heap.
3. Only then compute and return the cost.

If the remaining budget at the point of invocation is 1 (or any value below the actual cost of the operation), the node still performs all of the above work before the post-operator cost check fires and rejects the transaction. The attacker obtains CPU time and heap allocation for free relative to the budget they declared.

This is the direct analog to the reported "Unchecked Fund Provision" pattern: the resource (computation budget) is accepted as a parameter but never validated before the expensive action is taken. The action (shift + allocation) happens unconditionally, and the check occurs only after the resource has been consumed.

**Severity**: Medium. The shift is bounded (max ~8 KB output), so the per-invocation waste is bounded. However, a program can chain many such calls, and in consensus mode (where `allow_unknown_ops` is true and cost limits are enforced globally) this allows an attacker to force disproportionate CPU and allocator work relative to the declared cost budget, degrading node throughput.

---

### Likelihood Explanation

The entry path is direct: any CLVM program submitted to the network (e.g., as a puzzle or generator) can invoke `ash` or `lsh` with attacker-controlled operands. No special privilege is required. The operators are part of the standard opcode table dispatched by `ChiaDialect` and `RuntimeDialect`. The `_max_cost` naming convention confirms this is a pre-existing design gap, not a transient bug. [6](#0-5) 

---

### Recommendation

Add a pre-execution `check_cost` call in both `op_ash` and `op_lsh` using a conservative upper-bound cost estimate before performing the shift. The upper bound can be computed from the input size `l0` and the maximum possible output size (input bytes + shift_bits/8):

```rust
pub fn op_ash(a: &mut Allocator, input: NodePtr, max_cost: Cost, _flags: ClvmFlags) -> Response {
    let [n0, n1] = get_args::<2>(a, input, "ash")?;
    let (i0, l0) = int_atom(a, n0, "ash")?;
    let a1 = i32_atom(a, n1, "ash")?;
    if !(-65535..=65535).contains(&a1) {
        return Err(EvalErr::ShiftTooLarge(n1));
    }
    // Guard before the expensive shift
    let max_output_bytes = l0 + (a1.unsigned_abs() as usize / 8) + 1;
    let upper_cost = ASHIFT_BASE_COST + (2 * max_output_bytes as Cost) * ASHIFT_COST_PER_BYTE;
    check_cost(upper_cost, max_cost)?;
    // ... rest unchanged
```

Apply the same pattern to `op_lsh`.

### Proof of Concept

```python
# Pseudocode: craft a CLVM program that forces expensive ash with near-zero budget
# (ash <256-byte atom> 65535)
# Expected: node performs full bignum shift before discovering cost exceeded
import clvm_rs

program = assemble("(ash 0x" + "ff" * 256 + " 65535)")
env = assemble("()")

# Run with a budget of 1 -- op_ash still executes the full shift
try:
    result = run_program(program, env, max_cost=1)
except CostExceeded:
    pass  # error fires AFTER the shift and allocation completed
```

The node performs the full `num-bigint` left-shift of a 256-byte integer by 65535 bits (producing a ~8 KB result) and allocates it in the `Allocator` heap before the `CostExceeded` error is raised by the post-operator check in `run_program`. [1](#0-0) [2](#0-1) [7](#0-6)
