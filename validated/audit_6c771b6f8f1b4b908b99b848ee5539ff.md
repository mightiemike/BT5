### Title
Missing Upper-Bound Validation on `Tier.ExitDuration` Permanently Locks User Funds - (File: x/tieredrewards/types/tier.go)

### Summary
`Tier.Validate()` enforces only a lower-bound (`ExitDuration > 0`) on the exit commitment duration. There is no upper-bound check. A governance proposal can set `ExitDuration` to `math.MaxInt64` nanoseconds (≈ 292 years). Any user who subsequently calls `MsgTriggerExitFromTier` on a position in that tier has their `Position.ExitUnlockAt` stamped to `blockTime + 292 years`, permanently trapping their staked tokens in the module's per-position delegator account.

### Finding Description

`Tier.Validate()` in `x/tieredrewards/types/tier.go` only rejects non-positive durations:

```go
if t.ExitDuration <= 0 {
    return fmt.Errorf("exit duration must be positive")
}
``` [1](#0-0) 

No maximum is enforced. `ExitDuration` is a Go `time.Duration` (int64 nanoseconds), so any value up to `math.MaxInt64` (≈ 292 years) passes validation. This check is the only guard called by `SetTier` (invoked from both `AddTier` and `UpdateTier` in the message server): [2](#0-1) [3](#0-2) 

When a user calls `MsgTriggerExitFromTier`, the handler reads the **current** tier's `ExitDuration` and stamps it onto the position:

```go
tier, err := ms.getTier(ctx, pos.TierId)
...
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
``` [4](#0-3) 

`TriggerExit` writes `ExitUnlockAt` directly onto the position record:

```go
func (p *Position) TriggerExit(blockTime time.Time, duration time.Duration) {
    p.ExitTriggeredAt = blockTime
    p.ExitUnlockAt = blockTime.Add(duration)
}
``` [5](#0-4) 

`WithdrawFromTier` and `ExitTierWithDelegation` both gate on `CompletedExitLockDuration`, which checks `blockTime >= ExitUnlockAt`: [6](#0-5) 

With `ExitUnlockAt` set 292 years in the future, neither exit path is reachable.

The only escape valve is `MsgClearPosition`, which resets exit timestamps. However, `ClearPosition` is explicitly blocked when the tier has `CloseOnly = true`: [7](#0-6) 

A governance proposal that simultaneously sets `ExitDuration = math.MaxInt64` and `CloseOnly = true` leaves all positions that trigger exit with no recovery path whatsoever. Even without `CloseOnly`, a user who clears their exit and re-triggers it while the bad `ExitDuration` is still in effect is immediately re-trapped.

### Impact Explanation

The corrupted on-chain value is `Position.ExitUnlockAt`. Once stamped with a 292-year future timestamp, the position's staked tokens — held by the position's dedicated `DelegatorAddress` — are permanently inaccessible to the owner. The user cannot call `WithdrawFromTier`, `ExitTierWithDelegation`, or (if `CloseOnly`) `ClearPosition`. The staked principal is not lost from the chain, but it is functionally unrecoverable by the owner for the lifetime of the chain. [8](#0-7) 

### Likelihood Explanation

Requires a governance proposal to pass with an extreme `ExitDuration`. This is the same low-but-non-zero likelihood as the external report's Treasury `delay` finding: a miscalculation, unit confusion (seconds vs. nanoseconds), or a malicious but passing proposal is sufficient. The `ExitDuration` field is a raw protobuf `Duration`, making a unit-confusion mistake (e.g., treating the field as seconds and supplying `31536000000` intending "1 year in seconds" but actually encoding 31 billion nanoseconds ≈ 31 seconds — or the reverse) realistic.

### Recommendation

Add an upper-bound check in `Tier.Validate()` in `x/tieredrewards/types/tier.go`:

```go
const MaxExitDuration = 5 * 365 * 24 * time.Hour // 5 years

if t.ExitDuration > MaxExitDuration {
    return fmt.Errorf("exit duration must not exceed %s: got %s", MaxExitDuration, t.ExitDuration)
}
```

Apply the same check in genesis validation and in any migration handler that sets tiers.

### Proof of Concept

1. Governance submits and passes `MsgUpdateTier` with `ExitDuration = math.MaxInt64` (≈ 292 years) and `CloseOnly = true` for an existing tier.
2. `Tier.Validate()` passes because `math.MaxInt64 > 0`.
3. `SetTier` stores the tier.
4. User calls `MsgTriggerExitFromTier` on their position in that tier.
5. `TriggerExit` sets `pos.ExitUnlockAt = blockTime + 292 years`.
6. User attempts `MsgExitTierWithDelegation` → rejected: `CompletedExitLockDuration` returns false.
7. User attempts `MsgWithdrawFromTier` → rejected for the same reason.
8. User attempts `MsgClearPosition` → rejected: tier is `CloseOnly`.
9. User's staked tokens remain locked in the position's `DelegatorAddress` indefinitely. [9](#0-8) [10](#0-9)

func (t Tier) Validate() error {
	if t.Id == 0 {
		return ErrInvalidTierID
	}

	if t.ExitDuration <= 0 {
		return fmt.Errorf("exit duration must be positive")
	}

	if t.BonusApy.IsNil() {
		return fmt.Errorf("bonus apy cannot be nil")
	}

	if t.BonusApy.IsNegative() {
		return fmt.Errorf("bonus apy cannot be negative: %s", t.BonusApy)
	}

	// Cap BonusApy at 100% to prevent governance from draining the rewards pool.
	if t.BonusApy.GT(math.LegacyOneDec()) {
		return fmt.Errorf("bonus apy must not exceed 1.0 (100%%): got %s", t.BonusApy)
	}

	if t.MinLockAmount.IsNil() {
		return fmt.Errorf("min lock amount cannot be nil")
	}

	if t.MinLockAmount.IsNegative() {
		return fmt.Errorf("min lock amount cannot be negative: %s", t.MinLockAmount)
	}

	return nil
}

}

### CloseOnly Tiers

When a tier is marked `CloseOnly` by governance, these messages are **blocked**: LockTier, CommitDelegation, AddToPosition, TierRedelegate, ClearPosition. **Allowed**: TriggerExit, TierUndelegate, WithdrawFromTier, ClaimTierRewards, ExitTierWithDelegation. This lets governance sunset a tier while existing positions close out.
