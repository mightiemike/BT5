Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Position `owner` Instead of Actual Depositor, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the actual depositor address (first parameter) and instead validates the position `owner` (second parameter) against the allowlist. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts any non-zero `owner` while using `msg.sender` as the token payer, any unprivileged actor can bypass the pool admin's deposit allowlist by specifying an already-allowlisted address as `owner` while paying the tokens themselves.

## Finding Description

**Root cause — wrong parameter checked:**

`DepositAllowlistExtension.beforeAddLiquidity` (lines 32–42) discards the first argument (the actual `sender`) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`msg.sender` here is the pool (the extension caller), so the mapping lookups are `allowAllDepositors[pool]` and `allowedDepositor[pool][owner]`. The actual depositor — the address paying tokens — is never consulted.

**Pool call site confirms argument order:**

`MetricOmmPool.addLiquidity` (line 191) passes the real caller first and `owner` second:
```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

**Exploit path via `MetricOmmPoolLiquidityAdder`:**

`addLiquidityExactShares(pool, owner, ...)` (lines 56–68) accepts any non-zero `owner` (validated only by `_validateOwner` which rejects `address(0)`) and stores `msg.sender` as the token payer:

```solidity
_validateOwner(owner);          // only rejects address(0)
return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
```

Inside `_addLiquidity` (line 194–195), the pool is called with `positionOwner` as the `owner` argument:
```solidity
IMetricOmmPoolActions(pool).addLiquidity(positionOwner, salt, deltas, abi.encode(KIND_PAY), extensionData)
```

The pool then calls `_beforeAddLiquidity(MetricOmmPoolLiquidityAdder, allowlistedAddress, ...)`. The extension checks `allowedDepositor[pool][allowlistedAddress]` — which is `true` — and passes. The attacker's tokens are pulled via the callback; the position is minted to `allowlistedAddress`.

**Existing guards are insufficient:**
- `_validateOwner` only rejects `address(0)`.
- The extension never reads the first (sender) argument.
- No other hook or pool-level check enforces that the payer is allowlisted.

## Impact Explanation
The pool admin's deposit allowlist — an explicit access-control boundary set via `setAllowedToDeposit` / `setAllowAllDepositors` — is fully bypassed by any unprivileged caller. This constitutes an admin-boundary break: an unprivileged path circumvents a pool-admin-configured restriction. The attacker can force token deposits into a restricted pool, crediting positions to allowlisted addresses without those addresses initiating the action, violating the intended deposit gating invariant.

## Likelihood Explanation
Exploitation requires only: (1) knowledge of at least one allowlisted address for the target pool, and (2) sufficient token balance and approval on `MetricOmmPoolLiquidityAdder`. Both conditions are trivially satisfiable by any on-chain observer. The attack is repeatable, permissionless, and requires no privileged setup.

## Recommendation
Replace the discarded first parameter with a named `sender` variable and check it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns with the NatSpec comment ("Gates `addLiquidity` by depositor address") and the `isAllowedToDeposit` view function, which also keys on `depositor`.

## Proof of Concept
```solidity
// Preconditions:
// - pool has DepositAllowlistExtension configured
// - allowedDepositor[pool][alice] = true  (alice is allowlisted)
// - attacker (bob) is NOT allowlisted
// - bob has approved MetricOmmPoolLiquidityAdder for token0/token1

// Attack:
vm.prank(bob);
liquidityAdder.addLiquidityExactShares(
    pool,
    alice,      // owner = allowlisted address → passes extension check
    salt,
    deltas,
    max0,
    max1,
    extensionData
);
// Result: bob's tokens deposited, position credited to alice, allowlist bypassed
```
