### Title
Deprecated `verify_transaction_inclusion` Remains Callable Without Coinbase Guard, Enabling 64-Byte Merkle Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function is still reachable by any unprivileged NEAR caller. It performs no coinbase Merkle proof validation, meaning the guard that blocks the known 64-byte transaction forgery attack is simply absent — not activated. Any caller can bypass the protection added in `verify_transaction_inclusion_v2` by calling the old entry point directly, obtaining a `true` proof result for a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions). It does so by first verifying a coinbase Merkle proof before delegating to the old function: [1](#0-0) 

The coinbase proof check is the guard: [2](#0-1) 

However, the original `verify_transaction_inclusion` is still a live, public, `#[pause]`-gated (but not access-controlled) contract method. It is not blocked, not removed, and not made to always revert: [3](#0-2) 

The function's own warning acknowledges the risk but defers responsibility to callers: [4](#0-3) 

The analog to the Uniswap report is exact:

| Uniswap v4 | BTC Light Client |
|---|---|
| `beforeDonate: false` → hook call skipped | `verify_transaction_inclusion` deprecated but not blocked → coinbase check skipped |
| Donations proceed, fee growth inflated | Forged proofs accepted, `true` returned |
| Fix: activate hook to always revert | Fix: make deprecated function always revert |

The proof result — the return value of `verify_transaction_inclusion` — is the corrupted canonical value. Any recipient contract that consumes this result to authorize a cross-chain action (e.g., a bridge releasing funds) receives a `true` for a fabricated transaction.

---

### Impact Explanation

An attacker who knows the Merkle tree structure of any mainchain block can craft a 64-byte value that is simultaneously a valid internal Merkle node hash and a plausible `tx_id`. Supplying this as `args.tx_id` with a valid Merkle path to the block's `merkle_root` causes `verify_transaction_inclusion` to return `true` for a transaction that was never broadcast or mined. Recipient contracts that rely on this function's return value to release assets or record state transitions are directly exploitable. [5](#0-4) 

---

### Likelihood Explanation

The entry point is public and requires no privileged role. The `#[pause]` decorator means it is active by default and can only be disabled by a `PauseManager`. The 64-byte forgery technique is well-documented and tooling exists to construct such proofs. Any NEAR account can call `verify_transaction_inclusion` directly, bypassing `verify_transaction_inclusion_v2` entirely.

---

### Recommendation

Make `verify_transaction_inclusion` always revert, mirroring the Uniswap report's recommended mitigation of activating the hook to always revert:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] _args: ProofArgs) -> bool {
    env::panic_str("verify_transaction_inclusion is disabled; use verify_transaction_inclusion_v2");
}
```

This closes the bypass path while preserving the v2 entry point for legitimate callers.

---

### Proof of Concept

1. Identify any block `B` on the contract's mainchain with at least two transactions.
2. Locate an internal Merkle node `N` at depth ≥ 1 whose 32-byte hash, when interpreted as a transaction hash, is the value to forge.
3. Construct a valid Merkle path from `N` to `B.merkle_root` (this path is shorter by one level than a leaf proof).
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash)
   - `tx_block_blockhash = B`
   - `tx_index` = the index corresponding to `N`'s position
   - `merkle_proof` = the path from `N` to `merkle_root`
   - `confirmations` = any value ≤ current depth
5. The function computes `compute_root_from_merkle_proof(N, index, proof)` which equals `B.merkle_root`, and returns `true` — confirming a transaction that does not exist. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
