Looking at the exact code flow in `processEventsAndClaimBonus` and `slashRedelegationPosition`, I need to determine whether the checkpoint advancement and the pool-insufficient error interact in a way that permanently loses rewards.

The bug is real and concrete. Here is the exact state-change sequence:

**`processEventsAndClaimBonus` (claim_rewards.go:215 → 230):**
- Line 215 advances `pos.LastBonusAccrual` to `blockTime` **in-memory on the pointer** before the pool check
- Line 230 checks the pool and returns `ErrInsufficientBonusPool` — but the checkpoint mutation already happened

**`slashRedelegationPosition` (slash.go:54-77):**
- Line 54 passes `&pos` (pointer), so the mutation from `processEventsAndClaimBonus` is visible
- Lines 56-63 swallow `ErrInsufficientBonusPool` and continue
- Line 77 calls `k.setPositionWithState(ctx, pos, nil)` — persisting the position with the already-advanced `LastBonusAccrual` and no bonus paid

### Title
Checkpoint Advancement Before Pool Check in `slashRedelegationPosition` Causes Permanent Bonus Reward Loss on Partial Redelegation Slash — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary

When `BeforeRedelegationSlashed` fires and the bonus pool is empty, `processEventsAndClaimBonus` advances the position's `LastBonusAccrual` checkpoint to the current block time **before** checking pool sufficiency. The error is then silently swallowed in `slashRedelegationPosition`, and the position is persisted with the advanced checkpoint. The pre-slash bonus period is permanently unclaimable.

### Finding Description

The root cause is a sequencing error in `processEventsAndClaimBonus`:

```go
// claim_rewards.go:215-231
applyBonusAccrualCheckpoint(&pos.Position, blockTime)  // checkpoint advanced HERE
pos.UpdateLastKnownBonded(bonded)

if totalBonus.IsZero() {
    return sdk.NewCoins(), nil
}
// ...
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err  // ErrInsufficientBonusPool returned AFTER checkpoint mutation
}
``` [1](#0-0) 

The function mutates `pos.LastBonusAccrual` (via `applyBonusAccrualCheckpoint`) and `pos.LastKnownBonded` before it checks whether the pool can cover the computed bonus. When the pool is insufficient, it returns an error — but the in-memory `pos` struct (passed by pointer) already has its checkpoints advanced.

In `slashRedelegationPosition`, this error is deliberately swallowed:

```go
// slash.go:54-64
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // continues — does NOT return
    } else {
        return err
    }
}
``` [2](#0-1) 

Execution then falls through to the partial-slash branch, which persists the position with the already-advanced checkpoint:

```go
// slash.go:76-77
pos.Delegation.Shares = pos.Delegation.Shares.Sub(sharesToUnbond)
return k.setPositionWithState(ctx, pos, nil)
``` [3](#0-2) 

The position's `LastBonusAccrual` is now set to the slash block time, with zero bonus paid. All bonus accrued between the previous checkpoint and the slash block is permanently lost — the next `processEventsAndClaimBonus` call will start from the slash block time, not from the original `LastBonusAccrual`.

Note: the **full-slash** branch (`fullSlash == true`) is not affected because it calls `pos.ClearBonusCheckpoints()` which resets the checkpoints regardless. The bug is specific to **partial** redelegation slashes. [4](#0-3) 

### Impact Explanation

A position owner who has a pending redelegation entry (created via `MsgTierRedelegate`) loses all bonus rewards accrued since their last checkpoint when:
1. A partial redelegation slash fires (`BeforeRedelegationSlashed`)
2. The bonus pool is empty at that moment

The rewards are **permanently** lost — not deferred. The `LastBonusAccrual` checkpoint is advanced to the slash block time with no payment, so no future claim can recover the pre-slash period.

An adversary who can drain the pool (by legitimately claiming their own large bonus rewards via `MsgClaimTierRewards`, or by timing the attack around `BeginBlocker` top-up draining the pool) can cause targeted position owners to forfeit their pre-slash bonus. The attacker does not receive the victim's rewards directly, but causes their permanent forfeiture. [5](#0-4) 

### Likelihood Explanation

The pool being empty is a realistic operational condition — `BeginBlocker` drains it for base-reward top-up every block when there is a shortfall, and legitimate `MsgClaimTierRewards` calls drain it further. A deliberate attacker needs only to submit a `MsgClaimTierRewards` transaction in a block before the evidence block to drain the pool. Redelegation slashes occur whenever a validator with a pending redelegation entry is slashed for equivocation (double-signing), which is a known and observable on-chain event. [6](#0-5) 

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the pool sufficiency check and the bank send succeed. Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, explicitly reset the position's bonus checkpoints to their pre-call values (snapshot them before calling `processEventsAndClaimBonus`).

The simplest fix in `processEventsAndClaimBonus` is to defer checkpoint advancement until after the bank send:

```go
// After SendCoinsFromModuleToAccount succeeds:
applyBonusAccrualCheckpoint(&pos.Position, blockTime)
pos.UpdateLastKnownBonded(bonded)
```

For the zero-bonus early-return path (line 219-221), the checkpoint advancement is safe because no pool interaction occurs and the caller expects the checkpoint to advance. [7](#0-6) 

### Proof of Concept

Keeper test (Go, unmodified test setup):

```go
func (s *KeeperSuite) TestSlashRedelegation_EmptyPool_PermanentBonusLoss() {
    lockAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
    _, bondDenom := s.getStakingData()
    // Fund pool, then drain it — do NOT leave any balance
    s.fundRewardsPool(sdkmath.NewInt(1), bondDenom)
    // drain by claiming (or just don't fund at all)

    pos, _, unbondingID := s.setupRedelegatingPosition(lockAmount)
    owner := sdk.MustAccAddressFromBech32(pos.Owner)
    originalCheckpoint := pos.LastBonusAccrual

    // Advance time so bonus accrues
    s.ctx = s.ctx.WithBlockTime(s.ctx.BlockTime().Add(30 * 24 * time.Hour))

    balBefore := s.app.BankKeeper.GetBalance(s.ctx, owner, bondDenom)

    // Partial slash with empty pool
    sharesToUnbond := pos.Delegation.Shares.Quo(sdkmath.LegacyNewDec(2))
    err := s.keeper.Hooks().BeforeRedelegationSlashed(s.ctx, unbondingID, sharesToUnbond)
    s.Require().NoError(err) // hook does not error

    balAfter := s.app.BankKeeper.GetBalance(s.ctx, owner, bondDenom)
    // Owner received zero bonus
    s.Require().True(balAfter.Amount.Equal(balBefore.Amount), "owner received zero bonus")

    // But checkpoint was advanced — pre-slash period permanently lost
    updated, _ := s.keeper.GetPositionState(s.ctx, pos.Id)
    s.Require().True(updated.LastBonusAccrual.After(originalCheckpoint),
        "checkpoint advanced despite zero payment — rewards permanently lost")
}
```

This test will pass (demonstrating the bug) against the current unmodified code.
