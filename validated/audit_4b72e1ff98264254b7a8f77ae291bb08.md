### Title
`DepositAllowlistExtension` `beforeAddLiquidity` Guard Checks `sender` While LP Position Is Minted to Arbitrary `owner` — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address that receives the minted LP position shares, while passing `msg.sender` as `sender` to the `beforeAddLiquidity` extension hook. The `DepositAllowlistExtension` guard fires against `sender` (the router/caller), not `owner` (the actual beneficiary of the position). An allowlisted intermediary can therefore deposit on behalf of any non-allowlisted `owner`, bypassing the deposit allowlist entirely.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no restriction that it equal `msg.sender`: [1](#0-0) 

The hook is fired as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [2](#0-1) 

Inside `ExtensionCalling._beforeAddLiquidity`, the call to the extension encodes `sender = msg.sender` and `owner = <arbitrary>`: [3](#0-2) 

The `DepositAllowlistExtension` receives both values but, following the natural guard pattern, checks `sender` (the actor initiating the call). The LP shares, however, are credited to `owner` via `_positionBinShares` keyed by `(owner, salt, binIdx)`: [4](#0-3) 

Contrast this with `removeLiquidity`, which enforces `msg.sender == owner` before the hook fires: [5](#0-4) 

No equivalent guard exists on `addLiquidity`. The asymmetry means the allowlist gate is applied to the wrong identity for the add-liquidity path.

---

### Impact Explanation

A non-allowlisted address cannot call `addLiquidity` directly (the extension would reject `sender`). However, any allowlisted address (e.g., a public router, a whitelisted contract, or the attacker's own allowlisted EOA) can call `addLiquidity(owner = <non-allowlisted address>, ...)`. The LP position — and all future `removeLiquidity` proceeds — accrue to the non-allowlisted `owner`. The deposit allowlist invariant is broken: the protocol believes it controls who holds LP positions, but non-allowlisted parties gain full LP ownership and can later withdraw funds directly (since `removeLiquidity` only checks `msg.sender == owner`, which the non-allowlisted owner satisfies).

**Impact class:** Broken core pool functionality / admin-boundary break — allowlist guard bypassed by unprivileged path, resulting in unauthorized LP positions and eventual unauthorized withdrawal of pool assets.

---

### Likelihood Explanation

- Any allowlisted address (including a public periphery router) can trigger this with a single `addLiquidity` call.
- No special permissions, flash loans, or oracle manipulation are required.
- The attacker only needs to know the pool address and have an allowlisted `msg.sender` (trivially achievable if the router is allowlisted, which is the standard deployment pattern).

---

### Recommendation

In `addLiquidity`, add the same ownership check that `removeLiquidity` enforces, or change the extension hook to pass `owner` as the identity to be allowlist-checked (not `sender`):

```solidity
// Option A: restrict owner to msg.sender (like removeLiquidity)
if (msg.sender != owner) revert NotPositionOwner();

// Option B: pass owner as the guarded identity in the hook
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
// and have DepositAllowlistExtension check owner, not sender
```

The `DepositAllowlistExtension` should explicitly validate `owner` (the LP beneficiary) rather than `sender` (the transaction initiator), since the economic benefit of the position flows to `owner`.

---

### Proof of Concept

1. Deploy a pool with `DepositAllowlistExtension` configured; allowlist only `routerAddress`.
2. `attackerAddress` (not allowlisted) cannot call `addLiquidity` directly — extension reverts.
3. `routerAddress` calls:
   ```solidity
   pool.addLiquidity(
       owner = attackerAddress,   // non-allowlisted beneficiary
       salt  = 0,
       deltas = <valid bins/shares>,
       callbackData = ...,
       extensionData = ...
   );
   ```
4. Extension fires with `sender = routerAddress` (allowlisted) → passes.
5. LP shares are minted to `attackerAddress` in `_positionBinShares`.
6. `attackerAddress` later calls `pool.removeLiquidity(owner = attackerAddress, ...)` directly — `msg.sender == owner` check passes, no extension blocks removal — and withdraws pool tokens.

The allowlist is fully bypassed; `attackerAddress` gains and liquidates an LP position it was never authorized to hold.

  function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
  ) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
    if (deltas.binIdxs.length == 0) return (0, 0);
    if (deltas.binIdxs.length != deltas.shares.length) revert LiquidityDeltaLengthMismatch();
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);

    );
    _afterAddLiquidity(msg.sender, owner, salt, deltas, amount0Added, amount1Added, extensionData);
  }
