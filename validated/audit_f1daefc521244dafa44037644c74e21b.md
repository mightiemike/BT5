### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Guard — (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase proof. However, the original `verify_transaction_inclusion` (v1) remains a live, publicly callable NEAR contract method. Any unprivileged caller can invoke v1 directly, bypassing the coinbase proof check entirely and obtaining a `true` return value for a forged transaction inclusion claim.

### Finding Description
`verify_transaction_inclusion_v2` enforces a coinbase Merkle proof check before delegating to v1: [1](#0-0) 

The coinbase guard is: [2](#0-1) 

This check exists **only** in v2. The v1 function is marked `#[deprecated]` but carries no runtime access restriction — it is still a fully public NEAR contract method: [3](#0-2) 

`#[deprecated]` in Rust is a compile-time lint, not a runtime guard. On NEAR, the method remains callable by any account. The v1 function performs only a standard Merkle root comparison with no coinbase proof: [4](#0-3) 

The `ProofArgs` struct accepted by v1 contains no `coinbase_tx_id` or `coinbase_merkle_proof` fields: [5](#0-4) 

### Impact Explanation
The 64-byte transaction Merkle proof forgery (documented in the v2 deprecation notice at line 268) allows an attacker to supply a `tx_id` that is actually an internal Merkle tree node rather than a real transaction hash. Because `compute_root_from_merkle_proof` operates purely on hashes without distinguishing leaf nodes from internal nodes, a crafted 64-byte value can produce a valid Merkle path to the block's `merkle_root`. Calling v1 directly returns `true` for this forged proof. Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a financial action (e.g., releasing bridged funds upon claimed Bitcoin payment) can be deceived into accepting a transaction that never existed on-chain. [6](#0-5) 

### Likelihood Explanation
The entry path requires no privileges: any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte forgery technique is publicly documented (referenced in the contract's own comments at line 271). Constructing a valid internal-node proof requires only knowledge of the target block's Merkle tree, which is public Bitcoin data. Likelihood is **medium-high** for any deployment where downstream contracts consume the v1 return value.

### Recommendation
Remove `verify_transaction_inclusion` as a public contract method, or add a runtime `env::panic_str` / `require!(false, "use v2")` guard at its entry point so it cannot be called externally. All callers should be migrated to `verify_transaction_inclusion_v2`. Alternatively, apply `#[private]` to v1 so it is only callable as an internal function from v2, which is its only legitimate remaining use.

### Proof of Concept
1. Identify a Bitcoin block on the main chain tracked by the contract. Obtain its full Merkle tree.
2. Select any internal node `N` at depth ≥ 1 (a 32-byte hash). Construct a valid Merkle proof from `N` to the block's `merkle_root` using the sibling hashes at each level above `N`.
3. Call `verify_transaction_inclusion` with:
   - `tx_id` = `N` (the internal node hash, treated as a fake transaction ID)
   - `tx_block_blockhash` = the real block hash (on main chain, satisfying the `mainchain_header_to_height` check)
   - `tx_index` = the position of `N` in the tree at its level
   - `merkle_proof` = the sibling hashes from `N` up to the root
   - `confirmations` = any value ≤ `gc_threshold`
4. `compute_root_from_merkle_proof(N, index, proof)` returns the block's real `merkle_root`, so the function returns `true`.
5. The coinbase proof check in v2 is never reached because v1 was called directly. [7](#0-6)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
