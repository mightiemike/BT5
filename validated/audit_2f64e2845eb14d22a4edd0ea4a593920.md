Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` receives the `sender` argument forwarded verbatim from `MetricOmmPool.swap(msg.sender, ...)`, which is the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, that caller is the router contract, not the end user. A pool admin who allowlists the router so that their permitted users can trade through it inadvertently grants swap access to every user on the router, collapsing per-user access control to a per-router boundary.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to extensions.**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

**Step 2 — Extension receives that value as `sender`.**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (used as the mapping namespace) and `sender` is whatever was passed in: [2](#0-1) 

**Step 3 — Router is the `msg.sender` the pool sees.**

`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making the router the `msg.sender` observed by the pool: [3](#0-2) 

The same holds for every hop in `exactInput`: [4](#0-3) 

And for intermediate hops in `exactOutput`, where `_exactOutputIterateCallback` calls `pool.swap()` with the router as the effective caller: [5](#0-4) 

**Root cause:** The extension has no mechanism to recover the original EOA/contract that initiated the router call. The only identity it can gate is the router contract itself. A pool admin who wants their allowlisted users to be able to trade through the router must add `allowedSwapper[pool][router] = true`. The moment they do, every user on the router — allowlisted or not — passes the check, because the check only sees the router address.

**Existing guards are insufficient:** `allowAllSwappers` is a separate escape hatch and does not mitigate this. There is no mechanism in the extension or the pool to thread the original `tx.origin` or a user-supplied identity through the call in a trustworthy way.

## Impact Explanation

Any user can bypass a pool's swap allowlist by routing through `MetricOmmSimpleRouter` once the pool admin has allowlisted the router. Restricted pools — KYC-gated, institutional-only, or whitelist-only — lose their access control entirely for router-mediated swaps. This breaks the core invariant of `SwapAllowlistExtension` ("Gates `swap` by swapper address, per pool") and constitutes broken core pool functionality causing loss of the intended access-control guarantee. This directly matches the "Allowlist path" Smart Audit Pivot: allowlist checks must cover the exact actor/action intended and cannot be bypassed through the router.

## Likelihood Explanation

The scenario is realistic and repeatable. Any pool admin who deploys `SwapAllowlistExtension` and wants their permitted users to be able to use the protocol's own router faces a forced choice: either block the router entirely (making the router unusable for their pool) or allowlist the router (opening the pool to all users). An unprivileged attacker needs only to call `MetricOmmSimpleRouter.exactInputSingle` (or any multi-hop variant) targeting the pool. No special privileges, flash loans, or oracle manipulation are required.

## Recommendation

The extension must verify the actual end user, not the intermediary. Two approaches:

1. **Pass the original initiator through `extensionData`:** Have the router encode `msg.sender` (the user) into `extensionData` and have `SwapAllowlistExtension` decode and verify it. This requires the router to be trusted to supply honest data, which can be enforced by a router registry in the factory.

2. **Maintain a router registry and check `tx.origin` when `sender` is a known router:** If `sender` is a registered router, fall back to checking `allowedSwapper[pool][tx.origin]`. This is simpler but couples the extension to `tx.origin` semantics.

Either fix must be applied consistently across `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` (including the recursive callback path).

## Proof of Concept

```solidity
// 1. Pool admin deploys pool with SwapAllowlistExtension.
// 2. Pool admin allowlists alice: allowedSwapper[pool][alice] = true
// 3. Pool admin allowlists the router so alice can use it:
//    allowedSwapper[pool][router] = true
// 4. Bob (not allowlisted) calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: restrictedPool,
    recipient: bob,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    tokenIn: token0,
    deadline: block.timestamp,
    extensionData: ""
}));
// pool.swap() is called with msg.sender == router
// SwapAllowlistExtension checks allowedSwapper[pool][router] == true → passes
// Bob's swap succeeds despite not being on the allowlist.
```
