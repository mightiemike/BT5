Audit Report

## Title
`OracleValueStopLossExtension` Stop-Loss Bypassed via Stale `lastDecayTs` When Bin Transitions from Empty to Active — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

When a bin's `totalShares` is zero, `_afterSwapOracleStopLoss` skips it entirely via `continue`, leaving `highWatermarks[pool_][binIdx].lastDecayTs` frozen at the timestamp set by the last admin call or last active swap. If a non-zero `decayPerSecondE8` is configured and enough time elapses while the bin is empty, the watermark decays to zero on the first post-reactivation swap. With `hwm = 0`, `_applyWatermark` unconditionally returns `breached = false`, allowing any swap through the bin to proceed regardless of the actual oracle-derived metric value.

## Finding Description

**Root cause — empty-bin skip without timestamp update:**

In `_afterSwapOracleStopLoss` (lines 236–242), the loop skips bins with `totalShares == 0`:

```solidity
if (totalShares == 0) continue;
```

`_checkAndUpdateWatermarks` — the only function that writes `hwmS.lastDecayTs` during normal swap flow — is therefore never reached for empty bins. The only other writer is `executeOracleStopLossHighWatermarks` (line 174), which is an admin-only call.

**Decay calculation uses raw elapsed time:**

`_checkAndUpdateWatermarks` computes:
```solidity
uint256 dt = block.timestamp - hwmS.lastDecayTs;
```
If the bin was empty for `D` seconds, `dt` on the first post-reactivation swap equals `D` plus any prior elapsed time, inflating the decay factor.

**`_decayed` collapses to zero:**

```solidity
uint256 factor = ratePerSecondE8 * dt;
if (factor >= E8) return 0;
```
At `decayPerSecondE8 = 58`, `factor >= 1e8` after `ceil(1e8 / 58) ≈ 1,724,138 s ≈ 20 days`.

**`_applyWatermark` with `hwm = 0` never reports a breach:**

```solidity
if (metric >= hwm) return (metric, false);
```
Since `metric` is `uint256`, `metric >= 0` is always true. The function returns `(metric, false)`, setting the new watermark to the current metric and reporting no breach. The direction-aware revert in `_checkAndUpdateWatermarks` is never reached.

**Exploit flow:**
1. Admin calls `executeOracleStopLossHighWatermarks` for bin 1 with `decayPerSecondE8 = 58`; `lastDecayTs = T0`.
2. All LPs remove liquidity from bin 1 (`totalShares → 0`).
3. Swaps cross adjacent bins for ≥ 20 days; bin 1 is skipped each iteration; `lastDecayTs` stays at `T0`.
4. LPs re-add liquidity to bin 1.
5. Attacker swaps through bin 1: `dt ≥ 1,724,138 s`, `_decayed(hwm, 58, dt) = 0`, `_applyWatermark(metric, 0, floor) → (metric, false)`, no `OracleStopLossTriggered` revert.
6. Swap executes; attacker extracts token value from bin 1 below the admin-configured drawdown floor.

**Existing guards are insufficient:** The `drawdown == 0` early-exit guard (line 217) does not apply when drawdown is configured. The timelock on admin watermark resets does not help because the bypass occurs before the admin can react.

## Impact Explanation

The stop-loss is the primary on-chain mechanism preventing LP value from being extracted below the configured drawdown floor. When the watermark collapses to zero, `_applyWatermark` always returns `breached = false`, so `OracleStopLossTriggered` is never emitted and the swap is never reverted. An attacker can execute a swap through the reactivated bin at an oracle price that would otherwise be blocked, extracting LP principal below the floor. This is a direct loss of user principal matching the "bad-price execution" and "direct loss of LP assets" allowed impact categories.

## Likelihood Explanation

All preconditions are reachable through normal, unprivileged pool operation: LPs routinely rebalance by removing and re-adding liquidity; swaps naturally cross adjacent bins. The only privileged prerequisite is the admin having configured a non-zero `decayPerSecondE8`, which is the intended production configuration. The attacker requires no special access — only the ability to submit a swap transaction at the right moment. At the example rate of 58 (~5 %/day), the threshold is ~20 days of bin inactivity, a realistic window in any pool with episodic liquidity.

## Recommendation

Inside `_afterSwapOracleStopLoss`, when a bin is detected as empty (`totalShares == 0`), still advance `lastDecayTs` without touching the watermark values:

```solidity
if (totalShares == 0) {
    highWatermarks[pool_][binIdxs[i]].lastDecayTs = uint32(block.timestamp);
    continue;
}
```

This ensures the decay clock advances continuously regardless of bin activity, so the elapsed time on the first post-reactivation swap reflects only the time since the bin was last empty, not the entire dormant period.

## Proof of Concept

```
1. Deploy pool with OracleValueStopLossExtension, decayPerSecondE8 = 58.
2. Admin calls executeOracleStopLossHighWatermarks(pool, binIdx=1, hwm0=1000, hwm1=1000).
   → highWatermarks[pool][1] = {token0:1000, token1:1000, lastDecayTs: T0}
3. All LPs remove liquidity from bin 1 → totalShares[1] = 0.
4. Swaps cross bins 0 and 2 for 20+ days; bin 1 is skipped each time.
   → lastDecayTs for bin 1 remains T0.
5. LP adds liquidity back to bin 1 at time T0 + 20 days.
6. Attacker calls swap crossing bin 1.
   → dt = 1_728_000 s
   → factor = 58 * 1_728_000 = 100_224_000 >= 1e8
   → _decayed(1000, 58, 1_728_000) = 0
   → _applyWatermark(metricT0, 0, floorMultiplier) → (metricT0, false)
   → No OracleStopLossTriggered revert; swap proceeds.
7. Attacker drains LP value from bin 1 below the configured drawdown floor.
```
