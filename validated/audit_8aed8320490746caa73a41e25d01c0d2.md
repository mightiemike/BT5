### Title
Raw `approve()` in `DirectDepositV1.creditDeposit()` Permanently Bricks USDT Deposits When Allowance Is Left Unconsumed — (`core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls raw `token.approve(address(endpoint), balance)` before calling `endpoint.depositCollateralWithReferral(...)`. The endpoint silently returns without consuming the allowance when the deposit amount is below the minimum threshold. For USDT and other tokens that require the allowance to be zero before a new non-zero approval, the next call to `creditDeposit()` will permanently revert, bricking the DDA for that token.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`, for each product token the contract does:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(
    subaccount, productId, uint128(balance), "-1"
);
``` [1](#0-0) 

The endpoint's `depositCollateralWithReferral` contains an explicit early-return path:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked...
    // we can just skip the deposit and continue with the next asset.
    return;
}
``` [2](#0-1) 

When this early return fires, `handleDepositTransfer` is never called, meaning **zero tokens are pulled from the DDA** and the allowance set by `token.approve(...)` is left entirely unconsumed. The DDA now holds a non-zero allowance to the endpoint for that token.

USDT (and several other production ERC-20 tokens) implement the classic approve-race-condition guard: they revert if `approve(spender, nonZeroValue)` is called while `allowance(owner, spender) != 0`. On the next invocation of `creditDeposit()`, the line:

```solidity
token.approve(address(endpoint), balance);
``` [3](#0-2) 

will revert unconditionally because the residual allowance is non-zero, permanently preventing any future USDT deposit from this DDA.

---

### Impact Explanation

The DDA for the targeted subaccount becomes permanently unable to credit USDT (or any similarly-behaving token) to the protocol. All USDT sent to that DDA address after the first failed `creditDeposit()` call is permanently stranded — it cannot be deposited via `creditDeposit()` (reverts) and can only be recovered via the `onlyOwner` `withdrawFromDirectDepositV1` path. The subaccount loses access to the normal deposit flow for that token indefinitely.

---

### Likelihood Explanation

`creditDeposit()` has no access control — any address can call it. [4](#0-3) 

An attacker needs only to:
1. Send a dust amount of USDT (below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT`) to a victim's DDA address.
2. Call `creditDeposit()`.

Both steps are trivially executable by any unprivileged on-chain actor. USDT is the primary quote token in Nado (product ID 0), making this a high-value target. The `createDirectDepositV1` / `creditDepositV1` flow in `ContractOwner` is the documented deposit path for contract-owned subaccounts. [5](#0-4) 

---

### Recommendation

Before calling `approve`, reset the allowance to zero first, mirroring the pattern already used in `ContractOwner.wrapVaultAsset`:

```solidity
// In DirectDepositV1.creditDeposit():
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(...);
```

Alternatively, use OpenZeppelin's `safeIncreaseAllowance` (which avoids the race condition entirely) or set the approval to `type(uint256).max` once and skip re-approving on subsequent calls. [6](#0-5) 

---

### Proof of Concept

1. A victim subaccount `S` has a DDA deployed at address `D`.
2. Attacker sends 1 wei of USDT to `D` (below `MIN_FIRST_DEPOSIT_AMOUNT`).
3. Attacker calls `DirectDepositV1(D).creditDeposit()`.
4. Inside the loop for the USDT product:
   - `balance = USDT.balanceOf(D)` → 1 wei
   - `USDT.approve(endpoint, 1)` → succeeds (allowance was 0), USDT allowance = 1
   - `endpoint.depositCollateralWithReferral(S, 0, 1, "-1")` → `isValidDepositAmount` returns false → **silent return**, no transfer, allowance stays at 1
5. Later, a legitimate user sends 1000 USDT to `D` and calls `creditDeposit()`.
6. `USDT.approve(endpoint, 1000e6)` → **REVERTS** because USDT requires allowance == 0 before a new non-zero approval.
7. The DDA is permanently bricked for USDT. All future USDT deposits to `D` are stranded. [7](#0-6) [8](#0-7)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned

        }

            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
