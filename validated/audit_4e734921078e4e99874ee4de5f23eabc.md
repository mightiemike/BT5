### Title
AuxPoW Path Selection Bypassed via Caller-Controlled `Option<AuxData>` — (File: `contract/src/dogecoin.rs`)

### Summary
In the Dogecoin build of the BTC light client, `submit_block_header` selects between AuxPoW validation and standard PoW validation based solely on whether the caller-supplied `AuxData` is `Some` or `None`. The block header's AuxPoW version flag is never cross-checked against the presence or absence of `AuxData`. A malicious trusted relayer can submit an AuxPoW-flagged block with `aux_data = None`, causing the contract to apply standard scrypt PoW validation instead of the full AuxPoW chain verification, bypassing it entirely.

---

### Finding Description

`BlockHeader` for the Dogecoin build is defined as: [1](#0-0) 

```rust
pub type BlockHeader = (Header, Option<AuxData>);
```

The caller of `submit_blocks` fully controls whether `AuxData` is `Some` or `None`. Inside `submit_block_header`, the branch is: [2](#0-1) 

```rust
if !skip_pow_verification {
    self.check_target(&block_header, &prev_block_header);

    if let Some(ref aux_data) = aux_data {
        self.check_aux(&block_header, aux_data);
    } else {
        let pow_hash = block_header.block_hash_pow();
        require!(
            U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
            ...
        );
    }
}
```

The only place where the AuxPoW version flag is checked is **inside** `check_aux`: [3](#0-2) 

```rust
const BLOCK_VERSION_AUXPOW: i32 = 0x100;
require!(
    block_header.version & BLOCK_VERSION_AUXPOW != 0,
    "Aux POW block does not have AuxPoW flag set in version"
);
```

This check is **never reached** when `aux_data` is `None`. There is no reciprocal guard: when `aux_data` is `None`, the code never asserts that `block_header.version & BLOCK_VERSION_AUXPOW == 0`. The caller-supplied `Option<AuxData>` is the sole selector of the processing path, and the block's own version field is never validated against that choice — a direct structural analog to the ScrollChain `_version` parameter bug.

---

### Impact Explanation

For a legitimate Dogecoin AuxPoW block, the block header's own scrypt hash does **not** need to satisfy the PoW target — only the parent (Bitcoin) block's hash does. The Dogecoin header's nonce is normally 0 in AuxPoW blocks because the real nonce lives in the parent block.

However, an attacker can craft a block where:
- `version` has the AuxPoW flag set (`version | 0x100`)
- `prev_block_hash` points to the current chain tip
- `merkle_root` is set to an arbitrary value (e.g., the root of a fabricated transaction set)
- `nonce` is mined until `scrypt(block_header) <= target`

Submitted with `aux_data = None`, the contract accepts this block via the standard PoW branch. Because `merkle_root` is attacker-controlled, the attacker can subsequently call the public `verify_transaction_inclusion` with a crafted proof matching that root, obtaining a `true` result for a transaction that never existed on the real Dogecoin chain. [4](#0-3) 

This corrupts the canonical chain state and enables false transaction inclusion proofs — the primary security guarantee of the light client.

---

### Likelihood Explanation

The entry point requires `trusted_relayer` status: [5](#0-4) 

This is analogous to the `OnlySequencer` guard in the external report. The `trusted_relayer` macro exposes a `bypass_roles` path (`Role::UnrestrictedSubmitBlocks`) and a staking-based registration path, meaning the set of eligible callers is larger than a single admin key. If the staking threshold is low or if additional relayers are onboarded, the attack surface grows. The computational cost of mining a valid Dogecoin scrypt hash at current difficulty is feasible for a motivated attacker.

---

### Recommendation

In `submit_block_header` (Dogecoin build), add a reciprocal version-flag check in the `None` branch:

```rust
} else {
    // Reject AuxPoW-flagged blocks submitted without AuxData
    require!(
        block_header.version & BLOCK_VERSION_AUXPOW == 0,
        "AuxPoW flag set in version but no AuxData provided"
    );
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        "block should have correct pow"
    );
}
```

This mirrors the existing check inside `check_aux` and ensures the caller-supplied `Option<AuxData>` is always consistent with the block's declared version.

---

### Proof of Concept

1. Attacker registers as a trusted relayer (stakes the required deposit).
2. Attacker constructs a `Header` with:
   - `version = existing_version | 0x100` (AuxPoW flag set)
   - `prev_block_hash` = current mainchain tip hash
   - `merkle_root` = SHA256d of a fabricated transaction `T`
   - `bits` = current network target
   - Mines `nonce` until `scrypt(header_bytes) <= target_from_bits(bits)`
3. Attacker calls `submit_blocks([(header, None)])` — `aux_data` is `None`.
4. Contract executes the `else` branch at [6](#0-5)  — scrypt PoW check passes; block is stored as a valid mainchain block.
5. Attacker calls the public `verify_transaction_inclusion` with `tx_id = hash(T)`, `tx_block_blockhash = hash(fraudulent_header)`, `tx_index = 0`, `merkle_proof = []`.
6. Contract returns `true` for a transaction that never existed on the real Dogecoin chain, enabling theft in any downstream cross-chain application that trusts this result.
