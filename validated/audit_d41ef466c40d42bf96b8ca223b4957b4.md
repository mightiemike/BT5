Audit Report

## Title
`SwapAllowlistExtension` Bypassed via Router — Any User Can Swap Through Allowlisted Router Address - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` is used, `msg.sender` to the pool is always the router address, not the originating user. Any pool that allowlists the router — the natural configuration for router-mediated swaps — silently grants swap access to every user who calls through the router, regardless of individual allowlist status.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this `sender` value unchanged into the extension call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the first argument — the router address when the router calls the pool: [3](#0-2) 

`MetricOmmSimpleRouter.exactInput()` calls `pool.swap()` directly with no mechanism to forward the original caller's identity: [4](#0-3) 

The same applies to `exactInputSingle` and `exactOutputSingle`: [5](#0-4) 

The result: `sender` at the extension is always the router address. If `allowedSwapper[pool][router] == true`, the check passes for every user who routes through the router, regardless of whether that user is individually allowlisted.

## Impact Explanation
Any unprivileged user can bypass a per-user swap allowlist on any pool that has allowlisted the router. The pool admin's intent to restrict swaps to specific addresses is silently violated. Unauthorized users can trade in pools intended to be private or restricted (e.g., institutional pools, KYC-gated pools, or pools with restricted counterparties). This constitutes broken core access-control functionality — the `allowedSwapper` mapping is rendered meaningless for all router-mediated swaps.

## Likelihood Explanation
Allowlisting the router is the natural and expected configuration for any pool that wants to support router-mediated swaps while still restricting direct `pool.swap()` calls. The bypass requires no special privileges — any public user can call `exactInput`, `exactInputSingle`, or `exactOutputSingle` on the router. The condition is trivially reachable and repeatable.

## Recommendation
The router should forward the original `msg.sender` as part of `extensionData` (e.g., ABI-encoded as a prefix), and `SwapAllowlistExtension` should extract and check the originating user when the caller is a trusted router. Alternatively, the pool interface should carry an explicit `origin` address through hook arguments, or the extension should check both `sender` (the router) and an `origin` field extracted from `extensionData` when `sender` is a known trusted router.

## Proof of Concept
```solidity
// Pool configured with SwapAllowlistExtension
// Admin allowlists only the router (natural config for router-mediated swaps)
swapExtension.setAllowedToSwap(address(pool), address(router), true);
// nonAllowlistedUser is NOT individually allowlisted

// Non-allowlisted user calls exactInputSingle through the router
vm.prank(nonAllowlistedUser);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: tokenA,
    recipient: nonAllowlistedUser,
    amountIn: 1e18,
    amountOutMinimum: 0,
    zeroForOne: true,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// Swap succeeds — sender seen by extension is router (allowlisted)
// nonAllowlistedUser was never individually allowlisted but swapped successfully
```
