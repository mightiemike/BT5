### Title
Slow-Mode `WithdrawCollateral` Bypasses Product-Specific `withdrawFeeX18`, Causing Permanent Protocol Revenue Loss — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `processSlowModeTransactionImpl` handler for `WithdrawCollateral` omits the `chargeFee(…, withdrawFeeX18, productId)` call that is mandatory in the fast-path handler `processTransactionImpl`. Any user can route a withdrawal through the slow-mode queue, paying only the flat `SLOW_MODE_FEE` ($1) upfront, and receive the full withdrawal without the product-specific fee being deducted. The `sequencerFee[productId]` accounting bucket is permanently under-credited by the bypassed amount.

---

### Finding Description

**Fast path** (`processTransactionImpl`, `WithdrawCollateral` branch):

```
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,  // ← fee charged
    signedTx.tx.productId
);
clearinghouse.withdrawCollateral(…);
``` [1](#0-0) 

**Slow-mode path** (`processSlowModeTransactionImpl`, `WithdrawCollateral` branch):

```
validateSender(txn.sender, sender);
clearinghouse.withdrawCollateral(          // ← no fee charged at all
    txn.sender, txn.productId, txn.amount,
    address(0), nSubmissions
);
``` [2](#0-1) 

When a user calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload, `submitSlowModeTransactionImpl` falls into the `else` branch and charges only `SLOW_MODE_FEE` ($1) in USDC: [3](#0-2) 

After the 3-day delay, `_executeSlowModeTransaction` dispatches to `processSlowModeTransactionImpl`, which executes the withdrawal without ever calling `chargeFee` for `withdrawFeeX18`. [4](#0-3) 

The `chargeFee` helper credits `sequencerFee[productId]` and debits the sender's product balance. Because this call is absent in the slow-mode path, `sequencerFee[productId]` is permanently under-credited by `withdrawFeeX18` per exploited withdrawal. [5](#0-4) 

A secondary effect: in the fast path the fee deduction happens **before** `clearinghouse.withdrawCollateral` runs its `getHealth` check, so the health check is evaluated against `balance − fee − amount`. In the slow-mode path the health check is evaluated against `balance − amount` only. A user whose account would fail the initial-health check in the fast path (because the fee pushes them below the threshold) can pass it via slow mode, allowing a marginally larger withdrawal than the protocol intends. [6](#0-5) 

---

### Impact Explanation

- **Exact corrupted state**: `sequencerFee[productId]` is under-credited by `withdrawFeeX18` for every slow-mode withdrawal. The user's product balance retains the fee amount it should have lost.
- **Asset delta**: The protocol (sequencer fee pool) loses `withdrawFeeX18` per exploited withdrawal. For any product where `withdrawFeeX18 > SLOW_MODE_FEE` ($1), the attacker profits the difference.
- **Secondary**: Users near the initial-health boundary can withdraw slightly more collateral than the fast path would permit, because the fee is not subtracted before the health check.

---

### Likelihood Explanation

- The slow-mode submission path (`submitSlowModeTransaction`) is a public, permissionless entry point requiring no special role.
- The only cost is $1 USDC and a 3-day wait. For any withdrawal where `withdrawFeeX18` exceeds $1, the rational strategy is to use slow mode.
- The sequencer itself can execute slow-mode transactions early (`fromSequencer = true`), so the 3-day delay is an upper bound, not a guaranteed deterrent.
- No signature on the withdrawal payload itself is required in the slow-mode path — only `validateSender` (Ethereum `msg.sender` matching the subaccount address prefix), which is trivially satisfied by the legitimate account owner. [7](#0-6) 

---

### Recommendation

Add the same `chargeFee` call to the slow-mode `WithdrawCollateral` handler that exists in the fast-mode handler:

```solidity
// in processSlowModeTransactionImpl, WithdrawCollateral branch
validateSender(txn.sender, sender);
chargeFee(
    txn.sender,
    spotEngine.getConfig(txn.productId).withdrawFeeX18,
    txn.productId
);
clearinghouse.withdrawCollateral(…);
```

If slow-mode withdrawals are intentionally fee-exempt (e.g., as a sequencer-unavailability relief valve), this should be explicitly enforced by gating slow-mode `WithdrawCollateral` to periods when the sequencer is provably offline, and the exemption should be documented.

---

### Proof of Concept

1. Product P has `withdrawFeeX18 = F` (any non-zero value, e.g. $10 equivalent in the product token).
2. Attacker owns subaccount `S` with sufficient collateral.
3. Attacker calls `Endpoint.submitSlowModeTransaction(abi.encodePacked(uint8(WithdrawCollateral), abi.encode(WithdrawCollateral({sender: S, productId: P, amount: A}))))`, paying `SLOW_MODE_FEE` = $1.
4. After ≤ 3 days, `executeSlowModeTransaction()` is called (by anyone, or by the sequencer).
5. `processSlowModeTransactionImpl` executes: `validateSender` passes, `clearinghouse.withdrawCollateral` runs, health check passes (evaluated without fee deduction).
6. Attacker receives `A` tokens. `sequencerFee[P]` is credited $0 instead of `F`. Net gain to attacker: `F − $1`. [8](#0-7) [9](#0-8)

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
