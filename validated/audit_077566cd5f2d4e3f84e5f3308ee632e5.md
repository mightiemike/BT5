### Title
`ValidResourceBounds` Protobuf Round-Trip Silently Collapses `AllResources` to `L1Gas`, Producing a Different Transaction Hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserialization of `ValidResourceBounds` in the block-sync/consensus path silently converts `AllResources { l1_gas: X, l2_gas: ZERO, l1_data_gas: ZERO }` into `L1Gas(X)`. Because `get_tip_resource_bounds_hash` hashes a different number of resource-bound felts for each variant (2 for `L1Gas`, 3 for `AllResources`), the Poseidon hash of the fee fields — and therefore the entire transaction hash — changes after a protobuf round-trip. A transaction committed with hash H₁ (computed over `AllResources`) is re-hashed as H₂ ≠ H₁ after deserialization, breaking the canonicalization invariant that every representation of the same transaction must produce the same hash.

### Finding Description

**Step 1 — The lossy conversion.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `transaction.rs`:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // ← zero when absent
// ...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)                      // ← collapses AllResources
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

When a transaction carries `AllResources { l1_gas: X, l2_gas: ZERO, l1_data_gas: ZERO }` (a valid post-0.13.3 transaction that happens to use only L1 gas), the serializer emits three non-`None` protobuf fields (all with zero amounts for l2 and l1_data). The deserializer then sees `l1_data_gas.is_zero() && l2_gas.is_zero()` and silently returns `L1Gas(X)` instead of `AllResources`.

**Step 2 — The hash divergence.**

`get_tip_resource_bounds_hash` branches on the variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts total
    }
});
Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
``` [2](#0-1) 

- `AllResources { l1_gas: X, l2_gas: ZERO, l1_data_gas: ZERO }` → Poseidon(`tip`, `L1_GAS_packed(X)`, `L2_GAS_packed(0)`, `L1_DATA_GAS_packed(0)`) = **H₁**
- `L1Gas(X)` → Poseidon(`tip`, `L1_GAS_packed(X)`, `L2_GAS_packed(0)`) = **H₂**

H₁ ≠ H₂ because the Poseidon sponge absorbs a different number of field elements.

**Step 3 — How the two variants arise.**

All new transactions enter the system as `RpcInvokeTransactionV3` / `RpcDeclareTransactionV3` / `RpcDeployAccountTransactionV3`, which carry `resource_bounds: AllResourceBounds`. When converted to the canonical `InvokeTransactionV3` for storage, the conversion always wraps them as `ValidResourceBounds::AllResources(...)`:

```rust
impl From<RpcInvokeTransactionV3> for InvokeTransactionV3 {
    fn from(tx: RpcInvokeTransactionV3) -> Self {
        Self {
            resource_bounds: ValidResourceBounds::AllResources(tx.resource_bounds),
            ...
        }
    }
}
``` [3](#0-2) 

So a post-0.13.3 transaction with zero l2_gas and l1_data_gas is stored as `AllResources` and hashed as H₁. After protobuf round-trip through the block-sync path it becomes `L1Gas` and re-hashes as H₂.

**Step 4 — The serializer confirms the collision.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes both `L1Gas(X)` and `AllResources { l1_gas: X, l2_gas: ZERO, l1_data_gas: ZERO }` to the identical wire bytes:

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),          // ZERO
    l1_data_gas: Some(ResourceBounds::default().into()), // ZERO
},
``` [4](#0-3) 

Two distinct Rust values with different hashes are serialized to the same bytes, and the deserializer always picks `L1Gas`. The round-trip is not an identity function.

**Note on the parallel RPC-transaction converter.**

The `rpc_transaction.rs` converter used for mempool/consensus `RpcTransaction` objects is *not* affected: it always deserializes into `AllResourceBounds` directly and never produces `L1Gas`:

```rust
impl TryFrom<protobuf::ResourceBounds> for AllResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        Ok(Self {
            l1_gas: value.l1_gas.ok_or(missing(...))?.try_into()?,
            l2_gas: value.l2_gas.ok_or(missing(...))?.try_into()?,
            l1_data_gas: value.l1_data_gas.ok_or(missing(...))?.try_into()?,
        })
    }
}
``` [5](#0-4) 

The bug is isolated to the `transaction.rs` converter used for the canonical `InvokeTransactionV3` / `DeclareTransactionV3` / `DeployAccountTransactionV3` types in the block-sync path.

### Impact Explanation

Any node that receives a block via P2P sync and re-derives transaction hashes from the deserialized `InvokeTransactionV3` data will compute H₂ for transactions that were committed with H₁. This breaks the transaction-commitment verification step: the recomputed hash does not match the hash stored in the block header, causing the block to be rejected as invalid. A single valid transaction with `AllResources { l2_gas: 0, l1_data_gas: 0 }` is sufficient to make an entire block unverifiable by any syncing peer, constituting a wrong-state / wrong-receipt result from blockifier/execution logic for accepted input.

### Likelihood Explanation

The trigger condition — a V3 transaction with zero l2_gas and zero l1_data_gas — is a normal, valid transaction configuration (e.g., a simple ETH-fee-only invoke on a network that has not yet activated L2-gas billing). No special privileges are required; any user submitting such a transaction causes the invariant to be violated for every node that later syncs the block containing it.

### Recommendation

Remove the `L1Gas`/`AllResources` branching logic from the protobuf deserializer. The presence of all three fields in the wire format (even if zero) should always produce `AllResources`, preserving the original variant and the original hash:

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let l1_gas = value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?;
        let l2_gas = value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?;
        let l1_data_gas = value.l1_data_gas.ok_or(missing("ResourceBounds::l1_data_gas"))?.try_into()?;
        // Always AllResources when all three fields are present.
        Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
    }
}
```

The `L1Gas` variant should only be produced when deserializing genuinely pre-0.13.3 wire messages that lack the `l2_gas` field entirely (i.e., `value.l2_gas` is `None`).

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds,
};
use starknet_api::block::{GasAmount, GasPrice};
use apollo_protobuf::protobuf;

fn main() {
    // A post-0.13.3 transaction with zero l2_gas and l1_data_gas.
    let original = ValidResourceBounds::AllResources(AllResourceBounds {
        l1_gas: ResourceBounds {
            max_amount: GasAmount(1000),
            max_price_per_unit: GasPrice(1),
        },
        l2_gas: ResourceBounds::default(),      // zero
        l1_data_gas: ResourceBounds::default(), // zero
    });

    // Serialize to protobuf (transaction.rs path).
    let proto: protobuf::ResourceBounds = original.into();

    // Deserialize back (transaction.rs path).
    let roundtripped = ValidResourceBounds::try_from(proto).unwrap();

    // The variant has changed.
    assert!(matches!(roundtripped, ValidResourceBounds::L1Gas(_)),
        "Expected L1Gas after round-trip");

    // The hashes differ.
    let tip = starknet_api::transaction::fields::Tip(0);
    let h1 = starknet_api::transaction_hash::get_tip_resource_bounds_hash(&original, &tip).unwrap();
    let h2 = starknet_api::transaction_hash::get_tip_resource_bounds_hash(&roundtripped, &tip).unwrap();
    assert_ne!(h1, h2, "Hash changed after protobuf round-trip: {h1:?} vs {h2:?}");
}
```

The assertion `h1 != h2` holds because `AllResources` hashes 3 resource felts while `L1Gas` hashes only 2, and Poseidon is collision-resistant across inputs of different lengths.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
