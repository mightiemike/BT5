### Title
`init` Does Not Grant Any Administrative Role to a Human Account — (`contract/src/lib.rs`)

### Summary
The `init` function sets the contract account itself as the sole super admin but never grants any role (`DAO`, `PauseManager`, `RelayerManager`, etc.) to a human/DAO account. `InitArgs` accepts no `admin` parameter. After deployment, no human account holds any privileged role, leaving the contract unable to be paused, upgraded, or have its relayer list managed until a separate manual post-deployment step is executed.

### Finding Description
In `contract/src/lib.rs`, the `init` function bootstraps the access-control system by calling `acl_init_super_admin(env::current_account_id())`, which grants super-admin status to the contract account itself. [1](#0-0) 

The comment explicitly states this is done "to allow granting any role in the constructor," yet the constructor never follows through — no role is granted to any human account before `init` returns. [2](#0-1) 

`InitArgs` contains no `admin` or `dao` field, so there is no mechanism to pass an intended administrator at deployment time. [3](#0-2) 

The roles that remain unassigned after `init` gate every sensitive operation:

| Role | Blocked operation |
|---|---|
| `DAO` / `PauseManager` | `pause` / `unpause` |
| `DAO` / `CodeStager` / `CodeDeployer` | `up_stage_code` / `up_deploy_code` |
| `DAO` / `DurationManager` | `up_init_staging_duration` |
| `DAO` / `RelayerManager` | trusted-relayer management | [4](#0-3) [5](#0-4) 

### Impact Explanation
Between deployment and a manual `acl_grant_role` call, the contract has zero human administrators. If the deployer omits the post-deployment role-granting step (or if the contract account's keys are rotated or lost), the following become permanently impossible:

- **Emergency pause** — a live exploit cannot be halted.
- **Contract upgrade** — a critical security patch cannot be deployed.
- **Relayer management** — malicious or stale relayers cannot be removed.

These are not theoretical: the relayer service's `init_contract` path calls `init` and returns immediately with no follow-up role-granting logic. [6](#0-5) 

### Likelihood Explanation
High. Every deployment goes through `init`. There is no on-chain enforcement that roles must be granted before the contract becomes operational. The relayer's automated init path does not grant any role. A deployer who follows only the documented automated flow ends up with a live contract that no human account can administer.

### Recommendation
Accept an `admin: AccountId` (or a set of role-to-account mappings) in `InitArgs` and grant the required roles (`DAO`, `PauseManager`, `RelayerManager`) to that account inside `init` before returning, mirroring the pattern the comment already describes as the intent:

```rust
// After acl_init_super_admin succeeds:
contract.acl_grant_role(Role::DAO.into(), args.admin.clone());
contract.acl_grant_role(Role::PauseManager.into(), args.admin.clone());
contract.acl_grant_role(Role::RelayerManager.into(), args.admin);
```

### Proof of Concept

1. Deploy the contract via the relayer's `init_contract` path (or any standard deployment).
2. Call `init` with valid `InitArgs` — no `admin` field exists, so no role is granted.
3. Immediately after `init`, call `acl_get_grantees("DAO")` → returns empty list.
4. Attempt to call `up_stage_code` from any account → panics with "Unauthorized".
5. Attempt to call `pause` from any account → panics with "Unauthorized".
6. The contract is live and accepting block headers but is permanently unmanageable by any human account until a separate `acl_grant_role` transaction is manually issued by whoever controls the contract account's full-access key. [7](#0-6)
