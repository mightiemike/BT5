Audit Report

## Title
SwapAllowlistExtension Checks Router Address as `sender`, Allowing Per-User Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter against `allowedSwapper[pool][sender]`. When `MetricOmmSimpleRouter` calls `pool.swap(...)`, the pool sets `sender = msg.sender = router address`. If the router is allowlisted, any unprivileged, non-allowlisted user can bypass per-user swap restrictions by routing through the router. Conversely, if the router is not allowlisted, individually allowlisted users cannot use the router at all, breaking core swap functionality.

## Finding Description

**Root cause — wrong identity checked in `beforeSwap`:**

`SwapAllowlistExtension.beforeSwap` (line 37) checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool (correct) and `sender` is the first ABI-decoded parameter forwarded by the pool from `ExtensionCalling._beforeSwap`.

**How `sender` is set — pool derives it from `msg.sender` of the `swap` call:**

`ExtensionCalling._beforeSwap` (lines 149–177) encodes whatever `sender` address the pool passes. The pool's `swap` function is called by the router with no explicit `sender` argument; the pool must therefore set `sender = msg.sender` of the `swap` call, which is the router contract.

**Router call site — actual user address is never forwarded to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` (lines 71–80):
```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64, "", params.extensionData
);
```
`msg.sender` (the actual user) is stored only in transient storage for the payment callback. The `pool.swap(...)` call carries no user-identity argument. The pool sees `msg.sender = router`.

**Exploit path:**

1. Pool admin deploys pool with `SwapAllowlistExtension` to restrict swaps to KYC'd addresses.
2. Pool admin allowlists the router (`allowedSwapper[pool][router] = true`) so that legitimate users can swap via the router — a normal operational step.
3. Non-allowlisted attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the restricted pool.
4. Pool calls `_beforeSwap(router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
5. Attacker swaps successfully despite not being individually allowlisted.

**Contrast with `DepositAllowlistExtension` — correct design:**

`DepositAllowlistExtension.beforeAddLiquidity` (line 38) checks `owner` (the second parameter), which is the actual position owner forwarded through `MetricOmmPoolLiquidityAdder` unchanged. The deposit allowlist correctly identifies the real beneficiary. The swap allowlist does not.

**Existing guards are insufficient:**

There is no mechanism in the router or pool to forward the original `msg.sender` as the `sender` identity to the extension. The transient callback context stores the payer for token settlement only; it is never read by the extension layer.

## Impact Explanation

An unprivileged, non-allowlisted user can swap in a pool that the pool admin intended to restrict to specific addresses. This breaks the admin-enforced access control boundary: the pool admin's per-user allowlist is rendered ineffective whenever the router is allowlisted. Any swap volume, fee accrual, or price impact that the restricted pool was designed to prevent from non-allowlisted actors becomes reachable. This is a broken core pool functionality / admin-boundary break under the allowed impact gate.

## Likelihood Explanation

The precondition — router being allowlisted — is a standard operational step for any pool that wants to support router-based swaps. Pool admins who configure both a per-user allowlist and router support will unknowingly create the bypass. The attack requires no special privileges, no flash loans, and no oracle manipulation; any EOA can call the router.

## Recommendation

Pass the original caller's identity through the swap path so extensions can check it. One approach: add a `payer` or `originator` field to `extensionData` that the router populates with `msg.sender`, and have `SwapAllowlistExtension` decode and verify it. A cleaner approach is to add an explicit `sender` parameter to `IMetricOmmPoolActions.swap` (analogous to how `owner` is explicit in `addLiquidity`) so the pool forwards the true initiator rather than `msg.sender`. Alternatively, document that `SwapAllowlistExtension` gates by direct caller only and provide a separate extension that decodes a signed or router-attested user identity from `extensionData`.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup:
// 1. Deploy pool with SwapAllowlistExtension.
// 2. Pool admin allowlists the router:
//    swapAllowlist.setAllowedToSwap(pool, address(router), true);
// 3. Pool admin does NOT allowlist attacker:
//    allowedSwapper[pool][attacker] == false

// Attack:
// Attacker (non-allowlisted EOA) calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: restrictedPool,
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    tokenIn: token0,
    deadline: block.timestamp,
    extensionData: ""
}));
// pool.swap() is called with msg.sender = router
// beforeSwap receives sender = router
// allowedSwapper[pool][router] == true → passes
// Attacker swaps successfully, bypassing per-user allowlist
```
