### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Any Invoke When Any Transaction Exists in Mempool for Undeployed Account â€” (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway skips the `__validate__` entry-point execution for invoke transactions with `nonce=1` when the on-chain account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. The mempool check does not distinguish transaction types â€” it returns `true` for **any** transaction in the pool for that address, not exclusively a `DeployAccount`. An unprivileged attacker who observes a legitimate `DeployAccount` transaction in the mempool for address `A` can immediately submit a crafted invoke with `nonce=1` from `A` carrying an invalid signature. The gateway skips `__validate__`, admits the invalid invoke to the mempool, and the transaction is later executed by the blockifier where `__validate__` runs and fails â€” but the transaction has already been sequenced.

### Finding Description

`skip_stateful_validations` implements a UX special-case: when a user submits a `deploy_account + invoke` pair simultaneously, the invoke (nonce=1) should be admitted even though the account does not yet exist on-chain. [1](#0-0)

```
tx.nonce() == Nonce(Felt::ONE)
  && account_nonce == Nonce(Felt::ZERO)
  && account_tx_in_pool_or_recent_block(sender_address) == true
```

Neither `contains_account` nor `tx_pool.contains_account` checks the **type** of the transaction stored for that address. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning only holds for the legitimate user. A third-party attacker can exploit the presence of the legitimate user's `DeployAccount` in the pool to satisfy the check for their own crafted invoke.

### Impact Explanation

**High â€” Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can submit an invoke transaction with an arbitrary (invalid) signature for any address `A` that has a pending `DeployAccount` in the mempool. The gateway admits it without running `__validate__`. The invalid invoke enters the mempool and is eventually picked up by the batcher. During blockifier execution, `__validate__` runs with `execution_flags.validate=true` (the batcher sets default flags), fails, and the transaction reverts. Depending on the account's balance at execution time, a fee may be charged to the victim's account. At minimum, the attacker can flood the mempool with invalid invokes for every observed `DeployAccount`, consuming mempool capacity and batcher execution resources at negligible cost.

---

### Likelihood Explanation

**Medium.** The `DeployAccount` transactions are broadcast over P2P and visible to any mempool participant. The attacker only needs to observe a `DeployAccount` for address `A`, compute or read the sender address, and immediately submit a crafted invoke with `nonce=1`. No privileged access is required. The window is open from the moment the `DeployAccount` enters the mempool until it is committed on-chain (at which point the on-chain nonce becomes 1 and the bypass condition `account_nonce == 0` no longer holds).

---

### Recommendation

**Short term:** In `skip_stateful_validations`, replace the generic `account_tx_in_pool_or_recent_block` check with a type-specific query that verifies a `DeployAccount` transaction (not just any transaction) exists in the mempool for the sender address. This eliminates the ability of a third party to exploit another user's pending `DeployAccount` to bypass signature validation.

**Long term:** Avoid special-casing validation logic based on mempool state that does not encode the specific invariant being relied upon. The current check conflates "account is known to the mempool" with "a deploy_account for this account is pending," which are not equivalent from a security perspective. A dedicated `deploy_account_in_pool(address)` predicate would make the invariant explicit and auditable.

### Proof of Concept

1. Alice submits `RpcDeployAccountTransaction::V3` for address `A` (class hash `C`, salt `S`). The gateway validates and forwards it to the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.

2. Attacker observes `A` in the mempool (via P2P gossip or RPC). Attacker constructs:
   ```
   RpcInvokeTransaction::V3 {
       sender_address: A,
       nonce: 1,
       signature: [0xdead, 0xbeef],   // garbage
       resource_bounds: <above min_gas_price>,
       calldata: [],
       ...
   }
   ```

3. Attacker submits the invoke to the gateway. Stateless validation passes (no signature check). Stateful validation:
   - `validate_nonce`: `0 <= 1 <= max_allowed_nonce_gap` â†’ passes. [6](#0-5) 
   - `validate_by_mempool`: no duplicate nonce â†’ passes.
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` â†’ returns `true`. [7](#0-6) 
   - `run_validate_entry_point` called with `skip_validate=true` â†’ `__validate__` not executed.

4. Invalid invoke is forwarded to the mempool and admitted.

5. Alice's `DeployAccount` is committed. Batcher picks up the attacker's invoke. Blockifier runs `__validate__` with `validate=true` (default execution flags). `__validate__` fails (invalid signature). Transaction reverts; fee is charged to `A`.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
