### Title
AuxPoW Validation Unconditionally Skipped When Relayer Omits `aux_data` for AuxPoW-Flagged Dogecoin Blocks - (File: `contract/src/dogecoin.rs`)

---

### Summary

The Dogecoin build of `submit_block_header` branches entirely on whether the caller-supplied `aux_data` field is `Some` or `None`. When `None` is supplied, the full `check_aux` path is skipped and only a direct block-hash PoW check is performed. No guard enforces that a block whose `version` has the AuxPoW flag set (`version & 0x100 != 0`) must carry `aux_data`. A relayer supplying adversarial chain data can therefore omit `aux_data` for a legitimately AuxPoW-flagged block, causing every AuxPoW-specific consensus rule to be silently bypassed.

---

### Finding Description

In `contract/src/dogecoin.rs`, `submit_block_header` (lines 166–204) performs the following branch:

```rust
if let Some(ref aux_data) = aux_data {
    self.check_aux(&block_header, aux_data);
} else {
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        format!("block should have correct pow")
    );
}
``` [1](#0-0) 

The `aux_data` value is entirely caller-controlled — it is the second element of the `(Header, Option<AuxData>)` tuple that the relayer serialises and submits via `submit_blocks`. [2](#0-1) 

`check_aux` enforces every AuxPoW-specific consensus rule:

- The block's `version` must have the AuxPoW flag set (`version & 0x100 != 0`).
- The block's chain ID must match the configured Dogecoin chain ID.
- The parent block must **not** carry the Dogecoin chain ID.
- The coinbase transaction must commit to the correct chain merkle root.
- The chain merkle proof must be valid.
- The nonce/index must match the expected slot.
- The **parent block's** hash (not the Dogecoin block's own hash) must satisfy the PoW target. [3](#0-2) 

None of these checks are performed when `aux_data = None`. The only surviving check is the Dogecoin block's own hash against the target — a check that is semantically wrong for AuxPoW blocks, where the work is proved by the parent Bitcoin block's hash, not the Dogecoin block's own hash.

There is no guard of the form:

```rust
require!(
    block_header.version & BLOCK_VERSION_AUXPOW == 0,
    "AuxPoW-flagged block must supply aux_data"
);
```

before the `else` branch. The control-flow decision is therefore made entirely on an attacker-controlled field, mirroring the external report's pattern where an invalid sub-item caused the main validation path to be skipped.

---

### Impact Explanation

A relayer that submits a Dogecoin block with `version & 0x100 != 0` (AuxPoW flag set) and `aux_data = None` causes the contract to:

1. Accept the block without verifying chain ID, parent-block chain ID, coinbase commitment, chain merkle proof, or nonce slot.
2. Apply the wrong PoW check (the Dogecoin block's own hash instead of the parent Bitcoin block's hash).

If the block's own hash satisfies the target (i.e., the attacker mined it directly), the contract stores it as a valid canonical or fork header. The real Dogecoin network rejects any block that carries the AuxPoW flag without valid AuxPoW data, so the light client's canonical chain view diverges from the real chain. Downstream callers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` then operate against a corrupted chain state, producing incorrect SPV proof results. [4](#0-3) 

---

### Likelihood Explanation

**Medium-low.** The entry point is `submit_blocks`, which is gated by `#[trusted_relayer]`. [5](#0-4) 

A relayer that has staked and been approved can call this function. The report explicitly lists "relayer-path user supplying adversarial chain data" as a valid attacker class. A compromised or malicious approved relayer can craft the submission without any additional privilege escalation. The remaining barrier is mining: the attacker must produce a Dogecoin block whose own hash satisfies the current PoW target (equivalent to mining a regular Dogecoin block). This is non-trivial but not impossible for an entity with meaningful hash power, and the cost is the same as mining one legitimate block.

---

### Recommendation

Add an explicit guard in the `else` branch of `submit_block_header` (Dogecoin build) that rejects any block whose `version` has the AuxPoW flag set when `aux_data` is absent:

```rust
} else {
    const BLOCK_VERSION_AUXPOW: i32 = 0x100;
    require!(
        block_header.version & BLOCK_VERSION_AUXPOW == 0,
        "AuxPoW-flagged block must supply aux_data"
    );
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        format!("block should have correct pow")
    );
}
```

This mirrors the fix recommended in the external report: remove the control-flow path that allows untrusted input to bypass mandatory validation.

---

### Proof of Concept

1. Attacker (approved relayer) mines a Dogecoin block `B` with `version & 0x100 != 0` (AuxPoW flag set) such that `sha256d(B_header) ≤ target_from_bits(B.bits)` — i.e., the block's own hash satisfies the PoW target.
2. Attacker calls `submit_blocks([(B_header, None)])` on the NEAR contract.
3. `submit_block_header` is invoked with `aux_data = None`.
4. `check_target` passes (difficulty bits are valid).
5. The `else` branch executes: `block_hash_pow()` returns a hash below the target, so `require!` passes.
6. `check_aux` is never called. Chain ID, parent-block chain ID, coinbase commitment, chain merkle proof, and nonce slot are all unverified.
7. `submit_block_header_inner` stores `B` as a valid header. The real Dogecoin network would reject `B` because it carries the AuxPoW flag without valid AuxPoW data.
8. The light client's canonical chain now includes a block that does not exist on the real Dogecoin chain. Any subsequent `verify_transaction_inclusion` call referencing a transaction in `B` returns `true` against a fraudulent chain state. [6](#0-5)

    pub(crate) fn check_aux(&mut self, block_header: &Header, aux_data: &AuxData) {
        // The Dogecoin block must have the AuxPoW flag set (bit 8) when AuxPoW data is present.
        // https://github.com/dogecoin/dogecoin/blob/master/src/auxpow.h
        const BLOCK_VERSION_AUXPOW: i32 = 0x100;

        require!(
            aux_data.chain_merkle_proof.len() <= 30,
            "Aux POW chain merkle branch too long"
        );
        require!(
            block_header.version & BLOCK_VERSION_AUXPOW != 0,
            "Aux POW block does not have AuxPoW flag set in version"
        );

        let chain_id = self.get_config().aux_chain_id;

        require!(
            chain_id == block_header.get_chain_id(),
            format!(
                "block does not have our chain ID (got {}, expected {chain_id})",
                block_header.get_chain_id()
            )
        );

        require!(
            chain_id != aux_data.parent_block.get_chain_id(),
            "Aux POW parent has our chain ID"
        );

        let chain_root = merkle_tools::compute_root_from_merkle_proof(
            block_header.block_hash(),
            aux_data.chain_id,
            &aux_data.chain_merkle_proof,
        );

        let coinbase_tx = aux_data.get_coinbase_tx();
        let coinbase_tx_hash = coinbase_tx.compute_txid();

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                H256::from(coinbase_tx_hash.to_raw_hash().to_byte_array()),
                0,
                &aux_data.merkle_proof,
            ) == aux_data.parent_block.merkle_root
        );

        let script_sig = coinbase_tx
            .input
            .first()
            .unwrap()
            .script_sig
            .to_hex_string();
        let pos_merged_mining_header = script_sig.find(MERGED_MINING_HEADER);
        let mut pos_chain_root = script_sig
            .find(&chain_root.to_string())
            .expect("Aux POW missing chain merkle root in parent coinbase");

        match pos_merged_mining_header {
            Some(pos_merged_mining_header) => {
                if script_sig[pos_merged_mining_header + MERGED_MINING_HEADER.len()..]
                    .contains(MERGED_MINING_HEADER)
                {
                    env::panic_str("Multiple merged mining headers in coinbase");
                }

                require!(
                    pos_merged_mining_header + MERGED_MINING_HEADER.len() == pos_chain_root,
                    "Merged mining header is not just before chain merkle root"
                );
            }
            None => {
                require!(pos_chain_root <= 40, "Aux POW chain merkle root must start in the first 20 bytes of the parent coinbase");
            }
        }

        pos_chain_root += chain_root.to_string().len();
        require!(
            script_sig.len() - pos_chain_root >= 16,
            "Aux POW missing chain merkle tree size and nonce in parent coinbase"
        );

        let bytes = hex::decode(&script_sig[pos_chain_root..pos_chain_root + 8]).unwrap();
        let n_size = u32::from_le_bytes([bytes[0], bytes[1], bytes[2], bytes[3]]);
        require!(
            n_size == (1u32 << aux_data.chain_merkle_proof.len()),
            "Aux POW merkle branch size does not match parent coinbase"
        );

        let bytes = hex::decode(&script_sig[pos_chain_root + 8..pos_chain_root + 16]).unwrap();
        let n_nonce = u32::from_le_bytes([bytes[0], bytes[1], bytes[2], bytes[3]]);

        let chain_id = block_header.get_chain_id();

        let expected_index =
            Self::get_expected_index(n_nonce, chain_id, aux_data.chain_merkle_proof.len());

        require!(
            u32::try_from(aux_data.chain_id).ok() == Some(expected_index),
            "Aux POW wrong index"
        );
        let pow_hash = aux_data.parent_block.block_hash_pow();
        require!(
            self.skip_pow_verification
                || U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
            format!("block should have correct pow")
        );
    }

    pub(crate) fn submit_block_header(
        &mut self,
        header: (Header, Option<AuxData>),
        skip_pow_verification: bool,
    ) {
        let (block_header, aux_data) = header;

        let prev_block_header = self.get_prev_header(&block_header);
        let current_block_hash = block_header.block_hash();

        if !skip_pow_verification {
            self.check_target(&block_header, &prev_block_header);

            if let Some(ref aux_data) = aux_data {
                self.check_aux(&block_header, aux_data);
            } else {
                let pow_hash = block_header.block_hash_pow();
                // Check if the block hash is less than or equal to the target
                require!(
                    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
                    format!("block should have correct pow")
                );
            }
        }

        let (current_block_computed_chain_work, overflow) = prev_block_header
            .chain_work
            .overflowing_add(work_from_bits(block_header.bits));
        require!(!overflow, "Addition of U256 values overflowed");

        let current_header = ExtendedHeader {
            block_header: block_header.clone().into_light(),
            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        self.submit_block_header_inner(current_header, &prev_block_header);
    }
