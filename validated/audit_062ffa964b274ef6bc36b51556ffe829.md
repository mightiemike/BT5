### Title
Slow-Mode Transaction Queue Delay Exceeds `EndpointTx` Upgrade Timelock, Leaving Users Unable to Cancel Queued Transactions — (File: `core/contracts/Endpoint.sol`, `core/contracts/BaseProxyManager.sol`)

---

### Summary

Slow-mode transactions are queued with a mandatory 3-day delay (`SLOW_MODE_TX_DELAY`) and have no cancellation mechanism. The `EndpointTx` contract — which processes slow-mode transactions via `delegatecall` — can be upgraded with **zero timelock** through `ProxyManager.migrateAll()`. This creates a structural mismatch identical in class to the Nibbl finding: a user's queued slow-mode transaction will be executed against a different implementation than the one that existed when the user submitted it, with no ability to exit.

---

### Finding Description

**Step 1 — User is locked into a 3-day queue with no cancel path.**

When a user calls `submitSlowModeTransaction` (e.g., `WithdrawCollateral`), `EndpointTx.submitSlowModeTransactionImpl` queues the transaction with:

```solidity
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
    sender: sender,
    tx: transaction
});
``` [1](#0-0) 

The user also pays a non-refundable `SLOW_MODE_FEE` at submission time: [2](#0-1) 

There is no `cancelSlowModeTransaction` function anywhere in `Endpoint.sol` or `EndpointTx.sol`. Once queued, the transaction cannot be withdrawn.

**Step 2 — `EndpointTx` is upgraded via `delegatecall` with zero timelock.**

`ProxyManager.submitImpl` stages a new implementation immediately (no delay):

```solidity
function submitImpl(string memory name, address impl) external override onlySubmitter {
    require(pendingImpls[name] != address(0), "unsupported contract.");
    pendingImpls[name] = impl;
    pendingHashes[name] = _getCodeHash(impl.code);
}
``` [3](#0-2) 

`BaseProxyManager.migrateAll` can be called by the owner **immediately** after `submitImpl` — there is no enforced waiting period between the two calls: [4](#0-3) 

The `EndpointTx` upgrade path is `migrateAll` → `_migrateEndpointTx` → `proxyManagerHelper.upgradeEndpointTx` → `Endpoint.upgradeEndpointTx`, which sets `endpointTx = _endpointTx` in live storage: [5](#0-4) 

**Step 3 — Queued slow-mode transactions execute against the new implementation.**

When the slow-mode transaction is eventually executed (by the sequencer or anyone after the delay), `Endpoint.processSlowModeTransaction` routes it through `_delegatecallEndpointTx` using the **current** `endpointTx` address:

```solidity
function processSlowModeTransaction(address sender, bytes calldata transaction) public virtual {
    require(msg.sender == address(this));
    _delegatecallEndpointTx(
        abi.encodeWithSelector(
            EndpointTx.processSlowModeTransactionImpl.selector,
            sender,
            transaction
        )
    );
}
``` [6](#0-5) 

The `delegatecall` target is resolved at execution time from `endpointTx`, not at submission time. If the implementation was replaced during the 3-day window, the user's transaction is processed by the new logic.

**Broken invariant**: `EndpointTx upgrade timelock (0 days) < SLOW_MODE_TX_DELAY (3 days)`.

---

### Impact Explanation

A user who submits a slow-mode `WithdrawCollateral` (or `LinkSigner`, `DepositCollateral`, etc.) is committed for 3 days with no exit. During that window, the owner can upgrade `EndpointTx` to an implementation with different withdrawal logic, different fee handling, or altered validation. The user's transaction will be processed under the new rules they never consented to, and the non-refundable `SLOW_MODE_FEE` is already spent. The concrete corrupted state is the user's collateral balance and the execution semantics of their queued transaction.

---

### Likelihood Explanation

Requires the protocol owner to upgrade `EndpointTx` during the 3-day slow-mode window. This is the same trust assumption as the Nibbl finding (admin making an undesirable change). The structural mismatch means any routine upgrade during a period when slow-mode transactions are queued creates this exposure. Given that slow-mode is the censorship-resistance path — used precisely when users distrust the sequencer — the scenario where an upgrade coincides with a queued slow-mode transaction is realistic.

---

### Recommendation

Either:
1. **Add a cancel function** for slow-mode transactions that allows the original sender to remove their queued entry and recover the `SLOW_MODE_FEE` before `executableAt`, or
2. **Enforce a timelock on `EndpointTx` upgrades** of at least `SLOW_MODE_TX_DELAY` (3 days) between `submitImpl` and `migrateAll` for the `EndpointTx` name, so the invariant `EndpointTx upgrade delay >= SLOW_MODE_TX_DELAY` holds.

---

### Proof of Concept

1. User calls `submitSlowModeTransaction(withdrawTx)` — transaction queued with `executableAt = now + 3 days`, `SLOW_MODE_FEE` charged and non-refundable. [7](#0-6) 

2. Submitter calls `ProxyManager.submitImpl("EndpointTx", newEndpointTxAddr)` — stages new implementation instantly. [3](#0-2) 

3. Owner calls `ProxyManager.migrateAll([NewImpl("EndpointTx", newEndpointTxAddr)])` — `endpointTx` in `Endpoint` storage is updated immediately, zero delay enforced. [8](#0-7) 

4. User has no cancel function to call. After 3 days, anyone calls `executeSlowModeTransaction()` — `processSlowModeTransaction` `delegatecall`s into the **new** `EndpointTx`, processing the user's withdrawal under changed rules. [6](#0-5)

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }

    function migrateAll(NewImpl[] calldata newImpls) external onlyOwner {
        for (uint32 i = 0; i < newImpls.length; i++) {
            if (_isEqual(newImpls[i].name, CLEARINGHOUSE_LIQ)) {
                _migrateClearinghouseLiq(newImpls[i]);
            } else if (_isEqual(newImpls[i].name, ENDPOINT_TX)) {
                _migrateEndpointTx(newImpls[i]);
            } else {
                _migrateRegularProxy(newImpls[i]);
            }
            codeHashes[newImpls[i].name] = pendingHashes[newImpls[i].name];
        }
        require(!hasPending(), "still having pending impls to be migrated.");
    }
