### Title
Protobuf `ValidResourceBounds` Deserialization Incorrectly Collapses `AllResources` to `L1Gas`, Producing a Different Transaction Hash and Wrong Execution Mode - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation uses a value-based heuristic ã checking whether `l2_gas` and `l1_data_gas` are both zero ã to decide between `ValidResourceBounds::L1Gas` and `ValidResourceBounds::AllResources`. A transaction originally submitted as `AllResources` with zero `l2_gas` and `l1_data_gas` (valid, e.g., a user who sets no L2/data-gas budget) survives the serialization step with all three fields present, but is silently downgraded to `L1Gas` on deserialization. Because `get_tip_resource_bounds_hash` hashes a different number of elements for the two variants, the transaction hash changes across the protobuf boundary. Additionally, the blockifier's gas-vector computation mode flips from `All` to `NoL2Gas`, altering fee checks and potentially reversing a revert/success outcome.

For the same numeric values, `AllResources` hashes `[tip, l1_gas, l2_gas, l1_data_gas]` (4 felts) while `L1Gas` hashes `[tip, l1_gas, l2_gas]` (3 felts). Poseidon is sensitive to input length, so the two hashes are distinct.

**Execution-mode divergence** (`ValidResourceBounds::get_gas_vector_computation_mode`):

- `AllResources` ã `GasVectorComputationMode::All` ã L2 gas tracked separately, checked against the zero bound ã any L2 gas usage causes a revert.
- `L1Gas` ã `GasVectorComputationMode::NoL2Gas` ã L2 gas folded into L1 gas ã the zero L2 bound is never enforced ã the transaction may succeed instead of reverting.

### Impact Explanation

1. **Wrong transaction hash bound to the executable payload.** After a protobuf round-trip (P2P sync), the hash recomputed from the deserialized transaction differs from the hash the originating node computed and stored. The P2P sync client currently defers hash validation (`// TODO(eitan): Validate transaction hash from untrusted sources`, `crates/apollo_p2p_sync/src/client/transaction.rs` line 88), so the wrong hash is silently stored. When hash validation is enabled, or when the OS/prover recomputes the hash, the mismatch will surface as a block-level inconsistency.

2. **Wrong execution result from the blockifier.** A transaction with `AllResources` and a zero L2-gas bound that uses any L2 gas should revert. After deserialization as `L1Gas`, the same transaction succeeds (L2 gas is folded into L1 gas). This is a wrong revert/success outcome from accepted input, matching the Critical execution-result impact scope.

### Likelihood Explanation

The gateway accepts `RpcInvokeTransactionV3` which carries `AllResourceBounds` (not `ValidResourceBounds`), so any invoke V3 transaction submitted through the RPC is `AllResources`. A user who sets `l2_gas.max_amount = 0` and `l1_data_gas.max_amount = 0` (e.g., to signal "I do not consent to L2/data-gas charges") produces exactly the triggering condition. Such a transaction is valid and will be included in a block. Any peer that syncs the block via P2P will deserialize it as `L1Gas`, compute a different hash, and execute it under different semantics. The trigger requires no special privilege ã any user can submit such a transaction.

### Recommendation

The protobuf deserialization must not infer the `ValidResourceBounds` variant from field values. The canonical approach used by the JSON/storage path (`TryFrom<DeprecatedResourceBoundsMapping>`) is to treat the *presence* of the `L1DataGas` key as the discriminant. For protobuf, the same intent should be encoded explicitly:

- Add a boolean or enum discriminant field to the `ResourceBounds` protobuf message (e.g., `is_all_resources: bool`), **or**
- Treat the *presence* of a non-`None` `l1_data_gas` field as the `AllResources` signal, and always emit `l1_data_gas = None` for `L1Gas` transactions (matching the `unwrap_or_default` fallback path).

The current serializer already emits `l1_data_gas: Some(ResourceBounds::default().into())` for `L1Gas` and `l1_data_gas: Some(l1_data_gas.into())` for `AllResources`, so both paths produce a `Some`. The deserializer therefore cannot distinguish them by presence alone without a protocol change. The safest fix is to add an explicit variant tag to the wire format.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

#[derive(Clone, Copy, Debug, Eq, PartialEq, Hash, Ord, PartialOrd)]
pub enum ValidResourceBounds {
    L1Gas(ResourceBounds), // Pre 0.13.3. Only L1 gas. L2 bounds are signed but never used.
    AllResources(AllResourceBounds),
}

impl From<AllResourceBounds> for ValidResourceBounds {
    fn from(value: AllResourceBounds) -> Self {
        Self::AllResources(value)
    }
}

impl ValidResourceBounds {
    pub fn get_l1_bounds(&self) -> ResourceBounds {
        match self {
            Self::L1Gas(l1_bounds) => *l1_bounds,
            Self::AllResources(AllResourceBounds { l1_gas, .. }) => *l1_gas,
        }
    }

    pub fn get_l2_bounds(&self) -> ResourceBounds {
        match self {
            Self::L1Gas(_) => ResourceBounds::default(),
            Self::AllResources(AllResourceBounds { l2_gas, .. }) => *l2_gas,
        }
    }

    /// Returns the maximum possible fee that can be charged for the transaction.
    /// The computation is saturating, meaning that if the result is larger than the maximum
    /// possible fee, the maximum possible fee is returned.
    pub fn max_possible_fee(&self, tip: Tip) -> Fee {
        match self {
            ValidResourceBounds::L1Gas(l1_bounds) => {
                l1_bounds.max_amount.saturating_mul(l1_bounds.max_price_per_unit)
            }
            ValidResourceBounds::AllResources(AllResourceBounds {
                l1_gas,
                l2_gas,
                l1_data_gas,
            }) => l1_gas
                .max_amount
                .saturating_mul(l1_gas.max_price_per_unit)
                .saturating_add(
                    l2_gas
                        .max_amount
                        .saturating_mul(l2_gas.max_price_per_unit.saturating_add(tip.into())),
                )
                .saturating_add(
                    l1_data_gas.max_amount.saturating_mul(l1_data_gas.max_price_per_unit),
                ),
        }
    }

    pub fn get_gas_vector_computation_mode(&self) -> GasVectorComputationMode {
        match self {
            Self::AllResources(_) => GasVectorComputationMode::All,
            Self::L1Gas(_) => GasVectorComputationMode::NoL2Gas,
        }
    }
