### Title
Fee-on-Transfer Token Accounting Mismatch Overcredits Subaccount Collateral — (`core/contracts/EndpointStorage.sol`, `core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The deposit flow in Nado records the user-supplied `amount` parameter in the slow-mode transaction queue and later credits that same value to the subaccount in the `SpotEngine`, without ever measuring the tokens actually received. For any ERC-20 token that deducts a transfer fee (fee-on-transfer), the Clearinghouse receives fewer tokens than `amount`, yet the subaccount is credited the full `amount`. This inflates internal collateral accounting beyond the real token holdings, creating a solvency shortfall that other depositors ultimately bear.

---

### Finding Description

The deposit path is:

```
Endpoint.depositCollateral()
  → depositCollateralWithReferral()
      → EndpointStorage.handleDepositTransfer()   // actual token movement
      → slowModeTxs[...] = SlowModeTx{ amount: amount }  // records user-supplied value
          → Clearinghouse.depositCollateral()
              → spotEngine.updateBalance(..., amountRealized)  // credits user-supplied value
```

**Step 1 — Token transfer (`EndpointStorage.sol` lines 111–119):**

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);          // user → Endpoint
    safeTransferTo(token, address(clearinghouse), amount); // Endpoint → Clearinghouse
}
```

`safeTransferFrom` and `safeTransferTo` (via `ERC20Helper`) only verify that the call returned `true`; they do not compare the contract's balance before and after to confirm the actual received amount. [1](#0-0) 

**Step 2 — Slow-mode queue records user-supplied `amount` (`Endpoint.sol` lines 152–165):**

```solidity
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    ...
    tx: abi.encodePacked(
        uint8(TransactionType.DepositCollateral),
        abi.encode(
            DepositCollateral({
                sender: subaccount,
                productId: productId,
                amount: amount          // ← user-supplied, not actual received
            })
        )
    )
});
``` [2](#0-1) 

**Step 3 — Clearinghouse credits the user-supplied `amount` (`Clearinghouse.sol` lines 204–207):**

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
```

`txn.amount` is the value stored in the slow-mode queue — the original user-supplied `amount`, not the tokens actually held by the Clearinghouse. [3](#0-2) 

**`ERC20Helper.safeTransferFrom` does not check received amount (`ERC20Helper.sol` lines 23–42):**

```solidity
function safeTransferFrom(...) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.transferFrom.selector, from, to, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Only the boolean return value is checked; the actual token delta is never measured. [4](#0-3) 

---

### Impact Explanation

For a fee-on-transfer token with fee rate `f`:

| Step | Expected | Actual |
|---|---|---|
| User sends `amount` | Endpoint receives `amount` | Endpoint receives `amount × (1-f)` |
| Endpoint forwards `amount` | Clearinghouse receives `amount` | Clearinghouse receives `amount × (1-f)²` (or the second transfer reverts if Endpoint has no surplus balance) |
| Subaccount credited | `amount` | `amount` (overcredited by `amount × (1-(1-f)²)`) |

The subaccount's internal balance in `SpotEngine` exceeds the real token holdings of the Clearinghouse. Across many deposits, the cumulative overcredit grows. When users withdraw, the Clearinghouse pays out based on internal balances; eventually it cannot satisfy all withdrawals, and later withdrawers lose funds. The protocol's solvency invariant — that the sum of all subaccount spot balances equals the Clearinghouse's token holdings — is permanently broken.

---

### Likelihood Explanation

The trigger requires a fee-on-transfer token to be listed as a supported spot product. The protocol does not enforce any restriction against such tokens at the product-listing level. The `IERC20Base` interface is generic and imposes no fee-free guarantee. [5](#0-4)  Any operator or governance action that adds a fee-on-transfer token (e.g., a token that activates fees via an upgradeable proxy, as noted in the analogous report) immediately exposes all depositors to this accounting drift. Likelihood is **Medium**: not exploitable with standard tokens today, but a single token listing decision activates it with no further attacker action required.

---

### Recommendation

1. **Short term**: In `handleDepositTransfer`, measure the Clearinghouse's token balance before and after the transfer and pass the delta — not the user-supplied `amount` — into the slow-mode queue:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // return actualReceived to caller for use in the slow-mode tx
}
```

2. **Long term**: Establish and enforce a token integration checklist that explicitly disallows fee-on-transfer and rebasing tokens from being listed as supported spot products, or add a dedicated wrapper/adapter layer that normalizes such tokens before they interact with the accounting layer.

---

### Proof of Concept

1. A fee-on-transfer token `FeeToken` (1% fee per transfer) is listed as a supported spot product with `productId = 5`.
2. Alice calls `Endpoint.depositCollateral("alice\x00...", 5, 1000e18)`.
3. `handleDepositTransfer` executes:
   - `safeTransferFrom(FeeToken, Alice, 1000e18)` → Endpoint receives `990e18` (1% fee deducted).
   - `safeTransferTo(FeeToken, clearinghouse, 1000e18)` → Clearinghouse receives `980.1e18` (another 1% fee). If Endpoint has no surplus balance, this call reverts; if it does, it drains the surplus.
4. The slow-mode queue records `DepositCollateral{sender: Alice_subaccount, productId: 5, amount: 1000e18}`.
5. When the sequencer (or Alice after the delay) processes the slow-mode tx, `Clearinghouse.depositCollateral` runs: `amountRealized = 1000e18 * multiplier`, and `spotEngine.updateBalance(5, Alice_subaccount, 1000e18 * multiplier)`.
6. Alice's subaccount now shows `1000e18` collateral, but the Clearinghouse only holds `980.1e18` of `FeeToken`.
7. Alice withdraws `1000e18`. The Clearinghouse attempts to send `1000e18` but only holds `980.1e18`, either reverting or draining other users' deposits.
