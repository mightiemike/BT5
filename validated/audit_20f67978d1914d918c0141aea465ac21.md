### Title
Zero-Confirmation Transaction Verification Accepted — (`File: contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations` value of `0` with no minimum enforcement. This is a direct structural analog to the SwapManager `amountOutMinimum = 0` issue: a hardcoded or unchecked floor of zero means the protection mechanism provides no actual guarantee, and any downstream consumer of the verification result can be deceived.

### Finding Description

Both public verification entry points delegate to `verify_transaction_inclusion`, which enforces only an upper bound on `confirmations`: [1](#0-0) 

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

The lower bound check that follows is: [2](#0-1) 

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

When `args.confirmations = 0`, the expression `(anything) >= 0` is trivially true for all `u64` values. The check is vacuous. There is no `require!(args.confirmations >= MIN_CONFIRMATIONS, ...)` guard anywhere in the function.

`verify_transaction_inclusion_v2` — the current recommended entry point — calls the deprecated function directly after its coinbase proof check: [3](#0-2) 

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
```

So the zero-confirmation path is equally reachable through the non-deprecated API.

### Impact Explanation

A downstream NEAR contract (bridge, escrow, DEX settlement) that calls `verify_transaction_inclusion_v2` with `confirmations = 0` receives `true` for a transaction that sits in the current chain tip — the block most exposed to reorganization. An attacker who controls mining power (or exploits a natural reorg) can:

1. Broadcast a Bitcoin transaction T to the network.
2. Call the NEAR contract with `confirmations = 0`; the function returns `true` because the block is on the mainchain and the Merkle proof is valid.
3. The downstream contract releases funds / settles the trade.
4. The attacker mines a competing chain that excludes T, causing a reorg; the original block is evicted from the mainchain.
5. The attacker double-spends T on the new chain.

The corrupted state is the downstream contract's authorization decision: it accepted a transaction that is no longer canonical. The `mainchain_header_to_height` mapping is the concrete on-chain value that changes after the reorg, invalidating the earlier verification result. [4](#0-3) 

### Likelihood Explanation

Any unprivileged NEAR account can call `verify_transaction_inclusion_v2`. The `confirmations` field is part of the caller-supplied `ProofArgsV2` struct; there is no on-chain enforcement of a minimum. A caller that passes `0` — whether by mistake or by intent — receives a verification result that carries no confirmation-depth guarantee. Natural Bitcoin reorgs of 1–2 blocks occur regularly; a deliberate double-spend attack requires only modest hashrate for short reorgs.

### Recommendation

Add an explicit minimum-confirmations guard at the top of `verify_transaction_inclusion`, mirroring the recommendation in the SwapManager report (replace the hardcoded zero with a caller-supplied but bounded value):

```rust
const MIN_CONFIRMATIONS: u64 = 1; // or a protocol-defined constant

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    "Confirmations must be at least MIN_CONFIRMATIONS"
);
```

For security-sensitive consumers (bridges, escrows), document a recommended minimum (e.g., 6 for Bitcoin mainnet) and consider enforcing it on-chain rather than relying on callers to supply a safe value. [5](#0-4) 

### Proof of Concept

1. Deploy the contract (Bitcoin feature, mainnet, `skip_pow_verification = false`).
2. Submit a valid chain of headers so the mainchain tip is at height H.
3. Call `verify_transaction_inclusion_v2` with:
   - `tx_block_blockhash` = hash of the tip block (height H)
   - a valid Merkle proof for a transaction in that block
   - `confirmations = 0`
4. Observe the function returns `true`.
5. Submit a competing fork that reorganizes the tip block out of the mainchain.
6. The previously "verified" transaction is no longer in the canonical chain, but the downstream contract already acted on the `true` result.

The zero-confirmation path bypasses the only meaningful depth check: [2](#0-1) 

because `saturating_sub(...) + 1 >= 0` is always satisfied, making the guard a no-op when `confirmations = 0`.

```rust
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```
