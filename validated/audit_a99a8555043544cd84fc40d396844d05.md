Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on Caller-Supplied `owner` Instead of Actual `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and checks only `owner`, which is a freely caller-supplied parameter. Because any address can name an already-authorized `owner`, the allowlist check is trivially bypassed. The attacker pays tokens into the pool via the liquidity callback, the position is minted to the named owner, and the pool's bin liquidity distribution is permanently altered by an actor the pool admin explicitly excluded.

## Finding Description

`MetricOmmPool.addLiquidity` passes both the real caller and the caller-chosen owner to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but leaves it unnamed and unused, then gates only on `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

An attacker who is not on the allowlist calls `addLiquidity(owner = any_authorized_address, ...)`. The extension sees an authorized `owner`, passes the check, and the pool proceeds to pull tokens from the attacker via the callback and mint a position for the named owner. The attacker cannot reclaim the tokens because `removeLiquidity` enforces `msg.sender == owner`: [3](#0-2) 

The parallel `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller), confirming the asymmetry is unintentional: [4](#0-3) 

## Impact Explanation

The pool admin's deposit allowlist access-control invariant is completely broken for every pool configured with this extension. Any unprivileged address can: (1) add arbitrary liquidity to any bin, shifting the pool's depth profile and effective bid/ask spread for all subsequent swaps; (2) force a position onto an authorized LP address without that LP's consent; (3) conduct cross-market manipulation by depositing liquidity to move the pool's marginal price and profiting via an external position, accepting the token cost as a manipulation fee since the tokens are irrecoverably credited to the named owner. This constitutes an admin-boundary break (unprivileged path bypasses pool admin access control) and broken core pool functionality with direct fund-flow consequences.

## Likelihood Explanation

The bypass requires no special privilege — any EOA or contract can call `addLiquidity`. The only prerequisite is knowing one authorized owner address, which is discoverable on-chain from past `addLiquidity` transactions or from the `allowedDepositor` mapping directly. The attacker must supply real tokens, but those tokens are not lost — they are credited to the named owner, making the attack economically rational whenever manipulation value exceeds token cost. The attack is repeatable indefinitely.

## Recommendation

Replace the ignored first parameter with a named `sender` and gate on it, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to also gate on the position owner, add an explicit `owner` check in addition to the `sender` check.

## Proof of Concept

1. Deploy a pool with `DepositAllowlistExtension`. Add `authorizedLP` to the allowlist; leave `attacker` off it.
2. Deploy `AttackerRouter` (as described in the submission) holding token0/token1 balances.
3. Call `AttackerRouter.exploit(authorizedLP, deltas)`.
4. Inside `exploit`, `pool.addLiquidity(authorizedLP, 0, deltas, "", "")` is called. The extension checks `allowedDepositor[pool][authorizedLP]` — passes. The pool pulls tokens from `AttackerRouter` via `metricOmmModifyLiquidityCallback`. A position is minted under `authorizedLP`.
5. Verify: `pool.position(id, authorizedLP)` shows non-zero shares that `authorizedLP` never requested; the pool's bin liquidity has been modified by an address the pool admin explicitly excluded. `attacker` cannot call `removeLiquidity` to reclaim tokens since `msg.sender != owner` reverts.
