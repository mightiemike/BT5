### Title
`DepositInsurance` Slow Mode Fee Never Collected — Full Amount Credited to Insurance Without Fee Deduction - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `submitSlowModeTransactionImpl`, the `DepositInsurance` branch validates that `txn.amount >= SLOW_MODE_FEE` but then transfers and credits the **full** `txn.amount` to insurance without ever deducting the fee. The `slowModeFees` accumulator is never incremented for this path. This is a direct structural analog to the `initiateRepay` bug in the reference report: a fee sufficiency check exists, but the credited amount is never reduced by the fee, so the protocol never actually collects it.

---

### Finding Description

In `EndpointTx.submitSlowModeTransactionImpl`, the `DepositInsurance` case is handled as follows:

```solidity
} else if (txType == IEndpoint.TransactionType.DepositInsurance) {
    IEndpoint.DepositInsurance memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.DepositInsurance)
    );
    require(
        txn.amount >= uint128(SLOW_MODE_FEE),
        ERR_DEPOSIT_TOO_SMALL
    );
    handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
``` [1](#0-0) 

The check `txn.amount >= uint128(SLOW_MODE_FEE)` is intended to ensure the deposit covers the slow mode fee. However, the full `txn.amount` is transferred to the clearinghouse with no fee deduction, and `slowModeFees` is never incremented.

Compare this to every other non-owner, non-deposit slow mode transaction type, which falls into the `else` branch:

```solidity
} else {
    chargeSlowModeFee(_getQuote(), sender);
    slowModeFees += SLOW_MODE_FEE;
}
``` [2](#0-1) 

For all other user-submitted slow mode transactions, `chargeSlowModeFee` pulls `SLOW_MODE_FEE` from the sender and `slowModeFees` is incremented. For `DepositInsurance`, neither happens.

When the queued transaction is later executed via `processSlowModeTransactionImpl`, it calls `clearinghouse.depositInsurance(transaction)`:

```solidity
} else if (txType == IEndpoint.TransactionType.DepositInsurance) {
    clearinghouse.depositInsurance(transaction);
``` [3](#0-2) 

Inside `Clearinghouse.depositInsurance`, the full `txn.amount` is credited to `insurance` with no fee deduction:

```solidity
int128 amount = int128(txn.amount) * int128(multiplier);
insurance += amount;
``` [4](#0-3) 

The result: the user pays `txn.amount` tokens, the full `txn.amount` is credited to insurance, and the sequencer receives `0` in slow mode fees for this transaction type. The fee check only enforces a minimum deposit size — it does not cause the fee to be collected.

---

### Impact Explanation

The sequencer loses `SLOW_MODE_FEE` ($1 in quote token terms) for every `DepositInsurance` slow mode transaction. The `slowModeFees` state variable — which is later claimed by the sequencer via `DumpFees` — is never incremented for this path. Any caller can submit insurance deposits and bypass the slow mode fee entirely. Repeated calls drain the sequencer's expected fee revenue by $1 per call, with no lower bound on how many times this can be triggered.

The corrupted state delta is: `slowModeFees` is under-counted by `SLOW_MODE_FEE` per `DepositInsurance` submission, and `insurance` is over-credited by the same amount relative to what it should hold after fee deduction.

---

### Likelihood Explanation

The entry point is `Endpoint.submitSlowModeTransaction`, which is `external` and callable by any unprivileged user. No special role, signature, or governance action is required. The only precondition is holding enough quote tokens to meet the `txn.amount >= SLOW_MODE_FEE` check. This is trivially satisfiable and the path is reachable on every supported chain where the Endpoint is deployed.

---

### Recommendation

In `submitSlowModeTransactionImpl`, for the `DepositInsurance` branch, deduct the slow mode fee from the credited amount and increment `slowModeFees`:

```solidity
} else if (txType == IEndpoint.TransactionType.DepositInsurance) {
    IEndpoint.DepositInsurance memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.DepositInsurance)
    );
    require(
        txn.amount >= uint128(SLOW_MODE_FEE),
        ERR_DEPOSIT_TOO_SMALL
    );
    // Charge the slow mode fee separately before depositing
    chargeSlowModeFee(_getQuote(), sender);
    slowModeFees += SLOW_MODE_FEE;
    // Only transfer the net amount to the clearinghouse
    uint256 netAmount = uint256(txn.amount) - uint256(SLOW_MODE_FEE);
    handleDepositTransfer(_getQuote(), sender, netAmount);
    // Update txn.amount to reflect the net deposit
```

Alternatively, adjust the `DepositInsurance` transaction struct so that `txn.amount` represents the net deposit after fee, and enforce `txn.amount + SLOW_MODE_FEE` as the total transfer.

---

### Proof of Concept

1. Attacker (or any user) calls `Endpoint.submitSlowModeTransaction` with a `DepositInsurance` transaction where `txn.amount = SLOW_MODE_FEE` (minimum valid value, e.g., $1 in quote tokens).
2. `submitSlowModeTransactionImpl` passes the `require(txn.amount >= uint128(SLOW_MODE_FEE))` check.
3. `handleDepositTransfer` transfers the full `txn.amount` to the clearinghouse. `slowModeFees` is **not** incremented.
4. After the slow mode delay, anyone calls `executeSlowModeTransaction`.
5. `processSlowModeTransactionImpl` routes to `clearinghouse.depositInsurance(transaction)`.
6. `Clearinghouse.depositInsurance` credits the full `txn.amount` to `insurance`.
7. Net result: the user deposited $1 to insurance, the sequencer received $0 in slow mode fees. Repeating this N times costs the sequencer N × $1 in uncollected fees, with the full deposit amount going to insurance instead of `(deposit - fee)` to insurance and `fee` to the sequencer.     function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }
