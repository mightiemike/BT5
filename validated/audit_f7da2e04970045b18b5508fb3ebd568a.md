### Title
Sanctioned Subaccount Bypasses Withdrawal Restriction via Linked Signer — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `submitSlowModeTransactionImpl` function enforces a sanctions check only on `msg.sender` (the transaction submitter), not on the subaccount whose funds are being acted upon. Because the `linkedSigner` mechanism allows a non-sanctioned address to submit and sign slow-mode transactions on behalf of a sanctioned subaccount, a sanctioned party can withdraw collateral from the protocol by routing the action through their linked signer, bypassing the sanctions restriction entirely.

---

### Finding Description

The Nado protocol maintains a `linkedSigners` mapping that associates a subaccount with a delegated signing address: [1](#0-0) 

When a slow-mode transaction is submitted, the sanction check is performed against `msg.sender` — the address calling `submitSlowModeTransaction` — not against the subaccount owner whose funds are at stake: [2](#0-1) [3](#0-2) 

When the queued slow-mode `WithdrawCollateral` transaction is later processed, `validateSignedTx` is called with `allowLinkedSigner = true`, meaning the linked signer's signature is accepted as valid authorization for the sanctioned subaccount: [4](#0-3) 

The signature validation in `Verifier.validateSignature` accepts either the subaccount owner or the linked signer: [5](#0-4) 

Critically, the `LinkSigner` transaction itself is processed in the fast path (sequencer-submitted) with no sanction check on the subaccount being configured: [6](#0-5) 

This means a sanctioned subaccount can establish or retain a linked signer, and that linked signer — being unsanctioned — can freely submit and sign slow-mode `WithdrawCollateral` transactions on the sanctioned subaccount's behalf.

---

### Impact Explanation

A sanctioned address is able to withdraw collateral from the protocol by routing the withdrawal through a linked signer. This directly violates the protocol's sanctions compliance invariant. The `requireUnsanctioned` check in `depositCollateralWithReferral` explicitly checks both the depositor and the subaccount owner: [7](#0-6) 

The withdrawal path has no equivalent check on the subaccount owner, creating an asymmetric enforcement gap. The concrete asset delta is: full collateral balance of the sanctioned subaccount is exfiltrated.

---

### Likelihood Explanation

Any sanctioned subaccount that has a linked signer configured (either before or after sanctioning, since `LinkSigner` in the fast path has no on-chain sanction check) can execute this bypass. The attacker controls both the sanctioned subaccount and the linked signer address. No privileged access, governance capture, or external dependency is required. The only prerequisite is that the linked signer relationship exists, which is a normal protocol feature used by traders.

---

### Recommendation

The sanction check must be applied to the **subaccount owner** (the address embedded in the subaccount `bytes32`) at the point of slow-mode `WithdrawCollateral` processing, not only to `msg.sender` at submission time. Specifically:

1. In `processTransactionImpl` (and `processSlowModeTransactionImpl`) for `WithdrawCollateral`, extract `address(bytes20(signedTx.tx.sender))` and call `requireUnsanctioned` on it before proceeding.
2. Consider whether `LinkSigner` in the fast path should also enforce a sanction check on the subaccount, to prevent a sanctioned party from establishing a new linked signer post-sanctioning.

---

### Proof of Concept

```
1. Attacker controls sanctioned subaccount A (address 0xSANCTIONED) and linked signer B (address 0xCLEAN).
2. A sets linked signer to B via a fast-path LinkSigner transaction (no on-chain sanction check).
3. B calls submitSlowModeTransaction() with a WithdrawCollateral transaction
   signed by B (as linked signer for A), specifying A's subaccount as sender.
4. In submitSlowModeTransactionImpl:
     sender = msg.sender = B (not sanctioned) → requireUnsanctioned(B) passes.
     Transaction is queued with sender = B.
5. After SLOW_MODE_TX_DELAY (3 days), anyone calls executeSlowModeTransaction().
6. processSlowModeTransaction(B, tx) is called.
7. validateSignedTx(A_subaccount, nonce, tx, sig, true) passes because B is
   the registered linked signer for A_subaccount.
8. clearinghouse.withdrawCollateral executes, transferring A's collateral to
   the withdrawal address — sanctions restriction fully bypassed.
```
