### Title
`LiquidateSubaccount` Has No Permissionless Slow-Mode Trigger — (`core/contracts/EndpointTx.sol`)

---

### Summary

`processSlowModeTransactionImpl` in `EndpointTx.sol` does not handle `TransactionType.LiquidateSubaccount`. Because slow mode is the only permissionless fallback path when the sequencer is unavailable, there is no way for an unprivileged caller to trigger liquidations on underwater subaccounts without sequencer cooperation. This is a direct structural analog to the Malda M-11 finding: a critical action exists in the system but has no reachable trigger from the permissionless entry path.

---

### Finding Description

The Nado protocol routes all transactions through two paths:

1. **Sequencer path** — `submitTransactionsChecked` → `processTransaction` → `processTransactionImpl`. Only callable by the sequencer address.
2. **Slow-mode path** — `submitSlowModeTransaction` → `executeSlowModeTransaction` → `processSlowModeTransaction` → `processSlowModeTransactionImpl`. Callable by any user after a 3-day delay.

`processTransactionImpl` handles `LiquidateSubaccount` correctly: [1](#0-0) 

However, `processSlowModeTransactionImpl` does **not** include a `LiquidateSubaccount` branch. The function ends with: [2](#0-1) 

This means any slow-mode `LiquidateSubaccount` transaction silently reverts and is consumed from the queue without effect (the outer `try/catch` in `_executeSlowModeTransaction` swallows the revert in production): [3](#0-2) 

Compounding the issue, `submitSlowModeTransactionImpl` does **not** block `LiquidateSubaccount` from being submitted — it falls into the generic fee-charging branch: [4](#0-3) 

So a user pays a slow-mode fee, the transaction is queued, and after 3 days it silently fails with no liquidation occurring and no fee refund.

---

### Impact Explanation

If the sequencer is unavailable (maintenance, crash, censorship), underwater subaccounts accumulate unrealized losses with no permissionless mechanism to liquidate them. The insurance fund can become insolvent before the sequencer resumes. `liquidateSubaccountImpl` in `ClearinghouseLiq.sol` is fully implemented and callable — it simply has no reachable trigger from the slow-mode path: [5](#0-4) 

The corrupted state is: `insurance` balance in `ClearinghouseStorage` can go negative (socialization path) because timely liquidations that would have kept it solvent are blocked.

---

### Likelihood Explanation

Sequencer downtime is a realistic operational scenario (upgrades, bugs, network issues). The slow-mode mechanism exists precisely as a permissionless fallback for such events. The 3-day delay is already built into the design for other user actions (withdrawals, link-signer). The absence of liquidation support in this path is a concrete gap, not a theoretical one — any sequencer outage lasting more than the time needed for positions to go deeply underwater creates direct protocol insolvency risk.

---

### Recommendation

Add a `LiquidateSubaccount` branch to `processSlowModeTransactionImpl` in `EndpointTx.sol`, analogous to how `WithdrawCollateral` is handled. The branch should decode the transaction, validate that the sender matches `msg.sender` (or allow any caller for liquidations, since liquidation is permissionless by design), and call `clearinghouse.liquidateSubaccount(txn)`. Alternatively, explicitly revert in `submitSlowModeTransactionImpl` for `LiquidateSubaccount` to prevent users from paying fees for a no-op.

---

### Proof of Concept

1. Subaccount `0xBob...` goes underwater (maintenance health < 0).
2. Sequencer goes offline.
3. Liquidator calls `submitSlowModeTransaction(abi.encodePacked(uint8(TransactionType.LiquidateSubaccount), abi.encode(signedTx)))` — pays `SLOW_MODE_FEE`.
4. After 3 days, liquidator calls `executeSlowModeTransaction()`.
5. `processSlowModeTransactionImpl` hits `else { revert(); }` at line 328.
6. `_executeSlowModeTransaction`'s `try/catch` swallows the revert (line 207).
7. `Bob`'s position remains unliquidated; liquidator's fee is lost; `nSubmissions` increments.
8. Sequencer remains offline; `Bob`'s losses continue to grow against the insurance fund. [6](#0-5) [7](#0-6)
