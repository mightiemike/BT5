### Title
Sync peer can inject `fee_proposal_fri=Some(X)` for pre-V0_14_3 blocks via ungated protobuf converter, corrupting `fee_proposals_window` and `compute_fee_actual` — (`crates/apollo_protobuf/src/converters/header.rs`)

### Summary

The `TryFrom<protobuf::SignedBlockHeader> for SignedBlockHeader` converter accepts `fee_proposal_fri` for any `protocol_version` with no version gate. The `validate_proposal` path enforces `fee_proposal_required = starknet_version >= V0_14_3` and rejects `Some(_)` for pre-V0_14_3 blocks. No equivalent guard exists in the p2p sync path. A sync peer can send a pre-V0_14_3 block with `fee_proposal_fri=Some(X)`, which is stored verbatim, then read back into `fee_proposals_window`, corrupting `compute_fee_actual` for subsequent V0_14_3+ blocks.

### Finding Description

**Step 1 — Ungated converter.** [1](#0-0) 

Line 179 unconditionally maps the wire field to `Some(GasPrice(...))` regardless of `protocol_version`. The `starknet_version` is parsed at line 126–134 but is never consulted before assigning `fee_proposal_fri`.

**Step 2 — Contrast with `validate_proposal` gate.** [2](#0-1) 

The consensus validation path explicitly rejects `(Some(_), false)` (i.e., `fee_proposal_fri` present for pre-V0_14_3). No equivalent check exists in the protobuf converter or anywhere in the p2p sync pipeline.

**Step 3 — P2P sync stores the converted header verbatim.** [3](#0-2) 

`write_to_storage` calls `append_header` directly with the converted `SignedBlockHeader`. No version gate is applied.

**Step 4 — Storage persists `fee_proposal_fri` as-is.** [4](#0-3) 

`append_header` copies `fee_proposal_fri` from `block_header_without_hash` into `StorageBlockHeader` without any version check.

**Step 5 — `fee_proposals_window` is populated from storage without a version gate.** [5](#0-4) [6](#0-5) 

Both `initialize_fee_proposals_window` and `try_sync` read `fee_proposal_fri` from the stored `BlockHeaderWithoutHash` and record it directly into `fee_proposals_window`. A pre-V0_14_3 block that canonically should have `None` will instead have `Some(X)` if the sync peer injected it.

**Step 6 — `compute_fee_actual` is corrupted.** [7](#0-6) 

`compute_fee_actual` returns `None` if **any** window entry is `None` (pre-V0_14_3). By injecting `Some(X)` for a pre-V0_14_3 block, the attacker replaces a `None` entry with `Some(X)`, potentially making `compute_fee_actual` return a non-`None` corrupted median during the V0_14_2→V0_14_3 transition window.

**Step 7 — Corrupted `fee_actual` propagates into fee market validation and proposal building.** [8](#0-7) [9](#0-8) 

The corrupted `fee_actual` is used to validate fee_proposal bounds in `validate_proposal` and to compute the proposer's fee_proposal in `compute_proposer_fee_proposal`. A victim node with a corrupted window will disagree with honest peers on what constitutes a valid fee_proposal.

**Step 8 — The proto spec itself acknowledges the absence of cryptographic protection.** [10](#0-9) 

The proto comment explicitly states: *"WARNING: this field is currently not part of the block hash, so the value must be trusted."* The `BlockHeaderWithoutHash` struct also carries a TODO confirming this: [11](#0-10) 

### Impact Explanation

A sync peer can corrupt the `fee_proposals_window` on any node that syncs from it. During the V0_14_2→V0_14_3 transition, the window contains a mix of `None` (pre-V0_14_3) and `Some` (V0_14_3+) entries. Injecting `Some(X)` for a pre-V0_14_3 block replaces a `None` entry, causing `compute_fee_actual` to return a corrupted median instead of `None`. The corrupted `fee_actual` is then used to:

1. Validate fee_proposal bounds in `validate_proposal` — causing the victim node to reject valid proposals from honest proposers or accept out-of-bounds proposals.
2. Compute the proposer's own fee_proposal — causing the victim node to propose fees that diverge from the network's canonical fee market.

This constitutes an incorrect fee/gas accounting effect with economic impact (corrupted L2 gas price sliding window), matching the Critical impact scope.

### Likelihood Explanation

Any p2p sync peer can trigger this. No special privileges are required. The attack is effective during the V0_14_2→V0_14_3 transition window (when the `fee_proposals_window` contains pre-V0_14_3 blocks). The proto field is optional, so a malicious peer simply sets it on a pre-V0_14_3 block.

### Recommendation

Add a version gate in `TryFrom<protobuf::SignedBlockHeader> for SignedBlockHeader` mirroring the one in `validate_proposal`:

```rust
// After parsing starknet_version:
let fee_proposal_fri = if starknet_version >= StarknetVersion::V0_14_3 {
    value.fee_proposal_fri.map(|v| GasPrice(u128::from(v)))
} else {
    if value.fee_proposal_fri.is_some() {
        return Err(ProtobufConversionError::OutOfRangeValue {
            type_description: "fee_proposal_fri must be absent before V0_14_3",
            value_as_str: value.protocol_version,
        });
    }
    None
};
```

This ensures the sync path enforces the same canonicalization invariant as the consensus validation path.

    );
