### Title
Deprecated `verify_transaction_inclusion` Accepts Internal Merkle Node Hashes as Valid Transaction IDs, Enabling Proof Forgery — (`File: contract/src/lib.rs`)

---

### Summary

The public contract method `verify_transaction_inclusion` remains callable by any unprivileged NEAR account despite being deprecated. It performs no validation that the supplied `tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node hash. An attacker can supply an internal node hash as `tx_id` along with a valid sub-tree proof, causing the function to return `true` for a transaction that does not exist. Consumer contracts (bridges, atomic swaps) that rely on this method can be deceived into accepting forged inclusion proofs.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` but remains a live `pub` method on the deployed contract. In Rust, `#[deprecated]` only emits a compile-time warning for Rust callers; it has no effect on NEAR runtime dispatch. Any external account or contract can call it freely.

The function computes a Merkle root from the caller-supplied `tx_id` and `merkle_proof` and compares it to the stored block header's `merkle_root`: [1](#0-0) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` is a pure path-traversal function. It accepts any 32-byte starting hash and walks up the tree using the supplied proof nodes: [2](#0-1) 

There is no check that `tx_id` is a leaf node. The contract's own inline warning acknowledges this: [3](#0-2) 

Bitcoin's Merkle tree is a binary tree of double-SHA256 hashes. Every internal node is itself a 32-byte hash. An attacker who knows the Merkle tree of any confirmed block (all data is public) can:

1. Select any internal node `N` at depth `d` from the root.
2. Construct the `d`-step sibling path from `N` to the root — this is the valid `merkle_proof`.
3. Set `tx_id = N`, `tx_index` to the position of `N` at depth `d`, and `tx_block_blockhash` to the real block hash.
4. Call `verify_transaction_inclusion`. The function recomputes the root correctly and returns `true`.

The `ProofArgs` struct places no constraints on any of these fields: [4](#0-3) 

The fixed version `verify_transaction_inclusion_v2` mitigates this by requiring a coinbase proof at index 0, which forces the proof to be rooted at a real leaf. However, the old entry point is never removed or access-gated: [5](#0-4) 

---

### Impact Explanation

Consumer contracts (cross-chain bridges, atomic swaps, bridged asset protocols — the stated production use cases of this light client) that call `verify_transaction_inclusion` receive a `bool` result they treat as authoritative. A forged `true` result for a non-existent transaction can cause:

- Unauthorized release of bridged funds on NEAR.
- Acceptance of a fabricated cross-chain state transition.
- Bypass of any on-chain guard that conditions an action on Bitcoin transaction inclusion.

The corrupted value is the **proof result** (`true`) returned to the consumer contract for a `tx_id` that corresponds to no real Bitcoin transaction.

**Impact: High** — direct financial loss in any consumer contract that gates fund release on this verification result.

---

### Likelihood Explanation

- The entry point is public and requires no role, stake, or privileged key.
- All data needed to construct the attack (block Merkle trees) is publicly available from any Bitcoin full node or block explorer.
- The attacker only needs to know which block hash is stored in the contract's mainchain and pick any internal node from that block's Merkle tree.
- No mining, no key compromise, no social engineering required.

**Likelihood: Medium** — requires knowledge of the 64-byte Merkle forgery technique and the contract's stored chain state, but both are trivially obtainable.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract's public interface entirely, or gate it behind a role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts. Deprecation alone is insufficient because NEAR runtime dispatch is not affected by Rust's `#[deprecated]` attribute. The replacement `verify_transaction_inclusion_v2` should be the only callable proof-verification entry point.

---

### Proof of Concept

1. Identify any block hash `B` stored in the contract's mainchain (call `get_block_hash_by_height` for any recent height).
2. Fetch block `B`'s full transaction list from a public Bitcoin node/explorer. Compute the Merkle tree. Select any internal node `N` at depth 1 (i.e., the hash of the first pair of transactions). Its two children are `tx[0]` and `tx[1]`.
3. The proof for `N` at position 0 at depth 1 is a single-element `merkle_proof` containing the sibling of `N` at depth 1 (i.e., the hash of the next pair). The root is the block's `merkle_root`.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash — not a real transaction)
   - `tx_block_blockhash = B`
   - `tx_index = 0` (position of `N` among depth-1 nodes)
   - `merkle_proof = [sibling_of_N]`
   - `confirmations = 1`
5. The function returns `true`. No transaction with ID `N` exists on Bitcoin. [6](#0-5) [2](#0-1)

        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
