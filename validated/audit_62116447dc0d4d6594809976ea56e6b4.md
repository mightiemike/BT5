### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Draining usdcE from DirectDepositV1 Accounts - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` uses a raw, unchecked `transferFrom` to pull USDC from the caller, but uses `safeTransfer` to send usdcE out. If the USDC `transferFrom` returns `false` instead of reverting, the function continues execution, withdrawing usdcE from the target `DirectDepositV1` and sending it to the caller — with no USDC ever received in exchange.

---

### Finding Description

`replaceUsdcEWithUsdc` is an externally callable function (no `onlyOwner` modifier, only a `block.chainid == 57073` guard) that is intended to swap usdcE held in a `DirectDepositV1` contract for real USDC provided by the caller.

The function body is:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // line 616 — unchecked
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));       // line 617
IERC20Base(usdcE).safeTransfer(msg.sender, balance);                 // line 618 — checked
``` [1](#0-0) 

The project defines and uses `ERC20Helper.safeTransferFrom` elsewhere, which wraps the low-level call and checks the boolean return value. [2](#0-1) 

However, line 616 calls `transferFrom` directly on the `IERC20Base` interface, discarding the `bool` return. The outbound `safeTransfer` on line 618 is protected, but the inbound leg is not. [3](#0-2) 

---

### Impact Explanation

If the USDC token on chain 57073 returns `false` on a failed `transferFrom` (e.g., insufficient allowance or balance) rather than reverting, execution continues. The function then:

1. Calls `DirectDepositV1(directDepositV1).withdraw(usdcE)` — pulling all usdcE out of the deposit account into `ContractOwner`.
2. Calls `safeTransfer(msg.sender, balance)` — sending that usdcE to the attacker.

The attacker receives real usdcE tokens without having provided any USDC. Every `DirectDepositV1` account on chain 57073 that holds a usdcE balance is at risk of being drained.

---

### Likelihood Explanation

The function is callable by any address on chain 57073 (Ink chain) — there is no `onlyOwner` or `onlyDeployer` restriction. [4](#0-3) 

The hardcoded USDC address (`0x2D270e6886d130D724215A266106e6832161EAEd`) is a non-standard deployment. Whether this specific token returns `false` or reverts on failure is an external dependency, but the inconsistency is a concrete, reachable code defect regardless. Any token behavior that does not revert on `transferFrom` failure triggers the exploit path.

---

### Recommendation

Replace the raw `transferFrom` call on line 616 with `safeTransferFrom` from `ERC20Helper`, consistent with how the rest of the codebase handles token transfers:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [2](#0-1) 

---

### Proof of Concept

1. A `DirectDepositV1` exists for some `subaccount` on chain 57073 with a non-zero usdcE balance.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` returns `false` (no revert).
4. Execution continues: `DirectDepositV1.withdraw(usdcE)` transfers all usdcE to `ContractOwner`.
5. `safeTransfer(msg.sender, balance)` sends all usdcE to the attacker.
6. Attacker has received usdcE; no USDC was ever deposited. [1](#0-0)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);

        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

**File:** core/contracts/libraries/ERC20Helper.sol (L23-42)
```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
