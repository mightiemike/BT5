### Title
Unconditional `lastKnownBonded = true` in `TierRedelegate` Causes Bonus Overpayment When Redelegating to an Unbonding Validator — (`File: x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`TierRedelegate` resets the position's bonus checkpoint with `lastKnownBonded` hardcoded to `true`, regardless of the destination validator's actual bonded state. When the destination validator is currently unbonding, the next call to `processEventsAndClaimBonus` will treat the entire gap between the redelegation block and the validator's eventual re-bond event as a fully-bonded accrual period, paying out bonus rewards for time the validator was not bonded.

---

### Finding Description

In `TierRedelegate`, after claiming rewards against the source validator, the position's bonus checkpoint is reset for the destination validator:

```go
latestSeq, err := ms.getValidatorEventLatestSeq(ctx, dstValAddr)
...
sdkCtx := sdk.UnwrapSDKContext(ctx)
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), true)   // ← hardcoded true
``` [1](#0-0) 

`UpdateBonusCheckpoints` stores the three fields `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded`:

```go
func (p *Position) UpdateBonusCheckpoints(lastEventSeq uint64, t time.Time, lastKnownBonded bool) {
    p.LastEventSeq = lastEventSeq
    p.LastBonusAccrual = t
    p.LastKnownBonded = lastKnownBonded
}
``` [2](#0-1) 

`LastKnownBonded` is the seed value for the bonded-state replay in `processEventsAndClaimBonus`:

```go
bonded := pos.LastKnownBonded
segmentStart := pos.LastBonusAccrual
...
for _, entry := range events {
    evt := entry.Event
    if bonded {
        bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
        totalBonus = totalBonus.Add(bonus)
    }
    switch evt.EventType {
    case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND:
        bonded = false
    case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND:
        bonded = true
    ...
    }
    segmentStart = evt.Timestamp
``` [3](#0-2) 

**Attack scenario:**

1. Validator B begins unbonding. The staking hook `AfterValidatorBeginUnbonding` fires and appends an `UNBOND` event at sequence `N` for validator B.
2. A user redelegates their tiered position to validator B. Cosmos SDK permits redelegation to an unbonding (non-unbonded) validator.
3. `getValidatorEventLatestSeq` returns `N` (the UNBOND event is the latest). `UpdateBonusCheckpoints(N, blockTime, true)` is called — `LastKnownBonded = true`, `LastEventSeq = N`.
4. Validator B eventually re-bonds. `AfterValidatorBonded` fires and appends a `BOND` event at sequence `N+1`.
5. The user calls `ClaimTierRewards`. `processEventsAndClaimBonus` runs:
   - Seeds `bonded = true` (from `LastKnownBonded`).
   - Fetches events since seq `N`: only the `BOND` event at `N+1` is returned (the `UNBOND` event at `N` is not, because `getValidatorEventsSince` returns events strictly after `LastEventSeq`).
   - Because `bonded == true`, it computes bonus for the segment `[blockTime_of_redelegation, BOND_event_timestamp]` — the entire unbonding gap — and pays it out.

The `UNBOND` event that should have reset `bonded` to `false` is permanently skipped because `LastEventSeq` was set to `N` (the UNBOND event's own seq), so it is never replayed.

---

### Impact Explanation

The position owner receives bonus rewards from the `RewardsPoolName` module account for a time window during which the delegated validator was not bonded. Bonus is computed as `shares × tokensPerShare × BonusAPY × duration / SecondsPerYear`: [4](#0-3) 

The longer the validator remains unbonding before re-bonding, the larger the illegitimate payout. The funds are drawn directly from the `RewardsPoolName` module account via `bankKeeper.SendCoinsFromModuleToAccount`: [5](#0-4) 

The corrupted value is the `RewardsPoolName` module account balance — it is drained by an amount equal to the bonus that should not have accrued.

---

### Likelihood Explanation

- Cosmos SDK allows redelegation to a validator in the `Unbonding` state; only `Unbonded` validators are rejected.
- Validator unbonding periods are typically 21 days, giving a wide window for exploitation.
- No privileged role is required; any position owner can submit a `MsgTierRedelegate` transaction.
- The attacker does not need to control the validator — they only need to observe that a validator has begun unbonding (publicly visible on-chain) and redelegate before it completes.

---

### Recommendation

Before calling `UpdateBonusCheckpoints`, query the actual bonded state of the destination validator and pass it as `lastKnownBonded`:

```go
dstVal, err := ms.stakingKeeper.GetValidator(ctx, dstValAddr)
if err != nil {
    return nil, err
}
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), dstVal.IsBonded())
``` [1](#0-0) 

---

### Proof of Concept

1. Position P is delegated to validator A (bonded). `P.LastKnownBonded = true`.
2. Validator B begins unbonding → `UNBOND` event appended at seq `N`.
3. User submits `MsgTierRedelegate` for position P to validator B.
   - `claimRewards` settles all accrued bonus against validator A.
   - `getValidatorEventLatestSeq(dstValAddr)` → `N`.
   - `pos.UpdateBonusCheckpoints(N, T_redelegate, true)` → `P.LastKnownBonded = true`, `P.LastEventSeq = N`.
4. Validator B re-bonds at time `T_bond` → `BOND` event appended at seq `N+1`.
5. User submits `MsgClaimTierRewards` for position P.
   - `getValidatorEventsSince(valAddr, N)` → returns `[{seq: N+1, BOND, T_bond}]`.
   - Loop iteration: `bonded = true` → `computeSegmentBonus(P, tier, T_redelegate, T_bond, rate)` → non-zero bonus paid.
   - Bonus for the full unbonding gap `[T_redelegate, T_bond]` is transferred from `RewardsPoolName` to the position owner.
   - Correct behavior: zero bonus for this gap (validator was unbonding throughout). [6](#0-5) [7](#0-6)
