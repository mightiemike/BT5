Audit Report

## Title
Unvalidated `timelock` in `OracleValueStopLossExtension.initialize` enables `uint32` truncation in `_afterTimelock`, making the timelock immediately bypassable by the pool admin ‚Äî (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension.initialize` validates `drawdownE6` and `decayPerSecondE8` but applies no validation to the `timelock` parameter. Setting `timelock = type(uint32).max` causes `_afterTimelock` to silently truncate the computed `uint256` deadline back to a past timestamp, making `_requireElapsed` pass immediately. A pool admin can exploit this at pool creation or via `proposeOracleStopLossTimelock` to bypass the timelock that is the sole on-chain protection for LPs against sudden stop-loss parameter changes.

## Finding Description
In `initialize`, the three decoded fields are treated asymmetrically:

```solidity
// L56-62
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);    // capped at E6
_validateDecay(decayPerSecondE8); // capped at E8
// timelock: no validation
oracleStopLossConfig[pool] = PoolStopLossConfig({
    drawdownE6: drawdownE6, decayPerSecondE8: decayPerSecondE8, timelock: timelock, initialized: true
});
```

Every timelocked setter calls `_afterTimelock`:

```solidity
// L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

The addition is performed in `uint256` (no revert), then explicitly cast to `uint32`, which truncates silently in Solidity 0.8. With `timelock = type(uint32).max = 4,294,967,295` and `block.timestamp ‚âà 1,753,000,000`:

```
block.timestamp + type(uint32).max = 6,047,967,295
uint32(6,047,967,295)              = 1,752,999,999  ‚âà block.timestamp ‚àí 1
```

The guard at L301-303:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

evaluates `block.timestamp (uint256) < uint256(executeAfter = block.timestamp ‚àí 1)` ‚Üí **false** ‚Üí no revert. Every `execute*` call succeeds immediately.

The same unvalidated path exists in `proposeOracleStopLossTimelock` (L78-84), which accepts any `uint32 newTimelock` without a cap, enabling the exploit even after a pool is deployed with a legitimate timelock.

## Impact Explanation
The timelock is the only on-chain mechanism constraining the pool admin from immediately weakening or disabling stop-loss parameters. Once bypassed, the admin can immediately set `drawdownE6 = 1e6` (making `floorMultiplier = 0`, permanently disabling the stop-loss breach condition), reset all per-bin watermarks to zero, or set `decayPerSecondE8 = E8` (watermarks decay to zero within one second). With stop-loss disabled, LPs are exposed to unchecked adverse swaps resulting in direct loss of LP principal. This is a confirmed admin-boundary break: pool admin bypasses the timelock constraint that the protocol explicitly imposes on that semi-trusted role.

## Likelihood Explanation
Requires a malicious or compromised pool admin. The pool admin is semi-trusted; the timelock exists precisely because the protocol does not fully trust this role. The attack is non-obvious: `timelock = 4,294,967,295` reads as a "136-year timelock" to any observer, yet produces zero effective delay. The exploit is reachable through the normal `createPool` ‚Üí `extensionInitData` path with no special permissions beyond pool admin, and is repeatable via `proposeOracleStopLossTimelock` even on pools deployed with a legitimate timelock.

## Recommendation
1. **Validate `timelock` in `initialize`** against a sensible maximum (e.g., 365 days = 31,536,000 seconds, well within `uint32` range):
```solidity
uint32 private constant MAX_TIMELOCK = 365 days;
if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
```
2. **Apply the same cap in `proposeOracleStopLossTimelock`** to `newTimelock`.
3. **Harden `_afterTimelock`** to revert on overflow rather than truncate:
```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 result = block.timestamp + oracleStopLossConfig[pool_].timelock;
    require(result <= type(uint32).max, "timelock overflow");
    return uint32(result);
}
```

## Proof of Concept

1. Pool admin calls `factory.createPool(params)` with `extensionInitData = abi.encode(uint32(50_000), uint32(58), uint32(type(uint32).max))`.
