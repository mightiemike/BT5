### Title
Frontrunnable CREATE2 Deployment in `createDirectDepositV1` Causes Victim Transaction Revert — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.createDirectDepositV1()` is a `public` function that deploys a `DirectDepositV1` contract via CREATE2 using a **hardcoded salt** of `bytes32(uint256(1))`. Because the resulting address is fully deterministic and the function has no access control, any attacker can frontrun a legitimate DDA creation call with the same `subaccount` argument, causing the victim's transaction to revert when the CREATE2 opcode finds the target address already occupied.

---

### Finding Description

`createDirectDepositV1` in `ContractOwner.sol` deploys a `DirectDepositV1` contract using Solidity's `new … {salt: …}` syntax:

```solidity
DirectDepositV1 directDepositV1 = new DirectDepositV1{
    salt: bytes32(uint256(1))
}(address(endpoint), address(spotEngine), subaccount, wrappedNative);
directDepositV1Address[subaccount] = payable(directDepositV1);
``` [1](#0-0) 

The CREATE2 address is:

```
keccak256(0xff ++ address(ContractOwner) ++ bytes32(1) ++ keccak256(initcode ++ abi.encode(endpoint, spotEngine, subaccount, wrappedNative)))
```

All four inputs are either fixed protocol constants or the caller-supplied `subaccount`. The salt is **always** `bytes32(uint256(1))` — it never changes. For a given `subaccount`, the address is therefore fully predictable by any observer.

The function carries **no access modifier**:

```solidity
function createDirectDepositV1(bytes32 subaccount)
    public
    returns (address payable)
``` [2](#0-1) 

`creditDepositV1` and `wrapVaultAsset` are also unrestricted `external` entry points that lazily invoke `createDirectDepositV1`:

```solidity
function creditDepositV1(bytes32 subaccount) external {
    address payable directDepositV1 = directDepositV1Address[subaccount];
    if (directDepositV1 == address(0)) {
        directDepositV1 = createDirectDepositV1(subaccount);
    }
    DirectDepositV1(directDepositV1).creditDeposit();
}
``` [3](#0-2) 

An attacker who observes a pending `createDirectDepositV1(victim_subaccount)` (or `creditDepositV1(victim_subaccount)`) in the mempool can submit the same call with higher gas. The attacker's transaction deploys the DDA first, occupying the deterministic address. When the victim's transaction executes, the EVM's CREATE2 finds a non-empty account at that address and the deployment reverts, causing the entire victim transaction to fail.

---

### Impact Explanation

The victim's transaction reverts. The `DirectDepositV1` contract is still deployed at the correct address (by the attacker's call) and is owned by `ContractOwner` — the attacker gains no privileged access. The `directDepositV1Address[subaccount]` mapping is set correctly by the attacker's call, so subsequent `creditDepositV1` calls will succeed. No funds are lost. However, the victim's transaction fails silently, and the victim may believe DDA creation did not succeed, mirroring the exact bad-UX impact described in the reference report.

---

### Likelihood Explanation

Medium. The attack requires only mempool visibility and the ability to submit a transaction with higher gas — standard frontrunning capability available on any public-mempool EVM chain. The `subaccount` argument is visible in the pending transaction's calldata. No privileged access, leaked keys, or social engineering is required.

---

### Recommendation

Apply one of the following mitigations:

1. **Add access control**: Restrict `createDirectDepositV1` to `onlyOwner` or `onlyDeployer`. Since `creditDepositV1` and `wrapVaultAsset` are the intended public entry points, the internal creation helper does not need to be `public`.

2. **Use a unique salt**: Incorporate an ever-increasing nonce counter (analogous to the mitigation accepted in the reference report) or `keccak256(abi.encode(subaccount, nonce++))` as the CREATE2 salt, so no two calls share the same salt even for the same `subaccount`.

---

### Proof of Concept

1. Alice submits `createDirectDepositV1(alice_subaccount)` to the mempool.
2. Attacker observes the pending transaction and extracts `alice_subaccount` from calldata.
3. Attacker submits `createDirectDepositV1(alice_subaccount)` with a higher gas price.
4. Attacker's transaction is mined first; `DirectDepositV1` is deployed at the deterministic CREATE2 address; `directDepositV1Address[alice_subaccount]` is set.
5. Alice's transaction executes: `new DirectDepositV1{salt: bytes32(uint256(1))}(…)` targets the same address, which is now non-empty; the EVM rejects the deployment; the transaction reverts.
6. Alice's DDA exists and is functional, but Alice's transaction failed — she has no on-chain confirmation and must discover the deployed address out-of-band. [2](#0-1)
