Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of Originating User, Allowing Full Allowlist Bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool populates with its own `msg.sender`. When swaps are routed through `MetricOmmSimpleRouter`, that value is the router contract address, not the originating user. Any pool admin who allowlists the router to support legitimate router-mediated swaps simultaneously grants every user on the network the ability to bypass the curated pool's allowlist.

## Finding Description
`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist: [1](#0-0) 

The pool populates that `sender` slot with its own `msg.sender` in `simulateSwapAndRevert` (and identically in the live `swap` path through `_beforeSwap`): [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly on behalf of the user without forwarding the originating caller's identity: [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. In every case, the pool's `msg.sender` is the router contract, so the extension evaluates `allowedSwapper[pool][router]` and never checks `allowedSwapper[pool][user]`.

A pool admin faces an impossible choice: not allowlisting the router blocks all router-mediated swaps (including for legitimately allowlisted users), while allowlisting the router opens the gate to every user on the network.

`DepositAllowlistExtension` does not share this bug because it checks `owner` (the second argument — the LP position recipient supplied by the caller), not `sender`: [4](#0-3) 

## Impact Explanation
Any user excluded from a curated pool's allowlist can execute swaps on that pool by routing through `MetricOmmSimpleRouter`. Swaps execute at full oracle-anchored prices and settle real token transfers. The pool admin's curation policy (KYC gate, institutional-only pool, partner-only liquidity) is completely nullified. This is a direct, fund-impacting bypass of a core access-control mechanism — every swap that should have been blocked goes through, with no loss threshold.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entry point in the periphery. Any production curated pool that wants to support router users must allowlist it, automatically triggering the vulnerability. No privileged access, special token, or malicious setup is required — a standard router call suffices. The bypass is deterministic and repeatable on every block.

## Recommendation
The extension must gate the originating user, not the intermediary. Two complementary approaches:

1. **Pass the original caller through the router.** The router should forward the originating `msg.sender` as an explicit field in `extensionData` (authenticated by the pool's callback context), and the extension should decode and verify it.
2. **Check `tx.origin` as a fallback** (acceptable only if the threat model excludes contract callers; otherwise use approach 1).

## Proof of Concept
```
Setup
─────
1. Deploy a pool with SwapAllowlistExtension attached.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   → alice is allowlisted.
3. Pool admin calls setAllowedToSwap(pool, router, true)  → router allowlisted so alice can use it.
   (Without this step, alice cannot use the router either.)

Attack
──────
4. Bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(...) targeting the pool.
5. Router calls pool.swap(recipient=bob, ...).
6. Pool calls extension.beforeSwap(sender=router, ...).
7. Extension checks: allowedSwapper[pool][router] == true  → passes.
8. Swap executes; Bob receives tokens from the curated pool.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
```
