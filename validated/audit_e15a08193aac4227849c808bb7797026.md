### Title
Router-Mediated Swaps Bypass `SwapAllowlistExtension` by Substituting Router Address for Actual Swapper Identity - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against its per-pool allowlist. The pool passes `msg.sender` of its own `swap()` call as that `sender`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the end user. The allowlist therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`, making the identity check structurally wrong for every router-mediated swap.

### Impact Explanation

Two concrete failure modes arise from the same root cause:

**Mode A — Allowlist bypass (High):** A pool admin who wants to permit router-based swaps adds the router address to the allowlist (`setAllowedToSwap(pool, router, true)`). This is the only way to make the router work with the extension. Once done, *every* user — including those the admin explicitly excluded — can bypass the allowlist by routing through `MetricOmmSimpleRouter`. The curated pool becomes effectively public.

**Mode B — Broken core functionality (High):** If the admin does not add the router, every allowlisted user is silently blocked from using the supported periphery path. The `SwapAllowlistExtension` is a first-party periphery contract; its incompatibility with the first-party router breaks the core swap flow for any pool that deploys it.

Both outcomes satisfy the contest's required impact gate: broken core pool functionality causing loss of funds or unusable swap flows, and admin-boundary break where an unprivileged path bypasses a configured access control.

---

### Likelihood Explanation

The `SwapAllowlistExtension` is the primary access-control primitive for curated pools. Any pool that deploys it and expects to support router-based trading will encounter this issue immediately. No special preconditions, oracle manipulation, or non-standard tokens are required. The attacker only needs to call `MetricOmmSimpleRouter.exactInputSingle` targeting a pool whose allowlist includes the router address.

---

### Recommendation

The extension must receive the true originating user, not the immediate caller. Two options:

1. **Pass the original user through the pool's `swap` signature** — add a `swapper` parameter distinct from `msg.sender` so the router can supply `msg.sender` (the end user) explicitly. The pool would still use `msg.sender` for the callback, but pass the explicit `swapper` to extensions.

2. **Encode the original user in `extensionData`** — the router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires the extension to trust the encoding, which is only safe if the pool validates the caller is a known router.

Option 1 is cleaner and avoids trust assumptions.
