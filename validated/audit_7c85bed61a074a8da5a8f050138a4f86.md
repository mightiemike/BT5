### Title
Stale Slow-Mode `LinkSigner` Bypasses Ownership Check After Address Reuse — (`core/contracts/EndpointTx.sol`)

---

### Summary

The slow-mode `LinkSigner` execution path in `processSlowModeTransactionImpl` validates the submitter only by comparing a stored address value against the address portion of the subaccount bytes32. It performs no check on the *current* controller of that address at execution time. An attacker who briefly occupies an address via CREATE2 can queue a `LinkSigner` slow-mode transaction, destroy the contract, and have the linked signer set to their EOA after the 3-day delay — even after a victim has redeployed at the same address and deposited funds.

### Impact Explanation

Once `linkedSigners[subaccount_A] = attacker_EOA` is written, the attacker's EOA is accepted as a valid signer for all fast-path transactions on `subaccount_A`, including `WithdrawCollateral` and `LinkSigner` (which can further rotate the signer). The sequencer has no mechanism to distinguish a legitimately set linked signer from one set via this stale slow-mode trick — it only checks that the signature is valid and that `linkedSigners[subaccount_A]` matches the signer. If a victim has deposited funds into `subaccount_A` (after the attacker's contract was destroyed and redeployed at A), the attacker can drain those funds by submitting a signed withdrawal to the sequencer.

---

### Likelihood Explanation

The attack requires:

1. **Selfdestruct availability.** Pre-EIP-6780 chains allow post-deployment selfdestruct. On EIP-6780 chains (Cancun+), the attacker can still execute deploy → `depositCollateral` → `submitSlowModeTransaction(LinkSigner)` → `selfdestruct` atomically in a single constructor call, since EIP-6780 permits selfdestruct within the deployment transaction.
2. **Victim redeployment at address A.** The victim must deploy a new contract at the same CREATE2 address and deposit funds. This is realistic if the attacker sells or transfers the CREATE2 salt/deployer to the victim.
3. **Sequencer inclusion.** The sequencer must include the attacker's signed withdrawal. Since the linked signer is legitimately set in storage and the signature is valid, the sequencer has no on-chain signal to reject it.

The combination of prerequisites makes this a medium-likelihood attack, but the impact is full asset theft.

---

### Recommendation

At slow-mode `LinkSigner` execution time, verify that the current code or EOA at the subaccount's address is consistent with the original submitter. Concretely:

- Add an `extcodesize` / `extcodehash` snapshot at submission time and re-verify at execution time, OR
- Require the subaccount address to be an EOA (no code) at execution time if it was an EOA at submission, OR
- Most robustly: require a fresh signature from the subaccount address at execution time (not just a stored sender match), similar to how the fast-path `LinkSigner` uses `validateSignedTx`.

The root fix is that `validateSender` must not be the sole authorization gate for a privileged state mutation like setting a linked signer.

### Proof of Concept

```
1. Attacker deploys AttackerContract at address A via CREATE2 (known salt + deployer).

2. In AttackerContract's constructor (single atomic tx):
   a. Approve Endpoint to spend USDC.
   b. Call Endpoint.depositCollateral(subaccountName, QUOTE_PRODUCT_ID, MIN_FIRST_DEPOSIT_AMOUNT)
      → queues SlowModeTx[N]   { sender: A, tx: DepositCollateral{sender: subaccount_A} }
   c. Call Endpoint.submitSlowModeTransaction(
          abi.encodePacked(uint8(TransactionType.LinkSigner),
              abi.encode(LinkSigner{sender: subaccount_A, signer: bytes32(attacker_EOA)}))
      )
      → queues SlowModeTx[N+1] { sender: A, tx: LinkSigner{sender: subaccount_A, signer: attacker_EOA} }
   d. selfdestruct(attacker_EOA)   // clears code at A (EIP-6780: same tx as deploy)

3. Victim deploys VictimContract at address A (same CREATE2 salt).
   Victim calls Endpoint.depositCollateral(...) depositing 100,000 USDC into subaccount_A.

4. Advance time by 3 days.

5. Call Endpoint.executeSlowModeTransaction() twice:
   - First call processes SlowModeTx[N]:
       validateSender(subaccount_A, A) → passes (A == A)
       _recordSubaccount(subaccount_A) → subaccountIds[subaccount_A] = 1
   - Second call processes SlowModeTx[N+1]:
       validateSender(subaccount_A, A) → passes (A == A)
       requireSubaccount(subaccount_A) → passes (subaccountIds != 0)
       linkedSigners[subaccount_A] = attacker_EOA  ← STATE MUTATION

6. Assert: linkedSigners[subaccount_A] == attacker_EOA  ✓

7. Attacker signs WithdrawCollateral{sender: subaccount_A, amount: 100000e6} with attacker_EOA.
   Sequencer includes it → validateSignedTx passes (linked signer matches) → victim's funds drained.
```
