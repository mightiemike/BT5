### Title
Protobuf `ValidResourceBounds` silently downgrades `AllResources` to `L1Gas` when `l2_gas` and `l1_data_gas` are zero, producing a divergent transaction hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

When a post-0.13.3 `InvokeTransactionV3` carrying `AllResources` bounds with zero `l2_gas` and `l1_data_gas` is round-tripped through the protobuf layer, the converter silently collapses it to `ValidResourceBounds::L1Gas`. This causes `get_tip_resource_bounds_hash` to omit the `L1_DATA_GAS` field from the hash preimage, producing a hash that diverges from the one originally committed. The pattern is the direct sequencer analog of the external bug: a zero-value condition causes a crucial field update to be skipped, leaving one accumulator (the hash) out of sync with the state that was originally signed.

---

### Finding Description

**Step 1 — The zero-condition gate in the protobuf converter**

In `crates/apollo_protobuf/src/converters/transaction.rs`, `ValidResourceBounds::try_from(protobuf::ResourceBounds)` decides the variant based solely on whether the two newer gas fields are zero:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // ← silently defaults
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)                      // ← L1_DATA_GAS dropped
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The comment acknowledges this is a temporary shim: `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.` A pre-0.13.3 transaction legitimately has both fields zero and should become `L1Gas`. A post-0.13.3 transaction that happens to carry zero `l2_gas` and `l1_data_gas` is indistinguishable here and is silently downgraded.

**Step 2 — The hash function branches on the variant**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` includes `L1_DATA_GAS` in the Poseidon preimage only for `AllResources`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // ← L1_DATA_GAS absent
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [2](#0-1) 

`L1Gas` → 2-element preimage `[tip, L1_GAS_concat, L2_GAS_concat]`.  
`AllResources` → 3-element preimage `[tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]`.

These produce different Poseidon digests even when `l1_data_gas` and `l2_gas` are zero, because `poseidon_hash_update_with_nested_hash` over an empty/zero element still contributes to the state.

**Step 3 — The divergence**

A post-0.13.3 `InvokeTransactionV3` submitted through the gateway is stored internally as `InternalRpcInvokeTransactionV3` with `AllResourceBounds`. The `From` conversion to `InvokeTransactionV3` always wraps it as `AllResources`:

```rust
impl From<InternalRpcInvokeTransactionV3> for InvokeTransactionV3 {
    fn from(tx: InternalRpcInvokeTransactionV3) -> Self {
        Self {
            resource_bounds: ValidResourceBounds::AllResources(tx.resource_bounds),
            ...
        }
    }
}
``` [3](#0-2) 

The hash committed at ingestion time uses the 3-element preimage. After the transaction is serialized to protobuf and deserialized on a peer (P2P sync path), `ValidResourceBounds::try_from` produces `L1Gas`, and any subsequent call to `validate_transaction_hash` → `get_transaction_hash` → `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` recomputes the hash with the 2-element preimage. The two hashes differ. [4](#0-3) 

---

### Impact Explanation

**Impact: High — Transaction conversion or signature/hash logic binds the wrong hash/executable payload.**

Any node that receives a post-0.13.3 `InvokeTransactionV3` with zero `l2_gas` and `l1_data_gas` via the P2P sync path will recompute a hash that does not match the committed hash. `validate_transaction_hash` will return `false`, causing the node to reject a valid transaction (or the block containing it). This is a correctness break in the hash-binding invariant across the public-to-internal conversion boundary.

---

### Likelihood Explanation

**Likelihood: Low-Medium.**

The gateway's stateless validator explicitly allows individual resource bounds to be zero (test cases `valid_l1_gas`, `valid_l2_gas`, `valid_l1_data_gas` each set the other two to zero). [5](#0-4) 

A transaction with only `l1_gas` non-zero and both `l2_gas` and `l1_data_gas` zero is therefore a valid, accepted post-0.13.3 transaction. Any such transaction propagated via P2P sync triggers the divergence.

---

### Recommendation

The protobuf converter must not use the zero-value heuristic to select the `ValidResourceBounds` variant for transactions that are known to be post-0.13.3. Options:

1. **Carry a version tag in the protobuf message** so the deserializer knows which variant was originally used, rather than inferring it from field values.
2. **Always deserialize as `AllResources`** when the protobuf message originates from a post-0.13.3 block, and reserve the `L1Gas` downgrade only for blocks below the 0.13.3 activation height.
3. **Remove the heuristic entirely** once support for 0.13.2 is dropped (as the TODO comment already notes), and assert `l1_data_gas` is always present.

---

### Proof of Concept

1. Submit an `InvokeTransactionV3` with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` through the gateway. The gateway accepts it; the hash `H_orig` is computed using the 3-element `AllResources` preimage.
2. The transaction is included in a block and propagated to a peer via P2P sync as a protobuf `Transaction`.
3. The peer's `ValidResourceBounds::try_from(protobuf::ResourceBounds)` sees `l2_gas.is_zero() && l1_data_gas.is_zero()` → produces `ValidResourceBounds::L1Gas(l1_gas)`.
4. The peer calls `validate_transaction_hash(transaction, ...)` which calls `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `L1Gas` → 2-element preimage → hash `H_peer ≠ H_orig`.
5. `validate_transaction_hash` returns `false`; the peer rejects the transaction/block. [6](#0-5) [7](#0-6)

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
