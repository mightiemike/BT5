### Title
Linked Signer Not Revoked on Full Withdrawal Enables Stale Signer to Drain Re-deposited Funds — (`core/contracts/EndpointTx.sol`)

---

### Summary

When a user fully withdraws all collateral, the `linkedSigners` mapping in `EndpointStorage` is never cleared. A previously-linked signer retains indefinite signing authority over the subaccount. If the user re-deposits, the stale linked signer can immediately sign a `TransferQuote` transaction to redirect the new funds to an attacker-controlled subaccount.

---

### Finding Description

**State set on link:**

`linkedSigners[txn.sender]` is written in two places — the slow-mode path and the sequencer path: [1](#0-0) [2](#0-1) 

**State never cleared on withdrawal:**

Both `WithdrawCollateral` and `WithdrawCollateralV2` paths call `clearinghouse.withdrawCollateral(...)` but make no call to clear `linkedSigners[sender]`: [3](#0-2) [4](#0-3) 

**Stale signer used in `TransferQuote`:**

`TransferQuote` is processed with `allowLinkedSigner = true`, meaning the stale linked signer's signature is accepted as authoritative for the emptied subaccount: [5](#0-4) 

The recipient of `TransferQuote` is unconstrained — it can be any registered subaccount, including one controlled by the attacker: [6](#0-5) 

**Stale signer lookup:**

`getLinkedSigner` reads directly from the persistent `linkedSigners` mapping with no balance check: [7](#0-6) 

**Persistent mapping, never zeroed:** [8](#0-7) 

---

### Impact Explanation

A user who deposited, linked a signer (e.g., a trading bot or third-party service), and then fully withdrew all collateral leaves `linkedSigners[subaccount]` pointing to the old signer. If the user re-deposits — even from a fresh deposit — the old signer immediately holds full signing authority. The stale signer can submit a `TransferQuote` transaction (signed with the old key) to move the re-deposited funds to an attacker-controlled subaccount, from which a normal `WithdrawCollateral` drains them to an external address. This is a direct fund-theft path with no recovery mechanism for the victim.

---

### Likelihood Explanation

The scenario is realistic: users routinely link signers for automated trading, then exit the protocol and return later. The protocol provides no indication that the linked signer persists after full withdrawal, and no automatic cleanup occurs. The sequencer processes `TransferQuote` transactions signed by the linked signer as valid — it has no basis to distinguish a "stale" signer from an active one. The only barrier is that the attacker must control the previously-linked signer key, which is plausible if the key was shared with a third-party service, compromised, or the user intentionally linked a malicious party.

---

### Recommendation

In `Clearinghouse.withdrawCollateral`, after reducing the subaccount's balance to zero, emit a callback or return a flag to `EndpointTx` so that `linkedSigners[subaccount]` is set to `address(0)`. Alternatively, add an explicit check in `getLinkedSigner`: if the subaccount's total health (or quote balance) is zero, return `address(0)` to suppress stale signer authority. The slow-mode `LinkSigner` path already enforces `validateSender` (owner-only), so clearing on zero-balance withdrawal is the minimal fix needed.

---

### Proof of Concept

1. Alice calls `depositCollateral` → subaccount registered, funds credited.
2. Alice submits a `LinkSigner` slow-mode transaction → `linkedSigners[Alice_subaccount] = Bob`.
3. Alice submits `WithdrawCollateral` for her full balance → balance reaches zero. `linkedSigners[Alice_subaccount]` is **not** cleared; it still equals `Bob`.
4. Alice re-deposits collateral (e.g., next month).
5. Bob (stale signer, or attacker who compromised Bob's key) constructs a `SignedTransferQuote` with:
   - `sender = Alice_subaccount`
   - `recipient = Attacker_subaccount`
   - `amount = Alice's full balance`
   - Signed with Bob's key.
6. Sequencer includes the transaction in `submitTransactionsChecked` → `processTransactionImpl` reaches the `TransferQuote` branch → `validateSignedTx(..., allowLinkedSigner=true)` → `getLinkedSigner(Alice_subaccount)` returns `Bob` → signature validates → `clearinghouse.transferQuote(...)` moves Alice's funds to `Attacker_subaccount`.
7. Attacker submits `WithdrawCollateral` from `Attacker_subaccount` → funds exit the protocol. [7](#0-6) [9](#0-8) [8](#0-7)
