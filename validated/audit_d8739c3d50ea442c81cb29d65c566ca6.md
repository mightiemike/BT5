### Title
`verify_transaction_inclusion_v2` Coinbase Proof Bypass via `tx_id == coinbase_tx_id` — (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery attack by requiring a separate coinbase proof of equal length. However, there is no validation that `tx_id != coinbase_tx_id`. When an attacker supplies the same hash for both fields, the coinbase proof check becomes algebraically identical to the transaction proof check, providing zero additional security. The 64-byte forgery attack succeeds against v2 exactly as it does against the deprecated v1.

### Finding Description

`verify_transaction_inclusion_v2` performs two sequential Merkle root computations:

1. **Coinbase check** — `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == merkle_root`
2. **Transaction check** — `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == merkle_root` (via the v1 call) [1](#0-0) 

The only cross-field validation between the two proof sets is that their lengths are equal: [2](#0-1) 

There is no check that `args.tx_id != args.coinbase_tx_id`. [3](#0-2) 

**Concrete attack path** — Consider a Bitcoin block whose Merkle tree has depth 3 (8 transactions). The internal node at depth 2 is `N = SHA256d(T0 || T1)`. An attacker constructs:

| Field | Value |
|---|---|
| `tx_id` | `N` (internal node) |
| `coinbase_tx_id` | `N` (same internal node) |
| `tx_index` | `0` |
| `merkle_proof` | `[H(T2,T3), H(H(T4,T5),H(T6,T7))]` — length 2, not 3 |
| `coinbase_merkle_proof` | identical to `merkle_proof` |

Step 1 (coinbase check):
```
compute_root_from_merkle_proof(N, 0, [H(T2,T3), H(H(T4,T5),H(T6,T7))])
= H(H(N, H(T2,T3)), H(H(T4,T5),H(T6,T7)))
= H(H(H(T0,T1), H(T2,T3)), H(H(T4,T5),H(T6,T7)))
= merkle_root  ✓
```

Step 2 (transaction check): identical computation — also passes. The function returns `true` for a forged proof.

The `compute_root_from_merkle_proof` implementation in `merkle-tools/src/lib.rs` performs no depth or leaf-position validation; it simply walks the proof array: [4](#0-3) 

### Impact Explanation

Any NEAR contract or user that calls `verify_transaction_inclusion_v2` and trusts its boolean result can be deceived into accepting a Bitcoin transaction inclusion proof for a transaction that was never included in the block. The attacker supplies an internal Merkle node as the claimed transaction hash, bypasses the coinbase-depth guard by reusing the same hash for `coinbase_tx_id`, and receives a `true` return value. Downstream contracts that gate fund releases, cross-chain bridge actions, or other state transitions on this result are directly exploitable.

### Likelihood Explanation

The call is fully permissionless — `verify_transaction_inclusion_v2` carries no access-control restriction and is callable by any NEAR account. The attacker needs only a valid block header already stored in the contract (submitted by the relayer) and knowledge of any internal Merkle node in that block, both of which are public on-chain data. No privileged role, key material, or social engineering is required.

### Recommendation

Add an explicit guard at the top of `verify_transaction_inclusion_v2` that rejects calls where `tx_id` equals `coinbase_tx_id`:

```rust
require!(
    args.tx_id != args.coinbase_tx_id,
    "tx_id must not equal coinbase_tx_id"
);
```

Additionally, enforce `args.tx_index != 0` when the caller is not verifying the coinbase itself, or document and enforce that `coinbase_tx_id` must be the actual coinbase transaction of the block (index 0), which is structurally distinct from any non-coinbase `tx_id`.

### Proof of Concept

Given a real mainnet block with known Merkle tree structure:

1. Identify internal node `N = SHA256d(coinbase_txid || tx1_hash)`.
2. Call `verify_transaction_inclusion_v2` with:
   - `tx_id = N`
   - `coinbase_tx_id = N`
   - `tx_index = 0`
   - `merkle_proof = coinbase_merkle_proof = [tx1_sibling, depth1_sibling, ...]` (length = tree_depth − 1)
   - `tx_block_blockhash` = any valid block hash stored in the contract
   - `confirmations = 1`
3. The coinbase check passes because `N` at index 0 with the shortened proof reconstructs the root.
4. The transaction check passes for the same reason.
5. The function returns `true`, falsely asserting that `N` is an included transaction. [5](#0-4) [3](#0-2)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
