Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is the direct caller of `pool.swap`. When any user routes through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router`, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. If a pool admin allowlists the router to permit router-mediated swaps for their curated users, every unprivileged address can bypass the allowlist by routing through the same public router contract.

## Finding Description
`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)` directly: [1](#0-0) 

This makes the router the `msg.sender` to the pool. The pool passes `msg.sender` as the `sender` argument to `_beforeSwap` in `ExtensionCalling`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router — not the end user: [3](#0-2) 

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to curated addresses.
2. Admin calls `setAllowedToSwap(pool, router, true)` to allow router-mediated swaps for their allowlisted users.
3. Any unprivileged address calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting that pool.
4. The pool receives `msg.sender = router`; the extension checks `allowedSwapper[pool][router] == true` and passes.
5. The unprivileged user's swap executes successfully, bypassing the allowlist entirely.

No existing guard prevents this: `allowAllSwappers` is a separate flag, and there is no mechanism in the extension or the router to recover the original `msg.sender` (the end user) from the router call.

## Impact Explanation
The allowlist is the sole access-control mechanism for restricting who can swap in a pool. A full bypass means any unprivileged address can swap in a pool intended to be restricted, breaking the core pool access-control invariant. Depending on pool configuration, this can result in unauthorized trades against curated liquidity, direct loss of LP assets, or violation of regulatory/compliance constraints enforced by the pool admin.

## Likelihood Explanation
The condition is trivially reachable: any unprivileged user with knowledge that the router is allowlisted can call the public `MetricOmmSimpleRouter` functions. No special privileges, flash loans, or complex setup are required. The router is a public, permissionless contract. The attack is repeatable on every block.

## Recommendation
The extension should verify the actual end-user identity, not the intermediary. Two options:
1. **Pass the original user through the router:** Have `MetricOmmSimpleRouter` pass `msg.sender` (the end user) as a verified `sender` via `extensionData`, and have the extension decode and verify it — but this requires trust in the router.
2. **Check the caller of the router, not the pool:** Redesign the allowlist to check `tx.origin` (not recommended for security reasons) or, preferably, require users to call the pool directly when allowlist enforcement is needed.
3. **Preferred fix:** Do not allowlist the router address. Instead, require allowlisted users to call `pool.swap` directly, or introduce a dedicated allowlisted-router that forwards the original caller's identity in a verifiable way (e.g., signed message or trusted forwarder pattern checked by the extension).

// 3. Unprivileged attacker calls:

    pool: restrictedPool,
    recipient: attacker,

    deadline: block.timestamp,
