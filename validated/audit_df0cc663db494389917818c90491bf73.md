### Title
Missing Refund Mechanism for Failed Slow-Mode `DepositCollateral` Transactions — (`Endpoint.sol`)

---

### Summary

In `Endpoint.depositCollateralWithReferral`, user tokens are transferred to the contract immediately at call time, but the corresponding `DepositCollateral` credit is only applied when the queued slow-mode transaction is later executed. If that execution fails, the failure is silently swallowed by a `try/catch` with **no refund path**. The codebase itself contains the comment `// try return funds now removed` in the catch block, explicitly confirming a refund mechanism was removed.

---

### Finding Description

`Endpoint.depositCollateralWithReferral` performs two sequential steps:

**Step 1 — Immediate token transfer (assets committed):** [1](#0-0) 

**Step 2 — Slow-mode transaction queued (credit deferred):** [2](#0-1) 

The slow-mode transaction is executed later (minimum 3-day delay) via `_executeSlowModeTransaction`, which wraps execution in a `try/catch`: [3](#0-2) 

When the inner call reverts, the catch block does nothing — and the comment at line 226 reads:

```
// try return funds now removed
```

This is a direct admission that a refund mechanism existed and was deliberately removed. [4](#0-3) 

Concrete failure scenarios during the 3-day window between deposit and execution:

- The deposited `productId` is delisted via `delistProduct`, making `clearinghouse.depositCollateral` revert
- The subaccount or depositor address is added to the sanctions list (`requireUnsanctioned` check inside `processSlowModeTransaction`)
- Any other state change that causes `clearinghouse.depositCollateral` to revert

In all these cases, the user's ERC-20 tokens are already held by the `Endpoint` contract with no on-chain mechanism to recover them.

---

### Impact Explanation

The user's deposited tokens are permanently locked in the `Endpoint` contract. The balance in the `SpotEngine` is never credited (since the slow-mode tx failed), and there is no function to reclaim the transferred tokens. The asset delta is: user loses `amount` tokens of `productId` with zero credit to their subaccount.

---

### Likelihood Explanation

Medium. The 3-day `SLOW_MODE_TX_DELAY` creates a meaningful window during which protocol state can change. Product delisting is an explicit admin operation (`delistProduct` via `EndpointTx`), and sanctions screening is an active runtime check. Both are realistic triggers. The entry path requires only a standard user deposit call — no special privileges.

---

### Recommendation

Restore a refund mechanism in the catch block of `_executeSlowModeTransaction` for `DepositCollateral` transactions. The catch block should decode the transaction type, and if it is `DepositCollateral`, transfer the deposited tokens back to the original depositor. The original depositor address is recoverable from the queued `SlowModeTx.sender` field and the encoded `DepositCollateral.sender` payload.

---

### Proof of Concept

1. User calls `Endpoint.depositCollateral(subaccountName, productId, amount)`.
2. `depositCollateralWithReferral` is invoked; `handleDepositTransfer` pulls `amount` tokens from the user into the `Endpoint` contract.
3. A `SlowModeTx` is enqueued with `executableAt = block.timestamp + 3 days`.
4. Before the 3-day window expires, the owner calls `delistProduct` for `productId`.
5. After 3 days, anyone calls `executeSlowModeTransaction()`.
6. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(...)` inside a `try`.
7. `clearinghouse.depositCollateral` reverts because the product no longer exists.
8. The `catch` block executes — comment reads `// try return funds now removed` — no refund is issued.
9. User's `amount` tokens remain locked in `Endpoint` with no recovery path. [5](#0-4) [6](#0-5)

```text
    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });
        slowModeConfig = _slowModeConfig;
    }
```

```text
    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
