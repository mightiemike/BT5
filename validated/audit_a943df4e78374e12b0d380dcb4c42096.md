### Title
Lack of Slippage Control in `mintNlp` and `burnNlp` — Oracle Price Excluded from User's EIP-712 Digest Allows Sequencer to Execute at Arbitrary Price — (`Clearinghouse.sol` / `Verifier.sol`)

---

### Summary

The EIP-712 digest that users sign for `MintNlp` and `BurnNlp` transactions does **not** include `oraclePriceX18`. The sequencer freely supplies this price at execution time. Because the NLP amount received (for mint) and the quote amount received (for burn) are computed entirely from `oraclePriceX18`, users have no on-chain protection against receiving fewer NLP tokens or less quote than they expected. This is the direct Nado analog of the AuraVault slippage bug: users cannot commit to a minimum output, and the contract enforces no bound on the price used.

---

### Finding Description

**Root cause — EIP-712 digest omits `oraclePriceX18`**

In `Verifier.sol`, the typed-data signatures for NLP operations are defined as:

```solidity
string internal constant MINT_NLP_SIGNATURE =
    "MintNlp(bytes32 sender,uint128 quoteAmount,uint64 nonce)";
string internal constant BURN_NLP_SIGNATURE =
    "BurnNlp(bytes32 sender,uint128 nlpAmount,uint64 nonce)";
``` [1](#0-0) 

The corresponding digest computation for `MintNlp` hashes only `sender`, `quoteAmount`, and `nonce`:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(MINT_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.quoteAmount,
        signedTx.tx.nonce
    )
);
``` [2](#0-1) 

And for `BurnNlp`, only `sender`, `nlpAmount`, and `nonce`: [3](#0-2) 

The `oraclePriceX18` field present in `SignedMintNlp` and `SignedBurnNlp` is **never hashed into the digest the user signs**:

```solidity
struct SignedMintNlp {
    MintNlp tx;
    bytes signature;
    int128 oraclePriceX18;       // ← not covered by user's signature
    int128[] nlpPoolRebalanceX18;
}
``` [4](#0-3) 

**Execution path — sequencer injects arbitrary price**

In `EndpointTx.sol`, the sequencer decodes the full `SignedMintNlp` struct, validates the user's signature (which does not cover `oraclePriceX18`), and then passes the sequencer-chosen price directly to `Clearinghouse.mintNlp`:

```solidity
clearinghouse.mintNlp(
    signedTx.tx,
    signedTx.oraclePriceX18,   // ← sequencer-supplied, not user-committed
    nlpPools,
    signedTx.nlpPoolRebalanceX18
);
``` [5](#0-4) 

The same pattern applies to `burnNlp`: [6](#0-5) 

**State delta — NLP/quote amounts computed from unconstrained price**

In `Clearinghouse.mintNlp`, the NLP tokens credited to the user are:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [7](#0-6) 

A higher `oraclePriceX18` → fewer NLP tokens for the same `quoteAmount`. There is no `minNlpAmount` check anywhere.

In `Clearinghouse.burnNlp`, the quote credited to the user is:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [8](#0-7) 

A lower `oraclePriceX18` → less quote returned for the same `nlpAmount`. There is no `minQuoteAmount` check anywhere.

---

### Impact Explanation

- **`mintNlp`**: A user signs a transaction to spend `quoteAmount` of quote token. The sequencer can submit the transaction with an inflated `oraclePriceX18`, causing the user to receive materially fewer NLP tokens than the current market rate implies. The user's quote balance is debited in full; the NLP credit is silently reduced.
- **`burnNlp`**: A user signs a transaction to burn `nlpAmount` of NLP. The sequencer can submit with a deflated `oraclePriceX18`, causing the user to receive materially less quote than the current market rate implies. The NLP is burned in full; the quote credit is silently reduced.

In both cases the user's signed intent (spend X, receive fair-value Y) is not enforced on-chain. The corrupted state is the user's `QUOTE_PRODUCT_ID` and `NLP_PRODUCT_ID` balances in `SpotEngine`.

---

### Likelihood Explanation

The sequencer is the only entity that can submit sequenced transactions. The `oraclePriceX18` field is entirely under sequencer control and is not constrained by any on-chain bound or user commitment. Even without deliberate manipulation, price movements between the moment a user signs and the moment the sequencer executes can cause meaningful slippage, and the user has no mechanism to reject execution at an unfavorable price. The entry path is the normal `MintNlp`/`BurnNlp` transaction flow used by every NLP participant.

---

### Recommendation

Include `minNlpAmount` in the `MintNlp` struct and `minQuoteAmount` in the `BurnNlp` struct, add them to the EIP-712 type strings and digest computation in `Verifier.sol`, and enforce them in `Clearinghouse.mintNlp` / `Clearinghouse.burnNlp`:

```diff
// IEndpoint.sol
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
+   uint128 minNlpAmount;
    uint64 nonce;
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
+   uint128 minQuoteAmount;
    uint64 nonce;
}
```

```diff
// Verifier.sol
string internal constant MINT_NLP_SIGNATURE =
-   "MintNlp(bytes32 sender,uint128 quoteAmount,uint64 nonce)";
+   "MintNlp(bytes32 sender,uint128 quoteAmount,uint128 minNlpAmount,uint64 nonce)";

string internal constant BURN_NLP_SIGNATURE =
-   "BurnNlp(bytes32 sender,uint128 nlpAmount,uint64 nonce)";
+   "BurnNlp(bytes32 sender,uint128 nlpAmount,uint128 minQuoteAmount,uint64 nonce)";
```

```diff
// Clearinghouse.sol — mintNlp
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
+ require(uint128(nlpAmount) >= txn.minNlpAmount, "mintNlp: slippage");

// Clearinghouse.sol — burnNlp
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
+ require(uint128(quoteAmount) >= txn.minQuoteAmount, "burnNlp: slippage");
```

### Proof of Concept

1. Alice signs a `MintNlp` transaction: `quoteAmount = 1000e18`, `nonce = 5`. At signing time the NLP oracle price is `10e18`, so she expects to receive `100e18` NLP tokens.
2. The sequencer delays execution. The NLP price rises to `20e18`.
3. The sequencer submits `SignedMintNlp` with `oraclePriceX18 = 20e18`. Alice's signature is valid because `oraclePriceX18` is not in the digest.
4. `Clearinghouse.mintNlp` computes `nlpAmount = 1000e18 / 20e18 = 50e18`. Alice receives only 50 NLP instead of 100.
5. Alice's quote balance is debited by the full `1000e18`; she has lost half the expected NLP value with no on-chain recourse.

The same scenario applies symmetrically to `burnNlp` with a deflated oracle price reducing the quote returned to the user.
