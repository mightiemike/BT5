### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable Without Coinbase Proof Guard, Enabling 64-Byte Merkle Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is marked `#[deprecated]` in Rust, which is a compiler-only warning with no runtime enforcement. The function remains a live, publicly callable NEAR entry point. It is missing the coinbase Merkle proof check that `verify_transaction_inclusion_v2` introduces to block the 64-byte transaction Merkle proof forgery attack. Any unprivileged NEAR caller can invoke the deprecated function directly, bypassing the guard entirely, and obtain a forged `true` proof result for a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` is exposed as a public `#[pause]`-gated (but not role-gated) contract method:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    ...
    require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root
}
```

The only input guard added is `!args.merkle_proof.is_empty()`. There is no check that `args.tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node. The 64-byte forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) exploits exactly this: a 64-byte blob that is structurally identical to two concatenated 32-byte hashes can be submitted as `tx_id`. When `compute_root_from_merkle_proof` processes it, the computed root matches the block's real `merkle_root`, causing the function to return `true` for a non-existent transaction.

`verify_transaction_inclusion_v2` closes this gap by first verifying that the coinbase transaction (always at index 0) is present in the same Merkle tree:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
```

This guard is absent in v1. Because `#[deprecated]` is a Rust compiler attribute with no on-chain effect, v1 is callable by any NEAR account at any time. `verify_transaction_inclusion_v2` itself calls v1 internally (`#[allow(deprecated)] self.verify_transaction_inclusion(args.into())`), but that path goes through the coinbase check first. A direct call to v1 skips it entirely.

---

### Impact Explanation

Any dApp or cross-chain bridge that calls `verify_transaction_inclusion` to gate a security-sensitive action (e.g., releasing funds, minting tokens, confirming a Bitcoin payment) can be deceived. An attacker supplies a crafted `tx_id` equal to an internal Merkle tree node of a real, already-confirmed block, along with a valid partial Merkle path. The function returns `true`, and the consuming contract treats a non-existent transaction as confirmed. The corrupted value is the **proof result** returned by the contract.

---

### Likelihood Explanation

The 64-byte forgery technique is publicly documented and requires no privileged access. The attacker only needs: (1) a confirmed Bitcoin block whose Merkle tree structure is known (all blocks are public), (2) the ability to call a NEAR contract function (any account), and (3) knowledge that the deprecated v1 entry point still exists. No key compromise, social engineering, or special role is required.

---

### Recommendation

Add a runtime guard to `verify_transaction_inclusion` that unconditionally panics, preventing direct invocation:

```rust
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    env::panic_str("verify_transaction_inclusion is deprecated; use verify_transaction_inclusion_v2");
}
```

Alternatively, remove the function entirely from the public ABI. The coinbase-proof-validated path in `verify_transaction_inclusion_v2` is the only safe entry point and should be the sole callable method.

---

### Proof of Concept

1. Identify any confirmed Bitcoin block `B` with at least two transactions. Its `merkle_root` is stored in the contract's `headers_pool`.
2. Select any internal Merkle tree node `N` of `B` (e.g., the hash of `tx[0]` and `tx[1]`). `N` is 32 bytes and is a valid `H256`.
3. Construct `merkle_proof` as the sibling path from `N`'s position upward to the root (this is a standard Merkle path, publicly computable from the block).
4. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B.hash`, `tx_index = <position of N in the tree>`, `merkle_proof = <path>`, `confirmations = 1`.
5. `compute_root_from_merkle_proof(N, position, path)` reconstructs `merkle_root` exactly, so the function returns `true` — asserting that the non-existent "transaction" `N` is confirmed in block `B`. [1](#0-0) [2](#0-1) [3](#0-2)

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
