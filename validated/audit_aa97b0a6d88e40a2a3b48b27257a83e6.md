### Title
No On-Chain Staleness Check on `priceX18` Allows Stale Prices to Be Accepted in Slow-Mode Withdrawals — (`core/contracts/EndpointStorage.sol`, `core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`priceX18[productId]` in Nado is stored as a bare `int128` with no associated timestamp. The sequencer updates it via `UpdatePrice` transactions at its own discretion. There is no on-chain staleness threshold enforced anywhere prices are consumed. The `executeSlowModeTransaction()` function is callable by any unprivileged user after a 3-day delay and triggers a health check that uses these potentially stale prices. If the sequencer has not submitted a recent `UpdatePrice` transaction, a user can execute a slow-mode withdrawal that passes the health check using an outdated (favorable) price, withdrawing collateral that should be locked.

---

### Finding Description

**Price storage — no timestamp:**

`EndpointStorage.sol` stores oracle prices as a plain mapping with no timestamp field:

```solidity
mapping(uint32 => int128) internal priceX18;
``` [1](#0-0) 

The `UpdatePrice` struct carries only `productId` and `priceX18` — no timestamp:

```solidity
struct UpdatePrice {
    uint32 productId;
    int128 priceX18;
}
``` [2](#0-1) 

**Price update path — sequencer-only, no freshness guarantee:**

Prices are updated exclusively via `UpdatePrice` transactions submitted by the sequencer through `processTransactionImpl`:

```solidity
} else if (txType == IEndpoint.TransactionType.UpdatePrice) {
    (uint32 productId, int128 newPriceX18) = clearinghouse.updatePrice(transaction);
    if (productId != 0) {
        priceX18[productId] = newPriceX18;
    }
``` [3](#0-2) 

`getPriceX18()` performs only a zero-check — no staleness validation:

```solidity
function getPriceX18(uint32 productId) public override returns (int128 _priceX18) {
    _priceX18 = priceX18[productId];
    require(_priceX18 != 0, ERR_INVALID_PRODUCT);
    emit PriceQuery(productId);
}
``` [4](#0-3) 

**Unprivileged entry path — `executeSlowModeTransaction()`:**

Any address can call `executeSlowModeTransaction()` after the 3-day delay with no access control:

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);
    nSubmissions += 1;
    slowModeConfig = _slowModeConfig;
}
``` [5](#0-4) 

This executes queued `WithdrawCollateral` slow-mode transactions, which call `clearinghouse.withdrawCollateral()`: [6](#0-5) 

**Health check uses stale `risk.priceX18`:**

`withdrawCollateral()` enforces a health check:

```solidity
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [7](#0-6) 

`getHealth()` internally calls `getLiqPriceX18()` and `getSpreadLiqPriceX18()`, both of which read `risk.priceX18` directly from the engine — the same value last written by the sequencer's `UpdatePrice` transaction, with no freshness check:

```solidity
return (risk.priceX18.mul(ONE + penaltyX18), risk.priceX18);
``` [8](#0-7) 

---

### Impact Explanation

If the sequencer has not submitted `UpdatePrice` transactions recently (e.g., due to network congestion, high gas costs, or sequencer downtime), `priceX18` values in the engines become stale. A user who submitted a `WithdrawCollateral` slow-mode transaction when their collateral was priced favorably can execute it after 3 days using the stale (higher) price. The health check passes against the outdated price even though the user's actual collateral value has dropped, allowing them to withdraw funds that should be locked. This leaves the protocol undercollateralized and can cause losses to the insurance fund or other participants.

---

### Likelihood Explanation

The slow-mode path is a documented, user-accessible escape hatch. The sequencer's price update frequency is not enforced on-chain — it is purely an operational decision, exactly as in the Unitas report. Any period of sequencer inactivity or network congestion creates a window where stale prices persist. The 3-day slow-mode delay is long enough for significant price movement to occur. The attack requires no special privileges: any depositor can submit a slow-mode withdrawal and self-execute it after the timeout.

---

### Recommendation

1. Store a `uint128 priceTimestamp` alongside each `priceX18` entry (e.g., in a struct in `EndpointStorage`).
2. Update the timestamp whenever `UpdatePrice` is processed in `processTransactionImpl`.
3. In `withdrawCollateral()` (and any other health-check path reachable from slow-mode or user-callable functions), assert that `block.timestamp - priceTimestamp <= MAX_PRICE_AGE` before accepting the price.
4. Define `MAX_PRICE_AGE` as a protocol constant (e.g., 1 hour) with a per-product override setter, mirroring the fix applied in the Unitas protocol.

---

### Proof of Concept

1. At `t=0`: ETH price is $3000. Sequencer submits `UpdatePrice(ethProductId, 3000e18)`. User's account holds 1 ETH collateral and has a borrow near the initial margin limit.
2. At `t=1`: User submits `WithdrawCollateral` via `submitSlowModeTransaction()` for the maximum allowed amount at $3000.
3. At `t=2`: ETH price drops to $1500. Sequencer goes offline (or gas is too expensive) and does not submit a new `UpdatePrice`.
4. At `t=1 + 3 days`: User (or anyone) calls `executeSlowModeTransaction()`. The slow-mode tx is now executable (`executableAt <= block.timestamp`).
5. `processSlowModeTransactionImpl` → `clearinghouse.withdrawCollateral()` → `getHealth()` → `getLiqPriceX18()` reads `risk.priceX18 = 3000e18` (stale).
6. Health check passes at the stale $3000 price. Withdrawal succeeds.
7. At the true $1500 price, the account is now undercollateralized, causing a loss to the protocol.
