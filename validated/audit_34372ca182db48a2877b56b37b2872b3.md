Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded from the pool, which is the pool's `msg.sender` — the immediate caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, the router is that caller, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`. Any user not on the allowlist can bypass the gate by routing through the public router, completely defeating the access-control invariant the extension is meant to enforce.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes its own `msg.sender` as `sender`:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, so the `sender` argument reaching every extension is whoever called `pool.swap()` directly. [1](#0-0) 

**`ExtensionCalling._beforeSwap` forwards `sender` unchanged:**

The value is passed verbatim into `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))` with no transformation. [2](#0-1) 

**`SwapAllowlistExtension.beforeSwap` checks the wrong actor:**

The check `allowedSwapper[msg.sender][sender]` uses `sender` = whoever called `pool.swap()`, not the original EOA. [3](#0-2) 

**`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly:**

The router calls `pool.swap(...)` with no forwarding of the original `msg.sender` into `extensionData` or any other tamper-evident field, making the router the `msg.sender` seen by the pool. [4](#0-3) 

**Exploit flow:**

```
bannedUser → router.exactInputSingle()
               → pool.swap(recipient, ...)   // msg.sender = router
                    → _beforeSwap(router, ...)
                         → SwapAllowlistExtension.beforeSwap(sender=router)
                              → allowedSwapper[pool][router]  // ← router is allowlisted; check passes
```

The original `bannedUser` is never surfaced. The extension therefore checks whether the **router** is allowlisted, not whether the **user** is allowlisted.

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` — the position owner explicitly passed by the caller — not `sender`. This correctly identifies the economic actor regardless of who calls the pool. [5](#0-4) 

## Impact Explanation

Any unprivileged EOA can trade on a pool that is supposed to be restricted to an allowlist, breaking the core access-control invariant of `SwapAllowlistExtension`. LPs who deposited under the assumption that only vetted counterparties could trade against them are exposed to unauthorized swap counterparties, which can result in direct extraction of value from LP positions. This constitutes a broken core pool functionality / admin-boundary bypass with direct fund impact, meeting Critical/High Sherlock thresholds.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public swap entrypoint for the protocol. Any pool that configures `SwapAllowlistExtension` and needs to support router-based swaps (i.e., any real-world deployment) is immediately vulnerable. The bypass requires no special privileges — any EOA can call the router. The precondition (router allowlisted) is a necessary operational requirement, not an unusual admin mistake.

## Recommendation

The extension must gate on the **original initiator**, not the intermediate caller. Two concrete options:

1. **Router encodes originator in `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it when `sender` is a known router. This requires a coordinated convention.

2. **Add an `originator` field to the hook signature:** Have the pool (or router) surface the true economic actor in a tamper-evident way — e.g., a dedicated `originator` parameter in `beforeSwap` — that the extension can verify without relying on `sender`.

The simplest production fix is option 1: require the router to ABI-encode the original caller into `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and check it when present, falling back to `sender` only when `extensionData` is empty and `sender` is not a known router.

## Proof of Concept

```solidity
// Pool configured with SwapAllowlistExtension.
// Admin allowlists trustedUser and the router (required for trustedUser to use router).
swapExt.setAllowedToSwap(pool, address(router), true);
swapExt.setAllowedToSwap(pool, trustedUser, true);

// bannedUser is NOT on the allowlist.
vm.prank(bannedUser);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    recipient: bannedUser,
    deadline: block.timestamp,
    priceLimitX64: 0,
    extensionData: ""
}));
// Succeeds: extension sees sender=router, which IS allowlisted.
// bannedUser has bypassed the swap allowlist entirely.
```
