### Title
Missing Slippage Protection in `mintNlp` and `burnNlp` Allows Users to Receive Fewer Assets Than Expected — (`core/contracts/Clearinghouse.sol`, `core/contracts/interfaces/IEndpoint.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` EIP-712 signed structs contain no minimum output amount field. The `oraclePriceX18` that determines how many NLP tokens a user receives (or how much quote they get back) is **not part of the user's signed message** — it is appended by the sequencer at execution time. Because the oracle price can legitimately change between when the user signs and when the sequencer executes, users can receive materially fewer assets than they expected with no on-chain protection.

---

### Finding Description

The `MintNlp` struct that users sign contains only `sender`, `quoteAmount`, and `nonce`:

```
"MintNlp(bytes32 sender,uint128 quoteAmount,uint64 nonce)"
``` [1](#0-0) 

The `SignedMintNlp` wrapper that the sequencer submits on-chain adds `oraclePriceX18` and `nlpPoolRebalanceX18` — fields the user never signed:

```solidity
struct SignedMintNlp {
    MintNlp tx;
    bytes signature;
    int128 oraclePriceX18;       // NOT covered by user's signature
    int128[] nlpPoolRebalanceX18;
}
``` [2](#0-1) 

The EIP-712 digest for `MintNlp` is computed exclusively over `sender`, `quoteAmount`, and `nonce` — confirming `oraclePriceX18` is outside the signed scope: [3](#0-2) 

In `Clearinghouse.mintNlp()`, the NLP amount the user receives is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [4](#0-3) 

There is no check that `nlpAmount` meets any minimum the user specified. The user has no on-chain mechanism to express "I want at least X NLP for my Y quote."

The same pattern applies to `burnNlp`. The `BurnNlp` struct the user signs contains only `sender`, `nlpAmount`, and `nonce`: [5](#0-4) 

The quote returned is:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [6](#0-5) 

Again, no minimum quote amount check exists. The user burning NLP has no on-chain guarantee about the minimum quote they will receive.

---

### Impact Explanation

**`mintNlp`:** A user signs a transaction to spend `quoteAmount` USDC expecting to receive approximately `quoteAmount / currentPrice` NLP. If the oracle price rises between signing and sequencer execution (e.g., due to NLP NAV appreciation or oracle update), the user receives fewer NLP tokens than expected, with no floor.

**`burnNlp`:** A user signs a transaction to burn `nlpAmount` NLP expecting to receive approximately `nlpAmount * currentPrice - fee` USDC. If the oracle price falls between signing and execution, the user receives fewer USDC than expected, with no floor.

The corrupted state delta is the user's `QUOTE_PRODUCT_ID` or `NLP_PRODUCT_ID` balance in `SpotEngine` — they are debited the full signed amount but credited a lesser output than anticipated.

---

### Likelihood Explanation

NLP oracle prices update continuously as the pool's NAV changes. Any latency between a user signing a `MintNlp`/`BurnNlp` transaction and the sequencer including it in a batch is sufficient to cause a price discrepancy. This is a normal operating condition, not an attack. The larger the price movement or the longer the sequencer queue, the larger the shortfall. No adversarial action is required — the sequencer can be entirely honest and this still occurs.

---

### Recommendation

Add a `minAmountOut` field to both `MintNlp` and `BurnNlp` structs (and include it in the EIP-712 type hash so it is covered by the user's signature). In `Clearinghouse.mintNlp()` and `Clearinghouse.burnNlp()`, revert if the computed output falls below the user's specified minimum:

```solidity
// MintNlp
require(nlpAmount >= txn.minNlpAmount, "slippage: insufficient NLP out");

// BurnNlp
require(quoteAmount >= txn.minQuoteAmount, "slippage: insufficient quote out");
```

The EIP-712 type strings must be updated accordingly:

```
"MintNlp(bytes32 sender,uint128 quoteAmount,int128 minNlpAmount,uint64 nonce)"
"BurnNlp(bytes32 sender,uint128 nlpAmount,int128 minQuoteAmount,uint64 nonce)"
```

---

### Proof of Concept

1. NLP oracle price is currently `1.00 USDC` per NLP token.
2. Alice signs `MintNlp { sender: alice, quoteAmount: 1000e18, nonce: N }`, expecting to receive `~1000 NLP`.
3. Before the sequencer processes Alice's transaction, the oracle price updates to `1.10 USDC` per NLP (legitimate NAV increase).
4. Sequencer submits `SignedMintNlp` with `oraclePriceX18 = 1.10e18`.
5. `Clearinghouse.mintNlp()` computes `nlpAmount = 1000e18 / 1.10e18 ≈ 909e18`.
6. Alice's `NLP_PRODUCT_ID` balance is credited `909 NLP` instead of the expected `~1000 NLP` — a `~91 NLP` shortfall (~9.1% loss).
7. No revert occurs; no minimum output check exists anywhere in the execution path.

The same scenario applies to `burnNlp` with a falling oracle price: a user burning 1000 NLP at an expected price of `1.00 USDC` receives only `~899 USDC` (after fee) if the price drops to `0.90 USDC` before execution.
