The vulnerability is real. Here is the full trace and finding.

---

### Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing a non-allowlisted payer to bypass the deposit gate â€” (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` ignores the `sender` parameter (the actual payer/caller) and only checks `owner` (the LP share recipient). Because `pool.addLiquidity` accepts a caller-supplied `owner` that can differ from `msg.sender`, any non-allowlisted address can bypass the deposit gate by specifying an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address and passes both `msg.sender` (as `sender`) and `owner` to the extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol:191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The extension dispatcher encodes both values and calls the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol:95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but **discards it** (unnamed `address,`). It only checks `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol:32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

Here `msg.sender` inside the extension is the **pool** (the caller of the extension), so the check resolves to `allowedDepositor[pool][owner]`. If `owner` is allowlisted, the check passes regardless of who the actual payer (`sender`) is.

The `MetricOmmPoolLiquidityAdder` explicitly supports and tests the payer â‰  owner pattern:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol:56-68
function addLiquidityExactShares(address pool, address owner, ...) external payable {
    _validateOwner(owner);  // only checks owner != address(0)
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
``` [4](#0-3) 

The test `test_exactShares_usesMsgSenderAsPayerNotOwner` confirms alice pays while bob receives shares â€” with no allowlist in that test setup, but the same call path applies to allowlisted pools. [5](#0-4) 

---

### Impact Explanation

A non-allowlisted address B can:
1. Call `pool.addLiquidity(owner=A, ...)` directly, where A is allowlisted.
2. The extension checks `allowedDepositor[pool][A]` â†’ `true` â†’ passes.
3. B funds the deposit via the modify-liquidity callback; A receives LP shares.

This breaks the core invariant of the deposit allowlist: the pool admin intended to restrict **who funds deposits**, but the check is on the LP share recipient, not the payer. B can inject liquidity into curated bins, manipulate pool composition, and affect pricing â€” all without being allowlisted.

---

### Likelihood Explanation

The attack requires no special privileges. Any EOA or contract can call `pool.addLiquidity` directly with an arbitrary `owner`. The allowlisted address A does not need to cooperate (B simply names A as owner and pays). The path is fully public and requires no admin access, oracle manipulation, or non-standard token behavior.

---

### Recommendation

Gate on `sender` (the actual payer), not `owner`. Change the check in `beforeAddLiquidity`:

```solidity
// Current (wrong): checks owner
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) { ... }

// Fixed: checks sender (the actual payer/caller)
function beforeAddLiquidity(address sender, address owner, ...)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

If the intent is to gate both the payer and the recipient, check both `sender` and `owner`. The fix must be consistent with the pool admin's curation intent.

### Proof of Concept

```solidity
// Foundry integration test
function test_nonAllowlistedPayerBypassesDepositGate() public {
    address allowlistedA = makeAddr("allowlistedA");
    address nonAllowlistedB = makeAddr("nonAllowlistedB");

    // Only A is allowlisted
    depositExtension.setAllowedToDeposit(address(pool), allowlistedA, true);

    // Fund B with tokens and approve pool
    deal(address(token0), nonAllowlistedB, 1e18);
    deal(address(token1), nonAllowlistedB, 1e18);
    vm.prank(nonAllowlistedB);
    token0.approve(address(pool), type(uint256).max);
    vm.prank(nonAllowlistedB);
    token1.approve(address(pool), type(uint256).max);

    // B calls addLiquidity with owner=A (allowlisted)
    // Extension checks allowedDepositor[pool][A] â†’ true â†’ passes
    // B pays tokens, A receives LP shares
    vm.prank(nonAllowlistedB);
    pool.addLiquidity(
        allowlistedA,   // owner = allowlisted A
        0,              // salt
        deltas,
        callbackData,
        ""
    );

    // Assert: A has LP shares, B paid tokens
    assertGt(positionShares(allowlistedA, 0, bin), 0);
    assertLt(token0.balanceOf(nonAllowlistedB), 1e18); // B paid
}
```

The deposit succeeds because `DepositAllowlistExtension.beforeAddLiquidity` checks `allowedDepositor[pool][allowlistedA]` (true) while completely ignoring `nonAllowlistedB` (the actual payer passed as `sender`). [3](#0-2)
