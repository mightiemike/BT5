### Title
P2P Block Sync Computes Wrong Transaction Hash for V3 Transactions with Zero L2/L1DataGas Bounds — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter in the P2P block-sync path silently downgrades a V3 transaction's resource bounds to `ValidResourceBounds::L1Gas` whenever `l2_gas` and `l1_data_gas` are both zero. The gateway path, by contrast, always wraps the same values in `ValidResourceBounds::AllResources`. Because `get_tip_resource_bounds_hash` hashes a different number of fields depending on the variant, the two paths produce divergent transaction hashes for identical wire bytes, breaking the canonicalization invariant that every component must agree on a transaction's hash.

---

### Finding Description

**Step 1 — Gateway path always uses `AllResources`.**

`InternalRpcInvokeTransactionV3` stores resource bounds as `AllResourceBounds` (a plain struct, not the enum). Its `InvokeTransactionV3Trait::resource_bounds()` implementation unconditionally wraps the value:

```rust
// crates/starknet_api/src/rpc_transaction.rs:637-639
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
```

The hash is therefore always computed over three resource entries (L1\_GAS, L2\_GAS, L1\_DATA\_GAS), even when the last two are zero.

**Step 2 — P2P block-sync path can produce `L1Gas`.**

When a historical block is received over P2P, transactions are reconstructed from protobuf via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs:431-435
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

Any V3 transaction whose `l2_gas` and `l1_data_gas` fields are zero (a common pattern for L1-only-gas transactions) is silently converted to the `L1Gas` variant.

**Step 3 — `get_tip_resource_bounds_hash` produces different digests for the two variants.**

```rust
// crates/starknet_api/src/transaction_hash.rs:203-208
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2-element hash
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }                                                                  // 3-element hash
});
```

- **Gateway path** (`AllResources`): `poseidon(tip, L1_GAS_felt, L2_GAS_felt, L1_DATA_GAS_felt)` → **H1**
- **P2P sync path** (`L1Gas`): `poseidon(tip, L1_GAS_felt, L2_GAS_felt)` → **H2**

`H1 ≠ H2` even when `l2_gas = l1_data_gas = 0`, because Poseidon is not length-extension-safe in the same way — a 3-element input and a 2-element input produce distinct outputs.

**Step 4 — Divergence propagates to `validate_transaction_hash`.**

```rust
// crates/starknet_api/src/transaction_hash.rs:177-184
pub fn validate_transaction_hash(...) -> Result<bool, StarknetApiError> {
    let mut possible_hashes = get_deprecated_transaction_hashes(...)?;
    possible_hashes.push(get_transaction_hash(transaction, chain_id, transaction_options)?);
    Ok(possible_hashes.contains(&expected_hash))
}
```

The `expected_hash` carried in the block header is H1 (computed by the sequencer/gateway). The hash recomputed from the P2P-received `Transaction` object is H2. `possible_hashes.contains(&H1)` returns `false`, so the block is rejected.

---

### Impact Explanation

A non-leader node that receives a block containing any V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` will recompute a hash that does not match the block header. Depending on how the sync layer handles the mismatch:

- **If hash validation is enforced**: the node permanently rejects the block, halting its chain tip — a liveness failure.
- **If hash validation is skipped**: the node stores the transaction under hash H2 while the rest of the network uses H1, producing a divergent state — an integrity failure.

This matches the allowed impact: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."**

---

### Likelihood Explanation

V3 transactions that set only `l1_gas` and leave `l2_gas = l1_data_gas = 0` are the natural migration path from pre-0.13.3 transactions. Any such transaction included in a block triggers the mismatch on every syncing peer. The condition is unprivileged and requires no special attacker capability — a normal user submitting a standard L1-gas-only V3 transaction is sufficient.

---

### Recommendation

Remove the `L1Gas` downgrade from the P2P block-sync converter. When all three protobuf fields are present (even if zero), always produce `AllResources`:

```diff
// crates/apollo_protobuf/src/converters/transaction.rs
-Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
-    ValidResourceBounds::L1Gas(l1_gas)
-} else {
-    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
-})
+Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```
