### Title
Protocol Upgrade v83→v84 Breaks Contracts With Function Bodies > 196608 Bytes — (`runtime/near-vm-runner/src/prepare/prepare_v3.rs`)

### Summary

The introduction of `max_function_body_size = 196608` at protocol version 84, combined with the simultaneous switch from `prepare_v2` to `prepare_v3` (gated on `reftypes_bulk_memory`), creates a concrete backward-compatibility break: a contract deployed at v83 with a function body between 196609 and 4194304 bytes passes preparation under v83 rules but permanently fails with `PrepareError::FunctionBodyTooLarge` under v84 rules. No special privileges are required to deploy such a contract.

### Finding Description

**Prepare-path selection** is controlled by a single flag in `runtime/near-vm-runner/src/prepare.rs`:

```rust
if config.reftypes_bulk_memory || config.vm_kind == VMKind::Wasmtime {
    prepare_v3::prepare_contract(...)
} else {
    prepare_v2::prepare_contract(...)
}
``` [1](#0-0) 

At protocol version 84, `84.yaml` simultaneously sets:
- `reftypes_bulk_memory: { old: false, new: true }` — flips the gate to `prepare_v3`
- `max_function_body_size: { new: 196_608 }` — introduces the limit for the first time [2](#0-1) 

**prepare_v2** (`PrepareContext` struct) has no `function_body_size_limit` field and its `CodeSectionEntry` handler performs no body-size check: [3](#0-2) [4](#0-3) 

**prepare_v3** adds `function_body_size_limit: u64` and enforces it at every `CodeSectionEntry`: [5](#0-4) [6](#0-5) 

The limit is initialized from `limits.max_function_body_size.unwrap_or(u64::MAX)`, so before v84 (when the field is absent from the config) it is effectively unlimited, and at v84 it becomes 196608. [7](#0-6) 

**Concrete divergence:**

| Protocol | Prepare path | `max_function_body_size` | Result for body = 196609 bytes |
|---|---|---|---|
| v83 | prepare_v2 | absent (no check) | `Ok(prepared_bytes)` |
| v84 | prepare_v3 | 196608 | `Err(PrepareError::FunctionBodyTooLarge)` |

The raw Wasm bytes stored in the trie are identical; only the config applied at call time differs.

### Impact Explanation

Any contract deployed at v83 (or any earlier version using prepare_v2, i.e., `reftypes_bulk_memory = false`) with at least one function body whose encoded byte range exceeds 196608 bytes becomes permanently uncallable after the v84 upgrade. The `max_contract_size` is 4194304 bytes, so a 196609-byte function body is well within the deployment limit. The error surfaces as a `CompilationError::PrepareError(FunctionBodyTooLarge)` on every subsequent function call, with no recovery path short of redeploying a modified contract.

The cache-warming path (`precompile_contract_with_warming`) silently drops the `FunctionBodyTooLarge` error during the pre-upgrade warming pass, so the failure is not surfaced until the first call under v84. [8](#0-7) [9](#0-8) 

### Likelihood Explanation

Any user can deploy a contract with a large function body (e.g., generated/compiled Rust or AssemblyScript code with inlined data or large match arms). A 192KB function body is unusual but not impossible for machine-generated Wasm. The window is the entire period between the introduction of prepare_v2 (v62) and the v84 upgrade.

### Recommendation

Before activating `max_function_body_size` at v84, scan all deployed contracts in the state for function bodies exceeding the new limit and either:
1. Grandfather existing contracts (apply the limit only to contracts deployed at v84+), or
2. Raise the limit to a value that cannot be exceeded by any contract already in state, or
3. Document the breaking change and provide a migration path.

Additionally, the cache-warming path should log (not silently drop) `PrepareError::FunctionBodyTooLarge` so operators are alerted before the upgrade epoch boundary.

### Proof of Concept

```
1. At v83 (reftypes_bulk_memory=false, no max_function_body_size):
   - Build a Wasm module with one function whose encoded body range is 196609 bytes
     (e.g., 196609 nop instructions).
   - Call prepare_v2::prepare_contract(wasm, features, config_v83, kind) → Ok(_).

2. At v84 (reftypes_bulk_memory=true, max_function_body_size=196608):
   - Call prepare_v3::prepare_contract(same_wasm, features, config_v84, kind)
     → Err(PrepareError::FunctionBodyTooLarge).

3. On a live chain: deploy the contract in an epoch running v83, then wait for the
   v84 upgrade. Any subsequent FunctionCall to that contract returns
   CompilationError::PrepareError(FunctionBodyTooLarge).
``` [10](#0-9)

pub(crate) fn precompile_contract_with_warming(
    code: &ContractCode,
    current_config: Arc<Config>,
    next_config: Option<Arc<Config>>,
    cache: Option<&dyn ContractRuntimeCache>,
) {
    if let (Some(next_config), Some(cache)) = (next_config, cache) {
        if cache_keys_differ(Arc::clone(&current_config), Arc::clone(&next_config)) {
            spawn_cache_warming(code.clone(), next_config, cache.handle());
        }
    }
    let _ = precompile_contract(code, Arc::clone(&current_config), cache);
}
