### Title
Deprecated `verify_transaction_inclusion` Remains a Live NEAR Entrypoint and Accepts Forged Internal-Node Merkle Proofs — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is marked `#[deprecated]` in Rust but remains a fully callable public NEAR contract method. It accepts fully attacker-controlled `tx_id`, `tx_index`, and `merkle_proof` parameters and passes them directly to `compute_root_from_merkle_proof` with no check that `tx_id` is a leaf-level transaction hash. An unprivileged caller can supply an internal Merkle tree node hash as `tx_id` with a correctly constructed shorter proof path, causing the function to compute the correct Merkle root and return `true` for a transaction that does not exist in the block.

---

### Finding Description

**Root cause — attacker-controlled parameters flow into an unconstrained verifier:**

`verify_transaction_inclusion` accepts `ProofArgs` whose fields `tx_id`, `tx_index`, and `merkle_proof` are entirely caller-supplied. [1](#0-0) 

These are forwarded without any leaf-level constraint to `compute_root_from_merkle_proof`: [2](#0-1) 

`compute_root_from_merkle_proof` is a pure positional hash-chain: it takes whatever 32-byte value is given as `transaction_hash`, iterates over the proof hashes, and combines them left/right based solely on `current_position % 2`. It has no concept of "this must be a leaf": [3](#0-2) 

**Why `#[deprecated]` does not close the entrypoint:**

Rust's `#[deprecated]` attribute emits a compiler warning at call sites in Rust source; it does not remove the symbol from the compiled WASM or prevent NEAR from dispatching calls to it. The function is `pub` and decorated with `#[pause]` (not `#[private]`), so it is exported as a callable contract method for any NEAR account: [4](#0-3) 

**The forgery:**

For a real Bitcoin block with ≥ 2 transactions whose Merkle tree is:

```
Level 0 (leaves): T0, T1, T2, T3
Level 1 (internal): N01 = H(T0,T1),  N23 = H(T2,T3)
Level 2 (root):    R   = H(N01, N23)
```

An attacker sets:
- `tx_id`       = `N01`  (an internal node, not a real transaction)
- `tx_index`    = `0`    (even → hash left)
- `merkle_proof`= `[N23]`

`compute_root_from_merkle_proof(N01, 0, [N23])` computes `H(N01, N23) = R`, which equals the block's stored `merkle_root`. The function returns `true`.

The only guard against an empty proof is: [5](#0-4) 

There is no guard that `tx_id` must be a leaf, that `tx_index` must be consistent with the proof length, or that the proof length must equal `log2(tx_count)`.

**Why `verify_transaction_inclusion_v2` does not help here:**

`v2` adds a coinbase proof check and then delegates to `v1` via `args.into()`. But `v1` is still independently callable — a caller bypasses `v2` entirely by calling `verify_transaction_inclusion` directly: [6](#0-5) 

---

### Impact Explanation

Any NEAR contract that uses `verify_transaction_inclusion` as a gate — e.g., a Bitcoin bridge that releases wrapped tokens or NEAR funds upon proof of a Bitcoin deposit — can be tricked into accepting a forged proof. The attacker proves inclusion of a non-existent transaction, the bridge releases assets, and no corresponding Bitcoin deposit exists. The broken invariant is: *the function must return `true` only when `tx_id` is a real leaf-level transaction hash present in the block*. That invariant is violated by the internal-node substitution.

---

### Likelihood Explanation

High. The attack requires no privileged role, no leaked key, and no social engineering. All inputs needed — the block's Merkle tree structure, internal node hashes, and sibling hashes — are publicly available from any Bitcoin full node or block explorer. The attacker calls `verify_transaction_inclusion` directly as any NEAR account. The only prerequisite is that the target block is already tracked by the light client (i.e., it is in the main chain with sufficient confirmations), which is the normal operating state.

---

### Recommendation

1. **Remove `verify_transaction_inclusion` from the public NEAR API.** Change it from `pub fn` to `fn` (private), or add `#[private]` so only the contract itself can call it. Rust's `#[deprecated]` is insufficient — it must not be an externally dispatchable method.
2. **Enforce proof-length consistency.** In `verify_transaction_inclusion_v2`, require that `merkle_proof.len()` equals `coinbase_merkle_proof.len()` (already done) *and* that both lengths are consistent with the claimed `tx_index` (i.e., `tx_index < 2^proof_len`).
3. **Document the security boundary.** Any downstream contract consuming the boolean result of either function should be warned that the result is only as trustworthy as the proof inputs supplied by the caller.

---

### Proof of Concept

Given a real Bitcoin block already tracked by the light client (e.g., height 685452 in the test suite) with known transactions `T0, T1`:

1. Compute `N01 = double_sha256(T0 || T1)` (the level-1 internal node).
2. Compute `N23 = double_sha256(T2 || T3)` (its sibling).
3. Call `verify_transaction_inclusion` with:
   ```
   tx_id            = N01
   tx_block_blockhash = <real block hash in mainchain>
   tx_index         = 0
   merkle_proof     = [N23]
   confirmations    = 1
   ```
4. `compute_root_from_merkle_proof(N01, 0, [N23])` = `double_sha256(N01 || N23)` = Merkle root stored in the block header.
5. Function returns `true` — a non-existent transaction is "verified."

A bridge contract acting on this result would release funds with no corresponding on-chain Bitcoin deposit.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
