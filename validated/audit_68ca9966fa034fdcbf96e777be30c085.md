### Title
Stale Chain State Accepted Without Freshness Check in SPV Proof Verification - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` return SPV proof results based on the contract's stored mainchain state without checking whether that state is recent. If the relayer has fallen behind or stopped, the stored chain tip may be hours or days old. A transaction that was confirmed in the stored chain but subsequently reorganized out of the real Bitcoin chain will still verify as `true`, because the contract never compares the mainchain tip's timestamp against the current NEAR block time before returning a result.

### Finding Description

The `check_pow` path in every chain module (`bitcoin.rs`, `litecoin.rs`, `dogecoin.rs`, `zcash.rs`) already uses `env::block_timestamp_ms()` to enforce that submitted block headers are not too far in the future:

```rust
// contract/src/bitcoin.rs lines 35-39
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
    "time-too-new: block timestamp too far in the future"
);
``` [1](#0-0) 

However, the SPV verification entry points perform no equivalent freshness check on the stored chain tip before returning a result:

```rust
// contract/src/lib.rs lines 294-308
let heaviest_block_header = self
    .headers_pool
    .get(&self.mainchain_tip_blockhash)
    ...;
// No check: is heaviest_block_header.block_header.time recent?
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

The same omission exists in `verify_transaction_inclusion_v2`, which delegates to the deprecated `verify_transaction_inclusion` after the coinbase proof check: [3](#0-2) 

The `MAX_FUTURE_BLOCK_TIME_LOCAL` constant (7200 seconds / 2 hours) is already defined and used during block submission, but is never consulted during proof verification: [4](#0-3) 

### Impact Explanation

Any downstream NEAR contract or off-chain consumer that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` receives a boolean result it treats as authoritative. If the light client's stored chain is stale (relayer offline, censored, or lagging), the contract's mainchain tip may not reflect the real Bitcoin chain. A transaction confirmed in the stored chain but reorganized out of the real chain will return `true`. Downstream logic — e.g., a bridge contract releasing funds, a settlement contract marking a payment as received — will act on a false positive.

### Likelihood Explanation

Shallow Bitcoin reorgs (1–6 blocks) occur multiple times per year. A relayer outage of even a few hours is operationally plausible. The window between the relayer falling behind and the contract being queried for a proof is entirely attacker-exploitable: the attacker submits a transaction, waits for it to appear in the stored chain, then queries the contract during a period when the relayer is behind and a reorg has occurred. No privileged access is required to call `verify_transaction_inclusion`.

### Recommendation

Before returning a verification result, compare the stored mainchain tip's block timestamp against the current NEAR block time. Reject the call if the tip is older than an acceptable staleness threshold (e.g., `MAX_FUTURE_BLOCK_TIME_LOCAL` = 7200 seconds, or a chain-specific value):

```rust
let tip_time = heaviest_block_header.block_header.time;
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    current_timestamp <= tip_time + ACCEPTABLE_CHAIN_LAG,
    "ERR_STALE_CHAIN: light client chain tip is too old"
);
```

This mirrors the existing freshness guard already applied during `submit_blocks` / `check_pow` and closes the symmetric gap on the read path.

### Proof of Concept

1. Light client is initialized and synced to Bitcoin height H with tip timestamp T.
2. Relayer goes offline (network partition, crash, or deliberate griefing).
3. Bitcoin produces blocks H+1 … H+N; a shallow reorg removes a transaction `tx` that was in block H.
4. Attacker calls `verify_transaction_inclusion` with `tx_block_blockhash` = hash of block H, a valid Merkle proof for `tx`, and `confirmations = 1`.
5. The contract reads `heaviest_block_header` (still at height H, timestamp T), computes `H - H + 1 >= 1` → passes, verifies the Merkle proof → returns `true`.
6. The real Bitcoin chain no longer contains `tx`, but the NEAR contract has certified its inclusion.

The missing check — `current_timestamp > tip_time + ACCEPTABLE_CHAIN_LAG` — would have caused the call to revert, preventing the false positive. [5](#0-4) [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
