### Title
Unprivileged Caller Can Drain Canonical Chain Headers via `run_mainchain_gc` — (File: `contract/src/lib.rs`)

### Summary

`BtcLightClient::run_mainchain_gc` is a public, state-mutating function that carries no caller access control when the contract is not paused. Any unprivileged NEAR account can call it with an attacker-controlled `batch_size`, immediately pruning all mainchain headers that exceed `gc_threshold` in a single transaction. This permanently corrupts the canonical chain index, causing `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to panic for any pruned block and preventing legitimate chain reorganizations.

### Finding Description

`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

The `#[pause]` macro blocks the function only when the contract is in the paused state. When the contract is operating normally (unpaused), the macro imposes no caller restriction whatsoever. There is no `#[trusted_relayer]`, no `acl_require`, and no `env::predecessor_account_id()` check. Any NEAR account — with zero stake, zero role, and no deposit — can call this method.

The `batch_size` parameter is fully attacker-controlled. Inside the function:

```rust
let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
```

Passing `batch_size = u64::MAX` causes `selected_amount_to_remove` to equal `total_amount_to_remove`, the maximum possible removal in one call. The loop then deletes every header from `headers_pool` and `mainchain_height_to_header` for heights `start_removal_height..end_removal_height`, and rewrites `mainchain_initial_blockhash` to the new oldest block.

By contrast, the only other caller of `run_mainchain_gc` is `submit_blocks`, which is gated behind `#[trusted_relayer]` (requiring `Role::UnrestrictedSubmitBlocks` or `Role::DAO`). The existence of `Role::UnrestrictedRunGC` — defined specifically to allow GC calls on a paused contract — confirms the designers intended access control on this path, but they applied it only to the paused branch, leaving the normal-operation branch open.

### Impact Explanation

**Canonical chain state corruption.** After an attacker drains headers:

1. `verify_transaction_inclusion` panics with `"block does not belong to the current main chain"` for any transaction whose containing block was pruned. SPV proofs that were valid before the attack become permanently unverifiable.

2. `verify_transaction_inclusion_v2` calls `verify_transaction_inclusion` internally after the coinbase check; it fails identically.

3. Chain reorganization is broken. As documented in `contract/CLAUDE.md`: *"If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound`."* An attacker who triggers aggressive GC before a fork is resolved can permanently prevent the correct chain from being promoted.

4. `mainchain_initial_blockhash` is rewritten to a new value, permanently altering the canonical chain's lower bound. This state change cannot be reversed without a privileged migration.

### Likelihood Explanation

The attack requires no deposit, no role, no staking, and no prior knowledge beyond the contract's public ABI. The call is a single NEAR transaction. The contract is designed for production use tracking live Bitcoin/altcoin chains, where the mainchain routinely exceeds `gc_threshold` (recommended value: 52,704 blocks). Any time the chain is above threshold, the attack is immediately executable.

### Recommendation

Add an explicit caller restriction to `run_mainchain_gc` matching the trust level of `submit_blocks`. The simplest fix is to require `Role::UnrestrictedRunGC` (or `Role::DAO`) for direct external calls, while keeping the internal call from `submit_blocks` unrestricted (since `submit_blocks` is already gated). Alternatively, make `run_mainchain_gc` `pub(crate)` and expose a separate access-controlled public wrapper.

### Proof of Concept

1. Deploy the contract with `gc_threshold = 52704` and submit enough blocks so `mainchain_size > gc_threshold`.
2. From any unprivileged NEAR account (no roles granted), call:
   ```json
   { "method": "run_mainchain_gc", "args": { "batch_size": 18446744073709551615 } }
   ```
3. All headers beyond `gc_threshold` are deleted in one transaction.
4. Call `verify_transaction_inclusion` for any transaction in a pruned block — the contract panics with `"block does not belong to the current main chain"`.
5. Attempt to submit a fork block whose ancestor was pruned — the contract panics with `"PrevBlockNotFound"`.

Root cause: [1](#0-0)  — `#[pause]` only restricts the paused state; no role check guards the unpaused path.

Attacker-controlled parameter: [2](#0-1)  — `batch_size` is uncapped relative to caller identity.

State mutations performed: [3](#0-2)  — headers removed from pool and index, `mainchain_initial_blockhash` rewritten.

Contrast with the gated path: [4](#0-3)  — `submit_blocks` (the only intended caller of GC) requires `#[trusted_relayer]`.
