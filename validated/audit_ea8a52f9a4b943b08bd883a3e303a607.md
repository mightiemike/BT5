Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the originating caller, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist, where `sender` is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating EOA. If a pool admin allowlists the router to enable router-mediated swaps for curated users, every unprivileged address can bypass the allowlist by calling the same router, because the extension only sees the router address and approves it unconditionally.

## Finding Description

**Extension check (root cause):**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the extension's caller) and `sender` is whoever called `pool.swap()`:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

**What the pool passes as `sender`:**

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`, which `ExtensionCalling` faithfully forwards to every registered extension:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← becomes `sender` in the extension
    ...
```

**What the router passes to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly with no originating-user context. The pool therefore sees `msg.sender = router`:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The same pattern applies to `exactInput` (L104), `exactOutputSingle` (L136), and `exactOutput` (L165).

**Bypass path:**

A pool admin who wants allowlisted users to trade through the router must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, `allowedSwapper[pool][router] == true` for every swap arriving through it — regardless of who the originating caller is. Any non-allowlisted address can call `exactInputSingle` (or any other router entry point) and the extension passes because it only checks whether the router itself is allowed.

The existing test suite (`FullMetricExtension.t.sol`, `test_allowedSwapSucceeds`) only exercises direct pool calls via `TestCaller` and never a router-mediated path, so this bypass is untested.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties loses that restriction entirely for the router path the moment the admin allowlists the router. Any unprivileged address can execute swaps against the pool at oracle-anchored prices that LPs accepted only under the assumption that only vetted counterparties would trade. This is a direct allowlist bypass with potential for LP principal loss on pools designed for restricted access (e.g., institutional or KYC-gated pools). This matches the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" allowed impact.

## Likelihood Explanation

The trigger requires two steps, both entirely plausible in production:
1. Pool admin allowlists the router — a natural and expected action when the pool is meant to be accessible via the standard periphery for allowlisted users.
2. Any non-allowlisted user calls the router — zero additional privilege required.

The admin has no mechanism to allowlist specific users for router-mediated swaps only without simultaneously opening the pool to all router callers. The design gap is structural, not a misconfiguration edge case, and is repeatable by any address at any time after step 1.

## Recommendation

The router must propagate the originating user's identity to the extension layer. Two approaches:

1. **Extension-data convention**: The router encodes `msg.sender` into `extensionData` before forwarding to the pool, and `SwapAllowlistExtension.beforeSwap` decodes and checks that address instead of (or in addition to) `sender`.
2. **Dedicated sender field**: Add an `originSender` field to the swap call that the pool passes through to hooks, populated by the router from its own `msg.sender`.

Until one of these is implemented, the NatSpec for `SwapAllowlistExtension` and `MetricOmmSimpleRouter` must explicitly warn that allowlisting the router grants unrestricted swap access to all router callers.

## Proof of Concept

```
Setup:
  pool   = pool with SwapAllowlistExtension (extension2, beforeSwap order = 2)
  alice  = allowlisted user
  bob    = non-allowlisted user
  router = MetricOmmSimpleRouter

Step 1 (admin):
  swapExtension.setAllowedToSwap(pool, alice,  true)   // intended allowlist
  swapExtension.setAllowedToSwap(pool, router, true)   // to let alice use the router

Step 2 (bob, unprivileged):
  router.exactInputSingle({
      pool:          pool,
      zeroForOne:    true,
      amountIn:      X,
      recipient:     bob,
      extensionData: ""
  })

Execution trace:
  router.exactInputSingle()
    → pool.swap(recipient=bob, ...) [msg.sender = router]
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (no revert)
      → swap executes, bob receives tokens

Result: bob swaps successfully despite never being allowlisted.
```

A Foundry integration test can reproduce this by deploying the pool with `SwapAllowlistExtension`, deploying `MetricOmmSimpleRouter`, allowlisting the router via `setAllowedToSwap`, then calling `exactInputSingle` from an address not in the allowlist and asserting no revert occurs.
