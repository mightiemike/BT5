### Title
`ValidResourceBounds` Protobuf Deserialization Silently Mutates `AllResources` to `L1Gas`, Producing a Wrong Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation silently converts `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` into `L1Gas(X)` after protobuf round-trip. Because `get_tip_resource_bounds_hash` hashes a different number of elements for each variant, the transaction hash computed from the deserialized object diverges from the hash computed at submission time. Any downstream component that recomputes the hash from the deserialized `DeclareTransactionV3` or `DeployAccountTransactionV3` (e.g., `validate_transaction_hash` in the P2P sync path) will obtain a wrong value and reject a valid transaction.

### Finding Description

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` applies the following heuristic:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← assumed equivalent
} else {
    ValidResourceBounds::AllResources(...)
})
``` [1](#0-0) 

The assumption is wrong. `AllResources` and `L1Gas` are structurally distinct variants of `ValidResourceBounds`, and `get_tip_resource_bounds_hash` branches on that distinction to decide how many elements to include in the Poseidon hash:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 3 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 4 elements total
    }
});
``` [2](#0-1) 

A user can legitimately submit a `DeclareTransactionV3` or `DeployAccountTransactionV3` with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` through the gateway. The gateway's `RpcDeclareTransactionV3` and `InternalRpcDeclareTransactionV3` both carry `AllResourceBounds` (always `AllResources`), so the hash is computed over 4 elements at submission time. [3](#0-2) 

When this transaction is later serialized to protobuf for P2P sync via `DeclareTransactionV3Common` and deserialized on a peer, the `resource_bounds` field is reconstructed through `ValidResourceBounds::try_from(protobuf::ResourceBounds)`, which converts it to `L1Gas`. Any subsequent call to `calculate_transaction_hash` or `validate_transaction_hash` on the deserialized `DeclareTransactionV3` now hashes only 3 elements, producing a hash that does not match the one stored in the block. [4](#0-3) 

The same path applies to `DeployAccountTransactionV3`, which also uses `ValidResourceBounds` and goes through the same `DeclareTransactionV3Common` / `DeployAccountV3` protobuf converters. [5](#0-4) 

### Impact Explanation

**High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

A syncing node that receives a `DeclareTransactionV3` or `DeployAccountTransactionV3` with `AllResources { l2_gas: 0, l1_data_gas: 0 }` via P2P will deserialize it with `L1Gas` bounds. When `validate_transaction_hash` recomputes the hash from the deserialized `Transaction` object, it produces a value that differs from the hash stored in the block header / receipt. The node will therefore reject a valid, already-finalized transaction, breaking state sync for any block containing such a transaction. [6](#0-5) 

### Likelihood Explanation

**Medium.** The gateway's stateless validator explicitly permits `AllResourceBounds` with only `l1_gas` set (zero `l2_gas` and zero `l1_data_gas`), as confirmed by the `valid_l1_gas` test case. Any user who submits such a transaction triggers the divergence. The condition is not gated by any privilege; it requires only a standard, well-formed transaction submission. [7](#0-6) 

### Recommendation

Remove the heuristic that conflates `AllResources { l2_gas: 0, l1_data_gas: 0 }` with `L1Gas`. The protobuf wire format already carries an explicit `l1_data_gas` field; use its presence (or a dedicated version/type tag) to determine the correct variant, not the zero-value of the fields. Concretely, change the deserialization to always produce `AllResources` when all three fields are present in the protobuf message, reserving `L1Gas` only for legacy messages that genuinely omit `l1_data_gas`:

```rust
// Only fall back to L1Gas when l1_data_gas is absent from the wire message
// (pre-0.13.3 compatibility), not when it is present but zero.
let l1_data_gas_opt = value.l1_data_gas;
Ok(match l1_data_gas_opt {
    None => ValidResourceBounds::L1Gas(l1_gas),
    Some(raw) => {
        let l1_data_gas: ResourceBounds = raw.try_into()?;
        ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
    }
})
``` [8](#0-7) 

### Proof of Concept

1. Submit a `DeclareTransactionV3` with `AllResourceBounds { l1_gas: GasAmount(1)/GasPrice(1), l2_gas: zero, l1_data_gas: zero }` to the gateway. The gateway computes `tx_hash_A` using `get_tip_resource_bounds_hash` over 4 elements `[tip, l1_gas_packed, l2_gas_packed(0), l1_data_gas_packed(0)]`.

2. The transaction is stored as `DeclareTransactionV3 { resource_bounds: AllResources(...) }` and included in a block.

3. A peer node receives the block via P2P. The `DeclareTransactionV3` is serialized to `protobuf::transaction_in_block::DeclareV3WithoutClass`, then deserialized. Inside `DeclareTransactionV3Common::try_from`, `ValidResourceBounds::try_from(protobuf::ResourceBounds)` fires the `l1_data_gas.is_zero() && l2_gas.is_zero()` branch and returns `L1Gas(l1_gas)`.

4. The peer calls `validate_transaction_hash` on the deserialized `DeclareTransactionV3`. `get_tip_resource_bounds_hash` now hashes only 3 elements `[tip, l1_gas_packed, l2_gas_packed(0)]`, producing `tx_hash_B ≠ tx_hash_A`.

5. Validation fails; the peer rejects the block or marks the transaction as invalid, diverging from the canonical chain state. [1](#0-0) [9](#0-8) [10](#0-9)

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::DeclareV3Common> for DeclareTransactionV3Common {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::DeclareV3Common) -> Result<Self, Self::Error> {
        let resource_bounds = ValidResourceBounds::try_from(
            value.resource_bounds.ok_or(missing("DeclareV3Common::resource_bounds"))?,
        )?;

        let tip = Tip(value.tip);

        let signature = TransactionSignature(
            value
                .signature
                .ok_or(missing("DeclareV3Common::signature"))?
                .parts
                .into_iter()
                .map(Felt::try_from)
                .collect::<Result<Vec<_>, _>>()?
                .into(),
        );

        let nonce = Nonce(value.nonce.ok_or(missing("DeclareV3Common::nonce"))?.try_into()?);

        let compiled_class_hash = CompiledClassHash(
            value
                .compiled_class_hash
                .ok_or(missing("DeclareV3Common::compiled_class_hash"))?
                .try_into()?,
        );

        let sender_address = value.sender.ok_or(missing("DeclareV3Common::sender"))?.try_into()?;

        let nonce_data_availability_mode =
            enum_int_to_volition_domain(value.nonce_data_availability_mode)?;

        let fee_data_availability_mode =
            enum_int_to_volition_domain(value.fee_data_availability_mode)?;

        let paymaster_data = PaymasterData(
            value.paymaster_data.into_iter().map(Felt::try_from).collect::<Result<Vec<_>, _>>()?,
        );

        let account_deployment_data = AccountDeploymentData(
            value
                .account_deployment_data
                .into_iter()
                .map(Felt::try_from)
                .collect::<Result<Vec<_>, _>>()?,
        );

        Ok(Self {
            resource_bounds,
            tip,
            signature,
            nonce,
            compiled_class_hash,
            sender_address,
            nonce_data_availability_mode,
            fee_data_availability_mode,
            paymaster_data,
            account_deployment_data,
        })
    }
