### Title
Secp256k1/Secp256r1 Operator Flag Gate Bypassed via 4-Byte Opcode Alias - (File: src/chia_dialect.rs)

### Summary
`ChiaDialect::op` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` unconditionally when the operator atom is 4 bytes long (`0x13d61f00` / `0x1c3a8f00`), while the 1-byte opcode aliases (64 / 65) for the same functions are correctly gated behind `ClvmFlags::ENABLE_SECP_OPS`. An attacker who supplies CLVM bytes using the 4-byte opcode form can invoke secp signature verification even when the hard-fork flag has not been activated, bypassing the intended access control entirely.

### Finding Description
`ChiaDialect::op` in `src/chia_dialect.rs` contains two separate dispatch paths for the secp operators.

**Path 1 – 4-byte opcode (lines 157–182):** When the operator atom is exactly 4 bytes, the function matches against two hard-coded opcodes and dispatches the corresponding function with no flag check:

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
``` [1](#0-0) 

**Path 2 – 1-byte opcode (lines 248–249):** The same two operators are also reachable via 1-byte opcodes 64 and 65, but here the dispatch is correctly guarded:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The flag `ENABLE_SECP_OPS` is defined as a hard-fork gate:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

The 4-byte opcode path was introduced to give these operators a cost-formula-compatible encoding (the multiplier bytes encode the fixed costs 1,300,000 and 1,850,000), but the `ENABLE_SECP_OPS` guard that protects the 1-byte aliases was never applied to the 4-byte aliases. The two paths call the identical underlying functions `op_secp256k1_verify` / `op_secp256r1_verify` in `src/secp_ops.rs`. [4](#0-3) 

### Impact Explanation
Any caller that constructs a CLVM program using the 4-byte opcode form (`0x13d61f00` or `0x1c3a8f00`) will have secp signature verification execute regardless of whether `ENABLE_SECP_OPS` is set. This produces a consensus divergence:

- A node running **without** `ENABLE_SECP_OPS` rejects opcode 64/65 (falls through to `unknown_operator`) but **accepts** `0x13d61f00`/`0x1c3a8f00` and executes the full secp verification.
- A node running **with** `ENABLE_SECP_OPS` accepts both forms identically.

The concrete corrupted result is the `Response` returned by `run_program`: one node returns `Ok(Reduction(cost, nil))` (spend accepted) while another returns an error or a no-op cost (spend rejected), splitting consensus on the validity of any coin whose puzzle uses the 4-byte secp opcode form before the hard fork activates.

### Likelihood Explanation
The 4-byte opcode encoding is a documented feature of the CLVM cost model and is visible in the source comments. A researcher or adversarial coin author who reads `chia_dialect.rs` can discover the bypass without any privileged access. The attacker-controlled entry point is the raw CLVM bytes passed to `run_program` — fully reachable from the Python wheel API, the CLI, and the mempool generator path. No special node configuration or social engineering is required.

### Recommendation
Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the guard already present on the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both opcode aliases are subject to the same hard-fork gate, eliminating the divergence.

### Proof of Concept
1. Construct a CLVM program that uses the 4-byte operator atom `\x13\xd6\x1f\x00` (secp256k1_verify) with a valid pubkey, message digest, and signature as arguments.
2. Execute it via `run_program` with a `ChiaDialect` instantiated **without** `ENABLE_SECP_OPS` (e.g., `ChiaDialect::new(ClvmFlags::empty())`).
3. Observe that the program returns `Ok(Reduction(1300000, nil))` — the secp verification ran and succeeded.
4. Execute the same logic using the 1-byte opcode `\x40` (64) under the same flags.
5. Observe that the program returns an error (`Unimplemented` or unknown-op no-op), confirming the bypass.

The divergence between steps 3 and 5 under identical flags is the consensus-splitting impact. [5](#0-4) [6](#0-5)
