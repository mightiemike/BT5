### Title
Duplicate Position IDs in `ClaimTierRewards` Enable Double-Claiming of Bonus Rewards — (File: x/tieredrewards/keeper/msg_server.go)

---

### Summary
The `ClaimTierRewards` message handler loads all requested positions into a slice before processing them. Because there is no deduplication of `msg.PositionIds`, an attacker who includes the same position ID twice causes the same position state (with its stale `LastEventSeq`) to be processed twice in `claimRewardsAndUpdatesPositions`, resulting in bonus rewards being paid out twice from the `RewardsPoolName` module account.

---

### Finding Description

`ClaimTierRewards` in `msg_server.go` first collects all positions into a slice:

```go
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    ...
    positions = append(positions, pos)
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
``` [1](#0-0) 

If the same `posId` appears twice in `msg.PositionIds`, two copies of the same `PositionState` — both carrying the identical stale `LastEventSeq` — are appended to the slice.

`claimRewardsAndUpdatesPositions` then iterates over the slice:

```go
for i := range positions {
    pos := &positions[i]
    ...
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)
    ...
    if err := k.setPosition(ctx, pos.Position, nil); err != nil { ... }
}
``` [2](#0-1) 

On the **first** iteration for the duplicated position:
- `processEventsAndClaimBonus` fetches all validator events since `pos.LastEventSeq`, computes bonus, sends coins from `RewardsPoolName` to the owner, and advances `pos.LastEventSeq` **in the in-memory copy**.
- `setPosition` persists the updated `LastEventSeq` to the store. [3](#0-2) 

On the **second** iteration for the same position (the second slice element):
- `pos` now points to the **second, unmodified copy** in the slice, which still holds the old `LastEventSeq`.
- `getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)` returns the **same events** again.
- Bonus is recomputed and `SendCoinsFromModuleToAccount` is called a second time.
- `decrementEventRefCount` is called again for the same events. [4](#0-3) 

The root cause is that the slice is populated from the store **before** any processing, so in-store updates made during the first iteration are invisible to the second copy of the same position.

---

### Impact Explanation

The `RewardsPoolName` module account is drained at a rate proportional to the number of duplicate IDs submitted. An attacker with a single position can multiply their bonus payout by N by submitting the same ID N times, subject only to the pool's current balance (checked by `sufficientBonusPoolBalance`). This directly corrupts the bonus-rewards accounting invariant: the pool pays out more than the position has accrued, under-collateralizing the rewards pool for all other participants. [5](#0-4) 

---

### Likelihood Explanation

Any position owner can craft a `MsgClaimTierRewards` transaction with a repeated position ID. No special privilege is required. The only precondition is that the validator the position is delegated to has at least one other position (so event reference counts are ≥ 2, preventing an underflow error on `decrementEventRefCount` from aborting the transaction). This is the normal state of any live validator with multiple delegators in the tiered-rewards system.

---

### Recommendation

Deduplicate `msg.PositionIds` before loading positions, or add an explicit check inside the loading loop:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(types.ErrDuplicatePositionID, "position id %d", posId)
    }
    seen[posId] = struct{}{}
    ...
}
```

Alternatively, enforce uniqueness in `msg.Validate()` so the check is applied at the message-validation layer before any state is touched.

---

### Proof of Concept

1. Attacker owns position ID `42`, delegated to validator `V`. Several other users also have positions on `V`, so each validator event has a reference count ≥ 2.
2. Attacker submits `MsgClaimTierRewards{Owner: attacker, PositionIds: [42, 42]}`.
3. The loading loop appends two identical `PositionState` copies (both with `LastEventSeq = S`) to the slice.
4. **Iteration 1**: `processEventsAndClaimBonus` processes all events since seq `S`, pays bonus `B` to attacker, advances in-memory `LastEventSeq` to `S'`, calls `setPosition` (store now reflects `S'`).
5. **Iteration 2**: the second slice element still has `LastEventSeq = S`; `getValidatorEventsSince` returns the same events; bonus `B` is paid again; `decrementEventRefCount` is called again for the same events (succeeds because ref count was ≥ 2).
6. Attacker receives `2B` instead of `B`. Repeating with `[42, 42, 42, …]` multiplies the payout further, draining the rewards pool. [6](#0-5) [7](#0-6)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
