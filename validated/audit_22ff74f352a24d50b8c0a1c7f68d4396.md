Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Depositors to Bypass the Deposit Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead validates the caller-supplied `owner` address against the allowlist. Because `owner` is an arbitrary parameter any caller can set, any non-allowlisted address can bypass the deposit guard by naming any allowlisted address as the LP position recipient.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as two distinct arguments into `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both verbatim to the extension via `abi.encodeCall`: [2](#0-1) 

The extension's hook receives `sender` as its first positional argument but names it `_` (anonymous, discarded) and checks `owner` instead: [3](#0-2) 

`sender` — the address that actually calls `addLiquidity` and supplies tokens via the `IMetricOmmSwapCallback` — is never read. The analogous `SwapAllowlistExtension.beforeSwap` correctly reads and checks `sender`: [4](#0-3) 

## Impact Explanation
The deposit allowlist is rendered completely ineffective. Any non-allowlisted address can deposit into a restricted pool by setting `owner` to any allowlisted address. The non-allowlisted address supplies the tokens (via the swap callback), the check passes because `owner` is allowlisted, and the LP position is minted under the allowlisted address without their consent. This breaks the core access-control invariant of the extension — pools relying on it for permissioned liquidity (e.g., KYC/compliance gating) silently admit unchecked depositors, constituting a broken core pool functionality with direct fund-flow impact.

## Likelihood Explanation
Exploitation requires no special privilege. Any EOA can call `addLiquidity` with `owner` set to any allowlisted address. The allowlist is publicly readable via `allowedDepositor` or emitted events, so identifying a valid `owner` value is trivial. The attack is a single transaction with no preconditions.

## Recommendation
Rename the first parameter from `_` to `sender` and validate it instead of `owner`:

```solidity
// Fixed
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

## Proof of Concept
1. Deploy pool with `DepositAllowlistExtension` as `beforeAddLiquidity` hook; `allowAllDepositors[pool] = false`.
2. Pool admin calls `setAllowedToDeposit(pool, Bob, true)`. Alice is **not** allowlisted.
3. Alice calls `pool.addLiquidity(owner = Bob, salt, deltas, callbackData, extensionData)`.
4. Pool calls `extension.beforeAddLiquidity(sender = Alice, owner = Bob, ...)`.
5. Extension evaluates `allowedDepositor[pool][Bob]` → `true` → no revert.
6. `LiquidityLib.addLiquidity` executes; Alice's callback is invoked and Alice transfers tokens into the pool.
7. LP position is minted under `(Bob, salt)`.
8. Alice — non-allowlisted — has successfully deposited into a restricted pool; the guard is fully bypassed. [5](#0-4) [6](#0-5)
