### Title
Unbounded Slow Mode Queue Flooding via Minimal Deposits Enables Permanent Censorship of User Withdrawals - (File: `core/contracts/Endpoint.sol`)

---

### Summary

`depositCollateralWithReferral` in `Endpoint.sol` appends an entry to the slow mode queue for every deposit with no per-user rate limit, no queue size cap, and no slow mode fee. The deposited tokens are credited back to the attacker's subaccount, making the effective cost of each queue entry gas-only. An attacker can flood the slow mode queue with millions of entries, permanently blocking legitimate users from processing their slow mode withdrawals — the protocol's only censorship-resistance fallback when the sequencer is unavailable.

---

### Finding Description

The slow mode queue is the protocol's censorship-resistance mechanism: when the sequencer is offline or censoring, users submit transactions via `submitSlowModeTransaction` (which charges `SLOW_MODE_FEE = $1`) or via `depositCollateralWithReferral` (which charges **no slow mode fee**). The queue is processed FIFO via `executeSlowModeTransaction`, callable by anyone.

**Root cause — `depositCollateralWithReferral` (Endpoint.sol lines 123–167):**

```solidity
function depositCollateralWithReferral(
    bytes32 subaccount,
    uint32 productId,
    uint128 amount,
    string memory
) public {
    // ...
    handleDepositTransfer(...);          // tokens transferred to clearinghouse
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
        executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
        sender: sender,
        tx: abi.encodePacked(...)
    });
    slowModeConfig = _slowModeConfig;   // no slow mode fee charged
}
``` [1](#0-0) 

Contrast with `submitSlowModeTransactionImpl`, which explicitly reverts for `DepositCollateral` (preventing the $1 fee path from being used for deposits) and charges `SLOW_MODE_FEE` for all other user-submitted transaction types: [2](#0-1) 

The minimum deposit for an existing account is `MIN_DEPOSIT_AMOUNT = $0.10`: [3](#0-2) 

The deposited tokens are credited to the attacker's subaccount in the clearinghouse. When the sequencer is online, the sequencer processes the attacker's deposits quickly, returning the funds. The attacker's net cost per queue entry is **gas only**.

The slow mode queue is an unbounded `mapping(uint64 => SlowModeTx)` with a `uint64` counter — no cap exists: [4](#0-3) 

`executeSlowModeTransaction` processes entries strictly in FIFO order (`txUpTo` pointer), so legitimate user withdrawals queued behind attacker entries cannot be reached until all attacker entries are cleared: [5](#0-4) 

---

### Impact Explanation

The slow mode queue is the **only on-chain fallback** for users to withdraw funds without sequencer cooperation. If the queue is flooded with N attacker entries, users must call `executeSlowModeTransaction` N times (each costing gas) before their own withdrawal is reachable. On Ink Chain (L2), gas is cheap for the attacker but the cumulative processing burden on users is unbounded.

**Concrete impact:** If the attacker pre-floods the queue with 1,000,000 entries before the sequencer goes offline, users cannot withdraw their funds via slow mode until all 1,000,000 entries are processed. At one call per block, this takes millions of blocks. User funds are effectively frozen for the duration of any sequencer outage. This is a permanent lock of user assets under the realistic scenario of sequencer unavailability.

---

### Likelihood Explanation

**Attacker preconditions:** Any address holding a small amount of the quote token (USDC or equivalent). No privileged access required.

**Attack cost:** On an L2 like Ink Chain, gas per transaction is on the order of fractions of a cent. To create 1,000,000 queue entries at $0.10 minimum deposit each, the attacker needs $100,000 in tokens — but recovers all of it once the sequencer processes the deposits. The net cost is gas only, which at L2 rates is negligible (< $100 for 1M transactions).

**Trigger:** The attacker calls `depositCollateralWithReferral` in a loop (or via a contract) while the sequencer is online, flooding the queue. When the sequencer later goes offline, the queue is already poisoned.

**Likelihood: Medium-High.** The attack is cheap, permissionless, and requires no special knowledge beyond the public ABI.

---

### Recommendation

1. **Charge a non-refundable slow mode fee for deposits.** Add `chargeSlowModeFee` to `depositCollateralWithReferral` (or a separate `depositCollateralSlowMode` entry point), matching the `SLOW_MODE_FEE = $1` charged for other slow mode transaction types. This makes flooding 1M entries cost $1M, not gas-only.

2. **Enforce a per-address rate limit or per-block queue entry cap** to bound the rate at which any single address can enqueue slow mode transactions.

3. **Add a maximum queue depth** (e.g., `require(txCount - txUpTo < MAX_QUEUE_DEPTH)`) to prevent unbounded growth.

---

### Proof of Concept

**Setup:** Attacker holds 10,000 USDC. Sequencer is online.

**Step 1 — Flood the queue:**
```solidity
// Attacker calls in a loop (or via a batch contract):
for (uint i = 0; i < 100_000; i++) {
    endpoint.depositCollateralWithReferral(
        attackerSubaccount,
        QUOTE_PRODUCT_ID,
        100_000,          // $0.10 minimum deposit
        "-1"
    );
}
// 100,000 entries added to slowModeTxs[0..99999]
// Attacker's $10,000 is credited back by sequencer quickly
// Net cost: gas only (~$10 on L2)
```

**Step 2 — Sequencer goes offline.**

**Step 3 — Victim tries to withdraw:**
```solidity
// Victim's WithdrawCollateral slow mode tx is at index 100,000
// Victim calls executeSlowModeTransaction() — processes attacker entry #0
// Victim must call it 100,000 times before reaching their own withdrawal
// Each call costs gas; at 1 call/block this takes ~100,000 blocks (~3+ days on L2)
// Attacker can keep adding more entries to extend the delay indefinitely
```

**Broken invariant:** The slow mode queue, intended as a permissionless censorship-resistance mechanism, is rendered non-functional for legitimate users at negligible cost to the attacker.
