Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address as Swapper Instead of End-User, Enabling Full Allowlist Bypass — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` always sets to `msg.sender` — the direct caller of the pool. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` in the pool is the router contract, not the end-user. The extension therefore evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. This creates an irreconcilable dilemma: if the router is not allowlisted, legitimate allowlisted users cannot swap through it; if the router is allowlisted, any unprivileged user can bypass the allowlist entirely with a single router call.

## Finding Description

**Root cause — wrong actor bound to `sender`:**

`MetricOmmPool.swap` (line 230–231) passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← router address when called via router
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` (lines 149–177) forwards this value unchanged to every configured extension via `_callExtensionsInOrder`.

`SwapAllowlistExtension.beforeSwap` (line 37) then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router address — the end-user's address is never seen by the extension.

**Router call path:**

`MetricOmmSimpleRouter.exactInputSingle` (lines 72–80) calls:

```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
);
```

The router passes no originator identity. Inside the pool, `msg.sender` is the router. The same pattern applies to `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165).

**Two failure modes:**

| Admin configuration | Result |
|---|---|
| Router NOT allowlisted | `allowedSwapper[pool][router] = false` → all router-mediated swaps revert, even for allowlisted users → broken core swap functionality |
| Router IS allowlisted | `allowedSwapper[pool][router] = true` → any unprivileged user calling the router passes the check → full allowlist bypass |

No existing guard compensates: `SwapAllowlistExtension` has no mechanism to decode an originator from `extensionData`, and the pool interface carries no `originator` field.

## Impact Explanation

For any pool deploying `SwapAllowlistExtension` (KYC-gated, institutional, or restricted-access pools), an unprivileged user can bypass the allowlist by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point). If the admin allowlists the router to enable legitimate users to trade, the allowlist is rendered entirely ineffective — any address can execute swaps against the restricted pool. This constitutes a direct admin-boundary break: an access control configured by the pool admin is bypassed by an unprivileged path through the router. Unauthorized swaps extract value from LP positions under conditions the pool admin explicitly did not intend to permit.

## Likelihood Explanation

The router is the primary user-facing entry point for swaps. The bypass requires no special privileges, no flash loans, and no multi-transaction setup — a single `exactInputSingle` call suffices. Any user who observes that a pool has `SwapAllowlistExtension` configured and that the router is allowlisted can immediately exploit this. The condition (router allowlisted to support legitimate users) is the natural operational state for any pool that intends to use both the allowlist and the router simultaneously.

## Recommendation

The `sender` identity forwarded to extensions must reflect the economic actor, not the intermediary. The preferred fix is to extend the pool's `swap` interface with an explicit `originator` field that the router populates with `msg.sender` (the end-user), and have `SwapAllowlistExtension.beforeSwap` check `originator` instead of `sender`. Alternatively, `SwapAllowlistExtension` can decode a signed originator from `extensionData`, though this requires router cooperation and careful replay protection. Checking `recipient` instead of `sender` is not equivalent — `recipient` is the output destination, not necessarily the economic actor initiating the swap.

## Proof of Concept

```solidity
// Pool deployed with SwapAllowlistExtension.
// Admin allowlists only `trustedUser` and also allowlists the router
// so that trustedUser can trade through it:
//   extension.setAllowedToSwap(pool, trustedUser, true);
//   extension.setAllowedToSwap(pool, router, true);  // required for router path to work

// Attacker (not allowlisted) calls the router:
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    tokenIn: token0,
    extensionData: ""
}));
// MetricOmmPool.swap: msg.sender = router → _beforeSwap(router, attacker, ...)
// SwapAllowlistExtension.beforeSwap: checks allowedSwapper[pool][router] = true → passes
// Attacker successfully swaps against the restricted pool.
```
