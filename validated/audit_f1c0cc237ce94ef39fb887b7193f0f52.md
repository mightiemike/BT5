### Title
Dead `DISABLE_OP` Guard Makes Unconditional 256-Byte Limit Overly Restrictive in Consensus Mode for `op_div`, `op_divmod`, and `op_mod` — (`File: src/more_ops.rs`)

### Summary

In `op_div`, `op_divmod`, and `op_mod` (and their malachite variants), a `DISABLE_OP`-gated check at 2048 bytes is dead code because an unconditional 256-byte check immediately follows it and is always more restrictive. The intended design — allow `a0_len` up to 2048 bytes in consensus mode, restrict to 256 bytes in mempool mode — is never realized. Instead, the 256-byte limit is applied unconditionally in both modes, causing any CLVM program that uses these operators with a dividend/numerator atom between 257 and 2048 bytes to be incorrectly rejected even in consensus mode.

### Finding Description

In `src/more_ops.rs`, all three arithmetic operators share the same guard pattern:

```rust
// op_div (line 665-669), op_divmod (line 713-717), op_mod (line 769-773)
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {   // ← dead code
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
if a0_len > 256 || a1_len > 1024 {                             // ← always fires first
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
```

Because `256 < 2048`, any input that would reach the `DISABLE_OP` branch (`a0_len > 2048`) has already been rejected by the unconditional branch (`a0_len > 256`). The `DISABLE_OP` check is therefore unreachable dead code in all four variants (`op_div`, `op_div_malachite`, `op_divmod`, `op_divmod_malachite`, `op_mod`, `op_mod_malachite`).

The `DISABLE_OP` flag is included in `MEMPOOL_MODE` (the stricter mode):

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)          // ← stricter mempool flag
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

The intended semantics, consistent with how `DISABLE_OP` is used elsewhere (e.g., completely disabling `op_modpow` at opcode 60 in mempool mode), is:

- **Consensus mode** (no `DISABLE_OP`): allow `a0_len` up to 2048 bytes
- **Mempool mode** (`DISABLE_OP` set): restrict `a0_len` to 256 bytes

The correct code order should be:

```rust
if a0_len > 2048 || a1_len > 1024 {                           // unconditional consensus limit
    return Err(...);
}
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 256 {    // stricter mempool limit
    return Err(...);
}
```

Instead, the checks are inverted: the 256-byte limit is applied unconditionally, and the 2048-byte `DISABLE_OP` check is dead code. The same inversion exists in all six affected function bodies. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

### Impact Explanation

Any CLVM program whose puzzle or solution causes `op_div`, `op_divmod`, or `op_mod` to be called with a dividend/numerator atom of 257–2048 bytes is unconditionally rejected by `clvm_rs` — even in full-node consensus mode where the 2048-byte limit was intended to apply. Coins locked by such puzzles become permanently unspendable: the spend attempt is rejected at the CLVM evaluation layer before any signature or condition check is reached. If a future or alternative CLVM implementation correctly implements the 2048-byte consensus limit, nodes running `clvm_rs` would diverge from that implementation, accepting blocks the other rejects or vice versa.

### Likelihood Explanation

The entry path is fully attacker-controlled: an adversary submits a serialized CLVM program (via `run_serialized_chia_program` or the Rust `run_program` API) that invokes opcode 19 (`/`), 20 (`divmod`), or 61 (`%`) with a first argument atom of 257–2048 bytes. No privilege is required. The program bytes are the only input needed. A coin owner who legitimately designs a puzzle using large-integer division (e.g., for cryptographic or financial computations within the 2048-byte design limit) would find their coin permanently frozen with no recourse, directly mirroring the H-5 frozen-asset scenario.

### Recommendation

Swap the order of the two checks so the unconditional guard enforces the 2048-byte consensus limit and the `DISABLE_OP` guard enforces the stricter 256-byte mempool limit. Apply the fix to all six affected functions: `op_div`, `op_div_malachite`, `op_divmod`, `op_divmod_malachite`, `op_mod`, and `op_mod_malachite`.

```rust
// Correct pattern for op_div, op_divmod, op_mod (and malachite variants):
if a0_len > 2048 || a1_len > 1024 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 256 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
```

### Proof of Concept

```rust
use clvmr::allocator::Allocator;
use clvmr::chia_dialect::{ChiaDialect, ClvmFlags, MEMPOOL_MODE};
use clvmr::more_ops::op_div;
use clvmr::op_utils::get_args;

fn main() {
    let mut a = Allocator::new();

    // Build a 300-byte dividend atom (within the intended 2048-byte consensus limit)
    let big_dividend = vec![0x01u8; 300];
    let divisor = vec![0x01u8];

    let v0 = a.new_atom(&big_dividend).unwrap();
    let v1 = a.new_atom(&divisor).unwrap();
    let nil = a.nil();
    let args = a.new_pair(v0, nil).unwrap();
    let args = a.new_pair(v1, args).unwrap(); // wrong order, just for illustration
    // Correct arg list: (v0 . (v1 . nil))
    let args2 = a.new_pair(v1, nil).unwrap();
    let args2 = a.new_pair(v0, args2).unwrap();

    // Consensus mode: DISABLE_OP not set — should allow 300-byte dividend per design
    let result = op_div(&mut a, args2, u64::MAX, ClvmFlags::empty());
    // BUG: returns Err(InvalidOpArg) — incorrectly rejected
    assert!(result.is_err(), "Expected success in consensus mode, got error");

    // Mempool mode: DISABLE_OP set — correctly rejects 300-byte dividend
    let result2 = op_div(&mut a, args2, u64::MAX, MEMPOOL_MODE);
    assert!(result2.is_err());
}
```
