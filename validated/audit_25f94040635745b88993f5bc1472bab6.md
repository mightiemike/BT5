Audit Report

## Title
`OracleValueStopLossExtension` Uses Arithmetic Mid-Price While Pool Uses Geometric Mid-Price, Causing Stop-Loss to Underprotect LPs on `zeroForOne` Swaps - (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss()` computes the oracle mid-price as the arithmetic mean of bid and ask at line 218, while `SwapMath.midAndSpreadFeeX64FromBidAsk()` — used by `PriceVelocityGuardExtension` and `MetricOmmPool.swap()` — computes it as the geometric mean. By AM-GM inequality, arithmetic mean ≥ geometric mean when bid ≠ ask, so the stop-loss guard's `metricT0` value-per-share is systematically understated. This causes the guard to measure a smaller relative value loss than actually occurred for `zeroForOne == true` swaps, allowing LPs to lose more than the configured `drawdownE6` before the stop-loss triggers.

**Algebraic proof of underprotection:**

Let `p_a = Q64/arith_mid` and `p_g = Q64/geom_mid`. Since `arith_mid > geom_mid`, we have `p_a < p_g`.

For a `zeroForOne == true` swap (token0 in Δ0, token1 out Δ1), define:
- Before: `V_arith = A + B·p_a`, `V_geom = A + B·p_g`
- After: `V'_arith = (A+Δ0) + (B-Δ1)·p_a`, `V'_geom = (A+Δ0) + (B-Δ1)·p_g`

Cross-multiplying `V'_arith/V_arith` vs `V'_geom/V_geom`:

```
V'_arith·V_geom - V'_geom·V_arith = (p_g - p_a)(B·Δ0 + A·Δ1) ≥ 0
```

Therefore `V'_arith/V_arith ≥ V'_geom/V_geom` always. The guard-measured relative value is always at least as high as the true geometric-mid relative value, meaning the guard sees a smaller relative loss and its floor `hwm0 * floorMultiplier / E6` is set lower than the true geometric-mid floor.

## Impact Explanation

The contract's documented guarantee — *"value per share at oracle marks cannot fall faster than drawdown (one-time) + decay × t"* — is violated for `zeroForOne == true` swaps. The stop-loss guard permits the pool to lose more than `drawdownE6` of its true (geometric-mid) value before reverting. LP principal is directly at risk beyond the configured protection level. For a 10% oracle spread (bid=0.9, ask=1.1), the excess unprotected loss is approximately 0.5% of the t1 portfolio component per drawdown event. The excess is systematic and accumulates across every `zeroForOne == true` swap on every pool configured with this extension.

## Likelihood Explanation

This triggers on every swap through any pool configured with `OracleValueStopLossExtension` where the oracle spread is non-zero — i.e., all real deployments. No special attacker setup is required. Any public swap on such a pool exercises the discrepancy. The guard is silently less protective than configured for the entire lifetime of the pool.

## Recommendation

Replace the arithmetic mean in `_afterSwapOracleStopLoss` with the same geometric mean used everywhere else in the protocol:

```solidity
// Before (arithmetic mean — wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (geometric mean — consistent with pool execution):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64), uint256(askPriceX64)
);
```

Add `import {SwapMath} from "@metric-core/libraries/SwapMath.sol";` to the file.

## Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension` configured with `drawdownE6 = 50_000` (5%).
2. Set oracle bid = `0.9 * Q64`, ask = `1.1 * Q64` (10% spread). Geometric mid ≈ `0.9950 * Q64`; arithmetic mid = `1.0 * Q64`.
3. Seed the pool with `t0 = 1000`, `t1 = 1000` scaled units. The guard sets `hwm0` using arithmetic mid (lower value).
4. Execute a sequence of `zeroForOne == true` swaps that drain t1 at a price slightly worse than geometric mid.
5. Observe: the guard does not revert even though the true geometric-mid value has fallen by more than 5%, because the guard's `metricT0` (computed at arithmetic mid) shows a smaller loss than actually occurred.
6. Confirm: replacing line 218 with `SwapMath.midAndSpreadFeeX64FromBidAsk(...)` causes the guard to revert at the correct drawdown threshold.
