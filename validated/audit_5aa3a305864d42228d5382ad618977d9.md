### Title
`ValidResourceBounds::AllResources` with zero L2/L1-data gas silently collapses to `L1Gas` variant during protobuf deserialization, producing a divergent transaction hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a zero-value heuristic to decide which enum variant to reconstruct. When a V3 transaction carries `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the serialized protobuf bytes are byte-for-byte identical to those produced by `L1Gas(X)`. On deserialization the converter silently picks `L1Gas(X)`. Because `get_tip_resource_bounds_hash` includes `l1_data_gas` in the Poseidon preimage only for the `AllResources` variant, the two variants produce **different transaction hashes** for the same user-supplied bounds. A transaction accepted and hashed by the sequencer under the `AllResources` rule cannot be re-verified by any node that deserializes it through this protobuf path.

---

### Finding Description

**Step 1 – Serialization is lossy.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes both variants to the same wire bytes when l2_gas and l1_data_gas are zero:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  lines 471-489
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),   // zero
    l1_data_gas: Some(ResourceBounds::default().into()), // zero
},
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),
    },
```

When `l2_gas == 0` and `l1_data_gas == 0`, both arms emit identical bytes. [1](#0-0) 

**Step 2 – Deserialization always picks `L1Gas` for the zero case.**

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  lines 417-436
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
// ...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← always chosen when both are zero
} else {
    ValidResourceBounds::AllResources(...)
})
```

An `AllResources` transaction with zero l2/l1-data gas is reconstructed as `L1Gas`. [2](#0-1) 

**Step 3 – The two variants produce different Poseidon hashes.**

`get_tip_resource_bounds_hash` conditionally appends `l1_data_gas` to the hash preimage only for `AllResources`:

```rust
// crates/starknet_api/src/transaction_hash.rs  lines 203-208
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2-element preimage
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3-element preimage
    }
});
```

`poseidon(tip, L1_GAS||X, L2_GAS||0, L1_DATA_GAS||0)` ≠ `poseidon(tip, L1_GAS||X, L2_GAS||0)`. [3](#0-2) 

**Step 4 – The gateway accepts `AllResources` with zero l2/l1-data gas.**

The stateless validator explicitly allows transactions where only `l1_gas` is non-zero: [4](#0-3) 

**Step 5 – The `Transaction` type used in the sync path carries `ValidResourceBounds`.**

`InvokeTransactionV3.resource_bounds` is `ValidResourceBounds`. When the sequencer converts an incoming `RpcInvokeTransactionV3` (which uses `AllResourceBounds`) to the internal `InvokeTransactionV3`, it always wraps it as `ValidResourceBounds::AllResources(...)`. The hash is computed and stored under the `AllResources` rule. When a peer node receives this transaction over P2P and deserializes it through the protobuf converter above, it reconstructs `L1Gas` and recomputes a different hash. [5](#0-4) 

---

### Impact Explanation

A syncing node that receives a block containing a V3 transaction with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` will:

1. Deserialize the transaction as `L1Gas(X)`.
2. Recompute the transaction hash using the 2-element Poseidon preimage.
3. Obtain a hash that differs from the one committed by the sequencer (3-element preimage).
4. Fail hash validation and reject the block or store the transaction under the wrong hash.

This causes the syncing node to diverge from the canonical chain state for any block containing such a transaction. The RPC layer of the syncing node will then serve wrong receipts, wrong transaction data, and wrong block hashes — matching the "Wrong state / wrong value from RPC" High impact category.

---

### Likelihood Explanation

Any user can craft a valid V3 invoke/declare/deploy-account transaction with `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas` set). The gateway accepts it. Once included in a block, every node that syncs that block via P2P is affected. The trigger requires no special privilege and no malicious peer — a normal user transaction is sufficient.

---

### Recommendation

The serialization must encode the variant unambiguously. Two concrete options:

1. **Add a discriminant field** to `protobuf::ResourceBounds` (e.g., `bool all_resources`) so the deserializer can reconstruct the correct variant regardless of whether the numeric values are zero.

2. **Normalize at ingestion**: when the gateway converts `AllResourceBounds { l2_gas: 0, l1_data_gas: 0 }` to `ValidResourceBounds`, produce `L1Gas` immediately so the hash is computed under the `L1Gas` rule from the start, making the round-trip consistent.

Option 2 is simpler but changes which hash rule applies to new transactions; option 1 preserves the existing hash semantics without ambiguity.

---

### Proof of Concept

```
1. Submit RpcInvokeTransactionV3 with resource_bounds = AllResourceBounds {
       l1_gas: ResourceBounds { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas: ResourceBounds::default(),      // zero
       l1_data_gas: ResourceBounds::default(), // zero
   }

2. Sequencer converts to InvokeTransactionV3 with
       ValidResourceBounds::AllResources(AllResourceBounds { l1_gas: 1000/1, l2_gas: 0, l1_data_gas: 0 })
   Hash H1 = poseidon(0, L1_GAS||1000||1, L2_GAS||0||0, L1_DATA_GAS||0||0)

3. Transaction is included in block B.

4. Peer node receives block B via P2P protobuf sync.
   Deserializer sees l2_gas.is_zero() && l1_data_gas.is_zero() → picks L1Gas(1000/1).
   Hash H2 = poseidon(0, L1_GAS||1000||1, L2_GAS||0||0)

5. H1 ≠ H2 → hash validation fails → block B is rejected by the peer node.
```
