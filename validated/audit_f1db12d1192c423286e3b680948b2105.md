### Title
Trade Execution with Zero Quote Payment for Sub-Precision Priced Tokens — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`sizeIncrement` and `minSize` are stored in `OffchainExchange` with a hard precision floor of `1e9` (in 18-decimal representation) due to integer truncation in `updateMarket()`. This creates a minimum effective trade size of `1e9`. Combined with the fixed-point `mul()` computation in `matchOrders()`, any token whose `priceX18 < 1e9` (i.e., price < `1e-9` in human-readable terms) produces `quoteDelta = 0`. Trades for such tokens execute with zero quote payment: the taker receives base tokens and the maker loses base tokens with no quote compensation — a direct accounting corruption.

---

### Finding Description

**Root Cause — Precision Floor in `updateMarket()`**

In `OffchainExchange.updateMarket()`, both `sizeIncrement` and `minSize` are compressed into `int64` by dividing by `1e9` before storage:

```solidity
// OffchainExchange.sol lines 279–280
marketInfo[productId].minSize      = int64(minSize      / 1e9);
marketInfo[productId].sizeIncrement = int64(sizeIncrement / 1e9);
``` [1](#0-0) 

When read back, they are multiplied by `1e9`:

```solidity
// OffchainExchange.sol lines 112–113
m.minSize       = int128(market.minSize)       * 1e9;
m.sizeIncrement = int128(market.sizeIncrement) * 1e9;
``` [2](#0-1) 

Any `sizeIncrement` value whose magnitude is less than `1e9` (in 18-decimal) is silently truncated to `0` on write and reads back as `0`. The minimum non-zero representable `sizeIncrement` after the round-trip is exactly `1e9`.

**Broken Invariant — `quoteDelta` Truncates to Zero**

In `matchOrders()`, the filled amount is snapped to the nearest `sizeIncrement`, then `quoteDelta` is computed via fixed-point multiply:

```solidity
// OffchainExchange.sol lines 757–762
ordersInfo.taker.amountDelta -=
    ordersInfo.taker.amountDelta % market.sizeIncrement;
ordersInfo.maker.quoteDelta = ordersInfo.taker.amountDelta.mul(
    maker.order.priceX18
);
``` [3](#0-2) 

`MathSD21x18.mul()` computes `(x * y) / 1e18`:

```solidity
// MathSD21x18.sol lines 54–59
function mul(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        int256 result = (int256(x) * y) / ONE_X18;   // ONE_X18 = 1e18
        ...
    }
}
``` [4](#0-3) 

With the minimum non-zero `amountDelta = 1e9` and any `priceX18 < 1e9`:

```
quoteDelta = (1e9 × priceX18) / 1e18
           = priceX18 / 1e9
           = 0   (integer division truncates)
```

The only price guard in `_validateOrder()` is `priceX18 > 0`:

```solidity
// OffchainExchange.sol line 458
return ((order.priceX18 > 0) || _isTWAP(order.appendix)) && ...
``` [5](#0-4) 

There is no on-chain floor preventing `priceX18 = 1`. The `isHealthy` stub in the base contract always returns `true`, so no post-trade health gate catches the zero-quote fill:

```solidity
// OffchainExchange.sol lines 625–629
function isHealthy(bytes32 /* subaccount */) internal view virtual returns (bool) {
    return true;
}
``` [6](#0-5) 

After `quoteDelta = 0` is computed, both sides' balances are updated unconditionally:

```solidity
// OffchainExchange.sol lines 811–824
_updateBalances(callState, market.quoteId, taker.order.sender,
    ordersInfo.taker.amountDelta, ordersInfo.taker.quoteDelta);   // quoteDelta = 0
_updateBalances(callState, market.quoteId, maker.order.sender,
    ordersInfo.maker.amountDelta, ordersInfo.maker.quoteDelta);   // quoteDelta = 0
``` [7](#0-6) 

The taker's base balance increases by `amountDelta`; the maker's base balance decreases by `amountDelta`; neither party's quote balance changes.

---

### Impact Explanation

For any spot or perp product whose oracle price satisfies `priceX18 < 1e9` (i.e., token price below `1e-9` in human-readable units), every matched order at the minimum trade size results in:

- **Taker:** receives `amountDelta` base tokens, pays `0` quote tokens.
- **Maker:** loses `amountDelta` base tokens, receives `0` quote tokens.

This is a direct, concrete accounting corruption. The maker's position is drained without compensation. Repeated fills drain the maker's entire base balance to zero at no cost to the taker. For perp markets, the maker's `vQuoteBalance` is never credited, permanently corrupting unrealized PnL and downstream liquidation health calculations.

---

### Likelihood Explanation

Memecoins and micro-cap tokens with prices below `1e-9` (human-readable) are common. The threshold is `priceX18 < 1e9`, which in 18-decimal fixed-point corresponds to a token price below `0.000000001` of the quote asset. Any such product listed on Nado is immediately affected. The sequencer matches orders in good faith without an on-chain guard; no off-chain check is mandated by the protocol, and the sequencer has no reason to suspect the `quoteDelta` silently truncates to zero.

---

### Recommendation

1. **Enforce a minimum `quoteDelta` check in `matchOrders()`:** After computing `ordersInfo.maker.quoteDelta`, require it to be non-zero (or at least `>= 1`) before proceeding with balance updates.
2. **Enforce a minimum effective price at order validation time:** In `_validateOrder()`, require `order.priceX18.mul(market.sizeIncrement) >= 1` so that any order whose price would produce a zero `quoteDelta` at minimum fill size is rejected at the gate.
3. **Alternatively, raise the storage precision:** Store `sizeIncrement` and `minSize` as full `int128` values rather than compressing to `int64` after dividing by `1e9`, eliminating the precision floor entirely.

---

### Proof of Concept

1. A memecoin spot product is listed with `sizeIncrement = 1e9` (minimum non-zero value after the `int64` truncation round-trip) and oracle `priceX18 = 500` (price ≈ `5e-16` human-readable).
2. User A (taker) signs a buy order: `priceX18 = 500`, `amount = 1e9`.
3. User B (maker) signs a sell order: `priceX18 = 500`, `amount = -1e9`.
4. Sequencer submits `matchOrders`. Both orders pass `_validateOrder` (`priceX18 = 500 > 0`).
5. `amountDelta = 1e9`; `quoteDelta = (1e9 × 500) / 1e18 = 5e11 / 1e18 = 0`.
6. `_updateBalances` credits taker `+1e9` base, `0` quote; debits maker `-1e9` base, `0` quote.
7. `isHealthy` returns `true` for both. Trade settles. Maker has lost `1e9` base tokens and received nothing.
8. The taker repeats with fresh orders until the maker's entire base balance is drained at zero cost.
