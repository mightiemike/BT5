### Title
Gateway Rejects Valid Declare Transactions for Already-Compiled Classes When Compilation Semaphore Is Exhausted - (`crates/apollo_gateway/src/gateway.rs`)

### Summary

The gateway unconditionally acquires the `declare_compilation_semaphore` for every declare transaction before calling `add_class`. However, the class manager's `add_class` returns immediately (without compilation) when the class is already cached. When the semaphore is exhausted, valid declare transactions for already-compiled classes are rejected with `TransactionLimitExceeded`, even though they would consume zero compilation resources.

### Finding Description

In `crates/apollo_gateway/src/gateway.rs`, the semaphore is acquired for **all** declare transactions:

```rust
// lines 243–251
let compilation_permit = if matches!(tx, RpcTransaction::Declare(_)) {
    Some(self.declare_compilation_semaphore.try_acquire().map_err(|_| {
        let error = StarknetError::too_many_concurrent_declare_compilations();
        metric_counters.record_add_tx_failure(&error);
        error
    })?)
} else {
    None
};
``` [1](#0-0) 

The semaphore is acquired before `convert_rpc_tx_to_internal_and_executable_txs` is called, which internally calls `class_manager_client.add_class`. However, `add_class` in the class manager short-circuits immediately when the class is already stored:

```rust
// lines 74–79
if let Ok(Some(executable_class_hash_v2)) =
    self.classes.get_executable_class_hash_v2(class_hash)
{
    // Class already exists.
    return Ok(ClassHashes { class_hash, executable_class_hash_v2 });
}
``` [2](#0-1) 

This is confirmed by the class manager test comment: *"Add existing class; response returned immediately, without invoking compilation."* [3](#0-2) 

The semaphore is initialized with `max_concurrent_declare_compilations` (default: 40) permits: [4](#0-3) 

When all 40 permits are held by genuine compilations, a declare for a class that is already compiled is rejected with `KnownStarknetErrorCode::TransactionLimitExceeded`: [5](#0-4) 

The test `test_declare_compilation_concurrency_limit` confirms this rejection path is reachable and that the second declare is rejected while the first holds the single permit: [6](#0-5) 

### Impact Explanation

A valid declare transaction for a class that is already compiled and cached in the class manager is rejected at the gateway admission layer. The user receives a `TransactionLimitExceeded` error and must retry. This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

An attacker can flood the gateway with 40 concurrent declare transactions for novel (never-seen) Sierra classes. While those compilations are in progress, any legitimate user submitting a declare for an already-compiled class (e.g., a widely-used ERC-20 or account contract) is rejected. The default limit of 40 is reachable without special privileges. The attacker only needs to submit 40 large Sierra classes simultaneously to saturate the semaphore. [7](#0-6) 

### Recommendation

Before acquiring the semaphore, perform a lightweight pre-check against the class manager to determine whether the class is already compiled. If it is, skip semaphore acquisition entirely:

```rust
let class_already_compiled = if let RpcTransaction::Declare(RpcDeclareTransaction::V3(ref d)) = tx {
    self.class_manager_client
        .get_executable_class_hash_v2(d.contract_class.calculate_class_hash())
        .await
        .ok()
        .flatten()
        .is_some()
} else {
    false
};

let compilation_permit = if matches!(tx, RpcTransaction::Declare(_)) && !class_already_compiled {
    Some(self.declare_compilation_semaphore.try_acquire().map_err(|_| {
        StarknetError::too_many_concurrent_declare_compilations()
    })?)
} else {
    None
};
```

This mirrors the KUMASwap fix: only apply the capacity gate when the resource would actually be consumed (i.e., when the class is not already present in the set/cache).

### Proof of Concept

1. Configure the gateway with `max_concurrent_declare_compilations = 1`.
2. Submit a declare transaction for a novel class `A`. The compilation starts and holds the single permit.
3. While compilation of `A` is in progress, submit a second declare for a class `B` that was previously compiled and is already in the class manager cache.
4. The gateway calls `try_acquire()` on the semaphore, finds no permits available, and returns `TransactionLimitExceeded` — even though `add_class(B)` would have returned immediately without touching the compiler.
5. The legitimate declare for `B` is dropped from the gateway, forcing the user to retry.

The existing test `test_declare_compilation_concurrency_limit` already demonstrates step 4 (the second declare is rejected while the first holds the permit), and `class_manager_test.rs` confirms step 4's premise (already-compiled classes skip compilation). [8](#0-7) [3](#0-2)

#[tokio::test(flavor = "multi_thread")]
async fn test_declare_compilation_concurrency_limit(mut mock_dependencies: MockDependencies) {
    mock_dependencies.config.static_config.max_concurrent_declare_compilations = 1;

    // Both declares run stateless validation, so replace the fixture's single-call mock with one
    // that accepts repeated calls.
    let mut mock_stateless_transaction_validator = MockStatelessTransactionValidatorTrait::new();
    mock_stateless_transaction_validator.expect_validate().returning(|_| Ok(()));
    mock_dependencies.mock_stateless_transaction_validator = mock_stateless_transaction_validator;

    // The first declare's conversion performs the Sierra-to-CASM compilation while holding the
    // single permit. Make that conversion block so the second declare arrives while the permit is
    // still held: `compilation_started_sender` signals that the permit is held, and the conversion
    // then parks on `release_compilation_receiver` until the test lets it finish.
    let (compilation_started_sender, compilation_started_receiver) =
        tokio::sync::oneshot::channel();
    let (release_compilation_sender, release_compilation_receiver) = std::sync::mpsc::channel();
    mock_dependencies
        .mock_transaction_converter
        .expect_convert_rpc_tx_to_internal_rpc_tx()
        .return_once(move |_| {
            compilation_started_sender.send(()).unwrap();
            release_compilation_receiver.recv().unwrap();
            // Fail the conversion so the first declare short-circuits here instead of running the
            // full admission path; its outcome is irrelevant, so the specific error is arbitrary.
            Err(TransactionConverterError::ClassNotFound { class_hash: ClassHash::default() })
        });

    let gateway = Arc::new(mock_dependencies.gateway());

    // Spawn the first declare and wait until it holds the permit inside compilation.
    let first_declare_task = {
        let gateway = gateway.clone();
        tokio::spawn(async move { gateway.add_tx(declare_tx(), None).await })
    };
    compilation_started_receiver.await.unwrap();

    // The second declare cannot acquire a permit and must be rejected immediately.
    let second_declare_error = gateway.add_tx(declare_tx(), None).await.unwrap_err();
    assert_eq!(
        second_declare_error.code,
        StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::TransactionLimitExceeded)
    );

    // Release the first declare so its task and the blocked worker thread can wind down.
    release_compilation_sender.send(()).unwrap();
    let _ = first_declare_task.await;
