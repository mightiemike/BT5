### Title
Unbounded Validator Event Loop in `processEventsAndClaimBonus` Permanently Locks Bonus Rewards and Delegated Funds - (File: `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` iterates over every pending `ValidatorEvent` since a position's `LastEventSeq` with no upper bound. Because every mutation path (`MsgTierUndelegate`, `MsgTierRedelegate`, `MsgAddToTierPosition`, `MsgClearPosition`, `MsgExitTierWithDelegation`) calls `claimRewards` — which calls `processEventsAndClaimBonus` — before modifying state, a position whose event backlog exceeds the block gas limit becomes permanently frozen: the owner can neither claim bonus rewards nor recover their delegated principal.

---

### Finding Description

`processEventsAndClaimBonus` loads all events since `pos.LastEventSeq` via `getValidatorEventsSince` and processes them in a single unbounded loop:

```go
// x/tieredrewards/keeper/claim_rewards.go  lines 153–198
events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
...
for _, entry := range events {          // no cap
    ...
    if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
        return nil, err
    }
}
``` [1](#0-0) 

Each iteration performs at minimum one KV store read and one KV store write (`decrementEventRefCount`). The position's `LastEventSeq` checkpoint is only persisted after the entire loop completes and the caller calls `setPosition`; a mid-loop gas exhaustion reverts the transaction entirely, leaving the checkpoint unchanged. [2](#0-1) 

`getValidatorEventsSince` materialises the full event slice into memory before the loop begins, so both memory and gas grow linearly with the backlog size. [3](#0-2) 

Every user-facing mutation message calls `claimRewards` (which calls `processEventsAndClaimBonus`) as a mandatory first step before any state change: [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

There is no partial-claim path for the event backlog. `MsgClaimTierRewards` caps the number of *positions* per call at `MaxClaimPositionIds = 300`, but places no cap on the number of events processed per position. [9](#0-8) [10](#0-9) 

---

### Impact Explanation

If the event backlog for a position exceeds the block gas limit, every transaction that touches that position fails unconditionally. The position owner:

1. **Cannot claim bonus rewards** — `MsgClaimTierRewards` → `processEventsAndClaimBonus` exhausts gas.
2. **Cannot redelegate** — `MsgTierRedelegate` calls `claimRewards` first; same failure.
3. **Cannot undelegate** — `MsgTierUndelegate` calls `claimRewards` first; same failure.
4. **Cannot exit with delegation** — `MsgExitTierWithDelegation` calls `claimRewards` first; same failure.
5. **Cannot add to position** — `MsgAddToTierPosition` calls `claimRewards` first; same failure.

The position's delegator address is a module-derived account; the owner cannot sign standard staking `MsgUndelegate` from it. The result is:

- **Bonus rewards permanently locked** in the `RewardsPoolName` module account — the owner earned them but can never receive them.
- **Delegated principal permanently locked** — the owner cannot recover their staked tokens through any supported message path.

The corrupted invariant is: *"User-driven paths fail atomically; user retries after pool is replenished"* (ADR-006 §5). When gas exhaustion is the failure cause, retrying is impossible regardless of pool balance. [11](#0-10) 

---

### Likelihood Explanation

The scenario requires two concurrent conditions:

1. **A validator with a large event backlog.** Each jailing/unjailing cycle appends two events (`UNBOND` + `BOND`); each slash appends one. A validator that jails and unjails once per day accumulates ~730 events per year. Over a 5-year lock tier, that is ~3 650 events. At ~10 000 gas per event (KV read + write), this approaches or exceeds a typical Cosmos SDK block gas limit of 40 M–100 M gas.

2. **A position that has not claimed recently.** Long-term lock tiers (e.g., 5-year) are the primary use case of the module. Users who lock for years without frequent claims are the intended audience. The reference-count mechanism does not help: events are only garbage-collected after *all* positions on the validator process them, so a single long-dormant position keeps the entire backlog alive.

Neither condition requires attacker control. Both arise from normal, intended usage of the protocol.

---

### Recommendation

Add a per-call event processing cap inside `processEventsAndClaimBonus` and persist the `LastEventSeq` checkpoint after each processed batch, so that a subsequent call continues from where the previous one stopped. Expose this as a dedicated `MsgAdvanceEventCheckpoint` (or reuse `MsgClaimTierRewards` with a `max_events` parameter) so users can drain their backlog incrementally before attempting mutations. This mirrors the external report's recommendation to "allow users to claim their rewards partially in several batches."

---

### Proof of Concept

1. Deploy the chain with a 5-year tier (`ExitDuration = 5 years`, `BonusApy > 0`).
2. User A calls `MsgLockTier` on validator V. Position P is created with `LastEventSeq = N`.
3. Validator V is jailed and unjailed repeatedly (e.g., via a script that submits evidence). Each cycle appends two `ValidatorEvent` entries. After K cycles, `ValidatorEventSeq[V] = N + 2K`.
4. User A does not call `MsgClaimTierRewards` for an extended period.
5. User A submits `MsgClaimTierRewards{PositionIds: [P]}` with the maximum allowed gas. `processEventsAndClaimBonus` attempts to iterate all 2K events. When 2K is large enough, the transaction runs out of gas and is reverted. `LastEventSeq` for P remains at N.
6. User A attempts `MsgTierUndelegate` — fails for the same reason (calls `claimRewards` first).
7. User A attempts `MsgTierRedelegate` to a validator with zero events — fails for the same reason.
8. User A's bonus rewards remain in `RewardsPoolName`; their delegated tokens remain locked in the position's delegator address with no recovery path. [12](#0-11) [13](#0-12) [14](#0-13) [15](#0-14)

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil

func (k Keeper) processEventsAndClaimBonus(ctx context.Context, pos *types.PositionState) (sdk.Coins, error) {
	// Rewards should have been claimed before undelegation
	if !pos.IsDelegated() {
		return sdk.NewCoins(), nil
	}

	valAddr, err := sdk.ValAddressFromBech32(pos.Delegation.ValidatorAddress)
	if err != nil {
		return nil, err
	}

	events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	blockTime := sdkCtx.BlockTime()

	totalBonus := math.ZeroInt()
	// Use the persisted bonded state from the last replay, not a hardcoded default.
	// This prevents overpaying bonus for unbonded gaps between claims.
	bonded := pos.LastKnownBonded
	segmentStart := pos.LastBonusAccrual

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	for _, entry := range events {
		evt := entry.Event

		if bonded {
			// Compute bonus for the bonded segment [segmentStart, eventTime]
			// using the snapshot rate at the event.
			bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
			totalBonus = totalBonus.Add(bonus)
		}

		// Update bonded state based on event type.
		switch evt.EventType {
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND:
			bonded = false
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND:
			bonded = true
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_SLASH:
			// Slash doesn't change bonded state.
		}

		segmentStart = evt.Timestamp
		pos.UpdateLastEventSeq(entry.Seq)

		// Decrement reference count.
		if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
			return nil, err
		}
	}

func (k Keeper) getValidatorEventsSince(ctx context.Context, valAddr sdk.ValAddress, startSeq uint64) ([]EventEntry, error) {
	// Range from (valAddr, startSeq+1) to end of valAddr prefix.
	rng := collections.NewPrefixedPairRange[sdk.ValAddress, uint64](valAddr).
		StartExclusive(startSeq)

	var entries []EventEntry
	err := k.ValidatorEvents.Walk(ctx, rng, func(key collections.Pair[sdk.ValAddress, uint64], event types.ValidatorEvent) (bool, error) {
		entries = append(entries, EventEntry{Seq: key.K2(), Event: event})
		return false, nil
	})
	if err != nil {
		return nil, err
	}
	return entries, nil
}

		TokensPerShare: tokensPerShare,
		ReferenceCount: count,
	})
