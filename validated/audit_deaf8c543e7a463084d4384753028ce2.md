Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Unauthorized Depositors to Bypass the Allowlist Guard - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and gates access only on `owner` (the position beneficiary). Because `MetricOmmPool.addLiquidity` explicitly permits `msg.sender != owner`, any unprivileged actor can bypass the deposit allowlist by naming an allowlisted address as `owner`, nullifying the pool admin's configured access control.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as separate arguments to the extension hook:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` correctly encodes both into the call:

```solidity
// ExtensionCalling.sol line 97
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

However, `DepositAllowlistExtension.beforeAddLiquidity` discards the first parameter (`sender`) entirely and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol line 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

`removeLiquidity` enforces `msg.sender == owner` (line 206 of `MetricOmmPool.sol`), but `addLiquidity` has no such constraint — the operator pattern is explicitly documented. An attacker `bob` calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)` where `alice` is allowlisted. The pool fires `beforeAddLiquidity(bob, alice, ...)`. The extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert. `bob` pays tokens via callback; `alice` receives LP shares. `bob` is never evaluated.

The same bypass is available without a custom callback via `MetricOmmPoolLiquidityAdder.addLiquidityWeighted(pool, alice, salt, ...)`, which calls `pool.addLiquidity(alice, ...)` with `MetricOmmPoolLiquidityAdder` as `msg.sender` — also not on the allowlist.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender` (the actual swapper), discarding `recipient` — the exact pattern that should have been applied here.

## Impact Explanation
An unprivileged actor can deposit into any pool protected by `DepositAllowlistExtension` by naming any allowlisted address as `owner`. The pool admin's configured access control is silently nullified for the entire deposit path. This is a direct admin-boundary break: an unprivileged actor bypasses a factory-configured guard, constituting broken core pool functionality. The attacker contributes real tokens and mints LP shares under an allowlisted owner's position key, which may also interfere with that owner's position accounting.

## Likelihood Explanation
No special privileges, malicious setup, or non-standard tokens are required. The `addLiquidity` interface explicitly supports `msg.sender != owner` (documented in `IMetricOmmPoolActions` NatSpec). `MetricOmmPoolLiquidityAdder.addLiquidityWeighted` provides a public, no-custom-callback entry point. Any actor who knows one allowlisted address can exploit this immediately and repeatably.

## Recommendation
Name the first parameter `sender` and gate on it, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is instead to gate on `owner` (position beneficiary), the NatSpec, mapping name (`allowedDepositor`), and contract description ("Gates `addLiquidity` by depositor address") must all be corrected, and the `addLiquidityWeightedFor` path must be separately hardened.

## Proof of Concept
1. Deploy pool with `DepositAllowlistExtension`; pool admin calls `setAllowedToDeposit(pool, alice, true)`. `bob` is not allowlisted.
2. `bob` calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)` directly (implementing `IMetricOmmModifyLiquidityCallback` to pay from his own balance), or calls `liquidityAdder.addLiquidityWeighted(pool, alice, salt, ...)` (no custom callback needed).
3. Pool fires `extension.beforeAddLiquidity(bob, alice, salt, deltas, extensionData)`.
4. Extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. Pool invokes callback; `bob` transfers tokens; `alice`'s position receives LP shares.
6. `bob` has deposited into a restricted pool he is explicitly barred from, bypassing the allowlist entirely. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
