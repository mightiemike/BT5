### Title
`l2_gas_price_wei` in `BlockInfo` derived from proposer-supplied L1 gas price ratio instead of a trusted oracle — (`crates/apollo_consensus_orchestrator/src/utils.rs`)

---

### Summary

`convert_to_sn_api_block_info` back-calculates `eth_to_fri_rate` from the proposer-supplied `l1_gas_price_fri / l1_gas_price_wei` ratio and uses it to derive `l2_gas_price_wei`. The validator independently validates each L1 gas price field only within a percentage margin, but never validates the derived ratio or the resulting `l2_gas_price_wei`. A malicious proposer can skew the ratio to its extremes within the allowed margin, producing a systematically wrong `l2_gas_price_wei` that is committed into every block's `BlockInfo` and block hash.

---

### Finding Description

`convert_to_sn_api_block_info` converts a `ProposalInit` (proposer-supplied, received over the network) into a `starknet_api::block::BlockInfo` that is handed to the blockifier for execution:

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs  lines 318-328
let proposal_init_info = PreviousProposalInitInfo::from(init);
let eth_to_fri_rate = calculate_eth_to_fri_rate(&proposal_init_info)?;

let l2_gas_price_wei = NonzeroGasPrice::new(init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?)
    ...?;
```

`calculate_eth_to_fri_rate` computes:

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs  lines 503-531
eth_to_fri_rate = l1_gas_price_fri * WEI_PER_ETH / l1_gas_price_wei
```

Both `l1_gas_price_fri` and `l1_gas_price_wei` come from the proposer's `ProposalInit`. The validator's `is_proposal_init_valid` checks each field independently against its own oracle reading using `within_margin`:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs  lines 342-353
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(l1_data_gas_price_fri_proposed, l1_data_gas_price_fri, ...)
    && within_margin(l1_gas_price_wei_proposed, l1_gas_price_wei, ...)
    && within_margin(l1_data_gas_price_wei_proposed, l1_data_gas_price_wei, ...))
```

`within_margin` enforces a symmetric band `[ref*(1-m), ref*(1+m)]` around the validator's own oracle reading. The **ratio** `l1_gas_price_fri / l1_gas_price_wei` is never validated. A malicious proposer can simultaneously push `l1_gas_price_fri` to the top of its band and `l1_gas_price_wei` to the bottom of its band (or vice versa), both passing individual validation, while the ratio deviates by up to `(1+m)/(1-m)`. For the default 10% margin this is ≈ 1.22×.

The inflated/deflated `eth_to_fri_rate` is then applied to `l2_gas_price_fri` (which is validated exactly) to produce `l2_gas_price_wei`. This derived value is placed into `BlockInfo.eth_gas_prices.l2_gas_price` and is never independently verified against the validator's own oracle.

`convert_to_sn_api_block_info` is called on the proposer-supplied `init` in both the build path and the validate path:

```rust
// crates/apollo_consensus_orchestrator/src/build_proposal.rs  line 205
block_info: convert_to_sn_api_block_info(&init)?,

// crates/apollo_consensus_orchestrator/src/validate_proposal.rs  line 466
block_info: convert_to_sn_api_block_info(init)?,
```

The resulting `BlockInfo` is passed to the batcher/blockifier for execution, and the `l2_gas_price_wei` is also committed into the block header (and thus the block hash) via `gas_prices_to_hash`:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  lines 417-443
pub fn gas_prices_to_hash(
    l1_gas_price: &GasPricePerToken,
    l1_data_gas_price: &GasPricePerToken,
    l2_gas_price: &GasPricePerToken,   // includes price_in_wei
    block_hash_version: &BlockHashVersion,
) -> Vec<Felt>
```

---

### Impact Explanation

**Wrong ETH-denominated L2 gas fee in every executed block.** The blockifier uses `BlockInfo.eth_gas_prices.l2_gas_price` to compute the ETH cost of L2 gas consumption for transactions that pay in ETH. A manipulated `l2_gas_price_wei` causes systematic over- or under-charging of those users. With a 10% margin the proposer can deflate `l2_gas_price_wei` by ≈18%, undercharging ETH-paying users and extracting the difference, or inflate it by ≈22%, overcharging them.

**Wrong value committed into the block hash.** `gas_prices_to_hash` hashes `l2_gas_price.price_in_wei` into the block hash for versions ≥ V0_13_4. A manipulated value produces a block hash that diverges from what an honest re-execution would compute, breaking re-execution and proof verification.

This matches the allowed impact scope: *"Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact"* and *"Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."*

---

### Likelihood Explanation

Any consensus participant that wins a proposer slot can trigger this. No external oracle manipulation or privileged access is required — the proposer simply sets its own `l1_gas_price_fri` and `l1_gas_price_wei` fields to the extremes of the allowed margin. The attack is deterministic, repeatable every block the malicious node proposes, and leaves no on-chain trace distinguishing it from honest price variation.

---

### Recommendation

The validator should derive `eth_to_fri_rate` from its **own** oracle reading (already available from `get_l1_prices_in_fri_and_wei_and_conversion_rate`) rather than from the proposer-supplied ratio. Concretely, `convert_to_sn_api_block_info` should accept the locally-computed `eth_to_fri_rate` as a parameter instead of re-deriving it from `init.l1_gas_price_fri / init.l1_gas_price_wei`. Alternatively, add an explicit `within_margin` check on the derived `l2_gas_price_wei` against the validator's own oracle-derived value before passing `BlockInfo` to the batcher.

---

### Proof of Concept

Let `ref_fri` and `ref_wei` be the validator's own oracle readings for L1 gas price, and `m = l1_gas_price_margin_percent / 100`.

**Proposer sets:**
- `l1_gas_price_fri = ref_fri * (1 + m)` — passes `within_margin`
- `l1_gas_price_wei = ref_wei * (1 - m)` — passes `within_margin`

**`calculate_eth_to_fri_rate` computes:**
```
eth_to_fri_rate = ref_fri*(1+m) * WEI_PER_ETH / (ref_wei*(1-m))
               = honest_rate * (1+m)/(1-m)
```

For `m = 0.10`: `eth_to_fri_rate ≈ honest_rate * 1.222`

**`l2_gas_price_wei` becomes:**
```
l2_gas_price_wei = l2_gas_price_fri * WEI_PER_ETH / eth_to_fri_rate
                 ≈ honest_l2_gas_price_wei * (1-m)/(1+m)
                 ≈ honest_l2_gas_price_wei * 0.818
```

The blockifier executes the block with `l2_gas_price_wei` deflated by ≈18%. ETH-paying users are undercharged for L2 gas. The block hash commits this wrong value. The validator accepts the proposal because each individual L1 gas price field is within margin, but the derived ratio — and thus `l2_gas_price_wei` — is never checked. fn calculate_eth_to_fri_rate(
```rust
    proposal_init_info: &PreviousProposalInitInfo,
) -> Result<u128, StarknetApiError> {
    let eth_to_fri_rate = proposal_init_info
        .l1_prices_fri
        .l1_gas_price
        .0
        .checked_mul(WEI_PER_ETH)
        .ok_or_else(|| {
            StarknetApiError::GasPriceConversionError(format!(
                "Gas price in Fri should be small enough to multiply by WEI_PER_ETH. Previous \
                 proposal init info: {:?}",
                proposal_init_info
            ))
        })?
        .checked_div(proposal_init_info.l1_prices_wei.l1_gas_price.0)
        .ok_or_else(|| {
            StarknetApiError::GasPriceConversionError(format!(
                "Gas price in Wei should be non-zero. Previous proposal init info: {:?}",
                proposal_init_info
            ))
        })?;
    if eth_to_fri_rate == 0 {
        return Err(StarknetApiError::GasPriceConversionError(format!(
            "Eth to fri rate is zero. Previous proposal init info: {:?}",
            proposal_init_info
        )));
    }
    Ok(eth_to_fri_rate)
```

```rust
/// Returns whether `proposed` is within `margin_percent` of the locally-trusted `reference`,
/// i.e. within the symmetric band `[reference*(1-m), reference*(1+m)]`.
///
/// The band is anchored to `reference` (the node's own L1 oracle read), not to the
/// proposer-supplied `proposed`: anchoring to `proposed` would let a malicious proposer scale the
/// band width with its own input and widen it in its favor.
fn within_margin(proposed: GasPrice, reference: GasPrice, margin_percent: u128) -> bool {
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if proposed.0.abs_diff(reference.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    // Saturate: `reference.0 * margin_percent` can overflow u128 on large WEI prices.
    let margin = reference.0.saturating_mul(margin_percent) / 100;
    proposed.0.abs_diff(reference.0) <= margin
}
```
