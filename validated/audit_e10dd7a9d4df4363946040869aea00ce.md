### Title
Asymmetric Fee Calculation Overcharges Taker Buy Orders — (`File: core/contracts/OffchainExchange.sol`)

### Summary
In `OffchainExchange.sol::applyFee()`, the fee for taker buy orders (negative `meteredQuote`) is computed by dividing by `keepRateX18 = (1 - feeRate)` rather than multiplying by `(1 + feeRate)`. This mirrors the M-09 pattern exactly: the denominator `(1 - feeRate)` inflates the fee above the intended `meteredQuote * feeRate`, systematically overcharging every buyer.

### Finding Description

In `applyFee`, the post-fee quote is computed as:

```solidity
int128 keepRateX18 = ONE - feeInfo.feeRate;
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(keepRateX18)
    : meteredQuote.div(keepRateX18);
orderInfo.fee = meteredQuote - newMeteredQuote;
``` [1](#0-0) 

`MathSD21x18.mul(x, y)` computes `x * y / ONE` and `MathSD21x18.div(x, y)` computes `x * ONE / y`. [2](#0-1) 

**Sell path (`meteredQuote > 0`):**
- `newMeteredQuote = meteredQuote * (1 - feeRate)`
- `fee = meteredQuote * feeRate` ✓ correct

**Buy path (`meteredQuote < 0`, e.g. `meteredQuote = -100e18`, `feeRate = 2bps = 2e14`):**
- `newMeteredQuote = meteredQuote / (1 - feeRate) = -100e18 / 0.9998 ≈ -100.020004e18`
- `fee = -100e18 − (−100.020004e18) = 0.020004e18`
- **Intended fee:** `100e18 * 2e14 / 1e18 = 0.02e18`
- **Overcharge:** `fee_actual / fee_intended = feeRate / (1 - feeRate)` — always greater than `feeRate`

The correct post-fee amount for a buyer should be `meteredQuote * (1 + feeRate)`, not `meteredQuote / (1 - feeRate)`. These are only equal when `feeRate = 0`.

The computed fee is then deducted from `quoteDelta`, making the buyer's net quote obligation larger than intended:

```solidity
orderInfo.quoteDelta =
    orderInfo.quoteDelta -
    orderInfo.fee -
    orderInfo.builderFee;
``` [3](#0-2) 

### Impact Explanation

Every taker buy order is overcharged by a factor of `feeRate / (1 - feeRate)` relative to the intended fee. The excess is collected into `collectedFees` and ultimately credited to `X_ACCOUNT`, meaning the protocol accrues more fee revenue than its stated rates imply. The overcharge scales with both trade size and fee rate:

| Fee Rate | Intended Fee on $1M | Actual Fee | Overcharge |
|---|---|---|---|
| 2 bps (default) | $200 | $200.04 | $0.04 |
| 50 bps | $5,000 | $5,025.13 | $25.13 |
| 100 bps | $10,000 | $10,101.01 | $101.01 |

The overcharge is systematic, permanent, and accumulates across every buy-side fill on every market. [1](#0-0) 

### Likelihood Explanation

Every taker buy order (negative `meteredQuote`) triggers the `div` branch. This is the most common trade type on any DEX. No special role, privilege, or configuration is required. The default taker fee rate is 2 bps, and custom tiers can be higher. [4](#0-3) 

### Recommendation

Replace the `div` branch with a multiply by `(ONE + feeInfo.feeRate)` so that the fee is computed symmetrically as a fraction of the pre-fee quote amount for both directions:

```solidity
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(ONE - feeInfo.feeRate)   // seller: receives less
    : meteredQuote.mul(ONE + feeInfo.feeRate);  // buyer: pays more
orderInfo.fee = meteredQuote - newMeteredQuote;
```

This ensures `fee = |meteredQuote| * feeRate` in both cases, matching the stated fee rates exactly.

### Proof of Concept

Concrete values using the default taker rate of 2 bps (`takerRateX18 = 200_000_000_000_000`): [5](#0-4) 

```
meteredQuote    = -1_000_000e18  ($1M buy)
feeRate         = 200_000_000_000_000  (2e14, i.e. 2bps)
keepRateX18     = 1e18 - 2e14 = 999_800_000_000_000_000

// Current (buggy):
newMeteredQuote = -1_000_000e18 * 1e18 / 999_800_000_000_000_000
               ≈ -1_000_200.040008e18
fee             = -1_000_000e18 - (-1_000_200.040008e18)
               = 200.040008e18   ($200.04)

// Correct:
newMeteredQuote = -1_000_000e18 * (1e18 + 2e14) / 1e18
               = -1_000_200e18
fee             = -1_000_000e18 - (-1_000_200e18)
               = 200e18          ($200.00)

Overcharge per $1M buy: $0.04 (at 2bps)
```

The overcharge is non-zero for every buy order and grows quadratically with fee rate, constituting a systematic accounting corruption in the protocol's fee collection.
