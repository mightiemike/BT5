### Title
Order Digest Domain Separator Uses `productId` Instead of Contract Address, Enabling Cross-Instance Order Replay — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.getDigest` constructs the EIP-712 domain separator using `address(uint160(productId))` as the `verifyingContract` field instead of `address(this)`. Because `productId` is a small `uint32` integer, the resulting "contract address" is a near-zero address (e.g., `0x0000…0001`) that is identical across every `OffchainExchange` deployment sharing the same name, version, and chain ID. Any order signature produced for one `OffchainExchange` instance is therefore valid on any other instance for the same `productId`, bypassing the cross-contract binding that EIP-712's `verifyingContract` field is designed to enforce.

---

### Finding Description

`OffchainExchange.getDigest` is the sole function that produces the digest used to validate every taker and maker order signature in `matchOrders`:

```solidity
// OffchainExchange.sol lines 291-322
function getDigest(uint32 productId, IEndpoint.Order memory order)
    public view returns (bytes32)
{
    bytes32 structHash = keccak256(abi.encode(
        keccak256(bytes(structType)),
        order.sender, order.priceX18, order.amount,
        order.expiration, order.nonce, order.appendix
    ));

    bytes32 domainSeparator = keccak256(abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),       // "Nado"
        _EIP712VersionHash(),    // "0.0.1"
        block.chainid,
        address(uint160(productId))   // ← BUG: should be address(this)
    ));

    return ECDSAUpgradeable.toTypedDataHash(domainSeparator, structHash);
}
``` [1](#0-0) 

The `verifyingContract` slot in the domain separator is filled with `address(uint160(productId))`. For `productId = 1` this resolves to `0x0000000000000000000000000000000000000001`; for `productId = 2` it resolves to `0x0000000000000000000000000000000000000002`, and so on. None of these values is the actual `OffchainExchange` contract address.

The digest is then used directly in `_validateOrder` → `_checkSignature` to authenticate every order:

```solidity
// OffchainExchange.sol lines 655-656, 663-664
digest: getDigest(callState.productId, taker.order),
...
digest: getDigest(callState.productId, maker.order),
``` [2](#0-1) 

And signature recovery happens at:

```solidity
// OffchainExchange.sol lines 332-343
function _checkSignature(...) internal view virtual returns (bool) {
    address signer = ECDSA.recover(digest, signature);
    return (signer != address(0)) &&
           (signer == address(uint160(bytes20(subaccount))) || signer == linkedSigner);
}
``` [3](#0-2) 

Because the domain separator does not bind to `address(this)`, two distinct `OffchainExchange` deployments on the same chain (same `chainid`, same name `"Nado"`, same version `"0.0.1"`) produce an identical domain separator for the same `productId`. A signature valid on instance A is cryptographically valid on instance B.

The `filledAmounts` mapping that tracks partial fills is per-instance storage:

```solidity
mapping(bytes32 => int128) public filledAmounts;
``` [4](#0-3) 

A fresh instance starts with all `filledAmounts` at zero, so a fully-filled or cancelled order from instance A can be replayed in full on instance B.

---

### Impact Explanation

An attacker who observes a user's signed order on `OffchainExchange` instance A can submit that exact signature to a second `OffchainExchange` instance B (e.g., a redeployment, a parallel market deployment, or a post-upgrade fresh proxy). The signature passes `_checkSignature` because the digest is identical. The order executes against the user's subaccount balance, draining collateral or creating an unwanted position without the user's consent. The corrupted state is the user's spot or perp balance in `SpotEngine`/`PerpEngine`, which is decremented by the replayed trade. [5](#0-4) 

---

### Likelihood Explanation

The Nado protocol uses an upgradeable proxy architecture (`ProxyManager`). A second `OffchainExchange` proxy pointing to the same or a new implementation is a realistic operational event (e.g., a v2 market, a parallel deployment for a new collateral type, or a migration). All historical order signatures remain valid on the new instance. The attacker needs only to monitor on-chain events for signed orders and submit them to the new instance — no privileged access is required. [6](#0-5) 

---

### Recommendation

Replace `address(uint160(productId))` with `address(this)` in the domain separator construction inside `getDigest`:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(this)   // bind to this specific OffchainExchange instance
    )
);
```

Alternatively, remove the manual domain separator construction entirely and use the inherited `_domainSeparatorV4()` from `EIP712Upgradeable`, which already uses `address(this)` correctly. Note that `productId` can still be included in the struct hash (not the domain separator) if per-market order isolation is desired. [7](#0-6) 

---

### Proof of Concept

1. `OffchainExchange` instance A is deployed at address `0xAAAA…` and initialized with `__EIP712_init("Nado", "0.0.1")`.
2. User signs an order for `productId = 1` on instance A. The domain separator is `keccak256(abi.encode(_TYPE_HASH, nameHash, versionHash, chainId, address(0x1)))`.
3. The sequencer matches and fills the order on instance A; `filledAmounts[digest]` on A is now non-zero.
4. Protocol deploys a second `OffchainExchange` instance B at address `0xBBBB…` with the same name and version.
5. Attacker submits the user's original signed order to instance B via the sequencer (or waits for the sequencer to include it in a batch).
6. `getDigest` on instance B produces the identical digest (same `productId`, same name/version/chainId, same `address(uint160(1))`).
7. `_checkSignature` recovers the user's address and returns `true`.
8. `filledAmounts[digest]` on B is `0`, so the full order amount is executed.
9. The user's balance on instance B is debited without their knowledge or consent. [8](#0-7)
