### Title
`op_unknown` Cost Check Applied to Pre-Multiplied Value, Allowing Returned Cost to Exceed `max_cost` — (`File: src/more_ops.rs`)

---

### Summary

In `op_unknown`, the `check_cost` guard is called on the **base cost** before the `cost_multiplier` is applied. The function then multiplies the cost by `(cost_multiplier + 1)` and returns the inflated value. This is a direct arithmetic semantic mismatch: the enforcement check operates on the wrong scale, mirroring the `calculateDustAmount` double-scaling bug where a value already in one unit is scaled again before comparison.

---

### Finding Description

`op_unknown` in `src/more_ops.rs` computes a base cost from the opcode's `cost_function` bits, then applies a separate `cost_multiplier` extracted from the opcode bytes. The enforcement sequence is:

```rust
// line 260
check_cost(cost, max_cost)?;
// line 261
cost *= cost_multiplier + 1;
// line 262-265
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
``` [1](#0-0) 

`check_cost` is defined as a simple `cost > max_cost` guard: [2](#0-1) 

The `cost_multiplier` is decoded from the opcode atom bytes (up to 4 bytes, so up to `u32::MAX`): [3](#0-2) 

The base cost for `cost_function = 0` is `1`. With `cost_multiplier = u32::MAX - 1 = 4,294,967,294`, the sequence is:

1. `check_cost(1, max_cost)` — passes for any `max_cost ≥ 1`
2. `cost *= 4,294,967,295` → `cost = 4,294,967,295`
3. `cost > u32::MAX` → false
4. Returns `Reduction(4_294_967_295, nil)`

The returned cost is `4,294,967,295` regardless of `max_cost`. The check at line 260 is performed on the **pre-multiplied** value (`1`), not on the value that is actually returned and accumulated by the caller. This is the same class of bug as `calculateDustAmount`: a quantity is checked at one scale but used at another.

The `cost_function` bits and `cost_multiplier` are both fully attacker-controlled via the opcode bytes of any unknown opcode in lenient/softfork mode: [4](#0-3) 

---

### Impact Explanation

Any caller that passes a `max_cost` smaller than `u32::MAX` (e.g., a per-coin cost budget, a softfork sub-budget, or a test harness) will have its budget enforcement silently bypassed by `op_unknown`. The function returns a `Reduction` whose cost field can be up to `4,294,967,295` even when `max_cost` is `1`. The caller accumulates this value; if the caller's own post-call check is the only remaining guard, the program has already done the work and returned a result before the overrun is detected. In contexts where `run_program` passes the remaining budget as `max_cost` and trusts the operator's internal `check_cost` as the enforcement point, the budget is not enforced at all for unknown opcodes with large multipliers.

---

### Likelihood Explanation

Unknown opcodes are reachable in lenient mode (softfork extensions). An attacker submitting a CLVM program to a Chia full node can include an unknown opcode byte sequence with the high bytes set to encode a large `cost_multiplier` and `cost_function = 0`. No special privileges are required; the opcode bytes are part of the serialized program and are fully attacker-controlled.

---

### Recommendation

Move `check_cost` to **after** the multiplier is applied, so the enforcement is done on the value that is actually returned:

```rust
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}
check_cost(cost, max_cost)?;   // now checks the post-multiplied cost
Ok(Reduction(cost as Cost, allocator.nil()))
```

This mirrors the fix in the external report: normalize all values to the same scale before comparing them.

---

### Proof of Concept

Craft an unknown opcode atom with:
- Last byte = `0x00` → `cost_function = 0`, base cost = 1
- Preceding bytes = `0xFF 0xFF 0xFF` → `cost_multiplier = 0x00FFFFFF = 16,777,215`
- Returned cost = `1 × (16,777,215 + 1) = 16,777,216`

Call `op_unknown` with `max_cost = 10`:

```
check_cost(1, 10)  → passes (1 ≤ 10)
cost = 1 × 16,777,216 = 16,777,216
returned Reduction(16_777_216, nil)  ← 1,677,721× over budget
```

The `check_cost` at line 260 passes because it sees `cost = 1`, not `cost = 16,777,216`. The caller receives a `Reduction` whose cost field is `16,777,216`, which is `1,677,721×` the `max_cost` of `10`, with no error raised inside `op_unknown`. [5](#0-4)
