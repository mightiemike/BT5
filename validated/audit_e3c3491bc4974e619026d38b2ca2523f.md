Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which resolves to `msg.sender` of the `pool.swap()` call — the router's address when a user routes through `MetricOmmSimpleRouter`. Any pool admin who allowlists the router to enable router-based swaps for permitted users simultaneously grants unrestricted swap access to every unpermitted user, completely defeating the per-user curation invariant.

## Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
```

`SwapAllowlistExtension.beforeSwap` receives this as `sender` and checks it against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37-38
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant), the router is the entity that calls `pool.swap()`:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

So `sender` arriving at the extension is `address(router)`, not the actual end-user. The allowlist lookup becomes `allowedSwapper[pool][router]`. The router does not encode the real user's identity into `extensionData` — it only stores `msg.sender` in transient storage for the payment callback (`_setNextCallbackContext`), which is never forwarded to extensions.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly ignores `sender` (first argument, unnamed) and checks `owner` (the economic actor):

```solidity
// DepositAllowlistExtension.sol L32-38
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    ...
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

This demonstrates the protocol understands the distinction between the call-chain intermediary and the economic actor — but `SwapAllowlistExtension` does not apply the same logic.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties, whitelisted market makers) is completely bypassed the moment the router is added to the allowlist. Any unpermitted user can call `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` targeting the restricted pool; the extension sees `sender = router`, which is allowlisted, and passes. The swap executes as if the user were permitted, breaking the core curation invariant and allowing unauthorized trading that can drain LP value from a pool intended to be restricted to trusted counterparties. The loss is direct and repeatable.

## Likelihood Explanation

The trigger condition — the router being allowlisted — is not exotic. A pool admin who wants to support router-based swaps for their permitted users *must* allowlist the router, because without it, even permitted users cannot trade through the router (their individual addresses are not `sender`). The admin faces a forced choice: either block all router usage or open the allowlist to everyone. There is no correct configuration that achieves both goals with the current implementation, making the vulnerable state a natural operational outcome, not a misconfiguration edge case.

## Recommendation

The extension must resolve the actual end-user identity rather than the immediate caller. Two viable approaches:

1. **Pass the original user through `extensionData`:** The router encodes `msg.sender` (the real user) into `extensionData`; the extension decodes and checks it. This requires a convention between the router and the extension.
2. **Mirror the deposit allowlist pattern:** Introduce a separate `originator` field that the pool populates with the economically relevant actor, analogous to how `beforeAddLiquidity` receives `owner` rather than `sender`.

At minimum, the NatSpec of `SwapAllowlistExtension` must warn that allowlisting the router grants access to all router users, not just individually permitted ones.

## Proof of Concept

```
Setup:
  pool  = MetricOmmPool with SwapAllowlistExtension
  admin = pool admin
  alice = KYC-approved user
  bob   = unpermitted attacker

Step 1 – Admin configures allowlist:
  admin calls swapExt.setAllowedToSwap(pool, alice, true)
  admin calls swapExt.setAllowedToSwap(pool, router, true)
    ↳ intent: alice can trade directly or via router
    ↳ actual effect: every user can trade via router

Step 2 – Bob bypasses allowlist:
  bob calls MetricOmmSimpleRouter.exactInputSingle({
      pool: pool, recipient: bob, zeroForOne: true, amountIn: X, ...
  })

Step 3 – Call chain:
  router.exactInputSingle()
    → pool.swap(msg.sender=router, ...)          // MetricOmmPool.sol L230-231
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓ passes  // SwapAllowlistExtension.sol L37
      → swap executes, bob receives output tokens

Result: bob trades on a pool he is not individually permitted to access.
