Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates position `owner` instead of actual depositor `sender`, enabling allowlist bypass via `MetricOmmPoolLiquidityAdder` — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter and instead checks `allowedDepositor[pool][owner]`, where `owner` is the position recipient supplied by the caller. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a caller-controlled `owner` argument (validated only for non-zero), any unprivileged user can set `owner` to any allowlisted address, pass the allowlist check, pay tokens themselves, and mint LP shares to that address — fully circumventing the pool admin's deposit restriction.

## Finding Description

**Root cause — wrong identity checked:**

`IMetricOmmExtensions.beforeAddLiquidity` receives two identity parameters: `sender` (the direct pool caller, i.e. `msg.sender` inside `MetricOmmPool.addLiquidity`) and `owner` (the position recipient). The extension ignores `sender` entirely and checks `owner`:

```solidity
// DepositAllowlistExtension.sol line 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

The pool passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner`:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

**Exploit path via `MetricOmmPoolLiquidityAdder`:**

`addLiquidityExactShares(address pool, address owner, ...)` validates `owner` only for non-zero:

```solidity
// MetricOmmPoolLiquidityAdder.sol line 65-67
_validateOwner(owner);   // only checks owner != address(0)
...
return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
```

The callback stores `msg.sender` (the attacker) as `payer` and pulls tokens from the attacker, while the pool mints the position to the attacker-supplied `owner`.

**Full exploit flow:**

1. Pool is deployed with `DepositAllowlistExtension`; only `alice` is allowlisted (`allowedDepositor[pool][alice] = true`).
2. `bob` (not allowlisted) calls `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, data)`.
3. The adder calls `pool.addLiquidity(alice, salt, deltas, ...)`.
4. The pool calls `_beforeAddLiquidity(LiquidityAdder, alice, ...)`.
5. Extension checks `allowedDepositor[pool][alice]` → `true` → hook passes.
6. Pool mints LP shares to position `(alice, salt)`; callback pulls tokens from `bob`.
7. `bob` has successfully deposited into a restricted pool without being allowlisted.

**Why existing guards fail:**

- `_validateOwner` only rejects `address(0)`, not arbitrary allowlisted addresses.
- The `onlyPool` modifier on the base class is not present on the override, but `msg.sender` is used as the pool key, so the allowlist lookup is correctly scoped to the calling pool — the bug is purely the choice of `owner` over `sender`.
- `removeLiquidity` enforces `msg.sender == owner`, so `bob` cannot reclaim the tokens; the position is permanently minted to `alice`.

## Impact Explanation
This is an admin-boundary break: the pool admin's deposit allowlist — the sole mechanism restricting who may add liquidity — is bypassed by any unprivileged caller who knows any allowlisted address. Unauthorized liquidity can be injected into the pool, violating the invariant that only approved depositors may participate. Additionally, allowlisted users can have LP positions force-minted to them without consent, creating unwanted token exposure and potential tax/accounting consequences. Severity: **Medium** (admin-boundary bypass with no direct financial gain to the attacker, but clear circumvention of a core access-control invariant).

## Likelihood Explanation
The attack requires only: (1) knowledge of one allowlisted address (publicly readable from `allowedDepositor`), (2) willingness to spend tokens (attacker pays), and (3) a call to the publicly deployed `MetricOmmPoolLiquidityAdder`. No privileged role, flash loan, or complex setup is needed. The condition is trivially reachable on any pool using `DepositAllowlistExtension` without `allowAllDepositors = true`.

## Recommendation
Check `sender` (the actual pool caller) instead of `owner` (the position recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Note: if routing through `MetricOmmPoolLiquidityAdder` is intended to be allowlisted, the adder address itself must be added to the allowlist, or the adder must forward the original `msg.sender` through `extensionData` and the extension must decode it (with appropriate trust assumptions).

## Proof of Concept

```solidity
// Foundry test sketch
function test_allowlistBypassViaLiquidityAdder() public {
    // Setup: pool with DepositAllowlistExtension, only alice allowlisted
    vm.prank(poolAdmin);
    depositAllowlist.setAllowedToDeposit(address(pool), alice, true);

    // bob is NOT allowlisted
    assertFalse(depositAllowlist.isAllowedToDeposit(address(pool), bob));

    // bob approves the adder and calls with owner = alice
    vm.startPrank(bob);
    token0.approve(address(adder), type(uint256).max);
    token1.approve(address(adder), type(uint256).max);

    // This should revert if the allowlist gates the actual depositor,
    // but it succeeds because the extension checks owner (alice) not sender (adder/bob)
    adder.addLiquidityExactShares(address(pool), alice, 0, deltas, max0, max1, "");
    vm.stopPrank();

    // bob's tokens were spent; alice has an LP position she did not request
    assertGt(pool.positionShares(alice, 0, binIdx), 0);
}
```
