Audit Report

## Title
`SwapAllowlistExtension` checks the immediate `pool.swap()` caller instead of the end-user, allowing any unprivileged caller to bypass the swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` as the address that called `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` of `pool.swap()`, so the extension checks the router's address rather than the actual end-user. A pool admin who allowlists the router to enable router-based access for their intended users simultaneously grants every unprivileged user the ability to bypass the allowlist by calling the same public router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
  msg.sender,   // ← always the immediate caller of pool.swap()
  recipient, ...
);
```

`ExtensionCalling._beforeSwap` forwards that value verbatim to the extension via `abi.encodeCall`. `SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no user-identity forwarding — the router's own address becomes `sender` in the extension:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The bypass path is:
1. Pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists `trustedMM`.
2. Admin also calls `setAllowedToSwap(pool, address(router), true)` so `trustedMM` can use the standard router.
3. Any unprivileged `attacker` calls `router.exactInputSingle(...)` targeting the same pool.
4. The extension sees `sender = router`, finds `allowedSwapper[pool][router] = true`, and passes — the attacker swaps successfully.

The existing integration test `test_allowedSwapSucceeds` allowlists `address(callers[0])` (a `TestCaller` contract that is itself the direct caller of `pool.swap()`), confirming the check is keyed to the direct caller, not the end-user EOA. No test exercises the router-against-allowlisted-pool path.

## Impact Explanation
This is an admin-boundary break. A pool configured with `SwapAllowlistExtension` to restrict swaps to specific counterparties (KYC'd addresses, approved market makers, protocol-internal actors) can be fully bypassed by any user routing through `MetricOmmSimpleRouter`. Once the router is allowlisted, the curation boundary collapses to "anyone who calls the router." LP positions sized under the assumption that only vetted counterparties trade are exposed to adverse selection from unrestricted public flow, causing direct loss of LP value and violating the pool's intended access policy.

## Likelihood Explanation
The scenario requires only one intuitive admin action: allowlisting the router so that allowlisted users can access the pool via the standard periphery. Nothing in the interface, NatSpec, or documentation warns that this collapses the allowlist for all users. No privileged attacker capability is needed — any EOA can call the public router. The bypass is repeatable every block once the router is allowlisted.

## Recommendation
The extension must resolve the actual end-user identity rather than the immediate `pool.swap()` caller. Two viable approaches:

1. **Pass the real user in `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Requires a trusted encoding convention between router and extension.
2. **Router-aware identity resolution**: The extension inspects whether `sender` is a known router and, if so, reads the real user from a verified field in `extensionData`.
3. **Documentation-only mitigation (minimal)**: Document that the router must never be allowlisted and that allowlisted users must call `pool.swap()` directly. This breaks router UX for curated pools.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension; only trustedMM is allowlisted.
swapExt.setAllowedToSwap(pool, trustedMM, true);
// Admin also allowlists the router so trustedMM can use it:
swapExt.setAllowedToSwap(pool, address(router), true);

// Attack: attacker (not allowlisted) routes through the public router.
vm.prank(attacker);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             address(pool),
        tokenIn:          address(token0),
        tokenOut:         address(token1),
        zeroForOne:       true,
        amountIn:         1_000e18,
        amountOutMinimum: 0,
        recipient:        attacker,
        deadline:         block.timestamp + 1,
        priceLimitX64:    0,
        extensionData:    ""
    })
);
// Extension sees sender = router → allowedSwapper[pool][router] = true → passes.
// Attacker swaps successfully despite not being on the allowlist.
```
