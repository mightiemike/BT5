### Title
Freed Storage Deposits Accumulate Permanently in Contract After GC — (`contract/src/lib.rs`)

### Summary

Every call to `submit_blocks` pays a storage deposit for new block headers. The `run_mainchain_gc` function, called both internally from `submit_blocks` and directly as a public entry point, removes old headers and frees their storage staking. However, the freed NEAR tokens from those previously-paid deposits are never returned to anyone — they accumulate in the contract's balance indefinitely with no withdrawal mechanism. This is a direct structural analog to M-16: funds are released from staking but remain permanently locked inside the module.

### Finding Description

In `submit_blocks`, the storage measurement is taken **after** GC runs: [1](#0-0) 

The sequence is:
1. `initial_storage = env::storage_usage()` — snapshot before any changes
2. New headers are inserted (storage grows)
3. `run_mainchain_gc(num_of_headers)` removes old headers (storage shrinks)
4. `diff_storage_usage = env::storage_usage().saturating_sub(initial_storage)` — net delta

When GC removes headers that were paid for in **previous** calls to `submit_blocks`, the net storage delta for the current call is reduced (or zeroed via `saturating_sub`). The current caller's required deposit is therefore reduced or eliminated — but the NEAR tokens that previous callers deposited for those now-removed headers remain in the contract's balance. There is no accounting of which caller paid for which headers, and no withdrawal function exists anywhere in the contract.

`run_mainchain_gc` is also a standalone public entry point: [2](#0-1) 

When called directly, it frees storage (releasing NEAR from storage staking into the contract's available balance) but issues no refund to anyone. The freed tokens are simply absorbed into the contract's balance with no path out.

The contract has no `withdraw`, `claim_storage_refund`, or equivalent function anywhere in `lib.rs`, `utils.rs`, or any chain-specific module.

### Impact Explanation

NEAR tokens paid as storage deposits by relayers for block headers that are subsequently garbage-collected are permanently locked in the contract. The accumulation is proportional to chain activity:

- Tests use `STORAGE_DEPOSIT_PER_BLOCK = 500 milliNEAR` per header
- At Bitcoin's ~144 blocks/day and the recommended `gc_threshold = 52704`, GC begins removing ~144 headers/day after the first year
- That is ~72 NEAR/day, ~26,000 NEAR/year, permanently locked

The contract's `Upgradable` plugin could theoretically allow a DAO upgrade to add a withdrawal function, but in the current deployed state there is no retrieval path. The locked funds represent a direct, growing financial loss to the protocol.

### Likelihood Explanation

This requires no attacker. It is a structural logic flaw that activates automatically once the chain has been running long enough for GC to begin (i.e., once `mainchain_size > gc_threshold`). Any relayer submitting blocks triggers it. The condition is inevitable in any production deployment.

### Recommendation

Measure storage **before** GC runs, or track freed storage separately and issue a refund for it:

```rust
let initial_storage = env::storage_usage();

for header in headers {
    self.submit_block_header(header, self.skip_pow_verification);
}

// Measure storage BEFORE GC so freed storage is not silently absorbed
let storage_after_insert = env::storage_usage();
let diff_storage_usage = storage_after_insert.saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

// Now run GC; freed storage becomes a refundable surplus
self.run_mainchain_gc(num_of_headers);
let freed_storage = storage_after_insert.saturating_sub(env::storage_usage());
let freed_deposit = env::storage_byte_cost().saturating_mul(freed_storage.into());

let refund = amount.saturating_sub(required_deposit).saturating_add(freed_deposit);
```

Alternatively, add a DAO-callable `withdraw_surplus` function that transfers `contract_balance - storage_usage * storage_byte_cost` to a designated treasury address.

### Proof of Concept

1. Deploy the contract with `gc_threshold = 3`.
2. Call `submit_blocks` with 3 headers, attaching sufficient deposit. Contract stores 3 headers; deposit is consumed.
3. Call `submit_blocks` with 3 more headers. GC removes the first 3 headers inside the call. `diff_storage_usage` is 0 (net zero). Caller's entire deposit is refunded.
4. Observe: the contract's balance now contains the deposit from step 2 that covered the removed headers. No function exists to retrieve it.
5. Repeat steps 2–4 indefinitely. The contract's surplus balance grows monotonically with no withdrawal path. [3](#0-2) [4](#0-3)

```rust
    #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
```
