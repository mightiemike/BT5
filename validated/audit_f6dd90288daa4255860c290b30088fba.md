### Title
`calculate_next_base_gas_price` Uses `latest_constants()` Denominator Regardless of Block's Starknet Version, Producing Wrong `next_l2_gas_price` - (`crates/apollo_consensus_orchestrator/src/fee_market/mod.rs`)

---

### Summary

`calculate_next_base_gas_price` and `calculate_next_l2_gas_price_for_fin` unconditionally call `VersionedConstants::latest_constants()` to obtain `gas_price_max_change_denominator` and `gas_target`. These two values form the denominator of the EIP-1559 price-adjustment ratio. When the orchestrator's versioned constants change across Starknet versions (V0_14_0 → V0_14_4), any block whose `starknet_version` is not the latest will have its `next_l2_gas_price` computed with the wrong denominator, committing a wrong value to the block header and therefore charging every transaction in the following block the wrong L2 gas price.

---

### Finding Description

`calculate_next_l2_gas_price_for_fin` reads `gas_target` from `VersionedConstants::latest_constants()`:

```rust
// fee_market/mod.rs:70
let gas_target = VersionedConstants::latest_constants().gas_target;
```

It then calls `calculate_next_base_gas_price`, which again calls `VersionedConstants::latest_constants()` to obtain `gas_price_max_change_denominator` and `max_block_size`:

```rust
// fee_market/mod.rs:92
let versioned_constants = VersionedConstants::latest_constants();
// ...
let denominator =
    gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
let price_change = (price_u256 * gas_delta) / denominator;
```

Neither function accepts a `starknet_version` or a `VersionedConstants` argument. The orchestrator defines five distinct constant sets (V0_14_0 through V0_14_4), each in its own JSON file. When `gas_price_max_change_denominator` or `gas_target` differs between two consecutive versions, a block whose `ProposalInit.starknet_version` is not the latest will have its `next_l2_gas_price` computed with the wrong denominator.

The same pattern appears in the fee-proposal window management:

```rust
// sequencer_consensus_context.rs:313
let window_size = VersionedConstants::latest_constants().fee_proposal_window_size;
```

```rust
// sequencer_consensus_context.rs:785
VersionedConstants::latest_constants().fee_proposal_window_size,
```

```rust
// sequencer_consensus_context.rs:490
VersionedConstants::latest_constants().fee_proposal_margin_ppt,
```

```rust
// validate_proposal.rs:403
VersionedConstants::latest_constants().fee_proposal_margin_ppt,
```

If `fee_proposal_window_size` changes between versions, `compute_fee_actual` computes the median over the wrong number of blocks, producing a wrong `fee_actual` floor. If `fee_proposal_margin_ppt` changes, the proposer and validator may compute different bounds for the same `fee_actual`, causing valid proposals to be rejected or out-of-bounds proposals to be accepted.

The analog to the external report is exact: in PirexRewards, `globalRewards` is a time-weighted supply integral that does not account for a dynamic emission rate, so the ratio `userRewards / globalRewards` distributes the wrong share. Here, `gas_price_max_change_denominator` and `gas_target` are taken from the latest version rather than the block's actual version, so the ratio `price * gas_delta / (gas_target * denominator)` produces the wrong price adjustment.

---

### Impact Explanation

`next_l2_gas_price` is stored in `StorageBlockHeader` and becomes the L2 gas price for every transaction in the next block. A wrong value means every transaction in that block is charged the wrong L2 gas fee — an incorrect fee/gas/resource accounting effect with direct economic impact. This matches the Critical impact category: *"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

For the `fee_proposal_margin_ppt` divergence: a validator running the new binary rejects a proposer's `fee_proposal` that was computed correctly under the old constants, stalling block production. This matches the High impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The vulnerability is latent and activates at every Starknet version upgrade where `gas_price_max_change_denominator`, `gas_target`, `fee_proposal_window_size`, or `fee_proposal_margin_ppt` changes. The orchestrator already defines five versioned constant files (V0_14_0–V0_14_4), making future changes to these fields routine. No privileged access is required; the trigger is the normal version-upgrade deployment sequence.

---

### Recommendation

Pass the block's `starknet_version` (or the resolved `VersionedConstants` for that version) into `calculate_next_l2_gas_price_for_fin` and `calculate_next_base_gas_price` instead of calling `VersionedConstants::latest_constants()` inside those functions. Apply the same fix to `prune_fee_proposals_window`, `compute_fee_actual`, `compute_proposer_fee_proposal`, and `is_proposal_init_valid`: resolve the constants from `init_proposed.starknet_version` (already present in `ProposalInit`) rather than from `latest_constants()`.

---

### Proof of Concept

Assume V0_14_3 has `gas_price_max_change_denominator = 32` and V0_14_4 has `gas_price_max_change_denominator = 48` (the confirmed V0_14_4 value).

A block is finalized with `starknet_version = V0_14_3`, `current_l2_gas_price = 30_000_000_000`, `gas_used = 3/4 * gas_target` (high congestion).

**Correct computation (V0_14_3 constants, denominator = 32):**
```
gas_delta = gas_used - gas_target = gas_target / 4
price_change = price * (gas_target/4) / (gas_target * 32) = price / 128
next_price = 30_000_000_000 + 234_375_000 = 30_234_375_000
```

**Actual computation (latest = V0_14_4 constants, denominator = 48):**
```
price_change = price * (gas_target/4) / (gas_target * 48) = price / 192
next_price = 30_000_000_000 + 156_250_000 = 30_156_250_000
```

The committed `next_l2_gas_price` is `30_156_250_000` instead of `30_234_375_000` — a ~78 million FRI undercharge per gas unit applied to every transaction in the next block. The root cause is identical to the external report: a ratio denominator that does not account for the dynamic (version-dependent) rate.
```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;
```
