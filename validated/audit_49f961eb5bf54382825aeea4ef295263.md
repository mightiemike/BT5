Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and gates on `owner` instead. Because `owner` is a free caller-supplied parameter with no binding to `msg.sender`, any unprivileged address can pass the allowlist check by nominating an already-allowlisted address as the position owner. This breaks the pool curation invariant the extension is designed to enforce.

## Finding Description

The extension declares `sender` as an unnamed, ignored parameter and checks `owner` instead:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The pool passes `msg.sender` (the actual caller) as `sender` and the caller-supplied `owner` as `owner`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` correctly forwards both arguments to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
_callExtensionsInOrder(
    BEFORE_ADD_LIQUIDITY_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

The extension receives the real caller as its first argument but ignores it. The only guard on `owner` in the periphery rejects only `address(0)`:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L247-249
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
```

**Exploit path:** An attacker calls `pool.addLiquidity(owner=allowlisted_address, ...)` directly. The pool calls `_beforeAddLiquidity(sender=attacker, owner=allowlisted_address, ...)`. The extension checks `allowedDepositor[pool][allowlisted_address]`, which is `true`. The check passes. LP shares are minted to `allowlisted_address`; tokens are pulled from the attacker via the callback. The same path is reachable through `addLiquidityExactShares(pool, owner=allowlisted_address, ...)` and `addLiquidityWeighted(pool, owner=allowlisted_address, ...)` in the periphery adder.

The extension's own NatSpec states it "Gates `addLiquidity` by depositor address," but the depositor is `sender`, not `owner`.

## Impact Explanation

The pool admin deploys a curated pool with `DepositAllowlistExtension` to restrict who may provide liquidity. The invariant is broken: any address can call `pool.addLiquidity(owner=allowlisted_address, ...)` directly, or route through `addLiquidityExactShares`/`addLiquidityWeighted` with `owner=allowlisted_address`. The extension passes because `allowedDepositor[pool][allowlisted_address]` is `true`. This constitutes an admin-boundary break — the pool admin's access control is bypassed by an unprivileged path with no trusted-role assumption required. Secondary impact: an attacker can force LP positions onto allowlisted addresses without their consent (griefing), at the attacker's own token cost.

## Likelihood Explanation

The bypass requires only a public call to `pool.addLiquidity` or the periphery adder with a known allowlisted address as `owner`. No privileged role, oracle manipulation, or special token behavior is needed. The `allowedDepositor` mapping is public, so any observer can identify valid allowlisted addresses and execute the bypass immediately and repeatedly.

## Recommendation

Replace the `owner` check with a `sender` check in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns the check with the stated invariant ("gates by depositor address") and with the `sender` value the pool already passes as the first argument.

## Proof of Concept

```solidity
function test_depositAllowlistBypass() public {
    // Setup: pool with DepositAllowlistExtension; only `allowlisted` is allowed
    address allowlisted = makeAddr("allowlisted");
    depositExtension.setAllowedToDeposit(address(pool), allowlisted, true);

    // Attacker is NOT allowlisted
    address attacker = makeAddr("attacker");
    _fundAndApprove(attacker, address(pool));

    // Attacker calls addLiquidity with owner=allowlisted
    vm.prank(attacker);
    // Does NOT revert — extension checks owner (allowlisted), not sender (attacker)
    pool.addLiquidity(allowlisted, salt, deltas, callbackData, extensionData);

    // LP shares minted to allowlisted, tokens pulled from attacker
    assertGt(positionShares(pool, allowlisted, salt, bin), 0);
}
```

Call chain: `attacker → pool.addLiquidity(owner=allowlisted)` → `_beforeAddLiquidity(sender=attacker, owner=allowlisted)` → `extension.beforeAddLiquidity(_, owner=allowlisted)` → `allowedDepositor[pool][allowlisted] == true` → passes → shares minted to `allowlisted`, tokens pulled from `attacker`.
