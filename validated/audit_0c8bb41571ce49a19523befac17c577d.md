### Title
`verify_transaction_inclusion` Panics on GC-Evicted Blocks, Enabling Unprivileged Front-Running via Public `run_mainchain_gc` to Break On-Chain Consumer Verification Flows — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` both call `env::panic_str` (hard abort) when the target block has been evicted by garbage collection, rather than returning `false`. Because `run_mainchain_gc` carries no access-control restriction and is callable by any unprivileged NEAR account, an attacker who observes a pending cross-contract call to `verify_transaction_inclusion` can front-run it by invoking `run_mainchain_gc` with a large `batch_size`, evicting the target block and turning the consumer contract's expected `bool` result into an unrecoverable panic. This is the direct analog of the `_updateSponsor` front-run: the caller supplies a "slippage" guard (`confirmations`) that protects against one dimension of state change (chain length) but provides zero protection against the orthogonal state change (GC eviction of the very block being verified).

---

### Finding Description

**Root cause — panic instead of `false` on missing block**

`verify_transaction_inclusion` performs two lookups that abort on failure:

```rust
// lib.rs:299-301
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
``` [1](#0-0) 

```rust
// lib.rs:311-313
let header = self
    .headers_pool
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));
``` [2](#0-1) 

`verify_transaction_inclusion_v2` has the same hard abort on the first pool lookup:

```rust
// lib.rs:353-356
let header = self
    .headers_pool
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));
``` [3](#0-2) 

**Root cause — `run_mainchain_gc` is unrestricted**

`run_mainchain_gc` carries only `#[pause]` (disabled when the contract is paused) and no `#[trusted_relayer]` guard. Any NEAR account can call it at any time while the contract is live:

```rust
// lib.rs:376-377
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [4](#0-3) 

`remove_block_header` deletes the block from both `mainchain_header_to_height` and `headers_pool`, the exact two maps that `verify_transaction_inclusion` panics on when absent:

```rust
// lib.rs:659-662
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [5](#0-4) 

**The missing slippage guard**

The only caller-supplied protection is:

```rust
// lib.rs:289-292
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [6](#0-5) 

This check ensures `confirmations ≤ gc_threshold` but says nothing about whether the target block is still present in the pool. A block can have far more than `confirmations` confirmations and still be GC-evicted if the chain tip has advanced more than `gc_threshold` blocks beyond it. There is no caller-supplied guard analogous to a `max_tip_height` or `expected_tip_hash` that would let the consumer abort cleanly if the chain state has shifted.

---

### Impact Explanation

A consumer contract (bridge, atomic-swap, escrow) that issues a cross-contract call to `verify_transaction_inclusion` expecting a `bool` return will instead receive a failed promise when the function panics. NEAR cross-contract callbacks that do not explicitly handle promise failure will silently skip their success branch. Depending on the consumer's design, this can:

- Leave bridged or escrowed funds permanently locked (the release condition never fires).
- Allow a double-spend window if the consumer treats a failed callback as a no-op and retries with a different proof.
- Corrupt multi-step protocol state that assumed the verification would either succeed or return `false`, not abort.

The corrupted invariant is: *a consumer that supplies a valid Merkle proof for a block with at least `confirmations` confirmations must receive a deterministic `bool`, never a panic.* The GC path breaks this invariant.

---

### Likelihood Explanation

- **Trigger condition:** The stored chain must exceed `gc_threshold` so that at least one block is eligible for eviction. For Dogecoin (≈1 min/block, `gc_threshold = 52704`) this occurs after ≈36 days of relay operation; for Litecoin (≈2.5 min/block) after ≈91 days; for Bitcoin after ≈1 year.
- **Attacker cost:** Zero PoW work. The attacker only needs to submit a NEAR transaction calling `run_mainchain_gc(large_batch_size)` before the consumer's cross-contract call lands.
- **Observability:** NEAR transaction receipts are public; a consumer contract's pending cross-contract call is visible in the mempool, making the ordering attack straightforward.
- **Realistic scenario:** A bridge or atomic-swap contract verifying a Dogecoin transaction from 5–6 weeks ago (still within a reasonable settlement window) is a plausible production use case that falls squarely in the vulnerable range.

---

### Recommendation

1. **Return `false` instead of panicking.** Replace both `unwrap_or_else(|| env::panic_str(...))` calls in `verify_transaction_inclusion` (and the equivalent in `verify_transaction_inclusion_v2`) with `unwrap_or(return false)` or an explicit `if let Some(...) else { return false; }` pattern. This makes the function's contract (`bool`) hold unconditionally.

2. **Add a slippage guard parameter.** Introduce an optional `max_tip_height: Option<u64>` (or `expected_tip_hash: Option<H256>`) argument. If supplied, the function returns `false` immediately when the current tip exceeds the caller's expectation, giving consumer contracts the same protection that `minSPercent_` was intended to give in the original report.

3. **Restrict `run_mainchain_gc`.** Consider adding `#[trusted_relayer]` or a dedicated role check so that arbitrary accounts cannot trigger GC on demand.

---

### Proof of Concept

```
Setup:
  gc_threshold = 100
  Relayer submits blocks 0..200 (chain length = 201 > gc_threshold)
  → GC is now able to evict blocks 0..100

Consumer contract action (pending in NEAR mempool):
  verify_transaction_inclusion({
      tx_block_blockhash: HASH_OF_BLOCK_50,   // 150 confirmations — well above any threshold
      confirmations: 6,
      ...
  })

Attacker observes the pending receipt and submits first:
  run_mainchain_gc(batch_size = 60)
  → removes blocks 0..59 from mainchain_header_to_height and headers_pool
  → HASH_OF_BLOCK_50 is now absent from both maps

Consumer's verify_transaction_inclusion executes next:
  Line 299: mainchain_header_to_height.get(&HASH_OF_BLOCK_50) → None
  → env::panic_str("block does not belong to the current main chain")
  → consumer callback receives PromiseResult::Failed
  → consumer's fund-release branch never executes
  → funds remain locked
```
