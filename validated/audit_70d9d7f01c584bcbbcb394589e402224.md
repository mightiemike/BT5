Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates Caller-Supplied `owner` Instead of Actual Caller `sender`, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter and checks the caller-supplied `owner` address against the allowlist. Because `MetricOmmPool.addLiquidity` accepts `owner` as a free caller-controlled argument with no restriction (unlike `removeLiquidity`, which enforces `msg.sender == owner`), any unprivileged address can bypass the deposit guard by naming an already-allowlisted address as `owner`. `SwapAllowlistExtension` correctly gates on `sender`, confirming this is an implementation error.

## Finding Description
`DepositAllowlistExtension.beforeAddLiquidity` receives `(address sender, address owner, …)` but leaves `sender` unnamed and checks `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-controlled `owner` argument as `owner` at line 191, with no restriction on what `owner` may be:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

Contrast with `removeLiquidity`, which enforces `msg.sender == owner` before proceeding (line 206), a guard that is entirely absent from `addLiquidity`. Because `owner` is unconstrained in `addLiquidity`, any caller can set it to any allowlisted address and the extension check passes. `SwapAllowlistExtension.beforeSwap` correctly gates on `sender`, not the recipient:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

The wrong value is `allowedDepositor[msg.sender][owner]` — it should be `allowedDepositor[msg.sender][sender]`.

## Impact Explanation
The deposit allowlist is an admin-configured access-control boundary intended to restrict which addresses may provide liquidity to a pool. With this bug, any unprivileged address can deposit into any allowlisted owner's position by supplying that owner's address as the `owner` argument. This constitutes a broken core pool access-control invariant: pools deployed specifically to restrict participation (e.g., permissioned institutional pools) are open to any depositor who can observe an allowlisted address on-chain. An attacker can force liquidity into an allowlisted owner's position at the attacker's own token cost, altering the owner's share composition and potentially affecting expected withdrawal amounts. This is a direct breach of the admin-configured access boundary with protocol-level impact.

## Likelihood Explanation
Allowlisted depositor addresses are observable on-chain via `AllowedToDepositSet` events or direct `allowedDepositor` storage reads. Any actor can trivially construct the bypass call. The only cost to the attacker is the tokens they deposit (which they control and receive shares for). Likelihood is high whenever a pool is deployed with `DepositAllowlistExtension` and `allowAllDepositors` is false — the exact configuration for which the extension exists.

## Recommendation
Change the check to validate `sender` (the actual caller), consistent with `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate on the position owner rather than the caller, the mapping name (`allowedDepositor`), event name (`AllowedToDepositSet`), NatSpec, and the inconsistency with `SwapAllowlistExtension` must all be explicitly documented and justified.

## Proof of Concept
1. Pool is deployed with `DepositAllowlistExtension`; `allowAllDepositors[pool] = false`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` — only `alice` is permitted.
3. `bob` (not on allowlist) calls:
   ```solidity
   pool.addLiquidity(
       owner = alice,   // allowlisted address
       salt  = 0,
       deltas = ...,
       callbackData = ...,
       extensionData = ""
   );
   ```
4. Pool calls `_beforeAddLiquidity(msg.sender=bob, owner=alice, …)` → extension receives `(sender=bob, owner=alice, …)`.
5. Extension discards `sender=bob` and checks `allowedDepositor[pool][alice]` → `true` → no revert.
6. `bob`'s tokens are deposited into `alice`'s position; the allowlist is fully bypassed.

A Foundry test can confirm this by deploying the extension, setting only `alice` as allowed, then calling `addLiquidity` from `bob` with `owner=alice` and asserting no revert occurs.
