The code confirms the vulnerability path. Let me verify the exact cap enforcement comparison between `setPoolAdminFees` and `setPoolBinAdditionalFees`.

### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` on the pool admin's global spread fee. `setPoolBinAdditionalFees` — also callable by the pool admin — passes `addFeeBuyE6`/`addFeeSellE6` directly to the pool with no cap check, allowing the pool admin to impose per-bin fees up to `uint16.max` (65 535 = 6.5535 % in E6 terms) regardless of the factory-configured cap.

---

### Finding Description

`setPoolAdminFees` explicitly guards against the pool admin exceeding the factory cap: [1](#0-0) 

`setPoolBinAdditionalFees`, also gated by `onlyPoolAdmin`, contains no equivalent guard: [2](#0-1) 

`setBinAdditionalFees` on the pool itself also performs no cap validation — it only checks the bin index range: [3](#0-2) 

The stored `addFeeBuyE6` / `addFeeSellE6` values are then added directly to `baseFeeX64` in every swap path: [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

The same uncapped values are used in `getSellAndBuyPrices` and the periphery data provider: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

The factory owner sets `maxAdminSpreadFeeE6` to bound the maximum fee the pool admin may impose on traders. That bound is enforced for the global spread fee path but is entirely absent from the per-bin fee path. A pool admin can therefore set `addFeeBuyE6 = addFeeSellE6 = 65535` on any bin, adding up to 6.5535 % on top of the oracle-derived base spread for every swap through that bin. Traders pay more than the protocol-configured maximum — a direct loss of user principal. This is an admin-boundary break: the pool admin exceeds the factory-configured cap.

---

### Likelihood Explanation

The pool admin is a per-pool role (`poolAdmin[pool]`) that is set at pool creation and can be transferred. Any pool admin — including one who is malicious or compromised — can call `setPoolBinAdditionalFees` at any time with no timelock. The call requires no special pool state and succeeds immediately.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (uint256(addFeeBuyE6)  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (uint256(addFeeSellE6) > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap inside `setBinAdditionalFees` on the pool by reading the factory's `maxAdminSpreadFeeE6` at call time.
