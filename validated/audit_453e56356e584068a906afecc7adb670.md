### Title
`confirmations = 0` Bypasses Finality Check in Transaction Inclusion Verification — (`contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` (and `verify_transaction_inclusion_v2`) functions accept a caller-supplied `confirmations: u64` field with no lower-bound check. A caller can pass `confirmations = 0`, causing the confirmation depth check to be trivially satisfied for any block in the mainchain, including the chain tip itself. This is a direct analog to the M-01 pattern: a value is checked against an upper bound but has no enforced minimum, allowing a trivially small value to defeat the security purpose of the check.

---

### Finding Description

`ProofArgs` and `ProofArgsV2` both expose a `confirmations` field that is fully caller-controlled:

```rust
// btc-types/src/contract_args.rs
pub struct ProofArgs {
    pub tx_id: H256,
    pub tx_block_blockhash: H256,
    pub tx_index: u64,
    pub merkle_proof: Vec<H256>,
    pub confirmations: u64,   // <-- caller-controlled, no minimum enforced
}
```

Inside `verify_transaction_inclusion`, the only bounds check on `confirmations` is an upper-bound guard:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

The actual finality check is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

Because `confirmations` is `u64`, passing `0` makes the right-hand side `0`, and the left-hand side (a `u64`) is always `>= 0`. The check **always passes** regardless of how recently the block was included. There is no `require!(args.confirmations > 0, ...)` or `require!(args.confirmations >= MIN_CONFIRMATIONS, ...)` guard anywhere in the function.

The same flaw is inherited by `verify_transaction_inclusion_v2`, which delegates to `verify_transaction_inclusion` via `args.into()` after the coinbase proof check.

---

### Impact Explanation

Any unprivileged NEAR caller invoking `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with `confirmations = 0` receives a `true` result for any transaction whose block is currently in the mainchain — including the chain tip, which has zero depth and is maximally vulnerable to reorganization. Recipient contracts that consume this boolean result and gate asset releases or state transitions on it are exposed to double-spend attacks: an attacker submits a block, immediately calls `verify_transaction_inclusion` with `confirmations = 0`, and the contract returns `true` before the block has any finality depth.

---

### Likelihood Explanation

The entry path requires no privilege: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, `#[pause]`-gated (but unpaused in normal operation), and accept borsh-serialized arguments directly from any NEAR account. Constructing a `ProofArgs` with `confirmations = 0` is trivial. Any protocol or contract that relies on this function for finality assurance is immediately exploitable by any caller.

---

### Recommendation

Add an explicit minimum-confirmations guard at the top of `verify_transaction_inclusion`:

```rust
require!(args.confirmations > 0, "Confirmations must be at least 1");
```

Ideally, enforce a protocol-level minimum (e.g., 6 for Bitcoin mainnet) and expose it as a contract constant or configurable parameter, rather than leaving the security threshold entirely to the caller.

---

### Proof of Concept

1. Deploy the contract with any valid genesis and `skip_pow_verification = false`.
2. Submit one block header via `submit_blocks`.
3. Call `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = the hash of the just-submitted block (chain tip, 0 depth)
   - `confirmations = 0`
   - A valid Merkle proof for a transaction in that block
4. The function returns `true` despite the block having zero confirmation depth.
5. Any downstream contract gating on this result will accept the transaction as final while it is still trivially reorganizable.
