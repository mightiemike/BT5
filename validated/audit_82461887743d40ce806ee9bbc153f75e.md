### Title
`SwapAllowlistExtension` checks the router address instead of the actual user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` â€” (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks the router's address â€” not the actual user's address. If the router is allowlisted (or `allowAllSwappers` is set), any unprivileged user can bypass the curated pool's swap allowlist entirely.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value forwarded above: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point), the router calls `pool.swap(...)` directly: [4](#0-3) 

At that point `msg.sender` inside the pool is the **router**, not the original user. The extension therefore evaluates `allowedSwapper[pool][router]` â€” a check on the router's address, not the actual swapper's address. The real user's identity is never consulted.

---

### Impact Explanation

**Critical / High.** A pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict swaps to a specific set of addresses. If the router is added to the allowlist (a natural operational step so that allowlisted users can use the standard periphery), the allowlist is completely voided: any address can call `router.exactInputSingle` and the extension will pass because it only sees the router. Even without the router being explicitly allowlisted, the design means allowlisted users cannot use the router at all, breaking the intended user flow. The direct consequence is either unauthorized swaps draining a curated pool or a permanently broken swap path for legitimate users.

---

### Likelihood Explanation

**High.** `MetricOmmSimpleRouter` is the standard, publicly documented swap entry point for the protocol. Pool admins who want their allowlisted users to be able to trade will naturally allowlist the router. The bypass requires no special privileges, no flash loans, and no unusual token behavior â€” any address can call the router.

---

### Recommendation

The extension must gate the **original user**, not the intermediary. Two complementary fixes:

1. **Pass the original user through the router.** The router should forward the original `msg.sender` as an explicit `sender` argument to `pool.swap`, and the pool should pass that value (rather than its own `msg.sender`) to `_beforeSwap`. This requires a small interface change.

2. **Alternatively, check `sender` (the first argument) in the extension rather than the pool-level `msg.sender` chain.** The extension already receives `sender` as its first parameter; the allowlist lookup should use that value only when the caller is a trusted router registered with the factory.

The simplest safe fix is option 1: the router stores the original caller in transient storage (it already does this for the payer in `_setNextCallbackContext`) and passes it as the `sender` to the pool, which then forwards it to extensions.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` so that allowlisted users can trade via the standard periphery.
3. Attacker (not in the allowlist) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` â€” pool's `msg.sender` is the router.
5. Pool calls `_beforeSwap(sender=router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router]` â†’ `true` â†’ passes.
7. Attacker's swap executes on the curated pool despite never being allowlisted.

The check that was supposed to block the attacker â€” `allowedSwapper[pool][attacker]` â€” is never evaluated. The structural parallel to the seed bug is exact: just as the Cosmos loop body never executes when no coins are sent, the Solidity allowlist check never evaluates the real user's address when the call arrives through the router.
