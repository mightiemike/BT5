### Title
`SwapAllowlistExtension.beforeSwap` checks the router's address instead of the actual swapper, enabling allowlist bypass through `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user. The allowlist therefore checks the router's address, not the actual economic actor. This creates an irreconcilable wrong-actor binding: if the router is not allowlisted, allowlisted users cannot use the router at all; if the router is allowlisted to fix that, every user on the network can bypass the allowlist by routing through it.

---

### Finding Description

**Call path:**

```
User → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(recipient, zeroForOne, ..., extensionData)   [msg.sender = router]
             → ExtensionCalling._beforeSwap(msg.sender=router, ...)
                 → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                     → allowedSwapper[pool][router]  ← wrong actor checked
```

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← router address when called via router
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged:

```solidity
// ExtensionCalling.sol
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
```

`SwapAllowlistExtension.beforeSwap` then checks it against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (the extension is called by the pool), and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`.

The router never forwards the original user's identity to the pool; it only stores it in transient storage for the payment callback:

```solidity
// MetricOmmSimpleRouter.sol
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(...);   // pool sees msg.sender = router
```

There is no mechanism in the extension interface or the router to propagate the original caller's address to the `beforeSwap` hook.

**The two failure modes are mutually exclusive and both harmful:**

| Router allowlist state | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot swap through the router — broken core functionality |
| Router **allowlisted** | Every user bypasses the allowlist by routing through the router — security bypass |

---

### Impact Explanation

**Allowlist bypass (High):** A pool admin who wants allowlisted users to be able to use the standard router UX must add the router to the allowlist. Once the router is allowlisted, any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle()` (or any `exact*` variant) and the `beforeSwap` hook passes because `allowedSwapper[pool][router] == true`. The allowlist is completely defeated for all router-mediated swaps. On a curated pool (e.g., KYC-gated, institution-only, or regulatory-restricted), this allows arbitrary users to trade against LP funds that were deposited under the assumption that only vetted counterparties would interact.

**Broken functionality (Medium):** If the admin does not allowlist the router, allowlisted users are forced to call `pool.swap()` directly, bypassing the router's slippage protection, deadline checks, and multi-hop routing. This degrades the usability of any allowlisted pool and may cause allowlisted users to suffer worse execution.

---

### Likelihood Explanation

The router is the primary user-facing entry point for swaps. Any pool that deploys a `SwapAllowlistExtension` and also wants its allowlisted users to use the router will encounter this problem immediately. The admin's natural remediation — allowlisting the router — directly enables the bypass. The attacker requires no special privilege: any EOA can call `MetricOmmSimpleRouter.exactInputSingle()`.

---

### Recommendation

The `beforeSwap` hook should gate on the **original user**, not the intermediary. Two approaches:

1. **Forward the original caller via `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated change to the router and the extension.

2. **Check `recipient` instead of `sender` for router flows, or use `tx.origin` as a fallback:** Neither is ideal (`tx.origin` is generally discouraged; `recipient` is not always the swapper).

3. **Preferred — gate on `sender` but document that the router must never be allowlisted, and provide a router variant that calls `pool.swap()` with the user as `msg.sender` via a delegatecall or a dedicated trusted-forwarder pattern.**

The cleanest fix is to redesign the extension interface so that the original initiator's address is always passed as a distinct, unforgeable parameter separate from the immediate `msg.sender` of the pool call.

### Proof of Concept

```
Setup:
  pool P with SwapAllowlistExtension E
  allowedSwapper[P][alice] = true   (alice is the intended gated user)
  allowedSwapper[P][router] = true  (admin adds router so alice can use it)

Attack:
  bob (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool: P, recipient: bob, ...})

Execution:
  router → P.swap(bob, ...)          [msg.sender to pool = router]
  P → E.beforeSwap(sender=router, ...)
  E checks: allowedSwapper[P][router] == true  → passes
  bob's swap executes on the allowlisted pool
```

Bob, who is not in the allowlist, successfully swaps on a pool that was configured to restrict access to allowlisted users only. The allowlist extension provides no protection for any router-mediated swap once the router address is added to the allowlist. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
