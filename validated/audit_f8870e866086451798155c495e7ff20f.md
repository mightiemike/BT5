### Title
Non-Bond-Denom Tokens Sent to `rewards_pool` Module Account Are Permanently Locked — (`app/app.go`, `x/tieredrewards/keeper/abci.go`, `x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

The `tieredrewards` module's `rewards_pool` module account is explicitly whitelisted to receive external funds via `MsgSend` from any unprivileged account. However, the module exclusively disburses the **bond denom** from this pool (for both bonus rewards and base-rewards top-up). Any non-bond-denom token sent to the pool is permanently locked: no governance message, no admin sweep, and no module-level disbursement path exists to recover it.

---

### Finding Description

`app/app.go` registers two relevant facts about the `rewards_pool` module account:

1. It has `nil` permissions — it cannot mint or burn tokens.
2. It is placed in `moduleAccsAllowedToReceiveExternalFunds`, which bypasses the default Cosmos SDK block on `MsgSend` to module accounts. [1](#0-0) 

This means any user can send any token denomination to the `rewards_pool` address via a standard `MsgSend` transaction.

The module has exactly two disbursement paths from the pool:

**Path 1 — BeginBlocker base-rewards top-up** (`abci.go`): reads only the bond-denom balance and sends only `sdk.NewCoin(bondDenom, topUpAmount)` to the distribution module. [2](#0-1) 

**Path 2 — Bonus reward claim** (`bonus_rewards.go` / `claim_rewards.go`): computes bonus exclusively in bond denom (shares × tokensPerShare × APY) and calls `SendCoinsFromModule(rewards_pool, owner, bonus)` with a bond-denom-only `sdk.Coins`. [3](#0-2) 

The `sufficientBonusPoolBalance` check uses `poolBalance.IsAllGTE(bonus)` where `bonus` is always a single bond-denom coin. Non-bond-denom balances in the pool are never read, never disbursed, and never checked. [3](#0-2) 

The full set of governance messages for this module (`MsgAddTier`, `MsgUpdateTier`, `MsgDeleteTier`, `MsgUpdateParams`) contains no sweep or recovery function for the pool's non-bond-denom balance. [4](#0-3) 

---

### Impact Explanation

Any non-bond-denom token (IBC vouchers, other native denoms) sent to the `rewards_pool` module account is permanently locked. The `nil` module permissions prevent minting/burning, and no code path in the module ever reads or transfers non-bond-denom balances out of the pool. The corrupted value is the non-bond-denom balance at the `rewards_pool` module account address. This is a direct, irreversible loss of the sender's funds.

---

### Likelihood Explanation

The `moduleAccsAllowedToReceiveExternalFunds` whitelist is the explicit, documented mechanism for funding the rewards pool. A pool operator or community governance participant who intends to fund the pool with a different token (e.g., an IBC-bridged asset or a second native denom) will succeed in sending the tokens but will find them permanently unrecoverable. The entry path is a standard `MsgSend` transaction, requiring no special privileges. [5](#0-4) 

---

### Recommendation

Add a governance-gated sweep message (e.g., `MsgSweepRewardsPool`) that allows governance to transfer any non-bond-denom balance from the `rewards_pool` module account to a specified recipient address. Alternatively, restrict `moduleAccsAllowedToReceiveExternalFunds` to only the bond denom at the `AnteHandler` level, rejecting `MsgSend` transactions that attempt to deposit non-bond-denom tokens into the pool.

---

### Proof of Concept

1. Query the `rewards_pool` module account address:
   ```
   chain-maind query auth module-account rewards_pool
   ```
2. Send a non-bond-denom IBC token to it from any account:
   ```
   chain-maind tx bank send <sender> <rewards_pool_addr> 1000ibc/XXXX --from <key>
   ```
3. Observe the transaction succeeds (the whitelist permits it).
4. Query the pool balance:
   ```
   chain-maind query bank balances <rewards_pool_addr>
   ```
   The IBC token appears in the balance.
5. Attempt any module operation (claim rewards, BeginBlocker top-up) — none of them touch the IBC token balance.
6. Confirm there is no message type in `x/tieredrewards` that can move the IBC token out of the pool. The tokens are permanently locked. [6](#0-5) [7](#0-6)

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```
