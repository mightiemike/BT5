### Title
Residual Non-Zero Allowance in `DirectDepositV1.creditDeposit()` Permanently Bricks Deposits for USDT-like Tokens — (File: `core/contracts/DirectDepositV1.sol`)

### Summary
`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. `Endpoint.depositCollateralWithReferral()` contains an early-return path that skips the actual `transferFrom` when the deposit amount is below the minimum threshold, leaving the allowance non-zero. On any subsequent call to `creditDeposit()` for the same token, the bare `approve` call reverts for USDT-like tokens that require the allowance to be zero before a new non-zero value is set, permanently bricking the deposit path for that token on the affected DDA.

### Finding Description
In `DirectDepositV1.creditDeposit()`:

```solidity
// core/contracts/DirectDepositV1.sol, lines 91–99
if (balance != 0) {
    token.approve(address(endpoint), balance);          // sets allowance = balance
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),
        "-1"
    );
}
``` [1](#0-0) 

`Endpoint.depositCollateralWithReferral()` has an explicit early return when the deposit amount is below the minimum:

```solidity
// core/contracts/Endpoint.sol, lines 137–142
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked...
    // we can just skip the deposit and continue with the next asset.
    return;
}
``` [2](#0-1) 

When this early return fires, `handleDepositTransfer` is never called, so no `transferFrom` occurs and the allowance set by `token.approve(address(endpoint), balance)` is never consumed. The allowance remains at `balance` (non-zero). On the next invocation of `creditDeposit()`, the token balance is still non-zero (tokens were not moved), so the `if (balance != 0)` branch is entered again and `token.approve(address(endpoint), balance)` is called a second time against an already non-zero allowance — which reverts for USDT-like tokens.

The developers are already aware of this pattern: `ContractOwner.wrapVaultAsset()` correctly uses `approve(0)` before `approve(amount)`:

```solidity
// core/contracts/ContractOwner.sol, lines 530–531
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [3](#0-2) 

This mitigation was not applied to `DirectDepositV1.creditDeposit()`.

### Impact Explanation
Once the allowance is left non-zero, every subsequent call to `creditDeposit()` for that token reverts at the `approve` step. The DDA can no longer credit collateral for that token to the subaccount. Any USDT-like tokens sitting in the DDA are effectively frozen from the deposit path. Recovery requires the owner to call `withdrawFromDirectDepositV1`, which is an admin-only rescue path and not the intended protocol flow. The subaccount loses access to timely collateral crediting.

### Likelihood Explanation
`creditDeposit()` is `external` with no access modifier — any caller can invoke it. [4](#0-3) 

The trigger condition — a deposit amount below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT` — is realistic: a user may send a dust amount of a USDT-like token to the DDA, or the minimum deposit threshold may be raised by governance after a deposit is already sitting in the DDA. An attacker can also deliberately send a sub-minimum amount to a victim's DDA to trigger the stuck-allowance state before the victim's real deposit arrives.

### Recommendation
Reset the allowance to zero before setting the new value in `creditDeposit()`:

```solidity
if (balance != 0) {
    token.approve(address(endpoint), 0);           // reset first
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),
        "-1"
    );
}
```

This mirrors the pattern already used in `ContractOwner.wrapVaultAsset()` at lines 530–531. [3](#0-2) 

### Proof of Concept
1. A USDT-like token is a supported spot product in `SpotEngine`.
2. A small sub-minimum amount of that token is sent to a `DirectDepositV1` DDA (by the user or an attacker).
3. Anyone calls `DirectDepositV1.creditDeposit()`.
4. `token.approve(address(endpoint), balance)` sets allowance = `balance` (non-zero).
5. `endpoint.depositCollateralWithReferral(...)` hits the early return at line 141 of `Endpoint.sol` because `balance < MIN_DEPOSIT_AMOUNT`; no `transferFrom` occurs; allowance remains non-zero.
6. The user later sends a proper deposit amount to the DDA and calls `creditDeposit()` again.
7. `token.approve(address(endpoint), balance)` is called with a non-zero existing allowance → USDT reverts → the entire `creditDeposit()` call reverts.
8. The DDA is permanently unable to credit that token's collateral via `creditDeposit()`. [5](#0-4) [6](#0-5)
