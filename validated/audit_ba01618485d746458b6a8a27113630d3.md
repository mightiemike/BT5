### Title
Silent Bonus Reward Forfeiture with Checkpoint Advancement During Redelegation Slash — (`x/tieredrewards/keeper/slash.go`)

### Summary

In `slashRedelegationPosition`, when the rewards pool is insufficient to pay accrued bonus during a `BeforeRedelegationSlashed` hook, the `ErrInsufficientBonusPool` error is deliberately swallowed to avoid a chain halt. However, `processEventsAndClaimBonus` has already advanced the position's accounting checkpoints (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) in memory and decremented event reference counts in the store. The function then persists the updated position via `setPositionWithState`. The result is that the bonus owed to the position owner is permanently lost with no accounting record and no recovery path.

### Finding Description

`slashRedelegationPosition` calls `processEventsAndClaimBonus` to settle bonus before the slash modifies shares:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // continues — does NOT return
    } else {
        return err
    }
}
``` [1](#0-0) 

Inside `processEventsAndClaimBonus`, before the pool-balance check, the function:
1. Walks all pending validator events, calling `pos.UpdateLastEventSeq(entry.Seq)` on each
2. Calls `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` — a **store mutation** that may garbage-collect events
3. Calls `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` advancing `LastBonusAccrual` to the current block time
4. Calls `pos.UpdateLastKnownBonded(bonded)`

Only then does it check pool balance and return `nil, err` if insufficient:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
``` [2](#0-1) 

When `ErrInsufficientBonusPool` is caught and swallowed in `slashRedelegationPosition`, the `pos` variable already carries the advanced checkpoints. The function then falls through to `setPositionWithState`, which persists the modified position:

```go
// partial slash path
pos.Delegation.Shares = pos.Delegation.Shares.Sub(sharesToUnbond)
return k.setPositionWithState(ctx, pos, nil)
``` [3](#0-2) 

For the full-slash path, `ClearBonusCheckpoints()` is called, which zeros the checkpoints — but the bonus is still not paid. [4](#0-3) 

The net result in both cases: the bonus that accrued from the previous `LastBonusAccrual` to the slash block time is permanently unrecoverable. The position's checkpoints have moved forward (or been cleared), so a subsequent `MsgClaimTierRewards` will compute bonus only from the new checkpoint onward. The lost interval is gone from accounting entirely.

This is structurally identical to the `safeRewardTransfer` pattern in the external report: a "safe" fallback that prioritizes liveness over correctness silently forfeits user funds with no accounting record.

### Impact Explanation

The exact corrupted value is the position owner's bonus reward balance. For a position that has been delegated for a long period before the slash (e.g., months at a high APY tier), the forfeited bonus can be substantial. The loss is:

- **Permanent**: checkpoints are advanced past the unpaid interval; no retry is possible
- **Silent**: no error is returned to any caller; only a log line is emitted
- **Untracked**: no debt record, no event, no state that would allow governance or the protocol to identify or compensate affected users

### Likelihood Explanation

The trigger requires two concurrent conditions:
1. A tier position is in an active redelegation (user called `MsgTierRedelegate` and the SDK unbonding period has not elapsed)
2. The `RewardsPoolName` module account has insufficient balance to cover the accrued bonus at the moment the source validator is slashed

Both conditions are realistic in production:
- Redelegations are a normal user action
- The rewards pool is funded externally and can be depleted (e.g., if many positions claim simultaneously, or if the pool was not topped up)
- Validator slashing is a normal protocol event triggered by double-signing or extended downtime

The position owner has no control over when slashing occurs and cannot preemptively claim to avoid the loss.

### Recommendation

The checkpoint advancement and event reference-count decrements inside `processEventsAndClaimBonus` must not be persisted when the bonus cannot be paid. Two options:

1. **Preferred**: Run `processEventsAndClaimBonus` inside a cache context (`ctx.CacheContext()`). If it returns `ErrInsufficientBonusPool`, discard the cache — no checkpoints advance, no reference counts decrement, no bonus is paid. The position retains its old checkpoints and will re-compute the same bonus once the pool is replenished (analogous to the user-driven path behavior documented in ADR-006).

2**Alternative**: Separate checkpoint advancement from payment. Only call `setPositionWithState` with the advanced checkpoints after a successful `SendCoinsFromModuleToAccount`. If the pool is insufficient, persist the position with its original checkpoints.

The ADR-006 documentation already states the correct invariant for user-driven paths: "fail atomically. User retries after pool is replenished." The `BeforeRedelegationSlashed` hook path should honour the same invariant for checkpoint state, even if it cannot return an error to the SDK hook caller. [5](#0-4) 

### Proof of Concept

1. Alice creates a tier position via `MsgLockTier` with a high-APY tier and delegates to validator V1.
2. Alice calls `MsgTierRedelegate` to move to validator V2. A redelegation entry is created; `RedelegationMappings[unbondingId] = positionId` is stored.
3. One year passes. Alice's accrued bonus is large (e.g., 10,000 tokens).
4. The rewards pool is drained to near-zero (e.g., by other claimants or simply not replenished).
5. V1 double-signs. The Cosmos SDK calls `BeforeRedelegationSlashed(ctx, unbondingId, sharesToUnbond)`.
6. `slashRedelegationPosition` calls `processEventsAndClaimBonus`. Inside, `LastBonusAccrual` is advanced to the current block time, event ref-counts are decremented, but `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`.
7. The error is swallowed. `setPositionWithState` persists the position with the new `LastBonusAccrual`.
8. Alice later calls `MsgClaimTierRewards`. Bonus is computed only from the new `LastBonusAccrual` forward — the 10,000 tokens are gone with no record. [6](#0-5) [7](#0-6) [8](#0-7)
