### Title
Checkpoint Advancement Before Pool Check in `processEventsAndClaimBonus` Causes Permanent Bonus Loss on Redelegation Slash — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

When `BeforeRedelegationSlashed` fires and the `RewardsPoolName` is empty, `processEventsAndClaimBonus` advances the position's `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded` checkpoints **and** decrements event reference counts in the KV store **before** checking pool sufficiency. The error is then swallowed by `slashRedelegationPosition`, which persists the modified position. The accrued bonus is permanently unclaimable even after the pool is replenished.

---

### Finding Description

**Step 1 — Checkpoint mutation before pool check in `processEventsAndClaimBonus`:**

Inside `processEventsAndClaimBonus`, the function processes events in a loop, calling `pos.UpdateLastEventSeq(entry.Seq)` (line 193) and `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` (line 196) — the latter writes directly to the KV store — for every event. After the loop, `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` (line 215) and `pos.UpdateLastKnownBonded(bonded)` (line 217) mutate the in-memory `pos`. Only then, at line 230, does `sufficientBonusPoolBalance` run. If the pool is insufficient, `ErrInsufficientBonusPool` is returned — but by this point:

- `pos.LastBonusAccrual` has been advanced to `blockTime`
- `pos.LastEventSeq` has been advanced past all processed events
- `pos.LastKnownBonded` has been updated
- Event reference counts in the KV store have been decremented (events may be garbage-collected) [1](#0-0) [2](#0-1) [3](#0-2) 

**Step 2 — Error swallowed and mutated position persisted in `slashRedelegationPosition`:**

`slashRedelegationPosition` calls `processEventsAndClaimBonus` with `&pos` (pointer). When `ErrInsufficientBonusPool` is returned, the error is caught and logged but execution continues. The function then calls `k.setPositionWithState(ctx, pos, nil)` at line 77, persisting the already-mutated `pos` with its advanced checkpoints. [4](#0-3) [5](#0-4) 

**Step 3 — Bonus permanently unclaimable:**

On the next call to `processEventsAndClaimBonus` (e.g., via `MsgClaimTierRewards`), the function reads `pos.LastBonusAccrual` as the segment start. Since it was advanced to the slash block time, the pre-slash accrual period is skipped entirely. The events were also garbage-collected (reference counts decremented), so they cannot be replayed. The bonus for the entire pre-slash period is permanently lost. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

A position owner with an active redelegation permanently loses all bonus rewards accrued from `LastBonusAccrual` up to the slash block time whenever the `RewardsPoolName` is empty at the moment the slash fires. After the pool is replenished, `MsgClaimTierRewards` returns zero bonus for that period because `LastBonusAccrual` already points past it. This is a direct, irrecoverable fund loss proportional to the position size, tier APY, and elapsed time.

---

### Likelihood Explanation

The pool being empty is a realistic operational condition: it can be depleted by normal bonus claims from other positions, by the BeginBlocker consuming it for base-reward top-ups, or by an adversary who claims their own legitimate rewards to drain it before a known slash event (e.g., after a double-sign evidence is submitted). The `BeforeRedelegationSlashed` hook fires automatically during staking's `SlashRedelegation`, which is a standard production path triggered by evidence processing. No governance or privileged access is required. [8](#0-7) [9](#0-8) 

---

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the successful `SendCoinsFromModuleToAccount` call, and move `decrementEventRefCount` to after the pool check as well. This ensures that if the pool is insufficient, no state mutations are committed — the position retains its original checkpoints and the events retain their reference counts, allowing the owner to retry after the pool is replenished. Alternatively, in the `slashRedelegationPosition` error handler for `ErrInsufficientBonusPool`, explicitly restore the original checkpoint values on `pos` before calling `setPositionWithState`.

```go
func (k Keeper) decrementEventRefCount(ctx context.Context, valAddr sdk.ValAddress, seq uint64) error {
	key := collections.Join(valAddr, seq)
	event, err := k.ValidatorEvents.Get(ctx, key)
	if errors.Is(err, collections.ErrNotFound) {
		return nil // already cleaned up
	}
	if err != nil {
		return err
	}

	if event.ReferenceCount <= 1 {
		return k.ValidatorEvents.Remove(ctx, key)
	}

	event.ReferenceCount--
	return k.ValidatorEvents.Set(ctx, key, event)
}
```

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```
