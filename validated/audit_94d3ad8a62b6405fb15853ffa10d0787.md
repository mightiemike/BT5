### Title
Missing Validation for Duplicate Atom Entries and Non-Canonical Multi-Atom Groups in serde_2026 Deserializer — (`File: src/serde_2026/de.rs`)

### Summary

`deserialize_2026_body_from_stream` in `src/serde_2026/de.rs` accepts serde_2026 blobs that violate two format invariants: (1) the atom table may contain duplicate atom byte sequences (the same bytes at multiple indices), and (2) a multi-atom group (negative-length varint) may declare `count = 1`, which the format spec reserves for singleton groups (positive-length varint). Neither condition is rejected. The result is that multiple structurally distinct byte sequences can deserialize to the same CLVM tree, breaking the one-to-one blob-to-tree mapping the format is designed to guarantee.

### Finding Description

The serde_2026 format specification (`docs/serde-2026.md`) states two invariants for the atom table:

1. The table contains **all unique atoms** (except nil).
2. A negative-length varint introduces a group of **multiple** atoms of the same length; a positive-length varint introduces a **singleton** group.

The deserializer in `deserialize_2026_body_from_stream` enforces neither:

**Missing duplicate-atom check** — atoms are appended to the `atoms` Vec unconditionally:

```rust
for _ in 0..count {
    reader.read_exact(&mut buf)...;
    atoms.push(allocator.new_atom(&buf)?);   // no uniqueness check
}
```

An attacker can include the same byte sequence twice in the atom table. Both entries receive distinct indices. Instructions `N >= 2` can reference either index to push the same atom value, producing the same tree from a different blob.

**Missing `count > 1` check for multi-atom groups** — the only guard is `count == 0`:

```rust
if length == 0 || count == 0 {
    return Err(EvalErr::SerializationError);
}
```

A negative-length varint with `count = 1` is accepted. The canonical encoding for a single atom uses a positive-length varint (singleton group). Both encodings produce the same atom at the same index, but the blob bytes differ.

The serializer (`src/serde_2026/ser.rs`) always produces canonical blobs via `intern_tree` (which deduplicates atoms) and `write_atom_table` (which uses singleton groups for lone atoms). The deserializer imposes no matching constraint on input.

`serialized_length_serde_2026` mirrors the same missing checks, so it also returns `Ok` for non-canonical blobs, preserving the `Ok`/`Ok` pairing but not the canonicality invariant.

### Impact Explanation

Any system that uses the raw serde_2026 blob bytes as a unique identifier — transaction ID, mempool deduplication key, blob-level cache key, or content-addressed store — will treat two different blobs representing the same CLVM tree as distinct objects. Concretely:

- **Mempool deduplication bypass**: An attacker submits the same spend bundle twice, each time with a different non-canonical encoding of the same CLVM program. A mempool that deduplicates by blob hash admits both, doubling processing work or bypassing rate limits.
- **Cache aliasing / consensus divergence**: If one node caches a program result keyed by blob bytes and another node receives a differently-encoded blob for the same program, the cache miss causes re-execution with a potentially different cost accounting or result under future dialect changes.
- **Broken format invariant**: The format is explicitly designed to produce a unique encoding per tree (via atom deduplication and canonical group encoding). Accepting non-canonical blobs undermines any downstream assumption of uniqueness.

### Likelihood Explanation

The attack requires only crafting a valid serde_2026 blob with either a repeated atom entry or a multi-atom group with `count = 1`. Both are trivially constructable by any party who can submit CLVM bytes to a node. No privileged access is required. The entry points `deserialize_2026` and `deserialize_2026_body_from_stream` are public API, exposed through the Python wheel (`wheel/src/api.rs`) and directly callable from Rust.

### Recommendation

In `deserialize_2026_body_from_stream` (`src/serde_2026/de.rs`):

1. **Reject duplicate atoms**: maintain a `HashSet` of atom byte sequences seen so far; return `EvalErr::SerializationError` if a duplicate is encountered.
2. **Reject `count == 1` for multi-atom groups**: after parsing `count` for a negative-length group, add `if count < 2 { return Err(EvalErr::SerializationError); }`.

Apply the same two checks to `serialized_length_serde_2026` to preserve the documented invariant that `Ok` from the length function implies `Ok` from the deserializer under identical validation policy.

### Proof of Concept

**Duplicate atom (two blobs → same tree):**

```
Blob A (canonical):
  group_count=1, length=1, atom=0x41 ("A")
  instruction_count=1, inst=2  → push atom[0]="A"

Blob B (non-canonical, duplicate atom):
  group_count=2, length=1, atom=0x41; length=1, atom=0x41
  instruction_count=1, inst=3  → push atom[1]="A"
```

Both blobs deserialize to the atom `"A"`. Blob B is accepted without error by `deserialize_2026_body_from_stream` because there is no uniqueness check on atom bytes.

**Multi-atom group with count=1 (two blobs → same tree):**

```
Blob A (canonical):
  group_count=1, length=+1 (singleton), atom=0x41
  instruction_count=1, inst=2

Blob B (non-canonical, multi-atom group count=1):
  group_count=1, length=-1 (multi), count=1, atom=0x41
  instruction_count=1, inst=2
```

```rust
pub fn serialized_length_serde_2026(buf: &[u8], max_atom_len: usize, strict: bool) -> Result<u64> {
    if !buf.starts_with(&SERDE_2026_MAGIC_PREFIX) {
        return Err(EvalErr::SerializationError);
    }

    let data = &buf[SERDE_2026_MAGIC_PREFIX.len()..];
    let mut cursor = Cursor::new(data);

    let group_count = checked_usize(read_varint(&mut cursor, strict)?)?;
    for _ in 0..group_count {
        let length_val = read_varint(&mut cursor, strict)?;
        let skip = if length_val < 0 {
            if length_val == i64::MIN {
                return Err(EvalErr::SerializationError);
            }
            let atom_len = checked_bounded_usize(-length_val, max_atom_len)?;
            let count = checked_usize(read_varint(&mut cursor, strict)?)?;
            if atom_len == 0 || count == 0 {
                return Err(EvalErr::SerializationError);
            }
            (atom_len as u64)
                .checked_mul(count as u64)
                .ok_or(EvalErr::SerializationError)?
        } else {
            let atom_len = checked_bounded_usize(length_val, max_atom_len)?;
            if atom_len == 0 {
                return Err(EvalErr::SerializationError);
            }
            atom_len as u64
        };
        let new_pos = cursor
            .position()
            .checked_add(skip)
            .ok_or(EvalErr::SerializationError)?;
        if new_pos > data.len() as u64 {
            return Err(EvalErr::SerializationError);
        }
        cursor.set_position(new_pos);
    }

    let instruction_count = checked_usize(read_varint(&mut cursor, strict)?)?;
    // Mirror `deserialize_2026_body_from_stream`: instruction_count == 0
    // leaves the stack empty and is rejected there, so reject it here too.
    if instruction_count == 0 {
        return Err(EvalErr::SerializationError);
    }
    for _ in 0..instruction_count {
        read_varint(&mut cursor, strict)?;
    }

    Ok(SERDE_2026_MAGIC_PREFIX.len() as u64 + cursor.position())
}
```

```markdown
1. **Atom table** — all unique atoms (except nil), grouped by length

### Atom Table

Nil (the empty atom) is **not** included in the atom table — it has a dedicated
opcode (`0`) in the instruction stream.

The atom table begins with a varint encoding the number of atom groups.

For each group (in stream order):

- If the group contains **one** atom: a positive varint encoding the atom's byte
  length, followed by the atom's raw bytes.
- If the group contains **multiple** atoms of the same length: a negative varint
  encoding the negated byte length, then a positive varint encoding the count,
  then the raw bytes of each atom concatenated (each is exactly `length` bytes).

Atom lengths must be non-zero because nil is excluded from the atom table.
Deserializers enforce a configurable maximum atom length (default: 1 MiB) and a
maximum input byte budget (default: 10 MiB). Separate atom-group, atom-count,
instruction-count, stack-size, and pair-count limits are not needed for DoS
protection: every declared item must consume at least one input byte before it
can produce parser work or allocate a CLVM node. The input byte budget therefore
bounds all of those quantities.

Atoms are assigned indices starting from 0, in the order they appear in the
table.

The decoder accepts groups in any order. Multiple groups with the same byte
length are valid (they contribute separate atom indices). A serializer may
choose a specific ordering strategy (for example, sorting by frequency so
commonly-referenced atoms land in lower index ranges whose varint encodings are
shorter).
```
