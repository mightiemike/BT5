### Title
`submitFastWithdrawal()` incorrectly pulls fee from `msg.sender` (relayer) instead of deducting from withdrawal amount — (`File: core/contracts/BaseWithdrawPool.sol`)

### Summary
In `BaseWithdrawPool.submitFastWithdrawal()`, when a third-party relayer (where `sendTo != msg.sender`) processes a fast withdrawal, the contract attempts to pull the fee directly from `msg.sender` via `safeTransferFrom`. A relayer will not normally hold the withdrawal token, so the call reverts and fast withdrawals become non-functional for any relayer-submitted path. If a relayer does hold the token and has approved the contract, their tokens are drained as fees with no compensation.

### Finding Description

`submitFastWithdrawal()` is a public, permissionless function. Any address can call it to process a signed fast-withdrawal transaction on behalf of a user. The function resolves `sendTo` from the signed transaction data — it is the user's address, not the caller's.

The fee logic branches on whether the caller is also the recipient:

```solidity
if (sendTo == msg.sender) {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);          // fee deducted from what user receives
} else {
    safeTransferFrom(token, msg.sender, uint128(fee));  // fee pulled from relayer's wallet
}
```

In the `else` branch (relayer case), the fee is pulled from `msg.sender` (the relayer), while `handleWithdrawTransfer(token, sendTo, transferAmount)` sends the **full** `transferAmount` from the pool to the user. The relayer subsidises the fee with no mechanism for reimbursement, and the pool pays out the full amount.

The correct behaviour — consistent with the `sendTo == msg.sender` branch — is to always deduct the fee from `transferAmount` before sending it to `sendTo`, regardless of who `msg.sender` is.

### Impact Explanation

- **Fast withdrawal DoS**: Any relayer calling `submitFastWithdrawal()` for a user (`sendTo != msg.sender`) will have the transaction revert at `safeTransferFrom` because the relayer does not hold the withdrawal token or has not approved the pool. Fast withdrawals are entirely non-functional via the relayer path.
- **Relayer token drain**: A relayer that does hold the token and has approved the pool will have their tokens taken as fees with no compensation, constituting direct asset loss for the relayer.
- **Pool accounting corruption**: In the drain scenario, `fees[productId]` is credited from the relayer's wallet while the pool pays out the full `transferAmount` — the pool's liquidity is depleted by the full amount but only the relayer (not the pool) contributed the fee, breaking the pool's fee-accounting invariant.

### Likelihood Explanation

High. `submitFastWithdrawal()` is a public function with no access control. The relayer path (`sendTo != msg.sender`) is the primary intended use case — a relayer processes withdrawals for users. Any attempt to use this path triggers the bug. The `sendTo == msg.sender` path (user calls for themselves) is a narrow edge case.

### Recommendation

Always deduct the fee from `transferAmount`, regardless of who `msg.sender` is:

```solidity
require(transferAmount > uint128(fee), "Fee larger than balance");
transferAmount -= uint128(fee);
fees[productId] += fee;
handleWithdrawTransfer(token, sendTo, transferAmount);
```

Remove the `safeTransferFrom(token, msg.sender, ...)` call entirely. The fee is already retained in the pool because `handleWithdrawTransfer` sends only the reduced `transferAmount`.

### Proof of Concept

1. User signs a `WithdrawCollateral` transaction for `productId=1`, `amount=1000e6 USDC`, `sendTo=userAddress`.
2. Relayer (address `R`, holding no USDC) calls `WithdrawPool.submitFastWithdrawal(idx, transaction, signatures)`.
3. `resolveFastWithdrawal` returns `sendTo = userAddress`, `transferAmount = 1000e6`.
4. `sendTo != msg.sender` → code enters `else` branch.
5. `safeTransferFrom(token, R, fee)` reverts because `R` holds no USDC.
6. Fast withdrawal is blocked; the signed transaction cannot be processed by any relayer.

Alternatively, if `R` holds USDC and has approved the pool:
- `R`'s USDC is taken as the fee.
- Pool sends full `1000e6` to `userAddress`.
- `R` loses the fee amount with no compensation.
