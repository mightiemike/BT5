### Title
Non-Zero-Sum Funding Mechanism Creates Protocol Deficit When Open Interest Is Imbalanced — (`core/contracts/PerpEngineState.sol`)

---

### Summary

`updateStates` increments both `cumulativeFundingLongX18` and `cumulativeFundingShortX18` by the identical `paymentAmount`. Because `_updateBalance` applies each rate to the signed position size, the net change in total `vQuoteBalance` across all users equals `paymentAmount × (totalShortOI − totalLongOI)`. When this product is positive the protocol creates `vQuoteBalance` from nothing; that phantom balance inflates health scores and, once `availableSettle` has capacity, can be settled into real spot tokens — leaving the protocol undercollateralized.

---

### Finding Description

**Root cause — `updateStates`** (`PerpEngineState.sol` lines 130–132):

```solidity
int128 paymentAmount = priceDiffX18.mul(dtX18).div(ONE_DAY_X18);
state.cumulativeFundingLongX18 += paymentAmount;   // ← same delta
state.cumulativeFundingShortX18 += paymentAmount;  // ← same delta
``` [1](#0-0) 

**How `_updateBalance` materialises the funding** (`PerpEngineState.sol` lines 31–36):

```solidity
int128 diffX18 = cumulativeFundingAmountX18 - balance.lastCumulativeFundingX18;
int128 deltaQuote = vQuoteDelta - diffX18.mul(balance.amount);
``` [2](#0-1) 

For a long (`amount = a > 0`): `deltaQuote = −paymentAmount × a` → vQuoteBalance decreases by `paymentAmount × a`.
For a short (`amount = a < 0`): `deltaQuote = −paymentAmount × a = +paymentAmount × |a|` → vQuoteBalance increases by `paymentAmount × |a|`.

**Net change in total vQuoteBalance across all positions:**

```
Δ(total vQuoteBalance) = paymentAmount × (totalShortOI − totalLongOI)
```

When `totalShortOI > totalLongOI` and `paymentAmount > 0` (or `totalLongOI > totalShortOI` and `paymentAmount < 0`), this quantity is **positive**: the protocol credits shorts more than it debits longs, creating `vQuoteBalance` backed by no real collateral.

**Impact path 1 — health inflation** (`BaseEngine.sol` lines 163–168):

```solidity
(int128 amount, int128 quoteAmount) = _getBalance(productId, subaccount);
health += quoteAmount;   // quoteAmount IS vQuoteBalance
``` [3](#0-2) 

Inflated `vQuoteBalance` directly inflates the health score, allowing users to open positions larger than their real collateral supports and to evade liquidation.

**Impact path 2 — settlement drain** (`PerpEngine.sol` lines 88–100):

```solidity
(int128 canSettle, State memory state, Balance memory balance)
    = getSettlementState(productId, subaccount);   // min(positionPnl, availableSettle)
state.availableSettle -= canSettle;
balance.vQuoteBalance -= canSettle;
totalSettled += canSettle;
``` [4](#0-3) 

`availableSettle` is a shared pool per product that is **never updated inside `updateStates`**. However, it is consumed by every settlement. Once the pool has capacity (replenished by trading fees, deposits, or other settlements), the phantom `vQuoteBalance` created by the funding imbalance can be settled into real spot tokens via `_settlePnl` → `spotEngine.updateBalance(QUOTE_PRODUCT_ID, subaccount, amountSettled)`, draining real collateral. [5](#0-4) 

---

### Impact Explanation

The protocol's total `vQuoteBalance` liability grows beyond the real collateral backing it whenever OI is imbalanced and the funding rate favours the larger side. This manifests as:

1. **Undercollateralization via settlement**: phantom `vQuoteBalance` is converted to real spot tokens, reducing the collateral pool below what is owed to remaining participants.
2. **Undercollateralization via health inflation**: over-leveraged positions are opened or kept alive, and when they eventually close at a loss the protocol cannot cover them.

Both paths match the Critical scope: *accounting failure in funding/settlement that transfers value incorrectly or leaves the protocol undercollateralized*.

---

### Likelihood Explanation

OI imbalance is the normal state of any perpetual market. The sequencer calls `updateStates` on every funding tick (every few seconds to minutes). The deficit accumulates continuously and silently. No special privilege or attack setup is required beyond holding a position on the dominant side.

---

### Recommendation

Funding must be zero-sum. The standard fix is to scale the rate for the smaller side so that total payments equal total receipts:

```solidity
// longs pay: paymentAmount per unit (unchanged)
state.cumulativeFundingLongX18 += paymentAmount;

// shorts receive: scaled so total receipt == total payment
// totalShortReceiptPerUnit = paymentAmount * totalLongOI / totalShortOI
if (state.openInterest > 0) {
    int128 longOI = /* tracked separately */;
    int128 shortOI = state.openInterest - longOI; // if openInterest = longOI + shortOI
    if (shortOI > 0) {
        state.cumulativeFundingShortX18 += paymentAmount.mul(longOI).div(shortOI);
    }
}
```

Alternatively, track `longOpenInterest` and `shortOpenInterest` separately and apply the asymmetric rate in `updateStates`.

    function settlePnl(bytes32 subaccount, uint256 productIds)
        external
        returns (int128)
    {
        _assertInternal();
        int128 totalSettled = 0;

        while (productIds != 0) {
            uint32 productId = uint32(productIds & ((1 << 32) - 1));
            // otherwise it means the product is a spot.
            if (productId % 2 == 0) {
                (
                    int128 canSettle,
                    State memory state,
                    Balance memory balance
                ) = getSettlementState(productId, subaccount);

                state.availableSettle -= canSettle;
                balance.vQuoteBalance -= canSettle;

                totalSettled += canSettle;

                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
            productIds >>= 32;
        }
        return totalSettled;
    }
