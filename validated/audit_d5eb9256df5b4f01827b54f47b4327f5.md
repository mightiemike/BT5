### Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Any User to Bypass Swap Allowlist on Curated Pools - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router address**, not the actual end user. A pool admin who allowlists the router to enable router-mediated swaps inadvertently opens the gate to every user, defeating the curated-pool access control entirely.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value above: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly — the actual end user (`msg.sender` of the router call) is stored only in transient storage for the payment callback and is **never forwarded to the pool**: [4](#0-3) 

The full call chain is:

```
User → router.exactInputSingle(pool, recipient, ...)
         → pool.swap(recipient, ...)          [msg.sender = router]
           → _beforeSwap(sender=router, ...)
             → SwapAllowlistExtension.beforeSwap(sender=router, ...)
               → allowedSwapper[pool][router]  ← checked, NOT the user
```

A pool admin who wants to support router-mediated swaps must allowlist the router address. Once `allowedSwapper[pool][router] = true`, the check passes for **every caller** of the router, regardless of whether that caller is individually allowlisted. The allowlist is completely bypassed.

Note the asymmetry with `DepositAllowlistExtension`, which correctly checks the `owner` parameter (the LP position owner supplied by the caller), not `sender`: [5](#0-4) 

### Impact Explanation

Any user can trade on a pool whose admin intended to restrict swaps to a curated set of addresses. The pool admin has no way to simultaneously (a) allow router-mediated swaps and (b) enforce per-user restrictions: allowlisting the router opens the gate to everyone; not allowlisting it blocks even legitimately allowlisted users from using the router. Unauthorized trading on a curated pool can cause direct LP losses through adversarial order flow the pool was not designed to absorb.

### Likelihood Explanation

High. `MetricOmmSimpleRouter` is the standard periphery entry point. Any pool admin who deploys a curated pool and then allowlists the router to support normal UX inadvertently enables the bypass. The attacker needs only to call `router.exactInputSingle` — no special privileges, no malicious setup.

### Recommendation

The extension must gate on the **economic actor**, not the intermediary. Two options:

1. **Pass the payer/user through the pool**: Add an explicit `payer` or `user` field to the `swap` signature (or to `extensionData`) so the router can forward the actual end user's address, and update `SwapAllowlistExtension` to check that field.
2. **Mirror the deposit pattern**: Require callers to supply the swapper identity as a parameter (analogous to `owner` in `addLiquidity`), and check that parameter in the extension rather than `sender`.

### Proof of Concept

```solidity
// Pool admin sets up curated pool
SwapAllowlistExtension ext = new SwapAllowlistExtension(address(factory));
// Admin allowlists the router so legitimate users can swap via UI
ext.setAllowedToSwap(pool, address(router), true);
// Alice (not individually allowlisted) calls the router
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    ...
    recipient: alice
}));
// pool.swap is called with msg.sender = router
// extension checks allowedSwapper[pool][router] == true → passes
// Alice successfully swaps despite not being in the allowlist

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
