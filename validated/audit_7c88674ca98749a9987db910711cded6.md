### Title
Zero-Confirmation Bypass in `verify_transaction_inclusion` — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` accepts a caller-supplied `confirmations: u64` with no minimum enforcement. Passing `confirmations = 0` causes both confirmation guards to evaluate trivially true, allowing any transaction in any canonical mainchain block — including the unconfirmed tip — to pass as "verified."

### Finding Description

Two guards in `verify_transaction_inclusion` are both bypassed when `args.confirmations == 0`:

**Guard 1** — gc_threshold upper-bound check: [1](#0-0) 

`0 <= gc_threshold` is always true for any `u64` value of `gc_threshold`.

**Guard 2** — depth check: [2](#0-1) 

The expression `(heaviest_block_header.block_height).saturating_sub(target_block_height) + 1 >= args.confirmations` reduces to `(some_u64) >= 0`, which is always true in Rust for `u64`. When the target block IS the tip, `saturating_sub` returns `0`, so the left side is `1`, and `1 >= 0` passes.

After both guards pass, the function proceeds directly to Merkle proof verification: [3](#0-2) 

If the caller supplies a valid Merkle proof for a transaction in the tip block, the function returns `true` — with zero blocks built on top of it.

The `ProofArgs` struct has no minimum enforcement on `confirmations`: [4](#0-3) 

The function is publicly callable with no access control (`#[pause]` only gates it when paused, not by caller identity): [5](#0-4) 

Note: `verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase check, so it inherits the same bypass: [6](#0-5) 

### Impact Explanation

An unprivileged caller can obtain a `true` inclusion proof for a transaction in the current chain tip with zero subsequent confirmations. That tip block is maximally vulnerable to reorganization — it has no accumulated work on top of it. Any downstream system (bridge, DEX, payment processor) that calls this function and gates an action on the `true` result can be tricked into releasing funds or accepting a payment for a transaction that is subsequently erased by a reorg.

### Likelihood Explanation

The attack path is fully permissionless: no privileged role, no PoW forgery, no key compromise required. The attacker only needs to:
1. Submit a valid block (standard relayer path, PoW already verified by the contract).
2. Call `verify_transaction_inclusion` with `confirmations = 0` and a valid Merkle proof for a transaction in that block.
3. Trigger the downstream action gated on the `true` result.
4. Mine a competing chain of greater work to reorg the tip away.

The existing test suite already demonstrates `confirmations = 0` is accepted without error — it only returns `false` because the test uses an invalid Merkle proof (default hashes), not because the confirmation check rejects it: [7](#0-6) 

### Recommendation

Enforce a minimum confirmation count of at least 1 at the contract level. Add a `require!(args.confirmations >= 1, "confirmations must be at least 1")` guard before the existing gc_threshold check in both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2`. This ensures the depth expression `depth + 1 >= confirmations` can never be trivially satisfied by a zero right-hand side.

### Proof of Concept

```rust
// After init_contract() and submit_blocks([valid_block_with_tx]):
// The submitted block is the chain tip. Call verify_transaction_inclusion
// with confirmations=0 and a valid Merkle proof for a tx in that block.
// Expected (broken) result: returns true.
// Expected (correct) result: should panic/return false — 0 confirmations
// means the block could be reorganized away.

let result: bool = user_account
    .view(contract.id(), "verify_transaction_inclusion")
    .args_borsh(ProofArgs {
        tx_id: real_tx_hash,
        tx_block_blockhash: tip_block.block_hash(),
        tx_index: 0,
        merkle_proof: valid_merkle_proof,
        confirmations: 0,   // <-- bypass
    })
    .await?
    .json()?;

assert!(result); // passes — tip block, zero depth, zero confirmations required
```
