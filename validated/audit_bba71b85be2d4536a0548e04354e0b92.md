### Title
`ENABLE_SECP_OPS` Flag Not Checked for 4-Byte Secp Opcode Forms, Allowing Gate Bypass — (File: `src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` via two separate code paths: a 1-byte opcode path (opcodes 64/65) that correctly gates on `ClvmFlags::ENABLE_SECP_OPS`, and a 4-byte opcode path (opcodes `0x13d61f00`/`0x1c3a8f00`) that invokes the same functions **without checking `ENABLE_SECP_OPS` at all**. An attacker-controlled CLVM program can use the 4-byte encoding to execute secp signature verification regardless of whether the flag is set, bypassing the intended activation gate.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` has two distinct dispatch branches for secp operations.

**Branch 1 — 4-byte opcodes (lines 157–183):** When the opcode atom is exactly 4 bytes, the code matches two specific values and calls the secp functions directly, with no flag check:

```rust
// op_len == 4
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
``` [1](#0-0) 

**Branch 2 — 1-byte opcodes (lines 248–249):** When the opcode atom is 1 byte, opcodes 64 and 65 are gated behind `ENABLE_SECP_OPS`:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The flag's own documentation states it "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)" — but both 1-byte and 4-byte forms call the identical underlying functions `op_secp256k1_verify` / `op_secp256r1_verify` in `src/secp_ops.rs`. [3](#0-2) [4](#0-3) 

The `ENABLE_SECP_OPS` flag is set in one place (the 1-byte dispatch) but never checked in the other (the 4-byte dispatch) — a direct structural analog to the `isLpCreated` pattern in the reference report.

---

### Impact Explanation

**Impact: Medium**

`ENABLE_SECP_OPS` is a hardfork-style activation flag. When it is absent (e.g., in `MEMPOOL_MODE`, which does not include `ENABLE_SECP_OPS`), secp operations via 1-byte opcodes are correctly rejected. However, the same secp operations remain fully reachable via the 4-byte opcode encoding. This means:

1. **Gate bypass**: Any caller that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` (including mempool mode) intends to disallow secp verification. An attacker-supplied CLVM program using `0x13d61f00`/`0x1c3a8f00` bypasses this intent and executes real ECDSA verification.

2. **Consensus/mempool divergence**: A node enforcing `ENABLE_SECP_OPS` as a pre-activation guard will reject 1-byte secp opcodes but silently accept 4-byte secp opcodes, producing different execution outcomes than a node that treats both forms uniformly. [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Medium**

The 4-byte opcode values (`0x13d61f00`, `0x1c3a8f00`) are documented in code comments within the same file. Any party reading the source can construct a CLVM program using these byte sequences. The attacker-controlled entry path is direct: supply a CLVM program atom with the 4-byte opcode as the operator, followed by valid pubkey/message/signature arguments. No special privileges or side-channel access are required.

---

### Recommendation

Add the `ENABLE_SECP_OPS` check to the 4-byte opcode dispatch branch, mirroring the guard already present for the 1-byte forms:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both opcode encodings are subject to the same activation gate, eliminating the asymmetry.

---

### Proof of Concept

A CLVM program using the 4-byte opcode `0x13d61f00` as the operator, with valid secp256k1 pubkey, message digest, and signature as arguments, will successfully execute `op_secp256k1_verify` even when `ChiaDialect` is constructed with `ClvmFlags::empty()` (no `ENABLE_SECP_OPS`). The same program using 1-byte opcode `64` would fall through to `unknown_operator` and return a no-op or error depending on `NO_UNKNOWN_OPS`, demonstrating the inconsistency. The attacker-controlled bytes are the opcode atom and the three argument atoms passed to `run_program`. [6](#0-5) [7](#0-6)
