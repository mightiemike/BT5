### Title
Negative-PnL Settlement Inflates `availableSettle` and Drains Subaccount Spot Quote — (`core/contracts/PerpEngine.sol`)

### Summary

`PerpEngine.getSettlementState` returns `canSettle = min(positionPnl, state.availableSettle)` with no guard against a negative result. When `positionPnl < 0` and `state.availableSettle > 0`, `canSettle` is negative. The subsequent arithmetic in `settlePnl` then *increases* `state.availableSettle` and *decreases* the subaccount's spot quote balance, violating the invariant that settlement must only transfer quote from the pool to subaccounts with positive PnL.

### Finding Description

**`MathHelper.min` is a plain signed comparison with no floor at zero:** [1](#0-0) 

**`getSettlementState` passes the raw signed result directly as `canSettle`:** [2](#0-1) 

**`settlePnl` applies `canSettle` without checking its sign:** [3](#0-2) 

When `canSettle < 0`:
- `state.availableSettle -= canSettle` → **increases** `availableSettle` (subtracting a negative)
- `balance.vQuoteBalance -= canSettle` → **increases** the perp vQuote balance (loss reduced)
- `totalSettled += canSettle` → **negative** return value

**`Clearinghouse._settlePnl` then applies the negative `amountSettled` to the subaccount's spot quote:** [4](#0-3) 

This decreases the subaccount's real USDC balance.

### Impact Explanation

Two simultaneous accounting corruptions occur:

1. **`state.availableSettle` is inflated.** The pool that backs positive-PnL settlements now reports more capacity than it actually has. Every subsequent legitimate settler can drain quote that does not exist, creating **protocol bad debt**.

2. **Subaccount spot quote is decreased without a corresponding position change.** The subaccount's perp `vQuoteBalance` increases by the same magnitude, so the subaccount's *net value* is unchanged — but real USDC has been moved out of the subaccount's spot balance into the inflated pool, effectively transferring value from the victim to future settlers.

This matches the scoped critical impact: *unauthorized loss of funds and cross-contract state desynchronization between PerpEngine and SpotEngine*.

### Likelihood Explanation

The `SettlePnl` transaction type is not restricted to the sequencer. It is not in the owner-only list in `submitSlowModeTransactionImpl`: [5](#0-4) 

Any user can submit a `SettlePnl` slow-mode transaction for their own subaccount (or any subaccount) by paying the slow-mode fee. After the 3-day delay, it executes. A trader with a losing perp position and a positive `state.availableSettle` can trigger this path deterministically.

### Recommendation

Add a zero-floor guard in `getSettlementState` so `canSettle` is never negative:

```solidity
availableSettle = MathHelper.max(
    0,
    MathHelper.min(
        calculatePositionPnl(balance, productId),
        state.availableSettle
    )
);
```

Alternatively, add an explicit early-return in `settlePnl` when `canSettle <= 0`.

### Proof of Concept

```
Setup:
  productId = 2 (even, perp)
  state.availableSettle = +100e18
  balance.amount = -10e18  (short position)
  balance.vQuoteBalance = +50e18
  priceX18 = 20e18

Step 1: positionPnl = 20e18 * (-10e18) + 50e18 = -150e18
Step 2: canSettle = min(-150e18, +100e18) = -150e18
Step 3: state.availableSettle -= -150e18  →  availableSettle = +250e18  (inflated by 150)
Step 4: balance.vQuoteBalance -= -150e18  →  vQuoteBalance = +200e18
Step 5: totalSettled = -150e18
Step 6: Clearinghouse calls spotEngine.updateBalance(QUOTE_PRODUCT_ID, subaccount, -150e18)
        → subaccount loses 150 USDC from spot balance

Assert: state.availableSettle increased (250 > 100) — invariant broken.
Assert: totalSettled < 0 — invariant broken.
```

The fuzz condition is: any `(positionPnl < 0, availableSettle > 0)` pair reaches this path through the slow-mode `SettlePnl` flow.
