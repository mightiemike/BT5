### Title
SwapAllowlistExtension Gates Router Address Instead of Actual Swapper, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` — the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool sees the router as `msg.sender`, so the extension checks `allowedSwapper[pool][router]` instead of the actual user's address. If the pool admin allowlists the router to enable router-mediated swaps, any user — including one whose individual allowlist entry was revoked — can bypass the per-user swap gate by routing through the router.

---

### Finding Description

`SwapAllowlistExtension` is documented as gating `swap` **by swapper address, per pool**: [1](#0-0) 

The hook receives `sender` as its first argument and checks it against `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the caller of the extension) and `sender` is whatever the pool passed.

The pool's `swap` function passes `msg.sender` as `sender`: [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards that value unchanged: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly: [4](#0-3) 

So `msg.sender` inside the pool is the **router address**, not the originating user. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The same wrong-actor binding applies to `exactOutputSingle`, `exactInput`, and `exactOutput` multi-hop paths — in every case the router is the direct caller of `pool.swap()`.

**The replay-analog scenario (matching the KYCRegistry bug class):**

1. Pool admin deploys a curated pool with `SwapAllowlistExtension`.
2. Pool admin allowlists user A and also allowlists the router (so allowlisted users can trade via the router).
3. Pool admin revokes user A's individual entry: `setAllowedToSwap(pool, userA, false)`.
4. User A calls `router.exactInputSingle(...)` — the extension checks `allowedSwapper[pool][router]` → `true` → swap succeeds.

The revocation is ineffective as long as the router remains allowlisted, exactly mirroring the KYCRegistry pattern where a revoked status could be bypassed through a still-valid authorization path.

---

### Impact Explanation

Any user whose individual swap permission has been revoked can continue trading on a curated pool by routing through `MetricOmmSimpleRouter`. The allowlist invariant — "only explicitly permitted addresses may swap" — is broken for every pool that allowlists the router. Depending on the pool's purpose (e.g., compliance-gated, institutional-only, or risk-controlled pools), this allows unauthorized principals to move funds through the pool, defeating the entire purpose of the extension.

---

### Likelihood Explanation

Medium. The bypass requires the pool admin to have allowlisted the router. A pool admin who wants their allowlisted users to be able to trade via the router has no other option than to allowlist the router address — there is no mechanism to pass the originating user's identity through the router to the extension. The admin may not realize that allowlisting the router grants access to **all** router users, not just the individually allowlisted ones. Once the router is allowlisted, the bypass is trivially reachable by any user with no further preconditions.

---

### Recommendation

The extension must check the economically relevant actor — the originating user — not the intermediary contract. Two viable approaches:

1. **Extension-data forwarding**: Require the router to encode the originating user's address in `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and check that address instead of (or in addition to) `sender`. The pool admin would configure the extension to trust the router as a forwarding agent only.

2. **Separate router-aware allowlist**: Introduce a second allowlist tier keyed by `(pool, router, user)` so that the extension can verify both that the router is trusted and that the underlying user is permitted, using the decoded address from `extensionData`.

The `DepositAllowlistExtension` does not share this flaw because it checks `owner` (the position owner supplied by the caller), which is the economically relevant actor for deposits regardless of which periphery contract initiates the call. [5](#0-4)
