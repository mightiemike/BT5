Audit Report

## Title
`DepositAllowlistExtension` Validates Caller-Controlled `owner` Instead of Actual Depositor `sender`, Enabling Full Allowlist Bypass — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual depositor, bound to `msg.sender` in `MetricOmmPool.addLiquidity`) and instead validates the caller-supplied `owner` parameter. Because `owner` is a free argument accepted from any caller, any address excluded from the allowlist can bypass the deposit gate by supplying any allowlisted address as `owner`, providing funds via the callback, and receiving the LP position at that address.

## Finding Description
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` from the caller and passes `msg.sender` as `sender` to the extension hook:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both values:

```solidity
// ExtensionCalling.sol lines 95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

Inside `DepositAllowlistExtension`, the first positional argument (`sender`) is unnamed and discarded; only `owner` is checked:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`owner` is entirely attacker-controlled. The correct pattern is demonstrated by `SwapAllowlistExtension.beforeSwap`, which binds the check to `sender` (the actual swapper) and discards `recipient`:

```solidity
// SwapAllowlistExtension.sol lines 31-41
function beforeSwap(address sender, address, ...)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
```

The asymmetry confirms the deposit extension has the wrong parameter bound. No existing guard compensates: `addLiquidity` imposes no restriction on who may call it or what `owner` value is supplied.

## Impact Explanation
The deposit allowlist is the pool admin's mechanism to restrict who may provide liquidity to a permissioned pool. With this bug the restriction is completely ineffective. An address not on the allowlist calls `addLiquidity(owner=allowlistedAddress, ...)`, the extension approves because `allowedDepositor[pool][allowlistedAddress]` is `true`, the unauthorized address provides tokens via the callback, and LP shares are minted to `allowlistedAddress`. If `allowlistedAddress` is the attacker's own second account, the attacker later calls `removeLiquidity` (which enforces `msg.sender == owner`) to recover all funds — a complete allowlist bypass with no fund loss to the attacker. This breaks the pool admin's access-control invariant and constitutes an admin-boundary break by an unprivileged actor.

## Likelihood Explanation
Exploitation requires only a single direct call to `pool.addLiquidity` with `owner` set to any address appearing in `allowedDepositor[pool]`. No special privileges, flash loans, or complex setup are needed. The `allowedDepositor` mapping is publicly readable, so any attacker can enumerate valid `owner` values. The attacker needs only to control a second address that is on the allowlist, or cooperate with one. Likelihood is **High**.

## Recommendation
Bind the check to `sender` (the actual depositor) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup
─────
1. Pool deployed with DepositAllowlistExtension as a beforeAddLiquidity hook.
2. Pool admin calls setAllowedToDeposit(pool, alice, true).
   → allowedDepositor[pool][alice] = true
3. bob (controls alice as a second EOA) is NOT on the allowlist.

Attack
──────
4. bob calls pool.addLiquidity(
       owner        = alice,   // allowlisted address bob controls
       salt         = 0,
       deltas       = <bins bob wants to fill>,
       callbackData = "",
       extensionData= ""
   )

5. Pool calls _beforeAddLiquidity(msg.sender=bob, owner=alice, ...).

6. Extension receives (sender=bob [discarded], owner=alice).
   Check: allowedDepositor[pool][alice] == true → PASSES.

7. LiquidityLib.addLiquidity runs; bob's metricOmmSwapCallback is invoked;
   bob transfers tokens into the pool.

8. LP shares are minted to alice (owner).

9. alice calls pool.removeLiquidity(alice, 0, deltas, "")
   → msg.sender == owner check passes (MetricOmmPool.sol line 206)
   → alice recovers all tokens.

Net result: bob deposited into a pool he was explicitly barred from,
with zero fund loss to the attacker.
