### Title
CREATE2 Address Collision via User-Controlled `subaccount` Constructor Argument Enables Draining of `DirectDepositV1` Contracts — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.createDirectDepositV1` deploys `DirectDepositV1` via `new DirectDepositV1{salt: bytes32(uint256(1))}(...)` with a caller-supplied `bytes32 subaccount` constructor argument. Because `subaccount` is part of the initcode, it controls the resulting CREATE2 address. An attacker can brute-force `subaccount` values to find a collision with an attacker-controlled wallet, pre-set an ERC20 allowance at that address before the `DirectDepositV1` is deployed, and then drain all tokens deposited into it.

---

### Finding Description

`createDirectDepositV1` is `public` with no access control and accepts an arbitrary `bytes32 subaccount`:

```solidity
// ContractOwner.sol lines 486–500
function createDirectDepositV1(bytes32 subaccount)
    public
    returns (address payable)
{
    require(
        getDirectDepositV1BytecodeHash() ==
            0x7974df41bdca2be1539fa7d01f41277f0d728823b20230a18a31e40c707874e7,
        "dda hash"
    );
    DirectDepositV1 directDepositV1 = new DirectDepositV1{
        salt: bytes32(uint256(1))
    }(address(endpoint), address(spotEngine), subaccount, wrappedNative);
    directDepositV1Address[subaccount] = payable(directDepositV1);
    return payable(directDepositV1);
}
``` [1](#0-0) 

The CREATE2 address is determined by:

```
keccak256(0xff ++ address(ContractOwner) ++ bytes32(1) ++ keccak256(initcode))
```

where `initcode` = `DirectDepositV1` bytecode + `abi.encode(endpoint, spotEngine, subaccount, wrappedNative)`.

Since `subaccount` is a full `bytes32` caller-supplied value embedded in the initcode, the attacker has 256 bits of entropy to vary the initcode hash, and thus the resulting deployment address. This is equivalent in power to controlling the salt directly.

`creditDepositV1` is also `external` with no access control and will call `createDirectDepositV1` with any caller-supplied `subaccount`:

```solidity
// ContractOwner.sol lines 502–508
function creditDepositV1(bytes32 subaccount) external {
    address payable directDepositV1 = directDepositV1Address[subaccount];
    if (directDepositV1 == address(0)) {
        directDepositV1 = createDirectDepositV1(subaccount);
    }
    DirectDepositV1(directDepositV1).creditDeposit();
}
``` [2](#0-1) 

The `DirectDepositV1` contract holds ERC20 tokens temporarily before they are forwarded to the endpoint via `creditDeposit()`:

```solidity
// DirectDepositV1.sol lines 83–101
function creditDeposit() external {
    ...
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
    ...
}
``` [3](#0-2) 

**Attack steps:**

**Step 1 — Find a collision (off-chain):**
Using a meet-in-the-middle technique over `2^80` iterations:
- Brute-force `subaccount` values → compute resulting `DirectDepositV1` CREATE2 addresses → store in a Bloom filter.
- Brute-force attacker-controlled CREATE2 wallet salts → find any address in the stored set.

**Step 2 — First transaction (pre-collision setup):**
- Deploy the attacker's contract at `0xCOLLIDED`.
- Call `token.approve(attackerWallet, type(uint256).max)` for all relevant tokens.
- `selfdestruct` the contract in the same transaction (valid post-Dencun per EIP-6780 when created in the same tx).

**Step 3 — Second transaction (deploy the real DDA):**
- Call `createDirectDepositV1(collidedSubaccount)` or `creditDepositV1(collidedSubaccount)`.
- `DirectDepositV1` is now deployed at `0xCOLLIDED` with the pre-set allowance intact.

**Step 4 — Drain:**
- Wait for users to send tokens to `0xCOLLIDED` (or for `wrapVaultAsset` / `creditDepositV1` flows to route funds there).
- Call `transferFrom(0xCOLLIDED, attackerWallet, balance)` to drain all tokens.

The bytecode hash check (`getDirectDepositV1BytecodeHash() == 0x7974...`) does not mitigate this — it only ensures the `DirectDepositV1` bytecode is unmodified, which the attacker does not need to change. [4](#0-3) 

---

### Impact Explanation

Any `DirectDepositV1` contract deployed for a collided `subaccount` will have a pre-set ERC20 allowance to the attacker. All tokens deposited into it — by users sending tokens directly, by `wrapVaultAsset`, or by any other flow that routes funds to the DDA address — can be drained via `transferFrom` before or after `creditDeposit` is called. The attacker can choose the target `subaccount` freely and wait until sufficient funds accumulate.

---

### Likelihood Explanation

The attack requires approximately `2^80` hash operations off-chain, which is feasible given modern GPU/ASIC hardware (the BTC network alone exceeds `6.5 × 10^20` hashes/second). The attacker can pre-compute the collision at leisure, then execute the two on-chain transactions at any time. No privileged access is required — `createDirectDepositV1` and `creditDepositV1` are both unrestricted public/external functions.

---

### Recommendation

1. **Remove user control over the initcode entropy.** Do not accept an arbitrary `bytes32 subaccount` as a constructor argument that determines the CREATE2 address. Instead, derive the salt from protocol-controlled parameters (e.g., `keccak256(abi.encode(block.timestamp, block.number, subaccount, nonce))`).
2. **Add a nonce or timestamp to the salt** so that even if a collision is found, it is only valid at a specific block, making execution impractical without sequencer collusion.
3. **Restrict `createDirectDepositV1` to internal or owner-only** so that arbitrary `subaccount` values cannot be submitted by unprivileged callers.

---

### Proof of Concept

The PoC from the referenced report applies directly. Replace `PanopticFactory.deployNewPool` with `ContractOwner.createDirectDepositV1(subaccount)` and the collision target with the resulting `DirectDepositV1` address. The two EVM properties demonstrated — redeployment on a previously self-destructed address, and persistent allowances surviving `selfdestruct` — are identical and confirmed on Mainnet-fork environments.
