### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, making the allowlist bypassable or unusable for router-mediated swaps — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool. The pool always passes `msg.sender` (the direct caller of `pool.swap()`) as `sender`. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router, so the extension checks whether the **router** is allowlisted — not the actual end user. This creates an irreconcilable wrong-actor binding: either the router is allowlisted (any user bypasses the curated allowlist) or it is not (allowlisted users cannot use the router at all).

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap()`, the pool calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`_beforeSwap` in `ExtensionCalling` forwards that value verbatim as the `sender` argument to every configured extension: [2](#0-1) 

**Step 2 — Router is `msg.sender` at the pool.**

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()` directly. The pool therefore sees `msg.sender == router`: [3](#0-2) 

**Step 3 — Extension checks the router address, not the user.**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router: [4](#0-3) 

The actual end user's address is never visible to the extension.

---

### Impact Explanation

Two mutually exclusive failure modes exist for any pool that deploys `SwapAllowlistExtension`:

| Admin configuration | Outcome |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot swap through the router; they must implement `IMetricOmmSwapCallback` themselves — the standard periphery path is broken. |
| Router **allowlisted** | Every user on the network can bypass the curated allowlist by routing through `MetricOmmSimpleRouter`; the allowlist provides zero protection. |

In the second case — which is the natural fix an admin would reach for — the allowlist is completely nullified. Any non-allowlisted user can trade against a pool that was explicitly configured to restrict access (e.g., a KYC-gated or institutional pool), receiving tokens from the pool's reserves at oracle-derived prices. This is a direct loss of the pool's curation guarantee and constitutes broken core pool functionality with fund-impacting consequences.

---

### Likelihood Explanation

- `SwapAllowlistExtension` is a first-class production extension in the periphery, not a test mock.
- Any pool admin who deploys it and wants users to access the pool through the standard router will inevitably allowlist the router to unblock the UX, triggering the bypass.
- The attacker requires no special privilege: calling `exactInputSingle` on the router is a standard public action.
- No existing guard in the extension or the pool prevents this; the `onlyPool` modifier in `BaseMetricExtension` only verifies the pool is the caller, not the identity of the underlying user. [5](#0-4) 

---

### Recommendation

**Short term:** The pool must pass the economically relevant actor — the end user — to the extension, not the immediate `msg.sender`. One approach: the router encodes the real user's address into `extensionData`, and `SwapAllowlistExtension.beforeSwap` decodes and checks that address when `sender` is a known router. A simpler approach is for the pool to expose a separate `swapFor(address realUser, ...)` entry point that the router calls, passing the real user explicitly.

**Long term:** Never derive access-control identity from the immediate `msg.sender` of a multi-hop call chain. The extension framework should define a canonical "originating user" field that the router is required to populate and that the pool authenticates before forwarding to extensions — analogous to how `ERC-2771` meta-transactions handle trusted forwarders.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup:
// 1. Deploy pool with SwapAllowlistExtension.
// 2. Admin allowlists only `alice` for swaps:
//    swapExt.setAllowedToSwap(pool, alice, true);
// 3. Admin also allowlists the router so alice can use the standard UX:
//    swapExt.setAllowedToSwap(pool, address(router), true);

// Attack:
// 4. Eve (not allowlisted) calls the router directly:
router.exactInputSingle(ExactInputSingleParams({
    pool:           pool,
    tokenIn:        token0,
    recipient:      eve,
    zeroForOne:     true,
    amountIn:       1_000e18,
    amountOutMinimum: 0,
    priceLimitX64:  0,
    deadline:       block.timestamp,
    extensionData:  ""
}));
// Result: pool.swap() is called with msg.sender == router.
// Extension checks allowedSwapper[pool][router] == true → passes.
// Eve receives token1 from the curated pool despite never being allowlisted.
```

The `sender` the extension sees is the router address, not Eve's address, so the allowlist check passes and Eve's swap executes against the restricted pool.
