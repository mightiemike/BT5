### Title
`From<ResourceBoundsMapping> for ValidResourceBounds` Selects `L1Gas` Variant for V3 Transactions with Zero L2/L1-Data Gas, Producing a Different Transaction Hash in RPC Simulation vs. Gateway Admission — (`File: crates/apollo_rpc/src/v0_8/transaction.rs`)

---

### Summary

The RPC simulation/estimation path converts a V3 transaction's `ResourceBoundsMapping` to `ValidResourceBounds::L1Gas` whenever `l1_data_gas == 0 && l2_gas == 0`. The gateway admission path always uses `ValidResourceBounds::AllResources` for the same data. Because `get_tip_resource_bounds_hash` produces structurally different Poseidon hashes for these two variants — even when the numeric values are identical — the transaction hash computed during RPC simulation diverges from the hash computed and stored at the gateway. Signature verification inside the account contract during simulation then fails against the wrong hash, causing `starknet_simulateTransactions` / `starknet_estimateFee` to return an authoritative-looking failure for a transaction that is actually valid and will execute successfully.

---

### Finding Description

**Divergent variant selection**

`From<ResourceBoundsMapping> for ValidResourceBounds` in the RPC layer:

```rust
// crates/apollo_rpc/src/v0_8/transaction.rs  lines 188-199
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← simulation path
        } else {
            Self::AllResources(AllResourceBounds { ... })
        }
    }
}
``` [1](#0-0) 

The gateway admission path uses `RpcInvokeTransactionV3` / `InternalRpcInvokeTransactionV3`, both of which carry `resource_bounds: AllResourceBounds` and always return `ValidResourceBounds::AllResources(...)` from their `resource_bounds()` trait method:

```rust
// crates/starknet_api/src/rpc_transaction.rs  lines 636-638
impl InvokeTransactionV3Trait for InternalRpcInvokeTransactionV3 {
    fn resource_bounds(&self) -> ValidResourceBounds {
        ValidResourceBounds::AllResources(self.resource_bounds)  // always AllResources
    }
``` [2](#0-1) 

**Hash divergence**

`get_tip_resource_bounds_hash` branches on the variant:

```rust
// crates/starknet_api/src/transaction_hash.rs  lines 188-211
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 resources hashed
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 resources hashed
    }
});
Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
``` [3](#0-2) 

For a V3 transaction with `l2_gas = 0`, `l1_data_gas = 0`:

| Path | Variant | Elements hashed | Hash |
|---|---|---|---|
| Gateway (admission) | `AllResources` | `tip, L1_GAS, L2_GAS(0), L1_DATA_GAS(0)` | **H₁** |
| RPC simulation | `L1Gas` | `tip, L1_GAS, L2_GAS(0)` | **H₂** |

`poseidon(tip, L1_GAS, L2_GAS, L1_DATA_GAS=0) ≠ poseidon(tip, L1_GAS, L2_GAS)`, so **H₁ ≠ H₂**.

The same inconsistency exists in the protobuf deserialization path used for P2P mempool propagation:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  lines 431-435
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [4](#0-3) 

A transaction that enters the mempool as `AllResources` (hash H₁) is serialized to protobuf with `l1_data_gas = 0`, then deserialized back as `L1Gas` (hash H₂) on a peer node, breaking the hash invariant across the P2P boundary.

---

### Impact Explanation

**RPC simulation / fee estimation returns wrong value (High):** When a user calls `starknet_simulateTransactions` or `starknet_estimateFee` for a V3 transaction with zero L2 gas and zero L1 data gas without `SKIP_VALIDATE`, the sequencer computes H₂ internally and passes it to the account contract's `__validate__` entry point. The account verifies the user's ECDSA/Stark signature against H₂, but the signature was produced over H₁ (the hash the gateway stored). Verification fails, and the RPC returns an authoritative-looking `REVERTED` or error result for a transaction that will actually succeed on-chain.

**Mempool P2P admission rejects valid transactions (High):** A peer node that receives a propagated transaction via protobuf recomputes the hash as H₂ ≠ H₁. Any hash-equality check against the stored/announced hash causes the transaction to be silently dropped or rejected before sequencing.

---

### Likelihood Explanation

V3 transactions with zero L2 gas and zero L1 data gas are explicitly accepted by the gateway's stateless validator (test case `valid_l1_gas` in `crates/apollo_gateway/src/stateless_transaction_validator_test.rs` sets only `l1_gas` non-zero). Any unprivileged user can submit such a transaction. The divergence is deterministic and reproducible for every such transaction. [5](#0-4) 

---

### Recommendation

Remove the `L1Gas` branch from `From<ResourceBoundsMapping> for ValidResourceBounds` and from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. V3 transactions must always be represented as `AllResources`, regardless of whether individual bounds are zero. The `L1Gas` variant should only be constructed for pre-0.13.3 (V1/V2) transactions that structurally lack an L1-data-gas field. Alternatively, gate the `L1Gas` shortcut on the transaction version rather than on the numeric values of the bounds.

---

### Proof of Concept

1. Construct a V3 invoke transaction: `l1_gas = { max_amount: 1000, max_price: 1 }`, `l2_gas = { 0, 0 }`, `l1_data_gas = { 0, 0 }`.
2. Compute H₁ = `get_invoke_transaction_v3_hash(tx, chain_id, version)` with `ValidResourceBounds::AllResources(...)` — this is what the gateway computes and stores.
3. Sign H₁ with the account's private key.
4. Submit to the gateway → accepted, stored with hash H₁.
5. Call `starknet_simulateTransactions` with the same transaction body (no `SKIP_VALIDATE`).
6. The RPC converts `ResourceBoundsMapping { l1_data_gas: 0, l2_gas: 0 }` → `ValidResourceBounds::L1Gas(...)`.
7. Internally computes H₂ = `get_invoke_transaction_v3_hash(tx, chain_id, version)` with `L1Gas` variant — H₂ ≠ H₁.
8. Account contract's `__validate__` receives H₂, verifies signature against H₂ → fails.
9. `starknet_simulateTransactions` returns `REVERTED` for a transaction that will succeed on-chain.
