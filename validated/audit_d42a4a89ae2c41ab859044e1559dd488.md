Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` argument forwarded by the pool, which equals the `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` is the router's address, not the end user's. A pool admin who allowlists the router to enable legitimate router-mediated swaps inadvertently grants swap access to every user, because the extension cannot distinguish which end user is behind the router call.

## Finding Description

`SwapAllowlistExtension.beforeSwap()` performs its identity check as:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (the contract calling the extension), and `sender` is the value the pool forwarded — which is the `msg.sender` of `pool.swap()`. [1](#0-0) 

In `MetricOmmPool.swap()`, the pool passes its own `msg.sender` as the first argument to `_beforeSwap`: [2](#0-1) 

When a user calls `router.exactInputSingle()`, the router calls `pool.swap()` directly with no forwarding of the original caller: [3](#0-2) 

So the extension receives `sender = address(router)`, not the end user's address. The allowlist lookup becomes `allowedSwapper[pool][router]`.

**Bypass path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists `user1`, `user2` (trusted traders).
2. To let `user1`/`user2` use the router, the admin must also allowlist `address(router)`.
3. Once the router is allowlisted, any address — including `user3` who is not on the allowlist — can call `router.exactInputSingle()` and the extension will pass, because it only sees `sender = router`.

The pool admin has no way to selectively allow specific users to use the router while blocking others; it is all-or-nothing. The same issue applies to `exactInput`, `exactOutputSingle`, and `exactOutput` router entry points. [4](#0-3) 

## Impact Explanation

A curated pool's swap allowlist is completely bypassed for any user who routes through `MetricOmmSimpleRouter`. Unauthorized traders can execute swaps on pools intended to be restricted to specific counterparties. This breaks the core curation invariant and exposes LP funds to trades from actors the pool admin explicitly excluded. The impact is direct loss of LP value if the allowlist was protecting against adversarial or uninformed order flow — a broken core pool functionality causing potential loss of funds.

## Likelihood Explanation

Any pool that (a) uses `SwapAllowlistExtension` and (b) wants to support router-mediated swaps for its allowlisted users is vulnerable. The pool admin must allowlist the router to enable router usage, and doing so silently removes all per-user enforcement. This is a routine configuration that any operator of a curated pool would reach. No special attacker capability is required beyond calling the public router.

## Recommendation

The extension should receive and check the original end-user address, not the immediate caller of `pool.swap()`. Options:

1. Pass the end user's address through `extensionData` and verify it in the extension (with the router signing or forwarding it).
2. Add a dedicated `swapper` field to the `beforeSwap` hook arguments that the pool populates from a trusted source (e.g., transient storage set by the router before calling the pool), so the extension can always gate the true economic actor regardless of intermediary.
3. Alternatively, document clearly that allowlisting the router grants access to all users, and provide a separate router that forwards the original `msg.sender` as a verified parameter.

## Proof of Concept

```
Pool admin:
  allowedSwapper[pool][user1] = true
  allowedSwapper[pool][router] = true   // needed for user1 to use router

Attacker (user3, not allowlisted):
  router.exactInputSingle({
      pool: pool,
      tokenIn: token0,
      ...
  })
  // router calls pool.swap() → msg.sender = router
  // pool calls _beforeSwap(router, ...)
  // extension checks allowedSwapper[pool][router] → true
  // swap executes despite user3 not being allowlisted
```

A Foundry integration test can confirm this by: deploying a pool with `SwapAllowlistExtension`, allowlisting `user1` and `router`, then calling `router.exactInputSingle()` from an unallowlisted `user3` address and asserting the swap succeeds (no `NotAllowedToSwap` revert).

    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
      if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

      amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    }

    if (amount <= 0) revert InvalidSwapDeltas();
    amountOut = MetricOmmSwapInputs.int128ToUint128(amount);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
