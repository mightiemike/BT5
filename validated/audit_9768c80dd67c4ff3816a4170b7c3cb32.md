### Title
Bonus Reward Checkpoints Permanently Advanced Without Payment During Redelegation Slash When Pool Is Insufficient — (File: `x/tieredrewards/keeper/slash.go`)

---

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` fails with `ErrInsufficientBonusPool`, the error is deliberately swallowed to prevent a chain halt. However, `processEventsAndClaimBonus` mutates the `pos` pointer in-place — advancing `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded` — **before** the pool-balance check. Because the error is swallowed and `setPositionWithState` is subsequently called with the already-mutated `pos`, those advanced checkpoints are permanently committed to state without the corresponding bonus ever being paid. The accrued bonus for the period up to the slash is irrecoverably lost.

---

### Finding Description

`processEventsAndClaimBonus` mutates the `pos *types.PositionState` pointer throughout its execution: [1](#0-0) 

Inside the event loop, `pos.UpdateLastEventSeq(entry.Seq)` is called and `decrementEventRefCount` writes to the KV store — both before the pool-balance check. After the loop: [2](#0-1) 

`applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` further advance the in-memory `pos`. Only then is the pool checked: [3](#0-2) 

When the pool is insufficient, the function returns an error with `pos` already fully advanced. In the normal claim path this is safe — the error propagates and the transaction reverts, undoing all state changes. But in `slashRedelegationPosition`, the error is explicitly swallowed: [4](#0-3) 

Execution then falls through to `setPositionWithState`, which commits the mutated `pos` (with advanced checkpoints) to the KV store: [5](#0-4) 

Because `LastBonusAccrual` and `LastEventSeq` are now permanently advanced past the slash point, the user can **never** reclaim the bonus that accrued up to the slash — even after the pool is later refilled. The `decrementEventRefCount` calls inside the loop are also committed, so the underlying validator events may be pruned, making recovery impossible.

This is structurally analogous to the reported LoC vulnerability: in the solvent-LoC liquidation path, the fee is added on top without checking whether reserved collateral can cover the total, causing the transaction to revert at a later point and leaving the position stuck. Here, the slash path advances accounting checkpoints without checking whether the pool can cover the bonus, causing the bonus to be permanently forfeited rather than deferred.

---

### Impact Explanation

Any position that holds an active redelegation entry at the time a validator slash occurs, and for which the `BonusRewardsPool` balance is insufficient to cover the accrued bonus, will have its bonus checkpoints permanently advanced without payment. The user permanently loses all bonus rewards accrued from `LastBonusAccrual` up to the slash timestamp. This is an irreversible loss of entitled funds — not a deferral — because the checkpoints that would allow a future claim are overwritten.

**Corrupted value:** `Position.LastBonusAccrual` and `Position.LastEventSeq` in the KV store, advanced past the slash point without the corresponding `BonusRewardsPool → owner` transfer.

---

### Likelihood Explanation

The `BonusRewardsPool` is funded by inflation and governance. It can legitimately run low or empty under high claim volume, a governance funding gap, or a deliberate pool-drain by other participants. Validator slashes are routine Cosmos SDK events triggered by double-sign or downtime evidence — they are not attacker-controlled from the victim's perspective, but they are a normal production occurrence. The combination of a low pool balance and a slash event is realistic, particularly on a chain with many active tiered positions.

---

### Recommendation

`processEventsAndClaimBonus` should not mutate the `pos` pointer before confirming the pool can cover the computed bonus. Two safe approaches:

1. **Compute first, mutate after**: Separate the bonus computation (read-only pass over events) from the checkpoint-advancement (write pass). Only advance checkpoints after `sufficientBonusPoolBalance` succeeds.
2. **Defer checkpoints on pool failure**: In `slashRedelegationPosition`, if `ErrInsufficientBonusPool` is returned, do **not** call `setPositionWithState` with the mutated `pos`. Instead, preserve the original checkpoints so the user can claim the bonus once the pool is refilled.

The slash path's intentional "forgo bonus to prevent chain halt" design is acceptable, but it must not permanently destroy the user's claim by advancing checkpoints without payment.

---

### Proof of Concept

1. User creates a position and calls `MsgTierRedelegate`, creating an active `x/staking` redelegation entry tracked by `RedelegationMapping`.
2. The `BonusRewardsPool` balance is zero (naturally drained or not yet funded).
3. A validator slash occurs on the destination validator, triggering `BeforeRedelegationSlashed` → `slashRedelegationPosition`.
4. Inside `slashRedelegationPosition`, `processEventsAndClaimBonus` is called. It advances `pos.LastEventSeq`, `pos.LastBonusAccrual`, and `pos.LastKnownBonded` in memory, then fails at `sufficientBonusPoolBalance` with `ErrInsufficientBonusPool`.
5. The error is swallowed. `setPositionWithState` is called with the mutated `pos`, committing the advanced checkpoints to the KV store. No bonus is transferred.
6. The pool is later refilled by governance.
7. The user calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` starts from the now-advanced `LastBonusAccrual` and `LastEventSeq`, skipping the pre-slash accrual period entirely. The bonus for that period is permanently unclaimable. [6](#0-5) [7](#0-6) [8](#0-7)

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
