Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which resolves to `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router address, not the end user. Any pool admin who allowlists the router (required for router-based swaps to function on a curated pool) inadvertently grants every user — including explicitly denied ones — the ability to bypass the per-user swap allowlist.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
  msg.sender,   // <-- immediate caller, not end user
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to `IMetricOmmExtensions.beforeSwap`. `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool address and `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) is used, it calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making `address(router)` the `msg.sender` to the pool, and therefore `sender` in the extension. For a curated pool to accept any router-mediated swap at all, the admin must call `setAllowedToSwap(pool, address(router), true)`. Once that entry exists, `allowedSwapper[pool][router] == true` satisfies the check for every user routing through the contract, regardless of whether that individual user is individually denied.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on the `owner` parameter — the economically attributed actor — which is not substituted by the router.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. Disallowed users can execute swaps against the pool, draining LP value or violating compliance constraints the pool admin intended to enforce. This is a direct, fund-impacting loss of curation policy for LPs on restricted pools, meeting the "broken core pool functionality causing loss of funds" and "admin-boundary break" impact categories.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard supported periphery path for end users. Any pool admin who allowlists the router — a necessary operational step to enable normal router usage on a curated pool — unknowingly opens the bypass. No privilege is required: any unpermissioned user can call the router. The misconfiguration is latent in the design of actor-binding, not in any unusual or adversarial setup.

## Recommendation

`SwapAllowlistExtension` must gate on the end user, not the immediate pool caller. Two viable approaches:

1. **Encode the originating user in `extensionData`**: The router encodes `msg.sender` (the true end user) into `extensionData` before calling `pool.swap()`. The extension decodes and checks this value. Requires a trusted encoding convention enforced by the router.
2. **Check `recipient` instead of `sender`**: If pool swap semantics guarantee `recipient` is always the economic beneficiary and cannot be spoofed to a third party, gate on `recipient`. This must be verified to hold across all router paths including multi-hop exact-output flows.

Option 1 with a verified encoding is the cleanest fix. Alternatively, document and enforce at the factory/extension initialization level that `SwapAllowlistExtension` is incompatible with router-mediated swaps.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, address(router), true)` to enable normal router usage.
3. Pool admin does **not** call `setAllowedToSwap(pool, address(attacker), true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the curated pool.
5. The router executes `IMetricOmmPoolActions(pool).swap(recipient, ...)` — router is `msg.sender` to the pool.
6. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender=router, ...)`.
7. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router] == true` → passes.
8. Attacker's swap executes on the curated pool despite being individually denied.
