Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the actual trader, enabling full allowlist bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. The pool passes `msg.sender` of its own `swap` call as `sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original trader. If the pool admin allowlists the router to support router-mediated swaps, every unprivileged user can bypass the allowlist by calling the router, defeating the curation guarantee entirely.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)` — `msg.sender` = user.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — `msg.sender` seen by the pool = **router address**.
3. Pool calls `_beforeSwap(msg.sender, ...)` → `ExtensionCalling._beforeSwap(sender = router, ...)`.
4. Pool calls `SwapAllowlistExtension.beforeSwap(sender = router, ...)`.
5. Extension evaluates `allowedSwapper[msg.sender/*pool*/][sender/*router*/]`.

**Root cause in `SwapAllowlistExtension.beforeSwap` (line 37):**
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
`sender` is the router, not the original user. The pool passes `msg.sender` (line 231 of `MetricOmmPool.sol`) as `sender` to `_beforeSwap`, and `msg.sender` during a router-mediated swap is the router contract.

**Exploit flow:**
- Pool admin deploys pool with `SwapAllowlistExtension`, allowlists specific user addresses, and also allowlists the router address so that allowlisted users can trade via the standard periphery path.
- Any non-allowlisted attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the curated pool.
- The extension receives `sender = router`, finds `allowedSwapper[pool][router] == true`, and passes.
- The attacker's swap executes on the curated pool despite not being on the allowlist.

**Existing guards are insufficient:** `BaseMetricExtension.onlyPool` (line 19-24) only verifies the caller is a registered pool — it does not verify the identity of the original trader. The pool correctly passes `msg.sender` as `sender`, but `msg.sender` is the router, not the user.

## Impact Explanation
A curated pool using `SwapAllowlistExtension` to restrict trading to approved counterparties is fully bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. The wrong value is `allowedSwapper[pool][sender]` where `sender` is the router address rather than the actual trader address. This is a direct allowlist bypass causing broken core pool functionality and potential loss of funds for LPs who deposited under the assumption that only approved traders could interact with the pool.

## Likelihood Explanation
The bypass requires only that the pool admin has allowlisted the router (a natural and expected configuration for any pool that intends to support the standard periphery). No special privileges, flash loans, or unusual conditions are needed. Any unprivileged user can call the public router functions. The attack is repeatable on every block.

## Recommendation
Pass the original trader identity through the swap path. Options:
1. Have the pool store the original `msg.sender` in transient storage before calling extensions, and expose it to extensions as the "initiator" rather than the immediate caller.
2. Have the router pass the original `msg.sender` as `extensionData` and have `SwapAllowlistExtension` decode and check it — but this requires the extension to trust the router, which reintroduces the same trust problem.
3. The cleanest fix: change `SwapAllowlistExtension.beforeSwap` to check `recipient` or require the pool to pass the original initiator as a dedicated field in the extension interface, separate from `sender`.
