Let me analyze the external bug (wrong refund address in cross-chain transfers) and search for nearcore analogs involving fee/reward attribution, delegate actions, or meta-transactions where the wrong address is used.

### Title
`apply_delegate_action` Creates V1 Receipt Without `refund_to`, Sending Deposit Refunds to the Wrong Account in Meta-Transactions — (`runtime/runtime/src/actions.rs`)

### Summary

In NEAR's meta-transaction (NEP-366) flow, the relayer pays all token deposits attached to inner actions. When an inner action fails on the receiver's shard, the deposit refund is sent to `predecessor_id` of the receipt — which is Alice (the `DelegateAction` sender), not the relayer who actually paid. `apply_delegate_action` creates a legacy `ReceiptEnum::Action` (V1) receipt that has no `refund_to` field, so the runtime's `balance_refund_receiver()` always falls back to `predecessor_id`. The `ActionReceiptV2` type with its `refund_to: Option<AccountId>` field exists precisely to fix this class of problem, but `apply_delegate_action` never uses it.

### Finding Description

`apply_delegate_action` in `runtime/runtime/src/actions.rs` constructs the inner-action receipt as a legacy `ReceiptEnum::Action(ActionReceipt { … })`: [1](#0-0) 

The receipt's `predecessor_id` is set to `sender_id` (Alice), and `signer_id` is set to the relayer (inherited from the outer receipt). Because the receipt is `ReceiptEnum::Action` (discriminant 0), `Receipt::refund_to()` unconditionally returns `&None`: [2](#0-1) 

`balance_refund_receiver()` therefore resolves to `predecessor_id()` = Alice. When `refund_unspent_gas_and_deposits` fires a deposit refund, it calls `Receipt::new_balance_refund(receipt.balance_refund_receiver(), deposit_refund)`: [3](#0-2) 

The `ActionReceiptV2` struct already carries the `refund_to: Option<AccountId>` field for exactly this purpose: [4](#0-3) 

And `function_call.rs` correctly propagates `refund_to` when contracts create receipts via `promise_set_refund_to`: [5](#0-4) 

But `apply_delegate_action` never uses `ActionReceiptV2`, so there is no mechanism to route the deposit refund to the relayer.

The protocol documentation explicitly acknowledges the consequence: [6](#0-5) 

### Impact Explanation

Any unprivileged user (Alice) can craft a `DelegateAction` that attaches a large deposit to an inner action that is guaranteed to fail on the receiver's shard (e.g., calling a non-existent method on a contract). The relayer pays the deposit when it submits the wrapping transaction. When the inner action fails, the full deposit is refunded to Alice, not the relayer. The relayer loses the deposit with no recourse. This is a direct, repeatable token-theft vector against any public relayer service.

### Likelihood Explanation

The relayer is an unprivileged third party. Alice controls the content of the `DelegateAction` she signs, including the attached deposit and the target method. She can deliberately choose a method that does not exist or a receiver that has no contract, guaranteeing failure. No special access or timing is required. The only constraint is that the relayer must accept and submit the meta-transaction, which is the relayer's normal operating mode.

### Recommendation

Change `apply_delegate_action` to emit `ReceiptEnum::ActionV2(ActionReceiptV2 { refund_to: Some(action_receipt.signer_id().clone()), … })` instead of `ReceiptEnum::Action(ActionReceipt { … })`. Setting `refund_to` to the relayer's account ID (`action_receipt.signer_id()`) ensures that any deposit refund from a failed inner action is returned to the entity that paid it, mirroring how `function_call.rs` already handles `refund_to` for contract-created receipts. [7](#0-6) 

### Proof of Concept

1. Relayer submits a meta-transaction wrapping Alice's `DelegateAction`.
2. `DelegateAction` targets `bob.near` with `Action::FunctionCall { method_name: "nonexistent", deposit: 10 NEAR, … }`.
3. `apply_delegate_action` creates `ReceiptEnum::Action(ActionReceipt { predecessor_id: alice, signer_id: relayer, … })`.
4. On Bob's shard, the function call fails (method not found). `refund_unspent_gas_and_deposits` fires.
5. `receipt.balance_refund_receiver()` → `receipt.refund_to()` → `&None` (V1 receipt) → falls back to `receipt.predecessor_id()` → `alice`.
6. `Receipt::new_balance_refund(&alice, 10 NEAR)` is emitted.
7. Alice receives 10 NEAR; the relayer loses 10 NEAR. [8](#0-7) [9](#0-8)

        match self.receipt() {
            ReceiptEnum::Action(_)
            | ReceiptEnum::Data(_)
            | ReceiptEnum::PromiseYield(_)
            | ReceiptEnum::PromiseResume(_)
            | ReceiptEnum::GlobalContractDistribution(_) => &None,

```markdown
## Balance refunds in meta transactions

Unlike gas refunds, the protocol sends balance refunds to the predecessor
(a.k.a. sender) of the receipt. This makes sense, as we deposit the attached
balance to the receiver, who has to explicitly reattach a new balance to new
receipts they might spawn.

In the world of meta transactions, this assumption is also challenged. If an
inner action requires an attached balance (for example a transfer action) then
this balance is taken from the relayer.

The relayer can see what the cost will be before submitting the meta transaction
and agrees to pay for it, so nothing wrong so far. But what if the transaction
fails execution on Bob's shard? At this point, the predecessor is `Alice` and
therefore she receives the token balance refunded, not the relayer. This is
something relayer implementations must be aware of since there is a financial
incentive for Alice to submit meta transactions that have high balances attached
but will fail on Bob's shard.
```
