### Title
CEI Violation in `withdrawCollateral` Allows `nSubmissions` Corruption via Reentrancy with Hook-Bearing Tokens — (`File: core/contracts/Clearinghouse.sol`)

### Summary
`Clearinghouse.withdrawCollateral` transfers tokens to the recipient before decrementing the subaccount's SpotEngine balance, violating the Checks-Effects-Interactions pattern. When the withdrawal token implements a transfer hook (e.g., ERC777 `tokensReceived`), an attacker controlling `sendTo` can re-enter `Endpoint.executeSlowModeTransaction` during the transfer. Because `slowModeConfig` is written back to storage only after `_executeSlowModeTransaction` returns, the reentrant call increments `nSubmissions` a second time, permanently skipping a submission index. This leaves a gap in `BaseWithdrawPool.markedIdxs`, enabling a pre-signed fast withdrawal for the skipped index to be submitted and paid out a second time.

---

### Finding Description

`Clearinghouse.withdrawCollateral` (lines 408 → 412) performs the external token transfer before updating the subaccount balance:

```
handleWithdrawTransfer(token, sendTo, amount, idx);   // line 408 — external call chain
...
spotEngine.updateBalance(productId, sender, amountRealized); // line 412 — state update
```

`handleWithdrawTransfer` in `Clearinghouse` (lines 383–384) first pushes tokens to `withdrawPool`, then calls `BaseWithdrawPool.submitWithdrawal`, which in turn calls `token.safeTransfer(sendTo, amount)` (line 189 of `BaseWithdrawPool`). The final recipient is the user-controlled `sendTo` address.

`Endpoint.executeSlowModeTransaction` (lines 231–236) is permissionless and stores `slowModeConfig` as a memory copy that is written back to storage only after `_executeSlowModeTransaction` returns:

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;   // read
    _executeSlowModeTransaction(_slowModeConfig, false);
    nSubmissions += 1;                                         // incremented after
    slowModeConfig = _slowModeConfig;                          // written back after
}
```

`_executeSlowModeTransaction` deletes the slot from the `slowModeTxs` mapping before the external call (`delete slowModeTxs[_slowModeConfig.txUpTo++]`, line 194), but the storage variable `slowModeConfig.txUpTo` is not updated until the outer function returns.

**Reentrancy sequence:**

1. Attacker queues a slow-mode withdrawal; `sendTo` is an ERC777-aware contract.
2. Anyone calls `executeSlowModeTransaction`; `nSubmissions = N` at entry.
3. `withdrawCollateral` is reached; `idx = N` is passed to `submitWithdrawal`.
4. `BaseWithdrawPool.submitWithdrawal` sets `markedIdxs[N] = true`, `minIdx = N`, then calls `token.safeTransfer(sendTo, amount)`.
5. The ERC777 hook fires; attacker re-enters `executeSlowModeTransaction`.
6. Re-entrant call reads `slowModeConfig.txUpTo = N` from storage (not yet updated); `slowModeTxs[N]` is already deleted → processes an empty tx → reverts → caught by `try/catch`.
7. Re-entrant call still executes `nSubmissions += 1` (now `N+1`) and writes `slowModeConfig.txUpTo = N+1` to storage.
8. Re-entrant call returns; outer call resumes, decrements balance, passes health check.
9. Outer call executes `nSubmissions += 1` again (now `N+2`) and writes `slowModeConfig.txUpTo = N+1` (same value, no harm to queue pointer).

**Result:** `nSubmissions` is `N+2` instead of `N+1`. Index `N+1` is never passed to `submitWithdrawal`, so `markedIdxs[N+1]` is never set and `minIdx` remains `N`.

A pre-signed fast withdrawal for `idx = N+1` now satisfies both guards in `submitFastWithdrawal`:
- `!markedIdxs[N+1]` → true (never marked)
- `N+1 > minIdx (= N)` → true

The attacker submits the fast withdrawal, draining an additional `amount` from the `WithdrawPool`'s liquidity pool — tokens belonging to other users.

---

### Impact Explanation

The attacker receives their collateral twice: once via the slow-mode path and once via the fast-withdrawal path for the skipped index. The second payment is drawn from the `WithdrawPool`'s shared liquidity, directly stealing funds deposited by other users. The corrupted `nSubmissions` counter also permanently desynchronises the submission index, potentially blocking or misrouting future fast withdrawals.

---

### Likelihood Explanation

- `executeSlowModeTransaction` is permissionless; any user can trigger it.
- The attacker controls `sendTo` (explicitly set in `WithdrawCollateralV2.sendTo`).
- ERC777 tokens or tokens with transfer hooks are a realistic integration scenario for a multi-asset exchange.
- The sequencer routinely pre-signs fast withdrawals; a signature for `idx = N+1` is a normal operational artefact.
- No privileged keys, governance capture, or sequencer compromise is required.

---

### Recommendation

Apply the fix analogous to the Berachain resolution: move `spotEngine.updateBalance` and the health check **before** `handleWithdrawTransfer` in `withdrawCollateral`, or add a `ReentrancyGuard` to `executeSlowModeTransaction` (and/or `withdrawCollateral`). Additionally, increment `nSubmissions` and write `slowModeConfig` back to storage **before** calling `_executeSlowModeTransaction` to close the storage-lag window.

---

### Proof of Concept

```
Attacker contract (ERC777 tokensReceived hook):
  function tokensReceived(...) external {
      IEndpoint(endpoint).executeSlowModeTransaction();
      // nSubmissions now N+2; markedIdxs[N+1] unset
  }

Attack flow:
  1. attacker.submitSlowModeWithdrawal(productId, amount, sendTo=attackerContract)
  2. sequencer signs FastWithdrawal(idx=N+1, amount=amount)
  3. anyone calls endpoint.executeSlowModeTransaction()
     → tokensReceived fires → nSubmissions corrupted to N+2
  4. attacker calls withdrawPool.submitFastWithdrawal(N+1, signedTx, sigs)
     → markedIdxs[N+1] == false, N+1 > minIdx(N) → passes
     → attacker receives `amount` a second time from WithdrawPool liquidity
```

```text
    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
