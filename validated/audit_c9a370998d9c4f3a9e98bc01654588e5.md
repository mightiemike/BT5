### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the End-User, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. That argument is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. If the pool admin allowlists the router (the only way to let allowlisted users reach the pool through the router), every user — including those not on the allowlist — can bypass the gate by going through the router.

---

### Finding Description

**Step 1 — What the extension checks.**

`SwapAllowlistExtension.beforeSwap` uses `msg.sender` (the pool) as the mapping key and `sender` (the first argument forwarded by the pool) as the identity being gated:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

**Step 2 — What the pool passes as `sender`.**

`MetricOmmPool.swap` passes `msg.sender` (the immediate caller of `swap`) as the first argument to `_beforeSwap`, which encodes it as `sender` in the extension call:

```solidity
// metric-core/contracts/MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    ...
);
``` [2](#0-1) 

**Step 3 — What the router passes to the pool.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(recipient, ...)` directly. The original `msg.sender` (the end-user) is stored only in transient storage for the payment callback; it is never forwarded to the pool as a parameter:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData   // ← user's identity is NOT in here
    );
``` [3](#0-2) 

The same pattern applies to `exactInput` and `exactOutputSingle`. [4](#0-3) 

**The resulting identity mismatch.**

When a user calls the router, the pool's `msg.sender` is the router contract. The extension therefore evaluates:

```
allowedSwapper[pool][router_address]
```

instead of:

```
allowedSwapper[pool][actual_user]
```

A pool admin who wants allowlisted users to reach the pool through the router must add the router to the allowlist. The moment the router is allowlisted, the check becomes `allowedSwapper[pool][router] == true` for every caller — the per-user gate is permanently open for all router-mediated swaps.

---

### Impact Explanation

Any user — regardless of whether they appear in `allowedSwapper` — can execute swaps against a pool that has `SwapAllowlistExtension` configured, simply by routing through `MetricOmmSimpleRouter`. The allowlist access control is completely bypassed for router-mediated swaps whenever the router itself is allowlisted. This breaks the admin-boundary invariant: an unprivileged path (the public router) circumvents an admin-configured guard, allowing unauthorized actors to interact with a pool that was explicitly restricted.

---

### Likelihood Explanation

The bypass requires the router to be allowlisted. A pool admin who deploys a restricted pool and wants their allowlisted users to use the standard router will naturally add the router to the allowlist — this is the only way to make the router work for legitimate users. The misconfiguration is therefore the expected operational path, not an edge case. Any pool that (a) uses `SwapAllowlistExtension` and (b) allowlists the router is fully exposed.

---

### Recommendation

The extension must gate the actual end-user, not the intermediate router. Two viable approaches:

1. **Require the user's address in `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires a convention between the router and the extension.
2. **Check `sender` only when it is a registered pool-factory pool; otherwise check the decoded user from `extensionData`:** Allows direct-pool callers to be gated by their own address while router callers are gated by the address they supply in the extension payload.

Either way, the extension must not treat the router address as the identity to gate.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — Alice is the only intended swapper.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — necessary so Alice can use the router.
4. Bob (not in the allowlist) calls `router.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, ...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true`.
8. Bob's swap executes successfully; the allowlist is bypassed.

The root cause is structurally identical to the seeded bug: a predictable/shared address (the router, analogous to the pre-computable staking token address) is registered in a gating registry (the allowlist, analogous to `getVault`), and that registration opens the gate for all callers rather than only the intended one.
