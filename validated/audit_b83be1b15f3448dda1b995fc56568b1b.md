### Title
Cost Limit Check Applied to Pre-Multiplier Base in `op_unknown` — (`File: src/more_ops.rs`)

### Summary

In `op_unknown`, the `check_cost` guard is called on the raw base cost **before** the `cost_multiplier` is applied. The function then multiplies the cost by `(cost_multiplier + 1)` and returns the inflated value. This means the cost-limit enforcement is applied to the wrong (smaller) base, directly mirroring the external report's class: a fee/cost check that operates on the wrong quantity, allowing the actual charged amount to exceed the budget without triggering the guard inside the function.

### Finding Description

`op_unknown` in `src/more_ops.rs` computes a base cost from the opcode's `cost_function` field (one of four arithmetic-style formulas), then multiplies that base cost by `(cost_multiplier + 1)` where `cost_multiplier` is a 32-bit value encoded in the opcode bytes. The critical ordering is:

```
line 260: check_cost(cost, max_cost)?;      // checks PRE-multiplier cost
line 261: cost *= cost_multiplier + 1;      // inflates cost AFTER the check
``` [1](#0-0) 

The `check_cost` at line 260 validates the base cost (e.g., as low as `1` for `cost_function = 0`) against the remaining budget. If the base cost is within budget, the check passes. The function then multiplies by up to `u32::MAX + 1 = 4,294,967,296` and returns the inflated value — which can be up to `u32::MAX = 4,294,967,295` (capped by the `u32::MAX` guard at line 262). [2](#0-1) 

The cost constants confirm the base values are small: `ARITH_BASE_COST = 99`, `CONCAT_BASE_COST = 142`, `MUL_BASE_COST = 92`. For `cost_function = 0`, the base cost is always `1`. [3](#0-2) 

The returned inflated cost is then accumulated in `run_program`'s main loop:

```rust
cost += match op {
    Operation::Apply => self.apply_op(cost, effective_max_cost - cost)?,
``` [4](#0-3) 

The overshoot is caught only at the **top of the next loop iteration**:

```rust
if cost > effective_max_cost {
    return Err(EvalErr::CostExceeded);
}
``` [5](#0-4) 

This means `op_unknown` itself never enforces the cost limit correctly — it always delegates enforcement to the outer loop, one iteration late.

### Impact Explanation

The cost limit is not enforced at the operator boundary. `op_unknown` can return a cost up to `u32::MAX` (~4.3 billion) regardless of the remaining budget passed to it. The outer loop catches the overshoot, so programs do ultimately fail with `CostExceeded`. However:

1. **Cost accounting invariant broken**: The contract of `op_unknown` — that it raises `CostExceeded` if the operation would exceed `max_cost` — is violated. The function silently returns a cost that can be orders of magnitude larger than the remaining budget.
2. **Consensus divergence risk**: Any reference implementation (e.g., Python CLVM) that correctly checks the post-multiplier cost against `max_cost` inside the op handler will raise `CostExceeded` at a different point in execution. While both ultimately reject the program, the divergence in *where* the error is raised can affect softfork cost validation (`exit_guard` checks `current_cost == guard.expected_cost`), error attribution, and any caller that inspects the partial cost state.
3. **Softfork guard interaction**: Inside a softfork, `effective_max_cost = current_cost + expected_cost`. If `op_unknown` returns a cost that overshoots `expected_cost`, the loop raises `CostExceeded` before `exit_guard` can validate the cost match. This changes the error type from `SoftforkCostMismatch` to `CostExceeded`, which is a behavioral divergence from the intended softfork semantics. [6](#0-5) 

### Likelihood Explanation

`op_unknown` is reachable in any non-mempool execution context where `ClvmFlags::NO_UNKNOWN_OPS` is not set (i.e., consensus/lenient mode). An attacker-controlled CLVM program can include any opcode byte sequence that is not a known operator and does not start with `0xffff`. The `cost_multiplier` is fully attacker-controlled via the opcode bytes. A single-byte opcode with `cost_function = 0` (base cost = 1) and a 4-byte multiplier prefix encoding `u32::MAX / 2` is sufficient to trigger the discrepancy. This is trivially constructable. [7](#0-6) 

### Recommendation

Move `check_cost` to **after** the multiplier is applied:

```rust
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}
check_cost(cost, max_cost)?;   // check POST-multiplier cost
Ok(Reduction(cost as Cost, allocator.nil()))
```

This ensures the cost limit is enforced on the actual returned cost, not the pre-multiplier base.

### Proof of Concept

Craft an unknown opcode with:
- Opcode bytes: `[0x3F, 0xFF, 0xFF, 0xFF, 0x00]`
  - Last byte `0x00`: `cost_function = 0` (base cost = 1), 6 low bits ignored
  - Prefix bytes `[0x3F, 0xFF, 0xFF, 0xFF]`: `cost_multiplier = 0x3FFFFFFF = 1,073,741,823`
- Final cost = `1 * (1,073,741,823 + 1) = 1,073,741,824`

Set `max_cost = 500` (remaining budget). Call `op_unknown`:
- Base cost = 1 (cost_function 0)
- `check_cost(1, 500)` → passes (1 ≤ 500)
- `cost *= 1,073,741,824` → cost = 1,073,741,824
- `1,073,741,824 ≤ u32::MAX` → no `Invalid` error
- Returns `Reduction(1_073_741_824, nil)` — **exceeds `max_cost = 500` by ~2 million×**

The outer loop in `run_program` then catches `cost > effective_max_cost` one iteration later, but `op_unknown` itself returned successfully with a cost that violated the budget it was given. [8](#0-7) [1](#0-0)

    fn exit_guard(&mut self, current_cost: Cost) -> Result<Cost> {
        // this is called when we are done executing a softfork program.
        // This is when we have to validate the cost
        let guard = self
            .softfork_stack
            .pop()
            .expect("internal error. exiting a softfork that's already been popped");

        if current_cost != guard.expected_cost {
            #[cfg(test)]
            println!(
                "actual cost: {} specified cost: {}",
                current_cost - guard.start_cost,
                guard.expected_cost - guard.start_cost
            );
            return Err(EvalErr::SoftforkCostMismatch);
        }

        // restore the allocator to the state when we entered the softfork guard
        // This is an optimization to reclaim all heap space allocated by the
        // softfork program. Since the softfork always return nil, no value can
        // escape the softfork program, and it's therefore safe to restore the
        // heap
        self.allocator.restore_checkpoint(&guard.allocator_state);

        // the softfork always returns nil, pop the value pushed by the
        // evaluation of the program and push nil instead
        self.pop()
            .expect("internal error, softfork program did not push value onto stack");

        self.push(self.allocator.nil())?;

        Ok(0)
    }
