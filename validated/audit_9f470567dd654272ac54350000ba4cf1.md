### Title
`verify_transaction_inclusion_v2` Always Reverts for Single-Transaction Blocks - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion_v2` — the recommended, non-deprecated proof API — unconditionally delegates to `verify_transaction_inclusion` after its own coinbase validation. `verify_transaction_inclusion` contains a hard guard that rejects any empty `merkle_proof`. For a single-transaction block (only the coinbase), both the coinbase proof and the transaction proof are legitimately empty, so the coinbase check inside `verify_transaction_inclusion_v2` passes, but the inner call immediately panics. The result is that the recommended API always reverts for a valid and reachable class of Bitcoin blocks.

### Finding Description

`verify_transaction_inclusion_v2` is the successor to the deprecated `verify_transaction_inclusion`. Its purpose is to add a coinbase-proof step to defeat the 64-byte Merkle forgery attack. After validating the coinbase proof it delegates the rest of the work to the deprecated function via `self.verify_transaction_inclusion(args.into())`. [1](#0-0) 

The deprecated inner function contains this guard: [2](#0-1) 

For a single-transaction block the Merkle tree has exactly one leaf. The merkle root equals the coinbase transaction hash, and no sibling hashes are needed, so both `coinbase_merkle_proof` and `merkle_proof` are empty (`[]`).

Step-by-step execution for a single-tx block:

1. **Length check passes** — `0 == 0`. [3](#0-2) 

2. **Coinbase proof check passes** — `compute_root_from_merkle_proof` with an empty proof list simply returns the input hash unchanged. [4](#0-3)  Because `coinbase_tx_id == merkle_root` for a single-tx block, the `require!` succeeds. [5](#0-4) 

3. **Inner call panics** — `verify_transaction_inclusion` is called with `merkle_proof = []`. The `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard fires unconditionally. [2](#0-1) 

The `From<ProofArgsV2> for ProofArgs` conversion faithfully copies `args.merkle_proof` (the empty vec) into the inner args, so there is no opportunity for the empty proof to be corrected in transit. [6](#0-5) 

### Impact Explanation

Any NEAR caller invoking `verify_transaction_inclusion_v2` for a transaction in a single-transaction Bitcoin block receives a panic revert. Because `verify_transaction_inclusion` is explicitly deprecated and callers are directed to use `verify_transaction_inclusion_v2`, the recommended API is broken for this block class. Downstream contracts or relayers that rely on the return value to gate cross-chain actions (e.g., releasing funds after SPV proof) will be permanently blocked for affected blocks.

### Likelihood Explanation

Single-transaction blocks (coinbase only) are common in Bitcoin history and continue to occur on mainnet when miners produce empty blocks. The contract targets Bitcoin mainnet (`Network::Mainnet`). Any integrator attempting to prove a coinbase transaction from such a block via the recommended API will trigger the revert deterministically. No special attacker capability is required — a normal unprivileged NEAR caller supplying a valid proof for a real single-tx block is sufficient.

### Recommendation

Remove the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard from `verify_transaction_inclusion`, or move it only into the v1 path. For a single-transaction block the correct behaviour of `compute_root_from_merkle_proof` with an empty proof is to return the transaction hash itself, which is then compared against `merkle_root` — this comparison is the real correctness check and is sufficient on its own. [7](#0-6) 

Alternatively, `verify_transaction_inclusion_v2` can short-circuit before calling the inner function when both proofs are empty and `tx_id == header.block_header.merkle_root`.

### Proof of Concept

Caller submits `ProofArgsV2` for a real single-transaction Bitcoin block:

```
tx_id              = <coinbase_txid>          // only tx in the block
tx_block_blockhash = <block_hash>
tx_index           = 0
merkle_proof       = []                       // empty — no siblings exist
coinbase_tx_id     = <coinbase_txid>          // same tx
coinbase_merkle_proof = []                    // empty — no siblings exist
confirmations      = 1
```

Execution trace inside `verify_transaction_inclusion_v2`:
- `merkle_proof.len() == coinbase_merkle_proof.len()` → `0 == 0` ✓
- `compute_root_from_merkle_proof(coinbase_txid, 0, &[])` → returns `coinbase_txid` == `merkle_root` ✓
- Calls `verify_transaction_inclusion(args.into())`
  - `require!(!args.merkle_proof.is_empty(), ...)` → **PANIC: "Merkle proof is empty"**

The call reverts. The same block verified via the deprecated `verify_transaction_inclusion` (with `merkle_proof = []`) also reverts for the same reason, meaning there is no working code path for this valid input.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
