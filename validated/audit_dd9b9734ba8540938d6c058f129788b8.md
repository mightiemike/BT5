### Title
Users Cannot Fully Clear Spot Collateral Balance Due to Share-Based Accounting Mismatch — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`WithdrawCollateral` and `WithdrawCollateralV2` accept only a token-denominated `amount`, but `SpotEngine` stores balances as normalized shares (`amountNormalized`) scaled by a monotonically increasing `cumulativeDepositsMultiplierX18`. Because the multiplier grows with every `SpotTick`, a withdrawal signed for the user's full balance at time T will always leave a non-zero residual after the multiplier advances, making it structurally impossible to fully clear a spot balance in a single transaction.

---

### Finding Description

`SpotEngineState` stores every subaccount's spot balance as a normalized share:

```
balance.amount = amountNormalized × cumulativeDepositsMultiplierX18
``` [1](#0-0) 

The multiplier grows continuously via `_updateState`, which is triggered by every `SpotTick` transaction the sequencer submits: [2](#0-1) 

Both `WithdrawCollateral` and `WithdrawCollateralV2` structs carry only a token-denominated `amount` field — there is no share/normalized field: [3](#0-2) [4](#0-3) 

`Clearinghouse.withdrawCollateral` converts the token amount back to a balance delta using the **current** multiplier at execution time: [5](#0-4) 

Inside `_updateBalanceNormalized`, the new normalized balance is:

```
newAmount = amountNormalized × cumulativeMultiplier_T1  +  amountRealized
          = amountNormalized × T1  −  amountNormalized × T0
          = amountNormalized × (T1 − T0)
``` [6](#0-5) 

Because `T1 > T0` whenever a `SpotTick` has been processed between signing and execution, `newAmount > 0` — a residual normalized balance always remains.

---

### Impact Explanation

The residual is real value: it equals the interest accrued on the user's full deposit between the moment they signed the withdrawal and the moment the sequencer executed it. The user cannot withdraw this residual without signing a new transaction, which will itself leave another residual for the same reason. A user who wants to fully exit a spot position must repeatedly submit withdrawals, each time chasing a shrinking but never-zero dust balance. The funds are not lost, but they are structurally unclaimable in a single atomic operation, and the residual compounds as long as the position remains open.

---

### Likelihood Explanation

`SpotTick` is a routine sequencer operation processed continuously. Any withdrawal that is not the very first transaction in a batch — or that is submitted via the slow-mode path (3-day delay) — will have at least one `SpotTick` interleaved before execution. This condition is met for virtually every real withdrawal. The issue is not edge-case; it is the normal operating path. [7](#0-6) 

---

### Recommendation

Add a `withdrawCollateralByShares` path (or a boolean `useNormalized` flag on the existing structs) that accepts `amountNormalized` directly and deducts it without any multiplier-dependent conversion. When a user wants to fully exit, they pass their stored `amountNormalized` value; the contract sets the balance to zero unconditionally. The token amount transferred is then computed as `amountNormalized × cumulativeDepositsMultiplierX18` at execution time, ensuring the user receives the full accrued value and the on-chain balance is exactly zeroed.

---

### Proof of Concept

1. At time T0, `cumulativeDepositsMultiplierX18 = M0`. User's stored `amountNormalized = N`. Actual balance = `N × M0`.
2. User signs `WithdrawCollateral { amount = N × M0 / decimalsMultiplier }` (their full balance).
3. Sequencer processes a `SpotTick`; multiplier advances to `M1 > M0`.
4. Sequencer processes the withdrawal. `Clearinghouse.withdrawCollateral` computes:
   - `amountRealized = -(N × M0 / decimalsMultiplier) × decimalsMultiplier = -N × M0`
5. `_updateBalanceNormalized` computes:
   - `newAmount = N × M1 + (-N × M0) = N × (M1 − M0) > 0`
6. The subaccount retains a normalized balance of `N × (M1 − M0) / M1` shares — a non-zero residual proportional to the interest accrued during sequencer latency.
7. The user must sign another withdrawal to claim this residual, which will again leave a residual for the same reason. [8](#0-7) [9](#0-8) [10](#0-9)

    function _updateBalanceNormalized(
        State memory state,
        BalanceNormalized memory balance,
        int128 balanceDelta
    ) internal pure {
        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized -= balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized += balance.amountNormalized;
        }

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        int128 newAmount = balance.amountNormalized.mul(
            cumulativeMultiplierX18
        ) + balanceDelta;

        if (newAmount > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);

        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized += balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized -= balance.amountNormalized;
        }
    }

        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);
