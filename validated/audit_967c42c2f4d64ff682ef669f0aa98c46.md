### Title
`burnNlp` Uses Maintenance Health Check Instead of Initial Health Check, Allowing Voluntary Creation of Initial-Health-Negative Subaccounts — (`core/contracts/Clearinghouse.sol`)

### Summary
`Clearinghouse.burnNlp` enforces only a `MAINTENANCE` health check after burning NLP tokens, while every other voluntary user action (`withdrawCollateral`, `mintNlp`, `transferQuote`) enforces an `INITIAL` health check. A user can craft a burn that leaves their initial health negative while maintenance health remains non-negative, placing them in a "zombie" state that cannot open positions and cannot be liquidated, but is one price move away from bad debt.

### Finding Description
In `Clearinghouse.burnNlp`, after deducting `nlpAmount` from the sender's NLP balance and crediting `quoteAmount = nlpAmount * oraclePriceX18 - burnFee` to their quote balance, the only health guard is:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) 

The burn fee is `max(ONE, quoteAmount / 1000)` — at minimum 1 unit, or 0.1% of the gross quote value. [2](#0-1) 

Because `INITIAL` weights are strictly more conservative than `MAINTENANCE` weights (lower long-weight, higher short-weight), a user whose initial health is marginally positive before the burn can end up with initial health < 0 and maintenance health >= 0 after the burn fee is applied. The code comment on line 523 even acknowledges this possibility but uses the wrong guard.

By contrast, `mintNlp` correctly enforces `INITIAL`: [3](#0-2) 

And `withdrawCollateral` enforces `INITIAL` for all non-X_ACCOUNT senders: [4](#0-3) 

### Impact Explanation
A user in the resulting state (initial health < 0, maintenance health >= 0):
1. Cannot open any new positions (the offchain exchange enforces initial health >= 0 for order matching).
2. Cannot be liquidated (liquidation requires maintenance health < 0).
3. Is one adverse price move away from crossing the maintenance threshold, at which point the protocol absorbs bad debt because the user has no remaining buffer.

This directly matches the scoped critical impact: settlement outcomes are distorted in a way that transfers value incorrectly (losses socialized to the insurance fund / other LPs).

### Likelihood Explanation
The path is fully user-controlled: sign a `BurnNlp` transaction with a large enough `nlpAmount` such that the burn fee tips initial health negative. No admin access, no oracle manipulation, and no sequencer compromise is required. The user only needs to hold NLP and have perp/spot positions that create a gap between their initial and maintenance health.

### Recommendation
Replace the `MAINTENANCE` health check in `burnNlp` with an `INITIAL` health check, consistent with all other voluntary user actions:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) 

### Proof of Concept
1. User deposits collateral and opens a perp position such that: `getHealth(sender, INITIAL) = +ε` (barely positive) and `getHealth(sender, MAINTENANCE) = +δ` where `δ >> ε`.
2. User mints NLP worth `Q` quote. `mintNlp` checks `INITIAL >= 0` — passes.
3. User submits `BurnNlp` for the same `nlpAmount`. Burn fee = `max(ONE, Q/1000)`. Net quote credited = `Q - fee`.
4. After the burn: the NLP position is gone (its initial-weight contribution is removed) and the net quote received is less than the NLP's initial-health contribution by the burn fee amount. Result: `getHealth(sender, INITIAL) < 0`, `getHealth(sender, MAINTENANCE) >= 0`.
5. The `require(getHealth(sender, MAINTENANCE) >= 0)` check passes. Transaction succeeds.
6. Assert: `getHealth(sender, INITIAL) < 0` and `getHealth(sender, MAINTENANCE) >= 0` simultaneously — invariant broken. [5](#0-4)
