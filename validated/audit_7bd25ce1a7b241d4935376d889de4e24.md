Audit Report

## Title
`PriceVelocityGuardExtension` per-block velocity cap bypassed by multiple intra-block swaps due to unconditional `lastMidPriceX64` overwrite â€” (File: `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap` unconditionally overwrites `lastMidPriceX64` and `lastUpdateBlock` on every swap, including multiple swaps within the same block. Each intra-block swap is therefore velocity-checked against the immediately preceding swap's mid price rather than the block's starting mid price. With a permissionless push oracle (Pyth Lazer), an attacker can submit two valid signed price updates within the same block and execute two swaps that each individually satisfy the velocity constraint but collectively exceed the configured per-block cap by a factor of `n`.

## Finding Description

In `beforeSwap` (lines 53â€“58 of `PriceVelocityGuardExtension.sol`), the guard reads `prevMid` and `prevBlock` from storage, then **unconditionally overwrites both** before performing the velocity check:

```solidity

uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;          // written unconditionally
s.lastUpdateBlock = uint64(block.number); // written unconditionally

if (prevMid != 0) {
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        uint256 blockDiff = block.number - prevBlock;
        // ...
        uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
        if (actualSq > allowedSq) revert PriceVelocityExceeded(actualSq, allowedSq);
```

The Pyth Lazer oracle (`PythOracle.sol`) has a permissionless `fallback()` that accepts any Lazer-signed payload. `LazerConsumer._verifyAndStore` enforces per-feed monotonicity by `FeedUpdateTimestamp` â€” a newer signed timestamp overwrites the stored price. 2. Execute Swap 1 â†’ guard reads `prevMid = P0`, `blockDiff = 0`, checks `(1%)Â² â‰¤ (1%)Â² Ã— 1` â†’ **passes**, writes `lastMidPriceX64 = P1`

The transient reentrancy guard clears between sequential (non-nested) swap calls, so no reentrancy protection prevents this.

## Impact Explanation

The velocity guard's configured per-block price-movement cap degrades from "price cannot move more than `maxChangePerBlockE18` per block" to "price cannot move more than `maxChangePerBlockE18` per swap." An attacker can execute `n` swaps in one block, each at a freshly pushed oracle price, achieving `n Ã— maxChangePerBlockE18` total movement. Swaps execute at prices the guard was deployed to block, constituting bad-price execution against LP positions â€” a direct loss of LP principal that the extension was deployed to prevent. This matches the contest's allowed impact: bad-price execution where an unclamped oracle price reaches a pool swap.

## Likelihood Explanation

Pyth Lazer is a permissionless push oracle: anyone holding valid Lazer-signed price attestations (publicly available from Pyth's price service) can call `PythOracle.fallback()` to update the on-chain price. No privileged access is required. The only constraint is that each update must carry a strictly newer `FeedUpdateTimestamp` than the stored one â€” satisfied by using two consecutive real market price attestations. This is a realistic, low-barrier attack on any pool using a Pyth-backed price provider with `PriceVelocityGuardExtension` configured.

## Recommendation

When `block.number == lastUpdateBlock`, do not overwrite `lastMidPriceX64`. Keep the block's first observed mid price as the reference for all swaps within the same block:

```solidity
PriceVelocityState storage s = priceVelocityState[pool_];
uint128 prevMid = s.lastMidPriceX64;
uint64 prevBlock = s.lastUpdateBlock;

// Only advance the reference price when entering a new block.
if (block.number > prevBlock) {
    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);
}

if (prevMid != 0) {
    // velocity check unchanged
}
```

This ensures the per-block cap is measured from the block's first observed price, not the most recent swap's price.
