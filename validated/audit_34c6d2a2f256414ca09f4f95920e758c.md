### Title
Linked Signer Can Self-Escalate by Signing a New `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

### Summary

The `LinkSigner` fast-path in `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner=true`, which permits the *current* linked signer to sign a new `LinkSigner` transaction. This breaks the invariant that only the subaccount owner should be able to mutate the signer-link relationship.

---

### Finding Description

**Step 1 — Entry point.**
`processTransactionImpl` handles `TransactionType.LinkSigner` at: [1](#0-0) 

It calls `validateSignedTx(..., true)` — the final `true` is `allowLinkedSigner`.

**Step 2 — `validateSignedTx` delegates to `validateSignature` with `allowLinkedSigner=true`.** [2](#0-1) 

**Step 3 — `validateSignature` passes the current linked signer to the verifier.** [3](#0-2) 

When `allowLinkedSigner=true`, it passes `getLinkedSigner(sender)` as the accepted alternate signer.

**Step 4 — `Verifier.validateSignature` accepts either the subaccount owner OR the linked signer.** [4](#0-3) 

The check is `recovered == owner || recovered == linkedSigner`. There is no restriction that `LinkSigner` mutations must come from the owner only.

**Step 5 — The new linked signer is written unconditionally.** [5](#0-4) 

`linkedSigners[subaccount]` is overwritten with whatever address the attacker specifies in `signedTx.tx.signer`.

---

### Impact Explanation

A compromised or malicious `signerA` (the current linked signer) can:
1. Craft a `SignedLinkSigner` transaction with `signer = signerB` (attacker-controlled address).
2. Sign it with `signerA`'s key — the signature passes because `recovered == linkedSigner`.
3. Submit via the sequencer's `processTransactionImpl` path.
4. `linkedSigners[subaccount]` is now `signerB`.

`signerB` can then perform all actions that a linked signer is authorized for (trading, withdrawals, further re-linking). The original owner is **not permanently locked out** (they can still sign directly as `address(uint160(bytes20(subaccount)))`), but the linked signer slot is silently hijacked without the owner's consent, violating the intended access-control invariant.

---

### Likelihood Explanation

Any party that holds a linked signer key — including a compromised API key, a third-party trading bot, or a builder integration — can execute this with a single signed transaction. No admin access, governance capture, or leaked owner key is required.

---

### Recommendation

For `TransactionType.LinkSigner`, pass `allowLinkedSigner=false` to `validateSignedTx`, or add an explicit check that the recovered signer equals the subaccount owner address:

```solidity
// In processTransactionImpl, LinkSigner branch:
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // <-- must be false; only owner may mutate the link
);
```

This mirrors the slow-mode `LinkSigner` path, which uses `validateSender` (an on-chain `msg.sender` check) rather than accepting a linked signer signature. [6](#0-5) 

---

### Proof of Concept

```
1. Owner (0xOwner) creates subaccount S.
2. Owner signs LinkSigner tx: sender=S, signer=signerA. Submitted → linkedSigners[S] = signerA.
3. Attacker (holding signerA key) crafts: SignedLinkSigner { tx: { sender: S, signer: signerB, nonce: N }, signature: signerA.sign(digest) }.
4. Sequencer calls processTransactionImpl with this tx.
5. validateSignedTx(S, N, tx, sig, true):
     - validateNonce passes (nonce N is correct).
     - validateSignature: getLinkedSigner(S) = signerA; recovered = signerA → passes.
6. linkedSigners[S] = signerB.
7. Assert: getLinkedSigner(S) == signerB. ✓
8. signerB can now sign withdrawals, orders, and further LinkSigner txs on behalf of S.
```

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
