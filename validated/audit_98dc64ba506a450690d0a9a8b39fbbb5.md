### Title
Fee-on-Transfer Token Deposit Inflates Internal Accounting, Enabling Protocol Insolvency — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.depositCollateral()` credits the user's internal balance with the nominal `txn.amount` from the signed transaction struct, without measuring the actual token amount received by the contract. If a fee-on-transfer token is listed as a collateral product, every deposit inflates the protocol's internal accounting relative to its real token holdings, eventually making the contract unable to honor all withdrawals.

---

### Finding Description

In `Clearinghouse.depositCollateral()`, the credited amount is taken directly from the decoded transaction struct:

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [1](#0-0) 

No balance-before / balance-after check is performed. The actual token transfer into the contract (executed at the Endpoint layer before this function is called) may deliver fewer tokens than `txn.amount` if the token charges a transfer fee. The full nominal amount is nonetheless credited to the depositor's subaccount via `spotEngine.updateBalance()`.

The only on-chain solvency guard is `assertUtilization()` in `SpotEngine`, called on every withdrawal:

```solidity
require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
``` [2](#0-1) 

This check operates entirely on internal normalized accounting (`totalDepositsNormalized`, `totalBorrowsNormalized`). It does not compare internal accounting against the contract's real ERC-20 balance. Once the inflated deposit is recorded, `assertUtilization` will pass even though the contract holds fewer tokens than it has credited.

Similarly, `manualAssert` in `Clearinghouse` only verifies internal state hash consistency and the insurance balance — it does not cross-check actual token balances against internal deposit totals. [3](#0-2) 

There is no guard in `SpotEngine.addOrUpdateProduct()` that prevents a fee-on-transfer token from being listed as a collateral product. [4](#0-3) 

---

### Impact Explanation

Each deposit of a fee-on-transfer token creates a gap: the contract's real token balance grows by `amount * (1 - fee_rate)`, but internal accounting grows by `amount`. Over many deposits, the cumulative shortfall grows. When users withdraw, the contract will eventually be unable to satisfy all withdrawal requests. The last withdrawers lose funds proportional to the total accumulated fee shortfall. This is a direct solvency/accounting corruption: the invariant `actual_token_balance(clearinghouse) >= totalDepositsNormalized * cumulativeDepositsMultiplierX18` is silently broken.

---

### Likelihood Explanation

The likelihood depends on whether a fee-on-transfer token is ever listed as a collateral product. The protocol has no on-chain enforcement preventing this. The `addOrUpdateProduct` owner function accepts any ERC-20 address. If such a token is listed — even inadvertently — every ordinary user deposit (an unprivileged action) triggers the accounting discrepancy. No special attacker capability is required beyond making a standard deposit.

---

### Recommendation

1. In `depositCollateral`, measure the actual received amount using a balance-before / balance-after pattern:
   ```solidity
   uint128 before = _balanceOf(address(token));
   // (token transfer already occurred at Endpoint layer — read post-transfer balance)
   uint128 actualReceived = _balanceOf(address(token)) - before;
   // use actualReceived instead of txn.amount
   ```
   Alternatively, record the pre-transfer balance at the Endpoint layer and pass the delta to `depositCollateral`.

2. Add an explicit check in `addOrUpdateProduct` that rejects tokens whose `transfer` delivers less than the requested amount (e.g., a test transfer pattern), or document and enforce an off-chain policy that only non-rebasing, non-fee tokens may be listed.

---

### Proof of Concept

1. Owner calls `SpotEngine.addOrUpdateProduct()` listing a token with a 1% transfer fee as `productId = 5`.
2. Alice submits a deposit of 1000 tokens. The Endpoint pulls 1000 tokens; the token contract retains 10 as fee; Clearinghouse receives 990. `depositCollateral` credits Alice with `1000 * multiplier`.
3. Bob does the same. Clearinghouse now holds 1980 real tokens but internal accounting shows 2000 deposited.
4. Alice calls `WithdrawCollateral` for 1000 tokens. `assertUtilization` passes (internal: deposits=2000 ≥ borrows=0). Transfer of 1000 succeeds; contract now holds 980.
5. Bob calls `WithdrawCollateral` for 1000 tokens. `assertUtilization` passes (internal: deposits=1000 ≥ borrows=0). Transfer of 1000 attempted — contract only holds 980. The `safeTransfer` reverts, or if the pool is shared with other products, Bob's withdrawal silently drains another product's reserves. [5](#0-4) [6](#0-5) [2](#0-1)
