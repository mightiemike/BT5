### Title
`only_query` Hardcoded to `false` in `BroadcastedTransaction`→`ExecutableTransactionInput` Conversion Causes Wrong Signature-Hash Domain in RPC Simulation — (`File: crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

The `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` conversion in the RPC layer unconditionally sets `only_query = false` for every broadcasted transaction submitted to `starknet_estimateFee` and `starknet_simulateTransactions`. The Starknet protocol requires simulation/estimation to compute the transaction hash with the 128th-bit "query" flag set in the version field, creating a distinct signature domain from real execution. Because the flag is hardcoded to `false`, the sequencer computes the execution-domain hash instead of the simulation-domain hash, passes it to the account's `__validate__` entry point, and returns an authoritative-looking but wrong result.

---

Because `only_query` is always `false`, the hash fed into `__validate__` during simulation is the **execution-domain hash** (version without the query bit), not the **simulation-domain hash** (version | 2¹²⁸). The `only_query` field also propagates into `TransactionInfo` and is exposed to the account contract via the `get_execution_info` syscall, so the account sees `only_query = false` even during simulation. [4](#0-3) 

The call path for `starknet_simulateTransactions` is:

1. `simulate_transactions` in `api_impl.rs` converts `BroadcastedTransaction` → `ExecutableTransactionInput` via `tx.try_into()` (hardcoded `only_query=false`).
2. `exec_simulate_transactions` calls `execute_transactions` with `tx_hashes = None`.
3. `execute_transactions` calls `calc_tx_hashes(txs, chain_id)` to compute hashes from the `ExecutableTransactionInput` (using `only_query=false`).
4. The resulting execution-domain hash is passed to `to_blockifier_tx` and then to `blockifier_tx.execute(...)`, which calls `__validate__`. [5](#0-4) [6](#0-5)

### Impact Explanation

**Cross-domain signature acceptance in simulation (analog to the external report):** A user who signs a transaction for real execution (no query bit) submits that same signature to `starknet_simulateTransactions` with `SKIP_VALIDATE` absent. The sequencer computes the execution-domain hash (because `only_query=false`), which matches the execution signature. The simulation's `__validate__` call succeeds and the RPC returns an authoritative-looking trace and fee estimate — but this result was produced using the wrong hash domain. The simulation is supposed to use a distinct domain so that execution signatures cannot be replayed in simulation and vice versa.

**False-negative for correctly-signed simulation transactions:** A user who correctly follows the Starknet spec and signs with the query bit (version | 2¹²⁸) will have their simulation rejected with a signature failure, because the sequencer computes the execution hash instead of the simulation hash. This is a wrong authoritative value returned by the RPC.

This matches the impact category: **High — RPC simulation returns an authoritative-looking wrong value** (wrong hash domain used in `__validate__`, wrong `only_query` field exposed via `get_execution_info`).

---

### Likelihood Explanation

Any caller of `starknet_simulateTransactions` or `starknet_estimateFee` with `SKIP_VALIDATE` absent is affected. Wallets and SDKs that follow the Starknet spec and sign simulation transactions with the query bit will consistently receive wrong results. The trigger requires no privilege — it is reachable by any unprivileged RPC caller.

---

### Recommendation

In `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput`, set `only_query = true` for all variants, since broadcasted transactions submitted to `estimate_fee` and `simulate_transactions` are by definition simulation/query transactions:

```rust
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    type Error = ErrorObjectOwned;
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        match value {
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
            BroadcastedTransaction::DeployAccount(tx) => Ok(Self::DeployAccount(tx.into(), true)),
            BroadcastedTransaction::Invoke(tx) => Ok(Self::Invoke(tx.into(), true)),
        }
    }
}
```

The same fix must be applied to `TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput` for the `DeclareV1` variant (line 506–508 in `mod.rs`). [7](#0-6) 

---

### Proof of Concept

1. Deploy an account contract whose `__validate__` checks `get_execution_info().tx_info.transaction_hash` against a pre-signed value using the query-bit version (version | 2¹²⁸).
2. Call `starknet_simulateTransactions` with that transaction and the signature computed over the simulation hash (query bit set).
3. Observe that `__validate__` fails — the sequencer computed the execution hash (no query bit) and the signature does not match.
4. Now call `starknet_simulateTransactions` with the same transaction but with the signature computed over the execution hash (no query bit).
5. Observe that `__validate__` succeeds — the execution-domain signature is accepted in the simulation domain, and the RPC returns a successful trace.

This demonstrates that the simulation domain is not isolated from the execution domain: execution signatures are accepted in simulation, and simulation signatures are rejected — the inverse of the intended behavior.
