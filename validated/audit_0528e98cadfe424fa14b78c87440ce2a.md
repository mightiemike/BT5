Audit Report

## Title
`PriceVelocityGuardExtension.beforeSwap` Writes Storage Before Velocity Check, Causing Permanent Swap Deadlock on Rapid Price Moves — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`beforeSwap` unconditionally writes `lastMidPriceX64` and `lastUpdateBlock` to storage at lines 57–58 before performing the velocity check at lines 60–74. When the check reverts, the EVM rolls back those writes, restoring `lastMidPriceX64` to the pre-move reference price. Every subsequent swap attempt reads the same stale reference, computes the same excess velocity, and reverts identically, permanently freezing all swaps through the pool until the pool admin manually calls `setLastMidPrice`.

## Finding Description

In `PriceVelocityGuardExtension.beforeSwap`, the storage writes occur unconditionally before the guard check:

```solidity
// Lines 57–58: written before any check
s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        // ... velocity computation ...
        if (actualSq > allowedSq) {
            revert PriceVelocityExceeded(actualSq, allowedSq); // line 73
        }
    }
}
```

When the revert fires at line 73, the EVM rolls back the writes at lines 57–58, restoring `lastMidPriceX64` to `prevMid`. On the next block's swap attempt, the oracle still returns the new price `P₁`, the same `delta = |P₁ - prevMid|` is computed, the same `actualSq > allowedSq` condition holds, and the same revert fires. This cycle repeats indefinitely across all blocks for as long as the oracle price remains beyond the configured cap relative to the stale stored reference. The guard is also direction-agnostic (the `bool zeroForOne` parameter at line 39 is unnamed and unused), so swaps in the direction that would restore equilibrium are blocked equally.

The only escape is a privileged call to `setLastMidPrice` (lines 29–34), which requires the pool admin to act before any swap can proceed.

## Impact Explanation

All swaps through any pool with this extension active are completely blocked in both directions for the duration of the rapid price move. This satisfies the "broken core pool functionality causing loss of funds or unusable swap flows" impact gate. Traders cannot execute any swap regardless of direction or size; arbitrageurs cannot rebalance the pool; LPs who wish to exit via swap are blocked. If `OracleValueStopLossExtension` is also active, its `afterSwap` hook never fires (no swap reaches settlement), silently bypassing stop-loss protection while LP balances remain exposed to the new oracle price on the next withdrawal.

## Likelihood Explanation

Any genuine rapid market move — a flash crash, a large liquidation cascade, or a high-volatility news event — that moves the oracle mid-price beyond `maxChangePerBlockE18` in a single block triggers the deadlock. No privileged access is required; any public swap call against a pool with this extension active is sufficient. On chains with fast block times (e.g., HyperEVM), even moderate sustained moves can exceed a tightly configured cap across consecutive blocks. The trigger is fully unprivileged and repeatable.

## Recommendation

Move the storage writes to after the velocity check so that a failed check leaves `lastMidPriceX64` unchanged:

```solidity
if (prevMid != 0) {
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        uint256 blockDiff = block.number - prevBlock;
        uint256 delta = midPrice > prevMid ? uint256(midPrice - prevMid) : uint256(prevMid - midPrice);
        uint256 changeE18 = (delta * 1e18) / uint256(prevMid);
        uint256 actualSq = changeE18 * changeE18;
        uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
        if (actualSq > allowedSq) revert PriceVelocityExceeded(actualSq, allowedSq);
    }
}
// Write only after passing the check
s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);
```

Additionally, consider making the guard direction-aware so that a rapid price move in one direction does not block swaps in the opposite direction that would restore the pool toward equilibrium.

## Proof of Concept

1. Deploy a pool with `PriceVelocityGuardExtension` configured with `maxChangePerBlockE18 = 0.05e18` (5% per block).
2. Execute one swap at oracle mid-price `P₀`. Extension writes `lastMidPriceX64 = P₀`, `lastUpdateBlock = N`.
3. In block `N+1`, the oracle reports mid-price `P₁ = 0.80 * P₀` (20% crash — legitimate market move).
4. Any swap attempt:
   - Reads `P₁` from oracle; writes `lastMidPriceX64 = P₁` (temporary).
   - Computes `changeE18 = 0.20e18`, `actualSq = 0.04e36`, `allowedSq = 0.0025e36 * 2 = 0.005e36`.
   - `0.04e36 > 0.005e36` → `revert PriceVelocityExceeded`.
   - Storage rolls back; `lastMidPriceX64` stays at `P₀`.
5. Repeat step 4 in blocks `N+2`, `N+3`, … — every attempt reverts identically.
6. Pool is frozen. Only `setLastMidPrice(pool, P₁)` by the pool admin unblocks it. [1](#0-0) [2](#0-1) [3](#0-2)
