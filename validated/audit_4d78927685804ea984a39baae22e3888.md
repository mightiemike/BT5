### Title
`ValidResourceBounds` Variant Divergence in Protobuf Deserialization Produces Wrong Transaction Hash for V3 Invoke Transactions with Zero L2/Data-Gas Bounds — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion silently downgrades a post-0.13.3 V3 invoke transaction to the legacy `ValidResourceBounds::L1Gas` variant whenever `l2_gas` and `l1_data_gas` are both zero. Because `get_tip_resource_bounds_hash` hashes a different number of resource-bound elements depending on the variant (2 for `L1Gas`, 3 for `AllResources`), the Poseidon hash computed during P2P block-sync deserialization diverges from the hash computed at RPC admission. A legitimate user submitting a V3 invoke with zero L2/data-gas bounds triggers this deterministically.

**Admission path always uses `AllResources`:**

`RpcInvokeTransactionV3` and `InternalRpcInvokeTransactionV3` both store `resource_bounds: AllResourceBounds` (never `L1Gas`). The `InvokeTransactionV3Trait` impl for `InternalRpcInvokeTransactionV3` always wraps it as `ValidResourceBounds::AllResources(self.resource_bounds)`: [4](#0-3) [5](#0-4) 

So the gateway always computes the hash with 3 resource bounds (4 Poseidon inputs: tip + L1_GAS + L2_GAS + L1_DATA_GAS), while the P2P deserialization path computes it with 2 resource bounds (3 Poseidon inputs: tip + L1_GAS + L2_GAS) for the same transaction data.

**The OS Cairo code enforces exactly 3 resource bounds for all V3 transactions:**

```cairo
with_attr error_message("Invalid number of resource bounds: {n_resource_bounds}.") {
    assert n_resource_bounds = 3;
}
``` [6](#0-5) 

This confirms the canonical hash always includes L1_DATA_GAS, matching the RPC-admission hash (H1), not the protobuf-deserialized hash (H2).

### Impact Explanation

A V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` is accepted by the gateway with hash H1 (computed over 3 resource bounds). When the containing block is later synced to a peer via P2P, the peer deserializes the transaction as `ValidResourceBounds::L1Gas` and recomputes hash H2 (over 2 resource bounds). H1 ≠ H2. Any hash-validation step during block sync will reject the block as having an invalid transaction hash, causing the peer to diverge from the canonical chain. This matches the **High** impact: "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."

### Likelihood Explanation

The trigger requires no privilege: any user can submit a V3 invoke transaction with zero L2 and data-gas bounds. The gateway's stateless validator does not reject zero resource bounds. The hash divergence is deterministic and reproducible for every such transaction. The TODO comment in the code (`// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2`) confirms this backward-compat branch is intentionally still live.

### Recommendation

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion should not infer the `L1Gas` variant based solely on zero values. For V3 transactions (post-0.13.3), always produce `AllResources`. A version-aware gate (e.g., passing the Starknet version alongside the protobuf message) should determine whether `L1Gas` is appropriate. Alternatively, the conversion for `InvokeV3` in the block-sync path should mirror the RPC path and reject `L1Gas` with `DEPRECATED_RESOURCE_BOUNDS_ERROR`, as is already done for `DeclareV3`: [7](#0-6)

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
