Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of end-user, breaking per-user allowlist enforcement â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is always `msg.sender` at the pool call site. When users route through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This breaks per-user access control in both directions: allowlisting the router grants all users access regardless of individual status, while not allowlisting the router blocks all legitimate allowlisted users from using the standard periphery.

## Finding Description
The call chain is confirmed by the production code:

1. `MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` with no user address forwarded â€” `msg.sender` inside the pool becomes the router. [1](#0-0) 

2. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, passing the router address as `sender`. [2](#0-1) 

3. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router â€” never the actual end user. [3](#0-2) 

The extension's stated purpose is per-swapper gating, but the `sender` argument it receives is structurally wrong when any intermediary contract (router, aggregator, multicall) calls the pool. [4](#0-3) 

The same structural flaw exists in `_afterSwap`, which also passes `msg.sender` as `sender` to after-swap extensions. [5](#0-4) 

No existing guard compensates for this: the extension has no mechanism to recover the true caller, and the pool passes no additional context identifying the originating user.

## Impact Explanation
Two fund-impacting outcomes arise simultaneously for any pool using `SwapAllowlistExtension` with `MetricOmmSimpleRouter`:

**Scenario A â€” Allowlist bypass (unauthorized access):** Pool admin must allowlist the router for any router-based swap to work. Once `allowedSwapper[pool][router] = true`, every caller through the router passes the check unconditionally. Non-KYC'd or blacklisted addresses freely extract value from a restricted pool.

**Scenario B â€” Broken core swap functionality:** If the admin does not allowlist the router (attempting per-user enforcement), every `exactInputSingle` call from a legitimately allowlisted user reverts with `NotAllowedToSwap`. The primary swap interface is non-functional for all users of any pool with a per-user allowlist. This constitutes broken core pool functionality causing loss of funds and unusable swap flows.

## Likelihood Explanation
`SwapAllowlistExtension` is a production periphery contract deployed alongside `MetricOmmSimpleRouter`. Any pool that configures this extension and expects users to interact via the router immediately exhibits one of the two broken behaviors. No special attacker capability is required â€” a standard `exactInputSingle` call is sufficient. The condition is triggered on every router-mediated swap.

## Recommendation
The router must forward the actual end-user address so the extension can check it.

**Option 1 â€” Pass user address through `extensionData`:** `MetricOmmSimpleRouter` encodes `msg.sender` into `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap` decodes the user address from `extensionData` and checks it instead of `sender`.

**Option 2 â€” Explicit `msgSender` parameter on `swap`:** Add an explicit `msgSender` parameter to `IMetricOmmPoolActions.swap()`, validated against `msg.sender` or a trusted router registry, and pass it as `sender` to extensions.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is KYC'd
  allowedSwapper[pool][bob]   = false  // bob is not KYC'd
  allowedSwapper[pool][router] = true  // required for any router swap to work

Attack (Scenario A â€” bypass):
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  â†’ router calls pool.swap(recipient, ...)          // msg.sender at pool = router
  â†’ pool calls _beforeSwap(msg.sender=router, ...)
  â†’ SwapAllowlistExtension: allowedSwapper[pool][router] == true â†’ PASSES
  â†’ bob's swap executes despite not being allowlisted

Broken functionality (Scenario B):
  admin does NOT allowlist router (tries per-user enforcement)
  alice calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  â†’ router calls pool.swap(recipient, ...)          // msg.sender at pool = router
  â†’ pool calls _beforeSwap(msg.sender=router, ...)
  â†’ SwapAllowlistExtension: allowedSwapper[pool][router] == false â†’ REVERTS NotAllowedToSwap
  â†’ alice cannot use the standard periphery despite being allowlisted
```

Foundry test: deploy pool with `SwapAllowlistExtension`, set `allowedSwapper[pool][alice]=true`, do not allowlist the router, call `router.exactInputSingle` as alice, assert revert `NotAllowedToSwap`. Then set `allowedSwapper[pool][router]=true`, call as bob (not allowlisted), assert swap succeeds.
