Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Enabling Allowlist Bypass by Any Caller — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` discards the `sender` argument and instead checks the caller-supplied `owner` address against the allowlist. Because `MetricOmmPool.addLiquidity` accepts an arbitrary `owner` with no `msg.sender == owner` enforcement, any unprivileged caller can bypass the deposit allowlist by specifying any allowlisted address as `owner`, paying tokens via the callback, and minting shares into that address's position without authorization.

## Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the first parameter (`sender`) and gates on `owner`: [1](#0-0) 

`msg.sender` inside the extension is the pool contract, so the check resolves to: *"is `owner` allowlisted for this pool?"* — not *"is the actual caller allowlisted?"*

`MetricOmmPool.addLiquidity` accepts a fully caller-controlled `owner` with no ownership enforcement: [2](#0-1) 

Notably, `removeLiquidity` does enforce `msg.sender != owner` at L206, but `addLiquidity` has no equivalent guard. The pool passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to the extension via `_beforeAddLiquidity`, but the extension ignores `sender` entirely.

**Exploit path:**
1. Pool admin deploys a pool with `DepositAllowlistExtension`; only `alice` is allowlisted.
2. Unauthorized `bob` calls `pool.addLiquidity(owner=alice, salt=X, deltas=..., callbackData=..., extensionData=...)` directly.
3. Extension receives `owner=alice`, finds her on the allowlist → returns success.
4. Pool mints shares into position `(alice, X)`.
5. Bob's `metricOmmModifyLiquidityCallback` is invoked; Bob pays the tokens.
6. Allowlist bypassed: Bob deposited into a curated pool without being allowlisted, and Alice has an unsolicited position at a salt she never chose.

The `SwapAllowlistExtension` demonstrates the correct pattern — it checks `sender` (the actual caller), not the recipient/owner: [3](#0-2) 

## Impact Explanation

The pool admin's curation invariant is broken. Any unprivileged EOA or contract can deposit into a pool restricted to specific depositors by supplying any allowlisted address as `owner`. This constitutes an admin-boundary break (allowlist/role check bypassed by an unprivileged path). Secondary impacts include unsolicited position creation under a victim's address at arbitrary salts, and pool bin state manipulation affecting all LPs' returns.

## Likelihood Explanation

Exploitation requires no special privilege, no flash loan, and no oracle manipulation. Any caller can invoke `pool.addLiquidity` directly with `owner` set to any allowlisted address. Allowlisted addresses are discoverable on-chain from `AllowedToDepositSet` events. The attack is repeatable at will.

## Recommendation

Gate on `sender` (the actual caller/payer), not `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Because `MetricOmmPoolLiquidityAdder` is the `sender` when routing through the adder, the adder contract itself must also be allowlisted per pool, with user-level policy enforced by the adder's own `_validateOwner` / payer binding.

## Proof of Concept

```solidity
// Pool has DepositAllowlistExtension; only alice is allowlisted. Bob is NOT.

address alice = makeAddr("alice");
address bob   = makeAddr("bob");

vm.prank(poolAdmin);
depositAllowlist.setAllowedToDeposit(address(pool), alice, true);

deal(token0, bob, 1e18);
deal(token1, bob, 1e18);
vm.startPrank(bob);
IERC20(token0).approve(address(pool), type(uint256).max);
IERC20(token1).approve(address(pool), type(uint256).max);

// Bob calls addLiquidity with owner=alice — extension checks alice (allowlisted) → passes
// Bob's callback pays the tokens
pool.addLiquidity(
    alice,           // owner — allowlisted, so extension passes
    uint80(999),     // salt alice never used
    deltas,
    abi.encode(bob), // callback data: bob is payer
    ""
);
// Bob deposited into a curated pool without being allowlisted.
// Alice now has an unsolicited position at salt 999.
```
