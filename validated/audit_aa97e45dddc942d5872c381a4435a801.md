### Title
Reverting ERC20 Recipient Permanently Blocks Sequencer Withdrawal Processing ŌĆö (`core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.handleWithdrawTransfer()` uses a push-based `safeTransfer` to deliver tokens directly to the user-supplied `sendTo` address. If `sendTo` is a smart contract whose token-receipt hook reverts, the sequencer's `submitWithdrawal` call will always revert, permanently preventing that withdrawal index from being marked and blocking sequencer batch processing for all users sharing that batch.

---

### Finding Description

`BaseWithdrawPool.handleWithdrawTransfer` unconditionally pushes tokens to the recipient: [1](#0-0) 

```solidity
function handleWithdrawTransfer(
    IERC20Base token,
    address to,
    uint128 amount
) internal virtual {
    token.safeTransfer(to, uint256(amount));
}
```

This is called directly from `submitWithdrawal`, which is the sequencer's path for settling user withdrawals: [2](#0-1) 

```solidity
function submitWithdrawal(...) public {
    require(msg.sender == clearinghouse);
    if (markedIdxs[idx]) { return; }
    markedIdxs[idx] = true;
    minIdx = idx;
    handleWithdrawTransfer(token, sendTo, amount); // ŌåÉ push to external address
}
```

`markedIdxs[idx] = true` is set **before** the transfer. If `handleWithdrawTransfer` reverts (because `sendTo` is a contract that reverts on ERC20 receipt), the entire transaction reverts ŌĆö including the `markedIdxs[idx] = true` assignment. The withdrawal index is never marked, so the sequencer cannot advance past it and will keep failing on retry.

The same push pattern is present in the fast-withdrawal path: [3](#0-2) 

And in `Clearinghouse.withdrawCollateral`, which calls `handleWithdrawTransfer` **before** decrementing the user's on-chain balance: [4](#0-3) 

The `Clearinghouse.handleWithdrawTransfer` overload first moves tokens to `withdrawPool`, then calls `submitWithdrawal` to push to `sendTo`: [5](#0-4) 

If `sendTo` reverts, the entire `withdrawCollateral` call reverts atomically ŌĆö no balance is decremented, no withdrawal is settled, and the sequencer batch is blocked.

---

### Impact Explanation

A user who registers a reverting contract as their `sendTo` address (or whose `sendTo` becomes a reverting contract after the withdrawal is queued) causes the sequencer's `submitWithdrawal` call to always revert. Because `markedIdxs[idx]` is never durably set, the sequencer cannot skip or advance past the stuck index. If the Endpoint processes withdrawals in sequential batches (the standard pattern for this architecture), a single stuck withdrawal blocks all subsequent withdrawals for every user in the queue. Funds are not directly stolen, but all pending withdrawals become unrecoverable until the stuck entry is resolved ŌĆö which the protocol has no on-chain mechanism to do.

---

### Likelihood Explanation

Any user can supply an arbitrary `sendTo` address when initiating a withdrawal. A contract address that reverts on ERC20 `transfer` (e.g., a contract with no token-receipt logic, a paused contract, or a deliberately malicious contract) is trivially deployable. No privileged access is required. The trigger is a normal user withdrawal flow through the Endpoint.

---

### Recommendation

Replace the push pattern in `BaseWithdrawPool.handleWithdrawTransfer` with a pull pattern:

1. Instead of calling `token.safeTransfer(to, amount)` directly, store the pending amount in a `mapping(address => mapping(address => uint256)) pendingWithdrawals` (token ŌåÆ recipient ŌåÆ amount).
2. Add a permissionless `claimWithdrawal(address token)` function that lets recipients pull their own funds.
3. This ensures a single reverting recipient cannot block the sequencer's batch processing or other users' withdrawals. [1](#0-0) 

---

### Proof of Concept

1. Attacker deploys `RevertOnReceive` ŌĆö a contract with a `transfer`/`transferFrom` hook that always reverts.
2. Attacker opens a position on Nado and initiates a `withdrawCollateral` with `sendTo = address(RevertOnReceive)`.
3. The sequencer picks up the withdrawal and calls `Clearinghouse.withdrawCollateral` ŌåÆ `handleWithdrawTransfer` ŌåÆ `BaseWithdrawPool.submitWithdrawal` ŌåÆ `token.safeTransfer(RevertOnReceive, amount)`.
4. `RevertOnReceive` reverts. The entire transaction reverts. `markedIdxs[idx]` is never set.
5. The sequencer retries ŌĆö same result. The withdrawal index is permanently stuck.
6. All other users' withdrawals queued after this index are blocked indefinitely. [2](#0-1) [1](#0-0)
