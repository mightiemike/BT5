### Title
Fork Difficulty Validation Uses Stale Mainchain Ancestor Instead of Fork's Actual Ancestor - (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary

When validating the proof-of-work difficulty for a fork block, `get_next_work_required` calls `get_header_by_height` to retrieve the block at the start of the difficulty period. However, `get_header_by_height` always returns the **mainchain** block at that height, not the fork's actual ancestor. When a fork diverges before the difficulty adjustment boundary, the two differ, causing the contract to compute the expected difficulty using the wrong block's timestamp. This is a cross-module desynchronization: the mainchain height-to-header mapping is used as a proxy for fork ancestry, but it is stale with respect to the fork's own history.

The codebase itself acknowledges this with an explicit TODO in `dogecoin.rs`:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

### Finding Description

The `get_header_by_height` implementation always resolves through `mainchain_height_to_header`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

This is called from `get_next_work_required` in all three chain modules to obtain the block at the start of the difficulty period:

- **Bitcoin** (`bitcoin.rs` line 81): `blocks_getter.get_header_by_height(first_block_height)` where `first_block_height = prev_height - (interval - 1)`. [2](#0-1) 

- **Litecoin** (`litecoin.rs` line 88): same pattern. [3](#0-2) 

- **Dogecoin** (`dogecoin.rs` line 292–295): same pattern, with the TODO comment. [4](#0-3) 

The call chain is: `submit_blocks` → `submit_block_header` → `check_target` → `check_pow` → `get_next_work_required` → `get_header_by_height`. [5](#0-4) 

When a fork block is being validated, the fork has not yet been promoted to the mainchain. If the fork diverged at height H and the difficulty period boundary is at height B where B > H, then the fork has its own block at height B (different from the mainchain block at B). But `get_header_by_height(B)` returns the mainchain block at B, not the fork's ancestor. The timestamp of the mainchain block at B is used in the difficulty formula instead of the fork's block at B.

For **Dogecoin after height 145,000**, the difficulty adjusts every single block (`difficulty_adjustment_interval = 1`), so `height_first = prev_height - 1`. This means for any fork block at height H+2 or later (where the fork diverged at H), the difficulty is computed using the mainchain block at `H`'s timestamp rather than the fork's block at `H`'s timestamp. [6](#0-5) 

### Impact Explanation

An attacker submitting fork blocks can craft fork block timestamps such that, when combined with the fixed mainchain ancestor's timestamp (which is public on-chain), the Dogecoin difficulty formula yields a lower difficulty target than the fork's actual history warrants. The attacker mines these easier blocks, and the contract accepts them because the difficulty check passes against the wrong ancestor. If the fork accumulates sufficient chainwork, `reorg_chain` is triggered, replacing the mainchain with a chain whose blocks were validated against incorrect difficulty. After the reorg, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` operate on a corrupted canonical chain, potentially confirming transactions in blocks that did not meet the correct PoW requirement. [7](#0-6) 

### Likelihood Explanation

For Dogecoin (post-height 145,000), the trigger condition is met for every fork block at depth ≥ 2 from the divergence point — no special setup is required beyond submitting a fork. The entry point is the public `submit_blocks` function, callable by any NEAR account. For Bitcoin and Litecoin, the fork must span a full 2016-block difficulty period, which is a higher bar but not impossible for a well-resourced attacker. [8](#0-7) 

### Recommendation

Replace `get_header_by_height` (mainchain lookup) with an ancestor walk via `get_prev_header` when computing the difficulty period boundary block during fork validation. The correct approach is to walk backwards from the fork's `prev_block_header` using `prev_block_hash` links until reaching the target height, rather than using the height-indexed mainchain map. This is exactly what the TODO comment in `dogecoin.rs` flags as unverified. [9](#0-8) 

### Proof of Concept

**Dogecoin scenario (post-height 145,000):**

1. Mainchain has blocks at heights 0…N. Mainchain block at height N has timestamp `T_main_N`.
2. Attacker submits a fork block at height N that diverges from the mainchain (different `prev_block_hash` pointing to the same parent as the mainchain block at N-1). This fork block has timestamp `T_fork_N`.
3. Attacker submits a fork block at height N+1. Its difficulty is computed as:
   - `height_first = N` (prev_height - 1)
   - `get_header_by_height(N)` returns the **mainchain** block at N with timestamp `T_main_N`
   - Time difference used: `T_fork_N+1 - T_main_N` (wrong — should be `T_fork_N+1 - T_fork_N`)
4. Attacker sets `T_fork_N` to be much earlier than `T_main_N`. Then `T_fork_N+1 - T_main_N` is smaller than `T_fork_N+1 - T_fork_N`, yielding a **higher** difficulty than the fork's actual history warrants — causing valid fork blocks to be rejected.
5. Conversely, if `T_fork_N` is later than `T_main_N`, the computed time difference is larger, yielding a **lower** difficulty — allowing the attacker to mine easier blocks that pass the contract's check.
6. With enough such blocks, `current_header.chain_work > total_main_chain_chainwork` triggers `reorg_chain`, installing the attacker's low-difficulty chain as the new mainchain. [10](#0-9) [11](#0-10)

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
