### Title
Blacklisted Withdrawal Recipient Permanently Locks User Funds and Blocks Slow-Mode Queue — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitWithdrawal` and `BaseWithdrawPool.submitFastWithdrawal` both perform a direct push-payment `token.safeTransfer(to, amount)` to the caller-supplied `sendTo` address. If that address is blacklisted by the token contract (e.g., USDC's built-in blacklist), the transfer reverts, the sequencer's slow-mode withdrawal transaction cannot be executed, and the affected user's collateral is permanently unrecoverable through the normal withdrawal path.

---

### Finding Description

`BaseWithdrawPool.handleWithdrawTransfer` is the single exit point for all token withdrawals from the protocol:

```solidity
// BaseWithdrawPool.sol line 184-190
function handleWithdrawTransfer(
    IERC20Base token,
    address to,
    uint128 amount
) internal virtual {
    token.safeTransfer(to, uint256(amount));
}
```

This is called unconditionally from two paths:

**Path 1 — Slow-mode (sequencer-submitted) withdrawal:**

`Clearinghouse.withdrawCollateral` → `Clearinghouse.handleWithdrawTransfer` → `BaseWithdrawPool.submitWithdrawal` → `handleWithdrawTransfer(token, sendTo, amount)`. [1](#0-0) [2](#0-1) 

**Path 2 — Fast withdrawal (permissionless caller):**

`BaseWithdrawPool.submitFastWithdrawal` → `handleWithdrawTransfer(token, sendTo, transferAmount)`. [3](#0-2) 

In both paths, if `sendTo` is blacklisted by USDC (or any token with a blacklist), `token.safeTransfer` reverts. Because `ERC20Helper.safeTransfer` uses a low-level `call` and then `require(success && ...)`, the revert propagates up and the entire transaction fails. [4](#0-3) 

For the slow-mode path, the sequencer's transaction that calls `withdrawCollateral` reverts. The sequencer has no on-chain mechanism to skip or redirect this withdrawal — it must re-submit the same transaction, which will keep reverting. The user's collateral balance in `SpotEngine` is never decremented (the `updateBalance` call at line 412 of `Clearinghouse.sol` is part of the same transaction and also reverts), so the funds are frozen in the clearinghouse with no on-chain recovery path. [5](#0-4) 

---

### Impact Explanation

A user whose withdrawal address is blacklisted by USDC (or any supported blacklist-enabled token) cannot withdraw their collateral. The sequencer's slow-mode withdrawal transaction reverts on every attempt. There is no on-chain function allowing the user to redirect their withdrawal to a different address after the fact. The `WithdrawCollateralV2` transaction type does include a `sendTo` field, but submitting a new transaction requires the sequencer to process it — and if the original stuck withdrawal is blocking the sequential slow-mode queue, no subsequent transactions can be processed either. [6](#0-5) 

Funds are effectively locked: the user cannot withdraw, and if the slow-mode queue is sequential (consistent with the sequencer-based architecture), all other users' pending slow-mode withdrawals are also blocked.

---

### Likelihood Explanation

USDC blacklisting is a real, documented mechanism used by Circle. Addresses are blacklisted for sanctions compliance, Tornado Cash usage, or other regulatory reasons. The Nado protocol explicitly supports USDC as a collateral token. A single blacklisted user is sufficient to trigger this condition. No attacker action is required — a user can be blacklisted by Circle at any time after depositing.

---

### Recommendation

Replace the push-payment pattern with a pull-payment (withdrawal) pattern:

1. In `submitWithdrawal`, instead of immediately calling `handleWithdrawTransfer`, store the claimable amount in a mapping: `claimable[sendTo][productId] += amount`.
2. Add a `claimWithdrawal(uint32 productId, address alternateRecipient)` function that allows the original `sendTo` address to pull their funds to any address they specify.
3. For `submitFastWithdrawal`, apply the same pattern so a failed transfer does not revert the entire transaction and waste the provider's fee.

This mirrors the recommendation in the referenced Escrow report and is the standard mitigation for push-payment blacklist vulnerabilities.

---

### Proof of Concept

1. Alice deposits 10,000 USDC into Nado and has a positive collateral balance in `SpotEngine`.
2. Alice signs a `WithdrawCollateral` transaction. The sequencer queues it as a slow-mode transaction.
3. Circle blacklists Alice's address (e.g., for sanctions compliance).
4. The sequencer calls `Endpoint.executeSlowModeTransaction`, which routes to `Clearinghouse.withdrawCollateral`.
5. `Clearinghouse.handleWithdrawTransfer` calls `token.safeTransfer(withdrawPool, amount)` — succeeds.
6. `BaseWithdrawPool.submitWithdrawal` calls `handleWithdrawTransfer(token, alice, amount)` → `token.safeTransfer(alice, amount)` → **REVERT** (Alice is blacklisted).
7. The entire transaction reverts. Alice's `SpotEngine` balance is unchanged. The slow-mode queue index does not advance.
8. Every subsequent call by the sequencer to process this withdrawal reverts identically.
9. Alice's 10,000 USDC is permanently unrecoverable through the normal withdrawal path. If the slow-mode queue is sequential, all other users' pending withdrawals are also blocked. [7](#0-6) [1](#0-0)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
