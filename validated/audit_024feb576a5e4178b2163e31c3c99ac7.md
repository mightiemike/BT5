Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of originating user, making per-pool allowlist fully bypassable via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the pool's `swap` call. When any user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract. If the pool admin allowlists the router (the only way to enable router-mediated swaps for legitimate users), every address in the system can bypass the allowlist by calling the router. There is no configuration that simultaneously permits allowlisted users to use the router and blocks non-allowlisted users from doing the same.

## Finding Description

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, passing the direct caller of `swap` as `sender`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that `sender` value and forwards it verbatim to the configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — `msg.sender` here is the pool, `sender` is whoever called `pool.swap`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly, making the pool's `msg.sender` the router contract, not the originating user: [4](#0-3) 

The extension therefore checks `allowedSwapper[pool][router_address]`. If the admin allowlists the router (required for any legitimate user to swap via the router), the check passes for every caller regardless of their identity. The `extensionData` field forwarded by the router is user-controlled (`params.extensionData`) and carries no authenticated identity: [5](#0-4) 

No existing guard in `SwapAllowlistExtension` inspects the originating EOA or validates any signature in `extensionData`.

## Impact Explanation

Any pool deploying `SwapAllowlistExtension` to restrict swap access to vetted counterparties is rendered fully open once the router is allowlisted. Non-allowlisted addresses can execute swaps, receive output tokens, and trade against LP liquidity that was explicitly deposited under the assumption of restricted access. This is a direct, fund-impacting loss of the pool's primary access-control invariant — LPs suffer adverse selection from unrestricted counterparties they never consented to trade against.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point shipped with the protocol. Any pool admin who deploys a restricted pool and then tries to make it usable through the router will naturally allowlist the router address. The bypass requires no special privileges, no flash loans, and no unusual token behavior — any address can call `exactInputSingle` on the router. The misconfiguration is not obvious: the admin believes they are enabling router access for allowlisted users, not opening the pool to everyone.

## Recommendation

The extension must verify the identity of the human caller, not the intermediary contract. Two viable approaches:

1. **Require the actual user address in `extensionData` with an off-chain signature** — the extension verifies the signature and checks the recovered address against the allowlist. The router must forward the signed payload.
2. **Trusted router embedding** — require the router to embed `msg.sender` in `extensionData` in a tamper-evident way (e.g., verified against a factory-registered router registry), and have the extension read the user address from that field when `sender` is a known periphery contract.

The simplest safe default is to document that `SwapAllowlistExtension` is incompatible with any shared router and must only be used with direct pool calls or a purpose-built router that embeds and authenticates the originating user address.

## Proof of Concept

```
Setup:
  pool deployed with SwapAllowlistExtension
  pool admin: extension.setAllowedToSwap(pool, alice, true)
  pool admin: extension.setAllowedToSwap(pool, router, true)
    (required so alice can use the router)

Attack (bob is not allowlisted):
  bob calls: router.exactInputSingle({pool: pool, recipient: bob, ...})

  router calls: pool.swap(bob, zeroForOne, amount, priceLimit, "", extensionData)
    // pool.msg.sender == router

  pool calls: _beforeSwap(sender=router, ...)
  ExtensionCalling encodes sender=router and calls extension

  extension checks: allowedSwapper[pool][router] == true  ✓ passes

  bob receives output tokens despite never being allowlisted.
```

Foundry test plan: deploy `SwapAllowlistExtension`, configure a pool with it, allowlist only `alice` and the router, then call `exactInputSingle` from `bob` (not allowlisted) and assert the swap succeeds — confirming the allowlist is bypassed.
