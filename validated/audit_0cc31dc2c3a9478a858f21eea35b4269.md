### Title
Residual Positive Zero-Weight Spot Balances Bypass Finalization Checks, Allowing Socialization of Liabilities While Liquidatee Retains Assets — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`_finalizeSubaccount` unconditionally skips all spot products whose `longWeightInitialX18 == 0` in both its pre-finalization check loop and its `canLiquidateMore` guard loop. Neither `perpEngine.socializeSubaccount` nor `spotEngine.socializeSubaccount` clears a **positive** zero-weight spot balance. Finalization therefore completes while the liquidatee retains real token value, and their quote liabilities are socialized across other depositors.

---

### Finding Description

`_finalizeSubaccount` contains two loops over spot product IDs.

**Loop 1** (unconditional, lines 301–311):

```solidity
if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
    continue;   // zero-weight product: no check at all
}
require(balance.amount <= 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
``` [1](#0-0) 

**Loop 2** (only when `v.canLiquidateMore`, lines 372–384):

```solidity
if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
    continue;   // zero-weight product: no check at all
}
require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
``` [2](#0-1) 

In both cases a zero-weight spot product with `balance.amount > 0` is silently skipped.

After the loops, `perpEngine.socializeSubaccount` only touches perp `vQuoteBalance`: [3](#0-2) 

And `spotEngine.socializeSubaccount` only processes `balance.amount < 0` (negative/liability balances): [4](#0-3) 

A positive zero-weight spot balance is therefore **never touched** by any step in `_finalizeSubaccount`. The function returns `true` and the liquidatee retains the balance.

The normal liquidation path also cannot clear these positions: `_assertCanLiquidateAmount` explicitly reverts for zero-weight spot products:

```solidity
require(
    spotEngine.getRisk(spotId).longWeightInitialX18 != 0,
    ERR_INVALID_PRODUCT
);
``` [5](#0-4) 

This creates a complete dead-end: zero-weight spot balances cannot be liquidated through the normal path, and finalization ignores them.

---

### Impact Explanation

A liquidatee who holds a positive balance in a delisted (zero-weight) spot product can have their quote liabilities fully socialized across other depositors while retaining the token value of the zero-weight spot balance. This is a direct, measurable asset transfer from the insurance fund and/or other depositors to the liquidatee. The invariant stated in the code comments — *"all spot assets have closed"* — is violated. [6](#0-5) 

---

### Likelihood Explanation

The precondition is that at least one spot product with `longWeightInitialX18 == 0` exists and a subaccount holds a positive balance in it. Products are delisted by setting their weight to zero; any user who held the product before delisting retains a positive balance that can never be liquidated. This is a realistic, non-adversarial state that arises from normal protocol operation (product delisting). No privileged access or special configuration is required beyond the product existing in the engine's `productIds` array.

---

### Recommendation

In both loops, remove the `continue` for zero-weight products, or add an explicit assertion that their balance is also zero before allowing finalization to proceed:

```solidity
// In both loops, replace:
if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
    continue;
}
// With a check that covers zero-weight products too:
require(balance.amount <= 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
```

Alternatively, add a dedicated pre-finalization step that transfers or burns any residual positive zero-weight spot balance (e.g., routes it to the insurance fund) before the socialization steps execute.

---

### Proof of Concept

1. Deploy the protocol with a spot product `P` that initially has `longWeightInitialX18 > 0`.
2. Liquidatee deposits and holds `+100 P`.
3. Admin sets `P`'s `longWeightInitialX18 = 0` (delist).
4. Liquidatee takes on a large perp short that drives their maintenance health negative.
5. Liquidators close all perp positions and all non-zero-weight spot positions.
6. Liquidator submits `LiquidateSubaccount` with `productId = type(uint32).max`.
7. `_finalizeSubaccount` runs: both loops skip `P` (line 303 and line 379 `continue`).
8. `perpEngine.socializeSubaccount` and `spotEngine.socializeSubaccount` do not touch `P`'s positive balance.
9. Assert: `spotEngine.getBalance(P, liquidatee).amount == 100` — the balance is intact.
10. Assert: the liquidatee's quote liability has been socialized (other depositors' `cumulativeDepositsMultiplierX18` is reduced).

The liquidatee retains `+100 P` while their debt was borne by the protocol.

                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
        }
