Audit Report

## Title
Missing Minimum-Output Guard on `removeLiquidity()` While `addLiquidity` Path Is Protected — (`metric-core/contracts/MetricOmmPool.sol`, `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`MetricOmmPool.removeLiquidity()` accepts only a share count (`LiquidityDelta`) and returns token amounts with no `minAmount0`/`minAmount1` guard. `MetricOmmPoolLiquidityAdder` provides slippage-protected wrappers exclusively for `addLiquidity`, leaving LPs with no on-chain mechanism to bound the output of a removal. Because bin composition is determined by every swap that crosses the active bin, an LP can receive materially less than their simulation predicted with no revert path.

## Finding Description
`MetricOmmPoolLiquidityAdder` exposes two protected deposit entry points (`addLiquidityExactShares`, `addLiquidityWeighted`), both of which enforce `maxAmountToken0`/`maxAmountToken1` caps inside `metricOmmModifyLiquidityCallback` at lines 165–167:

```solidity
if (amount0Delta > max0 || amount1Delta > max1) {
    revert MaxAmountExceeded(amount0Delta, amount1Delta, max0, max1);
}
```

No analogous wrapper exists for withdrawals. `MetricOmmPool.removeLiquidity()` (lines 199–212) accepts `(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)`, calls `LiquidityLib.removeLiquidity`, and returns `(amount0Removed, amount1Removed)` with no minimum-output check before or after the library call:

```solidity
(amount0Removed, amount1Removed) = LiquidityLib.removeLiquidity(
    _liquidityContext(), owner, salt, deltas, binTotals, _binStates, _binTotalShares, _positionBinShares
);
_afterRemoveLiquidity(msg.sender, owner, salt, deltas, amount0Removed, amount1Removed, extensionData);
// ← no minAmount0 / minAmount1 check
```

The `_afterRemoveLiquidity` hook and `extensionData` do not provide a standard slippage gate; they are extension callbacks, not user-controlled output bounds. The LP's only recourse is off-chain simulation, which is invalidated by any swap that crosses the active bin between simulation and execution.

**Exploit path:**
1. LP holds shares in bin 0. Off-chain view returns `amount0 ≈ 1000 USDC`, `amount1 ≈ 0.5 WETH`.
2. Before the LP's transaction lands, a large exact-input swap buys all token0 from bin 0, setting `token0BalanceScaled = 0`.
3. LP's `removeLiquidity` executes. `LiquidityLib.removeLiquidity` computes proportional share of the now-depleted bin: `amount0Removed = 0`, `amount1Removed = 0.5 WETH`.
4. No revert occurs. LP receives 0 USDC instead of the expected 1000 USDC.

## Impact Explanation
Direct loss of LP principal: the LP burns shares and receives fewer tokens than their position was worth at simulation time, with no on-chain revert path. The lost value (the depleted token0 leg) is captured by the swap counterparty or MEV searcher. This meets the "direct loss of user principal / owed LP assets" impact gate.

## Likelihood Explanation
No privileged actor or malicious setup is required. Any swap that crosses the active bin — including normal sequencer ordering on an L2 — shifts bin composition. The Metric OMM pool is oracle-anchored and designed for active trading, so bin composition changes are frequent. Any LP who simulates a removal and submits the transaction is exposed. The condition is repeatable and requires only ordinary market activity.

## Recommendation
Add a periphery `removeLiquidity` wrapper in `MetricOmmPoolLiquidityAdder` that accepts `minAmountToken0` and `minAmountToken1` and reverts if the pool returns less:

```solidity
function removeLiquidityExactShares(
    address pool,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmountToken0,
    uint256 minAmountToken1,
    bytes calldata extensionData
) external returns (uint256 amount0Removed, uint256 amount1Removed) {
    (amount0Removed, amount1Removed) =
        IMetricOmmPoolActions(pool).removeLiquidity(msg.sender, salt, deltas, extensionData);
    if (amount0Removed < minAmountToken0 || amount1Removed < minAmountToken1)
        revert InsufficientOutput(amount0Removed, amount1Removed, minAmountToken0, minAmountToken1);
}
```

Alternatively, add `minAmount0`/`minAmount1` parameters directly to `MetricOmmPool.removeLiquidity` and revert after `LiquidityLib.removeLiquidity` returns.

## Proof of Concept
1. Deploy pool with token0=USDC, token1=WETH. LP deposits into bin 0 with balanced composition.
2. LP calls an off-chain view: estimated removal = 1000 USDC + 0.5 WETH.
3. Attacker (or natural trader) submits a large exact-input swap buying all token0 from bin 0 before the LP's tx lands.
4. LP's `removeLiquidity` executes at `MetricOmmPool` lines 199–212: `LiquidityLib.removeLiquidity` returns `amount0Removed = 0`, `amount1Removed = 0.5 WETH`.
5. No revert. LP loses the entire USDC leg with no recourse.
