### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Guard — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, callable public NEAR function despite being deprecated since v0.5.0. The `#[deprecated]` Rust attribute is a compiler-only warning; it imposes no runtime restriction. Any unprivileged NEAR caller can invoke v1 directly, completely bypassing the coinbase Merkle proof check that `verify_transaction_inclusion_v2` was introduced to enforce. This allows an attacker to obtain a `true` proof result for a transaction that was never included in any Bitcoin block.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions). It enforces a coinbase proof guard before delegating to v1:

```rust
// v2 guard — verifies coinbase is at index 0
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())   // then calls v1
``` [1](#0-0) 

However, v1 is still exposed as a public `#[pause]`-gated function with no access restriction beyond the pause flag:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [2](#0-1) 

The function's own docstring acknowledges the danger:

> **Warning:** This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. [3](#0-2) 

The root cause is that `compute_root_from_merkle_proof` in `merkle-tools` is a pure positional hash-chain computation. It has no mechanism to distinguish a leaf (real transaction hash) from an internal node hash:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
    current_hash
}
``` [4](#0-3) 

Because `compute_hash` simply concatenates two 32-byte values and double-SHA256s them — identical to how Bitcoin builds every internal node — any internal node hash `N` at tree depth `d` is a valid input that will produce the correct Merkle root when paired with the correct sibling proof path.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) to gate a fund release, cross-chain bridge payout, or other security-critical action can be deceived into accepting a forged proof. The attacker supplies an internal Merkle node hash as `tx_id` and a crafted proof path; the function returns `true` against a real, confirmed Bitcoin block without any corresponding on-chain Bitcoin transaction ever having occurred. The corrupted value is the **proof result** (`true` for a non-existent transaction), which downstream contracts consume as authoritative.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` directly. The only prerequisite is knowledge of a confirmed Bitcoin block's Merkle tree structure, which is fully public. The attack is deterministic and requires no brute-force. Downstream integrators who followed the original (pre-v2) API or who call the contract generically are silently exposed.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or gate it with `#[private]` so it is only callable by the contract itself (as v2 already does internally via `self.verify_transaction_inclusion`). The intended public entry point is `verify_transaction_inclusion_v2`.

```diff
-#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
-#[pause]
-pub fn verify_transaction_inclusion(&self, ...) -> bool {
+#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
+#[pause]
+#[private]   // only callable by the contract itself
+pub fn verify_transaction_inclusion(&self, ...) -> bool {
```

### Proof of Concept

Given a confirmed Bitcoin block with transactions `[T0, T1, T2, T3]`:

- Internal node `N01 = double_sha256(T0 || T1)`
- Internal node `N23 = double_sha256(T2 || T3)`
- Merkle root `R = double_sha256(N01 || N23)`

An attacker calls `verify_transaction_inclusion` (v1) directly with:

| Parameter | Value |
|---|---|
| `tx_id` | `N01` (hash of internal node — **not a real transaction**) |
| `tx_block_blockhash` | any confirmed block hash stored in the contract |
| `tx_index` | `0` |
| `merkle_proof` | `[N23]` |
| `confirmations` | `1` |

`compute_root_from_merkle_proof(N01, 0, [N23])` computes `double_sha256(N01 || N23) = R`, which equals the block's stored `merkle_root`. The function returns `true`. No such transaction exists on Bitcoin. The coinbase guard in v2 would have rejected this call because `N01` at index `0` is not the coinbase transaction. [5](#0-4) [6](#0-5)

        // Check requested confirmations. No need to compute proof if insufficient confirmations.

            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
