Audit Report

## Title
Single `MAX_REF_STALENESS` Applied to Both Legs of Synthetic Pair Allows Stale-Price Execution Against LPs — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary
`AnchoredPriceProvider` supports a two-feed synthetic ratio mode where `mid = price(baseFeedId) / price(quoteFeedId)`. Both legs are validated by `_readLeg`, which applies the single immutable `MAX_REF_STALENESS` to each feed. Because oracle feeds have different heartbeats, no single threshold can be simultaneously correct for both legs; when the deployer accommodates the slower feed, the faster feed can be significantly stale at swap time, producing a corrupted synthetic price that an unprivileged trader can exploit at LP expense.

## Finding Description
`_readLeg` (L277–295) is the sole staleness gate for every feed read. It unconditionally compares `refTime` against the single `MAX_REF_STALENESS` immutable (L283). `_getBidAndAskPrice` (L258–272) calls `_readLeg(baseFeedId)` and, when `quoteFeedId != 0`, `_readLeg(quoteFeedId)` — both through the same function with no per-leg staleness argument. The constructor (L129–151) accepts exactly one `_maxRefStaleness` value and stores it as `MAX_REF_STALENESS`; no second parameter exists. The `priceGuard` check (L290–292) is a price-range bound, not a freshness check, and provides no mitigation. When `baseFeedId` has a 24-hour heartbeat and `quoteFeedId` has a 1-hour heartbeat, the deployer must set `MAX_REF_STALENESS ≥ 24 h` to avoid DoS on the base leg. Under that setting, the quote leg's `refTime` can be up to 24 hours old while passing the staleness check, producing a stale `mid2`. The synthetic ratio `mid1 / mid2` is then computed from a stale denominator. An unprivileged trader who observes the real price divergence calls `swap` on the pool; `getBidAndAskPrice` returns a bid/ask derived from the stale ratio; the pool executes at that bad price, transferring value from LP reserves to the trader.

## Impact Explanation
The corrupted synthetic bid/ask reaches the pool swap path — a direct "bad-price execution" impact. The trader receives assets at a price materially below fair value; LPs bear the loss as a reduction in pool reserves. The loss is proportional to the price discrepancy (potentially 5–10%+ for volatile assets) and swap volume, constituting a direct, quantifiable loss of LP principal. This matches the allowed impact categories: bad-price execution and pool insolvency.

## Likelihood Explanation
`AnchoredPriceProvider` is documented as "the one standard provider for public pools" and explicitly supports synthetic ratio mode via the `quoteFeedId` constructor parameter. The protocol targets any EVM-compatible chain. Feed heartbeats differ across chains for the same asset (e.g., USDT/USD: ~24 h on Linea, ~27 s on Polygon). Any synthetic pool deployed on a chain where the two constituent feeds have materially different heartbeats is structurally affected. The factory admin has no code-level workaround — the contract provides no per-leg staleness parameter. The condition is repeatable and exploitable by any unprivileged trader who monitors oracle update times.

## Recommendation
Add a separate `MAX_QUOTE_STALENESS` immutable for the quote leg, passed as a distinct constructor argument. Parameterize `_readLeg` to accept a `maxStaleness` argument:

```solidity
uint256 public immutable MAX_REF_STALENESS;   // base leg
uint256 public immutable MAX_QUOTE_STALENESS; // quote leg (synthetic mode)

function _readLeg(bytes32 feedId, uint256 maxStaleness) internal returns (...) {
    ...
    if (_isStale(refTime, block.timestamp, maxStaleness)) return (..., false);
    ...
}
```

Call sites in `_getBidAndAskPrice`:
```solidity
_readLeg(baseFeedId,  MAX_REF_STALENESS);
_readLeg(quoteFeedId, MAX_QUOTE_STALENESS);
```

For single-feed deployments, `MAX_QUOTE_STALENESS` is unused and can be set to `type(uint256).max`.

## Proof of Concept
1. Deploy `AnchoredPriceProvider` in synthetic mode: `baseFeedId = BTC/USD` (24 h heartbeat), `quoteFeedId = ETH/USD` (1 h heartbeat), `MAX_REF_STALENESS = 24 hours`.
2. At `t = 0`, oracle publishes BTC/USD = 65 000 and ETH/USD = 3 000; synthetic BTC/ETH = 21.67.
3. At `t = 20 h`, ETH/USD rises to 3 300 on the real market; the stored oracle `refTime` for ETH/USD is `t = 0`.
4. Trader calls `swap`. `_readLeg(quoteFeedId)` checks `(20 h) > MAX_REF_STALENESS (24 h)` → false → staleness check passes. Stale ETH/USD = 3 000 is used.
5. Synthetic price = 65 000 / 3 000 = 21.67 (stale). Real price = 65 000 / 3 300 = 19.70.
6. Trader buys BTC/ETH at 21.67; pool sells BTC cheaply in ETH terms, extracting ~9% above fair value from LP reserves.
7. LPs suffer a direct loss proportional to the price discrepancy and swap volume.
