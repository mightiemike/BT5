### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery - (`contract/src/lib.rs`)

---

### Summary

The contract exposes two SPV proof verification functions: the deprecated `verify_transaction_inclusion` (v1) and the secure `verify_transaction_inclusion_v2`. The v1 function is explicitly marked `#[deprecated]` and documented as vulnerable to the 64-byte transaction Merkle proof forgery attack, yet it remains a live, publicly callable NEAR contract method. Any unprivileged NEAR caller can invoke v1 directly, bypassing the coinbase Merkle proof mitigation introduced in v2, and obtain a forged `true` proof result.

---

### Finding Description

`verify_transaction_inclusion` is annotated with both a doc-level deprecation warning and a Rust `#[deprecated]` attribute: [1](#0-0) 

Despite this, the function retains `pub` visibility and the `#[pause]` attribute, making it a fully live NEAR contract entry point: [2](#0-1) 

The function performs only a raw Merkle root comparison against the stored block header's `merkle_root`, with no coinbase proof check: [3](#0-2) 

The secure replacement, `verify_transaction_inclusion_v2`, first validates a coinbase Merkle proof before delegating to v1: [4](#0-3) 

The `#[deprecated]` Rust attribute only emits a compiler warning at the call site within Rust code. It does **not** remove the function from the compiled WASM binary or prevent it from being dispatched on-chain. Any NEAR caller can invoke `verify_transaction_inclusion` directly by name, completely bypassing `verify_transaction_inclusion_v2`.

The `ProofArgs` struct (used by v1) has no `coinbase_tx_id` or `coinbase_merkle_proof` fields: [5](#0-4) 

Whereas `ProofArgsV2` carries the coinbase fields required for the mitigation: [6](#0-5) 

---

### Impact Explanation

The 64-byte transaction Merkle proof forgery (CVE-2017-12842) allows an attacker to craft a 64-byte blob that is simultaneously a valid serialized internal Merkle tree node and a plausible `tx_id`. Because Bitcoin's Merkle tree uses the same double-SHA256 hash for both leaf nodes (transactions) and internal nodes, an attacker can supply an internal node hash as `tx_id` along with a valid Merkle path, causing `compute_root_from_merkle_proof` to reconstruct the correct `merkle_root` and return `true`. Without coinbase proof validation, the contract cannot distinguish a real leaf from a forged internal node.

Any downstream NEAR contract or application that calls `verify_transaction_inclusion` (v1) and acts on its `true` result — e.g., releasing funds, minting tokens, or unlocking cross-chain assets — can be deceived into accepting a Bitcoin transaction that was never broadcast or confirmed.

---

### Likelihood Explanation

The entry path requires no privileges: any NEAR account can call `verify_transaction_inclusion` directly. The attacker needs only a valid confirmed Bitcoin block hash (publicly available on-chain) and the ability to compute a 64-byte internal node that satisfies the Merkle path. The attack is well-documented and tooling exists. The deprecated function is still present in the deployed WASM, so the likelihood is high for any integrator that has not migrated to v2.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface entirely, or gate it with an access-control role that prevents unprivileged callers from invoking it. The `#[deprecated]` Rust attribute is insufficient — it only warns Rust callers at compile time and has no effect on on-chain dispatch. The function must be made `private`, removed, or replaced with a wrapper that unconditionally enforces the coinbase proof check before the Merkle root comparison.

---

### Proof of Concept

1. Identify a confirmed Bitcoin block whose hash is tracked by the contract (e.g., via `get_block_hash_by_height`).
2. Obtain the block's Merkle tree. Select any internal node `N` at depth `d` from the root.
3. Construct a `merkle_proof` path of length `d` from `N` up to the `merkle_root`.
4. Call `verify_transaction_inclusion` with:
   - `tx_id` = hash of `N` (the internal node, treated as a fake transaction ID)
   - `tx_block_blockhash` = the confirmed block hash
   - `tx_index` = the index corresponding to `N`'s position
   - `merkle_proof` = the path constructed in step 3
   - `confirmations` = 1
5. The function returns `true`, confirming inclusion of a transaction that does not exist.

The same call to `verify_transaction_inclusion_v2` would fail at the coinbase proof check: [7](#0-6) 

because the attacker cannot produce a valid coinbase Merkle proof that also reconstructs to the same `merkle_root` using a different path.

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
