**`BurnNFT`** (lines 141–161 of `keeper.go`) requires **both** guards to pass:
1. `IsOwner(ctx, denomID, tokenID, owner)` — sender must be the current NFT owner
2. `IsDenomCreator(ctx, denomID, owner)` — sender must be the denom creator

**`TransferOwner`** (lines 121–138) only requires `IsOwner(srcOwner)` — no creator check.

The scenario is concrete and reachable:

| Step | Action | State |
|---|---|---|
| 1 | Creator issues denom, mints NFT to self | creator = owner |
| 2 | Creator calls `MsgTransferNFT` → `TransferOwner(creator, victim)` | victim = owner |
| 3 | Victim calls `MsgBurnNFT` | `IsOwner(victim)` ✓, `IsDenomCreator(victim)` ✗ → error |
| 4 | Creator calls `MsgBurnNFT` | `IsOwner(creator)` ✗ → error |

Neither party can burn. The only escape hatch is `BurnNFTUnverified`, which is only called internally during IBC NFT transfers — not exposed as a user-callable `MsgBurnNFT` path.

### Title
`BurnNFT` dual-guard deadlock permanently prevents burning of transferred NFTs — (`x/nft/keeper/keeper.go`)

### Summary
`BurnNFT` enforces `IsOwner(sender) AND IsDenomCreator(sender)` simultaneously. Because `TransferOwner` has no creator restriction, any NFT transferred to a non-creator address enters a state where no party satisfies both guards, making the NFT permanently unburnable and the collection supply counter permanently inflated.

### Finding Description
`BurnNFT` at [1](#0-0)  requires the caller to pass both `IsOwner` and `IsDenomCreator`. `TransferOwner` at [2](#0-1)  only checks `IsOwner(srcOwner)` and places no restriction on the destination address, so the creator can freely transfer to any non-creator. After transfer:

- The new owner (victim) fails `IsDenomCreator` at [3](#0-2) 
- The original creator fails `IsOwner` at [4](#0-3) 

`BurnNFTUnverified` at [5](#0-4)  bypasses the creator check but is only invoked internally by the IBC module — it is not reachable via `MsgBurnNFT`.

### Impact Explanation
The NFT is permanently unburnable through any user-accessible transaction path. `decreaseSupply` at [6](#0-5)  is never called, so `GetTotalSupply` permanently over-counts. Any on-chain or off-chain logic relying on supply accuracy is corrupted. The NFT occupies the victim's account indefinitely with no recourse.

### Likelihood Explanation
Triggered by two ordinary, fully authorized production transactions: `MsgMintNFT` followed by `MsgTransferNFT`. No special privileges, governance, or key compromise required. The creator may do this inadvertently (e.g., gifting an NFT) or deliberately (griefing a recipient's collection view).

### Recommendation
Remove the `IsDenomCreator` check from `BurnNFT` and allow the current owner to burn unconditionally, matching the semantics of `TransferOwner`. If creator-only burn is a design requirement, then `TransferOwner` must be restricted to creator-controlled addresses, or a separate creator-override burn path must be added that does not require `IsOwner`.

func TestBurnAfterTransferDeadlock(t *testing.T) {
    ctx, k := setupKeeper(t)
    creator := sdk.AccAddress([]byte("creator"))
    victim  := sdk.AccAddress([]byte("victim"))

    _ = k.IssueDenom(ctx, "denom1", "Denom", "", "", creator)
    _ = k.MintNFT(ctx, "denom1", "nft1", "", "", "", creator, creator)

    // Creator transfers to victim — succeeds
    err := k.TransferOwner(ctx, "denom1", "nft1", creator, victim)
    require.NoError(t, err)

    // Victim cannot burn: IsDenomCreator(victim) fails
    err = k.BurnNFT(ctx, "denom1", "nft1", victim)
    require.Error(t, err) // ErrUnauthorized

    // Creator cannot burn: IsOwner(creator) fails
    err = k.BurnNFT(ctx, "denom1", "nft1", creator)
    require.Error(t, err) // ErrUnauthorized

    // Supply counter is permanently inflated
    require.Equal(t, uint64(1), k.GetTotalSupply(ctx, "denom1"))
}
