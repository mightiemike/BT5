Audit Report

## Title
SwapAllowlistExtension Bypass via Router — Allowlisted Router Grants Unrestricted Swap Access to All Users - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` of the `pool.swap()` call. When `MetricOmmSimpleRouter` mediates a swap, it becomes `msg.sender` of `pool.swap()`, so the extension checks the router's allowlist status rather than the end-user's. Any pool that allowlists the router — a prerequisite for router-mediated swaps — grants unrestricted swap access to every user on the network, completely defeating the per-user allowlist.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs the check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension caller) and `sender` is the first argument forwarded by the pool. `MetricOmmPool.swap` always passes `msg.sender` as `sender`: [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards that `sender` verbatim to every extension in `BEFORE_SWAP_ORDER`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` (and all other router swap entry points) calls `pool.swap(...)` directly, making the router the `msg.sender` of that call: [4](#0-3) 

**Direct swap path** (`user → pool.swap()`): extension receives `sender = user`; check is `allowedSwapper[pool][user]` — correctly gates the actual user.

**Router-mediated path** (`user → router.exactInputSingle() → pool.swap()`): extension receives `sender = router`; check is `allowedSwapper[pool][router]`. For any router-mediated swap to succeed on an allowlisted pool, the admin must call `setAllowedToSwap(pool, router, true)`. Once set, the check passes for every user who routes through the router, regardless of whether that user is individually allowlisted or explicitly blocked. [5](#0-4) 

## Impact Explanation
The swap allowlist is a core access-control feature. Its failure allows any address to trade against a pool that was explicitly configured to restrict counterparties. LPs who deployed the allowlist to exclude known toxic-flow addresses, non-KYC'd users, or specific counterparties receive no protection. Unauthorized swaps expose LPs to adverse selection and spread losses that the allowlist was designed to prevent. This constitutes broken core pool functionality causing loss of LP value, matching the "Admin-boundary break" and "Broken core pool functionality causing loss of funds" allowed impact categories.

## Likelihood Explanation
High. The bypass requires only that the pool admin has allowlisted the router — a necessary and expected step for any router-mediated swap to function on an allowlisted pool. The attacker needs no special privilege: calling `router.exactInputSingle(...)` or any other router entry point is a public, permissionless action available to any address. No admin cooperation beyond the standard setup, no malicious token behavior, and no non-standard conditions are required. The condition is structurally guaranteed whenever the pool uses both the allowlist extension and the router.

## Recommendation
The extension must verify the original end-user identity, not the immediate caller of `pool.swap()`. Two viable approaches:

1. **Extension-data forwarding**: Require the router to encode the original `msg.sender` into `extensionData`; the extension decodes and checks that address against `allowedSwapper[pool][originator]`. The pool admin must configure the router to always include this field.
2. **Separate originator field**: Add an `originator` parameter to the `beforeSwap` interface that the pool populates from a transient-storage context set by the router before calling `pool.swap()`, so the extension can check the true initiator independently of the immediate caller.

Until fixed, pools that need per-user swap gating must not allowlist the router; users must call `pool.swap()` directly.

## Proof of Concept
```
1. Pool admin deploys pool with SwapAllowlistExtension as BEFORE_SWAP extension.
2. Pool admin calls:
       extension.setAllowedToSwap(pool, router, true)   // router allowlisted (required for router use)
       // attacker address is NOT allowlisted
3. attacker calls router.exactInputSingle({pool: pool, ...}) 
       → router calls pool.swap(msg.sender=router)
4. pool calls extension.beforeSwap(sender=router, ...)
5. Extension evaluates: allowedSwapper[pool][router] == true → passes
6. Swap executes. attacker has bypassed the allowlist with zero privilege.

Direct call check (for comparison):
   attacker calls pool.swap() directly.
   Extension evaluates: allowedSwapper[pool][attacker] == false → reverts NotAllowedToSwap.
```
