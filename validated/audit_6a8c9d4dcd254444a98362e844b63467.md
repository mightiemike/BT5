### Title
Unbounded `NlpLockedBalanceQueue` Growth Enables Permanent Freeze of NLP Balance Operations Including Liquidation — (File: `core/contracts/SpotEngineState.sol`)

---

### Summary

The `NlpLockedBalanceQueue` per-subaccount queue has no cap on the number of entries it can accumulate. The `tryUnlockNlpBalance` function iterates all unlocked entries in a single unbounded `while` loop. A user who makes many small NLP deposits at different oracle timestamps over the 4-day lock window can inflate their queue to a size that causes any subsequent call to `tryUnlockNlpBalance` on their account to exceed the block gas limit, permanently freezing NLP balance operations — including liquidation of their account.

---

### Finding Description

`NlpLockedBalanceQueue` is defined in `ISpotEngine.sol` as:

```solidity
struct NlpLockedBalanceQueue {
    mapping(uint64 => NlpLockedBalance) balances;
    uint64 balanceCount;
    uint64 unlockedUpTo;
    Balance unlockedBalanceSum;
}
``` [1](#0-0) 

Each time a user deposits NLP tokens (`productId == NLP_PRODUCT_ID`) and the oracle timestamp has advanced since the last deposit, `handleNlpLockedBalance` appends a **new entry** to the queue and increments `balanceCount`:

```solidity
queue.balances[queue.balanceCount] = NlpLockedBalance({...});
queue.balanceCount++;
``` [2](#0-1) 

Consolidation only occurs when the last entry's `unlockedAt` exactly matches `getOracleTime() + NLP_LOCK_PERIOD`. Since oracle time advances with each block, every deposit in a different block creates a distinct entry. There is **no cap** on `balanceCount`.

`tryUnlockNlpBalance` processes all matured entries in a single `while` loop:

```solidity
while (
    queue.unlockedUpTo < queue.balanceCount &&
    queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
) {
    queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
    delete queue.balances[queue.unlockedUpTo];
    queue.unlockedUpTo++;
}
``` [3](#0-2) 

This function is called unconditionally at the start of every `handleNlpLockedBalance` invocation: [4](#0-3) 

And `handleNlpLockedBalance` is called from `updateBalance` whenever `productId == NLP_PRODUCT_ID`: [5](#0-4) 

`NLP_LOCK_PERIOD` is 4 days: [6](#0-5) 

---

### Impact Explanation

After the 4-day lock window expires, all accumulated entries become eligible for processing simultaneously. If a user has accumulated `N` entries (one per block over 4 days), the `while` loop must execute `N` iterations — each requiring a storage read, a storage delete, and arithmetic — in a single transaction. At ~2-second block times on Ink Chain, 4 days yields ~172,800 blocks, meaning ~172,800 queue entries. At ~7,500 gas per iteration, this totals ~1.3 billion gas, far exceeding any realistic block gas limit.

Once the queue reaches this state, **every subsequent call to `updateBalance` for `NLP_PRODUCT_ID` on that subaccount reverts**. This includes:

1. The user's own NLP withdrawals (self-inflicted lock).
2. Any liquidation path that calls `updateBalance` for `NLP_PRODUCT_ID` on the bloated subaccount — the liquidation transaction reverts, leaving an undercollateralized account permanently unliquidatable. The protocol absorbs losses that should have been covered by the liquidation mechanism.

---

### Likelihood Explanation

The attacker must make one NLP deposit per block for 4 days. The minimum deposit is `$0.1` (`MIN_DEPOSIT_AMOUNT`): [7](#0-6) 

On a low-fee EVM chain like Ink, the total cost is bounded by gas fees per deposit transaction multiplied by ~172,800 transactions. If gas is cheap enough, this is economically feasible for an attacker who stands to benefit from avoiding liquidation (e.g., a heavily leveraged position going underwater). The attacker controls the timing entirely and needs no privileged access — only the ability to call the standard NLP deposit flow.

---

### Recommendation

1. **Cap queue depth**: Enforce a maximum on `balanceCount - unlockedUpTo` (the number of pending locked entries). Reject new NLP deposits if the live queue depth exceeds a safe bound (e.g., 100 entries).
2. **Batch-limit the unlock loop**: Process at most `MAX_UNLOCK_BATCH` entries per call to `tryUnlockNlpBalance`, deferring the rest to subsequent calls.
3. **Coarser time bucketing**: Round `getOracleTime()` to a larger granularity (e.g., 1 hour) when computing `unlockedAt`, so deposits within the same hour always consolidate into a single entry, bounding queue growth to `NLP_LOCK_PERIOD / bucket_size` entries.

---

### Proof of Concept

1. Attacker opens a subaccount and takes a large leveraged position.
2. Attacker calls the NLP deposit entrypoint once per block for 4 days, each time depositing the minimum amount (`$0.1`). Because oracle time advances each block, `handleNlpLockedBalance` appends a new entry each time, growing `balanceCount` to ~172,800.
3. After 4 days, all entries mature (`unlockedAt <= getOracleTime()`).
4. The attacker's position goes underwater. A liquidator calls the liquidation entrypoint, which internally calls `updateBalance(NLP_PRODUCT_ID, attackerSubaccount, delta)`.
5. `handleNlpLockedBalance` calls `tryUnlockNlpBalance`, which enters the `while` loop and attempts to process ~172,800 storage slots in one transaction.
6. The transaction runs out of gas and reverts. The undercollateralized account cannot be liquidated. The protocol's insurance fund absorbs the bad debt. [8](#0-7) [9](#0-8)
