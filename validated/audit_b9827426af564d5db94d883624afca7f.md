### Title
Signature Verification Bypass via Zero-Address `_verifier` in `BaseWithdrawPool._initialize` — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool._initialize` accepts a `_verifier` address with no zero-address guard. If initialized with `address(0)`, the public `submitFastWithdrawal` function's only security check — `Verifier(verifier).requireValidTxSignatures(...)` — silently succeeds because calling a void-returning function on an address with no code returns success in the EVM. Any unprivileged caller can then drain the pool with a crafted withdrawal transaction.

---

### Finding Description

`BaseWithdrawPool._initialize` stores `_verifier` directly without validation: [1](#0-0) 

`submitFastWithdrawal` is the only public withdrawal path for fast withdrawals. Its sole authorization check is: [2](#0-1) 

`Verifier.requireValidTxSignatures` has no return value: [3](#0-2) 

When `verifier == address(0)`, the EVM executes the call against an account with no code. Because the function is `void` (no return data to ABI-decode), Solidity accepts the empty return as success and execution continues past the check. All subsequent logic — parsing the transaction, resolving `sendTo`/`amount`, and transferring tokens — is fully attacker-controlled.

---

### Impact Explanation

After bypassing `requireValidTxSignatures`, the attacker crafts a `WithdrawCollateral` or `WithdrawCollateralV2` transaction body pointing to any `productId` with pool liquidity and setting `sendTo` to their own address. `resolveFastWithdrawal` parses this attacker-supplied bytes blob: [4](#0-3) 

`handleWithdrawTransfer` then transfers the full requested amount to the attacker: [5](#0-4) 

**Impact: complete theft of all ERC-20 collateral held in `WithdrawPool`.**

---

### Likelihood Explanation

`WithdrawPool.initialize` is a plain `external` function with no access control beyond the `initializer` modifier: [6](#0-5) 

A deployment script that passes `address(0)` for `_verifier` — e.g., due to a misconfigured environment variable or a staged deployment where the verifier is not yet deployed — silently succeeds. There is no on-chain guard to catch this. The broken state is permanent (the `initializer` modifier prevents re-initialization). Any actor who observes the zero-address `verifier` storage slot can immediately exploit it.

---

### Recommendation

Add a zero-address guard in `BaseWithdrawPool._initialize`:

```solidity
function _initialize(address _clearinghouse, address _verifier)
    internal
    initializer
{
+   require(_clearinghouse != address(0), "_clearinghouse != address(0)");
+   require(_verifier != address(0), "_verifier != address(0)");
    __Ownable_init();
    clearinghouse = _clearinghouse;
    verifier = _verifier;
}
``` [1](#0-0) 

---

### Proof of Concept

1. Deploy `WithdrawPool` and call `initialize(validClearinghouse, address(0))`. No revert occurs.
2. Fund the pool with USDC for `productId = 1`.
3. Attacker constructs `transaction` bytes: `[uint8(TransactionType.WithdrawCollateral)] ++ abi.encode(SignedWithdrawCollateral{sender: attacker_as_bytes32, productId: 1, amount: poolBalance, nonce: 0}, sig: "")`.
4. Attacker calls `submitFastWithdrawal(1, transaction, [])`.
5. `Verifier(address(0)).requireValidTxSignatures(...)` calls into address(0), returns empty data, and does not revert (void return type — no ABI decoding step).
6. `resolveFastWithdrawal` decodes `sendTo = attacker`, `amount = poolBalance`.
7. `handleWithdrawTransfer` transfers the full pool balance to the attacker. [7](#0-6)

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }
```
