### Title
Merkle Proof Replay: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` Never Record Proof Consumption — (`contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are stateless view functions that perform Merkle proof verification but never record that a given `(tx_id, tx_block_blockhash, tx_index, merkle_proof)` tuple was already accepted. Any unprivileged NEAR caller can replay an identical valid proof an unlimited number of times and receive `true` on every call. Downstream contracts that gate state-changing actions (fund releases, token mints, cross-chain settlements) on a `true` result from this light client are directly exploitable.

---

### Finding Description

`BtcLightClient` exposes two public proof-verification entry points:

- `verify_transaction_inclusion` (`&self`, line 288) [1](#0-0) 
- `verify_transaction_inclusion_v2` (`&self`, line 347) [2](#0-1) 

Both functions take `&self` — they are read-only and mutate no state. The entire verification logic reduces to computing a Merkle root from the supplied proof and comparing it to the stored `header.block_header.merkle_root`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [3](#0-2) 

The `BtcLightClient` struct contains no `used_proofs`, `spent_txids`, or any `LookupSet<H256>` field for tracking consumed proofs:

```rust
pub struct BtcLightClient {
    mainchain_height_to_header: LookupMap<u64, H256>,
    mainchain_header_to_height: LookupMap<H256, u64>,
    mainchain_tip_blockhash: H256,
    mainchain_initial_blockhash: H256,
    headers_pool: LookupMap<H256, ExtendedHeader>,
    skip_pow_verification: bool,
    gc_threshold: u64,
    network: Network,
}
``` [4](#0-3) 

The only field that previously resembled proof-consumption tracking was `used_aux_parent_blocks` in the now-dropped `BtcLightClientV2` layout, which tracked AuxPoW parent blocks for Dogecoin — not transaction inclusion proofs — and was explicitly removed during migration. [5](#0-4) 

Because the block header referenced by `tx_block_blockhash` persists in `headers_pool` for `gc_threshold` blocks (default ~52,704), a valid proof remains replayable for the entire window during which the referenced block stays on the main chain. [6](#0-5) 

---

### Impact Explanation

The BTC light client is explicitly designed as an SPV verification primitive for cross-chain applications. Any NEAR contract that calls `verify_transaction_inclusion_v2` and uses the `true` result to gate a state-changing action — releasing bridged funds, minting wrapped tokens, crediting a cross-chain deposit — can be drained by replaying the same valid Merkle proof. The light client itself provides no mechanism to prevent this; the `true` return value is indistinguishable between a first use and the hundredth replay.

The corrupted invariant is the authorization assumption: **a `true` result from `verify_transaction_inclusion_v2` is intended to mean "this Bitcoin transaction occurred and has not yet been processed," but the contract enforces only the first half of that invariant.**

---

### Likelihood Explanation

The attack requires only a single valid Bitcoin transaction inclusion proof, which is public on-chain data. No privileged role, leaked key, or social engineering is needed. The attacker calls `verify_transaction_inclusion_v2` (or the deprecated v1) with the same arguments in a loop. The proof remains valid for the entire `gc_threshold` window. Any deployed consuming contract that does not implement its own `used_proofs` set is immediately exploitable.

---

### Recommendation

Add a `used_proofs: near_sdk::collections::LookupSet<H256>` field to `BtcLightClient`. Before returning `true`, compute a canonical digest of `(tx_id, tx_block_blockhash, tx_index)`, assert it is not already in the set, insert it, and then return `true`. This requires converting both verification functions from `&self` to `&mut self`. Alternatively, document explicitly that replay protection is the sole responsibility of every consuming contract and that the light client intentionally provides no such guarantee — but this shifts a non-obvious security burden onto all integrators.

---

### Proof of Concept

1. Deploy a consuming NEAR contract `BridgeReceiver` that calls `verify_transaction_inclusion_v2` and, on `true`, mints 1 wrapped-BTC to `env::predecessor_account_id()`.
2. Obtain a valid Bitcoin transaction inclusion proof `P` for any confirmed transaction in a block within the `gc_threshold` window.
3. Call `BridgeReceiver` with `P` → receives `true` → mints 1 wBTC.
4. Call `BridgeReceiver` again with the identical `P` → `verify_transaction_inclusion_v2` returns `true` again (no state was changed in the light client) → mints another 1 wBTC.
5. Repeat until the block is GC'd from `headers_pool` (after `gc_threshold` ≈ 52,704 blocks, roughly one year).

The root cause is that `verify_transaction_inclusion_v2` at line 347 takes `&self`, writes nothing, and the `BtcLightClient` struct at line 96 has no field to record proof consumption. [7](#0-6) [4](#0-3)

```rust
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
