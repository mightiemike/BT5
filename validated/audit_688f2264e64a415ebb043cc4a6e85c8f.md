### Title
4-Byte Secp Opcode Aliases Bypass `ENABLE_SECP_OPS` Flag Gate — (`File: src/chia_dialect.rs`)

### Summary
The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches secp256k1 and secp256r1 signature-verification operators through two independent opcode representations. The 1-byte opcodes (64 and 65) are correctly gated behind `ClvmFlags::ENABLE_SECP_OPS`, but the 4-byte aliases (`0x13d61f00` and `0x1c3a8f00`) dispatch to the identical operator functions with no flag check at all. An attacker who supplies attacker-controlled CLVM bytes using the 4-byte form invokes live secp verification regardless of whether the flag is set, bypassing the softfork gate entirely.

### Finding Description

`ChiaDialect::op()` has two separate dispatch branches keyed on opcode byte-length.

**Branch 1 — 4-byte opcodes (lines 157–183):** The code comments describe this region as "unknown operators with assigned cost." However, two specific 4-byte values are explicitly matched and routed to the real secp operator functions with no flag guard:

```rust
// src/chia_dialect.rs  lines 175-182
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,   // no ENABLE_SECP_OPS check
    0x1c3a8f00 => op_secp256r1_verify,   // no ENABLE_SECP_OPS check
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

**Branch 2 — 1-byte opcodes (lines 248–249):** The same two operator functions are reached only when the flag is present:

```rust
// src/chia_dialect.rs  lines 248-249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The two representations are non-unique identifiers for the same operator. The 4-byte form is the alias that escapes the gate. The structural parallel to the external report is exact: just as a token symbol string is a non-unique identifier that lets an attacker impersonate a privileged token, the 4-byte opcode is a non-unique identifier that lets an attacker impersonate a flag-gated operator. ### Impact Explanation

`ENABLE_SECP_OPS` is a hard-fork activation flag. When it is absent — as it is in `MEMPOOL_MODE` and on any node that has not yet activated the fork — opcodes 64 and 65 fall through to `unknown_operator`. The 4-byte aliases do not fall through; they execute `op_secp256k1_verify` / `op_secp256r1_verify` unconditionally.

Concrete consequences:

1. **Consensus divergence.** A pre-fork node (no `ENABLE_SECP_OPS`) accepts a block containing a puzzle that uses `0x13d61f00` and verifies a secp256k1 signature. A post-fork node with `ENABLE_SECP_OPS` set also accepts it via opcode 64. Both reach the same result, but the pre-fork node should have treated the operator as unknown/no-op. The two nodes agree on the outcome for the wrong reason, and any future change to the 4-byte branch would split them.

2. **Mempool/consensus mismatch.** `MEMPOOL_MODE` sets `NO_UNKNOWN_OPS` but not `ENABLE_SECP_OPS`. A transaction using the 4-byte secp opcode passes mempool validation (the 4-byte branch executes and succeeds), while the same transaction using opcode 64 is rejected (`unknown_operator` → `EvalErr::Unimplemented`). This asymmetry lets an attacker craft transactions that pass mempool checks in a way the protocol designers did not intend.

3. **Softfork gate nullified.** The entire purpose of `ENABLE_SECP_OPS` is to allow secp operations only after a governance-controlled activation. The 4-byte alias makes that gate optional for any caller who knows the alternate encoding. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

The entry path is fully attacker-controlled: the attacker supplies raw CLVM bytes to `run_program`. Encoding a 4-byte atom `\x13\xd6\x1f\x00` as the operator of a CLVM application form is trivial and requires no special privilege. The 4-byte values are already referenced in `clvm-fuzzing/src/make_tree.rs` and `tools/src/bin/benchmark-clvm-cost.rs`, confirming they are known, reachable values in the ecosystem. [5](#0-4) 

### Recommendation

Add the same `ENABLE_SECP_OPS` guard to the 4-byte branch before dispatching to the secp functions:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This mirrors the guard pattern already used for opcodes 64 and 65 in the 1-byte branch and eliminates the non-unique-identifier bypass. [6](#0-5) 

### Proof of Concept

Attacker constructs a CLVM program where the operator atom is the 4-byte sequence `\x13\xd6\x1f\x00` (i.e., `0x13d61f00`) followed by a valid secp256k1 pubkey, message digest, and signature as arguments. When evaluated by `run_program` with a `ChiaDialect` that does **not** have `ENABLE_SECP_OPS` set (e.g., `MEMPOOL_MODE`):

- The `op_len == 4` branch is entered.
- `opcode == 0x13d61f00` matches, dispatching to `op_secp256k1_verify`.
- The signature is verified and the program returns `nil` (success) with cost 1,300,000.

The identical program using 1-byte opcode `\x40` (64) under the same flags hits the `_ =>` arm in the 1-byte branch and returns `EvalErr::Unimplemented`, demonstrating the asymmetry. The 4-byte form is the alias that escapes the gate. [7](#0-6) [8](#0-7)
