### Title
Deprecated `verify_transaction_inclusion` Remains Callable, Allowing Selective Bypass of Coinbase Merkle Proof Validation — (`contract/src/lib.rs`)

### Summary

The contract exposes two transaction-inclusion verification paths: the deprecated `verify_transaction_inclusion` and the current `verify_transaction_inclusion_v2`. Any unprivileged NEAR caller can selectively invoke the deprecated path, which skips the coinbase Merkle proof check that guards against the 64-byte transaction forgery attack. This is a direct analog to the `forceDeallocate` bug class: just as a vault user could selectively withdraw from a profitable adapter while ignoring loss accounting in another, a proof consumer here can selectively invoke the weaker verification path to obtain a `true` result that the stronger path would reject.

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` in Rust source, but `#[deprecated]` only emits a compiler warning — it does **not** restrict runtime invocability. The function remains a fully public NEAR entry point protected only by `#[pause]`, which is inactive by default. [1](#0-0) 

`verify_transaction_inclusion_v2` adds a mandatory coinbase Merkle proof check before delegating to the deprecated function: [2](#0-1) 

The coinbase proof requirement exists specifically to defeat the [64-byte transaction Merkle forgery](https://www.bitmex.com/blog/64-Byte-Transactions): an internal Merkle tree node (64 bytes) can be presented as a `tx_id`, and `compute_root_from_merkle_proof` will compute a root that matches the block's `merkle_root`, causing the function to return `true` for a non-existent transaction. [3](#0-2) 

The Merkle computation itself has no awareness of whether the input is a leaf (real transaction) or an internal node: [4](#0-3) 

### Impact Explanation

A caller that wants a forged proof accepted can bypass the coinbase check entirely by calling `verify_transaction_inclusion` instead of `verify_transaction_inclusion_v2`. Any downstream NEAR contract that calls either function and treats the boolean result as authoritative will accept a fabricated transaction inclusion as valid. This corrupts the core security guarantee of the SPV light client: that a `true` result means the transaction genuinely exists in a Bitcoin block on the canonical chain.

### Likelihood Explanation

The entry point is fully permissionless — no role, stake, or special privilege is required. The 64-byte forgery technique is publicly documented and well-understood. An attacker only needs to: (1) identify a target block whose Merkle tree has an internal node whose 32-byte left and right children concatenate to a 64-byte value they control, and (2) supply that internal node as `tx_id` with a valid sibling path. The deprecated function is callable on any live deployment where the `PauseManager` has not explicitly paused it.

### Recommendation

Remove `verify_transaction_inclusion` as a callable public entry point. If backward compatibility is required, gate it behind an access-control role (e.g., `Role::DAO`) so it cannot be invoked by unprivileged callers. Alternatively, add the same coinbase proof validation inside `verify_transaction_inclusion` itself so both paths enforce the same security invariant, eliminating the selective-bypass opportunity.

### Proof of Concept

```
1. Attacker identifies block B on the canonical mainchain stored in the contract.
2. Attacker finds an internal Merkle node N in block B's transaction tree such that
   compute_root_from_merkle_proof(N, position, sibling_path) == B.merkle_root.
3. Attacker calls:
     verify_transaction_inclusion({
         tx_id: N,                  // internal node, not a real tx hash
         tx_block_blockhash: B,
         tx_index: position,
         merkle_proof: sibling_path,
         confirmations: 1,
     })
4. The function returns `true` because the Merkle root matches and no coinbase
   proof is checked.
5. Any downstream NEAR contract that called this entry point now believes the
   fabricated transaction N is confirmed in block B.
``` [5](#0-4)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]

    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
