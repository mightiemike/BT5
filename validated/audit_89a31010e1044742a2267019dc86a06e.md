### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Allows Draining usdcE from DDAs Without Paying USDC - (File: core/contracts/ContractOwner.sol)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(...)` without checking its return value. If the USDC token returns `false` on failure instead of reverting, the function continues to withdraw usdcE from the target DDA and send it to the caller — without the caller ever paying USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is an unguarded `external` function (no `onlyOwner` or similar modifier) that is intended to let a caller swap USDC-E held in a Direct Deposit Account (DDA) for native USDC. The swap logic is:

1. Read the usdcE balance of the target DDA.
2. Pull `balance` USDC from `msg.sender` into the DDA via `transferFrom`.
3. Withdraw all usdcE from the DDA to `ContractOwner`.
4. Send the usdcE to `msg.sender` via `safeTransfer`.

The critical flaw is at step 2:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

The return value of `transferFrom` is **never checked**. [1](#0-0) 

ERC-20 does not require `transferFrom` to revert on failure; returning `false` is fully compliant and is implemented by many real tokens (BAT, cUSDC, ZRX, etc.). If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on chain 57073 (Ink) returns `false` on an underfunded or unapproved call, execution silently continues through steps 3 and 4, draining usdcE from the DDA and delivering it to the caller for free.

Every other transfer in the codebase uses the project's own `ERC20Helper.safeTransfer` / `safeTransferFrom` wrappers, which explicitly check the return value and revert on failure. [2](#0-1) 

This single call is the only place in the production contracts that uses a raw, unchecked `transferFrom`. [3](#0-2) 

---

### Impact Explanation

An attacker with zero USDC (or zero allowance) calls `replaceUsdcEWithUsdc(subaccount)` for any DDA that holds usdcE. If `transferFrom` returns `false` silently:

- The DDA loses its entire usdcE balance (withdrawn and sent to the attacker).
- The attacker pays nothing.
- The DDA owner's deposited usdcE is permanently gone with no recourse.

The asset delta is: attacker gains `balance` usdcE; DDA loses `balance` usdcE; no USDC is ever transferred.

---

### Likelihood Explanation

- The function has **no access control** beyond a chain-ID check (`block.chainid == 57073`), so any EOA or contract on Ink can call it. [4](#0-3) 
- The exploit requires only that the USDC token at the hardcoded address returns `false` on failure rather than reverting. Many ERC-20 tokens exhibit this behavior, and the ERC-20 standard explicitly permits it.
- No privileged access, no governance capture, and no sequencer compromise is needed.

---

### Recommendation

Replace the raw `transferFrom` call with the project's existing `safeTransferFrom` helper, consistent with every other transfer in the codebase:

```diff
- IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
+ IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ERC20Helper.safeTransferFrom` already handles the return-value check and reverts with `ERR_TRANSFER_FAILED` on failure. [5](#0-4) 

---

### Proof of Concept

```
Precondition:
  - Chain ID = 57073 (Ink)
  - DDA for `subaccount` exists and holds N usdcE
  - Attacker has 0 USDC and 0 USDC allowance to ContractOwner

Attack:
  1. Attacker calls ContractOwner.replaceUsdcEWithUsdc(subaccount)
  2. Line 614: balance = N (usdcE balance of DDA)
  3. Line 616: IERC20Base(usdc).transferFrom(attacker, DDA, N)
               → USDC returns false (attacker has no balance/allowance)
               → return value ignored, execution continues
  4. Line 617: DirectDepositV1(DDA).withdraw(usdcE)
               → N usdcE transferred from DDA to ContractOwner
  5. Line 618: IERC20Base(usdcE).safeTransfer(attacker, N)
               → N usdcE transferred to attacker

Result:
  - Attacker receives N usdcE
  - DDA loses N usdcE
  - No USDC was ever transferred
```

    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }

        IERC20Base self,
        address from,
        address to,
        uint256 amount

        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(

                to,
                amount
            )
