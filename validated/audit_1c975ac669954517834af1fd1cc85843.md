### Title
`verify_transaction_inclusion` unconditionally rejects empty Merkle proofs, blocking valid SPV verification for single-transaction blocks — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` contains a blanket guard `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` that is applied to every call path, including calls delegated from `verify_transaction_inclusion_v2`. For single-transaction blocks (where the Merkle root equals the sole transaction hash), an empty proof is the only mathematically correct proof. The underlying `compute_root_from_merkle_proof` function handles this case correctly, but the guard fires first and panics, permanently blocking SPV verification for any such block.

---

### Finding Description

In `verify_transaction_inclusion` at line 315 of `contract/src/lib.rs`:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

This check is applied unconditionally to every caller. [1](#0-0) 

For a block containing exactly one transaction (the coinbase), Bitcoin's Merkle tree construction sets `merkle_root = coinbase_tx_id`. The correct proof for that transaction at index 0 is an empty vector — there are no sibling hashes to provide. The `compute_root_from_merkle_proof` function in `merkle-tools/src/lib.rs` handles this correctly: when `merkle_proof` is empty, the `for` loop does not execute and the function returns `transaction_hash` unchanged. [2](#0-1) 

The guard fires before `compute_root_from_merkle_proof` is ever reached, so the correct result is never computed. [3](#0-2) 

The same defect is inherited by `verify_transaction_inclusion_v2`, which validates the coinbase proof independently and then delegates to `verify_transaction_inclusion` via `self.verify_transaction_inclusion(args.into())`. [4](#0-3) 

In `verify_transaction_inclusion_v2`, for a single-transaction block:
1. `merkle_proof.len() == coinbase_merkle_proof.len()` → `0 == 0` → passes. [5](#0-4) 
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id`, which equals `header.block_header.merkle_root` → passes. [6](#0-5) 
3. Delegation to `verify_transaction_inclusion` → panics at the empty-proof guard.

The guard is therefore applied to a code path (single-transaction block verification) where it is not only unnecessary but actively incorrect, mirroring the M-10 pattern of a post-validation check applied uniformly across all operation types when it is only valid for a subset.

---

### Impact Explanation

Any NEAR caller or downstream smart contract invoking either `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to verify a transaction in a single-transaction block will receive a panic instead of a valid `true` result. The block is correctly stored in the contract's `headers_pool` and `mainchain_height_to_header` maps; the failure is entirely in the proof-verification path. There is no workaround: the caller cannot supply a non-empty proof for a single-transaction block because no valid non-empty proof exists for that Merkle tree structure.

---

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) are common throughout Bitcoin's history, particularly in early blocks and during low-activity periods. Any bridge, oracle, or SPV consumer that needs to verify coinbase transactions (e.g., for mining reward proofs or coinbase maturity checks) against this light client will be permanently blocked for all such blocks. The entry path is fully unprivileged: any NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with a valid block hash and an empty proof.

---

### Recommendation

Remove the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard. The `compute_root_from_merkle_proof` function already handles empty proofs correctly and will return the transaction hash unchanged, which will correctly match the stored `merkle_root` for single-transaction blocks. No additional logic is needed.

```diff
-    require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
-
     merkle_tools::compute_root_from_merkle_proof(
         args.tx_id,
         usize::try_from(args.tx_index).unwrap(),
         &args.merkle_proof,
     ) == header.block_header.merkle_root
```

---

### Proof of Concept

1. A Bitcoin block contains exactly one transaction (the coinbase) with hash `C`. Its `merkle_root = C`.
2. The block is submitted to the contract via `submit_blocks` and stored in `mainchain_height_to_header` and `headers_pool`.
3. A caller invokes `verify_transaction_inclusion` with `tx_id = C`, `tx_block_blockhash = <block hash>`, `tx_index = 0`, `merkle_proof = []`, `confirmations = 1`.
4. All checks before line 315 pass: the block is on the mainchain, confirmations are sufficient, the header is found.
5. Execution reaches `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` and panics. [1](#0-0) 
6. Without the guard, `compute_root_from_merkle_proof(C, 0, &[])` returns `C`, which equals `header.block_header.merkle_root`, and the function correctly returns `true`. [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
