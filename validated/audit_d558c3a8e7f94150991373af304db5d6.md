### Title
Removed Fund Recovery in `_executeSlowModeTransaction()` Catch Block Permanently Locks User Deposit Collateral - (File: `core/contracts/Endpoint.sol`)

### Summary

`Endpoint._executeSlowModeTransaction()` wraps `processSlowModeTransaction()` in a `try/catch`. When the inner call fails, the catch block silently discards the failure with no fund recovery. A code comment — `// try return funds now removed` — explicitly confirms that a refund mechanism was intentionally deleted. Because `depositCollateralWithReferral()` transfers user tokens to the `Clearinghouse` **before** queuing the slow-mode transaction, any failure during execution permanently locks those tokens with no on-chain recovery path.

### Finding Description

`Endpoint.depositCollateralWithReferral()` performs two sequential operations:

1. `handleDepositTransfer()` — pulls tokens from `msg.sender` and pushes them to `address(clearinghouse)`.
2. Enqueues a `DepositCollateral` slow-mode transaction in `slowModeTxs`. [1](#0-0) 

`handleDepositTransfer()` in `EndpointStorage` performs the actual ERC-20 transfer chain: `safeTransferFrom(token, from, amount)` then `safeTransferTo(token, address(clearinghouse), amount)`. [2](#0-1) 

When the slow-mode transaction is later executed via `_executeSlowModeTransaction()`, the call to `this.processSlowModeTransaction()` is wrapped in a `try/catch`. If the inner call reverts, the catch block does nothing — the transaction is consumed from the queue (`delete slowModeTxs[_slowModeConfig.txUpTo++]`) and execution continues silently. The comment `// try return funds now removed` at line 226 explicitly marks where a refund path was deleted. [3](#0-2) 

`processSlowModeTransaction` delegates to `EndpointTx.processSlowModeTransactionImpl`, which calls `clearinghouse.depositCollateral(txn)`. If that call reverts (e.g., the product was delisted during the mandatory 3-day `SLOW_MODE_TX_DELAY` window), the revert is caught, the slow-mode entry is deleted, and the tokens already sitting in the `Clearinghouse` are permanently orphaned. [4](#0-3) 

### Impact Explanation

A user who calls `depositCollateral()` or `depositCollateralWithReferral()` transfers real ERC-20 tokens to the `Clearinghouse` at submission time. If the corresponding slow-mode execution fails for any reason, those tokens are permanently locked: the slow-mode entry is deleted so it cannot be retried, and no on-chain mechanism exists to reclaim the funds. The user loses their entire deposited amount with no recourse.

### Likelihood Explanation

The 3-day `SLOW_MODE_TX_DELAY` creates a meaningful window during which protocol state can change. A product being delisted via `DelistProduct` between deposit submission and execution is a realistic trigger — the sequencer can submit a `DelistProduct` slow-mode transaction that executes before the user's `DepositCollateral` slow-mode transaction. Any other revert path inside `clearinghouse.depositCollateral()` (e.g., a health-check invariant, a product misconfiguration) also triggers the loss. The entry point `depositCollateralWithReferral()` is public and callable by any user, including via `DirectDepositV1.creditDeposit()`. [5](#0-4) 

### Recommendation

Restore the fund-recovery logic in the catch block of `_executeSlowModeTransaction()`. When `processSlowModeTransaction` fails for a `DepositCollateral` transaction, the `Clearinghouse` should transfer the deposited tokens back to the original sender. At minimum, the catch block must not silently discard failures that involve pre-transferred user funds. The removed refund path should be reinstated, conditioned on the transaction type being one that took custody of user assets at submission time.

### Proof of Concept

1. User calls `Endpoint.depositCollateral(subaccountName, productId, 1000e18)`.
2. `handleDepositTransfer` moves 1000e18 tokens from the user to `Clearinghouse`. Slow-mode entry is queued at index `N`.
3. Within the 3-day window, the sequencer submits and executes a `DelistProduct` slow-mode transaction for `productId`.
4. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`.
5. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(sender, tx)` inside `try/catch`.
6. `clearinghouse.depositCollateral()` reverts because the product is delisted.
7. The catch block executes: gas check passes, execution falls through to `// try return funds now removed` — nothing happens.
8. `slowModeTxs[N]` is deleted. `slowModeConfig.txUpTo` is incremented.
9. The user's 1000e18 tokens remain in `Clearinghouse` with no associated subaccount balance and no recovery path. [6](#0-5)
