### Title
`ValidResourceBounds::AllResources` with zero L2/L1_DATA gas silently collapses to `ValidResourceBounds::L1Gas` across the protobuf boundary, producing a different transaction hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` applies a heuristic that converts `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` into `L1Gas(X)`. Because `get_tip_resource_bounds_hash` produces structurally different poseidon preimages for these two variants (3 vs 4 elements), the transaction hash computed after a protobuf round-trip differs from the hash computed at submission time. Any node that re-derives the hash from the deserialized `Transaction` object — including blockifier re-execution, hash validation, and fee estimation — will obtain a wrong value.

### Finding Description

**Serialization path (proposer → protobuf)**

`ValidResourceBounds::AllResources` with `l2_gas = 0` and `l1_data_gas = 0` serializes to a `protobuf::ResourceBounds` with all three fields present but the last two zero: [1](#0-0) 

**Deserialization path (protobuf → syncing node)**

The deserializer checks whether `l1_data_gas` and `l2_gas` are both zero and, if so, returns `L1Gas` instead of `AllResources`: [2](#0-1) 

**Hash domain divergence**

`get_tip_resource_bounds_hash` produces a poseidon hash over a **different number of elements** depending on the variant:

- `L1Gas` → `poseidon([tip, L1_GAS_packed, L2_GAS_packed])` — **3 elements**
- `AllResources` → `poseidon([tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed])` — **4 elements** [3](#0-2) 

Because poseidon is collision-resistant, these two hashes are distinct for any non-trivial `l1_gas` value. The transaction hash committed to the block therefore differs from the hash a syncing node would compute from the deserialized `Transaction`.

**Trigger is unprivileged**

The gateway's stateless validator explicitly accepts `AllResourceBounds { l1_gas: NON_EMPTY, l2_gas: 0, l1_data_gas: 0 }` as a valid submission: [4](#0-3) 

The `RpcTransaction` and `InternalRpcInvokeTransactionV3` types always carry `AllResourceBounds` (never `ValidResourceBounds`), so the hash is always computed via the `AllResources` path at submission: [5](#0-4) 

But the P2P sync path deserializes into `Transaction` / `InvokeTransactionV3`, which carries `ValidResourceBounds`: [6](#0-5) 

After deserialization the `resource_bounds` field holds `L1Gas(X)` instead of `AllResources(X, 0, 0)`, so any subsequent call to `get_transaction_hash` or `validate_transaction_hash` on the deserialized object produces the wrong 3-element poseidon hash.

**Gas computation mode also changes**

`ValidResourceBounds::get_gas_vector_computation_mode` returns `NoL2Gas` for `L1Gas` and `All` for `AllResources`: [7](#0-6) 

This changes which resource bounds are enforced during fee pre-validation in the blockifier: [8](#0-7) 

A transaction originally executed under `All` mode (checking L1, L2, L1_DATA gas) would be re-executed under `NoL2Gas` mode (checking only L1 gas), producing a different fee result.

### Impact Explanation

Any syncing node that receives a V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` via P2P will hold a `Transaction` object whose `resource_bounds` is `L1Gas(X)`. If that node:

1. Calls `validate_transaction_hash` — it computes the 3-element poseidon hash and compares it to the 4-element poseidon hash stored in the block, returning `false` for a legitimately included transaction.
2. Re-executes the transaction (blockifier re-execution, RPC `starknet_simulateTransactions`, fee estimation) — it uses `NoL2Gas` mode, skipping L2 and L1_DATA gas bound checks, producing wrong fee/gas results.
3. Computes a block hash that depends on transaction hashes — the derived hash diverges from the canonical one.

This matches the **High** impact category: "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload" and "RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."

### Likelihood Explanation

The trigger requires only a standard V3 transaction submission with `l2_gas = 0` and `l1_data_gas = 0`, which the gateway explicitly accepts. Such transactions are common in the pre-0.13.3 era and remain valid today. No special privileges, malicious peers, or invalid bytes are required. Any honest user submitting a V3 transaction with only L1 gas bounds set will trigger this path.

### Recommendation

Replace the heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit discriminator field in the protobuf schema, or always deserialize into `AllResources` when all three fields are present (regardless of whether they are zero). The current check:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(...)
})
```

should be removed. Since the `ConsensusTransaction` path already correctly preserves `AllResourceBounds` end-to-end (it never passes through `ValidResourceBounds`), the fix should align the P2P sync `Transaction` deserialization to always produce `AllResources` when all three protobuf fields are present, matching the hash domain used at submission time.

### Proof of Concept

```
1. User submits RpcInvokeTransactionV3 with:
     resource_bounds = AllResourceBounds { l1_gas: {max_amount: 1000, max_price: 1}, l2_gas: 0, l1_data_gas: 0 }

2. Gateway calls convert_rpc_tx_to_internal → calculate_transaction_hash:
     InternalRpcInvokeTransactionV3::resource_bounds() returns ValidResourceBounds::AllResources(...)
     get_tip_resource_bounds_hash hashes [tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)]
     → H1 = poseidon_hash(4 elements)

3. Transaction is included in block with hash H1.

4. Syncing node receives the block via P2P. Protobuf carries:
     ResourceBounds { l1_gas: {1000,1}, l2_gas: {0,0}, l1_data_gas: {0,0} }

5. TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
     l1_data_gas.is_zero() && l2_gas.is_zero() → true
     → returns ValidResourceBounds::L1Gas({1000,1})   ← WRONG VARIANT

6. Syncing node calls get_transaction_hash on deserialized Transaction:
     get_tip_resource_bounds_hash hashes [tip, L1_GAS_packed, L2_GAS_packed(0)]
     → H2 = poseidon_hash(3 elements)

7. H1 ≠ H2  →  validate_transaction_hash returns false for a legitimately included transaction.
   Re-execution uses NoL2Gas mode instead of All mode → wrong fee result.
```

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
