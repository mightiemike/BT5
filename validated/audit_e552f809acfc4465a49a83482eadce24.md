### Title
`DISABLE_OP` in `MEMPOOL_MODE` Permanently Blocks `op_modpow` (Opcode 60) in Mempool, Creating Consensus/Mempool Divergence — (File: `src/chia_dialect.rs`)

---

### Summary

`MEMPOOL_MODE` unconditionally includes `ClvmFlags::DISABLE_OP`, which causes the operator dispatch for opcode 60 (`op_modpow`) to return `EvalErr::Unimplemented` in mempool mode. In consensus mode the same opcode executes normally. Any attacker-controlled CLVM program that uses opcode 60 is therefore permanently rejected by every mempool node while being accepted by every consensus node — a direct flag/operator wiring analog to the `cage()`/`live=0` pattern in the external report.

---

### Finding Description

**Root cause — `MEMPOOL_MODE` wires in `DISABLE_OP`:**

```rust
// src/chia_dialect.rs lines 72-76
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)          // ← kills opcode 60
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

**Root cause — operator dispatch hard-gates opcode 60 on that flag:**

```rust
// src/chia_dialect.rs lines 239-243
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;   // ← permanent rejection
    }
    op_modpow
}
```

`op_modpow` (opcode 60) is a first-class entry in the main operator table — it is not behind a softfork guard, it has its own fuzz target (`fuzz/fuzz_targets/modpow.rs`), its own test-vector generator (`tools/src/bin/generate-modpow-tests.rs`), and is listed by name in `f_table.rs` alongside every other hardforked operator. Opcode 61 (`op_mod`), its sibling, carries **no** `DISABLE_OP` guard. The flag name `DISABLE_OP` is generic and undocumented; no comment explains why it targets only opcode 60.

**Execution path for an attacker-supplied program:**

1. Caller (full node, wallet, or any Python consumer via `wheel/src/api.rs`) invokes `run_program` with `MEMPOOL_MODE` flags and a CLVM program whose byte stream contains opcode byte `0x3c` (= 60).
2. `ChiaDialect::op` is called with `flags = MEMPOOL_MODE`.
3. The `match op { 60 => … }` arm fires; `flags.contains(DISABLE_OP)` is `true`.
4. `EvalErr::Unimplemented(o)` is returned — the program is rejected.
5. The identical program submitted directly to a consensus node (no `DISABLE_OP`) reaches `op_modpow` and succeeds.

The attacker-controlled input is the raw CLVM byte sequence; no privileged access is required.

---

### Impact Explanation

Any coin whose puzzle uses `modpow` (opcode 60) is **permanently unspendable through the mempool**. Mempool nodes will never relay or include such a spend, even though consensus nodes would accept it. This is the direct analog of `daiJoin.cage()` setting `live = 0`: a flag-gated check (`DISABLE_OP` / `live == 0`) causes a hard revert on an operation that is otherwise valid, permanently blocking the user's ability to move funds. The corrupted result is the `EvalErr::Unimplemented` return value for opcode 60 under `MEMPOOL_MODE`; the broken invariant is that mempool validity must be a strict subset of consensus validity, not a disjoint set.

---

### Likelihood Explanation

`MEMPOOL_MODE` is the default flag set used by full nodes when evaluating incoming transactions (`wheel/src/api.rs` exposes it directly to Python callers). Any spend bundle whose generator or puzzle contains opcode 60 triggers the divergence unconditionally. No special timing, race condition, or low-probability event is required — the flag is always present in mempool mode.

---

### Recommendation

1. Remove `ClvmFlags::DISABLE_OP` from `MEMPOOL_MODE` if `op_modpow` is an activated consensus operator, restoring mempool/consensus parity.
2. If `modpow` is intentionally restricted in mempool mode (e.g., pending a hard-fork activation), rename the flag to `DISABLE_MODPOW` and add an explicit comment explaining the activation timeline, mirroring the pattern used for `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` and `ENABLE_SHA256_TREE`.
3. Add a property-based test asserting that every opcode accepted by consensus mode is also accepted by mempool mode (modulo cost limits), to catch future wiring errors of this class.

---

### Proof of Concept

Minimal CLVM program (hex): `ff3cff8001ff8002ff8003ff` — this encodes `(modpow 1 2 3)` using opcode byte `0x3c` (60).

- **Consensus mode** (`ClvmFlags::empty()`): `run_program` dispatches to `op_modpow`, returns `Reduction(cost, result_node)`.
- **Mempool mode** (`MEMPOOL_MODE`): `run_program` hits `flags.contains(ClvmFlags::DISABLE_OP) == true` at `src/chia_dialect.rs:240`, returns `Err(EvalErr::Unimplemented(o))`.
