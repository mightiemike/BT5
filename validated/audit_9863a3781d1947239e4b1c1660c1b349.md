### Title
Missing `state_diff_commitment` Verification in P2P State Diff Sync Allows Attacker Peer to Inject Wrong `compiled_class_hash` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`StateDiffStreamBuilder::parse_data_for_block` validates only that the sum of `StateDiffChunk::len()` values equals the `state_diff_length` stored in the block header. It never computes or compares the Poseidon `state_diff_commitment` that is also stored in the header. An attacker-controlled peer can therefore send chunks whose total length is correct but whose content (e.g., `compiled_class_hash`) differs from the canonical state diff. The fabricated `ThinStateDiff` is written verbatim to storage, poisoning `compiled_class_hash_table` with an attacker-chosen value.

---

### Finding Description

`parse_data_for_block` in `state_diff.rs` reads `target_state_diff_len` from the stored header and loops until the accumulated chunk lengths equal that target: [1](#0-0) [2](#0-1) 

The only post-assembly checks are:
1. `current_state_diff_len == target_state_diff_len` (count only)
2. No duplicate keys within a chunk type (`ConflictingStateDiffParts`)
3. No duplicate deprecated declared classes

The `state_diff_commitment` field — a Poseidon hash over the full state diff content including every `compiled_class_hash` — is stored in the header but is **never read or compared** inside `parse_data_for_block`. A `grep` of `state_diff_commitment` across the entire `apollo_p2p_sync` crate confirms it appears only in test assertions and in `convert_sync_block_to_block_data` (the internal-block path), never in the P2P validation path.

The header sync (`header.rs`) also does not cryptographically verify the block signature — it only checks block number ordering and that `signatures.len() == ALLOWED_SIGNATURES_LENGTH`: [3](#0-2) 

So an attacker peer can send either a legitimate canonical header (with a correct `state_diff_length`) or a fabricated one, and in either case the state diff content is never bound to the commitment.

The assembled `ThinStateDiff` is written directly to storage without any hash check: [4](#0-3) 

`append_state_diff` then writes the attacker-supplied `compiled_class_hash` into `compiled_class_hash_table`: [5](#0-4) 

The `calculate_state_diff_hash` function exists and covers `compiled_class_hash` values: [6](#0-5) 

It is used in the committer (`apollo_committer`) when `verify_state_diff_hash = true`, but it is entirely absent from the P2P sync client path.

---

### Impact Explanation

A wrong `compiled_class_hash` stored in `compiled_class_hash_table` causes the wrong CASM artifact to be selected for any subsequent execution that looks up the class by `class_hash`. This affects RPC fee estimation, simulation, tracing, and any execution path that reads compiled class state from the poisoned storage. This matches the allowed Critical impact: **"Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution."**

---

### Likelihood Explanation

Any peer the state sync node connects to can execute this attack. No special privileges are required — the attacker only needs to be a reachable P2P peer. The attack requires crafting a single `DeclaredClass` chunk with the correct `class_hash` (observable from the canonical chain) and an arbitrary `compiled_class_hash`, with `len()` summing to the header's `state_diff_length`. The peer is reported after the session, but the poisoned state diff is already committed to storage before any report is filed.

---

### Recommendation

After assembling the full `ThinStateDiff`, compute `calculate_state_diff_hash(&result)` and compare it against `block_header.state_diff_commitment`. Reject with `BadPeerError` if they differ. This is the same pattern already used in `apollo_committer` when `verify_state_diff_hash = true`.

---

### Proof of Concept

Concrete attack for a block whose canonical state diff contains one `DeclaredClass { class_hash: X, compiled_class_hash: CANONICAL }` (so `state_diff_length = 1`):

1. Attacker peer responds to the state diff query with:
   ```
   StateDiffChunk::DeclaredClass(DeclaredClass {
       class_hash: X,
       compiled_class_hash: EVIL_HASH,  // differs from canonical
   })
   ```
   followed by `DataOrFin(None)`.

2. `parse_data_for_block` computes `current_state_diff_len = 1 == target_state_diff_len = 1`. No duplicate key. Passes all checks.

3. `write_to_storage` calls `append_state_diff(block_number, ThinStateDiff { class_hash_to_compiled_class_hash: {X: EVIL_HASH}, ... })`.

4. `compiled_class_hash_table` now maps `X → EVIL_HASH`.

5. Any subsequent execution that resolves class `X` retrieves `EVIL_HASH` and selects the wrong (or nonexistent) CASM for execution.

pub fn calculate_state_diff_hash(state_diff: &ThinStateDiff) -> StateDiffCommitment {
    let mut hash_chain = HashChain::new();
    hash_chain = hash_chain.chain(&STARKNET_STATE_DIFF0);
    hash_chain = chain_deployed_contracts(&state_diff.deployed_contracts, hash_chain);
    hash_chain = chain_declared_classes(&state_diff.class_hash_to_compiled_class_hash, hash_chain);
    hash_chain =
        chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
    hash_chain = hash_chain.chain(&Felt::ONE) // placeholder.
        .chain(&Felt::ZERO); // placeholder.
    hash_chain = chain_storage_diffs(&state_diff.storage_diffs, hash_chain);
    hash_chain = chain_nonces(&state_diff.nonces, hash_chain);
    StateDiffCommitment(PoseidonHash(hash_chain.get_poseidon_hash()))
}
