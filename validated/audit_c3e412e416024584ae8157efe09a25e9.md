Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Position `owner` Instead of Transaction `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter and gates only on `owner`. Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint (unlike `removeLiquidity`), any address not on the allowlist can deposit into a permissioned pool by naming any allowlisted address as the LP position `owner`, completely defeating the pool admin's access control intent.

## Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` and passes `msg.sender` as `sender` to the extension hook with no equality check between the two:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

By contrast, `removeLiquidity` enforces `msg.sender == owner` before proceeding:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [2](#0-1) 

The extension hook signature names the first argument (the actual depositor/`sender`) as unnamed (`address,`) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [3](#0-2) 

The contract's own NatSpec and storage layout frame the guard around the "depositor" concept, but the actual depositor (`sender`) is never evaluated: [4](#0-3) 

**Exploit path:**
1. Pool is configured with `DepositAllowlistExtension`; `allowedDepositor[pool][alice] = true`; `bob` is NOT allowlisted.
2. `bob` calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. The hook receives `(bob, alice, ...)`, discards `bob`, and evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
4. `bob`'s tokens are transferred to the pool; `alice` receives LP shares.
5. `bob` has bypassed the deposit allowlist entirely.

## Impact Explanation

The deposit allowlist is a core access control mechanism for permissioned pools. Its complete bypass means any address — including sanctioned or otherwise excluded actors — can inject tokens and alter bin state. Additionally, the allowlisted victim (`alice`) receives LP shares in bins they never chose, and the attacker can repeat the deposit continuously to grief. This constitutes broken core pool functionality (access control) and enables forced pool-state manipulation, meeting the contest threshold for a valid finding.

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with `owner` set to any allowlisted address. No privileged access, flash loan, or special token is needed. The attacker must supply tokens (which enter the pool and are credited to the victim as LP shares), making the attack economically rational for griefing or pool-state manipulation. The barrier is trivially low and the attack is freely repeatable.

## Recommendation

Replace the unnamed first parameter with `sender` and check it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the design intent is to allow a whitelisted router to deposit on behalf of any `owner`, both `sender` and `owner` should be checked with an explicit policy (e.g., `sender` must be allowlisted, or `sender == owner`).

## Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  bob is NOT on the allowlist

Attack:
  bob calls pool.addLiquidity(
      owner        = alice,   // allowlisted — passes the guard
      salt         = 0,
      deltas       = { binIdxs: [0], shares: [1e18] },
      callbackData = ...,     // bob pays tokens in the callback
      extensionData = ""
  )

Result:
  - beforeAddLiquidity checks allowedDepositor[pool][alice] → true → no revert
  - bob's tokens are transferred to the pool
  - alice receives 1e18 LP shares in bin 0
  - bob has bypassed the deposit allowlist entirely
```

Foundry test: deploy pool with `DepositAllowlistExtension`, set `allowedDepositor[pool][alice] = true`, call `addLiquidity` from `bob` with `owner = alice`, assert the call succeeds and `bob`'s token balance decreases while `alice`'s position shares increase.
