### Title
Silent `AllResources`→`L1Gas` Downgrade in Protobuf `ValidResourceBounds` Deserialization Produces Wrong Transaction Hash Preimage — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter silently reclassifies an `AllResources` transaction whose `l2_gas` and `l1_data_gas` are both zero as `ValidResourceBounds::L1Gas`. Because `get_tip_resource_bounds_hash` hashes a different number of resource felts for the two variants (2 for `L1Gas`, 3 for `AllResources`), the transaction hash recomputed after deserialization diverges from the hash that was computed and stored at submission time. This is the sequencer analog of the ETH→WETH silent fallback: a type-level downgrade silently changes the canonical representation of the object, with downstream consequences for hash validation, fee accounting, and gas-vector computation mode.

---

### Finding Description

**Serialization path (sender side)**

`RpcInvokeTransactionV3` carries `AllResourceBounds` (never `ValidResourceBounds`). When a user submits a transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the gateway accepts it and the hash is computed via `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(...)`:

```rust
// crates/starknet_api/src/transaction_hash.rs:203-208
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
```

For `AllResources`, the hash preimage includes three resource felts: `L1_GAS`, `L2_GAS`, and `L1_DATA_GAS`. Even when `l1_data_gas = 0`, the zero-valued `L1_DATA_GAS` felt is included and changes the Poseidon digest.

**Deserialization path (receiver side)**

When the block is synced via P2P, the transaction is deserialized through `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`, which calls:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs:417-436
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        ...
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)   // ← silent downgrade
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
```

The serializer always emits `l1_data_gas = Some(zero)` for both `L1Gas` and `AllResources`-with-zero-data-gas:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs:471-489
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),
    l1_data_gas: Some(ResourceBounds::default().into()),  // zero
},
```

So a modern `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` transaction round-trips through protobuf as `L1Gas(X)`. The condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is satisfied, and the type is silently downgraded.

**Hash divergence**

After deserialization, if the hash is recomputed (e.g., in `validate_transaction_hash`), the `L1Gas` path produces a 2-element resource hash while the original `AllResources` path produced a 3-element resource hash. These are distinct Poseidon digests, so the recomputed hash does not match the stored hash.

Additionally, `get_gas_vector_computation_mode` returns `GasVectorComputationMode::NoL2Gas` for `L1Gas` and `GasVectorComputationMode::All` for `AllResources`, so fee computation, bouncer weights, and gas accounting all diverge.

**Contrast with the consensus path**

The consensus path (`TryFrom<protobuf::ResourceBounds> for AllResourceBounds` in `crates/apollo_protobuf/src/converters/rpc_transaction.rs`) correctly requires `l1_data_gas` to be present and never performs the downgrade. The vulnerability is isolated to the `ValidResourceBounds` converter used in the sync/storage path.

---

### Impact Explanation

A user submits a valid `AllResources` transaction with zero `l2_gas` and zero `l1_data_gas`. The sequencer accepts it, computes the correct `AllResources` hash, and includes it in a block. A syncing node deserializes the transaction as `L1Gas`, recomputes a different hash, and either:

1. Rejects the block due to hash mismatch (High: valid transactions rejected before sequencing / wrong receipt value returned by RPC).
2. Stores the transaction with the correct hash but executes it under `NoL2Gas` mode instead of `All` mode, producing wrong fee, gas, and bouncer accounting (Critical: incorrect fee/gas/resource accounting with economic impact).

---

### Likelihood Explanation

Any user can submit a V3 invoke transaction with `AllResourceBounds` where `l2_gas = 0` and `l1_data_gas = 0` (a pure L1-gas transaction expressed in the new format). This is a valid, accepted transaction type. No special privileges are required. The trigger is the ordinary P2P sync path, which is exercised by every syncing node.

---

### Recommendation

Replace the heuristic type-inference in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit discriminant. Options:

1. **Add a discriminant field to the protobuf `ResourceBounds` message** (e.g., a boolean `is_all_resources`) so the receiver can reconstruct the exact variant without guessing from zero values.
2. **Remove `ValidResourceBounds::L1Gas` from the sync path entirely**: since all modern transactions use `AllResources`, always deserialize as `AllResources` and let the hash-validation step reject genuinely malformed messages.
3. **At minimum**, remove the `unwrap_or_default()` fallback and the zero-based downgrade once 0.13.2 support is dropped (as the existing `TODO` comment acknowledges), and add a test that round-trips `AllResources { l2_gas: 0, l1_data_gas: 0 }` through protobuf and verifies the variant is preserved.

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
