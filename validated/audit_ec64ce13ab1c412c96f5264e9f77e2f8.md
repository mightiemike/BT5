Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End-User, Enabling Full Allowlist Bypass — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the immediate caller of `pool.swap`. When swaps are routed through `MetricOmmSimpleRouter`, the router contract becomes the pool's `msg.sender`. Any pool admin who allowlists the router (required for allowlisted users to reach the pool via the canonical periphery) simultaneously grants every unpermissioned user the ability to bypass the allowlist by routing through the same public router.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to `_beforeSwap`:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at lines 230–240. [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged to every extension:**

`ExtensionCalling._beforeSwap` encodes `sender` verbatim into the `abi.encodeCall` payload dispatched to each configured extension. [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` gates on `allowedSwapper[msg.sender][sender]`:**

`msg.sender` here is the pool; `sender` is the first argument — the immediate caller of `pool.swap`. The check passes if the pool has allowlisted that address. [3](#0-2) 

**Step 4 — `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the pool's `msg.sender`:**

The original end-user address (`msg.sender` of the router call) is stored only in transient storage for the payment callback and is never forwarded to the extension as `sender`. [4](#0-3) 

**Root cause:** The extension mechanism conflates the immediate pool caller (`msg.sender` of `pool.swap`) with the true end-user. When the router intermediates, these are different addresses. The same flaw applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Why existing guards fail:** The only check in `SwapAllowlistExtension` is `allowedSwapper[pool][sender]`. Once `sender` resolves to the router address and the router is allowlisted, the guard is permanently satisfied for every caller of the router, regardless of who they are.

**Contrast with `DepositAllowlistExtension`:** That extension gates on `owner`, which the liquidity adder passes explicitly and independently of the intermediate caller — it is not derived from `msg.sender` of the pool call. [5](#0-4) 

## Impact Explanation

Any user can execute swaps on a pool the admin explicitly restricted to a curated set of swappers. LP funds in the restricted pool are exposed to swap flows the pool admin prohibited. Depending on the pool's purpose (institutional-only, KYC-gated, rate-limited), this enables unauthorized price impact, extraction of LP value by excluded actors, and violation of regulatory or contractual restrictions. The loss is direct: LP principal is at risk from swap flows the allowlist was designed to block.

## Likelihood Explanation

The trigger requires the pool admin to allowlist the router — a natural and expected action for any allowlisted pool intended to be usable through the standard periphery. A pool admin who does not allowlist the router blocks all router-mediated swaps for everyone, including allowlisted users, making the allowlist effectively unusable with the canonical periphery. The admin is therefore forced into the vulnerable configuration to provide normal UX. The attack requires no special privileges: any unpermissioned EOA can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) targeting the restricted pool.

## Recommendation

Pass the original end-user address through the extension mechanism rather than the immediate pool caller. Two concrete options:

1. **Extension-data forwarding:** Require the router to encode the originating user address in `extensionData` and have `SwapAllowlistExtension` decode and verify it. The pool's `_beforeSwap` already forwards `extensionData` verbatim to extensions.
2. **Dedicated originator field:** Add an explicit `originator` field to the swap call signature (analogous to how `addLiquidity` separates `sender` from `owner`) so the pool can pass the true end-user to extensions independently of `msg.sender`.

Until fixed, pool admins must not allowlist the router on pools with `SwapAllowlistExtension`; they must require direct `pool.swap` calls from allowlisted EOAs only.

## Proof of Concept

```
Setup
─────
1. Deploy pool with SwapAllowlistExtension (beforeSwap order set).
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
3. Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack
──────
4. Bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool:          <restricted pool>,
           recipient:     bob,
           zeroForOne:    true,
           amountIn:      X,
           ...
       })

5. Router calls pool.swap(...) — pool's msg.sender = router.
6. Pool calls _beforeSwap(sender=router, ...).
7. Extension evaluates: allowedSwapper[pool][router] == true  →  passes.
8. Swap executes; Bob receives output tokens from the restricted pool.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
```
