Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Originating User, Allowing Allowlist Bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` from the pool's perspective — the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. If the pool admin allowlists the router to enable router-mediated swaps for legitimate users, any non-allowlisted user can bypass the gate by calling the router, because the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`.

## Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension is called by the pool), and `sender` is the argument forwarded from the pool's `swap` function.

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    recipient, ...
);
```

`ExtensionCalling._beforeSwap` then encodes this `sender` and calls the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
);
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` from the pool's perspective:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData   // caller-controlled, router does NOT inject msg.sender here
    );
```

The router stores the originating user (`msg.sender`) only in transient storage for the payment callback (`_setNextCallbackContext`), but never encodes it into any argument passed to `pool.swap()`. The extension therefore sees `sender = address(router)` for every router-mediated swap.

This creates an irresolvable dilemma for any pool admin who wants to use the allowlist with the router:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router — broken UX |
| **Allowlist the router** | Any non-allowlisted user bypasses the gate by calling the router |

The `DepositAllowlistExtension` does not share this flaw because it checks `owner` (the LP position owner), which is an explicit parameter passed by the caller to `addLiquidity` — not derived from `msg.sender`. The swap path has no equivalent owner-level argument; the only identity available is `sender = msg.sender`.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs, or whitelisted market makers) has that restriction completely bypassed. Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` targeting the restricted pool. The router forwards the swap to the pool; the extension sees `sender = router`; if the router is allowlisted, the swap executes. The non-allowlisted user receives pool output tokens at oracle-derived prices without authorization, constituting a direct loss of LP principal and a complete failure of the pool's access-control invariant. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact criteria.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface in the periphery. Any pool admin who deploys a swap-allowlisted pool and also wants to support the standard router path must allowlist the router — the protocol provides no other mechanism to pass the originating user identity to the extension. The bypass is therefore reachable on any production pool that uses both `SwapAllowlistExtension` and the router, which is the expected deployment pattern. No special privileges, flash loans, or non-standard tokens are required; a single public `exactInputSingle` call suffices.

## Recommendation

1. **Pass the originating user through `extensionData`**: Have `MetricOmmSimpleRouter` encode `msg.sender` into the `extensionData` bytes it forwards to the pool. Update `SwapAllowlistExtension.beforeSwap` to decode and check this address when `sender` is a known router, falling back to `sender` for direct calls.

2. **Alternatively, add a trusted-router registry and `originator` field**: The pool could accept an explicit `originator` address (verified via a trusted-router registry) and pass it as `sender` to extensions, preserving the allowlist semantics regardless of routing path.

3. **At minimum, document the incompatibility**: If neither fix is applied, the extension's NatSpec must warn that allowlisting the router defeats the allowlist for all users, and that router-mediated swaps are unsupported on allowlisted pools.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls swapExtension.setAllowedToSwap(pool, router, true)
    (to support router-mediated swaps for legitimate users)
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  1. attacker (not allowlisted) calls:
       router.exactInputSingle(ExactInputSingleParams({
           pool: restrictedPool,
           tokenIn: token0,
           ...
           recipient: attacker,
           extensionData: ""
       }))

  2. Router calls pool.swap(attacker_as_recipient, ...)
     with msg.sender = router

  3. Pool calls _beforeSwap(sender=router, ...)

  4. ExtensionCalling calls extension.beforeSwap(sender=router, ...)

  5. Extension checks allowedSwapper[pool][router] == true  ✓
     → does NOT revert

  6. Swap executes; attacker receives output tokens from the pool

Result:
  - attacker bypassed the swap allowlist
  - LP assets transferred to a non-allowlisted address
  - SwapAllowlistExtension invariant broken
```

Foundry test: deploy pool with `SwapAllowlistExtension` (as in `FullMetricExtension.t.sol`), allowlist the router address, then call `router.exactInputSingle` from a non-allowlisted EOA and assert the swap succeeds and output tokens are received.
