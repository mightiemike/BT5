### Title
`SwapAllowlistExtension` checks the router's address instead of the real user's address, allowing any caller to bypass the swap allowlist on curated pools — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool. Because `MetricOmmPool.swap` sets `sender = msg.sender` (the immediate caller of the pool), and `MetricOmmSimpleRouter` calls `pool.swap` directly, the extension sees the **router's address** as the swapper, not the real end-user. If the pool admin allowlists the router (the natural action to enable router-based swaps), every user—including those the allowlist was meant to exclude—can bypass the gate by routing through `MetricOmmSimpleRouter`.

---

### Finding Description

**Call chain:**

```
User → MetricOmmSimpleRouter.exactInputSingle()
         → IMetricOmmPoolActions(pool).swap(recipient, ..., extensionData)
              msg.sender to pool = MetricOmmSimpleRouter address
         → MetricOmmPool._beforeSwap(msg.sender, ...)
              sender = MetricOmmSimpleRouter address
         → SwapAllowlistExtension.beforeSwap(sender=router, ...)
              checks allowedSwapper[pool][router]   ← wrong actor
```

**Root cause — `MetricOmmPool.swap` binds `sender` to `msg.sender`:** [1](#0-0) 

`msg.sender` here is whoever called `pool.swap()`. When the call originates from `MetricOmmSimpleRouter`, that is the router contract, not the end-user.

**`ExtensionCalling._beforeSwap` forwards that value unchanged as `sender`:** [2](#0-1) 

**`SwapAllowlistExtension.beforeSwap` then checks the allowlist against `sender` (= router):** [3](#0-2) 

`msg.sender` inside the extension is the pool (correct), but `sender` is the router. The check `allowedSwapper[pool][sender]` therefore evaluates to `allowedSwapper[pool][router]`.

**Router calls `pool.swap` directly in every entry point:** [4](#0-3) [5](#0-4) 

In every hop of `exactInput`, `exactInputSingle`, `exactOutputSingle`, and `exactOutput`, the router is `msg.sender` to the pool. The real user's address is never forwarded to the extension.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` intends to restrict swaps to a specific set of addresses (e.g., KYC-verified counterparties). To allow those users to trade via the standard periphery, the admin must allowlist the router address. Once the router is allowlisted, **any address**—including those explicitly excluded from the allowlist—can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps against the restricted pool. The allowlist guard is completely nullified for all router-mediated swaps.

The consequence is direct loss of the curation policy: unauthorized users drain liquidity from a pool that was designed to serve only approved counterparties, and LP funds are exposed to trades the pool admin explicitly intended to block.

---

### Likelihood Explanation

- The `MetricOmmSimpleRouter` is the canonical, documented periphery swap path. Any pool admin who wants users to trade conveniently will allowlist it.
- No special privilege or setup is required from the attacker: any EOA can call `exactInputSingle` on the router.
- The bypass is deterministic and requires no timing, oracle manipulation, or multi-step preparation.

---

### Recommendation

The extension must gate on the **economically relevant actor** — the end-user — not the intermediate contract. Two complementary fixes:

1. **Pass the original user through the router.** Add a `payer` / `originator` field to the swap call or use a transient-storage pattern (already used for the callback payer in `MetricOmmSimpleRouter`) so the pool can forward the real user's address as `sender` to extensions.

2. **Check `recipient` or a separately forwarded originator in the extension.** Until the core is updated, the extension can require that `extensionData` carries the real user's address and verify it, though this is fragile.

The cleanest fix is to store `msg.sender` (the real user) in transient storage at the router entry point and expose it as a separate `originator` argument in the pool's `swap` signature, forwarding it to all extension hooks alongside `sender`.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup:
//   pool configured with SwapAllowlistExtension
//   pool admin calls: swapAllowlist.setAllowedToSwap(pool, address(router), true)
//   attacker = address NOT in allowedSwapper[pool]

// Step 1: attacker calls router directly — extension sees router as sender → passes
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:            address(pool),
        tokenIn:         token0,
        recipient:       attacker,
        zeroForOne:      true,
        amountIn:        1_000e18,
        amountOutMinimum: 0,
        priceLimitX64:   type(uint128).max,
        deadline:        block.timestamp + 1,
        extensionData:   ""
    })
);
// ✓ swap succeeds — attacker received token1 despite not being allowlisted

// Step 2: attacker calls pool directly — extension sees attacker as sender → reverts
pool.swap(attacker, true, int128(1_000e18), type(uint128).max, "", "");
// ✗ reverts with NotAllowedToSwap — confirming the allowlist exists but is bypassed via router
```

The allowlist check `allowedSwapper[pool][sender]` passes in Step 1 because `sender` is the router (allowlisted), and fails in Step 2 because `sender` is the attacker (not allowlisted). The guard is structurally bypassed for all router-mediated swaps.
