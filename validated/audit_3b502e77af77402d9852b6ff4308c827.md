### Title
`deser_auto` Silently Drops `max_atom_len` and `strict` Constraints for Legacy-Format Blobs — (`File: wheel/src/api.rs`)

### Summary

`deser_auto` accepts `max_atom_len` and `strict` parameters that callers expect to be enforced for all input formats. However, when the input blob is in legacy or backrefs format (i.e., does not start with `SERDE_2026_MAGIC_PREFIX`), the function falls back to `node_from_bytes_backrefs` which accepts neither parameter. Both constraints are silently discarded. An attacker who controls the serialized bytes can trivially bypass the caller-specified atom-size cap and strictness enforcement by omitting the magic prefix.

### Finding Description

`deser_auto` is the Python-facing auto-detecting deserializer exposed via the `clvm_rs` wheel. Its signature is:

```python
deser_auto(blob, *, max_atom_len=PY_DEFAULT_MAX_ATOM_LEN, strict=True)
```

Internally, the dispatch is: [1](#0-0) 

```rust
fn deser_auto(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
        deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)
            .map_err(eval_to_py)?
    } else {
        node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?   // ← max_atom_len and strict are dropped
    };
    Ok(LazyNode::new(Rc::new(a), node))
}
```

When the blob carries the serde_2026 magic prefix, `deserialize_2026_body_from_stream` receives both `max_atom_len` and `strict` and enforces them. When the blob is in legacy or backrefs format, `node_from_bytes_backrefs` is called with no size cap and no strictness flag. The caller's constraints are silently ignored.

The default cap is `PY_DEFAULT_MAX_ATOM_LEN = 1 << 20` (1 MiB): [2](#0-1) 

The `Allocator` itself has no per-atom size limit; its only hard ceiling is `heap_limit = u32::MAX` bytes: [3](#0-2) 

So a legacy-format blob containing a single atom of, say, 256 MiB will be accepted without error when passed to `deser_auto`, even though the caller specified `max_atom_len=1<<20`.

The inconsistency is structurally identical to the reference report: one code path (serde_2026 branch) enforces the constraint; the other code path (legacy/backrefs branch, the "constructor" analog) does not.

### Impact Explanation

**Memory exhaustion / DoS**: A caller relying on `max_atom_len` to bound memory consumption (e.g., a network service deserializing untrusted CLVM blobs) receives no protection when the attacker supplies a legacy-format blob. The allocator will happily extend `u8_vec` to the full atom size, potentially exhausting process memory.

**Strictness bypass**: The `strict` flag in the serde_2026 path enforces additional encoding invariants. For legacy blobs, no equivalent check exists. A caller that sets `strict=True` expecting canonical-encoding enforcement gets none of it for legacy-format input, which can cause consensus divergence if different nodes apply different validation rules.

### Likelihood Explanation

Likelihood is **high**. The attacker only needs to supply a blob that does not start with the six-byte `SERDE_2026_MAGIC_PREFIX`. Any legacy CLVM serialization (the dominant historical format) satisfies this condition. No special privileges or insider access are required. The Python API is the primary integration surface for the Chia node and wallet software, making this path reachable from any code that calls `deser_auto` on externally-supplied bytes.

### Recommendation

Apply `max_atom_len` and `strict` in the legacy/backrefs branch as well. Either:

1. Add a `max_atom_len`-aware wrapper around `node_from_bytes_backrefs` that rejects atoms exceeding the limit after deserialization (or during, if the deserializer is extended), and add a `strict` mode to the legacy deserializer; or
2. Document clearly in the function signature and docstring that `max_atom_len` and `strict` are **only** enforced for serde_2026 blobs, and rename the parameters or add a separate `deser_auto_uncapped` variant to make the asymmetry explicit; or
3. Reject legacy-format blobs in `deser_auto` entirely when `strict=True` or when a non-default `max_atom_len` is supplied, forcing callers to use format-specific functions.

### Proof of Concept

```python
import clvm_rs

# Craft a legacy-format atom of 4 MiB (well above the 1 MiB default cap).
# Legacy encoding: 0xfe <3-byte big-endian length> <data>
size = 4 * 1024 * 1024
header = bytes([0xfe, (size >> 16) & 0xff, (size >> 8) & 0xff, size & 0xff])
blob = header + b'\xAB' * size  # no SERDE_2026_MAGIC_PREFIX

# Caller expects max_atom_len=1<<20 to be enforced.
node = clvm_rs.deser_auto(blob, max_atom_len=1 << 20, strict=True)
# No exception is raised; a 4 MiB atom is returned, bypassing the cap.
print(len(node.atom))  # prints 4194304
```
