### Title
Missing Block-Participation Check in Base Rewards Top-Up Allocates Rewards to Absent Validators — (File: `x/tieredrewards/keeper/abci.go`)

---

### Summary

`topUpBaseRewards` in `abci.go` iterates over every entry in `sdkCtx.VoteInfos()` and calls `distributionKeeper.AllocateTokensToValidator` for each one, without ever checking whether the validator actually signed the block (`vote.BlockIdFlag`). Validators that were absent in the previous block receive a proportional share of the rewards-pool top-up they did not earn, draining the pool faster and reducing the allocation for validators that did sign.

---

### Finding Description

`topUpBaseRewards` collects the previous block's vote-info set and computes a `previousTotalPower` denominator that includes **all** validators — present and absent alike: [1](#0-0) 

It then loops over the same full set and unconditionally allocates a power-weighted share of `topUpAmount` to every validator: [2](#0-1) 

`sdkCtx.VoteInfos()` (backed by CometBFT's `LastCommitInfo`) includes every validator in the previous block's validator set, with `BlockIdFlag = BLOCK_ID_FLAG_ABSENT` for those that did not sign. The standard Cosmos SDK distribution module guards its own allocation loop with a `vote.BlockIdFlag == comet.BlockIDFlagCommit` check before crediting a validator. `topUpBaseRewards` has no equivalent guard.

The analog to the reported issue is direct:

| Reported issue | This repository |
|---|---|
| Oracle heartbeat updated even when `to_execute[idx]` is false | Rewards allocated to a validator even when `vote.BlockIdFlag` is `ABSENT` |
| Missing queue-membership check before processing | Missing participation check before allocating from the rewards pool |

---

### Impact Explanation

1. **Rewards pool drained by non-participants.** Every absent validator receives `(absentPower / totalPower) × topUpAmount` tokens transferred from `types.RewardsPoolName` to the distribution module and credited to that validator. These are real tokens from the module account that should only flow to signing validators.

2. **Signing validators under-compensated.** Because absent validators' power is included in `previousTotalPower`, the power-fraction denominator is inflated, so each signing validator's `powerFraction` is smaller than it should be, and they receive less than their fair share of the top-up.

3. **Accelerated pool depletion.** Over many blocks with one or more absent validators, the pool is consumed faster than the protocol intends, potentially exhausting it and leaving signing validators with no top-up at all.

---

### Likelihood Explanation

Any validator operator can cause this by simply not signing blocks. The action requires no special privilege — it is a normal, externally reachable condition (a validator node going offline or deliberately abstaining). The standard downtime-slashing risk exists, but the base-rewards top-up received may partially offset that penalty, and the impact on other validators' rewards is independent of whether the absent validator is acting maliciously.

---

### Recommendation

Before allocating rewards, skip validators whose `BlockIdFlag` is not `BLOCK_ID_FLAG_COMMIT`, mirroring the standard distribution module pattern:

```go
import comet "cosmossdk.io/core/comet"

for i, vote := range bondedVotes {
    if vote.BlockIdFlag != comet.BlockIDFlagCommit {
        continue
    }
    // existing allocation logic
}
```

Recompute `previousTotalPower` using only the signing validators so that power fractions remain correct.

---

### Proof of Concept

1. A validator set of two validators exists: V1 (power 90) and V2 (power 10).
2. V2 does not sign block N (its `BlockIdFlag` is `ABSENT`).
3. `topUpBaseRewards` runs at block N+1.
4. `previousTotalPower = 90 + 10 = 100` (absent V2 included).
5. `topUpAmount = 1000` tokens.
6. V1 receives `(90/100) × 1000 = 900` tokens; V2 receives `(10/100) × 1000 = 100` tokens.
7. V2 has received 100 tokens from the rewards pool despite not signing the block.
8. Had the check been present, V1 would have received the full 1000 tokens and V2 nothing.
