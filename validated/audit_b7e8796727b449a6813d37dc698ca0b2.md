Looking at the swap allowlist path: pool `swap()` → `_beforeSwap(msg.sender, ...)` → `SwapAllowlistExtension.beforeSwap(sender, ...)`.

The critical question is: what is `sender` when a user goes through the router?

In `MetricOmmPool.swap()`, `msg.sender` is whoever called the pool — the router when routed. The pool passes `msg.sender` as `sender` to the extension. The extension then checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router. So the allowlist gates the **router address**, not the actual user.

---

### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to its own `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, that value is the router's address, not the originating user. A pool admin who allowlists the router (a natural configuration to enable router-mediated swaps) inadvertently opens the pool to every user, completely defeating the allowlist.

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is whoever called the pool. When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) calls `pool.swap(...)`, the pool's `msg.sender` is the router: [4](#0-3) 

So the allowlist lookup becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The actual originating user is invisible to the extension.

### Impact Explanation

**Bypass path**: A pool admin who wants to allow router-mediated swaps adds the router to the allowlist (`allowedSwapper[pool][router] = true`). From that moment, every user — including those explicitly excluded — can swap by calling `MetricOmmSimpleRouter`. The allowlist provides zero protection against any user who routes through the public router.

**Inverse breakage**: If the admin does not allowlist the router, legitimately allowlisted users cannot use the router at all, breaking the primary swap UX path.

Both outcomes are fund-impacting: the bypass allows unauthorized parties to trade in a restricted pool (e.g., a KYC-gated or institutional pool), and the inverse breaks the withdraw/swap flow for legitimate users.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface. A pool admin configuring a `SwapAllowlistExtension` will almost certainly need to decide whether to allowlist the router. Allowlisting it is the natural choice to preserve usability, and nothing in the extension's interface or documentation signals that doing so opens the pool to all users. The trigger requires no special privilege — any user can call the public router.

### Recommendation

The extension must gate the **economic actor**, not the technical caller. Two viable approaches:

1. **Pass originator through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires router cooperation and a trusted encoding convention.

2. **Separate originator from caller in the pool interface**: Add an explicit `originator` field to the swap call (distinct from `sender`/`recipient`) that the pool populates with `msg.sender` for direct calls and that the router populates with its own `msg.sender`. The extension then gates on `originator`.

The simplest safe interim fix is to document that allowlisting the router is equivalent to `allowAllSwappers = true`, and provide a dedicated "router-aware" extension variant that decodes the originator from `extensionData`.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists only `alice` (allowedSwapper[pool][alice] = true)
  - Pool admin also allowlists the router (allowedSwapper[pool][router] = true)
    to let alice use the router

Attack:
  - bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(...)
  - Router calls pool.swap(recipient=bob, ...)  →  pool's msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - Extension checks allowedSwapper[pool][router] → true  →  swap proceeds
  - bob successfully swaps in a pool he was explicitly excluded from

Result:
  - allowedSwapper[pool][bob] is false, yet bob's swap executes
  - The allowlist invariant is broken; any user bypasses it via the router
``` [3](#0-2) [5](#0-4) [6](#0-5)
