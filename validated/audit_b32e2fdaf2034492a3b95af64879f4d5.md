### Title
Stale `byte_count` in `op_keccak256` Loop Allows Undercharged Cost Enforcement — (`File: src/keccak256_ops.rs`)

---

### Summary

`op_keccak256` uses a deferred `byte_count` accumulator in its per-iteration `check_cost` call. Each iteration checks cost using only the bytes from *previous* arguments, not the current one. The current argument's byte cost is added to `cost` only after the loop ends, with no subsequent `check_cost`. This is the direct analog of the `msg.value`-in-a-loop bug: a resource (byte cost) that should be fully accounted for per-iteration is instead applied with a one-iteration lag, allowing an attacker to force expensive hashing work past the cost gate.

---

### Finding Description

In `op_sha256` (the reference implementation), byte cost is added to `cost` **before** `check_cost` is called each iteration:

```rust
// src/more_ops.rs lines 400-406
cost += SHA256_COST_PER_ARG;
let blob = atom(a, arg, "sha256")?;
cost += blob.as_ref().len() as Cost * SHA256_COST_PER_BYTE;  // added first
check_cost(cost, max_cost)?;                                   // then checked
``` [1](#0-0) 

In `op_keccak256`, the order is inverted. `check_cost` is called with the **stale** `byte_count` from all prior iterations, and the current argument's bytes are added to `byte_count` only afterward. After the loop, the full byte cost is folded into `cost` with no `check_cost`:

```rust
// src/keccak256_ops.rs lines 26-35
cost += KECCAK256_COST_PER_ARG;
check_cost(
    cost + byte_count as Cost * KECCAK256_COST_PER_BYTE,  // byte_count is STALE
    max_cost,
)?;
let blob = atom(a, arg, "keccak256")?;
byte_count += blob.as_ref().len();   // updated AFTER the check
hasher.update(blob);                 // expensive work done
// ...
cost += byte_count as Cost * KECCAK256_COST_PER_BYTE;  // no check_cost here
``` [2](#0-1) 

On the first iteration `byte_count = 0`, so `check_cost(50 + 160, max_cost)` passes regardless of how large the argument atom is. The hasher then processes the full atom. After the loop, `cost` is updated to include all byte costs, but there is no `check_cost` call to enforce the limit before returning.

---

### Impact Explanation

An attacker supplies a single large atom (e.g., 1 MB) as the argument to `keccak256`. The per-iteration check passes (`check_cost(210, max_cost)` with `byte_count = 0`), the full 1 MB is fed into the Keccak hasher, and `op_keccak256` returns `Reduction(2_000_210, ...)` — a cost far exceeding `max_cost`. The `run_program` loop catches the overrun on the *next* iteration, but the expensive hashing work has already been performed. This breaks the invariant that cost enforcement gates work: a node can be forced to perform O(N) hashing work for an attacker-controlled N, bypassing the per-operator cost limit.

**Impact: Medium** — consensus-level cost enforcement is bypassed for the duration of the `op_keccak256` call; the overrun is caught by `run_program` afterward but only after the work is done. [3](#0-2) 

---

### Likelihood Explanation

**Likelihood: High** — `op_keccak256` is reachable via opcode `62` when `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` is set, and also inside softfork guards. An attacker only needs to craft a CLVM program that calls `keccak256` with one large atom argument. No special privileges or configuration are required beyond submitting a transaction. [4](#0-3) 

---

### Recommendation

Follow the pattern used by `op_sha256`: add the current argument's byte cost to `cost` **before** calling `check_cost`, so the check reflects the true cost of processing the current argument:

```rust
cost += KECCAK256_COST_PER_ARG;
let blob = atom(a, arg, "keccak256")?;
cost += blob.as_ref().len() as Cost * KECCAK256_COST_PER_BYTE;  // add first
check_cost(cost, max_cost)?;                                      // then check
byte_count += blob.as_ref().len();
hasher.update(blob);
```

Remove the post-loop `cost += byte_count * KECCAK256_COST_PER_BYTE` line, as the byte cost will already be accumulated inside the loop.

---

### Proof of Concept

Construct a CLVM program that calls `keccak256` with a single 500,000-byte atom. Set `max_cost = 500` (well above the base + per-arg cost of 210, but far below the true byte cost of `210 + 500_000 * 2 = 1_000_210`).

1. First loop iteration: `check_cost(50 + 160 + 0 * 2, 500)` → `check_cost(210, 500)` → **passes**.
2. `hasher.update(500_000_byte_blob)` — 500 KB of hashing work is performed.
3. Loop ends. `cost = 210 + 500_000 * 2 = 1_000_210`.
4. `op_keccak256` returns `Ok(Reduction(1_000_210, hash_node))`.
5. `run_program` adds `1_000_210` to its running `cost`, then on the next iteration detects `cost > max_cost` and returns `CostExceeded`.

The 500 KB hashing was performed despite the budget being 500. The broken invariant: `check_cost` inside `op_keccak256` never saw the true cost of the current argument. [5](#0-4) [6](#0-5)

pub fn op_keccak256(
    a: &mut Allocator,
    mut input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = KECCAK256_BASE_COST;

    let mut byte_count: usize = 0;
    let mut hasher = Keccak256::new();
    while let Some((arg, rest)) = a.next(input) {
        input = rest;
        cost += KECCAK256_COST_PER_ARG;
        check_cost(
            cost + byte_count as Cost * KECCAK256_COST_PER_BYTE,
            max_cost,
        )?;
        let blob = atom(a, arg, "keccak256")?;
        byte_count += blob.as_ref().len();
        hasher.update(blob);
    }
    cost += byte_count as Cost * KECCAK256_COST_PER_BYTE;
