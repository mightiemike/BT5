### Title
Sanctioned Recipient Bypass via Unchecked `sendTo` in `WithdrawCollateralV2` and `submitFastWithdrawal` — (File: `core/contracts/EndpointTx.sol`, `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

The Nado protocol enforces a sanctions check on both the depositor and the subaccount owner during `depositCollateralWithReferral`, but performs **no sanctions check on the `sendTo` recipient** in `WithdrawCollateralV2` or `submitFastWithdrawal`. Because `sendTo` is a user-controlled field embedded in the signed transaction, a non-sanctioned user can route protocol funds directly to a sanctioned address, bypassing the OFAC/sanctions enforcement the protocol explicitly implements.

---

### Finding Description

**Deposit path — sanctions enforced on both parties:**

In `Endpoint.depositCollateralWithReferral`, the protocol calls `requireUnsanctioned` on both the `msg.sender` (the depositor) and the `sender` (the subaccount owner extracted from the `subaccount` bytes32):

```solidity
requireUnsanctioned(msg.sender);
requireUnsanctioned(sender);
``` [1](#0-0) 

**Withdrawal path — `sendTo` recipient is never checked:**

In `EndpointTx.processTransactionImpl`, the `WithdrawCollateralV2` branch validates the sender's signature and charges a fee, then calls `clearinghouse.withdrawCollateral` with the user-supplied `signedTx.tx.sendTo` address. There is no `requireUnsanctioned(signedTx.tx.sendTo)` call anywhere in this path:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateralV2) {
    ...
    validateSignedTx(signedTx.tx.sender, ...);
    ...
    clearinghouse.withdrawCollateral(
        signedTx.tx.sender,
        signedTx.tx.productId,
        signedTx.tx.amount,
        signedTx.tx.sendTo,   // ← user-controlled, never sanctions-checked
        nSubmissions
    );
``` [2](#0-1) 

`Clearinghouse.withdrawCollateral` passes `sendTo` directly to `handleWithdrawTransfer` → `BaseWithdrawPool.submitWithdrawal` → `handleWithdrawTransfer` → `token.safeTransfer(to, ...)` with no sanctions check at any layer: [3](#0-2) [4](#0-3) 

**Fast-withdrawal path — same gap, directly user-callable:**

`BaseWithdrawPool.submitFastWithdrawal` is a `public` function with no access-control modifier. It resolves `sendTo` from the user-signed transaction via `resolveFastWithdrawal` and transfers tokens to it without any sanctions check:

```solidity
function submitFastWithdrawal(
    uint64 idx,
    bytes calldata transaction,
    bytes[] calldata signatures
) public {
    ...
    (uint32 productId, address sendTo, uint128 transferAmount) =
        resolveFastWithdrawal(transaction);
    ...
    handleWithdrawTransfer(token, sendTo, transferAmount);  // ← no sanctions check
}
``` [5](#0-4) 

`resolveFastWithdrawal` for `WithdrawCollateralV2` explicitly preserves the user-supplied `sendTo`:

```solidity
address resolvedSendTo = signedTx.tx.sendTo == address(0)
    ? address(uint160(bytes20(signedTx.tx.sender)))
    : signedTx.tx.sendTo;
return (signedTx.tx.productId, resolvedSendTo, signedTx.tx.amount);
``` [6](#0-5) 

---

### Impact Explanation

The protocol's sanctions enforcement is asymmetric: it blocks sanctioned addresses from depositing but does not block them from **receiving** funds. A non-sanctioned user who controls a sanctioned address (or who is colluding with one) can deposit collateral, sign a `WithdrawCollateralV2` transaction naming the sanctioned address as `sendTo`, and have the sequencer (or call `submitFastWithdrawal` directly) route the ERC20 tokens to the sanctioned address. The sanctioned address receives real on-chain token value with no protocol-level check. This directly undermines the sanctions compliance invariant the protocol explicitly implements via `ISanctionsList`. [7](#0-6) 

---

### Likelihood Explanation

The entry path is fully user-controlled: `sendTo` is a field in the EIP-712 signed struct that the user populates before signing. The sequencer submits the transaction as-is; it performs no on-chain sanctions check on `sendTo`. For the fast-withdrawal path, `submitFastWithdrawal` is `public` and callable by any address once valid signatures are assembled. No admin or governance action is required. The only friction is obtaining valid sequencer co-signatures for the fast-withdrawal path; the slow/sequencer-submitted path has no such friction.

---

### Recommendation

Add a `requireUnsanctioned` check on the resolved `sendTo` address in both paths:

1. **`EndpointTx.processTransactionImpl`** — after decoding `WithdrawCollateralV2`, add:
   ```solidity
   if (signedTx.tx.sendTo != address(0)) {
       requireUnsanctioned(signedTx.tx.sendTo);
   }
   ```

2. **`BaseWithdrawPool.submitFastWithdrawal`** — after `resolveFastWithdrawal`, add a sanctions check on `sendTo` before calling `handleWithdrawTransfer`. This requires exposing the sanctions list to `BaseWithdrawPool` (e.g., via the clearinghouse or a stored reference).

---

### Proof of Concept

1. Sanctioned address `S` is registered in the `ISanctionsList`.
2. Non-sanctioned address `A` calls `depositCollateralWithReferral` — passes both `requireUnsanctioned` checks.
3. `A` signs a `WithdrawCollateralV2` transaction with `sendTo = S` and `amount = X`.
4. The sequencer includes it in a `submitTransactionsChecked` batch. `processTransactionImpl` validates `A`'s signature, charges the fee, and calls `clearinghouse.withdrawCollateral(..., S, ...)`.
5. `Clearinghouse.withdrawCollateral` calls `handleWithdrawTransfer(token, S, X, idx)`.
6. `BaseWithdrawPool.submitWithdrawal` calls `token.safeTransfer(S, X)`.
7. `S` receives `X` tokens. No sanctions check was ever performed on `S`.

Alternatively, `A` (or any third party with the signed transaction) calls `BaseWithdrawPool.submitFastWithdrawal` directly with the signed `WithdrawCollateralV2` transaction, achieving the same result without sequencer involvement. [8](#0-7)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
