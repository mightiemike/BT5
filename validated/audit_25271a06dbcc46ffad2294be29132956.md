Audit Report

## Title
`_afterTimelock` uint32 truncation allows pool admin to bypass LP reaction window — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`_afterTimelock` casts `block.timestamp + timelock` directly to `uint32` without overflow validation. When `timelock = type(uint32).max`, the intermediate `uint256` sum exceeds `2^32` and truncates to a value approximately one second in the past, causing `_requireElapsed` to pass immediately for every subsequent timelocked operation. A pool admin can exploit this to disable the stop-loss mechanism with zero LP reaction time.

## Finding Description
`_afterTimelock` at L297-299 computes:
```solidity
return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
```
The addition is performed in `uint256` (no overflow), but the explicit `uint32(...)` cast silently truncates — Solidity 0.8's checked arithmetic does not protect explicit casts. With `block.timestamp ≈ 1,753,000,000` and `timelock = type(uint32).max = 4,294,967,295`, the sum is `6,047,967,295`, which truncates to `1,752,999,999 ≈ block.timestamp − 1`.

`_requireElapsed` at L301-303 then checks `if (block.timestamp < executeAfter)`, which evaluates to `false` immediately, bypassing the timelock entirely.

Neither `initialize` (L56-58) nor `proposeOracleStopLossTimelock` (L78-84) validates the `timelock` value — only `drawdownE6` and `decayPerSecondE8` are validated via `_validateDrawdown` and `_validateDecay`.

**Exploit path:**
1. Pool is initialized with `timelock = 0` (no validation, accepted as-is).
2. Admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)` — `_afterTimelock` returns `uint32(block.timestamp + 0) = block.timestamp`; `_requireElapsed` passes immediately.
3. Admin calls `executeOracleStopLossTimelock(pool)` — succeeds, sets `timelock = type(uint32).max`.
4. Admin calls `proposeOracleStopLossDrawdown(pool, 0)` — `_afterTimelock` wraps to `block.timestamp − 1`.
5. Admin calls `executeOracleStopLossDrawdown(pool)` — `_requireElapsed` passes immediately; `drawdownE6` set to 0.
6. Stop-loss is now disabled (`drawdown == 0` causes early return at L217); the same bypass applies to `proposeOracleStopLossDecay` (L133) and `proposeOracleStopLossHighWatermarks` (L162). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

## Impact Explanation
This is an admin-boundary break: the pool admin exceeds their intended capability by nullifying the LP reaction window that the timelock mechanism is designed to enforce. With `drawdownE6 = 0`, the stop-loss check short-circuits at L217 (`if (drawdown == 0) return`), disabling all value-loss protection for LPs. LPs are exposed to unbounded value extraction with no opportunity to exit, constituting a direct loss of LP principal above Sherlock thresholds. [5](#0-4) 

## Likelihood Explanation
The pool admin is a semi-trusted role constrained by timelocks. A pool initialized with `timelock = 0` (a valid and common default, accepted without validation) can be fully exploited in two transactions. No factory owner or oracle admin privilege is required. The exploit is fully on-chain, deterministic, and repeatable. The only precondition is holding the pool admin role.

## Recommendation
Add a maximum cap on the timelock value in both `initialize` and `proposeOracleStopLossTimelock`, ensuring `block.timestamp + timelock` cannot overflow `uint32`:

```solidity
uint256 private constant MAX_TIMELOCK = type(uint32).max / 2; // ~68 years, safe from overflow

function _validateTimelock(uint256 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Apply `_validateTimelock` in `initialize` alongside `_validateDrawdown` and `_validateDecay`, and in `proposeOracleStopLossTimelock` before storing `newTimelock`.
