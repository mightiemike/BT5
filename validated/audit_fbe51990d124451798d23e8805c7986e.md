### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` When `openInterest == 0` with Residual Negative `vQuoteBalance` — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when spreading a residual negative `vQuoteBalance`. If `openInterest == 0` (no open positions remain in the product) but the insolvent subaccount still carries a negative `vQuoteBalance` (realized loss from a previously closed position), the call reverts with `ERR_DIV_BY_ZERO`, permanently blocking subaccount finalization.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after insurance is exhausted, the code attempts to spread the remaining loss across all open-interest holders: [1](#0-0) 

```solidity
if (balance.vQuoteBalance < 0) {
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ← no zero-guard
    );
    state.cumulativeFundingLongX18 += fundingPerShare;
    state.cumulativeFundingShortX18 -= fundingPerShare;
    balance.vQuoteBalance = 0;
}
```

`MathSD21x18.div` unconditionally reverts when the denominator is zero: [2](#0-1) 

`state.openInterest` is the **global** sum of absolute position sizes across all subaccounts for a product, maintained in `_updateBalance`: [3](#0-2) 

When `balance.amount == 0`, a subaccount contributes `0` to `openInterest`. If no other subaccount holds an open position, `state.openInterest == 0`.

Crucially, `_finalizeSubaccount` **requires** `balance.amount == 0` for every perp product before calling `socializeSubaccount`: [4](#0-3) 

So the liquidatee's position is already closed (amount = 0) when `socializeSubaccount` is invoked. A negative `vQuoteBalance` with `amount == 0` is a normal realized-loss state: when a position is closed at a loss, `vQuoteBalance` captures the negative PnL while `amount` becomes 0.

The protocol itself acknowledges `openInterest` can be 0 — `updateStates` explicitly skips funding when `openInterest == 0`: [5](#0-4) 

But no equivalent guard exists in `socializeSubaccount`.

---

### Impact Explanation

The call chain `liquidateSubaccountImpl` → `_finalizeSubaccount` → `perpEngine.socializeSubaccount` reverts with `"DBZ"` every time it is attempted. The insolvent subaccount can never be finalized. The negative `vQuoteBalance` is never zeroed out, the insurance fund is never decremented to cover the loss, and the socialization that would spread the loss to other participants never executes. The protocol is permanently stuck: no liquidator can complete the finalization, and the broken accounting state persists indefinitely. [6](#0-5) 

---

### Likelihood Explanation

This is reachable in any product where the insolvent subaccount was the **last remaining participant** (or all others closed before finalization). This is realistic in:
- Low-liquidity or newly-launched perp markets
- Markets being wound down where participants exit before the insolvent account is resolved
- Any scenario where the insolvent account held the only open position and closed it at a loss before becoming insolvent (e.g., via funding payments accumulating into `vQuoteBalance`)

No special privileges or admin access are required. A standard liquidation flow triggers it.

---

### Recommendation

Add a zero-check before the division. If `state.openInterest == 0`, there is no one to socialize the loss to; the loss should simply be written off (set `vQuoteBalance = 0`) or handled as a protocol bad-debt event:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest != 0) {
        int128 fundingPerShare = -balance.vQuoteBalance.div(
            state.openInterest
        );
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
    }
    // If openInterest == 0, no one to socialize to; absorb as protocol bad debt
    balance.vQuoteBalance = 0;
}
```

---

### Proof of Concept

1. Deploy the protocol with a single perp product (productId = 2).
2. Subaccount A opens a long position: `amount = 100e18`, `vQuoteBalance = -10000e18` (paid 100 per unit).
3. Subaccount A closes the position at price 90: `amount = 0`, `vQuoteBalance = -1000e18` (realized loss of 1000). `state.openInterest = 0`.
4. Subaccount A's health drops below maintenance (e.g., insufficient spot quote balance).
5. Liquidator calls `liquidateSubaccountImpl` with `productId = type(uint32).max` to trigger `_finalizeSubaccount`.
6. `_finalizeSubaccount` passes the `balance.amount == 0` check at line 319.
7. `perpEngine.socializeSubaccount` is called with `insurance < 1000e18`.
8. After insurance is exhausted, `balance.vQuoteBalance` is still `< 0`.
9. `(-balance.vQuoteBalance).div(state.openInterest)` → `div(x, 0)` → revert `"DBZ"`.
10. Assert: the transaction reverts; repeat indefinitely — the subaccount can never be finalized.
