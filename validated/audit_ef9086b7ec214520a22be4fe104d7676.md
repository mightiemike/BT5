### Title
`ContractOwner` Cannot Manage Sequencer Keys Due to Missing Ownership Transfer in `Verifier` — (File: `core/contracts/Verifier.sol`, `core/contracts/ContractOwner.sol`)

---

### Summary

`Verifier.initialize()` sets ownership to `msg.sender` (the deployer EOA) via `__Ownable_init()` without transferring to `ContractOwner`. `ContractOwner.initialize()` stores a reference to `Verifier` but never calls `verifier.transferOwnership(address(this))`. As a result, `ContractOwner` is never the owner of `Verifier`, and the two `onlyOwner` wrapper functions `ContractOwner.assignPubKey()` and `ContractOwner.deletePubkey()` permanently revert on every call. The intended governance path for sequencer key rotation is irreversibly broken at the code level.

---

### Finding Description

`Verifier` is an `OwnableUpgradeable` contract. Its `initialize()` function calls `__Ownable_init()`, which sets `owner = msg.sender` — the deployer EOA — with no subsequent `transferOwnership` call:

```solidity
// Verifier.sol
function initialize(Point[8] memory initialSet) external initializer {
    __Ownable_init();   // owner = msg.sender (deployer EOA)
    for (uint256 i = 0; i < 8; ++i) {
        if (!isPointNone(initialSet[i])) {
            _assignPubkey(i, initialSet[i].x, initialSet[i].y);
        }
    }
}
```

The two privileged key-management functions on `Verifier` are gated by `onlyOwner`:

```solidity
function assignPubKey(uint256 i, uint256 x, uint256 y) public onlyOwner { ... }
function deletePubkey(uint256 index) public onlyOwner { ... }
```

`ContractOwner` is the protocol's governance contract. It stores a reference to `Verifier` and exposes wrapper functions intended to let the multisig manage sequencer keys:

```solidity
// ContractOwner.sol
function assignPubKey(uint256 i, uint256 x, uint256 y) public onlyOwner {
    verifier.assignPubKey(i, x, y);   // ← always reverts: ContractOwner ≠ Verifier.owner()
}

function deletePubkey(uint256 index) public onlyOwner {
    verifier.deletePubkey(index);      // ← always reverts: ContractOwner ≠ Verifier.owner()
}
```

`ContractOwner.initialize()` stores `verifier = Verifier(_verifier)` but never calls `verifier.transferOwnership(address(this))`:

```solidity
// ContractOwner.sol
function initialize(address multisig, address _deployer, ..., address _verifier, ...) external initializer {
    require(_deployer == msg.sender, "expected deployed to initialize");
    __Ownable_init();
    transferOwnership(multisig);
    ...
    verifier = Verifier(_verifier);   // stored, but ownership never transferred
    ...
}
```

Because `Verifier.owner()` remains the deployer EOA and `ContractOwner` is never granted ownership, every call through `ContractOwner.assignPubKey()` or `ContractOwner.deletePubkey()` hits the `onlyOwner` revert inside `Verifier`. The governance path is permanently broken at the code level — no deployment step inside either contract corrects this.

This is structurally identical to the reported `LiquidAccessories`/`Deployer` bug: an intermediary governance contract holds a reference to a target contract and exposes `onlyOwner` wrappers, but the target contract's ownership was never transferred to the intermediary, so the wrappers are permanently inoperable.

---

### Impact Explanation

The sequencer public keys stored in `Verifier` are the root of trust for all transaction validation in the protocol. `requireValidSignature()` uses these keys to verify every sequencer-submitted batch. If a sequencer key is compromised or needs rotation, the only on-chain governance path — `ContractOwner.assignPubKey()` / `ContractOwner.deletePubkey()` — permanently reverts. The multisig, which owns `ContractOwner`, cannot exercise the key-management authority the contract was designed to provide. The deployer EOA retains direct access to `Verifier`, but this is an unintended residual privilege, not the designed governance model, and represents a permanent desynchronization between the intended ownership model and the actual on-chain state.

---

### Likelihood Explanation

This manifests unconditionally from the moment `Verifier` and `ContractOwner` are initialized, with no special attacker action required. Any attempt by the multisig to call `ContractOwner.assignPubKey()` or `ContractOwner.deletePubkey()` will revert. The bug is latent until key rotation is needed, at which point it becomes a hard blocker. There is no self-healing path in the code.

---

### Recommendation

In `ContractOwner.initialize()`, after storing the `verifier` reference, transfer `Verifier`'s ownership to `address(this)`:

```solidity
verifier = Verifier(_verifier);
verifier.transferOwnership(address(this));
```

This mirrors the pattern already used for `ContractOwner` itself (`transferOwnership(multisig)`) and ensures `ContractOwner` can exercise the `onlyOwner` functions on `Verifier` as designed. Alternatively, add a one-time `onlyDeployer` function in `ContractOwner` that calls `verifier.transferOwnership(address(this))`, allowing the deployer EOA to perform the handoff post-deployment.

---

### Proof of Concept

1. Deployer EOA calls `Verifier.initialize(initialSet)` → `Verifier.owner() == deployerEOA`.
2. Deployer EOA calls `ContractOwner.initialize(multisig, deployer, ..., address(verifier), ...)` → `ContractOwner` stores `verifier` reference; `Verifier.owner()` is still `deployerEOA`, not `address(contractOwner)`.
3. Multisig calls `ContractOwner.assignPubKey(0, x, y)`.
4. `ContractOwner.assignPubKey` calls `verifier.assignPubKey(0, x, y)`.
5. Inside `Verifier.assignPubKey`, `onlyOwner` checks `msg.sender == owner()` → `address(contractOwner) != deployerEOA` → **revert**.
