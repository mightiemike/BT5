### Title
Unsigned `oraclePriceX18` and `nlpPoolRebalanceX18` in `MintNlp`/`BurnNlp` Digest Allows NLP Accounting Manipulation - (File: `core/contracts/Verifier.sol`)

---

### Summary

`Verifier.computeDigest` omits `oraclePriceX18` and `nlpPoolRebalanceX18` from the signed digest for `MintNlp` and `BurnNlp` transactions. These fields are present in `SignedMintNlp` and `SignedBurnNlp` but are never committed to by the user's signature. Because `processTransactionImpl` is declared `public` with no access-control modifier of its own, any caller who obtains a valid user signature can replay it with an arbitrarily chosen `oraclePriceX18`, directly corrupting the NLP token amount minted or the quote amount returned on burn.

---

### Finding Description

`SignedMintNlp` and `SignedBurnNlp` each carry two fields outside the inner `tx` struct:

```solidity
// IEndpoint.sol lines 118-136
struct SignedMintNlp {
    MintNlp tx;           // sender, quoteAmount, nonce  ← signed
    bytes   signature;
    int128  oraclePriceX18;       // ← NOT signed
    int128[] nlpPoolRebalanceX18; // ← NOT signed
}

struct SignedBurnNlp {
    BurnNlp tx;           // sender, nlpAmount, nonce    ← signed
    bytes   signature;
    int128  oraclePriceX18;       // ← NOT signed
    int128[] nlpPoolRebalanceX18; // ← NOT signed
}
``` [1](#0-0) 

`Verifier.computeDigest` builds the digest for both transaction types using only the inner `tx` fields:

```solidity
// Verifier.sol lines 378-398
digest = keccak256(abi.encode(
    keccak256(bytes(MINT_NLP_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.quoteAmount,
    signedTx.tx.nonce          // oraclePriceX18 absent
));
...
digest = keccak256(abi.encode(
    keccak256(bytes(BURN_NLP_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.nlpAmount,
    signedTx.tx.nonce          // oraclePriceX18 absent
));
``` [2](#0-1) 

`EndpointTx.processTransactionImpl` is declared `public` with no sequencer guard of its own. It decodes the full `SignedMintNlp` struct, validates only the inner-`tx` signature, and then passes the unsigned `oraclePriceX18` directly to `clearinghouse.mintNlp`:

```solidity
// EndpointTx.sol lines 534-553
validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce,
                 transaction, signedTx.signature, true);
...
clearinghouse.mintNlp(
    signedTx.tx,
    signedTx.oraclePriceX18,      // attacker-controlled
    nlpPools,
    signedTx.nlpPoolRebalanceX18  // attacker-controlled
);
``` [3](#0-2) 

Inside `Clearinghouse.mintNlp`, `oraclePriceX18` is used without any independent oracle validation to compute the NLP amount the user receives:

```solidity
// Clearinghouse.sol lines 462-466
spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);
int128 quoteAmount = int128(txn.quoteAmount);
int128 nlpAmount   = quoteAmount.div(oraclePriceX18);
``` [4](#0-3) 

The same pattern applies symmetrically to `BurnNlp`, where `oraclePriceX18` determines the quote returned to the burner. [5](#0-4) 

---

### Impact Explanation

An attacker who obtains a legitimate user signature for a `MintNlp` transaction (e.g., from the public mempool or off-chain order book) can call `processTransactionImpl` directly with the same signature but a manipulated `oraclePriceX18`:

- **Inflated mint**: setting `oraclePriceX18` to a near-zero value causes `quoteAmount.div(oraclePriceX18)` to overflow toward a very large `nlpAmount`, minting far more NLP tokens than the deposited quote warrants.
- **Deflated burn**: setting `oraclePriceX18` to a near-zero value on `BurnNlp` causes the user to receive almost no quote for their NLP tokens, effectively stealing the quote that should have been returned.
- **Pool rebalance manipulation**: `nlpPoolRebalanceX18` is also unsigned; an attacker can redirect quote flows among NLP pool subaccounts arbitrarily, draining specific pools.

The corrupted state delta is: `spotEngine` balances for `NLP_PRODUCT_ID` and `QUOTE_PRODUCT_ID` across the sender and all NLP pool subaccounts. [6](#0-5) 

---

### Likelihood Explanation

`processTransactionImpl` carries no access-control modifier; it is `public` on `EndpointTx`. [7](#0-6) 

Any user who has seen a valid `MintNlp` or `BurnNlp` signature (e.g., intercepted from the sequencer submission pipeline or the off-chain API) can front-run the sequencer by calling `processTransactionImpl` directly with a crafted `oraclePriceX18` before the nonce is consumed. The nonce check in `validateNonce` only prevents replay after the transaction is processed; it does not prevent a first-use substitution attack. [8](#0-7) 

---

### Recommendation

1. Include `oraclePriceX18` and `nlpPoolRebalanceX18` in the EIP-712 struct hash for `MintNlp` and `BurnNlp`, updating `MINT_NLP_SIGNATURE` and `BURN_NLP_SIGNATURE` accordingly.
2. Add an access-control guard (e.g., `requireSequencer`) directly to `processTransactionImpl` so that even if the digest is correct, only the authorized sequencer can submit these transactions.
3. Validate `oraclePriceX18` against an on-chain oracle bound inside `mintNlp`/`burnNlp` as a defense-in-depth measure. [9](#0-8) 

---

### Proof of Concept

1. Alice signs a `MintNlp` transaction: `{sender: alice, quoteAmount: 1000e18, nonce: 5}`. The signature commits only to these three fields.
2. The signed transaction (with `oraclePriceX18 = 1e18`, `nlpPoolRebalanceX18 = [1000e18]`) is broadcast to the sequencer.
3. Bob observes Alice's signature off-chain. Bob constructs a new `SignedMintNlp` with the same `tx` and `signature` but sets `oraclePriceX18 = 1` (1 wei).
4. Bob calls `EndpointTx.processTransactionImpl` directly with this crafted struct before the sequencer processes Alice's transaction.
5. `validateSignedTx` passes because the digest only covers `sender`, `quoteAmount`, `nonce` — all unchanged.
6. `clearinghouse.mintNlp` computes `nlpAmount = 1000e18 / 1 = 1000e18` NLP tokens instead of the intended `1000` NLP tokens, crediting Bob's (or Alice's) subaccount with a massively inflated NLP balance at the expense of the `N_ACCOUNT` reserve. [10](#0-9)
