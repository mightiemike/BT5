Audit Report

## Title
`_clampMetric` Sentinel Permanently Corrupts High-Watermark, Blocking Swaps or Suppressing Stop-Loss — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

When `_metrics` produces an overflow under extreme-but-legitimate oracle-price conditions, `_clampMetric` silently returns `type(uint104).max` (`METRIC_MAX`). Because `_applyWatermark` unconditionally ratchets the stored watermark up to the current metric, a single swap writes `type(uint104).max` into `hwmS.token0` or `hwmS.token1`. Every subsequent swap then either permanently false-triggers the stop-loss (blocking all swaps in one direction) or permanently suppresses real breaches (stop-loss blind), depending on whether the extreme price condition resolves.

## Finding Description

**Root cause — `_clampMetric`:** [1](#0-0) 

With `t1 = type(uint104).max`, `midPriceX64 = 1`, `shares = minShares = 1`:
- `mulDiv(t1, Q64, 1) = 2^104 * 2^64 = 2^168`
- `mulDiv(2^168, 1e6, 1) ≈ 2^188 >> type(uint104).max`

`_clampMetric` returns `METRIC_MAX`. [2](#0-1) 

**Watermark ratchet — `_applyWatermark`:**

When `metric = type(uint104).max` and stored `hwm < type(uint104).max`, the condition `metric >= hwm` is true, so `newHwm = type(uint104).max` is returned with `breached = false`. No guard prevents the sentinel from being written. [3](#0-2) 

**Storage write — `_checkAndUpdateWatermarks`:**

Since `breach0 = false`, no revert occurs, and the sentinel is unconditionally persisted: [4](#0-3) 

**Mode A — permanently blocked swaps:**

On the next swap with a normal metric (e.g., `metricT0 = 1000`), `_applyWatermark(1000, type(uint104).max, 950_000)` evaluates:
- `1000 >= type(uint104).max`? No.
- `breached = 1000 < (type(uint104).max * 950_000) / 1e6 ≈ 1.93e31` → TRUE.

`breach0 && zeroForOne` → `revert OracleStopLossTriggered`. Every `zeroForOne = true` swap is permanently blocked. [5](#0-4) 

**Mode B — permanently suppressed stop-loss:**

If the extreme price persists, every swap produces `metricT0 = type(uint104).max`. `_applyWatermark(type(uint104).max, type(uint104).max, ...)` → `metric >= hwm` → `breached = false`. Real value loss is invisible to the guard. [6](#0-5) 

**Existing test confirms the sentinel write but does not test subsequent behavior:** [7](#0-6) 

**Recovery requires pool-admin action through a timelocked proposal:** [8](#0-7) 

## Impact Explanation

- **Mode A**: Core swap functionality broken. All swaps in one direction revert with `OracleStopLossTriggered` regardless of actual pool health. LPs cannot rebalance; traders cannot trade. Recovery requires pool-admin action through `proposeOracleStopLossHighWatermarks` + `executeOracleStopLossHighWatermarks`, subject to the configured timelock — during which the pool remains unusable.
- **Mode B**: The stop-loss guard is silently disabled for the affected bin and direction. A real drawdown event goes undetected, allowing bad-price swaps to drain LP value without triggering the intended protection.

Both outcomes satisfy "broken core pool functionality causing loss of funds or unusable swap flows" and "bad-price execution" impact gates.

## Likelihood Explanation

The conditions are reachable without any privileged action:

1. **Extreme oracle price** — `midPriceX64` is derived from `(bidPriceX64 + askPriceX64) / 2`. For token pairs with a very large or very small price ratio (e.g., a high-value token0 paired with a micro-cap token1), `midPriceX64` can legitimately be 1 or near-zero in Q64.64 representation. [9](#0-8) 

2. **Large balance, few shares** — The code explicitly floors shares at `minShares` (minimum 1), so a bin with accumulated fees or partially drained shares but large token balances can trigger the overflow. [10](#0-9) 

3. **Trigger** — Any unprivileged trader executing a swap through the public pool interface causes `afterSwap` to fire, which calls `_afterSwapOracleStopLoss` and writes the corrupted watermark. [11](#0-10) 

## Recommendation

In `_checkAndUpdateWatermarks`, skip the watermark write when the returned `newHwm` equals `METRIC_MAX`, since the metric is outside the range where the stop-loss can make a meaningful comparison:

```solidity
(uint256 hwm0, bool breach0) = _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
if (breach0 && zeroForOne) revert OracleStopLossTriggered(...);
if (hwm0 < METRIC_MAX) {   // never persist the sentinel
    hwmS.token0 = uint104(hwm0);
}
```

Alternatively, propagate a `clamped` flag from `_clampMetric` and skip both the breach check and the watermark update when the metric is clamped.

## Proof of Concept

```solidity
// Setup: bin with near-max balance, 1 share, extreme (low) oracle price
_storeBin(0, type(uint104).max, type(uint104).max, 1);
_configure(50_000, 0);   // 5% drawdown, no decay

// Swap 1: extreme price → metric clamped → watermark set to type(uint104).max
_exposeStopLoss(0, 0, 1 /* midPriceX64 = 1 */, false);
(uint256 hwm0,) = extension.currentHighWatermarks(address(pool), 0);
assertEq(hwm0, type(uint104).max);   // confirmed by existing test

// Swap 2: normal price, normal balances → false breach permanently blocks swaps
_storeBin(0, 1000, 1000, 1e18);
vm.expectRevert(OracleStopLossTriggered.selector);
_exposeStopLoss(0, 0, uint128(Q64), true);   // zeroForOne blocked forever
```
