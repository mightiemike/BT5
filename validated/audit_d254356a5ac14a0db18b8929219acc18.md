Audit Report

## Title
Unchecked `newTimelock` in `proposeOracleStopLossTimelock` Enables uint32 Truncation, Bypassing LP-Protection Timelock ‚Äî (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`proposeOracleStopLossTimelock` accepts any `uint32` value for `newTimelock` with no upper-bound validation. Setting `newTimelock = type(uint32).max` causes `_afterTimelock` to produce a `uint32`-truncated `executeAfter` value that falls one second in the past, making every subsequent proposal immediately executable in the same block. The pool admin can then atomically set `drawdownE6 = E6`, reducing `floorMultiplier` to zero and permanently silencing the stop-loss guard for all LP positions.

## Finding Description

**Root cause 1 ‚Äî no cap on `newTimelock`:**

`proposeOracleStopLossTimelock` stores the caller-supplied value directly with no upper-bound check: [1](#0-0) 

**Root cause 2 ‚Äî explicit `uint32` truncation in `_afterTimelock`:**

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [2](#0-1) 

Solidity 0.8.x explicit casts do **not** revert on truncation. With `block.timestamp ‚âà 1,750,000,000` and `timelock = type(uint32).max = 4,294,967,295`:

```
sum  = 1,750,000,000 + 4,294,967,295 = 6,044,967,295   (uint256, no overflow)
uint32(6,044,967,295) = 6,044,967,295 mod 4,294,967,296 = 1,749,999,999
```

`executeAfter = 1,749,999,999` is one second in the past.

**Root cause 3 ‚Äî `_requireElapsed` passes immediately:** [3](#0-2) 

Since `block.timestamp (1,750,000,000) >= executeAfter (1,749,999,999)`, the check passes without revert.

**Root cause 4 ‚Äî `drawdownE6 = E6` is a valid input:** [4](#0-3) 

`_validateDrawdown` only rejects values strictly greater than `E6`, so `drawdownE6 = 1_000_000` passes.

**Root cause 5 ‚Äî `floorMultiplier = 0` silences the guard:**

In `_afterSwapOracleStopLoss`, `floorMultiplier = E6 - drawdown`. With `drawdown = E6`, `floorMultiplier = 0`. [5](#0-4) 

`breached = metric < (hwm * 0) / E6 = 0` is always `false` for any `uint256 metric`. The stop-loss guard is permanently silenced for all bins and both swap directions.

**Exploit flow:**
1. Pool is initialized with `timelock = T` (e.g., 7 days) and some `drawdownE6`.
2. Admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)`. The proposal's own `executeAfter` is computed with the current (legitimate) timelock `T`.
3. Admin waits `T` seconds, then calls `executeOracleStopLossTimelock`. Now `oracleStopLossConfig[pool].timelock = type(uint32).max`.
4. Any subsequent `proposeOracleStopLoss*` call computes `_afterTimelock` ‚Üí truncated past timestamp ‚Üí `executeAfter` is already elapsed.
5. Admin calls `proposeOracleStopLossDrawdown(pool, 1e6)` then immediately `executeOracleStopLossDrawdown(pool)` in the same block.
6. `drawdownE6 = E6`, `floorMultiplier = 0`, stop-loss never fires.

## Impact Explanation

LPs who accepted a pool specifically because it carried a meaningful timelock (e.g., 7 days) to protect against sudden stop-loss weakening lose that protection entirely. After the guard is disabled, the pool admin can execute swaps that drain LP-owned token balances without the `afterSwap` hook ever reverting. This is a direct loss of LP principal and an explicit admin-boundary break: the pool admin bypasses the timelock that is the sole LP safety mechanism for this extension. This matches the allowed impact gate: **admin-boundary break ‚Äî pool admin bypasses timelocks**.

## Likelihood Explanation

The pool admin is a semi-trusted role explicitly permitted to call `proposeOracleStopLossTimelock`. The only prerequisite is waiting for the current timelock to elapse once. After that single wait, all future parameter changes are immediately executable with no further delay. Any pool admin who turns adversarial after deployment can execute this path deterministically with no external dependencies.

## Recommendation

Add an upper-bound check in `proposeOracleStopLossTimelock` (and mirror it in `initialize`) to prevent values that would cause `_afterTimelock` to overflow:

```solidity
uint32 constant MAX_TIMELOCK = 365 days;

function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock)
    external onlyPoolAdmin(pool_)
{
    if (newTimelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(newTimelock);
    // ... existing logic
}
```

Apply the same guard inside `initialize` for the constructor-time `timelock` parameter.

## Proof of Concept

```solidity
// Pool initialized with timelock = 1 days, drawdownE6 = 50_000.

// Step 1: propose type(uint32).max timelock (uses current 1-day timelock for executeAfter)
extension.proposeOracleStopLossTimelock(pool, type(uint32).max);
vm.warp(block.timestamp + 1 days);
extension.executeOracleStopLossTimelock(pool);
// oracleStopLossConfig[pool].timelock == type(uint32).max

// Step 2: _afterTimelock now overflows ‚Üí executeAfter is in the past
// Propose and execute drawdown = 1e6 in the SAME block
extension.proposeOracleStopLossDrawdown(pool, 1e6);
extension.executeOracleStopLossDrawdown(pool); // succeeds immediately

// Step 3: verify stop-loss is dead
assertEq(oracleStopLossConfig[pool].drawdownE6, 1e6);
// floorMultiplier = E6 - 1e6 = 0 ‚Üí breach never fires ‚Üí afterSwap never reverts
```
