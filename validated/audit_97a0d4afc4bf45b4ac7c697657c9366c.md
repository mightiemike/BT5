### Title
Feeder-gateway client silently drops `proof_facts`, producing hash/body mismatch served by RPC — (`crates/apollo_starknet_client/src/reader/objects/transaction.rs`, `crates/starknet_api/src/transaction_hash.rs`)

---

### Summary

`get_invoke_transaction_v3_hash` conditionally appends the `proof_facts` hash to the transaction hash preimage only when `proof_facts` is non-empty. The feeder-gateway (central-sync) client silently defaults `proof_facts` to `ProofFacts::default()` (empty) whenever the field is absent from the gateway response. Because `proof_facts` is only returned by the gateway when the caller explicitly requests `response_flags: ["INCLUDE_PROOF_FACTS"]` (per Starknet RPC v0.10), any `InvokeTransactionV3` carrying non-empty `proof_facts` that is ingested through central sync will be stored with `proof_facts = []`. The block-level transaction hash (returned by the gateway and stored verbatim) was computed over the original non-empty `proof_facts`, so the stored hash and the stored transaction body are permanently inconsistent. The RPC then serves this inconsistent pair as an authoritative response.

---

### Finding Description

**Step 1 – Conditional hash inclusion (hash domain boundary)**

`get_invoke_transaction_v3_hash` in `crates/starknet_api/src/transaction_hash.rs` appends the `proof_facts` nested hash only when the field is non-empty:

```rust
if !transaction.proof_facts().0.is_empty() {
    let proof_facts_hash =
        HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
    hash_chain = hash_chain.chain(&proof_facts_hash);
}
```

This is the canonical Starknet hash rule: a transaction with `proof_facts = [a, b, c]` produces a **different** hash than the same transaction with `proof_facts = []`.

**Step 2 – Silent drop in the feeder-gateway client (serialization boundary)**

`TryFrom<IntermediateInvokeTransaction> for starknet_api::transaction::InvokeTransactionV3` in `crates/apollo_starknet_client/src/reader/objects/transaction.rs` uses:

```rust
// proof_facts is optional for backward compatibility with V3 transactions created
// before client-side proving was added. Old V3 txs don't have this field.
proof_facts: invoke_tx.proof_facts.unwrap_or_default(),
```

`IntermediateInvokeTransaction.proof_facts` is `Option<ProofFacts>`. When the gateway response omits the field (the default for any gateway that does not receive `response_flags: ["INCLUDE_PROOF_FACTS"]`), `unwrap_or_default()` silently substitutes `ProofFacts::default()` (empty). No error is raised, no hash re-verification is performed.

**Step 3 – Stored hash vs. stored body diverge**

The gateway also returns the transaction hash as part of the block header / transaction index. That hash was computed by the original submitter over the real (non-empty) `proof_facts`. The syncing node stores:

- **Transaction index**: original hash `H(…, proof_facts_hash)`.
- **Transaction body**: `InvokeTransactionV3 { proof_facts: [] }`.

These two values are now permanently inconsistent in the MDBX storage.

**Step 4 – RPC serves the inconsistent pair**

`get_transaction_by_hash` in `crates/apollo_rpc/src/v0_8/api/api_impl.rs` looks up the stored body by index and returns `TransactionWithHash { transaction: body, transaction_hash: stored_hash }`. The body has `proof_facts = []`; the hash was computed over non-empty `proof_facts`. Any consumer that recomputes the hash from the returned body obtains a completely different value.

The storage backward-compat shim in `crates/apollo_storage/src/serialization/serializers.rs` (which defaults `proof_facts` to empty when the trailing bytes are absent) compounds the problem: even if the body were re-read from disk after a restart, the empty `proof_facts` is preserved.

---

### Impact Explanation

The RPC endpoint `starknet_getTransactionByHash` returns an authoritative-looking `TransactionWithHash` whose `transaction_hash` field does not match the hash that would be computed from the returned `transaction` body. This is a direct instance of **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

Secondary consequences:
- Any re-execution or simulation of the stored transaction (e.g., `starknet_simulateTransactions`) uses the body with empty `proof_facts`, so `validate_proof_facts` passes trivially, but the hash exposed to the account contract's `__validate__` entry point is the original hash (with `proof_facts`). The account sees a hash it cannot reproduce from the `TxInfo` it receives.
- The Starknet OS `AssertTransactionHash` hint recomputes the hash from the transaction body; it would compute a hash that differs from the stored one, causing proving to fail for any block containing such a transaction.

---

### Likelihood Explanation

Central sync (feeder-gateway) is the primary mechanism by which Apollo nodes bootstrap from the existing Starknet chain. The Starknet RPC v0.10 specification makes `proof_facts` an opt-in field gated behind `response_flags: ["INCLUDE_PROOF_FACTS"]`. The test comment in `crates/blockifier_reexecution/src/state_reader/raw_rpc_json_test.rs` confirms this:

> "returned only when the request includes `response_flags: ["INCLUDE_PROOF_FACTS"]`, per Starknet RPC v0.10"

Unless the central-sync reader explicitly sets this flag on every transaction-fetch request, every `InvokeTransactionV3` with non-empty `proof_facts` ingested through central sync will trigger the mismatch. Because client-side proving (SNIP-36) is an active feature and `proof_facts` is already present in mainnet blocks (as shown by `crates/apollo_starknet_client/resources/reader/block_post_0_14_3.json`), the condition is reachable without any attacker action.

---

### Recommendation

1. **Request `proof_facts` explicitly**: The central-sync reader should include `response_flags: ["INCLUDE_PROOF_FACTS"]` in every `get_transaction` / `get_block_with_txs` request so that the gateway returns the field.

2. **Verify hash after deserialization**: After constructing an `InvokeTransactionV3` from the gateway response, recompute the transaction hash and assert it equals the hash stored in the block. Reject the block if they differ.

3. **Harden the fallback**: Replace `unwrap_or_default()` with an explicit error when the transaction's stored hash implies non-empty `proof_facts` (i.e., when the hash computed over the empty-`proof_facts` body does not match the block-level hash).

---

### Proof of Concept

1. A user submits `InvokeTransactionV3` with `proof_facts = [PROOF_VERSION_V1, VIRTUAL_SNOS, …]` to an Apollo sequencer node A. The gateway computes and stores hash `H_full = poseidon(…, poseidon(proof_facts…))`.

2. Node B starts central sync. It fetches the block from node A's feeder-gateway endpoint **without** `response_flags: ["INCLUDE_PROOF_FACTS"]`.

3. `TryFrom<IntermediateInvokeTransaction>` fires `proof_facts: invoke_tx.proof_facts.unwrap_or_default()` → `proof_facts = []`. [1](#0-0) 

4. Node B stores the body with `proof_facts = []` but indexes it under `H_full` (taken from the block header).

5. A client calls `starknet_getTransactionByHash(H_full)` on node B. The response is `{ transaction_hash: H_full, transaction: { …, proof_facts: [] } }`.

6. The client recomputes the hash from the returned body: `H_empty = poseidon(…)` (no `proof_facts` term appended because `proof_facts.is_empty()` is true). [2](#0-1) 

7. `H_empty ≠ H_full`. The RPC has returned an authoritative-looking but internally inconsistent transaction.
