### Title
Hardcoded `amount=1` in `getCoreRisk` Always Returns `longWeight`, Inflating Spread Health for Short Perp Positions — (`core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine.getCoreRisk` passes the hardcoded literal `1` to `RiskHelper._getWeightX18`, so the `CoreRisk.longWeight` field always holds the **long** weight regardless of the actual position direction. When `Clearinghouse.getHealth` computes the spread health adjustment, it uses this field for both `existingWeight` and `spreadWeight`. For a spread with a **short perp**, the correct short weight (`> 1`) is never used, causing the spread health contribution to be inflated — and in typical parameter ranges, to flip from negative to positive.

---

### Finding Description

**Root cause — `BaseEngine.getCoreRisk` (line 190):**

```solidity
// core/contracts/BaseEngine.sol:179-192
function getCoreRisk(...) external returns (IProductEngine.CoreRisk memory) {
    RiskHelper.Risk memory risk = _risk(productId);
    (int128 amount, ) = _getBalance(productId, subaccount);
    return IProductEngine.CoreRisk(
        amount,
        risk.priceX18,
        RiskHelper._getWeightX18(risk, 1, healthType)  // ← hardcoded 1, not amount
    );
}
``` [1](#0-0) 

`_getWeightX18` branches on `amount >= 0`. Since `1 >= 0` is always true, it unconditionally returns `longWeightInitialX18` or `longWeightMaintenanceX18`:

```solidity
// core/contracts/libraries/RiskHelper.sol:44-52
if (amount >= 0) {
    weight = healthType == IProductEngine.HealthType.INITIAL
        ? risk.longWeightInitialX18
        : risk.longWeightMaintenanceX18;
} else {  // ← never reached from getCoreRisk
    weight = healthType == IProductEngine.HealthType.INITIAL
        ? risk.shortWeightInitialX18
        : risk.shortWeightMaintenanceX18;
}
``` [2](#0-1) 

**Propagation — `Clearinghouse.getHealth` spread block (lines 125–135):**

```solidity
int128 existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2;
int128 spreadWeight   = RiskHelper._getSpreadWeightX18(perpCoreRisk, spotCoreRisk, healthType);
health += basisAmount.mul(spotCoreRisk.price + perpCoreRisk.price).mul(spreadWeight - existingWeight);
``` [3](#0-2) 

`_getSpreadWeightX18` also reads `perpCoreRisk.longWeight` directly:

```solidity
// core/contracts/libraries/RiskHelper.sol:66-67
if (spotCoreRisk.amount > 0) {
    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
``` [4](#0-3) 

Both `existingWeight` and `spreadWeight` are therefore computed with the **long** weight of the perp, even when `perpCoreRisk.amount < 0`.

---

### Impact Explanation

The individual health calculation in `_calculateProductHealth` correctly passes the actual `amount` to `_getWeightX18`, so a short perp is already penalized with `shortWeight > 1`. The spread adjustment is meant to partially relieve that penalty for a hedged position. Because `getCoreRisk` returns the wrong (lower) weight, the relief is over-stated.

**Concrete numbers** (typical perp: `longWeightInitial = 0.9`, `shortWeightInitial = 1.1`; spot: `longWeightInitial = 0.9`):

| | With bug (`amount=1`) | Correct (`amount < 0`) |
|---|---|---|
| `perpCoreRisk.longWeight` | 0.9 | 1.1 |
| `existingWeight` | (0.9+0.9)/2 = **0.90** | (0.9+1.1)/2 = **1.00** |
| `spreadWeight` | 1−(1−0.9)/5 = **0.98** | min(1−(1−1.1)/5, 0.99) = **0.99** |
| `spreadWeight − existingWeight` | **+0.08** | **−0.01** |
| Health delta sign | **positive (inflated)** | **negative (correct)** |

The spread health contribution flips sign. A trader holding a long spot + short perp spread receives a net health bonus instead of a net health penalty, allowing them to hold undercollateralized positions and evade liquidation.

---

### Likelihood Explanation

- The spread path is reached by any subaccount that holds opposing spot and perp positions in a registered spread pair — a standard, documented trading strategy on the protocol.
- No special privileges, admin access, or external dependencies are required.
- The bug is deterministic: every call to `getCoreRisk` for a short perp position produces the wrong weight.

---

### Recommendation

Replace the hardcoded `1` with the actual `amount` in `getCoreRisk`:

```solidity
// core/contracts/BaseEngine.sol
return IProductEngine.CoreRisk(
    amount,
    risk.priceX18,
    RiskHelper._getWeightX18(risk, amount, healthType)  // use actual amount
);
``` [5](#0-4) 

This ensures `CoreRisk.longWeight` reflects the correct directional weight, making both `existingWeight` and `spreadWeight` in `_getSpreadWeightX18` accurate for short positions.

### Proof of Concept

```solidity
// Hardhat test (unmodified deployment)
// Setup: register a spread pair (spotId, perpId)
// Trader: deposit collateral, open long spot + short perp of equal size

// 1. Record getHealth before spread
int128 healthBefore = clearinghouse.getHealth(subaccount, HealthType.INITIAL);

// 2. Manually compute correct health:
//    individualPerpHealth = perpAmount * shortWeightInitial * perpPrice  (negative)
//    correctExistingWeight = (spotLongWeight + perpShortWeight) / 2 = 1.0
//    correctSpreadWeight   = min(1 - (1 - perpShortWeight)/5, 0.99) = 0.99
//    correctSpreadDelta    = basisAmount * (spotPrice+perpPrice) * (0.99 - 1.0) < 0

// 3. Assert: getHealth returns a value HIGHER than the manually computed correct health
//    (demonstrating inflated credit for the short perp in the spread)
assertGt(healthBefore, correctHealth);

// 4. Show the trader can open additional positions that would be rejected
//    under correct health accounting, confirming undercollateralization.
```

The fuzz variant: iterate over `perpAmount < 0`, `spotAmount > 0`, compare `getHealth` output against the manually computed value using `shortWeightInitialX18` for the perp — the protocol value will always be strictly greater, confirming systematic over-crediting.
