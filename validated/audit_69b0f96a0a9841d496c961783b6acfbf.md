### Title
Softfork Cost Atom Non-Canonical Encoding Causes Consensus/Mempool Divergence — (`src/run_program.rs`)

### Summary

`uint_atom::<8>` is called with the dialect's live flags to parse the softfork cost argument. Because `MEMPOOL_MODE` includes `CANONICAL_INTS` while consensus mode (`ClvmFlags::empty()`) does not, a cost atom with a spurious leading zero (e.g., `[0x00, 0x01]` encoding the value `1`) is silently accepted in consensus mode but hard-rejected in mempool mode. The error propagates via `?` before any `allow_unknown_ops` rescue logic is reached, making the divergence unconditional.

### Impact Explanation

A spend bundle whose CLVM generator contains a `softfork` call with a non-canonical cost atom (any atom with a spurious leading `0x00` byte, e.g., `0x0001` for cost 1) is:

- **Accepted** by consensus-mode full nodes — the leading zero is stripped and execution proceeds normally.
- **Rejected** by mempool-mode nodes — the `InvalidOpArg` error is returned before any `allow_unknown_ops` rescue path is reached.

A farmer/miner can include such a spend bundle directly in a block. The block passes consensus validation on all full nodes. However, the spend bundle could never have been relayed through the standard mempool path, and any node that re-validates it in mempool mode (e.g., during re-org processing or wallet sync) will incorrectly classify it as invalid. This is a concrete consensus/mempool split.

---

### Likelihood Explanation

The atom encoding is fully attacker-controlled: CLVM programs are deserialized as raw byte sequences with no normalization applied before execution. Any farmer who controls block production can craft and include such a spend bundle. No special privileges beyond block production are required.

---

### Recommendation

Pass `ClvmFlags::empty()` (or a flags value explicitly stripped of `CANONICAL_INTS`) when calling `uint_atom::<8>` for the softfork cost argument, so that cost atom parsing is mode-independent. Canonical enforcement for the cost field should either be applied uniformly in both modes or not at all. The `CANONICAL_INTS` strictness is appropriate for operator arguments but must not create a consensus/mempool split on the cost field itself.

### Proof of Concept

use clvm_rs::chia_dialect::{ChiaDialect, ClvmFlags, MEMPOOL_MODE};
use clvm_rs::run_program::run_program;
use clvm_rs::allocator::Allocator;

// CLVM: (softfork (q . <cost_atom_0x0001>) (q . 0) (q . (q . ())) (q . ()))
// cost atom bytes = [0x00, 0x01]  (non-canonical encoding of 1)
// In practice, build the program tree directly in the allocator.

fn main() {
    let mut alloc = Allocator::new();
    // ... build softfork program with cost atom [0x00, 0x01] ...

    let consensus = ChiaDialect::new(ClvmFlags::empty());
    let mempool   = ChiaDialect::new(MEMPOOL_MODE);

    let r_consensus = run_program(&mut alloc, &consensus, prg, env, 10_000);
    let r_mempool   = run_program(&mut alloc, &mempool,   prg, env, 10_000);

    // r_consensus → Ok(...)   (leading zero stripped, cost = 1)
    // r_mempool   → Err(InvalidOpArg: "softfork requires u64 arg with no leading zeros")
    assert!(r_consensus.is_ok());
    assert!(r_mempool.is_err());
    // Divergence confirmed.
}
