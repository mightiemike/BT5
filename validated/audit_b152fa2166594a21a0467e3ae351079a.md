Audit Report

## Title
Router-Mediated Swap Passes Router Address as `sender` to `SwapAllowlistExtension`, Allowing Any User to Bypass Per-User Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the address that called `pool.swap`. When a user routes through `MetricOmmSimpleRouter`, the pool receives the router as `msg.sender`, so the extension checks the router's allowlist status rather than the user's. If the router is allowlisted (required for any router-mediated swap to succeed), the per-user allowlist is completely bypassed for all router-mediated swaps.

## Finding Description
In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` in `ExtensionCalling.sol` forwards this value directly to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called the pool — the router, not the original user: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly with no mechanism to forward the original user's identity: [4](#0-3) 

The same structural issue applies to `exactOutputSingle`, `exactInput`, and `exactOutput`. There are no existing guards that compensate for this identity confusion — the extension has no access to `tx.origin` or any forwarded user context.

## Impact Explanation
Two concrete failure modes exist. First, if the pool admin allowlists the router (required for any user to use the router), any unprivileged address can call `exactInputSingle` and have the hook see `sender = router` (allowlisted), completely defeating the per-user allowlist. This is a broken core pool functionality impact: the access control extension fails to enforce its stated invariant, allowing unauthorized swaps on pools that should have blocked them. Second, if the admin does not allowlist the router, legitimately allowlisted users cannot use the router at all — their swaps revert with `NotAllowedToSwap`. There is no configuration that correctly allows specific users to use the router while blocking others.

## Likelihood Explanation
Any pool deploying `SwapAllowlistExtension` with `allowAllSwappers = false` and expecting users to interact via the router is affected. The router is the standard public entrypoint for swaps in the periphery. The bypass requires only a single transaction from any unprivileged address. The pool admin has no in-protocol remedy without replacing the extension, since the identity confusion is structural.

## Recommendation
The extension must receive the original user's identity, not the intermediary's. The preferred fix is to have the router encode `msg.sender` into `extensionData` and have the extension decode and verify it — but this is forgeable unless the pool or a trusted forwarder signs the context. The cleanest fix is for `beforeSwap` to receive a separate `origin` parameter (analogous to EIP-2771 trusted forwarder patterns) so the hook can check the true initiator. Alternatively, the router should enforce allowlist checks at the router level before calling the pool, using a separate allowlist that the router consults.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)  // required for any router use
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack (single transaction):
  - attacker calls router.exactInputSingle({pool: pool, ...})
  - router calls pool.swap(recipient, ...)  // msg.sender = router
  - pool calls _beforeSwap(sender=router, ...)
  - extension checks allowedSwapper[pool][router] == true  ✅ (passes)
  - attacker's swap executes on an allowlisted pool they were never authorized to use
```
