### Title
Gateway OS-Constraint Stateless Checks Applied in RPC Path but Absent from Consensus Proposal Conversion Path — (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`, `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, `crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

The `StatelessTransactionValidator` in the gateway enforces several OS-level field constraints on RPC-submitted transactions before they are converted to their internal representation. The consensus proposal validation path calls `convert_consensus_tx_to_internal_consensus_tx()` — which internally calls the same `convert_rpc_tx_to_internal()` helper — but never invokes the stateless validator. A malicious proposer can therefore include transactions that carry non-empty `paymaster_data`, non-empty `account_deployment_data`, or non-L1 `nonce_data_availability_mode` / `fee_data_availability_mode`. These transactions pass conversion, reach the batcher, are executed by the blockifier, and then fail inside the Starknet OS, producing wrong receipts and wrong committed state.

---

### Finding Description

**Gateway path (RPC → Internal):**

`StatelessTransactionValidator::validate()` is called before conversion and enforces four OS-level invariants, each documented with an explicit comment that the OS enforces the constraint:

```
validate_empty_paymaster_data      // "The Starknet OS enforces that the paymaster data is empty."
validate_empty_account_deployment_data  // "The Starknet OS enforces that the deployer data is empty."
validate_nonce_data_availability_mode   // "The Starknet OS enforces that the nonce DA mode is L1."
validate_fee_data_availability_mode     // "The Starknet OS enforces that the fee DA mode is L1."
```

These checks reject the transaction at the gateway before it ever reaches the converter.

**Consensus path (ConsensusTransaction → InternalConsensusTransaction):**

In `validate_proposal.rs`, incoming proposal transactions are processed as:

```rust
transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
```

`convert_consensus_tx_to_internal_consensus_tx()` dispatches to the private `convert_rpc_tx_to_internal()` helper. That helper computes the transaction hash and, for Declare transactions, validates the compiled class hash — but it never calls `StatelessTransactionValidator::validate()` or any equivalent check. The four OS-level field constraints are therefore completely absent from this path.

The structural asymmetry is exact:

| Check | Gateway (RPC) path | Consensus path |
|---|---|---|
| `paymaster_data` empty | ✅ enforced | ❌ absent |
| `account_deployment_data` empty | ✅ enforced | ❌ absent |
| `nonce_data_availability_mode == L1` | ✅ enforced | ❌ absent |
| `fee_data_availability_mode == L1` | ✅ enforced | ❌ absent |
| `compiled_class_hash` match | ✅ enforced | ✅ enforced (via `add_class`) |

---

### Impact Explanation

The Starknet OS enforces all four constraints. When a transaction that violates them reaches the OS (after passing through the blockifier, which does not re-check these fields in `perform_pre_validation_stage`), the OS fails the transaction. The transaction is committed to the block with a failure receipt, and the resulting state diff and event log diverge from what a correctly-validated block would produce. This matches:

> **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

Concretely, a transaction with `fee_data_availability_mode != L1` carries a different fee-accounting semantic that the OS rejects at the OS level, producing a wrong revert result for an otherwise well-formed transaction. A transaction with non-empty `paymaster_data` causes the OS to fail the transaction outright, writing a failure receipt into a block that a correctly-operating node would never have included.

---

### Likelihood Explanation

Any Byzantine proposer node — one that has been legitimately selected as proposer for a consensus round — can craft such a transaction. No special privilege beyond proposer selection is required. The proposer constructs a `ConsensusTransaction::RpcTransaction` with the offending field set, streams it in a `ProposalPart::Transactions` batch, and the validator node's `handle_proposal_part` loop converts it without any field-level check. The attack is deterministic and requires no brute-force or timing dependency.

---

### Recommendation

Apply the same OS-constraint checks in the consensus conversion path. The cleanest fix is to call `StatelessTransactionValidator::validate()` (or an equivalent subset covering the four OS-enforced fields) inside `convert_consensus_tx_to_internal_consensus_tx()` before delegating to `convert_rpc_tx_to_internal()`, or to add the four field checks directly into `convert_rpc_tx_to_internal()` so they are enforced on every call site regardless of entry point.

---

### Proof of Concept

1. A Byzantine proposer constructs an `RpcInvokeTransactionV3` with `fee_data_availability_mode: DataAvailabilityMode::L2` (any value other than `L1`).
2. It wraps this as `ConsensusTransaction::RpcTransaction(tx)` and streams it in a `ProposalPart::Transactions` batch.
3. The validator node's `handle_proposal_part` calls `transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)`.
4. `convert_consensus_tx_to_internal_consensus_tx` dispatches to `convert_rpc_tx_to_internal`, which computes the transaction hash and returns `Ok(InternalConsensusTransaction::RpcTransaction(...))` — no field validation occurs.
5. The internal transaction is forwarded to the batcher via `send_txs_for_proposal`.
6. The blockifier executes the transaction; `perform_pre_validation_stage` does not check DA modes.
7. The Starknet OS, which enforces `fee_data_availability_mode == L1`, fails the transaction.
8. The committed block contains a wrong failure receipt for a transaction that should have been rejected at admission, producing divergent state from any correctly-operating sequencer.

                let (internal_tx, proof_data) = self.convert_rpc_tx_to_internal(tx).await?;
                let task = proof_data.map(|(proof_facts, proof)| {
                    self.spawn_verify_and_store_proof(proof_facts, proof)
                });
                Ok((InternalConsensusTransaction::RpcTransaction(internal_tx), task))
            }
            ConsensusTransaction::L1Handler(tx) => {
                let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
                Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))
            }
        }
    }

    async fn convert_rpc_tx_to_internal(
        &self,
        tx: RpcTransaction,
    ) -> TransactionConverterResult<(InternalRpcTransaction, Option<(ProofFacts, Proof)>)> {
        let (tx_without_hash, proof_data) = match tx {
            RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => {
                let proof_data = if tx.proof_facts.is_empty() {
                    None
                } else {
                    Some((tx.proof_facts.clone(), tx.proof.clone()))
                };
                (InternalRpcTransactionWithoutTxHash::Invoke(tx.into()), proof_data)
            }
            RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => {
                let ClassHashes { class_hash, executable_class_hash_v2 } =
                // TODO(Dori): Make this async and spawn a task to compile and add it to the class manager.
                    self.class_manager_client.add_class(tx.contract_class).await?;
                // TODO(Aviv): Ensure that we do not want to
                // allow declare with compiled class hash v1.
                if tx.compiled_class_hash != executable_class_hash_v2 {
                    return Err(TransactionConverterError::ValidateCompiledClassHashError(
                        ValidateCompiledClassHashError::CompiledClassHashMismatch {
                            computed_class_hash: executable_class_hash_v2,
                            supplied_class_hash: tx.compiled_class_hash,
                        },
                    ));
                }
                (
                    InternalRpcTransactionWithoutTxHash::Declare(InternalRpcDeclareTransactionV3 {
                        sender_address: tx.sender_address,
                        compiled_class_hash: tx.compiled_class_hash,
                        signature: tx.signature,
                        nonce: tx.nonce,
                        class_hash,
                        resource_bounds: tx.resource_bounds,
                        tip: tx.tip,
                        paymaster_data: tx.paymaster_data,
                        account_deployment_data: tx.account_deployment_data,
                        nonce_data_availability_mode: tx.nonce_data_availability_mode,
                        fee_data_availability_mode: tx.fee_data_availability_mode,
                    }),
                    None,
                )
            }
            RpcTransaction::DeployAccount(RpcDeployAccountTransaction::V3(tx)) => {
                let contract_address = tx.calculate_contract_address()?;
                (
                    InternalRpcTransactionWithoutTxHash::DeployAccount(
                        InternalRpcDeployAccountTransaction {
                            tx: RpcDeployAccountTransaction::V3(tx),
                            contract_address,
                        },
                    ),
                    None,
                )
            }
        };
        let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
        Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
    }
