### Title
Deprecated `verify_transaction_inclusion` Remains Callable, Bypassing Coinbase Merkle Proof Validation - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (v1) is still a live, unpermissioned public NEAR function. `verify_transaction_inclusion_v2` was introduced specifically to add a coinbase merkle proof check that mitigates the 64-byte transaction Merkle forgery attack. However, v1 is never removed or gated, so any caller can invoke it directly and bypass the coinbase proof requirement entirely — the exact same "alternative path that skips the guard" pattern as the external report.

### Finding Description

`verify_transaction_inclusion_v2` adds two guards absent from v1:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — enforces tree-depth consistency.
2. A full coinbase merkle proof check — proves the coinbase transaction (position 0) is a real leaf, blocking the 64-byte internal-node forgery.

After those guards pass, v2 delegates to v1 internally:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
```

But v1 itself is still a public `#[pause]`-gated (not role-gated) function:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
```

`#[deprecated]` in Rust is a compiler lint only — it does not remove the function from the NEAR ABI. Any NEAR account can call `verify_transaction_inclusion` directly, skipping the coinbase proof entirely.

The v1 docstring even acknowledges the gap explicitly:

> *"This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification."*

That assumption is not enforced anywhere in the contract.

### Impact Explanation

The 64-byte transaction forgery (https://www.bitmex.com/blog/64-Byte-Transactions) works because Bitcoin's Merkle tree hashes pairs of 32-byte values, so a crafted 64-byte blob can be interpreted as either a transaction or an internal node. Without the coinbase proof, `compute_root_from_merkle_proof` cannot distinguish between the two.

An attacker who calls v1 directly can supply a `tx_id` that is actually an internal Merkle node of a real mainchain block. The function will compute a root that matches `header.block_header.merkle_root` and return `true` — a false positive proof of inclusion. Any downstream NEAR contract or bridge that calls `verify_transaction_inclusion` (v1) to gate fund releases, token mints, or cross-chain state transitions will accept the forged proof.

### Likelihood Explanation

The entry path requires no privilege: `verify_transaction_inclusion` is callable by any NEAR account when the contract is not paused. The 64-byte forgery technique is publicly documented and tooled. The attacker only needs a real mainchain block (already stored in the contract) and the ability to craft a suitable 64-byte blob whose double-SHA256 matches an internal node — a purely offline computation. Likelihood is **medium-high** for any deployment where a downstream contract consumes v1.

### Recommendation

**Short term:** Add the same coinbase merkle proof requirement to `verify_transaction_inclusion` (v1), or gate it with a role that prevents unprivileged callers from reaching it.

**Long term:** Remove `verify_transaction_inclusion` (v1) from the public ABI entirely. The `#[deprecated]` annotation provides no on-chain enforcement; only removal or access control does.

### Proof of Concept

1. Identify any block `B` stored in `mainchain_header_to_height` with at least two transactions.
2. Offline, find an internal Merkle node `N` (64 bytes) in `B`'s transaction tree such that `compute_root_from_merkle_proof(N, position, sibling_path) == B.merkle_root`.
3. Call `verify_transaction_inclusion` directly with `tx_id = N`, `tx_block_blockhash = B`, the matching `merkle_proof`, and any `confirmations ≤ gc_threshold`.
4. The function returns `true` despite `N` not being a real transaction.
5. A downstream contract that calls v1 to authorize a cross-chain action will execute based on the forged proof.

    #[pause]
    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
