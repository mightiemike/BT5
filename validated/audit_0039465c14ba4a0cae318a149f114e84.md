### Title
Incomplete Post-State-Change Health Check in `forceRebalanceNlpPool` Skips First NLP Pool — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

In `Clearinghouse.forceRebalanceNlpPool`, `_applyNlpRebalance` updates balances for **all** NLP pools (starting at index 0), but the subsequent health-check loop starts at `i = 1`, permanently skipping the health verification of `nlpPools[0]`. This allows the first NLP pool's subaccount to be left with negative initial health after a rebalance, violating the protocol's solvency invariant.

---

### Finding Description

`forceRebalanceNlpPool` applies a rebalance to all NLP pools via `_applyNlpRebalance`, which iterates from `i = 0`: [1](#0-0) 

However, the post-state-change health check loop starts at `i = 1`: [2](#0-1) 

`nlpPools[0]` receives a balance update from `_applyNlpRebalance` but its health is **never verified**. A rebalance that pushes `nlpPools[0]`'s initial health below zero will succeed without reversion.

The full function: [3](#0-2) 

---

### Impact Explanation

The first NLP pool's subaccount can be left with negative initial health after any `forceRebalanceNlpPool` call. NLP pools hold user-deposited quote collateral. An unhealthy `nlpPools[0]` means the pool is operating in an under-collateralized state — the exact solvency corruption class of H-07. Downstream operations (burns, profit shares, withdrawals) that rely on the pool's health being non-negative will be operating against a corrupted accounting baseline, leading to potential loss of funds for NLP depositors.

---

### Likelihood Explanation

The sequencer submits `forceRebalanceNlpPool` transactions as a routine protocol operation. The bug is in the **protocol's own invariant enforcement**, not in the sequencer's intent. Because the protocol never checks `nlpPools[0]`, the sequencer has no on-chain signal that the first pool was left unhealthy. Any rebalance where `nlpPoolRebalanceX18[0]` is sufficiently negative to push `nlpPools[0]`'s health below zero will silently corrupt state. This is a latent accounting error that accumulates invisibly.

---

### Recommendation

Change the health-check loop to start at `i = 0` so all pools are verified after the rebalance:

```solidity
for (uint128 i = 0; i < nlpPools.length; i++) {
    require(
        getHealth(
            nlpPools[i].subaccount,
            IProductEngine.HealthType.INITIAL
        ) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
``` [2](#0-1) 

---

### Proof of Concept

1. `nlpPools[0].subaccount` holds borderline-healthy collateral (initial health just above zero).
2. Sequencer calls `forceRebalanceNlpPool` with `nlpPoolRebalanceX18[0]` set to a large negative value (withdrawing quote from pool 0).
3. `_applyNlpRebalance` executes `spotEngine.updateBalance(QUOTE_PRODUCT_ID, nlpPools[0].subaccount, nlpPoolRebalanceX18[0])`, pushing pool 0's health below zero.
4. The health-check loop begins at `i = 1` — `nlpPools[0]` is never checked.
5. Transaction succeeds. `nlpPools[0]` is now under-collateralized with no on-chain revert.
