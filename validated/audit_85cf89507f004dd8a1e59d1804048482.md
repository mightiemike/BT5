### Title
Difficulty Retarget for Fork Blocks Uses Mainchain Ancestor Timestamp Instead of Fork Ancestor Timestamp - (`contract/src/dogecoin.rs`)

### Summary

The `get_next_work_required` function in the Dogecoin (and Bitcoin/Litecoin) difficulty logic calls `blocks_getter.get_header_by_height(height_first)` to obtain the timestamp of the first block in the retarget interval. `get_header_by_height` is implemented to look up only the **mainchain** height-to-hash map. When the block being validated is on a **fork**, the correct reference block is the fork's own ancestor at that height, not the mainchain block at the same height. The code's own TODO comment acknowledges this uncertainty.

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes the expected difficulty for a new block: [1](#0-0) 

The call to `blocks_getter.get_header_by_height(height_first)` resolves through the `BlocksGetter` implementation in `lib.rs`: [2](#0-1) 

This always reads from `mainchain_height_to_header`, which maps only canonical-chain heights. When a fork block is being submitted, the block at `height_first` on the fork may differ from the mainchain block at the same height. The code itself flags this with a TODO: [3](#0-2) 

For Dogecoin post-height 145,000, `difficulty_adjustment_interval = 1`, meaning every block triggers a retarget with `blocks_to_go_back = 1`: [4](#0-3) 

So for any fork block at height `H+k` (k ≥ 3) where the fork diverged at height `H`, `height_first = H+k-2` falls inside the fork. The contract fetches the **mainchain** block at `H+k-2` instead of the fork block at `H+k-2`. These have different timestamps because the attacker controls the fork block's timestamp (within MTP and future-time bounds).

The same structural bug exists in `contract/src/bitcoin.rs` and `contract/src/litecoin.rs` at their respective retarget boundaries: [5](#0-4) [6](#0-5) 

The Zcash implementation does **not** share this bug because it traverses the chain using `get_prev_header` rather than `get_header_by_height`: [7](#0-6) 

### Impact Explanation

An attacker submitting a Dogecoin fork can craft fork block timestamps so that the time difference `prev_fork_block.time − mainchain_block_at_height_first.time` is larger than the correct difference `prev_fork_block.time − fork_block_at_height_first.time`. The Digishield retarget formula in `calculate_next_work_required` translates a larger observed timespan into a **higher target** (lower difficulty): [8](#0-7) 

The maximum upward adjustment is 50% (`max_timespan = retarget_timespan + retarget_timespan/2`). The contract will accept a fork block whose `bits` field encodes this artificially lowered difficulty, while the real Dogecoin network would reject it. Because `work_from_bits` is used to accumulate chainwork: [9](#0-8) 

…a fork built with inflated targets accumulates less chainwork per block, but the attacker can mine more blocks in the same wall-clock time. If the fork's total chainwork exceeds the mainchain's, `submit_block_header_inner` promotes it as the new canonical chain: [10](#0-9) 

This corrupts the canonical-chain pointer and all downstream SPV proofs (`verify_transaction_inclusion`, `verify_transaction_inclusion_v2`) that rely on `mainchain_header_to_height` for confirmation counting.

### Likelihood Explanation

The `submit_blocks` entry point is open to any NEAR caller (gated only by a trusted-relayer check that can be bypassed via `Role::UnrestrictedSubmitBlocks`): [11](#0-10) 

For Dogecoin mainnet (post-145,000), the bug activates on the **third** fork block submitted, requiring no special privileges. The attacker only needs to know the mainchain block timestamps (publicly available on-chain) to compute the exact `bits` value the contract will accept.

### Recommendation

Replace `get_header_by_height` with a chain-traversal loop that walks `get_prev_header` from the fork tip, mirroring the Zcash implementation. This ensures the timestamp used in the retarget calculation always belongs to the actual ancestor of the block being validated, regardless of whether it is on the mainchain or a fork.

### Proof of Concept

1. Mainchain has blocks at heights 0…N. Fork diverges at height N (common ancestor).
2. Attacker submits fork block at N+1 with timestamp `T_fork_N1 = mainchain_block_N1.time + 3600` (1 hour ahead, within the 2-hour future-time limit).
3. Attacker submits fork block at N+2 with any valid timestamp `T_fork_N2 > MTP(N+1)`.
4. Contract computes expected bits for fork block at N+3:
   - `height_first = N+1`
   - `first_block_time = mainchain_block_N1.time` ← **wrong; should be `T_fork_N1`**
   - `modulated_timespan = T_fork_N2 − mainchain_block_N1.time` (larger than correct value)
   - Result: `expected_bits` encodes a target up to 50% higher than the real Dogecoin network requires.
5. Attacker mines fork block at N+3 satisfying this inflated target (easier than real difficulty).
6. Contract accepts it; real Dogecoin nodes reject it.
7. Attacker repeats until fork chainwork exceeds mainchain chainwork, triggering `reorg_chain` and corrupting the canonical chain stored in the contract.
