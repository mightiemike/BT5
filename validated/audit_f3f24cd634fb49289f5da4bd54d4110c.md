### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling Merkle Internal-Node Forgery Without On-Chain Transaction Hash Validation — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR contract method despite being deprecated. It contains no on-chain mechanism to distinguish a real transaction hash from an internal Merkle tree node hash. The contract explicitly delegates that validation to "a higher level of verification" (off-chain), mirroring the M-03 pattern of relying on an external, bypassable mechanism instead of enforcing the invariant on-chain. Any unprivileged NEAR caller can supply an internal Merkle node as `tx_id` and receive a `true` proof result.

---

### Finding Description

`verify_transaction_inclusion` is annotated `#[deprecated(since = "0.5.0")]` but carries no access restriction — it is still decorated with `#[pause]` (not `#[private]`) and is therefore callable by any NEAR account. [1](#0-0) 

The function's own documentation acknowledges the missing on-chain mechanism:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification." [2](#0-1) 

The verification body does nothing more than recompute the Merkle root from the caller-supplied `tx_id`, `tx_index`, and `merkle_proof`, then compare it to the stored block header's `merkle_root`: [3](#0-2) 

`compute_root_from_merkle_proof` is a pure hash-chain computation with no structural check on whether the leaf is a transaction or an internal node: [4](#0-3) 

The partial mitigation — `verify_transaction_inclusion_v2` — adds a coinbase proof length check, but it does not remove or gate v1. v2 internally calls v1 via `#[allow(deprecated)]`: [5](#0-4) 

Because v1 is still reachable, any caller can bypass the coinbase-proof guard entirely by calling v1 directly.

---

### Impact Explanation

Any consumer contract (cross-chain bridge, token-minting gate, settlement contract) that calls `verify_transaction_inclusion` (v1) to authorize an action can be deceived. An attacker who knows the Merkle tree of any block already stored in the contract's `headers_pool` can:

1. Pick any internal Merkle node `N` at tree depth `d` and its corresponding position `p`.
2. Construct a valid `merkle_proof` of length `d` that walks from `N` up to the stored `merkle_root`.
3. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index = p`, and that proof.
4. The contract returns `true` — a forged proof result — because `compute_root_from_merkle_proof(N, p, proof) == merkle_root` is arithmetically true for a real internal node.

The corrupted value is the **proof result**: a `true` return for a transaction that was never included in any Bitcoin block.

---

### Likelihood Explanation

Likelihood is high. The entry point is fully public (no role, no stake, no privileged key required). Every block already accepted by the contract exposes its full Merkle tree structure through `get_block_hash_by_height` / `get_last_block_header`, giving an attacker all the data needed to pick a valid internal node and construct the forgery proof. No mining, no key compromise, and no social engineering is required.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or gate it with `#[private]` so it cannot be called externally. The `#[deprecated]` Rust attribute is a compiler hint only; it has no effect on NEAR's runtime dispatch. As long as the method exists in the compiled WASM with a public name, any NEAR account can invoke it. The on-chain validation that `tx_id` is a leaf-level transaction hash — not an internal node — must be enforced inside the contract, not delegated to callers.

---

### Proof of Concept

Given a block at height `H` already in the contract's mainchain with a known Merkle tree of depth `d ≥ 2`:

1. Let `N = internal_node_at_depth_1_position_0` (the hash of the left subtree root, one level below `merkle_root`).
2. Let `proof = [right_subtree_root_at_depth_1]` (length 1, sufficient to reconstruct `merkle_root`).
3. Call (Borsh-encoded):

```
verify_transaction_inclusion({
    tx_id:              N,
    tx_block_blockhash: <known block hash at height H>,
    tx_index:           0,
    merkle_proof:       [right_subtree_root],
    confirmations:      1,
})
```

4. `compute_root_from_merkle_proof(N, 0, [right_subtree_root])` computes `double_sha256(N || right_subtree_root)` which equals `merkle_root` by construction.
5. The function returns `true`. No Bitcoin transaction with hash `N` exists; the proof is forged. [6](#0-5) [4](#0-3)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
