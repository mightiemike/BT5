### Title
`finalize_epoch` Fallback Path Does Not Update `shard_layout` or `last_resharding` in `EpochInfoV5` When `proposals_to_epoch_info` Fails - (`File: chain/epoch-manager/src/lib.rs`)

---

### Summary

In `EpochManager::finalize_epoch()`, when `proposals_to_epoch_info()` returns `ThresholdError` or `NotEnoughValidators` (the "all validators unstaked" safety fallback), the recovery path clones `next_epoch_info` (epoch N+1) and only increments `epoch_height`. It does **not** write the freshly-computed `next_next_shard_layout` or the freshly-computed `last_resharding` into the resulting `EpochInfoV5` that is stored as epoch N+2. If a dynamic resharding was scheduled for epoch N+2, the stored epoch info carries the stale N+1 shard layout and the stale N+1 `last_resharding` value, silently dropping the split and corrupting the cooldown counter.

---

### Finding Description

`finalize_epoch()` computes two values before calling `proposals_to_epoch_info()`:

```rust
// chain/epoch-manager/src/lib.rs  lines 926-929
let has_same_shard_layout = next_next_shard_layout == next_shard_layout;
let last_resharding = (!has_same_shard_layout)
    .then(|| next_epoch_info.epoch_height() + 1)
    .or_else(|| next_epoch_info.last_resharding());
```

These are passed as arguments to `proposals_to_epoch_info()` and, on the happy path, end up in the freshly-constructed `EpochInfoV5` for epoch N+2. [1](#0-0) 

On the error path, however, the fallback is:

```rust
Err(EpochError::ThresholdError { .. }) => {
    let mut epoch_info = EpochInfo::clone(&next_epoch_info);  // clones N+1 info
    *epoch_info.epoch_height_mut() += 1;                      // only bumps height
    epoch_info                                                 // shard_layout & last_resharding are N+1 values
}
```

`next_epoch_info` is the **N+1** epoch info. Its `shard_layout` field is the N+1 layout; its `last_resharding` field is whatever was recorded for N+1. Neither `next_next_shard_layout` (the newly derived split layout for N+2) nor the freshly-computed `last_resharding` is written into the fallback object. [2](#0-1) 

The `EpochInfoV5` struct carries both fields: [3](#0-2) 

`get_shard_layout()` reads `EpochInfo::shard_layout()` first (V5 path) and only falls back to `EpochConfig` when the field is absent (V1â€“V4): [4](#0-3) 

`can_reshard()` reads `last_resharding()` from the **next** epoch's stored info: [5](#0-4) 

`EpochInfo::last_resharding()` returns `None` for V1â€“V4 and the stored field for V5: [6](#0-5) 

---

### Impact Explanation

**Shard-layout divergence (consensus-breaking).** When the fallback fires during an epoch that also scheduled a dynamic resharding, `get_shard_layout(epoch_N+2)` returns the N+1 layout instead of the derived `ShardLayoutV3`. Every shard-layout-dependent operation for epoch N+2 â€” chunk-producer assignment, trie-key routing, state-sync, resharding execution â€” uses the wrong layout. Nodes that independently recompute the layout from `block_info.shard_split()` will disagree with the stored value, producing a consensus split.

**Cooldown counter corruption.** `last_resharding` in the N+2 epoch info is not advanced to `next_epoch_info.epoch_height() + 1`. When `can_reshard()` is later called during epoch N+1 (to decide whether epoch N+3 may be split), it reads the stale N+1 `last_resharding` from the N+2 epoch info. This can allow a back-to-back resharding that the cooldown invariant is specifically designed to prevent. The documentation explicitly warns that back-to-back reshardings are unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk, triggering `InvalidChunkHeaderShardSplit`. [7](#0-6) 

---

### Likelihood Explanation

The trigger requires two conditions to coincide in the same epoch:

1. A dynamic resharding is scheduled (`next_next_shard_layout != next_shard_layout`).
2. `proposals_to_epoch_info()` returns `ThresholdError` or `NotEnoughValidators` â€” i.e., the total stake falls below the seat threshold or there are fewer validators than shards.

Condition 2 is the existing "all validators unstaked" safety path that has been in the codebase since before dynamic resharding. Condition 1 is new with `ProtocolFeature::DynamicResharding`. Their intersection is a new, previously non-existent invariant violation. On mainnet the probability of both occurring simultaneously is low but non-zero: a coordinated mass-unstake during an epoch that happens to be a resharding epoch would trigger it. The fallback path is reachable without any privileged role â€” it is driven purely by on-chain staking transactions.

---

### Recommendation

In both fallback arms of `finalize_epoch()`, after cloning `next_epoch_info` and bumping `epoch_height`, also apply the computed `next_next_shard_layout` and `last_resharding` to the fallback object. For `EpochInfo::V5`, this means setting `v5.shard_layout = next_next_shard_layout` and `v5.last_resharding = last_resharding`. A helper method (e.g., `EpochInfo::set_shard_layout_and_last_resharding`) should be added to keep the mutation encapsulated. Additionally, a test should be added that exercises the fallback path when `DynamicResharding` is enabled and a resharding is scheduled, asserting that the stored epoch info carries the correct `shard_layout` and `last_resharding`.

---

### Proof of Concept

1. Enable `ProtocolFeature::DynamicResharding` (protocol version â‰Ą 153 in nightly).
2. Configure a network with `min_epochs_between_resharding = 2` and a shard that exceeds `memory_usage_threshold`.
3. At the last block of epoch N, ensure `block_info.shard_split()` is `Some(shard_id, boundary_account)` so that `next_next_shard_layout != next_shard_layout`.
4. In the same epoch N, have all validators submit stake-zero proposals so that `proposals_to_epoch_info()` returns `ThresholdError`.
5. Observe that the `EpochInfo` stored for epoch N+2 (keyed by `EpochId(*last_block_hash)`) is a `V5` whose `shard_layout` equals the N+1 layout (not the derived split layout) and whose `last_resharding` equals the N+1 value.
6. Call `get_shard_layout(&epoch_N+2_id)` â€” it returns the stale N+1 layout.
7. Call `can_reshard(block_in_epoch_N+1, min_epochs_between_resharding)` â€” it reads the stale `last_resharding` from the N+2 epoch info and incorrectly permits a back-to-back resharding. [8](#0-7) [9](#0-8)

        &self,
        block_hash: &CryptoHash,
        min_epochs_between_resharding: NonZeroEpochHeight,
    ) -> Result<bool, EpochError> {
        let block_info = self.get_block_info(block_hash)?;
        let next_epoch_id = self.get_next_epoch_id_from_info(&block_info)?;
        let next_epoch_info = self.get_epoch_info(&next_epoch_id)?;
