### Title
`ValidResourceBounds` P2P Protobuf Deserializer Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash Preimage — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The P2P protobuf converter for `ValidResourceBounds` classifies any V3 transaction whose `l1_data_gas` and `l2_gas` are both zero as the legacy `L1Gas` variant. Because `get_tip_resource_bounds_hash` hashes a different number of resource-bound felts depending on the variant (`L1Gas` → 3 elements; `AllResources` → 4 elements), the hash recomputed after P2P deserialization diverges from the hash computed at submission time. Any component that re-derives the transaction hash from the deserialized `Transaction` object will obtain a wrong value.

---

### Finding Description

**Submission path (gateway → mempool → block)**

`RpcInvokeTransactionV3`, `RpcDeclareTransactionV3`, and `RpcDeployAccountTransactionV3` all carry `resource_bounds: AllResourceBounds` (not `ValidResourceBounds`). Their `TransactionHasher` implementations always wrap the field as `ValidResourceBounds::AllResources(...)` before calling `get_tip_resource_bounds_hash`. [1](#0-0) 

`get_tip_resource_bounds_hash` for `AllResources` appends the `l1_data_gas` packed felt, producing a 4-element Poseidon input: [2](#0-1) 

**P2P deserialization path**

The protobuf converter for the storage-level `Transaction` type reconstructs `ValidResourceBounds` with this logic:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← 3-element hash path
} else {
    ValidResourceBounds::AllResources(...)       // ← 4-element hash path
})
``` [3](#0-2) 

A V3 transaction with `l1_gas = {amount=N, price=P}`, `l2_gas = {0,0}`, `l1_data_gas = {0,0}` is valid at the gateway (non-zero `max_possible_fee` because `l1_gas` is non-zero): [4](#0-3) 

After P2P deserialization the same transaction is reconstructed as `ValidResourceBounds::L1Gas`, so `get_tip_resource_bounds_hash` omits the `l1_data_gas` packed felt and produces a **different hash**.

The `ValidResourceBounds` comment confirms the semantic distinction: [5](#0-4) 

---

### Impact Explanation

Any code path that re-derives a transaction hash from the deserialized `Transaction` object (e.g., `validate_transaction_hash`, block-hash computation, P2P block verification) will compute hash H2 ≠ H1 (the hash stored in the block). This can cause:

- **P2P block sync rejection**: a syncing node rejects a valid committed block because the recomputed transaction hash does not match the stored one.
- **Wrong receipt/state**: if the wrong hash is used downstream, the transaction is associated with an incorrect identifier, corrupting receipts, events, or storage lookups.

This falls under: *High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value* and *High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.*

---

### Likelihood Explanation

The trigger requires only a standard V3 transaction with non-zero `l1_gas` and zero `l2_gas`/`l1_data_gas`. This is a plausible user configuration (e.g., a user who sets only L1 gas bounds and leaves the others at zero). The gateway accepts it, it can be included in a block, and the divergence is deterministic once the block is synced via P2P.

---

### Recommendation

1. **Preserve the variant tag explicitly in protobuf.** Add a boolean or enum field (e.g., `resource_bounds_version`) to the protobuf message so the deserializer can distinguish `L1Gas` from `AllResources` without relying on value-level zero checks.

2. **Remove the zero-value heuristic.** Replace the `l1_data_gas.is_zero() && l2_gas.is_zero()` classification with an explicit field. The TODO comment already acknowledges this:
   ```rust
   // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
   ```
   Until that TODO is resolved, the heuristic creates a hash-domain collision between legacy `L1Gas` transactions and new `AllResources` transactions with zero data/L2 gas.

3. **Add a round-trip hash test** that submits a V3 transaction with `AllResources{l1_gas=nonzero, l2_gas=0, l1_data_gas=0}`, serializes it to protobuf, deserializes it, and asserts the recomputed hash equals the original.

---

### Proof of Concept

```
1. Construct a V3 invoke transaction:
     resource_bounds = AllResources {
         l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
         l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
         l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
     }

2. Compute hash H1 via InternalRpcInvokeTransactionV3::calculate_transaction_hash
   → get_tip_resource_bounds_hash called with AllResources
   → resource_felts = [tip, l1_gas_packed, l2_gas_packed(0), l1_data_gas_packed(0)]  (4 elements)
   → H1 = Poseidon(tip, l1_gas_packed, l2_gas_packed, l1_data_gas_packed)

3. Serialize the Transaction to protobuf (P2P block sync wire format).

4. Deserialize via TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
   → l1_data_gas = 0, l2_gas = 0
   → branch taken: ValidResourceBounds::L1Gas(l1_gas)

5. Recompute hash H2 via InvokeTransactionV3::calculate_transaction_hash
   → get_tip_resource_bounds_hash called with L1Gas
   → resource_felts = [tip, l1_gas_packed, l2_gas_packed(0)]  (3 elements — l1_data_gas omitted)
   → H2 = Poseidon(tip, l1_gas_packed, l2_gas_packed)

6. Assert H1 == H2  →  FAILS.
   The P2P-deserialized transaction carries a different hash than the one committed to the block.
```

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
