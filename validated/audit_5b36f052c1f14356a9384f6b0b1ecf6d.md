### Title
AuxPoW Flag Not Enforced When `aux_data` Is Absent, Allowing Chain-ID Bypass in Dogecoin Block Submission — (`contract/src/dogecoin.rs`)

### Summary

The Dogecoin `submit_block_header` function branches on whether the caller supplies `aux_data`. When `aux_data` is `Some`, the full `check_aux` path runs and enforces the AuxPoW flag, Dogecoin chain-ID, coinbase merkle proof, and parent-block PoW. When `aux_data` is `None`, only a bare scrypt-hash PoW check is performed — there is **no assertion that the block's version field does not carry the AuxPoW flag** (`version & 0x100 == 0`). An attacker who mines a Dogecoin block header with the AuxPoW bit set and a valid scrypt hash can submit it without `aux_data`, causing the contract to accept a block that bypasses every AuxPoW-specific invariant.

---

### Finding Description

`submit_block_header` in `dogecoin.rs` dispatches on `aux_data`:

```rust
if let Some(ref aux_data) = aux_data {
    self.check_aux(&block_header, aux_data);   // ✅ full AuxPoW validation
} else {
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        format!("block should have correct pow")
    );
    // ❌ no check: block_header.version & BLOCK_VERSION_AUXPOW == 0
}
``` [1](#0-0) 

`check_aux` enforces, among other things:

```rust
const BLOCK_VERSION_AUXPOW: i32 = 0x100;
require!(
    block_header.version & BLOCK_VERSION_AUXPOW != 0,
    "Aux POW block does not have AuxPoW flag set in version"
);
require!(
    chain_id == block_header.get_chain_id(),   // must be 0x0062 for Dogecoin
    ...
);
require!(
    chain_id != aux_data.parent_block.get_chain_id(),
    "Aux POW parent has our chain ID"
);
``` [2](#0-1) 

None of these checks run when `aux_data` is `None`. The chain-ID is extracted from the version field via `version >> 16`: [3](#0-2) 

A block with `version = 0x00000100` (AuxPoW flag set, chain-ID = 0) passes the `else` branch if its scrypt hash satisfies the target, even though Dogecoin's required chain-ID is `0x0062`.

The inconsistency is structurally identical to the external report:

| Path | Validation |
|---|---|
| `aux_data = Some` | Full AuxPoW: flag, chain-ID, coinbase merkle, parent PoW |
| `aux_data = None` | **Only scrypt hash** — AuxPoW flag and chain-ID silently skipped |

---

### Impact Explanation

A block accepted through the `aux_data = None` path with the AuxPoW flag set is stored in `headers_pool` and, if it accumulates sufficient chainwork, promoted to `mainchain_tip_blockhash` via `submit_block_header_inner`. [4](#0-3) 

Once on the canonical chain, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will treat any merkle proof against that block as valid: [5](#0-4) 

This means a block that is invalid under the real Dogecoin consensus (wrong chain-ID, no merged-mining commitment) can anchor false transaction-inclusion proofs accepted by downstream NEAR contracts that rely on this light client.

---

### Likelihood Explanation

The attacker must mine a Dogecoin block header with the AuxPoW bit set (`version & 0x100 != 0`) whose scrypt hash satisfies the current Dogecoin difficulty target. This requires real hashrate proportional to the current network difficulty. Likelihood is **low** but non-zero: the attack is computationally expensive, not theoretically impossible, and the cost scales with current Dogecoin difficulty rather than being prohibitive. A well-resourced attacker (e.g., one with access to significant scrypt mining capacity) could execute it.

The entry point `submit_blocks` is gated by `#[trusted_relayer]`: [6](#0-5) 

If the `omni_utils` staking mechanism is permissionless (any NEAR account can stake to become a relayer), the attacker controls the full submission path without any privileged role. The `Role::UnrestrictedSubmitBlocks` bypass role would also suffice.

---

### Recommendation

Add an explicit guard in the `else` branch to reject any block whose version carries the AuxPoW flag when no `aux_data` is provided:

```rust
} else {
    const BLOCK_VERSION_AUXPOW: i32 = 0x100;
    require!(
        block_header.version & BLOCK_VERSION_AUXPOW == 0,
        "Block has AuxPoW flag set but no AuxPoW data was provided"
    );
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        format!("block should have correct pow")
    );
}
```

This mirrors the symmetric check already present inside `check_aux` and closes the inconsistency.

---

### Proof of Concept

1. Construct a `Header` with `version = 0x00000100` (AuxPoW flag set, chain-ID = 0, not Dogecoin's 0x0062), valid `prev_block_hash` pointing to a known chain tip, and any `merkle_root`.
2. Mine the scrypt hash of this 80-byte header until `scrypt(header) <= target_from_bits(bits)`.
3. Call `submit_blocks([(header, None)])` as a trusted relayer (or staked relayer if permissionless).
4. The contract executes the `else` branch, passes the scrypt check, and stores the block in `headers_pool` and `mainchain_height_to_header` without ever verifying the chain-ID.
5. Call `verify_transaction_inclusion_v2` with a crafted merkle proof against this block's hash; the function returns `true` for a transaction that was never included in any real Dogecoin block. [7](#0-6)

    pub(crate) fn submit_block_header(
        &mut self,
        header: (Header, Option<AuxData>),
        skip_pow_verification: bool,
    ) {
        let (block_header, aux_data) = header;

        let prev_block_header = self.get_prev_header(&block_header);
        let current_block_hash = block_header.block_hash();

        if !skip_pow_verification {
            self.check_target(&block_header, &prev_block_header);

            if let Some(ref aux_data) = aux_data {
                self.check_aux(&block_header, aux_data);
            } else {
                let pow_hash = block_header.block_hash_pow();
                // Check if the block hash is less than or equal to the target
                require!(
                    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
                    format!("block should have correct pow")
                );
            }
        }

        let (current_block_computed_chain_work, overflow) = prev_block_header
            .chain_work
            .overflowing_add(work_from_bits(block_header.bits));
        require!(!overflow, "Addition of U256 values overflowed");

        let current_header = ExtendedHeader {
            block_header: block_header.clone().into_light(),
            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        self.submit_block_header_inner(current_header, &prev_block_header);
    }

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    fn submit_block_header_inner(
        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }
