### Title
Unchecked Return Value of `token.approve()` in `DirectDepositV1.creditDeposit()` Enables Silent Deposit Authorization Failure — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

In `DirectDepositV1.creditDeposit()`, the contract calls `token.approve(address(endpoint), balance)` without checking the returned `bool`. For ERC20 tokens that signal failure by returning `false` instead of reverting, the approval silently fails and execution proceeds unconditionally to `endpoint.depositCollateralWithReferral()`. This mirrors the exact vulnerability class in the report: an external call that does not revert on failure, whose return value is ignored, allowing the contract to proceed as if the operation succeeded.

---

### Finding Description

`DirectDepositV1.creditDeposit()` is an unrestricted external function — any caller may invoke it. Its purpose is to sweep any token balance held by the DDA contract into the protocol by approving the Endpoint and then calling `depositCollateralWithReferral`.

The local `IIERC20Base` interface declares `approve` as returning `bool`:

```solidity
function approve(address spender, uint256 amount) external returns (bool);
```

At line 92, the return value is silently discarded:

```solidity
token.approve(address(endpoint), balance);          // return value ignored
endpoint.depositCollateralWithReferral(             // proceeds regardless
    subaccount,
    productId,
    uint128(balance),
    "-1"
);
``` [1](#0-0) 

If `approve()` returns `false` (non-reverting ERC20 failure), the allowance is never set, yet `depositCollateralWithReferral` is called immediately after. The Endpoint will then attempt a `transferFrom` against the DDA contract with zero allowance. If the Endpoint's internal `transferFrom` path does not itself revert (i.e., uses a raw unchecked call), the Clearinghouse will credit the DDA's subaccount with a balance that was never actually transferred — a direct accounting desynchronization between on-chain token custody and the protocol's internal ledger.

The contrast with the rest of the codebase is notable: `ERC20Helper.safeTransferFrom` and `ERC20Helper.safeTransfer` — used throughout `BaseWithdrawPool` and `Clearinghouse` — explicitly check both the call success flag and the decoded boolean return value:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [2](#0-1) 

`DirectDepositV1` defines its own local `safeTransfer` helper that also checks the return value for `transfer` calls, but inconsistently omits the same check for `approve`. [3](#0-2) 

---

### Impact Explanation

**Worst case (Endpoint uses raw unchecked `transferFrom`):** The Clearinghouse credits the DDA's fixed `subaccount` with `uint128(balance)` worth of a spot product without the tokens ever leaving the DDA contract. The protocol's internal ledger diverges from actual token custody: the DDA subaccount holds inflated collateral, which can be used to open positions, withdraw quote, or participate in NLP minting — all backed by tokens that were never deposited. This is a direct solvency/accounting corruption.

**Conservative case (Endpoint uses `safeTransferFrom`):** The `depositCollateralWithReferral` call reverts due to insufficient allowance. The DDA's tokens remain stranded in the contract and cannot be credited to the protocol until the approval is retried. Any caller who triggers `creditDeposit()` during this window causes a failed deposit cycle with no on-chain indication of the root cause.

---

### Likelihood Explanation

- `creditDeposit()` has no access control — any unprivileged external caller can trigger it at any time.
- The bug activates for any ERC20 token registered as a spot product that returns `false` on a failed `approve` rather than reverting. Several widely-used tokens (e.g., USDT on certain deployments, BNB, and others) exhibit this behavior.
- The DDA contract is designed to be deployed per-subaccount and to automatically sweep incoming token balances, making it a routine operational target. [4](#0-3) 

---

### Recommendation

Replace the bare `token.approve()` call with a checked variant that reverts on failure, consistent with the pattern already used in `ERC20Helper`:

```solidity
(bool success, bytes memory data) = address(token).call(
    abi.encodeWithSelector(IIERC20Base.approve.selector, address(endpoint), balance)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    "Approve failed"
);
```

Alternatively, extract a `safeApprove` helper in `DirectDepositV1` mirroring the existing `safeTransfer` helper at lines 69–81.

---

### Proof of Concept

1. A non-standard ERC20 token (one that returns `false` on a failed `approve` instead of reverting) is registered as a spot product in `SpotEngine`.
2. Some balance of this token is sent directly to the DDA contract (e.g., via a user transfer or airdrop).
3. An unprivileged caller invokes `DirectDepositV1.creditDeposit()`.
4. `token.approve(address(endpoint), balance)` is called. The token returns `false` — the allowance is **not** set. No revert occurs; the return value is discarded.
5. `endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1")` is called with zero allowance.
6. If the Endpoint's internal transfer path uses a raw `transferFrom` without checking the return value, the token transfer silently fails while the Clearinghouse proceeds to credit `subaccount` with `amountRealized` via `spotEngine.updateBalance()`.
7. The DDA's subaccount now holds inflated collateral in the Clearinghouse's ledger, backed by tokens that remain in the DDA contract — a direct accounting desynchronization exploitable for undercollateralized trading or withdrawal. [5](#0-4) [4](#0-3)
