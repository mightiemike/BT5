The vulnerability is real and concrete. Let me trace through the math precisely.

**Tracing `compute_root_from_merkle_proof` for an odd-length tree (N=5 transactions, last tx = `tx4` at index 4):**

`merkle_proof_calculator` for position 4 produces proof = `[tx4, H(tx4,tx4), H(H(tx0,tx1),H(tx2,tx3))]`

**Call A: `compute_root_from_merkle_proof(tx4, 4, proof)`** (legitimate)
- pos=4 (even): `H(tx4, proof[0])` = `H(tx4, tx4)`; pos→2
- pos=2 (even): `H(H(tx4,tx4), proof[1])` = `H(H(tx4,tx4), H(tx4,tx4))`; pos→1
- pos=1 (odd): `H(proof[2], ...)` = merkle root ✓

**Call B: `compute_root_from_merkle_proof(tx4, 5, proof)`** (phantom index, same proof)
- pos=5 (odd): `H(proof[0], tx4)` = `H(tx4, tx4)`; pos→2
- pos=2 (even): `H(H(tx4,tx4), proof[1])` = `H(H(tx4,tx4), H(tx4,tx4))`; pos→1
- pos=1 (odd): `H(proof[2], ...)` = merkle root ✓

Both produce the same root. The vulnerability is confirmed.

### Title
Non-Injective Merkle Proof Verification Allows Same Transaction to Verify at Two Distinct Indices — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

### Summary
`compute_root_from_merkle_proof` is not injective over `(tx_id, tx_index, merkle_proof)` tuples for odd-length transaction trees. The last transaction in any odd-length block verifies successfully at both index `N-1` (its real index) and index `N` (a phantom, non-existent index) using the identical proof. Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` expose this to any external caller.

### Finding Description
In Bitcoin's Merkle tree construction, when the number of transactions is odd, the last transaction is duplicated: `H(tx_last, tx_last)` is used as the right child of the last pair at that level.

`merkle_proof_calculator` encodes this by placing `tx_last` itself as `proof[0]` for the last transaction. [1](#0-0) 

When `compute_root_from_merkle_proof` processes this proof:
- At index `N-1` (even): step 1 computes `H(current_hash, proof[0])` = `H(tx_last, tx_last)`
- At index `N` (odd): step 1 computes `H(proof[0], current_hash)` = `H(tx_last, tx_last)`

Both produce the identical intermediate hash, and all subsequent steps are identical, yielding the same root. [2](#0-1) 

The contract's `verify_transaction_inclusion` accepts any caller-supplied `tx_index` with no upper-bound check against the actual number of transactions in the block: [3](#0-2) 

`verify_transaction_inclusion_v2` inherits the same flaw since it delegates to `verify_transaction_inclusion` after only validating the coinbase proof: [4](#0-3) 

### Impact Explanation
Any bridge or settlement contract that:
1. Calls `verify_transaction_inclusion` (or `_v2`) to confirm a BTC transaction, and
2. Tracks processed transactions by `(tx_id, tx_index)` pairs

can be double-claimed. An attacker who legitimately settles `(tx_last, N-1)` can immediately re-submit `(tx_last, N)` with the identical proof and receive a second settlement for the same on-chain BTC transaction. This enables double-spend / duplicate settlement.

### Likelihood Explanation
- The precondition (odd-length block) is common — roughly half of all Bitcoin blocks have an odd number of transactions.
- No privileged role is required. `verify_transaction_inclusion` is a public view/call method gated only by `#[pause]`.
- The attacker only needs to know the legitimate proof for the last transaction, which is derivable from public Bitcoin block data.
- The exploit is deterministic and requires no brute force.

### Recommendation
1. **Reject out-of-bounds indices**: In `verify_transaction_inclusion`, require that `tx_index < number_of_transactions`. The block header does not store the tx count, so this requires passing it as an additional argument and committing to it (or deriving it from the proof length, which is insufficient alone).
2. **Normalize the proof**: In `merkle_proof_calculator`, when the last transaction is duplicated, store a sentinel or a different sibling value so that the proof for index `N-1` is structurally distinct from any proof for a phantom index `N`.
3. **Deprecate `verify_transaction_inclusion`** more aggressively — it is already marked deprecated but remains callable. Consider adding a hard revert for the phantom-index case.
