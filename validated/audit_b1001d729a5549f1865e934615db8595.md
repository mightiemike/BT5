Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Real User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender` — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, the router is that direct caller, so the extension evaluates `allowedSwapper[pool][routerAddress]` rather than `allowedSwapper[pool][realUser]`. A pool admin who allowlists the router to enable router-based swaps inadvertently grants every user on the internet the ability to bypass the individual allowlist.

## Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- direct caller of pool.swap(), not the real end-user
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards that value verbatim to the extension via `abi.encodeCall`. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly without forwarding the original `msg.sender`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData   // real user identity never encoded here
  );
```

The router stores the real user only in transient callback context for payment settlement — it is never passed to the pool as `sender`. The pool therefore passes the router's address as `sender` to the extension.

The integration test confirms the design only covers direct-caller scenarios: `callers[0]` is a `TestCaller` that calls the pool directly, and `address(callers[0])` is what gets allowlisted. No test covers the router path.

```solidity
// metric-periphery/test/extensions/FullMetricExtension.t.sol L68-74
swapExtension.setAllowedToSwap(address(pool), address(callers[0]), true);
_swap(0, users[0], false, int128(1000), type(uint128).max);
// callers[0] calls pool.swap() directly — router path never tested
```

## Impact Explanation

Two mutually exclusive failure modes exist for any pool combining `SwapAllowlistExtension` with `MetricOmmSimpleRouter`:

| Admin choice | Consequence |
|---|---|
| Allowlist the router | Every user bypasses the individual allowlist via the public router |
| Do not allowlist the router | All router-based swaps revert; the primary user-facing swap path is permanently broken |

In the bypass scenario, the allowlist — the sole access-control mechanism for curated pools — is completely defeated. Any non-allowlisted address (e.g., non-KYC'd, non-whitelisted) can execute swaps against a pool explicitly configured to restrict trading to a specific set of actors. This constitutes broken core pool functionality and an admin-boundary break reachable by any unprivileged trader.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical, documented swap entry point for end users. Any pool deploying `SwapAllowlistExtension` and expecting users to interact via the router will face this issue. No special privileges, flash loans, or unusual token behavior are required — a standard `exactInputSingle` call suffices. The bypass is reachable on every block once the router is allowlisted.

## Recommendation

The extension must resolve the real user identity rather than accepting the `sender` argument blindly. The preferred fix is router-aware identity forwarding: have `MetricOmmSimpleRouter` encode the real user address in `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and verify it — combined with a check that `msg.sender` (the pool) is a known factory-deployed pool so the data cannot be forged by a direct pool caller. Alternatively, modify the pool's `swap` signature to carry the original `msg.sender` from the router through to the extension.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true          // alice is the intended gated user
  allowedSwapper[pool][router] = true         // admin must set this for router swaps to work

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, tokenIn: ..., ...})

  router calls:
    pool.swap(recipient=bob, ...)             // router is msg.sender of pool.swap

  pool calls _beforeSwap(msg.sender=router, ...):
    extension.beforeSwap(sender=router, ...)  // sender = router, not bob

  extension checks:
    allowedSwapper[pool][router] == true      // PASSES — bob bypasses the allowlist

Result:
  bob executes a swap on a pool that was supposed to block him.
