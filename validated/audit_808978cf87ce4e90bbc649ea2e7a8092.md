Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Originating User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the address that called `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so `sender` is always the router address — never the originating user. Any pool admin who allowlists the router to support router-mediated swaps for legitimate users simultaneously opens the pool to every user on the network.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly: [4](#0-3) 

At that point, `msg.sender` inside `pool.swap` is the **router**, so `sender` forwarded to the extension is the router address. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The originating user is never evaluated.

For multi-hop `exactInput`, intermediate hops explicitly use `address(this)` (the router) as payer context, compounding the issue — the extension always sees the router regardless of the originating user: [5](#0-4) 

This creates an irreconcilable dilemma:

| Admin intent | Admin action | Actual result |
|---|---|---|
| Allow specific users, support router | Allowlist users + router | Router allowlisted → **any user bypasses via router** |
| Allow specific users, no router | Allowlist users only | Allowlisted users cannot use the router at all |
| Allow only router users | Allowlist router | **Any user bypasses via router** |

There are no existing guards in `SwapAllowlistExtension` that inspect `extensionData` for an originating user identity, nor any trusted-router registry that would trigger a secondary check.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties, institutional LPs, or protocol-controlled bots) can be bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. The bypassing user executes swaps at oracle-anchored prices against LP capital deposited under the assumption that only allowlisted counterparties would trade. This constitutes a direct loss of LP principal through unauthorized price-taking and fee extraction — a broken core allowlist invariant with direct fund impact, meeting the "Allowlist path" audit pivot: allowlist checks must cover the exact actor/action intended and cannot be bypassed through the router.

## Likelihood Explanation

The router is the primary user-facing swap interface. Any pool operator who deploys `SwapAllowlistExtension` and also wants allowlisted users to benefit from router features (slippage protection, multi-hop, deadline checks) must allowlist the router, which immediately opens the pool to all users. The bypass requires no special privileges, no flash loans, and no unusual token behavior — a single `exactInputSingle` call suffices. It is repeatable indefinitely.

## Recommendation

The extension must gate the **originating user**, not the immediate caller of `pool.swap`. Two sound approaches:

1. **Pass the real initiator through `extensionData`**: Have the router encode `msg.sender` into `extensionData` before forwarding to the pool, and have `SwapAllowlistExtension` decode and verify it. The extension must also verify that the pool's `sender` is a trusted router (to prevent spoofing by arbitrary callers encoding a fake user).

2. **Trusted-router registry in the extension**: Maintain a registry of trusted routers in the extension; when `sender` is a trusted router, require the real user identity to be supplied in `extensionData` and verified there. When `sender` is not a trusted router, fall back to checking `allowedSwapper[pool][sender]` directly.

The current design where `sender` is always the immediate `pool.swap` caller cannot correctly gate individual users when any public intermediary contract is involved.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup: pool with SwapAllowlistExtension configured.
// Admin allowlists alice and the router (to let alice use the router).
// Bob (not allowlisted) bypasses the allowlist via the router.

function test_swapAllowlistBypassViaRouter() public {
    // Admin allowlists alice and the router
    swapExtension.setAllowedToSwap(address(pool), alice, true);
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // Bob is NOT allowlisted
    assertFalse(swapExtension.isAllowedToSwap(address(pool), bob));

    // Bob calls the router — router becomes msg.sender to pool.swap()
    // Extension checks allowedSwapper[pool][router] == true → passes
    vm.prank(bob);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        recipient: bob,
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));
    // Bob's swap succeeds despite not being on the allowlist.
}
```

The extension's check at `allowedSwapper[pool][router]` returns `true` because the router is allowlisted, regardless of who called the router. The originating user `bob` is never evaluated.
