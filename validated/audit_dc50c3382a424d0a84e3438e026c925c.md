### Title
Unauthenticated P2P Peer Can Store Arbitrary `block_hash` in Headers Table via Unverified `SignedBlockHeader` — (`crates/apollo_p2p_sync/src/client/header.rs`)

### Summary

The P2P sync client accepts a `block_hash` field from any network peer's `SignedBlockHeader` protobuf message and writes it directly to storage without (a) recomputing the hash from the header fields, or (b) verifying the consensus signature over it. A malicious peer can supply `block_hash = Felt::ZERO` (or any arbitrary value), and the syncing node will store it verbatim, poisoning both the `headers` table and the `block_hash_to_number` reverse-lookup table.

### Finding Description

**Step 1 — Protobuf deserialization accepts `block_hash` verbatim.**

`TryFrom<protobuf::SignedBlockHeader> for SignedBlockHeader` in `crates/apollo_protobuf/src/converters/header.rs` decodes `block_hash` directly from the wire message and places it into `BlockHeader::block_hash` with no recomputation: [1](#0-0) 

**Step 2 — `parse_data_for_block` performs no hash or signature verification.**

`HeaderStreamBuilder::parse_data_for_block` in `crates/apollo_p2p_sync/src/client/header.rs` only checks that the block number matches and that exactly one signature is present. There is no call to `calculate_block_hash`, no call to `verify_block_signature`, and the existing TODO comment acknowledges that even `parent_hash` chain continuity is not yet checked: [2](#0-1) 

**Step 3 — `write_to_storage` passes the peer-supplied header directly to `append_header`.** [3](#0-2) 

**Step 4 — `append_header` stores the peer-supplied `block_hash` and inserts it into `block_hash_to_number`.**

`append_header` in `crates/apollo_storage/src/header.rs` copies `block_header.block_hash` into `StorageBlockHeader` and then calls `update_hash_mapping`, which inserts the attacker-controlled value as a key in the `block_hash_to_number` table: [4](#0-3) [5](#0-4) 

**The consensus signature is never verified in this path.** `verify_block_signature` (defined in `crates/starknet_api/src/block.rs`) signs `poseidon_hash(block_hash, state_diff_commitment)` under the sequencer's public key, but it is never called anywhere in `crates/apollo_p2p_sync`: [6](#0-5) 

### Impact Explanation

A malicious peer sends a `SignedBlockHeader` with `block_hash = Felt::ZERO` and any syntactically valid (but cryptographically unverified) signature. The syncing node:

1. Stores `block_hash = Felt::ZERO` in the `headers` table for the targeted block number.
2. Inserts `Felt::ZERO → block_number` into `block_hash_to_number`, so any RPC or P2P query by hash `Felt::ZERO` resolves to that block.
3. When acting as a P2P sync server, re-serves the corrupted header to downstream peers via `FetchBlockData for SignedBlockHeader`, propagating the wrong hash across the network. [7](#0-6) 

RPC endpoints that look up blocks by hash will return the wrong block, and the `block_hash_to_number` table is permanently poisoned for that hash value. This falls within the **High** impact scope: *RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

Note: the blockifier's `get_block_hash` syscall reads from `block_hash_contract_address` contract storage (populated by the OS during execution), not from the headers table, so direct execution-layer impact is not demonstrated. The impact is confined to the headers table, the reverse-lookup index, and downstream RPC/P2P consumers of those values.

### Likelihood Explanation

Any peer that can establish a P2P sync connection to the node can trigger this. No operator or sequencer privileges are required. The only prerequisite is that the peer sends a well-formed protobuf message with the correct block number and exactly one signature entry (both trivially satisfied).

### Recommendation

1. **Recompute and verify `block_hash`** inside `parse_data_for_block` (or inside `write_to_storage`) by calling `calculate_block_hash` over the received header fields and comparing against the peer-supplied value. Reject with `BadPeerError` on mismatch.
2. **Verify the consensus signature** by calling `verify_block_signature` with the known sequencer public key before accepting any `SignedBlockHeader` from the network.
3. Resolve the existing TODO at line 102 of `crates/apollo_p2p_sync/src/client/header.rs` to also enforce `parent_hash` chain continuity.

### Proof of Concept

```rust
// In a Rust integration test:
let malicious_header = SignedBlockHeader {
    block_header: BlockHeader {
        block_hash: BlockHash(Felt::ZERO),          // attacker-controlled
        block_header_without_hash: BlockHeaderWithoutHash {
            block_number: BlockNumber(0),
            ..Default::default()
        },
        state_diff_length: Some(0),
        ..Default::default()
    },
    signatures: vec![BlockSignature::default()],    // unverified
};
// Feed through parse_data_for_block → write_to_storage → append_header
// Then assert:
let stored = storage_reader.begin_ro_txn().unwrap()
    .get_block_header(BlockNumber(0)).unwrap().unwrap();
assert_eq!(stored.block_hash, BlockHash(Felt::ZERO)); // passes — wrong hash stored

let lookup = storage_reader.begin_ro_txn().unwrap()
    .get_block_number_by_hash(&BlockHash(Felt::ZERO)).unwrap();
assert_eq!(lookup, Some(BlockNumber(0)));             // passes — table poisoned
```
