### Title
Isolated Subaccount Can Be Closed With Remaining Funds After Re-org Reverts Withdrawal — (`core/contracts/EndpointTx.sol`)

### Summary

A user with an isolated subaccount who follows the correct two-step sequence — (1) submit `WithdrawCollateral`, then (2) submit `CloseIsolatedSubaccount` — can permanently lose their funds if a re-org causes the withdrawal to revert while the close still executes. The `CloseIsolatedSubaccount` path in `processTransactionImpl` calls `tryCloseIsolatedSubaccount` with no balance-zero guard at the dispatch layer, and the codebase itself contains a comment explicitly acknowledging that re-org attacks are being ignored.

---

### Finding Description

In `EndpointTx.sol`, the `processTransactionImpl` function handles the `CloseIsolatedSubaccount` transaction type by directly forwarding to `IOffchainExchange.tryCloseIsolatedSubaccount`: [1](#0-0) 

No balance check is performed at the `processTransactionImpl` dispatch layer before the close is executed. Compare this to the liquidation-triggered finalization path in `ClearinghouseLiq._finalizeSubaccount`, which explicitly iterates all spot and perp product IDs and requires every balance to be zero before proceeding: [2](#0-1) 

The liquidation path is guarded; the direct user-initiated close path is not.

Separately, the slow mode queue implementation in `submitSlowModeTransactionImpl` contains a developer comment that explicitly acknowledges re-org risk and states it is being ignored: [3](#0-2) 

The slow mode queue accepts `CloseIsolatedSubaccount` transactions from users (only `DepositCollateral` is explicitly blocked): [4](#0-3) 

This means a user can enqueue both a `WithdrawCollateral` and a `CloseIsolatedSubaccount` in the slow mode queue. If a re-org at the L1 level (Ink Chain is an L2; Arbitrum's own documentation confirms L1 re-orgs propagate to L2) causes the withdrawal transaction to be re-ordered and reverted — e.g., because a health check fails after the re-ordering — while the close transaction still executes, the user's collateral balance remains in the contract with no reachable withdrawal path.

---

### Impact Explanation

The isolated subaccount's collateral balance becomes permanently inaccessible. The `withdrawCollateral` function in `Clearinghouse.sol` requires the subaccount to pass a health check after the withdrawal: [5](#0-4) 

Once the subaccount is closed, there is no standard re-entry path for the user to recover the remaining balance. The funds are locked in the contract. This is a direct asset-loss impact for the affected user.

---

### Likelihood Explanation

Ink Chain is an EVM-compatible L2. As confirmed by Arbitrum's own glossary (referenced in the M-04 judge ruling), L1 re-orgs propagate to L2 state. While re-orgs are infrequent, the two-step pattern (withdraw then close) is the natural and expected user flow for exiting an isolated subaccount. The developer comment at `EndpointTx.sol:382–383` confirms the protocol team is aware of re-org risk in the slow mode queue and has not mitigated it. Any user following the correct exit sequence is exposed.

---

### Recommendation

Add a balance-zero guard inside `processTransactionImpl` before calling `tryCloseIsolatedSubaccount` for the `CloseIsolatedSubaccount` transaction type, mirroring the checks already present in `ClearinghouseLiq._finalizeSubaccount`. Specifically, iterate all spot and perp product IDs for the target subaccount and revert if any non-zero balance remains. This ensures the close cannot succeed unless the subaccount is fully empty, regardless of transaction ordering.

---

### Proof of Concept

1. Alice holds an isolated subaccount with 1000 USDC collateral.
2. Alice submits `WithdrawCollateral` (1000 USDC) to the slow mode queue.
3. Alice submits `CloseIsolatedSubaccount` to the slow mode queue immediately after.
4. A re-org occurs at the L1. The sequencer's batch that included the withdrawal execution is re-ordered; the withdrawal reverts because Alice's health check fails in the new ordering (e.g., a concurrent price update or another sequencer transaction changed her health).
5. The `CloseIsolatedSubaccount` transaction executes successfully via `processTransactionImpl` → `tryCloseIsolatedSubaccount`, with no balance check at the dispatch layer.
6. Alice's 1000 USDC remains in the contract under a closed subaccount with no recovery path. [1](#0-0) [6](#0-5)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        emit ModifyCollateral(amountRealized, sender, productId);
