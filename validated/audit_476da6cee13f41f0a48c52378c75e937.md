### Title
Native ETH Received Without Triggering `receive()` Is Not Deposited by `creditDeposit()` — (`core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` only sweeps ERC20 token balances into the protocol. It never checks or wraps the contract's native ETH balance. Meanwhile, `ContractOwner.isDirectDepositV1Ready()` explicitly adds `recipient.balance` (native ETH) to the readiness check for the `wrappedNative` product. This creates a concrete discrepancy: the off-chain readiness signal fires on native ETH, but the on-chain deposit function silently skips it. Native ETH that arrives without triggering `receive()` — most concretely via `selfdestruct` — is permanently undepositable and stuck.

---

### Finding Description

`DirectDepositV1` is a per-subaccount deposit address. Its `receive()` function wraps incoming ETH to `wrappedNative`: [1](#0-0) 

This works for normal ETH transfers. However, per the Solidity specification, a contract can receive ETH without `receive()` being called — specifically via `selfdestruct` from another contract. In that case, `address(this).balance` increases silently.

`creditDeposit()` iterates over all spot product IDs and deposits ERC20 balances: [2](#0-1) 

It calls `token.balanceOf(address(this))` for each product — it never reads `address(this).balance`. Native ETH that arrived without triggering `receive()` is invisible to this function and is never wrapped or deposited.

The critical evidence that the protocol *expects* native ETH to be depositable is in `ContractOwner.isDirectDepositV1Ready()`: [3](#0-2) 

When the product token is `wrappedNative`, the function adds `recipient.balance` (raw ETH) to the balance used for the readiness check. This means the off-chain infrastructure will observe `isDirectDepositV1Ready() == true` when native ETH is present, trigger a call to `creditDepositV1()`, which calls `creditDeposit()` — but `creditDeposit()` will not deposit the native ETH. The subaccount is not credited, and the ETH remains stuck.

The constructor handles a similar scenario (pre-existing ETH at deployment time) with a best-effort wrap: [4](#0-3) 

But no equivalent recovery path exists in `creditDeposit()` for ETH that arrives post-deployment without triggering `receive()`.

The only escape hatch is `withdrawNative()`, which is `onlyOwner`: [5](#0-4) 

This can rescue the ETH to the owner, but it does not credit the subaccount — the user's deposit is lost from the protocol's perspective.

---

### Impact Explanation

Native ETH that arrives at a `DirectDepositV1` address via `selfdestruct` (or any other mechanism that bypasses `receive()`) is permanently undepositable into the protocol for the associated subaccount. The subaccount is not credited. The `isDirectDepositV1Ready()` readiness check will falsely signal that a deposit is ready, causing the off-chain system to call `creditDepositV1()` with no effect on the stuck ETH. The owner can rescue the ETH via `withdrawNative()`, but the user's intended deposit is lost.

---

### Likelihood Explanation

`selfdestruct` is a standard EVM primitive. Any contract — including one controlled by a user or a third party — can force ETH into a `DirectDepositV1` address by self-destructing with it as the target. This requires no special privilege. The discrepancy between `isDirectDepositV1Ready()` (which counts native ETH) and `creditDeposit()` (which does not) confirms this is an unhandled case in the protocol's own deposit flow.

---

### Recommendation

Add native ETH wrapping logic inside `creditDeposit()`, mirroring the constructor's handling:

```solidity
function creditDeposit() external {
    // Wrap any native ETH balance first
    uint256 ethBalance = address(this).balance;
    if (ethBalance != 0) {
        (bool success, ) = wrappedNative.call{value: ethBalance}("");
        if (!success) {
            emit NativeTokenTransferFailed(ethBalance);
        }
    }

    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        // ... existing ERC20 deposit logic
    }
}
```

This ensures that native ETH present in the contract — regardless of how it arrived — is wrapped and deposited before the ERC20 sweep runs.

---

### Proof of Concept

1. Attacker (or any party) deploys a contract `Forcer` that holds 1 ETH and calls `selfdestruct(directDepositV1Address)`.
2. `DirectDepositV1.receive()` is **not** called. `address(directDepositV1).balance` becomes 1 ETH.
3. Off-chain infrastructure calls `ContractOwner.isDirectDepositV1Ready(directDepositV1, false)`. For the `wrappedNative` product, `balance = token.balanceOf(dda) + dda.balance = 0 + 1e18`. The function returns `true`.
4. Off-chain infrastructure calls `ContractOwner.creditDepositV1(subaccount)`, which calls `DirectDepositV1.creditDeposit()`.
5. `creditDeposit()` calls `token.balanceOf(address(this))` for each product. For `wrappedNative`, this returns `0` (no ERC20 balance). No deposit is made.
6. The 1 ETH remains in `DirectDepositV1`. The subaccount is not credited. The ETH can only be rescued by the owner via `withdrawNative()`, not deposited into the protocol.
