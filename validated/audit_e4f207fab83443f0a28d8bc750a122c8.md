### Title
Odd-leaf duplicate position allows same Merkle proof to verify two distinct `tx_index` values — (`merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` does not validate that `transaction_position` is within the actual number of transactions in the block. In an odd-width Bitcoin Merkle tree the last leaf is duplicated, so the same proof and the same `tx_id` hash produce the correct Merkle root for **both** the real last position `N` and the phantom duplicate position `N+1`. Because `verify_transaction_inclusion` is a public, unrestricted call and the contract stores only the 80-byte block header (which contains the Merkle root but not the transaction count), there is no guard that can reject the out-of-bounds index. A downstream bridge that uses `(tx_id, tx_index)` as its replay-protection key will accept the same Bitcoin transaction twice.

---

### Finding Description

**Root cause — `compute_root_from_merkle_proof`** [1](#0-0) 

The function iterates over the proof elements and at each step decides left/right placement purely from `current_position % 2`. It never checks that the initial `transaction_position` is less than the number of leaves. No such check is possible from the contract side either, because the stored `ExtendedHeader` contains only the 80-byte Bitcoin header fields (including `merkle_root`) and no transaction count. [2](#0-1) 

**Why the duplicate-leaf ambiguity arises**

Consider a 5-transaction block `[T0, T1, T2, T3, T4]`. Bitcoin's Merkle construction pads odd-width levels by duplicating the last element:

```
Level 0 (padded): T0  T1  T2  T3  T4  T4
Level 1 (padded): H01 H23 H44 H44
Level 2:          H0123   H4444
Root:             H(H0123, H4444)
```

`merkle_proof_calculator` for position 4 emits proof `P = [T4, H44, H0123]`. [3](#0-2) 

Tracing `compute_root_from_merkle_proof` with `(T4, position=4, P)`:

| step | position | parity | operation | result |
|------|----------|--------|-----------|--------|
| 1 | 4 | even | H(T4, P[0]=T4) | H44 |
| 2 | 2 | even | H(H44, P[1]=H44) | H4444 |
| 3 | 1 | odd | H(P[2]=H0123, H4444) | Root ✓ |

Tracing the **same call** with `(T4, position=5, P)`:

| step | position | parity | operation | result |
|------|----------|--------|-----------|--------|
| 1 | 5 | **odd** | H(P[0]=T4, T4) | H44 |
| 2 | 2 | even | H(H44, P[1]=H44) | H4444 |
| 3 | 1 | odd | H(P[2]=H0123, H4444) | Root ✓ |

Both calls return the same root and therefore both calls to `verify_transaction_inclusion` return `true`. Position 5 does not correspond to any real transaction.

---

### Impact Explanation

`verify_transaction_inclusion` is a public NEAR contract method with no `#[private]`, no `#[trusted_relayer]`, and no role gate — any account can call it. [4](#0-3) 

A downstream bridge that records processed events as `(tx_id, block_hash, tx_index)` will:
1. Accept `(T4, block_hash, 4)` → mark as processed, release funds.
2. Accept `(T4, block_hash, 5)` → different key, mark as processed, release funds again.

This is a cross-chain double-spend. The attacker needs only a real Bitcoin transaction in an odd-width block and the legitimate Merkle proof for it; no privileged role, no key leak, and no chain manipulation is required.

---

### Likelihood Explanation

- Odd-width blocks are common on Bitcoin mainnet (any block with an odd number of transactions).
- The attacker needs only a valid proof for the real last transaction, which is public on-chain data.
- The call is permissionless and costs only NEAR gas.
- The deprecated function remains callable; nothing in the contract prevents its use.

---

### Recommendation

1. **Short-term**: Require callers to supply the total transaction count (`tx_count`) alongside the proof. Validate `tx_index < tx_count` before calling `compute_root_from_merkle_proof`. Commit the `tx_count` into the proof verification so the root computation is position-unique.

2. **Medium-term**: Migrate all callers to `verify_transaction_inclusion_v2`. Note that v2 still delegates to the deprecated v1 internally and therefore inherits this ambiguity for the target transaction's position. [5](#0-4) 

3. **In `compute_root_from_merkle_proof`**: Add a `tx_count: usize` parameter and assert `transaction_position < tx_count` before the loop.

---

pub fn merkle_proof_calculator(tx_hashes: Vec<H256>, transaction_position: usize) -> Vec<H256> {
    let mut transaction_position = transaction_position;
    let mut merkle_proof = Vec::new();
    let mut current_hashes = tx_hashes;

    while current_hashes.len() > 1 {
        if current_hashes.len() % 2 == 1 {
            current_hashes.push(current_hashes[current_hashes.len() - 1].clone());
        }

        if transaction_position % 2 == 1 {
            merkle_proof.push(current_hashes[transaction_position - 1].clone());
        } else {
            merkle_proof.push(current_hashes[transaction_position + 1].clone());
        }

        let mut new_hashes = Vec::new();

        for i in (0..current_hashes.len() - 1).step_by(2) {
            new_hashes.push(compute_hash(&current_hashes[i], &current_hashes[i + 1]));
        }

        current_hashes = new_hashes;
        transaction_position /= 2;
    }

    merkle_proof
}

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
