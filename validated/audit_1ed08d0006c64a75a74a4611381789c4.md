### Title
`StatelessTransactionValidatorConfig` accepts inverted `min_sierra_version`/`max_sierra_version` without cross-field validation, causing all declare transactions to be permanently rejected at the gateway — (`crates/apollo_gateway_config/src/config.rs`)

---

### Summary

`StatelessTransactionValidatorConfig` derives `Validate` but carries no cross-field constraint asserting `min_sierra_version <= max_sierra_version`. If an operator (or a misconfigured deployment) sets these two fields in inverted order, the `validate_sierra_version` predicate in `StatelessTransactionValidator` becomes unsatisfiable for every possible Sierra version, and the gateway permanently rejects every declare transaction with `UnsupportedSierraVersion`.

---

### Finding Description

`StatelessTransactionValidatorConfig` is defined as:

```rust
#[derive(Clone, Debug, Deserialize, PartialEq, Serialize, Validate)]
pub struct StatelessTransactionValidatorConfig {
    ...
    pub min_sierra_version: VersionId,
    pub max_sierra_version: VersionId,
    ...
}
```

Neither field carries a `#[validate(...)]` attribute, and no custom `validate()` implementation exists. The `Validate` derive is therefore a no-op for these two fields. The parent struct `GatewayStaticConfig` calls `#[validate(nested)]` on this config, but that only delegates to the same empty `validate()`. [1](#0-0) 

The version range is consumed in `validate_sierra_version`:

```rust
if self.config.min_sierra_version <= sierra_version && sierra_version <= max_sierra_version {
    return Ok(());
}
Err(StatelessTransactionValidatorError::UnsupportedSierraVersion { ... })
``` [2](#0-1) 

When `min_sierra_version > max_sierra_version`, the conjunction `min <= v && v <= max` is logically impossible for any `v`. Every declare transaction therefore hits the `Err` branch regardless of the Sierra version embedded in its contract class.

`validate_sierra_version` is called unconditionally for every `RpcTransaction::Declare` that reaches the stateless validator: [3](#0-2) 

---

### Impact Explanation

**Matching scope**: *High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.*

With an inverted version range in the live config, the gateway's `StatelessTransactionValidator::validate` returns `UnsupportedSierraVersion` for every declare transaction, regardless of the Sierra version the submitter provides. No declare transaction can enter the mempool or be sequenced. New contract classes cannot be deployed on the network until the node is restarted with a corrected config. The misconfiguration is silent at startup — no error is logged, no panic occurs, and the node appears healthy.

---

### Likelihood Explanation

Low. The misconfiguration requires an operator to supply `min_sierra_version > max_sierra_version` in the node config file or via a deployment overlay. This can happen as an operational mistake (e.g., copy-paste error in a Kubernetes overlay, or a config-manager update that swaps the two version fields). The default values (`1.1.0` / `1.9.MAX`) are correct, so the risk is limited to non-default deployments. However, the absence of any startup-time guard means the error is not caught until declare transactions begin failing.

---

### Recommendation

Add a cross-field validation to `StatelessTransactionValidatorConfig` that asserts `min_sierra_version <= max_sierra_version`. Using the `validator` crate already in use:

```rust
impl StatelessTransactionValidatorConfig {
    #[allow(dead_code)]
    fn validate_version_range(&self) -> Result<(), ValidationError> {
        if self.min_sierra_version <= self.max_sierra_version {
            Ok(())
        } else {
            Err(ValidationError::new("min_sierra_version must be <= max_sierra_version"))
        }
    }
}
```

and annotate the struct with `#[validate(custom = "Self::validate_version_range")]`. This mirrors the pattern already used for `far_behind_proposal_threshold` in `ConsensusDynamicConfig`. [4](#0-3) 

---

### Proof of Concept

1. Start the node with a config that sets:
   ```json
   "gateway_config.static_config.stateless_tx_validator_config.min_sierra_version.minor": 9,
   "gateway_config.static_config.stateless_tx_validator_config.max_sierra_version.minor": 1
   ```
   (major = 1 for both, so `min = 1.9.0 > max = 1.1.0`).

2. The node starts without error. `config_validate` passes because `StatelessTransactionValidatorConfig::validate()` performs no cross-field check.

3. Submit any declare transaction carrying a Sierra contract class with any version in `[1.1.0, 1.9.0]`.

4. `validate_sierra_version` evaluates `1.9.0 <= v && v <= 1.1.0` → `false` for every `v`, and returns `UnsupportedSierraVersion`.

5. The gateway rejects the transaction. All declare transactions are permanently blocked until the node is restarted with a corrected config. [5](#0-4) [6](#0-5)

    fn validate_sierra_version(
        &self,
        sierra_program: &[Felt],
    ) -> StatelessTransactionValidatorResult<()> {
        // Any patch version is valid. (i.e. when check version for upper bound, we ignore the Z
        // part in a version X.Y.Z).
        let mut max_sierra_version = self.config.max_sierra_version;
        max_sierra_version.0.patch = usize::MAX;

        let sierra_version = VersionId::from_sierra_program(sierra_program)?;
        if self.config.min_sierra_version <= sierra_version && sierra_version <= max_sierra_version
        {
            return Ok(());
        }

        Err(StatelessTransactionValidatorError::UnsupportedSierraVersion {
            version: sierra_version,
            min_version: self.config.min_sierra_version,
            max_version: self.config.max_sierra_version,
        })
    }
