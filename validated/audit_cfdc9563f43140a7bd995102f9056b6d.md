### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Forgery Guard Added in `verify_transaction_inclusion_v2` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced as a secure wrapper around the original `verify_transaction_inclusion`. The wrapper adds a coinbase Merkle proof check specifically to block the 64-byte transaction Merkle proof forgery attack. However, `verify_transaction_inclusion` is still declared `pub` on the NEAR contract — the Rust `#[deprecated]` attribute is a compile-time lint only and does not restrict on-chain invocation. Any unprivileged NEAR caller can invoke the old function directly, completely skipping the forgery guard, and receive a `true` proof result for a fabricated transaction inclusion claim.

---

### Finding Description

`verify_transaction_inclusion_v2` is the secure entry point: [1](#0-0) 

It first validates a coinbase Merkle proof of the same depth as the target proof, then delegates to the original function: [2](#0-1) 

The coinbase-depth check is the entire mitigation for the 64-byte forgery attack: by requiring a valid coinbase proof of identical depth, it constrains the Merkle tree depth and prevents an attacker from supplying an internal node hash as `tx_id`.

The original function, however, remains `pub`: [3](#0-2) 

Its own doc comment acknowledges the vulnerability explicitly: [4](#0-3) 

The `#[deprecated]` attribute emits a Rust compiler warning to callers in Rust source code. It has zero effect on the NEAR ABI: the method is exported as a callable contract function and any NEAR account can invoke it via a signed transaction or a cross-contract call, with no restriction whatsoever.

The proof verification logic inside the old function only recomputes the Merkle root from the supplied `tx_id` and proof path and compares it to the stored header's `merkle_root`: [5](#0-4) 

There is no check that `tx_id` is a leaf node (a real transaction hash) rather than an internal node hash. A 64-byte internal node hash can be crafted to satisfy the root equation without corresponding to any real transaction.

---

### Impact Explanation

A recipient NEAR contract that calls `verify_transaction_inclusion` to gate a cross-chain action (e.g., releasing bridged funds, minting tokens, recording a payment) can be deceived into accepting a forged proof. The attacker supplies an internal Merkle tree node hash as `tx_id` along with a crafted proof path; the function returns `true`; the downstream contract acts on a Bitcoin transaction that never existed. This is a direct proof-verification forgery with concrete financial impact on any protocol that trusts the return value of this function.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte Merkle forgery technique is well-documented (https://www.bitmex.com/blog/64-Byte-Transactions) and has known tooling. The only prerequisite is knowledge of a valid block hash stored in the contract and the ability to construct a valid Merkle path from an internal node — both are achievable with public Bitcoin chain data. Likelihood is high for any deployment where downstream contracts rely on this function.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or gate it with an explicit `#[private]` attribute so it cannot be called as a NEAR contract method. The secure path through `verify_transaction_inclusion_v2` already delegates to it internally, so no functionality is lost. Alternatively, delete the old function body entirely and have `verify_transaction_inclusion_v2` contain the full logic inline.

```rust
// Option A: make it private so it can still be called internally
fn verify_transaction_inclusion(&self, args: ProofArgs) -> bool { ... }

// Option B: remove it entirely and inline the logic in v2
```

---

### Proof of Concept

1. Attacker identifies a Bitcoin block `B` whose header is stored in the contract (any confirmed block hash from `mainchain_header_to_height`).
2. Attacker selects an internal Merkle tree node `N` from block `B` whose 64-byte preimage can be crafted to satisfy the Merkle root equation (standard 64-byte attack).
3. Attacker constructs `ProofArgs { tx_id: N, tx_block_blockhash: B, tx_index: <crafted>, merkle_proof: <crafted path>, confirmations: 1 }`.
4. Attacker calls `verify_transaction_inclusion(args)` directly on the NEAR contract — bypassing `verify_transaction_inclusion_v2` entirely.
5. The function computes `compute_root_from_merkle_proof(N, index, path)`, which equals `B.merkle_root` by construction, and returns `true`.
6. Any downstream contract that called `verify_transaction_inclusion` and trusted the result now believes a non-existent Bitcoin transaction was confirmed.

The structural parallel to the external report is exact: `verify_transaction_inclusion_v2` is the `withdraw()` wrapper that adds the critical guard; `verify_transaction_inclusion` is the `withdrawToken()` inner function that is still `pub` and reachable directly, allowing the guard to be skipped entirely.     #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
