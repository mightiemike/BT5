### Title
Stale Mainchain Block Used as Difficulty-Interval Anchor for Fork Blocks — (`contract/src/dogecoin.rs`, `contract/src/litecoin.rs`, `contract/src/bitcoin.rs`)

### Summary

When computing the required difficulty for a block that falls on a difficulty-adjustment boundary, the contract fetches the start-of-interval block via `get_header_by_height`, which always reads from the **mainchain** mapping. For fork blocks whose chain diverged before that boundary, the mainchain's block at the interval start is a different block (with a different timestamp) than the fork's actual ancestor at the same height. The contract therefore computes the wrong expected difficulty for those fork blocks — a direct analog of the Compound "outdated cached borrowIndex" desynchronization.

### Finding Description

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // always the mainchain map
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

This is called in all three chain-specific difficulty routines to obtain the timestamp of the first block in the retarget window:

**Bitcoin** (`bitcoin.rs` lines 78–86):
```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [2](#0-1) 

**Litecoin** (`litecoin.rs` lines 86–93):
```rust
let first_block_height = prev_block_header.block_height - blocks_to_go_back;
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [3](#0-2) 

**Dogecoin** (`dogecoin.rs` lines 291–297) — the developers themselves flagged this with a TODO:
```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [4](#0-3) 

The correct approach is to walk the fork's own chain backwards from `prev_block_header` by `blocks_to_go_back` steps (following `prev_block_hash` links through `headers_pool`), not to look up the mainchain's block at the same height.

**When the bug is silent**: If the fork diverges at height F and the interval-start height S satisfies S ≤ F, the fork's ancestor at S is identical to the mainchain's block at S (shared history), so the timestamps agree and the bug has no effect.

**When the bug fires**: If S > F (the fork diverged before the interval start), the fork has its own block at height S stored only in `headers_pool` (not in `mainchain_height_to_header`). The contract silently uses the mainchain's block at S instead, producing a different `actual_time_taken` and therefore a different expected `bits` value.

### Impact Explanation

The computed `expected_bits` passed to `require!(expected_bits == block_header.bits, ...)` is wrong for fork blocks that cross a difficulty-adjustment boundary when the fork diverged before the interval start.

Two outcomes are possible depending on the sign of the timestamp error:

1. **Under-estimated difficulty (attacker-favorable)**: If the mainchain's block at S has an earlier timestamp than the fork's block at S, `actual_time_taken` is inflated, the computed target is looser, and the contract accepts a fork block whose `bits` field encodes less work than the protocol actually requires. An attacker who controls the fork's block timestamps (within MTP and future-time constraints) can engineer this condition.

2. **Over-estimated difficulty**: The inverse case causes a legitimately valid fork block to be rejected, a denial-of-service for honest relayers on that fork.

The security-critical outcome is (1): a fork block with insufficient PoW is accepted. If the attacker accumulates enough such blocks to exceed the mainchain's `chain_work`, `reorg_chain` promotes the fraudulent fork to the canonical chain, corrupting `mainchain_tip_blockhash` and `mainchain_height_to_header`. Subsequent calls to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` then attest to transactions in the fraudulent chain. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

The entry point is the public, unprivileged `submit_blocks` function. [7](#0-6) 

The attacker must build a fork that (a) diverges before a difficulty-adjustment boundary and (b) reaches that boundary. For Dogecoin (1-block retarget interval) this is trivially satisfied for every block after height 145,000. For Bitcoin/Litecoin (2016-block interval) the attacker must sustain a fork across at least one full retarget period, which is expensive but not impossible for a well-resourced adversary. The TODO comment in `dogecoin.rs` confirms the developers are aware the logic may be incorrect, raising confidence that this is a real gap rather than an intentional design choice.

### Recommendation

Replace the `get_header_by_height` call in all three difficulty routines with an ancestor walk that follows `prev_block_hash` links through `headers_pool` for exactly `blocks_to_go_back` steps starting from `prev_block_header`. This mirrors how Bitcoin Core's `GetAncestor` works and ensures the interval-start timestamp always belongs to the chain being validated, not the current mainchain.

### Proof of Concept

1. Deploy the contract (Bitcoin feature) with genesis at height 0.
2. Submit 2016 mainchain blocks (heights 0–2015); the mainchain's block at height 0 has timestamp T₀.
3. Submit a fork block at height 1 whose timestamp is T₀ + Δ (Δ > 0, within MTP/future-time limits). Continue building the fork up to height 2015 with a fork block at height 0 having timestamp T₀ + Δ.
4. Submit the fork's block at height 2016 (difficulty-adjustment boundary). The contract computes `actual_time_taken` using the **mainchain's** block at height 0 (timestamp T₀) instead of the fork's block at height 0 (timestamp T₀ + Δ). The computed timespan is Δ seconds larger than correct, yielding a looser `expected_bits`.
5. The fork block is accepted with `bits` encoding less work than the protocol requires. If the fork's cumulative `chain_work` exceeds the mainchain's, `reorg_chain` is triggered and the fraudulent chain becomes canonical. [8](#0-7) [9](#0-8)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

            .headers_pool
            .get(&args.tx_block_blockhash)

    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
