Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Caller-Supplied `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and gates access on `owner`, which is a free argument supplied by the caller. Because any address can pass an already-allowed address as `owner`, the deposit allowlist is fully neutralized and any unprivileged address can add liquidity to a permissioned pool.

## Finding Description

`MetricOmmPool.addLiquidity` dispatches the before-hook with the actual caller as `sender` and the caller-supplied `owner` as the second argument: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values verbatim to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, …)` but leaves `sender` unnamed (discarded) and checks only `owner`: [3](#0-2) 

The admin-facing setter `setAllowedToDeposit` names the gated entity a **depositor** — clearly the party calling `addLiquidity` — not the position owner: [4](#0-3) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller), not `recipient`: [5](#0-4) 

The asymmetry confirms the deposit check is wrong. Because `owner` is a free argument, any caller can pass an already-allowed address as `owner` and the guard returns the correct selector unconditionally.

## Impact Explanation

A pool admin deploys a permissioned pool with `DepositAllowlistExtension` to restrict who may provide liquidity (e.g., a KYC-gated or institutional pool). Any unprivileged address bypasses this control entirely by calling `pool.addLiquidity(allowedAddress, salt, deltas, callbackData, extensionData)`. The hook sees `owner = allowedAddress` (on the allowlist) and passes. The pool credits the LP position to `allowedAddress` while pulling tokens from the unauthorized caller via callback. This is a direct admin-boundary break: a pool-admin-configured guard is bypassed by an unprivileged path with no special role or token required. The unauthorized caller can also force an allowed address to hold an unwanted LP position that only that address can remove, constituting a griefing vector against legitimate LPs.

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with a publicly observable allowed address as `owner`. No flash loan, oracle manipulation, or privileged role is needed. Any address can trigger it at any time against any pool using `DepositAllowlistExtension` with `allowAllDepositors` set to `false`.

## Recommendation

Check `sender` (the actual caller) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` as an extension; `allowAllDepositors[pool] = false`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Only `alice` is supposed to deposit.
3. Attacker (not on allowlist) calls:
   ```solidity
   pool.addLiquidity(
       alice,       // owner — alice IS on the allowlist
       salt,
       deltas,
       callbackData,
       extensionData
   );
   ```
4. `_beforeAddLiquidity(msg.sender=attacker, owner=alice, ...)` is dispatched.
5. `DepositAllowlistExtension.beforeAddLiquidity` checks `allowedDepositor[pool][alice]` → `true`. Guard passes.
6. Pool credits the LP position to `alice`; attacker's tokens are pulled via callback.
7. Attacker has successfully deposited into a permissioned pool without being on the allowlist. The deposit allowlist is fully neutralized.
