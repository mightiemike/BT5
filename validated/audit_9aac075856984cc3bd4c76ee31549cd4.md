### Title
`ERC20Helper.safeTransfer` / `safeTransferFrom` Do Not Check Code Existence at Token Address, Enabling Silent Deposit Crediting Without Fund Transfer — (File: `core/contracts/libraries/ERC20Helper.sol`)

---

### Summary

Nado's custom `ERC20Helper` library implements `safeTransfer` and `safeTransferFrom` using a raw low-level `.call()` pattern that does not verify code exists at the token address. When called against an address with no deployed contract, the EVM returns `(success=true, data="")`, causing the `require` check to pass silently. This is the same root cause as the Olympus M-02 finding. The impact in Nado is that a deposit into a product whose token address has no code will silently succeed, causing `Clearinghouse.depositCollateral` to credit the depositor's subaccount with collateral that was never actually transferred.

---

### Finding Description

`ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` are implemented as:

```solidity
(bool success, bytes memory data) = address(self).call(
    abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [1](#0-0) 

When `address(self)` has no deployed bytecode, the EVM low-level call returns `(true, "")`. The condition `success && (data.length == 0 || ...)` evaluates to `true`, so the `require` passes without reverting. No actual token transfer occurs, but the caller believes the transfer succeeded.

This flaw propagates directly into `EndpointStorage.handleDepositTransfer`, which is the on-chain entry point for all user collateral deposits:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
}
``` [2](#0-1) 

The only guard is `address(token) != address(0)`. Any non-zero address with no deployed code bypasses this check. Both `safeTransferFrom` and `safeTransferTo` silently succeed, and execution continues as if real tokens were moved.

After `handleDepositTransfer` returns, `Clearinghouse.depositCollateral` unconditionally credits the depositor's subaccount:

```solidity
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [3](#0-2) 

The same flaw also exists in `BaseWithdrawPool.handleWithdrawTransfer` and `Clearinghouse.handleWithdrawTransfer`, where `token.safeTransfer` is called without a code existence check: [4](#0-3) [5](#0-4) 

Additionally, `DirectDepositV1.sol` contains its own inline copy of the same vulnerable pattern: [6](#0-5) 

---

### Impact Explanation

If a product is registered with a token address that has no deployed code (e.g., a self-destructed token, a pre-computed CREATE2 address before deployment, or a misconfigured product registration), any user can call `depositCollateral` for that product with an arbitrarily large `amount`. Both `safeTransferFrom` and `safeTransferTo` silently succeed, and the user's subaccount is credited with `amountRealized` of that product's collateral. The user can then:

- Trade against real liquidity using phantom collateral
- Withdraw real assets from other products if cross-collateral health allows
- Corrupt the protocol's accounting of total collateral, breaking `manualAssert` invariants

The corrupted state delta is: `spotEngine` balance for the depositor's subaccount is inflated by `amountRealized` with zero real token backing, directly violating the solvency invariant that on-chain token balances must back all credited subaccount positions.

---

### Likelihood Explanation

The trigger requires a product to be registered with a token address that has no code. Realistic conditions include:

1. A token contract that self-destructs (possible on EVM chains that support `SELFDESTRUCT`)
2. A product registered at a pre-computed CREATE2 address before the token contract is deployed
3. An operator misconfiguration during product registration

Once such a product exists, any unprivileged user can exploit it permissionlessly by calling the standard deposit entrypoint. No special privileges, leaked keys, or governance capture are required.

---

### Recommendation

Replace `ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` with OpenZeppelin's `SafeERC20`, which includes a code existence check via `Address.isContract` before performing the call. Alternatively, add an explicit check inside `ERC20Helper`:

```solidity
require(address(self).code.length > 0, "Token has no code");
```

This check should be added at the top of both `safeTransfer` and `safeTransferFrom` in `ERC20Helper.sol`, and the inline copy in `DirectDepositV1.sol` should be replaced with the same pattern or with OpenZeppelin's `SafeERC20`. [7](#0-6) 

---

### Proof of Concept

1. A product (e.g., `productId = 5`) is registered in the spot engine with `token = 0xDEAD...BEEF`, a non-zero address with no deployed bytecode.
2. Attacker calls `Endpoint.depositCollateral(productId=5, amount=1_000_000e18)`.
3. `EndpointStorage.handleDepositTransfer` is called with `token = IERC20Base(0xDEAD...BEEF)`.
4. `require(address(token) != address(0))` passes (address is non-zero).
5. `safeTransferFrom(token, attacker, 1_000_000e18)` executes `address(0xDEAD...BEEF).call(...)` → returns `(true, "")` → `require(true && (0 == 0 || ...))` passes. No tokens moved.
6. `safeTransferTo(token, clearinghouse, 1_000_000e18)` — same result. No tokens moved.
7. `Clearinghouse.depositCollateral` calls `spotEngine.updateBalance(5, attacker_subaccount, 1_000_000e18_normalized)`.
8. Attacker's subaccount now has `1_000_000e18` of product 5 collateral with zero real token backing.
9. Attacker uses this phantom collateral to pass health checks and withdraw real quote tokens from the protocol. [1](#0-0) [2](#0-1) [8](#0-7)
