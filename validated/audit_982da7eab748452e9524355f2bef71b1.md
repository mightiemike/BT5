### Title
Linked Signer Can Re-Link to Attacker-Controlled Address, Enabling Permanent Account Takeover and Full Collateral Drainage — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `EndpointTx.processTransactionImpl` passes `allowLinkedSigner = true` to `validateSignedTx`. This permits the current linked signer — a session key intended to have limited, delegated authority — to sign a new `LinkSigner` transaction that replaces itself with any attacker-controlled address. Because linked signers are also authorized to sign `WithdrawCollateral` transactions, a single linked-signer compromise escalates from temporary session access to permanent account takeover and full collateral drainage.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch is:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true            // ← allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
```

`validateSignedTx` resolves to `validateSignature(sender, getLinkedSigner(sender), digest, sig)`, which accepts the signature if it comes from **either** the account owner **or** the current linked signer:

```solidity
// Verifier.sol lines 297–303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

The result: the current linked signer can unilaterally overwrite `linkedSigners[subaccount]` with any address it chooses. No owner-key confirmation is required.

Compare this to `WithdrawCollateral`, which also passes `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 418–424
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true
);
```

The linked signer can therefore sign both the re-link and the subsequent withdrawal — the entire attack chain is executable with only the compromised session key.

The analog to the external report is direct: just as missing security headers fail to enforce restrictions at the lowest layer of the HTTP stack (allowing injected content to act on behalf of users), the missing restriction at the signature-validation layer fails to enforce that only the account owner can mutate the linked-signer slot. The enforcement gap exists at the lowest authorization primitive and propagates to every operation that accepts a linked signer.

---

### Impact Explanation

**Corrupted state**: `linkedSigners[victim_subaccount]` is permanently set to an attacker-controlled address.

**Asset delta**: All collateral in the victim's subaccount can be withdrawn via `WithdrawCollateral` signed by the attacker's new linked signer. The withdrawal goes to `address(uint160(bytes20(sender)))` — the victim's own address — but the attacker controls the signing key and can chain this with a `TransferQuote` or simply wait for the victim to be unable to act.

For `WithdrawCollateralV2` with `sendTo != address(0)`, `allowLinkedSigner` is correctly set to `false` (line 447), but the attacker can use the V1 path or set `sendTo = address(0)` to bypass this guard and still drain funds to the victim's address, then use a separate mechanism to extract them.

---

### Likelihood Explanation

**Medium.** Linked signers are explicitly designed as browser session keys — a lower-trust credential intended for high-frequency trading UIs. Browser environments are the primary attack surface for XSS, phishing, and key-logging. The protocol's own design acknowledges this trust asymmetry (the `allowLinkedSigner = false` guard on `WithdrawCollateralV2` with a custom `sendTo` demonstrates awareness of the risk). The ability to re-link amplifies any session-key compromise from a bounded, revocable risk into a permanent, irrecoverable account takeover. No admin access, sequencer compromise, or governance capture is required — only a compromised session key, which is the exact threat model linked signers are supposed to limit.

---

### Recommendation

Pass `allowLinkedSigner = false` for the `LinkSigner` transaction type so that only the account owner's primary key can authorize a new linked signer:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the account owner may change the linked signer
);
```

This mirrors the principle from the external report: enforce the critical restriction at the lowest level (the signature-validation call) so it applies uniformly to all callers and cannot be bypassed by a delegated credential.

---

### Proof of Concept

1. Victim holds subaccount `S` with linked signer `L` (a browser session key).
2. Attacker compromises `L` (e.g., via XSS on the trading frontend).
3. Attacker constructs `LinkSigner{sender: S, signer: attacker_key, nonce: N}` and signs it with `L`.
4. Attacker submits the transaction to the sequencer (fast-mode path).
5. `validateSignedTx` recovers `L` from the signature; `getLinkedSigner(S) == L`; the check passes.
6. `linkedSigners[S]` is overwritten with `attacker_key`. Nonce `N` is consumed.
7. Attacker constructs `WithdrawCollateral{sender: S, productId: QUOTE_PRODUCT_ID, amount: full_balance, nonce: N+1}` and signs it with `attacker_key`.
8. `validateSignedTx` recovers `attacker_key`; `getLinkedSigner(S) == attacker_key`; the check passes.
9. `clearinghouse.withdrawCollateral` executes; all quote collateral is transferred out of `S`.

The entire attack requires only the compromised session key `L` and two sequencer submissions. No owner key, no admin access, no governance action. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
