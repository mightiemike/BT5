Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of end user, enabling allowlist bypass or complete lockout — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps on the `sender` argument, which equals `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` mediates a swap, `msg.sender` of `pool.swap()` is the router contract, not the end user. This creates a binary failure: if the router is allowlisted, every user bypasses per-user restrictions; if it is not, every individually allowlisted user is locked out of the only public swap periphery.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` of that call: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. The swap interface provides no separate "actual user" parameter analogous to the `owner` parameter in `addLiquidity`, so there is no on-chain way to recover the true end user when the router is in the call stack.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` deliberately ignores `sender` and checks `owner` — the explicit beneficiary argument — which is immune to intermediary routing: [5](#0-4) 

## Impact Explanation
**Bypass path (router allowlisted):** A pool admin who allowlists the router so that legitimate users can access the standard periphery inadvertently grants every unprivileged user unrestricted swap access. `allowedSwapper[pool][router]` is `true` for every swap routed through it, regardless of who called the router. The per-user allowlist is rendered inoperative for all router-mediated swaps, allowing unauthorized traders to execute swaps in pools designed to be restricted — adverse selection against LPs and potential regulatory breach.

**Lockout path (router not allowlisted):** Individually allowlisted users who call `router.exactInputSingle(...)` receive `NotAllowedToSwap` even though they are on the allowlist. The only usable swap path requires calling `pool.swap()` directly and implementing `metricOmmSwapCallback` — the standard periphery is broken for every allowlisted pool. This constitutes broken core swap functionality causing an unusable swap flow for legitimate users.

Both outcomes meet the allowed impact gate: broken core pool functionality causing loss of funds or unusable swap flows, and admin-boundary break where the per-user restriction is bypassed by an unprivileged path.

## Likelihood Explanation
No special privilege is required. Any user can call `MetricOmmSimpleRouter.exactInputSingle` or `exactInput`. The bypass materialises the moment a pool admin allowlists the router — a natural operational step when deploying a pool intended for use with the standard periphery. The lockout materialises for every allowlisted pool that does not also allowlist the router, which is the default state. Both conditions are reachable by any unprivileged trader.

## Recommendation
1. **Preferred fix — pass actual user through `extensionData`:** Have the router encode `msg.sender` into `extensionData` before forwarding to the pool. `SwapAllowlistExtension` can decode and check the real end user when `extensionData` is present, falling back to `sender` for direct pool calls.
2. **Alternative — add a dedicated `actualSender` field to the swap extension interface:** Extend `IMetricOmmExtensions.beforeSwap` with an `actualSender` parameter that the router populates with `msg.sender` before calling the pool, similar to the transient-storage callback payer pattern already used in `MetricOmmSimpleRouter`.
3. **Minimum mitigation — documentation and factory guard:** Document that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` and add a factory-level check preventing deployment of a pool with `SwapAllowlistExtension` in its extension order unless `allowAllSwappers` is set.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension configured in BEFORE_SWAP_ORDER.
2. Pool admin calls swapExtension.setAllowedToSwap(pool, alice, true)
   — alice is the only allowlisted swapper.
3. Pool admin calls swapExtension.setAllowedToSwap(pool, address(router), true)
   — necessary so alice can use the standard periphery.
4. Bob (not allowlisted) calls:
       router.exactInputSingle(ExactInputSingleParams{pool: pool, ...})
   The router calls pool.swap(recipient, ...) with msg.sender = router.
   The pool calls _beforeSwap(router, ...).
   SwapAllowlistExtension checks allowedSwapper[pool][router] → true.
   Bob's swap succeeds despite not being on the allowlist.

Alternatively (lockout path):
3'. Pool admin does NOT allowlist the router.
4'. Alice calls router.exactInputSingle(...).
    The pool calls _beforeSwap(router, ...).
    SwapAllowlistExtension checks allowedSwapper[pool][router] → false.
    Revert: NotAllowedToSwap — alice is locked out of the standard periphery.
```
