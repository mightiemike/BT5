### Title
Linked Signer Delegation Has No Expiry, Enabling Permanent Unauthorized Subaccount Control — (File: `core/contracts/EndpointStorage.sol`)

---

### Summary

The `linkedSigners` mapping stores delegated signing authority as a plain `address` with no associated expiry timestamp. Once a linked signer is set, it retains full signing authority over the subaccount indefinitely. There is no on-chain mechanism to automatically expire stale delegations. A compromised linked signer key gives an attacker permanent ability to transfer collateral to attacker-controlled subaccounts and to replace the linked signer itself, locking out the legitimate owner.

---

### Finding Description

The `LinkSigner` struct in `IEndpoint.sol` contains only `sender`, `signer`, and `nonce` — no `expiry` or `validUntil` field:

```solidity
struct LinkSigner {
    bytes32 sender;
    bytes32 signer;
    uint64 nonce;
}
```

The `linkedSigners` mapping in `EndpointStorage.sol` stores a plain `address` with no associated timestamp:

```solidity
mapping(bytes32 => address) internal linkedSigners;
```

When a `LinkSigner` transaction is processed in `EndpointTx.sol`, the signer address is written unconditionally with no expiry:

```solidity
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
```

`getLinkedSigner` returns this address with no staleness check:

```solidity
function getLinkedSigner(bytes32 subaccount) public view virtual returns (address) {
    return RiskHelper.isIsolatedSubaccount(subaccount)
        ? linkedSigners[IOffchainExchange(offchainExchange).getParentSubaccount(subaccount)]
        : linkedSigners[subaccount];
}
```

The linked signer is accepted (`allowLinkedSigner = true`) for every high-impact transaction type:

| Transaction | Line in EndpointTx.sol |
|---|---|
| `WithdrawCollateral` | 423 |
| `LiquidateSubaccount` | 403 |
| `MintNlp` / `BurnNlp` | 544, 564 |
| `TransferQuote` | 604 |
| `LinkSigner` (self-replacement) | 586 |

The critical consequence of the last entry is that a compromised linked signer can sign a new `LinkSigner` transaction to replace itself with another attacker-controlled address, permanently locking out the legitimate owner.

---

### Impact Explanation

A compromised linked signer key gives an attacker:

1. **Direct fund transfer:** The linked signer can sign `TransferQuote` with an arbitrary `recipient` subaccount, moving the victim's quote balance to an attacker-controlled subaccount from which it can be withdrawn.
2. **Permanent takeover:** The linked signer can sign a new `LinkSigner` transaction (since `allowLinkedSigner = true` at line 586) to replace itself with another attacker-controlled address, permanently locking out the legitimate owner even after the original key is rotated at the application layer.
3. **Position manipulation:** The linked signer can sign `MintNlp`/`BurnNlp` and `LiquidateSubaccount` to manipulate the subaccount's positions.

The corrupted state is `linkedSigners[subaccount]` — an authorization that should be time-limited but is instead permanent.

**Impact: 3**

---

### Likelihood Explanation

Linked signers are the intended mechanism for automated trading bots, API integrations, and programmatic order submission. These keys are routinely stored in server environments, scripts, and browser-based tooling — all of which are higher-risk storage contexts than hardware wallets. Key compromise in these environments is a realistic, non-theoretical scenario. The absence of any expiry means the window of exploitation is unbounded once a key is leaked.

**Likelihood: 3**

---

### Recommendation

Add an `expiry` field to the `LinkSigner` struct and store it alongside the signer address in the `linkedSigners` mapping. Modify `getLinkedSigner` to return `address(0)` when the current block timestamp exceeds the stored expiry. This mirrors the existing `_expired` pattern already used for order expiration in `OffchainExchange.sol`:

```solidity
function _expired(uint64 expiration) internal view returns (bool) {
    return expiration <= getOracleTime();
}
```

Apply the same pattern to linked signer lookups.

---

### Proof of Concept

1. User Alice sets a linked signer (e.g., a trading bot key) via a `LinkSigner` transaction. `linkedSigners[alice_subaccount] = bot_address` is written with no expiry.
2. The bot key is later compromised (e.g., leaked from a server config).
3. Attacker signs a `TransferQuote` transaction using the compromised key, setting `sender = alice_subaccount` and `recipient = attacker_subaccount`. The sequencer processes it; `clearinghouse.transferQuote` moves Alice's quote balance to the attacker's subaccount.
4. Attacker signs a new `LinkSigner` transaction using the compromised key, setting `signer = attacker_address2`. `linkedSigners[alice_subaccount]` is overwritten. Alice can no longer recover control via the linked signer path.
5. Attacker withdraws from `attacker_subaccount` using their own key.

The root cause — `mapping(bytes32 => address) internal linkedSigners` with no expiry — is the necessary vulnerable step. Without it, the attack window would be bounded by a configurable session timeout. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
