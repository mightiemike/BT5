### Title
Deprecated `verify_transaction_inclusion` Remains Callable, Bypassing Coinbase Merkle Proof Validation Enforced in `verify_transaction_inclusion_v2` — (File: `contract/src/lib.rs`)

---

### Summary

The `BtcLightClient` contract exposes two public transaction-inclusion verification entry points. The newer `verify_transaction_inclusion_v2` enforces a coinbase merkle proof check specifically to prevent the 64-byte internal-node forgery attack. The older `verify_transaction_inclusion` (marked `#[deprecated]`) omits this check entirely but remains fully callable by any unprivileged NEAR account. An attacker can bypass the coinbase proof invariant by calling the deprecated function directly, obtaining a `true` result for a transaction that does not exist on-chain.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction Merkle proof forgery vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions). It enforces the coinbase proof check before delegating to the deprecated function: [1](#0-0) 

The deprecated `verify_transaction_inclusion` skips this check entirely and computes only the transaction merkle root: [2](#0-1) 

The contract's own warning acknowledges the risk: [3](#0-2) 

Despite the deprecation annotation, the function carries `#[pause]` (not `#[private]`), meaning it is gated only by the pause flag — not by caller identity. Any unprivileged NEAR account can invoke it at any time the contract is unpaused. [4](#0-3) 

The structural parallel to the reported `veRAACToken` bug is exact:

| `veRAACToken` | `BtcLightClient` |
|---|---|
| `lock()` enforces `MAX_TOTAL_SUPPLY` | `verify_transaction_inclusion_v2` enforces coinbase proof |
| `increase()` skips the check | `verify_transaction_inclusion` skips the check |
| Attacker calls `increase()` directly | Attacker calls `verify_transaction_inclusion` directly |
| Invariant: total locked ≤ cap | Invariant: tx proof requires coinbase validation |

---

### Impact Explanation

The core security guarantee of the light client is that `verify_transaction_inclusion` returns `true` only for transactions that genuinely appear in a confirmed Bitcoin block. By calling the deprecated function with a `tx_id` that is actually a 64-byte internal Merkle tree node hash, an attacker can produce a valid-looking Merkle path to the block's `merkle_root` without a real transaction. Any downstream NEAR contract or bridge application that calls `verify_transaction_inclusion` (rather than `v2`) to gate asset releases, cross-chain messages, or other security-critical actions will accept the forged proof and execute the protected action. [5](#0-4) 

---

### Likelihood Explanation

The deprecated function is a named, documented public method on the contract. A relayer, integrator, or attacker who reads the ABI or the source can call it directly without any special role or key. The 64-byte forgery technique is well-documented and the necessary crafted Merkle path can be constructed off-chain with no mining work. The only prerequisite is that the target block hash is already in the `headers_pool`, which is the normal operating state of the contract.

---

### Recommendation

Remove the `verify_transaction_inclusion` function from the public ABI entirely, or gate it with `#[private]` so it is only callable by the contract itself (as `verify_transaction_inclusion_v2` already calls it internally). If backward compatibility must be preserved for existing callers, add the same coinbase proof requirement directly inside `verify_transaction_inclusion` and remove the `#[deprecated]` bypass path. [6](#0-5) 

---

### Proof of Concept

1. Deploy or use an existing `BtcLightClient` instance with a known confirmed block `B` at height `H` on the mainchain.
2. Construct a 64-byte value `fake_tx` that is the hash of two adjacent leaf nodes in block `B`'s Merkle tree (an internal node). Build a Merkle path from `fake_tx` to `merkle_root` of `B` — this path is shorter by one level than a real leaf proof and is trivially computable from the block's transaction list.
3. Call `verify_transaction_inclusion` (the deprecated entry point) with:
   - `tx_id = fake_tx`
   - `tx_block_blockhash = B`
   - `tx_index` = the index corresponding to the internal node position
   - `merkle_proof` = the constructed path
   - `confirmations` = any value ≤ confirmed depth
4. The function returns `true` because it only checks `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.block_header.merkle_root` with no coinbase anchor. [7](#0-6) 

5. The same call to `verify_transaction_inclusion_v2` would panic at the coinbase proof check, confirming the bypass is exclusive to the deprecated path. [8](#0-7)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
