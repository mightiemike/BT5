### Title
Stale Light Client State Accepted Without Freshness Check in `verify_transaction_inclusion` - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` never compare the Bitcoin timestamp of the stored chain tip against the current NEAR block time. If the relayer stops submitting headers for an extended period (due to downtime, DoS, or a NEAR network halt), the light client's mainchain state becomes stale. A consumer contract calling either verification function will receive a `true` result based on an outdated chain view, enabling a double-spend attack against any bridge or settlement protocol that relies on these functions.

### Finding Description

The `verify_transaction_inclusion` function computes confirmation depth purely from stored block heights:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

It never reads `env::block_timestamp_ms()` and never compares the Bitcoin `time` field of `mainchain_tip_blockhash` against the current NEAR wall-clock time. The stored `ExtendedHeader` carries a `block_header.time` (a `u32` Unix timestamp) that could be arbitrarily old, and the function is blind to this.

By contrast, `check_pow` in `contract/src/bitcoin.rs` — which runs only during `submit_blocks` — explicitly reads `env::block_timestamp_ms()` to reject headers whose Bitcoin timestamp is more than `MAX_FUTURE_BLOCK_TIME_LOCAL` (2 hours) ahead of NEAR's clock. The same awareness is entirely absent from the verification path.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after validating the coinbase proof, so it inherits the same gap.

### Impact Explanation

A consumer contract (bridge, lending protocol, cross-chain settlement) that calls `verify_transaction_inclusion` to gate fund release can be deceived:

1. Attacker submits a Bitcoin transaction T₁ that is included in block B at height H.
2. The light client records B in its mainchain; the consumer contract's required `confirmations` are satisfied.
3. NEAR experiences downtime (relayer halted, NEAR network paused, or targeted DoS). The light client tip freezes at height H + k.
4. During downtime, the attacker broadcasts a conflicting transaction T₂ (double-spend). A Bitcoin reorg occurs; B is orphaned and T₂ is confirmed in the winning chain.
5. When NEAR resumes, the attacker immediately calls `verify_transaction_inclusion` for T₁ before the relayer can submit the reorg. The light client still maps B to its mainchain, the confirmation count check passes, and the Merkle proof for T₁ is valid against the stored (now-orphaned) merkle root.
6. The consumer contract releases funds for T₁ even though T₁ no longer exists in the real Bitcoin chain.

The corrupted invariant is: `mainchain_header_to_height` maps a block hash to a height, but after a reorg that the light client has not yet processed, that mapping no longer reflects the canonical Bitcoin chain. `verify_transaction_inclusion` treats this stale mapping as authoritative.

### Likelihood Explanation

The NEAR network has experienced downtime before (as has Bitcoin's mempool during fee spikes). The relayer is an off-chain service with no on-chain liveness guarantee. Bitcoin reorgs of 1–3 blocks occur occasionally on mainnet; on Dogecoin and Litecoin (also supported by this contract via feature flags) they are more frequent. The attack window is the gap between NEAR resuming and the relayer catching up — a window the attacker can widen by front-running with high gas.

### Recommendation

Inside `verify_transaction_inclusion`, after fetching `heaviest_block_header`, add a staleness guard:

```rust
let near_now_secs = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
let tip_time = heaviest_block_header.block_header.time;
require!(
    near_now_secs.saturating_sub(tip_time) <= MAX_ACCEPTABLE_TIP_AGE_SECS,
    "light client tip is too stale to verify"
);
```

`MAX_ACCEPTABLE_TIP_AGE_SECS` should be chosen conservatively (e.g., 2–4 hours for Bitcoin, shorter for Dogecoin). This mirrors the existing `MAX_FUTURE_BLOCK_TIME_LOCAL` guard already applied in `check_pow`.

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
