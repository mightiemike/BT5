### Title
Sanctions Check Performed Only at Slow-Mode Submission, Not at Execution — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The Nado `Endpoint` slow-mode queue validates `requireUnsanctioned(sender)` when a transaction is enqueued but never re-validates it when the transaction is dequeued and executed up to 3 days later. A user who is sanctioned after submitting a `WithdrawCollateral` slow-mode transaction will still have that withdrawal executed, bypassing the protocol's sanctions screening entirely.

---

### Finding Description

When a user calls `submitSlowModeTransaction`, the implementation in `submitSlowModeTransactionImpl` performs a single sanctions check before writing the transaction to the queue: [1](#0-0) 

```solidity
requireUnsanctioned(sender);
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: transaction
});
```

The `SLOW_MODE_TX_DELAY` is hardcoded to 3 days. [2](#0-1) 

When the queued transaction is later executed — either by the user calling `executeSlowModeTransaction()` or by the sequencer via `_executeSlowModeTransaction` — no sanctions check is performed: [3](#0-2) 

The execution path delegates to `processSlowModeTransactionImpl`, which for `WithdrawCollateral` directly calls `clearinghouse.withdrawCollateral(...)` with no re-validation of the sender's sanctions status: [4](#0-3) 

The `requireUnsanctioned` function is defined in `EndpointStorage` and queries the external `ISanctionsList` contract: [5](#0-4) 

There is no call to `requireUnsanctioned` anywhere in `_executeSlowModeTransaction`, `processSlowModeTransaction`, or `processSlowModeTransactionImpl`. The sanctions state captured at submission time is treated as permanently valid for the lifetime of the queued transaction.

The same gap applies to the `LinkSigner` slow-mode path: a sanctioned user's queued `LinkSigner` transaction will still execute, granting a linked signer the ability to continue trading on behalf of the sanctioned subaccount. [6](#0-5) 

---

### Impact Explanation

A sanctioned user can withdraw collateral from the protocol after being added to the sanctions list, as long as they submitted the `WithdrawCollateral` slow-mode transaction before the sanction was applied. The concrete asset delta is: sanctioned-address-controlled funds exit the protocol's custody to the sanctioned address, defeating the protocol's OFAC/sanctions compliance mechanism. For `LinkSigner`, the impact extends to continued trading activity by the sanctioned party through a proxy signer.

---

### Likelihood Explanation

The 3-day window between submission and execution is a realistic and intentional design feature, not an edge case. Sanctions list additions (e.g., OFAC SDN list updates) are routine and publicly announced. A user who monitors their own sanctions risk — or a malicious actor who anticipates imminent sanctioning — can trivially exploit this window. The `executeSlowModeTransaction()` function is callable by anyone with no access restriction, so the user can self-execute their own queued withdrawal once the delay expires. [7](#0-6) 

---

### Recommendation

Re-execute `requireUnsanctioned(txn.sender)` inside `processSlowModeTransactionImpl` for every transaction type that moves funds or grants permissions (at minimum `WithdrawCollateral` and `LinkSigner`). Alternatively, add a single `requireUnsanctioned(sender)` call at the top of `processSlowModeTransactionImpl` before the `txType` dispatch, mirroring the check already present in `submitSlowModeTransactionImpl`. This ensures the sanctions state is evaluated against the current list at the moment of settlement, not the moment of submission.

---

### Proof of Concept

1. Alice submits a `WithdrawCollateral` slow-mode transaction via `submitSlowModeTransaction`. `requireUnsanctioned(Alice)` passes — she is not yet sanctioned. [1](#0-0) 

2. Within the 3-day window, Alice is added to the Chainalysis sanctions list (e.g., OFAC SDN update).

3. After `executableAt` elapses, Alice (or anyone) calls `executeSlowModeTransaction()`. [7](#0-6) 

4. `_executeSlowModeTransaction` dequeues the transaction and calls `this.processSlowModeTransaction(txn.sender, txn.tx)`. No sanctions check occurs. [8](#0-7) 

5. `processSlowModeTransactionImpl` dispatches to the `WithdrawCollateral` branch and calls `clearinghouse.withdrawCollateral(...)` directly. Alice's funds are transferred to her address despite her sanctioned status. [4](#0-3)

```text
    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
