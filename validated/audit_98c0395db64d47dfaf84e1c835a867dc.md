### Title
Missing `max_l2_gas_amount` Validation for Declare Transactions Allows Unbounded Gas Claims at Gateway Admission — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary
The `StatelessTransactionValidator` explicitly skips the `max_l2_gas_amount` upper-bound check for `Declare` transactions. Any other transaction type (`Invoke`, `DeployAccount`) is rejected if `l2_gas.max_amount > config.max_l2_gas_amount` (default 1,210,000,000). A `Declare` transaction with `l2_gas.max_amount = u64::MAX` passes the stateless validator unconditionally, bypassing the admission gate that exists for every other transaction type.

### Finding Description

In `validate_resource_bounds`, the check is structured as:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The test `valid_l2_gas_amount_on_declare` explicitly confirms this is the intended (but unguarded) behavior: a Declare transaction with `max_amount: GasAmount(200)` passes even when `max_l2_gas_amount: 100` is configured. [2](#0-1) 

The default production config sets `max_l2_gas_amount = 1_210_000_000`. [3](#0-2) 

The `l2_gas.max_amount` field is directly consumed by the OS as the initial gas budget for the transaction:

```cairo
func get_initial_user_gas_bound(common_tx_fields: CommonTxFields*) -> felt {
    assert common_tx_fields.n_resource_bounds = 3;
    return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
}
``` [4](#0-3) 

For Declare transactions, this value is passed directly to `get_initial_user_gas_bound` before being capped by `VALIDATE_MAX_SIERRA_GAS`: [5](#0-4) 

The `max_possible_fee` computation uses saturating arithmetic over `max_amount * max_price_per_unit`. A Declare transaction with `max_l2_gas_amount = u64::MAX` and `max_price_per_unit = min_gas_price (8e9)` produces a saturated `max_possible_fee = u128::MAX`, which will fail the blockifier's balance pre-check. However, for moderate overages — e.g., `max_l2_gas_amount = 2 × config_limit = 2_420_000_000` — the resulting `max_possible_fee` is large but not impossible, and a sufficiently funded account passes the balance check, allowing the transaction to enter the mempool with an inflated gas claim.

The stateful validator at the gateway level only checks the L2 gas *price* against the previous block, not the gas *amount*: [6](#0-5) 

### Impact Explanation

A Declare transaction with `l2_gas.max_amount` above the configured limit but below the balance-check saturation threshold passes both the stateless and stateful gateway validators and is admitted to the mempool. The mempool and batcher use `max_l2_gas_amount` for gas accounting and block-space reservation. An attacker with a funded account can submit Declare transactions claiming arbitrarily large L2 gas amounts (up to the balance-check ceiling), inflating the apparent gas consumption of a block and potentially starving legitimate transactions from being included.

**Impact**: High — Mempool/gateway admission accepts invalid Declare transactions that should be rejected before sequencing.

### Likelihood Explanation

Any funded account can craft a `RpcDeclareTransaction::V3` with `resource_bounds.l2_gas.max_amount` set above `max_l2_gas_amount`. The bypass is unconditional — no special privilege or race condition is required. The only natural gate is the balance check, which is bypassable for moderate overages.

### Recommendation

Apply the same `max_l2_gas_amount` upper-bound check to Declare transactions:

```diff
-// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
-if let RpcTransaction::Declare(_) = tx {
-} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
+if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
     return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
         gas_amount: resource_bounds.l2_gas.max_amount,
         max_gas_amount: self.config.max_l2_gas_amount,
     });
 }
```

If Declare transactions legitimately require a higher gas ceiling (e.g., due to compilation overhead), introduce a separate `max_l2_gas_amount_declare` config parameter rather than removing the check entirely.

### Proof of Concept

The existing test already demonstrates the bypass:

```rust
// From stateless_transaction_validator_test.rs — passes today, should fail
fn valid_l2_gas_amount_on_declare(config, rpc_tx_args) {
    // config: max_l2_gas_amount = 100
    // rpc_tx_args: l2_gas.max_amount = GasAmount(200)  ← exceeds limit
    let tx_type = TransactionType::Declare;
    let tx_validator = StatelessTransactionValidator { config };
    let tx = rpc_tx_for_testing(tx_type, rpc_tx_args);
    assert_matches!(tx_validator.validate(&tx), Ok(()));  // ← admitted despite exceeding limit
}
``` [2](#0-1) 

To demonstrate mempool admission with a moderate overage, submit a `RpcDeclareTransaction::V3` with `l2_gas.max_amount = 2_420_000_000` (2× the production limit) and `max_price_per_unit = 8_000_000_000` from an account holding at least `2_420_000_000 × 8_000_000_000 = ~1.9 × 10^19` STRK-wei. The transaction passes both gateway validators and enters the mempool, claiming twice the permitted gas budget.
