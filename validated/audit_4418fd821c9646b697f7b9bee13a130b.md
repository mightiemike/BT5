### Title
`DepositAllowlistExtension` Checks `owner` But Not `sender`, Allowing Unauthorized Callers to Bypass the Deposit Allowlist via Owner/Sender Separation — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` enforces the deposit allowlist against the `owner` parameter (the position recipient) while silently ignoring the `sender` parameter (the actual `msg.sender` of `addLiquidity`). Because `addLiquidity` explicitly supports an operator pattern where `msg.sender ≠ owner`, any unprivileged caller can bypass the allowlist by supplying an allowlisted address as `owner`, paying the tokens themselves, and creating a position for that address in a pool the admin intended to be permissioned.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook:

- `sender` = `msg.sender` (the actual caller / token payer)
- `owner` = the caller-supplied position owner [1](#0-0) 

`_beforeAddLiquidity` forwards both to the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but discards it (unnamed `address,`). It only checks `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

Because the pool's `addLiquidity` allows `msg.sender ≠ owner` (the operator pattern), an unauthorized caller can:

1. Call `pool.addLiquidity(owner = allowlistedAddress, ...)`.
2. The extension evaluates `allowedDepositor[pool][allowlistedAddress]` → `true` → passes.
3. The pool issues the liquidity callback to `msg.sender` (the unauthorized caller), who pays the tokens.
4. The position is minted under `allowlistedAddress`.

The allowlist check is structurally misbound: it validates the wrong actor. [4](#0-3) 

---

### Impact Explanation

A pool admin deploys `DepositAllowlistExtension` to restrict liquidity provision to a curated set of addresses (e.g., KYC-verified LPs, institutional partners, or protocol-controlled addresses). The bypass allows any unprivileged address to inject liquidity into the pool, violating the admin-configured access boundary. Consequences include:

- **Broken permissioned-pool invariant**: The pool receives liquidity from sources the admin explicitly excluded.
- **Griefing via dust positions**: An attacker can create many tiny positions under an allowlisted address, forcing that address to manage or remove them.
- **Regulatory / compliance breach**: Pools relying on the allowlist for KYC/AML enforcement are silently open to non-compliant depositors.

This fits the allowed impact gate: *"Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path."*

---

### Likelihood Explanation

- The operator pattern (`msg.sender ≠ owner`) is explicitly documented and supported by the pool interface.
- Any unprivileged address can exploit this with a single `addLiquidity` call; no special setup, flash loan, or privileged access is required.
- The allowlisted address need not cooperate; the attacker only needs to know one allowlisted address (e.g., from on-chain events emitted by `setAllowedToDeposit`). [5](#0-4) 

---

### Recommendation

Change `beforeAddLiquidity` to check **both** `sender` (the actual caller/payer) **and** `owner` (the position recipient), or at minimum check `sender`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    bool poolAllowsAll = allowAllDepositors[msg.sender];
    if (!poolAllowsAll && (!allowedDepositor[msg.sender][sender] || !allowedDepositor[msg.sender][owner])) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The exact policy (check sender, owner, or both) should be documented and match the pool admin's intent. If the allowlist is meant to gate the payer, check `sender`. If it gates position holders, check `owner`. If both must be restricted, check both.

### Proof of Concept

```solidity
// Pool admin configures allowlist: only `alice` may deposit
extension.setAllowedToDeposit(address(pool), alice, true);

// bob is NOT on the allowlist
// bob calls addLiquidity with owner=alice
// Extension checks allowedDepositor[pool][alice] → true → passes
// bob's callback pays the tokens; alice receives the position
vm.prank(bob);
pool.addLiquidity(
    alice,          // owner = allowlisted address
    0,              // salt
    deltas,
    callbackData,   // bob's callback pays tokens
    ""
);

// Result: bob successfully deposited into a permissioned pool
// despite not being on the allowlist
assertTrue(pool._positionBinShares[positionKey(alice, 0, binIdx)] > 0);
``` [3](#0-2) [6](#0-5)
