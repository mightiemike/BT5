### Title
Unchecked `approve()` Return Value in `creditDeposit()` Silently Locks User Collateral in DDA — (`File: core/contracts/DirectDepositV1.sol`)

### Summary
`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without checking the return value. For ERC20 tokens that return `false` on failure instead of reverting, the approval silently fails, causing the subsequent `endpoint.depositCollateralWithReferral()` call to revert (no allowance granted). The deposited tokens remain permanently locked in the DDA contract, recoverable only by the multisig owner.

### Finding Description
In `DirectDepositV1.creditDeposit()`, the contract iterates over all spot product token IDs, checks its own balance, and attempts to approve the endpoint to pull those tokens before calling `depositCollateralWithReferral`. The `approve()` call at line 92 uses the raw `IIERC20Base.approve()` interface and discards the boolean return value:

```solidity
// core/contracts/DirectDepositV1.sol : DirectDepositV1.creditDeposit()
token.approve(address(endpoint), balance);   // return value ignored
endpoint.depositCollateralWithReferral(
    subaccount,
    productId,
    uint128(balance),
    "-1"
);
```

The `ERC20Helper` library used elsewhere in the protocol provides `safeTransfer` and `safeTransferFrom` wrappers that check return values, but provides **no `safeApprove` equivalent**. `DirectDepositV1` also defines its own `safeTransfer` internally but uses a bare `approve()` call.

If a supported spot token's `approve()` returns `false` without reverting (a known pattern in non-standard ERC20s such as USDT on some chains), the endpoint's `depositCollateralWithReferral` will attempt a `transferFrom` with zero allowance and revert. The entire `creditDeposit()` call reverts, leaving the token balance stranded in the DDA.

The same unchecked pattern appears in `ContractOwner.wrapVaultAsset()` at lines 530–531:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
```

If either `approve()` silently fails, the vault `deposit()` call will revert and the asset tokens remain stuck in the DDA.

### Impact Explanation
Tokens sent to a DDA are locked. The `withdraw()` function on `DirectDepositV1` is `onlyOwner` (the `ContractOwner` contract), and `ContractOwner.withdrawFromDirectDepositV1()` is `onlyOwner` (the multisig). An ordinary user has no self-service recovery path. Every subsequent call to `creditDepositV1()` will also revert for the affected token, permanently blocking that product's deposit flow for the subaccount until the multisig manually intervenes.

### Likelihood Explanation
`ContractOwner.creditDepositV1(bytes32 subaccount)` carries no access restriction — any address can call it for any subaccount. The Nado protocol is designed to support multiple spot collateral tokens. If any listed token is a non-reverting ERC20 (e.g., USDT variants, BNB, or future tokens added via `addOrUpdateProducts`), the silent-failure path is reachable by any caller, including the token depositor themselves triggering the credit flow.

### Recommendation
Add a `safeApprove` wrapper to `ERC20Helper` (mirroring the existing `safeTransfer`/`safeTransferFrom` pattern) and use it in both `DirectDepositV1.creditDeposit()` and `ContractOwner.wrapVaultAsset()`:

```solidity
function safeApprove(IERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Replace the bare `approve()` calls in `DirectDepositV1.creditDeposit()` and `ContractOwner.wrapVaultAsset()` with this wrapper.
