### Title
Linked Signer Can Overwrite Itself via `LinkSigner` Transaction — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `processTransactionImpl` passes `allowLinkedSigner = true` to `validateSignedTx`. This means the existing linked signer — a limited delegation credential (hot wallet / API key) — can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. Only the subaccount owner should be authorized to change the linked signer.

---

### Finding Description

In `EndpointTx.sol`, `processTransactionImpl` handles the `LinkSigner` transaction type at lines 576–590:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

`validateSignedTx` calls `validateSignature`, which in turn calls `verifier.validateSignature` with the second argument being `getLinkedSigner(sender)` when `allowLinkedSigner = true`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
```

`Verifier.validateSignature` accepts the signature if the recovered address equals either the subaccount owner address or the `linkedSigner` argument:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

The result: the existing linked signer can sign a valid `LinkSigner` transaction and overwrite `linkedSigners[subaccount]` with any address it chooses.

The protocol already demonstrates awareness of this distinction. `WithdrawCollateralV2` explicitly restricts the linked signer from withdrawing to external addresses:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    signedTx.tx.sendTo == address(0)  // linked signer blocked for external withdrawals
);
```

No equivalent restriction exists for `LinkSigner`.

---

### Impact Explanation

A compromised linked signer (leaked hot-wallet key, compromised API key) can:

1. Sign a `LinkSigner` transaction setting `signer` to an attacker-controlled address.
2. The new linked signer is now accepted for all operations that pass `allowLinkedSigner = true`: `WithdrawCollateral`, `LiquidateSubaccount`, `TransferQuote`, `MintNlp`, `BurnNlp`, and further `LinkSigner` changes.
3. The attacker permanently controls the subaccount. The real owner cannot revoke access because the attacker's new linked signer can immediately re-set the linked signer again.

Concrete corrupted state: `linkedSigners[subaccount]` is permanently overwritten to an attacker address, enabling full collateral drain via `WithdrawCollateral` (which also uses `allowLinkedSigner = true`).

---

### Likelihood Explanation

Linked signers are explicitly designed as hot wallets or API keys for automated trading — credentials with a higher exposure surface than the cold-wallet subaccount owner. A leaked API key or compromised hot wallet is a realistic operational event. No admin or governance action is required; the attacker only needs the existing linked signer's private key.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction handler so that only the subaccount owner can change the linked signer:

```solidity
// EndpointTx.sol — processTransactionImpl, LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
-   true
+   false   // only the subaccount owner may change the linked signer
);
```

This mirrors the existing pattern used in `WithdrawCollateralV2` where the linked signer is blocked from performing sensitive operations that affect the subaccount's trust configuration.

---

### Proof of Concept

1. Alice owns subaccount `S` and has set a linked signer `L` (a hot wallet) via a prior `LinkSigner` transaction.
2. Attacker obtains the private key of `L` (e.g., via a leaked `.env` file).
3. Attacker uses `L` to sign a `SignedLinkSigner` transaction with `sender = S`, `signer = attacker_address`, and a valid nonce.
4. The sequencer processes this as a `LinkSigner` transaction. `validateSignedTx` is called with `allowLinkedSigner = true`. `getLinkedSigner(S)` returns `L`. The signature from `L` is accepted.
5. `linkedSigners[S]` is now set to `attacker_address`.
6. Attacker signs a `WithdrawCollateral` transaction from `attacker_address` (the new linked signer), draining Alice's collateral.
7. If Alice tries to revoke by submitting her own `LinkSigner`, the attacker front-runs (or re-submits after) using the new linked signer to re-set it back to the attacker address.
