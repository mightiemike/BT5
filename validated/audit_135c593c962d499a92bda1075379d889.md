### Title
Non-Zero to Non-Zero ERC20 Approval in `creditDeposit` Permanently Bricks Deposits for USDT-Like Tokens — (`core/contracts/DirectDepositV1.sol`)

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. For tokens like USDT that revert on a non-zero → non-zero approval change, any residual allowance left after a prior deposit cycle will permanently prevent future calls to `creditDeposit()` from succeeding, locking deposited collateral in the DDA contract.

### Finding Description

`creditDeposit()` is an unrestricted public function that iterates over all registered spot product IDs, reads the DDA's token balance, approves the endpoint for that balance, and then calls `depositCollateralWithReferral` with `uint128(balance)`: [1](#0-0) 

The critical lines are: [2](#0-1) 

The `approve` is issued for `balance` (a `uint256`), but the deposit call passes `uint128(balance)`. If `balance > type(uint128).max`, the endpoint's `transferFrom` only consumes `uint128(balance)` tokens, leaving a residual allowance of `balance - uint128(balance)` on the endpoint. On the next invocation of `creditDeposit()`, the `approve` call will revert for USDT-like tokens because their `approve()` implementation requires the current allowance to be zero before setting a new non-zero value.

Even outside the overflow scenario, any execution path in which the endpoint does not consume the full approved amount (e.g., a partial transfer, a minimum-deposit rejection that does not revert the outer call, or a future upgrade to the endpoint) leaves a non-zero residual allowance that permanently bricks subsequent `creditDeposit()` calls for that token.

By contrast, `ContractOwner.wrapVaultAsset()` — which also approves a token — correctly resets the allowance to zero before setting a new value: [3](#0-2) 

`DirectDepositV1.creditDeposit()` does not follow this pattern.

### Impact Explanation

Once a residual allowance exists for a USDT-like spot token, every subsequent call to `creditDeposit()` reverts at the `approve` line. Tokens sent to the DDA contract can no longer be credited to the subaccount via the normal deposit flow. The funds are not immediately stolen, but they are effectively locked in the DDA until an admin manually withdraws them via `withdrawFromDirectDepositV1`, which requires owner access and is an out-of-band recovery path. The deposit pipeline for that subaccount and token is permanently broken until the allowance is manually cleared.

### Likelihood Explanation

`creditDeposit()` has no access control — any address can call it directly, or trigger it via `ContractOwner.creditDepositV1()`: [4](#0-3) 

The overflow trigger (`balance > type(uint128).max`) is unlikely for most tokens in practice, but the broader class of residual-allowance scenarios (endpoint partial consumption, future endpoint upgrades, or any revert-after-approve pattern) is realistic. The absence of a zero-reset is a latent bug that activates the moment any such condition is met.

### Recommendation

Reset the allowance to zero before setting a new value in `creditDeposit()`, mirroring the pattern already used in `wrapVaultAsset`:

```solidity
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
```

Alternatively, use OpenZeppelin's `SafeERC20.safeIncreaseAllowance` or `forceApprove` to handle non-standard ERC20 approval semantics safely.

### Proof of Concept

1. Deploy with a USDT-like token registered as a spot product.
2. Send tokens to the DDA such that `balance > type(uint128).max` (or engineer any scenario where the endpoint does not consume the full approved amount).
3. Call `creditDeposit()` — the approve succeeds, the deposit consumes only `uint128(balance)`, leaving a residual allowance.
4. Send any additional tokens to the DDA.
5. Call `creditDeposit()` again — the `token.approve(address(endpoint), balance)` call reverts because the current allowance is non-zero, permanently blocking the deposit flow for that token. [5](#0-4)

    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }
