### Title
Missing `ENABLE_SECP_OPS` Guard on 4-Byte Secp Opcode Dispatch Path Bypasses Hard-Fork Gate - (`File: src/chia_dialect.rs`)

### Summary

`ChiaDialect::op()` has two separate dispatch paths for secp256k1 and secp256r1 signature verification. The 1-byte opcode path (opcodes 64, 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (`0x13d61f00`, `0x1c3a8f00`) dispatches to the **same underlying functions** with no flag check at all. An attacker-controlled CLVM program using the 4-byte encoding invokes real secp verification regardless of whether `ENABLE_SECP_OPS` is set, bypassing the intended hard-fork gate.

### Finding Description

In `src/chia_dialect.rs`, the `Dialect::op()` implementation for `ChiaDialect` handles operators in two distinct branches:

**Branch 1 — 4-byte opcodes (lines 157–182):**

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // no flag check
        0x1c3a8f00 => op_secp256r1_verify,   // no flag check
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
``` [1](#0-0) 

**Branch 2 — 1-byte opcodes (lines 248–249):**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

Both branches call the identical functions `op_secp256k1_verify` and `op_secp256r1_verify` from `src/secp_ops.rs`. The 1-byte branch correctly enforces `ENABLE_SECP_OPS`; the 4-byte branch does not. This is structurally identical to the reported `unstake()` / `rageQuit()` pattern: two entry points to the same operation, one with a precondition guard and one without.

The `ENABLE_SECP_OPS` flag is documented as the gate for secp opcodes:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

The 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` encode the same secp operations using the "unknown operator with assigned cost" format (cost multiplier in the high 3 bytes, cost function 0 in the low byte). Before the explicit dispatch was added, these would have fallen through to `op_unknown`, which returns nil with a cost — a no-op. Now they execute real cryptographic verification, but without the flag guard that the 1-byte forms require.

### Impact Explanation

**Consensus divergence / hard-fork gate bypass.** `ENABLE_SECP_OPS` is a hard-fork flag: it is absent from `MEMPOOL_MODE` and must be explicitly set by callers who have activated the fork. [4](#0-3) 

When `ENABLE_SECP_OPS` is **not** set:
- 1-byte opcodes 64/65 fall through to `unknown_operator` → return nil (lenient) or error (strict).
- 4-byte opcodes `0x13d61f00`/`0x1c3a8f00` execute `op_secp256k1_verify`/`op_secp256r1_verify` and **return nil on success or `Secp256Failed` on failure**.

A CLVM program using the 4-byte encoding therefore produces a different result than one using the 1-byte encoding under the same dialect flags. Nodes that have not activated the secp hard fork will evaluate the 4-byte form as a real signature check, while the 1-byte form would be a no-op. This divergence in evaluation outcome for the same logical operation is a consensus-breaking inconsistency.

Concretely: an attacker can craft a spend bundle whose puzzle uses `0x13d61f00` to perform secp256k1 signature verification before the hard fork activates. Pre-fork nodes will execute the check (and reject on bad signature), while the intended behavior is to treat it as a no-op unknown operator.

### Likelihood Explanation

**Medium.** The 4-byte encoding is valid CLVM bytecode reachable via any `run_program` call. Any caller that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` (including mempool mode) is exposed. An attacker who knows the opcode encoding — which is derivable from the cost constants documented in the source — can craft the 4-byte atom directly. No privileged access is required; only the ability to submit a CLVM program.

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte dispatch path, mirroring the 1-byte path:

```diff
 let f = match opcode {
-    0x13d61f00 => op_secp256k1_verify,
-    0x1c3a8f00 => op_secp256r1_verify,
+    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
+    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
     _ => {
         return unknown_operator(allocator, o, argument_list, flags, max_cost);
     }
 };
``` [1](#0-0) 

This ensures that both encodings of the secp operations are uniformly gated by the same flag, eliminating the divergence.

### Proof of Concept

```rust
use clvmr::allocator::Allocator;
use clvmr::chia_dialect::{ChiaDialect, ClvmFlags};
use clvmr::run_program::run_program;

fn main() {
    let mut allocator = Allocator::new();

    // Build: (0x13d61f00 pubkey msg sig)
    // 0x13d61f00 is the 4-byte secp256k1_verify opcode
    let opcode = allocator.new_atom(&[0x13, 0xd6, 0x1f, 0x00]).unwrap();
    // ... construct valid pubkey/msg/sig atoms ...
    // Run WITHOUT ENABLE_SECP_OPS
    let dialect = ChiaDialect::new(ClvmFlags::empty()); // no ENABLE_SECP_OPS

    // This executes op_secp256k1_verify despite ENABLE_SECP_OPS being absent.
    // With 1-byte opcode 64, the same dialect would return nil (unknown op).
    let result = run_program(&mut allocator, &dialect, program, env, u64::MAX);
    // result: Err(Secp256Failed) on bad sig — real verification ran, not a no-op
}
```

The 4-byte opcode path at `src/chia_dialect.rs` lines 175–182 dispatches to `op_secp256k1_verify` / `op_secp256r1_verify` in `src/secp_ops.rs` without the `ENABLE_SECP_OPS` check that lines 248–249 enforce, making the two paths behaviorally inequivalent under the same dialect configuration. pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
