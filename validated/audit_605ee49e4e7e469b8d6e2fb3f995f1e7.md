### Title
Fast Withdrawal Bypasses Subaccount Health Check, Enabling Collateral Extraction from Unhealthy Accounts — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal()` does not verify subaccount health before executing a collateral transfer. The standard withdrawal path enforces a health check inside `Clearinghouse.withdrawCollateral()`, but the fast withdrawal path skips it entirely. A user holding a valid signed fast-withdrawal transaction can submit it after their account has become unhealthy, draining collateral that should be available for liquidation.

---

### Finding Description

**Standard withdrawal path** (health-gated):

`Endpoint` → `EndpointTx.processTransactionImpl()` → `Clearinghouse.withdrawCollateral()` → health check → `handleWithdrawTransfer()` → `BaseWithdrawPool.submitWithdrawal()`

Inside `Clearinghouse.withdrawCollateral()`, the health check is mandatory: [1](#0-0) 

```solidity
IProductEngine.HealthType healthType = sender == X_ACCOUNT
    ? IProductEngine.HealthType.PNL
    : IProductEngine.HealthType.INITIAL;

require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
```

**Fast withdrawal path** (no health check):

`BaseWithdrawPool.submitFastWithdrawal()` only enforces three guards: [2](#0-1) 

```solidity
require(!markedIdxs[idx], "Withdrawal already submitted");
require(idx > minIdx, "idx too small");
markedIdxs[idx] = true;
Verifier v = Verifier(verifier);
v.requireValidTxSignatures(transaction, idx, signatures);
// ... fee logic ...
handleWithdrawTransfer(token, sendTo, transferAmount);
```

There is no call to `clearinghouse.getHealth()` anywhere in this path.

**Why `idx > minIdx` does not close the gap:**

`minIdx` is updated only inside `submitWithdrawal`, which is called only when the Clearinghouse *successfully* processes a normal withdrawal: [3](#0-2) 

```solidity
function submitWithdrawal(...) public {
    require(msg.sender == clearinghouse);
    if (markedIdxs[idx]) { return; }
    markedIdxs[idx] = true;
    minIdx = idx;          // only reached on success
    handleWithdrawTransfer(token, sendTo, amount);
}
```

If the sequencer attempts the normal withdrawal and it reverts at the health check, `submitWithdrawal` is never reached, so `minIdx` is **not** incremented. The fast withdrawal with the same `idx` therefore still satisfies `idx > minIdx` and executes without restriction.

**Concrete attack path:**

1. User holds a healthy subaccount; requests a fast withdrawal for amount A (assigned `idx = N`).
2. Protocol signers produce valid signatures for the transaction.
3. Market moves adversarially; subaccount health drops below zero.
4. Sequencer submits the normal `WithdrawCollateral` transaction with `nSubmissions = N`; `Clearinghouse.withdrawCollateral()` reverts at the health check. `minIdx` stays at `N-1`.
5. User calls `submitFastWithdrawal(N, transaction, signatures)` directly on `BaseWithdrawPool`.
6. Guards pass: `!markedIdxs[N]` ✓, `N > N-1` ✓, signatures valid ✓.
7. `handleWithdrawTransfer` executes; collateral is sent to the user.
8. The subaccount is now deeper underwater; the insurance fund must cover the shortfall.

---

### Impact Explanation

A user extracts collateral from a subaccount whose health is below the initial threshold. The assets that should remain available for a liquidator are removed, leaving the protocol with an insolvent position that must be socialized through the insurance fund or loss mutualization. This constitutes direct theft of collateral / protocol assets, matching the "theft of unclaimed yield" impact class in the reference report.

---

### Likelihood Explanation

**Medium.** The preconditions are realistic:

- Fast-withdrawal signatures have no on-chain expiry; once obtained they remain valid indefinitely.
- Volatile market conditions (the primary use-case for fast withdrawals) are exactly when accounts are most likely to become unhealthy between signing and submission.
- The attacker controls the submission timing and can wait for the sequencer's normal-path attempt to fail before submitting the fast withdrawal.
- No privileged access is required beyond obtaining a signed fast-withdrawal transaction while the account is still healthy.

---

### Recommendation

Add an explicit health check inside `submitFastWithdrawal` after resolving the sender from the transaction bytes, before executing the transfer:

```solidity
// After resolveFastWithdrawal(transaction) returns (productId, sendTo, transferAmount)
bytes32 senderSubaccount = bytes32(bytes20(sendTo));
require(
    IClearinghouse(clearinghouse).getHealth(
        senderSubaccount,
        IProductEngine.HealthType.INITIAL
    ) >= 0,
    ERR_SUBACCT_HEALTH
);
```

Alternatively, update `minIdx` to the attempted `idx` even when the normal withdrawal reverts (e.g., by having the sequencer call a dedicated `markIdx` function on failure), so that a failed normal withdrawal permanently blocks the corresponding fast withdrawal.

---

### Proof of Concept

```
1. Setup: user subaccount has 100 USDC collateral, 80 USDC in open perp positions.
   getHealth(subaccount, INITIAL) > 0  ✓

2. User requests fast withdrawal of 50 USDC.
   Protocol signers produce signatures for (idx=10, WithdrawCollateral{amount=50}).

3. Price drops 30%. getHealth(subaccount, INITIAL) < 0.

4. Sequencer calls Endpoint.processTransaction(WithdrawCollateral, nSubmissions=10).
   → Clearinghouse.withdrawCollateral() reverts at:
     require(getHealth(sender, INITIAL) >= 0, ERR_SUBACCT_HEALTH);
   → submitWithdrawal never called; minIdx remains 9.

5. User calls:
   BaseWithdrawPool.submitFastWithdrawal(
       10,                  // idx > minIdx (10 > 9) ✓
       transaction,         // signed WithdrawCollateral
       signatures           // valid protocol signer sigs ✓
   );

6. No health check is performed. handleWithdrawTransfer sends 50 USDC to user.

7. Subaccount now has 50 USDC collateral against 80 USDC perp exposure.
   Insurance fund must cover the ~30 USDC shortfall during liquidation.
```
