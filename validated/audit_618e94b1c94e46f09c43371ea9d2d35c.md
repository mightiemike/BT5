### Title
Storage Deposits Paid by `submit_blocks` Callers Are Permanently Retained When GC Frees Their Headers — (`contract/src/lib.rs`)

---

### Summary

`submit_blocks` charges callers a NEAR storage deposit proportional to the **net** storage change after GC runs. Because GC frees storage that was paid for by **previous** callers, those callers' deposits are permanently retained in the contract's balance and never returned. This is the direct analog to M-02: just as leftover ERC20 sell tokens were silently kept by the Swap contract, storage deposits paid by earlier `submit_blocks` callers are silently kept by the light client contract after GC evicts their headers.

---

### Finding Description

In `submit_blocks` (`contract/src/lib.rs`, lines 166–198), the deposit accounting follows this sequence:

```
initial_storage  = env::storage_usage()          // (1) snapshot before any change
submit new headers                                // (2) storage grows by X bytes
run_mainchain_gc(num_of_headers)                 // (3) storage shrinks by Y bytes (prior callers' headers)
diff_storage_usage = storage_usage - initial     // (4) net = X - Y  (saturating, so ≥ 0)
required_deposit   = diff * storage_byte_cost    // (5) current caller pays only for net
refund             = amount - required_deposit   // (6) excess returned to current caller
``` [1](#0-0) 

Step (3) calls `run_mainchain_gc`, which deletes old `ExtendedHeader` entries from `headers_pool` and `mainchain_height_to_header`: [2](#0-1) 

The storage freed in step (3) was originally paid for by **prior** callers. Because the net diff in step (4) subtracts that freed storage, the **current** caller's required deposit is reduced — effectively subsidized by prior callers' deposits. Those prior deposits are now unlocked from NEAR storage staking and sit in the contract's available balance, but the contract has no mechanism to track or return them to the original payers.

The contract never records a mapping of `(caller → storage_bytes_paid)`. There is no withdrawal function, no per-caller refund path, and no event that signals freed storage back to original depositors. [3](#0-2) 

---

### Impact Explanation

Every caller who pays a storage deposit via `submit_blocks` permanently loses that deposit once GC evicts their headers. The freed NEAR tokens accumulate in the contract's own balance and are inaccessible to the original payers. Over the lifetime of the contract (with `gc_threshold` set to ~52,704 blocks per year), the total leaked value equals the sum of all storage deposits ever paid for headers that have since been GC'd. For a busy relayer submitting thousands of headers per day, this represents a continuous, compounding drain on depositor funds.

---

### Likelihood Explanation

GC is not optional — it is called unconditionally inside every `submit_blocks` invocation at line 181: [4](#0-3) 

Once the chain exceeds `gc_threshold` blocks, every subsequent `submit_blocks` call triggers eviction of prior headers. No attacker action is required; the loss occurs automatically as part of normal relayer operation. `run_mainchain_gc` is also a public, unpermissioned entry point, so any external caller can accelerate eviction: [5](#0-4) 

---

### Recommendation

Track storage deposits per caller and issue refunds when GC frees their headers. One approach:

1. Maintain a `LookupMap<H256, (AccountId, NearToken)>` mapping each block hash to the account that paid for it and the deposit amount.
2. In `run_mainchain_gc`, when a header is removed via `remove_block_header`, look up the original payer and issue a `Promise::transfer` refund.
3. Alternatively, adopt a flat per-call storage fee model where the contract absorbs storage costs from a pre-funded treasury, removing the per-caller deposit requirement entirely.

---

### Proof of Concept

**Setup:** `gc_threshold = 3`, contract initialized with genesis + 2 headers.

1. **Caller A** calls `submit_blocks([H3])`. Storage grows by ~200 bytes. Caller A pays `200 * storage_byte_cost ≈ 0.002 NEAR`. Refund: 0. Caller A's deposit is now locked for H3's storage.

2. **Caller B** calls `submit_blocks([H4])`. Before computing the diff:
   - H4 is inserted (storage +200 bytes).
   - GC runs: `amount_of_headers_we_store = 4 > gc_threshold = 3`, so H1 is removed (storage −200 bytes).
   - `diff_storage_usage = (initial + 200 − 200) − initial = 0`.
   - `required_deposit = 0`. Caller B pays nothing and receives a full refund of any attached deposit.

3. **Result:** Caller A paid ~0.002 NEAR for H3's storage. H3 is still present (only H1 was GC'd in step 2). But when H3 is eventually GC'd in a future call, Caller A's deposit — already consumed in step 1 — is freed into the contract's balance with no refund path. Caller A has permanently lost ~0.002 NEAR.
