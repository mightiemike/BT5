Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps on the `sender` argument, which is `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` is used, `msg.sender` inside the pool is the router contract, not the originating user. If the pool admin allowlists the router to enable router-based swaps for legitimate users, every user — including non-allowlisted ones — can bypass the allowlist by routing through the router, completely defeating the access control.

## Finding Description
In `MetricOmmPool.swap()`, `msg.sender` is forwarded as the first argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap()` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [2](#0-1) 

In `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly, making the router the `msg.sender` inside the pool: [3](#0-2) 

The same misbinding occurs in `exactInput` (all hops): [4](#0-3) 

And in `exactOutputSingle` and the recursive `exactOutput` hops via `_exactOutputIterateCallback`: [5](#0-4) [6](#0-5) 

The exploit path is: attacker calls `router.exactInputSingle()` → router calls `pool.swap()` → pool calls `_beforeSwap(sender=router)` → extension checks `allowedSwapper[pool][router]` → returns `true` if router is allowlisted → swap proceeds regardless of who the original caller is. There is no existing guard that recovers the original user identity; `extensionData` is passed through unchanged from the router caller and is not decoded by the extension.

## Impact Explanation
A pool admin deploying a curated pool with `SwapAllowlistExtension` (e.g., for KYC-verified counterparties or whitelisted institutions) must allowlist the router to let legitimate users access the pool via the standard periphery. Once `allowedSwapper[pool][router] = true`, the check `allowedSwapper[pool][router]` returns `true` for every swap routed through it, regardless of who the original caller is. Any non-allowlisted address can trade freely on the curated pool by routing through `MetricOmmSimpleRouter`, exposing LP funds to unauthorized counterparties and breaking any compliance or pricing assumption tied to the allowlist. This constitutes broken core pool functionality causing potential loss of funds and a broken admin-boundary access control.

## Likelihood Explanation
The trigger is a natural and expected admin action: allowlisting the router so that allowlisted users can use the standard periphery. There is no warning in the contract or NatSpec that doing so opens the gate to all users. Any pool that (a) uses `SwapAllowlistExtension` and (b) wants router support will hit this condition. The attacker requires no special privilege — a standard `exactInputSingle` call suffices, and the attack is repeatable indefinitely.

## Recommendation
The extension must gate on the economically relevant actor, not the intermediary. Two sound approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Requires a coordinated convention between router and extension.
2. **Check `recipient` instead of `sender`**: The `recipient` argument to `beforeSwap` is set to the original user by the router (`params.recipient`). Gating on `recipient` correctly identifies the economic beneficiary, though the pool admin must decide whether to gate on payer or recipient.
3. **Document and provide a router-aware allowlist extension** that decodes the original caller from `extensionData`, with a clear warning that allowlisting the router without this opens the gate to all users.

## Proof of Concept
```solidity
// Setup: curated pool with SwapAllowlistExtension
// Admin allowlists alice and the router (to let alice use the router)
swapAllowlist.setAllowedToSwap(pool, alice, true);
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attacker (not allowlisted) swaps directly → reverts correctly
vm.prank(attacker);
pool.swap(attacker, true, 1000, type(uint128).max, "", "");
// → NotAllowedToSwap ✓

// Attacker routes through the router → passes the allowlist check
vm.prank(attacker);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// → succeeds; allowlist check saw sender=router which is allowlisted ✗
// Attacker trades on a pool they were never authorized to access
```
