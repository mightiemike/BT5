Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded from the pool, which is the pool's `msg.sender` — the router, not the originating user. When a pool admin allowlists the router address (a natural operational step), every unpermissioned user who calls the router can execute swaps against the pool's full liquidity, defeating the allowlist entirely.

## Finding Description
In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to `SwapAllowlistExtension.beforeSwap`. The extension then evaluates: [2](#0-1) 

Here `msg.sender` is the pool and `sender` is the router. When a user routes through `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` with no identity forwarding of the originating user: [3](#0-2) 

The router's own address becomes the pool-level `sender`. If the pool admin has called `setAllowedToSwap(pool, address(router), true)`, the check `allowedSwapper[pool][router]` evaluates to `true` for every caller of the router, regardless of whether that caller was individually allowlisted. No existing guard in the extension, pool, or router checks the originating user's identity.

## Impact Explanation
A pool deploying `SwapAllowlistExtension` intends to restrict which addresses may trade against its liquidity. Allowlisting the router collapses that restriction to zero: any unpermissioned address can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the router and execute swaps against the pool's full liquidity at oracle-derived prices. This constitutes a broken core pool functionality (allowlist access control) causing direct exposure of LP funds to every public user — matching the admin-boundary break and broken core functionality impact criteria.

## Likelihood Explanation
The scenario is operationally natural. A pool admin who wants to restrict direct pool access while still supporting the standard periphery router would allowlist the router address. The `SwapAllowlistExtension` interface and `setAllowedToSwap` setter give no indication that doing so opens the pool to all users: [4](#0-3) 

The bypass requires no special privileges, no flash loans, and no unusual token behavior — only a call to the public router.

## Recommendation
The extension must gate the original transaction initiator, not the immediate pool caller. The cleanest fix: the router appends `abi.encode(msg.sender)` to `extensionData` before calling `pool.swap`, and `SwapAllowlistExtension.beforeSwap` decodes and checks that address when `sender` is a recognized router. Alternatively, the router can enforce the allowlist check itself before calling the pool, so the extension only needs to gate direct (non-router) callers.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, address(router), true)` to enable standard router usage.
3. Attacker (not individually allowlisted) calls `router.exactInputSingle(...)` targeting the pool.
4. Pool calls `_beforeSwap(msg.sender=router, ...)`.
5. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. Attacker receives pool output tokens; LP balances are reduced without the allowlist having gated the actual trader. [5](#0-4) [6](#0-5)
