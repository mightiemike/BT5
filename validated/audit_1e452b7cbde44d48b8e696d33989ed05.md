### Title
Merkle Internal-Node Forgery via Deprecated `verify_transaction_inclusion` Returns `true` for Non-Existent Transaction — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

The deprecated but still-publicly-callable `verify_transaction_inclusion` method performs no validation that the supplied `tx_id` is a leaf-level transaction hash. An attacker can supply an internal Merkle tree node hash as `tx_id` with a proof shortened by one level, causing `compute_root_from_merkle_proof` to reconstruct the correct Merkle root and the function to return `true` for a transaction that does not exist in the block.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` but remains a `pub` NEAR contract method with no `#[private]` guard and no trusted-relayer restriction. Any external NEAR account can call it directly. [1](#0-0) 

The Rust `#[deprecated]` attribute emits a compiler warning for Rust callers only. It has zero effect on NEAR's ABI or on external transaction-level callers. The method is fully reachable from any NEAR account.

The only guards inside the function are:

1. `confirmations <= gc_threshold` — trivially satisfied with `confirmations = 1`.
2. `tx_block_blockhash` must be on the mainchain — satisfied by using any real confirmed block hash.
3. `merkle_proof` must be non-empty — satisfied by a 1-element proof. [2](#0-1) 

After those checks, the function delegates entirely to `compute_root_from_merkle_proof`: [3](#0-2) 

`compute_root_from_merkle_proof` is a pure positional hash-chain computation. It accepts any 32-byte value as `transaction_hash` and iterates through the proof, combining hashes level by level. There is no minimum proof-length check, no tree-depth check, and no assertion that the starting value is a leaf: [4](#0-3) 

The docstring explicitly acknowledges this but treats it as an out-of-scope concern: [5](#0-4) 

---

### Impact Explanation

For a 4-transaction block with leaf hashes `[A, B, C, D]`:

```
Level 2 (root):  root = hash(H_AB, H_CD)
Level 1 (nodes): H_AB = hash(A,B)    H_CD = hash(C,D)
Level 0 (leaves): A    B              C    D
```

The attacker supplies:
- `tx_id = H_AB` (an internal node, not a real txid)
- `tx_index = 0`
- `merkle_proof = [H_CD]` (one element, not two)

`compute_root_from_merkle_proof(H_AB, 0, [H_CD])` executes:
- Iteration 1: position 0 is even → `current_hash = hash(H_AB, H_CD) = root`
- Returns `root`

This equals `header.block_header.merkle_root`, so the function returns `true`.

The attacker has obtained a `true` inclusion proof for a 64-byte value that is not a transaction. Any downstream bridge, asset-release contract, or verification system that calls `verify_transaction_inclusion` and acts on its return value will be deceived into releasing assets for a fabricated transaction.

The sibling hashes (`H_CD` in this example) are public data available from any Bitcoin block explorer or full node, requiring no privileged access.

---

### Likelihood Explanation

- **Entrypoint is fully public**: `pub fn verify_transaction_inclusion` with no `#[private]`, no `#[trusted_relayer]`, no role check.
- **All preconditions are trivially met**: any real confirmed mainchain block hash satisfies the chain-membership and confirmation checks.
- **No cryptographic barrier**: the attacker needs only the sibling subtree hashes, which are public Bitcoin blockchain data.
- **Proof of concept is mechanically straightforward**: construct `ProofArgs { tx_id: H_AB, tx_block_blockhash: <real block>, tx_index: 0, merkle_proof: vec![H_CD], confirmations: 1 }` and call the method.

---

### Recommendation

1. **Remove or gate the deprecated method**: Apply `#[private]` to `verify_transaction_inclusion` or remove it from the ABI entirely. The `#[deprecated]` attribute provides no runtime protection.
2. **Enforce proof-length equals tree depth**: In `compute_root_from_merkle_proof`, validate that `merkle_proof.len()` equals `ceil(log2(tx_count))` for the block, preventing shortened proofs.
3. **Migrate all callers to `verify_transaction_inclusion_v2`**: The v2 method requires a coinbase proof of the same length as the transaction proof, which prevents this attack by anchoring the tree depth.

---

### Proof of Concept

```rust
// 4-tx block: leaves = [A, B, C, D]
// H_AB = double_sha256(A || B)  -- internal node, NOT a real txid
// H_CD = double_sha256(C || D)  -- sibling, public blockchain data
// root = double_sha256(H_AB || H_CD) == block.merkle_root

let args = ProofArgs {
    tx_id: H_AB,                    // internal node passed as tx_id
    tx_block_blockhash: real_block, // any confirmed mainchain block
    tx_index: 0,
    merkle_proof: vec![H_CD],       // proof of length depth-1 = 1
    confirmations: 1,
};

// compute_root_from_merkle_proof(H_AB, 0, [H_CD]):
//   position 0 even → hash(H_AB, H_CD) = root  ✓
// Returns: true  ← false confirmation for non-existent transaction
let result = contract.verify_transaction_inclusion(args);
assert!(result); // passes — exploit confirmed
```

The `#[deprecated]` attribute on `verify_transaction_inclusion` is a Rust compile-time annotation only. [6](#0-5) 
It does not restrict NEAR transaction-level callers. The method remains in the deployed ABI and is callable by any account, making this a concrete, externally reachable critical vulnerability.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
