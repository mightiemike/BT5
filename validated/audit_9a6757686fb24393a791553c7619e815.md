### Title
Difficulty Retarget Uses Main-Chain Ancestor Instead of Fork-Chain Ancestor, Allowing Incorrect PoW Validation for Fork Blocks — (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary

When validating the difficulty (`bits`) of a submitted fork block, all three chain modules (`dogecoin.rs`, `bitcoin.rs`, `litecoin.rs`) fetch the retarget boundary block using `get_header_by_height`, which unconditionally reads from `mainchain_height_to_header`. For a fork block whose chain diverges from the main chain before the retarget boundary height, this returns the **main chain's** block at that height rather than the **fork chain's** actual ancestor. The difficulty check therefore uses the wrong block's timestamp, allowing fork blocks with incorrect `bits` values to pass PoW validation.

The code itself acknowledges this concern with an explicit TODO comment.

It always resolves height → hash via `mainchain_height_to_header`, then fetches from `headers_pool`. There is no mechanism to follow the fork chain's `prev_block_hash` links back to the correct ancestor.

In `dogecoin.rs`, the Digishield per-block retarget path calls: [2](#0-1) 

The TODO comment on line 291 explicitly flags this as unverified. `height_first` is computed as `prev_block_header.block_height - 1` (for Digishield, where `difficulty_adjustment_interval = 1`): [3](#0-2) 

For a fork that diverges at height D (fork's first block at D+1):
- Fork block at D+3: `prev = fork block at D+2`, `height_first = D+1` → `get_header_by_height(D+1)` returns the **main chain** block at D+1, not the fork's block at D+1.

The same structural bug exists in `bitcoin.rs` and `litecoin.rs` at their respective retarget boundaries: [4](#0-3) [5](#0-4) 

The difficulty check in `check_pow` then enforces `expected_bits == block_header.bits`: [6](#0-5) 

Because `expected_bits` is computed from the wrong ancestor's timestamp, a fork block with `bits` that would be rejected by the correct difficulty algorithm can pass this check.

### Impact Explanation

The contract's canonical chain state can be corrupted. A fork chain built with incorrect difficulty bits — bits that the real Dogecoin/Bitcoin/Litecoin network would reject — can be accepted by the contract. If that fork accumulates more `chain_work` than the main chain, `reorg_chain` promotes it: [7](#0-6) 

After the reorg, `mainchain_tip_blockhash` and `mainchain_height_to_header` reflect a chain that diverges from the real network's canonical chain. Any subsequent `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` call operates against this corrupted state, producing false inclusion proofs for transactions that are not confirmed on the real network.

### Likelihood Explanation

For **Dogecoin** (Digishield, per-block retarget), the bug activates on the **third fork block** (D+3) and every subsequent fork block. An attacker who can submit headers (as a trusted relayer, which the system is designed to allow via staking) can craft fork block timestamps to exploit the discrepancy between `time(fork_D+2) - time(main_D+1)` (buggy) vs. `time(fork_D+2) - time(fork_D+1)` (correct). By setting `fork_D+1.time` significantly later than `main_D+1.time` (up to the 2-hour future-time limit), the attacker reduces the buggy timespan, pushing `expected_bits` toward the minimum difficulty (easiest) bound. The attacker still needs valid PoW, but at a lower difficulty than the protocol requires. For **Bitcoin/Litecoin**, the bug only activates when a fork crosses a 2016-block retarget boundary, making it significantly harder to exploit in practice.

### Recommendation

Replace `get_header_by_height` calls in all three retarget functions with an ancestor walk that follows `prev_block_hash` links through `headers_pool` (i.e., `get_prev_header` in a loop) until reaching the block at `height_first`. This correctly resolves the retarget boundary block for both main-chain and fork-chain submissions, matching the reference implementations in Bitcoin Core, Litecoin, and Dogecoin.

### Proof of Concept

1. Deploy the Dogecoin variant of the contract (feature `dogecoin`).
2. Submit the genesis chain up to height D (main chain established).
3. Submit a fork block at D+1 with `time = main_D.time + 7200` (2 hours ahead, within the future-time limit).
4. Submit a fork block at D+2 with any valid timestamp.
5. For fork block D+3, compute `expected_bits` using `time(fork_D+2) - time(main_D+1)` (the buggy path). Because `time(fork_D+1)` was set 2 hours ahead of `main_D+1.time`, the buggy timespan is smaller, yielding a lower (easier) `expected_bits` than the correct computation `time(fork_D+2) - time(fork_D+1)` would produce.
6. Mine fork block D+3 with the easier `bits` value and valid PoW. Submit it. The contract accepts it because `expected_bits == block_header.bits` passes using the wrong ancestor.
7. Continue building the fork chain with the easier difficulty until `chain_work` exceeds the main chain, triggering `reorg_chain` and corrupting the canonical chain state.
