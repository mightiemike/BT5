Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End User, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` enforces the per-user allowlist against `sender`, which resolves to `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the end user. A pool admin who allowlists the router (the only way to make the router functional for allowlisted pools) inadvertently grants every unprivileged caller the ability to bypass the allowlist by routing through the public router, completely defeating the access-control invariant the extension is designed to enforce.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap` (lines 230–240 of `MetricOmmPool.sol`). `ExtensionCalling._beforeSwap` forwards this value unchanged as the first argument to every registered extension via `abi.encodeCall` (lines 160–176 of `ExtensionCalling.sol`). `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` directly (lines 72–80 of `MetricOmmSimpleRouter.sol`), so `sender = router`. If the pool admin has set `allowedSwapper[pool][router] = true`, the check passes for every caller regardless of their individual allowlist status. Without allowlisting the router, even individually allowlisted users cannot use the router (their call goes through the router, so `sender = router` fails the check), creating operational pressure on the admin to allowlist the router and activate the bypass. The `DepositAllowlistExtension` avoids this flaw by ignoring the first argument (the payer/caller) and checking `owner` — the position owner — which is the economically relevant identity and cannot be spoofed by an intermediary.

## Impact Explanation
A pool admin deploying a restricted pool (e.g., KYC-gated or institutional-only) and allowlisting the router so approved users can access the router UI inadvertently opens the pool to all callers. Any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` targeting the restricted pool; the allowlist check passes because `sender = router` is allowlisted. The swap executes at the live oracle price, allowing unauthorized counterparties to trade against pool liquidity. This is an admin-boundary break reachable by an unprivileged path: the pool admin's access-control configuration is silently nullified by a public, permissionless intermediary.

## Likelihood Explanation
The precondition — the pool admin allowlisting the router — is not a malicious or unusual action; it is the only way to make the router functional for an allowlisted pool. Without it, even approved users cannot use the router. The admin is therefore operationally compelled to set `allowedSwapper[pool][router] = true`. Once that single call is made, the bypass is immediately live and repeatable by any address with no further preconditions. The router is a public, permissionless contract with no caller restrictions.

## Recommendation
The extension must gate on the end user's identity, not the immediate pool caller. Two viable approaches:

1. **Standardized `extensionData` field**: Require the router to prepend the originating user address (`msg.sender` at the router entry point) to `extensionData`. The extension decodes and checks that address. This requires a coordinated convention between the router and the extension.

2. **Trusted-intermediary mapping**: Maintain a separate mapping for trusted intermediaries (routers). When `sender` is a trusted intermediary, decode the end user from `extensionData` and check the per-user allowlist against that decoded address.

The extension must never treat a public, permissionless intermediary contract as a sufficient identity for access control.

## Proof of Concept
```
Setup:
  pool = new MetricOmmPool(...)  // with SwapAllowlistExtension configured
  admin calls setAllowedToSwap(pool, router, true)   // to enable router for approved users
  admin calls setAllowedToSwap(pool, alice, true)    // alice is the only approved user
  // bob is NOT allowlisted

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      tokenIn: token0,
      zeroForOne: true,
      amountIn: 1_000e18,
      ...
  })

  → router calls pool.swap(recipient=bob, ...)
  → pool calls _beforeSwap(sender=router, ...)
  → SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][router] == true → PASSES
  → swap executes at oracle price
  → bob receives token1 output from the restricted pool
  → allowlist completely bypassed
```

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist only `alice` and the router, call `exactInputSingle` from `bob`, assert the swap succeeds and `bob` receives output tokens.
