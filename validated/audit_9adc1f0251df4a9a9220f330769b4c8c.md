### Title
`From<ChunkHeaderView> for ShardChunkHeader` silently downgrades V6 Spice inner to V3, producing a divergent chunk hash â€” (`core/primitives/src/views.rs`)

### Summary
When a `ShardChunkHeaderInner::V6` (Borsh discriminant 5, Spice tx-only) chunk is converted to `ChunkHeaderView` and back to `ShardChunkHeader`, the reverse conversion reconstructs a `ShardChunkHeaderInner::V3` (discriminant 2) because the match logic has no V6 arm and uses field presence as a proxy for version. The recomputed `chunk_hash` differs from the on-chain value.

### Finding Description
`ShardChunkHeaderInner::V6` has `has_proposed_split_field() = false`, `bandwidth_requests() = None`, and `congestion_info() = CongestionInfo::default()`. [1](#0-0) 

The forward conversion therefore produces `ChunkHeaderView` with `proposed_split = None`, `bandwidth_requests = None`, `congestion_info = Some(default)`. [2](#0-1) 

The reverse conversion matches `(None, None, Some(congestion_info))` and unconditionally reconstructs `ShardChunkHeaderInner::V3`: [3](#0-2) 

`header.init()` then recomputes the hash from the V3 inner bytes, producing a `ChunkHash` that differs from the original V6 chunk hash. [4](#0-3) 

The `TODO` comment at line 1174 acknowledges the design flaw explicitly: `// TODO: store the version in ChunkHeaderView instead of guessing it from which fields are populated.` [5](#0-4) 

### Impact Explanation
Any Rust code (external SDK consumer, indexer, bridge relay, or light-client verifier) that calls `ShardChunkHeader::from(chunk_header_view)` on a Spice chunk and then calls `.chunk_hash()` will obtain a `CryptoHash` that does not match the on-chain value. This breaks chunk identity verification for V6 chunks. The `chunk_hash` field stored in `ChunkHeaderView` is correct (it is copied verbatim from the original), so the mismatch is invisible in JSON but surfaces the moment the Rust `From` impl is used. [6](#0-5) 

### Likelihood Explanation
The bug is latent until `ProtocolFeature::Spice` is enabled and V6 chunks appear on-chain. Once Spice is live, any consumer using `From<ChunkHeaderView> for ShardChunkHeader` will silently compute wrong hashes. Nearcore's own production consensus path does not appear to use this conversion, so the immediate consensus risk is low; the risk is to external consumers and SDK users. [7](#0-6) 

### Recommendation
Add a `is_spice_chunk` flag or an explicit version discriminant to `ChunkHeaderView` (as the TODO at line 1174 suggests), and add a V6 arm to `From<ChunkHeaderView> for ShardChunkHeader` that reconstructs `ShardChunkHeaderInner::V6` when that flag is set. Alternatively, store the raw inner version number in `ChunkHeaderView` and use it directly in the reverse conversion.
