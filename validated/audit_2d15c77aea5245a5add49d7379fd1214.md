### Title
No Caller-Controlled Maximum Payment Bound in Liquidation — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

The `_handleLiquidationPayment` function in `ClearinghouseLiq.sol` computes the quote debit charged to the liquidator (`txn.sender`) entirely from the oracle price (`risk.priceX18`) read at execution time. The `LiquidateSubaccount` transaction struct contains no `maxPayment` or `maxPrice` field. Because the sequencer can process an `UpdatePrice` transaction between the moment the liquidator signs their transaction and the moment it is executed, the liquidator's quote balance can be debited by an amount substantially larger than what they agreed to when signing.

---

### Finding Description

The `LiquidateSubaccount` struct is defined as:

```solidity
struct LiquidateSubaccount {
    bytes32 sender;
    bytes32 liquidatee;
    uint32 productId;
    bool isEncodedSpread;
    int128 amount;
    uint64 nonce;
}
``` [1](#0-0) 

There is no `maxPayment`, `maxPrice`, or any price-bound field. The liquidator signs over `amount` (position size) but not over the quote cost they are willing to pay.

Inside `_handleLiquidationPayment`, for a spot liquidation the payment is computed as:

```solidity
(v.liquidationPriceX18, v.oraclePriceX18) = getLiqPriceX18(txn.productId, txn.amount);
v.liquidationPayment = v.liquidationPriceX18.mul(txn.amount);
v.liquidationFees    = (v.oraclePriceX18 - v.liquidationPriceX18).mul(LIQUIDATION_FEE_FRACTION).mul(txn.amount);
...
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -v.liquidationPayment - v.liquidationFees);
``` [2](#0-1) 

The same pattern applies to the perp path: [3](#0-2) 

`getLiqPriceX18` reads `risk.priceX18` directly from engine storage at call time:

```solidity
RiskHelper.Risk memory risk = IProductEngine(productToEngine[productId]).getRisk(productId);
...
return (risk.priceX18.mul(ONE + penaltyX18), risk.priceX18);
``` [4](#0-3) 

That storage value is written by `BaseEngine.updatePrice`, which is called by the clearinghouse whenever the sequencer processes an `UpdatePrice` transaction:

```solidity
function updatePrice(uint32 productId, int128 priceX18) external virtual {
    require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);
    _risk().value[productId].priceX18 = priceX18;
}
``` [5](#0-4) 

The sequencer processes `LiquidateSubaccount` and `UpdatePrice` transactions in the same `processTransactionImpl` pipeline: [6](#0-5) 

Because both transaction types flow through the same sequencer queue, a price update can be ordered immediately before a pending liquidation transaction, changing the oracle price the liquidator's payment is computed against — with no on-chain mechanism for the liquidator to reject the resulting debit.

---

### Impact Explanation

The liquidator's quote balance (`QUOTE_PRODUCT_ID`) is debited by `-v.liquidationPayment - v.liquidationFees`, both of which scale linearly with `risk.priceX18`. If the oracle price increases significantly between signing and execution, the liquidator pays far more quote than they intended for the same position size. There is no on-chain check that caps the debit to any liquidator-specified limit. The corrupted state is the liquidator's quote balance in `SpotEngine`.

---

### Likelihood Explanation

Oracle price updates are a routine, high-frequency operation in the Nado sequencer. Any liquidator operating on volatile assets faces this exposure on every liquidation transaction. A liquidator bot that signs a transaction at price P may have it executed at price P' >> P if a price update is sequenced first. This is not a theoretical edge case — it is the normal operating condition of the protocol.

---

### Recommendation

**Short term:** Add a `maxPayment` (or `maxPriceX18`) field to the `LiquidateSubaccount` struct and enforce it in `_handleLiquidationPayment`:

```solidity
require(
    v.liquidationPayment + v.liquidationFees <= txn.maxPayment,
    ERR_EXCEEDS_MAX_PAYMENT
);
```

**Long term:** As a general principle, any transaction whose quote cost is determined by a mutable oracle price at execution time should carry a caller-specified upper bound on that cost. This applies to all three liquidation paths (spread, spot, perp) in `_handleLiquidationPayment`.

---

### Proof of Concept

1. Oracle price for product P is 100 USDC. Liquidator Alice signs `LiquidateSubaccount { amount: 10, productId: P, ... }`, expecting to pay ~1000 USDC (plus penalty/fees).
2. Alice's signed transaction is submitted to the sequencer queue.
3. The sequencer processes an `UpdatePrice` transaction for product P, setting `risk.priceX18 = 200e18`.
4. The sequencer then processes Alice's `LiquidateSubaccount` transaction.
5. `getLiqPriceX18` returns a liquidation price derived from 200 USDC.
6. `spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -v.liquidationPayment - v.liquidationFees)` debits Alice ~2000+ USDC — double what she agreed to pay.
7. Alice has no recourse; the transaction succeeds and her quote balance is reduced by the higher amount. [7](#0-6)

    function getLiqPriceX18(uint32 productId, int128 amount)
        internal
        returns (int128, int128)
    {
        RiskHelper.Risk memory risk = IProductEngine(productToEngine[productId])
            .getRisk(productId);
        int128 penaltyX18 = (RiskHelper._getWeightX18(
            risk,
            amount,
            IProductEngine.HealthType.MAINTENANCE
        ) - ONE) / 5;
        if (penaltyX18.abs() < MIN_NON_SPREAD_LIQ_PENALTY_X18) {
            if (penaltyX18 < 0) {
                penaltyX18 = -MIN_NON_SPREAD_LIQ_PENALTY_X18;
            } else {
                penaltyX18 = MIN_NON_SPREAD_LIQ_PENALTY_X18;
            }
        }
        return (risk.priceX18.mul(ONE + penaltyX18), risk.priceX18);
```
