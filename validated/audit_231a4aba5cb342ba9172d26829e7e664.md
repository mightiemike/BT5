### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable Without Coinbase Validation, Enabling Merkle Proof Forgery — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR entry point despite being deprecated. It lacks the coinbase Merkle proof check that `verify_transaction_inclusion_v2` adds to prevent the 64-byte transaction Merkle proof forgery attack. Any unprivileged NEAR caller can invoke v1 directly, bypassing the guard entirely, and obtain a `true` result for a transaction that does not exist.

### Finding Description

The repository contains two public verification entry points:

- `verify_transaction_inclusion` (v1): only checks `#[pause]`, no coinbase validation.
- `verify_transaction_inclusion_v2`: validates a coinbase Merkle proof before delegating to v1. [1](#0-0) 

The `#[deprecated]` Rust attribute is a **compiler lint only**. It does not remove the function from the compiled WASM ABI or prevent any caller from invoking it at runtime. The function remains a fully reachable public entry point on-chain. [2](#0-1) 

`verify_transaction_inclusion_v2` adds the coinbase guard:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
``` [3](#0-2) 

This guard is absent in v1. The v1 function directly computes the Merkle root from the caller-supplied `tx_id` and `merkle_proof` and compares it to the stored `merkle_root`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [4](#0-3) 

`compute_root_from_merkle_proof` is a pure hash-chain computation with no structural constraint on what `tx_id` represents: [5](#0-4) 

Because Bitcoin Merkle trees hash 64-byte concatenations of child nodes, an internal node of the tree is itself a valid 32-byte hash. An attacker who knows the tree structure of a real block can supply an internal node as `tx_id` with a shortened `merkle_proof` path and obtain a `true` return value for a transaction that was never broadcast or mined.

The structural parallel to the reported issue is exact:

| Original report | This repository |
|---|---|
| `SwapperV2._executeSwaps()` has a dex whitelist | `verify_transaction_inclusion_v2` has coinbase validation |
| `Executor._executeSwaps()` has no whitelist | `verify_transaction_inclusion` (v1) has no coinbase validation |
| Both are callable; the unsafe one is not removed | Both are callable; the deprecated one is not removed |

### Impact Explanation

Any downstream NEAR contract or off-chain bridge that calls `verify_transaction_inclusion` (v1) — whether by mistake, by following outdated documentation, or because an attacker directs it there — will accept a forged proof as valid. A recipient contract that gates fund release or cross-chain state transitions on a `true` result from this function can be deceived into acting on a transaction that never existed on Bitcoin.

The contract's own warning acknowledges the class of attack:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [6](#0-5) 

### Likelihood Explanation

The entry point requires no privilege. Any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte Merkle forgery technique is publicly documented (BitMEX research, CVE-2017-12842). An attacker targeting a bridge or escrow contract that uses this light client only needs to identify which verification method the consumer calls and craft a valid-looking `ProofArgs` with an internal node as `tx_id`. [7](#0-6) 

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely. Because `#[deprecated]` does not restrict runtime access, the only effective fix is to make the function private or delete it. `verify_transaction_inclusion_v2` already delegates to v1 internally after performing the coinbase check, so the internal logic is preserved. Removing the public entry point closes the bypass path without any loss of functionality.

### Proof of Concept

1. Identify a real confirmed Bitcoin block whose header is stored in the contract (any block in `mainchain_header_to_height`).
2. Obtain the full transaction list for that block and compute its Merkle tree.
3. Select any internal node `N` at depth `d` from the root. Construct a `merkle_proof` of length `d` that walks from `N` up to the root.
4. Call `verify_transaction_inclusion` with:
   - `tx_id` = `N` (the internal node hash)
   - `tx_block_blockhash` = the real block hash
   - `tx_index` = the position consistent with the proof path
   - `merkle_proof` = the `d`-element path from `N` to the root
   - `confirmations` = any value ≤ `gc_threshold`
5. The function returns `true` because `compute_root_from_merkle_proof(N, index, proof) == merkle_root` holds by construction, yet no transaction with hash `N` exists. [8](#0-7) [5](#0-4)

        since = "0.5.0",

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
