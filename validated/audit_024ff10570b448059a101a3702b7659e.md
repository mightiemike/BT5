### Title
`_getMaxProductId()` Reverts on Empty `productIds` Array, Blocking All Health-Check-Dependent Operations — (`File: core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine._getMaxProductId()` accesses `productIds[productIds.length - 1]` without guarding against an empty array. Because `PerpEngine.initialize()` never pushes any product into `productIds`, the array starts empty. Any call to `getHealthContribution()` on the `PerpEngine` before the first perp product is registered will revert, blocking every protocol operation that depends on `Clearinghouse.getHealth()`.

---

### Finding Description

`BaseEngine._getMaxProductId()` is defined as:

```solidity
function _getMaxProductId() internal view returns (uint32) {
    return productIds[productIds.length - 1];
}
``` [1](#0-0) 

In Solidity 0.8+, `productIds.length - 1` when `productIds.length == 0` triggers a checked-arithmetic underflow and reverts immediately — before even reaching the array index.

`_getMaxProductId()` is called unconditionally at the top of `getHealthContribution()`:

```solidity
uint32 maxBitmapIndex = _getMaxProductId() / 256;
``` [2](#0-1) 

`SpotEngine.initialize()` pushes `QUOTE_PRODUCT_ID` into `productIds` during construction, so `SpotEngine` is never empty:

```solidity
productIds.push(QUOTE_PRODUCT_ID);
``` [3](#0-2) 

`PerpEngine.initialize()` does **not** push anything:

```solidity
function initialize(...) external {
    _initialize(_clearinghouse, _offchainExchange, _endpoint, _admin);
}
``` [4](#0-3) 

So `PerpEngine.productIds` is empty from deployment until the first `addOrUpdateProduct` (which is `onlyOwner`). Notably, the developer was already aware of the empty-array case in `_addOrUpdateProduct`, which correctly guards:

```solidity
if (productIds.length == 0 || productId > productIds[productIds.length - 1]) {
``` [5](#0-4) 

…but `_getMaxProductId()` has no equivalent guard.

---

### Impact Explanation

`Clearinghouse.getHealth()` calls `perpEngine.getHealthContribution()` unconditionally:

```solidity
health += perpEngine.getHealthContribution(subaccount, healthType);
``` [6](#0-5) 

Every protocol operation that calls `getHealth()` will revert while `PerpEngine.productIds` is empty. Affected operations include:

- `withdrawCollateral` — `require(getHealth(sender, healthType) >= 0, ...)` [7](#0-6) 
- `transferQuote` — via `_isAboveInitial` [8](#0-7) 
- `mintNlp` / `burnNlp` — health checks at lines 480 and 527 [9](#0-8) 
- `liquidateSubaccount` — via `ClearinghouseLiq` which delegates back to `getHealth`

No funds are permanently lost, but all health-check-dependent operations are completely blocked during the initialization window. A user who deposits collateral before any perp product is registered cannot withdraw until an admin adds a perp product.

---

### Likelihood Explanation

The window exists between `PerpEngine` deployment and the first `addOrUpdateProduct` call. Since `depositCollateral` does **not** call `getHealth`, a user can deposit into the protocol during this window and then be unable to withdraw. The likelihood is low-to-medium: it requires the protocol to accept user interactions before perp products are configured, which is a realistic deployment sequencing mistake.

---

### Recommendation

Add an empty-array guard in `_getMaxProductId()`, consistent with the pattern already used in `_addOrUpdateProduct`:

```diff
function _getMaxProductId() internal view returns (uint32) {
+   if (productIds.length == 0) return 0;
    return productIds[productIds.length - 1];
}
```

Alternatively, guard `getHealthContribution()` itself:

```diff
function getHealthContribution(...) public returns (int128 health) {
+   if (productIds.length == 0) return 0;
    uint32 maxBitmapIndex = _getMaxProductId() / 256;
    ...
}
```

---

### Proof of Concept

1. Deploy `Clearinghouse`, `SpotEngine`, `PerpEngine` via `addEngine` (which calls `PerpEngine.initialize()` — `productIds` is now empty).
2. A user calls `depositCollateral` — succeeds (no health check).
3. Before any owner calls `PerpEngine.addOrUpdateProduct(...)`, the user calls `withdrawCollateral`.
4. `withdrawCollateral` → `getHealth` → `perpEngine.getHealthContribution` → `_getMaxProductId()` → `productIds.length - 1` underflows → **revert**.
5. The user's funds are locked until an admin registers at least one perp product.
