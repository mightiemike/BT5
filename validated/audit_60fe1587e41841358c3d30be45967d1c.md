### Title
When `ExitDuration` is decreased by governance, positions that already triggered exit must still wait the old (longer) `ExitUnlockAt` — (`x/tieredrewards/keeper/msg_server.go`, `x/tieredrewards/keeper/msg_server_auth.go`)

---

### Summary

`MsgTriggerExitFromTier` snapshots `ExitUnlockAt = blockTime + tier.ExitDuration` into the `Position` record at trigger time. `MsgUpdateTier` updates only the `Tier` store entry and never back-fills existing positions. When governance decreases `ExitDuration`, every position that already triggered exit retains the old, longer `ExitUnlockAt` and cannot undelegate or exit until that stale timestamp elapses — even though the current protocol parameter is shorter.

---

### Finding Description

`TriggerExitFromTier` reads the live tier at call time and writes the computed deadline into the position:

```go
// x/tieredrewards/keeper/msg_server.go:361-367
tier, err := ms.getTier(ctx, pos.TierId)
...
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
```

`TriggerExit` stores the absolute timestamp:

```go
// x/tieredrewards/types/position.go:71-74
func (p *Position) TriggerExit(blockTime time.Time, duration time.Duration) {
    p.ExitTriggeredAt = blockTime
    p.ExitUnlockAt = blockTime.Add(duration)
}
```

`UpdateTier` only replaces the tier record; it never iterates positions to update their `ExitUnlockAt`:

```go
// x/tieredrewards/keeper/msg_server_auth.go:57-81
func (ms msgServer) UpdateTier(...) {
    ...
    if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
        ms.claimRewardsAndUpdateTierPositions(...)  // only for APY changes
    }
    ms.SetTier(ctx, msg.Tier)   // ExitDuration change: no position update
}
```

Both exit paths (`MsgTierUndelegate` and `MsgExitTierWithDelegation`) gate on the stored `ExitUnlockAt` via `CompletedExitLockDuration`:

```go
// x/tieredrewards/keeper/msg_validate.go:57-60
if !pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
    return types.ErrExitLockDurationNotReached
}
```

```go
// x/tieredrewards/types/position.go:61-63
func (p Position) CompletedExitLockDuration(blockTime time.Time) bool {
    return !p.ExitUnlockAt.IsZero() && !blockTime.Before(p.ExitUnlockAt)
}
```

The same stale-timestamp problem applies in the opposite direction: if governance **increases** `ExitDuration`, positions that triggered exit before the change retain the old, shorter `ExitUnlockAt` and can exit sooner than the new governance-mandated lock period — bypassing the updated security parameter.

---

### Impact Explanation

**Decrease direction (UX harm):** A user who triggered exit when `ExitDuration = 1 year` and governance later shortens it to `2 weeks` must still wait the full year. Their tokens remain locked in the tier module far longer than the current protocol parameter mandates.

**Increase direction (security bypass):** A user who triggered exit when `ExitDuration = 2 weeks` and governance later extends it to `1 year` can still exit after 2 weeks. The new, longer lock period — potentially enacted to prevent rapid capital flight or protect the protocol — is silently bypassed for all positions that triggered exit before the change.

The corrupted invariant in both cases is `position.ExitUnlockAt`, a persisted timestamp that diverges from `ExitTriggeredAt + currentTier.ExitDuration` after any governance update to `ExitDuration`.

---

### Likelihood Explanation

`ExitDuration` is a normal governance-managed field on `Tier`. The integration test suite already exercises `MsgUpdateTier` with `exit_duration` changes and explicitly notes "the new exit_duration applies to new positions" — confirming the stale-position behavior is present and untested for existing positions. Any governance proposal that adjusts `ExitDuration` triggers the issue for all positions that have already called `TriggerExitFromTier`.

---

### Recommendation

When `UpdateTier` changes `ExitDuration`, iterate all positions in the tier that have a pending exit (`HasTriggeredExit() && !CompletedExitLockDuration(blockTime)`) and recompute their `ExitUnlockAt` as `ExitTriggeredAt + newExitDuration`. This mirrors the existing `claimRewardsAndUpdateTierPositions` pattern used for APY changes. Alternatively, remove the stored `ExitUnlockAt` field and compute the deadline lazily as `pos.ExitTriggeredAt + currentTier.ExitDuration` at validation time, so it always reflects the live parameter.

---

### Proof of Concept

1. Governance sets `Tier 1` `ExitDuration = 1 year` via `MsgUpdateTier`.
2. Alice calls `MsgTriggerExitFromTier` on her position → `ExitUnlockAt = now + 1 year` stored.
3. Governance shortens `Tier 1` `ExitDuration = 2 weeks` via `MsgUpdateTier`. Alice's position is untouched.
4. Bob calls `MsgTriggerExitFromTier` → `ExitUnlockAt = now + 2 weeks` stored.
5. After 3 weeks: Bob calls `MsgTierUndelegate` → succeeds (`CompletedExitLockDuration` passes).
6. After 3 weeks: Alice calls `MsgTierUndelegate` → reverts with `ErrExitLockDurationNotReached` because her stored `ExitUnlockAt` is still ~11 months in the future.

Alice must wait ~11 more months despite the current protocol parameter being 2 weeks.

**Inverse PoC (security bypass):**

1. `ExitDuration = 2 weeks`. Alice triggers exit → `ExitUnlockAt = now + 2 weeks`.
2. Governance increases `ExitDuration = 1 year`.

```go
func (ms msgServer) UpdateTier(ctx context.Context, msg *types.MsgUpdateTier) (*types.MsgUpdateTierResponse, error) {
	if err := ms.requireAuthority(msg.Authority); err != nil {
		return nil, err
	}

	oldTier, err := ms.getTier(ctx, msg.Tier.Id)
	if err != nil {
		return nil, err
	}

	if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
		if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
			return nil, err
		}
	}

	if err := ms.SetTier(ctx, msg.Tier); err != nil {
		return nil, err
	}

	if err := ms.emitTierChangedEvent(ctx, types.TierChangeAction_TIER_CHANGE_ACTION_UPDATE, msg.Tier); err != nil {
		return nil, err
	}

	return &types.MsgUpdateTierResponse{}, nil
```
