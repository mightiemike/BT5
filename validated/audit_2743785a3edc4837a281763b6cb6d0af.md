### Title
`DepositAllowlistExtension` Checks Position `owner` Instead of Actual Payer `sender`, Allowing Unauthorized Depositors to Bypass the Guard - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` is documented as gating `addLiquidity` by **depositor** address, but it silently discards the `sender` parameter (the actual token payer) and validates only `owner` (the position beneficiary). Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender ≠ owner`, any address not on the allowlist can bypass the guard by supplying an allowed owner.

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct actors to the extension hook:

```
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the operator who triggers the call and pays tokens through the modify-liquidity callback; `owner` is the address whose position is credited. The pool enforces no constraint that they be equal. [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both as `(sender, owner, ...)` to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but discards it (unnamed `address,`), then checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

The contract's own NatSpec states it "Gates `addLiquidity` by depositor address", and its admin API is named `setAllowedToDeposit`/`allowedDepositor` — all pointing to the payer as the intended subject. The implementation checks the wrong actor.

### Impact Explanation

Any address not on the allowlist can call `pool.addLiquidity(allowedOwner, salt, deltas, ...)`, pay tokens through the callback, and have the position credited to `allowedOwner`. The `DepositAllowlistExtension` guard passes because it only validates `allowedOwner`, which is on the allowlist. The deposit restriction is completely nullified: every pool that relies solely on this extension for LP access control is open to any depositor who can identify one allowed owner address (trivially discoverable from on-chain events). This is an admin-boundary break — an unprivileged actor bypasses a pool-admin-configured access control gate.

### Likelihood Explanation

Likelihood is high. The operator pattern (`msg.sender ≠ owner`) is a first-class, documented feature of the pool. Allowed owner addresses are publicly visible from `LiquidityAdded` events. No special privilege, flash loan, or oracle manipulation is required — a standard `addLiquidity` call suffices.

### Recommendation

Replace the discarded first parameter with a named `sender` and validate it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict both the payer and the position owner, both should be checked independently.

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` configured on `beforeAddLiquidity`.
2. Admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is **not** added.
3. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(Bob, alice, ...)` → extension receives `sender=Bob, owner=alice`.
5. Extension discards `Bob`, checks `allowedDepositor[pool][alice]` → `true` → no revert.
6. Bob's tokens are pulled via the modify-liquidity callback; Alice's position is credited.
7. Bob has successfully deposited despite being absent from the allowlist. [3](#0-2) [4](#0-3) [2](#0-1)
