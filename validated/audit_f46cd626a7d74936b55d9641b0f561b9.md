Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any User to Bypass Per-User Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool populates with `msg.sender` of the `pool.swap()` call — the direct caller, not the originating EOA. When swaps are routed through `MetricOmmSimpleRouter`, the direct caller is the router contract. If the pool admin allowlists the router (required for allowlisted users to use multi-hop routing), every unprivileged user can bypass the per-user restriction by routing through the same public router.

## Finding Description

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

This means the `sender` value received by the extension is the direct caller of `pool.swap()`. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly: [2](#0-1) 

So from the pool's perspective, `msg.sender = router`, and the extension receives `sender = router`. The allowlist check in `beforeSwap` then evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`: [3](#0-2) 

The `allowedSwapper` mapping is keyed by `(pool, swapper)`: [4](#0-3) 

There is no configuration that achieves "only userA can swap, including via the router." If the admin allowlists the router to enable router-mediated swaps for userA, every user gains the same access. If the admin does not allowlist the router, userA cannot use the router at all.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to trusted counterparties (e.g., a KYC-gated pool, a private institutional pool) has its allowlist fully bypassed. Any unprivileged user can execute swaps against the pool's liquidity by routing through `MetricOmmSimpleRouter`, enabling unauthorized price-taking against LP positions and draining LP principal and fees beyond what the pool admin intended to permit. This is a direct loss of user (LP) principal, meeting the Critical/High threshold.

## Likelihood Explanation

`MetricOmmSimpleRouter` is a public, permissionless contract callable by anyone. The only precondition is that the pool admin has allowlisted the router address — a natural and expected action for any pool that wants its allowlisted users to access multi-hop routing. The `isAllowedToSwap` view function does not expose this implication: [5](#0-4) 

The admin has no way to know that allowlisting the router opens the pool to all users. The attack is repeatable and requires no special privileges.

## Recommendation

The pool should forward the originating sender to the extension rather than `msg.sender`. Add an explicit `sender` parameter to `pool.swap()` that the pool validates (e.g., must equal `msg.sender` for direct calls, or a forwarded address for trusted routers). The router would then pass the original `msg.sender` as the forwarded sender, and the extension would check that value. Alternatively, redesign `SwapAllowlistExtension` to check `tx.origin` as a fallback when the direct caller is a known router, though this introduces its own risks. The cleanest fix is a forwarded-sender pattern.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension as extension1, beforeSwap order = extension1
  - Pool admin calls setAllowedToSwap(pool, userA, true)
  - Pool admin calls setAllowedToSwap(pool, router, true)   ← required for userA to use router
  - Add liquidity to the pool

Attack:
  - userB (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(pool, userB, ...)
  - Router calls pool.swap(userB, ...)  →  msg.sender = router
  - Pool calls extension.beforeSwap(router, userB, ...)
  - Extension checks allowedSwapper[pool][router] == true  →  passes
  - userB's swap executes against LP funds

Expected: revert NotAllowedToSwap
Actual:   swap succeeds, LP funds accessible to unauthorized user
``` [6](#0-5) [7](#0-6)
