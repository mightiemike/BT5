### Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on LP-position `owner` instead of the actual depositor `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the address that actually calls `addLiquidity` and pays the tokens) and instead checks the caller-controlled `owner` argument (the address that will receive the LP shares). Any address not on the allowlist can bypass the gate by specifying an allowlisted address as `owner`, paying tokens themselves while the allowlisted address receives the LP position.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` parameter and passes both `msg.sender` (as `sender`) and `owner` to the extension hook:

```solidity
// MetricOmmPool.sol
function addLiquidity(
    address owner,          // caller-controlled
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) ... {
    ...
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    ...
}
```

`ExtensionCalling._beforeAddLiquidity` forwards both arguments verbatim to the extension:

```solidity
// ExtensionCalling.sol
_callExtensionsInOrder(
    BEFORE_ADD_LIQUIDITY_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but names it `_` (discards it) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper), not `recipient`. The deposit extension is inconsistent: it checks the LP-position recipient (`owner`) rather than the actual depositor (`sender`).

Because `owner` is freely chosen by the caller, any address can pass the allowlist check by supplying any allowlisted address as `owner`. The pool's callback then pulls tokens from the actual caller (`msg.sender` of `addLiquidity`), so the unauthorized caller pays the tokens while the allowlisted address receives the LP shares.

Note also that the override drops the `onlyPool` modifier present on the base-class stub, meaning the function is callable by any address — but the primary exploit path runs through the pool itself via the `owner` parameter, not through a direct call.

---

### Impact Explanation

The deposit allowlist is the pool admin's mechanism to restrict who may provide liquidity (e.g., KYC enforcement, regulatory compliance, or pool-composition control). Bypassing it allows any unprivileged address to inject liquidity into a restricted pool. Consequences include:

- Unauthorized liquidity provision into pools intended to be restricted.
- Griefing of the allowlisted `owner` address, which receives LP shares it did not request and cannot easily refuse.
- If the pool also uses `OracleValueStopLossExtension`, the injected liquidity alters per-bin token balances and per-share metrics, potentially resetting or distorting watermarks and undermining the stop-loss guard.

---

### Likelihood Explanation

The attack requires only knowledge of one allowlisted address (publicly readable from `allowedDepositor`) and willingness to pay the deposited tokens. No special role, flash loan, or privileged access is needed. The pool's `addLiquidity` is a public entry point.

---

### Recommendation

Check `sender` (the actual depositor) instead of `owner` (the LP-position recipient), consistent with how `SwapAllowlistExtension` gates on `sender`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    onlyPool   // restore the base-class guard
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

---

### Proof of Concept

1. Deploy a pool with `DepositAllowlistExtension`.
2. Pool admin calls `setAllowedToDeposit(pool, Alice, true)` — Alice is allowlisted, Bob is not.
3. Bob calls `pool.addLiquidity(owner = Alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(sender = Bob, owner = Alice, ...)`.
5. Extension evaluates `allowedDepositor[pool][Alice]` → `true`; check passes.
6. `LiquidityLib.addLiquidity` mints LP shares to Alice.
7. The pool's callback pulls tokens from Bob (the `msg.sender` of `addLiquidity`).
8. Result: Bob (not on the allowlist) has deposited tokens into the restricted pool; Alice holds LP shares she did not request. The deposit allowlist is fully bypassed.
