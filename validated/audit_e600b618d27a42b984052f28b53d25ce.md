### Title
Swapped Weight Variable References in `_getSpreadWeightX18` Corrupt Spread Position Health Calculations — (`core/contracts/libraries/RiskHelper.sol`)

### Summary

`RiskHelper._getSpreadWeightX18` uses the wrong leg's `longWeight` in each branch of its conditional. When the spot position is long (and the perp is short), the function uses `perpCoreRisk.longWeight` instead of `spotCoreRisk.longWeight`, and vice versa. This is a direct analog to the external report's copy-paste error: the wrong variable is referenced in each conditional branch, corrupting the spread health adjustment applied in `Clearinghouse.getHealth`.

---

### Finding Description

In `RiskHelper._getSpreadWeightX18`, the spread weight is computed as:

```solidity
if (spotCoreRisk.amount > 0) {
    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
} else {
    spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
}
``` [1](#0-0) 

The condition `spotCoreRisk.amount > 0` identifies a **long spot / short perp** spread. In this case the long leg is the spot, so the spread weight should be derived from `spotCoreRisk.longWeight`. Instead, the code uses `perpCoreRisk.longWeight` — the weight of the short leg. The `else` branch (short spot / long perp) makes the symmetric mistake: it uses `spotCoreRisk.longWeight` (the short leg) instead of `perpCoreRisk.longWeight` (the long leg).

`CoreRisk.longWeight` is always the **long** weight of the product, computed unconditionally with `amount = 1` in `getCoreRisk`:

```solidity
return IProductEngine.CoreRisk(
    amount,
    risk.priceX18,
    RiskHelper._getWeightX18(risk, 1, healthType)
);
``` [2](#0-1) 

So in both branches, `_getSpreadWeightX18` always uses the long weight of the **short** leg rather than the long weight of the **long** leg.

The result feeds directly into the spread health adjustment in `Clearinghouse.getHealth`:

```solidity
int128 existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2;
int128 spreadWeight = RiskHelper._getSpreadWeightX18(perpCoreRisk, spotCoreRisk, healthType);
health += basisAmount
    .mul(spotCoreRisk.price + perpCoreRisk.price)
    .mul(spreadWeight - existingWeight);
``` [3](#0-2) 

---

### Impact Explanation

The formula `ONE - (ONE - weight) / 5` maps a lower weight to a lower spread weight. When `spotLongWeight ≠ perpLongWeight`:

- **Long spot / short perp** (`spotCoreRisk.amount > 0`): if `perpLongWeight > spotLongWeight`, the current code produces a *higher* `spreadWeight` than correct, inflating the health bonus. A trader holding a long-spot/short-perp spread is credited with more health than they deserve, enabling undercollateralized positions that bypass initial margin checks.

- **Short spot / long perp** (`spotCoreRisk.amount ≤ 0`): the same asymmetry produces a *lower* `spreadWeight` than correct, deflating the health bonus. A legitimately healthy spread position may be incorrectly flagged as under-maintenance, triggering wrongful liquidation.

Both paths flow through `Clearinghouse.getHealth`, which is called on every deposit, withdrawal, trade settlement, and liquidation check. [4](#0-3) 

---

### Likelihood Explanation

Any user who holds a spread position (long spot + short perp, or short spot + long perp) across two products whose `longWeightInitial` or `longWeightMaintenance` values differ will be affected on every health evaluation. Because different products routinely carry different risk weights, this condition is met in normal protocol operation. No special permissions or unusual inputs are required — a standard matched trade that creates a spread position is sufficient.

---

### Recommendation

Swap the weight references so each branch uses the long weight of the **long** leg:

```diff
 if (spotCoreRisk.amount > 0) {
-    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
+    spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
 } else {
-    spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
+    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
 }
``` [1](#0-0) 

---

### Proof of Concept

Assume two products with asymmetric weights: `spotLongWeightInitialX18 = 0.8e18`, `perpLongWeightInitialX18 = 0.9e18`. A user holds 10 units long spot / 10 units short perp at price 1e18 each.

```
basisAmount = 10
existingWeight = (0.8e18 + 0.9e18) / 2 = 0.85e18

// Current (buggy) code — uses perpLongWeight for long-spot branch:
spreadWeight = 1e18 - (1e18 - 0.9e18) / 5 = 0.98e18
healthDelta = 10 * 2e18 * (0.98e18 - 0.85e18) = 10 * 2e18 * 0.13e18  ← inflated

// Correct code — uses spotLongWeight for long-spot branch:
spreadWeight = 1e18 - (1e18 - 0.8e18) / 5 = 0.96e18
healthDelta = 10 * 2e18 * (0.96e18 - 0.85e18) = 10 * 2e18 * 0.11e18  ← correct
```

The buggy path overstates the health contribution by ~18% relative to the correct value, allowing the user to open additional positions beyond what their collateral should permit. The symmetric error in the short-spot branch understates health by the same proportion, exposing a legitimately solvent spread holder to wrongful liquidation.
