### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Callers to Bypass the Deposit Gate — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension.beforeAddLiquidity` hook silently drops the `sender` parameter and validates the `owner` (position beneficiary) instead. Because `addLiquidity` accepts a caller-supplied `owner` address, any non-allowlisted address can bypass the gate by nominating an allowlisted address as `owner`, while the actual depositor (the `sender`) is never checked.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument to `_beforeAddLiquidity`:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The `DepositAllowlistExtension` override receives `(address sender, address owner, ...)` but discards `sender` entirely, binding only `owner` to a named variable:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

The admin-facing setter is named `setAllowedToDeposit` and its parameter is `depositor`, making the intent unambiguous: the allowlist is meant to gate the *caller*, not the position beneficiary. [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly binds and checks `sender`:

```solidity
// SwapAllowlistExtension.sol lines 31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [4](#0-3) 

The asymmetry between the two sibling extensions confirms the deposit check is applied to the wrong actor.

---

### Impact Explanation

**Access-control bypass with fund-level consequences:**

1. A non-allowlisted address calls `pool.addLiquidity(allowlistedOwner, salt, deltas, ...)`.
2. The extension checks `allowedDepositor[pool][allowlistedOwner]` → `true`; the gate passes.
3. The pool executes the callback on `msg.sender` (the non-allowlisted caller), pulling tokens from them.
4. The position is recorded under `allowlistedOwner`.
5. `allowlistedOwner` later calls `removeLiquidity` (the only address permitted to do so, per `msg.sender != owner` check) and returns the proceeds off-chain. [5](#0-4) 

This colluding pair can repeatedly deposit into a permissioned pool (e.g., a KYC-gated or regulated pool) without the non-allowlisted party ever appearing in the allowlist. The pool admin's access-control invariant — that only allowlisted addresses may supply liquidity — is broken. Pool liquidity depth and bin state are altered by an unauthorized actor, which can affect swap pricing and LP share dilution for existing, legitimately allowlisted LPs.

Additionally, the inverse failure exists: an allowlisted `sender` who specifies a non-allowlisted `owner` is incorrectly blocked, causing a DoS for legitimate depositors.

---

### Likelihood Explanation

- The bypass requires no special privilege: any EOA or contract can call `addLiquidity` with an arbitrary `owner`.
- Finding a single allowlisted address (e.g., the pool admin itself, which is always known) is trivial.
- The collusion step (owner returning funds) is a simple off-chain arrangement.
- The bug is present in every pool that deploys `DepositAllowlistExtension` with a non-empty allowlist.

---

### Recommendation

Bind and check `sender` (the actual caller) instead of `owner`, mirroring the `SwapAllowlistExtension` pattern:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1)

### Proof of Concept

```
Setup:
  pool deployed with DepositAllowlistExtension
  allowedDepositor[pool][ALICE] = true   // ALICE is allowlisted
  BOB is NOT allowlisted

Attack:
  BOB calls pool.addLiquidity(
      owner    = ALICE,   // allowlisted → check passes
      salt     = 0,
      deltas   = <desired bins/shares>,
      callback = BOB's contract (provides tokens)
  )

Result:
  - Extension checks allowedDepositor[pool][ALICE] → true → no revert
  - Pool calls BOB.metricOmmSwapCallback → BOB transfers tokens to pool
  - Position (ALICE, 0) is credited with shares
  - ALICE calls removeLiquidity → receives tokens → returns to BOB off-chain
  - BOB has effectively deposited into a permissioned pool without being allowlisted
``` [2](#0-1) [6](#0-5)
