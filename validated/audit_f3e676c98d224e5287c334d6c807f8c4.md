Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates Router Address Instead of End-User, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is the `msg.sender` of the `pool.swap()` call — the router's address when a user routes through `MetricOmmSimpleRouter`. Because the router does not forward the original user's identity to the pool or extension, the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][end_user]`. Any pool admin who allowlists the router to support router-mediated swaps for legitimate users simultaneously grants every blocked user the ability to bypass the per-user allowlist by routing through the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap` (L230–240 of `MetricOmmPool.sol`). `ExtensionCalling._beforeSwap` forwards this value unchanged as the first argument to each configured extension hook (L149–177 of `ExtensionCalling.sol`). `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router executes:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The original user's address is stored in transient storage only for the payment callback. The router calls `pool.swap()` with `msg.sender = router`, so the pool passes the router's address as `sender` to the extension. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

Existing guards are insufficient: there is no mechanism in the pool, extension interface, or router to distinguish the original user from the intermediary. The `extensionData` field is user-controlled and not authenticated, so it cannot serve as a trusted attestation channel without additional protocol changes.

## Impact Explanation
Any user explicitly blocked by `allowedSwapper[pool][user] = false` can bypass the restriction by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) instead of calling `pool.swap` directly, provided the router is allowlisted. The pool's curated access control is completely nullified for router-mediated swaps. Depending on the pool's restriction rationale (KYC compliance, whitelist-only LP protection, adversarial-flow prevention), this allows disallowed counterparties to drain LP value through adverse selection or to interact with a pool that was never intended to be public — constituting a broken core pool access-control mechanism with direct potential for loss of LP assets.

## Likelihood Explanation
The bypass requires the pool admin to have allowlisted the router. Any pool admin who wants allowlisted users to be able to use the primary user-facing swap interface (the router) is forced to allowlist the router — there is no alternative configuration. Any pool intended to be accessible via the router while also restricting specific users will inevitably reach this state. The bypass is triggered by a normal, unprivileged user action (calling a public router function) with no special permissions or capital requirements.

## Recommendation
The extension must check the economically relevant actor — the end user — not the intermediary. Two approaches:

1. **Pass the original caller through the pool**: Have `MetricOmmPool.swap` record `msg.sender` at entry and forward it as a distinct `originator` field to extension hooks, separate from `sender` (the direct caller). Extensions can then gate on `originator`.

2. **Extension-data attestation with trusted attestor**: Require the router to embed the original user's address in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it, with the pool or factory as the trusted attestor. This avoids core changes but requires the extension to authenticate the source of the attestation.

Until fixed, pool admins must be warned that `SwapAllowlistExtension` cannot enforce per-user restrictions for router-mediated swaps and should not allowlist the router on curated pools.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured as the `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — Alice is allowed.
3. Pool admin calls `setAllowedToSwap(pool, bob, false)` — Bob is explicitly blocked.
4. Pool admin calls `setAllowedToSwap(pool, router, true)` — Router is allowlisted so Alice can use it.
5. Bob calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: bob, ...})`.
6. Router calls `pool.swap(bob, ...)` with `msg.sender = router`.
7. Pool calls `ExtensionCalling._beforeSwap(router, bob, ...)`.
8. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap succeeds.
9. Bob has bypassed the allowlist and executed a swap on the curated pool.

Foundry test plan: deploy `SwapAllowlistExtension`, configure a pool with it, set `allowedSwapper[pool][router] = true` and `allowedSwapper[pool][bob] = false`, then call `MetricOmmSimpleRouter.exactInputSingle` from Bob's address and assert the swap succeeds (no revert), confirming the bypass.
