### Title
Cross-Leg Timestamp Discrepancy in Synthetic Ratio Mode Allows Stale Price to Reach Pool Swaps - (File: smart-contracts-poc/contracts/AnchoredPriceProvider.sol)

### Summary
`AnchoredPriceProvider._getBidAndAskPrice()` computes a synthetic ratio by dividing two independently-fetched oracle prices (base ÷ quote). Each leg is individually validated for staleness against `MAX_REF_STALENESS`, but the two legs' `refTime` values are never compared against each other. This allows the synthetic ratio to be computed from prices at significantly different timestamps, producing an inaccurate bid/ask that reaches live pool swaps.

### Finding Description
When `quoteFeedId != bytes32(0)`, `_getBidAndAskPrice()` calls `_readLeg(baseFeedId)` and `_readLeg(quoteFeedId)` independently. `_readLeg` returns four values including `refTime`, but in both call sites the `refTime` return value is silently discarded via the blank slot in the tuple destructuring:

```solidity
(uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
// ...
(uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
``` [1](#0-0) 

The only staleness check is per-leg inside `_readLeg`: each `refTime` must be within `MAX_REF_STALENESS` of `block.timestamp`. [2](#0-1) 

There is no check that the two `refTime` values are close to each other. The base feed can have `refTime = block.timestamp` (fresh) while the quote feed has `refTime = block.timestamp - MAX_REF_STALENESS` (at the staleness boundary but still individually valid). The synthetic ratio `mid1 / mid2` is then computed from prices that are up to `MAX_REF_STALENESS` apart in time.

The resulting (bid, ask) pair is returned to the pool's `_getBidAndAskPriceX64()` and used directly to execute the swap: [3](#0-2) 

No downstream guard in `_computeBidAsk` catches a cross-leg time skew — it only checks `MAX_SPREAD_BPS` and band-width invariants: [4](#0-3) 

### Impact Explanation
During volatile market conditions, a `MAX_REF_STALENESS`-second discrepancy between the two feeds produces a synthetic ratio that is materially off from the true market price. For example, if the base asset (BTC/USD) is fresh but the quote asset (ETH/USD) is stale and ETH has moved 10% in the interim, the computed BTC/ETH ratio is off by ~10%. This inaccurate bid/ask reaches live pool swaps:

- If the ratio is inflated (stale quote underprices the denominator asset), traders can sell the base asset at an above-market price, extracting value directly from LP reserves.
- If the ratio is deflated, traders receive less than fair value, but the pool's LP claims are still mispriced relative to the true market.

This is a direct loss of LP principal — the exact "swap conservation failure / bad-price execution" impact class listed in scope.

### Likelihood Explanation
This is reachable whenever:
1. A pool is configured with `AnchoredPriceProvider` in synthetic ratio mode (`quoteFeedId != bytes32(0)`), e.g., BTC/ETH = BTC/USD ÷ ETH/USD.
2. The two underlying feeds update at different rates or one is temporarily delayed (common during network congestion, oracle keeper delays, or RWA market-hours gaps — the Chainlink oracle already writes `spreadBps = ORACLE_BPS` for off-hours, but a delayed-but-not-stalled feed passes the per-leg check).
3. A public swap is executed during this window.

No privileged access is required. Any public swap on such a pool triggers this path.

### Recommendation
After both legs pass their individual staleness checks, compare their `refTime` values and fail closed if the difference exceeds a configured cross-leg threshold (e.g., `MAX_LEG_TIME_DELTA`):

```solidity
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spreadBps, uint256 refTime1, bool ok) = _readLeg(baseFeedId);
    if (!ok) return (0, type(uint128).max);

    bytes32 _quote = quoteFeedId;
    if (_quote != bytes32(0)) {
        (uint256 mid2, uint256 spreadBps2, uint256 refTime2, bool ok2) = _readLeg(_quote);
        if (!ok2 || mid2 == 0) return (0, type(uint128).max);

        // NEW: cross-leg timestamp alignment check
        uint256 timeDelta = refTime1 > refTime2 ? refTime1 - refTime2 : refTime2 - refTime1;
        if (timeDelta > MAX_LEG_TIME_DELTA) return (0, type(uint128).max);

        mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
        spreadBps += spreadBps2;
    }
    return _computeBidAsk(mid, spreadBps);
}
```

`MAX_LEG_TIME_DELTA` should be set to a value significantly smaller than `MAX_REF_STALENESS` — for example, the expected maximum update interval of the slower feed.

### Proof of Concept
1. Deploy a pool with `AnchoredPriceProvider` configured with `baseFeedId = BTC/USD` and `quoteFeedId = ETH/USD`. Set `MAX_REF_STALENESS = 3600` (1 hour).
2. At `T = block.timestamp`, update BTC/USD to `$100,000` with `refTime = T`.
3. At `T`, the ETH/USD feed has `refTime = T - 3600` (1 hour old, at the staleness boundary) with price `$2,000`. True ETH price at `T` is `$2,500`.
4. Call `getBidAndAskPrice()`. Both legs pass `_isStale` individually. `refTime` values are discarded.
5. Computed synthetic ratio: `100,000 / 2,000 = 50 BTC/ETH`. True ratio: `100,000 / 2,500 = 40 BTC/ETH`. The ratio is inflated by 25%.
6. The pool quotes bid/ask around 50 BTC/ETH. A trader sells BTC into the pool at this inflated price, receiving 25% more ETH than the true market rate. LP reserves are drained by the difference.

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }
