Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Real Swapper, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` which equals `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the router contract is `msg.sender` at the pool level, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. A pool admin who allowlists the router to enable their curated users to trade inadvertently opens the gate to every address on-chain.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to `_beforeSwap`:**

`MetricOmmPool.swap` at lines 230–240 calls `_beforeSwap(msg.sender, ...)`. When the caller is `MetricOmmSimpleRouter`, `msg.sender` is the router contract address. [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged:**

`ExtensionCalling._beforeSwap` at lines 160–176 encodes `sender` as the first argument to every configured extension without modification. [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` gates on that `sender` value:**

The check at line 37 is `!allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router. If the admin has called `setAllowedToSwap(pool, router, true)`, this check passes for every caller regardless of their individual allowlist status. [3](#0-2) 

**Step 4 — `MetricOmmSimpleRouter` calls `pool.swap(...)` directly for all swap variants:**

- `exactInputSingle` (lines 72–80): router calls `pool.swap(...)` directly. [4](#0-3) 

- `exactInput` (lines 104–112): same pattern for each hop. [5](#0-4) 

- `exactOutputSingle` (line 136–137): same pattern. [6](#0-5) 

- `exactOutput` recursive hops via `_exactOutputIterateCallback` (lines 220–228): same pattern. [7](#0-6) 

In all cases, the router is `msg.sender` at the pool level, so `sender` forwarded to the extension is the router address, not the originating user.

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the position beneficiary), not `sender` (the intermediary). `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` preserves `owner` as the caller-supplied address, so the deposit path does not share this flaw. [8](#0-7) [9](#0-8) 

**Existing guards are insufficient:** There is no mechanism in the pool, the extension, or the router to propagate the originating user's identity. The `extensionData` field is caller-controlled and not validated by the extension.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties loses that restriction entirely the moment the pool admin allowlists the router — the expected operational step for any pool that wants its allowlisted users to access the standard periphery. Any unprivileged address can then call `exactInputSingle` or any other router entry point and execute swaps against the pool, draining LP-owned liquidity at oracle-derived prices. This constitutes a direct loss of LP principal and a complete failure of the pool's core access-control invariant, meeting the "broken core pool functionality causing loss of funds" threshold.

## Likelihood Explanation

Medium. The pool admin must take the affirmative step of allowlisting the router. However, this is the natural and expected operational action for any pool that wants its allowlisted users to trade through the standard periphery. The bypass is therefore reachable on any production pool that follows the standard deployment pattern. No special attacker capability is required beyond calling a public router function.

## Recommendation

The extension must receive and check the **original user identity**, not the intermediary. Two complementary fixes:

1. **Router-side**: `MetricOmmSimpleRouter` should encode the originating `msg.sender` into the `extensionData` field (or a dedicated parameter) so extensions can decode and verify it.
2. **Extension-side**: `SwapAllowlistExtension.beforeSwap` should decode the real swapper from `extensionData` when `sender` is a known router, or the pool interface should expose a dedicated `originator` field that the router populates and the pool forwards to extensions.

The deposit allowlist's pattern of checking the economically meaningful actor (`owner`) rather than the intermediary (`sender`) should be mirrored in the swap allowlist.

## Proof of Concept

```solidity
// Pool is deployed with SwapAllowlistExtension.
// Pool admin allowlists the router so their users can trade:
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attacker (not individually allowlisted) calls the router directly:
router.exactInputSingle(ExactInputSingleParams({
    pool:             pool,
    recipient:        attacker,
    zeroForOne:       true,
    amountIn:         largeAmount,
    amountOutMinimum: 0,
    priceLimitX64:    0,
    tokenIn:          token0,
    extensionData:    "",
    deadline:         block.timestamp
}));
// MetricOmmPool.swap is called with msg.sender == router.
// _beforeSwap forwards sender == router to SwapAllowlistExtension.
// beforeSwap checks allowedSwapper[pool][router] == true → passes.
// Attacker receives token1 from the curated pool without being individually allowlisted.
```
