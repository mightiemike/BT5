### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Unprivileged Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently ignores the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead validates the `owner` argument (the position-owner address supplied by the caller). Because `owner` is a free caller-controlled parameter, any address not on the allowlist can bypass the guard by passing any allowlisted address as `owner`. The pool's callback then pulls tokens from the unauthorized caller, and the position is credited to the allowlisted `owner`. The allowlist invariant — "only permitted addresses may deposit into this pool" — is completely defeated.

---

### Finding Description

`MetricOmmPool.addLiquidity` calls the extension hook with both the real caller and the caller-supplied owner:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both values:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but discards it (unnamed `address,`) and only checks `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (the actual caller):

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [4](#0-3) 

The asymmetry is the root cause: the swap guard checks the real caller; the deposit guard checks a caller-controlled parameter.

After the hook passes, `LiquidityLib.addLiquidity` fires the modify-liquidity callback on `msg.sender` (the unauthorized caller), pulling tokens from them, while the position shares are credited to `owner`:

```solidity
IMetricOmmModifyLiquidityCallback(msg.sender)
    .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
``` [5](#0-4) 

```solidity
positionBinShares[posKey] = newUserShares;
``` [6](#0-5) 

Because `removeLiquidity` enforces `msg.sender == owner`, the unauthorized caller cannot reclaim the deposited tokens:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [7](#0-6) 

---

### Impact Explanation

1. **Allowlist guard fully bypassed (admin-boundary break):** Any address not on the allowlist can deposit into a restricted pool by supplying any allowlisted address as `owner`. The pool admin's access-control configuration is rendered ineffective.

2. **Forced position on an allowlisted LP:** The unauthorized caller can create or enlarge a position under any allowlisted `owner`/`salt` combination without the owner's consent. The owner is then exposed to impermanent loss on a position they never requested. They must actively discover and remove it to stop the exposure.

3. **Irrecoverable token loss for the caller:** Tokens paid by the unauthorized caller are locked in the pool under the allowlisted owner's position key. The unauthorized caller has no path to recover them (`removeLiquidity` requires `msg.sender == owner`).

4. **Repeated griefing:** Because the minimum-liquidity check only prevents the remaining balance from falling below `minimalMintableLiquidity` (not from being added to), an attacker can repeatedly top up the victim's position across multiple bins and salts, continuously enlarging the victim's unintended exposure and forcing repeated gas-costly removals.

---

### Likelihood Explanation

- The allowlisted addresses are readable on-chain from `allowedDepositor` or `allowAllDepositors`.
- No special privilege is required; any EOA or contract can call `addLiquidity`.
- The attacker only needs to implement `IMetricOmmModifyLiquidityCallback` and hold the required tokens.
- Pools that deploy `DepositAllowlistExtension` for access control (e.g., institutional or KYC-gated pools) are the direct target.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual caller) instead of `owner`, mirroring the pattern used in `SwapAllowlistExtension`:

```solidity
// Before (buggy):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) { ... }
}

// After (fixed):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) { ... }
}
```

If the intended semantic is "only allowlisted addresses may own positions" (rather than "only allowlisted addresses may call `addLiquidity`"), both `sender` and `owner` should be checked, and the NatSpec and `IDepositAllowlistExtension` interface should be updated to document the distinction.

---

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true   // alice is the only permitted depositor
  bob is NOT on the allowlist

Attack:
  1. bob deploys a contract implementing IMetricOmmModifyLiquidityCallback
     that transfers the required token amounts to the pool on callback.

  2. bob calls:
       pool.addLiquidity(
           owner    = alice,   // allowlisted — passes the guard
           salt     = 0,
           deltas   = { binIdxs: [0], shares: [minimalMintableLiquidity] },
           callbackData = "",
           extensionData = ""
       )

  3. DepositAllowlistExtension.beforeAddLiquidity checks allowedDepositor[pool][alice] → true → no revert.

  4. LiquidityLib fires metricOmmModifyLiquidityCallback on bob's contract;
     bob's contract transfers token0/token1 to the pool.

  5. Shares are credited to positionBinShares[keccak256(alice, 0, 0)].

  6. bob has bypassed the allowlist and deposited into the restricted pool.
     alice now holds an unexpected position she did not create.
     bob's tokens are irrecoverable (only alice can call removeLiquidity).
```
