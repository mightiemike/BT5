### Title
Unsafe `approve` Call Without Return-Value Guard Blocks Deposits for Non-Standard ERC20 Tokens - (`File: core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve()` directly through the `IIERC20Base` interface, which declares `approve` as returning `bool`. In Solidity ≥0.8.0, if the token does not return any data from `approve` (e.g., USDT-style tokens), the ABI decoder reverts. The `ERC20Helper` library correctly guards `transfer` and `transferFrom` with a low-level call pattern, but no equivalent `safeApprove` exists. This causes `creditDeposit()` to permanently revert for any registered spot token that omits the `approve` return value, locking deposited tokens in the DDA contract.

---

### Finding Description

`ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` both use the low-level call pattern:

```solidity
(bool success, bytes memory data) = address(self).call(...);
require(success && (data.length == 0 || abi.decode(data, (bool))), ...);
```

This correctly handles tokens that return no data. [1](#0-0) 

However, `DirectDepositV1.creditDeposit()` calls `approve` directly through the typed interface:

```solidity
token.approve(address(endpoint), balance);
``` [2](#0-1) 

The `IIERC20Base` interface declares `approve` as returning `bool`:

```solidity
function approve(address spender, uint256 amount) external returns (bool);
``` [3](#0-2) 

In Solidity ≥0.8.0, a high-level call to an interface function that declares a return value will cause the compiler to emit ABI-decoding logic on the return data. If the callee returns zero bytes (as USDT-style tokens do), the decoder reverts with an ABI decoding error — even though the `approve` itself succeeded on-chain.

The same pattern appears in `ContractOwner.wrapVaultAsset()`, which is also externally callable without an access modifier:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [4](#0-3) 

---

### Impact Explanation

Any non-standard ERC20 token registered as a spot product that omits the `bool` return from `approve` will cause `creditDeposit()` to revert unconditionally. Tokens already transferred into the DDA contract address cannot be forwarded to the endpoint. They are effectively stuck in the DDA contract until the owner manually calls `withdraw()`. The deposit flow is permanently broken for that token class through the DDA mechanism.

---

### Likelihood Explanation

`creditDeposit()` has no access modifier — any address can call it. [5](#0-4)  `ContractOwner.creditDepositV1()` also has no access modifier and routes through the same path. [6](#0-5)  If any spot product token (present or future) omits the `approve` return value, every call to `creditDeposit()` for that product will revert. The trigger requires only that such a token be listed — no privileged access is needed by the attacker.

---

### Recommendation

Add a `safeApprove` helper to `ERC20Helper` using the same low-level call pattern already used for `safeTransfer` and `safeTransferFrom`:

```solidity
function safeApprove(IERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Replace all direct `token.approve(...)` calls in `DirectDepositV1.creditDeposit()` and `ContractOwner.wrapVaultAsset()` with this safe wrapper.

---

### Proof of Concept

1. Deploy a non-standard ERC20 token whose `approve` function executes successfully but returns no data (zero-length return).
2. Register this token as a spot product in the protocol.
3. Transfer some amount of the token to a DDA address.
4. Call `creditDepositV1(subaccount)` or `DirectDepositV1.creditDeposit()` directly.
5. The call reverts at `token.approve(address(endpoint), balance)` because Solidity's ABI decoder fails on empty return data for a `bool`-typed return.
6. The tokens remain stuck in the DDA contract; the deposit never reaches the endpoint. [7](#0-6)
