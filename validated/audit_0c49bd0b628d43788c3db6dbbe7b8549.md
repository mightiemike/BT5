### Title
Single Trusted Relayer Is a SPOF: Stalled Chain Tip Permanently Blocks Transaction Verification - (File: contract/src/lib.rs)

### Summary
`submit_blocks` is gated exclusively to registered trusted relayers via the `#[trusted_relayer]` macro. There is no permissionless fallback path. If the single deployed relayer goes offline or selectively withholds specific Bitcoin block headers, the contract's canonical chain tip stalls permanently. Any downstream bridge or protocol that calls `verify_transaction_inclusion_v2` to release locked funds will fail indefinitely, with no user-accessible escape mechanism.

### Finding Description
The `submit_blocks` function is the sole write path for advancing the NEAR-side Bitcoin chain state. It is protected by two stacked guards:

1. `#[pause]` — blocks calls when the contract is paused (except for `Role::UnrestrictedSubmitBlocks`)
2. `#[trusted_relayer]` — rejects callers who are not registered as trusted relayers (bypass only via `Role::DAO` or `Role::UnrestrictedSubmitBlocks`) [1](#0-0) 

Both bypass roles (`Role::DAO`, `Role::UnrestrictedSubmitBlocks`) require an admin to grant them. An unprivileged NEAR account that attempts to call `submit_blocks` receives "Relayer is not active" and is rejected outright. [2](#0-1) 

The off-chain relayer is a single `Synchronizer` instance running an infinite loop. It holds the only signing key authorized to call `submit_blocks`. [3](#0-2) 

When the relayer is offline or withholds headers for specific block heights, `mainchain_tip_blockhash` stops advancing. The contract has no timeout, no expiry, and no permissionless submission path. The stall is permanent until an admin intervenes.

### Impact Explanation
`verify_transaction_inclusion_v2` checks two conditions against the live chain state:

1. The target block must be in `mainchain_header_to_height` (i.e., on the current main chain).
2. The tip height minus the target block height must meet the required confirmation count. [4](#0-3) 

If the relayer stalls after block N, any Bitcoin transaction confirmed at height > N cannot be proven. A bridge contract that calls `verify_transaction_inclusion_v2` to release user funds will panic with "block does not belong to the current main chain" or "Not enough blocks confirmed". The user's funds in the bridge remain locked with no recourse — there is no cancel, no timeout, and no permissionless path to submit the missing headers.

### Likelihood Explanation
The relayer is a single process with a single signing key. [5](#0-4) 

Any crash, key compromise, infrastructure outage, or deliberate censorship of specific block heights causes an indefinite stall. The relayer's own error handling only retries internally — it does not expose any on-chain fallback. [6](#0-5) 

### Recommendation
- Allow any NEAR account to call `submit_blocks` with a valid PoW-verified header (permissionless submission), removing the `#[trusted_relayer]` gate or adding an open fallback path when the trusted relayer has not submitted for a configurable number of blocks.
- Alternatively, register multiple independent trusted relayers so no single entity is a SPOF.
- Add an on-chain staleness check: if the chain tip has not advanced for a threshold number of NEAR blocks, allow permissionless submission.

### Proof of Concept
1. Deploy the contract and register exactly one trusted relayer account (the normal production setup).
2. The relayer submits headers up to Bitcoin height N; `mainchain_tip_blockhash` points to block N.
3. Stop the relayer process (or have it skip height N+1 onward).
4. A user sends BTC in a bridge transaction confirmed at height N+1.
5. The user calls `verify_transaction_inclusion_v2` with the block hash of height N+1.
6. The contract panics: `"block does not belong to the current main chain"` — because height N+1 was never submitted.
7. The user attempts to call `submit_blocks` directly with the valid header for N+1.
8. The contract panics: `"Relayer is not active"` — because the user is not a trusted relayer.
9. The user's bridge funds are permanently locked with no on-chain escape path. [7](#0-6)

    #[payable]
    #[pause]
    #[trusted_relayer]
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

    ($res:expr, $msg:expr, $sleep_time:expr, $label:tt) => {
