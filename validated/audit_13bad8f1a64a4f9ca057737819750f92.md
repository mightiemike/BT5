### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery — (`contract/src/lib.rs`)

### Summary
The contract exposes a deprecated `verify_transaction_inclusion` function that is still reachable by any unprivileged NEAR caller. It lacks the coinbase-proof guard introduced in `verify_transaction_inclusion_v2`, making it vulnerable to the well-known 64-byte transaction Merkle proof forgery attack. An attacker can supply an internal Merkle-tree node hash as `tx_id` and receive a `true` verification result for a transaction that was never included in any block.

### Finding Description
`verify_transaction_inclusion` is marked `#[deprecated]` in Rust, which only emits a compiler warning — it does not restrict runtime access. The function remains a live, `#[pause]`-gated public entry point callable by any NEAR account. [1](#0-0) 

The function computes a Merkle root from the caller-supplied `tx_id` and `merkle_proof` and compares it to the stored `block_header.merkle_root`: [2](#0-1) 

`compute_root_from_merkle_proof` in `merkle-tools` performs no validation that the leaf is a real transaction hash versus an internal node: [3](#0-2) 

The v2 variant mitigates this by first verifying a coinbase proof at index 0, which forces the attacker to commit to a real leaf at the tree's leftmost position: [4](#0-3) 

The v1 function has no such guard. The contract's own documentation acknowledges this: [5](#0-4) 

### Impact Explanation
Any downstream NEAR contract or off-chain consumer that calls `verify_transaction_inclusion` (v1) to authorize an action — e.g., releasing bridged funds, minting tokens, or unlocking collateral — can be deceived. An attacker constructs a valid Merkle path to an internal node of a real Bitcoin block, presents that node's hash as `tx_id`, and the function returns `true`. The corrupted proof result is indistinguishable from a legitimate one to the caller.

### Likelihood Explanation
The attack requires only knowledge of a real Bitcoin block's Merkle tree structure (publicly available) and the ability to call a NEAR contract function. No privileged role, private key, or social engineering is needed. The 64-byte technique is documented and has tooling available. Any integrator that chose v1 over v2 — or that was written before v2 existed — is immediately exploitable.

### Recommendation
Remove `verify_transaction_inclusion` (v1) entirely, or gate it with an unconditional `env::panic_str` so it cannot be called at runtime. Do not rely on `#[deprecated]` to prevent production use. All callers must be migrated to `verify_transaction_inclusion_v2`.

### Proof of Concept

1. Identify any real Bitcoin block `B` whose Merkle tree has at least two transactions.
2. Compute the internal node hash `N = SHA256d(tx0 || tx1)` (the parent of the first two leaves).
3. Construct a Merkle proof path from `N` to the Merkle root of `B` (one level shorter than a leaf proof).
4. Call `verify_transaction_inclusion` on the NEAR contract with:
   - `tx_id = N` (the internal node hash, not a real txid)
   - `tx_block_blockhash` = hash of block `B` (which must be on the stored mainchain)
   - `tx_index` = the position of `N` in the reduced tree
   - `merkle_proof` = the path from `N` to the root
   - `confirmations` = any value ≤ `gc_threshold`
5. The function returns `true`, falsely asserting that `N` is an included transaction. [6](#0-5) [7](#0-6)

        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
