### Title
Internal-Node Forgery via Odd-Tree Index Aliasing in `verify_transaction_inclusion` (v1) — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) accepts an internal Merkle node hash as `tx_id` and returns `true` for a non-existent transaction. For any block with an odd number of transactions (2k+1), an attacker can supply the duplicated-leaf internal node `N_last = H(T_last, T_last)` as `tx_id` at `tx_index = 2k+1` with a single-element proof, and `compute_root_from_merkle_proof` will reconstruct the correct Merkle root — confirming a transaction that does not exist.

---

### Finding Description

`compute_root_from_merkle_proof` performs no validation that the supplied `transaction_hash` is a leaf node, nor that `transaction_position` is within the valid leaf range. It simply iterates over the proof array, hashing left or right based on position parity: [1](#0-0) 

For a 3-transaction block `[T0, T1, T2]`:

```
Level 0 (padded): [T0, T1, T2, T2]
Level 1:          [N01=H(T0,T1),  N22=H(T2,T2)]
Root:              H(N01, N22)
```

**Attack call:** `compute_root_from_merkle_proof(N22, 3, [N01])`

Trace:
- `current_position = 3` (odd) → `current_hash = H(proof[0], N22) = H(N01, N22)` = **Root** ✓
- Loop ends after 1 step.

The function returns `Root`, so `verify_transaction_inclusion` returns `true` for `tx_id = N22` — an internal node, not a real transaction.

`verify_transaction_inclusion` passes `tx_id` and `tx_index` directly to `compute_root_from_merkle_proof` with no leaf-range or proof-depth guard: [2](#0-1) 

The function is a public NEAR contract method. `#[deprecated]` is a Rust compiler hint only — it does not prevent external callers. `#[pause]` is a runtime gate that is **not** engaged by default: [3](#0-2) 

---

### Impact Explanation

Any downstream contract or off-chain system that calls `verify_transaction_inclusion` (v1) and trusts its `true` return value can be deceived into believing a fabricated transaction was confirmed in a real Bitcoin block. The attacker needs only:

1. A real Bitcoin block with an odd transaction count (extremely common — the majority of Bitcoin blocks have an odd number of transactions).
2. Knowledge of any two adjacent leaf hashes to compute the internal node.
3. A single NEAR contract call with a 1-element forged proof.

This directly satisfies the target scope: *"Merkle proof or transaction inclusion validation flaw that returns true for a nonexistent transaction … forged internal node."*

---

### Likelihood Explanation

- The function is publicly callable with no access control beyond the optional runtime pause.
- Odd-length transaction trees are the norm in Bitcoin (most blocks have an odd tx count).
- The attack requires no privileged role, no key compromise, and no social engineering.
- The proof construction is trivial: compute `N22 = SHA256d(T_last || T_last)`, set `tx_index = N_leaves` (the out-of-range index), proof = `[N01_sibling]`.
- The codebase itself acknowledges the attack surface in both the inline `# Warning` and `CLAUDE.md`: [4](#0-3) 

---

### Recommendation

1. **Remove or hard-disable `verify_transaction_inclusion` (v1).** Deprecation warnings do not prevent external callers. Replace with a panic or unconditional revert.
2. **In `verify_transaction_inclusion_v2`**, enforce `merkle_proof.len() == coinbase_merkle_proof.len()` AND that `coinbase_tx_id` is verified against a known-good coinbase (e.g., require `tx_index == 0` for the coinbase call), so a forged coinbase_tx_id (itself an internal node) cannot be used to bypass the depth check.
3. **In `compute_root_from_merkle_proof`**, consider asserting `transaction_position < (1 << merkle_proof.len())` to reject out-of-range indices that alias into internal-node positions.

---

### Proof of Concept

```rust
// 3-tx tree: T0, T1, T2
let t0 = H256([0x01; 32]);
let t1 = H256([0x02; 32]);
let t2 = H256([0x03; 32]);

// Level 1 nodes
let n01 = compute_hash(&t0, &t1);          // H(T0, T1)
let n22 = compute_hash(&t2, &t2);          // H(T2, T2)  ← internal node, not a real tx
let root = compute_hash(&n01, &n22);       // Merkle root

// Forged proof: claim n22 is a "transaction" at index 3
let forged_proof = vec![n01];
let result = compute_root_from_merkle_proof(n22, 3, &forged_proof);

assert_eq!(result, root);  // PASSES — internal node accepted as valid tx
```

Calling `verify_transaction_inclusion` on a real block with `tx_id = n22`, `tx_index = 3`, `merkle_proof = [n01]` returns `true`, confirming a transaction that does not exist. [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
