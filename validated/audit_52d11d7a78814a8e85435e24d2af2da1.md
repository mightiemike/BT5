**Exact Call Chain:**

1. `user_EOA` calls `MetricOmmSimpleRouter.exactInputSingle(params)` ā€” `msg.sender = user_EOA`
2. Router calls `IMetricOmmPoolActions(params.pool).swap(...)` ā€” pool sees `msg.sender = router`
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` ā€” passes `sender = router`
4. `ExtensionCalling._beforeSwap` encodes `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))` with `sender = router`
5. `SwapAllowlistExtension.beforeSwap` receives `sender = router`, `msg.sender = pool`

This resolves to `allowedSwapper[pool][router]` ā€” **never** `allowedSwapper[pool][user_EOA]`.

**Is the prerequisite (router being allowlisted) a "trusted admin" rejection?**

The pool admin allowlisting the router is not malicious ā€” it is the only way to permit any user to swap through the router at all. The admin's intent is "allow router-based swaps while maintaining a per-user allowlist." The design makes this intent impossible: allowlisting the router is a binary that opens the pool to every user who calls through the router. This is a design flaw in the extension, not a malicious admin action.

### Title
SwapAllowlistExtension checks router address instead of originating user, allowing any EOA to bypass per-user swap allowlist via MetricOmmSimpleRouter ā€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` gates swaps on `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of `pool.swap`. When `MetricOmmSimpleRouter.exactInputSingle` is used, `sender` is always the router address, not the originating EOA. Allowlisting the router therefore grants unrestricted swap access to every user who routes through it, defeating the per-user allowlist entirely.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [3](#0-2) 

Meanwhile, `exactInputSingle` calls `pool.swap` directly without forwarding the original caller: [4](#0-3) 

The originating EOA (`msg.sender` of `exactInputSingle`) is stored only in transient storage for the payment callback ā€” it is never surfaced to the extension layer.

### Impact Explanation

A pool admin who configures `SwapAllowlistExtension` to restrict swaps to a specific set of addresses, and who also allowlists the router (the only way to permit router-based swaps), inadvertently opens the pool to every user who calls through the router. The per-user allowlist is completely non-functional in the router path. Any EOA can swap in a pool that the admin intended to restrict.

This breaks the core functionality of the `SwapAllowlistExtension` ā€” the allowlist does not gate the actual economic actor.

### Likelihood Explanation

Any pool that uses `SwapAllowlistExtension` and also wants to support the standard `MetricOmmSimpleRouter` must allowlist the router. This is the expected operational configuration. The bypass is therefore automatic and requires no attacker sophistication beyond calling `exactInputSingle`.

### Recommendation

The extension interface should be extended to carry the originating user identity separately from the immediate caller. One approach: add an `originator` field to the `beforeSwap` hook arguments (populated by the pool from a trusted router registry or from `tx.origin` with appropriate caveats). Alternatively, `SwapAllowlistExtension` should document that it is incompatible with router intermediaries and that pools using it must be called directly.
