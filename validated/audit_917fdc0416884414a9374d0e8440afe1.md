### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` parameter and validates `owner` against the allowlist instead. Because `addLiquidity` accepts a caller-supplied `owner` address, any unprivileged address can bypass the guard by naming an already-allowlisted owner. The sibling `SwapAllowlistExtension` correctly gates on `sender`, making the inconsistency a clear implementation error rather than a design choice.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives `(address sender, address owner, …)` but discards `sender` (unnamed `_`) and checks `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-controlled `owner` argument as `owner`:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [2](#0-1) 

Because `owner` is a free parameter supplied by the caller, any address can call `addLiquidity(owner = allowlistedAddress, …)` and the guard will pass, even though the actual depositor (`msg.sender`) is not on the allowlist.

Compare with `SwapAllowlistExtension`, which correctly gates on `sender`:

```solidity
// SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, …) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    …
}
``` [3](#0-2) 

The `_callExtensionsInOrder` dispatch in `ExtensionCalling` faithfully forwards whatever arguments the pool encodes; the error is entirely in the extension's own check. [4](#0-3) 

---

### Impact Explanation

The deposit allowlist is an admin-configured access-control boundary. With this bug, any unprivileged address can deposit into any allowlisted owner's position, fully bypassing the guard. Consequences include:

- **Allowlist invariant broken**: the pool admin's intent to restrict depositors is silently nullified.
- **Unwanted position inflation**: an attacker can force liquidity into an allowlisted owner's position (at the attacker's own token cost), altering the owner's share composition and potentially affecting their expected withdrawal amounts.
- **Protocol-level access control failure**: pools deployed specifically to restrict participation (e.g., permissioned institutional pools) are open to any depositor who knows an allowlisted address.

---

### Likelihood Explanation

Allowlisted owner addresses are observable on-chain (via `AllowedToDepositSet` events or direct `allowedDepositor` reads). Any actor can trivially construct the bypass call. The only cost to the attacker is the tokens they deposit (which they control). Likelihood is **high** whenever a pool is deployed with `DepositAllowlistExtension` and `allowAllDepositors` is false.

---

### Recommendation

Change the check to validate `sender` (the actual caller), consistent with `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate on the position owner (not the caller), the mapping name, event name, and NatSpec must be updated to reflect that, and the inconsistency with `SwapAllowlistExtension` must be explicitly documented.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; `allowAllDepositors[pool] = false`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` — only `alice` is permitted.
3. `bob` (not on allowlist) calls:
   ```solidity
   pool.addLiquidity(
       owner = alice,   // allowlisted address
       salt  = 0,
       deltas = ...,
       callbackData = ...,
       extensionData = ""
   );
   ```
4. Pool calls `extension.beforeAddLiquidity(sender=bob, owner=alice, …)`.
5. Extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
6. `bob`'s tokens are deposited into `alice`'s position; the allowlist is fully bypassed.
