### Title
Linked Signer Can Overwrite Its Own Authorization via `LinkSigner` Fast Path — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner: true`. This means the **current linked signer** of a subaccount can sign and submit a `LinkSigner` transaction to replace the linked signer with any address it controls. The slow-mode path for the same transaction type correctly restricts this action to the subaccount owner only. The missing owner-only check in the fast path is the direct analog of the yAxis M-13 finding, where `addStrategy`/`removeStrategy` lacked the `allowedVaults` guard that `reorderStrategies` enforced.

---

### Finding Description

`EndpointTx.processTransactionImpl` handles the `LinkSigner` transaction type at lines 576–590:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
```

`validateSignedTx` with `allowLinkedSigner: true` calls `validateSignature`, which in turn calls `verifier.validateSignature(sender, getLinkedSigner(sender), digest, signature)`. The verifier accepts a signature from **either** the subaccount owner address **or** the current linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

Because the linked signer is accepted as a valid signer for the `LinkSigner` transaction itself, a linked signer can craft and submit a `LinkSigner` transaction to replace itself (or set any arbitrary address as the new linked signer).

**Contrast with the slow-mode path** (`processSlowModeTransactionImpl`, lines 232–239):

```solidity
validateSender(txn.sender, sender);   // requires msg.sender == subaccount owner address
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

`validateSender` enforces `address(uint160(bytes20(txn.sender))) == sender` — only the actual owner of the subaccount can change the linked signer via slow mode.

**Contrast with `WithdrawCollateralV2`** (lines 442–448), which correctly restricts the linked signer when a custom `sendTo` address is specified:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    signedTx.tx.sendTo == address(0)   // linked signer blocked when sendTo != address(0)
);
```

The `LinkSigner` fast path applies no equivalent restriction, leaving the linked signer free to mutate its own authorization.

---

### Impact Explanation

A compromised or malicious linked signer (e.g., an API key or trading bot) can:

1. Sign a `LinkSigner` transaction naming any address as the new linked signer.
2. The sequencer processes it; the on-chain handler accepts the linked signer's signature.
3. `linkedSigners[victim_subaccount]` is overwritten with the attacker-controlled address.
4. The attacker's new linked signer can now sign `WithdrawCollateral` (fast path, `allowLinkedSigner: true`), `MintNlp`, `BurnNlp`, `TransferQuote`, and `LiquidateSubaccount` transactions on behalf of the victim subaccount.
5. The victim's collateral can be drained via `WithdrawCollateral` (funds go to `address(uint160(bytes20(sender)))`, i.e., the subaccount owner's address) or via `TransferQuote` to another subaccount of the same owner address — but more critically, the attacker can also sign `WithdrawCollateralV2` with `sendTo == address(0)` (which also allows linked signers) to withdraw to the owner's address, or manipulate NLP positions and liquidations.

The corrupted state is `linkedSigners[subaccount]` — a mapping that gates signing authority for all high-value operations on the subaccount.

---

### Likelihood Explanation

Any user who has ever set a linked signer (e.g., for automated trading via an API key) is exposed. The attacker only needs to be the current linked signer — no privileged protocol access is required. The attack is executable by submitting a single sequencer transaction. The linked signer role is explicitly designed for third-party or automated use, making this a realistic threat model.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner (the address embedded in the `sender` bytes32) should be permitted to change the linked signer:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // linked signer must NOT be allowed to change the linked signer
);
```

This aligns the fast path with the slow-mode path, which already enforces owner-only access via `validateSender`.

---

### Proof of Concept

1. Alice sets Bob (`0xBob`) as her linked signer via a slow-mode `LinkSigner` transaction. `linkedSigners[alice_subaccount] = 0xBob`.
2. Bob constructs a `LinkSigner` transaction: `sender = alice_subaccount`, `signer = 0xAttacker`, `nonce = current_nonce`.
3. Bob signs the EIP-712 digest for this transaction with his own key.
4. Bob submits the transaction to the sequencer.
5. `processTransactionImpl` is called. `validateSignedTx(..., true)` calls `verifier.validateSignature(alice_subaccount, 0xBob, digest, sig)`. The verifier recovers `0xBob` from the signature and accepts it because `recovered == linkedSigner`.
6. `linkedSigners[alice_subaccount]` is set to `0xAttacker`.
7. `0xAttacker` can now sign `WithdrawCollateral` transactions on Alice's behalf, draining her collateral to her own address — or sign `WithdrawCollateralV2` with `sendTo == address(0)` (linked signer allowed when `sendTo == address(0)`) to withdraw to Alice's address, or manipulate her NLP/perp positions.
