### Title
Unbounded Loop in `reorg_chain` Causes Gas Exhaustion, Permanently Corrupting Fork-Choice Canonical Mapping — (File: `contract/src/lib.rs`)

---

### Summary

The `reorg_chain` function contains two unbounded loops that iterate once per fork block, each performing multiple `LookupMap` storage operations. There is no on-chain cap on fork length; the only limit (`max_fork_len = 500`) lives in the off-chain relayer config and is not enforced by the contract. When a fork is long enough to exhaust NEAR's per-transaction gas budget, the transaction reverts, the reorg never completes, and `mainchain_tip_blockhash` is permanently left pointing to the lower-chainwork chain. This corrupts the canonical mapping that `verify_transaction_inclusion` depends on.

---

### Finding Description

**Root cause — `reorg_chain`, `contract/src/lib.rs` lines 575–647:**

The function contains two unbounded iteration paths:

**Loop 1** (lines 586–593): iterates over every main-chain block above the fork point, removing each one:

```rust
for height in (fork_tip_height + 1)..=last_main_chain_block_height {
    let current_main_chain_blockhash = self
        .mainchain_height_to_header
        .get(&height)
        ...
    self.remove_block_header(&current_main_chain_blockhash);
    self.mainchain_height_to_header.remove(&height);
}
```

**Loop 2** (lines 616–643): walks back through the entire fork chain, promoting each fork block to the main chain:

```rust
while !self
    .mainchain_header_to_height
    .contains_key(&fork_header_cursor.block_hash)
{
    ...
    self.mainchain_height_to_header.insert(&current_height, &current_block_hash);
    self.mainchain_header_to_height.insert(&current_block_hash, &current_height);
    if let Some(current_main_chain_blockhash) = main_chain_block {
        self.remove_block_header(&current_main_chain_blockhash);
    }
    fork_header_cursor = self.headers_pool.get(&prev_block_hash)...;
}
```

Each iteration of Loop 2 performs at minimum 5–6 `LookupMap` host-function calls (reads + writes). NEAR storage writes are among the most expensive host operations. With a sufficiently long fork, the cumulative gas cost of these iterations exceeds NEAR's 300 Tgas per-transaction limit, causing the entire `submit_blocks` call to revert.

**No on-chain cap exists.** The only fork-length limit is `max_fork_len = 500` in the relayer's off-chain config:

```toml
max_fork_len = 500
```

This is a relayer-side heuristic, not enforced by the contract. Any caller of `submit_blocks` (a trusted relayer, or an account with `Role::UnrestrictedSubmitBlocks`) can submit fork blocks one at a time across many transactions, accumulating a fork of arbitrary depth in `headers_pool`. The gas exhaustion only occurs when the final block triggers `reorg_chain` in a single transaction.

**State after failed reorg:**
- Fork blocks submitted in prior transactions remain in `headers_pool` (as fork/sidechain entries).
- `mainchain_tip_blockhash` is unchanged — it still points to the lower-chainwork chain.
- `mainchain_height_to_header` / `mainchain_header_to_height` remain consistent with the old (wrong) chain.
- There is no batched-reorg mechanism; the contract cannot complete the reorg across multiple calls.

---

### Impact Explanation

**Corrupted canonical mapping.** After the failed reorg, `mainchain_tip_blockhash` permanently points to the lower-chainwork chain. Every downstream consumer of the canonical state is affected:

- `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` check `mainchain_header_to_height` to confirm a block belongs to the main chain. With the wrong tip, they will accept SPV proofs for transactions on the abandoned (lower-chainwork) chain and reject proofs for transactions on the correct (higher-chainwork) chain.
- `get_last_block_header`, `get_last_block_height`, and `get_last_n_blocks_hashes` all return data from the wrong chain.

The broken invariant is: **the contract's canonical chain must always track the highest-chainwork chain**. After a gas-exhausted reorg, this invariant is permanently violated with no recovery path.

**Impact: High** — SPV proof verification is silently wrong; cross-chain applications consuming `verify_transaction_inclusion` would accept fraudulent inclusion proofs against the abandoned chain.

---

### Likelihood Explanation

**Likelihood: Low.**

- For Bitcoin mainnet, a reorg long enough to exhaust gas requires sustained 51% mining power — practically infeasible.
- For Dogecoin or Litecoin (lower hashrate), a multi-hundred-block reorg is more feasible, either via a real network event or a well-resourced attacker who stakes to become a trusted relayer and mines a competing chain.
- The `#[trusted_relayer]` attribute on `submit_blocks` restricts direct callers, but any account can stake to become a trusted relayer, and the `Role::UnrestrictedSubmitBlocks` bypass role further widens the surface.
- The vulnerability is also reachable through normal protocol operation: a legitimate long reorg on a lower-hashrate chain would trigger the same failure path via the honest relayer.

---

### Recommendation

Add an on-chain cap on the fork depth processed in a single `reorg_chain` call, mirroring the relayer's `max_fork_len`. If the fork depth exceeds the cap, panic with a descriptive error before any state is mutated. Alternatively, implement a batched reorg mechanism that can promote fork blocks across multiple transactions, updating `mainchain_tip_blockhash` only after the full promotion is complete.

---

### Proof of Concept

1. Deploy the contract (Bitcoin or Dogecoin feature flag).
2. As a trusted relayer (or account with `UnrestrictedSubmitBlocks`), submit `N` fork blocks via repeated `submit_blocks` calls, each building on the previous fork block. Each call stores one block in `headers_pool` as a fork entry. The fork's cumulative `chain_work` is kept just below the main chain's until the final block.
3. Submit the final fork block whose `chain_work` exceeds the main chain tip. This triggers `reorg_chain` inside `submit_block_header_inner`.
4. `reorg_chain`'s `while` loop (lines 616–643) iterates `N` times, each performing 5–6 `LookupMap` operations. For sufficiently large `N`, the transaction exceeds NEAR's 300 Tgas gas limit and reverts.
5. Observe: `get_last_block_header()` still returns the old (lower-chainwork) tip. The fork blocks remain in `headers_pool`. `verify_transaction_inclusion` now accepts proofs against the abandoned chain.

```rust
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
```

```rust
    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
```
