Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks `sender` (router) instead of `recipient` (trader), enabling allowlist bypass via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on `sender` — the direct `msg.sender` to the pool, which is the router contract when users swap via `MetricOmmSimpleRouter` — rather than `recipient`, the actual economic beneficiary of the swap. This is asymmetric with `DepositAllowlistExtension.beforeAddLiquidity`, which correctly gates on `owner` (the position owner). Any pool admin who allowlists the router to enable standard periphery swaps simultaneously opens the gate to every user, including those the allowlist was meant to exclude.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(msg.sender, recipient, zeroForOne, amountSpecified, ...);
```

`ExtensionCalling._beforeSwap` forwards this unchanged to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...)));
```

`SwapAllowlistExtension.beforeSwap` then checks `sender` (the router) and ignores `recipient` (the user):

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-39
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
```

When Mallory calls `MetricOmmSimpleRouter.exactInputSingle(...)`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,   // Mallory
    params.zeroForOne,
    ...
);
```

The pool's `msg.sender` is the **router**, not Mallory. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][mallory]`. If the router is allowlisted (necessary for any router-based swap to work), the check passes for every user regardless of their individual allowlist status.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` ignores `sender` and checks `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-39
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
```

The deposit path correctly gates on the economic actor; the swap path does not. The integration test confirms the swap allowlist is keyed on the caller contract (`callers[0]`, a `TestCaller`), not the recipient user:

```solidity
// metric-periphery/test/extensions/FullMetricExtension.t.sol L70
swapExtension.setAllowedToSwap(address(pool), address(callers[0]), true);
```

## Impact Explanation
A pool deploying `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (KYC'd addresses, institutional partners, compliance-gated users) cannot enforce that restriction when users interact through `MetricOmmSimpleRouter`. The admin has no mechanism to simultaneously allowlist the router (enabling standard periphery access) and restrict individual users — the two goals are mutually exclusive under the `sender`-based check. Unauthorized users can execute swaps in pools explicitly configured to exclude them, causing unauthorized price impact, unauthorized fee extraction, and violation of any regulatory or compliance constraints the allowlist was meant to enforce. This is a direct policy bypass on a core access-control extension.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary supported swap entrypoint for end users. Any pool that deploys `SwapAllowlistExtension` and allowlists the router (the expected operational pattern for a curated pool that still wants to support standard periphery) is immediately and fully vulnerable. No special attacker capability is required beyond calling the public router. The condition is the normal, expected admin configuration, not an edge case.

## Recommendation
Change `SwapAllowlistExtension.beforeSwap` to check `recipient` instead of `sender`, mirroring the `owner`-based check in `DepositAllowlistExtension`:

```solidity
// After: checks the actual trader — correct actor
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

This aligns the swap allowlist with the deposit allowlist's actor model: both gate on the economic beneficiary of the action regardless of which intermediary relays the call.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Admin calls `swapExtension.setAllowedToSwap(pool, routerAddress, true)` to enable router-based swaps.
3. Admin does **not** call `swapExtension.setAllowedToSwap(pool, malloryAddress, true)`.
4. Mallory calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: mallory, ...})`.
5. Pool calls `extension.beforeSwap(router, mallory, ...)`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. Mallory successfully swaps in a pool configured to exclude her.

The deposit allowlist does not share this bypass: even if Mallory routes through `MetricOmmPoolLiquidityAdder`, the extension checks `owner` (Mallory's address), which is not allowlisted, and the deposit reverts correctly — confirming the asymmetry is a defect in the swap path, not an intentional design difference.
