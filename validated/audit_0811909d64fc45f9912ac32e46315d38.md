### Title
Address-Based Claim Tracking Blocks Multi-Leaf Merkle Claims, Causing Permanent Token Loss — (`core/contracts/Airdrop.sol`)

---

### Summary

`Airdrop.sol` tracks whether a user has claimed using `claimed[week][sender]`, keyed on the claimant's address. The leaf hash, however, encodes both `sender` and `totalAmount`. If a weekly Merkle tree contains two leaves for the same address (e.g., two different reward tiers), the address-based guard permanently blocks the second leaf after the first is claimed, causing irreversible loss of the remaining tokens.

---

### Finding Description

The double-claim guard in `_verifyProof` checks:

```solidity
require(claimed[week][sender] == 0, "Already claimed.");
``` [1](#0-0) 

After a successful claim it writes:

```solidity
claimed[week][sender] = totalAmount;
``` [2](#0-1) 

The Merkle leaf, however, is constructed from **both** `sender` and `totalAmount`:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(keccak256(abi.encode(sender, totalAmount)))
);
``` [3](#0-2) 

This means two distinct, valid leaves `(A, 100)` and `(A, 500)` can coexist in the same week's Merkle tree. Once address `A` claims the first leaf, `claimed[week][A]` becomes non-zero and the second leaf is permanently unclaimable — the 500-token leaf is locked in the contract forever with no recovery path.

The `claimed` mapping is declared as:

```solidity
mapping(uint32 => mapping(address => uint256)) internal claimed;
``` [4](#0-3) 

There is no leaf-level granularity in this mapping. The contract never documents whether multiple leaves per address per week are intended or forbidden, leaving Merkle tree builders without a clear invariant to enforce off-chain.

---

### Impact Explanation

If a weekly Merkle tree is built with two leaves for the same address — whether intentionally (to offer a choice of reward tiers) or accidentally (duplicate entries) — the user can only ever receive one of them. The tokens allocated to the unclaimed leaf remain locked in the contract with no admin recovery function present. This is a direct, permanent loss of user funds proportional to the difference between the two leaf amounts.

---

### Likelihood Explanation

The `registerMerkleRoot` function imposes no structural constraint on the tree; any root is accepted. Off-chain tooling that aggregates rewards from multiple sources (e.g., trading fees + referral bonuses for the same address in the same week) could naturally produce duplicate-address leaves. Because the contract is silent on this invariant, tree builders have no on-chain signal that it is forbidden. The likelihood is medium: it depends on the off-chain pipeline, but the contract provides no guard and no documentation. [5](#0-4) 

---

### Recommendation

Two mutually exclusive fixes:

1. **Enforce one leaf per address per week off-chain and document it.** Add a NatSpec comment to `_verifyProof` and `registerMerkleRoot` stating that the Merkle tree must contain at most one leaf per address per week. This preserves the current address-based tracking but makes the invariant explicit.

2. **Switch to leaf-based tracking.** Replace `claimed[week][sender]` with `claimed[week][leafHash]`, where `leafHash` is the computed leaf. This allows multiple distinct leaves per address per week to each be claimed independently:

```solidity
// Replace:
mapping(uint32 => mapping(address => uint256)) internal claimed;
// With:
mapping(uint32 => mapping(bytes32 => bool)) internal claimed;
```

And update the guard and setter in `_verifyProof` accordingly.

---

### Proof of Concept

1. Owner registers a Merkle root for week 1 that contains two leaves:
   - Leaf L1: `keccak256(keccak256(abi.encode(Alice, 100e18)))`
   - Leaf L2: `keccak256(keccak256(abi.encode(Alice, 500e18)))`

2. Alice calls `claim([{week:1, totalAmount:100e18, proof:proofL1}])`.
   - `claimed[1][Alice] == 0` → passes.
   - `claimed[1][Alice]` is set to `100e18`. Alice receives 100 tokens.

3. Alice calls `claim([{week:1, totalAmount:500e18, proof:proofL2}])`.
   - `claimed[1][Alice] == 100e18 != 0` → **reverts with "Already claimed."**
   - The 500-token leaf is permanently blocked. Alice loses 400 tokens with no recourse. [6](#0-5)

    function _verifyProof(
        uint32 week,
        address sender,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        require(claimed[week][sender] == 0, "Already claimed.");
        require(
            merkleRoots[week] != bytes32(0),
            "Week hasn't been registered."
        );
        require(
            !ISanctionsList(sanctions).isSanctioned(sender),
            "address is sanctioned."
        );
        bytes32 leaf = keccak256(
            bytes.concat(keccak256(abi.encode(sender, totalAmount)))
        );
        bool isValidLeaf = MerkleProof.verify(proof, merkleRoots[week], leaf);
        require(isValidLeaf, "Invalid proof.");
        claimed[week][sender] = totalAmount;
    }
```
