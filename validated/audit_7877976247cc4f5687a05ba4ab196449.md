Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual user, allowing full allowlist bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` enforces `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router's address, not the actual user. Any pool admin who allowlists the router to enable router-based swaps for their permitted users inadvertently grants swap access to every user who calls the router, completely defeating the per-user allowlist.

## Finding Description

**Root cause — extension check:**

`SwapAllowlistExtension.beforeSwap` at line 37 enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (the extension's caller), so the effective check is `allowedSwapper[pool][sender]`. The value of `sender` is whatever address called `pool.swap()`. [1](#0-0) 

**Root cause — pool passes `msg.sender` verbatim:**

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to every configured extension: [2](#0-1) 

**Root cause — router is the pool's `msg.sender`:**

`MetricOmmSimpleRouter.exactInputSingle()` stores the actual user in transient storage for the payment callback only, then calls `pool.swap()` directly. The router becomes `msg.sender` to the pool:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [3](#0-2) 

The actual user's address is never passed to `pool.swap()` and therefore never reaches the extension. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

**Exploit flow:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to specific institutional counterparties (e.g., addresses A and B).
2. Pool admin calls `setAllowedToSwap(pool, A, true)` and `setAllowedToSwap(pool, B, true)`.
3. Pool admin also calls `setAllowedToSwap(pool, router, true)` so that A and B can use the router — a natural and expected configuration.
4. Unauthorized user C calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
5. The router calls `pool.swap(...)` with itself as `msg.sender`.
6. The pool calls `_beforeSwap(router, ...)` and the extension checks `allowedSwapper[pool][router]` — which is `true`.
7. User C's swap succeeds despite never being allowlisted.

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` checks `owner`, which is an explicit parameter that callers (including `MetricOmmPoolLiquidityAdder`) set to the actual beneficiary's address regardless of who the immediate caller is. This design correctly identifies the real actor. `SwapAllowlistExtension` has no equivalent mechanism. [5](#0-4) 

## Impact Explanation

Any user can bypass the swap allowlist of a curated pool by routing through `MetricOmmSimpleRouter`, provided the router is allowlisted. In a pool designed for specific institutional counterparties, unauthorized users gaining swap access exposes LPs to adverse selection at oracle prices and breaks the curation guarantee the pool admin configured. This is a broken core access-control invariant with direct fund impact on LPs (adverse selection at oracle-derived bid/ask prices from unintended counterparties).

## Likelihood Explanation

Medium. The bypass requires the pool admin to allowlist the router. However, allowlisting the router is the only mechanism available to enable router-based swaps for any permitted user — there is no per-user router delegation. Any pool admin who wants their allowlisted users to be able to use the router (a natural and expected use case) will make this mistake. The bypass is then repeatable by any user with no special privileges.

## Recommendation

The extension must identify the actual user, not the immediate pool caller. Two concrete options:

1. **Trusted router pattern**: Have the router encode the actual user in `extensionData` and have the extension decode and verify it, gating on a trusted-router registry so only the legitimate router can assert a user identity.
2. **Pass-through sender**: Extend the pool's `swap` interface to accept an explicit `onBehalfOf` address (similar to how `addLiquidity` accepts `owner`), and have the router populate it with `msg.sender`. The extension then checks this field instead of `sender`.

## Proof of Concept

```solidity
// Setup
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
// Admin allowlists permitted user and router
ext.setAllowedToSwap(pool, permittedUser, true);
ext.setAllowedToSwap(pool, address(router), true); // required for router use

// Attack: unpermitted user bypasses allowlist via router
vm.prank(unpermittedUser);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: unpermittedUser,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// Swap succeeds — allowlist bypassed
```

The extension sees `sender = address(router)`, which is allowlisted, so the check passes for `unpermittedUser`.
