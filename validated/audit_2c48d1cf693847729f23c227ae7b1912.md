### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension.beforeAddLiquidity` hook validates the `owner` parameter (the position beneficiary, a caller-supplied value) against the allowlist instead of the `sender` parameter (the actual caller who provides the tokens). Because `MetricOmmPool.addLiquidity` accepts `owner` as an arbitrary caller-supplied argument with no `msg.sender == owner` enforcement, any unauthorized address can bypass the allowlist entirely by setting `owner` to any allowlisted address.

---

### Finding Description

In `MetricOmmPool.addLiquidity`, `owner` is a free parameter — there is no check that the caller equals the owner: [1](#0-0) 

Both `msg.sender` (as `sender`) and the caller-supplied `owner` are forwarded to the extension: [2](#0-1) 

The extension receives both arguments but silently discards `sender` (unnamed first parameter) and checks only `owner`: [3](#0-2) 

Contrast this with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (the actual caller): [4](#0-3) 

The inconsistency is structural: `removeLiquidity` enforces `msg.sender == owner`, so the pool correctly treats `sender` as the authoritative actor for withdrawals. The deposit allowlist should mirror this by checking `sender`, not `owner`. [5](#0-4) 

---

### Impact Explanation

The deposit allowlist is completely bypassed. Any unauthorized address (`Bob`) can call `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)`. The extension checks `allowedDepositor[pool][allowlistedAddress]` → passes. Bob provides tokens via the swap callback; the position is minted under the allowlisted address. Bob has injected liquidity into a restricted pool without authorization.

Consequences:
- **Broken access control**: Pools configured for permissioned liquidity (e.g., KYC-gated, curated market-maker-only) accept deposits from any address.
- **Pool manipulation**: An unauthorized actor can add liquidity to specific bins, shifting `curBinIdx`/`curPosInBin` and distorting the oracle-anchored bid/ask execution price for subsequent swaps, harming existing LPs.
- **Coordinated bypass**: If the unauthorized caller controls or coordinates with the allowlisted `owner` address, they can add and then remove liquidity freely, making the allowlist a no-op.

---

### Likelihood Explanation

The bypass requires only a single call with `owner` set to any known allowlisted address. No special privileges, flash loans, or oracle manipulation are needed. Allowlisted addresses are typically discoverable on-chain via emitted events or `allowedDepositor` reads. Likelihood is high.

---

### Recommendation

Check `sender` (the actual caller) instead of `owner` in `beforeAddLiquidity`, mirroring the pattern used in `SwapAllowlistExtension`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol

function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to allowlist position owners (not callers), the pool must also enforce `msg.sender == owner` in `addLiquidity`, as it already does in `removeLiquidity`.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; `allowedDepositor[pool][alice] = true`, `allowAllDepositors[pool] = false`.
2. Bob (not allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. Extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
4. `LiquidityLib.addLiquidity` calls back into Bob to pull tokens; Bob's tokens are deposited.
5. Position shares are minted under `(alice, salt)`.
6. Bob has successfully deposited into a restricted pool. If Bob controls Alice (or coordinates with her), Alice calls `removeLiquidity` to return the tokens, completing a full round-trip through a supposedly gated pool.
