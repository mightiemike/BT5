Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates Position `owner` Instead of Token-Paying `sender`, Enabling Complete Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the address that calls `addLiquidity` and pays tokens via callback) and instead validates the `owner` parameter (the position recipient). Any non-allowlisted address can bypass the guard by supplying any allowlisted address as `owner`, depositing tokens into a restricted pool without authorization. This renders the deposit allowlist completely ineffective.

## Finding Description

`MetricOmmPool.addLiquidity` invokes the extension hook at line 191 as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

Here `msg.sender` is the **sender** — the address that called `addLiquidity` and is the counterparty for the token-pull callback — while `owner` is the **position recipient**, which may be a completely different address.

`DepositAllowlistExtension.beforeAddLiquidity` receives these two arguments but leaves the first unnamed (discarding it) and validates only the second:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

The guard passes as long as `owner` is allowlisted, regardless of who `sender` is. The token-pull callback is issued to `msg.sender` of `addLiquidity` (i.e., `sender`), so the entity actually paying tokens into the pool is never checked.

`SwapAllowlistExtension.beforeSwap` correctly validates the first parameter (`sender`), confirming the inconsistency is a bug:

```solidity
function beforeSwap(address sender, address, bool, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
``` [3](#0-2) 

## Impact Explanation

The pool admin's deposit allowlist is completely ineffective. Any non-allowlisted address can deposit tokens into a restricted pool by supplying any allowlisted address as `owner`. The non-allowlisted address pays the tokens and the allowlisted address receives the position. This breaks the core access-control invariant the extension is designed to enforce, fitting the "admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" category.

## Likelihood Explanation

The bypass requires no special privileges, no flash loans, and no external state manipulation. Any caller can supply an arbitrary `owner`. Allowlisted addresses are typically known (e.g., whitelisted LPs), making the bypass trivially constructable by any non-allowlisted actor. The attack is repeatable at will.

## Recommendation

Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

Also update `setAllowedToDeposit` / `isAllowedToDeposit` documentation to reflect that the allowlist governs the **caller of `addLiquidity`** (the token payer), not the position owner.

## Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; `allowAllDepositors[pool] = false`.
2. Admin calls `setAllowedToDeposit(pool, alice, true)`. Alice is the only allowlisted depositor.
3. Bob (non-allowlisted) calls `pool.addLiquidity(owner = alice, salt, deltas, ...)` directly.
4. `beforeAddLiquidity(/*sender=bob (discarded)*/, owner=alice, ...)` is invoked.
5. Extension checks `allowedDepositor[pool][alice]` → `true` → hook returns success.
6. `LiquidityLib.addLiquidity` proceeds; the callback pulls tokens from Bob.
7. Bob has deposited tokens into the restricted pool and Alice holds the resulting position — the allowlist is fully bypassed. [4](#0-3) [5](#0-4)
