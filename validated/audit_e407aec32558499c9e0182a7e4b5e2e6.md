### Title
Linked Signer Can Permanently Hijack Subaccount Signing Authority via Self-Signed `LinkSigner` Transaction â€” (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` handler inside `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner=true`. Because `Verifier.validateSignature` accepts a signature from either the subaccount owner **or** the current linked signer, a previously-delegated linked signer can craft and submit a new `SignedLinkSigner` payload â€” signed by itself â€” that overwrites `linkedSigners[subaccount]` with an arbitrary attacker-controlled address. No owner participation is required after the initial delegation.

---

### Finding Description

**Root-cause chain:**

**Step 1 â€” `LinkSigner` handler passes `allowLinkedSigner=true`** [1](#0-0) 

The handler decodes a `SignedLinkSigner`, calls `validateSignedTx(..., true)`, then unconditionally writes:
```solidity
linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
```

**Step 2 â€” `validateSignature` accepts the linked signer as a valid signer** [2](#0-1) 

When `allowLinkedSigner=true`, `getLinkedSigner(sender)` is forwarded to the verifier.

**Step 3 â€” Verifier accepts either owner or linked signer** [3](#0-2) 

`recovered == address(uint160(bytes20(sender)))` **OR** `recovered == linkedSigner` â€” both are accepted with equal authority. There is no distinction between "owner signing to delegate" and "delegate signing to re-delegate."

**Exploit path:**

| Step | Actor | Action |
|------|-------|--------|
| 1 | Owner A | Submits `LinkSigner{sender=subaccount_A, signer=B}` signed by A â†’ `linkedSigners[subaccount_A] = B` |
| 2 | Attacker B | Constructs `LinkSigner{sender=subaccount_A, signer=C}`, signs with B's key |
| 3 | Sequencer | Calls `processTransactionImpl`; `validateSignedTx` passes because `recovered == B == linkedSigner` |
| 4 | State | `linkedSigners[subaccount_A] = C` â€” attacker C now controls subaccount A |
| 5 | Attacker C | Signs `WithdrawCollateral`, `LiquidateSubaccount`, etc. on behalf of victim A |

The nonce check (`validateNonce`) does not prevent this â€” B simply uses the next valid nonce for `subaccount_A`. [4](#0-3) 

---

### Impact Explanation

Once `linkedSigners[subaccount_A] = C`, attacker C can sign any transaction that accepts `allowLinkedSigner=true`, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` â€” drain all collateral
- `LiquidateSubaccount` â€” force-liquidate the victim
- `TransferQuote` â€” transfer quote balance out
- Further `LinkSigner` calls â€” lock the owner out permanently

The victim owner A retains their private key but cannot reclaim control until they submit a new `LinkSigner` signed by A â€” however, if C front-runs or the sequencer processes C's transactions first, assets can be drained before recovery.

---

### Likelihood Explanation

Any address that has ever been granted linked-signer status can execute this with no additional privileges. The attack requires only:
1. Knowledge that one is a linked signer for a target subaccount (observable on-chain via `getLinkedSigner`)
2. Ability to submit a signed transaction to the sequencer (the normal user flow)

No admin access, no key compromise, no governance capture required.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner (the address embedded in the `sender` bytes32) should be permitted to reassign signing authority:

```solidity
// core/contracts/EndpointTx.sol â€” LinkSigner handler
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // <-- was true; linked signer must NOT be able to re-delegate
);
```

This mirrors the slow-mode `LinkSigner` path in `processSlowModeTransactionImpl`, which correctly uses `validateSender` (an `msg.sender` check against the owner address) rather than accepting a linked signer. [5](#0-4)

```solidity
it("linked signer hijacks subaccount authority", async () => {
    const [ownerA, signerB, attackerC] = await ethers.getSigners();

    // subaccount_A = bytes32(ownerA.address padded)
    const subaccountA = ethers.utils.hexZeroPad(ownerA.address, 32);

    // Step 1: Owner A links signer B (normal flow)
    const linkB = buildLinkSignerTx(subaccountA, signerB.address, nonce=0);
    await endpoint.processTransactionImpl(signOwner(linkB, ownerA));

    assert.equal(await endpoint.getLinkedSigner(subaccountA), signerB.address);

    // Step 2: Signer B re-links to attacker C â€” signed by B, not A
    const linkC = buildLinkSignerTx(subaccountA, attackerC.address, nonce=1);
    await endpoint.processTransactionImpl(signLinked(linkC, signerB));

    // Step 3: Assert hijack succeeded
    assert.equal(await endpoint.getLinkedSigner(subaccountA), attackerC.address);

    // Step 4: Attacker C withdraws victim's collateral
    const withdraw = buildWithdrawTx(subaccountA, amount, nonce=2);
    await endpoint.processTransactionImpl(signLinked(withdraw, attackerC));
    // withdrawal succeeds â€” victim drained
});
```

The exact storage mutation is `linkedSigners[subaccount_A]` changing from `B` to `C` without any signature or consent from owner A.
