Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of End User, Enabling Full Per-User Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on `sender`, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router contract address. If a pool admin allowlists the router to enable router-based swaps on a curated pool, every user — including those not individually allowlisted — passes the check. The per-user restriction is completely nullified through the canonical periphery swap path.

## Finding Description

**Call chain:**

`MetricOmmPool.swap()` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`_beforeSwap` in `ExtensionCalling.sol` forwards this value as `sender` to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` — the router is `msg.sender` of `pool.swap()`, so `sender` = router address, not the end user: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Contrast with `DepositAllowlistExtension`:**

The deposit extension correctly ignores `sender` (the `LiquidityAdder` contract, first unnamed param) and checks `owner` — the actual position owner passed explicitly: [5](#0-4) 

`MetricOmmPoolLiquidityAdder` always passes the actual user as `positionOwner` to `pool.addLiquidity`: [6](#0-5) 

The deposit allowlist correctly enforces per-user policy through the periphery; the swap allowlist does not. The `beforeSwap` interface has no dedicated "owner" slot — only `sender` (direct caller) and `recipient` (output destination) — so there is no field carrying the end user's identity when the router intermediates.

**No existing guard prevents this.** The `allowAllSwappers` short-circuit only widens access further. There is no mechanism in the router to embed user identity into `extensionData` for the extension to decode.

## Impact Explanation

A pool admin deploying a curated pool with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC'd counterparties) faces a complete bypass when the router is involved. If the admin allowlists `MetricOmmSimpleRouter` to enable router-based swaps — the natural and expected configuration — every user passes the check because the extension sees only the router address. Non-allowlisted users can trade against a pool intended to be private, extracting value from LP positions priced for a restricted counterparty set. This is broken core pool functionality: the curated-pool restriction, a primary purpose of `SwapAllowlistExtension`, is rendered unenforceable through the canonical periphery path.

## Likelihood Explanation

- `SwapAllowlistExtension` is a production periphery contract explicitly designed for curated pools.
- `MetricOmmSimpleRouter` is the canonical swap entry point for EOA users.
- A pool admin enabling router-based swaps on a curated pool will naturally allowlist the router address — the exact action that triggers the bypass.
- No privileged attacker is required; any EOA can call the router.
- The asymmetry with `DepositAllowlistExtension` confirms this is not an intended design choice.

## Recommendation

The `beforeSwap` hook must gate on the actual end user, not the intermediary. Two viable approaches:

1. **Check `recipient` as a proxy for swapper identity** — valid only if the protocol guarantees `recipient == user` for all router paths. Verify this invariant holds across `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput`.

2. **Decode user identity from `extensionData`** — require the router to embed the actual user address in `extensionData`, and have `SwapAllowlistExtension` decode and check it. This mirrors how `addLiquidity` carries `owner` explicitly, and is the robust solution.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true          // alice is individually allowed
  allowedSwapper[pool][bob]   = false         // bob is NOT allowed
  allowedSwapper[pool][router] = true         // admin enables router swaps

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
    → router calls pool.swap(recipient=bob, ...)          [MetricOmmSimpleRouter.sol L72-80]
    → pool calls _beforeSwap(sender=router, ...)          [MetricOmmPool.sol L230-240]
    → SwapAllowlistExtension checks allowedSwapper[pool][router] == true  [SwapAllowlistExtension.sol L37]
    → check passes
    → bob's swap executes against the curated pool
    → bob bypasses the individual allowlist entirely
```

The `allowedSwapper` mapping is keyed by `[pool][sender]`: [7](#0-6) 

and the check at line 37 evaluates `allowedSwapper[msg.sender][sender]` where `sender` is the router, not bob: [8](#0-7)
