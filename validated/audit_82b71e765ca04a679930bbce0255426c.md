After reading all production source files, I need to map the sandwich-attack vulnerability class — **missing input-bounds / slippage protection that lets an attacker supply an unconstrained value and receive an undeserved favourable outcome** — onto this repository.

The closest structural match is in `contract/src/lib.rs`: the deprecated `verify_transaction_inclusion` (v1) is still a live, publicly callable entry point that accepts any `tx_id` without the coinbase-proof guard that was added in v2 precisely to prevent internal-node forgery.

---

### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable Without Coinbase-Proof Validation, Enabling Merkle-Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary
`verify_transaction_inclusion` (v1) was superseded by `verify_transaction_inclusion_v2`, which adds a mandatory coinbase-proof check to prevent the 64-byte-transaction Merkle-proof forgery described at https://www.bitmex.com/blog/64-Byte-Transactions. Despite being marked `#[deprecated]`, v1 carries no access-control restriction and remains callable by any unprivileged NEAR account. An attacker can supply an internal Merkle-tree node hash as `tx_id`, pair it with a structurally valid sibling proof, and cause the function to return `true` for a transaction that was never included in the block.

---

### Finding Description

`verify_transaction_inclusion` (v1) accepts a caller-supplied `tx_id` and a caller-supplied `merkle_proof` and computes:

```
compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.merkle_root
``` [1](#0-0) 

The only guard on `tx_id` is that `merkle_proof` must be non-empty: [2](#0-1) 

There is **no check** that `tx_id` is a leaf-level hash rather than an internal node. Bitcoin's Merkle tree is built by double-SHA256-hashing pairs of 32-byte values; an internal node is itself a 32-byte hash that sits at a valid position in the tree. An attacker who knows the block's Merkle tree can pick any internal node `N` at depth `d`, supply the `d`-element sibling path that leads from `N` to the root, and the equality check passes.

`verify_transaction_inclusion_v2` was introduced specifically to close this gap by requiring a separate coinbase proof that anchors the tree at position 0: [3](#0-2) 

However, v1 is still `pub` with only `#[pause]` (not paused by default) and no `#[private]` or `#[trusted_relayer]` guard: [4](#0-3) 

The `#[deprecated]` attribute emits a Rust compiler warning; it does **not** prevent on-chain invocation. Any NEAR account can call the function directly via a signed transaction.

The `compute_root_from_merkle_proof` helper in `merkle-tools` performs no position-range validation: [5](#0-4) 

---

### Impact Explanation

A downstream NEAR contract that calls `verify_transaction_inclusion` (v1) to gate a cross-chain action (e.g., releasing bridged assets, recording a payment, updating state) will receive `true` for a fabricated `tx_id`. The attacker does not need to mine a block or control a relayer; they only need to read the Merkle tree of any already-accepted block from on-chain state and compute the internal-node proof off-chain. The corrupted proof result directly undermines the SPV guarantee that is the contract's core security property.

---

### Likelihood Explanation

The entry point is unrestricted and permissionless. The required inputs (block hash, Merkle tree structure) are fully observable from public NEAR state and Bitcoin chain data. No privileged role, private key, or social engineering is required. The only prerequisite is that the target block is already in the contract's `headers_pool`, which is the normal operating state.

---

### Recommendation

1. **Remove or gate v1**: Add `#[private]` to `verify_transaction_inclusion`, or delete it entirely. The `#[deprecated]` marker is insufficient.
2. **Redirect callers**: Any integration still using v1 must migrate to `verify_transaction_inclusion_v2`.
3. **Optionally harden v2**: Confirm that the `coinbase_merkle_proof.len() == merkle_proof.len()` equality check (currently the only structural guard) is sufficient for all tree shapes, including single-transaction blocks where the coinbase is also the only transaction.

---

### Proof of Concept

1. Attacker reads the `headers_pool` for a confirmed block `B` with known Merkle tree `[tx0, tx1, tx2, tx3]`.
2. The internal node `I = double_sha256(tx0 || tx1)` is at depth 1.
3. Attacker constructs `merkle_proof = [double_sha256(tx2 || tx3)]` (the sibling of `I` at depth 1).
4. Attacker calls `verify_transaction_inclusion` with:
   - `tx_id = I` (the internal node hash)
   - `tx_index = 0`
   - `tx_block_blockhash = B`
   - `merkle_proof = [double_sha256(tx2 || tx3)]`
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(I, 0, [sibling])` = `double_sha256(I || sibling)` = `merkle_root` of `B`. [6](#0-5) 
6. The function returns `true`. No valid transaction with hash `I` exists in block `B`.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
