### Title
`ValidResourceBounds::AllResources` with Zero L2/Data Gas Is Silently Downgraded to `L1Gas` During P2P Protobuf Deserialization, Producing a Wrong Transaction Hash Preimage - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic — checking whether `l2_gas` and `l1_data_gas` are both zero — to decide between the `L1Gas` (pre-0.13.3) and `AllResources` (post-0.13.3) variants. A post-0.13.3 `AllResources` transaction whose `l2_gas` and `l1_data_gas` happen to be zero is silently re-typed as `L1Gas` after a protobuf round-trip. Because `get_tip_resource_bounds_hash` includes `L1_DATA_GAS` in the hash preimage only for `AllResources`, the two variants produce structurally different hash preimages (3 vs 2 resource elements). Any node that syncs such a transaction via P2P will store it with the wrong variant, causing the OS/SNOS to compute a divergent transaction hash and fail to prove the block.

### Finding Description

**Root cause — protobuf deserialization heuristic:** [1](#0-0) 

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The protobuf wire format carries no type tag distinguishing `L1Gas` from `AllResources`; the deserializer infers the variant purely from whether the numeric values are zero. A post-0.13.3 `AllResources` transaction with `l2_gas = 0` and `l1_data_gas = 0` is indistinguishable from a pre-0.13.3 `L1Gas` transaction after this conversion.

**Serialization side — always emits `l1_data_gas`:** [2](#0-1) 

When an `AllResources` value is serialized, `l1_data_gas` is always written as `Some(zero)`. On the receiving side `unwrap_or_default()` makes it present, so the `None`-branch is never taken; the only discriminant is the zero check, which collapses the two variants.

**Hash preimage divergence:** [3](#0-2) 

`get_tip_resource_bounds_hash` appends `L1_DATA_GAS` to the Poseidon chain **only** for `AllResources`. For `L1Gas` the chain contains `[tip, L1_GAS, L2_GAS]` (3 felts); for `AllResources` it contains `[tip, L1_GAS, L2_GAS, L1_DATA_GAS]` (4 felts). Even when `L1_DATA_GAS = 0`, the extra element changes the Poseidon output.

**OS always expects 3 resource bounds:** [4](#0-3) 

The SNOS Cairo program asserts `n_resource_bounds = 3`. The Rust helper that constructs the OS input: [5](#0-4) 

returns only 2 elements for `L1Gas`, so the OS assertion fires and proving aborts.

**Attack path:**

1. Attacker submits `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The gateway accepts it; `AllResourceBounds` is always treated as `AllResources`. [6](#0-5) 

2. The gateway computes hash **H₁** (4-felt preimage, `AllResources` path) and stores the transaction. [7](#0-6) 

3. The block is proposed and synced to peer nodes via P2P as `protobuf::InvokeV3`. [8](#0-7) 

4. The peer deserializes the transaction: `l2_gas = 0`, `l1_data_gas = 0` → variant becomes `L1Gas`. The peer stores the transaction with the `L1Gas` variant.

5. When the peer runs the SNOS to prove the block, `valid_resource_bounds_as_felts` emits only 2 resource elements. The OS asserts `n_resource_bounds = 3` and aborts. The peer cannot produce a valid proof for the block.

### Impact Explanation

Any sequencer node that syncs a block containing such a transaction via P2P will be unable to prove that block. The SNOS will abort with an assertion failure on `n_resource_bounds`. Additionally, `starknet_traceTransaction` and `starknet_simulateTransactions` on the synced node will compute hash **H₂** ≠ **H₁**, returning an authoritative-looking wrong transaction hash in execution traces. This matches:

- **Critical**: Wrong state/revert result from blockifier/syscall/execution logic for accepted input (proving abort).
- **High**: RPC execution, tracing, simulation returns an authoritative-looking wrong value.

### Likelihood Explanation

Any unprivileged user can craft and submit an `AllResourceBounds` invoke transaction with `l2_gas = 0` and `l1_data_gas = 0`. The gateway imposes no lower bound on these fields. The transaction will be accepted, included in a block, and synced to peers. The trigger requires no special privilege and is reachable from the public RPC endpoint.

### Recommendation

Replace the value-based heuristic with an explicit type tag in the protobuf wire format. Add a boolean or enum field (e.g., `bool is_all_resources = 4`) to `ResourceBounds` so the deserializer can reconstruct the correct variant without inspecting numeric values. Until the wire format is updated, the deserializer should default to `AllResources` for any transaction received in a post-0.13.3 context (e.g., when the block number is above the 0.13.3 activation height), rather than silently downgrading to `L1Gas`.

### Proof of Concept

```rust
// Construct AllResources with zero l2/data gas — accepted by gateway
let resource_bounds = AllResourceBounds {
    l1_gas: ResourceBounds { max_amount: GasAmount(1000), max_price_per_unit: GasPrice(1) },
    l2_gas: ResourceBounds { max_amount: GasAmount(0), max_price_per_unit: GasPrice(0) },
    l1_data_gas: ResourceBounds { max_amount: GasAmount(0), max_price_per_unit: GasPrice(0) },
};
let valid = ValidResourceBounds::AllResources(resource_bounds);

// Serialize to protobuf and back
let proto: protobuf::ResourceBounds = valid.into();
let recovered = ValidResourceBounds::try_from(proto).unwrap();

// Type has changed: AllResources → L1Gas
assert!(matches!(recovered, ValidResourceBounds::L1Gas(_)));  // passes

// Hash preimage differs
let tip = Tip(0);
let h1 = get_tip_resource_bounds_hash(&valid, &tip).unwrap();
let h2 = get_tip_resource_bounds_hash(&recovered, &tip).unwrap();
assert_ne!(h1, h2);  // passes — different number of Poseidon inputs
```

The `AllResources` hash includes `L1_DATA_GAS = 0` as a fourth Poseidon input; the `L1Gas` hash omits it. The SNOS Cairo program then asserts `n_resource_bounds = 3` and aborts when fed the 2-element `L1Gas` representation.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

pub fn valid_resource_bounds_as_felts(
    resource_bounds: &ValidResourceBounds,
    exclude_l1_data_gas: bool,
) -> Result<Vec<ResourceAsFelts>, StarknetApiError> {
    let mut resource_bounds_vec: Vec<_> = vec![
        ResourceAsFelts::new(Resource::L1Gas, &resource_bounds.get_l1_bounds())?,
        ResourceAsFelts::new(Resource::L2Gas, &resource_bounds.get_l2_bounds())?,
    ];
    if exclude_l1_data_gas {
        return Ok(resource_bounds_vec);
    }
    if let ValidResourceBounds::AllResources(AllResourceBounds { l1_data_gas, .. }) =
        resource_bounds
    {
        resource_bounds_vec.push(ResourceAsFelts::new(Resource::L1DataGas, l1_data_gas)?)
    }
    Ok(resource_bounds_vec)
}
