### Title
AuxPoW Chain Merkle Root Byte-Order Mismatch Causes All Dogecoin AuxPoW Blocks to Be Rejected — (`contract/src/dogecoin.rs`)

### Summary

The `check_aux` function in the Dogecoin module searches for the chain merkle root inside the parent coinbase script using `chain_root.to_string()`. The `Display` implementation for `H256` reverses the byte array before hex-encoding (producing the human-readable "display" format), but the merged-mining specification embeds the chain merkle root in the coinbase script as raw little-endian bytes. The two representations never match, so the `find` always returns `None` and the contract panics, rejecting every valid AuxPoW block.

### Finding Description

`H256::fmt` (the `Display` impl) reverses the 32-byte array before encoding:

```rust
// btc-types/src/hash.rs  lines 20-24
fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
    let reversed: Vec<u8> = self.0.into_iter().rev().collect();
    write!(f, "{}", hex::encode(reversed))
}
``` [1](#0-0) 

`block_hash()` and `compute_root_from_merkle_proof` both operate on and return raw (little-endian internal) bytes:

```rust
// btc-types/src/btc_header.rs  lines 33-36
pub fn block_hash(&self) -> H256 {
    let block_header = self.get_block_header_vec();
    double_sha256(&block_header)
}
``` [2](#0-1) 

`check_aux` computes `chain_root` in internal format, then converts it to display format via `.to_string()` before searching the coinbase hex string:

```rust
// contract/src/dogecoin.rs  lines 78-104
let chain_root = merkle_tools::compute_root_from_merkle_proof(
    block_header.block_hash(),
    aux_data.chain_id,
    &aux_data.chain_merkle_proof,
);
...
let script_sig = coinbase_tx.input.first().unwrap()
    .script_sig.to_hex_string();          // raw bytes, hex-encoded (internal format)
...
let mut pos_chain_root = script_sig
    .find(&chain_root.to_string())        // reversed bytes, hex-encoded (display format)
    .expect("Aux POW missing chain merkle root in parent coinbase");
``` [3](#0-2) 

`script_sig.to_hex_string()` from the `bitcoin` crate produces the hex of the raw script bytes. The merged-mining specification embeds the chain merkle root as raw little-endian bytes inside the coinbase script. `chain_root.to_string()` produces the byte-reversed display format. These two hex strings are always different (unless the root is a palindrome, which is cryptographically negligible), so `find` always returns `None` and the `expect` always panics.

### Impact Explanation

Every Dogecoin block carrying AuxPoW data (`aux_data = Some(...)`) is unconditionally rejected with a panic. Dogecoin switched to mandatory merged mining at block 371 337; after that height every main-chain block is an AuxPoW block. The relayer cannot submit any main-chain block, so:

- `mainchain_tip_blockhash` is permanently stuck.
- `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` return false for every transaction on the real Dogecoin chain.
- Any consumer contract relying on the light client for cross-chain verification receives incorrect results.

### Likelihood Explanation

The bug is triggered by normal operation: the legitimate relayer submitting any post-371 337 Dogecoin block. No adversarial input is required. The mismatch is deterministic and affects 100 % of AuxPoW submissions.

### Recommendation

Replace `chain_root.to_string()` with the raw little-endian hex encoding so that the search string matches the bytes actually embedded in the coinbase script:

```rust
// Use internal-format hex (no byte reversal)
let chain_root_hex = hex::encode(&chain_root.0);
let mut pos_chain_root = script_sig
    .find(&chain_root_hex)
    .expect("Aux POW missing chain merkle root in parent coinbase");
```

### Proof of Concept

1. Take any real Dogecoin main-chain block after height 371 337 (all are AuxPoW).
2. Submit it via `submit_blocks` with the corresponding `AuxData`.
3. `check_aux` computes `chain_root` (internal format), calls `.to_string()` (reversed format), and calls `script_sig.find(...)`.
4. The coinbase script contains the chain root in internal format; the search string is in reversed format → `find` returns `None` → `expect` panics → block is rejected.
5. Repeat for every subsequent block: the contract's tip never advances past the last pre-AuxPoW block.
