### Title
IBC NFT Denom Creator Invariant Broken by Unprivileged Pre-Creation — (`x/nft-transfer/keeper/packet.go`)

### Summary

The `x/nft-transfer` module assumes that any IBC voucher denom stored in `x/nft` was created with the channel's escrow address as its `creator`. An unprivileged user can break this invariant by calling `MsgIssueDenom` with the deterministically predictable IBC denom ID before the first IBC packet for that class arrives. When the packet is later processed, `processReceivedPacket` silently skips denom creation (the ID already exists) but then calls `MintNFT` with the escrow address as `sender`; `IsDenomCreator` rejects it because the real creator is the attacker. The packet is permanently rejected with an error acknowledgement for every future attempt on that class.

### Finding Description

**Invariant that must hold:** Every IBC voucher denom in `x/nft` must have `Denom.Creator == GetEscrowAddress(destPort, destChannel)`. The `MintNFT` path in `processReceivedPacket` relies on this unconditionally.

**How the invariant is set:** In `processReceivedPacket`, when a new IBC class arrives for the first time:

```go
// x/nft-transfer/keeper/packet.go:167-168
if !k.nftKeeper.HasDenomID(ctx, voucherClassID) {
    if err := k.nftKeeper.IssueDenom(ctx, voucherClassID, voucherClassID, "", data.ClassUri, escrowAddress); err != nil {
        return err
    }
}
``` [1](#0-0) 

**How the invariant is broken:** The guard is `HasDenomID` only. If the denom already exists (with any creator), the `IssueDenom` call is skipped entirely. There is no follow-up check that `denom.Creator == escrowAddress`.

**Subsequent failure:** Immediately after the guard, `MintNFT` is called with `escrowAddress` as the `sender`:

```go
// x/nft-transfer/keeper/packet.go:181-183
for i, tokenID := range data.TokenIds {
    if err := k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, receiver); err != nil {
        return err
    }
}
``` [2](#0-1) 

`MintNFT` calls `IsDenomCreator(ctx, denomID, sender)`:

```go
// x/nft/keeper/keeper.go:76-78
_, err := k.IsDenomCreator(ctx, denomID, sender)
if err != nil {
    return err
}
``` [3](#0-2) 

`IsDenomCreator` compares `denom.Creator` against `sender` (the escrow address). If the attacker pre-created the denom, `denom.Creator` is the attacker's address, not the escrow address, so the check fails and the error propagates back through `OnRecvPacket`. [4](#0-3) 

**Attacker-controlled entry path:** `MsgIssueDenom` is a standard, unprivileged transaction. The keeper's `SetDenom` only enforces uniqueness of `id` and `name`; there is no restriction on the `ibc/` prefix:

```go
// x/nft/keeper/denom.go:26-39
func (k Keeper) SetDenom(ctx sdk.Context, denom types.Denom) error {
    if k.HasDenomID(ctx, denom.Id) { ... }
    if k.HasDenomNm(ctx, denom.Name) { ... }
    ...
}
``` [5](#0-4) 

The IBC voucher class ID is `ibc/<sha256(port/channel/baseClassID)>`, fully deterministic from public on-chain data (channel configuration and source-chain class ID). An attacker computes this hash, submits `MsgIssueDenom` with `id = "ibc/<hash>"` and any unique `name`, and permanently occupies the slot with themselves as creator.

### Impact Explanation

The corrupted on-chain value is `Denom.Creator` for the IBC voucher denom: it is set to the attacker's address instead of the channel escrow address. This desynchronizes the `x/nft` denom state from the `x/nft-transfer` module's assumption, permanently breaking `processReceivedPacket` for every future packet carrying that NFT class over that channel. The source-chain NFT is returned to the sender via the error-acknowledgement path (`refundPacketToken` → `TransferOwner` back to sender), so no tokens are destroyed, but the IBC NFT transfer capability for that class is permanently and irrecoverably blocked on this chain.

### Likelihood Explanation

The attack requires no privileged keys. Both inputs (IBC channel identifiers and source-chain NFT class ID) are public. The attacker only needs to act before the first IBC packet for that class is relayed — a window that can be days or weeks for newly announced NFT collections. The transaction cost is a single `MsgIssueDenom`.

### Recommendation

In `processReceivedPacket`, after the `HasDenomID` guard, verify that the existing denom's creator matches the expected escrow address before proceeding to mint:

```go
if !k.nftKeeper.HasDenomID(ctx, voucherClassID) {
    if err := k.nftKeeper.IssueDenom(ctx, voucherClassID, voucherClassID, "", data.ClassUri, escrowAddress); err != nil {
        return err
    }
} else {
    // Invariant check: existing denom must have been created by this channel's escrow address.
    denom, err := k.nftKeeper.GetDenom(ctx, voucherClassID)
    if err != nil {
        return err
    }
    creator, err := sdk.AccAddressFromBech32(denom.Creator)
    if err != nil || !creator.Equals(escrowAddress) {
        return sdkerrors.Wrapf(types.ErrInvalidDenom,
            "IBC denom %s exists but creator %s does not match escrow address %s",
            voucherClassID, denom.Creator, escrowAddress)
    }
}
```

Alternatively, restrict `MsgIssueDenom` from accepting IDs that match the `ibc/` prefix pattern at the message-validation layer.

### Proof of Concept

1. Observe (or predict) that chain B will receive NFT class `nftClass` from chain A over channel `channel-0` / port `nft-transfer`.
2. Compute `classTrace = "nft-transfer/channel-0/nftClass"`, `voucherClassID = "ibc/" + hex(sha256(classTrace))`.
3. Submit `MsgIssueDenom{id: voucherClassID, name: "attacker-denom", sender: attacker}` on chain B. This succeeds because the ID does not yet exist.
4. Relay the IBC packet from chain A to chain B. `processReceivedPacket` finds `HasDenomID(voucherClassID) == true`, skips `IssueDenom`, then calls `MintNFT(..., escrowAddress, receiver)`.
5. `IsDenomCreator` returns `ErrUnauthorized` because `denom.Creator == attacker ≠ escrowAddress`. The packet is acknowledged with an error.
6. Chain A's `OnAcknowledgementPacket` calls `refundPacketToken`, which calls `TransferOwner` to return the NFT to the original sender — confirming no fund loss but confirming the permanent block.
7. Every subsequent attempt to transfer any token of `nftClass` from chain A to chain B over `channel-0` fails identically.

func (k Keeper) SetDenom(ctx sdk.Context, denom types.Denom) error {
	if k.HasDenomID(ctx, denom.Id) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomID %s has already exists", denom.Id)
	}

	if k.HasDenomNm(ctx, denom.Name) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomName %s has already exists", denom.Name)
	}

	store := ctx.KVStore(k.storeKey)
	bz := k.cdc.MustMarshal(&denom)
	store.Set(types.KeyDenomID(denom.Id), bz)
	store.Set(types.KeyDenomName(denom.Name), []byte(denom.Id))
	return nil
```

```go
func (k Keeper) IsDenomCreator(ctx sdk.Context, denomID string, address sdk.AccAddress) (types.Denom, error) {
	denom, err := k.GetDenom(ctx, denomID)
	if err != nil {
		return types.Denom{}, err
	}

	creator, err := sdk.AccAddressFromBech32(denom.Creator)
	if err != nil {
		panic(err)
	}

	if !creator.Equals(address) {
		return types.Denom{}, sdkerrors.Wrapf(types.ErrUnauthorized, "%s is not the creator of %s", address, denomID)
	}

	return denom, nil
```
