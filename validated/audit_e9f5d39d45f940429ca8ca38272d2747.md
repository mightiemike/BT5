### Title
Rebasing Token Collateral Growth Permanently Locked in Clearinghouse With No Recovery Path — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

When a rebasing token (e.g., an Aave aToken whose `balanceOf()` increases passively over time) is listed as a supported spot collateral product, the actual token balance held by `Clearinghouse` grows continuously due to rebasing. However, `SpotEngine`'s internal accounting only tracks the originally deposited amount scaled by the protocol's own interest-rate model. The excess rebasing growth is never credited to any user's balance and is permanently locked in the `Clearinghouse` contract. No owner or user function exists to recover it.

---

### Finding Description

**Deposit path — only the stated amount is recorded:**

`Endpoint.depositCollateralWithReferral` calls `handleDepositTransfer`, which physically moves `amount` tokens into `Clearinghouse`. [1](#0-0) 

`Clearinghouse.depositCollateral` then records exactly `amountRealized = int128(txn.amount) * multiplier` into `SpotEngine` via `updateBalance`. [2](#0-1) 

**Withdrawal path — only the recorded amount is returned:**

`withdrawCollateral` transfers the caller-requested `amount` and decrements the stored balance by the same `amountRealized`. [3](#0-2) 

**Internal accounting never observes actual token balance:**

`SpotEngine.assertUtilization` only verifies `totalDeposits >= totalBorrows` — a purely internal invariant that does not compare against `token.balanceOf(clearinghouse)`. [4](#0-3) 

`Clearinghouse.manualAssert` checks that the sequencer's off-chain state matches the on-chain `State` struct (`cumulativeDepositsMultiplierX18`, `totalDepositsNormalized`, etc.), which is also purely internal and never reads the actual token balance. [5](#0-4) 

**The `cumulativeDepositsMultiplierX18` grows only from the protocol's own interest model**, not from rebasing token growth. [6](#0-5) 

**No recovery mechanism exists.** The entire `Clearinghouse.sol` contains no function to sweep or redistribute excess token balance. The only owner-callable token-recovery helper (`withdrawFromDirectDepositV1`) is scoped exclusively to `DirectDepositV1` satellite contracts, not to `Clearinghouse` itself. [7](#0-6) 

The `Airdrop` contract has the same structural gap: it distributes fixed Merkle-leaf amounts and has no sweep function, so any rebasing growth on the airdrop token balance is also permanently locked. [8](#0-7) 

---

### Impact Explanation

A user who deposits a rebasing token (e.g., aUSDC) as collateral will only ever be able to withdraw the amount the protocol internally recorded at deposit time, scaled by the protocol's own interest rate. The additional balance that accrues in `Clearinghouse` due to rebasing is untracked, uncredited, and unrecoverable. The broken invariant is:

```
actual token.balanceOf(clearinghouse)
  > (totalDepositsNormalized × cumulativeDepositsMultiplierX18)
    - (totalBorrowsNormalized × cumulativeBorrowsMultiplierX18)
    + insurance
```

The surplus is permanently locked with no path for any party — user or owner — to claim it.

---

### Likelihood Explanation

Likelihood is **low-to-medium**. It requires a rebasing token to be listed as a supported collateral product by the owner via `SpotEngine.addOrUpdateProduct`. The protocol imposes no on-chain guard against this; the `IERC20Base` interface is generic and accepts any ERC-20-compatible token. [9](#0-8)  If a rebasing token such as an Aave aToken is ever listed (a realistic scenario given Nado's positioning as a multi-collateral perpetuals protocol), every depositor of that token silently loses their rebasing yield with no recourse.

---

### Recommendation

1. **Explicit exclusion**: Document and enforce on-chain that rebasing tokens are not supported collateral. A simple check comparing `balanceOf` before and after `transferFrom` and reverting on a mismatch would reject rebasing tokens at deposit time.
2. **Excess-sweep function**: Add an owner-callable function on `Clearinghouse` that computes `token.balanceOf(address(this)) - trackedDeposits + trackedBorrows` and distributes or withdraws the surplus, analogous to the `withdrawExcessRewards()` pattern referenced in the external report.
3. **Airdrop**: Add a similar owner-callable sweep to `Airdrop.sol` for any token balance exceeding the sum of unclaimed Merkle amounts.

---

### Proof of Concept

1. Owner calls `SpotEngine.addOrUpdateProduct` listing an Aave aUSDC (rebasing) token as product `productId = 5`.

### Title
Rebasing Token Yield Permanently Locked in Clearinghouse with No On-Chain Recovery Mechanism — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

The `Clearinghouse` holds actual ERC20 tokens but tracks user balances through a normalized internal accounting system in `SpotEngine`. If a rebasing token (e.g., Aave aUSDC) is registered as a product, the rebasing yield that accrues to `Clearinghouse`'s actual token balance is invisible to the internal accounting, cannot be distributed to depositors, and has no on-chain withdrawal path. The excess is permanently locked or silently captured by the sequencer via `X_ACCOUNT`.

---

### Finding Description

**Deposit path — internal accounting records a fixed amount:**

When a user calls `depositCollateralWithReferral`, `handleDepositTransfer` moves tokens to `Clearinghouse` and `depositCollateral` records the deposited amount in `SpotEngine` as a normalized balance: [1](#0-0) [2](#0-1) 

The `SpotEngine` internal accounting grows only via the protocol's own interest rate model (`_updateState`), which computes `cumulativeDepositsMultiplierX18` from utilization ratios — entirely disconnected from the actual token balance held by `Clearinghouse`: [10](#0-9) 

**Rebasing yield accumulates silently:**

For a rebasing token, `token.balanceOf(Clearinghouse)` grows over time independently of the protocol's multipliers. The divergence — `actual_balance − (totalDepositsNormalized × cumulativeDepositsMultiplierX18 − totalBorrowsNormalized × cumulativeBorrowsMultiplierX18)` — is the rebasing yield. No on-chain function in `Clearinghouse` reads `token.balanceOf(address(this))` and reconciles it against internal accounting.

**No recovery mechanism exists:**

`assertUtilization` only checks internal accounting consistency (`totalDeposits >= totalBorrows`), not actual token balance: [4](#0-3) 

`manualAssert` checks that the sequencer's off-chain state matches on-chain `states` (the normalized multipliers), but never checks actual token balances: [5](#0-4) [11](#0-10) 

`Clearinghouse` has no `withdrawExcessTokens` or equivalent function. The only owner-level escape hatch is `BaseWithdrawPool.removeLiquidity`, which applies to `WithdrawPool`, not `Clearinghouse`: [12](#0-11) 

**Withdrawal path — user receives only the internally tracked amount:**

`withdrawCollateral` transfers exactly `amount` (the internally tracked value) and decrements the SpotEngine balance by `amountRealized`. The rebasing surplus is never touched: [3](#0-2) 

**`Airdrop` has the same structural gap:**

The `Airdrop` contract distributes a fixed `totalAmount` per Merkle leaf. If the airdrop token is rebasing, the yield that accrues to `Airdrop`'s balance between `registerMerkleRoot` and `claim` is locked with no withdrawal path: [13](#0-12) 

---

### Impact Explanation

Users who deposit rebasing tokens (e.g., aUSDC) lose all rebasing yield. The excess tokens accumulate in `Clearinghouse` with no on-chain distribution mechanism. The sequencer could credit the surplus to `X_ACCOUNT` off-chain and extract it via `rebalanceXWithdraw`, meaning the yield is silently captured as protocol revenue rather than returned to depositors. This is a direct, quantifiable financial loss proportional to the rebasing rate and the total deposited rebasing collateral.

---

### Likelihood Explanation

`SpotEngine.addOrUpdateProduct` accepts any ERC20 token address with no rebasing-token guard. If a rebasing token such as aUSDC or stETH is registered as a product — a realistic scenario given the protocol's stated goal of supporting diverse collateral — every depositor of that token is affected for the entire duration of their deposit. [14](#0-13) 

---

### Recommendation

1. **Explicit exclusion**: Add a guard in `addOrUpdateProduct` that rejects known rebasing token patterns, or document that rebasing tokens are unsupported and enforce this off-chain.
2. **Excess sweep function**: Add an owner-callable `sweepExcess(uint32 productId)` to `Clearinghouse` that computes `token.balanceOf(address(this)) − internallyTrackedBalance` and credits the difference to depositors pro-rata (or to a designated account), analogous to the `withdrawExcessRewards` pattern referenced in the external report.
3. **`manualAssert` extension**: Extend the sequencer's `manualAssert` check to include actual token balances, so any divergence caused by rebasing is surfaced as a desync error rather than silently accumulating.

---

### Proof of Concept

1. Owner calls `SpotEngine.addOrUpdateProduct` with `token = aUSDC` (Aave's rebasing USDC).
2. User deposits 1,000 aUSDC via `Endpoint.depositCollateralWithReferral`. `Clearinghouse` receives 1,000 aUSDC; SpotEngine records 1,000 units (normalized).
3. Six months pass. aUSDC rebases at 5% APY. `aUSDC.balanceOf(Clearinghouse)` is now 1,025. SpotEngine's internal accounting reflects only the protocol's own interest rate (e.g., 2% from utilization), so the tracked balance is ~1,010.
4. User calls `withdrawCollateral` for their full balance. They receive ~1,010 aUSDC. The 15 aUSDC rebasing surplus remains in `Clearinghouse`.
5. No on-chain function can distribute or recover this surplus. The sequencer may credit it to `X_ACCOUNT` off-chain, but there is no on-chain enforcement or user recourse.

    function _updateState(
        uint32 productId,
        State memory state,
        uint128 dt
    ) internal {
        int128 borrowRateMultiplierX18;
        int128 totalDeposits = state.totalDepositsNormalized.mul(
            state.cumulativeDepositsMultiplierX18
        );
        int128 totalBorrows = state.totalBorrowsNormalized.mul(
            state.cumulativeBorrowsMultiplierX18
        );
        int128 utilizationRatioX18 = totalBorrows.div(totalDeposits);
        int128 minDepositRateX18;
        {
            Config memory config = configs[productId];

            // annualized borrower rate
            int128 borrowerRateX18 = config.interestFloorX18;
            if (utilizationRatioX18 == 0) {
                // setting borrowerRateX18 to 0 here has the property that
                // adding a product at the beginning of time and not using it until time T
                // results in the same state as adding the product at time T
                borrowerRateX18 = 0;
            } else if (utilizationRatioX18 < config.interestInflectionUtilX18) {
                borrowerRateX18 += config
                    .interestSmallCapX18
                    .mul(utilizationRatioX18)
                    .div(config.interestInflectionUtilX18);
            } else {
                borrowerRateX18 +=
                    config.interestSmallCapX18 +
                    config.interestLargeCapX18.mul(
                        (
                            (utilizationRatioX18 -
                                config.interestInflectionUtilX18).div(
                                    ONE - config.interestInflectionUtilX18
                                )
                        )
                    );
            }

            // convert to per second
            borrowerRateX18 = borrowerRateX18.div(
                MathSD21x18.fromInt(31536000)
            );
            borrowRateMultiplierX18 = (ONE + borrowerRateX18).pow(int128(dt));
            minDepositRateX18 = config.minDepositRateX18;
        }

        external
        onlyOwner
    {
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");

            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");

                ""
            );

            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));
            require(postBalance > preBalance, "empty");
            IERC20Base(token).safeTransfer(
                msg.sender,
                postBalance - preBalance
            );
