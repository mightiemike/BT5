### Title
`Clearinghouse.initialize` Does Not Transfer Ownership to Designated Admin, Propagating Deployer Address as Admin to All Engines — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.initialize` calls `__Ownable_init()` but never calls `transferOwnership()`. The deployer (`msg.sender`) permanently retains ownership. When `addEngine` is subsequently called (which is `onlyOwner`), it passes `owner()` — the deployer — as the `_admin` argument to each engine's `initialize`, causing `SpotEngine` and `PerpEngine` to be permanently initialized with the deployer as their admin instead of the intended multisig.

---

### Finding Description

`Clearinghouse.initialize` accepts no `_admin` parameter and does not call `transferOwnership` after `__Ownable_init()`:

```solidity
function initialize(
    address _endpoint,
    address _quote,
    address _clearinghouseLiq,
    uint256 _spreads,
    address _withdrawPool
) external initializer {
    __Ownable_init();          // owner = msg.sender (deployer)
    setEndpoint(_endpoint);
    // ... no transferOwnership call
}
```

`Clearinghouse.addEngine` is `onlyOwner` and passes `owner()` directly as the admin to each engine's `initialize`:

```solidity
function addEngine(...) external onlyOwner {
    ...
    productEngine.initialize(
        address(this),
        offchainExchange,
        quote,
        getEndpoint(),
        owner()   // <-- passes current owner (deployer) as admin to engine
    );
}
```

`BaseEngine._initialize` then permanently sets the engine's owner to whatever `owner()` returned:

```solidity
function _initialize(..., address _admin) internal initializer {
    __Ownable_init();
    setEndpoint(_endpointAddr);
    transferOwnership(_admin);   // engine owner = deployer, not multisig
    ...
}
```

This contrasts with `ContractOwner.initialize`, which correctly transfers ownership to the multisig:

```solidity
function initialize(address multisig, ...) external initializer {
    __Ownable_init();
    transferOwnership(multisig);   // correct pattern
    ...
}
```

---

### Impact Explanation

**Impact: Medium**

The deployer retains `onlyOwner` control over `Clearinghouse`, including:
- `setWithdrawPool` — can redirect all user withdrawals to an arbitrary address, enabling theft of user funds in transit.
- `setSpreads` — can manipulate spread-based health calculations, enabling artificial insolvency or liquidation bypass for any subaccount.
- `addEngine` — controls which engine addresses are registered and passes itself as admin to each.

All engines (`SpotEngine`, `PerpEngine`) are initialized with the deployer as admin, giving the deployer `onlyOwner` control over `addOrUpdateProduct` on both engines, allowing arbitrary manipulation of risk weights, size increments, and minimum sizes for all trading products.

The intended multisig (set as owner of `ContractOwner`) has no ownership over `Clearinghouse` or the engines, making the intended governance structure non-functional until a manual `transferOwnership` is performed.

---

### Likelihood Explanation

**Likelihood: High**

There is no enforcement mechanism requiring the deployer to call `transferOwnership` after initialization. The deployer must remember to do so manually as a post-deployment step. The pattern is especially likely to be missed because `ContractOwner.initialize` does correctly transfer ownership to the multisig, creating a false sense of security that the ownership structure is properly configured. The absence of an `_admin` parameter in `Clearinghouse.initialize` (unlike `BaseEngine._initialize` which explicitly accepts `_admin`) means there is no compile-time or runtime reminder to set the correct admin.

---

### Recommendation

Add an `_admin` parameter to `Clearinghouse.initialize` and call `transferOwnership(_admin)` immediately after `__Ownable_init()`, mirroring the pattern already used in `BaseEngine._initialize`:

```solidity
function initialize(
    address _endpoint,
    address _quote,
    address _clearinghouseLiq,
    uint256 _spreads,
    address _withdrawPool,
    address _admin          // add this
) external initializer {
    __Ownable_init();
    transferOwnership(_admin);  // add this
    setEndpoint(_endpoint);
    ...
}
```

---

### Proof of Concept

1. Deployer calls `Clearinghouse.initialize(endpoint, quote, clearinghouseLiq, spreads, withdrawPool)` → `owner()` = deployer EOA.
2. Deployer calls `Clearinghouse.addEngine(spotEngineAddr, offchainExchange, SPOT)` → `SpotEngine.initialize(..., owner())` → `SpotEngine` owner permanently set to deployer.
3. Deployer calls `Clearinghouse.addEngine(perpEngineAddr, offchainExchange, PERP)` → `PerpEngine.initialize(..., owner())` → `PerpEngine` owner permanently set to deployer.
4. Deployer calls `Clearinghouse.setWithdrawPool(attackerAddress)` → all subsequent user withdrawals routed to attacker-controlled address.
5. Deployer calls `SpotEngine.addOrUpdateProduct(productId, ...)` with manipulated `longWeightInitial`/`shortWeightInitial` risk parameters → users' collateral health is miscalculated, enabling unfair liquidations or bypassing margin requirements.

The intended multisig (owner of `ContractOwner`) cannot call any `onlyOwner` function on `Clearinghouse` or the engines because it is not the owner of those contracts.
