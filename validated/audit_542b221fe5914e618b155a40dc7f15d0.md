### Title
`ShardLayoutV3` Ancestor-Only Check in `check_if_descendant_of_tracked_shard_impl` Silently Drops Directly-Configured Shard Tracking — (`File: chain/epoch-manager/src/shard_tracker.rs`)

---

### Summary

When the epoch's shard layout is `ShardLayoutV3` (activated by `DynamicResharding`), the function `check_if_descendant_of_tracked_shard_impl` takes an early-return fast path that compares only the shard's **ancestors** against `tracked_shards`. It never checks whether the shard itself is in `tracked_shards`. The non-V3 path correctly checks the shard itself first. The result is that any node configured with `TrackedShardsConfig::Shards([uid])` where `uid` is a current V3 shard will silently fail to track that shard after the first dynamic resharding event.

---

### Finding Description

`check_if_descendant_of_tracked_shard_impl` in `chain/epoch-manager/src/shard_tracker.rs` has two branches:

**Non-V3 path (correct):**
```rust
let mut shard_uid = ShardUId::from_shard_id_and_layout(shard_id, &shard_layout);
if tracked_shards.contains(&shard_uid) {   // ← checks self first
    return Ok(true);
}
// then walks layout history to check ancestors
```

**V3 fast path (broken):**
```rust
if let Some(ancestors) = shard_layout.ancestor_uids(shard_id) {
    let ancestors = HashSet::from_iter(ancestors);
    return Ok(!ancestors.is_disjoint(&tracked_shards));  // ← only checks ancestors, never self
}
``` [1](#0-0) 

`ShardLayoutV3::ancestor_uids` returns the parent, grandparent, etc. of a shard — it never includes the shard itself:

```rust
pub fn ancestor_uids(&self, shard_id: ShardId) -> Vec<ShardUId> {
    self.shards_ancestor_map
        .get(&shard_id)
        .map(|ancestor_ids| ancestor_ids.iter().map(|id| ShardUId::new(VERSION, *id)).collect())
        .unwrap_or_default()
}
``` [2](#0-1) 

`validate_and_derive_shard_ancestor_map` inserts every current shard into `shards_ancestor_map`, including those with no ancestors (empty `Vec`):

```rust
for shard_id in shard_ids {
    let mut ancestors = vec![];
    // ... walk parent chain ...
    shards_ancestor_map.insert(*shard_id, ancestors);  // empty vec for root shards
}
``` [3](#0-2) 

Because `ShardLayout::ancestor_uids` wraps the inner call in `Some(...)` for all V3 layouts, the fast path is always taken for V3, and the `is_disjoint` check on an empty ancestor set always returns `true` → the function returns `false` for any directly-configured current shard. [4](#0-3) 

A second compounding issue: `ancestor_uids` stamps all ancestor shard IDs with the hardcoded `VERSION = 3`, regardless of the layout version those ancestors actually belonged to (V1 or V2). A node that configured `tracked_shards` before V3 was introduced would have stored UIDs with `version=1` or `version=2`. The `is_disjoint` comparison against V3-stamped UIDs would always fail even for the ancestor check. [5](#0-4) 

---

### Impact Explanation

`check_if_descendant_of_tracked_shard_impl` is the sole gate for `TrackedShardsConfig::Shards` in `tracks_shard_at_epoch`: [6](#0-5) 

`tracks_shard_at_epoch` drives `cares_about_shard`, `will_care_about_shard`, `should_apply_chunk`, and `get_tracked_shards_for_non_validator_in_epoch`. A node that silently stops tracking a shard will:

- Stop applying chunks for that shard.
- Stop maintaining flat storage and memtrie for that shard.
- Fail to serve RPC state queries for accounts in that shard.
- Lose the ability to participate in stateless validation for that shard.

The result is a silent, permanent loss of shard coverage with no error or panic — the node continues running but is functionally broken for the affected shard.

---

### Likelihood Explanation

`DynamicResharding` is assigned protocol version 85, the current stable version: [7](#0-6) 

Once 80% of stake upgrades and the first dynamic resharding event fires, every node using `TrackedShardsConfig::Shards` is affected. `TrackedShardsConfig::Shards` is a public, documented config variant: [8](#0-7) 

No privileged role is required — any operator who configures `tracked_shards_config` with explicit shard UIDs triggers the bug automatically at the first V3 resharding epoch boundary.

---

### Recommendation

In `check_if_descendant_of_tracked_shard_impl`, add a self-check before the ancestor disjoint test, mirroring the non-V3 path:

```rust
if let Some(ancestors) = shard_layout.ancestor_uids(shard_id) {
    // Check the shard itself first (a shard is a descendant of itself).
    let shard_uid = ShardUId::from_shard_id_and_layout(shard_id, &shard_layout);
    if tracked_shards.contains(&shard_uid) {
        return Ok(true);
    }
    let ancestors = HashSet::from_iter(ancestors);
    return Ok(!ancestors.is_disjoint(&tracked_shards));
}
```

Additionally, `ancestor_uids` should reconstruct each ancestor's `ShardUId` using the layout version that ancestor actually belonged to, not the hardcoded `VERSION = 3`, so that nodes whose `tracked_shards` were configured under V1/V2 layouts can still match their ancestors correctly after a V3 transition.

### Proof of Concept

```
Setup:
  - Network activates DynamicResharding (protocol version 85).
  - One resharding event fires: shard 0 (V2, version=2) splits into
    shards 1 and 2 (V3, version=3).
  - Node config: tracked_shards_config = Shards([ShardUId { version: 3, shard_id: 1 }])

Execution path for tracks_shard_at_epoch(shard_id=1, epoch=post-reshard):
  1. shard_layout is ShardLayoutV3.
  2. shard_layout.ancestor_uids(1) → Some([ShardUId { version:3, shard_id:0 }])
  3. ancestors = { ShardUId{v3,s0} }
  4. tracked_shards = { ShardUId{v3,s1} }
  5. ancestors.is_disjoint(&tracked_shards) → true  (s0 ≠ s1)
  6. returns false  ← node does NOT track shard 1

Expected: returns true (shard 1 is explicitly configured).

For a root shard with no ancestors (e.g., shard 1 that was never split):
  2. shard_layout.ancestor_uids(1) → Some([])   (empty vec)
  3. ancestors = {}
  5. {}.is_disjoint(&anything) → true always
  6. returns false  ← node does NOT track shard 1 regardless of config
``` [9](#0-8) [10](#0-9)

fn check_if_descendant_of_tracked_shard_impl(
    shard_id: ShardId,
    tracked_shards: &Vec<ShardUId>,
    epoch_id: &EpochId,
    epoch_manager: &Arc<dyn EpochManagerAdapter>,
) -> Result<bool, EpochError> {
    let tracked_shards: HashSet<ShardUId> = tracked_shards.into_iter().cloned().collect();
    let protocol_version = epoch_manager.get_epoch_protocol_version(epoch_id)?;
    let shard_layout = epoch_manager.get_shard_layout(&epoch_id)?;

    // `ShardLayoutV3` stores all ancestor shards, no need to iterate through protocol versions
    if let Some(ancestors) = shard_layout.ancestor_uids(shard_id) {
        let ancestors = HashSet::from_iter(ancestors);
        return Ok(!ancestors.is_disjoint(&tracked_shards));
    }

    let mut shard_uid = ShardUId::from_shard_id_and_layout(shard_id, &shard_layout);
    if tracked_shards.contains(&shard_uid) {
        // We explicitly track `shard_id` (the shard is a descendant of itself).
        return Ok(true);
    }

    // `shard_uid` does not belong to `tracked_shards`, but it might be a descendant of one.
    // Iterate through consecutive pairs of historical shard layouts (newest to oldest) to trace
    // the ancestry. Each pair represents a resharding transition.
    let layout_history = epoch_manager.get_shard_layout_history(protocol_version, None);
    for window in layout_history.windows(2) {
        let current_layout = &window[0];
        let prev_layout = &window[1];
        let Some(parent_shard_id) = current_layout.try_get_parent_shard_id(shard_uid.shard_id())?
        else {
            debug_assert!(
                false,
                "Parent shard is missing for shard {} in shard layout {:?}",
                shard_uid, current_layout,
            );
            return Ok(false);
        };
        shard_uid = ShardUId::from_shard_id_and_layout(parent_shard_id, &prev_layout);
        if tracked_shards.contains(&shard_uid) {
            return Ok(true);
        }
    }
    Ok(false)
}
