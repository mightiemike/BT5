### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Guard Added in `verify_transaction_inclusion_v2` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a fully reachable public NEAR contract method. Any unprivileged caller can invoke it directly, bypassing the coinbase Merkle proof validation that `verify_transaction_inclusion_v2` was specifically introduced to enforce. This is a direct analog to M-09: a secondary execution path silently skips a security check that the primary path imposes.

---

### Finding Description

The contract exposes two public verification endpoints:

**v2 (guarded path)** — `verify_transaction_inclusion_v2` at line 347: [1](#0-0) 

It first validates a coinbase Merkle proof before delegating to v1: [2](#0-1) 

This coinbase check was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability (documented at https://www.bitmex.com/blog/64-Byte-Transactions), where an attacker crafts a 64-byte value that is simultaneously a valid internal Merkle tree node and a plausible "transaction hash," allowing them to forge an SPV inclusion proof for a non-existent transaction.

**v1 (unguarded path)** — `verify_transaction_inclusion` at line 288: [3](#0-2) 

The `#[deprecated]` Rust attribute at line 283 generates only a **compiler warning for Rust callers**. It does **not** prevent the function from being dispatched as a NEAR contract method. There is no `#[private]` attribute, no role guard, and no access control of any kind. Any NEAR account can call `verify_transaction_inclusion` directly, receiving a `true` result for a forged proof without ever touching the coinbase validation logic in v2.

The parallel to M-09 is exact:

| M-09 (Vault.sol) | This repository |
|---|---|
| `execute()` enforces caller permissions | `verify_transaction_inclusion_v2` enforces coinbase proof |
| `fallback()` skips those checks | `verify_transaction_inclusion` skips the coinbase proof |
| Any caller can use `fallback()` | Any NEAR caller can call v1 directly |

---

### Impact Explanation

Any downstream NEAR contract that gates a privileged action (token mint, cross-chain bridge release, custody unlock) on the result of `verify_transaction_inclusion` can be deceived by a forged 64-byte proof. The attacker supplies a crafted `tx_id` that is actually an internal Merkle node; the v1 function computes the Merkle root from it and matches the block's `merkle_root` field, returning `true` for a transaction that was never broadcast or mined. The corrupted proof result propagates to the consuming contract, which executes the privileged action on false premises. [4](#0-3) 

---

### Likelihood Explanation

- No privileged role, staked deposit, or special key is required. Any NEAR account can call the method.
- The 64-byte forgery attack is publicly documented and has known construction techniques.
- The v1 endpoint is permanently registered in the WASM dispatch table until a breaking upgrade removes it; `#[deprecated]` provides zero on-chain enforcement.
- Integrators reading the ABI or prior documentation may use v1 without realizing the security regression.

---

### Recommendation

Apply `#[private]` to `verify_transaction_inclusion` to make it callable only by the contract itself (i.e., only through the internal `self.verify_transaction_inclusion(...)` call inside v2):

```rust
#[private]
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    ...
}
```

This mirrors the fix recommended in M-09: track which execution paths require guards and enforce them on every reachable entry point, not just the primary one.

---

### Proof of Concept

1. Deploy the contract with `skip_pow_verification: true` and a known set of headers.
2. Construct a `ProofArgs` where `tx_id` is a crafted 64-byte value whose double-SHA256 hash equals an internal Merkle tree node of a real block's Merkle tree, and `merkle_proof` is the sibling path that reconstructs the block's `merkle_root`.
3. Call `verify_transaction_inclusion` directly (not v2) from any unprivileged NEAR account.
4. The function returns `true` for a transaction that does not exist.
5. Calling `verify_transaction_inclusion_v2` with the same `tx_id` and a mismatched `coinbase_merkle_proof` panics at line 362, confirming the guard is present only in v2. [5](#0-4)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
