### Title
`PerpEngineState.balances` Public Mapping Exposes Unsettled Funding Data, Diverging from `getBalance()` — (File: `core/contracts/PerpEngineState.sol`)

---

### Summary

`PerpEngineState` declares its `balances` mapping as `public`, generating a Solidity auto-getter that returns raw, unsettled `Balance` structs directly from storage. The canonical `getBalance()` function applies pending funding-rate settlement before returning the balance. These two access paths diverge as funding accumulates, creating an inconsistency analogous to the Hats `balanceOf` / `balanceOfBatch` desynchronization.

---

### Finding Description

`PerpEngineState.getBalance()` applies `_updateBalance(state, balance, 0, 0)` before returning, which settles accrued funding payments into `balance.vQuoteBalance`:

```solidity
// PerpEngineState.sol line 65-74
function getBalance(uint32 productId, bytes32 subaccount)
    public view returns (Balance memory)
{
    State memory state = states[productId];
    Balance memory balance = balances[productId][subaccount];
    _updateBalance(state, balance, 0, 0);   // <-- settles funding
    return balance;
}
```

The settlement logic inside `_updateBalance` computes:

```solidity
// PerpEngineState.sol line 34-36
int128 diffX18 = cumulativeFundingAmountX18 - balance.lastCumulativeFundingX18;
int128 deltaQuote = vQuoteDelta - diffX18.mul(balance.amount);
balance.vQuoteBalance += deltaQuote;
```

However, `balances` is declared `public`:

```solidity
// PerpEngineState.sol line 17
mapping(uint32 => mapping(bytes32 => Balance)) public balances;
```

This auto-generates a getter `balances(productId, subaccount)` that returns the raw `Balance` struct directly from storage — **without** applying the funding settlement. Any caller of `balances(productId, subaccount)` receives a stale `vQuoteBalance` that does not reflect accrued funding since the last on-chain write.

By contrast, `SpotEngineState.balances` is correctly declared `internal`:

```solidity
// SpotEngineState.sol line 12
mapping(uint32 => mapping(bytes32 => BalanceNormalized)) internal balances;
```

The asymmetry is the root cause.

---

### Impact Explanation

Any external integrator — another protocol, a liquidation bot, an off-chain risk engine, or a composing contract — that reads `PerpEngine.balances(productId, subaccount)` directly receives a `vQuoteBalance` that is missing all funding payments accrued since the last `updateBalance` call. The magnitude of the discrepancy is `diffX18 × balance.amount`, which grows with position size and time elapsed since the last on-chain settlement. This leads to:

- Incorrect position PnL assessments by external systems.
- Incorrect health/margin calculations by any integrating contract that reads raw `balances` instead of calling `getBalance()`.
- Potential under-detection of insolvent positions by off-chain liquidation infrastructure that relies on the public getter.

The internal Nado protocol is unaffected because `_getBalance()` → `getBalance()` is always used for health calculations. The exposure is entirely to the external-facing surface.

---

### Likelihood Explanation

Medium. The `balances` mapping is `public` and therefore part of the contract's ABI. Any developer integrating with `PerpEngine` would naturally call the auto-generated getter as the most direct way to read a position. The discrepancy is non-obvious because `getBalance()` and `balances()` share the same name prefix and return the same type, but produce different values whenever funding has accrued.

---

### Recommendation

Change the `balances` mapping in `PerpEngineState` from `public` to `internal`, matching the pattern used in `SpotEngineState`:

```solidity
// Before
mapping(uint32 => mapping(bytes32 => Balance)) public balances;

// After
mapping(uint32 => mapping(bytes32 => Balance)) internal balances;
```

All external consumers should be directed to use `getBalance(productId, subaccount)` or `getStateAndBalance(productId, subaccount)`, which apply funding settlement before returning.

---

### Proof of Concept

1. Funding accumulates: `updateStates` is called, advancing `state.cumulativeFundingLongX18` by some `paymentAmount > 0` for a long position.
2. A long position holder has `balance.lastCumulativeFundingX18` set to the old cumulative value.
3. Call `PerpEngine.balances(productId, subaccount)` → returns raw `Balance` with stale `vQuoteBalance` (funding not applied).
4. Call `PerpEngine.getBalance(productId, subaccount)` → returns `Balance` with `vQuoteBalance` reduced by `diffX18 × balance.amount` (funding applied).
5. The two `vQuoteBalance` values differ by exactly `diffX18 × balance.amount`, which can be arbitrarily large for a large position over a long funding interval.
6. An external liquidation bot reading step 3 sees a healthier position than actually exists, potentially missing a liquidation opportunity. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
