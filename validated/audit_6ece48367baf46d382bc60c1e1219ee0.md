### Title
Unsigned `feeX18` in `WithdrawCollateralV2` Allows Sequencer to Charge Maximum Fee Without User Consent - (File: `core/contracts/interfaces/IEndpoint.sol`, `core/contracts/EndpointTx.sol`, `core/contracts/Verifier.sol`)

---

### Summary

The `feeX18` field in `SignedWithdrawCollateralV2` is excluded from the EIP-712 signed digest. A user signs only `{sender, productId, amount, nonce, sendTo, appendix}`. The sequencer can freely attach any `feeX18` value up to `currentFeeX18` when submitting the transaction on-chain, without the user having committed to that fee amount. This is the direct Nado analog of the Argent "no fee limit for v3 transactions" class: a privileged party (the sequencer) can set a fee parameter that the user never signed, draining collateral beyond what the user consented to.

---

### Finding Description

`SignedWithdrawCollateralV2` is defined as:

```solidity
struct SignedWithdrawCollateralV2 {
    WithdrawCollateralV2 tx;
    CompactSignature signature;
    int128 feeX18;          // ← outside the signed inner struct
}
```

The EIP-712 type string used to compute the digest is:

```
"WithdrawCollateralV2(bytes32 sender,uint32 productId,uint128 amount,uint64 nonce,address sendTo,uint128 appendix)"
```

`feeX18` is absent from this type string. In `computeDigest`, only the fields of the inner `WithdrawCollateralV2 tx` struct are hashed:

```solidity
digest = keccak256(abi.encode(
    keccak256(bytes(WITHDRAW_COLLATERAL_V2_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.productId,
    signedTx.tx.amount,
    signedTx.tx.nonce,
    signedTx.tx.sendTo,
    signedTx.tx.appendix
));
```

When the sequencer submits the transaction via `processTransactionImpl`, it enforces only:

```solidity
int128 currentFeeX18 = spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18;
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
chargeFee(signedTx.tx.sender, signedTx.feeX18, signedTx.tx.productId);
```

The sequencer is free to set `feeX18` to any value in `[0, currentFeeX18]` for every `WithdrawCollateralV2` it processes. The user's signature covers none of this.

---

### Impact Explanation

A malicious or compromised sequencer can set `feeX18 = currentFeeX18` on every `WithdrawCollateralV2` transaction it processes. The fee is deducted from the user's subaccount balance via `spotEngine.updateBalance(productId, sender, -fee)` and credited to `sequencerFee[productId]`. The user signed a withdrawal for a specific `amount` but never committed to any fee value. The sequencer can silently extract the maximum protocol fee on every withdrawal without the user's knowledge or consent. For the quote product, `withdrawFeeX18` is initialized to `ONE` ($1 per withdrawal). For other products, the owner can set it higher. Across many users and withdrawals, this represents a systematic, unconsented extraction of collateral.

---

### Likelihood Explanation

The sequencer is a trusted party in the current Nado deployment, analogous to the "nice sequencer" in the Argent report. However, the protocol design should not rely on sequencer honesty for fee correctness. The attack path requires only that the sequencer submit a valid user-signed `WithdrawCollateralV2` transaction with `feeX18` set to `currentFeeX18` — no additional compromise is needed beyond the sequencer acting in its own interest. The V2 withdrawal path was explicitly designed to allow variable fees (unlike V1 which always charges the full `withdrawFeeX18`), but the mechanism for the user to consent to a specific fee is absent.

---

### Recommendation

Include `feeX18` in the EIP-712 type string and digest computation so that users explicitly sign the fee they accept:

```
"WithdrawCollateralV2(bytes32 sender,uint32 productId,uint128 amount,uint64 nonce,address sendTo,uint128 appendix,int128 feeX18)"
```

This mirrors the Argent fix: just as Argent introduced explicit on-chain limits on the fee fields that the Guardian could set, Nado should require the user to commit to the fee at signing time, preventing the sequencer from unilaterally choosing the fee amount.

---

### Proof of Concept

1. User signs a `WithdrawCollateralV2` for `productId=0` (USDC), `amount=1000e6`, `sendTo=userAddress`. The signed digest covers only `{sender, productId, amount, nonce, sendTo, appendix}`.

2. Sequencer receives the signed transaction off-chain. It constructs `SignedWithdrawCollateralV2` with `feeX18 = currentFeeX18` (e.g., `1e18` = $1 for the quote product).

3. Sequencer calls `submitTransactions([encodedWithdrawCollateralV2Tx])`.

4. `processTransactionImpl` decodes the transaction, verifies the signature (which is valid — `feeX18` was never part of the digest), checks `0 <= feeX18 <= currentFeeX18` (passes), and calls `chargeFee(sender, 1e18, 0)`.

5. `spotEngine.updateBalance(0, sender, -1e18)` deducts $1 from the user's quote balance. The user receives `amount - fee` in the withdrawal, having never agreed to pay $1.

6. The sequencer repeats this for every `WithdrawCollateralV2` it processes, extracting the maximum fee each time.
