### Title
Minimum Liquidation Penalty Floor Creates Counterproductive Health Degradation for Low-Risk Assets — (`File: core/contracts/ClearinghouseStorage.sol`)

---

### Summary

In `ClearinghouseLiq`, the liquidation price is computed by deriving a penalty from the maintenance weight of the product. A minimum penalty floor (`MIN_NON_SPREAD_LIQ_PENALTY_X18`) is enforced to guarantee liquidator profitability. For assets whose maintenance weight is close to 1 (low-risk assets), the floored penalty exceeds the natural penalty implied by the weight, causing each liquidation to worsen the liquidatee's maintenance health rather than improve it. This is a structural analog to the Aave counterproductive-incentive vulnerability: liquidators are still incentivized to act, but each action deepens insolvency.

---

### Finding Description

`getLiqPriceX18` in `ClearinghouseStorage.sol` computes the liquidation price as:

```
penaltyX18 = (maintenanceWeight - ONE) / 5
if |penaltyX18| < MIN_NON_SPREAD_LIQ_PENALTY_X18:
    penaltyX18 = ±MIN_NON_SPREAD_LIQ_PENALTY_X18
liquidationPrice = oraclePrice * (ONE + penaltyX18)
``` [1](#0-0) 

For a **long perp/spot position** with `longWeightMaintenance` close to 1 (e.g., 0.99):

- Natural penalty: `(0.99 − 1) / 5 = −0.002`
- Floored to: `−MIN_NON_SPREAD_LIQ_PENALTY_X18` (e.g., `−0.02`)
- Liquidation price: `oraclePrice × 0.98`

The liquidatee receives `0.98 × oraclePrice` per unit of position, but the maintenance health contribution of that position was `0.99 × oraclePrice` per unit. **Net health change per unit: `0.98 − 0.99 = −0.01`** — health worsens.

For a **spot/perp liability** with `shortWeightMaintenance` close to 1 (e.g., 1.01):

- Natural penalty: `(1.01 − 1) / 5 = 0.002`
- Floored to: `+MIN_NON_SPREAD_LIQ_PENALTY_X18` (e.g., `+0.02`)
- Liquidation price: `oraclePrice × 1.02`

The liquidatee pays `1.02 × oraclePrice` per unit to buy back the liability, but the health benefit from reducing the liability is only `(1.01 − 1) × oraclePrice = 0.01 × oraclePrice` per unit. **Net health change per unit: `0.01 − 0.02 = −0.01`** — health worsens.

The general condition for counterproductive liquidation is:

```
MIN_NON_SPREAD_LIQ_PENALTY_X18 > |maintenanceWeight − 1|
```

This is satisfied for any asset whose maintenance weight is within `5 × MIN_NON_SPREAD_LIQ_PENALTY_X18` of 1.

The post-liquidation check does **not** guard against this:

```solidity
require(!isAboveInitial(txn.liquidatee), ERR_LIQUIDATED_TOO_MUCH);
``` [2](#0-1) 

This only prevents over-liquidation (health going above initial), not health degradation. The entry check only requires maintenance health to be negative:

```solidity
require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
``` [3](#0-2) 

After a counterproductive liquidation, the liquidatee's maintenance health is more negative, so they remain eligible for further liquidations. The liquidator continues to profit from each step, while the protocol is pushed deeper into insolvency.

---

### Impact Explanation

For subaccounts holding low-risk assets (maintenance weight near 1), each liquidation event:

1. Worsens the liquidatee's maintenance health (increases insolvency depth)
2. Keeps the liquidatee eligible for further liquidation (maintenance health still < 0)
3. Drains the liquidatee's quote balance faster than the liability/position is resolved
4. Ultimately forces the protocol to cover a larger shortfall via the insurance fund or socialization

The corrupted state delta is: `maintenanceHealth(liquidatee)` decreases monotonically with each liquidation step, rather than converging toward zero. The insurance fund (`insurance` in `ClearinghouseStorage`) is exposed to a larger-than-necessary deficit. [4](#0-3) 

---

### Likelihood Explanation

- Any unprivileged liquidator can trigger this via `liquidateSubaccountImpl` → `_handleLiquidationPayment`.
- Low-risk assets (stablecoins, low-volatility perps) with maintenance weights near 1 are common in DEX protocols.
- The liquidator is always incentivized (they receive the penalty as profit), so there is no economic barrier to repeated counterproductive liquidations.
- No admin action or special privilege is required. [5](#0-4) 

---

### Recommendation

1. **Add a post-liquidation maintenance health check**: After `_handleLiquidationPayment`, verify that the liquidatee's maintenance health has not worsened (i.e., is greater than or equal to its pre-liquidation value), or at minimum that it has moved toward zero.

2. **Derive the minimum penalty from the weight gap**: Instead of a flat floor, set `MIN_NON_SPREAD_LIQ_PENALTY_X18` as a function of `|weight − 1|` so that the floored penalty never exceeds the health benefit of the liquidation.

3. **Cap the liquidation amount for deeply insolvent subaccounts**: For subaccounts where the maintenance weight gap is small, allow 100% liquidation in a single step (analogous to the Aave recommendation of allowing full liquidation in exchange for all remaining collateral) to avoid the spiral.

### Proof of Concept

**Setup**: Subaccount `A` holds a long perp position of size 1000 units in a product with:
- `oraclePrice = 1.0`
- `longWeightMaintenance = 0.99`
- `MIN_NON_SPREAD_LIQ_PENALTY_X18 = 0.02`
- Subaccount has no quote balance, maintenance health = `1000 × 1.0 × 0.99 − debt = −5` (under maintenance)

**Step 1**: Liquidator calls `liquidateSubaccountImpl` with `txn.amount = 100` (10% of position).

**In `getLiqPriceX18`**:
- `penaltyX18 = (0.99 − 1) / 5 = −0.002`
- `|−0.002| < 0.02` → floored to `−0.02`
- `liquidationPrice = 1.0 × 0.98 = 0.98`

**Balance changes for liquidatee**:
- Perp position: `−100` units
- vQuote: `+0.98 × 100 = +98`

**Health change**:
- Perp position contribution: `−100 × 1.0 × 0.99 = −99`
- vQuote contribution: `+98`
- Net: `−1` (health worsened by 1)

**Post-liquidation maintenance health**: `−5 − 1 = −6` (more negative than before)

**Check `!isAboveInitial`**: passes (health is still negative)

**Step 2**: Liquidator repeats. Each iteration worsens health by 1 per 100 units liquidated. The liquidatee spirals deeper into insolvency while the liquidator profits `(1.0 − 0.98) × LIQUIDATION_FEE_FRACTION × 100` per step. [6](#0-5) [7](#0-6)

    function getLiqPriceX18(uint32 productId, int128 amount)
        internal
        returns (int128, int128)
    {
        RiskHelper.Risk memory risk = IProductEngine(productToEngine[productId])
            .getRisk(productId);
        int128 penaltyX18 = (RiskHelper._getWeightX18(
            risk,
            amount,
            IProductEngine.HealthType.MAINTENANCE
        ) - ONE) / 5;
        if (penaltyX18.abs() < MIN_NON_SPREAD_LIQ_PENALTY_X18) {
            if (penaltyX18 < 0) {
                penaltyX18 = -MIN_NON_SPREAD_LIQ_PENALTY_X18;
            } else {
                penaltyX18 = MIN_NON_SPREAD_LIQ_PENALTY_X18;
            }
        }
        return (risk.priceX18.mul(ONE + penaltyX18), risk.priceX18);
    }
