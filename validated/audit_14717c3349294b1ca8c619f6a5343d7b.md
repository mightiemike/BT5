### Title
Difficulty Validation for Fork Blocks Uses Mainchain Reference Block Instead of Fork Ancestor — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary
When validating the difficulty (`bits`) of a fork block at a retarget boundary, `get_next_work_required` calls `get_header_by_height(first_block_height)` to obtain the reference block for the timespan calculation. `get_header_by_height` exclusively reads from `mainchain_height_to_header`, so it always returns the **mainchain** block at that height — not the fork's actual ancestor. Once a fork has grown past the reference height, the difficulty check uses the wrong block's timestamp, producing an incorrect expected target. This is the direct analog of the borrowing-fee inconsistency: a cumulative/derived value (expected difficulty) is computed against a stale or wrong checkpoint (mainchain block instead of fork ancestor), making the result path-dependent on which chain the reference block belongs to.

### Finding Description

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

Fork blocks are stored only in `headers_pool`, never in `mainchain_height_to_header`. [2](#0-1) 

For **Dogecoin** (per-block Digishield retarget after height 145,000), every fork block's difficulty is computed using:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [3](#0-2) 

where `height_first = prev_block_header.block_height - blocks_to_go_back` (typically `prev_height - 1`). [4](#0-3) 

The same pattern exists for Bitcoin and Litecoin at retarget boundaries (every 2016 blocks):

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header,
    interval_tail_extend_header.block_header.time.into())
``` [5](#0-4) [6](#0-5) 

**Concrete trigger (Dogecoin):**

Suppose the mainchain has blocks at heights H-1, H, H+1, H+2, … and a fork diverges at height H (fork block at H has a different hash and timestamp than the mainchain block at H):

| Fork block height | `prev_block_header` | `get_header_by_height(height-1)` returns | Correct? |
|---|---|---|---|
| H | mainchain[H-1] | mainchain[H-2] | ✓ (fork diverged at H) |
| H+1 | fork[H] | mainchain[H-1] | ✓ (fork ancestor at H-1 = mainchain[H-1]) |
| H+2 | fork[H+1] | **mainchain[H]** ≠ fork[H] | **✗ WRONG** |

Starting from the third fork block, `first_block_time` is taken from the mainchain block at H, not the fork block at H. The `calculate_next_work_required` function computes:

```rust
let modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
``` [7](#0-6) 

using `actual_timespan = fork[H+1].time - mainchain[H].time` instead of the correct `fork[H+1].time - fork[H].time`. If the attacker sets `fork[H].time` to differ from `mainchain[H].time`, the expected `bits` diverges from what the fork chain actually requires.

The developers themselves flagged this with a TODO comment acknowledging the concern. [8](#0-7) 

### Impact Explanation

**False acceptance of under-difficulty fork blocks:** If the attacker sets `fork[H].time` later than `mainchain[H].time`, the computed `actual_timespan` shrinks, `modulated_timespan` is clamped lower, and the resulting target is **easier** (higher). The contract then accepts fork blocks whose actual PoW is weaker than the chain's true difficulty requires. An attacker can mine a fork chain at lower cost and trigger a reorg, corrupting the canonical chain pointer (`mainchain_tip_blockhash`) and invalidating SPV proofs that downstream contracts rely on.

**False rejection of valid fork blocks:** The inverse case (fork timestamp earlier than mainchain) causes the contract to reject legitimately valid fork blocks, preventing honest relayers from submitting correct chain data.

### Likelihood Explanation

For **Dogecoin** (per-block retarget), the bug is triggered by the **third fork block** in any fork — a very low bar. Any unprivileged relayer can submit fork blocks via the public `submit_blocks` entrypoint. [9](#0-8) 

For **Bitcoin/Litecoin**, the bug requires a fork spanning a full 2016-block retarget period, which is a higher bar but still reachable by a well-resourced attacker.

### Recommendation

Replace `get_header_by_height` with a chain-traversal that walks back through `get_prev_header` from `prev_block_header` to find the actual ancestor at `height_first`. This ensures the reference block is always the fork's true ancestor, regardless of what the mainchain index contains at that height — mirroring how the reference implementation walks the chain by hash linkage rather than by height index.

### Proof of Concept

1. Deploy the Dogecoin variant of the contract.
2. Submit a mainchain sequence: blocks at heights H-2, H-1, H, H+1, H+2 (all with known timestamps).
3. Submit a fork block at height H with `fork[H].time = mainchain[H].time + Δ` (Δ > 0, within MTP constraints).
4. Submit fork block at H+1 (any valid timestamp).
5. Submit fork block at H+2 with `bits` set to the value computed from `mainchain[H].time` (easier target).
6. Observe that `check_target` passes because `get_header_by_height(H)` returns `mainchain[H]` with the original timestamp, making the expected bits match the attacker's chosen easier value.
7. The fork block at H+2 is accepted with a lower-than-correct difficulty, allowing the attacker to accumulate chainwork cheaply and trigger a reorg via `submit_block_header_inner`. [10](#0-9)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

    fn submit_block_header_inner(
        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }
