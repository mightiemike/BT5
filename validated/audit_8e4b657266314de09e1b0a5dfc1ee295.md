Audit Report

## Title
Unbounded `timelock` in `OracleValueStopLossExtension` allows pool admin to bypass LP-protection timelock via `uint32` truncation — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
Neither `initialize` nor `proposeOracleStopLossTimelock` validates an upper bound on the `timelock` value. In Solidity 0.8+, explicit narrowing casts truncate silently; `uint32(block.timestamp + type(uint32).max)` wraps to a past timestamp, causing `_requireElapsed` to pass immediately. A pool admin can exploit this to bypass the timelock on any subsequent config change — collapsing the drawdown floor and draining LP value with no reaction window.

## Finding Description
`initialize` validates `drawdownE6` and `decayPerSecondE8` but never validates `timelock`:

```solidity
// L56-62
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);   // checked
_validateDecay(decayPerSecondE8); // checked
// timelock — never checked
oracleStopLossConfig[pool] = PoolStopLossConfig({
    drawdownE6: drawdownE6, decayPerSecondE8: decayPerSecondE8, timelock: timelock, initialized: true
});
```

`proposeOracleStopLossTimelock` has the same gap — `newTimelock` is stored without any bound check (L78-84).

The overflow occurs in `_afterTimelock`:

```solidity
// L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; the addition is performed in `uint256` space. The explicit `uint32(...)` cast is a narrowing truncation — **not** a checked operation in Solidity 0.8+. When `timelock = type(uint32).max` (~136 years), `block.timestamp + type(uint32).max ≈ 6.04e9`, which truncates to `≈ 1.75e9 − 1` — a timestamp already in the past.

`_requireElapsed` then trivially passes:

```solidity
// L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;  // block.timestamp > wrapped past value → passes
}
```

**Exploit path:**
1. Pool is live with `timelock = 1 day` (legitimate setup).
2. Pool admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)` — must wait 1 day (normal timelock applies here).
3. After 1 day, calls `executeOracleStopLossTimelock(pool)` — `oracleStopLossConfig[pool].timelock = type(uint32).max`.
4. Calls `proposeOracleStopLossDrawdown(pool, 999_999)` — `_afterTimelock` returns a wrapped past timestamp.
5. Immediately calls `executeOracleStopLossDrawdown(pool)` — `_requireElapsed` passes, `drawdownE6 = 999_999`.
6. Stop-loss floor = `metric * (1e6 − 999_999) / 1e6 ≈ 0`. All subsequent swaps pass `_checkAndUpdateWatermarks` without reverting. LP value is drained.

## Impact Explanation
This is a confirmed admin-boundary break: the pool admin exceeds the timelock cap that LPs rely on for reaction time. After bypassing the timelock, the admin can set `drawdownE6 = 999_999`, collapsing the per-share value floor to near zero. Swaps then drain LP principal with no stop-loss protection. This is direct loss of user principal — Critical/High severity under Sherlock thresholds.

## Likelihood Explanation
Requires a malicious or compromised pool admin. However, the timelock mechanism exists precisely because the pool admin is semi-trusted; bypassing it is the exact admin-boundary break the scope flags as in-scope. An honest admin could also trigger the brick path (timelock ~68 years, no overflow but all future `executeAfter` values are unreachable) by accidentally supplying a large sentinel value.

## Recommendation
Add a `_validateTimelock` helper with a reasonable cap (e.g., 30 days) and call it in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MAX_TIMELOCK = 30 days;

function _validateTimelock(uint32 timelock_) private pure {
    if (timelock_ > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock_);
}
```

Call it alongside `_validateDrawdown` and `_validateDecay` in `initialize` (after L58), and at the top of `proposeOracleStopLossTimelock` before writing `sched.pendingTimelock`.

## Proof of Concept

```solidity
// Foundry test sketch
function test_timelockOverflowBypass() public {
    // Pool live with timelock = 1 day
    vm.warp(1_750_000_000);

    // Step 1: propose overflow timelock (must wait current 1-day timelock)
    vm.prank(poolAdmin);
    ext.proposeOracleStopLossTimelock(pool, type(uint32).max);

    // Step 2: wait 1 day, execute
    vm.warp(block.timestamp + 1 days);
    vm.prank(poolAdmin);
    ext.executeOracleStopLossTimelock(pool);
    // oracleStopLossConfig[pool].timelock == type(uint32).max

    // Step 3: propose near-total drawdown
    vm.prank(poolAdmin);
    ext.proposeOracleStopLossDrawdown(pool, 999_999);
    // _afterTimelock: uint32(1_750_086_400 + 4_294_967_295) wraps to past timestamp

    // Step 4: execute immediately — no wait
    vm.prank(poolAdmin);
    ext.executeOracleStopLossDrawdown(pool);
    // drawdownE6 == 999_999, stop-loss floor ≈ 0

    assertEq(ext.oracleStopLossConfig(pool).drawdownE6, 999_999);
}
