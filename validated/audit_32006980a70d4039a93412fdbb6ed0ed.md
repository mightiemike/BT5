### Title
`WithdrawCollateralV2` Missing from `processSlowModeTransactionImpl` Causes Silent Fee Loss and Permanent Slow-Mode Withdrawal Failure — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

`processTransactionImpl` correctly handles both `WithdrawCollateral` and `WithdrawCollateralV2`, but `processSlowModeTransactionImpl` only handles `WithdrawCollateral`. When a user submits a `WithdrawCollateralV2` transaction via the slow-mode path, `submitSlowModeTransactionImpl` accepts it (charges the slow-mode fee and enqueues it), but `processSlowModeTransactionImpl` hits the `else { revert(); }` fallback. The outer `try/catch` in `_executeSlowModeTransaction` silently swallows the revert. The user loses the slow-mode fee and their withdrawal never executes.

---

### Finding Description

`processTransactionImpl` in `EndpointTx.sol` handles `WithdrawCollateralV2` at line 437:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateralV2) {
    IEndpoint.SignedWithdrawCollateralV2 memory signedTx = abi.decode(...);
    ...
    clearinghouse.withdrawCollateral(
        signedTx.tx.sender, signedTx.tx.productId, signedTx.tx.amount,
        signedTx.tx.sendTo, nSubmissions
    );
```

`processSlowModeTransactionImpl` handles `WithdrawCollateral` at line 217 but has **no branch for `WithdrawCollateralV2`**:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    ...
    clearinghouse.withdrawCollateral(txn.sender, txn.productId, txn.amount, address(0), nSubmissions);
} else {
    revert();   // <-- WithdrawCollateralV2 always lands here
}
```

Meanwhile, `submitSlowModeTransactionImpl` has no explicit rejection for `WithdrawCollateralV2` either — it falls into the generic `else` branch that charges the slow-mode fee and enqueues the transaction:

```solidity
} else {
    chargeSlowModeFee(_getQuote(), sender);
    slowModeFees += SLOW_MODE_FEE;
}
```

When the enqueued transaction is later executed via `_executeSlowModeTransaction`, the `try/catch` silently absorbs the revert from `processSlowModeTransactionImpl`:

```solidity
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    // silent failure unless out-of-gas
}
```

The user's slow-mode fee is consumed, the withdrawal never settles, and the collateral remains locked.

---

### Impact Explanation

A user who submits `WithdrawCollateralV2` via slow mode:

1. **Pays the slow-mode fee** (deducted from their quote balance in `chargeSlowModeFee`).
2. **Has their transaction enqueued** with a 3-day delay.
3. **Receives no funds** — the execution silently reverts.
4. **Cannot withdraw to a custom `sendTo` address** via slow mode at all, even after the sequencer recovers, because the transaction is consumed from the queue.

The slow-mode path is the censorship-resistance escape hatch. Blocking `WithdrawCollateralV2` there means users who need to withdraw to a non-owner address (the key feature of V2) are permanently unable to do so without sequencer cooperation, defeating the purpose of slow mode for that transaction type.

---

### Likelihood Explanation

The trigger is fully unprivileged: any user holding collateral can call `submitSlowModeTransaction` with a `WithdrawCollateralV2` payload. The scenario is realistic whenever:

- The sequencer is unresponsive or censoring.
- The user wants to withdraw to a `sendTo` address different from the subaccount owner (the only reason to use V2 over V1).

No special role, leaked key, or governance action is required.

---

### Recommendation

Add a `WithdrawCollateralV2` branch to `processSlowModeTransactionImpl` in `EndpointTx.sol`, mirroring the existing `WithdrawCollateral` branch but decoding `SignedWithdrawCollateralV2` and forwarding `signedTx.tx.sendTo`:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateralV2) {
    IEndpoint.WithdrawCollateralV2 memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateralV2)
    );
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(
        txn.sender,
        txn.productId,
        txn.amount,
        txn.sendTo,
        nSubmissions
    );
}
```

Additionally, consider whether `submitSlowModeTransactionImpl` should explicitly reject unsupported transaction types rather than silently accepting them and charging a fee.
