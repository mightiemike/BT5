### Title
`DepositAllowlistExtension.beforeAddLiquidity` gates the LP position owner instead of the actual depositor (payer), allowing any non-allowlisted address to deposit tokens into a curated pool — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is designed to gate `addLiquidity` by depositor address. Its `beforeAddLiquidity` hook silently ignores the `sender` argument (the actual caller/payer) and checks only `owner` (the LP position recipient). Because `MetricOmmPool.addLiquidity` is a public function that accepts any caller-supplied `owner`, a non-allowlisted address can call `pool.addLiquidity(owner = allowlistedAddress, …)` directly, pass the guard, and deposit tokens into the pool without ever appearing on the allowlist.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` drops the first parameter (`sender`) as an unnamed blank and gates only on `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`MetricOmmPool.addLiquidity` is a fully public function. It passes `msg.sender` (the payer) as `sender` and the caller-supplied `owner` argument as `owner` to the extension dispatcher:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both:

```solidity
// ExtensionCalling.sol L95-98
_callExtensionsInOrder(
    BEFORE_ADD_LIQUIDITY_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

The extension receives `sender` (the non-allowlisted payer) as its first argument but never reads it. It reads only `owner`. Because `addLiquidity` imposes no restriction on what address a caller may supply as `owner`, any non-allowlisted address B can pass `owner = A` where A is allowlisted, satisfy the guard, and have the pool pull tokens from B via the liquidity callback while minting the LP position to A.

This is the direct analog of the `forceAllow()` bug: just as that function conflated "remove from denylist" with "add to allowlist" by assuming the two operations always apply to the same address, `DepositAllowlistExtension` conflates "who pays tokens" with "who holds the LP position," checking the wrong actor and letting the unchecked actor bypass the guard entirely.

---

### Impact Explanation

The `DepositAllowlistExtension` is the pool admin's mechanism to enforce a curated depositor set (e.g., KYC/compliance-gated pools). By checking `owner` instead of `sender`, the guard is rendered ineffective against any non-allowlisted address that supplies an allowlisted `owner`. The non-allowlisted address successfully deposits tokens into the pool — the core action the guard was meant to block — without appearing on the allowlist. This is an **admin-boundary break**: the pool admin's deposit restriction is bypassed by an unprivileged, permissionless path. Any pool relying on `DepositAllowlistExtension` for compliance or access control cannot enforce that restriction on the actual payer.

---

### Likelihood Explanation

The attack requires no special privileges. Any address can call `pool.addLiquidity` directly (the function is fully public, no `onlyFactory` or similar guard). The only prerequisite is knowing one allowlisted address to supply as `owner`. Allowlisted addresses are discoverable on-chain via the `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. The attacker must supply tokens (they pay into the pool), but they lose nothing if they collude with the allowlisted `owner` who can later call `removeLiquidity` (which enforces `msg.sender == owner`) and return the proceeds.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual payer/depositor) instead of `owner` (the LP position recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the payer and the LP position recipient, both `sender` and `owner` should be checked.

---

### Proof of Concept

1. Deploy a pool with `DepositAllowlistExtension` as an active extension.
2. Pool admin calls `extension.setAllowedToDeposit(pool, A, true)` — address A is allowlisted; address B is not.
3. Address B calls `pool.addLiquidity(owner = A, salt, deltas, callbackData, extensionData)` directly.
4. Pool calls `_beforeAddLiquidity(msg.sender = B, owner = A, …)`.
5. Extension evaluates `allowedDepositor[pool][A]` → `true` → guard passes.
6. Pool mints LP shares to position `(A, salt)`.
7. Pool invokes the liquidity callback on B; B pays `amount0` and `amount1` tokens.
8. B (non-allowlisted) has successfully deposited tokens into the curated pool. A holds the LP position and can later call `removeLiquidity` to return proceeds to B. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
