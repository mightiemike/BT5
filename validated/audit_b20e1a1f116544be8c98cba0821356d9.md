### Title
Internal Merkle Node Accepted as Valid Transaction in `verify_transaction_inclusion_v2` ‚ÄĒ (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` is supposed to mitigate the 64-byte Merkle proof forgery attack by requiring a coinbase proof. However, the coinbase check only verifies that `coinbase_tx_id` hashes to the correct Merkle root at position 0 ‚ÄĒ it does **not** verify that `coinbase_tx_id` is the actual coinbase transaction. An attacker can supply an internal Merkle tree node as `coinbase_tx_id` (with a shorter proof path that still reaches the root), then use the same internal node as `tx_id`, causing the function to return `true` for a transaction that does not exist in the block.

3. **Non-empty guard** (`line 315`): `[n23]` is not empty ‚Üí **PASSES**

4. **TX inclusion check** (`lines 318‚Äď322`):
   `compute_root_from_merkle_proof(n01, 0, [n23])`
   ‚Üí `hash(n01, n23)` = `merkle_root` ‚Üí **PASSES**

**Result:** `verify_transaction_inclusion_v2` returns `true` for `tx_id = n01`, which is an internal Merkle tree node ‚ÄĒ not a real Bitcoin transaction.

The root cause is that the coinbase proof check only verifies a hash-path relationship, not that `coinbase_tx_id` is the actual transaction at index 0. An internal node at a higher tree level can satisfy the same hash-path equation with a shorter proof, bypassing the depth-anchoring intent of the coinbase check entirely.

---

### Impact Explanation

Any consumer of `verify_transaction_inclusion_v2` (e.g., a BTC bridge, payment verifier, or cross-chain relay) that trusts its `true` return value to confirm a Bitcoin payment can be deceived. An attacker can fabricate proof of a non-existent payment by using a publicly computable internal Merkle node from any block with ‚Č•4 transactions. This directly breaks the core invariant of the light client: **inclusion verification must only return `true` for actual transactions**.

---

### Likelihood Explanation

- No privileged access is required. `verify_transaction_inclusion_v2` is a public `#[pause]`-gated view/call.
- All inputs needed (internal node hashes) are computable from the public block transaction list.
- Any Bitcoin block with ‚Č•4 transactions (the vast majority) is exploitable.
- No cryptographic hardness assumption is broken; the attack is purely structural.

---

### Recommendation

The coinbase proof check must be strengthened so that `coinbase_tx_id` is provably the **actual** coinbase transaction at leaf index 0, not merely any value that hashes to the root at position 0. Concretely:

1. **Enforce proof depth equality with the expected tree depth.** The length of `coinbase_merkle_proof` must equal `ceil(log2(tx_count))`. This prevents a shorter path through an internal node from satisfying the check.
2. **Alternatively, require `coinbase_tx_id != tx_id` when `tx_index != 0`**, and verify that the coinbase proof path is disjoint from the tx proof path at the leaf level ‚ÄĒ i.e., the two proofs must diverge at the first step.
3. The most robust fix: pass the total transaction count into the verifier and assert `coinbase_merkle_proof.len() == expected_depth` where `expected_depth = ceil(log2(tx_count))`.

---

### Proof of Concept

For a block with 4 transactions `[tx0, tx1, tx2, tx3]` already submitted to the contract:

```rust
// Compute internal nodes
let n01 = double_sha256([tx0 || tx1]);  // internal node
let n23 = double_sha256([tx2 || tx3]);  // internal node
// merkle_root = double_sha256([n01 || n23])  -- stored in block header

// Craft the malicious call
verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id:                 n01,    // internal node, NOT a real tx
    tx_block_blockhash:    <valid block hash>,
    tx_index:              0,
    merkle_proof:          vec![n23],
    coinbase_tx_id:        n01,    // same internal node
    coinbase_merkle_proof: vec![n23],
    confirmations:         0,
})
// Returns: true  ‚Üź n01 is not a real transaction
```

Both proofs have length 1 (satisfying the length check), the coinbase check passes because `hash(n01, n23) = merkle_root`, and the tx inclusion check passes for the same reason. The function returns `true` for a fabricated `tx_id` that has never existed as a Bitcoin transaction.

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
