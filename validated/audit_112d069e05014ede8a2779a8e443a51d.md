### Title
SwapAllowlistExtension gates the router address instead of the end user, enabling allowlist bypass via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router**, not the end user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. If the router is added to the allowlist (a natural operational step to "enable router access"), every unpermissioned user can bypass the curated-pool gate by routing through it.

---

### Finding Description

**Pool passes its direct caller as `sender` to extensions.**

In `MetricOmmPool.swap`:

```solidity
_beforeSwap(
  msg.sender,   // ← router address when called via router
  recipient,
  ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as the `sender` argument to every configured extension:

```solidity
abi.encodeCall(
  IMetricOmmExtensions.beforeSwap,
  (sender, recipient, ...)
)
``` [2](#0-1) 

**`SwapAllowlistExtension` checks that `sender` argument.**

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [3](#0-2) 

`msg.sender` here is the pool (correct for the mapping key), and `sender` is the first parameter — the router's address when the call originates from `MetricOmmSimpleRouter`.

**The router never forwards the original user's identity to the pool.**

`exactInputSingle` stores `msg.sender` in transient storage for the payment callback, but the `pool.swap()` call itself carries no user-identity argument:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [4](#0-3) 

The pool therefore sees `msg.sender = router`, and the extension checks `allowedSwapper[pool][router]`.

**Two concrete failure modes arise:**

| Scenario | Effect |
|---|---|
| Pool admin adds the router to the allowlist to "enable router access" | Every user — including those not individually allowlisted — can trade on the curated pool by routing through the router. The allowlist is fully bypassed. |
| Pool admin does not add the router | Individually allowlisted users cannot use the router at all; they must call the pool directly. The router is broken for any allowlisted pool. |

Note that `DepositAllowlistExtension` avoids this problem because `addLiquidity` accepts an explicit `owner` parameter that the caller sets to the LP's address, and the extension checks `owner` rather than `sender`. No equivalent explicit-user parameter exists in `swap`. [5](#0-4) 

---

### Impact Explanation

A curated pool using `SwapAllowlistExtension` to restrict trading to a known set of addresses can be fully bypassed by any unpermissioned user who routes through `MetricOmmSimpleRouter`. The attacker does not need any special privilege; they only need to call the public router. This constitutes a direct loss of curation control and allows unauthorized parties to drain or trade against the pool's liquidity at oracle-anchored prices, which is the core economic action the allowlist was designed to prevent.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary supported periphery entry point for swaps. Any pool operator who adds the router to the allowlist (a natural and expected operational step) immediately opens the gate to all users. Even without that step, the broken router path for allowlisted users is a reachable, unprivileged condition triggered by normal usage. No oracle manipulation, flash loan, or admin compromise is required.

---

### Recommendation

The extension must check the economically relevant actor — the end user — not the intermediary. Two viable approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a convention between router and extension.

2. **Check `recipient` instead of `sender`**: If the pool's design guarantees that `recipient` is always the end user (which it is not in multi-hop paths where intermediate recipients are the router itself), this would work for single-hop swaps only.

3. **Structural fix**: Add an explicit `payer` or `originator` field to the pool's `swap` signature (analogous to `owner` in `addLiquidity`) so the router can forward the real user's address and the extension can check it unambiguously.

---

### Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Pool admin allowlists only user1:
       setAllowedToSwap(pool, user1, true)
3. Pool admin also adds the router to enable router-based access:
       setAllowedToSwap(pool, router, true)
4. user2 (NOT individually allowlisted) calls:
       router.exactInputSingle({pool: pool, recipient: user2, ...})
   → router calls pool.swap(user2, ...)
   → pool calls extension.beforeSwap(router, user2, ...)
   → extension checks: allowedSwapper[pool][router] == true  ✓ PASSES
   → user2 successfully trades on the curated pool.

5. Conversely, if the router is NOT added:
       user1 calls router.exactInputSingle(...)
   → extension checks: allowedSwapper[pool][router] == false  ✗ REVERTS
   → user1 (individually allowlisted) cannot use the router at all.
```
