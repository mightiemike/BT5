### Title
Unchecked `transferFrom` Return Value Enables USDC.e Drain from DirectDepositV1 Accounts — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the boolean return value. The function is publicly callable (no access modifier) on Ink chain. If the USDC token returns `false` on failure rather than reverting, the function unconditionally proceeds to withdraw USDC.e from the DDA and transfer it to the caller — resulting in the caller receiving USDC.e without having provided USDC.

---

### Finding Description

In `ContractOwner.replaceUsdcEWithUsdc`, the token swap logic is:

1. Read USDC.e balance held by the DDA.
2. Pull USDC from `msg.sender` into the DDA via `transferFrom` — **return value not checked**.
3. Withdraw USDC.e from the DDA to `ContractOwner` via `DirectDepositV1.withdraw`.
4. Transfer USDC.e from `ContractOwner` to `msg.sender` via `safeTransfer`. [1](#0-0) 

The critical line is:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

This is a raw interface call — not routed through `ERC20Helper.safeTransferFrom` — and the `bool` return value is silently discarded. Steps 3 and 4 execute unconditionally regardless of whether step 2 succeeded. [2](#0-1) 

Contrast this with every other transfer site in the protocol, which uses `ERC20Helper.safeTransferFrom` or `ERC20Helper.safeTransfer`, both of which enforce:

```solidity
require(success && (data.length == 0 || abi.decode(data, (bool))), ERR_TRANSFER_FAILED);
``` [3](#0-2) 

The function carries no access modifier, making it reachable by any unprivileged caller on Ink chain (chainid 57073). [4](#0-3) 

---

### Impact Explanation

If the USDC token at the hardcoded address (`0x2D270e6886d130D724215A266106e6832161EAEd`) returns `false` on a failed transfer (e.g., insufficient allowance or balance) rather than reverting:

- The attacker calls `replaceUsdcEWithUsdc(victimSubaccount)` with zero USDC allowance.
- `transferFrom` returns `false`; no USDC moves.
- `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes, pulling all USDC.e from the DDA into `ContractOwner`.
- `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` sends the full USDC.e balance to the attacker.

**Corrupted asset delta**: The DDA's USDC.e balance is zeroed; the attacker receives it for free. The subaccount's pending deposit credit is also lost since the DDA no longer holds the collateral.

---

### Likelihood Explanation

The hardcoded USDC address on Ink chain is expected to be Circle's standard implementation, which reverts on failure rather than returning `false`. This reduces immediate exploitability. However:

- The code-level root cause is entirely within Nado (missing return value check), not in the token.
- If the token is upgraded, replaced, or if a non-standard bridged variant is deployed at that address, the vulnerability becomes directly exploitable.
- The function is permissionlessly callable by any address on Ink chain, with no rate limiting or access control.

Likelihood: **Low** under current token behavior; **High** if token behavior changes.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's own safe wrapper:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe, consistent with rest of protocol):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`safeTransferFrom` in `ERC20Helper` performs a low-level call and requires `success && (data.length == 0 || abi.decode(data, (bool)))`, matching the SafeERC20 pattern used elsewhere. [3](#0-2) 

---

### Proof of Concept

**Chain**: Ink (chainid 57073)  
**Precondition**: A DDA exists for `victimSubaccount` with non-zero USDC.e balance. Attacker has zero USDC allowance set for `ContractOwner`.

1. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(victimSubaccount)`.
2. `balance = IERC20Base(usdcE).balanceOf(directDepositV1)` → e.g. `1000e6`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, 1000e6)` → returns `false` (no allowance). **Return value ignored.**
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` → transfers `1000e6` USDC.e from DDA to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, 1000e6)` → attacker receives `1000e6` USDC.e.

Net result: attacker gains `1000e6` USDC.e; DDA is drained; no USDC was provided. [1](#0-0)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

**File:** core/contracts/libraries/ERC20Helper.sol (L23-42)
```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
