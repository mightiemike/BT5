### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Allows Draining DDA usdcE Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` is an unpermissioned external function that uses a raw `transferFrom` call whose return value is never checked. If the USDC token at the hardcoded address returns `false` instead of reverting on a failed transfer (e.g., insufficient allowance), execution continues, causing the function to drain the entire usdcE balance from the victim's `DirectDepositV1` (DDA) address and send it to the attacker — without the attacker ever providing any USDC.

---

### Finding Description

`ContractOwner.replaceUsdcEWithUsdc` is callable by **any** external address on chain 57073 (Ink). It has no `onlyOwner` or similar access control — only a `chainid` check. [1](#0-0) 

The function's intended logic is:
1. Read the usdcE balance held by the target DDA.
2. Pull an equivalent amount of USDC from `msg.sender` into the DDA.
3. Withdraw the usdcE from the DDA to `ContractOwner`.
4. Forward the usdcE to `msg.sender`.

The critical flaw is at step 2:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [2](#0-1) 

The return value of `transferFrom` is **never checked**. If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (Ink chain) returns `false` on a failed transfer rather than reverting — a known behavior of non-standard ERC-20 tokens — execution falls through to steps 3 and 4 unconditionally.

Step 3 calls `DirectDepositV1.withdraw`, which is `onlyOwner` — but `ContractOwner` is the owner of every DDA it deploys, so this succeeds: [3](#0-2) 

Step 4 then uses `safeTransfer` (which does check the return value) to send the usdcE to `msg.sender`: [4](#0-3) 

The `ERC20Helper.safeTransferFrom` wrapper used elsewhere in the codebase correctly checks the return value: [5](#0-4) 

But `replaceUsdcEWithUsdc` bypasses this wrapper and calls `transferFrom` directly on the raw `IERC20Base` interface, which only declares the function signature without enforcing return-value checking: [6](#0-5) 

---

### Impact Explanation

An attacker with zero USDC allowance calls `replaceUsdcEWithUsdc(subaccount)` targeting any subaccount whose DDA holds usdcE. The `transferFrom` silently fails (returns `false`), the DDA's entire usdcE balance is withdrawn to `ContractOwner`, and then forwarded to the attacker via `safeTransfer`. The DDA is left with zero usdcE. The subaccount's pending collateral credit — which depends on the DDA holding those tokens before `creditDeposit` is called — is permanently destroyed. This is a direct, irreversible asset theft from the protocol's deposit pipeline.

---

### Likelihood Explanation

The function is externally callable by any address on chain 57073 with no access control beyond the chain ID check. The exploitability depends on whether the USDC token at the hardcoded address returns `false` on failed transfers rather than reverting. Non-standard ERC-20 tokens with this behavior are well-documented (USDT on mainnet being the canonical example). If the Ink-chain USDC deployment exhibits this behavior, the attack is immediately executable by any unprivileged caller against any DDA with a non-zero usdcE balance.

---

### Recommendation

Replace the raw `transferFrom` call with the `safeTransferFrom` wrapper already present in `ERC20Helper`, which checks the return value and reverts on failure:

```solidity
// Before (vulnerable):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

Additionally, add an `onlyOwner` modifier to `replaceUsdcEWithUsdc` to restrict it to the protocol multisig, consistent with all other sensitive DDA management functions in `ContractOwner`.

---

### Proof of Concept

1. Identify any `subaccount` whose DDA (`directDepositV1Address[subaccount]`) holds a non-zero usdcE balance.
2. As an attacker with **zero** USDC allowance granted to `ContractOwner`, call:
   ```solidity
   ContractOwner.replaceUsdcEWithUsdc(subaccount)
   ```
   on chain 57073.
3. `IERC20Base(usdc).transferFrom(attacker, dda, balance)` returns `false` (no revert).
4. `DirectDepositV1(dda).withdraw(usdcE)` executes — transfers all usdcE from DDA to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, balance)` executes — attacker receives the full usdcE balance.
6. The DDA is emptied. The victim subaccount's pending deposit collateral is lost. The attacker paid nothing.

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
