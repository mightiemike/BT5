Audit Report

## Title
`SwapAllowlistExtension` Checks Intermediary Router Address Instead of Actual Swapper, Enabling Allowlist Bypass — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` resolves to the router's address rather than the end-user's address. Any pool admin who allowlists the router (a necessary step to enable router-mediated swaps for curated users) inadvertently grants every unprivileged user the ability to bypass the allowlist by routing through the public router.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
```

`ExtensionCalling._beforeSwap` forwards this value unchanged as `sender` to every configured extension:

```solidity
// ExtensionCalling.sol L160-165
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, ...)
    )
```

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` inside the pool:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The exploit path:
1. Pool admin deploys pool with `SwapAllowlistExtension` and allowlists `userA` via `setAllowedToSwap(pool, userA, true)`.
2. Pool admin allowlists the router via `setAllowedToSwap(pool, router, true)` — required so `userA` can use `MetricOmmSimpleRouter` (without this, even `userA`'s router calls fail the check).
3. `userB` (not allowlisted) calls `router.exactInputSingle({pool: pool, recipient: userB, ...})`.
4. Router calls `pool.swap(userB, ...)` with `msg.sender = router`.
5. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
6. `userB`'s swap executes against LP liquidity despite being excluded.

No existing guard prevents this: the extension has no mechanism to distinguish the router acting on behalf of an authorized user from the router acting on behalf of an unauthorized one.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC-verified addresses, whitelisted market makers) loses that restriction entirely for any user who routes through the public `MetricOmmSimpleRouter`. Disallowed users can execute swaps against LP liquidity, enabling arbitrage or adversarial trading that the allowlist was designed to prevent. This constitutes a direct loss of LP principal and a broken core pool invariant (broken allowlist gating on swap).

## Likelihood Explanation
The bypass requires the pool admin to have added the router to the allowlist. This is a natural and expected administrative action: without it, even allowlisted users cannot use the router. Any pool that intends to support router-mediated swaps for its curated users is therefore vulnerable. The unprivileged trigger (routing through the public router) requires no special access and is repeatable by any address.

## Recommendation
The extension must gate on the actual end-user's identity, not the intermediary's. Two viable approaches:

1. **Extension-data forwarding**: Require the router to embed the real user's address in `extensionData` and have the extension verify it (with a signature or trusted-router attestation).
2. **Recipient-based check**: Gate on `recipient` instead of `sender` when `sender` is a known router, or require the pool admin to allowlist individual users and never allowlist routers.

The root cause is that `sender = msg.sender` of `pool.swap()` is the wrong identity to check when an intermediary router is in the call path.

## Proof of Concept
```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, userA, true).
3. Pool admin calls setAllowedToSwap(pool, router, true)
   — necessary so userA can use MetricOmmSimpleRouter.
4. Disallowed userB calls:
       router.exactInputSingle({pool: pool, recipient: userB, ...})
5. Router calls pool.swap(userB, ...) with msg.sender = router.
6. Extension evaluates allowedSwapper[pool][router] == true → passes.
7. userB's swap executes against LP liquidity despite being excluded.
```

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist `userA` and the router, then call `exactInputSingle` from `userB` and assert the swap succeeds (no `NotAllowedToSwap` revert).
