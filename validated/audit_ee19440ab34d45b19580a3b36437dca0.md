Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Share Recipient (`owner`) Instead of Actual Caller (`sender`), Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first argument (the real caller/`msg.sender` of the pool) and gates access by the `owner` parameter — the address that will receive LP shares — which any caller may set to an arbitrary allowlisted address. Because `addLiquidity` imposes no restriction on who may supply an arbitrary `owner` value, any unprivileged address can bypass the deposit allowlist entirely by naming a known allowlisted address as `owner`, depositing tokens into a restricted pool, and forcing LP shares onto the allowlisted address.

## Finding Description

`MetricOmmPool.addLiquidity` calls the `beforeAddLiquidity` hook with `msg.sender` as the first argument (the real caller) and the caller-supplied `owner` as the second: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` discards the first argument and checks only `owner`: [2](#0-1) 

The check `allowedDepositor[msg.sender][owner]` uses `msg.sender` as the pool address (correct) but `owner` as the depositor identity (wrong — this is the LP share recipient, not the caller). The actual caller is never verified.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks the first argument (`sender`, the real caller): [3](#0-2) 

`removeLiquidity` enforces `msg.sender == owner`, preventing the same pattern there: [4](#0-3) 

But `addLiquidity` has no such restriction, so `owner` is a fully attacker-controlled parameter: [5](#0-4) 

**Exploit path:**
1. Pool `P` is deployed with `DepositAllowlistExtension`; `allowAllDepositors[P] = false`.
2. Admin calls `setAllowedToDeposit(P, alice, true)`. Alice is the only allowlisted depositor.
3. Bob (not allowlisted) calls `P.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. `_beforeAddLiquidity(bob, alice, ...)` is dispatched to the extension.
5. `beforeAddLiquidity` checks `allowedDepositor[P][alice]` → `true` → no revert.
6. Bob's swap callback transfers tokens into the pool; Alice receives LP shares she never requested.
7. Bob has deposited into a restricted pool without being on the allowlist.

## Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting which addresses may inject liquidity (e.g., KYC/AML gating, curated market-maker whitelists, regulatory compliance). With this bug, the allowlist is rendered completely ineffective: any address can deposit by naming any allowlisted address as `owner`. Additionally, the allowlisted `owner` receives LP shares they never requested, which can be used to grief them (force unwanted exposure, complicate accounting, or lock them into a position they must actively unwind). This is a direct admin-boundary break where an unprivileged path bypasses a pool admin access control, with concrete fund-flow consequences (unauthorized token injection into the pool, forced LP positions).

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with a known allowlisted address as `owner`. The `allowedDepositor` mapping is public, so any observer can identify allowlisted addresses. No special privileges, flash loans, oracle manipulation, or complex setup is needed. Any address that can call `addLiquidity` can exploit this immediately and repeatably.

## Recommendation

Change `beforeAddLiquidity` to check the first parameter (the actual caller/sender) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
// Fixed:
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Also update `setAllowedToDeposit`, `isAllowedToDeposit`, and the `allowedDepositor` mapping semantics to reflect that the allowlisted entity is the caller, not the LP share recipient.

## Proof of Concept

1. Deploy pool `P` with `DepositAllowlistExtension`; `allowAllDepositors[P] = false`.
2. Admin calls `setAllowedToDeposit(P, alice, true)`.
3. Bob (not allowlisted) calls `P.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Extension receives `beforeAddLiquidity(bob, alice, ...)`, checks `allowedDepositor[P][alice]` → `true` → passes.
5. Bob's callback transfers tokens; Alice receives LP shares.
6. Assert: Bob deposited into a restricted pool without being allowlisted; Alice holds unwanted LP shares.
