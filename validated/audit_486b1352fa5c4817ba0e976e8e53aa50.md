Audit Report

## Title
`SwapAllowlistExtension` Bypassed via Router: `msg.sender` (Router) Checked Instead of Actual User - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end user. If the router is allowlisted (required for any router-mediated swap to succeed), every user—including non-allowlisted ones—can bypass the per-user swap gate by routing through the router.

## Finding Description
**Root cause — `sender` binding:**

In `MetricOmmPool.swap` (line 231), the pool passes `msg.sender` as `sender` to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap()
    recipient, ...
);
```

`ExtensionCalling._beforeSwap` forwards this value verbatim to the extension hook (line 165). `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

**Router path:**

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)` directly (lines 72–80, 104–112, 136–137, 165–181). The pool therefore sees `msg.sender = router`. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

**Exploit flow:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to KYC'd addresses only.
2. Pool admin must allowlist the router (`allowedSwapper[pool][router] = true`) so that any legitimate user can use the router.
3. A non-allowlisted attacker calls `router.exactInputSingle(...)`.
4. Router calls `pool.swap(...)` → pool passes `sender = router` to the extension.
5. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. Attacker successfully swaps in a pool they are not individually authorized to access.

**Existing guards are insufficient:** The extension has no mechanism to recover the original `msg.sender` from the router context. The `onlyPool` modifier on the extension only verifies the pool is the caller, not the identity of the end user.

## Impact Explanation
The swap allowlist—a core access-control extension—is rendered ineffective for all router-mediated swaps. Any unprivileged user can bypass per-user swap restrictions by routing through the public `MetricOmmSimpleRouter`. This breaks the intended access-control invariant of the extension and constitutes broken core pool functionality. Pools designed to restrict swaps to trusted counterparties (e.g., for regulatory compliance or risk management) are fully exposed to unauthorized swappers, which can result in direct loss of LP assets if the pool's pricing or liquidity assumptions depend on counterparty identity.

## Likelihood Explanation
Exploitable by any unprivileged user with no special setup beyond using the publicly deployed router. The condition (router allowlisted) is the expected operational state for any pool that intends to support router-mediated swaps. Repeatable on every block.

## Recommendation
Pass the true end-user identity through the swap path. One approach: have the router forward the original `msg.sender` as part of `extensionData`, and update `SwapAllowlistExtension.beforeSwap` to decode and check that address when present. Alternatively, the pool could accept an explicit `swapper` parameter (separate from `msg.sender`) that the router populates, though this requires a core interface change. At minimum, document that allowlisting the router grants unrestricted swap access to all users.
