Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end user, making per-user allowlists bypassable — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's own `msg.sender` — the router contract — when a user swaps via `MetricOmmSimpleRouter`. The extension therefore gates the router's address rather than the actual end user's address. A pool admin who allowlists the router to enable router-mediated swaps for approved users inadvertently opens the pool to every address on the network, completely defeating the per-user allowlist invariant.

## Finding Description
`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

When the call originates from `MetricOmmSimpleRouter.exactInputSingle`, the pool's `msg.sender` is the router contract, not the end user.

`SwapAllowlistExtension.beforeSwap` then checks that `sender` value against the allowlist: [2](#0-1) 

Here `msg.sender` is the pool (correct key) and `sender` is the router — not the actual end user.

The router stores the real user's address only in transient storage for payment purposes and never surfaces it to the pool or extension as the swap initiator: [3](#0-2) 

`DepositAllowlistExtension` avoids this problem by checking `owner` (the LP position owner, passed as a distinct parameter) rather than the immediate caller: [4](#0-3) 

No analogous separation exists in `SwapAllowlistExtension`.

## Impact Explanation
**Bypass path (router allowlisted):** Any unprivileged address calls `exactInputSingle` on the router targeting a restricted pool. The extension sees `allowedSwapper[pool][router] == true` and approves the swap. The per-user allowlist is completely bypassed, exposing LP funds to unauthorized traders. This is a direct loss-of-funds risk for LPs who deployed the extension expecting a restricted counterparty set.

**DoS path (router not allowlisted):** Allowlisted users who attempt to swap through the router are permanently blocked because the extension sees the router address, which is not in the allowlist. The only usable path is a direct pool call requiring the caller to implement `IMetricOmmSwapCallback`, making the public router unusable for any allowlisted pool.

Both outcomes break the core invariant stated in the NatSpec: "Gates `swap` by swapper address, per pool." [5](#0-4) 

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary public entry point for swaps. Any pool that deploys `SwapAllowlistExtension` and expects per-user gating will encounter this on the first router-mediated swap. No special privileges or unusual conditions are required — a standard `exactInputSingle` call is sufficient to trigger either failure mode. The bypass requires only that the pool admin allowlists the router (the natural action to enable router usage), after which every address on the network can swap freely.

## Recommendation
The extension must check the actual end-user's address, not the immediate pool caller. Two viable approaches:

1. **Router forwards the real user via `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires a trusted encoding convention between the router and the extension.

2. **Pool exposes the original initiator:** Add a separate `initiator` field to the `beforeSwap` hook signature that the pool populates from a transient-storage context set at the top of `swap`, analogous to how the router already stores the payer in transient storage. The extension then checks `initiator` instead of `sender`.

## Proof of Concept
```
Setup:
  pool P with SwapAllowlistExtension E
  allowedSwapper[P][userA] = true   // intended: only userA may swap
  allowedSwapper[P][router] = true  // needed so userA can use the router

Attack:
  userB (not allowlisted) calls:
    router.exactInputSingle({pool: P, ...})

Execution trace:
  router.exactInputSingle()                        // msg.sender = userB
    → pool.swap(msg.sender = router, ...)
      → extension.beforeSwap(sender = router, ...) // msg.sender = pool
        → allowedSwapper[pool][router] == true ✓   // no revert
      → swap executes for userB

Result:
  userB successfully swaps in a pool they were never authorized to access.
  The per-user allowlist is completely bypassed.
```

A Foundry integration test can reproduce this by deploying a pool with `SwapAllowlistExtension`, allowlisting only `userA` and the router, then calling `exactInputSingle` from `userB` and asserting the swap succeeds (demonstrating the bypass) or asserting it reverts when the router is not allowlisted (demonstrating the DoS).
