### Title
SwapAllowlistExtension Checks Router Address as Swapper Instead of Actual User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the actual user. If the pool admin allowlists the router (required for any router-mediated swap to work), every unprivileged user can bypass the allowlist by routing through the router.

---

### Finding Description

The call chain for a router-mediated swap is:

```
User → MetricOmmSimpleRouter.exactInputSingle()
     → IMetricOmmPoolActions(pool).swap(recipient, ...)   // msg.sender = router
     → MetricOmmPool._beforeSwap(msg.sender=router, ...)
     → ExtensionCalling._callExtensionsInOrder(...)
     → SwapAllowlistExtension.beforeSwap(sender=router, ...)
```

In `MetricOmmSimpleRouter.exactInputSingle`, the pool's `swap()` is called with no mechanism to forward the actual user's identity to the extension layer: [1](#0-0) 

`MetricOmmPool.swap()` passes `msg.sender` (the router) as `sender` to `_beforeSwap`: [2](#0-1) 

`ExtensionCalling._beforeSwap` encodes that same `sender` (router) as the first argument to every extension: [3](#0-2) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[pool][sender]` where `sender` is the router: [4](#0-3) 

The allowlist is keyed `allowedSwapper[pool][swapper]`. When the router is the swapper seen by the extension, the pool admin faces an impossible choice:

- **Do not allowlist the router** → all router-mediated swaps revert for every user, including allowlisted ones.
- **Allowlist the router** → every user on the network can bypass the allowlist by calling `exactInputSingle` or `exactInput` through the router.

There is no path that allows router-mediated swaps for allowlisted users only.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, protocol-internal actors, or whitelist-only LPs) loses that protection entirely the moment the router is allowlisted. Any unprivileged user can:

1. Call `MetricOmmSimpleRouter.exactInputSingle` targeting the restricted pool.
2. The extension sees `sender = router`, which is allowlisted.
3. The swap executes as if the user were authorized.

This breaks the core allowlist invariant stated in the contest scope: *"deposit/swap allowlist checks must cover the exact actor/action intended and cannot be bypassed through router."* The consequence is unauthorized access to curated pool liquidity, potential regulatory/compliance failure for the pool operator, and direct loss of LP value if the pool's pricing or fee model was calibrated for a restricted counterparty set.

---

### Likelihood Explanation

The trigger is a standard, publicly documented periphery path (`MetricOmmSimpleRouter`). Any user who reads the contract can discover it. The only precondition is that the pool admin has allowlisted the router — a step that is operationally necessary for the pool to be usable via the standard interface. Pool admins who deploy an allowlisted pool and also want users to interact via the router will inevitably create this condition.

---

### Recommendation

The actual user's address must be forwarded through the hook call chain. Two viable approaches:

1. **Router-side**: Have `MetricOmmSimpleRouter` encode `msg.sender` (the actual user) into `extensionData` before calling `pool.swap()`, and have `SwapAllowlistExtension.beforeSwap` decode and check that address when `sender` is a known router.

2. **Extension-side**: Add a secondary check: if `sender` is a registered router/forwarder, decode the real originator from `extensionData` and apply the allowlist to that address instead.

Either approach must be applied consistently across `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` in the router.

---

### Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true          // alice is the intended gated user
  allowedSwapper[pool][router] = true         // admin adds router to support alice's router swaps
  allowedSwapper[pool][bob] = false           // bob is NOT allowlisted

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      recipient: bob,
      zeroForOne: true,
      amountIn: X,
      ...
  })

  → router calls pool.swap(bob, true, X, ...)
  → pool calls _beforeSwap(msg.sender=router, bob, ...)
  → SwapAllowlistExtension.beforeSwap(sender=router, ...)
  → allowedSwapper[pool][router] == true  → no revert
  → swap executes for bob despite bob not being allowlisted
``` [5](#0-4) [6](#0-5)

    _checkDeadline(params.deadline);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);

        "",

    amountOut = MetricOmmSwapInputs.int128ToUint128(out);

    _clearExpectedCallbackPool();
  }
