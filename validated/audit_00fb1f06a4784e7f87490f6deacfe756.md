Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Unauthorized Depositors to Bypass the Allowlist Guard - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and gates access only on `owner` (the position beneficiary). Because `MetricOmmPool.addLiquidity` explicitly permits `msg.sender != owner`, any address not on the allowlist can bypass the deposit guard by naming an allowlisted address as `owner`. This nullifies the pool admin's configured access control for the deposit path.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as separate arguments to `_beforeAddLiquidity`:

```solidity
// metric-core/contracts/MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol line 97
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` receives both but discards `sender` (the unnamed first parameter) and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol line 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The pool's own NatSpec documents the separation: `"msg.sender pays but need not equal owner (operator pattern)"`. The `addLiquidityWeighted(address pool, address owner, ...)` overload in `MetricOmmPoolLiquidityAdder` provides a ready-made public entry point that accepts an arbitrary `owner` while `msg.sender` (the actual payer) is stored as `payer` in transient context. An attacker calls this with an allowlisted `owner`; the extension evaluates `allowedDepositor[pool][allowlistedOwner]` → `true` and does not revert, while the actual depositor is never evaluated.

`SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper) and discards `recipient`, confirming the asymmetry is a defect in the deposit path, not an intentional design choice.

## Impact Explanation
Any address not on the allowlist can deposit tokens into a restricted pool by supplying an allowlisted address as `owner`. The actual depositor (`sender`) pays tokens via callback and receives LP shares credited to the allowlisted `owner`. The pool admin's configured deposit access control is silently bypassed for every deposit, constituting an admin-boundary break: an unprivileged actor circumvents a factory-configured guard. The allowlisted `owner` receives LP shares they did not authorize, and the pool's restricted-access invariant is violated.

## Likelihood Explanation
The exploit requires no special privileges, no malicious token behavior, and no non-standard setup. Any actor who knows one allowlisted address can execute it immediately via `MetricOmmPoolLiquidityAdder.addLiquidityWeighted(pool, allowlistedOwner, ...)` or by calling `pool.addLiquidity(allowlistedOwner, ...)` directly with a custom callback. The `addLiquidity` interface explicitly documents `msg.sender != owner` as a supported pattern, making this trivially reachable.

## Recommendation
Replace the unnamed first parameter with `sender` and gate on it, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate on `owner` (position beneficiary), the NatSpec, extension description, and `addLiquidityWeightedFor` path must all be separately hardened to prevent an unprivileged payer from depositing on behalf of an allowlisted owner.

## Proof of Concept
1. Deploy pool with `DepositAllowlistExtension`; pool admin calls `setAllowedToDeposit(pool, alice, true)`. `bob` is not allowlisted.
2. `bob` calls `MetricOmmPoolLiquidityAdder.addLiquidityWeighted(pool, alice, salt, deltas, max0, max1, ...)`.
3. `MetricOmmPoolLiquidityAdder` calls `pool.addLiquidity(alice, salt, weightDeltas, abi.encode(KIND_PROBE), extensionData)` — pool fires `extension.beforeAddLiquidity(liquidityAdder, alice, ...)`.
4. Extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. Probe reverts with `LiquidityProbe(need0, need1)`; adder rescales and calls `_addLiquidity(pool, alice, salt, scaled, bob, ...)`.
6. Pool calls `extension.beforeAddLiquidity(liquidityAdder, alice, ...)` again → passes.
7. Pool invokes `metricOmmModifyLiquidityCallback`; adder pulls tokens from `bob` (the stored `payer`) and transfers them to the pool; `alice` receives LP shares.
8. `bob` has deposited into a pool he is explicitly barred from, bypassing the allowlist entirely.
