### Title
SwapAllowlistExtension Allowlist Bypassed via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]` where `sender` is the immediate `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router`, so the extension checks the router's allowlist entry rather than the actual end-user's. If the pool admin allowlists the router (the only way to let allowlisted users reach the pool via the router), every unprivileged user can bypass the allowlist by routing through the same public contract.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct — enforced by `onlyPool`) and `sender` is the first argument forwarded by the pool from `ExtensionCalling._beforeSwap`:

```solidity
// ExtensionCalling.sol lines 149-176
function _beforeSwap(
    address sender,   // ← pool passes msg.sender of swap()
    address recipient,
    ...
) internal {
    _callExtensionsInOrder(
      BEFORE_SWAP_ORDER,
      abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
    );
}
```

`pool.swap()` has no explicit `sender` parameter:

```solidity
// IMetricOmmPoolActions.sol line 188
function swap(
    address recipient,
    bool zeroForOne,
    int128 amountSpecified,
    uint128 priceLimitX64,
    bytes calldata callbackData,
    bytes calldata extensionData
) external returns (int128 amount0Delta, int128 amount1Delta);
```

The pool therefore passes its own `msg.sender` — the immediate caller — as `sender` to `_beforeSwap`. When `MetricOmmSimpleRouter` calls `pool.swap(...)`, `sender = address(router)`.

The extension then evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

This creates an irresolvable dilemma for any pool admin who wants to run a restricted pool while still supporting the router:

| Router allowlist state | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users **cannot** use the router — broken core flow |
| Router **allowlisted** | **Every** user bypasses the allowlist via the router |

Note the contrast with `DepositAllowlistExtension`, which correctly gates on the `owner` parameter (the LP-share recipient, explicitly supplied by the caller) rather than on `sender`, making it router-agnostic. The swap extension has no equivalent identity anchor.

---

### Impact Explanation

Any user excluded from a restricted pool's swap allowlist can execute swaps by calling `MetricOmmSimpleRouter` instead of the pool directly. The pool admin's intended access boundary — which may exist to limit trading to specific counterparties, satisfy regulatory requirements, or protect a nascent liquidity position — is silently voided. This is an admin-boundary break: a factory-level role check (`allowedSwapper`) is bypassed by an unprivileged path (the public router).

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary user-facing swap interface; pool admins who want allowlisted users to use it must allowlist the router.
- No special privilege is required to call the router — any EOA or contract can do so.
- The bypass is silent: the extension sees a valid allowlist entry (the router) and does not revert.
- The pool admin has no on-chain mechanism to distinguish per-user router calls; the only fix available to them is to remove the router from the allowlist entirely, which breaks the router for all users.

---

### Recommendation

Gate on the actual end-user identity rather than the immediate caller. Two viable approaches:

1. **Pass the originating user through the router**: Have `MetricOmmSimpleRouter` call a pool entry-point that accepts an explicit `sender` argument (authenticated via `msg.sender == router` and a trusted forwarding pattern), and have the pool forward that address to `_beforeSwap` as `sender`.

2. **Check `recipient` instead of `sender`**: For swap allowlists, gate on `recipient` (the address receiving output tokens) rather than `sender` (the address paying input tokens). This is already a parameter in `beforeSwap` and is set by the end-user in the router call, making it a more stable identity anchor.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, user1, true)       // allowlist user1
3. Pool admin calls setAllowedToSwap(pool, router, true)      // required for user1 to use router
4. user2 (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle(
           pool, zeroForOne, amountIn, minOut, deadline, extensionData
       )
5. Router calls pool.swap(recipient=user2, ...) → msg.sender = router
6. Pool calls _beforeSwap(sender=router, ...)
7. Extension checks allowedSwapper[pool][router] → true → no revert
8. user2 swap executes successfully, bypassing the allowlist.
