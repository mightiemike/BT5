### Title
`ValidResourceBounds::AllResources` with zero L2/L1DataGas silently collapses to `L1Gas` on protobuf round-trip, producing a divergent transaction hash — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to distinguish the pre-0.13.3 `L1Gas` variant from the post-0.13.3 `AllResources` variant. When both `l2_gas` and `l1_data_gas` are zero, it always emits `L1Gas`, even if the original transaction was canonically `AllResources`. Because `get_tip_resource_bounds_hash` hashes a different number of resource fields for each variant (2 for `L1Gas`, 3 for `AllResources`), the transaction hash computed at the RPC gateway diverges from the hash computed by any node that receives the transaction over P2P.

---

### Finding Description

**Step 1 – Origination (RPC path)**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`, which is always wrapped as `ValidResourceBounds::AllResources` when converted to the internal `InvokeTransactionV3`. A user may legitimately set `l2_gas = 0` and `l1_data_gas = 0` while keeping a non-zero `l1_gas` bound; the gateway's stateless validator accepts this.

The hash is then computed by `get_invoke_transaction_v3_hash` via `get_tip_resource_bounds_hash`:

```
AllResources path → resource_felts = [l1_gas_packed, l2_gas_packed(=0), l1_data_gas_packed(=0)]
hash_H1 = poseidon(tip, l1_gas_packed, 0, 0)   // 3 resource elements
``` [1](#0-0) 

**Step 2 – Serialisation to protobuf**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serialises `AllResources` with zero l2/l1_data_gas as:
```
l1_gas = Some(l1_gas), l2_gas = Some(zero), l1_data_gas = Some(zero)
``` [2](#0-1) 

**Step 3 – Deserialisation (P2P path) — the bug**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies the heuristic:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant chosen
} else {
    ValidResourceBounds::AllResources(...)
})
``` [3](#0-2) 

The `AllResources(zero, zero)` message is indistinguishable from a `L1Gas` message at the protobuf level, so the receiving node reconstructs the transaction as `L1Gas`.

**Step 4 – Hash divergence**

When the receiving node recomputes the hash from the deserialized `L1Gas` representation:

```
L1Gas path → resource_felts = [l1_gas_packed, l2_gas_packed(=0)]
hash_H2 = poseidon(tip, l1_gas_packed, 0)       // 2 resource elements
```

`H1 ≠ H2` because `poseidon(tip, x, 0, 0) ≠ poseidon(tip, x, 0)`.

The same divergence applies to `get_declare_transaction_v3_hash` and `get_deploy_account_transaction_v3_hash`, which both call `get_tip_resource_bounds_hash`. [4](#0-3) 

---

### Impact Explanation

Any node that receives the transaction over P2P will recompute hash H2 from the `L1Gas` representation. If it compares H2 against the transmitted hash H1 (computed at the originating gateway), it will detect a mismatch and reject the transaction. A valid, correctly-signed transaction is therefore silently dropped by every peer that receives it via P2P, while the originating node retains it with hash H1. This is a transaction-conversion bug that binds the wrong hash to the executable payload, matching the **High** impact: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

The trigger requires only that a user submit a V3 `Invoke`, `Declare`, or `DeployAccount` transaction through the RPC gateway with `l2_gas = 0` and `l1_data_gas = 0` (non-zero `l1_gas` only). The gateway's stateless validator explicitly accepts this configuration. [5](#0-4) 

No special privileges are required; any unprivileged user can craft such a transaction.

---

### Recommendation

Replace the value-based heuristic with an explicit version/type tag. The protobuf `ResourceBounds` message should carry a boolean or enum field that records whether the sender intended `L1Gas` or `AllResources`, rather than inferring it from whether the numeric values happen to be zero. Until the protobuf schema is updated, the deserialiser must never emit `L1Gas` for a message that was serialised from an `AllResources` variant; one safe interim fix is to always emit `AllResources` when all three fields are present (even if zero), reserving `L1Gas` only for messages where `l1_data_gas` is absent (`None`).

---

### Proof of Concept

```
1. User submits RpcInvokeTransactionV3 {
       resource_bounds: AllResourceBounds {
           l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
           l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
           l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
       },
       ...
   }

2. Gateway converts to InternalRpcInvokeTransactionV3 (resource_bounds: AllResourceBounds).
   Hash H1 = poseidon(tip, l1_gas_packed, 0_l2_packed, 0_l1data_packed)
   [3 resource elements — AllResources branch of get_tip_resource_bounds_hash]

3. Transaction propagated via P2P as protobuf::InvokeV3 with
   resource_bounds { l1_gas=Some(1000,1), l2_gas=Some(0,0), l1_data_gas=Some(0,0) }

4. Receiving node deserialises:
   l1_data_gas.is_zero() && l2_gas.is_zero()  →  ValidResourceBounds::L1Gas(l1_gas)

5. Receiving node recomputes hash:
   H2 = poseidon(tip, l1_gas_packed, 0_l2_packed)
   [2 resource elements — L1Gas branch of get_tip_resource_bounds_hash]

6. H1 ≠ H2  →  transaction rejected / hash binding wrong.
``` [3](#0-2) [1](#0-0) [6](#0-5)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
