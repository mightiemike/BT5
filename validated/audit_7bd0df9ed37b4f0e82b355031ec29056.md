### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable Double-Claiming Bonus Rewards from Module Pool - (File: x/tieredrewards/keeper/msg_server.go)

---

### Summary
The `ClaimTierRewards` message handler in the tieredrewards module loads all requested position states before processing them, with no deduplication of the caller-supplied `PositionIds` slice. An unprivileged position owner can include the same position ID twice (or more) in a single `MsgClaimTierRewards` transaction. Because each copy of the position is loaded from state before any writes occur, both copies carry the same stale `LastEventSeq` and `LastBonusAccrual` checkpoint. The bonus-reward computation and `bankKeeper.SendCoinsFromModuleToAccount` call execute once per list entry, so the owner receives bonus rewards multiple times for a single accrual period, draining the `RewardsPoolName` module account beyond what is legitimately owed.

---

### Finding Description

`ClaimTierRewards` in `msg_server.go` iterates over `msg.PositionIds`, loads each position state, and appends it to a local slice: [1](#0-0) 

All positions are loaded **before** any state mutation. The slice is then passed to `claimRewardsAndUpdatesPositions`: [2](#0-1) 

Inside `claimRewardsAndUpdatesPositions`, each entry is processed independently: [3](#0-2) 

For each entry the function calls `processEventsAndClaimBonus`, which:

1. Reads validator events since `pos.LastEventSeq` from the store.
2. Computes a bonus amount.
3. Calls `bankKeeper.SendCoinsFromModuleToAccount` to transfer bonus coins from `RewardsPoolName` to the owner.
4. Calls `setPosition` to persist the updated checkpoint. [4](#0-3) 

When the same position ID appears at index 0 and index 1 in `positions`, the first iteration processes the position and writes the updated `LastEventSeq`/`LastBonusAccrual` to the store. The second iteration operates on `positions[1]`, which is a **separate in-memory copy** loaded before the first write, still carrying the original stale checkpoint. It therefore re-reads the same validator events, recomputes the same bonus, and issues a second `SendCoinsFromModuleToAccount` call for the same amount.

Additionally, even if all historical events have been consumed and their reference counts decremented to zero, the "current segment" bonus is computed unconditionally whenever the validator is bonded: [5](#0-4) 

This segment is computed from `blockTime` and `currentRate`, both of which are identical across both iterations, so it is paid out twice regardless of event state.

The second `setPosition` call then overwrites the store with the stale checkpoint from `positions[1]`, resetting `LastEventSeq` and `LastBonusAccrual` to their pre-claim values. This means the position's reward checkpoint is rolled back, allowing the same accrual window to be claimed again in future transactions.

---

### Impact Explanation

- **Corrupted value:** `types.RewardsPoolName` module account balance (bond denom).
- **Direct effect:** An attacker claiming a position with N duplicates in one transaction receives N× the legitimate bonus reward, draining the shared pool by `(N-1) × bonus_amount`.
- **Compounding effect:** Because the second `setPosition` resets the checkpoint to the pre-claim state, the attacker can repeat the attack in subsequent blocks, continuously re-draining the pool for the same accrual window.
- **Scope:** Any position owner. No privileged role required.

---

### Likelihood Explanation

The entry path is a standard signed `MsgClaimTierRewards` transaction submitted by any position owner. The `PositionIds` field is a caller-controlled repeated field with no uniqueness constraint enforced at the message-validation or handler level visible in the code. The attack requires only a single transaction with a duplicated position ID and is trivially constructable by any user with an active tiered-rewards position.

---

### Recommendation

Deduplicate `msg.PositionIds` before loading position states, or enforce uniqueness in `MsgClaimTierRewards.Validate()`. A simple approach:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest,
            "duplicate position id %d in claim request", posId)
    }
    seen[posId] = struct{}{}
    // ... existing load + validate logic
}
```

Alternatively, re-load each position state inside `claimRewardsAndUpdatesPositions` immediately before processing (after the previous entry's `setPosition` write) so that stale copies cannot be used.

---

### Proof of Concept

1. Alice owns position ID `42` with accrued bonus rewards of 100 BASECRO.
2. Alice submits `MsgClaimTierRewards{Owner: alice, PositionIds: [42, 42]}`.
3. Handler loads `pos42_copy1` and `pos42_copy2` from state (identical, both with `LastEventSeq = S`).
4. `claimRewardsAndUpdatesPositions` processes `pos42_copy1`: computes 100 BASECRO bonus, sends 100 BASECRO from `RewardsPoolName` to Alice, writes updated checkpoint (`LastEventSeq = S'`) to store.
5. `claimRewardsAndUpdatesPositions` processes `pos42_copy2`: still has `LastEventSeq = S`, re-reads the same events, computes 100 BASECRO bonus again, sends another 100 BASECRO from `RewardsPoolName` to Alice, overwrites store with stale checkpoint (`LastEventSeq = S`).
6. Alice receives 200 BASECRO instead of 100 BASECRO. The position's checkpoint is reset to `S`, so the same 100 BASECRO can be claimed again in the next block.
