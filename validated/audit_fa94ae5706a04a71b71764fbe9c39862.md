### Title
`ghost_heap` Not Updated in `new_substr()` SmallAtom Path — Heap-Limit Invariant Broken - (File: `src/allocator.rs`)

### Summary

`new_substr()` in `src/allocator.rs` fails to update `ghost_heap` when a `SmallAtom` is substrung into another `SmallAtom`. Every other allocation path that produces a `SmallAtom` — `new_atom()` and `new_small_number()` — updates **both** `ghost_atoms` and `ghost_heap`. The missing `ghost_heap` increment causes `heap_size()` to underreport true logical heap usage and makes the heap-limit guard progressively too permissive, breaking the consensus-critical invariant that heap limits are enforced identically regardless of the SmallAtom optimization.

---

### Finding Description

The `Allocator` maintains three "ghost" counters — `ghost_atoms`, `ghost_pairs`, and `ghost_heap` — to account for allocations that were optimized away (e.g., small integers stored inline in a `NodePtr` instead of on the heap). The code comment is explicit:

> "the ghost counters are pretend atoms/pairs, that were optimized out. We still account for them to not affect the limits of atoms and pairs. Those limits must stay the same for consensus purpose."

Every function that creates a `SmallAtom` is required to increment both `ghost_atoms` (by 1) and `ghost_heap` (by the byte-length of the value):

**`new_atom()` — correct:**
```rust
// src/allocator.rs lines 626-629
if let Some(ret) = fits_in_small_atom(v) {
    self.ghost_atoms += 1;
    self.ghost_heap += v.len();   // ← both counters updated
    Ok(self.mk_node(ObjectType::SmallAtom, ret as usize))
```

**`new_small_number()` — correct:**
```rust
// src/allocator.rs lines 647-648
self.ghost_atoms += 1;
self.ghost_heap += len;           // ← both counters updated
```

**`new_substr()` SmallAtom→SmallAtom branch — missing update:**
```rust
// src/allocator.rs lines 852-854
if let Some(new_val) = fits_in_small_atom(substr) {
    self.ghost_atoms += 1;
    // ← ghost_heap NOT updated; substr.len() bytes are silently lost
    Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
```

`substr` has length `end - start` (up to 4 bytes for a SmallAtom). Each call through this branch silently drops those bytes from `ghost_heap`.

---

### Impact Explanation

`ghost_heap` feeds directly into two consensus-critical mechanisms:

1. **`heap_size()` reporting** — `self.u8_vec.len() + self.ghost_heap` — returns a value lower than the true logical heap usage. External callers and analytics that rely on this value see a false picture.

2. **Heap-limit enforcement** — every subsequent allocation checks:
   ```rust
   // src/allocator.rs line 621
   if start as usize + self.ghost_heap + v.len() > self.heap_limit {
       return Err(EvalErr::OutOfMemory);
   }
   ```
   Because `ghost_heap` is undercounted by N bytes (one `substr` call on a 1–4 byte SmallAtom leaks up to 4 bytes), the guard allows N extra bytes of real heap to be allocated. A program that issues many `substr` calls on SmallAtoms can accumulate a deficit of millions of bytes, allowing it to exceed the consensus heap limit without triggering `OutOfMemory`.

A future corrected implementation would reject such programs, causing a consensus split between patched and unpatched nodes.

---

### Likelihood Explanation

The `substr` CLVM operator is a standard, documented operator. `op_substr()` in `src/more_ops.rs` calls `a.new_substr(a0, start as u32, end as u32)` directly and is reachable from any attacker-supplied CLVM program. A SmallAtom input (any 1–4 byte positive integer atom) is trivially constructable. The attacker simply needs to loop `substr` calls on a small atom to accumulate the `ghost_heap` deficit. No special privileges or configuration are required.

---

### Recommendation

In `new_substr()`, add `self.ghost_heap += substr.len();` in the `SmallAtom → SmallAtom` branch, mirroring the pattern in `new_atom()` and `new_small_number()`:

```rust
// src/allocator.rs ~line 852
if let Some(new_val) = fits_in_small_atom(substr) {
    self.ghost_atoms += 1;
    self.ghost_heap += substr.len(); // ← add this line
    Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
```
