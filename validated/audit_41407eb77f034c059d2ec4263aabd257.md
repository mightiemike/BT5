### Title
Hardcoded `price_unit: 0` (Wei) in P2P Receipt Serialization Always Emits Wrong Fee Currency for V3 Transactions — (File: `crates/apollo_protobuf/src/converters/receipt.rs`)

### Summary

Every protobuf receipt produced by the `From` trait implementations for transaction outputs hard-codes `price_unit: 0` (`PriceUnit::Wei`). V3 transactions (`InvokeV3`, `DeclareV3`, `DeployAccountV3`) pay fees in STRK (FRI, value `1`). The correct value is never written into the serialized receipt automatically; a separate, manually-invoked function `set_price_unit_based_on_transaction` must be called after the fact. The `From` implementations themselves carry an explicit comment acknowledging the defect. Any code path that serializes a receipt without calling the fixup emits a canonically wrong `price_unit` field over the P2P sync wire, causing syncing peers to store and re-serve receipts with the wrong fee-currency denomination.

### Finding Description

`create_proto_receipt_common_from_txn_output_fields` is the single shared helper that builds the `protobuf::receipt::Common` struct for every transaction type:

```rust

protobuf::receipt::Common {
    actual_fee: Some(actual_fee),
    price_unit: 0,          // ← always Wei, regardless of tx version
    messages_sent,
    execution_resources: Some(execution_resources),
    revert_reason,
}
``` [1](#0-0) 

Every `From` implementation for `InvokeTransactionOutput`, `DeclareTransactionOutput`, `DeployAccountTransactionOutput`, etc. calls this helper and carries the same comment:

```rust

    /// It can be fixed by calling set_price_unit_based_on_transaction

The fixup lives in a completely separate function in a different file:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  line 1085-1115
pub fn set_price_unit_based_on_transaction(
    receipt: &mut protobuf::Receipt,
    transaction: &protobuf::TransactionInBlock,
) {
    let price_unit = match &transaction.txn {
        Some(Txn::InvokeV3(_))        => protobuf::PriceUnit::Fri,
        Some(Txn::DeclareV3(_))       => protobuf::PriceUnit::Fri,
        Some(Txn::DeployAccountV3(_)) => protobuf::PriceUnit::Fri,
        Some(Txn::InvokeV1(_))        => protobuf::PriceUnit::Wei,
        …
    };
    …
    common.price_unit = price_unit.into();
}
``` [3](#0-2) 

The protobuf schema defines `Wei = 0` and `Fri = 1`:

```proto
enum PriceUnit {
    Wei = 0;
    Fri = 1;
}
``` [4](#0-3) 

Because `price_unit: 0` is the protobuf default, any serialization path that omits the fixup call silently emits `Wei` for every V3 receipt. The deserialization path (`parse_common_receipt_fields`) reads `actual_fee` but discards `price_unit`, so the wrong value is stored verbatim in whatever the syncing peer persists and later re-serves. [5](#0-4) 

### Impact Explanation

A syncing node that receives a `TransactionWithReceipt` for a V3 transaction over the P2P protocol will store a receipt whose `price_unit` field says `Wei` instead of `Fri`. When that node later answers RPC calls (`starknet_getTransactionReceipt`, `starknet_getBlockWithReceipts`, etc.) it returns an authoritative-looking receipt with the wrong fee denomination. A client that trusts the `price_unit` field to interpret `actual_fee` will misread the fee amount by a factor of roughly 10¹⁸ (the ETH/STRK magnitude difference). This matches the **High** impact tier: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

### Likelihood Explanation

The defect is structural: the `From` trait is the natural, idiomatic conversion path. Any caller that writes `receipt.into()` or `From::from(output)` without subsequently calling `set_price_unit_based_on_transaction` produces a wrong receipt. The explicit "isn't correct" comments confirm the developers know the default is wrong, but the two-step pattern is easy to omit in new code paths or during refactors. The P2P sync layer sends `TransactionWithReceipt` bundles; if any assembly point forgets the fixup, every V3 receipt it propagates carries the wrong denomination.

### Recommendation

1. Remove `price_unit: 0` from `create_proto_receipt_common_from_txn_output_fields` and instead require the fee type to be passed as a parameter, so the correct value is set at construction time.
2. Alternatively, encode the fee type into the transaction output types (`InvokeTransactionOutput`, etc.) so the `From` implementation can derive `price_unit` without needing a separate transaction object.
3. Delete `set_price_unit_based_on_transaction` once the root cause is fixed, to prevent future callers from relying on the two-step pattern.

### Proof of Concept

```
1. User submits an InvokeV3 transaction; it executes and pays 5000 FRI.
2. The sequencer builds a TransactionWithReceipt for P2P sync:
       let proto_receipt: protobuf::receipt::Invoke =
           InvokeTransactionOutput { actual_fee: Fee(5000), … }.into();
   → proto_receipt.common.price_unit == 0  (Wei, WRONG; should be 1 = Fri)
3. set_price_unit_based_on_transaction is NOT called (omitted in this path).
4. The receipt is sent to a syncing peer over P2P.
5. The syncing peer stores price_unit = Wei for this receipt.
6. A client calls starknet_getTransactionReceipt; the peer returns:
       { "actual_fee": { "amount": "0x1388", "unit": "WEI" } }
   The client interprets 5000 WEI ≈ $0.000000015 instead of 5000 FRI ≈ $0.0015.
```

pub fn set_price_unit_based_on_transaction(
    receipt: &mut protobuf::Receipt,
    transaction: &protobuf::TransactionInBlock,
) {
    let price_unit = match &transaction.txn {
        Some(protobuf::transaction_in_block::Txn::DeclareV1(_)) => protobuf::PriceUnit::Wei,
        Some(protobuf::transaction_in_block::Txn::DeclareV2(_)) => protobuf::PriceUnit::Wei,
        Some(protobuf::transaction_in_block::Txn::DeclareV3(_)) => protobuf::PriceUnit::Fri,
        Some(protobuf::transaction_in_block::Txn::Deploy(_)) => protobuf::PriceUnit::Wei,
        Some(protobuf::transaction_in_block::Txn::DeployAccountV1(_)) => protobuf::PriceUnit::Wei,
        Some(protobuf::transaction_in_block::Txn::DeployAccountV3(_)) => protobuf::PriceUnit::Fri,
        Some(protobuf::transaction_in_block::Txn::InvokeV1(_)) => protobuf::PriceUnit::Wei,
        Some(protobuf::transaction_in_block::Txn::InvokeV3(_)) => protobuf::PriceUnit::Fri,
        Some(protobuf::transaction_in_block::Txn::L1Handler(_)) => protobuf::PriceUnit::Wei,
        _ => return,
    };
    let Some(ref mut receipt_type) = receipt.r#type else {
        return;
    };

    let common = match receipt_type {
        protobuf::receipt::Type::Invoke(invoke) => invoke.common.as_mut(),
        protobuf::receipt::Type::L1Handler(l1_handler) => l1_handler.common.as_mut(),
        protobuf::receipt::Type::Declare(declare) => declare.common.as_mut(),
        protobuf::receipt::Type::DeprecatedDeploy(deploy) => deploy.common.as_mut(),
        protobuf::receipt::Type::DeployAccount(deploy_account) => deploy_account.common.as_mut(),
    };

    if let Some(common) = common {
        common.price_unit = price_unit.into();
    }
