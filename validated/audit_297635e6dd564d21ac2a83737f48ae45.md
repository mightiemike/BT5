### Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. The extension therefore checks whether the **router** is allowlisted, not whether the **user** is allowlisted. A pool admin who allowlists the router to enable router-mediated swaps inadvertently opens the pool to every user on-chain, completely defeating the curation policy.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as `sender` to every configured extension: [2](#0-1) 

**Step 2 — Extension checks `sender`, which is the router when the router is the caller.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the value received from the pool: [3](#0-2) 

**Step 3 — The router calls `pool.swap` directly, substituting its own address.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` with no mechanism to forward the original `msg.sender`: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. In every case the pool sees `msg.sender = router`, so the extension evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][user]`. [5](#0-4) 

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router (the natural step to enable router-mediated swaps) grants every on-chain address the ability to trade on that pool. The allowlist is completely bypassed: any user who calls `exactInputSingle` or any other router entry-point will pass the extension check because the router itself is allowlisted. Unauthorized users can extract value from LPs who expected only trusted counterparties, constituting a direct loss of LP principal and a broken core pool invariant.

Conversely, if the admin does **not** allowlist the router, individually allowlisted users cannot swap through the router at all, breaking the primary swap interface for legitimate participants.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the standard, documented periphery swap interface. Any user who discovers the allowlist can trivially route through it. No privileged access, no special setup, and no malicious pool configuration is required — only a normal router call. The bypass is unconditional once the router is allowlisted.

---

### Recommendation

The pool must forward the originating user's identity rather than its own `msg.sender`. Two complementary fixes:

1. **Router-side**: Store the originating `msg.sender` in transient storage at entry (similar to how the payer is already stored) and expose it so the pool can pass it as `sender` to extensions. This requires a protocol-level convention for trusted periphery contracts.

2. **Extension-side (simpler, self-contained)**: Change `SwapAllowlistExtension.beforeSwap` to accept an explicit `recipient` or `originator` field from `extensionData`, or gate on `recipient` rather than `sender` when the pool is known to be called through a router. However, this shifts trust to the caller-supplied payload.

The cleanest fix is for the router to pass the originating user as the `recipient` of the swap **and** for the extension to check `recipient` when `sender` is a known router — but this requires a registry of trusted routers. Alternatively, the pool interface could be extended to carry an explicit `originator` field that the router populates from its own `msg.sender`.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured on beforeSwap.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (to enable router-mediated swaps for allowlisted users).
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  - Attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
  - Router calls pool.swap(recipient, ...) — pool sees msg.sender = router.
  - Pool calls _beforeSwap(router, recipient, ...).
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true.
  - Swap executes for the attacker despite the attacker never being allowlisted.

Result:
  - Attacker trades on a curated pool intended only for trusted counterparties.
  - LP funds are exposed to unauthorized flow; curation policy is void.
