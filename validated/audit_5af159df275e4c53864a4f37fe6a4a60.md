### Title
Consensus `handle_vote` Accepts Arbitrary `voter` Address Without Signature Verification, Allowing Any Peer to Forge Votes for Any Validator - (File: crates/apollo_consensus/src/single_height_consensus.rs)

### Summary
`SingleHeightConsensus::handle_vote` accepts the `voter` field from an incoming `Vote` message as a trusted identity without verifying the cryptographic signature. Any network peer can craft a `Vote` with `voter` set to any committee member's address and a default/empty signature, and the vote will be counted toward quorum. This is the direct sequencer analog of `voteExtraordinary(account_, ...)` accepting an arbitrary account address instead of binding to `msg.sender`.

### Finding Description
The `Vote` struct carries both a `voter: ContractAddress` field (the claimed identity) and a `signature: RawSignature` field (the proof of identity). When a vote arrives from the network, `handle_vote` performs two checks: height equality and committee membership of `vote.voter`. It then explicitly skips signature verification with a `// TODO(Asmaa): verify the signature` comment and proceeds to count the vote's weight toward quorum.

```rust
pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
    // TODO(Asmaa): verify the signature          // ← signature never checked
    ...
    if !self.committee.members().iter().any(|s| s.address == vote.voter) {
        debug!("Ignoring vote from non validator: vote={:?}", vote);
        return VecDeque::new();
    }
    // vote.voter is now used to count weight toward quorum
    ...
    self.state_machine.handle_event(sm_vote)
}
```

Outgoing self-votes are also created with `signature: RawSignature::default()` (`// TODO(Asmaa): sign the vote`), confirming the signing half is also unimplemented.

The codebase itself acknowledges the consequence in `cache_future_vote`:

> "Since vote signatures are not yet verified, a peer can forge votes with arbitrary voter addresses; without this cap that would grow `future_votes` without bound and exhaust memory."

The `voter` field is accepted verbatim from the protobuf wire message:

```rust
impl TryFrom<protobuf::Vote> for Vote {
    fn try_from(value: protobuf::Vote) -> Result<Self, Self::Error> {
        ...
        let voter = value.voter.ok_or(missing("voter"))?.try_into()?;
        let signature = value.signature.map(...).unwrap_or_default();
        Ok(Vote { ..., voter, signature })
    }
}
```

The `voter` address is then used as the key in `prevotes`/`precommits` maps and its associated weight is summed toward the quorum threshold — with no binding between the claimed `voter` address and the actual network sender.

### Impact Explanation
A single malicious peer connected to the consensus P2P network can:
1. Enumerate all committee members (public information from the committee provider).
2. For each validator, craft a `Vote { voter: validator.address, signature: RawSignature::default(), proposal_commitment: attacker_chosen_block, ... }`.
3. Send these forged prevotes and precommits to a target node.
4. The target node's `handle_vote` passes the committee-membership check and counts each forged vote at full validator weight.
5. Once forged votes accumulate to quorum, `upon_decision` fires and `DecisionReached` is emitted for the attacker-chosen block.

This drives the node to commit a block the attacker controls, producing wrong state, wrong block hash, wrong receipts, and wrong L1 messages — matching the Critical impact scope: *"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."*

### Likelihood Explanation
The trigger requires only a peer connection to the consensus broadcast channel. The code comment explicitly states any peer can forge votes with arbitrary voter addresses. No stake, no key material, and no privileged access are required beyond network reachability. The `RawSignature::default()` (all-zero) value is accepted without error.

### Recommendation
- **Short term**: In `handle_vote`, after the committee-membership check, verify `vote.signature` against the public key associated with `vote.voter` in the committee. The `apollo_signature_manager` crate already provides `verify_precommit_vote_signature`; extend it to cover prevotes and integrate it here before the vote is forwarded to the state machine.
- **Long term**: Remove the `// TODO(Asmaa): sign the vote` placeholder in `make_self_vote` and implement actual ECDSA signing so that outgoing votes carry valid signatures, enabling peers to verify them symmetrically.
