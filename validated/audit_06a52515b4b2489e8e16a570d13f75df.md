Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of end user, allowing allowlist bypass via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates pool swaps by checking `sender`, which resolves to `msg.sender` of `pool.swap()` — the router address when swaps are routed through `MetricOmmSimpleRouter`. If the pool admin allowlists the router (the natural configuration to enable standard user-facing swaps), every unprivileged user can bypass the allowlist entirely. This is structurally inconsistent with `DepositAllowlistExtension`, which correctly checks `owner` (the actual position beneficiary) and works correctly through the router.

## Finding Description
`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

`sender` is populated by `MetricOmmPool.swap()`, which passes `msg.sender` of the `pool.swap()` call: [2](#0-1) 

`ExtensionCalling._beforeSwap` then forwards this value as the first argument to the extension: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap(params.recipient, ...)`, making `msg.sender` inside the pool the **router address**: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` checks `owner` (second argument — the actual position beneficiary), not `sender`: [5](#0-4) 

**Exploit path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to specific counterparties.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable standard router-mediated swaps.
3. Any unprivileged, non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. The router calls `pool.swap(recipient=user, ...)` with `msg.sender` = router.
5. The extension evaluates `allowedSwapper[pool][router]` = `true` → swap proceeds.
6. The non-allowlisted user has fully bypassed the allowlist.

The pool admin faces an impossible choice: either allowlist the router (allowing all users to bypass the allowlist) or do not allowlist the router (making the router unusable for all users, including allowlisted ones).

## Impact Explanation
This is an admin-boundary break: an unprivileged path (router) bypasses a configured access control gate. The swap allowlist is rendered completely ineffective whenever the router is allowlisted. Any user — regardless of allowlist status — can execute swaps in a pool intended to be restricted to specific counterparties. In pools designed for institutional or permissioned trading (e.g., with specific price agreements or regulatory requirements), this allows unauthorized parties to extract value at pool-set prices, directly harming LP principal and disrupting the pool's intended economic model.

## Likelihood Explanation
The trigger condition — allowlisting the router — is the natural and expected configuration for any pool that wants to support standard user-facing swaps. A pool admin who configures `SwapAllowlistExtension` to restrict swaps to specific addresses will almost certainly also allowlist the router to enable normal operation, unaware that doing so voids the allowlist entirely. The inconsistency with `DepositAllowlistExtension` (which works correctly through the router) reinforces the expectation that the swap allowlist should behave the same way.

## Recommendation
Replace the `sender` check with a check on the actual end user. The `recipient` argument (second parameter, currently unnamed/ignored) is the address that receives swap output and is set to the actual user by the router. The fix should mirror the pattern used by `DepositAllowlistExtension`, which checks `owner` (the actual beneficiary) rather than the direct caller:

```solidity
// Proposed fix: check recipient (actual end user) instead of sender
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

## Proof of Concept
```solidity
// Assume: pool configured with SwapAllowlistExtension
// Pool admin setup:
swapAllowlist.setAllowedToSwap(pool, address(router), true);  // enable router
// attacker (not individually allowlisted) calls:
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    tokenIn: token0,
    tokenOut: token1,
    pool: pool,
    recipient: attacker,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    extensionData: ""
}));
// Result: swap succeeds — attacker bypassed the allowlist
// Extension checked allowedSwapper[pool][router] == true, not allowedSwapper[pool][attacker]
