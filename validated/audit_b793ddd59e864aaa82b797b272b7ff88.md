### Title
`op_bls_verify` Signature Verification Invariant Bypassed When Zero (pk, msg) Pairs Are Supplied — (`File: src/bls_ops.rs`)

### Summary

`op_bls_verify` in `src/bls_ops.rs` accepts a G2 signature followed by a variable-length flat list of `(pk, msg)` pairs. When the pair list is empty (zero items), the operator calls `aggregate_verify` with an empty `items` vector. BLS aggregate verify over an empty set is mathematically defined to return `true` for the G2 identity element. The operator therefore succeeds — returning `nil` (success) — without performing any cryptographic check. Any CLVM coin puzzle that derives the `(pk, msg)` pair list from attacker-controlled solution bytes can be bypassed by supplying the G2 identity point as the signature and an empty pair list.

### Finding Description

In `src/bls_ops.rs`, `op_bls_verify` reads the first argument as the signature, then iterates over the remaining arguments as `(pk, msg)` pairs:

```rust
// src/bls_ops.rs lines 381-401
let mut items = Vec::<(PublicKey, Atom)>::new();
while !nilp(a, args) {
    let pk = a.g1(first(a, args)?)?;
    args = rest(a, args)?;
    let msg = atom(a, first(a, args)?, "bls_verify message")?;
    args = rest(a, args)?;
    cost += BLS_PAIRING_COST_PER_ARG;
    ...
    items.push((pk, msg));
}

if !aggregate_verify(&signature, items) {
    Err(EvalErr::BLSVerifyFailed(input))?
} else {
    Ok(Reduction(cost, a.nil()))
}
```

When the argument list after the signature is `nil`, the `while` loop body never executes, `items` remains empty, and `aggregate_verify(&signature, vec![])` is called. The BLS aggregate verify specification defines the empty-set case as returning `true` when the signature equals the group identity (G2 infinity point, compressed `0xc000…0000`). The operator therefore returns `Ok(Reduction(cost, a.nil()))` — success — without verifying any signature over any message.

This is confirmed by the existing test vector in `op-tests/test-bls-ops.txt` line 351:

```
; identity (no messages signed)
bls_verify 0xc000...0000 => 0 | 3000000
```

The invariant that `bls_verify` must verify at least one `(pk, msg)` pair is never enforced. An attacker who controls the CLVM argument list (e.g., via the solution to a coin puzzle) can supply `(bls_verify <G2-identity>)` with zero pairs and the operator succeeds unconditionally.

The same structural issue exists in `op_bls_pairing_identity` (lines 332–358): when called with zero arguments, `items` is empty, `aggregate_pairing([])` returns `true`, and the operator succeeds. The test vector at line 305 confirms: `bls_pairing_identity => 0 | 3000000`.

### Impact Explanation

Any CLVM coin puzzle that uses `bls_verify` to guard a spend path, where the `(pk, msg)` pairs are derived from the solution (attacker-controlled bytes), can be bypassed. The attacker supplies the G2 identity point as the signature and provides no `(pk, msg)` pairs. The operator returns success, the guard is defeated, and the coin can be spent without a valid BLS signature. This is a consensus-level integrity failure: nodes running `clvmr` will accept the spend as valid, enabling unauthorized movement of Chia coins. The impact is equivalent to the re-nft escrow drain: a check that must always run is silently skipped when the item count is zero, and the downstream state change (coin spend) proceeds as if verification passed.

### Likelihood Explanation

The attack requires a coin puzzle that (a) uses `bls_verify` and (b) allows the caller to supply the `(pk, msg)` pair list from the solution rather than hardcoding it. This pattern is realistic for puzzles that implement delegated or aggregated signing. The attacker needs only to craft a solution with the G2 identity point as the signature and an empty pair list — both are trivially constructable from public knowledge. No privileged access, key material, or side-channel is required.

### Recommendation

Add a minimum-argument guard at the start of `op_bls_verify` that rejects calls where the `(pk, msg)` pair list is empty:

```rust
// after extracting `signature` and advancing `args` past it:
if nilp(a, args) {
    return Err(EvalErr::InvalidOpArg(
        input,
        "bls_verify requires at least one (pk, msg) pair".to_string(),
    ));
}
```

Apply the same guard to `op_bls_pairing_identity` to reject zero-argument calls. Update the test vectors in `op-tests/test-bls-ops.txt` to expect `FAIL` for the zero-argument cases currently documented as "identity."

pub fn op_bls_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = BLS_PAIRING_BASE_COST;
    check_cost(cost, max_cost)?;

    let mut args = input;

    // the first argument is the signature
    let signature = a.g2(first(a, args)?)?;

    // followed by a variable number of (G1, msg)-pairs (as a flat list)
    args = rest(a, args)?;

    let mut items = Vec::<(PublicKey, Atom)>::new();
    while !nilp(a, args) {
        let pk = a.g1(first(a, args)?)?;
        args = rest(a, args)?;
        let msg = atom(a, first(a, args)?, "bls_verify message")?;
        args = rest(a, args)?;

        cost += BLS_PAIRING_COST_PER_ARG;
        cost += msg.as_ref().len() as Cost * BLS_MAP_TO_G2_COST_PER_BYTE;
        cost += DST_G2.len() as Cost * BLS_MAP_TO_G2_COST_PER_DST_BYTE;
        check_cost(cost, max_cost)?;

        items.push((pk, msg));
    }

    if !aggregate_verify(&signature, items) {
        Err(EvalErr::BLSVerifyFailed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
}
