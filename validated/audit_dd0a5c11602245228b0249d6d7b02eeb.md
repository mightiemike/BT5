### Title
Silent Drop of AddMod/MulMod/RangeCheck96 Builtin Counts in P2P Receipt Protobuf Serialization — (`crates/apollo_protobuf/src/converters/receipt.rs`)

### Summary

`From<HashMap<Builtin, u64>> for ProtobufBuiltinCounter` omits `AddMod`, `MulMod`, and `RangeCheck96` from the protobuf wire encoding. Any node that syncs blocks via the p2p protocol stores receipts with those three builtin counts silently zeroed, producing wrong `execution_resources` in every RPC receipt response for transactions that use those builtins.

### Finding Description

`starknet_api::execution_resources::Builtin` has 11 variants, including `AddMod`, `MulMod`, and `RangeCheck96`. [1](#0-0) 

The protobuf `BuiltinCounter` message only has fields for 8 builtins and carries an explicit `// TODO(alonl): add the missing builtins` comment: [2](#0-1) 

The Rust serializer `From<HashMap<Builtin, u64>> for ProtobufBuiltinCounter` maps only 7 builtins and hardcodes `output: 0`, with no entry for `AddMod`, `MulMod`, or `RangeCheck96`: [3](#0-2) 

The deserializer `TryFrom<ProtobufBuiltinCounter> for HashMap<Builtin, u64>` likewise never inserts those three keys: [4](#0-3) 

The p2p sync client receives `FullTransaction` (including `transaction_output`) via this protobuf path and writes it directly to storage: [5](#0-4) 

The blockifier's native execution path does produce non-zero `AddMod`, `MulMod`, and `RangeCheck96` counts: [6](#0-5) 

The storage serializer correctly encodes all three builtins (discriminants 8, 9, 10), so the sequencer's own DB is correct — only the p2p-synced copy is wrong: [7](#0-6) 

The developers have already acknowledged this in the p2p sync test, where the `transaction_outputs` assertion is commented out with the note: [8](#0-7) 

### Impact Explanation

Every full node that syncs blocks via p2p will store receipts with `AddMod`, `MulMod`, and `RangeCheck96` counts set to 0 instead of their true values. Those nodes then serve wrong `execution_resources` through `starknet_getTransactionReceipt` — an authoritative-looking wrong value from a public RPC endpoint. This matches the **High** impact criterion: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

The sequencer's own block hash computation uses its local DB (which is correct), so the block hash itself is not corrupted. However, any downstream system that recomputes the receipt commitment from p2p-synced data (e.g., a verifier or a node that re-derives the block hash from body data) would produce a divergent value.

### Likelihood Explanation

Any transaction executed by a Cairo contract that uses `add_mod`, `mul_mod`, or `range_check96` instructions triggers this. These builtins are active in all versioned constants from v0.13.1 onward and are used by Sierra-compiled contracts performing modular arithmetic. The path is fully automatic — no attacker action is required; the bug fires for every such transaction synced over p2p.

### Recommendation

1. Add `add_mod`, `mul_mod`, and `range_check96` fields to the `BuiltinCounter` protobuf message in `receipt.proto` (resolving the existing TODO).
2. Add the three missing entries to both `From<HashMap<Builtin, u64>> for ProtobufBuiltinCounter` and `TryFrom<ProtobufBuiltinCounter> for HashMap<Builtin, u64>` in `crates/apollo_protobuf/src/converters/receipt.rs`.
3. Uncomment the `actual_transaction_outputs` assertion in the p2p sync test once the fix is in place.

### Proof of Concept

```rust
use std::collections::HashMap;
use starknet_api::execution_resources::Builtin;

// Simulate what the blockifier produces for a contract using add_mod/mul_mod/range_check96
let mut original: HashMap<Builtin, u64> = HashMap::new();
original.insert(Builtin::AddMod, 42);
original.insert(Builtin::MulMod, 17);
original.insert(Builtin::RangeCheck96, 99);
original.insert(Builtin::RangeCheck, 5);

// Serialize to protobuf (p2p send path)
type ProtobufBuiltinCounter = protobuf::receipt::execution_resources::BuiltinCounter;
let proto = ProtobufBuiltinCounter::from(original.clone());

// Deserialize (p2p receive path)
let recovered = HashMap::<Builtin, u64>::try_from(proto).unwrap();

// These assertions FAIL — counts are silently zeroed
assert_eq!(recovered.get(&Builtin::AddMod).copied().unwrap_or(0), 42);   // got 0
assert_eq!(recovered.get(&Builtin::MulMod).copied().unwrap_or(0), 17);   // got 0
assert_eq!(recovered.get(&Builtin::RangeCheck96).copied().unwrap_or(0), 99); // got 0
// Only RangeCheck survives
assert_eq!(recovered.get(&Builtin::RangeCheck).copied().unwrap_or(0), 5); // ok
```
