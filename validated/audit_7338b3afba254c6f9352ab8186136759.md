The exploit hypothesis is **valid**. Here is the complete analysis:

---

### Title
Repeated Margin Drain via Distinct Digests on Existing Isolated Subaccount — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`createIsolatedSubaccount` guards against exact digest replay but not against multiple distinct digests targeting the same `productId`. When an isolated subaccount already exists for a `productId`, the function reuses it and still unconditionally executes the margin transfer for every new digest, allowing an attacker to drain the parent's quote balance across N transactions.

---

### Finding Description

The function has two independent code paths that are incorrectly coupled:

**Path A — Digest replay guard** (lines 1009–1011): [1](#0-0) 

This early-return only fires when the **exact same** digest is resubmitted. A new order with a different `nonce`, `priceX18`, or `amount` produces a different `digest` and passes straight through.

**Path B — Existing-subaccount reuse** (lines 1025–1038): [2](#0-1) 

When an isolated subaccount already exists for `txn.productId`, `newIsolatedSubaccount` is set to it and the loop breaks. Crucially, execution does **not** stop here.

**Path C — Unconditional margin transfer** (lines 1072–1087): [3](#0-2) 

`digestToSubaccount[digest] = newIsolatedSubaccount` is recorded and the margin transfer executes for **every** new digest — whether the isolated subaccount was freshly created or merely reused from Path B. There is no guard of the form `if (newIsolatedSubaccount was just created)` around the transfer block.

The result: submitting N orders for the same `productId` with distinct nonces produces N distinct digests, each of which:
1. Passes the digest-replay guard (different digest).
2. Finds and reuses the existing isolated subaccount.
3. Transfers `margin` from the parent to that isolated subaccount.

The parent's quote balance decreases by `N × margin`.

---

### Impact Explanation

- **Asset impact**: Parent subaccount's `QUOTE_PRODUCT_ID` balance is decremented by `margin` on every call. With `margin = 1000e6` and N = 5 calls, the parent loses `5000e6` quote units while the isolated subaccount accumulates them — a direct, measurable asset drain.
- **Invariant broken**: The protocol invariant that "margin allocated to an isolated position is bounded by one transfer per position per order" is violated. The same isolated subaccount can receive unbounded margin from a single parent.
- **Scope**: Critical — unauthorized repeated asset transfer from parent to isolated subaccount.

---

### Likelihood Explanation

The attacker only needs to be the owner of the parent subaccount (or a linked signer). They submit N valid signed orders for the same `productId` with different nonces through the normal `CreateIsolatedSubaccount` transaction path via the sequencer. No admin access, no oracle manipulation, no external dependency is required. [4](#0-3) 

---

### Recommendation

Wrap the margin transfer block so it only executes when a **new** isolated subaccount was created, not when an existing one was reused:

```solidity
// Only transfer margin when a new isolated subaccount is created
bool isNew = (newIsolatedSubaccount == bytes32(0)); // check BEFORE the creation block

// ... existing creation logic ...

digestToSubaccount[digest] = newIsolatedSubaccount;

if (isNew) {   // <-- add this guard
    int128 margin = int128(_isolatedMargin(txn.order.appendix));
    if (margin > 0) {
        digestToMargin[digest] = margin;
        spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
        spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
    }
}
```

Alternatively, record the digest-to-margin mapping before the subaccount lookup and skip the transfer if `digestToMargin[digest]` is already set for the reused subaccount.

### Proof of Concept

```solidity
// Attacker controls parentSubaccount with 5000e6 USDC quote balance
for (uint64 i = 0; i < 5; i++) {
    IEndpoint.Order memory order = IEndpoint.Order({
        sender: parentSubaccount,
        priceX18: 1e18,
        amount: 1e18,
        expiration: block.timestamp + 1 days,
        nonce: i,           // different nonce → different digest
        appendix: encodeIsolatedAppendix(1000e6)  // margin = 1000e6
    });
    endpoint.submitSlowModeTransaction(
        abi.encodePacked(
            uint8(IEndpoint.TransactionType.CreateIsolatedSubaccount),
            abi.encode(IEndpoint.CreateIsolatedSubaccount({
                order: order,
                productId: TARGET_PRODUCT_ID,
                signature: sign(order)
            }))
        )
    );
}
// After 5 txns: parent quote balance = 0, isolated subaccount balance = 5000e6
assertEq(spotEngine.getBalance(QUOTE_PRODUCT_ID, parentSubaccount), 0);
assertEq(spotEngine.getBalance(QUOTE_PRODUCT_ID, isolatedSubaccount), 5000e6);
```

Each iteration produces a distinct digest, bypasses the replay guard, reuses the same isolated subaccount, and transfers 1000e6 from parent to isolated — draining the parent by 5000e6 total. [5](#0-4) [6](#0-5)
