### Title
Unchecked `u64` Multiplication Overflow in `op_unknown` Cost Calculation Enables Cost Undercharge - (`File: src/more_ops.rs`)

### Summary

In `src/more_ops.rs`, the `op_unknown` function computes the final cost of an unknown opcode by multiplying a base cost by `(cost_multiplier + 1)` at line 261. This multiplication is performed on a `u64` (`Cost`) without overflow protection. Because `cost` can be up to `max_cost` (~11 billion in Chia's block limit) and `cost_multiplier + 1` can be up to `u32::MAX + 1 = 4,294,967,296`, the product can exceed `u64::MAX`, wrapping silently in release builds. The wrapped result can be small enough to pass the subsequent `u32::MAX` guard, causing the operation to return with a near-zero cost — a cost undercharge that bypasses the execution cost limit.

### Finding Description

`op_unknown` handles any opcode not recognized by the interpreter. Its cost is determined by two fields encoded in the opcode bytes:

- `cost_function` (2 bits): selects the cost model (constant, arith-like, mul-like, concat-like)
- `cost_multiplier` (up to 4 bytes): a `u32` value cast to `u64`, up to `0xFFFFFFFF` [1](#0-0) 

After computing the base cost via the selected cost model (with `check_cost` guards inside the loop), the function applies a final guard and then multiplies:

```rust
check_cost(cost, max_cost)?;      // line 260: ensures cost ≤ max_cost
cost *= cost_multiplier + 1;      // line 261: UNCHECKED multiplication
if cost > u32::MAX as u64 {       // line 262: post-multiplication guard
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
``` [2](#0-1) 

`Cost` is `u64`: [3](#0-2) 

The `[profile.release]` section in `Cargo.toml` does not enable `overflow-checks`: [4](#0-3) 

So in release mode, Rust's default wrapping semantics apply to the `*=` at line 261.

**Overflow condition:**

- `cost_multiplier + 1` max = `0xFFFFFFFF + 1 = 2^32 = 4,294,967,296`
- Overflow threshold for `cost`: `u64::MAX / 2^32 ≈ 4,294,967,295`
- With `max_cost = 11,000,000,000` (Chia block limit), `cost` can reach up to 11 billion after the inner loop, which is well above the overflow threshold of ~4.3 billion

**Concrete wrap-to-zero scenario:**

With `cost_multiplier = 0xFFFFFFFF` (so `cost_multiplier + 1 = 2^32`) and `cost` being any exact multiple of `2^32`:

```
cost * 2^32  mod  2^64  =  (cost mod 2^32) * 2^32  =  0 * 2^32  =  0
```

The wrapped result is `0`, which satisfies `0 > u32::MAX` → false, so the function returns `Ok(Reduction(0, nil))` — an operation that should cost billions returns with cost **zero**.

For `cost_function = 1` (arith-like), the base cost formula is:

```
cost = ARITH_BASE_COST + n * ARITH_COST_PER_ARG + byte_count * ARITH_COST_PER_BYTE
     = 99 + n * 320 + byte_count * 3
``` [5](#0-4) 

The attacker can tune `n` (number of arguments) and `byte_count` (total argument bytes) to hit a multiple of `2^32`. With `max_cost = 11 billion`, the inner cost can reach up to ~11 billion, giving a wide range of multiples of `2^32` (e.g., `4,294,967,296`, `8,589,934,592`) to target.

### Impact Explanation

**Cost undercharge / cost bypass:** An attacker who crafts an unknown opcode with `cost_multiplier = 0xFFFFFFFF` and `cost_function ∈ {1,2,3}`, and provides arguments that push the inner cost to a multiple of `2^32`, causes `op_unknown` to return with cost = 0 (or another small wrapped value ≤ `u32::MAX`). The CLVM interpreter adds this returned cost to the running total. The attacker can therefore execute an operation that should consume billions of cost units for free, bypassing the block cost limit.

**Consensus divergence:** In debug builds, Rust panics on integer overflow. In release builds, it wraps silently. A node running a debug build would crash (or reject the program with a panic) when processing this crafted program, while a release-mode node would accept it with cost = 0. This is a consensus-breaking divergence between nodes.

### Likelihood Explanation

The trigger requires:
1. An unknown opcode with specific byte encoding (`cost_multiplier = 0xFFFFFFFF`, `cost_function ≠ 0`) — fully attacker-controlled via CLVM program bytes.
2. Arguments that push the inner cost to a multiple of `2^32` — achievable by tuning the number and size of atom arguments.
3. A caller that passes a `max_cost` above ~4.3 billion — standard for Chia full-node block validation (11 billion limit).

All three conditions are under attacker control or are standard deployment conditions. The opcode bytes are directly embedded in the serialized CLVM program, making this straightforwardly reachable from any externally submitted transaction or block.

### Recommendation

Replace the unchecked multiplication at line 261 with a checked multiply that returns an error on overflow:

```rust
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

This mirrors the existing pattern of using `check_cost` to guard against excessive cost values, and eliminates the wrapping overflow before the `u32::MAX` guard can fire.

### Proof of Concept

**Crafted opcode bytes:** `[0xFF, 0xFF, 0xFF, 0xFF, 0x40]`
- Bytes `[0..4]` = `[0xFF, 0xFF, 0xFF, 0xFF]` → `cost_multiplier = 0xFFFFFFFF` via `u32_from_u8`
- Last byte `0x40` = `0b01000000` → `cost_function = (0x40 & 0xC0) >> 6 = 1` (arith-like) [1](#0-0) 

**Arguments:** Provide `n = 13,421,772` atom arguments of 1 byte each:
- `cost = 99 + 13,421,772 * 320 + 13,421,772 * 3 = 99 + 4,294,966,640 + 40,265,316`
- Adjust `n` and `byte_count` to reach the nearest multiple of `2^32 = 4,294,967,296`

**Execution path:**
1. `op_unknown` is called with the crafted opcode and argument list
2. Inner loop (cost_function=1) accumulates cost to ~`4,294,967,296`
3. `check_cost(cost, max_cost)?` at line 260 passes (cost ≤ 11 billion)
4. `cost *= cost_multiplier + 1` → `4,294,967,296 * 4,294,967,296 = 2^64 ≡ 0 (mod 2^64)`
5. `if 0 > u32::MAX` → false
6. Returns `Ok(Reduction(0, nil))` — zero cost charged for a multi-billion-cost operation [2](#0-1)
