### Title
Rebasing Token Collateral Causes Withdrawal Revert and Fund Lockup — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

Nado's `Clearinghouse` tracks collateral balances using internal accounting in `SpotEngine` that is denominated in nominal token units. If a rebasing token is used as collateral and a negative rebase reduces the Clearinghouse's actual token holdings, the internal accounting diverges from the real balance. Subsequent `withdrawCollateral` calls attempt to transfer the internally-recorded amount, which exceeds the contract's actual holdings, causing `safeTransfer` to revert and permanently locking user funds.

---

### Finding Description

**Deposit path:**

`Endpoint.depositCollateralWithReferral` pulls exactly `amount` tokens from the caller via `handleDepositTransfer` and enqueues a `DepositCollateral` slow-mode transaction encoding that same `amount`. [1](#0-0) 

When the sequencer processes the slow-mode tx, `Clearinghouse.depositCollateral` credits `amountRealized = int128(txn.amount) * int128(multiplier)` to the user's SpotEngine balance — derived entirely from the encoded `amount`, not from a live `balanceOf` check. [2](#0-1) 

**Rebase event:**

If the collateral token is rebasing, the Clearinghouse's actual token balance can decrease (negative rebase) or increase (positive rebase) at any time. No hook, callback, or reconciliation mechanism in Nado updates `totalDepositsNormalized` or any user's `amountNormalized` in `SpotEngineState` to reflect this change. [3](#0-2) 

**Withdrawal path:**

`Clearinghouse.withdrawCollateral` calls `handleWithdrawTransfer(token, sendTo, amount, idx)`, which executes `token.safeTransfer(withdrawPool, uint256(amount))` where `amount` is derived from the internal SpotEngine balance — not from `token.balanceOf(address(this))`. [4](#0-3) 

If the Clearinghouse holds fewer tokens than the internally-recorded sum (due to a negative rebase), this `safeTransfer` reverts. The `assertUtilization` guard only checks `totalDeposits >= totalBorrows` (a purely internal invariant) and does not compare internal accounting against the actual on-chain token balance. [5](#0-4) 

**Positive rebase:**

A positive rebase silently inflates the Clearinghouse's actual balance above the internally-tracked total. The surplus tokens are permanently stranded in the contract with no mechanism to credit them to depositors or recover them.

---

### Impact Explanation

- **Negative rebase:** The Clearinghouse's actual token balance falls below the sum of all internally-recorded user balances. The first users to withdraw succeed; subsequent withdrawers' `safeTransfer` calls revert. User funds are locked in the contract with no recovery path through normal protocol flows.
- **Positive rebase:** Surplus tokens accumulate in the Clearinghouse and are permanently inaccessible — neither credited to depositors nor recoverable via any standard protocol function.

The broken invariant is: `token.balanceOf(clearinghouse) >= sum(all user internal balances for that token)`. A rebase can violate this invariant silently, with no on-chain detection. [6](#0-5) 

---

### Likelihood Explanation

Likelihood is **medium**. The Nado protocol's `addOrUpdateProduct` accepts any ERC20 token address with no restriction against rebasing tokens. If a rebasing token (e.g., stETH, AMPL, or any yield-bearing wrapper) is listed as a supported collateral product, any negative rebase event — which can be triggered by the token's own governance or algorithmic mechanism, entirely outside Nado's control — will corrupt the accounting. The trigger requires no privileged access from an attacker; the rebase is an inherent property of the token type. [7](#0-6) 

---

### Recommendation

1. **Short term:** Explicitly document that rebasing tokens are unsupported as collateral and add an allowlist or token-type validation in `addOrUpdateProduct` to reject tokens whose `totalSupply` can change externally.
2. **Long term:** If rebasing tokens are to be supported, replace the nominal-amount accounting with a shares-based model (similar to how Aave wraps rebasing tokens into `aTokens`), or add a reconciliation function that snapshots `token.balanceOf(clearinghouse)` and adjusts `totalDepositsNormalized` proportionally after a detected rebase.

---

### Proof of Concept

1. Owner calls `SpotEngine.addOrUpdateProduct` to list a rebasing token (e.g., AMPL) as `productId = 5`.
2. Alice calls `Endpoint.depositCollateral(subaccountName, 5, 1000e18)`. The Clearinghouse receives 1000 AMPL; SpotEngine records `amountNormalized` corresponding to 1000e18 × multiplier for Alice.
3. Bob calls `Endpoint.depositCollateral(subaccountName, 5, 1000e18)`. Same result; total internal deposits = 2000e18 × multiplier.
4. AMPL undergoes a 20% negative rebase. Clearinghouse now holds 1600 AMPL; internal accounting still reflects 2000 AMPL.
5. Alice submits a withdrawal for her full balance (1000 AMPL). The sequencer processes `withdrawCollateral`. `handleWithdrawTransfer` calls `token.safeTransfer(withdrawPool, 1000e18)`. This succeeds (1600 ≥ 1000). Clearinghouse now holds 600 AMPL.
6. Bob submits a withdrawal for his full balance (1000 AMPL). `token.safeTransfer(withdrawPool, 1000e18)` reverts — only 600 AMPL remain. Bob's 1000 AMPL is permanently locked. [8](#0-7) [9](#0-8)

        bytes32 sender,

        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }

    function _updateBalanceNormalized(
        State memory state,
        BalanceNormalized memory balance,
        int128 balanceDelta
    ) internal pure {
        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized -= balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized += balance.amountNormalized;
        }

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        int128 newAmount = balance.amountNormalized.mul(
            cumulativeMultiplierX18
        ) + balanceDelta;

        if (newAmount > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);

        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized += balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized -= balance.amountNormalized;
        }
    }
