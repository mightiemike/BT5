### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Forgery Protection — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof of equal depth. However, the deprecated `verify_transaction_inclusion` function remains `pub` and is callable by any unprivileged NEAR account via RPC. The `#[deprecated]` Rust attribute only emits compile-time warnings for Rust callers; it has zero effect on external NEAR RPC callers. An attacker can call the deprecated function directly, skip the coinbase proof check entirely, and obtain a `true` result for a forged transaction inclusion proof.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces two security invariants absent from v1:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — ensures proof depth matches the actual tree depth.
2. The coinbase transaction (always at index 0) must be proven against the block's `merkle_root` — this is the core mitigation for the 64-byte attack. [1](#0-0) 

The deprecated v1 function performs neither of these checks. It only verifies that `merkle_proof` is non-empty and that the computed root matches the block's `merkle_root`: [2](#0-1) 

Critically, v1 is still declared `pub` with only a `#[pause]` guard (no role-based access control), meaning any NEAR account can call it: [3](#0-2) 

The `#[deprecated]` attribute is a Rust compiler hint. NEAR contract methods are dispatched by name over JSON/Borsh RPC — the deprecation annotation has no runtime enforcement and does not prevent external callers from invoking the function.

The 64-byte attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) works because Bitcoin's Merkle tree hashes 64-byte concatenations of child hashes. An attacker can craft a fake "transaction" whose double-SHA256 hash equals an internal Merkle node. A Merkle proof for that internal node will verify correctly against the block's `merkle_root`, causing `verify_transaction_inclusion` to return `true` for a transaction that does not exist.

`verify_transaction_inclusion_v2` defeats this by requiring the coinbase proof (always at depth 0) to have the same length as the target proof, anchoring the expected tree depth. Since v1 lacks this check, calling it directly bypasses the entire mitigation.

---

### Impact Explanation

Any downstream NEAR contract or dApp that calls `verify_transaction_inclusion` — or is directed by an attacker to call it — can be deceived into accepting a forged Bitcoin transaction inclusion proof. This can result in releasing bridged funds, minting tokens, or executing cross-chain actions based on a Bitcoin transaction that never existed. The contract itself returns a `bool`; the corrupted value is the proof result (`true` for a forged proof), which propagates to every consumer of the API.

---

### Likelihood Explanation

The entry path requires no privileges: any NEAR account can call `verify_transaction_inclusion` via a standard NEAR RPC `call` transaction. The 64-byte Merkle forgery technique is publicly documented and has known tooling. The attacker must identify a target block in the mainchain (trivially obtained via `get_block_hash_by_height` or `get_last_block_header`) and construct a valid internal-node proof for it. This is a realistic, non-theoretical attack path.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract API entirely, or replace its body with an unconditional panic directing callers to `verify_transaction_inclusion_v2`. A `#[deprecated]` attribute alone provides no on-chain enforcement. As long as the function is `pub` and unpaused, it is reachable by any caller.

---

### Proof of Concept

1. Attacker queries `get_block_hash_by_height(H)` to obtain a mainchain block hash `B`.
2. Attacker retrieves the block header for `B` from `headers_pool` (via `get_last_block_header` or equivalent) to obtain `merkle_root`.
3. Attacker identifies an internal Merkle node `N` in block `B`'s transaction tree and constructs a Merkle proof `P` such that `compute_root_from_merkle_proof(N, idx, P) == merkle_root`.
4. Attacker crafts a 64-byte fake transaction whose `double_sha256` equals `N`.
5. Attacker calls `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B`, `tx_index = idx`, `merkle_proof = P`, `confirmations = 1`.
6. `compute_root_from_merkle_proof` returns `merkle_root`; the function returns `true`.
7. No coinbase proof is requested or checked.
8. Any downstream contract that cross-calls `verify_transaction_inclusion` and acts on the `true` result is deceived into treating the forged transaction as confirmed. [4](#0-3) [5](#0-4)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
