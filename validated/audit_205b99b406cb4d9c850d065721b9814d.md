### Title
`SwapAllowlistExtension` checks the router address instead of the real user, allowing any actor to bypass a curated pool's swap allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument it receives from the pool. The pool always passes `msg.sender` of the `swap()` call as `sender`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the actual user. The extension therefore checks whether the router is allowlisted, not whether the real user is allowlisted. Any user can bypass a curated pool's swap allowlist by routing through the standard periphery router.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (used as the namespace key) and `sender` is the value forwarded from the pool — i.e., whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` at the pool: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**Result:** the extension sees the router address as the swapper. There are two broken outcomes:

1. **Allowlist bypass (primary impact):** If the pool admin allowlists the router address (the only way to let any user swap through the router), every user — including those not individually allowlisted — can trade on the curated pool simply by calling the router.
2. **Allowlisted users locked out:** If the pool admin allowlists individual EOA addresses but not the router, those users can never actually swap, because EOAs cannot implement `IMetricOmmSwapCallback` and therefore cannot call `pool.swap()` directly.

---

### Impact Explanation

A curated pool that deploys `SwapAllowlistExtension` to restrict trading to a known set of addresses provides no real restriction. Any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) and the allowlist check passes as long as the router is allowlisted — which it must be for the router to be usable at all. This is a direct, complete bypass of the pool's access-control invariant with no special preconditions beyond using the standard periphery router.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface. Any user who discovers the mismatch (or simply uses the router by default) bypasses the allowlist. No privileged access, no special token behavior, and no unusual setup is required. The bypass is reachable on every swap through the router.

---

### Recommendation

The pool must propagate the real initiating user through the call stack so extensions can gate on the correct actor. Two complementary approaches:

1. **Pass the payer/initiator as a separate field.** The router already stores the real payer in transient storage (`_getPayer()`). Add an `initiator` parameter to `pool.swap()` (or encode it in `callbackData`/`extensionData`) and forward it to `beforeSwap` alongside `sender`.

2. **Extension-side: check `recipient` or require the caller to supply the real user in `extensionData`.** As a short-term mitigation, `SwapAllowlistExtension` could read the real user from a standardized prefix in `extensionData`, though this shifts trust to the caller.

The cleanest fix is option 1: the pool passes both the immediate caller (`msg.sender`) and the economic initiator (supplied by the router from its transient payer context) to every extension hook, and `SwapAllowlistExtension` gates on the initiator.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (required so any router swap works at all).
  - Pool admin does NOT call setAllowedToSwap(pool, alice, true).

Attack:
  1. Alice (not allowlisted) calls router.exactInputSingle({pool: pool, ...}).
  2. Router calls pool.swap(...) — msg.sender at pool = router.
  3. Pool calls beforeSwap(sender=router, ...).
  4. Extension checks allowedSwapper[pool][router] == true → passes.
  5. Alice's swap executes on the curated pool despite not being allowlisted.

Expected: revert NotAllowedToSwap.
Actual:   swap succeeds.
```
