### Title
`DepositAllowlistExtension` checks `owner` (LP-share recipient) instead of `sender` (token provider), allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` enforces the allowlist against the `owner` parameter (the address that will receive LP shares), not against `sender` (the address that actually calls the pool and provides tokens via the swap callback). Because `addLiquidity` accepts a caller-supplied `owner` with no ownership constraint, any non-allowlisted address can bypass the guard by setting `owner` to any allowlisted address.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address from `msg.sender` and forwards both to the extension hook:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` passes both `sender` and `owner` verbatim to the extension:

```solidity
// ExtensionCalling.sol lines 95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` silently discards `sender` (first parameter, unnamed `address`) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`SwapAllowlistExtension.beforeSwap` — the parallel guard — correctly checks `sender` (the actual caller):

```solidity
// SwapAllowlistExtension.sol lines 31-41
function beforeSwap(address sender, address, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

The asymmetry is the root cause. In `addLiquidity`, `msg.sender` to the pool is the entity that:
1. Triggers the call
2. Receives the `metricOmmAddLiquidityCallback` and must transfer tokens into the pool

`owner` is merely the address credited with LP shares. Because `owner` is a free parameter supplied by the caller, any non-allowlisted address can pass the guard by setting `owner` to any address that is on the allowlist.

---

### Impact Explanation

**Access-control bypass with fund-flow consequence.** The deposit allowlist is the pool admin's mechanism to restrict which addresses may inject liquidity into the pool (e.g., private/institutional pools, KYC-gated pools, or pools that must exclude sanctioned addresses). With this bug:

- A non-allowlisted address calls `pool.addLiquidity(owner = allowlisted_address, ...)`.
- The guard checks `allowedDepositor[pool][allowlisted_address]` → passes.
- The non-allowlisted `msg.sender` provides tokens via callback and the allowlisted address receives LP shares it never requested.
- Tokens from a non-allowlisted source now reside in the pool, violating the invariant the admin configured.

A motivated attacker can use this to deposit into specific bins, shift the pool's liquidity distribution, and then extract value through swaps at prices that the manipulated bin state now permits — a direct LP-principal loss path. Even without active exploitation, the compliance/access-control guarantee the pool admin paid for is silently void.

---

### Likelihood Explanation

Exploitability requires only that:
1. A pool is deployed with `DepositAllowlistExtension` and at least one allowlisted address exists (trivially true for any live restricted pool).
2. The attacker calls `addLiquidity` with `owner` set to that allowlisted address.

No privileged access, no special token, no oracle manipulation is needed. Any EOA or contract can trigger this in a single transaction.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual caller) instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate on the LP-share recipient (`owner`) rather than the token provider (`sender`), the storage mapping and NatSpec must be updated to make that explicit, and a separate `sender` check should be added so that the token-provider path is also gated.

---

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  bob is NOT on the allowlist

Attack:
  bob calls pool.addLiquidity(owner = alice, salt, deltas, callbackData, extensionData)

Inside pool:
  _beforeAddLiquidity(msg.sender=bob, owner=alice, ...)
    → extension receives (sender=bob, owner=alice)
    → checks allowedDepositor[pool][alice] == true  ✓  (guard passes)

  LiquidityLib.addLiquidity credits LP shares to alice
  pool calls bob.metricOmmAddLiquidityCallback(...)
  bob transfers tokens into the pool

Result:
  bob (non-allowlisted) has deposited tokens into the restricted pool.
  alice holds LP shares she did not request.
  The deposit allowlist invariant is broken.
