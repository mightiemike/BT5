### Title
Pool admin can set uncapped per-bin additional fees via `setPoolBinAdditionalFees` to frontrun user swaps, bypassing the global fee cap system — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`setPoolBinAdditionalFees` allows the pool admin to set per-bin additional fees (`addFeeBuyE6`, `addFeeSellE6`) to any value up to `uint16` max (65,535 in E6 ≈ 6.55%) with no cap validation and no timelock. The global fee setter `setPoolAdminFees` enforces `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8` caps, but the per-bin setter has no analogous guard. A malicious pool admin can frontrun a user's pending swap by atomically raising the per-bin fee to `type(uint16).max`, capturing the excess as protocol/admin revenue, then resetting it — directly analogous to the SwapFacility `setSwapInFeeRate` frontrun.

### Finding Description

`MetricOmmPoolFactory.setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` straight to the pool with no upper-bound check:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [1](#0-0) 

The pool-side handler only validates the bin index, not the fee magnitude:

```solidity
// metric-core/contracts/MetricOmmPool.sol
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    onlyFactory
    nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

Contrast with `setPoolAdminFees`, which enforces explicit caps before writing:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [3](#0-2) 

The hard caps for the global fee system are:

```solidity
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000;   // 20%
uint24 internal constant HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000; // 1%
``` [4](#0-3) 

Per-bin fees are stored as `uint16` in `BinState` with no documented or enforced ceiling:

```solidity
uint16 addFeeBuyE6;   // Additional fee for buying token0; 1e6 = 100%
uint16 addFeeSellE6;  // Additional fee for buying token1; 1e6 = 100%
``` [5](#0-4) 

`uint16` max = 65,535 → 65,535 / 1,000,000 ≈ **6.55%** additional fee per bin, additive on top of the global spread fee. There is no timelock on this setter.

### Impact Explanation

A malicious pool admin can:
1. Observe a user's pending `swap` transaction in the mempool.
2. Frontrun it with `setPoolBinAdditionalFees(pool, curBin, 65535, 65535)`.
3. The user's swap executes with up to 6.55% extra fee deducted from their output.
4. The admin resets the per-bin fee to 0 in the same or next block.

The user receives materially less output than the oracle price and global spread fee would imply. If the user set `amountOutMinimum = 0` (as seen in test helpers) or a loose slippage bound, the full 6.55% is captured. This bypasses the global admin fee cap (`maxAdminSpreadFeeE6`) because per-bin fees are additive and uncapped. The total effective fee can reach `spreadFeeE6 + 65535` — exceeding the intended cap enforced by `setPoolAdminFees`. [6](#0-5) 

### Likelihood Explanation

- The pool admin is a semi-trusted role with no timelock on `setPoolBinAdditionalFees`.
- Pool admins are expected to be EOAs or multisigs per the docs; an EOA admin can execute the frontrun in a single block.
- Users interacting via the official UI (which the pool admin may control) may be presented with loose or zero slippage defaults, as seen in test fixtures (`amountOutMinimum: 0`).
- No on-chain mechanism prevents the admin from toggling per-bin fees arbitrarily between blocks. [7](#0-6) 

### Recommendation

1. **Add a cap** in `setPoolBinAdditionalFees` (factory side) analogous to the global fee cap:
   ```solidity
   uint16 internal constant MAX_BIN_ADDITIONAL_FEE_E6 = <sane_value>; // e.g. 10_000 (1%)
   if (addFeeBuyE6 > MAX_BIN_ADDITIONAL_FEE_E6 || addFeeSellE6 > MAX_BIN_ADDITIONAL_FEE_E6)
       revert BinFeeTooHigh();
   ```
2. **Add a timelock** on `setPoolBinAdditionalFees` (or at minimum a minimum-delay enforced by the factory) to prevent same-block frontrunning, consistent with the oracle rotation timelock already present for `proposePoolPriceProvider`.
3. **Collect accrued fees** before changing per-bin fees (as `setPoolAdminFees` already does) to prevent retroactive fee manipulation.

### Proof of Concept

```solidity
// Pool admin frontrun scenario
address pool = ...; // pool with active liquidity
int8 activeBin = 0; // bin the user's swap will traverse

// Step 1: Admin sees user's exactInputSingle tx in mempool
// Step 2: Admin frontruns with max per-bin fee (no cap, no timelock)
vm.prank(poolAdmin);
factory.setPoolBinAdditionalFees(pool, activeBin, 65535, 65535); // ~6.55% extra

// Step 3: User's swap executes — receives ~6.55% less output than expected
// (amountOutMinimum: 0 → no revert protection)
uint256 amountOut = router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool, amountIn: 1e18, amountOutMinimum: 0, ...
    })
);
// amountOut is ~6.55% lower than oracle price implies

// Step 4: Admin resets fee
vm.prank(poolAdmin);
factory.setPoolBinAdditionalFees(pool, activeBin, 0, 0);
