### Title
`StatelessTransactionValidator` Skips `max_l2_gas_amount` Admission Check for Declare Transactions While Enforcing It for Invoke and DeployAccount — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces an upper bound on `l2_gas.max_amount` for `Invoke` and `DeployAccount` transactions but explicitly skips that check for `Declare` transactions. A submitter can craft a `Declare` transaction with `l2_gas.max_amount = u64::MAX` and the gateway will admit it, bypassing the admission control that exists for every other transaction type.

---

### Finding Description

In `validate_resource_bounds`, the `max_l2_gas_amount` ceiling is guarded by a type-dispatch that silently no-ops for `Declare`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The default `max_l2_gas_amount` is `1_210_000_000`: [2](#0-1) 

The same `validate_resource_bounds` function correctly rejects `Invoke` and `DeployAccount` transactions whose `l2_gas.max_amount` exceeds this ceiling. The test `valid_l2_gas_amount_on_declare` explicitly documents and asserts that a Declare transaction with `max_amount = 200` passes even when the configured limit is `100`: [3](#0-2) 

The full `validate` entry point for the stateless validator shows the asymmetry: all three transaction types share the `validate_resource_bounds` call, but only Declare silently skips the amount ceiling inside it: [4](#0-3) 

---

### Impact Explanation

A caller can submit a `RpcDeclareTransaction::V3` with `resource_bounds.l2_gas.max_amount = u64::MAX`. The gateway's stateless validator passes it, the stateful validator passes it (it only checks that `max_amount >= minimal_gas_amount`, a trivially satisfied lower-bound), and the transaction enters the mempool. This violates the gateway's own admission invariant: the `max_l2_gas_amount` config value is supposed to be a hard ceiling on the L2 gas a single transaction may claim, but it is not enforced for Declare transactions.

Impact: **High — Mempool/gateway admission accepts an invalid transaction that should be rejected before sequencing.** [5](#0-4) 

---

### Likelihood Explanation

The trigger is entirely unprivileged: any user who can submit a `starknet_addDeclareTransaction` RPC call can set `l2_gas.max_amount` to an arbitrary value. No special capability, key, or prior state is required. The TODO comment confirms the gap is known but unresolved.

---

### Recommendation

Remove the `Declare` carve-out and apply the same `max_l2_gas_amount` ceiling uniformly:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher ceiling (e.g., because Sierra compilation consumes more L2 gas), introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.

---

### Proof of Concept

1. Construct a valid `RpcDeclareTransaction::V3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and any non-zero `max_price_per_unit` above `min_gas_price`.
2. Submit it to the gateway's `add_tx` endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for Declare transactions regardless of the configured `max_l2_gas_amount`.
4. The same transaction submitted as `RpcInvokeTransaction::V3` with identical resource bounds would be rejected with `MaxGasAmountTooHigh`.

The test `valid_l2_gas_amount_on_declare` already encodes this behavior as an expected pass, confirming the bypass is present in the production code path. [6](#0-5)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {

                gas_amount: resource_bounds.l2_gas.max_amount,
