### Title
Difficulty Adjustment Uses Stale Mainchain Height Index Instead of Actual Fork Ancestor — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary
When validating a fork block's difficulty target, all chain modules call `get_header_by_height()` to retrieve the "first block" of the difficulty adjustment interval. This function reads from `mainchain_height_to_header` — the mainchain index — not from the fork's actual ancestor chain. For fork blocks, this is the wrong block, analogous to using `_cachedTotalUnderlying` (a stale cached value) instead of `cachedTotalUnderlying()` (the fresh, correct value). The result is that difficulty validation for fork blocks uses the wrong timestamp, allowing an attacker to submit fork blocks with an incorrect (easier) difficulty target.

### Finding Description

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

This always returns the **mainchain** block at the given height. The Dogecoin difficulty module itself contains a developer `TODO` acknowledging this is potentially wrong:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [2](#0-1) 

The same pattern appears in Bitcoin and Litecoin: [3](#0-2) [4](#0-3) 

When a fork block at height `H+1` is submitted, `prev_block_header` is the fork block at height `H` (correctly fetched via `headers_pool` using `prev_block_hash`). But `height_first` for Dogecoin (post-145,000, per-block difficulty) is computed as:

```rust
let mut blocks_to_go_back = difficulty_adjustment_interval - 1; // = 0
...
let height_first = prev_block_header.block_height.checked_sub(blocks_to_go_back)...
// height_first == prev_block_header.block_height == H
``` [5](#0-4) 

So `get_header_by_height(H)` returns `mainchain_block_H`, while the correct value is `fork_block_H` (i.e., `prev_block_header` itself). The difficulty calculation then computes:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
// = fork_block_H.time - mainchain_block_H.time   (WRONG)
// should be: fork_block_H.time - fork_block_H.time = 0 (clamped to min_timespan)
``` [6](#0-5) 

### Impact Explanation

An attacker submits `fork_block_H` with a timestamp significantly later than `mainchain_block_H.time` (within the `MAX_FUTURE_BLOCK_TIME_LOCAL` constraint). The difficulty for `fork_block_{H+1}` is then computed using an inflated timespan (`fork_block_H.time - mainchain_block_H.time` instead of the correct near-zero value). A larger timespan produces a higher target (easier difficulty). The attacker mines `fork_block_{H+1}` with this easier target, which the contract accepts. The `chain_work` for `fork_block_{H+1}` is computed from the attacker-controlled `bits` field reflecting the easier difficulty:

```rust
let (current_block_computed_chain_work, overflow) = prev_block_header
    .chain_work
    .overflowing_add(work_from_bits(block_header.bits));
``` [7](#0-6) 

By repeating this across multiple fork blocks, the attacker can accumulate chainwork faster than expected (mining with lower difficulty per block), potentially triggering a reorg that promotes the attacker's fork to mainchain:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [8](#0-7) 

Once the attacker's fork is the mainchain, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will return `true` for transactions in the attacker's fork blocks and `false` for transactions in the legitimate chain, corrupting all downstream SPV proof results.

### Likelihood Explanation

**Dogecoin** (highest risk): Only a 2-block-deep fork is required to trigger the desynchronization. Any unprivileged NEAR caller can call `submit_blocks` with adversarial headers. The attacker controls `fork_block_H.time` (within MTP and `MAX_FUTURE_BLOCK_TIME_LOCAL` bounds), giving meaningful control over the difficulty calculation. The developer `TODO` comment confirms this path was identified as potentially incorrect during development.

**Bitcoin/Litecoin** (lower risk): The difficulty adjustment interval is 2016 blocks, so the fork must diverge before the retarget boundary — requiring the attacker to have submitted 2016+ fork blocks. This is computationally expensive but the same structural flaw exists.

### Recommendation

Replace `get_header_by_height(height_first)` with a chain-walk from `prev_block_header` backwards through `get_prev_header()` calls to reach the actual ancestor at `height_first`. This mirrors how the reference Bitcoin Core implementation walks the chain via `prev_block_hash` links rather than a height index. For Dogecoin's per-block difficulty (post-145,000), `height_first == prev_block_header.block_height`, so `first_block_time` should simply be `prev_block_header.block_header.time` directly, eliminating the lookup entirely.

### Proof of Concept

1. Mainchain has blocks at heights `H-1` (hash `M_{H-1}`), `H` (hash `M_H`, time `T_M`), `H+1`, ...
2. Attacker calls `submit_blocks([fork_block_H])` where `fork_block_H.prev_block_hash = M_{H-1}` and `fork_block_H.time = T_M + MAX_FUTURE_BLOCK_TIME_LOCAL` (maximum allowed future timestamp). This passes difficulty check because its own difficulty is computed from `mainchain_block_{H-1}` (correct).
3. Attacker calls `submit_blocks([fork_block_{H+1}])` where `fork_block_{H+1}.prev_block_hash = fork_block_H.hash`.
4. Contract computes `height_first = H`, calls `get_header_by_height(H)` → returns `mainchain_block_H` with time `T_M`.
5. `modulated_timespan = fork_block_H.time - T_M = MAX_FUTURE_BLOCK_TIME_LOCAL` (7200 seconds), far exceeding the 60-second Dogecoin target spacing.
6. `calculate_next_work_required` produces a much higher target (easier difficulty) for `fork_block_{H+1}`.
7. Attacker mines `fork_block_{H+1}` with this easier target and submits it; the contract accepts it.
8. Repeating steps 3–7 allows the attacker to build a fork with less actual PoW than the mainchain while accumulating sufficient `chain_work` to trigger `reorg_chain`, corrupting the canonical chain view and all subsequent `verify_transaction_inclusion` results.
