### Title
`ValidResourceBounds::AllResources` with zero L2/L1-data gas silently collapses to `L1Gas` after protobuf round-trip, producing a divergent transaction hash in the P2P block-sync path — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a zero-value heuristic to decide whether a transaction carries `L1Gas` or `AllResources` bounds. Because the serializer encodes `L1Gas` as three fields with the L2/L1-data fields zeroed, any `AllResources` transaction whose L2 and L1-data bounds happen to be zero is indistinguishable from an `L1Gas` transaction in the wire format. After a round-trip through the P2P block-sync path the type collapses from `AllResources` to `L1Gas`, and the transaction hash recomputed by the receiving node diverges from the hash committed by the sequencer, because `get_tip_resource_bounds_hash` chains a different number of resource-bound felts for the two variants.

---

### Finding Description

**Serialization ambiguity (root cause)**

`From<ValidResourceBounds> for protobuf::ResourceBounds` encodes `L1Gas` by writing zero into the `l2_gas` and `l1_data_gas` wire fields: [1](#0-0) 

An `AllResources` value whose `l2_gas` and `l1_data_gas` are both zero produces an identical byte sequence.

**Deserialization collapse**

`ValidResourceBounds::try_from(protobuf::ResourceBounds)` reconstructs the variant by inspecting those same fields: [2](#0-1) 

When both fields are zero the result is `L1Gas`, regardless of whether the originating transaction was `AllResources`.

**Hash divergence**

`get_tip_resource_bounds_hash` chains a different number of felts depending on the variant: [3](#0-2) 

- `L1Gas` → two resource felts (`L1_GAS`, `L2_GAS`)
- `AllResources` → three resource felts (`L1_GAS`, `L2_GAS`, `L1_DATA_GAS`)

For an `AllResources` transaction with zero L2 and zero L1-data bounds the extra `L1_DATA_GAS` felt (encoding zeros) is still hashed, so the Poseidon output differs from the two-felt `L1Gas` hash.

**Reachable trigger**

`InternalRpcInvokeTransactionV3` always carries `AllResourceBounds`: [4](#0-3) 

The gateway's stateless validator only requires that *at least one* resource bound is non-zero, so a transaction with `l1_gas > 0`, `l2_gas = 0`, `l1_data_gas = 0` passes admission: [5](#0-4) 

The sequencer computes and stores the hash using the `AllResources` path (three felts). When the block is later propagated over P2P block-sync the receiving node deserializes the same transaction via the `ValidResourceBounds::try_from` path (two felts), producing a different hash.

The mempool P2P path is **not** affected because it uses `AllResourceBounds::try_from` directly: [6](#0-5) 

---

### Impact Explanation

**High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

A syncing node that recomputes transaction hashes will observe a mismatch and either reject a valid block or store the transaction under the wrong hash. A node that trusts the stored hash but reconstructs the `ValidResourceBounds` type will subsequently serve incorrect fee-estimation and simulation results for that transaction, because `get_gas_vector_computation_mode` returns `NoL2Gas` for `L1Gas` and `All` for `AllResources`: [7](#0-6) 

---

### Likelihood Explanation

**Low.** The trigger requires a user to submit an `AllResources` invoke transaction with `l1_gas > 0` and both `l2_gas = 0` and `l1_data_gas = 0`. This is an unusual but valid combination that passes all current gateway checks. No privileged access is required.

---

### Recommendation

Remove the zero-value heuristic from `ValidResourceBounds::try_from`. Instead, add an explicit discriminant field to `protobuf::ResourceBounds` (e.g., a `resource_bounds_type` enum) so the serializer can faithfully round-trip both variants without relying on the absence of zero values. Until the protobuf schema is updated, the deserializer should default to `AllResources` when all three fields are present, reserving the `L1Gas` path only for the legacy case where `l2_gas` is absent from the wire message (matching the existing `TODO(Shahak)` comment at line 426).

---

### Proof of Concept

1. Craft an invoke V3 transaction: `resource_bounds = AllResources { l1_gas: {max_amount: 1000, max_price: 1}, l2_gas: {0,0}, l1_data_gas: {0,0} }`.
2. Submit to the gateway. The gateway accepts it; `InternalRpcInvokeTransactionV3` stores it as `AllResources`. `get_tip_resource_bounds_hash` chains three felts → hash `H_all`.
3. The sequencer seals a block containing this transaction with hash `H_all`.
4. A peer node receives the block via P2P block-sync. `ValidResourceBounds::try_from` sees `l2_gas = 0` and `l1_data_gas = 0` and returns `L1Gas`. `get_tip_resource_bounds_hash` chains two felts → hash `H_l1 ≠ H_all`.
5. The peer node's hash verification fails or it stores the transaction under `H_l1`, causing divergent state and wrong fee-estimation results for any subsequent RPC call referencing that transaction.

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
