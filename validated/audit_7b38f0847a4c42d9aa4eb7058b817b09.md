### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources(X,0,0)` to `L1Gas(X)`, Producing a Different Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf converter `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` silently reclassifies an `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` transaction as `L1Gas(X)` after a round-trip. Because `get_tip_resource_bounds_hash` hashes a **different number of resource felts** depending on the variant (2 for `L1Gas`, 3 for `AllResources`), the same transaction content produces two distinct transaction hashes: one computed at the originating node and a different one computed at any node that deserializes the transaction from protobuf. This is the sequencer analog of the MarginRouter double-credit: an estimation (hash) is committed under one representation, then the same payload is re-evaluated under a different representation, yielding a divergent result.

### Finding Description

**Step 1 – The hash preimage is variant-dependent.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` builds the poseidon preimage differently for the two `ValidResourceBounds` variants:

```
L1Gas(X)        → poseidon([tip, pack(X, L1_GAS), pack(0, L2_GAS)])          // 3 elements
AllResources(X,0,0) → poseidon([tip, pack(X, L1_GAS), pack(0, L2_GAS), pack(0, L1_DATA_GAS)])  // 4 elements
```

`pack(0, L1_DATA_GAS)` is non-zero because the resource-name bytes `b"L1_DATA"` are embedded in the felt. The two hashes are therefore always distinct for the same `l1_gas` value. [1](#0-0) 

**Step 2 – The protobuf deserializer silently changes the variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies the following rule:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← variant changed
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

A transaction originally stored as `AllResources(X, 0, 0)` is deserialized as `L1Gas(X)` because both `l2_gas` and `l1_data_gas` are zero. The comment `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.` confirms this path is intentionally kept open. [2](#0-1) 

**Step 3 – A valid user transaction triggers the divergence.**

`RpcInvokeTransactionV3` and `InternalRpcInvokeTransactionV3` carry `AllResourceBounds` (not `ValidResourceBounds`). The gateway accepts any combination of zero/non-zero resource bounds, including `{ l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`: [3](#0-2) 

When this transaction is converted to the storage `Transaction` type (e.g., `InvokeTransactionV3`), the `AllResourceBounds` is wrapped as `ValidResourceBounds::AllResources`: [4](#0-3) 

The originating node computes the transaction hash using the `AllResources` path (4-element poseidon preimage). When the block is propagated over P2P and the receiving node deserializes the `Transaction` from protobuf, the `ValidResourceBounds` becomes `L1Gas(X)`, and the hash is recomputed using the `L1Gas` path (3-element poseidon preimage). The two hashes differ.

**Step 4 – The serializer always emits all three fields, so the deserializer's zero-check is the sole discriminant.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` always sets `l1_data_gas: Some(...)`, so the protobuf wire message always carries all three fields. The receiving node's decision to produce `L1Gas` is driven entirely by the value being zero, not by any absent field: [5](#0-4) 

### Impact Explanation

Any node that receives a historical block over P2P and deserializes a `Transaction` containing `AllResources(X, 0, 0)` will store a **different `transaction_hash`** in its receipt and state than the originating node. This constitutes:

- **Wrong receipt/state**: the stored `transaction_hash` diverges from the canonical hash signed by the account and committed by the sequencer.
- **Broken hash-chain invariant**: block hash computation that includes transaction hashes will diverge between nodes, potentially causing consensus failures or incorrect state roots.

This matches: *"Critical. Wrong state, receipt, event … or revert result"* and *"High. Transaction conversion or signature/hash logic binds the wrong … hash … or executable payload."*

### Likelihood Explanation

Any unprivileged user can submit a V3 invoke/declare/deploy-account transaction with `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas` non-zero). The gateway's stateless validator explicitly accepts this combination. The divergence is triggered automatically whenever such a transaction is included in a block that is synced over P2P. [6](#0-5) 

### Recommendation

Replace the value-based heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit discriminant field in the protobuf schema, or always deserialize to `AllResources` when all three fields are present (regardless of their values). The TODO comment already acknowledges this path should be removed:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
```

Once 0.13.2 support is dropped, the converter should require `l1_data_gas` to be present and always produce `AllResources`, eliminating the variant-switching logic entirely. Alternatively, add a boolean discriminant field `is_all_resources` to the protobuf `ResourceBounds` message so the variant is preserved across serialization.

### Proof of Concept

```
1. User submits InvokeV3 with AllResourceBounds { l1_gas: {max_amount:100, max_price:1}, l2_gas: {0,0}, l1_data_gas: {0,0} }.

2. Gateway accepts it. TransactionConverter computes:
   tx_hash_A = poseidon(["invoke", version, sender, 
       poseidon([tip, pack({100,1}, L1_GAS), pack({0,0}, L2_GAS), pack({0,0}, L1_DATA_GAS)]),
       ...])
   Stores InternalRpcTransaction { tx_hash: tx_hash_A, ... }.

3. Transaction is included in block N. Block is committed with tx_hash_A in the receipt.

4. Block N is propagated over P2P. Receiving node deserializes the Transaction from protobuf:
   l1_data_gas.is_zero() && l2_gas.is_zero() → ValidResourceBounds::L1Gas({100,1})

5. Receiving node recomputes:
   tx_hash_B = poseidon(["invoke", version, sender,
       poseidon([tip, pack({100,1}, L1_GAS), pack({0,0}, L2_GAS)]),   // ← only 2 resource felts
       ...])

6. tx_hash_A ≠ tx_hash_B.
   Receiving node stores tx_hash_B in its receipt for block N.
   The two nodes now disagree on the transaction hash for this block.
``` [7](#0-6) [8](#0-7)

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));

        let Some(l2_gas) = value.l2_gas else {

        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;

            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
