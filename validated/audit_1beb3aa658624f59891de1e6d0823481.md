### Title
`burnNlp` Health Check Uses `MAINTENANCE` Instead of `INITIAL`, Leaving Positions at Immediate Liquidation Boundary — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp` function in `Clearinghouse.sol` enforces only a `MAINTENANCE` health check after reducing a subaccount's NLP balance. Because `MAINTENANCE` is the liquidation threshold (not the safe entry threshold), a user can exit NLP and leave their subaccount exactly at the liquidation boundary — immediately liquidatable by any liquidator.

---

### Finding Description

Nado's risk system has two distinct health thresholds:

- **`INITIAL`** — uses stricter collateral weights (lower effective LTV). This is the threshold required to *open* or *increase* a position.
- **`MAINTENANCE`** — uses looser collateral weights (higher effective LTV, closer to liquidation). This is the threshold below which a subaccount *becomes liquidatable*.

`mintNlp` correctly enforces `INITIAL` health: [1](#0-0) 

`burnNlp`, however, only enforces `MAINTENANCE` health: [2](#0-1) 

The comment in the code acknowledges the intent to prevent unhealthy subaccounts, but the chosen threshold (`MAINTENANCE`) is the liquidation floor, not the safe-entry floor. A subaccount that passes `MAINTENANCE >= 0` but fails `INITIAL >= 0` is already in the zone between "safe" and "liquidatable" — any small adverse price move or fee accrual pushes it below maintenance and makes it immediately liquidatable.

The weight difference between `INITIAL` and `MAINTENANCE` is defined in `RiskHelper`: [3](#0-2) 

`longWeightInitialX18 < longWeightMaintenanceX18` for long positions, meaning `INITIAL` health is strictly harder to satisfy. A position at `MAINTENANCE == 0` is at the exact liquidation boundary.

---

### Impact Explanation

A user with an existing leveraged position (e.g., spot borrows) can call `burnNlp` to reduce their NLP collateral until their subaccount sits exactly at `MAINTENANCE == 0`. This position is immediately liquidatable by any liquidator in the next block. The user loses their collateral to liquidation penalties, and the liquidator profits at the user's expense. The protocol's insurance fund may also be drawn upon if the position is insolvent.

**Impact: Medium** — direct loss of user funds via immediate liquidation.

---

### Likelihood Explanation

Any user who holds NLP alongside a leveraged spot or perp position can trigger this. No special privileges are required. The `burnNlp` path is a standard user-facing operation routed through `Endpoint`. The condition is reachable whenever a user's `INITIAL` health would go negative after the burn but `MAINTENANCE` health remains non-negative.

**Likelihood: Medium** — reachable by any unprivileged user with an NLP position and existing leverage.

---

### Recommendation

Replace the `MAINTENANCE` health check in `burnNlp` with an `INITIAL` health check, consistent with `mintNlp` and all other position-opening operations in the protocol:

```solidity
// Before (vulnerable):
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);

// After (fixed):
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

This ensures that after burning NLP, the subaccount remains at a safe collateral ratio with a meaningful buffer above the liquidation threshold — consistent with how all other collateral-reducing operations are gated in the protocol.

---

### Proof of Concept

1. User has a subaccount with: 100 USDC quote balance, 50 USDC NLP, and a spot borrow of 80 USDC worth of an asset. Suppose `INITIAL` health = +5, `MAINTENANCE` health = +15.
2. User calls `burnNlp` to redeem 50 USDC of NLP. After the burn, quote balance increases by ~50 USDC (minus fees), NLP balance goes to 0.
3. Suppose the burn fee causes net health impact such that `INITIAL` health = −3, `MAINTENANCE` health = +2.
4. The `burnNlp` check at line 527 passes (`MAINTENANCE >= 0`). The transaction succeeds.
5. The subaccount now has `INITIAL < 0` and `MAINTENANCE > 0` — it is in the liquidation-eligible zone.
6. A liquidator calls `liquidateSubaccountImpl`. The check `isUnderMaintenance` at line 603 passes as soon as any small price move or fee accrual pushes `MAINTENANCE` below 0. [4](#0-3) [5](#0-4)

```text
    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
```
