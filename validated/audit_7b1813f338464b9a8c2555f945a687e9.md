### Title
Internal-Node Merkle Proof Forgery via Unconstrained Proof Length in `verify_transaction_inclusion` — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is a public, unpaused NEAR contract method that delegates directly to `compute_root_from_merkle_proof` without validating that the supplied proof length matches the actual tree depth of the block. An unprivileged caller can supply an internal Merkle node hash as `tx_id` together with a proof that is shorter than the real tree depth, and the function will return `true` — falsely certifying that a fabricated hash is an included transaction in a canonical block.

---

### Finding Description

`compute_root_from_merkle_proof` iterates unconditionally over whatever proof elements are provided: [1](#0-0) 

There is no check that `merkle_proof.len()` equals `ceil(log2(n))` for the block's transaction count, and no check that `transaction_hash` is a leaf-level hash. The function simply hashes upward from whatever starting value is given.

`verify_transaction_inclusion` passes attacker-controlled inputs directly to this function. Its only guard on the proof is a non-empty check: [2](#0-1) 

`ProofArgs` places no structural constraint on `merkle_proof` length: [3](#0-2) 

The function is public (no `#[private]`), carries only `#[pause]` (not paused by default), and is callable by any NEAR account: [4](#0-3) 

The code's own `# Warning` comment acknowledges the exact flaw but defers responsibility to "a higher level of verification" that does not exist within the contract: [5](#0-4) 

---

### Impact Explanation

For a canonical block with 4 transactions `[tx0, tx1, tx2, tx3]` and merkle root `R = hash(node_01, node_23)` where `node_01 = hash(tx0, tx1)` and `node_23 = hash(tx2, tx3)`:

- An attacker sets `tx_id = node_01`, `tx_index = 0`, `merkle_proof = [node_23]`.
- `compute_root_from_merkle_proof(node_01, 0, [node_23])` computes `hash(node_01, node_23) = R`.
- `verify_transaction_inclusion` returns `true`.

`node_01` is not a transaction — it is an internal tree node. The contract has certified inclusion of a fabricated identifier in a real canonical block. Any downstream consumer (bridge, oracle, cross-chain protocol) that trusts this return value can be deceived into accepting a fabricated on-chain event.

---

### Likelihood Explanation

- No privilege is required. Any NEAR account can call `verify_transaction_inclusion` directly.
- The block's merkle tree structure is fully public (Bitcoin block data is public), so an attacker can compute all internal node hashes offline.
- The crafted proof requires only knowledge of the block's transaction hashes, which are available from any Bitcoin node.
- The function is deprecated but remains deployed and unpaused.

---

### Recommendation

1. **Remove or hard-pause `verify_transaction_inclusion`** at the contract level. Since `verify_transaction_inclusion_v2` exists as the replacement, the v1 method should be permanently disabled via the `PauseManager` role or removed in the next upgrade.
2. **Add proof-length validation** to `compute_root_from_merkle_proof`: reject any call where `merkle_proof.len()` does not equal the expected tree depth, or add this check in `verify_transaction_inclusion` before delegating.
3. **Do not rely on documentation warnings** as a security boundary. The `# Warning` comment does not prevent exploitation; it only documents the known flaw.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
