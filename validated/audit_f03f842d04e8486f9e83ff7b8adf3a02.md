### Title
Declare Transactions Bypass `max_l2_gas_amount` Gateway Admission Check — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` applies the `max_l2_gas_amount` ceiling only to non-`Declare` transactions. The guard is structured as an inverted exclusion — identical in shape to the external bug — so any `Declare` transaction with an arbitrarily large `l2_gas.max_amount` passes gateway admission unchecked.

### Finding Description

In `validate_resource_bounds`, the check reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // empty — no check
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The guard only blocks `Invoke` and `DeployAccount` transactions from exceeding `max_l2_gas_amount`. `Declare` transactions fall into the empty `if` branch and are never checked. This is structurally identical to the external bug: the check only prevents one specific party (non-`Declare` senders) from violating the bound, while the other party (`Declare` senders) is silently allowed through.

The `max_l2_gas_amount` field is not merely a fee cap. It is propagated as the initial Sierra gas counter for execution:

```rust
let remaining_validation_gas = &mut remaining_gas.limit_usage(
    tx_context.block_context.versioned_constants.os_constants.validate_max_sierra_gas,
);
``` [2](#0-1) 

A `Declare` transaction submitted with `l2_gas.max_amount = u64::MAX` passes `validate_resource_bounds`, passes `validate_tx_size` (which checks the serialized class size, not the gas field), and is admitted to the mempool and batcher with an unbounded gas limit.

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

A `Declare` transaction whose `l2_gas.max_amount` exceeds `config.max_l2_gas_amount` is admitted by the gateway when it should be rejected. The batcher then schedules it with a gas limit that the gateway was designed to cap. This can:

1. Allow a single `Declare` transaction to monopolize the L2 gas budget of a block, starving other transactions.
2. Cause the bouncer to accept a block whose actual gas consumption exceeds the intended per-transaction ceiling, producing wrong resource accounting.
3. Undermine the fee-estimation invariant: the gateway's `max_l2_gas_amount` bound is the contract between the admission layer and the execution layer; bypassing it for one transaction type breaks that contract.

### Likelihood Explanation

Any unprivileged user can submit a `Declare` transaction. No special role, key, or state is required. The only prerequisite is that the sender has a valid Sierra class to declare. The bypass is unconditional — it does not depend on any runtime state or configuration flag.

### Recommendation

Remove the `Declare` exemption and apply the same `max_l2_gas_amount` ceiling to all transaction types:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If `Declare` transactions legitimately require a higher gas ceiling (e.g., because compilation is more expensive), introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.

### Proof of Concept

```rust
// In a gateway stateless-validator test:
fn test_declare_bypasses_max_l2_gas_amount() {
    let config = StatelessTransactionValidatorConfig {
        validate_resource_bounds: true,
        max_l2_gas_amount: 1_000_000,   // gateway ceiling
        ..Default::default()
    };
    let validator = StatelessTransactionValidator { config };

    // Build a Declare V3 tx with max_amount far above the ceiling.
    let tx = declare_tx_with_l2_gas_amount(u64::MAX);

    // Should fail with MaxGasAmountTooHigh — but it passes.
    assert!(validator.validate(&RpcTransaction::Declare(tx)).is_ok());

    // Contrast: an Invoke tx with the same amount is correctly rejected.
    let invoke_tx = invoke_tx_with_l2_gas_amount(u64::MAX);
    assert!(validator.validate(&RpcTransaction::Invoke(invoke_tx)).is_err());
}

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {

            });
        }

    }
