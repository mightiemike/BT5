### Title
`fee_proposal_fri` Excluded from Block Hash Allows Undetected Manipulation of L2 Gas Price During P2P Sync — (`crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

---

### Summary

`fee_proposal_fri` is stored in every post-V0_14_3 block header and is the sole input to `compute_fee_actual`, which sets the floor for the EIP-1559 L2 gas price. However, `fee_proposal_fri` is deliberately excluded from `calculate_block_hash`. A syncing node that receives a block over P2P has no cryptographic way to verify the `fee_proposal_fri` field, because the canonical block hash does not commit to it. A malicious peer can serve blocks with arbitrary `fee_proposal_fri` values, causing the victim node to compute a wrong `fee_actual`, a wrong L2 gas price, and wrong fee estimates.

---

### Finding Description

**The missing commitment.** `calculate_block_hash` in `crates/starknet_api/src/block_hash/block_hash_calculator.rs` hashes: block-hash version, block number, state root, sequencer, timestamp, concatenated counts, state-diff commitment, transaction commitment, event commitment, receipt commitment, `gas_prices_to_hash(l1_gas_price, l1_data_gas_price, l2_gas_price)`, starknet version, a zero sentinel, and the parent hash. [1](#0-0) 

`fee_proposal_fri` is not in this list. The `BlockHeaderWithoutHash` struct carries the field with an explicit TODO:

```rust
// TODO(AndrewL): Add this field into the block hash.
/// Proposer's oracle-derived recommended L2 gas fee. `None` for pre-V0_14_3 blocks.
pub fee_proposal_fri: Option<GasPrice>,
``` [2](#0-1) 

The same field appears in `StorageBlockHeader`: [3](#0-2) 

**Why the field matters.** `fee_proposal_fri` is the only input to `compute_fee_actual`, which computes the median of the last `window_size` proposals and is used as the floor for the EIP-1559 L2 gas price update: [4](#0-3) 

The `fee_proposals_window` is populated at startup by reading `fee_proposal_fri` directly from stored block headers: [5](#0-4) 

**The consensus-only binding is insufficient for sync.** The developers added a `ProposalCommitment` that binds `fee_proposal_fri` at consensus time: [6](#0-5) 

But `ProposalCommitment` is a consensus-internal object. It is never verified by a syncing node. A syncing node verifies only the canonical `BlockHash`, which does not include `fee_proposal_fri`. The code comment confirms this is intentional (for now):

> "This computation lives in the orchestrator (not in `starknet_api`) because the fee_proposal is a consensus-time signal — `starknet_api`'s block-hash primitives remain unaware of it, and `BlockHash` is unaffected." [7](#0-6) 

**Attack path.**

1. Attacker controls one P2P peer.
2. Attacker serves post-V0_14_3 blocks with valid block hashes but with `fee_proposal_fri` values set to an extreme (e.g., `u128::MAX` or `1`).
3. The victim node accepts the blocks — the block hash verifies correctly.
4. The victim stores the manipulated `fee_proposal_fri` values in `StorageBlockHeader`.
5. On startup (or after sync), `initialize_fee_proposals_window` reads these values and populates `fee_proposals_window` with attacker-controlled data.
6. `compute_fee_actual` returns a wrong median → `calculate_next_l2_gas_price_for_fin` uses it as a floor → the node's L2 gas price diverges from the honest network.
7. `starknet_estimateFee` and `starknet_simulateTransactions` return authoritative-looking but wrong fee values.
8. If the victim is a validator, `is_proposal_init_valid` checks `fee_proposal_fri` against a wrong `fee_actual`, causing it to accept out-of-range proposals or reject valid ones. [8](#0-7) 

---

### Impact Explanation

A syncing node that receives manipulated `fee_proposal_fri` values will compute a wrong `fee_actual` and a wrong L2 gas price. Every subsequent call to `starknet_estimateFee` or `starknet_simulateTransactions` will return an authoritative-looking but incorrect fee. If the node is a validator, it will apply wrong bounds when checking incoming proposals, potentially accepting invalid proposals or rejecting valid ones. This matches:

- **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**
- **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing** (via wrong proposal-init validation bounds).

---

### Likelihood Explanation

The attack requires a single malicious P2P peer. Any node that syncs from an attacker-controlled peer is affected. The block hash verification passes unconditionally because `fee_proposal_fri` is not in the hash. No privileged access is required. The window size is a fixed constant, so the attacker only needs to serve `window_size` consecutive blocks with manipulated values to fully control `fee_actual`.

---

### Recommendation

Include `fee_proposal_fri` in `calculate_block_hash` for blocks at or above `StarknetVersion::V0_14_3`, gated by the same `BlockHashVersion` check already used for `gas_prices_to_hash`. This is already tracked as a TODO:

```rust
// TODO(AndrewL): Add this field into the block hash.
```

Until the block hash commits to `fee_proposal_fri`, syncing nodes should cross-validate the field against the `ProposalCommitment` received from consensus peers, or treat `fee_proposal_fri` as untrusted when initializing the `fee_proposals_window` from sync-only storage.

---

### Proof of Concept

```
1. Run a modified peer that serves valid blocks (correct block hash) but with
   fee_proposal_fri = GasPrice(u128::MAX) for window_size consecutive heights.

2. Connect a victim node to this peer via P2P sync.

3. The victim calls initialize_fee_proposals_window(start_height):
   - Reads fee_proposal_fri = u128::MAX for each height in the window.
   - compute_fee_actual returns GasPrice(u128::MAX).

4. calculate_next_l2_gas_price_for_fin uses fee_actual = u128::MAX as the
   effective_min, so the L2 gas price is driven to u128::MAX.

5. starknet_estimateFee on the victim node returns fees orders of magnitude
   higher than on honest nodes — authoritative-looking wrong value.

6. is_proposal_init_valid on the victim rejects every honest proposal because
   fee_proposal_bounds(u128::MAX, margin_ppt) produces bounds that no honest
   proposer can satisfy.
``` [9](#0-8) [10](#0-9)

pub fn compute_fee_actual(
    fee_proposals_window: &BTreeMap<BlockNumber, Option<GasPrice>>,
    height: BlockNumber,
    window_size: u64,
) -> Option<GasPrice> {
    let Some(start) = height.0.checked_sub(window_size) else {
        warn!(
            "Cannot compute fee_actual for height {height}: height is below window_size \
             ({window_size})"
        );
        return None;
    };
    let window_size_usize = usize::try_from(window_size).expect("window_size fits in usize");
    let mut window = Vec::with_capacity(window_size_usize);
    for source_height in (start..height.0).map(BlockNumber) {
        match fee_proposals_window.get(&source_height) {
            Some(Some(price)) => window.push(*price),
            Some(None) | None => {
                warn!(
                    "Cannot compute fee_actual for height {height}: fee_proposals_window has no \
                     recorded fee_proposal for height {source_height}"
                );
                return None;
            }
        }
    }
    window.sort();
    let mid = window_size_usize / 2;
    let median = if window_size_usize.is_multiple_of(2) {
        // Even: average of the two middle values, rounded down.
        // Overflow-safe averaging: a + (b - a) / 2 (safe because sorted, so b >= a).
        GasPrice(window[mid - 1].0 + (window[mid].0 - window[mid - 1].0) / 2)
    } else {
        window[mid]
    };
    Some(median)
}

pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    override_l2_gas_price_fri: Option<u128>,
    min_l2_gas_price_per_height: &[PricePerHeight],
    fee_actual: Option<GasPrice>,
) -> GasPrice {
    if let Some(override_value) = override_l2_gas_price_fri {
        info!(
            "L2 gas price ({}) is not updated, remains on override value of {override_value} fri",
            current_l2_gas_price.0

    let gas_target = VersionedConstants::latest_constants().gas_target;
    let config_min = get_min_gas_price_for_height(height, min_l2_gas_price_per_height);
    let effective_min = match fee_actual {
        Some(fa) => GasPrice(max(config_min.0, fa.0)),
        None => config_min,
    };
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
}
