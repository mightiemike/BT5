### Title
Zero `max_cost` Silently Bypasses All Execution Limits, Enabling Unbounded Computation - (File: `src/run_program.rs`)

### Summary

In `src/run_program.rs`, the `run_program()` function converts a caller-supplied `max_cost` of `0` into `Cost::MAX` (`u64::MAX`), silently removing all cost limits instead of rejecting the program immediately. This is the direct structural analog of H-06: a zero sentinel value that should mean "no budget / reject" instead means "unlimited." Any caller — including the Python `run_with_cost` API — that passes `max_cost = 0` expecting immediate rejection instead receives unbounded program execution.

### Finding Description

In `src/run_program.rs` at line 494:

```rust
pub fn run_program(&mut self, program: NodePtr, env: NodePtr, max_cost: Cost) -> Response {
    // ...
    let max_cost = if max_cost == 0 { Cost::MAX } else { max_cost };
```

When `max_cost == 0` is supplied, the function silently substitutes `u64::MAX` as the effective limit. The program then runs with no cost ceiling whatsoever.

This is directly inconsistent with how the `softfork` operator handles a zero cost argument in the same file:

```rust
if expected_cost == 0 {
    return Err(EvalErr::CostExceeded);
}
```

The `softfork` operator correctly rejects a zero cost as invalid. The top-level `run_program` does the opposite: it promotes zero to unlimited. A caller who reasons by analogy with the softfork behavior — or who uses `0` as a "disabled / no budget" sentinel — will instead receive a program that runs without any cost constraint.

The Python-facing entry point `run_with_cost(args, max_cost: int, flags: int = 0)` in `wheel/python/clvm_rs/program.py` passes `max_cost` directly through to the Rust layer with no guard, so `run_with_cost(args, max_cost=0)` silently enables unlimited execution.

### Impact Explanation

Any caller that passes `max_cost = 0` to `run_program` or `run_serialized_chia_program` expecting the call to fail immediately (or to enforce a zero-cost budget) instead receives a program execution with no cost ceiling. This can cause:

1. **Resource exhaustion**: An attacker-controlled CLVM program passed with `max_cost=0` runs unbounded, consuming arbitrary CPU and allocator memory.
2. **Consensus divergence**: If one node implementation treats `max_cost=0` as "reject" and another (using this library) treats it as "unlimited," the two nodes will disagree on whether a spend bundle is valid — a consensus-critical failure on the Chia blockchain.
3. **API contract violation**: The Python `Program.run_with_cost(args, max_cost=0)` call is syntactically valid and produces no error, but silently removes all safety limits.

### Likelihood Explanation

The Python API `run_with_cost` accepts `max_cost` as a plain `int` with no documented lower bound. A developer who passes `0` to mean "no budget" or "disabled" — a common sentinel pattern — triggers the unlimited path. The inconsistency with the `softfork` operator's explicit rejection of `expected_cost == 0` makes this confusion highly plausible. The attacker-controlled entry path is: supply attacker-crafted CLVM bytes to any caller that invokes `run_serialized_chia_program` or `run_program` with `max_cost=0`.

### Recommendation

Remove the silent `0 → Cost::MAX` promotion. Instead, treat `max_cost == 0` as an explicit rejection (matching the `softfork` operator's behavior), or document and enforce a minimum non-zero value at the API boundary:

```rust
// Before (dangerous):
let max_cost = if max_cost == 0 { Cost::MAX } else { max_cost };

// After (safe):
if max_cost == 0 {
    return Err(EvalErr::CostExceeded);
}
```

If the "no limit" semantic is intentionally needed, expose it through a distinct constant (e.g., `COST_UNLIMITED = u64::MAX`) and require callers to pass it explicitly, rather than overloading `0`.
