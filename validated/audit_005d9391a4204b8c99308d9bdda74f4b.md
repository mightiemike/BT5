### Title
Deprecated `verify_transaction_inclusion` Remains Fully Callable On-Chain, Bypassing the Coinbase Merkle Proof Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof. However, the original `verify_transaction_inclusion` is only marked `#[deprecated]` in Rust — a compile-time annotation with zero runtime effect. It remains a public, unguarded NEAR contract method. Any unprivileged NEAR caller can invoke it directly, bypassing the coinbase proof check entirely and re-exposing the 64-byte Merkle forgery path that `v2` was designed to close.

---

### Finding Description

`verify_transaction_inclusion_v2` (lines 347–369) adds a coinbase Merkle proof check before delegating to `verify_transaction_inclusion`:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [1](#0-0) 

The coinbase check is the sole guard against the 64-byte transaction forgery: it anchors the proof to a known-valid leaf (the coinbase at index 0), preventing an attacker from supplying an internal Merkle node as `tx_id`.

`verify_transaction_inclusion` is declared `pub` with only `#[deprecated]` and `#[pause]`:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [2](#0-1) 

`#[deprecated]` is a Rust compiler annotation. It emits a warning only when Rust source code calls the function. NEAR callers invoke contract methods through the NEAR runtime ABI (Borsh/JSON), where no such warning exists and no runtime gate is applied. The function is therefore indistinguishable from any other public method to any on-chain or off-chain caller.

The function's own doc comment acknowledges the forgery risk:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

The only check inside `verify_transaction_inclusion` that touches the supplied `tx_id` is the final Merkle root comparison:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [4](#0-3) 

No check distinguishes a leaf transaction hash from an internal node hash. An attacker who supplies an internal node as `tx_id` with a matching sibling path will satisfy this comparison and receive `true`.

**Desynchronization root cause (analog to M-11):** The protocol introduced a restriction (coinbase proof) in `verify_transaction_inclusion_v2` to close the forgery path, but left the unrestricted path (`verify_transaction_inclusion`) fully reachable. The restriction is enforced on one entry point and silently absent on the other — the same cross-path desynchronization as the Sentiment delisting bypass.

---

### Impact Explanation

A NEAR DApp consuming `verify_transaction_inclusion` (the deprecated path) receives a `true` result for a fabricated `tx_id` that corresponds to an internal Merkle node, not a real transaction. The DApp cannot distinguish this from a legitimate proof. Depending on the DApp's logic, this can authorize:

- Release of bridged BTC or wrapped assets for a transaction that never occurred.
- Acceptance of a double-spend or replay as a confirmed deposit.
- Any action gated on "this BTC transaction was confirmed."

The corrupted proof result is the exact state value the analog targets: `verify_transaction_inclusion` returns `true` for a forged inclusion claim against a valid main-chain block.

---

### Likelihood Explanation

- `verify_transaction_inclusion` is a public NEAR method with no runtime deprecation gate; any caller reaches it with a standard NEAR function call.
- The 64-byte transaction forgery technique is publicly documented (BitMEX research, referenced in the contract's own comments) and requires no privileged access.
- DApps integrated before `v2` was introduced, or DApps that call the method by name without auditing the deprecation note, are directly exposed.
- No attacker capability beyond constructing a valid Merkle sibling path for an internal node is required.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or replace its body with an unconditional `env::panic_str("use verify_transaction_inclusion_v2")`. Marking a security-critical function `#[deprecated]` without a runtime guard does not close the attack surface for on-chain callers. The coinbase proof check must be the only reachable code path for transaction inclusion verification.

---

### Proof of Concept

1. Identify any main-chain block `B` with known `merkle_root` stored in `headers_pool` and `mainchain_header_to_height`.
2. Locate an internal Merkle node `N` in block `B`'s transaction tree (e.g., the parent of two known leaf hashes). `N` is 32 bytes; a 64-byte transaction whose double-SHA256 equals `N` can be constructed per the BitMEX technique.
3. Build a valid Merkle sibling path from `N` up to `merkle_root`.
4. Call `verify_transaction_inclusion` (not `v2`) with:
   - `tx_id = N`
   - `tx_block_blockhash = B`
   - `tx_index` = index of `N` in the tree
   - `merkle_proof` = sibling path from step 3
   - `confirmations = 1`
5. The function passes the `mainchain_header_to_height` check (block is canonical), passes the `merkle_proof.is_empty()` check, and returns `true` because `compute_root_from_merkle_proof(N, index, path) == merkle_root`.
6. The calling DApp receives `true` for a transaction that does not exist. [5](#0-4) [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
