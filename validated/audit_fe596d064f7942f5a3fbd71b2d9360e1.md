### Title
Missing Bounds Validation in `updateRisk` Allows Inflated Collateral Weight to Enable Undercollateralized Borrowing — (File: `core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine.updateRisk` is missing the absolute bounds checks on `RiskStore` weight fields that exist in `BaseEngine._addOrUpdateProduct`. Because the stored weights have an implicit decimal scaling requirement ("real weight × 1e9") that is not enforced in `updateRisk`, an owner who sets a weight value off by decimal places produces an astronomically inflated X18 weight. An unprivileged user can then open a position and borrow far beyond their collateral, draining the protocol.

---

### Finding Description

`RiskHelper.RiskStore` documents its weight fields as:

> "these weights are all between 0 and 2 / these integers are the real weights times 1e9" [1](#0-0) 

When `_risk(uint32 productId)` converts stored values to X18, it multiplies each field by `1e9`: [2](#0-1) 

`_addOrUpdateProduct` enforces the absolute bounds on these stored values — long weights must be ≤ `10**9` (≤ 1.0 real) and short weights must be ≥ `10**9` (≥ 1.0 real): [3](#0-2) 

`updateRisk`, however, only checks the **ordering** relationship between initial and maintenance weights. The absolute bounds (`longWeightMaintenance <= 10**9` and `shortWeightMaintenance >= 10**9`) are entirely absent: [4](#0-3) 

This means an owner can call `updateRisk` with `longWeightMaintenance = 15e8` (intending 0.9 but off by one decimal place, or confusing the scale). The stored value `15e8` passes the ordering check (`longWeightInitial <= longWeightMaintenance`), but when converted to X18 it becomes `15e8 × 1e9 = 1.5e18`, giving a 150% collateral factor for long positions.

The health calculation in `_calculateProductHealth` then computes: [5](#0-4) 

The only guard against an inflated weight is the exact equality check `weight == 2 * ONE` (i.e., `weight == 2e18`). Any value strictly between `1e18` and `2e18` — which is exactly the range enabled by the missing bounds check — bypasses this guard entirely and silently inflates health. [6](#0-5) 

---

### Impact Explanation

A user who holds a long position in a product whose `longWeightMaintenance` was misconfigured to `15e8` (stored) / `1.5e18` (X18) will have their health computed as `amount × 1.5 × price` instead of the intended `amount × 0.9 × price`. This allows the user to borrow up to 1.5× the value of their collateral. The excess borrowed quote is real USDC (or the protocol's quote token) that is transferred out of the `SpotEngine`'s pool, leaving a permanent shortfall that is socialized across all depositors.

The `spotUpdateRisk` and `perpUpdateRisk` entry points in `ContractOwner` both route to `updateRisk` with no additional validation: [7](#0-6) 

---

### Likelihood Explanation

The discrepancy between `_addOrUpdateProduct` (full validation) and `updateRisk` (ordering-only validation) creates a silent trap. An operator updating risk parameters post-deployment has no on-chain signal that `updateRisk` enforces weaker invariants than `addOrUpdateProduct`. The comment "real weights times 1e9" is the only specification, and a single decimal-place error (e.g., `9e8` vs `9e9`) produces a stored value that passes `updateRisk`'s check while violating the protocol's invariant. This is the same class of mistake described in the reference report (off by 19 decimal places in FPI context; here off by 1 decimal place in the 1e9 scale context).

---

### Recommendation

Add the same absolute bounds checks to `updateRisk` that exist in `_addOrUpdateProduct`:

```solidity
require(
    riskStore.longWeightInitial <= riskStore.longWeightMaintenance &&
        riskStore.longWeightMaintenance <= 10**9 &&
        riskStore.shortWeightInitial >= riskStore.shortWeightMaintenance &&
        riskStore.shortWeightMaintenance >= 10**9,
    ERR_BAD_PRODUCT_CONFIG
);
```

Additionally, add NatSpec to `updateRisk` explicitly documenting that all weight fields must be expressed as "real weight × 1e9" (i.e., a value of `9e8` represents a 90% collateral factor).

---

### Proof of Concept

1. Owner calls `BaseEngine.updateRisk(productId, riskStore)` where `riskStore.longWeightInitial = 9e8`, `riskStore.longWeightMaintenance = 15e8` (intending 1.5 but the operator confused the scale, or intended 0.9 but typed `15e8` instead of `9e8`).
2. The `require` in `updateRisk` passes: `9e8 <= 15e8` ✓. The missing check `15e8 <= 10**9` is never evaluated.
3. `_risk(productId)` returns `longWeightMaintenanceX18 = 15e8 × 1e9 = 1.5e18`.
4. Attacker deposits 100 units of the token (worth 100 USDC at `priceX18 = 1e18`).
5. `_calculateProductHealth` computes: `health = 100e18 × 1.5e18 / 1e18 × 1e18 / 1e18 = 150e18` (150 USDC health).
6. Attacker borrows 150 USDC quote. `_isAboveInitial` passes because health = 150 > 0.
7. Attacker withdraws 150 USDC, having deposited only 100 USDC worth of collateral.
8. The 50 USDC shortfall is absorbed by the `SpotEngine`'s depositor pool via `socializeSubaccount` when the position is eventually liquidated.
