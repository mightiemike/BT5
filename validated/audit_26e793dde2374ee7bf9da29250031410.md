Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Unauthorized Depositors to Bypass the Allowlist Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual token-providing caller) and gates only on `owner` (the LP-position recipient). Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender != owner` with no equality constraint, any address can bypass the deposit allowlist by naming an already-allowlisted address as `owner`.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as distinct arguments to `_beforeAddLiquidity`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` forwards both to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first parameter but leaves it unnamed and never reads it, checking only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The operator pattern is explicitly documented and supported: the NatSpec on `IMetricOmmPoolActions.addLiquidity` states "`msg.sender` pays but need not equal `owner`." Token provision occurs via `IMetricOmmModifyLiquidityCallback` called on `msg.sender` inside `LiquidityLib.addLiquidity` at L147, meaning the unauthorized `sender` is the one actually transferring tokens into the pool. The guard never evaluates `allowedDepositor[pool][sender]`, so the unauthorized actor's address is never checked.

This is structurally inconsistent with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (the actual swap initiator) and ignores `recipient`.

## Impact Explanation
A pool operator deploys a pool with `DepositAllowlistExtension` to enforce that only KYC-approved or otherwise authorized addresses can supply liquidity. The allowlist invariant — that only approved addresses can deposit capital — is broken. An unauthorized address can supply tokens into the restricted pool and receive (or control) the resulting LP position. This constitutes a broken core pool functionality causing unauthorized capital entry into a permissioned pool, directly violating the access-control invariant the extension is designed to enforce. The wrong value is `allowedDepositor[pool][sender]`, which is never evaluated; the guard passes on `allowedDepositor[pool][owner]` for an address the attacker controls or names.

## Likelihood Explanation
- Any pool deploying `DepositAllowlistExtension` with a non-trivial allowlist is affected.
- The attacker needs only one address on the allowlist as `owner` (e.g., their own previously approved second wallet, or any allowlisted address they can name as recipient).
- No special permissions, flash loans, or oracle manipulation are required — a single direct call to `pool.addLiquidity(allowlistedOwner, ...)` suffices.
- The bypass is reachable by any unprivileged external caller implementing `IMetricOmmModifyLiquidityCallback`.
- The `MetricOmmPoolLiquidityAdder` periphery contract explicitly supports `owner != msg.sender` via `addLiquidityExactShares(pool, owner, ...)`, making this path trivially accessible.

## Recommendation
Replace the unnamed first parameter with `sender` and gate on it, consistent with `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  Pool P configured with DepositAllowlistExtension E
  E.allowedDepositor[P][alice] = true   // alice is authorized
  E.allowedDepositor[P][bob]  = false   // bob is NOT authorized
  bob controls alice (e.g., alice is bob's second EOA, or bob simply names any allowlisted address)

Attack (direct call):
  bob deploys a contract implementing IMetricOmmModifyLiquidityCallback
  bob calls P.addLiquidity(alice /*owner*/, salt, deltas, callbackData, extensionData)
    → pool calls E.beforeAddLiquidity(bob /*sender — ignored*/, alice /*owner — checked*/, ...)
    → E checks allowedDepositor[P][alice] == true  ✓  (bob's address never evaluated)
    → guard passes
    → pool calls bob's callback to collect tokens
    → LP shares credited to alice
    → bob (controlling alice) holds LP position funded by unauthorized capital

Attack (via MetricOmmPoolLiquidityAdder):
  bob calls helper.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "")
    → helper calls pool.addLiquidity(alice, ...) with msg.sender = helper
    → pool calls E.beforeAddLiquidity(helper /*sender*/, alice /*owner*/)
    → same bypass: allowedDepositor[P][alice] == true passes
    → bob's tokens (approved to helper) are pulled; LP credited to alice
