### Title
`tokens_burnt` in `ExecutionOutcome` Captures Pre-Reward Value, Overstating Burnt Tokens Committed to Outcome Merkle Tree — (File: `runtime/runtime/src/lib.rs`)

---

### Summary

In `apply_action_receipt`, the `tokens_burnt` snapshot used in `ExecutionOutcome` is taken **before** the contract's `receiver_reward` (30% of wasm-burnt gas) is subtracted from `tx_burnt_amount`. The post-subtraction value correctly feeds `stats.balance.tx_burnt_amount` → `prev_balance_burnt` → `total_supply`, but the pre-subtraction value is Borsh-hashed into the outcome Merkle leaf and committed to the block header's `prev_outcome_root`. Every function-call receipt therefore commits an inflated `tokens_burnt` to the outcome Merkle tree.

---

### Finding Description

In `apply_action_receipt` (`runtime/runtime/src/lib.rs`), `tx_burnt_amount` is built up across lines 960–969 (gas burn cost, refund penalty, create-account charge, action-level tokens burnt). At line 972 it is snapshotted:

```rust
// The amount of tokens burnt for the execution of this receipt. It's used in the execution
// outcome.
let tokens_burnt = tx_burnt_amount;   // ← snapshot taken HERE
``` [1](#0-0) 

Immediately after, `receiver_reward` (30% of `gas_burnt_for_function_call × gas_price`) is computed and subtracted from `tx_burnt_amount`:

```rust
tx_burnt_amount = tx_burnt_amount.checked_sub(receiver_reward).unwrap();
account.set_amount(safe_add_balance(account.amount(), receiver_reward)?);
``` [2](#0-1) 

The post-subtraction `tx_burnt_amount` is correctly accumulated into `stats.balance.tx_burnt_amount`: [3](#0-2) 

But the pre-subtraction `tokens_burnt` is placed into the `ExecutionOutcome`: [4](#0-3) 

`tokens_burnt` is a field of `PartialExecutionOutcome`, which is Borsh-hashed in `to_hashes()` and included in the outcome Merkle tree: [5](#0-4) 

`PartialExecutionOutcome` includes `tokens_burnt` directly: [6](#0-5) 

The outcome Merkle root is committed to in the block header's `prev_outcome_root`. The `total_supply` field in the block header is computed from `prev_balance_burnt` (which uses the correct post-subtraction value), creating a divergence: the outcome Merkle tree commits an inflated `tokens_burnt`, while `total_supply` is correct.

A test helper in the codebase documents the intended semantics explicitly — "minus the function-call reward that is paid back to the contract" — but the actual sum of `tokens_burnt` across outcomes includes the reward: [7](#0-6) 

---

### Impact Explanation

- **Outcome Merkle tree divergence**: Every function-call receipt where the contract account exists commits a `tokens_burnt` value inflated by `receiver_reward` (up to 30% of wasm gas burnt × gas price). This incorrect value is Borsh-hashed into the outcome Merkle leaf and committed to the block header's `prev_outcome_root`.
- **Light-client / bridge inconsistency**: Any light client or bridge that verifies an outcome Merkle proof and reads `tokens_burnt` to track token supply will see an overstated value. The block header's `total_supply` (computed from `prev_balance_burnt`) is correct, so the two protocol-level fields are semantically inconsistent.
- **RPC consumers**: The `ExecutionOutcomeView.tokens_burnt` field exposed via JSON-RPC is documented as "the amount of tokens burnt" but systematically overstates the actual burnt amount by `receiver_reward`.

---

### Likelihood Explanation

This affects every function-call receipt where the receiving contract account exists and `gas_burnt_for_function_call > 0` — the normal case for any call to a deployed contract. The error magnitude is `receiver_reward = floor(gas_burnt_for_function_call × 3/10) × gas_price`, which for a 300 Tgas wasm execution at minimum gas price is on the order of tens of microNEAR per receipt, and scales with execution cost.

---

### Recommendation

Move the `tokens_burnt` snapshot to **after** the `receiver_reward` subtraction:

```rust
if receiver_reward > Balance::ZERO {
    if let Some(ref mut account) = account {
        tx_burnt_amount = tx_burnt_amount.checked_sub(receiver_reward).unwrap();
        account.set_amount(...);
        set_account(...);
        state_update.commit(...);
    }
}

// Capture AFTER receiver_reward has been subtracted
let tokens_burnt = tx_burnt_amount;
```

This aligns `tokens_burnt` in the `ExecutionOutcome` with the value accumulated into `stats.balance.tx_burnt_amount` and with the documented semantics ("tokens actually removed from circulation").

---

### Proof of Concept

1. Submit a `FunctionCall` transaction to any deployed contract that burns wasm gas (e.g., a simple loop).
2. Retrieve the receipt's `ExecutionOutcomeView` via `EXPERIMENTAL_tx_status` or `tx`.
3. Read `outcome.tokens_burnt` from the RPC response.
4. Independently compute `actual_burnt = gas_burnt × gas_price` (for a simple transfer-only receipt with no wasm, these match; for a function call they diverge).
5. The divergence equals `receiver_reward = floor(gas_burnt_for_function_call × 0.3) × gas_price`.

Concretely: a function call burning 100 Tgas of wasm at gas price 1 yoctoNEAR/gas yields `receiver_reward = 30 × 10¹² yoctoNEAR`. The committed `tokens_burnt` in the outcome Merkle leaf will be 30 microNEAR higher than the tokens actually removed from circulation, while `block_header.total_supply` correctly reflects the lower burnt amount.

/// Sum of tokens burnt across the transaction and all its receipt outcomes - the fees the

/// function-call reward that is paid back to the contract).
fn total_tokens_burnt(outcome: &FinalExecutionOutcomeView) -> Balance {
    let mut sum = outcome.transaction_outcome.outcome.tokens_burnt;
    for receipt in &outcome.receipts_outcome {
        sum = sum.checked_add(receipt.outcome.tokens_burnt).unwrap();
    }
