### Title
Two-Hop Deposit Transfer Breaks for Fee-on-Transfer Tokens — (`core/contracts/EndpointStorage.sol`)

---

### Summary

`handleDepositTransfer` in `EndpointStorage.sol` routes a deposit through two sequential `safeTransfer` calls using the same nominal `amount` for both legs. If a supported token charges a transfer fee (a mechanism present in tokens like USDT, currently set to 0 but activatable by the token admin), the Endpoint receives `amount - fee` in the first leg but unconditionally attempts to forward the full `amount` in the second leg, causing an irreversible revert and permanently breaking deposits for that token.

---

### Finding Description

`handleDepositTransfer` performs a two-hop transfer:

```
user ──(amount)──► Endpoint ──(amount)──► Clearinghouse
``` [1](#0-0) 

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);                    // leg 1: user → Endpoint
    safeTransferTo(token, address(clearinghouse), amount);    // leg 2: Endpoint → Clearinghouse
}
```

For a fee-on-transfer token, leg 1 delivers `amount - fee` to the Endpoint. Leg 2 then calls `token.transfer(clearinghouse, amount)` — but the Endpoint only holds `amount - fee`. The `safeTransfer` wrapper in `ERC20Helper` enforces a hard `require` on the return value: [2](#0-1) 

The token's `transfer` either returns `false` or reverts outright because the Endpoint's balance is insufficient, causing `ERR_TRANSFER_FAILED` to propagate up through `depositCollateral` / `depositCollateralWithReferral`. [3](#0-2) 

The same two-hop pattern is also triggered via `DirectDepositV1.creditDeposit`, which calls `depositCollateralWithReferral` with the full `balance` it holds: [4](#0-3) 

---

### Impact Explanation

Every deposit of the affected token reverts at leg 2, making it impossible for any user to deposit that asset. This breaks the core collateral-deposit functionality of the protocol for the affected product. Because the revert happens atomically, no funds are permanently lost — but the deposit path is completely blocked for as long as the token fee is non-zero.

---

### Likelihood Explanation

USDT (Tether) is the canonical example: its contract contains a `basisPointsRate` and `maximumFee` storage variables that are currently set to 0 but are settable by the USDT owner at any time. If USDT is registered as a spot product in Nado and the USDT owner activates the fee, all USDT deposits immediately start reverting. This is structurally identical to the BUIDL `minUSTokens` scenario: the restriction is dormant today but is a live, admin-controlled switch on the token contract itself.

---

### Recommendation

Measure the actual received amount by snapshotting the Endpoint's balance before and after leg 1, then forward only what was actually received:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(this));
    safeTransferFrom(token, from, amount);
    uint256 received = token.balanceOf(address(this)) - before;
    safeTransferTo(token, address(clearinghouse), received);
}
```

This mirrors the standard "balance-delta" pattern used to handle fee-on-transfer tokens safely.

---

### Proof of Concept

1. USDT is registered as a spot product in Nado (productId `X`).
2. The USDT owner calls `setParams(basisPointsRate=10, maximumFee=...)`, activating a 0.1% transfer fee.
3. A user calls `Endpoint.depositCollateral(subaccountName, X, 1_000_000e6)`.
4. `handleDepositTransfer` is entered with `amount = 1_000_000e6`.
5. Leg 1 (`safeTransferFrom`): USDT deducts 1,000 USDT as fee; Endpoint receives `999_000e6`.
6. Leg 2 (`safeTransferTo`): Endpoint calls `USDT.transfer(clearinghouse, 1_000_000e6)`. Endpoint balance is only `999_000e6` → USDT returns `false`.
7. `ERC20Helper.safeTransfer` hits `require(success && ...)` → reverts with `ERR_TRANSFER_FAILED`.
8. All USDT deposits are now permanently broken until the fee is deactivated. [1](#0-0) [2](#0-1)

    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
