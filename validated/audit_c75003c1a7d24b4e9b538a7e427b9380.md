### Title
Unsafe `uint256`→`uint128` Silent Truncation in `DirectDepositV1.creditDeposit()` Corrupts Subaccount Collateral Accounting — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` reads a token's full `uint256` balance and silently truncates it to `uint128` before passing it to `Endpoint.depositCollateralWithReferral()`. If the balance exceeds `type(uint128).max` (~3.4 × 10^38), the subaccount is credited with a drastically smaller (wrapped) collateral amount while the full token quantity remains in the deposit contract, permanently corrupting the subaccount's on-chain collateral record.

---

### Finding Description

`DirectDepositV1.creditDeposit()` is an externally callable function with no access control. It iterates over all registered spot products, reads the contract's token balance as `uint256`, and passes it directly to the endpoint after a bare `uint128(balance)` cast:

```solidity
uint256 balance = token.balanceOf(address(this));
if (balance != 0) {
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),   // ← silent truncation, no bounds check
        "-1"
    );
}
``` [1](#0-0) 

The `uint128(balance)` cast silently discards the upper 128 bits of `balance`. There is no `require(balance <= type(uint128).max)` guard here, unlike the explicit guards present elsewhere in the protocol (e.g., `require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW)` in `Clearinghouse.depositCollateral`). [2](#0-1) 

The truncated value is then forwarded to `Endpoint.depositCollateralWithReferral`, which calls `handleDepositTransfer` with `uint256(amount)` — transferring only the truncated token count from the deposit contract — and enqueues a slow-mode `DepositCollateral` transaction crediting the subaccount with the truncated amount. [3](#0-2) 

A second unguarded truncation exists in `Clearinghouse._balanceOf`, which returns `uint128(IERC20Base(token).balanceOf(address(this)))` with no overflow check, potentially corrupting any internal accounting that depends on it. [4](#0-3) 

---

### Impact Explanation

If `balance > type(uint128).max`, the subaccount receives credit for `balance mod 2^128` tokens instead of the actual deposited amount. The remaining tokens (`balance - uint128(balance)`) are not transferred out of the `DirectDepositV1` contract in that call, but the subaccount's collateral record is permanently set to the wrong (much smaller) value. This is a direct accounting corruption: the subaccount's health, borrowing capacity, and liquidation threshold are all computed from the corrupted balance, enabling undercollateralized positions or blocking legitimate withdrawals.

---

### Likelihood Explanation

`type(uint128).max` ≈ 3.4 × 10^38. For tokens with 18 decimals this requires a nominal balance of ~3.4 × 10^20 tokens, which is unrealistic for most assets. However, for tokens with 0–6 decimals (e.g., USDC with 6 decimals, where 1 unit = 10^-6 tokens), the threshold drops to ~3.4 × 10^32 raw units — still large, but the code path is structurally unguarded. The function is callable by any unprivileged address with no preconditions, making the attack surface permanently open. The risk is low-likelihood but high-impact when triggered.

---

### Recommendation

Add an explicit bounds check before the cast, consistent with the pattern already used elsewhere in the protocol:

```solidity
require(balance <= type(uint128).max, "balance exceeds uint128");
endpoint.depositCollateralWithReferral(
    subaccount,
    productId,
    uint128(balance),
    "-1"
);
```

Apply the same fix to `Clearinghouse._balanceOf`:

```solidity
function _balanceOf(address token) internal view virtual returns (uint128) {
    uint256 bal = IERC20Base(token).balanceOf(address(this));
    require(bal <= type(uint128).max, ERR_CONVERSION_OVERFLOW);
    return uint128(bal);
}
```

---

### Proof of Concept

1. Deploy a token with 0 decimals and mint `2^128 + 1` units to a `DirectDepositV1` deposit address.
2. Call `DirectDepositV1.creditDeposit()` (no access control, callable by anyone).
3. `balance = 2^128 + 1`; `uint128(balance) = 1`.
4. The endpoint transfers only `1` token from the deposit contract and enqueues a `DepositCollateral` slow-mode tx crediting the subaccount with `1` unit.
5. The subaccount's collateral is recorded as `1` instead of `2^128 + 1`, corrupting its health and all downstream risk calculations. [5](#0-4)
