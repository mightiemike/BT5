Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual calling address) and gates access on `owner`, which is a free caller-supplied parameter with no allowlist validation. Any non-allowlisted address can add liquidity to an allowlist-protected pool by nominating any already-allowlisted address as `owner`, completely bypassing the pool admin's access control.

## Finding Description
`MetricOmmPool.addLiquidity` invokes the extension hook as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

where `msg.sender` is the actual caller and `owner` is the LP-position recipient passed in by the caller. The extension receives both but discards `sender`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

The guard checks `allowedDepositor[pool][owner]`, not `allowedDepositor[pool][sender]`. In `MetricOmmPoolLiquidityAdder.addLiquidityExactShares`, `owner` is accepted directly from the caller with only a zero-address check:

```solidity
_validateOwner(owner);   // only checks owner != address(0)
``` [3](#0-2) [4](#0-3) 

An attacker (bob) who is not allowlisted sets `owner` to any allowlisted address (alice). The extension sees `allowedDepositor[pool][alice] == true`, the guard passes, and bob's tokens are pulled via the callback (`payer = msg.sender` of the LiquidityAdder call). The LP position is minted to alice, but bob has successfully deposited into a restricted pool.

The same bypass works when calling `pool.addLiquidity` directly, since the pool passes `msg.sender` as `sender` and the attacker-supplied value as `owner`; the extension only inspects `owner`.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender`:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
``` [5](#0-4) 

The asymmetry between the two extensions confirms the deposit guard has the wrong field.

## Impact Explanation
The `DepositAllowlistExtension` is the sole on-chain mechanism by which a pool admin restricts who may add liquidity. Bypassing it constitutes an admin-boundary break: an unprivileged path circumvents an admin-configured access control. Any non-allowlisted address can add liquidity to bins of their choice, altering the pool's liquidity distribution and effective depth at any price level, and can interact with pools intentionally restricted to vetted LPs (e.g., KYC-gated pools), violating the admin's access-control intent.

## Likelihood Explanation
The bypass requires no special permissions, no flash loan, and no privileged role. The attacker only needs to know one allowlisted address, which is trivially discoverable from on-chain events or the public `allowedDepositor` mapping. The attack is a single transaction through the existing `MetricOmmPoolLiquidityAdder` router. Likelihood is high.

## Recommendation
Replace the `owner` check with a `sender` check in `beforeAddLiquidity`, mirroring the correct pattern in `SwapAllowlistExtension.beforeSwap`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

## Proof of Concept
```solidity
// Setup: pool has DepositAllowlistExtension; only `alice` is allowlisted.
// Attacker (bob) is NOT allowlisted.

// Bob calls the LiquidityAdder, nominating alice as owner.
liquidityAdder.addLiquidityExactShares(
    pool,
    alice,      // owner — allowlisted, passes the guard
    0,          // salt
    deltas,
    maxAmount0,
    maxAmount1,
    ""
);
// Extension checks allowedDepositor[pool][alice] == true → no revert.
// Bob's tokens are pulled via callback (payer = bob).
// LP position minted to alice.
// Bob has successfully added liquidity despite not being allowlisted.
```

The pool's `addLiquidity` passes `msg.sender` (the LiquidityAdder contract, acting as router for bob) as `sender` and `alice` as `owner` to the extension hook. The extension ignores `sender` and only validates `owner`, so the guard passes unconditionally for any caller who knows an allowlisted address.
