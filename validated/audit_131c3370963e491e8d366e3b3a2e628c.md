### Title
Child `ChunkExtra` Inherits Parent's Non-`None` `proposed_split` Across Resharding Boundary ŌĆö (`File: chain/chain/src/resharding/manager.rs`)

---

### Summary

When a shard is split during dynamic resharding, `process_memtrie_resharding_storage_update()` constructs each child's initial `ChunkExtra` by cloning the parent's `ChunkExtra` and only overwriting `state_root` and `congestion_info`. The `proposed_split` field ŌĆö which encodes a pending trie-split proposal for the *next* epoch boundary ŌĆö is silently inherited from the parent. If the parent's last chunk had a non-`None` `proposed_split` (i.e., the parent was itself near a memory threshold and proposed a split), both child shards start epoch N+1 with a stale, parent-scoped `proposed_split` value in their `ChunkExtra`. The first chunk header produced for each child shard will carry `proposed_split = None` (because `compute_proposed_split` returns `None` for a freshly-split child that is below threshold or inside the resharding cooldown), causing `validate_chunk_with_chunk_extra_and_receipts_root` to reject it with `InvalidChunkHeaderShardSplit`. This is a protocol-level deadlock: the child shards cannot produce any valid chunk until the stale field is corrected, which has no in-protocol recovery path.

---

### Finding Description

**Root cause ŌĆö partial field update in `process_memtrie_resharding_storage_update`:**

```rust
// chain/chain/src/resharding/manager.rs:258-260
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
```

The code clones the parent's `ChunkExtra` (a `ChunkExtraV5`) and updates only `state_root` and `congestion_info`. The `proposed_split: Option<TrieSplit>` field is **not reset to `None`**. The existing TODO comment at line 255 acknowledges this incompleteness:

```
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
```

**The invariant that is broken:**

`validate_chunk_with_chunk_extra_and_receipts_root` enforces that the `proposed_split` in an incoming chunk header exactly matches the `proposed_split` stored in the previous `ChunkExtra`:

```rust
// chain/chain/src/validate.rs:176-185
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
```

For the first chunk of a child shard in epoch N+1, `prev_chunk_extra` is the child's initial `ChunkExtra` written during resharding. If the parent had `proposed_split = Some(TrieSplit { boundary_account: "foo", ... })`, the child inherits that value. But `compute_proposed_split` for the child returns `None` (the child is a new shard, below threshold, and inside the resharding cooldown enforced by `can_reshard`). The chunk producer therefore emits a header with `proposed_split = None`, which mismatches the stored `Some(...)`, triggering `InvalidChunkHeaderShardSplit`.

**The trigger condition:**

The parent shard must have had `proposed_split = Some(...)` in its final `ChunkExtra` at the resharding block. This happens when:
1. The parent shard's memory usage exceeded `memory_usage_threshold` near the end of epoch N (so `compute_proposed_split` returned `Some`), AND
2. The parent shard was simultaneously selected for splitting (its `proposed_split` was picked by `get_upcoming_shard_split` and embedded in the block header as `shard_split`).

Both conditions can co-occur: the parent is large enough to be selected for splitting *because* it is large, and that same size causes `compute_proposed_split` to return `Some` for the parent's final chunk. The `proposed_split` in the parent's last `ChunkExtra` is then the exact value that gets cloned into both children.

**The analog to the external bug:**

The external bug: `adminAccountMigration()` updates `auction.seller` but not `buyPrice.seller`, leaving two coupled state fields out of sync. The nearcore analog: `process_memtrie_resharding_storage_update()` updates `state_root` and `congestion_info` in the child `ChunkExtra` but not `proposed_split`, leaving the child's stored `ChunkExtra` out of sync with what `compute_proposed_split` will produce for the child's first chunk header.

---

### Impact Explanation

**Severity: High / Critical**

- Both child shards of a split parent are simultaneously affected.
- The first chunk header produced for each child shard fails `InvalidChunkHeaderShardSplit` validation on every honest node.
- No in-protocol recovery path exists: the stale `proposed_split` is stored in `DBCol::ChunkExtra` and is the authoritative value used by the validator. There is no mechanism to overwrite it without a node-level intervention.
- The network cannot finalize any block that includes a chunk from either child shard, effectively halting all shards that depend on those children.
- The condition is reachable in production whenever a shard that is already near the memory threshold is selected for splitting ŌĆö precisely the scenario dynamic resharding is designed to handle.

---

### Likelihood Explanation

Dynamic resharding (`ProtocolFeature::DynamicResharding`, protocol version 85) is a nightly/production feature. The trigger requires the parent shard to be above `memory_usage_threshold` at the end of the epoch in which it is split. Since the split is *triggered by* the shard exceeding the threshold, the parent's final chunk will almost always have `proposed_split = Some(...)` ŌĆö the same condition that caused the split to be scheduled. The two conditions are therefore strongly correlated, making this a near-certain failure on the first dynamic resharding event in production.

---

### Recommendation

In `process_memtrie_resharding_storage_update`, after cloning the parent `ChunkExtra` and before saving it for each child, explicitly reset `proposed_split` to `None`:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// Reset proposed_split: child shards start fresh; the parent's split
// proposal is scoped to the parent's shard ID and epoch, which no
// longer exist after resharding.
*child_chunk_extra.proposed_split_mut() = None;
```

This requires adding a `proposed_split_mut()` accessor to `ChunkExtra` (mirroring the existing `state_root_mut()` and `congestion_info_mut()` pattern). The fix is minimal, targeted, and directly resolves the invariant violation caught by `validate_chunk_with_chunk_extra_and_receipts_root`.

### Proof of Concept

**Step 1 ŌĆö Arrange:** Configure a single-shard network with `DynamicResharding` enabled and set `memory_usage_threshold` low enough that the shard's `proposed_split` is non-`None` at the end of epoch N. Also set `force_split_shards` to include that shard so it is selected for splitting.

**Step 2 ŌĆö Trigger:** Advance to the last block of epoch N. `compute_proposed_split` returns `Some(TrieSplit { boundary_account: "mid", ... })` for the parent shard. This value is stored in `ChunkExtra` for the parent at the resharding block. Simultaneously, `get_upcoming_shard_split` picks the same shard and embeds `shard_split = Some(...)` in the block header.

**Step 3 ŌĆö Resharding:** At the epoch N ŌåÆ N+1 boundary, `process_memtrie_resharding_storage_update` clones the parent's `ChunkExtra` into both children. Both children's `ChunkExtra` now contain `proposed_split = Some(TrieSplit { boundary_account: "mid", ... })`.

**Step 4 ŌĆö First chunk of epoch N+1:** The chunk producer for each child shard calls `compute_proposed_split`. The resharding cooldown (`can_reshard` checks `last_resharding`) returns `false`, so `compute_proposed_split` returns `None`. The chunk header is produced with `proposed_split = None`.

**Step 5 ŌĆö Validation failure:** `validate_chunk_with_chunk_extra_and_receipts_root` compares `prev_chunk_extra.proposed_split()` (`Some(TrieSplit { ... })`) against `chunk_header.proposed_split()` (`None`). They differ. `InvalidChunkHeaderShardSplit` is returned. The chunk is rejected by all validators. The child shards are permanently stalled. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    ///
    /// Returns `true` if no resharding occurred in the last `min_epochs_between_resharding`
    /// epochs (including the next one).
    fn can_reshard(
        &self,
        block_hash: &CryptoHash,
        min_epochs_between_resharding: NonZeroEpochHeight,
    ) -> Result<bool, EpochError> {
        let block_info = self.get_block_info(block_hash)?;
        let next_epoch_id = self.get_next_epoch_id_from_info(&block_info)?;
        let next_epoch_info = self.get_epoch_info(&next_epoch_id)?;

        // last_resharding() returns `None` if no resharding happened since dynamic resharding
        // has been enabled. It is theoretically possible that a static resharding was scheduled
        // right before enabling dynamic resharding, but we assume this didn't happen.
        let can_reshard = next_epoch_info.last_resharding().is_none_or(|last_resharding| {
            next_epoch_info.epoch_height() - last_resharding >= min_epochs_between_resharding.get()
        });
        Ok(can_reshard)
    }
