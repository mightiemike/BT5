### Title
`WithdrawCollateralV2` Missing Handler in `processSlowModeTransactionImpl` Breaks Censorship-Resistance Escape Hatch — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

`WithdrawCollateralV2` is fully defined in the `TransactionType` enum and handled in the sequencer fast path (`processTransactionImpl`), but has no handler in `processSlowModeTransactionImpl`. Because `submitSlowModeTransactionImpl` does not block this type either, a user can pay the slow mode fee and queue the transaction — but when it executes, it silently reverts. The slow mode censorship-resistance mechanism is permanently broken for `WithdrawCollateralV2`.

---

### Finding Description

The `TransactionType` enum in `IEndpoint.sol` defines `WithdrawCollateralV2` (index 32) with a full struct (`WithdrawCollateralV2`, `SignedWithdrawCollateralV2`) that adds a `sendTo` field for withdrawing to a custom address. [1](#0-0) [2](#0-1) 

The sequencer fast path handles it correctly in `processTransactionImpl`: [3](#0-2) 

However, `processSlowModeTransactionImpl` — the function that executes queued slow mode transactions — has no branch for `WithdrawCollateralV2`. It handles `WithdrawCollateral` (V1) but not V2, and falls through to `else { revert(); }`: [4](#0-3) [5](#0-4) 

Meanwhile, `submitSlowModeTransactionImpl` does **not** block `WithdrawCollateralV2` — it is absent from the explicit branch list, so it falls into the `else` path that charges the slow mode fee and queues the transaction: [6](#0-5) 

When the queued transaction is later executed via `_executeSlowModeTransaction`, the revert from `processSlowModeTransactionImpl` is silently swallowed by the `try/catch`: [7](#0-6) 

The slow mode transaction is consumed (deleted from the queue at line 194) but never actually executed. [8](#0-7) 

---

### Impact Explanation

The slow mode path is the protocol's censorship-resistance mechanism: if the sequencer refuses to include a user's transaction, the user can submit it on-chain, wait 3 days, and force execution. For `WithdrawCollateralV2` — the only withdrawal type that supports a custom `sendTo` address — this escape hatch is completely non-functional. A user facing sequencer censorship on a `WithdrawCollateralV2` withdrawal:

1. Pays the slow mode fee (lost permanently).
2. Waits 3 days.
3. The transaction silently fails on execution.
4. Has no further recourse for withdrawing to a custom address on-chain.

The user's funds remain locked in the protocol with no censorship-resistant exit path for the `sendTo` feature. The `WithdrawCollateral` (V1) slow mode path still works, but V1 always sends to `address(uint160(bytes20(sender)))` — it cannot replicate the `sendTo` behavior of V2.

---

### Likelihood Explanation

Any user who:
- Wants to withdraw to an address different from their subaccount owner address, **and**
- Faces sequencer censorship or sequencer downtime

will trigger this. The `WithdrawCollateralV2` type is the current preferred withdrawal path (it is also supported in `BaseWithdrawPool.resolveFastWithdrawal`), making it the natural choice for users with custom `sendTo` requirements. [9](#0-8) 

---

### Recommendation

Add a `WithdrawCollateralV2` branch to `processSlowModeTransactionImpl` in `core/contracts/EndpointTx.sol`, mirroring the logic already present in `processTransactionImpl` (lines 437–465). The slow mode version should decode `SignedWithdrawCollateralV2`, validate the sender, and call `clearinghouse.withdrawCollateral` with the `sendTo` field. The compact-signature validation and `feeX18` cap check from the fast path should also be preserved.

---

### Proof of Concept

**Exact execution path:**

1. User calls `Endpoint.submitSlowModeTransaction(abi.encodePacked(uint8(32), abi.encode(signedWithdrawV2)))`.
2. `submitSlowModeTransactionImpl` is invoked via `delegatecall`. `WithdrawCollateralV2` (type 32) is not in any explicit branch → falls to `else` → slow mode fee charged, tx queued. [10](#0-9) 
3. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`.
4. `_executeSlowModeTransaction` deletes the queued tx and calls `this.processSlowModeTransaction(sender, tx)` inside a `try/catch`. [11](#0-10) 
5. `processSlowModeTransactionImpl` is invoked. `WithdrawCollateralV2` matches no branch → `else { revert(); }`. [5](#0-4) 
6. The `catch` block in `_executeSlowModeTransaction` absorbs the revert silently (gas check passes for a simple revert). The withdrawal never executes. The user's slow mode fee is gone and the withdrawal is permanently blocked via this path.
