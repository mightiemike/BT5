### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Draining `usdcE` Without Providing `usdc` - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` uses a raw, unchecked `IERC20Base(usdc).transferFrom(...)` call at line 616. If the USDC token returns `false` instead of reverting on failure, the function continues executing and transfers `usdcE` out of the `directDepositV1` contract to the caller — without the caller having actually deposited any `usdc`. The function has no access control and is callable by any unprivileged user.

---

### Finding Description

`replaceUsdcEWithUsdc` is a public, permissionless function on `ContractOwner` that is intended to atomically swap `usdcE` held in a `DirectDepositV1` address for `usdc` provided by the caller:

```solidity
// ContractOwner.sol line 608–620
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    ...
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← raw, unchecked
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);
    }
}
```

The raw `.transferFrom(...)` call at line 616 does **not** check the returned `bool`. The `ERC20Helper` library — which is already imported and attached to `IERC20Base` via `using ERC20Helper for IERC20Base` at line 24 — provides `safeTransferFrom` that properly validates the return value:

```solidity
// ERC20Helper.sol line 23–41
function safeTransferFrom(IERC20Base self, address from, address to, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(...);
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Despite this safe wrapper being available and used elsewhere in the codebase, line 616 bypasses it entirely. If `usdc.transferFrom` returns `false` without reverting (as non-standard or upgradeable ERC-20 tokens may do), execution continues: `usdcE` is withdrawn from `directDepositV1` and sent to the caller at no cost.

---

### Impact Explanation

A caller with zero USDC balance or zero allowance can call `replaceUsdcEWithUsdc(subaccount)` for any subaccount whose `directDepositV1` holds `usdcE`. If `transferFrom` silently fails:

- `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` pulls the full `usdcE` balance out of the DDA into `ContractOwner`.
- `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` sends it to the attacker.

The attacker receives real `usdcE` tokens without providing any `usdc`. The `directDepositV1` is drained. The subaccount's collateral backing is destroyed.

---

### Likelihood Explanation

The function is permissionless — no `onlyOwner` or `onlyDeployer` guard. Any externally-owned account can call it on chain ID 57073 (Ink). The USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink is a relatively new deployment; if it is an upgradeable proxy or a non-Circle implementation that returns `false` on failure rather than reverting, the exploit is immediately executable. Even if the current USDC implementation reverts, the unchecked pattern is a latent vulnerability that activates if the token is upgraded or replaced.

---

### Recommendation

Replace the raw `transferFrom` call at line 616 with the `safeTransferFrom` method already available via the `using ERC20Helper for IERC20Base` directive:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

This mirrors the pattern used for the outbound transfer on line 618 (`safeTransfer`) and is consistent with how all other token transfers in the codebase are handled.

---

### Proof of Concept

1. A `directDepositV1` for `subaccount` exists and holds `N` `usdcE` tokens.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC balance and zero allowance.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, N)` returns `false` (no revert).
4. Return value is not checked; execution continues.
5. `DirectDepositV1(directDepositV1).withdraw(usdcE)` transfers `N` `usdcE` to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(attacker, N)` sends `N` `usdcE` to the attacker.
7. Attacker has stolen `N` `usdcE`; the subaccount's DDA is empty. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
