Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual user, enabling complete allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against `allowedSwapper[pool][sender]`, but the pool always passes its own `msg.sender` — the immediate caller — as `sender`. When swaps route through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. Allowlisting the router (the only way to let users access the pool via the standard periphery) grants every on-chain user unrestricted access, completely defeating per-user access control.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender`:**

`MetricOmmPool.swap` at L230–231 passes its own `msg.sender` directly to `_beforeSwap`:

```solidity
_beforeSwap(
  msg.sender,   // always the direct caller of pool.swap
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` at L160–176 forwards this value unchanged into the ABI-encoded call dispatched to each extension.

**`SwapAllowlistExtension.beforeSwap` checks that value:**

```solidity
// L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router address when the call originates from `MetricOmmSimpleRouter`.

**`MetricOmmSimpleRouter.exactInputSingle` never forwards the original caller:**

```solidity
// L71–80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
);
```

`_setNextCallbackContext` stores `msg.sender` only for the payment callback; it is never passed as the `sender` argument to `pool.swap`. The pool therefore receives `msg.sender = address(router)` and propagates that to the extension.

**Two broken outcomes:**

1. **Broken functionality:** Admin allowlists individual user addresses. Those users call `exactInputSingle`. Pool sees `sender = router`; router is not allowlisted; swap reverts with `NotAllowedToSwap`. Allowlisted users cannot use the standard periphery.

2. **Complete bypass:** To fix (1), admin allowlists the router. Now `allowedSwapper[pool][router] = true`. Any user — including those explicitly excluded — calls `exactInputSingle` and the extension passes, because it only sees the router address. The wrong value checked is `allowedSwapper[pool][address(router)]` instead of `allowedSwapper[pool][address(actualUser)]`.

## Impact Explanation

A pool deploying `SwapAllowlistExtension` to create a permissioned trading venue (KYC-gated, institution-only, regulatory-restricted) cannot enforce per-user access control when `MetricOmmSimpleRouter` is in use. Any unprivileged user bypasses the allowlist by routing through the public router. This constitutes a complete failure of the access-control guard — unauthorized users can execute trades the pool was designed to prevent, directly exposing LP principal to counterparties the pool admin explicitly excluded. This matches the allowed impact: broken core pool functionality causing loss of funds, and admin-boundary break bypassed by an unprivileged path.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard, publicly deployed periphery entry point. No special privileges, no malicious setup, and no admin cooperation are required. Any user who calls `exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) on a pool that has allowlisted the router triggers the bypass immediately and repeatably on every such pool.

## Recommendation

Pass the original user's identity through the extension `sender` argument rather than the immediate `msg.sender`. One approach: the router stores the original caller in transient storage before calling the pool, and the pool reads it as the authoritative `sender` for extension dispatch (analogous to how `_setNextCallbackContext` already stores the payer). Alternatively, `SwapAllowlistExtension` should document explicitly that allowlisting the router grants access to all router users, and per-user enforcement requires direct pool calls only — though the correct fix is architectural: the pool must propagate the true originator, not the intermediary.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is KYC'd
  allowedSwapper[pool][router] = true  // admin adds router so alice can use it

Attack:
  bob (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
      → pool.swap(msg.sender = router, ...)          // MetricOmmPool.sol L231
      → _beforeSwap(sender = router, ...)            // ExtensionCalling.sol L160
      → SwapAllowlistExtension.beforeSwap(sender = router, ...)
      → allowedSwapper[pool][router] == true → passes // SwapAllowlistExtension.sol L37
      → bob's swap executes against LP funds
```

Foundry test outline:
1. Deploy pool with `SwapAllowlistExtension`.
2. Call `setAllowedToSwap(pool, alice, true)` and `setAllowedToSwap(pool, router, true)`.
3. Impersonate `bob` (not allowlisted); call `MetricOmmSimpleRouter.exactInputSingle`.
4. Assert swap succeeds — confirming bypass.
