### Title
CEI Violation in `withdrawCollateral()`: Token Transfer Precedes Balance Update and Health Check — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral()` calls `handleWithdrawTransfer()` — which executes an external `token.safeTransfer` to the recipient — before updating the subaccount's spot balance, asserting utilization, or performing the health check. This is a direct Checks-Effects-Interactions violation. If a token with transfer hooks (ERC-777 or similar) is listed, or if the Endpoint's reentrancy guard does not cover all reachable entry paths, an attacker can re-enter the protocol while the balance state is stale, bypassing the health check that is supposed to gate the withdrawal.

---

### Finding Description

In `Clearinghouse.withdrawCollateral()`, the execution order is:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` — performs the actual token transfer to the recipient.
2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)` — decrements the subaccount balance.
3. **Line 413** — `spotEngine.assertUtilization(productId)` — checks pool utilization.
4. **Line 419** — `require(getHealth(sender, healthType) >= 0, ...)` — enforces solvency. [1](#0-0) 

`handleWithdrawTransfer` in `Clearinghouse.sol` first calls `token.safeTransfer(withdrawPool, amount)`, then calls `BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx)`, which in turn calls `token.safeTransfer(sendTo, amount)` — the final external call to the user-controlled address. [2](#0-1) [3](#0-2) 

The reentrancy window opens at the `token.safeTransfer(sendTo, amount)` call inside `BaseWithdrawPool.handleWithdrawTransfer`. At this point, the call stack is:

```
Endpoint → Clearinghouse.withdrawCollateral
  → handleWithdrawTransfer (Clearinghouse)
    → token.safeTransfer(withdrawPool, ...)
    → withdrawPool.submitWithdrawal(...)
      → token.safeTransfer(sendTo, ...)   ← reentrancy window
```

During this window, the subaccount's balance in `SpotEngine` still reflects the **pre-withdrawal** amount. The health check at line 419 has not yet executed. [4](#0-3) 

The `Endpoint.sol` has some reentrancy protection (grep confirms lock-related patterns), but this does not eliminate the structural risk:

- **Cross-function reentrancy**: If any protocol entry point callable during the transfer window does not share the same reentrancy lock as the Endpoint's `submitTransactions` path, the stale balance can be read or acted upon.
- **ERC-777 / hook-bearing tokens**: If a listed token implements `tokensReceived` or `beforeTokenTransfer` hooks, the hook fires during `safeTransfer` and can invoke arbitrary logic in the attacker's contract before the balance is decremented.
- **Future upgrade risk**: The `withdrawCollateral` function is `public virtual`, meaning an override in an upgrade could alter the call path in a way that bypasses the Endpoint guard entirely. [5](#0-4) 

---

### Impact Explanation

During the reentrancy window, the subaccount balance has not been decremented and the health check has not run. An attacker whose `sendTo` address is a contract with a token hook can:

- Re-read the stale (inflated) balance to pass a second health check.
- If a cross-function path exists that is not covered by the Endpoint's reentrancy lock, trigger a second `withdrawCollateral` for the same subaccount before the first one's balance decrement is committed.

The concrete corrupted state: `SpotEngine` balance for the subaccount remains at the pre-withdrawal value during the external call, meaning the health invariant (`getHealth >= 0`) is evaluated against a balance that does not reflect the already-transferred funds.

**Severity**: High — direct collateral theft if a hook-bearing token is listed or if any non-Endpoint-gated path is reachable during the window.

---

### Likelihood Explanation

The Endpoint's reentrancy guard reduces the immediate attack surface for standard ERC-20 tokens. However:

- The protocol is designed to support multiple spot products; a future listing of an ERC-777 or hook-bearing token directly enables the exploit.
- The `public virtual` modifier on `withdrawCollateral` means an upgrade could introduce a path that bypasses the Endpoint guard.
- The structural CEI violation is present unconditionally — it is not gated on any configuration.

**Likelihood**: Medium — requires a hook-bearing token or an unguarded entry path, both of which are realistic given the protocol's extensibility.

---

### Recommendation

Move all state mutations and checks before the external transfer call in `withdrawCollateral`. The corrected order should be:

1. Compute `amountRealized`.
2. Call `spotEngine.updateBalance(...)`.
3. Call `spotEngine.assertUtilization(...)`.
4. Call `require(getHealth(...) >= 0, ...)`.
5. **Then** call `handleWithdrawTransfer(...)`.

This mirrors the fix applied in the referenced Timeswap H-07: move the external interaction to the end of the function, after all state mutations are complete.

---

### Proof of Concept

1. Attacker deploys a malicious contract `MaliciousRecipient` that implements an ERC-777 `tokensReceived` hook.
2. Attacker opens a subaccount with collateral just above the minimum health threshold.
3. Attacker submits a `WithdrawCollateral` transaction via the Endpoint with `sendTo = address(MaliciousRecipient)`.
4. Endpoint calls `Clearinghouse.withdrawCollateral(...)`.
5. `handleWithdrawTransfer` → `withdrawPool.submitWithdrawal` → `token.safeTransfer(MaliciousRecipient, amount)`.
6. `tokensReceived` fires in `MaliciousRecipient`. At this point, `spotEngine.balances[productId][subaccount]` still holds the pre-withdrawal value.
7. If any protocol function reachable from `MaliciousRecipient` reads this stale balance (e.g., a cross-function path not covered by the Endpoint lock), the attacker can trigger a second withdrawal or pass a health check that should fail.
8. Execution returns; `spotEngine.updateBalance` and `getHealth` run against the now-already-transferred funds — but the second withdrawal has already been initiated. [6](#0-5) [3](#0-2)
