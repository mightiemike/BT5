### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Forgery Mitigation — (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` was superseded by `verify_transaction_inclusion_v2` specifically to close the 64-byte transaction Merkle proof forgery vulnerability. However, the old function was only marked `#[deprecated]` — it was never made private or removed. Any unprivileged NEAR caller can invoke it directly, bypassing the coinbase-proof guard entirely and obtaining a `true` SPV result for a transaction that was never included in any block.

### Finding Description

`verify_transaction_inclusion_v2` was introduced to mitigate the well-known 64-byte Merkle forgery attack (https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring the caller to also supply a coinbase Merkle proof, which anchors the tree structure and prevents an internal node from being mistaken for a leaf (transaction) hash. [1](#0-0) 

The v2 function validates the coinbase proof first, then delegates to v1: [2](#0-1) 

The old v1 function, however, is still decorated with `#[pause]` (making it a live, callable contract method) and carries only a Rust-level `#[deprecated]` attribute, which has zero enforcement at the NEAR runtime level: [3](#0-2) 

The code itself documents the forgery risk in a `# Warning` block: [4](#0-3) 

The `compute_root_from_merkle_proof` function used by both versions is a pure hash-chain computation with no awareness of whether the leaf is a real transaction or an internal node: [5](#0-4) 

### Impact Explanation

An attacker who wants to falsely prove that transaction `T` was included in block `B` can:

1. Identify an internal Merkle-tree node `N` whose 64-byte serialization (two concatenated 32-byte child hashes) hashes to a value that, together with a crafted sibling path, produces block `B`'s `merkle_root`.
2. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B`, and the crafted `merkle_proof`.
3. `compute_root_from_merkle_proof(N, index, proof) == header.block_header.merkle_root` evaluates to `true`.
4. The function returns `true` — a false positive SPV result — for a transaction that does not exist.

Any downstream NEAR contract or protocol that calls `verify_transaction_inclusion` to gate asset releases, bridge withdrawals, or other value-bearing actions is directly exploitable.

### Likelihood Explanation

The entry path requires no privilege: `verify_transaction_inclusion` is a public, non-`#[private]` contract method gated only by the `#[pause]` macro (which is not paused in normal operation). The 64-byte forgery technique is publicly documented and has known tooling. The attacker only needs to construct a valid-looking Merkle path for an internal node, which is computationally trivial given knowledge of the block's transaction set.

### Recommendation

Remove the `#[pause]` attribute from `verify_transaction_inclusion` and replace it with `#[private]` (or delete the public export entirely), so it can only be called internally by `verify_transaction_inclusion_v2`. Alternatively, add a hard `env::panic_str("use verify_transaction_inclusion_v2")` at the top of the function body to make any direct external call revert. The `#[deprecated]` Rust attribute has no effect on NEAR method dispatch.

### Proof of Concept

```rust
// Attacker constructs an internal node N such that:
//   compute_root_from_merkle_proof(N, crafted_index, crafted_proof) == real_merkle_root_of_block_B
//
// Then calls (no special role required):
let result = contract.verify_transaction_inclusion(ProofArgs {
    tx_id: N,                    // internal node hash, not a real tx
    tx_block_blockhash: B,       // a real block on the mainchain
    tx_index: crafted_index,
    merkle_proof: crafted_proof,
    confirmations: 1,
});
// result == true  ← false positive; no such transaction exists
```

The same call routed through `verify_transaction_inclusion_v2` would require a valid coinbase Merkle proof, which the attacker cannot forge without controlling the block's coinbase transaction, making the forgery infeasible via the v2 path. [3](#0-2)

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
