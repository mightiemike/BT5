### Title
`SwapAllowlistExtension` Checks Router Address as `sender`, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `swap` call. When any user routes through `MetricOmmSimpleRouter`, the router is the direct caller of the pool, so the extension checks the **router's address**, not the end-user's address. A pool admin who allowlists the router to enable router-mediated swaps for approved users inadvertently opens the pool to every user, allowing any unprivileged actor to bypass the individual allowlist entirely.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap`, the `sender` forwarded to `_beforeSwap` is `msg.sender` of the pool call: [1](#0-0) 

**Step 2 — Extension checks `sender` (the router) against the allowlist.**

`SwapAllowlistExtension.beforeSwap` checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool, `sender` = whoever called `pool.swap`. When the router calls the pool, `sender` = router address. [2](#0-1) 

**Step 3 — Router never forwards the original user's identity.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly. The pool sees `msg.sender` = router; the original user's address is never surfaced to the pool or the extension. [3](#0-2) 

The same applies to `exactInput` (multi-hop), `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

**Step 4 — The natural admin action creates the bypass.**

A pool admin who wants allowlisted users to be able to swap through the standard router must call `setAllowedToSwap(pool, router, true)`. Once the router is in `allowedSwapper[pool]`, the extension's check passes for **every** user who routes through it, regardless of whether that user is individually allowlisted. [5](#0-4) 

---

### Impact Explanation

The `SwapAllowlistExtension` is the sole on-chain mechanism for restricting which addresses may trade against a pool's LP liquidity. Once bypassed, any unprivileged user can execute swaps on a pool that was intended to be restricted to specific counterparties. This breaks the admin-boundary invariant: an unprivileged path (router call) circumvents the role check the pool admin configured. LP funds are exposed to unauthorized counterparties, enabling adverse selection against LP positions and violating the pool's intended access model.

---

### Likelihood Explanation

The trigger is a natural, well-motivated admin action: allowlisting the router so that approved users can use the standard periphery. There is no documentation warning that allowlisting the router opens the pool to all users. Any pool that (a) deploys `SwapAllowlistExtension` and (b) allowlists the router is fully exposed. The bypass itself requires no special privilege — any user can call `MetricOmmSimpleRouter`.

---

### Recommendation

1. **Document the invariant explicitly**: allowlisting the router is equivalent to `setAllowAllSwappers(pool, true)` for router-mediated paths. Pool admins must understand this before allowlisting the router.
2. **Forward user identity through `extensionData`**: the router could encode `msg.sender` into `extensionData`, and the extension could decode and verify it (with a signature or trusted-forwarder pattern).
3. **Alternatively**, require that allowlist-gated pools only accept direct `pool.swap` calls (document that the router is incompatible with `SwapAllowlistExtension` unless `allowAllSwappers`
