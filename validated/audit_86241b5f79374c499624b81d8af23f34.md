### Title
`validate_cold_store_config` Bypasses `split_storage`/`cold_store` Consistency Check When `archive = true` — (File: `nearcore/src/config_validate.rs`)

---

### Summary

`validate_cold_store_config` in `nearcore/src/config_validate.rs` checks that `split_storage` is not configured only inside the `if !self.config.archive` branch. When `archive = true`, no check enforces that `split_storage` requires `cold_store` to be present. A node can therefore start with `split_storage` configured but `cold_store` absent, silently disabling both the split-storage view client and the cold-store copy loop while passing `validate_config` without error.

---

### Finding Description

`validate_cold_store_config` has two distinct validation regions:

1. **Unconditional check** (line 162): `cold_store.is_some() && save_trie_changes != Some(true)` — rejects a cold store that lacks trie-change saving.
2. **`!archive` block** (lines 170–185): rejects `save_trie_changes = false`, `cold_store`, and `split_storage` on non-archival nodes. [1](#0-0) 

When `archive = true` the entire `!archive` block is skipped. The only remaining check is the one at line 162, which fires only when `cold_store` is already present. There is **no check** for the inverse: `split_storage.is_some() && cold_store.is_none()` on an archival node.

This means the combination:

```
archive          = true
split_storage    = Some(SplitStorageConfig { enable_split_storage_view_client: true, … })
cold_store       = None
save_trie_changes = None   (defaults to !archive = false)
```

passes `validate_config` without any error.

At runtime, `get_split_store` in `nearcore/src/lib.rs` silently returns `None` when `cold_store` is absent: [2](#0-1) 

And `create_cold_store_actor` in `chain/client/src/archive/cold_store_actor.rs` also returns `None` when `cold_db` is absent: [3](#0-2) 

Both subsystems silently no-op. No log line at `ERROR` or `WARN` level is emitted at startup to indicate that the intended split-storage configuration is inoperative.

The structural parallel to the external report is exact:

| External (`VaultTokenized`) | nearcore (`validate_cold_store_config`) |
|---|---|
| Validation of `depositorWhitelistRoleHolder` only runs when `depositWhitelistSetRoleHolder == address(0)` | Validation of `split_storage` requiring `cold_store` only runs when `!archive` |
| Setting `depositWhitelistSetRoleHolder` bypasses the check | Setting `archive = true` bypasses the check |
| Result: whitelist enabled, no one can add to it | Result: split storage configured, cold store loop never starts |

---

### Impact Explanation

An archival node operator who configures `split_storage` (intending to serve historical queries via the split-store view client) but omits `cold_store`:

1. Passes `validate_config` at startup — no error, no warning.
2. The view client falls back to the hot store silently.
3. The cold-store copy loop never starts — no data is ever written to cold storage.
4. Hot-store GC runs normally (since `save_trie_changes` defaults to `false` for archival nodes without explicit override), pruning historical state.
5. After GC, historical trie state is permanently inaccessible via the view client.
6. The operator has no indication from the config validator that the node is misconfigured.

The consequence is silent, permanent loss of archival data availability on a node that was explicitly configured to be archival with split storage.

---

### Likelihood Explanation

The scenario requires an operator to set `split_storage` without `cold_store` on an archival node. This is plausible during a migration to split storage (operator adds `split_storage` to config but forgets `cold_store`), or when copying a partial config from documentation. The existing test suite covers `cold_store` without `save_trie_changes` and `split_storage` without `archive`, but there is no test for `archive=true, split_storage=Some, cold_store=None`. [4](#0-3) 

---

### Recommendation

Add a check inside `validate_cold_store_config` for the archival case:

```rust
fn validate_cold_store_config(&mut self) {
    if self.config.cold_store.is_some() && self.config.save_trie_changes != Some(true) {
        // existing check …
    }

    // NEW: split_storage on an archival node requires cold_store.
    if self.config.archive && self.config.split_storage.is_some() && self.config.cold_store.is_none() {
        let error_message =
            "`split_storage` is configured on an archival node, but `cold_store` is missing. \
             Split storage requires a cold store to be configured."
                .to_string();
        self.validation_errors.push_config_semantics_error(error_message);
    }

    if !self.config.archive {
        // existing checks …
    }
}
```

This mirrors the existing pattern used for `cloud_archival` reader mode: [5](#0-4)

// Safely get the split store while checking that all conditions to use it are met.
fn get_split_store(config: &NearConfig, storage: &NodeStorage) -> anyhow::Result<Option<Store>> {
    // SplitStore should only be used on archival nodes.
    if !config.config.archive {
        return Ok(None);
    }

    // SplitStore should only be used if cold store is configured.
    if config.config.cold_store.is_none() {
        return Ok(None);
    }

    // SplitStore should only be used in the view client if it is enabled.
    if !config.config.split_storage.as_ref().is_some_and(|c| c.enable_split_storage_view_client) {
        return Ok(None);
    }

    // SplitStore should only be used if the migration is finished. The
    // migration to cold store is finished when the db kind of the hot store is
    // changed from Archive to Hot.
    if storage.get_hot_store().get_db_kind() != Some(DbKind::Hot) {
        return Ok(None);
    }

    Ok(storage.get_split_store())
}
