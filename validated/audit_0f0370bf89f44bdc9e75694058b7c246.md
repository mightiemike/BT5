Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Allowing Any Caller to Bypass a Curated Pool's Swap Allowlist — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's `msg.sender` at the time `swap` is called. When `MetricOmmSimpleRouter` calls `pool.swap(...)`, the pool passes the router address as `sender` to the extension. Any pool that is reachable via the router must allowlist the router, so `allowedSwapper[pool][router]` is `true`, and every user who routes through the router passes the allowlist check unconditionally, regardless of whether they are individually allowlisted.

## Finding Description
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 230, passing the immediate caller (the router) as `sender`. `ExtensionCalling._beforeSwap` encodes this `sender` directly into the extension call at line 163–165. `SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router. For the pool to be usable via `MetricOmmSimpleRouter` at all, the pool admin must call `setAllowedToSwap(pool, router, true)`, which sets `allowedSwapper[pool][router] = true`. Once this entry exists, every user who calls any of `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the router passes the allowlist check, because the check resolves to `allowedSwapper[pool][router]` — always `true` — regardless of who the originating user is. The per-user allowlist is completely inoperative for all router-mediated swaps.

## Impact Explanation
A pool admin deploys a KYC-gated or institutionally restricted pool and configures `SwapAllowlistExtension` to restrict swaps to a specific set of addresses. Any non-allowlisted user can bypass this gate by calling `MetricOmmSimpleRouter` instead of the pool directly. The pool receives and settles the swap normally; the extension's guard silently passes because it sees the router, not the user. The pool's curated access policy is completely defeated for all router-mediated swaps. This is a direct policy bypass on curated pools with no fund loss floor requirement — the access control invariant is broken for any pool using this extension with the router.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard periphery entry point for swaps. Any pool that wants to be usable via the router must allowlist the router — this is the normal operational setup. No special knowledge, privileged access, or unusual conditions are required. Any user can call the router. The bypass is unconditional and repeatable for every non-allowlisted user.

## Recommendation
The pool's `swap` function should forward the originating user's address rather than `msg.sender` as the `sender` argument to extensions. The concrete fix is to add an explicit `sender` parameter to `IMetricOmmPoolActions.swap` that the router populates with `msg.sender` (the end user), and the pool forwards to `_beforeSwap`. Alternatively, the pool can pass `tx.origin` as a fallback, but this is generally unsafe in composable contexts. The correct fix is at the pool/router interface boundary: the router must supply the originating user address, and the pool must forward it to extensions rather than using its own `msg.sender`.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Call `extension.setAllowedToSwap(pool, router, true)` — required for the pool to be usable via `MetricOmmSimpleRouter`.
3. As a non-allowlisted user `Eve`, call `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
4. The router calls `pool.swap(...)` — pool sees `msg.sender = router`.
5. Pool calls `_beforeSwap(router, ...)` → `ExtensionCalling._beforeSwap` encodes `sender = router` → extension receives `sender = router`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true`.
7. Eve's swap executes successfully despite not being on the allowlist.
8. Repeat with any number of non-allowlisted users — all succeed via the router.
