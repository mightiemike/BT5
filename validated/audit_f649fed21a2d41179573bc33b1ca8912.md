### Title
Linked Signer Can Unilaterally Replace Itself via `LinkSigner` Transaction, Enabling Persistent Unauthorized Subaccount Control - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `processTransactionImpl` function in `EndpointTx.sol` processes `LinkSigner` transactions with `allowLinkedSigner = true`. This permits the **current linked signer** to sign a transaction that replaces itself with any attacker-controlled address. Once a linked signer is compromised, the attacker can permanently escalate and maintain signing authority over the subaccount — even after the original linked signer key is rotated — by substituting a new attacker-controlled address before the owner can revoke access.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch calls `validateSignedTx` with the fifth argument hardcoded to `true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which in turn calls `Verifier.validateSignature`:

```solidity
function validateSignature(
    bytes32 sender,
    address linkedSigner,
    bytes32 digest,
    bytes memory signature
) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),
        ERR_INVALID_SIGNATURE
    );
}
``` [2](#0-1) 

When `allowLinkedSigner = true`, the `linkedSigner` argument is populated with the current linked signer address:

```solidity
function validateSignature(
    bytes32 sender,
    bytes32 digest,
    bytes memory signature,
    bool allowLinkedSigner
) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [3](#0-2) 

The EIP-712 digest for `LinkSigner` covers `sender`, `signer` (the **new** linked signer), and `nonce`:

```
LinkSigner(bytes32 sender,bytes32 signer,uint64 nonce)
``` [4](#0-3) 

Because the current linked signer is accepted as a valid signer for this digest, it can produce a valid signature over a `LinkSigner` message where `signer` is any attacker-controlled address. The result is that `linkedSigners[subaccount]` is overwritten with the attacker's address.

Contrast this with `WithdrawCollateralV2`, where the protocol explicitly restricts the linked signer when funds are directed to a non-owner address (`allowLinkedSigner = sendTo == address(0)`), demonstrating that the developers are aware of the need to scope linked-signer permissions: [5](#0-4) 

No equivalent restriction exists for `LinkSigner`.

---

### Impact Explanation

A compromised linked signer can:

1. Sign a `LinkSigner` transaction naming an attacker-controlled address as the new `signer`.
2. The sequencer processes it; `linkedSigners[subaccount]` is overwritten.
3. The attacker's new linked signer can now sign:
   - `WithdrawCollateral` / `WithdrawCollateralV2` (funds sent to the subaccount owner's address — draining the account on demand once the owner deposits again).
   - `LiquidateSubaccount` (charging liquidation fees against the subaccount).
   - `MintNlp` / `BurnNlp` (manipulating NLP positions and incurring fees).
   - Further `LinkSigner` transactions (maintaining persistent access indefinitely).
4. Even if the original linked signer key is rotated or revoked externally, the attacker's address remains the active linked signer until the subaccount owner submits a new `LinkSigner` transaction.

The corrupted state is `linkedSigners[subaccount]` — a persistent mapping that governs all future signature authorization for the subaccount.

---

### Likelihood Explanation

**Medium.** The attack requires the linked signer key to be compromised first (e.g., via private key leak from a trading bot or hot wallet). However, linked signers are by design lower-security keys (hot wallets, automated bots) compared to the subaccount owner key. Once compromised, the escalation is trivial, silent, and permanent until the owner notices and revokes.

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in `processTransactionImpl`. Only the subaccount owner (the address embedded in the first 20 bytes of the `sender` subaccount) should be permitted to change the linked signer. The linked signer is a delegate, not an owner; it must not be able to modify its own authorization.

```solidity
// Before (vulnerable):
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // allows linked signer to replace itself
);

// After (fixed):
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only subaccount owner may change the linked signer
);
```

### Proof of Concept

1. User calls `submitSlowModeTransaction` (or the sequencer submits a `LinkSigner` tx) to set `linkedSigners[subaccount] = bot_address`.
2. Attacker obtains `bot_address`'s private key (e.g., from a compromised trading server).
3. Attacker constructs a `SignedLinkSigner` struct:
   - `tx.sender = subaccount`
   - `tx.signer = bytes32(uint256(uint160(attacker_address)))`
   - `tx.nonce = current_nonce`
   - `signature` = ECDSA signature over the EIP-712 digest, signed by `bot_address`
4. Attacker submits this to the sequencer (or via slow mode).
5. `processTransactionImpl` calls `validateSignedTx(..., true)`. `Verifier.validateSignature` recovers `bot_address` from the signature, which equals `linkedSigner` — validation passes.
6. `linkedSigners[subaccount]` is overwritten with `attacker_address`.
7. User rotates `bot_address` key — irrelevant, since `attacker_address` is now the active linked signer.
8. Attacker uses `attacker_address` to sign `WithdrawCollateral` transactions, draining the subaccount to the owner's address on every future deposit, and signs further `LinkSigner` transactions to maintain persistence.
