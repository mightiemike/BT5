Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual trader — allowlist bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` at the pool level — the router contract, not the originating EOA. If the pool admin allowlists `MetricOmmSimpleRouter`, every unprivileged user can bypass the per-user allowlist by routing through it. Conversely, individually allowlisted users cannot use the router at all, breaking the intended access-control model in both directions.

## Finding Description
**Call path:**
1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)` (or `exactInput` / `exactOutputSingle` / `exactOutput`).
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — at this point `msg.sender` inside the pool is the **router**, not the user.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` passing the router address as `sender` (line 231).
4. The pool calls `SwapAllowlistExtension.beforeSwap(router_address, ...)`.
5. The extension evaluates `allowedSwapper[msg.sender/*pool*/][sender/*router*/]`.

The actual originating user's address (`msg.sender` inside the router, stored only in transient callback context for payment) is **never forwarded** to the pool or the extension. The `extensionData` bytes field is caller-supplied and not used for identity by the extension.

**Why existing guards fail:**
- `allowedSwapper` is keyed on the direct caller of `pool.swap()`, which is always the router for router-mediated swaps.
- There is no `tx.origin` check, no signed-identity mechanism, and no forwarding of the real user address through `extensionData`.
- If the pool admin allowlists the router (a natural configuration for a pool that should be accessible via the official periphery), the check `allowedSwapper[pool][router] == true` passes for **every** user who calls through the router, regardless of whether that user is individually permitted.

**Exact wrong value:** `allowedSwapper[pool][sender]` where `sender` is the router address, not the trader's address — the extension decision is bound to the wrong identity.

## Impact Explanation
Any unprivileged trader can bypass a per-user swap allowlist on a pool that has `MetricOmmSimpleRouter` allowlisted. The allowlist extension's core invariant — that only explicitly permitted addresses may swap — is violated. This constitutes broken core pool functionality (access-control bypass on the swap path) and falls under the "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows" allowed impact category. Severity is Medium: the allowlist bypass is complete but requires the router to be allowlisted by the pool admin, which is a realistic and expected operational configuration.

## Likelihood Explanation
Any unprivileged EOA can exploit this by calling `MetricOmmSimpleRouter.exactInputSingle` targeting a pool whose allowlist includes the router address. No special privileges, flash loans, or oracle manipulation are required. The condition (router is allowlisted) is a standard operational setup for any pool intended to be used via the official periphery. The attack is repeatable on every block.

## Recommendation
Pass the originating user's address through the extension hook rather than the direct pool caller. One approach: add an `originSender` field to `extensionData` that the router populates with `msg.sender` before calling `pool.swap()`, and have `SwapAllowlistExtension.beforeSwap` decode and verify it. A stronger approach is to have the pool accept an explicit `originSender` parameter in `swap()` and forward it as the first argument to all extension hooks, replacing the current `msg.sender` forwarding. The extension should then check `allowedSwapper[pool][originSender]` instead of `allowedSwapper[pool][sender]`.

## Proof of Concept
```solidity
// Pool has SwapAllowlistExtension configured.
// Pool admin allowlists the router: allowedSwapper[pool][router] = true.
// Alice (not individually allowlisted) calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: restrictedPool,
    recipient: alice,
    tokenIn: token0,
    amountIn: 1e18,
    amountOutMinimum: 0,
    zeroForOne: true,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// pool.swap() is called with msg.sender = router.
// beforeSwap receives sender = router (allowlisted) → passes.
// Alice swaps successfully despite not being on the allowlist.
```
Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist only the router, attempt swap from an unlisted EOA directly (expect revert) and via the router (expect success — demonstrating bypass).
