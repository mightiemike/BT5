Audit Report

## Title
`SwapAllowlistExtension` Gates on Direct Pool Caller (`sender = msg.sender`) Instead of the Actual User, Allowing Allowlist Bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` at the pool boundary — the direct caller. When `MetricOmmSimpleRouter` routes a swap, the router becomes `msg.sender` to the pool, so the extension gates on the router address rather than the actual user. This creates two fund-impacting outcomes: (1) if the router is allowlisted, any unprivileged user bypasses the curated-pool swap gate entirely; (2) if the router is not allowlisted, individually allowlisted users cannot use the standard swap entrypoint at all.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (correct) and `sender` is the direct pool caller:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router `msg.sender` to the pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The same pattern holds for `exactInput`, `exactOutputSingle`, and `exactOutput`. The extension therefore checks whether the **router** is allowlisted, not whether the **user** (`msg.sender` to the router) is allowlisted. The test in `FullMetricExtension.t.sol` confirms this: it allowlists `callers[0]` (the `TestCaller` contract that directly calls the pool), not `users[0]` (the human behind it):

```solidity
// metric-periphery/test/extensions/FullMetricExtension.t.sol L70-73
swapExtension.setAllowedToSwap(address(pool), address(callers[0]), true);
_swap(0, users[0], false, int128(1000), type(uint128).max);
```

The `DepositAllowlistExtension` avoids this problem because `addLiquidity` accepts an explicit `owner` parameter that the pool passes separately from `msg.sender`, and the extension checks `owner` (the economically relevant actor) rather than `sender` (the direct caller). The `swap` interface has no equivalent explicit user parameter, so `SwapAllowlistExtension` has no correct actor to check other than `sender` — which is always the router when routing through `MetricOmmSimpleRouter`.

## Impact Explanation

**Scenario 1 — Allowlist bypass (High):** If the pool admin allowlists the router address (to enable standard-periphery usage), every unprivileged user can bypass the curated-pool swap gate by routing through `MetricOmmSimpleRouter`. The allowlist provides zero protection against unauthorized swappers on the curated pool. Any user can swap assets they should not be permitted to swap, violating the pool's curation invariant and potentially draining liquidity at oracle-quoted prices.

**Scenario 2 — Allowlisted users locked out (Medium):** If the router is not allowlisted, users who are individually allowlisted cannot use `MetricOmmSimpleRouter` at all — their swaps revert with `NotAllowedToSwap` even though they are authorized. The standard swap entrypoint is entirely unusable for the intended user set, constituting broken core swap functionality.

Both outcomes break the invariant that a curated pool enforces the same allowlist policy regardless of which supported public entrypoint reaches it.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Any pool that deploys `SwapAllowlistExtension` and expects users to interact via the standard router is immediately affected by Scenario 2. Scenario 1 requires the pool admin to allowlist the router — a natural operational step when the intent is to allow standard-periphery access while restricting direct callers — but the admin may not realize this grants universal bypass. No special privileges are required for the attacker in Scenario 1; any user can call the public router.

## Recommendation

The extension must gate on the economically relevant actor — the human or contract that initiated the swap — not the intermediate router. Two approaches:

1. **Extend the `swap` interface** to accept an explicit `sender` parameter (analogous to `owner` in `addLiquidity`) that the router populates with `msg.sender` (the actual user). The extension would then check this explicit parameter, mirroring `DepositAllowlistExtension`'s use of `owner`.
2. **Document incompatibility** and enforce at the extension level that `SwapAllowlistExtension` must not be used with `MetricOmmSimpleRouter`, or add a router-level forwarding mechanism (e.g., passing the originating user in `extensionData` with extension-side verification) that the extension can authenticate.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `beforeSwap` order.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` to allow the standard router.
3. Non-allowlisted user Alice calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the curated pool.
4. The router calls `pool.swap(...)` — the pool sets `sender = address(router)`.
5. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. Alice, who is not in the allowlist, successfully swaps on the curated pool, violating the curation invariant.

For Scenario 2: repeat with the router not allowlisted but Alice individually allowlisted — her router call reverts with `NotAllowedToSwap` despite her being authorized.
