Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of end-user, allowing any caller to bypass per-user swap restrictions — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which the pool sets to `msg.sender` of the `swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router address, not the end-user. Any pool admin who allowlists the router to enable router-mediated swaps for their curated users inadvertently grants unrestricted swap access to every address that calls the router, completely nullifying the per-user allowlist.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, recipient, ...)`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim as the `sender` argument to every configured extension: [2](#0-1) 

**Step 2 — Router is the immediate caller of `pool.swap`.**

`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` directly. The pool therefore sees `msg.sender = address(router)`, not the end-user's address: [3](#0-2) 

**Step 3 — Extension checks the wrong actor.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (correct) and `sender` is the router address (wrong — it should be the end-user). The lookup becomes `allowedSwapper[pool][router]`: [4](#0-3) 

**Step 4 — The bypass.**

A pool admin who wants to allow specific KYC'd users to swap via the router will:
1. Allowlist individual users: `setAllowedToSwap(pool, userA, true)`
2. Also allowlist the router so those users can use it: `setAllowedToSwap(pool, router, true)`

Once the router is allowlisted, **any** address can call `router.exactInputSingle(pool, ...)` and the extension will pass — because it only checks `allowedSwapper[pool][router]`, which is `true`. The actual end-user identity is never inspected.

**Contrast with `DepositAllowlistExtension`:** The deposit extension correctly gates on `owner` (the LP position recipient), which is the economic beneficiary and is preserved correctly through the `MetricOmmPoolLiquidityAdder` path. The swap extension has no equivalent correct actor to check because the router does not forward the originating user identity: [5](#0-4) 

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., KYC'd institutions, whitelisted market makers) loses that restriction entirely for any user who routes through the public `MetricOmmSimpleRouter`. Non-allowlisted users can drain liquidity, front-run allowlisted traders, or violate the pool's compliance model. This breaks the core curation invariant the extension was deployed to enforce and constitutes an admin-boundary break reachable by an unprivileged path — a direct allowed impact under the contest rules.

## Likelihood Explanation

The trigger requires the pool admin to allowlist the router — a natural and expected action for any pool that wants to support router-mediated swaps for its allowlisted users. The router is a first-party periphery contract deployed by the protocol. Any pool that combines `SwapAllowlistExtension` with router support is vulnerable. The attacker needs no privileged access; a standard `exactInputSingle` call suffices. The same flaw applies to `exactInput`, `exactOutputSingle`, and `exactOutput` paths in the same router.

## Recommendation

The extension must gate on the **end-user identity**, not the immediate caller. The preferred fix is to redesign `SwapAllowlistExtension.beforeSwap` to check `recipient` (the economic beneficiary of the swap) rather than `sender` (the technical caller), since `recipient` is correctly forwarded from the end-user through the router. Alternatively, introduce a trusted-forwarder pattern where the router encodes the real user identity into `extensionData` and the extension verifies it — but this requires the pool to enforce the encoding to prevent forgery.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, alice, true)       // alice is KYC'd
  pool admin calls setAllowedToSwap(pool, router, true)      // enable router for alice

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ..., recipient: bob})

Result:
  pool.swap(msg.sender=router, recipient=bob, ...)
  _beforeSwap(sender=router, ...)
  extension checks allowedSwapper[pool][router] == true  → passes
  bob receives output tokens despite never being allowlisted
```
