### Title
NFT Becomes Permanently Unburnable After Transfer to Non-Creator — (`x/nft/keeper/keeper.go`)

### Summary
`keeper.BurnNFT` enforces a conjunctive guard requiring the caller to be **both** the current NFT owner **and** the denom creator simultaneously. Once a denom creator transfers an NFT to any other address, no party can ever satisfy both conditions at once, making the NFT permanently unburnable through the only exposed user-facing path (`MsgBurnNFT`).

### Finding Description
In `keeper.BurnNFT` (lines 141–161 of `x/nft/keeper/keeper.go`), the function first calls `k.IsOwner(ctx, denomID, tokenID, owner)` and then `k.IsDenomCreator(ctx, denomID, owner)`, both against the same `owner` argument. The `MsgBurnNFT` message handler passes `msg.Sender` as that single `owner`. After a `MsgTransferNFT` moves the NFT from alice (creator) to bob:

- Bob's `MsgBurnNFT`: `IsOwner` passes, `IsDenomCreator` fails → `ErrUnauthorized`
- Alice's `MsgBurnNFT`: `IsOwner` fails (alice is no longer owner) → `ErrUnauthorized`

`BurnNFTUnverified` (which only checks `IsOwner`) is not reachable via any transaction message — it is an internal function called only by the IBC transfer module. The denom creator field is immutable after `IssueDenom`, so the deadlock cannot be resolved.

### Impact Explanation
- The NFT supply counter (`decreaseSupply`) is never called, permanently inflating the on-chain supply for the denom.
- The NFT record persists in the KV store indefinitely and cannot be removed by any standard user transaction.
- Any NFT in a denom that has ever been transferred away from its creator is frozen in an unburnable state, violating the module's own invariant that owners and creators have authority over NFTs they control.

### Likelihood Explanation
This is triggered by the entirely normal and intended workflow of: issue denom → mint NFT → transfer NFT. No special privileges, governance, or key compromise is required. Any user who receives an NFT from its denom creator will hold an unburnable token.

### Recommendation
Change `keeper.BurnNFT` to use an **OR** condition: allow burning if the caller is either the current NFT owner **or** the denom creator. Alternatively, align the burn semantics with `BurnNFTUnverified` for the owner case (only require `IsOwner`), and separately allow the denom creator to burn any NFT in their denom regardless of current ownership.

### Proof of Concept
```
1. alice issues denom "testdenom" (alice = creator)
2. alice mints NFT "token1" in "testdenom", recipient = alice
3. alice sends MsgTransferNFT: sender=alice, recipient=bob, denomId="testdenom", id="token1"
4. bob sends MsgBurnNFT: sender=bob, denomId="testdenom", id="token1"
   → keeper.BurnNFT: IsOwner(bob) ✓, IsDenomCreator(bob) ✗ → ErrUnauthorized
5. alice sends MsgBurnNFT: sender=alice, denomId="testdenom", id="token1"
   → keeper.BurnNFT: IsOwner(alice) ✗ (owner is bob) → ErrUnauthorized
6. NFT "token1" is permanently locked in state; supply counter never decremented.
```

func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {

	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	_, err = k.IsDenomCreator(ctx, denomID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
}
