### Title
Checkpoint Advance Without Bonus Payment in `slashRedelegationPosition` Causes Permanent Bonus Loss — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary
When `BeforeRedelegationSlashed` fires and the `RewardsPoolName` module account balance is insufficient, `processEventsAndClaimBonus` advances `pos.LastBonusAccrual`, `pos.LastEventSeq`, and `pos.LastKnownBonded` in memory before the pool balance check. `slashRedelegationPosition` swallows `ErrInsufficientBonusPool` and then persists the mutated position via `setPositionWithState`, permanently skipping the accrued bonus period with no recovery path.

### Finding Description
`processEventsAndClaimBonus` receives `pos *types.PositionState` by pointer. The event loop at line 193 calls `pos.UpdateLastEventSeq(entry.Seq)` and also calls `k.decrementEventRefCount` (consuming the event from the store) on every iteration. After the loop, line 215 calls `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` which sets `pos.LastBonusAccrual = blockTime`, and line 217 updates `pos.LastKnownBonded`. The pool sufficiency check at line 230 only runs after all these mutations. If it fails, the function returns `ErrInsufficientBonusPool` with the in-memory `pos` already fully advanced.

Back in `slashRedelegationPosition`, the error is caught at line 56, logged, and silently dropped. The code then unconditionally calls `setPositionWithState(ctx, pos, ...)` at line 71 or 77, persisting the advanced checkpoints. The bonus period is permanently lost: the next call to `processEventsAndClaimBonus` for this position will start from the already-advanced `LastBonusAccrual` and `LastEventSeq`, computing zero bonus for the skipped interval.

### Impact Explanation
The position owner permanently loses all accrued bonus rewards for the redelegation period that was processed during the slash. The invariant that `LastBonusAccrual` must not advance past a time for which bonus was actually paid is violated. There is no retry or recovery mechanism — the consumed events and advanced checkpoint are irreversible.

### Likelihood Explanation
The attack path is concrete and externally reachable:
1. Attacker creates a position via `LockTier` (standard `MsgServer` transaction).
2. Attacker redelegates via `TierRedelegate`, creating a `RedelegationMapping` entry for the unbonding ID.
3. Attacker (or natural pool depletion) drains `RewardsPoolName` to zero.
4. Attacker controls a validator and deliberately double-signs to trigger `BeforeRedelegationSlashed` with the relevant `unbondingId`.

An attacker controlling a validator can deliberately trigger this. Even without attacker control, any slash on a validator with an active redelegation mapping while the pool is depleted triggers the bug.

### Recommendation
Move `applyBonusAccrualCheckpoint`, `pos.UpdateLastKnownBonded`, and `pos.UpdateLastEventSeq` to execute **only after** a successful `SendCoinsFromModuleToAccount`. If the pool is insufficient, return the error without mutating the position's checkpoints, so the caller can persist the original (unadvanced) position state. In `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, explicitly reset `pos` to its pre-call state (or re-fetch it from the store) before calling `setPositionWithState`.

### Proof of Concept
1. Create a position via `LockTier` for owner A delegating to validator V.
2. Call `TierRedelegate` to redelegate to validator V2, recording a `RedelegationMapping(unbondingId → positionId)`.
3. Drain the `RewardsPoolName` module account balance to zero via `BankKeeper.SendCoinsFromModuleToAccount` in test setup.
4. Call `BeforeRedelegationSlashed(ctx, unbondingId, sharesToUnbond)`.
5. Assert: `pos.LastBonusAccrual == blockTime` (advanced) but owner A received zero bonus coins.
6. Call `ClaimTierRewards` for the same period: assert zero bonus is returned because `LastBonusAccrual` already equals `blockTime`. [7](#0-6) [5](#0-4)
