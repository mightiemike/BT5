Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the depositor allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` receives both `sender` (the actual `msg.sender` of `addLiquidity`, i.e., the capital provider) and `owner` (the LP share recipient), but silently discards `sender` and gates only on `owner`. Because `owner` is a free caller-controlled parameter with no access restriction on `pool.addLiquidity`, any non-allowlisted address can pass an allowlisted address as `owner`, satisfy the check, and inject their own capital into the pool. The depositor allowlist — the sole mechanism for pool curation — is rendered completely ineffective.

## Finding Description
`MetricOmmPool.addLiquidity` accepts a free `owner` parameter and places no restriction on who may call it:

```solidity
// metric-core/contracts/MetricOmmPool.sol L182-196
function addLiquidity(address owner, uint80 salt, ...) external nonReentrant(...) {
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    ...
}
```

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both `sender` (the real caller) and `owner` to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` then silently drops `sender` (unnamed first parameter) and gates only on `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

After the hook passes, `LiquidityLib.addLiquidity` invokes `metricOmmModifyLiquidityCallback` on `msg.sender` (the attacker's contract), pulling the attacker's tokens, while LP shares are minted to `owner` (`allowlistedAddress`). The periphery `MetricOmmPoolLiquidityAdder._validateOwner` only guards against `address(0)` and is bypassed entirely when the attacker calls the pool directly.

**Exploit flow:**
1. Attacker deploys a contract implementing `metricOmmModifyLiquidityCallback` that pays tokens from the attacker.
2. Attacker calls `pool.addLiquidity(owner=allowlistedAddr, salt, deltas, callbackData, "")` directly.
3. `_beforeAddLiquidity` calls the extension with `sender=attacker, owner=allowlistedAddr`.
4. Extension checks `allowedDepositor[pool][allowlistedAddr]` → `true`. Hook passes.
5. Pool calls `metricOmmModifyLiquidityCallback` on the attacker's contract, pulling attacker's tokens.
6. LP shares are minted to `allowlistedAddr`. Attacker's capital is now inside the pool.

## Impact Explanation
The depositor allowlist is the sole mechanism for restricting which addresses may provide capital to a curated pool. This bypass allows any unprivileged address to deposit capital into a restricted pool, breaking the admin-configured invariant that only approved depositors may provide liquidity. This constitutes a broken admin-boundary and broken core pool functionality. The allowlisted address receives unwanted LP shares they did not initiate; unauthorized capital enters the pool on every such call.

## Likelihood Explanation
The attack requires only a direct call to `pool.addLiquidity` with a known allowlisted address as `owner`. No privileged access, no special token behavior, and no off-chain data are needed. Allowlisted addresses are publicly discoverable via `AllowedToDepositSet` events or `allowedDepositor` view calls. The attacker only needs to deploy a contract implementing the callback interface. Likelihood is high.

## Recommendation
Check `sender` (the actual capital provider) instead of `owner` (the share recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```solidity
// Foundry integration test
function test_allowlistBypass() public {
    // Setup: pool with DepositAllowlistExtension; only `allowlisted` is approved
    address allowlisted = makeAddr("allowlisted");
    extension.setAllowedToDeposit(address(pool), allowlisted, true);

    address attacker = makeAddr("attacker");
    // attacker is NOT in allowedDepositor

    // Fund attacker's callback contract with tokens
    deal(token0, address(attackerCallback), 1_000e18);
    deal(token1, address(attackerCallback), 1_000e18);

    vm.startPrank(attacker);
    // Attacker calls pool directly with owner=allowlisted
    pool.addLiquidity(allowlisted, salt, deltas, callbackData, "");
    vm.stopPrank();

    // allowlisted received LP shares funded by attacker's capital
    assertGt(positionBinShares(address(pool), allowlisted, salt, binIdx), 0);
}
```
