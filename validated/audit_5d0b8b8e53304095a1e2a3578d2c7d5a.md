### Title
Wrong Delegation Shares Used in Historical Segment Bonus Calculation — (`x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

`computeSegmentBonus` uses the **current** `pos.Delegation.Shares` when computing bonus for every historical time segment, including segments that accrued before the current delegation amount was established. If a user increases their delegation (e.g., via `AddToPosition`) after a segment has already accrued, the bonus for that historical segment is computed using the inflated share count rather than the shares that were actually delegated during that period.

---

### Finding Description

In `processEventsAndClaimBonus`, the keeper iterates over validator events since `pos.LastEventSeq` and computes bonus for each bonded segment:

```go
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
``` [1](#0-0) 

Inside `computeSegmentBonus`, the token value for the segment is computed as:

```go
tokens := pos.Delegation.Shares.Mul(tokensPerShare)
``` [2](#0-1) 

Here, `tokensPerShare` is a **historical snapshot** captured at the validator event time, but `pos.Delegation.Shares` is the **current** delegation shares at the moment of the claim call. These two values belong to different points in time. If the user's shares increased between when the segment started and when `ClaimRewards` is called, the product `shares × tokensPerShare` overstates the token value that was actually delegated during that historical segment.

This is directly analogous to the reported Vader bug: the second-step check uses the input value of the first step (the original token amount) instead of the intermediate output (base tokens received), causing the check to operate on mismatched quantities. Here, the historical rate (`tokensPerShare`) is paired with the wrong quantity (current shares instead of shares-at-segment-time).

The `pos.Delegation` is fetched fresh from the staking keeper at call time: [3](#0-2) 

There is no per-event snapshot of the delegator's share count stored alongside `evt.TokensPerShare`, so there is no mechanism to recover the historically correct share count for each segment.

---

### Impact Explanation

A user who increases their delegation after a segment has accrued (via `AddToPosition` or redelegation) and then calls `ClaimRewards` will receive bonus computed as:

```
bonus = (X + Y) × tokensPerShare × BonusApy × duration / SecondsPerYear
```

instead of the correct:

```
bonus = X × tokensPerShare × BonusApy × duration / SecondsPerYear
```

The excess `Y × tokensPerShare × BonusApy × duration / SecondsPerYear` is paid from the `RewardsPoolName` module account, draining it faster than the protocol intends and at the expense of other participants who depend on that pool for their own bonus rewards. [4](#0-3) 

---

### Likelihood Explanation

The exploit path requires that `AddToPosition` (or a redelegation operation) does **not** settle historical bonus rewards before updating `pos.Delegation.Shares`. This is the standard vulnerability pattern in reward-accrual systems. The existence of `msg_server_add_to_position_test.go` and `msg_server_redelegate_test.go` confirms both operations are supported production entry points. Any unprivileged user with an active position can attempt this by submitting an `AddToPosition` transaction followed by a `ClaimRewards` transaction. The profit scales with the size of the added delegation and the length of the historical window since the last claim. [5](#0-4) 

---

### Recommendation

Store a snapshot of the delegator's share count at each validator event (alongside `TokensPerShare`), or require that `processEventsAndClaimBonus` is called and rewards are settled before any operation that changes `pos.Delegation.Shares`. This mirrors the standard "checkpoint before mutation" pattern used in Compound/Aave-style reward accounting.

---

### Proof of Concept

1. User creates a position with `X` shares at time `T0`. `pos.LastEventSeq` is set to the current latest event.
2. Time passes from `T0` to `T1`. Validator events are recorded with `TokensPerShare` snapshots. The accrued bonus for this window is proportional to `X`.
3. User submits `AddToPosition` at `T1`, increasing delegation to `X + Y` shares. If this operation does not first call `processEventsAndClaimBonus`, `pos.LastEventSeq` remains at the `T0` value.
4. User submits `ClaimRewards` at `T1`.
5. `processEventsAndClaimBonus` iterates all events since `T0`, computing each segment's bonus using `pos.Delegation.Shares = X + Y`.
6. The user receives bonus for the `[T0, T1]` window computed on `X + Y` shares instead of `X` shares — an excess of `Y × tokensPerShare × BonusApy × (T1−T0) / SecondsPerYear` drawn from the rewards pool. [6](#0-5)

func (k Keeper) processEventsAndClaimBonus(ctx context.Context, pos *types.PositionState) (sdk.Coins, error) {
	// Rewards should have been claimed before undelegation
	if !pos.IsDelegated() {
		return sdk.NewCoins(), nil
	}

	valAddr, err := sdk.ValAddressFromBech32(pos.Delegation.ValidatorAddress)
	if err != nil {
		return nil, err
	}

	events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	blockTime := sdkCtx.BlockTime()

	totalBonus := math.ZeroInt()
	// Use the persisted bonded state from the last replay, not a hardcoded default.
	// This prevents overpaying bonus for unbonded gaps between claims.
	bonded := pos.LastKnownBonded
	segmentStart := pos.LastBonusAccrual

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	for _, entry := range events {
		evt := entry.Event

		if bonded {
			// Compute bonus for the bonded segment [segmentStart, eventTime]
			// using the snapshot rate at the event.
			bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
			totalBonus = totalBonus.Add(bonus)
		}

		// Update bonded state based on event type.
		switch evt.EventType {
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND:
			bonded = false
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND:
			bonded = true
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_SLASH:
			// Slash doesn't change bonded state.
		}

		segmentStart = evt.Timestamp
		pos.UpdateLastEventSeq(entry.Seq)

		// Decrement reference count.
		if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
			return nil, err
		}
	}

	val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
	if err != nil {
		return nil, err
	}
	// Defensive: validator bond status check
	if bonded && val.IsBonded() {
		currentRate, err := k.getTokensPerShare(ctx, valAddr)
		if err != nil {
			return nil, err
		}
		bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
		totalBonus = totalBonus.Add(bonus)
	}

	applyBonusAccrualCheckpoint(&pos.Position, blockTime)

func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}
