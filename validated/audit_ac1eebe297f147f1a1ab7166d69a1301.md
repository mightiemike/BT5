### Title
Fork Difficulty Calculated Against Main-Chain Ancestor Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When a fork block is submitted at a difficulty-adjustment boundary, `get_next_work_required` fetches the start-of-period reference block via `get_header_by_height`, which always reads from the **main chain** (`mainchain_height_to_header`). For main-chain submissions the reference block is the correct ancestor; for fork submissions it is the main-chain block at the same height, which may be a completely different block with a different timestamp. This asymmetric treatment lets an attacker craft a fork whose difficulty-adjustment boundary block is validated against the wrong timestamp, causing the contract to accept that block at a lower (or higher) difficulty than the fork's actual ancestor chain requires.

---

### Finding Description

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // always the main chain
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

All three chain-specific difficulty functions call this to obtain the first block of the retarget window:

**Bitcoin** (`bitcoin.rs` lines 78–86):
```rust

    blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config, prev_block_header,

``` [2](#0-1) 

**Litecoin** (`litecoin.rs` lines 86–93): identical pattern. [3](#0-2) 

**Dogecoin** (`dogecoin.rs` lines 291–297): same pattern, with an explicit developer acknowledgement of the problem:
```rust
// TODO: check if it is correct to get block header by height from mainchain
//       without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header.time;
``` [4](#0-3) 

**The asymmetry:** when a main-chain block is submitted, the block at `first_block_height` in `mainchain_height_to_header` *is* the correct ancestor. When a fork block is submitted, the block at `first_block_height` in `mainchain_height_to_header` is the **main-chain block at that height**, which is a different block from the fork's ancestor at the same height. The fork's ancestor is stored in `headers_pool` but is never consulted for difficulty calculation.

This mirrors the Fei Protocol bug exactly: one direction of state transition (main-chain submission) uses the correct reference value, while the other direction (fork submission) uses a stale/wrong reference value, producing an asymmetric security outcome.

---

### Impact Explanation

Suppose the main chain has a block `M_H` at height `H` (the start of a 2016-block retarget window) with timestamp `T_M`. An attacker submits a fork that diverges at height `H`, placing a fork block `F_H` at height `H` with timestamp `T_F > T_M` (later than the main-chain block).

When the attacker submits the fork block at height `H + 2016` (the retarget boundary), the contract computes expected difficulty using `T_M` (the main-chain block's timestamp) as the window start, not `T_F`. Because `T_M < T_F`, the computed window appears *longer* than the fork's actual window, so the contract calculates a *lower* required difficulty (higher target) than the fork's ancestor chain actually demands.

The attacker can therefore mine the retarget-boundary fork block against this artificially relaxed target. With a lower per-block difficulty, each fork block contributes less chainwork, but the attacker can compensate by mining more blocks cheaply. More critically, the contract accepts a block that would be **invalid on the real Bitcoin network**, corrupting the light client's canonical chain. Any downstream `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` call that relies on the reorged chain returns results based on a fraudulent chain state. [5](#0-4) 

---

### Likelihood Explanation

- The entry point is `submit_blocks`, callable by any unprivileged NEAR account (subject only to the `trusted_relayer` gate, which is bypassable by accounts with `UnrestrictedSubmitBlocks` or `DAO` roles, and in practice the relayer role is open to stakers).
- A fork spanning a difficulty-adjustment boundary (every 2016 Bitcoin blocks, ~2 weeks) is a routine scenario; the attacker does not need to control the main chain.
- The attacker only needs to submit fork headers with a crafted timestamp at the divergence point; no private keys or privileged roles are required beyond the ability to call `submit_blocks`.
- The Dogecoin TODO comment confirms the developers are aware the logic is potentially wrong, raising confidence this is a real gap rather than a theoretical one. [6](#0-5) 

---

### Recommendation

Replace `get_header_by_height` (main-chain lookup) with an ancestor walk starting from `prev_block_header` when computing the retarget window for a fork block. Specifically, walk back `difficulty_adjustment_interval - 1` steps through `get_prev_header` (which follows `prev_block_hash` pointers stored in `headers_pool` and therefore correctly traverses the fork's own ancestor chain) to find the true first block of the retarget window. This is the approach used by Bitcoin Core's reference implementation and by the Zcash implementation already present in `zcash.rs`, which uses `get_prev_header` exclusively and never calls `get_header_by_height`. [7](#0-6)

### Proof of Concept

1. Contract is initialized with a Bitcoin main chain at height 0. Main-chain block at height 0 has timestamp `T0 = 1_000_000`.
2. Attacker submits 2015 fork blocks branching from height 0, each with timestamps slightly above `T0`. The fork block at height 0 (the divergence point) has timestamp `T_fork_0 = 1_100_000` (100 000 seconds later than the main-chain block at height 0).
3. At height 2016 (retarget boundary), the contract calls `get_next_work_required`. It calls `get_header_by_height(0)`, which returns the **main-chain** block at height 0 with `T0 = 1_000_000`.
4. The computed window duration is `T_prev - T0`. If the fork's block at height 2015 has timestamp `T_prev = 2_016_000`, the contract computes `actual_time = 2_016_000 - 1_000_000 = 1_016_000` seconds.
5. The **correct** computation using the fork's own ancestor at height 0 would be `2_016_000 - 1_100_000 = 916_000` seconds — a shorter window, requiring **higher** difficulty.
6. The contract instead accepts the fork block at height 2016 with the lower difficulty derived from the inflated window, even though that block would be rejected by the real Bitcoin network.
7. If the attacker's fork accumulates more chainwork than the main chain (easier with the relaxed difficulty), `reorg_chain` is triggered and the contract's canonical chain is replaced with the attacker's fraudulent chain. [8](#0-7) [9](#0-8)

```rust
fn get_next_work_required(
    config: &NetworkConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    if (prev_block_header.block_height + 1) % config.difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            if block_header.time
                > prev_block_header.block_header.time + 2 * config.pow_target_spacing
            {
                return config.proof_of_work_limit_bits;
            }

            let mut current_block_header = prev_block_header.clone();
            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            let last_bits = current_block_header.block_header.bits;
            return last_bits;
        }
        return prev_block_header.block_header.bits;
    }

    let first_block_height =
        prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

    let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
    calculate_next_work_required(
        config,
        prev_block_header,
        interval_tail_extend_header.block_header.time.into(),
    )
}
