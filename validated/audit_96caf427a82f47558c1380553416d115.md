### Title
`makerAccruesTakerFee` Always Returns `false` Due to Impossible `address(0)` Owner Check — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.makerAccruesTakerFee` contains a condition that checks whether the owner address embedded in a subaccount `bytes32` equals `address(0)`. Because all valid Nado subaccounts are constructed from a real `msg.sender`, this condition is permanently false, silently disabling the maker-accrues-taker-fee path in every `matchOrders` execution.

---

### Finding Description

In `OffchainExchange.sol`, the function `makerAccruesTakerFee` is used inside `matchOrders` to decide whether a maker should receive the taker's fee (instead of paying a standard maker fee):

```solidity
// OffchainExchange.sol lines 589–598
function makerAccruesTakerFee(bytes32 subaccount, uint32 productId)
    internal
    view
    returns (bool)
{
    return
        subaccount != X_ACCOUNT &&
        address(uint160(bytes20(subaccount))) == address(0) &&   // ← always false
        getTierFeeRateX18(feeTiers[address(0)], productId).makerRateX18 <=
        TAKER_FEE_ACCRUAL_RATE_X18;
}
``` [1](#0-0) 

The second operand, `address(uint160(bytes20(subaccount))) == address(0)`, extracts the first 20 bytes of the `bytes32` subaccount identifier — which is always the owner's Ethereum address — and compares it to the zero address.

All subaccounts in Nado are created by encoding `msg.sender` into the first 20 bytes:

```solidity
// Endpoint.sol lines 108–110
bytes32 subaccount = bytes32(
    abi.encodePacked(msg.sender, subaccountName)
);
``` [2](#0-1) 

Because `msg.sender` is never `address(0)` for any real deposit or transaction, the condition `address(uint160(bytes20(subaccount))) == address(0)` is permanently false. The third operand (`feeTiers[address(0)]`) is never evaluated due to short-circuit evaluation.

---

### Impact Explanation

Inside `matchOrders`, the result of `makerAccruesTakerFee` controls a critical fee-routing branch:

```solidity
// OffchainExchange.sol lines 780–794
if (makerAccruesTakerFee(maker.order.sender, callState.productId)) {
    ordersInfo.maker.fee = -ordersInfo.taker.fee;
    ordersInfo.maker.quoteDelta =
        ordersInfo.maker.quoteDelta +
        ordersInfo.taker.fee;
} else {
    applyFee(..., false);   // standard maker fee path
}
``` [3](#0-2) 

When `makerAccruesTakerFee` returns `true`, the maker's `quoteDelta` is increased by the taker's fee — the maker receives the taker fee as a rebate. Because the function always returns `false`, every maker is routed through `applyFee` instead. Any maker that was intended to benefit from this arrangement never receives the rebate; the taker fee flows to the protocol's collected fees via `updateCollectedFees` rather than to the maker's balance. This is a concrete, per-trade balance delta corruption: the maker's `quoteDelta` is systematically lower than the protocol intends.

---

### Likelihood Explanation

The broken condition is evaluated on every single `matchOrders` call. There is no configuration, admin action, or user input that can make `address(uint160(bytes20(subaccount))) == address(0)` true for any real subaccount. The feature is unconditionally dead from deployment.

---

### Recommendation

Replace the impossible `address(0)` owner check with the correct condition. Based on the surrounding logic (which reads `feeTiers[address(0)]`), the likely intent was to check whether the maker's own fee tier matches the zero-address tier, i.e.:

```solidity
address makerAddr = address(uint160(bytes20(subaccount)));
feeTiers[makerAddr] == feeTiers[address(0)]
```

or whatever the actual design intent for this special fee path is. The exact fix depends on the intended semantics, but the current check must be corrected.

---

### Proof of Concept

1. Any user calls `Endpoint.depositCollateral` → subaccount is `bytes32(abi.encodePacked(msg.sender, name))` where `msg.sender != address(0)`. [2](#0-1) 
2. Sequencer submits a `MatchOrders` transaction; `matchOrders` is called in `OffchainExchange`. [4](#0-3) 
3. `makerAccruesTakerFee(maker.order.sender, productId)` is evaluated. [5](#0-4) 
4. `address(uint160(bytes20(maker.order.sender)))` is the maker's real Ethereum address (non-zero) → condition is `false`. [6](#0-5) 
5. Execution always falls into the `applyFee` branch; the maker never receives the taker-fee rebate regardless of fee tier configuration. [7](#0-6)
