### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Invariant Required to Prevent 64-Byte Transaction Forgery — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (deprecated since v0.5.0) is still a live, unpermissioned public entry point on the NEAR contract. It omits the coinbase Merkle proof check that `verify_transaction_inclusion_v2` adds specifically to block the 64-byte transaction forgery attack. Any unprivileged NEAR caller can invoke the deprecated function directly, supply a crafted proof whose `tx_id` is an internal Merkle-tree node rather than a real transaction hash, and receive a `true` return value. Downstream NEAR DApps that consume this result are deceived into accepting a forged transaction inclusion proof.

### Finding Description

`verify_transaction_inclusion_v2` was introduced precisely because `verify_transaction_inclusion` is vulnerable to the 64-byte transaction Merkle-proof forgery (documented at the bitmex link cited in the code itself). The fix is the coinbase proof gate: [1](#0-0) 

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
```

This check is entirely absent from `verify_transaction_inclusion`: [2](#0-1) 

The function is still decorated `pub` and `#[pause]` (not `#[private]`). In Rust, `#[deprecated]` is a compile-time lint for Rust callers only — it has zero effect on external NEAR transaction callers. Any NEAR account can call `verify_transaction_inclusion` via a signed transaction today.

The only validation performed is:

1. Confirmations ≤ `gc_threshold`
2. Block hash is in `mainchain_header_to_height`
3. Enough confirmations have accumulated
4. `merkle_proof` is non-empty
5. `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.merkle_root` [3](#0-2) 

Step 5 is the broken invariant: because Bitcoin's Merkle tree is built by hashing 64-byte concatenations of child hashes, an attacker can supply an internal node hash as `tx_id` and a proof path that reconstructs the root correctly. The function returns `true` for a transaction that does not exist.

The analog to the external report is direct: just as `installModule` omits `state++` — leaving the anti-frontrunning invariant silent on that code path while other paths do update it — `verify_transaction_inclusion` omits the coinbase proof check that the v2 path enforces. In both cases, a security invariant is enforced on one code path but silently absent on another that remains reachable.

### Impact Explanation

Any NEAR DApp (bridge, escrow, payment gateway) that calls `verify_transaction_inclusion` to gate an asset release or state change can be fed a forged proof. The attacker does not need to mine a Bitcoin block or break any cryptographic primitive — the 64-byte collision is a structural property of Bitcoin's Merkle construction. The result is unauthorized asset release or state corruption in the consuming contract, proportional to whatever value is gated behind the proof check.

### Likelihood Explanation

- The entry point is public and requires no privileged key.
- The 64-byte attack is well-documented and tooled; crafting a valid internal-node proof for an existing Bitcoin block is straightforward offline work.
- Any integrator that has not migrated to v2 (or that calls v1 directly for any reason) is immediately exploitable.
- The function appears in the contract's public ABI and will be discoverable by any caller enumerating the contract's methods.

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or add `#[private]` to make it callable only by the contract itself (since `verify_transaction_inclusion_v2` already delegates to it internally). The coinbase proof check must be enforced on every externally reachable proof-verification path without exception. [4](#0-3) 

Concretely: change `pub fn verify_transaction_inclusion` to `fn verify_transaction_inclusion` (private) or add `#[private]`, so only `verify_transaction_inclusion_v2` can invoke it. This closes the bypass while preserving the internal delegation pattern already in use.

### Proof of Concept

**Preconditions:** The BTC light client contract is deployed and has a synced mainchain with at least `confirmations` blocks past some target block `B`. The attacker knows the Merkle tree of block `B`.

**Steps:**

1. Attacker selects a confirmed mainchain block `B` whose Merkle tree has depth ≥ 1.
2. Offline, the attacker picks any internal Merkle node `N` at depth 1 (the concatenation of two child hashes, double-SHA256'd). This is a 32-byte value that is a valid node in the tree but corresponds to no real transaction.
3. Attacker constructs a `ProofArgs`:
   - `tx_id` = `N` (the internal node hash)
   - `tx_block_blockhash` = hash of block `B`
   - `tx_index` = the index that places `N` correctly in the proof path
   - `merkle_proof` = the sibling hashes needed to reconstruct the Merkle root from `N`
   - `confirmations` = any value ≤ current depth
4. Attacker calls `verify_transaction_inclusion(args)` on the NEAR contract.
5. `compute_root_from_merkle_proof(N, tx_index, merkle_proof)` reconstructs the correct Merkle root of block `B`.
6. The function returns `true`.
7. The downstream NEAR DApp, trusting this result, releases assets or updates state as if the fake transaction `N` was confirmed in block `B`. [5](#0-4) [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
