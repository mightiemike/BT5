Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End-User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which `MetricOmmPool.swap` populates with its own `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. This creates two failure modes: allowlisted users cannot swap through the router, and once the admin allowlists the router to restore usability, any unprivileged address can bypass the restriction entirely.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes its direct caller as `sender`:**

`MetricOmmPool.swap` at lines 230–231 passes `msg.sender` (the immediate caller) as the first argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // router address when called via MetricOmmSimpleRouter
    recipient,
    ...
);
``` [1](#0-0) 

**`SwapAllowlistExtension.beforeSwap` checks that identity against the allowlist:**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

Here `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`.

**`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly:**

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [3](#0-2) 

The real user's address is stored only in transient callback context for payment purposes; it is never forwarded to the pool or extension as the swap originator. The `extensionData` is passed through from user input but the extension never decodes it — it only reads `sender`.

The same pattern applies to `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165) — all call `pool.swap` with `msg.sender = router`. [4](#0-3) 

**No existing guard compensates for this.** The extension has no `extensionData` decoding, no transient-storage originator lookup, and no fallback identity mechanism.

## Impact Explanation

A pool deploying `SwapAllowlistExtension` to restrict trading to a curated set of addresses (KYC-verified traders, institutional counterparties, whitelisted market makers) loses that restriction entirely once the admin allowlists the router to restore usability. Any unprivileged address can call `exactInputSingle` and execute swaps at oracle-derived bid/ask prices against the pool's full liquidity. LPs who deposited under the assumption that only vetted counterparties could trade are exposed to unauthorized swap execution — a direct loss of LP principal through broken core pool access control. This constitutes broken core pool functionality causing loss of funds and an admin-boundary break where the pool admin's intended restriction is silently voided by the standard periphery.

## Likelihood Explanation

The router is the standard, documented user-facing entry point. Any pool admin who discovers that allowlisted users cannot swap through the router (failure mode 1) will naturally allowlist the router to restore usability, unknowingly opening the bypass (failure mode 2). No special privilege is required: any address can call `MetricOmmSimpleRouter.exactInputSingle`. The trigger is a predictable admin action in response to a usability problem, making exploitation highly likely once the pool is live.

## Recommendation

The end-user's identity must be forwarded through the call chain so the extension can gate on it. Two complementary approaches:

1. **Router encodes originator in `extensionData`**: `MetricOmmSimpleRouter` ABI-encodes `msg.sender` into `extensionData` before calling `pool.swap`. `SwapAllowlistExtension.beforeSwap` decodes and checks that value when `extensionData` is non-empty, preserving the existing direct-pool path's behavior (where `sender` is already the end-user).

2. **Pool exposes originator via transient storage**: `MetricOmmPool.swap` writes `msg.sender` to a transient slot before calling `_beforeSwap`; extensions can read it. This is a more invasive core change but is transparent to all extensions.

The simplest correct fix is option 1: require the router to encode `msg.sender` into `extensionData` and have the extension decode and check that value when present.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the only allowed swapper
  allowedSwapper[pool][router] = false

Step 1 — alice swaps directly (works):
  alice → pool.swap(...)
  sender = alice → allowedSwapper[pool][alice] = true → passes

Step 2 — alice swaps via router (fails):
  alice → router.exactInputSingle({pool, ...})
         → pool.swap(...)   // msg.sender = router
  sender = router → allowedSwapper[pool][router] = false → REVERT

Step 3 — admin allowlists router to restore usability:
  setAllowedToSwap(pool, router, true)

Step 4 — eve (non-allowlisted) bypasses via router:
  eve → router.exactInputSingle({pool, ...})
        → pool.swap(...)   // msg.sender = router
  sender = router → allowedSwapper[pool][router] = true → PASSES
  eve executes swap in restricted pool — allowlist fully bypassed
```

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist only `alice`, confirm `eve` reverts on direct call, then allowlist the router, confirm `eve` succeeds via `exactInputSingle` — demonstrating the bypass.
