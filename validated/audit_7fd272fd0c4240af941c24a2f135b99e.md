Audit Report

## Title
`SwapAllowlistExtension` checks the router address as `sender`, allowing any user to bypass per-user swap restrictions via `MetricOmmSimpleRouter` â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so `sender` is the router address â€” not the actual user. Any pool admin who allowlists the router (required for allowlisted users to trade through it) simultaneously opens the gate to every non-allowlisted user, completely defeating the curation guarantee.

## Finding Description

**Step 1 â€” Pool captures `msg.sender` as `sender`:**

`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

**Step 2 â€” `ExtensionCalling._beforeSwap` forwards `sender` verbatim:**

`_beforeSwap` encodes `sender` directly into the `IMetricOmmExtensions.beforeSwap` call: [2](#0-1) 

**Step 3 â€” Extension checks `sender` against the allowlist:**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Step 4 â€” Router calls `pool.swap()` directly, becoming `msg.sender`:**

`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` with no user identity forwarded: [4](#0-3) 

The same pattern applies to `exactInput` (line 104) and `exactOutputSingle` (line 136). [5](#0-4) 

**Root cause â€” impossible dilemma for pool admin:**

| Router allowlist state | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot use the router at all |
| Router **allowlisted** | Every non-allowlisted user bypasses the guard via the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same.

**Why `DepositAllowlistExtension` is not affected:**

`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` â€” an address explicitly supplied by the caller as a parameter â€” not `sender` (the immediate pool caller). This means the depositor identity is preserved regardless of who calls `addLiquidity`: [6](#0-5) 

## Impact Explanation

A pool deploying `SwapAllowlistExtension` to restrict trading to KYC'd counterparties, institutional participants, or specific protocol addresses is completely defeatable by any user routing through `MetricOmmSimpleRouter`. The non-allowlisted user receives identical swap execution â€” including access to pool liquidity and oracle-priced fills â€” as an allowlisted user. This is a direct break of the admin-enforced curation guarantee and exposes LPs to counterparty risk they explicitly opted out of. This matches the allowed impact gate: **admin-boundary break where an unprivileged path bypasses a pool admin access control**.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard supported periphery path. Any user can call it permissionlessly with no special role, token balance, or prior state required. The bypass requires a single `exactInputSingle` call targeting the restricted pool. The only precondition is that the pool admin has allowlisted the router (which is necessary for any allowlisted user to use the router), making the bypass automatic once the router is enabled.

## Recommendation

The extension must identify the **economic actor** â€” the address whose funds are at risk â€” not the immediate caller of `pool.swap`. Two viable approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.
2. **Make the extension router-aware**: The extension checks `sender` when `sender != router`; when `sender == router`, it requires the router to attest the real user via an authenticated claim in `extensionData`.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` â€” only Alice is allowed.
3. Admin calls `setAllowedToSwap(pool, router, true)` â€” necessary so Alice can use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. `beforeSwap` receives `sender = router`; checks `allowedSwapper[pool][router]` â†’ `true`.
7. Bob's swap executes successfully despite not being on the allowlist.

If the admin does **not** add the router in step 3, Alice also cannot use the router â€” confirming the impossible dilemma. A Foundry integration test can reproduce this by deploying the pool with the extension, setting up the allowlist as above, and asserting that Bob's `exactInputSingle` call succeeds.
