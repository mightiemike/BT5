### Title
Slow Mode `LinkSigner` Transactions Have No Expiration, Allowing Stale Signer Updates to Overwrite Current Linked Signer State - (File: `core/contracts/EndpointTx.sol`)

---

### Summary
Slow mode transactions in Nado's `Endpoint` are queued with only a minimum execution delay (`executableAt = block.timestamp + SLOW_MODE_TX_DELAY`) but have no upper expiry bound. A `LinkSigner` slow mode transaction submitted by a user can sit in the queue indefinitely. If the user later revokes a linked signer via the fast path (sequencer), the stale slow mode `LinkSigner` transaction will eventually execute and silently overwrite `linkedSigners[sender]` back to the old (revoked) address, restoring unauthorized signing authority.

---

### Finding Description
The `SlowModeTx` struct stores only `executableAt` (a minimum unlock time), with no `expireAt` field:

```solidity
struct SlowModeTx {
    uint64 executableAt;
    address sender;
    bytes tx;
}
```

When a slow mode transaction is enqueued, the only time constraint set is a 3-day minimum delay:

```solidity
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
    sender: sender,
    tx: transaction
});
```

Execution only enforces the lower bound:

```solidity
require(
    fromSequencer || (txn.executableAt <= block.timestamp),
    ERR_SLOW_TX_TOO_RECENT
);
```

There is no upper bound check and no cancellation mechanism. The queue is strictly FIFO (`txUpTo` increments by one per execution). When a slow mode `LinkSigner` transaction eventually reaches the front of the queue, `processSlowModeTransactionImpl` unconditionally overwrites the current linked signer:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.LinkSigner)
    );
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

No check is made against the current value of `linkedSigners[txn.sender]` or whether the queued signer is still the user's intended signer.

---

### Impact Explanation
When the stale `LinkSigner` slow mode transaction executes, `linkedSigners[sender]` is overwritten with the old (revoked) signer address. A linked signer is trusted to sign `WithdrawCollateral`, `TransferQuote`, `MintNlp`, `BurnNlp`, `LiquidateSubaccount`, and order matching transactions on behalf of the subaccount. Restoring a revoked signer gives that address full signing authority over the victim's subaccount, enabling collateral theft via `WithdrawCollateral` or `WithdrawCollateralV2`, unauthorized position changes, and unauthorized quote transfers. The corrupted state is: `linkedSigners[victim_subaccount]` = revoked address A instead of the user's current intended signer B.

---

### Likelihood Explanation
Any user who has ever submitted a slow mode `LinkSigner` transaction and subsequently changed their linked signer via the fast path is affected once the stale tx reaches the front of the FIFO queue. The sequencer processes slow mode txs via `ExecuteSlowMode` in strict FIFO order, and any unprivileged caller can also trigger execution via `executeSlowModeTransaction()` after the 3-day delay. There is no cancellation mechanism. The attacker (the revoked signer address A) is incentivized to monitor the queue and call `executeSlowModeTransaction()` at the right moment to restore their own signing authority. This is a realistic, low-effort attack requiring no special privileges.

---

### Recommendation
Add an `expireAt` field to `SlowModeTx` (or embed an expiry in the transaction payload for `LinkSigner`). In `processSlowModeTransactionImpl`, reject execution of any slow mode transaction whose expiry has passed. A reasonable upper bound (e.g., 30 days after `executableAt`) would prevent stale state mutations. Additionally, consider adding a user-callable cancellation function that allows the original sender to remove their own queued slow mode transaction before it executes.

---

### Proof of Concept

1. User calls `submitSlowModeTransaction` with a `LinkSigner` payload setting `signer = address A`. The tx is enqueued at position N with `executableAt = now + 3 days`. The user pays the $1 slow mode fee.

2. The sequencer processes other transactions. The user decides to revoke address A and submits a fast-path `LinkSigner` transaction (via `processTransactionImpl`) setting `signer = address B`. `linkedSigners[user_subaccount]` is now B.

3. The slow mode queue eventually reaches position N (could be days, weeks, or months later). Anyone — including address A — calls `Endpoint.executeSlowModeTransaction()`.

4. `_executeSlowModeTransaction` checks `txn.executableAt <= block.timestamp` (satisfied after 3 days) and calls `processSlowModeTransaction` → `processSlowModeTransactionImpl`.

5. The `LinkSigner` branch executes: `linkedSigners[user_subaccount] = address(A)`. The revoked signer A is now restored.

6. Address A calls `submitTransactionsChecked` (via sequencer) or uses the fast path to submit a `WithdrawCollateralV2` signed by A, draining the user's collateral to an arbitrary `sendTo` address. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
