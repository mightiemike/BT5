### Title
Zero Confirmations Bypass in Transaction Inclusion Verification — (`File: contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` field with no lower-bound enforcement. Passing `confirmations = 0` makes the confirmation-depth guard trivially true for any `u64` arithmetic, causing the contract to certify a transaction as "confirmed" with zero block depth — directly analogous to passing `0` as the minimum-received-amount in a cross-chain swap.

---

### Finding Description

`ProofArgs` and `ProofArgsV2` both carry a `confirmations: u64` field. [1](#0-0) 

Inside `verify_transaction_inclusion`, the only guard on `confirmations` is an **upper-bound** check:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [2](#0-1) 

The actual depth check that follows is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [3](#0-2) 

Because `confirmations` is `u64`, when the caller supplies `0`, the expression evaluates to `any_u64 >= 0`, which is **always true** in Rust. The guard is completely bypassed. No lower-bound `require!(args.confirmations >= 1, ...)` exists anywhere in the function.

`verify_transaction_inclusion_v2` — the current, non-deprecated entry point — delegates directly to `verify_transaction_inclusion` after its coinbase proof check, inheriting the same flaw: [4](#0-3) 

---

### Impact Explanation

A recipient contract (e.g., a bridge or escrow) that calls `verify_transaction_inclusion_v2` and trusts its `true` return value to release funds will accept a transaction with **zero confirmation depth** — i.e., a block that was just submitted and has not been buried under any subsequent work. Such a block is trivially reversible by a chain reorganization. An attacker who controls the relayer (or who submits a competing fork) can submit a fraudulent block, immediately call the verification function with `confirmations = 0`, receive `true`, trigger the downstream action (e.g., token mint or fund release), and then allow the fraudulent block to be orphaned. The corrupted state is the verification result itself: the contract certifies inclusion in a block that may never become canonical.

---

### Likelihood Explanation

The `confirmations` field is fully caller-controlled and is not validated on-chain beyond the upper-bound check. Any unprivileged NEAR account can call `verify_transaction_inclusion_v2` with `confirmations = 0`. No privileged role, leaked key, or social engineering is required. The only prerequisite is that the attacker can submit a valid Merkle proof for a transaction in any mainchain block — which is the normal, intended use of the API. Likelihood is **high** for any downstream consumer that does not independently enforce a minimum confirmation count.

---

### Recommendation

Add an explicit lower-bound guard at the top of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "Confirmations must be at least 1");
```

Alternatively, enforce a protocol-level minimum (e.g., 6 for Bitcoin) and reject any call that requests fewer. The fix must be applied before the Merkle proof computation, mirroring the pattern used for the upper-bound check.

---

### Proof of Concept

1. Deploy the contract with any valid genesis chain.
2. Submit one block header via `submit_blocks` — the block is now the mainchain tip at height `H`.
3. Call `verify_transaction_inclusion_v2` with:
   - `tx_block_blockhash` = hash of the block at height `H` (the tip, zero blocks deep)
   - `confirmations = 0`
   - a valid Merkle proof for any transaction in that block
4. The depth check evaluates `(H - H) + 1 = 1 >= 0` → `true`.
5. The function returns `true`, certifying the transaction as confirmed despite having zero confirmation depth.
6. A downstream contract acting on this result would release funds for a transaction that can still be reorganized away.

The root cause is the absence of a lower-bound check on `confirmations` in `contract/src/lib.rs` at the `verify_transaction_inclusion` function entry. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );
```
