### Title
`verify_transaction_inclusion_v2` Validates Coinbase Merkle Proof Without Binding `coinbase_tx_id` to the Actual Coinbase Transaction — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery attack by requiring a coinbase proof of equal length to the target transaction proof. However, the function validates only that the caller-supplied `coinbase_tx_id` hashes to a valid Merkle path at position 0 — it never checks that `coinbase_tx_id` is the actual coinbase transaction of the block. An unprivileged NEAR caller can supply an internal Merkle tree node as `coinbase_tx_id`, constructing a proof of a shorter depth that satisfies all on-chain checks, and thereby cause the function to return `true` for a fabricated transaction inclusion claim.

---

### Finding Description

`verify_transaction_inclusion_v2` performs three checks:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — proof lengths must match.
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == merkle_root` — the coinbase proof must be valid at position 0.
3. `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == merkle_root` — the target tx proof must be valid. [1](#0-0) 

The design intent is: if the real coinbase (always a leaf at depth N) is proven at position 0 with a proof of length N, then any proof of the same length N must also reach a leaf, ruling out internal-node forgeries (which would need a shorter proof). This is the standard SPV mitigation for the 64-byte attack referenced in the code comments. [2](#0-1) 

The broken invariant: the function never verifies that `coinbase_tx_id` is the actual coinbase transaction. The contract stores only `ExtendedHeader` (which contains `merkle_root`, `block_hash`, `chain_work`, `block_height`) — it does not store the coinbase transaction hash. [3](#0-2) 

Because `coinbase_tx_id` is fully attacker-controlled, the attacker can supply any internal Merkle tree node as `coinbase_tx_id`. An internal node at depth `d` (where `d < N`, the full tree depth) is a valid node in the tree, so a Merkle proof of length `d` for it at position 0 will correctly reconstruct `merkle_root`. The length-equality check then constrains the target `tx_id` proof to also have length `d` — but the attacker can equally supply another internal node at depth `d` as `tx_id`, with a valid proof of length `d`. All three checks pass, and the function returns `true`.

The `compute_root_from_merkle_proof` function in `merkle-tools` is a pure path-reconstruction function with no leaf/internal-node distinction: [4](#0-3) 

This is structurally identical to the external report's bug: `validate_dealing_support` validated the signature over the modified dealing content without checking it matched the original dealing. Here, `verify_transaction_inclusion_v2` validates the Merkle proof over the attacker-supplied `coinbase_tx_id` without checking it matches the actual coinbase transaction.

---

### Impact Explanation

Any downstream NEAR contract or application that calls `verify_transaction_inclusion_v2` to gate an action on confirmed Bitcoin transaction inclusion (e.g., cross-chain bridge unlocks, oracle attestations, custody releases) can be deceived into accepting a fabricated proof. The function returns `true` for an internal Merkle tree node that was never a real Bitcoin transaction, corrupting the proof result — the exact corrupted value identified by the audit methodology.

---

### Likelihood Explanation

Medium. The attacker needs a target Bitcoin block where an internal Merkle tree node at some depth `d` can be interpreted as a meaningful payload (the 64-byte attack). This is a known, documented attack class for Bitcoin SPV clients. The attacker does not need any privileged role, private key, or social engineering — only the ability to call a public NEAR contract method with crafted `ProofArgsV2` arguments. The entry path is fully permissionless. [5](#0-4) 

---

### Recommendation

The contract must bind `coinbase_tx_id` to the actual coinbase transaction of the block. Since the contract does not store coinbase hashes, the recommended fix is to require the caller to supply the raw coinbase transaction bytes, compute `double_sha256` of them on-chain, and use that computed hash as `coinbase_tx_id` in the Merkle proof check. This ensures the proven content is the actual coinbase transaction, not an attacker-chosen internal node — closing the same content-binding gap that the external report recommends closing in `validate_dealing_support`.

---

### Proof of Concept

Given a real Bitcoin block with Merkle tree of depth `N` and `merkle_root = R`:

1. Let `Y` = the internal node at depth `d` (where `d < N`), position 0. `Y` is the hash of the left subtree root at depth `d`. Construct `coinbase_merkle_proof_d` of length `d` that proves `Y` at position 0 against `R`. This proof is valid because `Y` is a real node in the tree.

2. Let `X` = any internal node at depth `d` at some position `p`. Construct `merkle_proof_d` of length `d` that proves `X` at position `p` against `R`. This proof is also valid.

3. Call `verify_transaction_inclusion_v2` with:
   - `tx_id = X` (internal node, not a real transaction)
   - `tx_block_blockhash` = the block's hash (already in the contract's mainchain)
   - `tx_index = p`
   - `merkle_proof = merkle_proof_d` (length `d`)
   - `coinbase_tx_id = Y` (internal node, not the real coinbase)
   - `coinbase_merkle_proof = coinbase_merkle_proof_d` (length `d`)
   - `confirmations = 1`

4. Check 1 passes: `d == d`. Check 2 passes: `compute_root(Y, 0, coinbase_merkle_proof_d) == R`. Check 3 passes: `compute_root(X, p, merkle_proof_d) == R`. The function returns `true` for a transaction that does not exist. [1](#0-0) [4](#0-3)

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
