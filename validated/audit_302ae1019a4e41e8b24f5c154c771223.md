### Title
Stale `UpdatePrice` Transaction Applied Without Timestamp Validation on Sequencer Batch Retry — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `UpdatePrice` transaction struct carries no timestamp field, and `processTransactionImpl` applies no recency check before writing `priceX18[productId]`. If a sequencer batch containing an `UpdatePrice` payload fails due to an out-of-gas error and is resubmitted unchanged, the stale price from the original batch silently overwrites the current oracle price. This is the direct Nado analog of the `PushOracleReceiver.handle()` stale-data replay bug.

---

### Finding Description

`IEndpoint.UpdatePrice` is defined as:

```solidity
struct UpdatePrice {
    uint32 productId;
    int128 priceX18;
}
``` [1](#0-0) 

There is no `time` or `timestamp` field. When `processTransactionImpl` handles a `TransactionType.UpdatePrice` transaction, it unconditionally calls `clearinghouse.updatePrice(transaction)` and writes the result into `priceX18[productId]`:

```solidity
} else if (txType == IEndpoint.TransactionType.UpdatePrice) {
    (uint32 productId, int128 newPriceX18) = clearinghouse.updatePrice(transaction);
    if (productId != 0) {
        priceX18[productId] = newPriceX18;
    }
}
``` [2](#0-1) 

`Clearinghouse.updatePrice` similarly performs no recency check — it only validates `priceX18 > 0`:

```solidity
require(txn.priceX18 > 0, ERR_INVALID_PRICE);
engine.updatePrice(txn.productId, txn.priceX18);
``` [3](#0-2) 

**Contrast with `SpotTick` / `PerpTick`:** Both tick transaction types carry a `time` field:

```solidity
struct PerpTick { uint128 time; int128[] avgPriceDiffs; }
struct SpotTick  { uint128 time; }
``` [4](#0-3) 

Their handlers compute `dt = txn.time - t.spotTime/perpTime` and pass it to `updateStates`, which enforces `require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME)`: [5](#0-4) [6](#0-5) 

No equivalent staleness guard exists for `UpdatePrice`.

**Replay window via `nSubmissions`:** `submitTransactionsChecked` increments `nSubmissions` inside the transaction loop:

```solidity
for (uint256 i = 0; i < transactions.length; i++) {
    processTransaction(transaction);
    nSubmissions += 1;
}
``` [7](#0-6) 

If the call reverts (e.g., OOG), all increments are rolled back and `nSubmissions` stays at its pre-call value. `validateSubmissionIdx` then requires the next submission to use the same `idx`:

```solidity
function validateSubmissionIdx(uint64 idx) private view {
    require(idx == nSubmissions, ERR_INVALID_SUBMISSION_INDEX);
}
``` [8](#0-7) 

The sequencer's only valid next action is to resubmit with the same `idx`. If automated retry logic resubmits the original byte-identical batch, the `UpdatePrice` payload it contains carries the price from the original (now-stale) construction time and is applied without rejection.

---

### Impact Explanation

`priceX18[productId]` is the canonical oracle price consumed by:

- **Liquidations** — `getHealth()` uses `_risk(productId).priceX18` (set by `engine.updatePrice`) to compute initial and maintenance health; a stale low price can make healthy accounts appear underwater (wrongful liquidation) or a stale high price can hide insolvent accounts.
- **NLP mint/burn** — `mintNlp` and `burnNlp` use `oraclePriceX18` to compute NLP token amounts; a stale price directly corrupts the quote↔NLP exchange rate. [9](#0-8) 

- **Perp funding** — `PerpEngineState.updateStates` reads `_risk(productId).priceX18` as the index price for funding rate computation. [10](#0-9) 

Corrupted state delta: `priceX18[productId]` and `states[productId].priceX18` in both engines are set to a stale value, directly affecting every downstream health and settlement calculation.

---

### Likelihood Explanation

Low-to-medium. The trigger requires:
1. A sequencer batch containing `UpdatePrice` to fail due to OOG.
2. The sequencer's retry path to resubmit the same byte payload without refreshing the price.

Automated sequencer retry logic that resubmits the exact serialized batch (common in off-chain sequencer implementations) satisfies condition 2 without any malicious intent. The sequencer is not compromised — it is acting correctly by retrying a failed submission. The protocol provides no on-chain mechanism to reject the stale price.

---

### Recommendation

Add a `uint128 time` field to `UpdatePrice`:

```solidity
struct UpdatePrice {
    uint32 productId;
    int128 priceX18;
    uint128 time;
}
```

In `processTransactionImpl`, before applying the update, verify recency analogously to `SpotTick`:

```solidity
Times memory t = times;
require(txn.time > t.spotTime, ERR_INVALID_TIME);
```

This ensures that a resubmitted batch carrying an old `UpdatePrice` payload is rejected if a more recent tick has already advanced the protocol clock past the price's timestamp.

---

### Proof of Concept

1. `nSubmissions = N`. Current on-chain ETH price: `$2100`.
2. Sequencer constructs Batch A (`idx = N`) containing `UpdatePrice(ETH, $2000)` (price at batch-construction time T1).
3. Sequencer calls `submitTransactionsChecked(N, batchA, ...)`. The call reverts due to OOG mid-batch. `nSubmissions` remains `N`.
4. Real ETH price moves to `$2200`. No other batch can be submitted until `idx = N` is consumed.
5. Sequencer's automated retry resubmits the identical `batchA` bytes with `idx = N`.
6. `submitTransactionsChecked` succeeds. `processTransactionImpl` processes `UpdatePrice(ETH, $2000)` — no timestamp check — and writes `priceX18[ETH] = $2000`.
7. On-chain oracle price is now `$2000` while the true market price is `$2200`. Liquidators can now wrongfully liquidate accounts that are healthy at `$2200` but appear underwater at `$2000`.
