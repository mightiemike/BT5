### Title
`TrieKey::BufferedReceipt` Encodes `receiving_shard` as `u16` While `ShardId` Is `u64`, Causing Node Panic Under Dynamic Resharding — (`File: core/primitives/src/trie_key.rs`)

---

### Summary

`TrieKey::BufferedReceipt` hard-codes the `receiving_shard` field into a 2-byte `u16` in the protocol trie key. `ShardId` is a `u64` wrapper. Under `ProtocolFeature::DynamicResharding`, shard IDs are allocated monotonically (`max + 1`, `max + 2` per split). Once any live shard ID exceeds `u16::MAX` (65 535), every attempt to buffer a cross-shard receipt to that shard hits an unconditional `assert!` that panics the node.

---

### Finding Description

`ShardId` is defined as a `u64` newtype:

```rust
// core/primitives-core/src/types.rs
pub struct ShardId(u64);
```

`ShardLayoutV3::derive_impl` allocates new shard IDs as:

```rust
// core/primitives/src/shard_layout/v3.rs  line 271-273
let max_shard_id = *shard_ids.iter().max()…;
let new_shards = vec![max_shard_id + 1, max_shard_id + 2];
```

So shard IDs grow by 2 per split, unbounded in the `u64` domain.

The trie key serialization for buffered receipts is:

```rust
// core/primitives/src/trie_key.rs  lines 526-534
TrieKey::BufferedReceipt { index, receiving_shard } => {
    let receiving_shard = *receiving_shard;
    buf.push(col::BUFFERED_RECEIPT);
    // Use  u16 for shard id to reduce depth in trie.
    let receiving_shard: u64 = receiving_shard.into();
    assert!(receiving_shard <= u16::MAX as u64, "Shard ID too big.");
    let receiving_shard: u16 = receiving_shard as u16;
    buf.extend(&receiving_shard.to_le_bytes());
    buf.extend(&index.to_le_bytes());
}
```

The `assert!` is unconditional and unrecoverable — it panics the process. There is no `Result` return, no graceful error, and no protocol-level gate that prevents a shard with ID > 65 535 from receiving buffered receipts.

The `len()` arm for the same variant already bakes in `size_of::<u16>()`:

```rust
// core/primitives/src/trie_key.rs  lines 412-416
TrieKey::BufferedReceipt { index, .. } => {
    col::BUFFERED_RECEIPT.len()
        + std::mem::size_of::<u16>()
        + std::mem::size_of_val(index)
}
```

This is a frozen protocol schema: changing the width would alter every existing trie key for buffered receipts, requiring a migration.

A secondary, silent variant of the same class exists in `ShardUId::new`:

```rust
// core/primitives/src/shard_layout/mod.rs  line 485-487
pub fn new(version: ShardVersion, shard_id: ShardId) -> Self {
    Self { version, shard_id: shard_id.into() }  // Into<u32>: self.0 as u32
}
```

`ShardUId.shard_id` is `u32`; the `Into<u32>` impl silently truncates:

```rust
// core/primitives-core/src/types.rs  lines 156-159
impl Into<u32> for ShardId {
    fn into(self) -> u32 {
        self.0 as u32   // no bounds check
    }
}
```

`ShardUId` is the 8-byte DB key prefix for all state, flat-state, and trie-changes columns. Silent truncation after 2³² splits would cause two distinct live shards to share the same DB prefix, corrupting all state reads and writes for those shards.

---

### Impact Explanation

**`TrieKey::BufferedReceipt` (u16 limit):** Any node that tracks a shard whose ID exceeds 65 535 and receives a cross-shard receipt destined for another high-ID shard will panic unconditionally inside `append_into`. Because `append_into` is called on every trie write for buffered receipts, the node cannot process any chunk for that shard. All validators tracking the affected shard crash; the shard stalls. Impact: **High** (liveness failure for affected shard).

**`ShardUId` (u32 truncation):** After 2³² splits, two distinct `ShardId` values produce the same `u32`, so `ShardUId` equality, DB key prefixes, and Borsh-serialized epoch data all become wrong. State from two shards merges in the DB. Impact: **Critical** (silent state corruption).

---

### Likelihood Explanation

Starting from the current mainnet maximum shard ID (~3), reaching `u16::MAX` requires ≈ 32 766 splits. At one split per epoch (≈12 h on mainnet) that is ≈ 45 years. The u32 overflow requires ≈ 2 billion splits. Both thresholds are far beyond any near-term horizon.

However, the structural invariant is broken today: the protocol type (`ShardId`, `u64`) and the trie-key encoding (`u16`) are mismatched with no enforcement at the type level, no protocol-version gate, and no migration plan. The `assert!` is the only guard, and it terminates the process rather than returning an error. As the external report notes, what seems safely distant today can become reachable as the system evolves.

---

### Recommendation

1. **`TrieKey::BufferedReceipt`**: Replace the `u16` encoding with `u64` behind a protocol-version gate. Introduce a migration that rewrites existing `BufferedReceipt` trie keys from the old 2-byte shard prefix to the new 8-byte prefix at the epoch boundary where the new version activates. Remove the `assert!` and the `Into<u16>` impl for `ShardId`.

2. **`ShardUId`**: Change `shard_id: u32` to `shard_id: u64` (or keep `u32` and add a checked conversion with a protocol-enforced upper bound on shard IDs). The Borsh layout of `ShardUId` is part of the DB schema, so this requires a DB migration or a new `ShardUId` version.

3. **`Into<u32>` and `Into<u16>` for `ShardId`**: Replace both with `TryInto` returning `Result`, forcing every call site to handle the out-of-range case explicitly.

---

### Proof of Concept

**Invariant break (u16):**

```
Initial mainnet shard IDs: {0, 1, 2, 3}  →  max = 3
After split N:  max = 3 + 2*N
N = 32_766  →  max = 65_535 = u16::MAX
N = 32_767  →  max = 65_537 > u16::MAX

TrieKey::BufferedReceipt { receiving_shard: ShardId(65_537), index: 0 }
  → receiving_shard as u64 = 65_537
  → assert!(65_537 <= 65_535)  ← PANIC: "Shard ID too big."
```

Every node tracking any shard that tries to buffer a receipt to shard 65 537 crashes.

**Invariant break (u32 / ShardUId):**

```
ShardId(0x0000_0001_0000_0000_u64)  →  Into<u32>  →  0x0000_0000_u32
ShardId(0x0000_0000_0000_0000_u64)  →  Into<u32>  →  0x0000_0000_u32

ShardUId::new(v, ShardId(2^32))  ==  ShardUId::new(v, ShardId(0))
  → identical DB key prefix
  → state reads/writes for shard 2^32 silently alias shard 0
