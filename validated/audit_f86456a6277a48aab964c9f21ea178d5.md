### Title
Unused `referral` Parameter in `depositCollateralWithReferral` Silently Drops Caller-Supplied Referral Code — (File: `core/contracts/Endpoint.sol`)

---

### Summary
`Endpoint.depositCollateralWithReferral` accepts a `string memory` referral code as its fourth parameter, but the parameter is completely unnamed and never read anywhere in the function body — not in logic, not in events, not in the slow-mode transaction payload. Any caller who supplies a referral code expecting it to be recorded or attributed receives no error and no effect; the value is silently discarded.

---

### Finding Description
`depositCollateralWithReferral` is declared as:

```solidity
function depositCollateralWithReferral(
    bytes32 subaccount,
    uint32 productId,
    uint128 amount,
    string memory        // ← unnamed, never referenced
) public {
``` [1](#0-0) 

The function body constructs a `DepositCollateral` slow-mode transaction that encodes only `subaccount`, `productId`, and `amount`:

```solidity
tx: abi.encodePacked(
    uint8(TransactionType.DepositCollateral),
    abi.encode(
        DepositCollateral({
            sender: subaccount,
            productId: productId,
            amount: amount
        })
    )
)
``` [2](#0-1) 

The referral string is absent from the payload, from every event, and from every downstream call. The protocol itself acknowledges the concept of referral codes — `depositCollateral` passes a `DEFAULT_REFERRAL_CODE` constant into this function: [3](#0-2) 

And `DirectDepositV1.creditDeposit` passes the literal `"-1"` as a referral code: [4](#0-3) 

Both callers demonstrate that the parameter is expected to carry meaningful information, yet the receiving function discards it unconditionally.

---

### Impact Explanation
**Low.** The deposit itself is processed correctly; collateral is credited to the intended subaccount. However, any referral attribution, fee rebate, or off-chain incentive that depends on the referral code being recorded is silently lost. A user who calls `depositCollateralWithReferral` with a non-default referral code — expecting to receive or grant a referral benefit — will never receive an error and will never receive the benefit. The broken invariant is: *the referral code supplied by the caller is not the referral code recorded by the protocol*.

---

### Likelihood Explanation
**High.** The function is `public` and is the canonical deposit entry point for any caller who wants to supply a referral code. The existence of `DEFAULT_REFERRAL_CODE` and the `"-1"` literal in `DirectDepositV1` confirms the parameter is actively passed by multiple callers. Every such call silently fails to record the referral.

---

### Recommendation
Either:
1. Include the referral code in the slow-mode transaction payload and emit it in a `DepositCollateral` event so off-chain systems can attribute it; or
2. Remove the parameter entirely if referral tracking is not implemented, so callers are not misled into believing their referral code is being used.

---

### Proof of Concept
1. Alice calls `depositCollateralWithReferral(subaccount, productId, amount, "alice_referral")`.
2. The function executes without revert.
3. The slow-mode transaction payload contains only `{sender, productId, amount}` — `"alice_referral"` is absent.
4. No event records the referral code.
5. Any referral program expecting `"alice_referral"` to appear on-chain never sees it; Alice receives no referral credit.
