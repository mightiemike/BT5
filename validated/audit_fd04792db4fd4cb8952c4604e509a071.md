### Title
Missing Liquidator Health Check After `_finalizeSubaccount` Returns `true` — (`core/contracts/ClearinghouseLiq.sol`)

### Summary

`liquidateSubaccountImpl` returns immediately when `_finalizeSubaccount` returns `true`, skipping the only liquidator health guard (`require(!isUnderInitial(txn.sender), ERR_SUBACCT_HEALTH)`), which lives exclusively inside `_handleLiquidationPayment`. A liquidator at borderline initial health can call with `productId == type(uint32).max` against a liquidatee with closed perp positions and positive `vQuoteBalance`, causing PnL settlement to drain the liquidator's QUOTE spot balance below initial health with no revert.

### Finding Description

**Entry point:** `liquidateSubaccountImpl` is called with `txn.productId == type(uint32).max`.

**Control flow:** [1](#0-0) 

When `_finalizeSubaccount` returns `true`, the function executes `return` at line 626 — `_handleLiquidationPayment` is never called.

**The only liquidator health check** in the entire contract is: [2](#0-1) 

This check is inside `_handleLiquidationPayment`, which is only reached when `_finalizeSubaccount` returns `false`.

**PnL settlement inside `_finalizeSubaccount`:** When the liquidatee has closed perp positions (`amount == 0`) with positive `vQuoteBalance`, the loop at lines 322–338 calls `_settlePnlAgainstLiquidator`: [3](#0-2) 

Line 269 decreases the liquidator's QUOTE spot balance by `pnl`. If the liquidator started near the initial health boundary, this deduction pushes them below initial health. `_finalizeSubaccount` then returns `true` at line 412: [4](#0-3) 

`liquidateSubaccountImpl` returns immediately with no post-settlement health assertion on `txn.sender`.

### Impact Explanation

The protocol invariant — *a liquidator must remain at or above initial health after any liquidation action* — is broken. The liquidator absorbs the liquidatee's positive PnL (receiving `vQuoteBalance` credit) while paying USDC from their spot balance. If that USDC payment drops the liquidator below initial health, the protocol does not revert. The liquidator now holds a degraded position that can itself become under-maintenance, creating bad debt that must be socialized or covered by the insurance fund. This satisfies the Critical scope: incorrect liquidation outcome and bad debt creation.

### Likelihood Explanation

The preconditions are reachable in normal protocol operation:
- A liquidatee with zero perp `amount` and positive `vQuoteBalance` is a standard end-state after perp positions are closed out during a multi-step liquidation sequence.
- A liquidator at borderline initial health is not unusual — liquidators often operate with leveraged positions.
- `productId == type(uint32).max` is the documented finalization signal; no special privilege is required to pass it.

### Recommendation

Add a liquidator health check immediately after `_finalizeSubaccount` returns `true`, before the early `return`:

```solidity
if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
    require(
        txn.sender == N_ACCOUNT || !isUnderInitial(txn.sender),
        ERR_SUBACCT_HEALTH
    );
    if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
        IOffchainExchange(
            IEndpoint(getEndpoint()).getOffchainExchange()
        ).tryCloseIsolatedSubaccount(txn.liquidatee);
    }
    return;
}
```

This mirrors the existing guard in `_handleLiquidationPayment` and closes the gap.

### Proof of Concept

1. Deploy `Clearinghouse` with `ClearinghouseLiq` via `delegatecall`.
2. Set up **liquidator** with QUOTE spot balance just sufficient to be at initial health (e.g., health == 1 wei above threshold).
3. Set up **liquidatee** under maintenance health with: all perp `amount == 0`, one perp with `vQuoteBalance > 0` (positive unrealized PnL), and QUOTE spot balance near zero so it remains under maintenance after PnL settlement.
4. Call `liquidateSubaccountImpl` with `txn.productId == type(uint32).max`.
5. `_finalizeSubaccount` passes the `amount == 0` check, settles the positive `vQuoteBalance` via `_settlePnlAgainstLiquidator`, decreasing the liquidator's QUOTE spot balance by `pnl`, and returns `true`.
6. `liquidateSubaccountImpl` returns at line 626 with no revert.
7. Assert `isUnderInitial(txn.sender) == true` — the liquidator is now below initial health with no protocol enforcement. [1](#0-0) [5](#0-4) [6](#0-5)
