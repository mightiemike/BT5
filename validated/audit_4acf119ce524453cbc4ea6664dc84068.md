### Title
Permanent `deployer` Role in `ContractOwner` Never Renounced, Enabling Unauthorized Product Delisting — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.initialize()` stores the deployer address permanently in the `deployer` state variable and provides no mechanism to renounce or transfer this role post-deployment. While ownership is immediately transferred to a multisig, the `deployer` role persists indefinitely and gates `delistProduct()`, which submits a `DelistProduct` slow-mode transaction with fully caller-controlled product IDs, settlement prices, and subaccounts — bypassing multisig governance entirely.

---

### Finding Description

In `ContractOwner.initialize()`, the deployer address is stored permanently:

```solidity
function initialize(address multisig, address _deployer, ...) external initializer {
    require(_deployer == msg.sender, "expected deployed to initialize");
    __Ownable_init();
    transferOwnership(multisig);   // ownership transferred to multisig
    deployer = _deployer;          // deployer role stored permanently — never renounced
    ...
}
``` [1](#0-0) 

Ownership is correctly transferred to the multisig at line 60, but `deployer = _deployer` at line 61 permanently records the deployer address. There is no `renounceDeployer()`, `updateDeployer()`, or equivalent function anywhere in the contract.

The `onlyDeployer` modifier gates five functions: [2](#0-1) 

The most impactful is `delistProduct()`:

```solidity
function delistProduct(
    uint32[] calldata productIds,
    int128[] calldata pricesX18,
    bytes32[] calldata subaccounts
) external onlyDeployer {
``` [3](#0-2) 

This function submits a `DelistProduct` slow-mode transaction directly to the `Endpoint` with fully caller-controlled `pricesX18` and `subaccounts`. Unlike all other sensitive protocol operations (e.g., `withdrawInsurance`, `addNlpPool`, `spotUpdateRisk`), which are gated by `onlyOwner` (the multisig), `delistProduct()` bypasses multisig governance entirely and is executable by the deployer alone.

The broken invariant: after `initialize()` completes, the deployer hot wallet should hold no privileged role. Instead, it permanently retains the ability to delist any product at an arbitrary settlement price.

---

### Impact Explanation

A deployer acting maliciously or whose key is compromised can call `delistProduct()` with:
- Arbitrary `productIds` — targeting any active spot or perp product
- Arbitrary `pricesX18` — setting a manipulated settlement price that deviates from the true market price
- Arbitrary `subaccounts` — specifying which subaccounts are settled at that price

This submits a `DelistProduct` slow-mode transaction (3-day delay) to the `Endpoint`. If the multisig does not cancel it within the window, the product is delisted at the attacker-chosen price, causing direct asset loss to users holding positions in that product. The deployer can also call `clearSpotAddOrUpdateProductCalls()` and `clearPerpAddOrUpdateProductCalls()` to silently discard pending product configuration queued by the legitimate team. [4](#0-3) 

---

### Likelihood Explanation

The deployer is a hot wallet used during deployment — inherently less secure than the multisig. The role persists indefinitely with no expiry or revocation path. Any future key rotation, infrastructure compromise, or insider threat targeting the deployer key creates a permanent, unmitigated attack vector. The likelihood is **low-medium**: the deployer key compromise is not routine, but the design flaw creates unnecessary and permanent exposure that the protocol's own governance model (multisig for all sensitive ops) was designed to eliminate.

---

### Recommendation

Add a `renounceDeployer()` function callable only by the current deployer, and invoke it at the end of the deployment script after all product setup is complete:

```solidity
function renounceDeployer() external onlyDeployer {
    deployer = address(0);
}
```

Alternatively, migrate `delistProduct()` to `onlyOwner` (multisig) so it is subject to the same governance controls as all other sensitive protocol operations.

---

### Proof of Concept

1. Deploy `ContractOwner` and call `initialize(multisig, deployerEOA, ...)`.
2. Observe that `deployer == deployerEOA` and `owner() == multisig`.
3. From `deployerEOA`, call:
   ```solidity
   contractOwner.delistProduct(
       [targetProductId],
       [manipulatedPriceX18],   // e.g., 1 wei — far below market
       [victimSubaccounts]
   );
   ```
4. A `DelistProduct` slow-mode transaction is queued in `Endpoint.slowModeTxs` with a 3-day delay.
5. If the multisig does not respond within 3 days, `executeSlowModeTransaction()` settles all victim positions at the manipulated price, causing direct asset loss. [5](#0-4) [3](#0-2)
