### Title
`ENABLE_SECP_OPS` Flag Bypassed via 4-Byte Opcode Encoding — (`File: src/chia_dialect.rs`)

---

### Summary

The `ENABLE_SECP_OPS` flag is intended to gate access to `op_secp256k1_verify` and `op_secp256r1_verify`. The 1-byte opcode dispatch path (opcodes 64 and 65) correctly checks this flag. However, the 4-byte opcode dispatch path explicitly maps `0x13d61f00` and `0x1c3a8f00` to the same secp functions **without** checking `ENABLE_SECP_OPS`. An attacker-controlled CLVM program using the 4-byte encoding can invoke secp operations regardless of whether the flag is set, breaking the invariant that `ENABLE_SECP_OPS` controls access to these operators.

---

### Finding Description

In `ChiaDialect::op()`, there are two separate dispatch paths for secp operations:

**1-byte path (correctly gated):**
```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [1](#0-0) 

**4-byte path (ungated):**
```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
``` [2](#0-1) 

The 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` are explicitly recognized and dispatched to the secp verification functions with no `ENABLE_SECP_OPS` check. The `ENABLE_SECP_OPS` flag definition confirms it is intended to control access to these operators:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

Critically, `ENABLE_SECP_OPS` is absent from `MEMPOOL_MODE`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

This means that in mempool mode (or any mode without `ENABLE_SECP_OPS`), the 1-byte opcodes 64/65 are correctly rejected, but the 4-byte opcodes `0x13d61f00`/`0x1c3a8f00` silently succeed and execute the full secp verification logic.

---

### Impact Explanation

**Consensus divergence / hardfork gate bypass.** Before the secp hardfork activates (i.e., before `ENABLE_SECP_OPS` is set by the full node), a CLVM program using the 4-byte opcode encoding can invoke secp operations and succeed. A node running without `ENABLE_SECP_OPS` will:
- Reject `(64 ...)` / `(65 ...)` — correctly
- Accept `(0x13d61f00 ...)` / `(0x1c3a8f00 ...)` — incorrectly

This creates a split: nodes that have activated the secp hardfork and nodes that have not will disagree on the validity of programs using the 4-byte encoding. The corrupted result is a `Response` that should have been `Err(EvalErr::Unimplemented)` but instead returns a successful secp verification result.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM byte sequence submitted to `run_program` (via the Python wheel's `run_serialized_chia_program`, or directly via the Rust API) can encode the 4-byte opcode. The 4-byte encoding is valid CLVM — it is a 4-byte atom used as an operator. An attacker who knows the opcode values `0x13d61f00` and `0x1c3a8f00` can craft such a program trivially. This is not a theoretical path; it is a direct consequence of the dispatch table in `ChiaDialect::op()`.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that both opcode encodings are uniformly gated by the same flag, eliminating the bypass path.

---

### Proof of Concept

A CLVM program encoding `(0x13d61f00 <pubkey> <msg> <sig>)` submitted to a `ChiaDialect` instantiated **without** `ENABLE_SECP_OPS` will:
1. Enter `ChiaDialect::op()` with `op_len == 4`
2. Match `0x13d61f00 => op_secp256k1_verify` at line 176
3. Execute `op_secp256k1_verify` and return a successful `Response`

The same program encoded as `(64 <pubkey> <msg> <sig>)` (1-byte opcode) would fall through to `unknown_operator` and return `Err(EvalErr::Unimplemented)` because `ENABLE_SECP_OPS` is not set and the `if flags.contains(ClvmFlags::ENABLE_SECP_OPS)` guard at line 248 fails. [5](#0-4) [1](#0-0)
