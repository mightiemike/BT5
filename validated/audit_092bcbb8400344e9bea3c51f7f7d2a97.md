### Title
`max_l2_gas_amount` Upper Bound Not Enforced for Declare Transactions in Gateway Stateless Validation - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The gateway's `StatelessTransactionValidator` enforces a `max_l2_gas_amount` upper bound on the `l2_gas.max_amount` resource bound for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. This creates an asymmetric admission-control invariant: a Declare transaction with an arbitrarily large `l2_gas.max_amount` (e.g., `u64::MAX`) passes stateless validation and is admitted to the mempool, while an equivalent Invoke or DeployAccount transaction is rejected at the gateway.

### Finding Description

In `validate_resource_bounds` inside `StatelessTransactionValidator`, the check against `config.max_l2_gas_amount` is guarded by an explicit early-return for Declare transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The production default for `max_l2_gas_amount` is `1,210,000,000` gas: [2](#0-1) 

The test `valid_l2_gas_amount_on_declare` explicitly documents and asserts this bypass: a Declare transaction with `max_amount: GasAmount(200)` passes validation even when `max_l2_gas_amount: 100` is configured: [3](#0-2) 

The same check is correctly applied to Invoke and DeployAccount transactions: [4](#0-3) 

No compensating check exists in the stateful validator. The stateful `validate_resource_bounds` only checks the L2 gas *price* (not the gas *amount*) against the previous block's price: [5](#0-4) 

### Impact Explanation

A Declare transaction with `l2_gas.max_amount` set to any value above `max_l2_gas_amount` (including `u64::MAX`) passes all gateway validation stages and is admitted to the mempool. The `max_l2_gas_amount` limit exists to prevent transactions from claiming more gas than the block can accommodate (`receipt_l2_gas` block cap = 5,800,000,000 gas per the bouncer config): [6](#0-5) 

With the check bypassed, a Declare transaction claiming `l2_gas.max_amount = u64::MAX` is admitted to the mempool. During execution, the blockifier's fee check uses the claimed `max_amount` as the upper bound for actual gas usage — with `u64::MAX` this bound never triggers, so the transaction executes with an effectively uncapped gas budget up to the block's physical limit. This is the direct Starknet analog of the Optimism H01 bug: Sequencer-submitted transactions (Declare) bypass the size/resource limit that is enforced for the other admission path (Invoke/DeployAccount).

**Impact category:** High — Mempool/gateway admission accepts invalid Declare transactions that violate the `max_l2_gas_amount` admission invariant enforced for all other transaction types.

### Likelihood Explanation

Any unprivileged user can submit a Declare transaction via the public RPC endpoint with an arbitrarily large `l2_gas.max_amount`. No special access is required. The bypass is unconditional for all Declare transactions regardless of the configured `max_l2_gas_amount` value.

### Recommendation

Remove the Declare-specific exemption in `validate_resource_bounds` and apply the `max_l2_gas_amount` check uniformly to all transaction types, including Declare. The existing TODO comment acknowledges this gap:

```rust
// Remove the Declare exemption:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

The `max_l2_gas_amount` config value may need to be tuned for Declare transactions if their legitimate gas requirements differ from Invoke/DeployAccount, but the check must not be absent.

### Proof of Concept

1. Construct a valid `RpcDeclareTransactionV3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and a valid Sierra contract class.
2. Submit it to the gateway's `add_tx` endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never returned for Declare transactions regardless of the configured `max_l2_gas_amount`.
4. The transaction proceeds through stateful validation and is inserted into the mempool with an uncapped L2 gas claim.

The existing test already proves the bypass:

```rust
