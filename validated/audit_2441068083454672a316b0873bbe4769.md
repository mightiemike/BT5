Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End User, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the `pool.swap()` call. When `MetricOmmSimpleRouter` intermediates a swap, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router to support router-based swaps for permitted users, every address—including those not on the allowlist—can bypass the restriction by routing through the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
  msg.sender,   // ← always the direct caller of pool.swap()
  recipient,
  ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this `sender` value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. The `recipient` parameter (second argument) is silently discarded:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
  }
  ...
}
``` [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The `IMetricOmmExtensions.beforeSwap` interface exposes both `sender` and `recipient`, but the production extension only uses `sender`: [5](#0-4) 

**Exploit flow:** Admin deploys pool with `SwapAllowlistExtension` and allowlists the router (`allowedSwapper[pool][router] = true`) so that permitted users can swap via the router. The extension now evaluates `allowedSwapper[pool][router]` → `true` for every caller of the router, regardless of who they are. Any unpermitted address calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`, the router calls `pool.swap(attacker_address, ...)`, the pool passes `sender = router` to the extension, the extension approves, and the swap executes.

## Impact Explanation
The allowlist is completely neutralized for the router path. Any unprivileged address can execute swaps against a curated pool that was intended to restrict access to a specific set of addresses. The attacker receives output tokens and pays input tokens—a full economic interaction the allowlist was designed to prevent. This constitutes broken core pool functionality (access control bypass on swap) causing unauthorized economic interaction with the pool.

## Likelihood Explanation
Both preconditions are routine operational choices: (a) deploying `SwapAllowlistExtension` to restrict swaps, and (b) allowlisting the router so that permitted users can use the documented periphery entry point. Any pool satisfying both conditions is immediately exploitable by any address without any privileged access, malicious setup, or special knowledge. The `MetricOmmSimpleRouter` is the standard supported swap interface.

## Recommendation
The extension must authenticate the end user, not the intermediary. The simplest router-agnostic fix is to gate on `recipient` instead of `sender`, since `recipient` is the economically relevant actor receiving output tokens and cannot be spoofed by an intermediary:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
  }
  return IMetricOmmExtensions.beforeSwap.selector;
}
```

Alternatively, require callers (including the router) to encode the actual user address in `extensionData` and decode it in the extension, but this requires router cooperation and is more complex.

## Proof of Concept
1. Deploy pool with `SwapAllowlistExtension` as `extension1`; set `beforeSwap` order to extension 1.
2. Pool admin calls `swapExtension.setAllowedToSwap(pool, router, true)` — necessary to let any permitted user swap via the router.
3. Attacker (address not on the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: attacker, ...})`.
4. Router calls `pool.swap(recipient=attacker, ...)` — pool's `msg.sender` is the router.
5. Pool calls `_beforeSwap(sender=router, recipient=attacker, ...)`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
7. Swap executes; attacker receives output tokens.
8. The per-user allowlist provided zero protection.

A Foundry test can confirm this by: deploying the pool+extension, setting `allowedSwapper[pool][router]=true`, then calling `exactInputSingle` from an address that was never individually allowlisted and asserting the swap succeeds.
