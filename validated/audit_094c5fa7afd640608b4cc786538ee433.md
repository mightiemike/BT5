## Analysis

I need to trace the full attack path through the protobuf converter, storage serializer, and hash computation to determine if the divergence is concrete and reachable.

### Step 1: The Protobuf Heuristic

The converter at `crates/apollo_protobuf/src/converters/transaction.rs` applies: [1](#0-0) 

For a wire message with `l2_gas=0` and `l1_data_gas=0`, the condition at line 431 fires and the result is `ValidResourceBounds::L1Gas(l1_gas)` — even if the original transaction was signed as `AllResources`.

### Step 2: The Hash Divergence Is Concrete

`get_tip_resource_bounds_hash` branches on the enum variant: [2](#0-1) 

- For `L1Gas`: hashes `[tip, concat(L1Gas,X), concat(L2Gas,0)]` — **3 Poseidon inputs**
- For `AllResources(l2=0, l1_data=0)`: hashes `[tip, concat(L1Gas,X), concat(L2Gas,0), concat(L1DataGas,0)]` — **4 Poseidon inputs**

These produce different field elements. The divergence is not theoretical — Poseidon over 3 elements ≠ Poseidon over 4 elements even when the 4th is zero.

### Step 3: The Reachable Attack Path

The path does **not** require a malicious peer. An unprivileged user can trigger it:

1. **RPC submission**: `RpcInvokeTransactionV3` always uses `AllResourceBounds` (not `ValidResourceBounds`). A user submits `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. [3](#0-2) 

The gateway converts this to `ValidResourceBounds::AllResources(...)` and computes hash **H_AllResources** (4-element Poseidon).

2. **Block propagation**: The sequencer serializes the stored `AllResources` transaction to protobuf via `From<ValidResourceBounds> for protobuf::ResourceBounds`: [4](#0-3) 

For `AllResources(l2=0, l1_data=0)`, this emits `l2_gas=Some(0)` and `l1_data_gas=Some(0)`.

3. **P2P sync deserialization**: The syncing node calls `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`: [5](#0-4) 

Which calls `ValidResourceBounds::try_from(protobuf::ResourceBounds)` — the heuristic fires, producing `L1Gas(X)`.

4. **Transaction hash not validated**: The syncing node stores the peer-provided hash (H_AllResources) without recomputing it: [6](#0-5)

### Title
Protobuf `ValidResourceBounds` heuristic silently downcasts `AllResources(l2=0, l1_data=0)` to `L1Gas`, producing a stored transaction body whose recomputed hash diverges from the stored hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary
The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter applies a backward-compatibility heuristic: if `l2_gas == 0 && l1_data_gas == 0`, it returns `L1Gas` regardless of whether the original transaction was signed as `AllResources`. Because `get_tip_resource_bounds_hash` hashes a different number of Poseidon inputs for each variant (3 vs 4), the recomputed hash of the stored body diverges from the hash that was computed at submission time and stored alongside the body.

### Finding Description
`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` at line 431 of `crates/apollo_protobuf/src/converters/transaction.rs` converts any wire message with `l2_gas=0` and `l1_data_gas=0` to `ValidResourceBounds::L1Gas`, even when the originating node serialized an `AllResources` variant. The `From<ValidResourceBounds> for protobuf::ResourceBounds` serializer (lines 471–490) faithfully emits `l2_gas=Some(0)` and `l1_data_gas=Some(0)` for `AllResources(l2=0, l1_data=0)`, so the round-trip is lossy. The P2P sync client stores the peer-provided transaction hash without recomputing it (line 88–89 of `crates/apollo_p2p_sync/src/client/transaction.rs`), so the stored body (discriminant `L1Gas=0`) is permanently inconsistent with the stored hash (computed under `AllResources=1` semantics).

### Impact Explanation
- **Hash divergence**: Any code path that calls `get_invoke_transaction_v3_hash` on the stored body — including `validate_transaction_hash`, RPC transaction serving, and the OS execution hint loader (`valid_resource_bounds_as_felts` with `exclude_l1_data_gas=false`) — produces H_L1Gas ≠ H_AllResources. The difference is concrete: Poseidon over 3 field elements vs. 4.
- **Gas/execution divergence**: `get_gas_vector_computation_mode` returns `NoL2Gas` for `L1Gas` and `All` for `AllResources`. Any re-execution on the syncing node (e.g., in a multi-sequencer or OS-proof context) uses wrong gas semantics, causing incorrect fee, bouncer, and resource accounting.
- **RPC divergence**: The syncing node serves `resource_bounds` with the `L1Gas` representation while the sequencer's view is `AllResources`, producing an authoritative-looking wrong value for RPC clients.

### Likelihood Explanation
Any unprivileged user can trigger this by submitting a V3 transaction with `l2_gas=0` and `l1_data_gas=0` (a valid and economically rational choice for pre-0.13.3 style transactions). The transaction is accepted by the gateway, included in a block, and propagated via P2P. Every syncing node that receives the block will silently downcast the variant. No special peer access or operator privilege is required.

### Recommendation
1. **Remove the heuristic**: Once support for 0.13.2 wire format is dropped (as noted by the existing TODO at line 426), assert that `l1_data_gas` is always present and remove the `if l1_data_gas.is_zero() && l2_gas.is_zero()` branch. The variant should be determined by the presence/absence of the `l1_data_gas` field, not by its value.
2. **Validate transaction hashes on sync**: Implement the missing validation noted by `// TODO(eitan): Validate transaction hash from untrusted sources` in `crates/apollo_p2p_sync/src/client/transaction.rs`. Recompute the transaction hash from the deserialized body and reject the peer if it does not match the provided hash.
3. **Add a round-trip invariant test**: Assert that `AllResources(l2=0, l1_data=0)` survives a protobuf round-trip with the same variant discriminant.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

pub fn valid_resource_bounds_as_felts(
    resource_bounds: &ValidResourceBounds,
    exclude_l1_data_gas: bool,
) -> Result<Vec<ResourceAsFelts>, StarknetApiError> {
    let mut resource_bounds_vec: Vec<_> = vec![
        ResourceAsFelts::new(Resource::L1Gas, &resource_bounds.get_l1_bounds())?,
        ResourceAsFelts::new(Resource::L2Gas, &resource_bounds.get_l2_bounds())?,
    ];
    if exclude_l1_data_gas {
        return Ok(resource_bounds_vec);
    }
    if let ValidResourceBounds::AllResources(AllResourceBounds { l1_data_gas, .. }) =
        resource_bounds
    {
        resource_bounds_vec.push(ResourceAsFelts::new(Resource::L1DataGas, l1_data_gas)?)
    }
    Ok(resource_bounds_vec)
}
