### Title
`SwapAllowlistExtension` gates the router address instead of the actual swapper, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is documented as gating `swap` **by swapper address, per pool**. Its `beforeSwap` hook checks `sender` — the value the pool passes as `msg.sender` at the pool call boundary. When `MetricOmmSimpleRouter` intermediates the swap, `msg.sender` at the pool is the **router contract**, not the end user. A pool admin who allowlists the router (the natural action to enable router-mediated swaps for legitimate users) simultaneously opens the gate for every non-allowlisted address, because the router is now unconditionally approved for all callers it forwards.

---

### Finding Description

`MetricOmmPool.swap` captures `msg.sender` and forwards it as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value as the first positional argument of `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then uses that `sender` argument — together with `msg.sender` (the pool) — to look up the per-pool allowlist: [3](#0-2) 

When `MetricOmmSimpleRouter` calls `pool.swap(...)`, `msg.sender` at the pool boundary is the **router address**. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

The asymmetry with `DepositAllowlistExtension` is instructive: `addLiquidity` accepts `owner` as an explicit parameter, so the liquidity adder can pass the real user address and the allowlist check is correct. `swap` has no such parameter — `sender` is always `msg.sender` at the pool, making the router the checked identity. [4](#0-3) 

The NatDoc for `SwapAllowlistExtension` states "Gates `swap` by swapper address, per pool," but the implementation gates by **immediate pool caller**, which is the router when the periphery is used. [5](#0-4) 

---

### Impact Explanation

A pool admin who configures a curated pool with `SwapAllowlistExtension` and then allowlists the router (to let approved users reach the pool through the standard periphery path) inadvertently grants every address the ability to swap. Any non-allowlisted user routes through `MetricOmmSimpleRouter`; the extension sees `sender = router`, finds the router in the allowlist, and passes. The curated pool's access control is fully neutralised. Unauthorized traders can extract LP value at oracle-derived prices, violating the pool's intended curation policy and causing direct loss of LP principal.

---

### Likelihood Explanation

The trigger is a standard, expected admin action: allowlisting the router so that approved users can use the supported periphery path. No malicious setup is required. Any non-allowlisted user who discovers the bypass can exploit it immediately by calling `MetricOmmSimpleRouter` rather than the pool directly. The periphery is the primary user-facing entry point, making this path highly reachable.

---

### Recommendation

`SwapAllowlistExtension.beforeSwap` must check the **end user**, not the immediate pool caller. Two sound approaches:

1. **Pass the real user through the router**: Have `MetricOmmSimpleRouter` encode the originating user address in `extensionData` and have the extension decode and verify it — but this requires the extension to trust the encoding, which is fragile.

2. **Check `tx.origin` as a fallback** (not recommended for general use, but acceptable in a controlled periphery context).

3. **Preferred — mirror the `addLiquidity` pattern**: Add a `swapper` parameter to `IMetricOmmPoolActions.swap` (analogous to `owner` in `addLiquidity`) that the pool passes through to the extension. The router sets this to `msg.sender` before calling the pool. The extension then checks `allowedSwapper[pool][swapper]` against the real user identity.

---

### Proof of Concept

```solidity
// Pool admin sets up a curated pool with SwapAllowlistExtension.
// Only `approvedUser` is supposed to be able to swap.
swapAllowlist.setAllowedToSwap(address(pool), approvedUser, true);

// Admin also allowlists the router so approvedUser can use the periphery.
swapAllowlist.setAllowedToSwap(address(pool), address(router), true);

// attacker is NOT allowlisted.
// Direct call to pool reverts:
vm.prank(attacker);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(attacker, false, int128(1000), type(uint128).max, callbackData, "");

// But routing through MetricOmmSimpleRouter succeeds:
// pool.swap() sees msg.sender = router → allowedSwapper[pool][router] = true → passes.
vm.prank(attacker);
router.exactInputSingle(/* pool, attacker, zeroForOne, amount, ... */);
// attacker successfully swaps in the curated pool, bypassing the allowlist.
``` [6](#0-5)
