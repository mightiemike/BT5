### Title
`DelegateAction` Signing Domain Contains No Chain-Specific Binding, Enabling Cross-Fork Meta-Transaction Replay - (File: core/primitives/src/action/delegate.rs)

### Summary

The `DelegateAction` (meta-transaction) signing domain is a fixed NEP discriminant with no chain-specific identifier. A user's signature over a `DelegateAction` is valid on every NEAR chain that shares the same account state, allowing a relayer to replay the user's signed payload on a fork without the user's knowledge or consent. Regular `Transaction` objects are protected by a `block_hash` field inside the signed body; `DelegateAction` has no equivalent binding.

### Finding Description

`DelegateAction::get_nep461_hash()` computes the hash that the meta-transaction sender signs:

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("failed to serialize");
    hash(&bytes)
}
``` [1](#0-0) 

`SignableMessage` prepends a fixed 4-byte discriminant derived from the NEP number:

```
discriminant = MIN_ON_CHAIN_DISCRIMINANT + NEP_366_META_TRANSACTIONS
             = (1 << 30) + 366
             = 0x4000016E  (same on every NEAR network)
``` [2](#0-1) [3](#0-2) 

The `DelegateAction` payload itself contains only `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — none of which are chain-specific: [4](#0-3) 

The `VersionedDelegateActionPayload::get_nep461_hash()` path (used by `DelegateActionV2` / `Action::DelegateV2`) has the identical structure, substituting discriminant `(1<<30)+611`: [5](#0-4) 

By contrast, every regular `Transaction` (V0 and V1) includes a `block_hash` field inside the Borsh-serialized body that is signed: [6](#0-5) [7](#0-6) 

The outer transaction that carries a `DelegateAction` is signed by the **relayer**, not the user. The relayer freely chooses a `block_hash` from whichever chain they submit to. The user's inner signature covers only the `DelegateAction` body, which has no chain anchor.

### Impact Explanation

If a NEAR hard fork occurs and both chains share the same account state at the fork point (identical access-key nonces, identical balances):

1. Alice signs a `DelegateAction` on mainnet (e.g., transferring 1000 NEAR to Bob, `nonce=N`, `max_block_height=H`).
2. The fork occurs before block `H`; both chains have Alice's access key at nonce `N-1`.
3. A relayer on the fork constructs a fresh outer `Transaction` with a valid `block_hash` from the fork chain and embeds Alice's original `DelegateAction` + signature unchanged.
4. `SignedDelegateAction::verify()` passes on the fork because the hash `(1<<30)+366 || borsh(DelegateAction)` is identical on both chains.
5. Alice's 1000 NEAR is transferred on the fork without her consent.

Any action type is replayable: `Transfer`, `AddKey`, `DeleteKey`, `FunctionCall`, `DeleteAccount`. The severity scales with the value of the actions Alice signed.

### Likelihood Explanation

Requires a post-deployment fork that retains the same account state. NEAR has undergone multiple protocol upgrades; a contentious hard fork (analogous to ETH/ETC) is a realistic future event. The attack requires only a relayer who holds a previously submitted `DelegateAction` — relayer services routinely buffer signed payloads. No privileged role is needed beyond possessing the already-broadcast signed payload.

### Recommendation

**Short-term:** Add a chain-specific field to the `DelegateAction` signing domain. The cleanest approach is to include the genesis block hash (already available as `GenesisId::hash`) inside `SignableMessage` or as an explicit field in `DelegateAction`/`DelegateActionV2`. This mirrors how `TransactionV0`/`V1` bind to a specific chain via `block_hash`.

**Long-term:** Standardize a `chain_id` or genesis-hash binding in NEP-461/NEP-611 so all future signable message types inherit chain-domain separation by default.

### Proof of Concept

```
// Mainnet: Alice signs a DelegateAction
let delegate_action = DelegateAction {
    sender_id: "alice.near",
    receiver_id: "bob.near",
    actions: [Transfer { deposit: 1000 NEAR }],
    nonce: 42,
    max_block_height: 999_999,
    public_key: alice_pk,
};
// hash = sha256( borsh( 0x4000016E || delegate_action ) )
// -- identical on mainnet AND any fork with same account state --
let alice_sig = alice_sk.sign(delegate_action.get_nep461_hash());

// Fork occurs at block 500_000; both chains have alice nonce=41.

// Fork chain: malicious relayer replays with a fork-valid block_hash
let outer_tx = Transaction::V0(TransactionV0 {
    block_hash: fork_block_hash,   // valid on fork, chosen by relayer
    signer_id: "relayer.near",
    actions: [Action::Delegate(SignedDelegateAction {
        delegate_action,           // Alice's original payload
        signature: alice_sig,      // Alice's original signature — still valid
    })],
    ...
});
// Runtime accepts: SignedDelegateAction::verify() passes because
// get_nep461_hash() produces the same bytes on the fork.

pub struct SignedDelegateAction {

}

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
}
