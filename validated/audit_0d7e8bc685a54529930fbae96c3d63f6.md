### Title
Orphaned Fork Headers Are Permanently Locked in `headers_pool` With No Removal Path — (`contract/src/lib.rs`)

---

### Summary

Fork block headers submitted via `submit_blocks` that never accumulate enough chainwork to trigger a reorg are stored permanently in `headers_pool`. The GC mechanism (`run_mainchain_gc`) only iterates over `mainchain_height_to_header` and therefore never touches orphaned fork entries. No admin or public function exists to evict them. This is a direct structural analog to the "unpurchased INTX locked after vesting begins" bug: a state transition (fork submission without promotion) leaves assets (NEAR storage slots and their associated deposits) permanently stranded with no recovery path.

---

### Finding Description

When `submit_block_header_inner` determines that an incoming header does not extend the current mainchain tip, it routes the header through `store_fork_header`:

```rust
// contract/src/lib.rs  lines 560, 665-667
self.store_fork_header(&current_header);
...
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [1](#0-0) [2](#0-1) 

`store_fork_header` writes **only** into `headers_pool`. It does **not** write into `mainchain_height_to_header` or `mainchain_header_to_height`. If the fork's chainwork does not exceed the current mainchain tip's chainwork, `reorg_chain` is never called and the fork headers remain exclusively in `headers_pool`. [3](#0-2) 

The GC function `run_mainchain_gc` removes headers by iterating over `mainchain_height_to_header`:

```rust
// contract/src/lib.rs  lines 401-414
for height in start_removal_height..end_removal_height {
    let blockhash = &self
        .mainchain_height_to_header
        .get(&height)
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
    self.remove_block_header(blockhash);
    self.mainchain_height_to_header.remove(&height);
}
``` [4](#0-3) 

Because orphaned fork headers were never inserted into `mainchain_height_to_header`, GC never reaches them. `remove_block_header` is never called for them, so their entries in `headers_pool` are never freed. [5](#0-4) 

Scanning the entire public API of `BtcLightClient` — `submit_blocks`, `run_mainchain_gc`, `get_*`, `verify_*`, `migrate` — there is no function that accepts a fork block hash and removes it from `headers_pool`. [6](#0-5) 

---

### Impact Explanation

Every `ExtendedHeader` stored in `headers_pool` occupies NEAR on-chain storage. `submit_blocks` enforces a storage deposit to cover the cost of new entries:

```rust
// contract/src/lib.rs  lines 182-188
let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());
require!(amount >= required_deposit, ...);
``` [7](#0-6) 

For orphaned fork headers, that storage is never freed. The NEAR tokens backing those storage bytes are permanently locked inside the contract with no mechanism to release them. Over time, as natural Bitcoin forks are relayed, `headers_pool` grows without bound while `gc_threshold` only bounds the mainchain portion. The contract's total storage is therefore unbounded, and the associated NEAR deposits are irrecoverable.

---

### Likelihood Explanation

Bitcoin (and Litecoin, Dogecoin, Zcash) produce natural short forks continuously. The relayer is designed to detect and submit competing chains to allow the contract to track the heaviest chain. Every submitted fork that loses the chainwork competition leaves orphaned entries. This is not an adversarial edge case — it is the normal operating mode of the relayer. The likelihood of orphaned fork headers accumulating is **high** under any production deployment.

---

### Recommendation

Add a privileged (e.g., `DAO`-gated) function that accepts a list of block hashes, verifies each is present in `headers_pool` but absent from `mainchain_header_to_height` (i.e., confirmed orphan), and calls `remove_block_header` on each. This mirrors the `withdrawUnallocatedIntx` fix applied in the referenced report: provide an explicit recovery path for stranded state once it is confirmed unreachable by normal protocol flow.

---

### Proof of Concept

1. Contract is initialized with a genesis block at height H.
2. Relayer submits mainchain blocks H+1 … H+N (all stored via `store_block_header`, indexed in `mainchain_height_to_header`).
3. Relayer detects a competing fork and submits fork blocks F1, F2, F3 branching from H+K (K < N). Each is routed through `store_fork_header` → inserted into `headers_pool` only.
4. Fork's cumulative chainwork < mainchain tip's chainwork → `reorg_chain` is not called → F1, F2, F3 remain in `headers_pool` with no entry in `mainchain_height_to_header`.
5. `run_mainchain_gc` is called (automatically inside `submit_blocks` or directly). It iterates `mainchain_height_to_header` from `start_removal_height` to `end_removal_height`. F1, F2, F3 are invisible to this loop.
6. F1, F2, F3 remain in `headers_pool` indefinitely. The NEAR storage deposits paid for them are permanently locked. Repeating steps 3–5 with successive losing forks causes unbounded storage growth with no recovery path. [8](#0-7) [2](#0-1)

    /// Public call to run GC on a mainchain.
    /// `batch_size` is how many block headers should be removed in the execution
    ///
    /// # Panics
    /// If initial blockheader or tip blockheader are not in a header pool
    #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
