Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of actual end user, allowing full allowlist bypass via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates pool swaps by checking `sender`, which is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` equals the router address, not the actual end user. If the pool admin allowlists the router — the natural and necessary action to enable any router-mediated swap — every unprivileged, non-allowlisted user can bypass the allowlist entirely. This is structurally inconsistent with `DepositAllowlistExtension`, which correctly checks `owner` (the actual position beneficiary) and therefore works correctly through the router.

## Finding Description

**Root cause — wrong identity checked in `beforeSwap`:**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`sender` is populated by `MetricOmmPool.swap`, which passes `msg.sender` of the `pool.swap()` call:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` then forwards this value directly to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, recipient, ...)  // sender = msg.sender of pool.swap()
    )
);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` with `msg.sender` inside the pool being the **router address**:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,   // actual user as recipient
        params.zeroForOne,
        ...
    );
```

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Structural inconsistency with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` — the actual position beneficiary — not `sender`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Because `addLiquidity` passes the actual user as `owner`, the deposit allowlist works correctly through the router. The swap allowlist does not.

**Exploit path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to specific counterparties.
2. Pool admin calls `setAllowedToSwap(pool, address(router), true)` to enable standard router-mediated swaps — a necessary action, since without it no user can swap through the router at all.
3. Any unprivileged, non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. The router calls `pool.swap(recipient=user, ...)` with `msg.sender` = router.
5. The extension evaluates `allowedSwapper[pool][router]` = `true` → swap proceeds.
6. The non-allowlisted user has fully bypassed the allowlist.

The pool admin faces an impossible choice: either allowlist the router (allowing all users to bypass the allowlist) or do not allowlist the router (making the router unusable for all users, including allowlisted ones).

**Existing guards are insufficient:** There are no secondary checks in `beforeSwap`, `pool.swap`, or the router that would re-verify the actual caller's identity against the allowlist. The `allowAllSwappers` escape hatch only makes the problem worse (it disables the allowlist entirely). The `isAllowedToSwap` view function is not called on-chain during swap execution.

**Exact wrong value:** The extension decision (`allowedSwapper[pool][sender]`) is evaluated against the router address instead of the actual end user, causing the access control gate to return `true` for any user routing through an allowlisted router.

## Impact Explanation

The swap allowlist is rendered completely ineffective whenever the router is allowlisted. Any user — regardless of allowlist status — can execute swaps in a pool intended to be restricted to specific counterparties. In pools designed for institutional or permissioned trading (e.g., with specific price agreements or regulatory requirements), this allows unauthorized parties to extract value at pool-set prices, directly harming LP principal and disrupting the pool's intended economic model. This is an admin-boundary break: an unprivileged path (router) bypasses a configured access control gate. Severity: High — the core access control invariant of `SwapAllowlistExtension` is broken under the expected operational configuration.

## Likelihood Explanation

The trigger condition — allowlisting the router — is the natural and expected configuration for any pool that wants to support standard user-facing swaps. A pool admin who configures `SwapAllowlistExtension` to restrict swaps to specific addresses will almost certainly also allowlist the router to enable normal operation, unaware that doing so voids the allowlist entirely. The inconsistency with `DepositAllowlistExtension` (which works correctly through the router) reinforces the expectation that the swap allowlist should behave the same way. The exploit requires no special privileges, no flash loans, and no unusual token behavior — any EOA or contract can trigger it by calling the public router.

## Recommendation

Replace the `sender` check with a check on the actual end user. The `recipient` argument (second parameter, currently unnamed/ignored) is the address that receives swap output and is set to the actual user by the router. This mirrors the pattern used by `DepositAllowlistExtension`, which checks `owner` (the actual beneficiary) rather than the direct caller:

```solidity
// Proposed fix: check recipient (actual end user) instead of sender
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Alternatively, require the caller to pass the actual user identity in `extensionData` and verify it against the allowlist, though this requires router-side changes and is more complex.

## Proof of Concept

```solidity
// Setup: pool configured with SwapAllowlistExtension
// Pool admin allowlists the router (necessary for any router-mediated swap):
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attacker (not individually allowlisted) calls:
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    tokenIn: token0,
    tokenOut: token1,
    pool: pool,
    recipient: attacker,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    extensionData: ""
}));
// Result: swap succeeds — attacker bypassed the allowlist
// Extension checked allowedSwapper[pool][router] == true, not allowedSwapper[pool][attacker]

// Verify: direct pool.swap() by attacker (not through router) correctly reverts:
// pool.swap(attacker, ...) → sender = attacker → allowedSwapper[pool][attacker] == false → revert NotAllowedToSwap
// But through router: sender = router → allowedSwapper[pool][router] == true → swap proceeds
```

A Foundry integration test can confirm this by: (1) deploying a pool with `SwapAllowlistExtension`, (2) allowlisting only the router, (3) calling `router.exactInputSingle` from an unallowlisted EOA, and asserting the swap succeeds. Then calling `pool.swap` directly from the same EOA and asserting it reverts with `NotAllowedToSwap`.
