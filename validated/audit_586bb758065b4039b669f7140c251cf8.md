Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on `owner` (beneficiary) instead of `sender` (actual caller/payer), allowing any unlisted address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of the pool call and token payer) and validates only `owner` (the position beneficiary). Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender` pays but need not equal `owner`, any address not on the allowlist can call `addLiquidity` with a listed address as `owner`, pass the guard, and inject liquidity into a curated pool while paying with their own tokens.

## Finding Description
`MetricOmmPool.addLiquidity` passes two distinct identities to the extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the actual caller and token payer; `owner` is the position beneficiary supplied by the caller. These are explicitly decoupled by the operator pattern.

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but discards it (unnamed `address,`), checking only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The contract's own NatSpec states it "Gates `addLiquidity` by depositor address" — but the implementation gates by the beneficiary address (`owner`), not the depositor (`sender`). The `SwapAllowlistExtension` correctly gates on `sender` (first argument), demonstrating the intended pattern was not followed here.

An unlisted attacker `bob` calls `pool.addLiquidity(alice, salt, deltas, ...)` where `alice` is listed. The pool calls `_beforeAddLiquidity(bob, alice, ...)`. The extension checks `allowedDepositor[pool][alice]` → true → passes. Bob pays tokens via the modify-liquidity callback; alice receives LP shares she did not request. The pool's curation invariant is broken.

Additionally, `removeLiquidity` enforces `msg.sender == owner`, so the injected shares are permanently locked under alice's key — alice cannot cleanly exit her own position without also withdrawing the injected shares.

## Impact Explanation
The deposit allowlist — a production extension intended to restrict LP participation to a curated set of addresses — is completely defeated. Any address can participate as a liquidity provider in a pool configured with this extension. The unlisted depositor can manipulate bin composition (concentrating liquidity in specific bins to skew oracle-derived pricing or affect swap execution). The listed owner receives LP shares they did not request and cannot prevent, and their position becomes entangled with the injected shares due to the `msg.sender == owner` check on `removeLiquidity`. This constitutes broken core pool functionality causing loss of funds (locked/unwanted LP shares for the listed owner) and a complete bypass of an admin-configured access control boundary.

## Likelihood Explanation
- Requires a pool configured with `DepositAllowlistExtension` (a supported production extension).
- The attacker must hold and approve the pool tokens, but pays their own funds — no victim approval needed.
- The attack path is a direct call to the public `addLiquidity` entrypoint or through `MetricOmmPoolLiquidityAdder.addLiquidityExactShares`.
- The only prerequisite is knowing one listed owner address, which is observable on-chain from past `AllowedToDepositSet` events.
- No privileged role is required; any unprivileged address can execute this.

## Recommendation
Change `beforeAddLiquidity` to validate `sender` (the actual payer/caller) instead of `owner`, consistent with `SwapAllowlistExtension.beforeSwap`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  - Pool deployed with DepositAllowlistExtension as beforeAddLiquidity hook
  - allowedDepositor[pool][alice] = true   (alice is listed)
  - bob is NOT listed

Attack:
  1. bob calls pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)
     - pool calls _beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)
     - extension receives (bob, alice, ...) but checks allowedDepositor[pool][alice] → true → passes
     - pool mints shares for (alice, salt)
     - pool calls metricOmmModifyLiquidityCallback on bob
     - bob pays token0/token1 to the pool

  Result:
     - bob (unlisted) successfully deposited into the curated pool
     - alice received LP shares she did not request and cannot prevent
     - pool's curation invariant is broken
     - alice's position is entangled with injected shares; removeLiquidity
       requires msg.sender == owner, so alice cannot cleanly exit
```
