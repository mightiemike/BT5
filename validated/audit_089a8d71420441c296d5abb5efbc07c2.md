### Title
`verify_transaction_inclusion` panics for single-transaction blocks due to unconditional empty-proof guard — (File: `contract/src/lib.rs`)

### Summary

Both `verify_transaction_inclusion` (v1) and `verify_transaction_inclusion_v2` unconditionally reject an empty `merkle_proof` via a hard `require!` guard. For a block containing exactly one transaction (the coinbase), the merkle proof is legitimately empty — the transaction hash *is* the merkle root — yet the guard fires before the computation, causing the contract to panic and permanently blocking verification of any transaction in a single-transaction block.

### Finding Description

`verify_transaction_inclusion` contains an unconditional guard at line 315:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This guard fires *before* the merkle root computation. For a block with exactly one transaction, the merkle root equals the coinbase transaction hash and no sibling hashes are needed, so the correct proof is the empty vector. `compute_root_from_merkle_proof` already handles this correctly — when `merkle_proof` is empty the loop body never executes and `transaction_hash` is returned unchanged:

```rust
for proof_hash in merkle_proof {   // never entered when proof is empty
    ...
}
current_hash   // returns transaction_hash as-is
``` [2](#0-1) 

So `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[]) == merkle_root` would evaluate to `true` for a single-transaction block, but the `require!` guard prevents the code from ever reaching that point.

`verify_transaction_inclusion_v2` is equally affected. It first checks that `merkle_proof.len() == coinbase_merkle_proof.len()` (both 0 — passes), then verifies the coinbase proof (empty proof returns `coinbase_tx_id` which equals `merkle_root` for a single-tx block — passes), and then delegates to the deprecated v1 function which panics: [3](#0-2) 

### Impact Explanation

Any consumer contract or NEAR caller attempting to verify a transaction in a single-transaction block will receive a panic. Single-transaction (coinbase-only) blocks are valid and occur in practice on all supported chains. The verification API is the primary output of the light client; blocking it for a reachable class of valid inputs breaks the core protocol guarantee for those blocks.

### Likelihood Explanation

The trigger is fully unprivileged: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public `#[pause]`-gated methods callable by any NEAR account in normal operation. Any caller supplying a legitimately empty merkle proof for a single-transaction block hits the panic deterministically. No special role, key, or social engineering is required. [4](#0-3) [5](#0-4) 

### Recommendation

Remove the unconditional `require!(!args.merkle_proof.is_empty(), ...)` guard. The downstream `compute_root_from_merkle_proof` already handles the empty-proof case correctly (returning the transaction hash unchanged), so no additional logic is needed. If a guard is desired for other reasons, it should be conditioned on whether the block's merkle root equals the transaction hash directly, not on proof length alone.

### Proof of Concept

1. Identify any Bitcoin (or supported chain) block containing only the coinbase transaction. Its `merkle_root` equals `coinbase_tx_id`.
2. Submit that block header to the contract so it is stored in `headers_pool` and on the main chain.
3. Call `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_tx_id`
   - `tx_block_blockhash = <that block's hash>`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = coinbase_tx_id`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
4. The length check `0 == 0` passes. The coinbase proof check `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[]) == merkle_root` passes. Execution falls into the deprecated v1 call, which panics with `"Merkle proof is empty"` at line 315 before the root computation is reached. [6](#0-5)
