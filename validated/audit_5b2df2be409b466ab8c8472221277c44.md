### Title
Liquidator Can Bypass Spread Liquidation Logic by Supplying `isEncodedSpread=false` for a Spread-Registered Product — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

The `LiquidateSubaccount` transaction struct contains a user-supplied `isEncodedSpread` boolean that controls how `productId` is decoded and which liquidation code path executes. The contract validates that `isEncodedSpread=true` implies a valid spread-encoded `productId`, but never validates the inverse: that `isEncodedSpread=false` implies the `productId` is **not** part of a registered spread. A liquidator can sign a transaction with `isEncodedSpread=false` and a `productId` that is the spot leg of a registered spread, causing the contract to apply single-asset spot liquidation logic to a spread position — skipping the perp leg entirely, using the wrong liquidation price, and leaving the liquidatee with a naked short perp position.

---

### Finding Description

**Root cause — missing inverse validation of `isEncodedSpread`:**

In `_assertLiquidationAmount`, the spread-lookup loop runs unconditionally for both `isEncodedSpread=true` and `isEncodedSpread=false`:

```solidity
// ClearinghouseLiq.sol lines 73–87
for (uint256 mask = spreads; mask != 0; mask >>= 16) {
    uint32 low  = uint32(mask & 0xFF);
    uint32 high = uint32((mask & 0xFF00) >> 8);
    bool hit = false;
    if (txn.isEncodedSpread) {
        hit = (txn.productId == ((high << 16) | low));
    } else {
        hit = (txn.productId == low || txn.productId == high);  // ← matches spread leg
    }
    if (hit) { spotId = low; perpId = high; break; }
}
```

When `isEncodedSpread=false` and `productId` equals the spot leg of a registered spread, the loop sets both `spotId` and `perpId`. The subsequent guard:

```solidity
// lines 95–103
if (spotId == 0 || perpId == 0) {
    require(!txn.isEncodedSpread, ERR_INVALID_LIQUIDATION_PARAMS);
    ...
}
```

is **not entered** (both are non-zero), so no revert occurs. The code then falls through to the spot-engine branch:

```solidity
// lines 159–180
} else if (engine == address(spotEngine)) {
    ...
    maxLiquidatable = spotAmount;   // ← uses full spot balance, not basisAmount
}
```

And in `_handleLiquidationPayment`, the same `isEncodedSpread=false` path executes only spot balance updates:

```solidity
// lines 507–543
} else if (engine == address(spotEngine)) {
    spotEngine.updateBalance(txn.productId, txn.liquidatee, -txn.amount);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.liquidatee, v.liquidationPayment);
    spotEngine.updateBalance(txn.productId, txn.sender, txn.amount);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -v.liquidationPayment - v.liquidationFees);
    // ← perpEngine.updateBalance is NEVER called
}
```

The perp leg of the spread is never touched.

**Concrete exploit path:**

Assume a registered spread `(spotId=2, perpId=3)` and a liquidatee with:
- `spotBalance[2] = +100` (long spot)
- `perpBalance[3] = −100` (short perp, hedged spread)

A liquidator signs `LiquidateSubaccount { productId=2, isEncodedSpread=false, amount=100 }`. The sequencer submits it (it is a validly signed transaction; the sequencer orders transactions but does not deep-validate content). The contract:

1. Finds the spread → `spotId=2, perpId=3`
2. Computes `basisAmount = min(100, 100) = 100`, but sets `maxLiquidatable = spotAmount = 100` (spot path)
3. Applies spot liquidation price via `getLiqPriceX18(2, 100)` — **not** the spread price from `getSpreadLiqPriceX18`
4. Transfers 100 spot units to the liquidator at the spot discount price
5. Leaves `perpBalance[3] = −100` on the liquidatee — a naked short perp with no hedge

The `require(!isAboveInitial(txn.liquidatee))` post-check does not prevent this: the liquidatee was under maintenance health before and removing the spot asset does not make them healthy.

**Why `_assertCanLiquidateLiability` does not protect here:**

```solidity
// liquidateSubaccountImpl lines 629–638
if (
    (txn.amount < 0) &&
    (txn.isEncodedSpread || address(productToEngine[txn.productId]) == address(spotEngine))
) {
    _assertCanLiquidateLiability(...);
}
```

This guard only fires when `txn.amount < 0`. For asset liquidation (`amount > 0`), it is skipped entirely, leaving the exploit path fully open.

---

### Impact Explanation

1. **Wrong liquidation price**: The spot-only liquidation price (`getLiqPriceX18`) is applied instead of the spread price (`getSpreadLiqPriceX18`), which accounts for the offsetting perp position. The liquidator acquires the spot leg at a more favorable discount than the protocol intends for spread positions.

2. **Perp leg abandoned on liquidatee**: The liquidatee's short perp position is left intact after the spot is removed, converting a hedged spread into a naked short perp. This worsens the liquidatee's risk profile and may require additional liquidation rounds, draining the insurance fund faster.

3. **Over-liquidation of spot leg**: When `spotAmount > |perpAmount|` (partial hedge), `maxLiquidatable = spotAmount` instead of `basisAmount`, allowing the liquidator to seize more spot than the spread-aware limit would permit.

4. **Accounting desynchronization**: The spread position is partially dismantled without the protocol's spread accounting being updated, leaving the system in an inconsistent state that the spread-health calculation in `Clearinghouse.getHealth` does not anticipate.

---

### Likelihood Explanation

The `LiquidateSubaccount` transaction is signed by the liquidator and submitted by the sequencer. The sequencer's role is transaction ordering; it does not perform deep semantic validation of signed transaction content. Any liquidator can craft and sign a transaction with `isEncodedSpread=false` for a spread-registered product. The sequencer will include it as a valid signed transaction. No privileged access, key compromise, or social engineering is required — only the ability to sign a `LiquidateSubaccount` transaction, which is available to any address.

---

### Recommendation

Add an explicit check in `_assertLiquidationAmount` (or at the top of `liquidateSubaccountImpl`) that rejects a transaction where `isEncodedSpread=false` but `productId` resolves to a spread leg:

```diff
// After the spread-lookup loop in _assertLiquidationAmount:
+ if (!txn.isEncodedSpread && spotId != 0 && perpId != 0) {
+     revert(ERR_INVALID_LIQUIDATION_PARAMS); // productId is a spread leg; must use isEncodedSpread=true
+ }
```

This mirrors the recommended fix in the external report: validate that the user-supplied type flag matches the actual on-chain classification of the entity being operated on.

---

### Proof of Concept

1. Deploy with a registered spread `(spotId=2, perpId=3)` in the `spreads` bitmask.
2. Put a liquidatee under maintenance health with `spotBalance[2]=+100`, `perpBalance[3]=−100`.
3. Liquidator signs: `LiquidateSubaccount { sender=liquidator, liquidatee=victim, productId=2, isEncodedSpread=false, amount=100, nonce=N }`.
4. Sequencer submits the signed transaction via `submitTransactionsChecked`.
5. `liquidateSubaccountImpl` executes:
   - Spread loop: `spotId=2, perpId=3` (hit on `productId==low`)
   - `engine = productToEngine[2]` = spot engine
   - `if (spotId==0 || perpId==0)` → false, no revert
   - `maxLiquidatable = spotAmount = 100` (spot path)
   - `_handleLiquidationPayment`: only spot balances updated, perp untouched
6. Result: liquidator holds +100 spot at spot discount price; liquidatee holds `perpBalance[3]=−100` (naked short), no spot hedge. [1](#0-0) [2](#0-1) [3](#0-2)
