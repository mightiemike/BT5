### Title
Unsafe `transferFrom` in `replaceUsdcEWithUsdc` Enables USDC-e Drain Without Payment — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` uses a bare, unchecked `IERC20Base(usdc).transferFrom()` to pull USDC from the caller before releasing USDC-e. The rest of the codebase consistently uses `ERC20Helper.safeTransferFrom`, which validates the return value. If the USDC token at the hardcoded address returns `false` on failure instead of reverting (non-standard ERC20 behavior), the inbound USDC transfer silently fails while the outbound USDC-e transfer still executes, allowing any caller to drain USDC-e from a victim's `DirectDepositV1` contract without paying anything.

---

### Finding Description

`replaceUsdcEWithUsdc` is an unguarded `external` function (no `onlyOwner` or similar modifier) callable by any address on chain ID 57073. Its three-step body is:

```solidity
// Step 1 — pull USDC from caller (UNSAFE bare call)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);   // line 616

// Step 2 — withdraw USDC-e from directDepositV1 to ContractOwner
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));          // line 617

// Step 3 — push USDC-e to caller (safe)
IERC20Base(usdcE).safeTransfer(msg.sender, balance);                    // line 618
``` [1](#0-0) 

Step 1 uses the raw `IERC20Base.transferFrom` selector. The `IERC20Base` interface expects a `bool` return value and will ABI-decode it. For tokens that return `false` on insufficient balance/allowance (BAT-style), the call succeeds at the EVM level (no revert), the `bool` is decoded as `false`, but **the return value is never checked** — execution continues unconditionally into steps 2 and 3.

By contrast, every other inbound token transfer in the protocol goes through `ERC20Helper.safeTransferFrom`, which explicitly asserts the decoded boolean:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [2](#0-1) 

The same `ERC20Helper` is imported and used in `ContractOwner` for the outbound `safeTransfer` on line 618, making the omission on line 616 a clear inconsistency. [3](#0-2) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (Ink chain) returns `false` on failure rather than reverting:

- An attacker with zero USDC balance calls `replaceUsdcEWithUsdc(victimSubaccount)`.
- Step 1 silently returns `false`; no USDC moves.
- Step 2 withdraws the full USDC-e balance from the victim's `DirectDepositV1` to `ContractOwner`.
- Step 3 transfers that USDC-e to the attacker.

**Concrete asset delta:** the victim's `DirectDepositV1` loses its entire USDC-e balance; the attacker gains it at zero cost. Because `directDepositV1Address` is a public mapping, any funded subaccount is a discoverable target. [4](#0-3) 

---

### Likelihood Explanation

- The function has **no access control** — any EOA or contract can call it.
- The only precondition is a non-zero `directDepositV1Address[subaccount]` with a positive USDC-e balance, both of which are on-chain observable.
- The trigger depends on the specific USDC deployment at the hardcoded address returning `false` rather than reverting on failure. This is a deployment-specific property, but the code provides no protection regardless of which behavior the token exhibits.

---

### Recommendation

Replace the bare `transferFrom` on line 616 with the protocol's own `safeTransferFrom` wrapper:

```solidity
// Before (unsafe)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe)
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ERC20Helper.safeTransferFrom` is already imported via `using ERC20Helper for IERC20Base` at the top of `ContractOwner`, so no additional imports are needed. [3](#0-2) 

---

### Proof of Concept

1. Identify a subaccount `S` where `directDepositV1Address[S] != address(0)` and `IERC20Base(usdcE).balanceOf(directDepositV1Address[S]) > 0`.
2. Call `ContractOwner.replaceUsdcEWithUsdc(S)` from an address with zero USDC balance and zero USDC allowance.
3. If USDC returns `false` on the failed `transferFrom` (step 1), execution continues.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` transfers all USDC-e to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` transfers all USDC-e to the attacker.
6. Attacker receives `balance` USDC-e; victim's `DirectDepositV1` is drained; no USDC was paid. [5](#0-4)
