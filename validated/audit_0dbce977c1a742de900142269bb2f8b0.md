### Title
Tiered Rewards Voting Power Read at Tally Time, Not at Vote-Cast Time, Allows Post-Vote Position Inflation to Manipulate Governance Outcomes - (File: `x/tieredrewards/keeper/gov_tally.go`)

---

### Summary

The custom governance tally function `NewCustomTallyTierVotesFn` reads each voter's tiered positions at **tally time** (end of voting period) rather than at the time the vote was cast. An attacker can cast a vote with minimal tier power, then create a large new tiered position before the voting period closes, causing the tally to count inflated power for their already-cast vote.

---

### Finding Description

The Cosmos SDK governance tally is executed at the end of the voting period (in `EndBlocker`). The custom tally function in `gov_tally.go` iterates over all stored votes and, for each voter, calls `GetPositionStatesByOwner` to retrieve their **current** tiered positions at that moment: [1](#0-0) 

`GetPositionStatesByOwner` reads live state from the `PositionsByOwner` index and the staking delegation store — both of which reflect the chain state at the block when the tally runs, not when the vote was submitted: [2](#0-1) 

`positionVotingPower` then computes power from the position's **current** `Delegation.Shares` against the validator's **current** `BondedTokens`/`DelegatorShares`: [3](#0-2) 

There is no snapshot of tiered positions taken at proposal creation or at vote-cast time. The `PositionState` struct carries no historical record of shares at vote time: [4](#0-3) 

---

### Impact Explanation

An attacker who has already cast a vote can create one or more new large tiered positions (locking additional CRO) in the same voting period after voting. At tally time, all those positions are counted toward their already-cast vote option, inflating `results[OptionYes]` or `results[OptionNo]` beyond what their committed stake was at vote time.

The corrupted value is the `results` map accumulated in `NewCustomTallyTierVotesFn`: [5](#0-4) [6](#0-5) 

This can swing quorum calculations and Yes/No outcomes for governance proposals that control protocol-level parameters (inflation schedule, tier APY, etc.).

The symmetric attack also applies: a voter with a large tiered position can trigger position exit (beginning unbonding) after voting. Once the position's delegation is cleared, `IsDelegated()` returns `false` and `positionVotingPower` returns zero, so their vote is tallied with reduced power — useful for suppressing one's own abstain vote from counting toward quorum. [7](#0-6) 

---

### Likelihood Explanation

The entry path is a standard, unprivileged sequence of Cosmos SDK transactions:

1. `MsgVote` — cast a governance vote (any voter)
2. `MsgLockAndDelegate` (or equivalent position-creation message) — create a new large tiered position in the same voting period, after voting
3. Wait for the voting period to end; the `EndBlocker` tally reads the inflated positions

No privileged role, leaked key, or social engineering is required. The attacker only needs to hold sufficient CRO to create a meaningful position. Because the unbonding period (typically 21 days) is longer than many governance voting periods, the attacker's tokens are locked post-tally but the governance outcome is already finalized.

---

### Recommendation

Capture a snapshot of each voter's tiered position shares at the time `MsgVote` is processed (or at proposal submission), and use that snapshot in the tally instead of live state. This mirrors how the standard Cosmos SDK staking tally uses the `validators` map built from bonded state at tally time but does not allow delegations created after a vote to retroactively count.

Concretely, when a vote is stored, record the voter's current tiered delegation shares alongside the vote record. In `NewCustomTallyTierVotesFn`, use those stored shares rather than calling `GetPositionStatesByOwner` against live state.

---

### Proof of Concept

```
Block N:   Proposal enters voting period.

Block N+5: Alice calls MsgVote(proposalId, Yes).
           At this point Alice has 1 tiered position with 10,000 shares.
           The vote is stored; no snapshot of tier shares is taken.

Block N+10: Alice calls MsgLockAndDelegate with 500,000 CRO,
            creating a second large tiered position (500,000 shares).

Block N+VotingPeriodEnd: EndBlocker triggers tally.
    NewCustomTallyTierVotesFn runs.
    For Alice's stored vote, GetPositionStatesByOwner returns BOTH positions:
      - position 1: 10,000 shares  → ~10,000 voting power
      - position 2: 500,000 shares → ~500,000 voting power
    results[OptionYes] += 510,000   ← inflated; should be ~10,000

Block N+VotingPeriodEnd+1: Alice triggers exit on position 2.
    After unbonding period, Alice recovers her 500,000 CRO.
    Governance outcome was already decided with inflated power.
```

```go
func (k Keeper) GetPositionStatesByOwner(ctx context.Context, owner sdk.AccAddress) ([]types.PositionState, error) {
	ids, err := k.getPositionsIdsByOwner(ctx, owner)
	if err != nil {
		return nil, err
	}
	states := make([]types.PositionState, 0, len(ids))
	for _, id := range ids {
		state, err := k.getPositionState(ctx, id)
		if errors.Is(err, types.ErrPositionNotFound) {
			continue
		}
		if err != nil {
			return nil, err
		}
		states = append(states, state)
	}
	return states, nil
}
```

```go
func positionVotingPower(
	pos types.PositionState,
	bondedVals map[string]v1.ValidatorGovInfo,
) math.LegacyDec {
	if !pos.IsDelegated() {
		return math.LegacyZeroDec()
	}
	val, ok := bondedVals[pos.Delegation.ValidatorAddress]
	if !ok || val.DelegatorShares.IsZero() {
		return math.LegacyZeroDec()
	}
	return pos.Delegation.Shares.MulInt(val.BondedTokens).Quo(val.DelegatorShares)
}
```
