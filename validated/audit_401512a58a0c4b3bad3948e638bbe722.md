### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is still a live, externally callable NEAR contract method despite being marked `#[deprecated]`. Any unprivileged NEAR caller can invoke it directly, bypassing the coinbase merkle proof validation that `verify_transaction_inclusion_v2` was introduced to enforce. This allows an attacker to submit a forged Bitcoin transaction inclusion proof using the 64-byte internal-node substitution attack, causing the contract to return `true` for a transaction that was never included in the block.

---

### Finding Description

`verify_transaction_inclusion_v2` was added specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability (documented at https://www.bitmex.com/blog/64-Byte-Transactions). Its protection works by requiring a coinbase merkle proof of equal depth before delegating to `verify_transaction_inclusion`: [1](#0-0) 

The coinbase check anchors the tree depth: because the coinbase transaction is always at index 0 and its proof must match the block's `merkle_root`, an attacker cannot substitute an internal node hash for a leaf-level `tx_id` without also forging a consistent coinbase proof of the same depth.

However, `verify_transaction_inclusion` is merely annotated `#[deprecated]`—not `#[private]`, not removed, and not access-controlled: [2](#0-1) 

In Rust, `#[deprecated]` is a compiler lint, not a runtime restriction. In NEAR, any `pub` function in a `#[near]` impl block is a callable contract method regardless of deprecation. An attacker can therefore call `verify_transaction_inclusion` directly, supplying `ProofArgs` (which has no `coinbase_tx_id` or `coinbase_merkle_proof` fields): [3](#0-2) 

The function then performs only a standard merkle root recomputation with no coinbase anchor: [4](#0-3) 

The structural parallel to the reported bug is exact: `verify_transaction_inclusion_v2` is the "outer" function that performs the critical step (coinbase proof check) and then calls the "inner" public function. Because the inner function is itself public, callers can invoke it directly and the critical step is silently omitted—analogous to `userActions` being called directly with `unwrap=true`, skipping the token transfer.

---

### Impact Explanation

Bitcoin's merkle tree hashes pairs of 32-byte child hashes. An internal node at depth *d* is the double-SHA256 of two 32-byte children—a 64-byte input—producing a 32-byte output indistinguishable in type from a leaf transaction hash. An attacker who knows a block's transactions can compute every internal node hash. They can then supply one of those internal node hashes as `tx_id`, paired with a valid (shorter) merkle proof that reconstructs the correct `merkle_root`. `verify_transaction_inclusion` will return `true` for this forged proof because it only checks that the computed root matches the stored root—it has no coinbase anchor to detect the depth mismatch.

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate fund releases, cross-chain state transitions, or other security-critical actions can be deceived into accepting a proof for a Bitcoin transaction that never existed.

---

### Likelihood Explanation

- `verify_transaction_inclusion` is unconditionally reachable by any NEAR account (no role check, no `#[private]`, not paused by default).

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"

        let header = self

            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
