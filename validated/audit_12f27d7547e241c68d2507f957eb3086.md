### Title
Coinbase Proof Bypass via Internal Node Substitution Allows False Inclusion Proof — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` is supposed to prevent the 64-byte transaction forgery attack by requiring a valid coinbase merkle proof. However, the check only verifies that `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof) == merkle_root`. It does **not** verify that `coinbase_tx_id` is the actual coinbase transaction (the real leaf at index 0). An attacker can substitute an internal node of the merkle tree as `coinbase_tx_id`, craft a shorter-but-valid proof for it at position 0, and simultaneously prove a different internal node as `tx_id` — causing the function to return `true` for a hash that is not a real transaction.

---

`compute_root_from_merkle_proof` is a pure hash-chain computation with no leaf-vs-internal-node distinction:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;
    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
    current_hash
}
``` [2](#0-1) 

The only structural guard between the coinbase proof and the tx proof is a length equality check:

**Concrete attack on a 4-transaction block:**

```
         root
        /    \
      AB      CD
     /  \    /  \
    A    B  C    D
```

- `A` = real coinbase (tx[0]), `B` = tx[1], `C` = tx[2], `D` = tx[3]
- `AB = hash(A, B)`, `CD = hash(C, D)`, `root = hash(AB, CD)`

Attacker supplies:

| Field | Value | Why it passes |
|---|---|---|
| `coinbase_tx_id` | `AB` (internal node) | `compute_root_from_merkle_proof(AB, 0, [CD])` = `hash(AB, CD)` = `root` ✓ |
| `coinbase_merkle_proof` | `[CD]` (len=1) | — |
| `tx_id` | `CD` (internal node) | `compute_root_from_merkle_proof(CD, 1, [AB])` = `hash(AB, CD)` = `root` ✓ |
| `tx_index` | `1` | — |
| `merkle_proof` | `[AB]` (len=1) | — |
| length check | `1 == 1` | ✓ |

All three guards pass. `verify_transaction_inclusion` is then called via `args.into()` and also returns `true` because the tx proof is arithmetically valid. [4](#0-3) 

The deprecated `verify_transaction_inclusion` itself explicitly documents this weakness:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [5](#0-4) 

The v2 function was supposed to close this gap, but the coinbase check is insufficient because it never binds `coinbase_tx_id` to the actual leaf-level coinbase transaction.

---

### Impact Explanation

Any caller of the public `verify_transaction_inclusion_v2` method can obtain a `true` return value for a `tx_id` that is an internal merkle node, not a real Bitcoin transaction. Downstream contracts or applications that rely on this return value to authorize payments, bridge withdrawals, or other state transitions will accept a fabricated inclusion proof. This is the exact 64-byte transaction forgery attack the v2 function was designed to prevent.

---

### Likelihood Explanation

The attack requires only:
1. A block already stored in the contract's `headers_pool` (any relayed block).
2. Knowledge of that block's transaction list (public on-chain data).
3. A single public NEAR contract call with crafted `ProofArgsV2`.

No privileged role, private key, or social engineering is needed. The call path is fully open.

---

### Recommendation

The coinbase check must verify that `coinbase_tx_id` is the **actual leaf-level coinbase transaction**, not merely that some value at position 0 hashes to the merkle root. The standard mitigation is to require that the coinbase proof depth equals `ceil(log2(tx_count))` — i.e., the proof must reach from a true leaf to the root. Concretely:

- Require `coinbase_merkle_proof.len()` to equal the full tree depth (derivable from the number of transactions, which must be supplied and validated), **and** require `merkle_proof.len()` to equal the same depth.
- Alternatively, require the caller to supply the full raw coinbase transaction bytes, hash them on-chain, and compare to `coinbase_tx_id` before running the proof — eliminating the ability to substitute an internal node.
