### Title
Partial `TierUndelegate` Accumulates Multiple Unbonding Entries But `getUndelegatedAmount` Only Reads the First — (`File: x/tieredrewards/keeper/position_state.go`)

---

### Summary

The `x/tieredrewards` module supports partial undelegation via `TierUndelegate`. After multiple partial calls fully drain a position's delegation, the position delegator accumulates multiple unbonding entries in the Cosmos SDK staking module. However, `getUndelegatedAmount` only reads `ubds[0].Entries[0].Balance` — the first entry of the first unbonding delegation — silently ignoring all subsequent entries. This causes the position's reported amount to be systematically underreported once the position is fully undelegated.

---

### Finding Description

`transferDelegationFromPosition` supports partial undelegation: when the requested `amount` is less than the full position amount, it calls `ValidateUnbondAmount` and unbonds only that fraction, leaving the remaining shares delegated. [1](#0-0) 

Each partial `TierUndelegate` call appends a new `Entry` to the `UnbondingDelegation` object for the `(posDelAddr, valAddr)` pair in the staking module. After the final call that removes the last shares, `pos.IsDelegated()` becomes false and `getPositionAmount` delegates to `getUndelegatedAmount`: [2](#0-1) 

`getUndelegatedAmount` fetches at most one `UnbondingDelegation` object (limit=1) and returns only its first entry's balance: [3](#0-2) 

All entries beyond `Entries[0]` — representing the balances from earlier partial undelegations — are silently dropped. The function then falls through to `bankKeeper.SpendableCoins`, which is only correct after unbonding has completed and tokens have landed in the account. During the unbonding period, the spendable balance is zero, so the returned amount is `Entries[0].Balance` only, not the sum of all entries.

---

### Impact Explanation

`getPositionAmount` is the canonical function for determining a position's current value. Any operation that reads position value after full undelegation — including queries used by users to decide whether to exit, and any internal logic that gates `ClearPosition`/`Withdraw` on a minimum amount — will see an underreported balance. A user who performed N partial `TierUndelegate` calls will have their position value reported as `1/N` of the true amount. If downstream operations use this value to determine how much to return or whether a threshold is met, the user may be unable to recover their full funds or may receive less than owed. [4](#0-3) 

---

### Likelihood Explanation

The trigger is a normal user transaction sequence: call `TierUndelegate` with a partial amount multiple times (each call is valid and accepted by the protocol). The `ExitWithFullDelegation` check explicitly supports this partial path. No privileged role is required. Any delegator with a tiered position can reach this state. [5](#0-4) 

---

### Recommendation

Replace the single-entry read with a loop that sums all entries across all unbonding delegations for the position delegator:

```go
// Instead of:
ubds, err := k.stakingKeeper.GetUnbondingDelegations(ctx, delAddr, 1)
if len(ubds) > 0 && len(ubds[0].Entries) > 0 {
    return ubds[0].Entries[0].Balance, nil
}

// Use:
ubds, err := k.stakingKeeper.GetUnbondingDelegations(ctx, delAddr, maxUint16)
total := math.ZeroInt()
for _, ubd := range ubds {
    for _, entry := range ubd.Entries {
        total = total.Add(entry.Balance)
    }
}
if total.IsPositive() {
    return total, nil
}
```

Additionally, consider restricting `TierUndelegate` to full-exit only (disabling partial undelegation) until the accounting across multiple unbonding entries is fully supported — directly mirroring the report's short-term recommendation to disable partial operations until the protocol supports them end-to-end.

---

### Proof of Concept

1. User creates a tiered position with 300 CRO delegated to validator V.
2. User calls `TriggerExit`; waits for `ExitDuration`.
3. User calls `TierUndelegate(amount=100)` → staking creates `UnbondingDelegation{Entries: [{Balance: 100}]}`.
4. User calls `TierUndelegate(amount=100)` → staking appends `{Balance: 100}` → `Entries: [{100}, {100}]`.
5. User calls `TierUndelegate(amount=100)` → staking appends `{Balance: 100}` → `Entries: [{100}, {100}, {100}]`. Position is now fully undelegated.
6. Any call to `getPositionAmount` now invokes `getUndelegatedAmount`, which returns `ubds[0].Entries[0].Balance = 100` instead of `300`.
7. The position's reported value is 200 CRO short of the user's actual locked funds. [4](#0-3) [1](#0-0)

func (k Keeper) getUndelegatedAmount(ctx context.Context, delegatorAddress string) (math.Int, error) {
	delAddr, err := sdk.AccAddressFromBech32(delegatorAddress)
	if err != nil {
		return math.Int{}, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}
	ubds, err := k.stakingKeeper.GetUnbondingDelegations(ctx, delAddr, 1)
	if err != nil {
		return math.Int{}, err
	}
	if len(ubds) > 0 && len(ubds[0].Entries) > 0 {
		return ubds[0].Entries[0].Balance, nil
	}
	bondDenom, err := k.stakingKeeper.BondDenom(ctx)
	if err != nil {
		return math.Int{}, err
	}
	return k.bankKeeper.SpendableCoins(ctx, delAddr).AmountOf(bondDenom), nil
}
