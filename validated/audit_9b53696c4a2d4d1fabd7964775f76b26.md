### Title
`SwapAllowlistExtension.beforeSwap` checks the router address as `sender` instead of the actual end user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is the production swap-curation guard for Metric OMM pools. Its `beforeSwap` hook checks the `sender` argument, which equals `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so `sender` = router address. For router-based swaps to work on an allowlisted pool, the pool admin must add the router to the allowlist. Once the router is allowlisted, every user — including those explicitly excluded — can bypass the individual allowlist by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` gates swap access by checking `sender` against the per-pool allowlist: [1](#0-0) 

`sender` is the value forwarded by `ExtensionCalling._beforeSwap`, which is `msg.sender` of `MetricOmmPool.swap()`: [2](#0-1) [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly: [4](#0-3) 

The actual end user (`msg.sender` of `exactInputSingle`) is stored only in transient storage for the payment callback — it is never forwarded to the extension. The extension therefore receives `sender` = router address, not the real user.

For any pool that uses `SwapAllowlistExtension` and also supports router-based swaps, the pool admin must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, the check `allowedSwapper[pool][router]` passes for every user who routes through it, regardless of whether that user is individually allowed or explicitly excluded. [5](#0-4) 

The analog to the Cork bug is exact: Cork's `LVDepositNotPaused` checked `isWithdrawalPaused` instead of `isDepositPaused`, making the deposit pause ineffective. Here, `SwapAllowlistExtension` checks the router address instead of the actual user, making the swap allowlist ineffective for all router-mediated swaps.

---

### Impact Explanation

**High.** The swap allowlist is the primary curation mechanism for pools that restrict trading to specific counterparties (e.g., KYC'd addresses, institutional partners). Bypassing it via the router allows any user to trade on a curated pool. This constitutes a broken core pool functionality and an admin-boundary break: the pool admin's configured access control is silently defeated by the standard periphery path. Any token amounts swapped by disallowed users represent direct loss of the curation guarantee and potential regulatory or counterparty risk that the pool was designed to prevent.

---

### Likelihood Explanation

**High.** `MetricOmmSimpleRouter` is the standard, documented periphery entry point for swaps. Any pool that deploys `SwapAllowlistExtension` and expects users to swap through the router — the normal usage pattern — is immediately affected. No special attacker capability is required: any user simply calls `exactInputSingle` or `exactInput` on the router.

---

### Recommendation

The `beforeSwap` hook must check the actual end user, not the intermediary router. Two viable approaches:

1. **Pass the real initiator via `extensionData`**: The router encodes `msg.sender` into `extensionData` before forwarding to the pool. The `SwapAllowlistExtension` decodes and checks that address. This requires a convention between the router and the extension.

2. **Check `recipient` as a proxy for the user**: For `exactInputSingle`, `recipient` is often the user. However, this is not reliable for multi-hop paths where intermediate recipients are the router itself.

The cleanest fix is approach (1): the router should encode the original caller into `extensionData`, and the extension should decode and verify it when present, falling back to `sender` for direct pool calls.

---

### Proof of Concept

1. Pool is deployed with `SwapAllowlistExtension` configured on `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps.
3. Pool admin calls `setAllowedToSwap(pool, alice, false)` (or simply never adds Alice).
4. Alice calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, ...)` — `msg.sender` = router.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Alice successfully swaps on a pool she was explicitly excluded from.

The allowlist check that should have blocked Alice at step 7 instead passes because it sees the router address, not Alice's address. [1](#0-0) [6](#0-5)
