Audit Report

## Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Oracle Mid Price, Systematically Weakening the Token-1-Outflow Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the arithmetic mean of bid and ask at line 218, while the protocol's own `SwapMath.midAndSpreadFeeX64FromBidAsk` (used by `PriceVelocityGuardExtension` and the pool's swap engine) uses the geometric mean. Because AM ≥ GM, the stop-loss extension permanently uses a mid price that is too high, causing `metricT0` to be underestimated on every swap. This sets the high-watermark lower than the true per-share value, lowering the drawdown floor and allowing more token-1 to drain from the pool before the guard fires.

## Finding Description

**Root cause — `OracleValueStopLossExtension.sol` line 218:**

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;  // arithmetic mean
``` [1](#0-0) 

**Protocol standard — `SwapMath.midAndSpreadFeeX64FromBidAsk` line 70:**

```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);  // geometric mean
``` [2](#0-1) 

`PriceVelocityGuardExtension.beforeSwap` line 48 calls `SwapMath.midAndSpreadFeeX64FromBidAsk` and uses the geometric mean. [3](#0-2) 

**How the wrong mid price corrupts `metricT0`:**

The `_metrics` function computes:
```
metricT0 = t0ps + mulDiv(mulDiv(t1, Q64, midPriceX64), METRIC_SCALE, shares)
``` [4](#0-3) 

Because `midPriceX64_arithmetic ≥ midPriceX64_geometric`, the term `t1 × Q64 / midPriceX64` is smaller with the arithmetic mean, so `metricT0` is underestimated on every swap.

**How the wrong metric corrupts the watermark and guard:**

`_applyWatermark` ratchets the high-watermark up to the live metric value:
```solidity
if (metric >= hwm) return (metric, false);   // ratchet up
breached = metric < (hwm * floorMultiplier) / E6;
``` [5](#0-4) 

Because `metricT0` is always underestimated, `hwm0` is set to a value lower than the true per-share value. The drawdown floor `hwm0 × floorMultiplier / E6` is therefore also lower. The guard fires only when the live metric falls below this floor, so a larger real-world value drop is required before the stop-loss triggers for `zeroForOne == true` swaps (token-1 outflow). [6](#0-5) 

**Quantified error:** For spread `s` (ask = bid × (1 + s)), AM overestimates GM by approximately `bid × s²/8`. At 10% spread the mid price is overestimated by ~0.12%, causing `metricT0` to be underestimated by ~0.12% and the effective drawdown protection to be ~2.4% weaker relative to the configured threshold. At 20% spread the relative weakening reaches ~9%.

## Impact Explanation

The `OracleValueStopLossExtension` is the primary on-chain mechanism protecting LP principal from value drain between oracle updates. Because `metricT0` is systematically underestimated on every swap, the high-watermark for token-0-denominated value is set below the true per-share value, the drawdown floor is proportionally lower, and `zeroForOne == true` swaps (token-1 outflow) can proceed past the configured drawdown threshold without triggering the stop-loss revert. LPs suffer a larger-than-configured loss of token-1 principal before the guard fires. For pools with spreads ≥ 5% (common for RWA or low-liquidity pairs), the effective protection is materially weaker than the admin intended. This constitutes a direct loss of LP assets above Sherlock thresholds for affected pool configurations.

## Likelihood Explanation

The miscalibration is present on every swap that touches a pool with this extension configured and a non-zero spread. No special attacker action is required; the error is structural and permanent. Any swap that moves the pool through bins will underestimate `metricT0` and set a lower watermark, permanently weakening the guard for the lifetime of the pool. Any unprivileged trader executing `zeroForOne == true` swaps benefits from the weakened guard without any special setup.

## Recommendation

Replace the arithmetic mean with the same geometric-mean helper used by the pool and `PriceVelocityGuardExtension`:

```solidity
// Before (line 218):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After:
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This ensures the stop-loss extension uses the same price reference as the pool's swap engine and `PriceVelocityGuardExtension`, eliminating the systematic underestimation of `metricT0`.

## Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension`, `drawdownE6 = 50_000` (5%), and an oracle with bid = 1e18, ask = 1.1e18 (10% spread).
2. Execute a `zeroForOne == true` swap to set the initial watermark.
   - Extension computes `midPriceX64 = (1e18 + 1.1e18) / 2 = 1.05e18` (arithmetic mean).
   - Correct value: `sqrt(1e18 × 1.1e18) ≈ 1.04881e18` (geometric mean).
   - `metricT0` is underestimated by ≈ 0.12% relative to the correct value.
   - `hwm0` is set to this underestimated value.
3. Drain token-1 via further `zeroForOne == true` swaps until `metricT0` falls 5% below `hwm0`.
   - Because `hwm0` was set 0.12% too low, the actual per-share value has fallen 5.12% before the guard fires.
4. Observe that the stop-loss reverts only after the true drawdown exceeds the configured 5% threshold by the AM-GM error margin, confirming LPs lost more than the declared maximum drawdown.
