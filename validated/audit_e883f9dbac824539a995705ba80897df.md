### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` ignores the `sender` argument (the actual `msg.sender` of the `addLiquidity` call) and gates only on `owner` (the caller-supplied position owner address). Because `MetricOmmPool.addLiquidity` accepts any `owner` value from any caller with no `msg.sender == owner` restriction, an unauthorized address can pass an allowlisted address as `owner`, satisfy the extension check, and deposit liquidity into the pool — paying the tokens themselves but modifying pool bin state without permission.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts a free-form `owner` parameter and passes both `msg.sender` (as `sender`) and the caller-supplied `owner` to the extension hook:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first argument (`sender`) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

Unlike `removeLiquidity`, which enforces `msg.sender == owner`, `addLiquidity` has no such constraint: [3](#0-2) 

An attacker deploys a contract implementing `metricOmmModifyLiquidityCallback`, calls `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)`, and the extension evaluates `allowedDepositor[pool][allowlistedAddress]` — which is `true` — and passes. The attacker's contract pays the tokens in the callback. The position is minted under `allowlistedAddress`, and the attacker has successfully deposited into a permissioned pool without being on the allowlist.

The `MetricOmmPoolLiquidityAdder` also exposes this path: `addLiquidityExactShares(pool, owner, ...)` only validates `owner != address(0)` and then calls `pool.addLiquidity(owner, ...)` with `msg.sender = LiquidityAdder`, so the extension again sees only `owner`. [4](#0-3) 

---

### Impact Explanation

The deposit allowlist is the primary access-control mechanism for permissioned pools. Bypassing it allows an unauthorized actor to:

1. **Modify pool bin state** — add liquidity to specific bins, shifting `curBinIdx` / `curPosInBin` and altering the marginal price seen by subsequent swaps.
2. **Grief allowlisted LPs** — force LP shares into an allowlisted address's position under an arbitrary `salt`, diluting or interfering with that position's accounting.
3. **Undermine the stop-loss and velocity guards** — `OracleValueStopLossExtension` and `PriceVelocityGuardExtension` both operate on post-swap bin state; an unauthorized liquidity injection before a swap can shift the watermark baseline or the bin composition those guards read, weakening their protective effect.

The attacker forfeits the deposited tokens (they cannot call `removeLiquidity` because `msg.sender != owner`), but the pool's invariant — that only allowlisted addresses may hold positions — is broken, and bin-level state manipulation is achievable at the cost of the deposited amount.

---

### Likelihood Explanation

- No special privilege is required; any EOA or contract can call `pool.addLiquidity` directly.
- The only cost is implementing a trivial `metricOmmModifyLiquidityCallback` and paying the deposited tokens.
- The attacker needs to know one allowlisted address (observable on-chain via `AllowedToDepositSet` events).
- Likelihood is **medium**: the attacker sacrifices capital, so the attack is economically rational only when the bin-state manipulation yields a larger off-pool profit (e.g., arbitrage, front-running).

---

### Recommendation

Gate on `sender` (the actual caller) rather than `owner`, consistent with the contract's stated purpose of gating `addLiquidity` by depositor:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is instead to restrict which addresses may *own* positions (not which addresses may *call* `addLiquidity`), the pool itself must enforce `msg.sender == owner` in `addLiquidity` (mirroring the `removeLiquidity` guard), so that `sender` and `owner` are always the same address and checking either is equivalent.

### Proof of Concept

```solidity
// Attacker contract
contract BypassDeposit {
    IMetricOmmPool pool;
    address allowlistedLP; // observed from AllowedToDepositSet event

    function attack(LiquidityDelta calldata deltas) external {
        // owner = allowlistedLP → extension check passes
        // sender = address(this) → extension check is never evaluated
        pool.addLiquidity(allowlistedLP, 0, deltas, abi.encode(uint8(1)), "");
    }

    function metricOmmModifyLiquidityCallback(
        uint256 amount0, uint256 amount1, bytes calldata
    ) external {
        // pay tokens from attacker's balance
        IERC20(token0).transfer(msg.sender, amount0);
        IERC20(token1).transfer(msg.sender, amount1);
    }
}
```

Steps:
1. Deploy `BypassDeposit`, fund it with token0/token1.
2. Call `attack(deltas)` targeting any bin.
3. `DepositAllowlistExtension.beforeAddLiquidity` receives `sender = BypassDeposit`, `owner = allowlistedLP`; it checks only `allowedDepositor[pool][allowlistedLP]` → `true` → no revert.
4. Liquidity is minted under `allowlistedLP`; pool bin state is modified by the unauthorized actor. [2](#0-1) [5](#0-4) [6](#0-5)
