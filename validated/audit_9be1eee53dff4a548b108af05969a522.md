### Title
Stale Genesis-Era Seat Count Used as Reed-Solomon Chunk Encoding Parameter — (File: `chain/epoch-manager/src/adapter.rs`)

### Summary

`EpochManagerHandle::num_total_parts()` permanently reads `num_block_producer_seats` from the **genesis** protocol version's `EpochConfig`, not from the current epoch's config. Every call site that encodes, validates, distributes, or reconstructs chunk parts uses this frozen value. If a protocol upgrade ever changes `num_block_producer_seats`, the Reed-Solomon encoding width diverges from the actual validator-set size, breaking chunk availability for the entire network.

### Finding Description

`num_total_parts()` is the single source of truth for the Reed-Solomon total-part count used throughout the chunk pipeline:

```rust
// chain/epoch-manager/src/adapter.rs  lines 955-963
fn num_total_parts(&self) -> usize {
    let epoch_manager = self.read();
    let genesis_protocol_version = epoch_manager.config.genesis_protocol_version();
    let seats = epoch_manager
        .config
        .for_protocol_version(genesis_protocol_version)   // ← always genesis version
        .num_block_producer_seats;
    if seats > 1 { seats as usize } else { 2 }
}
``` [1](#0-0) 

The function unconditionally fetches `genesis_protocol_version` and resolves the seat count from that version's `EpochConfig`. There is no per-epoch lookup. The returned value is consumed by every critical chunk-encoding path:

- **Part validation** — `validate_part(…, num_total_parts)` in `process_partial_encoded_chunk` and `validate_partial_encoded_chunk_forward` [2](#0-1) 
- **Part distribution** — `for part_ord in 0..self.epoch_manager.num_total_parts()` in `distribute_encoded_chunk` [3](#0-2) 
- **Chunk reconstruction** — `EncodedShardChunk::from_header(header, self.epoch_manager.num_total_parts())` [4](#0-3) 
- **Completeness check** — `for part_ord in 0..self.epoch_manager.num_total_parts()` in `has_all_parts` [5](#0-4) 

`num_data_parts()` is derived from `num_total_parts()`, so it inherits the same stale value.

The analog to the Frax `userStakedFrax` bug is direct: Frax calls `getAmount0ForLiquidity` using the **LP-provider's tick range set at deposit time** rather than the current price range, producing a token-amount that does not reflect actual market state. Here, nearcore calls `for_protocol_version(genesis_protocol_version)` to obtain a seat count set **at chain genesis** rather than the current epoch's config, producing a part-count that does not reflect the actual validator-set size after a protocol upgrade.

`num_block_producer_seats` is a first-class field of `EpochConfig` and is legitimately changed by protocol upgrades:

```rust
// core/primitives/src/epoch_manager.rs  lines 107-108
pub struct EpochConfig {
    pub num_block_producer_seats: NumSeats,
    ...
``` [6](#0-5) 

The protocol-upgrade voting mechanism is a public, epoch-boundary interaction — not a privileged individual-validator action. Once 80% of stake votes for a new protocol version, the new `EpochConfig` (with a potentially different `num_block_producer_seats`) takes effect two epochs later. At that point `num_total_parts()` continues to return the genesis value, while the actual validator set has grown or shrunk.

### Impact Explanation

**Chunk availability breaks network-wide.** Reed-Solomon encoding requires every node to agree on `(data_parts, parity_parts)`. If the genesis value is, say, 100 and the upgraded epoch has 200 seats:

- Chunks are still encoded into 100 parts (genesis value).
- Part-owner assignment (`get_part_owner`) maps part ordinals 0–99 across 200 validators, leaving half the validator set with no owned parts.
- Validators that own no parts cannot satisfy `has_all_parts()` and will never approve blocks.
- Conversely, if seats decrease (e.g., 100 → 50), `num_total_parts` returns 100 but only 50 validators exist; part ordinals 50–99 are unowned and undeliverable, making chunk reconstruction impossible.

The result is a protocol-wide liveness failure: no new blocks can be finalized because no chunk can be declared complete. This is a **Critical** impact on chain liveness.

### Likelihood Explanation

**Medium.** `num_block_producer_seats` has not changed on mainnet since genesis (currently 100). However:

1. The field is a normal `EpochConfig` parameter with no code-level freeze.
2. Protocol upgrades that adjust validator-set sizing are a documented and expected governance action.
3. The bug is silent until the upgrade fires — no test or assertion catches the mismatch before deployment.
4. The upgrade boundary is a public, unprivileged event (validator voting), not a privileged admin action.

### Recommendation

Replace the genesis-version lookup with a per-epoch lookup, or establish a protocol-enforced invariant that `num_block_producer_seats` is immutable:

```rust
// Option A: use the current epoch's seat count
fn num_total_parts(&self) -> usize {
    // num_total_parts must be constant across all epochs for stored-chunk
    // compatibility; document and enforce that num_block_producer_seats
    // is immutable, or derive total_parts from a separate immutable constant.
    ...
}
```

If the intent is that `num_total_parts` must be epoch-invariant (for backward compatibility with stored chunks), add an explicit protocol-schema assertion that `num_block_producer_seats` cannot change across protocol versions, and document this constraint prominently. If the intent is that it should track the current epoch, fix the lookup to use the current epoch's protocol version.

### Proof of Concept

1. Deploy a local network with genesis `num_block_producer_seats = 4`.
2. Introduce a protocol upgrade that sets `num_block_producer_seats = 8` in the new `EpochConfig`.
3. After the upgrade epoch activates, observe that `num_total_parts()` still returns `4`.
4. A chunk producer encodes a chunk into 4 parts and distributes them.
5. Validators with part ordinals 4–7 (assigned by `get_part_owner` over 8 validators) never receive their parts.
6. `has_all_parts()` returns `false` for those validators; they never approve the block.
7. No block reaches the Doomslug finality threshold; the chain halts.

The divergent value is exact: `EpochConfig::num_block_producer_seats` at `genesis_protocol_version` vs. `EpochConfig::num_block_producer_seats` at the current epoch's protocol version. [7](#0-6)
