### Title
`starknet_estimateFee` Returns `overall_fee` Including L2 Gas Cost But Omits `l2_gas_consumed` From Breakdown, Breaking the Stated Fee Formula — (File: `crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` exposes `gas_consumed` (L1 gas only) and `data_gas_consumed` (L1 data gas only), but omits `l2_gas_consumed`. The `overall_fee` field is taken directly from the execution receipt and **includes** the L2 gas cost. The OpenRPC spec description explicitly states `overall_fee = gas_consumed*gas_price + data_gas_consumed*data_gas_price`, which is arithmetically wrong whenever L2 gas is non-zero. Any client that trusts this formula receives an authoritative-looking wrong value.

---

### Finding Description

In `tx_execution_output_to_fee_estimation` the response is assembled as:

```rust
Ok(FeeEstimation {
    gas_consumed:      gas_vector.l1_gas.0.into(),      // L1 gas only
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // L1 data gas only
    l1_data_gas_price,
    l2_gas_price,                                        // price present, amount absent
    overall_fee: tx_execution_output.execution_info.receipt.fee, // ALL gas types
    unit: tx_execution_output.price_unit,
})
``` [1](#0-0) 

`receipt.fee` is produced by `GasVector::cost`, which sums L1 gas, L1 data gas, **and** L2 gas (with tip applied to L2 gas price):

```rust
for (gas, price, resource) in [
    (self.l1_gas,      gas_prices.l1_gas_price,      Resource::L1Gas),
    (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
    (self.l2_gas,      tipped_l2_gas_price,           Resource::L2Gas),
] { ... }
``` [2](#0-1) 

The `FeeEstimation` struct itself carries a developer TODO acknowledging the gap:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
``` [3](#0-2) 

The OpenRPC schema description for `overall_fee` states:

> "equals to gas_consumed\*gas_price + data_gas_consumed\*data_gas_price" [4](#0-3) 

This formula is wrong for any V3 (`AllResources`) transaction where `l2_gas > 0`. The schema has no `l2_gas_consumed` field and does not list it as required. [5](#0-4) 

The analog to the Rubicon bug is exact: Rubicon checks `fill_amt >= min_fill_amount` against the pre-fee accumulator and only deducts fees afterward, so the check is against the wrong quantity. Here, the breakdown fields (`gas_consumed`, `data_gas_consumed`) represent the pre-L2-gas-addition quantities, while `overall_fee` is the post-L2-gas-addition total. The stated invariant (`overall_fee == breakdown * prices`) is checked/advertised against the pre-transformation values, but the actual value delivered is the post-transformation one.

---

### Impact Explanation

**High — RPC fee estimation returns an authoritative-looking wrong value.**

1. **Incorrect fee verification**: A client that computes `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` to cross-check `overall_fee` will always get a value strictly less than `overall_fee` when L2 gas is consumed. The discrepancy equals `l2_gas * (l2_gas_price + tip)`.

2. **Broken resource-bound construction**: A client that uses the breakdown fields to construct V3 (`AllResources`) resource bounds will set `l2_gas.max_amount = 0` (since no `l2_gas_consumed` is returned). When the transaction is submitted, the blockifier's post-execution check will fire `FeeCheckError::MaxGasAmountExceeded { resource: L2Gas, ... }` and revert the transaction, even though the fee estimation succeeded. [6](#0-5) 

3. **Tip amplification**: Because the L2 gas price is tipped (`tipped_l2_gas_price = l2_gas_price + tip`), the gap between the stated formula and `overall_fee` grows with the tip value, making the discrepancy larger for higher-priority transactions.

---

### Likelihood Explanation

Every V3 (`AllResources`) transaction that executes Cairo 1 code consumes non-zero L2 gas. The `GasVectorComputationMode::All` path is the default for V3 transactions. Any wallet, SDK, or dApp that calls `starknet_estimateFee` and then uses the returned breakdown to construct resource bounds for a V3 transaction will hit this. The TODO comment in the source confirms the team is aware the field is missing but has not yet added it.

---

### Recommendation

1. Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0` in `tx_execution_output_to_fee_estimation`.
2. Update the OpenRPC `FEE_ESTIMATE` schema to include `l2_gas_consumed` as a required field and correct the `overall_fee` description to `gas_consumed*l1_gas_price + data_gas_consumed*l1_data_gas_price + l2_gas_consumed*(l2_gas_price+tip)`.
3. Ensure the `FeeEstimation` serialization test (`validate_fee_estimation_schema`) validates the corrected schema.

---

### Proof of Concept

1. Submit any V3 invoke transaction to `starknet_estimateFee` on a node running with `GasVectorComputationMode::All` (the default for V3).
2. Observe the response: `overall_fee > gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` because `l2_gas * l2_gas_price` is silently folded into `overall_fee` but not exposed.
3. Construct a V3 transaction using the returned breakdown: set `l1_gas.max_amount = gas_consumed`, `l1_data_gas.max_amount = data_gas_consumed`, `l2_gas.max_amount = 0` (the only value derivable from the response).
4. Submit the transaction. The blockifier's `check_all_gas_amounts_within_bounds` will reject it with `MaxGasAmountExceeded { resource: L2Gas, max_amount: 0, actual_amount: N }` even though the fee estimation returned success. [7](#0-6) [8](#0-7)

pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
