### Title
Attacker-controlled P2P peer can force a Sierra class to be stored as a deprecated class, causing wrong contract code selected for execution — (`crates/apollo_protobuf/src/converters/state_diff.rs`, `crates/apollo_p2p_sync/src/client/class.rs`)

### Summary

The P2P sync client uses `compiled_class_hash.as_ref() == None` as the sole discriminant to classify a received `protobuf::DeclaredClass` as either a Cairo1 (`DeclaredClass`) or Cairo0 (`DeprecatedDeclaredClass`) class. No state diff commitment is verified against the block header. The class sync then reads the already-stored (attacker-influenced) state diff to decide which class type to accept, and does not verify the class hash against the class body. An attacker-controlled peer can therefore force an arbitrary deprecated class body to be stored under a Sierra class hash, causing wrong contract code to be selected for execution.

### Finding Description

**Step 1 — State diff type confusion:** [1](#0-0) 

When `compiled_class_hash` is `None`, the message is unconditionally decoded as `StateDiffChunk::DeprecatedDeclaredClass`. There is no cross-check against the block header's `state_diff_commitment`.

**Step 2 — State diff stored without commitment verification:** [2](#0-1) 

`append_state_diff` is called directly. The `parse_data_for_block` loop only validates `state_diff_length` (total count) and duplicate-key conflicts: [3](#0-2) 

Both `DeclaredClass` and `DeprecatedDeclaredClass` have `len() == 1`, so the length check passes even when the class type is wrong. [4](#0-3) 

**Step 3 — Class sync reads the poisoned state diff:** [5](#0-4) 

The class sync reads `deprecated_declared_classes` from the already-stored (attacker-influenced) state diff. X is now in that set.

**Step 4 — Deprecated class body accepted for X:** [6](#0-5) 

A `DeprecatedContractClass` is accepted if `deprecated_declared_classes.contains(&class_hash)`. Since X was forced into that set, the check passes.

**Step 5 — No class hash verification:** [7](#0-6) 

The TODO is explicit: class hash is never verified against the class body. Any arbitrary deprecated class body is stored under X. [8](#0-7) 

### Impact Explanation

After the attack:
- X is in `deprecated_declared_classes` instead of `class_hash_to_compiled_class_hash` — the `compiled_class_hash` for X is never stored.
- An arbitrary `DeprecatedContractClass` body is stored under X in the class manager.
- When any contract whose class is X is executed, the execution engine retrieves the deprecated class body and uses the Cairo0 execution path instead of the Sierra/CASM path.
- This is "Wrong compiled class / contract code selected for execution" — Critical impact.

The `calculate_state_diff_hash` function commits to both `class_hash_to_compiled_class_hash` and `deprecated_declared_classes` separately: [9](#0-8) 

Moving X between these two maps changes the state diff hash, but the P2P sync never verifies the state diff against the commitment in the block header.

### Likelihood Explanation

- Requires an attacker-controlled peer that the syncing node connects to.
- The attack is fully deterministic: no brute force, no hash collision, no cryptographic assumption.
- The two missing guards (state diff commitment check and class hash verification) are both explicitly noted as TODOs in the codebase, confirming they are known gaps.
- Any node running P2P sync (`p2p_sync_client_config` enabled) is exposed.

### Recommendation

1. **Verify state diff commitment**: After assembling the full `ThinStateDiff` for a block, compute `calculate_state_diff_hash` and compare it against the `state_diff_commitment` stored in the block header before calling `append_state_diff`.
2. **Verify class hash in class sync**: Implement the TODO at `class.rs:39` — compute the class hash from the received class body and reject the peer if it does not match the expected hash.
3. **Reject `DeclaredClass{compiled_class_hash: None}` if the state diff commitment says it should be Sierra**: The commitment check in (1) would subsume this, but an explicit guard at the protobuf layer is also advisable.
