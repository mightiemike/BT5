Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` validates `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist ‚Äî (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual token-paying caller) and validates only `owner` (the LP-share recipient). Because `addLiquidity` lets any caller freely specify `owner`, an address not on the allowlist can deposit tokens by nominating an allowlisted address as `owner`, fully bypassing the access-control gate the pool admin configured. The allowlisted address also receives an unwanted LP position it cannot immediately exit without actively calling `removeLiquidity`.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument and the caller-supplied `owner` as the second to the extension hook: [1](#0-0) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and discarded; only `owner` is checked against the allowlist: [2](#0-1) 

Because `owner` is freely chosen by the caller, any unprivileged address Bob can call `pool.addLiquidity(alice, ...)` where Alice is allowlisted. The hook sees `owner = alice`, the check passes, and the pool issues the liquidity callback to `msg.sender` (Bob), collecting Bob's tokens. Alice receives LP shares she never requested.

`SwapAllowlistExtension.beforeSwap` demonstrates the correct pattern ‚Äî it names and validates `sender`: [3](#0-2) 

The asymmetry between the two extensions is the root cause.

Additionally, `removeLiquidity` enforces `msg.sender == owner`: [4](#0-3) 

This means Alice must actively unwind a position she never consented to.

## Impact Explanation

The pool admin's deposit allowlist invariant ‚Äî that only explicitly permitted addresses can supply tokens ‚Äî is broken by any unprivileged EOA or contract. If the allowlist enforces regulatory compliance or restricts liquidity to trusted counterparties, the bypass has direct protocol-level consequences. Additionally, the griefing vector forces an unwanted LP position onto the allowlisted address, which can only be removed by that address calling `removeLiquidity` itself. This constitutes an admin-boundary break (allowlist bypass by an unprivileged path) and a broken core pool functionality (access control on liquidity provision is non-functional).

## Likelihood Explanation

No special privilege is required. The only prerequisite is knowing one allowlisted address, which is publicly readable from `allowedDepositor` or `allowAllDepositors`. Any EOA or contract can execute the bypass in a single transaction with no flash loan, oracle manipulation, or admin cooperation.

## Recommendation

Replace the unnamed first parameter with `sender` and validate it instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the token payer and the position recipient, both `sender` and `owner` should be checked independently.

## Proof of Concept

```
Setup

1. Pool P is deployed with DepositAllowlistExtension E.
2. Pool admin calls E.setAllowedToDeposit(P, alice, true).
   ‚Üí allowedDepositor[P][alice] = true
   ‚Üí allowedDepositor[P][bob]   = false  (bob is NOT allowed)

Attack
‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
3. Bob deploys a router contract R that:
   a. Calls P.addLiquidity(alice, salt, deltas, callbackData, "").
   b. Implements the liquidity callback to transfer required tokens to P.

4. Inside P.addLiquidity (MetricOmmPool.sol L191):
   - _beforeAddLiquidity(R, alice, ...) is called.
   - Extension E receives: sender=R (ignored), owner=alice.
   - Check: allowedDepositor[P][alice] == true ‚Üí passes.
   - LiquidityLib credits LP shares to alice.
   - Callback fires on R; R transfers tokens to P.

Result
‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
5. Bob (via R) has deposited tokens into P without being on the allowlist.
6. Alice holds an LP position she never requested and must call
   removeLiquidity herself to exit (MetricOmmPool.sol L206).
7. Pool admin's deposit restriction is silently violated.
```
