### Title
Deposit to Zero Subaccount Permanently Locks Tokens and Corrupts Accounting - (File: `core/contracts/Endpoint.sol`)

### Summary
`depositCollateralWithReferral` is a `public` function that accepts a caller-supplied `bytes32 subaccount` without validating that the embedded address (first 20 bytes) is non-zero. Calling it with `bytes32(0)` transfers real tokens from the caller into the protocol and queues a slow-mode `DepositCollateral` transaction crediting `bytes32(0)`. When the sequencer executes that transaction, `SpotEngine.updateBalance` inflates `totalDepositsNormalized` for a subaccount that can never be controlled or withdrawn from, permanently locking the deposited tokens.

### Finding Description
`Endpoint.depositCollateralWithReferral` is declared `public` and takes a raw `bytes32 subaccount` parameter:

```solidity
function depositCollateralWithReferral(
    bytes32 subaccount,
    uint32 productId,
    uint128 amount,
    string memory
) public {
    require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);
    address sender = address(bytes20(subaccount));   // address(0) when subaccount == bytes32(0)
    requireUnsanctioned(msg.sender);
    requireUnsanctioned(sender);                     // address(0) is not sanctioned → passes
    ...
    handleDepositTransfer(..., msg.sender, uint256(amount));  // tokens pulled from caller
    slowModeTxs[...] = SlowModeTx({
        sender: sender,                              // address(0)
        tx: abi.encodePacked(..., DepositCollateral({ sender: subaccount, ... }))
    });
``` [1](#0-0) 

There is no `require(address(bytes20(subaccount)) != address(0))` guard. When `subaccount = bytes32(0)`:

1. `sender = address(0)` — not sanctioned, so `requireUnsanctioned` passes.
2. `handleDepositTransfer` pulls `amount` tokens from `msg.sender` into the `Endpoint`/`Clearinghouse`.
3. A slow-mode `DepositCollateral` transaction is enqueued with `sender = bytes32(0)`.
4. When the sequencer executes it, `Clearinghouse.depositCollateral` calls `spotEngine.updateBalance(productId, bytes32(0), amountRealized)`. [2](#0-1) 

`SpotEngineState._setBalanceAndUpdateBitmap` then writes the balance into `balances[productId][bytes32(0)]` and increments `totalDepositsNormalized`. [3](#0-2) 

No withdrawal path exists for `bytes32(0)`: `withdrawCollateral` is `onlyEndpoint` and requires a valid signed transaction from the subaccount owner; `address(0)` has no private key.

Compare with `withdrawCollateral`, which explicitly handles the zero-address case for `sendTo` but has no equivalent guard on the deposit side for the subaccount itself: [4](#0-3) 

### Impact Explanation
- Deposited tokens are permanently locked inside the protocol — no one can ever withdraw them from `bytes32(0)`.
- `totalDepositsNormalized` for the affected product is inflated, corrupting the utilization ratio used to compute borrow/deposit interest rates for all users of that product.
- The `insurance` fund is unaffected, but the spot engine's accounting invariant (total deposits ≥ total withdrawable balances) is broken.

### Likelihood Explanation
The function is `public` with no access control beyond the sanctions check. Any EOA or contract integration that mistakenly passes `bytes32(0)` (e.g., an uninitialized variable, a failed address-to-bytes32 encoding) triggers the loss. A malicious actor can also deliberately call it to grief the interest-rate accounting of any spot product. Likelihood is medium: accidental miscalls from integrators are realistic; deliberate griefing requires only a minimum deposit amount.

### Recommendation
Add a zero-address guard at the top of `depositCollateralWithReferral`:

```solidity
function depositCollateralWithReferral(
    bytes32 subaccount,
    uint32 productId,
    uint128 amount,
    string memory referral
) public {
+   require(
+       address(bytes20(subaccount)) != address(0),
+       "deposit to zero address"
+   );
    require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);
    ...
```

This mirrors the existing pattern in `withdrawCollateral` where `sendTo == address(0)` is explicitly handled, and is directly analogous to the `require(_to != address(0), "mint to the zero address")` fix recommended in the external report.

        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;

            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)

    function _updateBalanceNormalized(
        State memory state,
        BalanceNormalized memory balance,
        int128 balanceDelta
    ) internal pure {
        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized -= balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized += balance.amountNormalized;
        }

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        int128 newAmount = balance.amountNormalized.mul(
            cumulativeMultiplierX18
        ) + balanceDelta;

        if (newAmount > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);

        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized += balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized -= balance.amountNormalized;
        }
    }
