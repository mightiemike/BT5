### Title
Fee-on-Transfer Token Deposit Credits Full Amount Without Balance-Delta Verification — (`core/contracts/EndpointStorage.sol`)

---

### Summary

`EndpointStorage.handleDepositTransfer` performs two sequential token transfers without checking the actual balance received. When a fee-on-transfer ERC20 is used as a spot product, the Clearinghouse receives `amount - fee` tokens but the depositor's subaccount is credited with the full `amount`, creating an unbacked collateral surplus that corrupts protocol solvency.

---

### Finding Description

The deposit path in Nado is:

1. **`Endpoint.depositCollateralWithReferral`** (Endpoint.sol:123–167) accepts a caller-supplied `amount`, calls `handleDepositTransfer`, then enqueues a slow-mode `DepositCollateral` transaction carrying that same `amount`.

2. **`EndpointStorage.handleDepositTransfer`** (EndpointStorage.sol:111–119) executes two transfers in sequence:
   ```solidity
   safeTransferFrom(token, from, amount);               // user → Endpoint
   safeTransferTo(token, address(clearinghouse), amount); // Endpoint → Clearinghouse
   ```
   Neither call checks the actual balance delta. With a fee-on-transfer token, the first call delivers only `amount - fee` to the Endpoint. The second call then attempts to forward the full `amount` to the Clearinghouse. If the Endpoint holds a residual balance from prior deposits, the second transfer silently succeeds by consuming other depositors' tokens; otherwise it reverts.

3. **`Clearinghouse.depositCollateral`** (Clearinghouse.sol:193–209) is later invoked by the sequencer processing the slow-mode queue. It reads `txn.amount` — the original, unverified figure — and calls `spotEngine.updateBalance(txn.productId, txn.sender, amountRealized)`, crediting the full `amount` regardless of what was actually received.

No balance-before/after check exists anywhere in this path.

---

### Impact Explanation

- The depositor's subaccount balance is inflated by `fee` tokens that were never deposited into the Clearinghouse.
- The Clearinghouse's aggregate token holdings fall below the sum of all credited subaccount balances — a direct solvency shortfall.
- Repeated deposits by the same or different users compound the discrepancy, eventually causing legitimate withdrawals to fail when the Clearinghouse cannot cover them.
- The corrupted collateral balance also inflates the depositor's health score, allowing them to open larger positions than their real collateral supports.

---

### Likelihood Explanation

The entry point is the public `depositCollateral` / `depositCollateralWithReferral` functions on `Endpoint.sol`, callable by any unprivileged user. The only prerequisite is that a fee-on-transfer token is registered as a spot product. Token registration is an admin action, but once any such token is listed, every deposit of that token by any user triggers the accounting discrepancy. The protocol does not document or enforce a restriction against fee-on-transfer tokens in spot product registration.

---

### Recommendation

In `EndpointStorage.handleDepositTransfer`, record the Clearinghouse's balance before and after the transfer and use the delta — not the input `amount` — as the credited quantity. Propagate the verified amount back to `depositCollateralWithReferral` so the slow-mode transaction is enqueued with the actual received amount:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal returns (uint256 actualReceived) {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    actualReceived = token.balanceOf(address(clearinghouse)) - before;
    require(actualReceived == amount, "Fee-on-transfer tokens not supported");
}
```

Alternatively, revert immediately if `actualReceived != amount` to explicitly block fee-on-transfer tokens, matching the intent of the protocol.

---

### Proof of Concept

**Trigger path:**

1. Admin registers a fee-on-transfer ERC20 (10% fee on every transfer) as a spot product via `SpotEngine`.
2. Alice calls `Endpoint.depositCollateral(subaccountName, productId, 1000e18)`.
3. `handleDepositTransfer` pulls 1000e18 from Alice; Endpoint receives 900e18 (fee taken). Endpoint then forwards 1000e18 to Clearinghouse — succeeds only if Endpoint has ≥100e18 residual balance, draining other users' tokens.
4. Slow-mode tx is queued with `amount = 1000e18`.
5. Sequencer processes the queue; `Clearinghouse.depositCollateral` credits Alice's subaccount with `1000e18 * multiplier`.
6. Clearinghouse holds 900e18 but Alice's balance reflects 1000e18 — 100e18 of unbacked collateral.
7. Alice can now trade or withdraw against the phantom 100e18, leaving the protocol insolvent by that amount per deposit.
