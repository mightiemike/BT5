### Title
`DISABLE_OP` Hardcoded in `MEMPOOL_MODE` Silently Blocks `op_modpow` (Opcode 60), Creating Consensus Divergence — (`File: src/chia_dialect.rs`)

---

### Summary

`ClvmFlags::DISABLE_OP` is unconditionally included in the `MEMPOOL_MODE` constant. The only operator that checks this flag to completely disable itself is `op_modpow` (opcode 60). As a result, any CLVM program that uses `op_modpow` is rejected by the mempool with `EvalErr::Unimplemented`, while the identical program succeeds in consensus (block-validation) mode. This is a direct analog to the reported Factory decimal-check bug: a hardcoded constant silently blocks an operator that the VM is fully implemented to support.

---

### Finding Description

**Root cause — `MEMPOOL_MODE` includes `DISABLE_OP`:**

```rust
// src/chia_dialect.rs lines 72-76
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)      // ← hardcoded, no comment
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [1](#0-0) 

**`DISABLE_OP` completely disables `op_modpow` (opcode 60):**

```rust
// src/chia_dialect.rs lines 239-243
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
``` [2](#0-1) 

`op_modpow` is a fully implemented, documented operator with its own cost model, size limits, and GC-candidate registration. It is listed in `gc_candidate` alongside all other standard operators: [3](#0-2) 

The `DISABLE_OP` flag itself carries no explanatory comment: [4](#0-3) 

**Secondary observation — `DISABLE_OP` checks in `op_div`/`op_divmod`/`op_mod` are dead code:**

In those three operators the `DISABLE_OP` guard checks `a0_len > 2048`, but the unconditional guard immediately below checks `a0_len > 256`. Because `a0_len > 2048` implies `a0_len > 256`, the `DISABLE_OP` branch can never be the first to fire; both branches return the same error. This means `DISABLE_OP` has no observable effect on `op_div`, `op_divmod`, or `op_mod` — its only real effect in the entire codebase is the complete suppression of `op_modpow`. [5](#0-4) 

---

### Impact Explanation

Any CLVM puzzle that calls `op_modpow` (opcode 60) is rejected by the Chia node's mempool with `EvalErr::Unimplemented`. The same puzzle, if included directly in a block, executes successfully under consensus rules (where `DISABLE_OP` is not set). This creates a **consensus/mempool divergence**: coins locked by puzzles that use `op_modpow` cannot be spent through the standard transaction submission path, even though the on-chain rules permit it. The corrupted result is the wrong `Response` — `Err(EvalErr::Unimplemented)` instead of a valid `Reduction(cost, result)`.

---

### Likelihood Explanation

Any caller that passes attacker-controlled CLVM bytes containing opcode `0x3c` (decimal 60) to `run_program` with `MEMPOOL_MODE` flags will trigger the rejection. The entry path is direct: the Chia full node calls `run_program` with `MEMPOOL_MODE` for every transaction it validates before adding it to the mempool. No special privileges are required; a normal coin spend is sufficient.

---

### Recommendation

Either:
1. Remove `ClvmFlags::DISABLE_OP` from `MEMPOOL_MODE` if `op_modpow` is intended to be available, or
2. Add an explicit, named flag (e.g., `DISABLE_MODPOW`) with a comment explaining the hard-fork activation condition, and apply it only to `op_modpow`.

The dead-code `DISABLE_OP` checks in `op_div`, `op_divmod`, and `op_mod` should also be removed or corrected to avoid future confusion.

### Proof of Concept

```rust
use clvmr::allocator::Allocator;
use clvmr::chia_dialect::{ChiaDialect, MEMPOOL_MODE};
use clvmr::run_program::run_program;

fn main() {
    let mut allocator = Allocator::new();

    // Build: (modpow (q . 3) (q . 10) (q . 7))
    // opcode 60 = 0x3c
    let opcode = allocator.new_atom(&[60]).unwrap();
    let base    = allocator.new_small_number(3).unwrap();
    let exp     = allocator.new_small_number(10).unwrap();
    let modulus = allocator.new_small_number(7).unwrap();
    // ... (build quoted args and program list) ...

    let dialect_mempool   = ChiaDialect::new(MEMPOOL_MODE);
    let dialect_consensus = ChiaDialect::new(clvmr::chia_dialect::ClvmFlags::empty());

    // Mempool: EvalErr::Unimplemented  ← bug
    let r1 = run_program(&mut allocator, &dialect_mempool, program, env, u64::MAX);
    assert!(r1.is_err());

    // Consensus: Ok(Reduction(cost, result_4))  ← correct
    let r2 = run_program(&mut allocator, &dialect_consensus, program, env, u64::MAX);
    assert!(r2.is_ok());
}
```

The divergence is triggered by any attacker-controlled CLVM byte sequence containing opcode `0x3c` submitted to the mempool path.
