### Title
Unprivileged Caller Can Arbitrarily Manipulate Any User's Fee Tier via Unguarded `UpdateFeeTier` Slow Mode Transaction — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `processSlowModeTransactionImpl` function in `EndpointTx.sol` handles `UpdateFeeTier` slow mode transactions without any sender authorization check. Unlike every other user-affecting slow mode transaction type (`WithdrawCollateral`, `LinkSigner`, `ClaimBuilderFee`), the `UpdateFeeTier` branch does not call `validateSender`. Combined with the fact that `UpdateFeeTier` is not in the owner-only submission guard in `submitSlowModeTransactionImpl`, any unprivileged caller can pay the slow mode fee, submit a `UpdateFeeTier` transaction targeting any arbitrary user address with any tier value, and after the 3-day delay execute it to permanently alter that user's fee tier.

---

### Finding Description

**Root cause — submission is not owner-gated:**

In `submitSlowModeTransactionImpl`, a set of privileged transaction types requires `sender == owner()`: [1](#0-0) 

`UpdateFeeTier` is absent from this list. It falls into the `else` branch, meaning any caller can submit it by paying the slow mode fee.

**Root cause — execution has no sender validation:**

When the slow mode transaction is later executed, `processSlowModeTransactionImpl` handles `UpdateFeeTier` as: [2](#0-1) 

The `sender` parameter (the original submitter, passed from `_executeSlowModeTransaction`) is completely ignored. There is no `validateSender(txn.user, sender)` call and no ownership check.

**Contrast with protected transaction types:**

Every other user-affecting slow mode transaction type calls `validateSender` to bind the submitter to the affected subaccount: [3](#0-2) [4](#0-3) [5](#0-4) 

`UpdateFeeTier` is the only user-affecting slow mode transaction type that skips this check entirely.

**The `validateSender` function itself:** [6](#0-5) 

**Fee tier lookup used at trade time:** [7](#0-6) 

The fee tier stored in `feeTiers[user]` is read directly at order matching time to determine the maker/taker rate applied to every trade. Changing it changes the actual fees charged.

**The `updateFeeTier` write path:** [8](#0-7) 

This function is callable only by the clearinghouse, which is called by the endpoint. The missing guard is upstream, in the slow mode submission and execution path.

---

### Impact Explanation

An unprivileged attacker can:

1. **Self-upgrade fee tier**: Submit a `UpdateFeeTier` slow mode transaction with `user = attacker_address` and `newTier = T` where tier `T` has been configured with 0 maker fees or a negative maker rebate. After execution, the attacker trades at reduced or zero cost indefinitely, extracting value from the protocol's fee revenue.

2. **Downgrade a victim's fee tier**: Submit a `UpdateFeeTier` transaction with `user = victim_address` and `newTier = 0` (or any tier with higher fees), stripping a privileged user of their fee discount. This is a direct financial harm to the victim on every subsequent trade.

The corrupted state is `feeTiers[user]` in `OffchainExchange`, which is read on every order match. The impact is continuous and accumulates with trading volume.

**Impact score: High** — direct, ongoing financial loss to protocol or users.

---

### Likelihood Explanation

The attack requires:
- Paying the slow mode fee (a small fixed USDC amount, `SLOW_MODE_FEE`)
- Waiting 3 days for the slow mode delay to expire
- Calling the public `executeSlowModeTransaction()` function

No privileged access, no leaked keys, no governance capture. Any EOA with enough USDC for the slow mode fee can execute this.

**Likelihood score: High** — low barrier, no privilege required, fully on-chain.

---

### Recommendation

Add a sender authorization check in the `UpdateFeeTier` branch of `processSlowModeTransactionImpl`, requiring that the submitter is the owner (matching the intent of the fast-path sequencer gate), or alternatively add `UpdateFeeTier` to the owner-only list in `submitSlowModeTransactionImpl`:

```solidity
} else if (
    txType == IEndpoint.TransactionType.WithdrawInsurance ||
    ...
    txType == IEndpoint.TransactionType.UpdateBuilder ||
    txType == IEndpoint.TransactionType.UpdateFeeTier  // ADD THIS
) {
    require(sender == owner());
```

This matches the access model of the fast path, where `UpdateFeeTier` is only reachable via `submitTransactionsChecked` which requires `msg.sender == sequencer` plus multi-signer verification. [9](#0-8) 

---

### Proof of Concept

1. Attacker identifies a fee tier `T` configured with 0 taker fees via `updateTierFeeRates` (admin-set, publicly readable from `feeRates[T][productId]`).
2. Attacker calls `Endpoint.submitSlowModeTransaction` with transaction bytes encoding `TransactionType.UpdateFeeTier` and payload `{user: attacker_address, newTier: T}`. Pays `SLOW_MODE_FEE` in quote token.
3. After 3 days, attacker (or anyone) calls `Endpoint.executeSlowModeTransaction()`.
4. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(attacker_address, txBytes)`.
5. `processSlowModeTransactionImpl` reaches the `UpdateFeeTier` branch: calls `clearinghouse.updateFeeTier(transaction)` — no `validateSender`, `sender` is ignored.
6. `Clearinghouse.updateFeeTier` → `IOffchainExchange.updateFeeTier(attacker_address, T)` → `feeTiers[attacker_address] = T`.
7. All subsequent orders from the attacker's subaccount are matched using tier `T` fee rates (0 taker fee), permanently reducing their trading cost.
