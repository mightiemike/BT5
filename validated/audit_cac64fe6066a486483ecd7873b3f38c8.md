### Title
Raw `approve()` Call in `DirectDepositV1.creditDeposit()` Reverts for Non-Standard ERC20 Tokens — (`File: core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve()` as a high-level Solidity call against the `IIERC20Base` interface, which declares `approve` as returning `bool`. Non-standard ERC20 tokens such as USDT return no data from `approve`. The Solidity ABI decoder will revert when it attempts to decode the empty return buffer as `bool`, causing the entire deposit loop to fail. The same contract already implements a `safeTransfer` helper that correctly handles this case using a low-level call, but `approve` is left unguarded.

---

### Finding Description

`IIERC20Base` in `DirectDepositV1.sol` declares `approve` with a `bool` return: [1](#0-0) 

Inside `creditDeposit()`, the contract iterates over every registered spot product and issues a direct high-level `approve` call: [2](#0-1) 

When the token at `tokenAddr` is a non-standard ERC20 (e.g., USDT, which returns `void` from `approve`), the Solidity runtime tries to ABI-decode the empty return data as `bool` and reverts. Because the loop has no per-token error isolation, a single non-standard token in the spot product list causes the **entire** `creditDeposit()` call to revert, blocking deposits for every token in that iteration.

The inconsistency is explicit: the same contract already implements a `safeTransfer` helper that uses a low-level `.call` and accepts `data.length == 0` as a success indicator: [3](#0-2) 

No equivalent safe wrapper exists for `approve`.

A secondary instance of the same root cause exists in `ContractOwner.wrapVaultAsset()`, which issues two raw `approve` calls against the underlying asset of an ERC4626 vault token, also with no access restriction: [4](#0-3) 

---

### Impact Explanation

Any caller invoking `creditDeposit()` (no access control) when a non-standard ERC20 is registered as a spot product will receive a revert. Because the loop is not try/catch-isolated, **all** tokens in the product list are blocked in that call, not just the non-standard one. Funds already sitting in the `DirectDepositV1` contract cannot be credited to the subaccount until the issue is resolved, effectively locking deposited collateral. The `wrapVaultAsset()` path similarly becomes permanently broken for any ERC4626 vault whose underlying asset is a non-standard ERC20.

---

### Likelihood Explanation

USDT is one of the most commonly deposited stablecoins on any trading protocol. If the Nado deployment lists USDT (or any other void-returning token) as a spot product — which is the expected production configuration for a multi-asset exchange — every call to `creditDeposit()` will revert. The entry path requires no privilege: any address can call `creditDeposit()` directly on the deployed `DirectDepositV1` contract, or trigger it via `ContractOwner.creditDepositV1()`. [5](#0-4) 

---

### Recommendation

Replace the raw `token.approve(...)` call in `creditDeposit()` with a low-level safe wrapper analogous to the existing `safeTransfer` helper, or use OpenZeppelin's `SafeERC20.forceApprove`. The wrapper must accept `data.length == 0` as a valid success response:

```solidity
function safeApprove(IIERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IIERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        "Approve failed"
    );
}
```

Apply the same fix to the two `assetToken.approve(...)` calls in `ContractOwner.wrapVaultAsset()`. [4](#0-3) 

---

### Proof of Concept

1. Deploy or fork a network where USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) is registered as a spot product in the Nado `SpotEngine`.
2. Send any amount of USDT to the `DirectDepositV1` contract address for a given subaccount.
3. Call `creditDeposit()` on that contract (no privilege required).
4. The call reverts at line 92 because `USDT.approve()` returns no data, but the Solidity ABI decoder expects a `bool`.
5. The USDT balance remains stuck in the `DirectDepositV1` contract; the subaccount receives no credit.

The existing `safeTransfer` at lines 69–81 of the same file demonstrates the correct pattern and confirms the developer was aware of non-standard token behavior for `transfer` but did not apply the same guard to `approve`. [3](#0-2) [6](#0-5)
