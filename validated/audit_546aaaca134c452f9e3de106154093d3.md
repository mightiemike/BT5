### Title
Duplicate-Last-Leaf Position Aliasing in `compute_root_from_merkle_proof` Allows Same Transaction to Verify at Two Distinct `tx_index` Values — (`merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` performs no bounds check on `transaction_position`. In an odd-width Bitcoin Merkle tree the last leaf is duplicated during tree construction. This means the same `(tx_id, merkle_proof)` pair produces the correct Merkle root for **both** `tx_index = N-1` (the real last leaf) and `tx_index = N` (the phantom out-of-bounds duplicate). `verify_transaction_inclusion` returns `true` for both calls, breaking the invariant that a Merkle proof binds to exactly one transaction position.

---

### Finding Description

**`compute_root_from_merkle_proof` — no position bounds check** [1](#0-0) 

The function accepts any `transaction_position` value and blindly uses `current_position % 2` to decide left/right at each level. There is no check that `transaction_position < tree_leaf_count`.

**Why the duplicate-leaf property creates an alias**

For a 3-transaction block `[T0, T1, T2]`, Bitcoin's Merkle construction pads to `[T0, T1, T2, T2]`. The proof for `T2` at position 2 is `[T2, H(T0,T1)]`.

| Step | Position 2 (real) | Position 3 (phantom) |
|------|-------------------|----------------------|
| Round 0 | pos=2 (even) → `H(T2, T2)`, pos→1 | pos=3 (odd) → `H(T2, T2)`, pos→1 |
| Round 1 | pos=1 (odd) → `H(H(T0,T1), H(T2,T2))` = root ✓ | pos=1 (odd) → `H(H(T0,T1), H(T2,T2))` = root ✓ |

Both paths produce the same root. `verify_transaction_inclusion` therefore returns `true` for both `tx_index=2` and `tx_index=3` with the same `tx_id` and `merkle_proof`.

**`verify_transaction_inclusion` passes the attacker-controlled `tx_index` directly** [2](#0-1) 

`args.tx_index` is caller-supplied and cast to `usize` without any upper-bound validation against the proof length or tree size.

**The canonical-chain check does not mitigate this** [3](#0-2) 

The check only confirms the block is on the canonical chain. It says nothing about whether `tx_index` is within the actual transaction count of that block.

---

### Impact Explanation

Any higher-level contract that calls `verify_transaction_inclusion` and uses `(tx_id, tx_block_blockhash, tx_index)` as a unique spend key — the standard pattern for BTC light-client bridges — can be double-spent. An attacker holding a valid proof for the last transaction in any odd-width block can submit:

1. `tx_index = N-1` → `true` → first withdrawal/release
2. `tx_index = N` (phantom) → `true` → second withdrawal/release for the same on-chain transaction

The "wrong-block" framing in the question is **not** achievable: the proof is still tied to the correct block's `merkle_root`, so cross-block forgery is not possible. The real impact is **intra-block double-spend via position aliasing**.

---

### Likelihood Explanation

- The function is public and callable by any unprivileged account.
- No special chain state is required beyond a canonical block with an odd transaction count (extremely common in Bitcoin).
- The attacker only needs a valid SPV proof for the last transaction in any such block, which is publicly available from any Bitcoin full node.
- The deprecated status of `verify_transaction_inclusion` does not prevent calls; the `#[deprecated]` attribute is a Rust lint warning only and does not restrict on-chain invocation. [4](#0-3) 

---

### Recommendation

In `compute_root_from_merkle_proof`, validate that `transaction_position < 2^merkle_proof.len()` (i.e., the position must be within the padded leaf count implied by the proof length). Concretely:

```rust
let max_position = 1usize << merkle_proof.len(); // padded tree width
assert!(transaction_position < max_position, "tx_index out of bounds");
```

Additionally, for odd-width trees, reject any `transaction_position` that equals the phantom duplicate slot. The padded width is `2^proof_len`; if the real leaf count is known, reject `transaction_position >= real_leaf_count`.

---

### Proof of Concept

```rust
// 3-tx odd-width tree: [T0, T1, T2]
// Padded: [T0, T1, T2, T2]
// Proof for T2 at index 2: [T2, H(T0,T1)]

let t2   = decode_hex("048f3897c16bdc59ec1187aa080a4b4aa5ec1afcb4b776cf8b8a214b01990a7b");
let t2_dup_proof = vec![
    t2.clone(),                          // sibling at level 0 (the duplicate)
    compute_hash(&t0_hash, &t1_hash),    // H(T0,T1) at level 1
];

// Both calls must return the same root (== block's merkle_root)
let root_at_2 = compute_root_from_merkle_proof(t2.clone(), 2, &t2_dup_proof);
let root_at_3 = compute_root_from_merkle_proof(t2.clone(), 3, &t2_dup_proof);
assert_eq!(root_at_2, root_at_3); // passes — same root, two positions
```

`verify_transaction_inclusion` returns `true` for both `tx_index=2` and `tx_index=3` against the same canonical block, enabling a double-spend against any bridge that keys on `(tx_id, blockhash, tx_index)`. [5](#0-4)

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
