### Title
CEI Violation in `withdrawCollateral` Allows Double-Spend of Collateral via ERC777 Token Callback — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` transfers tokens to the recipient **before** decrementing the sender's internal balance. For tokens with transfer-hook callbacks (ERC777), a malicious recipient contract can re-enter `Endpoint.depositCollateralWithReferral` during the callback, queue a slow-mode credit for the same amount, and then have the balance decremented to zero — only to have it restored later by the queued slow-mode transaction. The net effect is that the attacker withdraws real tokens while retaining the full on-chain balance.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the call to `handleWithdrawTransfer` (which ultimately calls `token.safeTransfer(sendTo, amount)` inside `BaseWithdrawPool.handleWithdrawTransfer`) is made **before** `spotEngine.updateBalance` is called to debit the sender's balance:

```
withdrawCollateral
  └─ handleWithdrawTransfer(token, sendTo, amount, idx)          ← external call chain
       └─ token.safeTransfer(withdrawPool, amount)
       └─ BaseWithdrawPool.submitWithdrawal(token, sendTo, amount, idx)
            └─ token.safeTransfer(sendTo, amount)                ← user receives tokens HERE
  └─ spotEngine.updateBalance(productId, sender, -amountRealized) ← balance decremented AFTER
  └─ getHealth(sender, healthType) >= 0
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

For any ERC777-compatible collateral token, `token.safeTransfer(sendTo, amount)` triggers the `tokensReceived` hook on `sendTo` if it is a contract implementing `IERC777Recipient`. At that moment, the sender's internal balance has **not yet been decremented**, so the system still shows the full pre-withdrawal balance.

During the hook, the attacker calls `Endpoint.depositCollateralWithReferral`, which:
1. Pulls the just-received tokens back from the attacker's contract into the clearinghouse via `handleDepositTransfer`.
2. Enqueues a `SlowModeTx` to credit the same amount back to the attacker's balance. [5](#0-4) 

After the hook returns, `withdrawCollateral` decrements the balance to zero and the health check passes. When the sequencer later processes the queued slow-mode transaction, the balance is restored to the full withdrawal amount — giving the attacker both the withdrawn tokens **and** a restored on-chain balance to withdraw again.

---

### Impact Explanation

An attacker who controls a contract address as their subaccount wallet and uses an ERC777 collateral token can:

- Withdraw their full collateral balance in real tokens.
- Simultaneously re-deposit those tokens during the callback, queuing a slow-mode credit.
- After the slow-mode credit is processed, withdraw the same amount a second time.

This is a direct theft of protocol collateral. The corrupted state delta is `spotEngine.balances[productId][sender]`: it is decremented to zero by the withdrawal but then re-credited by the slow-mode transaction, allowing a second withdrawal of the same funds.

---

### Likelihood Explanation

The trigger requires:
1. A supported collateral token that implements ERC777 transfer hooks (backward-compatible with ERC20).
2. The attacker's subaccount address to be a contract implementing `IERC777Recipient`.

The protocol places no restriction on ERC777 tokens in `IERC20Base` or anywhere in the collateral registration path. Any operator-added token with transfer hooks activates this path. The attacker controls both conditions (they deploy the contract and choose the token product). Likelihood is **medium** — conditional on an ERC777 token being listed, but the code is structurally vulnerable by design.

---

### Recommendation

Apply the Checks-Effects-Interactions pattern: decrement the internal balance **before** making the external token transfer.

### Proof of Concept

1. Attacker deploys `MaliciousWallet` implementing `IERC777Recipient`, using it as their subaccount address.
2. Attacker deposits `X` units of an ERC777 collateral token. Balance = `X`.
3. Attacker signs and submits a `WithdrawCollateral` transaction for `X`. Sequencer calls `withdrawCollateral`.
4. Inside `handleWithdrawTransfer` → `BaseWithdrawPool.submitWithdrawal` → `token.safeTransfer(MaliciousWallet, X)`.
5. ERC777 triggers `MaliciousWallet.tokensReceived`. At this point, `spotEngine.balances[productId][attacker] == X` (not yet decremented).
6. `MaliciousWallet` calls `Endpoint.depositCollateralWithReferral(subaccount, productId, X)`. Tokens are pulled back; a slow-mode tx is queued to credit `X`.
7. Hook returns. `spotEngine.updateBalance(productId, sender, -X)` → balance = `0`. Health check passes.
8. Sequencer processes the queued slow-mode tx → balance = `X` again.
9. Attacker submits a second withdrawal for `X`. Net gain: `X` tokens stolen from the protocol. [6](#0-5)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });

    }
