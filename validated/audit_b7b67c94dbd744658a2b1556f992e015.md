### Title
CEI Violation in `withdrawCollateral`: External Token Transfer Precedes Balance Update — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` transfers tokens to `sendTo` via an external call before decrementing the subaccount's balance in `spotEngine`. No reentrancy guard exists anywhere in the protocol. This mirrors the exact bug class in the reference report: an external call before a critical state update, enabling a malicious recipient contract to re-enter the system while the on-chain balance still reflects the pre-withdrawal amount.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` — triggers an external token transfer chain ending in `token.safeTransfer(sendTo, amount)` inside `BaseWithdrawPool.handleWithdrawTransfer`.
2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)` — the subaccount balance is decremented **only after** the external call returns. [1](#0-0) 

The internal `handleWithdrawTransfer` in `Clearinghouse` routes through `BaseWithdrawPool.submitWithdrawal`, which calls `token.safeTransfer(sendTo, amount)`: [2](#0-1) [3](#0-2) 

There is **no `nonReentrant` modifier** on any function in the protocol — confirmed by a full-codebase search returning zero matches for `ReentrancyGuard` or `nonReentrant`.

---

### Impact Explanation

If `sendTo` is a contract that implements a token receive hook (e.g., ERC777 `tokensReceived`, or any token with a transfer callback), the hook fires during `token.safeTransfer(sendTo, amount)` at line 189 of `BaseWithdrawPool.sol`. At that moment:

- `spotEngine` still records the **full pre-withdrawal balance** for `sender` (the deduction at line 412 has not yet executed).
- Any public protocol function that reads or acts on `spotEngine` balances will observe the stale, inflated state.

The most direct consequence: a malicious `sendTo` contract can call `BaseWithdrawPool.submitFastWithdrawal` (a fully public, permissionless function) during the callback with a pre-signed withdrawal transaction, drawing additional tokens from the fast-withdrawal pool against a balance that has not yet been decremented. The user receives tokens from both the regular withdrawal path and the fast-withdrawal pool, but `spotEngine` is only decremented once when the original `withdrawCollateral` resumes. [4](#0-3) 

---

### Likelihood Explanation

- The trigger requires a token with a transfer callback (ERC777 or equivalent). The protocol does not restrict supported token types to plain ERC20.
- The attacker controls `sendTo` (user-supplied via `WithdrawCollateralV2`) and can deploy a malicious recipient contract.
- `submitFastWithdrawal` is public with no access control beyond verifier signatures on the withdrawal transaction itself — a user can obtain a legitimately signed withdrawal transaction and hold it for use during the callback.
- No reentrancy guard exists anywhere in the codebase to block the re-entry. [5](#0-4) 

---

### Recommendation

Apply the CEI pattern: move `spotEngine.updateBalance` (line 412) and `spotEngine.assertUtilization` (line 413) **before** the `handleWithdrawTransfer` call (line 408). Additionally, add OpenZeppelin's `ReentrancyGuard` (`nonReentrant`) to `withdrawCollateral` and `BaseWithdrawPool.submitFastWithdrawal` as a defense-in-depth measure.

---

### Proof of Concept

```
1. Attacker deploys MaliciousRecipient implementing ERC777 tokensReceived hook.
2. Attacker submits WithdrawCollateralV2 with sendTo = MaliciousRecipient, amount = X.
3. Sequencer processes → Endpoint → Clearinghouse.withdrawCollateral.
4. handleWithdrawTransfer fires → token.safeTransfer(MaliciousRecipient, X).
5. tokensReceived hook fires. At this point spotEngine.getBalance(productId, sender) = X (not yet decremented).
6. MaliciousRecipient calls BaseWithdrawPool.submitFastWithdrawal with a pre-signed WithdrawCollateral tx for the same user.
7. submitFastWithdrawal transfers an additional X tokens from the fast-withdrawal pool to MaliciousRecipient.
8. Control returns to withdrawCollateral. spotEngine.updateBalance decrements balance by X (once).
9. Net result: attacker received 2X tokens, balance decremented by X only.
``` [6](#0-5) [7](#0-6)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }

        uint64 idx,

        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
