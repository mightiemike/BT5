### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Guard Introduced in v2 - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof before delegating to `verify_transaction_inclusion`. However, `verify_transaction_inclusion` retains `pub` visibility and is a fully reachable NEAR contract method. Any unprivileged caller can invoke it directly, bypassing the coinbase guard entirely. The `#[deprecated]` attribute is a Rust compile-time hint only; it has no effect on NEAR runtime dispatch.

---

### Finding Description

`verify_transaction_inclusion_v2` acts as the security gatekeeper:

1. It validates that `coinbase_merkle_proof` and `merkle_proof` have equal length.
2. It recomputes the Merkle root from the coinbase transaction at index 0 and asserts it matches the stored block's `merkle_root`.
3. Only after both checks pass does it delegate to `verify_transaction_inclusion`. [1](#0-0) 

`verify_transaction_inclusion` is the inner function. It is marked `#[deprecated]` but remains `pub` and is decorated only with `#[pause]` — no access restriction beyond the pause gate: [2](#0-1) 

The function performs a raw Merkle root recomputation with no coinbase anchor check: [3](#0-2) 

Because NEAR's runtime dispatches any `pub` method on a `#[near]` impl block as a callable contract method, `#[deprecated]` provides zero runtime enforcement. An attacker submits a NEAR transaction calling `verify_transaction_inclusion` directly with a crafted proof, and the coinbase guard in v2 is never reached.

---

### Impact Explanation

The 64-byte transaction vulnerability (documented in the code's own warning at line 278 and the bitmex link at line 268) allows an attacker to supply a 64-byte value that is a valid internal Merkle tree node rather than a real transaction hash. Without the coinbase anchor, the contract cannot distinguish an internal node from a leaf. The contract returns `true` for a transaction that was never included in the block. Any downstream NEAR dApp that calls `verify_transaction_inclusion` directly — or that an attacker tricks into using the v1 method — receives a forged proof result, enabling false SPV attestations. [4](#0-3) 

---

### Likelihood Explanation

The entry path requires no privileges: any NEAR account can submit a transaction calling `verify_transaction_inclusion`. The method is not paused by default. Constructing a valid 64-byte forgery requires knowledge of the target block's Merkle tree structure, which is public Bitcoin chain data. The attacker only needs to find a block where an internal node's 64-byte serialization can be presented as a `tx_id` with a matching partial proof path — a known, documented attack class with published tooling.

---

### Recommendation

Remove `pub` from `verify_transaction_inclusion` and make it `pub(crate)` or `fn` (private), so it is only reachable internally from `verify_transaction_inclusion_v2`. Alternatively, if backward compatibility with existing callers is required, add an explicit `env::panic_str` guard at the top of `verify_transaction_inclusion` that rejects direct external calls, or remove the method from the NEAR ABI entirely by moving it out of the `#[near] impl` block. [5](#0-4) 

---

### Proof of Concept

1. Identify a Bitcoin mainchain block stored in the contract (any `tx_block_blockhash` present in `mainchain_header_to_height`).
2. Obtain the block's Merkle tree. Locate a 64-byte internal node `N` at depth `d` whose hash, when used as `tx_id` with a crafted `merkle_proof` of length `d`, produces the block's `merkle_root` via `compute_root_from_merkle_proof`.
3. Submit a NEAR transaction calling `verify_transaction_inclusion` (not v2) with:
   - `tx_id = N` (the forged internal node hash)
   - `tx_block_blockhash` = the target block hash
   - `tx_index` = the index corresponding to `N`'s position
   - `merkle_proof` = the sibling path from `N` to the root
   - `confirmations` = 1
4. The contract returns `true`. No coinbase proof was required. The 64-byte forgery guard in `verify_transaction_inclusion_v2` was never invoked. [6](#0-5)

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );
