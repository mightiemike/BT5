### Title
GC Threshold Below `difficulty_adjustment_interval` Permanently Bricks Block Submission at Difficulty Boundaries — (`contract/src/lib.rs`, `contract/src/bitcoin.rs`)

---

### Summary

`InitArgs.gc_threshold` accepts any `u64` with no lower-bound validation. When set below `difficulty_adjustment_interval` (2016 for Bitcoin), the automatic GC triggered inside `submit_blocks` prunes the period-start block from `mainchain_height_to_header`. The next `submit_blocks` call that lands on a difficulty boundary then calls `get_header_by_height(first_block_height)`, which panics with `ERR_KEY_NOT_EXIST` because the required block no longer exists. The contract is permanently bricked for all future difficulty-boundary submissions.

---

### Finding Description

**Entry point — `init` accepts unconstrained `gc_threshold`:**

`InitArgs` in `btc-types/src/contract_args.rs` exposes `gc_threshold: u64` with no minimum. [1](#0-0) 

`init()` stores it verbatim with no validation against `difficulty_adjustment_interval`: [2](#0-1) 

The doc comment only *recommends* `gc_threshold = 52704`; it enforces nothing. [3](#0-2) 

---

**GC removes the period-start block — `run_mainchain_gc`:**

After every `submit_blocks` call, GC fires automatically with `batch_size = num_of_headers`: [4](#0-3) 

`run_mainchain_gc` removes blocks from `start_removal_height` up to `end_removal_height`, deleting them from both `mainchain_height_to_header` and `headers_pool` with no awareness of which heights are needed for future difficulty calculations: [5](#0-4) 

---

**Difficulty retarget requires the pruned block — `get_next_work_required`:**

When submitting the first block of a new difficulty period (`(prev_height + 1) % 2016 == 0`), Bitcoin's retarget logic computes:

```
first_block_height = prev_block_header.block_height - (difficulty_adjustment_interval - 1)
```

and immediately calls `get_header_by_height(first_block_height)`: [6](#0-5) 

---

**`get_header_by_height` panics if the block was GC'd:** [7](#0-6) 

---

### Impact Explanation

Once GC removes the period-start block, every future `submit_blocks` call that crosses a difficulty boundary panics with `ERR_KEY_NOT_EXIST`. The state is permanently corrupted: the block is gone from both `mainchain_height_to_header` and `headers_pool`, and there is no recovery path short of a contract migration. All SPV proofs requiring confirmation of blocks at or after the first pruned difficulty boundary become permanently unverifiable.

---

### Likelihood Explanation

The `init` function is `#[private]` and `#[init]`, so only the deployer can set `gc_threshold`. However, the vulnerability does not require a malicious actor — a legitimate operator who sets `gc_threshold = 100` (e.g., to reduce storage costs during testing or a low-traffic deployment) will trigger this silently. The contract provides no guard, no warning, and no runtime check. The existing test suite uses `gc_threshold = 3` and `gc_threshold = 1000` in unit tests without ever crossing a real difficulty boundary, so the bug is not caught by tests. [8](#0-7) 

---

### Recommendation

Add a `require!` in `init_genesis` (or `init`) that enforces `gc_threshold >= difficulty_adjustment_interval` for Bitcoin/Litecoin builds. For example, in `init_genesis` after the config is loaded:

```rust
#[cfg(any(feature = "bitcoin", feature = "litecoin"))]
require!(
    args.gc_threshold >= config.difficulty_adjustment_interval,
    format!(
        "gc_threshold ({}) must be >= difficulty_adjustment_interval ({})",
        args.gc_threshold, config.difficulty_adjustment_interval
    )
);
```

Additionally, `run_mainchain_gc` should never remove a block whose height is the start of the current difficulty period.

---

### Proof of Concept

```
1. Deploy Bitcoin contract with:
   - genesis_block_height = 0
   - gc_threshold = 100
   - skip_pow_verification = true   (to avoid real PoW)

2. Submit blocks at heights 1–2015 in a single submit_blocks() call (batch_size = 2015).
   - All 2015 blocks are appended first (no difficulty boundary crossed yet).
   - run_mainchain_gc(2015) fires:
       amount_stored = 2016
       total_to_remove = 2016 - 100 = 1916
       selected = min(1916, 2015) = 1916
     → Removes heights 0..=1915 from mainchain_height_to_header and headers_pool.
     → Block at height 0 (period-start) is gone.

3. Submit block at height 2016.
   - check_pow → get_next_work_required:
       (2015 + 1) % 2016 == 0  → difficulty adjustment triggered
       first_block_height = 2015 - 2015 = 0
       get_header_by_height(0) → mainchain_height_to_header.get(0) returns None
       → env::panic_str("ERR_KEY_NOT_EXIST")

4. Assert: transaction panics with ERR_KEY_NOT_EXIST.
   Contract is now permanently unable to accept any block at a difficulty boundary.
```
