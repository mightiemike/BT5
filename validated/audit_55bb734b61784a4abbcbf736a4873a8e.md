### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Transaction Merkle Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is marked `#[deprecated]` but remains a fully public, externally callable NEAR method. It omits the coinbase merkle proof check that was introduced in `verify_transaction_inclusion_v2` specifically to prevent the 64-byte transaction forgery attack. Any unprivileged NEAR caller can invoke the deprecated path directly, bypassing the coinbase guard entirely and obtaining a `true` return value for a fabricated transaction inclusion claim.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to close the well-known 64-byte transaction Merkle proof forgery vulnerability (documented at https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring the caller to also supply a coinbase merkle proof that must resolve to the same block merkle root: [1](#0-0) 

After that guard passes, it delegates to the deprecated function: [2](#0-1) 

However, `verify_transaction_inclusion` is still declared `pub` with only a `#[deprecated]` compiler hint and a `#[pause]` attribute — neither of which prevents an external NEAR caller from invoking it at runtime: [3](#0-2) 

The function itself acknowledges the risk in its own doc comment: [4](#0-3) 

The verification logic inside the deprecated function performs only a single merkle root comparison with no coinbase anchor: [5](#0-4) 

`compute_root_from_merkle_proof` is a pure hash-chain computation with no structural constraints on the leaf: [6](#0-5) 

An internal merkle node (the hash of two 32-byte children concatenated into 64 bytes) is a valid input to this function and will produce the correct root if the proof path is chosen accordingly.

---

### Impact Explanation

An attacker who knows the merkle tree of any mainchain block can:

1. Pick any internal node `N` at depth `d` in the tree.
2. Construct a `merkle_proof` of length `d` that walks from `N` up to the merkle root using the sibling hashes at each level.
3. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index` set to the position consistent with the chosen path, and the crafted `merkle_proof`.
4. The function returns `true`, asserting that a transaction with hash `N` is confirmed in that block.

Because `N` is not a real transaction hash, the claimed transaction does not exist on the Bitcoin/Dogecoin/Litecoin/Zcash chain. Any downstream bridge, escrow, or cross-chain settlement contract that trusts the return value of `verify_transaction_inclusion` will accept a fabricated payment or event as confirmed, enabling theft or double-spend of bridged assets.

---

### Likelihood Explanation

The entry path requires no privilege, no staked role, and no special configuration. The function is reachable by any NEAR account. The 64-byte attack is publicly documented, the required merkle sibling hashes are derivable from any block explorer, and the deprecated function's continued public exposure is a direct, one-step bypass of the mitigation that was already implemented in `verify_transaction_inclusion_v2`.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or change it to `pub(crate)` so it is only callable internally (as it is by `verify_transaction_inclusion_v2`). Alternatively, delete the function entirely and inline its logic inside `verify_transaction_inclusion_v2`. If backward compatibility must be preserved for existing callers, add the same coinbase proof requirement to the deprecated function body before delegating to the shared inner logic.

---

### Proof of Concept

Given a mainchain block at height `H` with merkle root `R` and two transactions `T0` (coinbase) and `T1`:

```
merkle root R = hash(T0 || T1)
internal node N = R  (for a 2-tx block, the root IS the only internal node above the leaves)
```

For a block with more transactions, pick any internal node at depth 1:

```
N = hash(T0 || T1)
proof = [hash(T2 || T3)]   # sibling at depth 1
tx_index = 0
```

Call (NEAR CLI):
```bash
near call <contract> verify_transaction_inclusion \
  '{"tx_id": "<N_hex>", "tx_block_blockhash": "<block_hash>", \
    "tx_index": 0, "merkle_proof": ["<sibling_hex>"], "confirmations": 1}' \
  --accountId attacker.near
```

The call returns `true`. No coinbase proof is requested, no AuxPoW or PoW computation is required, and the "transaction" `N` does not exist on the source chain.
