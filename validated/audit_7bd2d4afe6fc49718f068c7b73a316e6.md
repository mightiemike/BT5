### Title
`ENABLE_SECP_OPS` Flag Guard Is Ineffective — 4-Byte Opcode Path Unconditionally Dispatches Secp Operators (`src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` in `src/chia_dialect.rs` contains two separate dispatch paths for `op_secp256k1_verify` and `op_secp256r1_verify`. The 1-byte opcode path (opcodes 64 and 65) is correctly gated behind `ClvmFlags::ENABLE_SECP_OPS`. However, the 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) dispatches the same two functions **unconditionally**, with no check for `ENABLE_SECP_OPS`. Any attacker-controlled CLVM program can invoke full secp signature verification by using the 4-byte opcode encoding, bypassing the flag entirely.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` handles 4-byte opcodes before it handles 1-byte opcodes:

```rust
// lines 157–183
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // no ENABLE_SECP_OPS check
        0x1c3a8f00 => op_secp256r1_verify,   // no ENABLE_SECP_OPS check
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

The 1-byte path, by contrast, correctly checks the flag:

```rust
// lines 248–249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

`ENABLE_SECP_OPS` is defined as a hard-fork activation flag:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

The flag is intended to control whether secp signature verification is available. But because the 4-byte opcode path has no flag check, any CLVM program can invoke `op_secp256k1_verify` or `op_secp256r1_verify` by encoding the operator as the 4-byte atom `0x13d61f00` or `0x1c3a8f00`, regardless of whether `ENABLE_SECP_OPS` is set. The existing tests confirm this: the test at line 1313 of `src/run_program.rs` shows `secp256k1_verify` (4-byte form) succeeds with `ClvmFlags::empty()` — no `ENABLE_SECP_OPS` required.

---

### Impact Explanation

The `ENABLE_SECP_OPS` flag is rendered ineffective as a hard-fork activation guard. Any CLVM program submitted to a node that does not have `ENABLE_SECP_OPS` set (i.e., before the hard-fork activates) can still execute full secp256k1 or secp256r1 signature verification by using the 4-byte opcode encoding. The flag check exists but does not prevent the intended behavior — secp operator execution — because the 4-byte dispatch path is a complete bypass. This is a direct analog to the "Ineffective Whitelist" class: the guard is present for one encoding but absent for the functionally equivalent alternative encoding of the same operator.

Concrete corrupted invariant: a node running without `ENABLE_SECP_OPS` is supposed to reject secp operations, but it will silently execute them when the 4-byte opcode form is used, producing a valid `Reduction(cost, nil)` result instead of an error or unknown-op no-op.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM program bytes submitted to `run_program` can encode the operator as the 4-byte atom `0x13d61f00` or `0x1c3a8f00`. No special privilege, flag, or configuration is required. The bypass is mechanical and requires only knowledge of the opcode encoding, which is documented in the codebase and in the benchmark tool.

---

### Recommendation

Add an `ENABLE_SECP_OPS` check to the 4-byte opcode dispatch path, mirroring the check already present for the 1-byte aliases:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

If the 4-byte form is intentionally always available (hardforked in), the `ENABLE_SECP_OPS` flag documentation and the 1-byte guard should be updated to reflect that the flag only gates the 1-byte aliases, and the flag name should be clarified to avoid implying it gates all secp operations.

---

### Proof of Concept

A CLVM program using the 4-byte opcode `0x13d61f00` (secp256k1_verify) executes successfully with `ClvmFlags::empty()` — no `ENABLE_SECP_OPS` required:

```rust
// Confirmed by existing test in src/run_program.rs lines 1312–1318:
RunProgramTest {
    prg: "(secp256k1_verify (q . <pubkey>) (q . <msg>) (q . <sig>))",
    // secp256k1_verify compiles to 4-byte opcode 0x13d61f00
    args: "()",
    flags: ClvmFlags::empty(),   // ENABLE_SECP_OPS is NOT set
    result: Some("0"),           // succeeds — flag bypass confirmed
    cost: 1300061,
    err: "",
},
```

The 4-byte opcode `0x13d61f00` is dispatched at `src/chia_dialect.rs` lines 175–182 with no flag check, while the 1-byte alias (opcode 64) at line 248 correctly requires `ENABLE_SECP_OPS`. The same `op_secp256k1_verify` function is invoked in both cases. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
