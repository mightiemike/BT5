### Title
`SLOW_MODE_FEE` Constant Is Set 10¹² Times Below Its Documented $1 Value, Breaking Slow-Mode Spam Protection — (`File: core/contracts/common/Constants.sol`)

---

### Summary

`SLOW_MODE_FEE` is declared as `1000000` with a comment of `// $1`, but the protocol's entire internal accounting system uses 18-decimal fixed-point arithmetic where `ONE = 10**18` represents one dollar. The value `1000000` (10⁶) is therefore $0.000000000001, not $1 — a factor of 10¹² below the intended amount. This is the same bug class as the external report: a hardcoded constant whose actual numeric value silently diverges from its documented intent, corrupting a risk-management parameter.

---

### Finding Description

In `core/contracts/common/Constants.sol`, the protocol defines its unit system and all fee constants:

```solidity
int128 constant ONE = 10**18;           // 1 dollar in 18-decimal fixed-point

int128 constant SLOW_MODE_FEE = 1000000; // $1   ← WRONG: 10^6, not 10^18
int128 constant LIQUIDATION_FEE  = 1e18; // $1   ← correct
int128 constant HEALTHCHECK_FEE  = 1e18; // $1   ← correct
```

Every other dollar-denominated fee constant in the file uses `1e18` (= `ONE`) to represent $1. `SLOW_MODE_FEE` alone uses `1000000` (= 10⁶), which in the same 18-decimal system equals $0.000000000001 — effectively zero.

`SLOW_MODE_FEE` is deducted from a submitter's quote balance in `EndpointTx.sol` whenever a slow-mode transaction is submitted. Slow-mode is the protocol's censorship-resistance fallback: when the sequencer ignores a user, the user submits directly on-chain and the contract must process it after `SLOW_MODE_TX_DELAY` (3 days). The $1 fee is the only economic barrier preventing queue flooding.

With the fee at $0.000000000001, an unprivileged user can submit an unbounded number of slow-mode transactions at negligible cost, filling the queue before any censored legitimate user can get their transaction processed.

---

### Impact Explanation

An attacker who wants to prevent a specific user from exercising censorship resistance (e.g., to block a withdrawal while the sequencer is also censoring that user) can flood the slow-mode queue at a cost of ~$0.000000000001 per entry. The censored user's funds remain locked for the duration of the attack because their slow-mode transaction cannot be included. The direct asset impact is the inability to withdraw collateral through the only permissionless path available when the sequencer is uncooperative.

---

### Likelihood Explanation

The entry path requires no privilege: any EOA can call the slow-mode submission function on `Endpoint.sol` with a crafted transaction payload. The cost to sustain the attack is negligible. The attack is most valuable when combined with sequencer censorship of a target, which is a realistic adversarial scenario the slow-mode mechanism was explicitly designed to defend against.

---

### Recommendation

Change `SLOW_MODE_FEE` to use the same unit as every other dollar-denominated constant in the file:

```solidity
// Before (wrong):
int128 constant SLOW_MODE_FEE = 1000000; // $1

// After (correct):
int128 constant SLOW_MODE_FEE = 1e18; // $1
```

This aligns it with `LIQUIDATION_FEE`, `HEALTHCHECK_FEE`, and the `ONE` base unit.

---

### Proof of Concept

1. Observe that `ONE = 10**18` is the protocol's unit for $1.
2. Observe that `LIQUIDATION_FEE = 1e18` and `HEALTHCHECK_FEE = 1e18` both correctly represent $1.
3. Observe that `SLOW_MODE_FEE = 1000000 = 10**6`, which equals `ONE / 10**12` = $0.000000000001.
4. Call the slow-mode submission entry point on `Endpoint.sol` repeatedly from an unprivileged EOA. Each call deducts only $0.000000000001 from the caller's quote balance.
5. For the cost of $1 of quote collateral, an attacker can submit 10¹² slow-mode entries, saturating any practical queue bound and preventing a censored user's legitimate slow-mode transaction from being processed within the 3-day window. [1](#0-0) [2](#0-1) [3](#0-2)
