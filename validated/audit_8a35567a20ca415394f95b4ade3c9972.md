### Title
`DepositAllowlistExtension.beforeAddLiquidity` gates `owner` instead of `sender`, allowing any disallowed depositor to bypass the curated-pool allowlist — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is intended to restrict which addresses may add liquidity to a curated pool. Its `beforeAddLiquidity` hook silently drops the `sender` argument and checks `owner` (the LP-share recipient) instead. Because `sender` is the address that actually pays tokens via the callback, any disallowed address can bypass the gate by naming an allowed address as `owner`.

---

### Finding Description

`BaseMetricExtension` declares `beforeAddLiquidity` with an `onlyPool` modifier and two leading address parameters: `sender` (the caller of `addLiquidity`, who pays tokens) and `owner` (who receives LP shares). [1](#0-0) 

`DepositAllowlistExtension` overrides this function. It drops the `onlyPool` modifier and leaves the first `address` parameter unnamed (ignored). The allowlist check is performed against `owner`, not `sender`: [2](#0-1) 

The admin-facing setter names its parameter `depositor`, making the intent unambiguous — the gate is supposed to restrict who can *put tokens into* the pool (the sender/payer), not who can *receive LP shares*: [3](#0-2) 

This is the direct analog of the `InnerOwnedSet` / `outerSet` bug: in that report the modifier checked against an uninitialized (zero) variable so the guard was permanently mis-bound; here the hook checks against the wrong actor so the guard is permanently bypassable for any caller who knows an allowed `owner` address.

---

### Impact Explanation

A disallowed address (Bob) can call `pool.addLiquidity(alice, salt, deltas, ...)` where Alice is on the allowlist. The extension evaluates `allowedDepositor[pool][alice]` → `true` and returns success. The pool then invokes the liquidity-callback on `msg.sender` (Bob), Bob pays the tokens, and Alice receives LP shares she never requested. Bob has:

- Deposited into a curated pool without authorization, violating the pool admin's access-control policy.
- Gained the ability to manipulate per-bin liquidity distribution (choose exact bins, share amounts) on a pool that was supposed to be restricted — directly affecting the oracle-anchored price curve seen by subsequent swaps and harming existing LPs whose positions are now diluted or repriced.
- Forced LP shares onto Alice, who may suffer loss if the pool is in a deteriorating state.

This satisfies the **admin-boundary break** impact gate: an unprivileged path bypasses a factory-configured pool guard.

---

### Likelihood Explanation

Exploitation requires only knowing one address that the pool admin has allowlisted (trivially readable from on-chain events or `allowedDepositor` storage). No special role, flash loan, or complex setup is needed. Any public caller can execute this in a single transaction.

---

### Recommendation

Replace the unnamed first parameter with `sender` and gate on it instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    onlyPool          // re-apply the base modifier that the override dropped
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Apply the same correction to `SwapAllowlistExtension` if a parallel review reveals the same pattern there.

### Proof of Concept

```
Setup
─────
1. Deploy DepositAllowlistExtension(factory).
2. Pool admin creates a curated pool with this extension in beforeAddLiquidity order.
3. Pool admin calls setAllowedToDeposit(pool, alice, true).
   → allowedDepositor[pool][alice] = true
   → Bob is NOT in the allowlist.

Attack
──────
4. Bob (disallowed) deploys AttackerContract that implements the addLiquidity callback.
5. AttackerContract calls:
       pool.addLiquidity(
           alice,          // owner  ← allowlist check passes (alice is allowed)
           salt,
           deltas,         // Bob chooses exact bins and share amounts
           callbackData,
           extensionData
       )
6. Extension evaluates: allowedDepositor[pool][alice] == true → no revert.
7. Pool calls AttackerContract's callback; AttackerContract transfers tokens.
8. Alice receives LP shares; Bob has manipulated pool liquidity without authorization.

Result
──────
- Bob bypassed the deposit allowlist.
- Bob chose arbitrary bin positions, altering the pool's liquidity curve.
- Alice holds unwanted LP exposure.
``` [2](#0-1) [4](#0-3)
