### Title
`DISABLE_OP` Flag in `MEMPOOL_MODE` Blocks `op_modpow` (Opcode 60) in Mempool Validation, Causing Consensus/Mempool Divergence — (`File: src/chia_dialect.rs`)

---

### Summary

`MEMPOOL_MODE` includes the undocumented `DISABLE_OP` flag, which unconditionally disables `op_modpow` (opcode 60) during mempool validation. Because `op_modpow` is a fully hardforked operator available in consensus mode without any flag guard, any CLVM program using it is rejected by the mempool but accepted by consensus. This is a direct flag/operator wiring error: the mempool applies an overly restrictive check that blocks a legitimate operation that the consensus layer permits.

---

### Finding Description

In `src/chia_dialect.rs`, the `MEMPOOL_MODE` constant is defined as:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)       // ← undocumented flag
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [1](#0-0) 

The `DISABLE_OP` flag (value `0x200`) has no documentation comment, unlike every other flag in the same `bitflags!` block. Its sole effect is in the `op()` dispatch table:

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
``` [2](#0-1) 

Opcode 60 (`op_modpow`) sits in the **main hardforked operator table** — it requires no `ENABLE_*` flag to be set, unlike softfork-gated operators such as `op_keccak256` (opcode 62, requires `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`) or `op_sha256_tree` (opcode 63, requires `ENABLE_SHA256_TREE`). In consensus mode (no `DISABLE_OP`), `op_modpow` executes normally. In mempool mode (`DISABLE_OP` set), it returns `EvalErr::Unimplemented`, causing the program to be rejected. [3](#0-2) 

The `op()` function receives the `extension: OperatorSet` argument and merges additional flags for softfork contexts, but **never clears `DISABLE_OP`** regardless of the active extension:

```rust
let flags = self.flags
    | match extension {
        OperatorSet::Default => ClvmFlags::empty(),
        OperatorSet::Bls    => ClvmFlags::empty(),
        OperatorSet::Keccak => ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD,
    };
``` [4](#0-3) 

So even inside a softfork guard, `op_modpow` remains disabled in mempool mode.

---

### Impact Explanation

Any attacker-controlled CLVM byte sequence that invokes opcode 60 (`op_modpow`) will be:

- **Rejected** by any node running mempool validation (`MEMPOOL_MODE` / `DISABLE_OP` set).
- **Accepted** by any node running consensus validation (no `DISABLE_OP`).

This is a **consensus/mempool divergence**. A farmer can include a block containing a spend bundle that uses `op_modpow` directly (bypassing the mempool), and all full nodes will accept it at the consensus layer. However, the same spend bundle cannot propagate through the peer-to-peer mempool because every node's mempool rejects it. This means:

1. Legitimate users whose CLVM programs use `op_modpow` cannot get their transactions into the mempool and thus cannot get them confirmed through normal network propagation.
2. A farmer who constructs such a block directly will produce a valid block that other nodes accept, but the transactions in it were never visible to the mempool — breaking the assumption that mempool-accepted transactions are a superset of what consensus accepts.

---

### Likelihood Explanation

The trigger is purely attacker-controlled CLVM bytes: any program that encodes opcode byte `0x3c` (decimal 60) as an operator call. No special privileges, no social engineering, and no dependency on external configuration are required. Any caller that passes a program through the mempool path (i.e., uses `MEMPOOL_MODE` or any `ClvmFlags` set that includes `DISABLE_OP`) will hit this divergence. The likelihood is **moderate to high** for any deployment that uses `op_modpow` in production spend bundles.

---

### Recommendation

Remove `DISABLE_OP` from `MEMPOOL_MODE`, analogous to the Reserve Protocol fix of changing `notPausedOrFrozen` to `notFrozen`:

```rust
// Before (overly restrictive):
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)      // ← remove this
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);

// After:
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

If `DISABLE_OP` is intentionally gating a not-yet-activated hard fork, it must be documented with a comment explaining the activation condition, and the operator dispatch must be restructured to use an `ENABLE_MODPOW` flag (like `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`) rather than a `DISABLE_*` flag included in `MEMPOOL_MODE`.

---

### Proof of Concept

1. Construct a CLVM program that calls `op_modpow`: e.g., `(modpow (q . 2) (q . 10) (q . 1000))` — opcode 60 with three integer arguments.
2. Run it under consensus mode (`ClvmFlags::empty()`): execution succeeds, returns `512`.
3. Run the same program under `MEMPOOL_MODE`: execution fails with `EvalErr::Unimplemented`.
4. A farmer who includes this program in a block will have the block accepted by all consensus nodes, but no mempool node would have forwarded the spend bundle — demonstrating the divergence.

The exact broken invariant: `MEMPOOL_MODE` is documented as "the stricter mode" for mempool validation, but the invariant that *every program accepted by consensus is also accepted by the mempool* is violated for any program invoking opcode 60. [1](#0-0) [2](#0-1)
