### Title
Stale `_cached_sha256_treehash` Trusted Without Verification Causes `Program.tree_hash()` to Return Wrong Puzzle Hash — (`File: wheel/python/clvm_rs/program.py`, `wheel/python/clvm_rs/tree_hash.py`)

---

### Summary

`Program.wrap()` blindly copies `_cached_sha256_treehash` from any caller-supplied `CLVMStorage` object without verifying it against the object's actual content. `Program.tree_hash()` then returns that cached value unconditionally. A caller-controlled Python object with a pre-set wrong hash causes every downstream puzzle-hash comparison to silently use the wrong value.

---

### Finding Description

`Program.wrap()` snapshots the `_cached_sha256_treehash` attribute from the wrapped object at construction time:

```python
o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)
``` [1](#0-0) 

`Program.tree_hash()` then returns that value without ever recomputing it:

```python
def tree_hash(self) -> bytes:
    if self._cached_sha256_treehash is None:
        self._cached_sha256_treehash = sha256_treehash(self._unwrapped)
    return self._cached_sha256_treehash
``` [2](#0-1) 

The same trust is present in `Treehasher.sha256_treehash()`, which short-circuits on any non-`None` `_cached_sha256_treehash` it finds on a sub-object, without checking whether that value actually matches the object's `atom` or `pair`:

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    self.cache_hits += 1
    hash_stack.append(r)
    return
``` [3](#0-2) 

The `CLVMStorage` protocol explicitly documents `_cached_sha256_treehash` as an optional, caller-settable field:

```python
# `_cached_sha256_treehash: Optional[bytes]` is used by `sha256_treehash`
``` [4](#0-3) 

Because `CLVMStorage` is a structural protocol (not an abstract base class with enforcement), any Python object with the right attribute names satisfies it. There is no mechanism to validate that a pre-set `_cached_sha256_treehash` is consistent with the object's actual `atom`/`pair` content.

Additionally, `Treehasher` writes the computed hash back to sub-objects via `setattr`:

```python
try:
    setattr(obj, "_cached_sha256_treehash", r)
except AttributeError:
    pass
``` [5](#0-4) 

If a `CLVMStorage` object's `pair` property is non-deterministic or mutated after the first hash computation, the stale hash written by `Treehasher` will be reused on all subsequent calls, exactly mirroring the rebasing-token scenario in the external report.

---

### Impact Explanation

`Program.__eq__()` delegates entirely to `tree_hash()`:

```python
def __eq__(self, other) -> bool:
    ...
    return self.tree_hash() == other_obj.tree_hash()
``` [6](#0-5) 

A wrong `tree_hash()` result causes wrong equality comparisons. In the Chia wallet layer this means:

- Puzzle-hash validation against a coin's `puzzle_hash` field can return the wrong boolean.
- Spend-bundle construction that selects coins by puzzle hash can select the wrong coins.
- Any code path that calls `Program.wrap(external_obj)` and then compares or hashes the result operates on a corrupted hash for the lifetime of that `Program` object.

The corrupted value is sticky: once `_cached_sha256_treehash` is set on a `Program` instance it is never recomputed, so the error propagates to every caller that holds a reference to that object.

---

### Likelihood Explanation

The `CLVMStorage` protocol is intentionally open: the `clvm_tree_to_lazy_node` public API accepts any Python object satisfying the protocol, and `Program.wrap()` is called on the result of every deserialization path (`deser_auto`, `deser_legacy`, `deser_backrefs`, `deser_2026`). Any integration that passes an externally-sourced `CLVMStorage` object — including objects returned by third-party libraries or constructed from untrusted network data — can trigger this path. The `GeneratedTree` test fixture already demonstrates that non-trivial mutable `CLVMStorage` implementations are expected and tested. Likelihood is **Medium**: it requires attacker influence over a Python object passed to the API, which is realistic in wallet or RPC contexts.

---

### Recommendation

1. **Validate on wrap**: In `Program.wrap()`, do not copy `_cached_sha256_treehash` from the underlying object. Compute it lazily from the actual content on first access, or at minimum verify the copied value matches a fresh computation before trusting it.

2. **Document the invariant**: The `CLVMStorage` protocol documentation should state explicitly that `_cached_sha256_treehash`, if set, must equal the SHA-256 tree hash of the object's actual content. Implementations that violate this invariant produce undefined behavior.

3. **Restrict write-back**: `Treehasher.sha256_treehash()` should only write `_cached_sha256_treehash` back to objects it has fully traversed and verified, not to arbitrary sub-objects whose mutability is unknown.

### Proof of Concept

```python
from clvm_rs.clvm_storage import CLVMStorage
from clvm_rs.program import Program

# Craft a CLVMStorage atom whose cached hash is wrong
class MaliciousAtom(CLVMStorage):
    atom = b"real_content"
    pair = None
    # Pre-set a hash that does NOT correspond to b"real_content"
    _cached_sha256_treehash = b"\xff" * 32  # arbitrary wrong hash

obj = MaliciousAtom()
p = Program.wrap(obj)

# Program.tree_hash() returns the wrong pre-set hash without recomputing
assert p.tree_hash() == b"\xff" * 32  # wrong: should be sha256(b"\x01" + b"real_content")

# Program.__eq__ now uses the wrong hash
real = Program.new_atom(b"real_content")
# real.tree_hash() is correct; p.tree_hash() is wrong → equality is False
# even though both represent the same atom
assert p != real  # wrong result: should be equal
```

The same effect is reachable through `Treehasher` on a mutable `pair` property: compute the hash once (which writes the stale value back), mutate the object's children, then call `sha256_treehash` again — the stale cached value is returned instead of the hash of the new children.
