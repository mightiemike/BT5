### Title
Hardcoded `amount=1` in `getCoreRisk()` causes inflated spread health credit for short-spot/long-perp positions — (`core/contracts/BaseEngine.sol`)

### Summary
`getCoreRisk()` passes the literal `1` instead of the subaccount's actual `amount` to `RiskHelper._getWeightX18()`. Because `1 >= 0`, the function always returns the `longWeight` regardless of whether the position is short. This corrupts both `existingWeight` and `spreadWeight` in `Clearinghouse.getHealth()`, causing the spread health adjustment to be a large positive benefit (+1600) instead of the correct small penalty (-200) for a short-spot/long-perp spread.

### Finding Description

`getCoreRisk()` in `BaseEngine.sol` reads the actual `amount` from storage but then discards it when computing the weight:

```solidity
// BaseEngine.sol line 185, 190
(int128 amount, ) = _getBalance(productId, subaccount);
return IProductEngine.CoreRisk(
    amount,
    risk.priceX18,
    RiskHelper._getWeightX18(risk, 1, healthType)  // ← hardcoded 1, not amount
);
``` [1](#0-0) 

`_getWeightX18` branches on the sign of `amount`:

```solidity
// RiskHelper.sol lines 44-52
if (amount >= 0) {          // 1 >= 0 is always true
    weight = ... longWeight ...
} else {
    weight = ... shortWeight ...
}
``` [2](#0-1) 

So `CoreRisk.longWeight` is always the long weight. In `Clearinghouse.getHealth()`, this corrupted value feeds into two places:

1. `existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2` — uses `0.9` instead of `1.0` for a short-spot/long-perp spread.
2. `_getSpreadWeightX18(perpCoreRisk, spotCoreRisk, healthType)` — the `else` branch uses `spotCoreRisk.longWeight = 0.9` instead of `1.1`, producing `spreadWeight = 0.98` instead of `0.99`. [3](#0-2) 

The final adjustment `basisAmount * (spotPrice + perpPrice) * (spreadWeight - existingWeight)` becomes `+1600` instead of `-200` — an inflation of **+1800 per unit of notional**.

Note that `_calculateProductHealth` (used for the baseline health of individual positions) correctly passes `amount` to `_getWeightX18`, so the baseline health is computed correctly. Only the spread adjustment is wrong. [4](#0-3) 

### Impact Explanation

A trader holding a short-spot/long-perp spread receives a large spurious health bonus from the spread adjustment. This allows them to:
- Open a spread position that is significantly larger than the protocol's risk parameters permit.
- Withdraw collateral that should be locked against the spread risk.
- If the spread moves adversely, the position becomes insolvent before the protocol can liquidate it, creating bad debt absorbed by the insurance fund.

The same bug also inflates health for long-spot/short-perp spreads (the perp's short weight is similarly replaced by its long weight).

### Likelihood Explanation

The path is fully permissionless. Any trader can open a spread position (short spot + long perp on the same underlying) through normal order submission. The health check `getHealth()` is called on every withdrawal, order placement, and liquidation check, so the inflated credit is active at all times. No special privileges or sequencer compromise are required.

### Recommendation

Replace the hardcoded `1` with the actual `amount` in `getCoreRisk()`:

```solidity
// BaseEngine.sol
return IProductEngine.CoreRisk(
    amount,
    risk.priceX18,
    RiskHelper._getWeightX18(risk, amount, healthType)  // use actual amount
);
```

This mirrors the correct pattern already used in `_calculateProductHealth()` at line 167. [5](#0-4) 

### Proof of Concept

```
Setup:
  spotLongWeight = 0.9e18, spotShortWeight = 1.1e18
  perpLongWeight = 0.9e18, perpShortWeight = 1.1e18
  price = 1000e18 for both

Trader opens:
  spot position: amount = -10e18  (short)
  perp position: amount = +10e18  (long)

Call getHealth(subaccount, INITIAL):

  Baseline (correct in both versions):
    spot: -10 * 1.1 * 1000 = -11000
    perp: +10 * 0.9 * 1000 =  +9000
    total baseline = -2000

  Spread adjustment (BUGGY):
    spotCoreRisk.longWeight = 0.9  (wrong, should be 1.1)
    existingWeight = (0.9 + 0.9) / 2 = 0.9
    spreadWeight = 1 - (1 - 0.9)/5 = 0.98
    basisAmount = 10
    adjustment = 10 * 2000 * (0.98 - 0.9) = +1600

  Total health (buggy) = -2000 + 1600 = -400

  Spread adjustment (CORRECT):
    spotCoreRisk.longWeight = 1.1
    existingWeight = (1.1 + 0.9) / 2 = 1.0
    spreadWeight = min(1 - (1-1.1)/5, 0.99) = 0.99
    adjustment = 10 * 2000 * (0.99 - 1.0) = -200

  Total health (correct) = -2000 + (-200) = -2200

Assert: buggy_health - correct_health = 1800 ≠ 0  → vulnerability confirmed
```
