### Title
Consensus/Mempool Split via Opcode 62 Inside Softfork Extension 0 Guard — (`src/chia_dialect.rs`)

---

### Summary

A program using `softfork` with `extension=0` (`OperatorSet::Bls`) that invokes opcode 62 (keccak256) inside the guard will **succeed in consensus mode** (opcode 62 treated as unknown, returns nil via `op_unknown`) but **fail in mempool mode** (`NO_UNKNOWN_OPS` rejects it). This is a concrete, locally testable consensus/mempool split.

---

### Finding Description

The execution path is fully traceable through production code:

**Step 1 — Extension 0 is accepted as a known extension.**

`softfork_extension(0)` returns `OperatorSet::Bls`, which is not `OperatorSet::Default`, so `parse_softfork_arguments` returns `Ok` and the guard is entered normally. [1](#0-0) [2](#0-1) 

**Step 2 — `OperatorSet::Bls` adds no flags.**

Inside `ChiaDialect::op`, the `extension` parameter is matched and `OperatorSet::Bls` contributes `ClvmFlags::empty()` — no additional capability flags are OR'd in. [3](#0-2) 

**Step 3 — Opcode 62 requires `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`.**

The match arm for opcode 62 is guarded by `flags.contains(ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD)`. Since `OperatorSet::Bls` adds no flags, this guard fails and execution falls through to `unknown_operator`. [4](#0-3) 

**Step 4 — `unknown_operator` diverges on `NO_UNKNOWN_OPS`.**

```
fn unknown_operator(..., flags: ClvmFlags, ...) -> Response {
    if flags.contains(ClvmFlags::NO_UNKNOWN_OPS) {
        Err(EvalErr::Unimplemented(o))?   // mempool: hard error
    } else {
        op_unknown(allocator, o, args, max_cost)  // consensus: nil + cost
    }
}
``` [5](#0-4) 

**Step 5 — No special error recovery exists for inner-program failures.**

The only special error handling in `apply_op` for the softfork path covers `parse_softfork_arguments` failures (lines 400–413). Errors that arise during inner-program execution propagate unconditionally up the call stack. There is no catch-and-nil path for inner-program errors inside a known extension guard. [6](#0-5) 

**Step 6 — `current_extensions` is correctly set to `OperatorSet::Bls` inside the guard.** [7](#0-6) 

---

### Impact Explanation

| Mode | Opcode 62 inside ext=0 guard | Outcome |
|---|---|---|
| Consensus (`ClvmFlags::empty()`) | `op_unknown` → nil + cost | Guard exits, program returns nil — **accepted** |
| Mempool (`NO_UNKNOWN_OPS`) | `Err(Unimplemented)` | Program fails — **rejected** |

A transaction containing a `softfork` with `extension=0` and opcode 62 in the inner program is valid on-chain but rejected by every mempool node. This is a direct consensus/mempool split: the transaction can be mined into a block but will never propagate through the P2P mempool.

---

### Likelihood Explanation

The path requires only attacker-controlled CLVM data submitted through the normal transaction API. No special privileges, compromised nodes, or social engineering are needed. The cost value in the softfork invocation must be set correctly (matching `op_unknown`'s cost for a 1-byte operator), which is fully computable offline. The split is deterministic and reproducible.

---

### Recommendation

Inside a **known** softfork extension (extension 0 or 1), unknown opcodes should be treated consistently in both modes. The simplest fix is to OR `ClvmFlags::NO_UNKNOWN_OPS` into `flags` when `extension` is `OperatorSet::Bls` or `OperatorSet::Keccak`, so that unknown opcodes inside a known guard are rejected in both consensus and mempool mode rather than silently returning nil in consensus mode. Alternatively, the `apply_op` softfork path can be made to propagate inner-program errors uniformly regardless of `allow_unknown_ops`.
