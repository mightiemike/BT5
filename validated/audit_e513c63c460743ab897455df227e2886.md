### Title
Linked Signer Can Self-Escalate by Overwriting Its Own Authorization via `LinkSigner` — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in `processTransactionImpl` is validated with `allowLinkedSigner = true`, meaning the **current linked signer** (a session key) can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. The slow-mode path correctly restricts this to the subaccount owner, but the fast-mode (sequencer) path does not, creating an unenforced ownership invariant directly analogous to the Neptune Mutual staking bypass.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` passes the current linked signer to the `Verifier`, which accepts a signature from **either** the subaccount owner **or** the linked signer: [2](#0-1) 

This means the linked signer can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address it chooses.

By contrast, the slow-mode path for the same transaction type enforces that `msg.sender` must be the subaccount owner via `validateSender`: [3](#0-2) 

The `linkedSigners` mapping is the sole authorization record for session keys: [4](#0-3) 

There is no secondary check that prevents a linked signer from being the signer of a `LinkSigner` mutation. The invariant — *only the subaccount owner may change the linked signer* — is enforced on the slow path but silently absent on the fast path.

---

### Impact Explanation

A compromised linked signer can:

1. Sign a `LinkSigner` transaction naming an attacker-controlled address as the new signer.
2. The sequencer processes it; `linkedSigners[subaccount]` is overwritten.
3. The attacker's new key is now the authorized linked signer.
4. The attacker can sign `WithdrawCollateral`, `MatchOrders`, `LiquidateSubaccount`, `TransferQuote`, and `MintNlp`/`BurnNlp` transactions — all of which accept linked signer signatures (`allowLinkedSigner = true`). [5](#0-4) 

The corrupted state is `linkedSigners[subaccount]`, and the downstream asset delta is the full collateral balance of the subaccount, which can be drained via a signed `WithdrawCollateral` transaction.

---

### Likelihood Explanation

Linked signers are session keys used in automated trading bots, API integrations, and front-end sessions. Private key leakage for session keys is a realistic and common threat (e.g., leaked `.env` files, compromised CI/CD pipelines, browser storage exfiltration). Once a session key is leaked, the attacker has a single-transaction path to permanent account takeover before the owner can react.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` branch in `processTransactionImpl`. Only the subaccount owner (the address encoded in the `bytes32` sender) should be permitted to change the linked signer:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // linked signer must NOT be able to change the linked signer
);
```

This aligns the fast-mode path with the slow-mode path, which already enforces owner-only authorization via `validateSender`.

---

### Proof of Concept

1. Alice links session key `S` to her subaccount `A` via a `LinkSigner` transaction.
2. Attacker obtains `S`'s private key (e.g., leaked `.env`).
3. Attacker constructs a `SignedLinkSigner` transaction: `sender = A`, `signer = attacker_address`, signed by `S`.
4. Attacker submits this to the sequencer (or via `submitSlowModeTransaction` — but slow mode requires `msg.sender == owner`, so the fast path is the attack vector).
5. Sequencer calls `processTransactionImpl`; `validateSignedTx(..., true)` accepts `S`'s signature.
6. `linkedSigners[A]` is overwritten with `attacker_address`.
7. Attacker signs a `WithdrawCollateralV2` transaction with `sendTo = address(0)` (which permits linked signer, per line 447) and drains Alice's collateral. [6](#0-5)
