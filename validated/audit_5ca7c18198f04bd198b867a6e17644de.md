### Title
Stale `versioned_constants` in `PyBlockExecutor` after `set_block_casm_hash_v1_declares_in_vc` / `set_enable_casm_hash_migration_in_vc` called post-`setup_block_execution` causes wrong compiled class hash accepted into block state — (`File: crates/native_blockifier/src/py_block_executor.rs`)

---

### Summary

`PyBlockExecutor` holds `self.versioned_constants` as a mutable field. `setup_block_execution` **clones** that field into a `BlockContext` that is then frozen inside the running `TransactionExecutor`. Two Python-callable setters — `set_block_casm_hash_v1_declares_in_vc` and `set_enable_casm_hash_migration_in_vc` — mutate only `self.versioned_constants`, leaving the already-created executor's `BlockContext` with the old values. Any transaction executed after the setter call but before the next `setup_block_execution` runs under the stale flag, producing wrong compiled-class-hash validation and wrong migration output in the committed state diff.

---

### Finding Description

`PyBlockExecutor` is the Rust/Python bridge that the Python sequencer uses to build blocks. Its struct holds two independent copies of the relevant flags:

```
PyBlockExecutor {
    versioned_constants: VersionedConstants,   // ← setters write here
    tx_executor: Option<TransactionExecutor>,  // ← BlockContext cloned at setup time
}
```

**Step 1 — clone at setup time.**
`setup_block_execution` creates a `BlockContext` by cloning `self.versioned_constants` at that instant and passes it into `TransactionExecutor::pre_process_and_create`. After this point the executor owns an `Arc<BlockContext>` that is immutable for the lifetime of the block.

```rust
// crates/native_blockifier/src/py_block_executor.rs  lines 137-153
let block_context = BlockContext::new(
    next_block_info.try_into()?,
    self.chain_info.clone(),
    self.versioned_constants.clone(),   // ← snapshot taken here
    self.bouncer_config.clone(),
);
self.tx_executor = Some(TransactionExecutor::pre_process_and_create(
    state_reader, block_context, ...
)?);
```

**Step 2 — setters mutate only the outer copy.**
Both setters are `#[pymethods]` and write directly to `self.versioned_constants`, never touching `self.tx_executor`:

```rust
// lines 341-349
pub fn set_enable_casm_hash_migration_in_vc(&mut self, enable_casm_hash_migration: bool) {
    self.versioned_constants.enable_casm_hash_migration = enable_casm_hash_migration;
}
pub fn set_block_casm_hash_v1_declares_in_vc(&mut self, block_casm_hash_v1_declares: bool) {
    self.versioned_constants.block_casm_hash_v1_declares = block_casm_hash_v1_declares;
}
```

**Step 3 — execution reads the stale copy.**
Every transaction executed in the block reads `block_context.versioned_constants`, which is the snapshot from Step 1:

*Declare-hash gate* — `block_casm_hash_v1_declares` controls whether V3 declare transactions carrying a Poseidon-based (V1) compiled class hash are rejected:

```rust
// crates/blockifier/src/transaction/transactions.rs  lines 184-188
if context.tx_context.block_context.versioned_constants.block_casm_hash_v1_declares
    && self.version() >= TransactionVersion::THREE
{
    self.check_compile_class_hash_v2_declaration()?
}
```

*Migration gate* — `enable_casm_hash_migration` controls whether executed classes are migrated from V1 to V2 compiled class hashes at block finalization:

```rust
// crates/blockifier/src/blockifier/transaction_executor.rs  lines 267-272
if !block_context.versioned_constants.enable_casm_hash_migration {
    assert!(class_hashes_to_migrate.is_empty(), ...);
}
block_state.set_compiled_class_hash_migration(&class_hashes_to_migrate)?;
```

**Divergence scenario (concrete):**

| Call order | `self.versioned_constants` | executor's `BlockContext` |
|---|---|---|
| `create(...)` | `block_casm_hash_v1_declares = false` | — |
| `setup_block_execution(...)` | `false` | **snapshot: `false`** |
| `set_block_casm_hash_v1_declares_in_vc(true)` | `true` | still `false` ← stale |
| `execute(declare_v3_with_v1_hash)` | — | gate not triggered → **accepted** |

The V3 declare transaction carrying a V1 (Poseidon) compiled class hash is accepted into the block. The state diff records the V1 hash as the canonical compiled class hash for that class. The OS/prover expects V2 (Blake) hashes for blocks where `block_casm_hash_v1_declares = true`.

The reverse divergence (`enable_casm_hash_migration` set to `true` after setup but executor has `false`) causes the migration step in `finalize_block` to be skipped, so classes that should have their compiled class hash upgraded in the state diff are not upgraded.

---

### Impact Explanation

The direct output is a wrong compiled class hash written into the committed state diff — either a V1 hash accepted where only V2 is valid, or a missing V1→V2 migration entry. This maps to:

- **Wrong state / class hash / storage value from blockifier execution logic** — the `class_hash_to_compiled_class_hash` map in the state diff contains the wrong hash.
- **Wrong compiled class / CASM artifact selected for execution** — downstream calls to the declared class will resolve to the wrong compiled artifact.
- **Wrong receipt / revert result** — if `block_casm_hash_v1_declares` is stale-false when it should be true, a transaction that must revert (hash mismatch) instead succeeds, producing a wrong execution receipt.

---

### Likelihood Explanation

The Python sequencer calls `set_block_casm_hash_v1_declares_in_vc` and `set_enable_casm_hash_migration_in_vc` during version-transition blocks (e.g., the upgrade from `V0_14_0` → `V0_14_1` where both flags flip from `false` to `true`). The API provides no ordering guarantee and no guard that prevents calling the setter after `setup_block_execution`. The window is every block that straddles a version boundary where the Python layer calls the setter mid-block. An external user who submits a V3 declare transaction with a V1 CASM hash during that window is the unprivileged trigger.

---

### Recommendation

1. **Remove the post-setup setters.** Both flags are version-gated constants; they should be fixed at `setup_block_execution` time from the block's `StarknetVersion`, not mutated afterward.
2. **If runtime override is required**, propagate the change into the live executor:
   ```rust
   pub fn set_block_casm_hash_v1_declares_in_vc(&mut self, v: bool) {
       self.versioned_constants.block_casm_hash_v1_declares = v;
       if let Some(exec) = &mut self.tx_executor {
           Arc::make_mut(&mut exec.block_context).versioned_constants
               .block_casm_hash_v1_declares = v;
       }
   }
   ```
3. **Add an assertion** in `setup_block_execution` that `tx_executor` is `None` before proceeding, and in the setters that `tx_executor` is `None` (i.e., no block is in progress).

### Proof of Concept

```python
# Python sequencer pseudocode illustrating the stale-flag window

executor = PyBlockExecutor.create(
    ...,
    py_versioned_constants_overrides=overrides_for_v0_14_0,  # block_casm_hash_v1_declares=False
)

# Block N: version transition block
executor.setup_block_execution(block_info_N, old_hash)
# BlockContext snapshot: block_casm_hash_v1_declares = False

# Python detects version upgrade, tries to enable the gate
executor.set_block_casm_hash_v1_declares_in_vc(True)
# self.versioned_constants updated, but executor's BlockContext still has False

# Attacker submits V3 declare with V1 (Poseidon) compiled class hash
result = executor.execute(declare_v3_tx_with_v1_casm_hash, class_info)
# check_compile_class_hash_v2_declaration() is NOT called (stale False)
# Transaction accepted → V1 hash written to state diff

summary = executor.finalize()
# summary.state_diff.class_hash_to_compiled_class_hash contains V1 hash
# OS/prover rejects the block or executes the class with wrong bytecode
```
