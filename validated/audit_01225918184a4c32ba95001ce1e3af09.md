### Title
Slow-Mode `WithdrawCollateral` Fee Permanently Lost When Subaccount Health Drops During Cooldown — (`File: core/contracts/EndpointTx.sol`, `core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

When a user submits a `WithdrawCollateral` slow-mode transaction, a `SLOW_MODE_FEE` ($1 USDC) is charged from their wallet at submission time. The transaction is then queued for 3 days (`SLOW_MODE_TX_DELAY`). If the user's subaccount health drops below zero during that cooldown — due to adverse price movement or position changes — the withdrawal reverts silently on execution, the queued transaction is permanently deleted, and the fee is never refunded. There is no mechanism to cancel a queued slow-mode transaction.

---

### Finding Description

**Step 1 — Fee charged at submission time.**

In `EndpointTx.submitSlowModeTransactionImpl`, for any transaction type that is not owner-only or `DepositCollateral`, the fee is taken immediately from the caller's wallet:

```solidity
// EndpointTx.sol lines 369-371
} else {
    chargeSlowModeFee(_getQuote(), sender);
    slowModeFees += SLOW_MODE_FEE;
}
```

`chargeSlowModeFee` (in `EndpointStorage.sol`) performs a real ERC-20 `safeTransferFrom` of `SLOW_MODE_FEE = 1_000_000` (i.e., $1 in 6-decimal USDC) from the user's wallet to the contract.

**Step 2 — Transaction queued with 3-day delay.**

```solidity
// EndpointTx.sol lines 376-380
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // 3 days
    sender: sender,
    tx: transaction
});
```

**Step 3 — Execution deletes the tx before processing, catches failures silently.**

In `Endpoint._executeSlowModeTransaction`:

```solidity
// Endpoint.sol lines 193-194
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];
```

The transaction is deleted from the queue **before** execution. Then:

```solidity
// Endpoint.sol lines 207-227
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed
}
```

The comment `// try return funds now removed` is explicit evidence that a refund path existed and was deliberately removed. Any revert inside `processSlowModeTransaction` is silently swallowed.

**Step 4 — Health check inside `withdrawCollateral` can revert.**

`processSlowModeTransactionImpl` routes `WithdrawCollateral` to:

```solidity
// EndpointTx.sol lines 217-229
clearinghouse.withdrawCollateral(
    txn.sender, txn.productId, txn.amount, address(0), nSubmissions
);
```

Inside `Clearinghouse.withdrawCollateral`:

```solidity
// Clearinghouse.sol line 419
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
```

If the subaccount's health has fallen below zero during the 3-day cooldown (due to price moves, funding payments, or new positions), this `require` reverts. The catch block in `_executeSlowModeTransaction` absorbs the revert, the tx is already deleted, and the `SLOW_MODE_FEE` is permanently retained by the protocol.

**Step 5 — No cancellation path exists.**

There is no `cancelSlowModeTransaction` or equivalent function anywhere in `Endpoint.sol`, `EndpointTx.sol`, or `EndpointStorage.sol`. The queue is strictly FIFO and the user has no recourse once the fee is paid.

---

### Impact Explanation

A user who submits a `WithdrawCollateral` slow-mode transaction while healthy loses their `SLOW_MODE_FEE` ($1 USDC) if their subaccount health deteriorates during the mandatory 3-day cooldown. The withdrawal silently fails, the queued entry is deleted, and the fee is not refunded. The user must pay another `SLOW_MODE_FEE` to re-submit, and the same failure can recur. The corrupted state is the user's USDC balance: `SLOW_MODE_FEE` is permanently transferred to the protocol with no corresponding service rendered.

**Impact: Medium** — direct asset loss per occurrence is bounded at $1, but the structural trap is repeatable and the user has no escape path.

---

### Likelihood Explanation

Over a 3-day window, market volatility, funding rate payments, or the user opening additional positions can easily push a subaccount's initial health below zero. Any user who submits a slow-mode withdrawal and then trades or experiences adverse price movement is exposed. The slow-mode path is the only permissionless withdrawal path available to users (the fast path requires sequencer cooperation), making this a realistic scenario for any user attempting a self-custodied exit.

**Likelihood: Medium**

---

### Recommendation

1. Restore a refund path in the `catch` block of `_executeSlowModeTransaction` for `WithdrawCollateral` failures caused by health checks (the removed `// try return funds` logic).
2. Alternatively, add a `cancelSlowModeTransaction(uint64 idx)` function that allows the original sender to dequeue their own pending transaction and receive a fee refund, analogous to the M-08 recommendation of allowing users to dequeue escrowed stables when the blocking condition is active.

---

### Proof of Concept

1. Alice has a healthy subaccount and calls `Endpoint.submitSlowModeTransaction(withdrawCollateralTx)`.
2. `EndpointTx.submitSlowModeTransactionImpl` charges `SLOW_MODE_FEE = $1` from Alice's wallet via `chargeSlowModeFee`. The tx is queued with `executableAt = now + 3 days`.
3. Over the next 3 days, Alice's collateral asset price drops, pushing `getHealth(alice, INITIAL) < 0`.
4. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`.
5. `_executeSlowModeTransaction` deletes Alice's tx from the queue, then calls `processSlowModeTransaction` inside a `try/catch`.
6. Inside, `Clearinghouse.withdrawCollateral` hits `require(getHealth(alice, INITIAL) >= 0)` → reverts.
7. The `catch` block fires; the comment `// try return funds now removed` confirms no refund occurs.
8. Alice's tx is gone from the queue, her $1 fee is retained by the protocol, and her collateral was never withdrawn. She must pay another $1 to retry, with no guarantee the same failure won't recur.
