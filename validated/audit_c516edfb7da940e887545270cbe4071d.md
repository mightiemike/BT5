### Title
Missing Access Control on `processSlowModeTransaction` Allows Unauthorized Collateral Withdrawal and Linked Signer Hijacking — (`Endpoint.sol` / `EndpointTx.sol`)

---

### Summary

`Endpoint.sol` exposes `processSlowModeTransaction(address sender, bytes calldata transaction)` as a public/external function with no access control. Its implementation, `EndpointTx.processSlowModeTransactionImpl`, is also declared `public` with no modifier. Because the slow-mode execution path performs no signature verification — relying solely on `validateSender` to confirm that the `sender` address matches the subaccount owner — an attacker who controls both the `sender` argument and the `transaction` bytes can trivially satisfy that check and execute privileged operations (collateral withdrawal, linked-signer replacement) on behalf of any victim subaccount.

---

### Finding Description

`Endpoint.sol` calls `processSlowModeTransaction` via `this.processSlowModeTransaction(txn.sender, txn.tx)` inside `_executeSlowModeTransaction` to enable try/catch semantics. This pattern requires the function to be `external` or `public`. No access-control modifier (e.g., `require(msg.sender == address(this))`) is present. [1](#0-0) 

The delegatecall target, `EndpointTx.processSlowModeTransactionImpl`, is declared `public` with no modifier: [2](#0-1) 

Inside `processSlowModeTransactionImpl`, the only authorization check for `WithdrawCollateral` is `validateSender(txn.sender, sender)`, which verifies that `address(bytes20(txn.sender)) == sender`. Both `sender` (a function parameter) and `txn.sender` (decoded from the attacker-supplied `transaction` bytes) are fully attacker-controlled when the function is called directly: [3](#0-2) 

There is no EIP-712 signature check in the slow-mode execution path — that check exists only in `processTransactionImpl` for sequencer-submitted batches. The slow-mode path was designed to trust `sender` as the address that originally submitted the slow-mode transaction via `submitSlowModeTransaction`. Calling `processSlowModeTransaction` directly breaks that assumption entirely.

The same absence of access control applies to the `LinkSigner` branch, allowing an attacker to overwrite `linkedSigners[victimSubaccount]` with an attacker-controlled address: [4](#0-3) 

`processTransactionImpl` is also `public` with no modifier, though its branches include signature validation, making it a lower-severity surface: [5](#0-4) 

---

### Impact Explanation

An unprivileged attacker can:

1. **Steal any user's collateral** — call `Endpoint.processSlowModeTransaction(victimAddress, withdrawTx)` where `withdrawTx` encodes a `WithdrawCollateral` with `txn.sender = bytes32(bytes20(victimAddress))`. `validateSender` passes; no signature is required; `clearinghouse.withdrawCollateral` executes immediately, bypassing the 3-day slow-mode delay.

2. **Hijack any subaccount's linked signer** — submit a `LinkSigner` transaction in the same way, replacing `linkedSigners[victimSubaccount]` with the attacker's address, granting the attacker signing authority over all future sequencer-submitted transactions for that subaccount.

The corrupted state deltas are: `spotEngine` balance of the victim subaccount reduced to zero, and/or `linkedSigners[victimSubaccount]` set to an attacker address. [3](#0-2) 

---

### Likelihood Explanation

The entry point is a standard external function call on the deployed `Endpoint` proxy — no special role, no front-running dependency, no flash loan, no governance. Any EOA can execute the attack in a single transaction. The only prerequisite is knowing a victim's Ethereum address, which is public on-chain. [6](#0-5) 

---

### Recommendation

Add a self-call guard to `processSlowModeTransaction` in `Endpoint.sol`:

```solidity
function processSlowModeTransaction(address sender, bytes calldata transaction)
    external
{
    require(msg.sender == address(this), "only self");
    _delegatecallEndpointTx(
        abi.encodeWithSelector(
            EndpointTx.processSlowModeTransactionImpl.selector,
            sender,
            transaction
        )
    );
}
```

Equivalently, restrict `EndpointTx.processSlowModeTransactionImpl` to `internal` (since it is only ever reached via `delegatecall` from `Endpoint`), mirroring the pattern used for other internal helpers in the engine contracts. [2](#0-1)

### Proof of Concept

```solidity
// Attacker constructs a WithdrawCollateral slow-mode transaction for the victim
bytes32 victimSubaccount = bytes32(bytes20(victimAddress)); // default subaccount name = 0x00..00

IEndpoint.WithdrawCollateral memory txn = IEndpoint.WithdrawCollateral({
    sender:    victimSubaccount,
    productId: QUOTE_PRODUCT_ID,
    amount:    victimEntireBalance,
    nonce:     0
});

bytes memory transaction = abi.encodePacked(
    uint8(IEndpoint.TransactionType.WithdrawCollateral),
    abi.encode(txn)
);

// Direct call — no signature, no queue delay, no fee
endpoint.processSlowModeTransaction(victimAddress, transaction);
// validateSender passes: address(bytes20(victimSubaccount)) == victimAddress
// clearinghouse.withdrawCollateral executes; victim's collateral is gone
``` [3](#0-2) [7](#0-6)
