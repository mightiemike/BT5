### Title
`op_secp256k1_verify` Accepts High-S Signatures, Breaking Canonical Signature Uniqueness â€” (`src/secp_ops.rs`)

### Summary
`op_secp256k1_verify` does not enforce low-S normalization. The `k256` library's `Signature::from_slice` and `verify_prehash` both accept high-S signatures. For any valid `(pubkey, msg, sig=(r,s))` triple, the high-S variant `sig'=(r, n-s)` also passes verification, meaning two distinct 64-byte signature atoms produce `Ok` for the same `(pubkey, msg)` pair.

### Finding Description
In `src/secp_ops.rs`, `op_secp256k1_verify` parses the signature with `K1Signature::from_slice` and verifies with `verifier.verify_prehash`. Neither call enforces the low-S invariant (`s <= n/2`). The `k256 = "0.14.0-rc"` crate's compact signature parsing explicitly accepts any `s` in `[1, n-1]`; low-S normalization via `normalize_s()` is a separate, optional step that is never called here. As a result, for any valid canonical signature `(r, s)`, the non-canonical high-S variant `(r, n-s)` is also accepted by the operator. [1](#0-0) [2](#0-1) 

### Impact Explanation
The invariant "exactly one canonical 64-byte signature passes `secp256k1_verify` for a given `(pubkey, msg)` pair" is violated. An attacker holding a valid spend can produce a second, distinct spend with signature `(r, n-s)` that also passes CLVM evaluation. This enables:

- **Transaction malleability**: two spends with different signature atoms but identical semantic effect, producing different coin IDs or spend bundle hashes depending on how the outer layer hashes them.
- **Consensus splits**: any node or wallet that enforces low-S (e.g., via libsecp256k1 or a stricter CLVM implementation) will reject the high-S spend, while this implementation accepts it, causing divergence.

### Likelihood Explanation
The attack is trivially constructable: given any valid `(r, s)`, compute `s' = n - s` and submit `(r, s')`. No special privileges or access are required beyond the ability to submit CLVM programs â€” a standard, attacker-controlled API path. The `k256` library's behavior is documented and stable.

### Recommendation
After parsing with `K1Signature::from_slice`, add an explicit low-S check before proceeding to verification:

```rust
// Reject high-S signatures (non-canonical)
if sig.normalize_s().is_some() {
    return Err(EvalErr::InvalidOpArg(
        input,
        "secp256k1_verify: signature has high-S value (not canonical)".to_string(),
    ))?;
}
```

`normalize_s()` returns `Some(normalized)` if and only if the signature had a high-S value. Returning an error on `Some` enforces the low-S invariant without changing behavior for already-canonical signatures. [3](#0-2)

pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
