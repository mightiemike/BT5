Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of end-user, permanently breaking per-user swap access control for router-routed swaps — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` routes a swap, it is the entity that calls `pool.swap()`, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actual_user]`. This makes the per-user allowlist structurally broken for all router-routed swaps: allowlisted users are locked out, and allowlisting the router to compensate grants every user unrestricted access.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` in `ExtensionCalling.sol` forwards `sender` unchanged to `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router is the direct caller of `pool.swap()`. The actual end-user (`msg.sender` of `exactInputSingle`) is stored only in transient storage as the payer and is never forwarded to the extension: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router_address]` — structurally wrong for per-user access control. By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the semantically meaningful actor (`owner`, the position owner) rather than the structural caller (`sender`), making it immune to this class of bypass: [5](#0-4) 

## Impact Explanation
Two concrete broken states arise from this structural mismatch:

**State A — Allowlisted users locked out of the router.** A pool admin allowlists specific user addresses (e.g., KYC'd counterparties). Those users call `exactInputSingle` through the router. The extension checks `allowedSwapper[pool][router]`, which is `false`. Every swap reverts with `NotAllowedToSwap`. The router — the primary user-facing swap interface — is permanently unusable for any pool that has a per-user `SwapAllowlistExtension` configured.

**State B — Full allowlist bypass.** To unblock router users, the admin allowlists the router address: `setAllowedToSwap(pool, router, true)`. Now `allowedSwapper[pool][router] = true`, and the guard passes for every caller of `exactInputSingle`, regardless of who they are. Any unprivileged user can swap against a pool intended to be restricted to a specific set of counterparties.

State B directly satisfies the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" impact criterion. State A satisfies "Broken core pool functionality causing unusable swap flows." There is no correct configuration that avoids both broken states simultaneously.

## Likelihood Explanation
Any pool that deploys `SwapAllowlistExtension` with a `beforeSwap` order and expects users to interact via `MetricOmmSimpleRouter` hits this immediately. Both contracts are first-party periphery contracts shipped together. The admin has no in-protocol mechanism to pass the real user identity through the router to the extension. The exploit requires no special privileges — any unprivileged user calling `exactInputSingle` on a pool where the router has been allowlisted (State B) bypasses the restriction entirely.

## Recommendation
The extension must receive the real initiating user, not the intermediary:

1. **Pass the real user via `extensionData`.** The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap` decodes and checks that address. This requires a coordinated change to the router and the extension.
2. **Check `recipient` as a proxy.** If the protocol convention is that `recipient == initiating user`, the extension can check `recipient` instead of `sender`. This is weaker (recipient can be set to any address) but requires no router change.

The deposit allowlist's pattern — checking the semantically meaningful actor (`owner`) rather than the structural caller (`sender`) — is the correct model to follow.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension as extension1, beforeSwap order = 1.
2. Admin calls swapExtension.setAllowedToSwap(pool, alice, true).
   → allowedSwapper[pool][alice] = true
3. Alice calls MetricOmmSimpleRouter.exactInputSingle({pool, recipient: alice, ...}).
   → router calls pool.swap(alice, ...) with msg.sender = router
   → pool calls _beforeSwap(sender = router, ...)
   → SwapAllowlistExtension checks allowedSwapper[pool][router] → false
   → revert NotAllowedToSwap   ← Alice is allowlisted but cannot swap via router.

4. Admin calls swapExtension.setAllowedToSwap(pool, router, true) to fix Alice.
   → allowedSwapper[pool][router] = true
5. Bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({pool, ...}).
   → router calls pool.swap(...) with msg.sender = router
   → SwapAllowlistExtension checks allowedSwapper[pool][router] → true
   → swap succeeds   ← Bob bypasses the allowlist entirely.
```
