Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of Actual User, Enabling Full Allowlist Bypass via Router — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool's `swap()` call. When users route through `MetricOmmSimpleRouter`, `sender` resolves to the router's address rather than the actual user. If the pool admin allowlists the router to enable allowlisted users to access it, every user of the router — including those the admin intended to block — can bypass the allowlist entirely.

## Finding Description
**Root cause:** `MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`. [1](#0-0) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, `msg.sender` inside the pool is the router contract, not the originating user. [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` = `allowedSwapper[pool][router]`. [3](#0-2) 

The router forwards `params.extensionData` verbatim and adds no user-identity information, so the extension has no way to recover the real caller. [4](#0-3) 

**Exploit path:**
1. Admin deploys pool with `SwapAllowlistExtension` to restrict swaps to KYC-approved addresses.
2. Admin calls `setAllowedToSwap(pool, userA, true)` to allowlist a legitimate user.
3. Admin calls `setAllowedToSwap(pool, router, true)` — the only available mechanism to let `userA` use the canonical router.
4. `userB` (not allowlisted) calls `router.exactInputSingle({pool: pool, ..., recipient: userB})`.
5. Router calls `pool.swap(...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(sender = router, ...)`.
7. Extension checks `allowedSwapper[pool][router] == true` → passes.
8. Swap executes; `userB` receives token output. Allowlist is fully bypassed.

**Why existing guards are insufficient:** The only guard is `allowedSwapper[pool][sender]`. [3](#0-2) 
There is no secondary check on the originating EOA, no signed credential mechanism, and no way to distinguish which EOA triggered the router call. The `allowAllSwappers` escape hatch is a separate flag and does not mitigate this issue. [5](#0-4) 

**Contrast with `DepositAllowlistExtension`:** The deposit extension checks `owner` (the position owner passed explicitly by the pool), which is independent of who calls `addLiquidity`. The swap extension has no equivalent — it only has `sender`, which collapses to the router for all router-mediated swaps.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` for KYC-gating, institutional-only access, or any curated allowlist loses that restriction entirely for any user routing through `MetricOmmSimpleRouter` once the admin allowlists the router. Unauthorized users receive real token output from the pool; LP providers bear the other side of every unauthorized swap. This is a direct loss of user principal and broken core pool functionality (access-controlled swap flow), matching the allowed impact gate.

## Likelihood Explanation
- `SwapAllowlistExtension` is a production extension explicitly designed for access-controlled pools.
- `MetricOmmSimpleRouter` is the canonical user-facing entry point; allowlisted users will naturally want to use it.
- The only mechanism to enable router access for allowlisted users is `setAllowedToSwap(pool, router, true)`, which is the exact action that opens the bypass.
- No special privilege, malicious setup, or non-standard token behavior is required. Any unprivileged address can call the router.

## Recommendation
The extension must gate the actual user, not the intermediary. Two viable approaches:

1. **Router-injected identity in `extensionData`:** Have `MetricOmmSimpleRouter` prepend `msg.sender` (the real user) to `extensionData` before forwarding to the pool. `SwapAllowlistExtension.beforeSwap` decodes and checks that address. This requires the allowlist to trust the router as an honest forwarder, which is acceptable for a factory-deployed periphery contract.

2. **Signed user credential:** Provide a companion extension that decodes a signed user credential from `extensionData` for router paths, and document clearly that allowlisting the router grants access to all router users without this mechanism.

Either way, the current behavior — where `sender = router` for all router-mediated swaps — must be fixed before the extension is used in access-controlled production pools.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension (beforeSwap order = 1).
2. Admin: swapExt.setAllowedToSwap(pool, userA, true)   // allowlist userA
3. Admin: swapExt.setAllowedToSwap(pool, router, true)  // enable router for userA
4. userB (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ..., recipient: userB, extensionData: ""})
5. router → pool.swap(recipient=userB, ...) with msg.sender=router
6. pool → _beforeSwap(sender=router, ...)
7. swapExt checks: allowedSwapper[pool][router] == true  ✓
8. Swap executes. userB receives token output. Allowlist bypassed.
``` [6](#0-5) [7](#0-6) [8](#0-7)
