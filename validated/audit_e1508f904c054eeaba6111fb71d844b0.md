### Title
Unbounded `while` loop in `tryUnlockNlpBalance` allows NLP tokens to be permanently locked via gas exhaustion — (`File: core/contracts/SpotEngineState.sol`)

---

### Summary

An unbounded `while` loop in `SpotEngineState.tryUnlockNlpBalance` iterates over every matured entry in a per-subaccount `NlpLockedBalanceQueue`. Because there is no cap on how many entries can accumulate in the queue, a user who mints NLP many times at distinct oracle timestamps can grow the queue arbitrarily large. Once the lock period expires, any subsequent call to `burnNlp` will invoke `tryUnlockNlpBalance`, exhaust all block gas, and revert — permanently locking the user's NLP tokens.

---

### Finding Description

`NlpLockedBalanceQueue` is a per-subaccount data structure that records each NLP mint as a time-locked entry:

```solidity
struct NlpLockedBalanceQueue {
    mapping(uint64 => NlpLockedBalance) balances;
    uint64 balanceCount;
    uint64 unlockedUpTo;
    Balance unlockedBalanceSum;
}
``` [1](#0-0) 

`handleNlpLockedBalance` appends a new entry to the queue whenever a mint occurs at a different oracle timestamp than the last entry. There is no cap on `balanceCount`:

```solidity
queue.balances[queue.balanceCount] = NlpLockedBalance({
    balance: Balance({amount: amountDelta}),
    unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
});
queue.balanceCount++;
``` [2](#0-1) 

`tryUnlockNlpBalance` processes all matured entries in a single unbounded `while` loop:

```solidity
while (
    queue.unlockedUpTo < queue.balanceCount &&
    queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
) {
    queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
    delete queue.balances[queue.unlockedUpTo];
    queue.unlockedUpTo++;
}
``` [3](#0-2) 

`burnNlp` in `Clearinghouse.sol` calls `spotEngine.getNlpUnlockedBalance`, which unconditionally calls `tryUnlockNlpBalance`:

```solidity
require(
    spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
    ERR_UNLOCKED_NLP_INSUFFICIENT
);
``` [4](#0-3) 

```solidity
function getNlpUnlockedBalance(bytes32 subaccount) external returns (Balance memory) {
    tryUnlockNlpBalance(subaccount);
    ...
}
``` [5](#0-4) 

---

### Impact Explanation

**Impact: High**

If a user's `NlpLockedBalanceQueue` grows large enough (N entries all matured), every call to `burnNlp` will attempt to iterate N times in `tryUnlockNlpBalance`. When N is sufficiently large, the transaction exceeds the block gas limit and reverts. Since `tryUnlockNlpBalance` is always called before the burn check, there is no code path to burn NLP that bypasses this loop. The user's NLP tokens become permanently unburnable — a complete, irreversible loss of the ability to redeem locked value. [6](#0-5) 

---

### Likelihood Explanation

**Likelihood: Low**

Each queue entry requires a mint at a distinct oracle timestamp. A user would need to submit many small mint transactions spread across many different oracle time slots. This is achievable by a determined user over time, or could occur naturally for a highly active liquidity provider who mints NLP frequently across many blocks. The sequencer cannot refuse valid signed transactions without censoring the user. [7](#0-6) 

---

### Recommendation

Cap the maximum number of entries in `NlpLockedBalanceQueue` per subaccount (e.g., enforce `queue.balanceCount - queue.unlockedUpTo <= MAX_QUEUE_DEPTH` before appending). Alternatively, process only a bounded number of entries per call to `tryUnlockNlpBalance` and allow partial unlocking across multiple transactions. [2](#0-1) 

---

### Proof of Concept

1. User signs and submits `mintNlp` transactions at N distinct oracle timestamps (e.g., one per block over many blocks). Each mint appends a new entry to `nlpLockedBalanceQueues[subaccount]`, growing `balanceCount` to N with no cap enforced. [2](#0-1) 

2. After `NLP_LOCK_PERIOD` elapses, all N entries satisfy `unlockedAt <= getOracleTime()`. [8](#0-7) 

3. User submits a `burnNlp` transaction. `Clearinghouse.burnNlp` calls `spotEngine.getNlpUnlockedBalance(txn.sender)`, which calls `tryUnlockNlpBalance`. The `while` loop iterates N times, each iteration performing a storage read, a storage write (`unlockedBalanceSum`), a `delete`, and an increment. [4](#0-3) 

4. For sufficiently large N, the transaction exceeds the block gas limit and reverts. The user's NLP balance remains locked in the queue. All future `burnNlp` attempts fail identically — the tokens are permanently locked. [6](#0-5)
