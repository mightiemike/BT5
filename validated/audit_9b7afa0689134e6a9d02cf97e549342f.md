### Title
`op_unknown` cost_function=3 Omits `MALLOC_COST_PER_BYTE` Compared to Real `op_concat`, Enabling Undercharged Execution — (`src/more_ops.rs`)

---

### Summary

The `op_unknown` function's `cost_function=3` branch is documented as computing cost "like operator concat", but it charges only `CONCAT_COST_PER_BYTE` (= 3) per input byte. The real `op_concat` charges `CONCAT_COST_PER_BYTE + MALLOC_COST_PER_BYTE` (= 3 + 10 = 13) per input byte. This is a 4.3× undercharge per byte. An attacker can craft CLVM programs using unknown opcodes with `cost_function=3` and large atom arguments to consume significantly more resources per cost unit than the cost model intends, bypassing the cost ceiling in consensus mode.

---

### Finding Description

**Real `op_concat` per-byte cost** (`src/more_ops.rs`, line 905):
```rust
cost += len as Cost * (CONCAT_COST_PER_BYTE + MALLOC_COST_PER_BYTE);
// = len * (3 + 10) = len * 13
```

**`op_unknown` cost_function=3 per-byte cost** (`src/more_ops.rs`, line 250):
```rust
cost += CONCAT_COST_PER_BYTE * (len as Cost);
// = len * 3
```

The comment at line 188–189 explicitly states that cost_function=3 is "computed like operator concat". The `MALLOC_COST_PER_BYTE = 10` constant (defined in `src/op_utils.rs`, line 13) is the per-byte cost ascribed to heap allocation of new atoms. In `op_concat`, this is charged per input byte because the output size equals the total input size — the malloc cost is embedded in the per-byte loop rather than added at the end. The `op_unknown` branch copies the loop structure but silently drops the `MALLOC_COST_PER_BYTE` term, producing a cost that is 4.3× lower than the stated model.

The `cost_function` field is attacker-controlled: it is extracted from bits 6–7 of the last byte of the unknown opcode atom (line 201: `let cost_function = (op[op.len() - 1] & 0b11000000) >> 6`). Any opcode byte ending in `0xC0`–`0xFF` (with bits 6–7 set) selects `cost_function=3`. The `cost_multiplier` from the preceding bytes then scales the undercharged base, amplifying the gap. ### Impact Explanation

In consensus mode (lenient mode, i.e. without `NO_UNKNOWN_OPS`), unknown opcodes are permitted as no-ops with defined costs. The cost ceiling is the primary resource-exhaustion guard. Because `cost_function=3` undercharges by 10 cost units per input byte, an attacker can submit a program that passes N bytes through unknown ops at cost `3N` instead of the intended `13N`. For a 1 MB payload, the attacker pays ~3.1 M cost instead of ~13.6 M cost — freeing ~10.5 M cost units to be spent on additional computation. This is a concrete undercharged-execution issue: programs that should be rejected for exceeding the cost limit are instead accepted, and nodes spend more real CPU/memory time than the cost model guarantees. ---

### Likelihood Explanation

The trigger is fully attacker-controlled CLVM bytes submitted to any node running in consensus mode. No special privileges are required. The attacker only needs to know the opcode encoding scheme (documented in the inline comment at lines 171–189 of `src/more_ops.rs`) and craft an opcode atom whose last byte has bits 6–7 set. This is straightforward from the public specification. The `cost_multiplier` field allows the attacker to further scale the undercharged cost, amplifying the effect. [7](#0-6) 

---

### Recommendation

Change the `cost_function=3` branch in `op_unknown` to include `MALLOC_COST_PER_BYTE` in the per-byte cost, matching `op_concat`:

```rust
3 => {
    let mut cost = CONCAT_BASE_COST;
    while let Some((arg, rest)) = allocator.next(args) {
        args = rest;
        let len = atom_len(allocator, arg, "unknown op")?;
        cost += CONCAT_COST_PER_ARG;
        // Fix: add MALLOC_COST_PER_BYTE to match op_concat
        cost += (CONCAT_COST_PER_BYTE + MALLOC_COST_PER_BYTE) * (len as Cost);
        check_cost(cost, max_cost)?;
    }
    cost
}
```

Since this changes the cost of existing unknown opcodes in consensus mode, it must be deployed as a hard fork or soft fork with appropriate activation logic. [1](#0-0)

### Proof of Concept

Craft an unknown opcode with `cost_function=3`: any single-byte opcode value not in the known set (e.g. `0xC0`, whose bits 6–7 = `0b11` → `cost_function=3`, `cost_multiplier=0`). Pass a large atom argument, e.g. 100,000 bytes.

**Expected cost** (matching `op_concat`):
```
CONCAT_BASE_COST + CONCAT_COST_PER_ARG + (CONCAT_COST_PER_BYTE + MALLOC_COST_PER_BYTE) * 100000
= 142 + 135 + 13 * 100000 = 1,300,277
```

**Actual cost charged** by `op_unknown` cost_function=3:
```
CONCAT_BASE_COST + CONCAT_COST_PER_ARG + CONCAT_COST_PER_BYTE * 100000
= 142 + 135 + 3 * 100000 = 300,277
```

**Shortfall**: 1,000,000 cost units for a single 100 KB argument — a 4.3× undercharge. An attacker can chain many such calls within the cost budget to process far more data than the cost model intends. [8](#0-7) [3](#0-2) [1](#0-0)

```rust
pub fn op_unknown(
    allocator: &mut Allocator,
    o: NodePtr,
    mut args: NodePtr,
    max_cost: Cost,
) -> Response {
    // unknown opcode in lenient mode
    // unknown ops are reserved if they start with 0xffff
    // otherwise, unknown ops are no-ops, but they have costs. The cost is computed
    // like this:

    // byte index (reverse):
    // | 4 | 3 | 2 | 1 | 0          |
    // +---+---+---+---+------------+
    // | multiplier    |XX | XXXXXX |
    // +---+---+---+---+---+--------+
    //  ^               ^    ^
    //  |               |    + 6 bits ignored when computing cost
    // cost_multiplier  |
    // (up to 4 bytes)  + 2 bits
    //                    cost_function

    // 1 is always added to the multiplier before using it to multiply the cost, this
    // is since cost may not be 0.

    // cost_function is 2 bits and defines how cost is computed based on arguments:
    // 0: constant, cost is 1 * (multiplier + 1)
    // 1: computed like operator add, multiplied by (multiplier + 1)
    // 2: computed like operator mul, multiplied by (multiplier + 1)
    // 3: computed like operator concat, multiplied by (multiplier + 1)

    // this means that unknown ops where cost_function is 1, 2, or 3, may still be
    // fatal errors if the arguments passed are not atoms.

    let op_atom = allocator.atom(o);
    let op = op_atom.as_ref();

    if op.is_empty() || (op.len() >= 2 && op[0] == 0xff && op[1] == 0xff) {
        Err(EvalErr::Reserved(o))?;
    }

    let cost_function = (op[op.len() - 1] & 0b11000000) >> 6;
    let cost_multiplier: u64 = match u32_from_u8(&op[0..op.len() - 1]) {
        Some(v) => v as u64,
        None => {
            return Err(EvalErr::Invalid(o))?;
        }
    };
```

```rust
pub fn op_concat(
    a: &mut Allocator,
    mut input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = CONCAT_BASE_COST;
    let mut total_size: usize = 0;
    let mut terms = Vec::<NodePtr>::new();
    while let Some((arg, rest)) = a.next(input) {
        input = rest;
        let len = match a.sexp(arg) {
            SExp::Pair(_, _) => {
                return Err(EvalErr::InvalidOpArg(arg, "concat on list".to_string()))?;
            }
            SExp::Atom => a.atom_len(arg),
        };
        cost += CONCAT_COST_PER_ARG;
        cost += len as Cost * (CONCAT_COST_PER_BYTE + MALLOC_COST_PER_BYTE);
        check_cost(cost, max_cost)?;
        if len > 0 {
            // skip NIL arguments, as an optimization
            total_size += len;
            terms.push(arg);
        }
    }

    let new_atom = a.new_concat(total_size, &terms)?;
    Ok(Reduction(cost, new_atom))
```
