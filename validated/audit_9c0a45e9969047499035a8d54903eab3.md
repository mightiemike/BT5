### Title
4-byte secp opcodes bypass both `NO_UNKNOWN_OPS` and `ENABLE_SECP_OPS` guards — (`src/chia_dialect.rs`)

### Summary
In `ChiaDialect::op`, the 4-byte opcode dispatch for `0x13d61f00` (`secp256k1_verify`) and `0x1c3a8f00` (`secp256r1_verify`) calls the operator function unconditionally, without checking `ClvmFlags::ENABLE_SECP_OPS` or `ClvmFlags::NO_UNKNOWN_OPS`. This allows attacker-controlled CLVM programs to execute secp signature verification in any dialect, including `MEMPOOL_MODE`, regardless of whether those operators are supposed to be enabled.

### Finding Description
`ChiaDialect::op` has two separate dispatch paths for secp ops:

- **1-byte opcodes 64/65**: gated behind `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` — correct.
- **4-byte opcodes `0x13d61f00`/`0x1c3a8f00`**: matched and called directly with no flag check — incorrect.

```rust
// 4-byte path — no ENABLE_SECP_OPS check, no NO_UNKNOWN_OPS check
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,   // executes unconditionally
    0x1c3a8f00 => op_secp256r1_verify,   // executes unconditionally
    _ => {
        return unknown_operator(...);    // only non-secp 4-byte ops are blocked
    }
};
return f(allocator, argument_list, max_cost, flags);
```

The `unknown_operator` function, which enforces `NO_UNKNOWN_OPS`, is never reached for these two opcodes. [4](#0-3) 

### Impact Explanation
Any caller using `ChiaDialect::new(ClvmFlags::NO_UNKNOWN_OPS)` — including `MEMPOOL_MODE` — intends to reject all non-standard operators. Instead, secp256k1 and secp256r1 signature verification remain silently available via their 4-byte encodings. A CLVM program submitted to a node running in mempool mode can perform secp verification in a context where the operator should be unavailable, violating the softfork gating invariant and potentially causing consensus divergence between nodes with different flag configurations.

### Likelihood Explanation
The attacker only needs to craft a CLVM program using the 4-byte opcode `0x13d61f00` with valid secp256k1 arguments. No special privileges are required. The path is reachable through the standard CLVM execution API with attacker-controlled program bytes.

### Recommendation
Add an `ENABLE_SECP_OPS` guard in the 4-byte opcode dispatch path, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both encodings of the secp operators are subject to the same flag gating.
