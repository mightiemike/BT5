### Title
Linked Signer Can Overwrite Its Own Authority via `LinkSigner` Fast Path — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the current linked signer of a subaccount can sign a `LinkSigner` transaction to replace itself — or install any attacker-controlled address — as the new linked signer. Because linked signers are authorized to sign `WithdrawCollateral` and other sensitive transactions, this allows a compromised or malicious linked signer to permanently escalate its own authority and drain the subaccount before the owner can react.

---

### Finding Description

The `linkedSigners` mapping in `EndpointStorage` stores a delegated signing address per subaccount. It is the primary authorization state for the fast-path transaction pipeline. [1](#0-0) 

In `processTransactionImpl`, the `LinkSigner` case calls `validateSignedTx` with `allowLinkedSigner = true`: [2](#0-1) 

`validateSignedTx` with `allowLinkedSigner = true` passes the current linked signer to `Verifier.validateSignature`, which accepts a signature from either the subaccount owner **or** the linked signer: [3](#0-2) 

Therefore, the current linked signer can produce a valid `LinkSigner` transaction that writes any new address into `linkedSigners[subaccount]`.

This is inconsistent with the protocol's own defensive patterns elsewhere:

- `WithdrawCollateralV2` uses `allowLinkedSigner = signedTx.tx.sendTo == address(0)` — explicitly blocking linked signers from withdrawing to arbitrary addresses. [4](#0-3) 

- `TransferQuote` uses `allowLinkedSigner = true` but adds `require(bytes20(txn.sender) == bytes20(txn.recipient))` with the comment *"otherwise linked signers can transfer out"*. [5](#0-4) 

- The **slow-mode** `LinkSigner` path uses `validateSender(txn.sender, sender)`, which enforces that only the actual owner (`msg.sender`) can change the linked signer. [6](#0-5) 

The fast-path `LinkSigner` has no equivalent restriction.

---

### Impact Explanation

A linked signer (e.g., a session key, API key, or hot wallet) can:

1. Sign a `LinkSigner` transaction naming an attacker-controlled address as the new signer.
2. The sequencer processes it via `processTransactionImpl`; `validateSignedTx` accepts the linked signer's signature.
3. `linkedSigners[subaccount]` is overwritten with the attacker's address.
4. The attacker's address is now the linked signer and can sign `WithdrawCollateral` (fast path, `allowLinkedSigner = true`) to drain all collateral.
5. The legitimate owner must use the slow-mode path to revoke the linked signer, which has a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). During this window the attacker can fully drain the subaccount. [7](#0-6) 

The corrupted state is `linkedSigners[subaccount]` — a direct authorization key that gates all fast-path signed transactions including withdrawals.

---

### Likelihood Explanation

Any user who has ever set a linked signer is exposed. Linked signers are a core UX feature (session keys, trading bots, API integrations). A malicious or compromised linked signer can execute this attack in a single sequencer batch with no preconditions beyond having been previously authorized. The attacker-controlled entry path is `Endpoint.submitTransactionsChecked` → `processTransactionImpl` → `LinkSigner` branch.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Changing the linked signer is a privileged ownership operation and must only be authorized by the subaccount owner's key, not by the current linked signer:

```diff
- validateSignedTx(
-     signedTx.tx.sender,
-     signedTx.tx.nonce,
-     transaction,
-     signedTx.signature,
-     true
- );
+ validateSignedTx(
+     signedTx.tx.sender,
+     signedTx.tx.nonce,
+     transaction,
+     signedTx.signature,
+     false   // only the subaccount owner may change the linked signer
+ );
```

This is consistent with the slow-mode path, which already enforces owner-only authorization for `LinkSigner` via `validateSender`.

---

### Proof of Concept

1. Alice owns subaccount `alice_sub` and has set `linkedSigners[alice_sub] = sessionKey`.
2. `sessionKey` is compromised by an attacker.
3. Attacker uses `sessionKey` to sign a `LinkSigner` transaction: `{ sender: alice_sub, signer: attacker_address, nonce: currentNonce }`.
4. Sequencer submits this via `submitTransactionsChecked` → `processTransactionImpl`.
5. `validateSignedTx(..., true)` recovers `sessionKey` from the signature; `sessionKey == linkedSigners[alice_sub]` → passes.
6. `linkedSigners[alice_sub]` is set to `attacker_address`.
7. Attacker signs `WithdrawCollateral { sender: alice_sub, productId: QUOTE_PRODUCT_ID, amount: fullBalance, nonce: nextNonce }` with `attacker_address`.
8. `validateSignedTx(..., true)` recovers `attacker_address`; `attacker_address == linkedSigners[alice_sub]` → passes.
9. Alice's entire collateral is withdrawn to `attacker_address`.
10. Alice can only recover via slow mode (3-day delay), by which time the funds are gone. [2](#0-1) [8](#0-7)
