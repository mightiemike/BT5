### Title
Unbounded Iteration in `reorg_chain` Can Permanently Freeze the Contract on a Lower-Chainwork Chain — (File: contract/src/lib.rs)

---

### Summary

`reorg_chain` contains an unbounded `while` loop that walks every fork block from the fork tip back to the common ancestor, performing multiple storage writes per iteration. As the fork grows, the gas cost of the single transaction that triggers the reorg grows proportionally. Once the fork is long enough to exceed NEAR's per-transaction gas limit (~300 TGas), the reorg transaction always reverts, the contract is permanently stuck on the lower-chainwork chain, and all subsequent SPV proofs are verified against the wrong canonical chain.

---

### Finding Description

When a submitted fork block's `chain_work` exceeds the current mainchain tip's `chain_work`, `submit_block_header_inner` calls `reorg_chain`. [1](#0-0) 

Inside `reorg_chain`, two loops execute in the same transaction:

**Loop 1** — removes mainchain blocks above the fork tip height (bounded by the height delta between mainchain and fork): [2](#0-1) 

**Loop 2** — the critical unbounded loop — walks every fork block from the fork tip back to the common ancestor, performing `LookupMap::insert`, `LookupMap::insert`, and `remove_block_header` (which itself does two `LookupMap::remove` calls) per iteration: [3](#0-2) 

The number of iterations equals the fork length — the total number of fork blocks accumulated in `headers_pool` since the fork diverged from the mainchain. Fork blocks are stored one-by-one across many prior `submit_blocks` calls, so the collection grows over time without any cap. There is no batch-size limit, no pagination, and no mechanism to split the reorg across multiple transactions.

The `headers_pool` comment confirms that all ever-submitted headers (including forks) are retained: [4](#0-3) 

---

### Impact Explanation

If the fork is long enough to push the reorg transaction past NEAR's gas ceiling, the transaction reverts atomically. The triggering fork-tip block is not stored. Every subsequent attempt to submit that same block re-triggers the same reorg and reverts again. The contract is permanently frozen with `mainchain_tip_blockhash` pointing to the lower-chainwork chain.

All callers of `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` then receive results computed against the wrong canonical chain: [5](#0-4) [6](#0-5) 

Concretely: the contract accepts SPV inclusion proofs for transactions in stale/orphaned blocks and rejects proofs for transactions in the true canonical chain. Any downstream contract or bridge relying on these verification results is compromised.

---

### Likelihood Explanation

The entry path is `submit_blocks`, which is gated by the `#[trusted_relayer]` macro: [7](#0-6) 

A trusted relayer is the normal operational role for this contract. The scenario does not require a malicious actor: a legitimate relayer faithfully following the Bitcoin (or Dogecoin/Litecoin/Zcash) protocol during a real long reorganization would naturally submit many fork blocks over successive transactions, then submit the final block that triggers the reorg — causing the gas-limit failure. Bitcoin has historically experienced multi-block reorgs; Dogecoin and Litecoin have experienced longer ones. The longer the reorg, the more fork blocks accumulate in `headers_pool` before the reorg is triggered, and the more iterations the while loop must execute in a single transaction.

---

### Recommendation

Replace the single-transaction unbounded reorg with a paginated, multi-step approach:

1. Add a `pending_reorg` field to contract state that records the fork tip hash and the current cursor position.
2. Expose a separate `apply_reorg(batch_size: u64)` method that advances the cursor by at most `batch_size` steps per call, updating `mainchain_height_to_header` and `mainchain_header_to_height` incrementally.
3. Only update `mainchain_tip_blockhash` once the cursor reaches the common ancestor (reorg complete).
4. Alternatively, precalculate and store the common-ancestor height at fork-submission time so that `reorg_chain` can be bounded by a caller-supplied `batch_size` parameter, analogous to how `run_mainchain_gc` already accepts a `batch_size`: [8](#0-7) 

---

### Proof of Concept

1. Initialize the contract with `skip_pow_verification = true`.
2. Submit N mainchain blocks (e.g., N = 50) via `submit_blocks` to establish the mainchain tip.
3. Submit N+1 fork blocks (each with `prev_block_hash` pointing to the fork's previous block, starting from a common ancestor, with slightly higher per-block work) via repeated `submit_blocks` calls. Each call stores the fork block in `headers_pool` without triggering a reorg (fork chainwork < mainchain chainwork until the last block).
4. Submit the final fork block whose cumulative `chain_work` exceeds the mainchain tip. This triggers `reorg_chain`, which must iterate over all N+1 fork blocks in a single NEAR transaction.
5. Observe the transaction fails with an out-of-gas error. The contract's `mainchain_tip_blockhash` remains on the lower-chainwork chain.
6. Repeat step 4 — the same failure occurs every time. The contract is permanently stuck.
7. Call `verify_transaction_inclusion` with a proof for a transaction in one of the fork blocks — it returns `false` (block not on mainchain). Call it with a proof for a stale mainchain block — it returns `true`. The canonical chain mapping is permanently corrupted.

        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1

            .headers_pool
