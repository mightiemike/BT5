### Title
Builder Fees Permanently Locked When Builder Is Removed via `updateBuilder` ‚Äî (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

When a builder is removed by setting its `owner` to `address(0)` via `updateBuilder`, all accumulated fees stored in `collectedBuilderFee[productId][builderId]` become permanently unclaimable. The `claimBuilderFee` function enforces a strict ownership check that always fails for any real user once the builder's owner is zeroed out, mirroring the gauge-removal voting-power lock from the reference report.

---

### Finding Description

Builders accumulate fees in `collectedBuilderFee[quoteId][builderId]` every time an order that references their `builderId` is matched: [1](#0-0) 

The only mechanism to withdraw these fees is `claimBuilderFee`, which enforces: [2](#0-1) 

The critical check is:

```solidity
require(
    builders[builderId].owner == address(uint160(bytes20(sender))),
    ERR_UNAUTHORIZED
);
```

A builder is removed by the owner submitting an `UpdateBuilder` slow-mode transaction with `owner = address(0)`: [3](#0-2) 

After this call, `builders[builderId].owner == address(0)`. For any real user, `address(uint160(bytes20(sender)))` is always non-zero, so the ownership check permanently fails. There is no alternative code path to drain `collectedBuilderFee` ‚Äî `dumpFees` only handles `marketInfo[productId].collectedFees`, not builder fees: [4](#0-3) 

The `UpdateBuilder` slow-mode path is restricted to the contract owner: [5](#0-4) 

---

### Impact Explanation

All fees accumulated in `collectedBuilderFee[productId][builderId]` across every spot product are permanently frozen in the `OffchainExchange` contract. The builder's earned quote-token balance can never be credited to any subaccount. The asset delta is the full sum of `collectedBuilderFee[*][builderId]` at the time of removal ‚Äî a concrete, non-recoverable loss of earned protocol fees belonging to the builder.

---

### Likelihood Explanation

The trigger is a routine administrative action: removing or replacing a builder. The owner does not need to be malicious ‚Äî a builder offboarding, a key rotation, or a protocol upgrade that replaces a builder entry with `owner = address(0)` is sufficient. Any unclaimed fees at that moment are permanently lost. Builders who have not recently claimed fees (e.g., during high-volume periods) are most exposed.

---

### Recommendation

Before zeroing out the builder's owner in `updateBuilder`, transfer any outstanding `collectedBuilderFee` balance to the existing owner's subaccount, or record the previous owner so they can still claim after removal:

```solidity
function updateBuilder(bytes calldata transaction) external onlyEndpoint {
    IEndpoint.UpdateBuilder memory txn = abi.decode(transaction[1:], (IEndpoint.UpdateBuilder));
    
    // Flush any outstanding fees to the existing owner before overwriting
    address existingOwner = builders[txn.builderId].owner;
    if (existingOwner != address(0) && txn.owner == address(0)) {
        _flushBuilderFees(txn.builderId, existingOwner);
    }
    
    builders[txn.builderId] = Builder(txn.owner, txn.defaultFeeTier, txn.lowestFeeRate, txn.highestFeeRate);
}
```

Alternatively, allow `claimBuilderFee` to accept a proof of prior ownership, or store the previous owner address so they retain claim rights after removal.

---

### Proof of Concept

1. Builder `B` is registered with `builderId = 1` and `owner = alice`.
2. Over time, trades referencing `builderId = 1` accumulate `collectedBuilderFee[QUOTE_PRODUCT_ID][1] = 1000e18`.
3. Alice has not yet called `claimBuilderFee`.
4. The protocol owner submits a slow-mode `UpdateBuilder` transaction with `builderId = 1, owner = address(0)`.
5. `builders[1].owner` is now `address(0)`.
6. Alice calls `claimBuilderFee` (via the sequencer submitting a `ClaimBuilderFee` transaction):
   - The check `builders[1].owner == address(uint160(bytes20(alice_subaccount)))` evaluates to `address(0) == alice`, which is `false`.
   - The call reverts with `ERR_UNAUTHORIZED`.
7. `collectedBuilderFee[QUOTE_PRODUCT_ID][1]` remains `1000e18` forever with no recovery path. [6](#0-5) [3](#0-2) [7](#0-6)
