Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of original user, enabling allowlist bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap()` passes `msg.sender` (the immediate caller) as the `sender` argument to `_beforeSwap()`, which forwards it to `SwapAllowlistExtension.beforeSwap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` seen by the pool is the router contract, not the original user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. If the pool admin allowlists the router to enable router-mediated swaps, every unprivileged user—including those explicitly excluded from the allowlist—can bypass the gate by calling through the router.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)` (or `exactInput` / `exactOutputSingle` / `exactOutput`).
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — the pool sees `msg.sender = router`.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` — `sender = router address`.
4. `ExtensionCalling._beforeSwap()` encodes `sender` (router) and dispatches to `SwapAllowlistExtension.beforeSwap(sender=router, ...)` with `msg.sender = pool`.
5. `SwapAllowlistExtension.beforeSwap()` evaluates:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
       revert IMetricOmmPoolActions.NotAllowedToSwap();
   }
   // msg.sender = pool, sender = router
   ```
   The check resolves to `allowedSwapper[pool][router]`.

**Root cause:** `MetricOmmPool.swap()` at line 231 passes `msg.sender` (the immediate caller) as `sender`. There is no mechanism to propagate the original EOA or user address through the router into the extension.

**Why existing guards fail:** The `onlyPool` modifier in `BaseMetricExtension` correctly ensures only the pool can call the extension, but it does not validate that the `sender` argument represents the true economic actor. The pool itself has no knowledge of who called the router.

**Two concrete broken invariants:**

- **Bypass (critical path):** Pool admin allowlists the router (a natural operational step to let users use the periphery). Any non-allowlisted user calls the router → router is allowlisted → check passes → swap executes. The allowlist is fully bypassed.
- **Lockout (secondary path):** Pool admin allowlists individual users but not the router. Those users cannot swap through the router even though they are individually permitted, breaking core swap functionality for allowlisted pools.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control primitive. When the router is allowlisted (the only way to enable router-mediated swaps on an allowlisted pool), the extension no longer gates individual swappers — it gates the router contract, which any user can call. This is a direct admin-boundary break: the pool admin's intent to restrict swaps to specific addresses is bypassed by any unprivileged trader routing through the public `MetricOmmSimpleRouter`. Pools using this extension for KYC/compliance or whitelist-only liquidity access are fully compromised.

## Likelihood Explanation
The precondition — router being allowlisted — is the natural operational state for any allowlisted pool that also wants to support the standard periphery. A pool admin who sets `allowedSwapper[pool][router] = true` to enable router usage inadvertently opens the pool to all users. The attacker requires no special privilege: any EOA can call `MetricOmmSimpleRouter.exactInputSingle` with the target pool. The bypass is repeatable on every block.

## Recommendation
The pool should pass the original user's address as `sender` rather than `msg.sender`. One approach: add an explicit `sender` parameter to `IMetricOmmPoolActions.swap()` that the caller supplies, and have the pool validate it against `msg.sender` only when `msg.sender != sender` (i.e., require the caller to be an approved operator for that sender). Alternatively, the router should pass the original `msg.sender` through the `callbackData` or a dedicated field, and the pool should forward it as `sender` to extensions. The simplest fix consistent with the existing design is to add a `sender` argument to `swap()` and require `msg.sender == sender || isApprovedOperator(sender, msg.sender)`, so extensions always see the true economic actor.
