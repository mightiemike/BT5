### Title
`get_split_parent_shard_uids()` stamps parent `ShardUId` with the **child** layout's version instead of the parent layout's version, causing cold-storage mapping and memtrie preload to target a non-existent DB prefix — (`core/primitives/src/shard_layout/mod.rs`)

---

### Summary

`ShardLayout::get_split_parent_shard_uids()` constructs each parent `ShardUId` with `self.version()` — the **new** (child) layout's version — rather than the **previous** (parent) layout's version. The parent shard belongs to the previous layout and is stored in the DB under the previous layout's version prefix. When the network first enables dynamic resharding (V2 → V3 bootstrapping), the returned UIDs carry `version=3` instead of `version=2`, causing `update_state_shard_uid_mapping()` in cold storage to install a child→wrong-prefix mapping and `get_resharding_parent_shard_uid()` to hand a non-existent `ShardUId` to the memtrie preloader.

---

### Finding Description

**Root cause — wrong version in `get_split_parent_shard_uids()`** [1](#0-0) 

```rust
pub fn get_split_parent_shard_uids(&self) -> BTreeSet<ShardUId> {
    let parent_shard_ids = self.get_split_parent_shard_ids();
    parent_shard_ids
        .into_iter()
        .map(|shard_id| ShardUId::new(self.version(), shard_id))  // ← uses child layout version
        .collect()
}
```

`get_split_parent_shard_ids()` returns shard IDs that existed in the **previous** layout and were split to produce the current layout. Those shards are stored in the DB under the **previous** layout's version prefix. Tagging them with `self.version()` (the new layout's version) produces a `ShardUId` that does not correspond to any real DB key.

For **V3 → V3** dynamic resharding both layouts share `version=3`, so the bug is latent. For the **V2 → V3 bootstrapping** transition (the first dynamic resharding), the parent shard has `version=2` in the DB but the function returns `ShardUId { version: 3, shard_id: X }`.

**Propagation path 1 — cold-storage shard-UID mapping** [2](#0-1) 

`update_state_shard_uid_mapping()` calls `shard_layout.get_split_parent_shard_uids()` on the new layout, then calls `get_shard_uid_mapping(&cold_store, parent_shard_uid)` with the wrong UID. Because no mapping exists for `ShardUId { version: 3, shard_id: X }`, `get_shard_uid_mapping` returns the identity. Each child shard is then mapped to `ShardUId { version: 3, shard_id: X }` instead of `ShardUId { version: 2, shard_id: X }`. All subsequent cold-store `State` lookups for child shards use the wrong 8-byte prefix and find no data.

**Propagation path 2 — memtrie preload for resharding** [3](#0-2) 

`get_resharding_parent_shard_uid()` calls `next_layout.get_split_parent_shard_uids()` and returns the first element. The caller `maybe_start_memtrie_preload_for_resharding()` passes this wrong UID to `tries.spawn_background_memtrie_loading_for_shard(parent_shard_uid, ...)`. [4](#0-3) 

The background loader looks for a memtrie keyed by `ShardUId { version: 3, shard_id: X }`, which does not exist. The actual parent shard `ShardUId { version: 2, shard_id: X }` is never preloaded. When resharding executes, `process_memtrie_resharding_storage_update()` checks `parent_trie.has_memtries()` and, if `allow_resharding_without_memtries` is false, returns an error, halting resharding. [5](#0-4) 

**Contrast with the correct pattern**

The shard-tracker test constructs the parent UID correctly by using the **base** (pre-resharding) layout: [6](#0-5) 

```rust
let parent_shard_uid = ShardUId::from_shard_id_and_layout(
    *parent_shard_ids.first().unwrap(),
    &base_shard_layout,   // ← uses the layout the parent actually belongs to
);
```

`get_split_parent_shard_uids()` should do the same but has no access to the previous layout, so it silently uses the wrong version.

---

### Impact Explanation

- **Archival / cold-store nodes**: After the first dynamic resharding (V2 → V3), every cold-store `State` lookup for child shards resolves to a non-existent prefix. Historical trie queries for those shards return `MissingTrieValue`, permanently breaking archival access to post-resharding state.
- **All nodes**: The parent shard's memtrie is never preloaded. If `allow_resharding_without_memtries` is `false` (the production default), `process_memtrie_resharding_storage_update()` returns `Error::Other("Memtrie not loaded")`, causing the resharding block to fail processing and stalling the chain at the resharding boundary.

---

### Likelihood Explanation

The bug fires unconditionally on the first dynamic resharding event (V2 → V3 bootstrapping). No attacker action is required; the normal protocol upgrade process triggers it. Every node that processes the resharding boundary block is affected. The code path is in production (not nightly-only): `ShardLayout::derive_v3()`, `get_resharding_parent_shard_uid()`, and `update_state_shard_uid_mapping()` are all stable production code.

---

### Recommendation

`get_split_parent_shard_uids()` must use the **previous** layout's version, not `self.version()`. The simplest fix is to require the caller to pass the previous layout, or to add a paired function that accepts the previous layout:

```rust
pub fn get_split_parent_shard_uids_from_prev(
    &self,
    prev_layout: &ShardLayout,
) -> BTreeSet<ShardUId> {
    let parent_shard_ids = self.get_split_parent_shard_ids();
    parent_shard_ids
        .into_iter()
        .map(|shard_id| ShardUId::new(prev_layout.version(), shard_id))
        .collect()
}
```

All callers — `get_resharding_parent_shard_uid()` and `update_state_shard_uid_mapping()` — already have access to both the current and next layouts and can supply the correct previous layout.
