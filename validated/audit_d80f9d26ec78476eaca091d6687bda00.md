### Title
Deprecated `verify_transaction_inclusion` Remains Callable and Vulnerable to Merkle Pre-Image Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` function (v1) in `contract/src/lib.rs` is still publicly callable by any unprivileged NEAR account despite being marked `#[deprecated]`. It performs no validation of the Merkle proof depth, allowing an attacker to supply an internal Merkle tree node as the `tx_id` with a proof path that is one level shorter than the actual tree. `compute_root_from_merkle_proof` will still reconstruct the correct block Merkle root, causing the function to return `true` for a Bitcoin transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` in `contract/src/lib.rs` delegates to `merkle_tools::compute_root_from_merkle_proof`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

The only guard on the proof is:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` iterates over whatever proof elements are supplied without any constraint on their count:

```rust
for proof_hash in merkle_proof {
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
``` [2](#0-1) 

Because the proof length is never validated against the actual tree depth, an attacker can supply an **internal Merkle node** as `tx_id` together with a proof that is one level shorter than the real tree. The computation still reaches the correct root, so the function returns `true`.

The `#[deprecated]` Rust attribute is a **compile-time warning only**. It does not restrict runtime access. The function is `pub` and exposed as a NEAR contract method, callable by any account via a signed transaction. [3](#0-2) 

The v2 function (`verify_transaction_inclusion_v2`) was introduced specifically to mitigate this class of attack by requiring a valid coinbase proof of the same length, which implicitly anchors the tree depth. However, v1 remains reachable. [4](#0-3) 

---

### Impact Explanation

Any NEAR contract or off-chain consumer that calls `verify_transaction_inclusion` and acts on a `true` return value (e.g., releasing bridged funds, crediting a deposit, unlocking an asset) can be deceived into accepting a Bitcoin transaction that was never broadcast or confirmed. The attacker does not need to control any Bitcoin miner or any privileged NEAR role — only the ability to submit a NEAR transaction with crafted `ProofArgs`.

---

### Likelihood Explanation

The function is publicly callable with no access control beyond the `#[pause]` gate (which is inactive by default). Any NEAR account can call it. The required crafted input — an internal Merkle node hash and a shortened proof — is trivially derivable from any public Bitcoin block. The attack requires no special resources.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public contract interface entirely, or add a runtime `env::panic_str` at the top of the function body so it is unreachable at runtime. Do not rely on `#[deprecated]` to prevent on-chain calls. All callers should be migrated to `verify_transaction_inclusion_v2`.

---

### Proof of Concept

Consider a Bitcoin block with four transactions `[T0, T1, T2, T3]` and the following Merkle tree:

```
Level 2 (root): R  = SHA256d(N01 || N23)
Level 1:        N01 = SHA256d(H(T0) || H(T1))
                N23 = SHA256d(H(T2) || H(T3))
Level 0 (leaf): H(T0), H(T1), H(T2), H(T3)
```

An attacker calls `verify_transaction_inclusion` with:
- `tx_id       = N01`  ← an internal node, not a real transaction
- `tx_index    = 0`
- `merkle_proof = [N23]`  ← one element instead of two

`compute_root_from_merkle_proof(N01, 0, [N23])`:
- `position 0` is even → `current_hash = SHA256d(N01 || N23) = R`
- Loop ends; returns `R`

`R == header.block_header.merkle_root` → function returns **`true`**.

The contract has confirmed a Bitcoin transaction (`N01`) that does not exist on the Bitcoin blockchain. [5](#0-4) [6](#0-5)

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
