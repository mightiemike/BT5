### Title
Difficulty Calculation for Fork Blocks Reads Stale Mainchain Ancestor Instead of Fork Ancestor Chain, Causing Incorrect PoW Validation — (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

In all chain-specific `get_next_work_required` implementations, the difficulty interval's "first block" is resolved via `get_header_by_height`, which reads exclusively from `mainchain_height_to_header`. When a fork block is being validated, the mainchain maps have **not yet been updated** to reflect the fork's ancestor chain (the reorg happens later, inside `submit_block_header_inner`). The difficulty calculation therefore uses the wrong ancestor block — the mainchain block at that height rather than the fork's actual ancestor — producing an incorrect expected difficulty. This is the direct structural analog of the external report: a sub-function reads state that has not been settled for the current execution context.

---

### Finding Description

**Call flow for a fork block submission (non-Dogecoin):**

```
submit_blocks
  └─ submit_block_header                          (lib.rs:490)
       ├─ get_prev_header(header)                 ← correct: walks headers_pool by hash
       ├─ check_target → check_pow
       │    └─ get_next_work_required             ← reads mainchain_height_to_header  ← BUG
       └─ submit_block_header_inner
            └─ store_fork_header / reorg_chain    ← mainchain maps updated HERE, too late
```

`get_header_by_height` is implemented as:

```rust
// lib.rs:677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

It reads `mainchain_height_to_header`, which is only updated by `store_block_header` (mainchain path) or `reorg_chain`. Fork blocks are stored only in `headers_pool` via `store_fork_header`:

```rust
// lib.rs:665-667
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
```

**Bitcoin / Litecoin** (`bitcoin.rs:78-86`, `litecoin.rs:86-88`): at a difficulty adjustment boundary, `get_header_by_height(first_block_height)` is called to obtain the timestamp of the interval's first block. If the fork diverges before `first_block_height`, the mainchain block at that height is a different block than the fork's ancestor at that height.

**Dogecoin** (`dogecoin.rs:286-295`): after height 145,000 the adjustment interval is 1, so `height_first = prev_block_height − 1` for every block. The developers themselves flagged this with a `TODO`:

```rust
// dogecoin.rs:291
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

For a fork diverging at height K, the fork's block at K is stored only in `headers_pool`. When the second fork block (height K+2) is validated, `get_header_by_height(K)` returns the **mainchain** block at K, not the fork's block at K. The timestamp difference between these two blocks directly corrupts the computed `modulated_timespan` and therefore the expected `bits`.

---

### Impact Explanation

**Legitimate reorganizations silently rejected.** If the mainchain block at `height_first` has a later timestamp than the fork's block at that height, `actual_time_taken` is computed as smaller than it should be, yielding a harder expected difficulty. A legitimate relayer submitting a valid reorganization chain will have its fork blocks rejected even though they satisfy the actual Bitcoin/Dogecoin protocol difficulty. The light client becomes stuck on a stale chain, and subsequent calls to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` return results against the wrong canonical chain.

**Adversarial fork blocks accepted with under-difficulty.** If the fork's block at `height_first` has a later timestamp than the mainchain block (attacker-controlled, up to `MAX_FUTURE_BLOCK_TIME_LOCAL = 7200 s`), `actual_time_taken` is inflated, yielding an easier expected difficulty. For Dogecoin the Digishield formula caps the adjustment at `max_timespan = retarget_timespan + retarget_timespan/2`, so the maximum difficulty reduction is 1.5×. Blocks meeting this reduced difficulty are accepted into `headers_pool` with their actual (lower) chainwork. While a single such block cannot immediately trigger a reorg, a sustained stream of under-difficulty fork blocks corrupts the pool with headers that did not satisfy the real protocol rules, undermining the integrity guarantee that the light client provides to consuming contracts.

---

### Likelihood Explanation

- **Dogecoin (highest likelihood):** Every fork block at height K+2 and beyond is affected, because the per-block adjustment interval means `get_header_by_height` always looks up the block immediately before the fork's divergence point. Any relayer submitting a multi-block Dogecoin reorganization triggers this path.
- **Bitcoin / Litecoin (lower likelihood):** The bug manifests only when the fork diverges before a 2016-block difficulty adjustment boundary. Deep reorganizations are rare on mainnet but are a normal operational scenario the light client is designed to handle.

---

### Recommendation

Replace the height-based mainchain lookup with an ancestor walk that follows `prev_block_hash` links through `headers_pool`, mirroring how `get_prev_header` and `get_median_time_past` already operate:

```rust
// Instead of:
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header.time;

// Walk back from prev_block_header by blocks_to_go_back steps:
let mut cursor = prev_block_header.clone();
for _ in 0..blocks_to_go_back {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time = cursor.block_header.time;
```

This is already the pattern used by `get_median_time_past` (`utils.rs:10-25`) and by `zcash_get_next_work_required` (`zcash.rs:87-116`), which correctly walks back through `headers_pool` using `get_prev_header` rather than `get_header_by_height`.

---

### Proof of Concept

**Dogecoin — legitimate reorganization rejected:**

1. Mainchain tip is at height K+10. Mainchain block at height K has timestamp `T_main`.
2. A real Dogecoin reorganization exists: fork diverges at height K. Fork block at K has timestamp `T_fork < T_main` (e.g., 60 s earlier, a normal variance).
3. Relayer submits fork blocks K+1, K+2, … via `submit_blocks`.
4. For fork block at height K+2:
   - `prev_block_header` = fork block at K+1 (correct, from `headers_pool`).
   - `height_first = K`.
   - `get_header_by_height(K)` → mainchain block at K, timestamp `T_main`.
   - `modulated_timespan` uses `T_main` instead of `T_fork`; since `T_main > T_fork`, the computed interval is shorter → harder expected difficulty.
   - Fork block at K+2 carries the correct Dogecoin difficulty (computed from `T_fork`), which does not match the inflated expected difficulty.
   - `require!(expected_bits == block_header.bits, ...)` panics.
5. The reorganization is rejected. The light client remains on the stale mainchain. `verify_transaction_inclusion` returns `true` for transactions confirmed only on the stale chain and `false` (or panics) for transactions on the real chain.
