### Title
4-byte secp opcode dispatch bypasses `ENABLE_SECP_OPS` flag guard — (`src/chia_dialect.rs`)

### Summary

`ChiaDialect::op` contains two separate dispatch paths for secp ops. The 1-byte path (opcodes 64/65) is correctly gated behind `ENABLE_SECP_OPS`. The 4-byte path (opcodes `0x13d61f00` / `0x1c3a8f00`) calls the secp functions **unconditionally**, with no flag check at all.

### Finding Description

In `ChiaDialect::op`, when the opcode atom is 4 bytes long, the code matches against the two secp opcode values and dispatches directly: [1](#0-0) 

No `ENABLE_SECP_OPS` check is performed here. By contrast, the 1-byte dispatch path at lines 248–249 correctly guards the same functions: [2](#0-1) 

The `ENABLE_SECP_OPS` flag is defined as `0x0800` and is documented as the gate for secp opcodes 64 and 65: [3](#0-2) 

`op_secp256k1_verify` and `op_secp256r1_verify` both accept `_flags: ClvmFlags` but ignore it entirely — they perform no internal flag check: [4](#0-3) [5](#0-4) 

So the only enforcement point is in `ChiaDialect::op`, and the 4-byte branch skips it.

### Impact Explanation

An attacker-controlled CLVM program can use the 4-byte encoding `0x13d61f00` (secp256k1) or `0x1c3a8f00` (secp256r1) to invoke secp signature verification on any `ChiaDialect` instance, regardless of whether `ENABLE_SECP_OPS` is set. This violates the operator availability contract: a node running with `ClvmFlags::empty()` (or any flags combination without `ENABLE_SECP_OPS`) will successfully execute secp ops via the 4-byte path, while the 1-byte path would correctly reject them. This creates a consensus divergence vector: nodes that have not yet activated the secp hard fork (no `ENABLE_SECP_OPS`) will accept programs that should be invalid, diverging from nodes that enforce the flag correctly via the 1-byte path.

### Likelihood Explanation

The 4-byte opcodes are valid CLVM atoms that any attacker can embed in a CLVM program. No special privileges are required. The path is reachable through normal CLVM execution with attacker-controlled input.

### Recommendation

Add the `ENABLE_SECP_OPS` flag check inside the 4-byte dispatch branch, mirroring the guard already present in the 1-byte branch:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```
