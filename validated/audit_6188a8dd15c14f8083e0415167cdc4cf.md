Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual user — allowlist bypassed via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, so when a swap is routed through `MetricOmmSimpleRouter`, the `sender` seen by `SwapAllowlistExtension.beforeSwap` is the router contract address, not the originating user. Any user can bypass a per-user swap allowlist on a restricted pool by calling the router instead of the pool directly, provided the router address is itself allowlisted.

## Finding Description
**Root cause — wrong identity propagated to the hook:**

`MetricOmmPool.sol` line 231 passes `msg.sender` as `sender` to `_beforeSwap`:
```solidity
_beforeSwap(
    msg.sender,   // ← always the immediate caller of pool.swap()
    recipient, ...
);
```

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()` directly:
```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ..., params.extensionData
);
```
So `msg.sender` inside `pool.swap()` is the **router**, not the end user.

`SwapAllowlistExtension.beforeSwap` then checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`.

**Exploit path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only specific counterparties (e.g., KYC'd addresses). The router is also allowlisted so that allowlisted users can use it conveniently (`allowedSwapper[pool][router] = true`).
2. A non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
3. The router calls `pool.swap()`; pool passes `msg.sender = router` as `sender` to `_beforeSwap`.
4. `SwapAllowlistExtension` evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
5. The non-allowlisted user successfully swaps in a pool that was meant to be restricted.

**Existing guards are insufficient:** The only guard is `allowedSwapper[pool][sender]` inside the extension. There is no mechanism to unwrap the original EOA from a router call; the pool has no concept of a "real sender" beyond `msg.sender`.

## Impact Explanation
The `SwapAllowlistExtension` is the sole on-chain mechanism for restricting swap access to specific addresses per pool. Its bypass allows any unprivileged user to trade in a pool that the pool admin intended to gate. Depending on pool configuration this can mean unauthorized access to preferential pricing, circumvention of compliance/KYC restrictions, or draining of a pool that was designed to serve only specific counterparties. This is a broken core access-control invariant with direct fund-impact potential, matching the "admin-boundary break bypassed by an unprivileged path" allowed impact.

## Likelihood Explanation
The condition is reachable by any unprivileged user with no special privileges: they only need to call the public `MetricOmmSimpleRouter` functions. The only precondition is that the router address is allowlisted on the target pool, which is the natural setup whenever the pool admin wants allowlisted users to be able to use the router. The attack is repeatable every block and requires no flash loan or special timing.

## Recommendation
Pass the originating user through the call chain rather than `msg.sender`. One approach: add an optional `address swapper` parameter to `pool.swap()` that defaults to `msg.sender` when zero, and have the router forward `msg.sender` explicitly. Alternatively, `SwapAllowlistExtension` should not be used with a router that is itself allowlisted unless the pool admin accepts that all router users are implicitly allowed. The cleanest fix is for the pool to expose a `swapper` field (separate from the callback payer) that extensions can gate on, populated by the router with the originating `msg.sender`.

## Proof of Concept
```solidity
// 1. Deploy pool with SwapAllowlistExtension
// 2. Pool admin: allowedSwapper[pool][router] = true (router is allowed)
// 3. Pool admin: allowedSwapper[pool][attacker] = false (attacker is NOT allowed)
// 4. Attacker calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: restrictedPool,
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1000e18,
    ...
}));
// pool.swap() is called with msg.sender = router
// _beforeSwap(router, ...) → allowedSwapper[pool][router] = true → passes
// Attacker successfully swaps despite being individually blocked
```
Foundry test: deploy pool + `SwapAllowlistExtension`, set `allowedSwapper[pool][router]=true`, set `allowedSwapper[pool][attacker]=false`, call `router.exactInputSingle` from `attacker`, assert swap succeeds (no `NotAllowedToSwap` revert).
