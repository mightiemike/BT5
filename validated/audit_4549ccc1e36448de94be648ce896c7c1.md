### Title
`check_declare_permissions` Admission Guard Missing from Consensus Proposal Validation Path — (`crates/apollo_gateway/src/gateway.rs`)

---

### Summary

The gateway enforces two declare-transaction admission controls — `block_declare` and `authorized_declarer_accounts` — via `check_declare_permissions` in the `add_tx` path. The identical guard is entirely absent from the consensus proposal validation path. A malicious proposer can include declare transactions from unauthorized senders (or when declares are globally blocked) directly in a block proposal; the validator node will convert, execute, and vote to commit them without ever consulting the configured policy.

---

### Finding Description

**Gateway path — guard is present:**

In `GenericGateway::add_tx_inner`, before any conversion or stateless validation, the gateway calls `check_declare_permissions`:

```rust
if let RpcTransaction::Declare(ref declare_tx) = tx {
    if let Err(e) = self.check_declare_permissions(declare_tx) {
        metric_counters.record_add_tx_failure(&e);
        return Err(e);
    }
}
``` [1](#0-0) 

`check_declare_permissions` enforces two policies:

1. **`block_declare`** — if `true`, all declare transactions are rejected with `BLOCKED_TRANSACTION_TYPE`.
2. **`authorized_declarer_accounts`** — if set, only listed sender addresses may declare. [2](#0-1) 

Both flags are part of `GatewayStaticConfig` and are operator-configurable: [3](#0-2) 

**Consensus path — guard is absent:**

When a validator processes an incoming block proposal, `ProposalPart::Transactions` batches are handled in `validate_proposal.rs`. Each transaction is passed directly to `convert_consensus_tx_to_internal_consensus_tx`:

```rust
let conversion_results =
    futures::future::join_all(txs.into_iter().map(|tx| {
        transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
    }))
    .await
    ...
``` [4](#0-3) 

`convert_consensus_tx_to_internal_consensus_tx` dispatches to `convert_rpc_tx_to_internal`, which for a `Declare` transaction only validates the compiled class hash against the class manager — it never checks `block_declare` or `authorized_declarer_accounts`: [5](#0-4) [6](#0-5) 

The converted transactions are then forwarded to the batcher via `send_txs_for_proposal` and executed by the blockifier. The blockifier has no knowledge of `authorized_declarer_accounts` or `block_declare`; it executes any structurally valid declare transaction.

---

### Impact Explanation

A malicious proposer (any rotating validator in the BFT round) can craft a block proposal containing:

- A declare transaction from an address not in `authorized_declarer_accounts`, or
- Any declare transaction when `block_declare = true`.

The validator node will accept the proposal, convert the transactions, send them to the batcher, execute them via the blockifier, and vote to commit the block. The unauthorized contract class will be declared and committed to state. The `authorized_declarer_accounts` and `block_declare` operator policies are rendered ineffective for any transaction that arrives via the consensus path rather than the public gateway.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**, and potentially **Critical — Wrong state from execution logic for accepted input** (unauthorized class hash committed to the contract class trie).

---

### Likelihood Explanation

Exploiting this requires the attacker to act as a block proposer. In the BFT protocol used here, the proposer role rotates among all validators. Any validator that obtains a proposer slot can include arbitrary declare transactions in its proposal. No external user privilege is required beyond being a validator participant.

---

### Recommendation

Apply the `check_declare_permissions` logic (or an equivalent check against `block_declare` and `authorized_declarer_accounts`) inside the consensus proposal validation path, specifically when processing `ProposalPart::Transactions` containing `ConsensusTransaction::RpcTransaction(RpcTransaction::Declare(...))` variants. The check should be performed before or during `convert_consensus_tx_to_internal_consensus_tx`, mirroring the guard already present in `add_tx_inner`.

---

### Proof of Concept

1. Configure a validator node with:
   ```json
   "gateway_config.static_config.authorized_declarer_accounts": "0x1",
   "gateway_config.static_config.authorized_declarer_accounts.#is_none": false
   ``` [7](#0-6) 

2. As a malicious proposer (a validator that wins a proposer slot), construct a `ProposalPart::Transactions` batch containing a `ConsensusTransaction::RpcTransaction(RpcTransaction::Declare(...))` where `sender_address = 0x2` (not in the authorized list).

3. Stream the proposal to the target validator node.

4. The validator calls `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal`. The `check_declare_permissions` guard is never invoked. [8](#0-7) 

5. The converted transaction is forwarded to the batcher via `send_txs_for_proposal`, executed by the blockifier, and the unauthorized class hash is committed to state.

6. The validator votes to commit the block. The `authorized_declarer_accounts` policy has been bypassed entirely through the consensus path.
