### Title
`SwapAllowlistExtension` gates the router address instead of the end user, enabling allowlist bypass or breaking router-mediated swaps for authorized users - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is always `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the end user. The extension therefore gates the router's address rather than the actual trader's address, creating a direct wrong-actor binding: either the router is allowlisted (bypassing the guard for all users) or it is not (breaking the flow for legitimately allowlisted users).

---

### Finding Description

**Actor binding chain:**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```
_beforeSwap(msg.sender, recipient, ...)
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim as the first argument to every configured extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
``` [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks that forwarded `sender` against the per-pool allowlist:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [4](#0-3) 

The pool's `msg.sender` at that point is the **router contract address**. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

The same wrong-actor binding applies to multi-hop `exactInput` (intermediate hops use `address(this)` as payer, and the pool still sees the router as `msg.sender`) and `exactOutput` recursive hops. [5](#0-4) 

---

### Impact Explanation

Two fund-impacting outcomes follow from the same root cause:

**Scenario A — Allowlist bypass (High):** The pool admin allowlists the router address (the only way to make router-mediated swaps work for any user) via `setAllowedToSwap(pool, router, true)`. Because the extension checks the router's address, every user — including those the admin explicitly excluded — can now swap on the curated pool simply by routing through `MetricOmmSimpleRouter`. The entire curation policy is nullified. Disallowed users trade on a pool that was designed to exclude them, directly violating the protocol's access-control invariant and potentially draining LP value from a pool whose liquidity was provided under the assumption of a restricted counterparty set.

**Scenario B — Broken authorized flow (Medium/High):** The pool admin allowlists individual user addresses directly. Those users cannot swap through the router at all because the extension sees the router's address (not theirs) and reverts with `NotAllowedToSwap`. The only working path is a direct `pool.swap()` call, which requires the caller to implement `metricOmmSwapCallback` themselves. The router — the protocol's primary public swap surface — is effectively unusable for any allowlisted pool, breaking core swap functionality for legitimate users. [6](#0-5) 

---

### Likelihood Explanation

- The `MetricOmmSimpleRouter` is the primary public entrypoint for swaps; most users will use it rather than calling the pool directly.
- The `SwapAllowlistExtension` is a production extension explicitly designed for curated pools; any pool that deploys it and expects router-mediated swaps will hit this immediately.
- No special privileges, flash loans, or unusual token behavior are required. Any unprivileged user can trigger Scenario A by calling `exactInputSingle` on a pool where the router is allowlisted.
- The misconfiguration is forced: the admin must choose between a broken router path (Scenario B) or a fully bypassed allowlist (Scenario A). There is no correct configuration under the current code.

---

### Recommendation

The pool must forward the original end-user address as `sender` to extensions, not its own `msg.sender`. One correct approach is to have the router pass the originating user address in `callbackData` and have the pool extract and forward it. Alternatively, the pool can accept an explicit `sender` override parameter from trusted periphery contracts, or the extension can be redesigned to check `tx.origin` (with appropriate trust assumptions). The cleanest fix consistent with the existing architecture is for `MetricOmmSimpleRouter` to pass the end user's address through a dedicated field that the pool forwards to extensions as `sender`, rather than using the pool's raw `msg.sender`.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as `beforeSwap` extension.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — the only way to allow any router-mediated swap.
3. Pool admin calls `setAllowedToSwap(pool, alice, false)` (or simply never allowlists Alice).
4. Alice calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` — pool's `msg.sender` = router.
6. Pool calls `_beforeSwap(router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Alice, a disallowed user, successfully swaps on a curated pool, bypassing the allowlist entirely.

The same sequence with `exactInput` (multi-hop) or `exactOutput` produces the same bypass because all router paths call `pool.swap` with the router as `msg.sender`. [7](#0-6) [3](#0-2)
