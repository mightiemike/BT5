### Title
Fork Retarget Validation Uses Mainchain Height Index Instead of Fork Ancestor — (`contract/src/lib.rs::get_header_by_height` + `contract/src/litecoin.rs::get_next_work_required`)

---

### Summary

When `get_next_work_required` validates a fork block at a Litecoin retarget boundary, it calls `blocks_getter.get_header_by_height(first_block_height)` to retrieve the interval-tail block. The `BlocksGetter` implementation of `get_header_by_height` unconditionally resolves the height through `mainchain_height_to_header`, returning the **mainchain** block at that height rather than the fork's own ancestor. A malicious trusted relayer can craft a fork that diverges before `first_block_height`, place a fork-branch block with a manipulated timestamp at that height, and submit a retarget block whose `bits` field matches the difficulty computed from the **mainchain** ancestor's timestamp. The contract accepts the fork with an incorrect difficulty, and if the fork accumulates enough chainwork to trigger `reorg_chain`, the mainchain is replaced with a chain the source network would reject as canonical.

---

### Finding Description

**Call chain:**

```
submit_blocks (lib.rs:169)
  └─ submit_block_header (lib.rs:490)
       └─ check_target (lib.rs:518) → check_pow (litecoin.rs:19)
            └─ get_next_work_required (litecoin.rs:51)
                 └─ blocks_getter.get_header_by_height(first_block_height)  ← BUG
```

At the retarget boundary `get_next_work_required` computes `first_block_height` and fetches the interval-tail block: [1](#0-0) 

The `BlocksGetter` implementation resolves that height exclusively through the **mainchain** index: [2](#0-1) 

`mainchain_height_to_header` is only written by `store_block_header` (mainchain path) and `reorg_chain`; `store_fork_header` never touches it: [3](#0-2) 

So for any fork block at a retarget boundary whose fork diverged at or before `first_block_height`, the lookup silently returns the **mainchain** block at that height, not the fork's own ancestor. The retarget arithmetic in `calculate_next_work_required` then uses the mainchain ancestor's `time` field: [4](#0-3) 

By contrast, the min-difficulty walk (non-retarget path) uses `get_prev_header`, which follows `prev_block_hash` through `headers_pool` and correctly traverses the fork branch: [5](#0-4) 

Only the height-indexed retarget path is broken.

---

### Impact Explanation

A malicious trusted relayer submits a fork whose divergence point is at or before `first_block_height = retarget_height − 2016`. The fork's block at `first_block_height` carries a crafted timestamp `T_fork`. The mainchain block at the same height carries `T_main`. The contract computes expected `bits` using `T_main`; the attacker sets the fork retarget block's `bits` to match that computation. The contract stores the fork with a difficulty that does not correspond to the fork's own history. If the fork's chainwork exceeds the mainchain tip, `reorg_chain` promotes it: [6](#0-5) 

After promotion, `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` will confirm transactions on a chain the source network would never accept as canonical. A downstream bridge treating those confirmations as final suffers a **light client verification bypass**.

---

### Likelihood Explanation

- On Litecoin **testnet** (`pow_allow_min_difficulty_blocks = true`, very low difficulty), mining 2016+ scrypt-valid headers is computationally trivial.
- The attacker must hold trusted-relayer status (staked, approved by `RelayerManager`). This is an economic barrier, not a cryptographic one; a malicious relayer who has staked can exploit this.
- The `run_mainchain_gc` public call (callable by anyone) can advance `mainchain_initial_blockhash` between relayer submissions. If GC removes the mainchain block at `first_block_height` before the retarget block is submitted, `get_header_by_height` panics (`ERR_KEY_NOT_EXIST`), which is a secondary DoS vector but does not prevent the core exploit when GC has not yet reached that height.
- No DAO, code-deployer, or leaked-key compromise is required.

---

### Recommendation

Replace the height-indexed lookup in `get_next_work_required` with a backward traversal along the fork's own `prev_block_hash` chain. Concretely, walk from `prev_block_header` backwards `blocks_to_go_back` steps using `get_prev_header` (the same pattern already used correctly in the min-difficulty walk), rather than calling `get_header_by_height`. This ensures the interval-tail block is always the fork's own ancestor, regardless of what the mainchain holds at that height.

---

### Proof of Concept

1. Initialize the contract on Litecoin testnet with genesis at height `H` where `(H+1) % 2016 == 0`.
2. Extend the mainchain to height `H + 2016` (the next retarget boundary). Record the mainchain block at height `H` as `M_H` with timestamp `T_main`.
3. Build a fork that diverges at height `H − 1`. At height `H` on the fork, place block `F_H` with timestamp `T_fork` chosen so that `T_main − T_fork` differs significantly from `T_fork_prev − T_fork` (the fork's own interval span), yielding a different `bits` value.
4. Extend the fork to height `H + 2016` (the retarget block). Compute `expected_bits` using `T_main` (what the contract will compute) rather than `T_fork` (what the source chain would compute). Set the fork retarget block's `bits` to `expected_bits`.
5. Submit the fork blocks via `submit_blocks`. The contract calls `get_header_by_height(H)`, returns `M_H`, computes `expected_bits` from `T_main`, and accepts the fork retarget block.
6. Confirm that the stored fork retarget block carries `bits` that the Litecoin source chain would reject (because the source chain uses `T_fork`, not `T_main`).
7. Give the fork enough chainwork to trigger `reorg_chain`; observe that `mainchain_tip_blockhash` now points to the invalid fork tip, and `verify_transaction_inclusion` returns `true` for transactions on that fork.
