### Title
Missing Parent Health Check After Margin Deduction in `createIsolatedSubaccount` — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`createIsolatedSubaccount` deducts an attacker-controlled `margin` from the parent subaccount's `QUOTE_PRODUCT_ID` spot balance with no subsequent health check. A trader can sign an order encoding a margin value larger than their parent's quote balance, driving the parent's balance negative and leaving it undercollateralized.

---

### Finding Description

In `OffchainExchange.createIsolatedSubaccount`, after verifying the order signature, the function extracts `margin` from the order's `appendix` field and performs two raw balance updates: [1](#0-0) 

```
int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);   // parent drained
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin); // iso funded
}
```

The function then immediately returns `newIsolatedSubaccount` with no health assertion on the parent: [2](#0-1) 

The calling code in `EndpointTx.sol` also performs no health check — it only calls `_recordSubaccount` on the new isolated subaccount: [3](#0-2) 

The `margin` value is entirely attacker-controlled: the trader signs the `CreateIsolatedSubaccount` order themselves, and `_isolatedMargin` decodes it from `appendix` bits `[127:64]` (scaled by `1e12`). There is no cap enforced on-chain against the parent's actual balance.

Contrast this with `TransferQuote`, which routes through `clearinghouse.transferQuote` — a path that does enforce a health check on the sender: [4](#0-3) 

---

### Impact Explanation

- The parent subaccount's `QUOTE_PRODUCT_ID` balance goes negative.
- If the parent holds no other assets sufficient to offset the deficit, its initial health drops below zero.
- An undercollateralized parent is eligible for liquidation at a discount, or accumulates bad debt absorbed by the insurance fund.
- The isolated subaccount holds the excess margin, effectively transferring value out of the parent without solvency enforcement.

---

### Likelihood Explanation

The path is reachable via `Endpoint.submitSlowModeTransaction(CreateIsolatedSubaccount)`, a public function. The attacker is the trader themselves — no privileged access is required. The only prerequisite is a parent subaccount with a positive quote balance. The `appendix` field is part of the signed order struct, fully controlled by the signer. [5](#0-4) 

---

### Recommendation

After the `spotEngine.updateBalance` calls at lines 1077–1086, add an initial-health check on the parent subaccount:

```solidity
require(
    clearinghouse.getHealth(txn.order.sender, IProductEngine.HealthType.INITIAL) >= 0,
    "parent undercollateralized after margin transfer"
);
```

This mirrors the pattern already used in `transferQuote` and other balance-moving paths.

---

### Proof of Concept

1. Parent subaccount `P` holds exactly `X` USDC quote balance and no other assets.
2. Attacker (owner of `P`) crafts a `CreateIsolatedSubaccount` order for product `pid` with `appendix` bits `[127:64]` encoding `X + 1` (in the `1e12`-scaled unit), so `_isolatedMargin(appendix) = (X + 1) * 1e12`.
3. Attacker signs the order and submits via `Endpoint.submitSlowModeTransaction`.
4. `createIsolatedSubaccount` executes:
   - `spotEngine.updateBalance(QUOTE_PRODUCT_ID, P, -(X+1)*1e12)` → parent balance = `-1e12`
   - `spotEngine.updateBalance(QUOTE_PRODUCT_ID, iso, (X+1)*1e12)` → iso balance = `(X+1)*1e12`
5. No health check fires. `clearinghouse.getHealth(P, INITIAL) < 0`.
6. Parent `P` is now liquidatable. [6](#0-5)
