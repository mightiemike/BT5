### Title
Unconditional Non-Empty Merkle Proof Requirement Rejects Valid SPV Proofs for Single-Transaction Blocks — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (and by extension `verify_transaction_inclusion_v2`, which calls it) unconditionally panics when the caller supplies an empty `merkle_proof`. For a block containing exactly one transaction, the correct and complete SPV proof **is** an empty proof vector — the merkle root equals the transaction hash directly. The downstream `compute_root_from_merkle_proof` function handles this case correctly, but the preceding hard `require!` guard prevents it from ever being reached, permanently blocking verification of any transaction in a single-transaction block.

---

### Finding Description

In `contract/src/lib.rs` line 315, the guard is unconditional:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This fires before the merkle computation at lines 318–322:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

`compute_root_from_merkle_proof` is implemented as:

```rust
let mut current_hash = transaction_hash;
for proof_hash in merkle_proof { ... }
current_hash
``` [3](#0-2) 

When `merkle_proof` is empty, the loop body never executes and `current_hash` is returned unchanged — i.e., `compute_root_from_merkle_proof(tx_id, 0, &[]) == tx_id`. For a single-transaction block, `merkle_root == coinbase_txid`, so the empty-proof path is **mathematically correct**. The `require!` guard is the only thing that breaks it.

`verify_transaction_inclusion_v2` delegates to the deprecated function via `self.verify_transaction_inclusion(args.into())`, so it inherits the same defect: [4](#0-3) 

The `ProofArgsV2 → ProofArgs` conversion preserves `merkle_proof` unchanged: [5](#0-4) 

---

### Impact Explanation

Any NEAR contract or user that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to confirm a Bitcoin transaction residing in a single-transaction block will receive a hard panic. If downstream contracts gate cross-chain actions (asset releases, bridge confirmations, etc.) on this verification result, those actions are permanently unexecutable for that class of valid proofs. The contract's stated guarantee — trustless SPV verification — is broken for a well-defined, protocol-valid input class.

---

### Likelihood Explanation

Single-transaction blocks (coinbase only) occur regularly: every early Bitcoin block (heights 1–170), any block mined during low-mempool periods, and all blocks on chains with low transaction volume (Litecoin, Dogecoin, Zcash testnet). A relayer or bridge user attempting to prove inclusion of a coinbase transaction in such a block will trigger the panic deterministically and reproducibly.

---

### Recommendation

Remove the unconditional empty-proof guard, or condition it only on cases where an empty proof is genuinely invalid (i.e., when `tx_id != header.block_header.merkle_root`). The correct fix mirrors the Primex remediation: skip the strict check when the context makes it inapplicable. Concretely:

```rust
// Only require a non-empty proof when the block has more than one transaction.
// For single-transaction blocks, merkle_root == tx_id and an empty proof is valid.
if args.merkle_proof.is_empty() {
    return args.tx_id == header.block_header.merkle_root;
}
```

### Proof of Concept

1. Initialize the contract with a genesis at height 0 (Bitcoin block 0).
2. Submit Bitcoin block 1 (`00000000839a8e6886ab5951d76f411475428afc90947ee320161bbf18eb6048`), which contains exactly one transaction (the coinbase). Its `merkle_root` equals the coinbase txid `0e3e2357e806b6cdb1f70b54c3a3a17b6714ee1f0e68bebb44a74b1efd512098`.
3. Call `verify_transaction_inclusion` with:
   - `tx_id` = `0e3e2357...` (the coinbase txid)
   - `tx_block_blockhash` = block 1 hash
   - `tx_index` = `0`
   - `merkle_proof` = `[]` (correct empty proof)
   - `confirmations` = `1`
4. The contract panics at `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` despite the proof being cryptographically valid and complete. [6](#0-5)
