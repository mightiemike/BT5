### Title
Gateway `StatelessTransactionValidator::validate_resource_bounds` Skips `max_l2_gas_amount` Enforcement for `Declare` Transactions, Admitting Unbounded Gas Amounts - (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

The gateway's stateless validator enforces a `max_l2_gas_amount` ceiling on `l2_gas.max_amount` for `Invoke` and `DeployAccount` transactions but explicitly skips this check for `Declare` transactions. Any user can submit a `Declare` transaction with `l2_gas.max_amount = u64::MAX` and the gateway will admit it, violating the admission invariant that the gateway is supposed to enforce uniformly.

---

### Finding Description

In `StatelessTransactionValidator::validate_resource_bounds`, the check against `self.config.max_l2_gas_amount` is guarded by a type-branch that silently no-ops for `Declare`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The default configured ceiling is `1_210_000_000` L2 gas units: [2](#0-1) 

The test `valid_l2_gas_amount_on_declare` explicitly documents and asserts this gap — a `Declare` with `max_amount: GasAmount(200)` against a config of `max_l2_gas_amount: 100` returns `Ok(())`: [3](#0-2) 

By contrast, the same `max_amount: GasAmount(200)` on an `Invoke` or `DeployAccount` transaction is rejected with `MaxGasAmountTooHigh`: [4](#0-3) 

The blockifier does enforce a separate ceiling (`validate_max_sierra_gas`, currently `100_000_000`) at execution time, but this is a different, lower-level limit applied after admission: [5](#0-4) 

The two limits are independent: the gateway's `max_l2_gas_amount` is the admission gate; `validate_max_sierra_gas` is the execution cap. The gap between them (gateway admits up to `u64::MAX`; execution caps at `100_000_000`) is the divergent range.

---

### Impact Explanation

**High — Mempool/gateway admission accepts transactions that should be rejected before sequencing.**

A `Declare` transaction with `l2_gas.max_amount = u64::MAX` passes stateless validation unconditionally. The `max_possible_fee` computed from this field drives the stateful balance check (`verify_can_pay_committed_bounds`). When `max_price_per_unit = 0` for L2 gas (a valid configuration for client-side-proving flows), the fee contribution from L2 gas is zero regardless of `max_amount`, so the stateful check also passes. The transaction is then admitted to the mempool with a resource-bounds field that is orders of magnitude above the gateway's own policy ceiling — a field that is signed into the transaction hash and propagated to peers and the batcher.

The exact divergent value: for `Invoke`/`DeployAccount`, `l2_gas.max_amount > 1_210_000_000` → `MaxGasAmountTooHigh`; for `Declare`, `l2_gas.max_amount = 18_446_744_073_709_551_615` → `Ok(())`.

---

### Likelihood Explanation

Unprivileged. Any external caller can craft a `RpcDeclareTransactionV3` with an arbitrary `l2_gas.max_amount` and submit it to the public gateway endpoint. No special account state, key material, or peer relationship is required. The trigger is a single well-formed RPC call.

---

### Recommendation

Remove the `Declare`-specific exemption and apply the same `max_l2_gas_amount` ceiling uniformly:

```rust
// Remove the special-casing:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If `Declare` transactions legitimately require a higher ceiling (e.g., because class compilation consumes more L2 gas), introduce a separate `max_l2_gas_amount_declare` config field with an explicit, documented value rather than leaving the check absent entirely.

---

### Proof of Concept

1. Construct an `RpcDeclareTransactionV3` with:
   - `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)`
   - `resource_bounds.l2_gas.max_price_per_unit = GasPrice(0)` (zero price → zero fee contribution)
   - All other fields valid (correct Sierra class, valid signature, nonce = account nonce)
2. Submit to the gateway's `starknet_addDeclareTransaction` endpoint.
3. Observe: stateless validation returns `Ok(())` despite `u64::MAX >> 1_210_000_000`.
4. For comparison, submit the same `l2_gas.max_amount` on an `Invoke` transaction — the gateway returns `MaxGasAmountTooHigh`.

The divergence is confirmed by the existing test suite: `valid_l2_gas_amount_on_declare` asserts `Ok(())` for a `Declare` that exceeds the configured limit, while `test_invalid_max_l2_gas_amount` asserts `MaxGasAmountTooHigh` for `Invoke`/`DeployAccount` under identical conditions.
