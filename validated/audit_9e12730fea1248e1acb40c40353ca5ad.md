Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of end-user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of `MetricOmmPool.swap()`. When any user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][end_user]`. A pool admin who allowlists the router address — the only way to permit any approved user to trade through the router — simultaneously grants swap access to every unprivileged user who calls the same public router.

## Finding Description

**Step 1 — Extension checks `sender`, not the originating user:**

`SwapAllowlistExtension.beforeSwap` (L37) checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
`msg.sender` here is the pool (enforced by `onlyPool` in `BaseMetricExtension` L19-24). `sender` is the first argument forwarded from the pool.

**Step 2 — Pool passes its own `msg.sender` as `sender`:**

`MetricOmmPool.swap` (L230-231) calls:
```solidity
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    ...
);
```
`ExtensionCalling._beforeSwap` (L160-176) encodes this value as the first argument to `IMetricOmmExtensions.beforeSwap`.

**Step 3 — Router is `msg.sender` of `pool.swap()`:**

`MetricOmmSimpleRouter.exactInputSingle` (L72-80) calls `pool.swap()` directly with no originator field:
```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```
The pool receives `msg.sender = router`. It passes `router` as `sender` to the extension. The extension evaluates `allowedSwapper[pool][router]`.

**Step 4 — No mechanism exists to thread the original user through:**

The router stores `msg.sender` as the payer in transient storage (`TransientCallbackPool`, T_PAYER_SLOT) for callback settlement only. This value is never forwarded to extensions. The `extensionData` field is caller-supplied and not automatically populated with the originating user by the router.

**The operational trap:** To allow any approved user to trade through the router, the pool admin must call `setAllowedToSwap(pool, router, true)`. This single action makes `allowedSwapper[pool][router] = true`, which causes the extension to pass for every caller of the router — including those explicitly excluded from the allowlist.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set (KYC'd addresses, protocol-owned accounts, specific market makers) is fully bypassed by any user routing through `MetricOmmSimpleRouter`. Disallowed users execute real swaps against the pool's LP reserves at oracle-derived prices. LPs deposited under the assumption that only approved counterparties could trade; unauthorized swap execution constitutes direct loss of LP principal. This meets the allowed impact gate: broken core pool functionality causing loss of funds, and direct loss of user/LP principal above Sherlock thresholds.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary documented public swap entrypoint. Any pool admin who deploys a curated pool and then allowlists the router (the only way to enable router-based swaps for approved users) immediately exposes the pool. The attacker requires no special privileges, no flash loan, and no multi-step setup — a single call to `exactInputSingle` suffices. The condition (router allowlisted) is a natural and expected operational step, not an exotic misconfiguration.

## Recommendation
The extension must identify the economically relevant actor. Two approaches:

1. **Thread the originating user through the router:** Extend `TransientCallbackPool` with a dedicated originator slot (analogous to the existing payer slot). Have the router write `msg.sender` there before calling `pool.swap()`. Expose it via a standardized interface so extensions can read it during `beforeSwap`.

2. **Pass originator via `extensionData`:** Define a convention where the router encodes the originating user into `extensionData` and the extension decodes it. This is opt-in per call and requires router cooperation.

The simplest safe interim fix is to document explicitly that pools using `SwapAllowlistExtension` must not allowlist the router, and that allowlisted users must call `pool.swap()` directly. However, this is a documentation-only mitigation and does not fix the architectural mismatch.

## Proof of Concept
```
Setup:
  1. Deploy pool with SwapAllowlistExtension configured in beforeSwap order.
  2. Pool admin calls setAllowedToSwap(pool, allowedUser, true).
  3. Pool admin calls setAllowedToSwap(pool, router, true)
     (required so allowedUser can trade through MetricOmmSimpleRouter).

Attack:
  4. attacker (NOT in allowlist) calls:
       MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: attacker, ...})
  5. Router calls pool.swap() → msg.sender in pool = router address.
  6. Pool calls _beforeSwap(sender=router, ...).
  7. SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][router] → true.
  8. Swap executes; attacker receives output tokens from LP reserves.

Result:
  - attacker bypassed the allowlist entirely with a single public call.
  - LP funds were traded against an unauthorized counterparty.
  - allowedSwapper[pool][attacker] was never set to true.
```

Foundry test outline: deploy pool + extension, configure allowlist with only `allowedUser` and `router`, call `exactInputSingle` from `attacker` address, assert swap succeeds and `attacker` receives output tokens.
