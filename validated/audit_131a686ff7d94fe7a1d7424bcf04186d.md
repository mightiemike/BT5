### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash in P2P Sync — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a **value-based heuristic** to reconstruct the enum variant. When `l2_gas` and `l1_data_gas` are both zero, it always produces `ValidResourceBounds::L1Gas`, even if the original transaction carried `ValidResourceBounds::AllResources` with those fields explicitly set to zero. Because `get_tip_resource_bounds_hash` hashes a **different number of elements** depending on the variant (3 for `L1Gas`, 4 for `AllResources`), the Poseidon hash of the resource-bounds sub-tree changes after a protobuf round-trip. Any node that receives such a transaction over P2P sync stores a transaction body whose recomputed hash diverges from the canonical hash committed to the block, breaking proof generation and OS hash re-verification.

---

### Finding Description

**Root cause — value-based variant reconstruction in protobuf deserialization**

`crates/apollo_protobuf/src/converters/transaction.rs` lines 417–436:

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← always chosen when both are zero
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
``` [1](#0-0) 

The serializer (`From<ValidResourceBounds> for protobuf::ResourceBounds`) faithfully encodes both variants with explicit zero fields for `l2_gas` and `l1_data_gas`:

```rust
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),          // zero
        l1_data_gas: Some(l1_data_gas.into()), // zero
    },
``` [2](#0-1) 

After the round-trip, `AllResources { l2_gas: 0, l1_data_gas: 0 }` silently becomes `L1Gas`.

**Why the hash changes**

`get_tip_resource_bounds_hash` branches on the variant to decide how many elements to feed into the Poseidon hash chain:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 3 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 4 elements total
    }
});
Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
``` [3](#0-2) 

`poseidon_hash([tip, l1, l2])` ≠ `poseidon_hash([tip, l1, l2, l1_data])` even when `l2 = l1_data = 0`, because Poseidon is length-sensitive. The transaction hash embedded in every V3 transaction type (`get_invoke_transaction_v3_hash`, `get_declare_transaction_v3_hash`, `get_deploy_account_transaction_v3_hash`) calls this function.

**How `AllResources { l2_gas: 0, l1_data_gas: 0 }` enters storage**

The JSON/serde deserializer uses a **key-presence** rule, not a value rule:

```rust
match resource_bounds_mapping.0.get(&Resource::L1DataGas) {
    Some(data_bounds) => Ok(Self::AllResources(AllResourceBounds { ... })),  // key present → AllResources
    None => { if l2_bounds.is_zero() { Ok(Self::L1Gas(*l1_bounds)) } ... }
}
``` [4](#0-3) 

A node that syncs from the feeder gateway (central sync) deserializes a Starknet ≥ 0.13.3 transaction whose JSON body contains an explicit `"L1_DATA_GAS": {"max_amount": "0x0", ...}` key. This produces `AllResources { l2_gas: 0, l1_data_gas: 0 }` in local storage. When that node later serves the transaction over P2P block sync, the protobuf round-trip silently converts it to `L1Gas`.

**P2P sync stores the body without hash re-validation**

The P2P sync client writes the deserialized (now `L1Gas`) transaction body alongside the original (correct) hash without verifying consistency:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [5](#0-4) 

The stored pair `(L1Gas body, AllResources hash)` is internally inconsistent.

---

### Impact Explanation

When the blockifier or Starknet OS re-derives the transaction hash from the stored body (e.g., during proof generation via `starknet_transaction_prover`), it calls `get_tip_resource_bounds_hash` on the `L1Gas` variant and produces a 3-element Poseidon hash. The canonical hash in the block was computed from the `AllResources` variant (4-element hash). The two values differ, so:

- OS hash re-verification fails → proof generation aborts or produces an invalid proof.
- Any RPC call that re-derives the hash from the stored transaction body (e.g., `starknet_getTransactionByHash` with re-computation) returns a wrong hash.
- The `GasVectorComputationMode` also changes (`All` → `NoL2Gas`), altering fee-check logic and resource-bound enforcement for the affected transaction.

This matches: **Critical — Wrong state, receipt, or revert result from blockifier/execution logic for accepted input**, and **High — RPC execution or tracing returns an authoritative-looking wrong value**.

---

### Likelihood Explanation

The trigger requires a transaction with `AllResources { l2_gas: 0, l1_data_gas: 0 }` to reach storage via the feeder-gateway (central sync) path. This is a normal condition for any Starknet ≥ 0.13.3 transaction whose submitter explicitly set both optional bounds to zero. No privileged access is required; the divergence is caused entirely by the inconsistency between the JSON and protobuf deserialization rules, and is triggered automatically whenever such a transaction is re-served over P2P sync.

---

### Recommendation

Replace the value-based heuristic in the protobuf deserializer with an explicit discriminator. The protobuf schema should carry a boolean or enum field that records whether the original transaction used `L1Gas` or `AllResources` semantics, independent of the numeric values. Until the schema is updated, the deserializer should default to `AllResources` whenever `l1_data_gas` is present in the wire message (matching the JSON key-presence rule), and only produce `L1Gas` when `l1_data_gas` is absent (`None`):

```rust
let l1_data_gas_opt = value.l1_data_gas;
Ok(match l1_data_gas_opt {
    None => ValidResourceBounds::L1Gas(l1_gas),
    Some(raw) => ValidResourceBounds::AllResources(AllResourceBounds {
        l1_gas,
        l2_gas,
        l1_data_gas: raw.try_into()?,
    }),
})
```

This aligns with the JSON rule (`TryFrom<DeprecatedResourceBoundsMapping>`) and makes the round-trip lossless.

### Proof of Concept

```
1. Node A syncs block N from the feeder gateway.
   - Transaction T is a V3 Invoke with resource_bounds JSON:
       {"L1_GAS": {"max_amount": "0x100", "max_price_per_unit": "0x1"},
        "L2_GAS": {"max_amount": "0x0",   "max_price_per_unit": "0x0"},
        "L1_DATA_GAS": {"max_amount": "0x0", "max_price_per_unit": "0x0"}}
   - JSON deserializer sees L1DataGas key → stores AllResources{l1_gas=X, l2_gas=0, l1_data_gas=0}.
   - Hash H = poseidon([tip, l1_felt, l2_felt(0), l1_data_felt(0)]) — 4 elements.

2. Node B requests block N from Node A over P2P sync.
   - Node A serializes T to protobuf: l1_gas=X, l2_gas=0, l1_data_gas=0 (all present).
   - Node B deserializes: l1_data_gas.is_zero() && l2_gas.is_zero() → L1Gas(X).
   - Node B stores (L1Gas body, hash H).

3. Proof generation on Node B:
   - OS calls get_tip_resource_bounds_hash on L1Gas(X).
   - Computes H' = poseidon([tip, l1_felt, l2_felt(0)]) — 3 elements.
   - H' ≠ H → OS hash assertion fails → proof is invalid / generation aborts.

4. Alternatively, blockifier re-executes T on Node B with GasVectorComputationMode::NoL2Gas
   instead of ::All, producing different fee accounting and resource-bound checks.
```
