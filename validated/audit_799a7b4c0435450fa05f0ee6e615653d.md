### Title
Fork Retarget Validation Uses Mainchain Index Instead of Fork Branch — (`contract/src/lib.rs`, `contract/src/litecoin.rs`)

---

### Summary

`get_header_by_height` always resolves blocks through `mainchain_height_to_header`. When `get_next_work_required` calls it during fork-block validation at a retarget boundary, it fetches the **mainchain's** historical block at the lookback height, not the fork's block at that height. An attacker who controls a fork that diverges before the lookback height can craft a retarget block whose `bits` field satisfies the mainchain-anchored calculation but would be rejected by the real Litecoin network, which would use the fork's own historical block.

---

### Finding Description

**The asymmetry in `BlocksGetter`:**

`get_prev_header` correctly follows the fork chain by traversing `prev_block_hash` links through `headers_pool`: [1](#0-0) 

`get_header_by_height`, however, always resolves through the mainchain index: [2](#0-1) 

**Where the wrong block is consumed:**

In `get_next_work_required`, when `(prev_block_header.block_height + 1) % difficulty_adjustment_interval == 0`, the function computes `first_block_height` and calls `blocks_getter.get_header_by_height(first_block_height)`: [3](#0-2) 

The timestamp of the block returned at `first_block_height` is passed directly to `calculate_next_work_required` and determines the expected `bits` for the retarget block: [4](#0-3) 

**The divergence condition:**

`store_fork_header` stores fork blocks only in `headers_pool`, never in `mainchain_height_to_header`: [5](#0-4) 

So when the fork diverges at height `K < first_block_height`, the fork's block at `first_block_height` is in `headers_pool` but is **not** reachable via `mainchain_height_to_header`. The call to `get_header_by_height(first_block_height)` silently returns the mainchain block at that height instead.

**Attack construction (Litecoin testnet):**

On testnet, `pow_allow_min_difficulty_blocks` is `true`. The min-difficulty shortcut fires when: [6](#0-5) 

This lets the attacker mine all pre-retarget fork blocks at minimum difficulty (maximum target, trivial scrypt). The attack steps:

1. Fork diverges at height `K`, where `K < first_block_height = retarget_height - 2016`.
2. The fork's block at `K` carries a timestamp `T_fork` different from the mainchain's `T_main` at that height.
3. All fork blocks from `K` to `retarget_height - 1` use min-difficulty (timestamps spaced `> 2 × pow_target_spacing`).
4. At `retarget_height`, the attacker sets `bits` to the value `calculate_next_work_required` would produce using `T_main` (the mainchain's timestamp at `first_block_height`).
5. The contract's `check_pow` → `get_next_work_required` fetches the mainchain block at `first_block_height`, computes the same `bits`, and accepts the retarget block.
6. The real Litecoin network would compute `bits` using `T_fork` (the fork's block at `first_block_height`), producing a different value, and would reject the block.

If the fork accumulates more chainwork than the mainchain (feasible on testnet where the mainchain may also consist of min-difficulty blocks), `reorg_chain` promotes the fork to mainchain: [7](#0-6) 

After the reorg, `mainchain_height_to_header` and `mainchain_header_to_height` point to fork blocks with a `bits` value the real Litecoin network would reject.

---

### Impact Explanation

A downstream bridge calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` checks only that the target block hash is present in `mainchain_header_to_height`: [8](#0-7) 

After the reorg, the bridge would confirm transactions in fork blocks that the real Litecoin network considers invalid. This is a direct **light client verification bypass**: the contract's canonical chain diverges from the real chain, and finality proofs issued by the bridge are fraudulent.

---

### Likelihood Explanation

- On Litecoin testnet, `pow_allow_min_difficulty_blocks = true`, making fork block mining trivial.
- The relayer submits any scrypt-valid headers it observes on the testnet P2P network; the attacker does not need to compromise the relayer key — only to broadcast valid-looking headers.
- The only hard requirement is that the fork's cumulative chainwork exceeds the mainchain's. On a low-activity testnet where the mainchain is also composed of min-difficulty blocks, this is achievable with a modest number of blocks.
- The `trusted_relayer` gate is the primary barrier; however, the question's stated entrypoint is relayer-mediated submission, and the relayer's normal operation is to forward any chain-tip-extending headers it sees.

---

### Recommendation

Replace the height-based lookup in `get_next_work_required` with a backward traversal through `get_prev_header` starting from `prev_block_header`, walking back exactly `blocks_to_go_back` steps. This mirrors how `get_median_time_past` already correctly follows the fork branch: [9](#0-8) 

Concretely, in `get_next_work_required` replace:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
```

with a loop that calls `blocks_getter.get_prev_header` `blocks_to_go_back` times starting from `prev_block_header`. This ensures the retarget calculation always uses the block that is actually `blocks_to_go_back` steps behind the current tip **on the same branch**, regardless of whether that branch is the mainchain or a fork.

---

### Proof of Concept

```
State: mainchain heights 0..4031, retarget at 2016 and 4032.
Mainchain block at height 16 has timestamp T_main.

Attack:
1. Submit fork block at height 15 (diverges from mainchain at height 14).
   Fork block at height 16 has timestamp T_fork ≠ T_main.
   All fork blocks 15..4031 use min-difficulty (timestamps spaced 301s apart).

2. Submit fork block at height 4032 (retarget boundary).
   first_block_height = 4032 - 2016 = 2016.
   Contract calls get_header_by_height(2016) → returns mainchain block at 2016
   (timestamp T_main_2016), NOT the fork's block at 2016 (timestamp T_fork_2016).
   calculate_next_work_required uses T_main_2016 → produces bits B_main.
   Attacker sets fork block 4032's bits = B_main and mines valid scrypt PoW.
   Contract accepts the block.

3. Real Litecoin network would call get_header_by_height(2016) on the fork branch
   → gets T_fork_2016 → produces bits B_fork ≠ B_main → rejects the block.

4. Fork chainwork > mainchain chainwork (both min-difficulty, fork is longer).
   reorg_chain() fires. Contract mainchain now contains fork blocks with bits = B_main.

5. Bridge calls verify_transaction_inclusion for a tx in fork block 4032.
   mainchain_header_to_height contains fork block 4032 → proof returns true.
   Real Litecoin network rejects block 4032 → bridge accepted a fraudulent confirmation.
```

pub fn get_median_time_past(
    block_header: ExtendedHeader,
    prev_block_getter: &impl BlocksGetter,
) -> u32 {
    use btc_types::network::MEDIAN_TIME_SPAN;

    let mut median_time = [0u32; MEDIAN_TIME_SPAN];
    let mut current_header = block_header;

    for slot in &mut median_time {
        *slot = current_header.block_header.time;
        current_header = prev_block_getter.get_prev_header(&current_header.block_header);
    }

    median_time.sort_unstable();
    median_time[median_time.len() / 2]
}
