### Title
`reorg_chain` Permanently Deletes Displaced Mainchain Blocks from `headers_pool`, Corrupting Fork State and Breaking SPV Verification — (`contract/src/lib.rs`)

---

### Summary

During a chain reorganization, `reorg_chain` calls `remove_block_header` on every displaced mainchain block. `remove_block_header` removes the block from **both** `mainchain_header_to_height` and `headers_pool`. The mainchain-map removal is correct, but the `headers_pool` removal is not: displaced mainchain blocks become fork blocks after a reorg and must remain in `headers_pool`. Deleting them permanently corrupts fork state, breaks SPV verification for any transaction in a reorged-out block, and prevents the old chain from ever being re-promoted even if it later accumulates more work.

---

### Finding Description

`remove_block_header` is a shared helper used in two contexts: GC (where full deletion is correct) and reorg (where only the mainchain-map entry should be removed):

```rust
// contract/src/lib.rs:659-662
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);          // ← always deletes from pool
}
```

`reorg_chain` calls this helper in two places for displaced mainchain blocks:

**Place 1 — blocks taller than the fork tip** (lines 586–593):
```rust
for height in (fork_tip_height + 1)..=last_main_chain_block_height {
    let current_main_chain_blockhash = self
        .mainchain_height_to_header.get(&height)...;
    self.remove_block_header(&current_main_chain_blockhash); // deletes from headers_pool
    self.mainchain_height_to_header.remove(&height);
}
```

**Place 2 — blocks at the same height as fork blocks** (lines 634–636):
```rust
if let Some(current_main_chain_blockhash) = main_chain_block {
    self.remove_block_header(&current_main_chain_blockhash); // deletes from headers_pool
}
```

After the reorg completes, every block that was on the old mainchain is gone from `headers_pool`. The contract's own comment at line 632 acknowledges this: *"remove this block from the header pool and from the header → height map"* — but this is the wrong behavior for a reorg. Compare with `store_fork_header`, which only inserts into `headers_pool` without touching the mainchain maps:

```rust
// contract/src/lib.rs:665-667
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
```

The asymmetry is the bug: fork blocks are stored only in `headers_pool`, but when they become mainchain blocks and displace old mainchain blocks, those old blocks are deleted from `headers_pool` entirely instead of being demoted to fork status.

---

### Impact Explanation

**Impact: Medium**

Three concrete consequences follow from the permanent deletion:

1. **SPV verification panics for reorged-out blocks.** Both `verify_transaction_inclusion` (line 312) and `verify_transaction_inclusion_v2` (line 354) look up the block in `headers_pool` and call `env::panic_str("cannot find requested transaction block")` if it is absent. Any consumer contract that tries to verify a transaction whose block was reorged out will receive a panic rather than `false`, which can break downstream logic.

2. **Re-reorg is permanently blocked.** If the old chain later accumulates more work (e.g., more blocks are mined on it), a relayer cannot submit those blocks because `get_prev_header` (line 671–675) looks up the parent in `headers_pool` and panics with `"PrevBlockNotFound"`. The old chain can never be re-promoted, violating the "most-work" rule the contract is designed to enforce.

3. **Canonical-chain state is corrupted.** The contract's invariant is that `headers_pool` contains all ever-submitted valid headers. After a reorg, this invariant is broken: valid, previously-accepted headers are silently deleted.

---

### Likelihood Explanation

**Likelihood: Medium**

Any account registered as a trusted relayer can submit a batch of headers that forms a fork with more cumulative work than the current tip. This is a normal operational event (Bitcoin reorgs happen regularly). No privileged role, leaked key, or social engineering is required — only a valid PoW chain (or `skip_pow_verification = true` in test deployments). The trigger is a standard relayer action, not an exotic attack.

---

### Recommendation

In `reorg_chain`, do **not** call `remove_block_header` for displaced mainchain blocks. Instead, only remove the mainchain-map entries, leaving the block in `headers_pool` as a fork block:

```rust
// Place 1: blocks taller than fork tip
for height in (fork_tip_height + 1)..=last_main_chain_block_height {
    let blockhash = self.mainchain_height_to_header.get(&height)...;
    self.mainchain_header_to_height.remove(&blockhash); // demote, don't delete
    self.mainchain_height_to_header.remove(&height);
}

// Place 2: blocks displaced at the same height
if let Some(current_main_chain_blockhash) = main_chain_block {
    self.mainchain_header_to_height.remove(&current_main_chain_blockhash); // demote only
    // do NOT call self.headers_pool.remove(...)
}
```

`remove_block_header` (full deletion from both maps and pool) should remain reserved for GC (`run_mainchain_gc`), where permanent deletion is the intended behavior.

### Proof of Concept

Consider the following state before a reorg:

```
[m0] → [m1] → [m2] → [m3]   ← mainchain tip (chain_work = W)
          ↘
           [f1] → [f2] → [f3] → [f4]  ← fork (chain_work = W+1, stored in headers_pool only)
```

1. A trusted relayer calls `submit_blocks([f4])`. Since `f4.chain_work > W`, `reorg_chain` is triggered.
2. `reorg_chain` walks back from `f4` to `f1`. At each step, it calls `remove_block_header` on the displaced mainchain block (`m3`, `m2`, `m1`).
3. After the call, `headers_pool` no longer contains `m1`, `m2`, or `m3`.

**Consequence A — SPV breaks:**
```
verify_transaction_inclusion_v2({ tx_block_blockhash: m2_hash, ... })
→ headers_pool.get(&m2_hash) → None
→ env::panic_str("cannot find requested transaction block")  // panic, not false
```

**Consequence B — Re-reorg blocked:**
```
// Bitcoin network mines more blocks on the m-chain: m4, m5 (now m-chain has more work)
submit_blocks([m4])
→ submit_block_header(m4, ...)
→ get_prev_header(m4) → headers_pool.get(&m3_hash) → None
→ env::panic_str("PrevBlockNotFound")  // m3 was deleted; m-chain can never be re-promoted
```
