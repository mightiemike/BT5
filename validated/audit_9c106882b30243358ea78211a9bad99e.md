Audit Report

## Title
`PriceVelocityGuardExtension.beforeSwap()` Unchecked Squaring of `changeE18` Overflows `uint256`, Permanently Blocking All Swaps - (File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol)

## Summary

In `PriceVelocityGuardExtension.beforeSwap()`, the expression `changeE18 * changeE18` at line 69 is computed in Solidity 0.8+ checked arithmetic with no overflow guard. When the oracle mid-price moves dramatically relative to the stored reference price (e.g., a low-value token with a small `prevMid`), `changeE18` can exceed `2^128`, causing the squaring to overflow `uint256` and revert with an arithmetic panic. Because this hook sits in the `_beforeSwap` dispatch path, every subsequent `pool.swap()` call reverts until a pool admin intervenes.

## Finding Description

In `beforeSwap()`, the velocity check computes:

```solidity
uint256 changeE18 = (delta * 1e18) / uint256(prevMid);   // line 67
uint256 actualSq  = changeE18 * changeE18;                // line 69 — overflow
```

`midPrice` and `prevMid` are both `uint128` (max ≈ 3.4 × 10^38). `delta * 1e18` fits in `uint256` (max ≈ 3.4 × 10^56 < 2^256). However, `changeE18 = (delta * 1e18) / prevMid` is unbounded when `prevMid` is small. With `prevMid = 1e10` and `delta ≈ type(uint128).max`:

```
changeE18 ≈ (3.4e38 × 1e18) / 1e10 = 3.4e46
actualSq  ≈ (3.4e46)^2 = 1.16e93  >>  uint256 max (≈ 1.16e77)
```

Solidity 0.8+ checked arithmetic reverts with panic `0x11`. The state writes at lines 57–58 (`s.lastMidPriceX64 = midPrice`, `s.lastUpdateBlock = ...`) are rolled back by the revert, so `prevMid` remains at the old value. Every subsequent swap re-enters the same overflow path as long as the oracle continues to return the extreme price. No existing guard caps `changeE18` before squaring.

The hook is wired into `_callExtensionsInOrder(BEFORE_SWAP_ORDER, ...)` called by `MetricOmmPool.swap()`, so the panic propagates to every caller.

## Impact Explanation

All swaps on any pool configured with `PriceVelocityGuardExtension` are blocked. LPs cannot rebalance, traders cannot execute, and protocol fees stop accruing. LP positions are exposed to unhedged oracle price risk with no ability to exit via swap. This matches the allowed impact: **broken core pool functionality causing unusable swap flows**. Recovery requires the pool admin to call `setMaxChangePerBlock(pool, 0)` or `setLastMidPrice(pool, currentPrice)`, neither of which has a timelock, making the outage window proportional to admin response time.

## Likelihood Explanation

The overflow requires `changeE18 > sqrt(uint256.max) ≈ 1.16e38`, i.e., `delta / prevMid > 1.16e20`. This is reachable when a pool's token pair has a very low Q64.64 price (small `prevMid`) and the oracle delivers a large price jump in a single block (e.g., after a listing event, depeg, or large market move). No privileged action is required: any unprivileged user calling `pool.swap()` when the oracle mid-price satisfies the overflow condition triggers the panic. The condition can persist across blocks if the oracle price remains extreme.

## Recommendation

Cap `changeE18` before squaring, or use `Math.mulDiv` with an explicit overflow guard:

```solidity
// Guard before squaring:
if (changeE18 > type(uint128).max) {
    revert PriceVelocityExceeded(type(uint256).max, allowedSq);
}
uint256 actualSq = changeE18 * changeE18; // safe: changeE18 <= 2^128
```

Alternatively, compare without squaring using `Math.mulDiv` to keep all intermediate products within `uint256`.
