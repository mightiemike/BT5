### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Proof Check Enforced by v2 — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof. However, the original `verify_transaction_inclusion` (v1) is still a live, publicly callable contract method. Any unprivileged NEAR caller can invoke v1 directly, skipping the coinbase proof check entirely, and obtain a `true` SPV verification result for a non-existent Bitcoin transaction.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces two guards that v1 does not:

1. It requires `merkle_proof.len() == coinbase_merkle_proof.len()`, fixing the proof depth.
2. It validates that the coinbase transaction (index 0) hashes to the block's `merkle_root` at that depth before proceeding. [1](#0-0) 

Only after both checks pass does v2 delegate to v1 via `self.verify_transaction_inclusion(args.into())`. [2](#0-1) 

v1, however, performs no coinbase check. Its only guard is that `merkle_proof` is non-empty, then it directly computes the Merkle root from the caller-supplied `tx_id` and compares it to the block header's `merkle_root`: [3](#0-2) 

The `#[deprecated]` Rust attribute is a compile-time warning only — it imposes no runtime restriction. The function remains a fully reachable public contract entry point, gated only by `#[pause]` (which is inactive by default). [4](#0-3) 

**Exploit path (64-byte internal-node forgery):**

1. The attacker identifies a real mainchain block `B` with a known `merkle_root`.
2. They construct a `tx_id` that is actually the hash of an internal Merkle tree node (a 64-byte concatenation of two child hashes). Because Bitcoin's Merkle tree uses the same double-SHA256 for both leaf and internal nodes, such a hash can be made to satisfy a valid Merkle path to `merkle_root`.
3. They call `verify_transaction_inclusion` directly (v1) with the crafted `tx_id`, a valid `tx_block_blockhash` for block `B`, and a Merkle proof that routes the internal node to `merkle_root`.
4. v1 returns `true`.

A downstream bridge or application contract that calls v1 to gate fund releases will accept this forged proof and release assets for a Bitcoin transaction that never existed.

The `ProofArgsV2 → ProofArgs` conversion used inside v2 silently drops `coinbase_tx_id` and `coinbase_merkle_proof`, confirming that v1 has no awareness of the coinbase guard at all: [5](#0-4) 

---

### Impact Explanation

Any downstream contract or user that calls `verify_transaction_inclusion` (v1) — whether by choice or because they integrated before v2 existed — can be fed a forged SPV proof. The attacker receives a `true` result for a Bitcoin transaction that does not exist, enabling unauthorized asset releases from any bridge or escrow that trusts this result. This is a direct integrity failure in the core trust model of the light client.

---

### Likelihood Explanation

The entry point is fully public and requires no privileged role. The 64-byte Merkle forgery technique is well-documented (referenced in the v2 deprecation notice itself). Any integrator that has not migrated to v2 — or any attacker who deliberately targets v1 — can trigger this with a single crafted NEAR transaction. The risk is elevated because v1 is not removed, only deprecated, giving integrators a false sense that it is still safe for some use cases.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the contract entirely, or add a hard `env::panic_str` body that unconditionally reverts with a clear message directing callers to v2. A compile-time `#[deprecated]` attribute is insufficient as a security boundary for an on-chain method. If backward compatibility must be preserved, replicate the coinbase proof check inside v1 so both paths enforce the same invariant.

---

### Proof of Concept

```
// Attacker constructs:
// - real_block_hash: a block hash present in mainchain_header_to_height
// - internal_node_txid: H256 = SHA256d(left_child || right_child) for some
//   internal node in that block's Merkle tree
// - merkle_proof: the sibling path from that internal node to merkle_root
// - tx_index: the index corresponding to the internal node's position

let args = ProofArgs {
    tx_id: internal_node_txid,
    tx_block_blockhash: real_block_hash,
    tx_index: crafted_index,
    merkle_proof: crafted_path,   // routes internal_node_txid → merkle_root
    confirmations: 1,
};

// Direct call to v1 — no coinbase check, returns true
let result = contract.verify_transaction_inclusion(args);
assert!(result); // true, despite no real transaction
```

The coinbase guard in v2 would have rejected this because the coinbase proof (depth fixed to `merkle_proof.len()`) would not validate an internal node at an arbitrary position. v1 has no such guard. [6](#0-5)

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

                0usize,

            "Incorrect coinbase merkle proof"
