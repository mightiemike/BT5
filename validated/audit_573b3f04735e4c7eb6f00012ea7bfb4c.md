### Title
`MAX_FUTURE_BLOCK_TIME_MTP` Is 24× Smaller Than Its Documented Zcash Protocol Value, Causing Valid Block Rejection — (`File: btc-types/src/network.rs`)

---

### Summary

The constant `MAX_FUTURE_BLOCK_TIME_MTP` is hardcoded to `90 * 60` (5 400 seconds / 90 minutes), but the inline comment at the only call-site explicitly states the correct Zcash protocol value is **129 600 seconds (36 hours)**. The 24× discrepancy causes the Zcash light-client build to reject any valid Zcash block whose timestamp falls between 90 minutes and 36 hours ahead of the previous block's median-time-past (MTP), a range that is fully legal under Zcash consensus rules. The light client can therefore permanently diverge from the real Zcash chain, making all subsequent `verify_transaction_inclusion` calls unreliable.

---

### Finding Description

`btc-types/src/network.rs` defines two future-timestamp guards:

```
pub const MAX_FUTURE_BLOCK_TIME_MTP:   u32 = 90 * 60;      // 5 400 s  ← wrong
pub const MAX_FUTURE_BLOCK_TIME_LOCAL: u32 = 2 * 60 * 60;  // 7 200 s  ← correct
``` [1](#0-0) 

`contract/src/zcash.rs` imports `MAX_FUTURE_BLOCK_TIME_MTP` and applies it inside `check_pow`, which is called for every submitted Zcash header:

```rust
// MAX_FUTURE_BLOCK_TIME_MTP is typically 129600 seconds (36 hours) in Zcash
require!(
    block_header.time
        <= next_work_result.prev_block_median_time_past + MAX_FUTURE_BLOCK_TIME_MTP,
    "time-too-far-ahead-of-mtp: block timestamp is too far ahead of median-time-past"
);
``` [2](#0-1) 

The comment is the developer's own acknowledgement of the correct Zcash value. The constant is 24× too small. Any Zcash block with `timestamp ∈ (prev_mtp + 5 400, prev_mtp + 129 600]` is valid on the real Zcash network but will be unconditionally rejected by this contract.

`check_pow` is reached from `submit_block_header` (non-dogecoin path), which is called by the public `submit_blocks` entry point: [3](#0-2) 

---

### Impact Explanation

When the real Zcash chain produces a block whose timestamp is more than 90 minutes but less than 36 hours ahead of the previous MTP (a condition that is valid per Zcash consensus), the contract panics with `"time-too-far-ahead-of-mtp"`. The `mainchain_tip_blockhash` is never updated to that block. The light client is now behind the real chain tip. Every subsequent block builds on the rejected block, so the relayer cannot advance the contract state at all until a block with a "close enough" timestamp appears — which may never happen if the chain has moved on.

Downstream callers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will receive `false` (or a panic) for any transaction confirmed in or after the rejected block, because `mainchain_header_to_height` will not contain those block hashes. [4](#0-3) 

---

### Likelihood Explanation

Zcash's 36-hour MTP window exists precisely because real Zcash blocks do occasionally carry timestamps significantly ahead of MTP (e.g., during low-hashrate periods or after network partitions). Any such naturally occurring block submitted by the relayer triggers the bug without any attacker involvement. An adversarial proof-submitter can also deliberately craft a valid Zcash block header (with a timestamp in the 90-min–36-hour window) and submit it to freeze the light client.

---

### Recommendation

Change the constant to match the Zcash protocol value documented in the call-site comment:

```rust
// btc-types/src/network.rs
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 129_600; // 36 hours, per Zcash protocol
```

Alternatively, if 90 minutes is intentionally more restrictive, remove the misleading comment and document the deliberate deviation from the Zcash specification.

---

### Proof of Concept

1. Deploy the Zcash build of the contract.
2. Advance the chain to a state where `prev_mtp` is known (e.g., `prev_mtp = T`).
3. Submit a Zcash block header with `time = T + 6_000` (100 minutes ahead of MTP — valid on Zcash, since `6000 ≤ 129600`, but invalid here since `6000 > 5400`).
4. The contract panics: `"time-too-far-ahead-of-mtp: block timestamp is too far ahead of median-time-past"`.
5. `mainchain_tip_blockhash` is not updated; the light client is now permanently behind the real chain.
6. Calling `verify_transaction_inclusion` for any transaction in that block returns a panic (`"block does not belong to the current main chain"`), breaking all downstream consumers. [5](#0-4) [2](#0-1)
