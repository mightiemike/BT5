### Title
Child Shard `ChunkExtra.proposed_split` Inherits Parent's Non-`None` Value During Resharding, Causing `InvalidChunkHeaderShardSplit` on First Chunk - (File: chain/chain/src/resharding/manager.rs)

### Summary
During dynamic resharding, `process_memtrie_resharding_storage_update` creates each child shard's `ChunkExtra` by cloning the parent's `ChunkExtra` without resetting the `proposed_split` field. The parent's last `ChunkExtra` carries `proposed_split = Some(TrieSplit{...})` because the split was proposed near the epoch boundary. The child shard's first chunk in the new epoch freshly computes `proposed_split = None` (the epoch just started, so `is_next_block_possibly_last_in_epoch` returns `false`). The invariant enforced by `validate_chunk_with_chunk_extra_and_receipts_root` — that `prev_chunk_extra.proposed_split == chunk_header.proposed_split` — is broken, producing `InvalidChunkHeaderShardSplit` and halting the child shard.

### Finding Description

In `process_memtrie_resharding_storage_update`, the child `ChunkExtra` is built by cloning the parent and updating only `state_root` and `congestion_info`:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// proposed_split is NOT reset — it inherits Some(TrieSplit{...}) from parent
``` [1](#0-0) 

The parent shard's final `ChunkExtra` (at the resharding boundary block) has `proposed_split = Some(TrieSplit{...})` because `compute_proposed_split` ran near the epoch end and found the shard above threshold. This value is silently carried into both child shards' initial `ChunkExtra`. [2](#0-1) 

When the child shard's first chunk is produced in epoch N+1, `compute_proposed_split` is called:

```rust
if !self.epoch_manager.is_next_block_possibly_last_in_epoch(height, prev_block_hash)? {
    return Ok(None);
}
``` [3](#0-2) 

Because the epoch just started, `is_next_block_possibly_last_in_epoch` returns `false`, so `compute_proposed_split` returns `None`. The chunk header therefore carries `proposed_split = None`.

Chunk validation then compares:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [4](#0-3) 

`prev_chunk_extra.proposed_split = Some(TrieSplit{...})` ≠ `chunk_header.proposed_split = None` → `InvalidChunkHeaderShardSplit`.

The codebase's own documentation explicitly records this as an open defect:

> **`chain/chain/src/resharding/manager.rs:249`** — The resharding manager doesn't set all `ChunkExtra` fields (notably the new `proposed_split` field). [5](#0-4) 

The `DynamicReshardingConfig` comment about `min_epochs_between_resharding > 0` describes the same symptom but frames it as a back-to-back resharding hazard; the root cause — the uncleared `proposed_split` in the child `ChunkExtra` — is present after every single resharding event: [6](#0-5) 

The `proposed_split` field exists only in `ChunkExtraV5`, which is the only variant constructed when `DynamicResharding` is enabled: [7](#0-6) 

### Impact Explanation

When `ProtocolFeature::DynamicResharding` is active and any shard split executes, every child shard's first chunk will be rejected by all validating nodes with `InvalidChunkHeaderShardSplit`. Because the child shard's chunk producer cannot produce a chunk that passes validation, the child shard stalls. Since the new epoch's shard layout requires the child shards to produce chunks, this constitutes a chain halt for all accounts assigned to the split shard. The divergent Borsh value is exact: `Some(TrieSplit { boundary_account, left_memory, right_memory })` in `ChunkExtraV5.proposed_split` vs. `None` in `ShardChunkHeaderInnerV5.proposed_split`.

### Likelihood Explanation

`ProtocolFeature::DynamicResharding` is currently a nightly feature (not yet on mainnet). However, the defective code path is unconditionally present in production binaries and will trigger on the very first resharding event after the feature is enabled. No special attacker action is required — the bug fires automatically as part of normal epoch-boundary processing. The `min_epochs_between_resharding: NonZeroEpochHeight` constraint does not prevent the bug; it only prevents a second resharding from being scheduled before the first one's child shards stabilize, but the first child shard's `ChunkExtra` is already corrupted.

### Recommendation

In `process_memtrie_resharding_storage_update`, after cloning the parent `ChunkExtra`, explicitly clear `proposed_split` on the child:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
*child_chunk_extra.proposed_split_mut() = None; // child shard starts fresh
```

This mirrors the fix suggested in the external report's "single transaction" approach: the `proposed_split` value must be computed fresh for the child shard's own context, not inherited from the parent's boundary-block context.

### Proof of Concept

1. Build nearcore with `--features nightly` (enables `DynamicResharding`).
2. Configure `DynamicReshardingConfig` with a low `memory_usage_threshold` so a shard crosses the threshold within a few epochs.
3. Run the network until `compute_proposed_split` returns `Some(TrieSplit{...})` for a shard near an epoch boundary; confirm `ChunkExtra.proposed_split` is set.
4. Observe the epoch boundary: `process_memtrie_resharding_storage_update` clones the parent `ChunkExtra` into both child shards, preserving `proposed_split = Some(...)`.
5. In the next epoch, the child shard's chunk producer calls `compute_proposed_split`; `is_next_block_possibly_last_in_epoch` returns `false`; the function returns `None`; the chunk header carries `proposed_split = None`.
6. `validate_chunk_with_chunk_extra_and_receipts_root` fires `InvalidChunkHeaderShardSplit` because `prev_chunk_extra.proposed_split = Some(...)` ≠ `None`.
7. All nodes reject the child shard's first chunk; the child shard stalls.
