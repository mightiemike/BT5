### Title
`op_multiply` Undercharges Cost When Intermediate Results Have High Bit Set — (`File: src/more_ops.rs`)

### Summary

In `op_multiply`, after each multiplication step the running byte-length tracker `l0` is updated via `limbs_for_int(&total)`, which computes `total.bits().div_ceil(8)` — the magnitude-only byte count. For any positive intermediate result whose highest bit is set (e.g. 128, 255, 32768, …), the CLVM atom encoding requires one extra leading `0x00` sign byte, so the actual atom length is `limbs_for_int + 1`. Because `l0` is underestimated by 1, every subsequent multiplication step is cost-charged using a smaller operand size than the allocator actually stores, and the 1024-byte size-limit guard uses the same underestimated value.

### Finding Description

`limbs_for_int` is defined at line 100–102:

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

`Number::bits()` returns the number of bits in the **magnitude** (absolute value). For a positive number whose magnitude fills exactly N bytes but whose top bit is 1 (e.g. 128 = `0b10000000`), `bits()` = 8, so `limbs_for_int` = 1. However, `a.new_number(128)` stores the atom as `[0x00, 0x80]` (2 bytes) because CLVM uses big-endian two's-complement and a bare `0x80` would be interpreted as −128.

In `op_multiply` (lines 596–655), after each multiplication the accumulator size is updated:

```rust
l0 = limbs_for_int(&total);   // line 649
if l0 > 1024 {                // line 650
    return Err(...);
}
```

`l0` is then used in the **next** iteration's cost formula:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;          // line 615/623
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;  // line 616/624
```

When `total` is a positive number with the high bit set, `l0` is 1 less than the actual atom length. The cost charged for the next step is therefore computed against a smaller operand than what the allocator holds, undercharging by up to `MUL_LINEAR_COST_PER_BYTE * 1 + l0 / MUL_SQUARE_COST_PER_BYTE_DIVIDER` cost units per step. The size-limit guard `l0 > 1024` is also evaluated against the underestimated value, so a 1025-byte intermediate result passes through unchecked.

The mismatch is structurally identical to H-01: a quantity is **consumed** (cost charged, size checked) based on the *requested/computed* value (`limbs_for_int`), while the *actual* resource used (atom bytes in the allocator) is larger. The consumed amount is never corrected to match the actual output.

The project's own test confirms the discrepancy:

```rust
// limb_test_helper(&[0x00, 0xff]):
// bytes[0] == 0  →  expected = bytes.len() - 1 = 1
// but actual atom length = 2
assert_eq!(limbs_for_int(&bigint), expected);  // passes, but actual atom = 2 bytes
```

### Impact Explanation

- **Cost undercharging**: An attacker can craft a multi-argument `*` expression whose intermediate products repeatedly land on positive values with the high bit set (e.g. 128, 32768, 8388608 …). Each such step causes the next step's cost to be computed 1 byte short. Over a long chain this accumulates, allowing more computation than the declared cost budget permits.
- **Size-limit bypass**: The guard `if l0 > 1024` uses the underestimated `l0`, so a 1025-byte intermediate result is not rejected. This is a 1-byte bypass of the intended operand-size cap.
- **Consensus divergence**: Any implementation that tracks `l0` using the actual atom length (e.g. a Python reference implementation using `len(number_to_bytes(total))`) will compute a different cost for the same program, producing a consensus split on cost-boundary transactions.

### Likelihood Explanation

The trigger is straightforward: supply a CLVM program containing `(* A B C …)` where intermediate products are positive numbers with the high bit set. This is trivially constructable from attacker-controlled CLVM bytes and requires no special privileges. The condition (high bit set on a positive intermediate result) occurs for roughly half of all possible intermediate values, so it is not a rare edge case.

### Recommendation

Replace `limbs_for_int(&total)` with the actual atom length of the newly allocated node:

```rust
let total_node = a.new_number(total)?;
l0 = a.atom_len(total_node);   // actual CLVM-encoded byte length
if l0 > 1024 { return Err(...); }
// use total_node directly in the next iteration
```

This ensures the cost formula and size guard both operate on the same byte count that the allocator stores, eliminating the mismatch.
