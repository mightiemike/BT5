### Title
No Slippage Protection in `mintNlp`/`burnNlp` — Oracle Price Not Committed to by User Signature - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` transaction structs that users sign do not include a minimum output amount. The `oraclePriceX18` that determines how many NLP tokens are minted or how much quote is returned is provided by the sequencer and is **excluded from the EIP-712 digest the user signs**. Users have no on-chain slippage protection and may receive significantly fewer tokens than expected due to oracle price movement between signing and execution.

---

### Finding Description

In `Clearinghouse.sol`, `mintNlp` computes the NLP amount as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
```

And `burnNlp` computes the quote returned as:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
``` [1](#0-0) [2](#0-1) 

The `oraclePriceX18` is a field in `SignedMintNlp` and `SignedBurnNlp`, appended by the sequencer at submission time:

```solidity
struct SignedMintNlp {
    MintNlp tx;
    bytes signature;
    int128 oraclePriceX18;       // NOT signed by user
    int128[] nlpPoolRebalanceX18;
}
``` [3](#0-2) [4](#0-3) 

In `Verifier.sol`, the EIP-712 digest for `MintNlp` is computed over only `(sender, quoteAmount, nonce)` — `oraclePriceX18` is **not included**:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(MINT_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.quoteAmount,
        signedTx.tx.nonce          // oraclePriceX18 absent
    )
);
``` [5](#0-4) 

Similarly for `BurnNlp`, only `(sender, nlpAmount, nonce)` is signed: [6](#0-5) 

The `MintNlp` struct itself contains no `minNlpOut` field, and `BurnNlp` contains no `minQuoteOut` field:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint64 nonce;
}
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64 nonce;
}
``` [7](#0-6) 

Because `oraclePriceX18` is not committed to by the user's signature, the user cannot enforce any minimum output. The sequencer submits the price at execution time, which may differ materially from the price at signing time.

---

### Impact Explanation

**Impact: Medium.**

- For `mintNlp`: a user pays a fixed `quoteAmount` but receives `quoteAmount / oraclePriceX18` NLP tokens. If the oracle price rises between signing and execution, the user receives fewer NLP tokens than expected with no on-chain recourse.
- For `burnNlp`: a user burns a fixed `nlpAmount` but receives `nlpAmount * oraclePriceX18` quote (minus fee). If the oracle price falls between signing and execution, the user receives less quote than expected.

The corrupted state delta is the user's NLP balance (mint) or quote balance (burn) — both are direct asset losses with no minimum bound enforced. [8](#0-7) [9](#0-8) 

---

### Likelihood Explanation

**Likelihood: Medium.**

Oracle prices for NLP fluctuate continuously. There is inherent latency between when a user signs a `MintNlp`/`BurnNlp` transaction and when the sequencer processes and submits it on-chain. During volatile market conditions, the price difference can be significant. No special attacker capability is required — this is a structural gap that affects every user of the NLP mint/burn flow.

---

### Recommendation

1. Add a `minNlpOut` field to `MintNlp` and a `minQuoteOut` field to `BurnNlp`.
2. Include these fields in the EIP-712 type string and digest computation in `Verifier.sol`.
3. In `Clearinghouse.mintNlp`, assert `nlpAmount >= txn.minNlpOut`.
4. In `Clearinghouse.burnNlp`, assert `quoteAmount >= txn.minQuoteOut`.

---

### Proof of Concept

1. User signs `MintNlp{sender, quoteAmount=1000e18, nonce=N}` when NLP oracle price is `$1.00`, expecting ~1000 NLP tokens.
2. Oracle price rises to `$1.20` before the sequencer processes the transaction.
3. Sequencer submits `SignedMintNlp` with `oraclePriceX18 = 1.20e18`.
4. `mintNlp` executes: `nlpAmount = 1000e18 / 1.20e18 ≈ 833` NLP tokens — ~17% less than expected.
5. No on-chain check prevents this outcome; the user's signature is valid because `oraclePriceX18` was never part of the signed digest. [10](#0-9) [1](#0-0)
