### Title
Missing Timeout Validation in `MsgTransfer` Allows NFT Packets With No Expiry - (File: x/nft-transfer/types/msgs.go)

### Summary

`MsgTransfer.ValidateBasic()` in the `x/nft-transfer` module does not enforce that at least one of `TimeoutHeight` or `TimeoutTimestamp` is non-zero. A user can submit an IBC NFT transfer with both timeout fields set to zero, creating a packet that never expires. The NFT is immediately escrowed on the source chain, but the packet can be relayed at any future time — or never — giving a relayer full discretion over when (or whether) the transfer completes.

### Finding Description

`ValidateBasic` in `x/nft-transfer/types/msgs.go` validates port, channel, class ID, token IDs, sender, and receiver, but contains no check that at least one of `TimeoutHeight` or `TimeoutTimestamp` is non-zero: [1](#0-0) 

The comment at line 51 explicitly documents this as intentional: *"timeout height or timestamp values can be 0 to disable the timeout."* Neither `SendTransfer` in `keeper/relay.go` nor the `Transfer` handler in `keeper/msg_server.go` adds any such guard: [2](#0-1) [3](#0-2) 

The CLI enforces a non-zero timestamp only in the relative-timeout path (`!absoluteTimeouts`). When `--absolute-timeouts` is passed, no such check exists, so a caller can supply `--packet-timeout-height 0-0 --packet-timeout-timestamp 0` and produce a zero-timeout packet: [4](#0-3) 

The IBC core `SendPacket` call at line 89 of `relay.go` accepts both zeros without error, because the IBC protocol treats zero as "timeout disabled": [5](#0-4) 

### Impact Explanation

When `MsgTransfer` is executed with both timeouts zero:

1. The NFT is immediately escrowed (or burned for IBC-originated tokens) on the source chain.
2. The resulting IBC packet carries no expiry, so it remains valid indefinitely in the IBC commitment store.
3. A relayer can choose to relay the packet at any future time — including after the sender has changed their mind, after the receiver address has been compromised, or after the NFT's value has changed significantly.
4. If the packet is never relayed, the NFT is permanently locked in the module escrow account with no timeout to trigger `OnTimeoutPacket` and no refund path: [6](#0-5) 

The corrupted value is the NFT owner's asset: the NFT is removed from the owner's account at send time and either permanently locked in escrow or delivered to the receiver at an arbitrary future time chosen by the relayer, not the sender.

### Likelihood Explanation

Any NFT owner can trigger this by submitting `MsgTransfer` with both timeout fields zero. The `--absolute-timeouts` CLI flag provides a direct path. Programmatic callers (e.g., smart contracts or scripts using the protobuf message directly) can also omit both fields, since proto3 defaults both to zero. The likelihood is low-to-medium: it requires a user to deviate from the default CLI behavior (which sets a 10-minute relative timestamp), but the protocol offers no on-chain guard to prevent it.

### Recommendation

Add a validation in `MsgTransfer.ValidateBasic()` that rejects messages where both `TimeoutHeight` is zero and `TimeoutTimestamp` is zero:

```go
if msg.TimeoutHeight.IsZero() && msg.TimeoutTimestamp == 0 {
    return sdkerrors.Wrap(ErrInvalidTimeout,
        "both timeout_height and timeout_timestamp cannot be zero")
}
```

This check should be placed in `x/nft-transfer/types/msgs.go` inside `ValidateBasic`, mirroring the guard that the ICS-20 fungible token transfer module applies in ibc-go.

### Proof of Concept

1. Alice owns NFT `(classID="dragons", tokenID="001")` on the source chain.
2. Alice submits:
   ```
   chaind tx nft-transfer transfer nft channel-0 <receiver> dragons 001 \
     --absolute-timeouts \
     --packet-timeout-height 0-0 \
     --packet-timeout-timestamp 0 \
     --from alice
   ```
3. `MsgTransfer.ValidateBasic()` passes — no timeout check exists.
4. `SendTransfer` escrows the NFT in the module account immediately.
5. The IBC packet is committed with `TimeoutHeight={0,0}` and `TimeoutTimestamp=0`.
6. The relayer holds the packet. Alice's NFT is gone from her account. She cannot recover it because `OnTimeoutPacket` can never be triggered (no timeout condition is ever met).
7. The relayer relays the packet months later (or never), with Alice having no recourse. [7](#0-6) [4](#0-3)

// ValidateBasic performs a basic check of the MsgTransfer fields.
// NOTE: timeout height or timestamp values can be 0 to disable the timeout.
// NOTE: The recipient addresses format is not validated as the format defined by
// the chain is not known to IBC.
func (msg MsgTransfer) ValidateBasic() error {
	if err := host.PortIdentifierValidator(msg.SourcePort); err != nil {
		return newsdkerrors.Wrap(err, "invalid source port ID")
	}
	if msg.SourcePort != PortID {
		return newsdkerrors.Wrapf(ErrInvalidSourcePort, "source port must be %q", PortID)
	}
	if err := host.ChannelIdentifierValidator(msg.SourceChannel); err != nil {
		return newsdkerrors.Wrap(err, "invalid source channel ID")
	}

	if strings.TrimSpace(msg.ClassId) == "" {
		return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
	}

	if len(msg.TokenIds) == 0 {
		return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
	}

	for _, tokenID := range msg.TokenIds {
		if strings.TrimSpace(tokenID) == "" {
			return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
		}
	}

	// NOTE: sender format must be validated as it is required by the GetSigners function.
	_, err := sdk.AccAddressFromBech32(msg.Sender)
	if err != nil {
		return newsdkerrors.Wrapf(sdkerrors.ErrInvalidAddress, "string could not be parsed as address: %v", err)
	}
	if strings.TrimSpace(msg.Receiver) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "missing recipient address")
	}
	return nil
}

func (k Keeper) SendTransfer(
	ctx sdk.Context,
	sourcePort,
	sourceChannel,
	classID string,
	tokenIDs []string,
	sender sdk.AccAddress,
	receiver string,
	timeoutHeight clienttypes.Height,
	timeoutTimestamp uint64,
) error {
	if sourcePort != types.PortID {
		return sdkerrors.Wrapf(types.ErrInvalidSourcePort, "source port must be %q", types.PortID)
	}

	sourceChannelEnd, found := k.channelKeeper.GetChannel(ctx, sourcePort, sourceChannel)
	if !found {
		return sdkerrors.Wrapf(channeltypes.ErrChannelNotFound, "port ID (%s) channel ID (%s)", sourcePort, sourceChannel)
	}

	destinationPort := sourceChannelEnd.Counterparty.PortId
	destinationChannel := sourceChannelEnd.Counterparty.ChannelId

	// get the next sequence
	sequence, found := k.channelKeeper.GetNextSequenceSend(ctx, sourcePort, sourceChannel)
	if !found {
		return sdkerrors.Wrapf(
			channeltypes.ErrSequenceSendNotFound,
			"source port: %s, source channel: %s", sourcePort, sourceChannel,
		)
	}

	// See spec for this logic: https://github.com/cosmos/ibc/blob/master/spec/app/ics-721-nft-transfer/README.md#packet-relay
	packet, err := k.createOutgoingPacket(ctx,
		sourcePort,
		sourceChannel,
		destinationPort,
		destinationChannel,
		classID,
		tokenIDs,
		sender,
		receiver,
		sequence,
		timeoutHeight,
		timeoutTimestamp,
	)
	if err != nil {
		return err
	}

	if _, err := k.ics4Wrapper.SendPacket(ctx, sourcePort, sourceChannel, timeoutHeight, timeoutTimestamp, packet.GetData()); err != nil {

	}
