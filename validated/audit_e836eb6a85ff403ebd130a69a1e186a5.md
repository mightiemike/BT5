### Title
Unprivileged Caller Can Arbitrarily Prune Mainchain Headers via Missing Access Control on `run_mainchain_gc` - (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public, state-mutating NEAR contract method protected only by a pause-gate, with no caller restriction. Any unprivileged NEAR account can invoke it with an attacker-controlled `batch_size`, immediately removing all excess block headers from the canonical chain state and corrupting `mainchain_initial_blockhash`. This causes `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to panic for any transaction whose containing block was pruned, invalidating SPV proof results for legitimate users.

---

### Finding Description

`submit_blocks` — the only legitimate path for adding headers — is correctly gated behind `#[trusted_relayer]`: [1](#0-0) 

However, `run_mainchain_gc`, which is also called internally by `submit_blocks` and directly mutates the canonical chain state, carries only a `#[pause]` decorator with no caller restriction: [2](#0-1) 

The `#[pause(except(roles(Role::UnrestrictedRunGC)))]` attribute only checks whether the contract is paused. When the contract is live, any account — with no role, stake, or authorization — can call `run_mainchain_gc(batch_size)` with an arbitrary `batch_size`.

Inside the function, `selected_amount_to_remove` is bounded by `total_amount_to_remove = amount_of_headers_we_store - gc_threshold`, so the attacker cannot remove headers within the threshold window. However, passing `batch_size = u64::MAX` causes the function to remove every single "excess" header in one transaction: [3](#0-2) 

The loop calls `remove_block_header` (which deletes from `headers_pool` and `mainchain_header_to_height`) and `mainchain_height_to_header.remove` for each pruned height, then overwrites `mainchain_initial_blockhash` with the new tail: [4](#0-3) 

---

### Impact Explanation

After the attacker's call, any block hash that was in the pruned range is absent from both `mainchain_header_to_height` and `headers_pool`. When a user or downstream contract calls `verify_transaction_inclusion` for a transaction in a pruned block, the contract panics at: [5](#0-4) 

and again at: [6](#0-5) 

`verify_transaction_inclusion_v2` hits the same panic path via its internal delegation: [7](#0-6) 

The concrete corrupted state is:
- `mainchain_initial_blockhash` is advanced to a later block without any authorized relayer action.
- All pruned entries are permanently deleted from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`.
- SPV proof verification panics (not merely returns `false`) for any transaction whose block was in the pruned range, making the verification API unreliable for any consumer that has not yet queried those blocks.

---

### Likelihood Explanation

The entry point is a plain public NEAR method callable by any account with no deposit, no role, and no staking requirement. The only precondition is that the chain has accumulated more headers than `gc_threshold` (the default is 52 704 blocks, roughly one year of Bitcoin headers). Once that threshold is crossed — which is the normal operating state of a production deployment — any observer can trigger the full prune in a single transaction.

---

### Recommendation

Apply the same `#[trusted_relayer]` (or at minimum an explicit role check such as `Role::DAO` or `Role::RelayerManager`) to `run_mainchain_gc` that is applied to `submit_blocks`. The internal call from `submit_blocks` already passes through the trusted-relayer gate, so restricting the public entry point does not break the intended GC-on-submit flow. Alternatively, make `run_mainchain_gc` a private helper (`fn` instead of `pub fn`) and expose a separate, role-gated public wrapper if external GC triggering is genuinely required.

---

### Proof of Concept

1. Wait (or arrange) for the contract to hold more than `gc_threshold` mainchain headers (normal production state).
2. From any unprivileged NEAR account, call:
   ```
   run_mainchain_gc(batch_size: 18446744073709551615)   // u64::MAX
   ```
3. The contract removes every excess header in one call, advances `mainchain_initial_blockhash`, and deletes the corresponding entries from all three maps.
4. Any subsequent call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction in a pruned block panics with `"block does not belong to the current main chain"` or `"cannot find requested transaction block"`, permanently breaking SPV proof verification for those blocks.
