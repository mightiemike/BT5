### Title
`build_precommit_vote_message_digest` Omits `chain_id`, Height, and Round from Precommit Signature Preimage — (`File: crates/apollo_signature_manager/src/signature_manager.rs`)

---

### Summary

The precommit vote signature preimage is constructed from only a static domain tag (`b"PRECOMMIT_VOTE"`) and the raw `block_hash` bytes. It does not bind to `chain_id`, block height, or consensus round. A validator's precommit signature for a given block hash is therefore valid for any chain, any height, and any round where that same hash appears, directly mirroring the `updateProjectHash` pattern of missing domain context in a signed payload.

---

### Finding Description

`build_precommit_vote_message_digest` in `crates/apollo_signature_manager/src/signature_manager.rs` constructs the signed message as:

```
message = b"PRECOMMIT_VOTE" || block_hash.to_bytes_be()
``` [1](#0-0) 

The `Vote` struct that carries this signature contains `height`, `round`, `vote_type`, `proposal_commitment`, and `voter`: [2](#0-1) 

None of those fields — `height`, `round`, or `chain_id` — are included in the signed preimage. The verification function `verify_precommit_vote_signature` accepts only `block_hash`, `signature`, and `public_key`, with no chain or height context: [3](#0-2) 

The `sign_precommit_vote` entry point likewise takes only `block_hash`: [4](#0-3) 

The consensus manager does perform a height equality check before forwarding a vote to `SingleHeightConsensus`, but this check is on the unsigned `Vote.height` field, not on anything covered by the signature: [5](#0-4) 

The `chain_id` is present in the node config and is used correctly in transaction hash computation: [6](#0-5) 

But it is never threaded into the consensus signature domain.

---

### Impact Explanation

**Cross-chain signature reuse (primary impact):** A validator running the same key on two networks (e.g., a staging/testnet and mainnet) produces a precommit signature `σ = Sign(sk, H("PRECOMMIT_VOTE" || block_hash))`. Because `block_hash` does not embed `chain_id` in the signature preimage, `σ` is cryptographically valid on any chain where the same `block_hash` value appears. An adversary who observes a validator's precommit on one network can replay it verbatim on another network by constructing a `Vote` message with the same `proposal_commitment` and the captured signature. The receiving node's `verify_precommit_vote_signature` will accept it.

**Cross-height/round reuse (secondary impact):** The signature does not bind to `height` or `round`. A precommit for block hash `X` at height 100, round 0 is indistinguishable from a precommit for block hash `X` at height 200, round 3. The unsigned `Vote.height` field is checked by the consensus manager, but an attacker can freely set that field to any value when replaying the signature.

This matches the allowed impact: **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

---

### Likelihood Explanation

- Validators sharing a key across testnet and mainnet is a realistic operational scenario.
- Block hashes are computed from block contents; transactions include `chain_id` in their own hashes, making an exact block hash collision across chains unlikely but not impossible (e.g., empty blocks, genesis-adjacent blocks, or adversarially crafted block content).
- The missing `height`/`round` binding is exploitable whenever the same block hash recurs, which is a weaker but still real constraint.
- No privilege is required; any network observer can capture and replay a precommit signature.

---

### Recommendation

Include `chain_id`, `height`, and `round` in the precommit vote message preimage:

```rust
fn build_precommit_vote_message_digest(
    chain_id: &ChainId,
    height: BlockNumber,
    round: Round,
    block_hash: BlockHash,
) -> MessageDigest {
    let mut message = Vec::new();
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&Felt::try_from(chain_id).unwrap().to_bytes_be());
    message.extend_from_slice(&height.0.to_be_bytes());
    message.extend_from_slice(&(round as u64).to_be_bytes());
    message.extend_from_slice(&block_hash.to_bytes_be());
    MessageDigest(blake2s_to_felt(&message))
}
```

Update `sign_precommit_vote` and `verify_precommit_vote_signature` to accept and pass these fields. The `SignatureManager` should hold or receive `chain_id` at construction time, consistent with how `TransactionConverter` already carries `chain_id`: [7](#0-6) 

---

### Proof of Concept

1. Validator `V` with key `(sk, pk)` is connected to both `SN_GOERLI` (testnet) and `SN_MAIN` (mainnet).
2. On testnet, `V` precommits for a block with `block_hash = H`. The signature is `σ = Sign(sk, Blake2s(b"PRECOMMIT_VOTE" || H.to_bytes_be()))`.
3. An observer captures the wire-format `Vote { vote_type: Precommit, height: h_testnet, round: r, proposal_commitment: H, voter: V, signature: σ }`.
4. The observer replays this vote on mainnet, changing only `height` to a mainnet height `h_main` where a block with hash `H` is being proposed (or constructing a block whose hash equals `H`).
5. `verify_precommit_vote_signature(H, σ, pk)` returns `true` on mainnet because the preimage `b"PRECOMMIT_VOTE" || H.to_bytes_be()` is identical.
6. The mainnet consensus node counts `V`'s precommit without `V` ever having signed anything on mainnet, potentially contributing to a false quorum. [1](#0-0) [3](#0-2)
