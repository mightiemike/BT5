### Title
`ChunkExtra.proposed_split` Not Reset to `None` in Child Shards After Dynamic Resharding â€” (`File: chain/chain/src/resharding/manager.rs`)

### Summary

During a dynamic resharding event, the child shard `ChunkExtra` is cloned from the parent shard's `ChunkExtra` but the `proposed_split` field is never reset to `None`. The parent's `proposed_split` (a `Some(TrieSplit {...})` that triggered the split) is inherited verbatim. When the child shard's first chunk is produced in epoch N+2, the chunk producer correctly computes `proposed_split = None` (cooldown prevents immediate re-splitting), but the stored `ChunkExtra` still holds the parent's non-`None` value. The mandatory field-equality check in `validate_chunk_with_chunk_extra_and_receipts_root` then rejects every first chunk of every child shard with `InvalidChunkHeaderShardSplit`, halting chunk production for those shards after any dynamic resharding event.

---

### Finding Description

In `chain/chain/src/resharding/manager.rs`, the child `ChunkExtra` is constructed by cloning the parent and overwriting only `state_root` and `congestion_info`:

```rust
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
``` [1](#0-0) 

The `proposed_split` field â€” added in `ChunkExtraV5` for dynamic resharding â€” is never touched. The parent shard's `ChunkExtra` at the resharding block carries `proposed_split = Some(TrieSplit { boundary_account, ... })` (the very split that triggered resharding). Both child shards inherit this value verbatim. [2](#0-1) 

In epoch N+2, when the child shard's first chunk is produced, `compute_proposed_split` returns `None` because:
1. `can_reshard` checks `epoch_height - last_resharding >= min_epochs_between_resharding` â€” the cooldown was just set, so re-splitting is blocked.
2. `is_next_block_possibly_last_in_epoch` returns `false` for early-epoch blocks. [3](#0-2) 

The chunk header therefore carries `proposed_split = None`. The mandatory validation in `validate_chunk_with_chunk_extra_and_receipts_root` compares the stored `ChunkExtra` against the incoming chunk header:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    DYNAMIC_RESHARDING_VALIDATION_FAILURES.with_label_values(&["chunk_header"]).inc();
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [4](#0-3) 

`Some(TrieSplit {...}) != None` â†’ `InvalidChunkHeaderShardSplit` â†’ chunk rejected. This fires for every child shard's first chunk, every time dynamic resharding executes.

The exact divergent Borsh value is the `proposed_split` field of `ChunkExtraV5`: stored as `Some(TrieSplit { boundary_account: <AccountId>, left_memory_usage: u64, right_memory_usage: u64 })` in the child `ChunkExtra`, but expected as `None` by the chunk producer and the chunk header it signs. [5](#0-4) 

---

### Impact Explanation

After any dynamic resharding event, both child shards cannot produce their first valid chunk. `InvalidChunkHeaderShardSplit` is a hard rejection â€” the chunk is discarded, not retried with different parameters. Because the stored `ChunkExtra` is the authoritative source for the next chunk's field expectations, and it is never corrected, the mismatch persists for every subsequent attempt. This constitutes a **chain halt** for the affected shards: no chunk can be accepted, no state transitions can proceed, and the network cannot advance past the resharding boundary.

---

### Likelihood Explanation

The bug fires deterministically on every dynamic resharding event. Dynamic resharding is automatic â€” it triggers when a shard's trie memory usage crosses `memory_usage_threshold` â€” so no privileged action is required to reach the vulnerable state. The feature is gated by `ProtocolFeature::DynamicResharding` (nightly, protocol version 153+), active on betanet and in the nightly CI environment. Any node running a nightly build that experiences a resharding event will hit this path. [6](#0-5) 

---

### Recommendation

In `split_shard` (or immediately after `ChunkExtra::clone`), explicitly reset `proposed_split` to `None` on the child `ChunkExtra` before saving it:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
*child_chunk_extra.proposed_split_mut() = None;  // child shard starts fresh
```

This mirrors the invariant that `proposed_split` is computed fresh by each chunk producer and must not be inherited across a shard-split boundary. The existing TODO comment at line 255 of `manager.rs` already acknowledges that not all `ChunkExtra` fields are set; `proposed_split` must be added to the explicit reset list. [7](#0-6) 

---

### Proof of Concept

1. Enable `ProtocolFeature::DynamicResharding` (nightly build).
2. Configure `force_split_shards` in `DynamicReshardingConfig` to force a split on the next epoch boundary.
3. Run the network through one epoch. At the last block of epoch N, `shard_split` is embedded in the block header; `finalize_epoch` stores a new `EpochInfoV5.shard_layout` for epoch N+2.
4. During epoch N+1, `start_resharding` fires, calling `split_shard`. The child `ChunkExtra` entries are saved with `proposed_split = Some(TrieSplit { boundary_account: <forced_account>, ... })` cloned from the parent.
5. At the start of epoch N+2, the chunk producer for each child shard calls `compute_proposed_split` â†’ `can_reshard` returns `false` (cooldown) â†’ returns `None` â†’ chunk header has `proposed_split = None`.
6. `validate_chunk_with_chunk_extra_and_receipts_root` compares `Some(TrieSplit {...})` (stored) vs `None` (header) â†’ `Err(Error::InvalidChunkHeaderShardSplit(...))`.
7. Both child shards are permanently unable to produce accepted chunks; the network halts on those shards. [4](#0-3) [8](#0-7)
