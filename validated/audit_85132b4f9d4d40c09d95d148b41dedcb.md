### Title
ML-DSA-65 Gas-Key Exec-Fee Uses Wire Length Instead of Trie-ID Length in Host-Function Path — (`runtime/near-vm-runner/src/logic/logic.rs`)

### Summary

When a contract adds an ML-DSA-65 gas key via the `promise_batch_action_add_gas_key_with_full_access` or `promise_batch_action_add_gas_key_with_function_call` host functions, the exec-fee calculation receives `public_key_len as usize` — the raw wire/borsh length of the key in WASM memory (1953 bytes for ML-DSA-65) — instead of the on-trie identifier length returned by `PublicKey::trie_id_len()` (33 bytes). The transaction path (`permission_exec_fees` in `runtime/runtime/src/config.rs`) correctly uses `public_key.trie_id_len()`. This produces a ~59× exec-fee overcharge per nonce for ML-DSA-65 gas keys added via host functions, breaking the fee-parity invariant between the two paths and potentially causing out-of-gas failures for contracts that budget based on the transaction-path cost.

### Finding Description

`gas_key_add_key_exec_fee` models the trie-write cost of each nonce entry as:

```
nonce_key_len = access_key_key_len(account_id_len, public_key_len) + sizeof(NonceIndex)
per_byte_fee  = gas_key_byte.exec_fee() × (nonce_key_len + NONCE_VALUE_LEN) × num_nonces
```

The `public_key_len` argument is supposed to be the **on-trie** key-identifier length, because that is what is actually written to the trie for each nonce entry. For ML-DSA-65, the trie stores `[tag=3] ‖ sha3_256(domain ‖ raw_pubkey)` — 33 bytes — not the 1953-byte borsh-encoded full pubkey.

**Transaction path** (`runtime/runtime/src/config.rs`, `permission_exec_fees`): correctly passes `public_key.trie_id_len()` = 33.

**Host-function path** (`runtime/near-vm-runner/src/logic/logic.rs`, `promise_batch_action_add_gas_key_with_full_access`): passes `public_key_len as usize` = 1953 (the raw byte count of the borsh-encoded ML-DSA-65 key in WASM memory).

The divergent value is **1953 vs 33** — a difference of 1920 bytes per nonce, multiplied by `gas_key_byte` exec fee and `num_nonces`.

The same error is present in the wasmtime runner mirror at `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`.

The existing fee-parity test (`test_gas_key_fee_parity`) only exercises ED25519 keys, for which `len()` == `trie_id_len()` == 33, so the divergence is invisible in the test suite.

### Impact Explanation

Any contract that calls `promise_batch_action_add_gas_key_with_full_access` or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key will be charged ~59× more exec gas per nonce than the equivalent transaction. If the contract's attached gas budget is sized according to the documented/transaction-path cost, the receipt will run out of gas and the gas-key addition will fail — permanently blocking that code path. The fee-domain invariant (same operation, same cost regardless of invocation path) is broken at the protocol level.

### Likelihood Explanation

Requires `PostQuantumSignatures` protocol feature to be active (currently nightly/testnet only) and a contract to invoke the gas-key host functions with an ML-DSA-65 key. Once the feature is stabilised on mainnet, any contract that attempts this will hit the overcharge. No privileged role is required; any unprivileged contract author can trigger it.

### Recommendation

In both `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`, decode the public key first, then pass `public_key.decode()?.trie_id_len()` (or equivalently, call `trie_id_len()` on the decoded key before it is consumed) to `gas_key_add_key_exec_fee`, mirroring the transaction path in `permission_exec_fees`.

```diff
-    let exec_fee = gas_key_add_key_exec_fee(
-        &self.fees_config,
-        receiver_id.len(),
-        public_key_len as usize,   // wire length — wrong for ML-DSA-65
-        num_nonces,
-    );
+    let decoded_pk = public_key.decode()?;
+    let exec_fee = gas_key_add_key_exec_fee(
+        &self.fees_config,
+        receiver_id.len(),
+        decoded_pk.trie_id_len(),  // on-trie length — correct for all key types
+        num_nonces,
+    );
```

Apply the same fix in `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`.

### Proof of Concept

**Transaction path** — `permission_exec_fees` in `runtime/runtime/src/config.rs`: [1](#0-0) 

Uses `public_key.trie_id_len()` = 33 for ML-DSA-65.

**Host-function path** — `promise_batch_action_add_gas_key_with_full_access` in `runtime/near-vm-runner/src/logic/logic.rs`: [2](#0-1) 

Uses `public_key_len as usize` = 1953 for ML-DSA-65 (the borsh wire length).

**Fee function** — `gas_key_add_key_exec_fee` in `core/parameters/src/cost.rs`: [3](#0-2) 

`nonce_key_len` is computed from `public_key_len`, so the 1920-byte error propagates into every nonce's exec fee.

**The trie-id vs wire-length contract** — `PublicKey::trie_id_len` in `core/crypto/src/signature.rs`: [4](#0-3) 

For `MLDSA65`, returns `1 + ML_DSA_65_HASH_LENGTH` = 33, not 1953.

**Documented invariant** (post-quantum signatures architecture doc):
> Callers that still use `len()` for trie-storage costing will misprice ML-DSA-65 keys by ~1900 bytes.

The host-function path violates this invariant by using the raw wire length rather than `trie_id_len()`.

```rust
pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
```
