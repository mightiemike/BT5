### Title
NLP Pool `balanceWeightX18` Not Enforced During Mint/Burn — (`core/contracts/Clearinghouse.sol`)

---

### Summary

The `_validateNlpRebalance` function validates the `nlpPoolRebalanceX18` array only by checking that its sum equals the total quote delta. It never verifies that each pool's share is proportional to its `balanceWeightX18`. A user who signs a `MintNlp` or `BurnNlp` transaction can freely choose any distribution of quote across pools — concentrating deposits into one pool and withdrawing from another — transferring value between pools in violation of the intended weight-based allocation.

---

### Finding Description

In `Clearinghouse.sol`, `_validateNlpRebalance` is the sole on-chain guard for the per-pool rebalance amounts supplied during every NLP mint and burn: [1](#0-0) 

The function checks only that `sum(nlpPoolRebalanceX18[i]) == deltaQuoteAmount`. It never inspects `nlpPools[i].balanceWeightX18`.

The `nlpPoolRebalanceX18` array is part of `SignedMintNlp` / `SignedBurnNlp`, which the user signs: [2](#0-1) 

`validateSignedTx` hashes the entire transaction body — including `nlpPoolRebalanceX18` — so the user fully controls these values at signing time: [3](#0-2) 

The sequencer submits exactly what the user signed. The on-chain path from `mintNlp` to `_applyNlpRebalance` applies the user-chosen distribution without any weight check: [4](#0-3) 

During `burnNlp`, the only health check performed is on the **user's** subaccount, not on the pools being drained: [5](#0-4) 

This means a user can deposit all quote into pool A during mint and withdraw all quote from pool B during burn, with no on-chain rejection.

---

### Impact Explanation

The `balanceWeightX18` field is the protocol's mechanism for proportional profit-sharing and risk distribution among NLP pools: [6](#0-5) 

Because `_validateNlpRebalance` never enforces proportionality, a user can:

1. **Mint** 100 USDC of NLP with `nlpPoolRebalanceX18 = [100e18, 0]` — all quote goes to pool 0, none to pool 1.
2. **Burn** the same NLP with `nlpPoolRebalanceX18 = [0, -100e18]` — all quote is taken from pool 1.

Net result: 100 USDC is silently transferred from pool 1 to pool 0. Pool 1's quote balance is reduced by the full mint amount; pool 1's owner loses value proportional to the attack size. The `balanceWeightX18` invariant is broken for every mint/burn cycle where the user chooses a non-proportional distribution.

---

### Likelihood Explanation

The user signs `nlpPoolRebalanceX18` directly. The sequencer submits what the user signed. There is no slow-mode path for `MintNlp`/`BurnNlp` that would allow the owner to intercept the transaction: [7](#0-6) 

If the sequencer does not independently validate weight proportionality off-chain, any user can craft and sign a skewed rebalance. Because the on-chain code provides no backstop, the invariant is entirely dependent on off-chain sequencer policy — a fragile guarantee that is not documented as a security boundary.

---

### Recommendation

Add a proportionality check inside `_validateNlpRebalance`. Compute the total weight of all pools, then assert that each pool's rebalance amount does not deviate from `deltaQuoteAmount * pool.balanceWeightX18 / totalWeight` beyond an acceptable rounding tolerance. Alternatively, compute the rebalance amounts on-chain from the weights rather than accepting them as user-supplied inputs.

---

### Proof of Concept

**Setup**: Two NLP pools, each with `balanceWeightX18 = 0.5e18` (50 %), each holding 1 000 USDC in quote.

**Step 1 — Skewed mint**
User signs `MintNlp` with `quoteAmount = 100e18` and `nlpPoolRebalanceX18 = [100e18, 0]`.

`_validateNlpRebalance` passes: `100e18 + 0 == 100e18`. ✓

Pool 0 quote: +100 USDC. Pool 1 quote: unchanged.

**Step 2 — Skewed burn**
User signs `BurnNlp` with `nlpAmount` equivalent to 100 USDC and `nlpPoolRebalanceX18 = [0, -100e18]`.

`_validateNlpRebalance` passes: `0 + (−100e18) == −100e18`. ✓

Pool 1 quote: −100 USDC. Pool 0 quote: unchanged.

**Outcome**: Pool 0 net +100 USDC; Pool 1 net −100 USDC. The 50/50 weight distribution is violated. The pool 1 owner's backing assets are reduced by the full attack amount with no on-chain rejection at any step. [8](#0-7)

    function _validateNlpRebalance(
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18,
        int128 deltaQuoteAmount
    ) internal pure {
        require(
            nlpPools.length == nlpPoolRebalanceX18.length,
            ERR_INVALID_NLP_REBALANCE
        );
        int128 rebalanceAmount = 0;
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            rebalanceAmount += nlpPoolRebalanceX18[i];
        }
        require(deltaQuoteAmount == rebalanceAmount, ERR_INVALID_NLP_REBALANCE);
    }
