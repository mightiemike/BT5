### Title
`burnNlp` Uses Maintenance Health Check Instead of Initial Health Check, Allowing Users to Deliberately Create Under-Collateralized Subaccounts — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp` function in `Clearinghouse.sol` enforces a `MAINTENANCE` health check after burning NLP tokens, while the symmetric `mintNlp` function enforces an `INITIAL` health check. This inconsistency is the direct analog to the external report's flawed conditional: just as the UNCX `lock` function used a wrong boolean predicate to classify positions, `burnNlp` uses the wrong health type to gate the operation. Any user with NLP tokens can burn them and leave their subaccount with `INITIAL` health below zero — a state the protocol's invariant requires to be unreachable through user-initiated actions.

---

### Finding Description

`mintNlp` enforces the stricter `INITIAL` health gate after minting: [1](#0-0) 

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

`burnNlp` enforces only the weaker `MAINTENANCE` health gate after burning: [2](#0-1) 

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
```

The in-code comment at line 524–525 explicitly acknowledges the intent: *"This check prevents malicious actors from deliberately creating unhealthy subaccounts through NLP burns."* However, the check used (`MAINTENANCE`) does not fulfil that intent. `MAINTENANCE` health uses more lenient risk weights than `INITIAL` health. A subaccount can satisfy `MAINTENANCE >= 0` while simultaneously having `INITIAL < 0`.

The burn path that causes this:

1. A user holds NLP tokens and has `INITIAL` health slightly above zero.
2. The user calls `burnNlp`. A burn fee is deducted: [3](#0-2) 

```solidity
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
```

3. NLP tokens (which carry a health contribution) are removed from the sender and credited to `N_ACCOUNT`. The sender receives `quoteAmount` (net of fee) in quote.
4. If the NLP risk weight exceeds the quote risk weight, or if the fee is large enough relative to the health margin, the net effect is a decrease in `INITIAL` health.
5. The post-burn check passes because `MAINTENANCE >= 0`, even though `INITIAL < 0`.

The result is a user-controlled subaccount with `INITIAL` health below zero — a state the protocol otherwise only permits to arise from adverse market movements, not from user-initiated transactions.

Additionally, because `burnNlp` only checks `MAINTENANCE` health, a user whose `INITIAL` health is already below zero (from prior market movements) can continue to burn NLP. Each successive burn extracts quote from the NLP pools while the subaccount remains in an `INITIAL`-unhealthy state, bypassing the collateral adequacy gate that every other user-initiated state-changing operation (`withdrawCollateral`, `transferQuote`, `mintNlp`) enforces. [4](#0-3) 

---

### Impact Explanation

**Broken invariant**: Every user-initiated action that changes subaccount state must leave `INITIAL` health ≥ 0. `burnNlp` violates this invariant.

**Concrete state corruption**:
- A user can exit an NLP position and land in `INITIAL < 0`, `MAINTENANCE >= 0`. This subaccount cannot be liquidated (maintenance health is non-negative) but is under-collateralized by the protocol's own initial-margin standard.
- A user already at `INITIAL < 0` (from market moves) can keep burning NLP — converting NLP pool assets to quote in their own account — while the initial-health gate that should block all such operations is bypassed.
- The protocol's risk model relies on the invariant that `INITIAL < 0` is only reachable through market price changes, not through user transactions. Violating this assumption can corrupt downstream accounting that depends on it (e.g., spread health adjustments in `getHealth`, liquidation eligibility logic in `ClearinghouseLiq`). [5](#0-4) 

---

### Likelihood Explanation

The trigger requires no privileged access. Any user holding NLP tokens and whose subaccount has `INITIAL` health close to zero can reach this state in a single `burnNlp` transaction submitted through the normal `Endpoint` flow. The burn fee is only 0.1% (`quoteAmount / 1000`), so even a small health margin is sufficient to be eroded. The condition is reachable under normal market conditions whenever NLP carries a higher initial risk weight than the quote asset.

---

### Recommendation

Replace the `MAINTENANCE` health type with `INITIAL` in the post-burn health check in `burnNlp`, consistent with `mintNlp`:

```solidity
// Before (incorrect):
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);

// After (correct):
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [6](#0-5) 

This aligns `burnNlp` with `mintNlp` and with every other user-initiated operation that mutates subaccount state, restoring the protocol's invariant that `INITIAL` health must remain non-negative after any user action.

---

### Proof of Concept

**Setup**:
- Subaccount `A` holds NLP tokens. `INITIAL` health = `+ε` (just above zero). `MAINTENANCE` health = `+δ` (comfortably above zero, since maintenance weights are more lenient).

**Steps**:
1. Subaccount `A` submits a `BurnNlp` transaction through `Endpoint`.
2. `burnNlp` executes: NLP tokens are removed, quote is credited net of the 0.1% burn fee.
3. Because NLP carries a higher initial risk weight than quote, and/or the burn fee erodes the margin, `INITIAL` health after the burn = `−x` (below zero).
4. `MAINTENANCE` health after the burn = `+y` (still non-negative due to more lenient weights).
5. The `require(getHealth(..., MAINTENANCE) >= 0)` check passes.
6. Transaction succeeds. Subaccount `A` now has `INITIAL < 0`, `MAINTENANCE >= 0`.

**Verification**: Call `getHealth(A, INITIAL)` — returns negative. Call `getHealth(A, MAINTENANCE)` — returns non-negative. Subaccount `A` cannot be liquidated, cannot withdraw, cannot open positions, but the protocol accepted the burn as valid.

**Extended path**: Repeat step 1–6 while `MAINTENANCE >= 0`. Each iteration extracts additional quote from the NLP pools into subaccount `A` while bypassing the initial-health gate.
