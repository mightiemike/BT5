### Title
`verify_transaction_inclusion` (v1) Remains Publicly Callable, Bypassing Coinbase Merkle Proof Validation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase proof. However, the original `verify_transaction_inclusion` (v1) remains a `pub` NEAR method callable by any unprivileged account. Any caller — including a downstream recipient contract — can invoke v1 directly, skipping the coinbase check entirely and obtaining a `true` proof result for a forged transaction.

---

### Finding Description

The vulnerability class in the external report is a **required-step bypass**: a higher-level function enforces a critical invariant (registering a validator), but a lower-level inherited function is still publicly reachable and skips that invariant. The exact same pattern exists here.

`verify_transaction_inclusion_v2` enforces two things before accepting a proof:

1. The coinbase transaction must be proven at index 0 against the block's Merkle root.
2. Only then is the caller's `tx_id` proof checked (by delegating to v1). [1](#0-0) 

The coinbase check is the security-critical step. It prevents the [64-byte transaction forgery](https://www.bitmex.com/blog/64-Byte-Transactions): an attacker cannot supply a fake `tx_id` that is actually an internal Merkle node, because the coinbase proof at index 0 would not match.

However, v1 is still `pub` and carries no such guard: [2](#0-1) 

The contract's own `# Warning` comment on v1 acknowledges this: *"This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree."* [3](#0-2) 

The `#[deprecated]` Rust attribute is a compiler lint only; it has no effect on NEAR's ABI or on-chain callability. The method remains a fully reachable public entry point. [4](#0-3) 

The `ProofArgs` struct accepted by v1 contains no `coinbase_tx_id` or `coinbase_merkle_proof` fields, so there is no way to retrofit the check at the argument level. [5](#0-4) 

---

### Impact Explanation

An attacker who wants to falsely prove that a Bitcoin transaction `T` was included in block `B` can:

1. Identify an internal Merkle node `N` in block `B`'s transaction tree whose hash equals the hash they want to pass off as `T`.
2. Construct a valid Merkle path from `N` to the block's Merkle root (this path is shorter by one level than a leaf-level proof).
3. Call `verify_transaction_inclusion` directly with `tx_id = N`, a valid `tx_block_blockhash` for `B`, and the crafted proof.
4. The function computes `compute_root_from_merkle_proof(N, index, proof)`, which equals the real Merkle root, and returns `true`.

Any recipient contract that calls v1 to gate a cross-chain action (token unlock, bridge withdrawal, etc.) will accept this forged proof. The corrupted value is the **proof result** — a `true` return for a transaction that was never included in the block.

---

### Likelihood Explanation

- v1 is reachable by any unprivileged NEAR account with no staking, role, or deposit requirement beyond the `#[pause]` gate (which is off by default).
- Downstream contracts integrated before v2 was released (v0.5.0) call v1 by name and cannot be forced to upgrade.
- The 64-byte forgery technique is publicly documented and well-understood; constructing the required internal-node proof requires only knowledge of the target block's Merkle tree, which is public Bitcoin data.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` (v1) so it becomes a private helper callable only by `verify_transaction_inclusion_v2`. Since v2 already delegates to v1 internally after performing the coinbase check, no functionality is lost:

```rust
// Change:
pub fn verify_transaction_inclusion(&self, ...) -> bool { ... }

// To:
fn verify_transaction_inclusion(&self, ...) -> bool { ... }
```

This mirrors the recommended fix in the external report: override (or restrict) the lower-level entry point so the required security step cannot be bypassed.

### Proof of Concept

```
// Attacker calls v1 directly on the deployed NEAR contract:
near call <contract_id> verify_transaction_inclusion \
  '{"tx_id": "<internal_node_hash>",
    "tx_block_blockhash": "<real_block_hash>",
    "tx_index": <crafted_index>,
    "merkle_proof": [<crafted_sibling_hashes>],
    "confirmations": 1}' \
  --accountId attacker.near

// Returns: true
// The same call to verify_transaction_inclusion_v2 would panic
// with "Incorrect coinbase merkle proof".
```

The attacker supplies an `internal_node_hash` that is a real intermediate node in the target block's Merkle tree. `compute_root_from_merkle_proof` reconstructs the correct Merkle root from this node and the crafted sibling path, so the equality check at line 318–322 passes and the function returns `true`. [6](#0-5)

        since = "0.5.0",

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
