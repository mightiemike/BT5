Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of real user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument passed by the pool, which is always `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, that caller is the router contract, not the original user. Allowlisting the router — the required operational step to enable router-mediated swaps — renders the per-pool swap allowlist completely inoperative for every address on the network.

## Finding Description
**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`. When the router is the direct caller of `pool.swap()`, `msg.sender` is the router address.

**Step 2 — Router never forwards the real user to the pool.**

`MetricOmmSimpleRouter.exactInputSingle` stores the real user (`msg.sender`) in transient storage via `_setNextCallbackContext` solely for the payment callback. It then calls `pool.swap(params.recipient, ...)` with no field carrying the original user's identity to the pool or its extensions.

**Step 3 — Extension checks the wrong identity.**

`SwapAllowlistExtension.beforeSwap(address sender, ...)` evaluates `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][router]`. If the pool admin has allowlisted the router (the only way to permit router-mediated swaps), every caller passes the check regardless of individual allowlist status.

**Broken invariant:** `allowedSwapper[pool][user]` is intended to gate the economic actor initiating the swap. Via the router, the gated identity is the router contract, not the user, making the mapping semantically void.

## Impact Explanation
A pool deployer using `SwapAllowlistExtension` intends to restrict swaps to specific addresses (e.g., KYC-verified counterparties, institutional participants). Once the router is allowlisted, any address can call `router.exactInputSingle()` and swap against the restricted pool. LP assets are exposed to unauthorized counterparties at oracle-derived prices intended only for vetted participants. The loss is direct and unbounded — the full pool liquidity is accessible to any caller. This is a direct loss of LP principal and a broken core pool access-control invariant.

## Likelihood Explanation
No special privilege is required. Any user with tokens can call `MetricOmmSimpleRouter.exactInputSingle()`. The only precondition — the pool admin allowlisting the router — is the expected operational step for any pool that supports standard periphery UX. The bypass is reachable in every realistic deployment of a swap-allowlisted pool that also supports router-mediated swaps.

## Recommendation
The extension must gate on the original user, not the intermediary. Two complementary approaches:

1. **Forward the real initiator through the router.** The router already stores `msg.sender` in transient storage. It should also encode it in `extensionData` (or a dedicated field), and `SwapAllowlistExtension` should decode and verify it when the direct caller is a recognized router.
2. **Gate on `recipient`.** If the design intent is that only allowlisted addresses may receive swap output, gating on `recipient` (which the user controls and the router forwards correctly) is more robust, though it changes allowlist semantics.

The simplest safe fix is option 1: encode `msg.sender` in `extensionData` inside the router and have `SwapAllowlistExtension` decode and check it when `msg.sender` (the pool's direct caller) is a known router address.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin: swapExt.setAllowedToSwap(pool, router, true)   // enable router UX
  - Pool admin does NOT call swapExt.setAllowedToSwap(pool, attacker, true)
  - Pool admin adds liquidity

Attack:
  attacker (not allowlisted) calls:
    router.exactInputSingle({pool, tokenIn, tokenOut, zeroForOne, amountIn, recipient: attacker, ...})

  Execution path:
    router.exactInputSingle()
      → pool.swap(msg.sender=router, ...)
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router)
            → allowedSwapper[pool][router] == true  ✓  (no revert)
        → swap executes, attacker receives output token

Result:
  Attacker swaps on a pool they are not individually allowlisted for.
  Allowlist invariant broken; LP assets exposed to unauthorized counterparties.
```
