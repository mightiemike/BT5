### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables USDC.e Drain Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

### Summary
`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` without checking the return value or using the codebase's own `ERC20Helper.safeTransferFrom` wrapper. If the USDC token returns `false` instead of reverting on a failed transfer, the function continues to withdraw USDC.e from the target `directDepositV1` contract and send it to the caller — with no USDC ever deposited in exchange.

### Finding Description
`replaceUsdcEWithUsdc` is an `external` function with no access-control modifier beyond a chain-ID check (`block.chainid == 57073`). Its intended flow is:

1. Read `balance` of USDC.e held by a `directDepositV1` contract.
2. Pull `balance` USDC **from `msg.sender`** into `directDepositV1` via `transferFrom`.
3. Withdraw the USDC.e from `directDepositV1` back to `ContractOwner`.
4. Send that USDC.e to `msg.sender`.

The critical line is step 2:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [1](#0-0) 

The return value of `transferFrom` is silently discarded. The rest of the codebase consistently wraps every token transfer in `ERC20Helper.safeTransferFrom` / `ERC20Helper.safeTransfer`, which checks both the low-level call success and the decoded boolean return value:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [2](#0-1) 

Even the very next line in the same function uses `safeTransfer` for the USDC.e leg:

```solidity
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
``` [3](#0-2) 

The inconsistency is unambiguous: one leg of the swap is guarded, the other is not.

### Impact Explanation
If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (Ink chain 57073) returns `false` on a failed `transferFrom` (e.g., insufficient allowance, insufficient balance) rather than reverting, any caller can:

- Call `replaceUsdcEWithUsdc(subaccount)` for any `subaccount` whose `directDepositV1` holds USDC.e.
- The silent-fail `transferFrom` deposits **zero USDC** into `directDepositV1`.
- `DirectDepositV1(directDepositV1).withdraw(usdcE)` still executes, pulling all USDC.e out.
- `safeTransfer` sends the full USDC.e balance to `msg.sender`.

The corrupted asset delta: the caller receives `balance` USDC.e tokens for free; the `directDepositV1` contract loses its entire USDC.e balance; no USDC is deposited in return. This is a direct token theft from user-owned deposit accounts. [4](#0-3) 

### Likelihood Explanation
The function is reachable by any unprivileged external caller on chain 57073 with no further preconditions. The only gate is `block.chainid == 57073`. The trigger depends on whether the specific USDC deployment returns `false` vs. reverts on failure. Many USDC-adjacent or bridged USDC variants (including some wrapped/bridged versions on newer chains) do return `false` rather than revert. The codebase itself treats this as a real risk — that is precisely why `ERC20Helper.safeTransferFrom` exists and is used everywhere else.

### Recommendation
Replace the bare `transferFrom` call with the codebase's existing safe wrapper:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`safeTransferFrom` is already imported via `using ERC20Helper for IERC20Base` at the top of `ContractOwner.sol`. [5](#0-4) 

### Proof of Concept
1. A `directDepositV1` for some `subaccount` holds 1000 USDC.e.
2. Attacker calls `replaceUsdcEWithUsdc(subaccount)` without having approved any USDC to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, 1000)` returns `false` (no revert).
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes — 1000 USDC.e moves to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, 1000)` executes — attacker receives 1000 USDC.e.
6. Net result: attacker gains 1000 USDC.e; `directDepositV1` is drained; no USDC was ever deposited. [4](#0-3)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

**File:** core/contracts/libraries/ERC20Helper.sol (L23-42)
```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
