### Title
Non-Zero Allowance Reset Missing Before `approve()` in `DirectDepositV1.creditDeposit()` Blocks USDT Deposits — (`File: core/contracts/DirectDepositV1.sol`)

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. For USDT (and any token that enforces the non-zero-to-non-zero approval guard), a residual allowance from a prior incomplete deposit cycle causes the next `creditDeposit()` call to revert, permanently blocking USDT from being credited to the subaccount and leaving it stranded in the DDA contract.

### Finding Description

`creditDeposit()` is a publicly callable function with no access control. It iterates over all registered spot product tokens, approves the endpoint for the current balance, and calls `depositCollateralWithReferral`:

```solidity
// DirectDepositV1.sol line 92
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(
    subaccount,
    productId,
    uint128(balance),   // ← truncates if balance > type(uint128).max
    "-1"
);
``` [1](#0-0) 

The approved amount is `balance` (a `uint256`), but the deposited amount is `uint128(balance)`. If `balance > type(uint128).max`, the endpoint's `transferFrom` only consumes `uint128(balance)` tokens, leaving a non-zero residual allowance of `balance - uint128(balance)`. On the next call to `creditDeposit()`, the `approve(endpoint, newBalance)` call reverts for USDT because USDT's mainnet contract enforces:

```
require(!((_value != 0) && (allowed[msg.sender][_spender] != 0)));
```

The same codebase already applies the correct pattern in `ContractOwner.wrapVaultAsset()`:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [2](#0-1) 

This confirms developer awareness of the USDT approval constraint, making the omission in `creditDeposit()` a concrete oversight.

`ContractOwner.depositInsurance()` has the same missing reset at line 254, but is `onlyOwner` and therefore out of scope for unprivileged-caller impact. [3](#0-2) 

### Impact Explanation

USDT deposited to a DDA contract cannot be credited to the target subaccount once a residual allowance exists. The tokens remain stranded in the `DirectDepositV1` contract with no automatic recovery path. The `creditDeposit()` call reverts on every subsequent invocation for that token, permanently blocking the deposit flow for USDT on that DDA instance.

### Likelihood Explanation

The residual-allowance trigger requires `balance > type(uint128).max`. For USDT (6 decimals) this is ~3.4 × 10³² USDT — practically unreachable under normal conditions. However, the structural defect is present and the fix is trivial. If the endpoint's `depositCollateralWithReferral` implementation ever caps or partially consumes the transferred amount for any reason (e.g., internal deposit limits), the same residual-allowance condition arises at realistic balances. Likelihood is **low** under current conditions but the code is fragile by construction.

### Recommendation

Mirror the pattern already used in `wrapVaultAsset()`: reset the allowance to zero before setting the new value.

```solidity
if (balance != 0) {
    token.approve(address(endpoint), 0);          // reset first
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),
        "-1"
    );
}
``` [1](#0-0) 

### Proof of Concept

1. USDT is registered as a spot product token in `SpotEngine`.
2. A DDA is created for a subaccount via `ContractOwner.createDirectDepositV1()`.
3. An amount of USDT exceeding `type(uint128).max` is sent to the DDA (or, under a partial-consume endpoint scenario, any non-zero amount).
4. `creditDeposit()` is called: `approve(endpoint, balance)` succeeds; `depositCollateralWithReferral` consumes only `uint128(balance)`, leaving residual allowance.
5. Any subsequent call to `creditDeposit()` (e.g., via `ContractOwner.creditDepositV1()`) hits `token.approve(endpoint, newBalance)` with a non-zero existing allowance → USDT reverts → USDT is permanently stranded in the DDA. [4](#0-3) [5](#0-4)
