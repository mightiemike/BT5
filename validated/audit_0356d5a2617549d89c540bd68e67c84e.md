### Title
Double Internal-Node Forgery in `verify_transaction_inclusion_v2` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` is intended to mitigate the 64-byte / internal-node Merkle forgery attack by requiring a coinbase proof alongside the transaction proof. However, because `coinbase_tx_id` is fully attacker-controlled and never validated against any known on-chain value, an attacker can supply an internal Merkle node as the fake coinbase and a different internal node as the fake transaction. Both proofs reconstruct the real `merkle_root`, the equal-length guard passes, and the function returns `true` — even though neither supplied hash corresponds to a real leaf transaction.

---

### Finding Description

`verify_transaction_inclusion_v2` performs three checks:

1. **Equal-length guard** — `merkle_proof.len() == coinbase_merkle_proof.len()` [1](#0-0) 

2. **Coinbase proof check** — `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof) == merkle_root` [2](#0-1) 

3. **Transaction proof check** (delegated to `verify_transaction_inclusion`) — `compute_root_from_merkle_proof(tx_id, tx_index, &merkle_proof) == merkle_root` [3](#0-2) 

`compute_root_from_merkle_proof` is a pure hash-chain computation with no leaf/depth validation: [4](#0-3) 

`ProofArgsV2` shows that `coinbase_tx_id` is a plain caller-supplied `H256` with no binding to any stored or verified on-chain coinbase: [5](#0-4) 

**Concrete exploit for a 4-tx block `[tx0, tx1, tx2, tx3]`:**

```
Level 1:  h01 = hash(tx0, tx1)    h23 = hash(tx2, tx3)
Root:     root = hash(h01, h23)
```

Attacker submits:

| Field | Value | Proof length |
|---|---|---|
| `coinbase_tx_id` | `h01` (internal node) | — |
| `coinbase_merkle_proof` | `[h23]` | 1 |
| `tx_id` | `h23` (internal node) | — |
| `tx_index` | `1` | — |
| `merkle_proof` | `[h01]` | 1 |

Step-by-step evaluation:

- **Equal-length**: `1 == 1` ✓
- **Coinbase check**: `compute_root_from_merkle_proof(h01, 0, [h23])` → position 0 is even → `hash(h01, h23)` = `root` ✓
- **Tx check**: `compute_root_from_merkle_proof(h23, 1, [h01])` → position 1 is odd → `hash(h01, h23)` = `root` ✓
- **Returns `true`** — neither `h01` nor `h23` is a real leaf transaction.

The root cause is that the mitigation assumes `coinbase_tx_id` is the actual coinbase transaction (whose proof length is fixed by the real tree depth), but the code never enforces this. The equal-length guard is only meaningful when the coinbase is a genuine leaf; when both inputs are internal nodes one level above the leaves, both proofs are one step shorter than a real leaf proof, so the lengths match trivially.

---

### Impact Explanation

Any downstream contract or bridge that calls `verify_transaction_inclusion_v2` to gate asset release will accept a forged proof. An attacker can fabricate "proof" that an arbitrary internal node is a confirmed transaction in any block already stored in the light client, enabling false confirmations and unauthorized asset release. This matches the Critical scope: acceptance of a forged inclusion proof enabling false confirmations or downstream asset release.

---

### Likelihood Explanation

The attack requires only:
1. Knowledge of any block's transaction set (public on-chain data).
2. The ability to call the public `verify_transaction_inclusion_v2` NEAR contract method.
3. No privileged role, no key compromise, no social engineering.

The block's `merkle_root` is stored in the light client and the attacker computes `h01`/`h23` from public transaction data. The call is fully permissionless.

---

### Recommendation

The coinbase mitigation only works if `coinbase_tx_id` is provably a real leaf. The fix requires enforcing that the coinbase proof depth equals `ceil(log2(tx_count))` — i.e., the proof length must match the actual tree height — and that `coinbase_tx_id` is bound to a known, externally-verified coinbase txid (e.g., passed in from a trusted source or verified against a stored value). Without anchoring `coinbase_tx_id` to a real leaf, the equal-length guard is defeatable by supplying two internal nodes at the same tree level.

### Proof of Concept

```rust
// 4-tx block: tx0, tx1, tx2, tx3 (any 32-byte values)
let tx0 = H256([1u8; 32]);
let tx1 = H256([2u8; 32]);
let tx2 = H256([3u8; 32]);
let tx3 = H256([4u8; 32]);

// Build internal nodes
let h01 = compute_hash(&tx0, &tx1); // fake "coinbase_tx_id"
let h23 = compute_hash(&tx2, &tx3); // fake "tx_id"
let root = compute_hash(&h01, &h23); // real merkle_root

// Coinbase proof: h01 at position 0, sibling = h23
// compute_root_from_merkle_proof(h01, 0, [h23]) = hash(h01, h23) = root ✓

// Tx proof: h23 at position 1, sibling = h01
// compute_root_from_merkle_proof(h23, 1, [h01]) = hash(h01, h23) = root ✓

// Both proof lengths = 1, equal-length check passes ✓
// verify_transaction_inclusion_v2 returns true
// Neither h01 nor h23 is a real transaction.
assert!(verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id: h23,
    tx_block_blockhash: /* any stored block whose merkle_root == root */,
    tx_index: 1,
    merkle_proof: vec![h01],
    coinbase_tx_id: h01,
    coinbase_merkle_proof: vec![h23],
    confirmations: 1,
}));
```

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
