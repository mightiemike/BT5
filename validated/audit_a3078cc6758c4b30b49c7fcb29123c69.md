### Title
Gas-key exec-fee host functions pass wire length instead of trie-id length for ML-DSA-65 keys, causing ~59× overcharge — (`runtime/near-vm-runner/src/logic/logic.rs`)

### Summary

The host functions `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` pass the raw `public_key_len` (the borsh wire length of the encoded public key) to `gas_key_transfer_exec_fee` and `gas_key_add_key_exec_fee`. For ML-DSA-65 keys the wire length is **1953 bytes**, but the on-trie identifier is only **33 bytes** (a SHA3-256 hash). The exec-fee functions are documented to model the cost of reading/writing the **trie key**, so they must receive the trie-identifier length. The transaction path correctly calls `public_key.trie_id_len()`, but the host-function path passes `public_key_len`, producing a ~59× overcharge for every ML-DSA-65 gas-key operation issued from a contract.

---

### Finding Description

`PublicKey` exposes two distinct length methods:

- `len()` — borsh-encoded wire size: 33 / 65 / **1953** bytes for ED25519 / SECP256K1 / ML-DSA-65.
- `trie_id_len()` — on-trie identifier size: 33 / 65 / **33** bytes (ML-DSA-65 is stored as a SHA3-256 hash, not the raw key). [1](#0-0) [2](#0-1) 

The exec-fee helper `gas_key_transfer_exec_fee` is explicitly documented as computing the cost based on the **access-key trie key length**: [3](#0-2) 

It calls `access_key_key_len(account_id_len, public_key_len)`, where `public_key_len` is supposed to be the trie-identifier length. The same applies to `gas_key_add_key_exec_fee`: [4](#0-3) 

**Transaction path (correct):** `runtime/runtime/src/config.rs` calls `gas_key_add_key_exec_fee` with `public_key.trie_id_len()`: [5](#0-4) 

**Host-function path (wrong):** `runtime/near-vm-runner/src/logic/logic.rs` passes the raw `public_key_len` (wire length) to both exec-fee helpers: [6](#0-5) [7](#0-6) 

The same wrong value is passed in the wasmtime runner: [8](#0-7) [9](#0-8) 

For ED25519 and SECP256K1 keys `len() == trie_id_len()`, so the bug is latent. For ML-DSA-65 the divergence is 1953 vs 33 bytes — a factor of **~59×**.

The documentation explicitly warns about this class of mistake: [10](#0-9) 

The existing fee-parity test (`test_gas_key_fee_parity`) only exercises ED25519 keys and therefore does not catch the discrepancy: [11](#0-10) 

---

### Impact Explanation

Once `ProtocolFeature::PostQuantumSignatures` is enabled, any unprivileged contract can call `promise_batch_action_transfer_to_gas_key` (or the `add_gas_key` variants) with an ML-DSA-65 public key. The exec-fee component of the resulting action receipt is computed using 1953 bytes instead of 33 bytes for the trie-key length, producing a ~59× overcharge on the `gas_key_byte` exec-fee line. Because exec fees are fixed charges (not refunded based on actual trie I/O), the caller burns ~59× more gas than the equivalent transaction-path operation for the same key. This breaks the fee-parity invariant between the transaction and host-function paths, makes ML-DSA-65 gas-key operations economically prohibitive from contracts, and constitutes a deterministic, reproducible loss of funds for any contract user of these host functions with PQ keys.

---

### Likelihood Explanation

The trigger condition is `ProtocolFeature::PostQuantumSignatures` being active. The feature is fully implemented in the production codebase and gated only by protocol version. Once stabilized and voted in, every contract on the network can reach the overcharge path without any privileged role. The bug is silent — no error is returned, the transaction succeeds, and the overcharge is invisible to the caller.

---

### Recommendation

Replace `public_key_len as usize` with the decoded key's `trie_id_len()` in all three exec-fee call sites in both `logic.rs` and `wasmtime_runner/logic.rs`. The decoded `public_key` is already available at that point in each function:

```rust
// Before (wrong — wire length):
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,
);

// After (correct — trie-identifier length):
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key.decode()?.trie_id_len(),
);
```

Apply the same fix to the `gas_key_add_key_exec_fee` call sites. Add a variant of `test_gas_key_fee_parity` that exercises ML-DSA-65 keys to prevent regression.

---

### Proof of Concept

1. Enable `PostQuantumSignatures` (nightly build or after stabilization).
2. Deploy a contract that calls `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 public key (borsh-encoded length = 1953 bytes).
3. Observe `gas_burnt` on the resulting action receipt.
4. Issue the identical `TransferToGasKey` action via a signed transaction (which routes through `permission_exec_fees` → `public_key.trie_id_len()` = 33 bytes).
5. Compare the two `gas_burnt` values: the host-function path charges `gas_key_byte_exec_fee × (account_id_len + 2 + 1953 + value_len)` while the transaction path charges `gas_key_byte_exec_fee × (account_id_len + 2 + 33 + value_len)`. The ratio of the per-byte components is 1953/33 ≈ 59×, matching the `len()` vs `trie_id_len()` divergence documented in `core/crypto/src/signature.rs`. [12](#0-11) [13](#0-12)

```rust
/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
```

```markdown
### 5. Storage usage and fee plumbing

The storage-stake calculation

trie-byte-priced fee path was updated to call `trie_id_len()`.
```
