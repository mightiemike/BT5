### Title
`Airdrop._claim()` Hardcodes `msg.sender` as Recipient, Permanently Locking Allocations for Token-Blacklisted Users — (File: `core/contracts/Airdrop.sol`)

### Summary

`Airdrop._claim()` unconditionally transfers the airdrop token to `msg.sender` with no `recipient` parameter. If the claimant's address is blacklisted by the airdrop token (e.g., USDC), the transfer reverts and the user's entire allocation becomes permanently inaccessible with no recovery path.

### Finding Description

`_claim()` hardcodes `msg.sender` as the transfer destination:

```solidity
function _claim(
    uint32 week,
    uint256 totalAmount,
    bytes32[] calldata proof
) internal {
    _verifyProof(week, msg.sender, totalAmount, proof);
    SafeERC20.safeTransfer(IERC20(token), msg.sender, totalAmount);  // <-- hardcoded
    emit Claim(msg.sender, week, totalAmount);
}
``` [1](#0-0) 

The public entry point `claim()` provides no mechanism to specify an alternative recipient:

```solidity
function claim(ClaimProof[] calldata claimProofs) external {
    for (uint32 i = 0; i < claimProofs.length; i++) {
        _claim(claimProofs[i].week, claimProofs[i].totalAmount, claimProofs[i].proof);
    }
}
``` [2](#0-1) 

The Merkle leaf is constructed from `(sender, totalAmount)`, binding the allocation irrevocably to the claimant's address:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(keccak256(abi.encode(sender, totalAmount)))
);
``` [3](#0-2) 

The contract does check the OFAC sanctions list, but USDC's internal token blacklist is entirely separate from OFAC sanctions. A user can pass the sanctions check and still be blacklisted at the token level:

```solidity
require(
    !ISanctionsList(sanctions).isSanctioned(sender),
    "address is sanctioned."
);
``` [4](#0-3) 

Because `claimed[week][sender]` is set inside `_verifyProof` before the transfer, a revert on `safeTransfer` rolls back the entire transaction — the state is not corrupted, but the user is stuck in an infinite retry loop that always reverts. There is no admin recovery function, no alternative claim path, and no way to redirect the allocation to a non-blacklisted address.

### Impact Explanation

A user whose address is blacklisted by the airdrop token (e.g., USDC) permanently loses their entire airdrop allocation. The Merkle tree is keyed to their address, so no other address can claim on their behalf. The tokens remain locked in the `Airdrop` contract with no protocol-level recovery mechanism. This is a direct, permanent asset loss for the affected user.

### Likelihood Explanation

USDC blacklisting is a documented, real-world event that has affected hundreds of addresses. Any protocol distributing USDC (or any token with a transfer blacklist) as an airdrop is exposed to this scenario. The trigger requires only that a claimant's address be blacklisted between the time the Merkle snapshot is taken and the time they attempt to claim — a realistic window that can span weeks or months.

### Recommendation

Add an optional `recipient` parameter to `_claim()` and `claim()`. If `recipient` is `address(0)`, default to `msg.sender`. The proof verification must remain anchored to `msg.sender` to prevent unauthorized claims:

```diff
 function _claim(
     uint32 week,
     uint256 totalAmount,
-    bytes32[] calldata proof
+    bytes32[] calldata proof,
+    address recipient
 ) internal {
     _verifyProof(week, msg.sender, totalAmount, proof);
+    if (recipient == address(0)) recipient = msg.sender;
-    SafeERC20.safeTransfer(IERC20(token), msg.sender, totalAmount);
+    SafeERC20.safeTransfer(IERC20(token), recipient, totalAmount);
-    emit Claim(msg.sender, week, totalAmount);
+    emit Claim(msg.sender, week, totalAmount);
 }
```

### Proof of Concept

1. Protocol deploys `Airdrop` with USDC as the airdrop token.
2. Week 1 Merkle snapshot is taken; Alice's address `0xAlice` is included with `totalAmount = 1000e6`.
3. Before Alice claims, her address is added to USDC's blacklist (e.g., due to a regulatory action unrelated to OFAC).
4. Alice calls `claim([{week: 1, totalAmount: 1000e6, proof: [...]}])`.
5. `_verifyProof` passes — Alice is not on the OFAC sanctions list.
6. `SafeERC20.safeTransfer(IERC20(usdc), 0xAlice, 1000e6)` reverts because USDC's `transfer` checks its own blacklist.
7. The entire transaction reverts; `claimed[1][0xAlice]` remains `0`.
8. Alice retries indefinitely — every attempt reverts at step 6.
9. Alice's 1000 USDC allocation is permanently locked in the `Airdrop` contract with no recovery path.
