### Title
Deprecated `verify_transaction_inclusion` Ignores Coinbase Proof Requirement, Enabling 64-Byte Transaction Merkle Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function remains publicly callable on-chain and omits the coinbase Merkle proof validation that `verify_transaction_inclusion_v2` introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability. Any unprivileged NEAR caller can invoke the deprecated path and receive a `true` result for a fabricated `tx_id` that corresponds to an internal Merkle tree node rather than a real transaction.

---

### Finding Description

`verify_transaction_inclusion` (lines 288–323 in `contract/src/lib.rs`) verifies transaction inclusion by computing the Merkle root from the caller-supplied `tx_id` and `merkle_proof` and comparing it to the block's stored `merkle_root`. The function itself documents the gap:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

The function is still `pub`, gated only by `#[pause]` (not paused by default), and carries no access-control restriction: [2](#0-1) 

The Merkle root comparison at the end of the function accepts any hash that traverses to the stored root, whether it originates from a leaf (real transaction) or an internal node: [3](#0-2) 

`verify_transaction_inclusion_v2` (lines 347–369) was introduced specifically to close this gap. It requires a coinbase Merkle proof — a proof that the coinbase transaction (always at index 0) is included in the same block. Because the coinbase is always a leaf, a valid coinbase proof cryptographically constrains the Merkle tree to have real leaf nodes, making it impossible for an internal node to be passed off as a transaction hash: [4](#0-3) 

The deprecated function completely ignores this coinbase proof requirement. This is structurally identical to the Pyth analog: the coinbase proof is the quality/validity indicator (analogous to the confidence interval `σ`) that distinguishes a valid transaction hash from an uncertain/forged one. The deprecated path discards it entirely.

The Merkle computation itself in `merkle-tools/src/lib.rs` is position-driven and has no awareness of whether the starting hash is a leaf or an internal node: [5](#0-4) 

---

### Impact Explanation

An attacker can call `verify_transaction_inclusion` with a fabricated `tx_id` equal to an internal Merkle tree node hash, paired with a valid Merkle proof from that node to the root. The function returns `true`, falsely asserting that the fabricated transaction is included in the block. Any downstream NEAR contract that consumes this result for cross-chain Bitcoin transaction verification — e.g., to release funds, mint tokens, or record a settlement — is deceived into acting on a non-existent transaction.

The corrupted value is the boolean proof result returned to the caller: a `true` that should be `false`.

---

### Likelihood Explanation

The 64-byte transaction Merkle forgery is a well-documented, publicly known attack vector (referenced in the contract's own deprecation notice and the [BitMEX blog post](https://www.bitmex.com/blog/64-Byte-Transactions)). Exploiting it requires only:

1. Knowledge of a block's Merkle tree structure — entirely public on-chain Bitcoin data.
2. The ability to call a public NEAR contract function — no privileged role, no staking, no special key.

The deprecated function is reachable by any unprivileged NEAR account at any time the contract is not paused.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract entirely, or restrict it to a privileged role so it cannot be called by arbitrary callers. If backward compatibility is required, redirect its implementation to `verify_transaction_inclusion_v2` internally, so the coinbase proof is always enforced. The deprecation marker alone does not prevent on-chain invocation.

---

### Proof of Concept

1. Select any Bitcoin mainchain block stored in the light client that has more than one transaction (so the Merkle tree has internal nodes). The block hash must be present in `mainchain_header_to_height`.
2. Obtain the block's full transaction list from a Bitcoin node. Identify an internal Merkle tree node hash `N` at depth `d` and position `p` in the tree.
3. Construct a Merkle proof from `N` upward to the root — this is a standard `(d - depth)` element proof, computable from public data.
4. Call `verify_transaction_inclusion` on NEAR with:
   - `tx_id = N` (the internal node hash, not a real transaction)
   - `tx_block_blockhash` = the block's hash
   - `tx_index` = `p` (the position of `N` at its level)
   - `merkle_proof` = the proof constructed in step 3
   - `confirmations` = any value ≤ current tip height − block height + 1
5. `compute_root_from_merkle_proof(N, p, proof)` traverses correctly to the stored `merkle_root`; the function returns `true`. [6](#0-5) 

A downstream contract receiving this `true` result has no way to distinguish it from a legitimately verified transaction inclusion.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
