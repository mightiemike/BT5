### Title
Stale `lastLiquidationFees` Withholds Insurance from Socialization When Finalizing a Different Subaccount — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a persistent storage variable that is only ever written by `_handleLiquidationPayment` and is **never reset to zero**. When `_finalizeSubaccount` is called for a subaccount that had no preceding `_handleLiquidationPayment` call of its own, the stale value from a prior liquidation of a *different* subaccount is silently subtracted from the insurance amount passed to `perpEngine.socializeSubaccount`. This causes the protocol to socialize more loss onto market participants than is necessary, violating the invariant that all available insurance must be exhausted before any socialization occurs.

---

### Finding Description

`lastLiquidationFees` is declared as a storage slot in `ClearinghouseStorage`: [1](#0-0) 

It is written exactly once, at the end of `_handleLiquidationPayment`: [2](#0-1) 

The comment at lines 581–585 explains the design intent: the fees from the *current* liquidation step are withheld from socialization to prevent a re-entrancy-like accounting loop where insurance is refilled by fees and then immediately consumed.

Inside `_finalizeSubaccount`, the pattern is:

```
v.insurance = insurance;
v.insurance -= lastLiquidationFees;   // (1) withhold stale fees
...
v.insurance = perpEngine.socializeSubaccount(txn.liquidatee, v.insurance);
...
v.insurance += lastLiquidationFees;   // (2) add them back
insurance = v.insurance;
``` [3](#0-2) 

`_finalizeSubaccount` **never resets `lastLiquidationFees`**. After finalization of subaccount A, the variable retains `fees_A`.

`liquidateSubaccountImpl` calls `_finalizeSubaccount` first, before `_handleLiquidationPayment`: [4](#0-3) 

A subaccount with only a negative USDC balance (no perp positions, no spot assets) satisfies all preconditions for `_finalizeSubaccount` to proceed — all perp `balance.amount == 0` checks pass, all spot asset checks pass — and can be finalized **without any preceding `_handleLiquidationPayment` call for that subaccount**: [5](#0-4) 

In `perpEngine.socializeSubaccount`, the `insurance` argument is the only source of coverage before socialization kicks in: [6](#0-5) 

If `insurance` is understated by a stale `lastLiquidationFees`, the shortfall is socialized across all open-interest holders instead of being covered by the fund.

---

### Impact Explanation

**Concrete scenario:**

| Step | Action | `insurance` (storage) | `lastLiquidationFees` |
|------|--------|-----------------------|-----------------------|
| 0 | Initial state | 100 | 0 |
| 1 | Liquidate subaccount A (normal step) | 200 | 100 |
| 2 | Finalize subaccount A (A had 100 debt) | 100 | **100 (not reset)** |
| 3 | Finalize subaccount B directly (B has 150 USDC debt, no other positions) | — | — |

At step 3:
- `v.insurance = 100 − 100 = 0` (stale subtraction)
- `socializeSubaccount(B, 0)` → all 150 is socialized; returns 0
- `v.insurance += 100 = 100`
- `insurance = 100`

**Correct behavior:** 100 of insurance should have covered 100 of B's 150 debt; only 50 should have been socialized.

**Actual behavior:** 150 is socialized; the insurance fund retains 100 it should have spent. Perp market participants absorb 100 units of loss that the insurance fund was obligated to cover first.

This is an incorrect value transfer: loss is shifted from the insurance fund to open-interest holders, violating the protocol's own socialization invariant.

---

### Likelihood Explanation

The conditions are realistic and reachable in normal protocol operation:

1. Any prior liquidation (of any subaccount) leaves `lastLiquidationFees > 0`.
2. A subaccount with only a negative USDC balance and no open positions is a common end-state after partial liquidation sequences or after a subaccount's positions are closed by other means.
3. The sequencer submits `liquidateSubaccountImpl` with `productId == type(uint32).max` for such a subaccount — a standard finalization call.
4. The condition `insurance < lastLiquidationFees + B_debt` is easily satisfied when the insurance fund is under stress (exactly the scenario where socialization matters most).

No special privileges, governance capture, or key compromise are required.

---

### Recommendation

Reset `lastLiquidationFees` to zero at the start of `_finalizeSubaccount` (or immediately after the `v.insurance += lastLiquidationFees` / `insurance = v.insurance` write at lines 410–411). This ensures that a finalization call that has no associated `_handleLiquidationPayment` for the same subaccount does not inherit a stale fee reservation:

```solidity
// at the end of _finalizeSubaccount, after insurance = v.insurance:
lastLiquidationFees = 0;
```

Alternatively, scope `lastLiquidationFees` to the liquidatee being processed (e.g., store it as a mapping keyed by subaccount) so that a finalization of subaccount B never reads fees recorded for subaccount A.

---

### Proof of Concept

```solidity
// 1. insurance = 100, lastLiquidationFees = 0

// 2. Liquidate subaccount A (normal step, fees = 100):
//    insurance = 200, lastLiquidationFees = 100

// 3. Finalize subaccount A (A has 100 USDC debt):
//    v.insurance = 200 - 100 = 100
//    socializeSubaccount(A, 100) → covers 100, returns 0
//    v.insurance = 0 + 100 = 100
//    insurance = 100
//    lastLiquidationFees = 100  ← NOT reset

// 4. Subaccount B: only -150 USDC, no perp/spot positions, under maintenance.
//    liquidateSubaccountImpl(B, productId=type(uint32).max)
//    → _finalizeSubaccount called directly (no _handleLiquidationPayment for B)
//    v.insurance = 100 - 100 = 0   ← stale subtraction
//    socializeSubaccount(B, 0)     ← 150 fully socialized, 0 insurance used
//    v.insurance = 0 + 100 = 100
//    insurance = 100               ← fund retains 100 it should have spent

// Expected: 100 covered by insurance, 50 socialized.
// Actual:   150 socialized, 100 retained in insurance fund.
// Delta:    100 units incorrectly transferred from participants to insurance fund.
```
