### Title
IBC NFT Owner Cannot Burn (or Transfer/Edit) Their Token — `MsgBurnNFT.ValidateBasic` Calls `ValidateDenomID` Instead of `ValidateDenomIDWithIBC` — (`x/nft/types/msgs.go`)

### Summary

`ValidateGenesis` correctly uses `ValidateDenomIDWithIBC` to accept `ibc/<hash>` denom IDs, but every message `ValidateBasic` — including `MsgBurnNFT`, `MsgTransferNFT`, and `MsgEditNFT` — calls the stricter `ValidateDenomID`, which unconditionally rejects any IBC denom string. An IBC NFT stored in chain state is therefore permanently unburnable, untransferable, and uneditable by its owner.

### Finding Description

`ValidateDenomID` enforces three constraints that an IBC denom `ibc/<64-char-hex>` (total length 68) can never satisfy:

1. **Length**: `MaxDenomLen = 64`; an IBC denom is `IBCDenomLen = 68` chars. [1](#0-0) 
2. **Character set**: `IsAlphaNumeric` accepts only `[a-z0-9]`; the `/` separator and uppercase hex digits both fail. [2](#0-1) 
3. **Prefix**: `IsBeginWithAlpha` requires `[a-z]`; `ibc/` begins with `i` (passes), but the slash and uppercase hex still fail the alphanumeric check.

`ValidateDenomIDWithIBC` exists precisely to handle this case — it short-circuits on the `ibc/` prefix and validates only the hex hash: [3](#0-2) 

`ValidateGenesis` correctly calls `ValidateDenomIDWithIBC`: [4](#0-3) 

But all three mutating message types call the wrong validator:

- `MsgBurnNFT.ValidateBasic` line 221: [5](#0-4) 
- `MsgTransferNFT.ValidateBasic` line 86: [6](#0-5) 
- `MsgEditNFT.ValidateBasic` line 135: [7](#0-6) 

`ValidateDenomIDWithIBC` is **never** called from any message `ValidateBasic`. [3](#0-2) 

### Impact Explanation

Any NFT whose denom was stored under `ibc/<hash>` — whether via `InitGenesis` or via an IBC NFT-transfer packet that writes directly to the keeper — is permanently frozen. The owner's signed `MsgBurnNFT`, `MsgTransferNFT`, and `MsgEditNFT` transactions are all rejected at `ValidateBasic` before they reach the keeper. The NFT cannot be destroyed, moved, or modified. The owner loses all authorized control over the asset with no recovery path.

### Likelihood Explanation

The path is concrete and requires no privilege:
1. Chain starts (or migrates) with genesis containing an IBC-origin NFT collection under `ibc/<hash>`. `ValidateGenesis` accepts it.
2. The legitimate owner constructs `MsgBurnNFT{DenomId: "ibc/ABCD..."}` and signs it.
3. `ValidateBasic` calls `ValidateDenomID`, which returns `ErrInvalidDenom` (length 68 > 64, non-alphanumeric characters).
4. The transaction is rejected by every node's ante-handler before execution. No state change occurs, ever.

This is reproducible with a standard Go/Cosmos SDK unit test with no special setup.

### Recommendation

Replace `ValidateDenomID` with `ValidateDenomIDWithIBC` in the `ValidateBasic` methods of `MsgBurnNFT`, `MsgTransferNFT`, and `MsgEditNFT` in `x/nft/types/msgs.go`. This is the same fix already applied to `ValidateGenesis`.

func ValidateDenomIDWithIBC(denomID string) error {
	if strings.HasPrefix(denomID, IBCPrefix) {
		if len(denomID) != IBCDenomLen {
			return sdkerrors.Wrapf(ErrInvalidDenom, "the length of ibc denom(%s) only accepts value [%d]", denomID, IBCDenomLen)
		}
		if _, err := hex.DecodeString(denomID[len(IBCPrefix):]); err != nil {
			return sdkerrors.Wrapf(ErrInvalidDenom, "the hash of ibc denom(%s) must be valid hex", denomID)
		}

		return nil
	}

	return ValidateDenomID(denomID)
}
