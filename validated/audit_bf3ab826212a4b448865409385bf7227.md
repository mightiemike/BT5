Audit Report

## Title
Router-mediated swaps pass the router address as `sender` to `SwapAllowlistExtension`, breaking per-user allowlist gating — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` from the pool's perspective. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user. The hook therefore checks whether the router is allowlisted, not the actual economic actor. This creates two mutually exclusive broken configurations: allowlisting the router opens the pool to all users, while allowlisting individual users blocks them from using the router.

## Finding Description

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the `sender` argument:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- router address when called via MetricOmmSimpleRouter
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` passes this value unchanged as the first argument to the extension via `abi.encodeCall`:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

In `MetricOmmSimpleRouter.exactInputSingle`, the pool's `swap` is called with the router as `msg.sender`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The same applies to `exactInput` (L104), `exactOutputSingle` (L136), and `exactOutput` (L165). No existing guard in the extension, pool, or router propagates the original user's identity to the hook. The router does not encode `msg.sender` into `extensionData`, and the extension has no mechanism to verify a user-supplied identity claim.

## Impact Explanation

**Bypass path (allowlist rendered ineffective):** If the pool admin allowlists the router address — the natural step to enable router-mediated swaps — then every user, including those never individually allowlisted, can bypass the per-user gate by routing through `MetricOmmSimpleRouter`. The `SwapAllowlistExtension` is designed to gate access to curated pools (KYC-gated, institutional, or restricted). Allowlisting the router silently opens the pool to all users, completely defeating the extension's purpose.

**Denial path (allowlisted users blocked from router):** If the pool admin allowlists only individual user addresses and not the router, those allowlisted users cannot use the router at all — the hook sees the router address and reverts with `NotAllowedToSwap()`. The two configurations are mutually exclusive; there is no way to simultaneously allow specific users through the router while blocking others.

Both outcomes break the core access-control invariant of the extension and constitute broken core pool functionality for any pool deploying `SwapAllowlistExtension` alongside the official router.

## Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` and also wants to support `MetricOmmSimpleRouter` faces this dilemma immediately upon configuration. Allowlisting the router is the obvious and predictable step an admin takes to enable router support, and doing so silently opens the pool to all users. This requires no special attacker capability — any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` with a restricted pool as the target. The condition is reachable in normal production use.

## Recommendation

The `beforeSwap` hook should not rely on `sender` (the immediate caller) to identify the economic actor. Options:

1. Have the router encode `msg.sender` (the user) into `extensionData` before calling the pool, and redesign the extension to decode and verify this value — potentially with a signature or trusted-router registry to prevent spoofing.
2. Require direct pool interaction for allowlisted pools and document that `MetricOmmSimpleRouter` is incompatible with `SwapAllowlistExtension`.
3. Redesign the extension to check `recipient` or a signed attestation rather than `sender`, if the pool's use case permits recipient-based gating.

## Proof of Concept

```
Bypass path:
1. Deploy pool P with SwapAllowlistExtension; call setAllowedToSwap(P, router, true)
   (admin enables router support — the natural configuration).
2. Non-allowlisted attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...}).
3. Router calls P.swap(recipient, ...) — msg.sender to pool = router address.
4. Pool calls SwapAllowlistExtension.beforeSwap(sender=router, ...).
5. Hook checks allowedSwapper[P][router] → true → passes.
6. Attacker completes swap on a pool they were never individually allowlisted for.

Denial path:
1. Deploy pool P with SwapAllowlistExtension; call setAllowedToSwap(P, userA, true)
   (router NOT allowlisted).
2. UserA calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...}).
3. Hook checks allowedSwapper[P][router] → false → reverts NotAllowedToSwap().
4. UserA is blocked from using the router despite being individually allowlisted.
```
