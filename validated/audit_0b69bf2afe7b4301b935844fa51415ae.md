### Title
User-Supplied Denom ID Squatting via Transaction Priority in `MsgIssueDenom` — (`x/nft/keeper/denom.go`)

---

### Summary

The `x/nft` module's `MsgIssueDenom` handler accepts a fully user-controlled `id` string and registers it globally on a first-come-first-served basis. Because any unprivileged account can submit this message, an attacker who observes a pending `MsgIssueDenom` in the mempool can submit an identical `id` with a higher fee, be included first by the priority mempool, and permanently claim the denom ID — becoming its sole creator and gaining exclusive minting rights, while the victim's transaction fails.

---

### Finding Description

`MsgIssueDenom` is an open, permissionless Cosmos SDK message. Its handler calls `Keeper.IssueDenom`, which delegates directly to `SetDenom`:

```go
// x/nft/keeper/keeper.go
func (k Keeper) IssueDenom(ctx sdk.Context, id, name, schema, uri string, creator sdk.AccAddress) error {
    return k.SetDenom(ctx, types.NewDenom(id, name, schema, uri, creator))
}
```

`SetDenom` enforces uniqueness of both `id` and `name`, but does **not** bind either to the sender's address:

```go
// x/nft/keeper/denom.go
func (k Keeper) SetDenom(ctx sdk.Context, denom types.Denom) error {
    if k.HasDenomID(ctx, denom.Id) {
        return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomID %s has already exists", denom.Id)
    }
    if k.HasDenomNm(ctx, denom.Name) {
        return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomName %s has already exists", denom.Name)
    }
    store := ctx.KVStore(k.storeKey)
    bz := k.cdc.MustMarshal(&denom)
    store.Set(types.KeyDenomID(denom.Id), bz)
    store.Set(types.KeyDenomName(denom.Name), []byte(denom.Id))
    return nil
}
``` [1](#0-0) 

The `id` field is a plain lowercase alphanumeric string chosen entirely by the user:

```go
// x/nft/types/validation.go
func ValidateDenomID(denomID string) error {
    if len(denomID) < MinDenomLen || len(denomID) > MaxDenomLen { ... }
    if !IsBeginWithAlpha(denomID) || !IsAlphaNumeric(denomID) { ... }
    return nil
}
``` [2](#0-1) 

There is no binding between the denom `id` and the sender's address at any point in the message pipeline: [3](#0-2) 

The Cosmos SDK priority mempool orders transactions by fee. An attacker who monitors the mempool for a `MsgIssueDenom` with a desirable `id` can resubmit the same `id` with a higher fee, be included first, and register themselves as the denom's `creator`.

---

### Impact Explanation

The `creator` field stored in the `Denom` struct is the sole authorization gate for all subsequent NFT minting under that denom:

```go
// x/nft/keeper/keeper.go
func (k Keeper) MintNFT(..., sender, owner sdk.AccAddress) error {
    _, err := k.IsDenomCreator(ctx, denomID, sender)
    if err != nil {
        return err
    }
    return k.MintNFTUnverified(...)
}
``` [4](#0-3) 

By squatting the denom ID, the attacker:
1. Permanently becomes the sole creator of that denom — the only address that can ever mint NFTs under it.
2. Causes the victim's transaction to fail with `ErrInvalidDenom: denomID X has already exists`, consuming the victim's fee.
3. Prevents the victim from ever using their intended denom ID on this chain (IDs are globally unique and immutable once set).

The corrupted on-chain value is `Denom.Creator` — it is set to the attacker's address instead of the victim's. [5](#0-4) 

---

### Likelihood Explanation

- `MsgIssueDenom` is a standard, publicly documented CLI command (`chain-maind tx nft issue <denom-id>`), so denom creation transactions are common and observable.
- Denom IDs are short, human-readable strings (e.g., `myproject`, `artcollection`), making them predictable and worth squatting.
- The Cosmos SDK priority mempool (used in this chain) orders transactions by fee, making fee-based front-running straightforward.
- No special privilege is required — any funded account can execute this attack. [6](#0-5) 

---

### Recommendation

Bind the denom ID to the creator's address by deriving it deterministically from the sender, preventing any other account from claiming the same ID:

```go
// Derive denom ID as hash(sender_address + user_chosen_suffix)
denomID := fmt.Sprintf("%x", sha256.Sum256(append(creator.Bytes(), []byte(userSuffix)...)))
```

Alternatively, enforce that the denom `id` must be prefixed with the creator's bech32 address or a hash thereof, so no two accounts can produce the same `id`.

---

### Proof of Concept

1. Alice broadcasts:
   ```
   chain-maind tx nft issue artcollection --name="Art Collection" --from=alice --fees=1000basecro
   ```
2. Bob observes Alice's pending transaction in the mempool.
3. Bob broadcasts:
   ```
   chain-maind tx nft issue artcollection --name="Art Collection" --from=bob --fees=2000basecro
   ```
4. Bob's transaction is included first (higher fee → higher priority).
5. Alice's transaction fails: `ErrInvalidDenom: denomID artcollection has already exists`.
6. Bob is now the `creator` of `artcollection` and is the only address that can ever call `MintNFT` under that denom. [1](#0-0) [7](#0-6) [8](#0-7)

func (m msgServer) IssueDenom(goCtx context.Context, msg *types.MsgIssueDenom) (*types.MsgIssueDenomResponse, error) {
	sender, err := sdk.AccAddressFromBech32(msg.Sender)
	if err != nil {
		return nil, err
	}

	ctx := sdk.UnwrapSDKContext(goCtx)
	if err := m.Keeper.IssueDenom(ctx, msg.Id, msg.Name, msg.Schema, msg.Uri, sender); err != nil {
		return nil, err
	}

	ctx.EventManager().EmitEvents(sdk.Events{
		sdk.NewEvent(
			types.EventTypeIssueDenom,
			sdk.NewAttribute(types.AttributeKeyDenomID, msg.Id),
			sdk.NewAttribute(types.AttributeKeyDenomName, msg.Name),
			sdk.NewAttribute(types.AttributeKeyCreator, msg.Sender),
		),
		sdk.NewEvent(
			sdk.EventTypeMessage,
			sdk.NewAttribute(sdk.AttributeKeyModule, types.AttributeValueCategory),
			sdk.NewAttribute(sdk.AttributeKeySender, msg.Sender),
		),
	})

	return &types.MsgIssueDenomResponse{}, nil
}
