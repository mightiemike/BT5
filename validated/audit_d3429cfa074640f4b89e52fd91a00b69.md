Audit Report

## Title
`SwapAllowlistExtension` checks intermediary `sender` (router) instead of end-user, allowing any router caller to bypass per-user swap gate - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on `sender`, which the pool sets to `msg.sender` of `pool.swap()`. When swaps route through `MetricOmmSimpleRouter`, `sender` is the router contract address. A pool admin who allowlists the router — the standard step to enable router-based swaps — silently grants every router user unrestricted swap access, defeating the per-user allowlist entirely. The `recipient` parameter, which identifies the economic beneficiary, is available in `beforeSwap` but is ignored.

## Finding Description

**Call chain confirmation:**

`MetricOmmPool.swap` passes `msg.sender` (the direct caller of `pool.swap()`) as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards `sender` verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender /* pool */][sender /* caller of pool.swap() */]`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` of that call: [4](#0-3) 

Therefore `allowedSwapper[pool][router]` is the value checked — not `allowedSwapper[pool][end_user]`. Once the router is allowlisted, the check passes for every caller of the router.

**Contrast with `DepositAllowlistExtension`:**

The deposit-side extension correctly checks `owner` (the actual position beneficiary), not `sender` (the operator/router): [5](#0-4) 

The `recipient` parameter is already present as the second argument in `beforeSwap` but is silently discarded (`address,`), so the fix requires no interface change. [6](#0-5) 

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of counterparties (e.g., KYC'd institutions, whitelisted market-makers) is silently opened to the entire public the moment the pool admin allowlists the router. Any user can call `MetricOmmSimpleRouter.exactInputSingle` / `exactOutputSingle` / `exactInput` / `exactOutput` and the extension will pass them through. Unauthorized traders can execute swaps at oracle-anchored prices, causing direct LP principal loss from a pool explicitly configured to prevent open access. This constitutes broken core pool functionality and direct loss of LP assets above Sherlock thresholds.

## Likelihood Explanation

Any pool that (a) deploys `SwapAllowlistExtension` as a `beforeSwap` hook and (b) allowlists the router is vulnerable. Allowlisting the router is the standard operational step to enable router-based swaps; the pool admin has no on-chain signal that doing so collapses the per-user gate. The inconsistency with `DepositAllowlistExtension` (which checks `owner`, not `sender`) makes the mistake easy to miss during review. The exploit is repeatable by any unprivileged user with access to the router.

## Recommendation

Gate on `recipient` (the address receiving output tokens) instead of `sender`. `recipient` is already the second parameter of `beforeSwap` and represents the economic beneficiary of the swap — analogous to `owner` in `beforeAddLiquidity`. Change line 37 of `SwapAllowlistExtension.sol`:

```solidity
// current (broken)
function beforeSwap(address sender, address, bool, ...)
    ...
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {

// fixed
function beforeSwap(address, address recipient, bool, ...)
    ...
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
```

Alternatively, document and enforce that allowlisted pools must be called directly (not through a router that separates payer from recipient), and add a `require(sender == recipient)` guard for allowlisted pools.

## Proof of Concept

```
1. Pool admin creates pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls extension.setAllowedToSwap(pool, router, true)
   — intending to "enable router support."
3. Unauthorized user (not individually allowlisted) calls:
       router.exactInputSingle(ExactInputSingleParams{pool: pool, recipient: user, ...})
4. Router calls pool.swap(recipient=user, ...); msg.sender of pool.swap() = router.
5. Pool passes msg.sender (router) as `sender` to _beforeSwap → extension.
6. Extension checks allowedSwapper[pool][router] == true → passes.
7. Unauthorized user receives output tokens; LP suffers loss.
8. Verification: direct call pool.swap() by the same unauthorized user reverts
   (allowedSwapper[pool][user] == false), confirming the bypass is router-specific.
```
