### Title
Protobuf `ValidResourceBounds` Deserialization Uses Inline Value Heuristic Instead of Canonical Variant Discriminant, Corrupting Transaction Hash Preimage - (File: crates/apollo_protobuf/src/converters/transaction.rs)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation reconstructs the `ValidResourceBounds` variant using an inline value-based heuristic (`l1_data_gas.is_zero() && l2_gas.is_zero()`) instead of the canonical discriminant (transaction version / presence of the `l1_data_gas` field). This mirrors the external bug exactly: an inline formula is substituted for the designated canonical function. For any `AllResources` transaction whose `l2_gas` and `l1_data_gas` happen to be zero, the protobuf round-trip silently changes the variant to `L1Gas`. Because `get_tip_resource_bounds_hash` branches on the variant to decide whether to include the `L1_DATA_GAS` concat element in the Poseidon hash preimage, the reconstructed transaction produces a structurally different hash than the one computed at admission, breaking the hash/signature domain invariant.

### Finding Description

**Serialization side** (`From<ValidResourceBounds> for protobuf::ResourceBounds`):

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),       // always zero for L1Gas
    l1_data_gas: Some(ResourceBounds::default().into()), // always zero for L1Gas
},
ValidResourceBounds::AllResources(...) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(l2_gas.into()),
    l1_data_gas: Some(l1_data_gas.into()),
},
``` [1](#0-0) 

**Deserialization side** (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`):

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [2](#0-1) 

The canonical discriminant is the **transaction version / format**, not the runtime values of the bounds. The `ValidResourceBounds::L1Gas` variant is explicitly documented as "Pre 0.13.3. Only L1 gas." — it is a version-level concept, not a value-level concept. [3](#0-2) 

**Hash preimage divergence** — `get_tip_resource_bounds_hash` branches on the variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // L1DataGas omitted
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // L1DataGas included
    }
});
``` [4](#0-3) 

For an `AllResources` transaction with `l2_gas = 0` and `l1_data_gas = 0`:
- **At admission**: hash computed over `[tip, L1_GAS_concat, L2_GAS_concat(0), L1_DATA_GAS_concat(0)]` — 4 elements
- **After protobuf round-trip**: variant silently becomes `L1Gas`; hash recomputed over `[tip, L1_GAS_concat, L2_GAS_concat(0)]` — 3 elements

These are structurally different Poseidon hash inputs, producing a different `TransactionHash`.

Additionally, `get_gas_vector_computation_mode()` returns `GasVectorComputationMode::NoL2Gas` for `L1Gas` and `GasVectorComputationMode::All` for `AllResources`, affecting execution gas accounting. [5](#0-4) 

### Impact Explanation

**Wrong state / receipt / revert result from blockifier/execution logic** and **wrong compiled class / hash selected for execution**:

1. A transaction submitted as `AllResources` (Starknet ≥ 0.13.3) with `l2_gas.max_amount = 0` and `l1_data_gas.max_amount = 0` is admitted with a hash `H_allresources`.
2. When the transaction is serialized to protobuf for P2P consensus propagation and then deserialized by a peer, the variant is reconstructed as `L1Gas`.
3. Any component that recomputes the hash from the deserialized transaction (e.g., for block hash construction, OS hint `AssertTransactionHash`, or signature verification) will compute `H_l1gas ≠ H_allresources`.
4. The OS `AssertTransactionHash` hint will fail or accept the wrong hash, producing an incorrect execution result or an invalid block.
5. The `GasVectorComputationMode` mismatch (`NoL2Gas` vs `All`) causes the blockifier to apply the wrong gas accounting path, producing wrong receipts and fees.

### Likelihood Explanation

Any user submitting a V3 `AllResources` transaction (the only format accepted by the RPC since Starknet 0.13.3) with `l2_gas.max_amount = 0` and `l1_data_gas.max_amount = 0` triggers this path. This is a valid, well-formed transaction — the gateway does not reject zero-amount bounds. The trigger requires no privilege and is reachable through the normal public RPC submission path.

### Recommendation

Replace the inline value heuristic with the canonical discriminant. The correct approach is to preserve the variant identity through the protobuf wire format — either by adding an explicit discriminant field to the protobuf schema, or by deriving the variant from the transaction version field that is already present in the enclosing transaction message:

```rust
// Instead of:
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(...)
})

// Use the transaction version carried by the enclosing message, or add an
// explicit `resource_bounds_version` field to protobuf::ResourceBounds, so
// the variant is reconstructed from its origin, not from runtime values.
```

Alternatively, always reconstruct as `AllResources` when deserializing from a V3 transaction context (since `L1Gas` is a pre-0.13.3 format and should never appear in a V3 protobuf message), and only reconstruct as `L1Gas` when the enclosing transaction version is < 3.

### Proof of Concept

1. Construct an `RpcInvokeTransactionV3` with `resource_bounds = AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`.
2. Submit to gateway → `convert_rpc_tx_to_internal` computes `tx_hash` using `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `AllResources` → hash preimage includes `L1_DATA_GAS_concat(0)` → produces `H_allresources`.
3. Serialize the `InternalConsensusTransaction` to protobuf via `From<ValidResourceBounds> for protobuf::ResourceBounds` → `l1_data_gas = ResourceBounds::default()` (zero), `l2_gas = 0`.
4. Deserialize on a peer via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` → `l1_data_gas.is_zero() && l2_gas.is_zero()` → reconstructed as `ValidResourceBounds::L1Gas(l1_gas)`.
5. Recompute hash from deserialized transaction → `get_tip_resource_bounds_hash` with `L1Gas` → hash preimage omits `L1_DATA_GAS_concat` → produces `H_l1gas ≠ H_allresources`.
6. Block committed with `H_l1gas`; OS `AssertTransactionHash` hint receives `H_allresources` from the original submission → mismatch → wrong execution result or invalid proof. [6](#0-5) [7](#0-6)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
