### Title
Silent Swallowing of Permanent Errors in `_executeSlowModeTransaction` Causes Permanent User Token Lock — (File: `core/contracts/Endpoint.sol`)

### Summary
`_executeSlowModeTransaction` uses a `try/catch` that silently discards all non-OOG errors from `processSlowModeTransaction`. Because `depositCollateralWithReferral` transfers user tokens to the contract **before** queuing the slow mode transaction, and the slow mode transaction is **deleted from the queue before execution**, any permanent failure of a `DepositCollateral` slow mode transaction results in user tokens being permanently locked in the contract with no recovery path.

### Finding Description
The deposit flow in `Endpoint.sol` operates in two phases separated by a mandatory 3-day delay (`SLOW_MODE_TX_DELAY`):

**Phase 1 — `depositCollateralWithReferral` (lines 144–165):**
- `handleDepositTransfer` is called at line 144, pulling ERC-20 tokens from the user into the contract.
- A `SlowModeTx` of type `DepositCollateral` is enqueued at line 152.

**Phase 2 — `_executeSlowModeTransaction` (lines 185–229):**
- At line 194, `delete slowModeTxs[_slowModeConfig.txUpTo++]` permanently removes the transaction from the queue **before** any execution attempt.
- At line 207, `processSlowModeTransaction` is called inside a `try/catch`.
- The `catch` block (lines 208–227) only re-raises out-of-gas conditions. All other errors — including permanent, unrecoverable ones — are silently swallowed.
- The comment at line 226 (`// try return funds now removed`) explicitly confirms that a fund-return mechanism previously existed in this catch block and was removed.

`processSlowModeTransactionImpl` for `DepositCollateral` (EndpointTx.sol lines 209–216) calls `clearinghouse.depositCollateral(txn)`, which can fail permanently if, for example, the product is delisted, the subaccount becomes sanctioned, or clearinghouse-level invariants are violated in the 3-day window between deposit and execution.

When such a failure occurs:
1. The slow mode transaction has already been deleted (line 194) — it cannot be re-executed.
2. The error is silently swallowed (lines 207–227) — no revert, no event, no recovery.
3. `nSubmissions` is incremented (line 234) as if the transaction succeeded.
4. The user's tokens remain in the contract with no mechanism to reclaim them.

The vulnerability class is identical to the reference report: errors are not classified as transient vs. permanent. All failures are treated identically (silently dropped), and the execution path continues regardless of error type, leaving the system in a corrupted state (tokens held, deposit never credited).

### Impact Explanation
A user who calls `depositCollateralWithReferral` and whose `DepositCollateral` slow mode transaction subsequently fails permanently will have their ERC-20 tokens permanently locked in the `Endpoint` contract. The deposit is never credited to their subaccount. There is no on-chain recovery path: the queue entry is gone, the catch block does nothing, and the removed fund-return mechanism confirms no refund is issued. This is a direct, irreversible asset loss for the affected user.

### Likelihood Explanation
The 3-day `SLOW_MODE_TX_DELAY` creates a meaningful window for state changes that can cause permanent failure:
- A product can be delisted between deposit and execution.
- A subaccount address can be added to the sanctions list.
- Clearinghouse configuration changes (e.g., product weight updates) can cause `depositCollateral` to revert.

Any of these conditions, occurring naturally or triggered by a third party (e.g., a sanctions oracle update), will silently destroy the user's deposit. The entry path is fully permissionless: `depositCollateralWithReferral` is callable by any user with no access restrictions.

### Recommendation
Restore a fund-return mechanism in the `catch` block of `_executeSlowModeTransaction` for `DepositCollateral` transactions. When `processSlowModeTransaction` fails for a non-OOG reason and the queued transaction was a deposit (i.e., tokens were already transferred), the catch block should refund the deposited tokens to the original sender. Alternatively, classify errors within `processSlowModeTransactionImpl` and propagate permanent errors upward so they can be handled distinctly from transient ones, mirroring the fix applied in Mantle PR #192.

### Proof of Concept

1. User calls `depositCollateralWithReferral(subaccount, productId, amount)`.
   - `handleDepositTransfer` moves `amount` tokens from user to `Endpoint` contract. [1](#0-0) 
   - A `SlowModeTx` of type `DepositCollateral` is enqueued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY`. [2](#0-1) 

2. During the 3-day delay, the product is delisted or the subaccount is sanctioned.

3. After the delay, anyone calls `executeSlowModeTransaction()`.
   - `_executeSlowModeTransaction` is called; the transaction is deleted from the queue at line 194. [3](#0-2) 
   - `processSlowModeTransaction` is called inside `try/catch`. [4](#0-3) 
   - `processSlowModeTransactionImpl` calls `clearinghouse.depositCollateral(txn)`, which reverts due to the state change. [5](#0-4) 
   - The `catch` block does not re-raise (not OOG), does not refund, and silently returns. [6](#0-5) 

4. Result: user's tokens remain in the `Endpoint` contract. The slow mode queue entry is gone. The deposit is never credited. `nSubmissions` is incremented as if the transaction succeeded. [7](#0-6)
