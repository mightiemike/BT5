### Title
Unprotected `initialize()` on Implementation Contracts Allows Attacker to Seize Ownership and Brick Proxies — (`core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`, `core/contracts/OffchainExchange.sol`)

---

### Summary

`Endpoint`, `Clearinghouse`, and `OffchainExchange` are deployed as upgradeable proxies but their implementation contracts do not call `_disableInitializers()` in a constructor. An attacker can call `initialize()` directly on the bare implementation address, take ownership of the implementation, and — for UUPS-style proxies — upgrade the implementation to a self-destructing contract, permanently bricking the proxy and halting the entire protocol.

---

### Finding Description

OpenZeppelin's upgradeable pattern requires that implementation contracts call `_disableInitializers()` in their constructor so that the implementation itself can never be initialized. Without this guard, the implementation's `initialize()` function remains callable by anyone who discovers the implementation address (which is publicly readable from the proxy's EIP-1967 storage slot).

Contracts confirmed to be registered as proxies via `ProxyManager.registerRegularProxy` include `Clearinghouse` and `Endpoint`. `OffchainExchange` is wired into `Endpoint.initialize` and follows the same upgradeable pattern.

**`Clearinghouse.sol`** — no constructor, no `_disableInitializers()`: [2](#0-1) 

**`OffchainExchange.sol`** — no constructor, no `_disableInitializers()`: [3](#0-2) 

Contrast with contracts that are correctly protected — `Airdrop`, `BaseProxyManager`, `Verifier`, `BaseWithdrawPool` — all call `_disableInitializers()` in their constructors: [4](#0-3) [5](#0-4) 

The three vulnerable contracts are missing this protection entirely.

---

### Impact Explanation

**`Endpoint` implementation takeover** is the highest-severity path:

1. Attacker calls `Endpoint_impl.initialize(attacker, attacker, ...)` directly on the implementation address.
2. `__Ownable_init()` runs, setting `owner = attacker` in the *implementation's* storage.
3. For UUPS proxies, `upgradeTo` is gated by `onlyOwner` and lives in the implementation. Attacker calls `Endpoint_impl.upgradeTo(maliciousImpl)` where `maliciousImpl` contains a `selfdestruct` in its fallback or initializer.
4. Attacker triggers `selfdestruct` in the context of the implementation contract — the implementation is destroyed.
5. Every subsequent call through the proxy `delegatecall`s to a destroyed address, returning empty data and reverting. **The proxy is permanently bricked.**

Even without `selfdestruct`, the attacker-owned implementation can have its owner-gated functions called (e.g., `setSequencer`, `setVerifier` equivalents) in the implementation's own storage context, creating confusion and potential for further exploit chains.

`Clearinghouse` and `OffchainExchange` follow the same path: attacker takes ownership of the implementation, then either destroys it or manipulates its state to corrupt future upgrades.

---

### Likelihood Explanation

- The implementation address is publicly readable from the proxy's EIP-1967 slot (`0x360894...`).
- No privileged access is required — any EOA can call `initialize()` on the implementation.
- The attack window is the gap between implementation deployment and the proxy's `initialize()` call (which initializes the *proxy's* storage, not the implementation's). This window exists in every deployment.
- The `ProxyManager` deployment flow confirms implementations are registered after deployment, creating a non-atomic window. [6](#0-5) 

---

### Recommendation

Add a constructor with `_disableInitializers()` to each affected implementation contract:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

This must be added to `Endpoint.sol`, `Clearinghouse.sol`, and `OffchainExchange.sol`. This is the same pattern already correctly applied in `Airdrop`, `BaseProxyManager`, `Verifier`, and `BaseWithdrawPool`.

---

### Proof of Concept

```solidity
// Attacker script (pseudocode)
address impl = IProxy(endpointProxy).implementation(); // read EIP-1967 slot

// Front-run or call after deployment, before proxy initialize
Endpoint(impl).initialize(
    attacker,   // _sanctions (attacker-controlled)
    attacker,   // _sequencer
    attacker,   // _offchainExchange
    IClearinghouse(attacker), // _clearinghouse
    attacker,   // _verifier
    attacker    // _endpointTx
);

// Attacker now owns the implementation
// For UUPS: upgrade implementation to selfdestruct contract
Endpoint(impl).upgradeTo(address(new SelfDestructImpl()));
// Trigger selfdestruct on implementation
SelfDestructImpl(impl).destroy();

// All proxy calls now delegatecall to dead address → protocol halted
```
