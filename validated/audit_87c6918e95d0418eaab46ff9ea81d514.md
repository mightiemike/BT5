### Title
Strict Bonus Pool Check Blocks All Position Exit Paths When Pool Is Depleted — (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

The `processEventsAndClaimBonus` function enforces an all-or-nothing bonus pool check. When the rewards pool balance is less than the accrued bonus owed to a position, every user-driven exit operation (`TierUndelegate`, `TierRedelegate`, `ExitTierWithDelegation`, `ClearPosition`) reverts atomically. This permanently locks user funds in their positions until governance replenishes the pool, and prevents redelegation away from at-risk validators.

---

### Finding Description

In `processEventsAndClaimBonus`, after computing the total bonus owed, the code calls `sufficientBonusPoolBalance`: [1](#0-0) 

`sufficientBonusPoolBalance` performs a strict all-or-nothing check: [2](#0-1) 

If the pool balance is less than the full bonus owed, `ErrInsufficientBonusPool` is returned and the entire transaction reverts. This check is reached from `claimRewards`, which is called unconditionally at the start of every exit path in `msg_server.go`:

- `TierUndelegate` calls `claimRewards` before undelegating: [3](#0-2) 

- `TierRedelegate` calls `claimRewards` before redelegating: [4](#0-3) 

- `ExitTierWithDelegation` calls `claimRewards` before transferring delegation: [5](#0-4) 

- `ClearPosition` calls `claimRewards` before clearing the exit: [6](#0-5) 

The only path that bypasses `claimRewards` is `WithdrawFromTier`, but it requires the position to already be undelegated — which itself requires `TierUndelegate` to succeed first. There is no exit path for a delegated position that avoids the bonus pool check.

The contrast with the slash hook is instructive: `slashRedelegationPosition` explicitly forfeits the bonus silently when the pool is short, to avoid a chain halt: [7](#0-6) 

No equivalent leniency exists for user-driven exit paths.

---

### Impact Explanation

When the bonus pool is depleted:

1. **Funds locked**: Users with delegated positions cannot undelegate, redelegate, or exit with delegation. Their staked tokens are inaccessible until governance replenishes the pool.
2. **Slashing exposure**: If a validator is about to be jailed or slashed, users with positions on that validator cannot redelegate to a safer validator. The `TierRedelegate` call reverts with `ErrInsufficientBonusPool`, and the user's position is slashed — a direct, avoidable financial loss.
3. **No partial exit**: Unlike the slash hook which pays what it can, user paths have no fallback. Even if the pool has 99% of the owed bonus, the transaction reverts entirely.

The broken invariant is: *a position owner must always be able to exit or redelegate their delegated position*. The strict pool check violates this invariant whenever the pool balance falls below the accrued bonus.

---

### Likelihood Explanation

- The bonus pool is funded solely by governance. It drains continuously as users claim rewards.
- With high `BonusApy` tiers and large or long-lived positions, the pool can be exhausted before governance acts.
- The pool depletion is a natural, unprivileged outcome of normal protocol usage — no attacker action is required to trigger it.
- Governance response latency (proposal → voting → execution) can span days, during which all exit paths remain blocked.

---

### Recommendation

For exit-critical operations (`TierUndelegate`, `TierRedelegate`, `ExitTierWithDelegation`), replace the strict all-or-nothing check with a "pay what is available" approach — cap the bonus payout at `min(accrued_bonus, pool_balance)` and proceed with the exit regardless. This mirrors the already-implemented behavior in `slashRedelegationPosition`: [7](#0-6) 

Concretely, in `processEventsAndClaimBonus`, instead of returning an error when the pool is insufficient, clamp `bonusCoins` to the available pool balance and emit the partial payout. The position's `LastEventSeq` and `LastBonusAccrual` checkpoints are already advanced before the pool check, so the forfeited portion is not double-counted on a future claim.

---

### Proof of Concept

1. Alice creates a tier position with a high `BonusApy` tier by calling `MsgLockTier`.
2. Significant time passes; Alice accrues a large bonus (e.g., 10,000 CRO).
3. The bonus pool is drained to zero — either by other users legitimately claiming rewards, or simply because governance has not replenished it.
4. Alice's validator signals it will be jailed (e.g., downtime detected on-chain).
5. Alice submits `MsgTierRedelegate` to move her stake to a healthy validator.
6. Execution path: `TierRedelegate` → `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → pool balance `0 < 10,000 CRO` → returns `ErrInsufficientBonusPool`.
7. The transaction reverts. Alice cannot redelegate.
8. The validator is jailed; Alice's position is slashed. She suffers a direct token loss that was entirely avoidable.

The root cause is structurally identical to the external report: a strict bonus/reward calculation that requires the full bonus to be available causes a critical operation (exit/redelegate vs. liquidation) to revert when the bonus cannot be fully paid, leaving the user permanently unable to act until an external party (governance vs. collateral top-up) intervenes.

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```
