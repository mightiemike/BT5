### Title
Sequencer-Supplied `oraclePriceX18` Is Not Committed to in User Signature for MintNlp/BurnNlp, Enabling Execution at Stale or Unfavorable NLP Prices — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` transaction flows allow the sequencer to supply an arbitrary `oraclePriceX18` that is entirely absent from the EIP-712 digest the user signs. Neither struct carries an expiration field nor a user-specified price bound. If a significant time gap exists between when the user signs the request and when the sequencer executes it, the NLP mint or burn is settled at a price that may have moved materially against the user, causing a concrete asset loss with no on-chain recourse.

---

### Finding Description

When a user initiates a `MintNlp` or `BurnNlp` operation, they sign a struct that contains only `sender`, `quoteAmount`/`nlpAmount`, and `nonce`:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint64 nonce;
}
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64 nonce;
}
``` [1](#0-0) 

The outer `SignedMintNlp` / `SignedBurnNlp` wrappers carry `oraclePriceX18` and `nlpPoolRebalanceX18` as additional fields that are **appended by the sequencer** and are **not part of the signed payload**:

```solidity
struct SignedMintNlp {
    MintNlp tx;
    bytes signature;
    int128 oraclePriceX18;        // sequencer-supplied, not signed
    int128[] nlpPoolRebalanceX18; // sequencer-supplied, not signed
}
``` [2](#0-1) 

The `Verifier.computeDigest` for `MintNlp` confirms that `oraclePriceX18` is excluded from the digest:

```solidity
digest = keccak256(abi.encode(
    keccak256(bytes(MINT_NLP_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.quoteAmount,
    signedTx.tx.nonce          // oraclePriceX18 is absent
));
``` [3](#0-2) 

In `EndpointTx.processTransactionImpl`, after validating the user's signature (which covers only the three fields above), the sequencer-provided price is written directly to protocol state and forwarded to `Clearinghouse.mintNlp` / `burnNlp` without any staleness or bounds check:

```solidity
priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;
clearinghouse.mintNlp(
    signedTx.tx,
    signedTx.oraclePriceX18,   // unchecked sequencer value
    nlpPools,
    signedTx.nlpPoolRebalanceX18
);
``` [4](#0-3) 

`Clearinghouse.mintNlp` uses this price directly to compute the NLP amount the user receives:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [5](#0-4) 

And `burnNlp` uses it to compute the USDC the user receives:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
``` [6](#0-5) 

Unlike `Order`, which carries an `expiration` field the user signs, `MintNlp` and `BurnNlp` carry no expiration and no user-specified price bound. There is no on-chain mechanism to reject execution if the price has drifted beyond what the user implicitly expected at signing time. [7](#0-6) 

---

### Impact Explanation

**MintNlp:** The user commits to paying `quoteAmount` USDC. The NLP tokens received equal `quoteAmount / oraclePriceX18`. If the NLP price has risen since signing, the user receives materially fewer NLP tokens for the same USDC outlay — a direct, quantifiable asset loss.

**BurnNlp:** The user commits to burning `nlpAmount` NLP. The USDC received equals `nlpAmount × oraclePriceX18`. If the NLP price has fallen since signing, the user receives materially less USDC — again a direct asset loss.

In both cases the corrupted state delta is the user's quote or NLP balance in `SpotEngine`, and the loss flows entirely to the NLP pool (N_ACCOUNT), which benefits from the mispriced conversion.

---

### Likelihood Explanation

The sequencer processes a high-throughput queue. Any period of sequencer latency, congestion, or maintenance creates a window where a signed `MintNlp`/`BurnNlp` request sits pending while the NLP oracle price moves. Because the user has no expiration or slippage guard in their signed message, every such request is silently exposed to this drift for its entire pending lifetime. No adversarial sequencer behavior is required; the gap between signing time and execution time is sufficient.

---

### Recommendation

1. Add a `maxOraclePriceX18` (for mint) and `minOraclePriceX18` (for burn) field to `MintNlp` and `BurnNlp` structs, include them in the EIP-712 digest, and enforce them on-chain before computing the conversion.
2. Add an `expiration` field (analogous to `Order.expiration`) to both structs so users can bound the window during which their request remains valid.
3. In `EndpointTx.processTransactionImpl`, validate that `signedTx.oraclePriceX18` does not exceed the user-specified bound before calling `clearinghouse.mintNlp` / `burnNlp`.

---

### Proof of Concept

1. User signs `MintNlp{sender=A, quoteAmount=10_000e18, nonce=N}` at time T₁ when NLP price is 1.00 USDC. User expects ≈10,000 NLP tokens.
2. Sequencer queue is congested; the transaction is not submitted on-chain until T₂.
3. Between T₁ and T₂ the NLP price rises to 2.00 USDC (e.g., due to market movement reflected in the next `UpdatePrice` transaction).
4. Sequencer submits `SignedMintNlp` with `oraclePriceX18 = 2e18`. The user's signature is valid (it covers only `sender`, `quoteAmount`, `nonce`).
5. `Clearinghouse.mintNlp` computes `nlpAmount = 10_000e18 / 2e18 = 5_000`. The user receives 5,000 NLP tokens instead of 10,000 — a 50% loss — with no on-chain check that would reject or revert this execution. [8](#0-7) [9](#0-8)

    function mintNlp(
        IEndpoint.MintNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.quoteAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 quoteAmount = int128(txn.quoteAmount);
        int128 nlpAmount = quoteAmount.div(oraclePriceX18);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] >= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, -nlpAmount);

        spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -quoteAmount);
        _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);

        require(
            getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
