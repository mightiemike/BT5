### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables USDC-E Drain Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` at line 616 without checking the return value. The `ERC20Helper` library providing `safeTransferFrom` is already imported and used throughout the same contract. If `transferFrom` returns `false` instead of reverting, the function continues executing, withdrawing USDC-E from the `directDepositV1` address and sending it to the caller — without the caller ever providing USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is a permissionless function (only gated by `block.chainid == 57073`) that is designed to let any caller swap USDC-E held in a `directDepositV1` address for an equivalent amount of USDC. The three-step sequence is:

1. Pull USDC from `msg.sender` into `directDepositV1` via `transferFrom`
2. Withdraw USDC-E from `directDepositV1` to `ContractOwner` via `DirectDepositV1.withdraw`
3. Forward USDC-E from `ContractOwner` to `msg.sender` via `safeTransfer`

Step 1 uses a raw, unchecked `transferFrom`:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // line 616 — return value ignored
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));       // line 617
IERC20Base(usdcE).safeTransfer(msg.sender, balance);                 // line 618 — safe wrapper used
``` [1](#0-0) 

The `ERC20Helper` library is already applied to `IERC20Base` via `using ERC20Helper for IERC20Base` and provides `safeTransferFrom`, which checks both the call success flag and the decoded boolean return value:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [2](#0-1) 

Line 618 in the same block already uses `safeTransfer`, making the inconsistency on line 616 a clear oversight. [3](#0-2) 

---

### Impact Explanation

If `IERC20Base(usdc).transferFrom` returns `false` without reverting (e.g., the caller has not approved `ContractOwner`, or the USDC token on chain 57073 is a non-standard implementation that returns `false` on failure), execution continues. `DirectDepositV1.withdraw` transfers all USDC-E from `directDepositV1` to `ContractOwner`, and `safeTransfer` then forwards it to the caller. The caller receives the full USDC-E balance of the targeted `directDepositV1` address without providing any USDC. This is a direct, unrecoverable asset loss for the subaccount owner whose USDC-E was held in that `directDepositV1`.

---

### Likelihood Explanation

The function is callable by any unprivileged external address on chain 57073 (Ink). No role, signature, or ownership check protects it beyond the chain ID guard. [4](#0-3)  The USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink is a relatively new deployment; if it is a non-Circle-canonical USDC or a bridged variant that returns `false` on insufficient allowance rather than reverting, the exploit is immediately triggerable. Even if the current USDC deployment reverts, the code pattern is incorrect and creates fragility against any future token migration or upgrade on that chain.

---

### Recommendation

Replace the raw `transferFrom` on line 616 with the `safeTransferFrom` wrapper already available via `ERC20Helper`:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [5](#0-4) 

This is consistent with how every other token transfer in `ContractOwner` and `EndpointStorage` is handled. [6](#0-5) 

---

### Proof of Concept

1. A `directDepositV1` address for some `subaccount` holds `N` USDC-E tokens.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` on chain 57073 **without** having approved any USDC to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, N)` is called. If the USDC token returns `false` instead of reverting (e.g., due to zero allowance on a non-reverting implementation), the call silently fails and no USDC moves.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes successfully — `DirectDepositV1` is owned by `ContractOwner`, so the call is authorized. All `N` USDC-E tokens are transferred to `ContractOwner`. [7](#0-6) 
5. `IERC20Base(usdcE).safeTransfer(attacker, N)` executes successfully — attacker receives `N` USDC-E.
6. Net result: attacker gains `N` USDC-E, subaccount loses `N` USDC-E, no USDC was provided. [1](#0-0)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
