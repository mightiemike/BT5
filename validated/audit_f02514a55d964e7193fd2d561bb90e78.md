### Title
Dead-Code Mempool Size Pre-Check Uses Wrong Threshold, Leaving Intended Guard Unenforced — (`src/more_ops.rs`)

### Summary

In `op_div`, `op_divmod`, `op_mod`, and their malachite variants, a `DISABLE_OP`-gated pre-check intended to enforce a stricter mempool-mode size limit on the dividend uses a threshold of `2048` bytes. However, an unconditional check immediately below it rejects any dividend exceeding `256` bytes. Because `2048 > 256`, the pre-check is permanently dead code: the unconditional check always fires first. The intended mempool-mode guard is never reached and never enforced.

### Finding Description

In `src/more_ops.rs`, all six arithmetic division/modulo functions share the same two-check pattern:

```rust
// op_div (line 665-669)
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
if a0_len > 256 || a1_len > 1024 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
```

The same pattern appears in `op_div_malachite` (lines 690–694), `op_divmod` (lines 713–717), `op_divmod_malachite` (lines 742–746), `op_mod` (lines 769–773), and `op_mod_malachite` (lines 794–798).

**Error 1 — Wrong threshold:** The pre-check fires only when `a0_len > 2048`. The unconditional check below fires when `a0_len > 256`. Since `2048 > 256`, any input that would satisfy the pre-check (`a0_len > 2048`) already satisfies the unconditional check (`a0_len > 256`). The pre-check is unreachable.

**Error 2 — Wrong flag semantics:** `DISABLE_OP` (`0x200`) is defined and documented in `src/chia_dialect.rs` (line 56) as the flag that disables op 60 (`op_modpow`) in mempool mode (lines 240–243). Reusing it as a size-limit gate in `op_div`/`op_divmod`/`op_mod` conflates two unrelated enforcement concerns under one flag name, making the intent of the pre-check opaque and the flag's semantics inconsistent.

**Error 3 — The pre-check is entirely redundant:** Even if the threshold were corrected, the unconditional check at `256` already handles all oversized inputs in both consensus and mempool modes. The pre-check adds no protection in any code path.

`DISABLE_OP` is included in `MEMPOOL_MODE` (lines 72–76 of `src/chia_dialect.rs`):

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)   // ← activates the dead pre-check
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

### Impact Explanation

The mempool-mode size guard for `op_div`, `op_divmod`, and `op_mod` is silently absent. If the intent was to enforce a stricter dividend-size limit in mempool mode (e.g., to bound worst-case computation before a transaction is admitted), that limit is never applied. An attacker can craft CLVM programs that pass mempool validation with dividend atoms up to 256 bytes — the same ceiling as consensus mode — regardless of what the `DISABLE_OP`-gated check was supposed to restrict. The cost model bounds total execution cost, but a missing per-operand size gate in mempool mode can allow inputs that are disproportionately expensive relative to their declared cost, undermining the mempool's DoS-resistance posture.

### Likelihood Explanation

The entry path is direct: any caller of `run_program` with `MEMPOOL_MODE` flags and a CLVM program invoking op 19 (`/`), op 20 (`divmod`), or op 61 (`mod`) with a large dividend atom reaches the dead pre-check. The attacker controls the CLVM bytes entirely. No special privileges are required.

### Recommendation

Remove the dead pre-check entirely and rely on the unconditional check, which already handles all cases correctly — mirroring the fix applied in the referenced external report. If a stricter mempool-mode limit is genuinely required, introduce a dedicated flag (not `DISABLE_OP`) and set its threshold below `256`.

### Proof of Concept

For any `a0_len` in the range `(256, 2048]`:
- Pre-check: `DISABLE_OP && a0_len > 2048` → **false** (not triggered)
- Unconditional check: `a0_len > 256` → **true** (error returned)

For any `a0_len > 2048`:
- Pre-check: `DISABLE_OP && a0_len > 2048` → **true** (would return error)
- But the unconditional check at `a0_len > 256` fires **first** in the same function, returning the same error before the pre-check's branch is ever evaluated

The pre-check therefore has zero reachable effect in all cases. A CLVM program `(/ (q . <257-byte atom>) (q . 1))` submitted under `MEMPOOL_MODE` reaches the unconditional check, not the `DISABLE_OP` guard, demonstrating the guard is dead.
