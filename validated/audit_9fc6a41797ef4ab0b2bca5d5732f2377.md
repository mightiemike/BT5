### Title
`RelayerManager` Role Cannot Update Relayer Configuration Despite Documented Purpose — (`contract/src/lib.rs`)

---

### Summary

The `Role::RelayerManager` is documented as the role that "may manage trusted relayer staking: reject applications and **update relayer config**." However, the `#[trusted_relayer]` macro is configured with `config_roles(Role::DAO)` only, meaning `RelayerManager` accounts are silently blocked from calling the config-update entrypoint. Only `Role::DAO` — the super-admin — can update the relayer configuration. This is a direct structural analog to the reported `RAACToken` bug: a purpose-built operational role is rendered incomplete by a more-restrictive access-control guard on the functions it is supposed to call.

---

### Finding Description

In `contract/src/lib.rs`, the `#[trusted_relayer]` macro is applied to the `BtcLightClient` impl block with three role categories:

```rust
#[trusted_relayer(
    bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks),
    manager_roles(Role::DAO, Role::RelayerManager),
    config_roles(Role::DAO)                          // ← RelayerManager absent
)]
``` [1](#0-0) 

The `Role` enum documents `RelayerManager` as:

```
/// May manage trusted relayer staking: reject applications and update relayer config.
RelayerManager,
``` [2](#0-1) 

The `omni_utils::macros::trusted_relayer` macro generates two distinct privileged surfaces:

- **`manager_roles`** — gates functions that add/remove relayers from the trusted set and reject staking applications. `RelayerManager` is included here.
- **`config_roles`** — gates functions that update the relayer configuration (e.g., minimum stake, staking parameters). Only `Role::DAO` is listed here.

Because `config_roles` does not include `Role::RelayerManager`, any account granted `RelayerManager` can manage the relayer list but **cannot call the config-update function**. The only account that can update the relayer config is one holding `Role::DAO`.

---

### Impact Explanation

The `RelayerManager` role is operationally broken for its stated purpose. An operator who is granted `RelayerManager` to handle day-to-day relayer administration — including updating staking requirements — will find that every call to the config-update method panics with an authorization error. The only path to update relayer config is through `Role::DAO`, which is the contract's super-admin and is typically held behind a multisig or DAO governance process with a slow approval cycle. This means:

1. Relayer configuration (staking thresholds, etc.) cannot be updated through the dedicated operational role.
2. Every config change requires DAO-level governance, defeating the purpose of the `RelayerManager` role.
3. If the DAO key is unavailable (e.g., multisig quorum not reachable), relayer config is permanently frozen.

---

### Likelihood Explanation

Any deployment that grants `RelayerManager` to a dedicated operational account — which is the intended use of the role — will immediately encounter this failure the first time that account attempts a config update. The misconfiguration is in the contract source itself and is present in every deployment.

---

### Recommendation

Add `Role::RelayerManager` to `config_roles` in the `#[trusted_relayer]` macro so that the role can fulfill its documented purpose:

```diff
 #[trusted_relayer(
     bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks),
     manager_roles(Role::DAO, Role::RelayerManager),
-    config_roles(Role::DAO)
+    config_roles(Role::DAO, Role::RelayerManager)
 )]
``` [1](#0-0) 

---

### Proof of Concept

1. Deploy the contract and call `init`.
2. Grant `Role::RelayerManager` to account `alice.near` via `acl_grant_role`.
3. As `alice.near`, call the config-update method generated by the `trusted_relayer` macro (e.g., `update_relayer_config` or equivalent).
4. Observe the call panics with an authorization error, because `alice.near` holds `RelayerManager` but `config_roles` only permits `Role::DAO`.
5. Repeat step 3 as an account holding `Role::DAO` — it succeeds.

This mirrors the `RAACMinter` → `RAACToken` call failure in the reference report: the operational role (`RelayerManager` / `minter`) is blocked by a guard that only admits the owner-equivalent role (`DAO` / `onlyOwner`).
