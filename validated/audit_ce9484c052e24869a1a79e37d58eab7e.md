### Title
Missing `BlockIdFlag` Filter in `topUpBaseRewards` Allows Absent Validators to Receive Base Reward Top-Up — (`x/tieredrewards/keeper/abci.go`)

### Summary

`topUpBaseRewards` distributes base reward top-up to **all** validators present in `sdkCtx.VoteInfos()` without filtering by `BlockIdFlag`. In Cosmos SDK / CometBFT, `VoteInfos()` includes every validator in the active set — including those that were absent (did not sign the previous block, `BlockIdFlag == BLOCK_ID_FLAG_ABSENT`). As a result, absent validators receive a share of the `RewardsPoolName` top-up they did not earn, and participating validators receive a proportionally smaller share than they are entitled to.

### Finding Description

In `topUpBaseRewards`: [1](#0-0) 

`bondedVotes` is populated from `sdkCtx.VoteInfos()` with no predicate on `voteInfo.BlockIdFlag`. The subsequent power accumulation loop: [2](#0-1) 

sums the power of **every** entry, including absent validators. The distribution loop: [3](#0-2) 

then calls `AllocateTokensToValidator` for every entry in `bondedVotes`, again with no `BlockIdFlag` guard. There is no filtering anywhere in the function.

The test helper `ctxWithVoteInfos` confirms the pattern — it constructs `VoteInfo` entries with only `Address` and `Power` set, leaving `BlockIdFlag` at its zero value (`BLOCK_ID_FLAG_UNKNOWN`/`ABSENT`), and the existing tests pass without any absent-validator filtering: [4](#0-3) 

### Impact Explanation

- A validator V that was absent in block N-1 (present in `VoteInfos` with `Power > 0` but `BlockIdFlag == BLOCK_ID_FLAG_ABSENT`) receives `(V.Power / totalPower) × topUpAmount` from `RewardsPoolName` via `AllocateTokensToValidator`.
- Participating validators collectively receive `(participatingPower / totalPower) × topUpAmount` instead of the full `topUpAmount`.
- The shortfall is a direct, per-block transfer of bonded-denom tokens from the rewards pool to validators that did not earn them, at the expense of validators that did participate.
- The effect compounds every block a validator is absent, draining `RewardsPoolName` faster than intended and permanently under-rewarding active validators.

### Likelihood Explanation

Validator absence is a normal, recurring network event (network partitions, upgrades, missed blocks). No special privileges or governance action are required. The path is triggered automatically every `BeginBlock` whenever a shortfall exists and at least one validator is absent.

### Recommendation

Filter `bondedVotes` to include only validators that actually signed the previous block before computing `previousTotalPower` and distributing rewards:

```go
// In topUpBaseRewards, after line 84:
var bondedVotes []abci.VoteInfo
for _, vi := range sdkCtx.VoteInfos() {
    if vi.BlockIdFlag == cmtproto.BlockIDFlagCommit {
        bondedVotes = append(bondedVotes, vi)
    }
}
```

This mirrors the pattern used by the Cosmos SDK distribution module's own `AllocateTokens` logic, which only rewards validators whose `BlockIdFlag` indicates they signed the previous block.

### Proof of Concept

Extend `abci_test.go` with a keeper test:

1. Create two validators, V1 (participating) and V2 (absent).
2. Set `ctx.WithVoteInfos` with both validators, but set V2's `BlockIdFlag = cmtproto.BlockIDFlagAbsent`.
3. Drain the fee collector to force a shortfall.
4. Fund `RewardsPoolName`.
5. Call `BeginBlocker`.
6. Assert `GetValidatorOutstandingRewardsCoins` for V2 is zero (currently it will be non-zero, proving the bug). [5](#0-4)
