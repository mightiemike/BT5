Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` validates the router address instead of the true end-user, allowing any non-allowlisted user to bypass the swap allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is whatever `MetricOmmPool.swap()` passes as `msg.sender` — the immediate caller of the pool. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the allowlist check validates the router's address rather than the actual end-user. Any non-allowlisted user can bypass the per-user swap allowlist on a curated pool by routing through the public `MetricOmmSimpleRouter`, exposing LP funds to counterparties the pool admin explicitly excluded.

## Finding Description

**Step 1 — Pool passes its own `msg.sender` as `sender`:**
`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` at line 231. When the caller is `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the router's address.

**Step 2 — Extension receives the router address as `sender`:**
`ExtensionCalling._beforeSwap` (lines 149–177) encodes and forwards the `sender` argument unchanged to every configured extension via `_callExtensionsInOrder`.

**Step 3 — Allowlist checks the router, not the end-user:**
`SwapAllowlistExtension.beforeSwap` (line 37) evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender])
```
Here `msg.sender` is the pool (the extension's caller) and `sender` is the router address. The check never sees the original EOA.

**Step 4 — Router calls pool directly without forwarding the originator:**
`MetricOmmSimpleRouter.exactInputSingle` (line 72) and `exactInput` (line 104) call `IMetricOmmPoolActions(pool).swap(...)` directly. There is no mechanism to pass the original `msg.sender` (the end-user) through to the extension layer.

**Irreconcilable dilemma for pool admins:**
- If the router is **not** allowlisted: all router-mediated swaps revert with `NotAllowedToSwap`, even for users the admin intended to allow.
- If the router **is** allowlisted (the only way to make router swaps work): every user who routes through the router passes the check, regardless of whether they are on the allowlist. The allowlist is completely bypassed.

No existing guard compensates for this: the pool has no concept of a "trusted forwarder" and the extension has no access to the original transaction initiator.

## Impact Explanation
A pool admin who deploys a curated pool with `SwapAllowlistExtension` to restrict trading to a specific set of addresses (e.g., KYC-verified users, whitelisted counterparties) loses that protection entirely for any user who routes through `MetricOmmSimpleRouter`. The non-allowlisted user executes a real swap against the pool's liquidity, receiving output tokens and paying input tokens — a direct, fund-impacting bypass of the intended access control. LP funds are exposed to counterparties the pool admin explicitly excluded. This constitutes a broken core pool functionality causing loss of funds (LP assets traded against unauthorized counterparties) and an admin-boundary break where the pool admin's access control configuration is rendered ineffective by an unprivileged path.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary public swap entrypoint documented and supported by the protocol. Any user aware of the allowlist restriction has a trivial, one-step bypass: call `exactInputSingle` or `exactInput` on the router instead of calling `pool.swap()` directly. No special privileges, flash loans, or multi-transaction setups are required. The router is a deployed, permissionless contract callable by any EOA.

## Recommendation
The `sender` forwarded to extensions must represent the true end-user, not the immediate pool caller. Two complementary approaches:

1. **Pool-level fix**: Have `MetricOmmPool.swap()` accept an explicit `originator` parameter (the true initiator) and pass it as `sender` to `_beforeSwap`, similar to how `addLiquidity` separates `msg.sender` (payer) from `owner` (position holder). The router would then pass `msg.sender` as the originator.

2. **Extension-level mitigation**: Until the pool-level fix is in place, document clearly that allowlisting the router address grants unrestricted access to all router users, and consider exposing a "trusted forwarder" concept where the router encodes the true originator in `extensionData` and the extension reads it from there (with appropriate trust checks on the forwarder).

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // only alice is allowed
  - Pool admin calls setAllowedToSwap(pool, router, true)  // required for router to work at all

Attack:
  - Bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  - Router calls pool.swap(recipient=bob, ...) → pool's msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true ✓
  - Bob's swap executes successfully despite not being on the allowlist

Result:
  - Bob receives output tokens from the curated pool
  - The allowlist that was supposed to restrict trading to alice is fully bypassed
  - If the admin does NOT allowlist the router, alice also cannot use the router
```
