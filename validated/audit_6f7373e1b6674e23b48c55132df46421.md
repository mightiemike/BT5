### Title
Permanent NFT Metadata and Burn Lock After Transfer — (`x/nft/keeper/keeper.go`)

### Summary

`Keeper.EditNFT` and `Keeper.BurnNFT` both require the caller to simultaneously satisfy **two independent checks**: `IsOwner` (current NFT owner) AND `IsDenomCreator` (original denom creator). After any `TransferOwner`, these two roles are held by different addresses, making it impossible for either party to ever edit or burn the NFT.

### Finding Description

`EditNFT` enforces a conjunctive guard: [1](#0-0) 

`IsOwner` checks that `owner` matches the NFT's stored owner field: [2](#0-1) 

`IsDenomCreator` checks that `address` matches `denom.Creator`, which is set once at `IssueDenom` and never updated: [3](#0-2) 

`TransferOwner` only requires `IsOwner` — it has no `IsDenomCreator` guard and freely reassigns the NFT's owner: [4](#0-3) 

After a transfer, `denom.Creator` remains A while `nft.Owner` becomes B. No single address can satisfy both checks simultaneously, so `EditNFT` is permanently unreachable for that token. The same dual-requirement lock exists in `BurnNFT`: [5](#0-4) 

The `msgServer.EditNFT` entry point passes `msg.Sender` directly as `owner` with no alternative path: [6](#0-5) 

### Impact Explanation

After one `MsgTransferNFT`:
- **B (new owner)** calls `MsgEditNFT` → `IsOwner(B)` passes, `IsDenomCreator(B)` fails → `ErrUnauthorized`
- **A (denom creator)** calls `MsgEditNFT` → `IsOwner(A)` fails → `ErrUnauthorized`
- **B** calls `MsgBurnNFT` → same dual-lock → `ErrUnauthorized`
- **A** calls `MsgBurnNFT` → `IsOwner(A)` fails → `ErrUnauthorized`

The NFT's metadata is permanently frozen and the token cannot be burned by any party. The only remaining operation is further transfers, which are useless if the NFT's data is stale or incorrect.

### Likelihood Explanation

This triggers on every ordinary secondary-market transfer. No special privileges, governance, or attacker cooperation is required — a single `MsgTransferNFT` from the denom creator to any other address is sufficient to trigger the lock permanently.

### Recommendation

Separate the authorization model for `EditNFT` and `BurnNFT`. The most natural fix is to require **either** `IsOwner` **or** `IsDenomCreator`, not both. Alternatively, restrict `EditNFT` to the current owner only (removing the `IsDenomCreator` check), which matches the semantics of NFT ownership in every comparable standard.

### Proof of Concept

```
1. A calls MsgIssueDenom(id="denom1", sender=A)
   → denom.Creator = A (permanent)

2. A calls MsgMintNFT(denomId="denom1", id="token1", sender=A, recipient=A)
   → nft.Owner = A

3. A calls MsgTransferNFT(denomId="denom1", id="token1", sender=A, recipient=B)
   → nft.Owner = B, denom.Creator still = A

4. B calls MsgEditNFT(denomId="denom1", id="token1", sender=B)
   → IsOwner(B) = OK
   → IsDenomCreator(B) = ErrUnauthorized  ← blocked

5. A calls MsgEditNFT(denomId="denom1", id="token1", sender=A)
   → IsOwner(A) = ErrUnauthorized  ← blocked

Result: no address can ever call EditNFT or BurnNFT on token1 again.
```

```go
func (k Keeper) TransferOwner(
	ctx sdk.Context, denomID, tokenID string, srcOwner, dstOwner sdk.AccAddress,
) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, srcOwner)
	if err != nil {
		return err
	}

	nft.Owner = dstOwner.String()

	k.setNFT(ctx, denomID, nft)
	k.swapOwner(ctx, denomID, tokenID, srcOwner, dstOwner)
	return nil
}
```

```go
func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	_, err = k.IsDenomCreator(ctx, denomID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
}
```
