### Title
`op_modpow` Quadratic Exponent Cost Is Undercharged by Splitting Into Multiple Calls - (`File: src/more_ops.rs`)

### Summary

The `op_modpow` operator in `src/more_ops.rs` charges a cost that is **quadratic in the exponent byte-size** (`esize² * MODPOW_COST_PER_BYTE_EXPONENT`). Because modular exponentiation satisfies `(a^e1)^e2 ≡ a^(e1·e2) (mod m)`, an attacker-controlled CLVM program can split one large-exponent `modpow` call into N smaller-exponent calls, paying `N·(E/N)²·3 = E²/N·3` in exponent cost instead of `E²·3`, while performing the same actual computation. This is the direct analog of M-08: a non-linear cost formula applied to parts yields a lower cumulative charge than applying it once to the whole.

---

### Finding Description

`op_modpow` in `src/more_ops.rs` computes its cost as:

```rust
// src/more_ops.rs lines 1256-1263
let mut cost = MODPOW_BASE_COST;                                          // 17 000
cost += bsize as Cost * MODPOW_COST_PER_BYTE_BASE_VALUE;                  // linear in base
cost += (esize * esize) as Cost * MODPOW_COST_PER_BYTE_EXPONENT;         // QUADRATIC in exponent
check_cost(cost, max_cost)?;
cost += (msize * msize) as Cost * MODPOW_COST_PER_BYTE_MOD;              // quadratic in modulus
``` [1](#0-0) 

The constants are:

```
MODPOW_BASE_COST                = 17 000
MODPOW_COST_PER_BYTE_EXPONENT   =      3   (applied to esize²)
MODPOW_COST_PER_BYTE_MOD        =     21   (applied to msize²)
``` [2](#0-1) 

The size limit for each argument is 256 bytes:

```rust
if bsize > 256 || esize > 256 || msize > 256 {
    return Err(EvalErr::InvalidOpArg(input, "modpow".to_string()));
}
``` [3](#0-2) 

**The exploit**: Because `(a^{e_1})^{e_2} \equiv a^{e_1 \cdot e_2} \pmod{m}`, an attacker who wants to compute `modpow(a, e, m)` with a 256-byte exponent `e = e1 * e2` can instead write the CLVM program:

```
(modpow (modpow a e1 m) e2 m)
```

where `e1` and `e2` are each 128-byte numbers. The mathematical result is identical, but the cost charged differs:

| Strategy | Exponent cost formula | Value (E=256, 3 per byte²) |
|---|---|---|
| Single call `modpow(a, e, m)` | `256² × 3` | **196 608** |
| Two calls with `e1`, `e2` (128 bytes each) | `2 × 128² × 3` | **98 304** |
| **Savings** | | **98 304 (50%)** |

After subtracting the extra `MODPOW_BASE_COST` (17 000) and extra modulus cost for the second call, the net savings for a 32-byte (256-bit) modulus are approximately **59 762 cost units** — a ~25% reduction for the same computation.

The optimal number of splits N minimises `N·MODPOW_BASE_COST + E²/N·MODPOW_COST_PER_BYTE_EXPONENT`, giving `N_opt = E·sqrt(3/(17000)) ≈ 3` for E=256, yielding ~45% savings on the exponent component.

---

### Impact Explanation

The Chia blockchain enforces a per-block cost limit (11 billion cost units). An attacker who can submit coin-spend CLVM programs can craft programs that perform `modpow` with a 256-byte exponent split across 2–4 nested calls, performing the same actual computation (same number of modular squarings) while consuming 25–50% fewer cost units. This allows the attacker to pack more real computation into a single block than the cost model is designed to permit, degrading full-node performance beyond the intended bound — a consensus-relevant cost-accounting divergence.

---

### Likelihood Explanation

The attack requires only crafting a CLVM program with nested `modpow` calls. No special privileges, keys, or social engineering are needed. Any coin spend submitted to the Chia mempool can carry such a program. The mathematical identity `(a^{e_1})^{e_2} = a^{e_1 e_2}` is elementary and the split is trivially constructible by choosing `e1` and `e2` freely. The `modpow` operator is available in the standard `ChiaDialect` and is reachable from any CLVM program.

---

### Recommendation

Replace the quadratic exponent cost formula with one that is **linear** in `esize`, matching the actual O(esize) squaring count of the square-and-multiply algorithm:

```rust
// Before (quadratic — gameable by splitting):
cost += (esize * esize) as Cost * MODPOW_COST_PER_BYTE_EXPONENT;

// After (linear — splitting gives no advantage):
cost += esize as Cost * MODPOW_COST_PER_BYTE_LINEAR;
```

Alternatively, add a guard that prevents the exponent cost from being reduced by splitting: charge a minimum cost proportional to the maximum possible exponent size regardless of the actual `esize`. The modulus quadratic term (`msize²`) does not have this problem because `modpow` takes exactly one modulus argument and it cannot be split.

---

### Proof of Concept

The following CLVM pseudocode demonstrates the cost difference. Both programs compute `a^e mod m` for the same `a`, `e = e1*e2`, `m`:

```lisp
; Strategy A: single call — charged esize=256, cost += 256²×3 = 196608
(modpow a e m)

; Strategy B: split — charged 2×(esize=128), cost += 2×128²×3 = 98304
(modpow (modpow a e1 m) e2 m)
```

Concrete numbers with `MODPOW_COST_PER_BYTE_EXPONENT = 3`, `MODPOW_BASE_COST = 17000`, `msize = 32`:

- **Strategy A total exponent cost**: `256² × 3 = 196 608`
- **Strategy B total exponent cost**: `2 × 128² × 3 = 98 304`
- **Extra base cost for Strategy B**: `17 000`
- **Extra modulus cost for Strategy B**: `32² × 21 = 21 504`
- **Net saving**: `196 608 − 98 304 − 17 000 − 21 504 = 59 800 cost units`
