### Title
Sanctioned Address Can Receive Funds via Fast Withdrawal — Missing `requireUnsanctioned` Check in `submitFastWithdrawal` - (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal` is a publicly callable function that transfers collateral tokens to a `sendTo` address derived from a signed withdrawal transaction. Unlike `Endpoint.depositCollateralWithReferral`, which enforces `requireUnsanctioned` on both the caller and the subaccount owner, `submitFastWithdrawal` performs no sanctions check on any party — not the caller (`msg.sender`), not the subaccount owner (signer), and not the `sendTo` recipient. This allows a sanctioned address to receive protocol funds via the fast withdrawal path, directly contradicting the sanctions enforcement applied on the deposit path.

---

### Finding Description

`Endpoint.depositCollateralWithReferral` enforces two sanctions checks before crediting any collateral:

```solidity
// Endpoint.sol lines 134–135
requireUnsanctioned(msg.sender);
requireUnsanctioned(sender); // sender = address(bytes20(subaccount))
``` [1](#0-0) 

`EndpointTx.submitSlowModeTransactionImpl` similarly enforces a sanctions check on the submitter before queuing any slow-mode transaction:

```solidity
// EndpointTx.sol line 375
requireUnsanctioned(sender);
``` [2](#0-1) 

However, `BaseWithdrawPool.submitFastWithdrawal` — the public fast-withdrawal entry point — contains no `requireUnsanctioned` call anywhere in its body:

```solidity
function submitFastWithdrawal(
    uint64 idx,
    bytes calldata transaction,
    bytes[] calldata signatures
) public {
    require(!markedIdxs[idx], "Withdrawal already submitted");
    require(idx > minIdx, "idx too small");
    markedIdxs[idx] = true;

    Verifier v = Verifier(verifier);
    v.requireValidTxSignatures(transaction, idx, signatures);

    (uint32 productId, address sendTo, uint128 transferAmount) = resolveFastWithdrawal(transaction);
    ...
    handleWithdrawTransfer(token, sendTo, transferAmount);
}
``` [3](#0-2) 

The `sendTo` address is resolved from the signed transaction payload. For `WithdrawCollateralV2`, `sendTo` can be an arbitrary address explicitly set in the transaction (`signedTx.tx.sendTo`), distinct from the subaccount owner:

```solidity
address resolvedSendTo = signedTx.tx.sendTo == address(0)
    ? address(uint160(bytes20(signedTx.tx.sender)))
    : signedTx.tx.sendTo;
return (signedTx.tx.productId, resolvedSendTo, signedTx.tx.amount);
``` [4](#0-3) 

`requireUnsanctioned` is defined in `EndpointStorage` and is available throughout the contract hierarchy: [5](#0-4) 

---

### Impact Explanation

A sanctioned address can receive protocol collateral via the fast withdrawal path. Specifically:

- A non-sanctioned subaccount owner can sign a `WithdrawCollateralV2` transaction with `sendTo` set to a sanctioned address. Any caller can then submit this to `submitFastWithdrawal`, and the funds are transferred to the sanctioned recipient with no check.
- Additionally, a sanctioned address can itself call `submitFastWithdrawal` as `msg.sender` (e.g., to collect the fee rebate when `sendTo == msg.sender`), since there is no check on the caller.

This breaks the protocol's sanctions enforcement invariant: funds must not flow to or from sanctioned addresses. The deposit path enforces this; the fast withdrawal path does not.

---

### Likelihood Explanation

`submitFastWithdrawal` is a public, permissionless function callable by any address. The `WithdrawCollateralV2` transaction type explicitly supports a custom `sendTo` field. A sanctioned address that cannot deposit directly can still receive funds if a cooperating non-sanctioned subaccount owner signs a V2 withdrawal directed to them. No privileged access is required.

---

### Recommendation

Add `requireUnsanctioned` checks in `submitFastWithdrawal` for:
1. `msg.sender` — the caller submitting the fast withdrawal.
2. `sendTo` — the address that will receive the funds.

Optionally also check the subaccount owner address (`address(uint160(bytes20(signedTx.tx.sender)))`), consistent with the two-sided check in `depositCollateralWithReferral`.

---

### Proof of Concept

1. Sanctioned address `S` cannot call `depositCollateralWithReferral` — reverts at `requireUnsanctioned(sender)`.
2. Non-sanctioned subaccount owner `A` signs a `WithdrawCollateralV2` transaction with `sendTo = S` and `amount = X`.
3. Any caller submits this signed transaction to `WithdrawPool.submitFastWithdrawal(idx, transaction, signatures)`.
4. `v.requireValidTxSignatures` passes (valid signature from `A`).
5. `resolveFastWithdrawal` returns `sendTo = S`.
6. `handleWithdrawTransfer(token, S, X)` executes — sanctioned address `S` receives `X` tokens.
7. No revert occurs; no sanctions check was ever performed on `S`. [3](#0-2)
