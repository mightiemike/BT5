### Title
Sanctioned Subaccount Owner Bypass via Linked Signer in `WithdrawCollateral` — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `processTransactionImpl` function in `EndpointTx` processes `WithdrawCollateral` transactions with `allowLinkedSigner = true` but never calls `requireUnsanctioned` on the subaccount owner (`signedTx.tx.sender`). A linked signer — a non-sanctioned address previously authorized by the subaccount owner — can sign and trigger a withdrawal from a sanctioned subaccount, bypassing the protocol's on-chain sanctions enforcement.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `WithdrawCollateral` branch is:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // allowLinkedSigner
);
...
clearinghouse.withdrawCollateral(
    signedTx.tx.sender,
    signedTx.tx.productId,
    signedTx.tx.amount,
    address(0),
    nSubmissions
);
``` [1](#0-0) 

`validateSignedTx` internally calls only `validateNonce`, `validateSignature`, and `requireSubaccount`. None of these check whether `signedTx.tx.sender` (the subaccount owner whose funds are being withdrawn) is sanctioned. [2](#0-1) 

`validateSignature` with `allowLinkedSigner = true` resolves the linked signer via `getLinkedSigner(sender)` and passes it to `verifier.validateSignature`, which accepts a signature from either the subaccount owner **or** the linked signer. [3](#0-2) [4](#0-3) 

The linked signer is stored in `linkedSigners[subaccount]` and set via the `LinkSigner` transaction type. [5](#0-4) 

By contrast, `depositCollateralWithReferral` explicitly calls `requireUnsanctioned` on **both** `msg.sender` (the depositor) and `sender` (the subaccount owner), demonstrating the protocol's clear intent to enforce sanctions on both parties to any fund movement. [6](#0-5) 

`requireUnsanctioned` is defined in `EndpointStorage` and queries the OFAC-style `ISanctionsList`: [7](#0-6) 

`Clearinghouse.withdrawCollateral` has no sanctions check of its own; it is `onlyEndpoint` and trusts the caller to have already validated the sender. [8](#0-7) 

---

### Impact Explanation

A linked signer (non-sanctioned) can drain the full collateral balance of a sanctioned subaccount by signing a `WithdrawCollateral` transaction. When `sendTo == address(0)` (the only option in V1), `Clearinghouse.withdrawCollateral` resolves the destination to `address(uint160(bytes20(sender)))` — the sanctioned address itself — and transfers the tokens out of the protocol's custody via `handleWithdrawTransfer`. [9](#0-8) 

This directly corrupts the protocol's sanctions invariant: funds belonging to a sanctioned entity are moved out of the clearinghouse, defeating the purpose of the `requireUnsanctioned` mechanism. The asset delta is the full collateral balance of the sanctioned subaccount across any product.

---

### Likelihood Explanation

The `LinkSigner` feature is a supported, documented user flow. Any user who set a linked signer before being sanctioned is vulnerable. Sanctions can be applied at any time after the linked signer is registered. The sequencer is a trusted relay but may not have real-time OFAC data or may process the linked signer's transaction before its own sanctions filter is updated. The on-chain contract provides no backstop. The attack requires no special privileges beyond having previously been a linked signer of the sanctioned account.

---

### Recommendation

Add a `requireUnsanctioned` check on the subaccount owner address in the `WithdrawCollateral` (and `WithdrawCollateralV2`) branch of `processTransactionImpl`, mirroring the pattern already used in `depositCollateralWithReferral`:

```solidity
// In processTransactionImpl, WithdrawCollateral branch:
requireUnsanctioned(address(uint160(bytes20(signedTx.tx.sender))));
```

This should be applied consistently to all transaction types that use `allowLinkedSigner = true` and affect fund balances, including `MintNlp`, `BurnNlp`, and `TransferQuote`. [10](#0-9) 

---

### Proof of Concept

1. **Setup:** Alice (`0xAlice`) calls `submitSlowModeTransaction` with a `LinkSigner` transaction, setting Bob (`0xBob`) as her linked signer. `linkedSigners[aliceSubaccount] = 0xBob`. [11](#0-10) 

2. **Sanction:** Alice's address `0xAlice` is added to the `ISanctionsList`. `sanctions.isSanctioned(0xAlice) == true`.

3. **Exploit:** Bob signs a `WithdrawCollateral` transaction for `aliceSubaccount`. The sequencer includes it in `submitTransactionsChecked`. `processTransactionImpl` is called. [12](#0-11) 

4. **No sanctions check:** `validateSignedTx` accepts Bob's signature (linked signer path). `requireSubaccount` passes (Alice's subaccount exists). No `requireUnsanctioned(0xAlice)` is ever called. [13](#0-12) 

5. **Funds moved:** `clearinghouse.withdrawCollateral` executes, deducting Alice's balance and transferring tokens to `0xAlice` via `handleWithdrawTransfer`. The sanctioned account's collateral exits the protocol. [14](#0-13)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
