### Title
Linked Signer Can Overwrite Its Own Delegation Without Subaccount Owner Consent — (`core/contracts/EndpointTx.sol`)

---

### Summary
The fast-path `LinkSigner` transaction handler passes `allowLinkedSigner = true` to `validateSignedTx`, meaning the **current linked signer** can sign a `LinkSigner` transaction and replace itself with any arbitrary address. The subaccount owner never consented to this re-delegation. A compromised or malicious linked signer can silently hand off signing authority to an attacker-controlled address, which then drains the subaccount via `WithdrawCollateral` or `TransferQuote`.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch is:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature` / `validateCompactSignature` and passes `getLinkedSigner(sender)` as the accepted signer:

```solidity
// EndpointTx.sol lines 172–184
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

This means the **existing linked signer** is a valid signer for a `LinkSigner` transaction, allowing it to write any new address into `linkedSigners[subaccount]`.

Contrast this with the **slow-mode** `LinkSigner` path, which enforces `validateSender(txn.sender, sender)` — requiring the actual subaccount owner address to be the `msg.sender`:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    validateSender(txn.sender, sender);   // owner-only
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

The fast path has no equivalent owner-only guard, creating an asymmetry: the slow path correctly restricts `LinkSigner` to the subaccount owner, but the fast path allows the linked signer to re-delegate to any address.

---

### Impact Explanation

Once the linked signer replaces itself with an attacker address, the attacker becomes the accepted signer for all `allowLinkedSigner = true` operations on that subaccount, including:

- **`WithdrawCollateral`** (fast path, `allowLinkedSigner = true`, line 423) — attacker can withdraw all collateral to any address.
- **`WithdrawCollateralV2`** (fast path, `allowLinkedSigner = true` when `sendTo == address(0)`, line 447) — same.
- **`TransferQuote`** (fast path, `allowLinkedSigner = true`, line 604) — attacker can transfer quote balance to an attacker-controlled subaccount.
- **`MintNlp` / `BurnNlp`** (fast path, `allowLinkedSigner = true`, lines 539, 559) — attacker can manipulate NLP positions. [4](#0-3) [5](#0-4) 

The corrupted state delta is: `linkedSigners[victim_subaccount]` is overwritten to an attacker address, enabling full collateral theft from the victim subaccount.

---

### Likelihood Explanation

Linked signers are hot wallets used for automated trading bots. They are a well-known high-value target. A compromised linked signer key — through server breach, key exfiltration, or a malicious trading bot — can silently escalate to full account takeover without the owner ever being notified on-chain. The owner set the linked signer expecting it to have bounded authority (trade on their behalf), not unbounded authority (re-delegate to anyone). This is a realistic attack path for any user who has enabled a linked signer.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in the fast path, so only the subaccount owner's key can authorize a linked signer change:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This aligns the fast path with the slow path's intent and removes the re-delegation capability from the linked signer role.

---

### Proof of Concept

1. Alice sets Bob (`0xB0b`) as her linked signer via a `LinkSigner` slow-mode transaction. `linkedSigners[alice_subaccount] = 0xB0b`.
2. Bob's key is compromised. Attacker controls `0xB0b`.
3. Attacker crafts a `SignedLinkSigner` transaction with `sender = alice_subaccount`, `signer = attacker_address`, `nonce = current_nonce`, signed by `0xB0b`.
4. Attacker submits this to the sequencer (or waits for it to be included in a batch). The sequencer processes it via `processTransactionImpl`.
5. `validateSignedTx(..., true)` passes because `getLinkedSigner(alice_subaccount) == 0xB0b` and the signature is valid.
6. `linkedSigners[alice_subaccount] = attacker_address` is written.
7. Attacker now crafts a `SignedWithdrawCollateral` transaction signed by `attacker_address`, draining Alice's entire collateral balance.
8. Alice's funds are stolen. She had no on-chain warning and no ability to react before step 7. [6](#0-5)
