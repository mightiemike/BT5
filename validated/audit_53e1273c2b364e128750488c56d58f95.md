Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates the direct caller of `pool.swap()` as the `sender` argument. When users route through `MetricOmmSimpleRouter`, the router is that direct caller. If the pool admin allowlists the router so that allowlisted users can use it, every unprivileged user can bypass the swap allowlist by routing through the same router contract.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks: [2](#0-1) 

where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`. When a user swaps through `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly: [3](#0-2) 

The router is `msg.sender` to the pool, so `sender` in `beforeSwap` is the router's address. The extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. The pool admin faces an impossible choice: don't allowlist the router and allowlisted EOAs cannot use the router at all; allowlist the router and every user can bypass the allowlist. The same wrong-actor binding applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) [5](#0-4) 

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict swaps to specific counterparties is fully bypassed. Any unprivileged user calls `router.exactInputSingle()` on the curated pool; the router's address passes the allowlist check; the swap executes. LP funds in the curated pool are exposed to unrestricted trading that the pool admin explicitly intended to prevent. This is broken core functionality causing direct loss of policy enforcement above Sherlock thresholds.

## Likelihood Explanation
High. `MetricOmmSimpleRouter` is the standard periphery swap interface. Any user who wants to swap on a curated pool will naturally use it. No special privilege, flash loan, or multi-step setup is required — a single `exactInputSingle` call suffices.

## Recommendation
The extension must gate the economically relevant actor, not the intermediary. Options:

1. **Router-injected identity**: Have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension` decode and verify it (requires trusting the router, enforceable by checking `msg.sender` is a factory-registered pool).
2. **Transient payer forwarding**: The router already stores the real payer in transient storage for callback settlement; expose a read function so extensions can query it.
3. **Direct-pool-only policy**: Document that `SwapAllowlistExtension` only works correctly when users call `pool.swap()` directly, and add a guard that reverts if the caller is a known router contract.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension.
2. Admin: setAllowedToSwap(pool, alice, true)         // alice is the intended grantee
3. Admin: setAllowedToSwap(pool, router, true)         // required so alice can use the router
4. bob (non-allowlisted) calls:
       router.exactInputSingle({pool: pool, tokenIn: ..., amountIn: X, ...})
5. Router calls pool.swap(recipient, ...) — msg.sender to pool = router
6. Pool calls _beforeSwap(sender=router, ...)
7. SwapAllowlistExtension checks allowedSwapper[pool][router] → true
8. Bob's swap executes on the curated pool, bypassing the allowlist entirely.
```

The corrupted value is the identity checked by the guard: `router` is substituted for `bob`, so the allowlist invariant `allowedSwapper[pool][bob]` is never evaluated. [6](#0-5) [7](#0-6)
