### Title
Misconfigured `MAX_FUTURE_BLOCK_TIME_MTP` Constant Causes Systematic Rejection of Valid Zcash Block Headers - (File: `btc-types/src/network.rs`)

---

### Summary

The constant `MAX_FUTURE_BLOCK_TIME_MTP` is defined as `90 * 60 = 5,400 seconds` (90 minutes), but the Zcash protocol requires this value to be `129,600 seconds` (36 hours). The `zcash.rs` implementation uses this constant to gate a mandatory timestamp check against the median-time-past (MTP). Because the constant is 24× too small, any valid Zcash block header whose timestamp is more than 90 minutes ahead of the MTP will be permanently rejected by the light client, even though the Zcash protocol explicitly permits up to 36 hours.

---

### Finding Description

In `btc-types/src/network.rs`, the constant is defined as:

```rust
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60;   // 5,400 seconds
``` [1](#0-0) 

In `contract/src/zcash.rs`, this constant is applied in `check_pow` with an inline comment that contradicts the actual value:

```rust
// MAX_FUTURE_BLOCK_TIME_MTP is typically 129600 seconds (36 hours) in Zcash
require!(
    block_header.time
        <= next_work_result.prev_block_median_time_past + MAX_FUTURE_BLOCK_TIME_MTP,
    "time-too-far-ahead-of-mtp: block timestamp is too far ahead of median-time-past"
);
``` [2](#0-1) 

The comment itself acknowledges the correct Zcash protocol value (`129,600 s = 36 h`), but the constant actually used is `5,400 s = 90 min` — a 24× undercount. The Zcash reference implementation (linked in the file at `https://github.com/zcash/zcash/blob/v6.2.0/src/main.cpp#L5019`) enforces 36 hours for this bound. The `MAX_FUTURE_BLOCK_TIME_LOCAL` constant (2 hours, used for the wall-clock check) is a separate, correctly-valued guard. [3](#0-2) 

The `check_pow` function for Zcash is the only chain variant that imports and applies `MAX_FUTURE_BLOCK_TIME_MTP`; Bitcoin, Litecoin, and Dogecoin do not use it. [4](#0-3) 

---

### Impact Explanation

Every Zcash block header submitted to `submit_blocks` whose `time` field exceeds `prev_block_median_time_past + 5400` will be unconditionally rejected with `"time-too-far-ahead-of-mtp"`. Because the Zcash protocol permits up to 36 hours of forward drift from MTP, a large class of fully valid, consensus-conforming Zcash headers will be permanently refused. This breaks the canonical chain tracking for the Zcash deployment of the light client: the `mainchain_tip_blockhash` stalls, `get_last_block_header` returns a stale tip, and any downstream NEAR contract calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against a Zcash block that arrived after a slow-block period will receive incorrect results or a panic. [5](#0-4) 

---

### Likelihood Explanation

Zcash has a 75-second target block time. A gap of only 90 minutes between the MTP and a new block's timestamp is reachable whenever the network experiences a modest slowdown or when the 11-block MTP window lags behind real time — a routine occurrence during any brief mining disruption. The relayer will submit the header, `check_pow` will evaluate the MTP check with the wrong bound, and the call will revert. Because the constant is hardcoded, every subsequent submission of that header will also revert; there is no operator-adjustable parameter to work around it. [6](#0-5) 

---

### Recommendation

Change the constant in `btc-types/src/network.rs` to match the Zcash protocol specification:

```rust
// 36 hours — matches Zcash consensus rule (nMaxFutureBlockTime for MTP check)
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 129_600;
``` [1](#0-0) 

---

### Proof of Concept

1. Deploy the contract in Zcash mode (`feature = "zcash"`).
2. Initialize with a genesis block whose MTP is `T`.
3. Submit a valid Zcash block header with `time = T + 5401` (91 minutes ahead of MTP, well within the 36-hour Zcash rule).
4. `check_pow` evaluates `5401 <= 5400` → `false` → panics with `"time-too-far-ahead-of-mtp"`.
5. The same header submitted to a reference Zcash node is accepted without error.

The revert is deterministic and repeatable for any header in the window `(MTP + 5400, MTP + 129600]`, which covers the entire range of timestamps that Zcash considers valid but this contract rejects. [7](#0-6)

        require!(

            "time-too-old: block time is before the median time of the previous block"
        );

        // Check future timestamp soft fork rule introduced in v2.1.1-1.
        // This retrospectively activates at block height 2 for mainnet and regtest,
        // and 6 blocks after Blossom activation for testnet.
        //
        // MAX_FUTURE_BLOCK_TIME_MTP is typically 129600 seconds (36 hours) in Zcash
        require!(
            block_header.time
                <= next_work_result.prev_block_median_time_past + MAX_FUTURE_BLOCK_TIME_MTP,
            "time-too-far-ahead-of-mtp: block timestamp is too far ahead of median-time-past"

        // Check timestamp
        let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap(); // Convert to seconds

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
