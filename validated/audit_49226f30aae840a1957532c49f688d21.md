### Title
`fee_proposal_fri` Accepted Without Bounds During Startup Window Poisons `fee_actual` Computation â€” (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

During the first `fee_proposal_window_size` blocks (10 blocks at genesis or after a restart), the validator in `is_proposal_init_valid` skips all bounds enforcement on the proposer-supplied `fee_proposal_fri` field because `fee_actual` is `None`. A Byzantine proposer can set an arbitrarily extreme `fee_proposal_fri` value during this unguarded window. Those values are stored in `fee_proposals_window` and become the seed for the sliding-window `fee_actual` computation that governs all future SNIP-35 L2 gas price signals.

---

### Finding Description

In `is_proposal_init_valid`, the bounds check on `fee_proposal_fri` is gated on `fee_actual` being `Some`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(...);
    }
}
``` [1](#0-0) 

When `fee_actual` is `None` (the first `fee_proposal_window_size` = 10 blocks), the inner block is never entered. Any `fee_proposal_fri` value â€” including `u128::MAX` â€” passes validation as long as it is `Some` for V0_14_3+. [2](#0-1) 

The accepted `fee_proposal_fri` is then bound into the `ProposalCommitment` via `proposal_commitment_from(partial_block_hash, fee_proposal_fri)` = `Poseidon(partial, fee_proposal_fri)`, so the validator's commitment check at line 244 passes as long as the proposer's `ProposalFin` carries the matching hash â€” which the proposer controls. [3](#0-2) 

After `decision_reached`, the accepted `fee_proposal_fri` is recorded into `fee_proposals_window`. Once the window accumulates `fee_proposal_window_size` entries, `compute_fee_actual` produces a `fee_actual` derived from these potentially-poisoned values. All subsequent `fee_proposal_fri` values are then bounded within `fee_proposal_margin_ppt` (0.2 â€° per block) of this poisoned baseline. [4](#0-3) 

The proposer side explicitly documents the symmetric fallback: when `fee_actual` is `None`, it freezes at `l2_gas_price`. The validator side, however, does not enforce any corresponding floor/ceiling on the received value â€” it simply skips the check entirely. [5](#0-4) 

The `fee_proposal_window_size` is 10 and `fee_proposal_margin_ppt` is 2 across all deployed versioned-constants files, confirming the window is small and the per-block drift cap is tight â€” meaning a poisoned seed propagates for many hundreds of blocks before honest proposals can correct it.

---

### Impact Explanation

A Byzantine proposer who controls even a single block during the 10-block startup window can inject `fee_proposal_fri = u128::MAX` (or near-zero). Once the window fills, `fee_actual` is computed from these poisoned entries. All future honest proposals are then bounded within Â±0.2 â€° per block of the poisoned `fee_actual`. Because the drift cap is 0.2 â€°, correcting a `u128::MAX` seed back to a reasonable value would require on the order of tens of thousands of blocks. During this period, the SNIP-35 L2 gas price signal is inflated (or deflated), causing incorrect fee accounting with direct economic impact on users.

This matches: **Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

---

### Likelihood Explanation

The startup window is exactly 10 blocks. In a BFT network with N validators, a single Byzantine validator is expected to propose approximately 10/N blocks during this window. With a small validator set (e.g., N = 4), a single Byzantine node can inject 2â€“3 poisoned values. Because `compute_fee_actual` uses a sliding-window average (not a median), even a minority of extreme values significantly skews the result. The attack requires no special privilege beyond being a scheduled proposer â€” a normal role in BFT consensus.

---

### Recommendation

Add an absolute bounds check that applies even when `fee_actual` is `None`. For example, enforce that `fee_proposal_fri` lies within `[min_gas_price, some_configured_max]` unconditionally for V0_14_3+ blocks. This mirrors the `apply_fee_transformations` clamp already applied to L1 gas prices: [6](#0-5) 

A simple floor/ceiling on `fee_proposal_fri` at validation time â€” independent of whether `fee_actual` is available â€” would prevent extreme seed values from entering `fee_proposals_window` without breaking the intentional "no relative-margin enforcement during startup" design.

---

### Proof of Concept

1. Network starts at genesis (or after a restart). `fee_proposals_window` is empty; `fee_actual` is `None` for the first 10 blocks.
2. A Byzantine validator is scheduled as proposer for block 1. It sends `ProposalInit` with `fee_proposal_fri = Some(GasPrice(u128::MAX))` and `starknet_version >= V0_14_3`.
3. `is_proposal_init_valid` reaches the fee-proposal check at line 398. Because `proposal_init_validation.fee_actual` is `None`, the `if let` does not match and the bounds check is skipped entirely. [1](#0-0) 
4. The validator computes `built_block = proposal_commitment_from(partial_block_hash, Some(GasPrice(u128::MAX)))`. The Byzantine proposer sends a matching `ProposalFin`. The commitment check at line 244 passes.
5. `decision_reached` records `fee_proposal_fri = u128::MAX` into `fee_proposals_window[1]`.
6. After 10 blocks (with the Byzantine node injecting extreme values on its turns), `fee_actual` is computed from the poisoned window. All subsequent honest proposals are bounded within Â±0.2 â€° of this extreme baseline, locking the SNIP-35 fee signal at an economically incorrect level for hundreds of blocks.

    /// Compute the proposer's fee_proposal: clamp the oracle's `fee_target` to a margin around
    /// `fee_actual`. When `fee_actual` is `None` (window incomplete), freeze at `l2_gas_price`; the
    /// validator derives the same fallback so both sides agree.
    async fn compute_proposer_fee_proposal(
        &self,
        fee_actual: Option<GasPrice>,
        timestamp: u64,
        target_atto_usd_per_l2_gas: u128,
    ) -> GasPrice {
        SNIP35_FEE_TARGET_ATTO_USD.set_lossy(target_atto_usd_per_l2_gas);
        let Some(fee_actual) = fee_actual else {
            warn!("fee_actual unavailable, freezing fee_proposal at l2_gas_price");
            SNIP35_FEE_PROPOSAL_FRI.set_lossy(self.l2_gas_price.0);
            return self.l2_gas_price;
        };
        SNIP35_FEE_ACTUAL_FRI.set_lossy(fee_actual.0);

        let fee_target = self.resolve_fee_target(timestamp, target_atto_usd_per_l2_gas).await;

        let proposal = compute_fee_proposal(
            fee_target,
            fee_actual,
            VersionedConstants::latest_constants().fee_proposal_margin_ppt,
        );
        SNIP35_FEE_PROPOSAL_FRI.set_lossy(proposal.0);
        proposal
    }

pub(crate) fn apply_fee_transformations(
    price_info: &mut PriceInfo,
    gas_price_params: &GasPriceParams,
) {
    price_info.base_fee_per_gas = price_info
        .base_fee_per_gas
        .saturating_add(gas_price_params.l1_gas_tip_wei)
        .clamp(gas_price_params.min_l1_gas_price_wei, gas_price_params.max_l1_gas_price_wei);

    price_info.blob_fee = GasPrice(
        (gas_price_params.l1_data_gas_price_multiplier * price_info.blob_fee.0).to_integer(),
    )
    .clamp(gas_price_params.min_l1_data_gas_price_wei, gas_price_params.max_l1_data_gas_price_wei);
}
