Audit Report

## Title
`DepositAllowlistExtension` gates on caller-supplied `owner` instead of `sender`, allowing full allowlist bypass — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual token payer) and checks only `owner` (the position recipient, freely chosen by the caller). Because `owner` is an unconstrained caller-supplied parameter, any unprivileged address can bypass the deposit allowlist by nominating any allowlisted address as `owner`. The naming of the mapping (`allowedDepositor`) and setter (`setAllowedToDeposit`) confirms the intended semantics are about the caller, not the position recipient.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position recipient into `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly encodes both addresses and forwards them to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, …)` but the first parameter is unnamed (discarded) and the guard checks only `owner`: [3](#0-2) 

The admin-facing setter names the second argument `depositor`, confirming the intended semantics are about the caller, not the position recipient: [4](#0-3) 

The periphery's `_validateOwner` provides no protection — it only rejects `address(0)`, not arbitrary non-`msg.sender` addresses: [5](#0-4) 

**Exploit path:**
1. Bob (not allowlisted) calls `pool.addLiquidity(owner = Alice, …)` where Alice is allowlisted.
2. Pool calls `_beforeAddLiquidity(sender = Bob, owner = Alice, …)`.
3. `DepositAllowlistExtension` evaluates `allowedDepositor[pool][Alice]` → `true` → no revert.
4. Liquidity is minted into Alice's position; Bob's tokens are pulled via the callback.

The same bypass works through `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner = Alice, …)` since `_validateOwner` only checks non-zero. [6](#0-5) 

## Impact Explanation
The deposit allowlist is the primary access-control mechanism for restricted pools (KYC-gated, institutional, or curated LP sets). The bypass satisfies the **admin-boundary break** impact gate: a pool admin-configured guard is bypassed by an unprivileged path. Any unprivileged address can inject liquidity into a pool the admin explicitly locked down. Additionally, the attacker forces an unsolicited position onto the allowlisted address; since `removeLiquidity` requires `msg.sender == owner`, only Alice can withdraw those funds, making this a one-way griefing/donation vector that dilutes LP composition without admin authorization.

## Likelihood Explanation
No special privilege is required — any EOA or contract implementing `metricOmmModifyLiquidityCallback` can execute the bypass. The only prerequisite is knowing one allowlisted address, which is publicly readable from the `allowedDepositor` mapping or observable on-chain. The attacker must spend tokens (they go to the allowlisted owner's position), limiting purely profit-motivated attacks, but griefing or regulatory-evasion scenarios require no profit motive. Likelihood: **Medium**.

## Recommendation
Replace the `owner` check with a `sender` check to gate the actual depositor:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intended semantics are to gate position *owners* rather than callers, rename the mapping and setter to `allowedOwner`/`setAllowedOwner` to make the design explicit. Either way, the current implementation does not match the naming (`allowedDepositor`, `setAllowedToDeposit`) and allows a full allowlist bypass.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

contract BypassDeposit {
    IMetricOmmPoolActions pool;
    IERC20 token0;
    IERC20 token1;

    // Bob calls this; alice is an allowlisted address
    function exploit(address alice, LiquidityDelta calldata deltas) external {
        // sender passed to extension = address(this) (not allowlisted)
        // owner = alice (allowlisted) → extension checks allowedDepositor[pool][alice] = true → passes
        pool.addLiquidity(alice, 0, deltas, abi.encode(address(this)), "");
    }

    function metricOmmModifyLiquidityCallback(
        uint256 amount0Delta,
        uint256 amount1Delta,
        bytes calldata
    ) external {
        if (amount0Delta > 0) token0.transferFrom(msg.sender, address(pool), amount0Delta);
        if (amount1Delta > 0) token1.transferFrom(msg.sender, address(pool), amount1Delta);
    }
}
```

Bob's call succeeds despite not being allowlisted, because `DepositAllowlistExtension` only checks `owner = alice`.
