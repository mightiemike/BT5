### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. The pool always passes `msg.sender` of `pool.swap()` as `sender`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` in the pool is the **router contract**, not the actual end user. The extension therefore checks whether the **router** is allowlisted, not whether the actual user is allowlisted. If the pool admin allowlists the router (a natural step to let allowlisted users use the router), every user — including non-allowlisted ones — can bypass the per-user gate by routing through the router.

---

### Finding Description

**Pool → Extension sender binding:**

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

**The allowlist check:**

`SwapAllowlistExtension.beforeSwap` uses `sender` (the first argument) as the identity to gate: [3](#0-2) 

**The router's call site:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly. At that call, `msg.sender` inside the pool is the **router**, not the original user: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**Result:** The extension receives `sender = address(router)` and evaluates `allowedSwapper[pool][router]`. The actual user's allowlist status is never consulted.

---

### Impact Explanation

A pool admin who deploys a curated pool (e.g., KYC-only, institution-only) with `SwapAllowlistExtension` must allowlist the router if they want allowlisted users to be able to use the standard periphery. The moment the router is allowlisted, the gate is open to every user on-chain: any non-allowlisted address can call `router.exactInputSingle(...)` and the extension will pass because it sees the allowlisted router as the swapper. The non-allowlisted user can then execute swaps against the pool's LP liquidity at oracle prices, directly extracting value from LPs who deposited under the assumption that only vetted counterparties could trade.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the standard, documented periphery swap path. Any pool admin who wants allowlisted users to be able to use the router must add the router to the allowlist. This is the expected operational pattern. Once the router is allowlisted, the bypass is trivially reachable by any address with no special privileges — just call `exactInputSingle` on the router pointing at the curated pool.

---

### Recommendation

The `SwapAllowlistExtension` should gate on the **original user**, not the direct caller of `pool.swap()`. Two approaches:

1. **Pass the original user through `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.

2. **Check `recipient` instead of `sender`:** For single-hop swaps the recipient is often the user, but this is not reliable for multi-hop paths.

3. **Preferred — check `tx.origin` or a signed identity in `extensionData`:** The cleanest fix is to have the router encode the original `msg.sender` into `extensionData` and have the extension verify it, similar to how `MetricOmmPoolLiquidityAdder` stores the payer in transient context.

The `DepositAllowlistExtension` is **not** affected by this issue because it gates on `owner` (the position owner argument), which is explicitly passed by the caller and is not overwritten by the liquidity adder: [6](#0-5) 

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` as the `beforeSwap` extension.
2. Allowlist only `user1` and the router: `setAllowedToSwap(pool, user1, true)` and `setAllowedToSwap(pool, router, true)`.
3. As `user2` (not allowlisted), call `router.exactInputSingle({pool: pool, ...})`.
4. Inside the pool, `msg.sender == router`. The extension checks `allowedSwapper[pool][router] == true` → passes.
5. `user2`'s swap executes against LP liquidity despite never being allowlisted.

Contrast with a direct call: `pool.swap(...)` from `user2` → extension checks `allowedSwapper[pool][user2] == false` → reverts `NotAllowedToSwap`. The bypass is exclusively available through the router path. [7](#0-6) [8](#0-7)
