Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Full Allowlist Bypass - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` as seen by the pool — the router address when a swap is routed through `MetricOmmSimpleRouter`. If the pool admin allowlists the router to enable router-mediated swaps for legitimate users, every unprivileged user can bypass the allowlist by routing through the router, rendering the allowlist completely ineffective.

## Finding Description
In `MetricOmmPool.swap`, `_beforeSwap` is called with `msg.sender` as the first argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as `sender` to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly: [4](#0-3) 

So `msg.sender` inside the pool is the **router**, not the original user. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`. The admin faces an irreconcilable dilemma: not allowlisting the router breaks legitimate router usage; allowlisting the router lets any unprivileged user bypass the allowlist. `DepositAllowlistExtension` avoids this by checking `owner` (the explicitly supplied position owner), not `sender`: [5](#0-4) 

## Impact Explanation
A pool deployer using `SwapAllowlistExtension` to restrict trading to KYC-verified or whitelisted addresses loses all curation guarantees once the router is allowlisted. Any unprivileged address can trade against the pool without restriction. This constitutes broken core pool functionality — the swap guard silently fails open — and is a direct policy bypass impacting every pool relying on this extension with router support enabled.

## Likelihood Explanation
The trigger requires no special privilege. Any user with the pool address and the router address can exploit this. The only precondition is that the pool admin has allowlisted the router, which is the natural and expected administrative action for any pool intending to support the standard periphery. The router is a factory-validated, publicly known contract, making this precondition highly likely in production deployments.

## Recommendation
The extension must gate the **original user**, not the intermediary. The most straightforward fix is for the router to encode `msg.sender` (the original user) into `extensionData`, and for `SwapAllowlistExtension.beforeSwap` to decode and check that address when `sender` is a registered periphery contract. Alternatively, the factory could expose a registry of trusted periphery contracts, and the extension falls back to checking `extensionData`-encoded identity when `sender` is a registered router.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Admin calls setAllowedToSwap(pool, alice, true)  // alice is the only permitted swapper
  - Admin calls setAllowedToSwap(pool, router, true) // allowlist router for periphery support

Attack (executed by bob, a non-allowlisted address):
  1. bob calls MetricOmmSimpleRouter.exactInputSingle({pool, recipient: bob, zeroForOne: true, amountIn: X, ...})
  2. Router calls pool.swap(bob, true, X, ..., extensionData)
     → msg.sender inside pool = router
  3. Pool calls _beforeSwap(router, bob, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  5. Swap executes successfully for bob despite bob not being on the allowlist.

Result: allowlist fully bypassed; bob trades on a curated pool without authorization.
```
