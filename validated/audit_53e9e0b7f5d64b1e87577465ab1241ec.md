### Title
Linked Signer Can Unilaterally Re-Delegate Signing Authority to an Unauthorized Third Party - (File: core/contracts/EndpointTx.sol)

### Summary
The `LinkSigner` transaction in the sequencer path is validated with `allowLinkedSigner = true`, meaning the current linked signer of a subaccount can sign a new `LinkSigner` transaction to replace themselves with any arbitrary address. The subaccount owner never consented to this re-delegation, and the new address gains full signing authority over the subaccount — including the ability to withdraw collateral, place/cancel orders, and liquidate positions.

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is handled as follows: [1](#0-0) 

The critical detail is the final argument `true` passed to `validateSignedTx`, which sets `allowLinkedSigner = true`: [2](#0-1) 

This routes through `validateSignature`, which passes the current `getLinkedSigner(sender)` as the permitted alternate signer to the verifier: [3](#0-2) 

The result is that the verifier accepts a signature from either the subaccount owner **or** the current linked signer. If the linked signer signs a `LinkSigner` transaction pointing to a new address, the mapping is overwritten: [4](#0-3) 

The `linkedSigners` mapping is stored in `EndpointStorage`: [5](#0-4) 

By contrast, the slow-mode path for `LinkSigner` correctly restricts this operation to the subaccount owner only, using `validateSender` which checks `msg.sender` against the address embedded in the subaccount bytes32: [6](#0-5) 

The sequencer path has no equivalent restriction.

### Impact Explanation

Once a linked signer re-delegates to Charlie, Charlie can sign any transaction that accepts `allowLinkedSigner = true`, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` — drain all collateral to an arbitrary address [7](#0-6) 
- `BurnNlp` — burn the subaccount's NLP position [8](#0-7) 
- `TransferQuote` — move quote balance to another subaccount [9](#0-8) 
- `LiquidateSubaccount` — trigger liquidation of other accounts on behalf of the victim [10](#0-9) 

The subaccount owner (Alice) granted Bob limited signing authority. Bob silently re-delegates to Charlie. Alice has no on-chain visibility into this change until Charlie acts. The corrupted state is `linkedSigners[subaccount]`, and the asset delta is the full collateral balance of the subaccount.

### Likelihood Explanation

Any address that has been granted linked-signer status can trigger this immediately. No special privileges, governance access, or external conditions are required. The attacker-controlled entry path is: Bob (linked signer) constructs a valid EIP-712 `LinkSigner` digest for Alice's subaccount, signs it with Bob's key, and submits it to the sequencer. The sequencer includes it in a batch, and `processTransactionImpl` overwrites `linkedSigners[aliceSubaccount]` with Charlie's address. This is a realistic scenario for any protocol user who has ever granted a linked signer (e.g., for automated trading bots).

### Recommendation

Change the `allowLinkedSigner` flag for `LinkSigner` processing in the sequencer path from `true` to `false`. Only the subaccount owner (the address embedded in the first 20 bytes of the `bytes32` subaccount) should be permitted to change the linked signer. This is already the correct behavior in the slow-mode path.

```solidity
// In processTransactionImpl, change:
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // was: true — linked signer must NOT be able to re-delegate
);
```

### Proof of Concept

1. Alice owns subaccount `aliceSub = bytes32(alice_address ++ name)` and links Bob via a `LinkSigner` transaction. `linkedSigners[aliceSub] = Bob`.
2. Bob constructs a `SignedLinkSigner` struct: `{ tx: { sender: aliceSub, signer: bytes32(Charlie), nonce: currentNonce }, signature: Bob_signs_EIP712_digest }`.
3. Bob submits this to the sequencer. The sequencer includes it in a `submitTransactionsChecked` batch.
4. `processTransactionImpl` reaches the `LinkSigner` branch, calls `validateSignedTx(..., true)`. The verifier accepts Bob's signature because Bob is `getLinkedSigner(aliceSub)`.
5. `linkedSigners[aliceSub]` is overwritten with Charlie's address.
6. Charlie now signs a `WithdrawCollateralV2` transaction for `aliceSub` with `sendTo = Charlie_EOA`, draining Alice's collateral. [11](#0-10)
