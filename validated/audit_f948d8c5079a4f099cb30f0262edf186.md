### Title
Linked Signer Can Re-Delegate Signing Authority to Arbitrary Address, Enabling Persistent Subaccount Impersonation — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-path `LinkSigner` handler in `EndpointTx.sol` accepts a signature from the **currently linked signer** to overwrite the linked signer mapping. This allows any address that has been granted linked-signer status to silently re-delegate that authority to an attacker-controlled address, achieving persistent impersonation of the victim subaccount without the owner's knowledge or consent.

---

### Finding Description

In `EndpointTx.sol`, the fast-path handler for `TransactionType.LinkSigner` calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves to `validateCompactSignature`, which in turn calls `Verifier.validateCompactSignature`. That function accepts the signature as valid if the recovered address equals **either** the subaccount owner address **or** the currently registered `linkedSigner`: [2](#0-1) 

The `getLinkedSigner` lookup used to resolve the allowed signer is: [3](#0-2) 

Because the `LinkSigner` transaction itself is validated with `allowLinkedSigner = true`, the currently linked signer can craft and submit a new `LinkSigner` transaction that replaces the `linkedSigners[subaccount]` entry with any address they choose — including a fresh attacker-controlled key. The subaccount owner is never consulted.

The slow-path `LinkSigner` handler correctly restricts this to the subaccount owner via `validateSender` (an `msg.sender` check): [4](#0-3) 

The fast path has no equivalent restriction.

---

### Impact Explanation

Once the linked signer re-delegates to an attacker-controlled address, that address can sign any fast-path transaction that passes `allowLinkedSigner = true`, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` — drain collateral from the victim subaccount
- `TransferQuote` — transfer quote balance to an arbitrary recipient subaccount
- `MintNlp` / `BurnNlp` — manipulate NLP positions on behalf of the victim [5](#0-4) 

The corrupted state is `linkedSigners[victimSubaccount]` — once overwritten, the attacker-controlled address holds full signing authority over the victim's subaccount. The owner retains the ability to sign directly (their address is always accepted), but in practice users who rely on linked signers (API keys, trading bots) may not monitor for unauthorized re-delegation events.

**Impact: 4** — Confidentiality: Low, Integrity: High (collateral and quote balance can be drained), Availability: Low.

---

### Likelihood Explanation

The attack requires that the victim has already registered a linked signer (a common pattern for API-key-based trading). The linked signer must be malicious or compromised. This is a realistic scenario: third-party trading bots, market-making services, or compromised API keys are common in DEX ecosystems. No admin or sequencer compromise is required — the attacker only needs to control the currently registered linked signer address.

**Likelihood: 2** — Requires a prior linked-signer grant, but the attack path is fully on-chain and permissionless once that condition is met.

---

### Recommendation

The `LinkSigner` fast-path handler should **not** accept a signature from the current linked signer. Change the `validateSignedTx` call for `LinkSigner` to pass `false` for `allowLinkedSigner`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only the subaccount owner may change the linked signer
);
```

This aligns the fast path with the slow path, which already enforces owner-only authorization via `validateSender`. [6](#0-5) 

---

### Proof of Concept

1. Victim (`0xVICTIM`) owns subaccount `S` and registers linked signer `0xLS` via a `LinkSigner` slow-mode transaction. `linkedSigners[S] = 0xLS`.
2. Attacker controls `0xLS` (e.g., a compromised API key or malicious trading bot).
3. Attacker constructs a `SignedLinkSigner` transaction: `{ sender: S, signer: bytes32(uint256(uint160(0xATTACKER))), nonce: currentNonce }`, signed by `0xLS`.
4. Attacker submits this to the sequencer via the fast path. `validateSignedTx(..., true)` accepts the `0xLS` signature because `getLinkedSigner(S) == 0xLS`.
5. `linkedSigners[S]` is now set to `0xATTACKER`.
6. Attacker signs a `WithdrawCollateral` transaction from subaccount `S` using `0xATTACKER`. The verifier accepts it as a valid linked-signer signature.
7. Victim's collateral is withdrawn to an attacker-controlled address. The victim's `linkedSigners` state is permanently corrupted until the owner signs a corrective `LinkSigner` transaction directly. [1](#0-0) [7](#0-6)
