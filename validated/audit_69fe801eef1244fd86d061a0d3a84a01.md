### Title
Deprecated `verify_transaction_inclusion` Remains Callable, Bypassing 64-Byte Merkle Proof Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof. However, the original `verify_transaction_inclusion` (v1) was only marked `#[deprecated]` — a Rust compiler lint with zero runtime effect — and remains a fully callable public NEAR contract method. Any unprivileged caller can invoke v1 directly, bypassing the coinbase proof check entirely. This is a direct structural analog to the stFLUO `withdraw`/`redeem` bypass: the secure path (v2) exists, but the insecure path (v1) was never disabled.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces the coinbase Merkle proof check before delegating to v1: [1](#0-0) 

The check at lines 358–365 verifies that the coinbase transaction's Merkle path resolves to the block's `merkle_root`. This is the mitigation for the [64-byte transaction forgery](https://www.bitmex.com/blog/64-Byte-Transactions): an internal Merkle node is exactly 64 bytes and can be misinterpreted as a valid transaction hash, so anchoring the proof to the coinbase (always at index 0) prevents this.

v1, however, performs no such check: [2](#0-1) 

The `#[deprecated]` attribute at line 283 is a Rust compiler warning only. It does not prevent the function from being called at runtime. The function is `pub`, carries `#[pause]` (which only blocks calls when the contract is administratively paused), and is fully exposed as a NEAR contract method. Any unprivileged NEAR account can call it directly.

The contract's own documentation acknowledges the danger: [3](#0-2) 

---

### Impact Explanation

A recipient contract (e.g., a bridge, a cross-chain DEX, or any NEAR contract that gates fund release on Bitcoin transaction inclusion) that calls `verify_transaction_inclusion` receives a `bool`. If an attacker can make that function return `true` for a fabricated `tx_id`, the recipient contract is deceived into treating a non-existent Bitcoin transaction as confirmed.

Concrete exploit path:
1. Attacker identifies a real Bitcoin block already accepted by the light client.
2. Attacker selects a 64-byte internal Merkle node from that block's transaction tree and treats it as a fake `tx_id`.
3. Attacker constructs a valid Merkle sibling path from that internal node up to the block's `merkle_root` — this path is mathematically valid because the node genuinely exists in the tree.
4. Attacker calls `verify_transaction_inclusion` with the fake `tx_id`, the real block hash, and the crafted proof.
5. `compute_root_from_merkle_proof` resolves correctly to `merkle_root`; the function returns `true`.
6. Any recipient contract consuming this result releases funds or executes a privileged action for a Bitcoin transaction that never existed. [4](#0-3) 

---

### Likelihood Explanation

- **Entry path is fully open**: no role, stake, or whitelist is required to call `verify_transaction_inclusion`. Any NEAR account can call it.
- **The forgery technique is publicly documented** and the contract's own comments link to the BitMEX writeup.
- **The fix (v2) already exists**, confirming the developers recognized the attack as realistic. The only missing step is disabling v1.
- Likelihood is **high** for any deployment where a downstream recipient contract has not independently migrated to v2.

---

### Recommendation

Disable `verify_transaction_inclusion` at the contract level by overriding it to revert unconditionally, mirroring the stFLUO remediation pattern:

```rust
/// Disabled. Use `verify_transaction_inclusion_v2` instead.
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] _args: ProofArgs) -> bool {
    env::panic_str("verify_transaction_inclusion is disabled; call verify_transaction_inclusion_v2");
}
```

This ensures no caller — direct or indirect — can reach the unprotected Merkle proof path.

### Proof of Concept

```
# Attacker account: attacker.near
# Target: a NEAR bridge contract that calls verify_transaction_inclusion

# Step 1: Pick a real block already in the light client, e.g. block B at height H.
# Step 2: Extract an internal 64-byte Merkle node N from block B's tx tree.
# Step 3: Compute the sibling path from N to merkle_root(B). This is valid by construction.
# Step 4: Encode ProofArgs { tx_id: N, tx_block_blockhash: B, tx_index: <node_index>,
#          merkle_proof: <sibling_path>, confirmations: 1 }
# Step 5: Call:
near call <light_client_account> verify_transaction_inclusion \
  --args <borsh-encoded ProofArgs above> \
  --accountId attacker.near
# Returns: true

# Step 6: The bridge contract, which calls verify_transaction_inclusion before releasing
# funds, now releases funds for a Bitcoin transaction that does not exist.
```

The call succeeds because `compute_root_from_merkle_proof(N, index, siblings) == merkle_root(B)` is true — N is a genuine internal node. v1 has no mechanism to distinguish an internal node from a leaf transaction hash. [5](#0-4)

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
