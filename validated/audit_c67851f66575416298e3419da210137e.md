### Title
Reduce-Only Order Cap Applied to Local Memory Copy, Original Uncapped Amount Used in Matching — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

In `OffchainExchange._validateOrder`, the reduce-only position cap is computed and applied only to a **local memory copy** of `order.amount`. The original `signedOrder.order.amount` is never modified. The actual matching logic then uses the original uncapped amount, allowing a reduce-only order to fill beyond the user's current position size — potentially flipping or increasing a position in violation of the reduce-only invariant.

---

### Finding Description

In `_validateOrder`, a local copy of the order is created and the reduce-only cap is applied to it:

```solidity
IEndpoint.Order memory order = signedOrder.order;   // local copy
order.amount -= filledAmount;

if (_isReduceOnly(order.appendix)) {
    int128 amount = callState.isPerp
        ? callState.perp.getBalance(callState.productId, order.sender).amount
        : callState.spot.getBalance(callState.productId, order.sender).amount;
    if ((order.amount > 0) == (amount > 0)) {
        order.amount = 0;
    } else if (order.amount > 0) {
        order.amount = MathHelper.min(order.amount, -amount);   // cap on local copy
    } else if (order.amount < 0) {
        order.amount = MathHelper.max(order.amount, -amount);   // cap on local copy
    }
}

return
    ...
    (order.amount != 0) &&   // validation uses local copy
    !_expired(order.expiration);
``` [1](#0-0) 

Because `order` is a local `memory` copy of `signedOrder.order`, the cap is **never written back** to `signedOrder.order.amount`. The function only returns a `bool`.

The actual fill computation in the matching logic uses `taker.order.amount` and `maker.order.amount` directly from the original `SignedOrder` structs:

```solidity
ordersInfo.taker.amountDelta = MathHelper.min(
    taker.order.amount,
    -maker.order.amount
);
``` [2](#0-1) 

Since `_validateOrder` is `internal view` and takes `signedOrder` by memory value, it cannot modify the caller's struct. The matching proceeds with the original, uncapped `order.amount`.

This is structurally identical to the FertilizerFacet bug: a cap is computed into a shadow variable (`order` / `_amount`) but the original uncapped variable (`signedOrder.order.amount` / `amount`) is used in the downstream operation.

---

### Impact Explanation

A reduce-only order is intended to only decrease an existing position. Because the cap is not propagated to the matching logic, a reduce-only order with `|amount| > |position|` will fill the full original amount. This allows:

- **Position flip**: A user with a long of 10 units submits a reduce-only sell for 100 units. Validation passes (capped local copy = 10 ≠ 0). Matching fills 100 units. User ends up short 90 units — a net position increase in the opposite direction.
- **Margin bypass**: Reduce-only orders may be processed under different margin assumptions. Filling beyond position size corrupts the subaccount's balance accounting.
- **Broken invariant**: The on-chain reduce-only constraint — the protocol's last line of defense — is silently unenforced.

**Impact: 3/5** — Corrupts position accounting and breaks a core order-type invariant; bounded by available counterparty liquidity.

---

### Likelihood Explanation

Any trader who submits a signed reduce-only order with `|amount| > |current position|` triggers this path. The sequencer processes `MatchOrders` transactions on-chain; the on-chain code is the final enforcement boundary. A buggy or malicious sequencer that does not pre-filter reduce-only amounts off-chain will cause this to manifest. The entry path is fully reachable through the standard `processTransaction` → `matchOrders` flow.

**Likelihood: 3/5** — Requires the sequencer to submit a match where the reduce-only amount exceeds the position, which can occur through sequencer bugs or adversarial sequencer behavior.

---

### Recommendation

The reduce-only cap must be applied to `signedOrder.order.amount` before it reaches the matching logic, not only to a local validation copy. One approach: refactor `_validateOrder` to return the effective (capped) amount alongside the validity bool, and use that capped amount in `amountDelta` computation. Alternatively, enforce the reduce-only cap directly inside the fill-amount calculation:

```solidity
// Before computing amountDelta, enforce reduce-only cap on taker/maker
if (_isReduceOnly(taker.order.appendix)) {
    int128 pos = ...; // current position
    taker.order.amount = /* clamp to position */;
}
```

---

### Proof of Concept

1. User holds a perp long position of `+10` units.
2. User signs a reduce-only sell order for `-100` units.
3. Sequencer submits `MatchOrders` pairing this order against a maker with `+100` units available.
4. `_validateOrder` is called: local copy is capped to `-10`, returns `true` (non-zero).
5. Matching computes `amountDelta = min(-100, -100) = -100` using the original uncapped amount.
6. User's position is updated by `-100`, resulting in a `-90` short position.
7. The reduce-only invariant is violated: the user's position was increased (in magnitude, opposite direction) rather than reduced. [3](#0-2) [4](#0-3)
