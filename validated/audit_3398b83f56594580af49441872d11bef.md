### Title
Sanctioned User Check Missing at Slow Mode Execution Time Allows Sanctioned Users to Withdraw Collateral — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `requireUnsanctioned` check is enforced only when a slow mode transaction is **submitted** (`submitSlowModeTransactionImpl`), but is absent when the transaction is **executed** (`processSlowModeTransactionImpl`). A user who is sanctioned after submitting a slow mode `WithdrawCollateral` transaction can still have that transaction executed after the 3-day delay, bypassing the sanctioned control entirely.

---

### Finding Description

The Nado protocol uses a slow mode queue as a censorship-resistant fallback path. Any user can submit a slow mode transaction via `submitSlowModeTransaction`, which delegates to `submitSlowModeTransactionImpl`. At submission time, the function enforces a sanctioned check: [1](#0-0) 

```solidity
IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
requireUnsanctioned(sender);
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: transaction
});
```

However, when the queued transaction is later executed — either by the sequencer via `ExecuteSlowMode` or by anyone calling `executeSlowModeTransaction()` after the delay — the execution path goes through `processSlowModeTransactionImpl`, which contains **no** `requireUnsanctioned` check: [2](#0-1) 

The `WithdrawCollateral` branch in `processSlowModeTransactionImpl` directly calls `clearinghouse.withdrawCollateral` without any sanctioned re-validation: [3](#0-2) 

The public `executeSlowModeTransaction()` entry point on `Endpoint.sol` is callable by any address after the delay, with no access control or sanctioned check: [4](#0-3) 

---

### Impact Explanation

A sanctioned user can withdraw collateral from the protocol after being sanctioned. The sanctioned list is a compliance/risk control mechanism. Bypassing it allows a sanctioned address to extract funds that the protocol intended to freeze, directly corrupting the expected asset-custody invariant for sanctioned accounts.

---

### Likelihood Explanation

The attack requires only that the user submits a `WithdrawCollateral` slow mode transaction before being sanctioned (or anticipating sanctioning). After the 3-day `SLOW_MODE_TX_DELAY`, the user — or any third party — can call `executeSlowModeTransaction()` permissionlessly. No privileged access, leaked keys, or social engineering is required. The window between sanctioning and the 3-day delay is a realistic scenario in any compliance-driven sanctioning event.

---

### Recommendation

Add `requireUnsanctioned(sender)` at the start of `processSlowModeTransactionImpl`, using the `sender` argument already passed to the function, so that the sanctioned status is re-evaluated at execution time rather than only at submission time:

```solidity
function processSlowModeTransactionImpl(
    address sender,
    bytes calldata transaction
) public {
    requireUnsanctioned(sender); // <-- add this
    ...
}
```

This mirrors the pattern already used in `submitSlowModeTransactionImpl` and closes the gap between submission-time and execution-time enforcement.

---

### Proof of Concept

1. User `A` calls `submitSlowModeTransaction` with a `WithdrawCollateral` transaction. The `requireUnsanctioned` check passes; the transaction is queued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY`. [1](#0-0) 

2. Protocol owner sanctions user `A` (adds them to the sanctioned list).

3. After 3 days, user `A` (or any address) calls `executeSlowModeTransaction()` on `Endpoint.sol`. [4](#0-3) 

4. `_executeSlowModeTransaction` → `processSlowModeTransaction` → `processSlowModeTransactionImpl` executes the `WithdrawCollateral` branch, calling `clearinghouse.withdrawCollateral` with no sanctioned check. [3](#0-2) 

5. User `A` successfully withdraws collateral despite being sanctioned, bypassing the protocol's compliance control.
