### Title
`sha256tree` Operator (Opcode 63) Has No Softfork Extension Path — Silent Nil Return Inside Any Softfork Guard - (File: src/chia_dialect.rs)

### Summary

The `sha256tree` operator (opcode 63) is gated by the `ENABLE_SHA256_TREE` flag in `ChiaDialect::op()`, but no `OperatorSet` variant exists that injects this flag when executing inside a softfork guard. Unlike `keccak256` (opcode 62), which has both a softfork extension path (`OperatorSet::Keccak` → `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`) and a hard-fork flag, `sha256tree` has only a hard-fork flag and no softfork extension path. Any attacker-controlled CLVM program that invokes opcode 63 inside a softfork guard will silently receive `nil` instead of the actual tree hash in consensus mode, potentially bypassing hash-based puzzle verification.

### Finding Description

The `OperatorSet` enum in `src/dialect.rs` defines only three variants:

```rust
pub enum OperatorSet {
    Default,
    Bls,
    Keccak,
}
``` [1](#0-0) 

The `softfork_extension()` dispatch in `ChiaDialect` maps:
- Extension 0 → `OperatorSet::Bls`
- Extension 1 → `OperatorSet::Keccak`
- Everything else → `OperatorSet::Default` (which causes `UnknownSoftforkExtension` in mempool mode) [2](#0-1) 

In `ChiaDialect::op()`, the extension-to-flags mapping is:

```rust
let flags = self.flags
    | match extension {
        OperatorSet::Default => ClvmFlags::empty(),
        OperatorSet::Bls    => ClvmFlags::empty(),
        OperatorSet::Keccak => ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD,
    };
``` [3](#0-2) 

The operator dispatch for opcode 63 is:

```rust
63 if flags.contains(ClvmFlags::ENABLE_SHA256_TREE) => op_sha256_tree,
``` [4](#0-3) 

Because no `OperatorSet` variant ever injects `ENABLE_SHA256_TREE` into `flags`, opcode 63 can only be dispatched when `self.flags` (the base flags passed to `ChiaDialect::new()`) already contains `ENABLE_SHA256_TREE`. Inside a softfork guard, `self.flags` is the caller-supplied base flags — the extension arm never adds `ENABLE_SHA256_TREE`. Therefore, `sha256tree` is permanently inaccessible inside any softfork guard, regardless of which extension number is used.

Compare this to `keccak256` (opcode 62), which has a complete path: extension 1 → `OperatorSet::Keccak` → `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` → opcode 62 dispatched. No equivalent path exists for opcode 63. [5](#0-4) 

When opcode 63 is invoked inside a softfork guard without `ENABLE_SHA256_TREE` in the base flags, it falls to `unknown_operator`. In consensus mode (no `NO_UNKNOWN_OPS`), `unknown_operator` calls `op_unknown`, which returns `nil` with a cost — a silent wrong result. In mempool mode, it returns `Unimplemented`. [6](#0-5) 

### Impact Explanation

An attacker-controlled CLVM program can invoke `sha256tree` (opcode 63) inside a softfork guard. In consensus mode, the operator silently returns `nil` instead of the actual SHA-256 tree hash. If the puzzle logic uses this result in a comparison (e.g., `(= (sha256tree puzzle) expected-hash)`), and the attacker supplies `expected-hash = ()` (nil), the comparison succeeds even though the tree hash was never computed. This allows the attacker to bypass hash-based puzzle verification — a spend that should be rejected is accepted. The corrupted result is the `NodePtr` returned by opcode 63: it is `nil` instead of a 32-byte hash atom.

### Likelihood Explanation

Medium. The attacker entry path is direct: submit any CLVM program that calls opcode 63 inside a softfork guard. No special privileges are required. The condition is reachable whenever `ENABLE_SHA256_TREE` is not set in the base flags (i.e., before the hard-fork activates). The impact is conditional on the puzzle logic being exploitable via a nil tree hash, which is a realistic pattern for puzzles that use sha256tree for identity or authorization checks.

### Recommendation

Add a `Sha256Tree` variant to `OperatorSet` in `src/dialect.rs`, map a new softfork extension number (e.g., extension 2) to `OperatorSet::Sha256Tree` in `softfork_extension()`, and add a corresponding arm in the `ChiaDialect::op()` extension-to-flags match that injects `ENABLE_SHA256_TREE`. This mirrors the complete path already implemented for `keccak256` via `OperatorSet::Keccak`.

### Proof of Concept

```

; sha256tree returns nil (unknown op). Attacker passes nil as expected-hash.
; The (= nil nil) comparison succeeds, softfork guard passes, spend is accepted.

(softfork
  (q . COST)
  (q . 0)                          ; extension 0 (BLS) — no ENABLE_SHA256_TREE injected
  (q a
    (i (= (sha256tree (q . (some . tree)))  ; opcode 63 → nil (unknown op in consensus)
           (q . ()))                         ; attacker-supplied expected-hash = nil
       (q . 0)
       (q x))
    (q . ()))
  (q . ()))
```

Without `ENABLE_SHA256_TREE` in the base flags, opcode 63 falls to `unknown_operator` → `op_unknown` → returns `nil`. The comparison `(= nil nil)` is true, the softfork guard exits successfully, and the spend is accepted despite the tree hash never being verified. [3](#0-2) [7](#0-6) [2](#0-1)

    fn softfork_extension(&self, ext: u32) -> OperatorSet {
        match ext {
            // Extension 0 is for the BLS operators, and is still valid.
            // However, the extension doesn't add any addition opcodes,
            // because the BLS operators were hardforked into the main set.
            0 => OperatorSet::Bls,

            // Extension 1 is for the keccak256 operator.
            1 => OperatorSet::Keccak,

            // Extensions 2 and beyond are considered invalid by the mempool.
            // However, all future extensions are valid in consensus mode and reserved for future softforks.
            _ => OperatorSet::Default,
        }
