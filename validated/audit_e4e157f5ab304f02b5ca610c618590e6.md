### Title
Silent `uint256`→`uint128` Truncation in `creditDeposit()` Causes Partial Deposit and Funds Stuck in Contract — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary
`DirectDepositV1.creditDeposit()` reads the full `uint256` token balance but silently truncates it to `uint128` when calling `endpoint.depositCollateralWithReferral()`. If the balance exceeds `type(uint128).max`, only the lower 128 bits are deposited to the subaccount; the remainder stays locked in the `DirectDepositV1` contract. Recovery requires the owner to call `withdraw()`, introducing the same centralization/drain risk described in the reference report.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`:

```solidity
// core/contracts/DirectDepositV1.sol, lines 90–98
uint256 balance = token.balanceOf(address(this));
if (balance != 0) {
    token.approve(address(endpoint), balance);   // approves full uint256
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),                        // ← silent truncation
        "-1"
    );
}
``` [1](#0-0) 

Solidity `^0.8.0` reverts on implicit arithmetic overflow but **does not revert on explicit casts**. The expression `uint128(balance)` silently discards the upper 128 bits when `balance > 2^128 - 1`. The `approve` call on line 92 uses the full `uint256` balance, so the endpoint has approval for the entire amount, but only the truncated value is actually deposited. The surplus tokens remain in the `DirectDepositV1` contract with no automatic recovery path.

The only recovery mechanism is the `onlyOwner` function:

```solidity
// core/contracts/DirectDepositV1.sol, lines 103–106
function withdraw(IIERC20Base token) external onlyOwner {
    uint256 balance = token.balanceOf(address(this));
    safeTransfer(token, msg.sender, balance);
}
``` [2](#0-1) 

This mirrors the reference report's pattern exactly: incorrect amount passed to the deposit function → less collateral credited to the subaccount → remainder stuck in the contract → privileged admin required to drain it.

`creditDeposit()` has no access control — it is callable by any external address:

```solidity
// core/contracts/DirectDepositV1.sol, line 83
function creditDeposit() external {
``` [3](#0-2)

### Impact Explanation

If the truncation condition is met, the subaccount receives only `uint128(balance)` worth of collateral credit instead of the full deposited amount. The difference — potentially a large token quantity — is silently retained in the `DirectDepositV1` contract. A malicious or compromised owner can then call `withdraw()` to extract those tokens to an arbitrary address, bypassing the normal protocol accounting entirely. The subaccount's on-chain balance in the `SpotEngine` will permanently undercount the actual tokens deposited. [4](#0-3) 

---

### Likelihood Explanation

`uint128` max is `2^128 − 1 ≈ 3.4 × 10^38`. For a standard 18-decimal token this requires `~3.4 × 10^20` whole tokens in a single DDA contract — an astronomically large amount for most assets today. Likelihood is therefore **low** under current token economics. However, the structural risk is real and non-zero: the protocol supports multiple spot products via `spotEngine.getProductIds()`, and future token listings with non-standard decimal configurations (e.g., 0–2 decimals, or rebasing tokens with inflating raw balances) could bring the threshold within reach. The absence of any bounds check or revert means the failure mode is completely silent. [5](#0-4) 

---

### Recommendation

Add an explicit overflow guard before the cast, consistent with the rest of the protocol's pattern (e.g., `require(balance <= INT128_MAX, ERR_CONVERSION_OVERFLOW)` used throughout `Clearinghouse.sol`):

```solidity
require(balance <= type(uint128).max, "balance overflow");
endpoint.depositCollateralWithReferral(
    subaccount,
    productId,
    uint128(balance),
    "-1"
);
```

This causes the transaction to revert rather than silently deposit a truncated amount, preventing funds from becoming stranded. Additionally, consider adding access control to `creditDeposit()` (e.g., restrict to the owner or the `ContractOwner` contract) to prevent griefing via repeated partial-deposit calls. [1](#0-0) 

---

### Proof of Concept

1. A `DirectDepositV1` contract is deployed for `subaccount = S` via `ContractOwner.createDirectDepositV1(S)`.
2. An attacker (or organic flow) causes the DDA to hold a token balance `B` where `B > type(uint128).max`. For a 0-decimal token, this requires `B > ~3.4 × 10^38` raw units; for a 6-decimal token, `B > ~3.4 × 10^32` whole units.
3. Any address calls `creditDeposit()`.
4. `token.approve(endpoint, B)` succeeds for the full amount.
5. `endpoint.depositCollateralWithReferral(S, productId, uint128(B), "-1")` is called with `uint128(B) = B mod 2^128`, depositing only the truncated value.
6. The remaining `B - uint128(B)` tokens stay in the DDA contract; the subaccount `S` is credited less than the actual deposit.
7. The owner calls `withdraw(token)` and receives the stranded tokens to `msg.sender`, with no protocol-level accounting for the discrepancy. [6](#0-5) [7](#0-6)
