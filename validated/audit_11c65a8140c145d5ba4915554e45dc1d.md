### Title
Coinbase Proof Bypass in `verify_transaction_inclusion_v2` Nullifies 64-Byte Forgery Mitigation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a separate coinbase proof. However, because the function never enforces that `coinbase_tx_id ≠ tx_id` or that `tx_index ≠ 0`, an unprivileged caller can supply identical values for both the coinbase and transaction proof fields, collapsing the two-party check into one. The coinbase validation is trivially bypassed, making `verify_transaction_inclusion_v2` functionally equivalent to the deprecated `verify_transaction_inclusion` it was meant to supersede.

---

### Finding Description

`verify_transaction_inclusion_v2` performs two sequential Merkle root checks:

1. **Coinbase check** — proves `coinbase_tx_id` at index `0` against `coinbase_merkle_proof` equals `header.block_header.merkle_root`.
2. **Transaction check** — proves `tx_id` at `tx_index` against `merkle_proof` equals the same `merkle_root` (via the deprecated `verify_transaction_inclusion`). [1](#0-0) 

The only structural constraint between the two proof sets is that they share the same length: [2](#0-1) 

There is **no check** that `coinbase_tx_id ≠ tx_id`, that `coinbase_merkle_proof ≠ merkle_proof`, or that `tx_index ≠ 0`. A caller can therefore supply:

```
coinbase_tx_id      = tx_id
coinbase_merkle_proof = merkle_proof
tx_index            = 0
```

Under these inputs, both calls to `compute_root_from_merkle_proof` evaluate the identical expression:

```
compute_root_from_merkle_proof(tx_id, 0, merkle_proof) == merkle_root
``` [3](#0-2) 

The coinbase guard is satisfied by the same proof that satisfies the transaction guard, so the additional check provides zero security.

---

### Impact Explanation

The 64-byte transaction Merkle proof forgery works as follows. Given a block with real transactions `[tx0, tx1, tx2, tx3]`, the Merkle tree has an internal node `N = double_sha256(tx0 || tx1)`. An attacker constructs a fake 64-byte "transaction" whose body is `tx0 || tx1`; its transaction ID is `N`. A valid Merkle proof for `N` at leaf-position `0` is `[double_sha256(tx2 || tx3)]`, and `compute_root_from_merkle_proof(N, 0, [double_sha256(tx2||tx3)]) == merkle_root` holds exactly. [3](#0-2) 

With the bypass, the attacker sets `coinbase_tx_id = N`, `coinbase_merkle_proof = [double_sha256(tx2||tx3)]`, `tx_id = N`, `tx_index = 0`, `merkle_proof = [double_sha256(tx2||tx3)]`. Both checks pass and the function returns `true` for a transaction that does not exist. Any downstream NEAR contract that calls `verify_transaction_inclusion_v2` to gate a cross-chain action (e.g., token unlock, bridge settlement) can be deceived into accepting a fabricated transaction.

**Impact: Medium** — proof-verification forgery; downstream contracts consuming the `true` result are directly affected.

---

### Likelihood Explanation

All inputs required for the attack are derived from public Bitcoin blockchain data (real transaction hashes in any block). No privileged access, leaked keys, or social engineering is needed. The caller is an unprivileged NEAR account invoking a public, non-paused entry point. The only prerequisite is knowledge of the 64-byte vulnerability, which is publicly documented. [4](#0-3) 

**Likelihood: Medium** — requires awareness of the attack class but no special on-chain privileges.

---

### Recommendation

Add an explicit guard at the top of `verify_transaction_inclusion_v2` that rejects calls where the coinbase and transaction proofs are identical:

```rust
require!(
    args.coinbase_tx_id != args.tx_id || args.tx_index != 0,
    "tx_id must differ from coinbase_tx_id or tx_index must be non-zero"
);
```

A stricter fix enforces `args.coinbase_tx_id != args.tx_id` unconditionally, since the coinbase and the target transaction are always distinct in a valid Bitcoin block.

---

### Proof of Concept

Given a real Bitcoin block with transactions `[tx0, tx1, tx2, tx3]`:

1. Compute `N = double_sha256(tx0_bytes || tx1_bytes)` — this is an internal Merkle node.
2. Compute `sibling = double_sha256(tx2_bytes || tx3_bytes)`.
3. Verify: `double_sha256(N || sibling) == merkle_root` (standard Merkle arithmetic).
4. Call `verify_transaction_inclusion_v2` with:
   - `tx_id = N`
   - `tx_block_blockhash = <valid block hash stored in contract>`
   - `tx_index = 0`
   - `merkle_proof = [sibling]`
   - `coinbase_tx_id = N`  ← same as `tx_id`
   - `coinbase_merkle_proof = [sibling]`  ← same as `merkle_proof`
   - `confirmations = 1`
5. The coinbase check evaluates `compute_root_from_merkle_proof(N, 0, [sibling]) == merkle_root` → `true`.
6. The transaction check evaluates the identical expression → `true`.
7. The function returns `true` for the non-existent transaction `N`. [1](#0-0) [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
