Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address, but its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual `msg.sender` of the pool call) and instead validates the caller-supplied `owner` argument (the LP position recipient). Because `owner` is a free parameter, any unprivileged address can bypass the allowlist by supplying any allowlisted address as `owner`, depositing tokens into a pool they are explicitly barred from.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the LP position recipient to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes and forwards both values positionally to the extension: [2](#0-1) 

Inside the extension, the first positional parameter (`sender`) is unnamed and discarded; the allowlist check is performed only on `owner`: [3](#0-2) 

The NatSpec ("Gates `addLiquidity` by depositor address") and the setter name `setAllowedToDeposit(pool_, depositor, allowed)` make the intent unambiguous — the check should be on `sender` (the entity that calls the pool and funds the deposit), not `owner`: [4](#0-3) 

Because `owner` is freely chosen by the caller, any address can pass the guard by setting `owner` to any allowlisted address. The `removeLiquidity` path enforces `msg.sender == owner`, so the deposited tokens are locked under the allowlisted address's position and the attacker cannot directly recover them: [5](#0-4) 

## Impact Explanation

The deposit allowlist is the sole on-chain mechanism a pool admin has to restrict who may inject liquidity into a permissioned pool. With this bug the restriction is entirely ineffective. An unprivileged caller can deposit arbitrary token amounts into the pool, bypassing the intended access control. The pool admin's invariant — "only allowlisted addresses may add liquidity" — is broken for every pool that relies on this extension (e.g., regulatory-compliance pools, curated LP sets, or pools where unauthorized liquidity dilutes or distorts existing LP positions). This constitutes a broken core pool access-control functionality causing a broken admin-boundary invariant.

## Likelihood Explanation

No special role, privilege, or precondition is required. Any EOA or contract can call `pool.addLiquidity(allowlistedAddress, ...)` at any time. The only cost to the attacker is the tokens they supply via the callback, which they cannot recover without the allowlisted address's cooperation. The attack is trivially repeatable.

## Recommendation

Replace the discarded first parameter with `sender` and check it against the allowlist:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol

function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` and calls:
   ```solidity
   extension.setAllowedToDeposit(pool, alice, true);
   // bob is NOT allowlisted
   ```
2. Bob constructs a callback contract that funds the deposit and calls:
   ```solidity
   pool.addLiquidity(
       alice,        // owner — allowlisted, passes the broken check
       salt,
       deltas,
       callbackData, // Bob's contract funds the token transfer here
       extensionData
   );
   ```
3. Inside `beforeAddLiquidity`, `msg.sender` is the pool, the checked value is `owner = alice`:
   ```
   allowedDepositor[pool][alice] == true  →  guard passes
   ```
4. Bob's callback transfers tokens into the pool; Alice receives the LP shares.
5. Bob has deposited into a pool he is explicitly barred from, bypassing the allowlist entirely. [3](#0-2) [6](#0-5)
