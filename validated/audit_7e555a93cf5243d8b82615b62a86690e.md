Looking at the `DepositAllowlistExtension` hook plumbing, I need to trace how the `beforeAddLiquidity` guard is applied and whether the correct actor is checked.

**Key observation in `DepositAllowlistExtension.beforeAddLiquidity`:** [1](#0-0) 

The first parameter (`sender`) is silently discarded. The check is performed on `owner` (the position owner), not on `sender` (the actual caller of `addLiquidity`).

**The pool passes both parameters distinctly:** [2](#0-1) 

`msg.sender` is the depositing caller; `owner` is the position beneficiary — two different addresses.

**The admin setter names the parameter `depositor`:** [3](#0-2) 

The naming unambiguously signals the intent: restrict who *deposits*, i.e., the `sender`. But the hook checks `owner` instead.

**Compare with `SwapAllowlistExtension`, which correctly checks `sender`:** [4](#0-3) 

The swap guard checks `sender`; the deposit guard checks `owner`. This asymmetry is the bug.

---

### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension.beforeAddLiquidity` ignores the `sender` argument (the actual `msg.sender` of `addLiquidity`) and gates on `owner` (the position beneficiary). Any address can bypass the allowlist by supplying an allowlisted address as `owner` while acting as the real depositor.

### Finding Description
`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`, forwarding the actual caller as `sender` and the user-supplied position owner as `owner`. [5](#0-4) 

`DepositAllowlistExtension.beforeAddLiquidity` receives both but discards `sender` entirely:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [1](#0-0) 

Because the check is on `owner`, any unprivileged caller can pass the guard by setting `owner` to any allowlisted address. The caller pays the tokens (via the callback), the allowlisted address receives the position, and the guard never fires.

`removeLiquidity` enforces `msg.sender == owner`: [6](#0-5) 

So the allowlisted owner can later withdraw the deposited tokens, completing a griefing path where the unauthorized depositor loses funds and the allowlisted owner profits — all while the pool admin's access control is silently bypassed.

### Impact Explanation
The deposit allowlist is the pool admin's primary mechanism to restrict who may become an LP (e.g., KYC-gated pools, private pools, regulatory-compliant pools). Because the guard checks the wrong parameter, it provides zero protection against unauthorized depositors. Any address can inject liquidity into a restricted pool, violating the admin-enforced invariant. Additionally, an unauthorized depositor who is tricked or front-run into calling `addLiquidity` with an attacker-controlled `owner` loses their tokens permanently (the attacker-controlled owner withdraws them).

### Likelihood Explanation
Allowlisted owner addresses are discoverable on-chain from `AllowedToDepositSet` events. Any actor who knows one allowlisted address can immediately exploit this with a direct pool call, requiring no special privileges or complex setup.

### Recommendation
Change the `beforeAddLiquidity` implementation to check `sender` (the actual depositor) instead of `owner`, consistent with the admin setter's `depositor` parameter name and with how `SwapAllowlistExtension` handles its analogous check:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

### Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension` configured in `beforeAddLiquidity` order.
2. Admin calls `setAllowedToDeposit(pool, Alice, true)` — Alice is the only allowlisted depositor.
3. Bob (not allowlisted) calls `pool.addLiquidity(owner=Alice, salt, deltas, callbackData, extensionData)` directly.
4. `_beforeAddLiquidity(Bob, Alice, ...)` is dispatched to the extension.
5. Extension evaluates `allowedDepositor[pool][Alice]` → `true` → no revert.
6. Bob's callback is invoked; Bob's tokens are transferred to the pool under Alice's position key.
7. Alice calls `removeLiquidity` and withdraws Bob's tokens.
8. The deposit allowlist has been fully bypassed; Bob lost funds; the pool admin's access control is broken.
