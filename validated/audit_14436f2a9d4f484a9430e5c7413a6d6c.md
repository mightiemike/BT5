Looking at the codebase, I need to trace the reward calculation path and identify any hardcoded-value analog to the Real Wagmi bug.

### Title
`RewardCalculator::calculate_reward` Ignores Passed-In `protocol_version` and Uses Hardcoded `genesis_protocol_version` to Select `protocol_reward_rate`, Permanently Freezing Treasury Reward Share Across Protocol Upgrades — (`File: chain/epoch-manager/src/reward_calculator.rs`)

---

### Summary

`RewardCalculator::calculate_reward` accepts a `_protocol_version: ProtocolVersion` parameter (the current epoch's actual protocol version) but silently discards it — the leading underscore is intentional. Instead, the function uses `self.genesis_protocol_version` (frozen at node startup from genesis config) to decide the `protocol_reward_rate`. For any chain whose genesis protocol version equals `PROD_GENESIS_PROTOCOL_VERSION` (29), the treasury share is hardcoded to `Rational32::new_raw(1, 10)` regardless of what any protocol upgrade specifies. Meanwhile, `max_inflation_rate` — the other multiplicative factor in the same reward formula — is correctly sourced from the version-aware `epoch_config.max_inflation_rate`. This asymmetry means the treasury reward fraction is permanently frozen at genesis and is unresponsive to protocol upgrades, while the total reward pool it is applied to does change with upgrades.

---

`epoch_protocol_version` is derived from `epoch_info.protocol_version()` — the real, live protocol version of the epoch being finalized. `epoch_config.max_inflation_rate` is fetched via `self.get_epoch_config(epoch_protocol_version)`, making it fully version-aware.

**Inside `calculate_reward` — `chain/epoch-manager/src/reward_calculator.rs` (lines 56, 63–68):**

```rust
pub fn calculate_reward(
    &self,
    ...
    _protocol_version: ProtocolVersion,   // ← accepted, never read
    ...
) -> (HashMap<AccountId, Balance>, Balance) {
    let use_hardcoded_value =
        self.genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION;  // == 29
    let protocol_reward_rate = if use_hardcoded_value {
        Rational32::new_raw(1, 10)          // ← hardcoded 10 % forever
    } else {
        self.protocol_reward_rate           // ← frozen genesis config value
    };
```

`self.genesis_protocol_version` is set once at node startup from `config.protocol_version` (the genesis config field). It never changes. The passed-in `_protocol_version` — the actual epoch protocol version — is never consulted.

**Exact divergent values:**

| What the call site provides | What the function actually uses |
|---|---|
| `epoch_protocol_version` (live, changes with upgrades) | `self.genesis_protocol_version` (frozen at 29 for mainnet/testnet) |
| `epoch_config.max_inflation_rate` (version-aware) | `Rational32::new_raw(1, 10)` (hardcoded constant) |

The treasury reward is computed as:

```
epoch_protocol_treasury = epoch_total_reward × protocol_reward_rate
```

`epoch_total_reward` correctly reflects the version-aware `max_inflation_rate`, but `protocol_reward_rate` is permanently `1/10` for mainnet/testnet, making the treasury fraction unresponsive to any protocol upgrade.

---

### Impact Explanation

Every epoch finalization on mainnet and testnet calls `finalize_epoch`, which calls `calculate_reward` with the live `epoch_protocol_version`. Because that version is silently dropped, the treasury always receives exactly 10 % of epoch rewards — the hardcoded `Rational32::new_raw(1, 10)` — regardless of what any protocol upgrade specifies. Any protocol upgrade that intends to adjust the treasury share (e.g., reduce it from 10 % to 5 % as part of an economics rework) would be silently ignored: `max_inflation_rate` would change correctly (reducing total rewards), but the treasury fraction applied to those rewards would remain 10 %, causing the treasury to receive a disproportionate share relative to what the upgrade intended. Validators would correspondingly receive less than the upgraded protocol specifies. The minted token amounts recorded in `EpochInfo` would diverge from the protocol specification at every epoch boundary after such an upgrade.

Additionally, any non-mainnet/testnet chain that happens to set `genesis_protocol_version = 29` (equal to `PROD_GENESIS_PROTOCOL_VERSION`) will have its configured `protocol_reward_rate` silently overridden by the hardcoded `1/10`, regardless of what the operator configured.

---

### Likelihood Explanation

The code path executes on every epoch boundary — approximately every 12 hours on mainnet. The bug is latent today because no protocol upgrade has yet attempted to change `protocol_reward_rate` via `EpochConfig`. However, the asymmetry is structurally present: `max_inflation_rate` was successfully made version-aware (it lives in `EpochConfig` and is fetched via `get_epoch_config(epoch_protocol_version)`), but `protocol_reward_rate` was not. The next economics-related protocol upgrade that touches treasury share will silently fail to apply the new rate, with no error, no log, and no on-chain signal.

---

### Recommendation

Replace the genesis-version-based branch with a lookup from the version-aware `EpochConfig`, mirroring how `max_inflation_rate` is already handled:

1. Add `protocol_reward_rate: Rational32` to `EpochConfig` (alongside `max_inflation_rate`).
2. Pass it into `calculate_reward` as an explicit argument (or remove the `_protocol_version` dead parameter and derive it inside the function from the epoch config).
3. Remove the `self.genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION` branch; the mainnet epoch config files already encode the correct historical value.
4. Remove `genesis_protocol_version` from `RewardCalculator` if it has no other uses.

The call site in `finalize_epoch` already has `epoch_config` in scope and already passes `epoch_config.max_inflation_rate`; passing `epoch_config.protocol_reward_rate` is a one-line change there.
