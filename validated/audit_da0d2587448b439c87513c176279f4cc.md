### Title
Negative `positionPnl` Passed to `min()` in `getSettlementState` Allows Draining Spot Quote While Inflating `availableSettle` — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.getSettlementState` computes `canSettle = min(positionPnl, state.availableSettle)` with no lower-bound clamp at zero. When `positionPnl < 0` and `state.availableSettle > 0`, `canSettle` is negative. The subsequent arithmetic in `settlePnl` then: (1) inflates `state.availableSettle`, (2) artificially reduces the user's perp loss (`vQuoteBalance`), and (3) causes `Clearinghouse._settlePnl` to call `spotEngine.updateBalance` with a negative amount, draining the user's real spot-quote collateral.

---

### Finding Description

**`getSettlementState`** computes the amount to settle as:

```solidity
availableSettle = MathHelper.min(
    calculatePositionPnl(balance, productId),
    state.availableSettle
);
``` [1](#0-0) 

There is no `max(0, …)` clamp. If `positionPnl` is negative (e.g. `-100e18`) and `state.availableSettle` is positive (e.g. `50e18`), `min` returns `-100e18`.

**`settlePnl`** then applies this unclamped value:

```solidity
state.availableSettle -= canSettle;   // 50e18 - (-100e18) = 150e18  ← inflated
balance.vQuoteBalance -= canSettle;   // vQuote increases ← loss erased
totalSettled += canSettle;            // totalSettled = -100e18
``` [2](#0-1) 

**`Clearinghouse._settlePnl`** then applies the negative `totalSettled` to the spot engine:

```solidity
int128 amountSettled = perpEngine.settlePnl(subaccount, productIds);
_spotEngine().updateBalance(QUOTE_PRODUCT_ID, subaccount, amountSettled);
``` [3](#0-2) 

With `amountSettled = -100e18`, the user's real spot-quote balance is reduced by 100e18.

**Net state mutation for the scenario `positionPnl = -100e18`, `availableSettle = 50e18`:**

| Variable | Before | After | Delta |
|---|---|---|---|
| `state.availableSettle` | `50e18` | `150e18` | `+100e18` |
| `balance.vQuoteBalance` | `-100e18` | `0` | `+100e18` |
| Spot quote balance | `X` | `X - 100e18` | `-100e18` |

The user's perp loss is silently erased while their real collateral is drained, and the protocol's settlement pool is inflated by the same amount — a double accounting error.

---

### Impact Explanation

- **Solvency / accounting failure**: `state.availableSettle` is inflated, meaning future legitimate settlers can claim more than the protocol actually holds.
- **Value transfer**: real spot-quote collateral is removed from the user's account without any corresponding reduction in their perp liability (the vQuoteBalance was also increased, cancelling the loss).
- **Protocol undercollateralization**: the inflated `availableSettle` will allow over-settlement by other users, leaving the protocol short.

This matches the Critical scope: *solvency or accounting failure in settlement that transfers value incorrectly or leaves the protocol undercollateralized.*

---

### Likelihood Explanation

`Clearinghouse.settlePnl` is `onlyEndpoint`. [4](#0-3) 

The endpoint processes `SettlePnl` transactions submitted through the normal sequencer flow. There is no on-chain check that prevents the sequencer from issuing a `SettlePnl` for a subaccount with negative PnL, and no guard inside `settlePnl` or `getSettlementState` rejects or skips negative `canSettle` values. Any `SettlePnl` transaction targeting a subaccount with negative `positionPnl` and a product whose `state.availableSettle > 0` will trigger the bug.

---

### Recommendation

Clamp `canSettle` to zero before applying it:

```solidity
// In getSettlementState or at the call site in settlePnl:
availableSettle = MathHelper.max(
    0,
    MathHelper.min(
        calculatePositionPnl(balance, productId),
        state.availableSettle
    )
);
```

Alternatively, add an early-exit guard in `settlePnl`:

```solidity
if (canSettle <= 0) {
    productIds >>= 32;
    continue;
}
``` [5](#0-4)
