Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` as the `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. If the pool admin allowlists the router (required for router-mediated swaps to function), every user on the network can bypass the swap allowlist by calling the router. There is no configuration that simultaneously enables router-mediated swaps and enforces per-user allowlisting.

## Finding Description

**Root cause:** `MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value received — the router address, not the original user: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly: [4](#0-3) 

The pool's `msg.sender` is the router. The extension evaluates `allowedSwapper[pool][router]`, never seeing the originating user's address. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension`, sets `allowedSwapper[pool][alice] = true`.
2. Admin also sets `allowedSwapper[pool][router] = true` so alice can use the router.
3. Bob (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient=bob, ...)` — pool sees `msg.sender = router`.
5. Pool calls `_beforeSwap(sender=router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds for bob.

**Existing guards are insufficient:** The `onlyPoolAdmin` guard on `setAllowedToSwap` only restricts who can configure the allowlist, not who can exploit the mismatch. There is no mechanism in the extension or the router to propagate the original `msg.sender` to the extension check.

## Impact Explanation

A pool deployer who configures `SwapAllowlistExtension` to restrict swaps to KYC'd or permissioned addresses cannot enforce that restriction when `MetricOmmSimpleRouter` is used. Any unprivileged user can execute swaps against the restricted pool by calling the public router. LPs who deposited under the assumption that only allowlisted counterparties could trade against them are exposed to unrestricted swap flow, defeating the access-control invariant the extension was deployed to enforce. This constitutes broken core pool functionality with direct fund-impact consequences.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool using `SwapAllowlistExtension` that also wants to support router-mediated swaps for allowlisted users must allowlist the router — this is the expected operational configuration. The bypass requires no special privileges, no flash loans, and no unusual token behavior. Any EOA can call the router directly with no preconditions beyond having tokens to swap.

## Recommendation

The extension must check the **original initiating user**, not the immediate caller of `pool.swap()`. The cleanest fix: have the router encode the original `msg.sender` into `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and verify it. The `onlyPool` guard (via `msg.sender` being the pool in `beforeSwap`) ensures only the pool — not an attacker — can invoke the extension, so the encoded value from `extensionData` can be trusted if the router is the only allowlisted intermediary. Alternatively, document that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` unless the router is allowlisted (which defeats the purpose), and provide a router variant that forwards the original `msg.sender` as a verified parameter.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - allowedSwapper[pool][alice] = true   (alice is the only allowed swapper)
  - allowedSwapper[pool][router] = true  (router allowlisted so alice can use it)
  - Add liquidity to the pool

Attack:
  - bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(recipient=bob, ...) — msg.sender to pool = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  - Swap executes successfully for bob despite bob not being allowlisted

Result:
  - bob swaps against a pool intended to be restricted to alice only
  - The swap allowlist guard is completely bypassed
```
