Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks caller-controlled `owner` instead of `sender`, allowing full allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and gates access on `owner`, which is a free caller-supplied argument. Because any caller can set `owner` to any allowlisted address, the deposit allowlist is fully bypassed by any unprivileged address. The analogous `SwapAllowlistExtension.beforeSwap` correctly checks `sender`, confirming the asymmetry is a bug.

## Finding Description

`MetricOmmPool.addLiquidity` passes the real caller as the first argument to the hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both `sender` and `owner` to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` names the first argument `address` (unnamed/discarded) and checks only `owner`: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` and discards `recipient`: [4](#0-3) 

Exploit path:
1. Pool `P` is deployed with `DepositAllowlistExtension`; only address `A` is allowlisted via `setAllowedToDeposit(P, A, true)`.
2. Attacker `B` (not allowlisted) calls `pool.addLiquidity(owner = A, salt, deltas, callbackData, extensionData)`.
3. The extension evaluates `allowedDepositor[P][A]` → `true` → no revert.
4. `LiquidityLib.addLiquidity` mints the position to `A` and calls back on `B` (the actual `msg.sender`) to collect tokens.
5. `B` provides tokens; the position is created for `A` with `B`'s capital.
6. The deposit allowlist has been fully bypassed.

No existing guard prevents this: `removeLiquidity` does enforce `msg.sender == owner` at line 206, but `addLiquidity` has no such check, leaving the allowlist as the sole access control — which is broken. [5](#0-4) 

## Impact Explanation

This is an admin-boundary break: the pool admin's deposit allowlist — the primary mechanism to restrict who may add liquidity to a private or permissioned pool — is fully bypassed by any unprivileged address. Consequences include unauthorized alteration of the pool's liquidity distribution across bins (affecting swap routing and pricing), and creation of positions for allowlisted addresses without their knowledge or consent (griefing). The attacker's tokens enter the pool as LP principal under an allowlisted address's ownership, directly violating the access control invariant the extension is designed to enforce.

## Likelihood Explanation

Exploitation requires no special role, no privileged key, and no complex setup. Any EOA or contract can call `addLiquidity` directly on the pool. Allowlisted addresses are discoverable on-chain via `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. The attack is repeatable at any time a pool is deployed with `DepositAllowlistExtension` configured and at least one address allowlisted.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual caller) instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
// Before (buggy):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// After (fixed):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
```
