### Title
Silent Truncating Downcast of `ShardId` (u64) to `u32` in `ShardUId::new` Corrupts Trie DB Key Domain — (`File: core/primitives/src/shard_layout/mod.rs`, `core/primitives-core/src/types.rs`)

---

### Summary

`ShardId` is a `u64`-backed newtype. `ShardUId` stores `shard_id: u32`. The conversion `ShardId → u32` inside `ShardUId::new()` is performed via `impl Into<u32> for ShardId` which executes `self.0 as u32` — a silent, wrapping truncation with no overflow check. `ShardUId` is the 8-byte prefix for every trie-state DB key. Any `ShardId` value whose lower 32 bits collide with another shard's lower 32 bits produces an identical `ShardUId`, silently aliasing two distinct shards' entire trie key spaces in the database.

---

### Finding Description

`ShardId` is defined as a `u64` newtype: [1](#0-0) 

The `Into<u32>` implementation performs an unchecked cast: [2](#0-1) 

`ShardUId::new` calls this conversion unconditionally: [3](#0-2) 

`ShardUId` stores the result in a `u32` field: [4](#0-3) 

This `ShardUId` is then serialized as an 8-byte DB key prefix (4 bytes version + 4 bytes shard_id) used for all trie state: [5](#0-4) 

The codebase itself acknowledges the structural mismatch with a `TODO`: [6](#0-5) 

`ShardLayoutV2` and `ShardLayoutV3` assign new shard IDs as `max_shard_id + 1` and `max_shard_id + 2` on each resharding: [7](#0-6) [8](#0-7) 

Additionally, `ShardLayoutV2::new()` and `ShardLayoutV3::new()` accept arbitrary `Vec<ShardId>` from genesis config or protocol upgrade payloads, meaning a shard ID exceeding `u32::MAX` can be introduced at any protocol upgrade boundary without any validation gate.

---

### Impact Explanation

`ShardUId` is the namespace key for all trie state in the node database. If two distinct `ShardId` values share the same lower 32 bits (e.g., `ShardId(X)` and `ShardId(X + 2^32)`), `ShardUId::new()` produces identical byte representations for both. This causes:

1. **DB key collision**: reads and writes for shard `X + 2^32` silently operate on shard `X`'s trie data.
2. **State corruption**: account balances, contract storage, and receipts from two distinct shards become indistinguishable at the storage layer.
3. **Consensus failure**: nodes computing state roots over corrupted trie data will diverge from nodes that do not have the collision, breaking consensus.

The `get_block_shard_uid` function embeds this corrupted `ShardUId` into block-level DB keys: [9](#0-8) 

---

### Likelihood Explanation

The trigger requires a `ShardId` value exceeding `u32::MAX` (4,294,967,295). Under the current `derive()` logic, starting from shard IDs in the single digits, this requires approximately 2 billion resharding events — not reachable organically. However:

- `ShardLayoutV2::new()` and `ShardLayoutV3::new()` accept arbitrary `shard_ids: Vec<ShardId>` with no upper-bound validation. A genesis config or a protocol upgrade (a public upgrade-boundary interaction) that specifies a large shard ID triggers the truncation immediately.
- The `Into<u16>` conversion (`self.0 as u16`) has an even smaller overflow threshold (65,535), reachable far sooner. [10](#0-9) 

---

### Recommendation

1. Replace `impl Into<u32> for ShardId` and `impl Into<u16> for ShardId` with checked conversions that panic or return `Result` on overflow:
   ```rust
   impl TryFrom<ShardId> for u32 {
       type Error = TryFromIntError;
       fn try_from(id: ShardId) -> Result<Self, Self::Error> {
           u32::try_from(id.0)
       }
   }
   ```
2. Add a validation gate in `ShardLayoutV2::new()` and `ShardLayoutV3::new()` asserting all shard IDs fit in `u32`.
3. Either widen `ShardUId.shard_id` to `u64` (requiring a DB migration) or document and enforce the `u32` upper bound at every shard-layout construction site.

---

### Proof of Concept

```rust
use near_primitives_core::types::ShardId;
use near_primitives::shard_layout::{ShardUId, ShardVersion};

let shard_a = ShardId::new(1u64);
let shard_b = ShardId::new((1u64) + (u32::MAX as u64) + 1); // = 2^32 + 1

let version: ShardVersion = 3;
let uid_a = ShardUId::new(version, shard_a);
let uid_b = ShardUId::new(version, shard_b);

// Both produce shard_id = 1u32 after truncation
assert_eq!(uid_a.to_bytes(), uid_b.to_bytes()); // PASSES — identical DB key prefix
// All trie reads/writes for shard_b now silently alias shard_a's state
```
