### Title
Zero-Confirmation Bypass in `verify_transaction_inclusion` Allows Acceptance of Unconfirmed Tip Transactions ŌĆö (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2` which delegates to it) accepts `confirmations=0` from any unprivileged NEAR caller. When the target block is the current mainchain tip, the confirmation arithmetic evaluates to `1 >= 0`, which always passes. The function then returns `true` for a valid merkle proof, certifying a transaction that has zero burial depth and is trivially reorgable.

---

### Finding Description

`ProofArgs.confirmations` is a caller-supplied `u64` with no enforced minimum. [1](#0-0) 

Inside `verify_transaction_inclusion`, the first guard only checks an upper bound:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [2](#0-1) 

The second guard computes available confirmations as `(tip_height - target_height) + 1`:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [3](#0-2) 

When `target_block_height == heaviest_block_header.block_height` (the tip):
- `saturating_sub(target_block_height)` ŌåÆ `0`
- `0 + 1 = 1`
- `1 >= 0` ŌåÆ **passes unconditionally**

The function then proceeds to compute and return the merkle proof result: [4](#0-3) 

There is no lower-bound guard anywhere ŌĆö neither `confirmations >= 1` nor any equivalent check. The function has only `#[pause]` and no `#[trusted_relayer]` or `#[private]` restriction, so any NEAR account can call it with arbitrary `confirmations`. [5](#0-4) 

`verify_transaction_inclusion_v2` passes `confirmations` through unchanged via `From<ProofArgsV2> for ProofArgs`, so it is equally affected: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` (or v2) and either:
- allows the end-user to supply `confirmations`, or
- hard-codes `confirmations=0` for "immediate" settlement

will receive `true` for a transaction that exists only in the current unconfirmed tip. That block can be reorged out in the very next Bitcoin block. A recipient contract that releases funds or transitions state on this `true` result is vulnerable to a classic double-spend: the attacker broadcasts a conflicting Bitcoin transaction, the original tip is reorged, and the NEAR-side settlement has already executed.

---

### Likelihood Explanation

- The call requires no privilege ŌĆö any NEAR account can invoke `verify_transaction_inclusion` directly.
- `confirmations=0` is a natural default for callers who do not understand the confirmation semantics.
- The existing test suite already exercises `confirmations=0` and expects it to succeed: [8](#0-7) 
  This confirms the behavior is not accidental ŌĆö it is the current contract design, and callers relying on it will be exposed.

---

### Recommendation

Enforce a minimum of 1 confirmation at the contract level. Add the following check immediately after the `gc_threshold` guard in `verify_transaction_inclusion`:

```rust
require!(
    args.confirmations >= 1,
    "At least 1 confirmation is required"
);
```

This must be applied in `verify_transaction_inclusion` (which `verify_transaction_inclusion_v2` delegates to), so a single fix covers both entry points. Callers that currently pass `confirmations=0` must be updated to pass at least `1`.

---

//    - tx_block_blockhash = that block's hash  (== mainchain tip)
//    - confirmations = 0
//    - valid merkle proof for a tx in that block

    .view(contract.id(), "verify_transaction_inclusion")
    .args_borsh(ProofArgs {

        tx_block_blockhash: tip_block_hash,   // current tip
        tx_index: 0,
        merkle_proof: vec![sibling_hash],
        confirmations: 0,                      // zero confirmations
    })
    .await?
    .json()?;

assert!(result);  // succeeds ŌĆö tip transaction accepted with no burial
```
