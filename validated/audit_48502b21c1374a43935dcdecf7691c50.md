The key question is whether `\xFE` is handled identically by the Python and Rust deserializers. I now have all the code needed to answer this definitively.

**Tracing Python `_atom_from_stream(f, 0xFE, new_atom_f)`:**

`0xFE = 11111110`. The function enters the bit-counting loop:
- 7 iterations strip all 7 leading set bits → `bit_count = 7`, `b = 0x00`
- `size_blob = bytes([0x00])`
- `bit_count > 1` → reads 6 more bytes → 7-byte size prefix total
- Computes `size = int.from_bytes(7_bytes, 'big')`
- If `size < 0x400000000`, reads `size` atom bytes and returns the atom

Python **accepts** `\xFE` as a valid 7-leading-bit atom prefix. [1](#0-0) 

**Tracing Rust `node_from_bytes` → `parse_atom(0xFE)` → `decode_size_with_offset(0xFE)`:**

```
initial_b = 0xFE = 11111110
leading_ones() = 7  →  atom_start_offset = 7
7 >= 8? No → passes first guard
bit_mask = 0xff >> 7 = 0x01
b = 0xFE & 0x01 = 0x00
size_blob = &mut stack_allocation[..7]   ← 7 bytes
size_blob[0] = 0x00
reads 6 more bytes into size_blob[1..]
size_blob.len() = 7 > 6  →  return Err(SerializationError)
```

Rust **rejects** `\xFE` unconditionally. [2](#0-1) 

This is confirmed by an explicit test case in the Rust test suite: [3](#0-2) 

**`node_from_bytes_backrefs` behavior for `\xFE`:**

In the backrefs deserializer, `0xFE` is the `BACK_REFERENCE` sentinel — it is consumed as a path, not an atom. So `\xFE`-prefixed bytes are interpreted as a back-reference path, not as an atom at all. [4](#0-3) [5](#0-4)

### Title
Python `_atom_from_stream` Accepts `\xFE`-Prefixed Atoms That Rust `node_from_bytes` Unconditionally Rejects — (`wheel/python/clvm_rs/ser.py`)

### Summary
The Python deserializer in `ser.py` treats `\xFE` as a valid 7-leading-bit atom size prefix and successfully decodes the atom. The Rust deserializer in `src/serde/parse_atom.rs` rejects `\xFE` with `SerializationError` because its `size_blob` would be 7 bytes, which exceeds the hard limit of 6. This is a concrete, locally reproducible Python/Rust deserializer split.

### Finding Description
In `_atom_from_stream`, the bit-counting loop runs 7 times for `\xFE`, producing `bit_count=7` and a stripped value of `0x00`. The function then reads 6 additional size bytes (total 7-byte prefix), computes the atom size, and reads the atom body — returning a valid atom node as long as `size < 0x400000000`. [6](#0-5) 

In `decode_size_with_offset`, `0xFE.leading_ones() = 7`, so `atom_start_offset = 7`. The first guard (`>= 8`) passes. The function allocates `size_blob = &mut stack_allocation[..7]`, reads 6 more bytes, then hits the second guard `size_blob.len() > 6` and returns `EvalErr::SerializationError`. [2](#0-1) 

The Rust test suite explicitly documents this rejection: [7](#0-6) 

The comment at the top of `ser.py` even acknowledges `\xFE` as a special case ("if it's 0xfe, it's nil"), but the code does not implement that — it falls through to the atom-parsing path, making the comment wrong and the behavior divergent from Rust. [8](#0-7) 

### Impact Explanation
Any code path that uses the Python `sexp_from_stream` to validate or pre-parse a serialized CLVM program before handing the raw bytes to the Rust executor will accept programs that Rust unconditionally rejects. Concretely:

- `sexp_from_stream` on `\xFE\x00\x00\x00\x00\x00\x01\x3c` → succeeds, returns atom `\x3c`
- `node_from_bytes` on the same bytes → `SerializationError`
- `node_from_bytes_backrefs` on the same bytes → interprets `\xFE` as a back-reference path, producing a completely different tree

A validator that uses Python deserialization to accept/reject programs before Rust execution will accept programs the Rust executor will always reject, and vice versa — a direct API-equivalence / consensus split.

### Likelihood Explanation
The proof-of-concept bytes are trivially constructable. The divergence is deterministic and requires no special privileges — any caller of the Python wheel's `sexp_from_stream` (or the higher-level `Program.from_bytes` if it routes through this path) with attacker-controlled bytes can trigger it.

### Recommendation
Align the Python `_atom_from_stream` with the Rust `decode_size_with_offset` guard: reject any first byte whose `leading_ones()` count produces a `size_blob` longer than 6 bytes. Concretely, add an explicit check before the loop or after computing `bit_count`:

```python
if bit_count > 6:
    raise ValueError("bad encoding")
```

This matches the Rust guard at `src/serde/parse_atom.rs` line 37. [9](#0-8)

def _atom_from_stream(f: BinaryIO, b: int, new_atom_f: NEW_ATOM_F) -> CLVMStorage:
    if b == 0x80:
        return new_atom_f(b"")
    if b <= MAX_SINGLE_BYTE:
        return new_atom_f(bytes([b]))
    bit_count = 0
    bit_mask = 0x80
    while b & bit_mask:
        bit_count += 1
        b &= 0xFF ^ bit_mask
        bit_mask >>= 1
    size_blob = bytes([b])
    if bit_count > 1:
        blob = f.read(bit_count - 1)
        if len(blob) != bit_count - 1:
            raise ValueError("bad encoding")
        size_blob += blob
    size = int.from_bytes(size_blob, "big")
    if size >= 0x400000000:
        raise ValueError("blob too large")
    blob = f.read(size)
    if len(blob) != size:
        raise ValueError("bad encoding")
    return new_atom_f(blob)
