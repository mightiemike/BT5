### Title
Unchecked Transaction Leaf Validity in Merkle Proof Verification Allows Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` accepts any caller-supplied `tx_id` as a valid transaction hash without verifying it is an actual leaf of the Bitcoin merkle tree. An unprivileged NEAR caller can supply an internal merkle-tree node hash as `tx_id`, pair it with a valid sibling proof, and cause the function to return `true` — falsely certifying that a non-existent transaction is confirmed in a real mainchain block. Any consumer contract (bridge, atomic swap, cross-chain lending protocol) that gates fund release on this return value is directly exploitable.

---

### Finding Description

The vulnerability class from the reference report is **missing validation of a critical property of an accepted resource**: the NFT loan contract accepted a token account without checking `amount == 1`. The direct analog here is that `verify_transaction_inclusion` accepts a `tx_id` without checking that it is a real transaction hash (a leaf node), rather than an internal node of the Bitcoin merkle tree.

In `verify_transaction_inclusion`, the only guard on `tx_id` is:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

`compute_root_from_merkle_proof` is a pure hash-chain computation that treats its first argument as an opaque 32-byte value and makes no distinction between a leaf hash and an internal node hash:

```rust

    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    ...
    for proof_hash in merkle_proof {
        ...
        current_hash = compute_hash(&current_hash, proof_hash);
        ...
    }
    current_hash
}
``` [2](#0-1) 

Because the function is still a live, public, unpermissioned entry point (only `#[pause]`, no role gate), any caller can invoke it directly: [3](#0-2) 

The `#[deprecated]` attribute is a Rust compiler hint; it imposes no on-chain restriction. The function remains callable at runtime.

---

### Impact Explanation

A consumer contract (bridge, atomic swap, cross-chain lending) that calls `verify_transaction_inclusion` and releases funds when it returns `true` can be drained. The attacker never needs to broadcast any Bitcoin transaction; they only need the publicly available merkle tree of any already-confirmed mainchain block. The corrupted proof result is: `verify_transaction_inclusion` returns `true` for a `tx_id` that does not correspond to any real Bitcoin transaction.

---

### Likelihood Explanation

The attack requires no privileges, no key material, and no mining. The attacker only needs to read the merkle tree of any block already stored in the contract's `headers_pool`. All required data is public. The only limiting factor is whether a consumer contract still calls the deprecated v1 path; given that v2 was introduced as a fix, any integration predating v2 remains exposed.

---

### Recommendation

Remove or hard-disable `verify_transaction_inclusion` (v1) at the contract level so it cannot be called at runtime. All callers must use `verify_transaction_inclusion_v2`, which enforces the coinbase-at-position-0 constraint that bounds the tree depth and prevents internal-node substitution: [4](#0-3) 

Additionally, add an explicit check inside `verify_transaction_inclusion_v2` (and its internal call to v1) that `tx_index` is strictly less than `2^(merkle_proof.len())`, ensuring the claimed position is a valid leaf index for the proven tree depth.

---

### Proof of Concept

Take any mainchain block with four transactions T0–T3. Its merkle tree is:

```
Root = H(N01, N23)
  N01 = H(H(T0), H(T1))
  N23 = H(H(T2), H(T3))
```

Attacker calls `verify_transaction_inclusion` with:
- `tx_id    = N01`  ← internal node, not a real transaction
- `tx_index = 0`
- `merkle_proof = [N23]`
- `tx_block_blockhash` = hash of the real block
- `confirmations = 1`

`compute_root_from_merkle_proof(N01, 0, [N23])` computes `H(N01, N23) = Root`, which equals `header.block_header.merkle_root`. The function returns `true`. No Bitcoin transaction was ever broadcast; the attacker has forged a confirmed-transaction proof for a non-existent transaction. [5](#0-4) [2](#0-1)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
