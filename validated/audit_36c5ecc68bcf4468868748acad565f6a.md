### Title
`verify_transaction_inclusion_v2` reverts on valid empty Merkle proofs for single-transaction blocks — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` unconditionally delegates to `verify_transaction_inclusion`, which always panics when `merkle_proof` is empty. For single-transaction Bitcoin blocks (e.g., coinbase-only blocks), the correct Merkle proof is empty because the transaction hash IS the Merkle root. Any caller submitting a valid SPV proof for such a block is permanently rejected with a contract panic.

---

### Finding Description

`verify_transaction_inclusion_v2` (line 347) performs two checks before delegating:

1. It requires `merkle_proof.len() == coinbase_merkle_proof.len()` — this passes when both are empty (0 == 0).
2. It calls `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof)` — with an empty slice, the for-loop in `compute_root_from_merkle_proof` does not execute and returns `coinbase_tx_id` unchanged. If `coinbase_tx_id == merkle_root` (which is exactly true for a single-transaction block), this check passes. [1](#0-0) 

It then unconditionally calls `verify_transaction_inclusion`: [2](#0-1) 

Inside `verify_transaction_inclusion`, the following guard is hit unconditionally:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [3](#0-2) 

For a single-transaction block, the correct and only valid Merkle proof is an empty vector — the transaction hash equals the Merkle root by definition. There is no sibling node to include. The `require!` guard is structurally analogous to the M-14 pattern: an operation is executed unconditionally even when the "amount" (here, proof length) is zero, and the downstream function rejects zero unconditionally.

The root cause is that `verify_transaction_inclusion_v2` does not short-circuit or handle the empty-proof case before delegating to the v1 function, and the v1 function has no awareness of the single-transaction special case. [4](#0-3) 

---

### Impact Explanation

The SPV verification functionality (`verify_transaction_inclusion_v2`) is permanently broken for all single-transaction Bitcoin blocks. Any consumer contract or user that needs to verify a transaction in such a block — including coinbase-only blocks, which are common during low-fee periods — will always receive a contract panic. The core light client guarantee (trustless transaction inclusion verification) is unavailable for this class of valid inputs.

---

### Likelihood Explanation

Bitcoin blocks with only a coinbase transaction are common during periods of low mempool activity. Any bridge, atomic swap, or cross-chain lending protocol built on top of this light client that needs to verify a coinbase transaction or any transaction in a single-tx block will reliably trigger this revert. The entry path is fully unprivileged: any NEAR account can call `verify_transaction_inclusion_v2`.

---

### Recommendation

In `verify_transaction_inclusion_v2`, add an explicit guard before delegating to the v1 function:

```rust
// Single-transaction block: empty proof is valid; tx_id must equal merkle_root
if args.merkle_proof.is_empty() {
    return args.tx_id == header.block_header.merkle_root
        && args.tx_index == 0;
}
```

Alternatively, relax the `require!(!args.merkle_proof.is_empty(), ...)` guard in `verify_transaction_inclusion` to allow the empty-proof case and instead verify directly that `tx_id == merkle_root` when the proof is empty.

---

### Proof of Concept

1. A Bitcoin block is mined containing only the coinbase transaction: `tx_hash = merkle_root`.
2. The block is submitted to the light client via `submit_blocks`.
3. A consumer calls `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_tx_hash`
   - `tx_block_blockhash = block_hash`
   - `tx_index = 0`
   - `merkle_proof = []` ← correct empty proof for single-tx block
   - `coinbase_tx_id = coinbase_tx_hash`
   - `coinbase_merkle_proof = []` ← correct empty proof
   - `confirmations = 1`
4. Length check: `0 == 0` → passes. [5](#0-4) 
5. `compute_root_from_merkle_proof(coinbase_tx_hash, 0, &[])` returns `coinbase_tx_hash` = `merkle_root` → coinbase check passes. [6](#0-5) 
6. `verify_transaction_inclusion(args.into())` is called unconditionally. [7](#0-6) 
7. `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` **panics**. [3](#0-2) 
8. The call reverts despite the submitted proof being cryptographically correct.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
