### Title
Bonus Reward Overpayment Due to Stale Shares in Historical Segment Calculation - (`x/tieredrewards/keeper/bonus_rewards.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary

`computeSegmentBonus` uses the position's **current** delegation shares (`pos.Delegation.Shares`) together with a **historical** snapshot `tokensPerShare` rate from a past validator event. If the position's shares increased between the historical event and the claim (e.g., via `MsgAddToPosition`), the bonus for the historical segment is overpaid, draining the protocol's reward pool.

### Finding Description

The vulnerability class from the external report is **value inconsistency**: one quantity is computed from an adjusted/current value while a related quantity uses a stale/unadjusted value, and the two are combined in a single calculation.

The exact analog lives in `computeSegmentBonus`: [1](#0-0) 

```go
func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
    ...
    tokens := pos.Delegation.Shares.Mul(tokensPerShare)
    return tokens.Mul(tier.BonusApy).MulInt64(durationSeconds).QuoInt64(types.SecondsPerYear).TruncateInt()
}
```

`pos.Delegation.Shares` is the **current** shares at claim time, loaded live from the staking keeper. `tokensPerShare` is a **historical snapshot** captured at the time of a past validator event.

The caller in `processEventsAndClaimBonus` passes `*pos` (the live position state) alongside `evt.TokensPerShare` (the historical rate): [2](#0-1) 

```go
for _, entry := range events {
    evt := entry.Event
    if bonded {
        bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
        totalBonus = totalBonus.Add(bonus)
    }
    ...
}
```

The invariant that must hold for a correct calculation is:

> **shares used for a historical segment must be the shares that existed during that segment.**

The code breaks this invariant: it uses the shares that exist **at claim time**, not at the time of the historical segment.

### Impact Explanation

An attacker can:

1. Create a position with a small amount (e.g., 1 token → shares `S_small`).
2. Wait for validator events to accumulate over a long bonded period `T`.
3. Call `MsgAddToPosition` to add a large amount, increasing shares to `S_large`.
4. Immediately call `MsgClaimRewards`.

The bonus paid for the historical period `T` is:

```
S_large × tokensPerShare_historical × BonusApy × T / SecondsPerYear
```

But the correct amount is:

```
S_small × tokensPerShare_historical × BonusApy × T / SecondsPerYear
```

The overpayment is `(S_large - S_small) × tokensPerShare_historical × BonusApy × T / SecondsPerYear`, paid from the `RewardsPoolName` module account. [3](#0-2) 

The corrupted value is the `RewardsPoolName` module account balance — real bonded tokens are transferred out in excess of what was earned.

### Likelihood Explanation

The entry path is entirely unprivileged: any user who holds a tiered-rewards position and can call `MsgAddToPosition` followed by `MsgClaimRewards` can trigger this. Both messages are standard user-facing transactions. The longer the gap between position creation and the add-to-position call, and the larger the added amount, the greater the overpayment. This is straightforwardly exploitable on mainnet.

### Recommendation

Validator events should record the position's share count at the time of each event, or the bonus calculation should use the shares that were active during the historical segment rather than the current shares. One approach: snapshot `pos.Delegation.Shares` into each `ValidatorEvent` entry at the time the event is recorded (e.g., in the slash/bond/unbond hooks), and pass that snapshot into `computeSegmentBonus` instead of the live `pos.Delegation.Shares`.

### Proof of Concept

1. Alice creates a position with 1 `basecro` at block 1. Shares ≈ 1.
2. 365 days pass. The validator remains bonded. One `BOND` event is recorded at block 1 with `tokensPerShare = 1.0`.
3. Alice calls `MsgAddToPosition` with 1,000,000 `basecro`. Shares ≈ 1,000,001.
4. Alice calls `MsgClaimRewards`.
5. `processEventsAndClaimBonus` iterates the historical event. It calls `computeSegmentBonus(*pos, tier, block1Time, eventTime, 1.0)` where `pos.Delegation.Shares = 1,000,001`.
6. Bonus paid ≈ `1,000,001 × 1.0 × BonusApy × 365days / year` — nearly the full-year bonus on 1,000,001 tokens, despite Alice only having 1 token for that entire year.
7. The `RewardsPoolName` account is drained by the excess `~1,000,000 × BonusApy` tokens. [4](#0-3) [5](#0-4)

```go
func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}
```
