### Title
Missing Per-Week Deposit Verification Allows Cross-Week Token Drain — (`File: core/contracts/Airdrop.sol`)

---

### Summary
`Airdrop.sol` allows the owner to register a merkle root for a new week without verifying that the contract holds sufficient token balance to cover that week's total allocations. Because all weeks share a single token pool with no per-week balance accounting, users holding valid proofs for a newly registered week (whose tokens have not yet been deposited) can successfully claim against tokens deposited for a prior week, leaving prior-week claimants unable to redeem their entitled amounts.

---

### Finding Description
`registerMerkleRoot()` increments `pastWeeks` and stores a merkle root for a given week, but performs no check that the contract's token balance is sufficient to cover the new week's total allocations. [1](#0-0) 

`_verifyProof()` — the only gate before a transfer executes — checks only that the week's root is non-zero, that the caller has not already claimed, and that the proof is valid. It does not verify that the contract holds enough tokens for the week being claimed. [2](#0-1) 

`_claim()` then immediately executes `SafeERC20.safeTransfer` against the single shared token pool. [3](#0-2) 

There is no mapping of `week → deposited balance`, no minimum-balance guard on `registerMerkleRoot`, and no per-week reserve accounting anywhere in the contract. All weeks compete for the same `token` balance held by the contract. [4](#0-3) 

---

### Impact Explanation
A user with a valid proof for week N (whose tokens have not yet been deposited) can successfully drain tokens that were deposited for week M. Week M claimants who have not yet claimed lose their entitled allocation permanently — their `claimed[M][sender]` entry is still zero, but the contract balance is zero, so every subsequent claim reverts. This is a direct, irreversible loss of user funds with no recovery path inside the contract.

---

### Likelihood Explanation
The scenario requires the owner to register a new week's merkle root before depositing that week's tokens — a realistic operational ordering (e.g., announcing rewards before the token transfer settles, or a multi-step deployment). No malicious admin intent is required; a routine timing gap is sufficient. Any user who observes the new merkle root on-chain before the deposit arrives can immediately exploit the window by submitting a valid claim against the prior week's pool.

---

### Recommendation
Add per-week deposit tracking. Before or at the time `registerMerkleRoot` is called, record the amount deposited for that week and require it to be non-zero. In `_verifyProof`, add a check analogous to the FantiumClaimingV1 recommendation:

```solidity
require(weekDeposits[week] > 0, "Tokens not yet deposited for this week");
```

Alternatively, require the token transfer to occur atomically inside `registerMerkleRoot` itself, so a root can never be live without its backing funds.

---

### Proof of Concept

1. Owner deposits 1 000 NADO into `Airdrop`, then calls `registerMerkleRoot(1, root1)`. Contract balance: 1 000 NADO.
2. Week 1 user A calls `claim([{week:1, totalAmount:500, proof:...}])` — succeeds. Contract balance: 500 NADO.
3. Owner calls `registerMerkleRoot(2, root2)` (week 2 total allocation: 1 000 NADO) **before** depositing week 2 tokens. Contract balance still: 500 NADO.
4. Week 2 user B calls `claim([{week:2, totalAmount:500, proof:...}])`. `_verifyProof` passes — `merkleRoots[2] != bytes32(0)`, proof is valid. `safeTransfer` succeeds because 500 NADO is present. Contract balance: 0 NADO.
5. Week 1 user C calls `claim([{week:1, totalAmount:500, proof:...}])`. `_verifyProof` passes, but `safeTransfer` reverts — contract is empty. User C's 500 NADO entitlement is permanently unclaimable. [5](#0-4)

```text
    function _verifyProof(
        uint32 week,
        address sender,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        require(claimed[week][sender] == 0, "Already claimed.");
        require(
            merkleRoots[week] != bytes32(0),
            "Week hasn't been registered."
        );
        require(
            !ISanctionsList(sanctions).isSanctioned(sender),
            "address is sanctioned."
        );
        bytes32 leaf = keccak256(
            bytes.concat(keccak256(abi.encode(sender, totalAmount)))
        );
        bool isValidLeaf = MerkleProof.verify(proof, merkleRoots[week], leaf);
        require(isValidLeaf, "Invalid proof.");
        claimed[week][sender] = totalAmount;
    }
```
