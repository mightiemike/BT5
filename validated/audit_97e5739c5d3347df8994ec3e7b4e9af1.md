### Title
P2P Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary
The protobuf deserializer for `ValidResourceBounds` maps any `AllResources{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` to `L1Gas(X)`. The `get_tip_resource_bounds_hash` function produces structurally different Poseidon hash preimages for these two variants (4 vs 3 elements). A V3 transaction admitted through the gateway is hashed with the `AllResources` path (4 elements), but after a P2P protobuf round-trip the same transaction is reconstructed as `L1Gas` (3 elements), so any subsequent call to `validate_transaction_hash` on the deserialized object returns the wrong hash.

### Finding Description

**Step 1 – Gateway admission always uses `AllResources`.**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds` (not `ValidResourceBounds`). When the gateway converts an incoming RPC transaction to its internal form, `InternalRpcInvokeTransactionV3::resource_bounds()` unconditionally returns `ValidResourceBounds::AllResources(self.resource_bounds)`: [1](#0-0) 

The hash is therefore computed via `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with the `AllResources` branch, which hashes **4 elements**: `[tip, pack(L1_GAS), pack(L2_GAS=0), pack(L1_DATA_GAS=0)]`. [2](#0-1) 

**Step 2 – P2P protobuf deserialization silently downgrades the variant.**

When the same transaction is received via P2P block sync, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies the following rule:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← chosen when both are zero
} else {
    ValidResourceBounds::AllResources(...)
})
``` [3](#0-2) 

For a transaction with `l2_gas = {0, 0}` and `l1_data_gas = {0, 0}`, the result is `L1Gas(X)` — a different enum variant than what was used at admission.

**Step 3 – The two variants produce different Poseidon hashes.**

`get_tip_resource_bounds_hash` branches on the variant:

| Variant | `resource_felts` | Hash input length |
|---|---|---|
| `AllResources{l1_gas:X, l2_gas:0, l1_data_gas:0}` | `[pack(X,L1_GAS), pack(0,L2_GAS), pack(0,L1_DATA_GAS)]` | 4 (tip + 3) |
| `L1Gas(X)` | `[pack(X,L1_GAS), pack(0,L2_GAS)]` | 3 (tip + 2) | [4](#0-3) 

The Poseidon hash of a 4-element input differs from that of a 3-element input even when the extra element is zero, because Poseidon is not length-prefix-free in this encoding.

**Step 4 – The gateway admits the triggering transaction.**

The stateless validator only rejects transactions where `max_possible_fee(Tip::ZERO) == 0` (all three bounds zero) or where `l2_gas.max_price_per_unit < min_gas_price`. A transaction with `l1_gas = {amount: A, price: P}` (A,P > 0) and `l2_gas = {0, 0}`, `l1_data_gas = {0, 0}` passes both checks when `min_gas_price = 0` or when `validate_resource_bounds = false` (the documented bootstrap mode): [5](#0-4) [6](#0-5) 

**Step 5 – `validate_transaction_hash` returns the wrong result.**

`validate_transaction_hash` recomputes the hash from the transaction object and compares it to the stored hash. After the protobuf round-trip the recomputed hash uses the `L1Gas` path (3 elements) while the stored hash was computed with the `AllResources` path (4 elements); the comparison fails. [7](#0-6) 

The same mismatch affects `get_declare_transaction_v3_hash` and `get_deploy_account_transaction_v3_hash`, which both call `get_tip_resource_bounds_hash`. [8](#0-7) 

### Impact Explanation
Any syncing node that calls `validate_transaction_hash` on a P2P-received block containing a V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` will compute a hash that does not match the committed transaction hash. This causes the node to reject a valid block, breaking liveness for syncing nodes. Additionally, any component that re-derives the transaction hash from the deserialized `Transaction` object (e.g., for block hash computation or receipt matching) will produce an incorrect value, constituting a wrong state/receipt result from accepted input.

### Likelihood Explanation
The condition is reachable in two documented operational modes: (1) bootstrap mode where `validate_resource_bounds = false` is explicitly recommended, and (2) any deployment where `min_gas_price = 0`. The `RpcInvokeTransactionV3` struct accepts `AllResourceBounds` with zero `l2_gas` and `l1_data_gas` without any structural rejection. The protobuf round-trip is exercised by every P2P-syncing node.

### Recommendation
In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not use the zero-check heuristic to select the variant. Instead, require the sender to explicitly signal the variant (e.g., via a protobuf enum field or a version tag), or always deserialize as `AllResources` for transactions whose version is ≥ 0.13.3. Alternatively, add a canonical sentinel value (e.g., a non-zero `l1_data_gas.max_price_per_unit`) that distinguishes old `L1Gas` transactions from new `AllResources` transactions with zero data-gas amounts. [9](#0-8) 

### Proof of Concept

```
1. Construct a V3 invoke transaction with:
     l1_gas  = { max_amount: 1000, max_price_per_unit: 1 }
     l2_gas  = { max_amount: 0,    max_price_per_unit: 0 }
     l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }

2. Submit via the gateway (with validate_resource_bounds=false or min_gas_price=0).
   The gateway computes hash H1 using AllResources path (4-element Poseidon input).
   H1 is stored in the block.

3. Serialize the committed Transaction to protobuf (TransactionInBlock).
   The protobuf ResourceBounds message has l2_gas and l1_data_gas both zero.

4. Deserialize the protobuf on a syncing node.
   TryFrom<protobuf::ResourceBounds>: l1_data_gas.is_zero() && l2_gas.is_zero() → L1Gas(X).

5. Call validate_transaction_hash on the deserialized transaction.
   get_tip_resource_bounds_hash uses L1Gas path → 3-element Poseidon input → hash H2 ≠ H1.
   validate_transaction_hash returns false for a valid transaction.
```

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
