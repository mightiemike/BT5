### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Proof Forgery Mitigation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is marked `#[deprecated]` but carries no access restriction. Any unprivileged NEAR caller can invoke it directly, supplying fully attacker-controlled `tx_id`, `tx_index`, and `merkle_proof` without the coinbase-proof validation that `verify_transaction_inclusion_v2` was introduced to enforce. This allows an attacker to forge a `true` SPV proof result for a `tx_id` that is an internal Merkle tree node rather than a real transaction, deceiving any recipient contract that consumes the verification result.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions). Its mitigation requires two things:

1. `coinbase_merkle_proof.len() == merkle_proof.len()` — both proofs must span the same tree depth.
2. The coinbase proof must verify against the stored Merkle root — anchoring the claimed depth to a real leaf. [1](#0-0) 

Because the coinbase is always at index 0 and is always a real leaf at full tree depth D, requiring equal proof lengths forces the claimed transaction to also be at depth D — ruling out internal nodes, which sit at depth d < D.

The deprecated v1 function, however, performs none of these checks. It accepts `tx_id`, `tx_index`, and `merkle_proof` entirely from the caller and passes them directly to `compute_root_from_merkle_proof`: [2](#0-1) 

The only guard is `require!(!args.merkle_proof.is_empty(), ...)`. The contract's own documentation acknowledges the consequence:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

Critically, `#[deprecated]` in Rust is a compiler lint, not an access control mechanism. The function is `pub`, carries only `#[pause]` (which restricts nothing when the contract is not paused), and is therefore callable by any NEAR account.

`compute_root_from_merkle_proof` iterates over the supplied proof hashes and uses `current_position /= 2` to track left/right placement at each level: [4](#0-3) 

For an internal node N at depth d, the attacker supplies a proof of length d (the sibling path from N to the root). The function correctly recomputes the root from N's perspective and the comparison `== header.block_header.merkle_root` succeeds — returning `true`.

---

### Impact Explanation

A recipient contract (or any on-chain consumer) that calls `verify_transaction_inclusion` directly, or that trusts its `true` return value, can be deceived into accepting a forged SPV proof. The attacker does not need to find a hash preimage; they only need to read the Merkle tree of any confirmed block (public data) and supply an internal node with its correct sibling path. The corrupted state is the proof result itself: a `true` value for a `tx_id` that was never a real transaction in the block.

---

### Likelihood Explanation

The entry point is fully permissionless — any NEAR account can call `verify_transaction_inclusion` with arbitrary Borsh-encoded `ProofArgs`. All required inputs (block Merkle tree structure, internal node hashes, sibling paths) are derivable from public Bitcoin chain data. No privileged role, leaked key, or social engineering is required. The attack is deterministic and requires no brute force.

---

### Recommendation

Remove the public accessibility of `verify_transaction_inclusion`. Options in order of preference:

1. **Add `#[private]`** to `verify_transaction_inclusion` so it can only be called as an internal cross-contract call (effectively making it an internal helper for v2 only).
2. **Delete the public function entirely** and inline its logic into `verify_transaction_inclusion_v2`, since v2 already calls it as a subroutine via `self.verify_transaction_inclusion(args.into())`.
3. At minimum, add an explicit `require!(false, "use verify_transaction_inclusion_v2")` guard at the top of the function body to make the deprecation a hard runtime rejection rather than a compiler hint.

---

### Proof of Concept

Given a confirmed mainchain block B with a Merkle tree of depth D ≥ 2:

1. Read the Merkle tree of block B from public Bitcoin data. Identify any internal node N at depth d = 1 (i.e., the direct child of the root). N = `double_sha256(left_leaf || right_leaf)`.
2. The sibling of N at depth 1 is the other child of the root, call it S.
3. Construct `merkle_proof = [S]` (length 1). Set `tx_index = 0` (N is the left child).
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N`
   - `tx_block_blockhash = hash(B)`
   - `tx_index = 0`
   - `merkle_proof = [S]`
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, 0, [S])` computes `double_sha256(N || S)` = Merkle root of B.
6. The comparison succeeds; the function returns `true` for a `tx_id` that is an internal node, not a real transaction. [5](#0-4) [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
