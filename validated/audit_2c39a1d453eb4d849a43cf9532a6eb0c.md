### Title
Incorrect `LastKnownBonded=true` Initialization When Position Is Created After UNBOND Event Causes Phantom Bonus Accrual During Unbonded Validator Period - (`x/tieredrewards/keeper/position.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`createDelegatedPosition` unconditionally sets `LastKnownBonded=true` and sets `LastEventSeq` to the current latest event seq. When a position is created in a block where the validator's UNBOND event has already been appended (e.g., the event fires in EndBlock of block T and the position is created in block T+1, or the event fires in BeginBlock jailing and the position is created in the same block's transactions), the UNBOND event is permanently skipped. The position starts with `LastKnownBonded=true` even though the validator is already unbonding. When the validator later re-bonds, `processEventsAndClaimBonus` replays from `LastEventSeq` (which is ≥ the UNBOND event's seq), sees only the BOND event, and — because `bonded=true` — computes a bonus for the entire interval `[LastBonusAccrual, bondEventTime]` as if the validator was bonded throughout, directly draining the rewards pool.

---

### Finding Description

**Root cause — `createDelegatedPosition`:** [1](#0-0) 

`getValidatorEventLatestSeq` returns the seq of the most recently appended event (including an UNBOND event that just fired). `NewPosition` is then called with `lastKnownBonded=true` hardcoded: [2](#0-1) 

The proto comment confirms the intent: `"True when the position is first created (only bonded validators accepted)"`, but the code does not enforce that the validator is actually bonded at creation time. [3](#0-2) 

`validateNewPosition` checks only for vesting accounts, close-only tier, and minimum lock amount. **There is no check that the validator is currently bonded.** Cosmos SDK allows delegation to an unbonding (non-jailed) validator, so `ms.delegate` in `LockTier` succeeds.

**Replay logic — `processEventsAndClaimBonus`:** [4](#0-3) 

Events are fetched with `seq > pos.LastEventSeq`. If `LastEventSeq = N` (the UNBOND event's seq), the UNBOND event is never seen. [5](#0-4) 

The loop starts with `bonded = pos.LastKnownBonded = true`. When the BOND event (seq=N+1) is encountered, `bonded=true` so the segment `[LastBonusAccrual, bondEventTime]` is computed as a bonded segment — even though the validator was unbonded for that entire interval.

**The defensive final-segment guard does not help:** [6](#0-5) 

The `val.IsBonded()` check only guards the final open-ended segment after all events are processed. The bonus computed *inside* the events loop for the pre-BOND segment is not guarded.

---

### Impact Explanation

A user whose position is created after an UNBOND event (but with `LastEventSeq` pointing at that event) will receive bonus rewards for the entire unbonded interval `[position_creation_time, validator_rebond_time]`. This is a direct, quantifiable transfer from the `RewardsPoolName` module account to the attacker's address: [7](#0-6) 

The magnitude scales with: `shares × tokensPerShare × bonusApy × unbondedDurationSeconds / SecondsPerYear`. For a large position and a long unbonding period, this can be substantial.

---

### Likelihood Explanation

The precondition is reachable via a normal production path:

1. **Block T EndBlock:** A validator falls out of the active set (insufficient voting power, no jailing required). `AfterValidatorBeginUnbonding` fires, appending UNBOND event seq=N.
2. **Block T+1 Tx:** User submits `MsgLockTier` or `MsgCommitDelegationToTier` targeting that validator. `validateNewPosition` passes (no bonded check). `getValidatorEventLatestSeq` returns N. Position stored with `LastEventSeq=N`, `LastKnownBonded=true`.
3. **Block T+M:** Validator re-enters active set. `AfterValidatorBonded` fires, appending BOND event seq=N+1.
4. **User calls `MsgClaimTierRewards`:** Bonus paid for `[T+1, T+M]` as if bonded. [8](#0-7) [9](#0-8) 

A monitoring attacker who tracks validator voting power can reliably predict when a validator will be kicked from the active set and front-run the position creation in the next block. No privileged access is required.

---

### Recommendation

In `createDelegatedPosition`, after fetching `lastEventSeq`, query the validator's current bonded status and use it as the initial `lastKnownBonded` value instead of hardcoding `true`:

```go
lastEventSeq, err := k.getValidatorEventLatestSeq(ctx, valAddr)
// ...
val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
// ...
initialBonded := val.IsBonded()
pos := types.NewPosition(id, owner, tier.Id, delAddr.String(), blockHeight, lastEventSeq, blockTime, initialBonded, blockTime)
```

Additionally, `validateNewPosition` (or the callers `LockTier`/`CommitDelegationToTier`) should reject position creation on non-bonded validators, consistent with the stated invariant in the ADR. [1](#0-0) 

---

### Proof of Concept

```
Block T EndBlock:
  validator V falls out of active set
  → AfterValidatorBeginUnbonding fires
  → appendValidatorEvent: UNBOND event stored at seq=1
  → ValidatorEventSeq[V] = 1

Block T+1 Tx (MsgLockTier, validator=V):
  validateNewPosition: passes (no bonded check)
  delegate(posDelAddr, V, amount): succeeds (unbonding, not jailed)
  createDelegatedPosition:
    getValidatorEventLatestSeq(V) → 1
    NewPosition(..., lastEventSeq=1, lastKnownBonded=true)
  Position stored: LastEventSeq=1, LastKnownBonded=true, LastBonusAccrual=T+1

Block T+N Tx (some delegation restores V to active set):
  → AfterValidatorBonded fires
  → appendValidatorEvent: BOND event stored at seq=2

Block T+N+1 Tx (MsgClaimTierRewards):
  processEventsAndClaimBonus:
    getValidatorEventsSince(V, 1) → [BOND event at seq=2]
    bonded = pos.LastKnownBonded = true   ← BUG: should be false
    segmentStart = T+1
    for BOND event:
      bonded=true → computeSegmentBonus([T+1, T+N]) → NON-ZERO  ← phantom bonus
      bonded = true
      segmentStart = T+N
    val.IsBonded() = true → computeSegmentBonus([T+N, T+N+1]) → bonus
  Total bonus paid for [T+1, T+N+1] even though V was unbonded [T+1, T+N]
```

Assert: bonus received > 0 for the unbonded interval `[T+1, T+N]`. This violates the invariant that bonus accrues only during bonded validator periods and constitutes a direct economic loss from the rewards pool.

func (h Hooks) AfterValidatorBeginUnbonding(ctx context.Context, _ sdk.ConsAddress, valAddr sdk.ValAddress) error {
	count, err := h.k.getPositionCountForValidator(ctx, valAddr)
	if err != nil {
		return err
	}
	if count == 0 {
		return nil
	}

	tokensPerShare, err := h.k.getTokensPerShare(ctx, valAddr)
	if err != nil {
		return err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	_, err = h.k.appendValidatorEvent(ctx, valAddr, types.ValidatorEvent{
		Height:         sdkCtx.BlockHeight(),
		Timestamp:      sdkCtx.BlockTime(),
		EventType:      types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND,
		TokensPerShare: tokensPerShare,
		ReferenceCount: count,
	})
	return err
}
