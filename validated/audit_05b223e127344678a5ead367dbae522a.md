### Title
Unprotected `acl_init_super_admin` Can Be Front-Run Before `init`, Permanently Bricking the Contract - (`contract/src/lib.rs`)

### Summary

The `BtcLightClient` contract relies on `acl_init_super_admin` (generated by the `near-plugins` `#[access_control]` macro) being called inside `init` to establish the contract account as super admin. Because `acl_init_super_admin` is a public, permissionless method that writes to its own independent storage prefix — not the main contract state — it can be called by any NEAR account before `init` is ever invoked. An attacker who front-runs this call permanently prevents the contract from being initialized, requiring a full redeployment to a new account.

### Finding Description

The `init` function is decorated with `#[init]` and `#[private]`, which correctly prevents re-initialization and restricts who can call `init` itself. However, the ACL super-admin slot is set inside `init` via:

```rust
near_sdk::require!(
    contract.acl_init_super_admin(env::current_account_id()),
    "Failed to initialize super admin",
);
``` [1](#0-0) 

`acl_init_super_admin` is a public method generated by the `#[access_control(role_type(Role))]` macro from `near-plugins`. It writes to its own dedicated storage prefix, entirely independent of the main `BtcLightClient` state. This means:

1. `PanicOnDefault` does **not** protect it — `PanicOnDefault` only panics when the main contract state is deserialized, which ACL methods never do.
2. It can be called by **any** NEAR account at any time after the WASM is deployed, even before `init` is called.
3. It is a one-time setter: it returns `false` (and does nothing) if a super admin is already set. [2](#0-1) 

The deployment flow used by the relayer calls `init` as a **separate transaction** from the WASM deployment: [3](#0-2) 

This creates a window between WASM upload and the `init` call during which an attacker can call `acl_init_super_admin(attacker_account_id)`.

### Impact Explanation

Once an attacker calls `acl_init_super_admin(attacker_account_id)` before `init`:

- Every subsequent call to `init` panics at `require!(contract.acl_init_super_admin(...), "Failed to initialize super admin")` because the slot is already occupied.
- Since `#[init]` prevents calling `init` after state is written, and the state is never written (each `init` call panics and is rolled back), the contract is permanently stuck in an uninitialized state.
- No block headers can ever be submitted. No SPV proofs can ever be verified. The entire light client is dead.
- The only recovery is redeployment to a new contract account, which may require coordination with downstream consumers of the contract address. [4](#0-3) 

### Likelihood Explanation

NEAR transactions are publicly observable in the mempool. Any attacker monitoring for WASM deployments to the known contract account can detect the deployment and submit `acl_init_super_admin` before the deployer's `init` transaction is finalized. This requires no special privileges, no leaked keys, and no social engineering — only a standard NEAR function call with no attached deposit.

### Recommendation

Move the super-admin initialization out of the `init` body and into the deployment batch itself, or use a factory contract that atomically deploys and initializes in a single batch transaction (combining `DeployContract` + `FunctionCall` actions). Alternatively, add an explicit check inside `acl_init_super_admin` (or wrap it) so that it can only be called by `env::current_account_id()`, matching the `#[private]` guard already on `init`.

### Proof of Concept

1. Deployer uploads WASM to `btc-light-client.near` via a `DeployContract` action.
2. Attacker observes the deployment and immediately submits:
   ```
   FunctionCall { method: "acl_init_super_admin", args: {"account_id": "attacker.near"} }
   ```
   to `btc-light-client.near`. This succeeds — no state exists yet, `PanicOnDefault` is not triggered, and the ACL storage slot is empty.
3. Deployer submits the `init` transaction. Inside `init`, `contract.acl_init_super_admin(env::current_account_id())` returns `false` because the slot is already taken by `attacker.near`.
4. `require!(false, "Failed to initialize super admin")` panics. The transaction is rolled back. No state is written.
5. Every subsequent `init` attempt by the deployer produces the same panic. The contract is permanently uninitialized and must be redeployed to a new account. [5](#0-4)
