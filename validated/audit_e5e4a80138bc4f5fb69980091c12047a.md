### Title
`finalize_epoch` fallback paths silently drop scheduled shard-layout and `last_resharding` update on `ThresholdError`/`NotEnoughValidators` — (`File: chain/epoch-manager/src/lib.rs`)

### Summary

In `EpochManager::finalize_epoch()`, when `proposals_to_epoch_info()` fails with `EpochError::ThresholdError` or `EpochError::NotEnoughValidators`, the fallback clones `next_epoch_info` (epoch T+1) and only increments `epoch_height`. It does **not** apply the already-computed `next_next_shard_layout` or `last_resharding` to the cloned struct. When `DynamicResharding` is active and a shard split was scheduled for epoch T+2, the stored `EpochInfoV5` for T+2 carries the **old shard layout** (T+1's) and the **stale `last_resharding`** value, silently dropping the split and corrupting the resharding-cooldown invariant.

### Finding Description

`finalize_epoch()` computes three values before calling `proposals_to_epoch_info()`:

1. `next_next_shard_layout` — the derived `ShardLayoutV3` for epoch T+2 (may differ from T+1's layout when a split is scheduled).
2. `last_resharding` — set to `next_epoch_info.epoch_height() + 1` when `next_next_shard_layout != next_shard_layout`.
3. `strategy` — the `AssignmentStrategy` (possibly `StickyResharding`).

These are passed into `proposals_to_epoch_info()`. On success, `EpochInfo::new()` gates on `ProtocolFeature::DynamicResharding` and constructs an `EpochInfoV5` that embeds both `shard_layout` and `last_resharding`.

On failure, the two fallback arms are:

```rust

    *epoch_info.epoch_height_mut() += 1;
    epoch_info

```

`next_epoch_info` is epoch T+1's `EpochInfoV5`. Its `shard_layout` field is T+1's layout (N shards). Its `last_resharding` is T+1's value. Neither is updated to `next_next_shard_layout` (N+1 shards) or the newly computed `last_resharding`. The result is saved as T+2's authoritative epoch info. [1](#0-0) 

The `EpochInfoV5` struct that would have been written by the success path carries both fields: [2](#0-1) 

`EpochInfo::new()` populates them only on the success path: [3](#0-2) 

`get_shard_layout()` reads `EpochInfo::shard_layout()` first, falling back to the static config only for pre-V5 structs: [4](#0-3) 

`can_reshard()` reads `last_resharding()` from the stored `EpochInfo` to enforce the cooldown: [5](#0-4) 

### Impact Explanation

**Wrong shard layout stored for epoch T+2.** Every call to `get_shard_layout(&epoch_T2_id)` returns the old N-shard layout instead of the derived N+1-shard layout. All shard-layout-dependent operations for epoch T+2 — chunk producer assignment, state-witness shard routing, trie-key routing, resharding manager's `start_resharding()` — operate on the wrong layout. The scheduled split is silently dropped with no error.

**Resharding-cooldown invariant broken.** `last_resharding` in the stored T+2 info is not updated to `next_epoch_info.epoch_height() + 1`. `can_reshard()` therefore believes no resharding occurred, allowing the next resharding attempt to fire immediately (back-to-back reshardings), which the design explicitly prohibits because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`. [6](#0-5) 

### Likelihood Explanation

`NotEnoughValidators` is returned from `assign_chunk_producers_to_shards` when the number of active chunk producers falls below `minimum_validators_per_shard * num_shards`. During a resharding epoch, `num_shards` increases by one, raising this threshold. A validator (or coordinated set of validators) can deliberately submit zero-stake proposals during epoch T to push the active count below the new threshold, triggering the fallback at the epoch boundary. This is an unprivileged, publicly-available action (staking/unstaking). The window is the entire epoch T, giving ample time to coordinate. `ThresholdError` (total stake < seat count) is a less likely but structurally identical path. [7](#0-6) 

### Recommendation

In both fallback arms, after cloning `next_epoch_info`, apply the computed `next_next_shard_layout` and `last_resharding` to the cloned struct before saving. For `EpochInfoV5`, this means updating `v5.shard_layout` and `v5.last_resharding`. A helper method `epoch_info.set_shard_layout(next_next_shard_layout)` and `epoch_info.set_last_resharding(last_resharding)` should be added alongside the existing `epoch_height_mut()` mutator. Alternatively, construct a minimal valid `EpochInfoV5` from scratch using the same fields as the success path but carrying forward the previous validator set, which is the semantically correct behavior (keep validators, update layout metadata).

### Proof of Concept

1. Dynamic resharding is active (protocol version ≥ 85). Current epoch T has N shards.
2. A chunk producer proposes a split; `shard_split` is embedded in the last block of epoch T. `next_next_shard_layout` is derived as an N+1-shard `ShardLayoutV3`.
3. During epoch T, a validator submits a zero-stake proposal, reducing active chunk producers below `minimum_validators_per_shard * (N+1)`.
4. `finalize_epoch()` computes `next_next_shard_layout` (N+1 shards) and `last_resharding = epoch_height_T1 + 1`, then calls `proposals_to_epoch_info()`.
5. `proposals_to_epoch_info()` → `get_chunk_producers_assignment()` → `assign_chunk_producers_to_shards()` fails → `EpochError::NotEnoughValidators`.
6. Fallback fires: `EpochInfo::clone(&next_epoch_info)` copies T+1's `EpochInfoV5` with `shard_layout = N-shard layout` and `last_resharding = T+1's value`. Only `epoch_height` is incremented.
7. This is saved as T+2's epoch info via `save_epoch_info()`.
8. `get_shard_layout(&epoch_T2_id)` returns the N-shard layout. The split is silently dropped.
9. `can_reshard()` for the next epoch reads `last_resharding = None` (or the old value), bypassing the cooldown, and a new split can be scheduled immediately, violating the back-to-back resharding safety invariant. [8](#0-7)

        &self,
