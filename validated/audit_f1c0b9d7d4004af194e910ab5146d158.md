### Title
Duplicate-Last-Leaf Position Ambiguity in `compute_root_from_merkle_proof` Allows Same Proof to Verify Two Distinct `tx_index` Values — (`merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` performs no bounds check on `transaction_position` relative to the actual tree width. In any odd-count transaction tree, Bitcoin's canonical construction pads the last leaf by duplicating it. Because the proof element for the real last leaf and its phantom duplicate are identical, the same `(tx_id, merkle_proof)` tuple produces the correct Merkle root for **two distinct `tx_index` values** — the real index `N-1` and the phantom index `N`. `verify_transaction_inclusion_v2` inherits this flaw and returns `true` for both calls.

---

### Finding Description

**`compute_root_from_merkle_proof` — `merkle-tools/src/lib.rs` lines 34–52**

```
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
    current_hash
}
``` [1](#0-0) 

The function accepts any caller-supplied `transaction_position` and never validates it against the proof length or the actual tree width. No upper-bound check exists anywhere in the call chain.

**Concrete arithmetic for a 3-tx tree `[T0, T1, T2]`:**

Bitcoin pads to `[T0, T1, T2, T2]`, producing:
- Level 1: `[H(T0,T1), H(T2,T2)]`
- Root: `H(H(T0,T1), H(T2,T2))`

Proof for `T2` at **position 2** (real, even):
```
step 1: pos=2 (even)  → hash = H(T2, T2),          pos → 1
step 2: pos=1 (odd)   → hash = H(H(T0,T1), H(T2,T2)) = Root ✓
```

Proof for `T2` at **position 3** (phantom, odd):
```
step 1: pos=3 (odd)   → hash = H(T2, T2),          pos → 1
step 2: pos=1 (odd)   → hash = H(H(T0,T1), H(T2,T2)) = Root ✓
```

The same `merkle_proof = [T2, H(T0,T1)]` satisfies both calls. The coinbase guard in `verify_transaction_inclusion_v2` is irrelevant here — it only validates that `coinbase_tx_id` at position `0` reaches the root; it says nothing about the upper bound of `tx_index` for the target transaction. [2](#0-1) 

The inner call to `verify_transaction_inclusion` passes `tx_index` directly to `compute_root_from_merkle_proof` with no bounds check: [3](#0-2) 

---

### Impact Explanation

Any contract that gates an economic action on `verify_transaction_inclusion_v2` returning `true` and uses `(tx_id, tx_index, block_hash)` as a uniqueness key — the natural design for a bridge or peg-out contract — can be drained twice for the same on-chain Bitcoin event. The attacker submits two calls with identical `tx_id`, `tx_block_blockhash`, and `merkle_proof` but with `tx_index = N-1` and `tx_index = N` respectively. Both return `true` against the same stored Merkle root. No privileged role, no reorg, and no special chain state is required.

The "across reorg" framing in the question is a separate, weaker path (general replay after re-inclusion) and is not needed to trigger the duplicate-position issue. The duplicate-position exploit works entirely within a single canonical block.

---

### Likelihood Explanation

- The attacker controls `tx_index` directly as a caller-supplied argument with no on-chain validation.
- Any Bitcoin block with an odd transaction count (the majority of real blocks) is a valid target.
- The proof bytes are identical for both calls — no cryptographic work is required.
- The light client stores no redemption state, so there is no replay guard at this layer.

---

### Recommendation

1. **Bound-check `tx_index` against proof length.** The maximum valid leaf index in a tree whose proof has `k` levels is `2^k - 1`. Reject any `tx_index >= 2^(merkle_proof.len())`.
2. **Reject the phantom duplicate position explicitly.** After computing the root, verify that `tx_index` is strictly less than the actual transaction count. Because the light client does not store the transaction count, the simplest fix is to require that the proof element at the first level is **not equal to `tx_id`** when `tx_index` is even (i.e., the sibling must be a distinct hash, not the self-duplicate).
3. **Alternatively**, require callers to supply the block's transaction count and validate `tx_index < tx_count` before entering the proof loop.
