### Title
`verify_transaction_inclusion_v2` Coinbase Proof Bypass Enables 64-Byte Merkle Forgery — (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` is documented as fixing the 64-byte transaction Merkle proof forgery vulnerability by requiring a separate coinbase proof at index 0. However, the coinbase check does not enforce that `coinbase_tx_id` is distinct from `tx_id`. An unprivileged NEAR caller can supply the same internal Merkle node for both fields, causing both the coinbase check and the transaction check to pass for a transaction that does not exist in the block.

### Finding Description

`verify_transaction_inclusion_v2` (lines 347–369 of `contract/src/lib.rs`) is the replacement for the deprecated `verify_transaction_inclusion`. Its stated purpose is to prevent the 64-byte transaction Merkle proof forgery attack by requiring a valid coinbase proof at position 0 with the same proof length as the transaction proof. [1](#0-0) 

The coinbase guard is: [2](#0-1) 

This check only verifies that `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_proof) == merkle_root`. It does **not** verify that `coinbase_tx_id` is the actual coinbase transaction hash, nor that it differs from `tx_id`. Because `compute_root_from_merkle_proof` is a pure arithmetic function over caller-supplied inputs: [3](#0-2) 

an attacker can supply `coinbase_tx_id = tx_id = H` (an internal Merkle node) and `coinbase_merkle_proof = merkle_proof = P`, where `compute_root_from_merkle_proof(H, 0, P) == merkle_root`. Both the coinbase check and the subsequent call to the deprecated `verify_transaction_inclusion` resolve to the same computation and both return true.

The deprecated v1 function, which v2 calls internally, contains the explicit warning that it may return `true` for internal nodes: [4](#0-3) 

The length-equality guard (`merkle_proof.len() == coinbase_merkle_proof.len()`) is satisfied trivially when both arrays are identical. The non-empty guard in v1 is also satisfied as long as the proof has at least one element. [5](#0-4) 

### Impact Explanation

Any downstream contract or off-chain system that calls `verify_transaction_inclusion_v2` and treats a `true` return value as proof of a real Bitcoin transaction inclusion can be deceived. A concrete example is a cross-chain bridge that gates fund releases on a verified Bitcoin deposit transaction: an attacker can forge a proof for a non-existent transaction using a publicly computable internal Merkle node from any real Bitcoin block, receive a `true` result, and claim funds that were never deposited. The corrupted value is the proof result itself — the canonical chain state is not modified, but the verification output is falsified.

### Likelihood Explanation

The attack requires no privileged access. `verify_transaction_inclusion_v2` is a public, permissionless NEAR function callable by any account. All inputs needed for the attack (internal Merkle nodes of real Bitcoin blocks) are publicly available from any Bitcoin block explorer. The construction is mechanical: pick any block on the mainchain with at least two transactions, compute one level of internal nodes, and submit the forged proof. No cryptographic hardness assumption is broken.

### Recommendation

Add an explicit check that `coinbase_tx_id != tx_id` before the coinbase proof verification. More robustly, require the caller to supply the raw coinbase transaction bytes, deserialize them on-chain, hash them to derive `coinbase_tx_id`, and verify that the result equals the supplied `coinbase_tx_id`. This removes the attacker's freedom to choose `coinbase_tx_id` arbitrarily. The length-equality guard alone is insufficient because it is trivially satisfied when both proofs are identical.

### Proof of Concept

Consider a real Bitcoin block `B` on the mainchain with four transactions `[T0, T1, T2, T3]` and Merkle root `R`:

```
        R
       / \
    H01   H23
   /   \ /   \
  T0  T1 T2  T3
```

where `H01 = hash(T0, T1)`, `H23 = hash(T2, T3)`, `R = hash(H01, H23)`.

The attacker calls `verify_transaction_inclusion_v2` with:

```
tx_id                = H01   // internal node, not a real transaction
tx_block_blockhash   = B
tx_index             = 0
merkle_proof         = [H23]
coinbase_tx_id       = H01   // same internal node
coinbase_merkle_proof= [H23] // same proof
confirmations        = 1
```

Step-by-step execution:

1. Length check: `[H23].len() == [H23].len()` → passes.
2. Coinbase check: `compute_root_from_merkle_proof(H01, 0, [H23])` → position 0 is even → `hash(H01, H23) = R == merkle_root` → passes.
3. `verify_transaction_inclusion` is called with the same `tx_id = H01`, `tx_index = 0`, `merkle_proof = [H23]`.
4. Block is on mainchain, confirmations satisfied.
5. `compute_root_from_merkle_proof(H01, 0, [H23]) = R == merkle_root` → returns `true`.

The function returns `true` for `H01`, an internal Merkle node that is not a Bitcoin transaction, with no cryptographic forgery required.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
