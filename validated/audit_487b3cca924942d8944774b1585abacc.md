### Title
`SwapAllowlistExtension` checks router address as `sender`, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` ŌĆö the immediate caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the original user. If the pool admin allowlists the router (the only way to enable router-mediated swaps for their users), every unprivileged user can bypass the per-user allowlist by routing through the router.

---

### Finding Description

**Pool passes `msg.sender` as `sender` to the extension:** [1](#0-0) 

`_beforeSwap` is called with `msg.sender` as the first argument. When the router calls `pool.swap()`, `msg.sender` at the pool is the router address.

**`ExtensionCalling._beforeSwap` forwards it unchanged:** [2](#0-1) 

**`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`:** [3](#0-2) 

`msg.sender` here is the pool; `sender` is the first argument ŌĆö the router address when the swap is router-mediated.

**`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly:** [4](#0-3) 

The pool sees `msg.sender = router`. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**The same applies to every router entry point** (`exactInput`, `exactOutputSingle`, `exactOutput`), including intermediate hops where `msg.sender` at the pool is `address(this)` (the router itself): [5](#0-4) 

**Contrast with `DepositAllowlistExtension`**, which correctly checks `owner` (the economically relevant identity) rather than `sender` (the immediate caller): [6](#0-5) 

The deposit allowlist is not affected because it gates the position owner, which is independent of whether the liquidity adder is used as an intermediary.

---

### Impact Explanation

A pool admin who deploys a restricted pool (e.g., KYC-gated, institutional-only) and configures `SwapAllowlistExtension` with a per-user allowlist must also allowlist the router if they want their approved users to be able to use `MetricOmmSimpleRouter`. Allowlisting the router sets `allowedSwapper[pool][router] = true`, which causes the extension to pass for **any** caller who routes through the router ŌĆö including users who are explicitly not on the allowlist. The access-control invariant ("only approved addresses may swap") is broken for all router-mediated paths. Unauthorized users can trade in pools that are intended to be restricted, undermining the pool's access policy and any downstream compliance or strategy guarantees that depend on it.

---

### Likelihood Explanation

The scenario requires the pool admin to allowlist the router, which is the natural and expected configuration for any restricted pool whose approved users are expected to interact via the standard periphery. There is no in-protocol mechanism that warns the admin that allowlisting the router is equivalent to opening the pool to all users. The bypass is reachable by any unprivileged address with no special setup beyond calling the public router.

---

### Recommendation

Gate the allowlist on the **original user's identity**, not the immediate caller. Two concrete approaches:

1. **Extension-data forwarding**: Have the router encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it. The extension must also verify that `sender` (the immediate caller) is a trusted router before trusting the decoded identity.

2. **Separate allowlist tiers**: Expose a `trustedRouter` mapping in the extension. When `sender` is a trusted router, decode the real user from `extensionData` and check `allowedSwapper[pool][realUser]`. When `sender` is not a trusted router, check `allowedSwapper[pool][sender]` as today.

The `DepositAllowlistExtension` pattern (checking `owner` rather than `sender`) is the correct model: gate the economically relevant identity, not the transport layer.

---

### Proof of Concept

```
Setup:
  pool admin deploys pool with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, user1, true)   // allowlist user1
  pool admin calls setAllowedToSwap(pool, router, true)  // enable router for user1

Attack:
  attacker (user2, not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: attacker, ...})

Execution trace:
  router.exactInputSingle()
    ŌåÆ pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)
        msg.sender at pool = router
      ŌåÆ _beforeSwap(sender=router, ...)
        ŌåÆ SwapAllowlistExtension.beforeSwap(sender=router, ...)
            allowedSwapper[pool][router] == true  ŌåÉ passes
      ŌåÆ swap executes for attacker
```

`user2` successfully swaps in a pool they are not allowlisted for, because the extension sees `sender = router` (allowlisted) rather than `sender = user2` (not allowlisted).
