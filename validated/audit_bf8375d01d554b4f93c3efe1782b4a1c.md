### Title
Child `ChunkExtra.proposed_split` Inherited from Parent After Shard Split Causes Permanent `InvalidChunkHeaderShardSplit` on Child Shard's First Chunk - (File: chain/chain/src/resharding/manager.rs)

---

### Summary

When the resharding manager creates child `ChunkExtra` records after a dynamic shard split, it clones the parent's `ChunkExtra` and only updates `state_root` and `congestion_info`. The `proposed_split` field — which the chunk-header validation gate compares byte-for-byte — is never reset to `None`. The child shard's first chunk header will always carry `proposed_split = None` (because the resharding cooldown blocks `compute_proposed_split` from returning a non-`None` value), while the stored child `ChunkExtra` carries the parent's stale `Some(TrieSplit { … })`. The strict equality check in `validate_chunk_with_chunk_extra_and_receipts_root` then rejects every first chunk of every child shard with `InvalidChunkHeaderShardSplit`, permanently halting the child shard.

---

### Finding Description

**Root cause — `chain/chain/src/resharding/manager.rs` lines 255–260:**

```rust
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// proposed_split is NOT reset — it inherits parent's Some(TrieSplit{…})
``` [1](#0-0) 

The parent shard's last `ChunkExtra` has `proposed_split = Some(TrieSplit { boundary_account, left_memory, right_memory })` — that is precisely the value that triggered the resharding. The child `ChunkExtra` inherits this value verbatim.

**Validation gate — `chain/chain/src/validate.rs` lines 176–185:**

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    DYNAMIC_RESHARDING_VALIDATION_FAILURES.with_label_values(&["chunk_header"]).inc();
    return Err(Error::InvalidChunkHeaderShardSplit(format!(
        "header has {:?}, expected {:?} ...",
        chunk_header.proposed_split(),
        prev_chunk_extra.proposed_split(),
    )));
}
``` [2](#0-1) 

This check is called unconditionally for every new chunk in `chain.rs`: [3](#0-2) 

**Why the cooldown makes it worse, not better:**

`compute_proposed_split` checks `can_reshard()` before computing a split. Because a resharding just occurred, `can_reshard()` returns `false` for the entire cooldown window, so the child shard's first chunk header always embeds `proposed_split = None`. The divergent pair is:

| Source | Value |
|---|---|
| `prev_chunk_extra.proposed_split()` (child, inherited from parent) | `Some(TrieSplit { boundary_account, … })` |
| `chunk_header.proposed_split()` (child's first chunk, cooldown active) | `None` |

The wiki explicitly acknowledges this: *"allowing back-to-back reshardings is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`."* [4](#0-3) 

The cooldown prevents the *next* resharding from being selected, but it does not fix the stale `proposed_split` in the child's `ChunkExtra`. The mismatch is structural and permanent for the child shard's first chunk.

### Impact Explanation

Every child shard produced by a dynamic shard split will have its first chunk permanently rejected by `validate_chunk_with_chunk_extra_and_receipts_root` with `InvalidChunkHeaderShardSplit`. Since this validation runs on every node before chunk endorsement, no chunk producer can produce an accepted first chunk for the child shard. The child shard is rendered permanently unusable — an exact protocol-level DoS on the newly created shard, halting all transactions and receipts routed to it. Because dynamic resharding is designed to run automatically without a protocol upgrade, this fires on every split event once `ProtocolFeature::DynamicResharding` is active.

**Severity: High/Critical** — permanent shard halt triggered by the normal protocol resharding path.

---

### Likelihood Explanation

The bug fires on every dynamic shard split, unconditionally, as long as `ProtocolFeature::DynamicResharding` is enabled and a shard's memory usage crosses `memory_usage_threshold`. No adversarial action is required; the trigger is the automatic protocol-level resharding pipeline. The only prerequisite is that the feature flag is active (protocol version ≥ the `DynamicResharding` activation version).

**Likelihood: High** once the feature is live on mainnet/testnet.

---

### Recommendation

In `chain/chain/src/resharding/manager.rs`, after cloning the parent `ChunkExtra`, explicitly reset `proposed_split` to `None` for each child:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// FIX: child shard has no prior split proposal; reset to None
if let ChunkExtra::V5(ref mut v5) = child_chunk_extra {
    v5.proposed_split = None;
}
```

Additionally, add a `proposed_split_mut()` accessor to `ChunkExtra` (mirroring `state_root_mut()` and `congestion_info_mut()`) and use it here, so the reset is type-safe and version-aware. A regression test should verify that after a shard split, the child shard's first chunk passes `validate_chunk_with_chunk_extra_and_receipts_root`.

### Proof of Concept

**Step-by-step trace:**

1. Epoch N, last block: shard S is above `memory_usage_threshold`. `compute_proposed_split` stores `ChunkExtra.proposed_split = Some(TrieSplit { boundary_account: "mid.near", left_memory: X, right_memory: Y })` for shard S. This value is embedded in the chunk header and selected by `get_upcoming_shard_split()`, which writes `shard_split = Some((S, "mid.near"))` into `BlockHeaderInnerRestV6`.

2. Epoch N→N+1 boundary: `finalize_epoch()` creates `EpochInfoV5` for epoch N+2 with the new `ShardLayoutV3` (children `S_left`, `S_right`). `last_resharding` is set to `epoch_height + 1`.

3. Epoch N+1: `ReshardingManager::build_split_shard_storage_changes()` runs. For each child `S_left`, `S_right`:
   ```rust
   let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
   // parent_chunk_extra.proposed_split = Some(TrieSplit{…})
   *child_chunk_extra.state_root_mut() = trie_changes.new_root;
   *child_chunk_extra.congestion_info_mut() = child_congestion_info;
   // proposed_split is NOT reset → child inherits Some(TrieSplit{…})
   chain_store_update.save_chunk_extra(block_hash, &new_shard_uid, child_chunk_extra.into());
   ``` [7](#0-6) 

4. Epoch N+2, first block: chunk producer for `S_left` calls `compute_proposed_split`. `can_reshard()` checks `epoch_height - last_resharding < min_epochs_between_resharding` → returns `false`. `compute_proposed_split` returns `None`. Chunk header is produced with `proposed_split = None`.

5. Validation in `validate_chunk_with_chunk_extra_and_receipts_root`:
   - `prev_chunk_extra.proposed_split()` = `Some(TrieSplit { boundary_account: "mid.near", … })` (inherited)
   - `chunk_header.proposed_split()` = `None`
   - `Some(…) != None` → `Err(InvalidChunkHeaderShardSplit(…))` [2](#0-1) 

6. Every node rejects the chunk. No endorsements are collected. The child shard `S_left` (and `S_right`) cannot produce any accepted chunk. All accounts in those shards are permanently inaccessible.
