### Title
Protobuf `ResourceBounds` → `ValidResourceBounds` Conversion Uses Zero-Value Sentinel, Producing Wrong `L1Gas` Variant and Divergent Transaction Hash for V3 Transactions with Zero L2/Data Gas - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ResourceBounds` classifies a V3 transaction as `ValidResourceBounds::L1Gas` whenever `l1_data_gas == 0 && l2_gas == 0`. The canonical JSON/storage deserializer (`DeprecatedResourceBoundsMapping → ValidResourceBounds`) uses **key presence**, not zero-value, as the discriminant. Because `get_tip_resource_bounds_hash` produces structurally different hashes for the two variants (3 vs 4 Poseidon inputs), any V3 transaction with `AllResources { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` that travels through the P2P protobuf path is stored with the wrong variant and a hash that cannot be reproduced from the stored transaction body.

---

### Finding Description

**Canonical discriminant — key presence (`crates/starknet_api/src/transaction/fields.rs`)**

```rust
match resource_bounds_mapping.0.get(&Resource::L1DataGas) {
    Some(data_bounds) => Ok(Self::AllResources(AllResourceBounds { … })),
    None => {
        if l2_bounds.is_zero() { Ok(Self::L1Gas(*l1_bounds)) }
        else { Err(…) }
    }
}
```

If the `L1DataGas` key is present in the map — even with a zero value — the result is `AllResources`. [1](#0-0) 

**Protobuf discriminant — zero-value sentinel (`crates/apollo_protobuf/src/converters/transaction.rs`)**

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
…
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong for V3 AllResources
} else {
    ValidResourceBounds::AllResources(…)
})
``` [2](#0-1) 

The serializer (`From<ValidResourceBounds> for protobuf::ResourceBounds`) emits `l1_data_gas = Some(zero)` for **both** `L1Gas` and `AllResources { …, l2_gas: 0, l1_data_gas: 0 }`: [3](#0-2) 

The two variants are therefore **indistinguishable** in the wire format, making the round-trip lossy.

**Hash divergence — `get_tip_resource_bounds_hash` (`crates/starknet_api/src/transaction_hash.rs`)**

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 3-element Poseidon
    ValidResourceBounds::AllResources(all) =>
        vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?],   // 4-element Poseidon
});
``` [4](#0-3) 

For a transaction with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`:

| Path | Variant stored | Hash inputs | Hash value |
|---|---|---|---|
| Gateway (JSON write API) | `AllResources` | `[tip, L1_GAS, L2_GAS(0), L1_DATA_GAS(0)]` | `H_all` |
| P2P receiving node (protobuf) | `L1Gas` | `[tip, L1_GAS, L2_GAS(0)]` | `H_l1 ≠ H_all` |

The P2P path trusts the hash from the wire message (`tx_hash` field in `TransactionInBlock`) and stores it alongside the deserialized transaction body: [5](#0-4) 

The receiving node therefore holds an **inconsistent pair**: stored hash = `H_all` (correct, from wire), stored `resource_bounds` = `L1Gas` (wrong variant). Any subsequent call to `get_transaction_hash` on the stored transaction body recomputes `H_l1 ≠ H_all`.

**Gateway accepts V3 transactions with zero L2/data gas**

The stateless validator explicitly accepts `AllResourceBounds { l1_gas: NON_EMPTY, l2_gas: 0, l1_data_gas: 0 }`: [6](#0-5) 

The existing protobuf round-trip test (`assert_transaction_to_vec_u8_and_back`) only exercises `RESOURCE_BOUNDS_MAPPING` with non-zero L2 and data gas, so the lossy case is not covered: [7](#0-6) 

---

### Impact Explanation

A receiving node that syncs a block containing such a transaction via P2P stores `ValidResourceBounds::L1Gas` for a transaction whose committed hash was computed over `AllResources`. Any component that recomputes the hash from the stored body — block re-execution, proof generation (the Cairo OS `hash_fee_fields` asserts `n_resource_bounds = 3` for V3 transactions), or RPC `starknet_getTransactionByHash` hash verification — will produce `H_l1 ≠ H_all`, yielding a wrong or unverifiable transaction hash. This matches the **"Wrong state, receipt, event … or revert result from blockifier/syscall/execution logic for accepted input"** and **"RPC execution … returns an authoritative-looking wrong value"** impact categories.

---

### Likelihood Explanation

Any user can submit a valid V3 invoke, declare, or deploy-account transaction with `l2_gas = 0` and `l1_data_gas = 0` (L1-only fee). The gateway accepts it. Once included in a block and propagated via P2P, every syncing node is affected. No special privilege is required.

---

### Recommendation

Replace the zero-value sentinel in the protobuf deserializer with an explicit variant tag. The simplest fix is to always produce `AllResources` when deserializing a V3 transaction (the transaction version is available in the enclosing `TransactionInBlock` message), or to add a boolean `is_all_resources` field to the protobuf `ResourceBounds` message so the serializer can preserve the variant unambiguously. The serializer (`From<ValidResourceBounds> for protobuf::ResourceBounds`) must also be updated to emit a distinguishable representation for `L1Gas` vs `AllResources` with zero data/L2 gas.

---

### Proof of Concept

1. Construct a V3 invoke transaction with `resource_bounds = AllResourceBounds { l1_gas: GasAmount(1)/GasPrice(1), l2_gas: 0, l1_data_gas: 0 }`. Compute its hash using `get_invoke_transaction_v3_hash` — this calls `get_tip_resource_bounds_hash` with `AllResources`, producing a 4-element Poseidon hash `H_all`.

2. Submit the transaction via the RPC write API. The gateway accepts it (stateless validator passes for non-zero L1 gas only) and stores `(tx_hash = H_all, resource_bounds = AllResources)`.

3. Serialize the stored transaction to protobuf via `From<ValidResourceBounds> for protobuf::ResourceBounds`. The wire bytes have `l1_data_gas = Some(zero)`, `l2_gas = Some(zero)`.

4. Deserialize on a receiving node via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Because `l1_data_gas.is_zero() && l2_gas.is_zero()`, the result is `ValidResourceBounds::L1Gas(l1_gas)`.

5. Call `get_invoke_transaction_v3_hash` on the deserialized transaction. `get_tip_resource_bounds_hash` now takes the `L1Gas` branch, producing a 3-element Poseidon hash `H_l1`.

6. Assert `H_l1 ≠ H_all` — the stored hash and the recomputed hash diverge. The receiving node holds an inconsistent transaction record.

    type Error = StarknetApiError;
    fn try_from(
        resource_bounds_mapping: DeprecatedResourceBoundsMapping,

        if let (Some(l1_bounds), Some(l2_bounds)) = (
            resource_bounds_mapping.0.get(&Resource::L1Gas),
            resource_bounds_mapping.0.get(&Resource::L2Gas),

            match resource_bounds_mapping.0.get(&Resource::L1DataGas) {

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
