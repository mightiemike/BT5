### Title
Missing Vote Signature Verification Allows Forged Votes to Count Toward Quorum — (`crates/apollo_consensus/src/single_height_consensus.rs`)

### Summary

`handle_vote` explicitly skips cryptographic signature verification (marked `TODO`). Any network peer can craft a `Vote` with `voter` set to a valid committee member address and `signature = RawSignature::default()` (all-zero bytes). The vote passes every existing guard and is forwarded to the state machine, where it counts toward quorum weight. The codebase itself acknowledges this gap in two separate comments.

---

### Finding Description

The `Vote` struct carries a `signature: RawSignature` field intended to bind the voter's identity to the `proposal_commitment`: [1](#0-0) 

`handle_vote` performs three checks before accepting a vote:

1. Height equality check [2](#0-1) 
2. Committee membership check (only verifies `vote.voter` is a known address, not that the sender controls that address) [3](#0-2) 
3. Duplicate/conflict check via `state_machine.received_vote` [4](#0-3) 

Signature verification is explicitly deferred with a TODO and never performed: [5](#0-4) 

After passing those three checks, the vote is unconditionally forwarded to the state machine: [6](#0-5) 

The codebase itself acknowledges the gap in `cache_future_vote`:

> "Since vote signatures are not yet verified, a peer can forge votes with arbitrary voter addresses" [7](#0-6) 

---

### Impact Explanation

An attacker who is a network peer (unprivileged) can:

1. Enumerate committee members (public information from the staking contract).
2. Craft `N` `Vote` messages, one per committee member, with `signature = RawSignature::default()`.
3. Broadcast them via the gossipsub `BroadcastVoteChannel`.
4. Each forged vote passes height, committee-membership, and duplicate checks, then is counted toward quorum weight in the state machine.
5. If the attacker forges enough votes to reach the quorum threshold, the state machine emits `DecisionReached`, committing a block that was not actually approved by the real committee.

This maps to **High: Transaction conversion or signature/hash logic binds the wrong signer** — the vote signature is the mechanism that is supposed to bind a voter's identity to a `proposal_commitment`; without verification, any peer can impersonate any committee member.

---

### Likelihood Explanation

- **Path is fully unprivileged**: any gossipsub peer can send a `Vote` protobuf message.
- **No rate-limiting or rejection** of structurally valid votes from known committee addresses exists.
- **Exact exploit value is concrete**: `voter = <any committee member ContractAddress>`, `signature = [0u8; 64]` (or whatever `RawSignature::default()` produces).
- The only practical constraint is that the attacker must be a connected gossipsub peer, which is the normal operating assumption for a public network.

---

### Recommendation

Implement the signature verification called out by the TODO before the committee-membership check in `handle_vote`. The `apollo_signature_manager` crate already contains the verification infrastructure (`verify_signature`). The vote message to be signed should canonically commit to `(vote_type, height, round, proposal_commitment, voter)` under a domain-separated hash to prevent cross-type replay.
