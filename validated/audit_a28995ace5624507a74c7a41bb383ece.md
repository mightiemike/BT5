### Title
Deprecated `verify_transaction_inclusion` Remains Callable On-Chain, Allowing Any Unprivileged Caller to Bypass the 64-Byte Forgery Protection Enforced by `verify_transaction_inclusion_v2` — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two SPV proof-verification entry points with nearly identical core logic but different security guarantees. `verify_transaction_inclusion` (v1) omits the coinbase Merkle proof check that `verify_transaction_inclusion_v2` uses to block the 64-byte transaction forgery attack. Despite being annotated `#[deprecated]`, v1 carries only a Rust compiler warning; it remains a live, `#[pause]`-gated public method callable by any unprivileged NEAR account. An attacker can route a forged proof directly to v1, receive a `true` result, and present that result to any downstream consumer that trusts the contract's verification output.

---

### Finding Description

**Root cause — two parallel entry points with divergent security, no enforcement of the secure path.**

`verify_transaction_inclusion` (v1) is defined at `contract/src/lib.rs:288`:

```rust

`verify_transaction_inclusion_v2` is defined at `contract/src/lib.rs:347`:

#[pause]
pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
    // ... length equality check ...
    require!(
        merkle_tools::compute_root_from_merkle_proof(
            args.coinbase_tx_id.clone(), 0usize, &args.coinbase_merkle_proof,
        ) == header.block_header.merkle_root,
        "Incorrect coinbase merkle proof"   // <-- the only additional guard
    );
    #[allow(deprecated)]
    self.verify_transaction_inclusion(args.into())   // delegates to v1
}
```

The two functions are structurally identical except for the coinbase proof gate. `ProofArgs` (the v1 argument type) is a strict subset of `ProofArgsV2`; `From<ProofArgsV2> for ProofArgs` drops the coinbase fields entirely.

`#[deprecated]` in Rust is a lint attribute. It emits a compiler warning at call sites in Rust source code. It has **no effect on the compiled WASM binary** and does not prevent the method from being invoked via a NEAR RPC call. The method remains exported under the name `verify_transaction_inclusion` and is reachable by any NEAR account.

**Attack path — 64-byte internal-node forgery via v1:**

The 64-byte transaction attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) exploits the fact that Bitcoin's Merkle tree does not encode node depth. An internal node at depth *d* is itself a 32-byte hash; its "proof" is the set of siblings from depth *d+1* up to the root. If an attacker supplies this internal node hash as `tx_id` and the corresponding sibling path as `merkle_proof`, `compute_root_from_merkle_proof` will compute the correct merkle root and v1 will return `true`.

v2 blocks this by requiring a valid coinbase proof at index 0. Because the coinbase is always a leaf at the full tree depth, its proof length equals the tree height. An internal-node proof is shorter; the coinbase check forces the proof depth to be consistent with the full tree, making the internal-node trick fail.

An attacker bypasses this entirely by calling v1 directly:

1. Identify a block already accepted by the contract (its `merkle_root` is stored in `headers_pool`).
2. Compute an internal Merkle node *N* at depth *d* in that block's transaction tree.
3. Construct `ProofArgs { tx_id: N, tx_block_blockhash: <block>, tx_index: <position of N at depth d>, merkle_proof: <siblings from d+1 to root>, confirmations: 0 }`.
4. Call `verify_transaction_inclusion` directly via NEAR RPC.
5. The function returns `true` — the forged "transaction" is accepted as included.

No privileged role, no leaked key, no social engineering is required. The call is open to any NEAR account.

---

### Impact Explanation

Any downstream contract or off-chain application that calls `verify_transaction_inclusion` (v1) — or that an attacker can route to v1 — will receive a `true` result for a fabricated transaction. In cross-chain bridge, atomic swap, or payment-verification contexts this means an attacker can prove a Bitcoin transaction occurred when it did not, enabling theft of bridged assets or fraudulent settlement. The corrupted value is the boolean proof result returned by the contract, which downstream consumers treat as authoritative.

---

### Likelihood Explanation

The entry point is unconditionally reachable by any unprivileged NEAR account. The `#[deprecated]` annotation provides zero on-chain enforcement. The 64-byte forgery technique is publicly documented and well understood. The only precondition is knowledge of a block already stored in the contract's `headers_pool`, which is public state. Likelihood is high.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the deployed contract entirely. If the method must be retained for internal use by `verify_transaction_inclusion_v2`, mark it `#[private]` (NEAR SDK attribute that restricts calls to the contract itself) or make it a non-`pub` Rust function. The `#[deprecated]` annotation alone is insufficient — it is a compile-time hint with no runtime enforcement.

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
