### Title
Unconditional Negative `vQuoteBalance` Transfer to Parent on Isolated Subaccount Close — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`_tryCloseIsolatedSubaccount` transfers `vQuoteBalance` to the parent unconditionally, including when it is negative. In gap-risk scenarios where the isolated subaccount's realized perp loss exceeds the deposited margin, the parent absorbs the excess loss without consent, reducing parent health beyond the initial margin and potentially triggering parent liquidation.

---

### Finding Description

In `OffchainExchange._tryCloseIsolatedSubaccount`, when `balance.amount == 0` (position fully closed), the function checks only `balance.vQuoteBalance != 0` before transferring it to the parent: [1](#0-0) 

There is no guard for `balance.vQuoteBalance < 0`. The two `perpEngine.updateBalance` calls at lines 174–185 zero out the isolated subaccount's `vQuoteBalance` and add it (sign-preserved) to the parent's `vQuoteBalance`. If `vQuoteBalance` is negative (realized loss), the parent's perp balance is debited by the full loss amount.

This function is reachable via two production paths that perform **no prior settlement or socialization**:

1. **`CloseIsolatedSubaccount` transaction** — `EndpointTx.sol` calls `tryCloseIsolatedSubaccount` directly with no settlement step: [2](#0-1) 

2. **`delistProduct`** — closes the position at delist price then immediately calls `tryCloseIsolatedSubaccount` with no settlement step: [3](#0-2) 

By contrast, the **liquidation path** (`_finalizeSubaccount`) correctly settles positive PnL, settles negative PnL against the spot quote balance, and then calls `socializeSubaccount` to zero out any remaining negative `vQuoteBalance` before `tryCloseIsolatedSubaccount` is invoked: [4](#0-3) 

The `socializeSubaccount` function explicitly zeros out negative `vQuoteBalance` by spreading it across open interest: [5](#0-4) 

The `CloseIsolatedSubaccount` and `delistProduct` paths skip this entire settlement/socialization pipeline.

---

### Impact Explanation

**Accounting identity broken:**

Let `M` = initial margin deposited, `L` = realized loss on the isolated subaccount.

After `_tryCloseIsolatedSubaccount`:
- Parent spot `quoteBalance` delta: `+M` (margin returned)
- Parent perp `vQuoteBalance` delta: `-L`
- Net parent health delta: `M - L`

When `L > M` (gap risk), the parent's health decreases by `L - M` beyond what the parent consented to risk. The parent deposited `M` as isolated margin but absorbs `L` in health reduction.

Health is computed by `_calculateProductHealth`, which adds `quoteAmount` (including `vQuoteBalance`) directly: [6](#0-5) 

A sufficiently large negative `vQuoteBalance` on the parent can push it below maintenance health, triggering liquidation of the parent's entire portfolio — a disproportionate consequence of an isolated position.

---

### Likelihood Explanation

- **Gap risk** (sudden price move exceeding margin before liquidation) is a realistic market event, especially for high-leverage isolated positions.
- The `CloseIsolatedSubaccount` transaction is a normal sequencer-submitted transaction that fires after any position closure. No special attacker privilege is required.
- A user can open a highly leveraged isolated position with minimal margin, and if the market gaps, the position closes with `vQuoteBalance << -M`. The sequencer then submits `CloseIsolatedSubaccount`, and the parent absorbs the full loss.
- The `delistProduct` path is also reachable by protocol governance and can trigger this for all isolated subaccounts simultaneously.

---

### Recommendation

Before transferring `vQuoteBalance` to the parent in `_tryCloseIsolatedSubaccount`, apply the same settlement/socialization logic used in `_finalizeSubaccount`:

1. If `vQuoteBalance > 0`: settle it against `state.availableSettle` (convert to spot quote).
2. If `vQuoteBalance < 0`: settle it against the isolated subaccount's spot `quoteBalance` up to the available amount, then socialize any remainder via `perpEngine.socializeSubaccount`.
3. Only after this pipeline, transfer the (now zero or positive) `vQuoteBalance` to the parent.

Alternatively, add a hard cap: if `balance.vQuoteBalance < 0`, clamp the transfer to `max(balance.vQuoteBalance, -spotQuoteBalance)` and socialize the remainder, mirroring the liquidation path.

---

### Proof of Concept

```
Setup:
  parent = bytes32("alice-default")
  isolated = createIsolatedSubaccount(parent, productId=2, margin=100e18 USDC)

Step 1 — Open long position:
  matchOrders(isolated, long 1 BTC at 50_000e18)
  → isolated.amount = 1e18
  → isolated.vQuoteBalance = -50_000e18

Step 2 — Price gaps down to 49_800e18 (loss = 200 USDC > margin = 100 USDC):
  matchOrders(isolated, short 1 BTC at 49_800e18)  // close position
  → isolated.amount = 0
  → isolated.vQuoteBalance = -200e18
  → isolated.spotQuoteBalance = 100e18 (margin, no fees for simplicity)

Step 3 — Sequencer submits CloseIsolatedSubaccount:
  endpoint.submitTransactions([CloseIsolatedSubaccount{subaccount: isolated}])
  → _tryCloseIsolatedSubaccount fires
  → perpEngine.updateBalance(2, isolated, 0, +200e18)   // zero out isolated
  → perpEngine.updateBalance(2, parent,   0, -200e18)   // impose loss on parent
  → spotEngine.updateBalance(0, isolated, -100e18)
  → spotEngine.updateBalance(0, parent,   +100e18)

Assert:
  parent.vQuoteBalance[productId=2] == -200e18   // ✓ loss transferred
  parent.spotQuoteBalance == baseline - 100 + 100 == baseline  // margin returned
  parent health delta == -200e18                 // exceeds initial margin of 100e18
  // parent may now be under maintenance → liquidatable
```

The parent absorbs a 200 USDC health reduction despite only consenting to risk 100 USDC of isolated margin. The excess 100 USDC loss bypasses the socialization mechanism that the liquidation path correctly applies.
