### Title
Relayer Advances `first_block_height_to_submit` on Unfinalized NEAR RPC Response, Creating Permanent Gap in `mainchain_height_to_header` — (`relayer/src/near_client.rs`)

---

### Summary

`parse_submit_blocks_response` silently returns `Ok(SubmitResult { gas_burnt: 0 })` when `response.final_execution_outcome` is `None`. The NEAR RPC legitimately returns `final_execution_outcome: None` for transactions that have been broadcast but not yet finalized. The `submit_blocks` polling loop exits immediately on any `Ok(response)` — it only retries on `TimeoutError` or `UnknownTransaction` errors. The caller in `prepare_and_submit_batches` unconditionally advances `first_block_height_to_submit` to `tx.last_block_height + 1` for any `Ok(Ok(result))`, regardless of `gas_burnt`. This permanently skips re-submission of blocks the contract never accepted, leaving a gap in `mainchain_height_to_header` that causes `verify_transaction_inclusion` to panic for any transaction in those heights.

---

### Finding Description

**Step 1 — `parse_submit_blocks_response` returns `Ok` on `None` outcome:** [1](#0-0) 

When `response.final_execution_outcome` is `None`, the `if let Some(...)` branch is skipped entirely and the function falls through to line 340, returning `Ok(SubmitResult { gas_burnt: 0 })`. There is no error, no retry signal, and no distinction from a genuinely successful submission.

**Step 2 — `submit_blocks` loop exits immediately on any `Ok(response)`:** [2](#0-1) 

The loop only retries on `RpcTransactionError::TimeoutError` or `RpcTransactionError::UnknownTransaction`. If the NEAR RPC returns `Ok(response)` with `final_execution_outcome: None` (a documented partial-response state when a transaction is received but not yet finalized), the loop exits and propagates `Ok(SubmitResult { gas_burnt: 0 })` to the caller.

**Step 3 — Caller unconditionally advances `first_block_height_to_submit`:** [3](#0-2) 

On `Ok(Ok(result))`, line 205 stores `tx.last_block_height + 1` into `first_block_height_to_submit` unconditionally. The `gas_burnt > 0` check on line 206 only gates the adaptive batch sizer — it does **not** gate the height advancement. A `gas_burnt == 0` result (the exact value produced by the `None` branch) still advances the pointer.

**Step 4 — Contract state is never updated; gap is permanent:**

The blocks in `[tx.first_block_height, tx.last_block_height]` were never submitted to the contract. `mainchain_height_to_header` has no entries for those heights. On the next sync iteration, `first_block_height_to_submit` starts at `tx.last_block_height + 1`, so those heights are never retried.

**Step 5 — `verify_transaction_inclusion` panics on the gap:** [4](#0-3) 

Any call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction in the skipped block range hits `mainchain_header_to_height.get(...).unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"))` and panics, making SPV proofs permanently unverifiable for those heights.

---

### Impact Explanation

The relayer's canonical progress pointer (`first_block_height_to_submit`) advances past blocks the on-chain contract never recorded. The resulting gap in `mainchain_height_to_header` is permanent — no recovery path exists in the relayer loop. Any downstream bridge or settlement contract that calls `verify_transaction_inclusion` for a transaction in the skipped range receives a contract panic (execution failure), not a `false` return. This silently breaks SPV proof verification for an unbounded range of Bitcoin blocks.

---

### Likelihood Explanation

The NEAR RPC `tx` endpoint is documented to return `final_execution_outcome: None` when a transaction has been broadcast but not yet finalized. This is a normal transient state, not an exceptional one. Under any network congestion, RPC node lag, or brief NEAR epoch boundary, the polling loop can receive this partial response before the transaction finalizes. The bug requires no attacker — it is triggered by ordinary infrastructure variability.

---

### Recommendation

`parse_submit_blocks_response` must treat `final_execution_outcome == None` as a non-terminal state, not a success. Return a distinct error variant (e.g., `CustomError::NotFinalized`) so the `submit_blocks` polling loop can retry:

```rust
// relayer/src/near_client.rs
fn parse_submit_blocks_response(
    response: &RpcTransactionResponse,
) -> Result<SubmitResult, CustomError> {
    let Some(ref final_execution_outcome) = response.final_execution_outcome else {
        return Err(CustomError::NotFinalized); // NEW: treat as retriable, not success
    };
    // ... existing outcome handling ...
}
```

In `submit_blocks`, add a retry arm for `CustomError::NotFinalized` analogous to the existing `TimeoutError` arm. Additionally, the `first_block_height_to_submit` advancement in `prepare_and_submit_batches` should be gated on `gas_burnt > 0` as a defense-in-depth check.
