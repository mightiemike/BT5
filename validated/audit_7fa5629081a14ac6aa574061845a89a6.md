### Title
Rebasing Token Balance Snapshot Mismatch in `creditDeposit` Causes Permanent Deposit Lock — (`File: core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` snapshots a token balance via `balanceOf` and then passes that exact snapshot value to `endpoint.depositCollateralWithReferral`, which internally calls `transferFrom` for the same amount. For rebasing tokens (e.g., OUSD-like tokens) where `balanceOf` can return a different value between two calls in the same block, the `transferFrom` will attempt to pull more tokens than the DDA actually holds at execution time, causing the entire `creditDeposit` call to revert. Tokens deposited to the DDA become permanently inaccessible to the subaccount until privileged owner intervention.

---

### Finding Description

`DirectDepositV1.creditDeposit()` implements the following pattern:

```solidity
uint256 balance = token.balanceOf(address(this));   // snapshot at T1
if (balance != 0) {
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),                            // exact snapshot used
        "-1"
    );
}
``` [1](#0-0) 

`endpoint.depositCollateralWithReferral` calls `handleDepositTransfer(token, msg.sender, uint256(amount))`, which performs `token.safeTransferFrom(dda, clearinghouse, amount)` — attempting to pull exactly `balance` tokens from the DDA. [2](#0-1) 

For a rebasing token, the value returned by `balanceOf` at T1 (the snapshot) may differ from the actual balance at T2 (the `transferFrom` execution). If the token rebases downward between T1 and T2, the DDA holds fewer tokens than `balance`, causing `safeTransferFrom` to revert. Because `creditDeposit` has no partial-fill or retry logic, the entire call reverts and the tokens remain stranded in the DDA.

The function is callable by any unprivileged address, either directly or via `ContractOwner.creditDepositV1`:

```solidity
function creditDepositV1(bytes32 subaccount) external {
    ...
    DirectDepositV1(directDepositV1).creditDeposit();
}
``` [3](#0-2) 

The root cause is structurally identical to the reported OUSD bug: a balance value is read at one point in time and then used as an exact transfer amount at a later point, with no tolerance for the token's internal accounting drift.

---

### Impact Explanation

A user who sends a rebasing token to their DDA expecting it to be credited to their subaccount will find that every call to `creditDeposit` reverts. Their tokens are locked in the DDA contract and cannot be deposited into the protocol. Recovery requires the privileged owner to call `withdrawFromDirectDepositV1`, which is `onlyOwner`:

```solidity
function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
    external
    onlyOwner
{
    ...
    uint256 preBalance = IERC20Base(token).balanceOf(address(this));
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
    uint256 postBalance = IERC20Base(token).balanceOf(address(this));
    require(postBalance > preBalance, "empty");
    IERC20Base(token).safeTransfer(msg.sender, postBalance - preBalance);
}
``` [4](#0-3) 

Until the owner acts, the user's funds are inaccessible to the protocol. If the owner is slow to respond or the DDA is used by many users, this constitutes a meaningful, user-facing asset lock.

**Impact: 3** — Funds are locked and inaccessible to the user; recovery requires privileged intervention but no permanent loss occurs if the owner acts.

---

### Likelihood Explanation

The `DirectDepositV1` system is designed to accept any token registered in the `SpotEngine`. If any registered spot product uses a rebasing token as its underlying collateral, every DDA for that product is vulnerable. The trigger (a rebase event between `balanceOf` and `transferFrom`) is an ordinary protocol event for rebasing tokens and requires no attacker action — it occurs naturally.

**Likelihood: 3** — Conditional on a rebasing token being registered as a spot product, but no attacker action is required once that condition holds.

---

### Recommendation

Replace the exact-snapshot pattern with a balance-difference approach that measures what was actually received:

```solidity
uint256 balanceBefore = token.balanceOf(address(endpoint_or_clearinghouse));
token.approve(address(endpoint), type(uint256).max); // or a safe upper bound
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
uint256 actualReceived = token.balanceOf(address(endpoint_or_clearinghouse)) - balanceBefore;
```

Alternatively, use `type(uint256).max` as the approval and pass the actual post-transfer balance delta as the credited amount, rather than the pre-transfer snapshot. At minimum, add a try/catch around the `depositCollateralWithReferral` call so that a single rebasing token does not block crediting of other tokens in the loop.

---

### Proof of Concept

1. A rebasing token (e.g., OUSD-like) is registered as a spot product in `SpotEngine`.
2. A user sends 100 tokens to their `DirectDepositV1` DDA.
3. A rebase event occurs, reducing the DDA's balance to 99.999...e18.
4. Anyone calls `ContractOwner.creditDepositV1(subaccount)`.
5. `creditDeposit` reads `balance = 100e18` (pre-rebase snapshot still cached in the call frame), approves 100e18, and calls `depositCollateralWithReferral` with `amount = 100e18`.
6. `handleDepositTransfer` calls `token.safeTransferFrom(dda, clearinghouse, 100e18)`.
7. The DDA only holds 99.999...e18 — `safeTransferFrom` reverts.
8. The entire `creditDeposit` call reverts; tokens remain in the DDA.
9. Every subsequent call to `creditDeposit` repeats steps 3–8 as long as the rebase persists.
10. The user's tokens are inaccessible to the protocol until the owner calls `withdrawFromDirectDepositV1`. [1](#0-0) [3](#0-2)
