### Title
Replacing `withdrawPool` Without Checking Existing State Orphans Pending User Withdrawals — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.setWithdrawPool` allows the owner to overwrite the `withdrawPool` address with no guard against replacing an already-active pool. This is a direct structural analog to the MechMarketplace balance-tracker replacement bug: existing user claims and liquidity recorded in the old `WithdrawPool` instance become permanently inaccessible through the protocol's normal paths once the pointer is swapped.

---

### Finding Description

`Clearinghouse.sol` stores a single `withdrawPool` address and exposes an owner-callable setter:

```solidity
// Clearinghouse.sol L750-753
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));
    withdrawPool = _withdrawPool;
}
```

The only guard is a zero-address check on the *new* value. There is no check that `withdrawPool` is currently `address(0)` (i.e., that no pool is already active). [1](#0-0) 

The `withdrawPool` field is declared in `ClearinghouseStorage` and is the single authoritative pointer used by the clearinghouse for all withdrawal routing: [2](#0-1) 

The `ContractOwner.removeWithdrawPoolLiquidity` helper also resolves the pool dynamically via `clearinghouse.getWithdrawPool()`, meaning after a swap it will operate on the *new* pool, not the old one: [3](#0-2) 

When the pointer is replaced:

1. **Pending fast-withdrawal claims** recorded in the old `WithdrawPool` instance are no longer reachable through the clearinghouse. Users who initiated a fast withdrawal against the old pool cannot complete their claim via the new pool because the new pool holds no record of their request and holds none of their underlying tokens.
2. **Liquidity provider balances** deposited into the old pool are stranded. The `removeWithdrawPoolLiquidity` admin helper now targets the new pool, so the owner cannot use the standard path to drain the old pool on behalf of LPs.
3. **Token custody mismatch**: the old pool still holds real ERC-20 tokens (the fast-withdrawal float), but the clearinghouse no longer routes any activity to it, creating a permanent accounting desynchronization between on-chain token balances and protocol-tracked state.

---

### Impact Explanation

Users who have an in-flight fast withdrawal against the old pool lose the ability to claim their funds through any supported protocol entrypoint. Their tokens remain locked in the old pool contract with no protocol-level recovery path. Liquidity providers in the old pool face the same stranding. The severity scales with the total value of pending claims and LP deposits at the time of the swap.

---

### Likelihood Explanation

This requires the owner to call `setWithdrawPool` while the existing pool has active state — a realistic operational scenario (e.g., upgrading the pool contract, migrating to a new implementation). The absence of any guard makes it trivially easy to trigger accidentally. The external report's analog was acknowledged as a real design-level risk by the Valory team under identical conditions.

---

### Recommendation

Add a guard that prevents overwriting an already-active pool without explicit migration:

```solidity
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));
    require(withdrawPool == address(0), "WithdrawPool already set; migrate first");
    withdrawPool = _withdrawPool;
}
```

Alternatively, provide a migration path that drains all pending claims and LP balances from the old pool before the pointer is updated, analogous to the MechMarketplace recommendation to revert if `mapPaymentTypeBalanceTrackers[paymentTypes[i]]` is already non-zero.

---

### Proof of Concept

1. Owner deploys `WithdrawPool_v1` and calls `setWithdrawPool(WithdrawPool_v1)` during initialization.
2. Users deposit liquidity into `WithdrawPool_v1`; some users initiate fast withdrawals, creating pending claim records and transferring tokens into `WithdrawPool_v1`.
3. Owner deploys `WithdrawPool_v2` and calls `setWithdrawPool(WithdrawPool_v2)`. The call succeeds — no revert — because the only check is `_withdrawPool != address(0)`. [1](#0-0) 
4. `clearinghouse.getWithdrawPool()` now returns `WithdrawPool_v2`. [4](#0-3) 
5. Users with pending claims in `WithdrawPool_v1` call the claim function — the clearinghouse routes them to `WithdrawPool_v2`, which has no record of their claims and holds none of their tokens. Claims revert or return zero.
6. `removeWithdrawPoolLiquidity` targets `WithdrawPool_v2`, so LP funds in `WithdrawPool_v1` are permanently stranded with no protocol-level recovery path. [3](#0-2)
