### Title
Silent Catch in `_executeSlowModeTransaction()` Permanently Locks Deposited Collateral — (`File: core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint._executeSlowModeTransaction()` wraps `processSlowModeTransaction` in a `try/catch` that silently discards failures. Because collateral is transferred to the `Clearinghouse` **before** the slow-mode tx is queued, any failure inside `processSlowModeTransaction` (e.g., a delisted product) leaves the user's tokens permanently locked in the `Clearinghouse` with no balance credit and no recovery path. The in-code comment `// try return funds now removed` confirms a refund mechanism was deliberately deleted.

---

### Finding Description

**Token transfer happens before queuing.**
`depositCollateralWithReferral()` calls `handleDepositTransfer()`, which pulls tokens from the user and forwards them to the `Clearinghouse` immediately:

```solidity
// Endpoint.sol lines 144-148
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
```

`handleDepositTransfer` (EndpointStorage.sol lines 111-119) does:
```solidity
safeTransferFrom(token, from, amount);          // user → Endpoint
safeTransferTo(token, address(clearinghouse), amount); // Endpoint → Clearinghouse
```

Only after the transfer succeeds is a `SlowModeTx` queued. [1](#0-0) 

**Slow-mode tx is deleted before processing.**
`_executeSlowModeTransaction()` deletes the queued tx from storage before attempting to execute it:

```solidity
// Endpoint.sol line 194
delete slowModeTxs[_slowModeConfig.txUpTo++];
``` [2](#0-1) 

**The catch block does nothing — and the comment admits it.**
```solidity
// Endpoint.sol lines 207-227
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed
}
```

The comment `// try return funds now removed` is direct evidence that a refund path previously existed and was deleted. On any revert inside `processSlowModeTransaction`, the catch block does nothing: the slow-mode tx is already gone from the queue, and the tokens are already in the `Clearinghouse` with no subaccount credit. [3](#0-2) 

**What causes `processSlowModeTransaction` to revert for `DepositCollateral`?**
Inside `processSlowModeTransactionImpl`, the critical call is `clearinghouse.depositCollateral(txn)`:

```solidity
// EndpointTx.sol lines 209-216
if (txType == IEndpoint.TransactionType.DepositCollateral) {
    IEndpoint.DepositCollateral memory txn = abi.decode(...);
    validateSender(txn.sender, sender);
    _recordSubaccount(txn.sender);
    clearinghouse.depositCollateral(txn);   // ← can revert
}
``` [4](#0-3) 

This call reverts if the `productId` has been delisted between the time the user deposited (up to 3 days earlier, per `SLOW_MODE_TX_DELAY`) and the time the sequencer processes the tx. The same silent-catch path applies to `DepositInsurance` slow-mode txs, which also pre-transfer tokens before queuing. [5](#0-4) 

---

### Impact Explanation

**Impact: Medium.**
A user's deposited collateral tokens are permanently locked inside the `Clearinghouse` contract. No subaccount balance is credited, the slow-mode tx is deleted from the queue, and there is no admin or user-callable recovery function. The asset loss is total for the affected deposit amount. The `Clearinghouse` holds the tokens but has no accounting entry for them, so they cannot be withdrawn or recovered through any supported protocol path.

---

### Likelihood Explanation

**Likelihood: Medium.**
The `SLOW_MODE_TX_DELAY` is hardcoded to 3 days. During that window, a product can be delisted via a `DelistProduct` slow-mode tx (owner-only, but a normal protocol operation). Any user who deposited into a product that is subsequently delisted before their slow-mode tx is processed will lose their funds silently. This is a realistic operational scenario, not a contrived attack. Additionally, any future revert condition added to `clearinghouse.depositCollateral` would trigger the same loss.

---

### Recommendation

Restore a fund-return mechanism in the catch block of `_executeSlowModeTransaction()`. When `processSlowModeTransaction` fails for a `DepositCollateral` or `DepositInsurance` tx, the catch block should decode the transaction, identify the token and amount, and return the funds to the original sender. Alternatively, failed slow-mode txs should be re-queued or stored in a recoverable mapping so users can reclaim their tokens.

---

### Proof of Concept

1. Product `productId = 5` (e.g., WBTC spot) is live.
2. Alice calls `depositCollateral(subaccountName, 5, 1000e6)`.
   - `handleDepositTransfer` pulls 1000 USDC from Alice and sends it to `Clearinghouse`. [6](#0-5) 
   - A `SlowModeTx` with `TransactionType.DepositCollateral` is queued at index `N`. [7](#0-6) 
3. Within 3 days, the owner submits a `DelistProduct` slow-mode tx for product 5, which is processed by the sequencer.
4. After 3 days, the sequencer (or anyone) calls `executeSlowModeTransaction()` for Alice's tx at index `N`.
   - `delete slowModeTxs[N]` removes the tx from the queue. [8](#0-7) 
   - `processSlowModeTransaction` → `clearinghouse.depositCollateral(txn)` reverts because product 5 is delisted.
   - The catch block fires, checks gas, and exits silently. [9](#0-8) 
5. Alice's 1000 USDC is in the `Clearinghouse` with no balance credit, no queue entry, and no recovery path. The funds are permanently lost.
