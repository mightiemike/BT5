### Title
`verify_transaction_inclusion` Incorrectly Rejects Valid SPV Proofs for Single-Transaction Blocks — (`contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` function contains an overly restrictive guard that panics when the Merkle proof is empty. In Bitcoin, a block containing exactly one transaction (the coinbase) has a Merkle tree of depth zero, making an empty proof the only correct and valid proof. This guard blocks all SPV verification for that legitimate class of blocks. The same defect propagates into `verify_transaction_inclusion_v2`, which calls the deprecated function internally.

---

### Finding Description

In `contract/src/lib.rs` at line 315, `verify_transaction_inclusion` unconditionally rejects any call where `merkle_proof` is empty:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

In Bitcoin's Merkle tree construction, when a block contains exactly one transaction (always the coinbase), the Merkle root equals the transaction hash directly. No sibling hashes are needed to reconstruct the root, so the correct and only valid Merkle proof for that transaction is an empty list. The downstream `compute_root_from_merkle_proof` call would handle this correctly by returning the transaction hash unchanged — but it is never reached because the `require!` panics first. [2](#0-1) 

The same defect is inherited by `verify_transaction_inclusion_v2` (lines 347–368). That function:
1. Checks `merkle_proof.len() == coinbase_merkle_proof.len()` — passes when both are `0`.
2. Validates the coinbase Merkle proof against the block's `merkle_root` — passes for a single-transaction block because `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id`, which equals `merkle_root`.
3. Calls `#[allow(deprecated)] self.verify_transaction_inclusion(args.into())` — panics at the empty-proof guard. [3](#0-2) 

The structural parallel to the reported vulnerability is exact: just as `QodaToken._update` added `require(to != address(0))` that blocked the legitimate burn path through `super._update`, this contract adds `require!(!args.merkle_proof.is_empty())` that blocks the legitimate single-transaction proof path through `compute_root_from_merkle_proof`.

---

### Impact Explanation

Any unprivileged NEAR caller invoking `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a Bitcoin block that contains only a coinbase transaction receives a contract panic (`"Merkle proof is empty"`) instead of the correct `true` result. Downstream contracts or off-chain systems that rely on this function to gate actions (e.g., bridging, settlement, proof-of-payment) will incorrectly treat a provably valid Bitcoin transaction as unverifiable. The corrupted result is the SPV proof outcome: a valid inclusion proof is permanently rejected.

---

### Likelihood Explanation

Single-transaction blocks are uncommon but entirely valid and occur in production Bitcoin: early mainnet blocks (heights 0–~170), blocks mined during low-fee periods, and any block where a miner includes no mempool transactions. A relayer that submits such a block header to the contract will cause all subsequent SPV verification calls against that block to fail. The entry path requires no privilege — `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are both public, `#[pause]`-gated (not role-gated) functions callable by any NEAR account. [4](#0-3) [5](#0-4) 

---

### Recommendation

Remove the blanket rejection of empty Merkle proofs. The correct fix is to allow `merkle_proof` to be empty and let `compute_root_from_merkle_proof` handle the single-transaction case, which it already does correctly (returning the transaction hash directly). If a non-empty proof is still desired as a sanity check, the guard must be conditioned on the block's transaction count, not on the proof length alone.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. A relayer submits a valid Bitcoin block header for a block containing only the coinbase transaction (e.g., Bitcoin mainnet block 1, height 1). The header is accepted and stored normally.
2. An unprivileged NEAR caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id` = coinbase txid
   - `tx_block_blockhash` = hash of the single-transaction block
   - `tx_index` = 0
   - `merkle_proof` = `[]` (empty — the only correct proof)
   - `coinbase_tx_id` = same coinbase txid
   - `coinbase_merkle_proof` = `[]` (empty)
   - `confirmations` = 1
3. The length check passes (`0 == 0`). The coinbase proof check passes (`coinbase_tx_id == merkle_root`). The call delegates to `verify_transaction_inclusion`.
4. At line 315, `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` panics.
5. The caller receives a contract abort instead of `true`, permanently blocking SPV verification for this valid block. [6](#0-5)
