### Title
Gas-fee domain mismatch in `promise_batch_action_transfer_to_gas_key` host function: raw borsh wire length used instead of on-trie `trie_id_len()` for ML-DSA-65 keys — (File: `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The `promise_batch_action_transfer_to_gas_key` host function (and the sibling add-gas-key host functions) compute both the send fee and the exec fee using `public_key_len` — the raw borsh wire length of the public key as supplied by the WASM contract — instead of the decoded key's `trie_id_len()`. For ML-DSA-65 post-quantum keys the borsh wire length is **1953 bytes** while the on-trie identifier is only **33 bytes** (a SHA3-256 hash). The transaction-level path for the same action correctly calls `action.public_key.trie_id_len()`. The divergence is ~59× on the per-byte fee component and is reachable by any unprivileged WASM contract once `PostQuantumSignatures` (protocol version 85) and `GasKeys` (protocol version 85) are both active.

---

### Finding Description

**Transaction path (correct):**

In `runtime/runtime/src/config.rs`, `total_send_fees` dispatches `TransferToGasKey` as:

```rust
TransferToGasKey(action) => {
    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.trie_id_len())
        .total()
}
```

`trie_id_len()` for `PublicKey::MLDSA65` returns `1 + ML_DSA_65_HASH_LENGTH = 33`. [1](#0-0) 

**Host-function path (wrong):**

In `runtime/near-vm-runner/src/logic/logic.rs`, `promise_batch_action_transfer_to_gas_key` does:

```rust
let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
// ...
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← raw borsh wire length, not trie_id_len()
);
```

`public_key_len` here is the borsh-encoded length of the key as written into WASM memory by the contract. For `PublicKey::MLDSA65` that is `1 + 1952 = 1953` bytes. [2](#0-1) 

The same pattern appears in the wasmtime backend: [3](#0-2) 

And in both add-gas-key host functions (`promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`), which pass `public_key_len as usize` to `gas_key_add_key_exec_fee`: [4](#0-3) [5](#0-4) 

**Fee functions affected:**

`gas_key_transfer_send_fee` multiplies `gas_key_byte` by `public_key_len`: [6](#0-5) 

`gas_key_transfer_exec_fee` feeds `public_key_len` into `access_key_key_len`, which computes the trie-key length used to price the receiver-side trie read/write: [7](#0-6) 

`gas_key_add_key_exec_fee` similarly feeds `public_key_len` into `access_key_key_len` for each nonce entry: [8](#0-7) 

**The invariant that is broken:**

The documentation and the runtime explicitly state that all storage-fee paths must use `trie_id_len()` rather than `len()` for ML-DSA-65 keys, because the trie stores a 33-byte SHA3-256 hash, not the 1953-byte full key: [9](#0-8) 

`PublicKey::trie_id_len()` for `MLDSA65` returns `1 + ML_DSA_65_HASH_LENGTH = 33`: [10](#0-9) 

The runtime unit test confirms the expected 33-byte trie id: [11](#0-10) 

**Exact divergent values for ML-DSA-65:**

| Path | Value passed as `public_key_len` | Per-byte fee multiplier |
|---|---|---|
| Transaction (`total_send_fees`) | `trie_id_len()` = **33** | 33 × `gas_key_byte` |
| Host function (`promise_batch_action_transfer_to_gas_key`) | borsh wire length = **1953** | 1953 × `gas_key_byte` |

Ratio: **1953 / 33 ≈ 59×** overcharge on the per-byte component.

---

### Impact Explanation

Any WASM contract that calls `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key will be charged ~59× more gas for the per-byte component than the equivalent transaction-level action. Concretely:

1. **Out-of-gas failures**: A contract that budgets gas based on the documented/transaction-level pricing will exhaust its gas allowance and abort, even though the underlying trie operation is identical to the transaction path.
2. **Protocol-level fee inconsistency**: The same logical action (fund a gas key) costs dramatically different amounts of gas depending on whether it is submitted as a transaction or dispatched from a contract. This breaks the protocol invariant that gas fees reflect actual resource consumption.
3. **Locked funds**: If a contract's cross-contract call to fund a gas key fails due to out-of-gas, any attached deposit is refunded to the predecessor but the gas is burnt — a net loss to the caller.

---

### Likelihood Explanation

Both `GasKeys` and `PostQuantumSignatures` activate at protocol version 85, which is below the current stable version 86. The host functions are callable by any deployed WASM contract without special permissions. A contract author who follows the transaction-level gas model (or the documentation) will encounter the overcharge the first time they use an ML-DSA-65 key with a gas key host function. The trigger requires only:

1. Protocol version ≥ 85 (already satisfied on mainnet/testnet at stable v86).
2. A WASM contract that calls one of the three affected host functions.
3. An ML-DSA-65 public key passed as the argument.

---

### Recommendation

In all three host functions (`promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`) and their wasmtime counterparts, decode the public key first and use `decoded_key.trie_id_len()` instead of `public_key_len as usize` when calling the fee helpers:

```rust
// Before (wrong for ML-DSA-65):
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(&self.fees_config, receiver_id.len(), public_key_len as usize);

// After (correct):
let decoded_key = public_key.decode()?;
let key_trie_len = decoded_key.trie_id_len();
let send = gas_key_transfer_send_fee(&self.fees_config, sir, key_trie_len);
let exec = gas_key_transfer_exec_fee(&self.fees_config, receiver_id.len(), key_trie_len);
```

The same fix applies to `gas_key_add_key_exec_fee` calls in the add-gas-key host functions. This aligns the host-function path with the transaction path, which already correctly calls `action.public_key.trie_id_len()`.

### Proof of Concept

**Step 1 — Establish the divergence numerically.**

For an ML-DSA-65 key:
- `PublicKey::MLDSA65(_).len()` = 1 + 1952 = **1953** (borsh wire form, tag 2 + raw key)
- `PublicKey::MLDSA65(_).trie_id_len()` = 1 + 32 = **33** (tag 3 + SHA3-256 hash)

`gas_key_transfer_send_fee` with `public_key_len = 1953` charges `1953 × gas_key_byte_send` in per-byte fees; with `public_key_len = 33` it charges `33 × gas_key_byte_send`. The ratio is 59.18×.

**Step 2 — Show the transaction path uses 33.**

`total_send_fees` in `runtime/runtime/src/config.rs` line 115 passes `action.public_key.trie_id_len()` = 33 to `gas_key_transfer_send_fee`.

**Step 3 — Show the host-function path uses 1953.**

`promise_batch_action_transfer_to_gas_key` in `runtime/near-vm-runner/src/logic/logic.rs` lines 3091–3096 passes `public_key_len as usize` = 1953 to both `gas_key_transfer_send_fee` and `gas_key_transfer_exec_fee`.

**Step 4 — Trigger.**

Deploy a contract that calls `promise_batch_action_transfer_to_gas_key` with a borsh-encoded ML-DSA-65 public key (1953 bytes). Observe that `gas_burnt` on the resulting receipt is ~59× higher on the per-byte component than the equivalent `TransferToGasKey` transaction action for the same key. The contract will run out of gas if it was budgeted using the transaction-level pricing.

```rust
pub fn gas_key_transfer_send_fee(
```

```rust
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
```

```rust
pub fn gas_key_add_key_exec_fee(

) -> GasKeyAddFee {

        .checked_mul(num_nonces)
        .unwrap();
```
