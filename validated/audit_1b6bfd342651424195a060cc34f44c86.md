### Title
`ValidResourceBounds` Variant Destroyed by Protobuf Zero-Heuristic Causes Wrong Transaction Hash and Wrong RPC/Execution Values — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion in the P2P sync protobuf layer uses a zero-value heuristic to reconstruct the `ValidResourceBounds` enum variant. A valid `AllResources` Invoke V3 transaction whose `l2_gas` and `l1_data_gas` are both zero is silently downgraded to `L1Gas` after a protobuf round-trip. Because `get_tip_resource_bounds_hash` produces structurally different hash preimages for the two variants (two vs. three resource elements), the transaction hash re-computed from the deserialized object diverges from the hash that was originally signed and stored. Any node that syncs blocks via P2P and then re-computes or validates the hash — or uses the stored object for fee estimation, simulation, or tracing — operates on a wrong value.

---

### Finding Description

**Root cause — the heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`:** [1](#0-0) 

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The protobuf wire format carries three numeric fields (`l1_gas`, `l2_gas`, `l1_data_gas`) but no tag for the original Rust enum variant. The deserializer infers the variant from whether the latter two fields are zero. A transaction originally constructed as `AllResources` with `l2_gas = {max_amount: 0, max_price_per_unit: 0}` and `l1_data_gas = {max_amount: 0, max_price_per_unit: 0}` — which is a valid, gateway-accepted transaction — is reconstructed as `L1Gas` on the receiving side.

**Why the variant matters — `get_tip_resource_bounds_hash` produces different preimages:** [2](#0-1) 

```rust
let mut resource_felts = vec![
    get_concat_resource(&l1_resource_bounds, L1_GAS)?,
    get_concat_resource(&l2_resource_bounds, L2_GAS)?,
];
// For new V3 txs, need to also hash the data gas bounds.
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // ← 2 elements
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // ← 3 elements
    }
});
```

- `AllResources` with zero l2/data gas → Poseidon over `[tip, concat(l1_gas, L1_GAS), concat(0, L2_GAS), concat(0, L1_DATA_GAS)]`
- `L1Gas` → Poseidon over `[tip, concat(l1_gas, L1_GAS), concat(0, L2_GAS)]`

These are **different field counts** fed into the Poseidon hash, producing different digests. The original hash H₁ (computed with `AllResources`) and the re-computed hash H₂ (computed with `L1Gas`) are unequal.

**The P2P block sync path that carries `InvokeTransactionV3` (not `InternalRpcInvokeTransactionV3`):** [3](#0-2) 

The P2P sync server serializes `FullTransaction` (which contains `InvokeTransactionV3` with `ValidResourceBounds`) to protobuf. The client deserializes it back via the heuristic above. The `transaction_hash` field is transmitted alongside the transaction data and stored as-is (H₁), but the stored `InvokeTransactionV3` now carries `L1Gas` instead of `AllResources`. [4](#0-3) 

`InvokeTransactionV3::calculate_transaction_hash` calls `get_invoke_transaction_v3_hash(self, ...)`, which calls `get_tip_resource_bounds_hash(&transaction.resource_bounds(), ...)`. With the wrong variant stored, any re-computation of the hash from the stored object yields H₂ ≠ H₁.

**The `InternalRpcInvokeTransactionV3` path is NOT affected** because it always stores `AllResourceBounds` (not `ValidResourceBounds`) and its `InvokeTransactionV3Trait` impl always wraps it in `AllResources`: [5](#0-4) 

The vulnerability is specific to the `InvokeTransactionV3` (storage/blockifier type) path used in P2P block sync.

**The `ValidResourceBounds` variant also controls the gas vector computation mode:** [6](#0-5) 

```rust
pub fn get_gas_vector_computation_mode(&self) -> GasVectorComputationMode {
    match self {
        Self::AllResources(_) => GasVectorComputationMode::All,
        Self::L1Gas(_) => GasVectorComputationMode::NoL2Gas,
    }
}
```

A node that re-executes or simulates the transaction (e.g., via `starknet_simulateTransactions` or `starknet_estimateFee`) using the stored object would apply `NoL2Gas` fee logic instead of `All`, producing wrong fee estimates and wrong execution results.

---

### Impact Explanation

**High — RPC returns authoritative-looking wrong values:**

1. `starknet_getTransactionByHash` returns `L1Gas` resource bounds for a transaction that was signed and committed as `AllResources`. The returned object is structurally wrong.
2. `validate_transaction_hash` re-computes H₂ from the stored `InvokeTransactionV3` and returns `false` for a valid transaction whose stored hash is H₁.
3. `starknet_estimateFee` / `starknet_simulateTransactions` use the stored object with `L1Gas` variant, applying `NoL2Gas` computation mode and producing wrong fee/gas values.

**High — Transaction conversion binds the wrong executable payload:**

Any path that converts the stored `InvokeTransactionV3` to an executable blockifier transaction inherits the wrong `ValidResourceBounds` variant, causing wrong fee enforcement and wrong gas accounting during re-execution.

---

### Likelihood Explanation

The trigger condition — a V3 Invoke transaction with `AllResources` where both `l2_gas` and `l1_data_gas` are zero — is valid and accepted by the gateway: [7](#0-6) 

The gateway explicitly tests and accepts transactions with only `l1_gas` set. Any user who submits such a transaction (e.g., a legacy-style V3 transaction that only specifies L1 gas) triggers the condition. The condition is unprivileged and requires no special access.

---

### Recommendation

1. **Preserve the variant tag in the protobuf schema.** Add a boolean or enum field (e.g., `is_all_resources`) to `protobuf::ResourceBounds` so the deserializer can reconstruct the exact variant without inference.

2. **Alternatively, always serialize/deserialize as `AllResources` for `InvokeTransactionV3`.** Since the P2P sync path only carries post-0.13.3 transactions, the `L1Gas` variant is a legacy artifact. The serializer can always emit three resource fields and the deserializer can always produce `AllResources`, matching the behavior of the `AllResourceBounds` converter used in the RPC path: [8](#0-7) 

3. **Add a round-trip test** for `InvokeTransactionV3` with `AllResources` where `l2_gas = 0` and `l1_data_gas = 0`, asserting that the variant is preserved and the hash is identical before and after protobuf serialization.

---

### Proof of Concept

```
1. Construct a valid V3 Invoke transaction with:
   resource_bounds = AllResources {
       l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
   }

2. Submit to gateway → accepted. Gateway computes hash H₁ using AllResources
   (get_tip_resource_bounds_hash includes L1_DATA_GAS term → 3-element Poseidon input).

3. Transaction is included in a block. P2P sync server serializes it via
   From<InvokeTransactionV3> for protobuf::InvokeV3 → l2_gas = 0, l1_data_gas = 0.

4. P2P sync client deserializes via TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
   l1_data_gas.is_zero() && l2_gas.is_zero() == true → produces L1Gas(l1_gas).

5. Client stores InvokeTransactionV3 { resource_bounds: L1Gas(...), ... } with hash H₁.

6. Re-compute hash from stored object:
   get_tip_resource_bounds_hash(L1Gas) → 2-element Poseidon input → hash H₂ ≠ H₁.

7. validate_transaction_hash(stored_tx, ..., H₁) → false (H₂ not in possible_hashes).
   starknet_getTransactionByHash → returns L1Gas resource bounds (wrong).
   starknet_estimateFee on stored tx → uses NoL2Gas mode (wrong fee).
```

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));

        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));

        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;

            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
```

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```
