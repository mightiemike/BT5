### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Validation Enforced by `verify_transaction_inclusion_v2` — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two transaction-inclusion verification entry points with asymmetric security requirements. `verify_transaction_inclusion_v2` enforces a coinbase Merkle proof check to prevent the 64-byte transaction forgery attack, but the original `verify_transaction_inclusion` (v1) — which lacks this check — remains fully public and callable by any unprivileged NEAR account. Any caller can bypass the coinbase proof requirement by invoking v1 directly.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability (documented in the code's own comments referencing the BitMEX disclosure). It enforces two additional invariants over v1:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — proof-length parity check.
2. The coinbase transaction's Merkle proof must reconstruct the block's `merkle_root` at index 0 before the target transaction proof is evaluated. [1](#0-0) 

v1, however, performs neither of these checks. It only verifies that the supplied `tx_id` hashes up to the stored `merkle_root` via the caller-supplied proof path: [2](#0-1) 

Critically, v1 carries only a `#[deprecated]` Rust attribute — a compiler lint, not an access-control enforcement. The function remains `pub`, is decorated with `#[pause]` (pausable by `PauseManager` but otherwise open), and has no role guard. Any NEAR account can call it directly. [3](#0-2) 

The `From<ProofArgsV2> for ProofArgs` conversion used inside v2 silently drops `coinbase_tx_id` and `coinbase_merkle_proof` when delegating to v1, confirming that v1 never sees coinbase data regardless of the call path: [4](#0-3) 

---

### Impact Explanation

The 64-byte transaction attack allows an adversary to craft a 64-byte blob that is simultaneously a valid serialized internal Merkle tree node and a plausible transaction hash. By supplying this blob as `tx_id` with a carefully constructed `merkle_proof`, the attacker can make `compute_root_from_merkle_proof` return the correct `merkle_root` for a real mainchain block without the blob corresponding to any real transaction. v1 will return `true`.

Any recipient contract that calls `verify_transaction_inclusion` (v1) — or that accepts a boolean result from either verification function without distinguishing which was called — can be deceived into treating a non-existent Bitcoin transaction as confirmed. This is a proof-verification forgery with direct protocol impact: cross-chain bridges, custody contracts, or settlement layers consuming this result would release funds or update state based on a fabricated proof.

---

### Likelihood Explanation

The entry point is fully public and requires no privileged role, staked deposit, or special account. The 64-byte transaction attack is a well-documented technique. The attacker only needs to:

1. Identify a real mainchain block stored in the contract.
2. Construct a 64-byte internal-node blob whose Merkle path reconstructs that block's `merkle_root`.
3. Call `verify_transaction_inclusion` directly with `tx_id` = the crafted blob.

The deprecated label provides no on-chain barrier. Likelihood is **medium-high** for any deployment where a downstream contract consumes v1's result.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public interface entirely, or gate it behind a role that no external caller holds (e.g., `Role::DAO`). The `#[deprecated]` attribute must be accompanied by an access-control guard to have any security effect on-chain. Alternatively, have v1 unconditionally delegate to v2's logic so the coinbase check cannot be bypassed regardless of which entry point is called.

---

### Proof of Concept

1. Deploy the contract (Bitcoin feature, `skip_pow_verification = false`).
2. Submit a sequence of real mainchain headers so that block `B` at height `H` is in `mainchain_header_to_height`.
3. Construct a 64-byte blob `fake_tx` such that `compute_root_from_merkle_proof(fake_tx, idx, crafted_proof) == B.merkle_root` (standard 64-byte Merkle forgery construction).
4. Call `verify_transaction_inclusion` with `tx_id = fake_tx`, `tx_block_blockhash = B.hash`, `tx_index = idx`, `merkle_proof = crafted_proof`, `confirmations = 1`.
5. The function returns `true` despite `fake_tx` not being a real transaction in block `B`.
6. Calling `verify_transaction_inclusion_v2` with the same `tx_id` and no valid `coinbase_merkle_proof` would panic or return `false`, demonstrating the asymmetry. [5](#0-4) [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
