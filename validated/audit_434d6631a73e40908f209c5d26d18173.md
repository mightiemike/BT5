### Title
Shared Latest `VersionedConstants` Applied to All Blocks in Virtual Block Executor Produces Wrong Execution Results for Historical Transactions - (File: crates/starknet_transaction_prover/src/running/virtual_block_executor.rs)

---

### Summary

`RpcVirtualBlockExecutorConfig::use_latest_versioned_constants` defaults to `true`, causing the virtual block executor to apply the **latest** `VersionedConstants` to every block it re-executes, regardless of the block's actual Starknet version. This is the direct sequencer analog to the external report's shared `navProvider`/`feeManager` applied globally across different asset types: a single shared configuration is applied to all inputs even when different inputs require different parameters. The version-specific flags `block_casm_hash_v1_declares` and `enable_casm_hash_migration` changed from `false` to `true` between v0.14.0 and v0.14.1. Re-executing a v0.14.0 block under v0.14.1+ constants causes DeclareV3 transactions that carried a V1 (Poseidon) compiled class hash — valid and accepted at the time — to fail with `DeclareTransactionCasmHashMissMatch`, and causes the state diff to contain migration entries that were never part of the original committed state.

---

### Finding Description

**Root cause — shared config applied globally:**

`RpcVirtualBlockExecutorConfig` sets `use_latest_versioned_constants` to `true` by default: [1](#0-0) 

The comment explicitly acknowledges the invariant break: *"The OS currently always runs with the latest constants"* and defers a fix with a TODO.

**Version boundary that changes behavior:**

The diff regression file records that both `block_casm_hash_v1_declares` and `enable_casm_hash_migration` flipped from `false` to `true` at v0.14.1: [2](#0-1) 

**Execution gate that diverges:**

In `DeclareTransaction::run_execute()`, when `block_casm_hash_v1_declares = true` and the transaction is V3+, the blockifier calls `check_compile_class_hash_v2_declaration()`, which rejects any compiled class hash that is not a V2 (Blake2) hash: [3](#0-2) 

**Hash algorithm divergence:**

`HashVersion::V1` uses Poseidon; `HashVersion::V2` uses Blake2Felt252. They produce different `CompiledClassHash` values for the same contract: [4](#0-3) 

**Migration state-diff divergence:**

When `enable_casm_hash_migration = true`, `CasmHashMigrationData::from_state()` adds V1→V2 migration entries to the state diff. Under v0.14.0 constants this path is skipped entirely: [5](#0-4) 

**Concrete call path:**

1. Before v0.14.1, a user submits a DeclareV3 transaction whose `compiled_class_hash` is the Poseidon (V1) hash of the CASM. The gateway accepts it; the batcher executes it under v0.14.0 constants (`block_casm_hash_v1_declares = false`). The transaction succeeds and the V1 hash is committed to state.
2. The virtual block executor is asked to re-execute that block for proving. With `use_latest_versioned_constants = true` it loads v0.14.1+ constants.
3. `DeclareTransaction::run_execute()` sees `block_casm_hash_v1_declares = true`, calls `check_compile_class_hash_v2_declaration()`, finds the stored hash is V1, and returns `DeclareTransactionCasmHashMissMatch`.
4. The re-execution produces a **revert** where the original produced **success**, and the state diff diverges (missing the class declaration entry, plus spurious migration entries from `enable_casm_hash_migration = true`).

The test suite confirms the exact divergence: [6](#0-5) 

---

### Impact Explanation

The virtual block executor produces a wrong execution result — a revert with `DeclareTransactionCasmHashMissMatch` — for a transaction that was validly accepted and committed by the original sequencer. The resulting state diff differs from the committed state diff: the class declaration entry is absent and spurious V1→V2 migration entries may be present. Any proof generated from this re-execution attests to an incorrect state transition. This matches: **"Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."**

---

### Likelihood Explanation

Every DeclareV3 transaction submitted before Starknet v0.14.1 that carried a V1 (Poseidon) compiled class hash is a trigger. No privileged access is required; the trigger is any historical transaction already committed to the chain. The `TransactionConverter` now enforces V2 hashes for new transactions, but it does not retroactively change what is stored in state for pre-v0.14.1 blocks. Any proving run over those historical blocks hits this path.
