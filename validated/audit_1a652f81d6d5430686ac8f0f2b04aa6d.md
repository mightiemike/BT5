### Title
`only_query` Flag Silently Dropped in `BroadcastedTransaction → ExecutableTransactionInput` Conversion, Causing Wrong Hash Domain and Wrong Execution-Info Version for Query Transactions in RPC Fee Estimation and Simulation — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

When `starknet_estimateFee` or `starknet_simulateTransactions` receives a query transaction (version `0x100000000000000000000000000000003`), the conversion from `BroadcastedTransaction` to `ExecutableTransactionInput` unconditionally hard-codes `only_query = false`. The query-version bit is never extracted and forwarded. As a result, the transaction hash is computed under the wrong domain (version `0x3` instead of `0x100000000000000000000000000000003`), and the account contract's `get_execution_info` syscall returns the wrong version. When `SkipValidate` is not set, the account's `__validate__` computes a hash over the wrong version, mismatches the user's signature, and the RPC returns an authoritative-looking validation-failure result instead of a correct fee estimate or simulation trace.

---

### Finding Description

Starknet defines a "query-only" transaction variant whose version has bit 128 set (e.g., `0x100000000000000000000000000000003` for V3). This bit is carried in the RPC type `TransactionVersion3::Version3OnlyQuery` and is supposed to be forwarded as the `only_query: bool` field of `ExecutableTransactionInput`. The `only_query` flag has two downstream effects:

1. **Hash domain**: `calc_tx_hash` calls `get_transaction_hash(tx, chain_id, &TransactionOptions { only_query })`. Inside, `signed_tx_version_from_tx` sets bit 128 of the version field that enters the Poseidon hash chain when `only_query = true`. A query transaction signed by the user was signed over a hash that includes version `0x100000000000000000000000000000003`.

2. **Execution-info version**: `to_blockifier_tx` passes `only_query` into `ExecutionFlags`, which propagates to `get_execution_info` syscall responses. The account contract reads this version to reconstruct the hash during `__validate__`.

The conversion that loses the flag is:

```rust
// crates/apollo_rpc/src/v0_8/api/mod.rs  lines 329-338
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        match value {
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
            BroadcastedTransaction::DeployAccount(tx) => Ok(Self::DeployAccount(tx.into(), false)),
            BroadcastedTransaction::Invoke(tx) => Ok(Self::Invoke(tx.into(), false)),
        }
    }
}
```

And is acknowledged but unresolved in `to_blockifier_tx`:

The hash calculation path that uses the flag:

```rust

    match self.apply_on_transaction(|tx, only_query| {
        get_transaction_hash(tx, chain_id, &TransactionOptions { only_query })
    }) { ... }
}
```

`get_transaction_hash` calls `signed_tx_version_from_tx`, which sets bit 128 only when `only_query = true`. Because `only_query` is always `false`, the hash is always computed over version `0x3`, regardless of what the user submitted.

---

### Impact Explanation

**High** — `starknet_estimateFee` and `starknet_simulateTransactions` return authoritative-looking wrong values for any query transaction submitted without `SkipValidate`. Concretely:

- The account contract's `__validate__` receives version `0x3` via `get_execution_info` and computes a hash over that version.
- The user signed over version `0x100000000000000000000000000000003`.
- The two hashes differ; ECDSA verification fails.
- The RPC returns a `TRANSACTION_EXECUTION_ERROR` (validation failure) instead of a valid fee estimate or simulation trace.

A caller relying on the RPC to determine whether a query transaction is valid and what fee it would cost receives a false negative. This is an authoritative-looking wrong value from the RPC fee-estimation and simulation endpoints.

---

### Likelihood Explanation

**High** — Any unprivileged caller who submits a V3 query transaction (version `0x100000000000000000000000000000003`) to `starknet_estimateFee` or `starknet_simulateTransactions` without the `SkipValidate` simulation flag triggers the bug unconditionally. No special state or privilege is required. The `TransactionVersion3::Version3OnlyQuery` variant is a first-class part of the RPC schema and is expected to be used by wallets performing dry-run fee estimation.

---

### Recommendation

Derive `only_query` from the transaction version during `BroadcastedTransaction → ExecutableTransactionInput` conversion. Specifically, inspect whether the incoming transaction version has bit 128 set and pass the result as the `only_query` field:

```rust
BroadcastedTransaction::Invoke(tx) => {
    let only_query = tx.version_is_query(); // check bit 128
    Ok(Self::Invoke(tx.into(), only_query))
}
```

The same fix must be applied to `DeployAccount` and all `Declare` variants. The `into()` conversion that maps the RPC transaction to the starknet-api type must strip bit 128 from the stored version (since `InvokeTransactionV3` does not carry a version field; the version is implicit), while the extracted bit is forwarded as `only_query`.

---

### Proof of Concept

1. Construct an Invoke V3 transaction signed over version `0x100000000000000000000000000000003` (query bit set).
2. Submit it to `starknet_estimateFee` (without `SkipValidate`).
3. The conversion at `crates/apollo_rpc/src/v0_8/api/mod.rs:336` sets `only_query = false`.
4. `calc_tx_hash` at `crates/apollo_rpc_execution/src/lib.rs:469-476` computes the hash with `TransactionOptions { only_query: false }`, yielding a hash over version `0x3`.
5. `to_blockifier_tx` at `crates/apollo_rpc_execution/src/lib.rs:815-817` passes `only_query = false` into `ExecutionFlags`.
6. The account's `__validate__` receives version `0x3` from `get_execution_info` and computes the hash over `0x3`.
7. The user's signature was computed over version `0x100000000000000000000000000000003`; the hashes differ; ECDSA fails.
8. `starknet_estimateFee` returns a validation error instead of a fee estimate.

pub fn signed_tx_version(
    tx_version: &TransactionVersion,
    transaction_options: &TransactionOptions,
) -> TransactionVersion {
    // If only_query is true, set the 128-th bit.
    let query_only_bit = *QUERY_VERSION_BASE;
    assert_eq!(
        tx_version.0.to_biguint() & query_only_bit.to_biguint(),
        BigUint::from(0_u8),
        "Requested signed tx version with version that already has query bit set: {tx_version:?}."
    );
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}
