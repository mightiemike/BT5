### Title
Linked Signer Can Overwrite Its Own Entry in `linkedSigners` Mapping, Hijacking Subaccount Signing Authority — (`core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.sol`, the fast-path `LinkSigner` transaction handler calls `validateSignedTx` with `allowLinkedSigner = true`. This means the **currently registered linked signer** for a subaccount is permitted to sign a new `LinkSigner` transaction, overwriting `linkedSigners[subaccount]` with any address of their choosing — without the subaccount owner's knowledge or consent.

---

### Finding Description

The fast-path `LinkSigner` handler in `EndpointTx.sol` is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves to:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

The verifier accepts a valid signature from **either** the subaccount owner **or** the currently registered linked signer. Because the `LinkSigner` transaction itself is processed with `allowLinkedSigner = true`, the existing linked signer can sign a new `LinkSigner` transaction and replace `linkedSigners[subaccount]` with any address.

By contrast, the slow-mode `LinkSigner` path uses `validateSender`, which enforces that `msg.sender` is the actual subaccount owner address — so the slow path is not affected. [3](#0-2) 

There is no guard checking whether `linkedSigners[txn.sender]` is already set, nor any restriction preventing the linked signer from being the signer of a `LinkSigner` mutation.

---

### Impact Explanation

The `linkedSigners` mapping is the trust anchor for all fast-path transaction authorization. `getLinkedSigner` is consulted for every fast-path transaction type that passes `allowLinkedSigner = true`, including `MatchOrders`, `WithdrawCollateral`, `TransferQuote`, `MintNlp`, `BurnNlp`, and `CreateIsolatedSubaccount`. [4](#0-3) 

A compromised or malicious linked signer (e.g., a third-party trading bot or API key service) can:

1. Replace `linkedSigners[victim_subaccount]` with an attacker-controlled address.
2. Use that new address to sign withdrawals, trades, or transfers draining the subaccount.
3. The original owner is locked out of the linked-signer fast path until they submit a slow-mode `LinkSigner` reset — which has a sequencer-enforced delay.

The corrupted state is `linkedSigners[subaccount]`, and the asset delta is the full collateral balance of the victim subaccount.

---

### Likelihood Explanation

Any entity that has ever been granted linked-signer status — including third-party integrations, API key services, or automated market makers — can trigger this unilaterally. No additional privilege, leaked key, or social engineering is required beyond the linked-signer credential already in their possession. The attack is a single sequencer-submitted transaction.

---

### Recommendation

The fast-path `LinkSigner` handler should **not** allow the existing linked signer to authorize a linked-signer change. Change the `allowLinkedSigner` flag to `false` for this specific transaction type:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the slow-mode path, which already enforces owner-only authorization via `validateSender`. [3](#0-2) 

---

### Proof of Concept

1. Alice owns subaccount `0xAlice...` and grants linked-signer status to `0xBot` (a trading service).
2. `linkedSigners[0xAlice...] = 0xBot`.
3. `0xBot` constructs a `SignedLinkSigner` transaction: `sender = 0xAlice...`, `signer = 0xAttacker`.
4. `0xBot` signs the transaction with its own key and submits it to the sequencer.
5. The sequencer calls `processTransaction`, which reaches the fast-path `LinkSigner` branch.
6. `validateSignedTx(..., true)` passes because `0xBot` is the current linked signer for `0xAlice...`.
7. `linkedSigners[0xAlice...] = 0xAttacker` is written.
8. `0xAttacker` now signs a `WithdrawCollateral` transaction for `0xAlice...`; it passes `validateSignedTx(..., true)` and drains Alice's collateral.
9. Alice can only recover by submitting a slow-mode `LinkSigner` reset, but the funds are already gone. [1](#0-0) [5](#0-4)
