### Title
Linked Signer Can Self-Escalate via `LinkSigner` Fast-Mode Transaction, Enabling Subaccount Takeover — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction handler in `EndpointTx.sol` calls `validateSignedTx` with `allowLinkedSigner = true`. This means the **current linked signer** — not just the subaccount owner — can sign a new `LinkSigner` transaction to replace itself with any attacker-controlled address. Once replaced, the attacker-controlled address can sign withdrawal transactions and drain the subaccount. This is a direct analog to the external report's class: a mutable privileged-address slot that can be overwritten to redirect asset control, here triggered by the linked signer itself rather than the owner.

---

### Finding Description

In `EndpointTx.sol`, the fast-mode `LinkSigner` branch is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` delegates to `validateSignature`, which passes the current linked signer as the permitted alternate signer to the verifier:

```solidity
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`getLinkedSigner` resolves the currently registered linked signer for the subaccount (or its parent if isolated): [3](#0-2) 

The `linkedSigners` mapping is stored in `EndpointStorage`: [4](#0-3) 

**Contrast with the slow-mode path**, which correctly restricts `LinkSigner` to the subaccount owner via `validateSender` (checks `msg.sender == address(uint160(bytes20(txn.sender)))`) and additionally calls `requireSubaccount`: [5](#0-4) 

The slow-mode path never passes the linked signer as an authorized alternate — it requires the actual wallet owner to submit the transaction on-chain. The fast-mode path has no such restriction, allowing the linked signer to self-escalate.

**Broken invariant:** The linked signer is a delegated signing key intended for trading operations. It must not be able to mutate the `linkedSigners` slot for its own subaccount. Allowing it to do so gives it the ability to install an arbitrary replacement signer, which can then authorize any subsequent transaction including withdrawals.

---

### Impact Explanation

**Impact: High** — complete loss of subaccount funds.

Attack chain:
1. Victim sets a linked signer (e.g., a third-party trading service or API key).
2. The linked signer constructs and signs a `SignedLinkSigner` transaction where `tx.sender = victim_subaccount` and `tx.signer = attacker_address`.
3. The sequencer processes the transaction; `linkedSigners[victim_subaccount]` is overwritten with `attacker_address`.
4. The attacker signs a `WithdrawCollateral` or `TransferQuote` transaction on behalf of the victim subaccount.
5. The sequencer processes the withdrawal; victim's collateral is transferred out.

The corrupted state delta is `linkedSigners[victim_subaccount]`, which controls who can authorize all subsequent fast-mode transactions for that subaccount, including withdrawals.

---

### Likelihood Explanation

**Likelihood: Low** — requires the linked signer to be malicious or compromised.

Users routinely set linked signers to third-party services, automated bots, or hot wallets for convenience. The attack surface is realistic: a user who trusts a trading service with a linked signer key does not expect that key to be able to permanently replace itself with an attacker-controlled address. The user's mental model is that the linked signer can trade, not that it can mutate the signing authority itself.

---

### Recommendation

Change `allowLinkedSigner` to `false` in the fast-mode `LinkSigner` handler so that only the subaccount owner's key can authorize a linked-signer change:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This aligns the fast-mode path with the slow-mode path's intent, where `validateSender` already enforces that only the wallet owner can submit a `LinkSigner` transaction.

### Proof of Concept

```
Precondition:
  - victim_subaccount = bytes32(address(victim) || subaccount_name)
  - linkedSigners[victim_subaccount] = trading_bot_address  (set by victim)

Step 1 (trading_bot constructs malicious tx):
  tx = SignedLinkSigner {
      tx: LinkSigner {
          sender: victim_subaccount,
          signer: bytes32(attacker_address),
          nonce: current_nonce(victim_address)
      },
      signature: trading_bot_address signs EIP-712 digest of above
  }

Step 2 (sequencer processes tx):
  validateSignedTx(..., allowLinkedSigner=true)
    → getLinkedSigner(victim_subaccount) = trading_bot_address
    → verifier accepts trading_bot_address's signature ✓
  linkedSigners[victim_subaccount] = attacker_address   ← state corrupted

Step 3 (attacker drains subaccount):
  attacker signs WithdrawCollateral for victim_subaccount
  → validateSignedTx(..., allowLinkedSigner=true)
    → getLinkedSigner(victim_subaccount) = attacker_address ✓
  → victim's collateral withdrawn to attacker
``` [6](#0-5)
