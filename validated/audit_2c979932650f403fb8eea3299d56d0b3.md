### Title
GC Over-Eviction Destroys Retarget Interval-Tail Block in Single `submit_blocks` Call — (`contract/src/litecoin.rs`, `contract/src/lib.rs`)

---

### Summary

`submit_blocks` calls `run_mainchain_gc(num_of_headers)` after appending all submitted headers. When a batch of N = `gc_threshold + 1` headers is submitted and the chain is already at capacity, GC removes exactly N blocks from the tail. Because `difficulty_adjustment_interval == gc_threshold == 2016` for Litecoin, the removed tail includes the interval-tail block at height `new_tip − 2016`, which is the exact block `get_next_work_required` fetches via `get_header_by_height` for the immediately following retarget boundary. That lookup panics with `ERR_KEY_NOT_EXIST`, permanently breaking the contract's retarget capability.

---

### Finding Description

**Entry point:** `submit_blocks` — a `#[trusted_relayer]`-gated public method.

**Step 1 — `submit_blocks` appends then GCs:**

```
submit_blocks(N headers)
  → for each header: submit_block_header(...)   // GC not yet run
  → run_mainchain_gc(N)                         // GC runs with batch_size = N
``` [1](#0-0) 

**Step 2 — GC arithmetic with `gc_threshold = 2016`, chain already at capacity:**

```
amount_of_headers_we_store = (H + 4032) − H + 1 = 4033
total_amount_to_remove     = 4033 − 2016 = 2017
selected_amount_to_remove  = min(2017, 2017) = 2017
```

Removes heights `H .. H+2017` (exclusive end), so heights `H` through `H+2016` inclusive are deleted. New `mainchain_initial_blockhash` = block at `H+2017`. [2](#0-1) 

**Step 3 — Next retarget lookup panics:**

For the block at height `H+4033` (prev_height = `H+4032`, `(H+4032+1) % 2016 == 0`):

```
blocks_to_go_back   = 2016
first_block_height  = (H+4032) − 2016 = H+2016
```

Block at `H+2016` was removed by GC (it falls in `H..H+2017`). `get_header_by_height(H+2016)` panics. [3](#0-2) [4](#0-3) 

**Litecoin `difficulty_adjustment_interval` is exactly 2016**, equal to the recommended `gc_threshold`: [5](#0-4) 

There is no guard in `submit_blocks` limiting batch size, and no guard in `run_mainchain_gc` preserving the interval-tail block.

---

### Impact Explanation

After the oversized batch commits, the contract's `mainchain_height_to_header` and `headers_pool` no longer contain the block at `new_tip − 2016`. Every subsequent `submit_blocks` call for a retarget-boundary block will panic and revert. The state mutation from the first call is already committed and irreversible — there is no function to re-insert a historical block. The contract's retarget capability is permanently destroyed without a contract migration.

---

### Likelihood Explanation

The recommended `gc_threshold` in the `init` docstring is `52704` (one year of blocks), which is much larger than `2016`. At that threshold the attack requires a batch of `52704 + 1` headers, which is operationally implausible. However:

- The docstring example and test fixtures use `gc_threshold = 2016` (exactly `difficulty_adjustment_interval`).
- Any deployment with `gc_threshold` set to `difficulty_adjustment_interval` or any value in the range `[difficulty_adjustment_interval, 2 * difficulty_adjustment_interval − 1]` is vulnerable to a batch of size `gc_threshold + 1`.
- The relayer uses adaptive batching for catch-up sync; a large batch is a normal operational event, not an attack-specific action.
- With `skip_pow_verification = true` (common in staging/testnet deployments), the batch requires zero valid PoW.
- With `skip_pow_verification = false`, the retarget blocks *within* the batch are validated against interval-tail blocks that are still present (GC runs after all headers are appended), so the batch still succeeds and the post-GC state is still broken. [6](#0-5) 

---

### Recommendation

`run_mainchain_gc` must never evict a block that is within `difficulty_adjustment_interval` blocks of the current `mainchain_initial_blockhash` after removal. The fix is to enforce a hard floor on the new initial height:

```rust
// In run_mainchain_gc, before computing selected_amount_to_remove:
let config = self.get_config();
let safe_floor = tip_blockheader.block_height
    .saturating_sub(self.gc_threshold)
    .saturating_sub(config.difficulty_adjustment_interval);

let max_removable = initial_blockheader.block_height
    .saturating_sub(safe_floor)
    .min(total_amount_to_remove);
let selected_amount_to_remove = std::cmp::min(max_removable, batch_size);
```

Alternatively, enforce at initialization that `gc_threshold >= 2 * difficulty_adjustment_interval`.

---

### Proof of Concept

```rust
// Litecoin feature, skip_pow_verification = true, gc_threshold = 2016
// Genesis at height 2015 ((2015+1) % 2016 == 0, satisfying init requirement)
// Pre-populate to exactly gc_threshold blocks: heights 2015..=4030

// Step 1: submit 2017 headers (heights 4031..=6047) in one call
contract.submit_blocks(headers_4031_to_6047);  // succeeds; GC removes 2015..=4031

// Step 2: verify the interval-tail is gone
assert!(contract.get_block_hash_by_height(4031).is_none());

// Step 3: attempt to submit the next retarget-boundary block (height 6048, prev=6047)
// get_next_work_required needs block at 6047 - 2016 = 4031 → ERR_KEY_NOT_EXIST
contract.submit_blocks(vec![header_6048]);  // panics: ERR_KEY_NOT_EXIST
```
