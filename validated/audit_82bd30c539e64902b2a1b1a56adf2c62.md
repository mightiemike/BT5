Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Real User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed from the pool, which is the pool's `msg.sender`. When `MetricOmmSimpleRouter` is used, the pool's `msg.sender` is the router contract, not the actual user. A pool admin who allowlists the router to enable curated users to trade through it inadvertently grants swap access to every address on-chain, completely nullifying the allowlist.

## Finding Description
**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim as the first argument of `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

**Step 2 — Router is the pool's `msg.sender`.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly. The actual user (`msg.sender` of `exactInputSingle`) is stored only in transient storage via `_setNextCallbackContext` for the payment callback and is never forwarded to the pool or any extension: [3](#0-2) 

**Step 3 — Extension checks the router, not the user.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router. The actual user's address is never consulted: [4](#0-3) 

**Step 4 — The dilemma that creates the bypass.**

The pool admin faces two mutually exclusive outcomes:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all |
| **Allowlist the router** | Every user on-chain can bypass the allowlist |

There is no configuration that allows only allowlisted users to trade through the router.

**Contrast with `DepositAllowlistExtension`.**

The deposit extension correctly gates on `owner` (the position owner explicitly passed to `addLiquidity`), which the liquidity adder preserves faithfully. The swap extension has no equivalent mechanism to recover the real user identity: [5](#0-4) 

## Impact Explanation
Any user not on the swap allowlist can bypass the restriction on a curated pool by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutputSingle` / `exactOutput`) once the pool admin has allowlisted the router. The attacker receives pool output tokens at the oracle-anchored price, draining liquidity the pool admin intended to reserve for curated counterparties. This is a direct loss of LP assets and a complete failure of the allowlist protection — a High severity broken core pool functionality / direct loss of LP principal.

## Likelihood Explanation
The trigger is a pool admin allowlisting the router — a routine, non-malicious action taken whenever the admin wants their allowlisted users to access the standard periphery. The admin has no way to achieve the intended goal (allow specific users to use the router) without also opening the gate to everyone. Once the router is allowlisted, any unprivileged user can exploit the bypass with a single router call, repeatably and without any special capability.

## Recommendation
The extension must verify the **original user**, not the intermediary. Two viable approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling the pool. The extension decodes and verifies it. This is acceptable since the router is a known, immutable periphery contract.
2. **Separate router-level allowlist**: Deploy a thin wrapper that enforces the user-level allowlist before forwarding to the pool, removing the need for the extension to recover the original caller.

## Proof of Concept
```
Setup:
  pool = deploy MetricOmmPool with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)   // alice is the curated user
  admin calls setAllowedToSwap(pool, router, true)  // admin wants alice to use the router

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

  Execution:
    router calls pool.swap(bob, ...)          // router is msg.sender to pool
    pool calls _beforeSwap(msg.sender=router, ...)
    extension checks allowedSwapper[pool][router] == true  ✓
    swap executes — bob receives tokens despite not being allowlisted
```

    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);

      .swap(

        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
