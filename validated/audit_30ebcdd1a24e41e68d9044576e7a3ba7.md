### Title
Fee-on-Transfer Token Incompatibility in Two-Hop Deposit Transfer Causes Revert and Accounting Inflation — (`core/contracts/EndpointStorage.sol`)

---

### Summary

`handleDepositTransfer` in `EndpointStorage.sol` routes tokens through two sequential `safeTransfer` hops (user → Endpoint → Clearinghouse). For fee-on-transfer (FOT) tokens, the Endpoint receives `amount - fee` from the user but then attempts to forward the full `amount` to the Clearinghouse. This causes an unconditional revert for all FOT token deposits. A parallel two-hop issue exists on the withdrawal path in `Clearinghouse.handleWithdrawTransfer`, where tokens travel Clearinghouse → WithdrawPool → user, double-charging fees and causing the user to receive less than the amount debited from their on-chain balance.

---

### Finding Description

**Deposit path — `EndpointStorage.handleDepositTransfer`:**

```
User ──[amount]──► Endpoint  (receives amount - fee₁)
Endpoint ──[amount]──► Clearinghouse  ← REVERTS: only holds amount - fee₁
``` [1](#0-0) 

`safeTransferFrom` pulls `amount` from the user; the Endpoint receives `amount - fee₁`. The immediately following `safeTransferTo` then attempts to push the original `amount` to the Clearinghouse. Because the Endpoint only holds `amount - fee₁`, this second call reverts unconditionally for any FOT token, blocking all deposits of that asset.

The slow-mode transaction that would credit the subaccount is queued *after* `handleDepositTransfer` returns: [2](#0-1) 

Because the revert propagates, the slow-mode entry is never written and the user's tokens are not transferred either — the deposit is a complete no-op revert. However, if the Endpoint ever holds a residual FOT balance (e.g., from slow-mode fees collected in the same token), the second transfer succeeds but the Clearinghouse receives only `amount - fee₂`. `Clearinghouse.depositCollateral` then credits the full `txn.amount` to the subaccount: [3](#0-2) 

This inflates the subaccount's credited balance above the actual tokens held by the Clearinghouse, corrupting the solvency invariant.

**Withdrawal path — `Clearinghouse.handleWithdrawTransfer`:**

```
Clearinghouse ──[amount]──► WithdrawPool  (receives amount - fee₁)
WithdrawPool  ──[amount]──► user          (receives amount - fee₁ - fee₂)
``` [4](#0-3) 

`submitWithdrawal` in `BaseWithdrawPool` then calls `handleWithdrawTransfer(token, sendTo, amount)` with the original `amount`: [5](#0-4) 

The user receives `amount - fee₁ - fee₂` while the accounting in `withdrawCollateral` deducts exactly `amount` from the subaccount balance: [6](#0-5) 

The protocol permanently loses the double-fee delta on every FOT withdrawal.

---

### Impact Explanation

**Deposit path:**
- **Primary**: Unconditional revert for all FOT token deposits — complete DoS of the deposit entrypoint for those assets.
- **Secondary** (if Endpoint holds residual FOT balance): Subaccount credited with `amount` while Clearinghouse received `amount - fee`. Repeated exploitation inflates total credited balances above actual token reserves, eventually making the protocol insolvent for that asset.

**Withdrawal path:**
- User receives `amount - 2×fee` while the protocol deducts `amount`. Each withdrawal permanently drains the Clearinghouse's FOT token reserve by `2×fee`, leading to insolvency over time.

---

### Likelihood Explanation

Any unprivileged user calling `Endpoint.depositCollateral` or `Endpoint.depositCollateralWithReferral` with a FOT token triggers the deposit revert immediately. No special role or precondition is required beyond the token being a FOT type. The withdrawal path is triggered by any normal withdrawal of a FOT collateral asset. Both paths are reachable by any trader interacting with the protocol's primary public entrypoints. [7](#0-6) [8](#0-7) 

---

### Recommendation

**Deposit**: Replace the two-hop transfer with a direct `safeTransferFrom(token, from, address(clearinghouse), amount)` so the token moves in a single hop. Then measure the actual received amount using a before/after balance check and pass that measured amount — not the caller-supplied `amount` — into the slow-mode `DepositCollateral` transaction.

**Withdrawal**: Replace the two-hop Clearinghouse → WithdrawPool → user path with a single direct transfer from the Clearinghouse to the user (or measure the actual received amount at each hop and propagate the real value through the accounting).

Reference: Uniswap Router02's `swapExactTokensForTokensSupportingFeeOnTransferTokens` pattern, which uses before/after balance snapshots to determine actual received amounts.

---

### Proof of Concept

1. A FOT token (e.g., 1% fee per transfer) is listed as a supported collateral product in `SpotEngine`.
2. User approves `Endpoint` for `1000` tokens and calls `Endpoint.depositCollateral(subaccountName, productId, 1000)`.
3. `depositCollateralWithReferral` calls `handleDepositTransfer(token, msg.sender, 1000)`.
4. `safeTransferFrom(token, user, Endpoint, 1000)` executes — Endpoint receives `990` (1% fee deducted).
5. `safeTransferTo(token, clearinghouse, 1000)` executes — Endpoint only holds `990`, so the ERC-20 `transfer` call reverts with insufficient balance.
6. The entire `depositCollateral` transaction reverts. The user's tokens are returned (no state change), but the deposit is permanently blocked.

For the accounting-inflation variant: if the Endpoint holds ≥10 tokens of residual FOT balance, step 5 succeeds. The Clearinghouse receives `990` but `depositCollateral` credits `1000 * multiplier` to the subaccount — a `10 * multiplier` phantom balance that has no backing tokens. [9](#0-8) [10](#0-9)

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
