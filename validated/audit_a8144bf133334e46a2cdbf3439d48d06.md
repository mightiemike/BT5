Audit Report

## Title
`PriceVelocityGuardExtension.beforeSwap` resets `lastMidPriceX64` per-swap, allowing cumulative intra-block price movement to exceed the configured per-block cap — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap` unconditionally writes `s.lastMidPriceX64 = midPrice` and `s.lastUpdateBlock = block.number` at lines 57–58 before the velocity check executes. Each subsequent swap in the same block therefore measures its incremental delta against the *previous swap's* mid price rather than the start-of-block mid price. An attacker who can push multiple oracle price updates within one block (e.g., via Pyth's permissionless `updatePriceFeeds`) can chain N swaps, each individually within the configured `maxChangePerBlockE18` envelope, while the cumulative intra-block price movement reaches approximately N × `maxChangePerBlockE18` — directly contradicting the NatSpec invariant "Caps how fast the provided price can move **between blocks**."

## Finding Description

In `PriceVelocityGuardExtension.beforeSwap`, the storage state is committed unconditionally before the guard check:

```solidity
// lines 54–55: capture old values
uint128 prevMid   = s.lastMidPriceX64;
uint64  prevBlock = s.lastUpdateBlock;

// lines 57–58: WRITE new state before the check
s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);

// lines 60–75: check uses captured prevMid/prevBlock — correct for THIS swap
if (prevMid != 0) {
    uint256 blockDiff = block.number - prevBlock;   // 0 for same-block swaps
    uint256 changeE18 = (delta * 1e18) / uint256(prevMid);
    uint256 actualSq  = changeE18 * changeE18;
    uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
    if (actualSq > allowedSq) revert PriceVelocityExceeded(...);
}
```

Because `prevMid` and `prevBlock` are captured before the write, the check for the *current* swap is arithmetically correct. However, once the check passes, `lastMidPriceX64` is already set to the new mid price. The *next* swap in the same block therefore reads `prevMid = P₁` (not P₀) and `blockDiff = 0`, so it is allowed another full `maxChangePerBlockE18` step from P₁:

| Swap | `prevMid` used | `blockDiff` | `allowedSq` | cumulative move from P₀ |
|------|----------------|-------------|-------------|------------------------|
| 1 | P₀ | ≥0 | maxChange² × (1+blockDiff) | ≤ maxChange |
| 2 | **P₁** | 0 | maxChange² | ≤ 2 × maxChange |
| N | **P_{N-1}** | 0 | maxChange² | ≤ N × maxChange |

The NatSpec at line 10 states the guard "Caps how fast the provided price can move **between blocks**," but the implementation enforces only a per-swap cap. There is no mechanism that anchors the comparison baseline to the start-of-block mid price. [1](#0-0) [2](#0-1) 

## Impact Explanation

The velocity guard is the primary on-chain protection against oracle-price manipulation on pools that configure it. Bypassing it allows a swap to execute at a mid price that is arbitrarily far from the start-of-block oracle price. Because the pool's bin pricing is anchored to the oracle mid (derived from `bidPriceX64`/`askPriceX64` passed through `SwapMath.midAndSpreadFeeX64FromBidAsk`), a sufficiently large intra-block price shift lets the attacker drain token reserves from LP bins at stale prices — a direct loss of LP principal. This matches the allowed-impact gate: *bad-price execution: stale, inverted, unbounded, or unclamped bid/ask quote reaches a pool swap*. [3](#0-2) [4](#0-3) 

## Likelihood Explanation

Pyth Network price feeds are updated permissionlessly: any party holding a valid VAA (signed by Pyth's authorized publishers and freely available from the Pyth off-chain API) can call `updatePriceFeeds` on the Pyth contract. On chains with sub-second block times (Base, HyperEVM) or on Ethereum with flashbots bundles, an attacker can interleave multiple `updatePriceFeeds` + `pool.swap` calls within a single block. No privileged role is required; the attacker only needs to be an allowed swapper (or the pool has no swap allowlist). The `swap` function in `MetricOmmPool` is public and calls `_getBidAndAskPriceX64()` fresh on each invocation, so each swap in the bundle picks up the latest oracle price. [5](#0-4) 

## Recommendation

Track the **start-of-block** mid price separately from the last-swap mid price. Only update `lastMidPriceX64` when `block.number > lastUpdateBlock` (i.e., a new block has started). Within the same block, compare the current mid against the value stored at the beginning of the block and do **not** overwrite it:

```solidity
if (block.number > prevBlock) {
    // new block: update reference price and block number
    s.lastMidPriceX64  = midPrice;
    s.lastUpdateBlock  = uint64(block.number);
} else {
    // same block: do NOT update lastMidPriceX64; compare against start-of-block reference
}
```

Alternatively, store a separate `blockStartMidPriceX64` field that is written only once per block and always use it as the comparison baseline, keeping `lastMidPriceX64` for other purposes. [6](#0-5) 

## Proof of Concept

1. Deploy a pool configured with `PriceVelocityGuardExtension`, `maxChangePerBlockE18 = 0.01e18` (1% per block), open swap allowlist.
2. Oracle (Pyth) mid price at block N−1: P₀ = 1.00; `lastMidPriceX64 = P₀`, `lastUpdateBlock = N−1`.
3. Attacker bundles in block N (e.g., via flashbots):
   - `updatePriceFeeds(VAA for P₁ = 1.01)` → `pool.swap(...)` → velocity check: Δ = 1% from P₀, `blockDiff = 1`, `allowedSq = maxChange² × 2`, passes; `lastMidPriceX64 = P₁`.
   - `updatePriceFeeds(VAA for P₂ = 1.0201)` → `pool.swap(...)` → velocity check: Δ = 1% from **P₁**, `blockDiff = 0`, `allowedSq = maxChange²`, passes; `lastMidPriceX64 = P₂`.
   - Repeat 10 times → final mid ≈ 1.1046 (10.46% above P₀).
4. Each individual swap passed the 1% cap; cumulative intra-block movement is ~10.46%, far exceeding the intended 1% per-block limit.
5. LP bins priced at the stale P₀ are drained at a ~10% discount relative to the true oracle price, extracting LP principal.
