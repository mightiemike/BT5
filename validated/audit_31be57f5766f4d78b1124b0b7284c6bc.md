### Title
Difficulty Retarget Uses Stale Mainchain Boundary Block Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When validating a fork block's proof-of-work, the difficulty retarget calculation fetches the boundary block of the difficulty adjustment interval from the **mainchain** (`mainchain_height_to_header`) rather than from the **fork's own ancestor chain**. If the fork diverged before the difficulty boundary, the mainchain block at that height is a different block with a different timestamp than the fork's actual ancestor. This stale/wrong value corrupts the expected-bits calculation, which is the critical gate for accepting or rejecting a submitted block.

---

### Finding Description

In `get_next_work_required` (Bitcoin and Litecoin variants), the first block of the difficulty adjustment interval is fetched via:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [1](#0-0) 

The `get_header_by_height` implementation always resolves through `mainchain_height_to_header`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

`mainchain_height_to_header` is a map that is **only updated for mainchain blocks**, never for fork blocks: [3](#0-2) 

Fork blocks are stored only in `headers_pool` via `store_fork_header`: [4](#0-3) 

When a fork block is submitted whose fork point is **before** the difficulty adjustment boundary (e.g., the fork diverges at height N, and the boundary is at height N+K where K < 2016), the block at `first_block_height` on the fork is a different block than the mainchain block at that same height. The contract silently uses the mainchain block's timestamp, producing a wrong `expected_bits` value.

The same pattern exists in the Dogecoin implementation, where the codebase itself acknowledges the issue with a TODO comment:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [5](#0-4) 

The Litecoin variant has the identical structural flaw: [6](#0-5) 

---

### Impact Explanation

The `expected_bits` computed from the wrong boundary block timestamp is compared directly against the submitted block's `bits` field:

```rust
require!(
    expected_bits == block_header.bits,
    "bad-diffbits: incorrect proof of work"
);
``` [7](#0-6) 

If the mainchain boundary block has an **earlier** timestamp than the fork's actual boundary ancestor, the computed difficulty is **lower** (easier target). An attacker can submit fork blocks with a lower `bits` value (easier PoW) that passes both the `expected_bits` equality check and the hash-vs-target check. If this fork accumulates enough chainwork, `reorg_chain` promotes it to the canonical mainchain, corrupting the canonical chain state and invalidating all downstream `verify_transaction_inclusion` results. [8](#0-7) 

---

### Likelihood Explanation

Bitcoin's difficulty adjustment interval is 2016 blocks (~2 weeks). Any fork that diverges more than one block before a retarget boundary triggers this bug. A relayer or any NEAR caller with access to `submit_blocks` can submit such a fork — no privileged role is required. The `submit_blocks` function is open to any trusted relayer, and the `trusted_relayer` macro does not prevent adversarial chain data from being submitted. [9](#0-8) 

---

### Recommendation

When computing the difficulty boundary block for a fork submission, traverse the fork's ancestor chain (via `get_prev_header` walking backward through `headers_pool`) to find the actual ancestor at `first_block_height`, rather than calling `get_header_by_height` which unconditionally reads from the mainchain index. This mirrors how Bitcoin Core resolves the boundary block: by walking the chain of `prev_block_hash` pointers from the current block tip.

---

### Proof of Concept

1. The mainchain contains blocks at heights 0–2016, with the boundary block at height 0 having timestamp `T_main`.
2. An attacker constructs a fork diverging at height 1, where the fork's block at height 0 (the genesis, shared) has timestamp `T_main`, but the fork's block at height 1 has a much later timestamp, making the fork's effective retarget interval appear shorter.
3. At height 2016, the fork needs a retarget. `get_next_work_required` calls `get_header_by_height(0)`, which returns the mainchain block at height 0 (timestamp `T_main`). This is the same block, so no divergence here for Bitcoin at the genesis boundary.
4. More concretely: fork diverges at height 1000. The next retarget is at height 2016. `first_block_height = 2016 - 2015 = 1`. The mainchain block at height 1 has timestamp `T_main_1`. The fork's ancestor at height 1 has timestamp `T_fork_1`. If `T_fork_1 > T_main_1`, the fork's actual retarget would produce a harder target, but the contract computes an easier target using `T_main_1`, allowing the attacker to mine the fork block at lower difficulty.
5. The fork block passes `expected_bits == block_header.bits` (using the wrong easy target) and the hash check. It is stored and, if chainwork exceeds the mainchain, `reorg_chain` is triggered, corrupting the canonical chain. [10](#0-9)

    fn submit_block_header_inner(
        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }
