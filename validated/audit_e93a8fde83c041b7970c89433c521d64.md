### Title
Silent `LinkSigner` Update for Isolated Subaccounts Due to Storage Desynchronization — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `EndpointTx.sol` accepts isolated subaccounts as `sender` without restriction, writes to `linkedSigners[isolatedSubaccount]`, and returns without reverting. However, `getLinkedSigner` for isolated subaccounts always reads from `linkedSigners[parentSubaccount]` — a different storage slot — so the write is permanently silently ignored. Any user who submits a `LinkSigner` for their isolated subaccount receives no error but achieves no effect.

---

### Finding Description

`getLinkedSigner` in `EndpointTx.sol` branches on whether the subaccount is isolated: [1](#0-0) 

For isolated subaccounts it reads `linkedSigners[parentSubaccount]` (fetched via `IOffchainExchange.getParentSubaccount`), never `linkedSigners[isolatedSubaccount]`.

The slow-mode `LinkSigner` handler writes unconditionally to `linkedSigners[txn.sender]` with no guard against isolated subaccounts: [2](#0-1) 

The fast-mode `LinkSigner` handler does the same: [3](#0-2) 

Neither path checks `RiskHelper.isIsolatedSubaccount(txn.sender)`. Both paths succeed without reverting, increment the nonce, and charge the slow-mode fee (slow path), yet the written value `linkedSigners[isolatedSubaccount]` is never consulted by any downstream logic.

The `getParentSubaccount` lookup that drives the read path is: [4](#0-3) 

The isolated subaccount is registered (via `_recordSubaccount`) at creation time, so `requireSubaccount` passes. For the slow-mode path, `validateSender` also passes because `address(uint160(bytes20(isolatedSubaccount)))` equals the parent EOA address (the isolated subaccount bytes32 encodes the parent address in its top 20 bytes): [5](#0-4) 

---

### Impact Explanation

A user who submits `LinkSigner` targeting their isolated subaccount:

1. **Cannot independently set a linked signer** for the isolated subaccount. If the parent has no linked signer, the isolated subaccount will also have none, regardless of what the user writes. Orders signed by the "set" linked signer will be rejected at matching time.
2. **Cannot independently revoke a linked signer** for the isolated subaccount. If the parent's linked signer is compromised, the user cannot revoke it for the isolated subaccount alone — the revocation silently fails. The compromised key retains signing authority over the isolated subaccount's positions.
3. **Wastes slow-mode fees** (charged at submission, not execution) and increments the address nonce without any state change.

The corrupted invariant is: `linkedSigners[isolatedSubaccount]` holds a value the user believes is authoritative, but `getLinkedSigner` never reads it, creating a permanent divergence between user-visible state and enforced signer state.

---

### Likelihood Explanation

Any user who creates an isolated subaccount and attempts to manage its linked signer independently — a natural expectation given that `LinkSigner` accepts any registered subaccount as sender — will trigger this path. No special privileges are required. The slow-mode path is directly user-accessible via `Endpoint.submitSlowModeTransaction`.

---

### Recommendation

Add an explicit guard in both `LinkSigner` handlers to reject isolated subaccounts as senders:

```solidity
require(
    !RiskHelper.isIsolatedSubaccount(txn.sender),
    ERR_UNAUTHORIZED
);
```

This mirrors the guard already present in other handlers (e.g., `NlpProfitShare` at line 308, `claimBuilderFee` at line 873 in `OffchainExchange.sol`). Alternatively, if independent linked-signer control per isolated subaccount is desired, `getLinkedSigner` must be updated to check `linkedSigners[isolatedSubaccount]` first before falling back to the parent.

---

### Proof of Concept

1. Parent subaccount `P` has no linked signer (`linkedSigners[P] == address(0)`).
2. User creates isolated subaccount `I` (child of `P`).
3. User submits slow-mode `LinkSigner` with `sender = I`, `signer = signerA`. Transaction succeeds; `linkedSigners[I] = signerA`.
4. User submits an order for `I` signed by `signerA`.
5. At matching time, `getLinkedSignerOrNlpSigner(I)` calls `getLinkedSigner(I)`, which returns `linkedSigners[P] == address(0)`.
6. `_checkSignature` in `OffchainExchange` fails: `signerA != address(uint160(bytes20(I)))` and `signerA != address(0)`. The order is rejected.
7. The user's `LinkSigner` transaction had no effect; the slow-mode fee is lost. [6](#0-5) [7](#0-6)
