### Title
Difficulty Adjustment Uses Mainchain Timestamp Instead of Fork Ancestor Timestamp, Enabling Incorrect PoW Target Acceptance for Fork Blocks - (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary

The difficulty adjustment calculation in `get_next_work_required` combines two timestamps from different chain lineages without validating their consistency: `prev_block_header.time` (from the fork being validated) and `first_block_time` (always fetched from the **mainchain** via `get_header_by_height`). When a fork block is submitted that diverges before the difficulty adjustment boundary, the `first_block_time` used in the calculation belongs to a different block than the fork's actual ancestor at that height. This desynchronization produces an incorrect difficulty target, which can be exploited to get fork blocks accepted with less PoW than the protocol requires.

### Finding Description

`get_header_by_height` in `contract/src/lib.rs` always reads from `mainchain_height_to_header`: [1](#0-0) 

During fork block validation, `get_next_work_required` in `contract/src/bitcoin.rs` calls this to obtain `first_block_time`: [2](#0-1) 

The same pattern exists in `contract/src/litecoin.rs`: [3](#0-2) 

And in `contract/src/dogecoin.rs`, where the developers themselves flagged the issue with a TODO comment: [4](#0-3) 

When a fork block is being validated, `prev_block_header` is correctly resolved by following the fork's chain lineage through `headers_pool` via `prev_block_hash`. However, `first_block_time` is fetched from `mainchain_height_to_header` at the same height — which is a **different block** if the fork diverged before that height. The two timestamps therefore come from different chain lineages, and no consistency check is performed between them.

The desynchronized timestamps feed directly into `calculate_next_work_required`, which computes `actual_time_taken = prev_block_header.time - first_block_time`: [5](#0-4) 

The resulting `expected_bits` is then enforced as the only accepted difficulty for the submitted block: [6](#0-5) 

### Impact Explanation

An attacker who submits a fork that diverges before a difficulty adjustment boundary can cause the contract to compute an incorrect `expected_bits` for fork blocks at that boundary. If the mainchain's block at `first_block_height` has an earlier timestamp than the fork's actual ancestor at that height, `actual_time_taken` is inflated, producing a lower (easier) difficulty target. The attacker mines fork blocks at this reduced difficulty, submits them, and — if the fork accumulates more chainwork than the mainchain — triggers a chain reorg. The canonical chain state (`mainchain_tip_blockhash`, `mainchain_height_to_header`) is then corrupted to reflect a chain that was not validated at the correct difficulty. Downstream consumers of `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` would then verify proofs against a fraudulently promoted chain.

For Dogecoin specifically, `difficulty_adjustment_interval = 1` after block 145,000, meaning **every fork block** past the divergence point triggers this desynchronization: [7](#0-6) 

For Bitcoin and Litecoin, the issue is triggered at every 2016-block boundary that falls within the fork.

### Likelihood Explanation

The entry point is `submit_blocks`, callable by any trusted relayer (or any account if `UnrestrictedSubmitBlocks` is granted). Fork submission is an explicitly supported and expected code path — the contract's reorg logic is designed to handle it. No privileged key or social engineering is required; a relayer simply submits a sequence of fork headers that diverge before a difficulty adjustment boundary. For Dogecoin mainnet (post-block 145,000), every fork block past the first triggers the bug. The TODO comment in the Dogecoin code confirms the developers are aware the behavior is unverified. [8](#0-7) 

### Recommendation

Replace `get_header_by_height` (which reads from the mainchain index) with an ancestor-traversal that follows the fork's own chain lineage. Specifically, walk backwards from `prev_block_header` using `get_prev_header` for exactly `blocks_to_go_back` steps to obtain the correct `first_block_time`. This ensures both timestamps used in `calculate_next_work_required` originate from the same chain lineage, eliminating the desynchronization.

### Proof of Concept

1. The contract is initialized with a mainchain at height H (a difficulty adjustment boundary). The mainchain block at height `H - 2016` has timestamp `T_main`.
2. An attacker constructs a fork that diverges at height `H - 2016`, where the fork's ancestor at that height has timestamp `T_fork > T_main` (e.g., 2 hours later, within the allowed future-time window).
3. The attacker submits fork blocks up to height `H`. At height `H`, `get_next_work_required` is called.
4. `first_block_time` = `T_main` (from mainchain via `get_header_by_height(H - 2016)`).
5. `prev_block_header.time` = fork's block at `H - 1` (correct fork lineage).
6. `actual_time_taken = prev_block_header.time - T_main` is larger than the true `prev_block_header.time - T_fork`, producing a lower `expected_bits` (easier difficulty).
7. The attacker mines the fork block at height `H` against this reduced difficulty, submits it, and it passes `check_pow`.
8. If the fork's accumulated chainwork exceeds the mainchain's, `reorg_chain` is triggered, corrupting the canonical chain. [9](#0-8)

// source https://github.com/dogecoin/dogecoin/blob/2c513d0172e8bc86fe9a337693b26f2fdf68a013/src/dogecoin.cpp#L41
fn calculate_next_work_required(
    config: &DogecoinConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let retarget_timespan = config.pow_target_timespan;
    let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;

    let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;

    let min_timespan = retarget_timespan - (retarget_timespan / 4);
    let max_timespan = retarget_timespan + (retarget_timespan / 2);

    if modulated_timespan < min_timespan {
        modulated_timespan = min_timespan;
    } else if modulated_timespan > max_timespan {
        modulated_timespan = max_timespan;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(modulated_timespan).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target =
        new_target / U256::from(<i64 as TryInto<u64>>::try_into(retarget_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
