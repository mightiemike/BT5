### Title
Silent Transaction Loss via Unchecked Broadcast Return Value in `MempoolP2pPropagator::broadcast_queued_transactions` — (File: `crates/apollo_mempool_p2p/src/propagator/mod.rs`)

---

### Summary

`MempoolP2pPropagator::broadcast_queued_transactions` drains the internal transaction queue **before** attempting the network broadcast. When the broadcast channel is full (`err.is_full()`), the function silently returns `Ok(())` after permanently discarding all drained transactions. The caller receives a success signal while the transactions are irreversibly lost from the propagation queue — a direct sequencer-domain analog of the ERC20 unchecked-return-value pattern.

---

### Finding Description

In `crates/apollo_mempool_p2p/src/propagator/mod.rs`, the function `broadcast_queued_transactions` (lines 117–140) executes the following sequence:

```rust
async fn broadcast_queued_transactions(&mut self) -> MempoolP2pPropagatorResult<()> {
    // 1. STATE MUTATED FIRST — queue is permanently drained before broadcast succeeds
    let queued_transactions: Vec<RpcTransaction> = self.transaction_queue.drain(..).collect();
    if queued_transactions.is_empty() {
        return Ok(());
    }
    let number_of_transactions_in_batch = queued_transactions.len().into_f64();
    let result = self
        .broadcast_topic_client
        .broadcast_message(RpcTransactionBatch(queued_transactions))
        .await
        // 2. FAILURE SILENTLY SWALLOWED — returns Ok(()) when buffer is full
        .or_else(|err| {
            if !err.is_full() {
                warn!("Error broadcasting transaction batch: {:?}", err);
                return Err(MempoolP2pPropagatorError::NetworkSendError);
            }
            warn!("...Dropping the transaction batch.");
            Ok(())   // <-- silent success despite total transaction loss
        });
    MEMPOOL_P2P_BROADCASTED_BATCH_SIZE.record(number_of_transactions_in_batch);
    result
}
```

The structural parallel to the ERC20 bug is exact:

| ERC20 analog | Sequencer instance |
|---|---|
| `pool.tokenBalance -= amount` (state updated before transfer) | `self.transaction_queue.drain(..)` (queue emptied before broadcast) |
| `IERC20(token).transfer(receiver, amount)` returns `false` / reverts | `broadcast_message(...)` fails with `is_full()` |
| Caller sees no error; `pool.totalRedeemed += 1` incremented incorrectly | Caller receives `Ok(())`; transactions are permanently gone |

This function is called from two sites:

1. **`add_transaction`** (line 98) — triggered when the batch reaches `max_transaction_batch_size`. The mempool's `add_transaction` call returns `Ok(())` to the mempool wrapper, which in turn returns success to the gateway. The gateway has already committed the transaction to the mempool; the user receives a success response. The transaction will never be propagated to peers.

2. **`MempoolP2pRunner::start`** (timer tick, `crates/apollo_mempool_p2p/src/runner/mod.rs` line 75) — periodic flush. If the buffer is full at flush time, all queued transactions are silently dropped and the runner continues normally.

---

### Impact Explanation

**High — Mempool/gateway/RPC admission rejects valid transactions before sequencing (via silent propagation loss).**

A transaction accepted by the gateway and inserted into the local mempool will never reach peer sequencer nodes if the P2P broadcast channel is full at the moment of batch dispatch. Because the queue is drained before the broadcast attempt and `Ok(())` is returned on `is_full()`, there is no retry, no re-queue, and no error propagated to the submitter. The transaction exists only in the local mempool. If the local node is not the current block proposer, the transaction will never be sequenced. The submitter receives a valid transaction hash and a success response, but the transaction is effectively dead.

---

### Likelihood Explanation

**Medium.** The broadcast channel is a bounded async channel. Under normal load the buffer is unlikely to be full. However:
- A burst of transactions filling the batch exactly when the channel is saturated (e.g., during high network activity or a slow peer) triggers the silent drop.
- An adversary who can flood the P2P channel (e.g., by sending large numbers of transactions from another node) can reliably keep the buffer full, causing every batch flush to silently discard legitimate transactions.
- The `MempoolP2pRunner` timer path (line 75) is also affected: if the buffer is full at every tick, all queued transactions are repeatedly dropped with no recovery.

---

### Recommendation

1. **Do not drain the queue before confirming the broadcast succeeds.** Use a peek/clone approach, and only drain on success:

```rust
async fn broadcast_queued_transactions(&mut self) -> MempoolP2pPropagatorResult<()> {
    if self.transaction_queue.is_empty() {
        return Ok(());
    }
    let batch = RpcTransactionBatch(self.transaction_queue.clone());
    let n = self.transaction_queue.len().into_f64();
    match self.broadcast_topic_client.broadcast_message(batch).await {
        Ok(()) => {
            self.transaction_queue.clear();
            MEMPOOL_P2P_BROADCASTED_BATCH_SIZE.record(n);
            Ok(())
        }
        Err(err) if err.is_full() => {
            warn!("Broadcast buffer full; retaining {} transactions for next attempt.", self.transaction_queue.len());
            Err(MempoolP2pPropagatorError::NetworkSendError)
        }
        Err(err) => {
            warn!("Error broadcasting transaction batch: {:?}", err);
            Err(MempoolP2pPropagatorError::NetworkSendError)
        }
    }
}
```

2. **Propagate the error upward** rather than converting `is_full()` into `Ok(())`. The caller (`add_transaction` and the runner timer) should handle backpressure explicitly — either by retrying, alerting, or returning an error to the mempool so the transaction is not silently lost.

---

### Proof of Concept

**Trigger path:**

1. Mempool accepts N transactions (N = `max_transaction_batch_size`).
2. Each `add_transaction` call pushes to `transaction_queue`.
3. On the N-th call, `transaction_queue.len() == max_transaction_batch_size` → `broadcast_queued_transactions()` is called.
4. `self.transaction_queue.drain(..)` empties the queue (line 118). Transactions are now held only in the local `queued_transactions` variable.
5. `broadcast_message(RpcTransactionBatch(queued_transactions))` is called. The channel is full → returns `Err(is_full = true)`.
6. `.or_else` branch: `err.is_full()` is `true` → returns `Ok(())` (line 136).
7. `add_transaction` returns `Ok(())` to the mempool wrapper.
8. The mempool wrapper returns success to the gateway; the gateway returns the transaction hash to the user.
9. `self.transaction_queue` is now empty. The N transactions are permanently lost from the propagation layer.
10. No peer node ever receives these transactions. If the local node is not the proposer, the transactions are never sequenced.
