### Title
`SwapAllowlistExtension.beforeSwap` checks the router address as `sender` instead of the end user, enabling allowlist bypass or locking out allowlisted users via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender`. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the end user. This creates two fund-impacting failure modes: (1) allowlisted users cannot use the router at all, or (2) if the pool admin allowlists the router to enable router swaps, every non-allowlisted user can bypass the curated pool's access control.

---

### Finding Description

**Call path:**

```
User → MetricOmmSimpleRouter.exactInputSingle()
     → IMetricOmmPoolActions(pool).swap(recipient, ...)   // msg.sender = router
     → MetricOmmPool._beforeSwap(msg.sender=router, ...)
     → ExtensionCalling._callExtensionsInOrder(...)
     → SwapAllowlistExtension.beforeSwap(sender=router, ...)
```

In `MetricOmmPool.swap`, `msg.sender` is forwarded verbatim as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that same `sender` into the hook call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is in the allowlist: [3](#0-2) 

When the swap originates from `MetricOmmSimpleRouter.exactInputSingle`, the router calls the pool directly — so `sender` received by the extension is the **router address**, not the end user: [4](#0-3) 

The allowlist is administered per-user address by the pool admin: [5](#0-4) 

This creates an irreconcilable dilemma for any pool admin who deploys a curated pool with `SwapAllowlistExtension`:

**Scenario A — Lockout:** Admin allowlists specific user addresses but not the router. Allowlisted users who call `MetricOmmSimpleRouter.exactInputSingle` have their swap rejected because the extension sees `sender = router`, which is not in the allowlist. The supported periphery path is broken for all allowlisted users.

**Scenario B — Bypass:** Admin allowlists the router address (the only way to enable router-mediated swaps for their users). Now `allowedSwapper[pool][router] = true`. Any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle` targeting the curated pool; the extension sees `sender = router`, which is allowlisted, and the swap succeeds — completely bypassing the curation policy.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the position beneficiary), not `sender` (the payer/adder contract), so the deposit path does not share this flaw: [6](#0-5) 

---

### Impact Explanation

**Scenario B (bypass)** is the critical path: a non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle` on a curated pool whose admin was forced to allowlist the router. The extension passes, the swap executes at oracle price, and the user receives output tokens they were explicitly barred from receiving. This is a direct policy bypass on a production curated pool with real LP assets at stake — matching the "allowlist bypass" impact class (High, broken core pool functionality / unauthorized swap execution).

**Scenario A (lockout)** breaks the supported periphery path for legitimate users, constituting broken core swap functionality.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap entrypoint documented and deployed for the protocol. Any pool that configures `SwapAllowlistExtension` and expects users to swap through the router will encounter one of the two failure modes immediately upon first use. No special preconditions, flash loans, or privileged access are required — a standard `exactInputSingle` call is sufficient.

---

### Recommendation

The extension must check the **original end user**, not the intermediary router. Two complementary fixes:

1. **In `MetricOmmSimpleRouter`**: pass the original `msg.sender` as `callbackData` or a dedicated field so the extension can recover it. Alternatively, the pool could expose a `swapWithOriginator` variant that carries the originating user.

2. **In `SwapAllowlistExtension.beforeSwap`**: if the `sender` is a known router, decode the true originator from `extensionData` and check that address instead. This requires the router to forward the end user's address in `extensionData`.

The simplest correct fix is to have `MetricOmmSimpleRouter` forward `msg.sender` inside `extensionData` for every hop, and have `SwapAllowlistExtension` decode and check that address when `sender` is a recognized router.

---

### Proof of Concept

```solidity
// Pool configured with SwapAllowlistExtension.
// Admin allowlists the router so that allowlisted users can swap via it.
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// alice is NOT allowlisted
// assertFalse(swapExtension.isAllowedToSwap(address(pool), alice));

// alice calls the router — sender seen by extension = router (allowlisted)
vm.startPrank(alice);
token1.approve(address(router), type(uint256).max);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token1),
        recipient: alice,
        deadline: block.timestamp + 1,
        zeroForOne: false,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        extensionData: ""
    })
);
vm.stopPrank();
// alice successfully swapped on a pool she was explicitly barred from — allowlist bypassed.
```
