### Title
Difficulty Calculation for Fork Blocks Uses Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/dogecoin.rs`)

---

### Summary

In the Dogecoin (and Bitcoin/Litecoin) difficulty adjustment logic, `get_header_by_height` always resolves blocks from `mainchain_height_to_header`. When a fork block is submitted at depth ≥ 3 from the fork point (depth ≥ 2 for Dogecoin's per-block retarget protocol), the boundary block used for difficulty calculation is taken from the mainchain rather than the fork's actual ancestor chain. This desynchronization between the fork's ancestor state and the mainchain state produces an incorrect difficulty target, allowing an attacker to submit fork blocks with less PoW than the protocol requires.

---

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes the expected difficulty for a new block by measuring the elapsed time between the previous block and the first block of the adjustment interval: [1](#0-0) 

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

The `get_header_by_height` implementation always resolves through `mainchain_height_to_header`: [2](#0-1) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

For Dogecoin with `new_difficulty_protocol` (height ≥ 145,000), `difficulty_adjustment_interval = 1`, so every block is a retarget and `blocks_to_go_back = 1`: [3](#0-2) 

This means for a fork block at height `H+k` (k ≥ 3), `height_first = H+k-2`. The mainchain block at `H+k-2` diverges from the fork's ancestor at `H+k-2` whenever `k-2 ≥ 1`, i.e., `k ≥ 3`. The difficulty calculation then uses the wrong timestamp:

```
// Computed (wrong for deep forks):
modulated_timespan = fork[H+k-1].time − mainchain[H+k-2].time

// Correct:
modulated_timespan = fork[H+k-1].time − fork[H+k-2].time
``` [4](#0-3) 

The code itself acknowledges this with a `TODO` comment at line 291, confirming the developers recognized the ambiguity but left it unresolved.

The same structural issue exists in `contract/src/bitcoin.rs` and `contract/src/litecoin.rs`: [5](#0-4) [6](#0-5) 

For Bitcoin/Litecoin the interval is 2016 blocks, making exploitation impractical. For Dogecoin's per-block retarget, the desynchronization activates at fork depth 3.

---

### Impact Explanation

An attacker who submits a fork diverging at height `H` can craft fork blocks at `H+1` and `H+2` with strategically chosen timestamps. For the block at `H+3`, the difficulty is computed using `mainchain[H+1].time` instead of `fork[H+1].time`. By setting `fork[H+1].time` just above the MTP floor (making it small), while `mainchain[H+1].time` is a normal real-world timestamp (larger), the computed `modulated_timespan` becomes smaller than the true elapsed time. After Digishield damping, this reduces the required difficulty by up to 25% per retarget step. Compounded over a multi-block fork, the attacker can accumulate chainwork with less total PoW than the protocol requires. If the fork's `chain_work` exceeds the mainchain's, `reorg_chain` is triggered: [7](#0-6) 

This corrupts `mainchain_tip_blockhash` and `mainchain_height_to_header`, causing `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to evaluate proofs against an attacker-controlled canonical chain: [8](#0-7) 

---

### Likelihood Explanation

For Dogecoin with `new_difficulty_protocol` (all mainnet blocks above height 145,000), every block is a retarget and the desynchronization activates at fork depth 3. Any unprivileged NEAR caller can invoke `submit_blocks` with adversarial Dogecoin headers. No privileged role, leaked key, or social engineering is required. The attacker only needs to submit a sequence of fork headers — a normal relayer operation — with crafted timestamps. The `trusted_relayer` guard on `submit_blocks` can be bypassed if the attacker stakes as a relayer, or the contract may be configured without that restriction. [9](#0-8) 

---

### Recommendation

Replace `get_header_by_height` in difficulty calculations with an ancestor traversal that walks back from `prev_block_header` via `get_prev_header` calls until reaching `height_first`. This ensures the boundary block is always the fork's actual ancestor, not the mainchain's block at that height. The existing `get_prev_header` correctly resolves through `headers_pool` by hash: [10](#0-9) 

---

### Proof of Concept

1. Dogecoin mainchain is at height `H+10` (height ≥ 145,000, new difficulty protocol active).
2. Attacker calls `submit_blocks` with a fork block at `H+1` (valid, parent = mainchain block at `H`).
3. Attacker calls `submit_blocks` with a fork block at `H+2` (valid, parent = fork block at `H+1`; `height_first = H`, which is the shared fork point — correct).
4. Attacker calls `submit_blocks` with a fork block at `H+3`. Now `height_first = H+1`. `get_header_by_height(H+1)` returns `mainchain[H+1]`, not `fork[H+1]`.
5. Attacker set `fork[H+1].time` = MTP + 1 (minimum valid, e.g., 600 seconds after MTP). Mainchain `[H+1].time` is a real timestamp, e.g., 600 seconds later than `fork[H+1].time`.
6. Computed `modulated_timespan = fork[H+2].time − mainchain[H+1].time` is smaller than `fork[H+2].time − fork[H+1].time` by ~600 seconds.
7. After Digishield damping, the required difficulty for `H+3` is reduced.
8. Attacker repeats for `H+4`, `H+5`, … accumulating chainwork with reduced PoW per block.
9. Once `fork.chain_work > mainchain.chain_work`, `reorg_chain` fires, replacing the canonical chain with the attacker's fork.
10. `verify_transaction_inclusion` now confirms attacker-chosen transactions as included in the canonical chain.

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
