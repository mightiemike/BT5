### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every `beforeSwap` hook. When a user routes through the public `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the originating user. `SwapAllowlistExtension.beforeSwap` keys its allowlist lookup on that `sender` value. Any non-allowlisted user can therefore bypass a curated pool's swap gate by routing through the router, because the router's address — not the user's — is what the extension checks.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to every hook.**

`MetricOmmPool.swap` captures `msg.sender` and forwards it verbatim to `_beforeSwap`: [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` relays that value unchanged to every configured extension.** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` keys its allowlist lookup on `sender`.**

The research pivot in the repository confirms the lookup is `allowAll/allowedSwapper` keyed by `(pool, sender)`: [3](#0-2) 

**Step 4 — When a user routes through `MetricOmmSimpleRouter`, `sender` = router.**

The router is a public, permissionless contract. Any EOA can call it. When it calls `pool.swap(...)`, the pool sees `msg.sender = router`. The extension therefore checks whether the **router** is allowlisted, not whether the **originating user** is allowlisted.

**Resulting invariant break:**

| Scenario | What the extension checks | Outcome |
|---|---|---|
| Router is allowlisted (or `allowAll = true`) | Router address ✓ | Every non-allowlisted user bypasses the gate via the router |
| Router is not allowlisted | Router address ✗ | Every legitimate user who uses the router is DoS'd |

The pool admin configures the allowlist believing it gates specific human actors. The extension silently gates a different entity (the router), making the configured policy either universally bypassable or universally broken for router users.

---

### Impact Explanation

**Bypass path (router allowlisted):** A non-allowlisted user routes through `MetricOmmSimpleRouter`. The extension sees the router's address, which passes the allowlist check. The user executes swaps on a pool that was designed to be curated. LP funds in the pool are exposed to actors the pool admin explicitly intended to exclude — direct loss of LP principal or fee revenue above Sherlock thresholds.

**DoS path (router not allowlisted):** Legitimate allowlisted users who interact through the supported periphery router are blocked. Core swap functionality is broken for the standard user-facing entry point.

Both paths are fund-impacting. The bypass path is the higher-severity outcome.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract — any EOA can call it.
- No special privilege, token, or setup is required beyond knowing the pool address.
- The bypass is deterministic and repeatable on every block.
- Pool admins have no on-chain mechanism to distinguish "router called by allowlisted user" from "router called by non-allowlisted user" under the current design.

Likelihood: **High**.

---

### Recommendation

The allowlist must gate the economically relevant actor — the originating user — not the intermediary router. Two approaches:

1. **Check `recipient` instead of `sender`** if the pool admin intends to gate who receives output (partial fix only; `recipient` can also be set arbitrarily).
2. **Pass the originating caller through the router as an explicit parameter** (e.g., `extensionData`) and have the extension decode and verify that address. The router must attest the real caller, and the extension must enforce it.
3. **Require direct pool interaction** for allowlisted pools (document that the router must not be used with allowlisted pools and add a factory-level flag to enforce this).

The cleanest fix is option 2: the router encodes `msg.sender` into `extensionData`, and `SwapAllowlistExtension` decodes and checks that value when `sender` is a known router address.

---

### Proof of Concept

```
1. Pool admin deploys a pool with SwapAllowlistExtension configured.
2. Pool admin adds Alice (0xAlice) to the allowlist; Bob (0xBob) is NOT added.
3. Bob calls MetricOmmSimpleRouter.exactInput(..., pool, ...).
4. Router calls pool.swap(recipient=Bob, ...).
5. Pool calls _beforeSwap(sender=router, ...).
6. SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][router].
   - If router is allowlisted (admin added it as a trusted intermediary): check passes → Bob's swap executes on the curated pool.
   - If router is not allowlisted: check fails → Alice's swap also fails when she uses the router (DoS).
7. In the bypass case, Bob repeats this for every block, draining LP value or
   executing trades the pool admin intended to restrict.
```

The root cause is that `MetricOmmPool.swap` passes `msg.sender` (the router) as `sender` to the hook, while the allowlist is configured against human actor addresses. The extension has no way to recover the originating EOA from the call context. [4](#0-3) [5](#0-4) [6](#0-5)
