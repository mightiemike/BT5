### Title
`DirectDepositV1` Constructor Accepts Zero `_wrappedNative` Address, Causing Silent Permanent ETH Loss — (File: `core/contracts/DirectDepositV1.sol`)

### Summary

The `DirectDepositV1` constructor accepts `_wrappedNative` as an `address payable` parameter with no zero-address validation. If `address(0)` is passed, the `receive()` fallback silently succeeds (EVM calls to `address(0)` return `success = true`) while permanently burning all native ETH sent to the contract. This is a direct analog to the `MerchantSubscription` null-address constructor bug: a contract instantiated with a null address parameter that silently fails to function — but here the consequence is irreversible asset loss rather than mere non-activation.

---

### Finding Description

`DirectDepositV1.constructor()` stores `_wrappedNative` directly without any zero-address guard: [1](#0-0) 

The `receive()` function then unconditionally calls `wrappedNative.call{value: msg.value}("")`: [2](#0-1) 

In the EVM, a low-level `.call` to `address(0)` returns `(true, "")`. The `require(success, "Failed to wrap native token.")` check therefore passes, no revert occurs, no event is emitted, and the ETH is permanently burned. The same silent burn occurs in the constructor's own balance-forwarding block: [3](#0-2) 

The `ContractOwner.initialize()` function, which sets the `wrappedNative` value later forwarded to every DDA, also performs no zero-address check on `_wrappedNative`: [4](#0-3) 

`createDirectDepositV1()` is declared `public` with no access control, so any external caller can trigger DDA creation once `ContractOwner` is initialized: [5](#0-4) 

---

### Impact Explanation

Any native ETH sent to a DDA whose `wrappedNative` is `address(0)` is permanently and silently destroyed. The `receive()` fallback emits no event and does not revert, so the sender receives no indication of failure. Because `creditDepositV1` is also unrestricted and auto-creates a DDA on first call, users interacting with a misconfigured deployment lose funds with no on-chain signal. [6](#0-5) 

---

### Likelihood Explanation

Likelihood is **low but non-zero**. It requires `ContractOwner.initialize()` to have been called with `_wrappedNative = address(0)` — a deployment-time misconfiguration. Because neither `initialize()` nor the `DirectDepositV1` constructor enforces the invariant, a deployment error (e.g., wrong argument order, scripting mistake) silently produces a broken system with no on-chain revert to catch it. The absence of any guard means the failure mode is invisible until funds are lost.

---

### Recommendation

Add a zero-address check in the `DirectDepositV1` constructor for `_wrappedNative`:

```solidity
require(_wrappedNative != address(0), "wrappedNative cannot be zero address");
``` [1](#0-0) 

Also add the same guard in `ContractOwner.initialize()` for `_wrappedNative`:

```solidity
require(_wrappedNative != address(0), "wrappedNative cannot be zero address");
``` [4](#0-3) 

---

### Proof of Concept

1. Deploy `ContractOwner` and call `initialize(...)` passing `_wrappedNative = address(0)` (no revert).
2. Any user calls `ContractOwner.createDirectDepositV1(someSubaccount)` — succeeds, DDA deployed with `wrappedNative = address(0)`.
3. User sends 1 ETH to the DDA address.
4. `receive()` executes: `address(0).call{value: 1 ether}("")` → returns `(true, "")`.
5. `require(true, ...)` passes. No event. No revert. 1 ETH permanently burned.
6. `creditDepositV1` also auto-creates a DDA on first call, so the same path is reachable without step 2. [2](#0-1) [6](#0-5)
