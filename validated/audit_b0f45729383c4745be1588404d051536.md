### Title
`get_protocol_config` Clones Stale `genesis_config` and Selectively Patches It, Leaking Genesis-Era Values into Per-Epoch RPC Responses - (File: `chain/chain/src/runtime/mod.rs`)

### Summary

`NightshadeRuntime::get_protocol_config` is the backend for the public `EXPERIMENTAL_protocol_config` RPC endpoint. It builds a `ProtocolConfig` by cloning the node's cached `self.genesis_config` (a snapshot frozen at node startup) and then overwriting only a subset of fields from the live epoch manager. Several `GenesisConfig` fields that **can change across protocol versions** — `gas_limit`, `min_gas_price`, `max_gas_price`, `transaction_validity_period`, `protocol_reward_rate`, `num_blocks_per_year`, `total_supply`, `dynamic_resharding` — are **never patched** and are always returned with their genesis-era values, regardless of which epoch is queried.

### Finding Description

`NightshadeRuntime` stores a `genesis_config: GenesisConfig` field that is cloned from the genesis configuration at construction time and never updated. [1](#0-0) 

`get_protocol_config` starts by cloning this frozen snapshot: [2](#0-1) 

It then overwrites a specific list of fields from the live `epoch_manager` and `runtime_config_store`:
- `protocol_version`, `epoch_length`, `num_block_producer_seats`, `block_producer_kickout_threshold`, `chunk_producer_kickout_threshold`, `chunk_validator_only_kickout_threshold`, `target_validator_mandates_per_shard`, `max_kickout_stake_perc`, `online_min_threshold`, `online_max_threshold`, `fishermen_threshold`, `minimum_stake_divisor`, `protocol_upgrade_stake_threshold`, `shard_layout`, `minimum_validators_per_shard`, `minimum_stake_ratio`, `shuffle_shard_assignment_for_chunk_producers`, `max_inflation_rate`.

The following `GenesisConfig` fields that appear in the `ProtocolConfigView` response are **not patched** and are served from the genesis snapshot regardless of the queried epoch:

| Field | Exposed in `ProtocolConfigView` |
|---|---|
| `gas_limit` | Yes |
| `min_gas_price` | Yes |
| `max_gas_price` | Yes |
| `transaction_validity_period` | Yes |
| `protocol_reward_rate` | Yes |
| `num_blocks_per_year` | Yes |
| `dynamic_resharding` | Yes |
| `total_supply` | (not in view, but in genesis_config) |

The `ProtocolConfigView` conversion reads these directly from `genesis_config`: [3](#0-2) 

The `gas_limit` field is particularly significant: it is a per-block gas limit that is adjusted dynamically by the protocol's gas price adjustment mechanism. The value stored in `genesis_config.gas_limit` is the **initial** gas limit, not the current one. Any client using `EXPERIMENTAL_protocol_config` to determine the current gas limit for a given block will receive the genesis-era value.

The `transaction_validity_period` is also significant: it defines how many blocks a transaction remains valid. If this were ever changed via a protocol upgrade, clients reading it from this RPC would use the wrong value for transaction construction.

The analog to the external bug is exact: `BlueBerryBank` cached `feeManager` at initialization and never updated it; here `NightshadeRuntime` caches `genesis_config` at initialization and `get_protocol_config` only partially patches it, leaving multiple fields permanently stale.

### Impact Explanation

Any RPC client or indexer that calls `EXPERIMENTAL_protocol_config` to determine current protocol parameters (gas limit, min/max gas price, transaction validity period, reward rate) receives genesis-era values. This is the **only** public RPC endpoint for querying these parameters per-block/epoch. Clients that rely on `gas_limit` from this endpoint for gas estimation or fee calculation will use incorrect values. The `transaction_validity_period` field, if ever changed by a protocol upgrade, would cause clients to construct transactions with wrong validity windows, leading to rejected transactions.

**Severity: Medium** — The fields are currently stable on mainnet, but the structural defect means any future protocol upgrade that changes `gas_limit`, `min_gas_price`, `max_gas_price`, or `transaction_validity_period` will silently serve wrong values to all RPC consumers without any error.

### Likelihood Explanation

The `EXPERIMENTAL_protocol_config` endpoint is a public, unprivileged RPC call reachable by any user. No special role is required. The divergence is triggered automatically whenever the queried epoch differs from genesis in any of the un-patched fields. On current mainnet the impact is limited because these fields have not changed since genesis, but the structural defect is present and reachable today.

### Recommendation

In `get_protocol_config`, replace the pattern of cloning `self.genesis_config` and selectively patching fields with a construction that derives all epoch-variable fields from the live `epoch_manager` and `runtime_config_store`. Specifically, `gas_limit`, `min_gas_price`, `max_gas_price`, `transaction_validity_period`, `protocol_reward_rate`, and `num_blocks_per_year` should either be sourced from the per-epoch `RuntimeConfig`/`EpochConfig` (if they are versioned there) or explicitly documented as genesis-constant with a comment explaining why they cannot change. The `dynamic_resharding` flag should be derived from the epoch's `ShardLayoutConfig` variant rather than from the genesis snapshot.

The documentation in `parameter_definition.md` already states: *"It should therefore not be cached. It should be read from the store once per chunk and then passed down."* [4](#0-3) 

### Proof of Concept

1. Start a nearcore node with a genesis that has `gas_limit = 1_000_000_000_000_000` (1000 TGas).
2. Suppose a future protocol upgrade changes `gas_limit` to `2_000_000_000_000_000` (2000 TGas) via a `GenesisConfig` change.
3. Call `EXPERIMENTAL_protocol_config` with `block_reference` pointing to a block in the new epoch.
4. Observe that `config_view.gas_limit` returns `1_000_000_000_000_000` (genesis value) instead of `2_000_000_000_000_000` (current value).

The root cause is at: [5](#0-4) 

where `self.genesis_config.clone()` is used as the base, and the subsequent patch loop at lines 1583–1606 never touches `gas_limit`, `min_gas_price`, `max_gas_price`, or `transaction_validity_period`. [6](#0-5)

    fn get_protocol_config(&self, epoch_id: &EpochId) -> Result<ProtocolConfig, Error> {
