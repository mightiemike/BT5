### Title
Precommit Vote Signature Preimage Omits Voter Address, Height, Round, and Chain ID — (`File: crates/apollo_signature_manager/src/signature_manager.rs`)

### Summary
`build_precommit_vote_message_digest` hashes only a domain tag and the `block_hash`. The `voter` address, `height`, `round`, `vote_type`, and chain ID present in the `Vote` struct are never committed into the signature. This means a valid precommit signature is reusable across heights, rounds, and chains for the same block hash, and the `voter` field is entirely self-reported and uncovered by the cryptographic commitment — a direct analog to H01's "commitment not tied to the voter."

### Finding Description

`build_precommit_vote_message_digest` constructs the signed payload as:

```
PRECOMMIT_VOTE || block_hash.to_bytes_be()
``` [1](#0-0) 

The `Vote` struct carries `vote_type`, `height`, `round`, `proposal_commitment`, and `voter` as separate fields: [2](#0-1) 

None of those fields — voter address, height, round, vote type — are included in the message digest that is signed or verified. The signing entry point confirms only `block_hash` is passed: [3](#0-2) 

And the verification entry point mirrors this, accepting only `block_hash` + `public_key`: [4](#0-3) 

Signature verification in the consensus vote handler is currently a `TODO`: [5](#0-4) 

When that TODO is resolved using the existing `verify_precommit_vote_signature` API, the verification will succeed for any `Vote` message that carries the correct `block_hash` and a signature produced by the claimed validator's key — regardless of what `height`, `round`, or `voter` the message declares.

### Impact Explanation

An attacker who observes a legitimate precommit from validator A for `block_hash = X` at `(height=H, round=R)` can:

1. **Cross-height/round replay**: Construct a new `Vote { voter: A, height: H', round: R', proposal_commitment: X, signature: <copied> }`. Because the signature only covers `PRECOMMIT_VOTE || X`, it verifies correctly against A's public key at any `(H', R')` where the same block hash appears.
2. **Cross-chain replay**: The same signature is valid on any chain that produces the same `block_hash`, since chain ID is absent from the preimage.
3. **Voter field spoofing (future risk)**: Because `voter` is not in the preimage, any future code path that derives the public key from the `voter` field and then verifies the signature will bind the wrong signer — accepting a vote attributed to validator B that was actually signed by validator A, if both happen to share a key or if the lookup is misconfigured.

The impact matches: **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

### Likelihood Explanation

The attack requires observing a broadcast precommit vote (public P2P gossip) and replaying it with modified `height`/`round` fields. This is trivially achievable by any network participant. The only current mitigation is that signature verification is not yet enforced (`// TODO(Asmaa): verify the signature`), but the incomplete preimage is baked into the signing and verification API that will be wired in. The existing TODO comment and the `SignatureManagerRequest::SignPrecommitVote` RPC confirm this path is actively being built toward production use. [6](#0-5) 

### Recommendation

Include all vote-identifying context in the signed preimage:

```rust
fn build_precommit_vote_message_digest(
    voter: ContractAddress,
    height: BlockNumber,
    round: Round,
    vote_type: VoteType,
    chain_id: &ChainId,
    block_hash: BlockHash,
) -> MessageDigest {
    let mut message = Vec::new();
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&voter.to_bytes());
    message.extend_from_slice(&height.0.to_be_bytes());
    message.extend_from_slice(&round.to_be_bytes());
    message.extend_from_slice(&(vote_type as u8).to_be_bytes());
    message.extend_from_slice(chain_id.as_bytes());
    message.extend_from_slice(&block_hash.to_bytes_be());
    MessageDigest(blake2s_to_felt(&message))
}
```

Update `sign_precommit_vote` and `verify_precommit_vote_signature` to accept and pass these fields. This mirrors the fix applied in UMA PR#1217: binding the commitment to all relevant context so it cannot be duplicated or replayed.

### Proof of Concept

1. Validator A broadcasts a valid `Vote { vote_type: Precommit, height: 5, round: 0, proposal_commitment: X, voter: A, signature: S }`.
2. Attacker intercepts it over P2P gossip.
3. Attacker constructs `Vote { vote_type: Precommit, height: 6, round: 0, proposal_commitment: X, voter: A, signature: S }` (only `height` changed).
4. When the consensus manager calls `verify_precommit_vote_signature(X, S, pubkey_of_A)`, it computes `blake2s(PRECOMMIT_VOTE || X)` — identical to step 1 — and returns `true`.
5. The replayed vote is accepted as a fresh, valid precommit from A at height 6, round 0, allowing the attacker to influence quorum counting without A's participation. [1](#0-0) [7](#0-6)
