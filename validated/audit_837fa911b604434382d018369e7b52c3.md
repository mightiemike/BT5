### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently ignores the `sender` parameter (the actual caller who provides tokens via callback) and gates only on `owner` (the LP-position recipient). Any unprivileged address can bypass the allowlist by calling `pool.addLiquidity(allowlistedAddress, ...)`, causing the guard to pass while the unauthorized caller injects liquidity into a restricted pool.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address and passes both `msg.sender` (the actual depositor) and `owner` (the position recipient) into the extension hook: [1](#0-0) 

The extension receives `sender` as its first argument and `owner` as its second: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` discards `sender` entirely (unnamed `address`) and checks only `owner`: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller): [4](#0-3) 

The asymmetry is the root cause: the deposit guard checks the wrong identity.

---

### Impact Explanation

An unauthorized caller (not on the allowlist) can:

1. Call `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)`.
2. The extension evaluates `allowedDepositor[pool][allowlistedAddress]` → `true` → no revert.
3. The unauthorized caller satisfies the token pull via `metricOmmSwapCallback`.
4. LP shares are minted to `allowlistedAddress`; the unauthorized caller has injected liquidity into a pool that was supposed to be restricted.

Consequences:
- **Broken allowlist invariant**: The pool admin's deposit restriction is entirely ineffective. Any address can participate in a pool configured as restricted.
- **Pool state manipulation**: The unauthorized caller can target specific bins, altering `token0BalanceScaled`/`token1BalanceScaled` and `binTotals`, which directly affects oracle-anchored swap prices and LP value per share.
- **Stop-loss watermark manipulation**: If `OracleValueStopLossExtension` is also configured, injecting liquidity into specific bins changes per-share metrics and can artificially raise or suppress high-watermarks, defeating the stop-loss guard for legitimate LPs. [5](#0-4) 

---

### Likelihood Explanation

- No special privilege is required. Any EOA or contract can call `addLiquidity` on any pool.
- The attacker only needs to know one allowlisted address (publicly readable from `allowedDepositor` or event logs).
- The attacker bears the token cost of the deposit, but the pool state manipulation benefit (e.g., moving bin balances to influence subsequent swaps they execute) can exceed that cost.
- The `removeLiquidity` path enforces `msg.sender == owner`, so the attacker cannot reclaim the deposited tokens — but the pool-state distortion persists. [6](#0-5) 

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor) instead of `owner`:

```solidity
// current (wrong):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}

// fixed:
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

If the intent is to gate both the caller and the position recipient, both should be checked. The fix must match the semantic intent of the allowlist (gating the party who provides tokens, not the party who receives shares). [3](#0-2)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Assume:
//   pool          = address of a MetricOmmPool with DepositAllowlistExtension configured
//   allowlisted   = an address in allowedDepositor[pool]
//   attacker      = address NOT in allowedDepositor[pool]

contract AttackerBypassDeposit {
    IMetricOmmPool pool;
    IERC20 token0;
    IERC20 token1;

    constructor(address pool_, address t0, address t1) {
        pool = IMetricOmmPool(pool_);
        token0 = IERC20(t0);
        token1 = IERC20(t1);
    }

    function exploit(address allowlisted, LiquidityDelta calldata deltas) external {
        // attacker is NOT allowlisted, but passes allowlisted as owner
        // DepositAllowlistExtension checks allowedDepositor[pool][allowlisted] → true → no revert
        pool.addLiquidity(
            allowlisted,   // owner: allowlisted address — passes the guard
            0,             // salt
            deltas,
            abi.encode(address(this)),  // callbackData
            ""             // extensionData
        );
        // LP shares minted to `allowlisted`; attacker has injected liquidity into restricted pool
    }

    // Pool pulls tokens from attacker via this callback
    function metricOmmSwapCallback(int128 amount0Delta, int128 amount1Delta, bytes calldata) external {
        if (amount0Delta > 0) token0.transfer(msg.sender, uint128(amount0Delta));
        if (amount1Delta > 0) token1.transfer(msg.sender, uint128(amount1Delta));
    }
}
```

The `beforeAddLiquidity` hook fires with `owner = allowlisted`, the check passes, and the unauthorized attacker successfully injects liquidity into the restricted pool.
