### Title
`ENABLE_SECP_OPS` Flag Bypass via 4-Byte Opcode Aliases for Secp Verification — (`File: src/chia_dialect.rs`)

---

### Summary

In `ChiaDialect::op()`, the `ENABLE_SECP_OPS` flag gates the 1-byte opcodes 64 (`secp256k1_verify`) and 65 (`secp256r1_verify`). However, the same underlying functions are also reachable via the 4-byte opcode aliases `0x13d61f00` and `0x1c3a8f00`, which are dispatched in a separate branch that performs **no flag check at all**. An attacker-controlled CLVM program using the 4-byte form executes secp verification regardless of whether `ENABLE_SECP_OPS` is set.

---

### Finding Description

`ChiaDialect::op()` in `src/chia_dialect.rs` dispatches operators through two distinct branches depending on the byte-length of the opcode atom.

**Branch 1 — 4-byte opcodes (lines 157–182):** The function first checks `op_len == 4`. For the two recognized 4-byte values it dispatches directly to the secp functions with no flag check: [1](#0-0) 

**Branch 2 — 1-byte opcodes (lines 248–249):** The same two secp functions are also reachable via 1-byte opcodes 64 and 65, but here the dispatch is guarded by `ENABLE_SECP_OPS`: [2](#0-1) 

The flag itself is defined as: [3](#0-2) 

`MEMPOOL_MODE` — the stricter evaluation mode used for transaction validation — does **not** include `ENABLE_SECP_OPS`: [4](#0-3) 

This means that in mempool mode, opcode 64 or 65 falls through to `unknown_operator()` and is rejected (because `NO_UNKNOWN_OPS` is also set). But the 4-byte aliases `0x13d61f00` / `0x1c3a8f00` are matched in the `op_len == 4` branch **before** any unknown-operator logic is reached, and are dispatched unconditionally to `op_secp256k1_verify` / `op_secp256r1_verify`.

The underlying secp functions themselves accept `_flags: ClvmFlags` but ignore it entirely: [5](#0-4) 

---

### Impact Explanation

Any caller that constructs a `ChiaDialect` **without** `ENABLE_SECP_OPS` — including the standard `MEMPOOL_MODE` configuration — intends to prevent secp signature verification from executing. A CLVM program that uses the 4-byte opcode form bypasses this intent completely. The concrete corrupted result is: `op_secp256k1_verify` or `op_secp256r1_verify` executes and returns `Reduction(cost, nil)` (success) or a secp-failure error, when the caller's flag configuration mandates the operation should be treated as an unknown/unimplemented opcode. This is a flag/operator wiring error producing a **consensus or mempool-mode divergence**: two nodes evaluating the same program bytes under the same nominal flag set may reach different accept/reject decisions depending on which opcode form the program uses.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM program submitted to the mempool or evaluated by a node can embed the 4-byte opcode bytes `\x13\xd6\x1f\x00` or `\x1c\x3a\x8f\x00` as the operator atom. No special privilege is required. The bypass is mechanical and deterministic — it requires only knowledge of the two 4-byte opcode values, which are documented in the source comments themselves.

---

### Recommendation

Add an `ENABLE_SECP_OPS` guard to the 4-byte dispatch branch, mirroring the guard already present on the 1-byte path:

```rust
// in the op_len == 4 branch, before dispatching:
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that regardless of which opcode encoding a program uses, the `ENABLE_SECP_OPS` flag is the single authoritative gate for secp execution.

---

### Proof of Concept

A CLVM program encoding the operator atom as the 4-byte value `0x13d61f00` (secp256k1_verify) with valid pubkey/msg/sig arguments, run under `MEMPOOL_MODE` (which omits `ENABLE_SECP_OPS`), will execute the secp verification and return `nil` on success — whereas the same program using 1-byte opcode `0x40` (64) would be rejected with `Unimplemented` due to `NO_UNKNOWN_OPS`. The divergence is triggered purely by the choice of opcode encoding in the attacker-supplied CLVM bytes, with no other change to the program or environment. [6](#0-5) [7](#0-6)
