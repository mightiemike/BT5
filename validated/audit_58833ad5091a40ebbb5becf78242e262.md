### Title
Unguarded `initialize` on Engine Implementation Contracts Allows Unauthorized Ownership Takeover — (`File: core/contracts/SpotEngine.sol`, `File: core/contracts/PerpEngine.sol`)

---

### Summary
`SpotEngine` and `PerpEngine` expose public `initialize` functions that carry **no `initializer` modifier** at the function level. The only re-entrancy lock exists deep inside the internal `BaseEngine._initialize`, and neither contract calls `_disableInitializers()` in a constructor. Because the implementation contracts are never self-initialized at deploy time, any unprivileged caller can invoke `initialize` directly on the implementation, seize ownership, and call privileged `onlyOwner` functions against the live protocol.

---

### Finding Description

`SpotEngine.initialize` and `PerpEngine.initialize` are declared `external` with no modifier: [1](#0-0) [2](#0-1) 

The only initialization guard is the `initializer` modifier placed on the **internal** `BaseEngine._initialize`: [3](#0-2) 

Neither `SpotEngine` nor `PerpEngine` defines a constructor that calls `_disableInitializers()`. Contrast this with contracts that do protect themselves: [4](#0-3) [5](#0-4) 

Because no constructor exists in `SpotEngine` or `PerpEngine`, the OpenZeppelin `_initialized` flag on the **implementation** address is `0` at deployment. The proxy is initialized correctly, but the bare implementation is not. An attacker can therefore call `SpotEngine.initialize(...)` or `PerpEngine.initialize(...)` directly on the implementation contract, satisfying the `initializer` guard in `_initialize` on its first execution and completing the call successfully.

`_initialize` then:
- calls `__Ownable_init()` — setting `owner` to the attacker
- calls `transferOwnership(_admin)` — where `_admin` is attacker-supplied
- sets `_clearinghouse`, `endpoint`, and `canApplyDeltas` to attacker-controlled addresses [6](#0-5) 

---

### Impact Explanation

Once the attacker owns the implementation contract, they can invoke every `onlyOwner`-gated function on it:

- `addOrUpdateProduct` — registers new products and calls `_clearinghouse.registerProduct(productId)` and `IEndpoint.setInitialPrice`. With attacker-controlled `_clearinghouse` and `endpoint` set during the malicious `initialize`, these calls go to attacker contracts.
- `updateRisk` — overwrites `_risk().value[productId]` (stored at the deterministic `RISK_STORAGE` slot), corrupting the risk parameters that health calculations depend on. [7](#0-6) [8](#0-7) 

The corrupted ownership assumption is concrete: the `owner` slot of the implementation is now attacker-controlled, and any future upgrade that re-uses the implementation's storage (or any `delegatecall` path that reads `owner`) inherits the corrupted state.

---

### Likelihood Explanation

The implementation contract addresses are publicly visible on-chain. Any unprivileged caller can send a transaction to `SpotEngine.initialize(...)` on the implementation immediately after deployment, before any protocol-level initialization of the proxy. No special privilege, leaked key, or governance capture is required. The entry path is a direct external call to a public function with no access control at the function signature level.

---

### Recommendation

Add a constructor with `_disableInitializers()` to both `SpotEngine` and `PerpEngine`, and place the `initializer` modifier directly on their public `initialize` functions rather than relying solely on the internal `_initialize`:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}

function initialize(...) external initializer {
    _initialize(...);
}
```

This matches the pattern already correctly applied in `Verifier`, `Airdrop`, `BaseWithdrawPool`, `ContractOwner`, and `BaseProxyManager`.

---

### Proof of Concept

1. Deploy the protocol normally. The `SpotEngine` proxy is initialized; the `SpotEngine` implementation at address `implAddr` is not.
2. Attacker calls:
   ```solidity
   SpotEngine(implAddr).initialize(
       address(maliciousClearinghouse),
       address(maliciousOffchainExchange),
       address(maliciousQuote),
       address(maliciousEndpoint),
       attacker          // _admin → becomes owner
   );
   ```
3. `BaseEngine._initialize` executes: `__Ownable_init()` sets `owner = attacker`, `transferOwnership(attacker)` confirms it.
4. Attacker calls `SpotEngine(implAddr).updateRisk(productId, maliciousRiskStore)` — succeeds because `msg.sender == owner`.
5. The `_risk().value[productId]` slot on the implementation is now corrupted with attacker-chosen weights and price. [1](#0-0) [2](#0-1) [3](#0-2)
