### Title
Accumulated Fast Withdrawal Fees in `BaseWithdrawPool` Have No Withdrawal Mechanism — (`File: core/contracts/BaseWithdrawPool.sol`)

### Summary
The `BaseWithdrawPool` contract collects real ERC20 tokens as fast withdrawal fees and tracks them in the `fees` mapping, but provides no function to withdraw or claim those accumulated fees. The `fees` mapping grows monotonically and is never decremented, making the collected fee tokens permanently inaccessible through any intended fee-management path.

### Finding Description
In `submitFastWithdrawal`, fees are collected in two ways:

1. When `sendTo == msg.sender` (fast withdrawal provider is also the recipient): `transferAmount -= uint128(fee)` retains the fee portion inside the contract's ERC20 balance.
2. When `sendTo != msg.sender`: `safeTransferFrom(token, msg.sender, uint128(fee))` pulls the fee directly from the caller into the contract.

In both cases, `fees[productId] += fee` records the accumulation. [1](#0-0) [2](#0-1) 

Neither `BaseWithdrawPool` nor its concrete child `WithdrawPool` defines any function that reads `fees[productId]` to transfer tokens out or reset the counter. [3](#0-2) 

The only token-exit path is `removeLiquidity`, which is a blunt `onlyOwner` function that transfers an arbitrary amount of a product's token to an arbitrary address without consulting or decrementing `fees[productId]`. [4](#0-3) 

This means:
- The `fees` mapping is purely decorative — it accumulates but is never consumed.
- The actual fee tokens sit in the contract's ERC20 balance with no dedicated retrieval path.
- `removeLiquidity` conflates fee tokens with liquidity-provider tokens; using it to recover fees silently corrupts the liquidity accounting without resetting `fees[productId]`.

### Impact Explanation
Fast withdrawal fees paid by every fast-withdrawal provider accumulate in the `WithdrawPool` contract indefinitely. The protocol has no on-chain mechanism to claim these as revenue. If `removeLiquidity` is used as a workaround, the `fees` mapping diverges from the actual token balance, breaking any off-chain or future on-chain accounting that relies on it. In the worst case, an operator attempting to recover fees via `removeLiquidity` may inadvertently drain liquidity-provider funds because the two pools of tokens are indistinguishable at the ERC20 balance level.

### Likelihood Explanation
Every call to `submitFastWithdrawal` by any unprivileged fast-withdrawal provider triggers fee collection. The contract is already deployed and the fee accumulation path is exercised in normal protocol operation. The absence of a withdrawal function is a structural gap, not a conditional one. [5](#0-4) 

### Recommendation
Add a dedicated `withdrawFees(uint32 productId, address recipient)` function (with appropriate access control, e.g., `onlyOwner`) that:
1. Reads `fees[productId]`.
2. Resets `fees[productId] = 0`.
3. Transfers exactly that amount of the product's token to `recipient`.

This mirrors the GMX resolution: a withdraw function with proper access controls that operates on the tracked fee balance rather than the raw contract balance.

### Proof of Concept
1. Alice (fast withdrawal provider) calls `submitFastWithdrawal` with `sendTo != msg.sender`. The contract pulls `fee` tokens from Alice via `safeTransferFrom` and records `fees[productId] += fee`.
2. Bob calls `submitFastWithdrawal` with `sendTo == msg.sender`. The contract retains `fee` tokens from the withdrawal amount and records `fees[productId] += fee`.
3. After many such calls, `fees[productId]` holds a large accumulated value and the contract holds the corresponding ERC20 tokens.
4. Searching `BaseWithdrawPool` and `WithdrawPool` for any function that reads `fees[productId]` to transfer tokens yields zero results — the mapping is write-only from a token-flow perspective.
5. The only exit is `removeLiquidity(productId, amount, sendTo)`, which does not reference `fees[productId]` and does not reset it, leaving the accounting permanently stale after any manual recovery attempt. [4](#0-3) [6](#0-5)

```text
    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
```
