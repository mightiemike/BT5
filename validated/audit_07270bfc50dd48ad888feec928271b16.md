### Title
Silent Bonus Reward Forfeiture with Permanent Checkpoint Advancement in `BeforeRedelegationSlashed` When Bonus Pool Is Insufficient — (`x/tieredrewards/keeper/slash.go`)

---

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is deliberately swallowed to prevent a chain halt. However, because `pos` is passed by pointer, the in-memory checkpoint mutations made inside `processEventsAndClaimBonus` (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) are already reflected in the caller's `pos` variable. The function then calls `setPositionWithState` with the mutated `pos`, permanently persisting the advanced checkpoints. The staker's accrued bonus rewards for the pre-slash period are silently forfeited with no ability to retry.

---

### Finding Description

`slashRedelegationPosition` is invoked from the `BeforeRedelegationSlashed` staking hook. Its purpose is to settle bonus rewards against pre-slash shares before the SDK's `Unbond` reduces them. [1](#0-0) 

When `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is caught and only logged. Execution continues to `setPositionWithState`.

The critical issue is that `processEventsAndClaimBonus` receives `pos` by pointer (`&pos`): [2](#0-1) 

Inside `processEventsAndClaimBonus`, before the pool-balance check, the function has already:

1. Called `pos.UpdateLastEventSeq(entry.Seq)` for every event in the loop: [3](#0-2) 

2. Called `applyBonusAccrualCheckpoint` advancing `pos.LastBonusAccrual` to `blockTime`: [4](#0-3) 

3. Decremented event reference counts in the store (potentially garbage-collecting events).

The pool-balance check that fails is at: [5](#0-4) 

When this returns `nil, err`, the function exits without paying the bonus — but all the in-memory mutations to `*pos` are already done. Back in `slashRedelegationPosition`, after swallowing the error, the code falls through to: [6](#0-5) 

`setPositionWithState` is called with the mutated `pos`, persisting `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded` at their post-accrual values — even though no bonus was ever paid.

The ADR documents this as an intentional design choice: [7](#0-6) 

However, the design fails to account for the fact that the checkpoint advancement itself is the irreversible harm: the accrual window is permanently consumed without payment.

---

### Impact Explanation

The corrupted state is `pos.LastBonusAccrual` (advanced to `blockTime` without payment) and the decremented event `ReferenceCount` values. After the slash, the staker's position records a checkpoint past the accrual period. Any subsequent `MsgClaimTierRewards` call will compute zero bonus for the already-consumed window. The staker permanently loses all bonus rewards accrued between the previous `LastBonusAccrual` and the slash block time, with no mechanism to recover them.

---

### Likelihood Explanation

Two conditions must coincide: (1) a validator slash that triggers `BeforeRedelegationSlashed` for a tier position (requires a redelegating position on a validator that gets slashed for double-signing — a realistic protocol event), and (2) the bonus pool being insufficient at that block. The bonus pool is an externally funded module account that can be depleted through normal reward payouts. Both conditions are operationally plausible and can occur without any attacker action; a staker with a redelegating position is simply a victim of timing.

---

### Recommendation

When `ErrInsufficientBonusPool` is returned from `processEventsAndClaimBonus` in the slash hook, do not persist the mutated checkpoints. Options:

1. **Snapshot and restore**: Save `pos.LastBonusAccrual`, `pos.LastEventSeq`, and `pos.LastKnownBonded` before calling `processEventsAndClaimBonus`, and restore them if `ErrInsufficientBonusPool` is returned, so `setPositionWithState` persists the original checkpoints.
2. **Cache context**: Run `processEventsAndClaimBonus` in a branched cache context; commit only on success, discard on pool-insufficient error (note: event ref-count decrements would also need to be rolled back).
3. **Decouple checkpoint advancement from payment**: Advance checkpoints only after `SendCoinsFromModuleToAccount` succeeds.

---

### Proof of Concept

1. Alice creates a tier position and redelegates it to validator V2 via `MsgTierRedelegate`. A `RedelegationMappings` entry is created.
2. Time passes; Alice accrues bonus rewards.
3. The bonus pool is depleted (e.g., by other stakers claiming rewards).
4. Validator V2 double-signs. The staking module calls `BeforeRedelegationSlashed` → `slashRedelegationPosition`.
5. `processEventsAndClaimBonus` computes Alice's accrued bonus, advances `pos.LastBonusAccrual` to the current block time in-memory, decrements event reference counts in the store, then hits `sufficientBonusPoolBalance` and returns `ErrInsufficientBonusPool`.
6. `slashRedelegationPosition` logs the error and continues. `setPositionWithState` persists `pos` with the advanced `LastBonusAccrual`.
7. The pool is later replenished. Alice calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` computes zero bonus (the accrual window from step 2 is already consumed by the persisted checkpoint). Alice receives no bonus for the entire pre-slash period. [8](#0-7) [9](#0-8)
