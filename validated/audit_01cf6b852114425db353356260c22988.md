### Title
Stale `linkedSigner` Not Cleared After Subaccount Finalization Enables Persistent Unauthorized Signing Authority ‚Äî (File: `core/contracts/EndpointStorage.sol`, `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

The `linkedSigners` mapping in `EndpointStorage` is never cleared when a subaccount is liquidated and finalized. If the subaccount owner re-deposits into the same subaccount after finalization, the stale linked signer retains full signing authority ‚Äî including the ability to sign new `LinkSigner` transactions to permanently re-entrench their access, and to sign `WithdrawCollateral` transactions to drain collateral.

---

### Finding Description

In Nado, a subaccount owner can delegate signing authority to a `linkedSigner` via the `LinkSigner` transaction type. This address is stored in the `linkedSigners` mapping in `EndpointStorage`: [1](#0-0) 

The linked signer is set via the sequencer path in `EndpointTx.processTransactionImpl`: [2](#0-1) 

Or via the slow-mode path: [3](#0-2) 

The linked signer is accepted as a valid signer for all sequencer-path transactions where `allowLinkedSigner = true`, including `WithdrawCollateral`, `WithdrawCollateralV2` (when `sendTo == address(0)`), `LiquidateSubaccount`, `MatchOrders`, `TransferQuote`, `MintNlp`, `BurnNlp`, and ‚Äî critically ‚Äî `LinkSigner` itself: [4](#0-3) 

When a subaccount is liquidated and finalized via `_finalizeSubaccount` in `ClearinghouseLiq`, the function zeroes out all balances and socializes losses, but **never touches `linkedSigners`**: [5](#0-4) 

The `ClearinghouseLiq` contract has no reference to the `linkedSigners` storage slot (which lives in `EndpointStorage` on the `Endpoint` contract), and no callback to clear it is made anywhere in the liquidation flow. After finalization, `linkedSigners[subaccount]` retains its stale value indefinitely.

Because Nado subaccounts are identified by a `bytes32` encoding the owner's address, the same subaccount identifier is reused if the owner re-deposits. At that point, the stale linked signer immediately regains full signing authority over the re-funded subaccount.

The most dangerous capability of the stale linked signer is signing a new `LinkSigner` transaction (sequencer path, `allowLinkedSigner = true`) to change the linked signer to a fresh attacker-controlled address before the owner notices: [6](#0-5) 

This creates a persistent backdoor: even if the owner later tries to revoke by submitting their own `LinkSigner`, the attacker can race to re-set it via the sequencer path.

Additionally, the stale linked signer can sign `WithdrawCollateral` to drain collateral to the owner's address (disrupting positions), and can sign orders via `MatchOrders` to trade at unfavorable prices against an attacker-controlled counterparty subaccount: [7](#0-6) 

---

### Impact Explanation

**Impact: Medium.**

The stale linked signer cannot directly redirect `WithdrawCollateral` funds to an arbitrary address ‚Äî V1 always sends to the owner's address, and V2 only allows the linked signer when `sendTo == address(0)`: [8](#0-7) [9](#0-8) 

However, the stale linked signer can:
1. **Re-entrench access** by signing a new `LinkSigner` to a fresh attacker address, making revocation a race condition against the sequencer.
2. **Drain collateral** to the owner's address at a time that causes the subaccount to become undercollateralized, triggering liquidation of the re-funded account.
3. **Sign orders** at unfavorable prices against an attacker-controlled counterparty, extracting value from the subaccount (requires sequencer inclusion of the matched orders).

The concrete corrupted state is `linkedSigners[subaccount]` remaining non-zero after finalization, granting an unauthorized address persistent signing authority over a re-funded subaccount.

---

### Likelihood Explanation

**Likelihood: Medium.**

The preconditions are:
1. The subaccount owner previously set a `linkedSigner` (common for users using trading bots or market-making integrations).
2. The subaccount is liquidated and finalized.
3. The owner re-deposits into the same subaccount without explicitly clearing the linked signer.

Step 3 is the most likely failure mode: users who are liquidated and re-deposit are focused on re-establishing positions, not on reviewing stale authorization state. The protocol provides no warning or automatic cleanup.

---

### Recommendation

In `ClearinghouseLiq._finalizeSubaccount`, after zeroing all balances, emit a callback or event that causes the `Endpoint` to clear `linkedSigners[txn.liquidatee]`. Alternatively, add a `clearLinkedSigner(bytes32 subaccount)` function callable only by the `Clearinghouse`/`ClearinghouseLiq` (via the `Endpoint`) and invoke it at the end of `_finalizeSubaccount`.

A simpler mitigation: in `getLinkedSigner`, return `address(0)` if the subaccount has no active balances (i.e., has been finalized and not yet re-funded). However, the cleanest fix is explicit clearing at finalization time.

---

### Proof of Concept

1. Alice sets `linkedSigners[aliceSub] = mallory` via a `LinkSigner` transaction.
2. Alice's subaccount `aliceSub` falls below maintenance health and is liquidated. `_finalizeSubaccount` is called, zeroing all balances. `linkedSigners[aliceSub]` is **not** cleared.
3. Alice re-deposits collateral into `aliceSub` (same `bytes32` identifier).
4. Mallory, observing the deposit on-chain, immediately submits a sequencer-path `LinkSigner` transaction signed by `mallory` (the current linked signer), setting `linkedSigners[aliceSub] = mallory2` (a fresh attacker address).
5. Even if Alice submits her own `LinkSigner` to revoke, Mallory can race via the sequencer to re-set it.
6. Mallory2 now has persistent signing authority over Alice's re-funded subaccount and can sign `WithdrawCollateral` to drain collateral to Alice's address (disrupting her positions) or sign orders at unfavorable prices against a Mallory-controlled counterparty subaccount.

The root cause ‚Äî `linkedSigners[aliceSub]` surviving `_finalizeSubaccount` ‚Äî is confirmed at: [5](#0-4) 

with the stale signer lookup at: [10](#0-9)

    function _finalizeSubaccount(
        IEndpoint.LiquidateSubaccount calldata txn,
        ISpotEngine spotEngine,
        IPerpEngine perpEngine
    ) internal returns (bool) {
        if (txn.productId != type(uint32).max) {
            return false;
        }
        // check whether the subaccount can be finalized:
        // - all perps positions have closed
        // - all spread positions have closed
        // - all spot assets have closed
        // - all positive pnls have been settled

        FinalizeVars memory v;

        v.spotIds = spotEngine.getProductIds();
        v.perpIds = perpEngine.getProductIds();

        require(v.spotIds[0] == QUOTE_PRODUCT_ID);

        // all spot assets (except USDC) must be closed out
        for (uint32 i = 1; i < v.spotIds.length; ++i) {
            uint32 spotId = v.spotIds[i];
            if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
                continue;
            }
            ISpotEngine.Balance memory balance = spotEngine.getBalance(
                spotId,
                txn.liquidatee
            );
            require(balance.amount <= 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
        }

        for (uint32 i = 0; i < v.perpIds.length; ++i) {
            uint32 perpId = v.perpIds[i];
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                perpId,
                txn.liquidatee
            );
            require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
        }

        // settle all positive pnl
        for (uint32 i = 0; i < v.perpIds.length; ++i) {
            uint32 perpId = v.perpIds[i];
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                perpId,
                txn.liquidatee
            );
            if (balance.vQuoteBalance > 0) {
                _settlePnlAgainstLiquidator(
                    txn,
                    perpId,
                    balance.vQuoteBalance,
                    spotEngine,
                    perpEngine
                );
            }
        }

        ISpotEngine.Balance memory quoteBalance = spotEngine.getBalance(
            QUOTE_PRODUCT_ID,
            txn.liquidatee
        );

        // settle all negative pnl until quote balance becomes 0
        for (uint32 i = 0; i < v.perpIds.length; ++i) {
            uint32 perpId = v.perpIds[i];
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                perpId,
                txn.liquidatee
            );
            if (balance.vQuoteBalance < 0 && quoteBalance.amount > 0) {
                int128 canSettle = MathHelper.max(
                    balance.vQuoteBalance,
                    -quoteBalance.amount
                );
                _settlePnlAgainstLiquidator(
                    txn,
                    perpId,
                    canSettle,
                    spotEngine,
                    perpEngine
                );
                quoteBalance.amount += canSettle;
            }
        }

        v.insurance = insurance;
        v.insurance -= lastLiquidationFees;
        v.canLiquidateMore = (quoteBalance.amount + v.insurance) > 0;

        if (v.canLiquidateMore) {
            for (uint32 i = 1; i < v.spotIds.length; ++i) {
                uint32 spotId = v.spotIds[i];
                ISpotEngine.Balance memory balance = spotEngine.getBalance(
                    spotId,
                    txn.liquidatee
                );
                if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
                    continue;
                }
                require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
            }
        }

        v.insurance = perpEngine.socializeSubaccount(
            txn.liquidatee,
            v.insurance
        );

        // we can assure that quoteBalance must be non positive, because if quoteBalance.amount > 0,
        // there must be 1) no negative pnl in perps, and 2) no liabilities in spot after above actions.
        // however, in this case the liquidatee must be healthy and cannot pass the health check at
        // the beginning.
        int128 insuranceCover = MathHelper.min(
            v.insurance,
            -quoteBalance.amount
        );
        if (insuranceCover > 0) {
            v.insurance -= insuranceCover;
            spotEngine.updateBalance(
                QUOTE_PRODUCT_ID,
                txn.liquidatee,
                insuranceCover
            );
        }
        if (v.insurance <= 0) {
            spotEngine.socializeSubaccount(txn.liquidatee);
        }
        v.insurance += lastLiquidationFees;
        insurance = v.insurance;
        return true;
    }
