### Title
Linked Signer Can Self-Escalate by Signing a `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner=true`. Because `validateSignature` accepts either the subaccount owner **or** the current `linkedSigner` as a valid signer, an existing linked signer can sign a new `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with an attacker-controlled address — without any involvement from the subaccount owner.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch decodes a `SignedLinkSigner` and validates it with:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // <-- allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
``` [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes `getLinkedSigner(sender)` as the second acceptable signer when `allowLinkedSigner=true`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` then accepts the signature if the recovered address equals **either** the subaccount owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

There is no additional guard that restricts `LinkSigner` mutations to the subaccount owner only. The slow-mode path does enforce `validateSender` (an `msg.sender` check), but the fast/sequencer path has no equivalent owner-only restriction. [4](#0-3) 

---

### Impact Explanation

A compromised or malicious linked signer (`signerA`) can:
1. Construct a `SignedLinkSigner` transaction: `{sender: victimSubaccount, signer: attackerAddress, nonce: N}`.
2. Sign it with `signerA`'s key.
3. Submit it through the sequencer's `processTransactionImpl`.
4. `linkedSigners[victimSubaccount]` is overwritten with `attackerAddress`.

From this point the attacker controls the linked signer slot. They can sign `WithdrawCollateral`, `TransferQuote`, or any other signed-tx flow that accepts `allowLinkedSigner=true`, draining the subaccount. The original owner's ability to use signed-tx flows is also disrupted because their linked signer is now attacker-controlled.

---

### Likelihood Explanation

Any party that has ever been granted linked-signer status (e.g., a trading bot key, a compromised API key, a former operator) can execute this unilaterally. No owner cooperation, no governance action, and no sequencer compromise is required — only the ability to submit a transaction through the normal sequencer flow.

---

### Recommendation

Pass `allowLinkedSigner=false` when validating `LinkSigner` transactions so that only the subaccount owner (the address embedded in `sender`) can mutate the signer-link relationship:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [5](#0-4) 

---

### Proof of Concept

1. Deploy the protocol locally (Hardhat).
2. Link `signerA` to `subaccountS` via a valid owner-signed `LinkSigner` tx → `linkedSigners[subaccountS] == signerA`.
3. With `signerA`'s key, construct and sign: `SignedLinkSigner { tx: { sender: subaccountS, signer: attackerAddress, nonce: currentNonce }, signature: signerA_sig }`.
4. Submit via `processTransactionImpl`.
5. `validateSignedTx` passes because `ECDSA.recover(digest, signerA_sig) == signerA == getLinkedSigner(subaccountS)`.
6. Assert: `linkedSigners[subaccountS] == attackerAddress`. ✓
7. Now sign a `WithdrawCollateral` with `attackerAddress`'s key → funds drained.
