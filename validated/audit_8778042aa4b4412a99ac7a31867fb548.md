### Title
`max_atom_len` and `strict` Parameters Silently Dropped for Legacy/Backrefs Blobs in `deser_auto` - (File: `wheel/src/api.rs`)

### Summary

`deser_auto` accepts `max_atom_len` and `strict` parameters but forwards them only to the serde_2026 branch. When the input blob is in legacy or backrefs format, both parameters are silently dropped and `node_from_bytes_backrefs` is called with no atom-length cap and no strictness enforcement. This is a direct wiring analog of the ETH-not-forwarded bug: a value is accepted by an outer function but never passed to the inner call that actually needs it.

### Finding Description

In `wheel/src/api.rs`, `deser_auto` is declared with `max_atom_len` and `strict` parameters:

```rust
#[pyo3(signature = (blob, *, max_atom_len=PY_DEFAULT_MAX_ATOM_LEN, strict=true))]
fn deser_auto(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
        deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)
            .map_err(eval_to_py)?
    } else {
        node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?  // ← both params dropped
    };
    Ok(LazyNode::new(Rc::new(a), node))
}
``` [1](#0-0) 

When the blob starts with the serde_2026 magic prefix, `max_atom_len` and `strict` are correctly forwarded to `deserialize_2026_body_from_stream`. [2](#0-1) 

When the blob is in legacy or backrefs format (the `else` branch), `node_from_bytes_backrefs` is called with no `max_atom_len` argument at all — the function signature does not accept one. [3](#0-2) 

`node_from_bytes_backrefs` has no configurable atom-length cap; it calls `parse_atom` which reads whatever length the blob encodes. [4](#0-3) 

### Impact Explanation

A Python caller that invokes `deser_auto(blob, max_atom_len=N, strict=True)` with a custom cap `N` (e.g., a downstream wrapper like `chia_rs` supplying its own consensus limits) will have that cap silently bypassed for any blob not starting with the serde_2026 magic prefix. An attacker who controls the input blob can:

1. **Bypass `max_atom_len`**: Submit a legacy-format blob encoding a single atom of arbitrary size (e.g., 1 GB). The allocator will attempt to satisfy the allocation, potentially causing OOM or exhausting the heap limit, regardless of the caller-supplied cap.
2. **Bypass `strict`**: Non-canonical atom-length encodings in the legacy format are not rejected even when `strict=True` is passed.

The corrupted invariant is: the caller's `max_atom_len` constraint is not enforced for the legacy/backrefs code path, producing a `LazyNode` backed by an atom that violates the caller's stated size policy.

### Likelihood Explanation

The function is exposed as a Python API entry point and is the recommended convenience function for format-agnostic deserialization. The docstring explicitly notes that "consensus-aware callers should sniff the prefix themselves and use their own caps" — but `deser_auto` is precisely the function meant to abstract that sniffing. Any caller that passes a custom `max_atom_len` smaller than `PY_DEFAULT_MAX_ATOM_LEN` (1 MiB) and receives attacker-controlled bytes in legacy format is affected. The attacker only needs to ensure the blob does not start with `fd ff 32 30 32 36`. [5](#0-4) 

### Recommendation

Forward both parameters to the legacy/backrefs branch. Since `node_from_bytes_backrefs` does not accept `max_atom_len`, either:

1. Add a `max_atom_len`-aware variant of `node_from_bytes_backrefs` and call it here, or
2. Wrap the call with a post-parse atom-size validation walk, or
3. Inline the prefix check in the caller and document that `deser_auto` does **not** enforce `max_atom_len` for legacy blobs (and rename/deprecate accordingly).

The minimal fix matching the pattern of the reference report is:

```rust
} else {
    node_from_bytes_backrefs_bounded(&mut a, blob, max_atom_len).map_err(eval_to_py)?
};
```

where `node_from_bytes_backrefs_bounded` enforces the atom-length cap during parsing.

### Proof of Concept

```python
import clvm_rs

# Craft a legacy blob with a 2 MB atom (length prefix 0x82 0x20 0x00 followed by 2MB of zeros)
# Legacy format: atom length >= 0x80 uses multi-byte header
atom_len = 2 * 1024 * 1024  # 2 MB, exceeds max_atom_len=1024
header = bytes([0x8E]) + atom_len.to_bytes(3, 'big')  # simplified; actual encoding varies
large_atom_blob = header + b'\x00' * atom_len

# Caller expects max_atom_len=1024 to be enforced
node = clvm_rs.deser_auto(large_atom_blob, max_atom_len=1024, strict=True)
# No error raised; a 2 MB atom is silently allocated, violating the caller's cap.
# For deser_2026 with the same cap, this would raise an error.
```

The asymmetry is the root cause: `deser_2026` and the serde_2026 branch of `deser_auto` enforce `max_atom_len`; the legacy branch of `deser_auto` does not. [6](#0-5)

fn node_from_stream_backrefs_old(
    allocator: &mut Allocator,
    f: &mut Cursor<&[u8]>,
    mut backref_callback: impl FnMut(NodePtr),
) -> Result<NodePtr> {
    let mut values = allocator.nil();
    let mut ops = vec![ParseOp::SExp];

    let mut b = [0; 1];
    while let Some(op) = ops.pop() {
        match op {
            ParseOp::SExp => {
                f.read_exact(&mut b)?;
                if b[0] == CONS_BOX_MARKER {
                    ops.push(ParseOp::Cons);
                    ops.push(ParseOp::SExp);
                    ops.push(ParseOp::SExp);
                } else if b[0] == BACK_REFERENCE {
                    let path = parse_path(f)?;
                    let reduction = traverse_path(allocator, path, values)?;
                    let back_reference = reduction.1;
                    backref_callback(back_reference);
                    values = allocator.new_pair(back_reference, values)?;
                } else {
                    let new_atom = parse_atom(allocator, b[0], f)?;
                    values = allocator.new_pair(new_atom, values)?;
                }
            }
            ParseOp::Cons => {
                // cons
                // pop left and right values off of the "values" stack, then
                // push the new pair onto it
                let SExp::Pair(right, rest) = allocator.sexp(values) else {
                    panic!("internal error");
                };
                let SExp::Pair(left, rest) = allocator.sexp(rest) else {
                    panic!("internal error");
                };
                let new_root = allocator.new_pair(left, right)?;
                values = allocator.new_pair(new_root, rest)?;
            }
        }
    }
    match allocator.sexp(values) {
        SExp::Pair(v1, _v2) => Ok(v1),
        _ => panic!("unexpected atom"),
    }
}
