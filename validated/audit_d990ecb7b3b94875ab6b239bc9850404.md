### Title
Difficulty Calculation for Fork Blocks Uses Mainchain Epoch-Boundary Block Instead of Fork's Actual Ancestor, Enabling Invalid Header Acceptance — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

### Summary

In `bitcoin.rs`, `litecoin.rs`, and `dogecoin.rs`, the `get_next_work_required` function fetches the epoch-boundary (retarget-start) block via `get_header_by_height`, which unconditionally resolves height through `mainchain_height_to_header`. When a fork diverges before an epoch boundary, the fork's ancestor at that height differs from the mainchain's block at that height, but the contract silently uses the mainchain's block. This causes the expected difficulty for the fork's next epoch to be computed from the wrong block's timestamp, allowing an attacker to submit fork blocks whose `bits` field satisfies an incorrectly computed target — either too easy (enabling a low-work reorg) or too hard (blocking valid fork submissions). The developer explicitly flagged this in `dogecoin.rs` with a `TODO` comment.

---

### Finding Description

**Root cause — `get_header_by_height` is mainchain-only:** [1](#0-0) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← always mainchain
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

`mainchain_height_to_header` maps only confirmed mainchain heights to hashes. Fork blocks are stored only in `headers_pool` (keyed by hash), never in `mainchain_height_to_header`.

**Bitcoin / Litecoin — epoch-boundary lookup:** [2](#0-1) 

```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header,
    interval_tail_extend_header.block_header.time.into())
```

When `prev_block_header` is a fork block at height 4031 and the fork diverged at height 2000, `first_block_height = 2016`. `get_header_by_height(2016)` returns the **mainchain's** block at 2016, not the fork's block at 2016.

**Dogecoin — same pattern, developer-acknowledged:** [3](#0-2) 

```rust
// TODO: check if it is correct to get block header by height from mainchain
//       without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

For Dogecoin (per-block retarget after height 145 000), `height_first = prev_block_header.block_height - 1`. For a fork block at height H+2 where the fork diverged at H, `height_first = H`. The mainchain's block at H has a different timestamp than the fork's block at H.

**Litecoin — identical pattern:** [4](#0-3) 

**Contrast with Zcash — correct traversal:** [5](#0-4) 

Zcash walks the averaging window via `get_prev_header`, which follows `prev_block_hash` through `headers_pool` and therefore correctly traverses the fork's ancestor chain. Bitcoin, Litecoin, and Dogecoin do not.

### Impact Explanation

**Scenario A — lower difficulty accepted (chain-reorg enablement):**

If the mainchain's epoch-boundary block has a larger timestamp than the fork's actual ancestor at the same height (i.e., the mainchain epoch appeared slower), `calculate_next_work_required` produces a higher target (lower difficulty). The contract then enforces `expected_bits == block_header.bits` against this inflated target. The attacker submits fork blocks whose `bits` encodes the inflated target; each block requires less PoW. With enough such blocks the fork's `chain_work` can exceed the mainchain's, triggering `reorg_chain` and corrupting the canonical chain state — invalidating previously confirmed transactions visible to consumer contracts via `verify_transaction_inclusion`.

**Scenario B — higher difficulty enforced (valid fork rejection):**

If the mainchain's epoch-boundary block has a smaller timestamp, the computed target is lower (higher difficulty). Legitimate fork blocks with the correct difficulty are rejected with `"bad-diffbits"`, preventing any fork from being submitted across that epoch boundary.

Both outcomes violate the core invariant: *only headers satisfying the PoW chain's actual consensus rules are accepted*.

---

### Likelihood Explanation

- **Attacker entry point:** `submit_blocks` is gated by `#[trusted_relayer]`. The `omni_utils` trusted-relayer pattern is a staking mechanism open to any NEAR account; no privileged key is required. [7](#0-6) 
- **Trigger condition:** The attacker submits a fork that diverges before an epoch boundary (height divisible by 2016 for Bitcoin/Litecoin; any height ≥ 145 000 for Dogecoin). This is a normal relayer operation.
- **Timestamp selection:** The attacker controls the timestamps of all fork blocks they submit (within the MTP and future-time constraints). They can choose the fork's epoch-boundary block timestamp to maximize the divergence from the mainchain's block at the same height.
- **Observability:** The mainchain's epoch-boundary block timestamps are public on-chain data; the attacker can pre-compute the resulting difficulty before submitting.
- **PoW requirement:** Scenario A still requires mining blocks, but at reduced difficulty — the attacker's advantage scales with the timestamp divergence they can engineer.

---

### Recommendation

Replace `get_header_by_height` in all three `get_next_work_required` implementations with an ancestor walk starting from `prev_block_header` using `get_prev_header`. This mirrors the Zcash implementation and correctly resolves the epoch-boundary block regardless of whether the block being validated is on the mainchain or a fork.

```rust
// Correct: walk back blocks_to_go_back steps from prev_block_header
let mut cursor = prev_block_header.clone();
for _ in 0..blocks_to_go_back {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
let first_block_time: i64 = cursor.block_header.time.into();
```

This is O(interval) in the worst case (2016 steps for Bitcoin/Litecoin), which is acceptable given that it only executes once per epoch boundary. For Dogecoin's per-block retarget the walk is always exactly one step.

### Proof of Concept

**Bitcoin build — epoch-boundary timestamp substitution:**

1. Contract is initialized with genesis at height 0 (a difficulty-adjustment boundary, as required). [8](#0-7) 
2. Relayer submits mainchain blocks 0–4031 (two full 2016-block epochs). Mainchain block at height 2016 has timestamp `T_main`.
3. Attacker (trusted relayer) submits a fork starting at height 2000. Fork blocks 2001–2016 are crafted with the fork's block at height 2016 having timestamp `T_fork < T_main` (a faster epoch on the fork).
4. Attacker submits fork block at height 4032 (first block of epoch 3 on the fork). `check_pow` calls `get_next_work_required` with `prev_block_header` = fork block at 4031.
5. `first_block_height = 4031 - 2015 = 2016`. `get_header_by_height(2016)` returns the **mainchain's** block at 2016 (timestamp `T_main`), not the fork's block (timestamp `T_fork`).
6. `calculate_next_work_required` computes `actual_time_taken` using `T_main` (larger), producing a higher target (lower difficulty) than the fork's actual consensus requires.
7. Attacker's fork block at 4032 encodes this inflated target in `bits`; the contract accepts it (`expected_bits == block_header.bits` passes).
8. Attacker mines subsequent fork blocks at the reduced difficulty. Once `fork.chain_work > mainchain.chain_work`, `reorg_chain` is triggered, replacing the canonical chain.
