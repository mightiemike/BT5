### Title
Protobuf `ValidResourceBounds` Deserialization Uses Value Heuristic Instead of Explicit Type Tag, Causing Transaction Hash Mismatch - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf converter for `ValidResourceBounds` determines whether a transaction uses the pre-0.13.3 `L1Gas` variant or the post-0.13.3 `AllResources` variant by inspecting whether the deserialized `l2_gas` and `l1_data_gas` values are zero — a value-based heuristic — rather than an explicit type/version tag. A transaction legitimately submitted as `AllResources` with zero `l2_gas` and zero `l1_data_gas` (but non-zero `l1_gas`) will be silently reclassified as `L1Gas` upon P2P deserialization. Because `get_tip_resource_bounds_hash` produces a different Poseidon preimage for `L1Gas` (2 resource elements) versus `AllResources` (3 resource elements), the recomputed transaction hash diverges from the hash committed to the block, causing `validate_transaction_hash` to reject the transaction on any syncing node.

### Finding Description

`ValidResourceBounds` is a Rust enum with two variants:

```
L1Gas(ResourceBounds)          // Pre-0.13.3: only L1 gas; L2 bounds signed but unused
AllResources(AllResourceBounds) // 0.13.3+: L1, L2, and L1_data gas
```

The protobuf deserializer reconstructs this enum by inspecting the decoded field values:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The same heuristic is present in the RPC layer:

```rust
// crates/apollo_rpc/src/v0_8/transaction.rs
if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
    Self::L1Gas(value.l1_gas)
} else { ... }
``` [2](#0-1) 

The transaction hash function `get_tip_resource_bounds_hash` produces structurally different Poseidon preimages for the two variants:

- `L1Gas` → hashes `[tip, L1_GAS_packed, L2_GAS_packed]` (3 elements)
- `AllResources` → hashes `[tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed]` (4 elements) [3](#0-2) 

A transaction originally committed to the chain as `AllResources` (with `l2_gas = 0`, `l1_data_gas = 0`, `l1_gas ≠ 0`) carries a hash computed over 4 elements. When a syncing node deserializes it from protobuf, the heuristic reclassifies it as `L1Gas`, and `validate_transaction_hash` recomputes a 3-element hash — a different value — causing the validation to fail.

The gateway's stateless validator does not prevent this trigger: it only requires `max_possible_fee(Tip::ZERO) != 0`, which is satisfied by non-zero `l1_gas` alone, and the `min_gas_price` check applies to `l2_gas.max_price_per_unit` but not `l2_gas.max_amount`. A transaction with `l2_gas = {max_amount: 0, max_price_per_unit: 0}` and `l1_data_gas = {max_amount: 0, max_price_per_unit: 0}` passes all gateway checks. [4](#0-3) 

The TODO comment in the converter acknowledges the fragility of this approach:

```
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
``` [5](#0-4) 

### Impact Explanation

Any syncing node that receives a block containing an `AllResources` transaction with zero `l2_gas` and zero `l1_data_gas` will fail to validate the transaction hash. This causes the syncing node to reject a valid committed block, breaking state sync. The RPC layer exhibits the same misclassification, so `starknet_getTransactionByHash` and fee estimation would also recompute the wrong hash for such transactions, returning an authoritative-looking wrong value.

Impact: **High** — Transaction conversion/hash logic binds the wrong hash type to the executable payload; syncing nodes reject valid transactions/blocks.

### Likelihood Explanation

The trigger requires submitting a V3 transaction with `AllResourceBounds` where both `l2_gas` and `l1_data_gas` are fully zero. The gateway's `validate_resource_bounds` permits this as long as `l1_gas` is non-zero. Any user (no privilege required) can craft such a transaction via the public RPC `add_invoke_transaction` endpoint. Once included in a block, every syncing node is affected.

### Recommendation

Replace the value-based heuristic with an explicit version/type tag in the protobuf schema, or add a dedicated boolean/enum field to `ResourceBounds` that encodes whether the transaction is pre-0.13.3 (`L1Gas`) or post-0.13.3 (`AllResources`). Until the schema is updated, the converter should default to `AllResources` when `l1_data_gas` is absent (for 0.13.2 wire compatibility) rather than inferring the variant from zero-value fields:

```rust
// Proposed fix: absence of l1_data_gas field → L1Gas (0.13.2 wire); presence → AllResources
let variant = match value.l1_data_gas {
    None => ValidResourceBounds::L1Gas(l1_gas),
    Some(raw) => {
        let l1_data_gas: ResourceBounds = raw.try_into()?;
        ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
    }
};
```

Additionally, the gateway should explicitly reject `AllResources` transactions where both `l2_gas` and `l1_data_gas` are zero, since such transactions are indistinguishable from `L1Gas` transactions after serialization.

### Proof of Concept

1. Craft an `add_invoke_transaction` RPC call with:
   ```json
   "resource_bounds": {
     "l1_gas":      { "max_amount": "0x1000", "max_price_per_unit": "0x1" },
     "l2_gas":      { "max_amount": "0x0",    "max_price_per_unit": "0x0" },
     "l1_data_gas": { "max_amount": "0x0",    "max_price_per_unit": "0x0" }
   }
   ```
2. The gateway accepts the transaction (non-zero `l1_gas` satisfies `max_possible_fee > 0`).
3. The sequencer computes `tx_hash_A` using `AllResources` (4-element Poseidon: tip + L1_GAS + L2_GAS_zero + L1_DATA_GAS_zero).
4. The transaction is included in block B with `tx_hash_A`.
5. A syncing node requests block B via P2P. The protobuf `ResourceBounds` message has `l2_gas = {0,0}` and `l1_data_gas = {0,0}`.
6. `TryFrom<protobuf::ResourceBounds>` evaluates `l1_data_gas.is_zero() && l2_gas.is_zero() == true` → classifies as `L1Gas`.
7. `validate_transaction_hash` recomputes `tx_hash_B` using `L1Gas` (3-element Poseidon: tip + L1_GAS + L2_GAS_zero).
8. `tx_hash_A ≠ tx_hash_B` → validation fails → syncing node cannot accept block B. [6](#0-5) [7](#0-6)
