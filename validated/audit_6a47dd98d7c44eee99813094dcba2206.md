### Title
Pausing the contract blocks `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` with no exception mechanism, freezing all downstream consumer operations — (`File: contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are decorated with a bare `#[pause]` macro with no bypass role. When the contract is paused, every NEAR caller — including third-party contracts that use the light client as a trust anchor for releasing bridged assets, atomic swaps, or HTLCs — receives a hard revert. Unlike `run_mainchain_gc` (which defines `UnrestrictedRunGC`) and `submit_blocks` (which defines `UnrestrictedSubmitBlocks`), no `UnrestrictedVerify` role exists, so there is no privileged path to keep verification alive during a pause.

---

### Finding Description

The `Role` enum defines bypass roles for two of the three pausable public methods:

- `submit_blocks` → `UnrestrictedSubmitBlocks`
- `run_mainchain_gc` → `UnrestrictedRunGC`
- `verify_transaction_inclusion` → **no bypass role**
- `verify_transaction_inclusion_v2` → **no bypass role** [1](#0-0) 

The two verification entry points carry a plain `#[pause]` with no `except(roles(...))` clause: [2](#0-1) [3](#0-2) 

Compare with `run_mainchain_gc`, which preserves an escape hatch: [4](#0-3) 

There is a compounding effect: `verify_transaction_inclusion_v2` internally delegates to `verify_transaction_inclusion`: [5](#0-4) 

Because `near_plugins` injects the pause guard into the generated function body, the inner call re-checks the pause flag. Even if a future patch added an exception to `verify_transaction_inclusion_v2` alone, the inner call to `verify_transaction_inclusion` would still revert, making the fix non-trivial and easy to get wrong.

---

### Impact Explanation

The light client is explicitly designed as a trust anchor for third-party NEAR contracts performing cross-chain verification (atomic swaps, bridged-asset releases, HTLC settlement). Any such consumer contract that calls `verify_transaction_inclusion_v2` to confirm a Bitcoin payment and release funds will receive a hard revert for the entire duration of the pause. If the consumer contract enforces a time-lock (e.g., an HTLC with a NEAR block-height or timestamp deadline), the window to claim funds may expire while the light client is paused, permanently locking or redirecting those funds. The light client's own state is not corrupted, but every downstream protocol that depends on it for finality is effectively frozen.

---

### Likelihood Explanation

A pause is a routine operational action — it is the expected response to a discovered bug in `submit_blocks` or a consensus anomaly. The `PauseManager` role is designed to be used. Because the pause is contract-wide and the verification functions carry no exception, any legitimate pause event simultaneously disables all consumer-facing verification. The longer the pause lasts (e.g., while a fix is staged and deployed through the `Upgradable` flow, which itself has a staging duration), the higher the probability that a time-sensitive consumer operation expires.

---

### Recommendation

Add an `UnrestrictedVerify` role and apply it as an exception on both verification methods, mirroring the pattern already used for `run_mainchain_gc`:

```rust
// In Role enum
/// Allows verify_transaction_inclusion* even when contract is paused
UnrestrictedVerify,

// On both verification methods
#[pause(except(roles(Role::UnrestrictedVerify)))]
pub fn verify_transaction_inclusion(&self, ...) -> bool { ... }

#[pause(except(roles(Role::UnrestrictedVerify)))]
pub fn verify_transaction_inclusion_v2(&self, ...) -> bool { ... }
```

Alternatively, since both functions are pure read operations that do not modify contract state, consider removing `#[pause]` from them entirely. The pause mechanism is intended to halt state-mutating operations during emergencies; blocking read-only verification provides no additional safety and actively harms consumers.

---

### Proof of Concept

1. A NEAR HTLC contract is deployed. It calls `verify_transaction_inclusion_v2` on the BTC light client to confirm a Bitcoin payment and release locked NEAR tokens. The HTLC has a 24-hour expiry.
2. A bug is discovered in `submit_blocks`. The `PauseManager` legitimately pauses the contract.
3. The Bitcoin payment is confirmed on-chain. The HTLC beneficiary calls the HTLC contract to claim their NEAR tokens.
4. The HTLC contract calls `verify_transaction_inclusion_v2` → the call reverts with the `near_plugins` pause error.
5. The HTLC contract cannot release funds. The 24-hour window expires.
6. The HTLC refund path activates; the original depositor reclaims the NEAR tokens.
7. The beneficiary loses their expected payment despite the Bitcoin transaction being fully confirmed.

The root cause — `#[pause]` on `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` with no exception role — is the necessary and sufficient condition for this outcome. [6](#0-5) [7](#0-6)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
