Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Allowing Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap()` sets to its own `msg.sender`. When `MetricOmmSimpleRouter` is the caller, `sender` is the router address, not the originating EOA. If the pool admin allowlists the router so that approved users can trade through it, every non-allowlisted address can bypass the curated-pool restriction by calling the router directly.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that value against the per-pool allowlist: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` the pool sees. The originating user (`msg.sender` of the router call) is stored only in transient storage for payment settlement and is never forwarded to the pool or extensions: [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. The extension therefore sees `sender = address(router)`, not the EOA. If the admin allowlists the router (required for any legitimate user to trade through it), the check `allowedSwapper[pool][router] == true` passes for every caller regardless of their individual allowlist status.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` is a curated pool where only approved addresses may trade. Once the router is allowlisted, the allowlist is effectively nullified: any address can call `router.exactInputSingle` and trade freely. All swap-side curation guarantees — KYC-gated pools, private market-making pools, sanctioned-party exclusions — are broken. This constitutes broken core pool functionality and loss of the admin-enforced access control, matching the "Admin-boundary break" and "Broken core pool functionality" allowed impacts.

## Likelihood Explanation
The router is the standard, documented swap entrypoint. A pool admin deploying a curated pool will naturally allowlist the router so approved users can trade normally. The bypass requires only a standard router call with no special setup, making it trivially reachable by any unprivileged address once the router is allowlisted.

## Recommendation
The extension must gate the originating user, not the intermediary. The cleanest fix is for the router to encode `msg.sender` into `extensionData`, and for the extension to decode and verify it — but only when the call originates from a factory-approved router (to prevent spoofing). Alternatively, gate on `recipient` as a proxy for the economic actor, which closes the router bypass though it is imperfect since `recipient` can be set to any address.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the only approved trader
  allowedSwapper[pool][router] = true  // admin allowlists router so alice can use it

Attack (bob, not allowlisted):
  bob calls router.exactInputSingle({
    pool:      pool,
    recipient: bob,
    zeroForOne: true,
    amountIn:  X,
    ...
  })

  router calls pool.swap(bob, true, X, ...)
    pool: msg.sender = router
    _beforeSwap(sender=router, ...)
      SwapAllowlistExtension.beforeSwap(sender=router, ...)
        allowedSwapper[pool][router] == true  → passes ✓

  bob receives output tokens despite never being allowlisted.
```
