### Title
Silent Failure of Slow-Mode `DepositCollateral` Execution Permanently Locks User Funds — (File: `core/contracts/Endpoint.sol`)

---

### Summary

In `Endpoint.sol`, the `depositCollateralWithReferral` function transfers user tokens to the Clearinghouse **before** queuing a slow-mode transaction. When that queued transaction is later executed, any revert inside `clearinghouse.depositCollateral` is silently swallowed by a `try/catch` block that contains no fund-return logic. The slow-mode entry is deleted before execution. The result is permanent, irrecoverable locking of the deposited tokens inside the Clearinghouse contract.

---

### Finding Description

**Step 1 — Token transfer precedes queue entry.**

`depositCollateralWithReferral` calls `handleDepositTransfer` (transferring ERC-20 tokens from `msg.sender` to the Clearinghouse) and then pushes a `SlowModeTx` entry into `slowModeTxs`. [1](#0-0) 

**Step 2 — Slow-mode entry is deleted before execution.**

`_executeSlowModeTransaction` deletes the queued entry with `delete slowModeTxs[_slowModeConfig.txUpTo++]` **before** calling `processSlowModeTransaction`. If execution subsequently reverts, the entry is already gone and cannot be retried. [2](#0-1) 

**Step 3 — Reverts are silently swallowed; fund-return mechanism was explicitly removed.**

The execution is wrapped in a `try/catch`. The catch block contains only an out-of-gas guard and the comment `// try return funds now removed`, confirming that a prior fund-return path was deliberately deleted with no replacement. [3](#0-2) 

**Step 4 — Concrete revert triggers inside `clearinghouse.depositCollateral`.**

`Clearinghouse.depositCollateral` calls `_decimals(txn.productId)`, which calls `_tokenAddress` → `spotEngine.getConfig(productId).token` and then `require(address(token) != address(0), ERR_INVALID_PRODUCT)`. If the product's token address is zero (e.g., after a spot-engine upgrade or product reconfiguration between deposit and execution), this reverts. The 3-day `SLOW_MODE_TX_DELAY` window makes this a realistic race condition. [4](#0-3) 

Additionally, `spotEngine.updateBalance` is called unconditionally; any revert inside it (arithmetic overflow, product-state inconsistency) propagates up and is caught silently. [5](#0-4) 

---

### Impact Explanation

Deposited ERC-20 tokens are held by the Clearinghouse contract. If the corresponding slow-mode `DepositCollateral` transaction fails silently, those tokens remain in the Clearinghouse with no accounting entry crediting the user's subaccount and no admin-accessible recovery path (unlike the external report's scenario where an admin can raise the cap). The user suffers a total, permanent loss of the deposited amount. This is a direct asset-loss impact, not merely a denial-of-service.

---

### Likelihood Explanation

The 3-day `SLOW_MODE_TX_DELAY` creates a meaningful window during which protocol state can diverge from the state at deposit time. Realistic triggers include:

- A spot-engine upgrade that changes product configuration (token address becomes zero or product is removed).
- Any arithmetic revert inside `spotEngine.updateBalance` caused by concurrent state changes (e.g., interest-index updates that push a balance calculation past `int128` bounds).
- A token contract upgrade that changes `decimals()`, causing `require(decimals <= MAX_DECIMALS)` to fail.

The comment `// try return funds now removed` is direct evidence that the developers were aware of this failure mode and chose to remove the mitigation without providing an alternative. [6](#0-5) 

---

### Recommendation

1. **Restore a fund-return path**: In the `catch` block of `_executeSlowModeTransaction`, detect `DepositCollateral` transaction type and refund the deposited tokens to the original sender before discarding the entry.
2. **Separate custody from queuing**: Do not transfer tokens until the slow-mode transaction is actually executed (i.e., move `handleDepositTransfer` into `processSlowModeTransactionImpl` for `DepositCollateral`).
3. **Add a recovery mapping**: Record `(txUpTo → (token, sender, amount))` at queue time so that a failed execution can be claimed back by the depositor.

---

### Proof of Concept

1. Alice calls `depositCollateralWithReferral(subaccount, productId, amount, referral)`.
2. `handleDepositTransfer` moves `amount` tokens from Alice to the Clearinghouse. [7](#0-6) 
3. A `SlowModeTx` is pushed to `slowModeTxs[txCount]` with `executableAt = block.timestamp + 3 days`. [8](#0-7) 
4. During the 3-day window, the spot engine is upgraded and `productId`'s token address is set to `address(0)`.
5. After the delay, `executeSlowModeTransaction()` is called (by anyone).
6. `delete slowModeTxs[txUpTo++]` removes the entry. [9](#0-8) 
7. `this.processSlowModeTransaction(...)` calls `clearinghouse.depositCollateral`, which calls `_decimals(productId)`, which calls `require(address(token) != address(0))` → **reverts**. [10](#0-9) 
8. The `catch` block fires; `gasleft()` is well above the threshold; execution continues silently. [11](#0-10) 
9. Alice's tokens remain in the Clearinghouse. Her subaccount balance was never updated. No recovery mechanism exists.

    function _tokenAddress(uint32 productId) internal view returns (address) {
        return _spotEngine().getConfig(productId).token;
    }

    function _decimals(uint32 productId) internal virtual returns (uint8) {
        IERC20Base token = IERC20Base(_tokenAddress(productId));
        require(address(token) != address(0), ERR_INVALID_PRODUCT);
        return token.decimals();
    }

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
