### Title
Whale or Early Depositor Can Push Spot Utilization to 100%, Permanently Blocking All Withdrawals - (`core/contracts/SpotEngine.sol`)

---

### Summary

A user with sufficient collateral can borrow all available deposits of a spot product, pushing utilization to 100%. Once `totalBorrows >= totalDeposits`, `assertUtilization` reverts on every subsequent withdrawal call, locking all other users' deposited collateral in the protocol indefinitely.

---

### Finding Description

`SpotEngine.assertUtilization` enforces a hard invariant that `totalDeposits >= totalBorrows` on every withdrawal: [1](#0-0) 

This function is called unconditionally inside `Clearinghouse.withdrawCollateral` after the balance update but before the transaction completes: [2](#0-1) 

The `totalDepositsNormalized` and `totalBorrowsNormalized` tracked in the `State` struct are global across all users of a product: [3](#0-2) 

An attacker who deposits sufficient collateral can borrow enough of a spot asset (creating a large negative balance) to drive `totalBorrows` to equal `totalDeposits`. Once at 100% utilization, any withdrawal attempt reduces `totalDeposits` further, making `totalDeposits < totalBorrows`, causing `assertUtilization` to revert.

Additionally, the interest accrual mechanism in `_updateState` deducts a protocol fee fraction from depositor returns while borrowers pay the full rate: [4](#0-3) 

At 100% utilization, this fee wedge causes `totalBorrows` to grow faster than `totalDeposits` over time, meaning even if utilization starts just below 100%, sustained high utilization will eventually push `totalBorrows > totalDeposits` passively — mirroring exactly how interest accrual in Liquity V2 passively triggers Recovery Mode.

---

### Impact Explanation

All users of the affected spot product are permanently blocked from withdrawing their deposited collateral. Unlike the Liquity analog (which only blocks new borrowing), this vulnerability locks existing deposited funds. The `withdrawCollateral` path is the only on-chain exit for collateral; if `assertUtilization` reverts, there is no alternative withdrawal route. The attacker retains the borrowed assets and can deploy them elsewhere to earn yield, offsetting their interest cost.

---

### Likelihood Explanation

At protocol launch, total deposits per product are small. An attacker needs only enough collateral to borrow the entire deposited supply of a single product. The health check permits borrowing up to the collateral's weighted value, so a well-capitalized attacker can execute this with a single transaction. The attack is especially cheap in the early bootstrapping phase, directly mirroring the "Initial Depositor disables the branch" scenario from the Liquity report. A competing liquidity provider or whale has clear economic incentive: lock a competing product's withdrawals while earning yield on the borrowed assets elsewhere.

---

### Recommendation

1. **Cap maximum utilization on borrows**: Prevent any single borrow from pushing utilization above a configurable ceiling (e.g., 95%), analogous to how Liquity's CCR prevents borrowing that would push TCR below the critical threshold.
2. **Allow withdrawals that do not worsen utilization**: If a user is withdrawing collateral (not the borrowed asset), their withdrawal does not change `totalBorrows` and should not be blocked by `assertUtilization`.
3. **Seed liquidity at launch**: Ensure protocol-owned liquidity is deposited at launch to make the cost of pushing utilization to 100% prohibitively expensive.

---

### Proof of Concept

1. At protocol launch, 1,000 USDC is deposited across all users into the USDC spot product (`totalDeposits = 1000`).
2. Attacker deposits 1,200 USDC-equivalent of ETH collateral into their subaccount.
3. Attacker submits a borrow/sell transaction via the sequencer that creates a negative USDC balance of −1,000 for their subaccount. Health check passes (collateral covers the borrow). `totalBorrows = 1000`, `totalDeposits = 1000`.
4. Any user now calls `withdrawCollateral` for USDC. Inside `Clearinghouse.withdrawCollateral` (line 412), `spotEngine.updateBalance` reduces `totalDeposits` to 999. Line 413 calls `spotEngine.assertUtilization(productId)`, which evaluates `999 >= 1000` → **false** → reverts with `ERR_MAX_UTILIZATION`.
5. All USDC withdrawals are permanently blocked. The attacker holds 1,000 USDC and can deploy it in another protocol, earning yield that offsets their borrow interest cost. [5](#0-4) [1](#0-0)

    function assertUtilization(uint32 productId) external view {
        (State memory _state, ) = getStateAndBalance(productId, X_ACCOUNT);
        int128 totalDeposits = _state.totalDepositsNormalized.mul(
            _state.cumulativeDepositsMultiplierX18
        );
        int128 totalBorrows = _state.totalBorrowsNormalized.mul(
            _state.cumulativeBorrowsMultiplierX18
        );
        require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
    }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
