### Title
Fast Withdrawal Fees Permanently Locked in `BaseWithdrawPool` with No Recovery Mechanism — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal` collects fees into `fees[productId]` and holds the corresponding ERC-20 tokens in the contract, but no function exists to claim or withdraw these accumulated fees. The `fees` mapping is only ever incremented, never decremented, and no `claimFees` or equivalent function is implemented anywhere in the contract or its concrete child `WithdrawPool`.

---

### Finding Description

In `submitFastWithdrawal`, two fee-collection paths exist:

**Path A — self-serve fast withdrawal (`sendTo == msg.sender`):**
The fee is deducted from the user's withdrawal amount (`transferAmount -= uint128(fee)`), so the fee tokens remain in the contract's balance.

**Path B — third-party fast withdrawal provider (`sendTo != msg.sender`):**
The fee is pulled directly from the caller via `safeTransferFrom(token, msg.sender, uint128(fee))`, depositing it into the contract.

In both paths, `fees[productId] += fee` records the accumulation. However, searching the entire contract hierarchy (`BaseWithdrawPool`, `WithdrawPool`) reveals:

- `fees[productId]` is **never decremented** anywhere.
- No `claimFees`, `withdrawFees`, or equivalent function exists.
- The only token-exit path is `removeLiquidity`, which is `onlyOwner`, takes an arbitrary `amount` parameter, and **does not touch `fees[productId]`** — leaving the accounting permanently desynchronized after any call. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Fast withdrawal fees paid by third-party providers (Path B) are permanently locked in `WithdrawPool`. The provider transfers real ERC-20 tokens into the contract and has no on-chain mechanism to recover them. The `fees` mapping grows monotonically, and the token balance representing those fees is irrecoverable by any non-owner party. Even the owner's `removeLiquidity` escape hatch does not decrement `fees`, so the accounting state is permanently corrupted after any such call — the contract will always report fees as outstanding even after tokens have been removed. [4](#0-3) 

---

### Likelihood Explanation

Medium-High. `submitFastWithdrawal` is a publicly callable, permissionless function — any address can act as a fast withdrawal provider. The `sendTo != msg.sender` path is the standard use case for a third-party liquidity provider servicing a user's withdrawal. Every such call permanently locks the fee amount. As fast withdrawal volume grows, the locked token balance grows proportionally with no recovery path. [5](#0-4) 

---

### Recommendation

1. Implement a `claimFees(uint32 productId, address recipient)` function (restricted to owner or a designated fee recipient) that transfers `fees[productId]` tokens to `recipient` and resets `fees[productId] = 0`.
2. Alternatively, if fees are intended to be protocol revenue, route them directly to a treasury address at collection time rather than accumulating them in the contract.
3. Ensure `removeLiquidity` either decrements `fees[productId]` or is replaced by the dedicated fee-claim function to prevent accounting desynchronization.

---

### Proof of Concept

1. Alice (fast withdrawal provider) calls `submitFastWithdrawal(idx, transaction, signatures)` where the signed transaction specifies `sendTo = Bob` (not Alice).
2. `resolveFastWithdrawal` returns `sendTo = Bob`, so `sendTo != msg.sender` (Alice).
3. `safeTransferFrom(token, Alice, uint128(fee))` pulls the fee from Alice into the contract.
4. `fees[productId] += fee` records the fee — this is the only state change for the fee.
5. Alice searches for any function to reclaim her fee: `claimFees` — does not exist; `withdrawFees` — does not exist; `removeLiquidity` — `onlyOwner`, not callable by Alice.
6. Alice's fee tokens are permanently locked in `WithdrawPool` with no on-chain recovery path. [2](#0-1) [6](#0-5)

```text
    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
```
