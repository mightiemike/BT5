### Title
Unrestricted `run_mainchain_gc` Callable by Any Unprivileged NEAR Account, Enabling Aggressive Header Pruning That Invalidates SPV Proofs and Breaks Reorg Handling — (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public, state-mutating function protected only by a `#[pause]` decorator. When the contract is not paused, any unprivileged NEAR account can call it with an arbitrarily large `batch_size`. This allows an attacker to immediately prune all mainchain headers eligible for removal in a single call — far more aggressively than the controlled, relayer-triggered GC that runs inside `submit_blocks`. The result is that SPV proof verification panics for pruned blocks and chain reorganization breaks when the common ancestor has been removed.

---

### Finding Description

`submit_blocks` is correctly restricted to registered relayers via `#[trusted_relayer]` and `#[pause]`. Inside it, GC is triggered conservatively:

```rust
self.run_mainchain_gc(num_of_headers);  // batch_size == number of submitted headers (small)
``` [1](#0-0) 

However, `run_mainchain_gc` itself is separately exposed as a public method with only a pause guard:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [2](#0-1) 

The `#[pause(except(...))]` decorator only restricts callers when the contract is paused. When the contract is running normally, there is no caller check — no `#[trusted_relayer]`, no `#[private]`, no role assertion. Any NEAR account can call this function directly.

Inside the function, the removal is bounded by `gc_threshold` but not by caller identity:

```rust
let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
``` [3](#0-2) 

An attacker passing `batch_size = u64::MAX` removes every header eligible for pruning in a single transaction — the maximum possible GC in one call. The normal relayer path removes only `num_of_headers` (the count of blocks just submitted, typically 1–100) per `submit_blocks` call, leaving older-but-still-eligible blocks available for SPV proofs and reorg walks. The attacker collapses this window to exactly `gc_threshold` blocks instantly.

---

### Impact Explanation

**1. SPV proof verification is broken for pruned blocks.**

`verify_transaction_inclusion` looks up the target block in `mainchain_header_to_height`. `remove_block_header` deletes entries from both `mainchain_header_to_height` and `headers_pool`:

```rust
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [4](#0-3) 

After aggressive GC, any call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a pruned block panics:

```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
``` [5](#0-4) 

Downstream NEAR contracts that call `verify_transaction_inclusion_v2` to gate cross-chain actions receive a panic (transaction failure) instead of a boolean result, permanently blocking any action that depends on proving inclusion of a transaction in a pruned block.

**2. Chain reorganization is broken when the common ancestor is pruned.**

`reorg_chain` walks the fork chain backward until it finds a block present in `mainchain_header_to_height`. If the common ancestor was pruned by aggressive GC, the walk reaches a block whose `prev_block_hash` is no longer in `headers_pool` and panics:

```rust
fork_header_cursor = self
    .headers_pool
    .get(&prev_block_hash)
    .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
``` [6](#0-5) 

This means a valid, higher-chainwork fork submitted by a legitimate relayer causes the contract to panic and the chain tip is never updated, permanently corrupting the canonical chain view.

---

### Likelihood Explanation

The entry path requires no privileges, no deposit, and no special knowledge beyond the contract's public ABI. Any NEAR account can call `run_mainchain_gc(batch_size: u64::MAX)` at any time the contract is not paused. The attack is a single transaction. The only precondition is that the mainchain must have grown beyond `gc_threshold`, which is the normal operating state of a live deployment (the recommended `gc_threshold` is 52,704 blocks — approximately one year of Bitcoin blocks).

---

### Recommendation

Add a caller restriction to `run_mainchain_gc` equivalent to the one on `submit_blocks`. The simplest fix is to require the caller to hold a designated role (e.g., `UnrestrictedRunGC` already exists for the paused-contract bypass, or a new `GCManager` role) or to restrict it to the contract itself (i.e., `#[private]`), since the only legitimate external trigger is already the internal call from `submit_blocks`.

```rust
// Option A: restrict to contract-internal calls only
#[private]
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }

// Option B: require a role for external callers
// Add an acl_require check at the top of the function body
```

### Proof of Concept

1. Deploy the contract with `gc_threshold = 100` and allow the relayer to submit 200 blocks. The mainchain now holds 200 headers; 100 are eligible for GC.
2. From any unprivileged NEAR account (no role, no deposit), call:
   ```
   run_mainchain_gc(batch_size: 18446744073709551615)
   ```
3. The contract immediately removes all 100 eligible headers in one call, advancing `mainchain_initial_blockhash` to block 101.
4. Now call `verify_transaction_inclusion_v2` for any block at height ≤ 100. The call panics with `"block does not belong to the current main chain"` because `mainchain_header_to_height` no longer contains that block hash.
5. Alternatively, have a relayer submit a valid fork whose common ancestor with the main chain is at height ≤ 100. The `reorg_chain` walk panics with `"previous fork block should be there"`, permanently preventing the reorg from completing. [7](#0-6)

    #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
