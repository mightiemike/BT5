### Title
Deposited Tokens Permanently Locked When `DepositCollateral` Slow Mode Transaction Reverts Silently — (`File: core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` transfers ERC20 tokens from the caller to the contract before queuing a slow mode transaction. `_executeSlowModeTransaction` deletes the queued entry and then wraps `processSlowModeTransaction` in a `try/catch` that silently discards any revert. If `clearinghouse.depositCollateral` reverts during slow mode processing, the ERC20 transfer is already committed but the balance credit never applies, permanently locking the deposited tokens with no recovery path. The codebase itself acknowledges this with the comment `// try return funds now removed`.

---

### Finding Description

**Step 1 — Token transfer committed before slow mode tx is queued**

In `Endpoint.depositCollateralWithReferral`, `handleDepositTransfer` is called at line 144, pulling tokens from `msg.sender` into the contract. Only after that does the function enqueue a `DepositCollateral` slow mode transaction at lines 152–166. [1](#0-0) 

**Step 2 — Slow mode tx deleted before execution, revert silently swallowed**

In `_executeSlowModeTransaction`, the queued entry is deleted at line 194 (`delete slowModeTxs[_slowModeConfig.txUpTo++]`) before the call is attempted. The actual execution at line 207 is wrapped in `try this.processSlowModeTransaction(...) {} catch {}`. If the inner call reverts, the catch block does nothing. The comment at line 226 — `// try return funds now removed` — explicitly confirms that a prior fund-return mechanism was removed, leaving no recovery path. [2](#0-1) 

**Step 3 — `clearinghouse.depositCollateral` is the revert point**

`processSlowModeTransactionImpl` handles `DepositCollateral` by calling `clearinghouse.depositCollateral(txn)` at line 216. This call can revert if the product has been delisted in the 3-day window between deposit and slow mode processing, or if any other clearinghouse-side validation fails. [3](#0-2) 

**Broken invariant**: every `handleDepositTransfer` in `depositCollateralWithReferral` must be matched by a successful `clearinghouse.depositCollateral`. The try/catch breaks this invariant without any compensating refund.

---

### Impact Explanation

When the slow mode tx reverts and the catch block fires:

- The ERC20 tokens are already held by the `Endpoint` contract (irreversible).
- The slow mode queue entry has been deleted (line 194) and cannot be retried.
- `clearinghouse.depositCollateral` was never called, so no balance is credited in `SpotEngine`.
- The depositor's funds are permanently locked in the contract with no on-chain recovery mechanism.

This is a direct, concrete asset loss for any depositor whose `DepositCollateral` slow mode tx fails during the 3-day processing window. [4](#0-3) 

---

### Likelihood Explanation

The 3-day `SLOW_MODE_TX_DELAY` creates a meaningful window during which protocol state can change. Concrete revert triggers include:

- A product being delisted via a `DelistProduct` slow mode tx between deposit and processing.
- Any clearinghouse-side guard (e.g., product validity, sanction checks on the subaccount) that passes at deposit time but fails at processing time.

`depositCollateralWithReferral` is `public` with no `msg.sender == owner(subaccount)` check, so any caller depositing on behalf of a subaccount is exposed. The explicit `// try return funds now removed` comment confirms the developers were aware of this gap and chose not to address it. [5](#0-4) 

---

### Recommendation

Restore a fund-return path inside the catch block of `_executeSlowModeTransaction`. When a `DepositCollateral` slow mode tx fails, the contract should decode the original `DepositCollateral` struct from the deleted `SlowModeTx.tx` field and transfer the corresponding token amount back to `address(bytes20(txn.sender))`. Alternatively, do not delete the slow mode tx entry until after successful execution, so a failed tx can be retried or refunded. [6](#0-5) 

---

### Proof of Concept

1. User calls `Endpoint.depositCollateral(subaccountName, productId, amount)`.
2. `depositCollateralWithReferral` is entered; `handleDepositTransfer` moves `amount` tokens from the user to `Endpoint` (line 144). A `DepositCollateral` slow mode tx is queued with a 3-day delay (lines 152–166).
3. Before the 3-day window expires, the product is delisted via a `DelistProduct` slow mode tx processed by the sequencer.
4. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`. `_executeSlowModeTransaction` deletes the queued entry (line 194) and calls `try this.processSlowModeTransaction(...)`.
5. Inside `processSlowModeTransactionImpl`, `clearinghouse.depositCollateral(txn)` reverts because the product no longer exists.
6. The catch block fires; the comment `// try return funds now removed` confirms no refund is issued.
7. The user's tokens remain in `Endpoint` with no balance credit and no on-chain recovery path. [7](#0-6) [2](#0-1) [3](#0-2)

```text
    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
