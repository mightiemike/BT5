### Title
Bonus Rewards Permanently Lost When Bonus Pool Is Insufficient During Redelegation Slash - (`File: x/tieredrewards/keeper/slash.go`)

### Summary

In `slashRedelegationPosition`, when a redelegation slash fires and the bonus pool is insufficient, the function silently swallows `ErrInsufficientBonusPool` and then persists the position with already-advanced reward checkpoints. The accrued bonus for the slashed period is permanently unclaimable by the position owner.

### Finding Description

`processEventsAndClaimBonus` advances the position's `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded` checkpoints **before** it checks whether the bonus pool has sufficient funds: [1](#0-0) 

Specifically, `applyBonusAccrualCheckpoint` is called at line 215, which advances `pos.LastBonusAccrual` to `blockTime` (or `ExitUnlockAt`). Only after that does the function reach `sufficientBonusPoolBalance` at line 230. If the pool is insufficient, the function returns `ErrInsufficientBonusPool` — but the in-memory `pos` already has its checkpoints advanced past the accrual period. [2](#0-1) 

In `slashRedelegationPosition`, this error is explicitly swallowed to prevent a chain halt: [3](#0-2) 

Execution then continues and the position — now carrying the advanced checkpoints — is persisted via `setPositionWithState`: [4](#0-3) 

For a **partial slash**, the position remains delegated with `LastBonusAccrual` advanced past the period for which bonus was computed but never paid. No subsequent call to `ClaimTierRewards`, `TierUndelegate`, `TierRedelegate`, or `ClearPosition` can recover those rewards — the checkpoint has already moved forward, so `processEventsAndClaimBonus` will compute zero bonus for the skipped window on the next invocation. [5](#0-4) 

### Impact Explanation

The position owner permanently loses the bonus rewards accrued between `pos.LastBonusAccrual` and the slash block time. The corrupted value is the `LastBonusAccrual` timestamp stored in the `Position` collection: it is advanced to mark the period as "processed" even though no coins were transferred. The bonus pool retains the coins, but they are now unattributable to the affected position — a silent, irreversible loss of user funds.

### Likelihood Explanation

The bonus pool is funded by governance via `MsgFundRewardsPool` and depletes as rewards are paid. Any period where the pool balance drops below the accrued bonus for a position — a realistic operational condition — combined with a redelegation slash (a normal staking event) triggers the bug. No privileged access is required; the hook fires automatically via the staking module's `BeforeRedelegationSlashed` callback. [6](#0-5) 

### Recommendation

Do not advance the `LastBonusAccrual` checkpoint before confirming the pool can cover the bonus. Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to execute only after `sufficientBonusPoolBalance` succeeds and the `SendCoinsFromModuleToAccount` call completes. Alternatively, when `ErrInsufficientBonusPool` is caught in `slashRedelegationPosition`, explicitly revert the in-memory checkpoint changes on `pos` before calling `setPositionWithState`, so the period remains re-claimable once the pool is refunded.

### Proof of Concept

1. A delegator creates a tier position and redelegates to a new validator.
2. The bonus pool is depleted (e.g., all prior rewards have been claimed).
3. The source validator is slashed, triggering `BeforeRedelegationSlashed` → `slashRedelegationPosition`.
4. `processEventsAndClaimBonus` computes a non-zero bonus, advances `LastBonusAccrual` in `pos`, then returns `ErrInsufficientBonusPool`.
5. The error is swallowed; `setPositionWithState` persists `pos` with the advanced `LastBonusAccrual`.
6. The pool is later refunded by governance.
7. The delegator calls `MsgClaimTierRewards` — `processEventsAndClaimBonus` computes zero bonus because `segmentStart == LastBonusAccrual == blockTime` of the slash. The rewards for the slashed period are gone. [7](#0-6) [1](#0-0)
