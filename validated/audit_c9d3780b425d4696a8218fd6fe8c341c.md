### Title
NLP Health Calculation Uses Fee-Exclusive Oracle Price, Overstating Collateral Value — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`burnNlp()` deducts a burn fee from the quote amount a user receives when redeeming NLP tokens, but the NLP price stored in the risk engine (`risk.priceX18`) is set to the raw `oraclePriceX18` — with no fee deduction. Health calculations therefore overstate the redeemable value of every NLP position by the burn fee, allowing users to borrow slightly more than the protocol can actually recover from NLP redemption.

---

### Finding Description

In `Clearinghouse.burnNlp()`, the actual quote a user receives is:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [1](#0-0) 

The effective per-token redemption price is therefore `oraclePriceX18 - burnFee/nlpAmount`, not `oraclePriceX18`.

However, both `mintNlp()` and `burnNlp()` set the NLP price in the spot engine to the unadjusted oracle price:

```solidity
spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);
``` [2](#0-1) [3](#0-2) 

This `oraclePriceX18` is then stored as `risk.priceX18` and consumed directly by `_calculateProductHealth()`:

```solidity
health += amount.mul(weight).mul(risk.priceX18);
``` [4](#0-3) 

The health contribution for an NLP position is therefore `nlpAmount × oraclePriceX18 × weight`, while the actual redeemable value is `nlpAmount × oraclePriceX18 − burnFee`. The burn fee is never reflected in the price used for collateral valuation.

---

### Impact Explanation

Every NLP holder's health is overstated by `burnFee × weight`. Concretely:

- For positions ≥ 1000 USDC: `burnFee = quoteAmount / 1000` (0.1%), so health is overstated by `0.1% × weight` of the NLP position value.
- For positions < 1000 USDC: `burnFee = ONE` (minimum 1 USDC), so the overstatement is proportionally larger — up to 100% of the position value for dust positions.

A user can exploit this to borrow slightly more quote than the NLP position can actually cover on redemption. If the user's position is then liquidated, the liquidator receives NLP tokens whose actual burn proceeds are less than the health calculation assumed, leaving a residual shortfall that must be absorbed by the insurance fund. Over time, repeated exploitation of this gap accumulates protocol losses.

Additionally, the health check in `mintNlp()` uses the fee-exclusive price:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [5](#0-4) 

This means a user can mint NLP and immediately use the overstated health to borrow more than the NLP is actually worth on redemption.

---

### Likelihood Explanation

Any user who holds NLP tokens and also has open borrow positions is affected passively. An adversarial user can deliberately maximize NLP holdings while borrowing to the health limit, knowing the health calculation is inflated by the burn fee. The entry path is fully permissionless: `MintNlp` and `BurnNlp` are standard user-signed transactions processed through `Endpoint`. No privileged access is required.

---

### Recommendation

When computing health for NLP positions, use the fee-adjusted effective price rather than the raw oracle price. One approach is to store a separate `nlpEffectivePriceX18` that accounts for the burn fee:

```solidity
// effective price = oraclePriceX18 * (1 - 1/1000) = oraclePriceX18 * 999/1000
int128 effectivePriceX18 = oraclePriceX18.mul(999e15); // 0.999 in 1e18 fixed-point
spotEngine.updatePrice(NLP_PRODUCT_ID, effectivePriceX18);
```

Alternatively, apply the burn fee discount inside `_calculateProductHealth()` specifically for `NLP_PRODUCT_ID`, analogous to how Idle's `tokenPriceWithFee(msg.sender)` was recommended over `tokenPrice()`.

---

### Proof of Concept

1. User calls `MintNlp` with `quoteAmount = 1,000,000 USDC` (1e24 internal). They receive `nlpAmount = 1e24 / oraclePriceX18` NLP tokens. NLP price is set to `oraclePriceX18`.

2. Health calculation values the NLP position at `nlpAmount × oraclePriceX18 × weight = 1,000,000 × weight` USDC.

3. User borrows `1,000,000 × weight` USDC against the NLP (health = 0, at the initial health limit).

4. User calls `BurnNlp` to redeem: receives `1,000,000 − 1,000` USDC (burn fee = 0.1% = 1,000 USDC).

5. The user has borrowed `1,000,000 × weight` USDC but can only recover `999,000` USDC from NLP. For `weight = 0.9`, the user borrowed 900,000 USDC and can repay it, but the health model assumed 1,000 USDC more collateral than actually existed. If the position is liquidated instead, the liquidator acquires NLP worth 999,000 USDC but the system credited 1,000,000 USDC of collateral — the 1,000 USDC gap is borne by the insurance fund.     function _calculateProductHealth(
        uint32 productId,
        bytes32 subaccount,
        IProductEngine.HealthType healthType
    ) internal returns (int128 health) {
        RiskHelper.Risk memory risk = _risk(productId);
        (int128 amount, int128 quoteAmount) = _getBalance(
            productId,
            subaccount
        );
        int128 weight = RiskHelper._getWeightX18(risk, amount, healthType);
        health += quoteAmount;

        if (amount != 0) {
            if (weight == 2 * ONE) {
                return -INF;
            }
            health += amount.mul(weight).mul(risk.priceX18);
            emit PriceQuery(productId);
        }
