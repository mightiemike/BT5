### Title
SHA256 Fast-Path Charges Hardcoded Byte Count Instead of Actual Argument Length — (File: `src/more_ops.rs`)

### Summary
The `op_sha256` fast path in `src/more_ops.rs` computes the per-byte cost using a hardcoded constant (`num_bytes = 2`) instead of the actual byte lengths of the two arguments. For the second argument `val ≥ 128`, the signed big-endian encoding requires 2 bytes (a leading zero is mandatory because the high bit is set), so the true byte count is 3, not 2. The fast path silently undercharges by `SHA256_COST_PER_BYTE = 2` cost units per invocation, directly mirroring the external report's pattern of using a fixed value instead of the caller-supplied quantity.

### Finding Description

`op_sha256` contains a compile-time fast path (disabled only when the `no-fastpath` feature is set) that fires when `sha256` receives exactly two arguments where the first is the small integer `1` and the second is a small integer `val` whose index falls within the `PRECOMPUTED_HASHES` table:

```rust
// src/more_ops.rs  lines 386-397
#[cfg(not(feature = "no-fastpath"))]
if let Some([v0, v1]) = match_args::<2>(a, input)
    && a.small_number(v0) == Some(1)
    && let Some(val) = a.small_number(v1)
    && (val as usize) < PRECOMPUTED_HASHES.len()
{
    let num_bytes: Cost = if val > 0 { 2 } else { 1 };   // ← hardcoded
    cost += num_bytes * SHA256_COST_PER_BYTE + 2 as Cost * SHA256_COST_PER_ARG;
    check_cost(cost, max_cost)?;
    return new_atom_and_cost(a, cost, &PRECOMPUTED_HASHES[val as usize]);
}
``` [1](#0-0) 

The slow path (the `while` loop below) charges `SHA256_COST_PER_BYTE` for every byte of every argument it hashes:

```rust
cost += blob.as_ref().len() as Cost * SHA256_COST_PER_BYTE;
``` [2](#0-1) 

The CLVM integer encoding rules (signed big-endian, minimal length) mean that:

| `val` range | `len_for_value(val)` | fast-path `num_bytes` | slow-path byte count |
|---|---|---|---|
| 0 | 0 | 1 | 1 (only `v0=1`) |
| 1 – 127 | 1 | 2 | 2 ✓ |
| 128 – 32 767 | 2 | 2 ✗ | 3 |
| 32 768 – 8 388 607 | 3 | 2 ✗ | 4 |

`len_for_value` is used throughout the allocator and operator code to compute the canonical byte length of a SmallAtom value: [3](#0-2) 

For `val ≥ 128` the fast path charges `num_bytes = 2` but the slow path would charge `1 + len_for_value(val) ≥ 3`. The discrepancy is `(len_for_value(val) − 1) × SHA256_COST_PER_BYTE` cost units — at minimum 2 units per call.

The root cause is identical to the external report: a hardcoded literal (`2`) is used where the actual computed quantity (`1 + len_for_value(val)`) should appear.

### Impact Explanation

Cost accounting in CLVM is consensus-critical: every full node must compute the same cost for the same program. The fast path is gated only on a compile-time feature flag (`no-fastpath`); production binaries ship with the fast path enabled. Any program that reaches the cost limit via the slow-path cost model will be accepted (not rejected) by the fast path, because the fast path charges fewer units. An attacker who knows the cost budget of a block can craft a `sha256` call with `(1 . val)` where `val ≥ 128` and `val` is within the precomputed table, accumulating undercharged cost across many calls. If the table extends beyond index 127 (a common choice is 256 entries), the undercharge is reachable with attacker-controlled CLVM bytes.

**Impact: Medium** — the per-call undercharge is small (2 cost units), but it is systematic and attacker-controlled, and it corrupts the cost value returned in the `Reduction` tuple, which is the authoritative cost signal used by `run_program` and the Python wheel's `run_serialized_chia_program`.

### Likelihood Explanation

**Likelihood: Medium** — the trigger requires only that the `PRECOMPUTED_HASHES` table contain more than 128 entries and that the attacker pass `(sha256 1 <val>)` with `val ≥ 128`. Both conditions are easily satisfied by an attacker who can submit arbitrary CLVM programs (e.g., via the mempool). The fast path is active in all production builds.

### Recommendation

Replace the hardcoded `num_bytes` with the actual byte lengths of both arguments:

```rust
// correct: use the real encoded lengths
let num_bytes: Cost = 1 + len_for_value(val) as Cost;  // v0=1 is always 1 byte
cost += num_bytes * SHA256_COST_PER_BYTE + 2 as Cost * SHA256_COST_PER_ARG;
```

This matches the slow-path accounting exactly and eliminates the divergence.

### Proof of Concept

Construct a CLVM program that evaluates `(sha256 1 128)`. The integer `128` encodes as `[0x00, 0x80]` (2 bytes). The slow path charges:

```
SHA256_BASE_COST(87) + 2×SHA256_COST_PER_ARG(134) + 3×SHA256_COST_PER_BYTE(2) + 32×MALLOC_COST_PER_BYTE(10)
= 87 + 268 + 6 + 320 = 681
```

The fast path charges:

```
SHA256_BASE_COST(87) + 2×SHA256_COST_PER_ARG(134) + 2×SHA256_COST_PER_BYTE(2) + 32×MALLOC_COST_PER_BYTE(10)
= 87 + 268 + 4 + 320 = 679
```

The `Reduction` returned by the fast path carries cost `679` instead of the correct `681`. Repeating this call `N` times in a single program undercharges by `2N` cost units, allowing a program that should exceed the block cost limit to pass validation. [4](#0-3) [5](#0-4)
