### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist. However, `MetricOmmPool.swap` passes `msg.sender` (the immediate caller of the pool) as `sender`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` at the pool level. If the pool admin allowlists the router address to support router-mediated swaps, every user — including non-allowlisted ones — can bypass the allowlist by routing through the public router.

---

### Finding Description

**Call chain for a direct swap:**

```
user → pool.swap()
  msg.sender = user
  _beforeSwap(sender=user, ...)
  extension.beforeSwap(sender=user, ...)
  checks allowedSwapper[pool][user]  ← correct
```

**Call chain for a router-mediated swap:**

```
user → router.exactInputSingle()
  router → pool.swap()
    msg.sender = router
    _beforeSwap(sender=router, ...)
    extension.beforeSwap(sender=router, ...)
    checks allowedSwapper[pool][router]  ← wrong actor
```

In `MetricOmmPool.swap`, the pool unconditionally passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value verbatim to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` at the pool level: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**The dilemma for pool admins:**

- If the admin does **not** allowlist the router → allowlisted users cannot use the router at all (broken functionality).
- If the admin **does** allowlist the router → any user can bypass the allowlist by routing through the public router (security bypass).

There is no configuration that simultaneously supports router-mediated swaps for allowlisted users and blocks non-allowlisted users from using the router.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` is a curated pool intended to restrict swaps to specific addresses (e.g., institutional market makers, KYC'd users, or protocol-controlled addresses). Once the pool admin allowlists the router to support standard periphery usage, any unprivileged user can bypass the allowlist entirely by calling `router.exactInputSingle()` or any other router entry point. This allows unauthorized swaps on a pool whose LP funds were deposited under the assumption that only allowlisted counterparties would trade. Unauthorized swaps can extract value from LPs at oracle-derived prices without the curation guarantee the pool was designed to enforce.

---

### Likelihood Explanation

Medium-High. The `MetricOmmSimpleRouter` is the standard, publicly documented periphery entry point for swaps. Pool admins who want their allowlisted users to use the router (a natural expectation) must allowlist the router address, which immediately opens the bypass to all users. The attacker needs no special privileges, no flash loan, and no multi-step setup — a single `exactInputSingle` call through the router suffices.

---

### Recommendation

The `sender` identity passed to extension hooks must reflect the **original economic actor**, not the intermediate contract. Two approaches:

1. **Pass the original user through the router**: The router could pass the original `msg.sender` as a parameter to `pool.swap()` (e.g., via a dedicated `swapOnBehalf` entry point or an additional field in `extensionData`), and the pool could forward it as `sender` to extensions. This requires a protocol-level change.

2. **Check `recipient` instead of `sender` in the extension**: If the pool admin's intent is to gate who receives output tokens, checking `recipient` (which the user controls and the router forwards correctly) would be more robust. However, this changes the semantics of the allowlist.

3. **Allowlist the router and require extension-data proof**: The extension could require a signed proof in `extensionData` identifying the original user, verified inside `beforeSwap`. This is more complex but preserves the current interface.

---

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension, only `allowedUser` is allowlisted
swapExtension.setAllowedToSwap(address(pool), allowedUser, true);
// Pool admin also allowlists the router to support router-mediated swaps
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Attack: non-allowlisted attacker bypasses the allowlist via the router
vm.prank(attacker); // attacker is NOT in the allowlist
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: token0,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 10_000,
    amountOutMinimum: 0,
    recipient: attacker,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// Swap succeeds: extension saw sender=router (allowlisted), not attacker
// attacker receives token1 output despite not being on the allowlist
```

The extension receives `sender = address(router)`, which is allowlisted, so the check at `allowedSwapper[pool][router]` passes and the swap executes for the non-allowlisted attacker. [6](#0-5)
