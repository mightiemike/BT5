### Title
Missing Parent Subaccount Health Check After Margin Deduction in `createIsolatedSubaccount` — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`createIsolatedSubaccount()` in `OffchainExchange.sol` deducts a user-specified margin from the parent subaccount's quote balance and transfers it to the new isolated subaccount, but performs **no health check on the parent** after the deduction. A user with existing borrows or open perp positions can drain their parent subaccount's quote balance below the level required to cover those liabilities, leaving the parent undercollateralized with no on-chain enforcement to stop it.

---

### Finding Description

In `OffchainExchange.createIsolatedSubaccount()`, after the margin transfer:

```solidity
int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
}
``` [1](#0-0) 

There is no call to `_isAboveInitial()` or `getHealth()` on `txn.order.sender` after the balance update. This is in direct contrast to `Clearinghouse.transferQuote()`, which performs the analogous quote transfer and **does** enforce a health check:

```solidity
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -toTransfer);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.recipient, toTransfer);
require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH);
``` [2](#0-1) 

The `CreateIsolatedSubaccount` transaction type is dispatched in `EndpointTx.sol` with no surrounding health check on the parent either: [3](#0-2) 

The user controls the `margin` field via `_isolatedMargin(txn.order.appendix)` embedded in the signed order. The signature check only verifies the user authorized the order — it does not constrain the margin to a safe amount relative to existing liabilities.

---

### Impact Explanation

A parent subaccount that has:
- Borrowed spot tokens (negative spot balance), or
- Open perp positions reducing health

...can have its quote balance stripped down to or below zero by specifying a large `margin` in the isolated subaccount order. After the transfer, the parent's `getHealth()` returns negative, meaning:

1. The parent is immediately undercollateralized and eligible for liquidation.
2. If the margin was the primary collateral buffer, the parent may have insufficient assets to cover its borrows — protocol losses follow.
3. The isolated subaccount holds the extracted margin and can trade or eventually return it, but the parent's debt remains on-chain.

This is the same solvency/accounting corruption class as the reference report: existing liabilities are ignored when validating a collateral-reducing operation.

---

### Likelihood Explanation

The `CreateIsolatedSubaccount` transaction is user-signed and submitted through the sequencer or via `submitSlowModeTransaction`. The user controls the `margin` value in the order appendix. No off-chain sequencer policy can substitute for the missing on-chain health check — the invariant must be enforced in the contract. Any user with an existing borrow or perp position can trigger this by crafting an isolated subaccount order with a margin large enough to push their parent health negative. [4](#0-3) 

---

### Recommendation

Add a parent health check immediately after the margin transfer in `createIsolatedSubaccount()`, mirroring the pattern used in `transferQuote`:

```solidity
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
    // ADD: enforce parent remains healthy after margin deduction
    require(
        IClearinghouse(clearinghouse).getHealth(
            txn.order.sender,
            IProductEngine.HealthType.INITIAL
        ) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
``` [1](#0-0) 

---

### Proof of Concept

1. User deposits 1000 USDC as collateral on parent subaccount.
2. User borrows 700 USDC worth of spot tokens on the parent (health is positive but reduced).
3. User submits a `CreateIsolatedSubaccount` order with `margin = 400` (encoded in `appendix`).
4. `createIsolatedSubaccount()` deducts 400 from parent quote balance and credits the isolated subaccount — no health check fires.
5. Parent quote balance is now 600, but borrows are 700 → parent health is negative.
6. Parent is undercollateralized; the isolated subaccount holds 400 quote that can be used to trade or eventually returned, while the parent's debt is unrecoverable without liquidation at a loss to the protocol. [1](#0-0) [2](#0-1)
