Audit Report

## Title
`SwapAllowlistExtension` Bypass via Router: `sender` Is Router Address, Not Original User - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap()`, which forwards it to `SwapAllowlistExtension.beforeSwap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` in the pool is the router contract, not the original user. The allowlist therefore gates the router address rather than the individual swapper, making it impossible to restrict individual users while still permitting router-mediated swaps: if the router is allowlisted, every user bypasses the per-user gate.

## Finding Description

**Root cause — `sender` binding in `MetricOmmPool.swap()`:**

`MetricOmmPool.sol` line 231 passes `msg.sender` directly as `sender`:
```solidity
_beforeSwap(
    msg.sender,   // ← always the immediate caller of pool.swap()
    recipient, zeroForOne, amountSpecified,
    priceLimitX64, packedSlot0Initial, bidPriceX64, askPriceX64, extensionData
);
```

`ExtensionCalling._beforeSwap()` (lines 160–176) encodes that value verbatim into the extension call:
```solidity
abi.encodeCall(IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, ...))
```

**Guard in `SwapAllowlistExtension.beforeSwap()` (line 37):**
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
`msg.sender` here is the pool (correct); `sender` is whoever called `pool.swap()`.

**Router path — `MetricOmmSimpleRouter.exactInputSingle()` (line 72–80):**
```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64, "", params.extensionData
);
```
The router calls `pool.swap()` directly, so `msg.sender` inside the pool is `address(MetricOmmSimpleRouter)`. The `sender` forwarded to the extension is the router, not the original EOA.

**Exploit flow:**
1. Pool is deployed with `SwapAllowlistExtension` as a `beforeSwap` hook.
2. Pool admin allowlists the router (`allowedSwapper[pool][router] = true`) so that allowlisted users can trade via the router.
3. Non-allowlisted attacker calls `MetricOmmSimpleRouter.exactInputSingle()` targeting the restricted pool.
4. Pool calls `_beforeSwap(msg.sender=router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
5. Attacker successfully swaps on a pool they are not individually authorized to access.

**Existing guards are insufficient:** The `SwapAllowlistExtension` has no mechanism to recover the original EOA from the router call. There is no `tx.origin` fallback, no forwarded-sender field in `extensionData`, and no alternative hook path that exposes the real user identity.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control feature. Its bypass allows any unprivileged user to execute swaps on pools explicitly configured to restrict trading to a named set of addresses (e.g., KYC-gated, institutional, or compliance-restricted pools). This constitutes broken core pool functionality: the allowlist gate is rendered ineffective for all router-mediated swaps, directly contradicting the pool's configured security invariant. Unauthorized swaps can alter pool state (bin cursor, bin balances) in ways that harm allowlisted LPs and traders who rely on the restricted-access guarantee.

## Likelihood Explanation
Any user can call `MetricOmmSimpleRouter` — it is a public, permissionless contract. The only precondition is that the pool admin has added the router to the allowlist, which is the natural action a pool admin takes when they want allowlisted users to be able to trade via the router. The bypass is therefore reachable in any realistic deployment of a router-enabled allowlisted pool, requires no special privileges, and is repeatable on every swap.

## Recommendation
Pass the original user identity through the extension layer rather than the immediate `msg.sender`. Two concrete options:

1. **Preferred — forwarded-sender field:** Add an explicit `originalSender` parameter to the pool's `swap()` signature (or encode it in `extensionData` at the router level) and have `ExtensionCalling._beforeSwap()` forward it as `sender` instead of `msg.sender`. The router would pass `msg.sender` (the EOA) as `originalSender`.

2. **Alternative — check both router and user:** Extend `SwapAllowlistExtension` to decode an allowlisted-user address from `extensionData` when `sender` is a known router, and verify that decoded address against the allowlist. This is weaker because it relies on the router correctly populating `extensionData`.
