### Title
Merkle Root Registration Decoupled From Token Deposit Allows Airdrop Claims to Be Permanently Locked - (File: `core/contracts/Airdrop.sol`)

---

### Summary
`Airdrop.registerMerkleRoot` commits claimable token amounts for all users in a given week without verifying that the contract holds sufficient token balance to cover those claims. The accounting state (Merkle root) and the actual token balance are two independent operations with no enforced relationship, directly mirroring the whitelist/balance desynchronization class from the reference report.

---

### Finding Description
`registerMerkleRoot` writes a Merkle root that cryptographically encodes the `totalAmount` each user is entitled to claim for a given week:

```solidity
function registerMerkleRoot(uint32 week, bytes32 merkleRoot)
    external
    onlyOwner
{
    pastWeeks += 1;
    require(week == pastWeeks, "Invalid week provided.");
    merkleRoots[week] = merkleRoot;   // no balance check
}
``` [1](#0-0) 

There is no corresponding `deposit` or `fund` function in the contract. Tokens must be sent to the contract address separately, entirely outside the `registerMerkleRoot` call. The contract never checks that `IERC20(token).balanceOf(address(this))` is sufficient to cover the aggregate claimable amounts encoded in the newly registered root.

When a user calls `claim`, the execution path reaches:

```solidity
SafeERC20.safeTransfer(IERC20(token), msg.sender, totalAmount);
``` [2](#0-1) 

If the contract's token balance is less than `totalAmount` (because the root was registered before or without a matching deposit), `safeTransfer` reverts. The user's `claimed[week][sender]` mapping was already written to `totalAmount` in `_verifyProof` before the transfer:

```solidity
claimed[week][sender] = totalAmount;
``` [3](#0-2) 

Because `_verifyProof` and the transfer are in the same transaction, the revert unwinds the `claimed` write too — so the user is not permanently marked as claimed. However, the user is still completely blocked from receiving tokens as long as the contract is underfunded, and there is no on-chain mechanism to force the owner to fund the contract after a root is registered.

---

### Impact Explanation
Any user with a valid Merkle proof for a registered week will have their `claim` call revert if the contract holds fewer tokens than their `totalAmount`. If the contract is entirely unfunded (root registered, no tokens deposited), **all users for that week are locked out**. There is no recovery path for users — they cannot force a deposit, and the owner can register additional roots for future weeks without ever funding past ones, compounding the locked-claim surface.

---

### Likelihood Explanation
The two-step pattern (register root off-chain → separately transfer tokens on-chain) is operationally common and error-prone. A deployment script that registers the root but fails mid-execution before the token transfer, a miscalculated aggregate amount, or a simple operational oversight produces this state. No malicious intent is required; the contract architecture provides no guard against it.

---

### Recommendation
Either:
1. Add a `deposit` function that accepts tokens and associates them with a specific week, and require that the deposited amount covers the week's total before `registerMerkleRoot` finalizes, **or**
2. Combine the token transfer and root registration into a single atomic function:
```solidity
function registerMerkleRoot(uint32 week, bytes32 merkleRoot, uint256 weeklyTotal) external onlyOwner {
    pastWeeks += 1;
    require(week == pastWeeks, "Invalid week provided.");
    SafeERC20.safeTransferFrom(IERC20(token), msg.sender, address(this), weeklyTotal);
    merkleRoots[week] = merkleRoot;
}
```
This ensures the contract's token balance always reflects the sum of all registered-but-unclaimed amounts.

---

### Proof of Concept
1. Owner deploys `Airdrop`, initializes with `token = NADO`.
2. Owner constructs a Merkle tree where Alice is entitled to `1000e18` NADO for week 1.
3. Owner calls `registerMerkleRoot(1, merkleRoot)` — succeeds, `pastWeeks = 1`. **No tokens are deposited.**
4. Alice calls `claim([{week: 1, totalAmount: 1000e18, proof: [...]}])`.
5. `_verifyProof` passes (valid proof, week registered, not sanctioned).
6. `SafeERC20.safeTransfer(IERC20(token), alice, 1000e18)` reverts — contract balance is 0.
7. Alice's claim is blocked indefinitely. The owner can register week 2, week 3, etc. without ever funding week 1. [1](#0-0) [4](#0-3)
