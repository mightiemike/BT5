### Title
Difficulty Calculation for Fork Blocks Uses Mainchain Ancestor Timestamps Instead of Fork Ancestor Timestamps - (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary

When computing the expected difficulty for a submitted fork block, all three chain implementations call `get_header_by_height`, which unconditionally returns the **mainchain** block at the reference height. For fork blocks whose divergence point is before the reference height, the correct reference block is the **fork's own ancestor** at that height, not the mainchain block. This is the same class of bug as M-05: a calculation uses one value source (mainchain timestamps) while the actual context demands a different source (fork ancestor timestamps), causing the difficulty enforcement to produce incorrect results.

### Finding Description

The difficulty retarget logic in all three non-Zcash chain implementations fetches the reference block by height from the mainchain:

**Bitcoin** (`contract/src/bitcoin.rs`, lines 78–86):
```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
```

**Litecoin** (`contract/src/litecoin.rs`, lines 86–93):
```rust
let first_block_height = prev_block_header.block_height - blocks_to_go_back;
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
```

**Dogecoin** (`contract/src/dogecoin.rs`, lines 286–297) — the developers themselves flagged this with a TODO:
```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

It always resolves through `mainchain_height_to_header`, which only maps mainchain blocks. Fork blocks stored via `store_fork_header` are placed only in `headers_pool` without a height-to-hash mapping, so they are unreachable through this function.

**Concrete attack scenario (Dogecoin Digishield, height ≥ 145 000):**

With `difficulty_adjustment_interval = 1`, the reference block is always 1 height back (`height_first = prev_block_header.block_height - 1`). Suppose the fork diverges at mainchain height H:

- Fork block at H+1 is submitted by the attacker with timestamp `T_fork` (up to 2 hours ahead of the mainchain's block at H+1, timestamp `T_main`).
- When the contract validates fork block at H+2, it computes the timespan as `fork_block_H+2_time − mainchain_block_H+1_time` (using `get_header_by_height(H+1)` → mainchain block).
- The correct timespan is `fork_block_H+2_time − fork_block_H+1_time`.
- Because `T_fork > T_main`, the contract-computed timespan is **longer** than the correct one.
- A longer timespan produces a **higher target** (lower difficulty) in `calculate_next_work_required`.
- The attacker submits fork block H+2 with `bits` matching the incorrectly computed (easier) target and a valid PoW hash meeting that lower difficulty.
- This repeats for every subsequent fork block, compounding the difficulty reduction.

The same structural flaw applies to Bitcoin and Litecoin at every difficulty adjustment boundary.

### Impact Explanation

The contract accepts fork blocks whose PoW does not meet the difficulty that the fork's own history would require. Because `chain_work` is accumulated from the submitted `bits` field, an attacker who submits many low-difficulty fork blocks can accumulate chainwork faster than a legitimate fork would allow. If the fork's chainwork exceeds the mainchain's, `reorg_chain` is triggered, replacing the canonical chain with the attacker's fork. After the reorg, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` operate against the corrupted canonical chain, returning incorrect inclusion proofs for transactions that were never confirmed on the real Bitcoin/Dogecoin network.

### Likelihood Explanation

The attacker is an unprivileged proof submitter or relayer. The entry point is the public `submit_blocks` function. The attacker needs:
1. Enough hashrate to produce fork blocks meeting the (reduced) difficulty — the reduction is bounded by the 2-hour timestamp window, so the difficulty reduction per block is modest but cumulative.
2. The ability to submit a sequence of fork blocks going back to the divergence point, which the relayer protocol already supports (the contract explicitly handles fork submission and reorg).

No privileged role, leaked key, or social engineering is required. The Dogecoin Digishield case (1-block adjustment interval) is the most exploitable because the difficulty is recalculated every block, so the error compounds on every fork block submitted.

### Recommendation

Replace `get_header_by_height` in the difficulty calculation path with a function that walks the fork chain backward through `prev_block_hash` links to find the true ancestor at the reference height. Specifically, starting from `prev_block_header`, follow `block_header.prev_block_hash` links in `headers_pool` until reaching the block at `first_block_height`, and use that block's timestamp as the reference time. This mirrors how Bitcoin Core's `GetAncestor` works and is the correct approach for fork-aware difficulty computation.

### Proof of Concept

1. Deploy the Dogecoin contract variant (height ≥ 145 000, Digishield active).
2. Submit a fork block at height H+1 with `time = mainchain_block_H+1.time + 7200` (2 hours ahead, the maximum allowed by `MAX_FUTURE_BLOCK_TIME_LOCAL`).
3. Observe that `get_next_work_required` for height H+2 calls `get_header_by_height(H+1)`, which returns the mainchain block with the earlier timestamp.
4. The computed `modulated_timespan` is `(fork_H+2_time − mainchain_H+1_time)` instead of `(fork_H+2_time − fork_H+1_time)`, inflated by up to 7200 seconds.
5. `calculate_next_work_required` produces a higher target (lower difficulty) for H+2.
6. Submit fork block H+2 with `bits` matching the inflated target and a valid scrypt hash meeting that lower difficulty.
7. Repeat: each subsequent fork block benefits from the same inflation, allowing the attacker to accumulate chainwork at a rate exceeding what the fork's true difficulty would permit.
8. Once the fork's `chain_work` exceeds the mainchain's, `reorg_chain` is triggered and `verify_transaction_inclusion` begins returning results against the attacker's chain.
