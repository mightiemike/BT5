### Title
Dual `IsOwner` + `IsDenomCreator` Requirement in `EditNFT` and `BurnNFT` Creates Permanent Post-Transfer Lock — (`x/nft/keeper/keeper.go`)

### Summary

`Keeper.EditNFT` and `Keeper.BurnNFT` both require the caller to simultaneously satisfy two independent checks: `IsOwner` (caller must be the current NFT owner) **and** `IsDenomCreator` (caller must be the denom creator). Once the denom creator transfers an NFT to any third party, these two conditions can never be satisfied by the same address again, permanently locking both edit and burn capabilities.

### Finding Description

`Keeper.EditNFT` enforces both checks sequentially with no escape hatch: [1](#0-0) 

`IsOwner` verifies the caller is the current on-chain owner: [2](#0-1) 

`IsDenomCreator` verifies the caller matches the immutable `denom.Creator` field set at `IssueDenom` time and never updated: [3](#0-2) 

`TransferOwner` (called by `msgServer.TransferNFT`) updates the NFT's owner record but leaves `denom.Creator` unchanged: [4](#0-3) 

After the transfer:
- **B (new owner) calls `EditNFT`**: `IsOwner` passes, `IsDenomCreator` fails → `ErrUnauthorized`
- **A (denom creator) calls `EditNFT`**: `IsOwner` fails (A no longer owns it) → `ErrUnauthorized`

The same dual-requirement logic applies identically to `BurnNFT`: [5](#0-4) 

`BurnNFTUnverified` (used only for IBC) bypasses the creator check, but `BurnNFT` (the user-facing path via `msgServer.BurnNFT`) does not: [6](#0-5) 

### Impact Explanation

After any `TransferNFT` from the denom creator to a third party:
1. **Permanent metadata lock**: `name`, `URI`, and `data` fields of the NFT can never be updated by anyone.
2. **Permanent burn lock**: The NFT cannot be burned via the standard user path by anyone. It is stuck in the new owner's possession forever.

Both effects are irreversible — there is no admin override, no governance path, and no re-transfer that restores the capability (re-transferring back to A would restore burn/edit for A, but that requires B's cooperation and is not a protocol guarantee).

### Likelihood Explanation

This is triggered by the ordinary, intended use of `MsgTransferNFT`. Any denom creator who transfers an NFT to a buyer or recipient immediately loses all edit and burn capability, and the recipient never had it. This is a normal, expected workflow for NFT marketplaces and gifting.

### Recommendation

Remove the `IsDenomCreator` check from `EditNFT` and `BurnNFT`. Ownership (`IsOwner`) is the correct and sufficient authorization for editing and burning an NFT one holds. The creator check is appropriate only for `MintNFT` (minting into a denom one controls). Alternatively, allow either the current owner **or** the denom creator to edit/burn, using an OR condition instead of AND.

### Proof of Concept

Keeper-level test (no mocks needed):
1. A calls `IssueDenom` → denom creator = A
2. A calls `MintNFT` (sender=A, owner=A) → NFT minted to A
3. A calls `TransferOwner` (src=A, dst=B) → NFT owner = B
4. B calls `EditNFT` → `IsOwner` passes, `IsDenomCreator` fails → assert `ErrUnauthorized` ✓
5. A calls `EditNFT` → `IsOwner` fails → assert `ErrUnauthorized` ✓
6. B calls `BurnNFT` → `IsOwner` passes, `IsDenomCreator` fails → assert `ErrUnauthorized` ✓
7. A calls `BurnNFT` → `IsOwner` fails → assert `ErrUnauthorized` ✓

All four assertions pass on unmodified production code, confirming the permanent dual lock.

```go
func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	_, err = k.IsDenomCreator(ctx, denomID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
}
```
