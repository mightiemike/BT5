### Title
L1Handler `paid_fee_on_l1` Dust Check Uses Exact-Zero Equality Instead of Threshold, Allowing Near-Free L1→L2 Message Execution - (`crates/blockifier/src/transaction/l1_handler_transaction.rs`)

### Summary

The blockifier's L1Handler execution path checks `paid_fee_on_l1 == Fee(0)` to reject fee-free L1→L2 messages. Because the check is an exact equality against zero rather than a comparison against the actual execution cost, any attacker who pays 1 wei on L1 bypasses the check entirely and gets their L1Handler transaction executed at essentially no cost to themselves, while the sequencer bears the full execution cost.

### Finding Description

In `crates/blockifier/src/transaction/l1_handler_transaction.rs`, after a successful L1Handler execution, the only fee adequacy check is:

```rust
// For now, assert only that any amount of fee was paid.
// The error message still indicates the required fee.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee {
            paid_fee,
            actual_fee: receipt.fee,
        },
    )));
}
``` [1](#0-0) 

The condition `paid_fee == Fee(0)` is the exact structural analog of the auction's `if balance == 0 { close_auction }` pattern: it uses exact equality against zero rather than a threshold comparison. Any `paid_fee >= Fee(1)` passes unconditionally, regardless of how large `receipt.fee` (the actual execution cost) is.

The `paid_fee_on_l1` value is sourced directly from the ETH `value` field of the `sendMessageToL2` call on L1, parsed in `crates/papyrus_base_layer/src/eth_events.rs`:

```rust
let fee = Fee(event.fee.try_into().map_err(EthereumBaseLayerError::FeeOutOfRange)?);
Ok(L1Event::LogMessageToL2 { tx, fee, l1_tx_hash, block_timestamp })
``` [2](#0-1) 

The Starknet L1 core contract only requires `fee > 0` (any non-zero ETH value), not `fee >= actual_execution_cost`. Integration tests confirm 1 wei is accepted:

```rust
let fee = 1_u8;
let message_to_l2 = contract.sendMessageToL2(...).value(U256::from(fee));
``` [3](#0-2) 

The normal `enforce_fee` path is entirely bypassed for L1Handler transactions because `create_tx_info` sets `max_fee: Fee::default()` (zero), making `enforce_fee()` return `false`:

```rust
impl TransactionInfoCreator for L1HandlerTransaction {
    fn create_tx_info(&self) -> TransactionInfo {
        TransactionInfo::Deprecated(DeprecatedTransactionInfo {
            ...
            max_fee: Fee::default(),
        })
    }
}
``` [4](#0-3) 

```rust
TransactionInfo::Deprecated(context) => context.max_fee != Fee(0),
``` [5](#0-4) 

So the `paid_fee == Fee(0)` check at lines 106–113 is the **sole** fee adequacy gate for L1Handler transactions. The comment on line 104 explicitly acknowledges the insufficiency: *"For now, assert only that any amount of fee was paid."*

### Impact Explanation

An attacker who sends `sendMessageToL2` with `value = 1 wei` on L1 causes the sequencer to:

1. Accept the L1 event and create an `L1HandlerTransaction` with `paid_fee_on_l1 = Fee(1)`.
2. Execute the L1Handler (committing all state changes).
3. Pass the `paid_fee == Fee(0)` check (since `1 != 0`).
4. Receive only 1 wei ETH from the L1 contract as compensation.

The actual execution cost (`receipt.fee`) can be orders of magnitude larger. The sequencer bears the full execution cost while the attacker pays 1 wei. This matches the **"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact"** critical impact category.

### Likelihood Explanation

The attack requires only a standard L1 transaction calling `sendMessageToL2` with `value = 1 wei`. No privileged access, no special contract, no frontrunning. The L1 Starknet core contract imposes no minimum fee floor. The attack is cheap to mount repeatedly.

### Recommendation

Replace the exact-zero equality check with a comparison against the actual computed fee:

```rust
// Before (vulnerable):
if paid_fee == Fee(0) {

// After (fixed):
if paid_fee < receipt.fee {
```

Or, if the intent is to keep a minimal threshold while the full fix is deferred, at minimum compare against a configurable `min_l1_handler_fee` constant rather than zero, mirroring the `min_gas_price` threshold pattern already used in `validate_tx_l2_gas_price_within_threshold`: [6](#0-5) 

### Proof of Concept

1. Deploy any L2 contract with an L1Handler entry point.
2. Call `sendMessageToL2(l2_contract, selector, payload)` on the Starknet L1 core contract with `.value(1)` (1 wei ETH).
3. The sequencer scrapes the `LogMessageToL2` event; `paid_fee_on_l1 = Fee(1)`.
4. The blockifier executes the L1Handler. `receipt.fee` reflects the true gas cost (e.g., thousands of wei).
5. The check `if paid_fee == Fee(0)` evaluates to `false` (since `1 != 0`), so execution commits.
6. The sequencer receives 1 wei ETH from L1 while having borne the full execution cost. [7](#0-6)
