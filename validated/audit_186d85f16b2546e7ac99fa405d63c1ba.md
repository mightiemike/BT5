### Title
`ValidResourceBounds::L1Gas` Misclassification in Protobuf Deserialization Causes Consensus Validators to Reject Valid `AllResources` Transactions - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf-to-Rust conversion for `ResourceBounds` silently downgrades an `AllResources` transaction (with zero `l2_gas` and `l1_data_gas`) to `ValidResourceBounds::L1Gas`. The gateway accepts such a transaction and computes its hash under the `AllResources` preimage. When the proposer later serializes the transaction for consensus P2P, the validator's deserialization path converts it back to `L1Gas`, causing `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` to fail with `DEPRECATED_RESOURCE_BOUNDS_ERROR`. The validator rejects the entire proposal, producing a consensus failure for a legitimately admitted transaction.

### Finding Description

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` uses a value-based heuristic to distinguish old (`L1Gas`) from new (`AllResources`) transactions:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

This heuristic is structurally ambiguous: a modern V3 transaction submitted via the RPC gateway with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is indistinguishable from a legacy `L1Gas` transaction once serialized to protobuf. The gateway's stateless validator explicitly permits this shape (test case `valid_l1_gas` uses `AllResourceBounds { l1_gas: NON_EMPTY, ..Default::default() }`). [2](#0-1) 

The gateway computes the transaction hash using `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash`. For `AllResources`, this includes `l1_data_gas` in the Poseidon preimage; for `L1Gas`, it does not:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [3](#0-2) 

The hash stored in `InternalRpcTransaction.tx_hash` is therefore computed under the `AllResources` preimage. When the proposer serializes the transaction for consensus P2P, `From<ValidResourceBounds> for protobuf::ResourceBounds` emits `l2_gas = 0` and `l1_data_gas = 0`: [4](#0-3) 

On the validator side, `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` calls `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`, which re-runs the heuristic and produces `ValidResourceBounds::L1Gas`. The subsequent `snapi_invoke.try_into()` (i.e., `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`) rejects `L1Gas` with `DEPRECATED_RESOURCE_BOUNDS_ERROR`:

```rust
// This conversion can fail only if the resource_bounds are not AllResources.
Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
``` [5](#0-4) 

This propagates as a `ProtobufConversionError` out of `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction`: [6](#0-5) 

The validator cannot reconstruct the `ConsensusTransaction` and rejects the proposal.

### Impact Explanation

A valid `InvokeV3` transaction with `AllResources(l2_gas=0, l1_data_gas=0)` is accepted by the gateway, assigned a hash under the `AllResources` preimage, and forwarded to the mempool. When the proposer includes it in a block and broadcasts the proposal, every validator fails to deserialize the transaction batch. The proposal is rejected, and the round must be retried. Any user who can submit a transaction to the gateway can trigger this condition deterministically. This matches: **High — Transaction conversion or signature/hash logic binds the wrong type or executable payload.**

### Likelihood Explanation

The gateway's stateless validator explicitly allows `AllResourceBounds` with zero `l2_gas` and `l1_data_gas` (the `valid_l1_gas` test case). No gateway-side guard prevents such a transaction from reaching the mempool. The protobuf round-trip tests in `rpc_transaction_test.rs` only exercise non-zero resource bounds, leaving the zero-value case untested. [7](#0-6) 

### Recommendation

Replace the value-based heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with a structural one: produce `ValidResourceBounds::L1Gas` only when `l1_data_gas` is **absent** from the protobuf message (`None`), and produce `ValidResourceBounds::AllResources` whenever `l1_data_gas` is **present** (even if zero). This is consistent with the existing `TODO(Shahak)` comment and correctly distinguishes pre-0.13.3 (`l1_data_gas` field absent) from post-0.13.3 (`l1_data_gas` field present but possibly zero) transactions:

```rust
let l1_data_gas_opt = value.l1_data_gas;
let l1_data_gas: ResourceBounds = l1_data_gas_opt
    .unwrap_or_default()
    .try_into()?;
Ok(if l1_data_gas_opt.is_none() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

### Proof of Concept

1. Submit via the JSON-RPC gateway an `InvokeV3` transaction with:
   ```json
   "resource_bounds": {
     "L1_GAS": { "max_amount": "0x1", "max_price_per_unit": "0x1" },
     "L2_GAS": { "max_amount": "0x0", "max_price_per_unit": "0x0" },
     "L1_DATA_GAS": { "max_amount": "0x0", "max_price_per_unit": "0x0" }
   }
   ```
2. The gateway's stateless validator accepts it (matches the `valid_l1_gas` test shape). The hash is computed via `get_invoke_transaction_v3_hash` under `AllResources`, including the zero `l1_data_gas` field in the Poseidon preimage. [8](#0-7) 
3. The transaction enters the mempool and is selected by the batcher.
4. The proposer serializes the `ConsensusTransaction` to protobuf. `From<ValidResourceBounds> for protobuf::ResourceBounds` emits `l2_gas = ResourceLimits { max_amount: 0, ... }` and `l1_data_gas = ResourceLimits { max_amount: 0, ... }`. [4](#0-3) 
5. Each validator calls `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction`. Inside, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` evaluates `l1_data_gas.is_zero() && l2_gas.is_zero()` → `true` → returns `ValidResourceBounds::L1Gas`. [9](#0-8) 
6. `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` matches `ValidResourceBounds::L1Gas` against the `AllResources` arm → falls through to the error branch → `DEPRECATED_RESOURCE_BOUNDS_ERROR`. [10](#0-9) 
7. `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction` returns `Err(DEPRECATED_RESOURCE_BOUNDS_ERROR)`. The validator cannot reconstruct the transaction batch and rejects the proposal. The consensus round fails.

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
