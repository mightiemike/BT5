Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of end-user, enabling allowlist bypass or DoS — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` is intended to restrict which addresses may swap in a pool. Its `beforeSwap` hook checks the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When users interact via `MetricOmmSimpleRouter`, the router is `msg.sender` at the pool level, not the actual user. This creates two mutually exclusive failure modes: allowlisting the router bypasses per-user restrictions entirely, while allowlisting individual users prevents them from using the router at all.

## Finding Description
**Root cause:** `MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap` (L230–240). `ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension (L149–177). `SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool address and `sender` is the direct caller of `pool.swap()` — the router, not the end-user (L37).

**Exploit flow (bypass path):**
1. Pool admin deploys pool with `SwapAllowlistExtension` in the `beforeSwap` slot.
2. Admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps.
3. Unprivileged attacker calls `router.exactInputSingle({pool: pool, recipient: attacker, ...})`.
4. Router calls `pool.swap(attacker, ...)` — pool sees `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router] == true` → passes.
6. Swap executes; attacker receives tokens from a pool intended to be restricted.

**Exploit flow (DoS path):**
1. Admin calls `setAllowedToSwap(pool, alice, true)` to allowlist a specific user.
2. Alice calls `router.exactInputSingle({pool: pool, ...})`.
3. Router calls `pool.swap(alice, ...)` — pool sees `msg.sender = router`.
4. Extension checks `allowedSwapper[pool][router] == false` → `NotAllowedToSwap` revert.
5. Alice cannot use the router at all.

No existing guard distinguishes the router intermediary from the end-user. The `recipient` field is available but is not checked by the extension.

## Impact Explanation
**Allowlist bypass (admin-boundary break):** When the router is allowlisted to support the standard periphery flow, any unprivileged address can swap in a pool intended to be restricted to a curated set of counterparties. Restricted pools may hold concentrated liquidity at oracle-anchored prices for specific counterparties; opening them to arbitrary swappers exposes LP principal to uninvited adverse selection and drains owed LP assets — a direct loss of user/LP principal satisfying the admin-boundary break impact gate.

**Core functionality DoS:** When individual users are allowlisted instead, those users cannot use `MetricOmmSimpleRouter` at all, breaking the primary user-facing swap entry point for any allowlisted pool.

## Likelihood Explanation
`SwapAllowlistExtension` is a production periphery extension. Any pool that deploys it and expects users to interact via `MetricOmmSimpleRouter` (the standard entry point) is immediately affected. No special attacker capability is required — a normal `exactInputSingle` call suffices. Both the extension and the router are expected production components, making this a realistic configuration. Likelihood is **Medium**: the extension must be configured and the router must be in use, but both are the expected production setup.

## Recommendation
The extension must gate on the economically relevant actor, not the intermediate dispatcher. The cleanest fix is to have the router encode `msg.sender` into `extensionData` before calling the pool (e.g., `abi.encode(msg.sender)` appended), and have `SwapAllowlistExtension.beforeSwap` decode the true initiator from that field when `sender` is a known router address. This requires a coordinated convention between router and extension but preserves full per-user enforcement. Alternatively, document that pools using `SwapAllowlistExtension` must not allowlist the router and users must call the pool directly — this is the least invasive fix but breaks the standard UX.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension configured in beforeSwap slot.
2. Admin calls setAllowedToSwap(pool, router, true)  // enable router-mediated swaps
3. Attacker (not allowlisted) calls:
       router.exactInputSingle({
           pool: pool,
           recipient: attacker,
           zeroForOne: true,
           amountIn: X,
           ...
       })
4. Router calls pool.swap(attacker, true, X, ...) — pool sees msg.sender = router
5. Pool calls extension.beforeSwap(router, attacker, ...)
6. Extension checks allowedSwapper[pool][router] == true → passes
7. Swap executes. Attacker receives tokens from a restricted pool.
   Per-user allowlist is completely bypassed.
```
