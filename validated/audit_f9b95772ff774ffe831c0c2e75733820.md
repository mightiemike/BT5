### Title
Gateway `validate_resource_bounds` Rejects Valid Client-Side Proving Transactions by Applying `min_gas_price` Threshold Without Considering Proving Mode — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` unconditionally checks `l2_gas.max_price_per_unit` against `config.min_gas_price` for every incoming `RpcTransaction`. Client-side proving transactions are protocol-required to carry `max_price_per_unit = 0` on all resource bounds (no fees are charged; the OS enforces gas only through `l2_gas.max_amount`). When the gateway is configured with `allow_client_side_proving = true` and `min_gas_price > 0`, every valid client-side proving transaction is unconditionally rejected at stateless admission — the same error path that rejects spam — before the proving-mode check is ever reached.

### Finding Description

`StatelessTransactionValidator::validate` calls checks in this order:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs:33-53
self.validate_resource_bounds(tx)?;   // ← runs first
...
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;  // ← runs after
    ...
}
```

Inside `validate_resource_bounds`:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs:71-76
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
        gas_price: resource_bounds.l2_gas.max_price_per_unit,
        min_gas_price: self.config.min_gas_price,
    });
}
```

The check is applied uniformly to every transaction regardless of whether it carries proof data. It does not branch on `tx.proof_facts.is_empty()` or any other proving-mode indicator.

The prover side enforces the opposite invariant. `validate_zero_fee_resource_bounds` in `crates/starknet_transaction_prover/src/proving/virtual_snos_prover.rs` (lines 401–443) explicitly requires:

```
l2_gas.max_price_per_unit = 0   (mandatory — no fees charged)
l2_gas.max_amount > 0           (mandatory — OS gas limit)
```

So the two components impose contradictory requirements on the same field when `min_gas_price > 0`:

| Component | Requirement on `l2_gas.max_price_per_unit` |
|---|---|
| Prover (`validate_zero_fee_resource_bounds`) | Must be **0** |
| Gateway (`validate_resource_bounds`) | Must be **≥ min_gas_price** |

When `min_gas_price > 0`, no client-side proving transaction can satisfy both constraints simultaneously. The gateway rejects the transaction before `validate_client_side_proving_allowed` is ever evaluated, so the `allow_client_side_proving` flag has no effect on these transactions.

The analog to the external bug is exact: in the DeFi report, `params.amount` (cash) was compared against a credit-denominated threshold without considering `exactAmountIn` mode. Here, `l2_gas.max_price_per_unit` (a fee field) is compared against a fee-enforcement threshold without considering the proving mode that mandates the field be zero.

### Impact Explanation

**High — Mempool/gateway admission rejects valid transactions before sequencing.**

Any user or application that submits a well-formed client-side proving invoke transaction (non-empty `proof_facts`, `max_price_per_unit = 0` on all resources, `l2_gas.max_amount > 0`) will receive a `MaxGasPriceTooLow` rejection from the gateway. The transaction never reaches the mempool or blockifier. The `allow_client_side_proving` configuration flag is rendered inoperative for all such transactions when `min_gas_price > 0`.

### Likelihood Explanation

The condition requires two non-default configuration values to be set simultaneously: `allow_client_side_proving = true` (enabling the proving feature) and `min_gas_price > 0` (enforcing a gas price floor). Both are plausible production settings — the proving feature is actively developed and the gas price floor is a standard anti-spam measure. The block data in `block_post_0_14_2.json` and `block_post_0_14_3.json` shows real transactions with `l2_gas.max_price_per_unit = 0x0`, confirming this pattern exists on-chain.

### Recommendation

In `validate_resource_bounds`, skip the `min_gas_price` check when the transaction is a client-side proving transaction:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs
fn validate_resource_bounds(
    &self,
    tx: &RpcTransaction,
) -> StatelessTransactionValidatorResult<()> {
    if !self.config.validate_resource_bounds {
        return Ok(());
    }

    let resource_bounds = *tx.resource_bounds();

    // Determine if this is a client-side proving transaction.
    let is_proving_tx = matches!(
        tx,
        RpcTransaction::Invoke(RpcInvokeTransaction::V3(t)) if !t.proof_facts.is_empty()
    );

    if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        && !is_proving_tx  // proving txs legitimately have zero fee
    {
        return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
    }

    // Skip price floor check for proving transactions: they must have max_price_per_unit = 0.
    if !is_proving_tx
        && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price
    {
        return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
            gas_price: resource_bounds.l2_gas.max_price_per_unit,
            min_gas_price: self.config.min_gas_price,
        });
    }
    ...
}
```

### Proof of Concept

1. Configure the gateway with `allow_client_side_proving = true`, `validate_resource_bounds = true`, and `min_gas_price = 1`.

2. Submit an `RpcInvokeTransactionV3` with:
   - `proof_facts`: non-empty (valid proof facts)
   - `resource_bounds.l2_gas.max_price_per_unit = 0` (required by prover)
   - `resource_bounds.l2_gas.max_amount = 100_000_000` (required by prover)
   - All other `max_price_per_unit` fields = 0

3. `StatelessTransactionValidator::validate` is called. At line 71 of `stateless_transaction_validator.rs`, `0 < 1` is true, so `MaxGasPriceTooLow` is returned immediately.

4. `validate_client_side_proving_allowed` (line 46) is never reached. The transaction is rejected despite being a valid client-side proving transaction.
