### Title
Unbounded Bonus Drain via Repeated TriggerExitFromTier/ClearPosition Cycling — (`x/tieredrewards/keeper/msg_server.go`)

### Summary
A position owner can cycle `TriggerExitFromTier` → wait exit duration → `ClearPosition` indefinitely to collect bonus rewards for each exit window without ever undelegating, draining `RewardsPoolName` without bound.

### Finding Description

`ClearPosition` claims bonus rewards up to `ExitUnlockAt` and then resets `ExitUnlockAt` to zero via `ClearExit`. After this reset, `validateTriggerExit` permits a new `TriggerExitFromTier` call because `HasTriggeredExit()` returns false when `ExitUnlockAt` is zero.

The `applyBonusAccrualCheckpoint` function sets `LastBonusAccrual = ExitUnlockAt` (not `blockTime`) when the exit window has elapsed: [1](#0-0) 

This means after `ClearExit` resets `ExitUnlockAt` to zero and a new trigger sets a fresh `ExitUnlockAt`, the next `claimRewards` call computes bonus from the old `ExitUnlockAt` to the new one — paying a full exit window's worth of bonus per cycle.

`validateTriggerExit` only blocks re-triggering if `HasTriggeredExit()` is true: [2](#0-1) 

`validateClearPosition` does not prevent clearing when the exit window has elapsed and the position remains delegated — it only checks for unbonding/undelegated state: [3](#0-2) 

`computeSegmentBonus` caps `segmentEnd` at `ExitUnlockAt` only when it is non-zero, so the gap period between cycles also accrues bonus: [4](#0-3) 

The bonus payment is sent directly from `RewardsPoolName`: [5](#0-4) 

### Impact Explanation
Each cycle drains one exit window's worth of bonus from `RewardsPoolName`. With N cycles, the attacker extracts N × (exitDuration bonus) while keeping their delegation intact. The pool can be fully drained given sufficient cycles, constituting direct theft of protocol funds.

### Likelihood Explanation
The attack requires only two standard user-level transactions (`MsgTriggerExitFromTier`, `MsgClearPosition`) repeated in a loop. No privileged access, governance, or special conditions are needed. Any position owner can execute this on mainnet.

### Recommendation
Add a permanent "exit-triggered" flag to the position that is set by `TriggerExit` and is **never** cleared by `ClearExit`. `validateTriggerExit` should reject re-triggering if this flag is set, regardless of whether `ExitUnlockAt` has been reset. Alternatively, `ClearPosition` should only be callable before the exit window elapses (i.e., as a "cancel exit" action), and once the window elapses the position must proceed to undelegation/withdrawal without the ability to re-enter the bonus-accruing state.

func (k Keeper) validateClearPosition(ctx context.Context, pos types.PositionState, owner string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
		isUnbonding, err := k.isUnbonding(ctx, pos.DelegatorAddress)
		if err != nil {
			return err
		}
		if isUnbonding {
			return types.ErrPositionUnbonding
		}

		if !pos.IsDelegated() {
			return types.ErrPositionNotDelegated
		}

	}

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return err
	}

	if tier.IsCloseOnly() {
		return types.ErrTierIsCloseOnly
	}

	return nil
}
