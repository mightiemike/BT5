### Title
Missing Zero-Address Validation in `Endpoint.initialize()` for Critical Protocol Addresses — (`File: core/contracts/Endpoint.sol`)

---

### Summary
`Endpoint.initialize()` accepts six critical address parameters — `_sequencer`, `_offchainExchange`, `_clearinghouse`, `_verifier`, `_sanctions`, and `_endpointTx` — without any zero-address validation. No recovery setters exist for `sequencer`, `clearinghouse`, `verifier`, or `sanctions`. If any of these are accidentally set to `address(0)`, the corresponding protocol functionality is permanently bricked with no on-chain remediation path.

---

### Finding Description
`Endpoint.initialize()` assigns all six address parameters directly to storage without checking for `address(0)`:

```solidity
sequencer = _sequencer;
clearinghouse = _clearinghouse;
offchainExchange = _offchainExchange;
verifier = IVerifier(_verifier);
sanctions = ISanctionsList(_sanctions);
endpointTx = _endpointTx;
``` [1](#0-0) 

`sequencer` is stored in `EndpointStorage` as a plain `address internal sequencer` with no public setter anywhere in the codebase. [2](#0-1) 

Similarly, `Clearinghouse.initialize()` sets `_quote` and `_endpoint` (via `setEndpoint()`) without zero-address validation, and no setter for `quote` exists anywhere:

```solidity
setEndpoint(_endpoint);
quote = _quote;
clearinghouseLiq = _clearinghouseLiq;
withdrawPool = _withdrawPool;
``` [3](#0-2) 

`EndpointGated.setEndpoint()` is `internal` and also performs no zero-address check: [4](#0-3) 

`BaseWithdrawPool._initialize()` sets `_clearinghouse` and `_verifier` without zero-address checks and has no recovery setters: [5](#0-4) 

---

### Impact Explanation

**`sequencer = address(0)`**: The `onlySequencer` modifier gates all sequencer-submitted transaction batches. With `sequencer == address(0)`, `msg.sender == sequencer` can never be satisfied by any real caller, permanently preventing all sequencer transaction processing. The entire protocol's settlement, liquidation, and withdrawal pipeline is bricked. No setter exists to recover.

**`clearinghouse = address(0)` in `Endpoint`**: All calls to `clearinghouse.depositCollateral()`, `clearinghouse.withdrawCollateral()`, `clearinghouse.liquidateSubaccount()`, etc. revert. Every user-facing action that touches collateral or positions fails permanently.

**`quote = address(0)` in `Clearinghouse`**: `_decimals()` calls `IERC20Base(address(0)).decimals()`, which reverts. This breaks `depositCollateral`, `withdrawCollateral`, `withdrawInsurance`, and `checkMinDeposit` — the full deposit/withdrawal surface. No setter for `quote` exists.

**`verifier = address(0)` in `BaseWithdrawPool`**: `submitFastWithdrawal` calls `Verifier(address(0)).requireValidTxSignatures()`, which reverts, permanently disabling fast withdrawals. [6](#0-5) 

---

### Likelihood Explanation
Likelihood is low-to-medium. These are `initializer` functions called once during deployment, typically by a trusted deployer. However, deployment scripts are error-prone, and the absence of any on-chain guard means a single misconfigured deployment permanently bricks the affected contract with no recovery path. The original report's referenced contracts suffered the same class of issue and required a dedicated fix commit.

---

### Recommendation

Add explicit zero-address guards for all critical address parameters in each initializer:

```solidity
// In Endpoint.initialize()
require(_sequencer != address(0), "zero sequencer");
require(address(_clearinghouse) != address(0), "zero clearinghouse");
require(_verifier != address(0), "zero verifier");
require(_sanctions != address(0), "zero sanctions");
require(_offchainExchange != address(0), "zero offchainExchange");

// In Clearinghouse.initialize()
require(_endpoint != address(0), "zero endpoint");
require(_quote != address(0), "zero quote");
require(_clearinghouseLiq != address(0), "zero clearinghouseLiq");
require(_withdrawPool != address(0), "zero withdrawPool");

// In BaseWithdrawPool._initialize()
require(_clearinghouse != address(0), "zero clearinghouse");
require(_verifier != address(0), "zero verifier");
```

Additionally, add owner-restricted setters for `sequencer`, `quote`, `verifier`, and `sanctions` so that a misconfigured deployment can be corrected without redeployment.

---

### Proof of Concept

1. Deploy `Endpoint` proxy and call `initialize()` passing `address(0)` as `_sequencer`.
2. Attempt to call any sequencer-gated function (e.g., `submitTransactions`). The call reverts because `msg.sender != address(0)`.
3. No setter exists to update `sequencer`; the contract is permanently bricked.
4. Repeat with `_clearinghouse = address(0)` in `Clearinghouse.initialize()`: any call to `depositCollateral` or `withdrawCollateral` reverts at `_decimals()` → `IERC20Base(address(0)).decimals()`.
5. No `setQuote()` function exists anywhere in the codebase to recover. [7](#0-6) [8](#0-7)
