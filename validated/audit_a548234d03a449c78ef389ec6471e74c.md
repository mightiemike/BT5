### Title
Withdrawal Fee Bypass via Inconsistent Dual-Path Execution for `WithdrawCollateral` — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `WithdrawCollateral` transaction type is processed by two distinct execution paths in `EndpointTx.sol`: a fast path via `processTransactionImpl` (sequencer-submitted) and a slow path via `processSlowModeTransactionImpl` (user-submitted). The fast path charges the product-specific percentage-based `withdrawFeeX18`, while the slow path omits this fee entirely. Any unprivileged user can exploit this inconsistency to bypass the withdrawal fee on arbitrarily large withdrawals by routing through the slow mode path.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `WithdrawCollateral` branch charges the configured withdrawal fee before executing the withdrawal:

```solidity
// EndpointTx.sol lines 413–436
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.SignedWithdrawCollateral memory signedTx = abi.decode(...);
    validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, true);
    chargeFee(
        signedTx.tx.sender,
        spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,  // percentage fee
        signedTx.tx.productId
    );
    clearinghouse.withdrawCollateral(...);
}
``` [1](#0-0) 

In `EndpointTx.processSlowModeTransactionImpl`, the identical `WithdrawCollateral` branch omits the `chargeFee` call entirely:

```solidity
// EndpointTx.sol lines 217–229
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(...);
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(
        txn.sender, txn.productId, txn.amount, address(0), nSubmissions
    );
    // No chargeFee call — withdrawFeeX18 is never applied
}
``` [2](#0-1) 

The slow mode submission path in `submitSlowModeTransactionImpl` only charges a flat `SLOW_MODE_FEE` (not the percentage-based `withdrawFeeX18`) and is open to any non-sanctioned user:

```solidity
// EndpointTx.sol lines 369–372
} else {
    chargeSlowModeFee(_getQuote(), sender);
    slowModeFees += SLOW_MODE_FEE;
}
``` [3](#0-2) 

The `withdrawFeeX18` is a product-level configurable percentage fee stored in `SpotEngine` config. For large withdrawals, this fee is orders of magnitude larger than the flat `SLOW_MODE_FEE`. The two paths are supposed to execute the same economic operation but produce different fee outcomes, breaking the invariant that every `WithdrawCollateral` incurs `withdrawFeeX18`.

---

### Impact Explanation

A user withdrawing a large amount (e.g., $1,000,000 USDC) with a `withdrawFeeX18` of even 5 bps (0.05%) would owe $500 in fees via the fast path. By routing through slow mode, they pay only the flat `SLOW_MODE_FEE` (a fixed small amount in quote tokens) and zero `withdrawFeeX18`. The protocol's `sequencerFee` accounting is corrupted: fees that should have been credited to `sequencerFee[productId]` are never collected. This is a direct, repeatable asset delta against protocol fee revenue with no upper bound on the bypassed amount.

---

### Likelihood Explanation

The slow mode path is unconditionally accessible to any non-sanctioned user holding a valid subaccount. The only cost is a 3-day delay (`SLOW_MODE_TX_DELAY`) and the flat slow mode fee. For any withdrawal where `amount * withdrawFeeX18 > SLOW_MODE_FEE`, the slow path is strictly economically superior. This threshold is crossed at modest withdrawal sizes for any non-trivial `withdrawFeeX18`. The exploit requires no special privileges, no oracle manipulation, and no coordination — only knowledge of the two paths.

---

### Recommendation

Add the `withdrawFeeX18` charge to the slow mode execution path for `WithdrawCollateral`, mirroring the fast path:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(...);
    validateSender(txn.sender, sender);
    chargeFee(
        txn.sender,
        spotEngine.getConfig(txn.productId).withdrawFeeX18,
        txn.productId
    );
    clearinghouse.withdrawCollateral(...);
}
```

Alternatively, introduce a shared internal `_executeWithdrawCollateral` helper that both paths call, ensuring fee logic cannot diverge.

---

### Proof of Concept

1. User holds 1,000,000 USDC in their subaccount. `withdrawFeeX18` for USDC is 5 bps.
2. Fast path cost: `1,000,000 * 0.0005 = $500` in fees + signature required.
3. User instead calls `Endpoint.submitSlowModeTransaction(withdrawCollateralTx)`.
4. `submitSlowModeTransactionImpl` charges only the flat `SLOW_MODE_FEE` (e.g., $1 equivalent).
5. After 3 days, user or sequencer calls `executeSlowModeTransaction`.
6. `processSlowModeTransactionImpl` calls `clearinghouse.withdrawCollateral` with no `chargeFee` call.
7. User receives full 1,000,000 USDC minus only the flat fee. Protocol loses $499 in fee revenue.
8. Repeatable for every withdrawal, with no per-transaction limit. [2](#0-1) [4](#0-3)
