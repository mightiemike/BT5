### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Check Added to Prevent 64-Byte Proof Forgery — (`File: contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, unpermissioned NEAR contract method. Any caller can invoke it directly, bypassing the coinbase Merkle proof validation that was specifically introduced in `verify_transaction_inclusion_v2` to close the 64-byte transaction Merkle proof forgery vulnerability. Rust's `#[deprecated]` attribute is a compile-time lint only; it does not restrict on-chain method dispatch. The result is that the security fix in v2 can be trivially circumvented by any unprivileged NEAR account.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to fix a known Bitcoin SPV vulnerability: an attacker can supply a 64-byte internal Merkle tree node as a fake `tx_id` and construct a proof path that correctly hashes to the block's `merkle_root`, causing the verifier to return `true` for a transaction that does not exist. The fix — verifying a coinbase Merkle proof at index 0 — is applied only inside `verify_transaction_inclusion_v2`. [1](#0-0) 

However, `verify_transaction_inclusion` (v1) is still decorated `pub fn` with only `#[pause]` and `#[deprecated]`: [2](#0-1) 

`#[deprecated]` in Rust is a compiler-side lint warning. It has zero effect on NEAR contract ABI exposure. The method name `verify_transaction_inclusion` remains a valid callable entry point on the deployed contract. Any NEAR account — including an attacker — can call it directly with `ProofArgs`, which contains no `coinbase_tx_id` or `coinbase_merkle_proof` fields: [3](#0-2) 

The v1 function performs only a standard Merkle root recomputation with no coinbase anchor: [4](#0-3) 

This is structurally identical to the BorrowLogic.sol analog: a critical check (coinbase proof) is present in one code path (`verify_transaction_inclusion_v2`) but entirely absent in a parallel, still-reachable code path (`verify_transaction_inclusion`).

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate an asset release (e.g., a BTC bridge minting wrapped BTC) will receive `true` for a forged proof. The attacker can claim a Bitcoin transaction was confirmed in a real, stored block when no such transaction exists. This enables theft of bridged assets or any other action gated on SPV proof acceptance.

The contract's own documentation acknowledges the broken invariant in v1: [5](#0-4) 

---

### Likelihood Explanation

- **Entry path**: Fully open. Any NEAR account can call `verify_transaction_inclusion` with no role, stake, or permission requirement.
- **Attack complexity**: The 64-byte Merkle forgery technique is publicly documented (BitMEX blog, referenced in the contract source at line 268). The attacker only needs a real block already stored in the contract's `headers_pool` and `mainchain_header_to_height` index.
- **No privileged access required**: The attacker does not need to control the relayer, any admin role, or any private key. [6](#0-5) 

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract ABI entirely, or gate it so it is unreachable by external callers. The simplest fix is to change its visibility from `pub fn` to `pub(crate) fn` (or remove the `#[near]` export), forcing all callers — on-chain and off-chain — to use `verify_transaction_inclusion_v2`. The `#[deprecated]` attribute alone provides no on-chain protection.

---

### Proof of Concept

1. A real Bitcoin block `B` is stored in the contract (submitted by the relayer). Its `merkle_root` is `R`.
2. The attacker identifies two adjacent leaf hashes `H_L` and `H_R` in block `B`'s Merkle tree such that `SHA256d(H_L || H_R)` equals some internal node `N` at depth `d`.
3. The attacker constructs a Merkle proof path from `N` up to `R` (all intermediate nodes are real, taken from the block).
4. The attacker calls `verify_transaction_inclusion` with:
   - `tx_id = N` (the 64-byte internal node hash, not a real transaction)
   - `tx_block_blockhash = B`
   - `tx_index` = the index corresponding to `N`'s position
   - `merkle_proof` = the valid path from `N` to `R`
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, index, proof)` correctly recomputes `R`.
6. The function returns `true`.
7. Any bridge contract consuming this result releases funds for a transaction that never existed on Bitcoin. [7](#0-6) [8](#0-7)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

        );
