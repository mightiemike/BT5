### Title
SHA-1 Collision in `TreeCache` Atom Deduplication Enables Non-Canonical Serialization - (`src/serde/tree_cache.rs`)

### Summary

The `TreeCache` in `src/serde/tree_cache.rs` uses SHA-1 — a cryptographically broken 160-bit hash — to identify and deduplicate atoms during compressed CLVM serialization. A random 8-byte salt is applied as a mitigation, but SHA-1 is fundamentally broken and the salt only raises the bar rather than eliminating the collision risk. If an attacker crafts two distinct atoms whose salted SHA-1 hashes collide, the serializer silently treats them as identical, emitting a back-reference to the wrong node. The deserialized tree then differs from the original, enabling non-canonical serialization and potential consensus divergence on the Chia network.

### Finding Description

`TreeCache` is the shadow-tree data structure used by the incremental compressed serializer (`src/serde/incremental.rs`) to track which sub-trees have already been serialized and to emit back-references (`0xfe` tokens) instead of re-serializing duplicate content.

Atoms are identified by a 20-byte SHA-1 digest computed over a random 8-byte salt concatenated with the atom's raw bytes:

```rust
// src/serde/tree_cache.rs lines 16-21
fn hash_atom(salt: &[u8], blob: &[u8]) -> Bytes20 {
    let mut ctx = Sha1::default();
    ctx.update(salt);
    ctx.update(blob);
    ctx.finalize().into()
}
```

This digest is used as the sole key in `atom_lookup`:

```rust
// src/serde/tree_cache.rs line 110
atom_lookup: HashMap<Bytes20, u32, RandomState>,
```

When a new atom is encountered during `update()`, the code looks up its SHA-1 hash in `atom_lookup`. On a hit, the atom is unconditionally merged with the existing `NodeEntry` — **no byte-level content comparison is performed**:

```rust
// src/serde/tree_cache.rs lines 251-261
let ne = match self.atom_lookup.entry(hash) {
    Entry::Occupied(ne) => {
        // we already have a node with this hash
        let idx = *ne.get();
        e.insert(idx);
        stack.push(idx);
        continue;   // ← atom B silently aliased to atom A's NodeEntry
    }
    Entry::Vacant(ne) => ne,
};
```

The developers acknowledge the risk in the struct comment:

```rust
// src/serde/tree_cache.rs lines 134-137
/// We compute hash-trees using SHA-1 in order to determine whether the
/// trees are identical or not. To mitigate malicious SHA-1 hash collisions,
/// we salt the hashes
salt: [u8; 8],
```

The salt is generated once per `TreeCache` instance at construction time:

```rust
// src/serde/tree_cache.rs lines 141-149
pub fn new(sentinel: Option<NodePtr>) -> Self {
    let mut rng = rand::rng();
    Self {
        sentinel_node: sentinel,
        atom_lookup: HashMap::with_hasher(RandomState::default()),
        salt: rng.random(),
        ..Default::default()
    }
}
```

**Collision consequence**: When atom `B` collides with atom `A` under the salted SHA-1, `B` is mapped to `A`'s `NodeEntry`. Subsequently, when `find_path()` is called for `B`, it returns the back-reference path to `A`'s position in the parse stack. The serializer then emits `0xfe <path-to-A>` in place of `B`'s actual bytes. A deserializer following that back-reference reconstructs `A`'s content at the position where `B` should appear, producing a tree that differs from the original.

### Impact Explanation

The corrupted output is the serialized CLVM byte stream: the back-reference token `0xfe` followed by a path that resolves to the wrong atom. Any consumer that deserializes this stream — a full node, a mempool validator, a wallet — reconstructs a different CLVM tree than the one that was serialized. Concretely:

- **Consensus divergence**: A node that serializes a puzzle or solution and a node that deserializes the same bytes may evaluate different programs, disagreeing on coin validity.
- **Puzzle-hash mismatch**: If the serialized form is used to compute a puzzle hash (e.g., for coin creation), the hash will not match the hash of the program that actually executes.
- **Incorrect program execution**: A puzzle solution containing atom `B` could be serialized as if it contained atom `A`, causing the evaluator on the receiving end to run a different branch of the puzzle.

### Likelihood Explanation

SHA-1 is cryptographically broken. The SHAttered attack (2017) demonstrated practical chosen-prefix collisions requiring approximately 2^63 SHA-1 compression function calls. The 8-byte random salt raises the difficulty because an attacker must find a collision for an unknown salt value, converting the problem from a chosen-prefix collision to a second-preimage attack on a salted SHA-1. This is significantly harder than a plain SHA-1 collision and is not currently feasible with known techniques.

However, the analogy to the external report is direct: the external report's vulnerability was also considered infeasible at the time of filing (requiring millions of dollars of compute) yet was accepted as Medium severity because advancing hardware and algorithms make the attack progressively cheaper. SHA-1 is already broken at the chosen-prefix level; the salt is the only remaining barrier. As SHA-1 cryptanalysis advances and compute costs fall, the salted second-preimage gap will narrow. Furthermore, the `TreeCache` is used in the incremental serializer that processes attacker-controlled CLVM bytes, giving an adversary full control over the atom content on both sides of the desired collision.

### Recommendation

Replace SHA-1 with SHA-256 (already used elsewhere in the codebase, e.g., `src/treehash.rs`) for atom identity in `TreeCache`. SHA-256 is not known to be broken and provides 256-bit collision resistance. The `Bytes20` type alias and `atom_lookup` key type should be widened to `[u8; 32]`. The salt can be retained as an additional defense-in-depth measure but is no longer the primary protection.

Alternatively, replace the hash-based deduplication with a direct byte-content comparison (e.g., using a `HashMap<Vec<u8>, u32>` keyed on the raw atom bytes), which is collision-proof by construction.

### Proof of Concept

1. Obtain or observe a `TreeCache` instance's `salt` (e.g., via a debug build or by instrumenting the serializer).
2. Using known SHA-1 collision techniques adapted for the salted prefix, find two distinct byte strings `A` and `B` such that `SHA1(salt || A) == SHA1(salt || B)`.
3. Construct a CLVM program whose serialized form contains both atom `A` (serialized first) and atom `B` (serialized second, at a position where a back-reference would be shorter than re-serializing `B`).
4. Pass this program through the incremental serializer (`src/serde/incremental.rs` `Serializer`).
5. The serializer calls `tree_cache.find_path(node_B)`, which returns the path to `A`'s position because `B` was aliased to `A`'s `NodeEntry` in `atom_lookup`.
6. The output byte stream contains `0xfe <path-to-A>` at the position of `B`.
7. Deserializing the output reconstructs `A`'s bytes at `B`'s position, yielding a tree that differs from the original program.
