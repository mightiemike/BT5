### Title
Fork Difficulty Validation Reads Main-Chain Ancestor Instead of Fork Ancestor, Enabling Under-Difficulty Chain Reorg — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`)

### Summary

When a fork block is submitted, the difficulty calculation fetches the reference block at the start of the adjustment interval via `get_header_by_height`, which unconditionally reads from `mainchain_height_to_header`. If the fork diverges from the main chain before that interval boundary, the function returns the **main-chain** block at that height rather than the fork's own ancestor. The resulting difficulty target is computed against the wrong timestamp, allowing fork blocks whose `bits` field encodes a lower-than-required difficulty to pass `check_pow`/`check_target`. A fork built with systematically under-difficulty blocks can accumulate enough chainwork to trigger `reorg_chain`, corrupting the canonical chain and invalidating downstream `verify_transaction_inclusion` results.

---

### Finding Description

**Root cause — `get_header_by_height` is main-chain-only**

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // always the canonical map
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

Fork blocks are stored only in `headers_pool` (via `store_fork_header`), never in `mainchain_height_to_header`. [2](#0-1) 

By contrast, `get_prev_header` correctly follows the fork's own `prev_block_hash` chain: [3](#0-2) 

**Bitcoin path — called at every 2016-block boundary**

`get_next_work_required` (bitcoin.rs) calls `get_header_by_height(first_block_height)` to obtain the timestamp of the block at the start of the retarget window: [4](#0-3) 

If the fork diverged from the main chain before `first_block_height`, the returned block is the main-chain block at that height, not the fork's ancestor. The timestamp difference feeds directly into `calculate_next_work_required`, producing a wrong `bits` value.

**Dogecoin path — called on every block after height 145 000**

After block 145 000, Dogecoin adjusts difficulty every block (`difficulty_adjustment_interval = 1`). `get_next_work_required` (dogecoin.rs) calls `get_header_by_height(height_first)` where `height_first = prev_height − 1`: [5](#0-4) 

The developers themselves flagged this with a TODO:

> `// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor` [6](#0-5) 

For any Dogecoin fork block at height H where the fork diverged before H−2, the timestamp used for difficulty calculation is the main-chain block's timestamp, not the fork ancestor's.

**State mutation after wrong validation**

`submit_block_header` (dogecoin) runs `check_target` (which calls `check_pow` → `get_next_work_required`) and then unconditionally calls `submit_block_header_inner`, which may call `reorg_chain`: [7](#0-6) 

`reorg_chain` rewrites `mainchain_height_to_header`, `mainchain_header_to_height`, and `mainchain_tip_blockhash`: [8](#0-7) 

The canonical-chain state is therefore mutated based on a difficulty check that was performed against the wrong reference block — the direct analog of the external report's "state mutation after a validation that used the wrong phase context."

---

### Impact Explanation

- **Canonical chain corruption**: A fork whose blocks carry a `bits` value that encodes a lower difficulty than the protocol requires can pass `check_pow` because the expected `bits` was computed from the wrong (main-chain) ancestor timestamp. If the fork's cumulative chainwork exceeds the main chain's, `reorg_chain` promotes it.
- **False SPV proof results**: After the reorg, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` answer queries against the new (attacker-controlled) canonical chain. Transactions that were confirmed are no longer confirmed; attacker-planted transactions may appear confirmed.
- **Severity**: High — permanent canonical-chain corruption reachable by an unprivileged `submit_blocks` caller.

---

### Likelihood Explanation

- **Dogecoin**: Every fork block submitted for a fork that diverged more than two blocks ago is affected. No special timing or privileged access is required.
- **Bitcoin/Litecoin**: Triggered only at 2016-block retarget boundaries, but those occur regularly and a long-running fork will cross them.
- **Attacker cost**: The attacker must mine blocks with valid PoW, but at the incorrectly computed (lower) difficulty. The timestamp gap between the main chain and a fork with artificially early timestamps can be large enough to push the computed target to the protocol minimum (`proof_of_work_limit_bits`), making mining trivial.
- **Entry point**: `submit_blocks` — fully unprivileged, payable, open to any NEAR account. [9](#0-8) 

---

### Recommendation

Replace the `get_header_by_height` call inside `get_next_work_required` with an ancestor-traversal that walks the fork's own `prev_block_hash` chain (using `get_prev_header`) until it reaches the required height. This mirrors how `get_median_time_past` already correctly traverses the fork chain. The fix must be applied in both `bitcoin.rs` and `dogecoin.rs` (and `litecoin.rs` if it shares the same pattern).

---

### Proof of Concept

**Dogecoin scenario (most direct)**

1. The main chain is at height 200 000. The relayer has submitted blocks 199 990–200 000 to the contract.
2. An attacker constructs a fork starting at height 199 990 (divergence point D = 199 990). Fork blocks carry timestamps just above their own MTP (the minimum allowed), so fork timestamps are, say, 600 s earlier per block than the main chain.
3. The attacker submits fork block at height 200 000 (H = 200 000). `get_next_work_required` fetches `get_header_by_height(199 999)` — the **main-chain** block at 199 999, whose timestamp is ~600 s later than the fork's ancestor at 199 999.
4. `calculate_next_work_required` computes `modulated_timespan` using `time(fork_prev=199999_fork) − time(mainchain_199998)`. The inflated denominator shrinks the computed target, yielding a lower `bits` (easier difficulty).
5. The attacker mines the fork block against this easier target, satisfying `U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits)`.
6. Repeated for each subsequent fork block. Each block's difficulty is computed against the wrong ancestor, remaining artificially low.
7. Once the fork's `chain_work` exceeds the main chain's, `reorg_chain` is triggered, rewriting the canonical chain to the attacker's fork.
8. Subsequent calls to `verify_transaction_inclusion` now answer against the attacker's chain.

    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
