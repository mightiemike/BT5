### Title
Missing `expiration` enforcement in `createIsolatedSubaccount()` allows margin transfer after order expiry — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary
`createIsolatedSubaccount()` in `OffchainExchange.sol` processes user-signed `Order` structs that carry an `expiration` field, but never validates this field against the current oracle time before transferring user margin. Regular order matching enforces expiration on-chain; isolated subaccount creation does not, creating an inconsistency that allows user margin to be locked into an isolated position after the user's intended time window.

---

### Finding Description

The `IEndpoint.Order` struct includes an `expiration` field that is part of the EIP-712 signed digest:

```solidity
struct Order {
    bytes32 sender;
    int128 priceX18;
    int128 amount;
    uint64 expiration;   // user-specified time bound
    uint64 nonce;
    uint128 appendix;
}
``` [1](#0-0) 

For regular order matching, `_validateOrder()` enforces expiration on-chain as a hard gate before any balance mutation:

```solidity
return
    ...
    !_expired(order.expiration);
``` [2](#0-1) 

where `_expired` compares against oracle time:

```solidity
function _expired(uint64 expiration) internal view returns (bool) {
    return expiration <= getOracleTime();
}
``` [3](#0-2) 

However, `createIsolatedSubaccount()` only validates the signature and then unconditionally transfers margin — there is no call to `_expired` or any equivalent check:

```solidity
function createIsolatedSubaccount(
    IEndpoint.CreateIsolatedSubaccount memory txn,
    address linkedSigner
) external onlyEndpoint returns (bytes32) {
    ...
    require(
        _checkSignature(txn.order.sender, digest, linkedSigner, txn.signature),
        ERR_INVALID_SIGNATURE
    );
    // ← no _expired(txn.order.expiration) check here
    ...
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
}
``` [4](#0-3) [5](#0-4) 

The `CreateIsolatedSubaccount` struct wraps an `Order` (with its `expiration` field) and is dispatched through `processTransactionImpl` → `createIsolatedSubaccount`: [6](#0-5) 

---

### Impact Explanation

**Medium**: When the sequencer processes a `CreateIsolatedSubaccount` transaction after the order's `expiration` (due to queue delay, network congestion, or high load — not requiring malicious intent), the user's margin `M` is transferred from their main subaccount to the newly created isolated subaccount. The user's main account balance is reduced by `M` outside their intended time window, locking collateral in an unintended isolated position. This is a concrete, irreversible asset delta: `spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin)` executes unconditionally.

---

### Likelihood Explanation

**Medium**: The sequencer processes transactions sequentially and may experience delays between submission and on-chain execution. A user who signs a time-sensitive isolated margin order (e.g., tied to a specific market window) with a short `expiration` can have that order executed after expiry without any on-chain rejection. This is a protocol-level inconsistency: the same `Order` struct's `expiration` is enforced for `matchOrders` but silently ignored for `createIsolatedSubaccount`. No sequencer compromise is required — normal processing delay is sufficient.

---

### Recommendation

Add an expiration check in `createIsolatedSubaccount()` immediately after the signature validation, consistent with `_validateOrder()`:

```solidity
require(!_expired(txn.order.expiration), ERR_ORDER_EXPIRED);
```

This should be placed after the `digestToSubaccount` early-return check (to avoid rejecting idempotent re-submissions of already-created subaccounts) but before any balance mutations.

---

### Proof of Concept

1. User signs a `CreateIsolatedSubaccount` order with `expiration = T` and `margin = M` (encoded in `appendix`).
2. The signed transaction is submitted to the sequencer's queue at time `T - ε`.
3. Due to queue backlog, the sequencer submits the transaction on-chain at time `T + δ` (after expiration).
4. `processTransactionImpl` dispatches to `createIsolatedSubaccount()`.
5. `_checkSignature` passes (signature is valid regardless of time).
6. No expiration check exists — execution continues.
7. `spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -M)` executes, debiting `M` from the user's main subaccount.
8. `spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, M)` executes, crediting the isolated subaccount.
9. User's main account is short `M` after their intended expiration window, with funds locked in an isolated position they no longer intended to open. [7](#0-6) [5](#0-4)
