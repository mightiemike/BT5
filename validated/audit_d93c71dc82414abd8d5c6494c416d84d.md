### Title
Insufficient Bonus Pool Permanently Blocks `MsgUpdateTier` BonusApy Governance Updates — (`x/tieredrewards/keeper/msg_server_auth.go`, `claim_rewards.go`, `bonus_rewards.go`)

### Summary

`UpdateTier` unconditionally calls `claimRewardsAndUpdateTierPositions` when `BonusApy` changes. That function iterates every position in the tier and calls `processEventsAndClaimBonus` for each. If the rewards pool cannot cover any single position's accrued bonus, the entire governance transaction fails atomically and the tier parameter is never updated. An unprivileged attacker who creates positions and drains the pool can permanently prevent governance from reducing `BonusApy`.

### Finding Description

In `msg_server_auth.go`, `UpdateTier` contains:

```go
if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
    if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
        return nil, err   // ← governance tx fails here
    }
}
``` [1](#0-0) 

`claimRewardsAndUpdateTierPositions` iterates all positions and calls `processEventsAndClaimBonus` for each:

```go
for _, id := range ids {
    ...
    if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
        return err   // ← aborts entire sweep on first failure
    }
    ...
}
``` [2](#0-1) 

`processEventsAndClaimBonus` checks pool sufficiency **per position** before paying:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
``` [3](#0-2) 

`sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` if the pool cannot cover that single position's bonus:

```go
if !poolBalance.IsAllGTE(bonus) {
    return errorsmod.Wrapf(types.ErrInsufficientBonusPool, ...)
}
``` [4](#0-3) 

Because Cosmos SDK transactions are atomic, a failure at position N causes the entire `MsgUpdateTier` to revert. No positions are settled, the tier is not updated, and the pool balance is unchanged. The next governance attempt faces the same state.

### Impact Explanation

An attacker who:
1. Creates many positions via permissionless `MsgLockTier`,
2. Lets bonus accrue over time, and
3. Claims their own rewards (draining the pool to below the aggregate accrued bonus)

can ensure that every `MsgUpdateTier` with a different `BonusApy` fails with `ErrInsufficientBonusPool`. Governance cannot reduce `BonusApy` to protect the pool. If governance refunds the pool to retry, the attacker claims again, re-draining it. The invariant that governance can always update tier parameters is violated. If the pool is periodically refunded (e.g., by protocol inflation or community funds), it will be continuously depleted at the high APY rate that governance cannot lower.

### Likelihood Explanation

- `MsgLockTier` is permissionless; any user can create positions.
- Claiming rewards is a normal user action with no restrictions.
- The attacker's locked tokens earn bonus rewards, so the attack has near-zero net cost.
- The attack requires no privileged access, no front-running in the traditional sense (the pool just needs to be insufficient when the governance proposal executes, which the attacker can ensure at any time during the voting period).
- The pool address is public and queryable; the attacker can monitor it and claim rewards to keep it below the threshold.

### Recommendation

Decouple the reward settlement from the tier parameter update. Options:

1. **Skip positions with insufficient pool**: In `claimRewardsAndUpdateTierPositions`, catch `ErrInsufficientBonusPool` per position and skip (or partially settle) rather than aborting the entire sweep.
2. **Aggregate check before iteration**: Compute total accrued bonus for all positions first; if the pool is insufficient, either skip bonus settlement entirely or settle proportionally, then proceed with the tier update.
3. **Separate settlement from governance**: Remove the forced settlement from `UpdateTier` and instead record a "rate change checkpoint" that each position settles lazily on its next individual claim.

### Proof of Concept

```
1. Attacker calls MsgLockTier N times → N positions in tier 1 with BonusApy = 10%
2. Wait T seconds so each position accrues bonus B_i
3. Attacker calls MsgClaimTierRewards for N-1 positions → pool drained to < B_N
4. Governance submits MsgUpdateTier{tier: {Id:1, BonusApy: 5%}}
5. claimRewardsAndUpdateTierPositions iterates positions:
   - Positions 1..N-1: already claimed (zero bonus), pass
   - Position N: bonus = B_N, pool < B_N → ErrInsufficientBonusPool
6. MsgUpdateTier returns error; tier BonusApy remains 10%
7. Repeat step 3 after any pool refund → governance permanently blocked
```

The exact failure is `ErrInsufficientBonusPool` (error code 12) propagating from `bonus_rewards.go:56` through `claim_rewards.go:71` to `msg_server_auth.go:68-69`. [5](#0-4)
