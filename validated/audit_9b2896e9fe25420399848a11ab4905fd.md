### Title
SwapAllowlistExtension gates the router address instead of the actual user, allowing complete per-user allowlist bypass through MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is `msg.sender` at the pool level. When `MetricOmmSimpleRouter` intermediates a swap, the pool's `msg.sender` is the **router contract**, not the actual end-user. A pool admin who allowlists the router address (the natural action to enable router-mediated swaps) inadvertently grants every user on-chain the ability to bypass the per-user swap allowlist entirely.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
// SwapAllowlistExtension.sol line 37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension is called by the pool via `ExtensionCalling._callExtensionsInOrder`), and `sender` is the first argument the pool forwards — the direct caller of `pool.swap(...)`.

`ExtensionCalling._beforeSwap` passes `sender` verbatim to every extension:

```solidity
// ExtensionCalling.sol lines 149-177
function _beforeSwap(address sender, address recipient, ...) internal {
    _callExtensionsInOrder(
        BEFORE_SWAP_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
    );
}
```

When `MetricOmmSimpleRouter` executes a swap on behalf of a user, it calls `pool.swap(...)` directly. At that point `msg.sender` at the pool level is the **router**, so `sender` forwarded to the extension is the router's address, not the end-user's address.

The allowlist lookup therefore becomes:

```
allowedSwapper[pool][router]   // checked
allowedSwapper[pool][user]     // never checked
```

A pool admin who wants to enable router-mediated swaps for their curated pool must allowlist the router. Once `allowedSwapper[pool][router] = true`, the guard passes for **every** user who routes through `MetricOmmSimpleRouter`, regardless of whether that user is individually allowlisted.

The `DepositAllowlistExtension` does not share this flaw: it checks `owner` (the position owner explicitly supplied by the caller), which correctly identifies the economic beneficiary even when an operator/router pays.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties, institutional LPs, or whitelisted protocols) loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The attacker:

1. Calls `router.exactInputSingle(...)` or `router.exactInput(...)` targeting the curated pool.
2. The router calls `pool.swap(...)` with itself as `msg.sender`.
3. The extension checks `allowedSwapper[pool][router]`; if the router is allowlisted, the check passes.
4. The swap executes against LP assets that were deposited under the assumption that only curated counterparties could trade.

This constitutes a broken core pool functionality (curation invariant violated) and a direct loss of LP assets to uncurated counterparties — matching the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" and "Broken core pool functionality causing loss of funds" impact categories.

---

### Likelihood Explanation

The trigger is a **valid, semi-trusted action**: a pool admin allowlisting the router so that their curated pool is accessible via the standard periphery. This is the expected operational step for any pool that wants to support the router UX. The admin has no on-chain signal that doing so opens the allowlist to all users. Any unprivileged user can then exploit it by simply routing through `MetricOmmSimpleRouter`.

---

### Recommendation

The extension must receive and check the **economic actor** (the end-user), not the intermediary. Two complementary fixes:

1. **Pass the originating user through the router**: The router should accept an explicit `swapper` parameter and forward it to the pool as a separate field, or the pool's `swap` interface should accept a `payer` address distinct from `msg.sender` (mirroring the `owner`/`sender` split already present in `addLiquidity`).

2. **Extension-side fix**: `SwapAllowlistExtension` could check `recipient` instead of `sender` when the intent is to gate who benefits from the swap, or the pool could pass an additional `originator` field through the extension hook so the allowlist can key on the true user.

Until the interface is extended, document clearly that allowlisting the router is equivalent to `setAllowAllSwappers(pool, true)`.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice]  = true   // only alice should trade
  allowedSwapper[pool][router] = true   // admin enables router UX

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, tokenIn: weth, tokenOut: token1, ...})

  router calls:
    pool.swap(recipient=bob, ...)   // msg.sender = router

  pool calls extension:
    extension.beforeSwap(sender=router, ...)

  extension checks:
    allowedSwapper[pool][router] == true  → passes

  Result: bob's swap executes against LP assets despite not being allowlisted.
  The per-user curation invariant is broken; any user can trade via the router.
