### Title
Fork Difficulty Retarget Uses Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

### Summary

When validating a fork block that falls on a difficulty-retarget boundary, the contract fetches the retarget interval's first block via `get_header_by_height`, which unconditionally returns the **mainchain** block at that height. If the fork diverged before that height, the mainchain block and the fork's actual ancestor at that height are different blocks with different timestamps. The difficulty calculation therefore uses a stale, wrong timestamp, producing an incorrect `expected_bits` value. The contract then enforces this wrong difficulty against the submitted fork block, accepting blocks that are invalid under the real protocol or rejecting blocks that are valid.

### Finding Description

**Root cause — `get_header_by_height` always reads from the mainchain index**

`BlocksGetter::get_header_by_height` is implemented in `lib.rs`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // mainchain-only index
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

It reads from `mainchain_height_to_header`, which only contains blocks on the current main chain. Fork blocks are stored only in `headers_pool` (via `store_fork_header`), never in the height index.

**Where the wrong block is used — all three chain modules**

*Bitcoin* (`contract/src/bitcoin.rs`, line 81):
```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header,
    interval_tail_extend_header.block_header.time.into())
```

*Litecoin* (`contract/src/litecoin.rs`, line 88):
```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header,
    interval_tail_extend_header.block_header.time.into())
```

*Dogecoin* (`contract/src/dogecoin.rs`, lines 292–295, with an explicit TODO acknowledging the problem):
```rust
// TODO: check if it is correct to get block header by height from mainchain
//       without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header.time;
```

**The desynchronization**

The difficulty retarget formula is:
```
new_target = prev_target × (T_last − T_first) / pow_target_timespan
```

`T_last` comes from the fork's block at height `N−1` (correct — it is fetched via `get_prev_header`, which walks the `headers_pool` chain). `T_first` comes from `get_header_by_height(N−2015)`, which returns the **mainchain** block at that height, not the fork's ancestor.

When the fork diverged before height `N−2015`, the fork's block at that height has a different timestamp than the mainchain's block. The two independently validated pieces of state — the mainchain's retarget-boundary block and the fork's tip block — are used together in the retarget calculation without verifying that the mainchain block is actually an ancestor of the fork tip. This is the same class of desynchronization as M-01: a constraint is validated at submission time for each piece of state independently, but the two pieces are later combined in a computation that silently assumes they are consistent.

**Exploit path**

1. The mainchain has progressed past two retarget boundaries (e.g., heights 0–4031 for Bitcoin).
2. An unprivileged caller submits a fork starting at height `K < 2016` (before the first retarget boundary's first block at height 2016).
3. The fork's block at height 2016 carries timestamp `T_fork_2016`; the mainchain's block at height 2016 carries `T_main_2016 ≠ T_fork_2016`.
4. The caller submits fork blocks up to height 4031, then submits the fork block at height 4032 (second retarget boundary).
5. The contract computes `expected_bits` using `T_main_2016` (wrong) instead of `T_fork_2016` (correct).
6. The caller sets the fork block's `bits` field to match this wrong `expected_bits`.
7. The PoW check then enforces `hash ≤ target_from_bits(wrong_bits)` — a target that does not correspond to the correct Bitcoin protocol difficulty for this fork chain.

### Impact Explanation

**Case A — wrong `expected_bits` is easier than correct (mainchain first-block timestamp is earlier than fork's):**
The contract accepts a fork block whose `bits` encodes a lower difficulty than the protocol requires. The block would be rejected by any real Bitcoin node. The contract's `headers_pool` and potentially its main chain now contain a block that is invalid under the real protocol. Any downstream SPV proof (`verify_transaction_inclusion`) built on this fork chain returns results that do not correspond to real Bitcoin state.

**Case B — wrong `expected_bits` is harder than correct (mainchain first-block timestamp is later than fork's):**
A legitimately mined fork block — one that satisfies the correct Bitcoin protocol difficulty — is rejected by the contract because its `bits` does not match the inflated `expected_bits`. A genuine chain reorganization that should be tracked is silently dropped. The contract continues to report a stale main chain, causing `verify_transaction_inclusion` to return false negatives for transactions confirmed on the real chain.

Both cases corrupt the canonical-chain mapping and the SPV verification results that downstream NEAR contracts rely on.

### Likelihood Explanation

The condition is: a fork submission crosses a retarget boundary, and the fork diverged before the retarget interval's first block. This is a normal operational scenario — any fork that spans more than one difficulty epoch triggers it. No privileged access is required; `submit_blocks` is callable by any trusted relayer (a role reachable without leaked keys) and, during genesis initialization, by the deployer. The Dogecoin module's own TODO comment confirms the developers identified this as an open correctness question.

### Recommendation

Replace `get_header_by_height` calls inside `get_next_work_required` with an ancestor walk through `get_prev_header`, starting from the fork's `prev_block_header` and walking back `difficulty_adjustment_interval − 1` steps. This mirrors how Bitcoin Core actually computes the retarget first-block: by walking the chain backwards from the previous block, not by looking up a height in a global index. The corrected logic ensures the retarget calculation always uses the block that is actually an ancestor of the block being validated, regardless of whether it is on the main chain or a fork.

### Proof of Concept

```
Setup (Bitcoin feature):
  Genesis at height 0.
  Mainchain: heights 0..4031 submitted normally.
  mainchain block at height 2016 has time = T_main (e.g., 1_400_000_000).

Attack:
  Attacker submits fork blocks at heights 1..4031, diverging at height 1.
  Fork block at height 2016 has time = T_fork (e.g., 1_300_000_000, earlier than T_main).

  When attacker submits fork block at height 4032:
    first_block_height = 4031 − 2015 = 2016
    get_header_by_height(2016) → mainchain block, time = T_main = 1_400_000_000
    actual_time_taken = T_fork_4031 − T_main_2016
                      = T_fork_4031 − 1_400_000_000   (uses wrong timestamp)
    Correct value would be:
                      = T_fork_4031 − T_fork_2016
                      = T_fork_4031 − 1_300_000_000   (100_000_000 seconds larger)

  The wrong actual_time_taken is 100_000_000 s smaller than correct.
  new_target = prev_target × wrong_actual / timespan
             < prev_target × correct_actual / timespan

  The contract enforces a harder target than the protocol requires.
  A legitimately mined fork block at height 4032 (valid on real Bitcoin) is rejected.
  The contract fails to process a real chain reorganization.
```
