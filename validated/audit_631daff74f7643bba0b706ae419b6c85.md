### Title
Mandatory `claimRewards` in all exit paths permanently locks user funds when the bonus pool is depleted — (File: `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

Every position exit path in the `x/tieredrewards` module unconditionally calls `claimRewards`, which internally calls `processEventsAndClaimBonus`. That function checks `sufficientBonusPoolBalance` before disbursing accrued bonus rewards. If the `RewardsPoolName` module account has insufficient balance, the check returns an error that propagates upward and reverts the entire exit transaction. Because no exit path bypasses this check, any user who has accrued bonus rewards is permanently unable to retrieve their staked principal when the bonus pool is depleted.

---

### Finding Description

**Mandatory reward claim in every exit path**

`TierUndelegate` calls `claimRewards` before initiating the unbonding: [1](#0-0) 

`ExitTierWithDelegation` calls `claimRewards` before transferring the delegation back to the owner: [2](#0-1) 

`ClearPosition` calls `claimRewards` before clearing the exit flag: [3](#0-2) 

**The blocking check inside `processEventsAndClaimBonus`**

After computing `totalBonus`, the function checks pool solvency and then sends from the module account: [4](#0-3) 

If `sufficientBonusPoolBalance` returns an error (pool balance < `bonusCoins`), the error propagates through `claimRewards` and causes the calling exit message handler to return an error, reverting the entire transaction.

**No bypass exists**

`WithdrawFromTier` does not call `claimRewards`, but it requires the position to already be undelegated: [5](#0-4) 

Reaching `WithdrawFromTier` requires a prior successful `TierUndelegate`, which itself requires `claimRewards` to succeed. The `ForceFullExitWithDelegation` migration helper also calls `claimRewards` unconditionally: [6](#0-5) 

There is no production message path that allows a user to exit a delegated position while skipping bonus reward disbursement.

---

### Impact Explanation

A user who has accrued any non-zero bonus rewards and whose position remains delegated cannot call `TierUndelegate` or `ExitTierWithDelegation` when the `RewardsPoolName` module account is empty. Their staked principal — locked into the position's delegator sub-account via `lockFunds` — is inaccessible. The delegation itself cannot be moved or unwound without first passing through `claimRewards`. The corrupted value is the user's staked `basecro` principal held in the position delegator account. [7](#0-6) 

---

### Likelihood Explanation

The bonus pool is a finite module account funded by inflation. Under normal operation it can be depleted if many users claim simultaneously or if the inflation rate is reduced. Once depleted, every user with a non-zero pending bonus is blocked. The trigger is a standard, unprivileged `MsgTierUndelegate` or `MsgExitTierWithDelegation` transaction — no special role or key is required.

---

### Recommendation

Decouple reward disbursement from the exit path. Specifically:

1. In `TierUndelegate` and `ExitTierWithDelegation`, attempt to claim rewards but treat a pool-insufficient error as a non-fatal condition (skip bonus payout, record unclaimed bonus for later collection) rather than reverting the entire transaction.
2. Alternatively, add a dedicated emergency-exit message that skips bonus reward claiming entirely, allowing users to recover their principal at the cost of forfeiting unclaimed bonus.

---

### Proof of Concept

1. User calls `LockTier` → funds sent to position delegator account → delegated to validator.
2. Several blocks pass; `processEventsAndClaimBonus` accumulates a non-zero `totalBonus` for the position.
3. The `RewardsPoolName` module account balance reaches zero (through normal depletion by other claimants).
4. User calls `MsgTierUndelegate`:
   - `msg_server.go:166` → `claimRewards`
   - `claim_rewards.go:97` → `processEventsAndClaimBonus`
   - `claim_rewards.go:230` → `sufficientBonusPoolBalance` returns error
   - Transaction reverts; no unbonding is initiated.
5. User calls `MsgExitTierWithDelegation` → identical failure at `msg_server.go:532`.
6. User's staked principal remains locked in the position delegator account with no reachable exit. [8](#0-7)
