### Title
`op_bls_verify()` Succeeds With Zero (pk, msg) Pairs, Silently Bypassing Signature Verification — (File: `src/bls_ops.rs`)

---

### Summary

`op_bls_verify` accepts a call containing only the G2 identity point (compressed infinity) and no `(pk, msg)` pairs, and returns success (`0`/nil) without verifying any signature. This is the direct structural analog to the NFT bug: a function whose purpose is to perform a critical check silently succeeds when the loop over its inputs processes zero items, because no guard enforces a minimum of one item.

---

### Finding Description

In `src/bls_ops.rs`, `op_bls_verify` (lines 364–401) is structured as follows:

1. It reads the first argument as the G2 signature.
2. It iterates over the remaining arguments as `(G1 pk, msg)` pairs, accumulating them into `items`.
3. It calls `aggregate_verify(&signature, items)` and returns success if it returns `true`. [1](#0-0) 

When `items` is empty — i.e., no `(pk, msg)` pairs are provided — `aggregate_verify` is called with an empty vector. By the mathematical definition of BLS aggregate verification, an empty aggregate over the identity signature is valid: the product of zero pairings is the identity element in Gt, so `aggregate_verify(G2_identity, [])` returns `true`. The function then returns `Ok(Reduction(cost, a.nil()))` — success — without having verified any signature at all.

This behavior is confirmed by the official test vector in `op-tests/test-bls-ops.txt`:

```
; identity (no messages signed)
bls_verify 0xc000...0000 => 0 | 3000000
``` [2](#0-1) 

The G2 compressed infinity point (`0xc000...0000`, 96 bytes) is a valid atom accepted by `a.g2()`. After reading it as the signature, `rest(a, args)?` yields nil, the `while !nilp(a, args)` loop body is never entered, `items` remains empty, and `aggregate_verify` returns `true`.

The structural parallel to the NFT bug is exact:

| NFT bug (`_transferNFTs`) | clvm_rs analog (`op_bls_verify`) |
|---|---|
| Checks `supportsInterface(ERC721)` and `supportsInterface(ERC1155)` | Iterates over `(pk, msg)` pairs |
| If neither matches, loop body never executes | If no pairs provided, loop body never executes |
| Falls through without reverting | Falls through to `aggregate_verify(sig, [])` → `true` |
| Seller paid, no NFT transferred | Verification "passes", no signature checked |

### Impact Explanation

Any CLVM puzzle that uses `bls_verify` to authenticate a spend and allows the caller (solution provider) to control the full argument list passed to `bls_verify` is bypassable. An attacker provides:

- Argument 1 (signature): the 96-byte G2 identity point `0xc000...0000`
- Remaining arguments: nil (empty list)

`op_bls_verify` returns success. The puzzle's authentication gate is bypassed without possessing any private key.

A concrete vulnerable puzzle pattern:

```clvm
; puzzle: authenticate spend via bls_verify over caller-supplied args
(mod solution
  (bls_verify (f solution) (r solution))
)
```

The attacker passes `solution = (G2_identity . ())`. `(f solution)` = G2 identity, `(r solution)` = nil → zero pairs → `aggregate_verify` returns `true` → coin is spent without a valid signature.

The corrupted result is: `op_bls_verify` returns `Ok(Reduction(3000000, NIL))` — indistinguishable from a successful real verification — when no cryptographic work was performed. [3](#0-2) 

---

### Likelihood Explanation

The attack requires a puzzle that passes attacker-controlled data as the full argument list to `bls_verify`. Puzzles that hardcode the public key and message (e.g., `(bls_verify (f solution) HARDCODED_PK HARDCODED_MSG)`) are not vulnerable because the attacker cannot remove the pk/msg pairs. However, puzzles that forward solution-derived argument lists to `bls_verify` — a natural pattern for multi-key or flexible authentication schemes — are directly exploitable. The attacker-controlled entry path is the CLVM solution bytes, which are fully attacker-supplied on every spend attempt.

---

### Recommendation

Reject calls to `op_bls_verify` where the `(pk, msg)` pair list is empty. After the signature is read and before calling `aggregate_verify`, check that `items` is non-empty:

```rust
if items.is_empty() {
    return Err(EvalErr::BLSVerifyFailed(input));
}
```

This mirrors the recommended fix in the NFT report: "revert if neither the ERC-721 nor the ERC-1155 interface is supported." The analogous fix here is: revert (error) if no `(pk, msg)` pairs were provided, because no verification was performed. [4](#0-3)

```rust
pub fn op_bls_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = BLS_PAIRING_BASE_COST;
    check_cost(cost, max_cost)?;

    let mut args = input;

    // the first argument is the signature
    let signature = a.g2(first(a, args)?)?;

    // followed by a variable number of (G1, msg)-pairs (as a flat list)
    args = rest(a, args)?;

    let mut items = Vec::<(PublicKey, Atom)>::new();
    while !nilp(a, args) {
        let pk = a.g1(first(a, args)?)?;
        args = rest(a, args)?;
        let msg = atom(a, first(a, args)?, "bls_verify message")?;
        args = rest(a, args)?;

        cost += BLS_PAIRING_COST_PER_ARG;
        cost += msg.as_ref().len() as Cost * BLS_MAP_TO_G2_COST_PER_BYTE;
        cost += DST_G2.len() as Cost * BLS_MAP_TO_G2_COST_PER_DST_BYTE;
        check_cost(cost, max_cost)?;

        items.push((pk, msg));
    }

    if !aggregate_verify(&signature, items) {
        Err(EvalErr::BLSVerifyFailed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
}
```
