### Title
Deprecated `verify_transaction_inclusion` (v1) Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Guard Enforced Only in v2 - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (v1) is still a live, unrestricted public entry point on the contract. The coinbase Merkle proof check — introduced in `verify_transaction_inclusion_v2` specifically to block the 64-byte transaction forgery attack — is only enforced when a caller chooses to use v2. Any unprivileged NEAR caller can invoke v1 directly, bypassing that guard entirely and obtaining a `true` proof result for a transaction that does not exist on-chain.

### Finding Description

`verify_transaction_inclusion_v2` was introduced to close the 64-byte Merkle-node forgery vulnerability documented at https://www.bitmex.com/blog/64-Byte-Transactions. It does so by requiring the caller to also supply a coinbase Merkle proof, which anchors the tree structure and prevents an internal node from being passed off as a leaf transaction hash. [1](#0-0) 

The critical guard is the `require!` at lines 358–365 that verifies the coinbase proof against the block's stored `merkle_root` before delegating to v1.

However, v1 is still decorated only with `#[deprecated]` and `#[pause]` — it carries no caller restriction and remains a fully reachable public method: [2](#0-1) 

v1's Merkle verification at lines 318–322 computes the root from the caller-supplied `tx_id` and `merkle_proof` with no coinbase anchor: [3](#0-2) 

The contract's own docstring acknowledges the consequence:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."

The structural parallel to M-19 is exact: the security check (coinbase proof) applies only when the caller voluntarily uses v2; callers who invoke v1 directly face no such restriction.

### Impact Explanation

An attacker crafts a 64-byte value that is a valid internal Merkle node of a real Bitcoin block already tracked by the contract. They supply a Merkle proof path from that internal node to the block's stored `merkle_root`. `verify_transaction_inclusion` returns `true`. Any dApp or bridge contract that calls this function to gate asset releases, cross-chain settlements, or other value-bearing actions will accept the forged proof as genuine, authorizing an action for a Bitcoin transaction that never existed.

The corrupted state is the proof result itself: `true` is returned for a non-existent transaction, breaking the core security invariant of the light client — that `verify_transaction_inclusion == true` implies the transaction is actually included in a mainchain Bitcoin block.

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. Any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte forgery technique is publicly documented and has known tooling. The only prerequisite is that the attacker knows a real block hash already stored in the contract's `headers_pool` (trivially obtained from `get_last_block_header` or `get_block_hash_by_height`). Likelihood is high wherever a downstream consumer relies on v1.

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` (v1) or gate it with an access-control role so it cannot be called by external accounts. Alternatively, make v1 a private internal function callable only from v2, which already enforces the coinbase proof check. The deprecation notice alone does not prevent external callers from invoking it.

### Proof of Concept

1. Identify a real block hash stored in the contract: call `get_last_block_header()` → obtain `block_hash` and `merkle_root`.
2. From the corresponding Bitcoin block, select any two adjacent leaf hashes `L` and `R`. Concatenate them to form a 64-byte value `N = L || R`. This `N` is an internal Merkle node and also a valid SHA256d preimage.
3. Compute the Merkle proof path from `N` up to `merkle_root` (one level shorter than a leaf proof).
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node, not a real transaction)
   - `tx_block_blockhash = block_hash`
   - `tx_index` = index corresponding to `N`'s position in the tree
   - `merkle_proof` = the path from `N` to `merkle_root`
   - `confirmations = 0`
5. The function returns `true` because `compute_root_from_merkle_proof(N, index, proof) == merkle_root` holds by construction, yet no transaction with hash `N` exists in the Bitcoin block. [4](#0-3) [5](#0-4) [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {

            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"

        let header = self
            .headers_pool

            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );
