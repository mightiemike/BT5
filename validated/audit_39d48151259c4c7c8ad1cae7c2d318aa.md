Audit Report

## Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any unprivileged actor to bypass the deposit allowlist - (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter and checks `owner` instead, despite the contract's documented intent to gate by depositor address. Any unprivileged actor can bypass the allowlist by calling `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with an allowlisted address as `owner`, causing their tokens to be pulled and permanently credited to the allowlisted owner's position, which only the allowlisted owner can withdraw.

## Finding Description

`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`, passing `sender = msg.sender` (the actual caller, e.g., `MetricOmmPoolLiquidityAdder`) and `owner` as the position recipient: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as unnamed/discarded and checks only `owner`: [2](#0-1) 

This contradicts the NatSpec and storage variable name, both of which declare the intent is to gate by **depositor** address: [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a caller-supplied `owner` and stores `msg.sender` as the `payer` in transient context: [4](#0-3) 

In the callback, tokens are pulled from `payer` (the actual caller, BOB), not from `owner`: [5](#0-4) 

`removeLiquidity` enforces `msg.sender == owner`, so BOB cannot recover the deposited tokens: [6](#0-5) 

The existing guard (`allowedDepositor[msg.sender][owner]`) is insufficient because it checks the wrong actor — it validates the position recipient (ALICE) rather than the actual token payer (BOB). No other guard in the hook path checks `sender`.

## Impact Explanation

Direct, irreversible loss of user principal. BOB's tokens are permanently locked in ALICE's position; BOB has no path to recover them since `removeLiquidity` requires `msg.sender == owner`. The deposit allowlist — intended for regulatory compliance (KYC gating) or access control — is completely negated: any unprivileged actor can deposit into a restricted pool. This meets the Critical/High threshold for direct loss of user principal and broken core pool access-control functionality.

## Likelihood Explanation

Exploitation requires no privileged access, flash loan, or special setup. The only precondition is knowledge of one allowlisted address, which is publicly readable from `allowedDepositor`. Any actor can call the deployed `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` against any pool using `DepositAllowlistExtension` with `allowAllDepositors = false`. The attack is repeatable and permissionless.

## Recommendation

Replace the `owner` check with a `sender` check in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns the implementation with the documented intent ("Gates `addLiquidity` by depositor address") and the storage variable name `allowedDepositor`.

## Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; `allowAllDepositors[pool] = false`; only `ALICE` is in `allowedDepositor[pool]`.
2. Unauthorized `BOB` calls:
   ```solidity
   liquidityAdder.addLiquidityExactShares(
       pool,
       ALICE,   // owner = allowlisted address
       salt,
       deltas,
       maxAmount0,
       maxAmount1,
       extensionData
   );
   ```
3. `LiquidityAdder` calls `pool.addLiquidity(ALICE, salt, deltas, abi.encode(KIND_PAY), extensionData)`.
4. Pool calls `_beforeAddLiquidity(LiquidityAdder, ALICE, ...)`.
5. Extension evaluates `allowedDepositor[pool][ALICE]` → `true` → no revert.
6. Liquidity is added; callback pulls tokens from BOB (stored as `payer` in transient context).
7. BOB's tokens are credited to ALICE's position. BOB cannot call `removeLiquidity` (not the owner). BOB's principal is permanently lost.
