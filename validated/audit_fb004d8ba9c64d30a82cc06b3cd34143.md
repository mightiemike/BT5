### Title
Re-Execution of `_tryCloseIsolatedSubaccount` on Already-Closed Subaccount Sweeps Quote Balance to FEES_ACCOUNT — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`_tryCloseIsolatedSubaccount` does not guard against being called a second time on an already-closed isolated subaccount. After first closure, `parentSubaccounts[subaccount]` is set to `bytes32(0)`, which is the protocol's `FEES_ACCOUNT`. If the subaccount subsequently holds a non-zero spot quote balance (e.g., from a deposit), a second invocation sweeps that balance to `FEES_ACCOUNT` instead of the legitimate parent, permanently confiscating user collateral.

---

### Finding Description

**Step 1 — Confirmed: `FEES_ACCOUNT == bytes32(0)`** [1](#0-0) 

**Step 2 — Confirmed: First closure sets `parentSubaccounts[subaccount] = bytes32(0)`** [2](#0-1) 

**Step 3 — Confirmed: `getIsolatedProductId` is a pure function reading from the subaccount's bytes32 encoding, not from storage** [3](#0-2) 

This means after closure, `getIsolatedProductId(subaccount)` still returns a non-zero `productId`, so the early-return guard at line 162 does **not** fire on a second call.

**Step 4 — Confirmed: No guard against re-execution**

The function reads `parent = parentSubaccounts[subaccount]` after closure, which is `bytes32(0)`: [4](#0-3) 

Then unconditionally uses `parent` as the destination for the quote balance sweep: [5](#0-4) 

On a second call, `parent == bytes32(0) == FEES_ACCOUNT`, so `spotEngine.updateBalance(QUOTE_PRODUCT_ID, FEES_ACCOUNT, quoteBalance)` is executed, permanently transferring the user's collateral to the protocol fees account.

**Step 5 — Access path**

The external entry point is gated: [6](#0-5) 

However, the endpoint processes user-submitted transactions (slow-mode and sequencer paths), and `tryCloseIsolatedSubaccount` is also called from `Clearinghouse.sol` and `ClearinghouseLiq.sol` (2 call sites each) and `EndpointTx.sol` (1 call site). If any of these paths can be triggered for an already-closed subaccount — e.g., via a slow-mode transaction type that accepts an arbitrary `subaccount` argument, or via a liquidation path that processes the closed subaccount — the second invocation executes without revert.

---

### Impact Explanation

Any spot quote balance held by a closed isolated subaccount (e.g., from a deposit made after closure, or from residual balance not fully swept on first close) is permanently transferred to `FEES_ACCOUNT`. The user's parent subaccount receives nothing. This is a direct, irreversible loss of user collateral matching the Critical scope: *unauthorized mutation of another user's subaccount balances*.

---

### Likelihood Explanation

The precondition requires a non-zero quote balance on a closed isolated subaccount and a second triggering of `_tryCloseIsolatedSubaccount`. Whether deposits to closed isolated subaccounts are blocked by a separate guard (not found in the searched code) is the main uncertainty. If no such guard exists, the attacker can deposit quote to the closed subaccount and then trigger closure again via any supported endpoint path. The absence of a `require(parentSubaccounts[subaccount] != bytes32(0))` guard in `_tryCloseIsolatedSubaccount` itself is the root cause.

---

### Recommendation

Add an explicit guard at the top of `_tryCloseIsolatedSubaccount` to abort if the subaccount has already been closed:

```solidity
function _tryCloseIsolatedSubaccount(bytes32 subaccount) internal {
    uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
    if (productId == 0) return;

    // ADD THIS GUARD:
    require(
        parentSubaccounts[subaccount] != bytes32(0),
        "subaccount already closed"
    );
    // ... rest of function
}
```

Additionally, block deposits to isolated subaccounts whose `parentSubaccounts` mapping is `bytes32(0)`.

```text
    function _tryCloseIsolatedSubaccount(bytes32 subaccount) internal {
        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
        if (productId == 0) {
            return;
        }
        IPerpEngine.Balance memory balance = perpEngine.getBalance(
            productId,
            subaccount
        );
        if (balance.amount == 0) {
            uint8 id = RiskHelper.getIsolatedId(subaccount);
            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(
                    productId,
                    subaccount,
                    0,
                    -balance.vQuoteBalance
                );
                perpEngine.updateBalance(
                    productId,
                    parent,
                    0,
                    balance.vQuoteBalance
                );
            }
            int128 quoteBalance = spotEngine
                .getBalance(QUOTE_PRODUCT_ID, subaccount)
                .amount;
            if (quoteBalance != 0) {
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    subaccount,
                    -quoteBalance
                );
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    parent,
                    quoteBalance
                );
            }
            isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
            isolatedSubaccounts[parent][id] = bytes32(0);
            parentSubaccounts[subaccount] = bytes32(0);

            emit CloseIsolatedSubaccount(subaccount, parent);
        }
    }
```
