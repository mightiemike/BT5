### Title
Unchecked ERC20 `transferFrom` Return Value Enables Theft of usdcE from Direct Deposit Accounts - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value and without using `safeTransferFrom`. Because the function is callable by any unprivileged user on chain 57073, a caller who provides no USDC (or has no allowance set) can silently skip the inbound USDC transfer while still receiving the full usdcE balance extracted from the victim's Direct Deposit Account.

---

### Finding Description

`replaceUsdcEWithUsdc` is an externally callable function with no access control beyond a chain ID check: [1](#0-0) 

The critical line is: [2](#0-1) 

`IERC20Base(usdc).transferFrom(...)` returns a `bool` that is never inspected. If the USDC token on Ink chain returns `false` on failure (e.g., insufficient balance or zero allowance) rather than reverting, execution continues to:

1. `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` — pulls the entire usdcE balance from the DDA into `ContractOwner` (permitted because `ContractOwner` is the `Ownable` owner of every DDA it deploys). [3](#0-2) 

2. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — forwards the extracted usdcE to the attacker. [4](#0-3) 

Contrast this with every other transfer in the codebase, which consistently uses the `ERC20Helper.safeTransferFrom` wrapper that checks both the low-level call success and the decoded boolean return: [5](#0-4) 

The `using ERC20Helper for IERC20Base` directive is active in `ContractOwner`, making the safe variant directly available — yet the raw `transferFrom` is used instead. [6](#0-5) 

---

### Impact Explanation

An attacker can drain the full usdcE balance held in any DDA on chain 57073 without supplying any USDC. The stolen usdcE represents real collateral belonging to the subaccount owner. Because `directDepositV1Address` is a public mapping, every DDA address is enumerable on-chain, making all usdcE-holding DDAs targetable in a single block.

---

### Likelihood Explanation

The function is permissionless (any EOA or contract on chain 57073 can call it). The only precondition is that at least one DDA holds a non-zero usdcE balance. The attacker needs zero USDC and zero allowance — the only cost is gas. Likelihood is high whenever the migration window is open and DDAs hold usdcE.

---

### Recommendation

Replace the raw `transferFrom` call with the `safeTransferFrom` helper already imported and in scope:

```solidity
// Before (vulnerable)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe)
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

This mirrors the pattern used everywhere else in the codebase, including `EndpointStorage.chargeSlowModeFee` and `BaseWithdrawPool.safeTransferFrom`. [7](#0-6) 

---

### Proof of Concept

1. A subaccount `S` has a DDA at address `D` holding `N` usdcE (e.g., from a prior deposit before the migration).
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(S)` with zero USDC balance and zero USDC allowance.
3. `balance = IERC20Base(usdcE).balanceOf(D)` → `N`.
4. `IERC20Base(usdc).transferFrom(attacker, D, N)` → returns `false` (no allowance/balance); return value ignored; execution continues.
5. `DirectDepositV1(D).withdraw(usdcE)` → transfers `N` usdcE from `D` to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(attacker, N)` → transfers `N` usdcE to attacker.

Net result: attacker receives `N` usdcE; subaccount `S` loses its usdcE collateral; no USDC was ever deposited into `D`. [1](#0-0)

```text
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
