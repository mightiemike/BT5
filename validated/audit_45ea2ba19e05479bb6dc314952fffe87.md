Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of end user, allowing any caller to bypass swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`. When a swap originates from `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end user. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`, which resolves to the router address — so any user who calls the router bypasses the per-user allowlist entirely.

## Finding Description
**Step 1 — Pool passes `msg.sender` to the extension.**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at L230–231. When the call originates from `MetricOmmSimpleRouter`, `msg.sender` is the router contract address. [1](#0-0) 

**Step 2 — Extension checks the router address, not the end user.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool (correct) and `sender` is the router (wrong actor). The end user's address is never examined. [2](#0-1) 

**Step 3 — Router forwards no user identity.**

`exactInputSingle`, `exactInput`, and `exactOutputSingle` all call `pool.swap(...)` directly. The `extensionData` field is passed through from the caller's params but is not populated with the original caller's address by the router, and the extension does not decode it. [3](#0-2) 

**Root cause:** The allowlist maps `pool → swapper → bool`, where `swapper` is expected to be the end user. But the pool always supplies `msg.sender` (the immediate caller) as `sender`, so when an intermediary router is in the call stack, the extension sees the router, not the user. [4](#0-3) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (KYC'd users, whitelisted market makers, etc.) provides zero enforcement against any user who routes through `MetricOmmSimpleRouter`. The pool admin faces an impossible choice: allowlist the router (enabling all users to bypass the gate) or block the router (preventing all router-mediated swaps, including those from intended users). The access-control invariant the extension is designed to enforce is completely broken for the primary user-facing swap path. This constitutes a broken core pool functionality / admin-boundary bypass impacting any pool relying on the extension for access control.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. No special privilege, capital, or timing is required — any user can call `exactInputSingle` on the public router. The bypass is deterministic and repeatable. The pool admin has no on-chain mechanism to close the gap while keeping the router functional. [5](#0-4) 

## Recommendation
The extension must gate the economic actor, not the intermediary. Two viable approaches:

1. **`extensionData` convention**: Have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData` before calling `pool.swap`, and have `SwapAllowlistExtension.beforeSwap` decode and verify it when `sender` is a known router address.
2. **Pool-level original-sender field**: Extend the pool's swap interface to carry the original initiator separately, so extensions always receive the true economic actor regardless of routing depth.

Until fixed, pools requiring strict swap access control must not allowlist the router address and must require users to call `MetricOmmPool.swap` directly.

## Proof of Concept
```solidity
// Pool configured with SwapAllowlistExtension; admin allows only alice.
swapExtension.setAllowedToSwap(address(pool), alice, true);

// Bob (not allowlisted) calls pool directly — correctly reverts.
vm.prank(bob);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(bob, true, 1000, 0, "", "");

// Admin must allowlist the router for alice to use it.
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Bob routes through the router — extension sees sender=router (allowlisted), passes.
vm.prank(bob);
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool), recipient: bob, tokenIn: token0,
    zeroForOne: true, amountIn: 1000, amountOutMinimum: 0,
    priceLimitX64: 0, deadline: block.timestamp, extensionData: ""
}));
// Bob receives output tokens despite not being on the allowlist.
```

The check `allowedSwapper[pool][router]` evaluates to `true`, so the revert is never triggered for any caller using the router. [6](#0-5)
