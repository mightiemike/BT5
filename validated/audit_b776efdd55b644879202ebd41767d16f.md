### Title
Gateway `StatefulValidator::validate()` Skips `perform_pre_validation_stage`, Allowing Transactions with Invalid `proof_facts` Content to Bypass Admission — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful validator calls `StatefulValidator::validate()` directly, which only runs the account's `__validate__` entry point. It does **not** call `StatefulValidator::perform_validations()`, which would first call `AccountTransaction::perform_pre_validation_stage()` — the function that contains `validate_proof_facts()`. As a result, the gateway never checks whether the `proof_facts` content (program hash, config hash, block number, block hash) is semantically valid for the current protocol version. The blockifier **does** run `validate_proof_facts()` during block execution. A transaction with a cryptographically valid proof but an unauthorized program hash (not in `allowed_virtual_os_program_hashes`) passes gateway admission and enters the mempool, then fails during block execution with a non-reverted `TransactionPreValidationError::InvalidProofFacts`. Because the failure rolls back the nonce increment, the attacker can resubmit the same transaction indefinitely at no additional cost.

---

### Finding Description

**Gateway admission path** (`crates/apollo_gateway/src/stateful_transaction_validator.rs`, line 341):

```rust
let mut blockifier_validator = StatefulValidator::create(state, block_context);
blockifier_validator.validate(account_tx)   // ← calls only __validate__ entry point
```

`StatefulValidator::validate()` only runs `tx.validate_tx()` (the account's `__validate__` entry point). It does **not** call `perform_pre_validation_stage()`.

**Block execution path** (`crates/blockifier/src/blockifier/stateful_validator.rs`, lines 68–95):

```rust
pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
    match tx.tx {
        ApiTransaction::Invoke(_) => {
            let tx_context = Arc::new(...);
            tx.perform_pre_validation_stage(self.state(), &tx_context)?;  // ← validate_proof_facts here
            ...
            let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
            ...
        }
    }
}
```

`perform_pre_validation_stage()` (`crates/blockifier/src/transaction/account_transaction.rs`, lines 355–372) calls `validate_proof_facts()` (lines 291–351), which checks:

1. `allowed_proof_versions` — proof version must be in the versioned-constants list
2. `allowed_virtual_os_program_hashes` — program hash must be in the versioned-constants list
3. Block number not too recent (`current_block_number - STORED_BLOCK_HASH_BUFFER`)
4. Block hash matches the stored block hash in the block-hash contract
5. Config hash matches `virtual_os_config_hash()`

The gateway's cryptographic proof verifier (`starknet_proof_verifier::verify_proof`, `crates/starknet_proof_verifier/src/proof_verifier.rs`, lines 126–157) only checks that the proof is cryptographically valid for the given `proof_facts`. It reconstructs the output preimage from `proof_facts[2..]` (skipping the proof-version and variant markers) and verifies the V1 circuit. It does **not** check whether the program hash is in `allowed_virtual_os_program_hashes` or whether the config hash matches the node's current virtual OS config.

Therefore, an attacker can generate a valid V1 proof for a program hash that is **not** in `allowed_virtual_os_program_hashes` (e.g., a hash from a previous or unauthorized virtual OS build). `verify_proof` passes. `validate_proof_facts` is never called at the gateway. The transaction enters the mempool and is included in a block. During block execution, `validate_proof_facts` fails with `InvalidProofFacts("Virtual OS program hash X is not allowed")`. The failure is a hard pre-validation error; the transactional state rolls back the nonce increment. The attacker resubmits the same transaction indefinitely.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

A single proof generation (one-time cost) yields a transaction that:
- Always passes gateway admission (cryptographic proof is valid; `validate_proof_facts` is not called)
- Always fails during block execution (`validate_proof_facts` rejects the unauthorized program hash)
- Never permanently increments the nonce (pre-validation failure rolls back state)
- Can be resubmitted indefinitely at zero marginal cost

This allows an attacker to continuously fill the mempool and waste batcher execution cycles with transactions that will never succeed, degrading sequencer throughput.

---

### Likelihood Explanation

Generating a valid V1 proof for an unauthorized program hash requires running the proving circuit once — expensive but feasible for a motivated attacker. After that, the attack is free and perpetual. The structural gap (gateway calling `validate()` instead of `perform_validations()`) is unconditional and affects every Invoke V3 transaction carrying `proof_facts`.

---

### Recommendation

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, replace the direct call to `StatefulValidator::validate()` with `StatefulValidator::perform_validations()`:

```rust
// Before:
blockifier_validator.validate(account_tx)

// After:
blockifier_validator.perform_validations(account_tx)
```

`perform_validations()` already handles the Invoke case by calling `perform_pre_validation_stage()` first (which includes `validate_proof_facts()`), then calling `validate()` for the `__validate__` entry point. This mirrors the check ordering used during actual block execution and closes the admission gap.

---

### Proof of Concept

1. Obtain or generate a valid V1 SNOS proof for a virtual OS program whose hash is **not** in `allowed_virtual_os_program_hashes` for the current `VersionedConstants` (e.g., the hash from `blockifier_versioned_constants_0_14_2.json`: `0x3e98c2d7703b03a7edb73ed7f075f97f1dcbaa8f717cdf6e1a57bf058265473`, which is absent from `0_14_3.json`).

2. Construct an `RpcInvokeTransactionV3` with:
   - `proof_facts` containing the unauthorized program hash at index 2
   - `proof` containing the valid V1 proof bytes

3. Submit to the gateway's `add_tx` endpoint.

4. **Gateway result**: `verify_proof` passes (proof is cryptographically valid for the given `proof_facts`). `validate_proof_facts` is never called. Transaction is accepted and forwarded to the mempool.

5. **Block execution result**: `perform_pre_validation_stage` → `validate_proof_facts` → `"Virtual OS program hash 0x3e98... is not allowed"` → `TransactionPreValidationError::InvalidProofFacts`. Nonce rolled back.

6. Resubmit step 3 indefinitely. Each submission passes the gateway and fails during block execution at no additional cost.

```rust
    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
```

```rust
    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt())
        {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof version {} is not allowed under this protocol version.",
                snos_proof_facts.proof_version
            )));
        }

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let virtual_os_config_hash = block_context.virtual_os_config_hash();
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
```

```rust
        &self,
        state: &mut S,

        self.validate_proof_facts(&tx_context.block_context, state)?;
```

```rust
/// Reconstructs the output preimage from proof facts for circuit verification.
///
/// Proof facts layout: `[PROOF_VERSION_V*, variant, program_hash, ...task_output]`
/// Output preimage layout: `[num_tasks=1, output_size, program_hash, ...task_output]`
/// where `output_size = task_content.len() + 1` (includes itself).
pub fn reconstruct_output_preimage(
    proof_facts: &ProofFacts,
) -> Result<Vec<Felt>, VerifyProofError> {
    // Proof facts must contain at least [PROOF_VERSION_V*, variant, program_hash].
    if proof_facts.0.len() < 3 {
        return Err(VerifyProofError::ProofFactsTooShort { length: proof_facts.0.len() });
    }
    // Skip PROOF_VERSION_V* (index 0) and variant (index 1).
    let task_content = &proof_facts.0[2..];
    let output_size = Felt::from(
        u64::try_from(task_content.len() + 1).expect("task content length exceeds u64::MAX"),
    );
    Ok([Felt::ONE, output_size].into_iter().chain(task_content.iter().copied()).collect())
}
```
