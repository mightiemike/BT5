### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables USDC.e Drain Without USDC Payment — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the boolean return value. If the USDC token at the hardcoded Ink mainnet address returns `false` on failure rather than reverting, any caller can drain USDC.e from a victim's `DirectDepositV1` contract without providing any USDC in exchange.

---

### Finding Description

The function is designed as a one-for-one swap: the caller provides USDC and receives USDC.e held in a `DirectDepositV1` contract. The three-step sequence is:

1. Pull `balance` USDC from `msg.sender` into `directDepositV1` — **unchecked `transferFrom`**
2. Withdraw all USDC.e from `directDepositV1` into `ContractOwner` — `DirectDepositV1.withdraw`
3. Push USDC.e to `msg.sender` — `safeTransfer` (correctly checked)

The unchecked call is at line 616:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // return value discarded
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
``` [1](#0-0) 

The rest of the codebase uses `ERC20Helper.safeTransferFrom` for exactly this purpose — it low-level-calls `transferFrom` and requires both `success == true` and a truthy decoded return value:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [2](#0-1) 

`replaceUsdcEWithUsdc` bypasses this wrapper entirely and calls `.transfer`/`.transferFrom` directly on the raw interface, discarding the return value.

The function has **no access control** — it is `external` with only a `block.chainid == 57073` guard, making it callable by any address on Ink mainnet. [3](#0-2) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (Ink mainnet) returns `false` on a failed `transferFrom` rather than reverting:

- Step 1 silently fails — no USDC is deposited into `directDepositV1`.
- Step 2 still executes — USDC.e is withdrawn from `directDepositV1` into `ContractOwner`.
- Step 3 still executes — USDC.e is sent to the attacker.

The attacker receives USDC.e without paying USDC. The victim subaccount's `DirectDepositV1` is drained of its entire USDC.e balance. The corrupted asset delta is the full `balance` of USDC.e held in `directDepositV1` at the time of the call.

---

### Likelihood Explanation

The function is permissionlessly callable by any address on Ink mainnet. The exploitability condition is that the USDC token at the hardcoded address returns `false` on failure rather than reverting. Circle's canonical USDC implementation reverts, which would prevent exploitation. However, Ink is a relatively new chain and the token at that address may be a bridged or wrapped variant with non-standard failure semantics. The inconsistency within the same function — `safeTransfer` is used for the outbound USDC.e leg but raw `transferFrom` is used for the inbound USDC leg — indicates this was an oversight rather than an intentional design choice, and the pattern is concretely wrong regardless of current token behavior.

---

### Recommendation

Replace the raw `transferFrom` call with `ERC20Helper.safeTransferFrom`, consistent with the rest of the codebase:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [4](#0-3) 

---

### Proof of Concept

Precondition: USDC at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on a failed transfer (e.g., caller has zero USDC balance and zero allowance, and the token does not revert).

1. Victim subaccount `S` has a `DirectDepositV1` at `directDepositV1Address[S]` holding 1000 USDC.e.
2. Attacker (holding zero USDC, zero allowance) calls `ContractOwner.replaceUsdcEWithUsdc(S)`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, 1000e6)` returns `false` — silently ignored.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` transfers 1000 USDC.e to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, 1000e6)` sends 1000 USDC.e to the attacker.

Result: attacker gains 1000 USDC.e; victim's `DirectDepositV1` is emptied; no USDC was paid.

        require(block.chainid == 57073, ERR_UNAUTHORIZED);

        require(directDepositV1 != address(0), "no dda");

    }
