Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of end-user, making per-user swap gating unenforceable for router-mediated swaps - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` is designed to gate individual swappers on curated pools by checking `allowedSwapper[pool][sender]`. However, when swaps are routed through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router` and passes it as `sender` to `beforeSwap`. The extension therefore evaluates `allowedSwapper[pool][router]` — a single binary flag for the entire router — rather than `allowedSwapper[pool][actualUser]`, making per-user restrictions structurally unenforceable for the primary swap path.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` verbatim to the extension via `abi.encodeCall`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap(...)` directly with no forwarding of the original `msg.sender`: [4](#0-3) 

So `msg.sender` at the pool level is the router contract. The extension evaluates `allowedSwapper[pool][router]` — a single flag covering all users of the router — rather than `allowedSwapper[pool][user]`. No existing guard recovers the original end-user identity; the transient storage set by `_setNextCallbackContext` stores the payer for payment purposes only and is never consulted by the extension. [5](#0-4) 

## Impact Explanation
This creates two mutually exclusive failure modes for any pool using `SwapAllowlistExtension`:

1. **Router not allowlisted**: Individually allowlisted users cannot swap through the router at all, breaking the standard swap path for legitimate users.
2. **Router allowlisted**: Every user on the network can bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`, because the extension sees only the router's address and passes the check.

The exact wrong value is `allowedSwapper[pool][router]` being evaluated in place of `allowedSwapper[pool][user]`, causing the extension's access-control decision to be incorrect. This breaks the core pool functionality of curated/permissioned pools and allows unauthorized users to swap on pools explicitly configured to exclude them — a direct broken-functionality and unauthorized-access impact. [6](#0-5) 

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary public swap entrypoint for the protocol. Any pool that configures `SwapAllowlistExtension` and needs to support router-based swaps (the normal user flow) must allowlist the router, immediately opening the pool to all users. The trigger requires no special privileges — any user can call `exactInputSingle` on the router with any pool address. The condition is trivially reachable and repeatable. [7](#0-6) 

## Recommendation
The pool should propagate the original end-user identity through the swap path. Options:

1. Store the initiating user in transient storage at the router entry point and expose it via a dedicated interface that the extension can query (e.g., `ISwapInitiator(msg.sender).swapInitiator()`).
2. Accept an explicit `swapper` parameter in `pool.swap()` distinct from `msg.sender`, validated via a signed permit or callback, so the extension sees the real user.
3. As a simpler mitigation: document and enforce at the extension level that `SwapAllowlistExtension` is incompatible with router-mediated swaps by reverting when the caller (`msg.sender` at the extension, i.e., the pool's caller) is not itself the allowlisted address — effectively requiring direct pool interaction only. [8](#0-7) 

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps.
3. Pool admin calls `setAllowedToSwap(pool, alice, true)` intending to allow only Alice.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(params.recipient, ...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(router, ...)` — `sender = router`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Bob successfully swaps on a pool he was explicitly excluded from.

The check at line 37 passes because it sees the router (allowlisted) rather than Bob (not allowlisted): [3](#0-2)
