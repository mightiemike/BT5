### Title
Merkle Proof `tx_index` Malleability Allows Replay-Protection Bypass in `verify_transaction_inclusion` — (File: `merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` consumes only the low `n` bits of `transaction_position` (where `n` = proof length). Any two `tx_index` values that share the same low `n` bits — i.e., `tx_index` and `tx_index + k × 2ⁿ` for any integer `k ≥ 1` — produce an identical Merkle root with the same proof elements. Both calls to `verify_transaction_inclusion` (v1 and v2) therefore return `true` for the same underlying transaction with multiple distinct `tx_index` values. Any recipient contract that implements replay protection by storing a hash or key derived from the full `ProofArgs` tuple (including `tx_index`) can be bypassed by an unprivileged caller who re-submits the proof with a shifted index.

---

### Finding Description

In `merkle-tools/src/lib.rs`, `compute_root_from_merkle_proof` iterates over the proof elements and at each step uses only `current_position % 2` to decide left/right placement, then halves the position:

```rust
// merkle-tools/src/lib.rs  lines 39-51
let mut current_position = transaction_position;
for proof_hash in merkle_proof {
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
```

For a proof of length `n`, the function consumes exactly bits `0..n-1` of `transaction_position`. Bits `n` and above are never read. Therefore, for any valid `(tx_id, tx_block_blockhash, tx_index, merkle_proof)` that returns `true`, the tuple `(tx_id, tx_block_blockhash, tx_index + 2ⁿ, merkle_proof)` — and `tx_index + 2·2ⁿ`, `tx_index + 3·2ⁿ`, etc. — also returns `true` with no change to any proof element.

This is called by both public contract entry points:

- `verify_transaction_inclusion` (`contract/src/lib.rs`, line 318) calls `compute_root_from_merkle_proof` directly.
- `verify_transaction_inclusion_v2` (`contract/src/lib.rs`, line 368) calls `verify_transaction_inclusion` via `self.verify_transaction_inclusion(args.into())` after the coinbase check, inheriting the same malleability.

The `ProofArgs` struct (`btc-types/src/contract_args.rs`, lines 18–24) exposes `tx_index: u64` as a caller-controlled field with no upper-bound validation against the actual tree depth.

---

### Impact Explanation

`verify_transaction_inclusion` is the primary cross-chain verification API of this light client. Its intended consumers are bridge contracts and other NEAR contracts that verify Bitcoin transaction inclusion before releasing funds or minting tokens. The natural replay-protection pattern for such a consumer is to record a used-proof key derived from the full argument set — e.g., `hash(tx_id ‖ tx_block_blockhash ‖ tx_index ‖ merkle_proof)` or storing the `ProofArgs` struct directly.

Because `tx_index` is malleable, an attacker who has already consumed a valid proof `(tx_id, B, p, π)` can immediately construct a second valid proof `(tx_id, B, p + 2ⁿ, π)` (where `n = π.len()`). The light client returns `true` for both. The consumer's replay guard sees a new, never-seen key and processes the transaction a second time — enabling double-spend or double-mint against any bridge built on this API.

---

### Likelihood Explanation

The attack requires no privileged role, no private key, and no chain reorganization. Any unprivileged NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2`. The only prerequisite is possession of one valid proof, which is public on-chain data. Bitcoin Merkle trees for typical blocks have depth 10–13, so `2ⁿ` is at most 8192 — well within the `u64` range of `tx_index`. The malleability is deterministic and requires no brute force.

---

### Recommendation

Validate that `tx_index < 2ⁿ` (where `n = merkle_proof.len()`) before computing the root. This ensures the position is within the actual tree and eliminates all higher-bit aliases:

```rust
// In compute_root_from_merkle_proof or in verify_transaction_inclusion:
require!(
    transaction_position < (1usize << merkle_proof.len()),
    "tx_index exceeds tree size implied by proof length"
);
```

Additionally, document explicitly in the `verify_transaction_inclusion` NatSpec that callers must not use `tx_index` as part of a replay-protection key without first applying this bound check, mirroring the warning approach taken by the 1inch ECDSA library for compact-signature malleability.

---

### Proof of Concept

Given a real Bitcoin block with 4 transactions (depth = 2, `n = 2`):

- Valid proof: `tx_index = 0`, `merkle_proof = [sibling_0, parent_sibling]` → `compute_root_from_merkle_proof` returns the correct Merkle root → `verify_transaction_inclusion` returns `true`.
- Malleable proof: `tx_index = 4` (= 0 + 1·2²), same `merkle_proof` unchanged.
  - Step 0: `4 % 2 = 0` → same left/right as `0 % 2 = 0` ✓
  - Step 1: `4/2 = 2`, `2 % 2 = 0` → same left/right as `0/2 = 0`, `0 % 2 = 0` ✓
  - Root produced is identical → `verify_transaction_inclusion` returns `true`.

A bridge contract that stores `used[hash(tx_id, block_hash, 0, proof)] = true` after the first call will not block the second call with `tx_index = 4`, allowing the attacker to trigger the bridge action twice for the same Bitcoin transaction.
