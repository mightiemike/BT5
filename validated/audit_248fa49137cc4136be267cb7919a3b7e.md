### Title
SwapAllowlistExtension gates on router address instead of actual user, allowing any unprivileged user to bypass the swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is the `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool sees `msg.sender = router`, so the extension checks whether the **router** is allowlisted, not the actual user. If the pool admin allowlists the router to enable router-mediated swaps for permitted users, every non-allowlisted user can bypass the gate by routing through the same router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`, which is `msg.sender` of `pool.swap()`:

```solidity
_beforeSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    ...
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` as `msg.sender`:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [3](#0-2) 

So the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Attack path:**

1. Pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict swaps to KYC'd counterparties.
2. Admin allowlists specific users: `setAllowedToSwap(pool, userA, true)`.
3. Admin also allowlists the router so that permitted users can trade through the periphery: `setAllowedToSwap(pool, router, true)`.
4. Non-allowlisted `userB` calls `router.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap()` with `msg.sender = router`. The extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. `userB` successfully swaps in a pool they were never permitted to access.

The same bypass applies to `exactInput` (multi-hop) and `exactOutput` paths.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to enforce a curated or permissioned trading environment loses that guarantee entirely once the router is allowlisted. Any unprivileged user can execute swaps against the pool's liquidity, defeating the curation policy. Depending on the pool's purpose (e.g., institutional-only, regulatory-compliant, or strategically restricted), this constitutes a direct admin-boundary break: an unprivileged path bypasses a factory-configured access control that the pool admin explicitly set.

---

### Likelihood Explanation

The bypass requires the admin to allowlist the router. This is a natural operational step: a pool admin who wants permitted users to trade through the standard periphery must allowlist the router, because the extension sees `sender = router` for all router-mediated swaps. There is no other way to enable router access for allowed users without also enabling it for everyone. The bypass is therefore reachable in any production deployment that supports router-mediated swaps on an allowlisted pool.

---

### Recommendation

The extension must gate on the **actual user**, not the intermediary. Two approaches:

1. **Pass the original caller through the router**: The router stores the original `msg.sender` in transient storage (it already does this for the payer in `_setNextCallbackContext`). The pool or extension could read a `originalSender()` view on the router. However, this couples the extension to a specific router implementation.

2. **Check `sender` against the allowlist and reject the router address explicitly**: Require that `sender` is never a known router, forcing users to call the pool directly for allowlisted pools.

3. **Preferred — per-user allowlist enforced at the router level**: The router should forward the original `msg.sender` as part of `extensionData`, and the extension should decode and verify it. This preserves the operator pattern while correctly identifying the economic actor.

---

    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
