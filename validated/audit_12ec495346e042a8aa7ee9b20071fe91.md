### Title
Unvalidated Attacker-Controlled `tx_id` in `verify_transaction_inclusion` Enables Merkle Proof Forgery for Non-Existent Transactions — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is a publicly callable NEAR contract method that accepts a fully attacker-controlled `tx_id` and uses it directly in `compute_root_from_merkle_proof` without any validation that it represents a real leaf-level transaction hash. An unprivileged NEAR caller can supply an internal Merkle tree node as `tx_id` along with a crafted proof path, causing the function to return `true` for a transaction that does not exist in the block. Any downstream DApp consuming this result is deceived into accepting a forged inclusion proof.

---

### Finding Description

**Root cause — unvalidated attacker-controlled input used in a security-critical computation:**

`verify_transaction_inclusion` is still a live, publicly callable method on the contract: [1](#0-0) 

The function accepts `ProofArgs` entirely from the caller: [2](#0-1) 

The only guards applied before the cryptographic check are:

1. `args.confirmations <= self.gc_threshold`
2. The block hash must appear in `mainchain_header_to_height` (i.e., the block is on the canonical chain)
3. `!args.merkle_proof.is_empty()`

None of these guards validate that `args.tx_id` is a leaf-level transaction hash. The function then passes the raw, unvalidated `tx_id` directly into: [3](#0-2) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` treats whatever hash it receives as a leaf and walks up the tree using the supplied siblings: [4](#0-3) 

**Exploit mechanics (64-byte internal-node attack):**

Bitcoin's Merkle tree internal nodes are `SHA256d(left_child || right_child)` — a 64-byte input. An internal node `N` at depth `d` from the root has a valid proof path of length `d` (its `d` sibling hashes up to the root). An attacker who knows any real block on the canonical chain can:

1. Pick any internal Merkle node `N` at depth `d` from the root (e.g., the root's left child, depth = 1).
2. Compute the `d`-element sibling path from `N` to the root.
3. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index = position_of_N_at_depth_d`, `merkle_proof = [sibling_{d-1}, …, sibling_0]`, and a valid `tx_block_blockhash`.
4. `compute_root_from_merkle_proof(N, position, proof)` reproduces the stored `merkle_root` exactly.
5. The function returns `true`.

The code's own warning acknowledges this broken invariant: [5](#0-4) 

The `#[deprecated]` attribute in Rust emits only a compiler warning; it does not prevent runtime calls. The function remains fully reachable by any unprivileged NEAR account.

---

### Impact Explanation

Any external NEAR DApp that calls `verify_transaction_inclusion` (v1) to gate an asset release, state change, or authorization decision can be made to accept a forged proof for a transaction that never existed on Bitcoin. The attacker does not need to forge a Bitcoin block or break any cryptographic primitive — they only need to know the Merkle tree structure of any real block already accepted by the light client. The corrupted return value (`true`) directly propagates to the consuming contract, enabling unauthorized asset movement or state mutation in that contract.

---

### Likelihood Explanation

The attack requires no privileged access. The attacker needs only:
- Knowledge of any canonical Bitcoin block stored by the light client (publicly observable on-chain).
- The ability to call a public NEAR contract method.
- Basic Merkle tree arithmetic (no cryptographic break required).

Any DApp that has not migrated to `verify_transaction_inclusion_v2` remains exposed. The v1 function is not gated, rate-limited, or access-controlled.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) entirely from the contract, or add an `env::panic_str` body that unconditionally aborts, so it cannot be called at runtime. The `#[deprecated]` attribute alone is insufficient because it only affects compile-time callers of the Rust crate, not external NEAR accounts calling the deployed contract via RPC. All callers must be migrated to `verify_transaction_inclusion_v2`, which enforces equal proof lengths and validates the coinbase leaf to establish tree depth, preventing internal-node substitution.

---

### Proof of Concept

**Setup:** Light client is initialized with real Bitcoin mainnet blocks (e.g., heights 685440–685452 as shown in the test suite). Block at height 685452 is on the canonical chain with a known `merkle_root`.

**Steps:**

1. Obtain the Merkle tree for block 685452. Let `L` = left child of the root (an internal node at depth 1). Let `R` = right child of the root (the sibling of `L`).
2. Construct `ProofArgs`:
   - `tx_id = L` (an internal node, not a real transaction)
   - `tx_block_blockhash` = hash of block 685452 (on canonical chain)
   - `tx_index = 0` (position 0 at depth 1 → even, so left child)
   - `merkle_proof = [R]` (one sibling: the right child of the root)
   - `confirmations = 1`
3. Call `verify_transaction_inclusion` with these args.
4. Internally: `compute_root_from_merkle_proof(L, 0, [R])` = `SHA256d(L || R)` = `merkle_root`. The comparison succeeds.
5. **Result:** The function returns `true` for a transaction `L` that does not exist in the block.

**Expected correct result:** `false` — `L` is not a valid transaction hash.

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
