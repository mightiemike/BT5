Audit Report

## Title
`SwapAllowlistExtension` gates the router address instead of the actual user for router-mediated swaps, making per-user allowlist bypassable - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which is the `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` of `pool.swap()`, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. Any pool admin who allowlists the router to enable router-mediated swaps for their permitted users inadvertently opens the pool to all users, fully bypassing the per-user allowlist.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` — making the router the `msg.sender` of `pool.swap()`: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` — the economically relevant LP position holder — regardless of who calls `addLiquidity`: [5](#0-4) 

The swap allowlist has no equivalent mechanism to recover the actual user identity through the router. The `recipient` field is available in `beforeSwap` (second parameter, currently ignored) and is set by the user even through the router, but the extension does not use it.

## Impact Explanation

A pool admin deploying a private pool with `SwapAllowlistExtension` who allowlists the router — the natural and expected configuration step to enable router-mediated swaps for permitted users — inadvertently grants swap access to every address. Any unpermitted user can call `router.exactInputSingle` or `router.exactInput` targeting the pool; the extension sees `sender = router`, finds it allowlisted, and permits the swap. The per-user allowlist is entirely bypassed for all router-mediated paths. Unauthorized traders can extract value from the pool's LP positions through unrestricted swaps, constituting direct loss of LP principal. This meets the Sherlock threshold for a High-severity finding: broken core access-control functionality causing direct loss of user funds.

## Likelihood Explanation

The bypass requires only that the pool admin allowlists the router, which is the standard and expected configuration for any pool that intends to support router-mediated swaps alongside the allowlist. No special privileges, front-running, or unusual conditions are required for the attacker — any address can call the router. The misconfiguration is invisible to the pool admin because the allowlist appears correctly configured from their perspective. The condition is reachable in normal production use.

## Recommendation

Pass the original user identity through the swap path so the extension can gate the actual swapper. Two concrete options:

1. **Preferred**: Change `beforeSwap` to check `recipient` (the second parameter, currently ignored) instead of `sender`. The `recipient` is set by the user even through the router (`params.recipient` in `exactInputSingle`) and correctly identifies the economically relevant party.
2. **Alternative**: Mirror the deposit allowlist pattern — add an explicit `swapper` parameter to the pool's `swap` signature (analogous to `owner` in `addLiquidity`) so the caller can declare the economically relevant identity and the extension can gate on it.

## Proof of Concept

1. Deploy pool with `SwapAllowlistExtension` attached.
2. Pool admin calls `setAllowedToSwap(pool, userA, true)` — only `userA` is permitted.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — router allowlisted to enable router-mediated swaps for `userA`.
4. `userB` (not allowlisted) calls `router.exactInputSingle({pool: pool, recipient: userB, ...})`.
5. Router calls `pool.swap(userB, ...)` — `msg.sender` of `pool.swap()` is the router.
6. Pool calls `_beforeSwap(router, userB, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` = `true` → swap proceeds.
8. `userB` successfully trades in the private pool, bypassing the per-user allowlist entirely.
