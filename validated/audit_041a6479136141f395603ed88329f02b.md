### Title
CEI Violation in `withdrawCollateral` Enables Cross-Function Re-Entrancy Double Withdrawal - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs an external token transfer via `handleWithdrawTransfer` **before** decrementing the sender's balance in `SpotEngine`. This violates the Checks-Effects-Interactions pattern. An attacker who controls a re-entrant callback path (e.g., via a token with transfer hooks, or if `BaseWithdrawPool.submitWithdrawal` immediately pushes tokens to the recipient) can re-enter `Endpoint.executeSlowModeTransaction` while the balance is still at its pre-withdrawal value, causing a second queued withdrawal to pass the health check and drain double the collateral.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

1. **External call** — `handleWithdrawTransfer(token, sendTo, amount, idx)` (line 408)
2. **State update** — `spotEngine.updateBalance(productId, sender, amountRealized)` (line 412)
3. **Health check** — `getHealth(sender, healthType) >= 0` (line 419) [1](#0-0) 

`handleWithdrawTransfer` makes two external calls before any state is written:

```solidity
function handleWithdrawTransfer(...) internal virtual {
    token.safeTransfer(withdrawPool, uint256(amount));          // external call 1
    BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx); // external call 2
}
``` [2](#0-1) 

At the moment these external calls execute, `spotEngine.updateBalance` has **not yet been called**, so the sender's balance in `SpotEngine` still reflects the full pre-withdrawal amount.

The public re-entry path is `Endpoint.executeSlowModeTransaction`, which is callable by anyone and processes queued `WithdrawCollateral` slow-mode transactions through `processSlowModeTransaction` → `processSlowModeTransactionImpl` → `clearinghouse.withdrawCollateral`. There is no `nonReentrant` guard on any of these functions. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

An attacker who can trigger a callback during `handleWithdrawTransfer` (e.g., via an ERC-777 token or a `BaseWithdrawPool.submitWithdrawal` implementation that immediately pushes tokens to the recipient) can:

1. Queue two `WithdrawCollateral` slow-mode transactions for the same collateral.
2. Call `executeSlowModeTransaction` to process the first withdrawal.
3. During the external call in `handleWithdrawTransfer`, re-enter `executeSlowModeTransaction` to process the second withdrawal.
4. Because `spotEngine.updateBalance` has not yet run, the balance is still the original amount; the health check at line 419 passes for the second withdrawal.
5. Both withdrawals succeed, draining twice the deposited collateral.

This is a direct theft of funds from the protocol.

---

### Likelihood Explanation

The re-entrancy requires a callback during `handleWithdrawTransfer`. This is realistic if:
- Any supported collateral token implements ERC-777 or a custom `transferFrom` hook (common in DeFi ecosystems).
- `BaseWithdrawPool.submitWithdrawal` immediately transfers tokens to the `to` address (the file was not accessible for confirmation, but the pattern is architecturally plausible).

No privileged access is required. Any trader with a deposited balance can attempt this.

---

### Recommendation

Apply the Checks-Effects-Interactions pattern: update `spotEngine.updateBalance` and perform the health check **before** calling `handleWithdrawTransfer`.

```diff
 function withdrawCollateral(...) public virtual onlyEndpoint {
     ...
+    int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
+    int128 amountRealized = -int128(amount) * int128(multiplier);
+    spotEngine.updateBalance(productId, sender, amountRealized);
+    spotEngine.assertUtilization(productId);
+    require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

     handleWithdrawTransfer(token, sendTo, amount, idx);

-    int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
-    int128 amountRealized = -int128(amount) * int128(multiplier);
-    spotEngine.updateBalance(productId, sender, amountRealized);
-    spotEngine.assertUtilization(productId);
-    require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
     emit ModifyCollateral(amountRealized, sender, productId);
 }
```

Alternatively, add OpenZeppelin's `ReentrancyGuard` to `Clearinghouse` and apply `nonReentrant` to `withdrawCollateral`.

### Proof of Concept

1. Attacker deploys a contract with a re-entrant `receive()` or ERC-777 `tokensReceived` hook.
2. Attacker deposits collateral and queues **two** `WithdrawCollateral` slow-mode transactions (each for the full balance).
3. After the delay, attacker calls `executeSlowModeTransaction` to process the first withdrawal.
4. During `token.safeTransfer(withdrawPool, amount)` or `withdrawPool.submitWithdrawal(token, to, amount, idx)`, the callback fires.
5. In the callback, attacker calls `executeSlowModeTransaction` again to process the second withdrawal.
6. `spotEngine.updateBalance` has not yet run for the first withdrawal; the balance is still full. The health check at line 419 passes.
7. Both withdrawals complete. Attacker receives `2 × amount` while only having deposited `amount`. [5](#0-4)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
```
