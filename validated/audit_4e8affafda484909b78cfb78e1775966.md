Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Original Swapper, Allowing Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender` at swap time. When `MetricOmmSimpleRouter` intermediates a swap, the pool's `msg.sender` is the router contract, not the original EOA. If the pool admin allowlists the router (required for any legitimate router-mediated swap), every unprivileged user can bypass the per-user allowlist by routing through the router.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs its identity check as:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension caller) and `sender` is the first argument forwarded from the pool. In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as `sender`:

```solidity
_beforeSwap(
    msg.sender,   // pool's msg.sender — the router, not the original EOA
    recipient, ...
);
```

This flows through `ExtensionCalling._beforeSwap` unchanged into the extension call. When a user calls `MetricOmmSimpleRouter.exactInputSingle(...)`, the router calls `pool.swap(...)` directly — making the pool's `msg.sender` the router address. The router stores the original caller only in transient storage for the payment callback (`_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)`), but never encodes it into `extensionData` or any argument visible to the extension. The `params.extensionData` is passed through verbatim from the user's call with no router-injected identity.

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The pool admin faces an impossible choice: not allowlisting the router blocks all router-mediated swaps including for legitimately allowlisted users; allowlisting the router grants every address on the network the ability to bypass the per-user allowlist.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict swaps to specific counterparties (KYC-gated, institutional, or protocol-internal) has its access control rendered entirely ineffective for any user routing through `MetricOmmSimpleRouter`. Any unprivileged address can execute swaps on a restricted pool by calling `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the router, bypassing the admin-configured per-user allowlist. This constitutes a broken core pool functionality and admin-boundary break with direct fund-impact potential, as restricted pool liquidity becomes accessible to unauthorized counterparties.

## Likelihood Explanation
`MetricOmmSimpleRouter` is a public periphery contract with no access controls — any address can call it at any time. The bypass requires only that the pool admin has allowlisted the router, which is a necessary operational step for any legitimate router-mediated swap to function. The trigger is fully unprivileged, requires no special setup, and is repeatable indefinitely.

## Recommendation
The extension must gate on the original user identity, not the intermediary. Two viable approaches:

1. **Router encodes original caller into `extensionData`**: The router prepends `msg.sender` into `extensionData` before passing it to `pool.swap`; the extension decodes and checks it. Requires a convention between router and extension.
2. **Transient-storage forwarder pattern**: The router writes the original caller to a known transient-storage slot before calling the pool; the extension reads that slot and checks it, treating the router as a trusted forwarder.

Alternatively, document explicitly that `SwapAllowlistExtension` only enforces per-user access for direct pool calls, and that router-mediated swaps must be blocked by not allowlisting the router — accepting that allowlisted users must call the pool directly.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` and `setAllowedToSwap(pool, router, true)` so Alice can use the router.
3. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: restrictedPool, ...})`.
4. The router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — pool's `msg.sender` is the router.
5. Pool calls `_beforeSwap(router, ...)` → `ExtensionCalling._beforeSwap` encodes `sender = router` → extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. Bob successfully swaps on the restricted pool, bypassing the per-user allowlist entirely.
