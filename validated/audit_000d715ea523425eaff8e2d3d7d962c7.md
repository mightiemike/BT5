### Title
Attacker-Controlled `_cached_sha256_treehash` Bypasses Hash Computation in `sha256_treehash` — (`File: wheel/python/clvm_rs/tree_hash.py`, `wheel/python/clvm_rs/program.py`)

---

### Summary

The Python wheel's `sha256_treehash` function and `Program.wrap()` unconditionally trust the `_cached_sha256_treehash` attribute on any `CLVMStorage`-conforming Python object, without verifying it against the object's actual content. An attacker who supplies a custom `CLVMStorage` object with a pre-set wrong `_cached_sha256_treehash` can cause `Program.tree_hash()` to return an arbitrary hash, corrupting puzzle-hash verification, coin-ID derivation, and equality checks — all without touching the actual CLVM bytes.

---

### Finding Description

**Vulnerability class:** Cache/allocator aliasing — a stale or attacker-supplied cached value is consumed in place of a freshly computed one, analogous to the governance bug where current voting power is read instead of a fixed snapshot.

**Root cause — `tree_hash.py`:**

```python
# wheel/python/clvm_rs/tree_hash.py  line 57-61
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    self.cache_hits += 1
    hash_stack.append(r)
    return
```

`sha256_treehash` short-circuits immediately when `_cached_sha256_treehash` is present on the object. It performs **zero verification** that the cached bytes actually match `sha256(atom_prefix || atom)` or `sha256(pair_prefix || left_hash || right_hash)`. Any Python object that satisfies the structural `CLVMStorage` protocol (i.e., has `.atom` and `.pair`) and carries a pre-set `_cached_sha256_treehash` will have that value returned verbatim.

**Root cause — `program.py` `wrap()`:**

```python
# wheel/python/clvm_rs/program.py  line 141-142
o._cached_serialization = getattr(v, "_cached_serialization", None)
o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)
```

`Program.wrap()` propagates whatever `_cached_sha256_treehash` is present on the input `CLVMStorage` object directly onto the new `Program` instance, again without verification.

**Root cause — `program.py` `tree_hash()`:**

```python
# wheel/python/clvm_rs/program.py  line 284-286
def tree_hash(self) -> bytes:
    if self._cached_sha256_treehash is None:
        self._cached_sha256_treehash = sha256_treehash(self._unwrapped)
    return self._cached_sha256_treehash
```

Once `_cached_sha256_treehash` is set (either by `wrap()` or by a prior call), `tree_hash()` returns it unconditionally. The hash is never recomputed or validated.

**The `CLVMStorage` protocol explicitly documents this field as an optional optimization hint:**

```python
# wheel/python/clvm_rs/clvm_storage.py  line 20-21
# optional fields used to speed implementations:
# `_cached_sha256_treehash: Optional[bytes]` is used by `sha256_treehash`
```

Because the protocol is structural (duck-typed), any Python object with `.atom`, `.pair`, and `._cached_sha256_treehash` qualifies — including attacker-controlled objects.

---

### Impact Explanation

`Program.tree_hash()` is the canonical puzzle-hash function in the Chia Python stack. It is used for:

1. **Coin-ID derivation** — `coin_id = sha256(parent_id || puzzle_hash || amount)`. A wrong puzzle hash produces a wrong coin ID, breaking spend validation.
2. **Equality checks** — `Program.__eq__` delegates entirely to `tree_hash()`:
   ```python
   # program.py  line 188
   return self.tree_hash() == other_obj.tree_hash()
   ```
   Two programs with different content but the same injected hash compare as equal; two programs with the same content but different injected hashes compare as unequal.
3. **Curry-hash computation** — `curry_hash()` calls `self.tree_hash()` as its base, so curried puzzle hashes are also corrupted.
4. **Mempool/consensus validation** — any layer that calls `.tree_hash()` on a `Program` wrapping an attacker-supplied object will receive the wrong hash.

The corrupted hash is a `bytes` value, so it passes all downstream type checks silently.

---

### Likelihood Explanation

The `CLVMStorage` protocol is intentionally open to third-party implementations. Any code path that calls `Program.wrap(external_obj)` or `Program.to(external_obj)` where `external_obj` is not produced by the library's own deserializers is vulnerable. Realistic entry points include:

- Wallet or node code that wraps externally-deserialized puzzle objects.
- Any library that implements `CLVMStorage` and sets `_cached_sha256_treehash` from an untrusted source (e.g., a network message that includes a pre-computed hash field).
- A malicious dependency that returns a `CLVMStorage`-conforming object with a poisoned cache.

The attack requires no special privileges — only the ability to supply a Python object to `Program.wrap()` or `Program.to()`, which is the normal public API.

---

### Recommendation

1. **Remove unconditional cache trust in `sha256_treehash`:** Do not short-circuit on `_cached_sha256_treehash` unless the object was produced by a trusted internal path (e.g., `CLVMTree` deserialization, where hashes are computed from the blob). At minimum, add a debug-mode assertion that the cached value matches the freshly computed one.

2. **Do not propagate the cached hash in `Program.wrap()`:** Either drop the propagation entirely and always recompute on first call, or only propagate when the source object is a known-trusted type (e.g., `isinstance(v, CLVMTree)` or `isinstance(v, Program)`).

3. **Treat `_cached_sha256_treehash` as write-once after first computation:** Once `Program.tree_hash()` computes the hash, it should be immutable. Never accept it from an external object.

---

### Proof of Concept

```python
from clvm_rs.program import Program
from clvm_rs.clvm_storage import CLVMStorage

# Legitimate program: atom b"hello"
real = Program.new_atom(b"hello")
real_hash = real.tree_hash()
print("real hash:", real_hash.hex())

# Attacker-controlled CLVMStorage with wrong cached hash
class PoisonedStorage:
    atom = b"evil_content"
    pair = None
    _cached_sha256_treehash = real_hash  # inject the hash of b"hello"

poisoned = Program.wrap(PoisonedStorage())

# tree_hash() returns the injected hash, not the hash of b"evil_content"
assert poisoned.tree_hash() == real_hash, "hash was not injected"

# __eq__ now says poisoned == real, despite different content
assert poisoned == real, "equality check is corrupted"

# The actual bytes differ
assert bytes(poisoned) != bytes(real)
print("PoC: Program with content 'evil_content' reports hash of 'hello'")
