### Title
SwapAllowlistExtension gates the router address instead of the actual end-user, allowing any user to bypass the per-user swap allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is the pool's `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, `sender` equals the router's address, not the actual end-user. A pool admin who allowlists the router (the natural step to permit router-mediated swaps for their allowlisted users) inadvertently opens the gate to every user on the network.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured extension: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted for the calling pool: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` (and every other router entry-point) calls `pool.swap(...)` directly, making the pool's `msg.sender` the router contract, not the originating user: [3](#0-2) 

The actual end-user (`msg.sender` of the router call) is never forwarded to the pool or to the extension. The extension therefore sees only the router address. A pool admin who wants allowlisted users to be able to trade through the router must add the router to `allowedSwapper[pool][router]`. Once that entry exists, every user on the network can call `exactInputSingle` (or any other router function) and the extension will pass them through, because the router is allowlisted — not the individual user.

The `DepositAllowlistExtension` does not share this flaw: it gates on `owner` (the position owner explicitly supplied by the caller), which the liquidity adder preserves correctly. [4](#0-3) 

### Impact Explanation

Any pool that deploys `SwapAllowlistExtension` to restrict trading to a curated set of addresses (KYC'd users, institutional counterparties, whitelisted market-makers) loses that protection entirely the moment the router is allowlisted. An unprivileged user can trade against the pool's liquidity, extract value at the pool's quoted prices, and trigger any downstream accounting or regulatory consequence the allowlist was meant to prevent. This is a direct policy bypass with fund-impacting consequences for LP assets and protocol integrity.

### Likelihood Explanation

The scenario is realistic and likely. The `MetricOmmSimpleRouter` is the standard public entry-point for swaps. A pool admin who configures `SwapAllowlistExtension` and then wants their allowlisted users to be able to use the router will naturally add the router to the allowlist — the documentation and extension interface give no indication that doing so opens the gate to all users. The bypass requires no special privileges, no flash loans, and no unusual token behavior: any user calls a public router function.

### Recommendation

Pass the originating user through the swap call chain so extensions can gate on the actual economic actor. One approach: add an `originator` field to the `beforeSwap` extension interface (populated with the router's `msg.sender` when the router calls the pool, or with `msg.sender` directly when the pool is called directly). Alternatively, document explicitly that `SwapAllowlistExtension` cannot be used with the router and enforce this at the extension or factory level. A third option is to have the router pass the originating user as part of `extensionData` and have the extension decode it, but this requires a trusted encoding convention.

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, userA, true)` — only `userA` should be able to swap.
3. Admin calls `setAllowedToSwap(pool, router, true)` — intending to let `userA` use the router.
4. `userB` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. `_beforeSwap` passes `sender = router` to the extension.
7. The extension evaluates `allowedSwapper[pool][router] == true` → passes.
8. `userB`'s swap executes against the pool's liquidity with no restriction.
