### Title
`starknet_estimateFee` Returns Internally Inconsistent `FeeEstimation`: `overall_fee` Silently Includes L2 Gas Cost Absent from Breakdown Fields — (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

The `FeeEstimation` struct populated by `tx_execution_output_to_fee_estimation` exposes `l2_gas_price` but omits `l2_gas_consumed`. The `overall_fee` field is taken directly from `receipt.fee`, which is computed from the full three-component gas vector including L2 gas. For any V3 transaction that consumes L2 gas, `overall_fee` is strictly greater than `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` — the formula documented in both the Rust struct comment and the OpenRPC schema. The discrepancy equals `l2_gas_consumed * (l2_gas_price + tip)`. A client that reconstructs the fee from the breakdown fields, or that uses the breakdown to set `l2_gas.max_amount` in `AllResourceBounds`, will produce a wrong (too-low) value and have its transaction rejected at the gateway.

### Finding Description

**Root cause — `tx_execution_output_to_fee_estimation`:**

**Gateway enforcement uses the missing dimension:**

`check_fee_bounds` validates `l2_gas.max_amount` against `estimate_minimal_gas_vector(...).l2_gas`. ### Impact Explanation

`starknet_estimateFee` and `starknet_simulateTransactions` return an authoritative-looking `FeeEstimation` object whose `overall_fee` is provably larger than the sum implied by the two breakdown fields whenever `l2_gas > 0`. Any client that:

1. Reconstructs the fee from the breakdown to decide whether to submit a transaction, or
2. Derives `l2_gas.max_amount` from the fee-estimate response for a V3 `AllResourceBounds` transaction,

will compute a wrong (too-low) value. In case (2) the transaction is rejected at the gateway before sequencing. This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

### Likelihood Explanation

Every V3 transaction executed in `GasVectorComputationMode::All` (i.e., `AllResourceBounds`) produces non-zero `l2_gas` in its receipt. This is the standard mode for all post-0.13.3 transactions. The discrepancy is therefore present for the vast majority of modern transactions and is reproducible on any node running this code.

### Recommendation

1. Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it with `gas_vector.l2_gas.0.into()` in `tx_execution_output_to_fee_estimation`.
2. Update the `overall_fee` doc comment and the OpenRPC `FEE_ESTIMATE` schema description to reflect the correct three-term formula: `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price + l2_gas_consumed * l2_gas_price`.
3. Resolve the existing `TODO(Tzahi)` which already tracks this gap.

### Proof of Concept

1. Submit any V3 invoke transaction to `starknet_estimateFee` on a node running this code.
2. Observe the response: `overall_fee` is non-zero, `gas_consumed` and `data_gas_consumed` are present, but `l2_gas_consumed` is absent.
3. Compute `reconstructed_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`.
4. Assert `reconstructed_fee < overall_fee`; the difference is `l2_gas * (l2_gas_price + tip)`.
5. Construct a V3 transaction with `AllResourceBounds` where `l2_gas.max_amount` is set to `0` (the only value derivable from the fee-estimate breakdown for that field).
6. Submit to the gateway; it is rejected with `MaxGasAmountTooLow` because `estimate_minimal_gas_vector` returns a non-zero `l2_gas` lower bound that exceeds the user-supplied zero bound.

pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,

    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),

        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),

        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
