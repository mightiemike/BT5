### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas` When L2/Data-Gas Are Zero, Producing a Wrong Transaction Hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a zero-check on `l2_gas` and `l1_data_gas` to decide which variant to reconstruct. When both are zero it silently returns `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. Because `get_tip_resource_bounds_hash` hashes a **different number of elements** for the two variants (2 vs 3), any `AllResources` transaction whose `l2_gas` and `l1_data_gas` happen to be zero will produce a **different transaction hash** after a protobuf round-trip. This breaks hash/signature binding for every component that deserializes the transaction from protobuf (P2P block sync, consensus message handling).

### Finding Description

**Root cause — protobuf deserializer:**

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  L417-L436
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
``` [1](#0-0) 

The serializer always emits all three fields for both variants:

```rust
// L471-L489
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),          // zero when user set it to 0
        l1_data_gas: Some(l1_data_gas.into()), // zero when user set it to 0

``` [2](#0-1) 

So an `AllResources` transaction with `l2_gas = 0` and `l1_data_gas = 0` serializes to protobuf with all three fields present and zero, then deserializes back as `L1Gas` — a different variant.

**Hash divergence — `get_tip_resource_bounds_hash`:**

```rust
// crates/starknet_api/src/transaction_hash.rs  L188-L210
pub fn get_tip_resource_bounds_hash(resource_bounds: &ValidResourceBounds, tip: &Tip)
    -> Result<Felt, StarknetApiError>
{
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements
        ValidResourceBounds::AllResources(all) =>
            vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?],    // 3 elements
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
``` [3](#0-2) 

`AllResources` hashes `[tip, L1_gas, L2_gas, L1_data_gas]` (4 Poseidon inputs); `L1Gas` hashes `[tip, L1_gas, L2_gas]` (3 inputs). Even when `l2_gas = 0` and `l1_data_gas = 0`, the extra zero element changes the Poseidon digest. The two hashes are **always distinct**.

**`InvokeTransactionV3` carries `ValidResourceBounds` directly:**

```rust
// crates/starknet_api/src/transaction.rs  L663-L688
pub struct InvokeTransactionV3 {
    pub resource_bounds: ValidResourceBounds,   // ← holds the deserialized variant
    ...
}
impl TransactionHasher for InvokeTransactionV3 {
    fn calculate_transaction_hash(...) -> Result<TransactionHash, StarknetApiError> {
        get_invoke_transaction_v3_hash(self, chain_id, transaction_version)
    }

``` [4](#0-3) 

When the P2P block-sync path deserializes a `FullTransaction` from protobuf, the `InvokeTransactionV3.resource_bounds` is set to the deserialized `ValidResourceBounds::L1Gas`. Any subsequent call to `calculate_transaction_hash` (e.g., for block-hash verification or receipt matching) uses the 3-element Poseidon preimage instead of the 4-element one the signer used, producing hash H2 ≠ H1.

### Impact Explanation

A node receiving a synced block via P2P will recompute transaction hashes from the deserialized `Transaction` objects. For every `AllResources` transaction where `l2_gas = 0` and `l1_data_gas = 0`, the recomputed hash differs from the hash the sequencer stored in the block. This causes:

- **Wrong receipt/event/state attribution** — the stored `tx_hash` in receipts and events does not match the recomputed hash, corrupting the block's transaction index.
- **Wrong class hash / compiled class hash selection** — if the wrong hash is used to look up execution artifacts, the wrong CASM/native artifact may be selected.
- **Signature binding broken** — the account signed the `AllResources` preimage (H1); the verifier checks against the `L1Gas` preimage (H2). Account validation will fail or accept the wrong preimage.

This matches the impact category: **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

### Likelihood Explanation

The trigger condition is an `AllResources` V3 transaction with `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0`, and non-zero `l1_gas`. The gateway's `validate_resource_bounds` only rejects transactions where the **total** max possible fee is zero:

```rust
if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
    return Err(...ZeroResourceBounds...);
}
``` [5](#0-4) 

A transaction with non-zero `l1_gas` passes this check regardless of `l2_gas` and `l1_data_gas`. Any user who submits a V3 transaction specifying only an L1-gas bound (setting L2 and data-gas to zero) while using the `AllResources` encoding triggers the bug on every node that syncs the block via P2P.

### Recommendation

Remove the zero-value heuristic from the protobuf deserializer. The variant must be encoded explicitly in the protobuf message (e.g., a boolean `is_all_resources` flag, or a dedicated oneof), not inferred from field values. Until the wire format is extended, the deserializer should default to `AllResources` when all three fields are present, regardless of their values:

```rust
// Proposed fix
Ok(if value.l1_data_gas.is_none() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

This preserves backward compatibility with 0.13.2 peers (which omit `l1_data_gas`) while correctly reconstructing `AllResources` for any transaction that explicitly set the field to zero.

### Proof of Concept

1. Construct an `AllResources` V3 invoke transaction with `l1_gas = {max_amount: 1000, max_price_per_unit: 1}`, `l2_gas = {0, 0}`, `l1_data_gas = {0, 0}`.
2. Compute H1 = `get_invoke_transaction_v3_hash(tx, chain_id, version)` — uses `AllResources`, 4-element Poseidon.
3. Serialize `tx.resource_bounds` via `From<ValidResourceBounds> for protobuf::ResourceBounds` → all three fields present, l2/data-gas zero.
4. Deserialize via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` → returns `L1Gas(l1_gas)`.
5. Reconstruct `InvokeTransactionV3 { resource_bounds: L1Gas(...), ... }`.
6. Compute H2 = `get_invoke_transaction_v3_hash(tx, chain_id, version)` — uses `L1Gas`, 3-element Poseidon.
7. Assert H1 ≠ H2. The syncing node stores H2 while the sequencer committed H1, breaking all hash-indexed lookups for this transaction.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
