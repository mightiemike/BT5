### Title
Hardcoded `Fee(1)` in Consensus L1 Handler Conversion Produces Wrong `paid_fee_on_l1` in Executable Transaction — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

`convert_consensus_l1_handler_to_internal_l1_handler` hardcodes `paid_fee_on_l1` as `Fee(1)` instead of propagating the actual fee paid on L1. This is the direct Sequencer analog of the AmarokFacet `recovery: msg.sender` pattern: a critical per-transaction parameter is replaced with a fixed placeholder at a public-to-internal conversion boundary, causing the executable object to carry the wrong value.

---

### Finding Description

`ConsensusTransaction::L1Handler` wraps `transaction::L1HandlerTransaction`, which carries no `paid_fee_on_l1` field. When the converter promotes this to an `executable_transaction::L1HandlerTransaction`, it must supply the fee. Instead of sourcing it from the originating L1 event, the code unconditionally writes `Fee(1)`:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs  L473-483
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

The blockifier's L1 handler execution path then reads `self.paid_fee_on_l1` and performs the only fee guard:

```rust
// crates/blockifier/src/transaction/l1_handler_transaction.rs  L103-113
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee {
            paid_fee,
            actual_fee: receipt.fee,
        },
    )));
}
``` [2](#0-1) 

Because `Fee(1) ≠ Fee(0)`, the guard is permanently bypassed for every L1 handler transaction that enters through the consensus path. The `paid_fee_on_l1` field on the resulting `executable_transaction::L1HandlerTransaction` is structurally defined to carry the real value:

```rust
// crates/starknet_api/src/executable_transaction.rs  L380-396
pub struct L1HandlerTransaction {
    pub tx: crate::transaction::L1HandlerTransaction,
    pub tx_hash: TransactionHash,
    pub paid_fee_on_l1: Fee,
}
``` [3](#0-2) 

The native-blockifier Python path correctly reads the real value from the L1 event:

```rust
// crates/native_blockifier/src/py_l1_handler.rs  L37-39
let paid_fee_on_l1 = Fee(py_attr::<u128>(py_tx, "paid_fee_on_l1")?);
Ok(L1HandlerTransaction { tx, tx_hash, paid_fee_on_l1 })
``` [4](#0-3) 

The consensus converter path is the only path that substitutes a placeholder.

---

### Impact Explanation

**Incorrect fee/resource accounting with economic impact (Critical scope).**

1. The `paid_fee_on_l1 = Fee(1)` placeholder permanently satisfies the `paid_fee == Fee(0)` guard, so the blockifier never rejects an L1 handler transaction on fee grounds regardless of what was actually paid on L1.
2. The `InsufficientFee` error path, which is the sequencer's only enforcement point for L1 handler fee adequacy, is rendered dead code for all consensus-path L1 handler transactions.
3. The actual fee paid on L1 is not recorded in the executable object, so any downstream logic that reads `paid_fee_on_l1` (e.g., receipts, auditing, future stricter checks) will see `1` instead of the real value, producing wrong accounting.

---

### Likelihood Explanation

**Low.** L1 handler transactions originate from L1 contract events that already enforce non-zero fee payment at the Ethereum layer. In practice the real `paid_fee_on_l1` is almost always non-zero, so the bypass of the `Fee(0)` guard rarely changes observable execution outcomes today. The risk materialises if: (a) a stricter fee check is added later without fixing the placeholder, (b) the L1 contract's enforcement is weakened, or (c) a malicious proposer injects a crafted L1 handler transaction through the consensus P2P path where no independent L1-event cross-check is performed by validators.

---

### Recommendation

Carry `paid_fee_on_l1` through the consensus transaction type. Either:

- Add a `paid_fee_on_l1: Fee` field to `ConsensusTransaction::L1Handler` (or a wrapper) so the proposer can include the real value from the L1 event, and validators can verify it against their own L1 event view; or
- Have each validator independently look up the fee for the given L1 handler nonce from its own L1 event store before constructing the executable transaction, mirroring how the native-blockifier path sources the value.

Remove the `Fee(1)` placeholder and the associated TODO once the real value is plumbed through.

---

### Proof of Concept

1. A block proposer includes a `ConsensusTransaction::L1Handler` in a proposal.
2. Each validator calls `convert_consensus_tx_to_internal_consensus_tx`, which dispatches to `convert_consensus_l1_handler_to_internal_l1_handler`.
3. `L1HandlerTransaction::create(tx, &self.chain_id, Fee(1))` is called — `paid_fee_on_l1` is set to `1` regardless of the actual L1 event fee.
4. The blockifier executes the transaction; the guard `if paid_fee == Fee(0)` evaluates to `false` (since `1 ≠ 0`) and execution proceeds unconditionally.
5. The `TransactionExecutionInfo` receipt records `fee = Fee(0)` (set by `l1_handler_tx_execution_info`) while `paid_fee_on_l1` in the executable object is `1`, not the real L1 fee — producing divergent, wrong fee accounting. [1](#0-0) [5](#0-4) [6](#0-5)
