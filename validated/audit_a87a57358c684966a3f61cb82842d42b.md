### Title
`deser_auto` Silently Bypasses `max_atom_len` for Non-serde_2026 Blobs — (File: `wheel/src/api.rs`)

### Summary

`deser_auto` is a Python-facing auto-detecting deserializer that accepts a `max_atom_len` security parameter. The parameter is enforced only when the blob is in serde_2026 format; when the blob is in backrefs/legacy format, `node_from_bytes_backrefs` is called with no atom-size cap at all. An attacker who controls the serialized blob can trivially bypass the `max_atom_len` guard by omitting the serde_2026 magic prefix, causing unbounded allocation in the Rust `Allocator`.

### Finding Description

In `wheel/src/api.rs`, `deser_auto` is the Python convenience entry point for format-agnostic deserialization:

```rust
// wheel/src/api.rs  lines 147-157
#[pyfunction]
#[pyo3(signature = (blob, *, max_atom_len=PY_DEFAULT_MAX_ATOM_LEN, strict=true))]
fn deser_auto(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
        deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)
            .map_err(eval_to_py)?
    } else {
        node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?   // ← max_atom_len silently dropped
    };
    Ok(LazyNode::new(Rc::new(a), node))
}
``` [1](#0-0) 

The `max_atom_len` parameter (defaulting to `PY_DEFAULT_MAX_ATOM_LEN = 1 << 20`) is:

- **Enforced** for serde_2026 blobs via `deserialize_2026_body_from_stream`, which rejects any atom whose encoded length exceeds the cap.
- **Not enforced** for backrefs/legacy blobs: `node_from_bytes_backrefs` receives no size cap and will allocate atoms of arbitrary size. [2](#0-1) 

The structural parallel to the Argo bug is exact:

| Argo | clvm_rs |
|---|---|
| `argo_liquidate::liquidate_repay` (wrapper) enforces repayment-amount check | `deser_auto` (wrapper) accepts and enforces `max_atom_len` |
| `argo_engine::liquidate_repay` (underlying) requires no capability | `node_from_bytes_backrefs` (underlying) accepts no size cap |
| Attacker calls underlying directly, repays 1 token | Attacker omits magic prefix, atom size limit is never applied |

The `deser_legacy` and `deser_backrefs` siblings also carry no atom-size cap, but they do not advertise one either. `deser_auto` is the only function that accepts `max_atom_len` and then silently fails to forward it on one branch. [3](#0-2) 

### Impact Explanation

Any Python caller that invokes `deser_auto(blob, max_atom_len=N)` expecting a hard cap on atom size receives no such guarantee when the blob is in backrefs format. Concretely:

1. **Memory exhaustion**: A crafted backrefs blob encoding a single atom of, say, 2 GB causes the Rust `Allocator` to attempt a 2 GB allocation. The process OOMs or is killed by the OS before the Python caller can apply any downstream limit.
2. **Security-boundary bypass**: Downstream code (e.g., chia_rs, wallet software) that calls `deser_auto` with a reduced `max_atom_len` to enforce a protocol-level atom-size invariant has that invariant silently voided for any attacker-supplied non-serde_2026 blob. Any subsequent check that assumes atoms are bounded is operating on a false premise.

### Likelihood Explanation

Likelihood is high. The attacker's only precondition is the ability to supply bytes to any code path that calls `deser_auto` — the standard entry point for format-agnostic CLVM deserialization in the Python wheel. Omitting the six-byte serde_2026 magic prefix is trivial. No keys, privileges, or special network access are required.

### Recommendation

Forward `max_atom_len` to the backrefs branch, either by adding a size-capped wrapper around `node_from_bytes_backrefs`, or by performing an explicit post-parse walk that rejects oversized atoms. The fix should mirror the serde_2026 path so that the parameter is honoured regardless of which branch is taken:

### Proof of Concept

```python
import clvm_rs

# Craft a backrefs-format blob encoding a single 4 MB atom.
# CLVM atom encoding: 0b11_xxxxxx prefix for 3-byte length, then payload.
size = 4 * 1024 * 1024          # 4 MB
header = bytes([0b11_000000 | (size >> 16), (size >> 8) & 0xFF, size & 0xFF])
blob = header + b'\xAA' * size  # no serde_2026 magic prefix

# Caller believes max_atom_len=1024 is enforced for all formats.
node = clvm_rs.deser_auto(blob, max_atom_len=1024)
# Succeeds — 4 MB atom is returned despite the 1 KB cap.
print(len(node.atom))  # 4194304
```

The call succeeds and returns a 4 MB atom, violating the caller-specified 1 KB limit. Scaling `size` to hundreds of megabytes causes OOM.
