### Title
`ContractOwner` Cannot Receive Native ETH from `DirectDepositV1.withdrawNative()` Due to Missing `receive()` Function — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner` is the privileged admin contract that manages `DirectDepositV1` instances. It exposes `withdrawFromDirectDepositV1(subaccount, address(0))` to recover native ETH stuck in a DDA. However, `ContractOwner` itself has no `receive()` or `fallback()` function, so when `DirectDepositV1.withdrawNative()` attempts to push ETH back to `ContractOwner`, the transfer fails and the entire call reverts. Any native ETH stranded in a `DirectDepositV1` contract is permanently unrecoverable through the intended path.

---

### Finding Description

`DirectDepositV1` wraps incoming native ETH into `wrappedNative` inside its `receive()` hook. However, the constructor explicitly handles the case where wrapping fails silently: [1](#0-0) 

If the `wrappedNative.call` fails during construction, ETH remains in the `DirectDepositV1` contract and a `NativeTokenTransferFailed` event is emitted. The intended recovery path is `ContractOwner.withdrawFromDirectDepositV1` with `token == address(0)`: [2](#0-1) 

This calls `DirectDepositV1.withdrawNative()`: [3](#0-2) 

`withdrawNative()` pushes ETH to `msg.sender`, which is `ContractOwner`. But `ContractOwner` declares no `receive()` or `fallback()` function: [4](#0-3) 

A plain ETH transfer to a contract with no `receive()` or `fallback()` reverts. Therefore `withdrawNative()` always reverts with `"Failed to transfer native token to owner"` when called from `ContractOwner`, and the `require(postBalance > preBalance, "empty")` check is never reached.

---

### Impact Explanation

Native ETH stranded in any `DirectDepositV1` contract (due to a failed wrap-on-construction) is permanently locked. The `withdraw(IIERC20Base)` function only handles ERC20 tokens and cannot recover native ETH. `withdrawNative()` is the sole recovery mechanism, and it is broken when invoked through `ContractOwner`. This constitutes a loss of funds for the protocol or its users.

---

### Likelihood Explanation

The failure condition (ETH stuck in DDA) is explicitly anticipated by the codebase — the constructor emits `NativeTokenTransferFailed` precisely because this scenario was considered possible. Any time a `DirectDepositV1` is deployed and the `wrappedNative` call fails (e.g., the wrapped-native contract is paused, has a bug, or the DDA is deployed before the wrapped-native contract is live), ETH will be stranded and unrecoverable. The recovery function `withdrawFromDirectDepositV1` is callable by the multisig owner, making this a realistic operational path.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

This mirrors the fix applied to `WalletImpl` in the referenced report: implement the missing callback/receiver so the contract can accept the asset type it is designed to handle.

---

### Proof of Concept

1. A `DirectDepositV1` is deployed via `ContractOwner.createDirectDepositV1(subaccount)`.
2. During construction, `wrappedNative.call{value: balance}("")` fails (e.g., wrapped-native contract is not yet live). ETH remains in the DDA; `NativeTokenTransferFailed` is emitted.
3. The multisig owner calls `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`.
4. `ContractOwner` calls `DirectDepositV1(directDepositV1).withdrawNative()`.
5. Inside `withdrawNative()`: `(bool success, ) = msg.sender.call{value: balance}("")` — `msg.sender` is `ContractOwner`, which has no `receive()` function → `success == false`.
6. `require(success, "Failed to transfer native token to owner")` reverts.
7. The entire `withdrawFromDirectDepositV1` call reverts. ETH is permanently locked in the DDA.
