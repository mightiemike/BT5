### Title
`DBCol::SpiceEndorsementStats` Not Seeded at State-Sync Boundary — First Post-Sync Block Fails to Compute Endorsement Stats (`core/store/src/columns.rs`)

---

### Summary

`DBCol::SpiceEndorsementStats` is a running per-epoch accumulator written during normal block processing. State sync skips block processing for the sync-boundary block, so the accumulator row for that block is never written. When the first post-sync block is processed, `compute_spice_endorsement_stats` reads the missing parent row and returns a hard error, preventing the node from processing any further blocks in the epoch.

---

### Finding Description

`DBCol::SpiceEndorsementStats` stores a per-block running accumulator of chunk-endorsement statistics, keyed by block hash. It is written by `record_spice_endorsement_stats_for_block` during normal block processing. [1](#0-0) 

The column definition carries an explicit TODO acknowledging the gap:

> `TODO(spice): each block's value is derived from its parent's, so this must be seeded at the state-sync boundary block (same as UncertifiedChunks) once spice state sync exists, otherwise the first post-sync block can't compute it.`

State sync downloads the trie state snapshot for the sync-boundary block but does **not** execute `record_spice_endorsement_stats_for_block` for it, so no `SpiceEndorsementStats` row is written for that block hash.

When the first post-sync block is processed, `compute_spice_endorsement_stats` is called: [2](#0-1) 

If the previous block (the sync-boundary block) is in the same epoch, it calls `get_spice_endorsement_stats`, which performs a hard read: [3](#0-2) 

Because the row is absent, this returns `Err(Error::Other(...))`. That error propagates through `record_spice_endorsement_stats_for_block` → `postprocess_block` → block acceptance, causing the node to reject every block it attempts to process after state sync within a Spice epoch.

The block-header validation path also calls `validate_spice_chunk_endorsement_stats`, which calls `spice_chunk_endorsement_stats_for_next_block`, which calls `compute_spice_endorsement_stats` for the same missing parent: [4](#0-3) 

---

### Impact Explanation

Any node that performs state sync into a Spice epoch is permanently unable to process blocks until the missing seed row is written. This is not a transient race: the row is structurally absent because the state-sync path never calls the writer. The node stalls at the sync point and cannot participate in consensus or serve RPC queries for new blocks.

**Severity: High** — state sync is a normal, unprivileged operation triggered automatically whenever a node falls more than one epoch behind the network tip. [5](#0-4) 

---

### Likelihood Explanation

State sync is exercised on every node restart after a prolonged outage and on every new node joining the network. Once `ProtocolFeature::Spice` is activated on a network, any node that state-syncs into that epoch will hit this path deterministically. No special attacker action is required; the trigger is ordinary network participation. [6](#0-5) 

---

### Recommendation

Seed `DBCol::SpiceEndorsementStats` for the sync-boundary block during `set_state_finalize` (or the equivalent Spice state-sync finalization path), analogous to how `UncertifiedChunks` must be seeded. The seed value at an epoch boundary is an all-zero accumulator (the epoch resets), so the correct value is deterministic and requires no additional data download. The existing `fold_endorsement_stats` helper with `prev_stats = None` produces the correct reset value. [7](#0-6) 

---

### Proof of Concept

1. Run a Spice-enabled network (`ProtocolFeature::Spice` active, protocol version ≥ 180).
2. Let a node fall more than one epoch behind so that it triggers state sync.
3. State sync completes: the node downloads the trie snapshot for the sync-boundary block `B_sync` but does not call `record_spice_endorsement_stats_for_block` for it.
4. The node attempts to process block `B_sync + 1`. `compute_spice_endorsement_stats` is called with `prev_hash = B_sync`.
5. Because `B_sync` is in the same epoch as `B_sync + 1`, the code enters the `Some(get_spice_endorsement_stats(...))` branch.
6. `get_spice_endorsement_stats` finds no row in `DBCol::SpiceEndorsementStats` for `B_sync` and returns `Err(Error::Other("missing spice endorsement stats for ..."))`.
7. Block `B_sync + 1` is rejected. Every subsequent block in the epoch is also rejected. The node is permanently stalled. [8](#0-7) [9](#0-8)

pub fn compute_spice_endorsement_stats(
    chain_store: &ChainStoreAdapter,
    epoch_manager: &dyn EpochManagerAdapter,
    epoch_id: &EpochId,
    prev_hash: &CryptoHash,
) -> Result<Vec<SpiceChunkEndorsementStats>, Error> {
    let epoch_info = epoch_manager.get_epoch_info(epoch_id)?;
    let num_validators = epoch_info.validators_iter().len();
    let contribution =
        endorsement_contribution_of_block(chain_store, epoch_manager, &epoch_info, prev_hash)?;

    // Carry the previous block's accumulator only within the same epoch; at the
    // epoch boundary (and genesis) the per-epoch accumulator resets.
    let prev_header = chain_store.get_block_header(prev_hash)?;
    let prev_stats = if !prev_header.is_genesis() && prev_header.epoch_id() == epoch_id {
        Some(get_spice_endorsement_stats(chain_store, prev_hash)?)
    } else {
        None
    };
    fold_endorsement_stats(num_validators, prev_stats.as_deref(), &contribution)
}

/// Stores the running endorsement-stats accumulator for `block`, alongside its
/// uncertified chunks. Must be called for every spice block during processing.
pub fn record_spice_endorsement_stats_for_block(
    chain_store_update: &mut ChainStoreUpdate,
    epoch_manager: &dyn EpochManagerAdapter,
    block: &Block,
) -> Result<(), Error> {
    let stats = compute_spice_endorsement_stats(
        chain_store_update.chain_store(),
        epoch_manager,
        block.header().epoch_id(),
        block.header().prev_hash(),
    )?;
    let mut store_update = chain_store_update.chain_store().store_ref().store_update();
    store_update.insert_ser(
        DBCol::spice_endorsement_stats(),
        block.header().hash().as_ref(),
        &stats,
    );
    chain_store_update.merge(store_update);
    Ok(())

#### Step 3: state sync

After headers are synced, the node downloads the full state snapshot at a
sync hash (near the beginning of the most recent epoch). For each shard the
node tracks, it downloads state parts from peers and assembles the full state.

State sync creates a gap in the chain data on this node — blocks between the
epoch sync boundary and the sync point are not processed. This is fine for
non-archival nodes, as that data would be garbage collected after a few epochs
anyway.

This step never runs on archival nodes — they need the complete history and
cannot have gaps.
