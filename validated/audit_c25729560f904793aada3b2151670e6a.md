### Title
Missing Zero Address Check for `_sequencer` in `Endpoint.initialize()` Permanently Disables Sequencer Transaction Submission — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.initialize()` assigns `sequencer = _sequencer` with no zero address guard. If `address(0)` is passed, the `sequencer` state variable is permanently set to zero. Because `submitTransactionsChecked` enforces `require(msg.sender == sequencer)` and no `setSequencer` setter exists in `Endpoint.sol`, the primary sequencer-gated transaction submission path is permanently and irrecoverably disabled.

---

### Finding Description

`Endpoint.initialize()` accepts `_sequencer` and stores it directly: [1](#0-0) 

```solidity
function initialize(
    address _sanctions,
    address _sequencer,
    address _offchainExchange,
    IClearinghouse _clearinghouse,
    address _verifier,
    address _endpointTx
) external initializer {
    __Ownable_init();
    __EIP712_init("Nado", "0.0.1");
    sequencer = _sequencer;   // ← no zero address check
    ...
```

The `sequencer` variable is declared in `EndpointStorage`: [2](#0-1) 

`submitTransactionsChecked` enforces the sequencer identity check: [3](#0-2) 

```solidity
require(msg.sender == sequencer);
```

There is no `setSequencer` function anywhere in `Endpoint.sol`. The only other submission path, `submitTransactionsCheckedWithGasLimit`, always terminates via `verifier.revertGasInfo` and is used exclusively for gas estimation, not live transaction processing. [4](#0-3) 

---

### Impact Explanation

If `_sequencer` is `address(0)` at initialization:

- `sequencer` is permanently `address(0)`.
- `submitTransactionsChecked` requires `msg.sender == address(0)`, which is impossible for any EOA or contract.
- No recovery path exists within `Endpoint.sol` (no owner-callable setter for `sequencer`).
- The protocol's primary fast-path transaction processing is permanently disabled. Users are limited to slow-mode transactions only, which have a mandatory multi-day delay and cover a restricted subset of operations.

The corrupted state is: `sequencer == address(0)` → `submitTransactionsChecked` permanently uncallable → all sequencer-batched transactions (liquidations, settlements, price updates, fee claims, etc.) permanently blocked.

---

### Likelihood Explanation

Low. This requires the deployer to pass `address(0)` as `_sequencer` during the one-time `initialize()` call. This is an accidental deployment mistake, not an attacker-controlled path. However, the consequence is irreversible without a full contract upgrade, matching the severity class of the referenced report's `initialize()` finding.

---

### Recommendation

Add a zero address guard for `_sequencer` (and analogously for `_offchainExchange`, `_verifier`, and `_endpointTx`) in `Endpoint.initialize()`:

```solidity
require(_sequencer != address(0), "sequencer cannot be zero address");
require(_offchainExchange != address(0), "offchainExchange cannot be zero address");
require(_verifier != address(0), "verifier cannot be zero address");
require(_endpointTx != address(0), "endpointTx cannot be zero address");
```

Additionally, consider adding an owner-callable `setSequencer` function with a zero address check to allow recovery without a full upgrade.

---

### Proof of Concept

1. Deployer calls `Endpoint.initialize(sanctions, address(0), offchainExchange, clearinghouse, verifier, endpointTx)`.
2. `sequencer` is set to `address(0)`.
3. Any call to `submitTransactionsChecked(idx, transactions, e, s, signerBitmask)` hits `require(msg.sender == sequencer)` → `require(msg.sender == address(0))` → always reverts.
4. No function in `Endpoint.sol` allows the owner to update `sequencer`.
5. The sequencer submission path is permanently bricked; the protocol can only process slow-mode transactions. [5](#0-4) [6](#0-5)

        uint64 idx,
        bytes[] calldata transactions,
        uint256 gasLimit
    ) external {
        uint256 initialGas = gasleft();
        validateSubmissionIdx(idx);
        for (uint256 i = 0; i < transactions.length; i++) {
            bytes calldata transaction = transactions[i];
            processTransaction(transaction);
            uint256 gasUsed = initialGas - gasleft();
            if (gasUsed > gasLimit) {
                verifier.revertGasInfo(i, gasUsed);
            }
        }
        verifier.revertGasInfo(transactions.length, initialGas - gasleft());
    }
