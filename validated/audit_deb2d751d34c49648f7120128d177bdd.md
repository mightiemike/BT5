### Title
Checkpoint Advancement Before Pool Balance Check in `processEventsAndClaimBonus` Causes Permanent Bonus Loss When `topUpBaseRewards` Drains Pool in Same Block — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`, `x/tieredrewards/keeper/abci.go`)

---

### Summary

`topUpBaseRewards` (BeginBlocker) can drain the entire `RewardsPoolName` in a single block. Later in the same block, when `BeforeRedelegationSlashed` fires and calls `processEventsAndClaimBonus`, the function advances `LastBonusAccrual`, `LastEventSeq`, and decrements event reference counts in storage **before** it checks pool balance. When `ErrInsufficientBonusPool` is returned, `slashRedelegationPosition` swallows the error and persists the position with the already-advanced checkpoints. The position owner permanently loses their accrued bonus with no recourse.

---

### Finding Description

**Step 1 — `topUpBaseRewards` drains the pool**

In `abci.go` lines 105–113, when `poolBalance.Amount < shortFallAmount`, the entire pool is drained:

```go
if poolBalance.Amount.LT(shortFallAmount) {
    topUpAmount = poolBalance.Amount
}
err = k.bankKeeper.SendCoinsFromModuleToModule(ctx, types.RewardsPoolName, distributiontypes.ModuleName, ...)
``` [1](#0-0) 

This runs unconditionally in `BeginBlock` before any transaction processing or slashing module hooks.

**Step 2 — `processEventsAndClaimBonus` mutates state before the pool check**

In `claim_rewards.go`, the function:
- Walks all pending events, calling `pos.UpdateLastEventSeq(entry.Seq)` and `k.decrementEventRefCount(...)` (storage write) for each — lines 172–199
- Calls `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` advancing `LastBonusAccrual` — line 215
- Calls `pos.UpdateLastKnownBonded(bonded)` — line 217
- **Only then** calls `sufficientBonusPoolBalance` — line 230 [2](#0-1) 

When `ErrInsufficientBonusPool` is returned at line 230–232, all checkpoint fields on the in-memory `pos` pointer have already been advanced, and event reference count decrements have already been committed to the store.

**Step 3 — Error is swallowed in `slashRedelegationPosition`**

In `slash.go` lines 54–64:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
    } else {
        return err
    }
}
``` [3](#0-2) 

Execution continues with `pos` carrying the already-advanced checkpoints.

**Step 4 — Position is persisted with advanced checkpoints**

For a partial slash (the common case), `setPositionWithState(ctx, pos, nil)` is called at line 77, persisting the position with the advanced `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded`. [4](#0-3) 

The owner can never recover the lost bonus: the next `processEventsAndClaimBonus` call will start from the advanced checkpoint, skipping the period for which bonus was computed but never paid.

**Step 5 — `sufficientBonusPoolBalance` reads the post-drain pool**

`sufficientBonusPoolBalance` reads the live pool balance at call time:

```go
poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
if !poolBalance.IsAllGTE(bonus) {
    return errorsmod.Wrapf(types.ErrInsufficientBonusPool, ...)
}
``` [5](#0-4) 

After `topUpBaseRewards` has drained the pool, this check always fails for any non-zero bonus.

---

### Impact Explanation

For every redelegation slash that occurs in a block where `topUpBaseRewards` has drained the pool:

- The position owner's accrued bonus (computed correctly from shares × tokensPerShare × bonusApy × duration) is permanently forfeited.
- The `RewardsPoolName` balance flows entirely into `distribution` (via `topUpBaseRewards`) without paying owed bonuses.
- Event reference counts are decremented as if the bonus was paid, so events may be garbage-collected, making the loss irrecoverable even if the pool is later replenished.
- This is a direct accounting flaw: value that was owed to position owners leaves `RewardsPoolName` into `distribution` without the corresponding bonus payment.

---

### Likelihood Explanation

This does not require attacker control. It occurs naturally whenever:

1. The pool balance is less than the per-block shortfall (a normal operational condition as the pool depletes over time).
2. A redelegation slash occurs in the same block (a normal chain event for any misbehaving validator).

The `TargetBaseRewardsRate` does not need to be set to an extreme value — any rate that produces a shortfall larger than the remaining pool balance triggers the drain. The pool running low is an expected operational state.

---

### Recommendation

Move the `sufficientBonusPoolBalance` check (and the `bondDenom` fetch) to **before** the event-processing loop and checkpoint advancement in `processEventsAndClaimBonus`. If the pool is insufficient, return `ErrInsufficientBonusPool` without modifying any checkpoint fields or decrementing any event reference counts. This ensures the position owner can retry after the pool is replenished.

Alternatively, perform the pool balance check at the start of `slashRedelegationPosition` before calling `processEventsAndClaimBonus`, and skip the entire bonus settlement (without advancing checkpoints) if the pool is empty.

---

### Proof of Concept

1. Fund `RewardsPoolName` with an amount slightly less than one block's shortfall.
2. Set `TargetBaseRewardsRate` such that `topUpBaseRewards` will drain the entire pool in the next block.
3. Create a position with a non-zero accrued bonus (let time pass with the validator bonded).
4. Submit a `MsgSubmitEvidence` (or trigger downtime slashing) for the validator in the same block.
5. Observe: `BeginBlock` → `topUpBaseRewards` drains pool → tx processing → `BeforeRedelegationSlashed` → `processEventsAndClaimBonus` → `ErrInsufficientBonusPool` swallowed → `setPositionWithState` persists advanced `LastBonusAccrual`.
6. Assert: position owner received zero bonus despite non-zero accrual; `LastBonusAccrual` has advanced past the accrual period; pool is zero; distribution received the full pool amount.

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
