### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Drain from Direct Deposit Accounts — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without using `safeTransferFrom` and without checking the return value. If the USDC token at the hardcoded address returns `false` on a failed transfer instead of reverting, the function continues execution and drains usdcE from the target DDA to the caller — with no USDC provided in exchange. The function is callable by any unprivileged user on chain 57073 (Ink).

---

### Finding Description

`replaceUsdcEWithUsdc` is an `external` function with no `onlyOwner` guard — only a `block.chainid == 57073` check. Its intended logic is a 1:1 swap: receive USDC from the caller, deposit it into the DDA, then return the equivalent usdcE to the caller.

The execution sequence is:

```
1. balance = usdcE.balanceOf(directDepositV1)
2. IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)  ← unchecked
3. DirectDepositV1(directDepositV1).withdraw(usdcE)                     ← drains usdcE to ContractOwner
4. IERC20Base(usdcE).safeTransfer(msg.sender, balance)                  ← sends usdcE to caller
```

Step 2 uses a raw `.transferFrom()` call on the `IERC20Base` interface — not the `ERC20Helper.safeTransferFrom` wrapper used elsewhere in the codebase. The return value is silently discarded. If the USDC token returns `false` on failure (non-reverting failure path), steps 3 and 4 execute unconditionally, transferring usdcE out of the DDA to the caller without any USDC being deposited.

The rest of the codebase consistently uses `ERC20Helper.safeTransfer` / `safeTransferFrom` for all token movements. This call is the sole exception. [1](#0-0) 

The `ERC20Helper` library — used everywhere else — correctly handles non-returning tokens via low-level `.call()` with `data.length == 0 || abi.decode(data, (bool))`: [2](#0-1) 

The raw `transferFrom` call at line 616 bypasses this protection entirely. [3](#0-2) 

---

### Impact Explanation

**Impact: High.**

If the `transferFrom` silently returns `false`:
- No USDC enters the DDA.
- usdcE is fully withdrawn from the DDA to `ContractOwner` (step 3).
- usdcE is transferred to the attacker (step 4).

The attacker receives the full usdcE balance of any targeted DDA at zero cost. The DDA owner loses their usdcE with no USDC compensation. The corrupted asset delta is exact: `usdcE.balanceOf(directDepositV1)` is zeroed, and the attacker's usdcE balance increases by the same amount.

---

### Likelihood Explanation

**Likelihood: Low.**

The USDC token address is hardcoded to `0x2D270e6886d130D724215A266106e6832161EAEd` on chain 57073 (Ink). Standard Circle USDC reverts on failure rather than returning `false`, which would prevent silent failure. However:

1. The specific token at that address on Ink may not be canonical Circle USDC — its exact behavior is not verifiable from the contract code alone.
2. The code defect (unchecked return value diverging from the rest of the codebase) is concrete and present regardless of current token behavior.
3. The function is callable by any unprivileged user with no additional barriers.

---

### Recommendation

Replace the raw `transferFrom` call with `ERC20Helper.safeTransferFrom`, consistent with all other token transfer operations in the codebase:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ContractOwner` already imports and uses `ERC20Helper` via `using ERC20Helper for IERC20Base` at line 24, so no additional imports are needed. [4](#0-3) 

---

### Proof of Concept

1. Identify a DDA on chain 57073 with a non-zero usdcE balance (e.g., `directDepositV1Address[victim_subaccount]`).
2. Call `ContractOwner.replaceUsdcEWithUsdc(victim_subaccount)` without having approved `ContractOwner` to spend USDC (or with a USDC token that returns `false` on failure).
3. If `transferFrom` returns `false` silently:
   - `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes, pulling all usdcE from the DDA into `ContractOwner`.
   - `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` executes, sending all usdcE to the attacker.
4. The attacker receives the full usdcE balance of the victim DDA. No USDC was deposited. [1](#0-0)

```text
        require(directDepositV1 != address(0), "no dda");

        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
