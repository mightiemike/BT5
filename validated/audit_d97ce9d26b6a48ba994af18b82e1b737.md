### Title
Unvalidated `subaccount` Parameter in `depositCollateralWithReferral` Allows Attacker to Redirect Victim's Token Transfer to Attacker-Controlled Subaccount — (`core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` is a `public` function that accepts an arbitrary `bytes32 subaccount` without verifying that it is derived from `msg.sender`. Tokens are pulled from `msg.sender`, but the collateral credit is applied to the caller-supplied `subaccount`. A malicious contract can exploit this to drain a victim's pre-approved tokens into an attacker-controlled subaccount.

---

### Finding Description

`depositCollateral` (the safe entry point) correctly binds the subaccount to the caller:

```solidity
bytes32 subaccount = bytes32(abi.encodePacked(msg.sender, subaccountName));
``` [1](#0-0) 

`depositCollateralWithReferral`, however, is `public` and accepts an arbitrary `bytes32 subaccount` with no equivalent binding check:

```solidity
function depositCollateralWithReferral(
    bytes32 subaccount,
    uint32 productId,
    uint128 amount,
    string memory
) public {
    require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);
    address sender = address(bytes20(subaccount));
    requireUnsanctioned(msg.sender);
    requireUnsanctioned(sender);
    ...
    handleDepositTransfer(
        IERC20Base(spotEngine.getToken(productId)),
        msg.sender,          // ← tokens pulled from caller
        uint256(amount)
    );
    slowModeTxs[...] = SlowModeTx({
        sender: sender,      // ← credit goes to bytes20(subaccount), not msg.sender
        tx: abi.encodePacked(...DepositCollateral({ sender: subaccount, ... }))
    });
``` [2](#0-1) 

There is no `require(address(bytes20(subaccount)) == msg.sender)` guard anywhere in this function. The only identity check is a sanctions screen on both `msg.sender` and `address(bytes20(subaccount))`, which does not establish ownership. [3](#0-2) 

---

### Impact Explanation

Alice's ERC-20 tokens are transferred to the Endpoint contract (`handleDepositTransfer` pulls from `msg.sender`), but the resulting `DepositCollateral` slow-mode transaction credits an attacker-controlled subaccount (`eveSubaccount`). Once the slow-mode transaction executes, Eve's on-chain subaccount balance increases by the stolen amount while Alice's wallet is debited. This is a direct, irreversible asset transfer: Alice loses collateral, Eve gains it. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** Two preconditions must hold:

1. Alice has a pre-existing ERC-20 approval for the Endpoint contract — standard for any user who has ever deposited collateral.
2. Alice calls a malicious contract (e.g., a fake router, aggregator, or airdrop claimer) that internally calls `depositCollateralWithReferral(eveSubaccount, productId, amount)`.

This is the same attacker model as the TradeCallee report: a malicious proxy contract sits between the victim and the protocol. Because `depositCollateralWithReferral` is `public` (not `external` with a `msg.sender == owner` guard), any contract in the call stack can invoke it on Alice's behalf. No privileged access, governance capture, or key compromise is required.

---

### Recommendation

**Short term:** Add a binding check at the top of `depositCollateralWithReferral`:

```solidity
require(
    address(bytes20(subaccount)) == msg.sender,
    ERR_UNAUTHORIZED
);
```

This mirrors the invariant already enforced by `depositCollateral` and closes the mismatch between who pays and who is credited.

**Long term:** Audit all `public` entry points that accept caller-supplied identity parameters (`subaccount`, `sender`, `recipient`) and verify that each one either derives the identity from `msg.sender` or explicitly validates ownership before transferring assets.

---

### Proof of Concept

```
1. Alice approves Endpoint to spend 1000 USDC (standard deposit flow).

2. Eve deploys MaliciousRouter:
   function steal(address victim, bytes32 eveSubaccount) external {
       // msg.sender here is Alice (victim)
       IEndpoint(endpoint).depositCollateralWithReferral(
           eveSubaccount,   // Eve's subaccount
           USDC_PRODUCT_ID,
           1000e6
       );
   }

3. Eve tricks Alice into calling MaliciousRouter.steal(...).

4. Inside depositCollateralWithReferral:
   - handleDepositTransfer pulls 1000 USDC from Alice (msg.sender).
   - A SlowModeTx is queued: sender = address(bytes20(eveSubaccount)), 
     DepositCollateral.sender = eveSubaccount.

5. After SLOW_MODE_TX_DELAY, anyone executes the slow-mode tx.
   SpotEngine credits eveSubaccount with 1000 USDC of collateral.

6. Alice has lost 1000 USDC; Eve has gained 1000 USDC of tradeable collateral.
``` [2](#0-1)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });
        slowModeConfig = _slowModeConfig;
    }
