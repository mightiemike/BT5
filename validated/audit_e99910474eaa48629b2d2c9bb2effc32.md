### Title
Caller-Controlled `confirmations` With No Minimum Floor Allows Zero-Confirmation Transaction Verification — (`contract/src/lib.rs`)

### Summary
Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` field with no minimum value enforced. Any unprivileged NEAR caller — including a recipient contract consuming verification results — can pass `confirmations = 0` or `confirmations = 1` and receive a `true` result for a transaction that has no meaningful reorg protection. This is the direct analog of the reported vulnerability: instead of a fixed but insufficient depth, the contract here imposes *no* depth floor at all, delegating the entire safety decision to the caller.

---

### Finding Description

`ProofArgs` and `ProofArgsV2` both carry a `confirmations: u64` field that is fully caller-controlled: [1](#0-0) 

Inside `verify_transaction_inclusion`, the only bounds check on this value is an *upper* cap against `gc_threshold`: [2](#0-1) 

The confirmation-depth check that follows is: [3](#0-2) 

When `args.confirmations = 0`, the right-hand side of `>=` is `0`, and since the left-hand side is a `u64` expression that is always `≥ 0`, the check is **unconditionally true** — the function proceeds to Merkle verification and returns `true` for a transaction with zero confirmation depth. When `args.confirmations = 1`, only the block containing the transaction itself is required, providing no reorg protection whatsoever.

`verify_transaction_inclusion_v2` delegates directly to `verify_transaction_inclusion` after its coinbase-proof check, so it inherits the same flaw: [4](#0-3) 

---

### Impact Explanation

A recipient contract (e.g., a cross-chain bridge or atomic-swap settlement contract) that calls `verify_transaction_inclusion_v2` with `confirmations = 0` or `confirmations = 1` will receive `true` for a transaction that is still trivially reorg-able. If the contract releases funds or mints tokens upon a `true` result, an attacker who controls mining power (or exploits a natural short reorg) can:

1. Broadcast a BTC transaction in block B.
2. Have the relayer submit block B to the NEAR contract.
3. Call `verify_transaction_inclusion_v2` with `confirmations = 1`; the contract returns `true`.
4. The consumer contract releases assets.
5. The attacker mines a competing chain that omits the original transaction; the relayer updates the NEAR contract to the new tip.
6. The original transaction is no longer on the main chain, but assets have already been released.

The contract itself provides no protection against this because it enforces no minimum confirmation depth. The entire security burden is silently shifted to every downstream consumer.

---

### Likelihood Explanation

The entry path requires no privileged role: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, pausable-only methods callable by any NEAR account or contract. The `confirmations` field is a plain integer in the Borsh-serialized argument struct with no validation beyond the upper-bound `gc_threshold` check. Any consumer that passes a low value — whether by mistake or by adversarial intent — triggers the vulnerable path. Bitcoin 1-block reorgs occur regularly; a deliberate selfish-mining or fee-sniping attack makes even a 1-confirmation window exploitable.

---

### Recommendation

Enforce a protocol-level minimum confirmation depth inside `verify_transaction_inclusion`. A reasonable floor for Bitcoin mainnet is 6 confirmations (the conventional economic finality threshold); for higher-value settlements, a larger value (e.g., 30–100) is appropriate. The minimum should be a contract-level constant or a governance-controlled parameter, not a caller-supplied value:

```rust
const MIN_CONFIRMATIONS: u64 = 6;

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    format!("Confirmations must be at least {}", MIN_CONFIRMATIONS)
);
```

This check should be placed before the existing `gc_threshold` upper-bound check in `verify_transaction_inclusion`. [5](#0-4)

### Proof of Concept

```
// Attacker-controlled consumer contract on NEAR
// Step 1: relayer has submitted block B (height H) to the light client.
// Step 2: attacker calls verify_transaction_inclusion_v2 with confirmations=0.

let args = ProofArgsV2 {
    tx_id: <attacker_tx_hash>,
    tx_block_blockhash: <block_B_hash>,
    tx_index: 0,
    merkle_proof: <valid_merkle_proof>,
    coinbase_tx_id: <coinbase_hash>,
    coinbase_merkle_proof: <valid_coinbase_proof>,
    confirmations: 0,   // ← no minimum floor enforced
};

// Contract evaluates:
//   (tip_height - H) + 1 >= 0  →  always true for u64
// Returns: true
// Consumer contract releases funds.

// Step 3: attacker mines a reorg that excludes the transaction.
// Relayer updates NEAR contract to new tip.
// Transaction is gone from main chain; funds already released.
```

The `confirmations = 0` path is reachable by any unprivileged NEAR caller with a valid Merkle proof for a block that is currently on the main chain. [3](#0-2) [6](#0-5)

```rust
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );
```
