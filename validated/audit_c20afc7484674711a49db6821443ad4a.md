### Title
Deprecated `verify_transaction_inclusion` (v1) Accepts Internal Merkle Node Hash as Valid Transaction, Enabling False SPV Proof Confirmation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is a publicly callable NEAR contract method with no access control beyond the pause gate. It passes the caller-supplied `tx_id` directly into `compute_root_from_merkle_proof` without validating that it is a leaf node (real transaction hash). An attacker can supply any internal Merkle tree node hash as `tx_id` with a crafted shorter proof, causing the function to reconstruct the correct `merkle_root` and return `true` for a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` and documented with an explicit warning, but it remains a live, unrestricted public method on the contract: [1](#0-0) 

The sole verification step is: [2](#0-1) 

`compute_root_from_merkle_proof` treats its first argument as an opaque 32-byte starting hash and iterates upward through the proof without any check that the input is a leaf: [3](#0-2) 

The internal `compute_hash` function concatenates two 32-byte values and double-SHA256s them — identical to how Bitcoin computes every internal node: [4](#0-3) 

Because the function is symmetric with respect to leaf vs. internal node, any internal node `N` at depth `d` from the root can be used as `tx_id` with a proof of length `d` (instead of the full tree depth), and the computed root will equal the real `merkle_root`.

The contract's own documentation acknowledges this: [5](#0-4) 

`verify_transaction_inclusion_v2` mitigates this by requiring a coinbase proof at index 0, which forces the proof length to match the full tree depth: [6](#0-5) 

But v1 is never removed, never access-controlled, and never paused by default, so any caller can bypass v2 entirely by calling v1 directly.

---

### Impact Explanation

An attacker who knows the Merkle tree of any block already accepted by the contract can:

1. Pick any internal node `N` at depth `d` (e.g., `H01 = SHA256d(T0 || T1)` for a 4-tx block).
2. Construct a proof of length `d` from `N` to the root (e.g., `[H23]`).
3. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index = 0`, `merkle_proof = [H23]`.
4. The contract returns `true` despite no real transaction with that hash existing.

Any downstream bridge or dApp that calls v1 and gates asset release on its return value is directly exploitable. The `ProofArgs` struct accepts arbitrary caller-supplied `tx_id` with no type-level or runtime leaf-node guard: [7](#0-6) 

---

### Likelihood Explanation

- `verify_transaction_inclusion` is a public, non-`#[private]` method callable by any NEAR account.
- The Merkle tree of every submitted Bitcoin block is publicly derivable from on-chain data.
- No special privilege, key compromise, or social engineering is required.
- The attack requires only knowledge of the block's transaction set and a single contract call.

---

### Recommendation

1. **Remove or gate v1**: Apply `#[private]` to `verify_transaction_inclusion`, or add a role-based access control guard, so it cannot be called by arbitrary external accounts. Alternatively, remove it entirely if no internal caller depends on it.
2. **Enforce v2 as the only public path**: `verify_transaction_inclusion_v2` already implements the correct coinbase-proof check. It should be the sole externally callable verification method.
3. **Do not rely on documentation warnings as a security boundary**: The `# Warning` comment and `#[deprecated]` attribute do not prevent on-chain calls.

---

### Proof of Concept

Given a block with 4 transactions `[T0, T1, T2, T3]` and Merkle tree:

```
Level 0 (leaves): T0,  T1,  T2,  T3
Level 1:          H01 = SHA256d(T0||T1),  H23 = SHA256d(T2||T3)
Level 2 (root):   R   = SHA256d(H01||H23)
```

Attacker constructs:
- `tx_id        = H01`  (internal node, not a real transaction)
- `tx_index     = 0`
- `merkle_proof = [H23]`  (length 1, not the full depth-2 proof)

Call path:
```
verify_transaction_inclusion({ tx_id: H01, tx_index: 0, merkle_proof: [H23], tx_block_blockhash: <real block>, confirmations: 1 })
  → compute_root_from_merkle_proof(H01, 0, [H23])
      current_hash = H01, position = 0 (even)
      → compute_hash(H01, H23) = SHA256d(H01 || H23) = R
  → R == header.merkle_root  ✓
  → returns true
```

The contract confirms inclusion of `H01` — a hash that corresponds to no real Bitcoin transaction — enabling any downstream system gating on this result to release assets falsely.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
