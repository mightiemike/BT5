Audit Report

## Title
Router-Mediated Swaps Bypass `SwapAllowlistExtension` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool::swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`, so when `MetricOmmSimpleRouter` calls `pool.swap()`, the pool's `msg.sender` is the router. `SwapAllowlistExtension::beforeSwap` checks `allowedSwapper[pool][sender]`, which resolves to the router address rather than the original user. Any unprivileged user can bypass the allowlist by routing through a router that is itself allowlisted on the restricted pool.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the hook.**

`MetricOmmPool::swap` calls:

```solidity
_beforeSwap(
  msg.sender,   // whoever called pool.swap()
  recipient,
  ...
);
``` [1](#0-0) 

**Step 2 — `ExtensionCalling::_beforeSwap` forwards that `sender` to every configured extension.** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension::beforeSwap` checks `allowedSwapper[msg.sender][sender]`**, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

**Step 4 — The router calls `pool.swap()` directly**, storing the original caller only in transient callback context for payment, never forwarding it as the swap initiator:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ...);
``` [4](#0-3) 

The pool sees `msg.sender = router`, so the extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`. The original caller's identity is never verified by the allowlist.

## Impact Explanation

If a pool admin allowlists the router (necessary for any user to swap via the router on a restricted pool), every unprivileged user can bypass the allowlist by calling `MetricOmmSimpleRouter::exactInputSingle` (or any other `exact*` entry point). The core invariant — that only approved addresses may swap on a restricted pool — is fully nullified. This constitutes broken core pool functionality and unauthorized execution of swaps, matching the "Admin-boundary break" and "Broken core pool functionality" allowed impacts.

## Likelihood Explanation

The vulnerability is structural and requires no special attacker capability. Any pool using `SwapAllowlistExtension` that expects users to interact via the router faces this issue. The pool admin has no safe configuration: allowlisting the router opens the bypass; not allowlisting it breaks the router for all users. The exploit is repeatable by any address calling the public router.

## Recommendation

The router must forward the original caller's identity so the extension can check it. Two approaches:

1. **Pass original caller via `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap`. The extension reads and verifies it (with a pool-level trust policy for the router).
2. **Dedicated `swapper` override parameter**: Add an explicit `swapper` field to the pool's `swap` interface that the router populates with `msg.sender`, and the pool passes it as `sender` to hooks instead of its own `msg.sender`.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // only alice is allowed
3. Pool admin calls setAllowedToSwap(pool, router, true)  // router must be allowed for UX
4. Bob (not allowlisted) calls:
     MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
5. Pool calls _beforeSwap(msg.sender=router, ...)
6. Extension checks allowedSwapper[pool][router] → true → passes
7. Bob's swap executes despite not being in the allowlist.

Assert: Bob received output tokens. The allowlist invariant is violated.
```
