### Title
SwapAllowlistExtension Checks Router Address as Swapper Identity, Allowing Any EOA to Bypass Per-User Allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` as seen by the pool. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the originating EOA. A pool admin who allowlists the router address to enable router-mediated swaps inadvertently grants unrestricted swap access to every EOA that calls the router, completely defeating the per-user allowlist invariant.

---

### Finding Description

The call chain is:

```
EOA → MetricOmmSimpleRouter.exactInputSingle(msg.sender=EOA)
        → pool.swap(...)           [pool sees msg.sender = router]
            → _beforeSwap(msg.sender=router, ...)
                → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                    → checks: allowedSwapper[pool][router]   ← NOT allowedSwapper[pool][EOA]
```

**Step 1** — `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly with no forwarding of the originating EOA: [1](#0-0) 

**Step 2** — `MetricOmmPool.swap` passes `msg.sender` (the router) as the `sender` argument to `_beforeSwap`: [2](#0-1) 

**Step 3** — `ExtensionCalling._beforeSwap` forwards that `sender` value unchanged to the extension: [3](#0-2) 

**Step 4** — `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [4](#0-3) 

The check `allowedSwapper[pool][EOA]` is **never evaluated** for any router-mediated swap. The extension has no mechanism to recover the originating EOA.

This creates an inescapable design trap for pool admins:

| Admin intent | Admin action | Actual result |
|---|---|---|
| Allow specific EOAs via router | Allowlist the router | **All EOAs bypass the allowlist** |
| Allow specific EOAs directly | Allowlist those EOAs | Router-mediated swaps from those EOAs are **blocked** |

There is no configuration that achieves per-user gating for router-mediated swaps.

---

### Impact Explanation

Any EOA — including those explicitly excluded from `allowedSwapper` — can execute swaps against a pool's LP liquidity by routing through `MetricOmmSimpleRouter`, provided the router address is in the allowlist. LP funds are exposed to unauthorized traders. The pool curation invariant ("only allowlisted end-users may trade") is broken for the entire router-mediated swap surface, which is the primary intended usage path for end users.

---

### Likelihood Explanation

A pool admin who deploys a pool with `SwapAllowlistExtension` and wants to support the standard router flow has no choice but to allowlist the router. This is the natural, expected configuration. The bypass is then automatic for any EOA — no special knowledge or exploit tooling is required.

---

### Recommendation

The `sender` identity passed through the hook chain must reflect the originating EOA, not the intermediary router. Options:

1. **Router-side**: Store the originating `msg.sender` in transient storage and expose it via a standard interface (e.g., `IMetricOmmSimpleRouter.msgSender()`); the extension reads it during the hook call.
2. **Extension-side**: Check `recipient` instead of `sender` when `sender` is a known router, or require the router to pass the EOA via `extensionData` with a verifiable signature.
3. **Pool-side**: Add a separate `originator` field to the swap call that the router populates with `msg.sender`.

---
