Audit Report

## Title
Unspent ETH Stranded in Router When `priceLimitX64` Causes Partial Exact-Input Swap - (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

## Summary

`exactInputSingle` is `payable` and, when `tokenIn == WETH`, uses the router's native ETH balance to satisfy the swap callback. When a `priceLimitX64` causes the pool to partially consume the specified input, the callback pays only the actually consumed amount, leaving the remainder permanently in the router. No refund is issued after the swap, and the stranded ETH is immediately drainable by any address via the public `refundETH()`.

## Finding Description

**Step 1 – ETH enters the router.** `exactInputSingle` is `payable`; a user selling native ETH sends `msg.value` with the call. [1](#0-0) 

**Step 2 – Pool partially consumes input.** Both `_swapToken0ForToken1SpecifiedInput` and `_swapToken1ForToken0SpecifiedInput` break out of the swap loop early when the price limit is reached, returning only the actually consumed scaled amount rather than the full `amountSpecified`. [2](#0-1) [3](#0-2) 

This is confirmed by the pool's own invariant tests, which assert that consumed input can be strictly less than specified when a limit is hit. [4](#0-3) 

**Step 3 – Callback pays only the partial amount.** `_justPayCallback` calls `pay()` with `extractPositiveAmount(amount0Delta, amount1Delta)` — the real consumed delta, not `params.amountIn`. [5](#0-4) 

**Step 4 – `pay()` wraps ETH from the router's own balance.** When `token == WETH` and `address(this).balance >= value`, `pay()` wraps exactly `value` wei from the router's ETH balance and forwards WETH to the pool. The surplus ETH is never touched. [6](#0-5) 

**Step 5 – No refund after the swap.** After `pool.swap` returns, `exactInputSingle` only checks `amountOut < params.amountOutMinimum` and calls `_clearExpectedCallbackPool()`. There is no ETH refund. [7](#0-6) 

**Step 6 – Stranded ETH is publicly drainable.** `refundETH()` is an unrestricted external function that sends the router's entire ETH balance to `msg.sender`, not to the original swapper. [8](#0-7) 

## Impact Explanation

Direct loss of user principal. The unconsumed ETH fraction (proportional to how far the price limit is from the current price and the trade size) is permanently removed from the user's control the moment the transaction confirms. Any third party — including MEV bots monitoring the mempool — can immediately call `refundETH()` and receive the stranded funds. This meets the Sherlock critical/high threshold for direct loss of user principal with no privileged access required.

## Likelihood Explanation

`priceLimitX64` is a standard, documented parameter of `ExactInputSingleParams`. Any user who sets a non-trivial price limit while selling native ETH triggers the bug. No malicious setup, non-standard token behavior, or privileged role is required — only a normal `exactInputSingle{value: X}` call with `tokenIn == WETH` and a `priceLimitX64` that the pool's swap loop reaches before exhausting the full input.

## Recommendation

After `pool.swap` returns, compute the unspent ETH and refund it to `msg.sender`:

```solidity
// inside exactInputSingle, after the swap call:
int128 actualIn = MetricOmmSwapResults.extractAmountIn(
    params.zeroForOne, amount0Delta, amount1Delta
);
if (address(params.tokenIn) == WETH && msg.value > uint128(actualIn)) {
    _transferETH(msg.sender, msg.value - uint128(actualIn));
}
```

Alternatively, document — with an on-chain enforcement mechanism — that callers **must** append a `refundETH()` call via `multicall` whenever a price limit is set and ETH is the input token. The existing test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` already demonstrates this pattern for the `msg.value > amountIn` case but does not cover the price-limit partial-fill scenario. [9](#0-8) 

## Proof of Concept

```
1. ETH/TKN pool is at price P0. Alice calls:
     router.exactInputSingle{value: 1 ether}(ExactInputSingleParams{
       tokenIn:          WETH,
       tokenOut:         TKN,
       pool:             ethTknPool,
       recipient:        alice,
       amountIn:         1e18,
       amountOutMinimum: 0,
       priceLimitX64:    <price between current and upper bin boundary>,
       ...
     });

2. _swapToken0ForToken1SpecifiedInput (or token1-for-token0 counterpart) exits
   the while loop early at the price limit; only 0.6e18 of input is consumed.
   The pool fires metricOmmSwapCallback with amount0Delta = 0.6e18 (positive).

3. _justPayCallback calls pay(WETH, alice, pool, 0.6e18).
   pay() sees address(this).balance (1 ether) >= 0.6e18, wraps 0.6 ETH → WETH,
   transfers WETH to pool. Router now holds 0.4 ETH.

4. exactInputSingle returns; amountOut passes the minimum check.
   No refund is issued. 0.4 ETH is stranded in the router.

5. Bob calls router.refundETH() and receives Alice's 0.4 ETH.
```
