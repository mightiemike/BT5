### Title
`CongestionInfoV1::allowed_shard` Stored as `u16` Silently Truncates `ShardId` (u64) Under Dynamic Resharding — (`core/primitives/src/congestion_info.rs`)

---

### Summary

`CongestionInfoV1::allowed_shard` is declared as `u16` and Borsh-serialized into every chunk header. `ShardId` is `u64`. The conversion `ShardId → u16` in `finalize_allowed_shard` and `bootstrap_congestion_info` is a silent wrapping cast (`self.0 as u16`). Once `DynamicResharding` (protocol version 85, stable) allocates shard IDs above 65 535, the stored `allowed_shard` encodes the wrong shard, breaking the congestion-control invariant that exactly one shard may send large receipts to a fully-congested shard.

---

### Finding Description

`CongestionInfoV1` is the Borsh-serialized congestion record carried in every `ShardChunkHeader`:

```rust
// core/primitives/src/congestion_info.rs
pub struct CongestionInfoV1 {
    pub delayed_receipts_gas: u128,
    pub buffered_receipts_gas: u128,
    pub receipt_bytes: u64,
    pub allowed_shard: u16,   // ← narrowed from ShardId (u64)
}
``` [1](#0-0) 

`finalize_allowed_shard` selects a `ShardId` winner and stores it:

```rust
pub fn finalize_allowed_shard(&mut self, own_shard: ShardId, all_shards: &[ShardId], congestion_seed: u64) {
    let allowed_shard = Self::get_new_allowed_shard(own_shard, all_shards, congestion_seed);
    self.set_allowed_shard(allowed_shard.into());   // ShardId → u16, silent truncation
}
``` [2](#0-1) 

The `Into<u16>` implementation for `ShardId` is an unchecked cast:

```rust
impl Into<u16> for ShardId {
    fn into(self) -> u16 {
        self.0 as u16   // wraps silently for id > 65535
    }
}
``` [3](#0-2) 

`bootstrap_congestion_info` has the same truncation when initialising the field from the shard's own ID:

```rust
allowed_shard: shard_id.into(),   // ShardId → u16
``` [4](#0-3) 

The stored `u16` is later widened back to `ShardId` for comparison:

```rust
pub fn outgoing_size_limit(&self, sender_shard: ShardId) -> u64 {
    if sender_shard == ShardId::from(self.info.allowed_shard()) {
        self.config.outgoing_receipts_big_size_limit
    } else {
        self.config.outgoing_receipts_usual_size_limit
    }
}
``` [5](#0-4) 

`ShardId::from(u16)` zero-extends, so `ShardId::from(truncated_u16) ≠ original_shard_id` whenever the original exceeded 65 535.

The same structural truncation exists in `BandwidthRequest::to_shard: u16`, which is also Borsh-serialized into chunk headers:

```rust
pub struct BandwidthRequest {
    pub to_shard: u16,   // ShardId (u64) narrowed here
    ...
}
// construction:
Ok(Some(BandwidthRequest { to_shard: to_shard.into(), ... }))
``` [6](#0-5) [7](#0-6) 

The bandwidth scheduler reconstructs the receiver shard via `bandwidth_request.to_shard.into()`, recovering the truncated value instead of the real shard ID. [8](#0-7) 

A third related site is `TrieKey::BufferedReceipt`, which hard-asserts `receiving_shard ≤ u16::MAX` and panics otherwise — a node crash rather than silent misbehaviour:

```rust
assert!(receiving_shard <= u16::MAX as u64, "Shard ID too big.");
let receiving_shard: u16 = receiving_shard as u16;
``` [9](#0-8) 

---

### Impact Explanation

`CongestionInfoV1` is part of every `ShardChunkHeader` and is validated by all validators. When `allowed_shard` encodes the wrong shard (due to truncation), every validator independently computes the same wrong value (the computation is deterministic), so consensus is not broken. However, the congestion-control invariant is violated: the shard that is supposed to receive the large-bandwidth slot does not receive it, while a different shard (whose ID happens to share the low 16 bits) does. Under full congestion this means the intended shard makes no progress, while an unintended shard gets the big outgoing limit. The `BandwidthRequest::to_shard` truncation causes the bandwidth scheduler to misroute grants to the wrong shard pair. The `TrieKey::BufferedReceipt` assert causes a node panic (crash) on the first attempt to write a buffered receipt for a shard with ID > 65 535.

---

### Likelihood Explanation

`ProtocolFeature::DynamicResharding` is stable at protocol version 85. `ShardLayoutV3::derive` allocates new shard IDs as `max_existing_id + 1` and `max_existing_id + 2` on every split. Starting from today's mainnet shard IDs (single digits), reaching ID 65 535 requires approximately 32 765 resharding events. At the minimum cooldown of one epoch per resharding, this is a very long horizon. The bug is therefore a latent structural defect rather than an immediately exploitable condition, but the type mismatch is unconditional and no guard prevents it from manifesting once the threshold is crossed.

---

### Recommendation

1. Change `CongestionInfoV1::allowed_shard` from `u16` to `u64` (or introduce a `CongestionInfoV2` with the wider field) and update all serialization, views, and comparison sites.
2. Change `BandwidthRequest::to_shard` from `u16` to `u64` (or introduce `BandwidthRequestV2`).
3. Replace the `assert!` in `TrieKey::BufferedReceipt` with a proper `Result`-returning error so that an oversized shard ID produces a recoverable error rather than a panic.
4. Remove or restrict the `Into<u16> for ShardId` and `Into<u32> for ShardId` impls (both perform silent truncation) and replace them with checked conversions that return `Result` or `Option`.

---

### Proof of Concept

With dynamic resharding enabled, after enough splits the shard layout contains a shard with `id = 65536`. At the next chunk application:

```
finalize_allowed_shard(own_shard=ShardId(65536), all_shards=[...], seed=N)
  → get_new_allowed_shard returns ShardId(65536)
  → set_allowed_shard(ShardId(65536).into())
       = set_allowed_shard(65536u64 as u16)
       = set_allowed_shard(0u16)          ← truncated to 0
```

`CongestionInfoV1::allowed_shard` is serialized as `0x0000` in the chunk header. Every node reads it back as `ShardId::from(0u16) = ShardId(0)`. Shard 0 now receives the big outgoing-receipts limit, while shard 65536 (the intended allowed sender) is throttled to the small limit and cannot drain its buffered receipts, stalling cross-shard progress indefinitely.
