### Title
`ValidResourceBounds` Protobuf Deserialization Silently Converts `AllResources` to `L1Gas` When `l2_gas == 0 && l1_data_gas == 0`, Producing a Wrong Transaction Hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf converter `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` uses a zero-value heuristic to decide between the `L1Gas` and `AllResources` enum variants. When a transaction is submitted with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the gateway computes and stores a hash that includes the `L1_DATA_GAS` field. After the transaction is serialized to protobuf and deserialized by a syncing peer, the converter silently produces `L1Gas(X)` instead of `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The hash function `get_tip_resource_bounds_hash` omits `L1_DATA_GAS` for the `L1Gas` variant, so the recomputed hash diverges from the stored hash. Any code path that recomputes the hash from the deserialized transaction — including `validate_transaction_hash` during P2P block sync — will produce a wrong value.

### Finding Description

**Step 1 — Submission and original hash computation.**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds` directly. The gateway converts it to `InternalRpcInvokeTransactionV3`, which also stores `AllResourceBounds`. The hash is computed via `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(tx.resource_bounds)`. [1](#0-0) 

Inside `get_tip_resource_bounds_hash`, the `AllResources` branch appends a third element — `L1_DATA_GAS_concat` — to the hash preimage: [2](#0-1) 

So for `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the hash preimage is:
`poseidon(tip, L1_GAS_concat(X), L2_GAS_concat(0), L1_DATA_GAS_concat(0))`

**Step 2 — Protobuf serialization and deserialization.**

When the block is propagated via P2P sync, the transaction is serialized to `protobuf::InvokeV3`. The `l1_data_gas` and `l2_gas` fields are both zero. On the receiving side, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies this heuristic: [3](#0-2) 

Because `l1_data_gas.is_zero() && l2_gas.is_zero()`, the converter returns `ValidResourceBounds::L1Gas(l1_gas)` — discarding the information that the original transaction used the `AllResources` variant.

**Step 3 — Hash recomputation produces a divergent value.**

When `get_tip_resource_bounds_hash` is called with `L1Gas(X)`, the `L1Gas` branch emits only two elements: [4](#0-3) 

The hash preimage becomes:
`poseidon(tip, L1_GAS_concat(X), L2_GAS_concat(0))`

This is a **different Poseidon hash** from the original `AllResources` hash, because the `L1_DATA_GAS_concat(0)` element is absent.

**Step 4 — `validate_transaction_hash` fails.**

`validate_transaction_hash` recomputes the hash from the deserialized transaction and checks it against the stored hash. For V3 invoke transactions there are no deprecated hash alternatives, so the only candidate is the recomputed `L1Gas` hash. It will not match the stored `AllResources` hash. [5](#0-4) 

**Trigger condition.** A user submits `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The gateway's stateless validator accepts this (at least one resource bound is non-zero). The transaction is included in a block. Any peer that syncs the block via protobuf will recompute a wrong hash.

### Impact Explanation

The P2P sync path deserializes `InvokeTransactionV3` from protobuf and recomputes the transaction hash for validation. For any transaction with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the recomputed hash will not match the hash stored in the block. This causes syncing nodes to reject a valid block or to store the transaction under a wrong hash, breaking the canonical chain view. This matches: **High — Transaction conversion or signature/hash logic binds the wrong hash or executable payload.**

### Likelihood Explanation

The trigger requires a user to submit a V3 invoke transaction with non-zero `l1_gas` but zero `l2_gas` and zero `l1_data_gas`. The gateway's stateless validator permits this combination (the `valid_l1_gas` test case in the validator test suite explicitly covers it). No privileged access is required. The condition is an ordinary user-controlled input.

### Recommendation

Remove the zero-value heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Instead, add an explicit discriminant field to the `protobuf::ResourceBounds` message (e.g., a boolean `is_all_resources`) so the variant is preserved losslessly across serialization. Alternatively, always deserialize as `AllResources` when `l1_data_gas` is present in the protobuf message (even if zero), and only fall back to `L1Gas` when the field is absent (i.e., `value.l1_data_gas.is_none()`).

The existing comment `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.` confirms the intent to eventually enforce this, but the current `unwrap_or_default` path silently collapses the variant.

### Proof of Concept

1. Construct `RpcInvokeTransactionV3` with `resource_bounds = AllResourceBounds { l1_gas: ResourceBounds { max_amount: 100, max_price_per_unit: 1 }, l2_gas: ResourceBounds::default(), l1_data_gas: ResourceBounds::default() }`.

2. Compute `H_original` via `get_invoke_transaction_v3_hash` (uses `AllResources` → includes `L1_DATA_GAS_concat(0)` in preimage).

3. Serialize to `protobuf::ResourceBounds` with `l1_data_gas = Some(default_zero_limits)`.

4. Deserialize via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`: `l1_data_gas.is_zero() && l2_gas.is_zero()` → returns `L1Gas(l1_gas)`.

5. Compute `H_recomputed` via `get_invoke_transaction_v3_hash` (uses `L1Gas` → omits `L1_DATA_GAS_concat`).

6. Assert `H_original != H_recomputed` — the two Poseidon hashes differ because the preimage lengths differ (3 resource elements vs 2).

7. Call `validate_transaction_hash` with the deserialized transaction and `expected_hash = H_original` → returns `false`, causing the syncing node to reject the transaction. [6](#0-5) [7](#0-6)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
