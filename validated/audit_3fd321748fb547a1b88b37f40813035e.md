### Title
Missing DA-mode validation in consensus transaction conversion path allows OS-divergent transactions to be accepted — (`crates/apollo_gateway/src/stateless_transaction_validator.rs` / `crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

The `StatelessTransactionValidator` in the gateway explicitly validates that `nonce_data_availability_mode` and `fee_data_availability_mode` are `DataAvailabilityMode::L1` for every incoming `RpcTransaction`, with inline comments stating *"The Starknet OS enforces that the nonce/fee data availability mode is L1. We add this validation here in the gateway to prevent transactions from failing the OS."* The consensus block-validation path (`convert_consensus_tx_to_internal_consensus_tx`) processes the same `RpcTransaction` payload but never invokes the `StatelessTransactionValidator`, so these two invariant checks are silently absent. A malicious proposer can craft a V3 transaction with `nonce_data_availability_mode = L2` or `fee_data_availability_mode = L2`, sign it against the resulting (L2-DA-mode) hash, and include it in a block proposal. The validating node accepts and executes it through the blockifier (signature is valid for the L2-DA-mode hash), but the Starknet OS enforces L1-only DA mode and will fail the block during proving, creating a divergence between committed blockifier state and provable OS state.

---

### Finding Description

**Gateway path (validated):**

`Gateway::add_tx_inner` calls `self.stateless_tx_validator.validate(&tx)` before any conversion:

```
StatelessTransactionValidator::validate()
  → validate_nonce_data_availability_mode()   // asserts == L1
  → validate_fee_data_availability_mode()     // asserts == L1
  → …
  → convert_rpc_tx_to_internal()
``` [1](#0-0) [2](#0-1) 

**Consensus path (unvalidated):**

`TransactionConverter::convert_consensus_tx_to_internal_consensus_tx` for the `RpcTransaction` arm calls `convert_rpc_tx_to_internal()` directly, with no `StatelessTransactionValidator` step:

```rust
ConsensusTransaction::RpcTransaction(tx) => {
    let (internal_tx, proof_data) = self.convert_rpc_tx_to_internal(tx).await?;
    …
    Ok((InternalConsensusTransaction::RpcTransaction(internal_tx), task))
}
``` [3](#0-2) 

`convert_rpc_tx_to_internal` computes the transaction hash (which encodes the DA mode via `concat_data_availability_mode`) and returns the `InternalRpcTransaction` without ever checking whether the DA mode is `L1`: [4](#0-3) 

The DA-mode bits are baked into the hash preimage: [5](#0-4) 

**OS invariant:**

The Starknet OS Cairo program enforces `nonce_data_availability_mode == L1` at execution time. The gateway comment is the authoritative statement of this invariant: [6](#0-5) 

The blockifier's `perform_pre_validation_stage` does **not** check DA modes; it only checks nonce, fee bounds, and proof facts: [7](#0-6) 

---

### Impact Explanation

A malicious proposer constructs a V3 Invoke/Declare/DeployAccount transaction with `nonce_data_availability_mode = L2` (integer `1` instead of `0`). Because the DA mode is encoded in the hash, the proposer signs against the L2-DA-mode hash. The validating node's `convert_consensus_tx_to_internal_consensus_tx` recomputes the same L2-DA-mode hash and stores it as `tx_hash` in `InternalRpcTransaction`. The blockifier's signature validation passes (the stored hash matches the signature). The blockifier executes the transaction and commits a state diff. However, the Starknet OS enforces L1-only DA mode; when the prover runs the OS over the committed block, the OS fails on this transaction, making the block unprovable. This is a divergence between committed blockifier state and provable OS state — matching **Critical: Wrong state/receipt/revert result from blockifier/syscall/execution logic for accepted input**.

---

### Likelihood Explanation

Triggering this requires a malicious block proposer (a staked validator). In a BFT protocol tolerating up to `f` Byzantine validators, any of those `f` validators can act as proposer in their assigned round. No user collusion is needed: the proposer creates and signs the invalid-DA-mode transaction themselves. The validating node has no gate on the consensus path that would catch this before execution.

---

### Recommendation

Apply the same DA-mode invariant checks in the consensus conversion path. The simplest fix is to call `StatelessTransactionValidator::validate_nonce_data_availability_mode` and `validate_fee_data_availability_mode` (or an equivalent shared helper) inside `convert_consensus_tx_to_internal_consensus_tx` before calling `convert_rpc_tx_to_internal`, mirroring the gateway's `add_tx_inner` flow:

```diff
// in convert_consensus_tx_to_internal_consensus_tx
ConsensusTransaction::RpcTransaction(tx) => {
+   // Mirror the gateway's stateless DA-mode invariant checks.
+   validate_da_modes(&tx)?;   // assert nonce_da == L1, fee_da == L1
    let (internal_tx, proof_data) = self.convert_rpc_tx_to_internal(tx).await?;
    …
}
```

Alternatively, extract the DA-mode checks into a standalone function in `starknet_api` and call it from both `StatelessTransactionValidator::validate` and `TransactionConverter::convert_consensus_tx_to_internal_consensus_tx`.

---

### Proof of Concept

1. Craft an `RpcInvokeTransactionV3` with `nonce_data_availability_mode = DataAvailabilityMode::L2` and `fee_data_availability_mode = DataAvailabilityMode::L2`.
2. Compute the transaction hash — `concat_data_availability_mode` encodes `nonce_mode=1, fee_mode=1`, producing a hash distinct from the L1-DA-mode hash.
3. Sign the transaction against this L2-DA-mode hash with a valid account key.
4. Wrap it as `ConsensusTransaction::RpcTransaction(RpcTransaction::Invoke(…))` and include it in a block proposal.
5. The validating node calls `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal` → `calculate_transaction_hash` (recomputes the same L2-DA-mode hash) → stores `InternalRpcTransaction { tx_hash: <L2-DA-mode hash>, … }`.
6. The blockifier's `perform_pre_validation_stage` checks nonce and fee bounds only; DA mode is not checked. Signature validation passes (hash matches).
7. The transaction executes and a state diff is committed.
8. The Starknet OS, which enforces `nonce_data_availability_mode == L1`, fails on this transaction during proving, making the committed block unprovable. [2](#0-1) [3](#0-2) [5](#0-4) [7](#0-6)

    /// The Starknet OS enforces that the nonce data availability mode is L1. We add this validation
    /// here in the gateway to prevent transactions from failing the OS.
    fn validate_nonce_data_availability_mode(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let expected_da_mode = DataAvailabilityMode::L1;
        let da_mode = *tx.nonce_data_availability_mode();
        if da_mode != expected_da_mode {
            return Err(StatelessTransactionValidatorError::InvalidDataAvailabilityMode {
                field_name: "nonce".to_string(),
            });
        };

        Ok(())
    }

fn concat_data_availability_mode(
    nonce_mode: &DataAvailabilityMode,
    fee_mode: &DataAvailabilityMode,
) -> Felt {
    (data_availability_mode_index(fee_mode)
        + (data_availability_mode_index(nonce_mode) << DATA_AVAILABILITY_MODE_BITS))
        .into()
}
