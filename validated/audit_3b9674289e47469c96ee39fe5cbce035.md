### Title
Fast Withdrawal Provider Liquidity Permanently Stranded When `setWithdrawPool` Rotates Pool Reference — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.setWithdrawPool` can atomically redirect the `withdrawPool` pointer to a new contract. Any fast withdrawal provider who pre-paid a user from the **old** `WithdrawPool` and is waiting for sequencer reimbursement will never receive it: the sequencer's subsequent `handleWithdrawTransfer` call routes tokens to the **new** pool, the new pool's `markedIdxs` is clean so it pays the user a second time, and the old pool's balance is never replenished. The only recovery path (`removeLiquidity`) is `onlyOwner` and sends funds to the owner, not to the original providers.

---

### Finding Description

**Reimbursement mechanism (normal flow)**

`Clearinghouse.handleWithdrawTransfer` always does two things atomically:

```solidity
token.safeTransfer(withdrawPool, uint256(amount));
BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
``` [1](#0-0) 

`submitWithdrawal` checks `markedIdxs[idx]` and returns early if already set, leaving the just-transferred tokens inside the pool as reimbursement for whoever pre-paid the user via `submitFastWithdrawal`. [2](#0-1) 

**The pointer update**

`setWithdrawPool` is an `onlyOwner` function that replaces the stored `withdrawPool` address with no migration logic:

```solidity
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));
    withdrawPool = _withdrawPool;
}
``` [3](#0-2) 

`withdrawPool` is a plain storage slot with no pending-reimbursement accounting: [4](#0-3) 

**Desynchronization**

1. Provider deposits tokens into `WithdrawPool_A` and calls `submitFastWithdrawal(idx=N, …)` → user is paid, `markedIdxs[N] = true` in `WithdrawPool_A`.
2. Owner calls `setWithdrawPool(WithdrawPool_B)`.
3. Sequencer processes the withdrawal: `handleWithdrawTransfer` now calls `token.safeTransfer(WithdrawPool_B, amount)` + `WithdrawPool_B.submitWithdrawal(…, idx=N)`.
4. `WithdrawPool_B.markedIdxs[N]` is `false` → it pays the user **again** and sets `minIdx = N`.
5. `WithdrawPool_A` never receives the reimbursement tokens; the provider's pre-deposited liquidity is permanently stranded there.

The only recovery path is `removeLiquidity`, which is `onlyOwner` and sends funds to the owner, not to the original providers: [5](#0-4) 

`submitWithdrawal` enforces `msg.sender == clearinghouse`, so providers cannot call it directly on the old pool to trigger reimbursement themselves: [6](#0-5) 

---

### Impact Explanation

Fast withdrawal providers permanently lose the liquidity they pre-deposited into the old `WithdrawPool`. Simultaneously, the protocol suffers a double-payment loss: the user receives the withdrawal amount twice (once from the provider via `submitFastWithdrawal`, once from the new pool via `submitWithdrawal`). The corrupted state is: `WithdrawPool_A.balance` is depleted with no reimbursement path for providers, and `WithdrawPool_B` pays out tokens that should have been a no-op.

---

### Likelihood Explanation

`setWithdrawPool` is a realistic operational action (pool upgrade, bug fix, migration). The window of vulnerability is any period between a provider calling `submitFastWithdrawal` and the sequencer settling that same index. Given that sequencer settlement is not instantaneous and pool upgrades can happen at any time, the overlap is plausible during normal protocol maintenance.

---

### Recommendation

Before updating `withdrawPool`, verify that all fast-withdrawal indices that have been marked in the current pool have also been settled by the sequencer (i.e., `minIdx` in the old pool equals the highest fast-withdrawal index). Alternatively, implement a `migrateReimbursements` function that transfers the old pool's outstanding balance to the new pool, or add a `pendingReimbursements` counter that must reach zero before `setWithdrawPool` is permitted.

---

### Proof of Concept

```
1. Provider deposits 1 000 USDC into WithdrawPool_A.
2. Provider calls WithdrawPool_A.submitFastWithdrawal(idx=100, …)
   → user receives 1 000 USDC; markedIdxs[100] = true in WithdrawPool_A.
3. Owner calls Clearinghouse.setWithdrawPool(WithdrawPool_B).
4. Sequencer calls Clearinghouse.withdrawCollateral(…, idx=100)
   → handleWithdrawTransfer:
       token.safeTransfer(WithdrawPool_B, 1 000)   // tokens go to new pool
       WithdrawPool_B.submitWithdrawal(…, idx=100)  // markedIdxs[100] == false
       → user receives 1 000 USDC again             // double payment
5. WithdrawPool_A balance: 0 USDC (provider's 1 000 already paid to user in step 2,
   reimbursement never arrived).
6. Provider's 1 000 USDC is unrecoverable without owner calling removeLiquidity
   and manually redistributing — no direct provider recovery path exists.
``` [7](#0-6) [8](#0-7)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
```

    function submitWithdrawal(
        IERC20Base token,
        address sendTo,
        uint128 amount,
        uint64 idx
    ) public {
        require(msg.sender == clearinghouse);

        if (markedIdxs[idx]) {
            return;
        }
        markedIdxs[idx] = true;
        // set minIdx to most recent withdrawal submitted by sequencer
        minIdx = idx;

        handleWithdrawTransfer(token, sendTo, amount);
    }
