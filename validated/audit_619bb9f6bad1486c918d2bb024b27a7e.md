### Title
Deposited Collateral Permanently Locked When Slow-Mode `DepositCollateral` Execution Fails — (`core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` transfers real ERC-20 tokens from `msg.sender` into the contract at queue time, then defers the on-chain credit to the subaccount to a slow-mode execution step that runs up to three days later. The execution step runs inside a `try/catch` that silently discards any revert, and the in-code comment `// try return funds now removed` explicitly confirms the refund path was deleted. If the deferred `DepositCollateral` slow-mode transaction reverts at execution time for any reason, the deposited tokens are permanently locked in the `Endpoint` contract with no recovery path.

---

### Finding Description

**Queue-time path — funds taken, no validation of execution-time preconditions:**

`depositCollateralWithReferral` (Endpoint.sol lines 144–166) transfers the user's tokens into the contract and enqueues a `SlowModeTx` in a single atomic step:

```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: abi.encodePacked(...)
});
``` [1](#0-0) 

The only pre-queue checks are: isolated-subaccount guard, sanctions check, and `isValidDepositAmount`. None of these verify the execution-time preconditions inside `Clearinghouse.depositCollateral`.

**Execution-time path — silent failure, no refund:**

`_executeSlowModeTransaction` deletes the queued entry *before* attempting execution, then wraps the call in a `try/catch` that swallows all non-OOG reverts:

```solidity
delete slowModeTxs[_slowModeConfig.txUpTo++];
...
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed
}
``` [2](#0-1) 

The comment `// try return funds now removed` is an explicit acknowledgement that the refund mechanism was intentionally deleted. Once the entry is deleted and the catch branch is taken, the deposited tokens have no path back to the user.

**The deferred validation that can fail:**

`processSlowModeTransactionImpl` for `DepositCollateral` calls `clearinghouse.depositCollateral(txn)`: [3](#0-2) 

Inside `Clearinghouse.depositCollateral`, `_decimals(txn.productId)` is called:

```solidity
function _decimals(uint32 productId) internal virtual returns (uint8) {
    IERC20Base token = IERC20Base(_tokenAddress(productId));
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    return token.decimals();
}
``` [4](#0-3) 

This `require` is **not evaluated at queue time**. The queue-time check (`isValidDepositAmount` → `checkMinDeposit`) also calls `_decimals`, but it does so against the live state at queue time. The 3-day `SLOW_MODE_TX_DELAY` window means the product's token address can become zero (e.g., via a product configuration update, token migration, or any admin action that zeroes the token field) before execution. When that happens, `_decimals` reverts with `ERR_INVALID_PRODUCT`, the `try/catch` silently discards the revert, and the deposited tokens are permanently locked. [5](#0-4) 

---

### Impact Explanation

The deposited ERC-20 tokens are held by the `Endpoint` contract. After the slow-mode entry is deleted and the execution silently fails, there is no on-chain mechanism to recover them: no retry, no refund, no admin rescue path. The user's collateral is permanently locked. The corrupted state delta is: `Endpoint` holds `amount` tokens that are neither credited to the subaccount nor returnable to the depositor.

---

### Likelihood Explanation

`depositCollateralWithReferral` is `public` and is called both directly by users and by `DirectDepositV1.creditDeposit()` for every product ID. [6](#0-5) 

The 3-day `SLOW_MODE_TX_DELAY` creates a concrete window during which any product configuration change (token migration, product removal, or admin update) can cause the deferred execution to fail. The `// try return funds now removed` comment confirms the protocol team was aware of this failure mode and chose not to address it, making the silent-loss path a permanent structural property of the contract.

---

### Recommendation

Validate all execution-time preconditions (specifically, that `_decimals(productId)` will succeed) before taking custody of the user's tokens in `depositCollateralWithReferral`. Alternatively, restore a refund path in the `catch` block of `_executeSlowModeTransaction` for `DepositCollateral` transactions, so that if execution fails the tokens are returned to the original depositor recorded in the `SlowModeTx.sender` field.

---

### Proof of Concept

1. A spot product `P` exists with a valid token address. Its `_decimals` check passes.
2. User calls `Endpoint.depositCollateral(subaccountName, P, amount)`. `isValidDepositAmount` returns `true`. `handleDepositTransfer` moves `amount` tokens from the user into `Endpoint`. A `SlowModeTx` is enqueued with `executableAt = now + 3 days`.
3. Within the 3-day window, an admin updates the spot engine configuration for product `P` so that `getConfig(P).token == address(0)` (e.g., token migration or product decommission).
4. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`. `_executeSlowModeTransaction` deletes the entry, then calls `processSlowModeTransaction` inside `try/catch`. `clearinghouse.depositCollateral` calls `_decimals(P)`, which hits `require(address(token) != address(0), ERR_INVALID_PRODUCT)` and reverts. The `catch` branch is taken. The comment `// try return funds now removed` confirms no refund is issued.
5. The user's `amount` tokens remain in `Endpoint` indefinitely with no recovery path.

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
