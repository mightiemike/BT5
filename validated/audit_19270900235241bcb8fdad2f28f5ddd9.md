### Title
Linked Signer Can Unilaterally Hijack Its Own Delegation via `LinkSigner` Fast-Mode Transaction — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction handler in `EndpointTx.sol` passes `allowLinkedSigner = true` to `validateSignedTx`, permitting the current linked signer — a delegated, lower-privilege role — to overwrite the `linkedSigners` mapping for any subaccount it controls. This operation should be restricted exclusively to the subaccount owner. The slow-mode path correctly enforces owner-only access via `validateSender`, but the fast-mode path does not.

---

### Finding Description

In `EndpointTx.sol`, the fast-mode `LinkSigner` handler at lines 576–590 decodes a `SignedLinkSigner` transaction and calls:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
```

`validateSignedTx` delegates to `validateCompactSignature`, which in turn calls `verifier.validateCompactSignature(sender, allowLinkedSigner ? getLinkedSigner(sender) : address(0), digest, signature)`. The `Verifier.validateCompactSignature` accepts a signature from **either** the subaccount owner address **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

Because `allowLinkedSigner = true` is passed, the current linked signer's signature is accepted as valid for a `LinkSigner` transaction. This means the linked signer can craft and submit a `LinkSigner` transaction that replaces `linkedSigners[subaccount]` with any address of its choosing — including an attacker-controlled address or `address(0)`.

By contrast, the slow-mode `LinkSigner` handler at lines 232–239 uses `validateSender(txn.sender, sender)`, which enforces that `msg.sender == address(uint160(bytes20(txn.sender)))` — i.e., only the subaccount owner can submit it:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

The two paths are inconsistent. The fast-mode path, which is the primary production path (submitted by the sequencer via `submitTransactions`), incorrectly grants the linked signer the ability to mutate its own delegation.

---

### Impact Explanation

A compromised or malicious linked signer can:

1. **Hijack signing authority**: Replace `linkedSigners[subaccount]` with an attacker-controlled address. That new address then has full signing authority over the subaccount — it can submit orders, withdraw collateral (where linked signer is permitted), and transfer quote — without the owner's consent.
2. **Revoke owner's delegation**: Set `linkedSigners[subaccount] = address(0)`, silently removing the owner's linked signer and disrupting any automated trading or NLP operations that depend on it.
3. **Persistent takeover**: Because the linked signer can set a new linked signer, the attacker can chain the delegation to a permanent attacker-controlled key, making recovery difficult without the owner submitting a corrective slow-mode transaction.

The corrupted state is `linkedSigners[subaccount]` — a mapping that governs who can sign fast-mode transactions for the subaccount. Any subsequent transaction validated with `allowLinkedSigner = true` (orders, liquidations, transfers, burns, mints) will accept signatures from the attacker-controlled address.

---

### Likelihood Explanation

**Medium–High.** The linked signer is an externally-controlled address explicitly granted delegated access. Any linked signer that turns malicious (e.g., a compromised API key, a rogue trading bot, or a third-party integration) can immediately exploit this path. No admin access, governance capture, or sequencer compromise is required — the attacker only needs to control the current linked signer key and submit a sequencer transaction, which is the normal production flow.

---

### Recommendation

Change `allowLinkedSigner` to `false` in the fast-mode `LinkSigner` handler so that only the subaccount owner can update the linked signer mapping, consistent with the slow-mode path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only owner may update linked signer
);
```

This aligns the fast-mode path with the slow-mode path's intent and prevents a delegated signer from mutating its own delegation.

---

### Proof of Concept

1. Owner `0xAlice` has subaccount `alice_default` and has set `linkedSigners[alice_default] = 0xBot` (a trading bot).
2. `0xBot` (the linked signer) constructs a `SignedLinkSigner` transaction with:
   - `tx.sender = alice_default`
   - `tx.signer = bytes32(uint256(uint160(0xAttacker)))`
   - Signs the EIP-712 digest with `0xBot`'s private key.
3. `0xBot` submits this to the sequencer, which includes it in a `submitTransactions` batch.
4. `EndpointTx.processTransactionImpl` reaches the `LinkSigner` branch, calls `validateSignedTx(..., true)`.
5. `Verifier.validateCompactSignature` recovers `0xBot` from the signature, checks `recovered == linkedSigner` → passes.
6. `linkedSigners[alice_default]` is now set to `0xAttacker`.
7. `0xAttacker` can now sign any `allowLinkedSigner = true` transaction for `alice_default`, including `WithdrawCollateral` (standard path), orders, and transfers. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
