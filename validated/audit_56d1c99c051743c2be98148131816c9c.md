### Title
Empty Merkle Proof Guard Permanently Blocks SPV Verification of Single-Transaction Blocks — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (and by extension `verify_transaction_inclusion_v2`) contains a guard `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` that unconditionally rejects an empty proof. For a Bitcoin block containing exactly one transaction, the Merkle proof is legitimately empty — the transaction hash *is* the Merkle root — yet the guard causes the call to panic, permanently blocking SPV verification for that valid case.

### Finding Description

In `contract/src/lib.rs`, `verify_transaction_inclusion` enforces:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

Immediately after, it delegates to `merkle_tools::compute_root_from_merkle_proof`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

`compute_root_from_merkle_proof` is implemented as:

```rust
for proof_hash in merkle_proof {
    ...
}
current_hash  // returned unchanged when merkle_proof is empty
``` [3](#0-2) 

When `merkle_proof` is empty the loop body never executes and `current_hash` (the transaction hash) is returned directly. For a block with exactly one transaction, the Merkle root *is* the transaction hash, so an empty proof is the **correct and only valid proof**. The `compute_root_from_merkle_proof` logic handles this correctly, but the `require!` guard fires first and panics, making the correct result unreachable.

`verify_transaction_inclusion_v2` inherits the same defect because it converts its arguments and calls `verify_transaction_inclusion` internally:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [4](#0-3) 

### Impact Explanation

Any unprivileged NEAR caller invoking `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction in a single-transaction block will always receive a panic (`"Merkle proof is empty"`) regardless of whether the transaction is genuinely included. The SPV proof result is corrupted: a legitimately provable inclusion is permanently reported as invalid. Downstream contracts that rely on the verification result (e.g., bridge contracts gating asset releases on proof success) will be permanently blocked for all such blocks.

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) are common throughout Bitcoin's history — every early mainnet block and any block where no user transactions were included falls into this category. A relayer that has ingested such a block header into the contract creates a state where no caller can ever successfully verify the coinbase transaction's inclusion, even though the header is stored and the proof is mathematically trivial.

### Recommendation

Remove the blanket `require!(!args.merkle_proof.is_empty(), ...)` guard. An empty proof is valid when `tx_index == 0` and the block contains exactly one transaction. If a non-empty proof is required as a security invariant, the check should be conditioned on the tree depth, not on proof length alone. For example:

```rust
// Only require a non-empty proof when the tree has more than one leaf.
// An empty proof is valid iff tx_index == 0 and the tx_hash equals the merkle root.
```

Alternatively, validate the proof length against the expected tree depth derived from the stored header rather than rejecting zero-length proofs outright.

### Proof of Concept

1. Relay a Bitcoin block header for a block containing only the coinbase transaction (e.g., Bitcoin mainnet block 1, where `merkle_root == coinbase_txid`). The header is accepted and stored by `submit_blocks`.
2. Call `verify_transaction_inclusion` with:
   - `tx_id` = the coinbase transaction hash (= the block's `merkle_root`)
   - `tx_index` = 0
   - `merkle_proof` = `[]` (empty — the only correct proof)
3. Observed: the call panics with `"Merkle proof is empty"` before reaching `compute_root_from_merkle_proof`.
4. Expected: `compute_root_from_merkle_proof(tx_id, 0, &[])` returns `tx_id`, which equals `header.block_header.merkle_root`, so the function should return `true`. [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
