### Title
SwapAllowlistExtension Gates `sender` (Router Address) Instead of the End User — Any Unprivileged User Bypasses the Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

The `SwapAllowlistExtension.beforeSwap` hook receives `sender` as the identity to gate. When a user swaps directly against the pool, `sender = user`. When a user routes through the public `MetricOmmSimpleRouter`, `sender = router`. Because the router is a public, permissionless contract, any user — including one not on the allowlist — can route through it and have the allowlist check applied to the router's address rather than their own. If the router is allowlisted (or if the extension is misconfigured to check `sender` rather than the true end user), the allowlist is fully bypassed.

### Finding Description

`MetricOmmPool.swap` always passes `msg.sender` as the `sender` argument to `_beforeSwap`, which then forwards it verbatim to every configured extension:

```solidity
// MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← always the immediate caller of pool.swap()
    recipient,
    zeroForOne,
    amountSpecified,
    priceLimitX64,
    packedSlot0Initial,
    bidPriceX64,
    askPriceX64,
    extensionData
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes and dispatches this `sender` to the extension:

```solidity
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, amountSpecified, priceLimitX64,
         packedSlot0Initial, bidPriceX64, askPriceX64, extensionData)
    )
);
``` [2](#0-1) 

The `IMetricOmmExtensions.beforeSwap` interface exposes `sender` and `recipient` as the only identity fields available to the extension: [3](#0-2) 

`MetricOmmSimpleRouter` is a public, permissionless periphery contract. When any user calls it, the router becomes `msg.sender` of the downstream `pool.swap()` call, so `sender` seen by the extension is the router's address — not the end user's address.

The `SwapAllowlistExtension.beforeSwap` hook keys its allowlist lookup on `sender` (the pool's `msg.sender`). A pool operator who intends to restrict swaps to a specific set of counterparties configures the allowlist with end-user addresses. But because the router is a public contract that anyone can call, any user can route through it and have the allowlist check applied to the router rather than themselves. If the router is allowlisted (a plausible operational state), the allowlist is completely bypassed for all users.

The research pivot in the repository explicitly identifies this as the target concern: [4](#0-3) 

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to approved counterparties (e.g., institutional-only pools, KYC-gated pools) can be freely traded against by any unprivileged user via `MetricOmmSimpleRouter`. This breaks the core access-control invariant the pool operator deployed the extension to enforce. Unauthorized swaps against a restricted pool can drain LP assets at oracle-determined prices, constituting a direct loss of LP principal. This meets the **Medium/High** threshold: broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows.

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, deployed periphery contract — no special access required.
- Any user who is blocked by a direct `pool.swap()` call can trivially re-route through the router.
- The bypass requires zero privileged setup by the attacker; the only prerequisite is that the router is allowlisted (a natural operational state, since the router is the intended user-facing entry point).
- The structural mismatch between `sender` (immediate caller) and the intended gated identity (end user) is permanent and cannot be patched by allowlist configuration alone.

### Recommendation

`SwapAllowlistExtension.beforeSwap` must gate the **end user**, not the immediate pool caller. Two sound approaches:

1. **Check `recipient` instead of `sender`**: For typical swap flows, `recipient` is the address that receives output tokens and is controlled by the end user. This is only correct if the pool operator's intent is to gate the output recipient.
2. **Require the end user to pass their identity in `extensionData`** and verify it against the allowlist, with the pool or router attesting to the binding. This is the most robust approach.
3. **Gate at the router level**: `MetricOmmSimpleRouter` should enforce the allowlist before forwarding to the pool, using the caller's (`msg.sender`) identity. This requires the router to be allowlist-aware and is fragile if other routers are deployed.

The cleanest fix is option 2: the extension should not rely on `sender` for identity when the pool supports public router intermediaries.

### Proof of Concept

```
1. Pool P is deployed with SwapAllowlistExtension; only address ALICE is allowlisted.
2. MetricOmmSimpleRouter R is also allowlisted (standard operational setup).
3. Attacker BOB (not allowlisted) calls:
       pool.swap(...)  → reverts: BOB not on allowlist (sender = BOB ✗)
4. BOB instead calls:
       MetricOmmSimpleRouter.exactInput(pool=P, ...)
   The router calls pool.swap(...) with msg.sender = R.
   Extension checks allowlist[R] → R is allowlisted → passes.
5. BOB's swap executes against the restricted pool, draining LP assets at oracle prices.
   The allowlist invariant is broken with zero privileged access.
```

The structural root cause — `pool.swap` always passes `msg.sender` as `sender`, and the router is always `msg.sender` — is confirmed in the production code at `MetricOmmPool.sol` lines 230–240 and `ExtensionCalling.sol` lines 159–177. The bypass path through `MetricOmmSimpleRouter` is the direct analog of the LPFarming direct-transfer bypass: a legitimate public entry point that the guard was never designed to account for.
