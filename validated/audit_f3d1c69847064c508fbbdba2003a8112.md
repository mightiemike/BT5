Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end user, allowing any caller to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `pool.swap()`. When users enter through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the end user. If the pool admin allowlists the router to enable router-mediated swaps for permitted users, every unpermitted user can bypass the allowlist by routing through the same public router, completely defeating the access control.

## Finding Description

**Root cause — `sender` is the direct caller of `pool.swap()`, not the end user:**

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // <-- direct caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol:160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks this `sender` against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

**Router path — `sender` becomes the router, not the end user:**

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
```

Here `msg.sender` to the pool is the router contract. So `sender` in `beforeSwap` is the router address, not the end user (`msg.sender` of `exactInputSingle`).

**Exploit flow:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a set of permitted users (e.g., KYC'd addresses).
2. Pool admin allowlists specific user addresses (e.g., `alice`) for direct swaps.
3. Pool admin also allowlists the `MetricOmmSimpleRouter` address so that permitted users can swap via the router (otherwise router-mediated swaps by `alice` would revert, since the router is the `sender`).
4. Unpermitted user `bob` calls `MetricOmmSimpleRouter.exactInputSingle()` targeting the restricted pool.
5. The router calls `pool.swap()` with `msg.sender = router`.
6. Pool calls `_beforeSwap(sender = router, ...)`.
7. Extension checks `allowedSwapper[pool][router]` → `true` (admin allowlisted the router).
8. `bob`'s swap succeeds despite not being on the allowlist.

**Why existing guards fail:**

The `SwapAllowlistExtension` has no mechanism to recover the original end user's address. The `extensionData` field is passed through from the router caller but is never read by `SwapAllowlistExtension.beforeSwap` — it only checks `sender`. There is no on-chain binding between the router call and the originating EOA that the extension can verify.

The allowlist is structurally broken for router-mediated swaps: either (a) the admin does not allowlist the router, in which case no user can swap via the router even if they are individually permitted, or (b) the admin allowlists the router, in which case every user can bypass the allowlist.

## Impact Explanation

Any unpermitted user can execute swaps on a pool that is intended to be restricted to a specific set of addresses. This breaks the core access-control invariant of `SwapAllowlistExtension` — the pool admin's intent to gate swaps by user identity is completely defeated. Pools designed for permissioned trading (e.g., compliance-gated, institutional, or whitelist-only pools) become effectively public, allowing unauthorized parties to trade against pool liquidity. This constitutes broken core pool functionality and an admin-boundary break where an unprivileged path bypasses a required guard.

## Likelihood Explanation

The precondition — that the pool admin allowlists the router — is a natural and expected operational step. Any pool admin who wants to support router-mediated swaps for permitted users must allowlist the router, since the router is the direct caller of `pool.swap()`. This is not a misconfiguration; it is the only way to enable router usage on an allowlisted pool. The attack requires no special privileges: any EOA can call `MetricOmmSimpleRouter.exactInputSingle` with the target pool. The bypass is repeatable and deterministic.

## Recommendation

The `SwapAllowlistExtension` should gate on the end user's identity rather than the direct caller of `pool.swap()`. Two approaches:

1. **Check `tx.origin` instead of `sender`** (simple but has known limitations with smart contract wallets and meta-transactions).
2. **Decode the end user from `extensionData`**: Require the router (or any intermediary) to encode the originating user's address in `extensionData`, and have the extension verify it against the allowlist. The router would need to encode `msg.sender` into `extensionData` before forwarding to the pool.
3. **Remove router allowlisting as a pattern**: Document that the allowlist only works for direct `pool.swap()` callers, and that router-mediated swaps are incompatible with this extension unless the router itself enforces per-user allowlist checks before calling the pool.

The most robust fix is option 2: modify `SwapAllowlistExtension.beforeSwap` to decode and verify the end user from `extensionData` when `sender` is a known router, and update `MetricOmmSimpleRouter` to encode `msg.sender` into `extensionData` for allowlisted pools.

## Proof of Concept

```solidity
// Foundry test outline
function test_routerBypassesSwapAllowlist() public {
    // 1. Deploy pool with SwapAllowlistExtension
    // 2. Admin allowlists alice for direct swaps
    // 3. Admin allowlists MetricOmmSimpleRouter so alice can use the router
    extension.setAllowedToSwap(address(pool), alice, true);
    extension.setAllowedToSwap(address(pool), address(router), true);

    // 4. bob (not allowlisted) calls the router
    vm.prank(bob);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        recipient: bob,
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        tokenIn: token0,
        extensionData: ""
    }));
    // 5. Swap succeeds — bob bypassed the allowlist
    // Assert bob received output tokens despite not being allowlisted
}
```

The test demonstrates that `bob`, who is not in `allowedSwapper[pool]`, successfully swaps by routing through `MetricOmmSimpleRouter`, because the extension sees `sender = router` (which is allowlisted) rather than `sender = bob`.
