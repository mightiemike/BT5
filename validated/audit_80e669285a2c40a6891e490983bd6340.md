### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Validation — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) was deprecated and superseded by `verify_transaction_inclusion_v2`, which adds a mandatory coinbase Merkle proof check to close the 64-byte transaction forgery vulnerability. However, v1 remains `pub` and is callable by any unprivileged NEAR account at runtime. The Rust `#[deprecated]` attribute is a compiler-only warning with no runtime enforcement. Any caller can invoke v1 directly, bypassing the coinbase proof check entirely.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability (documented inline at line 328). It enforces a coinbase Merkle proof check before delegating to v1: [1](#0-0) 

v1 is marked `#[deprecated]` at line 283, but this attribute only produces a compiler warning — it imposes no runtime restriction. v1 remains `pub` and is guarded only by `#[pause]`, which is inactive by default: [2](#0-1) 

v1's own documentation explicitly acknowledges the forgery risk: [3](#0-2) 

The contract's own internal call to v1 from v2 uses `#[allow(deprecated)]` to suppress the compiler warning, confirming that the deprecation is purely advisory: [4](#0-3) 

The result is a desynchronized control structure: v2 is the intended, hardened path, but v1 — the vulnerable path — remains an equally reachable public entrypoint. The "replacement" mechanism (deprecation) fails to restrict the unintended operation at runtime, directly mirroring the external report's class: a single control mechanism that is supposed to gate two operations independently but does not.

---

### Impact Explanation

**Impact: High.** An attacker who calls `verify_transaction_inclusion` directly with a crafted 64-byte `tx_id` (an internal Merkle tree node hash) can cause the function to return `true` for a transaction that was never included in a Bitcoin block. Any NEAR DApp consuming this result — e.g., to release bridged funds, mint tokens, or authorize cross-chain actions — is deceived into accepting a forged proof. The coinbase proof check in v2, which is the only defense against this class of forgery, is completely absent in the v1 code path. [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: High.** The entry path requires no privileges. Any NEAR account can call `verify_transaction_inclusion` directly when the contract is not paused (the default state). The 64-byte transaction attack is publicly documented and the bypass is a single direct function call. No special knowledge of the contract internals is required beyond knowing that v1 exists and is callable.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or replace it with `pub(crate)` so it is only accessible internally (from v2). Alternatively, add a runtime `panic!` or `env::panic_str` at the top of v1 to unconditionally reject direct external calls. The simplest fix is to make v1 private and have v2 call the shared verification logic directly, eliminating the publicly reachable vulnerable path entirely.

---

### Proof of Concept

1. Attacker identifies a Bitcoin block whose Merkle tree has an internal node whose 32-byte hash, when interpreted as a `tx_id`, satisfies the Merkle proof check (the 64-byte transaction attack described at https://www.bitmex.com/blog/64-Byte-Transactions).
2. Attacker constructs a `ProofArgs` with this crafted `tx_id`, a valid `tx_block_blockhash` for a block already in the contract's `headers_pool`, a matching `merkle_proof`, and sufficient `confirmations`.
3. Attacker calls `verify_transaction_inclusion` directly on the NEAR contract (not `verify_transaction_inclusion_v2`).
4. The function executes with no coinbase proof check, computes the Merkle root from the crafted inputs, finds it matches the block's `merkle_root`, and returns `true`.
5. Any downstream NEAR DApp that trusted this result treats the forged transaction as confirmed on Bitcoin. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
