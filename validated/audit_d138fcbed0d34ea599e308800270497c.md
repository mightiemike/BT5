### Title
`gas_key_transfer_exec_fee` / `gas_key_add_key_exec_fee` Called with Borsh Wire Length Instead of Trie-ID Length for ML-DSA-65 Keys in Host-Function Path — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The exec-fee helpers `gas_key_transfer_exec_fee` and `gas_key_add_key_exec_fee` are documented to price trie I/O based on the **on-trie key length** (`trie_id_len()`). The transaction-conversion path (`runtime/runtime/src/config.rs`) correctly passes `public_key.trie_id_len()` (33 bytes for ML-DSA-65). The host-function path in both `logic.rs` and `wasmtime_runner/logic.rs` passes the raw `public_key_len` argument supplied by the contract (1953 bytes for ML-DSA-65). For ed25519/secp256k1 the two values are identical, so the bug is latent until `PostQuantumSignatures` activates. After activation, any contract that calls `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 key is overcharged by approximately `(1953 − 33) × gas_key_byte_exec_fee` gas per nonce entry.

---

### Finding Description

`gas_key_transfer_exec_fee` computes the execution fee as:

```
trie_key_len = access_key_key_len(account_id_len, public_key_len)
per_byte     = gas_key_byte_exec_fee × (trie_key_len + estimated_value_len)
```

Its docstring states: *"Based on the access key trie key length + estimated value length (what the receiver needs to read/write in the trie)."*

For ML-DSA-65 the actual trie key stores a 33-byte SHA3-256 hash (`trie_id_len() = 1 + 32`), not the 1953-byte borsh-encoded pubkey (`len() = 1 + 1952`).

**Transaction path** — correctly uses `trie_id_len()`:

```rust
// runtime/runtime/src/config.rs  line 348
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
// line 392
let nonce_fee = gas_key_add_key_exec_fee(
    fees, account_id.len(), public_key.trie_id_len(), gas_key_info.num_nonces,
);
```

**Host-function path** — passes raw `public_key_len` (borsh wire size):

```rust
// runtime/near-vm-runner/src/logic/logic.rs  line 3092
let exec = gas_key_transfer_exec_fee(
    &self.fees_config, receiver_id.len(), public_key_len as usize,  // ← 1953 for ML-DSA-65
);
// line 3155
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config, receiver_id.len(), public_key_len as usize, num_nonces,
);
// line 3226
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config, receipt_receiver_id.len(), public_key_len as usize, num_nonces,
);
```

The same pattern appears in `runtime/near-vm-runner/src/wasmtime_runner/logic.rs` at lines 3324, 3395, and 3499.

The exact divergent values for ML-DSA-65:
- Correct: `public_key_len = trie_id_len() = 33`
- Actual: `public_key_len = len() = 1953`
- Overcharge per nonce entry: `(1953 − 33) × gas_key_byte_exec_fee × (nonce_key_overhead + NONCE_VALUE_LEN)`

---

### Impact Explanation

Any unprivileged contract that calls the gas-key host functions with an ML-DSA-65 key after `PostQuantumSignatures` activates (protocol version 85) is overcharged for execution gas. The overcharge is deterministic and protocol-enforced: every node computes the same inflated fee, so the transaction succeeds but burns ~1920 × `gas_key_byte` extra exec gas per nonce entry. At the shipped `gas_key_byte` exec parameter (101,435,400 gas/byte), the overcharge is ≈ 0.195 Tgas per nonce per operation — a significant fraction of the per-receipt gas budget. The transaction-path fee and the host-function-path fee for the same ML-DSA-65 gas-key action are materially different, breaking the protocol invariant that action fees are representation-independent.

---

### Likelihood Explanation

The bug is latent for ed25519/secp256k1 (where `len() == trie_id_len()`). It becomes reachable the moment `PostQuantumSignatures` is live and any contract issues a gas-key host-function call with an ML-DSA-65 key. No privileged role is required; any deployed contract can trigger it. The discrepancy is not gated or documented as intentional.

---

### Recommendation

In all three host-function call sites, decode the public key first and pass `trie_id_len()` to the exec-fee helpers, mirroring the transaction path:

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let pk_decoded = public_key.decode()?;
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    pk_decoded.trie_id_len(),   // ← on-trie length, not wire length
);
```

Apply the same fix to `gas_key_add_key_exec_fee` call sites in both `logic.rs` and `wasmtime_runner/logic.rs`. The send-fee helper (`gas_key_transfer_send_fee`) intentionally uses the wire length ("what the sender sees") and should remain unchanged.

```rust
/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
```

```rust
    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {

            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
```
