### Title
Declare Transactions Bypass `max_l2_gas_amount` Cap in Stateless Validator, Admitting Transactions with Unbounded L2 Gas Amount — (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` explicitly skips the `max_l2_gas_amount` cap for `Declare` transactions. This is the direct sequencer analog of the external bug: just as reduce-only orders bypass collateral requirements but still have their `price × amount` product unconditionally counted toward `quoteOI`, Declare transactions bypass the `max_l2_gas_amount` cap but still have their `l2_gas.max_amount` unconditionally used in the `max_possible_fee = max_amount × max_price_per_unit` product. The gateway therefore admits Declare transactions with an arbitrarily large `l2_gas.max_amount`, violating the intended stateless admission invariant.

---

### Finding Description

In `validate_resource_bounds`, the code branches on transaction type and silently skips the cap for Declare:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
``` [1](#0-0) 

The `max_l2_gas_amount` cap exists precisely to bound the `l2_gas.max_amount` field before it is multiplied by `max_price_per_unit` in `ValidResourceBounds::max_possible_fee`:

```rust
l2_gas.max_amount.saturating_mul(l2_gas.max_price_per_unit.saturating_add(tip.into()))
``` [2](#0-1) 

For non-Declare transactions the cap is enforced at the stateless gate. For Declare transactions it is not, so a Declare with `l2_gas.max_amount = u64::MAX` and `l2_gas.max_price_per_unit = min_gas_price` produces a `max_possible_fee` that saturates to `Fee::MAX`. The stateless validator passes it; the stateful validator's `verify_can_pay_committed_bounds` then checks whether the account balance covers `Fee::MAX` and rejects it — but only after the more expensive stateful validation path has been entered.
