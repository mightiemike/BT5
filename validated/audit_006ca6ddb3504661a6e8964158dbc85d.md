### Title
`ENABLE_SECP_OPS` Flag Bypass via 4-Byte Opcode Forms Allows Premature Secp Execution - (File: `src/chia_dialect.rs`)

### Summary

`ChiaDialect::op` in `src/chia_dialect.rs` dispatches secp256k1/secp256r1 verification through two separate code paths. The 1-byte opcode path (opcodes 64 and 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (`0x13d61f00` / `0x1c3a8f00`) dispatches to the same real secp implementations with no flag check at all. An attacker-controlled CLVM program using the 4-byte opcode forms executes live secp cryptographic verification regardless of whether `ENABLE_SECP_OPS` is set, bypassing the hard-fork gate entirely.

### Finding Description

`ChiaDialect::op` has two distinct dispatch branches for secp operations.

**Branch 1 — 1-byte opcodes (correctly gated):** [1](#0-0) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

When `ENABLE_SECP_OPS` is absent, opcodes 64 and 65 fall through to `unknown_operator` — a no-op with cost in lenient mode, or an error in strict mode.

**Branch 2 — 4-byte opcodes (ungated):** [2](#0-1) 

```rust
if op_len == 4 {
    // these are unknown operators with assigned cost
    ...
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // no ENABLE_SECP_OPS check
        0x1c3a8f00 => op_secp256r1_verify,   // no ENABLE_SECP_OPS check
        _ => { return unknown_operator(...); }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

The comment explicitly labels this block "unknown operators with assigned cost," meaning these opcodes are supposed to behave as no-ops with a deterministic cost when the corresponding hard fork has not activated. Instead, they unconditionally call the real secp implementations.

Neither `op_secp256k1_verify` nor `op_secp256r1_verify` checks `ENABLE_SECP_OPS` internally — both accept `_flags: ClvmFlags` and ignore it: [3](#0-2) [4](#0-3) 

The flag is defined as a hard-fork guard: [5](#0-4) 

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

`MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`: [6](#0-5) 

### Impact Explanation

Before the secp hard fork activates (i.e., `ENABLE_SECP_OPS` is not set), any caller — including the Chia mempool and consensus engine — that runs a CLVM program containing the 4-byte opcode `0x13d61f00` or `0x1c3a8f00` will execute live secp256k1 or secp256r1 signature verification. The expected behavior is that these opcodes are treated as unknown no-ops with a fixed cost.

Concrete broken invariant: a program that uses the 4-byte secp opcode forms succeeds or fails based on cryptographic signature validity even when `ENABLE_SECP_OPS` is absent. The same program using the 1-byte forms (64/65) would be a no-op or error. This is a **consensus divergence** between the two opcode encodings of the same logical operation, and between nodes that have and have not activated the hard fork.

### Likelihood Explanation

The entry path is direct: any attacker who can submit a CLVM program (e.g., a coin spend) can craft the 4-byte opcode bytes `\x13\xd6\x1f\x00` or `\x1c\x3a\x8f\x00` as the operator atom. This is fully attacker-controlled input. No special privileges, compromised nodes, or social engineering are required. The 4-byte encoding is documented in the cost formula comment in the same function, making it discoverable.

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the 1-byte path:

```rust
// in the op_len == 4 branch:
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

When `ENABLE_SECP_OPS` is not set, the 4-byte secp opcodes fall through to `unknown_operator`, which applies the standard unknown-op cost formula (a no-op in lenient mode, an error in strict mode), consistent with the stated intent of the "unknown operators with assigned cost" block.

### Proof of Concept

```rust
use clvmr::allocator::Allocator;
use clvmr::chia_dialect::{ChiaDialect, ClvmFlags};
use clvmr::run_program::run_program;

fn main() {
    let mut allocator = Allocator::new();

    // Build (0x13d61f00 pubkey msg sig) — 4-byte secp256k1_verify opcode
    // without ENABLE_SECP_OPS set.
    // In a real exploit, supply a valid pubkey/msg/sig triple.
    // The operator executes regardless of the missing flag.

    // Dialect with NO ENABLE_SECP_OPS
    let dialect = ChiaDialect::new(ClvmFlags::empty());

    // Opcode atom: 4 bytes = 0x13d61f00
    let op = allocator.new_atom(&[0x13, 0xd6, 0x1f, 0x00]).unwrap();

    // With a valid (pubkey, msg, sig) triple this call succeeds,
    // proving secp verification runs without ENABLE_SECP_OPS.
    // Expected: treated as unknown no-op.
    // Actual:   op_secp256k1_verify executes.
    let _ = run_program(&mut allocator, &dialect, op, allocator.nil(), u64::MAX);
}
```

The 1-byte opcode 64 under the same `ClvmFlags::empty()` dialect falls through to `unknown_operator` and returns a no-op result, confirming the inconsistency between the two dispatch paths.
