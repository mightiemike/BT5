### Title
Storage Deposit Permanently Locked After Garbage Collection Frees Block Headers - (File: `contract/src/lib.rs`)

---

### Summary

`submit_blocks` collects a NEAR storage deposit from the caller to cover the on-chain storage cost of new block headers. When `run_mainchain_gc` later removes those same headers, the storage bytes are freed and the staked NEAR is released back to the contract's own account balance. No mechanism exists to return these freed tokens to the original depositors or to any designated recipient, so they accumulate permanently inside the contract.

---

### Finding Description

`submit_blocks` is `#[payable]` and enforces a storage deposit:

```rust
let initial_storage = env::storage_usage();
// ... headers inserted ...
self.run_mainchain_gc(num_of_headers);
let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());
require!(amount >= required_deposit, ...);
let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id()).transfer(refund).into()
}
```

The `required_deposit` portion is retained by the contract to cover storage staking on NEAR. Later, `run_mainchain_gc` removes the oldest mainchain headers:

```rust
self.remove_block_header(blockhash);
self.mainchain_height_to_header.remove(&height);
```

On NEAR Protocol, removing storage entries releases the staked NEAR back to the contract's account balance. However, `run_mainchain_gc` performs no transfer after freeing storage — it simply deletes the entries and returns. There is no withdrawal function, no treasury address, and no per-depositor accounting anywhere in the contract. The released NEAR tokens are permanently stranded in the contract's balance.

---

### Impact Explanation

**Impact: Medium**

Every relayer that has ever paid a storage deposit for headers that were subsequently garbage-collected loses those NEAR tokens permanently. With the recommended `gc_threshold = 52704` (approximately one year of Bitcoin blocks), the contract continuously cycles through headers: new ones are paid for, old ones are GC'd, and the freed deposits accumulate in the contract with no recovery path. Over the operational lifetime of the contract, this constitutes a material, irrecoverable loss of NEAR tokens for relayers.

---

### Likelihood Explanation

**Likelihood: High**

GC is a core, designed feature that runs automatically on every `submit_blocks` call once the mainchain exceeds `gc_threshold`. It is not an edge case — it is the steady-state behavior of a live deployment. Every relayer operating past the GC threshold will be affected on every submission batch that triggers pruning.

---

### Recommendation

Track the freed storage bytes inside `run_mainchain_gc` and transfer the corresponding NEAR value to a designated recipient (e.g., a treasury address stored in contract state, or back to `env::predecessor_account_id()` when called standalone):

```rust
pub fn run_mainchain_gc(&mut self, batch_size: u64) -> PromiseOrValue<()> {
    let storage_before = env::storage_usage();
    // ... existing removal logic ...
    let freed_bytes = storage_before.saturating_sub(env::storage_usage());
    let freed_near = env::storage_byte_cost().saturating_mul(freed_bytes.into());
    if freed_near > NearToken::from_near(0) {
        Promise::new(self.treasury).transfer(freed_near).into()
    } else {
        PromiseOrValue::Value(())
    }
}
```

Alternatively, maintain a per-depositor accounting map so freed storage can be refunded to the original submitter.

---

### Proof of Concept

1. Relayer calls `submit_blocks` with 85 headers, attaches `85 * storage_byte_cost * header_size` NEAR. The contract retains `required_deposit` and refunds only the excess.
2. The mainchain grows past `gc_threshold`. On the next `submit_blocks` call, `run_mainchain_gc(85)` removes 85 old headers.
3. NEAR storage staking for those 85 headers is released back to the contract's balance.
4. No transfer is issued. The freed NEAR is now permanently locked.
5. Confirm: search the entire contract for any `withdraw`, `transfer`, or `storage_deposit` refund path — none exists. [1](#0-0) [2](#0-1) [3](#0-2)

    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
