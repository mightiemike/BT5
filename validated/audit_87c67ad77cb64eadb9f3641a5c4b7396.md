### Title
Unprivileged Caller Can Drain the Canonical Chain Index via `run_mainchain_gc` - (File: contract/src/lib.rs)

### Summary
`run_mainchain_gc` is a public, mutable contract function that carries no caller-identity restriction when the contract is not paused. Any unprivileged NEAR account can call it with an arbitrarily large `batch_size`, removing all GC-eligible mainchain block headers from the canonical index in a single transaction. This corrupts the shared chain state that every downstream SPV consumer depends on and can permanently prevent legitimate chain reorganizations.

### Finding Description
`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. [1](#0-0) 

The `#[pause]` attribute from `near-plugins` restricts access only when the contract is **paused**; during normal (unpaused) operation it imposes no caller check at all. Compare this with `submit_blocks`, which carries the additional `#[trusted_relayer]` guard that enforces an active-relayer or bypass-role check regardless of pause state. [2](#0-1) 

Inside `run_mainchain_gc`, the only bound on how many blocks are removed is:

```
selected_amount_to_remove = min(total_amount_to_remove, batch_size)
``` [3](#0-2) 

Because `batch_size` is fully attacker-controlled, passing `u64::MAX` removes every block currently eligible for GC in one call. The function then:
- Deletes entries from `mainchain_height_to_header` and `headers_pool`
- Removes entries from `mainchain_header_to_height`
- Advances `mainchain_initial_blockhash` to the new oldest block [4](#0-3) 

### Impact Explanation
Two concrete state corruptions result:

**1. SPV verification forgery / denial for GC-eligible blocks.**
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the target block in `mainchain_header_to_height` to confirm it belongs to the canonical chain. [5](#0-4) 

Once an attacker GC's a block, that lookup panics with "block does not belong to the current main chain." Any downstream NEAR contract that gates a fund release on a successful SPV proof for a block within the GC-eligible window can be permanently blocked by an attacker who front-runs the proof call with a `run_mainchain_gc(u64::MAX)` call.

**2. Permanent reorg prevention.**
`reorg_chain` walks the fork chain back to the common ancestor by following `prev_block_hash` links through `headers_pool`. [6](#0-5) 

If the common ancestor has been GC'd, the walk panics with `PrevBlockNotFound`. An attacker who observes a relayer submitting a high-chainwork fork can front-run the final block submission with `run_mainchain_gc`, causing the reorg to fail. The lower-chainwork chain remains canonical, breaking the fundamental Bitcoin consensus invariant that the contract is supposed to enforce. The CLAUDE.md explicitly acknowledges this failure mode but treats it as an operational concern, not a security boundary. [7](#0-6) 

### Likelihood Explanation
The attack requires no privileged keys, no staking, and no special role. Any NEAR account can call `run_mainchain_gc` at any time the contract is not paused. The call costs only gas. A bot can monitor the NEAR mempool for pending `submit_blocks` or `verify_transaction_inclusion` calls and front-run them. The SPV-blocking variant is immediately exploitable against any downstream contract that relies on the light client for fund gating.

### Recommendation
Add an explicit caller restriction to `run_mainchain_gc` that mirrors the `#[trusted_relayer]` guard used on `submit_blocks`, or restrict it to accounts holding a designated role (e.g., `RelayerManager` or `DAO`). The `#[pause]` decorator alone is insufficient because it only gates access when the contract is paused, not during normal operation. The internal call from `submit_blocks` can bypass the guard via the same bypass-role mechanism already used for `UnrestrictedSubmitBlocks`.

### Proof of Concept
```
1. Deploy the contract with gc_threshold = 100 and submit 200 mainchain blocks
   (blocks 0–199 are stored; blocks 0–99 are GC-eligible).

2. A downstream contract calls verify_transaction_inclusion for block at height 50.
   This call is visible in the NEAR mempool.

3. Attacker front-runs with:
     near call <contract> run_mainchain_gc '{"batch_size": 18446744073709551615}' --accountId attacker.near

4. run_mainchain_gc removes blocks 0–99 from mainchain_header_to_height and headers_pool,
   advancing mainchain_initial_blockhash to block 100.

5. The downstream contract's verify_transaction_inclusion call now panics:
   "block does not belong to the current main chain"
   — the SPV proof permanently fails, blocking any fund release gated on it.

For the reorg-prevention variant:
1. A relayer is submitting a fork (blocks F1…F10) that has more chainwork than the mainchain.
   The common ancestor is at height 50 (GC-eligible).
2. Attacker calls run_mainchain_gc(u64::MAX) before the fork tip F10 is submitted.
3. When F10 is submitted, reorg_chain walks back through F9…F1 and then tries to fetch
   the common ancestor from headers_pool — panics with PrevBlockNotFound.
4. The reorg fails; the lower-chainwork mainchain remains canonical.
```
