### Title
Unprotected `initialize()` on ProxyManager Proxy Allows Attacker to Seize Upgrade Authority Over All Core Contracts - (File: `core/contracts/BaseProxyManager.sol`)

### Summary
`BaseProxyManager.initialize()` has no caller restriction. If the ProxyManager proxy is deployed without atomically calling `initialize()`, any attacker who observes the deployment transaction can front-run the initialization, claim ownership and the `submitter` role, and subsequently upgrade `Clearinghouse` and `Endpoint` to malicious implementations — draining all user funds.

### Finding Description
`BaseProxyManager.initialize()` is declared `external initializer` with no access control: [1](#0-0) 

The first caller becomes `owner` (via `__Ownable_init()`) and `submitter`. There is no check binding the caller to the deployer address, unlike `ContractOwner.initialize()` which explicitly enforces `require(_deployer == msg.sender, "expected deployed to initialize")`: [2](#0-1) 

The constructor correctly calls `_disableInitializers()` to protect the bare implementation: [3](#0-2) 

However, this only protects the implementation address. The proxy itself must still be initialized in a separate transaction, creating a front-runnable window between proxy deployment and `initialize()` execution.

Once an attacker owns the ProxyManager, they control the entire upgrade path. `migrateAll()` is `onlyOwner` and calls `ITransparentUpgradeableProxy(proxy).upgradeTo(newImpl.impl)` directly on the Clearinghouse and Endpoint proxies: [4](#0-3) [5](#0-4) 

The attacker also controls `submitImpl()` (gated to `submitter`) to stage the malicious implementation: [6](#0-5) 

### Impact Explanation
An attacker who seizes ProxyManager ownership can:
1. Call `submitImpl("Clearinghouse", maliciousImpl)` and `submitImpl("Endpoint", maliciousImpl)` as the self-appointed `submitter`.
2. Call `migrateAll()` as owner to upgrade both proxies to malicious implementations.
3. The malicious `Clearinghouse` can redirect `withdrawCollateral` to the attacker, or the malicious `Endpoint` can bypass all sequencer and verifier checks.

All user collateral held in the Clearinghouse/SpotEngine and all pending withdrawals in the WithdrawPool are at risk. The `ProxyManagerHelper` registered during the attacker-controlled `initialize()` would also be attacker-controlled, giving them the `upgradeClearinghouseLiq` and `upgradeEndpointTx` paths as well: [7](#0-6) [8](#0-7) 

### Likelihood Explanation
The attack requires monitoring the mempool for the ProxyManager proxy deployment transaction and submitting a competing `initialize()` call with higher gas before the deployer's initialization lands. On Ink Chain (an L2 with a sequencer), strict transaction ordering reduces but does not eliminate this risk — the sequencer itself could be a threat actor, or the deployment could occur during a sequencer downtime window where transactions are submitted to L1 directly. The deployment window is a known, well-documented attack surface for upgradeable proxy patterns.

### Recommendation
Add a deployer-binding check to `BaseProxyManager.initialize()`, mirroring the pattern already used in `ContractOwner.initialize()`:

```solidity
function initialize(address expectedOwner) external initializer {
    require(msg.sender == expectedOwner, "unauthorized initializer");
    __Ownable_init();
    submitter = msg.sender;
    proxyManagerHelper = new ProxyManagerHelper();
}
```

Alternatively, combine proxy deployment and initialization into a single atomic transaction using a factory or deployment script that calls the proxy constructor with encoded `initialize()` calldata (the standard OpenZeppelin `TransparentUpgradeableProxy` constructor accepts `_data` for this purpose).

### Proof of Concept
1. Deployer broadcasts: `deploy TransparentUpgradeableProxy(proxyManagerImpl, proxyAdmin, "")` — proxy deployed, `initialize()` not yet called.
2. Attacker observes the pending deployment in the mempool and broadcasts `proxyManagerProxy.initialize()` with higher gas.
3. Attacker's transaction is included first: `owner = attacker`, `submitter = attacker`, `proxyManagerHelper` is an attacker-controlled instance.
4. Attacker calls `submitImpl("Clearinghouse", maliciousImpl)` and `submitImpl("Endpoint", maliciousImpl)`.
5. Attacker calls `migrateAll([{name:"Clearinghouse", impl:maliciousImpl}, {name:"Endpoint", impl:maliciousImpl}])`.
6. Both core proxies now point to attacker-controlled logic; all user funds are accessible to the attacker. [1](#0-0) [9](#0-8)
