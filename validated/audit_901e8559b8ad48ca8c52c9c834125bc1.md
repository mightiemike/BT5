### Title
`SwapAllowlistExtension` gates the direct pool caller (`msg.sender`) as `sender`, not the end user — allowlisted EOAs are permanently locked out of swapping on curated pools - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`MetricOmmPool.swap()` passes `msg.sender` (the direct caller of the pool) as `sender` to `_beforeSwap()`. `SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted. When a user swaps through `MetricOmmSimpleRouter`, `sender` equals the router address, not the user. Because EOAs cannot implement `IMetricOmmSwapCallback` they cannot call `pool.swap()` directly either. The result is that allowlisted EOAs can never execute a swap on any pool guarded by `SwapAllowlistExtension`.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension dispatcher.**

In `MetricOmmPool.swap()`:

```solidity
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap(), not the end user
    recipient,
    ...
);
``` [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument.** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `sender` against the per-pool allowlist.**

```solidity
function beforeSwap(address sender, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [3](#0-2) 

**Step 4 — The router is the only viable swap path for EOAs.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` with the router as `msg.sender`: [4](#0-3) 

`pool.swap()` immediately calls back `IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(...)`: [5](#0-4) 

EOAs cannot implement `IMetricOmmSwapCallback`, so they **must** route through `MetricOmmSimpleRouter`. When they do, `sender = router address`. The router is not in the per-user allowlist, so `allowedSwapper[pool][router] == false` and the check reverts with `NotAllowedToSwap`.

**The two paths available to an allowlisted EOA:**

| Path | `sender` seen by extension | Result |
|---|---|---|
| Direct `pool.swap()` | EOA address (allowlisted ✓) | Reverts in callback — EOA cannot implement `IMetricOmmSwapCallback` |
| Via `MetricOmmSimpleRouter` | Router address (not allowlisted ✗) | Reverts with `NotAllowedToSwap` |

There is no reachable path through which an allowlisted EOA can successfully swap.

---

### Impact Explanation

Any pool that deploys `SwapAllowlistExtension` with per-address allowlisting is completely unusable for allowlisted EOAs. The swap flow — the core revenue-generating action of the protocol — is permanently broken for the intended privileged users. The pool admin cannot fix this without either (a) allowlisting the router address (which opens the pool to all users, defeating the allowlist entirely) or (b) requiring users to deploy custom callback contracts.

This matches the allowed impact gate: **broken core pool functionality causing an unusable swap flow**.

---

### Likelihood Explanation

Every pool that uses `SwapAllowlistExtension` with individual address allowlisting (the primary use case documented for the extension) will exhibit this behavior for every allowlisted EOA. No special attacker action is required; the breakage is structural and triggered by the normal user flow through the supported periphery.

---

### Recommendation

The extension must gate the economically relevant actor, not the direct pool caller. The cleanest fix is to have the pool admin allowlist the router and have the router forward the original `msg.sender` in `extensionData`, which the extension decodes and checks. Alternatively, the extension can be redesigned to check `recipient` (the address receiving output tokens) if that matches the intended gating semantics. At minimum, the `SwapAllowlistExtension` documentation must warn that allowlisted addresses must be direct pool callers (i.e., smart contracts implementing `IMetricOmmSwapCallback`), not EOAs.

---

### Proof of Concept

```solidity
// Pool is configured with SwapAllowlistExtension.
// Pool admin allowlists Alice (an EOA).
swapExtension.setAllowedToSwap(address(pool), alice, true);

// Alice tries to swap through the router (the only path available to an EOA).
vm.prank(alice);
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: alice,
    tokenIn: token1,
    zeroForOne: false,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// Reverts: NotAllowedToSwap
// Because pool.swap() is called by the router, so sender = router address.
// allowedSwapper[pool][router] == false → revert.

// Alice cannot call pool.swap() directly either:
// pool.swap() calls IMetricOmmSwapCallback(alice).metricOmmSwapCallback(...)
// Alice is an EOA → no code → revert.
// Alice is permanently locked out.
``` [6](#0-5) [7](#0-6) [8](#0-7)
