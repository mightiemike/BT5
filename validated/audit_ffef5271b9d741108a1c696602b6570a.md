### Title
Cross-Channel Refund Mint Fails Due to Denom Creator Mismatch, Causing Permanent NFT Loss — (`x/nft-transfer/keeper/packet.go`, `x/nft/keeper/keeper.go`)

---

### Summary

When a sink-chain voucher denom is created via one IBC channel and the NFT is later sent back (and burned) via a **different** channel, a timeout/refund will attempt to re-mint the voucher using the new channel's escrow address as the `sender` to `MintNFT`. Because `MintNFT` enforces `IsDenomCreator`, which compares against the **original** channel's escrow address stored at denom creation time, the check fails with `ErrUnauthorized`. The NFT was already burned in `createOutgoingPacket` (via `BurnNFTUnverified`, which has no denom-creator check), so the asset is permanently destroyed with no recovery path.

---

### Finding Description

**Step 1 — Denom creation on the sink chain (channel-0)**

When chain B first receives an NFT from chain A over `channel-0`, `processReceivedPacket` issues the voucher denom:

```go

// escrowAddress = GetEscrowAddress("nft", "channel-0")  →  escrow0
if !k.nftKeeper.HasDenomID(ctx, voucherClassID) {
    k.nftKeeper.IssueDenom(ctx, voucherClassID, ..., escrowAddress)
}
``` [1](#0-0) 

The denom's `Creator` field is permanently set to `escrow0 = GetEscrowAddress("nft", "channel-0")`.

**Step 2 — User sends NFT back via channel-1 (new channel after channel-0 is closed)**

`createOutgoingPacket` evaluates direction:

```go

// fullClassPath = "nft/channel-0/originalClass"
// IsAwayFromOrigin("nft", "channel-1", "nft/channel-0/originalClass")
//   → !strings.HasPrefix("nft/channel-0/...", "nft/channel-1/") → false
``` [2](#0-1) 

Because `isAwayFromOrigin = false`, the chain is treated as the sink and the NFT is **burned** via `BurnNFTUnverified`:

---

### Impact Explanation

The NFT is permanently destroyed. The user loses a potentially valuable asset with no on-chain recovery mechanism. The invariant that "a timeout/refund must always restore the sender's asset" is broken for any sink-chain voucher sent back over a channel different from the one that created the denom.

---

### Likelihood Explanation

This requires:
1. An NFT to have been received over channel-0 (creating the denom with `escrow0`).
2. Channel-0 to be closed (possible via normal IBC channel lifecycle or counterparty chain upgrade).
3. A new channel-1 to be opened between the same two chains.
4. The user to attempt to send the NFT back via channel-1 (a natural action — the user has no reason to know the channel matters for the denom creator).
5. The packet to time out or the channel to close before delivery.

All five conditions are reachable through normal IBC operations without any privileged access.

---

### Recommendation

Replace `MintNFT` (which enforces `IsDenomCreator`) with `MintNFTUnverified` in the `refundPacketToken` sink path, mirroring the pattern already used in `BurnNFTUnverified` for the outgoing direction. The IBC module itself is the authoritative caller here; the denom-creator check is appropriate for user-facing `MsgMintNFT` but not for protocol-internal refund paths. Alternatively, store the escrow address used at denom creation time and update it when a new channel is used, or use a module-level authority address as the denom creator instead of a channel-specific escrow address.

---

### Proof of Concept

```
1. Chain A sends NFT "class1/token1" to chain B over channel-0.
   → chain B: IssueDenom(voucherClassID, creator=escrow("nft","channel-0"))
   → chain B: MintNFT(voucherClassID, "token1", owner=userB)

2. channel-0 is closed. channel-1 is opened (same chains).

3. userB sends voucherClassID/token1 back to chain A via channel-1.
   → createOutgoingPacket: IsAwayFromOrigin("nft","channel-1","nft/channel-0/class1") = false
   → BurnNFTUnverified(voucherClassID, "token1", userB)  ← succeeds, no creator check
   → packet committed with seq=1, sourceChannel="channel-1"

4. Packet times out (or channel-1 closes before delivery).
   → OnTimeoutPacket → refundPacketToken
   → escrowAddress = GetEscrowAddress("nft","channel-1") = escrow1
   → IsAwayFromOrigin = false → MintNFT(voucherClassID,"token1","","uri","",escrow1,userB)
   → IsDenomCreator(voucherClassID, escrow1) → FAIL (creator=escrow0)
   → ErrUnauthorized returned; tx reverts

5. token1 is gone. userB has no NFT. No recovery path exists.
```

// BurnNFTUnverified deletes a specified NFT without verifying if the owner is the creator of denom
// Needed for IBC transfer of NFT
func (k Keeper) BurnNFTUnverified(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
```
