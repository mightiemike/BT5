Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Allowing Any Unprivileged Caller to Bypass the Deposit Allowlist — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and checks only `owner` (a caller-supplied parameter). Because any caller can freely choose any `owner` address, an unprivileged address can name an already-authorized owner to pass the allowlist check, pay tokens into the pool via the callback, and alter the pool's bin liquidity distribution — all without being on the allowlist. The parallel `SwapAllowlistExtension` correctly gates on `sender`, confirming the asymmetry is a defect.

## Finding Description
`MetricOmmPool.addLiquidity` passes both the real caller and the caller-chosen owner to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but leaves it unnamed and unused, checking only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

An attacker who is **not** on the allowlist calls `addLiquidity(owner = any_authorized_address, ...)`. The extension sees an authorized `owner`, passes the check, and the pool proceeds to pull tokens from the attacker via the liquidity callback and mint a position for the named owner. The attacker cannot reclaim the tokens because `removeLiquidity` enforces `msg.sender == owner`: [3](#0-2) 

`SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller), confirming the deposit extension's behavior is unintentional: [4](#0-3) 

## Impact Explanation
The deposit allowlist access-control invariant is broken for every pool configured with this extension. Any unprivileged address can add arbitrary liquidity to any bin, shifting the pool's depth profile and effective bid/ask spread for all subsequent swaps. This constitutes an admin-boundary break: the pool admin's explicit exclusion of an address from the allowlist is rendered ineffective by an unprivileged call path. Secondary effects include forcing positions onto authorized LP addresses without their consent and enabling cross-market manipulation where the attacker accepts the token cost as a manipulation fee (tokens are irrecoverably gifted to the named owner).

## Likelihood Explanation
The bypass requires no special privilege — any EOA or contract can call `addLiquidity`. The only prerequisite is knowing one authorized owner address, which is discoverable on-chain from past `addLiquidity` transactions or from the `allowedDepositor` mapping directly. The attacker must supply real tokens, but those tokens are not lost to the protocol; they are credited to the named owner, making the attack economically rational whenever manipulation value exceeds token cost. The attack is repeatable indefinitely.

## Recommendation
Replace the ignored first parameter with a named `sender` and gate on it, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate on the position owner (who will hold the LP shares), keep the `owner` check but add an explicit `sender` check as well, so both the caller and the beneficiary must be authorized.

## Proof of Concept
1. Deploy a pool with `DepositAllowlistExtension` configured.
2. Pool admin calls `setAllowedToDeposit(pool, authorizedLP, true)` — `authorizedLP` is on the allowlist; `attacker` is not.
3. `attacker` deploys `AttackerRouter` implementing `IMetricOmmModifyLiquidityCallback`, pre-funded with token0/token1.
4. `attacker` calls `pool.addLiquidity(authorizedLP, 0, deltas, "", "")`.
5. Extension checks `allowedDepositor[pool][authorizedLP]` → `true` → passes.
6. Pool pulls tokens from `attacker` via callback; mints position shares for `authorizedLP`.
7. Verify: `pool.position(id, authorizedLP)` shows non-zero shares `authorizedLP` never requested; pool bin totals reflect attacker-supplied liquidity; `attacker` cannot call `removeLiquidity` (reverts `NotPositionOwner`).
