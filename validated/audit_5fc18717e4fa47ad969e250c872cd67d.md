Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Allowing Any Unprivileged Caller to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[pool][sender]`. When `MetricOmmSimpleRouter` is the direct caller of `pool.swap`, the pool forwards `sender = router` to the extension. A pool admin who wants allowlisted users to access the pool via the router must add the router to the allowlist, which unconditionally opens the pool to every caller of the public router contract.

## Finding Description

`SwapAllowlistExtension.beforeSwap` performs its check as:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (enforced by `onlyPool` on other setters, and the extension is called only by the pool via `CallExtension.callExtension`). `sender` is whatever the pool received as `msg.sender` when `pool.swap` was called.

`ExtensionCalling._beforeSwap` passes `sender` directly from the pool's `msg.sender` context:

```solidity
// ExtensionCalling.sol L149-177
function _beforeSwap(address sender, ...) internal {
    _callExtensionsInOrder(BEFORE_SWAP_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeSwap,
            (sender, recipient, ...)));
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the `msg.sender` seen by the pool:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The same applies to `exactInput` (L104-112), `exactOutputSingle` (L136-137), and every hop of `exactOutput` (L165-181, L220-228).

**Exploit path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` as the `beforeSwap` hook.
2. Admin adds Alice (a vetted user) to `allowedSwapper[pool][alice]`.
3. Admin also adds the router to `allowedSwapper[pool][router]` so Alice can use the router.
4. Bob (non-allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` → pool sees `msg.sender = router` → passes `sender = router` to `_beforeSwap`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → passes.
7. Bob's swap executes at the oracle-anchored bid/ask price; Bob was never checked.

No existing guard prevents this. The `onlyPool` modifier on the extension only ensures the extension is called by a registered pool, not that the `sender` argument reflects the true originator.

## Impact Explanation

A pool protected by `SwapAllowlistExtension` is intended to restrict trading to a curated set of counterparties. Once the router is allowlisted (the only supported path for allowlisted users to use the router), the access-control invariant is completely broken: any unprivileged user can execute swaps at oracle-anchored prices. If the oracle price is stale, at a spread boundary, or during a volatile period, non-allowlisted users extract value from LPs who deposited under the assumption that only vetted counterparties could trade. This is a direct loss of LP principal and a broken core access-control invariant.

**Severity: High** — the bypass is unconditional once the router is allowlisted; no special privilege, token, or flash loan is required.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless periphery contract with no caller restrictions on `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput`.
- Pool admins who deploy a swap-allowlisted pool and want allowlisted users to use the router **must** add the router to the allowlist; there is no other supported path.
- Any attacker who observes `allowedSwapper[pool][router] = true` on-chain can immediately exploit the bypass with a standard router call.
- The condition is repeatable and requires no setup beyond a normal swap call.

**Likelihood: High.**

## Recommendation

Pass the originating user's address through the call path. Options in increasing robustness:

1. **Minimal fix:** Have the router encode `msg.sender` into `extensionData`; have `SwapAllowlistExtension.beforeSwap` decode and check that address when `sender` is a known router address.
2. **Robust fix:** Add an `originator` parameter to `IMetricOmmExtensions.beforeSwap` that the pool populates from a verified transient-storage slot written by the router before calling `swap`. The extension checks `originator` instead of (or in addition to) `sender`.

## Proof of Concept

```
Setup
─────
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin: allowedSwapper[pool][alice] = true  (vetted user)
3. Pool admin: allowedSwapper[pool][router] = true (so Alice can use router)

Attack
──────
4. Bob (non-allowlisted EOA) calls:
   MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: bob, ...})
5. Router calls pool.swap(bob, zeroForOne, amount, ...).
   → pool sees msg.sender = router → passes sender=router to _beforeSwap.
6. SwapAllowlistExtension.beforeSwap:
   allowedSwapper[pool][router] == true → check passes.
7. Bob's swap executes at oracle bid/ask price.

Result
──────
Bob, a non-allowlisted user, successfully swaps in a pool restricted to
vetted counterparties. The allowlist invariant is fully bypassed.
```
