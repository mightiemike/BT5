### Title
Unchecked Return Value of `transferFrom()` in `replaceUsdcEWithUsdc` Enables Draining usdcE from DDAs Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value. The function is callable by any unprivileged user. If the USDC token at the hardcoded address returns `false` on a failed transfer rather than reverting, an attacker can receive usdcE from any DDA without providing any USDC in exchange.

---

### Finding Description

`ContractOwner` declares `using ERC20Helper for IERC20Base` and the `ERC20Helper` library provides both `safeTransfer` and `safeTransferFrom` wrappers that check return values and revert on failure. Despite this, `replaceUsdcEWithUsdc` at line 616 calls the raw `IERC20Base.transferFrom()` directly, discarding its `bool` return value entirely:

```solidity
// line 616 — raw call, return value silently discarded
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// line 617 — DDA's usdcE is unconditionally pulled to ContractOwner
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));

// line 618 — usdcE is unconditionally sent to msg.sender using the safe wrapper
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
```

The outgoing usdcE transfer on line 618 correctly uses `safeTransfer`, but the incoming USDC pull on line 616 uses the raw interface method. The three steps are not atomic with respect to the USDC transfer's success: if `transferFrom` returns `false`, execution continues, the DDA's usdcE is withdrawn, and the caller receives it for free.

The `ERC20Helper.safeTransferFrom` that should have been used is defined at lines 23–42 of `ERC20Helper.sol` and would revert on a false return.

---

### Impact Explanation

An attacker targeting any subaccount whose DDA holds a non-zero usdcE balance can call `replaceUsdcEWithUsdc(subaccount)` and receive the full usdcE balance without transferring any USDC. The corrupted asset delta is: attacker gains `balance` usdcE, DDA loses `balance` usdcE, ContractOwner receives zero USDC. The DDA's usdcE holdings — which represent real user collateral awaiting credit — are drained.

---

### Likelihood Explanation

The function has no access control beyond `require(block.chainid == 57073)`, making it callable by any address on that chain. The USDC address is hardcoded to `0x2D270e6886d130D724215A266106e6832161EAEd` on chain 57073 (Ink mainnet). Standard Circle USDC reverts on failure, which would prevent exploitation under normal conditions. However, the code provides no safety net: if the token at that address ever behaves non-standardly (proxy upgrade, non-standard fork, or replacement), the vulnerability becomes immediately exploitable by any caller. The inconsistency — using `safeTransfer` for the outgoing leg but raw `transferFrom` for the incoming leg — is a concrete code defect regardless of current token behavior.

---

### Recommendation

Replace the raw `transferFrom` call with `ERC20Helper.safeTransferFrom`, consistent with how the rest of the codebase handles token transfers:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ERC20Helper.safeTransferFrom` is already available via the `using ERC20Helper for IERC20Base` directive declared at line 24 of `ContractOwner.sol`.

---

### Proof of Concept

1. A DDA for `subaccount` holds `N` usdcE (e.g., from a prior bridged deposit).
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` on chain 57073.
3. `balance = IERC20Base(usdcE).balanceOf(directDepositV1)` returns `N > 0`.
4. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, N)` is called. If USDC returns `false` (no revert), execution continues with no USDC transferred.
5. `DirectDepositV1(directDepositV1).withdraw(usdcE)` pulls all `N` usdcE from the DDA to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(attacker, N)` sends `N` usdcE to the attacker.
7. Attacker has gained `N` usdcE; the DDA and its depositor have lost `N` usdcE; no USDC was provided. [1](#0-0) [2](#0-1) [3](#0-2)

```text
    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

**File:** core/contracts/libraries/ERC20Helper.sol (L23-42)
```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
