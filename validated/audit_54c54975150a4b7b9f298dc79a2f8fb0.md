### Title
Linked Signer Can Overwrite Its Own Delegation Entry, Enabling Persistent Subaccount Takeover - (File: `core/contracts/EndpointTx.sol`)

### Summary

The `LinkSigner` fast-path transaction in `processTransactionImpl` is validated with `allowLinkedSigner = true`, meaning the **current linked signer** can sign a new `LinkSigner` transaction to replace itself with an attacker-controlled address. This is the on-chain analog of the "event-based" authentication bypass described in the external report: authentication is gated on a mutable state variable (`linkedSigners` mapping) that the delegated key itself can overwrite, rather than being cryptographically bound to the subaccount owner's key.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes the current linked signer as an accepted signer: [2](#0-1) 

`Verifier.validateSignature` accepts either the subaccount owner address **or** the linked signer address as a valid signer: [3](#0-2) 

After validation passes, the `linkedSigners` mapping is unconditionally overwritten: [4](#0-3) 

The `linkedSigners` mapping is the sole mutable state variable that controls the second accepted signer for every sensitive operation (withdrawals, transfers, liquidations, NLP mint/burn): [5](#0-4) 

This is structurally identical to the "event-based" authentication pattern in the external report: the authentication check reads from a mutable state variable (`linkedSigners`) rather than being cryptographically bound to the subaccount owner's key. The delegated key can rewrite that state variable, permanently redirecting authentication to an attacker-controlled address.

---

### Impact Explanation

A compromised linked signer key can:

1. Sign a `LinkSigner` transaction naming the attacker's address as the new linked signer.
2. Immediately use the new linked signer to sign `WithdrawCollateral` or `WithdrawCollateralV2` transactions, draining all collateral.

The subaccount owner's only immediate recourse is to submit a competing `LinkSigner` revocation via the fast path (sequencer-dependent) or via the slow mode path, which carries a hardcoded 3-day delay: [6](#0-5) 

During that window, the attacker's newly installed linked signer retains full signing authority over the subaccount. The corrupted state delta is `linkedSigners[victim_subaccount] = attacker_address`, which directly enables collateral theft via any subsequent signed withdrawal.

---

### Likelihood Explanation

Linked signers are the standard mechanism for automated trading bots and API keys in perpetual DEX architectures. Hot-wallet keys used for high-frequency signing are materially more likely to be compromised (phishing, server breach, key exfiltration) than the cold-wallet owner key. The attack requires only a single valid ECDSA signature from the compromised linked signer key — no privileged access, no governance capture, no sequencer compromise.

---

### Recommendation

Set `allowLinkedSigner = false` for the `LinkSigner` branch in `processTransactionImpl`. Changing the linked signer is a privileged administrative action that should require the subaccount owner's key, not the delegated key's key. The delegated key should be able to act *within* the scope of its delegation, but not *redefine* that scope.

```solidity
// In processTransactionImpl, LinkSigner branch:
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // owner key required; linked signer must not self-escalate
);
```

This mirrors the "result-based" authentication recommendation in the external report: bind the sensitive state-change operation to the root credential (owner key), not to the delegated credential (linked signer).

---

### Proof of Concept

1. Alice has `subaccount = 0xAlice...` with `linkedSigners[0xAlice...] = hotWallet`.
2. Attacker compromises `hotWallet`.
3. Attacker constructs `SignedLinkSigner { tx: { sender: 0xAlice..., signer: bytes32(attacker), nonce: N }, signature: hotWallet.sign(digest) }`.
4. Attacker submits this as a sequencer transaction. `validateSignedTx` passes because `recovered == hotWallet == getLinkedSigner(0xAlice...)`.
5. `linkedSigners[0xAlice...] = attacker` is written.
6. Attacker immediately signs `WithdrawCollateralV2 { sender: 0xAlice..., productId: USDC, amount: MAX, sendTo: address(0), nonce: N+1 }` with the `attacker` key.
7. `validateSignedTx` passes (`recovered == attacker == getLinkedSigner(0xAlice...)`), and Alice's collateral is transferred out. [7](#0-6)

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
