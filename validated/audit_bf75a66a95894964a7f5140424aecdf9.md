Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any User to Bypass Per-Pool Swap Allowlist via MetricOmmSimpleRouter â€” (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` equal to `msg.sender` of the pool's `swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, that value is the router's address, not the originating user. A pool admin who allowlists the router to enable router-mediated swaps for legitimate users simultaneously grants swap access to every user on the network, defeating the allowlist entirely.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` gates on that value as `sender`, with `msg.sender` being the pool: [2](#0-1) 

So the effective check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router `msg.sender` inside the pool: [3](#0-2) 

The same pattern holds for `exactInput` (all hops): [4](#0-3) 

For `exactOutputSingle`: [5](#0-4) 

And for `_exactOutputIterateCallback` (multi-hop exact output): [6](#0-5) 

Because `MetricOmmSimpleRouter` is a public, permissionless contract, `allowedSwapper[pool][router] = true` is functionally equivalent to `allowAllSwappers[pool] = true` for any user who routes through it. The existing guard in `beforeSwap` is insufficient because it checks the intermediary (router) rather than the economically relevant actor (end user).

## Impact Explanation
Any user not on the per-pool allowlist can trade in a curated pool by calling the router instead of the pool directly. LP providers who deployed a curated pool to restrict counterparties (e.g., KYC-only, institutional-only) lose that protection entirely. This constitutes broken core pool functionality: the allowlist â€” the primary access-control mechanism of `SwapAllowlistExtension` â€” is completely bypassed, potentially exposing LP funds to adversarial flow or violating compliance requirements.

## Likelihood Explanation
The `MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who enables router-mediated swaps for their allowlisted users (the expected operational setup) triggers the bypass automatically. No special knowledge or privileged access is required â€” any user can call `exactInputSingle` with the curated pool address. The condition is met in normal operation.

## Recommendation
Pass the originating user rather than the immediate caller as `sender` to the extension. Two options:

1. **Preferred:** Add a `swapper` parameter to `IMetricOmmPoolActions.swap()` that callers supply explicitly. The router passes `msg.sender` (the end user); direct callers pass `msg.sender` as well. The pool forwards this value to `_beforeSwap` instead of its own `msg.sender`.
2. **Alternative:** Have the router store the originating user in transient storage and expose a `currentSwapper()` view that the extension can read. The extension calls back into the router to retrieve the real user identity.

## Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   â€” intending to allow allowlisted users to use the router.
3. Pool admin calls setAllowedToSwap(pool, alice, true)
   â€” alice is the only intended swapper.
4. Bob (not allowlisted) calls:
     MetricOmmSimpleRouter.exactInputSingle({
       pool: curated_pool,
       recipient: bob,
       ...
     })
5. Router calls pool.swap(); msg.sender inside pool = router.
6. _beforeSwap passes sender = router to SwapAllowlistExtension.
7. Extension checks allowedSwapper[pool][router] == true â†’ passes.
8. Bob's swap executes despite not being on the allowlist.
```
