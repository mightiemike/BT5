### Title
Malicious IBC Counterparty Can Steal Escrowed NFTs via Crafted ClassId Prefix — (`x/nft-transfer/keeper/packet.go`)

---

### Summary

A malicious IBC counterparty chain can craft a `NonFungibleTokenPacketData.ClassId` that begins with its own source port/channel prefix (e.g., `nft-transfer/channel-0/realClass`). Because `processReceivedPacket` calls `IsAwayFromOrigin` using the **source** port/channel and the attacker-controlled `ClassId`, the function returns `false`, causing the unescrow path to execute instead of the mint path. If legitimate users have previously escrowed native NFTs on the receiving chain (by sending them to the attacker's chain), the attacker can drain those escrowed NFTs to an arbitrary receiver.

---

### Finding Description

`processReceivedPacket` determines the transfer direction with:

```go
isAwayFromOrigin := types.IsAwayFromOrigin(packet.GetSourcePort(), packet.GetSourceChannel(), data.ClassId)
``` [1](#0-0) 

`IsAwayFromOrigin` returns `false` (i.e., "back to origin") whenever `data.ClassId` starts with `sourcePort/sourceChannel/`:

```go
func IsAwayFromOrigin(sourcePort, sourceChannel, fullClassPath string) bool {
    prefixClassID := GetClassPrefix(sourcePort, sourceChannel)
    return !strings.HasPrefix(fullClassPath, prefixClassID)
}
``` [2](#0-1) 

The attacker controls the counterparty chain and therefore knows the exact source port (`nft-transfer`) and source channel (`channel-0`) for the established IBC channel. They also control `data.ClassId` freely — `ValidateBasic` only checks that it is non-blank:

```go
if strings.TrimSpace(nftpd.ClassId) == "" {
    return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
}
``` [3](#0-2) 

No format or prefix validation is applied to `ClassId`. By setting `ClassId = "nft-transfer/channel-0/realClass"`, the attacker makes `IsAwayFromOrigin` return `false`, and the unescrow branch executes:

```go
unprefixedClassID := types.RemoveClassPrefix(packet.GetSourcePort(),
    packet.GetSourceChannel(), data.ClassId)
// "nft-transfer/channel-0/realClass" → "realClass"

voucherClassID := types.ParseClassTrace(unprefixedClassID).IBCClassID()
// ParseClassTrace("realClass") → ClassTrace{Path:"", BaseClassId:"realClass"}
// IBCClassID() → "realClass"  (path is empty, no IBC hash prefix)

k.nftKeeper.TransferOwner(ctx, "realClass", tokenID, escrowAddress, receiver)
``` [4](#0-3) 

The escrow address used is `GetEscrowAddress(packet.GetDestPort(), packet.GetDestChannel())` — the same address that holds NFTs escrowed by legitimate users who previously sent their native `realClass` NFTs to the attacker's chain. [5](#0-4) 

---

### Impact Explanation

Any NFT of a native class (e.g., `realClass`) that has been escrowed on the victim chain — because a legitimate user sent it outbound to the attacker's chain — can be stolen. The attacker calls `TransferOwner` from the escrow address to an arbitrary receiver they control, permanently removing the NFT from the escrow and giving it to the attacker. The legitimate user's IBC voucher on the attacker's chain remains valid, meaning the attacker both holds the original NFT and retains the voucher — a double-spend of NFT ownership.

---

### Likelihood Explanation

The precondition is that at least one legitimate user has sent a native NFT from the victim chain to the attacker's chain (creating an escrow entry). This is the normal, intended use of the IBC NFT transfer module. The attacker needs only to:

1. Establish a valid IBC channel with the victim chain (standard IBC handshake, no privilege required).
2. Wait for any user to escrow an NFT via a normal outbound transfer.
3. Send a crafted packet with `ClassId = "<sourcePort>/<sourceChannel>/<nativeClass>"` and the victim's tokenID.

No governance, operator access, or key compromise is required. The exploit is executable by any chain that can open an IBC channel.

---

### Recommendation

In `processReceivedPacket`, validate that `data.ClassId` does **not** start with the source port/channel prefix before calling `IsAwayFromOrigin`. A packet arriving from chain A cannot legitimately claim that the class already carries chain A's own prefix — that prefix is only added by the **receiving** chain. Reject any packet where `ClassId` starts with `GetClassPrefix(packet.GetSourcePort(), packet.GetSourceChannel())`:

```go
if strings.HasPrefix(data.ClassId, types.GetClassPrefix(packet.GetSourcePort(), packet.GetSourceChannel())) {
    return types.ErrInvalidClassID.Wrap("classId must not be prefixed with source port/channel")
}
```

This mirrors the guard that the ICS-20 fungible token transfer module applies to denominations.

---

### Proof of Concept

```
Setup:
  Chain B (victim): native NFT class "realClass", tokenID "token1", owned by user Alice
  Chain A (attacker): IBC channel to Chain B, sourcePort="nft-transfer", sourceChannel="channel-0"

Step 1 — Alice sends her NFT from Chain B to Chain A (normal transfer):
  Chain B escrows "realClass"/"token1" at escrowAddress = GetEscrowAddress("nft-transfer", "channel-0")
  Chain A mints voucher ibc/{hash} to Alice

Step 2 — Attacker crafts malicious OnRecvPacket on Chain B:
  data.ClassId  = "nft-transfer/channel-0/realClass"
  data.TokenIds = ["token1"]
  data.Receiver = attacker_address
  packet.SourcePort    = "nft-transfer"
  packet.SourceChannel = "channel-0"
  packet.DestPort      = "nft-transfer"
  packet.DestChannel   = "channel-1"  (Chain B's channel)

Step 3 — Chain B processes the packet:
  IsAwayFromOrigin("nft-transfer", "channel-0", "nft-transfer/channel-0/realClass")
    → HasPrefix("nft-transfer/channel-0/realClass", "nft-transfer/channel-0/") = true
    → returns false  ← unescrow path taken

  RemoveClassPrefix → "realClass"
  ParseClassTrace("realClass").IBCClassID() → "realClass"
  TransferOwner("realClass", "token1", escrowAddress, attacker_address)  ← STOLEN

Result: Attacker holds Alice's original NFT. Alice's IBC voucher on Chain A is now unbacked.
```
