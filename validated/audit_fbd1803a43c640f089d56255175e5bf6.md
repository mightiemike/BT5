Audit Report

## Title
`DepositAllowlistExtension` Checks Caller-Controlled `owner` Instead of `sender`, Allowing Full Deposit Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual token-providing caller) and validates only `owner` (the LP position recipient), which is a free caller-supplied argument to `MetricOmmPool.addLiquidity`. Any unprivileged address can bypass the deposit allowlist entirely by naming any allowlisted address as `owner`, depositing unauthorized capital into a pool the admin intended to be permissioned.

## Finding Description

**Root cause — extension checks `owner`, not `sender`:**

In `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and discarded; the guard evaluates only `owner`: [1](#0-0) 

**`owner` is a free caller-supplied argument:**

`MetricOmmPool.addLiquidity` accepts `owner` as an arbitrary external parameter and passes `msg.sender` as `sender` and the caller-supplied `owner` verbatim to the hook: [2](#0-1) 

**The hook dispatch is correct — the bug is in the extension:**

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both `sender` and `owner` to the extension: [3](#0-2) 

**Token pull is on `msg.sender` (the actual caller), not `owner`:**

`LiquidityLib.addLiquidity` fires the callback on `msg.sender` of `addLiquidity` — the actual token provider — not on `owner`: [4](#0-3) 

This means `owner` only receives LP shares; the actual token pull targets the caller. The extension's check on `owner` therefore validates the wrong actor.

**Contrast with `SwapAllowlistExtension`, which correctly checks `sender`:** [5](#0-4) 

The swap guard checks the actual caller (`sender`). The deposit guard checks the position owner (`owner`). This asymmetry is the root cause.

**The periphery router (`MetricOmmPoolLiquidityAdder`) makes this trivially exploitable:**

`addLiquidityExactShares(pool, owner, ...)` only validates `owner != address(0)` and stores `msg.sender` as the payer. An unprivileged `bob` calls this with `owner = alice` (allowlisted); the router calls `pool.addLiquidity(alice, ...)`, the extension checks `alice` → passes, and the callback pulls tokens from `bob`. [6](#0-5) 

## Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for running a permissioned liquidity pool. Because the guard checks the wrong actor, it is fully bypassed: any unprivileged address can deposit into a pool configured to reject them, simply by naming any allowlisted address as `owner`. Unauthorized capital enters the pool, violating the invariant that only approved addresses can provide liquidity. This is a direct admin-boundary break: an unprivileged path circumvents the pool admin's configured access control with no special role or privilege required.

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call (directly or via `MetricOmmPoolLiquidityAdder`) with `owner` set to any address already on the allowlist. The allowlist mappings are public. No flash loans, reentrancy, or oracle manipulation are needed. Any address can execute the bypass in a single transaction.

## Recommendation

Replace the `owner` check with a `sender` check, consistent with how `SwapAllowlistExtension` operates:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the token provider and the position owner, check both `sender` and `owner`.

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension`; only `alice` is allowlisted via `setAllowedToDeposit(pool, alice, true)`.
2. `bob` (not allowlisted) calls `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, extensionData)`.
3. Router calls `pool.addLiquidity(owner=alice, ...)` with `msg.sender = LiquidityAdder`.
4. Pool calls `_beforeAddLiquidity(sender=LiquidityAdder, owner=alice, ...)`.
5. `DepositAllowlistExtension` evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
6. Pool executes liquidity addition; callback fires on `LiquidityAdder`, which pulls tokens from `bob` (the stored payer).
7. `alice` receives LP shares; `bob` has deposited into a pool that was supposed to reject them.
