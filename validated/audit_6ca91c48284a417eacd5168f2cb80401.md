### Title
Gateway `skip_stateful_validations` Allows Attacker to Front-Run Victim's Nonce-1 Invoke with Fake Signature During deploy_account Pending Window — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` UX feature, which allows a user to submit `deploy_account` + `invoke(nonce=1)` simultaneously by skipping the blockifier's `__validate__` entry point for the invoke, creates a front-running window. An attacker who observes a victim's pending `deploy_account` in the mempool can submit an `invoke(nonce=1)` for the victim's account address with a fake/invalid signature. Because signature validation is skipped (the account is not yet deployed), the attacker's transaction is accepted into the mempool. The victim's legitimate `invoke(nonce=1)` is then rejected with `DuplicateNonce`, disrupting the deploy-and-invoke flow.

---

### Finding Description

**The bridge-delay analog:**
In the external report, the "bridge delay" is the 10+ second window between a user's transaction being submitted on chain A and the cross-chain message arriving at chain B. An attacker observes the pending transaction on chain A and front-runs the identifier on chain B.

In the Sequencer, the equivalent delay is the window between when a `deploy_account` transaction is submitted to the Gateway and when it is actually executed by the Blockifier. During this window, the `skip_stateful_validations` feature is active for any `invoke(nonce=1)` targeting that account address.

**The vulnerable code path:**

`extract_state_nonce_and_run_validations` in `stateful_transaction_validator.rs` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`: [1](#0-0) 

`skip_stateful_validations` returns `true` (skip `__validate__`) when:
1. The transaction is an `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
4. `account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true` [2](#0-1) 

The fourth condition checks only whether the account address has **any** transaction in the mempool — it does not verify that the caller is the legitimate owner of the account: [3](#0-2) 

**Attack sequence:**

1. Victim submits `deploy_account` → accepted into mempool (account address is now in pool).
2. Attacker observes the pending `deploy_account` and submits `invoke(nonce=1, sender=victim_address, signature=[0,0], calldata=arbitrary)`.
3. Gateway stateless validation passes (signature format is a valid felt array).
4. `skip_stateful_validations` returns `true` because `victim_address` is in the mempool → `run_validate_entry_point` skips `__validate__` → attacker's fake-signature invoke is accepted into the mempool.
5. Victim submits `invoke(nonce=1)` with valid signature → `validate_fee_escalation` returns `DuplicateNonce` → rejected. [4](#0-3) 

6. Attacker's invoke fails at execution time (invalid signature → revert), but the victim's legitimate invoke was already rejected.

**Why existing protections do not cover this:**

The `validate_no_delayed_declare_front_run` guard protects only delayed **declare** transactions from front-running at the same `(address, nonce)`: [5](#0-4) 

There is no equivalent guard for the `skip_stateful_validations` invoke window. The mempool's `validate_by_mempool` call checks for duplicate hash and nonce range, but not signature validity: [6](#0-5) 

---

### Impact Explanation

**Matching impact:** *High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

- The gateway **accepts** an invalid transaction (attacker's `invoke` with a fake signature that would fail `__validate__`).
- The gateway **rejects** a valid transaction (victim's `invoke` with a correct signature) with `DuplicateNonce`.
- The victim's `deploy_account + invoke` UX flow is broken for at least one block until the attacker's reverted invoke is committed and cleaned up.
- Secondary: griefing with no profit motive, directly analogous to the external report's impact.

---

### Likelihood Explanation

- The `deploy_account` transaction is publicly visible in the mempool the moment it is accepted.
- The attacker needs only to observe the mempool, copy the victim's account address, and submit an `invoke(nonce=1)` with any signature before the victim does.
- No special privileges, keys, or funds are required beyond gas for the attacker's transaction.
- The `skip_stateful_validations` feature is active by default for all new account deployments. [7](#0-6) 

---

### Recommendation

Restrict the `skip_stateful_validations` bypass to transactions whose `tx_hash` was pre-registered alongside the `deploy_account` submission, or require the Gateway to verify that the `invoke(nonce=1)` was submitted by the same origin as the `deploy_account` (e.g., by storing the expected `tx_hash` of the paired invoke when the `deploy_account` is added to the mempool). Alternatively, mirror the `validate_no_delayed_declare_front_run` pattern: when a `deploy_account` is added to the mempool, record the `(address, nonce=1)` slot as reserved, and reject any `invoke(nonce=1)` for that address that does not match the pre-registered hash. [8](#0-7)

### Proof of Concept

```
Precondition: attacker monitors the mempool for new deploy_account transactions.

1. Victim calls POST /add_transaction with:
     deploy_account { class_hash, salt, constructor_calldata, nonce=0, signature=<valid> }
   → Gateway accepts; victim_address enters mempool.

2. Attacker calls POST /add_transaction with:
     invoke {
       sender_address: victim_address,   // copied from step 1
       nonce: 1,
       calldata: [<arbitrary>],
       signature: [0x0, 0x0],            // fake, any felt array passes stateless check
       resource_bounds: <minimal valid>
     }
   → Stateless validation: passes (signature is a valid felt array).
   → skip_stateful_validations: account_nonce=0, tx.nonce=1,
       account_tx_in_pool_or_recent_block(victim_address)=true → returns true.
   → run_validate_entry_point skips __validate__ → ACCEPTED into mempool.

3. Victim calls POST /add_transaction with:
     invoke { sender_address: victim_address, nonce: 1, signature: <valid>, ... }
   → validate_fee_escalation: (victim_address, nonce=1) already occupied
       → MempoolError::DuplicateNonce → REJECTED.

4. Block is built: deploy_account executes (account deployed), then attacker's
   invoke executes → __validate__ runs → signature invalid → REVERTED.
   Victim's invoke was never included.

5. Victim must resubmit invoke(nonce=1) in the next block.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
