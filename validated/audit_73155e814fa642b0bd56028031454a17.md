### Title
Attacker-controlled p2p peer can inject arbitrary `transaction_hash` values that are stored and served as canonical — (`crates/apollo_p2p_sync/src/client/transaction.rs`, `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The p2p sync client accepts `transaction_hash` directly from the untrusted protobuf wire field without recomputing or verifying it against the transaction body. The code itself contains an explicit acknowledgment of this gap. Because the header signature is also not cryptographically verified (only its length is checked), an attacker-controlled peer can inject a fully consistent fake block — fake header with fake `transaction_commitment`, fake transaction hashes — that passes all current guards and is committed to storage, after which the node serves the wrong hashes via RPC.

---

### Finding Description

**Step 1 — Protobuf deserialization takes `transaction_hash` verbatim.**

In `TryFrom<protobuf::TransactionInBlock>` for `(Transaction, TransactionHash)`:

```rust
let tx_hash = value
    .transaction_hash
    .clone()
    .ok_or(missing("Transaction::transaction_hash"))?
    .try_into()
    .map(TransactionHash)?;
``` [1](#0-0) 

The hash is taken from the wire field. No recomputation against the transaction body, no comparison with `calculate_transaction_hash(body, chain_id)`.

**Step 2 — The p2p sync client pushes the unverified hash directly into storage.**

```rust
block_body.transactions.push(transaction);
block_body.transaction_outputs.push(transaction_output);
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [2](#0-1) 

The TODO comment is the developers' own acknowledgment that this guard is missing.

**Step 3 — The block body (with the fake hash) is committed to storage unconditionally.**

```rust
storage_writer.begin_rw_txn()?.append_body(self.1, self.0)?.commit()?;
``` [3](#0-2) 

**Step 4 — The header is not cryptographically verified either.**

The only check on the `SignedBlockHeader` received from a p2p peer is that the block number matches and that exactly one signature is present:

```rust
if signed_block_header.signatures.len() != ALLOWED_SIGNATURES_LENGTH {
    return Err(ParseDataError::BadPeer(BadPeerError::WrongSignaturesLength { ... }));
}
``` [4](#0-3) 

The signature bytes are never verified against the block hash, and the block hash is never recomputed from the header fields. There is also an open TODO for parent-hash chain verification:

```
// TODO(shahak): Check that parent_hash is the same as the previous block's hash
// and handle reverts.
``` [5](#0-4) 

**Step 5 — No downstream cross-check between stored transaction hashes and the stored `transaction_commitment`.**

A grep across the entire codebase for `transaction_commitment.*verify`, `verify.*transaction_commitment`, and `check.*transaction_hash` returns zero production-code matches. The `transaction_commitment` stored in the header (which the attacker also controls) is never recomputed from the stored transaction hashes after sync.

**Step 6 — The wrong hashes are served authoritatively via RPC.**

`get_block_w_full_transactions` zips stored transactions with stored transaction hashes and returns them directly:

```rust
transactions
    .into_iter()
    .zip(transaction_hashes)
    .map(|(transaction, transaction_hash)| TransactionWithHash {
        transaction,
        transaction_hash,
    })
``` [6](#0-5) 

`get_transaction_by_hash` looks up by the stored (fake) hash, so a query for the canonical hash returns `TRANSACTION_HASH_NOT_FOUND`. [7](#0-6) 

---

### Impact Explanation

A node running in p2p-sync mode (i.e., with `p2p_sync_client_config` set) that connects to an attacker-controlled peer will store and serve wrong transaction hashes for every block that peer serves. Concretely:

- `starknet_getBlockWithTxs` returns transactions paired with attacker-chosen hashes.
- `starknet_getTransactionByHash` with the canonical hash returns "not found".
- Event and receipt queries that use the stored transaction hash as a key return wrong associations.
- Any downstream component (e.g., another node syncing from this node's p2p server) propagates the corruption.

This matches the allowed impact: **High — RPC execution / pending view returns an authoritative-looking wrong value**, and **High — transaction conversion or hash logic binds the wrong hash**.

---

### Likelihood Explanation

The p2p sync path is the production sync path for full nodes. Any peer the node dials (or that dials the node) can serve blocks. There is no allowlist or trust-level requirement beyond being a reachable libp2p peer. The attack requires no cryptographic capability — the attacker simply sets `transaction_hash` to an arbitrary `Felt` in the protobuf message. The TODO comment confirms the developers are aware the guard is absent.

---

### Recommendation

1. **Recompute and verify the hash at deserialization time.** In `TryFrom<protobuf::TransactionInBlock>`, after constructing `transaction`, call `transaction.calculate_transaction_hash(chain_id, &version)` and compare it to `tx_hash`. Reject with `ParseDataError::BadPeer` on mismatch.

2. **Alternatively, verify in `parse_data_for_block`.** Remove the TODO and add the check before `block_body.transaction_hashes.push(transaction_hash)` in `crates/apollo_p2p_sync/src/client/transaction.rs`.

3. **Verify the header signature cryptographically.** Use `verify_block_signature` (already present in `starknet_api::block`) against the sequencer's known public key before accepting a `SignedBlockHeader`.

4. **Cross-check `transaction_commitment` after body sync.** After writing the block body, recompute `calculate_transaction_commitment` from the stored hashes and compare it to the `transaction_commitment` stored in the header.
