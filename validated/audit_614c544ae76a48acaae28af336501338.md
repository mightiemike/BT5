### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Returns Forged Proof Results - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is explicitly marked `#[deprecated]` in the contract and superseded by `verify_transaction_inclusion_v2`, which was introduced to close the 64-byte transaction Merkle proof forgery vulnerability. Despite the deprecation, the old function remains a live, permissionless public entry point. Any unprivileged NEAR caller can invoke it directly with an adversarially crafted `tx_id` that is an internal Merkle tree node hash, causing the function to return `true` — a forged proof of transaction inclusion.

---

### Finding Description

`verify_transaction_inclusion` is annotated with `#[deprecated(since = "0.5.0", note = "Use verify_transaction_inclusion_v2 instead.")]` and carries an explicit `# Warning` in its doc comment:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

The function is gated only by `#[pause]`, which is a pausability guard, not an access-control restriction. No role check prevents an unprivileged NEAR account from calling it directly. [2](#0-1) 

The core verification logic computes a Merkle root from the caller-supplied `tx_id` and `merkle_proof` and compares it to the stored block header's `merkle_root`: [3](#0-2) 

Because Bitcoin block headers store only the Merkle root and not the transaction count, the proof depth cannot be validated on-chain. An attacker can supply an internal Merkle tree node hash as `tx_id` along with a valid sibling path that reconstructs the same root, and the function returns `true`.

The safe replacement, `verify_transaction_inclusion_v2`, closes this by requiring a coinbase Merkle proof that anchors the tree depth: [4](#0-3) 

The CLAUDE.md developer note confirms the vulnerability class explicitly: [5](#0-4) 

---

### Impact Explanation

Any dApp or recipient contract that calls `verify_transaction_inclusion` and acts on a `true` result can be deceived into believing a transaction is included in a Bitcoin block when it is not. This breaks the core SPV proof guarantee the contract is designed to provide. Concrete downstream consequences include: fraudulent cross-chain asset releases, false settlement of payment proofs, and acceptance of fabricated on-chain events — all triggered by a single permissionless NEAR call with crafted inputs.

---

### Likelihood Explanation

The entry path requires no privileges, no staked role, and no leaked keys. Any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte transaction Merkle forgery technique is publicly documented (referenced in the contract's own deprecation notice at the BitMEX blog link). The deprecated function is live in production state; the contract's own documentation acknowledges the vulnerability class. Likelihood is high.

---

### Recommendation

Remove `verify_transaction_inclusion` as a callable public method. It should not remain a live NEAR entry point after `verify_transaction_inclusion_v2` was introduced as its replacement. If backward compatibility is required during a transition window, gate the deprecated function behind an explicit access-control role (e.g., `Role::DAO`) so it cannot be invoked by unprivileged callers, and emit a deprecation event. Ultimately, the function should be deleted from the public ABI entirely.

---

### Proof of Concept

1. Identify a block `B` already accepted on the contract's mainchain with known `merkle_root`.
2. Obtain the raw Merkle tree for block `B`. Select any internal node `N` at depth `d` whose subtree hashes to `merkle_root` via a valid sibling path `P` of length `d`.
3. Call `verify_transaction_inclusion` as an unprivileged NEAR account:
   ```
   tx_id            = N   (internal node hash, not a real txid)
   tx_block_blockhash = hash(B)
   tx_index         = index consistent with path P
   merkle_proof     = P
   confirmations    = 1
   ```
4. `compute_root_from_merkle_proof(N, index, P)` reconstructs `merkle_root` exactly.
5. The function returns `true` — falsely asserting that `N` is an included transaction.
6. Any recipient contract consuming this result treats the forged proof as valid SPV confirmation. [6](#0-5)

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
