### Title
Unprivileged Caller Can Trigger Unbounded Mainchain GC, Permanently Invalidating SPV Proofs — (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public, state-mutating NEAR contract method protected only by the `#[pause]` decorator. When the contract is **not paused** — its normal operating state — any unprivileged NEAR account can call it with an arbitrary `batch_size`. This allows an attacker to force maximum-batch garbage collection in a single transaction, permanently deleting mainchain block headers from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`, and advancing `mainchain_initial_blockhash`. Any downstream contract or caller that subsequently calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a pruned block will receive a hard panic ("block does not belong to the current main chain"), permanently invalidating those SPV proofs.

---

### Finding Description

The `#[pause(except(roles(Role::UnrestrictedRunGC)))]` attribute on `run_mainchain_gc` only restricts access when the contract is **paused**: [1](#0-0) 

When the contract is running normally (unpaused), the decorator is a no-op for caller identity — it does not check `env::predecessor_account_id()` against any role. Any NEAR account can call this method.

Compare this to `submit_blocks`, the only other state-mutating public method, which carries both `#[pause]` **and** `#[trusted_relayer]`: [2](#0-1) 

The `#[trusted_relayer]` guard enforces that the caller is a registered, staked relayer. No equivalent guard exists on `run_mainchain_gc`. The existence of the `UnrestrictedRunGC` role in the role enum confirms the designers intended GC to be a privileged operation: [3](#0-2) 

But the role is only enforced in the paused path, leaving the unpaused path open to all callers.

Inside the function, the attacker-controlled `batch_size` is used directly in the removal loop: [4](#0-3) 

With `batch_size = u64::MAX`, `selected_amount_to_remove` is capped at `total_amount_to_remove = amount_of_headers_we_store - gc_threshold`, meaning all over-threshold headers are deleted in one call. The function then permanently advances `mainchain_initial_blockhash` to the new oldest block.

The design intent (per `CLAUDE.md`) is that GC runs automatically inside `submit_blocks` with `batch_size = num_of_headers` — a small, bounded value proportional to the current submission batch: [5](#0-4) 

The public endpoint breaks this by allowing any caller to collapse the entire over-threshold window in one transaction.

---

### Impact Explanation

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` both look up the target block in `mainchain_header_to_height`: [6](#0-5) 

`remove_block_header` deletes entries from both `mainchain_header_to_height` and `headers_pool`: [7](#0-6) 

After the attacker's GC call, any block that was pruned will cause `verify_transaction_inclusion` to panic with "block does not belong to the current main chain." This is a permanent, irreversible state change — deleted `LookupMap` entries cannot be restored without a migration. Any downstream NEAR contract that relies on SPV proof verification for pruned blocks loses that capability permanently.

**Impact: High** — canonical chain state is permanently corrupted for all blocks beyond the new `mainchain_initial_blockhash`; SPV proof verification is broken for those blocks for all consumers of the contract.

---

### Likelihood Explanation

The entry point is a standard NEAR contract call requiring no deposit, no role, and no special knowledge beyond the contract's account ID. The attacker needs only to call `run_mainchain_gc` with `batch_size = u64::MAX` while the contract is unpaused. The contract is designed to run unpaused in production. No privileged keys, social engineering, or external dependencies are required.

**Likelihood: High** — zero-privilege, single-transaction exploit against a publicly callable method on a live contract.

---

### Recommendation

Add a caller authorization check to `run_mainchain_gc` equivalent to the `#[trusted_relayer]` guard on `submit_blocks`, or introduce a dedicated role check (e.g., require `Role::UnrestrictedRunGC` or `Role::DAO` even when unpaused). The simplest fix is to replace the current decorator:

```rust
// Current — only restricts when paused:
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }

// Fixed — restrict to authorized callers at all times:
#[pause(except(roles(Role::UnrestrictedRunGC)))]
#[access_control_any(roles(Role::DAO, Role::UnrestrictedRunGC, Role::RelayerManager))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }
```

Alternatively, make `run_mainchain_gc` a private method and expose it only through `submit_blocks`, removing the public endpoint entirely.

---

### Proof of Concept

Precondition: contract is deployed and unpaused, mainchain has grown beyond `gc_threshold` (e.g., `gc_threshold = 52704`, mainchain has 60,000 headers).

```rust
// Attacker account — no roles, no stake, no deposit required
attacker_account
    .call(contract.id(), "run_mainchain_gc")
    .args_json(json!({ "batch_size": u64::MAX }))
    .transact()
    .await?;
// Result: all 7,296 over-threshold headers deleted in one call.
// mainchain_initial_blockhash advanced by 7,296 blocks.

// Any subsequent SPV proof for a pruned block now panics:
contract
    .view("verify_transaction_inclusion_v2")
    .args_borsh(proof_args_for_pruned_block)
    .await?;
// Panics: "block does not belong to the current main chain"
```

The test in `contract/tests/test_basics.rs` already demonstrates that `run_mainchain_gc` succeeds when called by the `user_account` (which holds only `UnrestrictedSubmitBlocks`, not `UnrestrictedRunGC`), confirming the absence of any caller check in the unpaused path: [8](#0-7)
