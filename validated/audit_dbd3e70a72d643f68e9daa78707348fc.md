### Title
Raw `approve()` Call in `DirectDepositV1.creditDeposit()` Permanently Locks Non-Standard ERC20 Tokens — (File: `core/contracts/DirectDepositV1.sol`)

### Summary
`DirectDepositV1.creditDeposit()` calls `token.approve()` via the `IIERC20Base` interface, which expects a `bool` return value. Non-standard ERC20 tokens (e.g., USDT) that do not return a value from `approve()` cause the ABI decoder to revert, permanently locking any such tokens deposited into the `DirectDepositV1` contract.

### Finding Description
`DirectDepositV1` is a per-subaccount deposit address contract. When a user sends tokens to their DDA and `creditDeposit()` is called, the function iterates over all registered spot product token addresses and calls `token.approve(address(endpoint), balance)` before forwarding the deposit to the endpoint.

The `IIERC20Base` interface in `DirectDepositV1.sol` declares `approve` as returning `bool`: [1](#0-0) 

When Solidity's ABI decoder attempts to decode the return data of a non-standard token's `approve()` (which returns no data), it reverts. This is the same root cause as the referenced report.

The `creditDeposit()` function has no access modifier — it is callable by any address: [2](#0-1) 

The raw `approve()` call at line 92 is the vulnerable site: [3](#0-2) 

By contrast, `DirectDepositV1` already implements a `safeTransfer` helper that uses low-level `.call()` to handle tokens that do not return a value: [4](#0-3) 

No analogous `safeApprove` helper exists. The project-wide `ERC20Helper` library also only provides `safeTransfer` and `safeTransferFrom`, with no `safeApprove`: [5](#0-4) 

A secondary instance of the same bug exists in `ContractOwner.wrapVaultAsset()`, which calls raw `assetToken.approve()` (via `IERC20Base`) twice without safe handling: [6](#0-5) 

`IERC20Base.approve` also declares a `bool` return: [7](#0-6) 

### Impact Explanation
Any non-standard ERC20 token registered as a spot product (e.g., USDT, BNB) that does not return a value from `approve()` will cause `creditDeposit()` to revert unconditionally. Tokens already transferred into the `DirectDepositV1` contract cannot be credited to the subaccount and are effectively locked. Recovery requires the multisig owner to call `ContractOwner.withdrawFromDirectDepositV1()`, which is an `onlyOwner` function: [8](#0-7) 

Until admin intervention, the deposited tokens are inaccessible to the user. The asset delta is: user's tokens transferred into DDA, protocol balance unchanged, user's on-chain subaccount balance uncredited.

### Likelihood Explanation
USDT is one of the most widely used stablecoins and is a canonical non-standard ERC20. If the Nado protocol lists USDT or any other non-returning-`approve` token as a spot product, every DDA deposit for that token will fail silently from the user's perspective (tokens leave their wallet, nothing is credited). The entry path requires no special privileges — any user can send tokens to their DDA and trigger `creditDeposit()`.

### Recommendation
Add a `safeApprove` helper to `DirectDepositV1` (mirroring the existing `safeTransfer` pattern) that uses a low-level `.call()` and accepts both empty and `bool` return data. Replace the raw `token.approve(address(endpoint), balance)` call at line 92 with this helper. Apply the same fix to `ContractOwner.wrapVaultAsset()` lines 530–531. Alternatively, extend `ERC20Helper` with a `safeApprove` function and use it consistently across all contracts.

### Proof of Concept
1. Register a USDT-equivalent token (no `bool` return from `approve`) as a spot product in `SpotEngine`.
2. Deploy or reuse a `DirectDepositV1` for any subaccount via `ContractOwner.creditDepositV1()`.
3. Transfer USDT to the `DirectDepositV1` address.
4. Call `DirectDepositV1.creditDeposit()` (permissionless).
5. The call reaches line 92: `token.approve(address(endpoint), balance)`. The token returns no data; Solidity's ABI decoder attempts to decode empty bytes as `bool` and reverts with a decoding error.
6. The USDT remains in the `DirectDepositV1` contract. The subaccount balance is not updated. The user has lost access to their funds until the multisig manually calls `withdrawFromDirectDepositV1`. [9](#0-8) [10](#0-9)

        uint256 amount

        (bool success, bytes memory data) = address(self).call(

        require(

            ERR_TRANSFER_FAILED

        address to,

                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }

    function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
        external
        onlyOwner
    {
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        if (token == address(0)) {
            uint256 preBalance = address(this).balance;
            DirectDepositV1(directDepositV1).withdrawNative();
            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");
            (bool success, ) = msg.sender.call{value: postBalance - preBalance}(
                ""
            );
            require(success, "xfer");
        } else {
            uint256 preBalance = IERC20Base(token).balanceOf(address(this));
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));
            require(postBalance > preBalance, "empty");
            IERC20Base(token).safeTransfer(
                msg.sender,
                postBalance - preBalance
            );
