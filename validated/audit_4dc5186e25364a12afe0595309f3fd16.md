### Title
Deprecated `verify_transaction_inclusion` Remains Callable and Vulnerable to 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` (v1) function is still reachable by any unprivileged NEAR caller and is structurally vulnerable to the 64-byte Merkle proof forgery attack. An attacker can supply a crafted `tx_id` that is the first 32 bytes of a real 64-byte Bitcoin transaction and construct a valid-looking Merkle proof that the contract accepts as genuine, falsely confirming inclusion of a non-existent transaction.

---

### Finding Description

`verify_transaction_inclusion` at `contract/src/lib.rs` line 288 is annotated `#[deprecated(since = "0.5.0")]` and carries an explicit self-warning in its doc comment:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."

Despite this, the function carries only `#[pause]` access control — meaning it is callable by **any unprivileged NEAR account** whenever the contract is not paused. The `#[deprecated]` Rust attribute is a compiler hint only; it imposes no on-chain access restriction.

The root cause lives in `merkle-tools/src/lib.rs`, `compute_root_from_merkle_proof` (line 34–52). The function accepts any 32-byte `transaction_hash` as a starting leaf and iteratively hashes it with sibling nodes from `merkle_proof`. It performs **no validation** that the input hash is a genuine leaf (transaction) rather than an internal Merkle tree node:

```rust
// merkle-tools/src/lib.rs:34-51
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;
    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
    current_hash
}
```

The 64-byte forgery attack exploits this directly:

1. A 64-byte Bitcoin transaction `T` has hash `H = double_sha256(T)`.
2. Split `T` into two 32-byte halves: `L = T[0:32]`, `R = T[32:64]`.
3. Because `compute_hash(L, R) = double_sha256(L || R) = double_sha256(T) = H`, the value `H` is simultaneously the hash of transaction `T` **and** the hash of an internal Merkle node whose children are `L` and `R`.
4. An attacker calls `verify_transaction_inclusion` with `tx_id = L`, `merkle_proof = [R, <proof from H to root>]`.
5. `compute_root_from_merkle_proof` computes `double_sha256(L || R) = H`, then walks the remaining proof elements to the real `merkle_root`.
6. The comparison `== header.block_header.merkle_root` succeeds, and the function returns `true`.

The only guard in v1 that could block this is:

```rust
// contract/src/lib.rs:315
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

This check is satisfied because the forged proof is non-empty (`[R, ...]`).

`verify_transaction_inclusion_v2` was introduced specifically to close this gap by requiring a coinbase proof of equal depth, but v1 remains a live, unguarded entry point.

---

### Impact Explanation

Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion` (v1) to gate a security-sensitive action — cross-chain asset release, bridge unlock, settlement — can be deceived into accepting a forged SPV proof. The attacker proves inclusion of `L` (a 32-byte value that is not a real transaction) in a real, finalized Bitcoin block. The canonical chain state in the contract is not corrupted, but the **proof result** returned to the caller is false, which is the primary security guarantee the contract is supposed to provide.

---

### Likelihood Explanation

64-byte Bitcoin transactions are syntactically valid. An attacker can craft and broadcast one, wait for it to be mined, and then use the resulting block as the forgery vehicle. No hash collision or cryptographic break is required. The attack requires only: (a) a mined block containing a 64-byte transaction, and (b) the ability to call a public NEAR contract method — both are within reach of an unprivileged actor.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the contract entirely, or gate it behind a role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts. All callers must migrate to `verify_transaction_inclusion_v2`. Keeping a deprecated but exploitable public method on a live contract is equivalent to the untested/unguarded code-path class identified in the reference report.

---

### Proof of Concept

```
1. Broadcast a 64-byte Bitcoin transaction T to mainnet/testnet.
   Let L = T[0:32], R = T[32:64], H = double_sha256(T).

2. Wait for T to be mined in block B.
   Obtain the Merkle proof from H to B's merkle_root: [p1, p2, ...].

3. Submit block B's headers to the NEAR contract via submit_blocks().

4. Call verify_transaction_inclusion() with:
     tx_id              = L          (not a real transaction)
     tx_block_blockhash = hash(B)
     tx_index           = <even index of T in the block>
     merkle_proof       = [R, p1, p2, ...]
     confirmations      = 1

5. Contract returns true.
   L has never existed as a Bitcoin transaction.

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
