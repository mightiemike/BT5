### Title
`_isAboveInitial` Uses `>= 0` While `isAboveInitial` Uses `> 0`, Allowing `transferQuote` to Drain Sender Health to the Under-Initial Boundary — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse._isAboveInitial` returns `true` when health == 0 (operator `>= 0`), while `ClearinghouseLiq.isAboveInitial` returns `false` for the same condition (operator `> 0`). This is a direct analog to the `_hasFundableKeys` bug: just as that function returned `true` when `funded == limit` (the operator was maxed out), `_isAboveInitial` returns `true` when health == 0 (the account is at the exact under-initial boundary). The result is that `transferQuote` permits a sender to drain their account to exactly zero initial health — a state the protocol's own liquidation-side function does not consider "above initial."

---

### Finding Description

`Clearinghouse.sol` defines:

```solidity
function _isAboveInitial(bytes32 subaccount) internal returns (bool) {
    // Weighted initial health with limit orders < 0
    return getHealth(subaccount, IProductEngine.HealthType.INITIAL) >= 0;
}
```

`ClearinghouseLiq.sol` defines:

```solidity
function isAboveInitial(bytes32 subaccount) public returns (bool) {
    // Weighted initial health with limit orders < 0
    return
        getHealthFromClearinghouse(
            subaccount,
            IProductEngine.HealthType.INITIAL
        ) > 0;
}
```

Both carry the identical comment. The only difference is `>= 0` vs `> 0`. The `_isAboveInitial` variant is the gate used in `transferQuote`:

```solidity
require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH);
```

Because `>= 0` is used, a sender whose health lands at exactly 0 after the transfer passes the check. The `isAboveInitial` variant used in liquidation (`require(!isAboveInitial(txn.liquidatee), ERR_LIQUIDATED_TOO_MUCH)`) treats health == 0 as *not* above initial — confirming the protocol's own liquidation logic does not regard zero health as a safe state.

The `stopped` / `funded` analogy maps directly: `stopped` was a counter-only field that, when subtracted from `funded`, made a maxed-out operator appear to have remaining capacity. Here, the `>= 0` boundary makes a zero-health account appear to be above initial, when the rest of the protocol (liquidation side) treats zero health as the exact threshold.

---

### Impact Explanation

A user can call `transferQuote` in a loop, moving quote collateral to another subaccount they control, until their initial health reaches exactly 0. At that point:

- `_isAboveInitial` returns `true` → the final transfer is accepted.
- `isUnderInitial` returns `false` (health is not `< 0`) → the account is not yet liquidatable.
- `isAboveInitial` (ClearinghouseLiq) returns `false` → the account is not considered healthy by the liquidation module.

The sender has extracted maximum collateral while sitting at the exact under-initial boundary. Any subsequent oracle price tick, funding rate accrual, or fee charge pushes them below zero, making them immediately liquidatable with no buffer. The collateral has already been moved to a sibling subaccount. This is a concrete accounting corruption: the health-gate function certifies safety at a state the protocol's own liquidation module does not consider safe.

---

### Likelihood Explanation

Any user with a non-zero subaccount balance can trigger this via the sequencer-processed `TransferQuote` transaction type. No special role, governance access, or external dependency is required. The entry path is `EndpointTx.processTransactionImpl` → `clearinghouse.transferQuote` → `require(_isAboveInitial(...))`. The condition (health landing at exactly 0) is achievable by choosing the transfer amount precisely, which is straightforward off-chain.

---

### Recommendation

Align the two functions. Since `ClearinghouseLiq.isAboveInitial` (used in liquidation) treats health == 0 as *not* above initial, `_isAboveInitial` in `Clearinghouse.sol` should use the same strict inequality:

```solidity
function _isAboveInitial(bytes32 subaccount) internal returns (bool) {
    return getHealth(subaccount, IProductEngine.HealthType.INITIAL) > 0;
}
```

This ensures `transferQuote` cannot leave a sender at the exact under-initial boundary.

---

### Proof of Concept

1. Attacker controls subaccount `A` (health = H > 0) and subaccount `B`.
2. Attacker submits a signed `TransferQuote` transaction moving quote from `A` to `B` such that `A`'s resulting initial health == 0.
3. `Clearinghouse.transferQuote` calls `require(_isAboveInitial(A))`.
4. `_isAboveInitial` computes `getHealth(A, INITIAL) >= 0` → `0 >= 0` → `true`. Transfer succeeds.
5. `A`'s health is now exactly 0. `isUnderInitial(A)` returns `false` (not `< 0`), so no liquidation is triggered.
6. `isAboveInitial(A)` (ClearinghouseLiq) returns `false` (not `> 0`), confirming the liquidation module does not consider `A` healthy.
7. The next oracle price update or funding tick that reduces `A`'s health by any positive amount makes `A` immediately liquidatable, with all collateral already transferred to `B`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
