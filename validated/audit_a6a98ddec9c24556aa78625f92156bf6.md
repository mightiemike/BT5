### Title
Fee-on-Transfer Token Deposit Over-Credits Subaccount Balance — (`core/contracts/Endpoint.sol` / `core/contracts/Clearinghouse.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` transfers tokens from the depositor using the caller-supplied `amount`, then enqueues a `DepositCollateral` slow-mode transaction encoding that same nominal `amount`. When the sequencer later processes the transaction, `Clearinghouse.depositCollateral` credits the full nominal `amount` to the subaccount balance without verifying how many tokens were actually received. For any ERC20 token that deducts a transfer fee (fee-on-transfer), the Clearinghouse receives fewer tokens than it credits, inflating the depositor's on-chain collateral.

---

### Finding Description

**Step 1 — Deposit entry point (`Endpoint.sol`)**

`depositCollateralWithReferral` calls `handleDepositTransfer` with the raw caller-supplied `amount`, then immediately encodes that same `amount` into the pending slow-mode queue: [1](#0-0) 

No balance snapshot is taken before or after the transfer. The value stored in the slow-mode queue is always the nominal `amount`, regardless of what the contract actually received.

**Step 2 — Clearinghouse crediting (`Clearinghouse.sol`)**

When the sequencer executes the queued transaction, `depositCollateral` reads `txn.amount` (the nominal value) and credits it directly to the subaccount: [2](#0-1) 

There is no comparison against an actual on-chain balance delta. The credited `amountRealized` is derived purely from `txn.amount`, which was set at deposit time from the caller's input.

**Root cause:** Neither `Endpoint.depositCollateralWithReferral` nor `Clearinghouse.depositCollateral` performs a `balanceBefore` / `balanceAfter` check to determine the true received amount. This is structurally identical to the reference bug, where `FunnelVault.swapTokens` trusted the router's returned `amountOut` instead of measuring the actual balance delta.

---

### Impact Explanation

A depositor using a fee-on-transfer token receives a subaccount credit of `amount` while the Clearinghouse only holds `amount − fee`. The inflated balance:

- Increases the depositor's health score, allowing larger positions or borrows than the protocol's actual collateral supports.
- Causes the protocol to become undercollateralized by the cumulative fee amount across all such deposits.
- On withdrawal, the Clearinghouse attempts to transfer the full credited amount, which it does not hold, leading to failed withdrawals for other users or depletion of insurance funds.

**Impact: Medium** — accounting corruption and solvency risk proportional to the fee rate and deposit volume.

---

### Likelihood Explanation

**Likelihood: High** — The deposit path is the primary, permissionless entry point for all users. Any token registered as a supported product that carries a transfer fee (including tokens whose fee switches are activated post-listing) immediately triggers this condition. No special role or privileged access is required; any ordinary depositor calling `depositCollateral` or `depositCollateralWithReferral` with a fee-bearing token triggers the discrepancy. [3](#0-2) 

---

### Recommendation

Replace the nominal-amount accounting with a balance-delta pattern in `Endpoint.depositCollateralWithReferral`:

```solidity
uint256 balanceBefore = IERC20Base(spotEngine.getToken(productId)).balanceOf(address(clearinghouse));
handleDepositTransfer(IERC20Base(spotEngine.getToken(productId)), msg.sender, uint256(amount));
uint256 actualReceived = IERC20Base(spotEngine.getToken(productId)).balanceOf(address(clearinghouse)) - balanceBefore;
// encode actualReceived (not amount) into the slow-mode DepositCollateral transaction
```

Alternatively, whitelist only non-fee-on-transfer tokens and enforce this invariant at product registration time.

---

### Proof of Concept

1. A fee-on-transfer token (e.g., 1% fee) is registered as a supported product in SpotEngine.
2. Attacker calls `Endpoint.depositCollateral(subaccountName, productId, 1000e18)`.
3. `handleDepositTransfer` pulls `1000e18` tokens from the attacker; the token's fee mechanism deducts 1%, so Clearinghouse receives `990e18`.
4. The slow-mode queue records `amount = 1000e18`.
5. Sequencer processes the transaction; `Clearinghouse.depositCollateral` credits `amountRealized` derived from `1000e18` to the attacker's subaccount.
6. Attacker's subaccount now shows `1000e18` collateral while the protocol only holds `990e18`.
7. Attacker opens a position or borrows against the inflated `1000e18` balance, extracting value exceeding actual collateral.
8. Repeated across many deposits, the protocol accumulates a growing shortfall between credited balances and held tokens. [4](#0-3) [5](#0-4)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });
        slowModeConfig = _slowModeConfig;
    }

**File:** core/contracts/Clearinghouse.sol (L193-209)
```text
    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
```
