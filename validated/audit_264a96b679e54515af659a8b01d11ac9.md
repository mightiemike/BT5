Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. A pool admin who allowlists the router to support standard periphery swaps inadvertently opens the pool to every user, completely defeating the allowlist. The `beforeSwap` hook interface exposes no separate field for the original transaction initiator, making this bypass structurally unavoidable.

## Finding Description

**Complete call chain:**

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```
MetricOmmPool.swap (L230-231):
  _beforeSwap(msg.sender, recipient, ...)
              ^^^^^^^^^^
              = router address when called via MetricOmmSimpleRouter
```

`ExtensionCalling._beforeSwap` (L160-176) encodes and forwards this `sender` value verbatim to every configured extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))`.

`SwapAllowlistExtension.beforeSwap` (L37) then checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
where `msg.sender` = pool and `sender` = router address. The end user's identity is never consulted.

`MetricOmmSimpleRouter.exactInputSingle` (L71-80) calls `pool.swap(params.recipient, ...)` with no mechanism to inject the original `msg.sender` (the user) into the pool's `sender` argument. The `IMetricOmmExtensions.beforeSwap` interface (L50-60) only exposes `sender` and `recipient` — there is no `swapper` or originator field.

**Irreconcilable dilemma for pool admins:**

| Admin action | Result |
|---|---|
| Allowlist individual users only | Those users cannot swap through the router (router not allowlisted → revert) |
| Allowlist the router address | **Any** user bypasses the allowlist by routing through the router |

**Contrast with `DepositAllowlistExtension`:** `addLiquidity` accepts an explicit `owner` parameter distinct from `msg.sender`, so `_beforeAddLiquidity(msg.sender, owner, ...)` carries the economic actor's identity separately. `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (L38), not `sender`. The swap path has no equivalent separate identity field.

**Existing guards are insufficient:** The `BaseMetricExtension.onlyPool` modifier only verifies the caller is a registered pool — it does not recover the original user identity. There is no on-chain mechanism within the current interface for the extension to distinguish a direct swap from a router-mediated swap or to recover the true initiator.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a specific set of addresses (e.g., KYC'd counterparties, whitelisted market makers, or protocol-controlled addresses) has that restriction completely bypassed. Any unprivileged user routes through `MetricOmmSimpleRouter`, which the admin allowlisted to support normal periphery usage. The user receives swap output, the pool's token balances change, and the curation policy is violated. This is a direct violation of the pool's intended access-control invariant with fund-impacting consequences: unauthorized parties trade against the pool's liquidity. Severity: **High**.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical periphery entry point for swaps. Any pool admin who deploys a swap-allowlisted pool and also wants to support normal user tooling (wallets, aggregators, front-ends using the router) will allowlist the router. The bypass requires no special knowledge or privileged access — any user calls `exactInputSingle` on the router with the target pool. The admin's mistake is structurally forced: there is no mechanism to allowlist specific users *and* allow them to use the router simultaneously. Likelihood: **High**.

## Recommendation

1. **Preferred fix:** Introduce a `swapper` parameter distinct from `sender` in the `IMetricOmmExtensions.beforeSwap` interface, where `swapper` is the original transaction initiator. `MetricOmmSimpleRouter` would pass `msg.sender` (the user) as `swapper` when calling the pool, and the pool would forward it to extensions. This mirrors how `DepositAllowlistExtension` uses `owner` rather than `sender`.

2. **Short-term:** Change `SwapAllowlistExtension.beforeSwap` to check `recipient` instead of `sender` if the intent is to gate who receives the output — though this is semantically weaker and can be gamed by setting a different recipient.

3. **Documentation guard (insufficient alone):** Document that `SwapAllowlistExtension` gates the direct pool caller, not the end user, and that allowlisting the router opens the pool to all users.
