### Title
Unprotected `initialize()` in `WithdrawPool` Allows Frontrunning to Seize Ownership and Drain Funds - (File: core/contracts/WithdrawPool.sol)

### Summary
`WithdrawPool.initialize()` is missing the `initializer` modifier. Any unprivileged caller can invoke it before the protocol does, set themselves as owner and supply a malicious `clearinghouse`/`verifier`, then drain all tokens via `removeLiquidity()`.

### Finding Description

`WithdrawPool` exposes a public `initialize()` entry point with no access control and no `initializer` guard: [1](#0-0) 

```solidity
contract WithdrawPool is BaseWithdrawPool {
    function initialize(address _clearinghouse, address _verifier) external {
        _initialize(_clearinghouse, _verifier);
    }
}
```

The `initializer` modifier lives only on the internal `_initialize` in `BaseWithdrawPool`: [2](#0-1) 

```solidity
function _initialize(address _clearinghouse, address _verifier)
    internal
    initializer          // ← guard is here, not on the public wrapper
{
    __Ownable_init();
    clearinghouse = _clearinghouse;
    verifier = _verifier;
}
```

Because the `initializer` modifier is absent from the public wrapper, any EOA or contract can race the protocol's deployment transaction and call `WithdrawPool.initialize()` first, supplying attacker-controlled `_clearinghouse` and `_verifier` addresses. OpenZeppelin's `initializer` modifier then permanently locks the slot; the protocol's own `initialize()` call reverts with "Initializable: contract is already initialized", leaving the proxy permanently owned by the attacker.

The same pattern exists in `SpotEngine.initialize()` and `PerpEngine.initialize()`, which also delegate to `BaseEngine._initialize()` (which carries the `initializer` guard) without placing any guard on their own public wrappers: [3](#0-2) [4](#0-3) 

### Impact Explanation

Once the attacker owns `WithdrawPool`:

1. **Direct fund drain** — `removeLiquidity()` is `onlyOwner` and transfers any token balance to an arbitrary address: [5](#0-4) 

2. **Bypass withdrawal accounting** — `submitWithdrawal()` gates on `msg.sender == clearinghouse`; with a malicious `clearinghouse` the attacker can mark arbitrary `idx` values as processed and drain tokens: [6](#0-5) 

3. **Bypass fast-withdrawal signature check** — `submitFastWithdrawal()` calls `Verifier(verifier).requireValidTxSignatures()`; a malicious `verifier` can approve any withdrawal: [7](#0-6) 

All user collateral held in the pool is at risk.

### Likelihood Explanation

The attack requires only that the proxy deployment and the `initialize()` call are submitted as separate transactions (the common pattern when using deployment scripts or multi-step factory flows). A mempool observer can frontrun the initialization call with a higher gas price. No special privilege, leaked key, or governance capture is needed — any unprivileged address can execute this.

### Recommendation

Add the `initializer` modifier to the public wrapper in `WithdrawPool`, `SpotEngine`, and `PerpEngine`:

```solidity
// WithdrawPool.sol
function initialize(address _clearinghouse, address _verifier)
    external
    initializer          // ← add this
{
    _initialize(_clearinghouse, _verifier);
}
```

Apply the same fix to `SpotEngine.initialize()` and `PerpEngine.initialize()`. Alternatively, collapse the two-level call into a single function that carries the modifier directly, matching the pattern used by every other contract in the codebase (`Endpoint`, `Clearinghouse`, `OffchainExchange`, etc.).

### Proof of Concept

1. Protocol deploys `WithdrawPool` proxy (transaction T1 in mempool).
2. Attacker observes T1, submits `WithdrawPool(proxy).initialize(attackerClearinghouse, attackerVerifier)` with higher gas (transaction T2).
3. T2 mines first: `_initialize` runs, `__Ownable_init()` sets `owner = attacker`, `clearinghouse = attackerClearinghouse`, `verifier = attackerVerifier`. `_initialized` slot is set to `1`.
4. T1 mines: `_initialize` reverts ("Initializable: contract is already initialized"). Protocol cannot recover ownership.
5. Attacker calls `removeLiquidity(productId, totalBalance, attacker)` → all tokens transferred to attacker.
