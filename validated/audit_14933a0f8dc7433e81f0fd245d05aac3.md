### Title
Fork Retarget Difficulty Computed from Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

`get_header_by_height` is implemented to read exclusively from `mainchain_height_to_header`. Fork blocks are stored only in `headers_pool` via `store_fork_header`, never in `mainchain_height_to_header`. When `get_next_work_required` is called for a fork block at a retarget boundary, it calls `get_header_by_height(first_block_height)` to obtain the timestamp of the interval's opening block. If the fork diverged before that opening height, the mainchain block at that height is returned instead of the fork's own ancestor, producing an incorrect `first_block_time` and therefore an incorrect expected `bits` value. The contract then enforces that the submitted block's `bits` field matches this incorrect value, meaning a fork block with bits that would be rejected by the real network can be accepted, and a fork block with bits that the real network would accept can be rejected.

---

### Finding Description

**Storage asymmetry between fork and mainchain blocks**

`store_block_header` writes to both `mainchain_height_to_header` and `headers_pool`: [1](#0-0) 

`store_fork_header` writes only to `headers_pool`: [2](#0-1) 

`get_header_by_height` reads only from `mainchain_height_to_header`: [3](#0-2) 

**Retarget calculation uses height-based lookup**

In `bitcoin.rs`, at a retarget boundary, `get_next_work_required` computes `first_block_height` and calls `get_header_by_height`: [4](#0-3) 

The same pattern exists in `litecoin.rs`: [5](#0-4) 

And in `dogecoin.rs`, where the developers themselves flagged the issue with a TODO: [6](#0-5) 

**Concrete broken invariant**

Consider Bitcoin with `difficulty_adjustment_interval = 2016`. A fork diverges at height 2000 (before the first retarget at 2016). The attacker submits fork blocks f2001…f4032. When submitting f4032 (the second retarget boundary):

- `prev_block_header` = f4031 (height 4031), stored in `headers_pool` only
- `first_block_height = 4031 − 2015 = 2016`
- `get_header_by_height(2016)` → returns **mainchain** block m2016 (from `mainchain_height_to_header`)
- The fork's own block at height 2016 is f2016, stored only in `headers_pool`, and is **never consulted**

`calculate_next_work_required` then uses `m2016.time` as `first_block_time`: [7](#0-6) 

The attacker controls f2016's timestamp (subject only to MTP and the 2-hour future cap). If the attacker sets f2016.time > m2016.time, then using m2016.time as `first_block_time` produces a larger `actual_time_taken`, which yields a lower (easier) `new_target` and lower required `bits`. The contract enforces that the submitted block's `bits` equals this incorrectly computed value: [8](#0-7) 

A fork block at the retarget boundary is therefore accepted with bits that do not match what the real Bitcoin network would require.

---

### Impact Explanation

The contract accepts fork blocks at retarget boundaries with a `bits` value derived from the mainchain's retarget-interval opening block rather than the fork's own ancestor at that height. This breaks the invariant that difficulty for a fork block must be computed from the fork's own chain history.

Concretely: the attacker can craft a fork whose second-interval retarget computes an easier target than the real network enforces. If the fork accumulates more chainwork than the mainchain (easier with reduced difficulty), `reorg_chain` promotes it to canonical: [9](#0-8) 

Once canonical, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will confirm transactions in fork blocks as if they were on the real Bitcoin chain, enabling forged transaction inclusion proofs.

---

### Likelihood Explanation

The entry point is `submit_blocks`, gated by `#[trusted_relayer]`. A trusted relayer is a valid attacker role per the scope rules. The attacker must mine 2016+ fork blocks, but the difficulty reduction (up to 4×) directly reduces the work required. The Dogecoin TODO comment at line 291 confirms the developers were already aware this path is unverified. The bug is deterministic and locally reproducible without any external dependency.

---

### Recommendation

Replace the height-based lookup in `get_next_work_required` with an ancestor traversal that follows `get_prev_header` links from `prev_block_header` backward by `difficulty_adjustment_interval` steps. This traversal uses `headers_pool` and correctly follows the fork's own chain regardless of what is in `mainchain_height_to_header`. The `get_header_by_height` shortcut is only valid when the block being validated is on the mainchain; it must not be used during fork validation.

---

### Proof of Concept

```
State setup (Bitcoin feature, skip_pow_verification = false):
  Genesis at height 0 (aligned to retarget interval 2016).
  Mainchain: heights 0..4032 submitted normally.
  mainchain_height_to_header[2016] = m2016 (timestamp T_m).

Fork construction:
  Fork diverges at height 2000.
  Submit f2001..f2015 as fork blocks (stored in headers_pool only).
  Submit f2016 with timestamp T_f = T_m + 8_weeks (maximally late, within 2-hour future cap at submission time).
    → difficulty for f2016 uses get_header_by_height(0) = genesis (same for both chains) → correct bits accepted.
  Submit f2017..f4031 as fork blocks.
  Submit f4032 at retarget boundary:
    → get_next_work_required calls get_header_by_height(2016) → returns m2016 (T_m), NOT f2016 (T_f).
    → actual_time_taken = f4031.time − T_m  (larger than f4031.time − T_f)
    → expected_bits computed with inflated actual_time_taken → lower difficulty than real network requires.
    → contract accepts f4032 with these easier bits.

Assert: the bits value accepted for f4032 differs from what Bitcoin consensus would compute
        using f2016's timestamp, confirming the fork's required difficulty is easier than enforced.
```

fn calculate_next_work_required(
    config: &NetworkConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let prev_block_time: i64 = prev_block_header.block_header.time.into();

    let mut actual_time_taken: i64 = prev_block_time - first_block_time;
    if actual_time_taken < config.pow_target_timespan / 4 {
        actual_time_taken = config.pow_target_timespan / 4;
    }
    if actual_time_taken > config.pow_target_timespan * 4 {
        actual_time_taken = config.pow_target_timespan * 4;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(actual_time_taken).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target = new_target
        / U256::from(<i64 as TryInto<u64>>::try_into(config.pow_target_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
