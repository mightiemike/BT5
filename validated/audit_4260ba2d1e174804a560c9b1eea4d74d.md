### Title
L1 Handler Consensus Conversion Hardcodes `paid_fee_on_l1 = Fee(1)`, Bypassing the Blockifier's Zero-Fee Guard - (File: crates/apollo_transaction_converter/src/transaction_converter.rs)

### Summary

When a `ConsensusTransaction::L1Handler` is converted back to an executable `L1HandlerTransaction`, the `paid_fee_on_l1` field is unconditionally set to the hardcoded constant `Fee(1)` instead of the actual fee that was paid on L1. Because the blockifier's only post-execution fee guard for L1 handler transactions is `paid_fee == Fee(0) → reject`, this hardcoded value permanently disables that guard for every L1 handler transaction that travels through the consensus path.

### Finding Description

`transaction::L1HandlerTransaction` (the wire/consensus type) carries no `paid_fee_on_l1` field. When the consensus orchestrator receives a `ConsensusTransaction::L1Handler` and needs to produce an executable transaction, it calls `convert_consensus_l1_handler_to_internal_l1_handler`:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

The resulting `executable_transaction::L1HandlerTransaction` always carries `paid_fee_on_l1 = Fee(1)`. The blockifier's post-execution check for L1 handler transactions is:

```rust
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
``` [2](#0-1) 

Because `Fee(1) != Fee(0)`, the guard is permanently satisfied regardless of what fee was actually paid on L1.

The conversion boundary is the exact analog of the `approveAndCall` bug: instead of forwarding the caller-supplied specific value (`amount`), the code substitutes a hardcoded sentinel (`Fee(1)` here, `type(uint256).max` there). The direction differs (too-low vs. too-high) but the structural invariant broken is identical — the conversion does not bind the correct domain value.

### Impact Explanation

Any L1 handler transaction that arrives through the consensus path (i.e., proposed by a sequencer peer and validated by this node) will have its `paid_fee_on_l1` silently replaced with `Fee(1)`. If the original L1 message was sent with zero fee on L1 (which the Starknet L1 contract permits — the fee is a payment, not a validity gate), the blockifier check that is supposed to revert such transactions is bypassed. The transaction executes, its state changes are committed, and the receipt reflects a successful execution that should have been a fee-error revert.

This fits: **High — Transaction conversion logic binds the wrong executable payload** (wrong `paid_fee_on_l1` value bound into the executable object) and potentially **Critical — Wrong receipt/revert result from blockifier execution logic for accepted input** (a transaction that should produce `InsufficientFee` instead produces a successful receipt with committed state).

### Likelihood Explanation

- The path is exercised for every L1 handler transaction processed through consensus (both proposal and validation sides).
- Sending an L1→L2 message with `msg.value = 0` on Ethereum is possible; the Starknet core contract does not enforce a minimum fee at the L1 level.
- The TODO comment confirms the placeholder is intentional but unresolved, meaning the broken invariant is present in every deployed version until the TODO is addressed.

### Recommendation

1. Add `paid_fee_on_l1` to the `ConsensusTransaction::L1Handler` wire representation (protobuf and Rust types) so the value survives the consensus round-trip.
2. In `convert_consensus_l1_handler_to_internal_l1_handler`, use the actual fee from the consensus message instead of the hardcoded `Fee(1)`.
3. Until (1) is done, at minimum document that the zero-fee guard is disabled for the consensus path and assess whether the L1 contract's own enforcement is sufficient.

### Proof of Concept

1. An attacker calls the Starknet L1 core contract's `sendMessageToL2` with `msg.value = 0` (zero ETH fee), targeting any L2 contract entry point.
2. The L1 events scraper picks up the event and creates an `executable_transaction::L1HandlerTransaction` with `paid_fee_on_l1 = Fee(0)`.
3. The proposing sequencer includes this transaction in a block proposal. The `InternalConsensusTransaction::L1Handler` wraps `transaction::L1HandlerTransaction` (no `paid_fee_on_l1` field).
4. The validating sequencer calls `convert_consensus_l1_handler_to_internal_l1_handler`, which produces an executable transaction with `paid_fee_on_l1 = Fee(1)`.
5. The blockifier executes the transaction. The post-execution check evaluates `Fee(1) == Fee(0)` → false → no error. The execution is committed.
6. The resulting receipt shows a successful execution; the state changes are applied. The correct outcome would have been a `TransactionFeeError::InsufficientFee` revert with no state change. [1](#0-0) [3](#0-2) [4](#0-3)
