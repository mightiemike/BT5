### Title
`LIMIT_HEAP` Flag Declared and Included in `MEMPOOL_MODE` but Never Enforced in Any Execution Path — (`File: src/chia_dialect.rs`)

---

### Summary

`ClvmFlags::LIMIT_HEAP` is defined with a documented purpose ("limits the number of atom-bytes allowed to be allocated, as well as the number of pairs"), included in the `MEMPOOL_MODE` constant used for strict mempool validation, but is **never checked anywhere in the execution path**. The flag is wired in but has no enforcement code, making the heap-limit protection completely inert. Any attacker-controlled CLVM program submitted to the mempool can allocate unbounded atom-bytes and pairs without triggering the intended limit.

---

### Finding Description

In `src/chia_dialect.rs`, `ClvmFlags::LIMIT_HEAP` is declared at bit `0x0004`:

```rust
/// When set, limits the number of atom-bytes allowed to be allocated,
/// as well as the number of pairs.
const LIMIT_HEAP = 0x0004;
``` [1](#0-0) 

It is included in the `MEMPOOL_MODE` composite constant, which is the entry point for strict mempool-mode validation:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [2](#0-1) 

A grep across all `src/*.rs` files for `LIMIT_HEAP` returns **exactly two matches, both in `src/chia_dialect.rs`**: the definition and the `MEMPOOL_MODE` constant. The flag is never read, tested, or acted upon in `run_program.rs`, `allocator.rs`, `more_ops.rs`, `bls_ops.rs`, or any other execution-path file.

By contrast, every other flag in `MEMPOOL_MODE` has concrete enforcement:

- `NO_UNKNOWN_OPS` → checked in `unknown_operator()` and `RuntimeDialect::op()` [3](#0-2) 
- `CANONICAL_INTS` → checked in `src/op_utils.rs` (2 matches)
- `DISABLE_OP` → checked at opcode 60 (`op_modpow`) dispatch and inside `op_div`, `op_divmod`, `op_mod` [4](#0-3) 
- `LIMIT_SOFTFORK` → checked in `apply_op()` before pushing a `SoftforkGuard` [5](#0-4) 

`LIMIT_HEAP` alone has **zero enforcement sites**.

---

### Impact Explanation

The flag's documented invariant — that mempool-mode execution caps total atom-byte and pair allocations — is silently broken. An attacker can craft a CLVM program that stays within the cost limit (which is enforced) but allocates a very large number of pairs or large atom buffers, consuming far more memory than the intended heap cap would allow. This bypasses the mempool's intended resource-isolation guarantee and can cause full-node memory pressure during mempool validation of attacker-submitted transactions. Because cost accounting charges per-byte via `MALLOC_COST_PER_BYTE`, there is a partial indirect bound, but the explicit pair-count and atom-byte-count limits the flag was designed to enforce are entirely absent.

**Impact: 2 / 10** — The cost model provides a partial backstop; the missing check does not enable arbitrary memory exhaustion, but it does silently remove a documented mempool-mode protection.

---

### Likelihood Explanation

`MEMPOOL_MODE` is the standard flag set passed by full nodes when validating incoming transactions. Any CLVM program submitted to the mempool exercises this path. No special attacker capability is required beyond the ability to submit a transaction. The broken invariant is triggered on every mempool evaluation.

**Likelihood: 5 / 10** — Routine mempool traffic exercises the code path; the missing check is always absent, not conditionally absent.

---

### Recommendation

Implement the heap-limit enforcement that `LIMIT_HEAP` documents. In the `Allocator` (or in `run_program`'s evaluation loop), track cumulative atom-bytes allocated and pair count. When `flags.contains(ClvmFlags::LIMIT_HEAP)` is true, reject programs that exceed the intended thresholds. Alternatively, if the flag is intentionally retired, remove it from `MEMPOOL_MODE` and from the `ClvmFlags` definition to eliminate the false documentation guarantee.

---

### Proof of Concept

```
// Attacker submits this CLVM program to the mempool.
// It allocates a large number of pairs via repeated cons,
// staying within cost limits but exceeding the intended heap cap.
// Because LIMIT_HEAP is never checked, no heap-limit error is raised.

(a (q 2 2 (c 2 (c 5 (c 11 ()))))
   (c (q 2 (i (= 11 ()) (q 1 . ()) (q 4 5 (a 2 (c 2 (c 5 (c (- 11 (q . 1)) ())))))) 1) 1))
// args: (nil 50000)
// Run with MEMPOOL_MODE flags.
// Expected (with LIMIT_HEAP enforced): EvalErr::HeapLimitExceeded (or equivalent)
// Actual: Ok(Reduction(...)) — heap limit silently unenforced
```

The root cause is in `src/chia_dialect.rs` lines 35–37 (flag definition) and lines 72–76 (`MEMPOOL_MODE` composition): `LIMIT_HEAP` is assembled into the flags word passed through `ChiaDialect::op()` and `run_program()`, but no code in the execution path ever reads it. [1](#0-0) [2](#0-1)
