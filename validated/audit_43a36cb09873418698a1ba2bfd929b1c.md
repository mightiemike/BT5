### Title
Users Can Permanently Lose Funds by Sending Unsupported Tokens to `DirectDepositV1` Due to Silent Skip in `creditDeposit()` - (File: core/contracts/DirectDepositV1.sol)

---

### Summary

The `DirectDepositV1` (DDA) contract is a personal deposit address that passively accepts any ERC-20 token via standard transfers. Its `creditDeposit()` function only iterates over tokens registered in `spotEngine.getProductIds()`, silently skipping any unsupported token. Unsupported tokens sent to a DDA become permanently locked with no automated recovery, mirroring the MultipliBridger vulnerability class exactly.

---

### Finding Description

`DirectDepositV1` is deployed per-subaccount as a deposit address. Users send ERC-20 tokens directly to their DDA, then `creditDeposit()` is called (by anyone — no access control) to forward those balances into the protocol via `endpoint.depositCollateralWithReferral`.

The critical path in `creditDeposit()`:

```solidity
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        uint32 productId = productIds[i];
        address tokenAddr = spotEngine.getToken(productId);
        require(tokenAddr != address(0), "Invalid productId.");
        IIERC20Base token = IIERC20Base(tokenAddr);
        uint256 balance = token.balanceOf(address(this));
        if (balance != 0) {
            token.approve(address(endpoint), balance);
            endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
        }
    }
}
```

The loop is bounded strictly to `spotEngine.getProductIds()`. Any token not registered as a spot product is never queried, never approved, and never forwarded. The function emits no event and does not revert when it encounters a non-zero balance of an unregistered token — it simply ignores it.

Because `DirectDepositV1` is a plain contract with a `receive()` function and no ERC-20 rejection mechanism, it cannot refuse incoming token transfers. Any ERC-20 sent to the DDA address lands there unconditionally.

The only recovery path is `withdraw(IIERC20Base token)` which carries `onlyOwner`:

```solidity
function withdraw(IIERC20Base token) external onlyOwner {
    uint256 balance = token.balanceOf(address(this));
    safeTransfer(token, msg.sender, balance);
}
```

The owner of every DDA is `ContractOwner`, so recovery requires the multisig to call `ContractOwner.withdrawFromDirectDepositV1(subaccount, token)` — a centralized manual intervention identical in structure to the `removeFunds` dependency in the MultipliBridger report.

---

### Impact Explanation

A user who sends an unsupported ERC-20 token to their DDA address loses those funds permanently unless the protocol's multisig manually intervenes. The `creditDeposit()` function provides zero feedback — no event, no revert — so the user receives no on-chain signal that their tokens are stuck. The asset delta is a direct, unrecoverable loss of the deposited token balance from the user's perspective.

---

### Likelihood Explanation

Realistic triggers include:
- A user sends a token that was once supported but has since been delisted (removed from `spotEngine.getProductIds()`).
- A user mistakenly sends the wrong ERC-20 to their DDA (e.g., the underlying asset of a vault token instead of the wrapped vault token itself — a scenario the codebase explicitly handles for supported products via `wrapVaultAsset`, but not for arbitrary tokens).
- A user sends a token before it is formally listed, expecting it to be processed once listed (it will not be retroactively credited).

All of these are reachable by an unprivileged user with no special access.

---

### Recommendation

1. In `creditDeposit()`, after processing all registered products, scan for any non-zero ERC-20 balance of tokens not in the registered set and emit a `StuckToken(address token, uint256 balance)` event so users and off-chain monitors can detect the condition.
2. Alternatively, maintain an explicit allowlist of accepted tokens in `DirectDepositV1` and revert (or emit and skip) when a non-allowlisted token balance is detected during `creditDeposit()`.
3. Document clearly that sending unsupported tokens to a DDA results in locked funds, and surface this in any front-end deposit flow.

---

### Proof of Concept

1. User calls `ContractOwner.directDepositV1Address[subaccount]` to obtain their DDA address.
2. User sends `1000 UNSUPPORTED_TOKEN` (any ERC-20 not in `spotEngine.getProductIds()`) to the DDA address via a standard ERC-20 `transfer`.
3. Anyone calls `DirectDepositV1.creditDeposit()` (or `ContractOwner.creditDepositV1(subaccount)`).
4. `creditDeposit()` iterates over `spotEngine.getProductIds()` — `UNSUPPORTED_TOKEN` is not present, so its balance is never read, approved, or forwarded.
5. `UNSUPPORTED_TOKEN` balance remains in the DDA indefinitely. No event is emitted.
6. The only recovery is the multisig calling `ContractOwner.withdrawFromDirectDepositV1(subaccount, UNSUPPORTED_TOKEN)`, which calls `DirectDepositV1.withdraw(IIERC20Base(UNSUPPORTED_TOKEN))` under `onlyOwner`.

    function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
        external
        onlyOwner
    {
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        if (token == address(0)) {
            uint256 preBalance = address(this).balance;
            DirectDepositV1(directDepositV1).withdrawNative();
            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");
            (bool success, ) = msg.sender.call{value: postBalance - preBalance}(
                ""
            );
            require(success, "xfer");
        } else {
            uint256 preBalance = IERC20Base(token).balanceOf(address(this));
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));
            require(postBalance > preBalance, "empty");
            IERC20Base(token).safeTransfer(
                msg.sender,
                postBalance - preBalance
            );
        }
    }
