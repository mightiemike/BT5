Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Allowlist Bypass via Router — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool populates with `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool sees the router as `msg.sender`, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`. Any pool admin who allowlists the router (the only way to let legitimate users trade via the router) simultaneously grants unrestricted swap access to every user, nullifying the per-user gate entirely.

## Finding Description

The call chain is fully confirmed in production code:

**Step 1 — Extension checks `sender`, not the end user:**
`SwapAllowlistExtension.beforeSwap` at L37 evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the first argument forwarded from the pool. [1](#0-0) 

**Step 2 — Pool passes its own `msg.sender` as `sender`:**
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, so whoever called `pool.swap()` becomes `sender` in the extension. [2](#0-1) 

**Step 3 — `ExtensionCalling._beforeSwap` forwards `sender` verbatim:**
The internal helper encodes `sender` directly into the `abi.encodeCall` payload sent to each extension. [3](#0-2) 

**Step 4 — Router calls `pool.swap()` directly, making itself `msg.sender`:**
`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` with no originator forwarding. The router is `msg.sender` of `pool.swap()`. [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

The extension has no mechanism to decode an originator from `extensionData`; the `bytes calldata` parameter is accepted but entirely ignored. [6](#0-5) 

**Existing guards are insufficient:** `allowAllSwappers` is a pool-wide bypass toggle, not a per-user check. There is no transient-storage originator field, no `extensionData` convention, and no secondary check on the actual end user. The `DepositAllowlistExtension` avoids this problem by checking `owner` (explicitly supplied by the caller), not `sender`; no equivalent field exists in the swap path. [7](#0-6) 

## Impact Explanation

Any non-allowlisted user can trade on a pool protected by `SwapAllowlistExtension` by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) once the pool admin has allowlisted the router. The pool admin's per-user access control is silently nullified for all router-mediated swaps. This is a confirmed admin-boundary break: an unprivileged actor bypasses the pool-level guard the admin configured and believed was enforced, matching the contest's allowed impact gate.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who wants allowlisted users to trade via the router must allowlist the router address — a natural, expected operational step. Once taken, the bypass is available to every address unconditionally and is repeatable without limit.

## Recommendation

The extension must verify the identity of the economic actor, not the intermediary. Two viable approaches:

1. **`extensionData` originator convention:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Requires a coordinated convention between router and extension.
2. **Transient-storage originator field:** The router writes the end user's address to a known transient storage slot before calling `pool.swap()`; the extension reads it directly. This avoids ABI coupling and is forgery-resistant because only the router (already trusted by the pool admin) writes the slot.

Until resolved, `SwapAllowlistExtension` must document that it only enforces per-user gating for direct `pool.swap()` calls and is entirely ineffective for router-mediated swaps.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][userA] = true   // legitimate user
  allowedSwapper[pool][router] = true  // admin adds this so userA can use the router

Attack:
  userB (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ...})

  router calls:
    pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)
    // msg.sender of pool.swap() == router

  pool calls:
    _beforeSwap(msg.sender=router, ...)
    → extension.beforeSwap(sender=router, ...)

  extension checks:
    allowedSwapper[pool][router] == true  →  passes

  Result: userB swaps successfully on a pool that should have blocked them.

  Direct pool call by userB would correctly revert:
    allowedSwapper[pool][userB] == false  →  NotAllowedToSwap
```

Foundry test plan: deploy pool with `SwapAllowlistExtension`, allowlist `userA` and the router, call `router.exactInputSingle` from `userB` (not allowlisted), assert the swap succeeds; then call `pool.swap` directly from `userB` and assert it reverts with `NotAllowedToSwap`.
