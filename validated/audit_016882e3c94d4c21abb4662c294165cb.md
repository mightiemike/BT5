### Title
Deprecated `verify_transaction_inclusion` Accepts Arbitrary `tx_id` Without Leaf-Node Validation, Enabling Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is a public, unprivileged-callable function that accepts a fully attacker-controlled `tx_id` and computes a Merkle root from it without ever checking that `tx_id` is a leaf node (a real transaction hash) rather than an internal Merkle tree node. An attacker can supply an internal node hash as `tx_id` with a crafted sibling proof, causing the function to return `true` for a transaction that does not exist. Any recipient NEAR contract that consumes this result is deceived into accepting a forged inclusion proof.

---

### Finding Description

`verify_transaction_inclusion` computes the Merkle root from the caller-supplied `tx_id`, `tx_index`, and `merkle_proof`, then compares it to the stored block's `merkle_root`: [1](#0-0) 

There is no check that `tx_id` is a leaf node. Because Bitcoin's Merkle tree is built from 32-byte hashes at every level, an internal node hash is indistinguishable from a transaction hash at the verification layer. The function's own `# Warning` comment acknowledges this explicitly: [2](#0-1) 

The function carries only a `#[pause]` guard, not a `#[trusted_relayer]` guard, so any unprivileged NEAR account can call it: [3](#0-2) 

`verify_transaction_inclusion_v2` was introduced to fix this via a mandatory coinbase co-proof that pins the tree depth, but the deprecated v1 function was never removed and remains callable: [4](#0-3) 

The `compute_root_from_merkle_proof` helper in `merkle-tools` performs no leaf-vs-internal-node distinction — it blindly hashes whatever `transaction_hash` it receives: [5](#0-4) 

---

### Impact Explanation

Any NEAR recipient contract (e.g., a cross-chain bridge, payment processor, or escrow) that calls `verify_transaction_inclusion` to gate fund release or state transitions can be deceived into accepting a forged proof. The attacker does not need to produce a real Bitcoin transaction — they only need to identify an internal Merkle node in any block already stored in `headers_pool` and supply the correct sibling path to that node. The function returns `true`, and the recipient contract acts as if a real transaction was confirmed.

The corrupted value is the **proof result** returned to the caller: a `true` that asserts transaction inclusion for a hash that is not a transaction.

---

### Likelihood Explanation

- The function is callable by any NEAR account without staking or role requirements.
- All Bitcoin block Merkle trees are fully public; computing internal node hashes requires only the block's transaction list, which is freely available.
- Any recipient contract that has not yet migrated from `verify_transaction_inclusion` to `verify_transaction_inclusion_v2` is immediately exploitable.
- The attack requires no privileged access, no key compromise, and no social engineering.

---

### Recommendation

Remove `verify_transaction_inclusion` entirely from the contract. If backward compatibility is required, gate it with the same coinbase co-proof check used in `verify_transaction_inclusion_v2`: [6](#0-5) 

The coinbase co-proof check pins the Merkle tree depth, making it impossible to supply an internal node at a shallower depth as a valid `tx_id`.

---

### Proof of Concept

Consider a block stored in `headers_pool` with four transactions `[tx0, tx1, tx2, tx3]`:

```
merkle_root = hash(h01, h23)
    h01 = hash(tx0, tx1)
    h23 = hash(tx2, tx3)
```

Attacker calls `verify_transaction_inclusion` with:
- `tx_id      = h01`  ← internal node, not a real transaction
- `tx_index   = 0`
- `merkle_proof = [h23]`
- `tx_block_blockhash` = the stored block hash
- `confirmations = 1`

Execution inside `compute_root_from_merkle_proof`:

```
current_hash = h01, position = 0
  → position even: current_hash = hash(h01, h23) = merkle_root
return merkle_root
```

`merkle_root == header.block_header.merkle_root` → function returns **`true`**.

The internal node `h01` is not a Bitcoin transaction. No such transaction exists. The recipient contract is deceived. [5](#0-4) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
