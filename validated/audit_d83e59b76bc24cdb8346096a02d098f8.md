Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of the pool call) and performs its allowlist check exclusively on `owner` (the caller-supplied position recipient). Because `MetricOmmPool.addLiquidity` accepts an arbitrary `owner` with no `msg.sender == owner` guard, any non-allowlisted address can bypass the allowlist by supplying any allowlisted address as `owner`. The pool admin's access control intent is entirely defeated by an unprivileged caller.

## Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` has the following signature and check:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The first parameter (`sender`, the actual depositor/payer) is unnamed and discarded. The check is performed on `owner`, the position recipient chosen by the caller. [1](#0-0) 

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner`, with no `msg.sender == owner` guard:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [2](#0-1) 

By contrast, `removeLiquidity` enforces `if (msg.sender != owner) revert NotPositionOwner();`, a guard that is entirely absent from `addLiquidity`. [3](#0-2) 

The periphery's `addLiquidityExactShares(pool, owner, ...)` overload accepts an arbitrary `owner` and only validates it is non-zero, making the bypass trivially accessible to EOAs: [4](#0-3) [5](#0-4) 

The NatSpec for the extension explicitly states its purpose is to gate by depositor address, confirming `sender` is the intended subject of the check: [6](#0-5) 

**Exploit path:**
1. Pool admin deploys pool with `DepositAllowlistExtension` and calls `setAllowedToDeposit(pool, alice, true)`. Bob is not allowlisted.
2. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `_beforeAddLiquidity(bob, alice, ...)` → extension checks `allowedDepositor[pool][alice]` = `true` → passes.
4. Bob's tokens are pulled via callback; Alice receives the LP position.
5. Bob has deposited into a restricted pool without being allowlisted.

The same bypass is reachable via `addLiquidityWeighted(pool, owner, ...)`: [7](#0-6) 

## Impact Explanation

The deposit allowlist access control is completely ineffective. Any unprivileged address can deposit into a restricted pool by naming any allowlisted address as `owner`. This constitutes an admin-boundary break: the pool admin's configured access control is bypassed by an unprivileged caller. Additionally, allowlisted addresses can have unwanted LP positions forced onto them without consent, exposing them to impermanent loss. Pools configured for regulatory/KYC compliance provide no actual restriction.

## Likelihood Explanation

`addLiquidity` is a public, permissionless function. The bypass requires only knowing one allowlisted address, which is publicly readable from the `allowedDepositor` mapping or emitted events. No special privileges, flash loans, or complex setup are required. The periphery `addLiquidityExactShares(pool, owner, ...)` makes the attack trivially accessible to EOAs. The attack is repeatable indefinitely.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor/payer) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Additionally, consider whether `addLiquidity` should enforce `msg.sender == owner` (as `removeLiquidity` does) to prevent forced position griefing, or whether the allowlist should check both `sender` and `owner` depending on intended semantics.

## Proof of Concept

1. Deploy pool with `DepositAllowlistExtension`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` — Alice allowlisted; Bob is not.
3. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)` directly, or via `liquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, extensionData)`.
4. Extension receives `(sender=bob, owner=alice)`, checks `allowedDepositor[pool][alice]` = `true` → no revert.
5. Bob's tokens are pulled; Alice receives the position. Bob has bypassed the allowlist.

A Foundry test can confirm this by: deploying the extension, setting only Alice as allowed, calling `addLiquidity` as Bob with `owner=alice`, and asserting the call succeeds and Alice holds the resulting position shares.
