### Title
CEI Pattern Violation in `withdrawCollateral` — External Token Transfer Before Balance Update and Health Check - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs an external token transfer via `handleWithdrawTransfer` **before** decrementing the sender's on-chain balance in `SpotEngine` and **before** executing the post-withdrawal health check. This is a direct structural analog to the `_refund()` CEI violation in the referenced report.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

```
handleWithdrawTransfer(token, sendTo, amount, idx);   // ← EXTERNAL CALL (line 408)

spotEngine.updateBalance(productId, sender, amountRealized);  // ← STATE CHANGE (line 412)
spotEngine.assertUtilization(productId);

require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);  // ← HEALTH CHECK (line 419)
```

`handleWithdrawTransfer` in `Clearinghouse` expands to two external calls:

```solidity
token.safeTransfer(withdrawPool, uint256(amount));                          // ERC20 transfer out
BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);   // cross-contract call
```

`submitWithdrawal` in `BaseWithdrawPool` then calls:

```solidity
handleWithdrawTransfer(token, sendTo, amount);  // token.safeTransfer(sendTo, amount)
```

The token transfer to `sendTo` is the final external call. At the moment this transfer executes, `spotEngine.updateBalance` has **not yet been called**, meaning the sender's balance in `SpotEngine` still reflects the pre-withdrawal amount. The health check at line 419 has also not yet run. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

During the `token.safeTransfer(sendTo, amount)` call, the sender's `SpotEngine` balance has not been decremented. If `sendTo` is a contract with a token-receive callback (e.g., ERC777 `tokensReceived`, or a token with hooks), the callback executes in a state where:

- The sender's balance appears at the **pre-withdrawal level** to any on-chain reader.
- The health check has **not yet been enforced**.

A malicious `sendTo` contract could exploit this window by calling `submitFastWithdrawal` (a public, permissionless function in `BaseWithdrawPool`) with a separately signed withdrawal transaction. At callback time, `markedIdxs[idx]` for the *current* withdrawal is already set (preventing exact replay), but a **different valid idx** for the same sender — whose balance still reads as full — could be submitted. The `WithdrawPool` holds pooled liquidity from multiple withdrawals, so it may have sufficient token balance to satisfy the second transfer. The net result is the sender receives tokens twice while the balance decrement and health check for the first withdrawal have not yet been committed. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

**Preconditions required:**

1. The collateral token must support receive-side callbacks (ERC777 or a custom ERC20 with hooks). Standard USDC/ERC20 does not trigger callbacks on `transfer`, so this is not exploitable with vanilla ERC20 tokens.
2. The attacker must control `sendTo` — possible via `WithdrawCollateralV2`, where `sendTo` is user-specified and the sequencer passes it through without filtering.
3. The attacker must hold a second valid signed withdrawal transaction (a different `idx`) that has not yet been marked. [7](#0-6) 

Condition 1 is the binding constraint. On the Ink network with standard ERC20 collateral, the likelihood is **low**. If the protocol ever lists an ERC777-compatible or hook-bearing token, likelihood rises to **medium-high**.

---

### Recommendation

Reorder `withdrawCollateral` to follow strict CEI:

1. Decrement the sender's balance via `spotEngine.updateBalance` first.
2. Run `spotEngine.assertUtilization` and `getHealth` checks.
3. Only then call `handleWithdrawTransfer`.

```solidity
// CORRECT ORDER
int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);   // Effects first
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

handleWithdrawTransfer(token, sendTo, amount, idx);            // Interaction last
``` [8](#0-7) 

---

### Proof of Concept

1. Attacker holds two valid signed `WithdrawCollateralV2` transactions: `idx=N` (amount=X) and `idx=M` (amount=X), both for the same subaccount. `idx=M` has not yet been submitted to the sequencer.
2. Attacker sets `sendTo` in `idx=N` to a malicious ERC777-receiver contract they control.
3. Sequencer processes `idx=N` → `Clearinghouse.withdrawCollateral` → `handleWithdrawTransfer` → `token.safeTransfer(withdrawPool, X)` → `submitWithdrawal` → `token.safeTransfer(maliciousContract, X)`.
4. ERC777 `tokensReceived` fires on `maliciousContract`. At this moment, `spotEngine` balance for the subaccount is still `+X` (not yet decremented).
5. `maliciousContract` calls `BaseWithdrawPool.submitFastWithdrawal(idx=M, signedTxM, validSigs)`. `markedIdxs[M]` is unset; the pool holds pooled liquidity. The fast withdrawal succeeds, transferring another `X` tokens to the attacker.
6. Control returns to `withdrawCollateral`, which now decrements the balance by `X` and runs the health check — but the attacker has already received `2X` tokens. [1](#0-0) [5](#0-4)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
