Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool populates with its own `msg.sender` — the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the end user. A pool admin who allowlists the router (required for any router-mediated swap to succeed) inadvertently grants every router user access, completely defeating the per-user allowlist.

## Finding Description

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` at line 231, forwarding its own `msg.sender` as `sender`: [1](#0-0) 

`ExtensionCalling._beforeSwap` passes this `sender` directly to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, the pool's `msg.sender` is the router contract, not the end user (`msg.sender` of `exactInputSingle` is stored only in transient callback context, never forwarded to the pool as `sender`): [4](#0-3) 

This creates an impossible dilemma: not allowlisting the router causes all router swaps to revert; allowlisting the router causes `allowedSwapper[pool][router] == true` to pass for every user, since the extension only ever sees `sender = router`. The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

## Impact Explanation

A pool admin deploys a pool with `SwapAllowlistExtension` to restrict swapping to a curated set of addresses (e.g., KYC'd counterparties). Once the router is allowlisted (required for normal operation), any unprivileged user can call `exactInputSingle` and execute swaps against the pool's LP liquidity. LP principal is exposed to the full universe of router users rather than the intended restricted set. This is a direct, on-chain access-control bypass with no mitigation once the router is allowlisted — matching the "broken core pool functionality causing loss of funds" impact criterion.

## Likelihood Explanation

The router is the standard user-facing entry point. Any pool using `SwapAllowlistExtension` that also supports router-mediated swaps must allowlist the router, triggering the bypass automatically. No special privilege is required — a single public call to `exactInputSingle` suffices. The condition is met by any pool that combines both features, which is the expected deployment pattern.

## Recommendation

The extension must gate on the end user, not the intermediary. The recommended fix is a trusted-router registry pattern: if `sender` is a registered trusted router, `SwapAllowlistExtension` decodes the actual swapper from `extensionData` (which the router must embed as `abi.encode(msg.sender)`); otherwise it checks `sender` directly. This requires the router to be modified to always populate `extensionData` with the originating user, and the extension to maintain a registry of trusted routers whose self-reported identity is accepted.

## Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Pool admin calls swapExtension.setAllowedToSwap(pool, router, true)
   — required so that any router swap passes the check.
3. Alice (not individually allowlisted) calls:
       router.exactInputSingle({pool: pool, ..., recipient: alice, ...})
4. Router calls pool.swap(alice, ...) — pool's msg.sender = router.
5. Pool calls _beforeSwap(sender=router, ...).
6. Extension checks allowedSwapper[pool][router] == true → passes.
7. Alice's swap executes against LP liquidity despite never being allowlisted.
```
