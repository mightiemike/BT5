Audit Report

## Title
Router Address Substitution Breaks `SwapAllowlistExtension` Per-User Guard — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating EOA. If the router is allowlisted for a pool, any EOA can bypass the per-user swap allowlist by routing through it, completely nullifying the access control the pool admin intended to enforce.

## Finding Description
`MetricOmmSimpleRouter.exactInputSingle` stores the originating `msg.sender` only in transient callback context for payment purposes, then calls `IMetricOmmPoolActions(params.pool).swap(...)` directly — making the router itself the `msg.sender` of that call. `MetricOmmPool.swap` unconditionally passes `msg.sender` as the `sender` argument to `_beforeSwap`, which ABI-encodes it and dispatches it to `SwapAllowlistExtension.beforeSwap`. The extension evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` = pool and `sender` = router address. The originating EOA is never visible to the extension. If the router is allowlisted (the natural configuration for a pool that wants to support router-mediated swaps), every EOA in existence passes the guard regardless of whether it is individually allowlisted.

## Impact Explanation
A pool admin deploying `SwapAllowlistExtension` to restrict swaps to KYC'd or whitelisted EOAs must also allowlist the router for those EOAs to use it. Once the router is allowlisted, any non-permitted EOA can call `router.exactInputSingle` and the guard passes because `allowedSwapper[pool][router] == true`. The per-user restriction is completely nullified: non-permitted actors execute swaps against a pool intended to be restricted, draining token0/token1 from a pool whose access control invariant is broken. This constitutes a broken core pool functionality and an admin-boundary break reachable by any unprivileged EOA.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical swap interface for the protocol. Any pool admin who deploys `SwapAllowlistExtension` and also wants to support router-mediated swaps will hit this condition. The router allowlist path is the natural, expected configuration, making this trivially reachable by any EOA with no special privileges or setup.

## Recommendation
Pass the originating user identity through the router to the pool, or have the extension resolve the true economic actor:
1. **Router-side:** Have `exactInputSingle` (and all other swap entry points) encode `msg.sender` into `extensionData` in a standardized prefix slot, and have `SwapAllowlistExtension.beforeSwap` decode and check that value when `sender` is a known router.
2. **Extension-side:** Introduce a separate `allowedRouter` mapping that, when set, causes the extension to decode and verify the originating user from `extensionData` rather than granting blanket access to all callers through the router.
