### Title
ML-DSA-65 Gas Key Host Functions Pass Wire Length Instead of Trie-ID Length to Fee Calculators, Causing ~59× Fee Divergence from Transaction Path — (File: `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` host functions pass the raw borsh wire length of the public key (`public_key_len as usize`) to `gas_key_transfer_exec_fee` and `gas_key_add_key_exec_fee`. Those functions treat the argument as the **on-trie identifier length**. For ML-DSA-65 keys the two values diverge by ~59×: wire = 1953 bytes, trie-id = 33 bytes. The transaction path in `runtime/runtime/src/config.rs` correctly passes `action.public_key.trie_id_len()` (= 33) to the same fee helpers. The result is that any contract invoking these host functions with an ML-DSA-65 key is charged ~59× more gas than the equivalent transaction, breaking the fee-parity protocol invariant and causing gas exhaustion for any contract that budgets based on the documented transaction cost.

---

### Finding Description

`PublicKey::trie_id_len()` and `PublicKey::len()` are identical for ED25519 (33 B) and SECP256K1 (65 B), but diverge for ML-DSA-65:

| method | ED25519 | SECP256K1 | ML-DSA-65 |
|---|---|---|---|
| `len()` | 33 | 65 | **1953** |
| `trie_id_len()` | 33 | 65 | **33** |

The ML-DSA-65 trie key stores a SHA3-256 hash (tag + 32 B), not the 1952-byte raw pubkey. [1](#0-0) 

**Transaction path** — both send and exec fees use `trie_id_len()`: [2](#0-1) [3](#0-2) 

**Host function path (Wasmer)** — both send and exec fees use the raw `public_key_len` (wire length): [4](#0-3) 

**Host function path (Wasmtime)** — same error: [5](#0-4) 

`gas_key_transfer_exec_fee` is documented as computing cost based on the **trie key length**: [6](#0-5) 

When `public_key_len = 1953` is passed instead of `33`, `access_key_key_len(account_id_len, 1953)` inflates the trie-key-length estimate by 1920 bytes, multiplied by `gas_key_byte` exec rate, yielding a ~59× exec-fee overcharge.

The same wrong argument is passed to `gas_key_add_key_exec_fee` in both `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`: [7](#0-6) [8](#0-7) 

The transaction path for `AddKey` with a gas-key permission correctly uses `trie_id_len()`: [9](#0-8) 

The fee-parity test (`test_gas_key_fee_parity`) only exercises ED25519 keys, so the ML-DSA-65 divergence is untested: [10](#0-9) 

---

### Impact Explanation

Once both `GasKeys` and `PostQuantumSignatures` (protocol version 85) are active, any contract can call these host functions with an ML-DSA-65 key. The exec fee charged is ~59× the amount charged for the identical operation submitted as a transaction. Contracts that budget gas based on the transaction-path cost will exhaust gas and fail. The fee-parity invariant — that the same action costs the same gas regardless of whether it is submitted as a transaction or dispatched via a host function — is broken for ML-DSA-65 keys across all three gas-key host functions.

---

### Likelihood Explanation

The trigger is any contract that holds or manages ML-DSA-65 gas keys and calls `promise_batch_action_transfer_to_gas_key` or the add-gas-key host functions. No privileged role is required. The divergence is deterministic and reproducible on every invocation.

---

### Recommendation

Decode the public key before computing fees, then use `decoded_key.trie_id_len()` in place of `public_key_len as usize` for all three host functions in both the Wasmer (`logic/logic.rs`) and Wasmtime (`wasmtime_runner/logic.rs`) paths:

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_key = public_key.decode()?;
let trie_key_len = decoded_key.trie_id_len();

let send = gas_key_transfer_send_fee(&self.fees_config, sir, trie_key_len);
let exec = gas_key_transfer_exec_fee(&self.fees_config, receiver_id.len(), trie_key_len);
// ... then append action using decoded_key (already decoded above)
```

Apply the same fix to `gas_key_add_key_exec_fee` calls in the add-gas-key host functions. Extend `test_gas_key_fee_parity` to cover ML-DSA-65 keys.

---

### Proof of Concept

For an ML-DSA-65 key targeting a 10-character account ID, the exec-fee byte count diverges as follows:

| path | `public_key_len` passed | `trie_key_len` computed | extra bytes vs correct |
|---|---|---|---|
| Transaction (`config.rs`) | — | `access_key_key_len(10, 33)` = 45 | 0 |
| Host function (`logic.rs`) | 1953 | `access_key_key_len(10, 1953)` = 1965 | **+1920** |

The per-byte exec fee (`gas_key_byte`) is multiplied by `trie_key_len + min_gas_key_borsh_len`. With 1920 extra bytes, the exec portion of the fee is inflated by `1920 × gas_key_byte` per call. At current mainnet `gas_key_byte` pricing this represents a significant overcharge that will exhaust any gas budget sized for the transaction path.

```rust
    /// entry owned by this public key. For ed25519/secp256k1 this matches

            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
```

```rust
/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
```
