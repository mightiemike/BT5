Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and checks `owner` (the position beneficiary) instead. Because `addLiquidity` accepts an arbitrary `owner` address with no on-chain constraint tying it to `msg.sender`, any non-allowlisted address can bypass the guard by naming an allowlisted address as `owner`. The deposit allowlist is rendered a no-op for the actual token provider.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and an arbitrary caller-supplied `owner` as the position beneficiary to the extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

There is no restriction requiring `msg.sender == owner` in `addLiquidity` (unlike `removeLiquidity`, which enforces `if (msg.sender != owner) revert NotPositionOwner()` at L206).

`DepositAllowlistExtension.beforeAddLiquidity` ignores the first parameter entirely and checks `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`allowedDepositor[pool][owner]` is checked — the position beneficiary — not `allowedDepositor[pool][sender]` — the actual token provider. The wrong variable is tested.

`SwapAllowlistExtension.beforeSwap` demonstrates the correct pattern, checking `sender`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

The asymmetry between the two extensions confirms the deposit extension checks the wrong address.

## Impact Explanation

Any address not on the deposit allowlist can call `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, "")`. The extension sees `owner = allowlistedAddress` → `allowedDepositor[pool][allowlistedAddress] == true` → no revert. The non-allowlisted caller supplies the tokens via the liquidity callback and the LP position is minted to the allowlisted address. Pools configured as restricted (KYC-gated, institutional-only, exploit-mitigation allowlists) are fully open to any depositor. The admin-boundary invariant of the extension is completely defeated: the allowlist is enforced on the position beneficiary, not on the token provider.

## Likelihood Explanation

The trigger requires only a standard `addLiquidity` call with any known allowlisted address as `owner`. The `allowedDepositor` mapping is public, so any observer can identify allowlisted addresses on-chain. No privileged access, flash loan, or special token behavior is needed. The bypass is trivially constructable by any address at any time.

## Recommendation

Replace the unnamed first parameter with `sender` and check it instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` in the `beforeAddLiquidity` order.
2. Admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is **not** allowlisted.
3. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
4. Extension receives `sender = bob` (ignored), `owner = alice`. It checks `allowedDepositor[pool][alice]` → `true` → no revert.
5. Bob's tokens are pulled via the liquidity callback; the LP position is minted to Alice.
6. Bob has deposited into a restricted pool without being on the allowlist. The guard is bypassed by supplying an allowlisted `owner`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
