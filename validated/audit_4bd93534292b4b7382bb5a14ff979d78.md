Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates on Router Address Instead of End-User, Allowing Any Router Caller to Bypass Per-User Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the immediate `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract, not the actual end user. If the router is allowlisted — a natural operational step — every caller of the router bypasses the per-user guard, completely nullifying the allowlist for router-mediated swaps.

## Finding Description
The full call chain is confirmed in production code:

1. `MetricOmmPool.swap()` passes `msg.sender` (the router) as `sender` to `_beforeSwap`. [1](#0-0) 

2. `ExtensionCalling._beforeSwap` forwards that `sender` (the router address) to the extension. [2](#0-1) 

3. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — `msg.sender` is the pool, `sender` is the router — never inspecting the actual end user. [3](#0-2) 

4. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly with no user-identity forwarding; the pool sees only the router as `msg.sender`. [4](#0-3) 

The asymmetry with `DepositAllowlistExtension` is structurally decisive: that extension correctly checks `owner` (the actual position beneficiary, passed as a distinct parameter), not `sender` (the caller). [5](#0-4) 

`addLiquidity` separates `msg.sender` (payer/operator) from `owner` (beneficiary), enabling the extension to gate on the correct identity. `swap` has no such separation — `sender` is always the immediate caller — so when a router intermediates, the extension gates on the wrong address. The protocol's own design pattern (used in `DepositAllowlistExtension`) is not applied here. [6](#0-5) 

## Impact Explanation
A pool deployer configures `SwapAllowlistExtension` to restrict swaps to KYC-verified or otherwise approved counterparties, and allowlists `MetricOmmSimpleRouter` so approved users can swap via the standard UX. Because the extension checks the router's address — which is allowlisted — every caller of the router passes the guard regardless of individual allowlist status. Unauthorized users can execute swaps in a pool intended to be access-controlled, extracting output tokens from LP reserves they should never have been able to touch. This is a direct loss of LP assets and a broken core pool access-control invariant, meeting the allowed impact gate for broken core pool functionality causing loss of funds. [3](#0-2) 

## Likelihood Explanation
Allowlisting the router is the natural operational step for any pool that wants to support standard UX while maintaining an allowlist. The pool admin has no reason to suspect this collapses the per-user guard, because the `DepositAllowlistExtension` pattern correctly handles the analogous case. The trigger requires only: (1) a pool with `SwapAllowlistExtension` configured, (2) the router allowlisted, and (3) any non-allowlisted user calling through the router. All three conditions are routine in production. No privileged attacker capability is required — any unprivileged router caller can exploit this once the configuration is in place. [7](#0-6) 

## Recommendation
Pass the actual end-user identity through the swap path so the extension can gate on it. Two options:

1. **Encode the originating user in `extensionData`** at the router level and have `SwapAllowlistExtension.beforeSwap` decode and verify it (requires a trusted encoding convention and that the pool enforces the encoding).
2. **Add a `swapFor(address user, ...)` entry point** on the pool that records the intended user separately from `msg.sender`, mirroring the `owner` pattern used by `addLiquidity`, and pass that identity to extensions as a distinct argument.

Either way, `SwapAllowlistExtension.beforeSwap` must check the actual end-user identity, not the immediate caller. [6](#0-5) 

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][router] = true        // router allowlisted for convenience
  allowedSwapper[pool][alice]  = true        // alice is individually approved
  allowedSwapper[pool][bob]    = false       // bob is NOT approved

Attack:
  bob calls router.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(...)            // msg.sender at pool = router
    → pool calls _beforeSwap(router, ...)
    → SwapAllowlistExtension checks allowedSwapper[pool][router] == true  ✓
    → swap executes, bob receives output tokens

Result:
  bob, who is not on the allowlist, successfully swaps through the pool.
  The SwapAllowlistExtension guard is bypassed entirely.
``` [4](#0-3) [3](#0-2)
