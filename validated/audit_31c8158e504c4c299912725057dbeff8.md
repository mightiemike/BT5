Audit Report

## Title
SwapAllowlistExtension Bypassed via Router: Any Unprivileged User Can Swap on Allowlisted Pools — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the immediate caller of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates a swap, the router becomes `msg.sender`, so the extension checks the router's allowlist status rather than the end user's. Any pool admin who allowlists the router (required for router-mediated swaps to succeed) inadvertently grants every unprivileged user the ability to bypass the per-user allowlist entirely.

## Finding Description

**Root cause — pool passes `msg.sender` as `sender` to the extension:**

In `MetricOmmPool.swap`, the `sender` forwarded to `_beforeSwap` is always `msg.sender` of the `pool.swap()` call: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes and forwards this `sender` verbatim to every configured extension via `_callExtensionsInOrder`: [2](#0-1) 

**Extension checks the wrong identity:**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Router is the sole on-chain identity the pool sees:**

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making the router `msg.sender` from the pool's perspective: [4](#0-3) 

The pool also issues the swap callback to `msg.sender` (the router), confirming the router is the sole on-chain identity the pool observes: [5](#0-4) 

**Exploit flow:**

1. Admin deploys pool with `SwapAllowlistExtension` as `BEFORE_SWAP` extension.
2. Admin calls `setAllowedToSwap(pool, Alice, true)` — Alice is the intended user.
3. Admin calls `setAllowedToSwap(pool, router, true)` — required for Alice to use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: Bob, ...})`.
5. Router calls `pool.swap(recipient=Bob, ...)` → `msg.sender = router`.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → passes.
8. Swap executes; Bob receives output tokens. `allowedSwapper[pool][Bob]` was never checked.

No existing guard recovers the originating EOA. The `extensionData` field is caller-supplied and unverified, so it cannot serve as a trusted identity channel without protocol-level enforcement.

## Impact Explanation

The `SwapAllowlistExtension` access control is completely defeated for any pool that permits router-mediated swaps. Unauthorized traders can execute swaps on pools explicitly configured to restrict access to a curated set of counterparties (e.g., KYC'd institutions). Consequences include front-running of institutional order flow, extraction of LP inventory at oracle mid-price by adversarial traders, and violation of the pool admin's explicit access policy. LP principal is at risk because pool liquidity was sized and priced for a controlled counterparty set. This constitutes broken core pool functionality causing potential loss of funds and an admin-boundary break where an unprivileged path bypasses the pool admin's access control configuration.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical periphery entry point; most users interact with pools through it. Any pool admin who enables both a swap allowlist and router access — a natural and expected combination — creates the bypass automatically and silently. No special privilege, flash loan, or oracle manipulation is required: a plain router call suffices. The bypass is repeatable and leaves no distinguishing on-chain trace.

## Recommendation

The extension must gate the economic actor, not the immediate caller. Viable approaches:

1. **Propagate the originating user via `extensionData` with router enforcement**: The router encodes `msg.sender` (the end user) into `extensionData` before forwarding to the pool. `SwapAllowlistExtension` decodes and checks that address only when `sender` is a known router. Requires a trusted router registry in the extension.

2. **Transient-storage forwarding**: The router writes the originating user address into a transient storage slot before calling `pool.swap()`. The extension reads from that slot when `sender` is a recognized router. Avoids calldata manipulation but requires EIP-1153.

3. **Document and enforce router separation**: Treat the router as a separate pool type (`allowAllSwappers = true`) and prohibit adding the router to per-user-gated pools. Enforce this invariant in the factory or extension setter.

─────

Attack
──────
4. Bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool:            pool,
           recipient:       Bob,
           amountIn:        X,
           ...
       })

5. Router calls pool.swap(recipient=Bob, ...) → msg.sender = router

6. Pool calls _beforeSwap(sender=router, ...)
   [MetricOmmPool.sol L230-240, ExtensionCalling.sol L149-177]

7. SwapAllowlistExtension.beforeSwap checks:
       allowedSwapper[pool][router]  →  true  ✓
   [SwapAllowlistExtension.sol L37]

Foundry test plan: deploy pool with `SwapAllowlistExtension`, allowlist only Alice and the router, call `exactInputSingle` from Bob's address, assert the swap succeeds and Bob receives tokens despite `allowedSwapper[pool][Bob] == false`.
