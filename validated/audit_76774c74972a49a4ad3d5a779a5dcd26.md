### Title
Duplicate-Last-Leaf Position Ambiguity Allows Proof Replay at Phantom `tx_index` — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`)

---

### Summary

`compute_root_from_merkle_proof` does not enforce that `tx_index` is within the actual bounds of the Merkle tree. In any block whose transaction count is odd, Bitcoin's tree construction duplicates the last leaf. The resulting proof for the real last leaf (position N-1) is mathematically identical to a proof for the phantom duplicate (position N). `verify_transaction_inclusion_v2` therefore returns `true` for both positions with the same `tx_id` and the same proof bytes. A downstream bridge that uses `(tx_id, tx_index)` as a composite replay-prevention key can be tricked into processing the same on-chain event twice.

---

### Finding Description

**`compute_root_from_merkle_proof` — position is not uniquely bound** [1](#0-0) 

The function iterates over `merkle_proof`, at each step choosing `H(current, sibling)` or `H(sibling, current)` based on `current_position % 2`. It never checks whether `transaction_position` is less than the actual number of leaves in the tree.

**Why the duplicate-leaf collapses two positions to the same root**

Consider a 3-transaction block `[T0, T1, T2]`. Bitcoin pads it to `[T0, T1, T2, T2]`:

```
Level 0:  T0    T1    T2    T2   (T2 duplicated)
Level 1:  H(T0,T1)   H(T2,T2)
Root:     H(H(T0,T1), H(T2,T2))
```

The canonical proof for T2 at **position 2** is `[T2, H(T0,T1)]`.

Verification at **position 2** (even):
```
step 1: pos=2 (even) → H(T2, proof[0]=T2) = H(T2,T2);  pos→1
step 2: pos=1 (odd)  → H(proof[1]=H(T0,T1), H(T2,T2)) = Root ✓
```

Verification at **position 3** (phantom) with the **same proof**:
```
step 1: pos=3 (odd)  → H(proof[0]=T2, T2) = H(T2,T2);  pos→1
step 2: pos=1 (odd)  → H(proof[1]=H(T0,T1), H(T2,T2)) = Root ✓
```

Both calls return the same root and therefore both pass the equality check in `verify_transaction_inclusion`. [2](#0-1) 

**The coinbase check in v2 does not close this gap**

`verify_transaction_inclusion_v2` validates that `coinbase_tx_id` at index 0 produces the correct root: [3](#0-2) 

This check only confirms the coinbase is in the tree. It says nothing about whether the caller's `tx_index` is within the real leaf count. The attacker supplies a valid coinbase proof (trivially obtained from any block explorer) alongside the replayed `tx_index`.

**The GC timing is a red herring**

`run_mainchain_gc` removes old blocks from both `mainchain_header_to_height` and `headers_pool`: [4](#0-3) 

`verify_transaction_inclusion` panics if the block is absent from `mainchain_header_to_height`: [5](#0-4) 

So GC'd blocks cannot be used. The duplicate-position attack works on **any retained block** with an odd transaction count; GC timing is irrelevant to the root cause.

### Impact Explanation

`verify_transaction_inclusion_v2` is a public, unpermissioned view call: [6](#0-5) 

Any caller can supply `tx_index = N` (the phantom position) for a block with `N` transactions (N odd). The function returns `true`. If a downstream bridge, mint, or withdrawal contract uses `(tx_id, tx_index)` as its replay-prevention key — a natural design since `tx_index` is part of the proof API — the attacker can replay the same Bitcoin transaction a second time by switching from `tx_index = N-1` to `tx_index = N`, causing a double-mint or double-withdrawal.

---

### Likelihood Explanation

- Odd transaction counts are common in real Bitcoin blocks.
- The attacker needs no special role, no relayer key, and no privileged access — only a public NEAR call.
- The canonical proof for the real last leaf is publicly available from any Bitcoin node or block explorer.
- The only prerequisite is that the downstream bridge uses `(tx_id, tx_index)` as a composite key rather than `tx_id` alone.

---

### Recommendation

1. **Encode tree size in the proof.** Pass the total leaf count alongside the proof and assert `tx_index < leaf_count` before computing the root.
2. **Alternatively, reject `tx_index` values that are ≥ the implied tree size.** The tree depth is `merkle_proof.len()`, so the maximum valid index is `2^depth - 1`, but the actual leaf count must be supplied or derived from the block header (Bitcoin does not store it in the 80-byte header, so it must be passed as an additional argument and committed to).
3. **Downstream bridges** should use only `tx_id` (or a hash of the full serialized transaction) as the replay-prevention key, not `(tx_id, tx_index)`.

### Proof of Concept

```
Block: 3 transactions [coinbase, T1, T2]
Merkle tree (padded): [coinbase, T1, T2, T2]

Canonical proof for T2 at index 2:
  proof = [T2, H(coinbase, T1)]

Call 1 (legitimate):
  verify_transaction_inclusion_v2(
      tx_id = T2, tx_index = 2,
      merkle_proof = [T2, H(coinbase,T1)],
      coinbase_tx_id = coinbase, coinbase_merkle_proof = [T1, H(T2,T2)],
      ...
  ) → true   ← bridge mints once

Call 2 (replay, same proof, phantom index):
  verify_transaction_inclusion_v2(
      tx_id = T2, tx_index = 3,       ← only change
      merkle_proof = [T2, H(coinbase,T1)],
      coinbase_tx_id = coinbase, coinbase_merkle_proof = [T1, H(T2,T2)],
      ...
  ) → true   ← bridge mints again
```

Both calls traverse identical arithmetic paths through `compute_root_from_merkle_proof` and produce the same root, so both return `true`. The light client has been bypassed for a nonexistent transaction position. [1](#0-0)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
