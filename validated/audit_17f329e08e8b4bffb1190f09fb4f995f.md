### Title
Missing Gateway Validation for Non-Empty `account_deployment_data` in Invoke/Declare V3 Transactions Causes OS Assertion Failure During Block Proving — (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The Starknet OS Cairo program asserts `account_deployment_data_size = 0` for both Invoke V3 and Declare V3 transactions because the field is not yet supported. The gateway's stateless validator enforces the analogous constraint for `paymaster_data` via `validate_empty_paymaster_data`, but no equivalent validation exists for `account_deployment_data`. An unprivileged user can submit a V3 transaction with non-empty `account_deployment_data`, have it admitted by the gateway, included in a block, and executed successfully by the blockifier — but the OS will abort with an assertion failure when it attempts to compute the transaction hash during block proving, preventing L1 settlement of that block.

### Finding Description

**OS-level assertion (Cairo):**

In `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`, both `compute_invoke_transaction_hash` and `compute_declare_transaction_hash` contain a hard assertion:

```cairo
// TODO(Noa, 01/01/2026): remove the following `assert` once the field is supported.
assert account_deployment_data_size = 0;
```

This assertion fires unconditionally before the hash is computed. Any non-zero `account_deployment_data` causes the OS execution to abort.

**Gateway validates `paymaster_data` but not `account_deployment_data`:**

`crates/apollo_gateway/src/stateless_transaction_validator.rs` contains `validate_empty_paymaster_data`, which rejects all V3 transactions with non-empty `paymaster_data` — precisely because the OS also asserts `paymaster_data_length = 0` in `hash_tx_common_fields`. However, no corresponding `validate_empty_account_deployment_data` function exists. The `account_deployment_data` field is present in `RpcInvokeTransactionV3` and `RpcDeclareTransactionV3` and is accepted without restriction.

**Blockifier includes the field in the hash but does not gate on it:**

`crates/starknet_api/src/transaction_hash.rs` `get_invoke_transaction_v3_hash` unconditionally hashes `account_deployment_data` into the transaction hash. The blockifier's pre-validation (`validate_proof_facts`) only checks proof facts; there is no check that `account_deployment_data` is empty. The transaction is therefore executed successfully and committed to the block with a hash that includes the non-empty field.

**Divergence at proving time:**

When the OS processes the block, it calls `compute_invoke_transaction_hash` and immediately hits `assert account_deployment_data_size = 0`. The OS aborts before producing a hash, so the block proof fails entirely. The blockifier-computed hash (which included the field) and the OS-computed hash (which cannot be computed) are irreconcilable.

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

Any unprivileged user can craft an `RpcInvokeTransactionV3` or `RpcDeclareTransactionV3` with a non-empty `account_deployment_data` array and submit it via the public `add_transaction` RPC endpoint. The gateway accepts it, the mempool queues it, the batcher includes it in a block, and the blockifier executes it without error. The block is then unproducible: the OS aborts on the assertion, and the block cannot be settled on L1. This is a denial-of-service vector against block proving.

### Likelihood Explanation

**High.** The `account_deployment_data` field is a first-class field in the public Starknet V3 transaction schema (documented in the OpenRPC spec). Any wallet or script that constructs a raw V3 transaction can populate it. The asymmetry with `paymaster_data` (which is validated) makes this easy to miss and easy to trigger.

### Recommendation

Add a `validate_empty_account_deployment_data` function in `crates/apollo_gateway/src/stateless_transaction_validator.rs`, mirroring the existing `validate_empty_paymaster_data`:

```rust
fn validate_empty_account_deployment_data(
    tx: &RpcTransaction,
) -> StatelessTransactionValidatorResult<()> {
    let account_deployment_data = match tx {
        RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => &tx.account_deployment_data,
        RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => &tx.account_deployment_data,
        RpcTransaction::DeployAccount(_) => return Ok(()), // field absent
    };
    if account_deployment_data.is_empty() {
        Ok(())
    } else {
        Err(StatelessTransactionValidatorError::NonEmptyField {
            field_name: "account_deployment_data".to_string(),
        })
    }
}
```

Call this from `validate` alongside `validate_empty_paymaster_data`. This mirrors the OS invariant at the admission boundary, preventing the divergence before it reaches the prover.

### Proof of Concept

1. Construct an `RpcInvokeTransactionV3` with `account_deployment_data: vec![Felt::ONE]` and valid signature/nonce/resource_bounds.
2. Submit via `starknet_addInvokeTransaction` to the gateway.
3. Observe: gateway returns success (no `NonEmptyField` error for `account_deployment_data`).
4. The transaction enters the mempool and is included in the next block.
5. The blockifier executes it and records a successful receipt; the block is finalized by consensus.
6. The OS prover attempts `compute_invoke_transaction_hash` → hits `assert account_deployment_data_size = 0` → OS execution aborts → block proof fails.
