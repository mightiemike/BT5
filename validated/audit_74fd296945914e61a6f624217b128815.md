Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of Actual Swapper, Enabling Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which equals the pool's own `msg.sender` — the immediate caller. When swaps are routed through `MetricOmmSimpleRouter`, the router becomes `msg.sender` at the pool, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actualUser]`. Any non-allowlisted address can bypass the per-user gate by calling the public router once the pool admin allowlists the router to support legitimate router-mediated swaps.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards that value unchanged to every configured extension via `_callExtensionsInOrder`. `SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool received as its own `msg.sender`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the `msg.sender` at the pool level: [3](#0-2) 

The same pattern holds for `exactInput` (all hops): [4](#0-3) 

And for `exactOutputSingle`: [5](#0-4) 

The extension therefore evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actualUser]`. This creates an irreconcilable dilemma: if the router is not allowlisted, allowlisted users cannot use the router at all; if the router is allowlisted, every user — allowlisted or not — can swap by routing through the public router, completely defeating the per-user gate. There are no additional guards in the extension or pool that authenticate the true originating user.

## Impact Explanation
Any non-allowlisted address can bypass a `SwapAllowlistExtension`-protected pool by calling `MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, or `exactOutputSingle`. The router is a public, permissionless contract. Once the pool admin allowlists the router to enable router-mediated swaps for legitimate users, the allowlist provides zero per-user protection. This breaks the core access-control invariant of pools explicitly configured to restrict swap access, and allows unauthorized parties to execute swaps in permissioned pools — potentially draining liquidity or extracting value from pools designed for restricted participants. This constitutes an admin-boundary break where an unprivileged path bypasses a pool admin's configured access control.

## Likelihood Explanation
The router is the primary user-facing entry point. A pool admin who deploys a `SwapAllowlistExtension` and wants allowlisted users to use the standard router will naturally allowlist the router address — this is the only way to permit router-mediated swaps for legitimate users. The bypass is then immediately reachable by any address with no special privileges, no flash loans, and no complex setup: a single `exactInputSingle` call suffices. The precondition (router allowlisted) is the expected operational state for any pool using both the extension and the router.

## Recommendation
Pass the economically relevant actor — the end user — rather than the immediate `msg.sender` through the extension chain. Two complementary approaches:

1. **Router-side:** Have the router encode the originating `msg.sender` in `extensionData` (e.g., as a standardized prefix), and have `SwapAllowlistExtension.beforeSwap` read and authenticate that field when `msg.sender` (the pool's caller) is a known router.
2. **Extension-side:** Define a convention where the pool or router encodes the true initiator in `extensionData`, and `SwapAllowlistExtension` reads that field (with appropriate authentication, e.g., verifying the caller is a factory-registered router) instead of the raw `sender` argument.

Until fixed, pool admins must be warned that `SwapAllowlistExtension` provides no per-user protection for router-mediated swaps.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice]  = true   // alice is the only intended swapper
  allowedSwapper[pool][router] = true   // admin adds router so alice can use it

Attack:
  bob (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
        pool: pool,
        recipient: bob,
        ...
    })

  Router executes (MetricOmmSimpleRouter.sol L72-80):
    pool.swap(bob, ...)   // msg.sender at pool = router

  Pool calls _beforeSwap(router, bob, ...) (MetricOmmPool.sol L230-240)

  Extension checks (SwapAllowlistExtension.sol L37):
    allowedSwapper[pool][router] == true  → passes

  Result: bob's swap succeeds despite not being allowlisted.
```
