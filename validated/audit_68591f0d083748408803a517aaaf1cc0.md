### Title
`SwapAllowlistExtension.beforeSwap` Gates the Router Address Instead of the End User, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `swap` call. When users route through `MetricOmmSimpleRouter`, that `sender` is the router contract address, not the originating user. If the pool admin allowlists the router to support router-mediated swaps for permitted users, every user — including non-allowlisted ones — can bypass the individual allowlist by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every configured extension:

```solidity
_beforeSwap(
    msg.sender,   // ← router address when called via MetricOmmSimpleRouter
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that argument against the per-pool allowlist:

```solidity
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool; `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()`, `sender = address(router)`.

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

**Forced configuration trap.** A pool admin who wants allowlisted users to be able to trade through the router must call `setAllowedToSwap(pool, router, true)`. The moment they do, the check `allowedSwapper[pool][router]` returns `true` for every caller — allowlisted or not — because the router is a single shared contract. There is no code path that lets the extension distinguish which end user initiated the router call.

The pool admin cannot simultaneously:
- Allow allowlisted users to trade via the router, and
- Block non-allowlisted users from trading via the router.

---

### Impact Explanation

A curated pool's swap allowlist is completely ineffective for router-mediated swaps once the router is allowlisted. Any user can call `router.exactInputSingle(pool, ...)` and the extension passes because it sees the allowlisted router address, not the non-allowlisted end user.

Consequences:
- **LP principal at risk.** The allowlist may have been configured to exclude toxic-flow sources, front-runners, or non-KYC'd addresses. Bypassing it exposes LP positions to exactly the trading patterns the guard was meant to block.
- **Broken core pool functionality.** The allowlist is a first-class pool security feature; its silent failure on the primary public swap path (the router) renders it useless for any pool that supports router access.
- **No on-chain signal of the bypass.** The `NotAllowedToSwap` error is never raised; the swap succeeds normally, so LPs have no indication the guard was circumvented.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who wants allowlisted users to be able to use the router (the expected production flow) must allowlist the router address. This is a routine, non-malicious configuration step. Once done, the bypass is available to every address on-chain with no further privilege required — a single `exactInputSingle` call suffices.

---

### Recommendation

The extension must check the originating end user, not the intermediate router. Two viable approaches:

1. **Caller forwarding via `extensionData`.** The router encodes `msg.sender` into `extensionData` before calling `pool.swap`. The extension decodes and checks that address. This requires a trusted encoding convention between the router and the extension.

2. **Transient-storage originator.** The router writes the originating caller into a transient storage slot before calling the pool. The extension reads that slot. This is already consistent with the protocol's EIP-1153 usage for callback context.

Either approach must be paired with a check that the `extensionData` or transient slot was actually set by a trusted router (e.g., by verifying `sender` is a factory-registered router), otherwise an attacker could forge the originator field.

---

### Proof of Concept

```
Setup:
  pool admin deploys pool with SwapAllowlistExtension as beforeSwap hook
  pool admin calls setAllowedToSwap(pool, userA, true)   // allowlist userA
  pool admin calls setAllowedToSwap(pool, router, true)  // enable router-mediated swaps

Attack (userB, not allowlisted):
  userB calls router.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...) with msg.sender = router
    → pool calls extension.beforeSwap(sender=router, ...)
    → extension checks allowedSwapper[pool][router] → true
    → swap executes; userB receives output tokens

Result:
  userB traded on a curated pool despite not being individually allowlisted.
  The NotAllowedToSwap guard was never triggered.
```
