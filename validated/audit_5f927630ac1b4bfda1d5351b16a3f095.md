### Title
`paid_fee_on_l1` Stripped During `InternalConsensusTransaction → ConsensusTransaction` Conversion; Validators Always Bind Hardcoded `Fee(1)` — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

The `paid_fee_on_l1` field of an L1Handler transaction is silently dropped when the proposer converts `InternalConsensusTransaction::L1Handler` to `ConsensusTransaction::L1Handler` for P2P broadcast. When a validator receives the message and converts back to `InternalConsensusTransaction`, it unconditionally substitutes a hardcoded `Fee(1)`. This is the direct sequencer analog of the LP-token/lock-data split: the "ownership" of the transaction (the raw `L1HandlerTransaction` body) crosses the consensus wire, but the associated state (`paid_fee_on_l1`) stays behind and is replaced with a synthetic value on the other side.

### Finding Description

**Step 1 — stripping on the proposer side.**

`convert_internal_consensus_tx_to_consensus_tx` discards `paid_fee_on_l1` by projecting only `tx.tx` (the raw `transaction::L1HandlerTransaction`, which carries no fee field):

```rust
InternalConsensusTransaction::L1Handler(tx) => {
    Ok(ConsensusTransaction::L1Handler(tx.tx))   // tx.paid_fee_on_l1 silently dropped
}
```

`executable_transaction::L1HandlerTransaction` has three fields — `tx`, `tx_hash`, and `paid_fee_on_l1` — but only `tx` survives the conversion.
