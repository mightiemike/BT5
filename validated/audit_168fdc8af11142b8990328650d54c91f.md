### Title
ML-DSA-65 Gas-Key Nonce Exec-Fee Uses Wire Length Instead of Trie-ID Length in Host Functions — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

When a contract calls `promise_batch_action_add_gas_key_with_full_access` or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key, the exec gas fee for writing gas-key nonce trie entries is computed using `public_key_len as usize` — the raw borsh wire length (1953 bytes) — instead of the actual on-trie identifier length (`trie_id_len()` = 33 bytes). The runtime path that processes the resulting `AddKey` action correctly uses `public_key.trie_id_len()`. This creates an exact, unprivileged-user-observable divergence of 1920 bytes per nonce between the two code paths, overcharging contracts that add ML-DSA-65 gas keys via host functions by a factor of ~59× on the nonce-key component of the exec fee.

---

### Finding Description

**The invariant.** `gas_key_add_key_exec_fee` takes a `public_key_len` parameter that is supposed to represent the on-trie key length for each gas-key nonce entry. The trie stores ML-DSA-65 access keys as a SHA3-256 hash (`[tag=3] || 32-byte digest` = 33 bytes), not as the 1953-byte borsh-encoded full pubkey. The design document and `PublicKey::trie_id_len()` both encode this invariant explicitly.

**The runtime path (correct).** In `permission_exec_fees` (`runtime/runtime/src/config.rs`), the exec fee for an `AddKey` action with a gas-key permission is computed as:

```rust
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // 33 for ML-DSA-65
    gas_key_info.num_nonces,
);
```

**The host-function path (wrong).** In both `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`, the exec fee is computed as:

```rust
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // 1953 for ML-DSA-65 (borsh wire length)
    num_nonces,
);
```

`public_key_len` is the raw byte count of the guest-memory buffer, which for ML-DSA-65 equals the borsh-encoded full pubkey length (1 tag byte + 1952 key bytes = 1953). The decoded key is available immediately after (`public_key.decode()?`) but its `trie_id_len()` is never consulted.

**The divergent value.** Inside `gas_key_add_key_exec_fee`:

```rust
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len) + size_of::<NonceIndex>();
```

For ML-DSA-65, `public_key_len = 1953` yields a `nonce_key_len` that is 1920 bytes larger than the actual trie key. The per-nonce exec fee is multiplied by `(nonce_key_len + NONCE_VALUE_LEN)`, so the overcharge scales linearly with `num_nonces`.

The same divergence exists in the wasmtime runner (`runtime/near-vm-runner/src/wasmtime_runner/logic.rs`) at the corresponding call sites.

---

### Impact Explanation

Any unprivileged contract that calls `promise_batch_action_add_gas_key_with_full_access` or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key is charged ~1920 × `num_nonces` × `gas_key_byte` exec gas more than the actual trie work warrants. With the shipped `gas_key_byte` exec fee and even a modest `num_nonces` (e.g., 10), the overcharge can exceed the practical gas budget for a single receipt, making ML-DSA-65 gas keys effectively impossible to add via contract host functions. The direct `AddKey` action path (not through a contract) is unaffected and charges the correct fee. This creates a permanent, protocol-level asymmetry: the same key type is priced differently depending on whether it is added by a contract or by a direct transaction, breaking the fee-model invariant introduced alongside `PostQuantumSignatures`.

---

### Likelihood Explanation

The `PostQuantumSignatures` protocol feature is stabilized at protocol version 85. After activation, any contract on mainnet or testnet can call these host functions with an ML-DSA-65 public key. No privileged role is required. The trigger is a normal contract call, fully under unprivileged-user control.

---

### Recommendation

In both `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call` (in both `logic.rs` and `wasmtime_runner/logic.rs`), decode the public key first and pass `trie_id_len()` to `gas_key_add_key_exec_fee`:

```rust
let public_key = self.get_public_key(public_key_ptr, public_key_len)?.decode()?;
// ...
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key.trie_id_len(),   // was: public_key_len as usize
    num_nonces,
);
// pass public_key directly to append_action_add_gas_key_with_full_access
```

This matches the runtime path in `permission_exec_fees` and the documented invariant in `docs/architecture/how/post_quantum_signatures.md` §5.

### Proof of Concept

**Exact divergent bytes:**

| Path | `public_key_len` passed to `gas_key_add_key_exec_fee` | Resulting `nonce_key_len` (10-char account) |
|---|---|---|
| Runtime (`permission_exec_fees`) | `33` (`trie_id_len()`) | `col + 10 + sep + 33 + 2 = ~47` |
| Host fn (`promise_batch_action_add_gas_key_with_full_access`) | `1953` (wire length) | `col + 10 + sep + 1953 + 2 = ~1967` |
| **Overcharge per nonce** | — | **~1920 bytes × `gas_key_byte` exec fee** |

**Reachability:** Deploy a contract that calls `promise_batch_action_add_gas_key_with_full_access` with an ML-DSA-65 key and `num_nonces = 10` after `PostQuantumSignatures` activates. Observe that the gas burnt by the receipt exceeds the gas burnt by an equivalent direct `AddKey` transaction by approximately `1920 × 10 × gas_key_byte_exec_fee` gas — a value that can be verified deterministically from the execution outcome. pub fn gas_key_add_key_exec_fee(
```rust
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
}
```

```markdown
### 5. Storage usage and fee plumbing

The storage-stake calculation
(`runtime/runtime/src/access_keys.rs::access_key_storage_usage`) and the
gas-key fee helpers (`gas_key_*_fee` in `runtime/runtime/src/config.rs`) use
`PublicKey::trie_id_len()` rather than `PublicKey::len()`:

- `len()` reports the borsh-encoded length (33 / 65 / 1953 across the
  three `PublicKey` variants).
- `trie_id_len()` reports the on-trie length (33 / 65 / **33**).

The two diverge only for `PublicKey::MLDSA65`. Every storage-stake and
trie-byte-priced fee path was updated to call `trie_id_len()`.
```
