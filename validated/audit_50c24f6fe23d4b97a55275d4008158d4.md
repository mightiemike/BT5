Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing non-allowlisted depositors to bypass the curated-pool deposit guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards its first parameter (`sender`, the actual caller who pays tokens) and only validates `owner` (the position recipient) against the per-pool allowlist. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts any non-zero `owner` from the caller while setting the payer to `msg.sender`, a non-allowlisted address can deposit its own tokens into a curated pool by nominating any allowlisted address as `owner`. The pool admin's deposit curation policy is completely voided.

## Finding Description

**Root cause — discarded `sender` parameter:**

`DepositAllowlistExtension.beforeAddLiquidity` names the first argument `address` (anonymous), making it unreachable:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
```

The guard only evaluates `owner`:

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
``` [1](#0-0) 

**Call chain that separates payer from owner:**

`MetricOmmPool.addLiquidity` passes `msg.sender` (the adder contract) as `sender` and the caller-supplied `owner` as `owner`: [2](#0-1) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts any non-zero `owner` from the caller and stores `msg.sender` (the real payer) in transient context: [3](#0-2) 

`_validateOwner` only rejects `address(0)`, imposing no allowlist check: [4](#0-3) 

`_addLiquidity` then calls `pool.addLiquidity(positionOwner, ...)`, making `msg.sender` inside the pool equal to the adder contract, while `owner` is the attacker-chosen allowlisted address: [5](#0-4) 

The extension evaluates `allowedDepositor[pool][alice]` (the allowlisted address chosen by the attacker) and passes without ever seeing the real payer's address.

**Why existing guards fail:**

- `_validateOwner` only rejects `address(0)` — no allowlist check.
- The extension's `msg.sender` is the pool (correct), but the pool's `msg.sender` (the adder) is passed as `sender` to the extension and then silently discarded.
- There is no check anywhere in the adder or pool that the real payer (`msg.sender` of the adder call) is allowlisted.

## Impact Explanation

A non-allowlisted address (Bob) can deposit its own tokens into any curated pool that has `DepositAllowlistExtension` active with the `beforeAddLiquidity` hook enabled. Bob's tokens are pulled from Bob (he is the payer stored in transient `T_SLOT_PAY_PAYER`), the LP position is minted to Alice (any allowlisted address Bob nominates), and the extension never evaluates Bob's address. This constitutes an admin-boundary break: the pool admin's deposit curation policy — the only on-chain mechanism restricting who may provide liquidity — is silently voided by an unprivileged caller. Any non-allowlisted actor can inject capital into the pool, alter bin balances, and affect the pool's price cursor and LP accounting, with direct fund-level consequences for existing LPs and the pool's solvency invariants.

## Likelihood Explanation

`MetricOmmPoolLiquidityAdder` is the standard periphery entry point for liquidity provision and is publicly callable by anyone. No privileged role, special token, or unusual setup is required. The attacker only needs to know one allowlisted address (readable from `allowedDepositor` events or the public mapping) and hold tokens to deposit. The bypass is reachable on every curated pool using `DepositAllowlistExtension` with the `beforeAddLiquidity` hook enabled.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual caller/payer) rather than — or in addition to — `owner`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender]
        && !allowedDepositor[msg.sender][sender]
        && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Because `sender` will be the adder contract when users route through periphery, pool admins must also decide whether to allowlist the adder contract (open to all adder users) or require direct pool calls for curated pools. The adder itself would need to enforce a per-user allowlist check if it is to be allowlisted.

## Proof of Concept

```
Setup:
  pool  = MetricOmmPool with DepositAllowlistExtension (beforeAddLiquidity hook enabled)
  alice = allowlisted depositor  (allowedDepositor[pool][alice] = true)
  bob   = NOT allowlisted

Attack:
  1. bob calls:
       adder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, extensionData)
     → _validateOwner(alice) passes (alice != address(0))
     → _addLiquidity called with payer = bob, positionOwner = alice
     → _setPayContext stores bob as payer in transient T_SLOT_PAY_PAYER

  2. adder calls:
       pool.addLiquidity(alice, salt, deltas, abi.encode(KIND_PAY), extensionData)
     → pool sees: msg.sender = adder, owner = alice

  3. pool calls:
       extension.beforeAddLiquidity(adder, alice, salt, deltas, extensionData)
     → extension evaluates: allowedDepositor[pool][alice] == true → NO REVERT

  4. pool mints LP shares to alice; callback pulls tokens from bob.

Result:
  - bob (non-allowlisted) successfully deposited tokens into the curated pool.
  - DepositAllowlistExtension never evaluated bob's address.
  - alice receives LP shares she did not initiate.
  - Pool admin's deposit curation policy is bypassed.
```
