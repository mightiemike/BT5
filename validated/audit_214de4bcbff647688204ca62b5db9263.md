### Title
Missing Minimum Shares Check in ERC4626 Vault Deposit Allows Slippage Loss of Collateral - (File: `core/contracts/ContractOwner.sol`)

### Summary
`ContractOwner.wrapVaultAsset` calls `IERC4626Base.deposit` without enforcing a minimum shares output. The function computes `previewDeposit` only as a non-zero guard, then discards the result. Any change in the vault's exchange rate between the preview and the actual deposit — via front-running or a concurrent donation — causes `directDepositV1` to receive fewer vault shares than expected, reducing the collateral ultimately credited to the target subaccount.

### Finding Description
In `ContractOwner.wrapVaultAsset`, the code path is:

```solidity
if (IERC4626Base(tokenAddr).previewDeposit(assetBalance) != 0) {
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(assetTokenAddr));
    IERC20Base assetToken = IERC20Base(assetTokenAddr);
    assetToken.approve(tokenAddr, 0);
    assetToken.approve(tokenAddr, assetBalance);
    IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
}
``` [1](#0-0) 

`previewDeposit(assetBalance)` is evaluated only to confirm it is non-zero; its return value is never stored or used as a minimum. The subsequent `deposit(assetBalance, directDepositV1)` call accepts whatever shares the vault chooses to mint. Standard ERC4626 `deposit` returns the actual shares minted but imposes no floor.

`wrapVaultAsset` carries no access modifier — it is `external` with no `onlyOwner` or `onlyDeployer` guard — so any caller can invoke it at any time. [2](#0-1) 

### Impact Explanation
The `directDepositV1` contract accumulates vault shares that are later deposited into the Nado protocol as collateral via `creditDeposit`. If the vault's share price is inflated at deposit time, `directDepositV1` receives fewer shares for the same asset amount. When `creditDeposit` is subsequently called, the subaccount is credited with a smaller collateral balance than the underlying asset value warrants. This is a direct accounting corruption: the subaccount's collateral is permanently understated relative to the assets that were consumed. [3](#0-2) 

### Likelihood Explanation
The function is permissionlessly callable by any address. An attacker who observes a pending `wrapVaultAsset` transaction in the mempool can front-run it by donating underlying assets directly to the ERC4626 vault, inflating the share price. The victim's deposit then mints fewer shares. On chains with a public mempool (Ethereum mainnet, Arbitrum with public sequencer fallback) this is straightforwardly executable. No privileged access is required.

### Recommendation
Capture the `previewDeposit` result before withdrawing assets from `directDepositV1`, then pass it as a minimum-shares floor to the deposit call (or assert the returned shares meet the floor):

```solidity
uint256 minShares = IERC4626Base(tokenAddr).previewDeposit(assetBalance);
if (minShares != 0) {
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(assetTokenAddr));
    IERC20Base assetToken = IERC20Base(assetTokenAddr);
    assetToken.approve(tokenAddr, 0);
    assetToken.approve(tokenAddr, assetBalance);
    uint256 sharesReceived = IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
    require(sharesReceived >= minShares, "insufficient shares");
}
```

Additionally, consider restricting `wrapVaultAsset` to `onlyOwner` to eliminate the permissionless front-running surface.

### Proof of Concept
1. A subaccount's `directDepositV1` holds 1000 USDC (the underlying asset of an ERC4626 vault token used as Nado collateral).
2. `previewDeposit(1000e6)` returns 1000 shares at the current 1:1 rate.
3. Attacker front-runs the `wrapVaultAsset` call by donating 1000 USDC directly to the vault contract, doubling the share price to 2 USDC/share.
4. `wrapVaultAsset` executes: `previewDeposit` check passes (non-zero), `deposit(1000e6, directDepositV1)` mints only 500 shares instead of 1000.
5. `creditDeposit` is called; the subaccount is credited with 500 vault-token units of collateral instead of 1000.
6. The subaccount's health and borrowing capacity are permanently reduced by ~50% relative to the assets consumed, with no recourse. [4](#0-3)
