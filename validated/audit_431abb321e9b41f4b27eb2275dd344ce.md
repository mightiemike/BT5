### Title
Linked Signer Can Overwrite Its Own Slot to Grant Arbitrary Third-Party Signing Access — (`File: core/contracts/EndpointTx.sol`)

---

### Summary
The `LinkSigner` fast-path handler in `EndpointTx.sol` accepts a signature from the **current linked signer** (`allowLinkedSigner = true`) to overwrite `linkedSigners[sender]` with any arbitrary address. Because `TransferQuote` also accepts linked-signer signatures, a malicious linked signer can silently replace itself with an attacker-controlled address and then drain the victim subaccount's quote balance via `TransferQuote`.

---

### Finding Description
In `processTransactionImpl`, the `LinkSigner` branch validates the transaction with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:], (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← linked signer accepted
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
```

`validateSignedTx` delegates to `Verifier.validateSignature`, which accepts either the subaccount owner **or** the current linked signer:

```solidity
// Verifier.sol lines 297–303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
    ((recovered == address(uint160(bytes20(sender)))) ||
     (recovered == linkedSigner)),          // ← linked signer can pass
    ERR_INVALID_SIGNATURE
);
```

There is no restriction preventing the linked signer from setting `signedTx.tx.signer` to an **arbitrary third-party address**. Once the slot is overwritten, the new address inherits full linked-signer privileges.

`TransferQuote` also passes `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 599–604
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true                          // ← linked signer accepted
);
```

This means the newly installed address can immediately sign `TransferQuote` transactions to move the victim's quote balance to any registered subaccount.

---

### Impact Explanation
A malicious or compromised linked signer can:

1. Submit a `LinkSigner` transaction signed by itself, setting `signer` to an attacker-controlled address.
2. The attacker-controlled address becomes the new linked signer for the victim subaccount.
3. The attacker signs `TransferQuote` transactions to drain the victim's quote balance to an attacker-owned subaccount.
4. The attacker withdraws the transferred balance.

The corrupted state is `linkedSigners[victimSubaccount]` — a direct mutation of another account's signer configuration by an unprivileged party. The downstream asset delta is the full quote balance of the victim subaccount.

**Impact: 3 / 5** — Direct quote-balance theft is possible, but the victim's subaccount owner can sign directly (without a linked signer) to restore the correct signer and halt further damage. Non-quote product balances are not directly reachable via `TransferQuote`.

---

### Likelihood Explanation
Linked signers are explicitly listed as in-scope callers. They are commonly automated bots, third-party trading services, or API keys. A compromised or malicious linked signer has a realistic path to exploit this without any additional privilege escalation.

**Likelihood: 3 / 5** — Requires a malicious or compromised linked signer, which is a realistic threat model for automated trading infrastructure.

---

### Recommendation
Restrict `LinkSigner` transactions so that only the **subaccount owner** (the address embedded in `sender`) can authorize a signer change. Change `allowLinkedSigner` to `false` for the `LinkSigner` branch:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the principle of least privilege: a linked signer is authorized to trade on behalf of an account, not to modify the account's trust configuration.

---

### Proof of Concept

**Setup**: Alice owns subaccount `A`. She links Bob (`linkedSigners[A] = Bob`). Bob is a malicious linked signer who wants to install Carol as the new linked signer.

**Step 1 — Bob crafts a `LinkSigner` transaction:**
```
sender = A
signer = bytes32(uint256(uint160(Carol)))
nonce  = nonces[address(uint160(bytes20(A)))]
```

**Step 2 — Bob signs the EIP-712 digest** (Bob's key is accepted because `allowLinkedSigner = true` and `linkedSigners[A] == Bob`).

**Step 3 — Bob submits to the sequencer.** The sequencer calls `processTransactionImpl`. `validateSignedTx` passes. `linkedSigners[A]` is overwritten with Carol's address.

**Step 4 — Carol signs a `TransferQuote` transaction:**
```
sender    = A
recipient = Carol's subaccount
amount    = Alice's full quote balance
nonce     = current nonce for A
```

Carol's signature is accepted because `linkedSigners[A] == Carol` and `allowLinkedSigner = true`.

**Step 5 — Carol withdraws** the transferred balance from her own subaccount.

**Corrupted invariant**: `linkedSigners[A]` was mutated by Bob (an unprivileged linked signer) without Alice's consent, analogous to `resetAllocationsByUser` deleting another account's state without that account's authorization. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
