### Title
Phantom-Index Merkle Proof Collision: `compute_root_from_merkle_proof` Accepts Out-of-Bounds `tx_index` for Odd-Length Trees — (`merkle-tools/src/lib.rs`)

### Summary

In blocks with an odd number of transactions, `compute_root_from_merkle_proof` accepts `tx_index = N` (one past the last real leaf) with the same `tx_id` and proof as the legitimate `tx_index = N-1`, producing the identical Merkle root. The contract has no guard that bounds `tx_index` against the actual transaction count of the block, so `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` both return `true` for this phantom index.

### Finding Description

`merkle_proof_calculator` handles odd-length levels by appending a duplicate of the last element before pairing: [1](#0-0) 

For a 5-leaf tree at position 4, this produces `proof[0] = L4` (the duplicate). In `compute_root_from_merkle_proof`, the first step for `tx_index=4` (even) computes `H(current_hash=L4, proof[0]=L4)`, and for `tx_index=5` (odd) computes `H(proof[0]=L4, current_hash=L4)`. Because both inputs are identical, both orderings yield the same digest `H44`. All subsequent steps are then identical, so the final root matches. [2](#0-1) 

The contract passes `tx_index` directly to this function with no upper-bound check against the block's transaction count: [3](#0-2) 

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after the coinbase check, so it inherits the same flaw: [4](#0-3) 

### Impact Explanation

An attacker can submit `(tx_id=L_{N-1}, tx_index=N, block=B, proof=proof_for_{N-1})` to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` and receive `true`, even though index `N` does not exist in the block. The contract's documented invariant — "True if `tx_id` is **at the claimed position**" — is broken. [5](#0-4) 

Concretely, if a downstream NEAR contract uses `(tx_id, tx_index)` as a unique key to track processed BTC transactions (a common pattern to prevent replay), an attacker can replay the same transaction under the phantom index `N` after it has already been processed under index `N-1`, bypassing the deduplication guard and triggering double-processing (e.g., double minting of wrapped BTC).

### Likelihood Explanation

The condition requires only that the target Bitcoin block has an odd number of transactions, which is common (roughly half of all blocks). No privileged access, key compromise, or special chain state is needed. The attacker only needs to know the txid and the legitimate proof for the last transaction, both of which are publicly available from any Bitcoin node.

### Recommendation

Add an explicit upper-bound check on `tx_index` before calling `compute_root_from_merkle_proof`. Since the Bitcoin block header does not encode transaction count directly, the proof length encodes the tree depth: a valid index must satisfy `tx_index < 2^(proof.len())`. A tighter bound requires the caller to supply the transaction count and validate `tx_index < tx_count`. Additionally, `compute_root_from_merkle_proof` itself should reject any `transaction_position >= 2^(merkle_proof.len())`.

pub fn merkle_proof_calculator(tx_hashes: Vec<H256>, transaction_position: usize) -> Vec<H256> {
    let mut transaction_position = transaction_position;
    let mut merkle_proof = Vec::new();
    let mut current_hashes = tx_hashes;

    while current_hashes.len() > 1 {
        if current_hashes.len() % 2 == 1 {
            current_hashes.push(current_hashes[current_hashes.len() - 1].clone());
        }

        if transaction_position % 2 == 1 {
            merkle_proof.push(current_hashes[transaction_position - 1].clone());
        } else {
            merkle_proof.push(current_hashes[transaction_position + 1].clone());
        }

        let mut new_hashes = Vec::new();

        for i in (0..current_hashes.len() - 1).step_by(2) {
            new_hashes.push(compute_hash(&current_hashes[i], &current_hashes[i + 1]));
        }

        current_hashes = new_hashes;
        transaction_position /= 2;
    }

    merkle_proof
}
