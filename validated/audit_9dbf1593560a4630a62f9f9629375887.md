### Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension` is intended to restrict which addresses may add liquidity to a pool. Its `beforeAddLiquidity` hook receives both `sender` (the pool's `msg.sender`, i.e., the actual depositing caller) and `owner` (the position owner). The hook silently ignores `sender` and checks `owner` against the allowlist. Because `owner` is a caller-controlled parameter, any unprivileged address can bypass the guard by supplying an allowlisted address as `owner` while remaining the actual depositor.

### Finding Description

`MetricOmmPool.addLiquidity` fires the `_beforeAddLiquidity` hook with `msg.sender` as `sender` and the caller-supplied `owner` as `owner`:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` forwards both values faithfully to every configured extension:

```solidity
// ExtensionCalling.sol lines 95-98
_callExtensionsInOrder(
  BEFORE_ADD_LIQUIDITY_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

`DepositAllowlistExtension.beforeAddLiquidity` then ignores `sender` entirely and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`owner` is a free parameter supplied by the caller. `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts an arbitrary `owner` address (validated only to be non-zero) and passes it straight through to the pool:

```solidity
// MetricOmmPoolLiquidityAdder.sol lines 56-68
function addLiquidityExactShares(
    address pool, address owner, uint80 salt, LiquidityDelta calldata deltas,
    uint256 maxAmountToken0, uint256 maxAmountToken1, bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _validateOwner(owner);   // only checks owner != address(0)
    ...
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
```

The contrast with `SwapAllowlistExtension` makes the mismatch explicit: the swap guard correctly checks `sender` (the actual caller), while the deposit guard checks `owner` (a caller-controlled field):

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) { ... }
```

### Impact Explanation

A pool configured with `DepositAllowlistExtension` intends to restrict liquidity provision to a curated set of addresses. Because the guard checks `owner` rather than `sender`, any address can pass the check by nominating any allowlisted address as `owner`. The allowlist invariant — "only permitted depositors may add liquidity" — is completely broken. An attacker can:

1. Identify any address `A` that is allowlisted for pool `P`.
2. Call `addLiquidityExactShares(P, A, ...)` through `MetricOmmPoolLiquidityAdder`.
3. The hook evaluates `allowedDepositor[P][A]` → `true` and does not revert.
4. The attacker's tokens are deposited; the LP position is credited to `A`.

This constitutes a griefing vector (forced liquidity injection into `A`'s position without `A`'s consent) and a complete defeat of the pool admin's access-control intent. It also enables manipulation of pool liquidity composition in pools that are supposed to be restricted to specific LPs.

### Likelihood Explanation

The trigger requires only knowledge of one allowlisted address and a call through the publicly available `MetricOmmPoolLiquidityAdder`. No privileged role, special token, or malicious setup is needed. Any external actor can execute this against any pool that has `DepositAllowlistExtension` configured.

### Recommendation

Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Pool admins who use the `MetricOmmPoolLiquidityAdder` periphery should then allowlist the adder contract address (as a trusted intermediary), consistent with how the swap allowlist handles the router.

### Proof of Concept

1. Pool `P` is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` extension.
2. Pool admin calls `setAllowedToDeposit(P, alice, true)`. Only `alice` is supposed to deposit.
3. Attacker `bob` (not allowlisted) calls:
   ```solidity
   liquidityAdder.addLiquidityExactShares(P, alice, salt, deltas, max0, max1, "");
   ```
4. Pool calls `_beforeAddLiquidity(liquidityAdder, alice, ...)`.
5. Extension evaluates `allowedDepositor[P][alice]` → `true` → no revert.
6. `bob`'s tokens are transferred into pool `P`; the LP position is minted to `alice`.
7. The deposit allowlist has been bypassed; `bob` has added liquidity to a pool that was supposed to exclude him. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
