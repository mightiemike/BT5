### Title
Deprecated `verify_transaction_inclusion` Accepts Internal Merkle Node as Valid Transaction Proof — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function remains callable by any unprivileged NEAR account. It does not validate that the supplied `tx_id` is a leaf node (an actual transaction) rather than an internal Merkle tree node. An attacker can supply an internal Merkle tree node hash as `tx_id` with a structurally valid proof for that internal node, causing the function to return `true` for a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` (lines 288–323 in `contract/src/lib.rs`) computes the Merkle root from the caller-supplied `tx_id`, `tx_index`, and `merkle_proof`, then compares the result to the block's stored `merkle_root`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

The underlying `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` (lines 34–52) is a pure hash-chain computation. It accepts any 32-byte value as the starting hash and iterates over the proof elements, combining them left or right based on position parity. There is no check that the starting hash is a leaf node (an actual transaction) rather than an internal node of the Merkle tree.

A Bitcoin Merkle tree internal node at depth `d` is the double-SHA256 of two 32-byte child hashes. That internal node hash is a valid 32-byte value. If an attacker supplies it as `tx_id` at the correct `tx_index` for that level, the proof chain will correctly reconstruct the Merkle root, and the function returns `true`.

The function is annotated `#[deprecated(since = "0.5.0")]` but Rust's `#[deprecated]` is a compiler lint only — it does not remove the function from the contract's public ABI. The function is gated only by `#[pause]`, not by `#[trusted_relayer]`, so any unprivileged NEAR account can call it when the contract is not paused.

The v2 replacement (`verify_transaction_inclusion_v2`, lines 347–369) closes this gap by requiring a coinbase proof of the same length as the transaction proof:

```rust
require!(
    args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
    "Coinbase merkle proof and transaction merkle proof should have the same length"
);
```

Because an internal node at depth `d` requires a proof of length `tree_depth - d`, while the coinbase proof always has length `tree_depth`, the length equality check prevents the forgery in v2. The v1 function has no such guard.

---

### Impact Explanation

Any consumer contract (bridge, atomic swap, cross-chain lending protocol) that calls `verify_transaction_inclusion` and acts on a `true` result can be deceived into treating a non-existent Bitcoin transaction as confirmed. The attacker does not need to mine a block or control any privileged role — they only need public information (the block's transaction list) to compute the internal node hash and construct the proof. The corrupted value is the **proof result**: the function returns `true` for a `tx_id` that is not a real transaction, breaking the invariant that a `true` result certifies transaction inclusion.

---

### Likelihood Explanation

High. The function is reachable by any unprivileged NEAR caller. The attack requires only public Bitcoin blockchain data (transaction IDs for a confirmed block). The 64-byte internal-node forgery technique is well-documented. Any consumer contract that has not migrated to `verify_transaction_inclusion_v2` is exposed.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract's public ABI entirely, or gate it with an access-control role so it cannot be called by unprivileged accounts. Do not rely on the `#[deprecated]` annotation as a security boundary. Consumer contracts must migrate to `verify_transaction_inclusion_v2`.

---

### Proof of Concept

Assume a confirmed mainchain block with four transactions: `tx0, tx1, tx2, tx3`.

**Merkle tree structure:**
```
Level 0 (leaves): tx0,  tx1,  tx2,  tx3
Level 1:          H01 = SHA256d(tx0||tx1),  H23 = SHA256d(tx2||tx3)
Level 2 (root):   R   = SHA256d(H01||H23)
```

**Attack:**
1. Compute `H01 = double_sha256(tx0 || tx1)` — this is an internal node, not a transaction.
2. Obtain `H23` from the block's public transaction list.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = H01`
   - `tx_block_blockhash = <block hash>`
   - `tx_index = 0`
   - `merkle_proof = [H23]`
   - `confirmations = 1`
4. `compute_root_from_merkle_proof(H01, 0, [H23])` computes `SHA256d(H01 || H23) = R`, which equals `header.block_header.merkle_root`.
5. The function returns `true`. `H01` is not a real transaction — it is an internal Merkle node — yet the contract certifies its inclusion.

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
