Looking at the `burnNlp` function and comparing it to other similar operations in `Clearinghouse.sol`.

### Title
`burnNlp` Uses `MAINTENANCE` Health Check Instead of `INITIAL`, Allowing Value Extraction Below Initial Health — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`burnNlp` is a value-extraction operation (NLP → quote) that enforces only a `MAINTENANCE` health check post-burn. Every other value-extraction path in the same contract enforces `INITIAL`. This asymmetry allows a subaccount that is already between initial and maintenance health (a state reachable via normal price movement) to burn NLP and extract quote, further eroding the safety buffer without triggering a revert.

---

### Finding Description

In `Clearinghouse.sol`, the post-burn health check at line 527 reads:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) 

Compare this to every analogous value-extraction function in the same file:

| Function | Health check used |
|---|---|
| `withdrawCollateral` (line 419) | `INITIAL` |
| `transferQuote` (line 249) | `INITIAL` (via `_isAboveInitial`) |
| `mintNlp` (line 480) | `INITIAL` |
| `nlpProfitShare` (line 563) | `INITIAL` |
| `forceRebalanceNlpPool` (line 543) | `INITIAL` | [2](#0-1) [3](#0-2) [4](#0-3) 

The protocol enforces `longWeightInitial <= longWeightMaintenance` for every product: [5](#0-4) 

This means `INITIAL health <= MAINTENANCE health` for any subaccount holding long positions. A subaccount can therefore be in the state `INITIAL health < 0 <= MAINTENANCE health` — the "twilight zone" — which is reachable through ordinary adverse price movement without any protocol violation.

The code comment at lines 523–525 explicitly acknowledges the health-decrease risk but uses the wrong health type to guard against it: [6](#0-5) 

---

### Impact Explanation

Once a subaccount enters the twilight zone (INITIAL < 0, MAINTENANCE >= 0) via price movement:

1. `withdrawCollateral` reverts — INITIAL check blocks it.
2. `burnNlp` **does not revert** — MAINTENANCE check passes.
3. The attacker burns NLP, receiving quote. MAINTENANCE health decreases but stays ≥ 0.
4. The subaccount now has less NLP collateral and a lower MAINTENANCE health buffer.
5. A further adverse price move pushes MAINTENANCE health below 0.
6. Liquidation is triggered, but the protocol has already allowed the attacker to extract value that `withdrawCollateral` would have blocked, leaving less margin to cover the debt.
7. If the liquidation proceeds at a loss, the shortfall is absorbed by the insurance fund or socialised as bad debt.

The net effect is that `burnNlp` bypasses the margin requirement that all other withdrawal paths enforce, allowing value to be extracted from an already-undercollateralised subaccount.

---

### Likelihood Explanation

- The twilight zone (INITIAL < 0, MAINTENANCE >= 0) is a normal market state, not an exotic edge case. Any subaccount with leveraged positions will pass through it during volatility.
- `burnNlp` is a standard user-initiated transaction routed through the Endpoint (`onlyEndpoint`), requiring no special privileges.
- The attacker does not need to manipulate prices or compromise any key — they only need to hold NLP and wait for a price move.
- The burn fee (`max(ONE, quoteAmount / 1000)`) is small (0.1%), so large NLP burns can extract substantial quote while only marginally reducing MAINTENANCE health.

---

### Recommendation

Replace the `MAINTENANCE` health type with `INITIAL` in `burnNlp`, consistent with every other value-extraction function:

```solidity
// Before (line 527):
getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0

// After:
getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0
``` [7](#0-6) 

---

### Proof of Concept

**Setup:**
- NLP `longWeightInitialX18 = 0.90`, `longWeightMaintenanceX18 = 0.95`
- Subaccount holds 10,000 NLP (worth 10,000 quote at oracle price 1.0) and a leveraged perp position.
- Price movement causes: `INITIAL health = -50`, `MAINTENANCE health = +200`.

**Attack:**
1. Attacker submits `BurnNlp` for 1,000 NLP (worth 1,000 quote).
2. `burnFee = max(1, 1000/1000) = 1`. `quoteAmount = 999`.
3. Balances updated: NLP −1,000, quote +999.
4. Health delta for MAINTENANCE: `1000 * (1 − 0.95) − 1 = 49`. New MAINTENANCE health = 249. ✓ passes.
5. Health delta for INITIAL: `1000 * (1 − 0.90) − 1 = 99`. New INITIAL health = 49. ✓ (in this example INITIAL also improves — but the subaccount was already at −50 before the burn, meaning the burn was permitted despite the pre-existing INITIAL < 0 condition).
6. Attacker repeats until MAINTENANCE health approaches 0, extracting quote each time.
7. Any further price drop pushes MAINTENANCE < 0; liquidation proceeds with a depleted margin buffer.

The invariant broken: `burnNlp` must not permit execution when `INITIAL health < 0`, just as `withdrawCollateral` does not.
