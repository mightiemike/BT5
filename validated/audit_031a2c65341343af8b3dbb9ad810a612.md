### Title
Linked Signer Can Replace Itself With Any Address via `LinkSigner` — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction handler in `EndpointTx.sol` validates the transaction with `allowLinkedSigner = true`, meaning the currently-set linked signer for a subaccount can sign a new `LinkSigner` transaction to overwrite the linked signer mapping with any address it chooses. Only the subaccount owner should be permitted to change the linked signer, but the overly permissive signature check allows the delegated signer to re-delegate to a new address entirely.

---

### Finding Description

In `EndpointTx.sol`, the fast-mode (sequencer-submitted) handler for `TransactionType.LinkSigner` is:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

The fifth argument `true` is `allowLinkedSigner`. This flows into `validateSignature` in `Verifier.sol`:

```solidity
// Verifier.sol lines 297–303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),   // <-- linked signer accepted
    ERR_INVALID_SIGNATURE
);
``` [2](#0-1) 

Because the linked signer's signature is accepted, the linked signer can craft and submit a `SignedLinkSigner` transaction where `signedTx.tx.sender` is the victim subaccount and `signedTx.tx.signer` is any address the attacker controls. The result is that `linkedSigners[victim_subaccount]` is overwritten with the attacker's new address.

**Contrast with slow-mode**: The slow-mode `LinkSigner` handler correctly restricts this operation to the subaccount owner only, using `validateSender` which checks that `msg.sender` is the address embedded in the subaccount bytes32:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // owner-only check
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
``` [3](#0-2) 

The fast-mode path has no equivalent owner-only restriction.

---

### Impact Explanation

A linked signer that becomes malicious (or is compromised) can:

1. Sign a `LinkSigner` transaction naming any attacker-controlled address as the new signer.
2. `linkedSigners[victim_subaccount]` is overwritten with the attacker's address.
3. The new linked signer can now sign `WithdrawCollateral`, `TransferQuote`, and all other `allowLinkedSigner = true` transactions on behalf of the victim subaccount.
4. The victim loses all assets in the subaccount.

The corrupted state is `linkedSigners[subaccount]` in `EndpointStorage.sol`: [4](#0-3) 

This is a direct, concrete asset-loss path: the new linked signer can immediately drain collateral via `WithdrawCollateral`.

---

### Likelihood Explanation

This is contingent on the subaccount owner having previously set a linked signer that becomes malicious or is compromised — identical to the ODSafeManager finding's contingency. The owner must have trusted and linked a signer. Once that signer is compromised, the attacker can silently rotate the linked signer to a fresh address before the owner can revoke it, making recovery impossible without the owner submitting a competing transaction first.

---

### Recommendation

Pass `allowLinkedSigner = false` when validating `LinkSigner` transactions in the fast-mode path, so only the subaccount owner's key can authorize a linked signer change:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the slow-mode path's owner-only enforcement and closes the permission escalation path.

---

### Proof of Concept

1. Owner of subaccount `S` sets linked signer to address `A` via a legitimate `LinkSigner` transaction.
2. Address `A` (now malicious or compromised) constructs a new `SignedLinkSigner` where:
   - `tx.sender = S` (victim subaccount)
   - `tx.signer = bytes32(uint256(uint160(attackerAddress)))` (attacker's new address)
   - `signature` = signed by `A` (the current linked signer)
3. `A` submits this to the sequencer as a fast-mode transaction.
4. `validateSignedTx(..., true)` accepts `A`'s signature because `recovered == linkedSigner`.
5. `linkedSigners[S]` is overwritten with `attackerAddress`.
6. `attackerAddress` immediately signs a `WithdrawCollateral` transaction for all of `S`'s collateral.
7. Funds are drained. The original owner has no recourse unless they front-run step 3 with a revocation. [5](#0-4)
