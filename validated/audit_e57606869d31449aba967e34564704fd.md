The code confirms the claim. All three code paths are verified:

1. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.
2. `MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as `sender` to `_beforeSwap`.
3. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `sender` seen by the extension.

The dilemma is real and inescapable in the current design.

Audit Report

## Title
`SwapAllowlistExtension` checks the immediate caller (`sender`) rather than the originating user, allowing any unprivileged caller to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router becomes `sender`. If the pool admin allowlists the router (the only way to permit router-mediated swaps for legitimate users), every caller — including those explicitly excluded — can bypass the allowlist by routing through the router. No configuration exists that simultaneously enables router-mediated swaps and enforces per-user access control.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` verbatim as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-231
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
```

`SwapAllowlistExtension.beforeSwap` checks that value against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

This makes the router the `msg.sender` seen by the pool, and therefore the `sender` seen by the extension. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. If the router is allowlisted (required for any router-mediated swap to succeed), the check passes for every caller regardless of their individual allowlist status. The end user's address (`msg.sender` of the router call) is never evaluated.

## Impact Explanation
A pool admin who deploys a curated pool with `SwapAllowlistExtension` to restrict trading to specific counterparties cannot enforce that restriction when the router is involved. Any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` (or any other `exact*` entry point) targeting the curated pool and execute swaps that the allowlist was meant to block. The pool's core access-control invariant — "only allowlisted addresses may trade" — is broken. This constitutes broken core pool functionality (the configured extension fails to enforce its stated gate) and an admin-boundary break (an unprivileged path bypasses the pool admin's access control configuration). Pool admins deploying allowlist-gated pools for regulatory compliance, KYC gating, or counterparty restrictions have no on-chain remedy short of removing the router from the allowlist and thereby disabling router access for all users.

## Likelihood Explanation
`MetricOmmSimpleRouter` is a public, permissionless periphery contract. Any user can call it with any pool address. The bypass requires only that the pool admin has allowlisted the router — the natural and necessary operational choice to support router-mediated swaps for legitimate users. No privileged access, no special token, and no multi-step setup is needed. The attacker simply calls the router with the target pool address.

## Recommendation
**Short term:** Require callers to embed the original end-user address in `extensionData` and have the extension decode and check that address. The router should forward `msg.sender` in a standardised slot of `extensionData` so the extension can verify it against the allowlist instead of (or in addition to) `sender`.

**Long term:** Introduce a canonical "originator" field in the swap call path so extensions always have access to the economic actor regardless of how many intermediary contracts are in the call stack. Alternatively, document explicitly that `SwapAllowlistExtension` cannot gate router-mediated swaps and require pool admins to deploy a custom extension that reads originator data from `extensionData`.
