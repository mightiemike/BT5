Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Originating User, Rendering Per-Pool Swap Allowlist Inoperative — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[pool][sender]`. When a user swaps through `MetricOmmSimpleRouter`, the pool's `swap()` receives `msg.sender = router`, so `sender` passed to the extension is the router address, not the user's EOA. This structurally breaks the allowlist: either every allowlisted user is blocked (router not allowlisted), or every user bypasses the allowlist (router allowlisted).

## Finding Description

**Confirmed call path:**

`MetricOmmPool.swap()` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called the pool — the router, not the user: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` stores the originating `msg.sender` only in transient storage for the payment callback, and calls `pool.swap()` directly with no user-identity field in the pool call itself: [4](#0-3) 

The extension has no mechanism to recover the originating user from transient storage or `extensionData`. The wrong identity (`router`) is the only value available at the check site.

## Impact Explanation

Any pool deploying `SwapAllowlistExtension` as a `beforeSwap` extension faces one of two mutually exclusive broken states:

- **Mode A (users frozen):** Admin allowlists user EOAs. Users call `exactInputSingle` through the router. Extension checks `allowedSwapper[pool][router]` → `false` → every swap reverts with `NotAllowedToSwap`. Because EOAs cannot implement `IMetricOmmSwapCallback`, they cannot bypass the router and call `pool.swap()` directly. Allowlisted users are permanently unable to swap — their token positions in the pool are inaccessible via the swap path (broken core pool functionality, loss of access to funds).

- **Mode B (allowlist bypassed):** To unblock users, admin must allowlist the router. This grants every caller — including those the allowlist was meant to exclude — unrestricted swap access, defeating the entire access-control mechanism.

Both outcomes constitute broken core pool functionality meeting the contest's allowed impact gate.

## Likelihood Explanation

No attacker action is required. Any normal `exactInputSingle` call through the documented router entry point triggers the failure. Every pool that configures `SwapAllowlistExtension` is affected by construction. The pool admin's configuration is correct; the bug is structural.

## Recommendation

The extension must check the originating user, not the intermediary. The `DepositAllowlistExtension` demonstrates the correct pattern — it gates `owner` (the position owner passed explicitly by the pool), not the raw caller: [5](#0-4) 

Two complementary fixes:

1. **Forward the originating user via `extensionData`.** The router already holds `msg.sender` at `exactInputSingle` entry; pass it as part of `extensionData` so `beforeSwap` can decode and verify it.
2. **Recover user identity from transient storage inside the extension.** The router stores the payer in transient storage (`_setNextCallbackContext`); expose a read accessor and have the extension call it to obtain the real user.

Either approach must be paired with a guard that rejects calls where the identity field is absent or spoofed.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` as the `beforeSwap` extension.
2. Admin calls `setAllowedToSwap(pool, userEOA, true)`.
3. `userEOA` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, …})`.
4. Router calls `pool.swap(recipient, …)` — pool sees `msg.sender = router`.
5. Pool calls `_beforeSwap(router, …)`.
6. Extension evaluates `allowedSwapper[pool][router]` → `false`.
7. Transaction reverts with `NotAllowedToSwap`.
8. `userEOA` cannot swap despite being explicitly allowlisted.

A Foundry integration test can confirm this by: deploying the extension, configuring a pool with it, allowlisting a test EOA, calling `exactInputSingle` from that EOA through the router, and asserting the revert.
