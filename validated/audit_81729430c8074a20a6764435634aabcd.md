### Title
Silent `ValidResourceBounds` Variant Downgrade in Protobuf Deserialization Produces Wrong Transaction Hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion silently collapses an `AllResources` variant into `L1Gas` whenever `l2_gas` and `l1_data_gas` are zero. Because `get_tip_resource_bounds_hash` hashes a different number of resource elements depending on the variant, the hash recomputed after a protobuf round-trip diverges from the hash the signer and proposer computed. This is the sequencer analog of the oracle reading the current pool state without canonicalization: the conversion reads the current wire values and makes an irreversible structural decision, discarding the original variant intent.

---

### Finding Description

**Root cause — the conversion gate** [1](#0-0) 

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← variant is CHANGED
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

A V3 transaction signed with `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` — a valid and common pattern for transactions that only consume L1 gas — is silently re-typed to `L1Gas(X)` on every protobuf decode. No error is raised; the caller receives a structurally different value.

**Why the variant matters for the hash**

`get_tip_resource_bounds_hash` branches on the variant to decide how many resource felts to include: [2](#0-1) 

```
L1Gas      → poseidon(tip, L1_GAS_felt, L2_GAS_felt)          // 3 elements
AllResources → poseidon(tip, L1_GAS_felt, L2_GAS_felt, L1_DATA_GAS_felt)  // 4 elements
```

Even when `l2_gas = 0` and `l1_data_gas = 0`, the poseidon hash of 3 elements ≠ poseidon hash of 4 elements. The two variants produce cryptographically distinct `tip_resource_bounds_hash` values, which propagate into every V3 transaction hash.

**Proposer vs. validator divergence**

The proposer computes the hash through `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash`: [3](#0-2) 

`InternalRpcInvokeTransactionV3.resource_bounds` is typed `AllResourceBounds`, and its `resource_bounds()` accessor always wraps it in `ValidResourceBounds::AllResources`: [4](#0-3) 

So the proposer always hashes with `AllResources` semantics (4 elements). The hash is committed to the block.

A validating or syncing peer receives the same transaction over P2P, decodes it through `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction`: [5](#0-4) 

For the state-sync path the transaction is decoded as `InvokeTransactionV3` (which holds `ValidResourceBounds`), going through the buggy conversion. The peer recomputes the hash with `L1Gas` semantics (3 elements) and obtains a different value.

`validate_transaction_hash` then compares the recomputed hash against the committed hash: [6](#0-5) 

The comparison fails for every such transaction, causing the peer to reject the block.

**Attacker-controlled trigger**

Any unprivileged user can submit a V3 invoke, declare, or deploy-account transaction with `AllResources` bounds where `l2_gas.max_amount = 0` and `l1_data_gas.max_amount = 0`. The gateway accepts it (the bounds are valid), the proposer includes it, and every syncing/validating peer that decodes it via protobuf will compute the wrong hash.

---

### Impact Explanation

The conversion binds the wrong hash to the transaction after every protobuf round-trip. Concretely:

1. **State-sync peers reject valid blocks** — `validate_transaction_hash` returns `false` for any block containing a V3 transaction with zero L2/L1-data gas bounds, stalling the syncing node.
2. **Wrong hash in execution context** — if the wrong `ValidResourceBounds` variant reaches the blockifier's `TxInfo`, the `transaction_hash` field exposed to the account contract's `__validate__` entry point is wrong, breaking signature verification for any account that re-derives the hash on-chain.
3. **RPC returns authoritative-looking wrong hash** — a node that stored the recomputed (wrong) hash would serve it via `starknet_getTransactionByHash`, matching the "High: RPC returns an authoritative-looking wrong value" impact.

This matches: **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

---

### Likelihood Explanation

**Medium-High.** The trigger is any V3 transaction with zero L2 and L1-data gas bounds. This is the natural form of a "legacy-style" V3 transaction (pre-0.13.3 semantics submitted as V3). No special privilege is required; a single such transaction in any block is sufficient to cause hash divergence on all syncing peers.

---

### Recommendation

Remove the heuristic variant decision from the protobuf conversion. The wire format must carry an explicit discriminator (e.g., a boolean `is_all_resources` flag, or a separate `AllResourceBounds` message type) so the decoder can reconstruct the exact variant the signer used. Until the wire format is extended, the conversion should default to `AllResources` when all three bounds are present on the wire, regardless of whether their values are zero:

```rust
// Always produce AllResources when the wire carries all three fields.
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

This is safe because `get_tip_resource_bounds_hash` already handles the case where `l2_gas` and `l1_data_gas` are zero under `AllResources` — it simply includes zero-valued felts in the hash, which is the correct behavior for transactions signed with that variant.

---

### Proof of Concept

1. Construct a V3 invoke transaction with `AllResources(l1_gas=1000, l2_gas=0, l1_data_gas=0)`.
2. Compute its hash via `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` — this uses `AllResources` semantics and produces hash **H₁** (4-element poseidon).
3. Serialize the transaction to `protobuf::ConsensusTransaction` (all `transaction_hash` fields are `None` per the serializer).
4. Deserialize via `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction` — the `ResourceBounds` conversion produces `L1Gas(1000)`.
5. Recompute the hash via `get_tip_resource_bounds_hash` — this uses `L1Gas` semantics and produces hash **H₂** (3-element poseidon).
6. Assert `H₁ ≠ H₂` — the assertion holds because `poseidon([tip, L1_felt, L2_felt, L1D_felt]) ≠ poseidon([tip, L1_felt, L2_felt])` even when `L2_felt = L1D_felt = 0`.
7. Call `validate_transaction_hash` with the original transaction (deserialized as `L1Gas`) and expected hash `H₁` — it returns `false`, demonstrating that a syncing peer rejects the valid block.

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
