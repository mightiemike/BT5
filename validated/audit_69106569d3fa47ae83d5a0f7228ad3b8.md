Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User — Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` from the pool's perspective — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router address (a natural step to enable router-based swaps for permitted users), every unprivileged user can bypass the per-user allowlist by routing through the router.

## Finding Description

**Call path:**

1. `MetricOmmSimpleRouter.exactInputSingle()` (or `exactInput`, `exactOutputSingle`, `exactOutput`) calls `IMetricOmmPoolActions(pool).swap(recipient, ...)` directly.
   - `MetricOmmSimpleRouter.sol:72-80`, `104-112`, `136-137`, `165-181`

2. Inside `MetricOmmPool.swap()`, `msg.sender` is the router. The pool passes `msg.sender` as `sender` to `_beforeSwap()`:
   - `MetricOmmPool.sol:230-240` — `_beforeSwap(msg.sender, recipient, ...)`

3. `ExtensionCalling._beforeSwap()` encodes and forwards `sender` (= router address) to the extension:
   - `ExtensionCalling.sol:160-176`

4. `SwapAllowlistExtension.beforeSwap()` receives `sender` = router address and checks:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
       revert IMetricOmmPoolActions.NotAllowedToSwap();
   }
   ```
   where `msg.sender` = pool, `sender` = router.
   - `SwapAllowlistExtension.sol:37-39`

**Root cause:** The extension receives the router's address as `sender`, not the end user's address. The allowlist mapping `allowedSwapper[pool][router]` is evaluated, not `allowedSwapper[pool][user]`.

**Exploit flow:**
- Pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists specific user addresses (e.g., KYC-gated addresses).
- Pool admin also calls `setAllowedToSwap(pool, router, true)` to permit router-based swaps for those users (a natural and expected configuration step).
- A non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle({pool: curatedPool, ...})`.
- The router calls `pool.swap()` as `msg.sender`; the extension sees `sender = router`, finds `allowedSwapper[pool][router] = true`, and allows the swap.
- The non-allowlisted user successfully trades on the curated pool, defeating the allowlist policy entirely.

**Existing guards are insufficient:** The `onlyPool` modifier on the extension entry point only verifies the caller is a registered pool — it does not recover the original end-user identity. There is no mechanism in the current design to propagate the true initiator through the router into the extension's `sender` argument.

## Impact Explanation
A curated pool's swap allowlist — intended to restrict trading to a defined set of addresses — is fully bypassed by any unprivileged user routing through the public `MetricOmmSimpleRouter`. The exact corrupted value is `allowedSwapper[pool][actualUser]` (which is `false` for the attacker) being substituted by `allowedSwapper[pool][router]` (which is `true`). This constitutes an admin-boundary break: an unprivileged path (the public router) circumvents a pool-level access control that the pool admin configured and expected to be enforced. Severity: **High** — the allowlist provides zero protection for router-mediated swaps once the router is allowlisted.

## Likelihood Explanation
The precondition — router being allowlisted — is not a malicious or exotic setup. Any pool admin who wants allowlisted users to be able to use the standard router must allowlist the router address, since without it, even permitted users cannot route through it. The bypass is then trivially reachable by any unprivileged user with no special capability, repeatable on every block, and requires no flash loans or oracle manipulation.

## Recommendation
The extension must gate on the true end-user identity, not the immediate pool caller. Options:
1. Pass the original `msg.sender` (end user) through the router as part of `extensionData` and have the extension decode and verify it — but this is forgeable unless the pool enforces it.
2. Have the router store the originating user in transient storage and expose a view function the extension can call to retrieve it during the hook.
3. Require direct pool interaction for allowlisted pools (document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and enforce this at the extension or factory level).
4. Change the allowlist to gate on `recipient` instead of `sender` if the pool's security model is recipient-based — but this must be validated against the full threat model.

## Proof of Concept

```solidity
// Foundry test sketch
function test_swapAllowlist_bypassViaRouter() public {
    // Setup: curated pool with SwapAllowlistExtension
    // Pool admin allowlists only `allowedUser`
    swapExtension.setAllowedToSwap(address(pool), allowedUser, true);
    // Pool admin also allowlists the router so allowedUser can use it
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // Attacker: not in allowlist
    address attacker = makeAddr("attacker");
    assertFalse(swapExtension.isAllowedToSwap(address(pool), attacker));

    // Attacker routes through the router — extension sees router as sender
    vm.prank(attacker);
    router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token1),
        recipient: attacker,
        deadline: block.timestamp + 1,
        amountIn: 1000,
        amountOutMinimum: 0,
        zeroForOne: false,
        priceLimitX64: type(uint128).max,
        extensionData: ""
    }));
    // Swap succeeds — allowlist bypassed
}
```
