### Title
Gas-key host functions pass wire-length instead of trie-id-length to exec-fee helpers for ML-DSA-65 keys, causing ~42× exec-gas overcharge — (File: `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The host functions `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, and `promise_batch_action_transfer_to_gas_key` pass `public_key_len as usize` — the raw borsh wire length of the public key as supplied by the contract — to `gas_key_add_key_exec_fee` / `gas_key_transfer_exec_fee`. Those helpers are documented to model **trie I/O cost** and expect the on-trie identifier length (`trie_id_len()`). For ML-DSA-65 keys the wire length is 1953 bytes but the trie ID length is 33 bytes, so the exec fee is computed with the wrong unit, producing an ~42× overcharge. The runtime path (transactions) correctly passes `public_key.trie_id_len()` for the same helpers; the host-function path does not.

---

### Finding Description

`gas_key_add_key_exec_fee` and `gas_key_transfer_exec_fee` in `core/parameters/src/cost.rs` accept a `public_key_len` parameter that is fed directly into `access_key_key_len(account_id_len, public_key_len)` to compute the trie key length that the receiver must read/write: [1](#0-0) 

The runtime path in `runtime/runtime/src/config.rs` correctly passes `public_key.trie_id_len()` (33 bytes for ML-DSA-65): [2](#0-1) 

But both the `VMLogic` host-function path and the wasmtime path pass `public_key_len as usize` — the raw wire length the contract wrote into memory — to the same exec-fee helpers: [3](#0-2) [4](#0-3) [5](#0-4) 

The same pattern appears in the wasmtime runner: [6](#0-5) [7](#0-6) 

The two lengths diverge only for ML-DSA-65:

| Key type | `len()` / wire | `trie_id_len()` / on-trie |
|---|---|---|
| ED25519 | 33 B | 33 B |
| SECP256K1 | 65 B | 65 B |
| **ML-DSA-65** | **1953 B** | **33 B** | [8](#0-7) 

The send-fee helper intentionally uses the wire length ("what the sender sees"), but the exec-fee helper is explicitly documented to model trie I/O cost: [9](#0-8) 

The design document confirms the invariant: every storage-stake and trie-byte-priced fee path must call `trie_id_len()`, not `len()`: [10](#0-9) 

---

### Impact Explanation

For an ML-DSA-65 gas key with a 10-byte account ID and `num_nonces = 1`:

- **Correct** (`trie_id_len = 33`): `nonce_key_len = access_key_key_len(10, 33) + 2 = 47`
- **Buggy** (`wire_len = 1953`): `nonce_key_len = access_key_key_len(10, 1953) + 2 = 1967`

The per-byte exec component is `1967 / 47 ≈ 41.9×` too large. The exec fee is charged as `used_gas` (prepaid gas for future execution). Any contract that calls these host functions with an ML-DSA-65 gas key must attach ~42× more gas than the actual trie work warrants. If it does not, the call fails with an out-of-gas error even though the underlying trie operation would succeed. This is a protocol-level denial-of-service for contracts using ML-DSA-65 gas keys via host functions, and a deterministic gas-pricing inconsistency between the transaction path and the contract path for the same action type.

---

### Likelihood Explanation

The `PostQuantumSignatures` protocol feature must be enabled for ML-DSA-65 keys to be admitted into state. Once enabled, any unprivileged contract can call these host functions with an ML-DSA-65 public key. The overcharge is deterministic: every valid ML-DSA-65 key is exactly 1953 bytes on the wire, so the ~42× overcharge applies to every such call without exception. No special attacker capability is required beyond submitting a function call that invokes one of the three affected host functions with an ML-DSA-65 key.

---

### Recommendation

In all three host functions (`promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, `promise_batch_action_transfer_to_gas_key`) in both `runtime/near-vm-runner/src/logic/logic.rs` and `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`, replace `public_key_len as usize` in the exec-fee call with `public_key.decode()?.trie_id_len()`. The send-fee call should continue to use `public_key_len as usize` because it intentionally models the wire cost of transmitting the key.

---

``` [3](#0-2) [2](#0-1) 

The exact divergent values for ML-DSA-65: `public_key_len as usize = 1953` vs `trie_id_len() = 33`. The exec fee helper uses this value to compute `access_key_key_len(account_id_len, public_key_len)`, which is the trie key length the receiver must read/write. Passing 1953 instead of 33 inflates the per-byte exec component by a factor of `(1 + account_id_len + 1 + 1953 + 2) / (1 + account_id_len + 1 + 33 + 2)` — approximately 41–42× for typical account ID lengths — causing every contract-initiated ML-DSA-65 gas-key operation to require ~42× more prepaid gas than the equivalent transaction-initiated operation.

/// Send fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the public key length (what the sender sees).
pub fn gas_key_transfer_send_fee(
    cfg: &RuntimeFeesConfig,
    sender_is_receiver: bool,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).send_fee(sender_is_receiver);
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .send_fee(sender_is_receiver)
        .checked_mul(public_key_len as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

    /// Length of this public key's borsh encoding, in bytes - that is,
    /// the on-the-wire size of the raw key bytes plus a 1-byte borsh
    /// discriminant tag (the leading `+ 1` in each arm).
    ///
    /// For storage-fee accounting use [`PublicKey::trie_id_len`] instead;
    /// for ML-DSA-65 those two diverge (1953 wire vs 33 on-trie).
    // `is_empty` always returns false, so there is no point in adding it
    #[allow(clippy::len_without_is_empty)]
    pub fn len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_PUBLIC_KEY_LENGTH,
        }
    }

    pub fn empty(key_type: KeyType) -> Self {
        match key_type {
            KeyType::ED25519 => {
                PublicKey::ED25519(ED25519PublicKey([0u8; ed25519_dalek::PUBLIC_KEY_LENGTH]))
            }
            KeyType::SECP256K1 => PublicKey::SECP256K1(Secp256K1PublicKey([0u8; 64])),
            KeyType::MLDSA65 => {
                PublicKey::MLDSA65(MlDsa65PublicKey(Box::new([0u8; ML_DSA_65_PUBLIC_KEY_LENGTH])))
            }
        }
    }

    pub fn key_type(&self) -> KeyType {
        match self {
            Self::ED25519(_) => KeyType::ED25519,
            Self::SECP256K1(_) => KeyType::SECP256K1,
            Self::MLDSA65(_) => KeyType::MLDSA65,
        }
    }

    fn key_tag(&self) -> KeyTag {
        match self {
            PublicKey::ED25519(_) => KeyTag::Ed25519,
            PublicKey::SECP256K1(_) => KeyTag::Secp256k1,
            PublicKey::MLDSA65(_) => KeyTag::MlDsa65Full,
        }
    }

    pub fn key_data(&self) -> &[u8] {
        match self {
            Self::ED25519(key) => key.as_ref(),
            Self::SECP256K1(key) => key.as_ref(),
            Self::MLDSA65(key) => key.as_ref(),
        }
    }

    pub fn unwrap_as_ed25519(&self) -> &ED25519PublicKey {
        match self {
            Self::ED25519(key) => key,
            Self::SECP256K1(_) | Self::MLDSA65(_) => panic!(),
        }
    }

    pub fn unwrap_as_secp256k1(&self) -> &Secp256K1PublicKey {
        match self {
            Self::SECP256K1(key) => key,
            Self::ED25519(_) | Self::MLDSA65(_) => panic!(),
        }
    }

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }

### 5. Storage usage and fee plumbing

The storage-stake calculation
(`runtime/runtime/src/access_keys.rs::access_key_storage_usage`) and the
gas-key fee helpers (`gas_key_*_fee` in `runtime/runtime/src/config.rs`) use
`PublicKey::trie_id_len()` rather than `PublicKey::len()`:

- `len()` reports the borsh-encoded length (33 / 65 / 1953 across the
  three `PublicKey` variants).
- `trie_id_len()` reports the on-trie length (33 / 65 / **33**).

The two diverge only for `PublicKey::MLDSA65`. Every storage-stake and
trie-byte-priced fee path was updated to call `trie_id_len()`.
