### Title
Linked Signer Can Self-Replace via `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

### Summary

The `LinkSigner` transaction handler in `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner=true`. The `Verifier.validateSignature` function accepts a signature from either the subaccount owner **or** the current linked signer. This means a linked signer can unilaterally sign a new `LinkSigner` transaction to replace themselves (or the existing linked signer) with any attacker-controlled address, permanently transferring signing authority without the subaccount owner's consent.

### Finding Description

In `EndpointTx.sol`, the `LinkSigner` branch of `processTransactionImpl` is:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(...);
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner=true` calls `validateSignature`, which passes the current linked signer address to the verifier:

```solidity
// EndpointTx.sol lines 172–184
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` then accepts the signature if it comes from **either** the subaccount owner **or** the linked signer:

```solidity
// Verifier.sol lines 291–304
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),   // <-- linked signer accepted
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

There is no additional check that restricts `LinkSigner` mutations to the subaccount owner only. The same `allowLinkedSigner=true` flag used for trading transactions is reused here, which is incorrect for an administrative operation that changes the trust relationship itself.

### Impact Explanation

A compromised or malicious linked signer can:
1. Sign `LinkSigner{sender: victimSubaccount, signer: attacker2}`.
2. Submit it through `submitTransactionsChecked` → `processTransactionImpl`.
3. `linkedSigners[victimSubaccount]` is overwritten to `attacker2`.
4. `attacker2` can now sign `WithdrawCollateral`, `TransferQuote`, and all other `allowLinkedSigner=true` transactions on behalf of the victim.
5. The original owner loses effective control until they submit a new owner-signed `LinkSigner` — but if the attacker front-runs or the owner is unaware, assets can be drained first.

This directly matches the scoped critical impact: **unauthorized transaction execution (withdrawals, trades) on behalf of a user without valid consent from the subaccount owner**.

### Likelihood Explanation

Any party that has ever been granted linked-signer status (e.g., a trading bot, a third-party integration, or a key that was later compromised) can execute this attack without any additional privileges. The entrypoint is the standard sequencer-submitted transaction path, which is fully supported in production.

### Recommendation

`LinkSigner` must **not** accept a linked signer's signature to authorize a new signer link. Change the `allowLinkedSigner` argument for the `LinkSigner` handler to `false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [4](#0-3) 

The same fix should be applied to the slow-mode `LinkSigner` path, which already enforces `validateSender` (msg.sender must match the subaccount owner address) and is therefore not vulnerable via that path — but consistency is important.

### Proof of Concept

```solidity
// 1. Owner links attacker1 as signer for victimSubaccount
//    linkedSigners[victimSubaccount] = attacker1

// 2. attacker1 constructs and signs:
//    LinkSigner{ sender: victimSubaccount, signer: attacker2, nonce: currentNonce }

// 3. Sequencer (or attacker1 via submitTransactionsChecked) submits the tx.
//    validateSignedTx passes because:
//      recovered == attacker1 == getLinkedSigner(victimSubaccount)

// 4. State after: linkedSigners[victimSubaccount] = attacker2

// 5. attacker2 signs WithdrawCollateral{ sender: victimSubaccount, ... }
//    → funds drained from victim's account

// Assert: linkedSigners[victimSubaccount] == attacker2 (not attacker1, not owner)
// Assert: owner-signed recovery requires a new LinkSigner tx, but assets may already be gone
```
