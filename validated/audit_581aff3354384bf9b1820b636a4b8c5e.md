### Title
Missing Custom `Authorization` Type for `MsgTransfer` Allows Authz Grantee to Transfer All Grantor NFTs — (`x/nft-transfer/types/msgs.go`, `x/nft-transfer/keeper/packet.go`)

---

### Summary

The `x/nft-transfer` module does not implement a custom `authz.Authorization` type for `MsgTransfer`. As a result, any authz grant for `MsgTransfer` is a fully unrestricted grant: a grantee can execute `MsgTransfer` on behalf of the grantor for **any** classID and any tokenID the grantor owns, including NFTs the grantor never intended to authorize for transfer.

---

### Finding Description

**`GetSigners` returns `msg.Sender` (the grantor):** [1](#0-0) 

When the authz module processes a `MsgExec` wrapping a `MsgTransfer`, it extracts the signer from `GetSigners()` — which is `msg.Sender` (the grantor) — and verifies the grantee holds a grant for the `MsgTransfer` message type from that grantor. If the grant exists, execution proceeds.

**No custom `Authorization` type exists in the module:**

A grep across the entire `x/nft-transfer` package returns zero matches for `Authorization`, `authz`, or `TransferAuthorization`. Compare this to the Cosmos SDK bank module's `SendAuthorization` or ibc-go's fungible `TransferAuthorization`, both of which carry spend/channel limits. The nft-transfer module has no equivalent. [2](#0-1) 

Only `MsgTransfer` itself is registered — no `Authorization` implementation is registered.

**Ownership check in `createOutgoingPacket` passes for the grantor:** [3](#0-2) 

The check `sender.Equals(owner)` uses `msg.Sender` (the grantor). Since the grantor legitimately owns the NFT, this check passes. The NFT is then escrowed or burned: [4](#0-3) 

**The `Transfer` handler passes `msg.Sender` directly as the sender:** [5](#0-4) 

There is no additional check that the actual transaction signer (the grantee) is authorized for the specific classID or tokenID being transferred.

---

### Impact Explanation

A grantee holding a generic `MsgTransfer` authz grant can:
1. Construct a `MsgTransfer` with `Sender = grantor`, `ClassId = <any class grantor owns>`, `TokenIds = [<any token>]`, `Receiver = attacker_address`.
2. Wrap it in `MsgExec` and submit it.
3. The authz module approves (grant exists for the message type).
4. `createOutgoingPacket` approves (grantor owns the NFT).
5. The NFT is escrowed/burned and an IBC packet is sent to the attacker-controlled receiver.

The grantor has no mechanism to issue a restricted grant (e.g., "only classID X, tokenID Y"). All NFTs owned by the grantor are implicitly in scope of any `MsgTransfer` grant.

---

### Likelihood Explanation

Authz grants for IBC transfer messages are a common pattern for marketplace integrations, automated bots, and delegated operations. A user granting `MsgTransfer` permission to a third-party service (e.g., an NFT marketplace contract or relayer) would unknowingly expose their entire NFT portfolio to that grantee.

---

### Recommendation

Implement a custom `TransferAuthorization` type for the nft-transfer module (analogous to ibc-go's `TransferAuthorization` for fungible tokens) that:
- Carries an allowlist of `(classID, []tokenID)` pairs the grantee is permitted to transfer.
- Implements `cosmos/cosmos-sdk/x/authz.Authorization` interface (`MsgTypeURL()`, `Accept()`, `ValidateBasic()`).
- Is registered in `codec.go` via `RegisterImplementations`.

The `Accept()` method should reject any `MsgTransfer` whose `ClassId` or `TokenIds` are not in the allowlist, and optionally decrement/remove the grant after use.

---

### Proof of Concept

```
1. Alice (grantor) owns NFT {classID: "crypto-punk", tokenID: "001"} (high-value)
   and NFT {classID: "crypto-punk", tokenID: "002"} (intended for transfer).

2. Alice grants Bob (grantee) a generic MsgTransfer authz grant:
   MsgGrant{
     Granter: alice,
     Grantee: bob,
     Grant: {Authorization: GenericAuthorization{MsgTypeURL: "/crypto_org_chain_chainmain.nft_transfer.v1.MsgTransfer"}}
   }

3. Bob submits:
   MsgExec{
     Grantee: bob,
     Msgs: [MsgTransfer{
       Sender:    alice,
       ClassId:   "crypto-punk",
       TokenIds:  ["001"],          // high-value NFT Alice never intended to transfer
       Receiver:  attacker_address,
       SourcePort: "nft-transfer",
       SourceChannel: "channel-0",
       ...
     }]
   }

4. Authz module: bob has a grant for MsgTransfer from alice → approved.
5. Transfer handler: sender = alice (from msg.Sender).
6. createOutgoingPacket: alice.Equals(owner of "001") → true → NFT escrowed.
7. IBC packet sent; on destination chain, attacker_address receives the NFT.
```
