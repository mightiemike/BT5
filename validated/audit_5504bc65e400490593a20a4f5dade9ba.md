Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual user, allowing full allowlist bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the originating user. Any pool admin who allowlists the router to support standard periphery usage inadvertently grants every user — allowlisted or not — unrestricted access to the pool.

## Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist: [1](#0-0) 

`MetricOmmPool.swap` always forwards its own `msg.sender` as the `sender` argument to `_beforeSwap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool: [3](#0-2) 

The router passes `params.extensionData` verbatim from the user's call — it does not encode the originating user's address into `extensionData`. The extension therefore has no mechanism to recover the real user. The allowlist check becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

The pool admin faces an impossible choice: not allowlisting the router blocks all router-mediated swaps (including for legitimately allowlisted users), while allowlisting the router opens the pool to every user unconditionally.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` is intended to be permissioned (e.g., KYC-gated or institutional-only). Once the router is allowlisted — the necessary configuration for any pool supporting the standard periphery — the allowlist provides zero protection. Any unprivileged user can call `router.exactInputSingle` or `router.exactInput` targeting the restricted pool and execute swaps at the oracle price, directly extracting LP principal. This constitutes a direct loss of user/LP funds and breaks the core access-control functionality the extension was deployed to enforce.

## Likelihood Explanation

No special privilege, flash loan, or oracle manipulation is required. Any EOA can call the router. The `MetricOmmSimpleRouter` is the primary user-facing entry point, so any pool admin who wants to support standard usage must allowlist it, triggering the bypass. The attack is repeatable and requires no malicious setup beyond a standard router call.

## Recommendation

The extension must gate the economically relevant actor — the end user — not the intermediary. Two sound approaches:

1. **Router encodes originating user in `extensionData`**: The router always encodes `msg.sender` (the actual user) into `extensionData` for each hop; the extension decodes and checks that address instead of `sender`.
2. **Fallback logic in the extension**: The extension inspects whether `sender` is a known router and, if so, reads the real user from `extensionData`, falling back to `sender` for direct pool calls.

The simplest correct fix is approach 1: the router encodes the originating user into `extensionData`, and `SwapAllowlistExtension.beforeSwap` decodes and checks that value when present.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack (executed by bob, who is NOT allowlisted):
  1. bob calls router.exactInputSingle({pool: pool, recipient: bob, extensionData: "", ...})
  2. Router calls pool.swap(bob, ...) — msg.sender to pool = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true ✓
  5. Swap executes; bob receives output tokens from the restricted pool.

Expected: revert NotAllowedToSwap
Actual:   swap succeeds — allowlist fully bypassed
```
