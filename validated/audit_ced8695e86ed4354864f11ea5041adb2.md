### Title
`starknet_estimateFee` Returns `overall_fee` That Includes L2 Gas Cost But Omits `l2_gas_consumed` From Breakdown, Making the Spec-Stated Formula Incorrect - (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`tx_execution_output_to_fee_estimation` builds the `FeeEstimation` response by populating `gas_consumed` (L1 gas) and `data_gas_consumed` (L1 data gas) from the gas vector, but silently omits `l2_gas_consumed`. The `overall_fee` field is taken directly from `receipt.fee`, which for V3 (`AllResources`) transactions already includes the L2 gas component. The OpenRPC spec description for `overall_fee` states it "equals to `gas_consumed*gas_price + data_gas_consumed*data_gas_price`", which is arithmetically false for any V3 transaction that consumes non-zero L2 gas. A client that trusts the spec formula to reconstruct or verify the fee will compute a value strictly less than `overall_fee`, and has no way to derive the correct `l2_gas.max_amount` for a follow-up submission.

### Finding Description

In `tx_execution_output_to_fee_estimation`:

```rust
let gas_vector = tx_execution_output.execution_info.receipt.gas;

Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l1_data_gas_price,
    l2_gas_price,
    overall_fee: tx_execution_output.execution_info.receipt.fee,  // includes l2_gas cost
    unit: tx_execution_output.price_unit,
})
```

`gas_vector.l2_gas` is present in the `GasVector` receipt but is never written into the response. The `overall_fee` is computed by the blockifier as:

```
overall_fee = l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * (l2_gas_price + tip)
```

The OpenRPC spec description for `overall_fee` says:

> "equals to gas_consumed\*gas_price + data_gas_consumed\*data_gas_price"

This formula omits the L2 gas term entirely. For any V3 transaction with non-zero L2 gas consumption, `overall_fee > gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`. The gap is the L2 gas cost, which is unexplained in the response. The developer TODO in the struct confirms the field is known to be missing:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
pub l2_gas_price: GasPrice,
```

### Impact Explanation

This matches: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

The `FeeEstimation` response is the canonical output of `starknet_estimateFee`. It exposes `l2_gas_price` (signalling that L2 gas is relevant) but withholds `l2_gas_consumed`. The `overall_fee` is numerically correct but the spec-stated decomposition formula is false. A client that:

1. Calls `starknet_estimateFee` to size resource bounds for a V3 transaction,
2. Reads `gas_consumed` → sets `l1_gas.max_amount`,
3. Reads `data_gas_consumed` → sets `l1_data_gas.max_amount`,
4. Has no `l2_gas_consumed` field → cannot set `l2_gas.max_amount` correctly,

will either set `l2_gas.max_amount = 0` (causing the transaction to revert with "Insufficient max L2Gas" in post-execution) or set it to an arbitrary large value (which is safe but defeats the purpose of bounded estimation). Neither outcome is derivable from the response alone. The `overall_fee` value is higher than the formula predicts, and the "excess" — the L2 gas cost — is silently absorbed into `overall_fee` with no breakdown field to explain it.

### Likelihood Explanation

Every V3 (`AllResources`) transaction on a network running Starknet ≥ 0.13.3 consumes L2 gas. The `starknet_estimateFee` endpoint is the standard pre-submission tool. Any client that follows the spec formula to verify or decompose the fee will observe a discrepancy. Clients that use the response to set `l2_gas.max_amount` have no correct value to use.

### Recommendation

Add `l2_gas_consumed` to `FeeEstimation` and populate it in `tx_execution_output_to_fee_estimation`:

```rust
pub struct FeeEstimation {
    pub gas_consumed: Felt,
    pub l1_gas_price: GasPrice,
    pub data_gas_consumed: Felt,
    pub l1_data_gas_price: GasPrice,
    pub l2_gas_consumed: Felt,          // add this
    pub l2_gas_price: GasPrice,
    pub overall_fee: Fee,
    pub unit: PriceUnit,
}

// in tx_execution_output_to_fee_estimation:
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l1_data_gas_price,
    l2_gas_consumed: gas_vector.l2_gas.0.into(),   // populate from gas_vector
    l2_gas_price,
    overall_fee: tx_execution_output.execution_info.receipt.fee,
    unit: tx_execution_output.price_unit,
})
```

Update the OpenRPC spec description for `overall_fee` to:

> "equals to `gas_consumed*l1_gas_price + data_gas_consumed*l1_data_gas_price + l2_gas_consumed*l2_gas_price`"

pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
