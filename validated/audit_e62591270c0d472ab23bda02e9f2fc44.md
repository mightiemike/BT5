### Title
Hardcoded `softfork_kw` in `RuntimeDialect` Silently Overrides User-Configured Operator at Opcode 36 — (`src/runtime_dialect.rs`)

---

### Summary

`RuntimeDialect::new()` accepts `quote_kw` and `apply_kw` as caller-supplied parameters but unconditionally hardcodes `softfork_kw: vec![36]`. Because the interpreter checks `softfork_kw` before dispatching to the operator table, any `RuntimeDialect` that maps opcode 36 to a custom operator will silently have that operator bypassed — the interpreter always treats opcode 36 as the softfork guard instead.

---

### Finding Description

`RuntimeDialect` is a public, configurable dialect intended for custom operator sets. Its constructor signature accepts `quote_kw` and `apply_kw` as parameters, making them runtime-configurable. However, `softfork_kw` is hardcoded to `vec![36]` regardless of what the caller passes: [1](#0-0) 

The struct stores `softfork_kw` as a `Vec<u8>` field — a design that implies it was intended to be configurable — but the constructor never exposes it as a parameter: [2](#0-1) 

In `run_program.rs`, the interpreter checks `softfork_kw` **before** dispatching to the dialect's `op()` method: [3](#0-2) 

This means that in any `RuntimeDialect` where opcode 36 is registered in `op_map` as a custom operator, the interpreter intercepts it as a softfork guard and never reaches the `op()` dispatch. The custom operator at opcode 36 is permanently unreachable.

The analog to the external report is direct: just as `@oft_admin` was hardcoded where `get_admin()` should have been called, `softfork_kw` is hardcoded to `36` where the caller-supplied value should have been used — and the struct even stores the field as if it were meant to be configurable.

---

### Impact Explanation

Any downstream code that constructs a `RuntimeDialect` and registers a custom operator at opcode 36 will silently receive incorrect execution: the softfork guard fires instead of the intended operator. Depending on the program being evaluated:

- The custom operator's logic is never executed.
- The softfork guard's cost accounting and `ExitGuard` logic runs instead, producing a different result and cost.
- If the custom dialect is used in a consensus-critical path (e.g., a custom chain or generator), this causes consensus divergence between nodes that expect the custom operator and the actual execution outcome. [4](#0-3) 

---

### Likelihood Explanation

`RuntimeDialect` is a public Rust API. Any downstream integrator building a custom CLVM dialect who assigns opcode 36 to a non-softfork operator — a reasonable choice given that `quote_kw` and `apply_kw` are fully configurable — will trigger this silently. The bug requires no special attacker capability beyond supplying CLVM bytes that invoke opcode 36, which is attacker-controlled input in any CLVM execution context.

---

### Recommendation

Add `softfork_kw` as an explicit parameter to `RuntimeDialect::new()`, mirroring the treatment of `quote_kw` and `apply_kw`:

```rust
pub fn new(
    op_map: HashMap<String, Vec<u8>>,
    quote_kw: Vec<u8>,
    apply_kw: Vec<u8>,
    softfork_kw: Vec<u8>,   // <-- add this
    flags: ClvmFlags,
) -> RuntimeDialect {
    RuntimeDialect {
        f_lookup: f_lookup_for_hashmap(op_map),
        quote_kw,
        apply_kw,
        softfork_kw,          // <-- use caller-supplied value
        flags,
    }
}
```

Callers that want the standard Chia softfork behavior can pass `vec![36]` explicitly; callers that want no softfork keyword can pass an empty or unused byte.

### Proof of Concept

```rust
use clvmr::chia_dialect::ClvmFlags;
use clvmr::runtime_dialect::RuntimeDialect;
use clvmr::run_program::run_program;

// Register a custom operator at opcode 36 (e.g., a no-op that returns nil)
let mut op_map = HashMap::new();
op_map.insert("my_op".to_string(), vec![36u8]);

let dialect = RuntimeDialect::new(
    op_map,
    vec![1],  // quote_kw
    vec![2],  // apply_kw
    ClvmFlags::empty(),
);

let mut allocator = Allocator::new();
// Program: (my_op) — invoke opcode 36 directly
// Expected: custom operator executes, returns nil
// Actual:   softfork guard fires (opcode 36 == hardcoded softfork_kw),
//           parses softfork arguments, fails or produces wrong cost/result
let program_bytes = hex::decode("ff24ff8080").unwrap(); // (36 ())
// run_program will treat opcode 36 as softfork, not as the custom operator
```
