### Title
`validate_delegate_action` validates inner actions against the outer transaction's `receiver_id` instead of `delegate_action.receiver_id`, breaking the identity domain for `DeterministicStateInitAction` in meta transactions — (`File: runtime/runtime/src/action_validation.rs`)

---

### Summary

`validate_delegate_action` passes the **outer transaction's `receiver_id`** as the `receiver` argument when recursively validating inner actions of a `DelegateAction`, instead of using `delegate_action.receiver_id()`. This is the exact nearcore analog of the ZetaChain bug: an intermediary/outer identity is substituted for the actual target identity, breaking the validation domain invariant. The consequence is twofold: (1) legitimate meta transactions carrying a `DeterministicStateInitAction` are incorrectly rejected at tx-admission, and (2) a crafted meta transaction can bypass tx-level validation entirely, only failing later at receipt execution.

---

### Finding Description

In `runtime/runtime/src/action_validation.rs`, `validate_delegate_action` receives `receiver: &AccountId`, which is the **outer transaction's `receiver_id`** (i.e., the account the relayer's transaction targets — the meta-tx sender's account). It then passes this value as the `inner_receiver` to `validate_actions_with_mode` for the inner actions:

```rust
fn validate_delegate_action(
    limit_config: &LimitConfig,
    delegate_action: VersionedDelegateActionRef<'_>,
    receiver: &AccountId,          // ← outer_tx.receiver_id, NOT delegate_action.receiver_id
    current_protocol_version: ProtocolVersion,
    mode: ValidateReceiptMode,
) -> Result<(), ActionsValidationError> {
    let actions = delegate_action.get_actions();
    let inner_receiver =
        if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
            delegate_action.receiver_id()   // correct post-fix
        } else {
            receiver                        // ← BUG: outer tx receiver used pre-fix
        };
    validate_actions_with_mode(limit_config, &actions, inner_receiver, ...)?;
```

`validate_deterministic_state_init` checks that `derived_id == receiver_id`. When `receiver_id` is the outer tx receiver (e.g., `det_account_b = derive(state_init_b)`) but the delegate action targets a different deterministic account (`det_account_a`), the check passes at tx admission — because `det_account_b == derive(state_init_b)` — even though the inner action will execute against `det_account_a`. The receipt-level re-validation then catches the mismatch and fails the receipt.

The reverse is equally broken: a legitimate user who wants to send `state_init_b` to `det_account_b` via a meta transaction where `outer_tx.receiver_id = det_account_b` (the meta-tx sender) will have their transaction **rejected** at tx admission, because the outer receiver happens to equal the correct derived ID but the validation path is still wrong in the general case.

The divergent values are exact:
- **Buggy path**: `inner_receiver = outer_tx.receiver_id` (the relayer's target, i.e., the meta-tx sender's account)
- **Correct path**: `inner_receiver = delegate_action.receiver_id()` (the actual target of the inner actions) [1](#0-0) 

The call site that passes the wrong `receiver` is in `validate_action_with_mode`: [2](#0-1)

---

### Impact Explanation

**Functional breakage (DoS on feature combination):** Any user attempting to use `DeterministicStateInitAction` inside a `DelegateAction` on a node running protocol version < 85 will have their transaction rejected at tx admission with `InvalidDeterministicStateInitReceiver`, even when the transaction is fully valid. The feature combination is completely unusable.

**Validation domain bypass:** An attacker can craft a meta transaction where `outer_tx.receiver_id = derive(state_init_b)` but `delegate_action.receiver_id = det_account_a` (a different deterministic account). This transaction passes tx-level validation and is included in a block, but fails at receipt execution. The attacker pays gas; no state is corrupted. However, the tx-admission gate — which is supposed to be the first line of defense — is bypassed, wasting block space and validator compute.

The test `test_deterministic_state_init_meta_tx_receiver_check_pre_fix` confirms the exploit path is reachable: [4](#0-3)

---

### Likelihood Explanation

Any unprivileged user can submit a `SignedTransaction` containing a `DelegateAction` wrapping a `DeterministicStateInitAction`. No special role is required. The bug is triggered at the public tx-admission boundary (`validate_actions` → `validate_delegate_action`). On protocol versions < 85 (the `FixDelegatedDeterministicStateInit` boundary), the wrong identity is used unconditionally.

The `FixDelegatedDeterministicStateInit` feature stabilized at protocol version 85: [6](#0-5) 

---

### Recommendation

The fix is already present: when `ProtocolFeature::FixDelegatedDeterministicStateInit` is enabled, `validate_delegate_action` correctly uses `delegate_action.receiver_id()` as the identity for inner-action validation. The old path is retained only for backward compatibility with pre-85 protocol versions. No further action is needed for current production, but the old code path should be documented as permanently broken for the `DeterministicStateInitAction` + meta-transaction combination on pre-85 nodes.

### Proof of Concept

The test `try_meta_tx_deterministic_receiver_exploit` in `test-loop-tests/src/tests/deterministic_account_id.rs` constructs the exact scenario:

1. Deploy `det_account_b = derive(state_init_b)` and add a full-access key to it.
2. Craft a `DelegateAction` with `sender_id = det_account_b`, `receiver_id = det_account_a` (wrong target), containing `state_init_b`.
3. Wrap it in an outer transaction with `receiver_id = det_account_b`.
4. On protocol version < 85: tx validation passes (outer receiver `det_account_b` matches `derive(state_init_b)`), transaction is included in a block, receipt fails with `InvalidDeterministicStateInitReceiver`.
5. On protocol version ≥ 85: tx validation correctly rejects the transaction immediately. [7](#0-6)

fn validate_delegate_action(
    limit_config: &LimitConfig,
    delegate_action: VersionedDelegateActionRef<'_>,
    receiver: &AccountId,
    current_protocol_version: ProtocolVersion,
    mode: ValidateReceiptMode,
) -> Result<(), ActionsValidationError> {
    let actions = delegate_action.get_actions();
    let inner_receiver =
        if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
            // This is the correct receiver id to use for the check.
            delegate_action.receiver_id()
        } else {
            // This is a bug fixed with `FixDelegatedDeterministicStateInit` that
            // validated against the wrong id. This makes it impossible to
            // initialize deterministic accounts from meta transactions.
            // The bug cannot be abused, if someone crafts a state init that passes
            // validation here, it will fail when it is checked as incoming receipt.
            receiver
        };
    validate_actions_with_mode(
        limit_config,
        &actions,
        inner_receiver,
        current_protocol_version,
        mode,
    )?;
    Ok(())
}

fn try_meta_tx_deterministic_receiver_exploit(
    protocol_version: ProtocolVersion,
) -> Result<FinalExecutionOutcomeView, InvalidTxError> {
    let mut env = TestEnv::setup_with_version(Balance::from_near(100), protocol_version);
    env.deploy_global_contract(GlobalContractDeployMode::AccountId);

    let (_state_init_a, det_account_a) = env.new_deterministic_account_with_data(small());
    let (state_init_b, det_account_b) = env.new_deterministic_account_with_data(big());
    assert_ne!(det_account_a, det_account_b);

    // Deploy det_account_b and add a full-access key so it can act as meta_tx_sender.
    let user_signer = create_user_test_signer(&env.user_account());
    let storage_balance = env.balance_for_storage(state_init_b.clone());
    let deploy_tx = SignedTransaction::deterministic_state_init(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        env.get_tx_block_hash(),
        state_init_b.clone(),
        storage_balance,
    );
    env.run_tx(deploy_tx);

    let meta_tx_sender_signer = create_user_test_signer(&det_account_b);
    let pk_base64 = near_primitives_core::serialize::to_base64(
        &borsh::to_vec(&meta_tx_sender_signer.public_key()).unwrap(),
    );
    let add_key_args = serde_json::json!([
        { "batch_create": { "account_id": det_account_b.as_str() }, "id": 0 },
        {
            "action_add_key_with_full_access": {
                "promise_index": 0,
                "public_key": pk_base64,
                "nonce": 0
            },
            "id": 0,
            "return": true
        }
    ]);
    let add_key_tx = SignedTransaction::call(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        Balance::from_near(2),
        "call_promise".to_owned(),
        serde_json::to_vec(&add_key_args).unwrap(),
        Gas::from_teragas(300),
        env.get_tx_block_hash(),
    );
    env.run_tx(add_key_tx);

    // Craft the exploit: outer_tx.receiver = det_account_b = derive(state_init_b).
    // Old check: det_account_b == derive(state_init_b) passes.
    // The delegate action targets det_account_a, which is the wrong account.
    // In no protocol version can this ever be allowed to be executed successfully.
    let relayer = env.independent_account();
    let relayer_signer = create_user_test_signer(&relayer);
    let inner_action = Action::DeterministicStateInit(Box::new(DeterministicStateInitAction {
        state_init: state_init_b,
        deposit: Balance::ZERO,
    }));
    let delegate_nonce = env.next_nonce_for(&det_account_b);
    let delegate_action = DelegateAction {
        sender_id: det_account_b.clone(),
        receiver_id: det_account_a,
        actions: vec![NonDelegateAction::try_from(inner_action).unwrap()],
        nonce: delegate_nonce,
        max_block_height: 1_000_000,
        public_key: meta_tx_sender_signer.public_key(),
    };
    let signed_delegate_action =
        SignedDelegateAction::sign(&meta_tx_sender_signer, delegate_action);
    let tx = SignedTransaction::from_actions(
        env.next_nonce(),
        relayer,
        det_account_b,
        &relayer_signer,
        vec![Action::Delegate(Box::new(signed_delegate_action))],
        env.get_tx_block_hash(),
    );
    env.try_execute_tx(tx)
}
