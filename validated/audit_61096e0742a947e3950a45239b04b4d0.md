### Title
Expired Declare Transactions Stranded in `delayed_declares` Block Valid Replacements and Bypass TTL — (`crates/apollo_mempool/src/mempool.rs`)

### Summary
`remove_expired_txs` purges expired transactions from `tx_pool` and `tx_queue` but never touches `delayed_declares`. When `declare_delay ≥ transaction_ttl`, a declare transaction can outlive its TTL while still sitting in `delayed_declares`. Two concrete consequences follow: (1) `validate_no_delayed_declare_front_run` reads the stale entry and rejects every valid replacement at the same `(address, nonce)` with `DuplicateNonce`; (2) when `add_ready_declares` eventually promotes the expired declare into `tx_pool`, `TimedTransactionMap::insert` stamps it with `clock.now()`, silently resetting its submission time and granting it a full second TTL window.

### Finding Description
The `Mempool` struct maintains two independent queues for declare transactions:

- `tx_pool` — backed by `TimedTransactionMap`, which records `submission_time` and is swept by `remove_txs_older_than` on every `add_tx` call.
- `delayed_declares` — a plain `AddTransactionQueue` (`VecDeque`) with no TTL enforcement whatsoever. [1](#0-0) 

`remove_expired_txs` only operates on `tx_pool`: [2](#0-1) 

`delayed_declares` is never consulted or pruned here. The `AddTransactionQueue` has no TTL-aware method: [3](#0-2) 

`add_ready_declares` promotes entries from `delayed_declares` to `tx_pool` based solely on `declare_delay`, with no TTL check: [4](#0-3) 

When it calls `add_tx_inner`, the transaction is inserted into `TimedTransactionMap` with `submission_time = clock.now()`: [5](#0-4) 

This resets the TTL clock. A declare submitted at T=0 with `TTL=60s` and `declare_delay=100s` expires at T=60 but is promoted at T=100 with a fresh `submission_time`, surviving until T=160.

Meanwhile, `validate_no_delayed_declare_front_run` performs a linear scan of `delayed_declares` with no freshness check: [6](#0-5) 

`delayed_declares.contains` has no TTL awareness: [7](#0-6) 

Any incoming transaction at the same `(address, nonce)` as the expired-but-stranded declare is rejected with `DuplicateNonce` for the entire window `[TTL_expiry, declare_delay]`.

`update_accounts_with_gap` also reads `delayed_declares` without a freshness check, causing gap tracking to silently suppress eviction eligibility for the affected account: [8](#0-7) 

### Impact Explanation
**High — Mempool admission rejects valid transactions before sequencing.**

A user whose declare TTL has expired cannot submit any replacement transaction (invoke, deploy-account, or a new declare) at the same nonce until either `declare_delay` elapses and the stale entry is promoted, or the node is restarted. The gateway returns `DuplicateNonce` for every such attempt. This is a liveness failure: a valid, correctly-signed transaction is permanently rejected by the mempool for a duration proportional to `declare_delay − TTL`.

Additionally, the TTL bypass means the mempool retains a transaction for up to `declare_delay + TTL` instead of `TTL`, violating the operator-configured eviction policy.

### Likelihood Explanation
Requires `declare_delay > transaction_ttl` in the node configuration. Both are operator-tunable `Duration` fields with no enforced ordering constraint. The code contains no assertion or validation that `declare_delay < transaction_ttl`. Any deployment that sets a long `declare_delay` (e.g., to slow front-running) relative to a short `transaction_ttl` (e.g., for a high-throughput network) hits this condition. The trigger is unprivileged: any user submitting a declare transaction and then attempting a replacement after TTL expiry.

### Recommendation
**Short term:** Add TTL pruning to `remove_expired_txs` (or a dedicated `remove_expired_delayed_declares`) that iterates `delayed_declares` and drops entries whose `submission_time + transaction_ttl < now`. Return the affected addresses in the `AddressToNonce` map so `update_accounts_with_gap` re-evaluates gap status correctly.

**Long term:** Add a configuration-time assertion `declare_delay < transaction_ttl` and a unit test covering the scenario where a delayed declare's TTL expires before `declare_delay` elapses, verifying that (a) a replacement transaction is accepted and (b) gap tracking reflects the correct state.

### Proof of Concept
```
TTL = 60s, declare_delay = 100s

T=0:   Alice submits Declare(nonce=0) → delayed_declares = [Declare(nonce=0, submitted=0)]
T=0:   Alice submits Invoke(nonce=2)  → tx_pool, gap detected (nonce 2 > account nonce 0)
       update_accounts_with_gap: delayed_declares.contains(alice, 0) == true → gap suppressed

T=65:  TTL expires. remove_expired_txs sweeps tx_pool only.
       delayed_declares still = [Declare(nonce=0, submitted=0)]

T=65:  Alice submits Invoke(nonce=0) to replace the expired declare.
       validate_no_delayed_declare_front_run: delayed_declares.contains(alice, 0) == true
       → Err(DuplicateNonce) — valid transaction rejected.

T=100: add_ready_declares promotes Declare(nonce=0) to tx_pool with submission_time=100.
       Declare now lives until T=160 (TTL bypass: 100s extra lifetime).
```

// A queue to hold transactions that are waiting to be added to the tx pool.
struct AddTransactionQueue {
    elements: VecDeque<(DateTime, AddTransactionArgs)>,
    // Keeps track of the total size of the transactions in this queue.
    size_in_bytes: u64,
}

impl AddTransactionQueue {
    fn new() -> Self {
        AddTransactionQueue { elements: VecDeque::new(), size_in_bytes: 0 }
    }

    fn push_back(&mut self, submission_time: DateTime, args: AddTransactionArgs) {
        self.size_in_bytes = self
            .size_in_bytes
            .checked_add(args.tx.total_bytes())
            .expect("Overflow when adding a transaction to AddTransactionQueue.");
        self.elements.push_back((submission_time, args));
    }

    fn pop_front(&mut self) -> Option<(DateTime, AddTransactionArgs)> {
        let removed_element = self.elements.pop_front();
        if let Some((_, args)) = &removed_element {
            self.size_in_bytes = self
                .size_in_bytes
                .checked_sub(args.tx.total_bytes())
                .expect("Underflow when removing a transaction from AddTransactionQueue.");
        }
        removed_element
    }

    fn front(&self) -> Option<&(DateTime, AddTransactionArgs)> {
        self.elements.front()
    }

    fn contains(&self, contract_address: ContractAddress, nonce: Nonce) -> bool {
        self.elements.iter().any(|(_, tx_args)| {
            let tx = &tx_args.tx;
            tx.contract_address() == contract_address && tx.nonce() == nonce
        })
    }

    fn len(&self) -> usize {
        self.elements.len()
    }

    fn size_in_bytes(&self) -> u64 {
        self.size_in_bytes
    }
}

    fn add_ready_declares(&mut self) {

    fn update_accounts_with_gap(&mut self, address_to_nonce: AddressToNonce) {
        for (address, account_nonce) in address_to_nonce {
            // If a delayed declare transaction exists at the account nonce, it is next to execute,
            // so no gap exists.
            if self.delayed_declares.contains(address, account_nonce) {
                self.remove_from_accounts_with_gap(address);
                continue;
            }

            // Gap exists when lowest transaction nonce is higher than account nonce.
            let gap_exists = match self.tx_pool.get_lowest_nonce(address) {
                Some(lowest_nonce) => account_nonce < lowest_nonce,
                None => false, // No transactions for the account, so no gap.
            };

            // Update the eviction tracking set accordingly.
            if gap_exists {
                if self.accounts_with_gap.insert(address) {
                    // Newly entered gap: all current pool txs for this account are now stuck.
                    let n_stuck = self.tx_pool.n_txs_for_address(address);
                    self.n_stuck_txs += n_stuck;
                    warn!(
                        "Account {address} has a nonce gap; {n_stuck} transaction(s) are now \
                         stuck."
                    );
                }
                // Stayed in gap: per-tx deltas were already applied at add/remove sites.
            } else {
                // Left gap: remaining pool txs for this account are no longer stuck.
                self.remove_from_accounts_with_gap(address);
            }
        }
    }
