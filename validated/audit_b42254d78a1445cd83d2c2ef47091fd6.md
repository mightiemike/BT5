### Title
`op_unknown` Cost Multiplier Multiplication Overflows `u64`, Enabling Undercharged Execution - (File: `src/more_ops.rs`)

### Summary

`op_unknown` in `src/more_ops.rs` computes a final cost by multiplying an accumulated base cost by `(cost_multiplier + 1)`, where both operands are `u64`. No overflow check is applied before this multiplication. In Rust release builds, `u64` overflow wraps silently. An attacker who controls both the opcode bytes (which set `cost_multiplier` up to `u32::MAX`) and the argument list (which drives the accumulated cost) can engineer a wrap-around that produces a final cost of 0 or any small value ≤ `u32::MAX`, causing the VM to accept the operation as nearly free while the actual resource consumption was bounded only by `max_cost`.

### Finding Description

`Cost` is defined as `u64`: [1](#0-0) 

In `op_unknown`, the opcode's leading bytes are decoded into `cost_multiplier` (a `u64` holding a value up to `u32::MAX = 4,294,967,295`): [2](#0-1) 

After the argument-scanning loop, `check_cost` confirms the accumulated `cost` is within `max_cost`, then the unchecked multiplication is performed: [3](#0-2) 

The only guard after the multiplication is `if cost > u32::MAX as u64`. In Rust release mode, `u64 *= u64` wraps on overflow — it does **not** panic and does **not** saturate. If the product exceeds `u64::MAX`, the result is `product mod 2^64`. A wrapped result ≤ `u32::MAX` passes the guard and is returned as the charged cost.

**Concrete overflow path:**

Set `cost_multiplier = u32::MAX` (opcode bytes `[0xff, 0xff, 0xff, 0xff, 0xXX]` where `XX` encodes `cost_function = 1`). With `cost_function = 1`, the accumulated cost is:

```
cost = ARITH_BASE_COST + n × ARITH_COST_PER_ARG + total_bytes × ARITH_COST_PER_BYTE
     = 99 + n × 320 + total_bytes × 3
``` [4](#0-3) 

The attacker chooses `n` and argument sizes so that `cost ≡ 0 (mod 2^32)` (i.e., `cost` is a multiple of `4,294,967,296`). Then:

```
cost × (cost_multiplier + 1)
= cost × 2^32
= (cost mod 2^32) × 2^32   (mod 2^64)
= 0 × 2^32
= 0
```

The function returns `Ok(Reduction(0, nil))` — cost zero — for an operation that consumed up to `max_cost` worth of resources.

Because `cost_function = 1` requires `cost ≥ ARITH_BASE_COST = 99`, and the Chia block cost limit is 11 billion (> `2^32 ≈ 4.3 billion`), the required accumulated cost of `2^32` is reachable when `max_cost` is the full block budget. The specific `n` satisfying `99 + n × 323 ≡ 0 (mod 2^32)` (for 1-byte arguments) is determined by the modular inverse of 323 mod `2^32`, which exists because `gcd(323, 2^32) = 1`.

Unknown ops are reachable in consensus mode (without `NO_UNKNOWN_OPS`): [5](#0-4) 

Mempool mode sets `NO_UNKNOWN_OPS` and blocks this path entirely: [6](#0-5) 

A malicious farmer can bypass the mempool and directly embed the crafted program in a block. Full nodes validate blocks in consensus mode, where `op_unknown` is reachable.

### Impact Explanation

A malicious farmer can include a CLVM program in a block that uses an unknown opcode engineered to wrap the cost counter to 0. Full nodes running in consensus mode accept the block, believing the program consumed 0 cost units. The farmer can repeat this within the same block, effectively bypassing the block cost limit entirely. This allows inclusion of arbitrarily expensive computation, causing full nodes to spend unbounded CPU/memory validating a block they believe is within cost limits — a consensus-critical resource exhaustion and cost-accounting integrity failure.

### Likelihood Explanation

Exploiting this requires a malicious farmer (block producer), which is a realistic attacker role in a proof-of-space blockchain. The farmer needs no privileged keys beyond their farming key. The opcode bytes and argument list are fully attacker-controlled. The arithmetic to find the right `n` is straightforward modular inverse computation. The only constraint is that the program must fit within the block's byte budget, which is separate from the cost budget and is large enough to accommodate the required argument count.

### Recommendation

Replace the bare multiplication at line 261 with a checked or saturating multiply, and treat overflow as an error:

```rust
// Before (vulnerable):
cost *= cost_multiplier + 1;

// After (safe):
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

This mirrors the pattern already used in the fast-path of `op_add` and `op_subtract`, which use `checked_add`/`checked_sub` to avoid silent wrapping. [7](#0-6) 

### Proof of Concept

#   cost_multiplier = u32_from_u8([0xff,0xff,0xff,0xff]) = 4294967295
#   cost_function   = (0x40 >> 6) & 0b11 = 1  (ARITH-like)
#
# cost_multiplier + 1 = 2^32 = 4294967296
#
# We need: 99 + n*320 + n*1*3 = 99 + n*323 ≡ 0 (mod 2^32)
#   => n*323 ≡ 4294967197 (mod 2^32)
#   => n = 4294967197 * modinv(323, 2^32) mod 2^32
#
# modinv(323, 2^32) can be computed; call the result N.
# Construct a CLVM program that calls opcode 0xffffffff40
# with N arguments each being a 1-byte atom.
#
# Accumulated cost = 99 + N*323 = 4294967296 (= 2^32, a multiple of 2^32)
# check_cost(4294967296, max_cost=11000000000) => Ok  (passes)
# cost *= 4294967296  =>  4294967296 * 4294967296 = 2^64 ≡ 0 (mod 2^64)
# cost = 0
# 0 > u32::MAX? No => Ok(Reduction(0, nil))   <-- cost charged: 0
``` [3](#0-2)
