### Title
`WithdrawCollateral` Slow-Mode Path Omits `withdrawFeeX18` Charge Present in Fast Path — (`core/contracts/EndpointTx.sol`)

### Summary

The `processSlowModeTransactionImpl` handler for `TransactionType.WithdrawCollateral` does not charge the product-specific `withdrawFeeX18` that the fast-path handler `processTransactionImpl` always charges. Any user can exploit this by submitting a `WithdrawCollateral` transaction through the slow-mode queue, paying only the flat `SLOW_MODE_FEE` and bypassing the per-product withdrawal fee entirely.

### Finding Description

**Fast path** (`processTransactionImpl`, `WithdrawCollateral` branch):

```solidity
validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce,
    transaction, signedTx.signature, true);
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,  // ← charged
    signedTx.tx.productId
);
clearinghouse.withdrawCollateral(
    signedTx.tx.sender, signedTx.tx.productId,
    signedTx.tx.amount, address(0), nSubmissions
);
``` [1](#0-0) 

**Slow-mode path** (`processSlowModeTransactionImpl`, `WithdrawCollateral` branch):

```solidity
validateSender(txn.sender, sender);
clearinghouse.withdrawCollateral(          // ← no withdrawFeeX18 charged
    txn.sender, txn.productId,
    txn.amount, address(0), nSubmissions
);
``` [2](#0-1) 

The slow-mode submission path (`submitSlowModeTransactionImpl`) only charges the flat `SLOW_MODE_FEE` upfront from the user's wallet; `WithdrawCollateral` is not in the owner-only list, so it falls into the `else` branch:

```solidity
} else {
    chargeSlowModeFee(_getQuote(), sender);
    slowModeFees += SLOW_MODE_FEE;
}
``` [3](#0-2) 

The `withdrawFeeX18` is a per-product flat fee stored in the spot engine config and credited to `sequencerFee`. It is never charged on the slow-mode path, creating a structural inconsistency between the two withdrawal code paths.

### Impact Explanation

A user who withdraws via slow mode pays only `SLOW_MODE_FEE` (a fixed quote-token amount taken from their wallet) and avoids the `withdrawFeeX18` that would otherwise be deducted from their subaccount balance in the product token. For any product where `withdrawFeeX18 > 0`, this is a direct fee bypass. The sequencer loses the fee revenue that the protocol intends to collect on every withdrawal. The corrupted state is `sequencerFee[productId]` — it is never incremented for slow-mode withdrawals, understating the sequencer's earned fees.

**Impact: Medium** — direct protocol fee revenue loss; no user funds at risk beyond the fee amount.

### Likelihood Explanation

Any unprivileged user can call `submitSlowModeTransaction` with a `WithdrawCollateral` payload. The only cost is the flat `SLOW_MODE_FEE` and a 3-day delay. For large withdrawals on products with a non-trivial `withdrawFeeX18`, the bypass is economically rational. The entry path requires no special privileges, no leaked keys, and no sequencer cooperation.

**Likelihood: Medium** — requires awareness of the fee difference and willingness to wait 3 days; economically rational for large withdrawals.

### Recommendation

Add the `withdrawFeeX18` charge to the slow-mode `WithdrawCollateral` handler in `processSlowModeTransactionImpl`, mirroring the fast-path logic:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:], (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
    // Add this:
    chargeFee(
        txn.sender,
        spotEngine.getConfig(txn.productId).withdrawFeeX18,
        txn.productId
    );
    clearinghouse.withdrawCollateral(
        txn.sender, txn.productId, txn.amount, address(0), nSubmissions
    );
```

Alternatively, explicitly document that slow-mode withdrawals are fee-exempt and ensure this is an intentional protocol design decision.

### Proof of Concept

1. Suppose `withdrawFeeX18` for product `P` is `F > 0` (e.g., 1 USDC equivalent in the product token).
2. User calls `Endpoint.submitSlowModeTransaction(abi.encodePacked(uint8(TransactionType.WithdrawCollateral), abi.encode(WithdrawCollateral({sender: subaccount, productId: P, amount: A}))))`.
3. `submitSlowModeTransactionImpl` charges only `SLOW_MODE_FEE` from the user's wallet and enqueues the transaction.
4. After 3 days, the user (or anyone) calls `executeSlowModeTransaction()`.
5. `processSlowModeTransactionImpl` executes the withdrawal: `clearinghouse.withdrawCollateral(...)` is called with no prior `chargeFee(sender, F, P)`.
6. The user receives the full withdrawal amount; `sequencerFee[P]` is not incremented by `F`.
7. Compare with the fast path: the sequencer would have submitted a `WithdrawCollateral` transaction that charges `F` before calling `clearinghouse.withdrawCollateral`. [2](#0-1) [1](#0-0) [4](#0-3)
