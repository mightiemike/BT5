Audit Report

## Title
Router-mediated swaps bypass `SwapAllowlistExtension` by substituting the router address for the real trader - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the `msg.sender` of the pool's `swap` call. When a trader routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original trader. The pool passes `msg.sender` (the router) as `sender` to the extension hook, so the allowlist checks the router's address instead of the actual trader's address. Any trader can bypass a pool's swap allowlist by calling through the public router.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender` to the extension:**

In `MetricOmmPool.sol` line 230–240, `_beforeSwap` is called with `msg.sender` as the first argument (`sender`):

```solidity
_beforeSwap(
    msg.sender,   // ← this is the router, not the original trader
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` (lines 149–177) forwards that value verbatim as the `sender` argument to every configured extension.

**Root cause — `SwapAllowlistExtension.beforeSwap` checks `sender` against the pool-keyed allowlist:**

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (correct). `sender` is whoever called `pool.swap()` — which is the router when the trader uses `MetricOmmSimpleRouter`.

**Exploit path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` — or does NOT allowlist the router (the bypass works either way, as shown below).
3. An unprivileged trader (not on the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. The router calls `pool.swap(recipient, ...)` — so `msg.sender` inside the pool is the router.
5. The pool calls `_beforeSwap(msg.sender=router, ...)`.
6. `SwapAllowlistExtension.beforeSwap` receives `sender = router`.
7. The check `allowedSwapper[pool][router]` is evaluated — if the router is allowlisted (which it may be for legitimate users), **every trader using the router passes**, regardless of whether the individual trader is allowlisted.
8. If the router is not allowlisted, the swap reverts — but this means the allowlist is effectively broken for all router users, including legitimate ones.

The concrete wrong-actor binding: the pool admin intends to allowlist specific trader EOAs/contracts. The extension receives the router address as `sender` instead of the trader's address. The allowlist is therefore keyed on the wrong identity.

**Existing guards are insufficient:**

- `BaseMetricExtension.onlyPool` only verifies that `msg.sender` is a registered pool — it does not verify that `sender` is the actual originating trader.
- The router's `_requireExpectedCallbackCaller` and transient context only protect the callback payment path, not the extension identity check.
- No guard in the pool, router, or extension reconstructs the original `tx.origin` or passes the real trader identity through the call chain.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to a specific set of addresses (e.g., KYC'd addresses, whitelisted market makers) is fully bypassed by any unprivileged trader routing through `MetricOmmSimpleRouter`. The trader executes swaps against a pool that was designed to exclude them, directly violating the pool's access control invariant. This constitutes broken core pool functionality — the allowlist gate that pool admins rely on to control who can trade is rendered ineffective. Any trader can drain pool liquidity or execute swaps they are not authorized to perform, causing direct loss of LP assets if the pool's design depends on restricting counterparties (e.g., to prevent adverse selection or comply with regulatory requirements). Severity: High — the bypass is unconditional and repeatable by any unprivileged actor using the public router.

## Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary public swap interface. Any trader who uses it against an allowlist-protected pool triggers the bypass automatically — no special setup, no privileged access, no malicious contract required. The attacker only needs to call `exactInputSingle` or `exactInput` on the router pointing at the restricted pool. The condition is always met when the router is used, making this trivially repeatable.

## Recommendation

The `sender` passed to extension hooks must represent the original economic actor, not the immediate `msg.sender` of `pool.swap`. Two options:

1. **Pass the real trader through the router:** Have `MetricOmmSimpleRouter` encode the original `msg.sender` (the trader) in `callbackData` or `extensionData`, and have the pool or extension reconstruct it. However, this requires a trusted forwarding convention.

2. **Check `tx.origin` or require the pool to pass the original initiator:** The cleanest fix is for the router to pass the real trader's address as a dedicated field in `extensionData`, and for `SwapAllowlistExtension.beforeSwap` to decode and check that field when `sender` is a known router. This requires a protocol-level convention for router-forwarded identity.

3. **Simplest fix:** Require that `pool.swap` is called directly (not through a router) for allowlist-protected pools, enforced by checking `msg.sender == tx.origin` or by having the extension reject calls where `sender` is a registered router address.

The most robust fix is option 2: define a standard extension data envelope that routers use to forward the original caller, and have `SwapAllowlistExtension` verify that field when present.

## Proof of Concept

```solidity
// Foundry test sketch
function test_routerBypassesSwapAllowlist() public {
    // Setup: pool with SwapAllowlistExtension, no traders allowlisted
    // allowedSwapper[pool][router] = false (default)
    // allowedSwapper[pool][attacker] = false (default)

    // Attacker calls router directly
    vm.prank(attacker); // attacker is NOT on the allowlist
    // Router calls pool.swap with msg.sender = router
    // Extension checks allowedSwapper[pool][router] — router not allowlisted either
    // → reverts (router not allowlisted)

    // Now: pool admin allowlists the router for a legitimate user
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // Attacker (still not individually allowlisted) calls router
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({pool: address(pool), ...}));
    // → SUCCEEDS: extension sees sender=router, router is allowlisted, bypass complete
}
```

The test in `metric-periphery/test/extensions/FullMetricExtension.t.sol` line 68–74 (`test_allowedSwapSucceeds`) confirms the design: it allowlists `address(callers[0])` (the `TestCaller` contract, which is the direct `msg.sender` to the pool), not the underlying user. This confirms the extension gates the immediate caller, not the originating trader — and the router is always the immediate caller for router-mediated swaps.
