### Title
Unbounded `slowModeTxs` Queue Inflation via Fee-Exempt `depositCollateralWithReferral` — (File: `core/contracts/Endpoint.sol`)

---

### Summary

The `slowModeTxs` queue has two insertion paths. The standard path (`submitSlowModeTransactionImpl`) charges a `SLOW_MODE_FEE` of $1 per entry as a rate-limiting mechanism. The deposit path (`depositCollateralWithReferral`) inserts into the same queue with no fee and no queue-size cap. An unprivileged attacker can flood the queue at a cost of $0.10 per entry (the minimum deposit, which they retain), indefinitely delaying all subsequent legitimate slow mode transactions — including `WithdrawCollateral` — and breaking the protocol's censorship-resistance guarantee.

---

### Finding Description

`submitSlowModeTransactionImpl` in `EndpointTx.sol` enforces a `SLOW_MODE_FEE` before enqueuing: [1](#0-0) 

```solidity
} else {
    chargeSlowModeFee(_getQuote(), sender);

}
// ...
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({...});
```

`SLOW_MODE_FEE` is $1: [2](#0-1) 

`depositCollateralWithReferral` in `Endpoint.sol` inserts into the **same** `slowModeTxs` mapping with **no fee and no size check**: [3](#0-2) 

```solidity
SlowModeConfig memory _slowModeConfig = slowModeConfig;
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: abi.encodePacked(...)
});
slowModeConfig = _slowModeConfig;
```

The queue is backed by an unbounded `mapping(uint64 => IEndpoint.SlowModeTx)` with a `uint64` counter: [4](#0-3) 

Execution is strictly FIFO: `_executeSlowModeTransaction` always pops `slowModeTxs[txUpTo]` and increments `txUpTo`: [5](#0-4) 

There is no eviction, no cap, and no mechanism to skip attacker-inserted entries. Any transaction submitted after the flood sits behind every attacker entry in the queue.

The minimum deposit for an existing subaccount is $0.10: [6](#0-5) 

The deposited funds are credited to the attacker's own subaccount — they are not lost. The net cost per queue entry is gas only.

---

### Impact Explanation

The slow mode queue is the protocol's sole censorship-resistance path. When the sequencer is unresponsive or censoring, users rely on `submitSlowModeTransaction` → `WithdrawCollateral` to recover their funds. If the queue is flooded with attacker-inserted `DepositCollateral` entries, the FIFO pointer (`txUpTo`) cannot advance past them until every attacker entry is processed. Legitimate `WithdrawCollateral` transactions submitted after the flood are delayed indefinitely, effectively locking user funds for as long as the attacker maintains the flood. This is a concrete, user-facing asset-access impact, not merely a gas or performance degradation.

---

### Likelihood Explanation

The entry point is `depositCollateralWithReferral`, a `public` function callable by any address without privilege. The attacker needs one initial deposit of $5 to register a subaccount, then can enqueue entries at $0.10 each (retained in their account). The cost to queue 10,000 entries is approximately $1,000 in deposited collateral (fully recoverable) plus gas. The attack is permissionless, repeatable, and economically rational for any actor wishing to censor a specific user's withdrawal.

---

### Recommendation

Apply the same `SLOW_MODE_FEE` charge inside `depositCollateralWithReferral` before enqueuing the slow mode transaction, consistent with `submitSlowModeTransactionImpl`. Alternatively, introduce a per-address or global cap on pending slow mode queue entries, analogous to the mempool `canAddTx` size check described in the reference report.

---

### Proof of Concept

1. Attacker calls `depositCollateral(subaccountName, productId, 5e6)` once to register a subaccount (cost: $5, retained).
2. Attacker calls `depositCollateralWithReferral(attackerSubaccount, productId, 1e5, "")` in a loop N times (cost: $0.10 × N, all retained in attacker's subaccount). Each call pushes one `DepositCollateral` entry into `slowModeTxs` at indices `[txCount, txCount+N)`.
3. Victim calls `submitSlowModeTransaction(withdrawTx)`. This charges $1 and enqueues the victim's `WithdrawCollateral` at index `txCount+N`.
4. The sequencer (or anyone calling `executeSlowModeTransaction`) must advance `txUpTo` through all N attacker entries before reaching the victim's entry.
5. With N = 100,000 entries and one execution per sequencer batch, the victim's withdrawal is delayed by 100,000 batch cycles. The attacker can continuously extend the flood to maintain the delay indefinitely.
