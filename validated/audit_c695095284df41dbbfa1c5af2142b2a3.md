### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Accepts Forged Merkle Proofs for Non-Existent Transactions — (`contract/src/lib.rs`)

---

### Summary

The function `verify_transaction_inclusion` is marked `#[deprecated]` but remains a live, publicly callable NEAR entry point. It accepts a fully user-controlled `tx_id` without verifying that it corresponds to a leaf node (real transaction) in the Merkle tree. An attacker can supply an internal Merkle tree node hash as `tx_id` together with a crafted proof that correctly reconstructs the block's `merkle_root`, causing the function to return `true` for a Bitcoin transaction that never existed.

---

### Finding Description

`verify_transaction_inclusion` in `contract/src/lib.rs` performs the following check:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

`args.tx_id`, `args.tx_index`, and `args.merkle_proof` are entirely user-supplied. There is no check that `tx_id` is a leaf node (an actual transaction hash) rather than an internal node of the Merkle tree. The code's own `# Warning` block acknowledges this:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [2](#0-1) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` simply walks the proof path, hashing left or right depending on position parity, with no leaf-vs-internal-node distinction:

```rust
for proof_hash in merkle_proof {
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
``` [3](#0-2) 

Despite being deprecated, the function carries `#[pause]` (not `#[private]`), meaning it is callable by any unprivileged NEAR account: [4](#0-3) 

The fixed version `verify_transaction_inclusion_v2` adds a coinbase proof check at position 0 to constrain the tree depth, but it does not remove or gate the v1 entry point: [5](#0-4) 

---

### Impact Explanation

Any consumer contract (atomic swap, bridged-asset protocol, cross-chain state verifier) that calls `verify_transaction_inclusion` to confirm a Bitcoin payment or deposit will accept a forged proof. An attacker can claim funds for a Bitcoin transaction that never occurred, leading to complete loss of assets locked in the consumer contract. The contract's own documentation identifies this as the 64-byte transaction Merkle proof forgery vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions).

---

### Likelihood Explanation

The attack requires zero brute-force computation. Given any Bitcoin block already stored in the contract, the attacker reads the block's public Merkle tree structure, computes an internal node hash, and constructs a one-step proof. The exploit is deterministic and requires only a standard NEAR function call. Any consumer that has not migrated to `verify_transaction_inclusion_v2` is immediately exploitable.

---

### Recommendation

- Remove `verify_transaction_inclusion` entirely, or restrict it with `#[private]` to prevent external calls.
- Alternatively, add a `require!` that the proof length equals `floor(log2(tx_count))` and that `tx_index < tx_count`, forcing the caller to prove knowledge of the full leaf set — though removal is the safest option.
- Ensure all consumer contracts are migrated to `verify_transaction_inclusion_v2`.

---

### Proof of Concept

Consider a Bitcoin block stored in the contract with four transactions `[T0, T1, T2, T3]` and Merkle tree:

```
        R
       / \
     N01  N23
     / \  / \
    T0 T1 T2 T3
```

where `N01 = SHA256d(T0 || T1)`, `N23 = SHA256d(T2 || T3)`, `R = SHA256d(N01 || N23)`.

**Attack call:**

```
verify_transaction_inclusion({
    tx_id:             N01,   // internal node, not a real transaction
    tx_block_blockhash: <real block hash stored in contract>,
    tx_index:          0,
    merkle_proof:      [N23],
    confirmations:     1,
})
```

**Execution trace in `compute_root_from_merkle_proof`:**

- `current_hash = N01`, `current_position = 0` (even)
- Iteration 1: `current_hash = SHA256d(N01 || N23) = R`
- Returns `R == header.block_header.merkle_root` → **`true`**

The function confirms a transaction `N01` that does not exist on the Bitcoin blockchain. A consumer contract paying out on this result would release funds to the attacker. [6](#0-5) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
