### Title
`ENABLE_SECP_OPS` Flag Not Enforced on 4-Byte Secp Opcode Dispatch Path, Allowing Secp Verification to Execute Without Activation - (File: `src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` has two separate dispatch paths for secp verification operators: a 1-byte opcode path (opcodes 64 and 65) and a 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`). The `ENABLE_SECP_OPS` flag is checked only on the 1-byte path. The 4-byte path dispatches `op_secp256k1_verify` and `op_secp256r1_verify` unconditionally, regardless of whether `ENABLE_SECP_OPS` is set. An attacker can craft CLVM bytes using the 4-byte secp opcodes to invoke secp signature verification in any execution context, including before the soft-fork activates and in mempool mode, bypassing the intended activation gate.

---

### Finding Description

In `src/chia_dialect.rs`, `ChiaDialect::op()` handles operator dispatch in two branches based on opcode byte length.

**4-byte opcode path (lines 157–182):** The secp operators are dispatched unconditionally when the opcode matches `0x13d61f00` or `0x1c3a8f00`:

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

No `ENABLE_SECP_OPS` check is present here.

**1-byte opcode path (lines 248–249):** The same operators are gated by the flag:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The flag is documented as "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)" and is absent from `MEMPOOL_MODE` (lines 72–76), meaning the intended design is that secp ops are disabled unless the flag is explicitly set. The 4-byte path completely bypasses this invariant.

The attacker-controlled entry path is direct: submit a CLVM program whose operator atom is the 4-byte encoding `0x13d61f00` (secp256k1) or `0x1c3a8f00` (secp256r1). The `op_len == 4` branch is entered, the opcode matches, and `op_secp256k1_verify` / `op_secp256r1_verify` executes regardless of the `ENABLE_SECP_OPS` flag state.

---

### Impact Explanation

**Consensus divergence / soft-fork bypass:** `ENABLE_SECP_OPS` is a soft-fork activation flag. Before activation, nodes without the flag treat 1-byte secp opcodes as unknown (no-ops in consensus mode, errors in mempool mode). But the 4-byte secp opcodes execute unconditionally on all nodes. A puzzle crafted with 4-byte secp opcodes will perform real secp signature verification even on pre-activation nodes, producing a different result than the intended "unknown op returns nil" behavior. This creates a consensus split between nodes that have activated the soft-fork and those that have not.

**Mempool bypass:** `MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`. A mempool node running without `ENABLE_SECP_OPS` will reject a program using 1-byte opcode 64/65 (as unimplemented, due to `NO_UNKNOWN_OPS`), but will silently execute and accept the same secp verification expressed via the 4-byte opcode. This allows an attacker to submit transactions that pass mempool validation under semantics that differ from consensus, undermining the mempool's role as a pre-filter.

---

### Likelihood Explanation

The 4-byte secp opcode values (`0x13d61f00`, `0x1c3a8f00`) are documented in the source code comments and are part of the public CLVM specification. Any attacker who reads the source or the Chia protocol documentation can construct a CLVM program using these opcodes. No privileged access is required — only the ability to submit a CLVM program (e.g., a coin puzzle) to the network.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that both opcode encodings for secp verification are uniformly gated by the activation flag, eliminating the inconsistency.
