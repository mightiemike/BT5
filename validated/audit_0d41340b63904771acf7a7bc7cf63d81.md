### Title
`starknet_estimateFee` Returns Unverifiable `overall_fee` With Incorrect Formula — Missing `l2_gas_consumed` Field - (File: crates/apollo_rpc_execution/src/objects.rs)

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` includes `l2_gas_price` but omits `l2_gas_consumed`. The `overall_fee` field is computed by the blockifier and includes L2 gas costs, but the struct's own docstring declares an incorrect formula (`gas_consumed * gas_price + data_gas_consumed * data_gas_price`) that omits the L2 gas term. A caller who trusts the documented formula to predict or verify the fee will compute a value that diverges from the actual charged fee for any post-0.13.3 transaction that consumes L2 gas.

### Finding Description

In `crates/apollo_rpc_execution/src/objects.rs`, the `FeeEstimation` struct is defined as:

```rust
pub struct FeeEstimation {
    pub gas_consumed: Felt,       // L1 gas
    pub l1_gas_price: GasPrice,
    pub data_gas_consumed: Felt,  // L1 data gas
    pub l1_data_gas_price: GasPrice,
    // TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
    // close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
    pub l2_gas_price: GasPrice,   // present, but l2_gas_consumed is absent
    /// The total amount of fee. This is equal to:
    /// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
    pub overall_fee: Fee,
    pub unit: PriceUnit,
}
``` [1](#0-0) 

The `tx_execution_output_to_fee_estimation` function populates this struct. It reads `l2_gas_price` from the block context but takes `overall_fee` directly from `tx_execution_output.execution_info.receipt.fee`, which is the blockifier's actual computed fee and **does** include the L2 gas component (`l2_gas_consumed × l2_gas_price`):

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l1_data_gas_price,
    l2_gas_price,
    overall_fee: tx_execution_output.execution_info.receipt.fee,  // includes L2 gas
    unit: tx_execution_output.price_unit,
    // l2_gas_consumed: gas_vector.l2_gas — NOT PRESENT
})
``` [2](#0-1) 

The `gas_vector` that is read contains `l2_gas` but it is never placed into the response:

```rust
let gas_vector = tx_execution_output.execution_info.receipt.gas;
// gas_vector.l2_gas is silently dropped
``` [3](#0-2) 

The OpenAPI spec for `FEE_ESTIMATE` in `crates/apollo_rpc/resources/V0_8/starknet_api_openrpc.json` also describes `overall_fee` as `gas_consumed*gas_price + data_gas_consumed*data_gas_price`, with no mention of L2 gas, and does not include an `l2_gas_consumed` field in the required properties: [4](#0-3) 

The internal TODO comment at line 104–105 explicitly acknowledges the gap and notes that the approximation may not hold:

> `TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.` [5](#0-4) 

### Impact Explanation

Any caller of `starknet_estimateFee` or `starknet_simulateTransactions` who uses the documented formula to predict the fee before submitting a transaction will compute:

```
predicted_fee = gas_consumed × l1_gas_price + data_gas_consumed × l1_data_gas_price
```

But the actual charged fee is:

```
actual_fee = gas_consumed × l1_gas_price
           + data_gas_consumed × l1_data_gas_price
           + l2_gas_consumed × l2_gas_price   ← missing from response
```

For post-0.13.3 transactions (which use `AllResources` bounds and consume L2 gas), the two values diverge. The caller receives an `overall_fee` that is correct but cannot be verified or reproduced from the other fields in the response. This is an authoritative-looking wrong value: the struct's own docstring asserts a formula that is false for the current protocol version.

This matches the allowed impact: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

All `AllResources` (V3) transactions on Starknet post-0.13.3 consume L2 gas. The test fixtures in `crates/starknet_transaction_prover/resources/rpc_records/` confirm that production responses already include non-zero `l2_gas_consumed` and `l2_gas_price` values (e.g., `l2_gas_consumed: 0xb56b6`, `l2_gas_price: 0x1dcd65000`), meaning the divergence is present on every modern transaction. [6](#0-5) 

### Recommendation

1. Add `l2_gas_consumed: Felt` to the `FeeEstimation` struct and populate it from `gas_vector.l2_gas.0.into()` in `tx_execution_output_to_fee_estimation`.
2. Update the `overall_fee` docstring to reflect the correct three-term formula.
3. Update the OpenAPI schema `FEE_ESTIMATE` to include `l2_gas_consumed` as a required field.
4. Remove the TODO comment once the fix is in place.

### Proof of Concept

Given a V3 invoke transaction on a post-0.13.3 block:

1. Call `starknet_estimateFee`. The response contains:
   - `gas_consumed = 0x0`
   - `l1_gas_price = 0xe8d4a51000`
   - `data_gas_consumed = 0x80`
   - `l1_data_gas_price = 0x3e8`
   - `l2_gas_price = 0x1dcd65000`
   - `overall_fee = 0x151eb86f3ed400`
   - *(no `l2_gas_consumed`)*

2. Apply the documented formula:
   ```
   predicted = 0x0 × 0xe8d4a51000 + 0x80 × 0x3e8 = 0x1f400
   ```

3. Compare to `overall_fee = 0x151eb86f3ed400`.

The values differ by the L2 gas component (`l2_gas_consumed × l2_gas_price`), which is not recoverable from the response. The caller cannot determine the correct fee, cannot verify the `overall_fee`, and cannot set resource bounds accurately — a direct sequencer-native analog to the TeleporterMessenger unpredictable-fee bug. [7](#0-6)

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
