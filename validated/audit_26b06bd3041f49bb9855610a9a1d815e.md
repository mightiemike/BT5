### Title
SwapAllowlistExtension checks the router's address as `sender`, allowing any user to bypass the swap allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the **direct caller of `pool.swap()`**. When a user swaps through `MetricOmmSimpleRouter`, the pool sees `msg.sender = router` and passes `sender = router` to the extension. If the pool admin allowlists the router to enable router-mediated swaps for legitimate users, any unprivileged user can bypass the allowlist entirely by routing through the router.

### Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted for the calling pool (`msg.sender` inside the extension is the pool): [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no mechanism to forward the original user's address: [4](#0-3) 

The same pattern holds for `exactInput` (all hops) and `exactOutputSingle`/`exactOutput`: [5](#0-4) 

Because the router is the direct caller of `pool.swap()`, the extension always sees `sender = router`, never the originating user. This creates an irreconcilable dilemma for any pool admin who wants to run a curated pool and also support the canonical router:

- **Router not allowlisted**: allowlisted users cannot use the router at all — broken core swap flow.
- **Router allowlisted**: the extension cannot distinguish between different users routing through the router, so **any** user can bypass the allowlist by calling `router.exactInputSingle(...)`.

The pool admin's natural mental model — "I'll allowlist the router so my approved users can use it" — silently opens the gate to every user on-chain.

### Impact Explanation
The pool admin's configured swap allowlist is bypassed by an unprivileged path. Any user can trade on a curated pool that was intended to be restricted to specific counterparties (e.g., KYC-gated, institutional-only, or MEV-protected pools). This constitutes an admin-boundary break: a factory-registered access-control guard is rendered ineffective by a supported periphery contract.

### Likelihood Explanation
Medium. The bypass requires the pool admin to allowlist the router, which is the natural step any admin would take when deploying a curated pool that is also expected to be usable through the protocol's own router. The router is a first-party, factory-aware periphery contract, so allowlisting it is a foreseeable and common configuration.

### Recommendation
The `SwapAllowlistExtension` must gate on the **original user**, not the direct pool caller. Two viable approaches:

1. **Extension-data forwarding**: require the router to encode the originating user in `extensionData` and have the extension decode and check that address. The extension should reject calls where `sender` is a known router but no valid user address is forwarded.
2. **Pool-level originator field**: add an originator address to the `beforeSwap` hook signature so the pool can propagate the true initiator through the router's callback context (already tracked in transient storage via `_getPayer()`).

Until fixed, pool admins should **not** allowlist the router on any pool that uses `SwapAllowlistExtension` with a non-trivial allowlist.

### Proof of Concept

```
Setup
─────
1. Deploy pool with SwapAllowlistExtension configured.
2. Pool admin allowlists user1 (legitimate trader) and the router
   (to let user1 use the router):
     allowedSwapper[pool][user1]  = true
     allowedSwapper[pool][router] = true

Attack
──────
3. user2 (not on the allowlist) calls:
     router.exactInputSingle({pool: pool, ...})
   → router calls pool.swap(recipient, ...)
   → pool calls _beforeSwap(msg.sender=router, ...)
   → extension checks allowedSwapper[pool][router] → true
   → swap executes successfully for user2

Result
──────
user2 swaps on a pool they were never authorized to access.
The allowlist is fully bypassed without any privileged action.
