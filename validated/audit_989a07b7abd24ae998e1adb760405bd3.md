Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual caller/payer) and gates only on `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` supports an operator pattern where `msg.sender` (payer) need not equal `owner` (position recipient), any non-allowlisted address can deposit into a restricted pool by naming any allowlisted address as `owner`, completely bypassing the access gate.

## Finding Description
In `DepositAllowlistExtension.sol` (L32–42), the hook signature receives `sender` as the first argument but leaves it unnamed and unused, checking only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
```

In `MetricOmmPool.addLiquidity` (L191), the pool calls `_beforeAddLiquidity(msg.sender, owner, ...)`, where `msg.sender` is the actual caller/payer and `owner` is the position recipient. These two values are explicitly decoupled by the operator pattern.

The check `allowedDepositor[msg.sender][owner]` (where `msg.sender` here is the pool address) evaluates the allowlist status of `owner`, not the actual depositor `sender`. A non-allowlisted `alice` calls `pool.addLiquidity(bob, salt, deltas, ...)` where `bob` is allowlisted. The extension receives `(alice, bob, ...)`, checks `allowedDepositor[pool][bob]` → `true` → no revert. Alice's callback pays the tokens; Bob receives the LP position. Bob can then call `removeLiquidity` (which enforces `msg.sender == owner` at L206) to withdraw the tokens.

By contrast, `SwapAllowlistExtension.beforeSwap` (L31–37) correctly checks `sender` (the actual swapper), not the recipient, making the inconsistency clear.

## Impact Explanation
The deposit allowlist — a pool admin's primary mechanism to restrict who can provide liquidity — is rendered completely ineffective. Any unprivileged address can inject liquidity into a restricted pool by naming any allowlisted address as `owner`. The allowlisted owner can subsequently withdraw the deposited tokens via `removeLiquidity`, making this a viable fund-extraction vector against the depositor (griefing) or a mechanism to force unwanted LP positions onto allowlisted addresses. This constitutes broken core pool access-control functionality with direct fund impact.

## Likelihood Explanation
The operator pattern (`sender != owner`) is a first-class, documented feature of the pool. `MetricOmmPoolLiquidityAdder` itself calls `pool.addLiquidity(owner, ...)` where the pool's `msg.sender` is the adder contract, not the user — demonstrating this flow is routine and expected. Any address that knows a single allowlisted address (which may be publicly observable on-chain) can exploit this without any privileged access, repeatedly and at will.

## Recommendation
Mirror the pattern used in `SwapAllowlistExtension`: check `sender` (the actual depositor/payer), not `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension` attached to `beforeAddLiquidity`.
2. Admin calls `setAllowedToDeposit(pool, bob, true)`. Alice is **not** allowlisted.
3. Alice calls `pool.addLiquidity(bob /*owner*/, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(alice /*sender*/, bob /*owner*/, ...)` → extension receives `(alice, bob, ...)`.
5. Extension evaluates `allowedDepositor[pool][bob]` → `true` → no revert.
6. Pool proceeds; Alice's `metricOmmModifyLiquidityCallback` pays tokens; Bob receives the LP position.
7. Bob calls `pool.removeLiquidity(bob, salt, deltas, "")` — passes `msg.sender == owner` check — and withdraws Alice's tokens.
8. Alice has deposited into a restricted pool without being on the allowlist; Bob has extracted her funds.
