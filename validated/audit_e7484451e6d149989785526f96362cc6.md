### Title
Unbounded Growth of BLS Validation Caches Across Softfork Guard Boundaries — (File: `src/allocator.rs`)

---

### Summary

The `Allocator` struct maintains two `HashSet` caches — `validated_g1_points` and `validated_g2_points` — that accumulate entries across softfork guard boundaries without ever being cleared during a `restore_checkpoint()` call. An attacker-controlled CLVM program can exploit this by repeatedly entering and exiting softfork guards, each time inserting unique BLS point entries into the caches. The allocator heap is freed on each guard exit, but the validation caches grow monotonically for the entire duration of `run_program()`, bypassing the heap limit and causing unbounded memory accumulation proportional to total BLS cost consumed.

---

### Finding Description

The `Allocator` struct declares two grow-only caches:

```rust
validated_g1_points: HashSet<[u8; 48]>,
validated_g2_points: HashSet<[u8; 96]>,
``` [1](#0-0) 

These caches are populated by four paths:

1. `validate_g1()` / `validate_g2()` — called from `op_bls_g1_negate` and `op_bls_g2_negate` to avoid re-validating the same point.
2. `add_validated_g1()` / `add_validated_g2()` — called after negation to mark the negated point as valid.
3. `new_g1()` / `new_g2()` — called from every BLS operator that produces a G1/G2 result (`g1_subtract`, `g1_multiply`, `g1_map`, `g2_add`, `g2_subtract`, `g2_multiply`, `g2_map`). [2](#0-1) [3](#0-2) 

The `restore_checkpoint()` function — which is called by `exit_guard()` every time a softfork guard completes — restores `u8_vec`, `pair_vec`, `atom_vec`, and ghost counters, but **does not touch `validated_g1_points` or `validated_g2_points`**:

```rust
pub fn restore_checkpoint(&mut self, cp: &Checkpoint) {
    self.restore_transparent_checkpoint(&cp.inner);
    self.ghost_atoms = cp.ghost_atoms;
    self.ghost_pairs = cp.ghost_pairs;
    self.ghost_heap = cp.ghost_heap;
}
``` [4](#0-3) 

The `Checkpoint` struct itself has no field for the validation cache state, so there is no mechanism to restore it: [5](#0-4) 

`clear_validation_caches()` is called exactly once — at the very end of `run_program()`, after the program has already completed:

```rust
self.allocator.clear_validation_caches();
Ok(Reduction(cost, self.pop()?))
``` [6](#0-5) 

The softfork guard mechanism in `exit_guard()` calls `restore_checkpoint()` to free heap allocations, but the validation caches survive intact: [7](#0-6) 

---

### Impact Explanation

An attacker-controlled CLVM program can craft a sequence of softfork guards, each invoking BLS operators on distinct G1/G2 points. On each guard exit, the allocator heap is freed (the intended behavior), but every unique BLS point validated or produced during that guard remains permanently in `validated_g1_points` / `validated_g2_points`. Across N sequential softfork guards, the cache accumulates entries from all N guards simultaneously, while the heap only reflects the current guard's allocations.

Each G1 entry consumes 48 bytes of raw data plus `HashSet` overhead (~80–100 bytes per entry). Each G2 entry consumes 96 bytes plus overhead. With Chia's block cost limit (~11 billion), and a per-operation cost of ~1,000–2,000 units for BLS operations, an attacker can force millions of unique cache entries, potentially consuming hundreds of megabytes to gigabytes of process memory that the heap limit does not account for and cannot reclaim until `run_program()` returns.

This is a **cache/allocator aliasing** class bug: the heap and the validation caches are logically coupled (both represent allocator state), but only the heap is restored on checkpoint rollback, leaving the caches permanently inflated.

---

### Likelihood Explanation

Any Chia transaction submitter can craft a CLVM puzzle that uses the `softfork` operator with BLS extension operators. The `softfork` operator is a standard, documented part of the CLVM dialect. No special privilege is required. The attacker only needs to submit a transaction to the mempool; full nodes processing the block will execute the program and accumulate the cache. The softfork stack depth is limited to 20 concurrent guards by `LIMIT_SOFTFORK`, but sequential (non-nested) softfork guards are unlimited, so the attacker can chain as many as the cost budget allows. [8](#0-7) 

---

### Recommendation

Include the validation cache state in the `Checkpoint` struct (e.g., record the cache size at checkpoint time), and truncate the caches back to that size on `restore_checkpoint()`. Alternatively, clear `validated_g1_points` and `validated_g2_points` inside `restore_checkpoint()` / `restore_transparent_checkpoint()` whenever a softfork guard exits, since the points validated inside a guard cannot be reused outside it (the corresponding atom bytes are freed from the heap).

---

### Proof of Concept

```clvm
; Pseudocode: chain N softfork guards, each validating K unique G1 points
(softfork (q . COST_1) (q . 1) (q . (
    (g1_negate (q . <unique_g1_point_1>))
    (g1_negate (q . <unique_g1_point_2>))
    ...
    (g1_negate (q . <unique_g1_point_K>))
)) (q . ()))
(softfork (q . COST_2) (q . 1) (q . (
    (g1_negate (q . <unique_g1_point_K+1>))
    ...
)) (q . ()))
; repeat...
```

After each `softfork` guard exits:
- `restore_checkpoint()` is called → `u8_vec`, `pair_vec`, `atom_vec` are truncated (heap freed).
- `validated_g1_points` is **not** cleared → all K entries from that guard remain.

After N guards, `validated_g1_points` holds N×K entries. The heap reports near-zero usage, but the process RSS has grown by N×K×(48 + HashSet overhead) bytes, entirely outside the allocator's accounting and limits.

    fn exit_guard(&mut self, current_cost: Cost) -> Result<Cost> {
        // this is called when we are done executing a softfork program.
        // This is when we have to validate the cost
        let guard = self
            .softfork_stack
            .pop()
            .expect("internal error. exiting a softfork that's already been popped");

        if current_cost != guard.expected_cost {
            #[cfg(test)]
            println!(
                "actual cost: {} specified cost: {}",
                current_cost - guard.start_cost,
                guard.expected_cost - guard.start_cost
            );
            return Err(EvalErr::SoftforkCostMismatch);
        }

        // restore the allocator to the state when we entered the softfork guard
        // This is an optimization to reclaim all heap space allocated by the
        // softfork program. Since the softfork always return nil, no value can
        // escape the softfork program, and it's therefore safe to restore the
        // heap
        self.allocator.restore_checkpoint(&guard.allocator_state);

        // the softfork always returns nil, pop the value pushed by the
        // evaluation of the program and push nil instead
        self.pop()
            .expect("internal error, softfork program did not push value onto stack");

        self.push(self.allocator.nil())?;

        Ok(0)
    }
