### Title
Silent Liquidation Bypass via Missing `extcodesize` Check on `clearinghouseLiq` Delegatecall — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.liquidateSubaccount` delegates execution to `clearinghouseLiq` without verifying the target address has deployed code. Neither `initialize` nor `upgradeClearinghouseLiq` enforce a non-zero address for `clearinghouseLiq`. If `clearinghouseLiq` is `address(0)` or an address with no code, the EVM returns `success = true` with empty returndata, the `require(success, ...)` guard passes, and the entire liquidation silently does nothing.

---

### Finding Description

In `Clearinghouse.liquidateSubaccount`, the delegatecall to `clearinghouseLiq` is performed with no `extcodesize` guard and no zero-address pre-check:

```solidity
// Clearinghouse.sol L654–L661
(bool success, bytes memory result) = clearinghouseLiq.delegatecall(
    liquidateSubaccountCall
);
require(success, string(result));
``` [1](#0-0) 

The EVM specification guarantees that a `delegatecall` to an address with no code (including `address(0)`) returns `success = true` and zero bytes of returndata. The `require(success, string(result))` check therefore passes unconditionally, and the call frame returns as if liquidation succeeded.

`clearinghouseLiq` is stored in `ClearinghouseStorage` and is set in two places, neither of which enforces a non-zero value:

**`initialize`** — sets `clearinghouseLiq = _clearinghouseLiq` with no zero-address check: [2](#0-1) 

**`upgradeClearinghouseLiq`** — sets `clearinghouseLiq = _clearinghouseLiq` with no zero-address check: [3](#0-2) 

The storage variable itself carries no invariant: [4](#0-3) 

Compare with `Endpoint._delegatecallEndpointTx`, which at least guards against `address(0)` before its own delegatecall: [5](#0-4) 

`Clearinghouse.liquidateSubaccount` has no equivalent guard at all.

---

### Impact Explanation

If `clearinghouseLiq` is `address(0)` or any codeless address, every sequencer-submitted `LiquidateSubaccount` transaction silently succeeds without executing `ClearinghouseLiq.liquidateSubaccountImpl`. Concretely:

- Unhealthy subaccounts are never actually liquidated.
- Bad debt accumulates unchecked against the insurance fund.
- The protocol's solvency invariant — that maintenance-unhealthy positions are closed before losses exceed insurance — is silently broken.
- No on-chain revert or event signals the failure; the sequencer observes a successful transaction.

The corrupted state delta is: `clearinghouseLiq` balance/position mutations that `liquidateSubaccountImpl` would have applied never occur, while the sequencer's submission counter advances as normal.

---

### Likelihood Explanation

Requires `clearinghouseLiq` to be `address(0)` or codeless, which can happen via:

1. **Deployment misconfiguration** — `initialize` called with `_clearinghouseLiq = address(0)` (no guard prevents it).
2. **Upgrade misconfiguration** — `upgradeClearinghouseLiq` called with `address(0)` or a stale/destroyed address (no guard prevents it).

This mirrors the original RocketPool finding exactly: a series of configuration failures is required, but the missing safety check means the failure mode is completely silent and undetectable from transaction receipts alone. Likelihood is low but non-negligible given the absence of any defensive check at the call site.

---

### Recommendation

Add an `extcodesize` check immediately before the delegatecall in `liquidateSubaccount`, and add zero-address guards in both `initialize` and `upgradeClearinghouseLiq`:

```solidity
// In liquidateSubaccount, before delegatecall:
uint256 codeSize;
assembly { codeSize := extcodesize(clearinghouseLiq) }
require(codeSize > 0, "clearinghouseLiq has no code");

// In initialize and upgradeClearinghouseLiq:
require(_clearinghouseLiq != address(0), "zero address");
```

---

### Proof of Concept

1. Deploy `Clearinghouse` via its proxy, passing `_clearinghouseLiq = address(0)` to `initialize` — no revert occurs.
2. Register engines and list products normally.
3. Create a subaccount that falls below maintenance health.
4. Sequencer submits a `LiquidateSubaccount` transaction targeting that subaccount.
5. `Clearinghouse.liquidateSubaccount` encodes the call and executes:
   ```solidity
   (bool success, bytes memory result) = address(0).delegatecall(callData);
   // success == true, result == ""
   require(true, ""); // passes
   ``` [6](#0-5) 
6. Transaction succeeds on-chain. The subaccount's positions are unchanged. `isUnderMaintenance` still returns `true`. The protocol has silently accepted a no-op liquidation.
