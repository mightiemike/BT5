### Title
Hardcoded `msg.sender` Recipient in Airdrop Claim Locks Tokens for Smart Contract Claimants — (`File: core/contracts/Airdrop.sol`)

---

### Summary

`Airdrop.sol` hardcodes `msg.sender` as both the verified claimant identity and the ERC20 token recipient in `_claim()`. There is no parameter to specify a separate recipient address. If a smart contract address is included in the Merkle tree as an eligible claimant and that contract calls `claim()`, the airdrop tokens are transferred directly to the calling contract. If the contract has no ERC20 withdrawal function, the tokens are permanently locked.

---

### Finding Description

In `_claim()`, the token transfer destination is unconditionally set to `msg.sender`:

```solidity
function _claim(
    uint32 week,
    uint256 totalAmount,
    bytes32[] calldata proof
) internal {
    _verifyProof(week, msg.sender, totalAmount, proof);
    SafeERC20.safeTransfer(IERC20(token), msg.sender, totalAmount);
    emit Claim(msg.sender, week, totalAmount);
}
``` [1](#0-0) 

The Merkle leaf is constructed from `(sender, totalAmount)` where `sender = msg.sender`, so the claimant address is baked into the proof verification:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(keccak256(abi.encode(sender, totalAmount)))
);
``` [2](#0-1) 

This means the address in the Merkle tree must be the `msg.sender` at claim time, and that same address unconditionally receives the tokens. There is no `recipient` parameter in either `_claim()` or the public `claim()` entry point. [3](#0-2) 

Once `claimed[week][sender]` is set to `totalAmount`, the claim is permanently consumed for that week and address: [4](#0-3) 

---

### Impact Explanation

If a smart contract address (e.g., a DAO treasury, a protocol-owned multisig, a vesting contract, or any contract without a generic ERC20 `transfer`/`rescue` function) is included in the Merkle tree as an eligible claimant, and that contract calls `claim()`:

1. `SafeERC20.safeTransfer` sends the airdrop tokens to the calling contract.
2. If the contract has no function to withdraw ERC20 tokens, the tokens are permanently locked inside it.
3. The claim is marked as consumed (`claimed[week][sender] = totalAmount`), so the same address cannot re-claim for that week.
4. The `registerMerkleRoot` function only allows new weeks to be added; it cannot retroactively fix a past week's locked tokens.

The corrupted asset delta is the full `totalAmount` of airdrop tokens for the affected claimant, permanently irrecoverable.

---

### Likelihood Explanation

Smart contract addresses are routinely included in token airdrops — protocol treasuries, multisigs, DAO governance contracts, and vesting contracts are common recipients. Not all such contracts implement a generic ERC20 rescue function. The trigger requires only that such a contract calls `claim()`, which is a normal, permissionless user action. No privileged access or external dependency failure is required.

---

### Recommendation

Add an optional `recipient` parameter to `_claim()` and `claim()`, defaulting to `msg.sender` if not specified. This mirrors the fix applied to the AmarokFacet issue and the pattern already used elsewhere in the Nado codebase (e.g., `WithdrawCollateralV2.sendTo`, `WithdrawInsurance.sendTo`):

```solidity
function _claim(
    uint32 week,
    uint256 totalAmount,
    bytes32[] calldata proof,
    address recipient
) internal {
    _verifyProof(week, msg.sender, totalAmount, proof);
    address to = recipient == address(0) ? msg.sender : recipient;
    SafeERC20.safeTransfer(IERC20(token), to, totalAmount);
    emit Claim(msg.sender, week, totalAmount);
}
```

The Merkle proof verification should remain tied to `msg.sender` (the authorized claimant), while the token destination becomes caller-specified.

---

### Proof of Concept

1. Protocol includes `address(contractA)` in the week-1 Merkle tree with `totalAmount = 1000e18`.
2. `contractA` has no ERC20 withdrawal function (e.g., a simple vesting contract or a minimal DAO executor).
3. `contractA` calls `Airdrop.claim([{week: 1, totalAmount: 1000e18, proof: [...]}])`.
4. `_verifyProof` passes because `msg.sender == address(contractA)` matches the leaf.
5. `SafeERC20.safeTransfer(IERC20(token), address(contractA), 1000e18)` executes successfully (ERC20 transfers to contracts do not revert unless the contract explicitly rejects them).
6. `claimed[1][address(contractA)] = 1000e18` is set.
7. The 1000e18 tokens are now inside `contractA` with no recovery path. The claim is permanently consumed. [1](#0-0)
