### Title
Protocol Withdrawal Fee Bypassed via Zero `feeX18` in `WithdrawCollateralV2` — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `WithdrawCollateralV2` transaction handler in `EndpointTx.sol` accepts a user-signed `feeX18` field and enforces only an upper bound (`feeX18 <= currentFeeX18`) with no minimum threshold. A user can sign a withdrawal with `feeX18 = 0`, and the on-chain contract will process it without charging any withdrawal fee, even when the product's configured `withdrawFeeX18` is non-zero.

---

### Finding Description

In `EndpointTx.sol`, the `WithdrawCollateral` (V1) path always charges the full configured fee:

```solidity
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,
    signedTx.tx.productId
);
```

The `WithdrawCollateralV2` path, however, reads `feeX18` directly from the user-signed transaction struct and validates only:

```solidity
int128 currentFeeX18 = spotEngine
    .getConfig(signedTx.tx.productId)
    .withdrawFeeX18;
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
chargeFee(
    signedTx.tx.sender,
    signedTx.feeX18,
    signedTx.tx.productId
);
``` [1](#0-0) 

There is no `require(signedTx.feeX18 >= currentFeeX18)` or any minimum threshold. The user signs the `feeX18` value as part of the `SignedWithdrawCollateralV2` struct, and `validateSignedTx` only verifies the signature over the full transaction bytes — it does not constrain the value of `feeX18` itself. [2](#0-1) 

The `IEndpoint.SignedWithdrawCollateralV2` struct exposes `feeX18` as a standalone field separate from the inner `WithdrawCollateralV2` tx, making it user-settable at signing time. [3](#0-2) 

---

### Impact Explanation

When a user signs a `WithdrawCollateralV2` with `feeX18 = 0`, the call `chargeFee(sender, 0, productId)` transfers nothing to the protocol. The protocol's configured `withdrawFeeX18` — intended to compensate the protocol for each withdrawal — is entirely bypassed. This is a direct revenue loss: every `WithdrawCollateralV2` processed with `feeX18 = 0` deprives the protocol of its expected withdrawal fee, undermining the fee model that funds protocol operations.

The contrast with V1 is stark: V1 always charges `withdrawFeeX18` with no user override. V2 introduced a user-controlled fee field with only an upper bound, creating an asymmetry that breaks the protocol's fee invariant.

---

### Likelihood Explanation

The user signs `feeX18` directly. The sequencer submits the signed transaction on-chain. The on-chain contract has no minimum threshold enforcement. If the sequencer's off-chain logic does not independently validate that `feeX18 == currentFeeX18`, any user who signs `feeX18 = 0` will have their withdrawal processed fee-free. There is no on-chain guard preventing this. The sequencer is a necessary intermediary but does not need to be compromised — it only needs to lack explicit off-chain rejection of `feeX18 < currentFeeX18`, which is not enforced by the contract itself.

---

### Recommendation

Enforce that the user-supplied fee equals the configured fee, removing the ability to undercut it:

```diff
require(signedTx.feeX18 >= 0);
- require(signedTx.feeX18 <= currentFeeX18);
+ require(signedTx.feeX18 == currentFeeX18, "fee must match configured withdrawal fee");
```

Alternatively, ignore `feeX18` entirely and always charge `currentFeeX18`, matching the V1 behavior.

---

### Proof of Concept

1. User constructs a `WithdrawCollateralV2` transaction with `feeX18 = 0` for a product where `withdrawFeeX18 = X > 0`.
2. User signs the full transaction bytes (including `feeX18 = 0`).
3. Sequencer submits the signed transaction via `processTransaction`.
4. `validateSignedTx` passes — the signature is valid over the bytes including `feeX18 = 0`.
5. `require(signedTx.feeX18 >= 0)` passes (0 ≥ 0). `require(signedTx.feeX18 <= currentFeeX18)` passes (0 ≤ X).
6. `chargeFee(sender, 0, productId)` is called — protocol collects zero fee.
7. `clearinghouse.withdrawCollateral(...)` executes the full withdrawal.

The protocol receives `0` instead of `X` for the withdrawal, with no revert or error. [4](#0-3)
