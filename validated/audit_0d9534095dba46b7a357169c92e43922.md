### Title
Softfork cost atom with unnecessary leading zero accepted in consensus but rejected in mempool, causing consensus/mempool divergence — (`src/run_program.rs`, `src/op_utils.rs`)

### Summary

A softfork invocation whose cost argument atom has an unnecessary leading zero byte (e.g., `b'\x00\x64'` = 100) is silently accepted in consensus mode but raises `InvalidOpArg` in mempool mode. This is a concrete, locally testable consensus/mempool divergence.

### Finding Description

In `apply_op` (`src/run_program.rs:385-390`), the softfork cost argument is parsed via:

```rust
let expected_cost = uint_atom::<8>(
    self.allocator,
    first(self.allocator, operand_list)?,
    "softfork",
    self.dialect.flags(),   // ← passes CANONICAL_INTS when in MEMPOOL_MODE
)?;
```

`uint_atom` (`src/op_utils.rs:67-86`) branches on `CANONICAL_INTS`:

- **Consensus (`CANONICAL_INTS` not set):** strips all leading zero bytes in a `while` loop, then reads the value. `b'\x00\x64'` → value 100. Execution continues normally.
- **Mempool (`CANONICAL_INTS` set, via `MEMPOOL_MODE`):** if `buf[0] == 0` and `(buf[1] & 0x80) == 0`, immediately returns `Err(EvalErr::InvalidOpArg(..., "softfork requires u64 arg with no leading zeros"))`. For `b'\x00\x64'`: `buf[0]=0x00`, `buf[1]=0x64`, `(0x64 & 0x80) == 0` → error returned, `?` propagates it, the entire softfork call fails.

`MEMPOOL_MODE` is explicitly defined as `NO_UNKNOWN_OPS | LIMIT_HEAP | DISABLE_OP | CANONICAL_INTS | LIMIT_SOFTFORK`: [1](#0-0) 

The existing test suite even documents the consensus-mode behavior explicitly:

```
// when parsing the cost argument, we ignore redundant leading zeroes
RunProgramTest {
    prg: "(softfork (q . 0x00000397) ...",
    flags: ClvmFlags::empty(),
    result: Some("()"),
    ...
},
``` [4](#0-3) 

There is no corresponding test for the same input under `MEMPOOL_MODE` / `CANONICAL_INTS`, and no guard in `apply_op` that normalizes the cost atom before passing it to `uint_atom`.

### Impact Explanation

An attacker can craft a CLVM program containing `(softfork (q . <non-canonical-cost-atom>) ...)` where the cost atom has an unnecessary leading zero. This program:

1. **Passes consensus validation** — full nodes running without `CANONICAL_INTS` accept it, compute the correct cost, and include the block.
2. **Fails mempool validation** — nodes running `MEMPOOL_MODE` reject it with `InvalidOpArg` before it ever enters the mempool.

The divergence direction (consensus accepts, mempool rejects) means such transactions can only reach the chain by being submitted directly to a block producer that bypasses the mempool. Once included in a block, all consensus-mode full nodes accept it. This undermines the mempool's role as a pre-filter for block validity and creates a class of on-chain-valid transactions that are permanently invisible to mempool-based fee/cost accounting and DoS protection.

### Likelihood Explanation

The attack requires only crafting a CLVM atom with a leading zero byte — trivially achievable by any party who can submit transactions. No special privileges, compromised nodes, or social engineering are needed. The divergence is deterministic and reproducible.

### Recommendation

Normalize the cost atom before the `CANONICAL_INTS` check, or apply the same leading-zero stripping in both modes at the `apply_op` level before calling `uint_atom`. Alternatively, add an explicit pre-check in `apply_op` that rejects non-canonical cost atoms in **both** consensus and mempool modes, so the behavior is uniform. The existing test at line 1067 should be extended to cover `MEMPOOL_MODE` / `CANONICAL_INTS` to catch this regression.

### Proof of Concept

```rust
// Consensus mode: accepts b'\x00\x64' as cost 100
let result_consensus = run_program(
    &mut allocator,

    &mut allocator,
    &dialect_with_flags(MEMPOOL_MODE),
    program_with_cost_atom(&[0x00, 0x64]),
    env,
    1000,
);
assert!(matches!(result_mempool, Err(EvalErr::InvalidOpArg(_, msg)) if msg.contains("no leading zeros")));
```
