### Title
Fast Withdrawal Fee Truncates to Zero for Low-Decimal Tokens via Integer Division — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.fastWithdrawalFeeAmount` converts a fee computed in X18 fixed-point back to native token decimals with a bare integer division. For tokens with fewer than 18 decimals, the divisor (`multiplier`) can be larger than the computed `feeX18`, causing the returned fee to truncate to zero. An unprivileged caller can exploit this by splitting a large fast withdrawal into many small ones, each below the truncation threshold, and pay no fast-withdrawal fee at all.

---

### Finding Description

`fastWithdrawalFeeAmount` in `BaseWithdrawPool.sol` computes the fee as follows: [1](#0-0) 

```
multiplier  = 10^(MAX_DECIMALS - decimals)   // e.g. 1e16 for a 2-decimal token
amountX18   = amount * multiplier
proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18)   // 0.1% of amountX18
feeX18      = max(proportionalFeeX18, minFeeX18)
return feeX18 / multiplier                   // ← integer division, can truncate to 0
```

`FAST_WITHDRAWAL_FEE_RATE` is `1e15` (0.1% in X18 scale). [2](#0-1) 

For a 2-decimal token (e.g. GeminiUSD), `multiplier = 1e16`. The proportional fee in X18 is:

```
proportionalFeeX18 = 0.001 * amount * 1e16
```

For this to be non-zero after the final division by `1e16`, we need `proportionalFeeX18 >= 1e16`, i.e. `amount >= 1000` (native units = 10.00 tokens). Any withdrawal of fewer than 10 tokens produces `proportionalFeeX18 < 1e16`, which integer-divides to **zero**.

The `minFeeX18` guard (`5 * withdrawFeeX18`) does not save this: if `withdrawFeeX18` is configured at a small value (e.g. `1e15`), then `minFeeX18 = 5e15 < 1e16`, and the final division still returns 0. [3](#0-2) 

In `submitFastWithdrawal`, when `fee == 0` and `sendTo == msg.sender`, the caller receives the full `transferAmount` with no deduction: [4](#0-3) 

---

### Impact Explanation

An unprivileged caller can bypass the `FAST_WITHDRAWAL_FEE_RATE` (0.1%) entirely for any low-decimal token by splitting a large withdrawal into many sub-threshold calls to `submitFastWithdrawal`. The protocol collects zero fast-withdrawal fees on those transactions. `fees[productId]` is never incremented, so the accounting is also corrupted.

---

### Likelihood Explanation

`submitFastWithdrawal` is a public function with no role restriction — any address can call it. [5](#0-4) 

Tokens with 2–6 decimals (USDC on some deployments, USDT, GeminiUSD, WBTC) are common. On L2s like Ink Chain where gas is cheap, splitting a withdrawal into dozens of small calls is economically rational. The threshold below which the fee is zero scales with `10^(18 - decimals)`, making the attack window larger the fewer decimals the token has.

---

### Recommendation

After computing `feeX18`, enforce a minimum that guarantees the native-decimal result is at least 1 unit before returning:

```solidity
// Ensure fee does not truncate to zero
int128 feeNative = feeX18 / int128(multiplier);
require(feeNative > 0, "Amount too small: fee rounds to zero");
return feeNative;
```

Alternatively, enforce a minimum withdrawal amount in native decimals so that `amount * FAST_WITHDRAWAL_FEE_RATE >= multiplier` always holds.

---

### Proof of Concept

Token: 2-decimal token, `withdrawFeeX18 = 1e15`, `FAST_WITHDRAWAL_FEE_RATE = 1e15`.

```
multiplier        = 10^(18-2) = 1e16
amount            = 999  (= 9.99 tokens in native units)
amountX18         = 999 * 1e16 = 9.99e18
proportionalFeeX18 = 1e15 * 9.99e18 / 1e18 = 9.99e15
minFeeX18         = 5 * 1e15 = 5e15
feeX18            = max(9.99e15, 5e15) = 9.99e15
fee               = 9.99e15 / 1e16 = 0  ← truncated to zero
```

The attacker calls `submitFastWithdrawal` repeatedly with `amount = 999`, withdrawing any total amount while paying zero fees each time.

```text
    function fastWithdrawalFeeAmount(
        IERC20Base token,
        uint32 productId,
        uint128 amount
    ) public view returns (int128) {
        uint8 decimals = token.decimals();
        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
        int128 amountX18 = int128(amount) * int128(multiplier);

        int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
        int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;

        int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
        return feeX18 / int128(multiplier);
    }
```
