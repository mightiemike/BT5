### Title
Linked Signer Authorization Has No Expiration, Enabling Indefinite Signing Authority After Intended Revocation — (File: `core/contracts/interfaces/IEndpoint.sol`)

---

### Summary

The `LinkSigner` struct in Nado contains no expiration field. Once a linked signer is authorized, it retains full signing authority over a subaccount indefinitely. A user who wishes to revoke a linked signer must submit a new `LinkSigner` transaction, which — via the slow mode path — takes 3 days to execute. During that window, the still-authorized linked signer can sign and submit `WithdrawCollateral`, `TransferQuote`, and other asset-moving transactions through the sequencer, draining the subaccount before revocation takes effect.

---

### Finding Description

The `LinkSigner` struct in `IEndpoint.sol` contains only `sender`, `signer`, and `nonce` — no expiration:

```solidity
struct LinkSigner {
    bytes32 sender;
    bytes32 signer;
    uint64 nonce;
}
``` [1](#0-0) 

The EIP-712 type string registered in `Verifier.sol` also omits any expiration field:

```solidity
string internal constant LINK_SIGNER_SIGNATURE =
    "LinkSigner(bytes32 sender,bytes32 signer,uint64 nonce)";
``` [2](#0-1) 

When the sequencer processes a `LinkSigner` transaction via `processTransactionImpl`, the linked signer is written to storage with no time bound:

```solidity
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [3](#0-2) 

The same unconditional write occurs in the slow mode path (`processSlowModeTransactionImpl`):

```solidity
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The linked signer is then consulted for every sensitive signed transaction — `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, `TransferQuote`, `MintNlp`, `BurnNlp`, and order matching — via `getLinkedSigner`:

```solidity
function validateSignature(
    bytes32 sender,
    bytes32 digest,
    bytes memory signature,
    bool allowLinkedSigner
) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        ...
    );
}
``` [5](#0-4) 

The only on-chain revocation mechanism is submitting a new `LinkSigner` transaction. Via `submitSlowModeTransaction`, the slow mode delay is hardcoded to 3 days:

```solidity
executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
``` [6](#0-5) 

Contrast this with `Order`, which does carry an `expiration` field that is actively enforced via `_expired()` in `OffchainExchange.sol`:

```solidity
struct Order {
    ...
    uint64 expiration;
    ...
}
``` [7](#0-6) 

```solidity
function _expired(uint64 expiration) internal view returns (bool) {
    return expiration <= getOracleTime();
}
``` [8](#0-7) 

The protocol already has the pattern for expiration-checked signatures; it is simply absent from `LinkSigner`.

---

### Impact Explanation

A linked signer authorized for a trading bot or third-party service retains full signing authority over `WithdrawCollateral`, `TransferQuote`, and all other user-signed transaction types indefinitely. If the user decides to revoke the linked signer — whether due to suspected compromise, end of service, or key rotation — the only on-chain path is a slow mode transaction with a 3-day execution delay. During that window, the still-valid linked signer can sign and submit withdrawal transactions through the sequencer, draining the subaccount's collateral before the revocation is applied. The corrupted state is `linkedSigners[subaccount]` remaining set to the malicious address, and the corrupted asset delta is the full collateral balance of the subaccount.

---

### Likelihood Explanation

Linked signers are the standard mechanism for trading bots and API-key-style delegations in Nado. Any user who has ever set a linked signer and later wishes to revoke it faces this window. The scenario is realistic: a third-party trading service is compromised, a bot key is leaked, or a user simply wants to rotate credentials. The 3-day slow mode delay is a protocol constant, not a user choice, making the window deterministic and exploitable.

---

### Recommendation

Add an `expiration` field to the `LinkSigner` struct and enforce it at processing time, mirroring the pattern already used for `Order`:

```solidity
struct LinkSigner {
    bytes32 sender;
    bytes32 signer;
    uint64 nonce;
    uint64 expiration; // add this
}
```

In `processTransactionImpl` and `processSlowModeTransactionImpl`, reject the transaction if `block.timestamp >= expiration`. Update `LINK_SIGNER_SIGNATURE` in `Verifier.sol` to include the new field so it is covered by the EIP-712 digest.

---

### Proof of Concept

1. User A calls `submitSlowModeTransaction` with a `LinkSigner` payload setting linked signer B (e.g., a trading bot).
2. After 3 days the sequencer executes it; `linkedSigners[A] = B` is written.
3. B's key is later compromised. A calls `submitSlowModeTransaction` with a new `LinkSigner` payload setting `signer = address(0)` to revoke B.
4. The revocation is queued with `executableAt = block.timestamp + 3 days`.
5. During those 3 days, the attacker holding B's key signs `WithdrawCollateralV2` transactions specifying A's subaccount as sender and an attacker-controlled `sendTo` address.
6. The sequencer, seeing valid EIP-712 signatures from the currently-authorized linked signer B, includes these withdrawals in a `submitTransactions` batch.
7. `validateSignedTx` passes because `getLinkedSigner(A)` still returns B; `WithdrawCollateralV2` is processed and A's collateral is transferred out.
8. A's revocation executes after 3 days, but the account is already drained. [9](#0-8)
