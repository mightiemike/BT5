### Title
Receipt `max_receipt_size` Hard Limit Bypassed via Post-Creation `output_data_receivers` Mutation, Forwarding Oversized Receipts Cross-Shard — (`runtime/runtime/src/lib.rs`)

---

### Summary

The nearcore runtime enforces `max_receipt_size` (4 MiB) as a hard limit on new receipts. However, two code paths mutate a receipt's Borsh-serialized size **after** the size check has already passed, producing receipts that exceed `max_receipt_size` and forwarding them cross-shard. The receiving shard processes them under `ValidateReceiptMode::ExistingReceipt`, which explicitly skips the size check. This is a confirmed, open protocol-invariant violation (nearcore issue #12606) with an unprivileged, contract-controlled trigger.

---

### Finding Description

**Root cause path 1 — `promise_return` appends `output_data_receivers` post-check**

When a contract executes `promise_return(C)` inside a promise chain `[A -then-> B]`, the runtime wires C into the DAG by extending C's `output_data_receivers` with B's data receivers. This mutation happens in `lib.rs` **after** C's receipt was already validated at creation time:

```rust
// runtime/runtime/src/lib.rs ~1019-1037
ReceiptEnum::ActionV2(new_action_receipt)
| ReceiptEnum::PromiseYieldV2(new_action_receipt) => new_action_receipt
    .output_data_receivers
    .extend_from_slice(&action_receipt.output_data_receivers()),
```

A contract can craft receipt C with size exactly equal to `max_receipt_size` (passes `NewReceipt` check), then `promise_return` it. The appended `DataReceiver` structs push the final Borsh size above `max_receipt_size` with no re-validation.

**Root cause path 2 — `value_return` creates a data receipt whose size exceeds `max_receipt_size`**

`max_length_returned_data` equals `max_receipt_size` (both 4 MiB). A contract returning exactly `max_length_returned_data` bytes passes the `ReturnedValueLengthExceeded` check in `VMLogic::value_return`. The runtime then wraps that value in a `DataReceipt`:

```rust
// runtime/runtime/src/lib.rs ~1044-1056
Receipt::V0(ReceiptV0 {
    ...
    receipt: ReceiptEnum::Data(DataReceipt {
        data_id: data_receiver.data_id,
        data: data.clone(),   // up to 4 MiB
    }),
})
```

The `DataReceipt` header overhead makes the total Borsh size exceed `max_receipt_size`, but no size check is performed on the newly created data receipt.

**Cross-shard forwarding without re-validation**

Both oversized receipts are placed in `result.new_receipts` and forwarded cross-shard. On the receiving shard, `process_incoming_receipts` calls `validate_receipt` with `ValidateReceiptMode::ExistingReceipt`:

```rust
// runtime/runtime/src/lib.rs ~2512-2518
validate_receipt(
    &processing_state.apply_state.config.wasm_config.limit_config,
    receipt,
    protocol_version,
    ValidateReceiptMode::ExistingReceipt,   // ← size check is skipped
)
```

The `ExistingReceipt` mode explicitly skips the `max_receipt_size` check:

```rust
// runtime/runtime/src/verifier.rs ~533-542
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 = borsh::object_length(receipt)...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
```

**Workaround confirms the bug is live**

The `try_forward` function in `congestion_control.rs` contains an explicit band-aid that clamps the oversized receipt's size to `max_receipt_size` to prevent it from getting permanently stuck in the outgoing buffer:

```rust
// runtime/runtime/src/congestion_control.rs ~413-427
// There is a bug which allows to create receipts that are above the size limit.
// See https://github.com/near/nearcore/issues/12606
let max_receipt_size = apply_state.config.wasm_config.limit_config.max_receipt_size;
if size > max_receipt_size {
    size = max_receipt_size;
}
```

The integration tests `test_max_receipt_size_promise_return` and `test_max_receipt_size_value_return` explicitly assert that oversized receipts **do** appear in the chain and that the runtime must handle them gracefully.

---

### Impact Explanation

- **Protocol invariant broken**: `max_receipt_size` is documented as a hard limit. Receipts above this limit appear in the chain and in `ChunkStateWitness::source_receipt_proofs`, violating the invariant that the uncompressed witness stays under 17 MiB.
- **Bandwidth accounting corrupted**: The clamping workaround in `try_forward` causes the outgoing-receipt size accounting to use a fictitious value (`max_receipt_size`) instead of the actual receipt size, making congestion-control decisions incorrect.
- **Cross-shard receipt forwarding delayed**: An oversized receipt (e.g., 4+ MiB) far exceeds `outgoing_receipts_usual_size_limit` (100 KiB). It can only be forwarded via the "allowed shard" big-limit slot (4.5 MiB), delaying delivery and reducing effective cross-shard throughput for other receipts.
- **Witness size bound violated**: A single oversized receipt included in `source_receipt_proofs` can push the `ChunkStateWitness` beyond the 17 MiB design ceiling, causing distribution and validation overhead.

---

### Likelihood Explanation

Any unprivileged contract can trigger both paths:

- **Path 1**: Deploy a contract that creates a receipt with args sized to `max_receipt_size - header_overhead`, then calls `promise_return` on it inside a `.then()` chain. This is standard NEAR contract API usage.
- **Path 2**: Deploy a contract that calls `value_return` with exactly `max_length_returned_data` bytes inside a `.then()` chain. The resulting data receipt will exceed `max_receipt_size` by the header overhead (~200 bytes).

No privileged role, validator key, or special permission is required. The trigger is reachable from any deployed contract on mainnet.

---

### Recommendation

1. **Re-validate after mutation**: After appending `output_data_receivers` to a receipt in `lib.rs`, re-compute the Borsh size and return an error if it exceeds `max_receipt_size`.
2. **Validate data receipts at creation**: When creating `DataReceipt` objects from `value_return` results, check the total Borsh size against `max_receipt_size` before adding them to `result.new_receipts`.
3. **Align `max_length_returned_data` with `max_receipt_size`**: Since a data receipt wrapping `max_length_returned_data` bytes always exceeds `max_receipt_size`, reduce `max_length_returned_data` by the maximum data receipt header size to close the gap.
4. **Remove the clamping workaround** in `try_forward` once the root cause is fixed, restoring correct congestion-control accounting.

---

### Proof of Concept

The bug is confirmed by the existing integration tests in `test-loop-tests/src/tests/max_receipt_size.rs`:

- `test_max_receipt_size_promise_return` (line 130): crafts a receipt at exactly `max_receipt_size`, triggers `promise_return`, and asserts via `assert_oversized_receipt_occurred` that a receipt above `max_receipt_size` appears in the chain.
- `test_max_receipt_size_value_return` (line 216): returns a value of size `max_receipt_size`, and asserts that the resulting data receipt above `max_receipt_size` appears in the chain.
