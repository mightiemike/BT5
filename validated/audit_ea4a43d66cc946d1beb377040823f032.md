### Title
No Staleness Threshold on `SignedAccountData` Allows Indefinitely Stale Validator Proxy Routing - (`chain/network/src/accounts_data/mod.rs`)

### Summary

`AccountDataCacheSnapshot::is_new()` and `try_insert()` accept any `SignedAccountData` whose `(version, timestamp)` tuple is lexicographically greater than what is already stored. Neither function ever compares the embedded `timestamp` against the current wall-clock time. There is no maximum age threshold. The code itself acknowledges this gap with the comment: *"eventually we might want to use the timestamp set expiration date on AccountData."* As a result, once a validator stops re-broadcasting its proxy information, the last-seen `SignedAccountData` — regardless of how old its timestamp is — remains the authoritative routing record for TIER1 messages indefinitely.

### Finding Description

`VersionedAccountData` carries a `timestamp: time::Utc` field whose stated purpose is to act as a tie-breaker for version collisions and, prospectively, as an expiry signal. [1](#0-0) 

The freshness check in `is_new()` only compares `(version, timestamp)` tuples between the incoming datum and the stored datum: [2](#0-1) 

`try_insert()` calls `is_new()` and then stores the datum unconditionally if it passes: [3](#0-2) 

The code comment in `set_local()` explicitly acknowledges the missing expiry: [4](#0-3) 

Contrast this with `OwnedAccount` in the handshake path, which **does** enforce a `MAX_CLOCK_SKEW` bound on its timestamp: [5](#0-4) 

No equivalent bound exists anywhere in the `SyncAccountsData` / `AccountDataCache` path.

### Impact Explanation

`SignedAccountData` is the mechanism by which TIER1 validators advertise their proxy addresses to the network. Stale entries cause TIER1 messages — chunk state witnesses, chunk endorsements — to be routed to unreachable or wrong proxy nodes. If a validator's proxy information is stale and the validator cannot be reached over TIER1, it cannot receive state witnesses and cannot produce chunk endorsements, degrading chunk production and potentially triggering validator kickout.

### Likelihood Explanation

Any peer that has received a `SignedAccountData` broadcast (which is public gossip) can re-broadcast it indefinitely via `SyncAccountsData`. Because `is_new()` only requires the incoming `(version, timestamp)` to exceed the stored value, a peer holding the highest-version datum for a given validator can keep that datum alive in the network forever, even after the validator has changed its proxy configuration or gone offline. No privilege is required beyond being a connected peer.

### Recommendation

Add a maximum-age check inside `is_new()` (or at the `insert` call site) that rejects any `SignedAccountData` whose `timestamp` is older than a configurable threshold (e.g., 2× the expected re-broadcast interval). The `clock` parameter already flows into `try_insert()` and `set_local()`, so the comparison `clock.now_utc() - d.timestamp > MAX_ACCOUNT_DATA_AGE` can be inserted without architectural changes. The threshold should be documented as a protocol constant alongside `MAX_ACCOUNT_DATA_SIZE_BYTES`. [6](#0-5) 

### Proof of Concept

1. Validator V broadcasts `SignedAccountData` version 3 with proxy address `P1` at time `T`.
2. At time `T + Δ`, V changes its proxy to `P2` and broadcasts version 4. All honest peers update their cache.
3. A malicious peer `M` retained version 3 (captured before step 2). `M` continuously sends `SyncAccountsData` containing version 3 to newly joining peers.
4. Because `is_new()` has no wall-clock staleness check, a newly joining peer that has not yet seen version 4 will accept version 3 and route TIER1 messages to `P1`.
5. `P1` is unreachable; TIER1 messages to V are dropped; V cannot receive state witnesses; V's chunk endorsements are missing.

The exact divergent field is `VersionedAccountData::timestamp` at: [2](#0-1) 

which is compared only against another stored `timestamp`, never against `clock.now_utc()`.

    /// The wrapped account data.
    pub data: AccountData,
    /// Account key of the validator signing this AccountData.
    pub account_key: PublicKey,
    /// Version of the AccountData. Each network node stores only
    /// the newest version of the data per validator. The newest AccountData
    /// is the one with the lexicographically biggest (version,timestamp) tuple:
    /// * version is a manually incremented version counter. In case a validator
    ///   (after a restart/crash/state loss) learns from the network that it has

    ///   override it by signing and broadcasting AccountData with a higher version.
    /// * timestamp is a version tie breaker, introduced only to minimize
    ///   the risk of version collision (see accounts_data/mod.rs).
    pub version: u64,

}

    fn try_insert(
        &mut self,
        clock: &time::Clock,
        d: Arc<SignedAccountData>,
    ) -> Option<Arc<SignedAccountData>> {
        if !self.is_new(&d) {
            return None;
        }
        let d = match &self.local {
            Some(local) if d.account_key == local.signer.public_key() => Arc::new(
                VersionedAccountData {
                    data: local.data.as_ref().clone(),
                    account_key: local.signer.public_key(),
                    version: d.version + 1,
                    timestamp: clock.now_utc(),
                }
                .sign(local.signer.as_ref())
                .unwrap(),
            ),
            _ => d,
        };
        self.data.insert(d.account_key.clone(), d.clone());
        Some(d)
    }
