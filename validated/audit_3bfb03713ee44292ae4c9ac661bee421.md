### Title
Cross-Subaccount `lastLiquidationFees` Contamination Causes Excess Socialization and Overstated Insurance — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a persistent storage variable that is written by every non-finalization liquidation step and read by every finalization step, with no scoping to the subaccount being finalized. A liquidator can deliberately sequence a large non-finalization liquidation of subaccount A immediately before a finalization of subaccount B, causing B's finalization to socialize losses that the insurance fund could have covered, while simultaneously leaving the insurance accounting overstated.

---

### Finding Description

`lastLiquidationFees` is declared as a bare storage slot in `ClearinghouseStorage`: [1](#0-0) 

It is unconditionally overwritten at the end of every call to `_handleLiquidationPayment`, regardless of which subaccount is being liquidated: [2](#0-1) 

The design intent (documented in the comment at lines 581–585) is to prevent the fees just collected during a single subaccount's liquidation from blocking that same subaccount's subsequent socialization. The subtract/add-back pattern in `_finalizeSubaccount` is meant to be a same-subaccount, same-session operation: [3](#0-2) 

However, `lastLiquidationFees` is never reset between different subaccounts. When `liquidateSubaccountImpl` is called with `productId == type(uint32).max` (finalization), it calls `_finalizeSubaccount` directly and returns — it never calls `_handleLiquidationPayment`, so `lastLiquidationFees` is never updated for the subaccount being finalized: [4](#0-3) 

This means the `lastLiquidationFees` value from a prior liquidation of a completely different subaccount is silently used in the subtract (line 369) and add-back (line 410) of the finalization logic.

---

### Impact Explanation

`perpEngine.socializeSubaccount` uses the passed `insurance` parameter as its coverage budget: [5](#0-4) 

When `v.insurance` is artificially reduced by a stale `lastLiquidationFees = F_A`, the perp engine socializes losses that the real insurance fund could have absorbed. After socialization, `F_A` is added back to `v.insurance` and written to `insurance` storage (line 410–411), leaving the insurance accounting overstated by exactly the amount that was incorrectly socialized.

**Concrete example:**
- `insurance = 1000`, `lastLiquidationFees = 800` (from subaccount A's prior liquidation)
- Subaccount B being finalized: perp `vQuoteBalance = -200`, spot `quoteBalance = -500`

| Step | Correct (`lastLiqFees = 0`) | Contaminated (`lastLiqFees = 800`) |
|---|---|---|
| `v.insurance` passed to perp socialize | 1000 | 200 |
| Perp vQuoteBalance covered by insurance | 200 | 200 |
| `v.insurance` after perp socialize | 800 | 0 |
| Quote balance covered by insurance | 500 | 0 |
| `spotEngine.socializeSubaccount` triggered | No | **Yes** (500 socialized) |
| Final `insurance` stored | 300 | **800** (overstated by 500) |

The 500-unit deficit is pushed onto all spot depositors via `cumulativeDepositsMultiplierX18` haircut, while the insurance fund retains 500 phantom units it should have spent. The protocol is undercollateralized by `F_A` (up to the size of the deficit).

---

### Likelihood Explanation

The attack requires only two sequential liquidation transactions routed through the normal `liquidateSubaccountImpl` public path — no special privileges, no governance, no oracle manipulation. Any liquidator bot can:

1. Submit a non-finalization liquidation of any subaccount with a large position to set `lastLiquidationFees = F` (large).
2. Immediately submit a finalization (`productId = uint32.max`) for a different subaccount that is ready to be finalized.

The sequencer processes these in order. The window is not race-dependent — the attacker controls both transactions. The larger the position in step 1, the larger `F` and the larger the socialization harm in step 2.

---

### Recommendation

Scope `lastLiquidationFees` to the subaccount being finalized. The simplest fix is to reset `lastLiquidationFees` to zero at the start of `_finalizeSubaccount` (or only use it when the finalization is for the same subaccount as the last non-finalization liquidation). Alternatively, key the storage by subaccount: `mapping(bytes32 => int128) internal lastLiquidationFees`.
