### Title
Asymmetric Fee Calculation in `applyFee` Overcharges Taker Buy Orders via Division by `(1 - feeRate)` — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

In `OffchainExchange.applyFee`, the fee for a negative `meteredQuote` (user paying quote, i.e., a taker or maker buying base) is computed by dividing by `keepRateX18 = ONE - feeRate` instead of multiplying. This inflates the fee by a factor of `feeRate / (1 - feeRate)` rather than the correct `feeRate`, systematically overcharging every buy-side order.

---

### Finding Description

Inside `applyFee`, the protocol computes how much quote the user keeps after fees:

```solidity
int128 keepRateX18 = ONE - feeInfo.feeRate;
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(keepRateX18)   // sell-side: correct
    : meteredQuote.div(keepRateX18);  // buy-side:  WRONG
orderInfo.fee = meteredQuote - newMeteredQuote;
``` [1](#0-0) 

**Sell-side path (`meteredQuote > 0`):**
The user receives quote and the fee is taken from it:
```
newMeteredQuote = meteredQuote * (1 - feeRate)
fee             = meteredQuote - meteredQuote*(1-feeRate)
               = meteredQuote * feeRate          ✓
```

**Buy-side path (`meteredQuote < 0`):**
Let `m = |meteredQuote|` (positive). The code divides:
```
newMeteredQuote = -m / (1 - feeRate)             ← more negative than -m
fee             = -m - (-m/(1-feeRate))
               = m * feeRate / (1 - feeRate)     ✗
```

The correct fee is `m * feeRate`. The actual fee is `m * feeRate / (1 - feeRate)`, which is strictly larger for any positive `feeRate`. This is the identical arithmetic error as the Basket.sol M-09 finding: dividing by `(BASE - feePct)` instead of `BASE`.

`meteredQuote` is negative whenever a taker is buying base (paying quote), which is set at line 763:

```solidity
ordersInfo.taker.quoteDelta = -ordersInfo.maker.quoteDelta;
``` [2](#0-1) 

For a taker buying base, `maker.quoteDelta > 0` so `taker.quoteDelta < 0`, making `meteredQuote < 0` inside `applyFee`. The same applies to a maker buying base.

---

### Impact Explanation

Every taker buy order and every maker buy order is overcharged. The excess fee is credited to `market.collectedFees` (protocol treasury):

```solidity
updateCollectedFees(callState.productId, market, true, ordersInfo.taker.fee, ...);
``` [3](#0-2) 

The user's `quoteDelta` is reduced by the inflated fee:

```solidity
orderInfo.quoteDelta = orderInfo.quoteDelta - orderInfo.fee - orderInfo.builderFee;
``` [4](#0-3) 

**Concrete example** (taker fee = 0.3% = `3e15`):

| | Correct | Actual (buggy) |
|---|---|---|
| Trade size | 1000 USDC | 1000 USDC |
| Fee | `1000 * 0.003 = 3.000 USDC` | `1000 * 0.003/0.997 ≈ 3.009 USDC` |
| Overcharge | — | ~0.009 USDC per 1000 USDC |

The overcharge scales with trade volume. Across all buy-side fills on all markets, the cumulative drain on user balances is material.

---

### Likelihood Explanation

This is triggered on every single taker buy order and every maker buy order processed through `matchOrders`. No special conditions, permissions, or attacker setup are required. The sequencer calls `matchOrders` as part of normal operation:

```solidity
function matchOrders(IEndpoint.MatchOrdersWithSigner calldata txn)
    external
    onlyEndpoint
``` [5](#0-4) 

Any user who places a buy order (the most common trade direction) is affected on every fill. Likelihood is **certain** given normal protocol usage.

---

### Recommendation

Replace the asymmetric `div` with `mul` for the negative branch, mirroring the sell-side calculation:

```solidity
// Before (buggy):
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(keepRateX18)
    : meteredQuote.div(keepRateX18);

// After (correct):
int128 newMeteredQuote = meteredQuote.mul(keepRateX18);
```

This yields `fee = meteredQuote * feeRate` for both positive and negative `meteredQuote`, which is the intended proportional fee.

---

### Proof of Concept

**Assumptions:**
- `ONE = 1e18`
- Taker fee rate = 0.3% → `feeRate = 3e15`
- `keepRateX18 = 1e18 - 3e15 = 997e15`
- Taker buys base worth 1000 USDC → `meteredQuote = -1000e18`

**Buggy path (`meteredQuote < 0` → divide):**
```
newMeteredQuote = -1000e18 / 997e15 * 1e18
               = -1000e18 * 1e18 / 997e15
               ≈ -1003.009e18

fee = -1000e18 - (-1003.009e18) = 3.009e18   ← 3.009 USDC charged
```

**Correct path (multiply):**
```
newMeteredQuote = -1000e18 * 997e15 / 1e18 = -997e18

fee = -1000e18 - (-997e18) = 3e18            ← 3.000 USDC charged
```

The division by `(1 - feeRate)` inflates the fee by `feeRate/(1-feeRate)` instead of `feeRate`, overcharging the user by ~0.009 USDC per 1000 USDC traded — a systematic accounting corruption on all buy-side fills. [6](#0-5)
