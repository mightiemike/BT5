### Title
`AllResources` Invoke V3 Transactions with Zero L2/L1-Data Gas Are Silently Downgraded to `L1Gas` During Protobuf Deserialization, Causing P2P Mempool Rejection — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf `ResourceBounds` wire format cannot distinguish between `ValidResourceBounds::L1Gas` (pre-0.13.3 transactions) and `ValidResourceBounds::AllResources` with zero `l2_gas` and `l1_data_gas`. When a receiving peer deserializes such a transaction, it silently downgrades it to `L1Gas`. The downstream `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion then hard-rejects it with `DEPRECATED_RESOURCE_BOUNDS_ERROR`, because `RpcInvokeTransactionV3` only accepts `AllResources`. A valid transaction accepted by the originating node is thus permanently rejected by every peer it is propagated to.

### Finding Description

`RpcInvokeTransactionV3` stores `resource_bounds` as `AllResourceBounds` (always `AllResources`). When the originating node serializes it for P2P mempool propagation, the conversion chain is:

```
RpcInvokeTransactionV3 { resource_bounds: AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 } }
  → InvokeTransactionV3 { resource_bounds: ValidResourceBounds::AllResources(...) }
  → protobuf::InvokeV3 { resource_bounds: ResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 } }
``` [1](#0-0) 

On the receiving peer, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies a zero-value heuristic:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← taken for AllResources with zero values
} else {
    ValidResourceBounds::AllResources(...)
})
``` [2](#0-1) 

The deserialized `InvokeTransactionV3` now carries `ValidResourceBounds::L1Gas`. The next step in the mempool P2P path calls `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`:

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => {
        return Err(StarknetApiError::OutOfRange { string: "resource_bounds".to_string() });
    }
},
``` [3](#0-2) 

This is invoked from the mempool P2P converter:

```rust
// This conversion can fail only if the resource_bounds are not AllResources.
Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
``` [4](#0-3) 

The peer returns `DEPRECATED_RESOURCE_BOUNDS_ERROR` and drops the transaction.

The hash domain is also broken. `get_tip_resource_bounds_hash` hashes **three** resource elements for `AllResources` (L1_GAS, L2_GAS, L1_DATA_GAS) but only **two** for `L1Gas` (L1_GAS, L2_GAS):

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [5](#0-4) 

So even if the peer did not reject the transaction outright, any hash recomputation on the peer would produce a different value than the one the user signed, binding the wrong hash to the transaction.

The existing P2P round-trip test only exercises `AllResourceBounds` with **non-zero** values and therefore does not catch this case:

```rust
static ref RESOURCE_BOUNDS_MAPPING: AllResourceBounds = AllResourceBounds {
    l1_gas: ...(0x5, 0x6),
    l2_gas: ...(0x6, 0x7),
    l1_data_gas: ...(0x7, 0x8),
};
``` [6](#0-5) 

### Impact Explanation

A user submits a valid `AllResources` invoke V3 transaction with zero `l2_gas` and `l1_data_gas` via JSON-RPC. The originating gateway accepts it (the stateless validator permits zero resource bounds). The transaction enters the originating node's mempool. When the mempool P2P propagator broadcasts it, every receiving peer rejects it with `DEPRECATED_RESOURCE_BOUNDS_ERROR`. The transaction is permanently stranded on the originating node. If that node is not the block proposer, the transaction is never sequenced.

This matches: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

Any user who submits an `AllResources` invoke V3 transaction specifying only L1 gas (leaving L2 gas and L1 data gas at zero) triggers this path. This is a natural usage pattern for users who do not need L2 gas or L1 data gas. The gateway imposes no minimum on these fields, so the condition is trivially reachable without any special privileges.

### Recommendation

Replace the zero-value heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit discriminator field in the protobuf `ResourceBounds` message (e.g., a `bool is_all_resources` or a `oneof` variant). Until the wire format is updated, the deserializer should treat a present-but-zero `l1_data_gas` field as `AllResources`, reserving the `L1Gas` path only for messages where `l1_data_gas` is `None` (the pre-0.13.3 absent-field case already guarded by the existing TODO comment).

### Proof of Concept

1. Submit via JSON-RPC v0.8:
   ```json
   {
     "type": "INVOKE",
     "version": "0x3",
     "resource_bounds": {
       "l1_gas": { "max_amount": "0x100", "max_price_per_unit": "0x1" },
       "l2_gas": { "max_amount": "0x0",   "max_price_per_unit": "0x0" },
       "l1_data_gas": { "max_amount": "0x0", "max_price_per_unit": "0x0" }
     },
     ...
   }
   ```
2. The originating node accepts the transaction; `InternalRpcInvokeTransactionV3.resource_bounds` is `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`.
3. The mempool P2P propagator serializes it to `protobuf::InvokeV3WithProof`. The wire bytes carry `l1_gas = X`, `l2_gas = 0`, `l1_data_gas = 0`.
4. A peer calls `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`. Because `l1_data_gas.is_zero() && l2_gas.is_zero()`, the result is `ValidResourceBounds::L1Gas(X)`.
5. The peer calls `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`. The `L1Gas` arm hits the `_` branch and returns `Err(DEPRECATED_RESOURCE_BOUNDS_ERROR)`.
6. The peer drops the transaction. The originating node's mempool retains it indefinitely; no other node will ever sequence it.

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
}
