### Title
`CendePreconfirmedTransaction` Invoke Conversion Unconditionally Emits `proof_facts: []` Instead of Omitting the Field for Non-Proving Transactions — (File: `crates/apollo_batcher/src/cende_client_types.rs`)

### Summary
In the `From<InternalRpcTransaction> for CendePreconfirmedTransaction` conversion, the `proof_facts` field of `IntermediateInvokeTransaction` is unconditionally set to `Some(tx.proof_facts)`, even when `tx.proof_facts` is empty (the default for every regular, non-client-side-proving invoke transaction). This causes the JSON payload sent to the Cende service to always include `"proof_facts": []` for regular invoke transactions, instead of omitting the field as the `#[serde(skip_serializing_if = "Option::is_none")]` annotation is designed to enforce.

### Finding Description

`IntermediateInvokeTransaction` in `crates/apollo_batcher/src/cende_client_types.rs` declares the field as:

```rust
#[serde(default, skip_serializing_if = "Option::is_none")]
pub proof_facts: Option<ProofFacts>,
``` [1](#0-0) 

The `skip_serializing_if = "Option::is_none"` annotation is the canonical way to omit the field when there are no proof facts. The field is `None` → absent from JSON; `Some(non-empty)` → present in JSON. This is the invariant the struct's design relies on.

However, the `From<InternalRpcTransaction> for CendePreconfirmedTransaction` implementation at line 505 unconditionally wraps `tx.proof_facts` in `Some(…)`:

```rust
proof_facts: Some(tx.proof_facts),
``` [2](#0-1) 

`ProofFacts` is defined as `pub struct ProofFacts(pub Arc<Vec<Felt>>)` with `#[derive(Default)]`, so `ProofFacts::default()` is an empty vector. [3](#0-2) 

For every regular invoke transaction (no client-side proving), `tx.proof_facts` is `ProofFacts::default()`. The conversion therefore produces `proof_facts: Some(ProofFacts(Arc::new(vec![])))`, which the serde machinery serializes as `"proof_facts": []` — a present-but-empty array — rather than omitting the field entirely.

The correct conversion should be:

```rust
proof_facts: if tx.proof_facts.is_empty() { None } else { Some(tx.proof_facts) },
```

The discrepancy is confirmed by the companion test in `apollo_consensus_orchestrator`, which explicitly uses `proof_facts: None` for the standard (non-proving) case:

```rust
// Test without proof_facts (standard flow without client-side proving).
proof_facts: None,
``` [4](#0-3) 

The `apollo_batcher` conversion path is the one that feeds the Cende preconfirmed-block endpoint; it is a separate code path from the `apollo_consensus_orchestrator` central-objects path and has no equivalent guard.

### Impact Explanation

The Cende service (Python central service) receives `"proof_facts": []` for **every** regular invoke transaction instead of the field being absent. This breaks the serialization contract at the batcher → Cende boundary.

If the Python side uses the **presence** of the `proof_facts` key (rather than its truthiness) to decide whether a transaction carries client-side-proving data — e.g., `if proof_facts is not None:` — it will incorrectly classify all regular invoke transactions as proof-carrying. Downstream effects include:

* Wrong transaction metadata stored in the Cende preconfirmed-block record, which is the authoritative source for pending-block RPC responses (`starknet_getTransactionByHash`, `starknet_getBlockWithTxs`, etc.) — matching **High: RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**.
* Potential schema-validation rejection of the preconfirmed block by the Python side if it enforces that `proof_facts` must be absent for non-proving transactions, which would prevent the sequencer from writing any preconfirmed block — matching **High: Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing**.

### Likelihood Explanation

Every regular invoke transaction (the overwhelming majority of all Starknet transactions) triggers this path. No special privileges, crafted inputs, or adversarial peers are required; the bug fires unconditionally for any invoke transaction that does not carry proof facts.

### Recommendation

In `crates/apollo_batcher/src/cende_client_types.rs`, change line 505 from:

```rust
proof_facts: Some(tx.proof_facts),
```

to:

```rust
proof_facts: if tx.proof_facts.is_empty() { None } else { Some(tx.proof_facts) },
```

This aligns the batcher's serialization with the struct's `skip_serializing_if = "Option::is_none"` contract and with the expected behavior documented in the `apollo_consensus_orchestrator` tests.

### Proof of Concept

1. Submit any standard invoke transaction (no `proof_facts`, no `proof`) to the sequencer gateway.
2. The gateway converts it to `InternalRpcTransaction`; `tx.proof_facts` is `ProofFacts::default()` (empty). [5](#0-4) 
3. The batcher receives the transaction and calls `CendePreconfirmedTransaction::from(internal_rpc_transaction)`.
4. Inside the `Invoke` arm, `proof_facts: Some(tx.proof_facts)` is set with an empty `ProofFacts`. [6](#0-5) 
5. `serde_json::to_value(...)` produces `"proof_facts": []` in the JSON body sent to the Cende recorder URL.
6. The Cende service receives `"proof_facts": []` for a transaction that carries no proof, diverging from the expected absent-field representation.
