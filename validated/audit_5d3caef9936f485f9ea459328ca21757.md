### Title
Missing `max_l2_gas_amount` Upper-Bound Check for `Declare` Transactions Allows Oversized Gas Bounds Through Gateway Admission - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator` enforces a `max_l2_gas_amount` ceiling on `l2_gas.max_amount` for Invoke and DeployAccount transactions, but this check is explicitly skipped for Declare transactions via an acknowledged TODO. A user can submit a `RpcDeclareTransactionV3` with an arbitrarily large `l2_gas.max_amount` (e.g., `u64::MAX`) and it will pass all gateway stateless validation, entering the mempool. This is the direct sequencer analog of the "Absence of Checks for Max TTL" pattern: a named maximum constant/config value exists and is enforced on some code paths but silently omitted on another.

### Finding Description

In `validate_resource_bounds`, the gateway checks that `l2_gas.max_amount` does not exceed `self.config.max_l2_gas_amount` (production value: `1,210,000,000`). However, the check is guarded by an explicit early-return for Declare transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The production config schema confirms `max_l2_gas_amount = 1,210,000,000` is a live, enforced limit: [2](#0-1) 

The test `valid_l2_gas_amount_on_declare` explicitly documents and asserts this bypass — a Declare transaction with `l2_gas.max_amount = 200` against a config of `max_l2_gas_amount = 100` returns `Ok(())`: [3](#0-2) 

For Invoke and DeployAccount, the same config value correctly rejects oversized amounts: [4](#0-3) 

### Impact Explanation

A Declare transaction with `l2_gas.max_amount` set to any value above `max_l2_gas_amount` (e.g., `u64::MAX`) passes all stateless gateway checks and is admitted to the mempool. The `max_l2_gas_amount` limit exists specifically to prevent transactions with unreasonably large gas bounds from entering the sequencer pipeline. Bypassing it for Declare transactions means:

1. **Mempool admission of policy-invalid transactions**: Declare transactions that should be rejected at the gateway boundary are accepted and queued.
2. **Fee pre-validation interaction**: `max_possible_fee` is computed as `l2_gas.max_amount * l2_gas.max_price_per_unit`. With `max_amount = u64::MAX` and the required `min_gas_price = 8,000,000,000`, this saturates to `u128::MAX`, causing the pre-validation balance check to reject the transaction only at execution time — after it has already consumed mempool resources. With a moderately oversized amount (e.g., `max_l2_gas_amount + 1`), the transaction can proceed all the way to execution.

This matches the **High** impact: "Mempool/gateway/RPC admission accepts invalid transactions before sequencing."

### Likelihood Explanation

Any unprivileged user submitting a `RpcDeclareTransactionV3` can set `l2_gas.max_amount` to any value. No special access, privileged key, or malformed bytes are required. The bypass is unconditional for all Declare transactions.

### Recommendation

Remove the Declare-specific early-return and apply the same `max_l2_gas_amount` upper-bound check uniformly to all transaction types, resolving the TODO:

```rust
fn validate_resource_bounds(&self, tx: &RpcTransaction) -> StatelessTransactionValidatorResult<()> {
    if !self.config.validate_resource_bounds {
        return Ok(());
    }
    let resource_bounds = *tx.resource_bounds();
    if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
        return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
    }
    if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
        return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
    }
    // Apply to ALL transaction types, including Declare:
    if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
        return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
    }
    Ok(())
}
```

### Proof of Concept

1. Construct a `RpcDeclareTransactionV3` with:
   - `l2_gas.max_amount = GasAmount(u64::MAX)` (or any value `> 1_210_000_000`)
   - `l2_gas.max_price_per_unit = GasPrice(8_000_000_001)` (above `min_gas_price`)
   - Any valid Sierra contract class
2. Submit to the gateway's `add_transaction` endpoint.
3. Observe: the transaction passes `StatelessTransactionValidator::validate` without error, despite `l2_gas.max_amount` exceeding `max_l2_gas_amount = 1_210_000_000`.
4. For comparison, submit an equivalent `RpcInvokeTransactionV3` with the same `l2_gas.max_amount` — it is correctly rejected with `MaxGasAmountTooHigh`.

The divergence is confirmed by the existing test `valid_l2_gas_amount_on_declare` which asserts `Ok(())` for a Declare transaction exceeding the configured limit: [3](#0-2)
