### Title
Stale Chain-Tip Used Without Freshness Check Allows False Transaction Inclusion Verification — (`contract/src/lib.rs`)

### Summary

The `BtcLightClient` contract's `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` functions derive their confirmation count from `mainchain_tip_blockhash`, a value that is exclusively updated by the off-chain relayer via `submit_blocks`. The contract stores no timestamp of the last header submission and performs no staleness check before answering proof queries. If the relayer halts for any reason while a Bitcoin reorg simultaneously removes a previously-confirmed transaction, the contract will continue to return `true` for that transaction's inclusion proof, because its internal chain state has not been updated to reflect the reorg.

### Finding Description

**Root cause — no staleness guard on the proof path.**

`verify_transaction_inclusion` reads `mainchain_tip_blockhash` to compute the confirmation depth: [1](#0-0) 

The tip is set only inside `submit_block_header_inner`, which is reachable exclusively through `submit_blocks`: [2](#0-1) 

`submit_blocks` is gated by `#[trusted_relayer]`, meaning only the off-chain relayer bot can advance the tip: [3](#0-2) 

The contract state struct contains no `last_submission_timestamp` or equivalent field: [4](#0-3) 

Therefore, `verify_transaction_inclusion` has no information about how old the stored chain state is, and no mechanism to reject or flag a query when the tip is stale.

**Attack path.**

1. Attacker submits Bitcoin transaction T; it is mined into block B at height H.
2. The relayer submits headers up to height H+C (C ≥ `confirmations`). The contract's `mainchain_header_to_height` maps B's hash to height H; `mainchain_tip_blockhash` points to height H+C.
3. A Bitcoin reorg of depth ≥ 1 occurs, orphaning block B and evicting T from the canonical chain.
4. The relayer fails (bug, network outage, infrastructure issue) before it can submit the reorg headers.
5. Attacker calls `verify_transaction_inclusion` (or an external DApp does so on the attacker's behalf) with T's proof.
6. The contract checks `mainchain_header_to_height` — B is still present because the reorg was never relayed — and computes `(H+C) - H + 1 >= confirmations`. The function returns `true`.
7. The consuming DApp releases assets or records settlement based on a transaction that no longer exists on Bitcoin.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after the coinbase check, so it is equally affected: [5](#0-4) 

### Impact Explanation

An external NEAR DApp that gates asset release on `verify_transaction_inclusion` returning `true` can be deceived into accepting a Bitcoin transaction that has been reorged out of the canonical chain. The contract provides no signal that its chain state is stale, so the DApp has no on-chain way to detect the condition. The corrupted value is the proof result (`bool`) returned to the caller, and the broken invariant is: *a `true` result must correspond to a transaction that is currently in the canonical Bitcoin chain with at least `confirmations` depth*.

### Likelihood Explanation

Relayer outages are a realistic and documented failure mode for off-chain bots (network partitions, RPC node downtime, software bugs). Bitcoin reorgs of 1–2 blocks occur several times per year on mainnet; shallow-confirmation DApps (e.g., `confirmations = 1` or `2`) are directly in the risk window. The attacker does not need to cause the relayer to fail — they only need to monitor for the condition and submit a proof query during the staleness window, which is observable on-chain by comparing the contract's reported tip height against a public Bitcoin block explorer.

### Recommendation

Add a `last_submission_timestamp_ms: u64` field to `BtcLightClient` and update it inside `submit_block_header_inner` (or `submit_blocks`) on every successful header ingestion. In `verify_transaction_inclusion` and `verify_transaction_inclusion_v2`, assert that `env::block_timestamp_ms() - last_submission_timestamp_ms` is below a configurable `max_staleness_ms` threshold before proceeding. Callers that need a guaranteed-fresh answer can be required to ensure the relayer is live; callers that can tolerate some lag can use a looser threshold. This mirrors the "max staleness" fix acknowledged by BadgerDAO in the referenced report.

### Proof of Concept

**Setup:**
- Deploy the Bitcoin variant of the contract with `confirmations` threshold of 1.
- Relayer submits headers for heights 100–101 (tip = 101). Transaction T is in block 100.
- Call `verify_transaction_inclusion` with T's proof and `confirmations = 1` → returns `true` (correct).

**Attack:**
- Bitcoin reorg occurs: block 100 is orphaned, T is evicted.
- Relayer goes offline before submitting the reorg.
- Contract state: `mainchain_tip_blockhash` still points to height 101; block 100's hash is still in `mainchain_header_to_height`.
- Call `verify_transaction_inclusion` again with the same proof → still returns `true` (incorrect).
- DApp releases assets based on a transaction that no longer exists on Bitcoin.

The contract has no code path that can detect or reject this scenario because `last_submission_timestamp_ms` does not exist in the state and no staleness guard exists in either verification function. [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
