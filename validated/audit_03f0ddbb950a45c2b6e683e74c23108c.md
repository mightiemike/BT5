### Title
`ghost_heap` Counter Not Updated in `Allocator::new_substr` for SmallAtom Input, Allowing Heap Limit Bypass — (File: src/allocator.rs)

---

### Summary

In `Allocator::new_substr`, when the input node is a `SmallAtom`, neither the `ghost_heap` counter nor the heap limit check is applied correctly. When the substr result fits in a `SmallAtom`, `ghost_atoms` is incremented but `ghost_heap` is not — directly contradicting the behavior of `new_atom` and `new_small_number`, which both update `ghost_heap` for SmallAtom results. When the substr result does *not* fit in a `SmallAtom`, bytes are written directly into `u8_vec` with no heap limit check at all. An attacker-controlled CLVM program can exploit `op_substr` (opcode 12) to underreport virtual heap usage and bypass the configured `heap_limit`.

---

### Finding Description

`Allocator::new_atom` enforces the heap limit before allocating and updates `ghost_heap` for SmallAtom results:

```rust
// src/allocator.rs ~619-638
pub fn new_atom(&mut self, v: &[u8]) -> Result<NodePtr> {
    let start = self.u8_vec.len() as u32;
    if start as usize + self.ghost_heap + v.len() > self.heap_limit {
        return Err(EvalErr::OutOfMemory);
    }
    ...
    if let Some(ret) = fits_in_small_atom(v) {
        self.ghost_atoms += 1;
        self.ghost_heap += v.len();   // ← ghost_heap updated
        Ok(self.mk_node(ObjectType::SmallAtom, ret as usize))
    } else { ... }
}
```

`new_small_number` does the same:

```rust
// src/allocator.rs ~640-650
pub fn new_small_number(&mut self, v: u32) -> Result<NodePtr> {
    let len = len_for_value(v);
    if self.u8_vec.len() + self.ghost_heap + len > self.heap_limit {
        return Err(EvalErr::OutOfMemory);
    }
    self.check_atom_limit()?;
    self.ghost_atoms += 1;
    self.ghost_heap += len;   // ← ghost_heap updated
    Ok(self.mk_node(ObjectType::SmallAtom, v as usize))
}
```

But `new_substr` for the `SmallAtom` input branch does neither:

```rust
// src/allocator.rs ~845-867
ObjectType::SmallAtom => {
    let val = node.index();
    let len = len_for_value(val) as u32;
    bounds_check(node, start, end, len)?;
    let buf: [u8; 4] = val.to_be_bytes();
    let buf = &buf[4 - len as usize..];
    let substr = &buf[start as usize..end as usize];
    if let Some(new_val) = fits_in_small_atom(substr) {
        self.ghost_atoms += 1;
        // ← ghost_heap NOT updated (missing: self.ghost_heap += substr.len())
        Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
    } else {
        let start = self.u8_vec.len();
        let end = start + substr.len();
        self.u8_vec.extend_from_slice(substr);   // ← bytes written to heap
        // ← NO heap limit check before this write
        let idx = self.atom_vec.len();
        self.atom_vec.push(AtomBuf { start: start as u32, end: end as u32 });
        Ok(self.mk_node(ObjectType::Bytes, idx))
    }
}
```

**Issue 1 (SmallAtom → SmallAtom):** `ghost_heap` is not incremented by `substr.len()`. Because `heap_size()` is defined as `u8_vec.len() + ghost_heap`, the virtual heap size is underreported by up to 4 bytes per call. Repeated `substr` calls on SmallAtoms accumulate this deficit, allowing subsequent `new_atom`/`new_small_number`/`new_concat` calls to pass the heap limit check when they should fail.

**Issue 2 (SmallAtom → non-SmallAtom):** When a SmallAtom substr produces a byte sequence with the high bit set (e.g., extracting `[0x80]` from `0x0180`), `fits_in_small_atom` returns `None` and bytes are written directly into `u8_vec` with no heap limit check whatsoever. This is a complete bypass of the limit for this allocation path.

The `ghost_heap` counter exists specifically for consensus-safe accounting of "virtual" allocations — bytes that are logically allocated but stored inline (SmallAtoms) or freed via checkpoint restore. Omitting the update breaks the invariant that `heap_size()` accurately reflects total virtual heap usage.

---

### Impact Explanation

The `heap_limit` enforced by `Allocator::new_limited` is the mechanism used to cap memory consumption per CLVM program execution. The `LIMIT_HEAP` flag is part of `MEMPOOL_MODE` (`ClvmFlags::NO_UNKNOWN_OPS | LIMIT_HEAP | DISABLE_OP | CANONICAL_INTS | LIMIT_SOFTFORK`), indicating that heap-limited allocators are used in mempool validation. A CLVM program submitted to the mempool can use `op_substr` (opcode 12) on SmallAtom values to:

- Underreport `ghost_heap` by up to 4 bytes per call (Issue 1), accumulating a deficit that allows later allocations to bypass the limit.
- Directly write bytes into `u8_vec` without any limit check (Issue 2), bypassing the limit entirely for that allocation.

The result is that the actual heap usage (`u8_vec.len() + ghost_heap`) can exceed the configured `heap_limit`, violating the resource bound that mempool mode is designed to enforce. This could allow a malicious transaction to consume more memory than permitted, potentially causing memory exhaustion or consensus divergence between nodes with different resource accounting.

---

### Likelihood Explanation

`op_substr` is a standard CLVM opcode (opcode 12), reachable by any attacker-controlled CLVM program. SmallAtoms are common — they represent small positive integers (up to 26 bits) and appear frequently in CLVM programs as constants, path indices, and intermediate values. Extracting a high-bit-set byte from a multi-byte SmallAtom (e.g., `substr(0x0180, 1, 2)` → `[0x80]`) is trivial to craft. The bypass is deterministic and requires no special privileges. The only precondition is that the caller uses `Allocator::new_limited` with a meaningful limit, which is the expected behavior in mempool mode.

---

### Recommendation

In `new_substr` for the `SmallAtom` input branch:

1. Add a heap limit check before any allocation (mirroring `new_atom`):
   ```rust
   if self.u8_vec.len() + self.ghost_heap + substr.len() > self.heap_limit {
       return Err(EvalErr::OutOfMemory);
   }
   ```

2. For the SmallAtom → SmallAtom path, add:
   ```rust
   self.ghost_heap += substr.len();
   ```
   to match the accounting in `new_atom` and `new_small_number`.

### Proof of Concept

```rust
// Demonstrates ghost_heap underreporting via repeated substr on SmallAtoms
let heap_limit = 100;
let mut a = Allocator::new_limited(heap_limit);

// 0x0180 fits in a SmallAtom (positive, 2 bytes, top bit clear)
let small = a.new_atom(&[0x01, 0x80]).unwrap();
// small is stored as SmallAtom; ghost_heap += 2 (correct in new_atom)

// Now extract [0x80] via substr — this is a non-SmallAtom result
// new_substr SmallAtom branch: NO heap limit check, bytes go into u8_vec
for _ in 0..50 {
    let _ = a.new_substr(small, 1, 2).unwrap(); // each call writes 1 byte, no limit check
}
// u8_vec.len() is now 50, exceeding the heap_limit of 100 only after ghost_heap
// is also considered — but ghost_heap was never updated for these calls,
// so subsequent new_atom calls see a falsely low ghost_heap and pass the check.

// For Issue 1: repeated SmallAtom→SmallAtom substr
let small2 = a.new_atom(&[0x01]).unwrap(); // ghost_heap += 1
for _ in 0..200 {
    let _ = a.new_substr(small2, 0, 1).unwrap();
    // ghost_atoms += 1 each time, but ghost_heap NOT incremented
    // heap_size() underreported by 1 byte per call
}
// After 200 calls, ghost_heap is underreported by 200 bytes,
// allowing 200 extra bytes to be allocated past the heap_limit.
