Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the initiating user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the direct caller of the pool. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the allowlist check evaluates the router's allowlist status rather than the actual initiating user's. Any unprivileged user can bypass a curated pool's swap allowlist by calling through the official periphery router.

## Finding Description
`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` receives that value as `sender` and checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called the pool: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` (and all other swap entry points) calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making the router `msg.sender` to the pool: [3](#0-2) 

The result is an irreconcilable actor mismatch:
- If the pool admin allowlists the router (the natural action to allow periphery usage), **every user** bypasses the individual allowlist by routing through it.
- If the pool admin does not allowlist the router, **no user** can swap through the router even if individually allowlisted, breaking periphery for curated pools.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the `owner` parameter (the actual position beneficiary), not the direct caller: [4](#0-3) 

The asymmetry between the two extensions confirms the swap path has the wrong actor bound.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC'd addresses, whitelisted institutions) can be freely traded against by any unprivileged user via `MetricOmmSimpleRouter`. The allowlist guard is silently bypassed, exposing LP funds to trades the pool admin explicitly intended to block. This constitutes broken core pool functionality and an admin-boundary break via an unprivileged path — both qualifying impacts under the contest rules.

**Severity: High**

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard, publicly documented periphery entry point. No special privilege is required — any user aware of the router can exploit this. The pool admin has no on-chain mechanism to simultaneously allow allowlisted users to trade through the router and block non-allowlisted users from doing the same.

**Likelihood: High**

## Recommendation
Gate on the actual initiating user rather than the direct pool caller. The preferred fix mirrors `DepositAllowlistExtension`'s pattern:

1. **Preferred**: Extend the `beforeSwap` hook signature (e.g., via `extensionData`) to carry the true initiating address, have the router populate it, and check that address in the extension.
2. **Alternative**: If swap semantics guarantee `recipient` equals the economic beneficiary, gate on `recipient` instead of `sender`. This is weaker if `recipient` can be set to a third party.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Admin calls `setAllowedToSwap(pool, router, true)` to allow the official router (necessary for any router-mediated swap to work).
3. Attacker (not individually allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the curated pool.
4. The router calls `IMetricOmmPoolActions(pool).swap(...)`, making itself `msg.sender` to the pool.
5. The pool calls `_beforeSwap(msg.sender=router, ...)`.
6. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` → `true` → no revert.
7. The attacker's swap executes successfully despite not being on the allowlist.
