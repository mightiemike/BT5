### Title
`SwapAllowlistExtension` checks the router address as the swapper, not the originating user ŌĆö allowlist is bypassed for all router-mediated swaps or blocks all allowlisted users from using the router - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed from the pool. The pool always passes its own `msg.sender` as `sender`. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router**, not the originating EOA. The allowlist therefore checks the router's address, not the individual user's address. This creates two mutually exclusive failure modes: (1) if the router is not allowlisted, every allowlisted user is silently blocked from using the router; (2) if the pool admin allowlists the router to restore router access, every non-allowlisted user can bypass the curated gate by routing through the router.

---

### Finding Description

**Call chain for a router swap:**

```
EOA ŌåÆ MetricOmmSimpleRouter.exactInputSingle()
        ŌåÆ pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)
              msg.sender = router
          ŌåÆ MetricOmmPool._beforeSwap(msg.sender /*= router*/, ...)
              ŌåÆ ExtensionCalling._callExtensionsInOrder(...)
                  ŌåÆ SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        checks: allowedSwapper[pool][router]
```

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool passed ŌĆö the router address when the swap originates from the router: [3](#0-2) 

The router never forwards the originating EOA's identity to the pool. It stores the EOA only in its own transient callback context (for payment settlement), but the pool's `swap` call receives `msg.sender = router`: [4](#0-3) 

The same wrong-actor binding applies to `exactInput` multi-hop swaps, where the first hop's payer is `msg.sender` (the EOA) but the pool still receives `msg.sender = router`: [5](#0-4) 

---

### Impact Explanation

**Failure mode A ŌĆö router not allowlisted (default):** Every user who is individually allowlisted on a curated pool is silently blocked from using `MetricOmmSimpleRouter`. The router is not in `allowedSwapper[pool]`, so every router-mediated call reverts with `NotAllowedToSwap`. Allowlisted users must call the pool directly, losing access to multi-hop routing, deadline enforcement, and slippage protection. This is broken core pool functionality for the intended user population.

**Failure mode B ŌĆö pool admin allowlists the router to restore router access:** The entry `allowedSwapper[pool][router] = true` passes for every EOA that routes through the router, regardless of whether that EOA is individually allowlisted. Any unprivileged user can bypass the curated gate by calling `exactInputSingle` or `exactInput` on the router. The allowlist provides zero protection against router-mediated swaps. This is a direct allowlist bypass with fund-impacting consequences on pools that rely on the gate for access control (e.g., KYC pools, private liquidity pools, or pools with restricted counterparties).

---

### Likelihood Explanation

The router is the primary user-facing swap interface documented and deployed alongside the pool. Pool admins who configure a `SwapAllowlistExtension` and also want their allowlisted users to access the router will inevitably encounter failure mode A and attempt to resolve it by allowlisting the router, triggering failure mode B. The trigger requires no special privilege ŌĆö any EOA can call `MetricOmmSimpleRouter.exactInputSingle` with a valid pool address.

---

### Recommendation

The pool must pass the **originating user's address** to the extension, not its own `msg.sender`. Two complementary fixes are needed:

1. **Router side:** `MetricOmmSimpleRouter` should encode the originating `msg.sender` into `extensionData` (or a dedicated field) so the pool can forward it to extensions.
2. **Extension side:** `SwapAllowlistExtension.beforeSwap` should read the originating user from the `extensionData` bytes rather than from the `sender` argument, or the pool interface should be extended with a dedicated `originator` field that the router populates.

Alternatively, the pool's `swap` signature can be extended with an explicit `originator` address that the router fills with `msg.sender` before calling the pool, and `ExtensionCalling._beforeSwap` forwards that field to extensions instead of the pool's `msg.sender`.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted to enable router access

Attack:
  - bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  - router calls pool.swap(...) with msg.sender = router
  - pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true  ŌåÆ passes
  - bob's swap executes successfully despite not being allowlisted

Result:
  - The curated pool's allowlist is completely bypassed for any user going through the router
  - alice's individual allowlist entry is irrelevant for router-mediated swaps
