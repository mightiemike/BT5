### Title
Mutable `Tier.BonusApy` Applied Retroactively to Historical Reward Segments in `processEventsAndClaimBonus` - (File: `x/tieredrewards/keeper/claim_rewards.go`)

### Summary
The `processEventsAndClaimBonus` function fetches the **current** tier state at claim time and applies its `BonusApy` to all historical time segments since the position's last accrual checkpoint. Because governance can update a tier's `BonusApy` via `SetTier`, any unclaimed historical bonus is retroactively recalculated at the new rate rather than the rate in effect during each segment. This mirrors the external report's root cause: a mutable reference whose update silently corrupts calculations that depend on historical state.

### Finding Description

`processEventsAndClaimBonus` walks the validator event log from `pos.LastEventSeq` forward, computing a bonus for each bonded segment. The tier is fetched once at the top of the function using the **current** store state: [1](#0-0) 

That single `tier` object — carrying the current `BonusApy` — is then passed into every call to `computeSegmentBonus` for every historical segment: [2](#0-1) 

`computeSegmentBonus` multiplies `tier.BonusApy` directly against the segment duration: [3](#0-2) 

Notice the asymmetry: `tokensPerShare` is a **historical snapshot** stored in the validator event at the time it was emitted, but `tier.BonusApy` is the **live value** at claim time. If governance calls `SetTier` between the last claim and the current claim, every historical segment is repriced at the new rate.

`SetTier` is the governance-accessible write path for tier parameters: [4](#0-3) 

There is a helper `claimRewardsAndUpdateTierPositions` that could checkpoint all positions before a tier update, but its invocation from the governance message handler (`msg_server.go`) could not be confirmed in this analysis. Even if it is called, any position created or modified between the checkpoint and the tier update would still be affected.

### Impact Explanation

Two concrete fund-loss scenarios arise:

1. **Overpayment (rewards pool drained):** Governance raises `BonusApy` from 5 % to 10 %. A user who has not claimed for a long period calls `ClaimRewards`. All historical segments — earned at 5 % — are recalculated at 10 %. The `RewardsPoolName` module account pays out up to 2× the legitimately accrued bonus.

2. **Underpayment (user funds lost):** Governance lowers `BonusApy` from 10 % to 5 %. The same user receives half the bonus they legitimately earned. The shortfall is permanently lost to the user because the accrual checkpoint is advanced after the (under)payment.

The corrupted value is the `bonusCoins` amount sent from `types.RewardsPoolName` to the position owner: [5](#0-4) 

### Likelihood Explanation

Governance parameter changes to tiers are an expected, documented operation (`SetTier` comment: *"Used by governance messages, genesis, and chain upgrades"*). Any governance proposal that adjusts `BonusApy` — even a well-intentioned one — silently reprices all unclaimed historical bonus for every position in that tier. The longer the interval since the last claim, the larger the discrepancy. This is not a theoretical edge case; it is triggered by routine protocol governance.

### Recommendation

Store a `BonusApy` snapshot inside each `Position` (or inside each validator event entry) at the time the position is created or the rate changes. `computeSegmentBonus` should use the snapshotted rate for each segment rather than the live tier value. Alternatively, mandate that `claimRewardsAndUpdateTierPositions` is called atomically before any `SetTier` write, and enforce this in the message server so no unclaimed historical segments can exist at the old rate when the tier is updated.

### Proof of Concept

1. Alice creates a position in tier 1 (`BonusApy = 0.05`, i.e. 5 %).
2. Alice delegates and accrues bonus for 180 days without claiming.
3. Governance passes `MsgUpdateTier` setting `BonusApy = 0.10` (10 %).
4. Alice calls `MsgClaimRewards`.
5. `processEventsAndClaimBonus` fetches tier 1 with `BonusApy = 0.10`.
6. All 180-day historical segments are computed at 10 % via `computeSegmentBonus`.
7. Alice receives ~2× the bonus she legitimately earned; the `RewardsPoolName` account is debited the excess.

The inverse (governance lowers `BonusApy`) causes Alice to receive ~½ the bonus she earned, with no recourse.
