### Title
Missing `validate_proof_size` call in `StatelessTransactionValidator::validate` allows oversized proofs through gateway admission — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator::validate` function never calls the `validate_proof_size` helper that enforces the `max_proof_size` config limit. A user can submit an Invoke V3 transaction carrying a proof payload larger than the configured 480,000-byte ceiling and the gateway will accept it, bypassing the only admission-time size gate.

### Finding Description

`StatelessTransactionValidator` defines a `validate_proof_size` method that checks `proof.0.len() > self.config.max_proof_size` and returns `StatelessTransactionValidatorError::ProofTooLarge` on violation. The `StatelessTransactionValidatorConfig` exposes `max_proof_size` (default 480,000 bytes, serialised in `config_schema.json`) as a first-class public parameter. Despite this, the top-level `validate` entry point never invokes `validate_proof_size`:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs  lines 31-53
pub fn validate(&self, tx: &RpcTransaction) -> StatelessTransactionValidatorResult<()> {
    Self::validate_contract_address(tx)?;
    Self::validate_empty_account_deployment_data(tx)?;
    Self::validate_empty_paymaster_data(tx)?;
    self.validate_resource_bounds(tx)?;
    self.validate_tx_size(tx)?;
    self.validate_nonce_data_availability_mode(tx)?;
    self.validate_fee_data_availability_mode(tx)?;

    if let RpcTransaction::Invoke(invoke_tx) = tx {
        self.validate_client_side_proving_allowed(invoke_tx)?;
        self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
        // ← validate_proof_size is never called here
    }
    ...
}
``` [1](#0-0) 

The dead method sits at:

```rust
// lines 265-278
fn validate_proof_size(
    &self,
    tx: &RpcInvokeTransaction,
) -> StatelessTransactionValidatorResult<()> {
    let RpcInvokeTransaction::V3(tx) = tx;
    let proof_size = tx.proof.0.len();
    if proof_size > self.config.max_proof_size {
        return Err(StatelessTransactionValidatorError::ProofTooLarge {
            proof_size,
            max_proof_size: self.config.max_proof_size,
        });
    }
    Ok(())
}
``` [2](#0-1) 

The config field and its schema entry confirm the limit is intentional and operator-visible: [3](#0-2) [4](#0-3) 

The test harness sets `max_proof_size: 10` to exercise small proofs, but no test exercises the rejection path because `validate_proof_size` is never wired in: [5](#0-4) 

### Impact Explanation

Any caller of the public `add_tx` RPC endpoint can submit an `RpcInvokeTransactionV3` whose `proof` field is arbitrarily large. The gateway accepts it, the transaction converter spawns a `spawn_proof_verification` task that passes the full oversized blob to `starknet_proof_verifier::verify_proof`, and the transaction is forwarded to the mempool. Neither `verify_proof` nor the proof manager imposes an independent byte-count guard: [6](#0-5) [7](#0-6) 

This matches the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

### Likelihood Explanation

The trigger is fully unprivileged: any user with RPC access can craft an `RpcInvokeTransactionV3` with an arbitrarily large `proof` byte vector. No special account, key, or network position is required. The `allow_client_side_proving` flag is `true` by default in production config, so the only remaining admission gate — `validate_client_side_proving_allowed` — passes without inspecting size. [8](#0-7) [9](#0-8) 

### Recommendation

Add the missing call inside the `RpcTransaction::Invoke` branch of `validate`:

```rust
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;
    self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
    self.validate_proof_size(invoke_tx)?;   // ← add this line
}
```

Add a corresponding test case that submits a proof whose length exceeds `max_proof_size` and asserts `StatelessTransactionValidatorError::ProofTooLarge` is returned.

### Proof of Concept

1. Construct an `RpcInvokeTransactionV3` with `proof: Proof(Arc::new(vec![0u8; 480_001]))` and valid `proof_facts`.
2. Submit via `POST /gateway/add_transaction` to a node with `allow_client_side_proving: true`.
3. Observe the gateway returns a transaction hash (accepted) rather than a `ProofTooLarge` error, confirming the size gate is absent.
4. Confirm `validate_proof_size` would have caught it: call it directly in a unit test with `max_proof_size: 10` and a proof of length 11 — it returns `Err(ProofTooLarge { proof_size: 11, max_proof_size: 10 })` — but this path is never reached from `validate`.

    fn validate_client_side_proving_allowed(
        &self,
        tx: &RpcInvokeTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if self.config.allow_client_side_proving {
            return Ok(());
        }

        // Reject V3 transactions with proofs when client-side proving is disabled.
        let RpcInvokeTransaction::V3(tx) = tx;
        let has_proof_data = !tx.proof_facts.is_empty() || !tx.proof.is_empty();
        if has_proof_data {
            return Err(StatelessTransactionValidatorError::ClientSideProvingNotAllowed);
        }

        Ok(())
    }
