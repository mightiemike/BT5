Audit Report

## Title
Stop-Loss Mid-Price Uses Arithmetic Mean While Oracle Mid Uses Geometric Mean, Causing Systematic Guard Miscalibration - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

## Summary
`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the arithmetic mean of bid and ask prices, while `SwapMath.midAndSpreadFeeX64FromBidAsk` (used by the pool for oracle mid derivation and by `PriceVelocityGuardExtension`) uses the geometric mean. By AM-GM inequality the arithmetic mean is always ‚Č• the geometric mean, so `metricT1` is systematically inflated, making `breach1` less likely to fire and allowing `zeroForOne == false` swaps (token0 in, token1 out) to proceed past the configured drawdown floor.

## Finding Description
In `_afterSwapOracleStopLoss` (line 218):
```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2; // arithmetic mean
```
The pool's oracle mid price is defined by `SwapMath.midAndSpreadFeeX64FromBidAsk` (line 70):
```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64); // geometric mean
```
`PriceVelocityGuardExtension.beforeSwap` (line 48) correctly calls `SwapMath.midAndSpreadFeeX64FromBidAsk`. The per-share metric formulas (lines 254‚Äď255) use `midPriceX64` multiplicatively for `metricT1`:
```solidity
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
```
Because `mid_arithmetic ‚Č• mid_geometric`, `metricT1` is inflated relative to the value the pool's own oracle mid would produce. The blocking check (lines 275‚Äď277):
```solidity
(uint256 hwm1, bool breach1) = _applyWatermark(metricT1, _decayed(hwmS.token1, decayRate, dt), floorMultiplier);
if (breach1 && !zeroForOne) { revert OracleStopLossTriggered(...); }
```
compares an inflated `metricT1` against the watermark floor, so `breach1` fires later than it should. The actual per-share value in token1 terms at the pool's oracle mid (geometric mean) can fall below the configured floor without triggering the stop-loss. No existing guard corrects for this: the arithmetic mean is hardcoded and there is no fallback check.

## Impact Explanation
The contract's documented invariant (lines 27‚Äď28) states: "value per share at oracle marks cannot fall faster than drawdown (one-time) + decay √ó t (ongoing)." Because `metricT1` is evaluated at a price higher than the pool's oracle mid, the guard compares the watermark floor against an inflated metric. The actual per-share value in token1 terms (at the geometric mean) can fall below the configured floor without triggering the stop-loss. LPs suffer direct loss of token1 principal through swaps the guard was configured to block. The discrepancy is `(AM ‚ąí GM) / GM ‚Čą s¬≤/8`; for a 10% spread this is ‚Čą 0.125%, for a 50% spread ‚Čą 3.1%, and it is always in the direction that suppresses the token1-side guard. This constitutes a broken core stop-loss invariant causing direct LP principal loss, qualifying as Medium/High under Sherlock thresholds.

## Likelihood Explanation
Every public swap on a pool with `OracleValueStopLossExtension` configured in its `afterSwap` order triggers this path via `afterSwap` ‚Üí `_afterSwapOracleStopLoss`. No special permissions, privileged roles, or unusual setup are required beyond the pool existing with the extension active. The discrepancy grows with the oracle spread, which is largest precisely when the guard is most needed (volatile or manipulated markets). Any unprivileged trader can execute swaps that exploit this miscalibration.

## Recommendation
Replace the arithmetic mean with the geometric mean, consistent with `SwapMath.midAndSpreadFeeX64FromBidAsk` and `PriceVelocityGuardExtension`:
```solidity
// Before (arithmetic mean ‚ÄĒ inconsistent with pool oracle mid):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (geometric mean ‚ÄĒ consistent with SwapMath and PriceVelocityGuardExtension):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64), uint256(askPriceX64)
);
```
Add the `SwapMath` import to `OracleValueStopLossExtension.sol`.

## Proof of Concept
1. Deploy a pool with `OracleValueStopLossExtension` in the `afterSwap` slot, `drawdownE6 = 50_000` (5% floor), and a 10% oracle spread (e.g., `bid = 1.0e18`, `ask = 1.1e18` in Q64.64 units).
2. Seed the pool with token0 and token1 so that `metricT1` is just above the 5% floor at the geometric mean price (`sqrt(1.0 √ó 1.1) ‚Čą 1.0488`).
3. Execute a `zeroForOne == false` swap. The extension computes `midPriceX64 = (1.0 + 1.1)/2 = 1.05`, which is ‚Čą 0.12% higher than the geometric mean.
4. The inflated `midPriceX64` raises `metricT1` by ‚Čą 0.06% (for a 50/50 bin), keeping it above the floor even though the actual value at the pool's oracle mid is below the floor.
5. The stop-loss does not revert; the swap completes and LP token1 value leaks past the configured drawdown limit.
6. Reproducible as a Foundry unit test: mock `bidPriceX64`/`askPriceX64`, set watermarks at the geometric-mean-derived metric, call `afterSwap` with `zeroForOne = false`, and assert no revert despite actual value being below the floor.
