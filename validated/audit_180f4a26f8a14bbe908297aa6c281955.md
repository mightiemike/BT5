### Title
`paid_fee_on_l1` Silently Dropped in `ConsensusTransaction → InternalConsensusTransaction` Conversion, Binding Wrong Executable Payload for L1Handler Transactions — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

The `ConsensusTransaction::L1Handler` variant carries only the raw `transaction::L1HandlerTransaction` (no `paid_fee_on_l1`). When a validator node converts a received `ConsensusTransaction::L1Handler` into an `InternalConsensusTransaction::L1Handler` (the executable form), the converter unconditionally substitutes a hardcoded `Fee(1)` for the actual fee paid on L1. This is a structural conversion-boundary defect: a semantically critical field is silently dropped at the public-to-internal boundary and replaced with a wrong constant, binding the wrong executable payload to every L1Handler transaction that traverses the consensus P2P path.

---

### Finding Description

**Type hierarchy at the boundary**

`ConsensusTransaction` (the P2P-facing type) holds:
```rust
pub enum ConsensusTransaction {
    RpcTransaction(RpcTransaction),
    L1Handler(transaction::L1HandlerTransaction),   // ← raw; NO paid_fee_on_l1
}
``` [1](#0-0) 

`InternalConsensusTransaction` (the execution-facing type) holds:
```rust
pub enum InternalConsensusTransaction {
    RpcTransaction(InternalRpcTransaction),
    L1Handler(executable_transaction::L1HandlerTransaction),  // ← has paid_fee_on_l1
}
``` [2](#0-1) 

The executable `L1HandlerTransaction` struct carries `paid_fee_on_l1: Fee` as a first-class field: [3](#0-2) 

**The lossy conversion**

`convert_consensus_tx_to_internal_consensus_tx` dispatches the L1Handler arm to:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [4](#0-3) 

The hardcoded `Fee(1)` replaces whatever the proposer's L1 provider recorded as the actual ETH fee paid on L1.

**Why the field is absent from the wire format**

The protobuf `ConsensusTransaction` message uses `L1HandlerV0`, which carries only `nonce`, `address`, `entry_point_selector`, and `calldata` — no `paid_fee_on_l1`: [5](#0-4) 

The protobuf deserializer confirms the field is absent: [6](#0-5) 

**The inverse direction also drops the field**

`convert_internal_consensus_tx_to_consensus_tx` strips `paid_fee_on_l1` on the way out:
```rust
InternalConsensusTransaction::L1Handler(tx) => {
    Ok(ConsensusTransaction::L1Handler(tx.tx))   // tx.paid_fee_on_l1 silently dropped
}
``` [7](#0-6) 

The round-trip `InternalConsensusTransaction → ConsensusTransaction → InternalConsensusTransaction` is therefore non-identity for L1Handler transactions: the output always has `paid_fee_on_l1 = Fee(1)` regardless of the input value.

**Where the wrong value is consumed**

The blockifier's `execute_raw` for L1Handler transactions checks:
```rust
let paid_fee = self.paid_fee_on_l1;
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(...));
}
``` [8](#0-7) 

A validator that reconstructs the transaction with `Fee(1)` will always pass this check, even if the proposer's copy had `paid_fee_on_l1 = Fee(0)` and therefore rejected the transaction. The proposer and validator bind different executable objects for the same consensus transaction.

**The `CendePreconfirmedTransaction` path also diverges**

The central-blob serialization (`CentralL1HandlerTransaction`) faithfully copies `paid_fee_on_l1` from the internal transaction: [9](#0-8) 

A validator node that produces a central blob (e.g., in a distributed topology) will emit `paid_fee_on_l1 = Fee(1)` instead of the real value, producing a wrong blob that diverges from the proposer's authoritative blob.

---

### Impact Explanation

**Wrong executable payload (High):** Every L1Handler transaction that traverses the consensus P2P path is reconstructed by validators with `paid_fee_on_l1 = Fee(1)` instead of the actual fee. The executable object bound for blockifier execution is structurally different from the one the proposer used.

**Execution divergence under reachable condition:** If the proposer's L1 provider supplies a transaction with `paid_fee_on_l1 = Fee(0)` (e.g., due to an L1 event with zero ETH value, or a future stricter fee gate), the proposer rejects the transaction (fee check fails) while the validator accepts it (`Fee(1) ≠ Fee(0)`). This produces a state divergence between proposer and validator without any privileged action.

**Wrong central-blob field:** Any node that serializes an `InternalConsensusTransaction::L1Handler` into a `CentralL1HandlerTransaction` after receiving it via P2P will emit the wrong `paid_fee_on_l1`, corrupting the authoritative record sent to the central layer.

---

### Likelihood Explanation

The defect is triggered by every L1Handler transaction that passes through the consensus P2P path (i.e., every L1Handler transaction in every block on a multi-node network). The hardcoded substitution is unconditional. The TODO comment confirms the developers are aware the value is wrong and intend to fix it, but no gate prevents the wrong value from being used in production today.

---

### Recommendation

1. Add `paid_fee_on_l1` to the `ConsensusTransaction::L1Handler` variant (or introduce a dedicated `ConsensusL1HandlerTransaction` wrapper that carries it).
2. Add a `paid_fee_on_l1` field to the protobuf `L1HandlerV0` / `ConsensusTransaction` message so the field survives serialization across the P2P boundary.
3. Remove the hardcoded `Fee(1)` in `convert_consensus_l1_handler_to_internal_l1_handler` and use the value carried in the deserialized message.
4. Add a round-trip test asserting that `paid_fee_on_l1` is preserved across `InternalConsensusTransaction → ConsensusTransaction → InternalConsensusTransaction`.

---

### Proof of Concept

```
1. Proposer's L1 provider delivers L1HandlerTransaction { ..., paid_fee_on_l1: Fee(500_000) }.
2. Proposer executes → paid_fee_on_l1 = Fee(500_000) ≠ Fee(0) → accepted.
3. Proposer serializes to protobuf ConsensusTransaction::L1Handler (L1HandlerV0).
   → paid_fee_on_l1 is NOT encoded (field absent from proto schema).
4. Validator deserializes → ConsensusTransaction::L1Handler(raw_tx).
5. Validator calls convert_consensus_tx_to_internal_consensus_tx:
   → convert_consensus_l1_handler_to_internal_l1_handler(raw_tx)
   → L1HandlerTransaction::create(raw_tx, &chain_id, Fee(1))   ← WRONG
6. Validator executes → paid_fee_on_l1 = Fee(1) ≠ Fee(0) → accepted.
   Execution result matches (current check is only ≠ 0), but the executable
   object is structurally wrong.

Divergence scenario:
1. Proposer's L1 provider delivers L1HandlerTransaction { ..., paid_fee_on_l1: Fee(0) }.
2. Proposer executes → paid_fee_on_l1 = Fee(0) → REJECTED (fee check fails).
   Proposer marks tx as rejected; does NOT include in block.
   [If proposer were to include it anyway, validator would accept with Fee(1)
    while proposer's execution produced a rejection — state divergence.]
```

    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::L1HandlerV0) -> Result<Self, Self::Error> {
        let version = L1HandlerTransaction::VERSION;

        let nonce = Nonce(value.nonce.ok_or(missing("L1HandlerV0::nonce"))?.try_into()?);

        let contract_address = value.address.ok_or(missing("L1HandlerV0::address"))?.try_into()?;

        let entry_point_selector_felt = Felt::try_from(
            value.entry_point_selector.ok_or(missing("L1HandlerV0::entry_point_selector"))?,
        )?;
        let entry_point_selector = EntryPointSelector(entry_point_selector_felt);

        let calldata =
            value.calldata.into_iter().map(Felt::try_from).collect::<Result<Vec<_>, _>>()?;

        let calldata = Calldata(calldata.into());

        Ok(Self { version, nonce, contract_address, entry_point_selector, calldata })
    }
