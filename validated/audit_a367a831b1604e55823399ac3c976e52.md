### Title
Deposit Allowlist Bypass via `owner` Parameter Substitution — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead validates the `owner` argument (the position beneficiary). Any unprivileged address can bypass the allowlist by passing an allowlisted address as `owner`.

---

### Finding Description

The call chain is:

1. Attacker (`0xDEAD`) calls `pool.addLiquidity(owner=0xBEEF, ...)`.
2. The pool calls `_beforeAddLiquidity(msg.sender=0xDEAD, owner=0xBEEF, ...)`. [1](#0-0) 

3. `ExtensionCalling._beforeAddLiquidity` encodes both `sender` and `owner` and forwards them to the extension. [2](#0-1) 

4. Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender = 0xDEAD`) is **unnamed and ignored**. The guard checks `allowedDepositor[msg.sender][owner]`, i.e., whether the *position owner* (`0xBEEF`) is allowlisted — not the actual caller. [3](#0-2) 

Because `0xBEEF` is allowlisted, the check passes and `0xDEAD`'s `addLiquidity` call succeeds. LP shares are minted to `0xBEEF`'s position, funded by tokens pulled from `0xDEAD` via the liquidity callback.

The storage layout confirms the intent: `allowedDepositor[pool][depositor]` is keyed by *depositor*, and `setAllowedToDeposit` is described as gating the depositor address — not the position owner. [4](#0-3) 

---

### Impact Explanation

The `DepositAllowlistExtension` is the sole mechanism for restricting who may add liquidity to a pool (e.g., KYC-gated, institutional-only, or compliance-restricted pools). The bypass is total: **every non-allowlisted address can deposit** by naming any allowlisted address as `owner`. The pool admin's access control intent is completely defeated. This constitutes broken core pool functionality under the contest's allowed impact gate.

---

### Likelihood Explanation

The entrypoint is the public `addLiquidity` function with no preconditions beyond token approval. The bypass requires only knowing one allowlisted address (observable on-chain via `AllowedToDepositSet` events or `allowedDepositor` reads). Exploitation is trivial and requires no privileged access.

---

### Recommendation

Name and validate `sender` instead of `owner` in `beforeAddLiquidity`:

```solidity
// Before (buggy):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}

// After (fixed):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```
