### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas` on Zero Values, Producing a Cross-Boundary Transaction Hash Mismatch — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` converts `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` to `ValidResourceBounds::L1Gas(l1_gas)` based on value equality rather than field presence. Because `get_tip_resource_bounds_hash` produces structurally different hash inputs for `AllResources` (three elements: L1\_GAS, L2\_GAS, L1\_DATA\_GAS) versus `L1Gas` (two elements: L1\_GAS, L2\_GAS), a transaction submitted via the RPC gateway with `AllResourceBounds` where `l2_gas = 0` and `l1_data_gas = 0` receives hash **H1** at the gateway but hash **H2 ≠ H1** after a protobuf round-trip. This is the sequencer-native analog of the ERC777 double-credit bug: two code paths (gateway admission vs. P2P deserialization) bind the same transaction to two different canonical identities.

---

### Finding Description

**Root cause — protobuf deserialization (`crates/apollo_protobuf/src/converters/transaction.rs`):**

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The check is purely value-based. Even when all three fields are explicitly present in the wire message with zero values, the deserializer silently selects the `L1Gas` variant.

**Hash function sensitivity (`crates/starknet_api/src/transaction_hash.rs`):**

```rust
let mut resource_felts = vec![
    get_concat_resource(&l1_resource_bounds, L1_GAS)?,
    get_concat_resource(&l2_resource_bounds, L2_GAS)?,
];
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements
    }
});
``` [2](#0-1) 

`AllResources` feeds three felts into the Poseidon hash; `L1Gas` feeds only two. The outputs are distinct even when the third element is the zero-packed `L1_DATA_GAS` felt.

**Gateway path always uses `AllResources`:**

`RpcInvokeTransactionV3` and `InternalRpcInvokeTransactionV3` both carry `AllResourceBounds` (never `L1Gas`). Their `InvokeTransactionV3Trait::resource_bounds()` implementations always return `ValidResourceBounds::AllResources(self.resource_bounds)`. [3](#0-2) 

The hash is computed once at conversion time:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
``` [4](#0-3) 

**P2P serialization preserves all three fields:**

```rust
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),
    },
``` [5](#0-4) 

All three fields are present on the wire. The downgrade to `L1Gas` happens purely on the receiving side, based on the zero-value check.

**Downstream conversion failure:**

`TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` rejects any non-`AllResources` variant:

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => {
        return Err(StarknetApiError::OutOfRange { string: "resource_bounds".to_string() });
    }
},
``` [6](#0-5) 

A receiving node that deserializes the protobuf and attempts to reconstruct the `RpcInvokeTransactionV3` will fail with an error, or — if the hash is recomputed from the downgraded `L1Gas` representation — will produce hash H2 ≠ H1.

**Cairo OS hash function mirrors the same conditional:**

```cairo
// For backward compatibility, we don't hash proof facts if they are empty.
if (proof_facts_size != 0) { ... }
```

The OS `hash_fee_fields` function also branches on the number of resource bounds elements, confirming that the two-element vs. three-element distinction is load-bearing in the canonical hash. [7](#0-6) 

---

### Impact Explanation

A user submits a valid `InvokeV3` transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The gateway accepts it and computes hash **H1** (three-element Poseidon input). When the transaction is propagated via P2P protobuf, the receiving node's deserializer silently converts the resource bounds to `L1Gas`, producing hash **H2** (two-element Poseidon input). H1 ≠ H2. The receiving node either:

1. **Rejects the transaction** because the downstream `TryFrom` conversion to `RpcInvokeTransactionV3` fails (valid transaction rejected before sequencing), or
2. **Binds the transaction to the wrong hash** H2, causing a hash mismatch with the proposer's block commitment.

Both outcomes match the allowed impact scope: *"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"* and *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

Any transaction that uses only L1 gas (setting `l2_gas = 0` and `l1_data_gas = 0`) while using the new `AllResourceBounds` wire format triggers this path. This is a common configuration for users migrating from pre-0.13.3 semantics or explicitly opting out of L2/data-gas usage. No special privileges are required; any unprivileged user can submit such a transaction via the public RPC endpoint.

---

### Recommendation

Change the protobuf deserialization to use **field presence** rather than **value equality** to select the `ValidResourceBounds` variant. Specifically:

- If `l1_data_gas` is **absent** from the protobuf message (i.e., `value.l1_data_gas` is `None`), use `ValidResourceBounds::L1Gas` (backward compatibility with 0.13.2 nodes that never sent this field).
- If `l1_data_gas` is **present** (even with a zero value), use `ValidResourceBounds::AllResources`.

```rust
// Proposed fix
Ok(match value.l1_data_gas {
    None => ValidResourceBounds::L1Gas(l1_gas),   // old 0.13.2 wire format
    Some(_) => ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }),
})
```

This preserves backward compatibility while correctly distinguishing "old transaction without data-gas field" from "new transaction with explicitly-zero data-gas field."

### Proof of Concept

1. Submit `RpcInvokeTransactionV3` with `resource_bounds = AllResourceBounds { l1_gas: GasAmount(1000)/GasPrice(1), l2_gas: GasAmount(0)/GasPrice(0), l1_data_gas: GasAmount(0)/GasPrice(0) }` to the gateway.

2. Gateway calls `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources` → Poseidon over `[tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)]` → **H1**.

3. Transaction is serialized to protobuf `InvokeV3` with all three `ResourceBounds` fields present (l1_data_gas = zero-valued `ResourceLimits`).

4. Receiving node deserializes: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas(l1_gas)`.

5. Receiving node calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::L1Gas` → Poseidon over `[tip, L1_GAS_packed, L2_GAS_packed(0)]` (no L1\_DATA\_GAS element) → **H2**.

6. **H1 ≠ H2**. The receiving node either errors on `TryFrom` conversion or stores the transaction under the wrong hash, breaking block-hash consistency between proposer and validator.
