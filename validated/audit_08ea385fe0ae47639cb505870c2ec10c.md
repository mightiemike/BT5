Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the direct caller of `MetricOmmPool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the originating user. A pool admin who allowlists the router so that vetted users can benefit from router features simultaneously opens the pool to every unprivileged user, because the extension evaluates the router's identity rather than the actual trader's identity.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap` (MetricOmmPool.sol L230–240). `ExtensionCalling._beforeSwap` encodes that value verbatim into the call to every configured extension (ExtensionCalling.sol L149–177). `SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()` (SwapAllowlistExtension.sol L37). When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` with `msg.sender = router` (MetricOmmSimpleRouter.sol L72–80). The pool therefore passes `sender = router` to the extension, which evaluates `allowedSwapper[pool][router]` — the actual user's address is never checked. The `extensionData` forwarded by the router is `params.extensionData` supplied by the caller (untrusted user input); the extension does not decode it to recover the originating user. There is no mechanism in the current interface for the extension to distinguish between a legitimate allowlisted user routing through the router and an unauthorized user doing the same.

## Impact Explanation
A curated pool configured with `SwapAllowlistExtension` in `BEFORE_SWAP_ORDER` loses its per-user access control entirely for any caller who routes through `MetricOmmSimpleRouter`. Non-allowlisted users can execute swaps against pool liquidity that the admin intended to reserve for vetted participants, constituting a direct admin-boundary break: an unprivileged path bypasses a pool-admin-configured access control and enables unauthorized swap execution and potential extraction of LP assets at oracle-quoted prices.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical periphery swap path providing slippage protection, multi-hop routing, and deadline enforcement. A pool admin deploying a curated pool will naturally add the router to the allowlist so that vetted users can use these features. Once `allowedSwapper[pool][router] = true` is observable on-chain, any attacker can immediately exploit the bypass by calling `router.exactInputSingle` with the target pool. No special privilege, flash loan, or oracle manipulation is required. The condition is self-reinforcing: the admin cannot simultaneously allow router-mediated swaps for allowlisted users and block non-allowlisted users.

## Recommendation
The extension must check the economically relevant actor — the user who initiated the transaction — not the intermediate contract. The cleanest fix is for the router to encode `msg.sender` into `extensionData` before calling `pool.swap`, and for the extension to decode and check that address. The pool's `onlyPool` guard on the extension ensures only a legitimate pool can invoke `beforeSwap`, so the decoded address cannot be spoofed by an attacker calling the extension directly. Alternatively, the `beforeSwap` interface could be extended with a dedicated `swapInitiator` field so the pool can pass both the direct caller and the original initiator.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is allowed to swap.
3. Pool admin calls `setAllowedToSwap(pool, address(router), true)` — router is added so Alice can use it.
4. Bob (not allowlisted) calls `router.exactInputSingle({pool: pool, recipient: bob, extensionData: "", ...})`.
5. Router calls `pool.swap(bob, zeroForOne, amount, priceLimit, "", "")` with `msg.sender = router`.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router] == true` → passes without checking Bob.
8. Bob's swap executes against the curated pool's liquidity despite never being allowlisted.
Step 3 is the realistic trigger: without it, Alice cannot use the router either, forcing the admin to choose between broken UX for allowlisted users and a fully open pool.
