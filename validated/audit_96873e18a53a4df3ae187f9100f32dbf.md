Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual User, Enabling Full Allowlist Bypass ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps on the `sender` parameter, which is `msg.sender` of the pool's `swap()` call ŌĆö the direct caller (e.g., `MetricOmmSimpleRouter`), not the end user. When a pool admin allowlists the router to permit authorized users to trade through it, every user who routes through that contract bypasses the per-user restriction, because the allowlist check resolves to the single router address rather than the individual user. This collapses a per-user guard into a per-router guard, allowing unauthorized users to execute swaps in pools intended to be private.

## Finding Description
`SwapAllowlistExtension.beforeSwap` checks `sender` against the allowlist:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (correct ŌĆö it identifies which pool's config to read), and `sender` is the first argument forwarded by `_beforeSwap`. In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` of the pool's own `swap()` call:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ŌåÉ this is the router, not the end user
    recipient,
    ...
);
```

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)`, the pool's `msg.sender` is the router. So `sender = router`, and the check becomes `allowedSwapper[pool][router]`. If the pool admin allowlisted the router (a natural action to let authorized users trade through the standard router), every user who calls the router passes the guard ŌĆö including completely unauthorized users.

**Contrast with `DepositAllowlistExtension`**, which correctly checks the actual beneficiary (`owner`, the LP position owner), not the intermediary caller:

```solidity
// DepositAllowlistExtension.sol L38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

The `beforeSwap` interface exposes both `sender` (the direct caller/router) and `recipient` (the swap output beneficiary), but neither is a reliable proxy for the true initiating user when a router is involved. The `DepositAllowlistExtension` pattern works because `addLiquidity` explicitly separates `sender` (payer/operator) from `owner` (position beneficiary), and the extension checks `owner`. The swap interface has no analogous dedicated `swapper` field.

The exploit path is fully confirmed by the production code:
1. `MetricOmmSimpleRouter.exactInputSingle` ŌåÆ `pool.swap(params.recipient, ...)` ŌĆö pool's `msg.sender` is the router.
2. Pool calls `_beforeSwap(msg.sender=router, recipient, ...)`.
3. Extension checks `allowedSwapper[pool][router]` ŌĆö passes if router is allowlisted.
4. Unauthorized user's swap executes.

## Impact Explanation
A pool admin who deploys a restricted pool (e.g., a private institutional pool with tight spreads) and allowlists `MetricOmmSimpleRouter` to allow authorized users to trade through it inadvertently grants swap access to **all users** of the router. Unauthorized users can execute swaps in a pool intended to be private, potentially draining LP value at below-market rates if the pool offers favorable pricing. This is a direct loss of LP principal to unauthorized counterparties ŌĆö a broken core pool access-control invariant with fund-impacting consequences.

## Likelihood Explanation
The trigger requires no privileged access: any user can call `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` targeting the restricted pool. The only precondition is that the pool admin has allowlisted the router (a natural and expected operational action). The divergence between admin intent and actual outcome is silent ŌĆö no on-chain warning is emitted. The path is repeatable by any address.

## Recommendation
Mirror the `DepositAllowlistExtension` pattern: check the actual user, not the intermediary. Options:
1. Add a dedicated `swapper` parameter to the `beforeSwap` interface (separate from `sender` and `recipient`) that the pool populates with the original `msg.sender` of the outermost call ŌĆö analogous to `owner` in the deposit path.
2. Alternatively, document clearly that `SwapAllowlistExtension` is incompatible with router-mediated swaps for per-user access control, and require direct pool calls for allowlisted pools.
3. At minimum, the extension should check `recipient` as a proxy for the beneficiary (though this is imperfect since `recipient` can be set to any address by the router caller).

## Proof of Concept
```
Setup:
  - Pool deployed with SwapAllowlistExtension in BEFORE_SWAP_ORDER.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (intending to allow authorized users to trade via MetricOmmSimpleRouter).
  - Unauthorized user (not on allowlist) calls:
      MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: unauthorizedUser, ...})
        ŌåÆ pool.swap(recipient=unauthorizedUser, ...)   [pool.msg.sender = router]
          ŌåÆ _beforeSwap(sender=router, ...)
            ŌåÆ allowedSwapper[pool][router] == true  Ō£ō (passes)
          ŌåÆ swap executes for unauthorized user

Verification:
  - allowedSwapper[pool][unauthorizedUser] == false (never set)
  - allowedSwapper[pool][router] == true (set by admin)
  - Guard passes because sender == router, which IS allowlisted.
  - Unauthorized user receives swap output; pool LPs bear counterparty risk
    they intended to restrict.

Foundry test sketch:
  1. Deploy pool with SwapAllowlistExtension.
  2. setAllowedToSwap(pool, address(router), true).
  3. vm.prank(unauthorizedUser); router.exactInputSingle({pool: pool, ...}).
  4. Assert swap succeeds (no NotAllowedToSwap revert).
  5. Assert allowedSwapper[pool][unauthorizedUser] == false.
