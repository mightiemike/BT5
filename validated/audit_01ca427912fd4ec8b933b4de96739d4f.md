### Title
Fast Withdrawal Fees Permanently Locked in `WithdrawPool` With No On-Chain Accounting Integration — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal` collects real ERC20 fees into the `WithdrawPool` contract and tracks them in `fees[productId]`, but no `claimFees` function exists to route these fees into the protocol's on-chain subaccount accounting system. Unlike sequencer fees — which are properly credited to `X_ACCOUNT` via `DumpFees` → `claimSequencerFees` — fast withdrawal fees are silently stranded in `WithdrawPool` with no equivalent claim path.

---

### Finding Description

In `submitFastWithdrawal`, after computing the fee, one of two branches executes:

- If `sendTo == msg.sender`: the fee is deducted from `transferAmount` and the tokens remain in the contract.
- If `sendTo != msg.sender`: the fee is pulled from `msg.sender` via `safeTransferFrom` into the contract.

In both cases, `fees[productId] += fee` is updated and the fee tokens accumulate in `WithdrawPool`. [1](#0-0) 

However, `BaseWithdrawPool` and `WithdrawPool` define no `claimFees` function. The only extraction path is `removeLiquidity`, which is `onlyOwner`, sends tokens to an arbitrary `sendTo` address, and does **not** update `fees[productId]` or credit any on-chain subaccount. [2](#0-1) 

Contrast this with sequencer fees, which follow a complete accounting loop: `chargeFee` → `sequencerFee[productId] += fee` → `DumpFees` transaction → `clearinghouse.claimSequencerFees(fees)` → `spotEngine.updateBalance(spotIds[i], X_ACCOUNT, fees[i] + feeBalance.amount)`. [3](#0-2) [4](#0-3) 

Fast withdrawal fees have no equivalent path into `X_ACCOUNT` or `FEES_ACCOUNT`.

---

### Impact Explanation

Fast withdrawal fees collected in real ERC20 tokens are permanently stranded in `WithdrawPool` with no on-chain accounting integration. The `fees[productId]` mapping grows indefinitely but is never reset or credited to any protocol subaccount. The protocol's on-chain balance sheet (`X_ACCOUNT`) never reflects fast withdrawal fee income, creating a permanent accounting desynchronization between actual token balances in `WithdrawPool` and the on-chain subaccount state. Any owner extraction via `removeLiquidity` bypasses the accounting system entirely, leaving `fees[productId]` permanently inflated relative to actual holdings.

---

### Likelihood Explanation

`submitFastWithdrawal` is a public, permissionless function callable by any unprivileged address. Every successful fast withdrawal call triggers the fee accumulation. The condition is reached on every fast withdrawal execution, making this a certainty rather than a probabilistic event. [5](#0-4) 

---

### Recommendation

Add a `claimFees` function (callable by the `Clearinghouse` or via a sequencer-submitted transaction type) that:
1. Reads `fees[productId]` for each product.
2. Calls `spotEngine.updateBalance(productId, X_ACCOUNT, fees[productId])` to credit the protocol treasury.
3. Resets `fees[productId] = 0`.

This mirrors the existing `claimSequencerFees` pattern in `Clearinghouse.sol`. [6](#0-5) 

---

### Proof of Concept

1. Alice calls `submitFastWithdrawal(idx, transaction, signatures)` where `sendTo != msg.sender`.
2. `safeTransferFrom(token, msg.sender, uint128(fee))` pulls `fee` tokens from Alice into `WithdrawPool`.
3. `fees[productId] += fee` is updated.
4. `handleWithdrawTransfer(token, sendTo, transferAmount)` sends the full amount to the user.
5. The `fee` tokens now sit in `WithdrawPool`. No transaction type exists to call `claimSequencerFees`-equivalent logic for these tokens.
6. The sequencer submits a `DumpFees` transaction: `claimSequencerFees` credits `X_ACCOUNT` with sequencer fees only — fast withdrawal fees in `WithdrawPool` are never included.
7. `fees[productId]` remains non-zero indefinitely; `X_ACCOUNT` never reflects the fast withdrawal fee income. [7](#0-6) [8](#0-7)
