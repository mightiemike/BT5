### Title
`AccountContract` Borsh Enum Lacks Explicit Discriminant Pinning — Trie-Stored Account State Silently Corrupted on Protocol Upgrade - (File: `core/primitives-core/src/account.rs`)

### Summary

`AccountContract`, the enum that records which contract (none, local, global-by-hash, or global-by-account-id) is attached to an on-chain account, derives `BorshSerialize`/`BorshDeserialize` without `#[borsh(use_discriminant = true)]` and without `#[repr(u8)]` explicit discriminants. Borsh therefore assigns discriminant tags by source-code position (0, 1, 2, 3). Because `AccountContract` is persisted inside `AccountV2` in the Merkle-Patricia Trie, any future protocol upgrade that inserts a new variant before `Global` or `GlobalByAccount`, or removes/reorders an existing variant, silently remaps every existing account's contract-type tag to a different meaning — exactly the storage-layout corruption class described in the seed report.

### Finding Description

`AccountContract` is defined as:

```rust
pub enum AccountContract {
    None,               // Borsh tag 0
    Local(CryptoHash),  // Borsh tag 1
    Global(CryptoHash), // Borsh tag 2
    GlobalByAccount(AccountId), // Borsh tag 3
}
``` [1](#0-0) 

It derives `BorshSerialize` and `BorshDeserialize` with no `#[borsh(use_discriminant = true)]` attribute and no `= N` explicit discriminant syntax. Borsh encodes the variant as its zero-based position index.

`AccountContract` is a field of `AccountV2`:

```rust
pub struct AccountV2 {
    amount: Balance,
    locked: Balance,
    storage_usage: StorageUsage,
    contract: AccountContract,   // ← stored inline, positional tag
}
``` [2](#0-1) 

`AccountV2` is persisted to the trie via `BorshVersionedAccount::V2(AccountV2)`:

```rust
enum BorshVersionedAccount {
    V2(AccountV2) = 0,   // explicit outer tag, but inner AccountContract is positional
}
``` [3](#0-2) 

The serialization path writes the sentinel `u128::MAX`, then the `BorshVersionedAccount` discriminant, then the raw `AccountV2` bytes — which include the positional `AccountContract` tag. [4](#0-3) 

Contrast this with every other protocol-critical enum in the codebase, which all use `#[borsh(use_discriminant = true)]` and `= N` pinned values:

```rust
// ReceiptEnum — correct pattern
#[borsh(use_discriminant = true)]
#[repr(u8)]
pub enum ReceiptEnum {
    Action(ActionReceipt) = 0,
    ...
    ActionV2(ActionReceiptV2) = 5,
    PromiseYieldV2(ActionReceiptV2) = 6,
}
``` [5](#0-4) 

The `protocol-schema-check` README explicitly documents the failure mode: "The real failure case was when we accidentally swapped the enum variants. If both are units, serialization don't break, but the impacted logic will be broken." [6](#0-5) 

This exact failure already occurred in production: PR #13155 unintentionally removed a `StateChangeCause` variant, shifting `BandwidthSchedulerStateUpdate` from Borsh tag 11 to tag 10, colliding with the removed slot. The workaround — keeping `_UnusedReshardingV2 = 10` as a tombstone — is still present:

```rust
/// Deprecated in #13155, we need to keep it to preserve enum variant tags for borsh serialization.
_UnusedReshardingV2 = 10,
/// Update persistent state kept by Bandwidth Scheduler after running the scheduling algorithm.
BandwidthSchedulerStateUpdate = 11,
``` [7](#0-6) 

And the view-layer workaround:

```rust
// Some nodes on testnet were upgraded to #13155 that included
// unintended removal of enum variants which changes borsh tag for
// BandwidthSchedulerStateUpdate from 11 to 10.
StateChangeCause::_UnusedReshardingV2 => Self::BandwidthSchedulerStateUpdate,
``` [8](#0-7) 

`StateChangeCause` was retrofitted with `#[borsh(use_discriminant = true)]` after the incident. `AccountContract` has not been.

### Impact Explanation

`AccountContract` is written into the trie for every account that uses a global contract (`AccountContract::Global` or `AccountContract::GlobalByAccount`). A protocol upgrade that inserts any new variant before position 2 would cause:

- Every existing `Global(hash)` account to deserialize as the new variant (wrong type, wrong payload interpretation).
- Every existing `GlobalByAccount(id)` account to deserialize as `Global` (a 32-byte hash read from what is actually a variable-length account-id string — producing a garbage hash or a deserialization panic).
- The `current_contract_code` host function (which maps `None→0`, `Local→1`, `Global→2`, `GlobalByAccount→3`) to return incorrect values to running contracts.

Accounts with global contracts would silently execute the wrong contract code or fail to execute any contract, with no on-chain error. The state root would still be accepted by validators running the new binary because the trie hash is computed over the corrupted bytes.

**Severity: High.** Corrupted account-contract bindings affect all accounts that opted into global contracts. The corruption is silent (no deserialization error for plausible byte sequences) and permanent (trie nodes are content-addressed; old nodes remain readable with the wrong interpretation).

### Likelihood Explanation

The trigger is a protocol upgrade that adds a variant to `AccountContract` at any position other than the end. The `protocol-schema-check` CI tool detects schema changes but (a) does not enforce backward-compatible ordering, (b) can be bypassed by updating the golden file without understanding the implication, and (c) was bypassed in the `StateChangeCause` incident. The global-contracts feature is active in stable protocol version 86, so affected accounts exist on mainnet today.

### Recommendation

Add `#[borsh(use_discriminant = true)]` and `#[repr(u8)]` with explicit `= N` discriminants to `AccountContract`:

```rust
#[borsh(use_discriminant = true)]
#[repr(u8)]
pub enum AccountContract {
    None               = 0,
    Local(CryptoHash)  = 1,
    Global(CryptoHash) = 2,
    GlobalByAccount(AccountId) = 3,
}
```

This pins each variant's Borsh tag independently of source-code position, matching the pattern already used by `ReceiptEnum`, `StateChangeCause`, `Block`, `ChallengeBody`, `FlatStorageStatus`, and every other protocol-critical enum in the codebase.

### Proof of Concept

**Current serialization (correct):**
```
AccountV2 { contract: AccountContract::Global(H) }
→ [sentinel u128] [BorshVersionedAccount::V2 tag=0] [amount u128] [locked u128] [storage_usage u64] [0x02] [H: 32 bytes]
                                                                                                       ^^^^
                                                                                              AccountContract::Global = position 2
```

**After a protocol upgrade inserts `NewVariant` at position 2 (before `Global`):**
```
AccountContract { None=0, Local=1, NewVariant=2, Global=3, GlobalByAccount=4 }
```

The existing byte `0x02` is now read as `NewVariant`, not `Global`. The 32 bytes of `H` are consumed as `NewVariant`'s payload. The account's contract binding is silently wrong. No error is returned; the trie root changes, but validators running the new binary all compute the same wrong root and reach consensus on corrupted state. [1](#0-0) [9](#0-8)

```rust
#[derive(BorshSerialize, BorshDeserialize)]
#[borsh(use_discriminant = true)]
#[repr(u8)]
enum BorshVersionedAccount {
    // V1 is not included since it is serialized directly without being wrapped in enum
    V2(AccountV2) = 0,
}

impl BorshDeserialize for Account {
    fn deserialize_reader<R: io::Read>(rd: &mut R) -> io::Result<Self> {
        // The first value of all Account serialization formats is a u128,
        // either a sentinel or a balance.
        let sentinel_or_amount = Balance::deserialize_reader(rd)?;
        if sentinel_or_amount == Account::SERIALIZATION_SENTINEL {
            let versioned_account = BorshVersionedAccount::deserialize_reader(rd)?;
            let account = match versioned_account {
                BorshVersionedAccount::V2(account_v2) => Account::V2(account_v2),
            };
            Ok(account)
        } else {
            // Legacy unversioned representation of Account
            let locked = Balance::deserialize_reader(rd)?;
            let code_hash = CryptoHash::deserialize_reader(rd)?;
            let storage_usage = StorageUsage::deserialize_reader(rd)?;

            Ok(Account::V1(AccountV1 {
                amount: sentinel_or_amount,
                locked,
                code_hash,
                storage_usage,
            }))
        }
    }
}

impl BorshSerialize for Account {
    fn serialize<W: io::Write>(&self, writer: &mut W) -> io::Result<()> {
        let versioned_account = match self {
            Account::V1(account_v1) => return account_v1.serialize(writer),
            Account::V2(account_v2) => BorshVersionedAccount::V2(account_v2.clone()),
        };
        let sentinel = Account::SERIALIZATION_SENTINEL;
        BorshSerialize::serialize(&sentinel, writer)?;
        BorshSerialize::serialize(&versioned_account, writer)
    }
}
