### Title
`SpotBalance` Event Emits Stale Pre-Accrual Balance for `FEES_ACCOUNT` During Interest Updates — (`File: core/contracts/SpotEngineState.sol`)

---

### Summary

In `SpotEngineState._updateState`, the cumulative interest multipliers are updated in memory before `_setBalanceAndUpdateBitmap` is called for `FEES_ACCOUNT`. The `_balanceUpdate` hook triggered inside `_setBalanceAndUpdateBitmap` reads `states[productId]` from **storage**, which still holds the old (pre-accrual) multipliers. As a result, the `SpotBalance` event emits an incorrect, understated balance for `FEES_ACCOUNT` on every interest accrual cycle.

---

### Finding Description

`SpotEngineState._updateState` is called from `updateStates` to accrue interest for each spot product. The execution order is:

1. **Lines 129–137** — `state.cumulativeBorrowsMultiplierX18` and `state.cumulativeDepositsMultiplierX18` are updated **in the in-memory `state` variable**.
2. **Line 144** — `_setBalanceAndUpdateBitmap(productId, FEES_ACCOUNT, feesAccBalance)` is called.
3. Inside `_setBalanceAndUpdateBitmap` (line 204), the new `feesAccBalance` is written to storage. Then `_balanceUpdate(productId, FEES_ACCOUNT)` is called (line 207).
4. `_balanceUpdate` (lines 210–231) calls `getStateAndBalance(productId, FEES_ACCOUNT)` (line 215), which reads `states[productId]` **from storage** (line 241).
5. At this point, `states[productId]` in storage still holds the **old multipliers** — `_setState(productId, state)` is only called after `_updateState` returns, back in `updateStates` at line 281.
6. `balanceNormalizedToBalance` (line 243) therefore computes `balance.amount = feesAccBalance.amountNormalized × OLD_cumulativeMultiplierX18`.
7. The `SpotBalance` event (lines 225–230) emits this stale, pre-accrual balance and the old `cumulativeMultiplierX18`.

The correct emission should use the new multipliers that were computed in lines 129–137 but not yet persisted.

---

### Impact Explanation

The `SpotBalance` event for `FEES_ACCOUNT` is emitted with a balance that is systematically lower than the true post-accrual balance by the amount of interest earned in that period. Off-chain systems — including the sequencer's accounting layer and any indexer reconstructing subaccount balances from events — will hold an incorrect view of accumulated protocol fees. This creates a persistent desynchronization between on-chain state and off-chain event-derived state for `FEES_ACCOUNT`, which can corrupt fee-accounting logic and any downstream decision that depends on the emitted balance (e.g., `claimSequencerFees` scheduling, fee-tier calculations, or monitoring).

---

### Likelihood Explanation

`updateStates` is called by the sequencer as a routine, high-frequency operation. Whenever there is non-zero borrowing utilization for any spot product, `feesAmt` will be non-zero and the bug triggers unconditionally. This is the normal operating condition of the protocol, making the likelihood **high**.

---

### Recommendation

Cache the updated multipliers before calling `_setBalanceAndUpdateBitmap`, or — preferably — call `_setState(productId, state)` **before** calling `_setBalanceAndUpdateBitmap` inside `_updateState`. This ensures that when `_balanceUpdate` reads `states[productId]` from storage, it retrieves the post-accrual multipliers and emits the correct balance.

Alternatively, pass the in-memory `state` directly into `_balanceUpdate` so it does not need to re-read from storage.

---
