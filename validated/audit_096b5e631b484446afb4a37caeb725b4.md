### Title
`op_modpow` (opcode 60) Permanently Locked in Mempool via Undocumented `DISABLE_OP` in `MEMPOOL_MODE` ‚Äî (`File: src/chia_dialect.rs`)

---

### Summary

`MEMPOOL_MODE` unconditionally includes `ClvmFlags::DISABLE_OP`, which causes opcode 60 (`op_modpow`) to return `EvalErr::Unimplemented` in every mempool evaluation. The same program is accepted and executed correctly in consensus mode (where `DISABLE_OP` is absent). Any coin puzzle that uses `op_modpow` is therefore permanently unspendable through the mempool ‚Äî the only path to spend it is for a miner/farmer to bypass the mempool and include the transaction directly in a block, a privileged action unavailable to ordinary users.

---

### Finding Description

**Root cause ‚Äî `src/chia_dialect.rs`, lines 56 and 72‚Äì76:**

```rust
const DISABLE_OP = 0x200;   // no documentation comment

pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)      // ‚Üê always present in mempool
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

**Operator dispatch ‚Äî `src/chia_dialect.rs`, lines 239‚Äì244:**

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;   // hard rejection
    }
    op_modpow
}
```

Opcode 60 (`op_modpow`) is a first-class, non-softforked operator present in the main dispatch table. It is not gated behind any softfork extension. In consensus mode (flags without `DISABLE_OP`) it executes normally. In mempool mode (`MEMPOOL_MODE`, which always carries `DISABLE_OP`) it is unconditionally rejected as `Unimplemented`.

The `DISABLE_OP` flag carries **no documentation comment** ‚Äî unlike every other flag in the same `bitflags!` block ‚Äî making it impossible to determine whether this asymmetry is intentional or an oversight.

**Attacker-controlled entry path:**

1. A user (or attacker demonstrating the issue) crafts a CLVM puzzle that calls opcode 60 (`op_modpow`).
2. The coin is broadcast to a full node.
3. The full node evaluates the spend bundle through `run_serialized_chia_program` with `MEMPOOL_MODE` flags.
4. `op_modpow` hits the `DISABLE_OP` guard and returns `Unimplemented`.
5. The mempool rejects the spend bundle.
6. The coin cannot be spent by any ordinary user; only a miner/farmer who constructs a block directly (bypassing the mempool) can include it.

---

### Impact Explanation

Any coin whose puzzle uses `op_modpow` (opcode 60) is **permanently locked** from the perspective of the mempool. The user has no mechanism to spend it through the standard transaction submission path. This is a direct analog to the external report's finding: a user-deposited resource that can only be released by a privileged actor (guardian/miner), not by the user themselves. The corrupted result is a concrete `EvalErr::Unimplemented` returned for a program that consensus would evaluate to a valid `Reduction`.

---

### Likelihood Explanation

`op_modpow` is a documented, non-softforked operator. Any Chialisp programmer who writes a puzzle using `(modpow ...)` will produce a coin that is unspendable through the mempool. The trigger requires only attacker-controlled (or user-authored) CLVM bytes containing opcode 60 ‚Äî no privileged keys, no social engineering, no dependency on external state.

---

### Recommendation

1. **Document the intent of `DISABLE_OP`** with a comment matching the style of all other flags in the `bitflags!` block. If the flag is a temporary transition measure (e.g., while `op_modpow` cost calibration is finalized), state that explicitly.
2. **If `op_modpow` is a fully activated operator**, remove `DISABLE_OP` from `MEMPOOL_MODE`. The mempool should reject programs that consensus rejects, not the reverse.
3. **If `op_modpow` is intentionally disabled in the mempool** (e.g., due to resource concerns), gate it behind a softfork extension so that coins cannot be created with it until it is fully activated in both paths.
