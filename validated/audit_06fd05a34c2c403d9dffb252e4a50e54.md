Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Immediate Caller (Router) Instead of Ultimate Swapper, Enabling Allowlist Bypass When Router Is Allowlisted — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which the pool sets to `msg.sender` of the `pool.swap()` call. When `MetricOmmSimpleRouter` is used, `sender` is the router's address, not the originating user. If a pool admin allowlists the router address (a natural configuration to support router-mediated swaps for permitted users), every unprivileged user can bypass the allowlist by routing through `MetricOmmSimpleRouter`, because the extension only verifies that the router is permitted, not the actual trader.

## Finding Description

**Root cause — `sender` binding in the pool:**

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol line 160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

**The extension checks this `sender`:**

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

**Router path — `sender` becomes the router:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
    );
```

The router is `msg.sender` to the pool, so `sender` delivered to `beforeSwap` is the **router address**, not the originating EOA.

**Exploit flow:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — a natural step to permit router-mediated swaps for allowlisted users.
3. Any unprivileged user (not in the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. The router calls `pool.swap(...)` with `msg.sender = router`.
5. `beforeSwap` receives `sender = router`, checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. The non-allowlisted user successfully swaps, bypassing the intended access control.

**Existing guards are insufficient:** The extension has no mechanism to recover the originating user from the router call. The `recipient` field is not the payer/swapper and is not checked. There is no forwarded identity in `extensionData` by default.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control primitive for pools that restrict swap participation. When the router is allowlisted (a reasonable operational choice), the allowlist is entirely ineffective: any address can execute swaps against a pool that was designed to be permissioned. This breaks the core pool functionality of restricting swap access and constitutes a broken allowlist invariant with direct fund-flow impact (unauthorized parties can trade against the pool's liquidity).

## Likelihood Explanation
The condition requires the pool admin to have allowlisted the router address. This is a natural and expected configuration for any pool that wants to support router-mediated swaps for its permitted users — the admin has no other way to enable router access. Once the router is allowlisted, the bypass is trivially reachable by any unprivileged caller with zero additional preconditions, and is repeatable on every swap.

## Recommendation
The extension should not rely on the `sender` argument alone when the caller may be a router. Two viable fixes:

1. **Pass the originating user via `extensionData`:** The router encodes `msg.sender` (the originating EOA) into `extensionData`; the extension decodes and checks it. This requires a convention between router and extension.
2. **Check `recipient` as the gated identity:** If the pool's design intent is to gate who *receives* output, check `recipient` instead of `sender`. This is only correct if recipient equals the economic beneficiary.
3. **Disallow router allowlisting / document the limitation:** Document that allowlisting a public router grants access to all users, and provide a router variant that forwards the originating caller identity.

## Proof of Concept

```solidity
// Foundry test sketch
function test_allowlistBypassViaRouter() public {
    // Setup: pool with SwapAllowlistExtension, router allowlisted
    extension.setAllowedToSwap(address(pool), address(router), true);
    // attacker is NOT in the allowlist
    assertFalse(extension.isAllowedToSwap(address(pool), attacker));

    // Attacker routes through MetricOmmSimpleRouter
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        recipient: attacker,
        tokenIn: token0,
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));
    // Swap succeeds — allowlist bypassed
}
```
