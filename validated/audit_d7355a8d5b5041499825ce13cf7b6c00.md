Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual pool-call initiator, i.e., `msg.sender` of `addLiquidity`) and validates only the caller-supplied `owner` (the LP position recipient). Because `owner` is a free parameter with no binding to the actual payer, any address can bypass the allowlist by nominating an already-allowlisted address as `owner` while routing through `MetricOmmPoolLiquidityAdder`. This breaks the pool admin's access-control invariant and allows unauthorized liquidity to enter the pool.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values verbatim to the extension: [2](#0-1) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and discarded; only `owner` is checked against `allowedDepositor`: [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (the two-argument overload) accepts any non-zero `owner` from the caller and stores `msg.sender` as the payer in transient storage, forwarding the caller-supplied `owner` directly to the pool: [4](#0-3) 

The only validation on `owner` is a zero-address check: [5](#0-4) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swap initiator), not the recipient: [6](#0-5) 

**Attack path:**
1. Pool is configured with `DepositAllowlistExtension`; Alice (`0xAlice`) is allowlisted, Bob (`0xBob`) is not.
2. Bob calls `LiquidityAdder.addLiquidityExactShares(pool, 0xAlice, salt, deltas, max0, max1, extData)`.
3. LiquidityAdder calls `pool.addLiquidity(0xAlice, salt, deltas, abi.encode(KIND_PAY), extData)`.
4. Pool calls `_beforeAddLiquidity(LiquidityAdder, 0xAlice, ...)`.
5. Extension evaluates `allowedDepositor[pool][0xAlice]` → `true` → hook passes.
6. Bob's tokens are pulled from Bob's wallet; the LP position is minted to Alice.

The `allowedDepositor` mapping is `public`, so any attacker can trivially read which addresses are allowlisted.

## Impact Explanation
The pool admin deploys `DepositAllowlistExtension` to enforce a closed LP set — for regulatory compliance, wash-trading prevention, or pool composition control. Any unprivileged address can circumvent this control by nominating any allowlisted address as `owner`. Unauthorized liquidity enters the pool, diluting existing LP positions and invalidating the admin's access-control invariant. This is a direct admin-boundary break: a pool-admin-configured guard is bypassed by an unprivileged path with no special permissions required. The `NotAllowedToDeposit` error documented at the interface level is never triggered for the actual depositor/payer. [7](#0-6) 

## Likelihood Explanation
`MetricOmmPoolLiquidityAdder` is the standard periphery entry point for EOA deposits and is publicly deployed. The bypass requires only: (1) knowing one allowlisted address (readable from the `public` `allowedDepositor` mapping), and (2) calling the standard `addLiquidityExactShares` overload with that address as `owner`. No privileged access, flash loans, or oracle manipulation is needed. The attack is repeatable at will by any address holding tokens and approval to the adder.

## Recommendation
Check `sender` (the actual pool-call initiator) in addition to — or instead of — `owner`. The minimal fix in `DepositAllowlistExtension.beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (
        !allowAllDepositors[msg.sender] &&
        !allowedDepositor[msg.sender][sender] &&
        !allowedDepositor[msg.sender][owner]
    ) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Alternatively, check only `sender` (mirroring `SwapAllowlistExtension`) and require integrators such as `MetricOmmPoolLiquidityAdder` to be individually allowlisted, giving the admin coarse-grained control over entry points.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Preconditions:
//   pool     — MetricOmmPool with DepositAllowlistExtension configured
//   adder    — MetricOmmPoolLiquidityAdder (public periphery)
//   alice    — allowedDepositor[pool][alice] == true
//   bob      — NOT allowlisted; holds token0/token1 and has approved `adder`

vm.startPrank(bob);
token0.approve(address(adder), type(uint256).max);
token1.approve(address(adder), type(uint256).max);

// bob is not allowlisted, but alice is — use alice as owner
adder.addLiquidityExactShares(
    pool,
    alice,   // owner: allowlisted → hook passes; bob (sender) is never checked
    0,
    deltas,
    max0,
    max1,
    ""
);
vm.stopPrank();

// Result: bob's tokens are in the pool; alice holds the LP position.
// DepositAllowlistExtension never checked bob.
assertEq(pool.positionShares(alice, 0, binIdx), expectedShares);
```
