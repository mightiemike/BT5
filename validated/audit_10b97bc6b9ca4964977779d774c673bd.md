### Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension` is documented as gating `addLiquidity` **by depositor address**. Its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual on-chain caller who provides tokens) and instead validates the `owner` argument (the LP-position recipient). Because `owner` is a free caller-supplied parameter in `MetricOmmPool.addLiquidity`, any unprivileged address can bypass the allowlist entirely by naming any allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes the real depositor as `sender` and the position recipient as `owner` to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values faithfully: [2](#0-1) 

Inside the extension, the first parameter (`sender`) is unnamed and discarded; only `owner` is checked: [3](#0-2) 

Because `owner` is a free argument supplied by the caller in `addLiquidity(address owner, ...)`, any address can pass `owner = <allowlisted_address>` and the guard approves the call. The actual depositor (`msg.sender` of the pool call, i.e., `sender`) is never checked.

The contract's own NatSpec confirms the intended invariant that is broken: [4](#0-3) 

---

### Impact Explanation

The deposit allowlist is rendered completely ineffective. Any address — regardless of allowlist status — can inject liquidity into a pool that is supposed to be restricted (KYC-gated, institutional-only, whitelist-only). The unauthorized depositor supplies the tokens; the named allowlisted `owner` receives the LP position. Consequences include:

- **Allowlist invariant broken**: the core guard the pool admin configured provides zero protection.
- **Unauthorized pool composition change**: an attacker can concentrate liquidity in specific bins to skew the marginal price curve, harming existing LPs on subsequent swaps.
- **Unsolicited LP exposure for the named owner**: tokens are locked in a position under Alice's address without Alice's consent; if the pool is adversarial or the oracle is later manipulated, Alice bears the loss.

This fits the allowed-impact gate: *"Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path"* and *"Broken core pool functionality."*

---

### Likelihood Explanation

Trivially exploitable. All allowlisted addresses are visible on-chain via `allowedDepositor` or `AllowedToDepositSet` events. No special privileges, flash loans, or multi-step setup are required — a single `addLiquidity` call suffices.

---

### Recommendation

Validate `sender` (the actual depositor) instead of `owner`:

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address /*owner*/, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; `allowAllDepositors[pool] = false`.
2. Pool admin calls `setAllowedToDeposit(pool, Alice, true)` — only Alice is allowlisted.
3. Bob (not allowlisted) calls:
   ```solidity
   pool.addLiquidity(
       owner = Alice,   // ← free parameter; Bob names Alice
       salt  = 0,
       deltas = <any valid delta>,
       callbackData = ...,
       extensionData = ""
   );
   ```
4. Pool calls `_beforeAddLiquidity(sender=Bob, owner=Alice, ...)`.
5. Extension evaluates `allowedDepositor[pool][Alice]` → `true` → **no revert**.
6. `LiquidityLib.addLiquidity` runs; Bob's tokens are pulled via the swap callback; Alice's position is credited.
7. Bob has added liquidity to a restricted pool without being on the allowlist. The allowlist guard is fully bypassed. [3](#0-2) [5](#0-4)
