### Title
SwapAllowlistExtension Checks Router Address as Swapper Identity, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract, not the original user. Any pool admin who adds the router to the allowlist (the only way to let legitimate users use the router) simultaneously opens the allowlist to every user on-chain.

---

### Finding Description

**Pool → Extension argument binding**

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

**Extension identity check**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool passed — the router address when the router is the direct caller: [3](#0-2) 

**Router call site**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly; the pool's `msg.sender` is the router, not the original EOA: [4](#0-3) 

The same pattern holds for `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**The dilemma**

A pool admin who deploys a curated pool with `SwapAllowlistExtension` faces an inescapable choice:

| Admin action | Effect |
|---|---|
| Do **not** add the router to the allowlist | Legitimate allowlisted users **cannot** use the router; they must call `pool.swap` directly |
| **Add the router** to the allowlist | Every on-chain address can bypass the allowlist by routing through the router |

There is no configuration that simultaneously allows router-mediated swaps for allowlisted users and blocks non-allowlisted users.

---

### Impact Explanation

A curated pool (e.g., KYC-gated, institutional-only, or regulatory-restricted) relies on `SwapAllowlistExtension` as its sole on-chain access gate. Once the router is allowlisted — the only way to support the standard periphery path — any unprivileged address can execute swaps on the pool. This breaks the core pool functionality the extension was deployed to enforce and exposes LP funds to counterparties the pool admin explicitly excluded. Because the pool is oracle-priced, the direct per-swap loss is bounded by the spread, but the aggregate exposure across all unauthorized swaps is unbounded and constitutes a broken invariant with direct LP-fund impact.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the canonical, documented swap entry point for end users. Any pool admin who wants their allowlisted users to use the standard router **must** add the router to the allowlist. This is the expected operational path, making the bypass reachable in every realistic curated-pool deployment. No privileged access, no malicious setup, and no non-standard tokens are required — a plain EOA calling `exactInputSingle` is sufficient.

---

### Recommendation

The extension must gate on the **original user**, not the intermediary caller. Two viable approaches:

1. **Pass original sender in `extensionData`**: The router encodes `msg.sender` (the EOA) into `extensionData`; the extension decodes and verifies it. This requires a trusted encoding convention between the router and the extension.
2. **Dedicated router wrapper**: The router exposes a verified `originalSender()` view (stored in transient storage alongside the existing callback context), and the extension reads it directly from the router. This is analogous to how the router already stores payer identity in transient slots.

The `DepositAllowlistExtension` correctly gates on `owner` (the position beneficiary) rather than `sender` (the caller), which is the right model for the deposit path. [6](#0-5) 

The swap extension should adopt an equivalent pattern keyed to the economically relevant actor.

---

### Proof of Concept

```
Setup
─────
1. Deploy pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, alice, true)      // alice is the only allowed swapper
3. Pool admin calls setAllowedToSwap(pool, router, true)     // required so alice can use the router

Attack
──────
4. bob (not in allowlist) calls:
       router.exactInputSingle({
           pool:          pool,
           recipient:     bob,
           zeroForOne:    true,
           amountIn:      X,
           ...
       })

5. Router calls pool.swap(bob, true, X, ...)
   → pool.msg.sender = router
   → _beforeSwap(sender=router, ...)
   → extension checks allowedSwapper[pool][router] == true  ✓
   → swap executes

Result: bob swaps on a pool he was explicitly excluded from.
        The allowlist is fully bypassed with a single standard router call.
```
