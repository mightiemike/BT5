### Title
Division-by-Zero in `socializeSubaccount` Permanently Blocks Insolvent Subaccount Finalization When `openInterest == 0` — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when socializing residual bad debt. If all other participants have closed their positions in a product where the insolvent subaccount holds a negative `vQuoteBalance` and insurance is insufficient, `state.openInterest == 0` causes `MathSD21x18.div` to revert with `"DBZ"`, permanently blocking the `_finalizeSubaccount` flow.

---

### Finding Description

The division occurs at: [1](#0-0) 

```solidity
if (balance.vQuoteBalance < 0) {
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ← reverts if == 0
    );
```

`MathSD21x18.div` explicitly guards against zero: [2](#0-1) 

```solidity
function div(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        require(y != 0, ERR_DIV_BY_ZERO);
```

**How `state.openInterest` reaches 0 while `balance.vQuoteBalance < 0`:**

`state.openInterest` is the sum of absolute position sizes across all participants, maintained in `_updateBalance`: [3](#0-2) 

`_finalizeSubaccount` requires `balance.amount == 0` for every perp product before calling `socializeSubaccount`: [4](#0-3) 

This check only gates `balance.amount`; it does **not** gate `balance.vQuoteBalance`. A subaccount can have `amount == 0` (position fully closed/liquidated) while retaining a negative `vQuoteBalance` (accumulated unrealized loss / bad debt). This is precisely the scenario `socializeSubaccount` is designed to handle.

When `getStateAndBalance` is called for the insolvent subaccount inside `socializeSubaccount`, `_updateBalance(state, balance, 0, 0)` is invoked. Because `balance.amount == 0`, the pre-update step subtracts `0.abs() == 0` from `state.openInterest` and the post-update step adds `0` back — the memory copy of `state.openInterest` equals the actual on-chain global value. If every other participant has closed their position in that product, the global `openInterest` is `0`. [5](#0-4) 

The call that triggers the revert: [6](#0-5)

### Impact Explanation

The entire `liquidateSubaccountImpl` transaction reverts. The insolvent subaccount cannot be finalized: its bad debt is never socialized, insurance is never applied, and the subaccount remains in a zombie state. The protocol is left undercollateralized because the loss is not distributed to other participants and the accounting identity is broken. This directly matches the scoped impact: *"solvency or accounting failure … that … leaves the protocol undercollateralized."*

---

### Likelihood Explanation

The precondition is reachable without any privileged access:

1. An insolvent subaccount has `balance.amount == 0` (already liquidated) and `balance.vQuoteBalance < 0` (bad debt) in product P.
2. Insurance is insufficient to cover the full loss (common in extreme market conditions).
3. All other participants close their positions in product P, driving `openInterest` to `0`. This can be done organically or engineered by a single actor who controls the only remaining open position.
4. Any liquidator calls `liquidateSubaccountImpl` with `productId == type(uint32).max`.

The attacker does not need admin keys, sequencer access, or any privileged role. The only cost is closing one's own position in the affected product.

---

### Recommendation

Add a zero-guard before the division. If `openInterest == 0` and there is still residual bad debt, there is no one to socialize against; the loss should be absorbed entirely by the insurance fund or written off:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; write off the residual
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

### Proof of Concept

```solidity
// Minimal Hardhat/Foundry fuzz test (pseudo-code)
function test_socialize_dbz(int128 vQuoteLoss) public {
    // vQuoteLoss < 0, e.g. -1e18
    vm.assume(vQuoteLoss < 0);

    // Set up: insolvent subaccount with amount=0, vQuoteBalance=vQuoteLoss
    // All other participants have closed → openInterest == 0 in storage
    perpEngine.setState(productId, State({
        cumulativeFundingLongX18: 0,
        cumulativeFundingShortX18: 0,
        availableSettle: 0,
        openInterest: 0          // ← key precondition
    }));
    perpEngine.setBalance(productId, insolventSubaccount, Balance({
        amount: 0,
        vQuoteBalance: vQuoteLoss,  // ← negative bad debt
        lastCumulativeFundingX18: 0
    }));

    int128 insurance = 0; // insufficient

    // Expect revert with "DBZ"
    vm.expectRevert("DBZ");
    vm.prank(address(clearinghouse));
    perpEngine.socializeSubaccount(insolventSubaccount, insurance);
}
```

The `require(y != 0, ERR_DIV_BY_ZERO)` in `MathSD21x18.div` fires, reverting the entire `liquidateSubaccountImpl` call and permanently blocking finalization until `openInterest` becomes non-zero again — a condition the attacker can maintain indefinitely by not opening new positions.
