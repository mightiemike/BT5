### Title
`ValidResourceBounds::AllResources` with zero `l2_gas`/`l1_data_gas` silently collapses to `ValidResourceBounds::L1Gas` in P2P protobuf deserialization, producing a divergent transaction hash and wrong gas computation mode - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic (`is_zero()`) to decide between the `L1Gas` and `AllResources` variants. A V3 transaction submitted via RPC with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is hashed as `AllResources` (hash chain includes the `L1_DATA_GAS` element). When that block is later synced over P2P, the receiving node deserializes the same resource bounds as `L1Gas(X)` (hash chain omits `L1_DATA_GAS`), producing a different transaction hash and a different `GasVectorComputationMode`, breaking hash canonicalization and fee/gas accounting for syncing nodes.

---

### Finding Description

**Serialization boundary — `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`** [1](#0-0) 

The converter reads `l1_data_gas` with `unwrap_or_default()` (backward-compat shim for pre-0.13.3 peers that omit the field), then classifies the variant purely by whether both values are zero:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong for AllResources(X,0,0)
} else {
    ValidResourceBounds::AllResources(...)
})
```

This collapses two structurally distinct variants — `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` and `L1Gas(X)` — into the same deserialized output, even though the serializer always emits `l1_data_gas: Some(...)` for `AllResources`: [2](#0-1) 

**Hash domain divergence — `get_tip_resource_bounds_hash`**

The hash function branches on the same enum variant: [3](#0-2) 

- `L1Gas(X)` → `poseidon(tip, L1_GAS_felt, L2_GAS_felt_zero)` — **2 resource felts**
- `AllResources(X, 0, 0)` → `poseidon(tip, L1_GAS_felt, L2_GAS_felt_zero, L1_DATA_GAS_felt_zero)` — **3 resource felts**

These produce different field elements. The originating sequencer computes hash H₁ (AllResources path). The syncing node recomputes hash H₂ (L1Gas path). H₁ ≠ H₂.

**Gas computation mode divergence** [4](#0-3) 

`L1Gas` → `NoL2Gas`; `AllResources` → `All`. The syncing node uses the wrong mode for fee pre-validation and post-execution resource-bounds checking.

**Trigger path**

1. User submits `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: X (non-zero), l2_gas: 0, l1_data_gas: 0 }`. The gateway accepts it (at least one resource bound is non-zero). [5](#0-4) 

2. `convert_rpc_tx_to_internal` computes the hash via `InternalRpcInvokeTransactionV3::calculate_transaction_hash`, which calls `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(self.resource_bounds)` → H₁ (3-element hash). [6](#0-5) 

3. Transaction is included in a block. Block is synced via P2P. `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` deserializes `resource_bounds` as `ValidResourceBounds::L1Gas(X)`. [7](#0-6) 

4. Any subsequent hash recomputation on the syncing node yields H₂ ≠ H₁.

---

### Impact Explanation

- **Wrong transaction hash**: The syncing node derives a different `TransactionHash` for the same transaction. If the node validates hashes against the block header, it rejects a valid block (chain split / wrong state). If it does not, it stores the transaction under the wrong hash, corrupting RPC responses (`starknet_getTransactionByHash`, `starknet_getTransactionReceipt`).
- **Wrong gas computation mode**: `NoL2Gas` is used instead of `All`, causing incorrect resource-bounds pre-validation and post-execution fee checks on the syncing node — wrong receipt/fee output for any re-execution or simulation of the transaction.
- **Wrong fee estimation**: `starknet_estimateFee` and `starknet_simulateTransactions` on the syncing node return authoritative-looking wrong values for such transactions.

---

### Likelihood Explanation

Any unprivileged user can craft and submit a valid V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas` needs to be non-zero to pass gateway resource-bounds validation). The transaction will be included in a block (possibly as a reverted transaction due to out-of-gas, but still committed with its hash). Every syncing node that receives the block via P2P will silently misclassify the resource bounds variant.

---

### Recommendation

Replace the value-based heuristic with a presence-based one. The `l1_data_gas` field in the protobuf `ResourceBounds` message is `optional`; its presence — not its value — should determine the variant:

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let l1_gas: ResourceBounds = value.l1_gas.ok_or(missing("l1_gas"))?.try_into()?;
        let l2_gas: ResourceBounds = value.l2_gas.ok_or(missing("l2_gas"))?.try_into()?;
        Ok(match value.l1_data_gas {
            // Field absent → legacy pre-0.13.3 transaction → L1Gas variant
            None => ValidResourceBounds::L1Gas(l1_gas),
            // Field present (even if zero) → AllResources variant
            Some(l1_data_gas) => ValidResourceBounds::AllResources(AllResourceBounds {
                l1_gas,
                l2_gas,
                l1_data_gas: l1_data_gas.try_into()?,
            }),
        })
    }
}
```

This preserves backward compatibility with 0.13.2 peers (which omit `l1_data_gas`) while correctly round-tripping `AllResources` transactions that happen to have zero `l2_gas` and `l1_data_gas`.

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds, GasAmount, GasPrice,
};
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
use starknet_api::transaction::fields::Tip;

// Transaction as submitted via RPC: AllResources with zero l2/l1_data
let all_resources = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas:     ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(1) },
    l2_gas:     ResourceBounds::default(),   // zero
    l1_data_gas: ResourceBounds::default(),  // zero
});

// Transaction as deserialized on syncing node after P2P round-trip: L1Gas
let l1_gas_only = ValidResourceBounds::L1Gas(
    ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(1) },
);

let tip = Tip(0);
let h1 = get_tip_resource_bounds_hash(&all_resources, &tip).unwrap();
let h2 = get_tip_resource_bounds_hash(&l1_gas_only,  &tip).unwrap();

// h1 ≠ h2: AllResources hashes 3 resource felts; L1Gas hashes 2.
assert_ne!(h1, h2, "Hash domain collision: same effective bounds, different hashes");

// Gas computation mode also diverges
assert_eq!(
    all_resources.get_gas_vector_computation_mode(),
    GasVectorComputationMode::All,
);
assert_eq!(
    l1_gas_only.get_gas_vector_computation_mode(),
    GasVectorComputationMode::NoL2Gas,
);
```

The protobuf round-trip that triggers the collapse:

```rust
// Serializer (originating node): AllResources → protobuf (l1_data_gas: Some(zero))
let proto: protobuf::ResourceBounds = all_resources.into();
assert!(proto.l1_data_gas.is_some());  // present, value = 0

// Deserializer (syncing node): unwrap_or_default → zero → is_zero() → L1Gas
let recovered = ValidResourceBounds::try_from(proto).unwrap();
assert!(matches!(recovered, ValidResourceBounds::L1Gas(_)));  // wrong variant

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
