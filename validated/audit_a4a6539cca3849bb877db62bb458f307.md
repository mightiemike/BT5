Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing un-allowlisted callers to bypass the deposit guard — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual caller) and checks `owner` (the LP position recipient) against the allowlist. Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint, any un-allowlisted address can call `addLiquidity` with an allowlisted address as `owner`, pass the guard, and deposit tokens into the pool — fully bypassing the pool admin's configured access control.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the LP position recipient to `_beforeAddLiquidity`: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives both `sender` and `owner` as distinct parameters per the `IMetricOmmExtensions` interface, but silently discards `sender` (unnamed first parameter) and checks only `owner`: [2](#0-1) 

The contract's own NatSpec states *"Gates `addLiquidity` by depositor address"* — the depositor is the caller (`sender`), not the LP position recipient (`owner`): [3](#0-2) 

`addLiquidity` has no `msg.sender == owner` guard (unlike `removeLiquidity`, which enforces `if (msg.sender != owner) revert NotPositionOwner()` at line 206), so any caller can supply an arbitrary `owner`: [4](#0-3) 

The correct pattern is demonstrated by `SwapAllowlistExtension.beforeSwap`, which checks `sender` (the actual swapper), not `recipient`: [5](#0-4) 

The allowlist state is publicly readable (`allowedDepositor` is a `public` mapping), so an attacker can trivially identify an allowlisted address to use as `owner`: [6](#0-5) 

## Impact Explanation
The pool admin's deposit allowlist is fully bypassed. An un-allowlisted address deposits tokens (paid via the `addLiquidity` callback on `msg.sender`) while the LP position is credited to the allowlisted `owner`. With a colluding allowlisted counterparty, the un-allowlisted address deposits tokens → the allowlisted address removes liquidity and returns the proceeds → a complete round-trip that circumvents the access control. This constitutes an admin-boundary break: the pool admin's configured guard is bypassed by an unprivileged path with no special permissions required.

## Likelihood Explanation
The root cause is structural and always present once the extension is deployed. `addLiquidity` imposes no `msg.sender == owner` constraint, allowlisted addresses are publicly enumerable from on-chain events and the `allowedDepositor` mapping, and the bypass requires only a single `addLiquidity` call with a chosen `owner`. No privileged access, flash loan, or oracle manipulation is needed.

## Recommendation
Check `sender` (the actual depositor/caller) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Pool is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(pool, Bob, true)` — only Bob is allowlisted; Alice is not.
3. Alice (un-allowlisted) calls `pool.addLiquidity(owner=Bob, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(sender=Alice, owner=Bob, ...)`.
5. Extension checks `allowedDepositor[pool][Bob]` → **passes** (Bob is allowlisted; Alice's address is never checked).
6. `LiquidityLib.addLiquidity` executes: Alice's callback pays the tokens; Bob's position is credited.
7. Bob calls `removeLiquidity` and transfers the proceeds to Alice.
8. Alice has deposited and withdrawn from the allowlisted pool without ever appearing on the allowlist.
