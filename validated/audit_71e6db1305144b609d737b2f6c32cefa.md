### Title
Linked Signer Can Overwrite Its Own Delegation, Hijacking Subaccount Signing Authority — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `EndpointTx.sol` validates the transaction signature with `allowLinkedSigner = true`. This means the **current linked signer** — a delegated, less-privileged hot-wallet — is accepted as a valid signer for a `LinkSigner` transaction. A malicious or compromised linked signer can therefore replace itself (or the subaccount owner's delegation) with an attacker-controlled address, granting that address full signing authority over the victim's subaccount, including the ability to withdraw all collateral.

---

### Finding Description

The `linkedSigners` mapping stores a per-subaccount delegation:

```
mapping(bytes32 => address) internal linkedSigners;
``` [1](#0-0) 

When a `LinkSigner` transaction is processed, the handler calls `validateSignedTx` with the fifth argument hardcoded to `true` (`allowLinkedSigner`):

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [2](#0-1) 

`validateSignature` propagates this flag by passing `getLinkedSigner(sender)` to the verifier when `allowLinkedSigner` is `true`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [3](#0-2) 

The verifier then accepts a signature from **either** the subaccount owner address **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [4](#0-3) 

The result is that the current linked signer can sign a valid `LinkSigner` transaction for the subaccount it is linked to, overwriting `linkedSigners[subaccount]` with any address it chooses.

---

### Impact Explanation

Once the linked signer replaces the delegation with an attacker-controlled address, that address inherits full signing authority. Critically, `WithdrawCollateral` and `WithdrawCollateralV2` are also validated with `allowLinkedSigner = true`: [5](#0-4) [6](#0-5) 

The attacker can therefore immediately sign `WithdrawCollateral` transactions to drain the victim subaccount's collateral to an arbitrary address. The corrupted state is `linkedSigners[victim_subaccount]`, and the asset delta is the full collateral balance of that subaccount.

---

### Likelihood Explanation

The linked signer role is explicitly designed for hot wallets and automated trading bots — entities with a higher compromise surface than the cold-wallet subaccount owner. Any party who gains access to the linked signer key (through key theft, a malicious bot operator, or a compromised API key) can execute this attack with a single sequencer-submitted transaction. No admin access, governance capture, or social engineering is required. The attacker only needs to know the current nonce for the subaccount owner address, which is publicly readable from `nonces[address(uint160(bytes20(sender)))]`. [7](#0-6) 

---

### Recommendation

Change the `allowLinkedSigner` flag for `LinkSigner` transaction validation from `true` to `false`. Only the subaccount owner (the address embedded in the `sender` bytes32) should be permitted to change the linked signer delegation:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [8](#0-7) 

Additionally, add a unit test that:
1. Links a signer to a subaccount.
2. Attempts to call `LinkSigner` signed by the linked signer (not the owner).
3. Asserts the transaction reverts.

---

### Proof of Concept

**Setup:**
- Alice owns subaccount `aliceSub = bytes32(abi.encodePacked(0xAlice, "default"))`.
- Alice calls `LinkSigner` (signed by `0xAlice`) to set `linkedSigners[aliceSub] = 0xBob`.

**Attack:**
1. Bob (the linked signer) constructs a `SignedLinkSigner` payload:
   - `tx.sender = aliceSub`
   - `tx.signer = bytes32(bytes20(0xCharlie))` (attacker-controlled)
   - `tx.nonce = nonces[0xAlice]` (read from chain)
2. Bob signs the EIP-712 digest for this `LinkSigner` struct with his own key (`0xBob`).
3. The sequencer submits the transaction. `validateSignedTx` accepts Bob's signature because `allowLinkedSigner = true` and `getLinkedSigner(aliceSub) == 0xBob`.
4. `linkedSigners[aliceSub]` is overwritten with `0xCharlie`.
5. Charlie signs a `WithdrawCollateral` transaction for `aliceSub` and drains Alice's collateral.

The broken invariant: `linkedSigners[subaccount]` must only be writable by `address(uint160(bytes20(subaccount)))` (the subaccount owner). The `allowLinkedSigner = true` flag on `LinkSigner` violates this invariant. [2](#0-1)
