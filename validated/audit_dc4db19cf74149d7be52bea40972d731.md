Audit Report

## Title
Pool Admin Can Reduce `OracleValueStopLossExtension` Timelock to Zero, Atomically Disabling Stop-Loss Guard With No LP Reaction Window — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension` uses a per-pool timelock as the sole mechanism giving LPs a reaction window before the pool admin can weaken or disable the stop-loss guard. However, no minimum timelock is enforced at initialization or on updates, allowing the timelock to be set to zero. With a zero timelock, `_afterTimelock` returns `block.timestamp`, `_requireElapsed` passes immediately, and the admin can propose and execute any parameter change atomically in a single transaction — eliminating the LP protection window entirely and enabling immediate stop-loss disablement.

## Finding Description

**Root cause — no minimum timelock validation:**

`initialize` decodes and stores the timelock with no floor check:

```solidity
// L56-62
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);    // validated
_validateDecay(decayPerSecondE8); // validated
// timelock: NO validation — zero accepted silently
oracleStopLossConfig[pool] = PoolStopLossConfig({
    drawdownE6: drawdownE6, decayPerSecondE8: decayPerSecondE8, timelock: timelock, initialized: true
});
``` [1](#0-0) 

`executeOracleStopLossTimelock` similarly writes any `uint32` value including zero with no floor:

```solidity
// L86-94
uint32 timelock = sched.pendingTimelock;
oracleStopLossConfig[pool_].timelock = timelock;
``` [2](#0-1) 

**Timelock bypass mechanics:**

`_afterTimelock` computes `block.timestamp + 0 = block.timestamp` when timelock is zero: [3](#0-2) 

`_requireElapsed` checks `block.timestamp < executeAfter`; when `executeAfter == block.timestamp`, this is `false` and the check passes immediately: [4](#0-3) 

**Stop-loss disablement path:**

With timelock=0, the admin calls `proposeOracleStopLossDrawdown(pool, E6)` then `executeOracleStopLossDrawdown(pool)` in the same transaction. This sets `drawdownE6 = 1e6`, making:

```solidity
uint256 floorMultiplier = E6 - drawdown; // = 1e6 - 1e6 = 0
``` [5](#0-4) 

The breach check in `_applyWatermark` becomes `metric < (hwm * 0) / E6 = 0`, which is always false — the stop-loss never triggers regardless of value loss: [6](#0-5) 

**Protocol test confirms the behavior:**

The test suite's `setUp` initializes with `timelock=0` and `test_decayTimelockZeroExecutesImmediately` explicitly demonstrates that propose+execute succeed in the same block: [7](#0-6) [8](#0-7) 

## Impact Explanation

LPs deposit into a pool with `OracleValueStopLossExtension` expecting the stop-loss guard to protect their principal from oracle-price-driven value drain. The timelock is the explicit mechanism bounding pool admin trust. Once the timelock is zero, the admin can atomically set `drawdownE6 = E6` in one transaction, making `floorMultiplier = 0` and permanently disabling the stop-loss. LP funds are then exposed to unlimited value loss with no protection and no exit window. This is a direct admin-boundary break: the pool admin bypasses the timelock mechanism that is specifically designed to cap their power, matching the contest's stated valid impact category ("Admin-boundary break: pool admin exceeds caps, bypasses timelocks").

**Severity: Medium** — requires adversarial pool admin action, but the timelock is the explicit trust boundary the contest rules identify as a valid bypass target.

## Likelihood Explanation

- **Path 1 (day-one):** Pool is deployed with `timelock = 0` directly in `initialize`. No waiting required. All LPs who deposit are immediately unprotected.
- **Path 2 (reduction):** For a pool with an existing non-zero timelock `T`, the admin calls `proposeOracleStopLossTimelock(pool, 0)`, waits `T` seconds, then `executeOracleStopLossTimelock(pool)`. This is a one-time cost that permanently removes the LP protection window for all future parameter changes.
- The README confirms: "The finding can be valid only if the Pool admin can bypass the caps or honestly configured timelocks." Reducing the timelock to zero is precisely such a bypass — it eliminates the timelock mechanism itself. [9](#0-8) 

## Recommendation

1. **Enforce a minimum timelock at initialization** in `initialize`:
```solidity
uint32 constant MIN_TIMELOCK = 1 days;
require(timelock >= MIN_TIMELOCK, "TimelockTooShort");
```

2. **Enforce the same floor in `executeOracleStopLossTimelock`** before writing the new value:
```solidity
uint32 timelock = sched.pendingTimelock;
require(timelock >= MIN_TIMELOCK, "TimelockTooShort");
oracleStopLossConfig[pool_].timelock = timelock;
``` [2](#0-1) 

3. **(Optional, stronger):** Require `newTimelock >= currentTimelock` so the timelock can only increase, never decrease.

## Proof of Concept

```solidity
// Scenario A: Pool initialized with timelock = 0
extension.initialize(pool, abi.encode(uint32(500_000), uint32(58), uint32(0)));
// LPs deposit expecting stop-loss protection

// Admin atomically disables stop-loss in one transaction:
extension.proposeOracleStopLossDrawdown(pool, 1e6);  // drawdown = 100%
extension.executeOracleStopLossDrawdown(pool);        // executes immediately (timelock = 0)
// floorMultiplier = 1e6 - 1e6 = 0 → breach check always false → stop-loss permanently disabled

// Scenario B: Pool with existing timelock T
extension.proposeOracleStopLossTimelock(pool, 0);
// wait T seconds
extension.executeOracleStopLossTimelock(pool);  // timelock now = 0
// Repeat Scenario A atomically
```

The protocol's own test suite confirms the zero-timelock immediate execution path at `OracleValueStopLossSubExtension.t.sol` L249–255. [8](#0-7)
