### Title
No On-Chain Eligibility Verification in Merkle Distributor Allows Ineligible Users to Claim Tokens — (`File: core/contracts/Airdrop.sol`)

---

### Summary
`Airdrop.sol` distributes protocol tokens using a weekly Merkle tree. Eligibility is determined entirely off-chain. The contract performs no on-chain verification that a claimant actually performed the required action to earn the airdrop. If the Merkle tree is constructed incorrectly — accidentally or deliberately — an ineligible user can drain tokens from the distribution pool with a valid proof.

---

### Finding Description
The `_verifyProof` function is the sole gatekeeper for claims. It checks four conditions: the user has not already claimed, the week has a registered root, the address is not sanctioned, and the Merkle proof is cryptographically valid against the stored root. [1](#0-0) 

None of these checks verify that the claimant actually performed the protocol action that entitles them to the airdrop (e.g., trading activity, liquidity provision, or any other on-chain behavior tracked by the protocol). The contract blindly trusts the off-chain Merkle tree as the sole source of truth for eligibility.

Notably, `Airdrop.sol` imports `Endpoint.sol` — the contract that records all user trading activity — but never uses it. [2](#0-1) 

This unused import indicates that on-chain eligibility verification via `Endpoint.sol` was either considered and abandoned, or is an oversight. The Endpoint contract holds the ground-truth record of user activity that could serve as an independent eligibility check, but `_verifyProof` never interrogates it.

---

### Impact Explanation
If the Merkle tree for any week includes an address that did not perform the required action — whether due to an off-chain generation bug, a data pipeline error, or a deliberate manipulation of the tree construction process — that address can call `claim()` and receive a `safeTransfer` of the full `totalAmount` from the contract's token balance. [3](#0-2) 

The impact is direct token theft from the airdrop pool. Legitimate claimants may find the pool partially or fully drained, permanently losing their entitled rewards. There is no on-chain mechanism to detect or revert such a claim after the fact, since `claimed[week][sender]` is set to `totalAmount` and the double-claim check prevents any correction. [4](#0-3) 

---

### Likelihood Explanation
The likelihood is medium. The Merkle tree is generated off-chain and registered by the owner via `registerMerkleRoot`. Any bug in the off-chain data pipeline (e.g., incorrect snapshot block, address aliasing, data source error) directly translates into an exploitable on-chain state. The protocol's own import of `Endpoint.sol` — without using it — suggests the eligibility data exists on-chain but is not being cross-referenced, increasing the probability of a divergence between the tree and actual protocol activity. [5](#0-4) 

---

### Recommendation
At claim time, cross-reference the claimant's address against on-chain activity recorded in `Endpoint.sol` (already imported). For example, verify that the claimant has a non-zero trading history or deposit record for the relevant week before executing the transfer. This provides a defense-in-depth layer that catches Merkle tree errors before tokens are transferred, without requiring the user to re-prove their activity beyond what is already stored on-chain.

---

### Proof of Concept

1. Owner calls `registerMerkleRoot(week, root)` where `root` is generated from a flawed off-chain snapshot that includes address `0xAttacker` with `totalAmount = 1_000_000e18`, despite `0xAttacker` having performed no qualifying protocol action.
2. `0xAttacker` constructs a valid `ClaimProof` using the Merkle path for their leaf: `keccak256(keccak256(abi.encode(0xAttacker, 1_000_000e18)))`.
3. `0xAttacker` calls `claim([ClaimProof{week, 1_000_000e18, proof}])`.
4. `_verifyProof` passes all four checks: not claimed, week registered, not sanctioned, proof valid.
5. `SafeERC20.safeTransfer(token, 0xAttacker, 1_000_000e18)` executes.
6. `claimed[week][0xAttacker] = 1_000_000e18` is set, making the claim irreversible.
7. Legitimate users who were entitled to those tokens find the pool short by `1_000_000e18`. [6](#0-5)

```text
    function _verifyProof(
        uint32 week,
        address sender,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        require(claimed[week][sender] == 0, "Already claimed.");
        require(
            merkleRoots[week] != bytes32(0),
            "Week hasn't been registered."
        );
        require(
            !ISanctionsList(sanctions).isSanctioned(sender),
            "address is sanctioned."
        );
        bytes32 leaf = keccak256(
            bytes.concat(keccak256(abi.encode(sender, totalAmount)))
        );
        bool isValidLeaf = MerkleProof.verify(proof, merkleRoots[week], leaf);
        require(isValidLeaf, "Invalid proof.");
        claimed[week][sender] = totalAmount;
    }
```
