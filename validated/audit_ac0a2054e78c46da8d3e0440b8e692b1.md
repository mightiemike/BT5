### Title
Anyone Can Call `run_mainchain_gc` to Prematurely Purge Mainchain Headers, Breaking Transaction Verification - (File: contract/src/lib.rs)

---

### Summary

`run_mainchain_gc` is a public, state-mutating NEAR contract method with no caller access control. Any unprivileged account can call it with an arbitrarily large `batch_size`, immediately purging all GC-eligible block headers from contract storage. This corrupts the canonical header set that `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` depend on, causing those functions to panic for any transaction whose block was prematurely removed.

---

### Finding Description

`run_mainchain_gc` is declared at `contract/src/lib.rs:377` as a fully public `&mut self` method:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

The `#[pause(...)]` attribute only governs behaviour when the contract is **paused** (restricting callers to `UnrestrictedRunGC` role holders). When the contract is **running normally**, the decorator imposes no caller restriction at all. Any NEAR account — with no staking, no role, and no deposit — can invoke this method.

The function deletes headers from three storage maps:

```rust
self.remove_block_header(blockhash);          // removes from mainchain_header_to_height + headers_pool
self.mainchain_height_to_header.remove(&height);
```

and advances `mainchain_initial_blockhash` to the new oldest stored block.

The intended design is that GC is triggered **only** by the trusted relayer, indirectly, through `submit_blocks`:

```rust
#[trusted_relayer]
pub fn submit_blocks(...) -> PromiseOrValue<()> {
    ...
    self.run_mainchain_gc(num_of_headers);   // batch_size == number of headers just submitted
    ...
}
```

`submit_blocks` is protected by `#[trusted_relayer]`, so only staked, approved relayers can call it. But `run_mainchain_gc` is separately exposed as a top-level public method with no equivalent guard.

An attacker calls `run_mainchain_gc(u64::MAX)` directly. The function computes:

```rust
let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
```

With `batch_size = u64::MAX`, `selected_amount_to_remove` equals `total_amount_to_remove` — every block currently eligible for GC is deleted in a single transaction, regardless of how gradually the relayer intended to remove them.

---

### Impact Explanation

After the attacker's call, any invocation of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction whose block was removed will panic:

```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
```

The block hash is no longer in `mainchain_header_to_height` (deleted by GC), so the call panics unconditionally. Consumer contracts that depend on the light client for cross-chain verification (atomic swaps, bridged asset releases, etc.) receive a hard failure instead of a boolean result, breaking their logic at an attacker-chosen moment.

The attacker does not need to corrupt the chain or forge a proof — they simply accelerate deletion of headers that the relayer had not yet removed, invalidating in-flight or pending verification requests for those blocks.

---

### Likelihood Explanation

The entry point is a public NEAR method requiring no deposit, no role, and no staking. Any NEAR account can call it at any time the contract is unpaused. The only precondition is that `amount_of_headers_we_store > gc_threshold`, which is the normal steady-state of a live deployment (the chain grows continuously). The attack is a single transaction.

---

### Recommendation

Restrict `run_mainchain_gc` to the trusted relayer or a dedicated role, mirroring the protection on `submit_blocks`. The simplest fix is to add `#[trusted_relayer]` (or an equivalent `acl_require_role` check for `UnrestrictedRunGC`) to the public method signature, so that direct external calls are rejected with the same guard that protects `submit_blocks`.

---

### Proof of Concept

1. Deploy the contract with `gc_threshold = 52704` and submit 52705 blocks (one block beyond the threshold).
2. From any unprivileged NEAR account (no role, no stake), call:
   ```
   run_mainchain_gc(18446744073709551615)   // u64::MAX
   ```
3. The function computes `total_amount_to_remove = 1`, removes block at height 0 from all three maps, and advances `mainchain_initial_blockhash`.
4. Now call `verify_transaction_inclusion` with `tx_block_blockhash` = the hash of the removed block.
5. The call panics: `"block does not belong to the current main chain"` — even though the block was legitimately part of the main chain and within the gc_threshold window just moments before.

The attacker controls the timing of this deletion, allowing them to invalidate verification calls that a consumer contract is about to make or is waiting on. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
