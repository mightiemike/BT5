### Title
Unchecked Multiplication Overflow in `op_unknown` Cost Computation Enables Undercharged Execution - (File: src/more_ops.rs)

---

### Summary

In `op_unknown`, the final cost is computed by multiplying a base cost by `(cost_multiplier + 1)` using an unchecked `*=` operator. In Rust release builds, this multiplication wraps silently. A subsequent `cost > u32::MAX` guard may then pass on the wrapped (tiny) value, causing the operator to return a near-zero cost to the evaluator. This allows attacker-crafted CLVM programs to execute expensive unknown-opcode operations while consuming almost no cost budget — an exact analog of the `u64` accumulator overflow in the original report.

---

### Finding Description

`op_unknown` in `src/more_ops.rs` computes the cost of unknown opcodes in two stages. First, a base cost is accumulated per-argument (cost functions 0–3). Then the base cost is scaled by `(cost_multiplier + 1)`, where `cost_multiplier` is a `u64` derived from up to 4 opcode bytes (max value `u32::MAX = 4_294_967_295`):

```rust
check_cost(cost, max_cost)?;          // line 260: base cost checked here
cost *= cost_multiplier + 1;          // line 261: UNCHECKED multiplication
if cost > u32::MAX as u64 {           // line 262: post-multiplication guard
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
``` [1](#0-0) 

The `check_cost` call on line 260 only validates the pre-multiplication base cost against `max_cost`. The multiplication on line 261 is a plain `*=` with no `checked_mul` or saturation. In Rust release mode, overflow wraps modulo 2⁶⁴.

**Concrete overflow scenario:**

- `cost_multiplier + 1 = 4_294_967_296` (i.e., `cost_multiplier = u32::MAX`)
- Base `cost` = `4_294_967_297` (achievable when `max_cost` is large, e.g., the Chia block limit of ~11 billion)
- `cost * (cost_multiplier + 1) = 4_294_967_297 × 4_294_967_296 = 18_446_744_077_294_419_968`
- Wraps modulo 2⁶⁴ to: `18_446_744_077_294_419_968 - 2⁶⁴ = 7_584_803_328`

If the wrapped value is ≤ `u32::MAX` (4_294_967_295), the guard on line 262 passes and the function returns the wrapped cost as if it were legitimate. The attacker can tune `cost` and `cost_multiplier` to produce a wrapped result of, e.g., 1.

`Cost` is defined as `u64`: [2](#0-1) 

The `check_cost` guard only fires before the multiplication: [3](#0-2) 

The returned cost delta is added directly to the global running cost in `run_program`: [4](#0-3)

### Impact Explanation

An attacker submits a CLVM program (as serialized bytes) containing an unknown opcode crafted so that `cost * (cost_multiplier + 1)` wraps to a value ≤ `u32::MAX`. The evaluator accepts the operation and adds the tiny wrapped cost to the running total instead of the true cost. This allows the program to consume far more computational resources than the declared cost budget permits.

**Concrete broken invariant**: The cost returned by `op_unknown` is supposed to equal `base_cost × (cost_multiplier + 1)`. After wrapping, it equals `(base_cost × (cost_multiplier + 1)) mod 2⁶⁴`, which can be arbitrarily small. The global cost counter in `run_program` is therefore incorrect, and the `cost > effective_max_cost` guard on line 514 is bypassed for subsequent operations. [5](#0-4) 

Additionally, debug builds panic on overflow while release builds wrap — creating a **consensus divergence** between nodes compiled differently.

---

### Likelihood Explanation

- `op_unknown` is reachable in consensus mode (lenient mode) whenever a CLVM program contains an opcode not in the standard set. The Chia protocol explicitly allows unknown ops in block validation.
- The attacker fully controls the opcode bytes (determining `cost_multiplier` and `cost_function`) and the argument list (determining base `cost`).
- With a typical Chia block `max_cost` of ~11 billion, a `cost_multiplier` of `u32::MAX` requires a base `cost` of only ~4.3 billion to trigger overflow — well within the budget.
- The test infrastructure itself acknowledges the wrap-around risk for the 64-bit cost counter: [6](#0-5) 

---

### Recommendation

Replace the unchecked multiplication with a checked or saturating variant:

```rust
// Replace:
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}

// With:
let Some(scaled) = cost.checked_mul(cost_multiplier + 1) else {
    return Err(EvalErr::Invalid(o));
};
if scaled > u32::MAX as u64 {
    return Err(EvalErr::Invalid(o));
}
Ok(Reduction(scaled as Cost, allocator.nil()))
```

This mirrors the pattern already used correctly in `op_add`'s fast path: [7](#0-6)

### Proof of Concept

Craft a CLVM program in lenient/consensus mode with an unknown opcode byte sequence such that:

- The last byte encodes `cost_function = 1` (arithmetic-like, bits `[7:6] = 01`)
- The preceding bytes encode `cost_multiplier = u32::MAX = 0xFFFFFFFF`
- The argument list contains enough large atoms to push base `cost` above `⌈u64::MAX / (u32::MAX + 1)⌉ = 4_294_967_297`

Call `run_program` with `max_cost = 0` (which becomes `Cost::MAX`) or any value ≥ 4.3 billion. The multiplication `cost *= 4_294_967_296` wraps. If the wrapped value is ≤ `u32::MAX`, `op_unknown` returns `Ok(Reduction(wrapped_cost, nil))` with a near-zero cost, and the evaluator continues without correctly accounting for the true resource consumption.
