Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Allowing Any Unauthorized Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of the `addLiquidity` call) and instead checks `owner`, a free caller-supplied argument. Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint, any address not on the allowlist can pass the gate by supplying any already-allowed address as `owner`, completely nullifying the pool admin's deposit access control.

## Finding Description
`MetricOmmPool.addLiquidity` (lines 182–196) accepts a caller-supplied `owner` parameter and forwards both `msg.sender` (as `sender`) and `owner` to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

No `msg.sender == owner` check exists for `addLiquidity` (contrast `removeLiquidity` at line 206, which does enforce `if (msg.sender != owner) revert NotPositionOwner()`).

`DepositAllowlistExtension.beforeAddLiquidity` (lines 32–42) discards `sender` entirely (unnamed first parameter) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`SwapAllowlistExtension.beforeSwap` (lines 31–41) correctly gates on `sender`:

```solidity
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

The two extensions are architecturally inconsistent. The deposit guard checks the wrong actor, making the allowlist trivially bypassable.

**Exploit path:**
1. Pool is deployed with `DepositAllowlistExtension`; `allowedDepositor[pool][alice] = true`; `bob` is not on the allowlist.
2. `bob` calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
3. Extension checks `allowedDepositor[pool][alice]` → `true` → passes.
4. Pool executes the liquidity callback against `bob` (the actual `msg.sender`), pulling tokens from `bob`.
5. Position is minted under `alice`'s key.
6. `alice` calls `pool.removeLiquidity(alice, salt, deltas, "")` and receives the tokens.

## Impact Explanation
The `DepositAllowlistExtension` is the sole mechanism by which a pool admin restricts who may provide liquidity (e.g., KYC-gated or curated pools). The bypass renders this control completely ineffective: any unprivileged address can inject liquidity into the pool by nominating any known allowed address as `owner`. This constitutes an admin-boundary break — an unprivileged path bypasses a pool-admin-configured access control — and breaks the core functionality of the extension. With collusion between `bob` and `alice`, unauthorized liquidity can be deposited and recovered, laundering unauthorized participation through the curated pool.

## Likelihood Explanation
The bypass requires only a single direct call to `pool.addLiquidity` with `owner` set to any address already in the public `allowedDepositor` mapping. No special privileges, flash loans, or complex setup are needed. The allowed address list is fully public on-chain. Any motivated actor can trivially identify an allowed address and execute the bypass repeatedly.

## Recommendation
Change `beforeAddLiquidity` to gate on `sender` (the actual depositor/payer) rather than `owner` (the position recipient), consistent with `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Also update `isAllowedToDeposit`, `setAllowedToDeposit`, and related events/docs to reflect that the checked address is the depositor/payer, not the position owner.

## Proof of Concept
```
Setup:
  - Pool deployed with DepositAllowlistExtension
  - allowedDepositor[pool][alice] = true
  - bob is NOT on the allowlist

Attack:
  1. bob calls pool.addLiquidity(alice, salt, deltas, callbackData, "")
  2. _beforeAddLiquidity(bob, alice, ...) is called on the extension
  3. Extension checks allowedDepositor[pool][alice] → true → passes (bob is never checked)
  4. Pool pulls tokens from bob via liquidity callback
  5. Position minted for alice
  6. alice calls pool.removeLiquidity(alice, salt, deltas, "") → receives tokens
  7. alice and bob split proceeds off-chain

Result: bob bypassed the deposit allowlist entirely with a single direct call.
```

Foundry test: deploy pool with `DepositAllowlistExtension`, whitelist `alice`, have `bob` (non-whitelisted) call `addLiquidity` with `owner = alice`, assert the call succeeds and tokens are pulled from `bob`.
