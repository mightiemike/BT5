### Title
`run_program` Silently Converts `max_cost = 0` to Unlimited Execution, Bypassing Cost Enforcement — (`File: src/run_program.rs`)

---

### Summary

The `run_program` engine in `src/run_program.rs` silently converts a caller-supplied `max_cost` of `0` into `Cost::MAX` (i.e., `u64::MAX`), completely disabling cost enforcement. This behavior is undocumented in the public Python API (`wheel/src/api.rs`), where `max_cost` is a plain caller-controlled `u64`. Any downstream Python caller that passes `max_cost = 0` — expecting it to mean "use a safe default" or "no limit needed" — will execute arbitrary CLVM programs with no cost ceiling, enabling unbounded CPU consumption.

---

### Finding Description

Inside `RunProgramContext::run_program`, the very first action taken on the caller-supplied `max_cost` is:

```rust
// max_cost is always in effect, and necessary to prevent wrap-around of
// the cost integer.
let max_cost = if max_cost == 0 { Cost::MAX } else { max_cost };
``` [1](#0-0) 

`Cost` is `u64`, so `Cost::MAX = 18_446_744_073_709_551_615`. The cost check at line 514 (`if cost > effective_max_cost`) will never fire for any realistic program when `max_cost` has been silently promoted to this sentinel value. [2](#0-1) 

This behavior is exposed directly through the Python-facing entry point:

```rust
pub fn run_serialized_chia_program(
    py: Python,
    program: &[u8],
    args: &[u8],
    max_cost: Cost,   // ← caller-controlled, no validation
    flags: u32,
) -> PyResult<(u64, LazyNode)> {
    ...
    Ok(py.detach(|| run_program(&mut allocator, &dialect, program, args, max_cost)))
}
``` [3](#0-2) 

The `max_cost` value is forwarded to `run_program` without any guard. There is no documentation in the Python module registration that warns callers about the zero-means-unlimited semantic. [4](#0-3) 

The asymmetry with the softfork operator is telling: the softfork handler **explicitly rejects** `expected_cost == 0` with `EvalErr::CostExceeded`, showing the developers recognise zero as a dangerous sentinel — yet the top-level `max_cost` path has the opposite, silent behaviour. [5](#0-4) 

---

### Impact Explanation

A node or mempool implementation that calls `run_serialized_chia_program(program, args, max_cost=0, flags)` — perhaps treating `0` as "no explicit limit set" or "use a safe default" — will execute every submitted CLVM program with no cost ceiling. An attacker who submits a maximally expensive but syntactically valid CLVM program (e.g., a deeply recursive multiply chain) to such a node causes unbounded CPU consumption per evaluation call, constituting a remote DoS against the node. Because the returned `cost` value is also used by callers to enforce fee/resource accounting, the silent bypass corrupts that accounting as well.

---

### Likelihood Explanation

The Python API exports `max_cost` as a plain `u64` with no type-level or docstring constraint. The zero-means-unlimited convention is only noted in a Rust-internal comment invisible to Python callers. A developer integrating the wheel who passes `0` as a "no-op" or "uncapped" sentinel — a natural interpretation — silently disables the only DoS guard in the engine. The CLI tool `tools/src/bin/run-clvm.rs` hard-codes `11_000_000_000` as its `max_cost`, demonstrating that callers are expected to supply a concrete non-zero value, but nothing in the API enforces this. [6](#0-5) 

---

### Recommendation

1. **Reject `max_cost = 0` explicitly** in `run_program` (or in `run_serialized_chia_program`) with a clear error, mirroring the softfork guard's treatment of `expected_cost == 0`.
2. Alternatively, document the zero-means-unlimited semantic prominently in the Python module docstring and in the `#[pyfunction]` signature so callers cannot accidentally opt out of cost enforcement.
3. Consider adding a `#[pyo3(signature = (program, args, max_cost, flags))]` annotation with a non-zero default or a range check to make the invariant machine-enforceable at the API boundary.

---

### Proof of Concept

```python
import clvm_rs

# A CLVM program that multiplies large numbers repeatedly — expensive but valid.
# Serialised hex of: (* (q . 99999999999999999999) (q . 99999999999999999999) ...)
expensive_program = bytes.fromhex("...")  # attacker-supplied bytes
args = bytes.fromhex("ff80")             # nil environment

# Passing max_cost=0 silently disables cost enforcement.
# run_program internally converts 0 → Cost::MAX (u64::MAX).
cost, result = clvm_rs.run_serialized_chia_program(
    expensive_program,
    args,
    max_cost=0,   # ← zero bypasses the cost limit entirely
    flags=0,
)
# cost is returned as the true accumulated cost, which may be arbitrarily large.
# No EvalErr::CostExceeded is ever raised regardless of program complexity.
print(f"Ran with no limit, accumulated cost = {cost}")
```

The concrete corrupted result is: `EvalErr::CostExceeded` is **never raised** for any program when `max_cost=0` is passed, because `effective_max_cost` is promoted to `u64::MAX` at line 494 of `src/run_program.rs`, and the guard `cost > effective_max_cost` at line 514 can never be satisfied.     py: Python,
    program: &[u8],
    args: &[u8],
    max_cost: Cost,
    flags: u32,

    let flags = ClvmFlags::from_bits_truncate(flags);
    let mut allocator = if flags.contains(ClvmFlags::LIMIT_HEAP) {
        Allocator::new_limited(500000000)

        Allocator::new()
    };

    let r: Response = (|| -> PyResult<Response> {
        let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
        let args = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
        let dialect = ChiaDialect::new(flags);

        Ok(py.detach(|| run_program(&mut allocator, &dialect, program, args, max_cost)))
    })()?;
