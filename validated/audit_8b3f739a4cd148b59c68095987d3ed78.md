### Title
Stale `lastDecayTs` After Decay Rate Change Causes Watermarks to Instantly Collapse to Zero, Bypassing the Stop-Loss Guard — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`executeOracleStopLossDecay` updates `oracleStopLossConfig[pool_].decayPerSecondE8` but never resets `highWatermarks[pool_][binIdx].lastDecayTs`. On the next swap, the new (higher) rate is applied retroactively to the full elapsed time since the last swap, collapsing watermarks to zero and silently disabling the stop-loss for that swap.

---

### Finding Description

`OracleValueStopLossExtension` protects LPs by tracking per-bin value-per-share watermarks and blocking swaps that breach a configured drawdown floor. The decay clock for each bin is stored in `BinHighWatermarks.lastDecayTs` and is updated in two places:

1. `executeOracleStopLossHighWatermarks` — explicitly sets `lastDecayTs: uint32(block.timestamp)`.
2. `_checkAndUpdateWatermarks` — sets `hwmS.lastDecayTs = uint32(block.timestamp)` after every swap. [1](#0-0) [2](#0-1) 

`executeOracleStopLossDecay`, however, only writes the new rate and clears the schedule — it never touches `lastDecayTs`: [3](#0-2) 

When the next swap arrives, `_checkAndUpdateWatermarks` computes:

```solidity
uint256 dt = block.timestamp - hwmS.lastDecayTs;   // stale: counts from last swap, not from rate change
```

and passes it to `_decayed`:

```solidity
function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;          // ← watermark collapses to zero
    return hwm - (hwm * factor) / E8;
}
``` [4](#0-3) 

If `ratePerSecondE8 * dt ≥ 1e8`, the watermark is zeroed. A zero watermark then enters `_applyWatermark`:

```solidity
function _applyWatermark(uint256 metric, uint256 hwm, uint256 floorMultiplier)
    private pure returns (uint256 newHwm, bool breached)
{
    if (metric >= hwm) return (metric, false);   // metric ≥ 0 is always true → no breach
    ...
}
``` [5](#0-4) 

`breached` is never set to `true`, so `afterSwap` does not revert, and the stop-loss is silently disabled for that swap. [6](#0-5) 

---

### Impact Explanation

The `afterSwap` hook is the sole enforcement point for the stop-loss. When the watermark collapses to zero, `_applyWatermark` always returns `breached = false`, meaning any swap — including one that drains LP value far below the configured drawdown floor — is permitted. LPs suffer direct principal loss beyond the threshold they configured and timelocked into place.

---

### Likelihood Explanation

The condition is reachable whenever:

- A pool runs with `decayPerSecondE8 = 0` (the default after initialization, since only `drawdownE6` and `decayPerSecondE8` are set at init time and zero decay is a valid config).
- The admin legitimately proposes and executes a non-zero decay rate via the timelocked flow.
- Sufficient time has elapsed since the last swap so that `newRate × dt ≥ 1e8`.

With `decayPerSecondE8 = 58` (≈5 %/day, the example cited in the NatDoc), the threshold is crossed after only `1e8 / 58 ≈ 1,724,138 seconds ≈ 20 days` of swap inactivity — a realistic gap for a low-volume pool. The trigger is a valid, timelocked admin action, not a malicious one.

---

### Recommendation

In `executeOracleStopLossDecay`, flush the accumulated decay under the **old** rate before installing the new one. The simplest correct fix is to iterate over all bins and update their watermarks and `lastDecayTs` at the moment of the rate change:

```solidity
function executeOracleStopLossDecay(address pool_) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    if (sched.pendingDecayExecuteAfter == 0) revert OracleStopLossNoPendingDecay(pool_);
    _requireElapsed(sched.pendingDecayExecuteAfter);

    // Flush decay under the OLD rate for every bin before switching rates.
    uint256 oldRate = oracleStopLossConfig[pool_].decayPerSecondE8;
    // (caller supplies the bin range, or the contract iterates known bins)
    // For each bin: apply _decayed(hwm, oldRate, dt) and reset lastDecayTs = block.timestamp

    uint32 decay = sched.pendingDecayPerSecondE8;
    oracleStopLossConfig[pool_].decayPerSecondE8 = decay;
    (sched.pendingDecayPerSecondE8, sched.pendingDecayExecuteAfter) = (0, 0);
    emit OracleStopLossDecaySet(pool_, decay);
}
```

Alternatively, store a `rateChangedAt` timestamp alongside the rate and split `dt` in `_decayed` into two segments (old rate × pre-change interval + new rate × post-change interval).

---

### Proof of Concept

1. Pool is initialized with `drawdownE6 = 50_000` (5 % drawdown) and `decayPerSecondE8 = 0` (no decay).
2. At block timestamp `T0`, a swap occurs. `_checkAndUpdateWatermarks` sets `lastDecayTs = T0` and watermarks to their current high values (e.g., `hwm0 = hwm1 = 1e6`).
3. No swaps occur for 25 days (`dt_idle = 2_160_000 s`).
4. Admin proposes `decayPerSecondE8 = 58` and, after the timelock, executes it at `T1 = T0 + 25 days`. `lastDecayTs` is **not** updated.
5. One second later (`T2 = T1 + 1`), an attacker (or any user) submits a swap that would normally breach the stop-loss (e.g., a large directional trade that drops `metricT1` well below `hwm1 × 0.95`).
6. `afterSwap` calls `_checkAndUpdateWatermarks`. `dt = T2 - T0 = 2_160_001 s`. `factor = 58 × 2_160_001 = 125_280_058 > 1e8`. `_decayed` returns `0`.
7. `_applyWatermark(metricT1_low, 0, 950_000)` → `metricT1_low ≥ 0` → returns `(metricT1_low, false)`. No revert.
8. The swap settles, LP value is drained beyond the 5 % drawdown floor, and the stop-loss never fired.
