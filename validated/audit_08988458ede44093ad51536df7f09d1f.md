### Title
Governance `BonusApy` update retroactively recalculates all pending unclaimed bonus rewards using the new rate — (`x/tieredrewards/keeper/claim_rewards.go`, `x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

The `x/tieredrewards` module computes bonus rewards for all historical time segments using the **current** tier `BonusApy` fetched at claim time, not the APY that was in effect when each segment was earned. Because governance can update `BonusApy` at any time via `MsgUpdateTier`, any pending unclaimed bonus is silently recalculated at the new rate, causing users to receive more or less than they actually earned.

---

### Finding Description

`processEventsAndClaimBonus` in `claim_rewards.go` fetches the live tier once at the start of the function:

```go
tier, err := k.getTier(ctx, pos.TierId)   // line 167 — always current state
```

It then passes this single `tier` object into `computeSegmentBonus` for **every** historical segment replayed from past validator events:

```go
// historical segments (lines 175-179)
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)

// current (live) segment (line 211)
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
```

Inside `computeSegmentBonus` (`bonus_rewards.go` line 42), the multiplier applied to every segment is `tier.BonusApy`:

```go
return tokens.
    Mul(tier.BonusApy).          // always the current APY
    MulInt64(durationSeconds).
    QuoInt64(types.SecondsPerYear).
    TruncateInt()
```

The `Position` proto does **not** store the `BonusApy` at creation time, and no per-segment APY snapshot is recorded. The `Tier` struct is a mutable governance-controlled object:

```go
type Tier struct {
    BonusApy      cosmossdk_io_math.LegacyDec   // updated freely by MsgUpdateTier
    MinLockAmount cosmossdk_io_math.Int
    ...
}
```

Governance can change `BonusApy` at any time via `MsgUpdateTier` (confirmed by integration tests in `test_tieredrewards_auth.py` lines 337–354 which show a live governance proposal updating tier parameters). There is no mechanism to snapshot or version the APY per earning period.

---

### Impact Explanation

- **APY decreased by governance**: A user who earned 6 months of bonus at 8% APY but has not yet claimed will have all 6 months recalculated at the new lower rate (e.g., 2%). They receive less than they earned — a direct loss of accrued bonus funds.
- **APY increased by governance**: The same user receives more than they earned, draining the rewards pool beyond what was budgeted.

The corrupted value is the **bonus reward amount sent from the `RewardsPoolName` module account to the position owner** via `bankKeeper.SendCoinsFromModuleToAccount` (`claim_rewards.go` line 239). The exact shortfall or excess equals `(newApy − oldApy) × shares × tokensPerShare × unclaimedDuration / SecondsPerYear` for each historical segment.

---

### Likelihood Explanation

- Governance proposals to adjust `BonusApy` are a normal, expected operation (the ADR explicitly lists `MsgUpdateTier` as a governance message).
- Users who claim infrequently (e.g., once per year, which is the natural cadence for a 1-year or 5-year lock tier) accumulate large unclaimed windows and are maximally exposed.
- No special attacker capability is required: any governance-passing proposal triggers the bug for all existing positions in the affected tier.

---

### Recommendation

Snapshot the `BonusApy` that applies to each earning segment. Two concrete approaches:

1. **Store APY in the position at creation time** and use it for all segments until the position is updated (analogous to saving the amount paid at purchase time).
2. **Record APY changes as validator-style events** (a `TIER_APY_CHANGE` event with a timestamp), so `processEventsAndClaimBonus` can apply the correct APY to each time window during replay, similar to how `tokensPerShare` is already snapshotted per validator event.

---

### Proof of Concept

1. Governance creates Tier 1 with `BonusApy = 0.08` (8%).
2. Alice calls `MsgLockTier` and locks 1,000,000 basecro. Her position is created; `LastBonusAccrual = T0`.
3. Six months pass (`T1 = T0 + 15,768,000s`). Alice does not claim.
4. Governance passes `MsgUpdateTier` setting `BonusApy = 0.02` (2%).
5. Alice calls `MsgClaimTierRewards` at `T1`.
6. `processEventsAndClaimBonus` fetches the current tier (`BonusApy = 0.02`) and computes:
   - `bonus = 1,000,000 × 0.02 × 15,768,000 / 31,536,000 = 10,000 basecro`
7. Alice receives 10,000 basecro. She should have received `1,000,000 × 0.08 × 0.5 = 40,000 basecro`.
8. Alice loses 30,000 basecro of legitimately earned bonus.
