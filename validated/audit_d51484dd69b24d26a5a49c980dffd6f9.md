Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end user, allowing any user to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool binds to `msg.sender` of the `pool.swap()` call. When `MetricOmmSimpleRouter` intermediates a swap, the pool's `msg.sender` is the router address, not the end user. If the router is allowlisted, every user — including those the admin intended to block — can bypass the restriction by routing through the router. Conversely, if the router is not allowlisted, legitimately allowlisted users who route through the canonical periphery are incorrectly blocked.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this `sender` value directly to the extension hook: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the `msg.sender` inside the pool: [4](#0-3) 

This is structurally broken compared to `DepositAllowlistExtension.beforeAddLiquidity`, which correctly checks the explicit `owner` parameter — the actual LP regardless of who calls the function: [5](#0-4) 

The `swap` interface has no explicit swapper identity parameter analogous to `owner` in `addLiquidity`: [6](#0-5) 

**Bypass path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict trading.
2. Pool admin allowlists `address(router)` to support router-mediated swaps.
3. `blockedUser` calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient=blockedUser, ...)` — pool's `msg.sender` is the router.
5. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. `blockedUser` successfully trades in the restricted pool.

**Dual failure:** If the router is NOT allowlisted, allowlisted users who route through the router are incorrectly blocked, breaking the core swap flow for the canonical periphery path.

## Impact Explanation

A curated pool's swap allowlist is the primary access-control boundary for restricted trading venues. Bypassing it allows unauthorized users to trade against LP positions never intended to be exposed to them. If the restriction was designed to prevent front-running, MEV extraction, or regulatory non-compliance, the bypass directly exposes LPs to losses from harmful trading patterns they opted out of. This constitutes broken core pool functionality and an admin-boundary break reachable by any unprivileged user via the canonical public router.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical public swap entrypoint. Pool admins who configure a swap allowlist and also want to support router-mediated swaps will naturally allowlist the router — the exact condition that enables the bypass. The trigger requires no privileged access beyond the pool admin's own intended configuration, and the exploit path is a standard public router call available to any user.

## Recommendation

Mirror the deposit extension's design: add an explicit `swapper` identity parameter to the pool's `swap` function (analogous to `owner` in `addLiquidity`), and have `MetricOmmSimpleRouter` forward `msg.sender` (the actual end user) as that parameter. The extension then checks the explicit swapper identity rather than the intermediary caller.

Alternatively, require the extension to decode the actual swapper from `extensionData` when the caller is a known router, and have the router inject the real user address into `extensionData` before forwarding to the pool.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, allowedUser, true)
  - Pool admin calls setAllowedToSwap(pool, address(router), true)
  - blockedUser is NOT on the allowlist

Attack:
  1. blockedUser calls router.exactInputSingle({pool: pool, recipient: blockedUser, ...})
  2. Router calls pool.swap(recipient=blockedUser, ...)
  3. Pool calls _beforeSwap(sender=address(router), ...)
  4. Extension checks allowedSwapper[pool][router] → true
  5. Swap executes — blockedUser successfully trades in the restricted pool

Expected: revert NotAllowedToSwap
Actual:   swap succeeds

Foundry test outline:
  function test_swapAllowlistBypassViaRouter() public {
      // deploy pool with SwapAllowlistExtension
      // allowlist router, not blockedUser
      // vm.prank(blockedUser)
      // router.exactInputSingle(...)
      // assert swap succeeded (no revert)
  }
```
