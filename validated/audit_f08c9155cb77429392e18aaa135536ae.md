### Title
`ValidResourceBounds::L1Gas` vs `AllResources` hash asymmetry across protobuf P2P boundary produces wrong transaction hash - (`File: crates/starknet_api/src/transaction_hash.rs`, `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

`get_tip_resource_bounds_hash` produces structurally different hash preimages for `ValidResourceBounds::L1Gas` and `ValidResourceBounds::AllResources` even when the numeric gas values are identical (both with zero `l2_gas` and `l1_data_gas`). The protobuf deserializer `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` silently collapses any `AllResources` whose `l2_gas` and `l1_data_gas` are both zero into `L1Gas`. A transaction submitted via RPC as `AllResources(l1_gas, l2_gas=0, l1_data_gas=0)` therefore receives a different hash on any node that reconstructs it from protobuf (P2P state sync), breaking hash canonicality across the network boundary.

---

### Finding Description

**Hash asymmetry in `get_tip_resource_bounds_hash`**

`get_tip_resource_bounds_hash` builds the resource-bounds sub-hash by conditionally appending `l1_data_gas`:

```
L1Gas(l1_gas)          → Poseidon(tip, concat(L1_GAS, l1_gas), concat(L2_GAS, 0))          // 2 resource felts
AllResources(l1, 0, 0) → Poseidon(tip, concat(L1_GAS, l1_gas), concat(L2_GAS, 0), concat(L1_DATA_GAS, 0))  // 3 resource felts
``` [1](#0-0) 

The two variants produce **different Poseidon digests** even when every numeric field is identical, because the input length differs.

**Protobuf converter silently changes the variant**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` maps any wire message with zero `l2_gas` and zero `l1_data_gas` to `ValidResourceBounds::L1Gas`, regardless of whether the sender intended `AllResources`: [2](#0-1) 

The comment `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.` confirms this is a deliberate backward-compat shim that is still active.

**The P2P state-sync path uses `InvokeTransactionV3` (with `ValidResourceBounds`)**

`TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` calls `ValidResourceBounds::try_from(...)` directly: [3](#0-2) 

So any `InvokeTransactionV3` received over P2P state sync whose `l2_gas = 0` and `l1_data_gas = 0` is reconstructed as `L1Gas`, not `AllResources`.

**Hash is recomputed from deserialized fields**

`validate_transaction_hash` recomputes the hash from the deserialized `Transaction` struct and checks it against the stored hash: [4](#0-3) 

Because the deserialized variant is now `L1Gas` instead of `AllResources`, `get_transaction_hash` → `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` produces a hash that does **not** match the hash originally computed at ingestion time (which used `AllResources`).

**Contrast with the RPC/mempool path**

`InternalRpcInvokeTransactionV3` and `RpcInvokeTransactionV3` both store `resource_bounds: AllResourceBounds` (not `ValidResourceBounds`), and their `InvokeTransactionV3Trait` impl always returns `ValidResourceBounds::AllResources(self.resource_bounds)`: [5](#0-4) 

So the RPC/mempool/consensus path always hashes with `AllResources` and is unaffected. Only the P2P state-sync path, which goes through `InvokeTransactionV3`, is vulnerable.

---

### Impact Explanation

A node syncing state via P2P will compute a wrong transaction hash for any `InvokeV3` transaction that was originally submitted with `AllResources` bounds where `l2_gas = 0` and `l1_data_gas = 0`. `validate_transaction_hash` will reject the transaction (hash mismatch), causing the syncing node to fail to ingest a valid block. This is a **wrong receipt/state result from blockifier/execution logic for accepted input** and a **transaction conversion that binds the wrong hash**.

---

### Likelihood Explanation

Any `InvokeV3` transaction submitted via the RPC gateway with `AllResourceBounds { l2_gas: 0, l1_data_gas: 0, l1_gas: X }` triggers the bug. This is a valid, accepted transaction (the gateway accepts it; the stateless validator does not reject zero l2/data gas). The condition is reachable by any unprivileged user. The bug fires deterministically on every P2P state-sync peer that receives such a transaction.

---

### Recommendation

1. **Remove the variant-collapsing logic** from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Once 0.13.2 support is dropped (as the TODO indicates), assert that `l1_data_gas` is always present and always deserialize to `AllResources` for V3 transactions.

2. **Alternatively**, make `get_tip_resource_bounds_hash` produce the same hash for `L1Gas` and `AllResources` with zero values by always including the `l1_data_gas` element (even when zero) for all V3 transactions, matching the canonical Starknet spec.

3. **Add a round-trip hash test** that serializes an `InvokeTransactionV3` with `AllResources(l1_gas, l2_gas=0, l1_data_gas=0)` to protobuf, deserializes it, and asserts the recomputed hash equals the original.

---

### Proof of Concept

```
1. User submits RpcInvokeTransactionV3 with:
     resource_bounds = AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }

2. Gateway converts to InternalRpcInvokeTransactionV3 (AllResourceBounds).
   Hash H_A = get_invoke_transaction_v3_hash(...)
            = Poseidon(..., get_tip_resource_bounds_hash(AllResources(X,0,0)), ...)
            // includes concat(L1_DATA_GAS, 0) as third resource felt

3. Transaction is committed to a block. Block is broadcast over P2P.

4. Syncing node receives protobuf::InvokeV3 with resource_bounds.l2_gas=0, l1_data_gas=0.
   TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
     l1_data_gas.is_zero() && l2_gas.is_zero() → ValidResourceBounds::L1Gas(X)

5. Syncing node calls validate_transaction_hash(tx, ..., H_A):
   H_B = get_invoke_transaction_v3_hash(...)
       = Poseidon(..., get_tip_resource_bounds_hash(L1Gas(X)), ...)
       // only 2 resource felts, no L1_DATA_GAS element

6. H_A ≠ H_B → validate_transaction_hash returns false → block rejected by syncing node.
```
