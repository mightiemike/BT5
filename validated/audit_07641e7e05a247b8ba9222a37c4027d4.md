### Title
`skip_pow_verification` Is Permanently Immutable After Initialization, With No Admin Setter — (`File: contract/src/lib.rs`)

### Summary

The `BtcLightClient` contract stores `skip_pow_verification` as a persistent state field set exclusively at `init` time. No privileged setter function exists anywhere in the contract to update it post-deployment. If the contract is deployed with `skip_pow_verification = true` — even accidentally — there is no on-chain recovery path short of a full contract upgrade. Any authorized relayer can then permanently submit fake Bitcoin headers with no PoW validation, corrupting the canonical chain and causing `verify_transaction_inclusion` to return `true` for fraudulent transactions.

### Finding Description

`BtcLightClient` stores three critical configuration fields set only at `init`:

- `skip_pow_verification: bool` — controls whether PoW is checked on every submitted header
- `gc_threshold: u64` — controls how many mainchain blocks are retained
- `network: Network` — mainnet vs. testnet [1](#0-0) 

All three are written once in `init`: [2](#0-1) 

A grep for any setter (`fn set_`, `fn update_`, `fn change_`, `fn toggle_`) returns zero matches across the entire repository. The `#[near]` impl block for `BtcLightClient` exposes no function that mutates `skip_pow_verification`, `gc_threshold`, or `network` after initialization.

`skip_pow_verification` is the most security-critical of the three. It is read directly in `submit_blocks` and passed to `submit_block_header`: [3](#0-2) 

Inside `submit_block_header`, when `skip_pow_verification` is `true`, both the difficulty target check (`check_target`) and the hash-vs-target comparison are entirely skipped: [4](#0-3) 

The same bypass applies to the Dogecoin AuxPoW path: [5](#0-4) 

The documentation explicitly warns this flag is for testing only: [6](#0-5) 

Yet there is no admin-callable function to correct it if it is set incorrectly in production. The `migrate` function preserves the field value unchanged: [7](#0-6) 

### Impact Explanation

If `skip_pow_verification = true` is deployed to production (by mistake or misconfiguration), any account holding the `UnrestrictedSubmitBlocks` or `DAO` role can submit arbitrary fake Bitcoin block headers — headers with any `bits`, any `nonce`, any `prev_block_hash` — and the contract will accept them, update `mainchain_tip_blockhash`, and record them in `headers_pool`. Downstream callers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will receive `true` for transactions that never existed on the real Bitcoin chain. The canonical chain state is permanently corrupted until a contract upgrade is staged and deployed, which itself requires the `CodeStager`/`CodeDeployer`/`DAO` role pipeline to be operational. [8](#0-7) 

### Likelihood Explanation

Medium. The `init` function is `#[private]`, so only the contract account itself can call it, but the relayer's `config.toml.example` exposes `skip_pow_verification` as a top-level init parameter: [9](#0-8) 

Deployment pipelines that copy a test config to production, or operators who set the flag for initial testing and forget to redeploy with `false`, will produce a permanently insecure contract. The absence of any corrective setter means the window of exposure is unbounded.

### Recommendation

Add a privileged setter restricted to `Role::DAO` (or a new `ConfigManager` role) that allows updating `skip_pow_verification` and `gc_threshold` post-deployment:

```rust
#[access_control_any(roles(Role::DAO))]
pub fn set_skip_pow_verification(&mut self, value: bool) {
    self.skip_pow_verification = value;
}

#[access_control_any(roles(Role::DAO))]
pub fn set_gc_threshold(&mut self, value: u64) {
    self.gc_threshold = value;
}
```

This mirrors the recommendation from the external report: all security-critical settings should be adjustable by the owner/DAO without requiring a contract upgrade.

### Proof of Concept

1. Deploy `BtcLightClient` with `skip_pow_verification: true` (e.g., copied from a test config).
2. Observe: no setter exists to flip it back to `false`. The entire public API of `BtcLightClient` contains no such function.
3. Grant `UnrestrictedSubmitBlocks` to a relayer account (normal operational step).
4. Relayer calls `submit_blocks` with a fabricated header: arbitrary `bits`, `nonce`, `prev_block_hash` pointing to the current tip, and a fake `merkle_root`.
5. The contract accepts it — `skip_pow_verification` causes both `check_target` and the hash ≤ target check to be bypassed.
6. `mainchain_tip_blockhash` now points to the fake header.
7. A downstream contract calls `verify_transaction_inclusion` with a fake `tx_id` and a trivially constructed single-element merkle proof matching the fake `merkle_root`. The function returns `true`. [10](#0-9)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
