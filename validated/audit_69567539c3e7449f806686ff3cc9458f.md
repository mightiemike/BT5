### Title
`gas_prices_to_hash` omits `l2_gas_consumed` and `next_l2_gas_price` from block hash preimage for Starknet versions > 0.14.0 - (File: crates/starknet_api/src/block_hash/block_hash_calculator.rs)

### Summary

`gas_prices_to_hash` in `crates/starknet_api/src/block_hash/block_hash_calculator.rs` is the sole function that serializes gas-price fields into the block hash preimage. For Starknet versions ≥ 0.13.4 it hashes only six fields (`l1_gas_price_wei/fri`, `l1_data_gas_price_wei/fri`, `l2_gas_price_wei/fri`) under the `STARKNET_GAS_PRICES0` domain. A developer TODO at line 416 explicitly acknowledges that `l2_gas_consumed` and `next_l2_gas_price` must be added "after 0.14.0", but the code was never updated. Meanwhile the external block-hash CLI invoked by `echonet/echo_center.py` already passes both fields for every 0.14.x block, and `StorageBlockHeader` stores them as first-class fields. The Rust sequencer therefore produces a structurally different (wrong) block hash for every block at version > 0.14.0 compared to what the protocol and external verifiers expect.

### Finding Description

`gas_prices_to_hash` is called from `calculate_block_hash` and determines the single `gas_prices_hash` felt that enters the Poseidon block-hash preimage:

```

    l1_gas_price: &GasPricePerToken,
    l1_data_gas_price: &GasPricePerToken,
    l2_gas_price: &GasPricePerToken,
    block_hash_version: &BlockHashVersion,
) -> Vec<Felt> {
    if block_hash_version >= &BlockHashVersion::V0_13_4 {
        vec![
            HashChain::new()
                .chain(&STARKNET_GAS_PRICES0)

                .chain(&l2_gas_price.price_in_wei.0.into())
                .chain(&l2_gas_price.price_in_fri.0.into())
                .get_poseidon_hash(),
        ]
    } else { ... }
}

`StorageBlockHeader` — the authoritative on-disk representation — stores both fields as mandatory members: [3](#0-2)

The `BlockHashVersion` enum has only two variants (`V0_13_2`, `V0_13_4`); all 0.14.x Starknet versions therefore fall through to the `V0_13_4` branch, which silently omits the two fee-market fields: [5](#0-4) 

### Impact Explanation

Every block produced at Starknet version > 0.14.0 receives a wrong `block_hash` because the `gas_prices_hash` felt inside the Poseidon preimage is computed over only six gas-price scalars instead of the eight (adding `l2_gas_consumed` and `next_l2_gas_price`) that the protocol requires. The wrong hash is stored in `StorageBlockHeader.block_hash`, returned by RPC, used in `PartialBlockHash` for consensus, and committed on-chain. Any external verifier or node that computes the hash correctly (as the echonet CLI already does) will disagree with the sequencer's stored value, causing state-root / block-hash divergence. This matches the **Critical** impact category: *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.*

### Likelihood Explanation

The defect is triggered by every block whose `starknet_version` maps to a version > 0.14.0. The versioned-constants registry already includes versions up to 0.14.4. No attacker action is required; the sequencer hits this path automatically for every block it produces under those versions. The TODO comment and the echonet CLI evidence confirm the omission is real and not guarded by any other gate.

### Recommendation

1. Add `l2_gas_consumed: GasAmount` and `next_l2_gas_price: GasPrice` to `PartialBlockHashComponents`.
2. Introduce a new `BlockHashVersion::V0_14_1` (or the appropriate activation version) and extend `gas_prices_to_hash` to chain both fields after `l2_gas_price_fri` when `block_hash_version >= V0_14_1`.
3. Update `PartialBlockHashComponents::new` to populate the two new fields from `BlockInfo` / `FeeMarketInfo`.
4. Add a regression test analogous to the existing `test_block_hash_regression` that covers a 0.14.x version and asserts the hash changes when either `l2_gas_consumed` or `next_l2_gas_price` changes.

### Proof of Concept

The divergence is directly observable by comparing the two code paths:

**Rust sequencer** (current, for any 0.14.x block):
```
Poseidon("STARKNET_GAS_PRICES0",
    l1_gas_price_wei, l1_gas_price_fri,
    l1_data_gas_price_wei, l1_data_gas_price_fri,
    l2_gas_price_wei, l2_gas_price_fri)          // 7 felts
```

**External CLI / echonet** (for the same 0.14.x block):
```
Poseidon("STARKNET_GAS_PRICES0",
    l1_gas_price_wei, l1_gas_price_fri,
    l1_data_gas_price_wei, l1_data_gas_price_fri,
    l2_gas_price_wei, l2_gas_price_fri,
    l2_gas_consumed, next_l2_gas_price)          // 9 felts
```

Because Poseidon is collision-resistant, the two hashes are distinct for any non-zero `l2_gas_consumed` or `next_l2_gas_price`. Since `FeeMarketInfo` in every real 0.14.x blob carries non-zero values for both fields (e.g. `l2_gas_consumed: 988191555`, `next_l2_gas_price: 0x1dcd65000` in the test fixture at `crates/apollo_starknet_client/resources/reader/block_post_0_14_3.json`), the sequencer's stored `block_hash` will always differ from the protocol-correct value for those blocks.
