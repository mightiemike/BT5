### Title
`SwapAllowlistExtension` Gates Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. If the pool admin allowlists the router (required for router-mediated swaps to function), every user on the network can bypass the swap allowlist by calling the router.

---

### Finding Description

**Call chain for a router-mediated swap:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` — `msg.sender` to the router is the user.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(...)` — `msg.sender` to the pool is the **router**.
3. Pool executes `_beforeSwap(msg.sender, ...)` — `sender` = **router address**.
4. `SwapAllowlistExtension.beforeSwap(sender=router, ...)` evaluates `allowedSwapper[pool][router]`.

The extension never sees the actual user's address. The check is entirely on the router's address.

```solidity
// MetricOmmPool.sol:230-240

  recipient,
  zeroForOne,
  amountSpecified,

```

```solidity
// SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
// msg.sender = pool, sender = router → checks allowedSwapper[pool][router]
```

The pool admin faces an impossible choice:

| Router allowlisted? | Effect |
|---|---|
| No | Router-mediated swaps revert for **everyone**, including allowlisted users |
| Yes | **Any** user can bypass the allowlist by calling the router |

There is no configuration that simultaneously enables router-mediated swaps and enforces per-user allowlisting.

---

### Impact Explanation

A pool deployer who configures `SwapAllowlistExtension` to restrict swaps to a set of KYC'd or permissioned addresses cannot enforce that restriction when `MetricOmmSimpleRouter` is used. Any unprivileged user can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the public router and execute swaps against the restricted pool. The pool's liquidity is exposed to all users, defeating the access-control invariant the extension was deployed to enforce. This constitutes broken core pool functionality with direct fund-impact consequences: LPs who deposited under the assumption that only allowlisted counterparties could trade against them are exposed to unrestricted swap flow.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface for the protocol. Any pool that uses `SwapAllowlistExtension` and also wants to support router-mediated swaps must allowlist the router. This is the expected operational configuration. The bypass requires no special privileges, no flash loans, and no unusual token behavior — any EOA can call the router directly.

---

### Recommendation

The `SwapAllowlistExtension` must check the **original user** rather than the immediate caller of `pool.swap()`. Two approaches:

1. **Pass the original initiator through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires router cooperation and is trust-dependent.

2. **Check `recipient` instead of `sender`**: For single-hop swaps the recipient is often the user, but this is not reliable for multi-hop paths.

3. **Preferred — check both `sender` and `recipient`, or require direct pool calls for allowlisted pools**: Document that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` unless the router is allowlisted (which defeats the purpose), and provide a router variant that forwards the original `msg.sender` as a verified parameter.

The cleanest fix is to have the router encode the original `msg.sender` into `extensionData` and have the extension verify it, with the `onlyPool` guard ensuring the pool (not an attacker) is the caller of `beforeSwap`.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - allowedSwapper[pool][alice] = true   (alice is the only allowed swapper)
  - allowedSwapper[pool][router] = true  (router allowlisted so alice can use it)
  - Add liquidity to the pool

Attack:
  - bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(recipient=bob, ...) — msg.sender to pool = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  - Swap executes successfully for bob despite bob not being allowlisted

Result:
  - bob swaps against a pool intended to be restricted to alice only
  - The swap allowlist guard is completely bypassed
