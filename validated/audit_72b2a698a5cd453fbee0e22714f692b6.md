### Title
`gas_prices_to_hash()` omits `l2_gas_consumed` and `next_l2_gas_price` from block hash for Starknet version ≥ 0.14.0 — (`crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

---

### Summary

The `gas_prices_to_hash()` function, which computes the gas-prices component committed into every block hash, is missing two fee-market fields — `l2_gas_consumed` and `next_l2_gas_price` — that the protocol requires for Starknet version ≥ 0.14.0. The `echonet` reference implementation already passes both fields to the block-hash CLI, creating a canonical divergence: the Rust sequencer produces a different block hash than the authoritative formula for every block at version ≥ 0.14.0.

---

### Finding Description

`gas_prices_to_hash()` is the single function that serialises gas-price data into the Poseidon hash chain used by `calculate_block_hash()`. For `BlockHashVersion >= V0_13_4` it hashes exactly six felts:

```
Poseidon("STARKNET_GAS_PRICES0",
         l1_gas_price_wei, l1_gas_price_fri,
         l1_data_gas_price_wei, l1_data_gas_price_fri,
         l2_gas_price_wei, l2_gas_price_fri)
```

A developer TODO at line 416 explicitly acknowledges the gap:

```rust
// TODO(Ayelet): add l2_gas_consumed, next_l2_gas_price after 0.14.0.
``` [1](#0-0) 

The `PartialBlockHashComponents` struct — the sole input to `calculate_block_hash()` — does not carry `l2_gas_consumed` or `next_l2_gas_price` at all, so the fields cannot reach the hash even if the function were patched in isolation: [2](#0-1) 

Meanwhile, `BlockHeaderWithoutHash` (the canonical in-memory block header) does carry both fields: [3](#0-2) 

The `echonet` OS-verification tool, which is the reference implementation used for mainnet replay, already passes both fields to the block-hash CLI for every block it processes: [4](#0-3) 

The production test fixture for version 0.14.4 confirms both fields are present in real blocks: [5](#0-4) 

The sequencer itself writes `l2_gas_consumed` and `next_l2_gas_price` into `BlockHeaderWithoutHash` when committing a block: [6](#0-5) 

The storage layer persists both fields: [7](#0-6) 

All the data is available; it is simply never fed into the hash.

---

### Impact Explanation

Every block produced at Starknet version ≥ 0.14.0 receives a wrong `block_hash`. The block hash is the root commitment used for:

- **Consensus**: validators verify the proposer's `PartialBlockHash` / `calculate_block_hash` output; a wrong hash breaks agreement.
- **State sync**: syncing nodes re-derive the hash from stored header fields and compare; the mismatch causes sync rejection.
- **RPC `starknet_getBlockWithTxHashes` / `starknet_getBlockWithTxs`**: the returned `block_hash` field is authoritative-looking but wrong.
- **`get_block_hash` syscall**: contracts that call `get_block_hash` for recent blocks receive the wrong value, which can break on-chain logic that depends on block-hash randomness or cross-chain proofs.

This matches the **Critical** impact category: *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input* (wrong block hash stored and served), and the **High** category: *RPC execution … returns an authoritative-looking wrong value*.

---

### Likelihood Explanation

- The network is already running at version 0.14.4 (evidenced by the production test fixture).
- The `echonet` reference tool already includes the two fields, confirming the external protocol expects them.
- The TODO comment confirms the Rust implementation knows it is incomplete.
- No version gate prevents the incomplete formula from being applied to 0.14.x blocks today; `gas_prices_to_hash` falls through to the `>= V0_13_4` branch for all 0.14.x blocks.

---

### Recommendation

1. Add `l2_gas_consumed: GasAmount` and `next_l2_gas_price: GasPrice` to `PartialBlockHashComponents`.
2. Introduce a new `BlockHashVersion::V0_14_0` (or the exact activation version) in the `BlockHashVersion` enum.
3. Update `gas_prices_to_hash()` to accept and chain both fields when `block_hash_version >= V0_14_0`:
   ```
   Poseidon("STARKNET_GAS_PRICES0",
            l1_gas_price_wei, l1_gas_price_fri,
            l1_data_gas_price_wei, l1_data_gas_price_fri,
            l2_gas_price_wei, l2_gas_price_fri,
            l2_gas_consumed, next_l2_gas_price)   // NEW
   ```
4. Propagate the two new fields from `BlockHeaderWithoutHash` through `PartialBlockHashComponents::new()` and all callers of `calculate_block_hash()`.
5. Add a regression test analogous to `test_block_hash_regression` for `BlockHashVersion::V0_14_0`.

---

### Proof of Concept

For a block at `starknet_version = "0.14.4"` with:
- `l2_gas_consumed = 988191555`
- `next_l2_gas_price = 0x1dcd65000`

The Rust sequencer computes:

```
gas_prices_hash = Poseidon("STARKNET_GAS_PRICES0",
                            l1_gas_wei, l1_gas_fri,
                            l1_data_wei, l1_data_fri,
                            l2_gas_wei, l2_gas_fri)
```

The `echonet` CLI (and the canonical protocol) computes:

```
gas_prices_hash = Poseidon("STARKNET_GAS_PRICES0",
                            l1_gas_wei, l1_gas_fri,
                            l1_data_wei, l1_data_fri,
                            l2_gas_wei, l2_gas_fri,
                            988191555,              // l2_gas_consumed
                            0x1dcd65000)            // next_l2_gas_price
```

The two Poseidon hashes are different, so `calculate_block_hash()` produces a different final `block_hash` than the canonical value. Every block at version ≥ 0.14.0 is affected. [8](#0-7) [9](#0-8)

    l1_data_gas_price: &GasPricePerToken,
    l2_gas_price: &GasPricePerToken,
    block_hash_version: &BlockHashVersion,

        vec![
            HashChain::new()
                .chain(&STARKNET_GAS_PRICES0)
                .chain(&l1_gas_price.price_in_wei.0.into())
                .chain(&l1_gas_price.price_in_fri.0.into())

        vec![
            l1_gas_price.price_in_wei.0.into(),

            l1_data_gas_price.price_in_fri.0.into(),
        ]
    }
