### Title
`RuntimeDialect::softfork_extension()` Silently Ignores Its `ext` Parameter, Causing False Assumptions About Softfork Extension Behavior — (File: `src/runtime_dialect.rs`)

---

### Summary

`RuntimeDialect::softfork_extension()` accepts an `ext: u32` parameter that is supposed to identify which softfork extension is active, but unconditionally returns `OperatorSet::Default` for every possible value. Simultaneously, `RuntimeDialect::op()` accepts an `_extensions: OperatorSet` parameter (the output of `softfork_extension`) but also ignores it entirely. Any caller using `RuntimeDialect` who passes a softfork extension number expecting it to enable additional operators (e.g., extension 1 for keccak256) will have that expectation silently violated — the extension is accepted without error but has zero effect on operator dispatch.

---

### Finding Description

The `Dialect` trait defines two cooperating methods for softfork extension handling:

1. `softfork_extension(&self, ext: u32) -> OperatorSet` — maps an extension number to an `OperatorSet` that signals which additional operators are available inside the softfork guard.
2. `op(..., extension: OperatorSet) -> Response` — uses the `OperatorSet` to adjust the active flag set before dispatching to an operator function.

`ChiaDialect` implements both correctly:

```rust
// src/chia_dialect.rs lines 144–154
let flags = self.flags
    | match extension {
        OperatorSet::Default => ClvmFlags::empty(),
        OperatorSet::Bls => ClvmFlags::empty(),
        OperatorSet::Keccak => ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD,
    };
```

`RuntimeDialect` breaks both halves of this contract:

```rust
// src/runtime_dialect.rs lines 73–75
fn softfork_extension(&self, _ext: u32) -> OperatorSet {
    OperatorSet::Default
}
```

```rust
// src/runtime_dialect.rs lines 40–61
fn op(
    &self,
    allocator: &mut Allocator,
    o: NodePtr,
    argument_list: NodePtr,
    max_cost: Cost,
    _extensions: OperatorSet,   // ← accepted, never read
) -> Response {
    ...
    return f(allocator, argument_list, max_cost, self.flags);  // ← self.flags only
    ...
}
```

The `_ext` parameter in `softfork_extension` and the `_extensions` parameter in `op` are both prefixed with `_`, the Rust idiom for intentionally unused variables. Neither influences the returned `OperatorSet` nor the flags passed to operator functions.

The execution path in `run_program.rs` that calls these methods is:

```rust
// src/run_program.rs (apply_op / softfork handling)
let extension = dialect.softfork_extension(ext);   // always Default for RuntimeDialect
...
dialect.op(allocator, o, argument_list, max_cost, extension)  // extension ignored
```

---

### Impact Explanation

**Impact: Low.**

A CLVM program running under `RuntimeDialect` that uses `(softfork cost 1 program env)` (extension 1 = keccak256) will have the keccak256 operator treated as unknown inside the guard. If `NO_UNKNOWN_OPS` is not set, the unknown operator returns nil with a defined cost instead of the actual hash, silently producing a wrong result. If `NO_UNKNOWN_OPS` is set, the program errors out. In neither case does the caller's extension argument produce the expected behavior. The impact is bounded because `RuntimeDialect` is not the consensus-critical path — `ChiaDialect` handles on-chain validation — but callers using `RuntimeDialect` for off-chain program evaluation or tooling will receive incorrect results without any error signal.

---

### Likelihood Explanation

**Likelihood: High.**

The `Dialect` trait's `softfork_extension` method exists precisely to allow callers to pass extension numbers and receive meaningful operator sets. Any caller who constructs a `RuntimeDialect`, runs a program containing a softfork guard with a non-zero extension number, and expects extension-specific operators (e.g., keccak256 under extension 1) to be available will always have that assumption violated. The parameter is accepted without complaint, so there is no error to alert the caller.

---

### Recommendation

1. In `RuntimeDialect::softfork_extension()`, either map known extension numbers to their corresponding `OperatorSet` values (mirroring `ChiaDialect`), or document explicitly that `RuntimeDialect` does not support softfork extensions and raise an error when a non-zero extension is encountered.

2. In `RuntimeDialect::op()`, consume the `extension: OperatorSet` parameter and merge it into the active flags before dispatching, consistent with how `ChiaDialect::op()` handles it.

---

### Proof of Concept

A CLVM program that uses the keccak256 operator inside a softfork guard (extension 1):

```
(softfork (q . <cost>) (q . 1) (q . (keccak256 (q . "hello"))) (q . ()))
```

Under `ChiaDialect` with no special flags: `softfork_extension(1)` returns `OperatorSet::Keccak`, which causes `op()` to add `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` to the active flags, making keccak256 available inside the guard.

Under `RuntimeDialect`: `softfork_extension(1)` returns `OperatorSet::Default` (the `_ext` argument is discarded). `op()` ignores the `_extensions` argument and dispatches with `self.flags` only. The keccak256 opcode (62) is not in the `f_lookup` table and is not enabled by any flag, so it is treated as an unknown operator — returning nil or raising an error — regardless of the extension number the caller supplied.
