### Title
`withdrawCollateral` Transfers Tokens Before Updating SpotEngine Balance, Exposing Stale Health/Balance State During ERC777 Callback Window — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs the token transfer to the recipient **before** calling `spotEngine.updateBalance`. If the collateral token is ERC777-compatible, the recipient contract's `tokensReceived` hook fires while the sender's SpotEngine balance still reflects the pre-withdrawal amount. Any external system that calls `getHealth` or `getBalance` during this window receives an inflated, incorrect value.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

1. `handleWithdrawTransfer(token, sendTo, amount, idx)` — triggers the full transfer chain ending in `token.safeTransfer(sendTo, amount)` inside `BaseWithdrawPool.handleWithdrawTransfer`
2. `spotEngine.updateBalance(productId, sender, amountRealized)` — deducts the withdrawn amount from the sender's internal balance
3. `spotEngine.assertUtilization(productId)`
4. `require(getHealth(sender, healthType) >= 0, ...)` [1](#0-0) 

The token transfer at step 1 routes through `Clearinghouse.handleWithdrawTransfer`: [2](#0-1) 

Which calls `BaseWithdrawPool.submitWithdrawal`, which calls `BaseWithdrawPool.handleWithdrawTransfer`: [3](#0-2) 

The final `token.safeTransfer(to, amount)` call is the callback point. If `to` is a contract and the token is ERC777-compatible, the `tokensReceived` hook executes **before** `spotEngine.updateBalance` has been called. At that moment:

- `SpotEngineState.getBalance(productId, sender)` returns the pre-withdrawal balance (inflated by `amount`)
- `Clearinghouse.getHealth(sender, ...)` returns health computed from that inflated balance

Neither `getHealth` nor `getBalance` carries a `nonReentrant` guard: [4](#0-3) [5](#0-4) 

The `ERC20Helper.safeTransfer` uses a raw low-level `call`, which will forward execution to any ERC777 `tokensReceived` hook registered by the recipient: [6](#0-5) 

---

### Impact Explanation

Any external protocol that reads `Clearinghouse.getHealth` or `SpotEngine.getBalance` as a health/collateral oracle during the callback window will observe the withdrawing subaccount as holding more collateral than it actually does. This allows the subaccount owner to appear solvent or over-collateralized to a third-party lending or margin system at the exact moment they are withdrawing funds, enabling them to borrow or open positions against collateral they no longer hold. The Nado protocol's own internal health check (line 419) is unaffected because it executes after `spotEngine.updateBalance`; only external consumers of the public state-reading functions are exposed.

---

### Likelihood Explanation

The trigger requires an ERC777-compatible token to be listed as a collateral product. ERC777 tokens are backward-compatible with ERC20 and can be added via `SpotEngine.addOrUpdateProduct` without any explicit ERC777 awareness. The `WithdrawCollateralV2` transaction type allows the withdrawer to specify an arbitrary `sendTo` address, meaning the attacker fully controls the callback recipient. The sequencer must include the withdrawal transaction, which is a normal user-initiated action. Likelihood is medium-low: it is gated on an ERC777 token being listed, but the code path is otherwise fully reachable by an unprivileged user. [7](#0-6) 

---

### Recommendation

Apply the checks-effects-interactions pattern: call `spotEngine.updateBalance` and `spotEngine.assertUtilization` **before** `handleWithdrawTransfer`. This ensures the SpotEngine balance is finalized before any external callback can observe it. Alternatively, add a reentrancy guard to `withdrawCollateral` and document that `getHealth` / `getBalance` may return stale values if called during a token transfer callback from a collateral token with hooks.

---

### Proof of Concept

1. Deploy a malicious ERC777 token `MalToken` and register it as a collateral product in SpotEngine.
2. Deploy an attacker contract `AttackerRecipient` that implements `tokensReceived`. Inside `tokensReceived`, call `clearinghouse.getHealth(senderSubaccount, INITIAL)` and record the result.
3. Fund `senderSubaccount` with 1000 `MalToken` via `depositCollateral`.
4. Submit a `WithdrawCollateralV2` transaction with `sendTo = address(AttackerRecipient)` and `amount = 1000`.
5. When the sequencer processes the transaction, `BaseWithdrawPool.handleWithdrawTransfer` calls `MalToken.transfer(AttackerRecipient, 1000)`, triggering `tokensReceived`.
6. Inside `tokensReceived`, `getHealth(senderSubaccount, INITIAL)` returns health computed from the full 1000-token balance — as if no withdrawal occurred.
7. After the transaction completes, `getHealth` correctly reflects the zero balance.
8. An external lending protocol that trusted the health value observed in step 6 would have granted credit against collateral that no longer exists. [8](#0-7) [9](#0-8)

        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
