The key to resolving this question is tracing exactly what `getCoreRisk` puts in the `longWeight` field of `CoreRisk`, and then checking whether `existingWeight` and `spreadWeight` are computed correctly for the short-spot/long-perp case.

---

## Root Cause Trace

### Step 1: `getCoreRisk` always returns `longWeight`

`BaseEngine.getCoreRisk` hardcodes `1` as the amount argument to `_getWeightX18`:

```solidity
return IProductEngine.CoreRisk(
    amount,
    risk.priceX18,
    RiskHelper._getWeightX18(risk, 1, healthType)  // ← hardcoded 1 (positive)
);
``` [1](#0-0) 

Because `_getWeightX18` branches on `amount >= 0`, passing `1` unconditionally means `CoreRisk.longWeight` always holds `longWeightInitialX18` (e.g. 0.9), **regardless of whether the actual position is long or short**. [2](#0-1) 

### Step 2: `existingWeight` is wrong for short-spot/long-perp

In `Clearinghouse.getHealth`:

```solidity
int128 existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2;
``` [3](#0-2) 

For a **short-spot/long-perp** position, the base health calculation in `_calculateProductHealth` uses `shortWeightInitialX18` (e.g. 1.1) for the short-spot leg. But `existingWeight` uses `longWeightInitialX18` (0.9) for both legs, giving `existingWeight = 0.9` instead of the correct `(1.1 + 0.9) / 2 = 1.0`. [4](#0-3) 

### Step 3: `_getSpreadWeightX18` also uses `longWeight` for the short-spot leg

For the short-spot/long-perp case (`spotCoreRisk.amount < 0`), the else branch fires:

```solidity
} else {
    spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
}
``` [5](#0-4) 

With `spotCoreRisk.longWeight = 0.9`: `spreadWeight = 1 - (1 - 0.9)/5 = 0.98`.

### Step 4: The health delta is a penalty, not a bonus

```solidity
health += basisAmount
    .mul(spotCoreRisk.price + perpCoreRisk.price)
    .mul(spreadWeight - existingWeight);
``` [6](#0-5) 

For short-spot/long-perp:

| Variable | Value | Sign |
|---|---|---|
| `basisAmount` | `-max(spotAmount, -perpAmount)` | **negative** |
| `spotPrice + perpPrice` | oracle prices | positive |
| `spreadWeight - existingWeight` | `0.98 - 0.9 = 0.08` | **positive** |

Result: `negative × positive × positive = negative` — a **health penalty**.

With the correct `existingWeight = 1.0`: `spreadWeight - existingWeight = 0.98 - 1.0 = -0.02`, giving `negative × positive × negative = positive` — the intended **health bonus**.

The magnitude of the error per unit of basis: `0.08` (actual) vs `-0.02` (correct), a swing of `0.10 × basisAmount × (spotPrice + perpPrice)` in the wrong direction.

### Title
Wrong `existingWeight` in `getHealth` turns spread bonus into penalty for short-spot/long-perp positions — (`core/contracts/Clearinghouse.sol` / `core/contracts/libraries/RiskHelper.sol`)

### Summary
`getCoreRisk` always populates `CoreRisk.longWeight` with the product's long weight (by passing a hardcoded `1` to `_getWeightX18`). `Clearinghouse.getHealth` then uses this field as `existingWeight` for both legs of a spread. For a short-spot/long-perp position the spot leg's actual health weight is `shortWeightInitialX18` (> 1), not `longWeightInitialX18` (< 1). This makes `existingWeight` too small, so `spreadWeight - existingWeight > 0`, and because `basisAmount < 0` for this leg orientation, the spread adjustment subtracts from health instead of adding to it.

### Finding Description
`BaseEngine.getCoreRisk` (line 190) calls `RiskHelper._getWeightX18(risk, 1, healthType)` — the literal `1` forces the long-weight branch regardless of the actual position sign. The `CoreRisk` struct has only one weight field (`longWeight`), so there is no way for callers to distinguish the actual weight applied in the base health sum.

`Clearinghouse.getHealth` computes `existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2`. For short-spot/long-perp this is `(longWeight_spot + longWeight_perp) / 2 ≈ 0.9`, but the base health sum already applied `shortWeight_spot ≈ 1.1` to the short-spot leg, so the correct `existingWeight` is `(1.1 + 0.9) / 2 = 1.0`.

`_getSpreadWeightX18` returns `≈ 0.98` for both spread orientations. With `existingWeight = 0.9`, `spreadWeight - existingWeight = +0.08`. Multiplied by a negative `basisAmount`, the adjustment is negative — a penalty of `0.08 × |basisAmount| × (spotPrice + perpPrice)` — instead of the intended bonus of `0.02 × |basisAmount| × (spotPrice + perpPrice)`.

### Impact Explanation
A trader holding a short-spot/long-perp spread has their INITIAL health understated by `0.10 × |basisAmount| × (spotPrice + perpPrice)`. Concretely:

1. **Withdrawal rejection**: `withdrawCollateral` checks `getHealth(sender, INITIAL) >= 0`. A solvent spread trader may fail this check and be unable to withdraw funds they are entitled to.
2. **Wrongful liquidation**: `liquidateSubaccount` (delegated to `clearinghouseLiq`) uses `getHealth` to determine whether a subaccount is liquidatable. A liquidator can liquidate a solvent short-spot/long-perp spread position, seizing the trader's assets at a discount.

Both paths are externally reachable production flows.

### Likelihood Explanation
Short-spot/long-perp (cash-and-carry short basis trade) is a standard hedging strategy. Any subaccount holding this position is affected every time `getHealth` is called. The error scales with position size, so large spread positions face proportionally larger health understatement. A liquidator can set up the scenario deterministically with no special privileges.

### Recommendation
Fix `getCoreRisk` to pass the actual `amount` (not `1`) to `_getWeightX18`, or add a `shortWeight` field to `CoreRisk` and populate it correctly. Then update `existingWeight` in `Clearinghouse.getHealth` to use the weight that matches the actual position direction for each leg:

```solidity
// For short-spot/long-perp, use shortWeight for spot and longWeight for perp
int128 existingWeight = (
    RiskHelper._getWeightX18(spotRisk, spotCoreRisk.amount, healthType) +
    RiskHelper._getWeightX18(perpRisk, perpCoreRisk.amount, healthType)
) / 2;
```

Similarly, `_getSpreadWeightX18` should derive its base weight from the short leg's actual short weight, not its long weight.

### Proof of Concept
```
Setup:
  spotLongWeightInitial  = 0.9e18
  spotShortWeightInitial = 1.1e18
  perpLongWeightInitial  = 0.9e18
  perpShortWeightInitial = 1.1e18
  spotPrice = perpPrice = 1000e18

Trader: short 1 BTC spot (amount = -1e18), long 1 BTC perp (amount = +1e18)

Base health (from engines):
  spot:  -1e18 * 1000e18 * 1.1  = -1100 (shortWeight applied)
  perp:  +1e18 * 1000e18 * 0.9  = +900  (longWeight applied)
  sum = -200

Spread adjustment (actual, buggy):
  basisAmount = -max(-1e18, -1e18) = -1e18
  existingWeight = (0.9 + 0.9) / 2 = 0.9
  spreadWeight = 1 - (1 - 0.9)/5 = 0.98
  delta = -1e18 * 2000e18 * (0.98 - 0.9) = -1e18 * 2000 * 0.08 = -160
  getHealth(INITIAL) = -200 + (-160) = -360  ← PENALTY applied

Correct spread adjustment:
  existingWeight = (1.1 + 0.9) / 2 = 1.0
  delta = -1e18 * 2000e18 * (0.98 - 1.0) = -1e18 * 2000 * (-0.02) = +40
  correct health = -200 + 40 = -160  ← BONUS applied

A liquidator observes getHealth = -360 < 0 and liquidates the position.
Correct health = -160 < 0 still negative here, but with sufficient quote balance
the trader would be solvent; the spread bonus is understated by 200 units,
directly enabling premature liquidation or withdrawal rejection.
```
