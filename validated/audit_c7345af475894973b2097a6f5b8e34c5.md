### Title
`BLS_G1_NEGATE_BASE_COST` and `BLS_G2_NEGATE_BASE_COST` Are Critically Undercharged Relative to Actual Execution Cost — (`File: src/bls_ops.rs`)

---

### Summary

`op_bls_g1_negate` (opcode 51) and `op_bls_g2_negate` (opcode 55) are assigned base cost constants of 916 and 1204 respectively, based on the assumption that the operation is a trivial bit-flip. However, in strict mode (the default), both operators perform full elliptic-curve point validation (`validate_g1` / `validate_g2`) before flipping the sign bit. The repository's own benchmark data shows these operations should cost approximately 850,000 and 2,400,000 cost units respectively — a factor of ~925× and ~2000× higher than what is actually charged. An attacker can craft CLVM programs that exhaust the block cost budget while consuming orders of magnitude more CPU time than the cost model predicts.

---

### Finding Description

**Root cause — mismatched constant vs. actual work:**

In `src/bls_ops.rs`, the negate cost constants are defined as:

```rust
// this is the same cost as XORing the top bit (minus the heap allocation of the
// return value, which the operator is adding back)
const BLS_G1_NEGATE_BASE_COST: Cost = 1396 - 480;   // = 916
``` [1](#0-0) 

```rust
// this is the same cost as XORing the top bit (minus the heap allocation of the
// return value, which the operator is adding back)
const BLS_G2_NEGATE_BASE_COST: Cost = 2164 - 960;   // = 1204
``` [2](#0-1) 

The developer's comment reveals the assumption: the cost was set to match a simple XOR of the top bit. But the actual implementation in strict mode (the default, when `RELAXED_BLS` is not set) calls `a.validate_g1` / `a.validate_g2` before the bit flip:

```rust
if strict {
    a.validate_g1(point, blob)?;
}
``` [3](#0-2) 

```rust
if strict {
    a.validate_g2(point, blob)?;
}
``` [4](#0-3) 

Point validation is a full curve-membership check — an expensive cryptographic operation. The repository's own benchmark data (three independent runs) consistently shows:

| Opcode | Benchmark-recommended cost | Actual constant | Ratio |
|---|---|---|---|
| `g1_negate` (51) | ~850,000 | 916 | **~928×** |
| `g2_negate` (55) | ~2,390,000 | 1,204 | **~1,985×** |

From `benchmarks.txt` (run 1):
```
opcode: g1_negate (51)
   time: base: 132316.16ns
   cost: base: 850233
opcode: g2_negate (55)
   time: base: 369885.46ns
   cost: base: 2376799
``` [5](#0-4) 

The benchmark tool derives recommended cost by multiplying measured nanoseconds by the cost scale (6.425771579565472 cost units/ns). All three runs agree within ~3%. [6](#0-5) 

**Operator registration — both opcodes are unconditionally reachable:**

Opcode 51 (`g1_negate`) and opcode 55 (`g2_negate`) are registered in `ChiaDialect::op` with no flag guard:

```rust
51 => op_bls_g1_negate,
...
55 => op_bls_g2_negate,
``` [7](#0-6) 

They are also listed in `f_table.rs` and are available in both consensus mode and mempool mode. [8](#0-7)

### Impact Explanation

The Chia blockchain enforces a maximum cost budget of 11,000,000,000 cost units per block (as seen in `tests/run.py` and `tools/src/bin/generate-stress-tests.rs`). [9](#0-8) 

With `BLS_G1_NEGATE_BASE_COST = 916`, an attacker can fit approximately:

```
11,000,000,000 / 916 ≈ 12,000,000 g1_negate calls per block
```

Each call takes ~132,000 ns of real CPU time, so the total wall-clock time to validate such a block is:

```
12,000,000 × 132,000 ns ≈ 1,584 seconds (~26 minutes)
```

For `g2_negate` (cost 1204, time ~370,000 ns):

```
11,000,000,000 / 1204 ≈ 9,100,000 calls × 370,000 ns ≈ 3,367 seconds (~56 minutes)
```

A legitimately-priced block should complete in seconds. This constitutes a **consensus-layer resource exhaustion** attack: a single crafted coin spend, accepted by the mempool as within budget, forces every full node to spend tens of minutes validating it, stalling block propagation and potentially causing chain forks or node timeouts. The impact is equivalent to a targeted denial-of-service against all Chia full nodes simultaneously.

---

### Likelihood Explanation

Likelihood is **high**. The attack requires no privileged access — any participant can submit a coin spend to the Chia mempool. The crafted program is syntactically valid CLVM, passes cost-budget checks (it is within the 11B limit), and uses a standard, unconditionally-enabled opcode. The attacker needs only a valid G1 or G2 point (publicly known) and the ability to construct a CLVM program that loops the negate call. No cryptographic secrets are required.

---

### Recommendation

1. **Immediate**: Update `BLS_G1_NEGATE_BASE_COST` and `BLS_G2_NEGATE_BASE_COST` to values consistent with the benchmark data — approximately 850,000 and 2,400,000 respectively. The comment claiming these are "the same cost as XORing the top bit" is incorrect for strict mode. [10](#0-9) 

2. **Long-term**: Establish a policy that every operator cost constant must be validated against the benchmark tool output before merging. The `benchmarks.txt` file already contains the correct methodology; the constants simply were not updated to match it.

---

### Proof of Concept

A CLVM program that exhausts the 11B cost budget using only `g1_negate` calls (opcode 51):

```
; Pseudocode: repeat g1_negate ~12,000,000 times
; Each call costs 916 cost units but takes ~132,000 ns
; Total cost: ~11,000,000,000 (within budget)
; Total wall time: ~1,584 seconds

(g1_negate (g1_negate (g1_negate ... (q . <valid_G1_point>) ...)))
```

Concretely, serialize a deeply-nested CLVM tree of `(51 . (51 . (51 . ... (1 . <48-byte G1 atom>) ...)))` using the backrefs serializer to keep the byte size small (avoiding the `cost_per_byte` pre-deduction). Submit as a coin spend to any Chia full node. The node's `run_program` call will accept it (cost check passes at 916 per call) but will spend ~26 minutes executing it, blocking all subsequent block validation. [11](#0-10) [12](#0-11)

```rust
pub fn op_bls_g1_negate(
    a: &mut Allocator,
    input: NodePtr,
    _max_cost: Cost,
    flags: ClvmFlags,
) -> Response {
    let strict = !flags.contains(ClvmFlags::RELAXED_BLS);
    let [point] = get_args::<1>(a, input, "g1_negate")?;

    let mut blob: [u8; 48] = atom(a, point, "G1 atom").and_then(|blob| {
        blob.as_ref().try_into().map_err(|_| {
            EvalErr::InvalidOpArg(point, "atom is not a G1 size, 48 bytes".to_string())
        })
    })?;
    if strict {
        a.validate_g1(point, blob)?;
    }

    if (blob[0] & 0xe0) == 0xc0 {
        // This is compressed infinity. negating it is a no-op
        // we can just pass through the same atom as we received. We'll charge
        // the allocation cost anyway, for consistency
        Ok(Reduction(
            BLS_G1_NEGATE_BASE_COST + 48 * MALLOC_COST_PER_BYTE,
            point,
        ))
    } else {
        blob[0] ^= 0x20;
        if strict {
            a.add_validated_g1(blob);
        }
        new_atom_and_cost(a, BLS_G1_NEGATE_BASE_COST, &blob)
    }
}
