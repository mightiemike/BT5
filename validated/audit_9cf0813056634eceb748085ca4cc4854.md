### Title
Loss of Accrued Bonus Rewards on Governance Tier Parameter Update — (File: x/tieredrewards/keeper/tier.go)

### Summary
When governance updates a tier's `BonusApy` via `SetTier()`, pending bonus rewards for all existing positions in that tier are not settled first. Any subsequent claim computes the entire unclaimed period at the new rate, causing position holders to lose bonus rewards accrued under the old rate.

### Finding Description
`SetTier()` in `x/tieredrewards/keeper/tier.go` is the single write path for tier parameters used by governance messages, genesis, and chain upgrades. It directly persists the updated `Tier` struct — including `BonusApy` — to the `Tiers` collection with no prior reward settlement step:

```go
// SetTier writes a tier after validation. Used by governance messages, genesis, and chain upgrades.
func (k Keeper) SetTier(ctx context.Context, tier types.Tier) error {
    if err := tier.Validate(); err != nil {
        return err
    }
    if err := k.Tiers.Set(ctx, tier.Id, tier); err != nil { // ← overwrites BonusApy immediately
        return errorsmod.Wrapf(err, "%s (tier id %d)", types.ErrTierStore.Error(), tier.Id)
    }
    return nil
}
``` [1](#0-0) 

Bonus reward computation in `computeSegmentBonus()` uses `tier.BonusApy` as passed by the caller at claim time:

```go
return tokens.
    Mul(tier.BonusApy).
    MulInt64(durationSeconds).
    QuoInt64(types.SecondsPerYear).
    TruncateInt()
``` [2](#0-1) 

There is no snapshot of `BonusApy` stored per position or per validator-event segment. The `tier` argument passed to `computeSegmentBonus` is fetched from the live `Tiers` store at claim time. When governance reduces `BonusApy`, the new value is immediately visible to all future claim computations, retroactively applying the lower rate to the entire unclaimed window — including the period that elapsed under the old, higher rate.

The `BeginBlocker` only handles base-reward top-ups and performs no bonus accrual or checkpoint flush for existing positions: [3](#0-2) 

Reward settlement (`claimRewards`) is only triggered lazily on user-initiated operations (`TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, `ExitTierWithDelegation`, `ClaimTierRewards`): [4](#0-3) 

Any position that has not been touched between the last claim and the governance update will have its entire unclaimed window repriced at the new rate.

### Impact Explanation
A governance proposal that reduces `BonusApy` for a tier causes every position holder in that tier to lose the difference between the old and new rate for the entire period since their last claim. The loss per position is:

```
loss = principal × (old_BonusApy − new_BonusApy) × elapsed_seconds / SecondsPerYear
```

This is a direct, quantifiable loss of bonus rewards that position holders had legitimately earned and that the protocol owes them from the `RewardsPoolName` module account. The funds are not redistributed — they simply remain in the pool, unearned by anyone.

### Likelihood Explanation
Governance proposals to adjust tier parameters (e.g., reducing `BonusApy` to manage pool sustainability) are an expected, routine chain operation. The comment on `SetTier` explicitly documents governance as a caller. Any such proposal, even one made in good faith, silently retroactively reprices all unclaimed bonus rewards for every active position in the affected tier. No malicious intent is required; the design flaw is structural.

### Recommendation
Before overwriting a tier's parameters in `SetTier()`, iterate over all positions belonging to that tier and settle their pending bonus rewards using the **old** tier parameters. Only after all positions have been checkpointed should the new tier be written to the store. This mirrors the pattern already used in user-initiated operations where `claimRewards` is called before any state-changing operation on a position. [1](#0-0) 

### Proof of Concept
1. Alice creates a position in Tier 1 (`BonusApy = 0.20`) at block time T0 with 10,000 tokens staked.
2. Six months pass (≈15,768,000 seconds). Alice does not claim. Her accrued bonus ≈ `10,000 × 0.20 × 15,768,000 / 31,536,000` = **1,000 tokens**.
3. Governance passes `MsgUpdateTier` for Tier 1, setting `BonusApy = 0.05`. `SetTier()` overwrites the tier immediately with no settlement.
4. Alice claims rewards the next block. `claimRewards` fetches the current tier (`BonusApy = 0.05`) and computes bonus for the full window [T0, T2] at 5%.
5. Alice receives ≈ `10,000 × 0.05 × 15,768,000 / 31,536,000` = **250 tokens** instead of 1,000 tokens.
6. Alice permanently loses **750 tokens** of bonus rewards she had earned under the old rate. The 750 tokens remain stranded in the `RewardsPoolName` account, unclaimable by anyone.
