### Title
No Liquidation Price Protection for Liquidators Allows Adverse Quote Drain — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

The `LiquidateSubaccount` struct contains no price-bound field. The liquidation payment charged to the liquidator's subaccount is computed entirely from the oracle price at execution time. Because the sequencer can update the oracle price in the same batch before executing the liquidation, the liquidator has no on-chain mechanism to cap the quote they spend, mirroring the missing slippage check in the reference report.

---

### Finding Description

The `LiquidateSubaccount` struct is defined as:

```solidity
struct LiquidateSubaccount {
    bytes32 sender;
    bytes32 liquidatee;
    uint32 productId;
    bool isEncodedSpread;
    int128 amount;
    uint64 nonce;
}
``` [1](#0-0) 

There is no `maxQuoteToSpend`, `minPriceX18`, or any equivalent price-protection field. Compare this to the `Order` struct, which carries `priceX18` as a limit price to protect traders from adverse execution. [2](#0-1) 

In `_handleLiquidationPayment`, the actual cost charged to the liquidator is computed at execution time from the live oracle price:

```solidity
(v.liquidationPriceX18, v.oraclePriceX18) = getLiqPriceX18(txn.productId, txn.amount);
v.liquidationPayment = v.liquidationPriceX18.mul(txn.amount);
v.liquidationFees = (v.oraclePriceX18 - v.liquidationPriceX18)
    .mul(LIQUIDATION_FEE_FRACTION)
    .mul(txn.amount);
``` [3](#0-2) 

`getLiqPriceX18` reads `risk.priceX18` — the current oracle price stored in the engine — and applies a penalty multiplier:

```solidity
return (risk.priceX18.mul(ONE + penaltyX18), risk.priceX18);
``` [4](#0-3) 

The oracle price is updated by the sequencer via `UpdatePrice` transactions processed in the same batch as the liquidation:

```solidity
} else if (txType == IEndpoint.TransactionType.UpdatePrice) {
    (uint32 productId, int128 newPriceX18) = clearinghouse.updatePrice(transaction);
    if (productId != 0) {
        priceX18[productId] = newPriceX18;
    }
}
``` [5](#0-4) 

The liquidator's quote balance is then debited without any check against a liquidator-specified bound:

```solidity
spotEngine.updateBalance(
    QUOTE_PRODUCT_ID,
    txn.sender,
    -v.liquidationPayment - v.liquidationFees
);
``` [6](#0-5) 

The only post-execution check on the liquidator is a health check (`ERR_SUBACCT_HEALTH`), which prevents the liquidator from going below initial health, but does **not** bound the price they pay. [7](#0-6) 

---

### Impact Explanation

A liquidator signs a `LiquidateSubaccount` transaction when the oracle price is `P1`. The sequencer includes an `UpdatePrice` transaction (setting price to `P2 > P1`) earlier in the same batch. When `_handleLiquidationPayment` executes, the liquidator pays `P2 × (1 + penalty) × amount + fees` instead of the `P1`-based cost they anticipated. The difference is a direct, unrecoverable loss from the liquidator's subaccount quote balance. For large liquidations on volatile assets, this loss can be material.

---

### Likelihood Explanation

The sequencer routinely includes `UpdatePrice` transactions in every batch to keep oracle prices current. This is not a malicious act — it is normal protocol operation. Any time a price update and a liquidation land in the same batch (which is the common case), the liquidator is exposed to the full price delta with no recourse. The liquidator cannot cancel or bound their signed transaction after submission.

---

### Recommendation

Add a `maxQuoteToSpend` (or `minPriceX18`) field to `LiquidateSubaccount`:

```solidity
struct LiquidateSubaccount {
    bytes32 sender;
    bytes32 liquidatee;
    uint32 productId;
    bool isEncodedSpread;
    int128 amount;
    uint64 nonce;
    int128 maxQuoteToSpend; // NEW: liquidator-specified cap
}
```

In `_handleLiquidationPayment`, enforce it before debiting the liquidator:

```solidity
require(
    txn.maxQuoteToSpend == 0 ||
    v.liquidationPayment + v.liquidationFees <= txn.maxQuoteToSpend,
    ERR_LIQUIDATION_SLIPPAGE
);
```

This mirrors the `priceX18` limit already present in `Order` and gives liquidators the same price-protection guarantee that traders have.

---

### Proof of Concept

1. Oracle price for product `P` is `1000 USDC`. Liquidator signs `LiquidateSubaccount { amount: 10, ... }` expecting to pay `~9800 USDC` (2% discount).
2. Sequencer builds a batch: `[UpdatePrice(P, 1100), LiquidateSubaccount(...)]`.
3. `getLiqPriceX18` returns `1100 × 0.98 = 1078` as `liquidationPriceX18`.
4. Liquidator is charged `1078 × 10 + fees = 10780+ USDC` instead of `9800 USDC`.
5. The liquidator loses ~980 USDC with no on-chain protection available. [8](#0-7)

        internal
        returns (int128, int128)
    {
        RiskHelper.Risk memory risk = IProductEngine(productToEngine[productId])
            .getRisk(productId);

            risk,

            IProductEngine.HealthType.MAINTENANCE

        if (penaltyX18.abs() < MIN_NON_SPREAD_LIQ_PENALTY_X18) {

                penaltyX18 = MIN_NON_SPREAD_LIQ_PENALTY_X18;
            }
        }
