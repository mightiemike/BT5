### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable N× Bonus Drain from `RewardsPoolName` - (`x/tieredrewards/keeper/claim_rewards.go`, `msg_server.go`)

---

### Summary

`ClaimTierRewards` loads each position ID from `msg.PositionIds` independently from the store with no deduplication check. If the same ID appears N times, N identical in-memory copies are built, each carrying the original `LastBonusAccrual` and `LastEventSeq` checkpoints. `claimRewardsAndUpdatesPositions` then calls `processEventsAndClaimBonus` on every copy, computing and paying the same bonus segment N times. The final `setPosition` call correctly advances the on-chain checkpoint, but by then `SendCoinsFromModuleToAccount` has already transferred N× the legitimate bonus out of `RewardsPoolName`.

---

### Finding Description

**Entry point — `ClaimTierRewards` (`msg_server.go` lines 429–468):**

The handler iterates `msg.PositionIds` and calls `getPositionState` for every element, appending each result to a `positions` slice. There is no uniqueness check. [1](#0-0) 

If `msg.PositionIds = [P, P, P]`, the slice becomes `[posP_copy0, posP_copy1, posP_copy2]`, all loaded from the store before any write, so all three carry the same `LastBonusAccrual = T0` and `LastEventSeq = X`.

**Processing loop — `claimRewardsAndUpdatesPositions` (`claim_rewards.go` lines 106–135):**

Each iteration takes a pointer to `positions[i]`, calls `processEventsAndClaimBonus`, then calls `setPosition`. [2](#0-1) 

Because `positions[i]` and `positions[j]` are independent slice elements (not the same pointer), updating `positions[0]` does not affect `positions[1]` or `positions[2]`. Each copy still holds the original `T0` / `X` when its turn comes.

**Bonus computation and payment — `processEventsAndClaimBonus` (`claim_rewards.go` lines 142–251):**

The function reads `pos.LastBonusAccrual` (= `T0`) and `pos.LastEventSeq` (= `X`) from the in-memory copy, computes the bonus for the segment `[T0, blockTime]`, checks the pool balance, and immediately sends coins. [3](#0-2) [4](#0-3) 

The pool-balance guard (`sufficientBonusPoolBalance`) is evaluated per-iteration against the live pool balance. As long as the pool holds ≥ N× the single-claim bonus, every iteration passes and transfers coins.

**Checkpoint write — `setPosition`:**

`setPosition` is called after each `processEventsAndClaimBonus`. The first call persists the advanced checkpoint. The second and third calls overwrite the store with an identically-advanced checkpoint — the store ends up correct, but the coins have already been sent N times.

**Validation gap — `validateClaimRewards` (`msg_validate.go` lines 175–181):**

The only per-position guard is an ownership check. There is no duplicate-ID rejection. [5](#0-4) 

**Secondary side-effect — `decrementEventRefCount`:**

Inside `processEventsAndClaimBonus`, `decrementEventRefCount` is called for every event entry on each iteration. With N duplicate copies, each event's reference count is decremented N times, potentially underflowing or prematurely pruning shared validator event records. [6](#0-5) 

---

### Impact Explanation

An attacker who owns a position with accrued bonus rewards can drain up to N× the legitimate bonus per transaction from `RewardsPoolName` by submitting `MsgClaimTierRewards` with `PositionIds = [P, P, …, P]`. The only practical ceiling is the pool balance and the per-transaction gas limit. This is a direct, quantifiable economic loss from a module account funded by protocol inflation.

---

### Likelihood Explanation

The exploit requires only a standard signed transaction with a repeated position ID — no privileged role, no governance action, no special chain state beyond a funded rewards pool and a position with accrued bonus. It is trivially constructable by any position owner.

---

### Recommendation

Deduplicate `msg.PositionIds` before processing, either in `MsgClaimTierRewards.Validate()` (reject if any ID appears more than once) or at the start of `ClaimTierRewards` before the load loop. A set-based check is sufficient:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, id := range msg.PositionIds {
    if _, dup := seen[id]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest,
            "duplicate position id %d", id)
    }
    seen[id] = struct{}{}
}
```

This guard belongs in `MsgClaimTierRewards.Validate()` so it is enforced at the message-validation layer before any keeper logic runs.

---

### Proof of Concept

1. Create position `P` with 30 days of accrued bonus (e.g., 1000 CRO expected).
2. Ensure `RewardsPoolName` holds ≥ 3000 CRO.
3. Submit `MsgClaimTierRewards{Owner: attacker, PositionIds: [P, P, P]}`.
4. Observe attacker balance increases by 3000 CRO; `RewardsPoolName` decreases by 3000 CRO.
5. Query position `P` — `LastBonusAccrual` is correctly advanced to `blockTime` (store is consistent, but coins are gone).
6. Assert invariant violation: bonus for segment `[T0, blockTime]` was paid 3× instead of 1×.
