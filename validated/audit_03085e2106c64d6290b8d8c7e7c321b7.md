### Title
Divide-by-Zero in `socializeSubaccount` When `openInterest == 0` Permanently Blocks Finalization — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when socializing a residual negative `vQuoteBalance`. There is no zero-guard. Because the finalization path **requires** `balance.amount == 0` for every perp position before calling `socializeSubaccount`, a product whose only remaining open interest belonged to the liquidatee will have `openInterest == 0` at the moment of the call. If insurance is insufficient to cover the full negative `vQuoteBalance`, the division reverts with `"DBZ"`, permanently blocking finalization.

---

### Finding Description

`MathSD21x18.div` explicitly reverts when the divisor is zero: [1](#0-0) 

Inside `socializeSubaccount`, after insurance is applied, any remaining negative `vQuoteBalance` is socialized by dividing by `state.openInterest` with no zero-check: [2](#0-1) 

The `state` used here comes from `getStateAndBalance`, which calls `_updateBalance(state, balance, 0, 0)`. Because the finalization path has already enforced `balance.amount == 0`, the pre-update subtracts `0.abs() == 0` and the post-update adds `0` back — so the returned `state.openInterest` equals the stored `states[productId].openInterest` exactly: [3](#0-2) 

The finalization path in `ClearinghouseLiq` **mandates** `balance.amount == 0` for every perp product before `socializeSubaccount` is ever called: [4](#0-3) 

`socializeSubaccount` is then called unconditionally: [5](#0-4) 

The developers already recognized that `openInterest` can be zero — `updateStates` skips products with zero open interest — but no equivalent guard exists in `socializeSubaccount`: [6](#0-5) 

---

### Impact Explanation

When the revert fires, the entire `liquidateSubaccountImpl` transaction reverts. Because the finalization is the only mechanism to clear the liquidatee's account, the subaccount is permanently stuck: it cannot be finalized, its remaining collateral cannot be recovered, and the protocol's insurance accounting for that subaccount is frozen. This is a direct, permanent loss of user funds and a solvency accounting break.

---

### Likelihood Explanation

The condition is reachable in normal operation:

1. A perp market has a single active participant (the liquidatee), or all other participants have closed their positions, driving `states[productId].openInterest` to `0`.
2. The liquidatee's position is closed (amount → 0) but `vQuoteBalance` remains negative (realized loss).
3. The protocol's insurance fund is depleted or insufficient to cover the full deficit.
4. Finalization is attempted — the division reverts.

This is not a contrived edge case. Thin markets with few participants are common, especially for newly listed or low-volume perp products.

---

### Recommendation

Add a zero-check before the division in `socializeSubaccount`. If `openInterest == 0` and `vQuoteBalance` is still negative after insurance, the loss cannot be socialized across other participants (there are none). The correct behavior is to absorb the loss into the protocol's accounting (e.g., leave `vQuoteBalance` at zero and record the unrecoverable deficit) or revert with a meaningful error only if the protocol explicitly cannot absorb it. The minimal fix:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No participants to socialize against; absorb or handle separately
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

### Proof of Concept

```
1. Deploy PerpEngine with one perp product (productId = 2).
2. Alice opens a long position: amount = 100e18, vQuoteBalance = -10000e18 (bought at a loss).
3. Alice's position is closed: amount = 0, vQuoteBalance = -500e18 (residual loss).
   → states[2].openInterest == 0 (Alice was the only participant).
4. Alice's maintenance health < 0; liquidator calls liquidateSubaccount with productId = type(uint32).max.
5. ClearinghouseLiq.finalizeSubaccount:
   a. Checks balance.amount == 0 for all perps → passes.
   b. Settles positive PnL → none.
   c. Settles negative PnL against quote → quote balance is 0.
   d. insurance = 0 (fund depleted).
   e. Calls perpEngine.socializeSubaccount(alice, 0).
6. Inside socializeSubaccount:
   - balance.vQuoteBalance = -500e18 < 0 → enters outer if.
   - insuranceCover = min(0, 500e18) = 0 → balance.vQuoteBalance still -500e18.
   - Enters inner if: fundingPerShare = 500e18.div(0) → REVERT "DBZ".
7. Entire finalization reverts. Alice's account is permanently unliquidatable.
``` [7](#0-6)

        external

            (State memory state, Balance memory balance) = getStateAndBalance(
                productId,
                subaccount
            );
            if (balance.vQuoteBalance < 0) {
                int128 insuranceCover = MathHelper.min(
                    insurance,
                    -balance.vQuoteBalance

                insurance -= insuranceCover;

                    balance.vQuoteBalance = 0;

                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
        }
        return insurance;
    }
