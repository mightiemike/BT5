### Title
`MAX_FUTURE_BLOCK_TIME_MTP` Set 24× Too Low, Causing Valid Zcash Blocks to Be Rejected — (`btc-types/src/network.rs`)

### Summary
The constant `MAX_FUTURE_BLOCK_TIME_MTP` is set to `90 * 60 = 5400 seconds` (90 minutes), but the in-code comment at the only call-site explicitly states the correct Zcash value is `129600 seconds` (36 hours). The constant is 24× too small. Any valid Zcash block whose timestamp falls between 90 minutes and 36 hours ahead of the previous block's median-time-past (MTP) is unconditionally rejected by `check_pow`, stalling the Zcash light-client and making SPV proof verification impossible for that chain segment.

---

### Finding Description

**Root cause — misconfigured constant:**

`btc-types/src/network.rs` defines two future-timestamp limits:

```rust
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60;   // 5 400 s  ← wrong
pub const MAX_FUTURE_BLOCK_TIME_LOCAL: u32 = 2 * 60 * 60; // 7 200 s  ← correct for BTC/LTC
``` [1](#0-0) 

`MAX_FUTURE_BLOCK_TIME_MTP` is consumed exclusively in the Zcash `check_pow` path:

```rust
// MAX_FUTURE_BLOCK_TIME_MTP is typically 129600 seconds (36 hours) in Zcash
require!(
    block_header.time
        <= next_work_result.prev_block_median_time_past + MAX_FUTURE_BLOCK_TIME_MTP,
    "time-too-far-ahead-of-mtp: block timestamp is too far ahead of median-time-past"
);
``` [2](#0-1) 

The comment is the developer's own acknowledgement of the correct value: **129 600 s = 36 hours**. The deployed constant is **5 400 s = 90 minutes** — a 24× shortfall, directly analogous to the 135× shortfall in M-12's `defaultNukeFactorIncrease`.

**How the check is reached:**

Any unprivileged NEAR caller (or the production relayer) invokes `submit_blocks`, which calls `submit_block_header` → `check_target` → `check_pow` for every Zcash header when `skip_pow_verification = false` (the recommended production setting). [3](#0-2) 

The MTP-future check is unconditional inside `check_pow`; there is no bypass path for a legitimate relayer. [4](#0-3) 

---

### Impact Explanation

Any Zcash block whose `time` field satisfies:

```
prev_MTP + 5400 < block.time ≤ prev_MTP + 129600
```

is a **valid block on the Zcash network** but is **rejected by this contract** with `"time-too-far-ahead-of-mtp"`. Because `submit_block_header` panics on rejection, the relayer cannot advance the mainchain tip past such a block. The contract's `mainchain_tip_blockhash` freezes, `get_last_block_height` returns a stale value, and every subsequent call to `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` that references a block beyond the stall point fails. Cross-chain SPV verification for Zcash becomes permanently unavailable until a contract upgrade is deployed. [5](#0-4) 

---

### Likelihood Explanation

Zcash targets a 75-second block interval (post-Blossom). The MTP is the median of the 11 most-recent block timestamps, so under normal conditions it lags actual time by roughly 4–5 blocks × 75 s ≈ 5–6 minutes. A block timestamped at wall-clock time is therefore only ~6 minutes ahead of MTP — well within 90 minutes.

However, the condition is reachable in practice:

1. **Hash-rate drops / network partitions.** If the Zcash network experiences a prolonged mining gap (e.g., 2–3 hours), the MTP freezes while wall-clock time advances. A miner resuming after a 2-hour gap will produce a block timestamped ~2 hours ahead of MTP — exceeding the 90-minute limit and triggering the bug.
2. **Testnet / low-difficulty environments.** Zcash testnet explicitly allows min-difficulty blocks after height 299 187; sparse mining is common, making large MTP–timestamp gaps routine.
3. **Relayer replay of historical segments.** A relayer initialising from a checkpoint that spans a historical mining gap will hit the same rejection.

---

### Recommendation

Change the constant to match the Zcash protocol specification cited in the call-site comment:

```rust
// btc-types/src/network.rs
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 129_600; // 36 hours, per Zcash spec
```

If the constant is also intended for non-Zcash chains (it currently is not used by Bitcoin or Litecoin paths), introduce a chain-specific value rather than a shared constant to avoid future cross-chain confusion.

---

### Proof of Concept

**Arithmetic demonstration** (no code change needed):

```
Zcash post-Blossom target spacing: 75 s
MTP = median of last 11 timestamps ≈ actual_time − 5 × 75 s = actual_time − 375 s

Scenario: 2-hour mining gap
  actual_time advances by 7200 s while MTP stays fixed.
  New block timestamp ≈ actual_time  →  block.time − MTP ≈ 7200 s

Contract check:
  7200 > MAX_FUTURE_BLOCK_TIME_MTP (5400)  →  PANIC, block rejected.

Zcash reference check:
  7200 ≤ 129600  →  PASS, block accepted.
```

A relayer submitting this otherwise-valid Zcash block to `submit_blocks` will receive a contract panic, the mainchain tip will not advance, and all downstream `verify_transaction_inclusion` calls for blocks at or above that height will fail.

        require!(
            next_work_result.expected_bits == block_header.bits,
            "bad-diffbits: incorrect proof of work"
        );

        // Check timestamp against prev
        require!(
            block_header.time > next_work_result.prev_block_median_time_past,
            "time-too-old: block time is before the median time of the previous block"
        );

        // Check future timestamp soft fork rule introduced in v2.1.1-1.
        // This retrospectively activates at block height 2 for mainnet and regtest,
        // and 6 blocks after Blossom activation for testnet.
        //
        // MAX_FUTURE_BLOCK_TIME_MTP is typically 129600 seconds (36 hours) in Zcash
        require!(
            block_header.time
                <= next_work_result.prev_block_median_time_past + MAX_FUTURE_BLOCK_TIME_MTP,
            "time-too-far-ahead-of-mtp: block timestamp is too far ahead of median-time-past"
        );

        // Check timestamp
        let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap(); // Convert to seconds
        require!(
            block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
            "time-too-new: block timestamp is too far ahead of local time"
        );

        // Check Equihash solution

        let input = block_header.get_block_header_vec_for_equihash();
