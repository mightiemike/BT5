### Title
`ReadCacheLookup.parent_lookup` entries never pruned when `count` reaches zero, causing O(N²) work in `find_paths()` for crafted CLVM trees — (`src/serde/read_cache_lookup.rs`)

---

### Summary

`ReadCacheLookup` maintains two maps — `count: HashMap<Bytes32, u32>` and `parent_lookup: HashMap<Bytes32, Vec<(Bytes32, bool)>>` — to track which tree-hash nodes are live on the deserialization stack. When `pop()` is called, the reference count for a hash is decremented but the entry is **never removed** from either map. `parent_lookup` in particular accumulates stale parent-edge vectors that are never pruned. `find_paths()` iterates over every entry in a hash's `parent_lookup` vector on every BFS step, including stale entries whose `count` is zero. For a crafted CLVM tree with many repeated identical subtrees, this produces O(N²) work inside a single `node_to_bytes_backrefs()` call.

---

### Finding Description

`ReadCacheLookup` is used by `node_to_stream_backrefs()` to find back-reference paths during compressed CLVM serialization.

**`push()` always appends to `parent_lookup`:**

```rust
// src/serde/read_cache_lookup.rs  lines 65-88
pub fn push(&mut self, id: Bytes32) {
    ...
    *self.count.entry(id).or_insert(0) += 1;
    *self.count.entry(new_root_hash).or_insert(0) += 1;

    self.parent_lookup.entry(id).or_default().push(new_parent_to_old_root);
    self.parent_lookup.entry(self.root_hash).or_default().push(new_parent_to_id);
    ...
}
```

**`pop()` decrements `count` but removes nothing from either map:**

```rust
// src/serde/read_cache_lookup.rs  lines 94-100
fn pop(&mut self) -> (Bytes32, Bytes32) {
    let item = self.read_stack.pop().expect("stack empty");
    *self.count.entry(item.0).or_insert(0) -= 1;
    *self.count.entry(self.root_hash).or_insert(0) -= 1;
    self.root_hash = item.1;
    item
}
```

The existing test at line 400 explicitly confirms that after `pop2_and_cons()`, a hash whose count has dropped to zero **remains in the map**:

```rust
// src/serde/read_cache_lookup.rs  line 400
assert_eq!(rcl.count.get(&hash_of_cons_9_cons_5_nil).unwrap(), &0);
```

**`find_paths()` iterates over every stale entry:**

```rust
// src/serde/read_cache_lookup.rs  lines 171-186
let parents = self.parent_lookup.get(node);
if let Some(items) = parents {
    for (parent, direction) in items.iter() {          // iterates ALL entries
        if *(self.count.get(parent).unwrap_or(&0)) > 0 // stale ones skipped but not avoided
            && !seen_ids.contains(parent)
        { ... }
        seen_ids.insert(parent);
    }
}
```

Every `push()` call appends at least two new edges to `parent_lookup`. For a CLVM tree containing N identical atoms (same tree hash), the `parent_lookup` vector for that hash grows to O(N) entries. `find_paths()` is called once per node in `node_to_stream_backrefs()`, so the total work becomes **O(N²)**.

---

### Impact Explanation

An attacker who supplies crafted CLVM bytes — e.g., a flat list of N identical atoms — causes `node_to_bytes_backrefs()` to perform O(N²) iterations inside `find_paths()`. Because `node_to_bytes_backrefs()` / `node_to_bytes_backrefs_limit()` are exposed as public API and called on attacker-controlled serialized programs, this is a reachable quadratic-cost amplification. The `parent_lookup` and `count` maps grow monotonically within the call and are never compacted, directly mirroring the unbounded-storage pattern in the reference report.

---

### Likelihood Explanation

Any caller that serializes attacker-supplied CLVM programs with back-reference compression is exposed. The Chia wallet and full-node both serialize spend bundles and puzzle reveals. A crafted program consisting of a long list of identical atoms is trivially constructable and requires no special privileges.

---

### Recommendation

In `pop()`, after decrementing a count, remove the entry from both `count` and `parent_lookup` when the count reaches zero:

```rust
fn pop(&mut self) -> (Bytes32, Bytes32) {
    let item = self.read_stack.pop().expect("stack empty");
    // decrement and prune item hash
    if let Entry::Occupied(mut e) = self.count.entry(item.0) {
        *e.get_mut() -= 1;
        if *e.get() == 0 {
            e.remove();
            self.parent_lookup.remove(&item.0);
        }
    }
    // decrement and prune current root hash
    if let Entry::Occupied(mut e) = self.count.entry(self.root_hash) {
        *e.get_mut() -= 1;
        if *e.get() == 0 {
            e.remove();
            self.parent_lookup.remove(&self.root_hash);
        }
    }
    self.root_hash = item.1;
    item
}
```

Apply the same pruning in `pop2_and_cons()` after the two `pop()` calls.

### Proof of Concept

```rust
use clvmr::serde::node_to_bytes_backrefs;
use clvmr::Allocator;

fn main() {
    let mut a = Allocator::new();
    // Build a flat list of N identical atoms: (atom . (atom . (atom . ... nil)))
    // All atoms share the same tree hash, so parent_lookup for that hash
    // accumulates O(N) entries, and find_paths() does O(N^2) work.
    let atom = a.new_atom(&[0xde, 0xad, 0xbe, 0xef]).unwrap();
    let mut node = a.null();
    for _ in 0..10_000 {
        node = a.new_pair(atom, node).unwrap();
    }
    // This call performs O(N^2) iterations inside find_paths()
    let _ = node_to_bytes_backrefs(&a, node).unwrap();
}
