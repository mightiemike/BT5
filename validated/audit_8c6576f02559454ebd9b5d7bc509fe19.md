### Title
Kickout-Exemption Sorting Uses Raw Endorsement Ratio While Reward Calculation Uses Binarized Ratio, Causing Wrong Validators to Be Protected from Kickout — (`chain/epoch-manager/src/validator_stats.rs`)

---

### Summary

`get_sortable_validator_online_ratio` always passes `endorsement_cutoff_threshold = None` to `get_validator_online_ratio`, while `finalize_epoch` passes `Some(chunk_validator_only_kickout_threshold)` to `calculate_reward`. The two code paths that operate on the same validator stats use divergent formulas: the kickout-exemption sort uses the raw endorsement ratio, while the reward calculation binarizes it. This causes the wrong validators to be shielded from kickout at epoch boundaries.

---

### Finding Description

In `chain/epoch-manager/src/validator_stats.rs`, `get_sortable_validator_online_ratio` unconditionally calls `get_validator_online_ratio(stats, None)`:

```rust
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);   // ← always None
    ...
}
``` [1](#0-0) 

This function is called inside `compute_validators_to_reward_and_kickout` to sort all validators by online ratio, which then drives `compute_exempted_kickout` — the mechanism that decides which validators are shielded from kickout when `validator_max_kickout_stake_perc` would otherwise be exceeded:

```rust
let mut sorted_validators = validator_block_chunk_stats
    .iter()
    .map(|(account, stats)| (get_sortable_validator_online_ratio(stats), account))
    .collect_vec();
sorted_validators.sort_by(validator_comparator);
let accounts_sorted_by_online_ratio = ...;
let exempted_validators = Self::compute_exempted_kickout(
    epoch_info, &accounts_sorted_by_online_ratio, ...
);
``` [2](#0-1) 

However, in `finalize_epoch`, the reward calculation uses a **binarized** endorsement ratio via `endorsement_cutoff_threshold: Some(chunk_validator_only_kickout_threshold)`:

```rust
let online_thresholds = ValidatorOnlineThresholds {
    online_min_threshold: epoch_config.online_min_threshold,
    online_max_threshold: epoch_config.online_max_threshold,
    endorsement_cutoff_threshold: Some(
        epoch_config.chunk_validator_only_kickout_threshold,
    ),
};
self.reward_calculator.calculate_reward(
    validator_block_chunk_stats, ..., online_thresholds, ...
)
``` [3](#0-2) 

The binarization logic in `get_endorsement_ratio` maps any endorsement ratio below the threshold to `(0, 1)` and any ratio at or above it to `(1, 1)`:

```rust
} else if let Some(threshold) = cutoff_threshold {
    if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
} else {
    (stats.produced, stats.expected)
}
``` [4](#0-3) 

The two code paths therefore compute fundamentally different online ratios for the same validator stats:

| Code path | Endorsement treatment | Formula used |
|---|---|---|
| Kickout-exemption sort | Raw ratio (e.g. 49/100 = 0.49) | `get_validator_online_ratio(stats, None)` |
| Reward calculation | Binarized (49/100 → 0 if threshold=50%) | `get_validator_online_ratio(stats, Some(threshold))` |

---

### Impact Explanation

Consider two validators with equal stake, both below their respective kickout thresholds, and `validator_max_kickout_stake_perc` set so only one can be kicked out:

- **Validator A**: blocks=90/100, chunks=90/100, endorsements=49/100 (below 50% cutoff)
- **Validator B**: blocks=85/100, chunks=85/100, endorsements=51/100 (above 50% cutoff)

**Current behavior (raw sort):**
- A's online ratio ≈ (0.90 + 0.90 + 0.49)/3 ≈ 0.763 → A sorted higher → A exempted from kickout
- B's online ratio ≈ (0.85 + 0.85 + 0.51)/3 ≈ 0.737 → B kicked out

**Correct behavior (binarized sort, matching reward formula):**
- A's online ratio ≈ (0.90 + 0.90 + 0.00)/3 ≈ 0.600 → A sorted lower → A kicked out
- B's online ratio ≈ (0.85 + 0.85 + 1.00)/3 ≈ 0.900 → B exempted from kickout

**For rewards (always binarized):**
- A receives zero reward (endorsement binarized to 0, uptime falls below `online_min_threshold`)
- B receives positive reward

The current code keeps A (zero-reward validator) in the validator set and ejects B (positive-reward validator). This is the wrong outcome: a validator that earns nothing is shielded from kickout while a better-performing validator is removed.

---

### Likelihood Explanation

This divergence is triggered at every epoch boundary where:
1. `validator_max_kickout_stake_perc` is configured below 100 (standard production configuration), and
2. At least one validator has an endorsement ratio near the `chunk_validator_only_kickout_threshold`.

Both conditions are routine in any epoch that includes chunk-validator-only nodes (stateless validation is active on mainnet). The bug fires silently — no error is raised, the wrong validator simply stays in the set.

---

### Recommendation

Pass the endorsement cutoff threshold into `get_sortable_validator_online_ratio` so the sorting formula matches the reward formula:

```rust
pub(crate) fn get_sortable_validator_online_ratio(
    stats: &BlockChunkValidatorStats,
    endorsement_cutoff_threshold: Option<u8>,   // add parameter
) -> BigRational {
    let ratio = get_validator_online_ratio(stats, endorsement_cutoff_threshold);
    ...
}
```

Then update the call site in `compute_validators_to_reward_and_kickout` to pass `Some(config.chunk_validator_only_kickout_threshold)`, matching the value used in `finalize_epoch`.

---

### Proof of Concept

The divergence is directly observable from the existing unit tests. `test_reward_stateless_validation_with_endorsement_cutoff` in `chain/epoch-manager/src/reward_calculator.rs` demonstrates that a validator with endorsement ratio 44.6% (below the 50% cutoff) receives **zero reward** when `endorsement_cutoff_threshold = Some(50)` is applied. [5](#0-4) 

Yet `get_sortable_validator_online_ratio` — used for kickout-exemption sorting — would compute a non-zero online ratio for that same validator (raw 44.6% endorsement included), ranking it above a validator whose endorsement ratio is just above the cutoff. The exemption sort therefore inverts the correct ordering, protecting the zero-reward validator and ejecting the positive-reward one. [6](#0-5) [7](#0-6)

}

fn get_endorsement_ratio(stats: &ValidatorStats, cutoff_threshold: Option<u8>) -> (u64, u64) {
    let (numer, denom) = if stats.expected == 0 {
        debug_assert_eq!(stats.produced, 0);
        (0, 0)
    } else if let Some(threshold) = cutoff_threshold {
        if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
    } else {
        (stats.produced, stats.expected)
    };
    (numer, denom)
}
