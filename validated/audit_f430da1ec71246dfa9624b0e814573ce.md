### Title
Storage Deposit Permanently Locked in Contract When GC Removes Blocks Paid for by Prior Relayers - (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc()` deletes block headers from on-chain storage without returning the NEAR storage deposit that was paid by the relayer who originally submitted those blocks. Because the deposit accounting in `submit_blocks()` is computed as a net delta (after GC has already run), the freed storage silently benefits the current caller while the original depositor's NEAR is permanently retained by the contract with no refund path.

---

### Finding Description

`submit_blocks()` is `#[payable]` and requires callers to attach NEAR tokens to cover the cost of new storage:

```rust
// contract/src/lib.rs:173-183
let amount = env::attached_deposit();
let initial_storage = env::storage_usage();

for header in headers {
    self.submit_block_header(header, self.skip_pow_verification);
}

self.run_mainchain_gc(num_of_headers);                          // GC runs first
let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage); // net delta
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());
```

The storage delta is measured **after** `run_mainchain_gc()` has already removed old blocks. If GC removes K blocks and the current batch adds M blocks, the net delta is `max(0, M - K)`. When `K >= M`, `diff_storage_usage = 0`, `required_deposit = 0`, and the current caller's full deposit is refunded.

The storage freed by GC corresponds to blocks that were paid for by a **previous** relayer call. That previous relayer's deposit entered the contract's balance and was consumed to satisfy NEAR's storage staking requirement. When GC removes those blocks, the storage requirement drops and those NEAR tokens become part of the contract's free balance — but the contract has no mechanism to track or return them to the original depositor.

`run_mainchain_gc()` is also directly callable by any unprivileged NEAR account (no role restriction, only the `#[pause]` guard which is inactive by default):

```rust
// contract/src/lib.rs:376-377
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

Inside, `remove_block_header()` unconditionally deletes entries from `headers_pool` and `mainchain_header_to_height` with no deposit accounting:

```rust
// contract/src/lib.rs:659-662
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
```

There is no per-block deposit ledger, no depositor address stored alongside each header, and no refund path anywhere in the contract.

---

### Impact Explanation

Every NEAR token deposited by a relayer to cover the storage cost of blocks that are subsequently garbage-collected is permanently retained by the contract. The original depositor has no way to recover those tokens. Over the lifetime of the contract — with `gc_threshold` set to ~52,704 blocks (one year of Bitcoin headers) — the cumulative locked deposit grows proportionally to the total number of blocks ever submitted. At NEAR's storage byte cost and typical header sizes, this represents a material and growing loss for relayers.

The broken invariant: **funds deposited to cover storage of a specific set of blocks must be returnable when those blocks are deleted**.

---

### Likelihood Explanation

GC is triggered automatically on **every** `submit_blocks()` call (line 181) with `batch_size = num_of_headers`. Once the mainchain exceeds `gc_threshold`, every new batch submission silently removes an equal number of old blocks, consuming the prior depositor's NEAR. This is not an edge case — it is the steady-state operation of the contract after the chain grows past the threshold. Additionally, any unprivileged NEAR account can call `run_mainchain_gc(batch_size)` directly to accelerate the removal of blocks whose storage was paid for by others.

---

### Recommendation

Introduce a per-block deposit ledger: store the depositor's account ID and the NEAR amount paid alongside each `ExtendedHeader` (or in a parallel `LookupMap<H256, (AccountId, NearToken)>`). In `remove_block_header()`, look up the recorded deposit and issue a `Promise::transfer` back to the original depositor. Apply the same refund logic inside `reorg_chain()` where old main-chain blocks are also removed without deposit accounting.

---

### Proof of Concept

1. Relayer A (authorized) calls `submit_blocks([h1, h2, ..., h100])` with a deposit of X NEAR. The contract stores 100 headers; `gc_threshold = 50` so GC removes 50 old blocks. Net storage delta = 50 headers. Required deposit = 50 × header_byte_cost. Excess is refunded. The deposit covering the 50 removed blocks is now absorbed into the contract's balance.

2. Relayer B (authorized) later calls `submit_blocks([h101, ..., h150])` with deposit Y. GC removes 50 more blocks (the ones Relayer A paid for in step 1). Net storage delta = 0. `required_deposit = 0`. Relayer B's full deposit Y is refunded. Relayer A's original deposit for those 50 blocks is permanently lost.

3. Alternatively, any unprivileged account calls `run_mainchain_gc(1000)` directly. Up to 1000 blocks are removed. No deposit is returned to any prior relayer. The freed NEAR remains in the contract's balance indefinitely.

The root cause is at:
- [1](#0-0)  — GC runs before the net storage delta is computed, silently zeroing the required deposit when GC frees as much as was added.
- [2](#0-1)  — `remove_block_header()` deletes state with no deposit refund.
- [3](#0-2)  — `run_mainchain_gc()` is publicly callable with no role restriction, allowing any account to trigger deposit-losing deletions.
