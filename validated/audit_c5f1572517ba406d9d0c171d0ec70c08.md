Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on LP-Position `owner` Instead of Transaction `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and checks `owner` (the LP-position recipient) against the allowlist instead. Any address not on the allowlist can bypass the guard by naming any allowlisted address as `owner`, depositing funds into a restricted pool while LP shares are credited to that named address. The pool admin's deposit restriction is fully ineffective from the moment of deployment.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` argument as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes and forwards both values to the extension: [2](#0-1) 

The interface confirms `sender` is the first parameter and `owner` is the second: [3](#0-2) 

`DepositAllowlistExtension.beforeAddLiquidity` leaves `sender` unnamed (discarded) and checks `owner`: [4](#0-3) 

The sibling `SwapAllowlistExtension.beforeSwap` correctly checks `sender` and discards `recipient`: [5](#0-4) 

The `setAllowedToDeposit` setter names its parameter `depositor`, confirming the intent is to gate the calling entity, not the LP-position owner: [6](#0-5) 

The `allowedDepositor` mapping is keyed `pool → depositor → bool`, and `isAllowedToDeposit` accepts a `depositor` argument — all confirming the intended gated identity is the caller, not the owner: [7](#0-6) 

No other guard in the `addLiquidity` path checks the caller's allowlist status. The `removeLiquidity` path enforces `msg.sender == owner`, but `addLiquidity` has no such constraint, making the owner/sender separation freely exploitable. [8](#0-7) 

## Impact Explanation

This is an admin-boundary break: a pool admin who deploys `DepositAllowlistExtension` to enforce a closed LP set (KYC gating, regulatory compliance, adversarial-liquidity prevention) receives zero protection. Any unprivileged address can supply liquidity to the restricted pool. The unauthorized caller's tokens enter the pool; LP shares are minted to the named allowlisted `owner`. The pool admin's configured access control is fully bypassed, violating the invariant that only authorized depositors can supply liquidity.

## Likelihood Explanation

The bypass requires no special privileges, no flash loan, and no oracle manipulation. The `allowedDepositor` mapping is public, so any external account can inspect it, pick any allowlisted address as `owner`, and execute the bypass in a single transaction. The only cost is gas and the deposited tokens (which go to the named `owner`). Every pool using `DepositAllowlistExtension` with a non-empty allowlist and `allowAllDepositors == false` is fully exposed from deployment.

## Recommendation

Replace the unnamed first parameter with `sender` and gate on it, matching `SwapAllowlistExtension`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` and calls `setAllowedToDeposit(pool, alice, true)`. `allowAllDepositors[pool] == false`.
2. Bob (not on the allowlist) calls `pool.addLiquidity(owner=alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `_beforeAddLiquidity(sender=bob, owner=alice, ...)` → extension receives `sender=bob` (discarded), `owner=alice`.
4. Check: `allowAllDepositors[pool] == false`, `allowedDepositor[pool][alice] == true` → no revert.
5. Bob's tokens are transferred into the pool via the liquidity callback; LP shares are minted to `alice`.
6. The deposit allowlist is fully bypassed: Bob added liquidity to a pool he was never authorized to touch.
