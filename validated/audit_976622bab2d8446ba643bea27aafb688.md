### Title
Linked Signer Can Overwrite Its Own Authorization to Hijack Subaccount Signing Authority — (`core/contracts/EndpointTx.sol`)

---

### Summary

The sequencer path for `LinkSigner` transactions in `EndpointTx.sol` accepts signatures from the currently linked signer (`allowLinkedSigner = true`). This allows a linked signer to craft and submit a `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with an attacker-controlled address. Because `WithdrawCollateral` also accepts linked signer signatures, the attacker can then drain the subaccount's collateral.

---

### Finding Description

`EndpointStorage.sol` maintains a mapping `linkedSigners` that maps a subaccount (`bytes32`) to a trusted signer address: [1](#0-0) 

When the sequencer processes a `LinkSigner` transaction in `processTransactionImpl`, it calls `validateSignedTx` with `allowLinkedSigner = true`: [2](#0-1) 

`validateSignedTx` (and its compact variant) passes `getLinkedSigner(sender)` — i.e., the currently stored `linkedSigners[sender]` — as the accepted alternate signer to the verifier: [3](#0-2) 

This means the currently linked signer can produce a valid `SignedLinkSigner` transaction (signed with its own key) that sets `signedTx.tx.signer` to any address it chooses, including a new attacker-controlled address. The result is that `linkedSigners[subaccount]` is unconditionally overwritten: [4](#0-3) 

There is no check that the signer of a `LinkSigner` transaction must be the subaccount owner (i.e., `address(uint160(bytes20(sender)))`). The slow-mode path does enforce this correctly via `validateSender`: [5](#0-4) 

But the sequencer path has no equivalent restriction.

---

### Impact Explanation

After overwriting `linkedSigners[subaccount]` with a new address it controls, the attacker's new address is accepted as a valid signer for `WithdrawCollateral`, which also uses `allowLinkedSigner = true`: [6](#0-5) 

The attacker can then sign `WithdrawCollateral` transactions to drain the subaccount's collateral to an arbitrary address. The subaccount owner retains their own signing key but faces a race condition against the attacker's withdrawal. The corrupted state is: `linkedSigners[subaccount]` points to an address the subaccount owner never authorized.

---

### Likelihood Explanation

Any address that has been granted linked signer status by a subaccount owner can execute this attack. Linked signers are commonly used for automated trading bots or delegated operators — entities that are trusted for routine operations but should not be able to escalate their own authority. The attack requires only that the linked signer submit a valid `SignedLinkSigner` transaction to the sequencer, which is a standard protocol entry point.

---

### Recommendation

The `LinkSigner` transaction type in the sequencer path should not accept linked signer signatures. Change `allowLinkedSigner` to `false` for `LinkSigner` processing:

```solidity
// EndpointTx.sol — processTransactionImpl, LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // was: true — linked signer must not be able to re-link itself
);
```

This mirrors the correct access control already present in the slow-mode path, where `validateSender` enforces that only the subaccount owner's address can submit a `LinkSigner` transaction. [7](#0-6) 

---

### Proof of Concept

1. Subaccount owner `0xOwner` owns subaccount `sub = keccak256(abi.encodePacked(0xOwner, subaccountName))`.
2. Owner submits a `LinkSigner` tx (slow mode or sequencer) setting `linkedSigners[sub] = 0xLinked`.
3. `0xLinked` crafts a `SignedLinkSigner` struct: `sender = sub`, `signer = bytes32(uint256(uint160(0xAttacker)))`, `nonce = currentNonce`, signed by `0xLinked`'s private key.
4. `0xLinked` submits this to the sequencer.
5. `processTransactionImpl` reaches the `LinkSigner` branch; `validateSignedTx(..., true)` calls `getLinkedSigner(sub)` → returns `0xLinked`; verifier accepts the signature.
6. `linkedSigners[sub]` is now `0xAttacker`.
7. `0xAttacker` signs a `SignedWithdrawCollateral` tx: `sender = sub`, `amount = fullBalance`, `nonce = currentNonce+1`.
8. `processTransactionImpl` reaches `WithdrawCollateral`; `validateSignedTx(..., true)` calls `getLinkedSigner(sub)` → returns `0xAttacker`; verifier accepts.
9. `clearinghouse.withdrawCollateral` transfers the subaccount's full collateral balance out. [8](#0-7)
