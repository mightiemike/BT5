### Title
`burnNlp` Enforces Only Maintenance Health Instead of Initial Health, Allowing Undercollateralized Subaccounts — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.burnNlp` checks `HealthType.MAINTENANCE` after burning NLP tokens, while every other user-initiated collateral-reducing operation (`withdrawCollateral`, `transferQuote`, `mintNlp`) enforces `HealthType.INITIAL`. A user can burn NLP tokens and deliberately land in a state where initial health is negative (undercollateralized by the protocol's own initial-margin standard) while remaining above the maintenance threshold and therefore unliquidatable.

---

### Finding Description

In `Clearinghouse.sol`, `burnNlp` deducts `nlpAmount` from the sender's NLP balance and applies a burn fee before crediting quote. The final health guard is:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) 

Compare this to `mintNlp`, which enforces the stricter `INITIAL` health type:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [2](#0-1) 

And `withdrawCollateral`, which also enforces `INITIAL` health (or `PNL` for the X account):

```solidity
IProductEngine.HealthType healthType = sender == X_ACCOUNT
    ? IProductEngine.HealthType.PNL
    : IProductEngine.HealthType.INITIAL;
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [3](#0-2) 

The protocol's two-tier health model is:
- **INITIAL** — required to open new positions, withdraw collateral, or transfer quote; enforced by `_isAboveInitial`.
- **MAINTENANCE** — the liquidation threshold; falling below it triggers liquidation.

By only checking MAINTENANCE in `burnNlp`, the protocol allows a user to exit the initial-margin safety band without triggering a revert or a liquidation. The comment in the code acknowledges the intent to block unhealthy subaccounts, but the wrong health type was used:

```solidity
// Burning NLP can decrease health if the burn fee exceeds the health improvement
// from the withdrawal. This check prevents malicious actors from deliberately
// creating unhealthy subaccounts through NLP burns.
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [4](#0-3) 

---

### Impact Explanation

After a successful `burnNlp` call that leaves the sender with `INITIAL health < 0` but `MAINTENANCE health >= 0`:

1. The subaccount is undercollateralized by the protocol's own initial-margin standard.
2. It cannot be liquidated (maintenance health is still non-negative), so the bad state persists indefinitely.
3. The subaccount is blocked from `withdrawCollateral` and `transferQuote` (both enforce INITIAL health), but the undercollateralized position remains open and accrues risk.
4. If market prices subsequently move against the user, the subaccount slides from `INITIAL < 0` to `MAINTENANCE < 0` with no intermediate safety net, potentially requiring insurance-fund coverage or socialization.

The corrupted state is: `spotEngine.balances[NLP_PRODUCT_ID][sender]` and `spotEngine.balances[QUOTE_PRODUCT_ID][sender]` are updated to values that produce `INITIAL health < 0`, violating the invariant that every user-initiated collateral-reducing action must leave `INITIAL health >= 0`.

---

### Likelihood Explanation

The entry path is fully user-controlled:

1. User signs a `BurnNlp` transaction (EIP-712, standard user action).
2. Sequencer submits it to `Endpoint.processTransactionImpl`.
3. `EndpointTx` validates the signature and calls `clearinghouse.burnNlp(...)`.
4. `burnNlp` applies the burn fee and checks only MAINTENANCE health. [5](#0-4) 

Any user who holds NLP tokens and simultaneously carries a leveraged spot or perp position can reach this state. The burn fee (`max(1e18, quoteAmount / 1000)`) is the mechanism that reduces health below the initial threshold while keeping it above maintenance. No privileged access, governance action, or sequencer compromise is required — the user simply submits a valid signed `BurnNlp` transaction.

---

### Recommendation

Replace `HealthType.MAINTENANCE` with `HealthType.INITIAL` in the post-burn health check inside `burnNlp`, consistent with every other collateral-reducing operation in the protocol:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) 

---

### Proof of Concept

1. Alice holds 1000 USDC of NLP tokens and a leveraged long perp position such that her `INITIAL health = +5 USDC` and `MAINTENANCE health = +50 USDC`.
2. Alice signs and submits a `BurnNlp` transaction for her full NLP balance.
3. `burnNlp` computes `burnFee = max(1e18, quoteAmount / 1000)`. Suppose the fee is 8 USDC.
4. Alice receives `quoteAmount - burnFee` quote tokens. Her NLP collateral is gone; the fee reduces her net health.
5. Post-burn: `INITIAL health = -3 USDC`, `MAINTENANCE health = +42 USDC`.
6. The `require(getHealth(..., MAINTENANCE) >= 0)` check passes (42 >= 0). The transaction succeeds.
7. Alice is now undercollateralized by initial-margin standards but cannot be liquidated. Her leveraged perp position remains open. If the market moves 0.1% against her, she crosses the maintenance threshold with no prior safety-net revert, and the insurance fund or socialization mechanism must absorb the shortfall. [6](#0-5)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
```
