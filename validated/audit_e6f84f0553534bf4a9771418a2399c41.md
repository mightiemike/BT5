### Title
Deposit Allowlist Bypassed via `owner`/`sender` Mismatch — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension.beforeAddLiquidity` checks the LP position `owner` parameter instead of the actual depositor (`sender`). Because `owner` is a free caller-controlled argument in `MetricOmmPool.addLiquidity`, any unprivileged address can bypass the allowlist by naming an already-allowlisted address as `owner`.

### Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` argument as `owner` into the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values verbatim: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first parameter (`sender`) and gates only on `owner`: [3](#0-2) 

The admin-facing setter is named `depositor`, signalling the intent is to gate the actual depositing address, not the position owner: [4](#0-3) 

Because `owner` is a free parameter with no other on-chain constraint (there is no `msg.sender == owner` check in `addLiquidity`, unlike `removeLiquidity`), any caller can supply an allowlisted address as `owner` and the guard passes unconditionally. [5](#0-4) 

### Impact Explanation

The deposit allowlist is the primary admin-configured access-control guard for restricted pools. Its complete bypass by any unprivileged address breaks the admin-boundary invariant. Concretely:

- An unauthorized actor can inject liquidity into a pool that is supposed to be closed to them, altering bin balances, per-share metrics, and the pool's token composition.
- If `OracleValueStopLossExtension` is co-configured, the injected liquidity changes `t0`/`t1` per-share metrics and can push or suppress watermark updates, interfering with the stop-loss guard for legitimate LPs.
- The depositor's tokens are irrecoverably locked in a position owned by the named `owner`; the depositor cannot retrieve them (only `owner` can call `removeLiquidity`).

### Likelihood Explanation

Exploitation requires only knowing one allowlisted address (discoverable from `AllowedToDepositSet` events emitted by `setAllowedToDeposit`) and calling `addLiquidity` with that address as `owner`. No privileged access, flash loan, or special setup is needed.

### Recommendation

Check `sender` (the actual depositor who pays tokens via callback) rather than `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to allow third-party deposits on behalf of allowlisted owners, both `sender` and `owner` should be checked and the semantics documented explicitly.

### Proof of Concept

1. Pool `P` is deployed with `DepositAllowlistExtension` as an extension.
2. Pool admin calls `setAllowedToDeposit(P, Alice, true)` — Alice is the only allowlisted depositor.
3. Bob (not allowlisted) calls:
   ```solidity
   pool.addLiquidity(
       owner = Alice,   // allowlisted address
       salt  = 0,
       deltas = <non-zero shares>,
       callbackData = <Bob pays via callback>,
       extensionData = ""
   );
   ```
4. `beforeAddLiquidity` is invoked with `sender = Bob`, `owner = Alice`.
5. The extension checks `allowedDepositor[P][Alice]` → `true`; the guard passes.
6. Bob's tokens are transferred into the pool; the LP position is minted to Alice.
7. Bob's tokens are permanently locked in Alice's position; Alice can call `removeLiquidity` to extract them. [3](#0-2)
