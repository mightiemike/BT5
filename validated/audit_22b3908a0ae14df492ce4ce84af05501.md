### Title
Bonus Rewards Permanently Lost When `ErrInsufficientBonusPool` Is Swallowed After State Mutation in `slashRedelegationPosition` — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` mutates the position's `LastEventSeq` and `LastBonusAccrual` checkpoints **and** decrements event `ReferenceCount` values in the store **before** checking whether the bonus pool has sufficient balance. When the pool is empty, it returns `ErrInsufficientBonusPool`. The caller `slashRedelegationPosition` deliberately swallows that error and then calls `setPositionWithState`, persisting the already-advanced checkpoints. All future calls to `processEventsAndClaimBonus` for that position start from the advanced `LastEventSeq`, so the processed segments are never replayed and the bonus is permanently lost.

---

### Finding Description

**Step 1 — Entry point.**
`BeforeRedelegationSlashed` (hooks.go:128) delegates unconditionally to `slashRedelegationPosition`. [1](#0-0) 

**Step 2 — `processEventsAndClaimBonus` mutates state before the pool check.**
Inside the event loop (claim_rewards.go:172–198), for every event the function:
- calls `pos.UpdateLastEventSeq(entry.Seq)` — advancing the in-memory checkpoint on the pointer (line 193), and
- calls `decrementEventRefCount` — **persisting** the decremented `ReferenceCount` to the KV store (line 196). [2](#0-1) 

After the loop, `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` further mutate the `pos` pointer (lines 215–217). [3](#0-2) 

Only then is `sufficientBonusPoolBalance` called (line 230). If the pool is empty, the function returns `nil, ErrInsufficientBonusPool` — **after** all mutations have already occurred. [4](#0-3) 

**Step 3 — The caller swallows the error and persists the mutated state.**
`slashRedelegationPosition` catches `ErrInsufficientBonusPool`, logs it, and continues execution (slash.go:56–63). It then calls `setPositionWithState` with the already-mutated `pos` (lines 71 or 77), writing the advanced `LastEventSeq` and updated `LastBonusAccrual` to the store. [5](#0-4) 

**Step 4 — Future claims skip the processed segments.**
`getValidatorEventsSince` uses `StartExclusive(pos.LastEventSeq)` (validator_events.go:70), so any subsequent call to `processEventsAndClaimBonus` starts strictly after the already-advanced sequence number. The events whose `ReferenceCount` was decremented are never replayed. [6](#0-5) 

---

### Impact Explanation

The position owner permanently loses all accrued bonus rewards for every bonded segment that was processed during the failed `slashRedelegationPosition` call. There is no recovery path: the checkpoints are advanced, the event reference counts are decremented (and events may be deleted if `ReferenceCount` reaches zero), and no retry mechanism exists. The `RewardsPoolName` module account balance is unaffected — the funds remain in the pool but are never disbursed to the rightful owner.

---

### Likelihood Explanation

The precondition — `RewardsPoolName` balance being zero or insufficient — is realistic. The pool is funded by governance or protocol allocation; it can be depleted by prior claims or simply never topped up. A validator double-sign slash (which fires `BeforeRedelegationSlashed` for any active redelegation on that validator) is an ordinary chain event. No privileged access or social engineering is required. The combination is uncommon but entirely reachable in production.

---

### Recommendation

Move all state mutations (checkpoint advancement, `decrementEventRefCount`) to **after** the pool balance check and the `SendCoinsFromModuleToAccount` call succeed. Alternatively, if the pool is insufficient, return the error without mutating `pos` or decrementing any reference counts, so the caller can either halt or retry without data loss. The `ErrInsufficientBonusPool` swallow in `slashRedelegationPosition` is intentional to prevent chain halts, but it must only be safe to swallow if no irreversible state has been written.

---

### Proof of Concept

1. Fund `RewardsPoolName` to zero (or drain it).
2. Create a tier position delegated to validator V.
3. Redelegate the position to validator V2; record the `unbondingId`.
4. Append at least one validator event for V2 with `ReferenceCount ≥ 1` so `totalBonus > 0`.
5. Call `BeforeRedelegationSlashed(ctx, unbondingId, smallShares)`.
6. Assert `processEventsAndClaimBonus` returned `ErrInsufficientBonusPool` (observable via the logged error).
7. Read `pos.LastEventSeq` from store — assert it advanced past the event appended in step 4.
8. Read the event's `ReferenceCount` from `ValidatorEvents` — assert it was decremented.
9. Fund `RewardsPoolName` with sufficient balance.
10. Call `ClaimTierRewards` for the position owner.
11. Assert the bonus paid is **zero** for the segment processed in step 5, confirming permanent loss.

func (k Keeper) getValidatorEventsSince(ctx context.Context, valAddr sdk.ValAddress, startSeq uint64) ([]EventEntry, error) {
	// Range from (valAddr, startSeq+1) to end of valAddr prefix.
	rng := collections.NewPrefixedPairRange[sdk.ValAddress, uint64](valAddr).
		StartExclusive(startSeq)

	var entries []EventEntry
	err := k.ValidatorEvents.Walk(ctx, rng, func(key collections.Pair[sdk.ValAddress, uint64], event types.ValidatorEvent) (bool, error) {
		entries = append(entries, EventEntry{Seq: key.K2(), Event: event})
		return false, nil
	})
	if err != nil {
		return nil, err
	}
	return entries, nil
}
