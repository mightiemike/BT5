### Title
Silent Bonus Checkpoint Advance on Insufficient Pool During Redelegation Slash Permanently Destroys Accrued Rewards - (File: x/tieredrewards/keeper/slash.go)

### Summary

When a redelegation slash fires against a tier position and the bonus rewards pool is empty, `slashRedelegationPosition` swallows the `ErrInsufficientBonusPool` error returned by `processEventsAndClaimBonus`. However, `processEventsAndClaimBonus` has already advanced the position's accrual checkpoints (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) in memory before it checks pool sufficiency. Those advanced checkpoints are then persisted by the subsequent `setPositionWithState` call. The accrued bonus is permanently destroyed: the position's accounting records the period as settled, but no tokens were ever transferred to the owner.

---

### Finding Description

`processEventsAndClaimBonus` in `x/tieredrewards/keeper/claim_rewards.go` updates the position's checkpoints unconditionally before it checks whether the pool can cover the computed bonus:

```go
// line 215-217
applyBonusAccrualCheckpoint(&pos.Position, blockTime)
pos.UpdateLastKnownBonded(bonded)
```

The event-sequence cursor is also advanced inside the loop (line 193: `pos.UpdateLastEventSeq(entry.Seq)`) and event reference counts are decremented in persistent state (`decrementEventRefCount`, line 196) before the pool check at line 230:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err   // checkpoints already mutated; ref-counts already decremented
}
```

`slashRedelegationPosition` calls this function and deliberately swallows `ErrInsufficientBonusPool` to avoid a chain halt:

```go
// slash.go lines 54-64
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
    } else {
        return err
    }
}
```

After swallowing the error, execution falls through to `setPositionWithState`, which persists the in-memory `pos` — including the already-advanced checkpoints — to the KV store:

```go
// slash.go lines 76-77 (partial slash path)
pos.Delegation.Shares = pos.Delegation.Shares.Sub(sharesToUnbond)
return k.setPositionWithState(ctx, pos, nil)
```

The result is that the position's `LastBonusAccrual` is set to the current block time, `LastEventSeq` is advanced past all pending events, and `LastKnownBonded` is updated — all without any bonus being paid. The next call to `processEventsAndClaimBonus` for this position will start from the advanced checkpoint, so the destroyed bonus period is never recoverable.

---

### Impact Explanation

The corrupted value is `pos.LastBonusAccrual` / `pos.LastEventSeq` / `pos.LastKnownBonded` stored in the tiered-rewards KV store. These checkpoints are the sole source of truth for which bonus period has been settled. Once they are advanced without payment, the corresponding accrued bonus tokens are permanently unclaimable by the position owner.

This is the direct analog to the external report's "unaccounted loss" pattern: the protocol's internal accounting (checkpoints) records the bonus as consumed, but the physical assets (pool tokens) were never transferred. Positions that claim bonus before the pool is drained receive their full entitlement; positions whose `BeforeRedelegationSlashed` hook fires while the pool is empty suffer a permanent, unrecoverable loss — a "first-out" advantage identical in structure to the Yearn divestment scenario.

---

### Likelihood Explanation

The preconditions are realistic and can arise in normal chain operation:

1. A user creates a tier position and calls `MsgTierRedelegate`, creating a redelegation entry tracked in `RedelegationMappings`.
2. The destination validator is slashed for double-signing or downtime during the redelegation's maturation window — a routine chain event requiring no privileged access.
3. The `BeforeRedelegationSlashed` hook fires. If the `tieredrewards` rewards pool has been drained by the `BeginBlocker` base-rewards top-up (which drains the pool every block when there is a shortfall), the pool balance is zero.
4. `processEventsAndClaimBonus` advances checkpoints, decrements event reference counts, then returns `ErrInsufficientBonusPool`.
5. The error is swallowed; the position is saved with advanced checkpoints and no bonus paid.

The pool can be empty legitimately (it is funded externally and drained by the BeginBlocker), so this is not a contrived edge case.

---

### Recommendation

**Short term:** In `slashRedelegationPosition`, use a branched (cached) context when calling `processEventsAndClaimBonus` so that all in-memory and persistent state mutations (checkpoint advances, reference-count decrements) are rolled back when the pool is insufficient. Only commit the cache if the call succeeds. If the pool is empty, leave the position's checkpoints unchanged so the accrued bonus remains claimable once the pool is replenished.

**Long term:** Separate the checkpoint-advance logic from the payment logic in `processEventsAndClaimBonus`. Checkpoints should only be advanced atomically with a successful payment. Alternatively, track "owed but unpaid" bonus as a debt on the position, payable when the pool is next replenished, mirroring the external report's recommendation to accumulate protocol debt.

---

### Proof of Concept

1. Position P is created on validator V; `LastBonusAccrual = T0`, `LastEventSeq = 0`.
2. Owner calls `MsgTierRedelegate`; a `RedelegationMappings[unbondingId] = P.Id` entry is created.
3. Bonus pool is drained to zero (e.g., by BeginBlocker top-up over several blocks).
4. Validator V is slashed; `BeforeRedelegationSlashed(unbondingId, sharesToUnbond)` fires.
5. `slashRedelegationPosition` calls `processEventsAndClaimBonus(&pos)`:
   - Loop advances `pos.LastEventSeq` past all pending events; `decrementEventRefCount` commits to store.
   - `applyBonusAccrualCheckpoint` sets `pos.LastBonusAccrual = blockTime` (T1).
   - `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`.
   - Function returns `nil, ErrInsufficientBonusPool`.
6. Error is swallowed; `setPositionWithState(ctx, pos, nil)` persists `LastBonusAccrual = T1`, `LastEventSeq = N`.
7. Owner later calls `MsgClaimTierRewards` after pool is replenished: `processEventsAndClaimBonus` starts from `LastBonusAccrual = T1`, `LastEventSeq = N` — the entire `[T0, T1]` bonus period is gone.
