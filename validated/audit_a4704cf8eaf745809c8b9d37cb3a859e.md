### Title
Imbalanced Open Interest Causes Non-Zero-Sum Funding Payments, Creating Protocol Funding Deficit — (`core/contracts/PerpEngineState.sol`)

---

### Summary

`updateStates` increments both `cumulativeFundingLongX18` and `cumulativeFundingShortX18` by the same scalar `paymentAmount`. Because `_updateBalance` multiplies that delta by each position's signed `amount`, the total vQuoteBalance transferred from longs does not equal the total vQuoteBalance transferred to shorts whenever long open interest ≠ short open interest. The difference is neither credited to nor debited from any protocol reserve, creating a real funding deficit (or surplus) on every funding tick where OI is imbalanced.

**`_updateBalance` — per-position application** [2](#0-1) 

```solidity
int128 cumulativeFundingAmountX18 = (balance.amount > 0)
    ? state.cumulativeFundingLongX18
    : state.cumulativeFundingShortX18;
int128 diffX18 = cumulativeFundingAmountX18 - balance.lastCumulativeFundingX18;
int128 deltaQuote = vQuoteDelta - diffX18.mul(balance.amount);
```

For a long of size `A`: `deltaQuote = -paymentAmount * A` (longs pay).  
For a short of size `-S` (S > 0): `deltaQuote = +paymentAmount * S` (shorts receive).

**Net vQuoteBalance change across all users:**

```
Σ(deltaQuote) = paymentAmount × (totalShortOI − totalLongOI)
```

- When `totalShortOI > totalLongOI` and `paymentAmount > 0`: shorts receive more than longs pay → **unbacked positive vQuoteBalance** created out of nothing.
- When `totalLongOI > totalShortOI` and `paymentAmount < 0`: longs receive more than shorts pay → same deficit in the opposite direction.

`openInterest` is a single combined field (long + short absolute notional), so the protocol has no per-side tracking to detect or correct this imbalance. [3](#0-2) 

---

### Impact Explanation

The excess vQuoteBalance held by the dominant side represents a claim on the protocol that has no corresponding asset backing it. When those positions are closed and `settlePnl` is called, the excess vQuoteBalance is converted to spot quote tokens, draining real collateral from the protocol. [4](#0-3) 

The `availableSettle` cap limits *immediate* extraction per settlement call, but it does not prevent the excess vQuoteBalance from inflating health scores, allowing the dominant-side users to open additional positions. Over time, as positions are closed and `availableSettle` is replenished by trade flows, the excess is fully extractable. The protocol has no mechanism to recapture the surplus or fund the deficit — `socializeSubaccount` only runs during liquidation and only adjusts the rates asymmetrically for a specific insolvent subaccount. [5](#0-4) 

---

### Likelihood Explanation

OI imbalance is the normal state of any real perpetual market. The funding rate exists precisely *because* OI is imbalanced. Every single `updateStates` call in a live deployment with unequal long/short OI silently creates a deficit or surplus. No special attacker action is required beyond holding a position on the dominant side.

---

### Recommendation

Track long OI and short OI separately. Apply `paymentAmount` scaled by the ratio of the opposite side's OI to the position's own side's OI, so that total payments equal total receipts:

```
fundingLong  += paymentAmount × (totalShortOI / totalLongOI)   // longs pay proportionally more when shorts are smaller
fundingShort += paymentAmount × (totalLongOI / totalShortOI)   // shorts receive proportionally less when longs are smaller
```

This ensures `totalLongOI × fundingLong = totalShortOI × fundingShort` at every tick, making funding zero-sum regardless of OI imbalance.

### Proof of Concept

```solidity
// Setup: 10 longs of size 1, 1 short of size 1
// paymentAmount = P > 0 (positive priceDiff)

// After updateStates:
//   cumulativeFundingLongX18  += P
//   cumulativeFundingShortX18 += P

// _updateBalance for each long (amount = 1):
//   deltaQuote = -P * 1 = -P
//   Total for 10 longs: -10P

// _updateBalance for the short (amount = -1):
//   deltaQuote = -P * (-1) = +P
//   Total for 1 short: +P

// Net vQuoteBalance change: -10P + P = -9P
// (longs paid 10P, shorts received only 1P — 9P vanishes)

// Reverse: 1 long, 10 shorts, positive priceDiff
// Net vQuoteBalance change: -P + 10P = +9P
// (shorts received 10P, longs paid only 1P — 9P created from nothing)
// → Protocol owes 9P with no backing asset
```

Assert: after `updateStates`, sum all `vQuoteBalance` deltas across all subaccounts. The sum is `paymentAmount × (totalShortOI − totalLongOI)`, which is non-zero whenever OI is imbalanced. This violates the zero-sum invariant required for protocol solvency. [6](#0-5)
