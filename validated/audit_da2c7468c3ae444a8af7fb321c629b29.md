### Title
Hardcoded `LastKnownBonded=true` at Position Creation Allows Bonus Accrual Over Unbonded Validator Periods — (`x/tieredrewards/keeper/position.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`createDelegatedPosition` unconditionally sets `LastKnownBonded=true` for every new position, regardless of the validator's actual bonded state at creation time. When `processEventsAndClaimBonus` later processes a BOND event for a validator that was already unbonded at position creation, it treats the entire gap from `LastBonusAccrual` to the BOND event timestamp as a bonded segment and pays bonus for it. This drains `RewardsPoolName` by the over-paid amount.

---

### Finding Description

**Root cause — hardcoded `true` in `createDelegatedPosition`:** [1](#0-0) 

```go
pos := types.NewPosition(id, owner, tier.Id, delAddr.String(), blockHeight, lastEventSeq, blockTime, true, blockTime)
```

The eighth argument (`lastKnownBonded`) is always `true`, independent of whether the validator is bonded, unbonding, or unbonded at the moment the position is created.

**How `processEventsAndClaimBonus` consumes this flag:** [2](#0-1) 

```go
bonded := pos.LastKnownBonded   // always true for a freshly-created position
segmentStart := pos.LastBonusAccrual
``` [3](#0-2) 

For each pending event, if `bonded == true` the code computes a bonus segment **before** updating the bonded flag from the event type. So when the first pending event is a BOND event (validator just unjailed), the code computes:

```
bonus = computeSegmentBonus(pos, tier, segmentStart, bondEventTime, evt.TokensPerShare)
```

…for the entire period `[position creation time, bond event time]`, during which the validator was actually unbonded. The `tokensPerShare` used is the rate captured at the moment of bonding (from `AfterValidatorBonded`). [4](#0-3) 

**The "defensive" trailing-segment guard does not help here:** [5](#0-4) 

```go
if bonded && val.IsBonded() {
    // trailing segment — only reached after all events are processed
}
```

This guard only prevents over-payment for the *trailing* (post-last-event) segment. It does nothing for intermediate segments computed inside the event loop.

**Genesis validation does not check consistency of `LastKnownBonded`:** [6](#0-5) 

`InitGenesis` validates `LastEventSeq` ordering and reference counts but never asserts that `LastKnownBonded` matches the validator's actual bonded state, so the same flaw is also reachable via a crafted genesis/migration import.

---

### Impact Explanation

The `RewardsPoolName` module account is debited by the over-paid bonus: [7](#0-6) 

```go
k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins)
```

The over-payment equals `shares × tokensPerShare × BonusApy × unbondedDuration / SecondsPerYear`. For a large position and a long unbonded period this can be substantial. The pool balance decreases by exactly this amount, constituting direct fund loss from the rewards pool.

---

### Likelihood Explanation

The attack path is fully reachable through normal `MsgServer` transactions:

1. Validator V is jailed → `AfterValidatorBeginUnbonding` fires, appending an UNBOND event (seq N) for existing positions. V is now unbonded.
2. Attacker creates a position on V via `MsgCommitDelegation` (or equivalent). `createDelegatedPosition` sets `LastEventSeq = N` (skipping the UNBOND event) and `LastKnownBonded = true`.
3. V unjails → `AfterValidatorBonded` fires, appending a BOND event (seq N+1) with `ReferenceCount` including the new position.
4. Attacker calls `MsgClaimRewards`. `processEventsAndClaimBonus` starts with `bonded=true`, processes the BOND event, and computes bonus for `[position creation time, unjail time]` — a period during which V was unbonded.

No governance, operator access, or privileged key is required.

---

### Recommendation

In `createDelegatedPosition`, query the validator's actual bonded state and pass it as `lastKnownBonded`:

```go
val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
if err != nil {
    return types.Position{}, err
}
pos := types.NewPosition(..., val.IsBonded(), blockTime)
```

Additionally, add a genesis validation check that `pos.LastKnownBonded` is consistent with the validator's bonded state at import time, or conservatively default it to `false` during `InitGenesis` when the validator is not bonded.

---

### Proof of Concept

```
1. Chain starts with validator V in bonded state.
2. V gets jailed → UNBOND event seq=1 appended (ReferenceCount=0 since no positions yet).
3. Attacker calls MsgCommitDelegation targeting V.
   → createDelegatedPosition: LastEventSeq=1, LastKnownBonded=true.
4. V unjails → BOND event seq=2 appended (ReferenceCount=1 for attacker's position).
5. Attacker calls MsgClaimRewards.
   → processEventsAndClaimBonus:
       bonded = true  (from LastKnownBonded)
       segmentStart = position creation time T0
       event[seq=2]: BOND at time T1
       → computeSegmentBonus(T0, T1, tokensPerShareAtBond) > 0  ← WRONG
       → bonus paid from RewardsPoolName
6. Assert: RewardsPoolName balance decreased; attacker received bonus for unbonded period.
```
