### Title
Missing Liquidator Health Check in `_finalizeSubaccount` Allows Liquidator Insolvency via Cumulative Positive PnL Settlement — (`core/contracts/ClearinghouseLiq.sol`)

### Summary

`_finalizeSubaccount` settles all positive `vQuoteBalance` from the liquidatee's perp products by debiting the liquidator's spot quote balance, but unlike `_handleLiquidationPayment`, it performs **no health check on the liquidator** after these transfers. If the liquidatee holds multiple perp products with large cumulative positive `vQuoteBalance`, the liquidator's quote balance can go deeply negative without any protocol enforcement, creating bad debt socialized across all depositors.

---

### Finding Description

**Entrypoint:** `liquidateSubaccountImpl` in `ClearinghouseLiq.sol`, called via `processTransactionImpl` in `EndpointTx.sol` when `txType == LiquidateSubaccount` and `txn.productId == type(uint32).max`.

**Flow:**

1. `liquidateSubaccountImpl` (line 603) checks `isUnderMaintenance(txn.liquidatee)` — the liquidatee must be under maintenance health. [1](#0-0) 

2. `_finalizeSubaccount` is entered (line 620). It first requires all perp `amount` fields to be zero (line 319), but **`vQuoteBalance` can remain non-zero** even when `amount == 0` — this represents accumulated unrealized PnL from closed positions. [2](#0-1) 

3. Lines 322–338 loop over all perp products. For each with `balance.vQuoteBalance > 0`, `_settlePnlAgainstLiquidator` is called: [3](#0-2) 

4. `_settlePnlAgainstLiquidator` (lines 259–270) transfers `pnl` quote **from the liquidator to the liquidatee** for each product: [4](#0-3) 

5. After all these transfers, `_finalizeSubaccount` returns `true` at line 412, and `liquidateSubaccountImpl` returns immediately at line 626 — **no health check on `txn.sender` is ever performed**. [5](#0-4) 

**Contrast with `_handleLiquidationPayment`**, which explicitly guards the liquidator's solvency: [6](#0-5) 

This check (`txn.sender == N_ACCOUNT || !isUnderInitial(txn.sender)`) is entirely absent in the finalization path.

---

### Impact Explanation

If the liquidatee has N perp products each with `vQuoteBalance = V`, the liquidator's quote balance is decremented by `N * V` cumulatively. If this sum exceeds the liquidator's available quote, the liquidator's quote balance goes negative. Since no health check is enforced post-finalization, the liquidator's account is left insolvent. This bad debt is subsequently socialized across all depositors via `socializeSubaccount` when the liquidator is eventually liquidated.

---

### Likelihood Explanation

The liquidatee state required (all perp `amount == 0`, multiple positive `vQuoteBalance`, negative spot quote making them under maintenance) is reachable through normal protocol usage: a user opens perp positions, accumulates positive funding/PnL, closes positions (zeroing `amount` while retaining `vQuoteBalance`), and ends up with a negative spot quote balance. The liquidator signs the finalization transaction, potentially without knowing the exact total `vQuoteBalance` at execution time (state can shift between signing and sequencer processing). The N_ACCOUNT bypass at line 575 also means the protocol's own liquidation bot is exempt from this check, but regular liquidators are not. [6](#0-5) 

---

### Recommendation

Add a liquidator health check at the end of `_finalizeSubaccount`, mirroring the guard in `_handleLiquidationPayment`:

```solidity
require(
    txn.sender == N_ACCOUNT || !isUnderInitial(txn.sender),
    ERR_SUBACCT_HEALTH
);
```

This should be inserted after the positive PnL settlement loop (after line 338) and before returning `true`.

---

### Proof of Concept

```
Setup:
- Liquidatee has 5 perp products, each with amount=0 and vQuoteBalance=+1000 USDC
- Liquidatee has spot quote balance = -3000 USDC (under maintenance health)
- Liquidator has spot quote balance = +2000 USDC

Execution:
- Liquidator signs LiquidateSubaccount with productId = type(uint32).max
- _finalizeSubaccount enters the positive PnL loop
- For each of 5 perps: _settlePnlAgainstLiquidator transfers 1000 USDC from liquidator to liquidatee
- Total deducted from liquidator: 5000 USDC
- Liquidator quote balance: 2000 - 5000 = -3000 USDC
- No health check fires; function returns true
- Liquidator is now insolvent; bad debt = 3000 USDC socialized to depositors

Assert: liquidator.quoteBalance >= 0 after finalization → FAILS
``` [4](#0-3) [3](#0-2)
