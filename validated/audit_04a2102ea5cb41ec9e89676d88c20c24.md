### Title
Negative `liquidationFees` on Liability Liquidations Drains Insurance Fund — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

Every spot (and spread) **liability** liquidation (`txn.amount < 0`) computes a negative `v.liquidationFees` and then executes `insurance += v.liquidationFees`, unconditionally decreasing the insurance fund instead of increasing it.

---

### Finding Description

**Step 1 — `getLiqPriceX18` always produces `liquidationPriceX18 < oraclePriceX18` for liabilities.**

For `amount < 0`, `_getWeightX18` returns `risk.shortWeightMaintenanceX18`. For every normal asset this weight is strictly less than `ONE` (e.g. 0.9e18). Therefore:

```
penaltyX18 = (shortWeightMaintenance - ONE) / 5   →  negative
```

The minimum-penalty clamp only enforces a floor on the absolute value, keeping the sign negative:

```solidity
if (penaltyX18 < 0) penaltyX18 = -MIN_NON_SPREAD_LIQ_PENALTY_X18;  // still negative
``` [1](#0-0) 

So `liquidationPriceX18 = oracle * (ONE + negative_penalty) < oraclePriceX18` for every liability liquidation.

**Step 2 — The fee formula multiplies a positive spread by a negative amount.**

```solidity
v.liquidationFees = (v.oraclePriceX18 - v.liquidationPriceX18)   // > 0
    .mul(LIQUIDATION_FEE_FRACTION)                                  // > 0
    .mul(txn.amount);                                               // < 0  ← sign flip
``` [2](#0-1) 

Result: `v.liquidationFees < 0` for every liability liquidation.

**Step 3 — The negative fee is unconditionally added to insurance.**

```solidity
insurance += v.liquidationFees;   // insurance DECREASES
lastLiquidationFees = v.liquidationFees;
``` [3](#0-2) 

There is no sign check, no `abs()`, and no guard anywhere in the call chain that prevents a negative value from reaching this line.

The same sign error exists in the spread branch: [4](#0-3) 

and the perp branch: [5](#0-4) 

**Step 4 — `lastLiquidationFees` does not compensate.**

`_finalizeSubaccount` temporarily subtracts `lastLiquidationFees` from the working insurance copy and adds it back at the end. When `lastLiquidationFees` is negative, this temporarily *inflates* the socialization pool and then deflates it again — it does not restore the insurance balance drained by step 3. [6](#0-5) 

---

### Impact Explanation

**Concrete example (spot liability, non-spread):**

| Variable | Value |
|---|---|
| `oraclePriceX18` | 1 000e18 ($1 000) |
| `shortWeightMaintenance` | 0.9e18 |
| `penaltyX18` | −0.02e18 |
| `liquidationPriceX18` | 980e18 |
| `txn.amount` | −10e18 (10 ETH liability) |
| `liquidationFees` | (1000−980)×0.5×(−10) = **−$100** |
| `insurance` change | **−$100** per call |

`LIQUIDATION_FEE_FRACTION = 500_000_000_000_000_000` (50%). [7](#0-6) 

Every liability liquidation drains the insurance fund by `(oracle − liqPrice) × 50% × |amount|`. An attacker who controls a subaccount with a large short spot position can repeatedly trigger liability liquidations (or simply wait for organic liquidations) to drain the insurance fund to zero, after which insolvent accounts can no longer be covered and socialization losses are imposed on all LPs.

---

### Likelihood Explanation

Liability liquidations are a normal, intended protocol operation. The bug fires on **every** such liquidation without any special precondition beyond the liquidatee being under maintenance health with a negative spot balance — a routine market event. No privileged access, oracle manipulation, or unusual configuration is required.

---

### Recommendation

Replace the raw `txn.amount` with its absolute value in the fee formula, or restructure the sign convention so that fees are always non-negative before being added to insurance:

```solidity
// Correct: use abs(txn.amount) so the fee is always non-negative
v.liquidationFees = (v.oraclePriceX18 - v.liquidationPriceX18)
    .mul(LIQUIDATION_FEE_FRACTION)
    .mul(txn.amount.abs());
```

Apply the same fix to all three branches (spot, spread, perp) in `_handleLiquidationPayment`. [8](#0-7)
