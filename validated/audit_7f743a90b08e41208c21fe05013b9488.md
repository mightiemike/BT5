### Title
`gc_threshold` Can Be Initialized to Zero, Permanently Disabling SPV Confirmation Enforcement — (File: `contract/src/lib.rs`)

---

### Summary

The `init` function accepts `gc_threshold` from `InitArgs` without validating that it is greater than zero. If initialized with `gc_threshold = 0`, the contract permanently disables meaningful SPV confirmation checks and causes the GC to aggressively remove all historical blocks on every `submit_blocks` call, with no recovery path.

---

### Finding Description

In `contract/src/lib.rs`, the `init` function stores `args.gc_threshold` directly into contract state without any validation:

```rust
gc_threshold: args.gc_threshold,
``` [1](#0-0) 

The `gc_threshold` field is used in two critical places:

**1. Confirmation bound check** in `verify_transaction_inclusion`:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [2](#0-1) 

If `gc_threshold = 0`, only `confirmations = 0` passes this check. Any caller requesting ≥ 1 confirmation is permanently rejected.

**2. GC trigger** in `run_mainchain_gc`:

```rust
if amount_of_headers_we_store > self.gc_threshold {
    let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
    let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
``` [3](#0-2) 

If `gc_threshold = 0`, the condition `amount_of_headers_we_store > 0` is always true (the chain always has ≥ 1 block). `total_amount_to_remove` equals the full chain size. Since `submit_blocks` calls `run_mainchain_gc(num_of_headers)`, the GC removes exactly as many blocks as were just submitted, keeping the chain at a constant size of 1 block. [4](#0-3) 

The `InitArgs` struct itself has no validation constraints on `gc_threshold`:

```rust
pub struct InitArgs {
    ...
    pub gc_threshold: u64,
    ...
}
``` [5](#0-4) 

---

### Impact Explanation

With `gc_threshold = 0`:

- `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` permanently reject any call with `confirmations >= 1`, making SPV proof verification with any security margin impossible for all consumer contracts.
- The GC removes all newly submitted blocks on every `submit_blocks` call, keeping only 1 block in the chain at all times.
- There is no setter for `gc_threshold` after initialization, so the broken state is permanent and unrecoverable without a full contract redeployment.

---

### Likelihood Explanation

The `init` function is called exactly once during deployment. A deployer who accidentally passes `gc_threshold: 0` — for example, via a default-initialized struct, a misconfigured deployment script, or a copy-paste error — permanently breaks the contract. Because no post-deployment setter exists for `gc_threshold`, the damage cannot be corrected without redeployment and state migration.

---

### Recommendation

Add a validation check in the `init` function immediately after receiving `args`:

```rust
require!(args.gc_threshold > 0, "gc_threshold must be greater than 0");
``` [6](#0-5) 

---

### Proof of Concept

1. Deploy the contract with `InitArgs { gc_threshold: 0, genesis_block_height: 0, ... }`.
2. Call `verify_transaction_inclusion` with `confirmations: 1`.
   - The call panics: `"The required number of confirmations exceeds the number of blocks stored in memory"`.
3. Call `submit_blocks` with N = 5 headers.
   - Internally, `run_mainchain_gc(5)` is called.
   - `amount_of_headers_we_store > 0` → true.
   - `total_amount_to_remove = amount_of_headers_we_store`, `selected_amount_to_remove = min(..., 5) = 5`.
   - 5 blocks are removed from the bottom of the chain.
   - The chain returns to size 1.
4. Repeat step 3 indefinitely — the chain never grows beyond 1 block, and SPV verification with any confirmation count ≥ 1 is permanently broken.
