### Title
Deposit Collateral to Zero-Address Subaccount Permanently Locks Funds - (File: `core/contracts/Endpoint.sol`)

### Summary
`depositCollateralWithReferral` accepts an arbitrary `bytes32 subaccount` parameter without validating that the embedded owner address (first 20 bytes) is non-zero. An attacker or mistaken user can deposit collateral into a subaccount owned by `address(0)`, permanently locking the transferred tokens inside the protocol with no recovery path.

### Finding Description
`depositCollateralWithReferral` is a `public` function in `Endpoint.sol` that accepts a caller-supplied `bytes32 subaccount`. It extracts the owner address as `address sender = address(bytes20(subaccount))`. If the caller passes a subaccount whose first 20 bytes are all zeros (e.g., `bytes32(0x000...000<12-byte suffix>)`), `sender` resolves to `address(0)`.

The function then calls `requireUnsanctioned(sender)`, but `address(0)` is not on any sanctions list, so this check passes silently. [1](#0-0) 

Tokens are then transferred from `msg.sender` to the clearinghouse via `handleDepositTransfer`. A `SlowModeTx` is queued with `sender: address(0)`. [2](#0-1) 

When the sequencer processes the slow-mode transaction, `processSlowModeTransactionImpl` calls `validateSender(txn.sender, sender)`, which checks `address(uint160(bytes20(txn.sender))) == sender`. Since both sides resolve to `address(0)`, this passes. [3](#0-2) 

`_recordSubaccount(txn.sender)` then registers the zero-address subaccount in the global registry, and `clearinghouse.depositCollateral(txn)` credits the balance to it. [4](#0-3) 

The `_recordSubaccount` function has no zero-address guard: [5](#0-4) 

No one controls `address(0)`, so no signed withdrawal transaction can ever be produced for this subaccount. The credited balance is permanently stranded.

Note: the non-referral `depositCollateral` entry point is safe because it constructs the subaccount from `msg.sender` directly and cannot produce a zero-address owner. The vulnerability is exclusive to the `public` `depositCollateralWithReferral` path. [6](#0-5) 

### Impact Explanation
Any tokens deposited to a zero-address subaccount are transferred to the `Clearinghouse` contract and credited to an uncontrollable subaccount. There is no admin recovery path for these funds in the contract. The `withdrawCollateral` path requires a valid signed transaction from the subaccount owner, which is impossible for `address(0)`. Deposited funds are permanently lost.

### Likelihood Explanation
`depositCollateralWithReferral` is a `public` function callable by any EOA or contract, including `DirectDepositV1.creditDeposit()` which calls it with an operator-supplied `subaccount`. A user or integrator who accidentally passes `bytes32(0)` or a misconfigured subaccount with zero address prefix will lose funds with no on-chain warning. The likelihood is medium: the path is reachable without any privilege, and the `DirectDepositV1` integration pattern makes accidental misconfiguration realistic. [7](#0-6) 

### Recommendation
Add a zero-address check at the top of `depositCollateralWithReferral` in `Endpoint.sol`:

```solidity
address sender = address(bytes20(subaccount));
require(sender != address(0), "ERR_ZERO_ADDRESS_SUBACCOUNT");
```

This mirrors the fix recommended in the external report and prevents any subaccount whose embedded owner is `address(0)` from being registered or credited. [8](#0-7) 

### Proof of Concept
1. Attacker (or mistaken user) calls:
   ```solidity
   endpoint.depositCollateralWithReferral(
       bytes32(0),   // subaccount: first 20 bytes = address(0)
       QUOTE_PRODUCT_ID,
       1_000_000,    // e.g. 1 USDC
       ""
   );
   ```
2. `sender = address(bytes20(bytes32(0))) = address(0)`. Sanctions check passes.
3. `handleDepositTransfer` pulls 1 USDC from the caller and sends it to `Clearinghouse`.
4. A `SlowModeTx` is queued with `sender = address(0)` and `txn.sender = bytes32(0)`.
5. Sequencer processes the tx: `validateSender(bytes32(0), address(0))` passes; `_recordSubaccount(bytes32(0))` registers the subaccount; `clearinghouse.depositCollateral` credits 1 USDC to `bytes32(0)`.
6. No withdrawal is ever possible. 1 USDC is permanently locked. [9](#0-8)

        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned

        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
