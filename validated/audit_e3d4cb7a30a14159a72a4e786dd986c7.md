### Title
Inconsistent Flag-Gating of Secp Opcodes Allows Bypass of `NO_UNKNOWN_OPS` in Mempool Mode - (File: `src/chia_dialect.rs`)

---

### Summary
The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches `secp256k1_verify` and `secp256r1_verify` via two structurally different paths with inconsistent flag enforcement. The 4-byte opcode path (opcodes `0x13d61f00` / `0x1c3a8f00`) executes the secp functions unconditionally, bypassing both the `ENABLE_SECP_OPS` flag and the `NO_UNKNOWN_OPS` check. The 1-byte opcode path (opcodes 64 / 65) is correctly gated by `ENABLE_SECP_OPS`. Because `ENABLE_SECP_OPS` is absent from `MEMPOOL_MODE` and `NO_UNKNOWN_OPS` is present, the 1-byte secp opcodes are rejected in mempool mode while the 4-byte secp opcodes execute unconditionally — the same underlying cryptographic operation is reachable through one encoding but not the other, with no flag able to close the gap.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method has two separate dispatch branches:

**Branch 1 — 4-byte opcodes (lines 157–183):** [1](#0-0) 

The code matches `0x13d61f00` → `op_secp256k1_verify` and `0x1c3a8f00` → `op_secp256r1_verify` and calls them directly. There is no check for `ENABLE_SECP_OPS`, and crucially, the branch returns before reaching `unknown_operator`, so `NO_UNKNOWN_OPS` is never consulted.

**Branch 2 — 1-byte opcodes (lines 248–249):** [2](#0-1) 

Opcodes 62 (keccak256), 63 (sha256tree), 64 (secp256k1_verify), and 65 (secp256r1_verify) are all gated by their respective flags. Without the flag, they fall through to `unknown_operator`, which then enforces `NO_UNKNOWN_OPS`.

**`MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`:** [3](#0-2) 

So in mempool mode:
- Opcode 64 / 65 → falls to `unknown_operator` → rejected by `NO_UNKNOWN_OPS` ✓
- Opcode `0x13d61f00` / `0x1c3a8f00` → dispatched directly → **executes unconditionally** ✗

The `unknown_operator` helper that enforces `NO_UNKNOWN_OPS`: [4](#0-3) 

The 4-byte secp path never calls this function.

The Python API exposes `ENABLE_SECP_OPS` as a public constant and accepts raw `u32` flags via `from_bits_truncate`, giving callers the impression that this flag fully controls secp availability: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

A CLVM program supplied by an attacker that encodes secp operations using the 4-byte opcode form (`0x13d61f00` or `0x1c3a8f00`) will:

1. **Execute in mempool mode** despite `NO_UNKNOWN_OPS` being set — the secp functions run, cost is charged, and the program succeeds or fails on signature validity alone.
2. **Bypass `ENABLE_SECP_OPS`** — any downstream caller (e.g., `chia_rs`) that withholds `ENABLE_SECP_OPS` expecting to disable secp operations is wrong; the 4-byte encoding remains fully active.
3. **Create an inconsistent mempool/consensus surface**: the mempool rejects 1-byte secp opcodes but accepts 4-byte secp opcodes for the same operation, making the mempool's "stricter" guarantee incorrect for this opcode class.

The corrupted invariant is: *"in mempool mode, `NO_UNKNOWN_OPS` causes all non-core operators to be rejected unless explicitly whitelisted."* The 4-byte secp opcodes violate this invariant silently.

---

### Likelihood Explanation

The attacker-controlled entry path is direct: the `program` and `args` byte slices passed to `run_serialized_chia_program` (Python) or `run_program` (Rust) are fully attacker-controlled. Encoding a secp call using the 4-byte opcode form requires only knowledge of the opcode values (`0x13d61f00`, `0x1c3a8f00`), which are documented in the source comments. No privileged access, social engineering, or dependency compromise is required. Likelihood is **medium** — the inconsistency is exploitable by any party who can submit CLVM programs, but the secp operations themselves do not produce harmful outputs beyond the inconsistency in flag enforcement.

---

### Recommendation

Apply the same flag guard to the 4-byte secp dispatch path that is applied to the 1-byte path. Specifically, after matching `0x13d61f00` or `0x1c3a8f00`, check `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` before dispatching, and fall through to `unknown_operator` if the flag is absent. This makes both encoding paths subject to identical access control, eliminating the inconsistency.

Alternatively, if the 4-byte secp opcodes are intentionally always-on (pre-hard-fork encoding), document this explicitly, remove `ENABLE_SECP_OPS` from the public Python API surface (or rename it to clarify it only controls the 1-byte aliases), and add a comment in the 4-byte dispatch block explaining why `NO_UNKNOWN_OPS` is intentionally not enforced there.

### Proof of Concept

Construct a CLVM program that invokes secp256k1_verify using the 4-byte opcode encoding and run it under `MEMPOOL_MODE`:

```python
import clvm_rs

# 4-byte secp256k1_verify opcode: 0x13d61f00
# Program: (0x13d61f00 pubkey msg sig)
# Encoded as a CLVM list with the 4-byte atom as operator.
# Even with MEMPOOL_MODE (which sets NO_UNKNOWN_OPS), this executes.

flags = clvm_rs.MEMPOOL_MODE  # includes NO_UNKNOWN_OPS, excludes ENABLE_SECP_OPS

# A minimal serialized CLVM program calling the 4-byte secp opcode.
# The secp call will fail on invalid inputs, but it will NOT be rejected
# as an unknown operator — proving the NO_UNKNOWN_OPS bypass.
result = clvm_rs.run_serialized_chia_program(
    program_bytes,  # encodes (0x13d61f00 <pubkey> <msg> <sig>)
    args_bytes,
    max_cost=10_000_000,
    flags=flags,
)
# Expected under correct behavior: EvalErr::Unimplemented (unknown op rejected)
# Actual behavior: EvalErr::Secp256Failed or success — secp function was called
```
