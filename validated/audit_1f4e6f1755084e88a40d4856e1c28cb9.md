### Title
Division-by-Zero in `socializeSubaccount` When `state.openInterest == 0` Permanently Blocks Insolvent Subaccount Finalization — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when a product's `vQuoteBalance` remains negative after insurance coverage. No guard exists for the case where `state.openInterest == 0`. Because `MathSD21x18.div` enforces `require(y != 0, ERR_DIV_BY_ZERO)`, the call reverts, permanently blocking `_finalizeSubaccount` for any insolvent subaccount whose negative `vQuoteBalance` exists in a product with zero global open interest.

**How `state.openInterest == 0` with `balance.vQuoteBalance < 0` is reachable:**

`state.openInterest` is the sum of `abs(balance.amount)` across all subaccounts for a product. It equals zero when every subaccount has `balance.amount == 0` — i.e., all positions are closed. `balance.vQuoteBalance` is the accumulated realized PnL and is independent of `balance.amount`; it persists as a negative value after a position is closed at a loss. [3](#0-2) 

Concrete state construction:
1. Alice opens long 1 unit at price 100 → `amount=1`, `vQuoteBalance=-100`.
2. Alice closes at price 80 → `amount=0`, `vQuoteBalance=-20`. `state.openInterest` returns to 0 (assuming no other open positions).
3. Alice's spot quote balance is also negative (she is insolvent).
4. Liquidator calls `liquidateSubaccountImpl` with `productId == type(uint32).max`.

**Finalization path — `ClearinghouseLiq.sol`:**

`_finalizeSubaccount` first requires `balance.amount == 0` for every perp product (line 319), which is satisfied. It then calls `perpEngine.socializeSubaccount` at line 386 with whatever insurance remains. [4](#0-3) [5](#0-4) 

Inside `socializeSubaccount`, for the product where `balance.vQuoteBalance < 0` and `state.openInterest == 0`, if insurance is insufficient to fully cover the deficit, execution reaches the division and reverts with `ERR_DIV_BY_ZERO`. There is no guard between lines 164 and 166 that skips or handles the zero-open-interest case. [6](#0-5) 

---

### Impact Explanation

The revert is **permanent**: the state that causes it (`state.openInterest == 0`, `balance.vQuoteBalance < 0`, insufficient insurance) cannot be changed by any other protocol path once all positions are closed. The insolvent subaccount can never be finalized. Its negative `vQuoteBalance` is never socialized, meaning the loss is never distributed to the protocol — the accounting identity that all losses must eventually be absorbed is broken. The subaccount remains in a zombie insolvent state indefinitely, violating the core solvency invariant.

---

### Likelihood Explanation

The scenario requires a product where all positions have been closed (low-activity or end-of-life market) while one subaccount retains a negative `vQuoteBalance` and becomes insolvent. This is a realistic edge case in any market that winds down or experiences a period of zero open interest. No privileged access, governance action, or external dependency is required — only normal trading followed by a liquidation attempt.

---

### Recommendation

Add a zero-check guard before the division in `socializeSubaccount`. If `state.openInterest == 0`, there are no other participants to socialize the loss against; the remaining deficit should either be absorbed by the protocol (written off) or the loop iteration should be skipped:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; write off the loss.
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(
            state.openInterest
        );
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
    _setState(productId, state);
    _setBalanceAndUpdateBitmap(productId, subaccount, balance);
}
```
